## Summary

Keep open: current main and v2026.6.9 still lack first-class MCP binary-to-channel relay, and the active implementation candidate is open rather than merged. The current code coerces MCP audio/resources to text, does not stage embedded MCP binary content as channel-deliverable media, and still needs maintainer product/security decisions for the relay contract.

## What This Changes

The issue requests a first-class OpenClaw path for MCP non-text tool results to be preserved, relayed as channel attachments, and rewritten out of model/transcript state.

## Best Possible Solution

Define and land a narrow maintainer-approved MCP media relay contract that stages safe MCP binary content as OpenClaw-owned artifacts, reuses existing outbound channel delivery, and rewrites model/transcript state without exposing a broad byte-sending API.

## Reproduction Assessment

Yes, at source level. Current main routes MCP tool results through `toAgentToolResult`, coerces audio/resource blocks to text, lacks MCP binary staging in `details.media`, and filters MCP-provenance local media paths before channel delivery.

## Solution Assessment

Unclear as proposed. The direction is valid, but the best permanent shape needs maintainer-approved product and security boundaries; the safest current path is the narrower host-owned artifact staging approach in the open candidate PR, with channel and hook semantics reviewed deliberately.
