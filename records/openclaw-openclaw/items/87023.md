## Summary

Keep open: current main and the latest release still do not provide first-class MCP binary-to-channel relay. Valid MCP images can remain in tool-result content, but audio/resources are coerced to text, embedded base64 is not staged as sendable channel media, MCP-provenance local media remains untrusted, and the linked PR #94663 is still open as a partial candidate implementation.

## What This Changes

Request first-class relay of non-text MCP tool-result content to user channel attachments through a documented hook/channel attachment contract with transcript rewrite semantics.

## Best Possible Solution

Define and land a narrow maintainer-approved MCP media relay contract that stages safe MCP binary content into OpenClaw-owned media, routes through existing channel outbound adapters, rewrites transcript/model context explicitly, and keeps credentialed channel sending behind a clear trust boundary.

## Reproduction Assessment

Yes, by source inspection on current main: MCP audio/resource content is converted to text, base64 image content is not converted to sendable media, and external MCP media paths are filtered unless HTTP.

## Solution Assessment

No current solution is complete; the right path needs a narrow SDK/media/channel contract and security review rather than an unrestricted public byte sink or hardcoded MCP forwarding.
