## Summary

Keep open: current main and the latest release still do not provide first-class MCP binary-to-channel relay, and the active implementation candidate at https://github.com/openclaw/openclaw/pull/94663 is open and security/product-sensitive rather than merged.

## What This Changes

The issue requests first-class handling of MCP image, audio, and resource content so OpenClaw can stage or relay binary tool output as channel attachments instead of flattening, filtering, or dropping it.

## Best Possible Solution

Define and land a narrow maintainer-approved MCP media relay contract that stages safe MCP binary content as OpenClaw-owned media, reuses existing outbound channel delivery, and rewrites model/transcript state without exposing unrestricted byte-sending APIs.

## Reproduction Assessment

Yes. Source inspection is enough to show the missing capability: current main coerces MCP audio/resources to text, does not stage embedded base64 image data as media, and filters MCP-provenance local media paths.

## Solution Assessment

Unclear as proposed: the direction is valid, but the best permanent shape needs maintainer-approved API and security boundaries. The safer current path is the narrower host-owned staging approach in the open candidate PR, with native channel and hook behavior reviewed as follow-up product/API decisions.
