## Label Justifications

- `impact:message-loss`: The issue is about non-text MCP tool output not reaching the user's chat as attachments.

## Summary

Keep open: current main has useful partial foundations for MCP content preservation, tool hooks, and bundled session attachments, but it does not yet provide the requested first-class MCP binary relay, public channel attachment sink, or transcript rewrite contract for this user path.

## What This Changes

Request first-class handling of non-text MCP tool results so binary content can be routed as channel attachments and rewritten out of transcripts through documented plugin or hook APIs.

## Best Possible Solution

Design a narrow plugin-SDK/media relay contract that preserves MCP binary content, lets trusted code deliver through the active channel without duplicating credentials, and rewrites persisted transcript content with focused tests across at least one direct and one gateway-style channel.

## Reproduction Assessment

No live current-main repro was run in this read-only review. Source inspection shows the missing end-to-end relay path: content can be preserved, but the available attachment API is bundled-only and after_tool_call cannot return a rewritten tool result.

## Solution Assessment

Unclear until maintainers choose the permanent API shape. The requested direction is plausible, but the safest solution is a narrow SDK/channel contract rather than hard-coding MCP media forwarding in core.
