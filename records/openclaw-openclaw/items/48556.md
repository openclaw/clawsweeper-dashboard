## Summary

Keep open. The PR targets a valid proxy/FormData bug, and current main still does not normalize FormData inside `makeProxyFetch()` or `resolveProxyFetchFromEnv()`, but this exact branch is stale against current proxy-env handling and needs concrete fixes before merge. The related #66243 is still open and broader, so it does not make this PR implemented or safely superseded yet.

## What This Changes

The PR adds a proxy-fetch helper that copies global `FormData` bodies into `undici.FormData` for both proxy fetch wrappers, with focused unit tests for conversion and passthrough behavior.

## Best Possible Solution

Land a narrow proxy-fetch fix that converts non-Undici FormData bodies in both wrappers, preserves current env proxy option semantics, clears stale multipart framing headers when rebuilding the body, keeps existing Undici FormData untouched, adds focused regression tests, and records the user-facing fix in `CHANGELOG.md`. Broader Request/request-like normalization should stay in a separate review path unless maintainers explicitly choose to combine it.

## Reproduction Assessment

Yes. The linked issue includes a focused Node/Undici reproduction, and current main still routes proxied media audio through `resolveProxyFetchFromEnv()` while the proxy wrapper forwards the global `FormData` body unchanged into `undiciFetch`. That is enough to reproduce the reported `[object FormData]` serialization on runtimes where global and Undici FormData classes differ.

## Solution Assessment

Partly. Fixing `src/infra/net/proxy-fetch.ts` is the right boundary for a drop-in proxy fetch wrapper, but this branch is not the best mergeable form until it preserves current env proxy options, clears stale multipart headers, handles filenames deliberately, and adds changelog coverage.
