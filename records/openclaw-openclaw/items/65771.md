Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep cron JSON errors raw:** `src/cli/cron-cli/shared.ts:51`
  - body: This changes the existing `error` JSON field from the raw run-log error into a display string prefixed with `Cause: ...`. The patch already adds a separate `cause` field, so keep `error` unchanged and put display text in a new/additive field or in human-readable alert text; otherwise existing `cron runs --json` consumers can break on upgrade.
  - confidence: 0.88
- **[P2] Use the full cron reason contract:** `src/cron/run-log.ts:78-89`
  - body: `CRON_FAILOVER_REASONS` omits valid `FailoverReason` / `CronFailoverReasonSchema` members such as `server_error`, `empty_response`, `no_error_details`, and `unclassified`. A run-log entry with one of those valid persisted reasons and no raw error text will lose `errorReason` before `cron.runs` or the CLI can surface it, so share the full enum or include every protocol-valid value with regression coverage.
  - confidence: 0.91
- **[P3] Handle the wait-run JSON shape:** `src/cli/cron-cli/shared.ts:34-35`
  - body: The helper returns unchanged unless the top-level payload has an `entries` array, but `cron run --wait` prints the completed entry under a nested `run` property. A failed manual run can therefore still omit the new cause display; normalize that shape too or add a focused test proving it is intentionally out of scope.
  - confidence: 0.78
