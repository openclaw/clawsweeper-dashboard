## Summary

Keep open. Current main still lacks the proposed full-transcript identifier survival pass, and the PR is pointed at the right compaction surface, but the patch has a blocking blind spot: it builds the source identifier set after pruning, so identifiers from pruned chunks can still disappear without triggering the strict retry path.

## What This Changes

The PR adds compaction helpers that extract opaque identifiers from transcript text, tool-call arguments, and tool-result content, then retries strict summaries when more than two identifiers are missing, with tests and a changelog entry.

## Best Possible Solution

Keep the narrowed compaction change, but update it so strict identifier survival is validated against every source message represented by the returned compaction summary, including pruned dropped chunks and the final capped output, while preserving the existing toolResult.details exclusion and policy-off behavior.

## Reproduction Assessment

Yes. The original gap is reproducible with a strict compaction summary that passes existing section/latest-ask checks while omitting more than two identifiers from tool arguments or results; the remaining PR bug is reproducible by putting those identifiers in messages moved to pruned.droppedMessagesList before the new source-set calculation.

## Solution Assessment

No. The proposed direction is maintainable, but this implementation computes the source set too late in the pipeline; the safer fix is to collect identifiers from the sanitized full compaction input or merge identifiers from droppedMessagesList before retry validation, then ensure the final capped summary still contains them.
