## Summary

The PR should stay open: it is a useful fix direction for the documented compaction identifier-preservation contract, but current head still permits identifier loss through pruned history, the strict-policy `> 2` loss threshold, and validation before final summary capping.

## What This Changes

The PR adds compaction identifier extraction across message content, tool calls, and tool results, then retries strict summarization when more than two identifiers are missing, with tests and a changelog entry.

## Best Possible Solution

Use the PR's richer extraction helpers, but validate the final returned summary against the full pre-compaction identifier set and make strict mode retry on any lost identifier, with regression coverage for pruning and capping.

## Reproduction Assessment

Yes. Source inspection shows current main has the documented strict-preservation contract but only audits a recent text-only sample, and PR head still leaves concrete identifier-loss paths through pruning, the strict threshold, and final capping.

## Solution Assessment

No, not yet. The helper extraction direction is maintainable, but the patch needs to move the source set earlier, retry strict summaries on any lost identifier, and validate the final capped return value before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Include pruned history in identifier survival checks:** `src/agents/pi-hooks/compaction-safeguard.ts:1153-1156`
  - body: The survival set is built from `messagesToSummarize` and `turnPrefixMessages` after oversized-history pruning and recent-turn splitting. If an identifier exists only in `pruned.droppedMessagesList` and the dropped-summary pass omits it, this new check never sees it and the retry does not fire.
  - confidence: 0.91
- **[P2] Retry strict summaries on any lost identifier:** `src/agents/pi-hooks/compaction-safeguard.ts:1262`
  - body: Strict compaction is documented as preserving literal identifiers, but this patch accepts one or two missing values because `identifierSurvivalFailed` only becomes true when more than two are lost. A single run ID, URL, commit, or session key can be the exact value the next turn needs.
  - confidence: 0.9
- **[P2] Validate identifiers on the returned summary:** `src/agents/pi-hooks/compaction-safeguard.ts:1260`
  - body: This computes lost identifiers against `summaryWithoutPreservedTurns`, before the returned summary is rebuilt and capped. A long summary can pass the new check and then lose the trailing `## Exact identifiers` body during `capCompactionSummaryPreservingSuffix`, so the returned compaction summary can still omit identifiers.
  - confidence: 0.84
