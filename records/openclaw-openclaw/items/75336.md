## Summary

Keep open. Current main still lacks the proposed identifier-survival check, and the PR is useful, but the patch misses identifiers from messages that the safeguard prunes into `droppedSummary` before the new extraction runs, so it does not fully validate the pre-compaction transcript it claims to cover.

## What This Changes

The PR adds compaction-safeguard helpers and runtime retry logic to extract opaque identifiers from transcript text, tool calls, and tool results, compare them to the generated summary, and retry strict-policy summaries with a preservation hint, plus tests and a changelog entry.

## Best Possible Solution

Update the PR so the survival source set is captured from the original pre-prune/pre-summary messages, or explicitly carries identifiers from `pruned.droppedMessagesList` into the comparison set, while still excluding stripped `toolResult.details`. Add a regression that forces the history-share pruning path, then rebase the changelog entry and run the compaction safeguard test plus the changed gate.

## Reproduction Assessment

Yes. The existing PR test demonstrates the base issue for normal summaries, and the remaining blocker is reproducible by forcing `pruneHistoryForContextShare`, placing more than two identifiers only in dropped older messages, and showing the new retry does not fire because extraction happens after pruning.

## Solution Assessment

No, not as currently patched. The direction is narrow and maintainable, but the implementation should source identifiers before pruning or merge dropped-message identifiers into the new comparison before it can claim full pre-compaction transcript survival.
