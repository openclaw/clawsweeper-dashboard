## Summary

Keep open: the PR is aimed at a real compaction bug, but the current patch still misses identifier-loss paths and lacks required after-fix real behavior proof.

## What This Changes

The PR adds identifier extraction from message text, tool-call arguments, and tool-result content, then retries compaction summaries when lost identifiers are detected, with tests and a changelog entry.

## Best Possible Solution

Keep the PR open and adjust the guard to scan the complete pre-compaction source set, enforce strict identifier survival consistently, validate the final returned summary, keep stripped details excluded, and add pruning/capping regressions plus real behavior proof.

## Reproduction Assessment

Yes, source-reproducible. Current main separates pruned dropped messages from kept messages, while the PR head only scans the kept post-prune set and also permits one or two lost strict identifiers.

## Solution Assessment

No, not yet. The retry hook fits the existing quality guard, but the patch must cover dropped chunks, strict single-identifier loss, and final summary capping before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Scan dropped history before declaring identifiers survived:** `src/agents/pi-hooks/compaction-safeguard.ts:1150-1154`
  - body: This survival set is built after pruning can replace `messagesToSummarize` with `pruned.messages`, while `pruned.droppedMessagesList` is only fed through `effectivePreviousSummary`. If an identifier appears only in a dropped chunk and the dropped-summary pass omits it, `lostIdentifiers` never sees it and the retry will not fire, leaving the full pre-compaction transcript guarantee incomplete.
  - confidence: 0.91
- **[P2] Retry strict summaries for every lost identifier:** `src/agents/pi-hooks/compaction-safeguard.ts:1262`
  - body: Under strict identifier preservation, accepting summaries with one or two lost identifiers still lets a single missing run ID, path, URL, or hash break the next turn. The linked bug asks the guard to retry when identifiers are lost, so this threshold keeps the central failure mode for small but critical losses.
  - confidence: 0.86
- **[P2] Validate identifier survival after final capping:** `src/agents/pi-hooks/compaction-safeguard.ts:1260`
  - body: This check compares against `summaryWithoutPreservedTurns`, but the returned summary is later rebuilt from `lastHistorySummary || summary` and capped. A long valid summary can pass the new guard and then lose the trailing `## Exact identifiers` section during `capCompactionSummaryPreservingSuffix`, so the final compaction result can still drop identifiers.
  - confidence: 0.78
- **[P3] Move the changelog entry out of Breaking:** `CHANGELOG.md:544`
  - body: The added entry describes a compaction quality-guard fix, not an incompatible config or runtime break. Publishing it under `### Breaking` would misclassify the release note; it belongs under Changes or Fixes during landing.
  - confidence: 0.82
