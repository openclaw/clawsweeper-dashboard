## Summary

Keep open. The approach is aimed at a real compaction bug, but the current PR still misses identifiers from pruned dropped history, allows one or two strict-policy identifiers to disappear, validates before final summary capping, and still lacks after-fix real behavior proof.

## What This Changes

The branch adds compaction identifier extraction from message text, tool-call arguments, and tool-result content, retries strict summaries when more than two identifiers are lost, adds tests, and adds a changelog entry.

## Best Possible Solution

Keep the PR open and adjust the guard to validate the final returned summary against the full pre-compaction source set, including pruned dropped messages, while excluding stripped toolResult.details and covering the pruning/capping paths with tests and real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main only seeds the existing strict audit from a sampled text view, while the PR still uses a post-prune source set and a >2 lost-identifier threshold that leaves reproducible loss cases.

## Solution Assessment

No, not yet. The extraction helper is directionally useful, but the fix needs to scan the complete pre-compaction source, retry for any strict-policy loss, and validate after final capping before it is the narrow maintainable solution.
