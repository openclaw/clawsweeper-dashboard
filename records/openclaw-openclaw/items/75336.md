## Summary

Keep open: the extraction-helper direction is useful, but PR head still has source-visible compaction paths where strict summaries can lose identifiers from pruned or preserved history, tolerate one or two missing identifiers, or pass validation before final capping removes identifiers.

## What This Changes

The branch adds compaction identifier extraction across message text, tool-call arguments, and tool results, then retries strict compaction summaries when lost transcript identifiers exceed its threshold, with focused tests.

## Best Possible Solution

Repair before merge by collecting identifiers from the full pre-prune compaction input, retrying strict mode on any lost identifier unless a different policy is explicitly documented, and validating the final capped returned summary with regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main has a strict identifier-preservation contract but seeds the audit from a sampled text-only message view, and PR head still has concrete source-visible loss paths through pruning, thresholding, and final capping.

## Solution Assessment

No, not yet. The richer extraction-helper direction is maintainable, but the patch needs full-transcript source selection, any-lost strict retries, and final-return validation before it is the best fix.

  - confidence: 0.9

  - body: This computes lost identifiers against `summaryWithoutPreservedTurns`, before the returned summary is rebuilt and capped. A long summary can pass the new check and then lose trailing `## Exact identifiers` content during final capping.
  - confidence: 0.84
