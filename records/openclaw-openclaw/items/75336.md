## Summary

Keep open. The PR targets a real documented compaction bug, but the current head still has source-visible loss paths through pruned messages, strict-mode thresholding, final summary capping, and credential-shaped tool arguments, and the supplied proof is test output rather than a real compaction run.

## What This Changes

The PR adds message-level identifier extraction, lost-identifier validation with retry hints in compaction, regression tests, and a changelog entry.

## Best Possible Solution

Validate the final returned summary against a redacted full pre-compaction identifier set, retry strict summaries on any lost required identifier, add pruning/capping/credential regression tests, and supply redacted live compaction proof.

## Reproduction Assessment

Yes. Current main is source-reproducible because it samples a recent text-only identifier seed, and PR head is source-reproducible for the remaining pruned-history, threshold, final-capping, and unredacted-argument paths.

## Solution Assessment

No, not yet. The extraction-helper direction is useful, but the proposed fix needs full-source coverage, any-loss strict retries, final-summary validation, and credential-aware filtering before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Include pruned history in survival checks:** `src/agents/pi-hooks/compaction-safeguard.ts:1153-1156`
  - body: The new survival set is built from `messagesToSummarize` plus `turnPrefixMessages` after oversized-history pruning. If an identifier exists only in `pruned.droppedMessagesList` and that dropped-summary pass omits it, the retry path never sees the loss.
  - confidence: 0.91
- **[P2] Retry strict summaries on any lost identifier:** `src/agents/pi-hooks/compaction-safeguard.ts:1262`
  - body: Strict preservation should not accept one or two missing opaque identifiers. A single run ID, commit SHA, URL, or session ID can be the exact value the next turn needs, and the existing strict audit fails whenever a supplied identifier is missing.
  - confidence: 0.89
- **[P2] Validate identifiers after final capping:** `src/agents/pi-hooks/compaction-safeguard.ts:1260`
  - body: The PR checks `summaryWithoutPreservedTurns`, but the returned summary is rebuilt and capped later with `capCompactionSummaryPreservingSuffix`. A long summary can pass the survival check and then lose the trailing exact-identifiers content during the final cap.
  - confidence: 0.82
- **[P2] Filter credentials before extracting tool arguments:** `src/agents/pi-hooks/compaction-safeguard-quality.ts:191-196`
  - body: `JSON.stringify(args)` feeds every tool argument value into opaque-identifier extraction. That can preserve credential-shaped values from fields such as `apiKey`, `authorization`, or `token` in the compaction summary and retry hint, which conflicts with the existing hardening that removed tokens/API keys from default preservation wording.
  - confidence: 0.78
