## Summary

Keep open. The branch is a useful fix direction, but latest head e9956e4 still has source-visible compaction correctness gaps: it builds the survival set after pruning, lets strict mode accept one or two lost identifiers, and validates before the final returned summary is capped.

## What This Changes

The branch adds opaque-identifier extraction from compaction messages, lost-identifier checks after summarization, strict-mode retry hints, and focused compaction safeguard tests.

## Best Possible Solution

Repair the guard so it validates the full pre-compaction identifier set, retries strict mode on any missing identifier, and rechecks the final capped summary with regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has a strict identifier-preservation contract but only audits a recent text-only sample, and PR head still has concrete source-visible loss paths through pruning, thresholding, and final capping.

## Solution Assessment

No, not yet. The extraction-helper direction is maintainable, but the patch needs full-transcript source selection, any-lost strict retries, and final-return validation before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Include pruned history in identifier checks:** `src/agents/pi-hooks/compaction-safeguard.ts:1153-1156`
  - body: The survival set is built from `messagesToSummarize` and `turnPrefixMessages` after oversized-history pruning and recent-turn splitting. If an identifier exists only in dropped messages and the dropped-summary pass omits it, this check never sees it and the retry cannot fire.
  - confidence: 0.91
- **[P2] Retry strict summaries on every lost identifier:** `src/agents/pi-hooks/compaction-safeguard.ts:1262`
  - body: Strict compaction is documented as preserving literal identifiers, but this patch accepts one or two missing values because `identifierSurvivalFailed` only becomes true when more than two are lost. A single run ID, URL, commit, or session key can be the exact value the next turn needs.
  - confidence: 0.9
- **[P2] Validate the final returned summary:** `src/agents/pi-hooks/compaction-safeguard.ts:1260`
  - body: This computes lost identifiers against `summaryWithoutPreservedTurns`, before the returned summary is rebuilt and capped. A long summary can pass the new check and then lose trailing `## Exact identifiers` content during final capping.
  - confidence: 0.84
