## Summary

Keep open: the linked bug is valid and the PR moves the Z.AI payload toward the right `thinking` object, but the patch now sends `reasoning_effort` for every Z.AI reasoning model. Official Z.AI docs and current OpenClaw provider docs scope effort control to GLM-5.2+, so this can regress GLM-5.1/default Z.AI setups.

## What This Changes

This PR changes the OpenAI-compatible Z.AI request payload from `enable_thinking` to `thinking: { type }` plus `reasoning_effort`, updates the compat type comment, and adds a focused OpenAI-completions test.

## Best Possible Solution

Keep the `thinking: { type }` correction, gate `reasoning_effort` to GLM-5.2+ or an explicit provider capability, and add coverage for both GLM-5.2 and a non-GLM-5.2 Z.AI model before asking for redacted live provider proof.

## Reproduction Assessment

Yes, source-level: current main still emits the stale Z.AI `enable_thinking` field, and the PR's new branch visibly sends `reasoning_effort` for all Z.AI reasoning models. I did not run a live credentialed Z.AI call in this read-only review.

## Solution Assessment

No. The `thinking: { type }` part is the right direction, but the safest fix must preserve the existing GLM-5.2-only effort boundary instead of applying `reasoning_effort` to all Z.AI models.
