Keep open: current main still serializes ordinary Telegram messages by chat/topic and still runs same-key inbound debounce chains, so the Telegram interrupt/steer delivery bypass is not implemented. ## What This Changes

The PR adds a Telegram chat-session cache, active-run-aware per-message grammY sequential keys, and an inbound-debouncer bypass for active interrupt/steer sessions, plus supporting run-active and queue-mode helpers.

## Best Possible Solution

Port the Telegram-specific delivery bypass onto current main as a narrow change, after or alongside the core active-run prerequisite, keeping unrelated branch churn out of the patch and preserving the maintainer-scoped behavior: faster Telegram interrupt/steer delivery, not Telegram polling-stall recovery.
