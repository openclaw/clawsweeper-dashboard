## Summary

Keep open. Current main still emits and persists generated Voice Call sessions under raw `voice:*` keys while the Gateway startup migration canonicalizes raw keys into `agent:<agentId>:...`, and an open linked PR currently references this issue but has not landed.

## What This Changes

The issue asks Voice Call `sessionScope` generated session keys to use the canonical agent-scoped format so per-phone caller continuity survives Gateway restarts.

## Best Possible Solution

Fix the Voice Call-owned resolver and regression tests so generated per-phone and per-call keys are canonical agent-scoped keys while preserving explicit caller-provided keys.

## Reproduction Assessment

Yes, source-reproducible. Current main generates raw Voice Call keys, persists them directly, and the startup migration canonicalizes raw keys to agent-scoped keys, while docs promise per-phone memory continuity.

## Solution Assessment

Yes, the narrowest maintainable solution is in the Voice Call resolver and its tests, not a core migration special case; the already linked PR should either be corrected and landed or closed before a replacement is started.
