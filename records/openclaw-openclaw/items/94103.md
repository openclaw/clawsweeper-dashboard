## Summary

Keep open: the linked Tailscale Serve duplicate-entry bug is still real on current main, but this PR changes the no-service startup path to run a broad `tailscale serve reset` by default, which can remove operator-managed Serve routes and still lacks live Tailscale proof.

## What This Changes

The PR adds best-effort Tailscale Serve cleanup before gateway Serve enablement and expands gateway tests around cleanup ordering and mode exclusions.

## Best Possible Solution

Land one canonical fix for the linked bug that removes only OpenClaw-owned stale Serve state or makes broad cleanup an explicit operator choice, with redacted live Tailscale before/after proof.

## Reproduction Assessment

Yes, there is a clear source-backed reproduction path: configure Serve mode, start and restart the gateway, then compare `tailscale serve status`. I did not execute a live Tailscale daemon here, so this is source-reproducible rather than freshly live-reproduced.

## Solution Assessment

No. Pre-cleaning before enablement is plausible, but this patch is too broad because the no-service branch turns an opt-in cleanup into a default reset of operator-owned Serve state.
