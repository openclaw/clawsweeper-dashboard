## What This Changes

The issue reports that default Telegram long-message auto-splitting in v2026.6.9 still truncates long structured assistant replies instead of delivering all chunks, with an optional request for per-chunk logging.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/49889 as the canonical tracker and land, rebase, or replace https://github.com/openclaw/openclaw/pull/84563 with live Telegram proof; track per-chunk logging separately only if maintainers still want that after the delivery fix.

## Reproduction Assessment

Yes at source level: current main can mark Telegram partial-stream finals delivered before follow-up chunks settle, while fallback logic later trusts that delivered flag. I did not run a live Telegram reproduction during this read-only review.

## Solution Assessment

No as a separate tracker: the best fix path is the existing canonical issue and open candidate PR that delay delivered-state success until all final chunks settle. The requested auto-split behavior remains the right end state, but duplicate tracking would fragment the repair and proof path.
