## Summary

Keep open: the PR targets a real gateway restart/update failure, but the SIGUSR1 token cleanup is still too narrow and leaves an external-allowed stale-token path unfixed.

## What This Changes

The PR clears ignored SIGUSR1 restart state, skips redundant macOS package-update restarts after a healthy LaunchAgent refresh, adjusts LaunchAgent timing and HTTP bind retries, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the focused lifecycle hardening, but clear the stale SIGUSR1 token immediately after failed authorization so both rejected and externally allowed paths can recover before the macOS update/restart changes land.

## Reproduction Assessment

Yes, at source level: an unconsumed emitted restart token causes later schedules to coalesce, and the PR still leaves the failed-authorization external-allowed branch able to call the scheduler before clearing that token. I did not run a live macOS launchd reproduction.

## Solution Assessment

No. The broader macOS restart direction is reasonable, but the SIGUSR1 rollback must cover all failed-authorization paths, not only the branch where external SIGUSR1 restarts are disabled.

## Review Findings
