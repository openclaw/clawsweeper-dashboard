## Summary

Keep open. Current main and the latest release still reject active local multimodal memory because only Gemini advertises structured multimodal embedding support, so this remains a valid provider/product decision rather than cleanup to close.

## What This Changes

The issue asks to restore or clarify `memorySearch.multimodal` behavior for configurations that used `provider: "local"` in 2026.2.12 but now require a Gemini-capable adapter.

## Best Possible Solution

Choose one explicit contract for multimodal memory: either implement and test local structured-input embeddings before relaxing the guard, or make the Gemini-only requirement clearer in validation, doctor/setup output, and docs.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"`, active `memorySearch.multimodal.enabled`, and image or audio modalities reaches the current validation guard; the local adapter lacks multimodal capability and the runtime also lacks `embedBatchInputs`.

## Solution Assessment

Unclear. The current guard is the narrowest safe behavior for the adapters present on main, but it does not decide whether OpenClaw should restore the old local workflow or clearly declare it unsupported.
