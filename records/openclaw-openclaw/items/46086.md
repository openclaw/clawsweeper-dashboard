## Summary

Keep open: current main still intentionally gates multimodal memory to providers that advertise cross-modal embedding support, and the built-in local provider does not. This should not be auto-closed as implemented; supporting local multimodal memory would need a product/API decision rather than a narrow bug cleanup.

## What This Changes

The issue asks for `memorySearch.multimodal` to keep working with the local memory embedding provider after an upgrade made it require a Gemini-capable provider.

## Best Possible Solution

Decide whether local cross-modal memory embeddings are in scope; if yes, add an explicit local/provider capability with tests and docs, otherwise document the Gemini-only contract more prominently and close by maintainer decision.

## Reproduction Assessment

Yes from source inspection: current main rejects multimodal memory for any configured provider adapter that lacks multimodal embedding support, and the built-in local adapter lacks that support. I did not run a live local embedding setup because this review is read-only.

## Solution Assessment

No, simply allowing `provider: "local"` through validation would not be the best fix because the downstream embedding path requires structured input support. The maintainable path is either real local multimodal embedding support or a clearer Gemini-only contract.
