## Summary

Keep open. Current main and the latest shipped release still reject active local multimodal memory, so this remains a real provider/product decision rather than a cleanup item with a safe close reason.

## What This Changes

Issue reports that enabling multimodal memory search stopped working with the local embedding provider after 2026.2.12 and now requires Gemini.

## Best Possible Solution

Choose one explicit multimodal memory contract: either add tested local structured-input embedding support that advertises the capability, or keep the fail-closed guard and make the Gemini-only requirement clearer in setup and diagnostics.

## Reproduction Assessment

Yes from source inspection. A config with `memorySearch.provider: "local"`, active image/audio `memorySearch.multimodal`, and `fallback: "none"` reaches the current validation guard; if bypassed, indexing still fails because the local provider lacks `embedBatchInputs`.

## Solution Assessment

Unclear. The current Gemini-only guard is the narrowest safe behavior for the adapters implemented on main, but maintainers still need to decide whether OpenClaw should restore a local multimodal workflow or document it as unsupported.
