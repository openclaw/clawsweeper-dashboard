## Summary

Keep open. Current main and the latest release still make active multimodal memory a Gemini-only contract, while the local provider remains text-only, so this is not a cleanup-close item and needs a provider/product decision.

## What This Changes

This issue asks whether local-provider multimodal memory search should keep working after the upgrade that made active multimodal memory require Gemini support.

## Best Possible Solution

Choose one explicit multimodal memory contract: implement tested local structured-input embeddings that advertise capability, or keep Gemini-only and improve upgrade diagnostics/docs around the unsupported local path.

## Reproduction Assessment

Yes, from source inspection. A config with provider local, active image/audio multimodal settings, and fallback none reaches the current provider-capability guard; if bypassed, indexing still fails because the local provider lacks embedBatchInputs.

## Solution Assessment

Unclear. The current Gemini-only fail-closed behavior matches the implemented adapters and docs, but restoring the reported local workflow would require a deliberate local multimodal embedding capability or an explicit decision to keep it unsupported.
