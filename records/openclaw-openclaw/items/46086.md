## Summary

Keep open. Current main still rejects active local multimodal memory because only adapters that advertise structured multimodal embedding support pass validation, and the bundled local adapter/runtime remain text-only; this is a real provider/product decision rather than cleanup to close.

## What This Changes

The issue asks to restore or clarify `memorySearch.multimodal` behavior for `provider: "local"` after it stopped working across the 2026.2.12 to 2026.2.13 upgrade path.

## Best Possible Solution

Choose one explicit contract: either implement local structured-input embeddings with adapter capability tests, or keep the Gemini-only guard and make the provider requirement unmistakable in validation, doctor/setup output, and memory docs.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"`, enabled image or audio multimodal settings, and `fallback: "none"` reaches the current validation guard, and the local runtime also lacks the structured-input method needed if that guard were bypassed.

## Solution Assessment

Unclear. The current guard is the narrowest safe behavior for the adapters present on main, but it does not decide whether OpenClaw should restore local multimodal behavior or explicitly declare it unsupported.
