## Summary

Keep open: current main intentionally limits active multimodal memory to adapters that advertise structured multimodal embedding support, while the built-in local provider is still text-only. Treating local multimodal memory as supported would be a new provider/product decision, not conservative cleanup.

## What This Changes

The issue asks to restore or allow `memorySearch.multimodal` with the local embedding provider after an upgrade began requiring a Gemini-capable provider.

## Best Possible Solution

Keep the current Gemini-only multimodal contract unless owners intentionally add a local adapter that advertises multimodal support, implements structured image/audio embedding inputs, and updates docs plus regression coverage.

## Reproduction Assessment

Yes for the current rejection path: source inspection shows `provider: "local"` with active multimodal memory is rejected because the local adapter lacks multimodal capability and structured input embedding. The asserted prior local multimodal behavior is not high-confidence because the issue lacks the exact model/config/logs and v2026.2.12's local embedding path was text-only.

## Solution Assessment

No for an automated bug fix: removing the guard would allow startup to pass but still fail when structured image/audio chunks reach a provider without `embedBatchInputs`. The maintainable solution is either an intentional local multimodal provider implementation or an explicit not-supported decision.
