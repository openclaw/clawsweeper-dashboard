## Summary

Keep open for maintainer product/provider decision: current source and docs intentionally gate multimodal memory to adapters that advertise multimodal support, while the report does not prove that `provider: "local"` is a currently supported multimodal contract.

## What This Changes

The issue asks to restore or allow `memorySearch.multimodal` with `provider: "local"` after an upgrade began requiring Gemini for multimodal memory.

## Best Possible Solution

Keep the current Gemini-only contract unless maintainers intentionally add a local multimodal embedding adapter; if local multimodal support is desired, implement an adapter that advertises multimodal support, implements structured input embedding, and updates docs and regression tests.

## Reproduction Assessment

Yes for the current failure path: source inspection shows `provider: "local"` with enabled multimodal memory is rejected because the local adapter does not advertise multimodal support or implement structured embedding inputs. I did not run a live command because this review is read-only and the reported local model/config were not provided.

## Solution Assessment

No for an automated bug fix: current docs and tests define multimodal memory around Gemini Embedding 2, so restoring local behavior needs a maintainer decision about product scope or a plugin/provider implementation path.
