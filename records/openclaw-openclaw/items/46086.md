## Summary

Keep open. Current main and the latest release still reject active local multimodal memory, and the remaining choice is a provider/product decision rather than a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to restore or clarify `memorySearch.multimodal` behavior for the built-in `local` memory embedding provider after newer versions require Gemini.

## Best Possible Solution

Define one explicit contract: either add tested local structured-input embedding support with capability metadata, or keep Gemini-only support and improve diagnostics/docs for local text-only embeddings.

## Reproduction Assessment

Yes, from source inspection. A config with `memorySearch.provider: "local"`, active image/audio `memorySearch.multimodal`, and `fallback: "none"` reaches the capability guard; if bypassed, indexing still fails because the local provider lacks `embedBatchInputs`.

## Solution Assessment

Unclear. The current fail-closed Gemini-only guard matches the implemented adapters and docs, but maintainers need to decide whether to restore local multimodal behavior or document it as unsupported.
