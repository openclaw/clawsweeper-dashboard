## Summary

Keep open: current main still limits active multimodal memory to provider adapters that advertise multimodal embedding support, and the built-in local embedding path remains text-only. Restoring local multimodal memory would be a provider/product-scope decision, not conservative cleanup.

## What This Changes

The issue asks to restore `memorySearch.multimodal` for `provider: "local"` after an upgrade from 2026.2.12 began requiring a Gemini-capable provider.

## Best Possible Solution

Keep the current Gemini-only multimodal contract unless maintainers intentionally add a local multimodal embedding adapter that advertises support, implements structured inputs, and updates docs plus regression tests.

## Reproduction Assessment

Yes for the current failure path: source inspection shows `provider: "local"` with active multimodal memory is rejected because the local adapter does not advertise multimodal support or implement structured input embedding. I did not run a live local model because this read-only review lacks the reporter's exact model and config.

## Solution Assessment

No for an automated bug fix: the current docs, tests, and runtime guard define multimodal memory around adapters with explicit structured-input support, currently Gemini. The maintainable path is a provider-backed implementation if maintainers choose to support local multimodal memory, not removing the guard.
