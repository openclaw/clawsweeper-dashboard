## Summary

Keep open. Current main and the latest shipped tag still make active multimodal memory a provider-capability-gated feature, while the bundled local provider remains text-only; the remaining question is whether OpenClaw should restore local multimodal support or document/validate the Gemini-only contract more clearly.

## What This Changes

The issue reports that enabling `memorySearch.multimodal` now requires Gemini even though the local memory provider worked for the reporter on 2026.2.12.

## Best Possible Solution

Choose one explicit contract for multimodal memory. If local multimodal is unsupported, improve validation, doctor/setup text, and docs so local-provider users get a clear Gemini or supported-provider migration path; if it should be supported, add a local adapter capability and real `embedBatchInputs` handling with dependency-backed tests before relaxing the guard.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"`, active `memorySearch.multimodal.enabled`, and image or audio modalities reaches `resolveMemorySearchConfig` and fails because the local adapter lacks multimodal capability; if that were bypassed, structured indexing would still fail because the local provider lacks `embedBatchInputs`.

## Solution Assessment

Unclear. The current Gemini-only guard is the narrowest safe behavior for the provider implementations present on main, but it does not answer whether the local-provider workflow should be restored or explicitly documented as unsupported.
