## Summary

Keep open: this is a focused fix for a real message-loss race, but it is not merge-ready because the new assistant acknowledgement is only wired into the direct active-requester completion wake while the completion fallback steer can still report delivery at the transcript-commit boundary.

## What This Changes

The PR adds an optional assistant-response wait after transcript commit for queued embedded-agent steering and enables it for direct active requester subagent completion wakes, with focused regression expectations.

## Best Possible Solution

Land one consumed-or-recoverable acknowledgement invariant for all completion active-wake paths, with fallback-steer coverage and redacted runtime proof showing the direct-primary failure then fallback case.

## Reproduction Assessment

Yes at source level: direct completion delivery can fail, dispatch can call fallback steering, fallback steering can queue with only transcript-commit proof, and `steered` maps to delivered before assistant consumption. I did not run an instrumented live reproduction in this read-only review.

## Solution Assessment

No. The assistant-acknowledgement helper is a plausible narrow fix, but applying it only to the direct active wake is too narrow because the completion fallback steer still has the original acknowledgement boundary.
