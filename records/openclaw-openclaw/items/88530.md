## Summary

Keep open: this is a focused iMessage idle-scan fix with useful proof, but the current patch can suppress all later no-target approval recovery after one transient first-discovery failure because it records completion before the RPC discovery pass succeeds.

## What This Changes

The PR adds per-account state so iMessage no-target approval recent-chat discovery runs once instead of every discovery interval, with a regression test for the bounded idle path.

## Best Possible Solution

Keep the idle scan bounded, but make the completed-account marker retry-safe by recording it only after a successful no-target discovery pass and adding a failed-first-call regression test.

## Reproduction Assessment

Yes from source inspection: on the PR head, a no-target discovery call marks the account complete before RPC discovery runs, and monitor-provider catches poll failures before the next 60-second discovery tick returns early. I did not run a live imsg reproduction in this read-only review.

## Solution Assessment

No. The owner boundary is right and the idle-scan goal is reasonable, but the completion marker is set too early; the safer fix preserves the bound while retaining retry recovery after a failed first discovery pass.
