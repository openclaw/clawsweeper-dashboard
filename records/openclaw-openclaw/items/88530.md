## Summary

Keep open: the PR targets a real iMessage idle-scan performance problem, but the current patch marks no-target discovery complete before recent-chat discovery succeeds, so a transient first `chats.list` failure can disable restart recovery until process restart.

## What This Changes

The PR adds per-account process-local state so iMessage no-target approval discovery scans recent chats once, plus a regression test for the bounded idle path.

## Best Possible Solution

Land a version that records no-target discovery completion only after successful discovery work, adds a failed-first-scan retry test, and explicitly accepts one-time startup recovery semantics.

## Reproduction Assessment

Yes by source inspection: current main schedules recurring iMessage discovery polls and the poller lists recent chats on every no-target discovery tick. I did not run live macOS/imsg proof in this read-only review.

## Solution Assessment

No: bounding idle discovery in the poller is the right layer, but marking completion before `listRecentChatIds` succeeds is not the safest implementation. The best fix is to mark completion after a successful discovery boundary and cover the failed-first retry case.
