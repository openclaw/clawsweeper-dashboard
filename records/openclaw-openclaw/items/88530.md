## Summary

Keep open: the PR addresses the previous `chats.list` retry bug, but it still marks no-target discovery complete before `messages.history` succeeds, so a first-pass history failure can suppress restart recovery until the gateway process restarts.

## What This Changes

The branch adds process-local per-account state to let iMessage no-target approval recent-chat discovery run once and adds tests for bounded scans plus retry after a first `chats.list` failure.

## Best Possible Solution

Keep the bounded idle scan design, but only mark no-target discovery complete after the recent-history pass finishes without fetch failures and leave the one-pass recovery policy as an explicit maintainer-approved tradeoff.

## Reproduction Assessment

Yes from source: current main schedules no-target discovery on startup and every 60 seconds, and the current poller lists recent chats whenever no targets exist and discovery is allowed. I did not run the gateway because this review is read-only.

## Solution Assessment

No, not yet: bounding idle scans in the poller is the right layer, but the completion marker is too early because per-chat history failures are swallowed after the account is marked complete.
