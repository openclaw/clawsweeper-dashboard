## Summary

Keep open: the latest branch fixes the prior retry defect, but it still intentionally changes iMessage no-target approval recovery from recurring discovery to one process-local pass, so maintainers need to accept that tradeoff before merge.

## What This Changes

The PR adds per-account process state to bound iMessage no-target approval recent-chat discovery to one successful pass and adds regression tests for bounded scans plus chat-list/history retry failures.

## Best Possible Solution

Land the poller guard only if maintainers accept one-pass no-target recovery; otherwise use a bounded backoff or freshness design that reduces idle scans while preserving periodic recovery.

## Reproduction Assessment

Yes from source: current main schedules iMessage approval discovery immediately and every 60 seconds, and the poller lists recent chats whenever no targets exist and discovery is allowed. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainer acceptance: the poller-local guard is the narrow performance mitigation and the latest patch fixes retry defects, but it is only the best fix if maintainers accept dropping recurring no-target recovery within a process.
