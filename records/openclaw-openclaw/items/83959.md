## Summary

Keep open: current main still has a small fixed Codex app-server startup-close retry window with no deliberate warm-up backoff, and the fresh v2026.5.18 production logs match that failure mode. The related shared-client work in https://github.com/openclaw/openclaw/issues/79495 is relevant but does not fully replace this narrower startup lifecycle issue.

## What This Changes

The issue requests a bounded backoff and larger startup-close retry window so Codex app-server replacement warm-up does not fail scheduled/background turns prematurely.

## Best Possible Solution

Add a bounded startup-close backoff and retry window in the existing Codex app-server startup lifecycle, preserve hard failure with a lifecycle-specific terminal error after exhaustion, and cover the path with focused startup retry and multi-agent cold-start tests.

## Reproduction Assessment

Yes, at source level: current main still has a three-attempt startup-close loop with no explicit backoff, and v2026.5.18 reporter logs show repeated startup closes exhausting that window. I did not run a live multi-agent Codex gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to adjust the existing startup-close recovery loop rather than changing provider routing or fallback policy. Broader multi-agent cold-spawn behavior should still be reconciled with https://github.com/openclaw/openclaw/issues/79495 and the active related PR.
