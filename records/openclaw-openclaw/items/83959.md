## Label Justifications

- `P1`: The issue describes a current-release Codex harness startup failure that can abort scheduled/background agent turns before task work starts.
- `impact:crash-loop`: The affected path repeatedly restarts and exhausts the Codex app-server startup loop, making it a process availability failure.

## Summary

Keep open: current main still has the Codex app-server startup-close loop capped at three attempts with no explicit warm-up backoff, and the new v2026.5.18 logs match that exhaustion path. Related shared-client work helps adjacent multi-agent churn, but the startup lifecycle retry window is not resolved yet.

## What This Changes

The issue asks OpenClaw to tolerate a short Codex app-server replacement warm-up window during startup-close recovery before failing a scheduled or background turn.

## Best Possible Solution

Repair the startup-close lifecycle path with a bounded backoff and sufficient retry window, keep the hard failure after that window, improve the terminal error signal, and coordinate with the open shared-client lease work so one implementation owns the process lifecycle.

## Reproduction Assessment

Yes, source-level: v2026.5.18 and current main both use the same three-attempt startup-close loop with no explicit backoff, and reporter logs show that loop exhausting on v2026.5.18. I did not run a live multi-agent Codex gateway in this read-only review.

## Solution Assessment

Yes directionally: the narrow maintainable fix is to adjust the existing startup-close retry window and tests rather than changing unrelated model/task behavior. The implementation should be reconciled with https://github.com/openclaw/openclaw/pull/83639 so shared-client lifetime and retry timing do not diverge.
