## Summary

Keep open: current main and the latest tagged release still have a three-attempt Codex app-server startup-close loop with no explicit warm-up backoff, while the issue comments add fresh v2026.5.18 production logs showing retry exhaustion during scheduled/background Codex turns. The shipped keyed shared-client work addresses the broader cross-agent eviction class, but it does not close this narrower startup lifecycle retry-window failure.

## What This Changes

The issue requests a bounded Codex app-server startup-close retry window that tolerates transient replacement-server warm-up before failing scheduled/background turns.

## Best Possible Solution

Add a small bounded backoff or readiness-aware wait plus enough startup-close attempts on the existing Codex app-server startup path, keep a hard terminal failure with a lifecycle-specific error after the bounded window, and cover both startup-close exhaustion and concurrent cold-spawn behavior with focused tests.

## Reproduction Assessment

Yes at source level: force the Codex startup path to throw `codex app-server client is closed` for three startup attempts and current main immediately exhausts the retry loop. A live multi-agent current-main run was not performed in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to adjust the existing startup lifecycle retry window and diagnostics rather than changing task/model fallback behavior. Any fix should be reconciled with the broader multi-agent cold-spawn work and the open timeout-recovery PR before landing.
