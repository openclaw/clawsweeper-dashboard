## Label Justifications

- `P1`: The issue describes current-release Codex harness startup failures that can abort scheduled/background agent turns before user work starts.
- `impact:crash-loop`: The core failure mode is repeated Codex app-server startup-close restart attempts ending in process availability failure.

## Summary

Keep open: current main and v2026.5.22 still use a three-attempt Codex app-server startup-close retry loop with no bounded warm-up backoff, and the latest field report says the same exhaustion is still visible on v2026.5.22. The keyed shared-client work related to https://github.com/openclaw/openclaw/issues/79495 helps a broader multi-agent eviction problem, but it does not resolve this narrower replacement-server readiness window.

## What This Changes

The issue asks OpenClaw to tolerate a short Codex app-server replacement warm-up window after startup-close failures before failing scheduled or background turns.

## Best Possible Solution

Add a bounded backoff and longer warm-up window to the existing Codex app-server startup-close loop, preserve a hard lifecycle-specific failure after the window, and cover the delayed-readiness case with focused regression tests.

## Reproduction Assessment

No live current-main reproduction was performed in this read-only review. Source-level reproduction is high confidence because three consecutive `codex app-server client is closed` startup errors exhaust the loop immediately and rethrow before any warm-up backoff can elapse.

## Solution Assessment

Yes, the narrow maintainable fix is to extend the existing startup lifecycle retry window with bounded backoff and clearer exhaustion diagnostics. This should avoid changing shared-client keying or adding a PI fallback policy in the same repair.
