## Summary

Keep open: the current Codex app-server startup path still has a small bounded startup-close retry loop with no explicit warm-up backoff, and the fresh v2026.5.18 logs match that exhaustion path. The related shared-client pool work is adjacent, but it does not by itself satisfy this issue's narrower retry-window behavior.

## What This Changes

Codex app-server startup should tolerate transient startup-close replacement-server warm-up with bounded backoff and a clearer terminal lifecycle failure.

## Best Possible Solution

Add bounded startup-close backoff and a larger warm-up retry window in the existing Codex app-server startup path, keep the hard failure after exhaustion, emit a distinct lifecycle-exhaustion signal, and cover the behavior with focused tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence and supported by v2026.5.18 logs: current main still exhausts three startup-close attempts without explicit backoff. I did not run a live multi-agent Codex gateway in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: adjust the startup lifecycle retry window and tests rather than changing provider fallback or broad shared-client ownership again. Related shared-client pooling may reduce one trigger, but it does not replace bounded startup warm-up handling.
