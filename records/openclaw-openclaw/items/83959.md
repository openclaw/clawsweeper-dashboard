## Summary

Keep open: current main still has a three-attempt immediate Codex app-server startup-close loop with no bounded warm-up backoff, and fresh v2026.5.18 logs show the same lifecycle failure after the replacement server later recovered.

## What This Changes

Request to make Codex app-server startup-close recovery tolerate a short replacement-server readiness window and report lifecycle exhaustion distinctly.

## Best Possible Solution

Add a bounded abort-aware backoff/readiness window to the existing Codex app-server startup-close retry path, keep a lifecycle-specific hard failure after exhaustion, and coordinate with the adjacent shared-client lifecycle PR before landing.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection shows the current retry window can exhaust without any bounded warm-up delay, and the issue comments provide production logs with the same startup-close failure shape.

## Solution Assessment

Yes directionally: the narrow maintainable fix is to adjust the existing startup-close recovery path with bounded backoff/readiness and focused tests, not to add a broad fallback or unrelated shared-client redesign.
