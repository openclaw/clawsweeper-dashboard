## Label Justifications

- `P1`: The issue reports current-release Codex harness startup failures that can abort scheduled/background and channel-triggered agent turns before user work starts.
- `impact:crash-loop`: The central failure is an app-server startup-close restart loop that exhausts and leaves the Codex runtime unavailable for the turn.
- `impact:message-loss`: When this hits an inbound channel or scheduled lane, the user-visible agent turn can fail before producing a reply or task result.

## Summary

Keep open: current main and the latest release still use a three-attempt Codex app-server startup-close loop without an explicit warm-up backoff, and fresh issue comments report the same exhaustion shape on v2026.5.22. The related keyed shared-client work addresses cross-agent eviction, but it does not close this narrower startup retry-window bug.

## What This Changes

Codex app-server startup-close recovery should tolerate a bounded replacement-server warm-up window before failing scheduled or channel-triggered agent turns.

## Best Possible Solution

Add a bounded backoff and larger startup-close warm-up window in the existing Codex app-server startup recovery path, keep a hard lifecycle-specific failure after exhaustion, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes at source level, but not by running a live gateway in this read-only review. Current main can be reproduced with a startup client that throws `codex app-server client is closed` for the current three-attempt window, and the issue comments include live v2026.5.18 and v2026.5.22 field reports with the same exhaustion shape.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the existing startup-close recovery loop, not in another shared-client ownership change. The fix should add bounded backoff and diagnostics while preserving hard failure when the app-server remains unavailable.
