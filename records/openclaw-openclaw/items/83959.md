## Summary

Keep open: current main still has a bounded Codex app-server startup-close retry loop without any explicit warm-up backoff, and the latest issue comment adds a second production log showing the same lifecycle exhaustion shape on v2026.5.18. The related shared-client keyed-pool work helps the broader cross-agent eviction story, but it does not settle this narrower startup retry window.

## What This Changes

The issue asks OpenClaw to tolerate a short Codex app-server replacement warm-up window after startup connection closes, with bounded retry/backoff and clearer lifecycle exhaustion errors.

## Best Possible Solution

Add bounded startup-close backoff and a larger retry window in the existing Codex app-server startup loop, keep a hard lifecycle failure after that window, and cover success-after-delayed-readiness plus exhaustion with focused tests while leaving broader cross-agent pooling policy to https://github.com/openclaw/openclaw/issues/79495.

## Reproduction Assessment

Yes, at source level: make `startOrResumeThread` throw `codex app-server client is closed` through the current three startup attempts and the loop rethrows without any bounded warm-up delay. I did not run a live multi-agent gateway in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the existing startup-close recovery path: add bounded delay/budget and a clear lifecycle exhaustion error. Changing fallback-to-embedded behavior or cross-agent cache policy would be broader work and should stay separate unless maintainers choose that path.
