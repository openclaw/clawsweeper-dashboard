## Label Justifications

- `P1`: The issue describes current-release Codex harness startup failures that can abort scheduled/background agent turns before user work starts.
- `impact:crash-loop`: The failure mode is repeated app-server startup-close recovery ending in retry exhaustion and process-level unavailability for the turn.

## Summary

Keep open: current main and the latest release still use a three-attempt Codex app-server startup-close loop with no explicit warm-up backoff, and fresh field reports show the same retry-exhaustion shape on released builds.

## What This Changes

The issue requests bounded Codex app-server startup-close recovery so scheduled/background turns do not fail before a replacement app-server is ready.

## Best Possible Solution

Add a small bounded backoff and larger startup-close retry window on the existing Codex app-server startup path, keep a hard terminal failure after the window, and cover the behavior with focused regression and live/release proof.

## Reproduction Assessment

Yes at source level: make `startOrResumeThread` throw `codex app-server client is closed` for more than the current three startup attempts and the loop rethrows without backoff. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to adjust the existing startup-close recovery loop rather than change shared-client ownership again. Current keyed shared-client behavior helps the broader multi-agent issue, but it does not solve the retry-window exhaustion here.
