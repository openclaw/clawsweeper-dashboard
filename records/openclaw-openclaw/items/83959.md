## Label Justifications

- `P1`: The issue reports current-release Codex harness startup failures that can abort scheduled/background turns before any task work starts.
- `impact:crash-loop`: The failure mode is repeated Codex app-server startup-close restart attempts ending in process-lifecycle exhaustion.

## Summary

Keep open: current main and the latest shipped release still use a three-attempt Codex app-server startup-close loop with no explicit warm-up backoff, and fresh v2026.5.18 logs match the reported retry exhaustion. The related keyed shared-client work helps the broader multi-agent churn, but it does not by itself close this narrower startup retry-window failure.

## What This Changes

The issue asks OpenClaw to make Codex app-server startup-close recovery tolerate a short replacement-server warm-up window before failing scheduled or background turns.

## Best Possible Solution

Add bounded startup-close backoff or readiness-aware waiting in the existing Codex app-server startup path, preserve a hard terminal failure after the window, and cover delayed replacement readiness without changing shared-client ownership again.

## Reproduction Assessment

No live current-main reproduction was run in this read-only review. The source path is high-confidence: current main and v2026.5.19 still have three immediate startup-close attempts, and the issue comments provide v2026.5.18 logs matching exhaustion.

## Solution Assessment

Yes directionally: the narrow maintainable fix is to adjust the existing startup-close recovery window and tests. It should be coordinated with the open lifecycle-hardening PR rather than solved by a broad shared-client redesign or PI fallback change.
