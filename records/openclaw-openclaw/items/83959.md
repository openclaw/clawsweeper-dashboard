## Label Justifications

- `P1`: The issue describes current-release Codex harness startup exhaustion that can abort scheduled/background agent turns before task work starts.
- `impact:crash-loop`: The reported failure is an app-server startup-close retry/exhaustion loop that affects Codex harness process availability.

## Summary

Keep open: current main and v2026.5.18 still use a three-attempt Codex app-server startup-close retry loop without an explicit warm-up backoff, and fresh reporter logs show the same exhaustion under multi-agent load. The shipped keyed shared-client registry helps the broader cross-agent eviction issue, but it does not resolve this narrower startup retry-window failure.

## What This Changes

Increase or tune Codex app-server startup-close retries/backoff so transient replacement-server warm-up does not prematurely abort agent turns.

## Best Possible Solution

Add a bounded startup-close backoff and larger retry window in the existing Codex app-server startup path, keep hard failure after the window, make exhaustion distinguish lifecycle startup failure, and leave broader multi-agent pool/cold-start ownership to the related follow-up work.

## Reproduction Assessment

Yes, at source level. A focused test can force startup `thread/resume` to throw `codex app-server client is closed` beyond the current three-attempt window and observe retry exhaustion; I did not run a live multi-agent gateway in this read-only review.

## Solution Assessment

Yes for the narrow issue: adjusting the existing startup-close retry window with bounded backoff is the maintainable fix. Broader cache-key, prewarm, or cold-spawn serialization changes should stay coordinated with https://github.com/openclaw/openclaw/issues/79495 rather than being bundled into this fix.
