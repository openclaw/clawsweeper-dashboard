## Label Justifications

## Summary

Keep open: current main and v2026.6.6 still have a three-attempt immediate Codex app-server startup-close retry loop with no backoff or readiness gate, while the related keyed shared-client work only fixes cross-agent eviction.

## What This Changes

The issue reports that Codex app-server startup-close recovery can exhaust before a replacement server is ready, failing scheduled or background turns before user work starts.

## Best Possible Solution

Add a readiness-aware bounded warm-up/backoff path to Codex app-server startup-close recovery, keep a hard lifecycle-specific failure after the window, and cover the path with focused regression tests.

## Reproduction Assessment

Yes for source-level reproduction: inject repeated `codex app-server client is closed` errors through `startCodexAttemptThread` and the current loop exhausts after three immediate attempts. A live current-main gateway reproduction was not run in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the existing Codex app-server startup lifecycle path, not in provider fallback or the already-fixed shared-client keying. The repair should add readiness/backoff behavior without adding a new config knob unless tests show a fixed bounded default cannot work.
