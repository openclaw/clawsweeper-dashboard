## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny config and cron sessions do not inherit `tools.subagents.tools`; the remaining behavior is per-job `payload.toolsAllow` plus delivery-mode handling for the message tool, so this is a valid enhancement that needs maintainer design.

## What This Changes

Request a cron-scoped tool restriction config, such as `tools.cron.tools.deny`, so isolated cron jobs cannot use tools denied for scheduled runs.

## Best Possible Solution

Design a cron-scoped tool policy or explicit cron inheritance rule, then update schema/types/help/docs and regression tests for isolated cron message-tool denial across delivery modes.

## Reproduction Assessment

Yes. Source inspection reproduces the current gap: strict config/types have no cron tool-policy subtree, and the runtime only applies `tools.subagents.tools` to subagent envelope sessions rather than cron sessions.

## Solution Assessment

Unclear until maintainers choose the policy shape. Adding `tools.cron.tools.allow/deny` or explicit inheritance from `tools.subagents.tools` is plausible, but precedence against existing policy layers and per-job `toolsAllow` needs a product/API decision first.
