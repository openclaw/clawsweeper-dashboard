## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny policy or explicit inheritance from subagent tool policy; the remaining shipped behavior is per-job `toolsAllow` plus delivery-mode message-tool handling, so the requested public config/security contract is still unresolved.

## What This Changes

Add a cron-scoped tool restriction option, or equivalent inheritance from subagent tool policy, so isolated scheduled jobs can deny tools such as `message`.

## Best Possible Solution

Define a public cron tool-policy contract, wire it into the fail-closed policy pipeline, and update schema/types/help/docs/generated config metadata plus focused cron regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main has no cron-scoped policy subtree in schema/types/docs, and runtime tool policy applies subagent policy only to subagent envelopes while cron relies on per-job `payload.toolsAllow`.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
