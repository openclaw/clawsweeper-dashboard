## Summary

Keep open: current main and the latest release still lack a cron-scoped `tools.cron.tools.allow/deny` policy or documented inheritance from `tools.subagents.tools`; current `toolsAllow` and delivery-message safeguards are useful partial controls, not the requested system-level cron policy.

## What This Changes

Request a cron-scoped tool deny configuration so isolated scheduled jobs can be prevented from using tools such as `message` even when subagent policy does not apply.

## Best Possible Solution

Maintain this as the cron-scoped policy tracker, then implement either explicit `tools.cron.tools.allow/deny` or documented cron inheritance from subagent policy after maintainers choose precedence and upgrade semantics.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes at source level for the missing capability: current main has no `tools.cron.tools` config and cron does not enter the subagent policy path.

## Solution Assessment

Unclear until maintainers choose the product/security shape; `tools.cron.tools.deny` is plausible, but the best fix must define precedence against existing policy layers and per-job `toolsAllow`.
