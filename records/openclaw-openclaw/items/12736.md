## Summary

Keep open: current main and v2026.6.8 still lack a cron-scoped tool policy such as `tools.cron.tools.allow/deny`, and cron sessions still do not inherit `tools.subagents.tools`. Current code has useful partial safeguards for per-job `payload.toolsAllow` and duplicate announce handling, but the requested operator-level cron deny policy remains a security/product decision.

## What This Changes

The issue requests a cron-scoped tool deny policy so isolated cron jobs can be prevented from using tools such as `message` directly.

## Best Possible Solution

Keep this open for maintainer product/security design, then implement a single cron tool-policy layer with schema/types/help/docs and regression tests across isolated cron delivery modes.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 have no `tools.cron.tools.allow/deny` config surface, and the runtime only applies `tools.subagents.tools` to subagent envelopes. I did not run a live cron job because the review is read-only.

## Solution Assessment

Unclear pending maintainer direction: `tools.cron.tools.deny` is a plausible narrow solution, but cron inheritance from `tools.subagents.tools` or delivery-mode-specific defaults may be safer depending on the desired precedence and upgrade behavior.
