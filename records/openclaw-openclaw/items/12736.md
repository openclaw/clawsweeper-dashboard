## Summary

Keep open: current main and v2026.6.5 still do not expose a cron-scoped tool policy or make cron runs inherit `tools.subagents.tools`. Existing cron delivery code has per-job `toolsAllow` and duplicate-delivery mitigation, but first-class hard restriction for cron jobs remains a product/security design item.

## What This Changes

Request to add `tools.cron.tools.deny` or equivalent cron inheritance so isolated cron jobs can be blocked from tools such as `message`.

## Best Possible Solution

Add one canonical cron tool-policy design in core, update schema/types/help/docs, define precedence with `toolsAllow` and delivery modes, and cover isolated cron `message` denial plus announce fallback behavior with focused regression tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current schema/types omit `tools.cron.tools`, and runtime subagent policy is only applied to subagent envelope sessions. I did not run a live cron job because this review is read-only.

## Solution Assessment

Unclear as an implementation choice: a cron policy or explicit inheritance is the right class of fix, but maintainers still need to choose precedence and default behavior across existing policy layers and cron delivery modes.
