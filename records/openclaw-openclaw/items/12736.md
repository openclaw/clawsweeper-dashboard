## Summary

Keep open: current main still has per-job cron `toolsAllow` and message-delivery safeguards, but no cron-scoped config policy or documented inheritance from `tools.subagents.tools`; the remaining work is a security/product design decision, not stale cleanup.

## What This Changes

Request to add a cron-scoped tool restriction policy such as `tools.cron.tools.deny` or make isolated cron runs inherit the existing subagent tool policy.

## Best Possible Solution

Keep the issue open until maintainers choose a cron tool-policy design, then implement an additive `tools.cron.tools.allow/deny` or explicit subagent-inheritance path with schema/types/docs and regression tests for announce-mode message denial and per-job `toolsAllow` precedence.

## Reproduction Assessment

Not applicable as a strict bug repro because this is a feature/config request, but the source path is high-confidence: current main lacks `tools.cron.tools.*` and cron sessions do not inherit the subagent policy path. Existing sanitized runtime evidence supports the user-visible duplicate-message motivation.

## Solution Assessment

Unclear: a cron-scoped deny option is plausible and source-backed, but maintainers need to decide whether cron should get its own policy layer or inherit subagent policy and how that composes with per-job `toolsAllow` and delivery modes.
