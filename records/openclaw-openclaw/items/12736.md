## Label Justifications

- `P2`: This is a meaningful cron/tool-policy improvement with security and message-delivery impact, but it is a feature/config design request rather than an urgent regression.
- `impact:security`: The request changes the enforcement boundary for which model-visible tools isolated cron jobs can call.
- `impact:message-loss`: The discussion includes duplicate visible cron delivery when direct `message` sends and automatic announce delivery both fired for one run.

## Summary

Keep open: current main and latest release still lack `tools.cron.tools.*` or documented inheritance from `tools.subagents.tools`; recent cron tool-surface and duplicate-announce safeguards are useful adjacent mitigations, but they do not implement the requested cron-scoped operator policy.

## What This Changes

The issue requests a cron-scoped tool deny policy, either as `tools.cron.tools.deny` or by making cron sessions inherit `tools.subagents.tools`.

## Best Possible Solution

Keep the existing runtime mitigations, then implement an additive cron tool-policy contract only after maintainers choose the precedence model and update schema, exported types, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing requested surface: current main and v2026.6.8 source show no `tools.cron.tools.*` config and no subagent-policy inheritance for cron sessions. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear: `tools.cron.tools.allow/deny` is a plausible explicit fix, but maintainers need to decide whether cron gets its own policy layer or inherits subagent policy and how it composes with per-job `toolsAllow` and delivery modes.
