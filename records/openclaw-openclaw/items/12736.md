## Summary

This issue should stay open: current main and the latest release still lack a cron-scoped tool policy or cron inheritance from `tools.subagents.tools`, while the discussion includes sanitized evidence that unrestricted cron `message` access can duplicate announce delivery output.

## What This Changes

The issue requests a `tools.cron.tools.deny` configuration surface, or equivalent inheritance from subagent policy, so cron jobs can be prevented from using tools such as `message` directly.

## Best Possible Solution

Design one canonical cron tool-policy path in core config/runtime, with explicit precedence and regression coverage for `message` denial under cron announce delivery.

## Reproduction Assessment

Yes, at source level: current main has no `tools.cron.tools.allow/deny` schema/type/runtime path, and cron sessions do not enter the subagent-policy branch. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy shape. A cron-specific allow/deny subtree or explicit subagent-policy inheritance could both solve the request, but either needs precedence and upgrade-safety decisions.
