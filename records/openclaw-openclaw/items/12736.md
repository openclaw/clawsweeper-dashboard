## Summary

Keep open. Current main still has no cron-scoped `tools.cron.tools.allow/deny` config surface or inheritance from `tools.subagents.tools`; the newer per-job `payload.toolsAllow` and delivery safeguards only partially address the reported operator need.

## What This Changes

The issue requests a cron-scoped tool restriction policy, such as `tools.cron.tools.deny`, or explicit cron inheritance from subagent tool policy.

## Best Possible Solution

Design a first-class cron tool policy or explicit cron inheritance rule, document precedence against existing policy layers and per-job `toolsAllow`, and add focused cron announce/message-tool regression coverage.

## Reproduction Assessment

Yes for the missing capability: current source and docs show no `tools.cron.tools.allow/deny` config and cron sessions do not inherit subagent policy. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy shape. A cron-wide config knob is plausible, but the safest solution must first define precedence and whether cron should inherit subagent policy or own a separate policy layer.
