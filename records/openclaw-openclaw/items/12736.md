## Summary

Keep open: current main and v2026.6.8 still do not provide a cron-scoped tool allow/deny policy or cron inheritance from `tools.subagents.tools`; the existing per-job `toolsAllow` and delivery safeguards are adjacent but do not satisfy the requested security/config boundary.

## What This Changes

Request a `tools.cron.tools.deny`-style config option, or explicit cron inheritance from subagent tool policy, so isolated cron jobs can be centrally restricted from tools such as `message`.

## Best Possible Solution

Preserve current per-job `toolsAllow` and delivery safeguards, then add one maintainer-approved cron policy or inheritance layer with explicit precedence, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request: current source inspection proves the requested cron config/runtime policy surface is absent. The motivating duplicate-delivery symptom is supported by sanitized discussion evidence and adjacent current tests/safeguards.

## Solution Assessment

Unclear: a cron allow/deny option is plausible, but maintainers still need to decide whether cron gets its own policy layer, inherits subagent policy, or relies on per-job `toolsAllow` plus delivery safeguards.
