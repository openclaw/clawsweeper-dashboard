## Summary

Keep open. Current main still has per-job cron `toolsAllow` plus global/per-agent, sandbox, subagent, and inherited tool-policy layers, but no cron-scoped allow/deny config or explicit cron inheritance rule; this remains a real feature/policy request rather than an implemented cleanup target.

## What This Changes

The issue requests a cron-scoped tool deny policy such as `tools.cron.tools.deny`, or explicit inheritance from subagent tool restrictions, so scheduled jobs cannot use selected tools.

## Best Possible Solution

Define either a `tools.cron.tools.allow/deny` contract or an explicit cron-inherits-subagent policy, document precedence, wire it into shared policy resolution, and add regression coverage for isolated cron jobs that deny `message` across delivery modes.

## Reproduction Assessment

Yes. Source inspection on current main shows no cron-scoped config/type/schema/docs surface, and the runtime policy pipeline has subagent and sandbox layers but no cron layer.

## Solution Assessment

Unclear. `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are both plausible, but maintainers need to choose the config shape and fail-closed precedence rules before implementation.
