## Summary

Keep open. Current main still lacks a cron-scoped tool policy and does not make cron sessions inherit `tools.subagents.tools`; isolated cron jobs are only constrained by global/agent policy plus per-job `payload.toolsAllow` and delivery-mode message-tool handling.

## What This Changes

Add a cron-scoped tool allow/deny configuration, or make isolated cron jobs inherit subagent tool restrictions, so cron sessions can be blocked from tools such as `message`.

## Best Possible Solution

Keep this open and define an explicit cron policy surface, preferably `tools.cron.tools.allow/deny`, then wire it through schema/types/help/docs and both tool-policy resolution paths while preserving per-job `toolsAllow` as an additional narrowing layer.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configuring `tools.subagents.tools.deny` cannot affect `agent:<id>:cron:<jobId>` sessions because cron and subagent session classifiers are separate and the runtime only applies subagent policy to subagent envelopes.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation needs maintainer choice between explicit `tools.cron.tools` and inheritance from subagent policy, plus a documented precedence rule.
