## Summary

Keep open: current main and latest release have the focused duplicate-announce safeguard, but they still do not provide a cron-scoped allow/deny policy or subagent-policy inheritance for cron sessions. The remaining work is a security-oriented product/config decision, not cleanup-close material.

## What This Changes

The issue requests a cron-scoped tool restriction policy such as `tools.cron.tools.deny`, or equivalent inheritance from subagent tool restrictions, to prevent cron jobs from using tools like `message` directly.

## Best Possible Solution

Preserve the current verified-delivery safeguard, then add one maintainer-approved cron policy layer only if OpenClaw wants restrictions beyond per-job `toolsAllow`, with schema/types/help/docs and focused tests.

## Reproduction Assessment

Not applicable as a feature request; the requested behavior is absent by current source inspection rather than a failing current-main bug reproduction. The motivating duplicate-delivery symptom is supported by sanitized v2026.5.18 discussion evidence and current regression tests for the narrower delivery safeguard.

## Solution Assessment

Unclear: a cron allow/deny option is plausible, but maintainers need to decide whether cron should get a general policy layer or whether per-job `toolsAllow` plus verified delivery is the right product boundary.
