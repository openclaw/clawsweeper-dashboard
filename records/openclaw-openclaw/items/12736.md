## Summary

Keep open. Current main still lacks a cron-scoped allow/deny policy or an explicit inheritance rule from subagent tool policy; the only cron-specific restriction surface I found is per-job `payload.toolsAllow` plus delivery-mode message handling, so this remains a valid config/product design request.

## What This Changes

Request a cron-scoped tool policy such as `tools.cron.tools.deny`, or cron inheritance from `tools.subagents.tools`, so isolated cron jobs cannot use restricted tools.

## Best Possible Solution

Define a cron-specific policy or explicit inheritance contract, update schema/types/help/docs/generated metadata, wire it into the shared policy pipeline with fail-closed precedence, and add regression coverage for isolated cron message-tool denial across delivery modes.

## Reproduction Assessment

Yes, by source inspection. Current main has no cron tool-policy subtree in schema/types/docs, and the runtime subagent policy path only applies to subagent envelope sessions while cron relies on per-job `payload.toolsAllow`.

## Solution Assessment

Unclear. `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are both plausible, but maintainers need to choose the public config shape and precedence rules before implementation.
