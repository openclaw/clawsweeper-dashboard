## Summary

Current main now prevents the reported duplicate announce by verifying direct `message` delivery to the configured target before fallback delivery, but it still has no general cron-scoped allow/deny policy for other tools. The remaining work is therefore a valid security-oriented feature that needs a maintainer product decision on scope, precedence, and defaults rather than a narrow automated fix.

## What This Changes

Add a cron-scoped tool policy, such as `tools.cron.tools.deny`, so operators can restrict tools available to isolated cron jobs.

## Best Possible Solution

Preserve the target-verified announce-delivery safeguard, then add one canonical cron allow/deny layer only if maintainers want restrictions beyond `message`, with documented precedence, fresh-install behavior, upgrade behavior, schema metadata, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. The motivating duplicate-delivery behavior is nevertheless supported by sanitized v2026.5.18 evidence and focused current-main regression tests.

## Solution Assessment

Unclear. The current source-delivery safeguard is the narrower solution for duplicate announce delivery, while a general cron allow/deny policy needs maintainer judgment about restrictions for other tools and policy precedence.
