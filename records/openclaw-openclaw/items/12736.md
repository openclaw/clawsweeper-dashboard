## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny policy or explicit inheritance from `tools.subagents.tools`, while recent sanitized evidence shows the missing enforcement can produce duplicate cron announce messages.

## What This Changes

Request a public `tools.cron.tools.deny` config option, or cron inheritance from subagent tool policy, so isolated cron jobs can be restricted from tools such as `message`.

## Best Possible Solution

Define a public cron tool-policy contract with fail-closed precedence, then wire schema/types/help/docs/generated config metadata and focused cron delivery regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main has no cron-scoped policy subtree in schema/types/docs, and cron runtime only threads per-job `toolsAllow` plus delivery-mode message-tool handling.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
