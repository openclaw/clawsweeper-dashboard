## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny config surface or explicit inheritance from subagent tool policy; per-job `payload.toolsAllow` and recent delivery safeguards address adjacent symptoms but do not provide the requested system-level enforcement.

## What This Changes

Request a cron-scoped tool policy, such as `tools.cron.tools.deny`, so isolated cron jobs can be prevented from using tools like `message` directly.

## Best Possible Solution

Define a first-class cron tool-policy contract with documented fail-closed precedence, then wire schema, exported types, help/labels/generated metadata, docs, and focused cron delivery/tool-policy regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has no cron-scoped policy in schema/types/docs, and cron session keys do not satisfy subagent policy checks; no live run was attempted because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented precedence contract first.
