## Summary

Keep open. Current main still has no cron-scoped tool policy and cron sessions do not inherit `tools.subagents.tools`, so the requested capability remains actionable.

## What This Changes

The issue requests a cron-scoped tool deny option, or cron inheritance from subagent tool policy, so isolated cron jobs can be restricted from tools such as `message`.

## Best Possible Solution

Define an explicit cron policy surface, preferably `tools.cron.tools.allow/deny`, then wire it through schema, exported types, help/generated metadata, docs, and both tool-resolution paths while leaving per-job `toolsAllow` as an additional narrowing layer.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `tools.subagents.tools.deny` cannot affect agent-scoped `cron:` sessions because cron and subagent classifiers are separate and the runtime only applies subagent policy to subagent envelopes.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation needs maintainer choice between an explicit cron policy and subagent-policy inheritance, plus a documented precedence rule.
