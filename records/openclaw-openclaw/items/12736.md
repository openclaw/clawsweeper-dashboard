## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny config or explicit inheritance from subagent tool policy, and the remaining design affects scheduled-job authorization and message delivery.

## What This Changes

The issue requests a `tools.cron.tools.deny` config option, or equivalent cron inheritance from `tools.subagents.tools`, so cron jobs can be system-restricted from tools such as `message`.

## Best Possible Solution

Define a cron-specific policy or explicit inheritance contract, wire it into the shared policy pipeline with fail-closed precedence, and update schema/types/help/docs/generated metadata plus focused cron regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main has no cron tool-policy subtree in schema/types/docs, and the runtime subagent-policy branch only applies to subagent envelope sessions while cron relies on per-job `payload.toolsAllow`.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented precedence contract first.
