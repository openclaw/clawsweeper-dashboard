## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny config policy or explicit cron inheritance from subagent policy; the existing per-job `toolsAllow` and cron delivery safeguards only partially cover the reported operator-control problem.

## What This Changes

Request a config-level cron tool restriction such as `tools.cron.tools.deny`, or explicit inheritance from `tools.subagents.tools`, so isolated scheduled jobs can be denied tools before model execution.

## Best Possible Solution

Define a first-class cron tool-policy contract, or explicit cron inheritance from subagent policy, with documented fail-closed precedence plus schema, exported types, docs, metadata/help, and focused cron delivery/tool-policy regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows no cron-scoped policy subtree in schema/types/docs, and cron sessions do not enter the subagent-policy path; no live run was attempted because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented precedence contract first.
