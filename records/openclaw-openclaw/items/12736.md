## Summary

Keep open: current main and v2026.6.5 still lack a cron-scoped tool policy or inheritance from `tools.subagents.tools`; per-job `payload.toolsAllow` and current delivery mitigations do not provide the requested hard operator deny policy.

## What This Changes

Request a cron-scoped tool restriction such as `tools.cron.tools.deny`, or cron inheritance from `tools.subagents.tools`, so scheduled agent runs can be prevented from using tools such as `message`.

## Best Possible Solution

Add one canonical cron tool-policy design in core, define its precedence against existing policy layers and delivery modes, update schema/types/help/docs, and cover isolated cron `message` denial plus announce fallback behavior with focused regression tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current schema/types omit `tools.cron.tools`, and runtime subagent policy is only applied to subagent envelope sessions. I did not run a live cron job because this review is read-only.

## Solution Assessment

Unclear as a final implementation choice: a cron policy or explicit inheritance is the right class of fix, but maintainers still need to choose precedence and default behavior across existing policy layers and cron delivery modes.
