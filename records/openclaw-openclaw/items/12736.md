## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny policy or explicit inheritance from `tools.subagents.tools`, and the latest issue discussion adds sanitized `v2026.5.18` evidence that cron announce delivery can duplicate visible messages when the model also uses the `message` tool.

## What This Changes

Request a cron-scoped tool restriction policy, such as `tools.cron.tools.deny` or explicit subagent-policy inheritance, for isolated cron runs.

## Best Possible Solution

Define a first-class cron tool-policy contract with documented fail-closed precedence, then wire schema/types/help/docs/generated config metadata and focused cron delivery regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main has no cron-scoped policy subtree in schema/types/docs, and cron runtime only threads per-job `toolsAllow` plus delivery-mode message-tool handling while subagent policy checks exclude cron sessions.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
