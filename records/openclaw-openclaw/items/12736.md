## Summary

Current main still has no cron-scoped tool policy. Recent cron delivery changes mitigate duplicate announce delivery but intentionally keep direct `message` tool use available, so the broader authorization request remains distinct and requires maintainer product/security judgment.

## What This Changes

Add a cron-scoped tool deny policy, or defined inheritance from subagent restrictions, for isolated cron jobs.

## Best Possible Solution

Add cron restrictions through the canonical tool-policy composition, define deny precedence against per-job `toolsAllow` and other scopes, align schema/types/help/docs, and cover fresh configuration plus existing-job upgrade behavior.

## Reproduction Assessment

Yes, at high-confidence source and documented-contract level: current main has per-job allow selection and delivery safeguards but no cron-scoped deny policy, and direct message-tool use remains supported.

## Solution Assessment

The enforcement goal is appropriate, but the exact proposed key is not yet proven to be the best shape; maintainers should first choose canonical policy precedence and compatibility behavior.
