## Summary

Keep open: current main and v2026.6.10 still lack a cron-scoped `tools.cron.tools.allow/deny` policy or documented inheritance from `tools.subagents.tools`; current per-job `toolsAllow` and cron delivery safeguards are useful partial controls, not the requested system-level cron restriction.

## What This Changes

The issue requests a cron-scoped tool-deny configuration so isolated scheduled jobs cannot use tools such as `message` unless explicitly allowed.

## Best Possible Solution

Keep this as the cron-scoped policy tracker, then implement either explicit `tools.cron.tools.allow/deny` or documented cron inheritance from subagent policy after maintainers choose precedence and upgrade semantics.

## Reproduction Assessment

Not applicable as a bug reproduction, but source inspection gives high confidence for the missing capability: current main has no `tools.cron.tools` config and cron sessions do not enter the subagent policy path.

## Solution Assessment

Unclear until maintainers choose the product/security shape; `tools.cron.tools.deny` is plausible, but the best fix must define precedence against existing policy layers and per-job `toolsAllow`.
