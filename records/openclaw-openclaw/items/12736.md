## Summary

Keep open. Current main and v2026.5.12 still lack a cron-scoped allow/deny policy or inheritance from subagent tool policy; the request remains actionable, but it needs maintainer design for config shape and authorization precedence.

## What This Changes

Request a cron-scoped tool allow/deny config option, or subagent-policy inheritance, so isolated cron jobs can be restricted from tools such as `message`.

## Best Possible Solution

Define either `tools.cron.tools.allow/deny` or explicit cron inheritance from `tools.subagents.tools`, document precedence, update schema/types/docs/generated metadata, and add regression coverage for isolated cron tool denial across delivery modes.

## Reproduction Assessment

Yes, as a source-verifiable behavior rather than a strict bug repro. Current main has no cron policy subtree in strict config/types, and the runtime only applies the subagent policy to subagent envelope sessions while cron runs use per-job `toolsAllow`.

## Solution Assessment

Unclear. A new `tools.cron.tools` policy and explicit inheritance from `tools.subagents.tools` are both plausible, but maintainers need to choose the public config contract and fail-closed precedence before implementation.
