## Summary

Keep open: current main and the latest release still lack a system-level cron tool policy such as `tools.cron.tools.allow/deny`; per-job `toolsAllow` and announce/message safeguards are useful partial controls, not the requested enforceable cron-scoped deny policy.

## What This Changes

Requested cron-scoped tool deny configuration so scheduled jobs can be prevented from using tools such as `message`.

## Best Possible Solution

Keep this as the cron-scoped policy tracker until maintainers choose either explicit `tools.cron.tools.allow/deny` or documented cron inheritance from subagent policy with clear precedence and upgrade behavior.

## Reproduction Assessment

Yes for the missing capability from source inspection: current main has no `tools.cron.tools` config surface and cron sessions do not enter the subagent policy path. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product/security shape; `tools.cron.tools.deny` is plausible, but the best implementation must define precedence against existing policy layers and per-job `toolsAllow`.
