## Label Justifications

- `P2`: This is a normal-priority security and delivery-control improvement with bounded cron-specific impact and a partial mitigation already on main.
- `impact:security`: Cron sessions still lack the requested enforceable scoped deny policy for least-privilege tool access.
- `impact:message-loss`: The motivating released behavior can duplicate scheduled messages when direct tool output and announce delivery both fire.

## Summary

Current main mitigates the reported duplicate announce symptom by suppressing fallback delivery after a matching direct message-tool send, but it still deliberately exposes `message` for announce jobs and has no general cron-scoped tool policy. The remaining least-privilege request is valid, aligned with the project’s security direction, and needs an explicit maintainer product decision rather than closure.

## What This Changes

Add enforceable cron-scoped tool restrictions so isolated scheduled jobs can be prevented from using selected tools such as `message`.

## Best Possible Solution

Keep the main-only same-target deduplication safeguard, then define one canonical cron least-privilege rule with explicit precedence and operator override—preferably a delivery-aware secure default plus per-job policy unless a general `tools.cron.tools` layer is clearly needed.

## Reproduction Assessment

Yes at source level: current main clearly has no cron policy layer, deliberately enables `message` for announce runs, and contains focused tests for suppressing duplicate fallback delivery. The discussion also provides a sanitized released-runtime reproduction, though no live run was executed during this read-only review.

## Solution Assessment

No, adding `tools.cron.tools.deny` verbatim is not yet proven to be the best solution. Maintainers should first decide whether a delivery-aware secure default composed with existing per-job `toolsAllow` is narrower and avoids another global config surface.
