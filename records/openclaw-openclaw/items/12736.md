## Label Justifications

- `P2`: This is a meaningful cron/tool-policy improvement with security and message-delivery impact, but it is a feature/config design request rather than an urgent regression.
- `impact:security`: The request changes the enforcement boundary for which model-visible tools isolated cron jobs can call.
- `impact:message-loss`: The discussion includes duplicated visible cron delivery when direct `message` sends and automatic announce delivery both fired for one run.

## Summary

Keep open: current main and v2026.6.8 still provide per-job cron `toolsAllow` and message-delivery safeguards, but not a cron-scoped config policy or documented inheritance from `tools.subagents.tools`; the remaining work is a security-sensitive product/config decision.

## What This Changes

Request adding a cron-scoped tool policy config option, such as `tools.cron.tools.deny`, or making cron sessions inherit `tools.subagents.tools` so isolated cron jobs can be prevented from using tools like `message`.

## Best Possible Solution

Keep the issue open until maintainers choose a cron policy model, then implement an additive `tools.cron.tools.allow/deny` or explicit subagent-inheritance path with schema/types/help/docs and regression coverage for announce-mode message denial and per-job `toolsAllow` precedence.

## Reproduction Assessment

Not applicable as a strict bug repro because this is a feature/config request, but the source path is high-confidence: current main lacks `tools.cron.tools.*` and cron sessions do not inherit the subagent policy path. Existing sanitized runtime evidence supports the duplicate-message motivation.

## Solution Assessment

Unclear: `tools.cron.tools.allow/deny` is a plausible explicit fix, but maintainers need to decide whether cron gets its own policy layer or inherits subagent policy and how it composes with per-job `toolsAllow` and delivery modes.
