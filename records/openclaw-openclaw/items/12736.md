## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny policy or explicit inheritance from subagent tool policy, while the existing cron surface is only per-job `toolsAllow`, so the requested security/config contract remains unresolved.

## What This Changes

Add a system-level cron tool restriction option such as `tools.cron.tools.deny`, or make cron sessions inherit `tools.subagents.tools`, so scheduled isolated jobs cannot use denied tools.

## Best Possible Solution

Define a public cron tool-policy contract, wire it into the fail-closed policy pipeline, and update schema/types/help/docs/generated metadata plus focused cron regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main has no cron tool-policy subtree in schema/types/docs, and the runtime subagent-policy branch applies only to subagent envelope sessions while cron relies on per-job `payload.toolsAllow`.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented precedence contract first.
