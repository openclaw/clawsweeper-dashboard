## Summary

Keep open. Current main and latest shipped tag still lack a cron-scoped tool allow/deny policy or explicit inheritance from subagent tool policy, while issue discussion includes sanitized duplicate announce delivery evidence when `message` remained available to cron.

## What This Changes

The issue requests a cron-scoped tool allow/deny policy so isolated scheduled jobs can be prevented from calling tools such as `message` directly.

## Best Possible Solution

Define a first-class cron tool-policy contract with documented fail-closed precedence, then wire schema/types/help/docs/generated config metadata and focused cron delivery regression coverage.

## Reproduction Assessment

Yes, by source inspection and discussion evidence. Current main has no cron-scoped policy subtree in schema/types/docs, cron sessions do not enter the subagent-policy path, and sanitized v2026.5.18 evidence reports duplicate announce delivery when the message tool remained available.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
