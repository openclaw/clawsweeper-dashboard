## Summary

Keep open. Current main and the latest release still expose per-job cron `toolsAllow` plus subagent/sandbox scoped policy, but there is no cron-scoped allow/deny config or inheritance rule, so this remains a valid feature/policy request requiring maintainer precedence design.

## What This Changes

Add a cron-scoped tool restriction policy, or explicit inheritance from subagent tool policy, so isolated cron jobs can be denied tools such as `message`.

## Best Possible Solution

Define a cron-scoped policy contract or explicit inheritance rule, update schema/types/help/docs/generated metadata, wire it into the shared tool-policy paths, and add regression coverage for isolated cron message-tool denial across delivery modes.

## Reproduction Assessment

Yes. Source inspection on current main shows no cron tool-policy subtree in strict config/types, while runtime subagent policy only applies to subagent envelope sessions and cron jobs rely on per-job `payload.toolsAllow`.

## Solution Assessment

Unclear. `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are both plausible, but maintainers need to choose the config shape and precedence rules before implementation.
