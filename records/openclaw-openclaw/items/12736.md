## Summary

Keep open. Current main still has no cron-scoped tool policy and cron sessions do not inherit `tools.subagents.tools`, so the requested capability remains actionable but needs a public config precedence decision.

## What This Changes

Request adding a cron-scoped tool restriction config, such as `tools.cron.tools.deny`, or making isolated cron jobs inherit subagent tool restrictions.

## Best Possible Solution

Define a documented cron tool-policy layer, wire it through schema/types/help/generated metadata/docs and both tool-resolution paths, and keep per-job `toolsAllow` as an additional narrowing layer.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `tools.subagents.tools.deny` cannot affect agent-scoped `cron:` sessions because cron and subagent classifiers are separate and the runtime only applies subagent policy to subagent envelopes.

## Solution Assessment

Unclear. A cron policy surface is the right problem area, but maintainers still need to choose explicit `tools.cron.tools.allow/deny` versus subagent-policy inheritance and document precedence before implementation.
