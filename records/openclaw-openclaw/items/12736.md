## Summary

Keep open: current main and v2026.6.10 still lack a cron-scoped `tools.cron.tools.allow/deny` policy or documented inheritance from `tools.subagents.tools`; per-job `toolsAllow` and cron message-delivery safeguards are useful partial controls, not the requested system-wide cron policy.

## What This Changes

Request to add a cron-scoped tool deny policy, or make cron sessions inherit subagent tool restrictions, so isolated scheduled jobs cannot use disallowed tools such as `message`.

## Best Possible Solution

Keep this as the canonical cron-scoped tool-policy tracker, then implement explicit `tools.cron.tools.allow/deny` or documented cron inheritance after maintainers decide precedence and upgrade behavior.

## Reproduction Assessment

Yes for source-level verification of the missing capability: current main has no `tools.cron.tools` config and cron sessions do not enter the subagent policy path. A live cron run was not executed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product/security shape. `tools.cron.tools.deny` is plausible, but the best fix must define policy precedence and upgrade behavior against existing per-job `toolsAllow` and delivery safeguards.
