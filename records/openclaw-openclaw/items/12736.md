## Summary

Keep open: current main and v2026.6.5 mitigate verified duplicate announce delivery, but neither provides a cron-scoped deny policy or makes cron inherit subagent tool restrictions; this remains a distinct security-sensitive product/config decision.

## What This Changes

Add enforceable cron-specific tool restrictions so operators can deny selected tools for isolated scheduled agent runs.

## Best Possible Solution

Add a cron scope to the canonical tool-policy composition with documented deny precedence over per-job allow requests, while keeping source-delivery verification responsible only for duplicate-delivery handling.

## Reproduction Assessment

Yes at source level: current main explicitly exposes only per-job `toolsAllow` and delivery-specific message controls, with no cron-wide deny policy or subagent-policy inheritance. The supplied v2026.5.18 evidence also describes the resulting duplicate-message workflow.

## Solution Assessment

The requested capability remains necessary, but a standalone special case is not yet the best implementation; the maintainable solution is a cron scope in canonical policy composition with explicit precedence and delivery-mode coverage.
