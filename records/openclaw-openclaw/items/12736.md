## Summary

Current main still lacks a cron-scoped tool policy, and the discussion includes a concrete shipped-version report of duplicate announce delivery when a cron agent directly uses `message`; the item remains necessary but requires maintainer product and security decisions before implementation.

## What This Changes

Add enforceable cron-specific tool restrictions, especially denial of direct `message` use for isolated jobs that already use automatic delivery.

## Best Possible Solution

Define one canonical cron policy rule at the tool-resolution boundary, specify precedence with existing policy layers and per-job capabilities, preserve intentional direct-send workflows, and prevent duplicate announce delivery through a safe default or explicit mode with schema, docs, and regression coverage.

## Reproduction Assessment

Yes for the policy gap: current source gives a high-confidence path showing no cron-scoped deny layer and cron sessions bypassing subagent policy. The duplicate-message symptom has concrete v2026.5.18 evidence but was not live-reproduced on current main.

## Solution Assessment

Unclear. A cron config wrapper is plausible, but a delivery-aware canonical policy may solve the duplicate-send problem with less configuration; maintainers must decide precedence and compatibility.
