## Summary

Keep open: current main and the latest checked release still lack an operator-level cron tool policy or subagent-policy inheritance for cron sessions; existing `toolsAllow` and delivery safeguards only cover narrower per-run behavior.

## What This Changes

Add an operator-level cron tool restriction policy such as `tools.cron.tools.deny`, or make cron sessions inherit an existing restricted tool policy.

## Best Possible Solution

Design a single cron tool-policy layer, then update schema, types, help, docs, and regression tests for isolated cron delivery modes.

## Reproduction Assessment

Yes at source level: current schema/types expose no cron tool policy, and runtime applies `tools.subagents.tools` only to subagent envelopes; no live cron run was executed because this review is read-only.

## Solution Assessment

Unclear pending maintainer direction: `tools.cron.tools.deny` is plausible, but maintainers need to choose the policy layer, inheritance behavior, delivery defaults, and precedence rules.
