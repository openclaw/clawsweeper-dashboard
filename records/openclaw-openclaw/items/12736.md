## Summary

Keep open. Current main still lacks a cron-scoped tool policy and does not make cron sessions inherit `tools.subagents.tools`; the remaining implementation is per-job `payload.toolsAllow` plus delivery-mode message-tool handling, so the feature request remains actionable.

## What This Changes

The issue requests a cron-scoped tool restriction config such as `tools.cron.tools.deny`, or equivalent inheritance from subagent tool restrictions, for isolated cron sessions.

## Best Possible Solution

Define an explicit cron policy surface, preferably `tools.cron.tools.allow/deny`, then wire it through schema/types/help/generated metadata/docs and all tool-policy resolution paths while preserving per-job `toolsAllow` as an additional narrowing layer.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configuring `tools.subagents.tools.deny` cannot affect agent-scoped `cron:` sessions because cron and subagent classifiers are separate and the runtime only applies subagent policy to subagent envelopes.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation needs maintainer choice between explicit `tools.cron.tools` and inheritance from subagent policy, plus a documented precedence rule.
