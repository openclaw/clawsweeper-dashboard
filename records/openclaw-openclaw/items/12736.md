## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny config policy or explicit cron inheritance from subagent policy; per-job `toolsAllow` and newer cron message-delivery safeguards only partially address the reported operator-control problem.

## What This Changes

The issue requests a cron-scoped tool policy such as `tools.cron.tools.deny`, or explicit inheritance from `tools.subagents.tools`, so isolated scheduled jobs can be denied tools like `message`.

## Best Possible Solution

Define a first-class cron tool-policy contract, or explicit cron inheritance from subagent policy, with documented fail-closed precedence plus schema, exported types, docs, metadata/help, and focused cron delivery/tool-policy regression coverage.

## Reproduction Assessment

Yes for the reported missing config surface. Source inspection on current main shows no cron-scoped policy subtree in schema/types/docs, and cron sessions do not enter the subagent-policy path; no live run was attempted because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
