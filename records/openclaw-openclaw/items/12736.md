Keep open: current `main` and the latest release still lack a cron-scoped `tools.cron.tools.allow/deny` contract or explicit cron inheritance from subagent tool policy, so the central request remains actionable and security-sensitive.

## What This Changes

The issue requests a cron-scoped tool deny policy so scheduled isolated sessions can be prevented from using tools such as `message`.

## Best Possible Solution

Define a first-class cron tool-policy contract, or explicit cron inheritance from subagent policy, with documented fail-closed precedence plus schema, exported types, docs/help metadata, and focused cron delivery/tool-policy regression coverage.

## Reproduction Assessment

Yes for the reported missing config surface: source inspection on current `main` shows no cron-scoped policy subtree in schema/types/docs, and cron sessions do not enter the subagent-policy path. No live run was attempted because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
