## Summary

Keep open: current main and v2026.6.10 still lack a cron-scoped tool policy or cron inheritance from `tools.subagents.tools`, while the discussion includes sanitized evidence that unrestricted cron `message` access can duplicate announce delivery output.

## What This Changes

The issue requests a cron-scoped tool deny policy, such as `tools.cron.tools.deny`, so scheduled isolated jobs can restrict tools independently of subagent sessions.

## Best Possible Solution

Design one canonical cron tool-policy layer in core config/runtime, then update schema/types/help/docs and add regression coverage for `message` denial under cron announce delivery.

## Reproduction Assessment

Yes, at source level: current main has no `tools.cron.tools.allow/deny` schema/type/runtime path, and cron sessions do not enter the subagent-policy branch. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy shape. A cron-specific allow/deny subtree or explicit inheritance from `tools.subagents.tools` could both solve the request, but either needs precedence, upgrade-safety, and security-boundary decisions.
