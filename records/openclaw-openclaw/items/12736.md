## Summary

Keep open. Current main still has only global/provider/agent/group/sandbox/subagent tool policy plus per-job cron `toolsAllow`; it does not provide a cron-scoped `tools.cron.tools.allow/deny` layer or make cron sessions inherit subagent policy.

## What This Changes

The issue requests a cron-scoped tool restriction config such as `tools.cron.tools.deny`, or explicit inheritance from `tools.subagents.tools`, for isolated cron jobs.

## Best Possible Solution

Define and document a cron tool-policy layer, then wire it through schema, exported types, generated config metadata, docs, runtime policy resolution, and regression coverage for isolated cron message-tool denial.

## Reproduction Assessment

Yes for the reported limitation. Source inspection shows cron session keys are distinct from subagent keys, no `tools.cron.tools` schema/type exists, and the runtime only applies subagent policy to subagent envelopes.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers need to choose explicit cron-scoped policy versus subagent-policy inheritance and define policy precedence before code changes.
