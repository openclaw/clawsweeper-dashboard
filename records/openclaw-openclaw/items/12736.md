## Summary

Keep open: current main and v2026.6.10 still lack a cron-scoped `tools.cron.tools.allow/deny` policy or documented inheritance from `tools.subagents.tools`; current per-job `toolsAllow` and message-delivery safeguards are useful partial controls but do not solve the requested system-level cron policy.

## What This Changes

Request a cron-scoped tool deny policy so isolated cron sessions can restrict tools independently of subagent-only restrictions.

## Best Possible Solution

Keep this as the cron-scoped policy tracker, then implement either explicit `tools.cron.tools.allow/deny` or documented cron inheritance from subagent policy after maintainers choose precedence, upgrade behavior, schema/docs/tests, and any required doctor migration.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but yes for source-level verification: current main has no `tools.cron.tools` config and cron sessions do not enter the subagent policy path.

## Solution Assessment

Unclear until maintainers choose the product/security shape; `tools.cron.tools.deny` is plausible, but the best fix must define precedence against existing policy layers and per-job `toolsAllow`.
