## Summary

Keep open. Current main still lacks a cron-scoped tool allow/deny config, and the runtime only applies the subagent policy to subagent/ACP envelope sessions; this is a valid enhancement that needs config/API precedence design.

## What This Changes

Request a cron-scoped tool policy such as `tools.cron.tools.deny`, or explicit inheritance from `tools.subagents.tools`, so isolated cron runs can be prevented from using tools like `message`.

## Best Possible Solution

Design a cron-scoped policy or explicit inheritance rule, update schema/types/help/docs/generated config metadata, wire it into the PI/gateway effective policy paths, and add regression coverage for isolated cron message-tool denial across delivery modes.

## Reproduction Assessment

Yes. Source inspection on current main shows strict config/types have no cron tool-policy subtree and the runtime only resolves `tools.subagents.tools` for subagent/ACP envelope sessions, while cron jobs use per-job `payload.toolsAllow`.

## Solution Assessment

Unclear until maintainers choose the policy shape. `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are both plausible, but the precedence rules need product/API design first.
