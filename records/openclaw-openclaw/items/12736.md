## Summary

Keep open: current main and the latest release still lack a cron-scoped tool policy or documented inheritance from `tools.subagents.tools`, while the issue discussion includes runtime evidence that announce-mode cron jobs can duplicate visible messages when the message tool remains available.

## What This Changes

The issue requests a cron-scoped tool restriction config such as `tools.cron.tools.deny` or explicit inheritance from subagent tool policy for cron jobs.

## Best Possible Solution

Keep this open until maintainers choose a cron tool-policy design, then implement an additive `tools.cron.tools.allow/deny` or explicit inheritance rule with schema/types/docs and regression coverage for announce-mode message denial plus per-job `toolsAllow` precedence.

## Reproduction Assessment

Yes for the source-level gap: current main has no `tools.cron.tools.*` schema/type/docs surface, and cron sessions do not inherit the subagent policy path. No live run was executed in this read-only review, but the discussion includes sanitized runtime evidence for the duplicate-message motivation.

## Solution Assessment

Unclear: a cron-scoped deny option is plausible and source-backed, but maintainers need to decide whether cron should get its own policy layer or inherit subagent policy and how that composes with per-job `toolsAllow` and delivery modes.
