## Summary

Keep open: current main and v2026.6.8 still lack a cron-scoped `tools.cron.tools.*` policy or documented inheritance from `tools.subagents.tools`; recent cron tool-surface work is useful mitigation but does not implement the requested operator policy.

## What This Changes

The issue requests a cron-scoped tool restriction policy, such as `tools.cron.tools.deny`, so scheduled cron sessions can block tools like `message` without relying on prompt instructions.

## Best Possible Solution

Keep this open for a maintainer-approved cron tool-policy design that either adds `tools.cron.tools.allow/deny` or explicitly defines cron inheritance from `tools.subagents.tools`, with precedence, schema/types/help/docs, and regression coverage.

## Reproduction Assessment

Yes for the source-level gap: current main and v2026.6.8 have no `tools.cron.tools.*` config and no subagent-policy inheritance for cron sessions. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear: `tools.cron.tools.allow/deny` is a plausible explicit fix, but maintainers need to decide whether cron gets its own policy layer or inherits subagent policy and how it composes with per-job `toolsAllow` and delivery modes.
