- `impact:security`: The issue is about enforcing operator-defined tool access boundaries for isolated cron sessions.
- `impact:message-loss`: The discussion includes duplicate visible cron messages and message routing mistakes when cron jobs can call `message` directly.

## Summary

Keep open: current main and the latest release still lack a cron-scoped `tools.cron.tools.allow/deny` policy or documented inheritance from `tools.subagents.tools`; current `payload.toolsAllow` and cron delivery safeguards are useful partial mitigations, not the requested system-level cron policy.

## What This Changes

Request adds a cron-scoped tool allow/deny policy, or explicit subagent-policy inheritance, so isolated cron jobs can block tools such as `message`.

## Best Possible Solution

Keep this issue open for a maintainer-designed cron tool-policy surface that defines precedence against existing policy layers and per-job `toolsAllow`, then updates schema, types, docs, metadata, and regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current schemas have no `tools.cron.tools` field, and the runtime only applies `tools.subagents.tools` to subagent envelope sessions. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainer product/security direction is chosen: `tools.cron.tools.allow/deny` is plausible, but explicit inheritance from subagent policy or a broader per-session policy could be the better long-term shape.
