Keep open. Current main still lacks a cron-scoped tool allow/deny policy or explicit subagent-policy inheritance for cron runs, and the remaining work is a public config/security contract rather than conservative cleanup.

## What This Changes

The issue requests a cron-scoped tool deny policy so scheduled isolated jobs can be restricted from tools such as `message`.

## Best Possible Solution

Define a public cron tool-policy contract with fail-closed precedence, then wire schema/types/help/docs/generated config metadata and focused cron regression coverage.

## Solution Assessment

Unclear until maintainers choose the public policy shape. Both `tools.cron.tools.allow/deny` and explicit inheritance from `tools.subagents.tools` are plausible, but the maintainable fix needs a documented fail-closed precedence contract first.
