## Summary

Keep open. Current main still lacks a preventative CLI-wide root-execution guard, the root HOME/state and doctor-to-systemd user-service paths remain reachable, and open PR #67509 is already the direct closing implementation candidate, so this issue should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to prevent or strongly warn against direct root invocation on DigitalOcean 1-Click deployments because it can create root-owned state, conflicting user systemd units, and backup/config failures.

## Best Possible Solution

Review and land PR #67509, or replace it with an equivalent early CLI guard before config/state resolution, with an explicit override, root help/version behavior, targeted tests, and DigitalOcean recovery or wrapper guidance if maintainers want image-level mitigation too.

## Reproduction Assessment

Yes. The issue provides deterministic droplet steps, and current-main code inspection confirms the required paths still exist: root HOME resolves root state, doctor repair can install a service using process.env, and systemd writes a user unit under the effective home. I did not run the destructive live root/DigitalOcean scenario during this read-only review.

## Solution Assessment

Yes at the issue level. An early CLI root guard with an explicit override is the narrowest repo-side prevention; docs or a marketplace wrapper alone would not reliably stop direct invocation of the installed openclaw binary.
