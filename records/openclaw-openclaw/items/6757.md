## Summary

Keep open: current main and the latest release still have manual, automatic, SDK/admin, and context-engine compaction paths, but they do not expose the requested in-session agent-triggered own-session compaction action. The prior implementation attempts are closed unmerged, so this remains a product and security-boundary decision rather than conservative cleanup to close.

## What This Changes

The issue requests a first-class way for an agent to compact its own active session, such as `session_status(action="compact")`, an equivalent Gateway tool action, or a dedicated CLI/API surface.

## Best Possible Solution

Implement a first-class own-session compaction action, likely `session_status(action="compact")` or an equivalent mutating agent tool, using direct in-run compaction when appropriate and preserving session visibility, mutation tracking, compaction counters, token snapshots, and the admin RPC boundary.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.20` lack a `session_status` compact action, model-facing Gateway compaction action, or dedicated compact CLI command, while existing compaction remains manual, automatic, plugin/context-engine, SDK, or admin/control-plane scoped.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. A constrained own-session `session_status(action="compact")` or equivalent mutating tool looks narrow, but forwarding the existing admin RPC unchanged would not be the right solution.
