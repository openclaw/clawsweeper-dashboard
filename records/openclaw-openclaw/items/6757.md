## Summary

Keep open: current main still lacks an agent-callable self-compaction path. The existing compaction surfaces are manual sender-authenticated chat commands, automatic/runtime compaction, SDK/control-plane RPCs, or admin-scoped Gateway methods, not a first-class in-session tool an agent can call on its own active run.

## What This Changes

The issue requests a new agent-facing way for a session to trigger its own context compaction, such as `session_status(action="compact")`, a Gateway action, or a dedicated CLI command.

## Best Possible Solution

Add a first-class mutating in-session compaction action with explicit permission semantics, direct active-run compaction, counter/token updates, docs, changelog, and focused regression tests.

## Reproduction Assessment

Not applicable as a feature request rather than a broken documented behavior. Source inspection on current main gives high confidence that the requested surface is absent: `session_status` has no compact action and the agent-facing `gateway` tool cannot call `sessions.compact`.

## Solution Assessment

Unclear until a maintainer chooses the public surface and safety model. `session_status(action="compact")` looks like the narrowest candidate and matches https://github.com/openclaw/openclaw/pull/43401 prior art, but it still needs product and permission sign-off.

## Risks / Open Questions

- The public surface is still a product/API choice: `session_status(action="compact")`, an agent-facing Gateway action, and a CLI command have different permission and active-run semantics.
- Forwarding the existing `sessions.compact` RPC as-is would likely be wrong for this use case because it is admin-scoped and interrupts active runs before compaction.
- A new mutating agent-facing compaction action needs explicit visibility, authorization, and replay/mutation tracking semantics before implementation.
