## Summary

Keep this issue open: current main and v2026.6.5 still provide manual, automatic, plugin-hook, and control-plane compaction, but no agent-callable same-run self-compaction action. The closed implementation PR remains useful prior art, but maintainers must decide the authorization and post-compaction continuation contract before implementation.

## What This Changes

Add an agent-callable action that compacts the agent’s own active session without requiring an authorized external sender.

## Best Possible Solution

Add one mutating, current-session-only agent action—preferably on `session_status`—that delegates to the canonical embedded compaction runtime, atomically adopts successor session state, and returns a defined continuation result without exposing arbitrary control-plane targeting.

## Reproduction Assessment

Yes, at the source level. Current main and v2026.6.5 expose no agent-callable compact action through `session_status` or the agent-facing gateway tool; this is a missing capability rather than a failing existing contract.

## Solution Assessment

Yes, conditionally: a current-session `session_status` action is narrower and more maintainable than exposing arbitrary admin RPC access or adding another CLI surface. It should proceed only after maintainers define authorization, active-run continuation, and successor-state adoption semantics.

## Risks / Open Questions

- A model-triggered compaction action is a privileged session-state mutation; without a strict current-session boundary it could allow an agent to mutate another visible session.
- Compaction can replace the active transcript/session identity, so the continuation behavior of the tool call and current run must be explicitly defined to avoid stale or mis-associated state.
- The implementation must preserve context-engine hooks, compaction checkpoints, counters, token freshness, model/auth overrides, and transcript-DAG invariants now handled by the canonical runtime.
