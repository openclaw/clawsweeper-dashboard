## Summary

Keep open: current `main` and the latest release still have manual, automatic, plugin/context-engine, and admin Gateway compaction paths, but no first-class agent-facing action that lets an agent compact its own active session from inside the run.

## What This Changes

The issue requests an agent-callable way to compact its own active session, such as `session_status(action="compact")`, a Gateway RPC, or a dedicated CLI command.

## Best Possible Solution

Keep this as the canonical request until maintainers choose a first-class agent-facing self-compaction contract, preferably a narrow `session_status` action or equivalent in-run tool that preserves visibility, mutation tracking, active-run behavior, and session compaction counters.

## Reproduction Assessment

Yes, by source inspection: current `main` exposes `session_status` as status/model only, while Gateway compaction is admin-scoped and absent from the agent-facing `gateway` tool actions. A live repro is not necessary because this is a missing API surface rather than a runtime-only failure.

## Solution Assessment

Unclear until maintainers choose the authority contract. `session_status(action="compact")` remains a plausible narrow shape, but exposing the existing admin `sessions.compact` method directly to agents would be the wrong boundary without a product/security decision.
