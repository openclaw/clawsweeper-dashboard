## Summary

Keep open. Current main still lacks a source-backed fix or targeted regression for the reported Control UI CJK streaming parse failure; the browser client continues to parse each Gateway WebSocket message independently, while the server-side framing path normally sends complete JSON string frames, so the actual malformed boundary still needs to be proven before choosing a repair.

## What This Changes

Streaming Control UI webchat responses containing Chinese/CJK UTF-8 text should not break with JSON parse errors when content is split near multi-byte characters.

## Best Possible Solution

Add deterministic coverage for the Control UI GatewayBrowserClient/webchat event path with long CJK content and, if possible, a captured malformed-frame scenario; then fix the exact boundary where malformed JSON enters while preserving final history recovery and adding useful diagnostics instead of blindly buffering every WebSocket message.

## Reproduction Assessment

Unclear. The issue has credible manual steps from v2026.4.2, but I did not find a deterministic current-main test, captured raw Control UI WebSocket frame, or live reproduction proving that CJK UTF-8 boundary splitting reaches GatewayBrowserClient as malformed JSON.

## Solution Assessment

Unclear. A raw WebSocket buffering fix is not clearly the best solution because the documented Gateway protocol sends JSON text frames and browsers normally deliver complete text messages; the safer path is to prove the malformed boundary first, then fix that specific parser or transport seam.
