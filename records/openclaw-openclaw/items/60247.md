## Summary

Keep open. Current main still does not prove the reported Control UI CJK streaming parse failure is fixed: the browser client parses each Gateway WebSocket message as standalone JSON and drops malformed frames, while the server broadcast path sends complete JSON text frames; the newer related items cover separate OpenAI SSE, Anthropic failover, and static skill-language problems rather than a canonical closure for this UI path.

## What This Changes

The issue requests that Control UI streaming handle Chinese/CJK UTF-8 content without JSON.parse failures when stream boundaries split multi-byte data.

## Best Possible Solution

Capture a raw failing Control UI Gateway message or add a deterministic malformed-frame test with long CJK content, then fix the exact browser client, server framing, provider stream, or renderer boundary and leave provider failover work to https://github.com/openclaw/openclaw/issues/60593.

## Reproduction Assessment

No. The report has credible manual steps from v2026.4.2, but I did not establish a deterministic current-main reproduction or captured raw failing WebSocket payload; source inspection only shows how the browser client would drop a malformed frame if one arrives.

## Solution Assessment

Unclear. Browser-side reassembly is not yet proven to be the best fix because the documented Gateway contract is complete JSON text frames and the server broadcast path sends one JSON string; the safer path is to prove the malformed boundary first.
