## Summary

Keep open: current main and v2026.5.20 still omit a `memorySearch.local.maxThreads` config carrier and `getLlama` passthrough, while upstream `node-llama-cpp` exposes `maxThreads`. The related https://github.com/openclaw/openclaw/pull/60683 overlaps the local llama tuning surface but is still open/conflicting and does not implement this thread-limit request.

## What This Changes

Request to add a `memorySearch.local.maxThreads` setting for the local `node-llama-cpp` embedding provider so users can tune CPU thread usage.

## Best Possible Solution

Add a validated `memorySearch.local.maxThreads` setting through public types, strict schema, help/labels, docs, resolved config, provider create options, and the node-llama wrapper, forwarding it only when configured while preserving current defaults.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than broken documented behavior. The missing capability is high-confidence from source inspection of schema, resolved config, provider option types, runtime call, docs, release tag, and upstream node-llama-cpp types.

## Solution Assessment

Yes, if maintainers accept the public knob. The narrow maintainable path is to follow the existing `contextSize` carrier pattern and explicitly decide OpenClaw's validation for upstream's `0` no-limit behavior.
