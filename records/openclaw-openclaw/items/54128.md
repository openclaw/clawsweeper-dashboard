## Summary

Keep open. Current main still has no `memorySearch.local.maxThreads` config carrier or runtime passthrough for local node-llama-cpp embeddings, while the upstream `getLlama` options surface supports `maxThreads`; the overlapping PR at https://github.com/openclaw/openclaw/pull/60683 is still open and does not implement this requested thread limit.

## What This Changes

Request adding a `memorySearch.local.maxThreads` setting for local node-llama-cpp embeddings and forwarding it to `getLlama`.

## Best Possible Solution

Add a narrow, validated `memorySearch.local.maxThreads` setting, preserve existing defaults when unset, forward it only to `getLlama`, and update config schema/help/docs plus focused tests.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken documented behavior. The gap is high-confidence from source inspection of the schema, resolved config, provider option types, node-llama shim, docs, and runtime call.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable path if maintainers accept the public config knob. It should follow the existing `contextSize` config/resolution/runtime/docs/test pattern and avoid mixing in unsupported or unrelated llama tuning changes.
