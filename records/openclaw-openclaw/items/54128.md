## Summary

Keep open. Current main still has no `memorySearch.local.maxThreads` config carrier or runtime passthrough for local node-llama-cpp embeddings, and the related local-llama settings PR at https://github.com/openclaw/openclaw/pull/60683 is open but does not cover this specific thread-limit request.

## What This Changes

Request adding a `memorySearch.local.maxThreads` config option for the local node-llama-cpp embedding provider so users can tune CPU thread usage.

## Best Possible Solution

Add a narrow, validated `memorySearch.local.maxThreads` setting, preserve existing defaults when unset, forward it only to `getLlama`, and update schema/help/docs plus focused tests after maintainers choose validation semantics.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a broken documented behavior. Source inspection gives high-confidence proof that the requested config key is currently absent from schema, resolution, docs, and runtime passthrough.

## Solution Assessment

Yes, if maintainers accept the public knob: the narrow maintainable path is to follow the existing local `contextSize` pattern and pass `maxThreads` to `getLlama`. The only unresolved part is the public validation/default semantics, especially upstream's `0` no-limit value.
