## Summary

Keep open: current `main` and latest `v2026.5.19` still omit a `memorySearch.local.maxThreads` config carrier and `getLlama` passthrough, while node-llama-cpp exposes `maxThreads` at the Llama options boundary. The related https://github.com/openclaw/openclaw/pull/60683 overlaps the local llama tuning surface, but it is still open/conflicting and does not close or implement this specific thread-limit request.

## What This Changes

Add a `memorySearch.local.maxThreads` option for local node-llama-cpp embeddings so users can tune CPU thread usage during memory indexing.

## Best Possible Solution

Add a validated `memorySearch.local.maxThreads` setting through public types, strict schema, help/labels, docs, resolved config, provider create options, and the node-llama wrapper, forwarding it to `getLlama` only when configured while preserving current defaults.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than broken documented behavior. The missing capability is high-confidence from source inspection of the strict schema, resolved config, provider option types, runtime call, docs, shipped release state, and upstream node-llama-cpp contract.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if maintainers accept the public knob. The implementation should follow the existing `contextSize` pattern while deciding how OpenClaw should validate upstream's `0` no-limit thread semantics.
