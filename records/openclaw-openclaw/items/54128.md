## Summary

Keep open. Current main still has no `memorySearch.local.maxThreads` config/schema/type carrier or node-llama-cpp passthrough, while the pinned upstream node-llama-cpp runtime supports `maxThreads` on `getLlama`; related PR #60683 remains open and overlaps local llama settings but does not implement this requested knob.

## What This Changes

The issue requests a configurable `memorySearch.local.maxThreads` setting for local node-llama-cpp embeddings so users can tune or limit CPU thread usage during memory indexing.

## Best Possible Solution

Add a validated `memorySearch.local.maxThreads` field for the built-in local embedding provider, preserve current defaults when unset, carry the value through resolved/default and per-agent config plus provider option types, pass it to `getLlama` only when configured, and update tests, generated config metadata, public docs, and changelog.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: `memorySearch.local.maxThreads` is rejected by the strict schema today, and the local embedding runtime cannot forward it because `getLlama` is called with only `logLevel`.

## Solution Assessment

Yes. A narrow schema/type/resolution/runtime/docs/test passthrough is the best maintainable fix; PR #60683 is adjacent but not sufficient because it omits the requested thread-limit option.
