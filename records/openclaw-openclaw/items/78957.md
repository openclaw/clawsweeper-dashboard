## Summary

Keep open: current main still pins the Android gateway TLS fingerprint by stable endpoint, and the setup-code/onboarding reset paths clear auth/device tokens but not stored fingerprints, so a renewed certificate on the same host can still fail closed with a mismatch.

## What This Changes

The issue reports that Android keeps a cached gateway TLS fingerprint after certificate renewal and asks for QR re-scan or reset recovery.

## Best Possible Solution

Add Android recovery that clears stored gateway TLS fingerprints during an explicit reset/onboarding restart, then reprobes and prompts before saving the renewed certificate fingerprint.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: store a `gateway.tls.manual|host|port` pin, reconnect to the same host after the certificate changes, and the reset/setup-code path leaves the old pin in place so TLS throws `gateway TLS fingerprint mismatch`. I did not run a live Android device.

## Solution Assessment

Is this the best way to solve the issue? Yes: clearing pins only through an explicit reset/onboarding recovery path is narrower and safer than silently accepting a QR or discovery-provided replacement, and it matches the existing iOS reset behavior.
