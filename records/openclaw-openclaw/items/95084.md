## Summary

Keep open: the fix direction is right for the generic outbound adapter, but the patch misses the default Google Chat inbound reply path. With the shipped/default typing-message flow, final replies can bypass durable outbound delivery and go through `deliverGoogleChatReply()` directly, so the reported internal tool-trace banner can still reach users.

## What This Changes

The branch composes Google Chat outbound adapter `sanitizeText` with `sanitizeAssistantVisibleText()` before `sanitizeForPlainText()` and adds focused sanitizer tests.

## Best Possible Solution

Apply the canonical assistant-visible sanitizer before both durable outbound and direct Google Chat reply delivery, then prove the default `typingIndicator: "message"` path no longer leaks trace banners.

## Reproduction Assessment

Yes, at source level. Current main shows Google Chat defaults to a typing-message reply path that disables durable outbound delivery and calls `deliverGoogleChatReply()` directly, where the PR's adapter sanitizer is not invoked.

## Solution Assessment

No. The adapter hook is a good part of the fix, but the best fix must also sanitize the direct Google Chat reply delivery path or prepare payloads before the durable/fallback split.

- **[P1] Sanitize the default reply path too:** `extensions/googlechat/src/channel.adapters.ts:201-202`
  - body: Putting the sanitizer only on the outbound adapter misses the default inbound reply path. Google Chat creates a typing message by default, and `resolveGoogleChatDurableReplyOptions()` returns `false` while that typing message is active, so final replies fall back to `deliverGoogleChatReply()`, which sends chunks from `reply.text` without invoking this adapter hook.
