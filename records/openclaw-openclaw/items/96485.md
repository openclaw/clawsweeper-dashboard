## Summary

Keep open: current main and the latest release still have the reported Anthropic OAuth callback host/redirect mismatch. The security framing should be narrowed because SECURITY.md treats attacker-controlled process environment as trusted host/operator control, but the provider-login footgun is real and source-reproducible.

## What This Changes

The issue reports that Anthropic OAuth binds its local callback server from OPENCLAW_OAUTH_CALLBACK_HOST while always sending a localhost redirect URI.

## Best Possible Solution

Align Anthropic OAuth with the existing OpenAI loopback-host pattern: allow only localhost, 127.0.0.1, or ::1, derive the redirect URI from that resolved host, and add focused regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the server binds with CALLBACK_HOST while the authorize request and token exchange use a fixed localhost REDIRECT_URI; I did not run a live Anthropic OAuth login because that requires an external provider flow.

## Solution Assessment

Yes, if narrowed to callback consistency and loopback hardening. The existing OpenAI OAuth helper is the cleaner in-repo pattern than adding another config surface or preserving arbitrary callback hosts.
