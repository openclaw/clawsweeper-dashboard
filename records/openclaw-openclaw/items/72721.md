## Summary

Keep open. Current main still routes the reported Feishu bot ping and startup/status probes through the Lark SDK HTTP path, WebSocket mode remains the documented/default setup path, and the current code bounds or logs probe timeouts rather than proving the Alibaba Cloud ECS / Node.js 24 axios transport hang is fixed; related PR #67466 is still only an open mitigation candidate.

## What This Changes

The issue reports Feishu WebSocket-mode gateway startup timing out on Lark SDK axios calls to open.feishu.cn from Alibaba Cloud ECS / Node.js 24 while native HTTPS succeeds, with webhook mode as a workaround.

## Best Possible Solution

Keep the fix inside the Feishu plugin by making the startup identity probe, setup/status probe, and SDK HTTP path deterministic and abortable, while treating webhook mode as an operational workaround rather than replacing WebSocket as the default.

## Reproduction Assessment

Unclear. The report has concrete steps, diagnostics, and corroborating logs from another user, but high-confidence reproduction still needs Feishu credentials plus the reported Alibaba Cloud ECS / Node.js 24 environment or a deterministic local axios-hang fixture.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable direction is plugin-local because the Feishu plugin owns the Lark SDK client wrapper, ping probe, setup/status probe, and monitor startup lifecycle; current main only bounds timeout handling.
