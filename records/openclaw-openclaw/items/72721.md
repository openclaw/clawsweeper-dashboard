## Summary

Keep open. Current main still routes the reported Feishu `openclaw_bot/ping` startup/status probe through the Lark SDK HTTP instance, and the recent main changes mitigate packaging/timeout handling without proving that the Alibaba Cloud ECS / Node.js 24 axios transport timeout is fixed for https://github.com/openclaw/openclaw/issues/72721.

## What This Changes

The issue reports Feishu websocket-mode gateway startup timeouts when the Lark SDK axios transport calls `open.feishu.cn` on Alibaba Cloud ECS with Node.js 24.

## Best Possible Solution

Keep the fix inside the Feishu plugin: land or replace the focused startup/status timeout work, then bypass or replace the SDK axios path for the affected Feishu probe/API calls only if dependency or live proof confirms that transport as the root cause.

## Reproduction Assessment

Unclear for a high-confidence live reproduction. The source path and field logs match current main, but this review did not reproduce the Alibaba Cloud ECS / Node.js 24 axios timeout without that environment and Feishu credentials.

## Solution Assessment

Not applicable for a patch on this issue. The narrow maintainable direction is plugin-local because the Feishu plugin owns the SDK client wrapper, ping probe, startup probe, setup status check, and monitor lifecycle.
