## Summary

Keep open. Current main still routes Feishu bot identity, setup/status, and channel-status probes through the Lark SDK HTTP instance that wraps the SDK default axios transport, so the reported gateway timeout path remains live rather than implemented away.

## What This Changes

The issue reports Feishu WebSocket-mode startup timing out in the Lark SDK axios path on Alibaba Cloud ECS / Node.js 24, while native HTTPS and webhook mode work.

## Best Possible Solution

Fix this inside the Feishu plugin by making the bot ping, status, and startup SDK HTTP path deterministic and abortable or replacing it with a native fetch/HTTPS adapter, while keeping WebSocket as the default and treating webhook mode as an operational workaround.

## Reproduction Assessment

Unclear. The report has concrete steps, diagnostics, and corroborating logs from another user, but high-confidence reproduction still needs Feishu credentials plus the reported Alibaba Cloud ECS / Node.js 24 environment or a deterministic axios-hang fixture.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable direction is plugin-local because the Feishu plugin owns the Lark SDK client wrapper, ping probe, setup/status probe, and monitor startup lifecycle.
