## Summary

Keep open. Current main still routes the Feishu `openclaw_bot/ping` probe through the Lark SDK-backed HTTP instance, WebSocket mode remains supported and default, and the second-user timeout stack matches the same endpoint. The current code mitigates startup failure by treating probe timeouts as non-fatal, but it does not prove the Alibaba Cloud ECS / Node.js 24 axios hang is fixed, and the related startup/status timeout PR is still open.

## What This Changes

The issue reports Feishu WebSocket-mode gateway startup timeouts when `@larksuiteoapi/node-sdk`/axios posts to `https://open.feishu.cn/open-apis/bot/v1/openclaw_bot/ping` on Alibaba Cloud ECS with Node.js 24.

## Best Possible Solution

Fix this inside the Feishu plugin with a deterministic transport and timeout policy for startup identity probing, setup/status probing, and SDK HTTP calls, while keeping WebSocket as a supported default and webhook mode as an operational workaround.

## Reproduction Assessment

Unclear. The report has concrete field steps, corroborating logs from a second user, and current source still exercises the same SDK-backed ping path, but high-confidence reproduction requires Feishu credentials and the reported Alibaba Cloud ECS / Node.js 24 environment or a deterministic axios-hang fixture.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable direction is a Feishu plugin-local fix because the plugin owns the Lark SDK dependency, ping probe, setup/status probe, and monitor lifecycle; current main only mitigates timeout handling rather than replacing or proving the problematic transport fixed.
