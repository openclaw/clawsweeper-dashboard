## Summary

Keep open. Current main still sends the reported Feishu `openclaw_bot/ping` probe through the Lark SDK HTTP path, WebSocket remains the documented default, and recent main-only Feishu changes add timeout/status handling rather than proving the Alibaba Cloud ECS / Node.js 24 axios transport hang is fixed.

## What This Changes

The issue reports a Feishu WebSocket-mode gateway startup timeout where the Lark SDK axios POST to `/open-apis/bot/v1/openclaw_bot/ping` hangs on Alibaba Cloud ECS / Node.js 24 while native HTTPS succeeds.

## Best Possible Solution

Keep the fix inside the Feishu plugin: bound and degrade startup/status probes, then bypass or replace the SDK axios path for the affected Feishu probe/API calls only if live or deterministic dependency proof confirms that transport is the root cause.

## Reproduction Assessment

Unclear. The report and corroborating logs provide a credible field path, and current source still uses the same SDK-backed ping, but this review did not establish the Alibaba Cloud ECS / Node.js 24 axios hang on current main without the reported environment and Feishu credentials.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable direction is plugin-local because the Feishu plugin owns the SDK client wrapper, ping probe, setup/status probe, and monitor startup lifecycle.
