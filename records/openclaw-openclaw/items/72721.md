## Summary

Keep open. Current main still sends the reported Feishu bot ping through the Lark SDK HTTP path, WebSocket remains the documented default, and the code currently bounds/logs probe timeouts rather than proving the Alibaba Cloud ECS / Node.js 24 axios hang is fixed; related PR #67466 is still an open mitigation candidate.

## What This Changes

Report a Feishu WebSocket-mode gateway startup timeout where the Lark SDK axios POST to `/open-apis/bot/v1/openclaw_bot/ping` hangs on Alibaba Cloud ECS / Node.js 24 while native HTTPS succeeds.

## Best Possible Solution

Keep the fix inside the Feishu plugin by making startup/status probes bounded and non-blocking, then either replace or bypass the SDK axios path for affected probes if live or deterministic dependency proof confirms that transport is the root cause.

## Reproduction Assessment

Unclear. The issue has concrete steps, diagnostics, and corroborating user logs, but high-confidence reproduction still needs Feishu credentials plus the reported Alibaba Cloud ECS / Node.js 24 environment or a deterministic fixture for the SDK axios hang.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable direction is plugin-local because the Feishu plugin owns the Lark SDK client wrapper, ping probe, setup/status probe, and monitor startup lifecycle.
