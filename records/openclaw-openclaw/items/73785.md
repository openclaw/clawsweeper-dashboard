## Summary

Keep open. The PR addresses a real current-main heartbeat/no-op leak, but it needs rebase/adaptation and a changelog entry, and the proposed `systemEvent` message-mode switch would stop suppressing short no-op summaries that end in `HEARTBEAT_OK`.

## What This Changes

The PR adds shared exact no-op sentinel detection for heartbeat/system-event replies and tests suppression of `NO_REPLY`, `NO_NEW_AUDIO`, `SESSION_WATCHDOG_OK`, and exact `HEARTBEAT_OK` while preserving meaningful exec summaries.

## Best Possible Solution

Land a rebased narrow exact-sentinel suppression that preserves short `HEARTBEAT_OK` no-op summary suppression, keeps relayable exec summaries deliverable, adds regression coverage and a changelog entry, and leaves broader job-defined no-reply semantics to the canonical follow-up issue.

## Reproduction Assessment

Yes, source-level. Current main only normalizes `HEARTBEAT_OK` in the heartbeat runner, so exact `NO_NEW_AUDIO` or `SESSION_WATCHDOG_OK` replies remain ordinary deliverable text, and relayable exec fallback can still re-expose exact `HEARTBEAT_OK`.

## Solution Assessment

No, not as-is. Central exact-sentinel detection is the right direction, but the patch should be rebased and should not switch all cron/system-event replies to message-mode stripping because that can leak short no-op summaries the existing heartbeat contract suppresses.
