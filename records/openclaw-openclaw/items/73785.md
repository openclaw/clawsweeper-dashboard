## Summary

Keep open: the branch addresses the exact relayable exec `HEARTBEAT_OK` leak, but it still routes cron/system-event replies through message-mode stripping, which can make short no-op summaries visible again.

## What This Changes

The PR adds exact no-op sentinel detection for heartbeat replies, applies it before delivery and exec fallback, and adds heartbeat-runner regression tests.

## Best Possible Solution

Keep the exact sentinel suppression, but use message-mode stripping only for relayable exec summaries; cron and non-relayable no-op events should preserve heartbeat-mode `ackMaxChars` suppression with a regression test.

## Reproduction Assessment

Yes, source-level. Current main can restore an exact `HEARTBEAT_OK` through the relayable exec fallback, and the PR regression is visible from the `hasCronEvents` message-mode switch; no tests were run in this read-only review.

## Solution Assessment

No, not as written. The sentinel helper is a narrow fit for the exec leak, but the mode switch should not include cron/no-op events that rely on heartbeat-mode short-ack suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve heartbeat-mode suppression for cron events:** `src/infra/heartbeat-runner.ts:1804`
  - body: Passing `hasExecCompletion || hasCronEvents` makes cron/system-event replies use message mode. In current `stripHeartbeatToken`, heartbeat mode drops short ack remainders via `ackMaxChars`, but message mode returns the stripped text; a cron/no-op reply such as `Watchdog clean HEARTBEAT_OK` would now send `Watchdog clean`. Limit message mode to relayable exec summaries and keep cron/no-op events on heartbeat mode.
  - confidence: 0.87
