## Summary

Keep open. Current main still lacks the no-op sentinel detector and suppression behavior this PR adds, but the PR needs repair before merge because its broad exec-system-event suppression can hide relayable exec completions that current main now distinguishes.

## What This Changes

The PR adds a shared exact no-op sentinel detector, uses it in heartbeat reply normalization for heartbeat/cron/exec system events, and adds runHeartbeatOnce tests for sentinel suppression and exec summaries.

## Best Possible Solution

Update the PR so exact no-op sentinels are suppressed at the heartbeat/system-event delivery boundary while preserving current main's relayable exec-completion behavior for captured output and failures; keep a broader structured noReply/suppressDelivery job-result API as a separate maintainer-scoped follow-up.

## Reproduction Assessment

Yes. The linked issue has a source-level trace for the sentinel leak, and the PR adds focused runHeartbeatOnce tests; the review finding is also reproducible by following current main's hasRelayableExecCompletion path for failed or output-bearing exec events into the PR's broad exact-sentinel skip.

## Solution Assessment

No, not as currently proposed against current main. The central exact-sentinel detector is a good narrow direction, but the implementation should be rebased and constrained so relayable exec completions are not silently swallowed, and it needs the required changelog entry.
