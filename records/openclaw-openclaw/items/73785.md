## Summary

Keep open: the PR’s exact no-op sentinel suppression is useful, but the latest head still routes cron/system-event replies through message-mode stripping, which bypasses the existing heartbeat-mode short-ack suppression. The supplied proof is also mocked test output rather than a real gateway/channel run.

## What This Changes

The PR adds shared no-op sentinel detection, applies it in heartbeat-runner reply normalization, and adds focused heartbeat-runner regression tests.

## Best Possible Solution

Keep the exact sentinel suppression, but preserve heartbeat-mode ackMaxChars suppression for cron/no-op events and use message-mode handling only where relayable exec summaries need it, then provide real runtime/channel proof.

## Reproduction Assessment

Yes, source-level. Current main can strip an exact HEARTBEAT_OK and then restore it through the relayable exec fallback, and the PR regression is visible where cron events are switched to message mode.

## Solution Assessment

No, not as written. The no-op sentinel helper is a narrow fit for the exec leak, but cron events should keep heartbeat-mode short-ack suppression instead of being treated like message-mode summaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve heartbeat-mode suppression for cron events:** `src/infra/heartbeat-runner.ts:1804`
  - body: Passing `hasExecCompletion || hasCronEvents` makes cron/system-event replies use message mode. In the current stripping contract, heartbeat mode drops short ack remainders via `ackMaxChars`, but message mode returns the stripped text; a cron/no-op reply like `Watchdog clean HEARTBEAT_OK` would now send `Watchdog clean`. Limit message mode to the exec-summary case and keep cron events on heartbeat mode.
  - confidence: 0.88
