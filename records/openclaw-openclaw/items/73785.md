## Summary

Keep open: the PR targets a valid heartbeat no-op leak, but the current diff still has a P2 correctness regression for cron/system-event acks and the supplied proof is only a mocked Vitest run.

## What This Changes

The PR adds no-op sentinel detection in token helpers, applies it in heartbeat-runner normalization, and adds regression tests for exact sentinels plus meaningful exec summaries.

## Best Possible Solution

Keep the exact sentinel suppression, but limit message-mode stripping to relayable exec summaries, preserve heartbeat-mode ackMaxChars suppression for cron/no-op events, and add real channel/runtime proof.

## Reproduction Assessment

Yes, source-level. Current main can restore an exact HEARTBEAT_OK through the relayable exec fallback, and the PR regression is visible from the hasCronEvents message-mode switch; no tests were run in this read-only review.

## Solution Assessment

No, not as written. The sentinel helper is a narrow fit for the exec leak, but cron/no-op events should keep heartbeat-mode short-ack suppression instead of being switched to message mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve heartbeat-mode suppression for cron events:** `src/infra/heartbeat-runner.ts:1804`
  - body: Passing `hasExecCompletion || hasCronEvents` makes cron/system-event replies use message mode. In current `stripHeartbeatToken`, heartbeat mode drops short ack remainders via `ackMaxChars`, but message mode returns the stripped text; a cron/no-op reply such as `Watchdog clean HEARTBEAT_OK` would now send `Watchdog clean`. Limit message mode to relayable exec summaries and keep cron/no-op events on heartbeat mode.
