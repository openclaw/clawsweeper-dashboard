## Summary

Keep open: this targets a real source-reproducible heartbeat leak, but the patch still has a correctness regression in system-event normalization and lacks required after-fix real behavior proof.

## What This Changes

The PR adds exact no-op sentinel detection for heartbeat replies, removes the exec fallback that re-delivers stripped ack tokens, and adds heartbeat-runner regression tests.

## Best Possible Solution

Rebase the narrow sentinel suppression onto current main, remove the exec ack fallback only for no-op tokens, preserve heartbeat-mode suppression for cron/no-op summaries, and add real runtime proof before merge.

## Reproduction Assessment

Yes, source-level. Current main can strip a pure HEARTBEAT_OK reply and then restore it through the relayable exec fallback before delivery; I did not run a live repro in this read-only review.

## Solution Assessment

No, not as written. The exact sentinel helper is a good narrow direction, but applying message-mode stripping to all system events bypasses existing heartbeat-mode ack suppression for cron/no-op summaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve heartbeat-mode ack suppression:** `src/infra/heartbeat-runner.ts:513`
  - body: Passing `systemEvent` for both exec and cron switches `stripHeartbeatToken` to message mode. For a cron/no-op reply like `Watchdog clean HEARTBEAT_OK`, current heartbeat mode drops the short ack summary via `ackMaxChars`, but this patch strips the token and sends `Watchdog clean`, reintroducing visible no-op noise. Limit message mode to relayable exec summaries or keep heartbeat mode for cron events.
  - confidence: 0.84
