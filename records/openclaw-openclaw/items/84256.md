## Summary

Keep open: current main can still reinstall the recorded npm spec during `plugins update --all`, so a manually advanced on-disk plugin can be moved back to an older recorded target. The requested “always latest” behavior conflicts with documented exact-pin semantics, so this needs a maintainer decision on downgrade protection rather than automatic cleanup.

## What This Changes

The issue asks `openclaw plugins update --all` to avoid downgrading npm plugins that were manually updated ahead of their recorded install spec.

## Best Possible Solution

Add downgrade-safe behavior that preserves exact pin semantics, such as warning, failing, or requiring an explicit npm spec before reinstalling a lower version, and document the supported reset path.

## Reproduction Assessment

Yes, source-reproducible: seed an npm install record pinned to an older spec, manually place a newer package version at the install path, then run the current `--all` selection path. The updater resolves the older recorded target and proceeds because the installed version does not exactly match that target.

## Solution Assessment

No, the requested “always latest” behavior is not clearly the best fix because current docs intentionally preserve exact pins. A safer solution is an explicit downgrade guard or maintainer-approved distinction between pinned records and stale official/default-line records.
