## Summary

Keep open: current main still has a source-reproducible path where `plugins update --all` can select the recorded npm spec and reinstall a lower version over a newer out-of-band package. The fix should not simply make `--all` chase latest, because current docs deliberately preserve exact pins and an overlapping open PR needs maintainer alignment rather than a second automatic fix branch.

## What This Changes

The issue asks `openclaw plugins update --all` to avoid reinstalling an older recorded npm plugin spec over a newer manually updated package.

## Best Possible Solution

Add a downgrade-safe managed npm update contract that preserves explicit pins, refuses or warns before installing a lower resolved version unless the user supplied an explicit downgrade spec, and covers the out-of-band manual-update case with regression tests and docs.

## Reproduction Assessment

Yes, source-reproducible: with an npm install record pinned to an older exact spec and a newer package on disk, `--all` supplies no override, the updater resolves the older recorded spec, the equal-version skip fails, and the installer path can run. I did not run the live npm flow because this review is read-only.

## Solution Assessment

No, the issue body's “always latest” expectation is not clearly the best fix because documented behavior intentionally preserves exact pins. The maintainable solution is a downgrade guard or record reconciliation that distinguishes explicit pins from stale out-of-band managed npm state.
