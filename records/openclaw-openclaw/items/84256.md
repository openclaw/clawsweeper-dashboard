- `P2`: The issue is a real plugin CLI update bug with user-visible downgrade risk, but the blast radius is limited to managed plugin update flows and needs a product decision.

## Summary

Keep open: current main and the latest release still route `plugins update --all` through tracked install records, so a newer out-of-band npm install can be reinstalled back to an older recorded exact spec. The source path is clear, but the right fix needs a maintainer contract decision because current docs also promise exact pins remain pinned.

## What This Changes

The issue reports that `openclaw plugins update --all` can reinstall an older recorded npm plugin spec over a newer manually updated plugin version.

## Best Possible Solution

Add a downgrade-safe update contract that preserves intentional pins but skips, warns, or fails before reinstalling a lower npm version unless the user supplies an explicit downgrade target, with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: `--all` selects tracked ids only, the updater resolves the recorded exact npm spec, and the unchanged guard only skips when the installed version equals the resolved target.

## Solution Assessment

No, not as stated. Always resolving to latest is not clearly the best fix because current docs intentionally preserve exact pins; the maintainable path is a maintainer-approved downgrade guard or an explicit distinction between pinned records and default-release-line records.
