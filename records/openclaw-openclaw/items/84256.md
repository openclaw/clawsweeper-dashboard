## Label Justifications

- `P2`: The issue describes a real CLI/plugin update bug with limited blast radius but meaningful user impact when managed plugin code is silently downgraded.

## Summary

Keep open: current main still selects each tracked plugin's recorded npm spec for `plugins update --all`, so a newer manually-updated on-disk plugin can be reinstalled back to an older recorded target. The safe fix needs maintainer judgment because the docs intentionally preserve exact pinned install records.

## What This Changes

Request that `openclaw plugins update --all` avoid downgrading npm plugins from a newer manually-installed version to the older version stored in the plugin index.

## Best Possible Solution

Add an explicit downgrade guard or prompt that preserves documented exact-pin semantics, then document the supported way to reset stale official/default-line records.

## Reproduction Assessment

Yes, source-reproducible: with an npm install record pinned to an older spec and a newer on-disk package, `--all` selects the record, resolves the older target, fails the unchanged check, and installs the older spec. I did not run the live npm flow in this read-only review.

## Solution Assessment

No, the requested “always latest” behavior is not clearly the best fix because current docs intentionally preserve exact pins. A safer solution is a maintainer-approved downgrade guard or explicit distinction between stale records and intentionally pinned records.
