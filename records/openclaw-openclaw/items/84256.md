## Summary

Keep open: current main and the latest shipped tag still have a source-reproducible path where `plugins update --all` can select an older recorded npm spec and reinstall it over a newer manually-updated package. The remaining fix needs a maintainer contract decision because “always latest” conflicts with documented exact-pin reuse.

## What This Changes

The issue reports that `openclaw plugins update --all` can reinstall an older recorded npm plugin spec over a newer manually-updated package version.

## Best Possible Solution

Add a downgrade-safe update contract that preserves explicit pins, such as skip/warn/fail before reinstalling a lower npm version unless the user supplies an explicit downgrade spec, with regression coverage and matching CLI docs.

## Reproduction Assessment

Yes, source-reproducible: with a tracked npm install record pinned to an older exact spec and a newer package on disk, current main selects the record, resolves the older target, fails the equal-version skip, and can install the older version. I did not run a live npm flow because this review is read-only.

## Solution Assessment

No, the requested “always latest” behavior is not clearly the best fix because current docs intentionally preserve exact pins. The safer path is a maintainer-approved downgrade guard or explicit distinction between pinned records and stale official/default-line records.
