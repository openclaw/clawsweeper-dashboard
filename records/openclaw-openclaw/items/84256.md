## Summary

Keep open: current `origin/main` still has a source-reproducible path where `plugins update --all` selects the recorded npm install spec and can reinstall a lower version after an out-of-band npm update. The remaining fix needs a maintainer contract decision because the reported “always latest” expectation conflicts with documented exact-pin reuse.

## What This Changes

Make `openclaw plugins update --all` avoid downgrading npm plugins that were manually updated ahead of their recorded install spec.

## Best Possible Solution

Add a downgrade-safe update contract that preserves explicit pins, such as skip/warn/fail before reinstalling a lower npm version unless the user supplies an explicit downgrade spec, with regression coverage and matching CLI docs.

## Reproduction Assessment

Yes, source-reproducible: with a tracked npm install record pinned to an older exact spec and a newer package on disk, `--all` selects the record, resolves the older target, fails the equal-version skip, and can install the older version. I did not run the live npm flow because this review is read-only.

## Solution Assessment

No, the requested “always latest” behavior is not clearly the best fix because current docs intentionally preserve exact pins. The safer path is a maintainer-approved downgrade guard or explicit distinction between pinned records and stale official/default-line records.
