## Summary

Keep open: current main still selects the recorded npm install spec for `plugins update --all` and can reinstall that lower target when the on-disk package was manually moved ahead. The report conflicts with documented exact-pin behavior, so the remaining decision is how to add downgrade protection without breaking pinned plugin installs.

## What This Changes

The issue asks for `openclaw plugins update --all` to avoid downgrading npm plugins that were manually updated outside OpenClaw to a newer version than the recorded install spec.

## Best Possible Solution

Add downgrade-safe behavior that preserves explicit pin semantics, such as warning/failing or requiring an explicit package spec before reinstalling a lower npm version, and document the supported way to reset a pinned record to the default release line.

## Reproduction Assessment

Yes, source-reproducible: with an npm install record pinned to an older spec and a newer on-disk package, `--all` selects the record, probes the older target, fails the unchanged check, and installs that older spec. I did not execute the live npm flow because the checkout review is read-only.

## Solution Assessment

No, the requested “always latest” behavior is not clearly the best fix because current docs intentionally preserve exact pins. A safer solution is an explicit downgrade guard or maintainer-approved distinction between pinned records and stale official/default-line records.
