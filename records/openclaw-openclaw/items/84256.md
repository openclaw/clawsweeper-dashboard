## Summary

Keep open: current main and latest release still route standalone `openclaw plugins update --all` through the recorded plugin spec, and the docs/tests intentionally preserve exact pins for manual plugin updates. The remaining fix needs a maintainer decision on downgrade protection versus exact-pin reuse, with related field evidence adding the beta-channel variant.

## What This Changes

The issue asks for `openclaw plugins update --all` to avoid downgrading npm plugins when the recorded install spec points to an older version than the package currently installed on disk.

## Best Possible Solution

Add a maintainer-approved downgrade-safe plugin update contract that preserves intentional exact pins while refusing, warning, or repairing stale official/default-line records before reinstalling a lower npm version, with CLI docs and regression tests for exact-record and beta-channel cases.

## Reproduction Assessment

Yes, source-reproducible: `plugins update --all` selects installed plugin ids, the CLI passes no official sync/channel policy, and the updater falls back to `record.spec`. I did not run a live mutating npm downgrade flow in this read-only review.

## Solution Assessment

No, the literal “always latest” expectation is not the best unqualified fix because current docs intentionally preserve exact pins. The best solution is a maintainer-approved downgrade guard or official/default-line repair path that distinguishes intentional pins from stale exact records.
