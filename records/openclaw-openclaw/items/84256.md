## Summary

Keep open: current main still has a source-reproducible path where `plugins update --all` can select an older recorded npm spec and reinstall it over a newer manually updated package. The fix should not be auto-queued until maintainers decide how downgrade protection should coexist with documented exact-pin reuse.

## What This Changes

The issue requests that `openclaw plugins update --all` not reinstall an older recorded npm plugin spec after the package has been manually updated outside OpenClaw.

## Best Possible Solution

Add a downgrade-safe update contract that preserves explicit pins but refuses, warns, or requires an explicit override before installing a lower npm version, with regression coverage and matching CLI docs.

## Reproduction Assessment

Yes, source-reproducible: with an older exact npm install record and a newer package on disk, `--all` selects the id, resolves the older recorded spec, fails the equal-version skip, and can run the installer. I did not run the live npm flow because this review is read-only.

## Solution Assessment

No, the requested “always latest” behavior is not clearly the best fix because current docs intentionally preserve exact pins. The safer path is a maintainer-approved downgrade guard or explicit distinction between pinned records and stale official/default-line records.
