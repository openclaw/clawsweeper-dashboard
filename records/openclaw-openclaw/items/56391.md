## Summary

Keep this PR open, but do not merge it as-is. Current main still does not implement the PR’s requested blank-name handling, while the proposed literal `_blank` fallback introduces concrete streaming recovery, dispatch-collision, replay-sanitization, and security regressions.

## What This Changes

The PR changes embedded runner tool-call normalization to convert blank or whitespace-only tool names to `_blank`, treats that sentinel as incomplete for unknown-tool loop classification, and updates related stream normalization tests.

## Best Possible Solution

Revise blank-name handling so malformed blank tool calls stop retrying without becoming a callable tool name. The final implementation should preserve late id-based recovery, use a non-dispatchable/internal invalid-name path or direct loop-guard rewrite, keep replay pruning semantics intact, and add focused coverage for blank-first/id-later streaming, `_blank` name collisions, and no-allowlist replay pruning.

## Reproduction Assessment

Yes. The linked issue provides provider logs for the user-visible blank-name loop, and current main has focused source-level tests around blank names, id inference, unknown-tool loop guarding, and replay sanitization; the PR regressions can be reproduced with targeted unit tests without live credentials.

## Solution Assessment

No. The requested bug fix is valid, but a normal `_blank` string is not the safest or narrowest solution because it mutates recoverable stream state and remains a legal registered tool name.
