## Summary

Keep open. The blank-tool-name loop is real, but this PR is not mergeable as-is because the literal `_blank` fallback remains a normal tool name and can regress streamed ID recovery, replay pruning, and dispatch safety.

## What This Changes

The PR changes embedded runner tool-call normalization to map unrecoverable blank or whitespace-only tool names to `_blank`, treats that sentinel as incomplete for unknown-tool loop classification, and updates related stream tests.

## Best Possible Solution

Handle unrecoverable blank tool calls through an internal, non-dispatchable invalid-tool path that stops retry loops while preserving id-based recovery and replay pruning, with focused coverage for streaming, replay, and sentinel-collision cases.

## Reproduction Assessment

Yes. The linked issue has concrete provider log evidence for the user-visible loop, and the PR regressions have deterministic source-level paths: blank-first/id-later streaming, a registered `_blank` tool, and no-allowlist replay sanitization.

## Solution Assessment

No. A normal string sentinel is not the narrowest maintainable fix because it mutates in-flight stream state and remains callable; a deferred invalid-name handling path or non-dispatchable sentinel would be safer.
