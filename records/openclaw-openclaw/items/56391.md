## Summary

Keep this PR open, but the current patch should not merge as-is. The linked blank-tool-name loop remains a valid bug area on current main, while the literal `_blank` fallback introduces concrete streaming recovery, replay sanitization, and unintended tool-dispatch regressions.

## What This Changes

The branch changes embedded runner tool-call normalization to map blank or whitespace-only tool names to `_blank`, treats that sentinel as incomplete for unknown-tool loop classification, and updates related stream normalization tests.

## Best Possible Solution

Revise blank-name handling so unrecoverable blank tool calls stop retrying without becoming a dispatchable real tool name. The end state should preserve late id-based recovery, make any sentinel non-callable or internal, keep replay pruning semantics intact, and add focused coverage for blank-first/id-later streaming, `_blank` tool-name collision, and no-allowlist replay pruning.

## Reproduction Assessment

Yes. The linked issue provides provider logs for the user-visible loop, and current main has source-level test paths around `wrapStreamFnTrimToolCallNames` for blank names, id inference, ambiguous ids, and whitespace-only names; the PR regressions can be reproduced with targeted stream/replay tests without requiring live credentials.

## Solution Assessment

No. The requested direction is valid, but a normal `_blank` string is not the narrowest maintainable fix because it mutates in-flight stream state and remains a legal registered tool name; a safer solution should defer terminal invalid-name handling or use a non-dispatchable failure path.

  - confidence: 0.88
