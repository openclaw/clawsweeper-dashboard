## Label Justifications

- `impact:other`: The issue affects CLI-backend and subagent throughput, which is meaningful but outside the specific data, session, auth, delivery, and availability categories.

## Summary

Current main still collapses independent fresh Claude CLI invocations in the same workspace onto one strictly serialized queue, while the configured higher-level concurrency is applied before that bottleneck. The linked symptom report corroborates this root cause, no open fixing PR was found, and the issue is a focused, valid core runtime bug.

## What This Changes

Allow fresh Claude CLI runs in one workspace to execute concurrently while continuing to serialize turns that resume the same CLI session.

## Best Possible Solution

Model the queue around the mutable session resource: use the resumed session ID for continuation turns and a unique run ID for fresh invocations, with separate tests and live proof for both concurrency and exclusion.

## Reproduction Assessment

Yes at source level with high confidence: current main deterministically assigns one workspace queue key to fresh same-workspace Claude CLI runs, and the reporter provides a concrete four-child timing reproduction on v2026.6.5. A credentialed current-main execution was not performed in this read-only review.

## Solution Assessment

Yes, with one safeguard: session-or-run keying is the narrowest maintainable solution, but it must retain and test serialization for concurrent resumes of the same Claude session.
