## Label Justifications

- `P1`: The shipped backend silently defeats configured parallelism across subagents and normal multi-session traffic, causing severe latency in active workflows.
- `impact:other`: The concrete impact is broad runtime throughput and responsiveness degradation outside the narrower owned loss, state, auth, security, or crash categories.

## Summary

Current main and v2026.6.5 retain the reported workspace-wide serialization, and no canonical fix PR currently owns the work. The bug is well supported, but later discussion shows that the safe repair requires an explicit runtime-owner decision because fresh subagent and ordinary chat runs share this lane while some CLI artifacts may require serialization.

Likely related people: @steipete (high confidence, introduced the Claude CLI queue hardening) and @vincentkoc (medium confidence, recent CLI-runner contributor).

## What This Changes

Fresh Claude CLI sessions sharing a workspace are serialized behind one backend queue key even when higher-level concurrency lanes permit parallel execution.

## Best Possible Solution

Make every mutable CLI artifact explicitly session- or run-owned, retain serialization for turns resuming the same Claude session, and permit unrelated fresh runs to overlap consistently across subagent and ordinary traffic paths.

## Reproduction Assessment

Yes. Current main deterministically maps fresh same-workspace Claude CLI runs to one strict queue key, focused tests assert that behavior, and the reporter supplied matching v2026.6.5 queue and execution timestamps; this review did not independently run the external Claude CLI.

## Solution Assessment

No final proposed option is yet proven best. A run-id-only key is safe only after shared mutable state is isolated, while per-spawn workspaces are incomplete for the broader fresh-session contention demonstrated in the discussion.
