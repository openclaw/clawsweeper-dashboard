## Summary

Current main and the latest release still serialize fresh same-workspace Claude CLI runs behind one queue key. The report has strong source and runtime evidence, and the specifically linked fixing PR remains open, so this issue should stay open until that implementation is reviewed and either merged or replaced.

## What This Changes

Scope Claude CLI queueing so independent fresh sessions sharing a workspace can run concurrently while resumed work for the same Claude session remains serialized.

## Best Possible Solution

Land one canonical queue identity derived from the existing Claude live-session owner boundary, with focused coverage proving parallel independent fresh sessions, serialized same-owner and resumed turns, and safe behavior for callers lacking complete owner identity.

## Reproduction Assessment

Yes. Current main deterministically maps fresh same-workspace Claude CLI runs onto one strict queue key, and the reporter supplied instrumented runtime timestamps showing those runs executing serially.

## Solution Assessment

Yes in direction. Reusing the established Claude live-session owner identity is narrower and safer than globally disabling serialization, adding a configuration escape hatch, or creating a temporary workspace for every spawn; the exact linked PR still needs review of its fallback and resume paths.
