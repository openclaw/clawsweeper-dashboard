- `P1`: A shipped core backend silently turns configured concurrent subagent fan-out into serial execution for real users.

## Summary

Current main and v2026.6.5 retain the reported workspace-wide queueing behavior. Source, tests, documentation, and the linked live report support treating this as a valid core runtime bug that defeats configured Claude CLI subagent concurrency.

## What This Changes

Fix Claude CLI queueing so independent fresh sessions in one workspace can run concurrently while turns for the same live or resumed session remain serialized.

## Best Possible Solution

Use the existing per-session lifecycle identity for fresh Claude runs, retain persisted Claude-session serialization on resume, and prove parallel fresh-session execution plus serialized same-session turns.

## Reproduction Assessment

Yes at source level. Current main deterministically collapses fresh Claude runs in one workspace onto one strict queue key, and https://github.com/openclaw/openclaw/issues/91940 supplies observed v2026.6.5 timing evidence; this read-only review did not execute a live Claude CLI account.

## Solution Assessment

Yes, with the queue keyed at the existing live-session ownership boundary. Adding a config escape hatch or documenting the limitation would leave the broken default concurrency semantics intact.
