## Summary

Keep open: current main still lacks operator-approvals TLS fingerprint propagation, but this PR is not merge-ready because it can apply a configured remote fingerprint to a local fallback Gateway and only has test/CI proof.

## What This Changes

The PR adds TLS fingerprint selection to the operator approvals Gateway client and adds focused TLS unit coverage.

## Best Possible Solution

Land a rebased shared approval-client fix that passes the correct TLS fingerprint for local WSS, remote config, env override, and CLI override cases while keeping local fallback on the local runtime fingerprint.

## Reproduction Assessment

Yes, at source level. Current main routes approval resolution through `withOperatorApprovalsGatewayClient`, whose `GatewayClient` options omit `tlsFingerprint`, while `GatewayClient` only enables pinned self-signed WSS handling when that option is supplied.

## Solution Assessment

No. The direction is right, but the PR should guard remote fingerprint use on an actual remote target or env override, preserve local fallback behavior, and be rebased onto current main before merge.
