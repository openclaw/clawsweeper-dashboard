## Summary

Keep open. Current main still lacks TLS fingerprint propagation for operator approval Gateway clients, so the PR addresses a real gap, but the branch should not merge as-is because the new remote fingerprint branch can override the local TLS fingerprint in the remote-mode/missing-remote-url fallback path.

## What This Changes

The PR adds TLS fingerprint propagation and focused tests for operator approval Gateway clients, plus updates the existing bootstrap mock for URL source handling.

## Best Possible Solution

Port the TLS fingerprint resolution onto current main, keep remote pins limited to real remote/env targets, let local fallback use the local TLS runtime fingerprint, add fallback coverage, and add the required changelog entry.

## Reproduction Assessment

Yes. The issue can be reproduced by inspection with `gateway.tls.enabled=true` and a WSS local operator-approval bootstrap: current main never passes `tlsFingerprint`, while the PR's added logic can be driven through the remote-mode/missing-remote-url fallback to select the wrong remote pin.

## Solution Assessment

No. The direction is right, but the PR is not the best final fix until it is rebased onto current main and guards remote TLS pinning on a resolved remote target or env override so the fallback path remains local.
