## Summary

Keep open: current main still lacks operator-approvals TLS fingerprint propagation, but this PR still has a remote-fallback logic bug and lacks real behavior proof before merge.

## What This Changes

The PR adds TLS fingerprint propagation to the shared operator approvals Gateway client and focused tests for local WSS and URL override cases.

## Best Possible Solution

Land a rebased shared approval-client fix that uses the local TLS runtime for local WSS, uses the remote fingerprint only for actual remote or env-targeted cases, and covers the remote-missing local fallback.

## Reproduction Assessment

Yes, at source level. Current main routes approval resolution through `createOperatorApprovalsGatewayClient`, which omits `tlsFingerprint`, while `GatewayClient` requires that option for pinned self-signed WSS handling.

## Solution Assessment

No, not as currently written. The direction is right, but the PR should guard remote fingerprint use on an actual remote target or env override and preserve local fallback behavior before merge.
