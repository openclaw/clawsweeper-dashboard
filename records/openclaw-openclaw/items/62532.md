## Summary

Keep open: current main still omits TLS fingerprint propagation in the shared operator approvals Gateway client, and this PR targets the right bug, but the head has a remote-fallback correctness blocker, lacks real after-fix runtime proof, and is currently conflicting.

## What This Changes

The PR adds TLS fingerprint resolution to the shared operator approvals Gateway client and focused tests for local WSS, env override, and CLI override cases.

## Best Possible Solution

Land or replace this with a rebased shared approval-client fix that uses the local TLS runtime for local non-override WSS, applies remote pinning only to actual remote or env-targeted cases, and includes focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main routes approval resolution through createOperatorApprovalsGatewayClient, which omits tlsFingerprint, while GatewayClient requires that option to accept a pinned local self-signed WSS certificate.

## Solution Assessment

No, not as currently written. The direction is right, but the PR should guard remote fingerprint use on an actual remote target or env override and preserve local fallback behavior before merge.
