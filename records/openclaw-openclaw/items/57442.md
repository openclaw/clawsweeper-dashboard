## Summary

Keep open. Current main still omits the requested timing from both relevant surfaces: the shared `/status` and `session_status` renderer only shows the session key plus last update age, and the per-response usage footer has no duration input even though the run diagnostics record `durationMs`. The related #68226 issue and #49762 PR are still open and cover narrower pieces, so this needs maintainer coordination rather than cleanup closure.

## What This Changes

Restore elapsed or response timing in the status card shown by `session_status`, `/status`, and the response footer after the 2026.3.28 upgrade.

## Best Possible Solution

Keep this as the broad timing-observability tracker until maintainers choose the canonical path. The durable end state should add session elapsed time in the shared status renderer using `sessionStartedAt` with lifecycle fallback, and handle response duration in the usage footer in coordination with #49762 or a replacement PR.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the current status/session-status render path never reads session start data, and the response usage formatter has no duration parameter while diagnostics already record `durationMs`.

## Solution Assessment

Unclear for the whole report. The maintainable direction is split by surface: session lifetime belongs in the shared status renderer, while response duration belongs in the usage footer; the open #68226 tracker and #49762 PR need maintainer coordination first.
