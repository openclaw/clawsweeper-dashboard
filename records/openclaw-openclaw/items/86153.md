## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Align broker capability config with the SDK shape:** `extensions/channel-broker/src/config-schema.ts:5-22`
  - body: `ResolvedChannelBrokerAccount.capabilities` is typed and consumed as broker platform capabilities with nested `delivery`, `live`, `receive`, and `native` maps, and `brokerPlatformSupports` only reads those keys. This strict schema instead allows flat keys like `threads`, `draftPreview`, and `receipts`, so SDK-shaped config is rejected while schema-accepted config is ignored by the support checks. Please make this layer use the SDK capability shape instead of relying on a later stacked PR.
  - confidence: 0.9
- **[P2] Validate provider receipts before trusting them:** `extensions/channel-broker/src/runtime.ts:46-50`
  - body: `sendBrokerOutboundHttp` parses provider JSON and accepts any object as `BrokerReceiptV1`. `sendChannelBrokerText` then reads `receipt.messageIds[0]` and maps `receipt.messageIds`, so a 200 response like `{ "status": "sent" }` throws an unstructured TypeError and malformed status/request/provider fields can enter receipt metadata. Validate and normalize the receipt shape before returning it from the external provider boundary.
  - confidence: 0.88
