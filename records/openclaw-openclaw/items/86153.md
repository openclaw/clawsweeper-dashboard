Full review comments:

- **[P2] Align configured capabilities with the SDK shape:** `extensions/channel-broker/src/config-schema.ts:5-22`
  - body: `BrokerProviderCapabilities` is grouped under `delivery`, `live`, `receive`, and `native`, but the channel config schema accepts flat keys like `threads`, `draftPreview`, and `progressPreview`. Those values are copied into resolved account capabilities but ignored by the SDK support checks, so an operator can configure capability support that the broker then treats as unsupported or inconsistent. Please make the config schema/types/docs use the SDK shape and add coverage that configured capabilities drive `brokerPlatformSupports`.
  - confidence: 0.9
- **[P2] Use pre-auth limits before verifying broker webhooks:** `extensions/channel-broker/src/http-routes.ts:80-84`
  - body: This route reads the body with the `post-auth` profile before the HMAC signature is checked. The shared guard contract reserves that looser 1 MiB/30s profile for authenticated requests; unauthenticated webhook callers should stay on the tighter pre-auth limits until the signature succeeds, otherwise this new public route has a larger unauthenticated resource-use window.
  - confidence: 0.92
