## Label Justifications

- `P1`: This is a user-facing regression for credential-backed skill CLI authentication on headless hosts with multiple user confirmations.
- `impact:security`: The issue concerns whether password/token-shaped skill values enter host process and child-process environments.
- `impact:auth-provider`: The affected values are authentication credentials supplied through skill env, apiKey, and SecretRef-style configuration.

## Summary

Keep open: current main and the latest release still block credential-shaped keys from `skills.entries.<skill>.env` unless the skill metadata declares the same key, and the remaining fix overlaps with an open inverse secret-leak boundary that needs maintainer security/product judgment.

## What This Changes

The issue requests that per-skill environment variables configured under `skills.entries.*.env` reach subprocesses spawned by the `exec` tool.

## Best Possible Solution

Land a maintainer-approved credential boundary where intended skill-owned host exec commands can receive configured skill env credentials while unrelated generic exec children do not inherit active skill secrets.

## Reproduction Assessment

Yes, source-backed. Current main blocks `GOG_KEYRING_PASSWORD`-style configured keys at the skill env sanitizer unless metadata declares them, while allowed host env values are forwarded into `exec` child environments.

## Solution Assessment

Unclear as a final product choice. Adding configured keys to the sensitive allowlist is the narrow availability fix, but it is not the best complete solution until paired with a deliberate generic-exec secret-isolation boundary.
