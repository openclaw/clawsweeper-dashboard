## Summary

Keep open: current main and the latest release still block sensitive-name user-configured skill env keys such as `GOG_KEYRING_PASSWORD` unless the key is declared in skill metadata, while the linked availability and security fixes remain open or conflicting.

## What This Changes

The issue reports that skill-scoped environment variables from `skills.entries.<skill>.env` are not available to commands run through the agent `exec` tool.

## Best Possible Solution

Define one skill-owned credential delivery boundary: intended skill commands receive declared/configured credentials, while generic `exec` and unrelated harnesses do not inherit active skill secrets, with docs and tests covering host and sandbox behavior.

## Reproduction Assessment

Yes. Source inspection shows current main blocks sensitive-name skill env keys unless skill metadata declares them, and the bundled gog skill does not declare the reported key; this read-only review did not run a live secret-bearing CLI command.

## Solution Assessment

No. Blanket inheritance into generic `exec` is not the best fix because it conflicts with the established child-process anti-leak boundary; the safer solution is scoped credential delivery for the owning skill command path.
