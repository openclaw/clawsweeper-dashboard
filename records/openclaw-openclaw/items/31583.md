## Summary

Keep open: current main still blocks credential-shaped keys from `skills.entries.*.env` unless the skill metadata also declares that exact env key, and the open fix path needs maintainer security review because a related open issue tracks the inverse leak of skill secrets into generic exec children.

## What This Changes

The issue asks OpenClaw to pass skill-scoped `skills.entries.*.env` variables, including credential-like keys, into `exec` subprocesses for skill CLI authentication.

## Best Possible Solution

Land a maintainer-approved scoped credential-delivery fix that gives the intended skill CLI access to explicit `skills.entries.*.env` values while stripping skill-scoped secrets from unrelated generic exec children.

## Reproduction Assessment

Yes from source inspection: a `skills.entries.*.env` key matching `_PASSWORD` or `_TOKEN` is blocked unless the skill metadata also declares that key, matching the reported `GOG_KEYRING_PASSWORD` failure. I did not run a live `gog` reproduction in this read-only review.

## Solution Assessment

No, the current open availability PR is only a plausible partial fix until maintainers reconcile it with the open secret-leak issue. The best fix should define the intended skill-command boundary rather than simply making all configured skill secrets process-wide.
