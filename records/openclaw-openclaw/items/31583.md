## Summary

Keep open. Current main still blocks the reported sensitive configured skill env key before host exec can inherit it, and the issue is paired with an open fix PR plus an open security follow-up that need maintainer/security review.

## What This Changes

The issue asks for `skills.entries.*.env` values to be available to subprocesses spawned by the `exec` tool so skill-owned CLIs can authenticate non-interactively.

## Best Possible Solution

Land or replace the linked fix with a scoped env boundary that lets selected skill-owned commands receive explicit configured env values while generic exec strips active skill secrets and rejects unsafe host-env conflicts.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare it as required env metadata, and current main blocks the configured override before host exec can inherit `process.env`; no live `gog` auth run was performed.

## Solution Assessment

No, not as a broad inherit-all fix. The maintainable solution needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
