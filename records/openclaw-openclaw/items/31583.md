## Summary

Keep this open: current main still blocks sensitive-pattern keys from `skills.entries.*.env`, and an open fix PR already references this issue with GitHub closing syntax.

## What This Changes

The issue reports that skill-scoped environment variables configured under `skills.entries.*.env` are not available to commands run through the `exec` tool.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50960 or an equivalent narrow change that admits explicitly configured skill env secrets for eligible host runs while preserving hard blocks and restoration coverage.

## Reproduction Assessment

Yes, by source inspection against current main. A key like `GOG_KEYRING_PASSWORD` is staged from `skills.entries.gog.env`, then blocked by the sensitive-pattern sanitizer because the `gog` metadata does not declare it.

## Solution Assessment

Yes. The narrow maintainable fix is in skill env override admission and scoping, because host `exec` already inherits `process.env` after those overrides are applied.
