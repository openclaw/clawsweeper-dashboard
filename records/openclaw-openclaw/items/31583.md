## Summary

Keep this open: current main still blocks sensitive-pattern keys from `skills.entries.*.env`, and https://github.com/openclaw/openclaw/pull/50960 is still open with closing syntax for this issue.

## What This Changes

The issue asks OpenClaw to pass per-skill environment variables from `skills.entries.*.env` into host `exec` subprocesses.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50960 or an equivalent narrow fix that admits explicitly configured skill env secrets only for eligible runs while preserving hard blocks and conflict handling.

## Reproduction Assessment

Yes. Source inspection shows `GOG_KEYRING_PASSWORD` is staged from `skills.entries.gog.env` and then blocked by the sensitive-pattern sanitizer because configured env keys are not added to `allowedSensitiveKeys`.

## Solution Assessment

Yes for the ownership boundary, but not by closing this issue yet. The narrow maintainable fix belongs in skill env override admission and scoping, because host `exec` already inherits `process.env` after those overrides are applied.
