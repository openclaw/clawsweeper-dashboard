## Label Justifications

- `P1`: This confirmed regression blocks authentication for multiple CLI-backed skills, and its repair must safely resolve an active credential-exposure boundary.
- `impact:security`: The issue directly concerns whether skill-scoped credentials are exposed through process and child-command environments.

## Summary

Keep open: current main and v2026.6.6 still reject credential-like keys supplied only through `skills.entries.*.env`, while the available fix conflicts with the unresolved requirement to keep skill secrets out of unrelated generic exec children.

## What This Changes

Allow user-configured per-skill environment variables, including credential-like names, to reach commands launched through the exec tool.

## Best Possible Solution

Introduce an owner-scoped execution contract that supplies only the invoking skill's declared environment to its command while generic exec children exclude active skill credentials, with focused tests covering host and sandbox execution.

## Reproduction Assessment

Yes, at source level: configure a credential-like key such as `GOG_KEYRING_PASSWORD` only in `skills.entries.gog.env`; the sanitizer rejects it unless the installed skill metadata declares that exact key, leaving it absent from the environment inherited by exec.

## Solution Assessment

No, neither open patch is presently the complete best fix: broad admission restores availability but expands generic secret exposure, while blanket stripping preserves isolation but breaks the reported skill command path.
