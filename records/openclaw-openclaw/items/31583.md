## Label Justifications

- `P1`: This is a source-backed regression that breaks skill CLI authentication for real users and is still present in the latest release.
- `impact:security`: The issue is about whether configured skill secrets are blocked from or exposed to child process environments.
- `impact:auth-provider`: The reported failure prevents skill-specific authentication secrets such as keyring passwords or API tokens from reaching the intended CLI subprocess.

## Summary

Keep open. Current `main` and latest release `v2026.5.12` still block the reported configured sensitive skill env key before host `exec` can inherit it, and the remaining work is already paired with open PR/security follow-up paths that need maintainer security review.

## What This Changes

The issue reports that `skills.entries.*.env` values such as `GOG_KEYRING_PASSWORD` are not available to subprocesses launched through the agent `exec` tool.

## Best Possible Solution

Land or replace the linked fix with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets and unsafe host-env overrides.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare that key as required metadata, and current `main` blocks it before host `exec` can inherit `process.env`; no live `gog` auth run was performed.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
