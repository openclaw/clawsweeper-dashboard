## Summary

Keep open. Current `main` still blocks the reported sensitive configured key before host `exec` can inherit it, and this issue is paired with open implementation and security follow-up work rather than a cleanup-close candidate.

## What This Changes

The issue reports that `skills.entries.<skill>.env` values are not inherited by subprocesses spawned through the `exec` tool, breaking skill-owned CLI authentication on host runs.

## Best Possible Solution

Land or replace the linked fix with scoped behavior that lets explicit selected-skill env values reach the intended skill-owned host command path while stripping skill secrets from unrelated generic exec children.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare it as a required env key, and the current sanitizer blocks it before host `exec` can inherit `process.env`; no live `gog` auth run was performed.

## Solution Assessment

Unclear as a broad inherit-all fix. The safer direction is a scoped fix that admits explicitly configured sensitive env only for the intended skill-owned execution path and preserves generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
