## Summary

Keep this open: current main still blocks sensitive-pattern keys from `skills.entries.*.env`, and https://github.com/openclaw/openclaw/pull/50960 remains open with a closing reference to this issue.

## What This Changes

The issue reports that skill-scoped env vars configured under `skills.entries.*.env` should be inherited by subprocesses spawned by the `exec` tool.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50960, or an equivalent narrow fix, then close this issue after the fix is merged and verified.

## Reproduction Assessment

Yes. Source inspection at the review SHA shows `GOG_KEYRING_PASSWORD` is staged from `skills.entries.gog.env`, then blocked by the sensitive-pattern sanitizer because that config key is not added to `allowedSensitiveKeys`.

## Solution Assessment

Yes at the issue level: the narrow maintainable fix belongs in skill env override admission because host `exec` already inherits `process.env` after admission. The linked PR should still be reviewed for secret scoping and hard-block preservation before merge.
