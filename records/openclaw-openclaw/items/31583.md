## Summary

Keep open: current `main` and `v2026.5.22` still block the reported sensitive `skills.entries.*.env` path, while the open usability and security PRs need maintainer/security reconciliation before this can safely close.

## What This Changes

The issue reports that skill-scoped environment variables configured under `skills.entries.*.env` are not available to commands spawned through the agent `exec` tool.

## Best Possible Solution

Land or replace the linked fixes with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic exec strips unrelated active skill secrets and handles duplicate sensitive keys deliberately.

## Reproduction Assessment

Yes, source-backed. Configure `skills.entries.gog.env.GOG_KEYRING_PASSWORD`: the key matches the sensitive-name sanitizer, the bundled `gog` skill does not declare it in metadata, and current main blocks it before host `exec` can inherit it.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
