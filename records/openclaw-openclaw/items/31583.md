## Label Justifications

- `P1`: This is a user-facing regression for credential-backed skill CLI authentication on headless hosts, with multiple confirmations and security-sensitive credential handling.
- `impact:security`: The issue concerns whether password/token-shaped skill values are blocked from or leaked into host process and child-process environments.
- `impact:auth-provider`: The affected values are authentication credentials supplied through skill env, apiKey, and SecretRef-style configuration.

## Summary

Keep open: current main and the latest release still block credential-shaped keys from `skills.entries.<skill>.env` unless the skill metadata declares the same key, and the issue has a protected security/auth boundary plus an open linked fix candidate that needs maintainer judgment.

## What This Changes

The issue requests that per-skill configured env vars, including credential-shaped keys, be available to `exec` subprocesses for skill CLI authentication.

## Best Possible Solution

Land a maintainer-approved credential boundary where intended skill-owned CLI/tool invocations can receive configured skill env credentials while unrelated generic `exec` children do not inherit active skill secrets, with regression tests for both sides.

## Reproduction Assessment

Yes, source-backed. A `GOG_KEYRING_PASSWORD`-style key matches the sanitizer's credential-name block pattern, and current main only sensitive-allowlists metadata-declared env names before applying configured `skills.entries.<skill>.env`.

## Solution Assessment

Unclear as a final product choice. Adding configured env keys to `allowedSensitiveKeys` is the narrow availability fix, but the best complete fix must also define how generic exec avoids inheriting unrelated skill secrets.
