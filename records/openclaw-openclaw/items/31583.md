## Summary

Keep this open. Current `main` still blocks user-configured skill env keys such as `GOG_KEYRING_PASSWORD` when they match the shared sensitive-key pattern but are not declared by skill metadata, and open PR #50960 still references this issue with closing syntax.

## What This Changes

The issue reports that `skills.entries.<skill>.env` values are not inherited by `exec` subprocesses, breaking CLI-backed skills such as `gog` on headless hosts.

## Best Possible Solution

Land #50960 or an equivalent maintainer-reviewed fix that allows explicitly configured skill secrets only for the run's eligible skill set, keeps dangerous host env names blocked, and adds regression coverage for undeclared `_PASSWORD`/`_TOKEN` keys reaching host `exec`.

## Reproduction Assessment

Yes. Source inspection is enough: configure `GOG_KEYRING_PASSWORD` under `skills.entries.gog.env`, and current `main` copies it into pending overrides, then blocks it through the sensitive-key sanitizer before host `exec` can inherit it.

## Solution Assessment

Yes, the maintainable direction is to fix skill env override admission and scoping rather than generic `exec` inheritance. The open PR points at the right root cause, but it needs maintainer/security review for eligible-skill scoping and duplicate sensitive-key conflicts.
