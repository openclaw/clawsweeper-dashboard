## Summary

Keep this open: current main still blocks user-configured skill env keys that match sensitive patterns unless the key is declared in skill metadata, and open PR #50960 already references this issue with closing syntax.

## What This Changes

The issue asks for `skills.entries.<skill>.env` variables, including sensitive names like `GOG_KEYRING_PASSWORD`, to be available to host `exec` subprocesses during an agent run.

## Best Possible Solution

Land #50960 or an equivalent maintainer-reviewed change that admits explicitly configured skill env secrets for eligible host runs while preserving dangerous env blocks, restoring `process.env`, and adding focused regression coverage.

## Reproduction Assessment

Yes, by source inspection on current main. A key like `GOG_KEYRING_PASSWORD` is copied from `skills.entries.gog.env` into pending overrides, then blocked by the sensitive-pattern sanitizer because `gog` metadata does not declare it.

## Solution Assessment

Yes. The narrow fix belongs in skill env override admission/scoping, not generic `exec` inheritance, because host `exec` already starts from `process.env`.
