- `impact:auth-provider`: The failure prevents skill-specific authentication material such as keyring passwords or API tokens from reaching the intended CLI subprocess.

## Summary

Keep open. Current main and the latest release still block the reported sensitive per-skill env key before host `exec` can inherit it, and the linked fix path needs to be reconciled with the open generic-exec secret-leak work before this can close safely.

## What This Changes

The issue reports that `skills.entries.*.env` values such as `GOG_KEYRING_PASSWORD` are not inherited by subprocesses spawned through the `exec` tool.

## Best Possible Solution

Land or replace the linked fixes with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets and unsafe host-env overrides.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare that key as required metadata, and current main blocks it before host `exec` can inherit `process.env`.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
