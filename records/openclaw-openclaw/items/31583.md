- `impact:security`: The issue is about whether password/token-shaped skill values enter process and child-process environments.
- `impact:auth-provider`: The affected values are authentication credentials supplied through skill env, `apiKey`, and SecretRef-style configuration.

## Summary

Keep open: current main and the latest release still block credential-shaped `skills.entries.<skill>.env` keys unless skill metadata declares the same key, and the open fix path needs maintainer security/product judgment because it intersects the inverse secret-leak issue.

## What This Changes

The issue requests that `exec` subprocesses receive skill-scoped `skills.entries.*.env` variables such as `GOG_KEYRING_PASSWORD` for headless CLI authentication.

## Best Possible Solution

Land a scoped credential boundary where intended skill-owned host exec commands can receive configured skill env credentials, while unrelated generic exec children do not inherit active skill secrets.

## Reproduction Assessment

Yes, source-backed. Current main blocks `GOG_KEYRING_PASSWORD`-style configured keys at the skill env sanitizer unless metadata declares them, while allowed env values are forwarded into host exec child environments.

## Solution Assessment

Unclear as a final product choice. Allowlisting configured sensitive keys is the narrow availability fix, but the best solution must also preserve the generic exec secret-isolation boundary tracked by the related security issue.
