## Summary

This remains necessary: current main still blocks user-configured sensitive-name skill variables unless the skill metadata declares them, but forwarding them through generic exec would conflict with the shipped anti-leak boundary and the open credential-exposure work. The issue should stay open while maintainers define one skill-owned execution contract that fixes authentication without exposing credentials to unrelated commands.

## What This Changes

Ensure skill-scoped authentication environment variables are available when the corresponding skill invokes CLI commands through exec.

## Best Possible Solution

Bind only the selected skill’s declared credentials to a skill-owned command invocation, while generic exec and unrelated harnesses exclude active skill secrets and dangerous host variables remain unconditionally blocked.

## Reproduction Assessment

Yes, at source level with high confidence: current main only re-allows sensitive variables declared by skill metadata, matching the reported blocked GOG_KEYRING_PASSWORD path, and multiple users supplied consistent release-specific runtime confirmations. This read-only review did not execute a live secret-bearing CLI command.

## Solution Assessment

No, blanket inheritance into generic exec is not the best solution because it conflicts with the established child-process anti-leak invariant and the open generic-exec credential-exposure report. The narrow maintainable solution is a skill-owned invocation context carrying only that skill’s credentials.
