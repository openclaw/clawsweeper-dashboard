## Label Justifications

- `P1`: This source-backed regression blocks skill CLI authentication for real users and remains present in the latest release.
- `impact:security`: The issue concerns which configured skill secrets are blocked from or exposed to child process environments.
- `impact:auth-provider`: The failure prevents skill-specific authentication material such as keyring passwords or API tokens from reaching the intended CLI subprocess.

## Summary

Keep open: current main and v2026.5.20 still block the reported `GOG_KEYRING_PASSWORD` path, and the live linked fix needs to be reconciled with the open generic exec secret-leak work before this can close safely.

## What This Changes

The issue asks OpenClaw to make `skills.entries.<skill>.env` variables available to `exec` subprocesses for skill-owned CLI authentication.

## Best Possible Solution

Land or replace the linked fixes with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets, rejects unsafe host-env overrides, and handles duplicate sensitive keys deliberately.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare that key as required metadata, and current main blocks it before host `exec` can inherit `process.env`.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
