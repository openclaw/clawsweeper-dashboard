## Summary

Keep this open: current main still admits only metadata-declared sensitive env keys, so a user-configured key like `GOG_KEYRING_PASSWORD` is copied into pending overrides and then blocked before `exec` can inherit it. Open PR #50960 already targets this issue with closing syntax, so the issue should stay open until that fix or an equivalent replacement lands.

## What This Changes

The issue asks for `exec` subprocesses to inherit env vars configured under `skills.entries.<skill>.env` during eligible skill runs.

## Best Possible Solution

Land #50960 or an equivalent maintainer-reviewed change that admits explicitly configured skill env secrets while preserving dangerous env blocks and adding focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. Configure `GOG_KEYRING_PASSWORD` under `skills.entries.gog.env`: current main copies it into pending overrides, blocks it via the sensitive-key pattern, and never writes it to `process.env` for host `exec` to inherit.

## Solution Assessment

Yes, the narrow fix belongs in skill env override admission/scoping rather than generic `exec` inheritance, because host `exec` already starts from the run process environment.
