## Summary

Keep this open. Current main still drops user-configured skill env keys that match the shared sensitive-key pattern unless the key is declared in skill metadata, and open PR #50960 still targets this issue with closing syntax.

## What This Changes

Allow `exec` subprocesses to inherit environment variables configured under `skills.entries.<skill>.env`, including sensitive-looking keys such as `GOG_KEYRING_PASSWORD`.

## Best Possible Solution

Land #50960 or an equivalent maintainer-reviewed fix that admits explicitly configured skill secrets for eligible skills while preserving hard blocks for dangerous host/runtime env names and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `GOG_KEYRING_PASSWORD` under `skills.entries.gog.env`, and current main copies it into pending overrides, then drops it because it matches the sensitive-key pattern and is not metadata-declared.

## Solution Assessment

Yes. The narrow maintainable fix is in skill env override admission/scoping, not in generic `exec` inheritance, because host `exec` already builds from the run process environment.
