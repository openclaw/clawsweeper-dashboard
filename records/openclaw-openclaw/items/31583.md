## Summary

Keep this open. Current main still drops user-configured skill env keys whose names match the shared sensitive-key pattern unless the skill metadata also declares those keys, and the bundled `gog` skill does not declare `GOG_KEYRING_PASSWORD`. The issue is also paired with open PR #50960 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that per-skill environment variables configured through `skills.entries.<skill>.env` should reach host `exec` subprocesses, including sensitive-looking keys such as `GOG_KEYRING_PASSWORD`.

## Best Possible Solution

Land #50960 or an equivalent maintainer-reviewed fix that adds explicitly configured `skills.entries.<skill>.env` keys to the skill-env sensitive-key allowlist before sanitization, while keeping the always-blocked host/runtime env policy intact and adding regression coverage for undeclared configured `_PASSWORD` and `_TOKEN` keys reaching host `exec`.
