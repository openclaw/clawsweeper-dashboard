## Label Justifications

- `P1`: This source-backed regression blocks skill CLI authentication for real users and remains present in the latest release.
- `impact:security`: The issue concerns which configured skill secrets are blocked from or exposed to child process environments.
- `impact:auth-provider`: The failure prevents skill-specific authentication material such as keyring passwords or API tokens from reaching the intended CLI subprocess.

## Summary

Keep open. Current main and the latest release still block the reported sensitive per-skill env key before host `exec` can inherit it, while the open closing PR needs reconciliation with the related generic-exec secret-leak issue and fix.

## What This Changes

The issue asks for `exec` subprocesses to receive intended `skills.entries.<skill>.env` values, including secret-like keys needed by skill CLIs such as `gog`.

## Best Possible Solution

Land or replace the linked fixes with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets and unsafe host-env overrides.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare that key as required metadata, and current main blocks it before host `exec` can inherit `process.env`; no live `gog` auth run was performed.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
