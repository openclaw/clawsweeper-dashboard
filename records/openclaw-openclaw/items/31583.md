## Summary

Keep this open: current `main` still blocks sensitive-pattern keys from `skills.entries.*.env`, and https://github.com/openclaw/openclaw/pull/50960 is still open with closing syntax for this issue. The remaining fix is real, but it needs to preserve hard env blocks and reconcile the related skill-secret scoping concern in https://github.com/openclaw/openclaw/issues/78528.

## What This Changes

Allow environment variables configured under `skills.entries.<skill>.env`, including sensitive-pattern keys such as `GOG_KEYRING_PASSWORD`, to reach `exec` subprocesses during skill use.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50960 or an equivalent narrow fix that admits explicitly configured skill env secrets only through the intended scoped path while preserving hard blocks and conflict handling.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` is staged from `skills.entries.gog.env`, then blocked by the sensitive-pattern sanitizer because configured env keys are not added to `allowedSensitiveKeys`; I did not run a live `gog` command in this read-only review.

## Solution Assessment

Unclear as a landed solution because the linked PR still needs security-sensitive scoping review. The narrow ownership boundary is correct, but the final fix should preserve dangerous-key blocks and avoid broad process-wide leakage of unrelated skill secrets.
