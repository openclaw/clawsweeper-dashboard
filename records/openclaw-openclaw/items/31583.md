## Summary

Keep open. Current main still source-reproduces the reported env inheritance failure for sensitive configured keys, and the issue is paired with an open fix PR plus a related security follow-up that needs maintainer/security review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make `skills.entries.<skill>.env` values available to subprocesses launched through the agent `exec` tool.

## Best Possible Solution

Land or replace the linked fix with a scoped env boundary that lets intended skill-owned commands receive explicit configured env values while generic exec strips active skill secrets and rejects unsafe conflicts.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare it as a required env key, and current main blocks it before host exec can inherit `process.env`.

## Solution Assessment

No, not as a broad inherit-all fix. The maintainable path needs scoped admission for intended skill-owned execution plus preservation of generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
