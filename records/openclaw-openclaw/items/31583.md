## Summary

Keep open: current main and the latest shipped release still block the reported `GOG_KEYRING_PASSWORD` path, and the linked usability/security PRs have not settled a safe env boundary yet.

## What This Changes

The issue reports that per-skill environment variables configured under `skills.entries.*.env` do not reach `exec` tool subprocesses, blocking skill CLI authentication such as GOG keyring unlocks.

## Best Possible Solution

Land or replace the linked fixes with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic `exec` strips unrelated active skill secrets, rejects unsafe host-env overrides, and handles duplicate sensitive keys deliberately.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare that key as required metadata, and current main blocks it before host `exec` can inherit it from `process.env`.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
