## Summary

Keep this open: current main still blocks sensitive-pattern keys from `skills.entries.*.env`, and an open implementation PR already references this issue with closing syntax.

## What This Changes

The issue requests that `skills.entries.<skill>.env` values, including secret-looking keys, be available to `exec` subprocesses during skill-invoked runs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50960 or an equivalent narrow change that admits explicitly configured skill env secrets for eligible host runs while preserving hard blocks and restoration coverage.

## Reproduction Assessment

Yes. By source inspection, `GOG_KEYRING_PASSWORD` is staged from `skills.entries.gog.env`, then blocked by the sensitive-pattern sanitizer because configured env keys are not added to `allowedSensitiveKeys`.

## Solution Assessment

Yes. The narrow maintainable fix is in skill env override admission and scoping, because host `exec` already inherits `process.env` after those overrides are applied.
