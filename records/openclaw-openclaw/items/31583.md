## Label Justifications

- `P1`: This is a high-impact regression for skill CLI authentication on headless hosts and is entangled with a credential-boundary security issue.
- `impact:security`: The requested behavior controls whether password/token-shaped values are injected into process and child-process environments.
- `impact:auth-provider`: The affected values are authentication credentials used by skills and SecretRef/apiKey-style configuration.

## Summary

Keep open: current main and the latest release still block credential-shaped keys from `skills.entries.<skill>.env` unless the skill metadata declares that exact key, and the remaining fix is tied to the inverse secret-leak boundary tracked in a separate open security issue.

## What This Changes

The issue requests that `exec` subprocesses inherit environment variables configured under `skills.entries.<skill>.env`, including credential-shaped keys such as `GOG_KEYRING_PASSWORD`, so skill CLI authentication works on headless hosts.

## Best Possible Solution

Keep this open until maintainers land a scoped credential boundary where intended skill-owned host exec commands can receive configured skill env credentials, while unrelated generic exec children do not inherit skill secrets.

## Reproduction Assessment

Yes. Source inspection on current main shows `GOG_KEYRING_PASSWORD`-style keys match the sanitizer's sensitive patterns and are blocked unless the skill metadata declares the same key, while ordinary allowed env values are passed into host exec child environments.

## Solution Assessment

Unclear as a final product choice. Adding configured env keys to the sensitive allowlist is a narrow availability fix, but maintainers must reconcile it with the open secret-leak issue so unrelated generic exec children do not receive skill-scoped credentials.
