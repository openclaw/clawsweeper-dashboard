## Label Justifications

- `P1`: This source-backed regression blocks skill CLI authentication for real users and intersects credential exposure rules.
- `impact:security`: The issue is about which configured skill secrets are blocked from or exposed to subprocess environments.
- `impact:auth-provider`: The failure prevents skill-specific authentication material such as keyring passwords or API tokens from reaching intended CLI subprocesses.

## Summary

Keep open: current main and the latest shipped release still block the reported `GOG_KEYRING_PASSWORD` path, and the open linked fixes have not yet settled the safe skill-env versus generic-exec secret boundary.

## What This Changes

The issue asks for `exec` subprocesses to receive environment variables configured under `skills.entries.<skill>.env` while preserving skill-secret isolation.

## Best Possible Solution

Land or replace the linked fixes with a scoped env boundary that lets intended skill-owned commands receive explicit configured env values while generic `exec` strips unrelated active skill secrets and duplicate sensitive keys are handled deliberately.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare that env key, and current main blocks it before host `exec` can inherit it from `process.env`.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable fix needs scoped admission for intended skill-owned execution while preserving generic `exec` secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
