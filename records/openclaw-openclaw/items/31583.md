## Summary

Keep open. Current main and the latest release still block the reported sensitive configured skill env key before host `exec` can inherit it, while the issue is paired with an open closing PR and a related security follow-up that need maintainer/security review.

## What This Changes

The issue requests that `skills.entries.<skill>.env` values from `openclaw.json` be available to subprocesses spawned by the `exec` tool for the active skill.

## Best Possible Solution

Land or replace the linked fix with a scoped env boundary that lets intended skill-owned subprocesses receive explicit configured env values while generic exec keeps stripping unrelated active skill secrets and unsafe host-env overrides.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare it as required env metadata, and current main blocks the configured override before host exec can inherit `process.env`; no live `gog` auth run was performed.

## Solution Assessment

No, not as a broad inherit-all change. The maintainable solution needs scoped admission for intended skill-owned execution while preserving generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
