## Summary

Keep open: current main and the latest release still block credential-shaped `skills.entries.<skill>.env` keys like `GOG_KEYRING_PASSWORD` unless the skill metadata declares the same key, and the remaining fix needs maintainer security/product judgment because it overlaps the inverse secret-leak issue.

## What This Changes

The issue requests that per-skill `skills.entries.<skill>.env` variables be available to commands spawned through the exec tool, including credential-shaped CLI auth keys.

## Best Possible Solution

Land a maintainer-approved scoped fix that lets user-configured sensitive skill env keys reach their intended skill-owned host exec flow while preserving hard blocks for dangerous host env names and preventing unrelated generic exec secret leakage.

## Reproduction Assessment

Yes. Source review on current main shows `GOG_KEYRING_PASSWORD` matches the credential-pattern block, the gog skill does not declare it in metadata, and the latest release has the same code path; I did not run the live Ubuntu keyring flow in this read-only review.

## Solution Assessment

No, current main is not the best end state because it preserves a safe default by blocking undeclared credential-shaped keys but breaks documented per-skill env delivery. The best fix needs a maintainer-approved boundary that handles intended skill credential delivery without reopening unrelated exec child-env leaks.
