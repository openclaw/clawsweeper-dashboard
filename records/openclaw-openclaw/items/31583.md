## Summary

Keep open. Current main still blocks the reported sensitive configured key before host exec can inherit it, and the issue is paired with an open fix PR plus a newer security follow-up that needs maintainer/security review.

## What This Changes

The issue requests that `skills.entries.<skill>.env` values, including sensitive keys such as `GOG_KEYRING_PASSWORD`, reach subprocesses spawned by the `exec` tool.

## Best Possible Solution

Land or replace the linked fix with a scoped env boundary that lets selected skill-owned commands receive explicit configured env values while generic exec strips active skill secrets and rejects unsafe conflicts.

## Reproduction Assessment

Yes, source-backed. `GOG_KEYRING_PASSWORD` matches the sensitive-name block, the bundled `gog` skill does not declare it as a required env key, and current main blocks it before host exec can inherit `process.env`; no live `gog` auth run was performed.

## Solution Assessment

No, not as a broad inherit-all fix. The maintainable solution needs scoped admission for intended skill-owned execution plus preservation of generic exec secret stripping, especially alongside https://github.com/openclaw/openclaw/issues/78528 and https://github.com/openclaw/openclaw/pull/82512.
