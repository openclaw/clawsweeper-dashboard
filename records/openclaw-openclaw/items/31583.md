## Summary

Keep this open. Current main still drops user-configured skill env keys whose names match the shared sensitive-key pattern unless the skill metadata also declares those keys, and the bundled `gog` skill does not declare `GOG_KEYRING_PASSWORD`. The issue is paired with open PR #50960 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that per-skill environment variables configured through `skills.entries.<skill>.env` reach host `exec` subprocesses, including sensitive-looking keys such as `GOG_KEYRING_PASSWORD`.

## Best Possible Solution

Land #50960 or an equivalent maintainer-reviewed fix that admits explicitly configured `skills.entries.<skill>.env` sensitive-pattern keys only for the active run's eligible skill set, preserves hard blocks for dangerous host/runtime variables, defines duplicate-key conflict behavior, and adds regression coverage for undeclared configured `_PASSWORD` and `_TOKEN` keys reaching host `exec`.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: configure a skill env key such as `GOG_KEYRING_PASSWORD` that is not declared in skill metadata, and the shared sanitizer blocks the `_PASSWORD` pattern because configured env keys are not added to `allowedSensitiveKeys`. No live gog run was needed for this review because the drop occurs deterministically before host exec builds its child environment.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the skill env override sanitizer path rather than changing generic exec inheritance. The linked PR points at the right root cause, but it should be reviewed for active-skill scoping, duplicate sensitive-key conflicts, and the existing dangerous-env policy before merge.

## Risks / Open Questions

- The fix is security-sensitive: explicitly configured skill env keys should be allowed without permitting dangerous host/runtime env overrides such as `BASH_ENV`, `SHELL`, `NODE_OPTIONS`, loader vars, proxy/TLS controls, or sandbox-control variables.
- Allowing every configured sensitive key before active-skill scoping could expose secrets from disabled, allowlist-blocked, or otherwise unselected skills to host exec subprocesses.
- Duplicate sensitive env keys across multiple skills need a defined conflict policy so one skill does not silently inherit another skill's credential.
- Docs distinguish host runs from sandbox runs; a narrow fix should not silently forward all host skill secrets into sandbox containers.
