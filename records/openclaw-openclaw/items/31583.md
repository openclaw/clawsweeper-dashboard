## Label Justifications

- `P1`: This is a user-facing regression for skill CLI authentication on headless hosts and is tied to a credential-boundary decision.
- `impact:security`: The issue is about whether password/token-shaped skill values enter process and child-process environments.
- `impact:auth-provider`: The affected values are authentication credentials supplied through skill env, apiKey, and SecretRef-style configuration.

## Summary

Keep open: current main and the latest release still block credential-shaped `skills.entries.<skill>.env` keys such as `GOG_KEYRING_PASSWORD` unless the skill metadata declares that key, and the linked fix path needs maintainer security/product review because it overlaps the inverse secret-leak issue.

## What This Changes

The issue requests that `skills.entries.<skill>.env` values, including credential-shaped keys such as `GOG_KEYRING_PASSWORD`, be available to host `exec` subprocesses for skill CLI authentication.

## Best Possible Solution

Land or replace the linked fix with a scoped credential boundary where intended skill-owned host exec commands can receive configured skill env credentials, while unrelated generic exec children do not inherit active skill secrets.

## Reproduction Assessment

Yes, source-backed. Current main blocks `GOG_KEYRING_PASSWORD`-style configured keys at the skill env sanitizer unless metadata declares them, while allowed env values are forwarded into host exec child environments.

## Solution Assessment

Unclear as a final product choice. Allowlisting configured sensitive keys is the narrow availability fix, but the best solution must also preserve the generic exec secret-isolation boundary tracked by https://github.com/openclaw/openclaw/issues/78528.
