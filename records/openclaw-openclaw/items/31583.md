## What This Changes

The issue requests that `skills.entries.<skill>.env` values be available to `exec` subprocesses so CLI-backed skills can authenticate on headless hosts.

## Best Possible Solution

Review and land a scoped version of the open skill-env fix that lets intended skill-owned CLI commands receive configured env values while preventing unrelated generic exec children from inheriting skill secrets.

## Reproduction Assessment

Yes, source-backed. Current main stages `GOG_KEYRING_PASSWORD` from `skills.entries.*.env`, then blocks it through the sensitive-name sanitizer because configured env keys are not metadata-allowlisted.

## Solution Assessment

Unclear as a final implementation. Allowlisting explicit configured keys is directionally necessary, but the fix must be scoped with the related generic exec secret-stripping work before it is the best maintainable solution.
