## Summary

Keep open: current main and the latest release still block credential-shaped keys from `skills.entries.<skill>.env` unless the skill metadata declares the same key, and the remaining fix overlaps with an open inverse secret-leak boundary that needs maintainer security/product judgment.

## What This Changes

The issue reports that per-skill environment variables from `skills.entries.<skill>.env` do not reach subprocesses spawned through the exec tool.

## Best Possible Solution

Make one maintainer-owned decision for the per-skill secret boundary, then either land the linked fix candidate with regression coverage or update the skill metadata/docs to match the intended allowlist model.

## Reproduction Assessment

Yes from source: configure `skills.entries.gog.env.GOG_KEYRING_PASSWORD`, use the bundled gog skill, and the current sanitizer blocks that credential-shaped key before `exec` builds its subprocess environment. I did not run the real gog flow because this is read-only triage and requires live credentials.

## Solution Assessment

Unclear as a product/security decision: allowing user-configured keys through matches the docs and reported need, but maintainers need to confirm that this is the desired secret boundary rather than requiring skill metadata declarations.
