## Summary

Keep open: this is a focused security hardening PR that moves Runway create and poll control JSON reads onto the existing bounded provider JSON reader, has convincing terminal proof, and has no line-level correctness findings. The remaining maintainer decision is whether to accept the intentional shared 16 MiB fail-closed cap for Runway control responses.

## What This Changes

The branch routes Runway create-submit and task-poll JSON reads through `readProviderJsonResponse` and updates Runway tests plus the shared provider HTTP test mock to use real `Response` bodies for that reader.

## Best Possible Solution

Land the narrow shared-reader fix after maintainer acceptance of the 16 MiB Runway control-JSON cap; keep the broader conflicting batch out of the Runway canonical path.

## Reproduction Assessment

Yes, at source and proof level. Current main calls `response.json()` for Runway create and poll success bodies, and the PR body shows a real loopback stream proof where the new reader cancels near 16 MiB while the old path buffers the whole oversized body.

## Solution Assessment

Yes. Reusing `readProviderJsonResponse` is the best local fix because it uses the established provider JSON cap and parse-error behavior instead of adding another Runway-specific bounded parser.
