## Summary

Keep open; the PR targets a real source-reproducible image-loss path, but it should not merge as-is because it lacks real behavior proof and the patch likely still misses sandboxed media URI hydration while duplicating the shared media URI contract.

## What This Changes

The PR resolves `media://inbound/...` image references through the media store before workspace or sandbox path handling in `loadImageFromRef` and adds two regression tests.

## Best Possible Solution

Fix `loadImageFromRef` so managed media-store URIs use the shared media-reference/loadWebMedia contract without workspace path mangling, cover both sandbox and non-sandbox cases, then verify a real Telegram/Gateway image reaches a vision-capable agent.

## Reproduction Assessment

Yes by source inspection: current main detects `media-uri` refs, but `loadImageFromRef` treats `media://inbound/...` as a non-absolute path and resolves it under the workspace before `loadWebMedia` can apply its media-store resolver. I did not run a live Telegram/Gateway reproduction in this read-only review.

## Solution Assessment

No: the proposed direction fixes the right area, but the implementation should reuse the shared managed-media resolver/read path and prove sandbox behavior instead of adding a parallel URI parser plus direct store lookup.
