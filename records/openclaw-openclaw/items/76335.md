## Summary

Keep open. Current main still resolves Telegram account tokens before disabled filtering in both account discovery and Gateway runtime snapshots; non-env SecretRefs can throw from that path, and the five-minute health monitor calls the snapshot path without catching the rejection.

## What This Changes

The issue reports Gateway unhandled rejections when disabled Telegram accounts use exec SecretRef bot tokens and are resolved during snapshot/account discovery.

## Best Possible Solution

Keep the repair Telegram-owned: filter or inspect disabled accounts before strict token resolution during snapshot/action discovery, add regression coverage for disabled exec/file SecretRef bot tokens, and leave shared SecretRef semantics unchanged unless a minimal generic seam is proven necessary.

## Reproduction Assessment

Yes. Current main source inspection gives a high-confidence path: a disabled Telegram account with an exec/file botToken reaches resolveTelegramToken through getRuntimeSnapshot or listEnabledTelegramAccounts before enabled filtering, then strict non-env SecretRef resolution throws.

## Solution Assessment

Yes. The narrowest maintainable fix is to keep discovery/status paths on raw enabled checks or non-throwing Telegram inspection while preserving strict token resolution for active startup/send paths.
