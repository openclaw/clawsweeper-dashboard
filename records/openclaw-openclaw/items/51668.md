## Summary

Keep this PR open, but it is not mergeable as submitted: the security boundary is still bypassable after hooks and through structured payloads, the advertised config option is not wired into the schema, and the external contributor has not provided real outbound behavior proof.

## What This Changes

The PR adds a gateway-layer outbound secret sanitizer with named reference handles and entropy fallback for the delivery-layer request tracked at https://github.com/openclaw/openclaw/issues/50718.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks and rendering, recursively covers user-visible payload strings, derives handles from canonical sensitive or SecretRef metadata, and updates schema, docs, changelog, regression tests, and live channel proof together.

## Reproduction Assessment

Yes for the review findings via source inspection: the PR sanitizes only early payload.text, while current main can rewrite content afterward through message_sending and can deliver structured strings through sendPayload. I did not run a live channel delivery path in this read-only review.

## Solution Assessment

No. A gateway-layer sanitizer is the right direction, but this implementation should move to the final-send boundary, cover structured payloads, use canonical sensitive metadata, and include the advertised config contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Re-sanitize after message_sending hooks:** `src/infra/outbound/deliver.ts:324-326`
  - body: The sanitizer runs during early payload normalization, but message_sending hooks run later and can replace outbound text. Hook-generated content containing a secret would reach the final send without another redaction pass.
  - confidence: 0.93
- **[P2] Sanitize structured payload strings too:** `src/infra/outbound/deliver.ts:324-326`
  - body: This only rewrites payload.text, while presentation, interactive, and channelData can still be delivered through sendPayload. Secrets in labels, placeholders, values, or presentation text would bypass the sanitizer.
  - confidence: 0.9
- **[P1] Use canonical sensitive metadata for handles:** `src/infra/outbound/sanitize-secrets.ts:82-108`
  - body: buildCredentialIndex walks only cfg.auth and top-level channel strings, but configured secrets also live under providers, tools, gateway auth, cron, nested channel accounts, and SecretRef-backed surfaces. Low-entropy values there can miss both named handles and entropy redaction.
  - confidence: 0.88
- **[P1] Split quoted structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:150-154`
  - body: The tokenizer keeps quotes, braces, and colons attached to candidate values. Common JSON/YAML like {"token":"ghp_..."} or password: "..." can miss exact-value and prefix matching.
  - confidence: 0.86
- **[P2] Declare messages.sanitizeSecrets in schema:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: isSecretSanitizationEnabled reads messages.sanitizeSecrets, but the strict MessagesSchema and MessagesConfig omit that field. The advertised opt-out would be rejected by config validation.
  - confidence: 0.91

## Security Review
