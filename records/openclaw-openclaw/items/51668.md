## Summary

Keep open: current main still lacks the outbound named-handle sanitizer and https://github.com/openclaw/openclaw/issues/50718 was closed as superseded by this PR, but this branch should not merge as-is because the sanitizer runs too early, misses structured/config surfaces, lacks the advertised config contract, is conflicting, and has only test proof.

## What This Changes

The PR adds a default-on outbound secret sanitization pass in `src/infra/outbound` that maps detected `payload.text` secrets to `key(...)` or `[REDACTED:high-entropy]`, plus focused unit tests.

## Best Possible Solution

Land a redesigned final-send sanitizer that runs after hooks, rendering, and adapter normalization, recursively covers outbound string-bearing payload fields, uses canonical sensitive/SecretRef metadata, declares the config contract, and includes upgrade plus live channel proof.

## Reproduction Assessment

Yes, at source level. Current main normalizes payloads before `message_sending` hooks, rendering, and final sends, while the PR inserts the sanitizer only in the early normalization step and only for `payload.text`.

## Solution Assessment

No. Gateway-layer sanitization is the right direction, but this implementation needs to move to the final-send boundary, cover structured fields, use canonical sensitive metadata, and expose the advertised config option before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Run redaction at the final send boundary:** `src/infra/outbound/deliver.ts:324-329`
  - body: This sanitizer runs during early payload normalization, but `message_sending` hooks and presentation rendering can replace outbound content later. A hook-generated or renderer-generated secret would bypass the only sanitizer pass and then reach final channel delivery.
  - confidence: 0.93
- **[P1] Sanitize structured payload strings before sendPayload:** `src/infra/outbound/deliver.ts:324-329`
  - body: The transform only rewrites `payload.text`, but the delivery path can send presentation, interactive, and `channelData` through `sendPayload`. Secrets in labels, placeholders, values, card text, URLs, or channel-specific fields remain unsanitized.
  - confidence: 0.9
- **[P1] Build handles from canonical sensitive config:** `src/infra/outbound/sanitize-secrets.ts:81-108`
  - body: `buildCredentialIndex` walks only `cfg.auth` and top-level channel string values. Current schemas mark many provider, tool, gateway, nested channel, and SecretRef fields as sensitive, so low-entropy configured secrets there can still be delivered verbatim.
  - confidence: 0.88
- **[P1] Declare the sanitizeSecrets config option:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: `isSecretSanitizationEnabled` reads `messages.sanitizeSecrets`, but `MessagesSchema` is strict and `MessagesConfig` omits that field. Users cannot rely on the advertised opt-out through validated config, which is merge-critical for a default-on content rewrite.
  - confidence: 0.91
- **[P2] Split quoted JSON and YAML values before matching:** `src/infra/outbound/sanitize-secrets.ts:148-151`
  - body: The tokenizer keeps quotes, braces, and colons attached to candidate values. JSON or YAML such as `{"token":"ghp_..."}` can miss exact-value and known-prefix matching because the token no longer starts with the secret prefix.
  - confidence: 0.86
