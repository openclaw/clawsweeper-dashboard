## Summary

Keep this PR open, but it is not ready to merge: the proposed gateway sanitizer is useful security work, yet source inspection confirms bypasses after hooks, through structured payloads, and through incomplete credential/config coverage.

## What This Changes

The PR adds `src/infra/outbound/sanitize-secrets.ts`, tests, and delivery wiring to replace outbound `payload.text` secrets with named `key(...)` handles or `[REDACTED:high-entropy]`, plus a `messages.sanitizeSecrets` opt-out helper.

## Best Possible Solution

Land a revised final-send sanitizer that runs at the effective outbound boundary, recursively covers user-visible strings, uses canonical sensitive/SecretRef metadata for handles, and updates schema, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: a `message_sending` hook can introduce a secret after the sanitizer pass, structured payload strings can reach `sendPayload`, and `messages.sanitizeSecrets` is absent from the strict config schema.

## Solution Assessment

No. A gateway-level outbound sanitizer is the right direction, but this implementation runs too early, covers only `payload.text`, and builds handles from an incomplete ad hoc index rather than canonical sensitive metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Re-sanitize after message_sending hooks:** `src/infra/outbound/deliver.ts:624-625`
  - body: The sanitizer runs during payload normalization, but `message_sending` hooks run later and can replace the payload before `sendPayload`, `sendFormattedText`, or chunked `sendText`. A hook-supplied secret therefore bypasses the only sanitizer pass; redaction needs to happen at the final effective-send boundary too.
  - confidence: 0.93
- **[P2] Sanitize structured payload strings too:** `src/infra/outbound/deliver.ts:324-326`
  - body: This only calls `sanitizeSecrets` for `sanitizedPayload.text`, leaving `presentation`, `interactive`, and `channelData` strings untouched even though those fields can be delivered through `sendPayload`. Secrets in button labels, option values, placeholders, presentation blocks, or channel-specific payloads would still be sent verbatim.
  - confidence: 0.9
- **[P1] Build handles from canonical secret metadata:** `src/infra/outbound/sanitize-secrets.ts:82-108`
  - body: `buildCredentialIndex` only walks `cfg.auth` and top-level `cfg.channels` strings, but many supported plaintext credentials live under sensitive/SecretRef surfaces such as provider keys, hooks, tools, gateway auth, and nested channel account fields. Low-entropy configured secrets on those paths can miss both named handles and entropy redaction.
  - confidence: 0.88
- **[P1] Tokenize quoted and structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:151-154`
  - body: The scanner splits on whitespace, commas, semicolons, pipes, and `=`, but keeps quotes, braces, and colons attached. Common JSON/YAML/code output like `{"token":"ghp_..."}` or `password: "shared-password-0000"` will not exact-match configured credentials or known prefixes, so those values can bypass redaction.
  - confidence: 0.86
- **[P2] Declare messages.sanitizeSecrets in the config schema:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: The helper reads `messages.sanitizeSecrets`, but `MessagesSchema` is strict and `MessagesConfig` does not declare that field. Users cannot set the advertised opt-out in `openclaw.json` without schema validation rejecting it, so the schema, type, docs, and generated help need to be updated together.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The patch changes the outbound delivery security boundary while leaving concrete bypasses through hooks, structured payloads, tokenization, and incomplete credential indexing.

Concerns:

- **[high] Hook rewrites bypass redaction:** `src/infra/outbound/deliver.ts:624`
  - body: Because redaction is applied before `message_sending` hooks, hook-supplied replacement content can introduce a secret after the only sanitizer pass and still reach channel delivery.
  - confidence: 0.93
- **[medium] Structured outbound payloads bypass redaction:** `src/infra/outbound/deliver.ts:324`
  - body: The sanitizer only rewrites `payload.text`, while structured payload fields can carry user-visible strings through `sendPayload`; this leaves a practical exfiltration path for buttons, presentations, and channel-specific payloads.
  - confidence: 0.9
- **[medium] Incomplete credential index misses configured secrets:** `src/infra/outbound/sanitize-secrets.ts:82`
  - body: The ad hoc index omits many registered sensitive and SecretRef config surfaces, so known low-entropy secrets outside top-level auth/channel strings may not receive named handles and may not be caught by entropy rules.
  - confidence: 0.88
- **[medium] Quoted and structured tokens can bypass matching:** `src/infra/outbound/sanitize-secrets.ts:151`
  - body: Keeping JSON/YAML punctuation attached to candidate values prevents known-prefix and exact-value matching for common agent outputs that quote or structure secrets.
  - confidence: 0.86
