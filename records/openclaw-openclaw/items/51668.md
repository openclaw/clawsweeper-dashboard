## Summary

Keep open: the PR addresses a real security boundary, but the current patch still has concrete bypasses, lacks the advertised config contract, has no real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds a gateway outbound secret sanitizer, wires it into delivery normalization, advertises a `messages.sanitizeSecrets` opt-out, and adds focused unit tests.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks/rendering, recursively sanitizes outbound payload strings, indexes canonical sensitive or SecretRef-backed values, declares the config contract/defaults, and includes live channel proof.

## Reproduction Assessment

Yes, at source level: the PR sanitizes during early normalization, while current main can rewrite text afterward via `message_sending` hooks and can deliver structured payload strings through `sendPayload`. I did not run a live channel delivery path in this read-only review.

## Solution Assessment

No. Gateway-layer sanitization is the right direction, but this implementation needs to move to the final-send boundary, cover structured fields, use canonical sensitive metadata, and expose the advertised config option before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Re-sanitize at the final send boundary:** `src/infra/outbound/deliver.ts:331-335`
  - body: The new redaction pass runs during early normalization, but `message_sending` hooks run afterward and can replace outbound text. Hook-generated content containing a secret would reach final channel delivery without another sanitizer pass.
  - confidence: 0.93
- **[P2] Sanitize structured payload strings too:** `src/infra/outbound/deliver.ts:331-335`
  - body: This only rewrites `payload.text`, while payloads with `interactive`, `presentation`, or `channelData` still go through `sendPayload`. Secrets in labels, placeholders, values, card text, or channel-specific fields would bypass redaction.
  - confidence: 0.9
- **[P1] Use canonical sensitive metadata for handles:** `src/infra/outbound/sanitize-secrets.ts:81-108`
  - body: `buildCredentialIndex` only walks `cfg.auth` and top-level channel string values. Current schemas mark secrets under providers, tools, gateway auth, cron, nested channel accounts, and SecretRef-backed fields, so configured low-entropy secrets there may still be delivered verbatim.
  - confidence: 0.88
- **[P1] Split quoted structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:150`
  - body: The tokenizer keeps quotes, braces, and colons attached to candidate values. JSON or YAML such as `{"token":"ghp_..."}` can miss exact-value and known-prefix matching because the token no longer starts with the secret prefix.
  - confidence: 0.86
- **[P2] Declare messages.sanitizeSecrets in the config contract:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: `isSecretSanitizationEnabled` reads `messages.sanitizeSecrets`, but the strict messages schema and exported `MessagesConfig` type do not declare that field. Users would not have the advertised opt-out through validated config.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: Needs attention: the patch is security hardening, but its current placement and coverage leave concrete secret-redaction bypasses.

Concerns:

- **[high] Hook rewrites bypass redaction:** `src/infra/outbound/deliver.ts:331`
  - body: Because the sanitizer runs before `message_sending` hooks, hook-supplied replacement content can introduce a secret after the only sanitizer pass and still reach channel delivery.
  - confidence: 0.93
- **[medium] Structured outbound payloads bypass redaction:** `src/infra/outbound/deliver.ts:331`
  - body: The sanitizer only rewrites `payload.text`, while structured payload fields can carry user-visible strings through `sendPayload`.
  - confidence: 0.9
- **[medium] Configured secrets are incompletely indexed:** `src/infra/outbound/sanitize-secrets.ts:81`
  - body: The ad hoc credential index omits many registered sensitive and SecretRef config surfaces, so known low-entropy secrets outside top-level auth/channel strings may not be redacted or mapped to handles.
  - confidence: 0.88
- **[medium] Quoted structured tokens can bypass matching:** `src/infra/outbound/sanitize-secrets.ts:150`
  - body: Keeping JSON/YAML punctuation attached to candidate values prevents known-prefix and exact-value matching for common agent outputs that quote or structure secrets.
  - confidence: 0.86
