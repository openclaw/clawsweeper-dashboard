## Summary

Keep open. Current main does not implement this sanitizer, and the PR is useful source work, but it is not merge-ready because the current diff leaves confirmed outbound secret-bypass paths in hook rewrites, structured payloads, quoted/structured tokens, and canonical SecretRef-backed credential surfaces.

## What This Changes

The PR adds a gateway outbound secret sanitizer and tests, then wires delivery normalization to replace detected `payload.text` secrets with named `key(...)` handles or `[REDACTED:high-entropy]`.

## Best Possible Solution

Land a revised gateway final-send sanitizer that runs after hooks, presentation rendering, payload normalization, and channel-specific rendering; recursively sanitizes all outbound user-visible string surfaces; derives named handles from canonical SecretRef/sensitive metadata and runtime auth-profile resolution; and updates schema, generated help/docs, and focused regression tests.

## Reproduction Assessment

Yes. The bypasses are reproducible by source inspection: the PR sanitizer is placed in early normalization and only rewrites `payload.text`, while current delivery later applies `message_sending` hooks and can send structured payloads via `sendPayload`.

## Solution Assessment

No. A gateway-layer final-send boundary is the right direction, but this PR's early text-only transform and ad hoc config index are not the narrowest maintainable security fix; the safer design is a final effective-payload sanitizer backed by canonical secret metadata.

Full review comments:

- **[P1] Re-sanitize after hook rewrites:** `src/infra/outbound/deliver.ts:625`
  - body: The sanitizer runs before `message_sending` hooks, but those hooks can replace outbound text before the final transport calls. A hook that adds or substitutes a secret therefore bypasses the new boundary entirely, so redaction needs to run at the final effective-payload send point.
  - confidence: 0.93
- **[P2] Sanitize structured payload fields too:** `src/infra/outbound/deliver.ts:328-329`
  - body: This only rewrites `sanitizedPayload.text`; `presentation`, `interactive`, and `channelData` strings can still be delivered through `sendPayload`. Secrets in labels, option values, placeholders, presentation blocks, or channel-specific data would be sent verbatim.
  - confidence: 0.9
- **[P1] Build the index from canonical secret surfaces:** `src/infra/outbound/sanitize-secrets.ts:82-108`
  - body: `buildCredentialIndex` only walks `cfg.auth` and top-level channel string properties, omitting many registered SecretRef/sensitive paths such as model provider keys, gateway auth, tool keys, talk providers, and nested channel account credentials. Low-entropy configured secrets on those surfaces can miss both named handles and entropy redaction.
  - confidence: 0.88
- **[P1] Tokenize quoted structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:151`
  - body: The tokenizer keeps quotes, braces, and colons attached to values, so common JSON/YAML output such as `{"token":"ghp_..."}` or `password: "shared-password-0000"` will not exact-match configured credentials or known prefixes. Those values need to be split or matched inside structured punctuation.
  - confidence: 0.86
- **[P2] Declare the opt-out in the config contract:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: The helper reads `messages.sanitizeSecrets`, but `MessagesSchema` is strict and `MessagesConfig` does not declare that field. Users cannot set the advertised opt-out in `openclaw.json` without schema/type/docs coverage.
  - confidence: 0.91
