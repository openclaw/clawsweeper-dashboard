## Summary

Keep open: this is useful security work, but the current branch is not merge-ready because the sanitizer runs too early, misses structured outbound payload strings, uses an incomplete credential index, lacks an accepted config contract, has no real outbound proof, and is currently conflicting with main.

## What This Changes

The PR adds a default-on outbound secret sanitizer that rewrites detected secrets in outbound delivery text to named key handles or high-entropy redaction markers.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks, rendering, and adapter normalization, recursively sanitizes outbound string-bearing payload fields, builds handles from canonical sensitive or SecretRef metadata, declares the config contract, and includes upgrade plus live channel proof.

## Reproduction Assessment

Yes, at source level: current main normalizes before hooks, rendering, and final sends, while the PR inserts the sanitizer only in the early normalization step. I did not run a live channel send in this read-only review.

## Solution Assessment

No. Gateway-layer sanitization is the right direction, but this implementation needs to move to the final-send boundary, cover structured fields, use canonical sensitive metadata, and expose the advertised config option before merge.
