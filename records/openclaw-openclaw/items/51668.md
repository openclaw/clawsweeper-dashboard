## Summary

Keep open. Current main still lacks delivery-layer secret-handle sanitization, and this external PR is the canonical implementation path, but the proposed patch has concrete security bypasses and no real behavior proof.

## What This Changes

The branch adds a new outbound secret sanitizer, integrates it into delivery normalization, and adds unit coverage for prefix, named-handle, entropy, delimiter, and opt-out behavior.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks and rendering, recursively covers user-visible structured strings, derives handles from canonical sensitive or SecretRef metadata, declares the config contract, and includes regression tests plus live channel proof.

## Reproduction Assessment

Yes, for the PR review findings via source inspection: the branch sanitizes early `payload.text`, while current main can rewrite content later through `message_sending` and can deliver structured strings through `sendPayload`. I did not run a live channel delivery path in this read-only review.

## Solution Assessment

No. Gateway-layer sanitization is a reasonable direction, but this implementation needs to move to the final-send boundary, cover structured strings, use canonical sensitive metadata, and expose the advertised config contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Re-sanitize after message_sending hooks:** `src/infra/outbound/deliver.ts:331-332`
  - body: The new redaction pass runs during normalization, but delivery applies `message_sending` hooks later and those hooks can replace outbound content. Hook-generated text containing a secret would reach `sendText` or `sendPayload` without another sanitizer pass.
  - confidence: 0.93
- **[P2] Sanitize structured payload strings too:** `src/infra/outbound/deliver.ts:331-332`
  - body: This only rewrites `payload.text`, while payloads with `interactive`, `presentation`, or `channelData` still go through `sendPayload`. Secrets in labels, placeholders, values, or structured text fields would bypass redaction.
  - confidence: 0.9
- **[P1] Use canonical sensitive metadata for handles:** `src/infra/outbound/sanitize-secrets.ts:81-108`
  - body: `buildCredentialIndex` only walks `cfg.auth` and top-level channel strings. Current config schemas mark secrets under providers, tools, cron, gateway auth, nested channel accounts, and SecretRef-backed fields, so low-entropy configured secrets there may be delivered verbatim.
  - confidence: 0.88
- **[P1] Split quoted structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:150`
  - body: The tokenizer keeps quotes, braces, and colons attached to candidate values. Common JSON/YAML output such as `{"token":"ghp_..."}` can miss exact-value and known-prefix matching because the token no longer starts with the secret prefix.
  - confidence: 0.86
- **[P2] Declare sanitizeSecrets in the messages schema:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: `isSecretSanitizationEnabled` reads `messages.sanitizeSecrets`, but the strict messages schema and exported `MessagesConfig` type do not declare that field. The advertised opt-out would be rejected by config validation.
  - confidence: 0.91

## Security Review

Status: needs_attention

Concerns:

- **[high] Hook rewrites bypass redaction:** `src/infra/outbound/deliver.ts:331`
  - body: Because the sanitizer runs before `message_sending` hooks, hook-supplied replacement content can introduce a secret after the only sanitizer pass and still reach channel delivery.
  - confidence: 0.93
