## Summary

Keep this PR open, but not merge-ready: it targets a real outbound secret-leak boundary, yet the patch runs too early, misses structured payload fields and canonical sensitive config surfaces, advertises an undeclared opt-out, and still lacks real outbound proof.

## What This Changes

The branch adds a default-on gateway outbound secret sanitizer with tests, replacing detected secrets in payload text with named `key(...)` handles or high-entropy redaction markers.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks, rendering, and adapter normalization, recursively covers outbound string-bearing fields, uses canonical sensitive/SecretRef metadata for handles, declares the config contract, and includes upgrade plus live channel proof.

## Reproduction Assessment

Yes, at source level. Current main normalizes before hooks, rendering, and final sends, while the PR inserts the sanitizer only in the early normalization step; I did not run a live channel send in this read-only review.

## Solution Assessment

No. Gateway-layer sanitization is the right direction, but this implementation needs to move to the final-send boundary, cover structured fields, use canonical sensitive metadata, and expose the advertised config option before merge.

  - body: This sanitizer runs during early payload normalization, but `message_sending` hooks and presentation rendering can replace outbound content later. A hook-generated or renderer-generated secret would bypass the only sanitizer pass and then reach final channel delivery.
  - confidence: 0.93

  - body: The transform only rewrites `payload.text`, but the delivery path can send presentation, interactive, and `channelData` through `sendPayload`. Secrets in labels, placeholders, values, card text, URLs, or channel-specific fields remain unsanitized.
  - confidence: 0.9

  - body: `buildCredentialIndex` walks only `cfg.auth` and top-level channel string values. Current schemas mark many provider, tool, gateway, nested channel, and SecretRef fields as sensitive, so low-entropy configured secrets there can still be delivered verbatim.
  - confidence: 0.88
- **[P1] Split quoted JSON and YAML values before matching:** `src/infra/outbound/sanitize-secrets.ts:148`
  - body: The tokenizer keeps quotes, braces, and colons attached to candidate values. JSON or YAML such as `{"token":"ghp_..."}` can miss exact-value and known-prefix matching because the token no longer starts with the secret prefix.
  - confidence: 0.86
- **[P1] Declare the sanitizeSecrets config option:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: `isSecretSanitizationEnabled` reads `messages.sanitizeSecrets`, but `MessagesSchema` is strict and `MessagesConfig` omits that field. Users cannot rely on the advertised opt-out through validated config, which is merge-critical for a default-on content rewrite.
