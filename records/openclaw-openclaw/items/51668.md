## Summary

Keep open: the security goal is valid and not implemented on current main, but this PR is not safe to merge because the sanitizer runs too early, only covers `payload.text`, misses canonical secret surfaces, and lacks real behavior proof.

## What This Changes

This PR adds a gateway outbound secret sanitizer, wires it into outbound delivery normalization, and adds sanitizer tests for prefix, named credential, entropy, and opt-out behavior.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks and final payload rendering, recursively covers user-visible payload strings, derives handles from canonical sensitive/SecretRef metadata, and updates schema, docs, changelog, and regression tests together.

## Reproduction Assessment

Yes for the review findings via source inspection: `message_sending` can replace text after the PR's pass, `sendPayload` carries structured strings, and the strict messages schema lacks `sanitizeSecrets`. I did not run a live delivery path in this read-only review.

## Solution Assessment

No. A gateway-level sanitizer is the right direction, but this implementation is too early, too text-only, and based on ad hoc credential discovery instead of the canonical secret metadata contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Re-sanitize after message_sending hooks:** `src/infra/outbound/deliver.ts:324-326`
  - body: The sanitizer runs during early payload normalization, but `message_sending` runs later and can replace outbound text. Hook-generated content containing a secret would reach the final send without another redaction pass.
  - confidence: 0.93
- **[P2] Sanitize structured payload strings too:** `src/infra/outbound/deliver.ts:324-326`
  - body: This only rewrites `payload.text`, while `presentation`, `interactive`, and `channelData` can still be delivered through `sendPayload`. Secrets in labels, placeholders, values, or presentation text would bypass the sanitizer.
  - confidence: 0.9
- **[P1] Use canonical sensitive metadata for handles:** `src/infra/outbound/sanitize-secrets.ts:82-108`
  - body: `buildCredentialIndex` walks only `cfg.auth` and top-level channel strings, but configured secrets also live under providers, tools, gateway auth, cron, and nested channel accounts. Low-entropy values there can miss both named handles and entropy redaction.
  - confidence: 0.88
- **[P1] Split quoted structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:150-154`
  - body: The tokenizer still keeps quotes, braces, and colons attached to candidate values. Common JSON/YAML like `{"token":"ghp_..."}` or `password: "..."` can miss exact-value and prefix matching.
  - confidence: 0.86
- **[P2] Declare messages.sanitizeSecrets in schema:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: `isSecretSanitizationEnabled` reads `messages.sanitizeSecrets`, but the strict `MessagesSchema` and `MessagesConfig` omit that field. The advertised opt-out would be rejected by config validation.
  - confidence: 0.91

## Security Review
