## Summary

Keep open: this is the canonical implementation PR for the closed tracking issue, but it is not merge-ready because source inspection and prior review comments show concrete security-boundary bypasses, a missing config contract, no real outbound proof, and current merge conflicts with main.

## What This Changes

The PR adds a default-on outbound secret sanitizer with named credential handles, wires it into outbound delivery normalization, and adds unit tests for prefix, entropy, named-handle, and config opt-out behavior.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks/rendering/adapter normalization, recursively sanitizes outbound payload strings, builds handles from canonical sensitive or SecretRef metadata, declares the config contract, and includes upgrade plus live channel proof.

## Reproduction Assessment

Yes, at source level: current main normalizes payloads before `message_sending` hooks, presentation rendering, later adapter normalization, and `sendPayload`, while the PR inserts its sanitizer only in the early normalization step. I did not run a live channel send in this read-only review.

## Solution Assessment

No. Gateway-layer sanitization is the right direction, but this implementation needs to move to the final-send boundary, cover structured fields, use canonical sensitive metadata, and expose the advertised config option before merge.

  - body: This sanitizer runs during early payload normalization, but `message_sending` hooks and presentation rendering can replace outbound content later. A hook-generated secret would bypass the only sanitizer pass and then reach final channel delivery.
  - confidence: 0.93

  - body: `buildCredentialIndex` walks only `cfg.auth` and top-level channel string values. Current schemas mark many provider, tool, gateway, cron, nested channel, and SecretRef fields as sensitive, so low-entropy configured secrets there can still be delivered verbatim.
  - confidence: 0.88

  - body: The tokenizer keeps quotes, braces, and colons attached to candidate values. JSON or YAML such as `{"token":"ghp_..."}` can miss exact-value and known-prefix matching because the token no longer starts with the secret prefix.
  - confidence: 0.86

## Security Review
