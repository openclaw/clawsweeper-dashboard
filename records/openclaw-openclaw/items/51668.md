## Summary

Keep open: the security hardening direction is valid and linked to https://github.com/openclaw/openclaw/issues/50718, but the current PR is not merge-ready because the sanitizer runs too early, covers only text, indexes sensitive config incompletely, lacks the advertised config contract, is currently conflicting with main, and has no real outbound proof.

## What This Changes

The branch adds a gateway-layer outbound secret sanitizer, wires it into delivery normalization, advertises a `messages.sanitizeSecrets` opt-out, and adds unit tests.

## Best Possible Solution

Land a revised final-send sanitizer that runs after hooks/rendering, recursively sanitizes outbound payload strings, indexes canonical sensitive or SecretRef-backed config values, declares the config contract, and includes redacted live channel proof.

## Reproduction Assessment

Yes, at source level: the PR sanitizes during early normalization, while current main can rewrite content afterward through `message_sending` hooks and can deliver structured strings through `sendPayload`. I did not run a live channel send in this read-only review.

## Solution Assessment

No. Gateway-layer sanitization is the right direction, but this implementation needs to move to the final-send boundary, cover structured fields, use canonical sensitive metadata, and expose the advertised config option before merge.
