## Summary

Keep open. Current main still propagates optimizer failures before an under-cap original image can be sent, so the PR is useful, but the head is stale over current media diagnostics and still lacks the required user-facing changelog entry.

## What This Changes

The PR changes shared media loading to return original under-cap non-HEIC image bytes when optimization fails, preserve original MIME/file metadata, keep guard failures fail-closed, retain optimizer cause details, and add focused media regression tests.

## Best Possible Solution

Land the shared media fallback after rebasing it onto current main, preserving the existing aggregated diagnostics and guard behavior, adding a single-line changelog fix entry, and then closing the directly covered optimizer-failure issues while leaving native crash work separate.

## Reproduction Assessment

Yes. Current main has a high-confidence code path where an under-cap image still propagates optimizer backend failure because `optimizeAndClampImage` has no catch around `optimizeImageWithFallback`; the PR's mock-backed test models that failure path.

## Solution Assessment

Mostly yes, but not in the current branch state. The `loadWebMedia`/`optimizeAndClampImage` boundary is the right place to preserve original MIME metadata, but the final patch should be rebased over current diagnostics and include the required changelog entry.
