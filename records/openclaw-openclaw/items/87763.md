## Summary

Keep open: current main and the latest release still route model-provider requests through the SSRF guarded fetch path with pinned dispatchers that inherit Node/Undici family autoselection and collapse automatic dual-stack pinned lookups to IPv4 when IPv4 exists. The issue remains the canonical tracker for this guarded-fetch address-family failure; no merged fix or better canonical open item owns it yet.

## What This Changes

The issue reports that SSRF-guarded model fetches can time out or reset on dual-stack providers because DNS pinning and address-family selection interact poorly.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose and validate a shared SSRF guarded-fetch family-selection repair that preserves DNS pinning and resolved-address validation across model fetch and overlapping guarded-fetch surfaces.

## Reproduction Assessment

Yes for source-level reproduction: two commenters provided live dual-stack provider reproductions, and current main still follows the implicated model-fetch to pinned-dispatcher path. I did not run credentialed provider calls locally, so this is not marked reproduced.

## Solution Assessment

Unclear as an implementation choice: the issue is valid, but the best repair must preserve SSRF DNS pinning while deciding whether the shared dispatcher, lookup policy, or model retry behavior owns alternate-family handling.
