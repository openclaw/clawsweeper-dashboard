## Summary

Keep open: current main and v2026.6.6 still keep model-provider requests on the shared SSRF pinned-dispatcher path with IPv4-preferred pinned lookup records and Node/Undici family autoselection applied. The added hydragpt and OpenRouter live reports make the issue stronger, and the related web-search timeout report at https://github.com/openclaw/openclaw/issues/87505 overlaps but does not supersede this model-fetch-specific failure.

## What This Changes

The issue asks OpenClaw to fix model-provider fetch timeouts caused by the SSRF pinned DNS dispatcher interacting badly with Node autoSelectFamily on dual-stack provider hosts.

## Best Possible Solution

Fix the shared pinned-dispatcher family-selection behavior while preserving SSRF host and resolved-IP checks, then validate model fetch plus the related guarded-fetch surfaces such as web_search on Node 24 dual-stack networking.

## Reproduction Assessment

Yes, there is a high-confidence manual reproduction path from the hydragpt and OpenRouter comments, and current source still follows the implicated model-fetch -> SSRF pinned-dispatcher path. I did not run the credentialed provider call myself, so the structured status is source_reproducible rather than reproduced.

## Solution Assessment

Unclear as proposed: a model-only IPv4/IPv6 fallback could reduce the symptom, but the maintainable fix should settle the shared pinned lookup and family-selection behavior without weakening SSRF checks.
