## Summary

Keep open: current main still routes model-provider requests through the shared SSRF pinned-dispatcher path with IPv4-preferring pinned lookup records and inherited Node/Undici family autoselection. ## What This Changes

The issue reports that SSRF-guarded model fetches can time out or reset on dual-stack OpenAI-compatible providers because pinned DNS lookup behavior conflicts with address-family selection.

## Best Possible Solution

Keep this issue as the canonical tracker until a shared SSRF pinned-dispatcher family-selection repair is chosen, implemented, and validated across model fetch plus overlapping guarded-fetch surfaces.

## Reproduction Assessment

Yes for source-level reproduction: two commenters provided live dual-stack provider reproductions, and current main still follows the implicated model-fetch to pinned-dispatcher path. I did not run credentialed provider calls locally, so this is not marked reproduced.

## Solution Assessment

Unclear as an implementation choice: the issue is valid, but the best repair must preserve SSRF DNS pinning while deciding whether the shared dispatcher, lookup policy, or model retry behavior owns alternate-family handling.
