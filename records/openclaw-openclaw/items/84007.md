## Summary

Keep open: this addresses a real documented subagent thinking-inheritance bug and the supplied live proof is useful, but the branch still treats caller-inherited thinking as an explicit gateway override, which can make mixed-model subagent spawns fail on unsupported thinking levels instead of following normal stored-session fallback behavior.

## What This Changes

The PR changes subagent spawn planning so child sessions can inherit the requester session thinking level after explicit spawn thinking and configured subagent thinking defaults are absent.

## Best Possible Solution

Preserve the documented inheritance order, but carry thinking source metadata so caller-inherited thinking is persisted as child session state without being sent as an explicit gateway override; explicit spawn thinking and configured subagent policy should still win.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current docs promise caller inheritance, but current main's resolver and spawn path never pass requester session thinking into the subagent thinking plan. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The direction is right, but the implementation should distinguish caller-inherited session state from explicit/configured thinking before merge so unsupported target models keep the normal fallback/remap behavior.
