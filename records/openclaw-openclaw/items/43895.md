## Summary

Keep open: the PR targets a real source-reproducible status display bug, but it is based on an older status-command shape, bypasses the current lazy status runtime boundary, and still lacks real multi-agent status output proof.

## What This Changes

The PR adds a `variesByAgent` marker to status session defaults, updates CLI/TUI rendering to show `varies by agent`, and adds divergent/uniform model-default regression tests.

## Best Possible Solution

Land a rebased fix that computes per-agent effective defaults through the lazy status runtime, propagates a variation marker through JSON/plain/TUI shared renderers, keeps status startup read-only, and includes focused tests plus redacted real output.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure multiple agents with different `agents.list[].model` values and inspect `openclaw status --json` or the plain/TUI summary; current main computes `sessions.defaults` once without an agent id.

## Solution Assessment

No. The intended behavior is right, but this branch should be rebased through `status.summary.runtime.ts` and `status-overview-values.ts` instead of editing the old inline renderer and importing model helpers directly.
