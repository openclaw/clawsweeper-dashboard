## Summary

Keep open: the reported status display problem is still source-reproducible on current main, but this branch is not mergeable as-is because it updates an old renderer path, bypasses the current lazy runtime boundary, and lacks contributor real-behavior proof.

## What This Changes

The branch aggregates effective default models across configured agents, adds a `variesByAgent` status field, and renders `varies by agent` in CLI/TUI summaries when agent defaults differ.

## Best Possible Solution

Land a rebased implementation that computes per-agent effective defaults through `status.summary.runtime.ts`, updates JSON/CLI/TUI/shared overview handling, and includes focused tests plus redacted real status proof.

## Reproduction Assessment

Yes, source-reproducible: current main builds `sessions.defaults` from `resolveConfiguredStatusModelRef` without `agentId`, while the current agent model contract and runtime helpers support per-agent effective defaults. I did not run a live CLI reproduction in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is right, but the patch needs to be rebased onto the current shared overview renderer and lazy status runtime boundary before it is the narrow maintainable fix.
