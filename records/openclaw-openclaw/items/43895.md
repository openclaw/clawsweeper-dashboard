## Summary

The PR is addressing a real status display bug, but it is stale against current main: the status renderer and model-resolution boundary have moved, so this branch would not update the active human status path and would reintroduce static imports on the lazy status path. It also still lacks after-fix real behavior proof from a real multi-agent status run.

## What This Changes

The branch aggregates effective session default models across configured agents, adds a `variesByAgent` status flag, renders `varies by agent` in CLI/TUI status output, and adds regression coverage.

## Best Possible Solution

Land a rebased implementation that computes per-agent effective defaults through `status.summary.runtime.ts`, carries the variation state through JSON/CLI/TUI including `buildStatusSessionsOverviewValue()`, and adds focused tests plus redacted real status proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the aggregate default path resolves one model without `agentId`, while the runtime resolver and docs support per-agent model overrides.

## Solution Assessment

No, not as currently proposed. The direction is right, but the branch targets an older inline renderer and bypasses the current lazy status runtime boundary; a rebased runtime-surface aggregation plus shared renderer update is safer.
