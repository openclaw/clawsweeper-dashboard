## Summary

Keep open: the automation problem is real, but this PR is not mergeable as-is because the Copilot PAT path is wired to stale/currently inactive resolver plumbing, the required request headers are not attached, non-interactive token validation regresses, and real behavior proof is missing.

## What This Changes

Adds non-interactive token inputs to `models auth paste-token` and attempts GitHub Copilot PAT support by bypassing token exchange and using the enterprise Copilot endpoint.

## Best Possible Solution

Implement Copilot PAT support in the active provider-auth/runtime path with prepared request headers, and let https://github.com/openclaw/openclaw/pull/63050 or an equivalent reviewed patch own the `paste-token --token` CLI surface.

## Reproduction Assessment

Yes for the source-level current-main gaps: `paste-token` has no token flag and Copilot runtime auth still exchanges every GitHub token. A live PAT-backed Copilot failure path was not reproduced in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR is not the best current-main solution because it patches stale resolver plumbing, omits runtime request headers, and bypasses existing token validation for scripted input.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Attach PAT headers to runtime requests:** `extensions/github-copilot/index.ts:162-167`
  - body: The PAT branch switches to the enterprise endpoint but returns only `apiKey`, `baseUrl`, and `expiresAt`; it never passes the new `COPILOT_EDITOR_HEADERS` to runtime requests. Since the PR says those headers are required, PAT-backed Copilot calls can still run without the required editor identity even though current runtime auth supports request overrides.
  - confidence: 0.88
- **[P2] Patch the active Copilot token resolver:** `src/agents/github-copilot-token.ts:108-118`
  - body: This adds PAT detection to `src/agents/github-copilot-token.ts`, but current main routes the GitHub Copilot plugin through the SDK/provider-auth resolver re-export. After rebasing, leaving that active resolver unchanged would keep sending PATs to `/copilot_internal/v2/token`.
  - confidence: 0.84
- **[P2] Validate scripted token input before storing:** `src/commands/models/auth.ts:383-392`
  - body: The flag, stdin, and env-var paths trim and persist tokens directly instead of using the prompt validator and Anthropic setup-token normalization. That lets automated Anthropic setup store malformed token text that the interactive path rejects or normalizes.
  - confidence: 0.82
