## Summary

Keep open: the underlying automation and Copilot PAT requests are still plausible, but this PR is not merge-ready because it omits PAT request headers, patches stale Copilot auth plumbing for current main, skips non-interactive token validation, and lacks real behavior proof.

## What This Changes

The PR adds non-interactive token sources to `models auth paste-token` and attempts GitHub Copilot PAT support by bypassing token exchange and using the enterprise Copilot endpoint.

## Best Possible Solution

Implement PAT support in the active Copilot SDK/provider-auth path with runtime `request.headers`, and let https://github.com/openclaw/openclaw/pull/63050 or an equivalent reviewed patch handle the `paste-token --token` CLI surface.

## Reproduction Assessment

Yes, source-reproducible for the current-main gaps: `paste-token` lacks a token flag and Copilot runtime auth still exchanges every GitHub token. I did not establish a live PAT-backed Copilot failure path.

## Solution Assessment

No. The direction is useful, but this PR is not the best current-main solution because it omits runtime headers, patches stale resolver plumbing, and skips validation for non-interactive token input.
