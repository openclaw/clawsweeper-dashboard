## Summary

Keep open. Current main and the latest release still hardcode the Dream Diary narrative wait at 60 seconds, and existing tests confirm a timeout warns and leaves `DREAMS.md` unwritten.

## What This Changes

Raise or make configurable the Dream Diary narrative wait timeout so dreaming phases with embedded-agent prep can finish instead of timing out at 60 seconds.

## Best Possible Solution

Add a memory-core-owned narrative timeout policy that feeds the Dream Diary `waitForRun` call, preserves bounded cron behavior, and updates the strict schema, docs, regression tests, and changelog together.

## Reproduction Assessment

Yes, source-backed: current main still uses a 60-second Dream Diary `waitForRun` budget, `agent.wait` returns timeout after that caller budget, and timeout tests leave `DREAMS.md` unwritten. I did not run a live 15-workspace gateway reproduction in this read-only review.

## Solution Assessment

Unclear until a maintainer chooses the public contract. Raising the default and/or adding `dreaming.narrativeTimeoutMs` would address the observed path, but a 4-minute default trades fewer false timeouts for longer stalled cron jobs.
