## Summary

Keep open. Current main and the latest release still hardcode the Dream Diary narrative wait at 60 seconds, timeout coverage still leaves `DREAMS.md` unwritten, and the open fix PRs have not landed.

## What This Changes

The issue asks memory-core to stop Dream Diary narrative phases from timing out by raising or overriding the hardcoded 60s narrative subagent wait, preferably with a supported config knob.

## Best Possible Solution

Review and land or replace a single memory-core fix that gives Dream Diary narrative runs enough bounded time, updates schema/docs/tests/changelog if a public timeout surface is added, and preserves bounded cron behavior.

## Reproduction Assessment

Yes, source-backed: current main and v2026.5.7 still use a fixed 60s Dream Diary `waitForRun` budget, and the timeout tests show `DREAMS.md` remains absent on timeout. I did not run a live 15-workspace gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public timeout contract. Raising the default and/or adding an env/config override would address the observed path, but the supported knob and longer cron stall budget need product review before this issue closes.
