## Summary

Keep open. Current main and the latest release still hardcode the Dream Diary narrative wait at 60 seconds, and the issue is paired with open PR #76342 using closing syntax, so it should remain open until that PR lands or maintainers choose a different timeout/config contract.

## What This Changes

Raise or make configurable the Dream Diary narrative timeout so dreaming phases have enough time to append `DREAMS.md` entries on large workspaces.

## Best Possible Solution

Review and land #76342 if an env override plus 240-second default is the chosen contract, or extend that PR with a documented memory-core config field before merging.

## Reproduction Assessment

Yes, source-backed: current main still uses a 60-second Dream Diary `waitForRun` budget, `agent.wait` returns `status: "timeout"` when that budget expires, and tests confirm timeout leaves `DREAMS.md` unwritten. I did not rerun the reporter's live 15-workspace gateway scenario.

## Solution Assessment

Unclear until maintainers choose the contract. Raising the default and adding an operator override matches the observed failure, but the issue also asks for a config-schema field and that public setting decision belongs with memory-core maintainers.
