## Summary

Keep open. Current main still hardcodes the Dream Diary narrative wait at 60 seconds, and the linked fix PR is still open, so this issue should remain open until that PR or a successor lands.

## What This Changes

The issue asks memory-core Dream Diary narrative runs to use a longer or tunable timeout instead of the hardcoded 60-second subagent wait budget.

## Best Possible Solution

Review and land #76342 or a successor that adjusts the memory-core narrative timeout policy and documents any accepted public tuning surface.

## Reproduction Assessment

Yes, source-backed: current main still hardcodes `60_000`, passes it to `waitForRun`, `agent.wait` returns timeout after the caller budget, and tests verify timeout leaves `DREAMS.md` unwritten. I did not run a live 15-workspace replay in this read-only review.

## Solution Assessment

Unclear until a maintainer chooses the public contract. Raising the default and adding an env override addresses the central timeout path, but the requested config-schema knob changes the supported tuning surface.
