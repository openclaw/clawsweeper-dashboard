## Summary

Keep open. Current main and the shipped v2026.5.2 release still hardcode the Dream Diary narrative wait at 60 seconds, the timeout path leaves `DREAMS.md` unwritten, and open PR #76342 is the active implementation candidate that should be reviewed before this issue closes.

## What This Changes

Raise or make configurable the memory-core Dream Diary narrative wait budget so larger dreaming sweeps stop timing out before writing `DREAMS.md`.

## Best Possible Solution

Review and land #76342 or a replacement memory-core fix that preserves bounded cron behavior, updates tests, and adds docs/changelog/schema coverage if a public timeout knob is accepted.

## Reproduction Assessment

Yes. Source inspection of current main shows a 60-second Dream Diary `waitForRun` budget, and existing tests show the timeout path warns and leaves `DREAMS.md` unwritten; I did not run a live 15-workspace gateway reproduction in this read-only pass.

## Solution Assessment

Unclear. #76342 is a narrow candidate for the observed bug, but a maintainer still needs to choose whether env-only tuning and a 240-second default are the right public contract or whether a schema-backed config field is required.
