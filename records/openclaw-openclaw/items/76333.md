## Summary

Keep open. Current main and the latest release still use a hardcoded 60-second Dream Diary narrative wait, the timeout path drops `DREAMS.md` output, and two open PRs are already linked as implementation candidates.

## What This Changes

Raise or make configurable the Dream Diary narrative wait budget so dreaming phases with high embedded-agent prep costs can append `DREAMS.md` instead of timing out.

## Best Possible Solution

Review and land one narrow memory-core fix that increases the narrative wait budget and plumbs the chosen timeout policy through schema, docs, tests, and changelog as needed, then close or supersede the duplicate PR.

## Reproduction Assessment

Yes, source-backed: current main hardcodes a 60-second `waitForRun` budget, gateway `agent.wait` returns timeout at that caller budget, and the existing timeout test verifies no `DREAMS.md` is written. I did not run a live 15-workspace gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public timeout contract. The linked env-var-plus-240s PRs address the observed timeout path, but a schema-backed config field would need broader docs/config/test follow-through.
