## Summary

Keep open: current main still relays node exec-approval management directly to the node without first checking that the connected node advertises `system.execApprovals.get/set`, so the requested graceful unsupported-capability path is not implemented.

## What This Changes

The issue requests structured, actionable CLI failure for `openclaw approvals get --node ... --json` when the selected node lacks node exec-approvals management commands.

## Best Possible Solution

Add a gateway-side unsupported-capability guard for node approvals get/set, return structured JSON/text errors, and cover the CLI plus gateway relay with focused regression tests.

## Reproduction Assessment

No, not as a live end-to-end reproduction in this review. The source path is clear: the node approvals gateway handler invokes `system.execApprovals.get/set` without checking the connected node's advertised commands, and tests do not cover the missing-capability case.

## Solution Assessment

Yes, the best fix is a narrow capability check at the gateway relay boundary, with CLI payload/error handling kept structured for `--json`. This should not enable approvals management for unsupported nodes or add a new config option.
