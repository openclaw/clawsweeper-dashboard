## Summary

Keep open: current main still forwards `exec.approvals.node.get/set` directly to the selected node without first checking that the connected node advertises `system.execApprovals.get/set`, so the reported raw TypeError path is still plausible and the requested structured unsupported-capability error is not implemented.

## What This Changes

The issue requests graceful structured errors when `openclaw approvals get --node` targets a connected node that lacks exec-approvals management commands.

## Best Possible Solution

Add a gateway-side unsupported-capability check for node approvals get/set, keep raw `node.invoke system.execApprovals.*` blocked, and return structured CLI/JSON errors without enabling approvals management for unsupported nodes.

## Reproduction Assessment

No, not as a live end-to-end reproduction in this read-only review. The source path is clear: the gateway node approvals relay invokes `system.execApprovals.get/set` without first checking the connected node's advertised commands.

## Solution Assessment

Yes, the best fix is a narrow capability guard at the gateway relay boundary plus structured CLI/JSON error handling. A broader node advertisement fix is useful but does not replace graceful handling for unsupported nodes.
