## Summary

Keep open: current main still relays node exec-approval get/set calls without first checking the selected node's advertised command surface, so a node that lacks `system.execApprovals.get/set` can still hit an unhelpful failure path.

## What This Changes

The issue asks OpenClaw CLI/Gateway to return a structured unsupported-capability error when `openclaw approvals get --node ... --json` targets a node that does not advertise exec-approval management commands.

## Best Possible Solution

Add a narrow gateway-side command-capability guard for `exec.approvals.node.get/set`, return a structured unsupported-capability error for text and `--json`, and cover the missing-command case with focused CLI/gateway tests.

## Reproduction Assessment

No, not as a live end-to-end reproduction in this review. The source path is clear: current main relays `system.execApprovals.get/set` without checking the node's advertised commands, while sibling `node.invoke` already has the structured command-declared guard.

## Solution Assessment

Yes, the best fix is narrow gateway-boundary validation before the node invoke, not a new config option or enabling approvals management for unsupported nodes.
