## Summary

Keep open: current main and v2026.6.9 handle the generated `gateway.systemd.env` path for state-dir `.env` values, and the new comment usefully confirms that path can work, but the reported update-time outage is not fully closed because Linux update-mode systemd repair still leaves live units unchanged and the `openclaw.env` versus generated-env-file contract needs maintainer/security direction.

## What This Changes

The issue asks OpenClaw to keep Linux user-systemd Gateway auth env wiring valid across update-triggered restarts when `gateway.auth.token` resolves from `OPENCLAW_GATEWAY_TOKEN`.

## Best Possible Solution

Decide the supported Linux systemd service env-file contract, then align install, doctor, update-mode repair, docs, focused tests, and Linux restart proof around that contract without broadening secret loading by accident.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the missing-env failure path: an unresolved active env SecretRef fails startup when the service lacks `OPENCLAW_GATEWAY_TOKEN`, but the exact WSL2 update restart sequence still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the env-file contract; preserving the generated `gateway.systemd.env` safety path plus explicit provenance/update handling looks safer than blindly loading `openclaw.env` or adding a token fallback.
