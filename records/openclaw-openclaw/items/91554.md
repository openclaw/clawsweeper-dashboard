## Summary

Keep open: current main and latest v2026.6.9 still leave a meaningful Linux/systemd update gap for env-backed Gateway auth, and the remaining fix needs maintainer choice around service env-file provenance and credential safety.

## What This Changes

The issue reports that a Linux/WSL systemd user Gateway service can restart after update without loading the env-backed Gateway auth token, leaving the Gateway down until the unit/env wiring is repaired.

## Best Possible Solution

Choose and document the supported Linux systemd service env-file contract, then align install/update/doctor behavior with focused systemd tests, fail-closed SecretRef semantics, and Linux restart proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: source inspection shows the missing-env startup failure and update-mode deferral path, but the exact WSL2 update restart sequence still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide the supported env-file contract; a narrow install/update/doctor contract is safer than blindly broadening env-file loading or adding auth fallback.
