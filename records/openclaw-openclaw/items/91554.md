## Label Justifications

- `P2`: A bounded Linux/WSL service configuration can leave the gateway unavailable, but the mismatch with the supported service shape still needs reproduction and provenance.
- `impact:crash-loop`: The report shows repeated gateway startup exits followed by systemd start-limit failure.
- `impact:auth-provider`: The startup failure is caused by an unresolved environment-backed gateway authentication token.

## Summary

Keep open: the reported outage is credible, but neither v2026.6.1 nor current main reproduces the stated managed-service shape because both stage the documented state-dir `.env` into `gateway.systemd.env` and render an `EnvironmentFile`; maintainers need to establish what created `openclaw.env` and the unit without that directive.

## What This Changes

Ensure Linux/WSL systemd gateway services retain access to an environment-backed gateway auth token across update-triggered restarts.

## Best Possible Solution

Identify the producer of `openclaw.env` and the pre-update unit, then either fix that official installer with a doctor migration into canonical `.env`/`gateway.systemd.env`, or document the unsupported local layout and add a targeted diagnostic instead of a permanent fallback.

## Reproduction Assessment

No. The failure mechanism is clear, but v2026.6.1 and current main already generate an EnvironmentFile from canonical `~/.openclaw/.env`; no high-confidence path currently reproduces the reported `openclaw.env` plus missing-directive unit.

## Solution Assessment

Unclear. Loading arbitrary `openclaw.env` is not the best fix unless a shipped OpenClaw installer is proven to create it; the safer path is producer repair or migration into the existing canonical env flow.
