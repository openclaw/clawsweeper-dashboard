## Summary

Keep open: current main and v2026.6.9 have safer state-dir `.env` to `gateway.systemd.env` handling, but they do not fully settle the reported update-time Linux/WSL systemd auth outage because update-mode doctor still leaves live systemd units unchanged and the supported legacy/env-file contract needs maintainer security judgment.

## What This Changes

The issue asks OpenClaw to keep env-backed Gateway auth token wiring durable for Linux systemd user services across update-triggered restarts.

## Best Possible Solution

Choose and document the supported Linux systemd service env-file contract, then align install/update/doctor behavior with focused systemd tests and Linux restart proof while preserving filtered credential handling.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the failure path: when the systemd process lacks `OPENCLAW_GATEWAY_TOKEN`, env SecretRef resolution fails and startup aborts. The exact WSL2 update/restart sequence still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear; directly broadening env-file loading or adding an auth fallback is riskier than a maintainer-chosen systemd env-file/update contract with targeted migration or operator guidance.
