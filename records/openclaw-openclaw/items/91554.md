## Summary

Keep open: current main and v2026.6.9 have improved fresh/forced systemd installs and stale literal env-reference cleanup, but update-mode doctor still deliberately leaves live Linux systemd units unchanged, so a legacy or drifted unit missing the needed EnvironmentFile can survive an update-triggered restart.

## What This Changes

Ensure Linux/WSL systemd gateway services load durable env-backed gateway auth tokens across update-triggered restarts.

## Best Possible Solution

Keep the current generated-env-file and literal-reference protections, then add a maintainer-approved update/doctor path that safely repairs legacy env-backed gateway service units without overwriting operator-owned systemd directives.

## Reproduction Assessment

No live high-confidence WSL2 reproduction was run in this read-only review. Source inspection does show the remaining update-mode path still leaves live Linux systemd units unchanged, which preserves the reported failure shape for legacy or drifted units.

## Solution Assessment

Unclear until maintainers choose the service-update contract. Fresh and forced installs have the right generated-env-file direction, but the best remaining fix needs safe update-mode migration or generated-unit provenance rather than an unconditional rewrite.
