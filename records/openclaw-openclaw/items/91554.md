## Summary

Keep open: current main has the generated `gateway.systemd.env`/`EnvironmentFile` path for staged systemd services, but the reported update-time failure is not clearly fixed because current doctor/update behavior still leaves legacy or operator-modified systemd units unchanged without `--force`, and the supported env-file/security contract needs maintainer review.

## What This Changes

The issue reports that Linux user-systemd gateway restarts can fail after update when env-backed gateway auth tokens are not loaded into the service process.

## Best Possible Solution

Keep the generated env-file model, then resolve the update-time contract through maintainer-approved provenance or explicit operator repair so env-backed gateway auth is consistently available without silently overwriting custom units.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes for the missing-env failure path: unresolved active env SecretRefs fail startup, but the exact WSL2 update restart sequence still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear; generated `gateway.systemd.env` is the right default path, but automatic repair of legacy or operator-modified units needs the provenance/product decision tracked by the related open issue.
