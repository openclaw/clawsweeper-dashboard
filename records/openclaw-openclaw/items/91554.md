## Label Justifications

- `P2`: A bounded Linux/WSL systemd service path can leave the Gateway unavailable after update, but it is not a universal runtime outage.
- `impact:auth-provider`: The failure is caused by env-backed Gateway auth SecretRef resolution not receiving `OPENCLAW_GATEWAY_TOKEN` in the managed service process.
- `impact:crash-loop`: The reported systemd service repeatedly exits and reaches start-limit failure after the update restart.
- `impact:security`: The remaining fix decision involves which credential-bearing env files are trusted by the managed Gateway service.

## Summary

Keep open: current main and the latest release still only wire the canonical state-dir `.env` and generated `gateway.systemd.env` into the systemd service, while the reported `openclaw.env` source remains unsupported and env SecretRefs still fail closed when the process lacks `OPENCLAW_GATEWAY_TOKEN`.

## What This Changes

Make managed Linux systemd Gateway installs and update restarts keep env-backed `OPENCLAW_GATEWAY_TOKEN` available to the service process.

## Best Possible Solution

Keep this open until maintainers define the supported env-file contract and the install, update, and doctor paths either preserve that source or emit a clear repair diagnostic with live systemd proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection clearly reproduces the missing-env failure path when the token exists only in `openclaw.env`, but the exact WSL2 update-restart sequence still needs live proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the env-file contract; blindly loading `openclaw.env` would change credential trust, while a doctor/install/update contract may be safer.
