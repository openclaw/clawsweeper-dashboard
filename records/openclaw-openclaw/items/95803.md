Keep this PR open: it targets a real user-visible duplicate media-delivery bug, but the patch is not merge-ready because it can leave stale pending-media metadata after removing the queued URL, and the external PR still lacks after-fix real behavior proof.

## Best Possible Solution

Keep the handler-level success/failure gate, but make the pending media URL queue and its metadata flags stay consistent after deduplication, then add redacted real behavior proof showing the media is delivered once.

## Reproduction Assessment

Yes, from source inspection: current main separately tracks successful message-tool media and pending fallback tool media, and the fallback queue is later consumed at message or agent end. I did not execute the regression test in this read-only review.

## Solution Assessment

No as written. `handleToolExecutionEnd` is a reasonable layer because it sees successful versus failed message-tool sends, but the fix must also clear or recompute pending media metadata when it removes queued URLs.
