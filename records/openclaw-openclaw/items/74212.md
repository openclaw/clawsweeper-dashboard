## Summary

Keep open. Current main and the latest tagged release still mask the OpenAI Codex device-pairing code in remote mode, and open PR #74258 is already paired with this issue using closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Fix `openclaw models auth login --provider openai-codex` so Device Pairing over SSH shows the upstream device code in the terminal instead of masking it.

## Best Possible Solution

Land or replace #74258 with the provider-local fix: include `Code: ${userCode}` in the `prompter.note` remote-mode output, keep `ctx.runtime.log(...)` URL-only, update the provider test to assert both terminal visibility and log hygiene, and include the normal user-facing changelog entry for the bug fix.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction is the existing OpenAI Codex device-code auth path with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The narrowest maintainable solution is provider-local because the helper already has `userCode` and the CLI already marks SSH as remote; #74258 matches that boundary better than logging the code or changing core auth plumbing.
