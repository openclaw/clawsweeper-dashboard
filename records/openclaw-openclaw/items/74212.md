## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote/SSH mode, the latest release tag still has the same placeholder, and open PR #74258 is already paired with this issue using closing syntax, so cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that `openclaw models auth login --provider openai-codex` display the OpenAI Codex device-pairing code in SSH-visible terminal output instead of replacing it with `[shown on the local device only]`.

## Best Possible Solution

Land or replace PR #74258 with the provider-local fix: include `Code: ${userCode}` in the user-facing `prompter.note` output for remote mode, keep the persistent `ctx.runtime.log(...)` output URL-only, update the provider tests to cover both terminal visibility and log hygiene, and include the normal user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is the existing OpenAI Codex device-code auth unit path with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and the test currently asserts that broken output.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-local because the helper already has `userCode` and the CLI already passes remote/SSH state into provider auth; changing core auth plumbing or writing the code to persistent runtime logs would be broader and riskier.
