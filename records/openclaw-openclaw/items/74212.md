## Summary

Keep open: current main still sends the placeholder to the remote prompter, and #74258 is an open unmerged closing PR for the same provider-local fix, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that OpenAI Codex device-code authentication over SSH hides the pairing code behind a placeholder, blocking completion of `openclaw models auth login --provider openai-codex`.

## Best Possible Solution

Land #74258 or an equivalent provider-local change that prints the short-lived device code on the interactive `prompter.note` surface for remote sessions, keeps `runtime.log` free of the code, updates the focused provider tests, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the existing provider unit flow with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and asserts the real code is absent.

## Solution Assessment

Yes. The paired PR's provider-local prompter change is the narrowest maintainable fix because the provider already receives `userCode`; changing core auth plumbing or putting the code into `runtime.log` would be broader and riskier.
