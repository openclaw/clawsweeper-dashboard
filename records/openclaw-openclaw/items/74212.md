## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote mode, and open PR #74258 is explicitly paired with this issue using closing syntax, so the issue should stay open until that PR or an equivalent provider-local fix lands.

## What This Changes

The issue reports that `openclaw models auth login --provider openai-codex` hides the OpenAI Codex device-pairing code in SSH/remote sessions, making the auth flow unrecoverable.

## Best Possible Solution

Land or replace #74258 with the provider-local fix: show `Code: ${userCode}` in the `prompter.note` terminal output for remote mode, keep persistent `ctx.runtime.log(...)` free of the device code, update the focused OpenAI provider tests for both invariants, and include the normal user-facing changelog entry.

## Reproduction Assessment

Yes. The high-confidence repro path is the existing OpenAI Codex device-code auth path with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The paired PR’s provider-local approach is the narrowest maintainable fix because the provider already receives `userCode`, and the core CLI already supplies the remote/SSH signal; changing core auth plumbing or logging the code persistently would be broader and riskier.
