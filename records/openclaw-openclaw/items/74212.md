## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote mode, and open PR #74258 is explicitly paired with this issue using closing syntax, so the issue should remain open until that PR or an equivalent provider-local fix lands.

## What This Changes

The issue asks OpenClaw to surface the OpenAI Codex device-pairing user code in the SSH-visible terminal prompt instead of replacing it with a placeholder during `models auth login --provider openai-codex`.

## Best Possible Solution

Land or replace #74258 with the provider-local fix: print the short-lived device code through the user-facing `prompter.note` terminal surface in remote mode, keep `ctx.runtime.log` free of the code, update the focused OpenAI provider tests for both invariants, and include the normal user-facing changelog entry.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the existing OpenAI Codex device-code auth flow with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-local because the provider already receives `userCode` and the CLI already supplies the remote/SSH signal; changing core auth plumbing or putting the code in persistent runtime logs would be broader and riskier.
