## Summary

Keep open. Current main and the latest release still mask the OpenAI Codex device-pairing code in remote mode, and open PR #74258 is already paired with this issue using closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Make `openclaw models auth login --provider openai-codex` surface the OpenAI Codex device-pairing code in SSH/remote sessions instead of masking it in the prompt.

## Best Possible Solution

Land or replace #74258 with a provider-local fix that includes `Code: ${userCode}` in the SSH-visible `prompter.note`, keeps `ctx.runtime.log(...)` URL-only, updates the OpenAI provider tests to assert both terminal visibility and log hygiene, and adds the normal user-facing changelog entry for the bug fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is the existing OpenAI Codex device-code auth path with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-local because the helper already supplies `userCode` and the CLI already marks SSH sessions as remote; changing the prompt note while preserving persistent log hygiene is the right boundary.
