## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote mode, the latest tagged release still has the same placeholder, and open PR #74258 already uses closing syntax for this exact bug.

## What This Changes

Request to surface the OpenAI Codex device-pairing user code during `models auth login --provider openai-codex` over SSH while keeping persistent logs free of the code.

## Best Possible Solution

Land or replace #74258 with the same narrow provider-local fix: print `Code: <userCode>` through the prompter note for remote mode, leave `ctx.runtime.log(...)` URL-only, and keep unit coverage for both terminal visibility and log hygiene.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction is to run the OpenAI Codex device-code auth method with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The proposed direction in #74258 is the narrowest maintainable fix because the provider helper already has the user code and the CLI already marks SSH as remote; this should stay in the OpenAI plugin instead of changing core auth or persistent logging.
