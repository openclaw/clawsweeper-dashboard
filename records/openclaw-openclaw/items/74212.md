## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote mode, and open PR #74258 uses closing syntax for the same bug, so this issue should remain open until that provider-local fix or an equivalent lands.

## What This Changes

The issue requests that OpenClaw's `openai-codex` device-pairing auth surface the user code in SSH-visible terminal output instead of replacing it with a placeholder.

## Best Possible Solution

Land or replace #74258 with the narrow OpenAI provider change: show the device code in the `prompter.note` terminal surface for remote sessions, keep persistent `runtime.log` free of the user code, update the focused provider tests for both invariants, and add the normal user-facing changelog entry.

## Reproduction Assessment

Yes. The high-confidence repro path is the existing provider unit path with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and explicitly asserts the real code is absent from that prompt.

## Solution Assessment

Yes. The narrowest maintainable solution is provider-local presentation logic because the provider already receives `userCode`; changing core auth plumbing or logging the code persistently would be broader and riskier.
