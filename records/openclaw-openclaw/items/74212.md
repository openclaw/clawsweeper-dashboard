## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote prompter output, and open PR #74258 uses closing syntax for the same bug, so this issue should remain open until that PR or an equivalent provider-local fix lands.

## What This Changes

The issue requests that `openclaw models auth login --provider openai-codex` surface the OAuth device-pairing code in SSH-visible terminal output instead of replacing it with a placeholder.

## Best Possible Solution

Review and land #74258, or replace it with the same provider-local fix: show the code on the interactive `prompter.note` surface for remote sessions, keep `runtime.log` free of the code, update the focused provider tests, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the existing OpenAI Codex provider unit path with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and asserts the real code is absent.

## Solution Assessment

Yes. The paired PR’s provider-local prompter change is the narrowest maintainable solution because the provider already receives `userCode`; changing core auth plumbing or putting the code in `runtime.log` would be broader and riskier.
