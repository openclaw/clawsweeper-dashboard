## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote sessions, and open PR #74258 uses closing syntax for the same bug, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that `openclaw models auth login --provider openai-codex` surface the OpenAI Codex device-pairing code in SSH/headless sessions instead of rendering the remote-mode placeholder.

## Best Possible Solution

Land #74258 or an equivalent OpenAI-plugin-local fix that shows `userCode` in the prompter for remote mode, preserves runtime-log redaction, and includes the focused regression tests plus changelog.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `loginOpenAICodexDeviceCode`, running the device-code auth method with `isRemote: true`, and observing that current main asserts the placeholder instead of `Code: CODE-12345`.

## Solution Assessment

Yes. The provider already receives `userCode`, so changing only the OpenAI Codex prompter note while preserving the runtime-log boundary is the narrowest maintainable solution.
