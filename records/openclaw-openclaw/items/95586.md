## Summary

Keep open: the issue reports a real auth/provider mismatch for Kimi in managed systemd Gateway mode, and current main still leaves the contract ambiguous between config env, generated service env, runtime process env, and `models status` diagnostics. The fix should not be guessed by automation because it affects credential materialization and status semantics.

## What This Changes

The issue requests that Kimi API-key auth reported by `models status --agent` match real systemd-managed Gateway runtime behavior when the key is configured via `env.KIMI_API_KEY`.

## Best Possible Solution

Define one managed-Gateway credential contract, then align systemd service-env rendering, Gateway startup env application, `models status --agent` wording/JSON, Kimi docs, and focused regression tests around that contract.

## Reproduction Assessment

Yes at source level, but not as a live systemd Kimi run in this review. Current code and tests show config env keys can be listed as managed without being rendered into the systemd runtime environment, while status reports env-backed auth from `process.env`.

## Solution Assessment

Unclear until maintainers choose the intended daemon credential contract. The safest fix is a coordinated service-env, runtime-env, status, docs, and test change rather than a Kimi-only patch.
