## Summary

Keep open: current main still has a real mismatch between config/env auth diagnostics and systemd Gateway runtime credential materialization for Kimi-style API-key providers. The issue is source-reproducible, but the repair needs maintainer direction because the safest fix could be status semantics, service-env rendering, docs/auth-profile guidance, or a combination.

## What This Changes

The issue reports that Kimi Coding API-key auth can appear available in `openclaw models status --agent <id>` while a systemd-managed Gateway lacks `KIMI_API_KEY` at runtime and fails over with an auth failure.

## Best Possible Solution

Define one managed-Gateway credential contract, then align systemd service-env rendering, `models status --agent` wording/JSON, Kimi docs, and regression tests so config/env auth is either truly usable by the Gateway or clearly reported as CLI/config-only with auth-profile or state-dir `.env` guidance.

## Reproduction Assessment

Yes at source level, but not through a live systemd Kimi run in this review. Current main's tests and service-env rendering show config env keys can be listed as managed without the value being injected into the systemd runtime environment.

## Solution Assessment

Unclear until maintainers choose the intended daemon credential contract. The safest fix is likely a coordinated service-env/status/docs change rather than a Kimi-only patch.
