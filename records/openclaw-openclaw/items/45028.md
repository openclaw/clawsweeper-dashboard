## Summary

Keep open. Current main still routes ordinary Control UI/WebChat sends as internal, non-delivering input and suppresses external route replies unless an explicit deliver route is set; the related mirroring implementation remains open and needs maintainer policy/proof review.

## What This Changes

Requests that replies typed in Control UI while viewing a Telegram-backed session deliver to the underlying Telegram route instead of remaining on WebChat.

## Best Possible Solution

Define a product-approved Control UI operator-reply or WebChat mirroring policy for externally backed sessions, then implement it narrowly with regression tests and live Telegram proof without weakening existing route-safety guards.

## Reproduction Assessment

Yes, at source level. The high-confidence path is Control UI `chat.send` with `deliver:false`, WebChat client origin resolution, and dispatch suppression unless `ExplicitDeliverRoute` is true.

## Solution Assessment

Unclear. The one-line fallback proposed in discussion conflicts with current route-safety tests, so the safer solution is a maintainer-approved operator-reply or mirroring design with focused tests and live Telegram proof.
