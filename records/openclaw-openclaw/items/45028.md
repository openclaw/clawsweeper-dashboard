## Summary

Keep open. Current main still treats ordinary Control UI/WebChat composer sends as internal WebChat input, suppresses route-reply delivery without an explicit deliver route, and preserves external session delivery metadata only to avoid overwrites; the related broad implementation for WebChat mirroring remains open, so the Telegram operator-reply behavior is not implemented on main.

## What This Changes

The issue requests that replies typed in the Control UI while viewing a Telegram-backed direct session be delivered back to the underlying Telegram chat instead of staying in WebChat.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions that preserves WebChat-native route safety while intentionally delivering selected Control UI replies to the stored external route.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run. The high-confidence path is Control UI chat.send with deliver:false, Gateway origin resolution to internal WebChat, and dispatch suppression unless ExplicitDeliverRoute is true.

## Solution Assessment

Unclear. The narrow fallback proposed in discussion conflicts with current WebChat route-safety tests, so the maintainable path is an explicit operator-reply or mirroring design reviewed against https://github.com/openclaw/openclaw/pull/46356.
