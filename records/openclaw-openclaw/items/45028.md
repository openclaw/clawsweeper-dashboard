## Summary

Keep open. Current main still routes ordinary Control UI/WebChat composer sends as internal WebChat traffic and suppresses external delivery unless explicit delivery metadata is set; the adjacent route-preservation fix prevents Telegram-style routes from being overwritten, but does not make normal Control UI replies deliver back to Telegram.

## What This Changes

Make Control UI replies sent while viewing a Telegram-backed session deliver to the underlying Telegram chat instead of staying on the internal WebChat surface.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions that preserves WebChat-native safety while allowing intentional delivery to the stored external route, with regression coverage for Telegram-style sessions, WebChat-native sessions, explicit delivery, and stale-route leakage.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run. The high-confidence path is Control UI `chat.send` with `deliver: false`, Gateway origin resolution to internal WebChat, and dispatch suppression of route-reply delivery unless `ExplicitDeliverRoute` is true; current tests assert this behavior.

## Solution Assessment

Unclear. The narrow fallback proposed in discussion is not the safest fix because it conflicts with existing WebChat route-safety behavior; the maintainable path is to review or split #46356 into an explicit operator-reply or mirroring design.
