## Summary

Keep open. Current main preserves stored external route metadata after WebChat/Control UI access, but ordinary Control UI composer replies still stay on internal WebChat and current docs/tests explicitly encode that route-safety behavior; the open #46356 implementation is broad enough that maintainer policy and split/landing judgment are still needed.

## What This Changes

Request that Control UI/WebChat replies in a Telegram-backed session preserve or explicitly target the underlying Telegram delivery route instead of staying on WebChat.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions that preserves WebChat-native route safety while intentionally delivering selected Control UI replies to the stored external route.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run. The high-confidence path is Control UI `chat.send` with `deliver: false`, Gateway origin resolution to internal WebChat, and dispatch suppression unless `ExplicitDeliverRoute` is true.

## Solution Assessment

Unclear. The narrow fallback proposed in discussion conflicts with current WebChat route-safety tests and docs, so the maintainable path is an explicit operator-reply/mirroring design reviewed against #46356.
