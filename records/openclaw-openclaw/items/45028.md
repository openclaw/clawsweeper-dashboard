## Summary

Keep open. Current main still sends ordinary Control UI composer messages as non-delivering WebChat input, preserves external delivery metadata only to avoid overwrites, and explicitly suppresses external route-reply delivery unless an explicit deliver route is set; the related WebChat mirroring implementation remains open at https://github.com/openclaw/openclaw/pull/46356.

## What This Changes

The issue requests that Control UI replies in a Telegram-backed session deliver back to the underlying Telegram conversation instead of staying on WebChat.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions, then implement it without weakening the existing route-safety guards for WebChat-native and unrelated channel-scoped sessions.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run. The high-confidence path is Control UI chat.send with deliver:false, gateway origin resolution to internal WebChat, and dispatch suppression unless ExplicitDeliverRoute is true.

## Solution Assessment

Unclear. The narrow fallback proposed in discussion conflicts with current route-safety tests, so the maintainable path is an explicit operator-reply or mirroring design reviewed against https://github.com/openclaw/openclaw/pull/46356.
