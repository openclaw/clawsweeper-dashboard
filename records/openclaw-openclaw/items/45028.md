## Summary

Keep open. Current main still treats ordinary Control UI composer sends as internal WebChat traffic and suppresses external route delivery unless explicit delivery metadata is set; the existing route-preservation code protects stored Telegram-style routes from being overwritten, but it does not make normal Control UI replies deliver back to Telegram. The directly related implementation candidate, #46356, is still open and unshipped.

## What This Changes

Make Control UI replies sent while viewing a Telegram-backed session deliver back to the underlying Telegram chat instead of staying on the internal WebChat surface.

## Best Possible Solution

Define an explicit operator-reply or WebChat mirroring policy for externally backed sessions that preserves WebChat-native safety while allowing intentional delivery to the stored external route, with regression coverage for Telegram-style sessions, WebChat-native sessions, explicit delivery, and stale-route leakage.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is Control UI `chat.send` with `deliver: false`, Gateway origin resolution to internal WebChat, and dispatch suppression of route-reply delivery unless `ExplicitDeliverRoute` is true; current tests assert this behavior.

## Solution Assessment

Unclear. The simple fallback proposed in discussion is not the safest fix because it conflicts with route-safety behavior already encoded in tests; the better path is to review or split #46356 into an explicit operator-reply or mirroring design.
