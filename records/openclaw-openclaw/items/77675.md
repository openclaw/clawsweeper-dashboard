## Label Justifications

- `P1`: The reported regression can block all embedded channel replies for operators using SecretRef-backed provider request headers in a model fallback chain.
- `impact:message-loss`: The issue causes embedded channel turns to fail before producing a reply, so channel messages can be effectively dropped into an error path.
- `impact:auth-provider`: The failure is in model provider routing/auth SecretRef handling across fallback candidates.
- `impact:security`: The documented workaround is to put the header credential in plaintext config, which weakens the intended SecretRef credential boundary.

## Summary

Keep open: the report describes a real embedded-channel/model-provider SecretRef failure mode, and current main has partial runtime-snapshot plumbing but not enough exact embedded fallback coverage or post-reporter fix history to close it as implemented.

## What This Changes

Fix embedded agent model fallback so SecretRef-backed `models.providers.<id>.request.headers` are resolved from the active runtime snapshot per provider and do not fail unrelated fallback providers.

## Best Possible Solution

Add a focused embedded reply/model-fallback regression for a fallback chain containing one provider with SecretRef-backed `request.headers`, then fix the first path that still passes source config into provider request resolution.

## Reproduction Assessment

No full high-confidence current-main live reproduction was run in this read-only review. Source inspection gives a focused repro target: if embedded fallback model resolution sees raw `request.headers` SecretRefs instead of the active runtime snapshot, current request sanitization throws the exact error reported.

## Solution Assessment

Unclear until the focused regression is added. The maintainable fix should stay in the existing runtime snapshot and embedded fallback config path, not add a new config knob or provider-specific fallback.
