## Summary

Keep open: current main still has a source-reproducible path where raw `models.providers.*.request.headers.*` SecretRefs can reach fail-closed provider request sanitization during embedded model resolution, and no linked fixing PR is merged for this model-provider request-header surface.

## What This Changes

The issue asks for model provider `request.headers` SecretRefs to resolve from the active runtime snapshot without breaking embedded agent fallback candidates that do not use that provider.

## Best Possible Solution

Keep the fail-closed SecretRef guard, but make embedded model resolution consume the active resolved runtime snapshot and avoid sanitizing unrelated provider request overrides before the selected provider is known.

## Reproduction Assessment

Yes, at source level. A source config with a raw `models.providers.ollama.request.headers` SecretRef can be passed through `runEmbeddedPiAgent` into `resolveModelAsync`, where inline model construction sanitizes every configured provider before selecting the fallback candidate.

## Solution Assessment

Yes, the narrow maintainable fix is to keep strict `assertSecretInputResolved` behavior while ensuring embedded model resolution uses resolved runtime config and scopes request sanitization to the relevant provider. Adding a config flag or weakening SecretRef validation would be the wrong direction.
