## Summary

Keep open: current main still has an eager model-provider request sanitization path that can throw on raw `request.headers` SecretRefs before provider-specific fallback selection, while docs and command targets treat that surface as supported. This is a narrow bug-fix candidate, not a cleanup close.

## What This Changes

Fix embedded agent runs so a SecretRef in one model provider's `request.headers` does not fail unrelated providers or fallback candidates.

## Best Possible Solution

Keep the supported model-provider SecretRef surface, use the resolved runtime snapshot for embedded reply runs, and make request override sanitization provider-candidate scoped with regression coverage for fallback chains.

## Reproduction Assessment

No high-confidence live reproduction was run, but source inspection gives a focused reproduction path: a raw `models.providers.<id>.request.headers` SecretRef reaches eager inline-provider sanitization and throws before provider selection. The issue's release/config/logs match that path closely.

## Solution Assessment

Yes, the requested direction is the right maintainable fix: do not let one provider's unresolved request SecretRef poison unrelated fallback candidates. The safer implementation is candidate-scoped/runtime-snapshot resolution while preserving fail-closed behavior for the selected provider.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77675. Current main supports `models.providers.*.request.headers.*` SecretRefs, but `buildInlineProviderModels` eagerly sanitizes every configured provider request and `provider-request-config` fails closed on raw SecretRefs, so one provider's `request.headers` SecretRef can poison unrelated fallback candidates in embedded agent runs. Implement the narrow fix in the embedded model/provider request path so unresolved raw request SecretRefs on non-selected providers do not fail other providers, while the selected provider still uses the active runtime snapshot and fails closed if its own required SecretRef is unresolved. Add regression coverage with a primary/fallback chain where only an `ollama` fallback has `request.headers` SecretRef and unrelated `openai-codex`/`github-copilot` candidates can resolve/build without throwing. Do not add product policy, new config options, bundled provider special cases, or plaintext secret persistence. Add a `CHANGELOG.md` fix entry because this is user-facing.
