## Label Justifications

- `P1`: The reported regression can block embedded channel agent replies for operators using SecretRef-backed provider request headers in model fallback chains.

## Summary

Keep open: current main has broad runtime-snapshot and provider request SecretRef plumbing, but the exact embedded-channel plus model-fallback plus `request.headers` SecretRef failure mode is still not proven fixed on current main, and the reporter’s latest human update says it still appears present.

## What This Changes

The issue requests that embedded agent fallback runs resolve `models.providers.*.request.headers` SecretRefs lazily per selected model provider instead of failing every fallback candidate from one unresolved raw SecretRef.

## Best Possible Solution

Add or run focused embedded-channel fallback proof with one fallback provider using SecretRef-backed `request.headers`, then fix the first path that still passes raw provider request SecretRefs into embedded model resolution while preserving strict fail-closed behavior for genuinely unresolved secrets.

## Reproduction Assessment

No, not yet with high confidence against current main. The report has concrete steps and logs, and source inspection confirms the supported SecretRef surface plus fail-closed raw SecretRef behavior, but the complete embedded fallback failure was not reproduced or covered by an exact current-main test.

## Solution Assessment

Yes on direction, but not yet on proof. The maintainable fix belongs in the existing runtime snapshot, command-secret resolution, and embedded provider fallback paths rather than a provider-specific workaround or new config option.
