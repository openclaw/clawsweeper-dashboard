## Summary

Keep open: current main still has a raw-config embedded model-resolution path that can throw the reported unresolved SecretRef before the selected provider is invoked, while the documented secrets contract expects runtime requests to read the active resolved snapshot.

## What This Changes

The issue requests that SecretRefs under `models.providers.<id>.request.headers` resolve safely for embedded agent model fallback chains without contaminating unrelated providers.

## Best Possible Solution

Align embedded model resolution with the active runtime config snapshot before provider request sanitization, while keeping the fail-closed raw SecretRef guard and adding regression coverage for fallback chains.

## Reproduction Assessment

Yes, at source level: a raw `models.providers.*.request.headers` SecretRef passed through embedded `resolveModelAsync` reaches `sanitizeConfiguredRequestString` and throws. A live channel run was not executed in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to make embedded model resolution consume the resolved runtime snapshot, similar to the existing skill-runtime pattern, rather than weakening SecretRef validation or adding lazy hot-path resolution.

## Repair Work Prompt

Fix the embedded-agent SecretRef leak reported in https://github.com/openclaw/openclaw/issues/77675. Current source shows embedded model resolution can pass raw `params.config` into `resolveModelAsync`, and provider request sanitization then throws on `models.providers.*.request.headers.*` SecretRefs. Expected boundary: embedded agent turns should resolve models from the active runtime config snapshot when caller config still contains raw SecretRefs, so a SecretRef on one fallback provider does not fail unrelated candidates. Keep `sanitizeConfiguredRequestString` fail-closed; do not remove SecretRef validation, do not store plaintext in source config, and do not introduce lazy hot-path SecretRef resolution. Add focused regression coverage for a config with primary `openai-codex/gpt-5.5`, fallback `github-copilot/gpt-5.5` and `ollama/...`, and `ollama.request.headers` as a SecretRef where the runtime snapshot has the resolved header.
