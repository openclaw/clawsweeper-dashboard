## Summary

Keep open: current main still has a source-reproducible embedded model-resolution path where raw `models.providers.*.request.headers.*` SecretRefs can reach the fail-closed request sanitizer before the selected provider is invoked. The earlier related channel SecretRef reports are useful context, but they do not close this model-provider request-header surface.

## What This Changes

The issue requests that embedded agent model fallback resolution use resolved runtime config so a provider-level `request.headers` SecretRef cannot fail unrelated providers.

## Best Possible Solution

Keep the fail-closed SecretRef guard, but ensure embedded model resolution consumes the active resolved runtime snapshot and avoids sanitizing unrelated provider request overrides during fallback candidate selection.

## Reproduction Assessment

Yes, at source level: a source config with `models.providers.ollama.request.headers` as a SecretRef can flow from `runEmbeddedPiAgent` into `resolveModelAsync`, where inline model construction sanitizes all providers before matching the selected fallback candidate. A live channel run was not executed in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable direction is to use the active runtime config snapshot for embedded model resolution and avoid cross-provider eager request sanitization, without weakening `assertSecretInputResolved`.
