## Summary

Keep open: the report is still actionable and not safely superseded. Current main has lower-level SecretRef runtime support, but I did not find exact embedded-channel plus model-fallback coverage for SecretRef-backed provider `request.headers`, and the related open PR is unmerged, conflicting, and aimed at generated `models.json` marker hardening rather than the full embedded fallback failure.

## What This Changes

The issue reports embedded channel agent turns failing across all model fallback candidates when any configured model provider has SecretRef-backed `models.providers.<id>.request.headers`.

## Best Possible Solution

Add a focused current-main embedded reply/model-fallback regression for a chain where only one provider has SecretRef-backed `request.headers`, then fix the first source-config or generated-marker leak while preserving strict credential failures for real request paths.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection plus reporter logs give a clear repro target: if raw provider `request.headers` SecretRefs reach provider request sanitization during embedded model fallback, current code throws the reported unresolved SecretRef error.

## Solution Assessment

Unclear until the focused regression is added. The maintainable fix should stay in the existing runtime snapshot, generated models marker, and embedded fallback config paths rather than adding a new provider-specific workaround.
