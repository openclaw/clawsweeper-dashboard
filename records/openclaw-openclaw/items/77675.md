## Label Justifications

- `P1`: The reported regression can block embedded channel replies for users with SecretRef-backed provider request headers in fallback chains.
- `impact:security`: The issue involves credential SecretRef handling and the reported workaround stores a credential-bearing header in plaintext config.
- `impact:message-loss`: The failure occurs before embedded agent output, so inbound channel messages can end without a real assistant reply.
- `impact:auth-provider`: The affected behavior is provider request SecretRef resolution across model fallback routing.

## Summary

Keep open: current main has general provider request SecretRef collection and queued-reply runtime config resolution, but I did not find exact current-main proof for the reported embedded-channel plus model-fallback plus `request.headers` SecretRef failure mode, and the reporter recently said it still appears to be an issue.

## What This Changes

The issue reports embedded agent model fallback turns failing when any provider in the fallback chain has a SecretRef-backed `request.headers` value, even for providers without such headers.

## Best Possible Solution

Add or run focused embedded-channel fallback proof with one fallback provider using SecretRef-backed `request.headers`, then fix the first path that still passes raw provider request SecretRefs into embedded model resolution.

## Reproduction Assessment

No full high-confidence current-main reproduction was run. The report has concrete steps and logs, and source inspection confirms the supported SecretRef surface plus fail-closed raw SecretRef behavior, but the complete embedded fallback failure was not reproduced on current main.

## Solution Assessment

Yes on direction, not yet on proof. The maintainable solution belongs in the existing runtime snapshot and queued embedded-reply execution-config path with focused regression coverage, not in a provider-specific workaround or new config option.
