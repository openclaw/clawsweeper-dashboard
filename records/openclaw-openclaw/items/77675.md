## Summary

Keep open: current main still has a source-reproducible model-resolution path where raw `models.providers.*.request.headers.*` SecretRefs can reach fail-closed request sanitization before the selected fallback provider is invoked. The related channel SecretRef fixes are useful context, but they do not close this model-provider request-header surface.

## What This Changes

Embedded agent turns should resolve model-provider `request.headers` SecretRefs from the active runtime snapshot and avoid letting one provider's raw SecretRef break unrelated fallback providers.

## Best Possible Solution

Keep the fail-closed SecretRef guard, but ensure embedded model resolution receives the active resolved runtime snapshot and does not sanitize unrelated provider request overrides while selecting a fallback candidate.

## Reproduction Assessment

Yes, at source level: a config with `models.providers.ollama.request.headers` as a SecretRef can reach `resolveModelAsync`, where inline model construction sanitizes every configured provider before matching the requested fallback candidate. I did not execute a live channel turn in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to use the active runtime config snapshot for embedded model resolution and avoid cross-provider eager request sanitization, without weakening `assertSecretInputResolved`. A broader config option or plaintext fallback would be the wrong direction.
