- `P1`: The report describes silent loss of agent-to-agent messages in production when session write-lock contention hits an existing 60s timeout.

- `maturity:stable`: The issue affects Transcript Persistence, whose taxonomy coverage includes session write locks and durability and is marked LTS-supported in the maturity scorecard.

## Summary

Keep open: current main still has a source-level path where `sessions_send` can return accepted/pending while later A2A or write-lock failures are only logged, and the overlapping lock-delivery tracker at https://github.com/openclaw/openclaw/issues/86538 plus its open PR do not specifically resolve the direct caller-visible `sessions_send` acknowledgement gap.

## What This Changes

Report or reject `sessions_send` delivery when the target session write lock later times out instead of returning accepted/pending and silently dropping the message.

## Best Possible Solution

Define the `sessions_send` delivery contract for post-accept write-lock failures, then implement caller-visible failure or retryable durable delivery status aligned with the existing session write-lock and delivery-outbox work.

## Reproduction Assessment

Yes at source level: current main can return accepted/pending before async A2A delivery is proven, the A2A flow catches and logs later failures, and session write-lock acquisition still throws after the documented timeout. I did not run a live timing-sensitive lock-contention scenario in this read-only review.

## Solution Assessment

Unclear as the final implementation shape: surfacing the failure is necessary, but maintainers need to choose between waiting before acceptance, returning a retryable rejection/timeout, or adding durable post-accept delivery status. A narrow fix should avoid creating a second, inconsistent delivery contract.
