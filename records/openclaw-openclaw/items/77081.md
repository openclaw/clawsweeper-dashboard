## Summary

Keep open: current main still has a source-visible path where Telegram MEDIA image replies fail during image optimization before any raw image or document delivery fallback can run.

## What This Changes

Telegram MEDIA image replies should deliver when optional sharp is missing or unresolved, instead of falling back to “No response generated.”

## Best Possible Solution

Make outbound media loading tolerate a missing optimizer by sending valid in-limit images raw or as documents, while keeping sharp only for optional resizing/format conversion and adding a regression test.

## Reproduction Assessment

Yes, by source inspection: a Telegram final reply containing a local image MEDIA path on an install without resolvable sharp reaches `loadWebMedia`, attempts image optimization, throws from the sharp-backed resize path, and then the Telegram dispatcher emits the fallback text. The issue’s v2026.5.2 logs match that path exactly.

## Solution Assessment

No, current main is not the best solution because it still lets optional optimization abort basic media delivery. The narrow maintainable fix is a raw/document fallback for images that are already within channel limits, with tests around the missing-sharp case.
