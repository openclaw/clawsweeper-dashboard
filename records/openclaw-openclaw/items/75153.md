## Summary

Keep open. Current main still lacks standalone channel lifecycle CLI commands and lacks Gateway RPC methods for `channels.stop` or `channels.restart`; the existing `channels.start` RPC is only wired through login reconciliation, so the reporter's non-destructive recovery path remains missing.

## What This Changes

Expose `openclaw channels start`, `stop`, and `restart` so operators can recover a stopped but still-linked channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Add generic, admin-scoped channel lifecycle controls: CLI commands for start/stop/restart, typed Gateway protocol schemas and validators, server-method handlers backed by the existing channel manager, focused tests, docs, and a user-facing changelog entry. Keep the longer-term health-monitor retry/backoff policy separate.

## Reproduction Assessment

Yes for the source-backed gap: current main exposes only `channels.start`, only login reconciliation calls it from the CLI, and no standalone lifecycle CLI or stop/restart RPC exists. The live network-blip sequence itself was not replayed.

## Solution Assessment

Yes. An additive generic CLI/RPC wrapper around existing channel manager start/stop hooks is the narrowest maintainable fix; WhatsApp-specific retry-budget behavior should be handled separately.
