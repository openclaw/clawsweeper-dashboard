## Summary

Keep open. Current main still lacks the requested standalone `openclaw channels start`, `stop`, and `restart` recovery commands, and the provided GitHub context shows open PR #75560 explicitly closes this issue, so this should remain open until that PR or an equivalent implementation lands.

## What This Changes

The issue requests standalone `openclaw channels start`, `stop`, and `restart` commands to recover an existing linked channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Land PR #75560 or an equivalent focused change that adds admin-scoped channel lifecycle CLI commands and Gateway RPC coverage for start, stop, and restart while preserving auth state. Keep automatic retry/backoff policy changes as a separate follow-up if maintainers want that broader behavior.

## Reproduction Assessment

Yes for the actionable gap: source and docs inspection shows no standalone `openclaw channels start`, `stop`, or `restart` commands on current main. The transient WhatsApp network failure path itself was not live-reproduced in this review.

## Solution Assessment

Yes. An additive operator lifecycle surface is the narrowest maintainable fix because it reuses existing channel runtime primitives and avoids mixing WhatsApp-specific retry policy into the same change.
