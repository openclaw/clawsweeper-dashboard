## Summary

Keep open. Current main still has the operator recovery gap: the Gateway exposes `channels.start` and `channels.stop`, but the channel CLI and docs do not expose standalone lifecycle commands, `channels.restart` is not registered as a Gateway method, and open PR #75560 is the active unmerged implementation that explicitly closes this issue.

## What This Changes

The issue requests standalone `openclaw channels start`, `stop`, and `restart` commands to recover an existing linked channel account without QR re-pairing or container restart.

## Best Possible Solution

Review and land PR #75560, or an equivalent focused implementation, adding admin-scoped channel lifecycle CLI commands and Gateway RPC coverage while preserving existing auth state; keep automatic retry/backoff policy changes as a separate follow-up.

## Reproduction Assessment

Yes for the actionable gap: source and docs inspection on current main shows no standalone `openclaw channels start`, `stop`, or `restart` commands. The transient WhatsApp network failure path itself was not live-reproduced in this read-only review.

## Solution Assessment

Yes. An additive operator lifecycle surface over the existing generic channel runtime primitives is the narrowest maintainable fix, and it avoids mixing a broader health-monitor retry-policy redesign into this issue.
