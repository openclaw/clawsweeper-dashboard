## Summary

Keep open: current main and the latest release still lack the requested standalone channel lifecycle CLI commands, and https://github.com/openclaw/openclaw/pull/93028 is an open closing PR rather than a merged fix.

## What This Changes

Request `openclaw channels start`, `stop`, and `restart` commands that wrap existing Gateway lifecycle RPCs so operators can recover a linked wedged channel without QR re-pairing or restarting the container.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93028 once proof is added, or replace it with an equivalent narrow CLI wrapper over the existing Gateway start/stop methods while leaving automatic retry/backoff policy to the separate lifecycle tracker.

## Reproduction Assessment

Yes for the missing command surface: current main has Gateway `channels.start` and `channels.stop` RPCs but no standalone `openclaw channels start|stop|restart` commands. The original live WhatsApp DNS-blip wedge was not reproduced here.

## Solution Assessment

Yes, the narrow maintainable direction is to expose the existing Gateway lifecycle RPCs through operator CLI commands. The broader automatic recovery/backoff behavior should stay separate from this CLI recovery surface.
