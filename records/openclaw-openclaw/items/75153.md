## Summary

Keep open. Current main still lacks the requested standalone channel lifecycle CLI/RPC recovery path, and open PR #75560 explicitly closes this issue, so this should stay open until that PR or an equivalent implementation is merged or rejected.

## What This Changes

Request standalone `openclaw channels start`, `stop`, and `restart` commands so operators can recover an existing linked channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Merge PR #75560 or an equivalent focused implementation that adds admin-scoped channel lifecycle CLI commands and Gateway RPCs backed by the existing generic channel manager primitives, with docs, protocol artifacts, tests, and changelog. Keep automatic retry/backoff or health-event policy as a separate follow-up if maintainers want that broader behavior.

## Reproduction Assessment

Yes for the actionable gap: current main can be inspected directly and lacks standalone `openclaw channels start`, `stop`, or `restart`, while `channels.start` exists only as a gateway method used indirectly. The full transient WhatsApp network-wedge scenario was not live-reproduced here.

## Solution Assessment

Yes. An additive CLI/RPC lifecycle surface is the narrowest maintainable fix for operator recovery because it reuses generic channel lifecycle primitives and does not bundle WhatsApp-specific retry policy changes into the same work.
