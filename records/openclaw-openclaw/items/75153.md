## Summary

Keep open. Current main still lacks the requested standalone channel lifecycle CLI and stop/restart Gateway RPC surface, and the provided GitHub context shows open PR #75560 explicitly closing this issue, so cleanup should wait for that PR or an equivalent implementation to merge or be rejected.

## What This Changes

Expose `openclaw channels start`, `stop`, and `restart` so operators can recover an existing linked channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Land PR #75560 or an equivalent focused implementation that adds admin-scoped channel lifecycle CLI commands and Gateway RPCs using the existing generic channel manager primitives, while keeping automatic retry/backoff or health-event policy as a separate follow-up.

## Reproduction Assessment

Yes for the actionable gap: current main can be inspected directly and lacks standalone `openclaw channels start`, `stop`, or `restart`, while `channels.start` exists only as a Gateway method used indirectly after login. The full transient WhatsApp network-wedge scenario was not live-reproduced here.

## Solution Assessment

Yes. An additive CLI/RPC lifecycle surface is the narrowest maintainable fix for operator recovery because it reuses generic channel lifecycle primitives and avoids mixing WhatsApp-specific retry policy changes into the same work.
