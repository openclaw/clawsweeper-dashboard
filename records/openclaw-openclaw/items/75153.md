## Summary

Keep open. Current main still lacks the requested standalone channel lifecycle CLI/RPC recovery path, and open PR #75560 explicitly closes this issue, so cleanup should wait until that PR or an equivalent implementation is merged or rejected.

## What This Changes

Expose `openclaw channels start`, `stop`, and `restart` for linked channel accounts so operators can recover wedged channel runtimes without QR re-pairing or restarting the container.

## Best Possible Solution

Review and land PR #75560 or an equivalent focused implementation that adds admin-scoped channel lifecycle CLI commands and Gateway RPCs backed by existing generic channel manager primitives, with protocol schemas/artifacts, docs, tests, and changelog. Keep automatic retry/backoff or high-severity health-event policy as a separate follow-up if maintainers want that broader behavior.

## Reproduction Assessment

Yes for the actionable gap: current main inspection shows no standalone `openclaw channels start`, `stop`, or `restart` commands, while `channels.start` exists only as a Gateway method used indirectly after login. The full transient WhatsApp network failure was not live-reproduced here.

## Solution Assessment

Yes. An additive CLI/RPC lifecycle surface is the narrowest maintainable fix for non-destructive operator recovery because it reuses generic channel lifecycle primitives and avoids mixing broader retry/backoff policy into the same change.
