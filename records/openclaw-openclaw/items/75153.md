## Summary

Keep open. Current main still lacks standalone `openclaw channels start`, `stop`, and `restart` recovery commands, and the provided context shows open PR #75560 explicitly closes this issue, so cleanup should wait for that PR or an equivalent implementation to land.

## What This Changes

Expose `openclaw channels start`, `stop`, and `restart` so operators can recover an existing linked channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Land PR #75560 or an equivalent focused change that adds admin-scoped channel lifecycle CLI commands backed by Gateway start/stop/restart behavior, preserves existing auth state, updates protocol/docs/tests, and keeps automatic retry policy changes as a separate follow-up.

## Reproduction Assessment

Yes for the actionable gap: source and docs inspection on current main show no standalone `openclaw channels start`, `stop`, or `restart` commands. The transient WhatsApp network failure itself was not reproduced live in this read-only review.

## Solution Assessment

Yes. An additive operator lifecycle CLI/RPC surface is the narrowest maintainable fix because it reuses existing channel runtime primitives and avoids mixing WhatsApp-specific recovery policy into the same change.
