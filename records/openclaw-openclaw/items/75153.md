## Summary

Keep open. Current main still lacks the requested standalone channel lifecycle CLI/RPC recovery path, and open PR #75560 explicitly closes this issue, so this should stay open until that PR or an equivalent implementation is merged or rejected.

## What This Changes

Request to expose channel start, stop, and restart operations through the CLI so operators can recover a stopped linked channel without QR re-pairing, account reconfiguration, or container restart.

## Best Possible Solution

Land PR #75560 or an equivalent focused implementation that adds admin-scoped channel lifecycle CLI commands and Gateway RPCs backed by the existing generic channel-manager primitives. Keep automatic retry/backoff and health-event policy changes as separate follow-up work if maintainers want that broader behavior.

## Reproduction Assessment

Yes for the actionable gap: current main inspection shows no standalone `openclaw channels start`, `stop`, or `restart`, while `channels.start` exists only as a Gateway method used indirectly by login reconciliation. The full transient WhatsApp network failure scenario was not live-reproduced here.

## Solution Assessment

Yes. An additive CLI/RPC lifecycle surface is the narrowest maintainable solution for the reported operator recovery problem because it reuses existing generic channel lifecycle primitives and avoids bundling WhatsApp-specific retry policy into the same change.
