## Summary

Keep open: current `main` still has workspace-scoped snapshot rejection paths that can force fallback plugin metadata scans. The setter-only root-cause theory is incomplete, but the reported performance regression remains actionable and is not implemented on `main`.

## What This Changes

The issue requests a fix for plugin metadata snapshot reuse misses that can rebuild manifest metadata on hot Gateway/model/tool paths.

## Best Possible Solution

Preserve workspace-plugin correctness while adding a bounded workspace-aware metadata reuse path, or explicit compatible snapshot threading, so hot Gateway/model/tool paths do not rebuild manifest metadata repeatedly; add regression coverage for default, non-default, and no-workspace callers.

## Reproduction Assessment

Yes, at source level: current code and tests show that workspace-scoped snapshots are rejected for omitted or nonmatching `workspaceDir`, and hot catalog paths can then fall back to metadata loading. I did not establish a live strace reproduction in this read-only sweep.

## Solution Assessment

No: passing `workspaceDir` only to the setter is not the best fix because the stored lookup table already carries the default workspace. The safer solution needs a maintainer-approved workspace-aware reuse design or explicit snapshot threading that does not hide workspace-local plugins.
