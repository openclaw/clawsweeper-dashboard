## Summary

Keep open: current `main` still has workspace-scoped snapshot rejection and fallback metadata-scan paths relevant to the reported Gateway/model-catalog stalls. The issue's setter-only root-cause claim is incomplete because current code fingerprints with `snapshot.workspaceDir`, but the broader performance regression remains actionable.

## What This Changes

The issue reports a regression where workspace-scoped plugin metadata snapshot checks miss the Gateway startup snapshot and force repeated manifest/index rebuilds on model/catalog hot paths.

## Best Possible Solution

Preserve workspace-plugin correctness while adding bounded workspace-aware metadata reuse or explicit snapshot threading through the hot Gateway/model/tool paths, with regression coverage for default workspace, explicit workspace, omitted workspace, and changed plugin roots.

## Reproduction Assessment

Yes, at source level: current `main` has a clear path where a workspace-scoped current snapshot can be rejected and `loadManifestModelCatalog` can fall back to metadata loading. I did not establish the reporter's live syscall profile in this read-only sweep.

## Solution Assessment

No, the setter-only Option A is not sufficient because current code already uses the snapshot's workspace when the setter option is omitted. The safer fix is an explicit workspace-aware reuse design or direct snapshot threading that does not hide workspace-local plugin changes.
