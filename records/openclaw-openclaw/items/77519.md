## Summary

Keep open: current main still has source-level paths where workspace-scoped Gateway/plugin metadata callers can miss the current snapshot and fall back to rebuilding plugin metadata. The issue's setter-only explanation is incomplete, but the reported performance regression remains actionable and is not implemented on main.

## What This Changes

The issue asks OpenClaw to preserve Gateway plugin metadata snapshot reuse across workspace-aware model/tool hot paths so repeated dispatch/RPC calls do not rescan bundled plugin manifests.

## Best Possible Solution

Add a bounded workspace-aware metadata reuse path, or explicit snapshot threading, that keeps default and per-agent hot paths off repeated manifest scans without bypassing workspace-plugin freshness rules; cover default, non-default, and no-workspace callers.

## Reproduction Assessment

Yes, at source level: non-default workspace callers can pass a workspaceDir that differs from the single default-workspace startup snapshot, causing getCurrentPluginMetadataSnapshot to return undefined and fall back to loadPluginMetadataSnapshot. I did not run a live strace or tests because this sweep was read-only.

## Solution Assessment

No, the closed two-line setter-only PR is not clearly the best fix because the stored lookup table already carries snapshot.workspaceDir. The safer solution needs maintainer-approved workspace-aware reuse or explicit snapshot threading that preserves workspace plugin correctness.
