## Summary

Keep open. Current main still lacks explicit-URL/non-loopback follow retry, but this PR is not merge-ready because its follow-mode catch path retries every `fetchLogs` failure, including fatal auth, policy, config, and protocol errors; the protected `maintainer` label also prevents cleanup closure.

## What This Changes

The PR adds backoff/reconnect notices for `openclaw logs --follow`, updates logs docs and changelog, and adds two CLI tests for explicit-URL retry behavior.

## Best Possible Solution

Keep the localized CLI follow retry, but route it through a transient-only classifier for close/timeout/startup cases and preserve the existing fatal path for auth, pairing, policy, config, and protocol errors.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, `openclaw logs --follow --url ...` exits through the catch block after a `fetchLogs` error, while the PR changes that same path to retry. The review finding is source-reproducible by making `fetchLogs` throw a fatal non-transient error such as a 1008 close or malformed `logs.tail` response.

## Solution Assessment

No, not as written. Retrying inside `src/cli/logs-cli.ts` is the right narrow boundary, but the PR needs a transient classifier and fatal-error regression coverage before it is the maintainable fix.
