## Summary

Keep open for maintainer handling. The PR is relevant to the remaining explicit-URL `logs --follow` disconnect gap, but it carries the protected `maintainer` label and the patch currently retries every follow-mode fetch failure rather than only transient Gateway transport failures.

## What This Changes

The PR adds `openclaw logs --follow` retry/backoff notices for Gateway failures, updates the logs docs and changelog, and adds two CLI tests for retry behavior.

## Best Possible Solution

Keep the localized CLI follow-loop retry approach, but gate retries through a clear transient transport classifier for close/timeout cases, preserve the implicit-loopback file fallback, keep auth/pairing/policy/protocol errors fatal, and validate with focused logs CLI tests plus the changed gate before merge.

## Reproduction Assessment

Yes. The target bug has a code-level reproduction by making `callGatewayFromCli` reject during `openclaw logs --follow --url ...`; current main reaches `process.exit(1)`, while the PR changes that path to retry. The review finding is also reproducible by throwing a non-transient error, such as a malformed `logs.tail` response or close code 1008, and observing the PR head retry instead of failing.

## Solution Assessment

No, not as written. Retrying inside `src/cli/logs-cli.ts` is the right boundary, but the current patch needs a transient-only classifier and regression coverage for fatal close/protocol cases before it is the narrow maintainable fix.
