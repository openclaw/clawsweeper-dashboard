## Summary

Keep open: the PR addresses the right UI bug, but the Sessions-table half still formats only `row.thinkingDefault`, while real `sessions.list` lightweight rows can leave that field unset and carry the effective default on the list defaults.

## What This Changes

The PR adds shared thinking-label helpers, applies inherited/override labels to chat and Sessions thinking selectors, and updates UI tests, thinking docs, and changelog.

## Best Possible Solution

Expose or pass the effective inherited thinking default to both UI surfaces, preferably per session row, while preserving existing clear-override `sessions.patch` values and provider option labels.

## Reproduction Assessment

Yes. Source inspection and current tests show real `sessions.list` rows can have `defaults.thinkingDefault` populated while `row.thinkingDefault` is undefined, and the PR's Sessions label path reads only the row field.

## Solution Assessment

No. The chat-side approach is narrow, but the Sessions-side fix needs the effective default from the real RPC payload or gateway row enrichment before the linked UI bug is actually fixed.

## Review Findings

- **[P2] Use the effective default for Sessions inherited labels:** `ui/src/ui/views/sessions.ts:98`
