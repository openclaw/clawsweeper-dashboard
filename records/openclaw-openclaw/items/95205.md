- `P1`: The linked stale Anthropic OAuth user-agent bug breaks real Claude subscription-backed provider runs, and this PR touches the active auth-provider fix path.

- `merge-risk: 🚨 auth-provider`: The changed header is part of Anthropic OAuth bearer authentication and directly affects provider routing for Claude subscription-backed requests.
- `merge-risk: 🚨 availability`: A synchronous `claude --version` lookup with a timeout on the auth path can stall or fail Anthropic request/client construction.

## Summary

This PR remains useful and focused, but the latest diff still moves Claude version resolution into the Anthropic OAuth request/header construction path via a synchronous ambient `claude --version` lookup, so it should not merge until that request-path execution/latency risk is moved to a setup-owned or cached runtime boundary.

## What This Changes

The PR replaces hardcoded Anthropic OAuth `claude-cli/2.1.75` user-agent values in the legacy provider and native messages transport with a shared resolver that reads the installed Claude Code version, adds tests, and adds a repro script.

## Best Possible Solution

Keep the shared two-call-site fix, but resolve the Claude Code version at an owned setup/runtime boundary with bounded caching and clear diagnostics before OAuth requests are built.

## Reproduction Assessment

Yes from source: current main sends `claude-cli/2.1.75` from both Anthropic OAuth builders, and the linked issue gives a concrete Windows 11/OpenClaw 2026.6.8 reproduction with Claude CLI 2.1.177. I did not run a live Anthropic OAuth request in this read-only review.

## Solution Assessment

No: replacing the stale constant in both OAuth builders is the right scope, but doing `claude --version` synchronously from header construction is not the best boundary. The safer fix is to prepare or cache the version outside the request path and feed both builders the prepared value.
