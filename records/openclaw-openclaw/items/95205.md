## Summary

Keep open: this PR fixes the right stale Anthropic OAuth header surfaces, but the current implementation resolves the Claude version by running a hardcoded PATH command from OAuth header construction, leaving compatibility, auth-provider, availability, and security-sensitive merge risk.

## What This Changes

The branch replaces stale hardcoded `claude-cli/2.1.75` user-agent values in both Anthropic OAuth builders with a shared runtime resolver, tests, and a repro script.

## Best Possible Solution

Keep the two-call-site stale-header fix, but resolve and cache the Claude Code version from the owned CLI backend or auth setup boundary using the configured command, then pass the prepared value into both OAuth builders.

## Reproduction Assessment

Yes, from source: current main and v2026.6.8 still send `claude-cli/2.1.75` from both Anthropic OAuth builders, and the linked report gives a concrete failing setup. I did not run a live Anthropic OAuth request in this read-only review.

## Solution Assessment

No: replacing the stale value in both builders is the right scope, but resolving it by running a hardcoded PATH command during header construction is not the best boundary. The safer fix uses the configured CLI backend command and a cached setup/provider-initialization value.
