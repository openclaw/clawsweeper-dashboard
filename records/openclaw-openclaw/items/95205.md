## Summary

Keep open: this is a focused fix for a real Anthropic OAuth regression, but the current patch still resolves the Claude Code version by running a PATH command from OAuth header construction, which leaves a compatibility, auth-provider, availability, and security-sensitive blocker before merge.

## What This Changes

The PR replaces two hardcoded `claude-cli/2.1.75` Anthropic OAuth user-agent headers with a shared runtime Claude Code version resolver, adds helper/transport tests, and adds a repro script.

## Best Possible Solution

Keep the two-call-site stale-header fix, but resolve and cache the Claude Code version at an owned setup/provider-initialization boundary with clear diagnostics before OAuth requests are constructed.

## Reproduction Assessment

Yes from source: current main sends `claude-cli/2.1.75` from both Anthropic OAuth builders, and the linked issue gives a concrete Windows 11/OpenClaw v2026.6.8 reproduction with Claude CLI 2.1.177. I did not run a live Anthropic OAuth request in this read-only review.

## Solution Assessment

No: replacing the stale constant in both OAuth builders is the right scope, but doing synchronous `claude --version` discovery from header construction is not the best boundary. The safer fix is to prepare or cache the version outside the request path and feed both builders the prepared value.
