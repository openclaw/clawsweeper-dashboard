## Summary

Keep this PR open and request changes. Current main still lacks the proposed centralized production guard, so the work is not obsolete, but the PR branch is not merge-ready because it applies the guard to only one debug read while documenting several flags as covered, and it mixes unrelated Control UI/session and Docker smoke-test changes into a security-motivated patch.

## What This Changes

The PR adds an agents debug-env helper that disables debug flags under `NODE_ENV=production`, routes model-catalog timing through it, and also changes Control UI session-limit typing plus an install-smoke timeout assertion and CSS/app-render formatting churn.

## Best Possible Solution

Define the production debug-flag policy first, place the helper in a shared boundary that all affected packages can legally import, migrate every in-scope debug read or document deliberate exceptions, add focused production/non-production regression tests, and split the session-limit and Docker changes into separate PRs.

## Reproduction Assessment

Yes for the PR defects: code inspection shows `NODE_ENV=production` would only suppress the model-catalog timing read through the new helper while the other listed debug flags still read env directly. The session-limit regression is also reproducible from the unchanged string initializer plus the patched direct controller forwarding into an integer-only gateway schema.

## Solution Assessment

No. Centralizing one call site is not the narrowest maintainable security fix because it creates partial coverage; the safer path is a shared helper plus complete migration or explicit exceptions, with unrelated UI/Docker work separated.
