## Summary

Keep open. The PR is still useful source work, but it does not meet its stated security goal: the new production guard is only wired into model-catalog timing while the listed debug flags still have direct reads elsewhere. The existing review discussion already identified this incomplete migration, and the author acknowledged feedback, so this needs maintainer review/request-changes rather than cleanup closure.

## What This Changes

The PR adds an agents debug-env helper that disables listed debug flags when `NODE_ENV=production`, routes model-catalog timing through it, and also changes Control UI session-limit typing/input behavior plus install-smoke timeout/CSS/render formatting churn.

## Best Possible Solution

Put the debug-flag helper in a shared boundary that core, plugins, and packages can legally consume; migrate every in-scope debug read or document explicit exceptions; add focused production/non-production tests; and split the Control UI session-limit work from the security hardening before merge.

## Reproduction Assessment

Yes for the PR defects. Code inspection of the PR diff plus current main shows `NODE_ENV=production` would only suppress the model-catalog timing read, and the sessions path can still start from a string `sessionsFilterLimit` while the gateway schema requires an integer `limit`.

## Solution Assessment

No. A helper under `src/agents` applied to one call site is not the narrowest maintainable security fix because it cannot cover the listed plugin/package flags without crossing boundaries, and the unrelated session-limit refactor introduces a separate invalid-params regression.

## Review Findings
