## Summary

Keep open: this recent external security-hardening PR has useful intent, but it is not mergeable as-is because the production debug guard only covers one call site, the sessions-limit refactor can send a string limit to the gateway, and contributor real-behavior proof is missing.

## What This Changes

The PR adds an agents debug-env helper for production-blocking debug flags, switches model-catalog to use it, changes Control UI sessions limit typing/input handling, reformats UI/CSS, and updates an install-smoke timeout assertion.

## Best Possible Solution

Define an agreed shared debug-flag boundary or explicit owner-local helpers, migrate or intentionally exempt every in-scope debug read with focused coverage, and split unrelated sessions UI changes from the security hardening.

## Reproduction Assessment

Yes for the PR defects by source inspection. The PR head routes only model-catalog through the new production guard while listed debug flags remain direct, and the sessions.list path can still receive a string limit from PR-head state sources.

## Solution Assessment

No. An agents-local helper applied to one call site is not the narrow maintainable fix for a cross-core/plugin debug policy, and the unrelated sessions/UI changes should not ride along with security hardening.
