## Summary

Keep this PR open: it is aimed at a real status/update bug, but the patch only changes manager detection and still leaves the reported npm-shrinkwrap package install showing a missing lockfile downstream.

## What This Changes

The PR makes package-manager detection prefer lock files over package.json and adds npm-shrinkwrap handling plus detector tests.

## Best Possible Solution

Complete the fix by making status/dependency checks carry or resolve the actual npm-shrinkwrap lock path for published npm packages, with an update-status regression and real npm package/global-install proof before merge.

## Reproduction Assessment

Yes. From source, the PR makes npm-shrinkwrap.json select npm, while the downstream npm deps marker still checks only package-lock.json and therefore still reaches the reported missing-lockfile path for shrinkwrap-only installs.

## Solution Assessment

No. Reordering detector priority is a useful part of the fix, but the maintainable solution also needs the status/deps lockfile path to recognize npm-shrinkwrap.json and a regression at the checkUpdateStatus surface.
