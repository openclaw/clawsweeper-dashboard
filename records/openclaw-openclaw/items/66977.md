## Summary

Keep open. Current main and the latest shipped release still route builtin memory through `node:sqlite` and sqlite-vec loadable-extension APIs; the fallback behavior mitigates total failure but does not resolve the reported macOS sqlite-vec native loadability gap.

## What This Changes

The issue asks OpenClaw to make sqlite-vec load successfully for builtin memory on macOS despite `node:sqlite` extension-loading limitations, or to document a supported workaround.

## Best Possible Solution

Keep this as the canonical sqlite-vec/node:sqlite tracker until maintainers either support a validated macOS SQLite loading strategy for builtin memory, or explicitly document the limitation and supported workaround while preserving degraded fallback behavior.

## Reproduction Assessment

Yes, with an affected macOS Node runtime: the issue provides concrete `node:sqlite` and `openclaw memory status` steps, an independent commenter reported the same load-extension failure class, and current source still uses the same extension-loading boundary. I did not reproduce the macOS compile-option failure on this Linux checkout.

## Solution Assessment

No current-main solution is sufficient for the requested native sqlite-vec acceleration. The in-process cosine fallback is useful degradation, but the maintainable fix needs a maintainer decision between supported runtime selection, a different SQLite loading strategy, or explicit limitation/workaround docs.
