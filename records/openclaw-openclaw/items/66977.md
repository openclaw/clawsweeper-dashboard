## Summary

Keep open. Current main still runs builtin memory on `node:sqlite` and loads sqlite-vec through loadable-extension APIs; the implemented fallback avoids total failure but does not resolve the reported macOS native sqlite-vec loadability gap.

## What This Changes

The issue asks OpenClaw to make sqlite-vec vector search load and function on macOS when the running `node:sqlite` build cannot enable loadable extensions.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers either support a macOS SQLite loading strategy that can load sqlite-vec or explicitly document the limitation and supported workaround while preserving degraded fallback behavior.

## Reproduction Assessment

Yes, for an affected macOS Node runtime: the issue gives concrete `node:sqlite` and `openclaw memory status` steps, and an independent commenter reported the same load-extension failure class. This Linux checkout did not reproduce the macOS compile-option failure directly, but current source still uses the same extension-loading boundary.

## Solution Assessment

No current-main solution is sufficient for the requested native acceleration. The fallback is useful degradation, but the maintainable fix needs a maintainer decision between supported macOS runtime selection, a different SQLite binding/loading strategy, or explicit limitation/workaround docs.
