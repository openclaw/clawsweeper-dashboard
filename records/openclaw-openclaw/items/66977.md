## Summary

Keep open. Current main and the latest release still use `node:sqlite` plus sqlite-vec loadable-extension APIs, so fallback behavior mitigates recall but does not resolve the reported macOS native sqlite-vec loading gap.

## What This Changes

The issue requests a fix or documented support boundary for macOS sqlite-vec loading when Node's built-in SQLite cannot load extensions.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers validate a supported macOS runtime, change the builtin memory SQLite loading strategy, or explicitly document native acceleration as unsupported with a clear workaround.

## Reproduction Assessment

Yes, at source/report level: the issue gives concrete macOS `node:sqlite` and `openclaw memory status` steps, and current main still depends on the same extension-loading boundary. I did not run a live affected macOS repro in this checkout.

## Solution Assessment

No current-main solution is sufficient for the requested native sqlite-vec acceleration. The in-process cosine fallback is useful mitigation, but a durable fix needs a maintainer decision on runtime support, backend strategy, or documented support boundaries.
