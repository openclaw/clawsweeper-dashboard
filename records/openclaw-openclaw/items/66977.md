## Summary

Keep open. Current main and the latest release still route builtin memory through node:sqlite plus sqlite-vec loadable-extension APIs, so the in-process cosine fallback mitigates recall but does not resolve the reported macOS native sqlite-vec loading gap.

## What This Changes

The issue reports that macOS node:sqlite builds cannot load sqlite-vec, leaving native vector acceleration unavailable and forcing degraded memory search.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers either validate a supported macOS runtime, change the builtin memory SQLite loading strategy, or document native acceleration as unsupported with a clear workaround.

## Reproduction Assessment

Yes, at source/report level: the issue gives concrete macOS node:sqlite and OpenClaw status steps, and current main still depends on the same node:sqlite extension-loading boundary. I did not run a live current-main macOS repro in this checkout.

## Solution Assessment

No current-main solution is sufficient for the requested native sqlite-vec acceleration. The in-process cosine fallback is useful mitigation, but a durable fix needs a maintainer decision on runtime support, backend strategy, or documented support boundaries.
