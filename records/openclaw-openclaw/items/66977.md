## Summary

Keep open. Current main and v2026.5.7 still route builtin memory through `node:sqlite` and sqlite-vec loadable-extension APIs, so the newer platform-package fallback and in-process cosine fallback mitigate symptoms but do not settle the reported macOS native sqlite-vec loadability gap.

## What This Changes

The issue asks OpenClaw to restore sqlite-vec vector acceleration on macOS Node runtimes whose `node:sqlite` cannot enable loadable extensions.

## Best Possible Solution

Keep this as the canonical sqlite-vec/native-load tracker until maintainers either validate a supported macOS Node/runtime combination, change builtin memory to a SQLite loading strategy that supports sqlite-vec on macOS, or document the limitation and supported workaround.

## Reproduction Assessment

Yes, with the reported affected macOS Node runtime: the issue gives concrete `node:sqlite` and `openclaw memory status` steps, and current source still depends on the same extension-loading boundary. I did not reproduce the macOS failure locally, and the local Linux Node v24.14.1 probe succeeds with `allowExtension:true`.

## Solution Assessment

No current-main solution is sufficient for the requested native sqlite-vec acceleration. The platform-variant loader and in-process cosine fallback are useful mitigation, but a durable fix needs a maintainer runtime/backend/support-boundary decision plus macOS validation.
