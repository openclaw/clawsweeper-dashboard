## Summary

Keep open. Current main still routes builtin memory through node:sqlite plus sqlite-vec extension-loading APIs and only degrades to fallback search when loading fails; I did not find a macOS-capable alternate SQLite runtime, supported Homebrew workaround, or documentation that resolves the reported Node/macOS extension-loading failure.

## What This Changes

The issue asks OpenClaw to make sqlite-vec vector acceleration work on macOS when node:sqlite cannot enable extension loading, or to document a supported workaround.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers either support a macOS runtime/binding that can load sqlite-vec, or explicitly document the limitation and supported workaround while preserving the degraded fallback path.

## Reproduction Assessment

Yes for a macOS maintainer machine: the issue gives concrete node:sqlite and OpenClaw memory-status commands, and a commenter independently hit a macOS vector-load failure with a PATH-sensitive SQLite workaround. This Linux checkout verifies the shared extension-loading code path but cannot directly reproduce the macOS compile-option failure.

## Solution Assessment

No current-main solution is sufficient. The fallback path prevents total memory failure, but the best fix needs a maintainer decision between supported macOS runtime selection, an alternate SQLite binding/loading strategy, or explicit docs/workaround support.
