## Summary

Keep open. Current main still uses the same node:sqlite plus sqlite-vec loadable-extension path and only degrades to fallback search when extension loading fails; I did not find a macOS runtime repair, alternate SQLite binding, or supported documented workaround that resolves the reported native sqlite-vec load failure.

## What This Changes

The issue requests macOS support or documented guidance for sqlite-vec vector acceleration when node:sqlite cannot enable loadable extensions.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers either support a macOS SQLite loading strategy that can load sqlite-vec or explicitly document the limitation and supported workaround while preserving degraded fallback behavior.

## Reproduction Assessment

Yes for a macOS maintainer machine: the issue gives concrete node:sqlite and OpenClaw memory-status commands, and an independent commenter reported the same vector-load failure class. This Linux checkout verifies the current source still uses that extension-loading path, but it cannot directly reproduce the macOS compile-option failure.

## Solution Assessment

No current-main solution is sufficient. The existing fallback prevents total memory failure, but the maintainable fix needs a maintainer decision between supported macOS runtime selection, a different SQLite binding/loading strategy, or explicit limitation/workaround docs.
