## Summary

Keep open. Current main and the latest release still route builtin memory through node:sqlite plus sqlite-vec loadable-extension APIs, then degrade to fallback search when loading fails; I did not find a macOS-capable SQLite runtime, alternate binding, or supported builtin-memory workaround. The related closed #74435/#74401 context is useful, but it does not supersede this broader macOS extension-loading tracker, and the commenter’s Homebrew PATH workaround applies to QMD/PATH-driven setups rather than the current builtin node:sqlite loader.

## What This Changes

The issue requests a supported macOS path for sqlite-vec vector search when node:sqlite cannot load SQLite extensions.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers either validate a supported macOS runtime where builtin sqlite-vec loads, change builtin memory to a SQLite loading strategy that supports sqlite-vec on macOS, or explicitly document the macOS limitation and supported workaround while preserving the degraded fallback path.

## Reproduction Assessment

Yes for a macOS maintainer machine: the issue gives concrete node:sqlite and openclaw memory status commands, and a commenter corroborated a macOS vector-load failure with PATH-sensitive SQLite behavior. This Linux checkout verifies the shared node:sqlite extension-loading gate but not the macOS compile option itself.

## Solution Assessment

No current best fix is present on main. The existing fallback is useful degradation, but the durable solution needs a maintainer decision between supported macOS runtime selection, an alternate SQLite binding/loading strategy, or explicit docs/workaround support.
