## Summary

Keep open. Current main has improved fallback and packaging around sqlite-vec, but it still depends on the active `node:sqlite` runtime being able to enable/load a SQLite extension and does not yet provide a confirmed macOS runtime repair or a documented supported workaround for this report.

## What This Changes

Requests macOS sqlite-vec vector-search support when Node's built-in SQLite cannot load the sqlite-vec extension.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec tracker until maintainers either validate a supported macOS Node/runtime combination, change builtin memory to a macOS-capable SQLite loading strategy, or document the limitation and supported workaround while preserving degraded fallback behavior.

## Reproduction Assessment

No, not with high confidence in this review environment. The report gives concrete macOS commands and another user reported the same failure class, but this Linux checkout only verifies that current main still uses the sqlite extension-loading path and cannot prove the macOS OMIT_LOAD_EXTENSION condition.

## Solution Assessment

No current-main solution is sufficient for native macOS sqlite-vec acceleration. The fallback keeps memory usable, but the durable fix needs a maintainer choice between runtime support, an alternate SQLite binding/loading strategy, or explicit docs for the supported limitation/workaround.
