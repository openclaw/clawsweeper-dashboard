## Summary

Keep open. Current main still routes builtin memory through `node:sqlite` and the sqlite-vec loadable-extension path, then degrades to fallback search when loading fails; I did not find a macOS-specific runtime repair or supported Homebrew workaround in code/docs, and related #74435 was closed unmerged.

## What This Changes

The issue requests fixing or documenting macOS sqlite-vec load failures in builtin memory when `node:sqlite` cannot enable extension loading.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec tracker until maintainers validate the supported macOS Node/SQLite matrix and either implement a supported loading strategy or document the limitation and workaround explicitly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for a macOS Node setup matching the report: the issue gives direct `node:sqlite` and OpenClaw memory probes, while this Linux checkout verifies the same source path but cannot live-reproduce the macOS binary behavior.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution is sufficient; fallback search is useful degraded behavior, but the maintainable resolution needs a supported macOS loading strategy or explicit documented limitation after macOS validation.
