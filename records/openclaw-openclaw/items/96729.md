## Summary

Keep open: the PR targets a real MCP argument bug, but the current patch is not merge-ready because it drops every top-level null after validation, which can erase schema-meaningful nullable MCP arguments, and the proof still does not show the changed OpenClaw MCP runtime path in a real affected setup.

## What This Changes

The branch changes `packages/llm-core/src/validation.ts` to prefer exact primitive union matches and changes bundle-MCP materialization to strip top-level null-valued tool arguments before `callTool`, with unit tests for both paths.

## Best Possible Solution

Make MCP null handling schema-aware: preserve explicit and required null by default, omit only schema-proven optional null-as-absent fields, keep the validation change deliberately covered, and require redacted real OpenClaw MCP proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection and reporter evidence show current main can coerce nullable string-union nulls to empty strings before bundle-MCP dispatch, then forward record inputs to the MCP SDK.

## Solution Assessment

No. Preserving null through validation is plausible, but unconditional top-level null stripping is too broad; the safer fix is schema-aware optional-null omission or preserving required nullable values by default.
