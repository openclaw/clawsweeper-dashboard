## Summary

Keep open: the PR touches the right MCP dispatch boundary, but the implementation is overbroad because it drops every top-level JSON null even though the linked bug report says explicit null is valid for the observed server and only an empty string fails; contributor proof is also mock-only.

## What This Changes

The PR strips top-level null-valued properties from materialized bundle-MCP tool inputs before calling the session MCP runtime and adds a mocked materialization regression test.

## Best Possible Solution

Trace the null-to-empty-string path and preserve explicit JSON null unless maintainers intentionally add a schema-aware omission rule for optional default-null fields, with focused regression tests and redacted real MCP proof.

## Reproduction Assessment

No high-confidence live reproduction from this review. Source inspection shows current main forwards null-containing records through the MCP tool path and the linked issue has release logs, but the raw wire payload and real awslabs EKS MCP run were not reproduced here.

## Solution Assessment

No. Unconditional null stripping is a plausible workaround, but the safer fix is to preserve explicit JSON null or make omission narrowly schema-aware after proving where null becomes an empty string.
