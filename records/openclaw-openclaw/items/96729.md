## Summary

Keep open: the patch is not merge-ready because it strips nulls after validated MCP tool arguments reach `execute(...)`, which likely misses the reported null-to-empty-string path and can change valid explicit-null MCP calls.

## What This Changes

The branch strips top-level `null` values before materialized MCP tools call `SessionMcpRuntime.callTool(...)` and adds a direct materialized-tool unit test for that filtering.

## Best Possible Solution

Fix the argument validation/coercion boundary so nullable-string MCP schema values keep their intended semantics, or omit only schema-proven optional default-null fields, with validated-path regression coverage and real MCP proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The issue is source-reproducible with medium confidence because current main validates tool args before `execute(...)` and forwards record inputs to MCP, but I did not run the awslabs EKS MCP server or capture the wire payload.

## Solution Assessment

No. The PR is a plausible mitigation at the wrong boundary; the maintainable fix should preserve or intentionally omit nullable MCP arguments before validation turns them into a different value.
