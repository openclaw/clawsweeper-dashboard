## Summary

Keep open: this addresses a real MCP argument bug, but the patch filters after tool-argument validation, where the reported null-to-empty-string conversion can already have happened, and it broadly changes explicit-null MCP semantics without real MCP proof.

## What This Changes

The PR filters top-level null properties out of materialized bundle-MCP tool inputs before `runtime.callTool(...)` and adds a mocked materialization regression test.

## Best Possible Solution

Preserve explicit JSON null through validation and MCP dispatch, and only omit fields when schema/default semantics explicitly say absence is desired, with real MCP proof.

## Reproduction Assessment

No high-confidence live reproduction from this review. Source inspection supports a medium-confidence path: current validation can coerce null to an empty string before MCP execution, but I did not run the awslabs EKS MCP server or capture a raw wire payload.

## Solution Assessment

No. The PR is a plausible workaround, but the safer fix is to preserve nullable values in validation/coercion or apply schema-aware omission before MCP dispatch rather than unconditionally stripping nulls after validation.
