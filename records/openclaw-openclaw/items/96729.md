## Summary

Keep open: this is a relevant candidate fix for the canonical MCP optional-null bug, but the branch is not merge-ready because the runtime filter drops every top-level null regardless of the MCP tool schema, and the after-fix proof is a standalone script rather than full OpenClaw MCP behavior.

## What This Changes

The branch changes JSON-schema union coercion to preserve exact null matches, strips top-level null-valued bundle-MCP tool arguments before dispatch, and adds unit coverage for those paths.

## Best Possible Solution

Keep the validation-layer null preservation if maintainers accept the broader union semantics, but make MCP omission schema-aware so required or meaningful nulls are preserved, then add regression coverage and redacted real OpenClaw MCP proof before merge.

## Reproduction Assessment

Yes, with medium confidence from source inspection and reporter evidence: current main can coerce nullable string-union nulls to empty strings before bundle-MCP dispatch, and then forwards record inputs to the MCP SDK. I did not run the awslabs EKS MCP server or capture a live OpenClaw wire payload.

## Solution Assessment

No. Preserving null through validation is plausible, but unconditional top-level null stripping is too broad; the safer fix is schema-aware optional-null omission or preserving required nullable values by default.
