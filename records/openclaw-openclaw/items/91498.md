## Summary

Keep open: current main and the latest release still provide provider-wide OpenAI auth ordering plus session/request profile overrides, but not the requested model alias/runtime-scoped default auth profile.

## What This Changes

The issue requests a declarative model alias, model reference, or runtime scoped auth-profile preference so two OpenAI-backed aliases can use different OpenAI auth profiles under the same provider key.

## Best Possible Solution

Define one canonical model-scoped auth-profile preference contract, keep auth.order.<provider> as fallback, and update schema, docs, status/probe output, and regression tests around that precedence.

## Reproduction Assessment

Yes. Source inspection shows the strict model config schema lacks an auth-profile selector and auth resolution currently uses provider-wide order plus session/request overrides.

## Solution Assessment

Unclear until maintainers choose the permanent contract. A model-scoped preference with provider-wide fallback is plausible, but precedence against session overrides and Codex/OpenClaw runtime selection needs product review.
