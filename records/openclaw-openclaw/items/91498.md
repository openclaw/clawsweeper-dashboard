## Summary

Keep open: current main and the latest release still only support provider-wide OpenAI auth ordering plus session/request auth-profile overrides, not the requested model alias/runtime-scoped default auth profile. This remains a valid feature request, but it needs maintainer product direction because it adds auth/config precedence semantics.

## What This Changes

The issue requests a declarative model alias, model ref, or runtime-scoped way to prefer different auth profiles under the same provider auth key.

## Best Possible Solution

Define one canonical model-scoped auth-profile preference contract, keep auth.order.<provider> as the fallback, and update schema, docs, status/probe output, and focused tests around the chosen precedence.

## Reproduction Assessment

Yes, at source level. Current main’s strict model-entry schema lacks an auth-profile selector, and runtime auth selection falls back to session overrides or provider-wide order.

## Solution Assessment

Unclear until maintainers choose the permanent contract. A model-scoped preference with provider-wide fallback is plausible, but precedence against session overrides and existing auth.order behavior needs product review.
