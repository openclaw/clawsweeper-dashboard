## What This Changes

The branch threads `audioAsVoice` through iMessage media sending, adds `--audio` to `imsg send-attachment`, marks voice receipts, advertises iMessage CAF/MP3 TTS voice capability, and adds focused tests.

## Best Possible Solution

Before merge, route `audioAsVoice` media with reply targets through `imsg send-attachment --audio --reply-to` with regression coverage, or keep iMessage TTS voice capability unadvertised until that path works.

## Reproduction Assessment

Yes from source, not from a live run by this review. Current reply plumbing defaults to threaded replies, iMessage inbound supplies a message id, and the PR only applies `--audio` inside the non-threaded attachment branch.

## Solution Assessment

No. The owner boundary and `imsg --audio` primitive are right, but the implementation is too narrow because normal iMessage auto-replies carry `replyToId` and bypass the new `send-attachment --audio` path.
