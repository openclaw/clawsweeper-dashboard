## Summary

Keep open: current main and v2026.5.5 still rewrite `openai-codex/*` agent routes to `openai/*` with PI, while the provider/runtime docs define `openai-codex/*` as the PI OAuth route and `openai/*` plus PI as the API-key route.

## What This Changes

The issue asks `openclaw update` and `openclaw doctor --non-interactive --fix` to preserve OAuth `openai-codex/*` agent model refs instead of rewriting them to `openai/*` API-key routes.

## Best Possible Solution

Align doctor/update with the documented provider/runtime split: preserve PI OAuth `openai-codex/*` routes unless the repair can safely migrate the route to native Codex runtime, and update tests, docs, and changelog together.

## Reproduction Assessment

Yes, from source inspection: `openclaw update` invokes `doctor --non-interactive --fix`, and the doctor repair path rewrites `openai-codex/gpt-5.5` to `openai/gpt-5.5` with `agentRuntime.id: "pi"`. I did not run the mutating command because this review is read-only.

## Solution Assessment

No, the current repair is not the best way to solve the issue because it can move an intentional PI OAuth route onto the documented API-key route. The narrow fix is to stop the PI fallback rewrite and keep the native-Codex migration only when the Codex runtime is actually selected and OAuth-ready.
