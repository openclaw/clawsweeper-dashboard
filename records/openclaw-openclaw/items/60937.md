## Summary

Keep this PR open for maintainer review. Current main still does not add baseline headers in the standalone `startCanvasHost` server, but the proposed `X-Frame-Options: DENY` changes the frame policy for canvas content that OpenClaw currently treats as embeddable, so the patch is not safe to merge as written.

## What This Changes

The PR adds `X-Content-Type-Options`, `Referrer-Policy`, and `X-Frame-Options` headers to the standalone canvas host server and adds a test covering 200 and 404 responses.

## Best Possible Solution

Land a rebased version that applies a canvas-safe standalone baseline before routing, covering success, not-found, and error responses with `X-Content-Type-Options` and `Referrer-Policy`, while leaving frame/CSP restrictions out unless maintainers explicitly change the canvas embedding policy.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to start `startCanvasHost` on current main and fetch a served canvas URL plus a missing URL; the handler path does not set the proposed baseline headers before routing. I did not run live tests because this review was required to remain read-only.

## Solution Assessment

No. The useful hardening is narrow, but `X-Frame-Options: DENY` conflicts with the current iframe-based canvas/A2UI embedding model; the safer solution is a canvas-specific baseline without frame denial or an explicit maintainer-approved policy change.
