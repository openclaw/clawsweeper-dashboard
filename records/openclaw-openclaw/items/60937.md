## Summary

Keep open: the standalone canvas-host security hardening is still useful, but this PR should not merge as-is because it adds frame denial to embeddable canvas content, targets the pre-refactor source path, and lacks real behavior proof.

## What This Changes

The branch sets `X-Content-Type-Options`, `Referrer-Policy`, and `X-Frame-Options: DENY` in the old standalone canvas-host server path and adds a 200/404 header test.

## Best Possible Solution

Add frame-compatible standalone canvas-host hardening in `extensions/canvas/src/host`, with focused 200/404/error coverage and no frame/CSP restrictions unless maintainers intentionally change the embedding policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: start `startCanvasHost` from `extensions/canvas/src/host/server.ts` and fetch a served canvas URL plus a missing path to observe missing baseline headers. I did not run the live server because this review is read-only.

## Solution Assessment

No, not as proposed. The narrow maintainable fix is to add the frame-compatible baseline to the current Canvas plugin host and tests, while dropping `X-Frame-Options: DENY` unless a maintainer/security decision changes canvas embedding policy.
