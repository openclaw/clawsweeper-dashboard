## Summary

Keep open: the standalone canvas host still lacks the requested baseline headers on current main, but this PR should not merge as-is because `X-Frame-Options: DENY` conflicts with the current iframe-based canvas/A2UI embedding model and the gateway’s frame-compatible baseline policy.

## What This Changes

The PR adds `X-Content-Type-Options`, `Referrer-Policy`, and `X-Frame-Options: DENY` to standalone canvas-host responses, plus a 200/404 header test.

## Best Possible Solution

Add standalone canvas-host hardening with the same frame-compatible baseline as gateway responses, including focused 200/404/error coverage, and omit frame/CSP restrictions unless maintainers explicitly change the canvas embedding policy.

## Reproduction Assessment

Yes. By source inspection, starting `startCanvasHost` on current main and fetching both `CANVAS_HOST_PATH/` and a missing canvas path would exercise response paths that do not set baseline security headers; I did not run live tests because this review must remain read-only.

## Solution Assessment

No, not as proposed. The standalone hardening is the right narrow direction, but `X-Frame-Options: DENY` should be dropped or explicitly approved as a product/security policy change because current canvas content is designed to load in sandboxed iframes.
