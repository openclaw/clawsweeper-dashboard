## Summary

Keep this PR open, but do not merge it as-is. The standalone canvas host still lacks the requested baseline headers on current main, so the hardening direction is valid; the blocker is that the PR adds `X-Frame-Options: DENY` on a canvas/A2UI surface that current gateway policy, UI code, tests, and docs treat as iframe-embeddable.

## What This Changes

The PR adds `X-Content-Type-Options`, `Referrer-Policy`, and `X-Frame-Options: DENY` to standalone canvas host responses, plus a canvas-host test for 200 and 404 responses.

## Best Possible Solution

Add standalone canvas-host hardening with the same frame-compatible baseline policy as gateway responses: set `X-Content-Type-Options` and `Referrer-Policy` early for success, not-found, and error responses, consider `Permissions-Policy` if maintainers want standalone parity, and leave frame/CSP restrictions out unless the canvas embedding policy is explicitly changed.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to start `startCanvasHost` on current main and fetch both a served canvas URL and a missing URL; the request handler currently has no early baseline header setup before those response paths. I did not run live tests because this review was required to remain read-only.

## Solution Assessment

No, this PR is not the best way as proposed. The narrow standalone hardening is right, but `X-Frame-Options: DENY` is unsafe for the current embeddable canvas/A2UI model; the safer fix is a frame-compatible header set plus focused tests.
