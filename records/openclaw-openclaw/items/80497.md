## Summary

Keep open: the feature is source-reproducible and the provided logs are good, but the PR still exposes plugin-forgeable trusted model telemetry through a public diagnostic-runtime emitter, so it needs API/security judgment or a provenance fix before merge.

## What This Changes

The branch adds a diagnostic-runtime/compat `onModelDiagnosticEvent` SDK export, tests, and a changelog entry to expose five trusted `model.*` lifecycle diagnostics to plugins.

## Best Possible Solution

Expose a narrow model diagnostic stream with non-forgeable runtime provenance, or explicitly document and name it as non-integrity telemetry in the public SDK docs.

## Reproduction Assessment

Yes. Current main source shows `onDiagnosticEvent` filters trusted events while the embedded runner emits the relevant model lifecycle through trusted diagnostic paths, and the PR body supplies deployed after-fix logs showing delivery.

## Solution Assessment

No, not yet. The narrow opt-in listener is the right shape, but trusting `metadata.trusted` is not enough while plugin code can publicly call `emitTrustedDiagnosticEvent`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use non-forgeable provenance for model telemetry:** `src/infra/diagnostic-events.ts:934-941`
  - body: `onModelDiagnosticEvent` treats any allowlisted event with `metadata.trusted` as runtime model telemetry, but `openclaw/plugin-sdk/diagnostic-runtime` still publicly exports `emitTrustedDiagnosticEvent`. A plugin can emit `model.usage` or `model.call.completed` through that public subpath and have the new stream deliver it as trusted, corrupting downstream cost or attribution consumers. Please add non-forgeable core provenance, or explicitly scope this SDK contract as non-integrity-safe telemetry before exposing it.
  - confidence: 0.91
- **[P3] Document the diagnostic-runtime listener:** `src/plugin-sdk/diagnostic-runtime.ts:14`
  - body: This adds a public diagnostic-runtime SDK export, but the plugin SDK docs still only describe the subpath generically and the migration table only lists diagnostic gating helpers. Please document `onModelDiagnosticEvent`, its allowed event types, and its provenance limits so plugin authors can safely discover and use the new API.
  - confidence: 0.78
