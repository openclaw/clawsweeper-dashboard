## Summary

Keep open: the reported picker pollution is still source-reproducible on current main, but this PR needs real behavior proof and should not merge with the current string-heuristic internal-session filter.

## What This Changes

The PR adds UI-side filtering in `resolveSessionOptionGroups`, a `spawnDepth` UI row field, and focused tests to hide inactive internal sessions while preserving the active internal selection.

## Best Possible Solution

Add gateway-produced internal visibility/classification for runtime sessions, propagate it through the typed session row contract, and have the normal chat picker hide or filter inactive internals while admin/debug surfaces can still show them explicitly.

## Reproduction Assessment

Yes, from source inspection. Current main still has no internal visibility/default-hide check in `resolveSessionOptionGroups`; it filters active-agent rows, global/unknown rows, and cron rows, then shows or nests remaining runtime rows.

## Solution Assessment

No, not as submitted. UI-side mitigation is a reasonable interim surface, but the PR should use explicit visibility metadata or parsed/shared key helpers and needs maintainer direction now that current main groups subagents visibly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

  - body: `isInternalSessionKey` returns `normalized.startsWith("node-")` for every non-agent key, so documented default `acp:<uuid>` sessions are not hidden even though the PR says ACP rows are internal. Use the shared ACP-key helper or explicit visibility metadata so bare ACP rows do not remain in the picker.
  - confidence: 0.88

  - body: The `includes(":subagent:")` and `includes(":acp:")` checks hide any flexible key containing those segments, not only internal agent-rest prefixes. Use parsed `agent:<id>` rest-prefix semantics or explicit visibility metadata so legitimate session keys with those path segments remain selectable.
  - confidence: 0.84
