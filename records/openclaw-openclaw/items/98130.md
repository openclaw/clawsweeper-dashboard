## Label Justifications

- `P2`: This is a normal-priority hardening fix for a local control-plane socket OOM/hang path with limited blast radius.
- `merge-risk: 🚨 compatibility`: The new 1 MiB requester cap can reject existing valid macOS exec-host responses because that producer currently serializes uncapped stdout/stderr into one JSONL line.

## Summary

Keep open: the OOM guard is useful, but the patch is not merge-ready because it enforces a 1 MiB client-side protocol cap before all current exec-host producers are aligned to that cap.

## What This Changes

Adds a 1 MiB response buffer cap to `requestJsonlSocket` and a regression test for oversized newline-free socket responses.

## Best Possible Solution

Bound pending newline-free socket data while preserving a documented exec-host response contract: either cap/truncate all producers under the requester limit or set the requester limit to a size all valid producers already satisfy.

## Reproduction Assessment

Yes. Current main clearly buffers newline-free socket data until timeout or close, and the PR body provides a real Unix-socket before/after proof, though I did not run that proof in this read-only review.

## Solution Assessment

No. The shared requester is the right place to bound pathological pending data, but this implementation rejects valid oversized producer responses before the exec-host response contract is aligned.
