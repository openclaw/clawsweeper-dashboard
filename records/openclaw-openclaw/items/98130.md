Keep open: the requester OOM guard is useful, but the current PR still has blocking compatibility and correctness issues around silent macOS exec-output truncation and cap ordering before complete-line parsing.

## What This Changes

The branch bounds `requestJsonlSocket` response buffering, adds a regression test, and caps macOS exec-host stdout/stderr before encoding JSONL responses.

## Best Possible Solution

Keep the requester OOM guard, enforce the limit only on the remaining unterminated pending line after complete-line parsing, and land macOS output limits only with an accepted/documented exec-output contract plus real macOS proof.

## Reproduction Assessment

Yes, source-reproducible. Current `main` and `v2026.6.11` accumulate JSONL response data until newline, close, or timeout, and the PR body includes a Unix-socket negative control; this read-only review did not rerun it.

No, not yet. A requester-side guard is the right layer for hostile newline-free peers, but this patch enforces it before complete-line parsing and adds a silent macOS exec-output truncation contract without accepted proof.
