## Summary

Keep open: the requester OOM guard is useful, but current live head is not merge-ready because it still caps the whole accumulated JSONL buffer before parsing complete lines and silently tail-truncates macOS exec-host stdout/stderr without an accepted/proven output contract.

## What This Changes

The PR adds a 16 MiB buffer cap to `requestJsonlSocket`, a regression test for newline-free oversized responses, and a 1 MiB macOS exec-host stdout/stderr limiter with Swift coverage.

## Best Possible Solution

Keep the requester OOM guard, enforce the size limit only on the remaining unterminated pending line after complete-line parsing, and land macOS output limits only with an accepted/documented exec-output contract plus redacted real macOS proof.

## Reproduction Assessment

Yes, source-reproducible. Current main and v2026.6.11 accumulate JSONL response data until newline, close, or timeout, and the PR body includes a Unix-socket negative control; this read-only review did not rerun it.

## Solution Assessment

No, not as written. A requester-side guard is the right layer for hostile newline-free peers, but this patch enforces it before complete-line parsing and adds a silent macOS exec-output truncation contract without accepted real behavior proof.
