## Summary

Keep open: the OOM guard is useful and the real-socket proof covers the hostile newline-free case, but the submitted shared 1 MiB cap can make valid macOS exec-host command responses resolve as null because the current producer still serializes unbounded stdout/stderr into one JSONL line.

## What This Changes

The PR adds a 1 MiB pending-buffer cap to the shared Unix-domain JSONL socket requester and a regression test for newline-free oversized responses.

## Best Possible Solution

Keep the requester OOM guard, but land it only with a verified shared JSONL response budget: either cap/truncate exec-host producer output below the requester limit with tests/proof, or choose an owner-approved cap that preserves supported outputs.

## Reproduction Assessment

Yes, at source level. Current main buffers newline-free socket data until newline, close, or timeout, and the PR body includes a real Unix-socket negative control showing the old path waits the full timeout.

## Solution Assessment

No, not as submitted. The requester cap is the right layer for hostile newline-free peers, but the shared 1 MiB limit needs a producer/output contract because the current macOS exec-host path can emit larger valid responses.
