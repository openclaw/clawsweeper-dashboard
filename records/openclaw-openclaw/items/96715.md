## What This Changes

The PR adds an auto-reply session guard that treats an implicitly stale daily-reset entry as reusable when the same session id has an active reply run, plus a regression test for that rollover race.

## Best Possible Solution

Land a focused guard that reuses the active same-session stale entry only for implicit rollover, with regression coverage and redacted after-fix proof, while keeping explicit resets and idle-session rollover semantics intact.

## Reproduction Assessment

Yes at source level: current main computes daily staleness in session initialization and can archive the prior transcript before active-run admission. I did not run a full live provider rollover repro in this read-only review.

## Solution Assessment

Yes: deferring only implicit stale rollover at the initializer blocks the destructive archive before the later queue/admission logic and keeps explicit reset behavior unchanged. A scheduler-only fix would not cover this inbound archive path.
