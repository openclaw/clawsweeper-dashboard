## Summary

Keep open for maintainer landing judgment: the patch is focused, source-backed, and has sufficient exact-head proof, with no discrete actionable defect found in this review. The remaining question is whether maintainers accept the session-state invariant that an active reply run keeps its current session through implicit daily or idle rollover.

## What This Changes

The PR makes auto-reply session initialization reuse a stale existing session when a matching reply run is active, and adds regression coverage proving the live transcript is not archived during that active run.

## Best Possible Solution

Land this focused guard after maintainer review accepts active-run session preservation, keep https://github.com/openclaw/openclaw/issues/96546 open until a fix merges, and recheck against https://github.com/openclaw/openclaw/pull/84009 if it lands first.

Yes at source level: current main evaluates daily staleness and captures the previous transcript for archive before the later active-run admission path. I did not run a full wall-clock provider reproduction in this read-only review.

## Solution Assessment

Yes: reusing the active stale session entry is narrower and more coherent than an archive-only guard that still mints a replacement session. A scheduler-only change would not cover this inbound auto-reply path.
