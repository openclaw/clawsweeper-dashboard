## Summary

Keep open. Active Memory's concrete prompt-bloat path is now narrowed, and fresh local onboarding uses a narrower profile, but current runtime still treats missing tool policy, omitted embedded allowlists, and the `full` profile as unrestricted. Changing that default remains a product/security compatibility decision.

## What This Changes

The issue requests least-privilege agent tool defaults so agents and embedded subagents receive only explicitly configured tools instead of every available tool.

## Best Possible Solution

Define a maintainer-approved narrow-by-default tool-policy contract with explicit unrestricted opt-in, migration or doctor guidance, docs/schema/help updates, and regression coverage across agents, subagents, embedded runs, plugins/MCP, cron, and channels.

## Reproduction Assessment

Yes, as a source reproduction: current main still allows tools when policy is missing and builds all embedded tool families when `toolsAllow` is omitted. The Active Memory-specific example is narrowed now, so the remaining reproduction is the broader default-permission contract.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. Caller-specific allowlists like Active Memory's are useful mitigations, but they are not a repo-wide least-privilege default.
