## Summary

Keep open. Current main has narrowed the Active Memory example, but the broader runtime contract is still permissive when no tool policy or embedded `toolsAllow` is configured; changing that default is a product/security compatibility decision, not a conservative cleanup close.

## What This Changes

Request default-deny or least-privilege tool permissions for agents and embedded sub-agents instead of broad tool exposure by default.

## Best Possible Solution

Define an intentional narrow-by-default tool-policy contract with a documented unrestricted escape hatch, then update config schema/help/docs, doctor or migration guidance, and regression coverage across agent, sub-agent, embedded-run, plugin, MCP, cron, and channel surfaces.

## Reproduction Assessment

Yes, as a source reproduction: current main still allows tools when policy is missing and builds all embedded tool families when `toolsAllow` is omitted. The Active Memory-specific path is narrowed now, so the remaining reproduction is the broader default-permission contract rather than that original example alone.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. Caller-specific allowlists like Active Memory's are useful mitigations, but they are not the same as a repo-wide least-privilege default.
