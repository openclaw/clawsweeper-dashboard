## Summary

Current main still preserves broad default tool access for unset or unrestricted profiles and for missing or empty embedded-run allowlists. The related Active Memory prompt-bloat symptom has a targeted memory-only allowlist now, but the issue's broader disabled-by-default / explicit-enable request is not implemented and should remain open for maintainer policy work.

## What This Changes

The issue requests least-privilege agent tool permissions where tools are disabled by default and explicitly enabled per agent instead of exposing every available tool under broad defaults.

## Best Possible Solution

Define explicit default semantics for agent, sub-agent, and embedded-run tool policy; keep a clearly named unrestricted escape hatch; add migration or doctor warnings for broad existing configs; and update schema/help/docs plus regression coverage for core tools, plugin/MCP tools, Active Memory, sub-agents, and runtime `toolsAllow` behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main by inspecting profile resolution, the policy matcher, embedded-run `toolsAllow` handling, and tests/docs that preserve broad defaults.

## Solution Assessment

No complete solution is present yet. The Active Memory allowlist and local onboarding `coding` default are useful mitigations, but the requested disabled-by-default model needs an intentional compatibility and security-policy design rather than more one-off allowlists.
