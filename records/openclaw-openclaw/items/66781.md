## Summary

Keep open. Current main has important mitigations for the originally named Active Memory symptom and for fresh local onboarding, but the broader request for disabled-by-default, explicitly enabled per-agent tools is still unresolved in core runtime semantics, tests, and docs.

## What This Changes

The issue requests least-privilege agent tool policy semantics where tools are disabled by default and enabled explicitly per agent or embedded run.

## Best Possible Solution

Define an explicit tool-policy contract that makes narrow defaults consistent across normal agents, sub-agents, embedded runs, and bundled MCP/plugin tools; preserve a clearly named unrestricted escape hatch, add migration or doctor warnings for broad existing configs, update docs/schema/help, and add regression coverage for profile resolution, runtime `toolsAllow`, Active Memory, sub-agents, and plugin/MCP reachability.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main by inspecting profile resolution, the policy matcher, embedded `toolsAllow`, and tests/docs that preserve broad unset/full/empty allow semantics.

## Solution Assessment

No complete solution is present yet. The Active Memory allowlist and onboarding `coding` default are useful mitigations, but disabled-by-default semantics would require coordinated changes in the policy pipeline, embedded runner, bundle MCP handling, docs, and compatibility guidance.
