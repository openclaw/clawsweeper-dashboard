## Summary

Keep open. Current main has fixed the Active Memory example by passing a narrow `toolsAllow`, but the broader runtime contract is still permissive when no tool policy or embedded allowlist is configured, and changing that default is a product/security compatibility decision.

## What This Changes

The issue requests least-privilege tool exposure by default for agents and embedded/sub-agent runs, with tools enabled explicitly per agent or run.

## Best Possible Solution

Define an intentional narrow-by-default tool-policy contract with a documented unrestricted escape hatch, migration or doctor guidance, docs/schema/help updates, and regression coverage across agents, subagents, embedded runs, plugins/MCP, cron, and channels.

## Reproduction Assessment

Yes, as a source reproduction: current main still allows tools when policy is missing and builds all embedded tool families when `toolsAllow` is omitted. The Active Memory-specific example is now narrowed, so the remaining reproduction is the broader default-permission contract.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. Caller-specific allowlists like Active Memory's are useful mitigations, but they are not a repo-wide least-privilege default.
