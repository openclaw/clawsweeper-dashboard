## Summary

Keep open. Current main has narrowed the Active Memory example and now treats explicit empty embedded `toolsAllow` as no tools, but the broader request remains unresolved because unset policy and `full` still mean permissive tool access, and docs/tests still encode that behavior.

## What This Changes

Request default least-privilege agent tool permissions so agents only receive explicitly enabled tools instead of broad default tool inventories.

## Best Possible Solution

Define intentional default-deny or narrow-by-default semantics across agent, sub-agent, embedded-run, plugin, and MCP tool policy, keep an explicit unrestricted escape hatch such as `tools.profile: "full"`, and update docs, schema/help, migration warnings, and regression coverage together.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: unset policy and `full` still resolve to permissive behavior, omitted embedded `toolsAllow` still constructs all tool families, and docs/tests preserve those defaults.

## Solution Assessment

No complete solution is present yet. The Active Memory allowlist and explicit empty-allowlist handling are useful targeted fixes, but the requested least-privilege default is a coordinated compatibility and security-policy change rather than another caller-specific allowlist.
