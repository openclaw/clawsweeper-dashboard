## Summary

Keep open. Current main still documents and implements permissive tool exposure when no restrictive policy is configured, while the issue asks for a default-deny or narrow-by-default permission model across agents and embedded sub-agents; that is a real product/security-policy change, not a cleanup close.

## What This Changes

The issue requests least-privilege agent tool permissions so agents and embedded sub-agents receive only explicitly enabled tools instead of broad default access.

## Best Possible Solution

Define an intentional narrow-by-default tool-policy contract with a documented unrestricted escape hatch, then update config schema/help/docs, migration or doctor guidance, and regression coverage across agent, sub-agent, embedded-run, plugin, MCP, and cron surfaces.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: unset policy and omitted embedded `toolsAllow` still take permissive paths, while tests and docs on current main preserve that behavior.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. The Active Memory allowlist and local onboarding `coding` default are useful mitigations, but the requested least-privilege default is broader than another caller-specific filter.
