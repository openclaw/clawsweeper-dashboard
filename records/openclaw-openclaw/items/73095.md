## Summary

Keep open. Current main still resolves the configured context engine in the native subagent spawn preparation path without first initializing the built-in registry, and open PR #73904 already targets this exact bug with closing syntax, so the issue should remain open until that fix or an equivalent patch lands.

## What This Changes

Native `sessions_spawn` should initialize built-in context engines before resolving the configured context engine so cliBackend-only cold starts do not fail with `legacy` unregistered.

## Best Possible Solution

Merge PR #73904 or an equivalent narrow patch that initializes built-in context engines at the native spawn-prep resolution boundary, with a regression test for the empty-registry cold path. Keep PR #75129 as the separate companion for CLI compaction unless maintainers intentionally combine both lanes.

## Reproduction Assessment

Yes. A cold process that reaches native `sessions_spawn` before any PI embedded path initializes context engines will call `resolveContextEngine()` against an empty registry and hit the exact `legacy` error; the issue also has multiple matching user confirmations.

## Solution Assessment

Yes. Initializing at the spawn-side context-engine resolution boundary is the narrowest maintainable fix for this path; making legacy registration a top-level side effect would have broader runtime and bundling implications.
