## Summary

Keep open. Current main still has the reported init-order bug in the native `sessions_spawn` preparation path, and open PR #73904 references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Ensure native `sessions_spawn` context-engine preparation initializes the built-in context-engine registry before resolving the default `legacy` engine on cliBackend-only cold starts.

## Best Possible Solution

Land PR #73904 or an equivalent narrow patch that initializes built-in context engines before `prepareContextEngineSubagentSpawn()` resolves the configured engine, with a regression test for the cold registry path. Keep the CLI compaction sibling tracked by #75129 unless maintainers choose to coordinate both fixes together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a cold cliBackend-only process can enter `sessions_spawn`, reach `prepareContextEngineSubagentSpawn()`, and call bare `resolveContextEngine()` before the legacy engine is registered, producing the exact missing-engine error.

## Solution Assessment

Yes. Initializing built-in engines at the spawn-preparation resolution boundary is the narrowest maintainable fix for this specific path; making registration an import side effect would be broader and riskier.
