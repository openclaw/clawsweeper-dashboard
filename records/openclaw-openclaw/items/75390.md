## Summary

Keep open. Current main still rejects boolean values at both reported `visibleReplies` config paths, and the open paired PR #75392 is the active closing implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests accepting boolean `messages.visibleReplies` and `messages.groupChat.visibleReplies` values by mapping `true` to `"automatic"` and `false` to `"message_tool"` instead of invalidating the whole config.

## Best Possible Solution

Land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes booleans at both config paths while preserving downstream enum types and adding focused schema coverage for canonical strings, booleans, `undefined`, and invalid values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a boolean at either reported path reaches strict `z.enum` schemas through `OpenClawSchema.safeParse`, which rejects the config and causes the gateway reload path to skip the invalid snapshot.

## Solution Assessment

Yes. Coercing at the schema boundary is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary while channel runtime code can keep using the existing enum contract.
