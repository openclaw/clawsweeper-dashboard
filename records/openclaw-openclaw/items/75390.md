## Summary

Keep this issue open. Current main still rejects boolean values at both reported `visibleReplies` config paths, the shared config validation path still turns that into an invalid snapshot, and the linked same-author PR #75392 is the active closing implementation candidate rather than a reason to close the issue early.

## What This Changes

The issue asks for `messages.visibleReplies` and `messages.groupChat.visibleReplies` boolean values to be normalized instead of invalidating config and dropping channel availability.

## Best Possible Solution

Land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes `true` to `"automatic"` and `false` to `"message_tool"` at both config paths while preserving downstream enum types and adding focused schema coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a boolean at either reported path reaches the current strict Zod enum through `OpenClawSchema.safeParse`, which rejects the config and causes the hot-reload path to skip the invalid snapshot.

## Solution Assessment

Yes. Schema-boundary normalization is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary, while channel runtime code can continue consuming the existing enum values.
