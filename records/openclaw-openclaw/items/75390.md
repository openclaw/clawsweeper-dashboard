## Summary

Keep open. Current main still defines both reported `visibleReplies` paths as strict string enums, invalid config still fails the shared schema validation path, and the open paired PR #75392 is the active closing implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests accepting or coercing boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so `true`/`false` do not invalidate config and disrupt channel reload/status behavior.

## Best Possible Solution

Normalize `true` to `"automatic"` and `false` to `"message_tool"` at the shared config schema boundary for both paths, keep downstream runtime types as the existing string union, and add focused schema coverage for canonical strings, booleans, `undefined`, and invalid strings.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through `OpenClawSchema.safeParse`, which rejects the config before reload can apply it.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths all share schema validation; handling booleans downstream would not help invalid snapshots that are rejected before runtime use.
