## Summary

Keep this issue open. Current main still rejects boolean values at both reported `visibleReplies` config paths, invalid configs still fail the shared schema validation path, and the open paired PR #75392 is the active closing implementation candidate rather than a reason to close this issue early.

## What This Changes

The issue requests accepting or coercing boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so an intuitive toggle value does not invalidate config and disrupt channel handling.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes `true` to `"automatic"` and `false` to `"message_tool"` at both config paths while preserving downstream enum types and adding targeted config schema coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through `OpenClawSchema.safeParse`, which rejects the config and causes the reload path to skip the invalid snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable solution because startup, hot reload, doctor, and config mutation paths share that validation boundary, while channel runtime code can keep using the existing enum values.
