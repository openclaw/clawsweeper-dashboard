## Summary

Keep this issue open. Current main still defines both reported `visibleReplies` config paths as string-only enums, invalid configs still fail through the shared schema validation path, and the paired PR #75392 is open with closing syntax for this issue.

## What This Changes

The issue reports that boolean values for `messages.visibleReplies` or `messages.groupChat.visibleReplies` are rejected by strict config schemas, making the gateway treat the config as invalid and drop channel reload state without actionable field guidance.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes `true` to `"automatic"` and `false` to `"message_tool"` at both config paths while keeping downstream runtime types as the existing enum values and adding focused schema coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through `OpenClawSchema.safeParse`, which rejects the config and causes reload to skip the invalid snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share the same validation boundary, while downstream channel behavior can remain typed to `"automatic" | "message_tool" | undefined`.
