## Summary

Keep this issue open. Current main still rejects boolean `visibleReplies` values at both reported config paths, invalid config still flows through shared schema validation and causes gateway reload to skip the new snapshot, and open PR #75392 is already the paired implementation candidate with closing syntax.

## What This Changes

The issue requests accepting boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` instead of letting them invalidate the config and disrupt channel reloads.

## Best Possible Solution

Land PR #75392 or an equivalent narrow schema-boundary fix that normalizes `true` to `"automatic"` and `false` to `"message_tool"` at both config paths, keeps downstream types as the existing enum values, and adds focused config schema coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a boolean at either reported path reaches the strict `z.enum` schemas through `OpenClawSchema.safeParse`, which marks the config invalid and causes the gateway reload handler to skip the snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation flows share the same validation boundary, while the runtime can continue consuming normalized enum values.
