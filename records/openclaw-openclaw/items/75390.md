## Summary

Keep open: current main still rejects booleans at both reported visibleReplies config paths, and #75392 is an open linked fix PR with closing syntax, so this should remain active until that PR or an equivalent fix lands.

## What This Changes

The issue requests boolean compatibility for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so `true` or `false` values do not invalidate config and drop channel responsiveness.

## Best Possible Solution

Keep the issue open until #75392 or an equivalent narrow schema-boundary compatibility fix lands with focused schema tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A boolean at either reported config path reaches enum-only Zod fields on current main, so raw config validation returns issues before a usable snapshot is applied.

## Solution Assessment

Yes. Coercing booleans at the shared config schema boundary is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime code can keep the enum type.
