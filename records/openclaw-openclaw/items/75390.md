## Summary

Keep this open. Current main still rejects boolean values at both reported `visibleReplies` config paths, and the linked open PR #75392 is the active closing implementation candidate rather than a reason to close the issue before merge.

## What This Changes

Accept boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies`, mapping `true` to `"automatic"` and `false` to `"message_tool"` so the config does not invalidate channel startup or reload.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary normalization, so both visibleReplies config paths accept boolean input while downstream runtime types remain the existing enum values and targeted config schema coverage protects the behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a boolean at either reported config path reaches the current strict enum through `OpenClawSchema.safeParse`, producing an invalid snapshot that the gateway reload path skips.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because the relevant startup, hot reload, doctor, and config mutation paths share that validation boundary, while channel runtime code can keep using the existing enum contract.
