## Summary

Current main still rejects boolean values at both reported visibleReplies config paths, and the shared validation/reload path still turns that into an invalid-config reload skip. Because PR #75392 is open with closing syntax for this issue, it should remain open until that PR or an equivalent fix lands.

## What This Changes

Accept or coerce boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so a natural boolean toggle does not invalidate channel configuration.

## Best Possible Solution

Land PR #75392 or an equivalent narrow schema-boundary fix that normalizes boolean values at both config paths, preserves the downstream enum type, and adds targeted schema coverage for the accepted legacy/natural boolean inputs.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through shared config validation and is rejected before hot reload can apply the snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime channel code can keep using the existing enum modes.
