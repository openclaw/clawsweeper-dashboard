## Summary

Keep this issue open. Current main still defines both reported `visibleReplies` paths as strict string enums, invalid config still fails schema validation, and the open paired PR #75392 is the active implementation candidate with closing syntax, so this issue should stay open until that PR or an equivalent fix merges.

## What This Changes

The issue requests accepting boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` without invalidating gateway configuration.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes `true` to `"automatic"` and `false` to `"message_tool"` at both config paths while keeping downstream runtime types as the existing enum values and adding targeted config schema coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through `OpenClawSchema.safeParse`, which rejects the config and causes reload to skip the invalid snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share the same schema validation boundary, while downstream channel behavior can remain typed to `"automatic" | "message_tool" | undefined`.
