## Summary

Keep this open: current main still rejects boolean values at both reported `visibleReplies` config paths, and the invalid-config reload path still skips that snapshot. The same-author fix PR #75392 is open with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Allow or clearly handle boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so invalid config does not silently drop channel handling.

## Best Possible Solution

Review and land #75392 or an equivalent narrow schema-boundary fix that normalizes booleans at both config paths, adds focused schema coverage, and preserves the downstream enum contract.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a boolean at either reported path reaches enum-only Zod schemas, causes `OpenClawSchema.safeParse` to fail, and then the gateway reload handler skips the invalid snapshot.

## Solution Assessment

Yes. Coercing booleans at the shared config schema boundary is the narrowest maintainable direction because startup, hot reload, doctor, and config mutation paths share that validation surface while runtime code can keep the enum type.
