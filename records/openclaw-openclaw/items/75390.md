## Summary

Keep open: current main and v2026.5.2 still reject Boolean `visibleReplies` values, and #75392 is the open closing PR for the same failure, so this should remain active until that PR or an equivalent fix lands.

## What This Changes

The report asks for `messages.visibleReplies` and `messages.groupChat.visibleReplies` to accept Boolean values without invalidating config and dropping channel availability.

## Best Possible Solution

Land #75392 or an equivalent narrow config-boundary compatibility fix that normalizes Boolean input at both visible-reply paths, preserves the runtime enum type, and updates generated schema/docs artifacts.

## Reproduction Assessment

Yes. Setting either `messages.groupChat.visibleReplies` or `messages.visibleReplies` to `true` or `false` reaches enum-only Zod fields on current main and makes raw config validation fail before a usable snapshot is produced.

## Solution Assessment

Yes, with one caveat. Schema-boundary normalization is the narrow maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary, but the linked PR still needs the generated schema artifacts and public schema shape handled correctly.
