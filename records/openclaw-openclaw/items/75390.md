## Summary

Keep open: current main and v2026.5.2 still reject Boolean `visibleReplies` values, and #75392 is an open closing PR for the same failure path.

## What This Changes

The issue requests compatibility or clearer handling for Boolean `messages.visibleReplies` and `messages.groupChat.visibleReplies` values so `true`/`false` do not invalidate config and drop channel handling.

## Best Possible Solution

Land #75392 or an equivalent narrow config-boundary compatibility fix that normalizes Boolean input at both visible-reply paths while keeping runtime config typed as the existing string enum.

## Reproduction Assessment

Yes. Source inspection shows both schema sites are enum-only and raw validation uses `OpenClawSchema.safeParse`, so a Boolean at either path fails config validation; I did not run a live gateway/status scenario in this read-only review.

## Solution Assessment

Yes. Schema-boundary normalization at both visible-reply fields is the narrow maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary.
