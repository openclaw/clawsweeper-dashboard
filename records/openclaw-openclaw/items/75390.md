## Summary

Current main still rejects boolean values at both reported `visibleReplies` config paths, and the invalid-config path still skips gateway hot reload. Because PR #75392 is open, unmerged, and explicitly paired with this issue, the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests boolean compatibility for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so `true` or `false` no longer invalidates config and drops channel responsiveness.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes boolean values at both config paths while preserving the downstream `"automatic" | "message_tool"` runtime contract and adding focused schema coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: a boolean value reaches the current strict Zod enum at either reported path, `OpenClawSchema.safeParse` fails, and the gateway reload handler skips the invalid snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable direction because startup, hot reload, doctor, and config mutation paths all share validation while runtime delivery code can keep using the existing enum modes.
