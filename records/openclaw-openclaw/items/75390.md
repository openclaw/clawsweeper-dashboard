## Summary

Current main still rejects boolean values at both reported `visibleReplies` config paths, so the config-invalid reload skip remains a valid source-level bug. This issue should stay open because #75392 is already an open, unmerged paired fix PR with closing syntax for the same report.

## What This Changes

Normalize boolean `messages.visibleReplies` and `messages.groupChat.visibleReplies` values to the existing visible-reply modes instead of invalidating config.

## Best Possible Solution

Review and land #75392, or an equivalent narrow schema-boundary normalization with focused tests and a changelog entry if the fix is user-facing.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a boolean at either reported path reaches the strict `z.enum` schemas through shared config validation and is rejected before gateway reload can apply the snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share the validation boundary while downstream runtime code can keep using the existing enum modes.
