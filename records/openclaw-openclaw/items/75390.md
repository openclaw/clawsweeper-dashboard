## Summary

Keep open. Current main still rejects boolean `visibleReplies` values at both reported config paths and still skips invalid hot-reload snapshots, while PR #75392 is an open, unmerged closing implementation candidate. Repository policy keeps this issue open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that boolean values for `messages.visibleReplies` or `messages.groupChat.visibleReplies` are rejected by config validation, causing channel reloads to stop without an actionable field-level message.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix that accepts booleans at both config paths, normalizes them to the existing enum values, preserves downstream runtime types, and adds focused schema coverage for boolean compatibility.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: putting `true` or `false` at either reported config path reaches a strict `z.enum` schema and fails shared validation before the hot-reload snapshot can apply.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable direction because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime code can keep the existing `"automatic" | "message_tool"` contract.
