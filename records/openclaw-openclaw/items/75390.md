## Summary

Keep this open. Current main still defines both reported `visibleReplies` config paths as string-only enums, and the shared validation plus gateway reload path still turns a boolean value into an invalid-config reload skip. The issue is also paired with open PR #75392 using closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Accept or normalize boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so natural boolean config does not invalidate the gateway configuration and interrupt channel handling.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix, that coerces `true` to `"automatic"` and `false` to `"message_tool"` at both config paths, preserves the downstream enum type, adds focused schema tests, and includes the expected user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: putting a boolean at either reported config path reaches the current strict Zod enum and is rejected before the gateway hot reload can accept the snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable direction because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime channel code can continue consuming the existing enum modes.
