## Summary

Current main still rejects boolean values at both reported `visibleReplies` config paths, and the shared validation/reload path turns that into an invalid-config reload skip. This issue should stay open because #75392 is an open, unmerged fix PR with closing syntax for the same report.

## What This Changes

Accept boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies`, or otherwise make the invalid config failure actionable instead of leaving channels down.

## Best Possible Solution

Review and land #75392 or an equivalent narrow schema-boundary fix that normalizes booleans at both config paths while preserving the downstream enum type and adding focused schema tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: setting either reported path to `true` or `false` reaches current enum-only Zod schemas through shared config validation, which rejects the value before hot reload can apply it.

## Solution Assessment

Yes. Schema-boundary normalization is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation layer while runtime channel code can keep using the existing enum modes.
