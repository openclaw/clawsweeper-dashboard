## Summary

Keep open: current main still rejects booleans at both `visibleReplies` config paths, and #75392 is already the open paired implementation candidate. Closing before that PR or an equivalent fix lands would lose a valid, source-confirmed regression report.

## What This Changes

Accept boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so intuitive JSON5 toggles do not make the config invalid and drop channel handling.

## Best Possible Solution

Accept boolean compatibility at the shared config schema boundary for both paths, preserve normalized enum values internally, and keep this issue linked to #75392 until the fix lands.

## Reproduction Assessment

Yes. A boolean at either reported path reaches enum-only Zod fields on current main, causing raw config validation to return issues before a usable snapshot can be applied.

## Solution Assessment

Yes. Normalizing booleans at the schema boundary is the narrowest maintainable direction because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime code keeps the enum contract.
