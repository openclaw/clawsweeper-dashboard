## Summary

Keep this issue open. Current main still rejects boolean `visibleReplies` values at both reported config paths, the shared config validator still fails the whole config snapshot, and reload still skips invalid snapshots; the already-open paired PR #75392 is the active implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests accepting boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so accidental `true` or `false` config values do not invalidate all channel configuration.

## Best Possible Solution

Review and land PR #75392, or an equivalent narrow schema-boundary fix, that normalizes `true` to `"automatic"` and `false` to `"message_tool"` at both config paths, keeps downstream runtime types unchanged, and adds focused schema tests plus any required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a boolean at either reported path reaches strict `z.enum` validation through `OpenClawSchema.safeParse`, which rejects the config and causes the gateway reload path to skip that snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable solution because startup, hot reload, doctor, and config mutation paths share the same validation boundary while runtime reply routing can continue consuming only the existing enum values.
