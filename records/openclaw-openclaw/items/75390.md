## Summary

Current main still rejects booleans at both reported visibleReplies config paths, and the shared config validation path still turns that into an invalid-config reload skip. This issue should stay open because #75392 is already an open, unmerged fix PR with closing syntax for the same report.

## What This Changes

Accept boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` instead of letting them invalidate config and interrupt channel operation.

## Best Possible Solution

Land #75392 or an equivalent narrow schema-boundary fix that normalizes booleans at both config paths, preserves the downstream enum type, and adds focused schema tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through shared validation and is rejected before hot reload can apply the snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime channel code can keep using the existing enum modes.
