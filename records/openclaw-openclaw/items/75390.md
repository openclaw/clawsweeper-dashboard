## Summary

Keep open: current main still rejects boolean values at both reported visibleReplies config paths, and #75392 is the open linked fix PR with closing syntax. The source-level failure is clear enough to keep this issue active until that PR or an equivalent fix lands.

## What This Changes

The issue requests boolean-compatible handling for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so boolean values do not invalidate config and disrupt channels.

## Best Possible Solution

Land #75392 or an equivalent narrow schema-boundary normalization that accepts booleans at both visibleReplies paths, preserves the downstream enum type, and adds focused schema coverage.

## Reproduction Assessment

Yes. A boolean at either reported config path reaches enum-only Zod fields on current main and causes raw config validation to return issues before a usable snapshot is produced.

## Solution Assessment

Yes. Coercing booleans at the shared config schema boundary is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths all share that validation boundary while runtime code keeps the enum contract.
