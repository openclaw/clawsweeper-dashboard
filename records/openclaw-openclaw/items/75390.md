## Summary

Current main still rejects boolean values at both reported `visibleReplies` config paths, and #75392 is already an open, unmerged fix PR with closing syntax for this issue. The issue should remain open until that PR, or an equivalent narrow fix, lands.

## What This Changes

The issue requests accepting or clearly handling boolean `true`/`false` values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so an intuitive config toggle does not invalidate all channel reloads silently.

## Best Possible Solution

Land #75392 or an equivalent schema-boundary normalization that coerces booleans at both config paths, preserves the downstream enum type, and adds focused schema tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a boolean at either reported config path reaches the current strict `z.enum` schemas through shared validation and is rejected before hot reload can apply the snapshot.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths share that validation boundary while runtime channel code can keep using the existing enum modes.
