## Summary

Keep open: current main and v2026.5.2 still reject Boolean `visibleReplies` values, and #75392 is the open closing PR for the same failure, so this should remain active until that PR or an equivalent fix lands.

## What This Changes

The issue requests Boolean compatibility for `messages.visibleReplies` and `messages.groupChat.visibleReplies` so `true`/`false` do not invalidate config and disrupt channel reloads.

## Best Possible Solution

Review and land #75392 or an equivalent narrow config-boundary compatibility fix that normalizes Booleans at both visible-reply paths while keeping the runtime enum type and generated schema/docs in sync.

## Reproduction Assessment

Yes from source inspection. Setting either visible-reply field to `true` or `false` reaches enum-only Zod schemas, makes `OpenClawSchema.safeParse` fail, and causes the gateway reload path to skip the invalid snapshot.

## Solution Assessment

Yes. Schema-boundary normalization is the narrow maintainable fix because startup, hot reload, doctor, and config mutation paths share raw validation, but generated schema/docs artifacts must stay consistent.
