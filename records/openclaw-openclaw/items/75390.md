## Summary

Keep this issue open. Current main and the latest tagged release still reject boolean visibleReplies values through strict string-enum schemas, and the linked open PR #75392 is the active implementation candidate with closing linkage for this issue, so it should not be closed before that PR merges or is replaced.

## What This Changes

The issue requests that boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` be accepted or normalized instead of invalidating config and dropping channel availability.

## Best Possible Solution

Keep the issue open and review or land #75392, or an equivalent narrow fix, that normalizes booleans at the config schema boundary while preserving the downstream `"automatic" | "message_tool" | undefined` runtime type and adding targeted tests for both config paths.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: setting either reported path to `true` or `false` reaches strict Zod enum validation through `OpenClawSchema.safeParse`, producing an invalid config snapshot that hot reload skips.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable fix because startup, hot reload, doctor, and config command paths all share schema validation, while downstream channel logic can keep the existing enum contract.
