## Summary

Keep this open: current main still rejects boolean values at both reported visibleReplies config paths, and the invalid-config reload path still skips applying that snapshot. The paired fix PR #75392 is open with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Boolean values in `messages.visibleReplies` or `messages.groupChat.visibleReplies` should not invalidate the config and take channel reload/status behavior down.

## Best Possible Solution

Land #75392 or an equivalent narrow schema-boundary fix that normalizes booleans at both config paths, preserves the downstream enum type, and adds focused schema coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a boolean at either reported config path reaches the current enum-only Zod schemas, fails validation, and the gateway reload path skips invalid snapshots.

## Solution Assessment

Yes. Coercing booleans at the shared schema boundary is the narrowest maintainable fix because startup, hot reload, doctor, and config mutation paths all share that validation boundary while runtime code can keep the enum contract.
