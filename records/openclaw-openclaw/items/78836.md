## Summary

Keep open: the NVIDIA NIM token-field problem is source-proven on current main, but this PR hard-codes a provider host in core and lacks required real behavior proof from the contributor's setup.

## What This Changes

The PR adds NVIDIA NIM baseUrl detection to OpenAI-completions compatibility defaults and tests that NIM uses `max_tokens` while unrelated custom endpoints keep `max_completion_tokens`.

## Best Possible Solution

Carry the NIM `max_tokens` default through NVIDIA-owned compatibility metadata or a generic manifest/providerRequest field, with focused regression coverage and redacted real NIM proof before merge.

## Reproduction Assessment

Yes for source-level reproduction: current main routes NVIDIA through OpenAI-completions, the payload builder follows `compat.maxTokensField`, and the NVIDIA manifest does not set that field. I did not run a live NIM request in this read-only review.

## Solution Assessment

No: the PR likely fixes the wire field symptom, but hard-coding `integrate.api.nvidia.com` in core conflicts with the provider/plugin ownership model. A provider-owned compat entry or a small generic manifest request seam is the safer direction.
