## Summary

Keep this PR open, but not merge-ready: the reported NVIDIA NIM token-field bug is source-reproducible on current main, while the proposed fix hard-codes a provider host in core and still lacks contributor-supplied real behavior proof.

## What This Changes

The PR adds NVIDIA NIM baseUrl detection to OpenAI-completions compatibility defaults and tests that NIM uses `max_tokens` while unrelated custom endpoints keep `max_completion_tokens`.

## Best Possible Solution

Model the NIM `max_tokens` requirement in NVIDIA-owned compat metadata or a generic manifest/providerRequest seam, with focused regression coverage and redacted real NIM proof before merge.

## Reproduction Assessment

Yes for the current-main source path: NVIDIA routes through OpenAI-completions, the payload builder follows `maxTokensField`, current defaults emit `max_completion_tokens`, and NVIDIA docs for the named endpoint list `max_tokens`. ([docs.api.nvidia.com](https://docs.api.nvidia.com/nim/reference/qwen-qwen2_5-coder-32b-instruct-infer)) A live NIM request still needs contributor proof.

## Solution Assessment

No: the PR fixes the outbound field selection, but hard-coding `integrate.api.nvidia.com` in core conflicts with the plugin/provider ownership model; a provider-owned manifest/model-compat path or a generic request metadata seam is safer.
