## Best Possible Solution

Land a version that keeps core sender fallback generic, uses plugin-owned or generic identity classification for WhatsApp group IDs, and makes the intentional fail-closed upgrade behavior visible to maintainers.

## Reproduction Assessment

Yes. Source inspection shows current main adds `ctx.From` to elevated sender tokens unconditionally, while WhatsApp group contexts use `From` for the group conversation and separate sender identity fields for the actor.

## Solution Assessment

No. Guarding `From` is the right repair direction, but hard-coding WhatsApp `@g.us` parsing in core is not the best owner-boundary shape.
