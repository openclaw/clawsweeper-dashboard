## Summary

Keep open. Current main still exposes session storage as a filesystem path with direct JSON file IO and path-based helper APIs; the requested pluggable backend/config/API contract is still unimplemented and needs maintainer design.

## What This Changes

Request to add a configurable session-store adapter or driver contract for backends like file, SQLite, Redis, Convex, and custom modules.

## Best Possible Solution

Define an additive, versioned session persistence contract with the current filesystem store as the default, one supported backend as proof, and aligned config, schema, docs, doctor/migration behavior, and regression coverage.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug reproduction. Source and docs inspection on current main gives high confidence that session.store is still a string filesystem path and no driver or adapter seam exists.

## Solution Assessment

Unclear: the proposed SessionStoreAdapter direction is plausible, but maintainers need to decide the core vs plugin SDK boundary, credential handling, migration/durability rules, and consistency model before implementation.
