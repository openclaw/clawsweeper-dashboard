## Summary

Keep open. Current main still has a path-only filesystem session store with locked JSON load/save paths and only helper-style plugin access, so the requested backend adapter contract is not implemented. The existing discussion shows maintainer and community interest, but this remains a broad storage/API design item rather than conservative cleanup.

## What This Changes

The issue requests a pluggable session store adapter/config contract for file, database, cloud, and custom session persistence backends.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported, backward-compatible session-store adapter contract with the existing file store as the default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: inspect the config type/schema, resolver, load/save/lock implementation, plugin SDK surface, docs, and targeted symbol search to confirm there is no backend adapter seam.

## Solution Assessment

Unclear. A backend adapter contract is plausible, but the best solution depends on maintainer choices around public API shape, plugin ownership, credentials, migrations, durability/recovery semantics, and overlap with #51946.
