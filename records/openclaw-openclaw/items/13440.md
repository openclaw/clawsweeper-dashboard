## Summary

Keep open. Current main still has a path-only filesystem session store and plugin-facing helpers built around `sessions.json`; the closed adapter PR did not merge, and the open directory-backed PR only addresses local storage contention rather than external backend drivers.

## What This Changes

Request pluggable session store backends so sessions can use file, SQLite, Redis, Convex, custom, or other shared storage drivers instead of only a local JSON file.

## Best Possible Solution

Define an additive, versioned session persistence contract with the filesystem store as the compatibility default, one supported backend as proof, and aligned config, schema, docs, doctor/migration behavior, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source, docs, live issue state, and related PR inspection give high confidence that current main still lacks a session-store backend driver or adapter registration seam.

## Solution Assessment

Unclear; a SessionStoreAdapter-style direction is plausible, but maintainers first need to choose the core/SDK/plugin boundary and persistence semantics. The narrowest maintainable path is a deliberate public contract rather than an automated one-off backend implementation.
