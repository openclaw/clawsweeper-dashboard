## Summary

Keep open. Current main still has a string/path-only session store with JSON file load/save helpers and no backend adapter registration seam; the request remains a valid broad storage API/product design item, and #51946 is adjacent but narrower.

## What This Changes

The issue requests a pluggable session-store adapter/config contract so file storage can be replaced by backends such as SQLite, Redis, Convex, or custom modules.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a backward-compatible session-store adapter contract with the existing file store as the default, or narrow the roadmap to a supported local storage model and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: config, schema, resolver, load/save code, SDK exports, docs, and symbol search show no backend adapter seam.

## Solution Assessment

Unclear. A SessionStoreAdapter-style seam is plausible, but the best solution depends on maintainer choices around API shape, plugin ownership, credentials, migrations, durability/recovery behavior, and the relationship to #51946.
