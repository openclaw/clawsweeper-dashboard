## Summary

Keep open. Current main still exposes session storage as a local path-backed JSON store plus helper-style SDK access, not a configurable backend or adapter registration contract; the related directory-store work is narrower and still needs maintainer handling.

## What This Changes

Request to add a pluggable session-store backend interface for file, SQLite, Redis, Convex, or custom storage instead of only a filesystem store path.

## Best Possible Solution

Define an additive, versioned session persistence contract with the current file store as the default, one supported backend path as proof, and aligned config/schema/docs plus session/transcript regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence that current main still lacks a driver/config/API seam for session-store backends.

## Solution Assessment

Unclear; a SessionStoreAdapter-style direction is plausible, but maintainers need to choose the core versus plugin boundary, credential model, migration behavior, and distributed consistency guarantees first.
