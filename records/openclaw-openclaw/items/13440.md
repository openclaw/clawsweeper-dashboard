## Summary

Keep open. Current main still has a string/path-only, JSON file-backed session store with lock-backed writes and helper-style plugin access, while the requested backend adapter/driver contract remains a broad core/SDK design decision; #51946 is adjacent but narrower.

## What This Changes

Add a backward-compatible session store adapter/driver interface so deployments can use file, SQLite, Redis, Convex, custom, or other backends instead of only `sessions.json`.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a backward-compatible session-store adapter contract with the current file store as the default, or narrow the roadmap to a built-in local storage model and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: config, schema, resolver, load/write/lock code, docs, and SDK surfaces show no backend adapter or driver registration seam.

## Solution Assessment

Unclear. A `SessionStoreAdapter`-style seam is plausible, but the best solution depends on maintainer choices around API shape, plugin ownership, credentials, migrations, durability/recovery semantics, and overlap with #51946.
