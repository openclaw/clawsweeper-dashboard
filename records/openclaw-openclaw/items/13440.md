## Summary

Keep open. Current main still has a string/path-only filesystem session store with JSON load/save and lock-backed writes; the existing plugin seams expose helper access to session data, not a backend adapter registration contract. The request remains a valid broad storage API/product design item, and #51946 is adjacent but narrower.

## What This Changes

Request a pluggable session-store backend contract so the file store remains the default while backends like SQLite, Redis, Convex, or custom modules can load and update session state.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a backward-compatible session-store adapter contract with the existing file store as the default, or narrow the roadmap to a built-in local storage model and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: config, schema, resolver, load/save/lock code, plugin SDK docs, and symbol search show no backend adapter seam.

## Solution Assessment

Unclear. A SessionStoreAdapter-style seam is plausible, but the best solution depends on maintainer choices around API shape, plugin ownership, credentials, migrations, durability/recovery semantics, and overlap with #51946.
