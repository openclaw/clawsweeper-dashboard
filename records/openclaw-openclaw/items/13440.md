## Summary

Keep open. Current main still uses a path-only `sessions.json` filesystem store with helper-style SDK access; the writer queue reduces one local lock-contention path, but no configurable session backend or plugin adapter contract for Redis, Convex, SQLite, or multi-instance storage has landed.

## What This Changes

Requests a configurable `SessionStoreAdapter`-style session persistence contract so the current JSON-file store can be replaced by backends such as Convex, SQLite, Redis, or a custom adapter.

## Best Possible Solution

Define an additive, versioned session persistence contract with the filesystem store as the default, one supported backend as proof, and aligned config, schema, docs, migration/doctor behavior, durability rules, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection gives high confidence that current main still lacks a driver/config/API seam for session store backends.

## Solution Assessment

Unclear; a `SessionStoreAdapter` direction is plausible, but maintainers first need to choose the core-vs-plugin ownership, credential, migration, durability, and consistency boundaries.
