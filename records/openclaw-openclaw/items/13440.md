## Summary

Keep open. Current main has improved file-store writer semantics, but `session.store` still resolves to a `sessions.json` path and plugin/runtime APIs expose file-path helpers rather than a backend registration contract, so external session backends remain a product/API design item.

## What This Changes

Request adding a configurable SessionStoreAdapter-style backend contract for file, SQLite, Redis, Convex, custom, and similar session stores.

## Best Possible Solution

Keep this as a maintainer design item: either define an additive, versioned session-store backend contract with the current file store as the compatibility default, or explicitly narrow the roadmap to local file/directory storage and document external backends as out of scope.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing behavior report. Source inspection gives high confidence that current main still lacks the requested adapter/config path because session storage is resolved and operated as a file path.

## Solution Assessment

Unclear; a SessionStoreAdapter-style seam is plausible, but not automatically the best fix until maintainers choose the core/SDK/plugin boundary and persistence semantics. A safer near-term path is to keep local file-store improvements separate from the backend contract design.
