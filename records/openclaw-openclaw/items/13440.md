## Summary

Keep open. Current main still exposes session storage as a filesystem `sessions.json` path with direct JSON file IO and file-path helper APIs; recent writer-queue work reduces one lock-contention path, but the requested pluggable backend/config/API contract has not landed.

## What This Changes

Request an additive session-store adapter/backend contract so config can select file, SQLite, Redis, Convex, custom, or similar stores instead of only a filesystem path.

## Best Possible Solution

Define an additive, versioned session persistence contract with the filesystem store as the compatibility default, one supported backend as proof, and aligned config, schema, docs, doctor/migration behavior, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection gives high confidence that current main still lacks a driver/config/API seam for session store backends.

## Solution Assessment

Unclear; a SessionStoreAdapter direction is plausible, but maintainers first need to choose the core, SDK/plugin, credential, migration, durability, and consistency boundaries.
