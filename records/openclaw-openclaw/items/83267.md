## What This Changes

The PR relaxes model provider config validation for bundled provider timeout-only overlays, updates the provider config type, guards configured-provider fallback and Microsoft Foundry model selection, and adds focused regression tests.

## Best Possible Solution

Keep bundled provider overlays, but make the validation source of truth cover manifest-owned provider IDs and aliases so existing built-in aliases are not left behind.

## Reproduction Assessment

Yes, source inspection plus the linked issue and PR proof give a high-confidence path: current main requires baseUrl/models for timeout-only provider entries, so the documented OpenAI overlay fails validation before startup proceeds.

## Solution Assessment

No. The overlay direction is right, but the static allowlist should include or derive existing manifest-owned provider aliases such as azure-openai-responses, moonshotai, and moonshot-ai before merge.
