## Summary

Keep this PR open: it fixes the reported OpenAI overlay path and includes usable terminal proof, but the new static bundled-provider allowlist misses supported manifest aliases such as Azure OpenAI and Moonshot aliases, so the patch is not merge-ready.

## What This Changes

The PR makes `models.providers.<id>.timeoutSeconds` valid as a standalone overlay for bundled model providers while keeping unknown custom providers strict.

## Best Possible Solution

Keep the overlay behavior, but validate bundled overlays using existing provider/manifest alias metadata or shared provider normalization rather than a static core allowlist, and add regression coverage for alias providers.

## Reproduction Assessment

Yes. Current main requires `baseUrl` and `models` in `ModelProviderSchema`, while docs and schema help expose `models.providers.<id>.timeoutSeconds`, so the reported startup validation failure is source-reproducible.

## Solution Assessment

No. The intended fix is right, but the static lowercased provider list misses manifest-owned aliases and creates drift from the plugin registry; the safer solution should use existing provider/alias metadata or normalization with alias tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Accept manifest provider aliases as bundled overlays:** `src/config/zod-schema.core.ts:440-441`
  - body: The overlay allowlist only lowercases the raw provider key, but bundled manifests already declare aliases such as `azure-openai-responses` for OpenAI and `moonshotai` / `moonshot-ai` for Moonshot, and those aliases are treated as model-catalog provider owners. With this patch, `models.providers.azure-openai-responses.timeoutSeconds` still falls through to the custom-provider errors, so supported bundled aliases cannot use the new overlay path. Drive this from manifest/normalization metadata or add the alias IDs with regression coverage.
