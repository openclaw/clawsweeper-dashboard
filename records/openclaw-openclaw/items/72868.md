## Summary

Keep this PR open for revision. The core bug is real on current main and the PR is in the right shared sanitizer area, but the current branch still uses a negative exclusion guard that would inject `thinkingBudget: 0` for non-budget Google model IDs and it is missing the required user-facing changelog entry.

## What This Changes

The PR updates shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` for `thinkingLevel: "off"` requests and adds regression tests for config, generationConfig, no-container, and excluded-model paths.

## Best Possible Solution

Revise this PR in place: keep the shared sanitizer fix for Gemini 2.5 Flash and Flash-Lite thinking-off requests, but make both injection sites require the budget-capable Gemini 2.5 family, add negative coverage for non-budget Google IDs, and add a single-line Unreleased Fixes changelog entry. Keep the Vertex/Gemini CLI transport widening in #73416 separate.

## Reproduction Assessment

Yes. The original bug is reproducible by unit-calling `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })` on current main and observing no mutation; the PR-introduced blocker is reproducible by the same inspection with `modelId: "gemini-2.0-flash"` on PR head, which would inject `thinkingBudget: 0`.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the maintainable fix is to gate forced budget injection on a positive budget-capability helper such as `isGoogleGemini25ThinkingBudgetModel(...)` in both branches, with matching tests and release-note coverage.
