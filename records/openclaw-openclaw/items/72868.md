## Summary

Keep this PR open. It fixes a real current-main Google thinking-off gap, but the PR head injects `thinkingBudget: 0` for any non-excluded Google model ID instead of positively gating on Gemini 2.5 budget-capable models, and it is missing the required changelog entry for a user-facing provider fix.

## What This Changes

The PR updates shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` when `thinkingLevel` is `off`, imports the shared Gemma 4 model-id matcher, and adds regression tests for config, generationConfig, and no-container payloads.

## Best Possible Solution

Revise this PR rather than close it: keep the shared sanitizer fix for Gemini 2.5 Flash and Flash-Lite thinking-off requests, but make both injection paths positively require the Gemini 2.5 thinking-budget family, add regression coverage for non-budget Google IDs, and add a single-line Unreleased Fixes changelog entry. Leave the Vertex/Gemini CLI wrapper widening to #73416.

## Reproduction Assessment

Yes. The original bug is reproducible by inspecting or unit-calling `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })`, which current main leaves unchanged. The PR-introduced blocker is reproducible the same way with `modelId: "gemini-2.0-flash"`, where PR head would inject a thinking budget outside the documented budget-capable family.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the maintainable fix is to gate forced `thinkingBudget: 0` on `isGoogleGemini25ThinkingBudgetModel(...)` in both the no-container fallback and in-container branch, with matching negative tests and release-note coverage.
