## Summary

Keep this PR open. The underlying Google thinking-off bug is real on current main and the PR is aimed at the right shared sanitizer, but the current diff still applies `thinkingBudget: 0` through a negative exclusion guard instead of a positive Gemini 2.5 budget-capability guard, and it is missing the required user-facing changelog entry.

## What This Changes

The PR changes shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` for `thinkingLevel: "off"` requests and adds regression tests for config, generationConfig, no-container, and excluded-model paths.

## Best Possible Solution

Revise this PR in place: keep the shared sanitizer fix for Gemini 2.5 Flash and Flash-Lite thinking-off requests, but make both injection sites require the budget-capable Gemini 2.5 family, preserve the existing exclusions for thinking-required and alternate-schema models, add negative coverage for non-budget Google IDs, and add a single-line Unreleased Fixes changelog entry. Keep the Vertex/Gemini CLI transport widening in #73416 separate.

## Reproduction Assessment

Yes. The original bug is reproducible by source-level unit inspection of `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })` on current main, which leaves the payload unchanged. The PR-introduced blocker is reproducible by the same call shape with a non-budget ID such as `gemini-2.0-flash` on PR head, where the negative guard would inject `thinkingBudget: 0`.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the maintainable fix is to gate forced budget injection on `isGoogleGemini25ThinkingBudgetModel(...)` in both branches, with matching negative tests and release-note coverage.
