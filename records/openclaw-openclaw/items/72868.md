## Summary

Keep this PR open for revision. The reported current-main bug is real and the branch has useful coverage, but the patch still injects `thinkingBudget: 0` by broad negative exclusions rather than by a positive budget-capable model guard, so non-thinking Google IDs can receive an unsupported thinking payload; it also still lacks the required user-facing changelog entry.

## What This Changes

The PR changes shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` when `thinkingLevel` is `off`, imports the shared Gemma 4 model matcher, and adds regression tests for config, generationConfig, no-container, and model-skip paths.

## Best Possible Solution

Revise the PR rather than close it: keep the shared sanitizer fix for Gemini 2.5 Flash and Flash-Lite thinking-off requests, but make both new injection paths positively require a budget-capable model family, add a negative regression test for a non-budget Google ID such as `gemini-2.0-flash`, and add a single-line Unreleased Fixes changelog entry. Keep the Vertex/Gemini CLI wrapper widening separate in #73416.

## Reproduction Assessment

Yes. Current main can be reproduced with a focused unit call to `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })`, which leaves `thinkingConfig` absent; the PR-introduced blocker is the same call with `modelId: "gemini-2.0-flash"`, which PR head would mutate to include `thinkingBudget: 0`.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the maintainable fix is a positive budget-capable guard with matching negative coverage, plus the required changelog entry.
