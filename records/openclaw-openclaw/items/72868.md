## Summary

Keep this PR open. The reported Gemini 2.5 Flash thinking-off bug is real on current main, but the latest PR head still injects `thinkingBudget: 0` through a negative exclusion guard that can affect non-budget Google model IDs; it also still lacks the required user-facing changelog entry.

## What This Changes

The PR updates shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` for `thinkingLevel: "off"` requests and adds regression tests for config, generationConfig, no-container, and excluded-model paths.

## Best Possible Solution

Revise this PR in place: keep the shared sanitizer fix for Gemini 2.5 Flash and Flash-Lite thinking-off requests, but make both injection sites require the budget-capable Gemini 2.5 family, add negative coverage for non-budget Google IDs, and add a single-line changelog entry. Keep the Vertex/Gemini CLI transport widening in #73416 separate.

## Reproduction Assessment

Yes. Current main can be reproduced by unit-calling `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })` and observing no mutation; the PR-introduced blocker is reproducible by inspecting the latest PR head with `modelId: "gemini-2.0-flash"`, which would inject `thinkingBudget: 0`.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the maintainable fix is to gate forced budget injection on a positive budget-capability helper such as `isGoogleGemini25ThinkingBudgetModel(...)` in both branches, with matching tests and release-note coverage.
