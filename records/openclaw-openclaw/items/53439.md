## Summary

Keep open. Current main still matches the reported Synology Chat incompatibility: HEAD probes take the non-POST 405 path, successful POST deliveries still ACK with 204/no body, and open implementation PRs using closing syntax are still unmerged.

## What This Changes

Synology Chat webhook handling should accept HEAD probes and acknowledge successful POST deliveries with HTTP 200 plus {"success":true} instead of 204/no body.

## Best Possible Solution

Keep this issue open until #53441 lands or is explicitly superseded by a narrow Synology Chat plugin fix that accepts HEAD probes, returns 200 with {"success":true} for successful webhook ACKs, preserves token, allowlist, rate-limit, and in-flight behavior, and updates webhook plus integration tests that currently expect 204.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a HEAD request to createWebhookHandler follows the 405 non-POST branch, and a valid POST follows the accepted path that calls the 204/no-body ACK helper; current tests encode both old expectations.

## Solution Assessment

Yes. The requested plugin-local handler change is the narrowest maintainable solution because gateway wiring and docs show this behavior belongs to the bundled Synology Chat webhook route, not core HTTP routing.
