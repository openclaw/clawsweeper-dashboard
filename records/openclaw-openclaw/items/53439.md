## Summary

Keep open. Current `main` still matches the reported Synology Chat incompatibility: the active webhook handler rejects `HEAD` through the non-POST 405 path and still acknowledges accepted webhook POSTs with `204` and no body. The issue is also paired with open implementation PR #53441 using closing syntax, so it should remain open until that PR or a focused replacement lands.

## What This Changes

Change the Synology Chat webhook handler to accept `HEAD` probes and return `200 {"success":true}` for successful POST acknowledgements instead of `204 No Content`.

## Best Possible Solution

Land #53441, or explicitly supersede it with a focused Synology Chat plugin change that accepts `HEAD` probes, returns `200 {"success":true}` for successful POST ACKs, preserves existing token, allowlist, rate-limit, and async delivery behavior, and updates the webhook plus integration tests that currently expect 204.
