## Summary

Keep open. Current main has adjacent ACP cleanup, but the bound persistent-session repair leak is still not resolved: active bound persistent ACP sessions are preserved, the ensure path still re-enters the runtime without process-tree cleanup, and OpenClaw still delegates adapter process lifecycle to the bundled acpx runtime.

## What This Changes

Request cleanup of orphaned `claude-agent-acp` and `plugin-tools-serve.js` processes when bound persistent ACP sessions are repaired or the gateway shuts down.

## Best Possible Solution

Fix ACPX process supervision so adapter launches have tracked PID/process-group metadata and close the full adapter/tool descendant tree before replacement and on shutdown, then bump OpenClaw's bundled acpx package and keep any process-name reaper as defense-in-depth only.

## Reproduction Assessment

Yes. The issue provides concrete restart/repair steps and a `ppid=1` process-table check for orphaned ACP descendants, and current source inspection still shows the relevant cleanup gap; I did not run the live ACP loop in this read-only review.

## Solution Assessment

No. The current-main behavior is not the best solution; the maintainable fix belongs in the acpx process lifecycle contract plus an OpenClaw dependency bump rather than relying on manual `ps` cleanup or metadata-only maintenance sweeps.
