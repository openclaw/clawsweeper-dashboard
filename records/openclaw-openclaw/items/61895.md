## Summary

Keep open. Current main has adjacent ACP cleanup and a newer bundled acpx runtime, but the specific bound persistent-session repair leak is not resolved: persistent sessions with active bindings are intentionally preserved, the OpenClaw ensure path still re-enters `runtime.ensureSession(...)` without a pre-replacement process-tree cleanup, and the pinned acpx 0.6.1 runtime still terminates only its direct child process rather than supervising a detached process group or reaping orphan descendants.

## What This Changes

Persistent ACP sessions should clean up replaced or orphaned `claude-agent-acp` and `plugin-tools-serve.js` process trees during `ensureSession` repair and gateway shutdown.

## Best Possible Solution

Fix the ACPX process supervisor so each ACP adapter launch has tracked PID/process-group metadata and closes the entire adapter plus descendant harness/tool tree before replacement and on gateway shutdown. Then bump OpenClaw's bundled acpx package, expose enough runtime status for OpenClaw cleanup paths, and keep any process-name orphan reaper as defense-in-depth rather than the primary lifecycle contract.

## Reproduction Assessment

Yes. The report gives concrete restart/repair steps and a process-table check for `ppid=1` orphaned ACP processes, and current source/dependency inspection shows the relevant cleanup gap remains plausible; I did not run the live macOS ACP loop in this read-only review.

## Solution Assessment

No. Current main is not the best solution for this issue; it has useful adjacent cleanup and dependency staging, but the maintainable fix is a process-tree lifecycle contract in acpx plus an OpenClaw bump/integration rather than relying on metadata sweeps or manual `ps` cleanup.
