## Summary

Current main now implements ownership-checked ACPX process cleanup for the reported orphan-process leak: generated wrappers kill their child process group when signaled or orphaned, runtime close cleans recorded process trees, and startup reaps stale PPID=1 OpenClaw-owned ACPX orphans. The fix is current-main-only in this checkout and is not in the latest v2026.5.7 tag.

## What This Changes

Persistent ACP repair and shutdown should clean up old ACP adapter/tool process trees so claude-agent-acp/plugin-tools-serve orphans do not accumulate.

## Best Possible Solution

Keep the current ownership-limited acpx process supervision on main and let the next release carry it; broader PID observability or unrelated ACP leak reports can stay in their narrower follow-up issues.

## Reproduction Assessment

Yes for the original bug: the report gives concrete persistent ACP restart/repair steps and PPID=1 process-table checks, and the old source path lacked process-tree cleanup. I did not reproduce the leak on current main because the inspected source and tests now cover close, startup, and wrapper-orphan cleanup.

## Solution Assessment

Yes. The current-main fix is the maintainable boundary: OpenClaw-owned generated wrappers and lease-gated process-tree cleanup avoid a broad process-name reaper while covering shutdown, close, and stale startup orphans.
