## Summary

Keep open: current main and v2026.5.22 address several surrounding failure modes, but the central boolean `channels.telegram.retry.jitter` compatibility problem is still present. The remaining fix needs maintainer direction because schema coercion, strict validation plus `doctor --fix`, and SecretRef/security follow-up are different product and security choices.

## What This Changes

The issue reports a compound operational failure: Telegram retry jitter boolean config causing gateway restart loops, misleading doctor SecretRef diagnostics, CDP helper restart storms, and multi-user state-root confusion.

## Best Possible Solution

Narrow this to the remaining supported upgrade path: either approve explicit schema compatibility for boolean jitter or keep the schema strict and add a `doctor --fix` migration/warning that converts observed boolean jitter to numeric values, while preserving the existing service-loop and SecretRef diagnostic hardening.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.22 still wire Telegram retry config to a Zod numeric-only `jitter` schema, and Zod 4.4.3 rejects non-number input for `z.number()`. I did not run a live gateway because this review is read-only.

## Solution Assessment

No final solution is selected yet: the closed PR's schema coercion would address the parse failure, but repo policy points legacy config repair toward `doctor --fix` unless maintainers approve runtime compatibility. A maintainer should choose that boundary before a new fix PR.
