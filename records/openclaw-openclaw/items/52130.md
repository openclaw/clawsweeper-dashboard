The issue asks OpenClaw to prevent a Telegram-related restart storm caused by boolean `channels.telegram.retry.jitter`, align doctor SecretRef diagnostics with runtime behavior, rate-limit service restart loops, and warn about multi-user state/CDP conflicts.

## Best Possible Solution

Keep the issue open or split it, with the immediate code fix focused on making boolean retry jitter either valid and normalized in the shared retry schema or rejected before startup with an actionable doctor/config repair path and regression tests.

Yes. A config containing `channels.telegram.retry.jitter: true` follows Telegram's `retry: RetryConfigSchema` path into `OpenClawSchema.safeParse`, where current main still accepts only numeric jitter values.

## Solution Assessment

No for current main. The existing hardening reduces surrounding operational damage, but the narrow maintainable fix is still to handle boolean retry jitter at the config compatibility/repair boundary without changing Telegram runtime retry behavior.
