## Summary

Keep open: current main still has a source-proven boolean `channels.telegram.retry.jitter` validation hole. Telegram retry config still reaches the shared number-only retry schema, and the doctor compatibility migrations do not repair boolean jitter values; the related attempted fix was closed unmerged.

## What This Changes

The issue asks OpenClaw to prevent Telegram retry-jitter boolean configs, misleading SecretRef diagnostics, and service helper restart loops from cascading into an apparent hung state.

## Best Possible Solution

Add a Telegram doctor compatibility migration that rewrites boolean `retry.jitter` values to the numeric contract at top-level and account scopes, while leaving broader SecretRef/CDP/recovery hardening to their narrower tracks.

## Reproduction Assessment

Yes, at source level. A config with `channels.telegram.retry.jitter: true` reaches `RetryConfigSchema` through the Telegram schema, and that schema still accepts only numbers while doctor migrations have no boolean repair.

## Solution Assessment

Yes for a focused repair. The maintainable fix is a doctor-owned compatibility migration that preserves the runtime numeric schema instead of loosening the canonical config contract.
