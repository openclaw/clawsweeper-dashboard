## Summary

Keeping this open: current main has mitigations for systemd restart limiting, Telegram doctor SecretRef warnings, CDP stale SingletonLock cleanup, and multiple-gateway diagnostics, but the concrete `channels.telegram.retry.jitter: true` config remains unhandled and there is no doctor legacy migration for it. The directly related fix PR was closed unmerged, so a focused follow-up is still useful.

## What This Changes

The issue asks OpenClaw to prevent a Telegram-related restart storm from boolean `retry.jitter`, align doctor SecretRef diagnostics with runtime behavior, limit service restart loops, isolate CDP profile state, and warn about multiple active state roots.

## Best Possible Solution

Add a doctor-owned legacy migration and regression coverage that rewrites boolean Telegram retry jitter to the numeric contract, while preserving the public numeric schema and leaving broader runtime recovery work on its narrower canonical tracks.

## Reproduction Assessment

Yes, at source level. The current Telegram config path reaches `RetryConfigSchema`, and that schema still requires `retry.jitter` to be a number while no doctor migration handles the boolean form.

## Solution Assessment

No, not as the old schema-coercion PR proposed. The safer repository-aligned fix is a `doctor --fix` legacy migration or actionable repair path that keeps runtime config validation strict and numeric.
