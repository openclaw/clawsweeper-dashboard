## Summary

Keep open. Current main has several operational hardening pieces, but the remaining Telegram config path still rejects boolean `channels.telegram.retry.jitter` before startup can proceed or doctor can repair it.

## What This Changes

The issue asks OpenClaw to prevent Telegram restart storms from boolean `retry.jitter`, align doctor SecretRef diagnostics with runtime behavior, harden CDP profile locking, and warn about multi-runtime state drift.

## Best Possible Solution

Keep the existing service, doctor, and browser hardening, then add a narrow raw-config or doctor repair path for boolean `retry.jitter` with regression coverage while preserving numeric public config docs.

Yes, at source level. A config with `channels.telegram.retry.jitter: true` reaches `OpenClawSchema.safeParse`, while Telegram still points at a number-only `RetryConfigSchema`; I did not run a live gateway in this read-only review.

## Solution Assessment

No, current main is not the complete fix. The restart-loop and diagnostics hardening reduce the operational blast radius, but the narrow remaining fix is to normalize or repair boolean retry jitter before schema validation rejects it.
