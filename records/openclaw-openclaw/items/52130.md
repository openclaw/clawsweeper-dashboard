## Summary

Keeping this open: current main still rejects `channels.telegram.retry.jitter: true`, and the focused doctor migration for that legacy/invalid value has not landed. Several surrounding symptoms now have mitigations, but the remaining jitter repair is still concrete and narrow.

## What This Changes

The issue asks OpenClaw to avoid restart storms and misleading diagnostics when Telegram retry jitter is boolean, Telegram bot-token SecretRefs resolve differently between doctor and runtime, CDP profile locks fail, and multiple state roots are active.

## Best Possible Solution

Add a doctor-owned legacy migration and regression coverage that rewrites boolean Telegram retry jitter to the numeric contract while keeping runtime schema validation strict.

## Reproduction Assessment

Yes, at source level. Current main routes `channels.telegram.retry` to `RetryConfigSchema`, and that schema still rejects boolean `jitter` while doctor has no migration for the boolean form.

## Solution Assessment

Yes for a focused doctor repair, not for a broad schema-loosening or watchdog bundle. The narrow maintainable fix is to migrate legacy boolean values before validation and leave unrelated recovery work on its existing tracks.
