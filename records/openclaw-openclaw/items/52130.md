## Summary

Keep open: current main has mitigations for the SecretRef diagnostic and systemd restart-loop parts, but `channels.telegram.retry.jitter: true` still reaches a number-only schema and no doctor migration repairs that value.

## What This Changes

The issue asks OpenClaw to prevent Telegram retry-jitter config type drift from causing restart storms and to align Telegram SecretRef diagnostics with runtime behavior.

## Best Possible Solution

Add a doctor-owned legacy migration and regression coverage that rewrites boolean Telegram retry jitter to the numeric contract while keeping runtime schema validation strict.

## Reproduction Assessment

Yes, at source level. Current main routes `channels.telegram.retry` to `RetryConfigSchema`, which still rejects boolean `jitter`, and doctor has no migration for that boolean form.

## Solution Assessment

Yes for a focused doctor repair. The narrow maintainable path is to migrate the legacy/invalid boolean value before validation, not to bundle broader watchdog, CDP, or SecretRef behavior into one change.
