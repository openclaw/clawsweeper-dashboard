## Summary

Keep open: current main still rejects boolean Telegram `retry.jitter` through the shared numeric retry schema, and neither the Telegram nor shared doctor compatibility path repairs that value before validation. Some service restart limiting and SecretRef handling moved since the report, but the remaining jitter crash path is still source-proven.

## What This Changes

The issue asks OpenClaw to stop Telegram config restart storms from boolean `retry.jitter`, align doctor SecretRef diagnostics with runtime token resolution, and surface restart/CDP multi-user failures more clearly.

## Best Possible Solution

Add a doctor-owned compatibility migration for boolean Telegram `retry.jitter` at top-level and account scopes, keep the canonical runtime schema numeric, and handle SecretRef/CDP recovery only through narrower proven follow-ups.

## Reproduction Assessment

Yes, at source level. `channels.telegram.retry.jitter: true` still reaches `RetryConfigSchema.jitter`, which is number-only, and the inspected doctor migration paths do not normalize the boolean first.

## Solution Assessment

Yes for the remaining narrow bug if implemented as a doctor compatibility repair. Runtime schema coercion would blur the canonical numeric config contract; `openclaw doctor --fix` is the safer upgrade path under repo policy.
