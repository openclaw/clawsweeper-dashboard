## Summary

Keep open: current main mitigates the systemd restart-loop and Telegram SecretRef diagnostic parts, but `channels.telegram.retry.jitter: true` still reaches a number-only schema and no doctor legacy migration repairs that value.

## What This Changes

The issue requests hardening for a Telegram `retry.jitter` boolean config mismatch, misleading Telegram SecretRef doctor diagnostics, service restart storms, and multi-user state-root warnings.

## Best Possible Solution

Keep the runtime retry schema numeric and add a doctor legacy migration plus regression coverage that rewrites boolean Telegram retry jitter to the numeric contract.

## Reproduction Assessment

Yes, at source level. Current main wires Telegram `retry` to `RetryConfigSchema`, whose `jitter` field is still `z.number()`, and the doctor channel migrations have no boolean jitter repair.

## Solution Assessment

Yes for a focused repair. A doctor-owned legacy migration is safer than loosening the runtime schema because config repair belongs in `openclaw doctor --fix` while runtime paths keep the canonical numeric contract.
