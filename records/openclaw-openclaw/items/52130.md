Fix the remaining boolean retry jitter compatibility bug from https://github.com/openclaw/openclaw/issues/52130. Current main still rejects `channels.telegram.retry.jitter: true` because Telegram uses `RetryConfigSchema` and `jitter` is numeric-only. Keep the fix narrow: accept or repair legacy boolean retry jitter for channel retry config (`true` -> default jitter factor, `false` -> 0) without adding a new config option or changing retry runtime behavior. Prefer existing config/doctor compatibility patterns. Add focused regression coverage for Telegram and any shared-schema channel coverage that falls out naturally. Do not attempt the broader SecretRef architecture, rescue watchdog, CDP service redesign, or multi-user runtime policy in this PR. Include a changelog entry because this is user-visible config compatibility.
