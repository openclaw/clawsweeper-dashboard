## Summary

Keep open: this is a real stale session/message-delivery fix with strong proof, but the gateway drop guard currently runs after the rebound session has already been touched by the agent handler, leaving a session-state side effect to fix before merge.

## What This Changes

The PR threads approval-time session IDs through exec approval follow-up targets, adds an optional gateway protocol field, drops rebound follow-ups, updates Swift protocol models, and adds regression tests.

## Best Possible Solution

Keep the session-bound follow-up guard, but make the gateway decide stale-versus-deliver before any session-store mutation or accepted-run side effects.

## Reproduction Assessment

Yes, source-reproducible. Current main follows up by sessionKey while /new and /reset rotate the stored sessionId, and the PR body includes before/after gateway logs for approved and denied paths.

## Solution Assessment

No. The session-id guard is the right approach, but the gateway path should decide to drop before session-store mutation rather than after accepted-run setup has begun.
