## Summary

Keep open: current main has a credible source-level path where a cron job can request `toolsAllow: ["exec"]` and resolve onto the Codex app-server runtime without any usable exec-capable tool being exposed, so this is not conservative cleanup-close material.

## What This Changes

The issue reports that isolated cron agent-turn jobs with a requested model and `toolsAllow: ["exec"]` can start without an exec tool, leaving `toolCount` at 0 and preventing the local command from running.

## Best Possible Solution

After cron resolves the model/runtime, validate whether the selected runtime can actually satisfy the requested allowlist; expose a safe exec-capable path when supported, otherwise fail before the model call with an explicit cron/runtime error instead of starting a no-tool Codex turn.

## Reproduction Assessment

Yes, at source level: an isolated cron run can carry `toolsAllow: ["exec"]` into the Codex app-server runtime, where narrow allowlists disable native code mode and default dynamic-tool filtering excludes OpenClaw `exec`. I did not run the live cron job because this review is read-only.

## Solution Assessment

Unclear without maintainer policy input: making `exec` work under Codex may require a safe bridge or sandbox-backed tool, while unconditional native-code-mode enablement could widen the tool boundary. A fail-fast validation error is the safer minimum if the runtime cannot honor the allowlist.
