## Summary

Keep open: current main still has a credible source-level path where an isolated cron job can request `toolsAllow: ["exec"]`, resolve to the Codex app-server runtime, and start without any exec-capable tool. This needs a maintainer policy choice, not cleanup close.

## What This Changes

The issue reports isolated cron `agentTurn` jobs with `payload.model` and `toolsAllow: ["exec"]` resolving to Codex with `toolCount: 0` and failing before the requested command runs.

## Best Possible Solution

After cron resolves the runtime, validate whether the selected Codex runtime can satisfy the requested allowlist; expose a safe exec-capable path when supported, otherwise fail before the model turn with a clear cron/runtime error.

## Reproduction Assessment

Yes, at source level: an isolated cron run can forward `toolsAllow: ["exec"]` into Codex, where non-wildcard allowlists disable native code mode and default dynamic-tool filtering excludes OpenClaw `exec`. I did not run a live cron job in this read-only review.

## Solution Assessment

Unclear until maintainers choose the Codex runtime policy. The safe minimum is fail-fast validation when the runtime cannot honor the allowlist; exposing exec should use a sandbox-backed or otherwise explicitly approved path.
