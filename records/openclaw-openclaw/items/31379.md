## Label Justifications

- `P2`: The issue affects user-facing channel discovery in the Control UI but does not break active message delivery or core runtime availability.
- `impact:other`: Users can lose visibility of built-in channel setup cards after enabling only a custom channel plugin.

## Summary

Keep open: current main and latest release still treat `channelMeta` or `channelOrder` as the complete Channels card list, so a custom-only snapshot can still hide built-in channel cards. The existing open implementation candidate is fork-local, so maintainers should review/adopt/replace it rather than close this issue.

## What This Changes

The issue requests that the Control UI Channels page always render built-in channel cards alongside snapshot-provided custom channel cards.

## Best Possible Solution

Land or replace a narrow UI resolver fix that appends missing built-in channel IDs after snapshot-provided channels, with regression coverage and real Control UI proof, while keeping backend `channels.status` semantics unchanged.

## Reproduction Assessment

Yes, source-level: current main maps exactly the resolver output to channel cards, and the resolver returns a custom-only `channelMeta` or `channelOrder` list as complete. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes, the narrow UI resolver merge remains the best fix shape: preserve backend loaded-plugin semantics, keep snapshot/custom channels first, append missing built-in discovery cards once, and cover custom-only plus mixed-order snapshots.
