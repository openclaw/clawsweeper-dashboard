## Label Justifications

- `P2`: The issue affects user-facing channel discovery in the Control UI but does not break active message delivery or core runtime availability.
- `impact:other`: Users can lose visibility of built-in channel setup cards after enabling only a custom channel plugin.

## Summary

Keep open: current main and the latest release still let a custom-only Channels snapshot become the full rendered card list, so built-in channel cards can still disappear. The open linked implementation candidate at https://github.com/BingqingLyu/openclaw/pull/544 uses closing syntax and should be reviewed, landed, or replaced before this issue closes.

## What This Changes

The issue requests that the Control UI Channels settings page show built-in channel cards alongside custom channel plugins even when the backend snapshot only lists the custom channels.

## Best Possible Solution

Land or replace the linked UI resolver fix so custom snapshot channels stay first, missing built-in channel cards are appended once, and regression coverage proves custom-only plus mixed-order snapshots without changing backend status semantics.

## Reproduction Assessment

Yes, source-level: current main maps resolveChannelOrder() directly to rendered cards, and the resolver returns a custom-only channelMeta or channelOrder list as complete. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes, the frontend resolver merge remains the narrowest maintainable solution: preserve backend loaded-plugin semantics, keep custom channels first, and append missing built-in discovery cards with de-duplication.
