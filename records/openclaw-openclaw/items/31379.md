## Label Justifications

- `P2`: The issue affects user-facing channel discovery in the Control UI but does not break active message delivery or core runtime availability.
- `impact:other`: Users can lose visibility of built-in channel setup cards after enabling only a custom channel plugin.

## Summary

Keep open: current main and v2026.6.6 still treat snapshot `channelMeta` or `channelOrder` as the complete Channels card list, so a custom-only channel snapshot can still hide built-in cards. A linked open implementation candidate exists at https://github.com/BingqingLyu/openclaw/pull/544, so the issue should wait for that PR to land or be replaced.

## What This Changes

Requests that the Control UI Channels settings page keep built-in channel cards visible alongside custom channel plugins.

## Best Possible Solution

Land or replace the linked UI resolver fix so snapshot-provided channels stay first, missing built-in channel cards are appended once, and regression coverage proves custom-only plus mixed-order snapshots without changing backend status semantics.

## Reproduction Assessment

Yes, source-level: current main maps exactly the resolver output to channel cards, and the resolver returns a custom-only `channelMeta` or `channelOrder` list as complete. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

Yes, the frontend resolver merge remains the narrowest maintainable solution: preserve backend loaded-plugin semantics, keep custom channels first, and append missing built-in discovery cards with de-duplication.
