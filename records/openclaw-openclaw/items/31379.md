## Label Justifications

- `P2`: The latest release has a bounded Control UI bug that hides built-in channel configuration choices but does not disrupt message transport.
- `impact:other`: The issue affects integration discovery and configuration visibility without fitting the delivery, state, auth, security, or availability categories.

## Summary

Keep open: current main and v2026.6.6 still treat non-empty snapshot channel metadata as the complete card list, so a custom-only snapshot hides built-in channel cards; an open linked PR remains the implementation candidate.

## What This Changes

Keep built-in channel discovery cards visible on the Channels settings page when gateway metadata contains custom or configured channels.

## Best Possible Solution

Preserve the gateway's loaded-plugin catalog semantics and update the Control UI resolver to append every missing built-in channel once after snapshot IDs, with custom-only and mixed-order regression coverage.

## Reproduction Assessment

Yes at source level: a custom-only channelMeta array is returned unchanged as the complete order, and renderChannels creates cards only for those IDs. No live gateway and browser reproduction was run during this read-only review.

## Solution Assessment

Yes, the frontend resolver is the narrowest maintainable owner boundary because gateway metadata intentionally describes loaded plugins while the UI owns built-in discovery cards; the merge should preserve order and deduplicate IDs.
