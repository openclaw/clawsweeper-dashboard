## Summary

Keep this PR open for normal review and landing; the patch appears correct and directly addresses a real current-main performance bug. Current main reconciles inspectable tasks twice during one status summary, while the PR changes that to a single per-call snapshot with backward-compatible helper fallbacks and focused regression coverage. Reviewed item: https://github.com/openclaw/openclaw/pull/73650

## What This Changes

The PR adds optional reconciled task snapshot parameters to the task registry and audit summary helpers, updates status summary generation to reconcile once and pass the shared snapshot, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land this PR or an equivalent narrow main-branch fix after exact-head validation: status should reconcile inspectable tasks once per summary build, derive registry and audit summaries from that snapshot, and keep zero-argument helper behavior for tasks and gateway call sites.

## Reproduction Assessment

Yes. The current-main reproduction is source-level: one getStatusSummary() call reaches both task summary helpers, and each helper independently calls reconcileInspectableTasks(), which reaches listTaskRecords() clone/sort work.

## Solution Assessment

Yes. A per-status-call shared snapshot is the narrowest maintainable fix because it removes the duplicated work without adding module-level cache invalidation, TTL, or stale-reference behavior.
