## Summary

Keep open. Current main still runs legacy iMessage through Gateway/Node-spawned `imsg rpc` for probe, send, and monitor paths, and the docs still tell users to grant Full Disk Access to the process context running OpenClaw/`imsg`; the related open PR improves permission-denied handling but does not provide the requested stable macOS TCC process boundary.

## What This Changes

Request that the legacy iMessage `imsg` integration access `~/Library/Messages/chat.db` from a stable `imsg` or helper process identity instead of requiring Full Disk Access for the parent Node/OpenClaw process.

## Best Possible Solution

The durable end state should either run legacy `imsg` database access behind a stable signed OpenClaw/`imsg` helper or app-owned XPC boundary with updated TCC docs, or explicitly retire the legacy path in favor of BlueBubbles with migration diagnostics.

## Reproduction Assessment

No live current-main reproduction was established because it needs macOS TCC state, `~/Library/Messages/chat.db`, an `imsg` FDA grant, and a Homebrew Node resolved-path change. Static source inspection does confirm the requested stable process boundary is still missing because OpenClaw spawns `imsg rpc` from the Gateway/Node process.

## Solution Assessment

Unclear. The request targets a real failure mode, but the best implementation requires maintainer architecture/security judgment among a signed helper, app/XPC-owned execution, LaunchAgent changes, or retiring legacy `imsg` in favor of BlueBubbles.
