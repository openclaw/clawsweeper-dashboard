## Summary

Keep open. Current main and the latest release still use the gateway/Node-launched `imsg` process model, and the remaining request is a security-sensitive macOS TCC architecture decision rather than a stale or already-implemented cleanup item.

## What This Changes

Request that the iMessage `imsg` path read `chat.db` from a stable process context so Homebrew Node upgrades do not invalidate the required Full Disk Access grant.

## Best Possible Solution

Define the supported macOS TCC model for iMessage, then either route `imsg` through a stable signed user-session broker/app context or document the parent-launcher FDA requirement with a supported workaround.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence reproduction was run, but source inspection gives a credible current-main path: configure iMessage, run the gateway under a Homebrew Node or LaunchAgent parent, and OpenClaw still spawns `imsg rpc` from that process context while docs/upstream require parent-launcher FDA.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested direction may be right, but the maintainable fix needs maintainer/security judgment on signed app mediation, LaunchAgent support, upstream `imsg` behavior, or explicit docs before automation changes this surface.
