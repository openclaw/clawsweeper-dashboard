## Summary

Keep open. Current main still spawns legacy `imsg rpc` from the Gateway/Node process and the docs still require Full Disk Access for that process context; the related open PR only improves permission-denied handling, not the requested stable macOS TCC identity.

## What This Changes

Request legacy iMessage `chat.db` access to run in `imsg`'s own stable process context so Homebrew Node upgrades do not invalidate Full Disk Access.

## Best Possible Solution

The durable end state is either a stable signed OpenClaw/`imsg` helper or app-owned XPC/LaunchAgent boundary for legacy `imsg` database access with updated docs, or an explicit product decision to retire that legacy path in favor of BlueBubbles.

## Reproduction Assessment

No live macOS TCC reproduction was established; it requires Messages `chat.db`, FDA state, an `imsg` grant, and a Homebrew Node resolved-path change. Source inspection does confirm the requested boundary is absent because probe, send, and monitor all create an `IMessageRpcClient` that spawns `imsg rpc` from Node.

## Solution Assessment

Unclear. The request targets a real failure mode, but the best implementation requires choosing among a signed helper, app/XPC-owned execution, LaunchAgent changes, or retiring legacy `imsg`, rather than a narrow autonomous bug fix.
