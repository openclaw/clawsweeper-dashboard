## Summary

Keep open. Current main still runs legacy iMessage by spawning `imsg rpc` from the OpenClaw Gateway/Node process, and the docs still tell users that Full Disk Access must be granted to the process context running OpenClaw/`imsg`; there is no implemented stable signed helper, XPC bridge, app-owned service, or deprecation path that resolves the Node-path TCC churn described here.

## What This Changes

The issue requests that legacy `imsg` read `~/Library/Messages/chat.db` under its own stable process identity so an `imsg` Full Disk Access grant is sufficient after Node/Homebrew upgrades.

## Best Possible Solution

The durable end state should either move legacy `imsg` database access behind a stable signed OpenClaw- or `imsg`-owned process boundary with explicit TCC documentation, or make a maintainer-backed decision to retire the legacy `imsg` path in favor of BlueBubbles with migration guidance and clear permission-denied diagnostics.

## Reproduction Assessment

No high-confidence live reproduction was performed. Static inspection is enough to confirm the requested process-boundary change is still missing, but the reported failure itself requires macOS TCC state, a real `~/Library/Messages/chat.db`, an `imsg` FDA grant, and a Homebrew Node path change.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation depends on maintainer choices around signed helpers, XPC/app-owned execution, LaunchAgent behavior, least-privilege FDA, and whether the legacy `imsg` channel should continue beside the recommended BlueBubbles path.
