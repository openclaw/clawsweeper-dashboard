## Summary

Keep open. Current main still runs legacy iMessage through Gateway/Node-spawned `imsg rpc` for probe, send, and monitor paths, and the docs still require Full Disk Access for the process context running OpenClaw/`imsg`; the related open PR only improves permission-denied handling, not the requested stable TCC process boundary.

## What This Changes

Make legacy iMessage `imsg` database access rely on an `imsg`- or OpenClaw-owned process identity instead of the parent Node process's Full Disk Access grant.

## Best Possible Solution

The durable end state should either put legacy `imsg` database access behind a stable signed OpenClaw/`imsg` helper or app-owned XPC boundary with clear TCC docs, or explicitly retire the legacy path in favor of BlueBubbles with migration and diagnostics.

## Reproduction Assessment

No. A live reproduction needs macOS TCC state, Messages `chat.db`, an `imsg` FDA grant, and a Homebrew Node resolved-path change; static inspection does confirm the requested process-boundary implementation is still missing on current main.

## Solution Assessment

Unclear. The request targets the right failure mode, but the best implementation requires maintainer architecture/security judgment around signed helpers, XPC/app-owned execution, LaunchAgent behavior, and whether legacy `imsg` should continue.
