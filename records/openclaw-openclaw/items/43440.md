## Summary

Close as duplicate/superseded: the parent-Node Full Disk Access problem is the same macOS TCC responsible-process failure now tracked by the broader open LaunchAgent/FDA/Automation issue, while the related open PR only improves permission-denied diagnostics and does not implement process isolation.

## What This Changes

The issue asks for iMessage database access to rely on the `imsg` process Full Disk Access grant instead of the parent Node process grant.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/51299 as the canonical place to decide the supported signed-app/session-broker/MDM or unsupported-headless stance, and preserve this issue's Homebrew/Node-upgrade detail there.

## Reproduction Assessment

Yes at source level, but not with live macOS validation: current main runs `imsg rpc` as a Node/gateway child, and upstream imsg documents the same parent-launcher FDA behavior after Homebrew, Node, or app updates. I did not reproduce TCC prompts on macOS in this checkout.

## Solution Assessment

No, keeping this separate satellite open is not the best path. The maintainable route is to track the shared macOS TCC responsible-process decision in https://github.com/openclaw/openclaw/issues/51299 and treat https://github.com/openclaw/openclaw/pull/72898 as separate diagnostic hardening.
