## Summary

Keep open. Current main still launches `imsg rpc` as a child of the OpenClaw Node/gateway process and the docs still require Full Disk Access for the process context running OpenClaw/`imsg`, so the requested parent-process-independent TCC model is not implemented and needs maintainer/security product direction.

## What This Changes

Request that the iMessage integration make `imsg` access Messages data in an authorization context where the `imsg` binary's own Full Disk Access grant is sufficient, instead of depending on the parent Node process grant.

## Best Possible Solution

Define a supported iMessage TCC architecture: either route `imsg` through a signed macOS app/session broker, document the parent-launcher FDA requirement as the supported model, or explicitly mark headless LaunchAgent use unsupported until a secure broker exists.

## Reproduction Assessment

No live high-confidence reproduction was established in this Linux checkout. Source inspection does show the current-main path that would reproduce the reported design problem on macOS: the gateway spawns `imsg rpc` as a child process and docs require FDA for that same process context.

## Solution Assessment

Unclear. The requested own-process context is a plausible product direction, but a signed app/session broker or explicit unsupported stance is safer than a small plugin-local change because this touches macOS TCC and Automation consent.
