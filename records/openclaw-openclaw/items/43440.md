## Summary

Keep open. Current main still launches `imsg rpc` as a Gateway/Node child process and the docs still require Full Disk Access for that process context, while the related open PR only improves fatal permission-denied handling rather than providing a stable macOS TCC identity.

## What This Changes

The issue requests that iMessage database access run under a stable `imsg`-owned macOS process identity so Homebrew Node upgrades do not invalidate Full Disk Access grants.

## Best Possible Solution

Choose a supported macOS iMessage boundary: route legacy `imsg` database access through a stable signed helper/app-owned TCC context or document a supported app-wrapped/unsupported LaunchAgent stance, then update iMessage and macOS permission docs with validation proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does confirm that current main still lacks the requested boundary because probe, send, chat actions, and monitor startup all create `IMessageRpcClient`, which spawns `imsg rpc` from the Gateway/Node process.

## Solution Assessment

Unclear. The request targets a real failure mode, but the best fix requires choosing among a signed helper, app/XPC-owned execution, app-wrapped LaunchAgent guidance, or a documented unsupported stance rather than a narrow code patch.
