## Summary

Current main still uses the gateway Node process to spawn `imsg rpc`, and the public iMessage docs still require Full Disk Access for the OpenClaw/`imsg` process context. The related open PR improves permission-denied detection only; the requested TCC process-context redesign is still unresolved and needs maintainer/security product judgment.

## What This Changes

The issue requests that the iMessage integration read Messages chat.db under the `imsg` process’s own macOS Full Disk Access grant instead of depending on the parent Node process’s TCC grant.

## Best Possible Solution

Define a supported iMessage permission model for headless macOS installs, then either route `imsg` through a stable signed GUI/session context or document the current parent-launcher FDA requirement and a safer supported workaround.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection does show the current design that triggers the report: OpenClaw starts `imsg rpc` as a child process and docs require FDA for the OpenClaw/`imsg` process context.

## Solution Assessment

Unclear. The requested direction could be right, but the maintainable fix needs a maintainer/security decision about signed app mediation, LaunchAgent support, upstream `imsg` behavior, or explicit docs before automation should change this surface.
