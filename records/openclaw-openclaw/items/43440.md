## Summary

Keep open. Current main still launches `imsg rpc` from the Gateway/Node process and the docs still require Full Disk Access for that process context; the related open PR improves permission-denied handling but does not provide the requested stable macOS TCC identity.

## What This Changes

The issue requests that iMessage `chat.db` access run under an `imsg`-owned or otherwise stable process identity so Homebrew Node upgrades do not require re-granting Full Disk Access to the parent Node process.

## Best Possible Solution

Choose a supported macOS TCC owner for legacy `imsg` access, such as a stable signed helper or app-owned IPC/LaunchAgent boundary, then update iMessage runtime handling, docs, and focused tests accordingly.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection does show the current boundary: iMessage probe/send/watch paths still spawn `imsg rpc` from the Gateway/Node process while docs require FDA for that process context.

## Solution Assessment

Unclear. The requested direction addresses a real TCC stability problem, but the best implementation needs a maintainer choice between signed helper, app/XPC-owned execution, LaunchAgent changes, or a documented support boundary.
