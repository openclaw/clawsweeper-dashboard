## Summary

Keep open. Current main still runs `imsg rpc` as a gateway-spawned child process and both OpenClaw plus upstream `imsg` docs still require Full Disk Access for the launching process context; the related open PR improves fatal permission-denied diagnostics but does not solve the requested TCC process-context redesign.

## What This Changes

The issue requests that the iMessage plugin access `chat.db` in an `imsg`-owned macOS TCC process context so Homebrew Node upgrades do not invalidate the parent process Full Disk Access grant.

## Best Possible Solution

Define the supported macOS TCC model for iMessage, then either route `imsg` through a stable signed GUI/session broker or document the parent-launcher FDA requirement with a safer supported workaround.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection gives a credible current-main path: configure iMessage, run the gateway as the parent process or LaunchAgent, and OpenClaw still spawns `imsg rpc` from that process while docs/upstream require FDA for that launcher context.

## Solution Assessment

Unclear. The requested direction may be right, but the maintainable fix needs a maintainer/security decision about signed app mediation, LaunchAgent support, upstream `imsg` behavior, or explicit docs before automation changes this surface.
