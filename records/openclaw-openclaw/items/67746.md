## Summary

Keep open. Current main still has the reported CLI/runtime mismatch: CLI logout clears WhatsApp auth through the plugin-local helper, but does not call the live gateway logout path that stops the running channel runtime. The related fix PR was closed unmerged, and `channels remove` currently mutates config only without coordinating with the running gateway session.

## What This Changes

The issue requests WhatsApp logout/remove cleanup to stop the live gateway session, remove or disable persisted channel config as intended, and clear WhatsApp Web credentials without manual file edits or process kills.

## Best Possible Solution

Route CLI logout through the generic gateway `channels.logout` RPC when a local gateway is reachable, keep an explicit local credential-cleanup fallback for unreachable default-local gateways, and add remove-path coordination so disabling/deleting a channel account cannot leave an already-running gateway listener active. Preserve documented `--delete` config deletion semantics unless maintainers intentionally change the CLI UX.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with a running gateway, `openclaw channels logout --channel whatsapp` follows `src/cli/channel-auth.ts` and calls the local plugin logout helper instead of the gateway `channels.logout` method that stops the active runtime.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing generic gateway channel lifecycle seam rather than adding WhatsApp-specific core behavior. The repair should also cover `channels remove` because config mutation alone does not stop the live in-memory runtime.
