## Summary

Keep this PR open: the latest head addresses the prior explicit OpenAI-only bypass, but the new explicit-order path can still forward an unusable stored Codex profile before an OpenAI backup, and the supplied real behavior proof still does not show the Pi runtime auth-routing path.

## What This Changes

The PR adds OpenAI Pi auth-profile selection for `openai-codex` entries in OpenAI auth order, external CLI auth scope handling for profile IDs, a `models auth login --device-code` flag with docs, and focused regression tests.

## Best Possible Solution

Make the OpenAI Pi selector walk `auth.order.openai` with the same eligibility and cooldown semantics as auth-profile order resolution, then keep the device-code CLI/docs additions with focused tests and redacted Pi runtime proof.

## Reproduction Assessment

Yes, source-level. Current main returns the normal OpenAI auth provider for an OpenAI Pi run without a session auth profile, and the linked issue provides a concrete Codex-first `auth.order.openai` configuration where that skips the Codex transport.

## Solution Assessment

No. The direction is right and the prior explicit OpenAI-only bypass was patched, but the helper should validate candidate eligibility before forwarding and the PR still needs real Pi runtime proof.
