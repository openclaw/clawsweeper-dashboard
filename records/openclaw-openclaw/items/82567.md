## Summary

Keep open. The PR targets a valid documented bug and adds relevant tests plus a CLI alias, but the current diff can select a stored Codex OAuth profile ahead of an explicit OpenAI-only auth order and still lacks real behavior proof for the main Pi routing path.

## What This Changes

The PR adds OpenAI Pi auth-profile selection logic, external CLI scope handling for profile IDs, a `models auth login --device-code` flag/docs, and regression tests.

## Best Possible Solution

Keep the PR open, make the OpenAI auth order authoritative for Pi selection, add a regression for OpenAI-only order with a stored Codex profile, and provide redacted real-run proof.

## Reproduction Assessment

Yes, source-level. The linked issue has live before/after configuration evidence, and current source/tests show the ordering edge that this PR would mishandle without needing private credentials.

## Solution Assessment

No. The direction is right, but the helper should walk the OpenAI provider's ordered candidates and select Codex only when that candidate appears before the first OpenAI-compatible profile, rather than reusing generic `openai-codex` ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve OpenAI-only auth order before selecting Codex:** `src/agents/command/auth-profile-selection.ts:37-43`
  - body: This resolves candidates by asking the `openai-codex` provider for its order. Existing auth-order behavior prepends stored native Codex profiles when `auth.order.openai` contains only `openai:default`, so an OpenAI Pi run with a stored `openai-codex:*` profile and an explicit OpenAI-only order would still auto-select Codex. That breaks the advertised order preservation and can route prompts through the wrong auth profile; resolve the OpenAI order directly or only select Codex IDs that appear before the first OpenAI-compatible entry.
  - confidence: 0.91
