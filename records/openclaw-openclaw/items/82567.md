## Summary

Keep open: the linked bug is valid and this PR targets the right auth/CLI surface, but the latest helper still bypasses existing auth-order eligibility semantics and the supplied proof does not show the main OpenAI Pi runtime path after the fix.

## What This Changes

The PR changes OpenAI Pi auth-profile selection so `auth.order.openai` can forward `openai-codex:*` OAuth profiles through the Codex transport, adds a `models auth login --device-code` CLI flag, updates docs, and adds focused tests.

## Best Possible Solution

Reuse or extend the existing auth-order resolver semantics for the OpenAI Pi compatibility route, keep the device-code CLI/docs addition, and require redacted runtime proof for the Pi auth path.

## Reproduction Assessment

Yes, source-level: configure `openai/gpt-5.5` with PI runtime and `auth.order.openai` starting with `openai-codex:*`; current main returns the normal OpenAI auth path unless a Codex profile is already selected. The linked issue also provides matching live `/status` evidence.

## Solution Assessment

No. The direction is right, but the explicit-order branch should walk to the next usable ordered profile with the same eligibility and cooldown rules as the existing resolver, and the PR still needs real Pi runtime proof.
