## Summary

Keep this PR open: the latest head addresses the CLI flag and some auth-order cases, but the Pi selection helper still has a blocking OpenAI-only order edge and the main auth-routing behavior lacks after-fix real-run proof.

## What This Changes

The PR adds OpenAI Pi auth-profile selection for openai-codex entries in OpenAI auth order, external CLI scope handling for profile IDs, a `models auth login --device-code` flag/docs, and regression tests.

## Best Possible Solution

Keep the PR focused on making `auth.order.openai` authoritative for OpenAI Pi selection, add the OpenAI-only regression case, keep the device-code CLI/docs alignment, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes, source-level: the linked issue has live before/after config evidence, and current main's alias-order test shows the exact ordering edge that the PR helper still inherits.

## Solution Assessment

No. The direction is right, but the helper should walk the OpenAI auth order directly, or select Codex only when a Codex entry appears before the first OpenAI-compatible entry, instead of resolving the generic `openai-codex` order.
