## Summary

Keep open: the PR targets a real documented auth-order bug and covers much of it, but the latest helper still forwards the first existing `openai-codex:*` profile from explicit `auth.order.openai` without the existing eligibility/cooldown checks, and the supplied proof does not show the OpenAI Pi runtime path after the patch.

## What This Changes

The PR adds an OpenAI Pi auth-profile selection helper, external CLI auth-scope updates, a `models auth login --device-code` flag with docs, and regression coverage for Codex-first OpenAI auth order.

## Best Possible Solution

Reuse the existing auth-order eligibility/cooldown resolver when selecting from explicit `auth.order.openai`, keep the CLI flag/docs/tests, and require redacted real OpenAI Pi runtime output before merge.

## Reproduction Assessment

Yes, source-level. Current main returns the normal OpenAI auth provider for an OpenAI Pi run without a selected session profile, and the linked issue supplies live configs showing API-key selection despite Codex-first auth order.

## Solution Assessment

No, not yet. The direction is right, but the explicit-order branch must apply existing profile eligibility/cooldown semantics before forwarding and the PR still needs real Pi runtime proof.
