## Summary

Keep open: current main still has the reported hot path for status/session row rendering, where persisted model parsing can enter manifest-backed normalization without config/workspace context and fall back to uncached plugin metadata loading. Related status-performance issues are adjacent, but this report adds a narrower gateway/session-row cache-miss path and fresh profiling evidence.

## What This Changes

The issue requests eliminating repeated plugin manifest normalization/cache misses that make status and session/model listing surfaces block for seconds when many session rows exist.

## Best Possible Solution

Land a narrow hot-path fix that either carries prepared config/workspace/manifest context into model normalization or disables manifest normalization for shallow status/session display paths, and reuse already-built rows where possible.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source plus reporter profiling: current main still has the per-row status/session model-normalization path and the no-context manifest snapshot fallback that the report timed in a real 2026.5.7 install, though I did not run the live timing locally.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction, but the safer implementation should prefer prepared config/workspace context or a deliberate shallow-path manifest opt-out over adding a broad process-wide plugin metadata cache.
