## Summary

Keep open. Current main still treats a non-zero default poll duration as poll intent on `message.send`, so the reported GPT-5.4-style default-filled input can still hit the poll-only guard; Discord all-empty modal defaults also still throw if they reach the Discord component parser.

## What This Changes

The issue requests robust message-tool handling for GPT-5.4-style default-filled optional parameters across poll detection, Discord components, and local media paths.

## Best Possible Solution

Normalize semantic no-op defaults at the message-tool boundary and owner-specific parsers: make shared poll intent depend on real poll content, make Discord treat an all-empty default component object as absent, and keep local media access within the existing agent/workspace trust model.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `message.send` with empty poll content plus `pollDurationHours: 24` still satisfies `hasPollCreationParams` and reaches the send-only poll guard; an all-empty `components.modal.fields: []` object still throws in the Discord parser.

## Solution Assessment

Yes, the general direction in the report is the right fix, but it should be implemented through narrowly scoped normalization rather than broad media-root relaxation. The safest split is shared poll-default handling plus Discord-owned empty component-default handling, with media policy changes avoided unless a focused test proves a remaining gap.
