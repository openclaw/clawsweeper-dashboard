## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire Railway startup to the Docker runtime:** `railway.json:4`
  - body: `railway.json` only selects the Dockerfile builder; it does not execute the new `start.sh`. The final image still uses the existing Dockerfile CMD, so Railway would run `node openclaw.mjs gateway --allow-unconfigured` and never create the Discord/DeepSeek config or switch to port 8080.
  - confidence: 0.93
- **[P2] Write config where OpenClaw can read it:** `start.sh:7`
  - body: Even if this script is wired in, it writes `/root/.openclaw/config.json`. OpenClaw reads `~/.openclaw/openclaw.json` unless `OPENCLAW_CONFIG_PATH` is set, and the Docker runtime runs as `node`, so `/root` is the wrong state home for the image.
  - confidence: 0.91
- **[P2] Keep the generated Discord config valid:** `start.sh:12`
  - body: The script sets `dmPolicy` to `open` without adding `allowFrom: ["*"]`. The Discord schema tests reject that shape, so a wired script would make startup fail validation instead of enabling Discord.
  - confidence: 0.89
- **[P2] Preserve stable Docker cache IDs:** `Dockerfile:65`
  - body: The repo has a cache-layout test requiring `id=openclaw-pnpm-store` for dependency installs. Replacing it with a generated path-like id breaks that contract and loses the stable shared cache key.
  - confidence: 0.88
- **[P3] Keep plugin source edits out of the install layer:** `Dockerfile:33`
  - body: The ext-deps stage is designed to expose the bundled plugin tree only long enough to copy selected `package.json` files, so unrelated plugin source edits do not invalidate `pnpm install`. Copying the whole bundled plugin directory before install undoes that cache boundary.
  - confidence: 0.82
