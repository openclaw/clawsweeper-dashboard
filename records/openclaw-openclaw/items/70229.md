## Summary

Keep open: current main still hardcodes Docker `no-new-privileges` and the linked bug remains open, but this branch needs refresh for the current dangerous-config collector plus real behavior proof and security review before merge.

## What This Changes

The PR adds `sandbox.docker.noNewPrivileges`, threads it through Docker create args, config schema, docs, and tests, and reports `false` as a dangerous configuration value.

## Best Possible Solution

Land a refreshed secure-default compatibility knob, or equivalent, that omits `no-new-privileges` only when explicitly false, documents it as break-glass in docs and config metadata, and reports the downgrade through the core dangerous-config diagnostics.

## Reproduction Assessment

Yes for review purposes: the linked issue gives a focused Docker command that fails only with `--security-opt no-new-privileges`, and current main unconditionally applies that flag before `sleep infinity`. I did not rerun the host-specific Docker/runc failure locally.

## Solution Assessment

Yes in direction, but not as the current branch. A default-on explicit opt-out is narrow, but this PR must move diagnostics and metadata to current main's core paths and needs real behavior proof plus maintainer/security approval.
