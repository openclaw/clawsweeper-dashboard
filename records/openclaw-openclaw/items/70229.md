## Summary

Keep open: current main and the latest release still hardcode Docker `--security-opt no-new-privileges`, and the linked host-compatibility bug remains plausible. The PR is still useful in direction, but the branch is stale/conflicting, puts dangerous-config reporting on the old collector path, and needs real behavior proof plus maintainer/security review before merge.

## What This Changes

The PR adds default-true `sandbox.docker.noNewPrivileges` config, wires it into Docker create args, config schema, docs, and tests, and reports `false` as dangerous.

## Best Possible Solution

Land a refreshed secure-default compatibility knob or equivalent replacement that omits `no-new-privileges` only when explicitly set false, documents it as break-glass, and reports the downgrade through the current core dangerous-config diagnostics.

## Reproduction Assessment

Yes. The linked issue gives a host-level Docker command that fails only with `--security-opt no-new-privileges`, and current main unconditionally appends that flag before running `sleep infinity`; I did not rerun the host-specific EPERM locally.

## Solution Assessment

Yes in direction, but not as the current branch. A default-on explicit opt-out with dangerous-config reporting is the narrowest maintainable path, but the warning must move into the current core collector and the security tradeoff needs maintainer approval.
