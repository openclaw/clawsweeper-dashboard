## Summary

Keep open: current main still has a source-level path where a configured `channels.feishu` can select the built-in `feishu` id for auto-enable, append it to a restrictive `plugins.allow`, and then warn because that plugin id is not installed. The existing ClawSweeper comment already identified the same narrow config ownership path, and the timeline usefully routes the Feishu/Lark surface toward @m1heng as an adjacent owner.

## What This Changes

The issue requests that plugin auto-enable avoid adding the built-in `feishu` plugin id to `plugins.allow` when an installed `openclaw-lark` plugin already owns `channels.feishu`.

## Best Possible Solution

Resolve configured channel ownership from installed/trusted manifest claims before falling back to a built-in channel id, while preserving `preferOver` and explicit user selection semantics.

## Reproduction Assessment

Yes. By source inspection on current main, `channels.feishu` plus an installed manifest owner `openclaw-lark` without an installed `feishu` plugin can still append `feishu` to `plugins.allow`, after which validation emits the reported warning.

## Solution Assessment

Yes for a generic manifest-owner fix, not for a hardcoded `feishu` to `openclaw-lark` mapping. The narrow maintainable path is to choose the installed channel-owning plugin before the built-in id fallback unless `preferOver` or explicit selection says otherwise.
