## Summary

Keep open: the report matches a current source-level path where channel auto-enable can select the built-in channel id as a plugin id, then validation warns when that plugin id is not actually installed. The narrow fix should be generic and manifest-driven, not a hardcoded `openclaw-lark` mapping.

## What This Changes

The issue asks OpenClaw to stop auto-adding `feishu` to `plugins.allow` when `channels.feishu` is served by the installed `openclaw-lark` plugin.

## Best Possible Solution

Teach channel auto-enable and validation to respect an installed, trusted manifest owner for a configured channel before appending a built-in channel id to `plugins.allow`, while preserving `preferOver` and explicit user selection semantics.

## Reproduction Assessment

Yes, by source inspection: with `channels.feishu` and a registry that has `openclaw-lark` claiming `feishu` but no installed `feishu` plugin, auto-enable can still append `feishu` to `plugins.allow`, and validation then emits the reported warning. I did not run a live Gateway because this was a read-only review.

## Solution Assessment

No to a hardcoded map from `feishu` to `openclaw-lark`; the maintainable fix is a generic manifest-owner path that uses installed/trusted channel owners and keeps the documented `preferOver` replacement contract intact.
