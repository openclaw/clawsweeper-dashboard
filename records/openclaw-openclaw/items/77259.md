## Summary

Keep open: current main still has a source-level path where channel auto-enable can append the built-in `feishu` id to a restrictive `plugins.allow`, and validation then warns when no installed plugin manifest has id `feishu`.

## What This Changes

Gateway should stop auto-injecting the built-in `feishu` plugin id into `plugins.allow` when an installed Lark/Feishu plugin such as `openclaw-lark` owns `channels.feishu`.

## Best Possible Solution

Teach channel auto-enable and validation to respect an installed, trusted manifest owner for a configured channel before appending a built-in channel id, while preserving `preferOver`, deny/disable, and explicit user selection semantics.

## Reproduction Assessment

Yes, by source inspection: with `channels.feishu`, a restrictive `plugins.allow`, and an installed manifest id `openclaw-lark` claiming channel `feishu` but no installed `feishu` manifest, auto-enable can append `feishu` and validation emits the reported warning. I did not run a live Gateway in this read-only review.

## Solution Assessment

No to a hardcoded `feishu` to `openclaw-lark` map; the narrow maintainable fix is a generic manifest-owner path that uses installed trusted channel ownership and preserves existing `preferOver` fallback behavior.
