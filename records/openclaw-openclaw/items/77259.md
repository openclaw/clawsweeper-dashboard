## Summary

Keep open: current main still has a source-level path that can select the built-in `feishu` channel id as a plugin id, append it to a restrictive `plugins.allow`, and then warn when no installed plugin has that id. The existing issue comment correctly points toward a generic manifest-owner fix rather than a hardcoded `openclaw-lark` mapping.

## What This Changes

Using `@larksuite/openclaw-lark` with `channels.feishu` should not cause Gateway to inject an unavailable `feishu` plugin id into `plugins.allow` and warn on every startup.

## Best Possible Solution

Teach channel auto-enable to prefer an installed, trusted manifest owner for a configured channel before appending a built-in channel id, while preserving `preferOver`, explicit enable/disable, and restrictive allowlist semantics.

## Reproduction Assessment

Yes, at source level: a `channels.feishu` config plus a registry where `openclaw-lark` claims `feishu` but no installed `feishu` plugin exists drives auto-enable to add `feishu` to `plugins.allow`, then validation emits the reported warning. A focused unit test can reproduce this without the live third-party package.

## Solution Assessment

Yes, the best fix is a generic manifest-owner path shared with the existing doctor ownership policy, not a hardcoded `feishu` to `openclaw-lark` alias or blanket warning suppression.

## Repair Work Prompt

Fix the config auto-enable bug reported at https://github.com/openclaw/openclaw/issues/77259. When `channels.feishu` or another channel is configured and an installed/trusted plugin manifest already owns that channel, do not auto-inject the built-in channel id into `plugins.allow` unless the built-in plugin is the effective owner or explicit user selection requires it. Keep the fix generic and manifest-driven; do not hardcode `openclaw-lark` or Feishu-only aliases. Preserve `preferOver`, explicit enable/disable, denylist, and restrictive allowlist behavior. Add regression coverage with a synthetic manifest fixture for `id: "openclaw-lark"`, `channels: ["feishu"]`, no installed `feishu` plugin, and existing `plugins.allow`, plus any validation/doctor coverage needed to prove the warning is gone. Add a user-facing `CHANGELOG.md` fix entry unless the final change is tests-only.
