## Summary

Keep open: current main still has a source-level path where configured channel auto-enable can choose the built-in `feishu` id for `channels.feishu`, then add that missing id to `plugins.allow` and trigger the reported stale-plugin warning. The earlier ClawSweeper comment is consistent with the current code path; the fix should be generic and manifest-driven, not a hardcoded `openclaw-lark` alias.

## What This Changes

Prevent Gateway plugin auto-enable from adding `feishu` to `plugins.allow` when `channels.feishu` is already served by an installed `openclaw-lark` channel plugin.

## Best Possible Solution

Teach channel auto-enable and validation to respect an installed, configured manifest owner for a configured channel before appending a built-in channel id to `plugins.allow`, while keeping `preferOver` fallback behavior intact.

## Reproduction Assessment

Yes, by source inspection: with `channels.feishu`, a restrictive `plugins.allow`, and an installed manifest record for `openclaw-lark` claiming `feishu` but no installed `feishu` plugin, current main can append `feishu` and then warn that it is missing. I did not run a live Gateway in this read-only review.

## Solution Assessment

No to a hardcoded `feishu` to `openclaw-lark` map. The narrow maintainable fix is a generic manifest-owner path that avoids stale allowlist injection while preserving `preferOver` and explicit user choices.
