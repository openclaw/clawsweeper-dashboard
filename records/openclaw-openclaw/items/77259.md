## Summary

Keep open: current main still contains the source-level path described in the report, where `channels.feishu` can auto-enable `feishu` into `plugins.allow` even when an installed `openclaw-lark` manifest already claims the Feishu channel, producing the stale-plugin warning.

## What This Changes

The issue requests that channel auto-enable avoid injecting an unavailable `feishu` plugin id when `channels.feishu` is satisfied by an installed `openclaw-lark` channel plugin.

## Best Possible Solution

Teach channel auto-enable to prefer an installed/trusted manifest owner for a configured channel before falling back to the built-in channel id, while preserving explicit disable/deny and `preferOver` semantics.

## Reproduction Assessment

Yes, by source inspection on current main: `channels.feishu` can select the built-in `feishu` id, append it to a restrictive `plugins.allow`, and then validation warns when that plugin id is absent. I did not execute a live Gateway in this read-only review.

## Solution Assessment

Yes for a generic manifest-owner fix; no for a hardcoded `feishu` to `openclaw-lark` map. The maintainable path is to reuse installed plugin channel ownership and keep the documented `preferOver` replacement contract intact.
