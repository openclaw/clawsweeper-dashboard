## Summary

Keep open: current main still has a source-level path where configured `channels.feishu` can auto-enable and allowlist the built-in `feishu` id even when an installed manifest such as `openclaw-lark` owns the channel, and validation then warns because `feishu` is absent. Existing doctor coverage already treats `openclaw-lark` as a valid `feishu` owner, so this should be fixed in the generic auto-enable path.

## What This Changes

The issue asks Gateway to stop re-adding stale `plugins.allow` entry `feishu` when `channels.feishu` is handled by the installed third-party `openclaw-lark` plugin.

## Best Possible Solution

Update channel auto-enable to choose an installed, allowlist-eligible manifest owner for a configured channel before falling back to a built-in channel id, while preserving `preferOver`, explicit enable/disable, denylist, and bundled fallback semantics.

## Reproduction Assessment

Yes. Source inspection shows `channels.feishu` can produce candidate `feishu`, materialization can append it to restrictive `plugins.allow`, and validation warns when the registry lacks plugin id `feishu` even though `openclaw-lark` claims the channel.

## Solution Assessment

Yes for a generic manifest-owner fix. A hardcoded `feishu` to `openclaw-lark` mapping would be brittle and would bypass the existing `preferOver` and explicit-selection contracts.
