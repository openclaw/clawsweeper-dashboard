## Summary

Keep open: the prior ClawSweeper source-level finding still matches current main, where configured-channel auto-enable can choose the built-in `feishu` id and then warn because that plugin id is absent while `openclaw-lark` owns `channels.feishu`.

## What This Changes

The issue requests that `channels.feishu` with third-party `openclaw-lark` stop re-injecting missing `feishu` into `plugins.allow` and emitting a stale-plugin warning.

## Best Possible Solution

Route channel auto-enable through effective trusted manifest owners for configured channels before falling back to built-in channel ids, while preserving `preferOver`, deny/disabled handling, and explicit user selection semantics.

## Reproduction Assessment

Yes. By source inspection, `channels.feishu` with a registry plugin claiming `feishu` as `openclaw-lark` but no installed `feishu` plugin can still select `feishu`, append it to `plugins.allow`, and then hit the existing stale-plugin warning path.

## Solution Assessment

No to a hardcoded map from `feishu` to `openclaw-lark`. The narrow maintainable fix is a generic manifest-owner selection path that reuses the same ownership concepts already present in doctor/channel presence logic.
