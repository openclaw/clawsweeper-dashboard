## Summary

Keep open. Current main still routes Microsoft/Edge speech synthesis through `node-edge-tts@1.2.10`, whose published transport sends the reported SSML WebSocket frame, and there is no OpenClaw-side protocol correction or regression proof that the generated audio omits XML markup.

## What This Changes

The issue asks OpenClaw to stop Microsoft/Edge TTS from reading SSML/XML markup aloud when Feishu voice-note replies are synthesized.

## Best Possible Solution

The Microsoft speech plugin should use a protocol-compatible Edge synthesis path that produces audio only for the requested text, while leaving Feishu as downstream voice-note delivery.

## Reproduction Assessment

Yes, this is source-reproducible with high confidence. The reporter supplied concrete Feishu plus Edge TTS steps and a transcript, and current main still delegates to the dependency path that builds the same SSML frame.

## Solution Assessment

No current implementation solves it. The narrow maintainable fix belongs in `extensions/microsoft` around the Edge speech transport or wrapper, not in Feishu or generic outbound media delivery.
