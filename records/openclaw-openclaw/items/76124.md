## Summary

Keep open. Current main and v2026.4.29 still route Microsoft/Edge TTS through `node-edge-tts@1.2.10`, whose transport constructs the same SSML WebSocket frame described in the report, and OpenClaw has no provider-side guard or test proving the spoken audio excludes XML markup.

## What This Changes

Edge/Microsoft TTS should speak only the reply text for Feishu voice replies instead of synthesizing SSML/XML markup as audible words.

## Best Possible Solution

Fix the Microsoft provider's Edge transport locally so it sends protocol-compatible speech requests and proves the resulting audio speaks only escaped user text, leaving Feishu delivery/transcoding unchanged.

## Reproduction Assessment

Yes. The reporter provided concrete Feishu plus Edge TTS steps and a Whisper transcript of spoken XML; source review confirms current main still uses the dependency path that builds and sends the SSML frame.

## Solution Assessment

No current implementation solves it. The narrow maintainable fix belongs in `extensions/microsoft` by owning or correcting the Edge WebSocket request path, not in Feishu or generic media delivery.
