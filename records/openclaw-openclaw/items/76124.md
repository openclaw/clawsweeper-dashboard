## Summary

Keep open. Current main still routes Microsoft/Edge speech through `node-edge-tts@1.2.10`, whose published `ttsPromise` builds the same SSML WebSocket frame described by the reporter, and OpenClaw does not yet own or test a provider-side transport fix that proves markup is not spoken.

## What This Changes

The issue asks OpenClaw to stop Microsoft Edge TTS from speaking SSML/XML request markup aloud and synthesize only the reply text for Feishu voice replies.

## Best Possible Solution

Own or correct the Microsoft Edge transport inside `extensions/microsoft` so the outgoing request is protocol-compatible and verified to speak only user text, while leaving Feishu delivery/transcoding unchanged.

## Reproduction Assessment

Yes. The reporter provided concrete Feishu plus Microsoft Edge TTS steps and a Whisper transcript of spoken XML; source and dependency review confirms current main still uses the same SSML WebSocket frame path.

## Solution Assessment

Yes, the best fix direction is provider-local. Feishu only receives/transcodes synthesized audio, so the maintainable repair belongs in the Microsoft Edge transport rather than channel delivery or generic outbound media code.
