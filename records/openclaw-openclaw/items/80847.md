## Summary

Keep open: this is a fresh, concrete regression report for an existing plugin-tool contract, and current main still has the source path that can throw the reported `plugin tool runtime missing` error after registry replacement/standalone reload. A later comment confirms the current-main path, and the already-merged cached-dispatch fix covers a different root cause, not the sub-agent archival/registry-lifetime failure described here.

## What This Changes

The issue requests that manifest-declared plugin tools remain available in sub-agent sessions for the lifetime of the gateway process, including after sub-agent archival and respawn.

## Best Possible Solution

Add a focused regression for cached descriptor plugin-tool execution after sub-agent archival or registry replacement, then make that path reuse or pin the durable gateway/channel registry instead of relying on a non-installed forced standalone load.

## Reproduction Assessment

Yes at source level: the current code contains the reported throw site and registry fallback path, and the issue discussion includes a current-main confirmation with exact line references. I did not live-run the custom lossless-claw plugin or force the archival timing cycle in this read-only review.

Unclear until a regression is written: reusing or pinning the gateway registry is likely the right direction, but the fix should prove cached descriptor execution after registry replacement while preserving the merged runtime-name dispatch behavior from https://github.com/openclaw/openclaw/pull/78716.
