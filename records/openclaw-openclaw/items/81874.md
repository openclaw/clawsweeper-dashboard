## Summary

Keep open: current main still has a source-reproducible Telegram topic leak where some forum-topic send paths retry after `message_thread_id` failure by dropping the topic id, which can place a visible message in General or the wrong topic despite a topic-qualified session.

## What This Changes

The issue reports Telegram forum-topic replies escaping the source topic even when inbound routing and session keys include the correct topic id.

## Best Possible Solution

Make every Telegram forum-topic reply/send path either include the resolved topic id or fail visibly; keep only the documented General-topic `threadId=1` omission.

## Reproduction Assessment

Yes from source inspection: a forum-topic draft preview with `message_thread_id` can retry after `message thread not found` by deleting the topic id and sending again. I did not run live Telegram, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: the narrow fix is to remove threadless retry for topic-qualified Telegram sends and add regression coverage for preview and outbound send paths, while preserving the General-topic special case.
