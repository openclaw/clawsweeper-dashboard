## Label Justifications

- `P1`: The original report described a real Telegram channel workflow where replies could land in the wrong forum topic for active users.
- `impact:message-loss`: Replies could be misdelivered to General or another topic instead of the originating Telegram topic.
- `impact:session-state`: The visible reply leaving the topic made the conversation appear to lose its topic-qualified session context.
- `impact:security`: Cross-topic delivery can expose assistant output in the wrong Telegram forum surface.

## Summary

Current main removes the Telegram forum-topic leak paths identified here, and the fix is present in the shipped v2026.5.18 release. The earlier focused PR was superseded by merged maintainer fixes that fail closed on missing topic threads and preserve topic-qualified inbound origins for follow-up delivery.

## What This Changes

The issue asks OpenClaw to keep Telegram forum-topic replies, previews, final sends, and fallbacks bound to the topic-qualified session target instead of jumping to General or another topic.

## Best Possible Solution

Keep the shipped fail-closed send fallback and topic-qualified origin routing, while letting broader async or cron cleanup continue only if maintainers still want separate follow-up work.

## Reproduction Assessment

No current-main failing path remains: the former source-level retry leak is now covered by regression tests that assert one topic-qualified call and a surfaced error. I did not run a live Telegram forum probe in this read-only review.

## Solution Assessment

Yes. The shipped solution is the narrow maintainable fix: fail closed when explicit topic sends fail, preserve real forum topic origin targets for follow-ups, and keep the documented General-topic exception.
