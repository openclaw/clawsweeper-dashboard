## Summary

Keep open: current main still has a source-reproducible Telegram forum-topic send leak, and https://github.com/openclaw/openclaw/pull/81885 is an open implementation candidate that explicitly closes this issue.

## What This Changes

The issue reports that Telegram forum-topic replies can appear in General or another topic even when inbound routing and the session key are topic-qualified.

## Best Possible Solution

Review and land or reject the linked Telegram fix PR; the desired end state is that topic-qualified sends keep `message_thread_id` or fail visibly while preserving the documented General-topic exception.

## Reproduction Assessment

Yes from source inspection: current main has Telegram forum-topic send paths that can retry after `message thread not found` by removing `message_thread_id`. I did not run live Telegram, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: removing threadless retry for topic-qualified Telegram sends is the narrow maintainable fix, with regression coverage for preview and outbound send paths and the General-topic `threadId=1` exception preserved.
