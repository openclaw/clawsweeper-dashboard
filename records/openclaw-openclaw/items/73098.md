## Summary

Keep open. Current main still rejects the safe cache-trace config mutation and still lets cache-trace content env values override explicit config false, while this PR directly addresses the linked bug with guarded code, regression tests, and a changelog entry; I found no blocking review finding.

## What This Changes

The PR allows agent-facing gateway config patch/apply calls to change cache-trace diagnostics only when raw message, prompt, and system capture stay disabled, and adds tests plus a changelog entry.

## Best Possible Solution

Land this PR after normal maintainer review, preserving the post-merge cache-trace guard, the config-disabled runtime precedence, and the newer current-main gateway allowlist entries.

## Reproduction Assessment

Yes. On current main, a patch that changes diagnostics.cacheTrace.enabled and the include* leaves reaches changed-path filtering with no matching allowlist entry, so the gateway tool rejects it before forwarding config.patch.

## Solution Assessment

Yes. The PR is narrowly scoped to exact cache-trace leaves, evaluates the resulting config instead of only the raw patch, keeps raw content capture protected, and backs the behavior with gateway-tool and cache-trace runtime tests.
