## Summary

Keep open for maintainer review. Current main still rejects the safe cache-trace patch described in the linked issue, while this PR is a narrow, tested fix with green PR checks and no blocking review findings.

## What This Changes

The PR allows agent-facing gateway config.patch/config.apply to change diagnostics.cacheTrace enabled/include fields only when resulting content capture stays disabled, hardens cache-trace env precedence, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land this PR after normal maintainer review, preserving the resulting-config guard and the runtime rule that config-disabled cache-trace content capture cannot be re-enabled by env overrides. A small docs clarification for env precedence can follow separately.

## Reproduction Assessment

Yes. Against current main, the safe patch from #72883 changes diagnostics.cacheTrace leaves that are absent from ALLOWED_GATEWAY_CONFIG_PATHS, so assertGatewayConfigMutationAllowed rejects it before forwarding config.patch.

## Solution Assessment

Yes. The PR is narrowly scoped to cacheTrace enabled/include leaves, evaluates the merged resulting config instead of only the raw patch, and closes the remote/env mismatch by making explicit config false authoritative at runtime.
