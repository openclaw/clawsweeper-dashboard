## Summary

Keep open. Current main still blocks the safe cache-trace config patch described in #72883, and this PR is a narrow, test-covered fix with no blocking correctness or security finding. One low-priority edge remains: the new runtime precedence makes config-disabled content capture authoritative, but the gateway guard still rejects safe patches when cache-trace content env vars are truthy.

## What This Changes

The PR special-cases agent-facing gateway config mutations to allow safe `diagnostics.cacheTrace` changes, updates cache-trace env/config precedence, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land the PR after maintainer review, with the cache-trace exception kept narrow and post-merge-state based. The maintainer should decide whether to remove the now-redundant env gate before merge or accept it as a conservative limitation and document the precedence explicitly.

## Reproduction Assessment

Yes. On current main, the linked safe `diagnostics.cacheTrace` patch changes paths absent from the agent gateway allowlist, so `assertGatewayConfigMutationAllowed` rejects it before forwarding `config.patch`.

## Solution Assessment

Yes, with a small follow-up. The PR is the narrowest maintainable shape because it checks the resulting config, keeps raw content capture protected, and updates runtime precedence; the remaining question is whether the env-truthy safe case should still be blocked.
