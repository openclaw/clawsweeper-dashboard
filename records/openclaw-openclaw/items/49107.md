## Summary

Keep this draft PR open. Current main still lacks non-interactive trusted-proxy onboarding, but the submitted patch is not merge-ready because it preserves incompatible token state, drops the Control UI origins flag through a Commander key mismatch, and documents a loopback trusted-proxy example without the required explicit loopback opt-in.

## What This Changes

The PR adds non-interactive `openclaw onboard` flags, option types, config application, tests, and docs for gateway trusted-proxy auth, trusted proxy IPs, `allowUsers`, and Control UI allowed origins.

## Best Possible Solution

Land a focused, rebased implementation that reuses or mirrors `buildGatewayAuthConfig`, wires the Commander option under the actual parsed key, and only documents loopback proxy setup when the explicit `allowLoopback` policy is exposed and tested.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main rejects `gatewayAuth: "trusted-proxy"` in non-interactive onboarding, and the PR regression follows directly from applying the diff to an existing token-auth config.

## Solution Assessment

No, not as submitted. Non-interactive onboard is the right layer, but this patch needs safer auth-mode replacement, correct Commander flag plumbing, and explicit loopback trust handling before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Drop stale credentials when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223-224`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-onboarding from token auth preserves `gateway.auth.token`. Current startup rejects `mode: "trusted-proxy"` when a shared token is also configured, leaving users with a config that fails until they manually remove the token.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's actual key:** `src/cli/program/register.onboard.ts:212`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, the documented flag is silently dropped before it reaches non-interactive config application.
  - confidence: 0.9
- **[P2] Make loopback proxy setup explicit:** `docs/cli/onboard.md:115`
  - body: The new example trusts `127.0.0.1`, but the runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups will still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Concerns:

- **[medium] Mixed trusted-proxy and shared-token state:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The PR preserves an existing `gateway.auth.token` when switching to trusted-proxy mode; current startup treats that mixed state as invalid, and auth-mode transitions should deliberately remove incompatible credentials.
