## Summary

## What This Changes

The PR adds non-interactive onboarding CLI/type/config/docs/test support for trusted-proxy gateway auth, allowUsers, and Control UI allowed origins.

## Best Possible Solution

Refresh this PR or replace it with a narrow onboarding change that reuses the existing gateway auth builder/parsing patterns, drops incompatible auth state, handles loopback trust intentionally, updates focused tests/docs, and includes redacted runtime proof.

## Reproduction Assessment

Yes, source-level. On current main, `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk` reaches `applyNonInteractiveGatewayConfig` and exits at the token/password-only validation before writing trusted-proxy config.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the patch needs the auth-state, Commander-key, loopback-documentation, proof, and security-review fixes first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. Current startup rejects trusted-proxy plus a shared token, so this can write a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Needs attention because the PR changes scripted trusted-proxy gateway setup and currently has concrete trust-boundary issues.

Concerns:

- **[medium] Mixed trusted-proxy and shared-token state:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The PR preserves an existing `gateway.auth.token` when switching to trusted-proxy mode, while current startup treats trusted-proxy plus a shared token as invalid; auth-mode transitions should deliberately drop incompatible credentials.
  - confidence: 0.92
- **[low] Loopback trust boundary documented but not configured:** `docs/cli/onboard.md:115`

  - confidence: 0.86
