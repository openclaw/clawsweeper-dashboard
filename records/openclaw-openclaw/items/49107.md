## Summary

Keep open. The PR targets a real current-main feature gap, but the submitted patch still has blocking defects in trusted-proxy state migration, Commander option wiring, loopback documentation, and lacks real after-fix behavior proof.

## What This Changes

The branch adds non-interactive onboard flags, config writing, tests, and docs for trusted-proxy gateway auth, trusted proxies, allowUsers, and Control UI origins, plus a generated plugin-sdk export addition already present on main.

## Best Possible Solution

Rebase and update this PR or a replacement to mirror the interactive trusted-proxy builder, drop incompatible token state, map Commander keys correctly, avoid loopback examples without an explicit allowLoopback surface, add focused regression coverage, and provide redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main rejects gatewayAuth: "trusted-proxy" in applyNonInteractiveGatewayConfig, and the submitted patch defects are traceable from the PR diff plus current gateway auth startup and loopback checks.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the patch must preserve current trust-boundary contracts before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-onboarding from token auth preserves `gateway.auth.token`. Current startup rejects `mode: "trusted-proxy"` when a shared token is also configured, leaving users with a config that fails until they manually remove the token.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, the documented flag is silently dropped before it reaches non-interactive config application.
  - confidence: 0.9
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups will still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88
