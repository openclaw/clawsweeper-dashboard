## Summary

Keep open. Current main still lacks non-interactive trusted-proxy onboarding, and this draft PR remains the active implementation path, but the patch has concrete correctness/security blockers and no after-fix real behavior proof.

## What This Changes

The PR adds non-interactive onboarding flags, config writing, docs, and unit tests for trusted-proxy gateway auth, trusted proxy IPs, allowed users, and Control UI allowed origins.

## Best Possible Solution

Update or replace this PR so non-interactive onboarding writes the same fresh trusted-proxy config shape as interactive onboarding, maps Commander keys correctly, avoids loopback examples without allowLoopback support, adds focused regression coverage, and includes redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main reaches applyNonInteractiveGatewayConfig for non-interactive onboarding and rejects gatewayAuth="trusted-proxy" at the token/password-only validation before writing config.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the patch needs the trust-boundary and option-mapping fixes before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. `assertGatewayAuthConfigured` rejects trusted-proxy when a shared token is present, so this can write a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander derives the option key by camel-casing the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff touches gateway trusted-proxy setup and has concrete trust-boundary issues that need correction before merge.

Concerns:

- **[medium] Mixed trusted-proxy and shared-token state:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The PR preserves an existing `gateway.auth.token` when switching to trusted-proxy mode, while current startup treats trusted-proxy plus a shared token as invalid; auth-mode transitions should deliberately drop incompatible credentials.
  - confidence: 0.92
