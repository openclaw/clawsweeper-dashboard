## Summary

Keep open: current main and the latest release still lack non-interactive trusted-proxy onboarding, but this draft PR is not merge-ready because the patch can write invalid auth state, drops the Control UI origins flag, documents a failing loopback example, and lacks real behavior proof.

## What This Changes

The PR branch adds non-interactive trusted-proxy onboarding CLI options, config writes for trusted proxies/users/Control UI origins, docs, tests, and a plugin-sdk lazy-runtime export that current main already contains.

## Best Possible Solution

Refresh or replace this PR with a narrow non-interactive onboarding implementation that reuses the existing trusted-proxy config builder patterns, deliberately handles stale auth state and loopback trust, and lands with focused tests, docs, and redacted runtime proof.

## Reproduction Assessment

Yes, source-level: current main routes `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk` to `applyNonInteractiveGatewayConfig`, which rejects anything except token/password before writing config.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the patch needs the auth-state, Commander-key, loopback-documentation, proof, and security-review issues fixed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. Current startup rejects trusted-proxy plus a shared token, so this can write a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88
