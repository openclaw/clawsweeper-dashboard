## Summary

Keep open. Current main still lacks the requested non-interactive trusted-proxy onboarding path, but this external draft PR is not mergeable as submitted because it can write invalid trusted-proxy config, drops a documented flag through Commander key mismatch, documents a loopback setup without the required opt-in, and lacks real behavior proof.

## What This Changes

The PR adds non-interactive onboarding flags and config writing for trusted-proxy gateway auth, trusted proxy IPs, allowed users, Control UI origins, docs, and unit coverage.

## Best Possible Solution

Update or replace this PR so non-interactive onboarding writes the same fresh trusted-proxy config shape as interactive onboarding, maps Commander option keys correctly, avoids loopback examples without allowLoopback support, adds focused regression coverage, and includes redacted runtime proof.

## Reproduction Assessment

Yes, source-level: current main reaches applyNonInteractiveGatewayConfig for non-interactive onboarding and exits when gatewayAuth is "trusted-proxy" because only token and password are accepted. I did not run a live onboard command because this review kept the checkout read-only.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the patch must preserve the trusted-proxy trust-boundary contracts before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-onboarding from token auth preserves `gateway.auth.token`. Current startup rejects mode `trusted-proxy` when a shared token is also configured, leaving users with a config that fails until they manually remove the token.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, the documented flag is silently dropped before it reaches non-interactive config application.
  - confidence: 0.9
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88
