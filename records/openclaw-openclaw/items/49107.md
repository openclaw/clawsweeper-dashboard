## Summary

Keep this PR open for maintainer follow-up. Current main still lacks the requested non-interactive trusted-proxy onboarding surface, and the PR is paired with the open feature request the author linked, but the draft patch is dirty and has blocking auth/config issues before it can land.

## What This Changes

This draft PR adds non-interactive onboard flags, types, config application, tests, and docs for trusted-proxy gateway auth, trusted proxy IPs, allowUsers, and Control UI allowed origins.

## Best Possible Solution

Keep or replace this implementation with a rebased, focused PR that adds non-interactive trusted-proxy onboarding in the current command structure, clears incompatible token state when switching modes, exposes an explicit loopback opt-in if same-host proxies are supported, handles Control UI origins with the correct CLI option key, and updates targeted tests/docs for the trusted-proxy security model.

## Reproduction Assessment

Yes. A focused static check against current main shows `gatewayAuth` is typed and validated as only `token | password`, and a unit-level reproduction would call `applyNonInteractiveGatewayConfig` with `gatewayAuth: "trusted-proxy"` and observe the current rejection path.

## Solution Assessment

No, not as submitted. Adding this support to non-interactive onboard is the right layer, but the current diff leaves stale auth state, misses Commander option-key behavior for the Control UI flag, and documents a loopback proxy example without writing the required `allowLoopback` opt-in.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Drop stale token when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223-224`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the mode, so a normal re-onboard from token auth preserves `gateway.auth.token`. Current startup rejects `mode: "trusted-proxy"` when a token is still configured, so this writes a config that will fail to start until the token is manually removed.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander’s actual option key:** `src/cli/program/register.onboard.ts:212`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, users can pass the documented flag and it will be silently dropped before reaching `applyNonInteractiveGatewayConfig`. Rename the flag to `--gateway-control-ui-allowed-origins` or read the actual key.
  - confidence: 0.9
- **[P2] Make loopback proxy setup explicit:** `docs/cli/onboard.md:115`
  - body: The new example includes `127.0.0.1` as a trusted proxy, but the runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups will still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88
