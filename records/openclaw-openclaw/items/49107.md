## Summary

Keep open. Current main still lacks non-interactive trusted-proxy onboarding, so the PR targets a real gap, but the submitted patch is draft/conflicting, lacks contributor real behavior proof, and has blocking auth/security correctness issues.

## What This Changes

The PR adds non-interactive onboard flags, config writes, docs, and unit tests for trusted-proxy gateway auth, allowed users, and Control UI allowed origins.

## Best Possible Solution

Refresh or replace this PR with a narrow implementation that writes the same trusted-proxy config shape as interactive onboarding, preserves runtime guardrails, and lands with focused tests, docs, and redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk` reaches `applyNonInteractiveGatewayConfig` and exits at the token/password-only validation on current main.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the patch needs the auth-state, Commander-key, loopback-documentation, proof, and security issues fixed first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. Current startup rejects trusted-proxy plus a shared token, leaving users with a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch changes scripted trusted-proxy setup and currently allows invalid mixed auth state while documenting a loopback proxy shape without the runtime opt-in.

Concerns:

- **[medium] Mixed trusted-proxy and shared-token state:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The PR preserves an existing `gateway.auth.token` when switching to trusted-proxy mode, while current startup treats trusted-proxy plus a shared token as invalid; auth-mode transitions should deliberately drop incompatible credentials.
  - confidence: 0.92
