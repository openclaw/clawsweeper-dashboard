## Summary

Keep open. Current main and the latest release still lack non-interactive trusted-proxy onboarding, but this draft PR is not merge-ready because it is conflicting, lacks real behavior proof, and has blocking trusted-proxy trust-boundary defects.

## What This Changes

This draft PR adds non-interactive onboard flags, config application, tests, docs for trusted-proxy gateway auth and Control UI origins, plus a generated plugin-sdk export addition already present on main.

## Best Possible Solution

Rebase and update the PR or a replacement to mirror the interactive gateway auth builder, wire Commander option keys correctly, avoid unsafe loopback examples, add focused regression coverage, and provide redacted runtime proof before security review.

## Reproduction Assessment

Yes, source-level. Current main rejects `gatewayAuth: "trusted-proxy"` in `applyNonInteractiveGatewayConfig`, and the submitted PR defects are traceable from the diff plus current gateway auth startup checks.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right surface, but this patch must replace incompatible auth fields, read the parsed Commander key, and respect the explicit loopback trust boundary before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-onboarding from token auth preserves `gateway.auth.token`. Current startup rejects `mode: "trusted-proxy"` when a shared token is also configured, leaving users with a config that fails until they manually remove the token.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, the documented flag is silently dropped before it reaches non-interactive config application.
  - confidence: 0.9
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but the runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups will still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88
