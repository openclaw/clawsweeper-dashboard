## Summary

Keep this PR open. Current main still lacks non-interactive trusted-proxy onboarding, and the PR is aimed at the right CLI/config surface, but the submitted patch is not merge-ready because it leaves incompatible auth state behind, drops one documented flag due to Commander option-key behavior, and documents a loopback proxy setup without an explicit loopback trust opt-in.

## What This Changes

This PR adds non-interactive `openclaw onboard` flags, option types, config application, tests, and docs for gateway trusted-proxy auth, trusted proxy IPs, `allowUsers`, and Control UI allowed origins.

## Best Possible Solution

Land a focused, rebased implementation that adds non-interactive parity for the existing trusted-proxy runtime contract, reuses or mirrors `buildGatewayAuthConfig` so incompatible token/password state is cleared, handles the Control UI origin flag with the actual CLI option key, and exposes/docs any same-host loopback proxy support as an explicit opt-in with targeted tests.

## Reproduction Assessment

Yes. Current main can be checked statically by calling `applyNonInteractiveGatewayConfig` with `gatewayAuth: "trusted-proxy"` and observing the rejection path; the PR regression can be reproduced by applying trusted-proxy options to a config that already has `gateway.auth.token` and then running the existing startup auth validation.

## Solution Assessment

No, not as submitted. Non-interactive onboard is the right layer for this feature, but the patch needs a safer auth-mode transition, correct Commander flag plumbing, and explicit loopback trust handling before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Drop stale credentials when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223-224`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-onboarding from token auth preserves `gateway.auth.token`. Current startup rejects `mode: "trusted-proxy"` when a shared token is also configured, leaving users with a config that fails until they manually remove the token.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's actual key:** `src/cli/program/register.onboard.ts:212`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, the documented flag is silently dropped before it reaches non-interactive config application.
  - confidence: 0.9
- **[P2] Make loopback proxy setup explicit:** `docs/cli/onboard.md:115`
  - body: The new example trusts `127.0.0.1`, but the runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups will still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88
