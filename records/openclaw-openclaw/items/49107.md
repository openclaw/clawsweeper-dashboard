## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Drop stale tokens when selecting trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223-224`
  - body: This spreads the existing `gateway.auth` object into the new trusted-proxy config, so a normal re-onboard from token auth can keep `gateway.auth.token` while setting `mode: "trusted-proxy"`. Current gateway startup rejects that mixed state, so the scripted setup can write a config that fails before the gateway starts; build the auth object through the existing guardrail helper or explicitly omit token fields.
  - confidence: 0.91
- **[P2] Merge Control UI origins instead of replacing them:** `src/commands/onboard-non-interactive/local/gateway-config.ts:264`
  - body: The new flag is documented as origins to add, but this assignment replaces `gateway.controlUi.allowedOrigins` outright. Re-running onboarding to add a proxy origin can silently remove existing localhost, LAN, or tailnet origins and break previously working Control UI access; merge and de-duplicate with the existing list instead.
  - confidence: 0.84
- **[P2] Do not show loopback proxy setup without allowLoopback:** `docs/cli/onboard.md:115`
  - body: The example writes `127.0.0.1` into `gateway.trustedProxies`, but current trusted-proxy auth rejects loopback-source proxy identity unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Either add an explicit non-interactive allowLoopback flag and warning, or remove loopback from the example so users do not copy a config that still rejects their proxy.
  - confidence: 0.87
