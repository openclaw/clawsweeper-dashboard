## Summary

Keep open. The PR targets a real non-interactive onboarding gap, but it is not merge-ready because the patch can write invalid mixed token/trusted-proxy auth, documents a loopback proxy shape that the new flags cannot enable, replaces existing Control UI origins, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds non-interactive onboarding flags and docs for trusted-proxy gateway auth, trusted proxy allow-users, and Control UI allowed origins.

## Best Possible Solution

Rebase and make the non-interactive path construct trusted-proxy config with the same runtime guardrails as current gateway auth: drop incompatible token state, preserve only valid auth fields, merge existing Control UI origins, handle allowLoopback explicitly or document it out of scope, and prove a real scripted onboarding run.

## Reproduction Assessment

Yes. Source inspection shows current main rejects `--gateway-auth trusted-proxy` in non-interactive onboarding, and the PR's re-onboard failure is source-reproducible because it spreads an existing `gateway.auth.token` into trusted-proxy mode.

## Solution Assessment

No. The direction matches the existing runtime and interactive onboarding surface, but this patch is not the safest implementation until it handles auth-mode migration, loopback proxy semantics, existing origin preservation, and real behavior proof.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0.91

Full review comments:

- **[P1] Drop stale shared auth when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223-224`
  - body: This spreads the existing `gateway.auth` object into the new trusted-proxy config. A normal re-onboard starts from token mode, so the written config can contain both `mode: "trusted-proxy"` and `auth.token`; current gateway startup rejects that mixed state, making the new non-interactive switch fail for existing installs. Build the trusted-proxy auth object from a clean base and preserve only fields that remain valid.
  - confidence: 0.93
- **[P2] Handle loopback trusted proxies explicitly:** `docs/cli/onboard.md:115`
  - body: The new example includes `127.0.0.1`, but current auth rejects loopback-source trusted-proxy headers unless `gateway.auth.trustedProxy.allowLoopback` is true. Since these flags never write that field, the documented same-host setup fails closed with `trusted_proxy_loopback_source`; add an explicit non-interactive allow-loopback flag with tests/docs, or remove loopback from the example and state it is not covered here.
  - confidence: 0.9
- **[P2] Merge Control UI origins instead of replacing them:** `src/commands/onboard-non-interactive/local/gateway-config.ts:264`
  - body: The docs say this flag adds origins, but the code replaces `gateway.controlUi.allowedOrigins` with only the flag values. Re-running onboarding to add one public origin can silently drop existing origins and break previously working Control UI clients; merge and dedupe with the existing list or document replacement semantics.
  - confidence: 0.86
