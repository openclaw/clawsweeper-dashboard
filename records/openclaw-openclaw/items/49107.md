## Summary

Keep open. Current main still lacks the requested non-interactive trusted-proxy onboarding surface, but this draft PR has blocking correctness issues on the gateway auth trust boundary and needs an update/rebase before it is mergeable.

## What This Changes

The PR adds non-interactive onboard flags, option types, config application, tests, and docs for trusted-proxy gateway auth, trusted proxy IPs, allowUsers, and Control UI allowed origins.

## Best Possible Solution

Land a focused, rebased implementation that mirrors the existing gateway auth builder semantics, wires Commander option keys correctly, and treats loopback trusted-proxy support as an explicit opt-in rather than implied docs behavior.

## Reproduction Assessment

Yes. Source inspection shows current main rejects `--gateway-auth trusted-proxy` in non-interactive onboarding, and the PR regressions follow directly from the submitted diff and current gateway auth startup checks.

## Solution Assessment

No, not as submitted. Non-interactive onboard is the right layer, but the patch needs safer auth-mode replacement, correct Commander flag plumbing, and explicit loopback trust handling before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-onboarding from token auth preserves `gateway.auth.token`. Current startup rejects `mode: "trusted-proxy"` when a shared token is also configured, leaving users with a config that fails until they manually remove the token.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212`
  - body: Commander camel-cases `--gateway-controlui-allowed-origins` to `gatewayControluiAllowedOrigins`, but the action reads `opts.gatewayControlUiAllowedOrigins`. As written, the documented flag is silently dropped before it reaches non-interactive config application.
  - confidence: 0.9
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The new example trusts `127.0.0.1`, but the runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback` is explicitly true. Since this PR does not expose or write that setting, copied same-host proxy setups will still fail with `trusted_proxy_loopback_source`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency or supply-chain change was found, but the diff changes gateway trusted-proxy auth setup and introduces concrete trust-boundary/configuration concerns.

Concerns:

- **[medium] Mixed trusted-proxy and shared-token state:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The PR preserves an existing `gateway.auth.token` when switching to trusted-proxy mode; current startup treats that mixed state as invalid, and auth-mode transitions should deliberately remove incompatible credentials.
  - confidence: 0.92
- **[low] Loopback trust boundary is documented but not configured:** `docs/cli/onboard.md:115`
  - body: The docs example suggests trusting a loopback proxy, but the patch does not expose or set the runtime's `allowLoopback` opt-in, so users may copy a setup that fails or misunderstand the same-host proxy trust boundary.
  - confidence: 0.86
