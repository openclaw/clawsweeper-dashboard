## Summary

Keep open. Current main still lacks non-interactive trusted-proxy onboarding, and this draft PR is the active same-author implementation path, but it is not mergeable as submitted because it can preserve stale token state, drops the Control UI origins flag, documents a loopback setup it does not configure, and lacks real behavior proof.

## What This Changes

The PR adds non-interactive onboard flags, config writing, docs, and unit tests for trusted-proxy gateway auth, trusted proxy IPs, allowed users, and Control UI allowed origins.

## Best Possible Solution

Update this PR or a replacement to reuse the existing gateway auth builder/parsing patterns, expose a clear trusted-proxy flag set with intentional loopback handling, add regression tests for auth-mode transitions and Commander key mapping, and include redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main's non-interactive flow calls applyNonInteractiveGatewayConfig, which rejects gatewayAuth="trusted-proxy" at the token/password-only validation before any trusted-proxy config can be written.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for an existing gateway auth mode, but this patch needs the auth-state, Commander-key, loopback-documentation, and proof fixes before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. Current startup rejects trusted-proxy plus a shared token, so this can write a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88
