## Summary

Keep open: current main still lacks the requested non-interactive trusted-proxy onboarding path, but this draft PR is not merge-ready because it preserves invalid auth state, drops one documented flag, documents a loopback proxy setup without the required opt-in, and lacks real behavior proof.

## What This Changes

The PR adds non-interactive onboard flags and config writes for trusted-proxy gateway auth, allowed users, Control UI allowed origins, plus docs and unit tests.

## Best Possible Solution

Refresh or replace this PR with a narrow non-interactive onboarding implementation that reuses existing trusted-proxy config contracts, deliberately clears incompatible auth fields, exposes loopback trust only as an explicit opt-in if desired, and lands with focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection shows `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk` reaches `applyNonInteractiveGatewayConfig`, which still rejects anything except token/password on current main.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but the PR needs auth-state cleanup, the Commander key fix, loopback documentation/config alignment, and real behavior proof first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. Current startup rejects trusted-proxy plus a shared token, leaving users with a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88
