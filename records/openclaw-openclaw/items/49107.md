## Summary

Keep open. Current main and the latest release still lack the requested non-interactive trusted-proxy onboarding path, and this active draft PR is the paired implementation path, but it still has blocking correctness/security findings and no real behavior proof.

## What This Changes

The branch adds CLI options, type plumbing, config writing, docs, and unit tests for non-interactive trusted-proxy gateway auth, plus a generated package export update for `plugin-sdk/lazy-runtime`.

## Best Possible Solution

Refresh this PR or replace it with a narrow onboarding change that reuses the existing gateway auth builder/parsing patterns, intentionally handles loopback trust, drops incompatible auth state, updates focused tests/docs, and includes redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main reaches `applyNonInteractiveGatewayConfig` for `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk` and exits at the token/password-only validation before writing config.

## Solution Assessment

No, not as submitted. Non-interactive onboarding is the right core surface for this existing gateway auth mode, but this patch needs the auth-state, Commander-key, loopback-documentation, proof, and security-review fixes first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Drop stale tokens when switching to trusted-proxy:** `src/commands/onboard-non-interactive/local/gateway-config.ts:223`
  - body: The trusted-proxy branch spreads the existing `gateway.auth` object before setting the new mode, so re-running onboarding from token auth keeps `gateway.auth.token`. Current startup rejects trusted-proxy plus a shared token, so this can write a config that fails until edited by hand.
  - confidence: 0.93
- **[P2] Read the Control UI flag from Commander's parsed key:** `src/cli/program/register.onboard.ts:212-215`
  - body: Commander camel-cases the literal long flag. `--gateway-controlui-allowed-origins` parses as `gatewayControluiAllowedOrigins`, but the action reads `gatewayControlUiAllowedOrigins`, so the documented flag is silently dropped before config application.
  - confidence: 0.91
- **[P2] Do not document loopback proxy setup without an opt-in:** `docs/cli/onboard.md:115`
  - body: The example trusts `127.0.0.1`, but trusted-proxy auth rejects loopback-source requests unless `gateway.auth.trustedProxy.allowLoopback` is true. This PR neither exposes nor writes that opt-in, so the copied same-host proxy example still fails with `trusted_proxy_loopback_source`.
  - confidence: 0.88
