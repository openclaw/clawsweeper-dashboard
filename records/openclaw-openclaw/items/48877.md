## Summary

Keep open: the feature is not implemented on current main, and this branch still needs real Telegram proof, a rebase onto the current Telegram bot-core/config surfaces, and tighter command-scoped callback authorization before it is mergeable.

## What This Changes

The PR adds Telegram `customCommands` `menus`/`routes` config, menu-backed slash handlers, and inline keyboard callback routing without an AI round trip.

## Best Possible Solution

Land a refreshed Telegram-owned implementation that uses `bot-core.ts` and local `command-config.ts`, validates callback payloads, preserves command-specific authorization, documents the public config, and includes live Telegram proof.

## Reproduction Assessment

Yes for source-level PR review: current main lacks `menus`/`routes`, docs describe custom commands as menu entries only, and the PR head shows the callback fallback path. I did not run live Telegram because this read-only review found no posted proof artifact to inspect.

## Solution Assessment

No. The direction is reasonable, but this branch is not the best mergeable solution until it is rebased onto current Telegram wiring, keeps callback actions within command-specific authorization, validates payloads, documents the config, and posts real Telegram proof.

## Review Findings

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on callback fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a custom menu button is not in `routes`, this forwards its `callback_data` to `processMessage` after only generic callback authorization. In shared groups, a member who could not invoke the restricted slash command can still click an existing menu action, so the fallback needs command-specific auth or a non-AI behavior.
  - confidence: 0.9
- **[P1] Move menu wiring into bot-core:** `extensions/telegram/src/bot.ts:496-535`
  - body: This threads `validatedCustomCommandIndices` through the old full `bot.ts` body, but current main's `bot.ts` is now a wrapper and `createTelegramBotCore` owns command and handler registration. Rebase this wiring into `bot-core.ts` or the handler will not receive the validated set.
  - confidence: 0.89
- **[P1] Use the current Telegram command-config surface:** `extensions/telegram/src/bot-native-commands.ts:9-13`
  - body: The new import expects `resolveValidatedCustomCommandIndices` from `openclaw/plugin-sdk/config-runtime`, but current main no longer exposes Telegram command helpers from that broad surface and the extension uses local `command-config.ts`. Rebase the helper into the current Telegram surface before merging.
  - confidence: 0.87
- **[P2] Validate callback payloads before sending menus:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts any string, while the new send/edit paths pass menu buttons directly to Telegram. Blank or over-64-byte payloads should fail config parsing instead of turning a configured slash command into a Telegram API failure at runtime.
  - confidence: 0.86
