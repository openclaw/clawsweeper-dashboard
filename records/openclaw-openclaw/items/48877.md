## Summary

Keep this PR open. Current main still lacks declarative Telegram `customCommands` menu routing, so the feature is not obsolete, but the proposed patch is not ready to merge because the latest diff imports a helper from a SDK surface that does not export it and accepts invalid Telegram callback payloads that can fail at runtime.

## What This Changes

The PR adds Telegram `customCommands` `menus`/`routes` config fields, registers menu-backed slash-command handlers, routes matching callback queries to menu edits, and adds a validated custom-command index helper.

## Best Possible Solution

Land a refreshed Telegram plugin implementation that adds typed `menus` and `routes` on the current config surfaces, exports any new validation helper through the narrow Telegram command-config SDK surface or keeps it extension-local, validates menu button text and callback payloads against Telegram constraints, routes only authenticated originating menu callbacks, updates generated config metadata and public Telegram docs, and adds focused schema/native-command/callback tests.

## Reproduction Assessment

Yes. Current main can be checked by source and config inspection: `menus`/`routes` are absent from the type, rejected by the strict schema, and callback payloads still fall through as synthetic messages. The PR's import issue is reproducible by comparing the proposed `config-runtime` import with the current exports.

## Solution Assessment

No, not as currently proposed. The feature belongs in the Telegram plugin/config surface, but the patch should be rebased onto current `bot-core`/SDK structure and should validate Telegram callback payloads before merge.

  - body: `resolveValidatedCustomCommandIndices` is imported from `openclaw/plugin-sdk/config-runtime`, but the patch does not update that SDK subpath to export it.   - confidence: 0.9

  - confidence: 0.86
