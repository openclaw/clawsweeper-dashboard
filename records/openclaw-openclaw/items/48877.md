Full review comments:

- **[P1] Recheck custom-command auth on callback fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a custom menu button is not in `routes`, this forwards its `callback_data` as a synthetic message after only callback-scope authorization. In restricted groups, another member can click an existing menu action and drive the command even though they could not invoke the slash command, so fallback actions need command-specific auth or a non-AI behavior.
  - confidence: 0.9

  - body: Current main's `createTelegramBot` now delegates to `createTelegramBotCore`, but this patch threads `validatedCustomCommandIndices` through the older full `bot.ts` body. Rebase the registration and handler plumbing into `bot-core` so the callback handler receives the validated set after merge.
  - confidence: 0.89
- **[P1] Use the current Telegram command-config surface:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: This imports `resolveValidatedCustomCommandIndices` from `openclaw/plugin-sdk/config-runtime` and adds the helper under `src/config`, but current main moved Telegram command config to extension-local `command-config.ts` and deprecated that broad SDK surface for bundled extension code. Rebase the helper into the local Telegram config surface before merging.
  - confidence: 0.87
- **[P2] Validate custom-menu callback payloads:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts blank or over-64-byte strings, while the new send/edit paths pass menu buttons directly to Telegram. Existing Telegram interactive button handling sanitizes overlong callback data, so config should reject invalid menu payloads before a slash command turns into a Telegram API failure.
  - confidence: 0.86
