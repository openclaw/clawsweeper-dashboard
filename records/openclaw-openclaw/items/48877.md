## Summary

This PR should stay open because current main still does not expose Telegram `customCommands` menu pages/routes, but the branch is not merge-ready after the current-main bot-core refactor and still has callback authorization, config ownership, payload-validation, and real-proof blockers.

## What This Changes

The PR adds optional Telegram `customCommands` `menus`/`routes` config, registers menu-backed slash handlers, and routes inline callback data to menu edits or AI fallback.

## Best Possible Solution

Rework this as Telegram-owned config/API support inside bot-core, with command-specific callback authorization, callback payload validation, focused tests/docs, and redacted live Telegram proof.

## Reproduction Assessment

Yes for source-level PR review: current main lacks `menus`/`routes`, and the PR diff shows the new routing and fallback paths. No live Telegram reproduction was established because there is no proof artifact to inspect.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the best mergeable shape until it is rebased into bot-core, keeps Telegram config helpers extension-local, secures callback fallback auth, validates payloads, and provides real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on menu fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1531-1532`
  - body: When a menu button is not listed in `routes`, this new router falls through to the generic synthetic callback message path after only callback-scope authorization. In shared groups, a user who could not invoke the original slash command can click an action button from an authorized user's menu and drive the command's AI behavior, so fallback actions need command-specific auth or should not fall through.
  - confidence: 0.9
- **[P1] Move menu registration into bot-core:** `extensions/telegram/src/bot.ts:493-531`
  - body: Current main's `bot.ts` is only a wrapper around `createTelegramBotCore`, but this branch still threads `validatedCustomCommandIndices` through the old inlined `bot.ts` body. Rebase the registration and handler plumbing into `bot-core` so the active runtime path actually receives the validated set.
  - confidence: 0.88
- **[P2] Keep command validation in Telegram's local helper:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: The PR imports and extends the deprecated broad config-runtime/core helper surface, while current Telegram production code owns custom command parsing in `extensions/telegram/src/command-config.ts`. Keep the index-preserving helper extension-local instead of widening stale SDK/core config API surface.
  - confidence: 0.86
- **[P2] Validate callback payloads before sending menus:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts blank and overlong strings, and the new send/edit paths pass menu buttons directly to Telegram. Reject invalid payloads during config parsing so a bad menu config fails early instead of becoming a Telegram API failure when the command is clicked.
  - confidence: 0.86
