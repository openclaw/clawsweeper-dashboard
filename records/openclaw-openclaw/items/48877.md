## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on callback fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a custom menu button is not in `routes`, this forwards its `callback_data` to `processMessage` after only generic callback authorization. In a restricted group, another member can click an existing menu action and drive the command even though they could not invoke the slash command, so fallback actions need command-specific auth or should not fall through to AI.
  - confidence: 0.9
- **[P1] Move menu wiring into bot-core:** `extensions/telegram/src/bot.ts:496`
  - body: Current main's `createTelegramBot` is a wrapper around `createTelegramBotCore`, but this branch threads `validatedCustomCommandIndices` through the older `bot.ts` body. Rebase the registration and handler plumbing into `bot-core` so the callback handler receives the validated set after merge.
  - confidence: 0.88
- **[P1] Use the current Telegram command-config surface:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: This imports a new helper from deprecated `openclaw/plugin-sdk/config-runtime` and adds it under the old core config helper path. Current Telegram production code uses extension-local `command-config.ts`, so the helper should live there instead of expanding the deprecated SDK surface.
  - confidence: 0.86
- **[P2] Validate custom-menu callback payloads:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts blank or over-64-byte strings, and the new send/edit paths pass menu buttons directly to Telegram instead of the existing sanitizer. Reject invalid payloads during config parsing so a bad menu config fails early instead of at the Telegram API call.
  - confidence: 0.86
