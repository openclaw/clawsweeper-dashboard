## Summary

Keep this PR open for maintainer/contributor follow-up: current main still does not implement Telegram `customCommands` menus, but the branch is conflicting, lacks inspectable real Telegram proof, and still has merge-blocking callback authorization and current-surface issues.

## What This Changes

The PR adds `menus` and `routes` fields to Telegram `customCommands`, registers menu-backed slash command handlers, and routes inline keyboard callbacks for menu navigation.

## Best Possible Solution

Land a refreshed Telegram implementation in `bot-core` and the extension-local command-config surface, with command-specific callback authorization, payload validation, docs/tests, and real Telegram proof.

## Reproduction Assessment

Yes for source-level PR review: current main lacks `menus`/`routes`, and the PR head's callback fallback plus stale wiring/import surfaces are visible from the diff. No live Telegram behavior was established because the PR has no proof artifact to inspect.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the best mergeable solution until it is rebased onto current Telegram wiring, keeps fallback actions within command-specific authorization, validates callback payloads, documents the config, and posts real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on callback fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a custom menu button is not in `routes`, this forwards its `callback_data` to `processMessage` after only the generic callback authorization path. In a restricted group, another member can click an existing menu action and drive a command they could not invoke via the slash command, so the fallback needs command-specific auth or should not fall through to AI.
  - confidence: 0.9
- **[P1] Move menu wiring into bot-core:** `extensions/telegram/src/bot.ts:496`
  - body: Current main's `createTelegramBot` is only a wrapper around `createTelegramBotCore`, but this branch threads `validatedCustomCommandIndices` through the older `bot.ts` body. Rebase the registration and handler plumbing into `bot-core` so the callback handler actually receives the validated set after merge.
  - confidence: 0.88
- **[P1] Use the current Telegram command-config surface:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: This imports a new helper from deprecated `openclaw/plugin-sdk/config-runtime` and adds it under the old core config helper path. Current Telegram production code uses extension-local `command-config.ts`, so the helper should live there rather than expanding the deprecated SDK surface.
  - confidence: 0.86
- **[P2] Validate custom-menu callback payloads:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts blank and overlong strings, and the new send/edit paths pass menu buttons directly to Telegram instead of the existing sanitizer. Reject invalid payloads during config parsing so a bad menu config fails early instead of at the Telegram API call.
  - confidence: 0.86
