## What This Changes

The PR adds Telegram `customCommands` `menus`/`routes` config, registers menu-backed slash commands, and routes custom menu callbacks without an AI round trip.

## Best Possible Solution

Land a refreshed Telegram-owned implementation on current main that uses the current `bot-core.ts` and command-config surfaces, validates callback payloads, preserves command-specific authorization, documents the public config, and includes real Telegram proof.

## Reproduction Assessment

Yes for source-level review: current main lacks `menus`/`routes`, docs describe custom commands as menu entries only, and the PR head's changed source shows the remaining callback, wiring, and schema defects. I did not run live Telegram because this review is read-only.

## Solution Assessment

No. The feature direction is reasonable, but this branch needs a rebase onto current Telegram wiring, current config API placement, stricter callback authorization, and callback payload validation before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on callback fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a custom menu button is not in `routes`, this forwards its `callback_data` to `processMessage` after only generic callback authorization. In shared groups, a member who could not invoke the restricted slash command can still drive an existing menu action posted by someone else, so the fallback needs command-specific auth or a safer non-AI behavior.
  - confidence: 0.9
- **[P1] Rebase menu wiring through bot-core:** `extensions/telegram/src/bot.ts:496-535`
  - body: This wires `validatedCustomCommandIndices` through the old full `bot.ts`, but current main's `bot.ts` is a wrapper and `createTelegramBotCore` owns command and handler registration. After rebase, this needs to move into `bot-core.ts` or the handler will not receive the validated set.
  - confidence: 0.89
- **[P1] Move the validated-index helper to the current command surface:** `extensions/telegram/src/bot-native-commands.ts:12`
  - body: The new import comes from `openclaw/plugin-sdk/config-runtime`, but current main no longer exposes the old `src/config/telegram-custom-commands.ts` surface there. Port the helper to the current Telegram command-config surface or keep it plugin-local so the rebased branch builds.
  - confidence: 0.87
- **[P2] Validate custom-menu callback payloads before sending:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts any string, but the new send/edit paths pass menu buttons straight to Telegram. Blank or over-64-byte payloads should fail config parsing instead of becoming missing buttons or Telegram API failures at runtime.
  - confidence: 0.86
