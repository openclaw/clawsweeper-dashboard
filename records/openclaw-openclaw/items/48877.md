## Summary

Keep open. Current main still lacks Telegram custom-command menus/routes, so the feature is not obsolete, but this PR is not mergeable yet because the branch is stale against the current Telegram bot-core split, has a callback authorization regression, accepts invalid callback payloads until runtime, and lacks inspectable real Telegram proof.

## What This Changes

The PR adds Telegram `customCommands` `menus`/`routes` config, menu-backed slash-command handlers, callback menu navigation, and a validated custom-command index helper.

## Best Possible Solution

Land a refreshed Telegram-owned implementation on current main that adds typed and validated menu config, wires through `bot-core.ts`, preserves command authorization for menu actions, documents/changelogs the public config, and includes real Telegram proof.

## Reproduction Assessment

Yes for source-level review: current main lacks `menus`/`routes` in the type, schema, callback router, and docs, and the PR branch's remaining defects are visible in the PR head source. I did not run live Telegram because this review is read-only.

## Solution Assessment

No. The feature direction is reasonable, but the current patch is not the best mergeable form until it is rebased onto the current Telegram wiring, fixes callback authorization and payload validation, and provides real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on callback fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a menu button is not in `routes`, this forwards `callback_data` to `processMessage` after only generic callback authorization. In a group, another authorized room member can drive a menu opened by someone who passed the stricter `/command` auth, so menu action callbacks need command-specific auth or should not fall through this way.
  - confidence: 0.9
- **[P1] Move bot wiring onto the current core entry point:** `extensions/telegram/src/bot.ts:496-535`
  - body: The branch wires validated custom-command indices through the old full `bot.ts`, but current main's `bot.ts` is only a wrapper and `createTelegramBotCore` owns `registerTelegramNativeCommands` and `registerTelegramHandlers`. Rebase this through `bot-core.ts` or the handler will not receive the validated set on current main.
  - confidence: 0.9
- **[P1] Port the validated-index helper to the current command surface:** `extensions/telegram/src/bot-native-commands.ts:12`
  - body: This imports `resolveValidatedCustomCommandIndices` from `openclaw/plugin-sdk/config-runtime`, but current main no longer wildcard-exports the old Telegram config module from that SDK surface. The helper needs to live on the current Telegram command-config surface, or remain plugin-local, before this can build after rebase.
  - confidence: 0.88
- **[P2] Validate custom-menu callback payloads before sending:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts any string, but the new send/edit paths pass menu buttons directly to Telegram. Existing Telegram helpers enforce the 64-byte callback-data limit, so blank or overlong menu payloads should fail config validation instead of becoming runtime API failures.
  - confidence: 0.86
