Full review comments:

- **[P1] Recheck custom-command auth on menu fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a menu button is not in `routes`, this branch forwards its `callback_data` to `processMessage` after only generic callback authorization. In shared groups that path does not reapply `commands.allowFrom`, so another member can click an existing restricted command menu and trigger the fallback action.
  - confidence: 0.9
- **[P1] Move menu registration into bot-core:** `extensions/telegram/src/bot.ts:493`
  - body: Current main's `bot.ts` is only a wrapper around `createTelegramBotCore`, while bot-core owns native command and handler registration. Threading `validatedCustomCommandIndices` through the old `bot.ts` body will not wire the active runtime after rebase; put this plumbing in `bot-core`.
  - confidence: 0.88
- **[P2] Use the extension-local command config helper:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: Telegram production code now owns custom command parsing in `extensions/telegram/src/command-config.ts`, and `openclaw/plugin-sdk/config-runtime` is deprecated for bundled extension production imports. Add the validated-index helper to the Telegram-local command config/API path instead of widening the stale SDK/core config surface.
  - confidence: 0.86
- **[P2] Validate callback payloads before sending menus:** `src/config/zod-schema.providers-core.ts:130`
  - body: `callback_data` accepts blank and overlong strings, and the new send/edit paths pass menu buttons directly to Telegram. Reject invalid payloads during config parsing so a bad menu config fails early instead of at the Telegram API call.
  - confidence: 0.84
