## Summary

Keep open: current main still does not implement Telegram `customCommands` menus/routes, but this PR is not mergeable as-is because it is stale against the bot-core refactor, lacks required real Telegram proof, and still has an authorization-sensitive callback fallback.

## What This Changes

Adds optional Telegram `customCommands` menus/routes config, menu-backed slash-command replies, and callback routing for inline menu navigation.

## Best Possible Solution

Implement this as Telegram-owned custom-command menu support in the current `bot-core` and `command-config` paths, with structured callback ownership, command-specific auth on fallback, payload validation, focused tests/docs, and live Telegram proof.

## Reproduction Assessment

Yes for source-level PR review: current main lacks menus/routes, and the PR diff shows the new menu routing and fallback path. No live Telegram reproduction was established because no proof artifact is attached.

## Solution Assessment

No. The feature direction is plausible, but this patch should be rebased into current bot-core/extension-local config, secure the callback fallback, validate payloads, and provide real Telegram proof before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth on menu fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1576-1585`
  - body: When a menu button is not in `routes`, this forwards `callback_data` to `processMessage` after only generic callback authorization. In a restricted group, a different member can click an existing menu action and trigger behavior the slash-command path would have denied, so fallback needs command-specific auth or no AI fallthrough.
  - confidence: 0.9
- **[P1] Move menu wiring into bot-core:** `extensions/telegram/src/bot.ts:493`
  - body: Current main's `bot.ts` is only a wrapper around `createTelegramBotCore`, but this branch threads `validatedCustomCommandIndices` through the old `createTelegramBot` body. Rebase the registration and handler plumbing into `bot-core` so the menu callbacks are wired in the active runtime.
  - confidence: 0.88
- **[P2] Keep command config helpers extension-local:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: The PR widens the old core/plugin-sdk config-runtime surface for Telegram custom-command validation, while current Telegram owns parsing in `extensions/telegram/src/command-config.ts`. Add the indices helper there instead of recreating the removed core helper path.
  - confidence: 0.86
- **[P2] Validate callback payloads before sending menus:** `src/config/zod-schema.providers-core.ts:130`
  - body: `menus.buttons[*].callback_data` accepts blank and over-64-byte strings, and the new send/edit paths pass the keyboard directly to Telegram. Bad config should fail during parsing instead of turning `/command` into a Bot API failure.
  - confidence: 0.86
