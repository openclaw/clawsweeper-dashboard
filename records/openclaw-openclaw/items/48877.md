## Summary

Keep open: current main does not implement multi-level Telegram custom-command menus, but this PR is not merge-ready because source review still shows callback authorization and callback payload validation blockers, the branch conflicts with current main, and real Telegram proof is missing.

## What This Changes

The PR adds optional Telegram `customCommands` `menus`/`routes` config, registers menu-backed Telegram bot commands, and routes matching inline keyboard callbacks to configured menu pages.

## Best Possible Solution

Rebase onto current main, model custom menus as one command-owned Telegram interactive surface, validate callback payloads before send/edit, add focused callback/config tests, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes for the patch defects by source inspection: the PR diff shows menu-owned non-route callbacks falling through to generic callback handling, and the schema accepts raw `callback_data`. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but the implementation should preserve command-owned authorization across the whole menu interaction, validate Telegram callback payloads, and rebase onto the current command-config boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Recheck command auth before forwarding menu callbacks:** `extensions/telegram/src/bot-handlers.runtime.ts:1531-1532`
  - body: When a custom-menu button is not in `routes`, this block leaves it to the generic callback-to-agent fallback after only callback-scope/allowlist authorization. In shared group chats, that can let another callback-authorized room member drive a command that only an authorized user could open, so carry the owning custom-command authorization through before forwarding fallback payloads.
  - confidence: 0.88
- **[P2] Reject invalid menu callback payloads:** `src/config/zod-schema.providers-core.ts:130-131`
  - body: `callback_data: z.string()` accepts empty and over-64-byte values, while the new send/edit paths pass configured `menu.buttons` directly to Telegram. Existing inline-button paths sanitize callback data before sending, so menu config should reject invalid payloads instead of making a configured command fail at runtime.
  - confidence: 0.83
