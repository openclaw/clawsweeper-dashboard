## Summary

Keep this PR open: current main still lacks Telegram custom-command menu support, but the branch is not mergeable as-is because it targets stale wiring/import surfaces and leaves Telegram callback payload validation too loose.

## What This Changes

The PR adds Telegram `customCommands` `menus`/`routes` config, menu-backed slash-command handlers, callback navigation, and a helper for filtering validated custom-command entries.

## Best Possible Solution

Land a refreshed Telegram-owned implementation on current main that adds typed and validated menu config, wires through `bot-core.ts`, documents/changelogs the public config, and preserves existing command auth and callback scoping.

## Reproduction Assessment

Yes. Source inspection on current main shows only `command`/`description` custom commands, a strict schema without `menus`/`routes`, and docs that explicitly state custom commands are menu entries only.

## Solution Assessment

No. The feature direction is reasonable, but the current patch is not the best mergeable form because it targets stale wiring/import surfaces and accepts invalid Telegram callback payloads until runtime.

This will fail to build until the helper is added or re-exported from the surface being imported, or kept plugin-local without the SDK import.
  - confidence: 0.9

  - body: `callback_data` accepts any string while the menu send/edit paths pass buttons to Telegram. Telegram callback data is limited to 64 bytes and existing interactive buttons sanitize through the callback-data helper, so blank or overlong menu payloads should be rejected during config parsing or sanitized before API calls.
  - confidence: 0.86
