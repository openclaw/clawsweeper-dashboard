## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth before fallback:** `extensions/telegram/src/bot-handlers.runtime.ts:1583-1585`
  - body: When a custom menu button is not in routes, this path forwards callback_data to processMessage after only generic callback authorization. In a restricted group, another member can click an existing menu action and drive behavior they could not invoke through the slash-command path, so fallback needs command-specific auth or should not fall through to AI.
  - confidence: 0.9
- **[P1] Move menu registration into bot-core:** `extensions/telegram/src/bot.ts:493`
  - body: Current main's bot.ts is only a wrapper around createTelegramBotCore, but this branch threads validatedCustomCommandIndices through the older bot.ts body. Rebase the registration and handler plumbing into bot-core so the callback handler actually receives the validated set after merge.
  - confidence: 0.88
- **[P2] Keep command config helpers local to Telegram:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: The PR adds the new validation helper through the old core/plugin-sdk config-runtime surface, while current Telegram production code owns custom command parsing in extensions/telegram/src/command-config.ts. Put the index-preserving helper there instead of widening the deprecated core/SDK config surface.
  - confidence: 0.86
- **[P2] Validate callback payloads before sending menus:** `src/config/zod-schema.providers-core.ts:130`
  - body: callback_data accepts blank and overlong strings, and the new send/edit paths pass menu buttons directly to Telegram. Reject invalid payloads during config parsing or reuse the existing sanitizer so a bad menu config fails early instead of at the Telegram API call.
  - confidence: 0.86
