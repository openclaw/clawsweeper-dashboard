## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recheck custom-command auth before fallback actions:** `extensions/telegram/src/bot-handlers.runtime.ts:1536-1537`
  - body: For menu buttons that are not in routes, this code skips menu handling and lets the existing synthetic-message fallback process callback_data after only generic callback authorization. In a group with command restrictions, another member can click an existing menu action and drive behavior they could not invoke through the slash-command path, so fallback needs command-specific auth or should not fall through to AI.
  - confidence: 0.9
- **[P1] Move menu wiring into bot-core:** `extensions/telegram/src/bot.ts:493`
  - body: Current main's createTelegramBot is only a wrapper around createTelegramBotCore, but this patch passes validatedCustomCommandIndices through the older bot.ts body. Rebase the registration and handler plumbing into bot-core so the active Telegram runtime receives the validated set after merge.
  - confidence: 0.88
- **[P2] Keep custom-command validation extension-local:** `extensions/telegram/src/bot-native-commands.ts:9-14`
  - body: The branch adds the validation helper through the old plugin-sdk config-runtime surface, while current Telegram production code owns custom command parsing in extensions/telegram/src/command-config.ts. Put the full-shape helper there instead of widening the deprecated SDK/core config surface.
  - confidence: 0.86
- **[P2] Validate callback_data before sending menus:** `src/config/zod-schema.providers-core.ts:130`
  - body: The schema accepts blank and overlong callback_data values, and the new send/edit paths pass menu buttons straight to Telegram. Reuse the existing Telegram callback-data validation so bad menu config fails early instead of making the slash command fail at the Bot API call.
  - confidence: 0.86
