## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks declarative Telegram custom-command menus, so the branch remains useful source work; the latest patch still has a build-surface blocker and a config-validation gap.

## What This Changes

The PR adds Telegram `customCommands` `menus`/`routes` config plus menu-backed slash-command handlers and callback navigation.

## Best Possible Solution

Land a refreshed Telegram-owned implementation that targets the current command-config/SDK layout, validates callback payloads before Telegram API calls, documents the public config shape, and preserves native command auth and callback scoping.

## Reproduction Assessment

Yes. Source inspection on current main shows only `command`/`description` custom commands, a strict schema without `menus`/`routes`, and docs that explicitly say custom commands are menu entries only.

## Solution Assessment

No. The feature direction is reasonable, but the current patch is not the best mergeable form because it imports a helper from the wrong/currently unexported surface and accepts invalid Telegram callback payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Export the validated-index helper from the imported surface:** `extensions/telegram/src/bot-native-commands.ts:12`
  - body: The new import asks `openclaw/plugin-sdk/config-runtime` for `resolveValidatedCustomCommandIndices`, but this PR only adds the helper in `src/config/telegram-custom-commands.ts` and does not re-export it from the imported SDK surface. Current main has moved this logic to `src/plugin-sdk/telegram-command-config.ts`, so the branch will fail to build until the helper is added or re-exported from the current surface.
  - confidence: 0.9
- **[P2] Validate custom-menu callback payloads before sending:** `src/config/zod-schema.providers-core.ts:129`
  - body: The schema accepts any string for button `callback_data`, while the menu send/edit paths pass these buttons directly to Telegram. Blank or over-64-byte payloads already get dropped by the existing inline-button sanitizer, so this new config path should reject or sanitize them during parsing to avoid runtime Telegram API failures.
  - confidence: 0.86
