## Summary

Keep open: the Telegram menu feature is plausible and not already present on current main, but this PR is not merge-ready because it is now conflicting, lacks inspectable real Telegram proof, and still has callback authorization and callback payload validation blockers.

## What This Changes

The branch adds optional Telegram customCommand menus and routes, registers menu-backed slash command handlers, and routes inline callback queries to configured menu pages.

## Best Possible Solution

Rebase onto current main, model custom menus as a command-owned Telegram interactive surface, enforce command authorization for both route and fallback callbacks, validate callback payloads before Telegram API calls, add focused tests, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes for the patch defects by source inspection: the PR route block only handles mapped routes and leaves non-route menu callbacks to generic callback dispatch, and the schema accepts arbitrary callback_data. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but the current patch should preserve command authorization across the whole menu interaction and validate Telegram callback payloads before it is merge-safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Recheck command auth before forwarding menu callbacks:** `extensions/telegram/src/bot-handlers.runtime.ts:1533`
  - body: When a custom menu button belongs to a command but is not mapped in `routes`, this branch continues and the callback falls through to generic callback-to-agent dispatch after only button-event authorization. In a restricted group command, another callback-authorized member could drive the posted menu, so fallback callbacks need the owning command authorization before `processMessage`.
  - confidence: 0.88
- **[P2] Validate custom-menu callback payloads before Telegram sends:** `src/config/zod-schema.providers-core.ts:130`
  - body: The schema accepts any string for `callback_data`, but the new send/edit paths pass configured buttons directly to Telegram. Existing inline-button conversion sanitizes Telegram callback data, so blank or over-64-byte payloads should fail config validation instead of making the command fail at runtime.
  - confidence: 0.83
