## Summary

Keep open. Current main has session-pruning docs, cache-TTL pruning knobs, inline tool-result trim markers, and a Control UI context-pressure notice, but it still lacks the requested pruning preview/history, lifecycle event or hook, user-facing pruning notifications, and keyword/tag retention strategy. Related open issues #8606 and #9409 cover narrower pieces but do not supersede this umbrella request.

## What This Changes

Request better visibility and control for context pruning, including preview/logging, configurable retention strategy, UI token indicators, and pruning notifications.

## Best Possible Solution

Keep this as a design umbrella unless maintainers split it: add safe pruning metadata to the pruning path, expose a pruning lifecycle event, surface concise Control UI/chat notices or history, document the behavior, and keep raw pruned content out of previews unless explicitly authorized.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs checks give high-confidence evidence that pruning runs today but only returns transformed messages, while UI/docs expose context pressure and configuration rather than pruning history or notifications.

## Solution Assessment

Unclear until maintainers approve the product boundary. The narrowest maintainable direction is metadata/events first, with UI/docs layered on top and privacy-preserving summaries rather than raw pruned-output previews by default.
