## Summary

Keep open. Current main has pruning docs, cache-TTL trim knobs, inline tool-result trim markers, `/context` diagnostics, and a Control UI context-pressure notice, but it still lacks the central requested pruning preview/history, lifecycle metadata or hook, pruning notifications, and keyword/tag retention strategy.

## What This Changes

The issue requests better visibility and control for context pruning, including preview/logging, configurable retention, UI token indicators, and pruning notifications.

## Best Possible Solution

Keep this as a design umbrella unless maintainers split it: add safe pruning metadata/events first, then layer Control UI/chat notices, docs, and optional redacted history on top.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs checks give high-confidence evidence that pruning runs today but only returns transformed messages, while UI/docs expose context pressure and configuration rather than pruning history or notifications.

## Solution Assessment

Unclear until maintainers approve the product boundary. The narrowest maintainable direction is metadata/events first, with privacy-preserving summaries rather than raw pruned-output previews by default.
