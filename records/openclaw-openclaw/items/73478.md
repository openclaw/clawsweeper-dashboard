## Summary

Keep open. Current main still projects chat-linked assistant Gateway events onto the WebSocket `chat` stream as text-only payloads, even though assistant event data can carry media URLs and WebChat already has managed outgoing image machinery for non-streaming replies.

## What This Changes

The issue reports that Gateway WebSocket chat.send/chat.history output for generated images should include image/media information instead of only the assistant text.

## Best Possible Solution

Add a generic Gateway agent-event media projection path that carries assistant event media into WebSocket final chat output and chat.history through the existing managed outgoing image attachment machinery, with focused tests for mixed text+media and media-only assistant events.

## Reproduction Assessment

Yes. A focused source-level reproduction can register a chat-linked run, emit an assistant event with `data.text` plus `data.mediaUrls`, then emit lifecycle end; current main builds the chat delta/final payloads from text buffers only.

## Solution Assessment

Yes. Reusing or factoring the existing managed image projection is the narrowest maintainable solution; provider-specific MiniMax handling or a docs-only change would leave the Gateway contract gap in place.
