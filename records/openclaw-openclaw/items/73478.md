## Summary

Keep open. Current main still accepts assistant media in agent events, but the Gateway agent-event-to-WebSocket chat bridge emits only text blocks for delta, flush, and final chat payloads, so the reported generated-image output gap remains valid and narrow enough for a focused repair PR.

## What This Changes

The issue requests Gateway WebSocket `chat` and `chat.history` output to include generated image information for an agent image-generation reply.

## Best Possible Solution

Project assistant `mediaUrls` from Gateway agent events into WebSocket `chat` final output and `chat.history` through the existing managed outgoing image attachment path, keeping the bridge provider-neutral and avoiding raw base64 or local file paths in client-visible payloads.

## Reproduction Assessment

Yes. A focused source-level reproduction can register a chat-linked run, emit an assistant event with text plus `data.mediaUrls`, then end the lifecycle; current main's chat payload construction only emits text content.

## Solution Assessment

Yes. Reusing or factoring the existing managed image projection used by `chat.send` is the narrowest maintainable fix; a MiniMax-specific change or docs-only change would leave the Gateway contract gap in place.
