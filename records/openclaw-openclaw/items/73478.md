## Summary

Keep open. Current main still has a Gateway projection gap: assistant events can carry `mediaUrls`, but the chat-linked WebSocket `chat` delta/final path still builds text-only messages and does not reuse the managed outgoing image attachment path used by non-agent WebChat replies.

## What This Changes

The issue requests that Gateway WebSocket chat output and `chat.history` include generated image/media attachment information for `chat.send` agent replies instead of returning only assistant text.

## Best Possible Solution

Add a generic Gateway agent-event media projection path that carries assistant event `mediaUrls` into live/final chat output and persisted WebChat history through the existing managed outgoing image attachment machinery, without MiniMax-specific handling.

## Reproduction Assessment

Yes. A focused source-level reproduction can register a chat-linked run, emit an assistant event with `data.text` plus `data.mediaUrls`, then end the lifecycle; current main reads only text/delta for the live and final chat projection.

## Solution Assessment

Yes. Reusing or factoring the existing managed WebChat image projection is the narrowest maintainable fix; provider-specific handling or docs-only changes would leave the Gateway contract gap in place.
