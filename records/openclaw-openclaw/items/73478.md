## Summary

Current main still has a source-reproducible gateway chat media-loss bug: assistant `mediaUrls` are allowed by the protocol and preserved on the lower-level `agent` event path, but the user-facing `chat` delta/final payloads are still built as text-only content. The linked fixing PR is open and conflicting, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that gateway WebSocket chat/history output drops generated image information and only includes assistant text content.

## Best Possible Solution

Land or replace the linked gateway fix so assistant media URLs are projected into gateway chat content alongside text while preserving text-only behavior.

## Reproduction Assessment

Yes, by source inspection: a chat-linked assistant event with `data.text` and `data.mediaUrls` reaches the assistant stream path, but the emitted `chat` message content is constructed as a single text block and media-only assistant events are not mirrored into chat deltas.

## Solution Assessment

Yes in direction: the fix belongs in the gateway chat projection path that already owns delta, throttle flush, and final chat payloads; the linked PR targets that boundary, but it still needs normal PR review and conflict resolution.
