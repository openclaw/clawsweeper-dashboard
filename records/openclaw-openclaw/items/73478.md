## Summary

Keep open. Current main still projects chat-linked assistant Gateway events onto the WebSocket `chat` stream as text-only payloads, while the assistant event data model can carry `mediaUrls` and the non-agent WebChat final-reply path already has managed image attachment support. The prior ClawSweeper comment is still consistent with current source; no open canonical fix PR was found in the provided/search context.

## What This Changes

The issue requests that Gateway WebSocket `chat` events and `chat.history` expose generated image information for a MiniMax-backed `chat.send` flow instead of only assistant text.

## Best Possible Solution

Add a generic Gateway agent-event media projection path for chat-linked runs that carries assistant event media into final WebSocket chat output and persisted `chat.history` through the existing managed outgoing image attachment machinery, with focused tests for mixed text+media and media-only assistant events.

## Reproduction Assessment

Yes. A focused source-level reproduction can register a chat-linked run, emit an assistant event with `data.text` plus `data.mediaUrls` or a media-only assistant event, then emit lifecycle end; current main builds the live chat payloads from text buffers only.

## Solution Assessment

Yes. Reusing or factoring the existing managed image projection is the narrowest maintainable fix because the assistant event schema already carries media URLs; a MiniMax-specific workaround or docs-only change would leave the Gateway contract gap in place.
