## Summary

Keep open. Current main still has a concrete Gateway behavior gap: assistant agent events can carry media URLs, but the chat-linked WebSocket projection builds live `chat` delta/final payloads from text buffers only, so generated images can be absent from the `chat` stream and follow-on history unless they arrive through the separate non-streaming reply-dispatch path.

## What This Changes

Gateway WebSocket `chat` and `chat.history` should include generated image/media information instead of only text when a `chat.send` run produces image output.

## Best Possible Solution

Add a generic Gateway agent-event media projection path that carries assistant event `mediaUrls` into live final chat output and persisted WebChat history through the existing managed outgoing image attachment machinery, without MiniMax-specific handling.

## Reproduction Assessment

Yes. A focused source-level reproduction can register a chat-linked run, emit an assistant event with `data.text` plus `data.mediaUrls`, then emit lifecycle end; current main reads only `text`/`delta` for live chat and final projection.

## Solution Assessment

Yes. Reusing or factoring the existing managed image projection is the narrowest maintainable fix; provider-specific handling or a docs-only change would leave the Gateway contract gap in place.
