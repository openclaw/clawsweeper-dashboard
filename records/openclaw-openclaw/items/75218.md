## Summary

Keep open. Current main still lacks a Tavily `web_fetch` provider, and the PR is a useful extension-owned direction, but the branch is not merge-ready: the provider returns Tavily Extract's nested `results[]` payload instead of the top-level `text` shape that core normalizes, the shared Tavily credential/base-url resolvers would make fetch-specific overrides affect Tavily search/extract, and the user-facing feature is missing the required changelog entry.

## What This Changes

The PR adds a Tavily `web_fetch` provider, Tavily manifest/docs/secret metadata for `webFetch`, Tavily Extract `format` forwarding, and tests for the new provider path.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` fallback after the provider maps a single Tavily Extract result into the existing top-level provider payload contract, uses fetch-specific credential/base-url resolution while preserving current Tavily search/extract behavior, and records the feature in the active changelog.

## Reproduction Assessment

Yes. Force the `web_fetch` provider fallback path with Tavily selected: current core reads only `payload.text`, while the PR provider returns Tavily Extract's `results[]` shape, yielding an empty final text field. The resolver leak is also reproducible by setting `plugins.entries.tavily.config.webFetch.*` and invoking Tavily search or `tavily_extract`.

## Solution Assessment

No. The product direction is reasonable, but the implementation should satisfy the existing provider fallback contract and add fetch-scoped Tavily resolvers instead of changing shared search/extract resolution.
