## Summary

Keep open. Current main still lacks Tavily as a native `web_fetch` provider, so this PR targets a real gap, but the head still has blocking provider-shape and shared-config issues plus a missing changelog entry; the existing ClawSweeper comment already identified the same narrow blockers.

## What This Changes

The PR adds a bundled Tavily `web_fetch` provider, Tavily manifest/docs/secret metadata, `format` forwarding for Extract, and Tavily provider tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after it maps Extract into the existing provider fallback payload shape, keeps fetch-only config resolution scoped to fetch, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. Force `web_fetch` through the Tavily provider fallback: core normalizes only `payload.text`, while this PR returns Tavily Extract's nested `results[]` shape, so the final fetch text can be empty despite a successful extract.

## Solution Assessment

No. The product direction is reasonable, but the implementation should adapt to the existing provider contract and add fetch-specific Tavily credential/base URL resolution instead of changing shared search/extract helpers.
