## Summary

Keep this PR open: the Tavily `web_fetch` direction fits the plugin-owned provider boundary, but the branch still breaks the provider fallback payload contract, leaks fetch-specific overrides into existing Tavily search/extract paths, and lacks the required changelog entry.

## What This Changes

The PR adds a bundled Tavily `web_fetch` provider backed by `runTavilyExtract`, with provider registration, manifest/config/docs/secret metadata, and Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after it maps Extract output into the existing top-level provider payload shape, scopes fetch overrides to fetch only, and records the feature in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Force `web_fetch` onto the Tavily provider fallback path: core consumes only provider `payload.text`, while this PR returns Tavily Extract `results[]`, so the final tool text can normalize to empty.

## Solution Assessment

No. The extension-owned native provider is a maintainable direction, but this implementation needs narrower payload mapping and fetch-specific credential/base URL resolution before merge.
