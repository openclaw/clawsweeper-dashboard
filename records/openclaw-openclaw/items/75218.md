## Summary

Keep open. Current main does not already implement Tavily as a `web_fetch` provider, so the PR is not obsolete, but the proposed patch is not merge-ready because the provider returns the Tavily Extract result shape instead of the top-level `web_fetch` provider payload shape and it scopes new fetch overrides through shared Tavily search resolution.

## What This Changes

The PR adds Tavily as a bundled `web_fetch` provider with provider registration, manifest contracts, `webFetch` config/docs/secretref metadata, Tavily Extract format support, and extension tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after it maps one Tavily Extract result into the existing top-level provider fallback contract, keeps search/extract and fetch credential resolution separate, preserves existing Tavily tool behavior, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is enough: select Tavily for provider fallback and force the fallback path; current core reads only `payload.text`, while the PR provider returns nested Tavily Extract results. The resolver leak is reproduced by configuring `plugins.entries.tavily.config.webFetch.*` and then invoking Tavily search, which uses the same shared helpers in the PR.

## Solution Assessment

No. The product direction is reasonable, but this implementation is not the narrowest maintainable solution because it violates the provider fallback payload contract and changes shared Tavily search resolution. A safer fix is a provider-local adapter plus fetch-specific credential/base-url helpers.
