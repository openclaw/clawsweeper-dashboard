## Summary

Keep open. Current main has no native Tavily `web_fetch` provider, and the PR is a plausible feature, but the branch still returns a successful empty provider result when Tavily Extract has no usable content and it lacks real after-fix behavior proof.

## What This Changes

The PR registers a Tavily `web_fetch` provider backed by Tavily Extract, adds Tavily fetch config and secret metadata, updates docs/changelog, and extends Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after empty or failed Extract responses become provider failures with regression coverage and after redacted real `web_fetch` proof is added.

## Reproduction Assessment

Yes, for the blocking review finding from source inspection: make the PR provider's `runTavilyExtract` return no usable `results`, and the provider returns `text: ""`. Current core normalizes and caches returned provider payloads as final.

## Solution Assessment

No. The native Tavily provider direction fits the existing plugin/provider seam, but the implementation should throw or surface Tavily failure state when Extract returns no content instead of returning a successful empty `web_fetch` result.
