## Summary

Keep open. Current main still exposes Tavily as a web_search provider and explicit tools only, and this PR is the active implementation attempt, but the patch is not merge-ready because the new provider returns Tavily Extract's nested result shape while core consumes top-level `payload.text`, and fetch overrides would change Tavily search/extract globally through shared resolvers.

## What This Changes

The branch registers a Tavily `web_fetch` provider, adds Tavily webFetch config, manifest, docs, and SecretRef metadata, routes provider calls through Tavily Extract, and extends Tavily tests.

## Best Possible Solution

Land Tavily as extension-owned `web_fetch` fallback support after it maps a single Tavily Extract result into the existing provider payload contract, keeps fetch credential/base-url resolution separate from search resolution, and records the user-facing feature in the active changelog.

## Reproduction Assessment

Yes. Force the PR branch's `web_fetch` provider fallback path with Tavily selected and a mocked `runTavilyExtract` returning `results[0].rawContent`; core normalization reads `payload.text`, so the final response text is empty. The resolver leak is reproducible by setting `plugins.entries.tavily.config.webFetch.*` and then invoking Tavily search, which uses the same shared resolver.

## Solution Assessment

No. The product direction is reasonable, but the implementation should satisfy the existing provider fallback contract and use fetch-scoped Tavily credential/base-url helpers instead of changing shared search helpers.
