## Summary

Keep open. Current main still lacks Tavily as a `web_fetch` provider, so this PR is not obsolete, but the head is not merge-ready: it returns Tavily Extract's nested `results` shape to a core fallback that only reads top-level `payload.text`, and it changes shared Tavily config resolvers so fetch-only overrides affect `web_search` and `tavily_search`. The remaining work is a narrow extension-owned repair plus changelog.

## What This Changes

The PR adds a Tavily `web_fetch` provider registration, manifest contract/config and secretref/docs updates, a web-fetch contract artifact, Tavily Extract format plumbing, and tests covering provider metadata and argument mapping.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after it maps Tavily Extract output into the existing provider fallback payload contract, separates fetch-specific credential/base URL resolution from search resolution, and records the user-facing feature in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A source-level reproduction is to select the Tavily provider through `createWebFetchTool`: core normalizes only top-level `payload.text`, while the PR provider returns Tavily Extract `results`, so the final `text` is empty. The config leak is reproducible by setting `plugins.entries.tavily.config.webFetch.*` and invoking Tavily search, which uses the same shared resolvers.

## Solution Assessment

No. The product direction is reasonable, but the implementation should satisfy the existing provider fallback contract and use fetch-scoped Tavily credential/base-url helpers instead of changing shared search resolution.
