## Summary

Keep open: the native Tavily provider direction is reasonable, but the current PR can still return and cache a successful empty `web_fetch` result when Tavily Extract has no usable content.

## What This Changes

The PR adds a bundled Tavily `web_fetch` provider backed by Tavily Extract, plus Tavily config schema, docs, secret metadata, tests, and a changelog entry.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after empty or failed Extract responses are treated as provider failures with regression coverage.

## Reproduction Assessment

Yes, from source inspection. Force the direct/readability path to call the Tavily provider and mock Tavily Extract to return no result with `rawContent` or `content`; the PR builds `text: ""`, and current core returns/caches any provider payload.

## Solution Assessment

No. The native Tavily provider is a maintainable direction, but it should throw or surface Tavily `failedResults` when no usable extracted content exists instead of producing a successful empty `web_fetch`.
