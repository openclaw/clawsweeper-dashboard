## Summary

Keep open. Current main still lacks Tavily as a `web_fetch` provider, so the feature is useful, but the PR is not merge-ready because the provider returns Tavily Extract’s nested result shape to a core fallback that consumes only top-level `text`, and its fetch override plumbing changes existing Tavily search/extract credential behavior.

## What This Changes

The PR adds a bundled Tavily `web_fetch` provider, Tavily manifest/docs/secret metadata, extract `format` forwarding, and Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` fallback after it maps a single Extract result into the existing top-level provider payload, keeps fetch overrides scoped to fetch, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. Force the provider fallback path with Tavily selected: current core passes the provider payload into `normalizeProviderWebFetchPayload`, which reads only `payload.text`, while the PR provider returns Tavily Extract’s `results[]` shape.

## Solution Assessment

No. The product direction is reasonable, but the implementation should satisfy the existing provider payload contract and add fetch-specific Tavily resolvers instead of changing shared search/extract resolution.
