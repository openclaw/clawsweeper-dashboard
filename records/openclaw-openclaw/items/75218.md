## Summary

Keep open. The Tavily web_fetch feature is not on current main, but this PR still lacks real behavior proof and the latest diff can turn an empty or failed Tavily Extract response into a cached successful empty web_fetch result.

## What This Changes

The PR adds a Tavily web_fetch provider to the bundled Tavily plugin, including provider registration, manifest/config metadata, Tavily client wiring, docs/secret-surface updates, and tests.

## Best Possible Solution

Land Tavily as an extension-owned web_fetch provider after empty or failed Extract responses become provider failures with regression coverage and after redacted real web_fetch proof is added.

## Reproduction Assessment

Yes, for the blocking review finding: source inspection shows that if runTavilyExtract returns no usable rawContent/content, the PR provider returns text: "" and current core caches that as a successful web_fetch payload.

## Solution Assessment

No. The native Tavily provider direction fits the existing plugin/provider seam, but the implementation should throw or surface Tavily failure state when Extract returns no content instead of returning a successful empty result.
