## Summary

Keep open. The Tavily `web_fetch` direction fits the existing bundled Tavily plugin, but this PR is not merge-ready because the provider returns Tavily Extract's nested `results[]` payload while core fallback normalization consumes only top-level `text`, and the new fetch override is wired into shared Tavily search/extract credential resolution.

## What This Changes

The PR adds Tavily as a bundled `web_fetch` provider with manifest/docs/secret metadata, fetch config, Extract format forwarding, and Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after it conforms to the existing provider payload shape, scopes fetch-only credentials to fetch, and records the user-facing change in the active changelog.

## Reproduction Assessment

Yes. Force the provider fallback path with Tavily selected; current core passes the raw provider result to `normalizeProviderWebFetchPayload`, which reads only `payload.text`, while the PR provider returns Tavily Extract's nested `results[]` shape.

## Solution Assessment

No. The product direction is reasonable, but the implementation should map a single Extract result into the existing top-level web-fetch provider payload and use fetch-specific Tavily credential/base URL resolvers.
