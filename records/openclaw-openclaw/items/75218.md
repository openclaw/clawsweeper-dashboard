## Summary

Keep open. Current main still lacks Tavily as a bundled `web_fetch` provider, so the PR is useful, but the proposed patch is not merge-ready: it returns Tavily Extract's nested result shape where core expects top-level `payload.text`, changes shared Tavily credential/base URL helpers used by search, and omits the required user-facing changelog entry.

## What This Changes

The PR adds a Tavily `web_fetch` provider registration, manifest/docs/secret metadata, `webFetch` config support, Tavily Extract `format` forwarding, and Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` fallback after the provider maps a single Tavily Extract result into the existing top-level provider payload contract, uses fetch-specific credential/base-url resolution, preserves current Tavily search behavior, and records the feature in the changelog.

## Reproduction Assessment

Yes. Force the `web_fetch` provider fallback path with Tavily selected: current core reads only `payload.text`, while the PR provider returns Tavily Extract's `results[].rawContent`/`content` shape, yielding an empty final `text` field. The resolver leak is also reproducible by setting `plugins.entries.tavily.config.webFetch.*` and invoking Tavily search.

## Solution Assessment

No. The product direction is reasonable, but the implementation should satisfy the existing provider fallback contract and add fetch-scoped Tavily resolvers instead of changing shared search/extract resolution.
