## Summary

Keep this PR open: the Tavily web_fetch provider is a plausible feature on the existing plugin/provider seam, but the current head still turns no-content Tavily Extract responses into successful empty web_fetch results and the external PR still lacks required real behavior proof.

## What This Changes

The PR adds a bundled Tavily web_fetch provider with Tavily Extract wiring, config/docs/secret metadata, and Tavily plugin tests.

## Best Possible Solution

Land Tavily as a plugin-owned web_fetch provider after no-content or failed Extract responses become provider failures with regression coverage and after redacted live web_fetch proof is added.

## Reproduction Assessment

Yes, for the blocking review finding: source inspection at PR head shows empty text is returned as a normal provider payload, and current main then normalizes and caches that payload as a successful web_fetch result. I did not run tests because this review is read-only.

## Solution Assessment

No. The Tavily provider direction fits the existing plugin/provider seam, but the provider should throw or surface Tavily failure state when Extract returns no usable content instead of returning a successful empty result.
