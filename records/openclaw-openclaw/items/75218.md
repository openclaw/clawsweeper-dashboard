## Summary

## What This Changes

The PR registers Tavily as a bundled web_fetch provider, adds Tavily webFetch config/docs/SecretRef surface, and extends tests for provider selection and extract mapping.

## Best Possible Solution

Keep the Tavily provider addition, but make the fetch override credential path/resolution match the documented config and add redacted live web_fetch proof before merge.

## Reproduction Assessment

Yes for the review finding at source level: a webFetch.apiKey SecretRef is read as the primary Tavily fetch credential while the generic resolver materializes the first advertised path, webSearch.apiKey, leaving the fetch SecretRef to throw on strict runtime read. I did not run a live Tavily call because this is a read-only review and the PR lacks a redacted live artifact.

## Solution Assessment

No. The feature belongs in the Tavily plugin seam, but the credential path should be made path-safe, preferably by treating webFetch.apiKey as the primary fetch credential and webSearch.apiKey as a configured fallback like Firecrawl, before adding real behavior proof.
