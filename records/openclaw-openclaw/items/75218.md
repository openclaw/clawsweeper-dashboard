## Summary

Keep open; the branch still has merge-blocking implementation issues in the Tavily provider payload shape and credential/base URL scoping, plus a required changelog entry is missing.

## What This Changes

The PR adds a bundled Tavily `web_fetch` provider backed by Tavily Extract, updates Tavily manifest/config/docs/secret metadata, forwards Extract `format`, and adds Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after it returns the existing top-level fallback payload shape, keeps fetch-only credential/base URL overrides scoped to fetch, avoids double-wrapping extracted content, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. Force the provider fallback path with Tavily selected: core passes the provider result into `normalizeProviderWebFetchPayload`, which reads `payload.text`, while the PR provider returns Tavily Extract `results[]` without top-level `text`.

## Solution Assessment

No. The native Tavily `web_fetch` direction is reasonable, but this implementation should satisfy the existing provider fallback contract and add fetch-specific Tavily credential/base URL resolution instead of changing shared search/extract resolution.
