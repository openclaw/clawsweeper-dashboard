## Summary

Keep open. Current main does not implement Tavily as a `web_fetch` provider, and the PR still has merge-blocking issues: empty Tavily Extract responses become successful empty `web_fetch` payloads, and the SecretRef surface docs no longer match the generated credential matrix.

## What This Changes

The PR registers Tavily as a bundled `web_fetch` provider backed by `/extract`, adds Tavily fetch config and SecretRef/docs updates, and extends Tavily extension tests plus the changelog.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after empty or failed Extract responses are treated as provider failures and the SecretRef generated docs stay in sync with the registry.

## Reproduction Assessment

Yes, from source inspection. Mock or force `runTavilyExtract` to return no usable `results`; the PR provider returns `text: ""`, and current core accepts that as the final cached provider result.

## Solution Assessment

No. The native Tavily provider direction is reasonable, but the PR should fail or surface Tavily Extract failures when no content exists and should list the generated `plugins.entries.tavily.config.webFetch.apiKey` SecretRef path.
