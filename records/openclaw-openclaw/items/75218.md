## Summary

Keep open. Current main still lacks a Tavily `web_fetch` provider, and this PR is a reasonable extension-owned feature, but the branch is not merge-ready because the provider does not return the current `web_fetch` payload contract, fetch-specific config overrides leak into Tavily search, and the user-facing feature lacks a changelog entry.

## What This Changes

The PR registers Tavily as a bundled `web_fetch` provider, adds Tavily `webFetch` config/docs/secretref metadata, routes provider calls through Tavily Extract, and extends Tavily tests.

## Best Possible Solution

Land the Tavily provider as extension-owned `web_fetch` fallback support after it maps the single Tavily Extract result into the existing top-level `web_fetch` response shape, keeps search and fetch credential/base-url resolution separate, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is to force the `web_fetch` provider fallback path with Tavily selected: current core reads only `payload.text`, while the PR provider returns nested Tavily Extract results, yielding an empty final `text` field. The config leak is reproducible by setting `plugins.entries.tavily.config.webFetch.*` and then invoking Tavily search, which uses the same shared resolver.

## Solution Assessment

No. The product direction is reasonable, but the implementation should satisfy the existing provider fallback contract and use fetch-scoped Tavily resolvers instead of changing shared search resolution.
