## Summary

Keep open: the Tavily web_fetch provider is still useful and current main does not implement it, but PR head still has a blocking empty-success path and lacks required real behavior proof.

## What This Changes

The PR adds Tavily as a bundled web_fetch provider backed by Tavily Extract, with plugin registration, manifest/config metadata, secret-surface docs, user docs, changelog, and tests.

## Best Possible Solution

Land Tavily as an extension-owned web_fetch provider after no-content Extract responses become provider failures with regression coverage and after the contributor adds redacted live web_fetch/Tavily proof.

## Reproduction Assessment

Yes, for the blocking review finding: source inspection of PR head shows no-content Extract payloads produce text: "", and current core wraps/caches any provider payload as success. I did not run a live Tavily repro in this read-only review, and the PR still lacks contributor live proof.

## Solution Assessment

No. The native Tavily provider direction fits the existing plugin-owned web_fetch seam, but the implementation should reject or surface failed/no-content Extract responses instead of returning a successful empty result.
