## Summary

Keep open: the current PR head addresses the earlier top-level text, fetch-scoped resolver, and changelog blockers, but it can still turn a failed or empty Tavily Extract response into a successful empty `web_fetch` result.

## What This Changes

The PR adds a bundled Tavily `web_fetch` provider backed by `/extract`, plus Tavily config, docs, secret metadata, tests, and a changelog entry.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after empty or failed Extract responses are treated as provider failures with regression coverage, while preserving the current resolver and changelog fixes.

## Reproduction Assessment

Yes, from source inspection. Force the provider fallback path and have Tavily Extract return no usable `results`; the PR provider returns `text: ""`, and current core accepts that as the final provider result.

## Solution Assessment

No. The native Tavily provider direction is reasonable, but the implementation should fail or surface Tavily's `failedResults` when no extracted content exists instead of caching a successful empty `web_fetch`.
