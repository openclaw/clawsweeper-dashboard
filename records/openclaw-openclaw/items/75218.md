## Summary

Keep open. Current main still lacks Tavily as a `web_fetch` provider, and the PR is useful but not merge-ready because the fallback result shape and shared Tavily config resolver would regress current behavior.

## What This Changes

The PR adds a Tavily `web_fetch` provider, Tavily manifest/docs/secret metadata, Extract `format` forwarding, and Tavily tests.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` fallback after it maps Extract output into OpenClaw's top-level fallback payload, scopes fetch-only credentials to the new provider, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. Force direct fetch to fail with Tavily selected as the provider fallback; core will pass the provider payload into `normalizeProviderWebFetchPayload`, which reads only top-level `payload.text`.

## Solution Assessment

No, not as-is. The product direction is reasonable, but the implementation should satisfy the existing provider payload contract and avoid changing shared Tavily search/extract credential resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

  - body: Core fallback normalization reads only `payload.text` before returning the final `web_fetch` details.
