## Summary

This PR should stay open: current main does not yet provide Tavily as a `web_fetch` provider, and the PR adds the right extension-owned surfaces, but the provider still treats missing Tavily Extract content as a successful empty fetch.

## What This Changes

The PR registers Tavily as a bundled `web_fetch` provider backed by Tavily Extract, adds Tavily `webFetch` config/secret metadata/docs/changelog, and expands Tavily tests for provider mapping and credential behavior.

## Best Possible Solution

Land Tavily as an extension-owned `web_fetch` provider after empty or failed Extract responses are treated as provider failures with regression coverage, while preserving the current config/docs boundaries.

## Reproduction Assessment

Yes, from source inspection of the PR branch path. Force the provider fallback path and have Tavily Extract return no usable `results`; the PR provider returns `text: ""`, and current core accepts that as the final cached provider result.

## Solution Assessment

No. The native Tavily provider direction is reasonable, but the implementation should throw or surface Tavily `failedResults` when no result has usable `rawContent` or `content` instead of returning an empty success.
