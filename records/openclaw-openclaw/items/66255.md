## Summary

Keep open. Current main still has the crash-prone dependency path: OpenClaw pins Pi 0.71.1, the Pi root export still reaches its clipboard utility, the upstream clipboard utility still loads `@mariozechner/clipboard` on Linux display sessions, and several gateway/agent paths still statically import the Pi root package without an OpenClaw-owned guard. The Fedora/KDE/Wayland follow-up also shows the remaining crash surface is wider than the original headless framing.

## What This Changes

The issue asks OpenClaw to prevent Gateway crash loops caused by Pi's optional native Linux clipboard addon and to make that path guarded, optional, lazy, or isolated.

## Best Possible Solution

Add an OpenClaw-owned Pi import boundary that prevents the native clipboard addon from loading in gateway/background contexts unless clipboard support is actually needed, with regression coverage and without making `TERMUX_VERSION` the supported public workaround.

## Reproduction Assessment

Yes for triage/source reproduction: current main still imports the Pi root package in gateway and embedded-agent startup paths, and upstream Pi 0.71.1 still evaluates a native clipboard require on Linux display sessions. I did not run a live native crash reproduction.

## Solution Assessment

Yes, a narrow OpenClaw-owned pre-import guard or lazy boundary is the best maintainable direction. Renaming the `.node` file or setting `TERMUX_VERSION` is useful workaround evidence, but not a durable OpenClaw contract.
