## Label Justifications

- `P1`: Two users reported Linux gateway crash loops on a supported runtime path, and current main/release still expose the same native clipboard load boundary.
- `impact:crash-loop`: The issue is specifically about repeated gateway process crashes during startup or early channel/runtime initialization.

## Summary

Keep open. Current main and the latest stable release still pin Pi 0.75.1, still carry the optional @mariozechner/clipboard Linux native package, and still have gateway/embedded-runner paths that evaluate the Pi root package; the upstream Pi root export still reaches clipboard-native.js, which requires the native clipboard package on Linux when DISPLAY or WAYLAND_DISPLAY is set. Two reporters supplied crash-loop/coredump evidence and a TERMUX_VERSION workaround, and I found no canonical merged fix for this specific clipboard-native startup path.

## What This Changes

Prevent gateway startup from loading the @mariozechner/clipboard Linux native addon through the embedded Pi runtime on Linux environments where clipboard support is unsafe or irrelevant.

## Best Possible Solution

Keep this open until gateway/background startup can use Pi session/runtime APIs without evaluating Pi's clipboard-native path on Linux, preferably via an upstream Pi lazy/export fix or a narrow OpenClaw import boundary with real Linux proof.

## Reproduction Assessment

Yes for source-level reproduction, but not for a live current-main crash in this review. Current main and v2026.5.18 still evaluate the Pi root package from gateway/embedded-runner paths, and Pi 0.75.1 still reaches clipboard-native.js, while two reporters supplied native coredump/workaround evidence.

## Solution Assessment

No proposed patch is under review. The best direction is to stop startup/background paths from evaluating Pi clipboard-native at all; the TERMUX_VERSION workaround is useful field evidence but should not become the OpenClaw contract.
