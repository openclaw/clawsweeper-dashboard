## Summary

Keep open. Current main still carries the crash-prone path: OpenClaw pins Pi 0.74.0, Pi's root export still reaches the native clipboard loader, and gateway/agent startup code still imports the Pi root package.

## What This Changes

Prevent Linux gateway crash loops caused by Pi loading the optional native `@mariozechner/clipboard` addon during gateway or embedded-agent startup.

## Best Possible Solution

Add an OpenClaw-owned Pi import/runtime boundary, or adopt an upstream Pi fix, so gateway/background startup cannot load the native clipboard addon unless clipboard functionality is actually needed.

## Reproduction Assessment

Yes for source reproduction: gateway and embedded-agent startup paths still import the Pi root package, and Pi 0.74.0 still reaches `clipboard-native.js`, which requires `@mariozechner/clipboard` on Linux display sessions. I did not live-trigger the native crash.

## Solution Assessment

Yes, the narrow maintainable fix is an import boundary, lazy load, or upstream Pi update that keeps native clipboard loading out of gateway startup. The `TERMUX_VERSION` workaround is useful evidence but should not become the OpenClaw contract.
