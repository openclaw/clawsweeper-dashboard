## Summary

Keep open. Current main still has the crash-prone dependency path: OpenClaw pins Pi 0.70.6, the lockfile still includes the Linux native clipboard optional package, upstream Pi still requires `@mariozechner/clipboard` when Linux has `DISPLAY` or `WAYLAND_DISPLAY` and `TERMUX_VERSION` is unset, and gateway/session/embedded paths still import Pi before any OpenClaw-owned clipboard opt-out exists.

## What This Changes

The issue asks OpenClaw to prevent Linux gateway crash loops caused by the embedded Pi runtime loading the native `@mariozechner/clipboard-linux-x64-gnu` addon.

## Best Possible Solution

Add an OpenClaw-owned pre-import guard or lazy boundary that prevents Pi’s native clipboard addon from loading in gateway/background contexts where clipboard access is unnecessary, or upgrade to an upstream Pi release with an explicit supported disable switch. Keep normal desktop clipboard behavior unless a clearly documented OpenClaw opt-out or server/headless mode applies.

## Reproduction Assessment

Yes for triage: the issue has a concrete Linux gateway startup crash path, coredump frames in `clipboard.linux-x64-gnu.node`, and two workarounds that specifically avoid the upstream native require. I did not live-reproduce the signal in this read-only checkout.

## Solution Assessment

Yes, the narrow maintainable direction is an OpenClaw-owned opt-out or import-order guard before top-level Pi imports, with tests that prove the native clipboard package is not evaluated in the protected gateway path. Using `TERMUX_VERSION` as the public workaround is not the best long-term solution because it is an upstream/Termux sentinel rather than an OpenClaw contract.
