## Summary

Keep open. Current main has upgraded Pi to 0.71.1, but the crash-prone dependency path still exists: OpenClaw statically imports the Pi root package from gateway/session and embedded-runner code, Pi's root export still reaches its clipboard utility, and the upstream clipboard-native module still requires `@mariozechner/clipboard` on Linux display sessions unless `TERMUX_VERSION` is set. The issue discussion also adds a second Fedora/KDE/Wayland reproduction, so this is not resolved by the original headless framing.

## What This Changes

The issue asks OpenClaw to prevent gateway crash loops caused by Pi's native Linux clipboard addon loading in gateway/background contexts where clipboard access is unsafe or unnecessary.

## Best Possible Solution

Add an OpenClaw-owned guard or lazy Pi import boundary that prevents Pi's native clipboard addon from loading in gateway/background contexts unless clipboard support is explicitly needed, while preserving normal interactive desktop clipboard behavior. Prefer a clear OpenClaw opt-out or upstream-supported Pi switch over documenting `TERMUX_VERSION` as the public workaround.

## Reproduction Assessment

Yes for triage. The issue has coredump evidence and two independent workarounds that specifically avoid the upstream native clipboard require, and current source still exposes the same dependency/import path; I did not run a live native crash reproduction.

## Solution Assessment

Yes, the best maintainable direction is a narrow pre-import guard or lazy boundary around Pi root imports in gateway/background paths, with tests proving the native clipboard package is not evaluated under the protected runtime. Renaming the `.node` file or using `TERMUX_VERSION` directly is useful workaround evidence but not the right long-term OpenClaw contract.
