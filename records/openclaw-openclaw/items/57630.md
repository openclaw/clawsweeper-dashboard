## Summary

Keep open. Current main has already absorbed the additive `appendContext` half of the request, but the `prompt` replacement field is still absent from the public plugin hook contract and runtime prompt assembly, so this PR is not obsolete; the EdgeClaw/GuardClaw use case remains valid, but the branch needs a current-main rebase or successor patch before it is merge-ready.

## What This Changes

This PR adds optional `appendContext` and `prompt` fields to the prompt-build hook result, merges them, applies them in the embedded runner, and updates hook documentation.

## Best Possible Solution

Land a rebased or replacement patch that adds `prompt?: string` to the current prompt hook result contract, merges it with higher-priority-wins semantics, applies it consistently in embedded and CLI prompt assembly while preserving system-generated warnings and prefixes, updates current English docs, adds focused tests, and includes the required changelog credit.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction path: a plugin cannot type-return `prompt`, the shared prompt resolver does not return it, and both embedded and CLI runners only apply prepend/append context; the PR diff then shows the remaining ordering and CLI gaps.

## Solution Assessment

No. The proposed direction is narrow and compatible, but the branch is not the best current solution because it applies replacement only in the embedded path and overwrites an already-built prompt that can include bootstrap truncation warnings; a current-main patch should implement replacement before wrapping or explicitly re-preserve those generated warnings.

Add the same replacement semantics and coverage to the CLI preparation path.
