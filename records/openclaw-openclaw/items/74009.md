## Summary

Keep open: this PR targets a real `sessions_send` bug that is still present on current `main` and in v2026.5.7, but it should not merge as submitted because the patch routes mixed sessionId/label inputs into a denied-access path that can echo a resolved canonical key and it adds a leaking test mock.

## What This Changes

The PR removes the `sessions_send` `sessionKey`+`label` early rejection, updates the tool description and changelog, and adds regression tests for `sessionKey`/sessionId precedence over redundant `label`/`agentId` hints.

## Best Possible Solution

Land a narrow sessions tool fix that treats explicit `sessionKey`/sessionId as authoritative while preserving denied-access confidentiality, keeping current thread-target guidance, and adding isolated regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current `main` and v2026.5.7 return the reported mutual-exclusion error for non-empty `sessionKey` plus `label`, and a production comment confirms the same failure path.

## Solution Assessment

No as submitted. The precedence direction is the right narrow fix, but the PR should avoid echoing resolved canonical keys on denied sends and reset or one-shot the gateway mock used by the new regression test.

## Review Findings

Overall correctness: patch is incorrect
