## Summary

Keep this PR open, but do not merge the current patch: it fixes the right mixed-selector failure, yet it opens a denied-access key-disclosure path and adds a test mock that can leak into later cases.

## What This Changes

This PR removes the `sessions_send` mixed `sessionKey`/`label` guard, updates the tool description and changelog, and adds regression tests for explicit `sessionKey`/sessionId precedence over redundant `label`/`agentId` hints.

## Best Possible Solution

Land a narrow sessions tool fix that treats explicit `sessionKey`/sessionId as authoritative while preserving denied-access confidentiality, with isolated regression tests and a changelog entry.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction: `sessions_send` with both `sessionKey` and `label` reaches the mutual-exclusion guard before send resolution; the PR’s denial-path regression is also traceable by following sessionId resolution into the existing denied branch.

## Solution Assessment

No as submitted. Preferring explicit `sessionKey` is the right direction, but the implementation should first avoid returning resolved canonical keys on denied sends and reset or one-shot the new gateway mock implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid echoing resolved keys on denied sends:** `src/agents/tools/sessions-send-tool.ts:106`
  - body: Deleting this guard lets sessionId-shaped `sessionKey` inputs with redundant `label`/`agentId` resolve before visibility/A2A checks. If access is denied, the existing branch returns the resolved `displayKey`, so mixed-selector probes can learn a canonical key they cannot send to; return the original selector or omit `sessionKey` on denial.
  - confidence: 0.84
- **[P2] Reset the gateway mock between gating tests:** `src/agents/tools/sessions.test.ts:687-695`
  - body: The added sessionId regression installs a persistent `callGatewayMock.mockImplementation(...)`, but the local `beforeEach` only calls `mockClear()`. Later tests in the same describe can inherit this gateway behavior, so reset the mock or use one-shot implementations.
  - confidence: 0.9
