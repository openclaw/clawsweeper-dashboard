## Label Justifications

- `P1`: The issue reports an active agent workflow race where a user-approved exec follow-up can fail as `UNAVAILABLE` after subagent cleanup retires its runtime.
- `impact:session-state`: The failure is caused by delayed approval work outliving the session/runtime state it is bound to.
- `impact:message-loss`: The approved follow-up result can fail instead of reaching the resumed session or user-visible channel.
- `maturity:stable`: The issue affects Gateway runtime exec approval behavior, and Gateway runtime is scored M4 Stable in the checked-in maturity scorecard.

## Summary

Keep open: current main still has explicit subagent bundle-MCP retirement paths that can dispose a session runtime without draining or terminally cancelling pending exec approval follow-up work. The merged session-rebind fix is adjacent, but it only drops follow-ups after `/new` or `/reset`; it does not establish the approval/runtime-retirement invariant requested here.

## What This Changes

Report that an exec approval follow-up can resume a subagent session after its bundle-MCP runtime has been retired, producing `UNAVAILABLE` and losing the approval result.

## Best Possible Solution

Define one approval/runtime lifecycle invariant and implement it in the core exec follow-up plus bundle-MCP retirement paths, with regression coverage for subagent finishes-before-approval, late approval after runtime retirement, duplicate approval resolution, and user-visible terminal outcomes.

## Reproduction Assessment

Yes at source level: current main still allows subagent cleanup to retire a bundle-MCP runtime while exec approval follow-up work can later resume against that session. I did not run a live timing reproduction in this read-only pass.

## Solution Assessment

Unclear as an implementation choice: the issue identifies the right invariant, but maintainers should decide whether late approvals should pin/drain the runtime or be terminally cancelled as runtime-retired before cleanup completes.
