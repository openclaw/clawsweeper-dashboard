## Summary

Keep open: current main and the latest release still contain the unguarded approval-follow-up versus bundle-MCP retirement path, and the related PRs/issues cover adjacent session-reset, fallback, or bookkeeping problems rather than this runtime-lifecycle invariant.

## What This Changes

The issue reports that an approved exec follow-up can run after a run-mode subagent's bundle-MCP runtime is disposed, causing a generic unavailable error and losing the approval result.

## Best Possible Solution

Add one maintainer-approved invariant: pending exec approvals either hold or reacquire a valid runtime generation until follow-up drains, or bundle-MCP retirement atomically moves those approvals to an explicit terminal user-visible state before they can be consumed.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a run-mode subagent can request exec approval, finish so cleanup retires its bundle-MCP runtime, and then the later approved follow-up resumes the same session path. I did not execute the live user flow in this read-only cleanup pass.

## Solution Assessment

No final implementation choice is confirmed yet: a retry or fallback-only fix would be too narrow. The maintainable fix should bind approval consumption to a live runtime generation/lease or terminalize the pending approval during runtime retirement.
