## Label Justifications

- `P1`: A delayed approval can be lost and surface as UNAVAILABLE in an active agent/subagent workflow, which is an urgent broken workflow for real users.
- `impact:message-loss`: The reported failure drops or fails the user-approved exec follow-up instead of delivering the result back to the session or channel.
- `impact:session-state`: The race is caused by stale asynchronous work targeting a subagent session after that session's bundle-MCP runtime has been retired.

## Summary

Keep open: current main still has the reported lifecycle gap between async exec approval follow-ups and run-mode subagent bundle-MCP retirement, and the related open items cover adjacent reset, transport-close, or follow-up UX paths rather than this specific pending-follow-up disposal race.

## What This Changes

The issue requests a guard so delayed exec approval follow-ups for subagents cannot run against a bundle-MCP runtime that subagent cleanup has already retired.

## Best Possible Solution

Add a central per-session lifecycle gate for exec approval follow-ups so run-mode subagent bundle-MCP retirement either waits for tracked follow-ups to drain or rejects/cancels them with an explicit recoverable user-facing outcome.

## Reproduction Assessment

Yes, source-reproducible: current main still resumes exec approval follow-ups by sessionKey while run-mode subagent cleanup can retire the same bundle-MCP runtime, and explicit runtime disposal does not drain active leases or pending follow-ups.

## Solution Assessment

Unclear: the issue's pin/drain/cancel directions are plausible, but the safest permanent fix needs maintainer agreement on lifecycle semantics rather than a local disposed-runtime catch.
