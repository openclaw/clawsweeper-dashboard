## Label Justifications

- `P1`: A real subagent approval workflow can lose an approved exec follow-up and surface a generic unavailable error to users.
- `impact:session-state`: The issue is about pending approval work outliving the subagent session/runtime state it targets.
- `impact:message-loss`: The approved exec result can fail before it is delivered or converted into a clear terminal message.

## Summary

Keep open: current main and the latest release still have the approval-follow-up path resuming through the session while run-mode subagent cleanup can retire that session's bundle-MCP runtime; related items cover adjacent rebind, fallback, or task-bookkeeping problems rather than this lifecycle invariant.

## What This Changes

The issue asks OpenClaw to prevent approved exec follow-up work from resuming against a subagent session whose bundle-MCP runtime has already been retired.

## Best Possible Solution

Add one maintainer-approved lifecycle invariant: pending exec approvals either hold or reacquire a valid runtime generation until follow-up drains, or runtime retirement atomically moves those approvals to a clear terminal user-visible state.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a run-mode subagent can request exec approval, finish so cleanup retires its bundle-MCP runtime, and then a later approval dispatches the follow-up through that same session path; I did not run the live user flow in this read-only review.

## Solution Assessment

No final implementation choice is confirmed yet. A retry or fallback-only fix would be too narrow; the maintainable fix should bind approval consumption to a live runtime generation/lease or terminalize the pending approval during runtime retirement.
