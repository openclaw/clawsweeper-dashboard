## Summary

Keep open: current main and the latest release still allow an exec approval follow-up to target a session whose run-mode subagent cleanup can retire the same bundle-MCP runtime without draining or terminalizing the pending approval. The related open PR at https://github.com/openclaw/openclaw/pull/64316 is adjacent reset-time cleanup, not a canonical fix for this approval-lifecycle race.

## What This Changes

This issue requests a fix for an exec approval follow-up race where a run-mode subagent’s bundle-MCP runtime can be retired before the user resolves the approval.

## Best Possible Solution

Add one approval-lifecycle invariant: pending exec approvals either retain or reacquire a valid runtime lease until the follow-up drains, or cleanup atomically moves them to a terminal user-visible state before bundle-MCP disposal can make them consumable.

## Reproduction Assessment

Yes at source level: spawn a run-mode subagent, trigger an exec approval, let the subagent finish so cleanup retires its bundle-MCP runtime, then approve the request. I did not execute the live scenario, but current source and tests still contain the unguarded lifetime path and disposed-runtime rejection shape.

## Solution Assessment

Unclear as a final implementation: the desired invariant is right, but maintainers should choose whether pending approvals block retirement, hold a runtime generation/lease, or are terminalized on retirement. A retry/fallback-only fix would be too narrow.
