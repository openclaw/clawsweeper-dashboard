## Label Justifications

- `impact:session-state`: The bug is caused by pending follow-up work outliving the session runtime state it targets.

## Summary

Keep open: current main still has the central approval-lifecycle race, with explicit subagent runtime retirement able to invalidate a later exec-approval follow-up and surface a generic `UNAVAILABLE` instead of a terminal approval outcome.

## What This Changes

The issue requests fixing an exec approval follow-up race where a subagent's bundle MCP runtime can be disposed before the user resolves the pending approval.

## Best Possible Solution

Add one lifecycle invariant for exec approval follow-ups: an approval either owns a valid runtime/session generation until it executes, or cleanup moves it to an explicit terminal user-visible state before the runtime is retired.

## Reproduction Assessment

Yes, at source level. Current code and tests show materialization can reject after runtime disposal, while the exec follow-up and subagent cleanup paths still compose that race without a live-or-terminal approval guard.

## Solution Assessment

Unclear until maintainers choose the lifecycle invariant. The strongest direction is not a retry-only fallback; it should bind approval follow-ups to a valid runtime/session generation or terminalize them before cleanup completes.
