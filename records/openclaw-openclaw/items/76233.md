## Summary

Keep open. Current main still has an unguarded lifecycle edge between delayed exec approval follow-up delivery and run-mode subagent bundle-MCP cleanup: the follow-up resumes by session key after the approval resolves, while subagent completion explicitly retires that session’s bundle-MCP runtime, and explicit disposal is not blocked by active runtime leases.

## What This Changes

Reports a race where an exec approval follow-up for a completed subagent can hit a disposed bundle-MCP runtime and surface `UNAVAILABLE`.

## Best Possible Solution

Add a lifecycle guard so run-mode subagent cleanup cannot dispose the session bundle-MCP runtime while an exec approval follow-up for that session is pending or active, with clear cancellation/direct-delivery behavior for expired follow-ups.

## Reproduction Assessment

Yes. The source-level reproduction path is to start a run-mode subagent, trigger a gateway exec approval, let the subagent complete and clean up before the approval resolves, then approve so the follow-up resumes the child session with bundle-MCP enabled.

## Solution Assessment

Unclear until the lifecycle policy is chosen. A lease-aware or pending-follow-up-aware runtime retirement fix is the most maintainable direction; a fallback-only error-message change would reduce the `UNAVAILABLE` surface but still risk losing the approved continuation.
