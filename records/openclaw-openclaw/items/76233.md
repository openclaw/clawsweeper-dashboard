## Summary

Keep open. Current main still has the reported lifecycle race: delayed exec approval follow-up resumes the subagent session by session key while run-mode subagent cleanup can retire and dispose that same session's bundle-MCP runtime before the follow-up materializes tools.

## What This Changes

The issue requests lifecycle handling for delayed exec approval follow-ups so they cannot resume against a disposed run-mode subagent bundle-MCP runtime.

## Best Possible Solution

Add a lifecycle gate that tracks pending and active exec approval follow-ups by session key/session id, then either blocks bundle-MCP retirement until they drain or cancels/direct-delivers them before disposal, with a regression test for the subagent interleaving.

## Reproduction Assessment

Yes. The source-level reproduction path is to start a run-mode subagent, trigger an exec approval, let the subagent complete and retire its bundle-MCP runtime, then approve so the follow-up resumes that child session.

## Solution Assessment

Unclear because there is no patch yet and the policy choice matters. A pending-follow-up-aware runtime retirement gate is the most maintainable direction; only changing the fallback error would leave the approved continuation lossy.

- A fix needs an explicit lifecycle policy for approvals that resolve after subagent completion: pin the runtime, cancel the follow-up, or retry/direct-deliver with clear semantics.
