## Summary

Keep open. Current main still has the race shape described: exec approval follow-ups are scheduled later against a session key, while subagent cleanup can retire the same session's bundle-MCP runtime before the follow-up run starts.

## What This Changes

The issue requests a fix for delayed exec approval follow-ups failing with `UNAVAILABLE` when subagent cleanup has already disposed the target session's bundle-MCP runtime.

## Best Possible Solution

Add a source-level lifecycle guard so pending exec approval follow-ups either keep the subagent bundle-MCP runtime alive until resolution or are cancelled/downgraded with a clear user-visible result before runtime disposal.

## Reproduction Assessment

Yes. The issue gives a specific sequence, and current code confirms the needed ingredients: delayed approval follow-up by session key plus subagent cleanup that retires the same bundle-MCP runtime; this still needs a deterministic regression test.

## Solution Assessment

Unclear for a specific patch because no patch is proposed. The best direction is a narrow lifecycle fix in the exec follow-up and bundle-MCP/subagent cleanup paths, preferably using the existing runtime lease concept rather than broad session migration.

## Repair Work Prompt

Fix the exec approval follow-up race where a subagent can finish and retire its bundle-MCP runtime before a pending approval resolves, causing the later `exec-approval-followup` run to fail with `bundle-mcp runtime disposed for session ...` / `UNAVAILABLE`. Keep the change narrow to the exec approval follow-up, subagent cleanup, and bundle-MCP runtime lifecycle paths. Prefer pinning the session bundle-MCP runtime while an approval follow-up is pending, using or extending the existing runtime lease concept, so subagent cleanup/idle disposal cannot dispose the runtime before the follow-up resolves; if pinning is not viable, cancel/downgrade pending follow-ups at cleanup with a clear user-visible result and internal diagnostics. Add regression coverage for: pending approval created in a subagent session; subagent cleanup runs before approval resolution; approval follow-up does not hit a disposed bundle-MCP runtime or leak a generic `UNAVAILABLE`; existing direct fallback behavior remains intact. Include a `CHANGELOG.md` fix entry because this is user-facing. Do not change plugin APIs, provider defaults, unrelated task-registry maintenance, or the separate #76162/#59349/#72143/#67777 behaviors except as needed for shared test fixtures.
