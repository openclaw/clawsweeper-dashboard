## Label Justifications

- `P1`: A real subagent approval workflow can lose an approved exec follow-up and surface `UNAVAILABLE` to users.
- `impact:session-state`: The bug is a session/runtime lifecycle race where pending approval work can outlive the subagent runtime it targets.
- `impact:message-loss`: The approved exec result can fail before it is delivered or converted into a clear terminal message.

## Summary

Keep open: current main and v2026.6.8 still allow an approved exec follow-up to resume a subagent session after that session's bundle-MCP runtime has been explicitly retired; related work such as https://github.com/openclaw/openclaw/pull/64316 covers reset-time cleanup, not this approval-lifecycle race.

## What This Changes

The issue reports a race where a pending exec approval follow-up can outlive subagent bundle-MCP runtime cleanup and return `UNAVAILABLE`, effectively losing the approved command result.

## Best Possible Solution

Add one maintainer-approved invariant: pending exec approvals either hold or reacquire a valid runtime generation until follow-up drains, or bundle-MCP retirement atomically moves those approvals to an explicit terminal user-visible state before they can be consumed.

## Reproduction Assessment

Yes, at source level: spawn a run-mode subagent, trigger an exec approval, let the subagent finish so cleanup retires its bundle-MCP runtime, then approve the request. I did not execute the live scenario, but current main and v2026.6.8 still contain the unguarded lifetime path and disposed-runtime error shape.

## Solution Assessment

Unclear as an implementation choice: the desired invariant is correct, but maintainers still need to choose whether approvals block retirement, carry a runtime generation/lease, or terminalize on retirement. A retry/fallback-only fix would be too narrow.
