## Label Justifications

- `P1`: The issue reports a broken approval/channel workflow where a user's approved exec follow-up can be lost and returned as `UNAVAILABLE`.
- `impact:session-state`: The failure is caused by session-scoped runtime lifecycle state being retired while delayed work still targets that session.
- `impact:message-loss`: The approved follow-up result can fail before reaching the user, effectively losing the approval outcome.

## Summary

Keep open: current main still has the approval-followup/runtime-retirement race shape. The source now has adjacent hardening for session rebinding and idle leases, but explicit subagent/embedded-run runtime retirement can still dispose a runtime while a follow-up is materializing it, and a current test even locks in rejection when catalog materialization is disposed in flight.

## What This Changes

The issue requests fixing an exec approval follow-up race where subagent bundle-MCP runtime cleanup can invalidate the execution context and surface `UNAVAILABLE`.

## Best Possible Solution

Define one approval-lifecycle invariant in the agent runtime: pending exec approvals for a subagent session must either keep a valid runtime/generation until execution drains or become an explicit terminal result before bundle-MCP retirement completes.

## Reproduction Assessment

Yes, at source level. Current code and tests show that disposing a session runtime during catalog materialization rejects with the disposed-runtime error, while subagent run cleanup still retires the same runtime path.

## Solution Assessment

Unclear. The issue correctly identifies the race, but the best fix is not merely a disposed-runtime check; maintainers should choose the durable approval/runtime lifecycle invariant before implementation.
