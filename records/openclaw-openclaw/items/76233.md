## Label Justifications

- `P1`: The issue affects real agent/subagent approval workflows and can turn a user-approved command follow-up into a generic unavailable failure.
- `impact:session-state`: Pending approval work can outlive and target session runtime state that has already been retired.
- `impact:message-loss`: The approved follow-up result can be lost or replaced by an internal `UNAVAILABLE` error instead of reaching the session/user.

## Summary

Keep open: current main and v2026.6.10 still allow subagent run cleanup to retire the session bundle-MCP runtime without draining or terminally cancelling pending exec approval follow-ups, so the reported disposed-runtime path remains actionable.

## What This Changes

The issue reports a race where a late exec approval follow-up resumes against a disposed subagent bundle-MCP runtime and surfaces `UNAVAILABLE`, losing the approved follow-up.

## Best Possible Solution

Add an explicit exec-approval lifecycle invariant so pending follow-ups either hold a valid runtime/session generation until execution or are terminally cancelled before bundle-MCP retirement completes, with regression tests for late, duplicate, and retired-runtime approvals.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a delayed exec approval follow-up can target the same session key while subagent cleanup explicitly retires that session's bundle-MCP runtime. I did not run a live end-to-end repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the lifecycle invariant. The issue's pin-or-cancel directions are the right shape, but current main only handles session rebind and idle-sweep leases, not pending approval ownership during explicit retirement.
