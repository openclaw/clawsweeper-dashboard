## Label Justifications

- `P1`: The issue affects active agent/subagent approval workflows and can turn a human-approved command follow-up into a generic unavailable failure.
- `impact:session-state`: The failure is caused by pending follow-up work outliving the session runtime state it targets.
- `impact:message-loss`: The approved follow-up result can be lost or reported as an internal unavailable error instead of reaching the user/session.

## Summary

Keep open: current main and the latest release still have explicit bundle-MCP retirement paths that can dispose a subagent session runtime without draining or terminally cancelling pending exec approval follow-ups, and the related items cover adjacent follow-up or MCP lifecycle problems rather than this runtime-retirement invariant.

## What This Changes

The issue reports an exec approval follow-up race where a delayed approval can resume a subagent after its bundle-MCP runtime has already been retired, producing `UNAVAILABLE` and losing the approved work.

## Best Possible Solution

Make exec approval follow-ups own a valid runtime/session lifecycle record until they execute, or transition them to an explicit terminal state before bundle-MCP cleanup completes, with regression coverage for late approvals, duplicate approvals, and subagent run-end cleanup.

## Reproduction Assessment

Yes for source-level reproduction: the current path schedules a delayed exec approval follow-up by session key while run-end cleanup can retire that same session runtime, and the tests already show explicit disposal rejects in-flight materialization. I did not run a live end-to-end repro in this read-only review.

## Solution Assessment

No current implementation is not sufficient: it guards session-key rebinding and idle-sweep leases, but not pending approval follow-ups against explicit runtime retirement. The best fix is a lifecycle invariant, not only a retry or prettier fallback error.
