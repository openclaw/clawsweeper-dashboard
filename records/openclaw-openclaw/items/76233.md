## Label Justifications

- `P1`: A real subagent approval workflow can lose an approved exec follow-up and surface an unavailable error to users.
- `impact:message-loss`: The approved exec result can fail before delivery or before conversion into a clear terminal message.
- `impact:session-state`: The issue is about pending approval work outliving the subagent session/runtime state it targets.

## Summary

Keep open: current main and the latest release still have the approval follow-up path resuming the same session without a runtime-retirement/generation guard, while run-mode subagent cleanup can still retire the bundle-MCP runtime before a delayed approval is consumed.

## What This Changes

The issue reports a race where delayed exec approval follow-up work can run after a subagent bundle-MCP runtime has been disposed, surfacing UNAVAILABLE and losing the approval result.

## Best Possible Solution

Adopt one explicit lifecycle invariant: pending exec approvals either hold or reacquire a valid runtime generation until follow-up drains, or runtime retirement atomically transitions them to a terminal user-visible state before they can be consumed.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a run-mode subagent can leave an exec approval pending, finish, and trigger bundle-MCP retirement before the later follow-up resumes the same session path. I did not run the live timing scenario in this read-only pass.

## Solution Assessment

No final implementation choice is confirmed yet: a retry or fallback-only fix would be too narrow. The maintainable fix should bind approval consumption to a live runtime generation/lease or terminalize the pending approval during runtime retirement.
