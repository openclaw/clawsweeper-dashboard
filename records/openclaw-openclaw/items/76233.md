## Label Justifications

- `P1`: The issue describes a user approval result being lost with a gateway `UNAVAILABLE` failure in an active agent/subagent workflow.
- `impact:message-loss`: The approved exec follow-up can fail after the user responds, effectively dropping the approval outcome.
- `impact:session-state`: The failure is tied to subagent session/runtime lifecycle state outliving or outpacing pending follow-up work.

## Summary

Keep open. Current main has adjacent session-rebind and follow-up hardening, but it still allows explicit subagent bundle-MCP retirement to dispose a runtime while an exec approval follow-up is materializing tools, so the central race is not implemented or stale.

## What This Changes

The issue requests an exec approval follow-up lifecycle fix so a user approval cannot target a disposed subagent bundle-MCP runtime and fail as generic `UNAVAILABLE`.

## Best Possible Solution

Add one approval-lifecycle invariant: every pending approval follow-up must either own a valid runtime lease through execution or be made terminal with a user-visible retired/cancelled outcome before subagent runtime cleanup completes.

## Reproduction Assessment

Yes, source-level: current main still permits explicit subagent bundle-MCP retirement while a follow-up agent turn can be materializing the same session runtime. I did not run a live gateway reproduction in this read-only pass.

## Solution Assessment

No, current main is not a complete solution; the session-rebind guard fixes a neighboring stale-session bug but not the disposed-runtime race. The best fix should bind approval follow-up state to runtime retirement rather than adding only generic retry or fallback text.
