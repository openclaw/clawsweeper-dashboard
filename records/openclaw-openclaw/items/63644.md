## Summary

Keep open: current main still has the iPhone Safari chat viewport gaps this PR targets, but the PR head has a stale textarea selector that does not match the active chat composer on current main.

## What This Changes

The PR changes Control UI CSS to lock root/app/chat heights to the dynamic viewport, hide root overflow, remove a chat-shell min-height, and add an iOS WebKit textarea font-size override.

## Best Possible Solution

Land one clean Control UI fix that keeps this PR's viewport/body/chat-shell improvements, retargets the iOS 16px input rule to the current composer and affected text controls, adds focused CSS assertions, and coordinates with the open input-zoom PR.

## Reproduction Assessment

Yes, source inspection plus the PR's iPhone Safari screenshots give a high-confidence path: current main still has the root viewport gap, chat-shell min-height conflict, and sub-16px active composer textarea. I did not run a live iPhone Safari reproduction in this read-only review.

## Solution Assessment

No, not as currently patched. The viewport direction is useful, but the iOS input-focus fix must target `.agent-chat__composer-combobox > textarea` and any other intended text-entry controls rather than the stale direct-child selector.
