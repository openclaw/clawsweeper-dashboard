## Summary

Keep open. Current main still lacks a first-class global setup/onboarding hook for credentialed tool-only plugins: manifest setup metadata is provider/CLI-backend shaped, setup registry resolution only collects those setup surfaces, the public plugin API has no tool-plugin setup registration hook, and the generic plugin-config wizard deliberately skips sensitive fields. The linked #61551 is a closed runtime hook-runner bug and does not supersede this public SDK/setup API gap.

## What This Changes

The issue requests a backward-compatible global `openclaw setup` contract for credentialed tool-only plugins, including manifest descriptors, setup-time registration, and plugin-owned credential persistence helpers.

## Best Possible Solution

Define an additive, manifest-first tool-plugin setup contract that cheaply declares credentialed tool setup needs, registers setup-time handlers without broad runtime loading, reuses SecretInput-safe prompting, persists under plugin-owned config, and updates SDK docs plus contract tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/API gap. The absence is high-confidence from source and docs: current main has no tool-plugin setup descriptor or registration hook, and the existing wizard skips sensitive plugin config fields.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current outcome. The requested additive SDK/setup seam is directionally aligned with OpenClaw’s plugin model, but credential storage and wizard UX should be settled before queuing an automated fix PR.
