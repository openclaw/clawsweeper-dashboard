## Summary

Keep open. Current main still lacks a first-class global setup/onboarding hook for credentialed tool-only plugins: the manifest and setup registry are provider/CLI-backend shaped, the public plugin API has no tool-plugin setup registration method, and the generic plugin config wizard deliberately skips sensitive fields.

## What This Changes

The issue requests an additive manifest/setup SDK hook that lets credentialed tool-only plugins participate in global `openclaw setup` and persist plugin-owned credentials/config.

## Best Possible Solution

Define a backward-compatible, manifest-first tool-plugin setup contract that cheaply declares credentialed tool setup needs, registers setup-time handlers without broad runtime loading, reuses SecretInput-safe prompting, persists under plugin-owned config, and updates SDK docs plus contract tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/API gap. The absence is high-confidence from current-main source review: there is no tool-plugin setup descriptor or registration hook, and the existing generic wizard excludes the sensitive API-key path described by the issue.

## Solution Assessment

Keeping this open for maintainer design is the best current outcome. The requested additive SDK/setup seam is directionally aligned with the projectâ€™s plugin API model, but the final credential storage and wizard UX should be decided explicitly before a repair PR is queued.
