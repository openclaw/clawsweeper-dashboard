## Summary

Keep open. Current main still has explicit provider request proxy support, but it does not emit the requested startup warning when proxy environment variables are present and configured remote model providers lack an explicit proxy override; the directly related implementation PR is closed and unmerged.

## What This Changes

Add a warning-only gateway startup diagnostic for env proxy variables plus remote model providers without `request.proxy.mode = "env-proxy"`.

## Best Possible Solution

Implement a one-time startup warning that reuses existing proxy-env helpers, inspects configured remote HTTP(S) model providers without explicit proxy config, and points users to `models.providers.<provider>.request.proxy.mode = "env-proxy"`.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to set `HTTPS_PROXY`, configure a remote `models.providers.*.baseUrl` without `request.proxy`, and call the current startup logger; the inspected source has no matching warning path or test coverage.

## Solution Assessment

Yes. A startup diagnostic is the narrowest maintainable fix because proxy routing already exists and should remain explicit; auto-enabling env proxy would change network semantics the report explicitly rejects.
