## Summary

Keep open. Current main still does not implement the requested startup warning: the gateway startup logger has no proxy/provider diagnostic branch, while model provider env-proxy routing remains explicit and the prior implementation candidate is closed unmerged.

## What This Changes

Add a non-breaking gateway startup warning when proxy environment variables are present but configured remote model providers do not explicitly use `models.providers.<provider>.request.proxy.mode = "env-proxy"`.

## Best Possible Solution

Implement a warning-only gateway startup diagnostic that reuses the existing proxy-env helpers, inspects configured model providers for remote HTTP(S) targets without explicit request proxy configuration, emits one actionable remediation message, and covers warning plus suppression cases in focused startup-log tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to set `HTTPS_PROXY`, configure a remote `models.providers.openai.baseUrl` without `request.proxy`, and call `logGatewayStartup`; the current source has no branch that can emit the requested warning.

## Solution Assessment

Yes. A startup warning is the narrowest maintainable solution because the env-proxy capability already exists and provider request routing should remain explicit; changing dispatcher defaults would be broader and riskier.
