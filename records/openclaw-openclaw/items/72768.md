## Summary

Keep open. Current main does not implement the requested event, and the PR still has blocking correctness and security issues around comparing the wrong config snapshots, shallow redaction of emitted values, and incomplete protocol schema exposure.

## What This Changes

The PR branch adds a Gateway config diff helper, ADMIN-scoped `config.changed` broadcast wiring from runtime config write listeners, event registration, a protocol payload schema, and close-handler cleanup.

## Best Possible Solution

Revise the feature so the event is emitted from the hot-reload comparison path, or from an extended notification that carries an explicit previous snapshot. The payload should be deeply redacted at every emitted boundary, registered in the public protocol schema surface, covered by focused tests for watcher and in-process writes, documented as a common event if public, and recorded in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config write notification has only post-write source/runtime projections, so this hook cannot compute a previous-to-next diff; a nested secret under a nonsensitive parent can also reach emitted `before`/`after` values at the mixed-type or depth-cap branch.

## Solution Assessment

No. The ADMIN-scoped event direction is plausible, but the implementation hooks too low in the write-notification path and does not fully redact arbitrary diff values; the safer design is to emit from the reload comparison state or extend the notification contract deliberately.
