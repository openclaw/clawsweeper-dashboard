## Summary

Keep open. Current main still lacks this event, but the PR has blocking correctness and security issues in how it derives and broadcasts the diff, so it is not ready for merge.

## What This Changes

The PR adds an admin-scoped `config.changed` Gateway event with a structured config diff payload, broadcast subscription wiring, cleanup state, protocol payload schema, and feature-list registration for config hot-reload notifications.

## Best Possible Solution

Revise the PR so the Gateway retains or receives a true previous compare snapshot for config writes, emits a deeply redacted previous-to-next diff, wires the payload schema through the public protocol registry/docs/generated-client path, and covers the event with targeted gateway/config/protocol tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a config write whose post-write source and runtime snapshots are equal makes `diffConfigs(evt.sourceConfig, evt.runtimeConfig)` return no changes, so the advertised event is not emitted for the actual edit.

## Solution Assessment

No. The listener and broadcast seam are reasonable, but this implementation needs previous-state data, deep value-wide redaction, and protocol/changelog wiring before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build the diff from the previous config:** `src/gateway/server-runtime-subscriptions.ts:101`
  - body: `sourceConfig` and `runtimeConfig` are both post-write snapshots, so a normal edit where runtime matches source produces an empty diff and no `config.changed` event. Capture the previous compare snapshot or extend the notification contract to carry before/after state.
  - confidence: 0.93
- **[P1] Deep-redact emitted values before broadcasting:** `src/gateway/config-diff.ts:68-77`
  - body: `redactIfSensitive` only checks the immediate key before recursion. If a nonsensitive parent changes type, is added/removed as a whole object, or hits the depth cap, nested fields like `apiKey` or `token` can be sent unredacted in `before`/`after`.
  - confidence: 0.91
- **[P2] Export and register the protocol schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: The new payload schema is added as a standalone module but is not exported from the schema barrel or registered in `ProtocolSchemas`, leaving schema consumers and generated clients unable to discover the new event contract.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/gateway/events.ts:4`
  - body: This is a user-facing Gateway protocol feature, but the PR does not update `CHANGELOG.md`. Add an Unreleased change entry with contributor credit before merge.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete sensitive-data exposure risk in the new admin-scoped config-change broadcast payload.

Concerns:

- **[medium] Nested secrets can bypass redaction:** `src/gateway/config-diff.ts:68`
  - body: When a nonsensitive parent object containing secret-like nested keys changes type, is emitted at the depth cap, or is added/removed as a whole object, `before`/`after` can contain unredacted nested credentials and broadcast them to admin-scoped clients.
  - confidence: 0.9
