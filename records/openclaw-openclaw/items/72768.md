## Summary

Keep open. Current main still lacks the requested `config.changed` Gateway event, and this PR is useful feature work, but the proposed patch has blocking correctness, security, and protocol-contract gaps before it can merge.

## What This Changes

The PR adds an admin-scoped `config.changed` Gateway event, a structured config diff helper, broadcast subscription cleanup, feature-list registration, and a new protocol payload schema for config hot-reload notifications.

## Best Possible Solution

Revise the PR so `config.changed` is emitted from a true previous-to-next config comparison, deep-redacts every emitted value, registers and exports the protocol schema, updates user-facing docs/changelog, and adds focused Gateway/config tests while preserving the ADMIN event scope.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config write notification contains only post-write `sourceConfig` and `runtimeConfig`, and a nonsensitive parent object containing nested `apiKey` or `token` fields can bypass the PR's immediate-key redaction when emitted as a whole value.

## Solution Assessment

No. The listener and admin broadcast seam are reasonable, but the PR needs previous-state data, value-wide deep redaction, and protocol export wiring before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Compare against the previous config snapshot:** `src/gateway/server-runtime-subscriptions.ts:101`
  - body: `evt.sourceConfig` and `evt.runtimeConfig` are both post-write snapshots. For a normal edit this can report no diff, or source-vs-runtime projection noise, instead of the user's actual before/after config change. Capture or pass the previous compare snapshot before broadcasting.
  - confidence: 0.93
- **[P1] Deep-redact emitted diff values:** `src/gateway/config-diff.ts:76`
  - body: The helper only redacts by immediate key before recursion, then emits whole `before`/`after` values for mixed-type and depth-cap changes. A nonsensitive parent containing nested credentials can be broadcast unredacted to admin clients.
  - confidence: 0.9
- **[P2] Export the new protocol schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: The PR adds a public event payload schema but does not export it from the schema barrel or register it in `ProtocolSchemas`, leaving protocol consumers and generated clients unable to discover the new contract.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/gateway/events.ts:4`
  - body: This is a user-facing Gateway protocol feature, but the PR file list does not include `CHANGELOG.md`. Repo policy requires `feat` changes to carry a changelog entry with appropriate attribution.
  - confidence: 0.84
