Adds an ADMIN-scoped Gateway `config.changed` event with structured config diffs, event discovery/scope wiring, a protocol schema file, and subscription cleanup.

## Best Possible Solution

Emit the event from the real previous/next reload boundary, recursively redact every emitted value, register the protocol schema/types, add focused tests, and require redacted real Gateway proof.

## Solution Assessment

No. The feature direction is plausible, but this implementation should move to the real hot-reload previous/next boundary, deep-redact payloads, and wire the schema into the protocol contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Compare against the previous config snapshot:** `src/gateway/server-runtime-subscriptions.ts:104`
  - body: This listener only observes runtime config write notifications, while ordinary file-watcher reloads use the reloader's `currentCompareConfig` to `nextCompareConfig` path. It also compares source-vs-runtime projections, so the event can report resolver/default noise instead of a true before/after config diff.
  - confidence: 0.93
- **[P1] Deep-redact every emitted diff value:** `src/gateway/config-diff.ts:52`
  - body: `redactIfSensitive` checks only the immediate key before emitting whole values. Added, removed, type-changed, or depth-capped objects under nonsensitive parents can still carry nested API keys, tokens, passwords, or credentials into the ADMIN broadcast.
  - confidence: 0.91
- **[P2] Register the config changed schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: Adding this schema file does not make the payload part of the protocol contract. Without exporting it from `schema.ts` and adding it to `ProtocolSchemas`, codegen and validator consumers cannot discover or validate the new event payload.
  - confidence: 0.86
