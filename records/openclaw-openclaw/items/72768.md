## Summary

Keep open: current main still lacks the new event, and the PR has blocking correctness and security issues around previous-state semantics, shallow redaction, and protocol registration.

## What This Changes

Adds a Gateway `config.changed` event, config-diff payload/schema, ADMIN-scoped broadcast wiring from runtime config write listeners, and shutdown cleanup for the subscription.

## Best Possible Solution

Emit `config.changed` from the reload comparison path or pass a real previous snapshot, deep-redact all emitted values, register/export the protocol schema, document/changelog the event, and add focused regression coverage before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: normal writes can compare two post-write projections instead of previous-to-next config, and object-valued diff entries can expose nested credentials because emitted values are not deep-redacted.

## Solution Assessment

No. The event direction is plausible, but the runtime write-listener hook is not the narrowest maintainable place for hot-reload before/after semantics; the existing reload comparison path already owns the needed state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Compare against the previous reload snapshot:** `src/gateway/server-runtime-subscriptions.ts:104`
  - body: `evt.sourceConfig` and `evt.runtimeConfig` are both post-write projections, and file-watcher hot reloads do not flow through this listener. The advertised before/after diff can be empty or source-vs-runtime noise; emit from the reload path's previous/next state or pass an explicit previous snapshot.
  - confidence: 0.93
- **[P1] Deep-redact emitted diff values:** `src/gateway/config-diff.ts:76`
  - body: The helper redacts only by the immediate key before emitting whole `before`/`after` values for mixed-type and depth-capped changes. A nonsensitive parent object containing nested credentials can still be broadcast to ADMIN clients, so every emitted value needs deep redaction.
  - confidence: 0.91
- **[P2] Register the config diff schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: This adds a public event payload schema, but the new schema is not exported from the schema barrel or registered in `ProtocolSchemas`. Protocol and codegen consumers that rely on the exported schema map cannot discover or validate the new contract.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/gateway/server-methods-list.ts:183`
  - body: Adding a Gateway event is a user-facing feature, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to include a changelog entry before merge.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete sensitive-data exposure risk in the new admin-scoped config-change broadcast payload.

Concerns:

- **[medium] Nested secrets can bypass redaction:** `src/gateway/config-diff.ts:76`
  - body: When a nonsensitive parent object containing secret-like nested keys is added, removed, changes type, or hits the depth cap, `before` and `after` can contain unredacted credentials and broadcast them to ADMIN-scoped clients.
  - confidence: 0.9
