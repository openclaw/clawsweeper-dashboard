## Summary

Keep open. Current main still lacks the `config.changed` Gateway event, and the PR head is not merge-safe because it builds the advertised diff from post-write source/runtime projections instead of previous-to-next config state and still has a nested-secret exposure path in emitted diff values. The author addressed the depth off-by-one comment, but the deeper redaction, previous-state, and protocol registration issues remain.

## What This Changes

The PR branch adds a Gateway config diff helper, a `config.changed` event payload, ADMIN-scoped broadcast wiring from runtime config write listeners, close-handler cleanup, and feature-list registration for config hot-reload notifications.

## Best Possible Solution

Revise the PR so `config.changed` is emitted from the config reload comparison path, or from an extended notification that carries a true previous snapshot, and ensure every emitted value is deeply redacted before broadcast. The protocol schema should be exported/registered for generated consumers, focused tests should cover in-process and watcher-driven reloads, and the user-facing Gateway event should get a changelog entry if it remains part of the public operator contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config write notification has only post-write `sourceConfig` and `runtimeConfig`, so the PR cannot produce a true before/after diff; a nested secret under a nonsensitive parent can also reach the emitted `before`/`after` values at the mixed-type or depth-cap branch.

## Solution Assessment

No. The ADMIN-scoped broadcast direction is plausible, but this implementation hooks too low-level a write event and does not fully redact arbitrary diff values; the narrower maintainable solution is to emit from the reload path that already owns previous-to-next compare state and reload/no-op decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Compare against the previous config snapshot:** `src/gateway/server-runtime-subscriptions.ts:104`
  - body: `evt.sourceConfig` and `evt.runtimeConfig` are both post-write projections, so this can suppress ordinary config edits or report source-vs-runtime noise instead of the advertised before/after config change. Emit from the reload path's `currentCompareConfig` -> `nextCompareConfig` state, or pass an explicit previous snapshot through the notification.
  - confidence: 0.93
- **[P1] Deep-redact emitted diff values:** `src/gateway/config-diff.ts:76`
  - body: The helper redacts only by immediate key before recursion, then emits whole `before`/`after` values for mixed-type and depth-cap changes. A nonsensitive parent object containing nested credentials can still be broadcast unredacted to admin clients.
  - confidence: 0.9

  - body: This adds a public event payload schema, but it is not exported from the schema barrel or registered in `ProtocolSchemas`, so protocol schema export/codegen consumers cannot discover the new contract.
  - confidence: 0.86
