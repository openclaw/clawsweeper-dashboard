## Summary

Keep this PR open. The requested custom `session.mainKey` behavior is still not implemented on current `main`, and the PR is a useful implementation candidate, but it is stale against the current routing/channel layout and still misses delivery-continuity handling for custom main keys.

## What This Changes

The PR preserves configured `session.mainKey`, threads it through selected route/session-key builders for config, routing, outbound, Telegram, and WhatsApp paths, and adds focused config/routing tests.

## Best Possible Solution

Rebase or port the feature onto current `main`, preserve configured `session.mainKey` through runtime materialization, thread the normalized key through the central SDK route builders and current outbound/Telegram/WhatsApp paths, update main-session alias and delivery-continuity guards for custom keys plus legacy `main`, align docs/generated config help, and add regression tests across config preservation, route resolution, outbound/channel routing, and delivery continuity.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a config with `session.mainKey: "telegram:direct:123456"`: current `materializeRuntimeConfig()` rewrites it to `"main"`, and the central route builders still hard-code `DEFAULT_MAIN_KEY`.

## Solution Assessment

No, not as currently proposed. The direction is right, but the PR needs to be rebased/ported to current paths and extended to delivery guards and docs before it is the narrow maintainable fix.
