## Summary

Keep open: current main still ignores custom `session.mainKey`, and this PR remains the canonical implementation candidate, but it is not mergeable as-is because it is stale against current route-builder paths, misses delivery-guard coverage, and lacks real behavior proof.

## What This Changes

The PR preserves configured `session.mainKey`, threads it through selected routing/outbound/Telegram/WhatsApp route builders, and adds config/routing tests for custom main-session keys.

## Best Possible Solution

Port the feature onto current main by preserving normalized `session.mainKey`, teaching all main-session builders/guards/plugin route paths to use the configured alias, aligning user-facing docs, and proving real Telegram/WhatsApp/system-event routing.

## Reproduction Assessment

Yes. Source inspection on current main shows `applySessionDefaults()` rewrites custom `session.mainKey` to `main`, while route builders and delivery guards still hard-code or recognize only the default key.

## Solution Assessment

No. The direction is maintainable, but this PR is not the best fix as-is because it needs a current-main port, delivery-guard changes, docs/test alignment, and real behavior proof.
