## Summary

Keep open. Current main still does not honor configured `session.mainKey`, and related #41790 was closed as superseded by this PR rather than as implemented. The PR has real product value, but it is stale and incomplete against current main: it misses moved route-builder paths and leaves main-session delivery guards literal-`main` only, which creates a privacy-sensitive routing risk if merged as-is.

## What This Changes

The PR preserves configured `session.mainKey`, threads it through selected config/routing/outbound/Telegram/WhatsApp session-key builders, and adds focused config and route-resolution tests.

## Best Possible Solution

Port the feature onto current main by preserving configured `session.mainKey` during runtime config materialization, threading the normalized key through central SDK route builders and current outbound/Telegram/WhatsApp paths, making main-session alias and delivery-continuity guards custom-key aware while preserving legacy `main`, and updating docs, generated config help, changelog, and regression tests across routing and delivery.

## Reproduction Assessment

Yes. On current main, a config with `session.mainKey: "telegram:direct:123456"` is rewritten to `"main"` by `materializeRuntimeConfig()` via `applySessionDefaults()`, and route resolution still hard-codes `DEFAULT_MAIN_KEY`; the branch-level delivery risk is reproducible from a custom `agent:*:<custom>` main key plus existing external `lastChannel`/`lastTo` state.

## Solution Assessment

No, not as currently proposed. The direction is the right narrow product fix, but the branch must be rebased or replaced with a current-main port that updates moved builders and delivery guards before it is safe to merge.
