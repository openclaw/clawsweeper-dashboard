## Summary

Keep open. The report matches a current-main/source path where `text_end` can mark a block reply as already handled before Telegram has actually sent anything, then `message_end` suppresses the safety send; the related #66459 is useful context but does not cover this supergroup-topic/off-mode regression tightly enough to close this issue as a duplicate.

## What This Changes

Fix Telegram supergroup forum-topic replies so a completed agent turn always produces an actual Telegram outbound delivery when user-visible text exists.

## Best Possible Solution

Fix delivery accounting so `message_end` is suppressed only after a block/text-end reply actually reaches Telegram or a preview is visibly finalized, with regression coverage for forum topics in streaming `off` and `partial` modes.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. The reporter supplied concrete steps and logs, and source inspection shows a plausible current-main path that can suppress the final send before Telegram transport succeeds.

## Solution Assessment

Yes, the best fix is narrow: repair the delivery-state contract between Pi `text_end` block replies, Telegram block delivery, and `message_end` fallback rather than changing Telegram topic routing or adding a config switch.
