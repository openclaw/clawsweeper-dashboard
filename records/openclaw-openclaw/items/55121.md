## Summary

Keep this PR open. Current main and the latest release still omit the `file` alias in the TypeScript tool-display resolver and embedded read-start diagnostic, so the branch is not obsolete; the diff is small and I found no blocking correctness or security issue, but the alias-display contract should get maintainer sign-off because execution validation remains canonical `path` only and native/mobile display helpers are still `path`-only.

## What This Changes

The PR adds `file` as a TypeScript read/write/edit display path candidate, extends the embedded read-start diagnostic to recognize `filePath` and `file`, and adds a focused read-summary assertion for `{ file: ... }`.

## Best Possible Solution

Land the PR as a scoped TypeScript display/logging compatibility fix if maintainers want verbose summaries to show intended file paths from alias-shaped model args, then either add native/mobile display coverage before closing the linked issue or explicitly narrow the issue to the server/web verbose path.

## Reproduction Assessment

Yes. On current main, the static unit path is high-confidence: `resolvePathArg()` ignores `record.file`, so `formatToolDetail(resolveToolDisplay({ name: "read", args: { file: "/tmp/a.txt" } }))` cannot produce the requested path detail; the PR adds the missing assertion.

## Solution Assessment

Unclear but likely acceptable for the TypeScript surface. Adding `record.file` to the shared resolver is the narrowest display-layer fix, while the embedded diagnostic and native/mobile scope need maintainer agreement because execution remains canonical `path` only.
