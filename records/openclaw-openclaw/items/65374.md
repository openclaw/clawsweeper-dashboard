## Label Justifications

- `P1`: The issue describes a real multi-agent memory workflow where one agent's session history can be synthesized into another agent's dreaming output.
- `impact:session-state`: The reported bug can associate session snippets and dream narratives with the wrong agent identity.
- `impact:data-loss`: The derived dreaming corpus and recall artifacts can be silently polluted with another agent's memory material.

## Summary

Keep open: current main and v2026.6.8 still resolve all agents attached to a workspace and write their session snippets into one workspace-level dreaming corpus. The linked implementation candidate is still open, draft, and conflicting, so this issue should close only after a reviewed fix lands.

## What This Changes

The issue asks for memory-core dreaming to keep session ingestion, recall, and generated dream narratives isolated per agent instead of pooling sessions across multi-agent setups.

## Best Possible Solution

Land a bounded memory-core fix that gives shared-workspace dreaming an agent-level ingestion, recall, and promotion boundary, then close this issue from merged current-main proof.

## Reproduction Assessment

Yes. Source inspection on current main shows that a shared workspace resolves all attached agent IDs, ingests every agent's session files, and appends them to one workspace-level `session-corpus/<day>.txt`.

## Solution Assessment

Yes for the requested end state, but not yet for a specific patch. The maintainable fix is an agent-level boundary in memory-core dreaming ingestion, recall, and promotion; the open draft PR is the current candidate to review or replace.
