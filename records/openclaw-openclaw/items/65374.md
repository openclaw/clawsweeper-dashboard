## Label Justifications

- `P1`: Real multi-agent users report silent built-in dreaming memory contamination that can corrupt an agent workflow without an obvious failure signal.
- `impact:data-loss`: The shared corpus can pollute derived dream and memory artifacts by mixing another agent's session snippets into the querying agent's state.
- `impact:session-state`: The issue affects transcript-derived recall, dreaming corpus state, and first-person agent identity across agents sharing a workspace.

## Summary

Keep open: current main and the latest release still have a source-reproducible shared-workspace dreaming isolation gap, and the linked implementation path is an open draft PR that is not merge-ready.

## What This Changes

The issue requests agent-level isolation for memory-core dreaming so session corpus snippets and first-person dream synthesis cannot mix experiences across agents.

## Best Possible Solution

Land a focused memory-core fix that carries current agent identity through dreaming ingestion, recall, and promotion, isolates shared-workspace corpus data per agent, and provides repair or migration handling for existing polluted artifacts.

## Reproduction Assessment

Yes. Source inspection shows shared workspace agent IDs are returned together, the ingestion loop reads every returned agent's sessions, snippets are written into one workspace-level corpus file, and promotion ranking has no current-agent scope; no live filesystem repro was run in this read-only review.

## Solution Assessment

Yes on direction: agent-scoped dreaming isolation is the maintainable fix boundary, not a docs-only workaround. The current open implementation PR still needs maintainer review or a successor because it is draft, conflicting, and missing accepted real behavior proof.
