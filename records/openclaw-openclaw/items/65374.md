## Label Justifications

- `P1`: The issue describes a real multi-agent memory workflow where one agent’s session history can affect another agent’s dreaming output.
- `impact:session-state`: The bug can associate session snippets and dream narratives with the wrong agent identity.
- `impact:data-loss`: The derived dreaming corpus and recall artifacts can be silently polluted with another agent’s memory material.

## Summary

Keep open: current main and the latest release still have the shared-workspace dreaming path that can ingest multiple agents into one workspace-level session corpus. The linked fix PR is still open and draft, so this issue should close only after a reviewed fix lands.

## What This Changes

The issue reports that memory-core dreaming can pool session snippets across agents sharing one gateway/workspace, causing cross-agent dream narratives and identity/session contamination.

## Best Possible Solution

Land a bounded memory-core fix that gives shared-workspace dreaming an agent-level ingestion and recall boundary, then close this issue from the merged PR or current-main proof.

## Reproduction Assessment

Yes. A source reproduction path exists on current main: configure multiple agents with a shared workspace, enable memory-core dreaming, and the ingestion path resolves all workspace agent ids, reads each agent’s sessions, and writes one workspace-level day corpus.

## Solution Assessment

Yes for the requested end state, but not yet for a specific patch. The maintainable solution is an agent-level boundary in memory-core dreaming ingestion/recall, and the linked draft PR is the current candidate to review rather than a reason to close this issue now.
