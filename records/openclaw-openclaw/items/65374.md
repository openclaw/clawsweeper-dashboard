## Summary

Keep open. Current main still resolves dreaming work by workspace, then ingests every agent attached to that workspace into one session corpus and records recall under a workspace/day query without an agent-scoped filter. The related replay/self-ingestion fixes reduce other contamination paths, but they do not resolve the shared-workspace multi-agent isolation bug described here.

## What This Changes

The issue asks memory-core dreaming to prevent cross-agent session corpus pooling and first-person identity contamination in multi-agent gateway setups.

## Best Possible Solution

Keep this as the canonical shared-workspace dreaming isolation bug. Maintainers should decide whether workspace-level pooling is intentional; if not, thread an agent scope through dreaming workspace resolution, session corpus storage, __dreaming_sessions__ recall keys, ranking/promotion, and Dream Diary generation, then add a regression with two active agents sharing one workspace proving one agent cannot ingest or promote the other's transcript.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure two agents with the same workspace and session transcripts for both; current main resolves them into one dreaming workspace, iterates both agent IDs, appends both transcript streams into one corpus file, and records one unscoped recall query.

## Solution Assessment

Unclear until maintainers choose the product boundary. The current workspace-pooled implementation is not the best solution for the reported identity-isolation expectation, but a durable fix should either make dreaming agent-scoped or explicitly document workspace pooling with an opt-out/control.
