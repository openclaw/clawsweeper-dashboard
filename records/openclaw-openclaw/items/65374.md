## Label Justifications

- `P1`: Real multi-agent users report silent identity and memory contamination in a built-in dreaming workflow that remains present in the latest release.
- `impact:session-state`: The issue corrupts agent memory, transcript-derived recall, and first-person identity state across agents sharing a workspace.
- `impact:data-loss`: The shared corpus can pollute or corrupt derived memory data by mixing another agent's session snippets into the querying agent's dream artifacts.

## Summary

Keep open: current main and the latest release still group multiple agents that share one workspace into one dreaming corpus/ranking path, while the active fix is an open draft PR that is conflicting and still needs maintainer review.

## What This Changes

The issue requests agent-level isolation for memory-core dreaming so shared-workspace multi-agent sessions do not contaminate one shared corpus or first-person dream narrative.

## Best Possible Solution

Land a focused memory-core fix that carries the current agent identity through dreaming ingestion, recall, and promotion, isolates shared-workspace corpus data per agent, and proves upgrade behavior for existing corpus artifacts.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: shared-workspace agent IDs are returned together, then session snippets are written and ranked through one workspace corpus path. I did not run a filesystem repro because this review is read-only.

## Solution Assessment

Yes on direction: agent-scoped dreaming isolation is the maintainable fix boundary, not a docs-only workaround. The exact open PR still needs maintainer review because it is draft, conflicting, and touches upgrade-sensitive memory artifacts.
