## Label Justifications

- `P1`: The issue describes a real multi-agent memory workflow where one agent's session history can be synthesized into another agent's dreaming output.
- `impact:session-state`: The bug can associate session snippets and dream narratives with the wrong agent identity.
- `impact:data-loss`: The derived dreaming corpus, recall artifacts, and promotion candidates can be silently polluted with another agent's memory material.
- `impact:security`: Cross-agent corpus pooling can expose one agent's session content to another agent's memory and narrative path.

## Summary

Keep open: current main and the latest release still let a shared memory-core dreaming workspace resolve multiple agent IDs, ingest those agents' session transcript snippets, and write them into one workspace-level dreaming corpus before recall and promotion. The directly linked fix candidate is still open, draft, and conflicting, so this should close only after a reviewed fix lands.

## What This Changes

The issue reports that memory-core dreaming can pool session snippets from multiple agents and synthesize them as first-person memories for the wrong agent in multi-agent setups.

## Best Possible Solution

Land a bounded memory-core fix that preserves per-agent boundaries through dreaming ingestion, recall, and promotion for shared-workspace multi-agent runs, then close this from merged current-main proof.

## Reproduction Assessment

Yes. Source inspection on current main shows shared-workspace agent IDs are resolved together, every resolved agent's session files are ingested, and the results are written and recalled from a workspace-level corpus rather than an agent-scoped corpus.

## Solution Assessment

Yes for the desired end state, but not yet for a specific patch. The maintainable fix is to thread agent identity through memory-core dreaming ingestion, recall, and promotion; the open draft PR is a candidate to review, refresh, or replace.
