## Summary

Keep open. Current main and the latest release still pool shared-workspace agents into one dreaming corpus/recall namespace, and the open draft fix PR for this bug has not merged.

## What This Changes

The issue reports that memory-core dreaming can ingest snippets from multiple agents sharing one workspace and synthesize them as one agent's first-person memory.

## Best Possible Solution

Land or replace the linked fix with a consistent agent-boundary contract across workspace resolution, session corpus storage, dreaming recall keys, ranking/promotion, Dream Diary generation, and shared-workspace regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure two agents with the same workspace and separate session transcripts, then current main resolves both ids, reads both session stores, and records one unscoped dreaming recall query.

## Solution Assessment

No. The current workspace-pooled implementation is not the right solution for the documented multi-agent isolation contract; the linked agent-scoped dreaming PR or a narrower replacement should be finished first.
