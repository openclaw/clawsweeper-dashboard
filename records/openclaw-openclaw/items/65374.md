## Summary

Keep open. Current `main` still preserves the shared-workspace dreaming behavior that lets multiple agents feed one workspace-level session corpus, and the linked draft fix at https://github.com/openclaw/openclaw/pull/76140 remains open rather than merged.

## What This Changes

The issue reports that `memory-core` dreaming pools session snippets from multiple agents into one workspace-level corpus, causing first-person dream output to synthesize another agent's sessions as local memory.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/76140 with one consistent agent-scoped dreaming contract across workspace resolution, session corpus storage, recall keys, ranking/promotion, diary generation, CLI/UI surfaces, and shared-workspace regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure two agents with the same workspace and separate session transcripts; current `main` resolves both IDs, reads both session stores, writes one corpus file, and records `__dreaming_sessions__:<day>` without an active-agent discriminator.

## Solution Assessment

No. The current workspace-pooled implementation is not the narrow maintainable solution for the documented isolation contract; the right direction is to finish an agent-scoped dreaming fix and verify ranking and promotion paths too.
