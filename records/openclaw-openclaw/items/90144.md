## Label Justifications

- `P1`: The affected subagent completion path can leave parent sessions waiting on completed children, which is a broken agent workflow affecting real users.

## Summary

Keep open. The PR targets a real subagent completion-delivery failure, but the proposed fallback marks the completion delivered after only writing an in-memory queue entry, without also waking the yielded requester that must start the next turn to drain it; the external PR also still needs real after-fix behavior proof.

## What This Changes

The PR routes direct-pending subagent completion announcements into the requester system-event queue, adds a `durable_queue` delivery path, and updates harness/Codex durability tests.

## Best Possible Solution

Commit the durable queue fallback together with an explicit requester wake/heartbeat and proof that a yielded parent resumes, drains the queued completion, and clears the pending descendant state.

## Reproduction Assessment

Yes at source level: current main returns `completion_handoff_pending` for the non-terminal direct announce path after a non-streaming requester, and the PR adds deterministic tests for that path. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. Using the existing system-event queue is the right layer, but the delivery commit also needs to wake the requester; otherwise it can be marked durable without the turn that drains it.
