## Label Justifications

- `P1`: The affected subagent completion path can leave parent sessions waiting on completed children, which is a broken agent workflow affecting real users.

## Summary

Keep open: the PR still addresses a real direct-pending subagent completion gap that current main does not implement, but it is not merge-ready because it can mark the completion delivered after only queueing an in-memory event, without waking the requester to drain it, and the external PR still has mock/test-only proof.

## What This Changes

The PR routes non-terminal direct subagent completion handoffs into the requester's system-event inbox and treats the new `durable_queue` path as durable for harness/Codex retry logic.

## Best Possible Solution

Land the durable queue fallback only after it also wakes the requester immediately, proves the queued completion drains and clears pending descendant state, and includes redacted real runtime proof.

## Reproduction Assessment

Yes at source level: current main has a `completion_handoff_pending` branch for the non-terminal direct announce path after a non-streaming requester, and the PR adds deterministic tests around the yielded-parent shape. I did not run tests because this is a read-only review.

## Solution Assessment

No as submitted. The existing system-event queue is a reasonable layer, but the fix also needs an immediate requester wake/heartbeat and real after-fix proof before the delivery can safely be treated as durable.
