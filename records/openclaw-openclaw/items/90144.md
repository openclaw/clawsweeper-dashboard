Keep open: the fallback is still relevant, but as submitted it can mark a completion delivered before the idle requester is woken to drain it, and the external real behavior proof is still missing.

## What This Changes

The PR adds a `durable_queue` subagent announcement path that enqueues pending direct completion handoffs into system events and marks that path durable in the agent harness runtime, with focused tests.

## Best Possible Solution

Land the durable queue fallback with an immediate requester heartbeat or equivalent wake, coverage that an idle parent drains the queued completion and clears pending state, and redacted real behavior proof before merge.

No as submitted. The existing system-event queue is the right layer, but the delivery commit also needs a requester wake or equivalent guarantee before the durable consumer stops retrying.
