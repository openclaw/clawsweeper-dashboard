## Summary

Keep open. Current main still has the reported loop-detection gap, and open PR #60248 is a narrow unmerged implementation candidate rather than landed behavior.

## What This Changes

The issue asks for `genericRepeat`/`generic_repeat` loop detection to escalate to critical at `criticalThreshold`, include `process(action="list")` in known polling, and consider a separate tool-only-turn breaker.

## Best Possible Solution

Land or replace #60248 with a scoped fix that escalates `generic_repeat` to critical with enough no-progress evidence, updates regression tests/docs, and explicitly scopes any `process(action="list")` or tool-only-turn follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: record `CRITICAL_THRESHOLD` repeated non-polling same-argument calls, then `detectToolCallLoop` still returns `generic_repeat` with `level: "warning"`.

## Solution Assessment

Unclear. The open PR's no-progress guard is safer than blocking solely by repeated arguments, but maintainer review should decide the exact false-positive boundary and whether the `process(action="list")` classification is in scope.
