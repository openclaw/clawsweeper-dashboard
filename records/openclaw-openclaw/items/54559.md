## Label Justifications

- `impact:crash-loop`: The issue is about an agent/tool loop continuing indefinitely instead of being blocked at the configured critical threshold.

## Summary

Close as implemented: current main and the v2026.5.18 release now add a no-progress-gated `generic_repeat` critical path at `criticalThreshold`, and critical loop results are blocked by the before-tool-call hook.

## What This Changes

The issue requests that `tools.loopDetection` make `genericRepeat` escalate from warning to blocking at `criticalThreshold`, with related notes about `process(action="list")` and a broader tool-only-turn circuit breaker.

## Best Possible Solution

Keep the shipped no-progress-gated `generic_repeat` blocking behavior, and open separate targeted issues only if current v2026.5.18+ behavior still misses `process(action="list")`, varied-output loops, or tool-only-turn abort policy.

## Reproduction Assessment

No current-main failing reproduction remains. Source inspection shows the previous path now returns `critical` for non-poll generic no-progress repeats at `criticalThreshold`, and unit/e2e tests assert that behavior.

## Solution Assessment

Yes. The landed implementation blocks only after identical outcomes prove no progress, which is safer than blocking every repeated same-argument call; process-list special-casing and a broader tool-only-turn circuit breaker are separate policy choices.
