## Summary

Keep open. Current main still has the reported gap: `generic_repeat` is warning-only at `criticalThreshold`, warning loop detections do not block tool execution, and `process(action="list")` is still outside known polling handling. PR #60248 remains the narrow open implementation candidate rather than landed behavior.

## What This Changes

The issue asks OpenClaw to make `generic_repeat` escalate to critical/blocking at `criticalThreshold`, optionally classify `process(action="list")` as known polling, and consider a broader consecutive tool-only-turn circuit breaker.

## Best Possible Solution

Land or replace the scoped loop-detection fix so `generic_repeat` can become critical at `criticalThreshold` when repeated calls show no progress, update unit/e2e expectations and docs/config help, and handle `process(action="list")` plus any tool-only-turn breaker as explicit follow-up scope if they are not included.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: record `CRITICAL_THRESHOLD` repeated non-polling calls and `detectToolCallLoop` still returns `generic_repeat` with `level: "warning"`; the existing unit test encodes that behavior.

## Solution Assessment

Unclear as proposed. Adding a critical branch for `generic_repeat` is the right narrow direction, but the safer implementation should account for no-progress evidence or other false-positive controls rather than blocking every repeated same-args call solely by count.
