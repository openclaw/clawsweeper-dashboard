## Summary

Keep open. Current main still has the reported enabled-loop-detection gap: `generic_repeat` remains warning-only, warning detections do not block tool execution, and the latest shipped tag still contains that behavior. The direct unconfigured case is consistent with loop detection being disabled by default, but the enabled `criticalThreshold` behavior remains unresolved; PR #60248 is the open implementation candidate rather than merged behavior.

## What This Changes

The issue reports that `genericRepeat` warns at `warningThreshold` but never blocks at `criticalThreshold`, and asks for critical escalation plus related `process(action="list")` polling coverage and a possible tool-only-turn breaker.

## Best Possible Solution

Land or replace the scoped loop-detection fix so `generic_repeat` can become critical at `criticalThreshold` when repeated calls show no progress, then align tests, docs, and changelog. Treat `process(action="list")` classification and the tool-only-turn breaker as explicit follow-up decisions unless the final PR intentionally includes them.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: record `CRITICAL_THRESHOLD` repeated non-polling same-tool/same-args calls with identical outcomes, and `detectToolCallLoop` still returns `generic_repeat` with `level: "warning"`; the current unit test asserts that behavior.

## Solution Assessment

No, current main is not the best solved state. The open PR’s no-progress-gated critical path is a safer narrow direction than blocking purely by repeated arguments, but maintainers still need to review the exact scope and follow-ups.
