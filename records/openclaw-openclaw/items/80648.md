## Summary

Keep open: the report is fresh, concrete, and plausible, but current source also shows that Feishu group replies are intentionally suppressed when the active route is message-tool-only and the model does not call `message(action=send)`. The remaining question is whether `coach` is unexpectedly entering that route, missing a delivery fallback, or only exposing the documented group-visible-reply behavior while `assist-agent` differs through agent/tool policy or session state.

## What This Changes

Investigate why Feishu webhook visible replies are delivered for `assist-agent` but not for `coach` under reportedly identical channel and bot configuration.

## Best Possible Solution

Diagnose the two agents side by side on current main, recording effective visible-reply mode, message-tool availability, route session key, Feishu reply target, and send result; fix the routing/policy code only if those facts show `coach` diverges unexpectedly from `assist-agent`.

## Reproduction Assessment

No. We have a high-confidence source path for the visible symptom when a Feishu group turn is `message_tool_only` and the model emits only final text, but we do not yet have a current-main reproduction proving that `coach` and `assist-agent` diverge under identical effective policy.

## Solution Assessment

Unclear. Investigating the Feishu route/session/tool-policy path is the right direction, but a code fix is not yet justified until diagnostics distinguish expected message-tool-only behavior from a true agent-specific delivery regression.
