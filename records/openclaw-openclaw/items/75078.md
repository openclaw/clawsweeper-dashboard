## Summary

Keep this PR open. Current main still keys Feishu inbound receive, debounce, and final processing on the raw message id, so the reported audio drop is valid; the PR addresses that path with media-aware keys and regression coverage, but it still needs the required active-version changelog entry before merge.

## What This Changes

The PR changes Feishu inbound receive, debounce, suppressed-message, and final dedupe paths to include media file/image keys for media messages, and adds a regression test for two audio events sharing one message id with different file keys.

## Best Possible Solution

Land the Feishu-owned media-aware dedupe fix after adding one active-version changelog bullet, preserving raw message-id behavior for non-media events and true retries while allowing distinct media attachments with reused Feishu ids to reach the agent.

## Reproduction Assessment

Yes. Current main has a high-confidence trace reproduction: two Feishu audio events for the same account with the same message_id and different file_key values collide in both the early processing claim and final persistent dedupe paths.

## Solution Assessment

Yes, with one release-note cleanup. The implementation stays inside the Feishu plugin and uses existing media-key parsing; a shared helper could reduce future drift, but no core API or broader channel change is needed for this fix.
