## Summary

Keep open. Current main still dedupes Feishu inbound audio by bare `message_id`, and this PR targets the right Feishu-owned bug from #75057, but the branch is not mergeable because it imports a non-exported helper and still lacks the required user-facing changelog entry.

## What This Changes

The PR changes Feishu inbound receive, debounce, and final dedupe to include media `file_key`/`image_key` values in the dedupe key and adds an audio regression test.

## Best Possible Solution

Keep the fix inside the Feishu plugin, extract or export one shared Feishu-local dedupe-key helper used by receive/debounce/final paths, preserve bare `message_id` behavior for non-media and malformed media payloads, add the active-version changelog bullet, then run focused Feishu validation before merge.

## Reproduction Assessment

Yes. Current main has a high-confidence trace reproduction: two Feishu audio events for the same account with the same `message_id` and different `file_key` values collide in both the early processing claim and final persistent dedupe paths.

## Solution Assessment

No, not as patched. The implementation direction is appropriate, but the unexported `parseMediaKeys` import blocks the build; a shared exported helper or dedicated `dedupe-key.ts` module is the narrowest maintainable repair.
