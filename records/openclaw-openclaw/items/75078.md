## Summary

Keep open. The PR targets a real Feishu media dedupe bug and the media-aware key direction is appropriate, but the current patch still imports a non-exported helper and lacks the required user-facing changelog entry.

## What This Changes

The PR changes Feishu inbound and final dedupe to use `message_id` plus media file/image keys for media events and adds an audio reused-message-id regression test.

## Best Possible Solution

Centralize a Feishu-local media-aware dedupe-key helper and use it consistently across receive, debounce, suppression recording, persistent finalization, and regression coverage, with a single-line changelog entry.

## Reproduction Assessment

Yes. Current main has a high-confidence source trace: same account, same Feishu `message_id`, and different audio `file_key` values collide in both the early claim path and final persistent dedupe path.

## Solution Assessment

No, not as currently patched. The media-aware key is the right narrow fix, but the branch must first export or extract the media-key helper and add the required changelog entry before exact-head validation.
