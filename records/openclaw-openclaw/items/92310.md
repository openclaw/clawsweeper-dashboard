## Summary

Keep open: the Telegram ledger regression is real, but this PR is not merge-ready because it only records the first delivered message id and the automatic dispatch path still does not pass `cfg` into `deliverReplies`. A broader open fix at https://github.com/openclaw/openclaw/pull/92420 covers the same issue more completely, but it still needs live Telegram proof before becoming a safe canonical landing path.

## What This Changes

The PR imports `recordSentMessage`, writes one sent-message ledger entry after a successful Telegram `deliverReplies` send, and adds a single `wasSentByBot` regression test.

## Best Possible Solution

Record every delivered Telegram message id at a shared delivery success point, forward `cfg` from automatic dispatch, cover single-message, multi-message, fallback, and cfg-scope cases, then attach redacted live Telegram or equivalent ledger proof before merge.
