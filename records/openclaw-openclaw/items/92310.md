## Label Justifications

- `P1`: The linked regression affects a real Telegram channel workflow where successful delivery replies are not persisted for downstream reply/reaction checks.
- `merge-risk: 🚨 message-delivery`: A too-narrow ledger write can keep downstream Telegram reply or reaction routing wrong for multi-message deliveries.
- `merge-risk: 🚨 session-state`: The sent-message ledger is persisted runtime state used by `wasSentByBot`, so incomplete writes can leave session-derived routing state stale.

## Summary

Keep this PR open, but it is not merge-ready: current main and the latest release still lack the Telegram delivery ledger write, while this patch has a broken test harness, records only one message id for paths that can send multiple Telegram messages, and lacks real Telegram proof.

## What This Changes

The PR adds `recordSentMessage` to Telegram delivery replies and adds a regression test checking `wasSentByBot` after a successful delivery send.

## Best Possible Solution

Record finalized durable Telegram message ids with the correct config at each successful delivery send site, install an isolated sent-message test store in the regression test, and require redacted real Telegram ledger/readback proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current `deliverReplies` can send and emit `message_sent` without recording the sent-message ledger, and the proposed test harness would call runtime-backed cache clearing without a Telegram runtime or test store.

## Solution Assessment

No as submitted. Restoring ledger writes in Telegram delivery is the right owner boundary, but the patch should record each finalized send id and fix the test-store setup before it is the best solution.
