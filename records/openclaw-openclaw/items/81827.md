## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Evaluate node-host shell payloads before policy fallback:** `src/node-host/invoke-system-run.ts:407`
  - body: The deny check only receives `parsed.shellPayload`, but CI shows `/bin/sh -lc "cat ~/.openclaw/secrets/foo.env"` falling through as `allowlist-miss` instead of `deny-path-pattern`. Recompute the shell-wrapper payload for deny matching or use the shared resolver before allowlist policy runs.
  - confidence: 0.93
- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway deny gate parses command text with POSIX `splitShellArgs()` before matching. On Windows, unquoted paths such as `C:\Users\alice\.ssh\id_rsa` can be backslash-unescaped before the matcher sees them, so Windows deny globs can miss protected files.
  - confidence: 0.9
- **[P1] Scan cmd and PowerShell payloads in the gateway gate:** `src/agents/bash-tools.exec.ts:1472-1475`
  - body: The gateway gate extracts inner payloads with a local helper that only recognizes POSIX shells. Denied paths inside `cmd.exe /c` or `pwsh -Command` payloads can stay hidden in a single command string instead of being tokenized and denied.
  - confidence: 0.88
- **[P1] Match shell backslash semantics before denying paths:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` treats every backslash as a generic escape and preserves escaped newlines. That corrupts Windows paths inside shell payloads and misses POSIX backslash-newline continuations where the shell joins a denied path before execution.
  - confidence: 0.86
- **[P2] Ignore unquoted shell comments during tokenization:** `src/infra/exec-deny-path.ts:149-155`
  - body: The payload tokenizer never stops at an unquoted `#` that begins a shell comment, so a path that appears only in a shell comment can cause a false denial. Match the existing shell argv comment behavior to avoid blocking commands the shell would ignore.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The PR adds a useful hard-deny control, but the changed command-execution boundary still has concrete parser and wrapper-resolution bypasses.

Concerns:

- **[high] Node-host shell payloads can miss the deny gate:** `src/node-host/invoke-system-run.ts:407`
  - body: CI shows a configured deny pattern did not fire for `/bin/sh -lc "cat ~/.openclaw/secrets/foo.env"`; the command reached normal allowlist handling instead of the hard-deny path.
  - confidence: 0.93
- **[high] Windows paths can bypass gateway deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway gate feeds command text through a POSIX-style splitter before matching, which can transform unquoted Windows backslash paths before the deny matcher sees them.
  - confidence: 0.9
- **[high] cmd and PowerShell wrappers hide denied payloads:** `src/agents/bash-tools.exec.ts:1472`
  - body: The gateway pre-dispatch gate uses a POSIX-only payload extractor instead of the shared shell-wrapper resolver, leaving cmd and PowerShell payloads outside proper deny-token scanning.
  - confidence: 0.88
- **[high] Backslash-newline can hide POSIX denied paths:** `src/infra/exec-deny-path.ts:130`
  - body: The payload tokenizer preserves a backslash-newline as a literal newline, while POSIX shells join it before execution, so a denied path can be split across lines and miss the glob.
  - confidence: 0.86
