## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Evaluate node-host shell payloads before policy fallback:** `src/node-host/invoke-system-run.ts:407`
  - body: The deny check only passes `parsed.shellPayload`, which can be null for legacy `/bin/sh -lc` argv without `rawCommand`; CI shows the new test falling through to `allowlist-miss` instead of `deny-path-pattern`. Recompute the payload for deny matching or use the shared shell-wrapper resolver so approved node-host runs cannot execute an inner denied path.
  - confidence: 0.93
- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway deny gate parses command text with POSIX `splitShellArgs()` before matching. On Windows, unquoted paths such as `C:\Users\alice\.ssh\id_rsa` can be backslash-unescaped before the matcher sees them, so Windows deny globs may miss the protected path.
  - confidence: 0.9
- **[P1] Use shared wrapper parsing for cmd and PowerShell payloads:** `src/agents/bash-tools.exec.ts:1472-1475`
  - body: The gateway gate extracts inner payloads with a local helper that only recognizes POSIX shells, so `cmd.exe /c` and `pwsh -Command` payloads are not tokenized for denied paths. Reuse the existing shared shell-wrapper resolver so Windows wrapper payloads get the same deny scan.
  - confidence: 0.89
- **[P1] Match shell backslash semantics before deny matching:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` treats every backslash as a generic escape and preserves escaped newlines. That corrupts Windows paths inside shell payloads and misses POSIX backslash-newline continuations where the shell joins a denied path before execution.
  - confidence: 0.86
- **[P2] Keep shell comments out of deny tokenization:** `src/infra/exec-deny-path.ts:142-156`
  - body: The payload tokenizer splits whitespace and operators but never stops at an unquoted `#` that begins a shell comment, so a path appearing only in a comment can cause a false denial. Match the existing `splitShellArgs()` comment behavior when scanning payload tokens.
  - confidence: 0.82
- **[P2] Do not resolve every bare token against cwd:** `src/infra/exec-deny-path.ts:236-243`
  - body: When `cwd` is set, every non-flag token is resolved against it, including executable names and plain words. Running from a denied directory can therefore reject unrelated commands with messages such as `arg="echo"`; restrict cwd resolution to plausible path arguments or skip the executable token.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR adds a useful security control, but the changed command-execution boundary still has concrete deny-bypass paths and a failing CI regression.

Concerns:

- **[high] Node-host shell payloads can miss the deny gate:** `src/node-host/invoke-system-run.ts:407`
  - body: CI shows `/bin/sh -lc "cat ~/.openclaw/secrets/foo.env; echo ok"` produced `reason: "allowlist-miss"` instead of `reason: "deny-path-pattern"`, so the proposed hard deny can fail before execution is blocked.
  - confidence: 0.93
- **[high] Windows paths can bypass gateway deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway gate feeds command text through POSIX-style argv splitting before deny matching, which can transform unquoted Windows backslash paths before the deny matcher sees them.
  - confidence: 0.9
- **[high] cmd and PowerShell wrappers hide denied payloads:** `src/agents/bash-tools.exec.ts:1472`
  - body: The gateway pre-dispatch gate uses a POSIX-only payload extractor instead of the shared shell-wrapper resolver, leaving cmd and PowerShell payloads outside proper deny-token scanning.
  - confidence: 0.89
- **[high] Backslash-newline can hide POSIX denied paths:** `src/infra/exec-deny-path.ts:130`
  - body: The tokenizer preserves a backslash-newline split while POSIX shells join it before execution, so a denied path can be split across physical lines and miss the glob.
  - confidence: 0.86
