## Summary

Keep open: current main still lacks the requested exec path-deny gate, and this PR is the active replacement for https://github.com/openclaw/openclaw/pull/75101, but the patch still has concrete parser/wrapper bypasses and the current PR body lacks real behavior proof.

## What This Changes

Adds `tools.exec.denyPathPatterns` config/schema/docs support plus pre-exec runtime deny checks for gateway, node, and sandbox exec paths.

## Best Possible Solution

Repair the deny gate to reuse established wrapper parsing and shell semantics, add current-PR real behavior proof, then land after focused security review.

## Reproduction Assessment

Yes. Source inspection of the PR head gives concrete paths for Windows backslash corruption, cmd/PowerShell payload bypasses, POSIX backslash-newline bypasses, and shell-comment false denials.

## Solution Assessment

No. A default-off hard-deny config is the right direction for the linked security request, but this implementation needs parser and wrapper-resolution fixes before it is the maintainable merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway deny gate parses command text with `splitShellArgs()`, which applies POSIX-style backslash escaping before matching. On a Windows gateway, a command like `type C:\Users\alice\.ssh\id_rsa` can be corrupted before the matcher sees it, so configured Windows deny globs may miss the secret path.
  - confidence: 0.9
- **[P1] Scan cmd and PowerShell payloads in the gateway gate:** `src/agents/bash-tools.exec.ts:1472-1475`
  - body: The gateway gate extracts inner payloads only through the POSIX-shell helper, so `cmd.exe /c` and `pwsh -Command` payloads can remain a single command-string argument instead of being tokenized for denied paths. Use the existing shared wrapper resolver so Windows shell payloads get the same deny scan.
  - confidence: 0.87
- **[P1] Match shell backslash semantics before denying paths:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` treats every backslash as a generic escape and preserves escaped newlines. That corrupts Windows paths inside shell payloads and misses POSIX backslash-newline continuations where the shell joins a denied path before execution.
  - confidence: 0.86
- **[P2] Ignore unquoted shell comments during tokenization:** `src/infra/exec-deny-path.ts:149-155`
  - body: The tokenizer never stops at an unquoted `#` comment, so `bash -c 'echo ok # ~/.openclaw/secrets/foo.env'` is denied even though the shell ignores that path. Match the existing shell parsing behavior to avoid false denials.
  - confidence: 0.8

## Security Review

- **[high] Windows paths can bypass gateway deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway gate feeds command text through a POSIX-style splitter before matching, so unquoted Windows backslash paths can be transformed before the deny matcher sees them.
  - confidence: 0.9
- **[high] cmd and PowerShell wrappers hide denied payloads:** `src/agents/bash-tools.exec.ts:1472`
  - body: The gateway pre-dispatch gate uses a POSIX-only payload extractor instead of the shared shell-wrapper resolver, leaving cmd and PowerShell payloads outside proper deny-token scanning.
  - confidence: 0.87
- **[high] Backslash-newline can hide POSIX denied paths:** `src/infra/exec-deny-path.ts:130`
  - body: The payload tokenizer preserves a backslash-newline as a literal newline, while POSIX shells join it before execution, so a denied path can be split across lines and miss the glob.
  - confidence: 0.86
