## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Resolve node-host shell payloads before policy fallback:** `src/node-host/invoke-system-run.ts:407`
  - body: The deny check only scans `parsed.shellPayload`, but that can be null for legacy shell-wrapper argv like `/bin/sh -lc ...`; the PR's own CI shows the denied path falling through as `allowlist-miss`. Reuse the shared shell-wrapper resolver or recompute the payload before allowlist handling so approved node-host runs cannot execute the inner denied path.
  - confidence: 0.92
- **[P1] Preserve Windows path backslashes before deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway gate parses the raw command with POSIX `splitShellArgs()` before matching. On Windows, unquoted paths such as `C:\Users\alice\.ssh\id_rsa` can be backslash-unescaped before the matcher sees them, causing Windows deny globs to miss protected paths.
  - confidence: 0.9
- **[P1] Use shared wrapper parsing for cmd and PowerShell:** `src/agents/bash-tools.exec.ts:1472-1475`
  - body: The gateway gate extracts inner payloads with the local POSIX-only `extractShellWrappedCommandPayload`, so `cmd.exe /c` and `pwsh -Command` payloads are not tokenized for denied paths. Use the existing shared shell-wrapper resolver so Windows wrappers receive the same deny scan.
  - confidence: 0.88
- **[P1] Match shell backslash-newline semantics:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` treats every backslash as a generic escape and preserves escaped newlines. POSIX shells join backslash-newline before execution, so a denied path split across physical lines can miss the deny glob.
  - confidence: 0.86
- **[P2] Ignore unquoted comments while tokenizing payloads:** `src/infra/exec-deny-path.ts:149-155`
  - body: The tokenizer splits whitespace and operators but never stops at an unquoted `#`, so a path appearing only in a shell comment can cause a false denial. Match the existing shell parsing behavior for comments when scanning payload tokens.
  - confidence: 0.8
- **[P2] Require path-like tokens before cwd resolution:** `src/infra/exec-deny-path.ts:236-242`
  - body: When `cwd` is present, every non-flag token is resolved against it, including plain executable names like `echo`. Running a harmless command from a denied directory can therefore match a file glob with `arg="echo"`; keep cwd resolution scoped to tokens that plausibly name paths.
  - confidence: 0.82
