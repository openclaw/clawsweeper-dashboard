The PR adds a default-off `tools.exec.denyPathPatterns` config/schema/docs surface and pre-exec path matching in bash exec and node-host `system.run`, plus focused tests.

## Best Possible Solution

Land a repaired default-off implementation that reuses the shared wrapper parsing/path helpers, covers gateway/node/sandbox paths with focused tests, and includes redacted real behavior proof for actual exec and system.run denial.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Evaluate legacy node-host shell payloads before policy fallback:** `src/node-host/invoke-system-run.ts:407`
  - body: For legacy `/bin/sh -lc` argv without `rawCommand`, `parsed.shellPayload` can be null, and CI shows this path emits `allowlist-miss` instead of `deny-path-pattern`. Resolve the shell payload for deny matching, or use the shared wrapper resolver, before policy fallback so approved or `security: full` node-host runs cannot execute the inner denied path.
  - confidence: 0.94
- **[P1] Preserve Windows paths before deny matching:** `src/agents/bash-tools.exec.ts:1461`
  - body: The gateway gate runs `splitShellArgs()` over the command text before matching. That POSIX parser consumes backslashes outside quotes, so an unquoted Windows path such as `C:\Users\alice\.ssh\id_rsa` can be transformed before the deny matcher sees it and miss Windows deny globs.
  - confidence: 0.9
- **[P1] Reuse the shared wrapper resolver for gateway deny payloads:** `src/agents/bash-tools.exec.ts:1472-1475`
  - body: The local payload extractor only recognizes POSIX shells, while current exec policy already has shared cmd and PowerShell wrapper handling. As written, `cmd.exe /c` and `pwsh -Command` payloads are not tokenized for denied paths in the gateway gate.
  - confidence: 0.89
- **[P1] Match shell backslash-newline semantics before denying paths:** `src/infra/exec-deny-path.ts:130-132`
  - body: `tokenizeShellPayload()` preserves a backslash-newline as a newline in the candidate token, but POSIX shells remove that continuation before execution. A denied path split across the physical line can therefore miss both exact globs and `**` regexes, because JavaScript `.` does not match newlines by default.
  - confidence: 0.86
- **[P2] Ignore unquoted shell comments while scanning payload tokens:** `src/infra/exec-deny-path.ts:149-155`
  - body: The tokenizer splits whitespace and operators but never stops at an unquoted `#`. A denied path that appears only in a shell comment can trigger a false denial, unlike the existing shell argv parser's comment behavior.
  - confidence: 0.8
- **[P2] Skip non-path argv tokens when resolving against cwd:** `src/infra/exec-deny-path.ts:236-242`
  - body: When `cwd` is present, every non-flag token is resolved against that directory, including the executable token. With a directory deny glob such as a secrets directory, a harmless command run from that cwd can be denied with `arg="echo"`; keep cwd resolution limited to tokens that plausibly name a path.
  - confidence: 0.76
