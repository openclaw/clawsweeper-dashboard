## Summary

Keep this PR open: the requested filter is useful, but the current patch would hide default-agent cron jobs whose stored `agentId` is omitted when filtering for `main` or the configured default agent.

## What This Changes

The PR adds optional `agentId` filtering to cron list across the CLI, agent tool, gateway protocol, cron service, generated Swift protocol models, docs, tests, and changelog.

## Best Possible Solution

Keep the new filter, but compare each job's effective agent id by treating an omitted `job.agentId` as the configured default agent, with regression coverage for `--agent main` and default-agent sessions.

## Reproduction Assessment

Yes, from source inspection: current cron semantics treat omitted `agentId` as the default agent, while the PR's strict equality filter would exclude those jobs when filtering for `main` or the configured default agent. I did not execute tests under the read-only review constraint.

## Solution Assessment

No. The proposed surface is the right narrow direction, but the service filter should use effective agent ids with the default-agent fallback before this is safe to automerge.
