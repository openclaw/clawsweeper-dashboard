## Summary

Keep open: current main still lacks the native Slack plan/task-card progress path, but the current PR head is draft, has stale real-behavior proof after a later merge conflict resolution, and still has two source-visible P2 correctness defects.

## What This Changes

The PR changes Slack progress draft rendering and native Slack streaming to use plan/task_card blocks plus plan_update/task_update chunks, with progress-line identity updates and focused tests.

## Best Possible Solution

Land the native Slack plan/task-card direction after stable task identities, success-aware completion/error handling, fresh redacted Slack proof against the current head, and focused Slack regression coverage are in place.

## Reproduction Assessment

Yes for the review findings: PR-head source shows completion chunks emitted without success/error gating, and task IDs still include a mutable array index. I did not run a live Slack failure path.

## Solution Assessment

No, not as currently patched. The native Slack plan/task-card approach matches the intended direction, but it needs stable task IDs, failure-aware finalization, and fresh current-head proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Only complete progress tasks after successful finals:** `extensions/slack/src/monitor/message-handler/dispatch.ts:844-851`
  - body: This path builds completion chunks for every streamed final append, including error finals, and the stop path repeats completion after `dispatchError` can already be captured. Failed turns can therefore leave Slack showing in-flight task cards as `complete`; gate completion on successful final dispatch or emit error-state chunks instead.
  - confidence: 0.87
- **[P2] Keep native task update IDs stable:** `extensions/slack/src/progress-blocks.ts:102`
  - body: `task_id` and `task_update.id` still include the task's current array index. When max-line pruning drops an older line, the same running item can change from an ID like `cmd_2_2` to `cmd_2_1`, causing Slack to treat it as a different task and leaving stale cards; use the stable line id directly when present.
  - confidence: 0.84
