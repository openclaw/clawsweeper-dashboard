## Summary

Keep open: this is an active draft PR for a valid Slack feature request, but it still needs real Slack proof and the proposed plan block currently reuses a fixed block identity across draft updates.

## What This Changes

The PR changes Slack rich progress drafts to render as a single Slack `plan` block with task entries and adds focused renderer/dispatch tests.

## Best Possible Solution

Keep the focused Slack renderer approach, but make the plan block identity safe for repeated message updates and require redacted live Slack proof before the PR is ready.

## Reproduction Assessment

Not applicable: this is a Slack UX feature request, not a broken existing behavior report. Source inspection confirms current main still uses generic rich progress fields and the PR changes that surface.

## Solution Assessment

No, not as currently patched: the renderer seam is the right place, but the fixed plan `block_id` conflicts with Slack's update contract and should be removed or regenerated per update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Regenerate the plan block id on draft updates:** `extensions/slack/src/progress-blocks.ts:118`
  - body: The progress draft message is updated in place whenever new tool progress arrives, but this constant `block_id` is reused for every rendered plan block. That conflicts with Slack's update contract for block IDs and can leave Slack with stale block identity during repeated draft edits; omit the block ID or derive a fresh one for each update iteration.
  - confidence: 0.84
