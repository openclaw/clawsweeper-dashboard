## Summary

Keep open: this is an active draft PR for a valid linked Slack UX feature, but the proposed renderer currently regresses the existing hidden-label contract and still lacks the required redacted live Slack proof before merge.

## What This Changes

The PR changes Slack rich progress drafts from section-field blocks to one Slack `plan` block with task entries and adds focused renderer/dispatch coverage.

## Best Possible Solution

Update the PR to preserve `label: false`, avoid stale Slack block identity on repeated draft edits, add focused regression coverage, and attach redacted live Slack proof before review-to-merge.

## Reproduction Assessment

Yes for the review finding: source inspection shows `label: false` resolves to `undefined` today, while the PR substitutes a visible default plan title on the changed line. The broader request is a feature, so normal bug reproduction is not applicable.

## Solution Assessment

No as written: the existing Slack rich-progress hook is the right narrow seam, but the branch needs to preserve the existing hidden-title contract and should not reuse a constant `block_id` across edited draft iterations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve hidden progress labels:** `extensions/slack/src/progress-blocks.ts:120`
  - body: When users configure `streaming.progress.label: false`, `resolveChannelProgressDraftLabel` passes `undefined` and current rich rendering omits the label section while still showing progress rows. This change always sends a plan block titled `OpenClaw is working`, so hidden-label configs become visibly titled again. Please keep that contract, for example by falling back to the old section-row shape or otherwise avoiding a synthetic title when the label is disabled.
  - confidence: 0.91
- **[P3] Avoid reusing one Slack block id for every draft edit:** `extensions/slack/src/progress-blocks.ts:119`
  - body: Progress drafts edit the same Slack message repeatedly as task state changes, but this constant `block_id` is sent on every update. Slack documents `block_id` as an iteration-specific identifier for updated messages/views, so please omit it here or vary it per draft update to avoid stale block identity in Slack clients.
  - confidence: 0.74
