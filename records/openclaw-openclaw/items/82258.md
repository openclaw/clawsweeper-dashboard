## Summary

Keep open: current main does not include the Slack plan/task-card progress behavior, and this active draft PR is the implementation path, but the current patch has a correctness blocker around native Slack task IDs.

## What This Changes

The PR changes the Slack plugin to render progress drafts as Slack plan/task_card blocks and stream native plan_update/task_update chunks, with Slack renderer and dispatch tests.

## Best Possible Solution

Keep the native Slack plan/task-card direction, but generate stable per-progress-item stream IDs and cover capped or long-running progress updates before marking the PR ready.

## Reproduction Assessment

Yes for the review finding: PR-head source shows task_update IDs are derived from the current visible index, and the dispatch path caps/slides progress lines before emitting chunks. I did not run a live Slack repro in this read-only review.

## Solution Assessment

No, not as currently patched. Native Slack stream chunks are the right surface for the requested UX, but task_update needs stable IDs across updates instead of index-derived IDs that can shift during long runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep task update IDs stable across capped progress:** `extensions/slack/src/progress-blocks.ts:102`
  - body: `task_update` uses `id` to update existing Slack tasks, but this derives the ID from the current visible array index after progress lines have already been capped. Once a long run drops an older line, remaining tasks are renumbered, so Slack can update the wrong task card or leave stale cards in the native plan. Use a stable progress-line identity for streamed task IDs rather than the current index.
  - confidence: 0.84
