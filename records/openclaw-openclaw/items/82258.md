## Summary

Keep open: current main does not implement native Slack plan/task-card progress rendering, but this draft PR is not merge-ready because PR head appears to break the reply callback type contract and still marks native progress tasks complete on error/failure paths. The supplied Slack Web API/session-log proof is relevant, but it predates the latest head commit.

## What This Changes

The PR enables native Slack progress streaming with plan/task chunks, adds Slack progress chunk/rendering helpers, threads progress-line IDs through the SDK helper, and adds Slack/SDK tests plus a local trajectory proof script.

## Best Possible Solution

Land the Slack-native progress work through the paired feature issue after the callback contract is aligned, task finalization is success/error-aware, and fresh current-head Slack proof demonstrates visible progress and final task state.

## Reproduction Assessment

Yes for the review findings: PR-head source shows reads from callback fields that are not in the current onToolStart contract and completion chunks emitted without success/error guards. I did not run live Slack in this read-only review.

## Solution Assessment

No, not as currently patched. The native Slack plan/task-card direction fits the request, but the implementation should update the callback contract or avoid those fields, gate completion on successful finals or emit error chunks, and provide fresh current-head proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Update the progress callback contract before reading tool IDs:** `extensions/slack/src/monitor/message-handler/dispatch.ts:1482-1483`
  - body: `dispatchInboundMessage` types `replyOptions` as `Omit<GetReplyOptions, "onBlockReply">`, and `GetReplyOptions.onToolStart` currently exposes only `name`, `phase`, `args`, and `detailMode`. Reading `payload.itemId` and `payload.toolCallId` here will break the extension typecheck unless the callback contract and producer plumbing are updated; otherwise do not read those fields on this path.
  - confidence: 0.91
- **[P2] Gate native task completion on successful finals:** `extensions/slack/src/monitor/message-handler/dispatch.ts:845-858`
  - body: This builds completion chunks for any appended final text, and the stop path repeats the same behavior after dispatch settles, without checking `payload.isError` or `dispatchError`. Failed Slack runs can show active task cards as complete; gate completion on successful finals or emit error-state chunks.
  - confidence: 0.9
