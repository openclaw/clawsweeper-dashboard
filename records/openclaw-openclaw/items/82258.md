## Summary

Keep open. Current main still renders Slack rich progress drafts as legacy section/field blocks and disables native streaming for progress mode, so this PR contains unique Slack UX work, but PR-head source has blocking correctness issues around failed-turn finalization and stable native task IDs.

## What This Changes

The PR changes Slack progress rendering and channel-streaming progress line handling to render plan/task_card blocks and stream plan_update/task_update chunks for native progress mode.

## Best Possible Solution

Keep the native plan/task_card direction, but revise completion handling to preserve failed/error state and use stable task IDs before marking the draft ready and landing with focused Slack tests plus repaired-path proof.

## Reproduction Assessment

Yes for the review findings: PR-head source shows completion chunks are emitted without checking failed final/dispatch state, and task_update IDs include the current array index that changes after pruning. I did not run a live Slack failure case in this read-only review.

## Solution Assessment

No, not as currently patched. The Slack-owned plan/task_card and structured chunk direction is maintainable, but the implementation needs stable task identities and success-aware finalization before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Only complete progress tasks after successful finals:** `extensions/slack/src/monitor/message-handler/dispatch.ts:831-839`
  - body: This path builds completion chunks for every streamed final, including error finals, and the same completion helper is later used after `dispatchError` has already been captured. Failed turns can therefore leave Slack showing all in-flight task cards as `complete`; gate completion on a successful final/dispatch or emit error-state chunks instead.
  - confidence: 0.86
- **[P2] Keep native task update IDs stable:** `extensions/slack/src/progress-blocks.ts:102`
  - body: `task_id`/`task_update.id` includes the task's current array index, but progress lines are capped and sliced. Once an older line drops, a still-running task changes from IDs like `item_2_2` to `item_2_1`, so Slack treats it as a different task and can leave stale cards instead of updating the existing one; use the line's stable id when present and only synthesize an index for id-less lines.
  - confidence: 0.82
