## Summary

Keep open: current main still uses the old Slack progress renderer, and this active draft PR is the paired implementation for the open Slack task-card feature, but the patch still has failure-state correctness gaps before merge.

## What This Changes

The PR changes the Slack plugin to render progress drafts as native Slack plan/task_card blocks and stream plan_update/task_update chunks, with focused renderer, dispatch, and streaming tests.

## Best Possible Solution

Keep the native Slack plan/task-card direction, but revise completion so failed/error progress remains error or unfinished, then cover that path with focused Slack dispatch tests and fresh live proof after revision.

## Reproduction Assessment

Yes for the review findings: PR-head source shows error states can be present while completion chunks still mark in-flight native progress tasks complete. I did not run a live failing Slack turn in this read-only review.

## Solution Assessment

No, not as currently patched. The plan/task_card and structured Slack chunk approach is the right product direction, but finalization needs to preserve failure/error state before this is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve failure state when ending native progress:** `extensions/slack/src/monitor/message-handler/dispatch.ts:1502-1505`
  - body: `dispatchError` can already be set here, but the finalizer still builds completion chunks for every in-flight progress line before rethrowing. A failed tool/model turn can leave Slack showing completed task cards for work that actually failed; skip completion or emit error-state chunks when `dispatchError` is present.
  - confidence: 0.86
- **[P2] Do not complete tasks for error final replies:** `extensions/slack/src/monitor/message-handler/dispatch.ts:835-838`
  - body: This completion chunk path runs for any final text payload, including `payload.isError`. Native progress mode can therefore append an error reply while marking unfinished tasks complete, unlike the existing preview-finalization path that refuses to promote error finals; gate completion on a non-error final or emit error/unfinished task state.
  - confidence: 0.78
