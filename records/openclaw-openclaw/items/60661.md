- `merge-risk: 🚨 session-state`: The change is in accepted gateway run handling, embedded fallback behavior, and run/session timeout ownership, where a wrong merge can mis-associate or duplicate session work.

## Summary

Keep this PR open because current main and the latest release still forward the CLI timeout into the gateway-owned agent run, so the central timeout-ownership problem is not already implemented. The branch is not merge-ready as submitted: it imports a gateway guard that does not exist, conflicts with current main, lacks inspectable real behavior proof, and changes `--timeout` semantics in a compatibility-sensitive session path.

## What This Changes

The PR removes the forwarded `timeout` from gateway agent params and adds tests plus accepted-run handling so `openclaw agent --timeout` remains a client-side wait budget after gateway acceptance.

## Best Possible Solution

Rework this on top of current main so client wait budget and gateway-owned run timeout semantics are explicit, tested, and proven without losing the newer accepted-run abort and fallback handling.

## Reproduction Assessment

Yes, from source: current main and `v2026.6.6` forward `--timeout` as both `params.timeout` and `timeoutMs`, and the gateway server uses `request.timeout` for run expiry. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No, not as submitted. The timeout-removal idea may be the right direction, but the branch must be rebased and the missing gateway guard plus timeout-ownership compatibility question need resolution first.
