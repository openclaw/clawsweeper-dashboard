## Summary

Keep open: the PR still targets a real Gateway CLI timeout boundary, but the current patch is not mergeable because it imports a non-existent accepted-ownership guard, lacks inspectable real behavior proof, and conflicts with current documented `--timeout` ownership semantics.

## What This Changes

The PR removes the `openclaw agent --timeout` value from Gateway `agent` params, adds an accepted-ownership catch branch, and updates gateway CLI unit tests for client-side timeout behavior.

## Best Possible Solution

Decide the Gateway `openclaw agent --timeout` contract first, then land a current-main patch that either keeps the documented fresh-session fallback or implements a real accepted-run handoff without forwarding run timeout, with docs and regression tests aligned.

## Reproduction Assessment

Yes for source-level checks: current main forwards `--timeout` into Gateway `agent` params and the server consumes it as run timeout, while the PR head imports a guard that production `src/gateway/call.ts` does not export. I did not run live Gateway because this review is read-only and the PR lacks inspectable proof artifacts.

## Solution Assessment

No. The timeout-boundary direction may be valid, but this patch adds a mocked-only accepted-ownership branch and leaves the CLI `--timeout` contract/docs unresolved; production call/error plumbing or a simpler current-main repair with aligned docs would be safer.
