## Summary

Keep open. The requested timeout-ownership change is not implemented on current main, and the PR cannot land as-is because its new accepted-ownership guard is imported from a module that does not export it; the existing review comments correctly identify a build/runtime blocker.

## What This Changes

The PR removes the forwarded Gateway `agent` timeout, adds regression coverage for that request shape, and adds an accepted-ownership fallback path in `agentCliCommand`.

## Best Possible Solution

Implement the accepted-run ownership behavior against current main: once the Gateway has accepted an agent run, a CLI wait deadline or post-accepted transport close should not abort or duplicate the Gateway-owned run, and the CLI/docs/tests should clearly separate client wait budget from agent runtime timeout.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main still forwards `timeout` into the Gateway `agent` params and the server consumes it as the run timeout, while the PR diff imports a guard that current `src/gateway/call.ts` does not export.

## Solution Assessment

No. The PR is aimed at the right timeout ownership boundary, but the implementation is incomplete because tests mock the missing export; a safer fix should add real accepted-ownership plumbing or adjust the current timeout/fallback path with tests that exercise the actual gateway call module.
