## Summary

Keep open: the PR is still necessary and the patch looks correct. Current main still lets store-stripped custom OpenAI Responses routes replay prior item ids, while this branch fixes both the direct transport builder and provider context wrapper with focused coverage and before/after runtime proof.

## What This Changes

The PR disables OpenAI Responses item-id replay for custom store-stripped Responses routes while preserving stateless `call_id` linking and existing store-enabled replay behavior.

## Best Possible Solution

Land after maintainers accept that `compat.supportsStore=false` means no server-side Responses item-id replay by default, while keeping provider-owned opt-ins for documented exceptions such as Foundry.

## Reproduction Assessment

Yes. Source inspection shows current main ignores `shouldStripStore` in both replay decision paths, and the PR body supplies before/after terminal proof against a storeless Responses-compatible endpoint.

## Solution Assessment

Yes. The patch uses the existing resolved payload policy in the two payload-construction paths, while leaving explicit provider-owned opt-ins available for exceptions instead of adding a new config surface.
