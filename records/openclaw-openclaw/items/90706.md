## Summary

Keep open: current main still has the storeless OpenAI Responses replay gap, and this PR is the active focused fix with no blocking patch findings. The remaining concern is maintainer compatibility acceptance for changing the default payload shape on `compat.supportsStore=false` custom Responses routes.

## What This Changes

The PR disables prior OpenAI Responses item-id replay when the resolved payload policy strips `store`, applies the same gate in the OpenAI Responses wrapper, and adds direct/wrapper regression coverage.

## Best Possible Solution

Land the focused fix if maintainers accept that `compat.supportsStore=false` means no server item-id replay by default, with provider-owned opt-ins for known exceptions such as Microsoft Foundry.

## Reproduction Assessment

Yes. Source inspection on current main shows `compat.supportsStore=false` sets `shouldStripStore=true` while the direct builder still gates replay only on `explicitStore !== false`, and the wrapper leaves replay undefined in that path.

## Solution Assessment

Yes. The PR uses the existing payload policy at both replay handoff points; stripping ids later inside message conversion would be a weaker layer because it lacks the resolved store policy and provider-wrapper override boundary.
