## What This Changes

The PR updates the OpenAI Responses transport replay gate to omit prior response item IDs when the resolved payload policy strips `store`, and adds a regression test for storeless custom providers.

## Best Possible Solution

Land this narrow transport fix after normal CI, while keeping sibling strict-provider and stream-normalization issues tracked separately.

Yes, source-level reproduction is high confidence: current main’s replay default ignores `shouldStripStore`, and the PR adds a payload-capture regression for `compat.supportsStore=false`. I did not rerun the tests because this review was read-only.

## Solution Assessment

Yes. Reusing the existing payload policy’s `shouldStripStore` decision in the Responses builder is the narrowest maintainable fix, and preserving `call_id` pairing avoids weakening tool continuation.
