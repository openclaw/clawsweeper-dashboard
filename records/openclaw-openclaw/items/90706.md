## Summary

This PR should stay open for maintainer compatibility review and normal landing. The patch appears correct and the supplied proof is sufficient, but it intentionally changes the default outbound payload for custom storeless OpenAI Responses-compatible routes, so the remaining decision is whether maintainers accept that compatibility contract.

## What This Changes

The PR disables OpenAI Responses item-id replay for `compat.supportsStore=false` storeless routes in the direct builder and context-management wrapper, and adds regression coverage for storeless, store-capable, Azure, and Foundry paths.

## Best Possible Solution

Land this after maintainer acceptance that `compat.supportsStore=false` means no server item-id replay by default, with provider-owned opt-ins retained for known exceptions like Foundry.

## Reproduction Assessment

Yes. Source inspection shows current main resolves `shouldStripStore=true` for `compat.supportsStore=false`, while the direct builder still gates replay only on `explicitStore !== false` and the wrapper leaves replay undefined in that path.

## Solution Assessment

Yes. The best bounded fix is to use the existing payload policy in both replay decision points; moving the logic into message conversion alone would not know the resolved store policy or wrapper override contract.
