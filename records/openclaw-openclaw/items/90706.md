## Summary

Keep open for maintainer review: current main still has the storeless replay gap, and this PR fixes the direct Responses builder plus the provider wrapper with focused regression coverage and credible terminal proof. I found no blocking code or security findings, but maintainers should explicitly accept the compatibility contract because the PR changes default outbound payloads for custom `openai-responses` routes that declare `compat.supportsStore=false`.

## What This Changes

The PR disables OpenAI Responses item-id replay when the resolved payload policy strips store support, updates the provider context-management wrapper to force replay off for storeless routes, and adds regression tests for direct and wrapped paths.

## Best Possible Solution

Land the focused replay gate after maintainers accept `compat.supportsStore=false` as the storeless replay contract, while keeping provider-owned opt-ins like Microsoft Foundry for known exceptions.

## Reproduction Assessment

Yes. Source inspection on current main shows `shouldStripStore` can be true while replay is still allowed, and the PR body supplies before/after transport-path terminal proof for that exact payload shape.

## Solution Assessment

Yes. The fix is narrow and placed at the two replay decision points that already consume the payload policy; changing store stripping itself or adding a new config knob would be broader and more likely to create parallel policy.
