## Summary

Keep open: the PR is still useful and current main still has the storeless Responses replay gap; the patch is a focused fix with strong proof, but it intentionally changes custom-provider payload defaults and needs maintainer compatibility acceptance before merge.

## What This Changes

The PR disables prior OpenAI Responses item-id replay when the resolved payload policy strips `store`, applies the same gate in the OpenAI Responses wrapper, and adds direct/wrapper regression tests for storeless and store-enabled routes.

## Best Possible Solution

Land the focused fix if maintainers accept that `compat.supportsStore=false` means no server item-id replay by default, with provider-owned opt-ins for known exceptions like Foundry and normal required checks green on the current head.

## Reproduction Assessment

Yes. Source inspection on current main shows `compat.supportsStore=false` sets `shouldStripStore=true` while the direct builder still gates replay only on `explicitStore !== false`, and the wrapper leaves replay undefined in that path.

## Solution Assessment

Yes. The best bounded fix is to use the existing payload policy at both replay handoff points; stripping ids later in message conversion would lose the resolved store policy and provider-wrapper override boundary.
