## Summary

Current main and the latest stable release now bound Telegram `getMe` startup calls, pass cached/probed bot info into grammY so `bot.init()` does not repeat `getMe` on healthy startup, and restart stalled polling workers with channel-status error patches, so the reported silent freeze is addressed well enough to close.

## What This Changes

The issue reports an intermittent Telegram long-polling freeze after `getMe` fetch timeouts where no channel exit, reconnect, or health degradation is observed.

## Best Possible Solution

Keep the shipped v2026.6.6 Telegram startup and polling liveness implementation; reopen only with fresh v2026.6.6-or-newer logs showing a remaining silent freeze.

## Reproduction Assessment

No, I did not establish a current-main failing reproduction. The reporter’s production logs match the old failure shape, while current source, dependency contract, regression tests, and release provenance show the `getMe` startup and polling-stall paths are now bounded.

## Solution Assessment

Yes, the current solution is the right layer: Telegram plugin startup owns the `getMe` timeout/botInfo handoff, and polling-session liveness owns stalled worker restart and channel-status error surfacing without adding a new config knob or external watchdog.
