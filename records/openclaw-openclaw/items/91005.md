## Label Justifications

- `impact:message-loss`: A frozen Telegram long-poll worker can silently stop receiving user messages.

## Summary

Current main and the latest stable release bound Telegram startup `getMe` calls, avoid duplicate grammY `getMe` on healthy startup via `botInfo`, and force-restart stalled polling workers with status errors, so the reported silent freeze is already addressed.

## What This Changes

The issue reports Telegram long-polling becoming silently inactive after a `fetch-timeout` on `getMe`, with no channel exit, reconnect, or health degradation.

## Best Possible Solution

Keep the shipped v2026.6.6 Telegram startup and polling liveness behavior, and reopen only with fresh v2026.6.6-or-newer logs showing a remaining silent freeze.

## Reproduction Assessment

No high-confidence current-main failing reproduction was established. The reporter's production logs match the old failure shape, while current source, dependency contract, regression coverage, and v2026.6.6 provenance show the `getMe` startup and polling-stall paths are now bounded.

## Solution Assessment

Yes, the shipped solution is the right layer: Telegram plugin startup owns the `getMe` timeout and botInfo handoff, while polling-session liveness owns stalled worker restart and status surfacing without adding a new watchdog script or config surface.
