## Label Justifications

- `P2`: The issue describes a Telegram liveness failure with real message-delivery impact, but the scope is limited to one channel path and is now implemented.
- `impact:message-loss`: The reported freeze can leave Telegram inbound polling inactive and messages unhandled.
- `impact:crash-loop`: The report is about channel worker liveness, restart behavior, and process-level availability after timeout failures.

## Summary

Current `main` and the shipped `v2026.6.6` tag contain the Telegram startup and polling-liveness behavior that covers the reported silent `getMe` timeout freeze: startup `getMe` is bounded, bot identity is handed to grammY when available, polling stalls publish unhealthy status, and stalled workers/runners restart instead of wedging silently.

## What This Changes

The issue reports intermittent silent freezes in the Telegram long-polling worker after `getMe` fetch timeouts, with no restart signal and health still reporting live.

## Best Possible Solution

Keep the shipped/current Telegram startup and polling-liveness implementation, and ask for a new report with a diagnostics export only if `v2026.6.6` or later still shows a silent `getMe` freeze.

## Reproduction Assessment

No high-confidence live reproduction was established during this read-only cleanup. The reporter's production logs are credible for older builds, and current source plus regression coverage now exercise the bounded startup and polling-stall recovery paths.

## Solution Assessment

Yes. The implementation is at the right owner boundary: Telegram startup owns the bounded `getMe`/botInfo handoff, while Telegram polling liveness owns stalled-worker status and restart without adding a new config knob or external watchdog.
