## Summary

Current main already addresses the central silent Telegram polling freeze: getMe/control-plane calls are bounded, grammY startup gets botInfo when available, polling stalls are surfaced into channel status, and both classic and isolated polling paths request a restart instead of staying silently wedged.

## What This Changes

The issue reports that Telegram long-polling can stop delivering messages silently after a getMe fetch timeout, leaving health checks green and requiring an external watchdog restart.

## Best Possible Solution

Keep the current-main Telegram polling recovery and status implementation; use `/ready`, `openclaw channels status --probe`, and the channel health monitor for channel readiness rather than changing the intentionally shallow `/health` liveness probe.

## Reproduction Assessment

No high-confidence current-main failing reproduction remains. The reporter's logs are credible for older builds, while current source and regression coverage exercise bounded startup getMe handling, polling-stall status, and restart paths.

## Solution Assessment

Yes. The current implementation solves this at the owner boundary: Telegram startup owns bounded getMe/botInfo setup, Telegram polling owns stall detection and restart, and gateway health policy consumes the resulting channel status.
