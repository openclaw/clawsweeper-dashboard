## Summary

Keep open. Current main has the pieces for Claude CLI Keychain reads, but the runtime auth paths still default to non-Keychain discovery and the external CLI overlay refuses to replace a still-future-dated local OAuth credential, which matches the reported forced macOS re-login/stale-token gap rather than proving it fixed.

## What This Changes

The issue requests that macOS keychain-backed `anthropic:claude-cli` OAuth profiles be refreshed from Claude CLI Keychain state during gateway/runtime startup so stale cached tokens are not used after re-login.

## Best Possible Solution

Keep the issue open and add a targeted runtime refresh path for the Claude CLI external-auth profile that can adopt a newer same-identity Keychain credential for `anthropic:claude-cli` without changing read-only status/probe semantics or broadening external CLI scans.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction: a stored `anthropic:claude-cli` OAuth credential with a future expiry plus a different fresh Keychain credential will not be overlaid because current policy treats the local credential as usable. A live macOS Keychain run would still be useful before merging.

## Solution Assessment

No, the exact proposed `source: keychain` boot-wide sync is not the best fit for current main because the auth profile type does not expose that source field and boot-wide Keychain prompting would be too broad. The safer fix is scoped Claude CLI runtime discovery/adoption with explicit no-Keychain behavior preserved for status and probes.
