## Summary

Keep open. Current main still has the reported macOS gap: Claude CLI Keychain reading exists, but the provider-auth runtime path disables Keychain reads and the external OAuth adoption policy keeps a future-dated local `anthropic:claude-cli` token even when the external CLI has fresher same-source credentials.

## What This Changes

The issue requests that gateway startup/runtime auth for keychain-backed `anthropic:claude-cli` profiles re-read macOS Keychain credentials so stale cached OAuth tokens in `auth-profiles.json` do not cause 401 failures after Claude CLI login refreshes.

## Best Possible Solution

Add a narrow Claude CLI runtime refresh/adoption path that can read the in-scope macOS Keychain credential for `anthropic:claude-cli` provider auth and adopt a fresher same-identity credential even when the stored token has a future expiry, without adding broad gateway boot scans or Keychain reads in status/probe paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: store a future-dated local `anthropic:claude-cli` OAuth credential, provide a fresher same-identity Claude CLI Keychain credential, and follow the current provider-auth path; it passes `allowKeychainPrompt: false` and keeps the usable-by-expiry local token.

## Solution Assessment

No, the issue's proposed boot-wide `source: keychain` sync is not the narrowest fit because current auth profile credential types do not carry that source field and docs intentionally keep read-only/status paths off Keychain. The better fix is scoped Claude CLI runtime refresh/adoption for the configured provider/profile.
