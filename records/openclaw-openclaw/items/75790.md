## Summary

Keep open. Current main still has the macOS gap described here: Claude CLI Keychain reading exists, but the runtime auth path defaults to non-Keychain discovery and the external CLI overlay keeps a still-usable-by-expiry local OAuth credential instead of adopting a refreshed same-source credential.

## What This Changes

The issue requests refreshing macOS Keychain-backed Claude CLI OAuth credentials for `anthropic:claude-cli` during gateway/profile hydration so stale cached `auth-profiles.json` tokens are not used after `claude /login`.

## Best Possible Solution

Add a narrow Claude CLI runtime refresh/adoption path that can read the in-scope macOS Keychain credential and overlay or update `anthropic:claude-cli` when it is the same external identity, while preserving no-Keychain behavior for read-only status/probe paths and avoiding broad boot-wide credential scans.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: store a future-dated local `anthropic:claude-cli` OAuth credential, have a different fresh Claude CLI Keychain credential, then follow the current runtime auth path; it defaults to no Keychain read and would also skip replacement while the local expiry is usable.

## Solution Assessment

No, the exact proposed boot-wide `source: keychain` sync is not the best fit because current auth profile types do not expose that source field and docs intentionally keep status/probe paths off Keychain prompts. The safer solution is a scoped Claude CLI runtime refresh/adoption path for the configured provider/profile.
