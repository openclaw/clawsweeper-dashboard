## Summary

Keep open. The reported Gemini-behind-proxy schema-cleaning gap still exists on current main because the normalizer receives modelId but Gemini detection still only checks modelProvider; however, this PR should not merge as written because the new modelId branch takes the Gemini early return and can bypass modelCompat keyword stripping, and the branch is stale against the refactored current file layout.

## What This Changes

This PR adds modelId-based Gemini detection to tool parameter schema normalization and adds focused tests for Gemini models reached through custom OpenAI-compatible proxy providers.

## Best Possible Solution

Update the current normalizer in src/agents/pi-tools-parameter-schema.ts to detect Gemini from a normalized modelProvider or modelId, then compose provider cleaning so Gemini cleanup and modelCompat unsupported-keyword stripping can both run when both apply. Add regression coverage for custom provider plus Gemini modelId, patternProperties/const cleanup, non-Gemini custom providers, and Gemini modelId with modelCompat-only unsupported keywords, plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call normalizeToolParameterSchema with modelProvider set to a custom proxy name and modelId containing gemini; current main receives modelId but does not use it for Gemini detection, so Gemini-only cleanup is skipped. The PR adds tests for that path, but I did not run them because this review is read-only.

## Solution Assessment

No, not as written. The modelId check is the right boundary, but the implementation should apply Gemini cleaning and then still honor modelCompat.unsupportedToolSchemaKeywords instead of returning early, and it must be rebased onto the current split normalizer file.
