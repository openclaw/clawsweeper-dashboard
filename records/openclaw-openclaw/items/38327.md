## Summary

Keep open: current main and the latest shipped tag still only sanitize Google thinking payloads for `google-generative-ai`, while the reported Vertex path can share the same wrapper as `google-vertex` or resolve through `google-gemini-cli`. The focused replacement PR that widened this guard was closed unmerged, so the central bug remains actionable.

## What This Changes

The issue reports a regression where Google Vertex Gemini models crash embedded agent runs with `Cannot convert undefined or null to object` after upgrading to OpenClaw 2026.3.2.

## Best Possible Solution

Land a narrow Google-family wrapper guard widening with regression tests for `google-generative-ai`, `google-vertex`, and `google-gemini-cli`, while keeping any gaxios dependency behavior in a separate dependency-approved follow-up.

## Reproduction Assessment

Yes for source reproduction: current main routes Google-family providers through the shared wrapper, the wrapper only sanitizes `google-generative-ai`, and existing tests show Vertex can resolve with `api: "google-gemini-cli"`. I did not run a live Vertex credential repro.

## Solution Assessment

Yes for the wrapper gap: widening the existing Google-family guard to a small set of known Google API shapes is the narrow maintainable fix. The gaxios/native-fetch path should stay separate because it changes dependency behavior and has a different ownership boundary.
