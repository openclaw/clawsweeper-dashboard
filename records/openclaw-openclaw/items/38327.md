## Summary

Keep open: current main still only invokes the Google thinking payload sanitizer for `google-generative-ai`, while the reported Vertex and Gemini CLI paths share the wrapper and remain source-reproducible. The issue is already paired with the open closing PR https://github.com/openclaw/openclaw/pull/73416, so maintainers should review, land, or reject that PR rather than close this issue early.

## What This Changes

The issue reports a regression where Google Vertex Gemini models crash embedded agent runs with `Cannot convert undefined or null to object`.

## Best Possible Solution

Review and land the narrow guard widening with regression coverage from https://github.com/openclaw/openclaw/pull/73416, then split any remaining gaxios/native-fetch failure into a separate dependency-focused follow-up if it still reproduces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a negative Gemini thinking payload through `google-vertex` or `google-gemini-cli` reaches the shared Google wrapper but bypasses sanitization because current main only matches `google-generative-ai`.

## Solution Assessment

Yes for the wrapper gap. Widening the existing Google-family guard with focused wrapper regression tests is the narrow maintainable fix; the possible gaxios/native-fetch path should stay separate unless independently reproduced.
