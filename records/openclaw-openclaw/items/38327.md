## Summary

Keep open: current main and the latest release still only run the Google thinking payload sanitizer for `google-generative-ai`, while the reported Vertex/Gemini paths can reach the same stream wrapper as `google-vertex` or `google-gemini-cli`. The linked pull request https://github.com/openclaw/openclaw/pull/83991 is open with closing syntax and targets that exact gap, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue reports an embedded agent crash on every message after selecting `google-vertex/gemini-3.1-pro-preview`, caused by Google Vertex/Gemini payload handling after the 2026.3.2 update.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/83991 for the wrapper guard gap, then track any validated gaxios/native-fetch failure as a separate dependency/auth-stack item.

## Reproduction Assessment

Yes, at source level. Current main has a concrete path where `google-vertex` and `google-gemini-cli` can share the Google stream wrapper while bypassing the sanitizer guarded only for `google-generative-ai`; live Vertex credentials were not exercised.

## Solution Assessment

Yes for the wrapper gap. The narrow maintainable fix is to widen the existing Google-family sanitizer gate with regression coverage, while keeping the separate gaxios/native-fetch concern out of this issue unless maintainers approve that dependency/auth-stack work.
