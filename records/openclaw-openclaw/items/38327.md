## Summary

Keep open: current main and v2026.5.7 still only run the Google thinking payload sanitizer for `google-generative-ai`, so the reported `google-vertex` and `google-gemini-cli` paths remain source-reproducible. The open closing PR at https://github.com/openclaw/openclaw/pull/73416 is the right implementation candidate, but it is not merged or shipped.

## What This Changes

The issue reports that `google-vertex/gemini-3.1-pro-preview` embedded agent runs crash with `Cannot convert undefined or null to object` after upgrading to OpenClaw 2026.3.2.

## Best Possible Solution

Review and land the narrow guard widening with regression coverage from https://github.com/openclaw/openclaw/pull/73416, then split any remaining gaxios/native-fetch crash into a separately approved dependency follow-up if it still reproduces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a negative `thinkingBudget` payload through `google-vertex` or `google-gemini-cli` reaches the shared Google wrapper path but bypasses sanitizer logic because the current guard only matches `google-generative-ai`.

## Solution Assessment

Yes for the wrapper gap. Widening the existing Google-family guard and adding focused wrapper regression tests is the narrow maintainable fix; the possible gaxios/native-fetch path should stay separate unless it is independently reproduced.
