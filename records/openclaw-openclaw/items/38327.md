## Summary

Keep open: current main and the latest release still gate the Google thinking payload sanitizer only on `google-generative-ai`, while the reported Vertex/Gemini CLI paths can reach the same wrapper with `google-vertex` or `google-gemini-cli`. The open linked pull request https://github.com/openclaw/openclaw/pull/83991 now targets this exact guard gap with closing syntax, so this issue should remain open until that PR lands or is rejected.

## What This Changes

The issue reports a regression where `google-vertex/gemini-3.1-pro-preview` embedded agent runs fail on every message with `Cannot convert undefined or null to object` after upgrading to 2026.3.2.

## Best Possible Solution

Review and land the linked narrow PR that widens the Google thinking payload guard to `google-vertex` and `google-gemini-cli` with regression coverage, then track any remaining gaxios/native-fetch failure separately.

## Reproduction Assessment

Yes, at source level. Current main has a concrete path where `google-vertex` and `google-gemini-cli` can share the Google stream wrapper while bypassing the sanitizer guarded only for `google-generative-ai`; live Vertex credentials were not exercised.

## Solution Assessment

Yes for the wrapper gap. The narrowest maintainable fix is to widen the existing Google-family guard and keep the separate gaxios/native-fetch concern out of this issue unless maintainers explicitly approve that dependency/auth-stack work.
