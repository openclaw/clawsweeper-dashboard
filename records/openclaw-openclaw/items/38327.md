## Summary

Keep open: current main still only sanitizes Google thinking payloads when `model.api` is `google-generative-ai`, while the reported Vertex path can reach the same wrapper as `google-vertex` or `google-gemini-cli`; both focused fix PRs were closed unmerged.

## What This Changes

The issue reports a regression where google-vertex Gemini preview models crash embedded agent runs with `Cannot convert undefined or null to object` after upgrading to OpenClaw 2026.3.2.

## Best Possible Solution

Land a narrow Google API guard widening with wrapper regression tests, then handle the gaxios/native-fetch path as a separate dependency-approved follow-up if it still reproduces.

## Reproduction Assessment

Yes for source reproduction: `google-vertex` is routed through the shared Google wrapper, current main only sanitizes `google-generative-ai`, and tests show Vertex can resolve with `api: "google-gemini-cli"`. I did not run a live Vertex credential repro.

## Solution Assessment

Yes for the wrapper gap: widening the existing Google-family guard to include `google-vertex` and `google-gemini-cli` is the narrow maintainable fix. The gaxios/native-fetch workaround should stay separate because it changes dependency behavior.
