## Summary

Keep open: current main and latest stable release still sanitize Google thinking payloads only for `google-generative-ai`, while the reported `google-vertex` and related `google-gemini-cli` paths share the Google wrapper and remain source-reproducible. The issue is paired with open closing PR https://github.com/openclaw/openclaw/pull/73416, so the right next action is to review, land, or reject that PR rather than close this issue early.

## What This Changes

The issue reports a regression where `google-vertex/gemini-3.1-pro-preview` crashes embedded agent runs with `Cannot convert undefined or null to object` after upgrading to OpenClaw 2026.3.2.

## Best Possible Solution

Review and land the narrow guard widening with regression coverage from https://github.com/openclaw/openclaw/pull/73416, then track any remaining gaxios/native-fetch failure as a separate dependency follow-up if it still reproduces.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a negative Gemini thinking payload through `google-vertex` or `google-gemini-cli` reaches the shared Google wrapper but bypasses sanitization because current main only matches `google-generative-ai`.

## Solution Assessment

Yes for the wrapper gap. Widening the existing Google-family guard with focused wrapper regression tests is the narrow maintainable fix; the possible gaxios/native-fetch path should be split unless separately reproduced.
