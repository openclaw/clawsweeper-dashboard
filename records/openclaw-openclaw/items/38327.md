## Summary

Keep open: current main still limits the Google thinking payload sanitizer to `google-generative-ai`, while the reported Vertex path shares the Google wrapper and can also resolve through `google-vertex` or `google-gemini-cli`. The focused fixes in https://github.com/openclaw/openclaw/pull/33428 and https://github.com/openclaw/openclaw/pull/73416 were closed unmerged, and the issue discussion includes multiple later confirmations plus production analysis for the same Google-family failure path.

## What This Changes

The issue reports that OpenClaw 2026.3.2 and later can fail every embedded-agent message with `Cannot convert undefined or null to object` when using `google-vertex/gemini-3.1-pro-preview`.

## Best Possible Solution

Land a narrow Google-family wrapper guard widening with regression tests for `google-generative-ai`, `google-vertex`, and `google-gemini-cli`, then track any remaining gaxios/native-fetch or OpenAI-compatible Vertex variants separately.

## Reproduction Assessment

Yes, source-reproducible: current main routes the Google-family providers through the shared wrapper, but the wrapper only sanitizes `google-generative-ai` while existing coverage shows Vertex can resolve to `google-gemini-cli`. I did not establish a live Vertex credential repro in this read-only review.

## Solution Assessment

Yes for the wrapper gap: widening the existing Google-family guard is the narrow maintainable fix. Bundling dependency patching or OpenAI-compatible Vertex behavior into the same automated repair would cross a different approval or product boundary.
