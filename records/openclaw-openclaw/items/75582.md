## Summary

Keep open. The PR is aimed at a real Discord/Slack reaction flicker problem, but the current head still has a concrete cleanup bug: normal restore cleanup can delete every known status emoji instead of only reactions this turn added, and the required user-facing changelog entry is still missing.

## What This Changes

The PR changes the shared status reaction controller and related tests so remove-capable status reactions stay visible during an agent turn, duplicate adds are skipped, and cleanup is deferred to final clear or restore paths.

## Best Possible Solution

Keep the shared-controller approach, but make restore cleanup remove only tracked active reactions while preserving the initial emoji, add the required Unreleased Fixes changelog bullet, then rerun the focused controller, Slack lifecycle, and Discord process tests plus the changed gate before automerge.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with the PR head, a normal Discord reply reaches setDone()/restoreInitial(), and restoreInitial() calls a helper that deletes every known status emoji even when only a subset was added in that turn.

## Solution Assessment

No, not at the current head. Deferring cleanup in the shared controller is the right narrow design, but cleanup must be based on activeEmojis rather than knownEmojis to avoid unnecessary Discord delete calls and logs.
