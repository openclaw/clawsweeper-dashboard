## Summary

Keep open. The PR targets a real shared Discord/Slack status-reaction UX issue and the controller-level approach matches the current call sites, but it is still a draft and the user-facing fix is missing the required changelog coverage.

## What This Changes

The PR changes the shared status reaction controller to track active remove-capable emojis, defer removals until `clear()` or `restoreInitial()`, skip duplicate reaction adds, and updates shared and Slack lifecycle tests for the new behavior.

## Best Possible Solution

Keep the shared-controller implementation direction, add the missing Unreleased changelog entry for the Discord/Slack status-reaction flicker fix, then mark the PR ready and land it after the targeted controller, Slack lifecycle, and Discord process tests pass.

Yes. The actual visual flicker would still need live Discord/Slack observation for UI proof.

## Solution Assessment

Mostly yes. Solving this in `src/channels/status-reactions.ts` is the narrow maintainable fix because Discord and Slack share the controller and already centralize cleanup through `clear()`/`restoreInitial()`; the remaining requirement is release-note coverage, not a different product design.
