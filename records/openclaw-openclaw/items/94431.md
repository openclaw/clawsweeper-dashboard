## Summary

Keep open: this is a focused fix for a real shipped CLI regression, but the new hoist helper has one blocking parser edge case where a parent value option with no value is moved before the subcommand and can consume the subcommand or next flag as its value.

## What This Changes

The PR adds a shared lazy-reparse helper that hoists parent or ancestor command options placed after a lazy subcommand, with focused CLI and browser regression coverage.

## Best Possible Solution

Keep the reparse-level parent-option hoist, but make value-option hoisting require an actual value so ambiguous or malformed invocations stay on Commander’s normal error path.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.8 still register `--browser-profile` on the browser parent and reparse lazy subcommands without parent-option hoisting; the PR body also supplies before/after terminal proof for the reported order.

## Solution Assessment

No, not yet. The reparse layer is the right place for the fix, but the helper must not hoist value options when no value follows because Commander will then consume the subcommand or next flag as the parent option value.
