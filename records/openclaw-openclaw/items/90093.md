## Summary

Keep open: current main still has a source-backed path where `openai-chatgpt-responses` can persist encrypted reasoning, preserve it when replay provenance matches, and send it on the next OpenClaw-runtime turn. The issue is valid, but several open PRs already touch this same Responses reasoning replay surface, so this should be coordinated rather than closed or duplicated.

## What This Changes

The issue asks OpenClaw to stop replaying stale `openai-chatgpt-responses` encrypted reasoning in native/OpenClaw-runtime sessions, or to retry safely without it.

## Best Possible Solution

Fold this into one canonical Responses replay fix: native ChatGPT/Codex OpenClaw-runtime requests should drop or safely retry unverifiable encrypted reasoning while preserving custom/proxy replay contracts, with focused regression coverage.

## Reproduction Assessment

Yes, with source-level confidence rather than a live run: the issue's v2026.6.1 steps match current code that persists encrypted reasoning, preserves it when replay metadata matches, and sends it on the next `openai-chatgpt-responses` OpenClaw-runtime request.

## Solution Assessment

The local mitigation is plausible but probably too broad if applied only at generic transcript policy; the best fix should target the native ChatGPT/Codex request/retry path while preserving custom Responses backends that intentionally support encrypted replay.
