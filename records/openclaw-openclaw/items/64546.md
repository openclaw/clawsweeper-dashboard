## Label Justifications

- `P1`: The PR addresses a high-impact Mattermost callback authorization weakness but can break an existing channel interaction workflow if merged as-is.
- `merge-risk: 🚨 compatibility`: The PR changes Mattermost interaction-secret defaults and adds a new account config surface that existing users do not have.
- `merge-risk: 🚨 message-delivery`: Legitimate Mattermost button callbacks can be rejected when signing and verification use different process-local secrets.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root for the plugin-auth Mattermost callback route, which is the route’s authorization boundary.

## Summary

This PR remains useful for the linked Mattermost security issue, but it is not merge-ready because the unconfigured fallback generates a process-local signing secret that can reject legitimate callbacks across CLI/gateway process or restart boundaries.

## What This Changes

The PR adds optional Mattermost `interactions.secret` account config and changes interaction token signing from bot-token-derived HMAC secrets to configured or generated per-account secrets.

## Best Possible Solution

Land a maintainer-approved version that uses a durable per-account Mattermost interaction secret, or explicitly fails closed with setup/doctor/docs guidance, plus regression proof for old-forgery rejection and restart or separate-process verification.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main derives Mattermost interaction signing from the bot token, while PR head generates an unconfigured process-local fallback; I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

No. Removing the bot-token-derived signing root is the right direction, but a process-local fallback is not the best fix because it is not durable across the existing CLI/gateway process boundary or restarts.
