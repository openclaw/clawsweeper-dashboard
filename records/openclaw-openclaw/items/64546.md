## Summary

Keep open: the PR addresses a real Mattermost callback signing vulnerability, but the unconfigured fallback is process-local and the new secret surface lacks an upgrade/setup contract, so existing button callbacks can fail after a restart or across separate CLI and gateway processes.

## What This Changes

The PR replaces Mattermost bot-token-derived interaction HMAC secrets with a configured or generated per-account secret and adds schema/tests for `interactions.secret`.

## Best Possible Solution

Land a maintainer-approved fix that uses a separate durable per-account Mattermost interaction secret, with setup/doctor/docs and tests for fresh install, upgrade or restart, cross-process callbacks, and old-forgery rejection.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main derives Mattermost interaction signing from the bot token, while PR head creates an unconfigured process-local fallback that can diverge across send and gateway lifecycles.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but the current PR is not the best fix until the fallback secret is durable or missing configuration fails closed through an explicit upgrade path.
