## Summary

Keep this PR open. The security direction is valid and the supplied terminal proof shows old bot-token-derived forgery is rejected, but the patch is not merge-ready because the fallback signing secret is process-local and the new user-facing secret surface lacks setup, secret-registry, docs, and upgrade coverage.

## What This Changes

The PR adds Mattermost account-level `interactions.secret` support and switches interactive button HMAC signing from bot-token derivation to configured or generated per-account secrets.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing while either provisioning a durable per-account interaction secret through setup/doctor/secret registry/docs or explicitly failing closed with operator guidance and upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows current `main` still derives Mattermost interaction signing from the bot token with a public HMAC key, and the linked issue gives a concrete forged-token path; I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. Removing the bot-token-derived signing root is the right direction, but a process-local random fallback is not the best fix because it is not durable across the existing send/gateway process boundary or restarts.
