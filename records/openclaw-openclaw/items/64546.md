## Label Justifications

- `P1`: The PR addresses a high-impact Mattermost callback authorization weakness but can break an existing channel interaction workflow if merged as-is.
- `merge-risk: 🚨 compatibility`: The PR changes Mattermost interaction-secret defaults and adds a new account config surface that existing users do not have.
- `merge-risk: 🚨 message-delivery`: Legitimate Mattermost button callbacks can be rejected when signing and verification use different process-local secrets.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root for the plugin-auth Mattermost callback route, which is the route’s authorization boundary.

## Summary

Keep open: the security direction is right and the proof covers the focused crypto path, but the process-local fallback can break legitimate Mattermost button callbacks across CLI/gateway process boundaries or restarts, and the new secret config lacks a maintainer-approved upgrade/setup contract.

## What This Changes

This PR adds an optional Mattermost `interactions.secret` config, resolves it through account config, and changes interaction signing from bot-token-derived HMAC secrets to configured or process-local generated secrets.

## Best Possible Solution

Land a maintainer-approved version that uses a separate durable per-account Mattermost interaction secret, with setup/docs/doctor or explicit fail-closed migration behavior and regression proof for old-forgery rejection plus restart or separate-process verification.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main derives Mattermost interaction signing from the bot token, and PR head generates an unconfigured process-local fallback; I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

No. Removing the bot-token-derived signing root is the right direction, but a process-local fallback is not the best fix because it is not durable across the existing CLI/gateway process boundary or restarts.
