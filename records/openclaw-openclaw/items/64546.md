- `P1`: The PR is for a high-impact Mattermost callback security bug, but the current branch can break existing interactive-button delivery without an upgrade-safe secret path.

## Summary

Keep this PR open: it addresses a real Mattermost interaction-token security bug, but the current patch replaces the old stable bot-token-derived fallback with an unset-config random in-memory secret that can break button verification across send/gateway processes or restarts.

## What This Changes

The PR changes Mattermost interaction signing to use a configured or generated per-account interaction secret, adds `interactions.secret` config parsing, updates Mattermost send/monitor call sites, and adjusts related tests.

## Best Possible Solution

Merge only after the Mattermost interaction secret has a maintainer-approved durable per-account storage or explicit fail-closed upgrade path, with docs/metadata updates and proof that old bot-token-derived forgeries fail while real buttons still verify across process and restart boundaries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no `interactions.secret`, the PR can sign a button in the send path with one process-local random secret while the gateway monitor verifies with another, causing the handler to reject the `_token`.

## Solution Assessment

No. A separate Mattermost interaction secret is the right direction, but the unset-config random fallback is not the best fix because it trades the security bug for broken cross-process and restart behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve a durable interaction secret fallback:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is unset, this creates only a process-local random secret. Mattermost buttons can be signed in the outbound send path and verified later by the gateway monitor, or after a gateway restart; those paths will not share this random value and valid clicks will be rejected as `Invalid token`. The fix needs a durable per-account secret or an explicit fail-closed migration before replacing the stable fallback.
  - confidence: 0.93
- **[P2] Document and surface the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive HMAC secrets from the bot token and the generated config metadata was not updated in the PR. Since the new field is the only durable escape hatch in this branch, operators and external integrations need the docs/metadata to match the chosen upgrade behavior.
  - confidence: 0.86
