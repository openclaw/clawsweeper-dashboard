## Summary

Keep this PR open. Current main still uses the reported bot-token-derived Mattermost interaction secret and the linked security issue is still open, but the PR should not merge as-is because its random in-memory fallback breaks the documented cross-process/restart stability that Mattermost button signing currently depends on.

## What This Changes

The PR adds a Mattermost `interactions.secret` config/type field, resolves it per account, replaces bot-token-derived interaction secrets with configured or generated secrets, and updates Mattermost signing tests plus send/monitor call sites.

## Best Possible Solution

Keep this PR and the linked security issue open, then revise the Mattermost fix around a persistent per-account interaction secret that is not derived from the bot token, works across send/gateway process boundaries and restarts, supports SecretRef or credential-store persistence, updates docs/setup/schema/types, and adds regression coverage for bot-token forgery resistance plus cross-process/restart verification.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: with a known Mattermost bot token, compute the documented derived secret and HMAC over the sorted interaction context, then the handler's `_token` check reaches the accepted-click path before system-event dispatch.

## Solution Assessment

No. The PR moves in the right direction by separating interaction signing from the bot token, but the random in-memory fallback is not the narrowest maintainable fix because it does not persist or synchronize the signing key across the existing CLI/gateway boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve cross-process interaction secret stability:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` generates a fresh process-local secret when `interactions.secret` is unset. The send path signs button payloads before the gateway monitor verifies callbacks, so CLI-created buttons or messages created before a restart can be rejected as `Invalid token`; use a persisted per-account secret or fail configuration instead of a random in-memory fallback.
  - confidence: 0.93
- **[P2] Update the Mattermost HMAC docs for configured secrets:** `extensions/mattermost/src/config-schema-core.ts:112-113`
  - body: Adding `interactions.secret` changes how external scripts must sign Mattermost button contexts, but `docs/channels/mattermost.md` still tells users to derive the secret from the bot token with the public constant. Merging this would leave the public docs both insecure and incorrect for the new contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff is limited to Mattermost code/tests/types and does not add dependency, CI, package, or supply-chain changes, but the proposed signing-key fallback is not a stable security control across process boundaries.

Concerns:

- **[medium] Process-local fallback weakens the interaction-token design:** `extensions/mattermost/src/mattermost/interactions.ts:194`
  - body: When no configured secret exists, the PR creates a random in-memory secret in whichever process calls `ensureInteractionSecret`. Because sending and gateway verification can happen in different processes or across restarts, the security boundary becomes unavailable unless users preconfigure persistence, so the fix needs a persistent or explicit fail-closed contract.
  - confidence: 0.91
