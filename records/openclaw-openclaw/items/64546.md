## Summary

Keep open: the PR addresses a real Mattermost callback-token forgery direction, but the current branch still has merge-blocking security and compatibility issues around secret persistence, docs, and inherited interaction settings.

## What This Changes

The branch adds Mattermost `interactions.secret` config and changes interactive button token signing from bot-token-derived HMACs to configured or generated per-account secrets.

## Best Possible Solution

Use a non-bot-token per-account interaction signing secret that is durable across senders, the gateway, and restarts, then document and test configured, unconfigured, upgrade, and multi-account inheritance behavior.

## Reproduction Assessment

Yes. The reported vulnerability is source-reproducible on current main, and the PR regression is source-reproducible because unconfigured send and monitor paths independently generate process-local random secrets.

## Solution Assessment

No. A separate signing secret is the right direction, but this PR's random in-memory fallback and undocumented config contract are not the best merge shape for a supported Mattermost security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep fallback interaction secrets stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a process-local random secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after a separate `ensureInteractionSecret` call, so CLI-sent buttons or post-restart callbacks can be rejected as `Invalid token`.
  - confidence: 0.91
- **[P2] Document the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive the HMAC secret from the bot token and public key string. After merge those scripts will generate rejected `_token` values, and operators will not know how to configure or rotate the new secret.
  - confidence: 0.86
- **[P2] Merge interaction overrides with inherited callback settings:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Adding the secret can silently drop callback reachability or allowlist settings for that account.
  - confidence: 0.82
