## Label Justifications

- `P1`: The PR addresses a high-impact Mattermost callback-token forgery path but can also break live button workflows if the replacement secret is not durable.
- `merge-risk: 🚨 compatibility`: Existing deployments without a configured interaction secret may start rejecting valid callbacks after restart or split-process sends.
- `merge-risk: 🚨 message-delivery`: Mattermost button callbacks can be rejected as invalid when the sender and verifier do not share the generated fallback secret.
- `merge-risk: 🚨 security-boundary`: The patch changes the credential and HMAC boundary that authorizes Mattermost interactive button callbacks.

## Summary

Keep open: the security fix is still necessary because current main and v2026.6.11 still use the public bot-token-derived HMAC path, but this PR remains blocked by the process-local unset-secret fallback, missing supported credential surfaces, and a dirty merge state.

## What This Changes

The PR changes Mattermost interactive button token signing from bot-token-derived HMAC secrets to configured or generated per-account interaction secrets, with schema/types and tests for the new secret path.

## Best Possible Solution

Choose an upgrade-safe Mattermost interaction-secret model, then wire setup, SecretRef/reference docs, generated metadata, restart or split-process coverage, and a clean branch refresh before merge.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.11 derive the Mattermost interaction HMAC secret from the bot token with a public key, and the PR's unset-secret regression is source-reproducible because generated secrets are process-local.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but an in-memory random fallback plus hidden config is not the best upgrade-safe solution for a credential shared by sender and callback verifier.
