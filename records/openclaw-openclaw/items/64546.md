## Summary

Keep open: the branch addresses the Mattermost token-forgery direction, but it is not merge-ready because unset deployments can lose valid button callbacks across restarts or split sender/verifier processes, the new credential is not wired through setup/SecretRef/docs surfaces, and GitHub reports the PR as conflicting.

## What This Changes

The PR adds a Mattermost `interactions.secret` config field and switches interaction HMAC signing from bot-token-derived secrets to configured or generated per-account secrets.

## Best Possible Solution

Implement a durable or explicitly fail-closed Mattermost interaction-secret model, then wire setup, SecretRef registry, docs, restart/split-process tests, and proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main derives the Mattermost interaction HMAC secret from the bot token with a public key, and the PR's unset-secret regression is source-reproducible because generated fallback secrets are process-local.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but an in-memory random fallback plus hidden config is not the best upgrade-safe solution for a credential shared by sender and callback verifier.

  - body: When `interactions.secret` is unset, this generates a fresh module-local secret. `sendMessageMattermost` can sign buttons in a different process from the gateway callback verifier, and any gateway restart also loses the generated value, so existing configs can start returning `Invalid token` for legitimate button clicks. Please either provision a durable shared secret or fail closed with explicit setup/upgrade guidance and coverage.
  - confidence: 0.93
