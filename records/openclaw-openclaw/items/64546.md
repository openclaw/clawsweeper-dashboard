## Label Justifications

- `P1`: This PR fixes a real Mattermost callback authorization bug, but the current patch can break live button workflows and changes a security boundary.
- `merge-risk: 🚨 compatibility`: Merging as-is can require users or external scripts to add a new `interactions.secret` or lose callback verification compatibility.
- `merge-risk: 🚨 message-delivery`: Buttons signed in one process can be rejected by a different gateway verifier process, dropping the intended button-click action.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root that protects Mattermost interaction callback authorization.

## Summary

Keep open. The PR moves Mattermost interaction signing away from the forgeable bot-token-derived HMAC, but it still has security/compatibility blockers around process-local fallback secrets, undocumented upgrade behavior, and interaction config inheritance.

## What This Changes

The PR adds a Mattermost `interactions.secret` SecretRef config option and replaces bot-token-derived HMAC interaction secrets with configured or generated per-account secrets in send and monitor paths.

## Best Possible Solution

Use a non-bot-token per-account interaction signing secret that stays stable across button producers, the gateway verifier, and restarts, then document and test fresh-install, upgrade, and external-script behavior.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: Mattermost derives the verifier secret from a public HMAC key and the bot token, and `verifyInteractionToken` gates callback dispatch on that value.

## Solution Assessment

No. A separate signing secret is the right direction, but this PR's random in-memory fallback and undocumented config surface are not the best fix for a callback verifier that must work across senders, the gateway, and restarts.

Full review comments:

- **[P1] Keep fallback interaction secrets stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a process-local random secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after a separate `ensureInteractionSecret` call, so CLI-sent buttons or post-restart callbacks can be rejected as `Invalid token`.
  - confidence: 0.91
- **[P2] Document the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive the HMAC secret from the bot token and public key string. After merge those scripts will generate rejected `_token` values, and operators will not know how to configure or rotate the new secret.
  - confidence: 0.86
- **[P2] Merge interaction overrides with inherited callback settings:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Adding the secret can silently drop callback reachability or allowlist settings for that account.
  - confidence: 0.82
