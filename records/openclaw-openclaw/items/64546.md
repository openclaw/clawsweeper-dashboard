## Summary

Keep open. Current main still has the Mattermost bot-token-derived HMAC issue, and this PR targets the right boundary, but the proposed transient fallback breaks existing cross-process/restart interaction verification and leaves config/docs contract gaps.

## What This Changes

The PR adds a Mattermost `interactions.secret` config/type field, resolves it per account, replaces bot-token-derived interaction secrets with configured or generated secrets, and updates Mattermost signing tests plus send/monitor call sites.

## Best Possible Solution

Revise the Mattermost fix around a durable per-account interaction secret that is not derived from the bot token, works across send/gateway processes and restarts, preserves callback URL config merging, and updates docs/setup/schema/types plus regression tests.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: with a known Mattermost bot token, compute the documented derived secret, sign the sorted interaction context, and the handler's `_token` check reaches the accepted-click path before system-event dispatch.

## Solution Assessment

No. The PR separates interaction signing from the bot token, but a random in-memory fallback is not the narrowest maintainable fix because the existing design requires a shared signing secret across process boundaries and restarts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve cross-process interaction secret stability:** `extensions/mattermost/src/mattermost/interactions.ts:193-195`
  - body: `ensureInteractionSecret` generates a fresh process-local secret when `interactions.secret` is unset. The send path signs button payloads before the gateway monitor verifies callbacks, so CLI-created buttons or messages created before a restart can be rejected as `Invalid token`; use a persisted per-account secret or fail configuration instead of a random in-memory fallback.
  - confidence: 0.93
- **[P2] Keep interaction config fields merged:** `extensions/mattermost/src/mattermost/accounts.ts:100-105`
  - body: Resolving an account-level `interactions.secret` from the merged config currently lets that account's `interactions` object replace a top-level `interactions.callbackBaseUrl`, because the account merge only deep-merges `commands`. A common setup with a shared reachable callback URL plus per-account secrets would silently fall back to the default gateway URL and break button clicks.
  - confidence: 0.87
- **[P2] Update the Mattermost HMAC docs for configured secrets:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes how external scripts must sign Mattermost button contexts, but `docs/channels/mattermost.md` still tells users to derive the secret from the bot token with the public constant. Merging this would leave the public docs both insecure and incorrect for the new contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff is limited to Mattermost code/tests/types and does not add dependency or CI supply-chain changes, but the proposed signing-key fallback is not a stable security control across process boundaries.

Concerns:

- **[medium] Process-local fallback weakens the interaction-token design:** `extensions/mattermost/src/mattermost/interactions.ts:193`
  - body: When no configured secret exists, the PR creates a random in-memory secret in whichever process calls `ensureInteractionSecret`. Because sending and gateway verification can happen in different processes or across restarts, the security boundary becomes unavailable unless users preconfigure persistence, so the fix needs a persistent or explicit fail-closed contract.
  - confidence: 0.91
