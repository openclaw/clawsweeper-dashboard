## Summary

Keep open. The PR targets a real Mattermost callback-signing security bug and the after-fix proof is useful, but the branch still needs maintainer security/compatibility decisions for durable secret persistence, restart/cross-process behavior, and the documented external-script token contract before merge.

## What This Changes

The PR changes Mattermost interaction signing from a bot-token-derived HMAC secret to a configured or generated per-account interaction secret.

## Best Possible Solution

Use a separate per-account Mattermost interaction signing secret that is stable across senders and the gateway without deriving from the bot token, then update docs and tests for fresh installs, existing configs, external scripts, and restart/cross-process delivery.

## Reproduction Assessment

Yes, at source level. Current main derives the Mattermost interaction signing secret from the bot token with a public HMAC key, and the PR proof demonstrates that old bot-token-derived forged tokens no longer verify against the new generated secret.

## Solution Assessment

No. A separate interaction signing secret is the right direction, but the PR's random in-memory fallback is not the best fix because Mattermost senders and the gateway verifier can run in different processes; the fix needs a persisted or explicitly configured secret contract with upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the fallback secret stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a fresh module-local secret. `sendMessageMattermost` can sign button props in one process while the gateway monitor verifies callbacks in another or after restart, so CLI-sent buttons and older messages can be rejected as `Invalid token`; use a persisted/shared secret or an explicit configured-secret requirement before removing the bot-token-derived fallback.
  - confidence: 0.91
- **[P2] Document the new signing secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` changes the public token-generation contract, but the Mattermost docs still tell external scripts to derive the signing secret from the bot token and public HMAC string. After this merge those scripts will generate rejected `_token` values, and the public security guidance will describe the retired scheme.
  - confidence: 0.86
- **[P2] Merge interaction config before reading the secret:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces channel-level `interactions.callbackBaseUrl` and `allowedSourceIps` for that account. Operators adding a per-account secret can silently lose their callback URL or allowlist and break button delivery; include `interactions` in the nested merge and cover the mixed top-level/per-account case.
  - confidence: 0.82
