## Label Justifications

- `P1`: The PR addresses a real Mattermost callback forgery issue and can also break real button callback workflows if merged as-is.
- `merge-risk: 🚨 compatibility`: Existing Mattermost configs without the new secret can change from stable bot-token-derived signing to process-local random signing.
- `merge-risk: 🚨 message-delivery`: Invalid `_token` verification would suppress legitimate Mattermost button-click delivery into the agent workflow.
- `merge-risk: 🚨 security-boundary`: The patch changes the credential used as the sole HMAC authorization check for Mattermost interaction callbacks.

## Summary

Keep open: the vulnerability is real and the core rejection proof is useful, but the PR is not merge-ready because the fallback signing secret is process-local and the new credential is not wired through setup, secret-contract, or docs surfaces.

## What This Changes

The PR changes Mattermost interaction signing to use a configured or generated per-account secret instead of deriving the signing secret from the Mattermost bot token.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing while making the replacement Mattermost interaction secret durable or explicitly required across setup, secret contracts, docs, upgrade behavior, and restart or split-process proof.

## Reproduction Assessment

Yes, source-level. Current main derives the Mattermost interaction signing secret from a bot token using a public HMAC key, and PR-head source shows the fallback replacement is process-local when no configured secret exists.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but the patch is not the best merge shape until durable provisioning or explicit fail-closed setup/docs behavior is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Make the fallback interaction secret durable:** `extensions/mattermost/src/mattermost/interactions.ts:198-199`
  - body: When `interactions.secret` is unset, this generates a random value held only in the current process. Mattermost button payloads can be signed in the send path and verified later by the gateway monitor or after a restart, so existing configs can start rejecting valid clicks as invalid tokens. Persist the secret or fail closed with an explicit setup requirement before signing buttons.
  - confidence: 0.92
- **[P1] Wire the new signing secret through setup and docs:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` to the schema creates a new credential boundary, but the Mattermost secret contract, setup flow, generated config metadata/help, and docs still only cover the bot token and the old bot-token-derived HMAC. Operators need a supported way to provision this durable secret and external integrations need updated signing guidance before this can merge.
  - confidence: 0.9
