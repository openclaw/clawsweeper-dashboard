## Summary

Keep this PR open: it addresses a real Mattermost callback-token security bug, but the current branch is not merge-ready because its unconfigured random fallback is process-local and GitHub reports the head as conflicting with current main.

## What This Changes

The branch adds account-level Mattermost `interactions.secret` support and switches button HMAC signing from bot-token derivation to configured or generated per-account secrets.

## Best Possible Solution

Land a maintainer-approved fix that uses a separate durable per-account Mattermost interaction secret, with setup/secret registry/docs/doctor or explicit fail-closed upgrade behavior plus proof for upgrade, restart or separate-process callbacks, and old-forgery rejection.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 still derive Mattermost interaction signing from the bot token with a public HMAC key, and PR head replaces that with configured or process-local generated secrets; I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

No. Removing the bot-token-derived signing root is the right direction, but process-local generation is not the best fix because it is not durable across the existing CLI/gateway process boundary or restarts.
