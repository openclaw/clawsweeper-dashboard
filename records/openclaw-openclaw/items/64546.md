## Label Justifications

- `P1`: This PR targets a high-impact Mattermost callback security bug but currently risks breaking existing interactive button delivery.
- `merge-risk: 🚨 compatibility`: The default signing secret changes from a deterministic shipped behavior to a process-local random value unless operators add new config.
- `merge-risk: 🚨 message-delivery`: Mattermost button clicks can be rejected as invalid when a message is signed in one process and verified in another.
- `merge-risk: 🚨 security-boundary`: The changed secret is the authorization boundary for Mattermost interaction callbacks registered with plugin auth.

## Summary

Keep open. The PR addresses the Mattermost token-forgery direction, but its unconfigured default creates a process-local random signing secret that can make valid Mattermost button clicks fail across the CLI/gateway process boundary or after restart.

## What This Changes

The PR replaces Mattermost interaction secrets derived from bot tokens with generated or configured per-account secrets and adds an `interactions.secret` config surface plus focused tests.

## Best Possible Solution

Use a separate durable per-account interaction signing secret with an explicit upgrade and storage path, then update Mattermost docs, generated config metadata/help, and tests for fresh installs, upgrades, external scripts, and cross-process button delivery.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `ensureInteractionSecret` creates a random in-memory secret when config is absent, while send and monitor paths call it independently for signing and verification.

## Solution Assessment

No. The security direction is right, but the current fallback is only process-local; the maintainable fix needs a durable per-account secret lifecycle plus docs/generated metadata and upgrade proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve cross-process interaction secrets:** `extensions/mattermost/src/mattermost/interactions.ts:198-199`
  - body: When `interactions.secret` is unset, this now generates a process-local random secret. `sendMessageMattermost` signs button payloads in the outbound send path, while the gateway monitor verifies callbacks from its own process and also calls `ensureInteractionSecret`; those two processes can therefore disagree and reject otherwise valid button clicks as `Invalid token` after process separation or restart.
  - confidence: 0.94
- **[P2] Document the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` changes the public signing contract, but the PR leaves the Mattermost external-script docs on the old bot-token-derived HMAC flow and does not refresh the generated config/help surface. Users following the docs will either create rejected tokens under the new runtime or keep using the insecure derivation this PR is trying to retire.
  - confidence: 0.84
