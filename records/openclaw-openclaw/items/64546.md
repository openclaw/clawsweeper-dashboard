## Summary

Keep this PR open: it addresses a real Mattermost interaction-token security flaw, but the current patch can reject legitimate button callbacks when no durable `interactions.secret` is configured, and the new config surface is not wired through setup, metadata, or docs.

## What This Changes

The PR replaces Mattermost bot-token-derived interaction signing with configured or generated per-account secrets and updates Mattermost config, tests, and send/monitor wiring.

## Best Possible Solution

Land a refreshed Mattermost security fix that removes bot-token-derived signing while using a durable per-account interaction secret, or an explicit fail-closed setup path, with setup/docs/metadata and upgrade/restart proof covered before merge.

## Reproduction Assessment

Yes, source-level. Current main derives Mattermost interaction signing from the bot token with a public HMAC key and verifies callback `_token` values against that derived secret; I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No, not as currently proposed. Removing bot-token-derived signing is the right direction, but a process-local random fallback is not durable across the existing send/gateway boundary or restarts.
