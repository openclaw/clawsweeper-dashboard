## Summary

Keep open. The PR targets a real Mattermost signing vulnerability, but the proposed fix still has blocking correctness/security gaps: the random fallback is process-local, account interaction config is not merged safely, public docs still describe the old bot-token derivation, and the contributor has not provided real Mattermost behavior proof.

## What This Changes

The PR adds Mattermost `interactions.secret` config support, resolves it per account, replaces bot-token-derived HMAC secrets with configured or generated secrets, and updates Mattermost tests/types/send/monitor call sites.

## Best Possible Solution

Revise the fix around a durable per-account Mattermost interaction signing secret independent of the bot token, stable across send/gateway processes and restarts, with config merging, docs, and regression coverage updated together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a Mattermost bot token and known context, compute the documented derived secret and token HMAC, and current main's `verifyInteractionToken` can accept it before dispatch.

## Solution Assessment

No. The PR addresses the right root problem, but a random process-local fallback is not a stable signing control, and the account config merge plus public HMAC docs need to be fixed in the same change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve cross-process interaction secret stability:** `extensions/mattermost/src/mattermost/interactions.ts:193-195`
  - body: `ensureInteractionSecret` generates a fresh in-memory secret when `interactions.secret` is unset. The send path can sign buttons in a different process from the gateway verifier, or before a restart, so valid clicks can be rejected as `Invalid token`; use a durable per-account secret or fail configuration instead of a random process-local fallback.
  - confidence: 0.93
- **[P2] Keep interaction config fields merged:** `extensions/mattermost/src/mattermost/accounts.ts:100-105`
  - body: Resolving an account-level `interactions.secret` from the merged config lets that account's `interactions` object replace a shared `interactions.callbackBaseUrl`, because the account merge only deep-merges `commands`. A shared callback URL plus per-account secrets would silently fall back to the default gateway URL and break button clicks.
  - confidence: 0.87
- **[P2] Update the Mattermost HMAC docs for configured secrets:** `extensions/mattermost/src/config-schema-core.ts:148`
  - body: Adding `interactions.secret` changes how external scripts must sign Mattermost button contexts, but `docs/channels/mattermost.md` still tells users to derive the secret from the bot token with the public constant. Merging this would leave the public docs both insecure and incorrect for the new contract.
  - confidence: 0.86

## Security Review
