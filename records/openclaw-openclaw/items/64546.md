## Label Justifications

- `P1`: The PR addresses a security-sensitive Mattermost interaction signing bypass but still needs design and proof before merge.
- `merge-risk: 🚨 compatibility`: The process-local fallback can break existing Mattermost button setups that do not configure a new interaction secret.
- `merge-risk: 🚨 message-delivery`: Valid Mattermost button clicks can be rejected when the sender and gateway verifier do not share the generated in-memory secret.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing boundary while leaving stale public signing instructions and an unsettled fallback contract.

## Summary

Keep open for maintainer security review: current main still has the vulnerable deterministic Mattermost interaction signing path, but this PR can break valid button clicks by using a process-local fallback secret, leaves the public bot-token HMAC recipe in docs, and lacks real behavior proof.

## What This Changes

The PR adds Mattermost `interactions.secret` config support and replaces bot-token-derived interaction HMAC secrets with configured or randomly generated per-account secrets in send and monitor paths plus tests.

## Best Possible Solution

Land or replace this with a reviewed Mattermost security fix that uses a durable per-account signing secret independent of the bot token, or an explicit fail-closed `interactions.secret` contract, with docs, upgrade tests, and real Mattermost proof together.

## Reproduction Assessment

Yes. Source inspection of current main shows `deriveInteractionSecret` uses a public constant with the bot token, and the verifier accepts tokens generated from that derived secret; I did not run a live Mattermost exploit.

## Solution Assessment

No. Separating interaction signing from the bot token is the right direction, but the process-local fallback, shallow interaction config merge, stale docs, and missing real behavior proof make this PR unsafe to merge as-is.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use a durable interaction signing secret:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` creates a random process-local secret when `interactions.secret` is unset. The send path can sign buttons in one process while the gateway verifies them in another, or verification can happen after a restart, so valid clicks can be rejected as invalid; use a durable per-account secret or an explicit fail-closed contract instead.
  - confidence: 0.93
- **[P2] Preserve shared interaction config when reading secrets:** `extensions/mattermost/src/mattermost/accounts.ts:103-108`
  - body: The new `merged.interactions?.secret` read relies on account merging that still deep-merges only `commands`. If shared `interactions.callbackBaseUrl` is paired with an account-level `interactions.secret`, the account object replaces the shared callback URL and source-IP settings, so button callbacks can use the wrong gateway URL or allowlist.
  - confidence: 0.87
- **[P2] Update the Mattermost signing docs:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes the external signing contract, but the Mattermost docs still instruct scripts to derive the signing key from the bot token with the public constant. Merging this leaves users with an insecure and incorrect recipe for the new verifier contract.
  - confidence: 0.86
- **[P3] Fix the stale interaction-secret error:** `extensions/mattermost/src/mattermost/interactions.ts:213`
  - body: After this API change, the error still tells callers to use `setInteractionSecret(accountId, botToken)`. Following that text would seed the verifier with the wrong value instead of the configured interaction signing secret.
  - confidence: 0.78
