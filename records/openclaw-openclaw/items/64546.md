## Summary

Keep open: the PR is pointed at a real Mattermost callback-signing security flaw, but it is not merge-ready because the replacement fallback secret is process-local, the new credential is not wired through setup/secret-contract/docs surfaces, and GitHub currently reports the branch as conflicting.

## What This Changes

The PR adds a Mattermost `interactions.secret` config field and switches interaction token signing from bot-token-derived HMAC secrets to configured or generated per-account secrets, with related runtime and tests.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing and either provisions a durable per-account Mattermost interaction secret through setup, secret contracts, docs, and tests, or explicitly fails closed with clear upgrade behavior and proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 derive the Mattermost interaction signing secret from the bot token with a public HMAC key and verify callback `_token` values against that derived secret. I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but this is not the best merge shape until the replacement secret is durable or explicitly required across setup, docs, upgrade, and restart behavior.
