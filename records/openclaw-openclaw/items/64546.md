## Summary

Keep open. This PR targets a real Mattermost interaction-token forgery bug, but the proposed patch still has blocking signing-secret stability, config-merge, docs, and real-behavior-proof gaps.

## What This Changes

Adds Mattermost `interactions.secret` config support and replaces bot-token-derived interaction HMAC secrets with configured or generated per-account secrets in send and monitor paths, with related schema, type, and test updates.

## Best Possible Solution

Land a reviewed security fix that separates Mattermost interaction signing from the bot token with a durable per-account secret, preserves config merging, updates schema/types/docs/tests together, and includes redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a Mattermost bot token and known interaction context, compute the documented derived secret and `_token` HMAC, and current main's verifier will accept it before the handler proceeds.

## Solution Assessment

No. Separating interaction signing from the bot token is the right direction, but this implementation uses an unstable process-local fallback and misses config-merge, docs, and real-behavior-proof follow-through.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use a durable interaction signing secret:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` creates a random process-local secret when `interactions.secret` is unset. The send path can sign buttons in one process while the gateway verifies in another, or verification can happen after a restart, so valid clicks can be rejected as invalid; use a durable per-account secret or fail closed instead of an in-memory fallback.
  - confidence: 0.93
- **[P2] Preserve shared interaction config when reading secrets:** `extensions/mattermost/src/mattermost/accounts.ts:103-108`
  - body: The new `merged.interactions?.secret` read relies on a merge helper that still deep-merges only `commands`. If a shared `interactions.callbackBaseUrl` is paired with an account-level `interactions.secret`, the account object replaces the shared callback URL and button callbacks can fall back to the wrong gateway URL.
  - confidence: 0.87
- **[P2] Update the Mattermost signing docs:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes the external signing contract, but the Mattermost docs still instruct scripts to derive the signing secret from the bot token with the public constant. Merging this leaves users with an insecure and incorrect recipe for the new verifier contract.
  - confidence: 0.86
- **[P3] Fix the stale interaction-secret error:** `extensions/mattermost/src/mattermost/interactions.ts:213`
  - body: After this API change, the error still tells callers to use `setInteractionSecret(accountId, botToken)`. Following that text would seed the verifier with the wrong value instead of the configured interaction signing secret.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no supply-chain surface, but the security fix is incomplete because the fallback secret is process-local and public docs retain the old forgeable signing recipe.

Concerns:

- **[medium] Process-local fallback weakens the token design:** `extensions/mattermost/src/mattermost/interactions.ts:192`
  - body: When no configured secret exists, the PR creates a random in-memory secret in whichever process calls `ensureInteractionSecret`; send and gateway verification can run in different processes or across restarts, so the fix needs a persistent or explicit fail-closed contract.
  - confidence: 0.91
- **[medium] Public docs keep the old forgeable signing recipe:** `docs/channels/mattermost.md:435`
  - body: The PR adds a new interaction secret config field but does not update the Mattermost HMAC docs, leaving users with instructions that still derive the signing key from the bot token and public constant.
  - confidence: 0.84
