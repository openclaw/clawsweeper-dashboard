## Summary

## Best Possible Solution

Land a maintainer-approved fix that uses a separate durable per-account interaction secret, preserves or explicitly migrates upgrade behavior, updates metadata/docs/tests, and proves fresh-install plus restart/cross-process Mattermost button callbacks.

## Solution Assessment

No. A separate per-account secret is the right direction, but this implementation is not the best mergeable fix until the default secret is durable and the config/docs/generated/inheritance surfaces are aligned.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve a durable interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is unset, this path creates only a process-local random secret. `sendMessageMattermost` signs button payloads in the send path while the gateway monitor verifies callbacks later and can run in another process or after a restart, so existing no-extra-config buttons can start returning 403 `Invalid token`; keep a durable shared secret path or require and migrate an explicit configured secret before enabling buttons.
  - confidence: 0.91
- **[P2] Regenerate the bundled config metadata:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` only to the source schema leaves the checked-in generated Mattermost schema without that property, while canonical config validation consumes `GENERATED_BUNDLED_CHANNEL_CONFIG_METADATA`. A user following the new mitigation can have the SecretRef rejected or hidden from config tooling until the generated metadata is updated.
  - confidence: 0.87
- **[P2] Deep-merge interaction account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` still deep-merges only `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Multi-account users adding the secret can silently fall back to localhost or lose their callback source allowlist, which breaks button callbacks in the same path this PR is trying to secure.
  - confidence: 0.84
- **[P2] Update the external-script HMAC contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This PR changes the signing secret away from bot-token derivation, but the Mattermost docs at the PR head still tell external scripts to derive the secret from the bot token and say random bytes are wrong. Those examples will generate `_token` values the gateway rejects after merge, so the docs need the new configured-secret contract and migration guidance.
  - confidence: 0.86
