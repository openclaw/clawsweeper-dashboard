## Summary

Keep open. This PR addresses a real Mattermost callback-auth security bug, but it is not merge-ready because the no-config fallback now creates process-local random secrets that can break existing button callbacks, and the new `interactions.secret` contract is not fully wired through config metadata, docs, or inherited account settings.

## What This Changes

Replaces Mattermost interaction signing with generated or configured per-account secrets, adds `interactions.secret` config support, and updates Mattermost account/send/monitor tests.

## Best Possible Solution

Land a maintainer-approved fix that uses a separate durable per-account interaction secret, preserves or explicitly migrates upgrade behavior, updates generated metadata/docs/tests, and proves fresh install plus restart/cross-process Mattermost button callbacks.

## Reproduction Assessment

Yes, source-reproducible. Current main derives the Mattermost interaction signing root from the bot token with a public HMAC key, and the PR body includes terminal output showing the old bot-token-derived forged token no longer verifies after the patch.

## Solution Assessment

No. A separate per-account secret is the right direction, but this implementation is not the best mergeable fix until the default secret is durable and the config/docs/generated/inheritance surfaces are aligned.

Full review comments:

- **[P1] Preserve a durable default interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: `ensureInteractionSecret` now falls back to a new process-local random value when `interactions.secret` is unset. `sendMessageMattermost` signs button payloads before posting, while the gateway monitor verifies callbacks later and can run in a different process or after a restart; those paths no longer share a secret, so existing no-extra-config buttons can start returning 403 `Invalid token`. Keep a durable shared secret path, or require and migrate a configured secret before enabling buttons.
  - confidence: 0.91
- **[P2] Regenerate the bundled config metadata:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` only to the extension schema leaves the checked-in generated Mattermost schema without that property, while canonical config validation consumes `GENERATED_BUNDLED_CHANNEL_CONFIG_METADATA`. A user following the new mitigation can have the SecretRef rejected or hidden from config tooling until the generated metadata is updated.
  - confidence: 0.87
- **[P2] Deep-merge interaction account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` still deep-merges only `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Multi-account users adding the secret can silently fall back to localhost or lose their callback source allowlist, which breaks button callbacks in the same path this PR is trying to secure.
  - confidence: 0.84
- **[P2] Update the external-script HMAC contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This PR changes the signing secret away from bot-token derivation, but the Mattermost docs at the PR head still tell external scripts to derive the secret from the bot token and say no configuration is needed. Those examples will generate `_token` values the gateway rejects after merge, so the docs need the new configured-secret contract and migration guidance.
  - confidence: 0.86
