## Summary

Keep open: the PR addresses a real Mattermost callback-signing security bug, but the current patch replaces the shipped shared secret behavior with a process-local random fallback when `interactions.secret` is unset, which can break valid CLI/gateway button verification and restart compatibility.

## What This Changes

The branch adds a Mattermost `interactions.secret` SecretRef and changes interactive-button HMAC signing to use configured or generated per-account secrets instead of bot-token-derived secrets.

## Best Possible Solution

Use a durable per-account Mattermost interaction secret shared by CLI and gateway, migrate or generate it for existing accounts, reject old bot-token-derived forgeries, and cover configured, unconfigured, restart, and separate-process button flows in tests and proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the unconfigured path generates a random in-memory secret, while the sender and gateway verifier call that helper independently; I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. A separate Mattermost interaction signing secret is the right security direction, but the current process-local fallback is not the best fix for a CLI/gateway channel because it is not durable or upgrade-safe.
