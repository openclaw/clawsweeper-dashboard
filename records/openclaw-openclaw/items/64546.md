## Summary

Keep open: current main still has the Mattermost HMAC issue, but this PR is not merge-ready because its fallback is process-local, the config/docs contract is incomplete, and contributor proof is absent.

## What This Changes

The PR adds Mattermost `interactions.secret` config support and replaces bot-token-derived interaction HMAC secrets with configured or generated secrets in Mattermost send/monitor paths.

## Best Possible Solution

Land a reviewed security fix that uses a durable per-account Mattermost interaction signing secret independent of the bot token, preserves account callback config, and updates schema/types/docs/tests together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a Mattermost bot token and known interaction context, compute the documented derived secret and token HMAC, and current main's `verifyInteractionToken` can accept it before dispatch.

## Solution Assessment

No. The PR targets the right root problem, but the random process-local fallback is not a stable signing control, and the account config merge plus public HMAC docs need to be fixed in the same change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve cross-process interaction secret stability:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` generates and stores a random per-process secret when `interactions.secret` is unset. The send path can sign buttons in a different process from the gateway verifier, or before a restart, so valid clicks can be rejected as `Invalid token`; use a durable per-account secret or fail configuration instead of a random process-local fallback.
  - confidence: 0.93
- **[P2] Deep-merge interaction config before reading secrets:** `extensions/mattermost/src/mattermost/accounts.ts:103-108`
  - body: Resolving an account-level `interactions.secret` from the merged config lets that account's `interactions` object replace a shared `interactions.callbackBaseUrl`, because the account merge only deep-merges `commands`. A shared callback URL plus per-account secrets would silently fall back to the default gateway URL and break button clicks.
  - confidence: 0.87
- **[P2] Update Mattermost HMAC docs for configured secrets:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes how external scripts must sign Mattermost button contexts, but `docs/channels/mattermost.md` still tells users to derive the secret from the bot token with the public constant. Merging this would leave the public docs both insecure and incorrect for the new contract.
  - confidence: 0.86
- **[P3] Update the interaction-secret initialization error:** `extensions/mattermost/src/mattermost/interactions.ts:213`
  - body: The error still tells callers to use `setInteractionSecret(accountId, botToken)`, but this PR changes that second argument to a raw secret. Following the message would seed the verifier with the wrong value instead of the configured interaction secret.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no supply-chain surface, but the generated process-local fallback is not a stable security control for the Mattermost interaction signing boundary.

Concerns:

- **[medium] Process-local fallback weakens interaction-token design:** `extensions/mattermost/src/mattermost/interactions.ts:192`
  - body: When no configured secret exists, the PR creates a random in-memory secret in whichever process calls `ensureInteractionSecret`; send and gateway verification can run in different processes or across restarts, so the fix needs a persistent or explicit fail-closed contract.
