## Label Justifications

- `P1`: This PR addresses a real Mattermost callback authorization bug but currently risks breaking live interactive button workflows.
- `merge-risk: 🚨 compatibility`: Existing users and external scripts may need a new `interactions.secret` or changed signing procedure after merge.
- `merge-risk: 🚨 message-delivery`: Buttons signed in one process can be rejected by the gateway verifier in another process when no explicit secret is configured.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root that protects Mattermost interaction callback authorization.

## Summary

Keep open: this PR fixes a real Mattermost callback-token forgery primitive in the right general direction, but it is not merge-ready because the unconfigured fallback signing secret becomes process-local and the new operator-facing secret contract lacks upgrade/docs coverage.

## What This Changes

The PR adds a Mattermost `interactions.secret` SecretInput and replaces bot-token-derived HMAC interaction signing with configured or generated per-account secrets across send and monitor paths, with focused tests.

## Best Possible Solution

Use a separate per-account interaction signing secret that is durable or explicitly required across senders, gateway restarts, and upgrades, then document the contract and preserve inherited interaction settings.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main and v2026.6.6: Mattermost derives the interaction verifier secret from the bot token with a public HMAC key, and the callback handler gates dispatch on that token.

## Solution Assessment

No. A separate signing secret is the right direction, but this PR's random in-memory fallback is not the best fix because Mattermost senders and the gateway verifier can run in different processes and across restarts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep fallback interaction secrets stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a process-local random secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after a separate initialization, so CLI-sent buttons or post-restart callbacks can be rejected as `Invalid token`.
  - confidence: 0.91
- **[P2] Document the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive the HMAC secret from the bot token and public key string. After merge those scripts will generate rejected `_token` values, and operators will not know how to configure or rotate the new secret.
  - confidence: 0.86
- **[P2] Merge interaction overrides with inherited settings:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Adding the secret can silently drop callback reachability or allowlist settings for that account.
  - confidence: 0.82
