## Summary

Keep open. The PR addresses a real Mattermost callback security bug and the contributor supplied useful runtime proof, but the patch still changes the signing-secret contract in a way that can break existing button delivery and needs maintainer security/compatibility design before merge.

## What This Changes

The PR adds Mattermost `interactions.secret` config and changes button-token signing from bot-token-derived HMACs to configured or generated per-account interaction secrets.

## Best Possible Solution

Use a non-bot-token per-account interaction signing secret that stays stable across senders, the gateway, and restarts, then document and test the upgrade path for fresh installs, existing configs, and external scripts.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: Mattermost derives the verifier secret from a public HMAC key and the bot token, and `verifyInteractionToken` gates callback dispatch on that value.

## Solution Assessment

No. A separate signing secret is the right direction, but this PR's random in-memory fallback is not the best fix because Mattermost senders and the gateway verifier can run in different processes.

Overall confidence: 0.88

Full review comments:

- **[P1] Keep fallback interaction secrets stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a process-local random secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after a separate `ensureInteractionSecret` call, so CLI-sent buttons or post-restart callbacks can be rejected as `Invalid token`.
  - confidence: 0.91
- **[P2] Document the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive the HMAC secret from the bot token and public key string. After merge those scripts will generate rejected `_token` values, and operators will not know how to configure or rotate the new secret.
  - confidence: 0.86
- **[P2] Merge interaction overrides with inherited callback settings:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Adding the secret can silently drop callback reachability or allowlist settings for that account.
  - confidence: 0.82
