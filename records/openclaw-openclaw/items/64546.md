## Summary

Keep this PR open, but it is not merge-ready: it fixes the forgeability directionally, while the new process-local secret fallback can break real Mattermost button callbacks and the new signing-secret contract is not wired into setup, docs, env metadata, or the secret registry.

## What This Changes

The PR adds Mattermost account-level `interactions.secret` support and switches interactive button HMAC signing from bot-token derivation to configured or generated per-account secrets.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing while either provisioning a durable per-account interaction secret through setup/doctor/secret registry/docs or explicitly failing closed with operator guidance and upgrade proof.

## Reproduction Assessment

Yes. Current main source derives Mattermost interaction signing from the bot token with a public HMAC key and uses that secret for token generation and verification; no live Mattermost server was run in this read-only review.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but a process-local random fallback is not the best fix because it is not durable across the existing send/gateway process boundary or restarts.

- **[P1] Persist or explicitly require the interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a fresh value only in the current process.   - confidence: 0.92
