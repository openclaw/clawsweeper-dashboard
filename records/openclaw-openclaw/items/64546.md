## Summary

Keep this PR open: it addresses a real Mattermost callback-token security flaw, but the proposed process-local fallback can break legitimate button callbacks across CLI/gateway process boundaries or restarts and the new secret surface is not yet wired into setup, docs, or secret provisioning.

## What This Changes

The PR adds `channels.mattermost.accounts.<id>.interactions.secret` and changes Mattermost button HMAC signing from bot-token derivation to configured or generated per-account secrets.

## Best Possible Solution

Use a durable per-account Mattermost interaction secret, or explicitly fail closed with setup/doctor/docs guidance when it is missing, then prove fresh install, upgrade, restart or separate-process delivery, and rejection of old bot-token-derived forged tokens.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.9` still derive Mattermost interaction signing from the bot token with a public HMAC key, and PR head replaces that with configured or process-local generated secrets; I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

No. Removing the bot-token-derived signing root is the right direction, but process-local generation is not the best fix because it is not durable across the existing CLI/gateway process boundary or restarts.
