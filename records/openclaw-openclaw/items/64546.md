## Label Justifications

- `P1`: This is a security-sensitive Mattermost callback fix, but the current patch can break real interactive-button delivery for existing unconfigured setups.
- `merge-risk: 🚨 compatibility`: Existing setups without `interactions.secret` can move from stable bot-token-derived signing to per-process random signing that rejects old or cross-process buttons.
- `merge-risk: 🚨 message-delivery`: Mattermost button clicks can fail before dispatching the intended agent system event when the sender and verifier generated different secrets.
- `merge-risk: 🚨 security-boundary`: The PR changes the HMAC signing root for the plugin-auth Mattermost callback route, which is the authorization boundary for interaction callbacks.

## Summary

Keep this PR open: current main and the latest release still use the forgeable bot-token-derived Mattermost interaction secret, so the security fix is still necessary, but the proposed random fallback is not merge-safe for unconfigured cross-process button flows.

## What This Changes

The PR changes Mattermost interactive button token signing from a bot-token-derived HMAC secret to a configured or generated interaction secret, with schema/types/tests and send/monitor call-site updates.

## Best Possible Solution

Keep the security direction, but land it only after Mattermost has a stable cross-process interaction secret path or an intentional fail-before-send setup path, with docs and regression coverage for upgrade behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use Mattermost without `interactions.secret`, sign a button through the direct outbound sender, then verify the callback in a gateway process or after a gateway restart where the fallback generated a different secret.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but a process-local random fallback is not the narrowest maintainable fix for the shipped cross-process interaction flow.

Full review comments:

- **[P1] Preserve a cross-process signing root:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is omitted, this branch signs Mattermost buttons with a process-local random secret. Mattermost outbound delivery is direct, so CLI/tool sends can create buttons in one process while the gateway monitor verifies callbacks in another; those legitimate clicks will fail with `Invalid token` after a restart or cross-process send. Use a generated persisted/shared secret or fail before posting buttons with clear migration guidance.
  - confidence: 0.9
- **[P2] Update the public HMAC instructions:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` changes the external signing contract, but the Mattermost docs at this PR head still tell external scripts to derive the HMAC secret from the bot token. That keeps users on the vulnerable pattern and will generate invalid tokens once this patch lands, so the docs need to describe the new secret configuration and signing steps.
  - confidence: 0.86
