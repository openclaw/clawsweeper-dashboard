## Label Justifications

- `P1`: The PR addresses an urgent Mattermost callback security issue, but the current branch can also break interactive button workflows during upgrade.
- `merge-risk: 🚨 compatibility`: The branch changes Mattermost interaction credential semantics and adds a new config key without a complete upgrade contract.

## Summary

This PR is still necessary because current main and the latest release still have the Mattermost bot-token-derived interaction secret, but this branch is not merge-ready: its no-config fallback is process-local, the new account secret can drop inherited interaction callback settings, and the new config surface is not aligned with generated metadata/docs.

## What This Changes

The PR replaces Mattermost button HMAC signing from bot-token-derived secrets with configured or generated per-account interaction secrets, adds an `interactions.secret` config field, and updates Mattermost tests and call sites.

## Best Possible Solution

Land a maintainer-approved Mattermost interaction-secret contract that is separate from the bot token, durable across process/restart boundaries, preserves inherited interaction settings, and updates tests, generated config metadata, and user docs together.

## Reproduction Assessment

Yes. Source reproduction is high-confidence: current main still derives the Mattermost interaction signing root from the bot token with a public HMAC key, and the callback verifier accepts tokens generated from that secret.

## Solution Assessment

No. A separate Mattermost interaction secret is the right direction, but this PR is not the best merge-ready fix until the fallback is durable, account interaction config is preserved, and config metadata/docs are updated.

Full review comments:

- **[P1] Persist the fallback interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-199`
  - body: `ensureInteractionSecret` now falls back to a random value stored only in this process. `sendMessageMattermost` can sign button payloads in a different process from the gateway verifier, and any gateway restart clears the verifier's generated secret, so existing no-extra-config buttons can start returning 403 `Invalid token`; use a durable per-account secret or an explicit maintainer-approved migration/fail-fast contract instead.
  - confidence: 0.92
- **[P1] Preserve inherited interaction settings:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: Adding `interactions.secret` makes account-level `interactions` overrides likely, but Mattermost account resolution still deep-merges only `commands`. An account that sets only `interactions.secret` will replace inherited `callbackBaseUrl` and `allowedSourceIps`, which can move callbacks back to localhost or drop source filtering.
  - confidence: 0.87
- **[P2] Align the new secret config surface:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: The runtime schema accepts `interactions.secret`, but the PR leaves generated bundled config metadata without that key and leaves the Mattermost docs telling external scripts to derive the signing secret from the bot token. That makes the durable mitigation undiscoverable/inconsistent and leaves the documented external-script HMAC contract broken after upgrade.
  - confidence: 0.9
