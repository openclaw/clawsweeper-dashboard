## Summary

Keep this PR open: it addresses a real Mattermost interaction-token forgery, but the current patch can generate process-local signing secrets that break valid callbacks after restart or split-process send/gateway execution, and the new secret config is not yet wired through setup, secret contracts, or docs.

## What This Changes

The PR replaces Mattermost bot-token-derived interaction signing with configured or generated per-account interaction secrets and updates related schema, runtime, and tests.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing, uses either a durable per-account interaction secret or an explicit fail-closed setup requirement, wires setup/secret-target/docs coverage, and proves upgrade, restart or split-process callbacks, and old-forgery rejection.

## Reproduction Assessment

Yes, source-level. Current main derives the Mattermost interaction signing secret from the bot token with a public HMAC key, and PR-head source shows the unset-secret path can create process-local secrets independently in sender and gateway paths.

## Solution Assessment

No. Stopping bot-token-derived signing is the right direction, but this is not the best merge shape until the new signing secret is durable or explicitly required and setup, SecretRef, docs, upgrade, and restart behavior are covered.
