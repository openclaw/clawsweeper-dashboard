- `P1`: The PR addresses a security-sensitive Mattermost interaction authorization flaw but can break real button workflows if merged as-is.

## Summary

Keep open: the security direction is valid and the supplied terminal proof shows old-forgery rejection, but the patch still has P1 upgrade and security-contract blockers because absent config generates process-local secrets that can reject legitimate callbacks, the new secret contract is not wired through setup/docs/secret surfaces, and GitHub currently reports the head as conflicting.

## What This Changes

The PR changes Mattermost interaction signing from bot-token-derived HMAC secrets to configured or generated per-account interaction secrets, with schema and test updates.

## Best Possible Solution

Land a refreshed Mattermost security fix that removes bot-token-derived signing while either provisioning a durable per-account interaction secret through setup/secret registry/docs or explicitly failing closed with operator guidance and upgrade proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives Mattermost interaction signing from the bot token with a public HMAC key and verifies that token before dispatching button-click system events; no live Mattermost server was run in this read-only review.

## Solution Assessment

No. Removing bot-token-derived signing is the right security direction, but a process-local random fallback is not the best fix because it is not durable across the existing send/gateway process boundary or restarts.

Full review comments:

- **[P1] Persist or explicitly require the interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When interactions.secret is absent, this generates a fresh value only in the current process. Mattermost buttons can be signed by sendMessageMattermost in one process and verified by the gateway monitor in another, or clicked after a gateway restart, so existing setups can start returning 403 for legitimate callbacks. Make the secret durable/provisioned, or fail with an explicit setup/migration path before replacing the stable fallback.
  - confidence: 0.92
- **[P1] Update the public signing-secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`

Wire those surfaces to the chosen durable or fail-closed policy so configured users and external scripts do not produce invalid callbacks.
  - confidence: 0.88
