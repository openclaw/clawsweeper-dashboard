## Label Justifications

- `P1`: The PR addresses a high-impact Mattermost callback authorization weakness but can break existing channel button workflows if merged as-is.
- `merge-risk: 🚨 compatibility`: The PR changes Mattermost interaction-secret defaults and adds a new account config surface that existing users do not have.
- `merge-risk: 🚨 message-delivery`: Legitimate Mattermost button callbacks can be rejected when signing and verification use different process-local secrets.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root for the plugin-auth Mattermost callback route, which is the route's authorization boundary.

## Summary

Keep open: the security direction is valid, but the branch is not merge-ready because the replacement secret is process-local when unconfigured, the upgrade/setup contract is unresolved, and the public Mattermost HMAC docs still describe the old bot-token-derived verifier.

## What This Changes

The PR changes Mattermost interactive-button signing to use configured or generated interaction secrets instead of deriving the signing secret from the bot token.

## Best Possible Solution

Land a mergeable security fix that removes bot-token-derived signing while either provisioning a durable per-account interaction secret or failing closed with explicit setup/migration guidance, then update the Mattermost HMAC docs and prove upgrade, restart, separate-process, and old-forgery behavior.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.9 still derive Mattermost interaction signing from the bot token with a public HMAC key, and PR head replaces that with configured or process-local generated secrets; I did not run a live Mattermost server reproduction in this read-only review.

## Solution Assessment

No. Removing the bot-token-derived signing root is the right direction, but process-local generation is not the best fix because it is not durable across the existing CLI/gateway process boundary or restarts.
