## Label Justifications

- `P1`: This PR handles a real Mattermost interaction callback security bug, but the current default can break a live channel button workflow.
- `merge-risk: 🚨 compatibility`: Merging as-is can require existing users or external scripts to add a new `interactions.secret` or lose callback verification compatibility.
- `merge-risk: 🚨 message-delivery`: Unconfigured buttons signed in one process can be rejected by the gateway verifier, dropping the intended button-click action.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root that protects Mattermost interaction callback authorization.

## Summary

Keep open: the PR fixes the reported forgeable-token primitive in the crypto path, but the default replacement secret is process-local and can break Mattermost button callbacks across CLI/gateway processes or gateway restarts unless maintainers choose and document a stable credential contract.

## What This Changes

The PR adds a Mattermost `interactions.secret` SecretInput, resolves it into account config, and replaces bot-token-derived interaction HMAC secrets with configured or generated per-account secrets in send and monitor flows.

## Best Possible Solution

Use a non-bot-token per-account signing secret that is stable across senders, the gateway, and restarts, then document and test configured, unconfigured, and mixed account interaction settings.

## Reproduction Assessment

Yes. The current-main source derives the Mattermost interaction secret from a public HMAC key plus bot token, and the verifier gates callback dispatch on that derived value.

## Solution Assessment

No. A separate signing secret is the right direction, but this PR's random in-memory fallback is not the best fix because Mattermost senders and the gateway verifier can run in different processes.

- **[P1] Keep fallback interaction secrets stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a process-local random secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after a separate `ensureInteractionSecret` call, so CLI-sent buttons or post-restart callbacks can be rejected as invalid.
  - confidence: 0.91

  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive the HMAC secret from the bot token and public key string. After merge those scripts will generate rejected `_token` values, and operators will not know how to configure or rotate the new secret.
  - confidence: 0.86
- **[P2] Merge interaction overrides with inherited callback settings:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces inherited `interactions.callbackBaseUrl` and `allowedSourceIps`. Adding the secret can silently drop callback reachability or allowlist settings for that account.
  - confidence: 0.82
