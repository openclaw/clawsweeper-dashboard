## Label Justifications

- `P1`: The PR targets a security-sensitive Mattermost interaction signing bypass, but the patch still needs design and proof before merge.
- `merge-risk: 🚨 compatibility`: The generated in-memory fallback can break existing Mattermost button setups that do not configure a new interaction secret.
- `merge-risk: 🚨 message-delivery`: Valid Mattermost button clicks can be rejected when signing and verification do not share the generated process-local secret.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing boundary while leaving an incomplete fallback contract and stale public signing instructions.

## Summary

Keep this PR open, but it is not merge-ready: the security direction is useful, yet the current patch can break Mattermost button verification across processes or restarts, leaves the public signing contract stale, and still lacks real Mattermost proof.

## What This Changes

The PR adds Mattermost `interactions.secret` config plumbing and replaces bot-token-derived interaction HMAC secrets with configured or generated per-account secrets across schema, account resolution, send, monitor, interaction handling, and tests.

## Best Possible Solution

Land or replace this with a reviewed Mattermost security fix that uses a durable per-account signing secret independent of the bot token, or an explicit fail-closed config contract, with docs, tests, migration guidance, and real Mattermost proof together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with a Mattermost bot token and known interaction context, current main derives the same public-constant HMAC secret and will verify a matching `_token`.

## Solution Assessment

No. Separating interaction signing from the bot token is the right direction, but this PR's process-local fallback, config merge gap, stale docs, and missing real behavior proof make it unsafe to merge as-is.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use a durable interaction signing secret:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` creates a random process-local secret when `interactions.secret` is unset. The send path can sign buttons in one process while the gateway verifies them in another, or verification can happen after a restart, so valid clicks can be rejected as invalid; use a durable per-account secret or an explicit fail-closed contract instead of an in-memory fallback.
  - confidence: 0.93
- **[P2] Preserve shared interaction config when reading secrets:** `extensions/mattermost/src/mattermost/accounts.ts:103-108`
  - body: The new `merged.interactions?.secret` read relies on account merging that still deep-merges only `commands`. If shared `interactions.callbackBaseUrl` is paired with an account-level `interactions.secret`, the account object replaces the shared callback URL and button callbacks can fall back to the wrong gateway URL.
  - confidence: 0.87
- **[P2] Update the Mattermost signing docs:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes the external signing contract, but the Mattermost docs still instruct scripts to derive the signing key from the bot token with the public constant. Merging this leaves users with an insecure and incorrect recipe for the new verifier contract.
  - confidence: 0.86
- **[P3] Fix the stale interaction-secret error:** `extensions/mattermost/src/mattermost/interactions.ts:213`
  - body: After this API change, the error still tells callers to use `setInteractionSecret(accountId, botToken)`. Following that text would seed the verifier with the wrong value instead of the configured interaction signing secret.
  - confidence: 0.78
