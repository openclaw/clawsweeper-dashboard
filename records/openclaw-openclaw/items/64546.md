## Summary

Keep open: this PR targets a real Mattermost interaction-signing flaw, but the current diff has blocking correctness/security gaps and only test/check evidence rather than real Mattermost behavior proof.

## What This Changes

The PR adds a Mattermost `interactions.secret` config field and changes interaction HMAC setup to use configured or generated per-account secrets in send and monitor paths with tests.

## Best Possible Solution

Land a reviewed security fix that uses a durable per-account Mattermost interaction signing secret independent of the bot token, preserves interaction config merging, updates docs/schema/types/tests together, and includes redacted real Mattermost proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the secret is derived from the bot token with a public HMAC key and `verifyInteractionToken` accepts tokens generated from that derived secret for known contexts.

## Solution Assessment

No. Separating interaction signing from the bot token is the right direction, but this implementation uses an unstable process-local fallback and misses config/docs follow-through needed for a maintainable security fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve cross-process interaction secret stability:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` generates and stores a random process-local secret when `interactions.secret` is unset. The send path can sign buttons in a different process from the gateway verifier, or before a restart, so valid clicks can be rejected as `Invalid token`; use a durable per-account secret or fail configuration instead of a random in-memory fallback.
  - confidence: 0.93
- **[P2] Deep-merge interaction config before reading secrets:** `extensions/mattermost/src/mattermost/accounts.ts:103-108`
  - body: Resolving an account-level `interactions.secret` from the merged config lets that account's `interactions` object replace a shared `interactions.callbackBaseUrl`, because the account merge only deep-merges `commands`. A shared callback URL plus per-account secrets would silently fall back to the default gateway URL and break button clicks.
  - confidence: 0.87
- **[P2] Update Mattermost HMAC docs for configured secrets:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes how external scripts must sign Mattermost button contexts, but the Mattermost docs still tell users to derive the secret from the bot token with the public constant. Merging this would leave the public docs both insecure and incorrect for the new contract.
  - confidence: 0.86
- **[P3] Update the interaction-secret initialization error:** `extensions/mattermost/src/mattermost/interactions.ts:213`
  - body: The error still tells callers to use `setInteractionSecret(accountId, botToken)`, but this PR changes that second argument to a raw secret. Following the message would seed the verifier with the wrong value instead of the configured interaction secret.
  - confidence: 0.78
