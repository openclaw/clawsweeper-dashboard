## Summary

Keep open: this PR fixes a real Mattermost interaction-token forgery, but the current patch is not merge-ready because the new default generated secret is process-local and can break button verification across CLI/gateway processes or after a gateway restart when `interactions.secret` is not configured.

## What This Changes

The branch adds a Mattermost `interactions.secret` config field and changes interaction signing from a bot-token-derived HMAC root to a configured or generated per-account interaction secret.

## Best Possible Solution

Use a separate per-account Mattermost interaction signing secret that is stable across senders and the gateway without deriving from the bot token, then document and test fresh-install, upgrade, external-script, restart, and cross-process behavior.

## Reproduction Assessment

Yes. The vulnerability is source-reproducible on current main because `setInteractionSecret(accountId, botToken)` derives the verifier secret from a public HMAC key and the bot token, and the PR regression is source-reproducible because the new unconfigured fallback is module-local process state.

## Solution Assessment

No. A separate Mattermost interaction signing secret is the right direction, but this PR's random in-memory fallback is not the best mergeable fix until maintainers approve a persisted, shared, or explicitly required secret contract with upgrade proof.

Full review comments:

- **[P1] Keep the fallback secret stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is not configured, this branch generates a fresh module-local secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after separately calling the same helper, so CLI-sent buttons or buttons created before a gateway restart can be rejected as `Invalid token`. The fallback needs a persisted/shared secret or an explicit configured-secret requirement before the bot-token derivation can be removed.
  - confidence: 0.91
- **[P2] Document the new signing secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` changes the public token-generation contract, but the Mattermost docs still tell external scripts to derive the signing secret from the bot token and public HMAC string. After this merge those scripts will generate rejected `_token` values, and the public security guidance will describe the retired scheme.
  - confidence: 0.86
- **[P2] Merge interaction config before reading the secret:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces channel-level `interactions.callbackBaseUrl` and `allowedSourceIps` for that account. Operators adding a per-account secret can silently lose their callback URL or allowlist and break button delivery; include `interactions` in the nested merge and cover the mixed top-level/per-account case.
  - confidence: 0.82
