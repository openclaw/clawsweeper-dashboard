## Summary

Keep open. The PR targets a real Mattermost interaction-signing security bug, but the proposed random in-memory fallback breaks the existing cross-process signing contract, misses a config-merge edge case, and leaves the public HMAC docs stale.

## What This Changes

The PR adds a Mattermost `interactions.secret` config/type field, resolves it per account, replaces bot-token-derived interaction secrets with configured or generated secrets, and updates Mattermost signing tests plus send/monitor call sites.

## Best Possible Solution

Revise the Mattermost fix around a durable per-account interaction signing secret that is independent of the bot token, stable across send/gateway processes and restarts, preserves interaction config merging, and updates setup docs plus regression tests.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: with a known Mattermost bot token, compute the published derived secret, sign the canonicalized context, and the same `_token` path can pass verification before system-event dispatch.

## Solution Assessment

No. The PR addresses the right root problem, but a random process-local fallback is not the best fix because Mattermost interaction signing needs a durable shared secret across send and gateway processes; the docs and account config merge also need to be updated together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve cross-process interaction secret stability:** `extensions/mattermost/src/mattermost/interactions.ts:193-195`
  - body: `ensureInteractionSecret` generates a fresh process-local secret when `interactions.secret` is unset. The send path signs button payloads before the gateway monitor verifies callbacks, so CLI-created buttons or messages created before a restart can be rejected as `Invalid token`; use a persisted per-account secret or fail configuration instead of a random in-memory fallback.
  - confidence: 0.93
- **[P2] Keep interaction config fields merged:** `extensions/mattermost/src/mattermost/accounts.ts:100-105`
  - body: Resolving an account-level `interactions.secret` from the merged config currently lets that account's `interactions` object replace a top-level `interactions.callbackBaseUrl`, because the account merge only deep-merges `commands`. A shared reachable callback URL plus per-account secrets would silently fall back to the default gateway URL and break button clicks.
  - confidence: 0.87
- **[P2] Update the Mattermost HMAC docs for configured secrets:** `extensions/mattermost/src/config-schema-core.ts:148`
  - body: Adding `interactions.secret` changes how external scripts must sign Mattermost button contexts, but `docs/channels/mattermost.md` still tells users to derive the secret from the bot token with the public constant. Merging this would leave the public docs both insecure and incorrect for the new contract.
  - confidence: 0.86
