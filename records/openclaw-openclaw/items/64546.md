## Summary

Keep open: the PR addresses a real Mattermost interaction-token forgery, but the current patch is not merge-ready because the generated fallback secret is process-local, the new credential is not wired through supported setup/SecretRef/docs surfaces, and GitHub reports the branch as conflicting.

## What This Changes

The PR adds a Mattermost `interactions.secret` config field and changes interaction-token signing from bot-token-derived HMAC secrets to configured or generated per-account secrets.

## Best Possible Solution

Choose and implement a durable or explicitly fail-closed Mattermost interaction-secret model, then wire setup, SecretRef targets, docs, restart/split-process tests, and branch refresh together.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 derive the Mattermost callback HMAC key from a public constant plus the bot token, and the PR's process-local fallback regression is source-reproducible from the send and monitor call sites.

## Solution Assessment

No. Replacing the forgeable derivation is necessary, but an in-memory random fallback plus hidden config is not the best upgrade-safe solution for a callback credential that must survive restarts and split sender/verifier execution.

## Review Findings

Full review comments:

- **[P1] Make the fallback interaction secret durable:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is unset, this generates a fresh module-local secret. `sendMessageMattermost` signs buttons in its own process and the Mattermost gateway monitor verifies callbacks in another process or after a restart, so existing configs can start returning `Invalid token` for legitimate button clicks. Please either provision a durable shared secret for the unset case or make missing `interactions.secret` an explicit setup-time/runtime failure with upgrade guidance and coverage.
  - confidence: 0.93
- **[P1] Wire the new signing secret through setup and secret targets:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` to the schema makes it a user-facing credential, but the branch does not add a Mattermost setup credential, SecretRef target/assignment, or docs update. Users who need the new stable signing credential will not discover or configure it through the supported setup/secrets surfaces, and the public docs still describe button HMAC verification as automatic with no config needed.
  - confidence: 0.9
