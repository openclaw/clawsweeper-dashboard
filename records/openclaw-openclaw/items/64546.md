## Summary

Keep open: the PR addresses a real Mattermost interaction-token security flaw, but the current patch is not merge-ready because its process-local generated fallback can break legitimate button callbacks and the new signing-secret config surface is not wired through setup, secret registry, metadata, or docs.

## What This Changes

The PR changes Mattermost interaction signing from bot-token-derived HMAC secrets to configured or generated per-account secrets, adds `interactions.secret`, and updates Mattermost tests and send/monitor call sites.

## Best Possible Solution

Land a refreshed Mattermost security fix that removes bot-token-derived signing while using a durable per-account interaction secret or explicit fail-closed setup, with setup, secret registry, docs, generated metadata, upgrade behavior, and old-forgery rejection covered before merge.

## Reproduction Assessment

Yes. Source inspection shows current main derives Mattermost interaction signing from the bot token with a public HMAC key and verifies callback `_token` values against that derived secret; I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. Removing bot-token-derived signing is the right security direction, but a process-local random fallback is not the best fix because it is not durable across the existing send/gateway process boundary or restarts.

  - body: When `interactions.secret` is absent, `ensureInteractionSecret` generates a random value only in the current process. Without those surfaces, operators can configure a field that setup/audit/docs do not understand and callbacks may fail after upgrade.
