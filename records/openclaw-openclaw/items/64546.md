## Label Justifications

- `P1`: The PR addresses a high-impact Mattermost callback authorization weakness but can break an existing channel interaction workflow if merged as-is.
- `merge-risk: 🚨 compatibility`: The PR changes Mattermost interaction-secret defaults and adds a new account config surface that existing users do not have.
- `merge-risk: 🚨 message-delivery`: Legitimate Mattermost button callbacks can be rejected when signing and verification use different process-local secrets.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root for the plugin-auth Mattermost callback route, which is the route’s authorization boundary.

## Summary

Keep this PR open, but do not merge it as-is: the security direction is right, yet the fallback random secret is process-local and can break Mattermost interactive button verification for existing setups that do not already configure a separate interaction secret.

## What This Changes

The PR adds an optional Mattermost `interactions.secret`, resolves it through account config, and changes interaction signing to use configured or generated secrets instead of bot-token-derived HMAC secrets.

## Best Possible Solution

Land a maintainer-approved version that uses a separate durable per-account Mattermost interaction secret, with setup/docs/upgrade behavior and tests for old-forgery rejection plus cross-process or restart verification.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main derives Mattermost interaction signing from the bot token, and PR head generates a process-local fallback when no `interactions.secret` is configured.

## Solution Assessment

No. Removing the bot-token-derived secret is the right direction, but the unconfigured fallback is not the best fix because it is not durable across the existing CLI/gateway process boundary or restarts.
