## Summary

Keep open. The linked Mattermost security bug is real and still present on current main/latest release, but this PR is not merge-ready because the replacement signing root can be process-local, undocumented for external button scripts, and lossy for mixed interaction config.

## What This Changes

The PR adds a Mattermost `interactions.secret` config field and changes interaction token signing from bot-token-derived HMAC secrets to configured or generated per-account interaction secrets.

## Best Possible Solution

Use a separate durable per-account Mattermost interaction signing secret that is stable for all senders and the gateway, then update docs and tests for fresh installs, upgrades, external scripts, restarts, and mixed top-level/per-account interaction config.

## Reproduction Assessment

Yes. Current main and v2026.6.6 both derive the Mattermost interaction signing root from the bot token with a public HMAC key, and current tests assert deterministic derivation.

## Solution Assessment

No. A separate interaction signing secret is the right direction, but the PR's random in-memory fallback is not the best fix because senders and the gateway can run in different processes or across restarts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep unconfigured secrets stable across processes:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, this generates a fresh module-local secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after its own `ensureInteractionSecret` call, so CLI-sent buttons or buttons created before a gateway restart can be rejected as `Invalid token`. The fallback needs a persisted/shared secret or an explicit configured-secret requirement before the bot-token derivation is removed.
  - confidence: 0.91
- **[P2] Document the new signing secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` changes the public token-generation contract, but the Mattermost docs on the PR head still tell external scripts to derive the signing secret from the bot token and public HMAC string. After this merges those scripts will generate rejected `_token` values, and the public security guidance will describe the retired scheme.
  - confidence: 0.87
- **[P2] Merge interaction config before reading the secret:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: `resolveMergedAccountConfig` only deep-merges `commands`, so an account-level `interactions.secret` replaces channel-level `interactions.callbackBaseUrl` and `allowedSourceIps` for that account. Operators adding a per-account secret can silently lose their callback URL or allowlist and break button delivery; include `interactions` in the nested merge and cover the mixed top-level/per-account case.
  - confidence: 0.83
