## Label Justifications

- `P1`: The PR addresses a real Mattermost callback forgery issue but can also break real button callback workflows if merged as-is.
- `merge-risk: 🚨 compatibility`: Existing Mattermost configs without the new secret can change from stable bot-token-derived signing to process-local random signing.

## Summary

Keep this PR open, but it is not merge-ready: the security direction is right, yet the fallback signing secret is process-local when no new config is set, the new credential lacks setup/SecretRef/docs coverage, and GitHub currently reports the branch as conflicting.

## What This Changes

The PR changes Mattermost interaction signing to use a configured or generated per-account interaction secret instead of deriving it from the bot token, and updates related schema, runtime, and tests.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing while making the replacement per-account secret durable or explicitly required through setup, secret registry, docs, and upgrade proof.

## Reproduction Assessment

Yes, source-level: current main derives the Mattermost interaction signing secret from the bot token with a public HMAC key, and PR-head source shows the replacement fallback is process-local when no configured secret exists.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but this is not the best merge shape until the replacement secret is durable or explicitly required across setup, docs, upgrade, and restart behavior.
