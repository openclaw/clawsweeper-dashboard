## Label Justifications

- `P1`: This is an urgent Mattermost channel security fix, but the current patch can break real interactive-button workflows.
- `merge-risk: 🚨 compatibility`: Existing Mattermost users without a new `interactions.secret` can see previously working interactive buttons fail after this change.
- `merge-risk: 🚨 message-delivery`: The patch can cause CLI-created or pre-restart Mattermost button callbacks to be rejected before they dispatch the intended agent action.
- `merge-risk: 🚨 security-boundary`: The diff changes the HMAC signing root for the plugin-auth Mattermost callback route, which is the callback authorization boundary.

## Summary

Keep open: the patch is directionally right for the Mattermost token-forgery bug, but the default replacement secret is process-local and can break existing CLI/gateway and restart interaction flows unless a new secret is manually configured.

## What This Changes

The branch replaces Mattermost interaction signing secrets derived from the bot token with configured or generated per-account secrets, adds `interactions.secret` config parsing, and updates related tests/call sites.

## Best Possible Solution

Land a maintainer-approved security fix that uses a durable per-account Mattermost interaction secret, proves old bot-token-derived forgery rejection plus CLI/gateway and restart verification, and documents or migrates any required operator action.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run Mattermost without `interactions.secret`, create a button from the CLI/send path, then verify it in the gateway or after restart where `ensureInteractionSecret` can generate a different in-memory secret.

## Solution Assessment

No. A separate interaction signing secret is the right security direction, but a process-local random fallback is not the best fix for a CLI/gateway channel; the best fix needs a durable shared secret or an explicit maintainer-approved fail-closed migration.
