## Summary

Keep open. The PR targets a real Mattermost callback-forgery security flaw, but the current patch is not safe to merge because missing `interactions.secret` falls back to a process-local random secret and the new config/SecretRef contract is not wired through setup, secret registry, plugin metadata, or docs.

## What This Changes

The PR replaces Mattermost bot-token-derived interaction signing with configured or generated per-account secrets, adds `interactions.secret`, and updates Mattermost tests plus send/monitor call sites.

## Best Possible Solution

Land a refreshed Mattermost security fix that removes bot-token-derived signing while using a durable per-account interaction secret or explicit fail-closed setup, with the config, SecretRef, docs, and upgrade behavior wired before merge.

## Reproduction Assessment

Yes. Source inspection shows current main derives Mattermost interaction signing from the bot token with a public HMAC key and verifies callback `_token` values against that derived secret; I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. Removing bot-token-derived signing is the right security direction, but a process-local random fallback is not the best fix because it is not durable across the existing send/gateway process boundary or restarts.

## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Persist or explicitly require the interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, `ensureInteractionSecret` generates a random value only in the current process. Mattermost buttons can be signed by `sendMessageMattermost` and verified by the gateway monitor in another process, or clicked after restart; those existing setups will start rejecting valid callbacks. Provision a durable per-account secret, or fail closed with setup/migration guidance before replacing the stable fallback.
  - confidence: 0.93
- **[P1] Wire the new signing-secret contract through setup and docs:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: The new `interactions.secret` field accepts SecretRefs, but Mattermost's secret registry still only lists bot-token targets, setup only provisions bot token/base URL, plugin env metadata still lists only `MATTERMOST_BOT_TOKEN`/`MATTERMOST_URL`, and public docs do not explain the required secret. Without those surfaces, operators can configure a field that setup/audit/docs do not understand and callbacks may fail after upgrade.
  - confidence: 0.89
