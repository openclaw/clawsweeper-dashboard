## Label Justifications

- `P1`: This PR addresses a real Mattermost callback authorization weakness but currently risks breaking legitimate interactive button workflows.
- `merge-risk: 🚨 compatibility`: Existing Mattermost users without a new `interactions.secret` can lose working callbacks after upgrade, restart, or split CLI/gateway operation.
- `merge-risk: 🚨 message-delivery`: Mismatched sender and gateway secrets can cause valid Mattermost button clicks to be rejected as invalid tokens.
- `merge-risk: 🚨 security-boundary`: The PR changes the interaction callback signing boundary and adds a new secret-bearing config field without complete secret-contract/setup coverage.

## Summary

Keep this PR open, but it is not merge-ready: it removes the bot-token-derived HMAC path in the right direction, while the default replacement is process-local and the new secret config is not fully wired through setup, secret-target discovery, audit, docs, and upgrade behavior.

## What This Changes

The PR replaces Mattermost bot-token-derived interaction signing with configured or generated per-account secrets, adds an `interactions.secret` config field, and updates Mattermost send, monitor, account, schema, and interaction tests.

## Best Possible Solution

Land a refreshed security fix that stops bot-token-derived signing, uses either a durable per-account interaction secret or explicit fail-closed setup, wires setup/doctor/secret-target/docs coverage, and proves fresh install, upgrade, restart or separate-process callbacks, and old-forgery rejection.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.10 derive the Mattermost interaction secret from the bot token with a public HMAC key, and PR-head source shows unset `interactions.secret` generates independent in-memory secrets at sender and gateway call sites; I did not run a live Mattermost server.

## Solution Assessment

No. Stopping bot-token-derived signing is the right direction, but this PR is not the best merge shape until the secret is durable or explicitly required and the setup, secret-contract, docs, upgrade, and restart behavior are covered.
