## Summary

Keep open: the security direction is valid, but the replacement makes unconfigured Mattermost signing process-local, leaves the new credential outside setup/docs/secret discovery, and the branch is currently conflicting.

## What This Changes

This PR adds a Mattermost `interactions.secret` config field and switches interaction signing from bot-token-derived HMAC secrets to configured or generated per-account secrets.

## Best Possible Solution

Land a refreshed security fix that removes bot-token-derived signing and either provisions a durable per-account interaction secret through setup, SecretRef/docs/tests, or explicitly fails closed with clear upgrade behavior and restart or split-process proof.

## Reproduction Assessment

Yes. Source inspection of current main shows the Mattermost interaction secret is deterministically derived from the bot token with a public HMAC key, and the PR body includes a focused runtime proof that the old forged token no longer verifies on the branch.

## Solution Assessment

No. Removing bot-token-derived signing is the right direction, but this is not the best merge shape until the replacement secret is durable or explicitly required across setup, docs, upgrade, and restart or split-process behavior.

- **[P1] Make fallback interaction secrets durable:** `extensions/mattermost/src/mattermost/interactions.ts:198-199`
  - body: When `interactions.secret` is unset, this generates a random secret stored only in the current process. Mattermost button props can be signed by `sendMessageMattermost` and verified later by the gateway monitor or after a restart, so existing configs can start rejecting valid clicks as invalid tokens. Add the setup/SecretRef/docs path or remove the config surface until operators have a supported upgrade path.
