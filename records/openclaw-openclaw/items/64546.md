## Summary

Keep open. The PR addresses a real Mattermost callback authorization bug and the terminal proof is sufficient, but it is not merge-ready because the unconfigured fallback becomes process-local, the new signing-secret contract is undocumented, and inherited interaction settings can be replaced when adding the new secret.

## What This Changes

The PR adds an optional Mattermost `interactions.secret` SecretInput and switches interaction token signing from bot-token-derived HMAC secrets to configured or generated per-account secrets in the send and monitor paths.

## Best Possible Solution

Use a separate per-account Mattermost interaction signing secret that is durable or explicitly required across senders, gateway restarts, and upgrades, then update docs/tests and preserve inherited interaction settings.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main and v2026.6.6: Mattermost derives the interaction verifier secret from the bot token with a public HMAC key, and the callback handler gates dispatch on that token.

## Solution Assessment

No. A separate signing secret is the right direction, but this PR's random in-memory fallback is not the best fix because Mattermost senders and the gateway verifier can run in different processes and across restarts.

  - body: When `interactions.secret` is absent, this generates a process-local random secret. `sendMessageMattermost` signs button props in the send path while the gateway monitor verifies callbacks after separate initialization, so CLI-sent buttons or post-restart callbacks can be rejected as `Invalid token`.

  - body: This adds `interactions.secret`, but the Mattermost docs still tell external scripts to derive the HMAC secret from the bot token and public key string. After merge those scripts will generate rejected `_token` values, and operators will not know how to configure or rotate the new secret.

  - confidence: 0.82
