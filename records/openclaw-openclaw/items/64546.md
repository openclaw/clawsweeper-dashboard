## Summary

Keep this PR open, but it is not merge-ready: current main and the latest release still have the Mattermost bot-token-derived signing weakness, while this branch fixes forgeability only by adding a process-local fallback and an incomplete new secret/config contract.

## What This Changes

The PR replaces Mattermost bot-token-derived interaction signing with configured or generated per-account secrets, adds an `interactions.secret` config field, and updates Mattermost tests plus send/monitor call sites.

## Best Possible Solution

Land a refreshed Mattermost security fix that removes bot-token-derived signing while either provisioning a durable per-account interaction secret through setup/secret registry/docs or explicitly failing closed with operator guidance and upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows current main derives Mattermost interaction signing from the bot token with a public HMAC key and verifies callback `_token` values against that derived secret; I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. Removing bot-token-derived signing is the right security direction, but a process-local random fallback is not the best fix because it is not durable across the existing send/gateway process boundary or restarts.

Full review comments:

- **[P1] Persist or explicitly require the interaction secret:** `extensions/mattermost/src/mattermost/interactions.ts:198-200`
  - body: When `interactions.secret` is absent, `ensureInteractionSecret` generates a random value only in the current process. Mattermost buttons can be signed by `sendMessageMattermost` and verified by the gateway monitor in another process, or clicked after restart; those existing setups will start rejecting valid callbacks. Provision a durable per-account secret, or fail closed with setup/migration guidance before replacing the stable fallback.
  - confidence: 0.93
- **[P1] Wire the new signing-secret contract through setup and docs:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: The new `interactions.secret` field accepts SecretRefs, but Mattermost's secret contract still only registers bot-token paths, setup only provisions bot token/base URL, plugin env metadata still lists only `MATTERMOST_BOT_TOKEN`/`MATTERMOST_URL`, and public docs still tell external scripts to derive from the bot token. Without those surfaces, configured SecretRefs can fail resolution and external/scripted button users get invalid callback tokens.
  - confidence: 0.89
