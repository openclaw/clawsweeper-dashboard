## Summary

Keep open. The PR addresses a real Mattermost signing vulnerability, but this patch still uses process-local fallback secrets, can lose shared interaction callback config, leaves the public HMAC recipe stale, and lacks after-fix real Mattermost proof.

## What This Changes

The PR adds Mattermost `interactions.secret` config support and changes interactive button HMAC signing from bot-token-derived secrets to configured or generated per-account secrets across schema, types, send, monitor, interaction, and tests.

## Best Possible Solution

Land or replace this PR with a durable per-account Mattermost interaction signing secret independent of the bot token, preserving cross-process/restart behavior and updating schema, types, docs, tests, and real Mattermost proof together.

## Reproduction Assessment

Yes. Source inspection on current main shows the interaction secret is derived from the bot token with a public HMAC constant, and tokens generated from that derived secret are accepted by `verifyInteractionToken` before the handler proceeds.

## Solution Assessment

No. Separating interaction signing from the Mattermost bot token is the right direction, but this PR needs a durable or explicitly fail-closed secret contract plus aligned docs and real Mattermost proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use a durable interaction signing secret:** `extensions/mattermost/src/mattermost/interactions.ts:192-194`
  - body: `ensureInteractionSecret` generates and stores a random process-local secret when `interactions.secret` is unset. The send path can sign buttons in one process while the gateway verifies in another, or verification can happen after a restart, so valid clicks can be rejected as invalid; use a durable per-account secret or fail closed instead of an in-memory fallback.
  - confidence: 0.93
- **[P2] Preserve shared interaction config when reading secrets:** `extensions/mattermost/src/mattermost/accounts.ts:103-108`
  - body: The new `merged.interactions?.secret` read relies on account merging that still deep-merges only `commands`. If shared `interactions.callbackBaseUrl` is paired with an account-level `interactions.secret`, the account object replaces the shared callback URL and button callbacks can fall back to the wrong gateway URL.
  - confidence: 0.87
- **[P2] Update the Mattermost signing docs:** `extensions/mattermost/src/config-schema-core.ts:113`
  - body: Adding `interactions.secret` changes the external signing contract, but the Mattermost docs still instruct scripts to derive the signing secret from the bot token with the public constant. Merging this leaves users with an insecure and incorrect recipe for the new verifier contract.
  - confidence: 0.86
- **[P3] Fix the stale interaction-secret error:** `extensions/mattermost/src/mattermost/interactions.ts:213`
  - body: After this API change, the error still tells callers to use `setInteractionSecret(accountId, botToken)`. Following that text would seed the verifier with the wrong value instead of the configured interaction signing secret.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no supply-chain surface, but the security fix is incomplete because fallback signing secrets are process-local and public docs retain the old forgeable recipe.

Concerns:

- **[medium] Process-local fallback weakens the signing contract:** `extensions/mattermost/src/mattermost/interactions.ts:192`
  - body: When no configured secret exists, the PR creates a random in-memory secret in whichever process calls `ensureInteractionSecret`; send and gateway verification can run in different processes or across restarts, so the fix needs a persistent or explicit fail-closed contract.
  - confidence: 0.91
