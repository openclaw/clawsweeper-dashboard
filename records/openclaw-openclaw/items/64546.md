## Label Justifications

- `P1`: This PR targets a high-impact Mattermost callback security bug but currently risks breaking existing interactive button delivery and secure configuration.
- `merge-risk: 🚨 compatibility`: The default signing secret changes from shipped deterministic behavior to a process-local random value unless operators add new config.
- `merge-risk: 🚨 message-delivery`: Mattermost button clicks can be rejected as invalid when signing and verification happen in different processes or after a restart.
- `merge-risk: 🚨 security-boundary`: The changed secret is the authorization boundary for Mattermost interaction callbacks registered with plugin auth.

## Summary

Keep open. The PR addresses a real Mattermost callback-forgery bug, but the current patch is not merge-ready because its unconfigured fallback is process-local and its new SecretRef/config contract is incomplete.

## What This Changes

This PR replaces Mattermost bot-token-derived interaction HMAC secrets with generated or configured per-account secrets, adds an `interactions.secret` config field, and updates focused Mattermost tests/send wiring.

## Best Possible Solution

Use a durable per-account Mattermost interaction secret with explicit fresh-install and upgrade behavior, registered SecretRef support, updated docs/generated metadata, and cross-process regression coverage before merge.

## Reproduction Assessment

Yes, source-level. On the PR head, the unconfigured secret is randomly generated in process-local memory, while the send and monitor paths initialize it independently for signing and verification.

## Solution Assessment

No. The security direction is right, but the current implementation is not the best fix until the secret is durable, SecretRef handling is registered, and docs/generated metadata describe the new contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve cross-process interaction secrets:** `extensions/mattermost/src/mattermost/interactions.ts:198-199`
  - body: When `interactions.secret` is unset, this now creates a random secret only in the current process. `sendMessageMattermost` signs button payloads from the outbound send path, while the gateway monitor verifies callbacks from its own process and also calls `ensureInteractionSecret`, so CLI-sent buttons or buttons clicked after a gateway restart can be rejected as `Invalid token`.
  - confidence: 0.94
- **[P1] Register interaction secrets for SecretRef resolution:** `extensions/mattermost/src/mattermost/accounts.ts:118-123`
  - body: This strict read allows the new `interactions.secret` field to be a SecretRef, but Mattermost's secret contract still only registers and collects `botToken`. A configured SecretRef at `channels.mattermost.accounts.<id>.interactions.secret` can therefore remain unresolved and throw at runtime instead of providing the durable signing secret this PR expects.
  - confidence: 0.9
- **[P2] Document the new interaction secret contract:** `extensions/mattermost/src/config-schema-core.ts:150`
  - body: Adding `interactions.secret` changes the public signing contract, but the PR leaves the Mattermost external-script docs on the old bot-token-derived HMAC flow and does not refresh generated config/help metadata. Users following the docs will create rejected tokens or keep using the insecure derivation this PR is retiring.
  - confidence: 0.86
