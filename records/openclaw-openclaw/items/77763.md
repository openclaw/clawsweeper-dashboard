## Summary

This PR targets a valid source-reproducible multi-agent routing bug, but it should stay open because the patch introduces a raw `agentId` override on a write-scoped gateway method and the PR body only contains test/check proof, not after-fix real behavior proof.

## What This Changes

The PR threads the originating agent id through Google Meet and Voice Call call creation, persists it on `CallRecord`, and adds regression tests for per-agent voice routing.

## Best Possible Solution

Keep the per-call `CallRecord` routing direction, but derive the answering agent only from trusted plugin/runtime context or an authenticated internal hop, add the required changelog, and require live multi-agent proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main drops the caller agent before the Voice Call gateway and both classic and realtime voice paths fall back to configured `agentId` or `main`; no live Twilio/Meet run was provided in the PR.

## Solution Assessment

Is this the best way to solve the issue? No. The centralized `CallRecord` helper design is sound, but accepting `agentId` and `sessionKey` from raw gateway params is not the safest trust boundary for multi-agent isolation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Authenticate agent overrides before persisting them:** `extensions/voice-call/index.ts:606-607`
  - body: `voicecall.start` now accepts `params.agentId` from the gateway request and freezes it into the call record. That method is registered with `operator.write`, so a write-scoped caller can select another agent workspace even though the new type comment says the value comes from trusted tool context. Derive this value from trusted plugin/internal context or reject external overrides.
  - confidence: 0.86
