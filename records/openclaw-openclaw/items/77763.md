## Summary

Keep open: the linked bug is source-reproducible and the per-call `CallRecord.agentId` direction is sound, but the current PR head still exposes caller-supplied plugin owner identity through a public plugin-runtime export and its real behavior proof is stale relative to the latest head.

## What This Changes

The PR threads tool `ctx.agentId` and session keys through Google Meet and Voice Call startup, stores `CallRecord.agentId`, centralizes downstream agent resolution, adds tests/proof script coverage, and updates the changelog.

## Best Possible Solution

Keep the per-call `CallRecord.agentId` model, but bind gateway-dispatch owner identity to trusted plugin runtime scope or an internal allowlist, then refresh current-head real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows tool context carries `agentId`, but Google Meet and Voice Call do not serialize/store it, and downstream response generation still falls back to the configured Voice Call agent.

## Solution Assessment

No for the current PR head. The per-call routing design is the narrow maintainable fix, but the new public dispatch helper accepts owner identity from its caller instead of deriving it from trusted runtime scope, and the current head lacks fresh real behavior proof.
