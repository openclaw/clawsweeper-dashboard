## Summary

This PR should stay open: current main clearly loses the originating agent on the Google Meet and Voice Call outbound paths, and the PR now has changelog plus terminal proof, but the latest patch still introduces a public plugin-runtime gateway dispatch helper that can stamp caller-supplied plugin owner identity.

## What This Changes

The branch threads tool `ctx.agentId` through Google Meet and Voice Call outbound call creation, persists `CallRecord.agentId`, centralizes call-agent resolution, adds tests/proof coverage, and adds a changelog entry.

## Best Possible Solution

Keep the per-call `CallRecord.agentId` resolver design, but make gateway owner identity come from trusted runtime scope or a private/whitelisted handoff instead of caller-provided helper options.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main drops `ctx.agentId` before the Google Meet and Voice Call outbound call paths, then downstream voice response code resolves from configured agent id or `main`.

## Solution Assessment

Is this the best way to solve the issue? No. Freezing `CallRecord.agentId` is the right shape, but the new public dispatch helper should not accept caller-supplied plugin owner identity as the trust marker.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Derive gateway owner identity from runtime scope:** `src/plugins/runtime/gateway-dispatch.runtime.ts:35-37`
