## Summary

Keep open: the underlying routing bug is valid and the per-call CallRecord direction is sound, but this patch still exposes a public helper that lets plugin code supply the owner id used as a gateway trust marker.

## What This Changes

The PR threads the initiating agent id through Google Meet and Voice Call outbound call creation, stores it on CallRecord, routes downstream voice responses through a resolver, adds tests/proof, and updates the changelog.

## Best Possible Solution

Keep the per-call CallRecord.agentId design, but make the gateway handoff derive and authorize plugin identity from trusted runtime scope or an internal allowlist before honoring agentId/sessionKey.

## Reproduction Assessment

Yes, from source inspection: current main exposes toolContext.agentId, but Google Meet and Voice Call drop it before CallRecord creation, and downstream response code reads the configured/default agent.

## Solution Assessment

No. The routing shape is the narrow maintainable fix, but exposing a public dispatch helper that accepts pluginRuntimeOwnerId as an option is not the safest trust boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Derive gateway owner identity from trusted scope:** `src/plugins/runtime/gateway-dispatch.runtime.ts:36`
  - body: `dispatchPluginGatewayRequest` is exported through the public plugin-runtime SDK but copies `options.pluginRuntimeOwnerId` into the synthetic gateway client. Any plugin importing this helper can claim another owner before calling trust-gated methods such as `voicecall.start`; derive the owner from active runtime scope or keep this handoff internal/allowlisted before using it as authority.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a public plugin-runtime gateway dispatch helper that forwards caller-supplied plugin owner identity into a trust-gated path.

Concerns:

- **[medium] Public helper can mint plugin owner identity:** `src/plugins/runtime/gateway-dispatch.runtime.ts:36`
  - body: The proposed helper forwards `options.pluginRuntimeOwnerId` into `dispatchGatewayMethod`, and `voicecall.start` treats the resulting internal marker as the gate for accepting per-call `agentId` and `sessionKey`. That broadens a trust boundary unless the owner id comes from trusted runtime scope or a private allowlisted handoff.
