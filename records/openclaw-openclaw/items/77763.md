## Summary

Keep open: the PR targets a real current-main per-agent Voice Call/Google Meet routing bug, but it is not merge-ready because the new public plugin-runtime gateway dispatch helper lets callers supply an arbitrary plugin runtime owner id and weakens the trust boundary that `voicecall.start` now relies on.

## What This Changes

Threads the initiating agent id through Google Meet and Voice Call, persists it on `CallRecord`, adds an in-process plugin gateway dispatch helper, expands tests/proof, and edits `CHANGELOG.md`.

## Best Possible Solution

Keep the per-call `CallRecord` routing design, but make the gateway dispatch identity derive from the active plugin runtime scope, preserve existing gateway-method auth boundaries, remove the release-owned changelog edit, then rebase and validate the real call path.

## Reproduction Assessment

Yes. Current-main source inspection shows the Google Meet and Voice Call paths do not serialize or persist a per-call `agentId`, and downstream response paths resolve from config defaults instead.

## Solution Assessment

No as submitted. The per-call `CallRecord` approach is the right fix shape, but the public dispatch helper should not let callers choose `pluginRuntimeOwnerId`; that trust boundary must be fixed first.
