## Summary

Keep open: current main has the reported per-agent routing defect and the PR is directionally useful with sufficient terminal proof, but the patch introduces a security-sensitive raw RPC agent/session override and still needs the required changelog entry.

## What This Changes

The PR threads originating agent identity through Google Meet and Voice Call outbound call creation, persists it on call records, centralizes call-agent resolution, adds regression tests, and adds a proof script.

## Best Possible Solution

Keep the per-call CallRecord resolution design, but make cross-plugin agent identity come from trusted runtime context or an authenticated internal hop, then add the user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main drops ctx.agentId before the Voice Call gateway hop and downstream response paths resolve from configured agentId or main.

## Solution Assessment

Is this the best way to solve the issue? No. Freezing call.agentId is the right shape, but accepting agentId/sessionKey from raw write-scoped voicecall.start params is not the safest trust boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Authenticate agent overrides before persisting them:** `extensions/voice-call/index.ts:607-608`
  - body: voicecall.start now accepts params.agentId and params.sessionKey from the gateway request and freezes them into call creation. Because this method is write-scoped, a write-scoped caller can select another agent workspace; derive these values from trusted plugin/runtime context or reject external overrides before persisting them.
  - confidence: 0.86
