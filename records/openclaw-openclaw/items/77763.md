## Summary

Keep open: the PR addresses a valid source-confirmed multi-agent Voice Call/Google Meet routing bug, but the current head still has a blocking gateway trust-boundary issue and its real behavior proof is stale to an earlier head.

## What This Changes

The PR threads tool `ctx.agentId` through Google Meet and Voice Call call creation, stores per-call `CallRecord.agentId`, centralizes voice agent resolution, adds tests/proof script coverage, updates the changelog, and exposes a plugin-runtime gateway dispatch helper.

## Best Possible Solution

Keep the per-call `CallRecord.agentId` and centralized resolver design, but bind gateway owner identity to the actual plugin runtime scope or a narrow internal allowlist, then refresh current-head real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main has `toolContext.agentId`, but Google Meet and Voice Call drop it before call creation and downstream voice response code falls back to the configured/default agent.

## Solution Assessment

No for the current PR head. The per-call routing model is the narrow maintainable fix, but the new gateway dispatch helper should not accept caller-supplied owner identity through the public plugin runtime SDK.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Derive gateway owner identity from runtime scope:** `src/plugins/runtime/gateway-dispatch.runtime.ts:35-36`
