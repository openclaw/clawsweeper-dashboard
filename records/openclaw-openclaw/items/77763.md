## Summary

Keep open: the PR addresses a real current-main multi-agent routing bug, but the proposed cross-plugin hop still lets a write-scoped gateway caller choose the answering agent, so it needs security/maintainer review before merge.

## What This Changes

The PR threads the originating agent id through Google Meet and Voice Call call creation, persists it on voice-call records, centralizes call-agent resolution, adds regression coverage, and adds a local proof script.

## Best Possible Solution

Keep the per-call CallRecord resolution design, but make the cross-plugin agent identity come from trusted runtime context or an authenticated internal hop, then add the required changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: current main drops ctx.agentId before the Voice Call gateway hop and downstream voice response paths resolve from configured agentId or main.

## Solution Assessment

Is this the best way to solve the issue? No. Freezing the resolved agent on CallRecord is the right direction, but accepting agentId/sessionKey from raw voicecall.start params is not the safest trust boundary.
