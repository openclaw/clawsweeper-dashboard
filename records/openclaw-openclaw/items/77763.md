## Summary

Keep open: current main still drops the initiating agent id for Google Meet and Voice Call routing, and this PR targets the right bug, but the PR head still has a trust-boundary issue in the new gateway dispatch helper and its real-behavior proof predates later gateway/security changes.

## What This Changes

The PR threads originating agentId/sessionKey through Google Meet and Voice Call creation, stores agentId on CallRecord, resolves downstream voice responses from the call record, adds tests/proof, and updates the changelog.

## Best Possible Solution

Keep the per-call CallRecord.agentId design, but derive plugin owner identity from trusted runtime/registry scope before honoring agentId/sessionKey and refresh current-head runtime proof.

## Reproduction Assessment

Yes. From source inspection, current main exposes toolContext.agentId, but Google Meet and Voice Call do not serialize/store it for outbound calls and downstream response generation still falls back to voiceConfig.agentId ?? "main".

## Solution Assessment

No for the current PR head. The per-call routing model is the narrow maintainable fix, but the new public dispatch helper accepts owner identity from the caller instead of deriving it from registered plugin runtime scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Derive gateway owner identity from runtime scope:** `src/plugins/runtime/gateway-dispatch.runtime.ts:35-36`
  - body: `dispatchPluginGatewayRequest` is exported through `openclaw/plugin-sdk/plugin-runtime`, but it copies the caller-provided `pluginRuntimeOwnerId` into the synthetic gateway client. Any plugin importing this helper can stamp an arbitrary owner before calling trust-gated methods such as `voicecall.start`; derive the owner from registered runtime scope or keep this handoff internal/allowlisted.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR still introduces a public plugin-runtime helper that can mint a plugin owner marker used by trust-gated gateway handlers.

Concerns:

- **[medium] Public helper can mint plugin owner identity:** `src/plugins/runtime/gateway-dispatch.runtime.ts:35`
  - body: The helper forwards caller-supplied `pluginRuntimeOwnerId` into `dispatchGatewayMethod`, while `voicecall.start` treats the resulting internal marker as the gate for accepting per-call `agentId` and `sessionKey`. That broadens a trust boundary unless the owner id comes from trusted runtime scope or a private allowlisted handoff.
