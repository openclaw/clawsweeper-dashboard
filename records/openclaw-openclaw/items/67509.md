## Summary

Keep open: the PR addresses a real root-state corruption path, but it is not merge-ready because the global root refusal still conflicts with published root-session DigitalOcean/VPS instructions and needs the required user-facing changelog entry.

## What This Changes

The PR adds a UID/EUID-based root guard to the modern and legacy CLI entrypoints, with explicit root and container-forwarding bypasses plus unit coverage.

## Best Possible Solution

Settle the intended root-execution policy, then land a guard scoped to that policy with matching DigitalOcean/VPS docs, recovery guidance, tests, and changelog coverage.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to ~/.openclaw, and the linked DigitalOcean report gives a deterministic root-shell path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. A guard is plausible, but the PR needs a maintainer policy choice between global CLI refusal and a narrower managed-image guard, plus aligned docs and changelog coverage.
