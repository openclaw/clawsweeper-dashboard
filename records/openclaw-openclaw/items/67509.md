## Summary

Keep open: current main still lacks the requested root-execution guard, but this PR is not merge-ready because it would make documented root-based DigitalOcean/VPS flows fail and the user-facing behavior has no changelog coverage.

## What This Changes

The PR adds a CLI root guard, wires it into modern and legacy entrypoints, and tests UID/EUID plus explicit root and container-forwarding bypass behavior.

## Best Possible Solution

Land a root-execution policy that prevents DigitalOcean 1-Click state corruption while keeping supported VPS docs, escape hatches, and changelog coverage aligned.

## Reproduction Assessment

Yes, at source level. Current main enters both CLI paths without UID/EUID checks, defaults mutable state to ~/.openclaw, and the linked bug gives a deterministic root-shell doctor --fix path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. The guard addresses the state corruption path, but it needs an explicit root/VPS policy plus docs and changelog updates before it can be the narrow maintainable fix.
