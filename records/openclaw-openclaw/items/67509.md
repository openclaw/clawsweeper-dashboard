## Summary

Keep open: current main still lacks a root-execution guard, and the PR has addressed several earlier review comments, but it should not merge while it globally refuses root execution without updating the root-based DigitalOcean/VPS setup path or adding the required changelog entry.

## What This Changes

The PR adds a CLI root guard for UID/EUID 0 with explicit root and container-forwarding bypasses, wires it into modern and legacy entrypoints, and adds focused tests.

## Best Possible Solution

Decide the intended root-execution policy, then land either a global guard with matching docs/changelog or a narrower DigitalOcean/doctor wrapper that prevents the 1-Click corruption path without breaking supported VPS setup.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to ~/.openclaw, and the linked report gives a deterministic root-shell DigitalOcean path; I did not live-run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard is plausible, but the PR must first reconcile global root refusal with published root-session install docs and add changelog coverage, or narrow the fix to the managed-image/doctor path.
