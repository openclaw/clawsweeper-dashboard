## Summary

Keep this PR open. Current main still lacks the root-execution guard for the CLI, and the PR is a focused implementation for the linked DigitalOcean state-corruption bug, but it is not merge-ready because the root-session DigitalOcean docs and required changelog entry are still missing and the root override policy needs maintainer sign-off.

## What This Changes

The PR adds a CLI root guard and tests, then wires modern and legacy CLI entrypoints to block real or effective UID 0 unless an explicit root override or container-forwarding environment is present.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with docs and changelog aligned: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override DigitalOcean workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH reproduction steps, and current main still enters CLI startup without a UID/EUID guard while state defaults through the invoking user's home directory.

## Solution Assessment

No, not as submitted. The root-guard shape is a plausible narrow fix, but the PR should also reconcile the DigitalOcean setup docs and changelog, and maintainers should explicitly accept the root override and container bypass policy.
