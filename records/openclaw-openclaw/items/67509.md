## Summary

Keep open. Current main still lacks the root guard and the PR addresses a real, reproducible root-state bug, but the submitted behavior conflicts with published DigitalOcean root-shell setup steps and still needs docs/changelog alignment before merge.

## What This Changes

The PR adds a CLI root guard for real or effective UID 0, allows explicit and container-forwarded bypasses, wires both CLI entrypoints, and adds unit tests.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with docs and changelog aligned: guard both entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override DigitalOcean workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic DigitalOcean root-SSH reproduction steps, and current main still starts both CLI entrypoints without a UID/EUID guard while mutable state defaults through the invoking user's home directory.

## Solution Assessment

No, not as submitted. The guard shape is a plausible narrow fix, but it must be reconciled with DigitalOcean/VPS install guidance and the required changelog before it is the best maintainable solution.
