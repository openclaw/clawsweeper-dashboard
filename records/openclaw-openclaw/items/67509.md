## Summary

Keep open. Current main still lacks a root-execution guard and the linked root-state corruption path remains source-reproducible, but this PR is not merge-ready because the new refusal behavior conflicts with current DigitalOcean setup docs and still needs changelog/test cleanup plus maintainer policy sign-off.

## What This Changes

The PR adds a CLI root guard at the modern and legacy entrypoints, with tests for real/effective UID checks, explicit root override, and container-forwarded bypass handling.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both entrypoints guarded before mutable state side effects, real/effective UID coverage retained, intentional container forwarding preserved, and DigitalOcean docs plus the changelog aligned.

## Reproduction Assessment

Yes, by source inspection and the linked bug steps. Current main lets root reach CLI startup, and state defaults to ~/.openclaw, matching the deterministic DigitalOcean root-shell corruption path even though this read-only review did not run a live droplet reproduction.

## Solution Assessment

No, not as-is. Early guarding in both entrypoints is a plausible narrow fix, but the PR must reconcile the root-session DigitalOcean workflow, changelog policy, and root/effective-UID test coverage before merge.
