## Summary

Keep this PR open: it addresses a real source-reproducible root-execution corruption path, but the current patch is not merge-ready because it would make published DigitalOcean setup steps fail and it lacks the required changelog entry.

## What This Changes

The PR adds an `assertNotRoot` CLI guard with tests and invokes it from modern and legacy CLI entrypoints, with explicit and container-forwarding bypasses.

## Best Possible Solution

Align the root policy with setup guidance: land an early guard with non-root VPS/DigitalOcean instructions and a changelog entry, or scope the guard to the image wrapper if maintainers want root-based manual installs to keep working.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to `~/.openclaw`, and #67478 gives a deterministic DigitalOcean root-shell path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard direction is plausible for the corruption bug, but the branch must first reconcile published root-session setup docs and add the required changelog entry.
