## Summary

Keep open. Current main still lacks the requested root-execution guard, and this PR is a relevant implementation candidate, but it is not merge-ready because the new CLI-wide refusal conflicts with current DigitalOcean root-session install docs and the branch lacks the required user-facing changelog entry.

## What This Changes

The PR adds a CLI root guard and tests, then wires it into the modern and legacy CLI entrypoints to block real or effective UID 0 unless an explicit root override or container-forwarding env pair is present.

## Best Possible Solution

Land this PR or an equivalent early guard after maintainer sign-off, with DigitalOcean/VPS docs and the changelog aligned: block root before mutable state/config side effects, cover both entrypoints, preserve intentional container forwarding, and document the supported non-root or explicit-override workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH reproduction steps, and current main still routes root invocations through normal CLI startup while state defaults through the invoking user's home directory.

## Solution Assessment

No, not as submitted. The early guard shape is a plausible narrow fix, but the branch should reconcile the documented root-based DigitalOcean setup flow and user-facing changelog before it is the best merge candidate.
