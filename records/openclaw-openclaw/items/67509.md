## Summary

Keep open. The PR addresses a real current-main gap, but it is not merge-ready because the new root refusal conflicts with existing DigitalOcean root-shell setup docs, lacks the required user-facing changelog entry, and the latest PR-head checks include failures.

## What This Changes

The PR adds a CLI root guard and tests, then wires it into modern and legacy entrypoints to block real or effective UID 0 unless an explicit root override or container-forwarding environment is present.

## Best Possible Solution

Land this PR or an equivalent early-entrypoint guard after maintainer sign-off, with DigitalOcean/VPS docs, changelog, and latest-head validation brought into alignment.

## Reproduction Assessment

Yes, at source level. Current main lacks a CLI-wide UID/EUID guard, state defaults through `HOME` to `~/.openclaw`, and user-scoped systemd service installation remains reachable; I did not run the destructive droplet scenario.

## Solution Assessment

Mostly yes for the code direction, but not as a complete merge candidate. Guarding both entrypoints before mutable CLI startup is the narrow fix, while the docs/changelog gaps, root-policy decision, and failing checks still need resolution.
