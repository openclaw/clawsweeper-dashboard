## Summary

Keep this PR open: the guard is a useful fix for the linked root-state corruption bug, but the current branch still conflicts with documented DigitalOcean root-shell setup steps and is missing the required user-facing changelog entry.

## What This Changes

Adds an `assertNotRoot` CLI guard with UID/EUID checks, tests override/container-forwarding cases, and wires it into modern and legacy CLI entrypoints.

## Best Possible Solution

Land a root-execution policy that prevents the DigitalOcean 1-Click corruption path while keeping Linux/VPS setup docs, the override behavior, and the changelog aligned.

## Reproduction Assessment

Yes, at source level. Current main enters both CLI paths without a UID/EUID guard, defaults mutable state to `~/.openclaw`, and #67478 provides deterministic root-shell `doctor --fix` steps; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. A root guard may be the right fix, but this branch needs the documented root-based DigitalOcean flow and changelog reconciled, or the guard narrowed to the managed-image path.
