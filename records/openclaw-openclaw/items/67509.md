## Summary

Keep this PR open: the root guard addresses the #67478 failure mode at source level, and the latest branch has resolved several earlier bypasses, but it is still not merge-ready because it would break current root-run Linux/DigitalOcean install paths and lacks the required changelog entry.

## What This Changes

The PR adds an `assertNotRoot` CLI guard, wires it into the modern and legacy CLI entrypoints, and adds tests for real/effective UID checks plus explicit and container-forwarding bypasses.

## Best Possible Solution

Land the UID/EUID guard only after the Linux/DigitalOcean installer and docs route users through a supported non-root workflow, or narrow the guard to the 1-Click wrapper if maintainers want root-based manual installs to continue.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, mutable state defaults to `~/.openclaw`, and #67478 provides a deterministic root-shell DigitalOcean path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard direction is plausible for the reported state corruption, but it must first be reconciled with supported root-run installer/DigitalOcean flows and documented in the changelog.
