## Summary

Keep open: the root guard addresses a real source-reproducible failure mode, but the PR is not ready because it globally refuses root while current install docs still instruct root-session OpenClaw commands, and it lacks the required changelog entry.

## What This Changes

The PR adds an early CLI root guard with UID/EUID checks, explicit root and container-forwarding bypasses, tests, and calls from both modern and legacy CLI entrypoints.

## Best Possible Solution

Decide the intended root-execution policy, then land either a global guard with matching docs/changelog or a narrower 1-Click wrapper/doctor path that prevents #67478 without breaking documented VPS setup.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to ~/.openclaw, and #67478 gives a deterministic root-shell DigitalOcean path; I did not live-run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard direction is plausible, but the patch must be reconciled with published root-session setup docs and changelog coverage, or narrowed to the managed-image path.
