## Summary

Keep open: the root guard addresses a source-reproducible corruption path, but the submitted patch conflicts with current root-based DigitalOcean setup docs and lacks required changelog coverage, so it should not merge as-is.

## What This Changes

The PR adds a CLI root guard, wires it into modern and legacy entrypoints, and tests UID/EUID plus explicit root and container-forwarding bypass behavior.

## Best Possible Solution

Land a root-execution policy that protects the DigitalOcean 1-Click path while keeping Linux/VPS docs, escape hatches, and changelog aligned.

## Reproduction Assessment

Yes, at source level. Current main enters both CLI paths without UID/EUID checks, defaults mutable state to `~/.openclaw`, and #67478 gives a deterministic root-shell `doctor --fix` path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. A global guard may be the right direction, but the patch must reconcile root-based install docs and changelog coverage, or narrow the guard to the managed-image path.
