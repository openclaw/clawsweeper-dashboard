## Summary

Keep open: current main still lacks a root/effective-root CLI guard, and this PR is a plausible fix for #67478, but it is not merge-ready because it conflicts with published root-session DigitalOcean setup docs and lacks required changelog coverage.

## What This Changes

The branch adds an `assertNotRoot` helper, tests, and modern/legacy CLI entrypoint calls to refuse root or effective-root execution unless an explicit or container-forwarding bypass is present.

## Best Possible Solution

Align the root policy with setup guidance: land an early UID/EUID guard with non-root VPS/DigitalOcean instructions and changelog coverage, or scope the guard to the image wrapper if root-based manual installs should keep working.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to `~/.openclaw`, and #67478 gives a deterministic DigitalOcean root-shell path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard direction is plausible for the corruption bug, but the branch must first reconcile published root-session setup docs and add the required changelog entry.
