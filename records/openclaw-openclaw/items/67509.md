## Summary

Keep open. This PR addresses a real root-state corruption path and the author has already handled earlier bypass concerns, but it is not merge-ready while published DigitalOcean/VPS docs still instruct root-session `openclaw` usage and the user-facing CLI change lacks a changelog entry.

## What This Changes

The PR adds a UID/EUID root guard with explicit and container-forwarding bypasses, wires it into modern and legacy CLI entrypoints, and adds unit coverage.

## Best Possible Solution

Ship the root-state protection only after the intended root policy is settled, with matching install docs, recovery guidance if needed, tests, and a changelog entry.

## Reproduction Assessment

Yes, at source level. Current main has no root guard in either CLI entrypoint, mutable state defaults under the invoking user's home directory, and #67478 gives a deterministic DigitalOcean root-shell path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. A global guard may be the right direction, but it must be reconciled with the root-based setup docs and changelog, or narrowed to the managed-image/wrapper/doctor path.
