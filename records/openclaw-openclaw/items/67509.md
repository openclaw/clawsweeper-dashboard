## Summary

Keep open. The PR targets a real, still-unfixed root execution bug, but it should not merge as-is because the new refusal path conflicts with current DigitalOcean root-session setup docs, lacks the required user-facing changelog entry, and leaves one UID test dependent on the runner's effective UID.

## What This Changes

The PR adds a CLI root guard module and tests, then wires it into modern and legacy CLI entrypoints to block real or effective UID 0 unless an explicit root override or container-forwarding hints are present.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both CLI entrypoints guarded before mutable state side effects, container forwarding preserved, and DigitalOcean setup docs or installer wrappers aligned to the chosen non-root workflow.

## Reproduction Assessment

Yes. The linked bug gives deterministic DigitalOcean root SSH plus doctor --fix steps, and current main shows both CLI entrypoints can still reach startup while mutable state resolves under the invoking user's home directory.

## Solution Assessment

No, not yet. Blocking root early in both entrypoints is a plausible narrow fix, but the branch still needs setup-doc alignment, a changelog entry, and a runner-independent eUID test before it is the maintainable solution.
