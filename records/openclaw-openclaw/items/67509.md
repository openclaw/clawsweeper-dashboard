## Summary

Keep open. The PR targets a real, source-reproducible root-execution bug, but it still conflicts with the documented DigitalOcean root onboarding flow, lacks the required changelog entry, and has no after-fix real behavior proof.

## What This Changes

Adds an `assertNotRoot` CLI startup guard for uid/euid 0, integrates it into both CLI entrypoints with explicit allow-root and container-child bypasses, and adds unit tests.

## Best Possible Solution

Land an early root-execution guard only after the root install/docs path, release notes, and redacted real Linux proof are aligned; keep the linked bug open until an equivalent fix merges.

## Reproduction Assessment

Yes, source-reproducible. Current main has no uid/euid guard in either CLI entrypoint, and `runCli` can resolve the root user's state directory before command dispatch; I did not run the destructive root state-mutation path in this read-only review.

## Solution Assessment

No, not as submitted. The early guard is a plausible fix direction, but the branch needs DigitalOcean docs alignment, a changelog entry, and real behavior proof before it is the maintainable solution.
