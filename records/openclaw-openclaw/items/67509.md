## Summary

Keep open. Current main still has no root-execution guard, and this PR is a focused implementation for the linked root-state corruption bug, but the proposed behavior would make the existing DigitalOcean setup docs fail and the PR still lacks the required user-facing changelog entry.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks, explicit and container-forwarding bypasses, unit tests, and wiring for both modern and legacy CLI entrypoints.

## Best Possible Solution

Land this PR or an equivalent core fix after the install docs and changelog are reconciled: guard both CLI entrypoints before mutable state/config work, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override workflow for DigitalOcean users.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH reproduction steps, and current main still reaches CLI startup without a UID/EUID guard while default state resolves through the effective home directory.

## Solution Assessment

No, not as submitted. The core guard is a plausible narrow fix, but the PR needs DigitalOcean install guidance and a changelog entry to avoid shipping a visible CLI refusal that contradicts current docs.
