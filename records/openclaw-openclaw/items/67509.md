## Summary

Keep this PR open: it targets a real root-state corruption path from #67478, but the submitted global root guard still conflicts with published DigitalOcean root-shell setup steps and lacks the required user-facing changelog entry.

## What This Changes

Adds a CLI root guard module, wires it into the modern and legacy CLI entrypoints, and tests UID/EUID handling plus explicit and container-forwarding bypasses.

## Best Possible Solution

Decide the intended root-execution policy, then land either a global guard with matching install docs and changelog or a narrower managed-image guard that prevents #67478 without breaking documented setup.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard, state defaults to ~/.openclaw, and #67478 provides a deterministic root-shell DigitalOcean path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. The guard direction is plausible, but the patch needs maintainer policy alignment plus docs and changelog coverage before it is the narrowest maintainable fix.
