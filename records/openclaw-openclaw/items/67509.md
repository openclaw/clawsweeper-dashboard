## Summary

Keep open. Current main still lacks a root-execution guard and the linked root-state corruption bug remains source-reproducible, but this PR is not merge-ready because the new refusal path conflicts with current DigitalOcean setup docs and still needs a changelog/test cleanup plus maintainer policy sign-off.

## What This Changes

This PR adds a CLI root guard at the modern and legacy entrypoints, with tests, to block real or effective UID 0 unless explicitly overridden or container-forwarded.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both entrypoints guarded before mutable state side effects, real/effective UID coverage retained, intentional container forwarding preserved, and DigitalOcean docs plus the changelog aligned.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection and the linked bug steps: current main lets root reach CLI startup, and state defaults to ~/.openclaw, matching the deterministic DigitalOcean root-shell corruption path, though this read-only review did not run a live root droplet reproduction.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. Early guarding in both entrypoints is a plausible narrow fix, but the PR must reconcile the root-session DigitalOcean workflow, changelog policy, and root/effective-UID test coverage before merge.
