## Summary

Keep open. Current main does not implement the requested root guard and the linked root-state corruption report remains source-backed; this PR is still useful, but it is not merge-ready until the root-based DigitalOcean setup flow and changelog are aligned.

## What This Changes

The PR adds a CLI root guard for real or effective UID 0, allows explicit and container-forwarded bypasses, wires the guard into the modern and legacy CLI entrypoints, and adds unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with docs and changelog aligned: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override DigitalOcean workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH reproduction steps, and current main still starts both CLI entrypoints without a UID/EUID guard while mutable state defaults through the invoking user's home directory.

## Solution Assessment

No, not as submitted. The guard shape is a plausible narrow fix, but the PR must reconcile the root refusal with DigitalOcean/VPS install guidance and add the required changelog entry before it is the best maintainable solution.
