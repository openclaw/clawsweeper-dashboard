## Summary

Keep open: this PR addresses a real root-execution bug and has handled earlier review comments, but it still conflicts with published root-based DigitalOcean setup docs and lacks the required changelog entry.

## What This Changes

Adds a CLI root guard module, wires it into the modern and legacy CLI entrypoints, and tests UID/EUID handling plus explicit root and container-forwarding bypasses.

## Best Possible Solution

Decide the intended root-execution policy, then land either a global guard with matching VPS/DigitalOcean docs and changelog or a narrower managed-image guard that prevents #67478 without breaking documented setup.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to ~/.openclaw, and #67478 gives a deterministic root-shell DigitalOcean path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. The guard direction is plausible, but the patch must be reconciled with published root-based setup docs and changelog coverage, or narrowed to the managed-image path.
