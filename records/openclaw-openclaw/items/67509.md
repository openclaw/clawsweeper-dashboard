## Summary

Keep this PR open. The PR addresses a real root-execution failure mode and incorporates earlier review feedback for the legacy entrypoint, container forwarding, and effective UID handling, but it is not ready to merge because the public DigitalOcean setup docs still instruct root-shell `openclaw` commands that this guard would now reject, and the global root-refusal/override policy needs maintainer sign-off.

## What This Changes

The PR adds a CLI root guard module and tests, wires it into the modern and legacy CLI entrypoints, and allows explicit or container-forwarded root bypasses.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with docs and changelog aligned: guard both entrypoints before mutable state/config side effects, check real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override DigitalOcean workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH reproduction steps, and current main still starts CLI paths without a UID/EUID guard while mutable state defaults through the invoking user's home directory.

## Solution Assessment

No, not as submitted. The guard shape is a plausible narrow fix, but it must be reconciled with root-based DigitalOcean/VPS documentation and the root override/container bypass policy before merge.
