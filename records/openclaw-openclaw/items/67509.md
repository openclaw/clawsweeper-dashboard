## Summary

Keep open. Current main still lacks the root-execution guard this PR adds, and the linked bug remains plausibly valid, but the proposed global CLI refusal still conflicts with current DigitalOcean setup docs and is missing the required user-facing changelog entry.

## What This Changes

The PR adds a CLI root guard that blocks real or effective UID 0 unless an explicit root override or container-forwarding hints are present, wires it into modern and legacy CLI entrypoints, and adds unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with documentation and changelog aligned: guard both CLI entrypoints before mutable state/config side effects, check real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override workflow for DigitalOcean users.

## Reproduction Assessment

Yes. The linked bug gives deterministic DigitalOcean root-SSH steps, and current main still reaches CLI startup without a UID/EUID guard while default state resolves through the effective home directory.

## Solution Assessment

No, not yet. The guard itself is a plausible narrow core fix, but the PR should not merge while the public DigitalOcean setup flow still tells users to run blocked root commands and the changelog omits this user-visible fix.
