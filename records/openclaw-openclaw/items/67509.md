## Summary

Keep this PR open. Current main still lacks a root-execution guard for the CLI, and the linked root-state corruption report remains valid, but the proposed broad refusal conflicts with current DigitalOcean setup docs that still run OpenClaw commands from a root SSH session.

## What This Changes

The PR adds a real/effective UID root guard for modern and legacy CLI entrypoints, allows explicit root and container-forwarding bypasses, and adds unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with matching DigitalOcean/1-Click guidance: guard both CLI entrypoints before mutable state/config side effects, check real and effective UID, preserve intentional container forwarding, and document the supported non-root service-user or wrapper workflow plus the explicit override.

## Reproduction Assessment

Yes. The linked report has deterministic root SSH reproduction steps, and current main confirms root invocations can still reach CLI startup while default state resolves under the effective home directory.

## Solution Assessment

No, not as submitted. A core root guard is a narrow and plausible fix, but the PR should also reconcile the documented DigitalOcean root-session commands or explicitly scope the guard to the intended image/wrapper behavior.
