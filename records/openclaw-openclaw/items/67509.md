## Summary

Keep open. The PR targets a valid root-state corruption bug, but the current patch blocks documented/root installer onboarding paths without updating or reshaping those workflows first.

## What This Changes

The PR adds an `assertNotRoot` helper, wires it into the modern and legacy CLI entrypoints, and tests uid/euid, allow-root, forwarded-container, and Windows-safe cases.

## Best Possible Solution

Land early root-execution prevention only after the install/onboarding and container contracts explicitly route root users to a non-root service account, safe wrapper, or deliberate override.

## Reproduction Assessment

Yes. Source inspection shows the new guard exits for uid/euid 0 while current DigitalOcean docs and the install script still run `openclaw onboard` from a root shell; I did not run the destructive droplet scenario.

## Solution Assessment

No. An early guard is the right direction for the reported corruption, but this PR needs the root install/onboarding contract reconciled and release notes added before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve documented root installs:** `src/entry.ts:98-100`
  - body: The guard exits before normal CLI handling for any uid/euid 0 invocation, but the current DigitalOcean docs still tell users to SSH as root and run `openclaw onboard --install-daemon`, and `scripts/install.sh` execs `openclaw onboard` during first setup. A root install that currently follows the docs will now stop at the guard unless the user sets `OPENCLAW_ALLOW_ROOT=1`, which sends them back into the state-corruption path this PR is trying to prevent. Please update or reshape the root install/onboarding path before enabling the CLI-wide guard.
