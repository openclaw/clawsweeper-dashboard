## Summary

Keep open. The related root-execution bug is valid and current main still lacks a CLI-wide uid/euid guard, but this external PR is not merge-ready because it conflicts with the documented DigitalOcean root setup flow, lacks the required changelog entry, and still has no after-fix real behavior proof.

## What This Changes

The PR adds an `assertNotRoot` CLI guard, wires it into the modern and legacy entrypoints, and adds unit coverage for uid/euid, override, and container-forwarding cases.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off: guard both CLI entrypoints before mutable state/config side effects, preserve intentional container forwarding, document the supported non-root/wrapper path and root override, add a changelog entry, and include real root-run proof.

## Reproduction Assessment

Yes, source-reproducible. Current main has no uid/euid guard in either entrypoint and `runCli()` can resolve state via `resolveStateDir()` before command dispatch; I did not run the destructive root-state path in this read-only review.

## Solution Assessment

No, not merge-ready as submitted. A core guard is a plausible narrow fix, but the docs/changelog/proof gaps and root override semantics need to be resolved before this is the best maintainable solution.
