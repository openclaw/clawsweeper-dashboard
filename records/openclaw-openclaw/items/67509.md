## Summary

Keep this PR open. The root-guard direction addresses a real, still-unfixed root-state corruption path, but the current branch still conflicts with published DigitalOcean setup guidance and lacks the required changelog/test cleanup before it should merge.

## What This Changes

This PR adds a CLI root guard, tests it, and wires modern plus legacy CLI entrypoints to reject real or effective UID 0 unless an explicit root or container-forwarding bypass is present.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both entrypoints guarded before mutable state/config side effects, real and effective UID covered, intentional container forwarding preserved, and DigitalOcean/1-Click guidance plus changelog aligned.

## Reproduction Assessment

Yes. #67478 gives deterministic root SSH and doctor steps, and current main still lets both CLI entrypoints reach startup while mutable state resolves under the invoking user's home directory.

## Solution Assessment

No, not yet. A root guard is a plausible narrow fix, but the branch should first update the DigitalOcean setup path, add the required changelog entry, and make the geteuid test independent of the host runner UID.
