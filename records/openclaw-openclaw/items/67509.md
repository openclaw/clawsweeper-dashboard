## Summary

Keep open. Current main still does not implement the requested root-execution guard, and this PR is a focused implementation candidate, but it is not merge-ready because the documented DigitalOcean setup flow still runs mutating OpenClaw commands from a root SSH session.

## What This Changes

The PR adds a CLI root guard that blocks real or effective UID 0 unless explicitly overridden or container-forwarded, wires it into the modern and legacy CLI entrypoints, and adds unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with docs and changelog aligned: guard both CLI entrypoints before mutable startup side effects, check real and effective UID, preserve intentional container forwarding, and document the supported non-root or explicit-override workflow for DigitalOcean users.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH reproduction steps, and current main still lets root enter CLI startup while state defaults through the invoking user's home directory.

## Solution Assessment

No, not as submitted. The root guard is a plausible narrow code fix, but the PR needs DigitalOcean install guidance and release-note alignment before it is the best maintainable solution.
