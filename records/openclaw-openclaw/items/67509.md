## Summary

Keep open. Current main still lacks the root-execution guard this PR targets, and the linked root/1-Click failure is a real implementation candidate, but the patch should not merge as-is because it changes global CLI behavior while the DigitalOcean/VPS setup docs still instruct root-session OpenClaw commands and the required changelog entry is missing.

## What This Changes

The PR adds a CLI root guard that blocks real or effective UID 0 unless an explicit root override or container-forwarding hints are present, wires it into modern and legacy entrypoints, and adds unit tests for the guard behavior.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with matching docs and release-note coverage: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root service-user or wrapper workflow plus the explicit override.

## Reproduction Assessment

Yes. The linked bug gives deterministic root SSH reproduction on a DigitalOcean 1-Click image, and current main shows root invocations still enter CLI startup with default state under ~/.openclaw and user-systemd repair paths before any root refusal exists.

## Solution Assessment

No, not as submitted. A core guard is a plausible narrow fix for the corruption path, but the PR must also reconcile the root-based DigitalOcean/VPS setup flow or intentionally scope the guard to the install/image behavior maintainers want to support.
