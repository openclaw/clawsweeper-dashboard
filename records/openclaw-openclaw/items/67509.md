## Summary

Keep open: the guard is a plausible fix for the linked root-run corruption bug, but this PR is not merge-ready while published DigitalOcean setup docs still instruct root-session `openclaw` commands and the required changelog entry is missing.

## What This Changes

The PR adds an `assertNotRoot` CLI guard with UID/EUID checks, explicit root and container-forwarding bypasses, tests, and calls from both modern and legacy CLI entrypoints.

## Best Possible Solution

Decide the intended root-execution policy, then land the early UID/EUID guard with matching DigitalOcean/VPS guidance and changelog coverage, or narrow the guard to the affected managed-image wrapper path.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to `~/.openclaw`, and the linked bug gives a deterministic root-shell DigitalOcean path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard direction is plausible, but it must be reconciled with published root-session setup docs and the required changelog entry before merge.
