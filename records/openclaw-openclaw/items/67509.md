## Summary

Keep open: the PR addresses a real source-level root-state corruption path, but the submitted global refusal still conflicts with published DigitalOcean root setup docs and lacks required changelog coverage.

## What This Changes

Adds a CLI root guard module, wires it into modern and legacy entrypoints, and tests UID/EUID plus explicit root and container-forwarding bypass behavior.

## Best Possible Solution

Land a root-execution policy that protects the 1-Click image while keeping install docs and changelog aligned, either by documenting a supported non-root VPS flow or narrowing the guard to managed-image usage.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to `~/.openclaw`, and #67478 gives a deterministic DigitalOcean root-shell path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. The guard direction is plausible, but the PR must be reconciled with published root-based DigitalOcean setup docs and changelog coverage, or narrowed to the managed-image path.
