## Summary

Keep open: the guard addresses a real root-state corruption path, but the current branch would newly refuse commands still documented for a root DigitalOcean session and lacks required changelog coverage.

## What This Changes

The PR adds an `assertNotRoot` CLI guard with UID/EUID checks, explicit root and container-forwarding bypasses, tests, and calls from both modern and legacy CLI entrypoints.

## Best Possible Solution

Decide the intended root-execution policy, then land the early UID/EUID guard with matching DigitalOcean/VPS guidance and changelog coverage or narrow the guard to the affected managed-image path.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to `~/.openclaw`, and the linked bug gives a deterministic root-shell DigitalOcean path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as a complete merge candidate. The guard direction is plausible, but it must first be reconciled with published root-session setup docs and the required changelog entry.
