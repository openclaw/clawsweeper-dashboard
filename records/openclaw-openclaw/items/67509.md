## Summary

Keep open: this PR targets a real root-state corruption path, but the submitted global root refusal still conflicts with published root-based DigitalOcean setup docs and lacks the required user-facing changelog entry.

## What This Changes

Adds a CLI root guard to modern and legacy entrypoints with UID/EUID checks, explicit root and container-forwarding bypasses, and focused root-guard tests.

## Best Possible Solution

Decide the intended root-execution policy, then land either a global guard with matching install docs and changelog or a narrower managed-image guard that prevents #67478 without breaking documented VPS setup.

## Reproduction Assessment

Yes, at source level. Current main has no UID/EUID guard in either CLI entrypoint, state defaults to ~/.openclaw, and #67478 gives a deterministic root-shell DigitalOcean path; I did not run the destructive droplet scenario.

## Solution Assessment

No, not as submitted. The guard direction is plausible, but it needs maintainer policy alignment with published root-based setup docs plus changelog coverage before merge.
