## Summary

Keep open. Current main and the latest release still expose Active Memory as a blocking embedded-agent recall path with no accepted single-shot mode, and the related open prompt-envelope PR plus broader reliability issue do not replace this narrow product/API request.

## What This Changes

Request an opt-in Active Memory single-shot mode that bypasses the embedded agent loop and produces preflight memory injection with one memory search plus one LLM call.

## Best Possible Solution

Keep this as the narrow tracker for an opt-in no-embedded-agent Active Memory mode, with maintainer-approved config semantics and regression coverage that preserves the existing blocking mode for recall-heavy use cases.

## Reproduction Assessment

Yes, at source level: current main has no accepted Active Memory mode selector and still routes eligible recall through the embedded agent runner. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainers choose the API semantics. The single-shot direction is plausible, but a maintainable fix should make the new path explicitly opt-in, preserve the existing blocking mode, and define how direct memory search plus the LLM call are routed.
