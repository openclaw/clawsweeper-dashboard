## Summary

Keep open. Current main still has Active Memory as a blocking embedded-agent recall path with no accepted single-shot mode, and the related open reliability threads and prompt-trimming PR do not supersede this narrower product/API request.

## What This Changes

Request an opt-in Active Memory config mode that performs one direct memory search plus one LLM completion instead of running the embedded recall agent loop.

## Best Possible Solution

Keep this as the canonical tracker for an opt-in Active Memory single-shot recall mode, implemented inside the plugin after maintainers choose the config/API semantics and validation bar.

## Reproduction Assessment

Yes, source-level: current main has no accepted Active Memory mode selector and still routes eligible recall through the blocking embedded-agent runner. I did not rerun the live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainers choose the product semantics. The single-shot direction is plausible because current main has memory-search and runtime LLM completion seams, but it changes recall quality, config surface, and provider/model policy.
