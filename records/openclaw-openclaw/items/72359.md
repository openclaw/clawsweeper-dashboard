## Summary

Keep open. Current main and v2026.6.10 still expose Active Memory as a blocking embedded-agent recall path with no accepted single-shot mode, and the open related prompt-trimming PR does not implement this narrower config/API request.

## What This Changes

Request an opt-in Active Memory single-shot recall mode that bypasses the embedded agent loop for lower-latency preflight memory injection.

## Best Possible Solution

Keep this as the canonical tracker for an opt-in Active Memory single-shot recall mode, implemented inside the plugin after maintainers choose the config/API semantics and validation bar.

## Reproduction Assessment

Yes, source-level: current main has no Active Memory mode selector and still routes eligible recall through the blocking embedded-agent runner. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainers choose the product semantics. The single-shot direction is plausible, but it adds config surface and changes recall quality, provider/model policy, and the plugin runtime path.
