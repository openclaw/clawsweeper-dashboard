## Summary

Keep open. Current main still lacks the requested opt-in Active Memory single-shot mode: config, docs, and runtime still describe and execute the blocking embedded memory sub-agent path, even though related latency work has improved the existing path.

## What This Changes

Request an opt-in Active Memory config mode that performs one direct memory search and one LLM completion instead of the embedded memory sub-agent loop.

## Best Possible Solution

Add a backward-compatible Active Memory mode selector if maintainers want this product path, keep embedded recall as the default, implement single-shot through plugin-owned memory search plus api.runtime.llm.complete, and update schema, docs, and focused tests together.

## Reproduction Assessment

Yes for the missing capability: current main has no accepted config.mode or single-shot schema entry, no normalized mode discriminator, and eligible turns still await maybeResolveActiveRecall. I did not run a live latency benchmark because source inspection is enough to prove the requested mode is absent.

## Solution Assessment

Unclear pending maintainer product approval. The requested opt-in mode is plausible and now has an available api.runtime.llm.complete seam, but it still changes the Active Memory config contract and recall semantics.
