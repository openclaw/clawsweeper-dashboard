## Summary

Keep open. Current main still implements Active Memory as a bounded blocking recall sub-agent with timeout/circuit-breaker mitigations, but it does not provide the requested opt-in single-shot mode that performs one memory search and one direct model completion without `runEmbeddedPiAgent`. The related timeout work from #72347 is not enough to satisfy this architecture request, and #65309 remains an open symptom tracker for foreground Active Memory latency.

## What This Changes

Request an opt-in Active Memory `single-shot` mode that performs one memory/QMD search and one model call without the embedded agent loop for low-latency preflight recall.

## Best Possible Solution

Add a documented, backward-compatible Active Memory mode with the current embedded sub-agent as the default and a single-shot branch for low-latency recall. The branch should use the plugin-safe memory search facade, perform one bounded direct model completion, return `NONE` or a capped summary, update manifest/schema/docs/changelog, and keep recall-heavy/contextual embedded behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `mode: "single-shot"` is not accepted by the strict manifest schema, is not normalized into runtime config, and eligible prompt-build turns still await the embedded Active Memory recall path.

## Solution Assessment

Likely yes. An opt-in single-shot mode is the narrowest product shape for the reported preference-only/preflight latency case, but the exact implementation should be reviewed because the current public plugin runtime lacks an obvious direct completion helper.
