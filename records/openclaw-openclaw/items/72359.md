## Summary

Keep open. Current main still has no Active Memory mode selector or single-shot recall branch, and eligible turns still use the blocking embedded memory sub-agent; the request is a real feature/config and recall-architecture decision, not conservative cleanup.

## What This Changes

Request an opt-in Active Memory config mode that performs one memory search and one LLM completion instead of running an embedded Pi agent loop.

## Best Possible Solution

Add a backward-compatible opt-in Active Memory mode selector, keep the embedded sub-agent as the default, implement the single-shot branch through existing memory-search and runtime LLM seams, and document/test the latency and recall-quality tradeoff.

## Reproduction Assessment

Yes, at source level for the missing capability: current main has no accepted config.mode, no normalized discriminator, and no direct search-plus-completion branch. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed single-shot path is plausible with existing llm.complete and memory-search seams, but mode naming, defaults, provider compatibility, and recall-quality tradeoffs need approval.
