## Summary

Keep open. Current main still lacks the requested opt-in Active Memory single-shot mode: the strict plugin schema has no `config.mode`, config normalization has no discriminator, and eligible turns still go through the embedded memory sub-agent path.

## What This Changes

The issue requests an opt-in Active Memory single-shot mode that avoids the embedded agent loop for lower-latency preflight memory injection.

## Best Possible Solution

Add a backward-compatible Active Memory mode selector, keep the embedded sub-agent as the default, and implement the opt-in single-shot path through public memory-search and `api.runtime.llm.complete` seams with docs and focused tests.

## Reproduction Assessment

Yes for the missing capability: current main has no accepted or normalized `config.mode`, and eligible prompt-build hooks still call the embedded recall path. I did not run a live latency benchmark because source inspection is enough to show the requested mode is absent.

## Solution Assessment

Unclear pending maintainer approval. A single-shot mode is a plausible narrow direction now that public LLM and memory-search seams exist, but the user-facing config contract and recall-quality tradeoff need maintainer product/API judgment first.
