## Summary

Keep open. Current main still lacks the requested opt-in Active Memory single-shot mode: the strict schema has no `config.mode`, normalization has no discriminator, and eligible turns still await the embedded memory sub-agent path. The related hard-timeout and bounded-query work helped the same latency family, but it did not implement this architecture change.

## What This Changes

The issue requests an opt-in Active Memory single-shot mode that performs one memory search and one LLM completion instead of using the embedded agent loop.

## Best Possible Solution

Add a backward-compatible Active Memory mode selector, keep the embedded sub-agent as the default, and implement the opt-in single-shot path through public memory-search and `api.runtime.llm.complete` seams with docs and focused tests.

## Reproduction Assessment

Yes for the missing capability: current main has no accepted or normalized Active Memory mode value, and eligible prompt-build hooks still call the embedded recall path. I did not rerun a live Telegram/provider latency benchmark because source inspection is enough to show the requested mode is absent.

## Solution Assessment

Unclear pending maintainer approval. A single-shot mode is a plausible narrow direction now that public LLM and memory-search seams exist, but the user-facing config contract, fallback semantics, and recall-quality tradeoff need product/API judgment first.
