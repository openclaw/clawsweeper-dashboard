## Summary

Keep open. Current main and the latest release still expose Active Memory as a blocking embedded-agent recall path with no accepted single-shot mode, while related reliability items and prompt-trimming work do not implement this narrower product/API request.

## What This Changes

The issue requests an opt-in Active Memory single-shot recall mode that avoids the embedded agent loop for lower-latency preflight context injection.

## Best Possible Solution

Keep this as the canonical tracker for an opt-in Active Memory single-shot recall mode, implemented inside the plugin after maintainers choose the config/API semantics and validation bar.

## Reproduction Assessment

Yes, source-level: current main has no Active Memory mode selector and still routes eligible recall through the blocking embedded-agent runner. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainers choose the product semantics. The single-shot direction is plausible, but it adds config surface and changes recall quality, provider/model policy, and the plugin's runtime path.
