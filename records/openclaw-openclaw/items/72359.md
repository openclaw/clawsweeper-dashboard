## Summary

Keep open. Current main and latest release still expose Active Memory as a blocking embedded-agent recall path with no accepted single-shot mode, and the related open reliability issue and prompt-trimming PR do not supersede this narrow product/API request.

## What This Changes

The issue requests an opt-in Active Memory single-shot recall mode that avoids the embedded agent loop for lower-latency preflight memory injection.

## Best Possible Solution

Keep this issue as the narrow product/API tracker for an opt-in no-embedded-agent Active Memory recall mode, coordinated with the broader Active Memory reliability investigation.

## Reproduction Assessment

Yes, at source level: current main has no accepted Active Memory mode selector and still routes eligible recall through the embedded agent runner. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainers choose the API semantics. The proposed single-shot path is plausible, but adding a new config mode and bypassing tool-based recall affects recall quality and provider/runtime policy.
