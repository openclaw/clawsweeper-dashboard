## Summary

Keep open. Current main and v2026.6.10 still expose Active Memory as a blocking embedded-agent recall path with no accepted single-shot config mode, and the related open PR only trims recall prompt input rather than replacing the embedded loop.

## What This Changes

Request an opt-in Active Memory single-shot mode that performs one memory search and one LLM call instead of the current embedded recall agent loop.

## Best Possible Solution

Add an Active Memory plugin-owned, opt-in single-shot/preflight mode only after maintainers approve the config shape, direct model-call seam, docs, tests, and live latency proof while preserving the current blocking mode as an explicit behavior.

## Reproduction Assessment

Yes at source level. Current main has no Active Memory single-shot config mode and still awaits the embedded recall path inside before_prompt_build; I did not run a live Telegram latency repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product shape. A single-shot mode is a plausible narrow solution, but the best implementation needs an approved config/API contract and direct model-call boundary rather than an ad hoc second path.
