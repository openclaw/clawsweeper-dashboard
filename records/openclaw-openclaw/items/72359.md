## Summary

Keep open. Current main still rejects/ignores any Active Memory `single-shot` mode and eligible prompt-build turns still route through the embedded recall sub-agent; #72347 addressed timeout enforcement and #65309 tracks broader latency, but this narrower opt-in architecture request remains valid and actionable.

## What This Changes

Add an opt-in Active Memory single-shot mode that performs one memory search plus one bounded LLM completion instead of running the embedded agent loop.

## Best Possible Solution

Add a backward-compatible `config.mode` with the existing embedded recall as the default and an opt-in `single-shot` branch that uses plugin-owned memory search and simple completion seams, with docs, manifest schema, tests, and changelog aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `mode: "single-shot"` is not in the strict manifest schema, is not normalized into runtime config, and eligible prompt-build turns still call the embedded recall path.

## Solution Assessment

Yes. An opt-in single-shot mode is the narrowest maintainable path for low-latency preflight recall; replacing the embedded default would be riskier because existing recall-heavy/contextual behavior depends on tool-driven recall.

## Repair Work Prompt

Implement a backward-compatible Active Memory `plugins.entries.active-memory.config.mode` option with the existing embedded sub-agent behavior as the default and an opt-in `single-shot` mode for low-latency preflight recall. Keep the change extension-owned under `extensions/active-memory`; do not move Active Memory policy into core and do not change the default embedded path for recall-heavy/contextual behavior. The single-shot branch should derive the query with the existing query-mode helpers, perform one bounded memory search through the public memory-host search seam, build a prompt containing the user message/query plus compact hits, call the public simple-completion runtime once with the resolved Active Memory model and timeout signal, then return `NONE` or a `maxSummaryChars`-capped summary through the existing hidden prompt-prefix/status/debug flow. Update the manifest schema/UI hints, runtime config types/normalization, tests for manifest acceptance/rejection and runtime behavior, docs at `docs/concepts/active-memory.md`, and add a user-facing `CHANGELOG.md` entry crediting the reporter when appropriate. Preserve existing cache, circuit-breaker, logging, transcript, and timeout semantics unless a test shows a single-shot-specific adjustment is needed.
