## Summary

Keep open. Current main still lacks the requested opt-in Active Memory single-shot mode; the schema, normalized config, docs, and prompt hook remain centered on the blocking embedded recall sub-agent path, and related #65309 still tracks the broader latency symptoms rather than superseding this specific design request.

## What This Changes

Request an opt-in Active Memory config mode that performs one memory search and one LLM completion instead of using the embedded recall agent loop.

## Best Possible Solution

Add a backward-compatible Active Memory mode config with the current embedded recall path as default and an opt-in single-shot branch that uses the memory search manager plus one bounded provider completion, with docs and focused regression coverage.

## Reproduction Assessment

Yes for the missing capability. Current main has no accepted or normalized `config.mode`, and eligible prompt-build hooks still await `maybeResolveActiveRecall`, which calls `runEmbeddedPiAgent`; I did not rerun live latency measurements.

## Solution Assessment

Yes directionally: an opt-in plugin-owned single-shot mode is the narrowest user-facing behavior while preserving the existing recall-heavy embedded path. The exact implementation still needs maintainer API judgment because current plugin runtime inspection did not show an obvious direct one-shot LLM completion helper.
