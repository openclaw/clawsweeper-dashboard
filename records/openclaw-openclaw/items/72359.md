## Summary

Keep open: current main and the latest release still expose Active Memory as a blocking embedded memory sub-agent with no `single-shot` mode, and the related open Codex/Active Memory issue adds current production evidence rather than superseding this narrower request.

## What This Changes

The issue requests an opt-in Active Memory single-shot mode that avoids the embedded agent loop for lower-latency preflight memory injection.

## Best Possible Solution

Define a maintainer-approved Active Memory mode contract for low-latency recall, keep the current embedded mode intact for recall-heavy use cases, and land schema/runtime/docs/tests together.

## Reproduction Assessment

Yes from source at medium confidence: current main has no accepted single-shot config and still dispatches Active Memory through the embedded recall sub-agent. I did not run a live Telegram/Codex latency reproduction in this read-only review.

## Solution Assessment
