## Summary

Keep open. Current main still only exposes the embedded Active Memory recall path: the strict config schema, normalized runtime config, docs, and before-prompt hook have no opt-in single-shot mode, while related timeout work only bounds the existing loop instead of replacing it for low-latency preflight recall.

## What This Changes

Request an opt-in Active Memory `single-shot` mode that performs one memory search and one LLM completion instead of running the embedded recall agent loop.

## Best Possible Solution

Add a backward-compatible plugin-owned `config.mode` that defaults to the existing embedded recall path and supports an opt-in `single-shot` branch using public memory-search and simple-completion seams, with docs, manifest schema, tests, and changelog aligned.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `config.mode` is absent from the strict manifest and normalized runtime config, and eligible prompt-build turns still await the embedded recall path.

## Solution Assessment

Yes. An opt-in plugin mode is the narrowest maintainable direction because it preserves the existing embedded default for recall-heavy use cases while giving preference-style preflight recall a lighter path.

## Repair Work Prompt

Implement the Active Memory single-shot mode requested in https://github.com/openclaw/openclaw/issues/72359. Add an opt-in `plugins.entries.active-memory.config.mode` with the existing embedded recall path as the default and a `single-shot` branch that derives the query with the current query-mode logic, performs one memory search through public memory-search SDK/facade APIs, then performs one simple completion through the public simple-completion runtime to return `NONE` or one compact summary bounded by `maxSummaryChars`. Preserve fail-open behavior, timeout/abort handling, circuit-breaker semantics, logging/debug output where applicable, prompt-prefix sanitization, qmd search-mode handling, and the current embedded behavior for default/recall-heavy/contextual use. Update manifest schema/UI hints, runtime normalization, Active Memory tests, config schema tests, docs, and CHANGELOG.md. Do not deep-import core internals or memory-core private files; if the public SDK surfaces are insufficient, stop and request an additive SDK seam instead.
