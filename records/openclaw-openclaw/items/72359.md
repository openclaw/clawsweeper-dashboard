## Summary

Keep open. Current main and v2026.5.18 still expose Active Memory as a blocking embedded sub-agent with no opt-in single-shot mode; the related reliability tracker and draft PR address timing/QMD startup mitigations, not this requested no-agent-loop path.

## What This Changes

Request an opt-in Active Memory single-shot mode that performs one memory search plus one LLM completion without `runEmbeddedPiAgent`.

## Best Possible Solution

Add a backward-compatible opt-in Active Memory mode selector, keep the embedded sub-agent as the default, implement the single-shot branch through existing memory-search and runtime LLM seams, and document/test the latency versus recall-quality tradeoff.

## Reproduction Assessment

Yes, at source level: current main has no accepted `config.mode`, no normalized discriminator, and eligible Active Memory recall still calls `runEmbeddedPiAgent`. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainer product/API review. The proposed single-shot path is plausible because runtime LLM completion and memory-search seams exist, but current timeout and reliability mitigations do not replace the requested no-agent-loop mode.
