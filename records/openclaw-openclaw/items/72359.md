## Summary

Keep open. Current main and the latest shipped release still expose Active Memory as a blocking embedded sub-agent with no `config.mode` or single-shot search-plus-completion branch; the related reliability work at https://github.com/openclaw/openclaw/issues/72015 and draft https://github.com/openclaw/openclaw/pull/73667 addresses latency budgets/QMD startup, not this requested no-agent-loop mode.

## What This Changes

The issue requests an opt-in Active Memory single-shot mode that performs one memory search and one LLM call instead of running the embedded agent loop.

## Best Possible Solution

Add a backward-compatible opt-in Active Memory mode selector, keep the embedded sub-agent as the default, implement the single-shot branch through existing query/memory-search and `api.runtime.llm.complete` seams, and document/test the latency and recall-quality tradeoff.

## Reproduction Assessment

Yes, at source level: current main has no accepted `config.mode`, no normalized discriminator, and eligible Active Memory recall still calls `runEmbeddedPiAgent`. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear until maintainer product/API review. The proposed single-shot path is plausible because runtime LLM completion exists, but current timeout and reliability mitigations do not replace the requested no-agent-loop mode.
