## Summary

Keep open. Current main still has no Active Memory single-shot mode, and eligible preflight turns still await the embedded recall sub-agent; the hard-timeout work from #72347 does not implement the requested low-latency architecture.

## What This Changes

Request an opt-in Active Memory `config.mode: "single-shot"` that performs one memory search and one LLM completion instead of running the embedded recall agent loop.

## Best Possible Solution

Add a backward-compatible plugin-owned `config.mode` with embedded recall as the default and an opt-in single-shot branch using existing memory search and simple-completion seams.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `mode: "single-shot"` is absent from the strict manifest and normalized runtime config, and eligible prompt-build turns still call the embedded recall path.

## Solution Assessment

Yes. An opt-in single-shot mode is the narrowest maintainable solution for preference-style preflight latency; replacing the embedded default would be riskier for recall-heavy and contextual use cases.

## Repair Work Prompt

Implement the Active Memory single-shot mode requested in https://github.com/openclaw/openclaw/issues/72359. Add a backward-compatible `plugins.entries.active-memory.config.mode` enum with the current embedded-agent behavior as the default and an opt-in `single-shot` value. For `single-shot`, preserve existing eligibility, query derivation, timeout/fail-open, cache/circuit-breaker/status, hidden prompt-prefix, model resolution, and `maxSummaryChars` contracts. Use one plugin-owned memory search through the public memory-host search seam and one simple completion through the public simple-completion runtime seam; do not call `runEmbeddedPiAgent`, expose tools, or create embedded-agent transcripts in this mode. Build a focused prompt from the query plus bounded search hits that returns exactly `NONE` or one plain-text summary within `maxSummaryChars`. Update `extensions/active-memory/openclaw.plugin.json`, Active Memory runtime/tests, manifest config tests, `docs/concepts/active-memory.md`, and `CHANGELOG.md`. Do not move Active Memory behavior into core except through existing documented plugin SDK seams; avoid memory-core internals unless a minimal typed adapter is truly required.
