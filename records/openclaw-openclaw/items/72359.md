## Summary

Keep open. Current main still implements Active Memory as a bounded blocking recall sub-agent with hard-timeout, cache, and circuit-breaker mitigations, but it does not provide the requested opt-in single-shot path that performs one direct memory query and one model completion without the embedded agent loop.

## What This Changes

The issue requests an opt-in Active Memory single-shot mode that performs one memory search and one LLM completion instead of the current embedded-agent recall loop.

## Best Possible Solution

Add a documented, backward-compatible Active Memory config mode with the current embedded-agent behavior as the default and a single-shot branch for low-latency preflight recall. The new branch should stay inside the plugin/runtime boundary, use the plugin-safe memory search facade, perform one bounded model completion, return NONE or a capped summary, and leave recall-heavy/contextual multi-turn behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: adding mode: "single-shot" is not accepted by the strict manifest schema or read by runtime config, and an eligible prompt-build path still awaits maybeResolveActiveRecall followed by runEmbeddedPiAgent.

## Solution Assessment

Likely yes. An opt-in single-shot mode is the narrowest maintainable direction for the preference-only/preflight latency case, provided any missing direct-completion runtime seam is kept small, documented, and backward-compatible.
