## Summary

Keep open. Current main still implements Active Memory as a bounded blocking embedded recall sub-agent and has timeout/cache/circuit-breaker mitigations, but it does not provide the requested opt-in single-shot mode that performs one memory search plus one direct model completion without `runEmbeddedPiAgent`.

## What This Changes

Request adding an opt-in Active Memory single-shot mode that bypasses the embedded recall sub-agent and returns `NONE` or a capped summary from one memory search plus one model call.

## Best Possible Solution

Add a documented, backward-compatible Active Memory `config.mode` with the current embedded sub-agent behavior as the default and a `single-shot` branch for low-latency preference/preflight recall. The new branch should use public SDK memory-search and simple-completion seams, perform one bounded search plus one bounded completion, return `NONE` or a `maxSummaryChars`-capped summary, and leave recall-heavy/contextual embedded behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `mode: "single-shot"` is not accepted by the strict manifest schema, is not normalized into runtime config, and eligible prompt-build turns still await the embedded recall path.

## Solution Assessment

Yes. An opt-in single-shot mode is the narrowest maintainable solution for the preference-only/preflight latency case; replacing the default embedded sub-agent would be riskier because recall-heavy and contextual styles still benefit from tool-driven behavior.
