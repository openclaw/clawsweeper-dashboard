## Summary

Keep open. Current main still has no Active Memory `config.mode` or single-shot recall branch, and eligible turns still route through the embedded memory sub-agent; the remaining work is a user-facing config and recall-architecture decision rather than conservative cleanup.

## What This Changes

Request an opt-in Active Memory `single-shot` mode that performs one QMD search plus one LLM completion instead of running the embedded memory sub-agent loop.

## Best Possible Solution

Add a backward-compatible opt-in Active Memory mode selector, keep the embedded sub-agent as the default, implement the single-shot branch through existing memory-search and `api.runtime.llm.complete` seams, and document/test the latency and recall-quality tradeoff.

## Reproduction Assessment

Yes for the missing capability: current main has no accepted `config.mode`, no normalized discriminator, and the eligible prompt hook still awaits the embedded recall path. I did not rerun the reporter's live Telegram/provider latency benchmark.

## Solution Assessment

Unclear pending maintainer product/API review. A single-shot path is plausible and narrow using existing `api.runtime.llm.complete` and memory-search seams, but mode naming, defaults, fallback behavior, and recall-quality tradeoffs need approval.
