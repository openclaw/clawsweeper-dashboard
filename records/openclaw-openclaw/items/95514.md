## Best Possible Solution

Keep the narrow Feishu HTTP-wrapper normalization, but convert Buffer media parts through a type-correct BlobPart-compatible byte representation and rerun exact-head extension checks before merge.

## Reproduction Assessment

No independent live replay in this read-only review. The linked issue reports deterministic Feishu 2026.6.6 media-send failures, and current source traces `filePath`/`media` sends into Lark SDK Buffer multipart uploads through the timeout wrapper.

## Solution Assessment

No as submitted: the wrapper boundary is the right place to normalize the SDK multipart request, but `new Blob([value])` is not type-correct under the current Node/DOM type contract. The safer fix is to keep the same narrow normalization and convert the Buffer to a BlobPart-compatible byte view with type coverage.
