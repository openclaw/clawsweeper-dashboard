## Summary

Keep open: the patch targets a plausible migration-provider gap, but it currently bypasses the documented global plugin-disable policy for non-bundled native plugins and has no after-fix real behavior proof.

## What This Changes

The PR changes manifest contract resolution and migration provider loading so migrationProviders can be discovered and force-enabled even when installed plugins are disabled, with regression tests for disabled installed providers.

## Best Possible Solution

Preserve disabled-plugin trust defaults and either scope automatic migration-provider compat to bundled providers or add an explicit documented opt-in for disabled external migration providers with upgrade and security proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the merge risk: the PR's own disabled-installed-provider case would make compat config set `plugins.enabled=true` and enable that plugin during migration loading. I did not run tests because this review is read-only.

## Solution Assessment

No. The current patch broadens a bundled-compat helper to all migration provider plugin ids; the maintainable path is bundled-only compat or an explicit opt-in policy for external disabled providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Do not force-enable disabled external migration plugins:** `src/plugins/migration-provider-runtime.ts:71`
  - body: Passing `resolution.pluginIds` into the bundled compat config makes `openclaw migrate` turn `plugins.enabled: false` into `plugins.enabled: true` and add enablement entries for installed migration plugins. That bypasses the documented global plugin-disable switch for non-bundled native plugins, which are in-process code, so a disabled external plugin can be imported by a migration/list command. Keep the compat override limited to `resolution.bundledCompatPluginIds` or add an explicit opt-in path for external migration providers.
