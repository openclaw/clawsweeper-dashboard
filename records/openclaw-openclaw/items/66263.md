## Summary

Keep open: current main fixes some stale-token display and updater paths, but the Codex-native compaction path still starts app-server compaction without a post-count or explicit unknown marker, and gateway transcript fallback can still promote stale usage as fresh.

## What This Changes

The issue asks OpenClaw to treat Codex-native post-compaction context usage as fresh only with trustworthy post-compaction counts, otherwise preserving an unknown/stale state across status and transcript fallback.

## Best Possible Solution

Make one explicit compaction-boundary invariant across Codex native compaction start/completion, session store updates, gateway rows, transcript fallback, and status rendering.

## Reproduction Assessment

Yes, at source level. Current main can start Codex-native compaction without tokensAfter while leaving existing fresh session totals in place, and gateway transcript fallback can still derive fresh-looking totals without a compaction-boundary guard.

## Solution Assessment

Yes, the requested boundary invariant is the best fix shape. The current main fixes are useful but partial, so closing this as implemented would leave the native Codex/gateway accounting path unresolved.
