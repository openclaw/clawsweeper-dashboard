## Summary

Keep open: current main still has a source-reproducible Codex native compaction accounting gap because native compaction can complete without `tokensAfter`, store/update paths do not consistently persist an explicit stale/unknown state, and some fallback paths can still treat transcript-derived usage as fresh.

## What This Changes

Ensure Codex native compaction records reliable `tokensAfter` when available and otherwise preserves unknown/stale context usage without reusing pre-compaction totals as fresh.

## Best Possible Solution

Make the native compaction boundary explicit end to end: persist a finite trusted `tokensAfter`, otherwise mark the post-compaction context as unknown/stale and prevent transcript or stored pre-boundary usage from becoming fresh current context.

## Reproduction Assessment

Yes, at source level. Complete Codex native compaction without `tokensAfter`; current main can either leave old totals untouched in auto-compaction accounting or drop the explicit freshness marker in manual compaction while fallback code can still promote transcript-derived usage as fresh.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve an explicit post-compaction unknown/stale state and make transcript/status fallback boundary-aware, while coordinating with the existing broader compaction invariant and status PRs.
