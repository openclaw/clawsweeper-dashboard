## Summary

Keep open: current main and the latest release still treat Codex native compaction as a pending start without a post-compaction token count, and the gateway pending-path test preserves the old fresh total. Some stale-status and generic missing-tokensAfter handling has improved, but the Codex-native boundary and transcript/status accounting invariant requested here is not fully implemented.

## What This Changes

The issue asks OpenClaw to record reliable Codex post-compaction token usage when available, and otherwise preserve an unknown/stale context state instead of reviving pre-compaction usage.

## Best Possible Solution

Define one explicit Codex native compaction boundary that invalidates stale usage when tokensAfter is unavailable, consumes reliable post-compaction token evidence when Codex provides it, and keeps gateway/status/transcript fallback from presenting pre-boundary totals as fresh.

## Reproduction Assessment

Yes, at source level. Current main starts Codex native compaction without tokensAfter and the gateway pending-compaction test preserves a previous fresh totalTokens value afterward; I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

Yes, the requested boundary invariant is the right fix direction. It should not be closed as implemented until maintainers choose and implement a single pending/unknown-state contract across Codex, gateway rows, transcript fallback, and status output.
