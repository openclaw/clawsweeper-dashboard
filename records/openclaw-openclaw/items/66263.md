## Summary

Keep open. Current main still has the reported Codex native compaction accounting gap: the native compaction result lacks `tokensAfter`, the gateway clears the stored token fields without preserving an explicit stale/unknown state, and session listing can rebuild a fresh-looking total from transcript usage after that boundary.

## What This Changes

Request that Codex native compaction record reliable post-compaction token counts when available and otherwise keep context usage unknown/stale instead of reporting stale pre-compaction usage as fresh.

## Best Possible Solution

Honor the native compaction boundary: persist a finite reliable `tokensAfter` as fresh, otherwise mark context usage unknown/stale and prevent transcript fallback from crossing that boundary.

## Reproduction Assessment

Yes, at source level. Complete Codex native compaction without `tokensAfter`, let `sessions.compact` clear stored token freshness, then session listing can read prior assistant transcript usage and expose it as fresh context again.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve an explicit stale/unknown post-compaction state and teach transcript fallback to respect the compaction boundary, rather than fabricating a precise context percentage.
