## Summary

Keep open. Current main still has the reported Codex native compaction accounting gap: the native compaction path records completion but not a reliable post-compaction token snapshot, manual compaction clears freshness when `tokensAfter` is absent, and session listing can still recover transcript-derived pre-boundary usage as fresh. A newer generic auto-compaction fix preserves provider-side `tokensAfter` estimates, but it does not solve this Codex native missing-`tokensAfter` path.

## What This Changes

The issue requests that Codex native compaction capture reliable post-compaction token counts when available and otherwise preserve an explicit unknown/stale context state instead of displaying stale pre-compaction usage as fresh.

## Best Possible Solution

Fix the Codex plugin and gateway accounting boundary together: capture a trustworthy post-compaction usage snapshot when the app-server emits one, persist an explicit stale/unknown state when it does not, and make transcript usage fallback honor a compaction boundary so pre-compaction assistant usage cannot be promoted as current context. Keep the #64669 cumulative-usage behavior intact and coordinate the shared invariant with #61864.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a successful Codex native compaction completion without any token usage, observe that `maybeCompactCodexAppServerSession` returns no `tokensAfter`, then see `sessions.compact` remove stored freshness and session listing recover transcript usage as fresh.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. The safer path is not to fabricate a percentage: use app-server usage only when it is explicitly a current post-compaction snapshot, otherwise surface unknown/stale and block transcript fallback across the compaction boundary.
