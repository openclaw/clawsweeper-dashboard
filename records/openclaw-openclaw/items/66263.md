## Summary

Keep open: current main still has the Codex native compaction accounting gap, because a successful native compaction can return no `tokensAfter`, the session store loses the explicit stale/unknown marker, and transcript fallback can still re-promote older usage as fresh.

## What This Changes

The issue requests reliable post-Codex-native-compaction context accounting, including `tokensAfter` capture when available and explicit unknown/stale state when it is not.

## Best Possible Solution

Honor the native compaction boundary end to end: capture a finite app-server `tokensAfter` when present, otherwise persist an explicit unknown/stale state and prevent transcript fallback from crossing that boundary as fresh current usage.

## Reproduction Assessment

Yes, at source level. Complete Codex native compaction without `tokensAfter`; `sessions.compact` then clears token freshness, and session listing can derive older transcript usage and mark it fresh again.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve an explicit post-compaction unknown/stale state and make transcript fallback boundary-aware, rather than fabricating a precise context percentage from pre-compaction transcript data.
