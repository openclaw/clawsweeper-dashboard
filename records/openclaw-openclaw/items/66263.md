## Summary

Keep open: current main and the latest release still treat native Codex compaction as a pending `thread/compact/start` signal, and the gateway test preserves the old fresh token total after that pending result. The newer related issue adds useful live evidence, but it is not a maintainer-designated replacement for this broader stale/unknown post-compaction accounting issue.

## What This Changes

The issue asks OpenClaw to avoid reporting stale pre-compaction Codex context usage as fresh when native compaction completes without a reliable post-compaction token count.

## Best Possible Solution

Define one Codex-native compaction state contract that bridges reliable native completion/usage into session state, marks missing usage as unknown, and prevents status/transcript fallback from presenting pre-compaction totals as fresh.

## Reproduction Assessment

Yes, at source level: current main starts native Codex compaction as pending and the gateway pending-compaction test preserves the old fresh total afterward. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

Yes on direction, but the exact implementation needs maintainer choice because current docs/tests intentionally keep native Codex completion separate from OpenClaw completed-compaction bookkeeping. A native-specific pending/completed/unknown state bridge looks safer than pretending the existing completed-compaction path already applies.
