## Label Justifications

- `impact:session-state`: The issue is about stale or unknown context usage after compaction, which can mislead session status and continuation decisions.

## Summary

Keep open: current main and v2026.6.8 still treat Codex native compaction as a pending `thread/compact/start` request, without consuming completion/token-usage notifications or invalidating the prior fresh session total on the manual compact/status paths.

## What This Changes

The issue requests reliable Codex post-compaction context accounting: use `tokensAfter` when available, otherwise mark context usage unknown or stale instead of showing pre-compaction totals as fresh.

## Best Possible Solution

Define one Codex-native compaction boundary that invalidates stale usage when `tokensAfter` is unavailable, consumes reliable post-compaction token evidence when Codex provides it, and prevents status/transcript fallback from presenting pre-boundary totals as fresh.

## Reproduction Assessment

Yes, at source level: current main starts Codex native compaction without `tokensAfter`, has tests proving the request does not consume completion/token notifications, and the gateway pending-path test preserves the old fresh total. The closed duplicate also adds a fresh Telegram direct repro.

## Solution Assessment

Yes, the requested boundary invariant is the right solution direction. The safer implementation should not fabricate precision; it should either consume reliable Codex token evidence or mark usage stale/unknown at the native compaction boundary.
