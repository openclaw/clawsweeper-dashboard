- `impact:session-state`: The issue is about stale or unknown context usage after compaction, which can mislead session status and continuation decisions.

## Summary

Keep open: current main and the latest release still start native Codex compaction as an async pending request and preserve the prior fresh session total on the manual/gateway status paths, so the central stale-or-unknown post-compaction accounting problem remains.

## What This Changes

The issue asks OpenClaw to treat native Codex post-compaction usage as reliable only when Codex provides a fresh snapshot and otherwise mark status/context usage stale or unknown.

## Best Possible Solution

Define one native Codex compaction boundary that consumes reliable post-compaction token evidence when available, otherwise marks usage stale/unknown and prevents status or transcript fallback from presenting pre-boundary totals as fresh.

## Reproduction Assessment

Yes, at source level: current main starts Codex native compaction without `tokensAfter`, has tests proving completion/token notifications are not consumed, and preserves the previous fresh total in the gateway pending path. I did not run a live Codex session in this read-only review.

## Solution Assessment

Yes, the requested boundary invariant is the right solution direction. The current implementation solves some generic stale-token paths, but not the native Codex manual/status bridge where pending compaction can leave pre-compaction usage marked fresh.
