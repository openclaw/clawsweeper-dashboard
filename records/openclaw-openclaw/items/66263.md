## Summary

Keep this open. Current main still has the reported Codex native compaction accounting gap: native compaction completion does not capture a reliable post-compaction token count, manual compaction clears stored totals when `tokensAfter` is absent, and transcript fallback can still repopulate old usage as fresh.

## What This Changes

The issue requests reliable Codex native compaction context accounting after app-server compaction, including a trustworthy `tokensAfter` path or an explicit unknown/stale state that status reporting preserves.

## Best Possible Solution

Implement the fix at the Codex plugin/gateway accounting boundary: capture a trustworthy post-compaction current/last token snapshot when the app-server provides one, persist an explicit stale/unknown context state when it does not, and make transcript fallback honor a compaction boundary instead of reading pre-boundary usage as current.

## Reproduction Assessment

Yes. A source-level reproduction is clear: emit a successful Codex native compaction notification without any token usage, observe that `maybeCompactCodexAppServerSession` returns no `tokensAfter`, then see `sessions.compact` clear totals and session listing recover full-transcript usage as fresh.

## Solution Assessment

Yes, the issue’s requested direction is the narrowest maintainable fix. Current main’s stale-display guard only helps when `totalTokensFresh` is explicitly false; it does not prevent missing post-compaction totals from being replaced by stale transcript-derived usage.
