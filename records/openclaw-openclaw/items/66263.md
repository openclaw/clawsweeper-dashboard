## Summary

Keep open. Current main still has the Codex native compaction accounting gap: native app-server compaction reports success without a post-compaction token snapshot, manual session compaction drops stored usage freshness when `tokensAfter` is absent, and session listing can still promote transcript-derived pre-compaction usage as fresh current context.

## What This Changes

The issue asks OpenClaw to record reliable post-Codex-native-compaction context usage, or preserve an explicit unknown/stale state when `tokensAfter` is unavailable.

## Best Possible Solution

Teach the Codex native compaction path and gateway session accounting to use only trustworthy post-compaction snapshots, persist unknown/stale state when none exists, and block transcript fallback from crossing a native compaction boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to complete Codex native compaction without usage metadata, observe no `tokensAfter` in the compact result, then see session listing recover transcript usage as fresh after the store cleared the token fields.

## Solution Assessment

Yes. The requested fix is the narrow maintainable path: do not fabricate a percentage, use app-server usage only when it is a current post-compaction snapshot, and otherwise surface unknown/stale while honoring a compaction boundary.
