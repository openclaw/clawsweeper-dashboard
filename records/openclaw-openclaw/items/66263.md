## Summary

Keep open: current main and v2026.6.8 still start Codex native compaction as a pending `thread/compact/start` request without consuming completion/token-usage notifications or invalidating the prior fresh session total. The related fresh Telegram repro was closed as a duplicate of this issue, so this remains the canonical tracker.

## What This Changes

The issue requests reliable post-compaction context accounting for Codex-native compaction, including `tokensAfter` when available and unknown/stale state when it is not.

## Best Possible Solution

Define one explicit Codex-native compaction boundary that invalidates stale usage when `tokensAfter` is unavailable, consumes reliable post-compaction token evidence when Codex provides it, and prevents status/transcript fallback from presenting pre-boundary totals as fresh.

## Reproduction Assessment

Yes, at source level. Current main starts Codex native compaction without `tokensAfter`, has tests proving completion/token notifications are not consumed by that request, and the gateway pending-path test preserves a prior fresh `totalTokens` value.

## Solution Assessment

Yes, the requested boundary invariant is the right solution direction. The generic completed-compaction stale-total fix is not enough because the Codex-native path is intentionally pending and currently skips the updater that would mark totals unknown/stale.
