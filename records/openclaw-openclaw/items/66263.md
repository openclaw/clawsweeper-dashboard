## Summary

Keep open: current main and v2026.6.10 still start Codex native compaction as an async pending request and preserve the prior fresh session token total, so the stale-or-unknown post-compaction accounting problem remains. The closed duplicate has live Telegram evidence that Codex completed compaction while OpenClaw status still showed the pre-compaction total as fresh.

## What This Changes

The issue asks OpenClaw to record reliable Codex post-compaction token counts when available, and otherwise mark context usage unknown/stale instead of reviving pre-compaction totals.

## Best Possible Solution

Define a native Codex compaction boundary that consumes reliable post-compaction token evidence when available, otherwise marks usage stale/unknown and prevents status or transcript fallback from presenting pre-boundary totals as fresh context.

## Reproduction Assessment

Yes, at source level: current main starts Codex native compaction without tokensAfter, and current tests assert that pending native compaction preserves totalTokensFresh:true. I did not run a live Codex session in this read-only review.

## Solution Assessment

Yes on direction, but the exact implementation needs maintainer choice. The narrow maintainable fix is a native compaction boundary that either bridges reliable Codex completion/token evidence or explicitly keeps usage unknown until such evidence exists.
