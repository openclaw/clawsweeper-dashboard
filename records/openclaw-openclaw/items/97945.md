## Summary

Keep open: current `main` and latest release still scan raw Memory Wiki markdown for wikilinks after removing only the managed related block, so fenced-code and inline-code `[[...]]` text can still become false `Broken wikilink target` warnings.

## What This Changes

The issue asks for `openclaw wiki lint` to stop treating Obsidian-style `[[...]]` text inside Markdown code fences and inline code spans as broken wikilinks.

## Best Possible Solution

Land a focused Memory Wiki extractor fix that masks or skips fenced code blocks and inline code before wikilink extraction, with regression coverage for fenced Scala/bash examples and inline code spans.

## Reproduction Assessment

Yes from source inspection: current `main` runs the wikilink regex over raw markdown without stripping fenced or inline code, then reports those extracted targets as broken links. I did not run the CLI because this review is read-only and must not create checkout artifacts.

## Solution Assessment

Yes for the focused issue: masking Markdown code fences and inline code inside the Memory Wiki extractor is the narrow owner-boundary fix. The broader title-resolution behavior from the older issue should stay with its separate PR unless maintainers intentionally expand this fix.
