## Summary

Keep open: current main and the latest release still have the Memory Wiki extractor path that scans code-bearing markdown, and an open PR now formally owns the fix path, so this issue should close only after a fix PR lands.

## What This Changes

The issue requests that Memory Wiki lint stop reporting wikilink-like text inside fenced code blocks and inline code spans as broken wikilink targets.

## Best Possible Solution

Land one focused Memory Wiki extractor fix that masks fenced code blocks and inline code before link extraction, then leave title/alias/source-path resolver normalization to the separate resolver issue.

## Reproduction Assessment

Yes, at source level: current main extracts wikilinks from markdown after removing only the related block, and lint reports unknown extracted targets as broken links. I did not run the CLI because this review is read-only and `wiki lint` can write reports/logs.

## Solution Assessment

Yes for the narrow issue: masking Markdown code spans before Memory Wiki stores `linkTargets` is the right owner boundary. The active fix PRs should stay scoped to extractor markdown-awareness and not fold in the separate resolver-normalization work.
