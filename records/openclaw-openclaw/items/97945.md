## Summary

Keep open: current `main` and v2026.6.10 still run Memory Wiki wikilink extraction over markdown that includes fenced and inline code, while https://github.com/openclaw/openclaw/pull/97954 is an open candidate fix and should land before this issue closes.

## What This Changes

The issue reports false `broken-wikilink` warnings from Memory Wiki lint when `[[...]]` text appears inside fenced code blocks or inline code spans.

## Best Possible Solution

Land the focused extractor masking fix in https://github.com/openclaw/openclaw/pull/97954, then close this issue once the fix reaches `main`; keep resolver normalization tracked separately.

## Reproduction Assessment

Yes, at source level: current `extractWikiLinks` removes only the related block before matching `[[...]]`, and lint reports unmatched extracted targets as `broken-wikilink`. I did not run `openclaw wiki lint` because this review is read-only and that command writes reports and logs.

## Solution Assessment

Yes for the focused issue: masking fenced and inline code before Memory Wiki link extraction is the narrow owner-boundary fix. Title, alias, slug, and source-path resolver normalization should remain separate unless maintainers choose to merge scopes.
