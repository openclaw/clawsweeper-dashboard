## Summary

Keep open: current `main` and v2026.6.10 still run Memory Wiki wikilink extraction over raw markdown after removing only the managed related block, so fenced-code and inline-code `[[...]]` text can still become false `Broken wikilink target` warnings.

## What This Changes

The issue asks for Memory Wiki lint to ignore wikilink-looking text inside fenced code blocks and inline code spans before reporting broken wikilink targets.

## Best Possible Solution

Land a focused Memory Wiki extractor fix that masks or skips fenced code blocks and inline code before wikilink extraction, with regression coverage for fenced bash/Scala examples and inline code spans.

## Reproduction Assessment

Yes, at source level: current `extractWikiLinks` runs the wikilink regexes over markdown that still includes fenced and inline code, then lint reports unmatched extracted targets. I did not run the CLI because this review is read-only and `wiki lint` can write reports/logs.

## Solution Assessment

Yes for the focused issue: Markdown-code masking belongs in the Memory Wiki extractor before lint sees `linkTargets`. The title/alias resolver behavior is adjacent and should remain separate unless maintainers choose to combine the work.
