## Summary

Keep open: current main and v2026.6.10 still run Memory Wiki wikilink extraction over markdown after removing only the managed related block, so code-block or inline-code `[[...]]` text can still become false `broken-wikilink` warnings. The right cleanup path is to keep this issue open while maintainers choose and land one of the open candidate PRs that target the extractor bug.

## What This Changes

Requests that Memory Wiki `wiki lint` ignore wikilink-like `[[...]]` text inside fenced code blocks and inline code spans before reporting broken wikilink targets.

## Best Possible Solution

Land one focused Memory Wiki extractor fix that masks or skips fenced code blocks and inline code before wikilink extraction, with regression coverage; keep title, alias, and source-path resolver normalization tracked separately.

## Reproduction Assessment

Yes, source-reproducible: current `extractWikiLinks` strips only the managed related block before regex extraction, and lint reports unmatched extracted targets. I did not run the full CLI because this review is read-only and `wiki lint` can write reports or logs.

## Solution Assessment

Yes for the focused bug: the best fix is at the Memory Wiki extraction boundary before `linkTargets` reaches lint. Broader title and alias resolver normalization is adjacent and should remain separate unless maintainers choose to combine it.
