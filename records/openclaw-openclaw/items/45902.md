## Summary

Keep this PR open: current main still lacks the requested Telegram future-thread model-default behavior, but the PR branch is broad, stale against current main, and has unresolved correctness plus supply-chain review blockers.

## What This Changes

The branch adds Telegram `/model` future-thread default persistence/seeding, DM-topic thread-id routing, memory and gateway diagnostics changes, and optional Telegram live E2E/recovery scripts.

## Best Possible Solution

Land a narrow Telegram-only implementation that persists a same-chat parent future-thread model default from all `/model` paths, seeds only genuinely new Telegram forum-topic and DM-topic sessions, avoids cross-chat leakage, handles metadata pre-created rows and `direct_messages_topic`, and adds focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main has no parent future-thread default fields, the `/model` paths write only the active session, and new session creation has no parent-default lookup, so a new Telegram topic/thread cannot inherit a model set in a sibling/current thread.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best way to solve it because the branch is broad and still has concrete runtime and supply-chain blockers; a smaller Telegram-focused patch is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope main-thread defaults by Telegram chat:** `src/sessions/session-key-utils.ts:173`
  - body: For `agent:...:main:thread:<chatId>:<topicId>` keys, the helper collapses the parent to the shared `main` key. That lets `/model` in one Telegram DM chat seed future threads in another chat, so the parent key needs to retain chat scope or skip this key shape.
  - confidence: 0.91
- **[P2] Seed defaults before metadata rows block inheritance:** `src/auto-reply/reply/session.ts:474`
  - body: Telegram inbound metadata can create a session row before `initSessionState` runs. Because seeding requires `!hasExistingSessionEntry`, the first real turn for a brand-new topic can skip the parent default and start without the inherited model.
  - confidence: 0.86
- **[P2] Gate parent default writes on actual `/model` changes:** `src/auto-reply/reply/directive-handling.impl.ts:428`
  - body: The directive-only write callback persists the parent entry whenever it exists, even for non-model directives. A concurrent `/queue`, `/think`, or similar directive can write a stale parent snapshot and revert a `/model` future-thread default.
  - confidence: 0.84
- **[P2] Gate inline parent writes on actual `/model` changes:** `src/auto-reply/reply/directive-handling.persist.ts:236`
  - body: The inline directive path has the same stale-parent overwrite window: any updated inline directive writes the loaded parent entry back even when this request did not modify the parent default.
  - confidence: 0.82
- **[P2] Use live Telegram context for directive defaults:** `src/auto-reply/reply/directive-handling.impl.ts:321`
  - body: Directive-only `/model` derives the Telegram hint only from persisted session fields, so older main-scoped Telegram thread entries without `channel` or `lastChannel` skip parent default persistence even though the live context is Telegram.
  - confidence: 0.78
- **[P2] Use live Telegram context for inline defaults:** `src/auto-reply/reply/directive-handling.persist.ts:88`
  - body: Inline `/model` also uses only persisted session channel hints. Migrated main-scoped Telegram threads without those fields update the current session but never persist the future-thread default for new Telegram threads.
  - confidence: 0.77
- **[P2] Preserve remote device identity for authenticated probes:** `src/gateway/probe.ts:74`
  - body: The gateway probe change suppresses device identity for non-loopback authenticated URLs, which breaks remote paired-device token/probe flows that require the signed device identity already preserved by current main.
  - confidence: 0.8
