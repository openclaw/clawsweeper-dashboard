## Summary

Keep this PR open. Current main still does not implement the Telegram future-thread model default behavior, and the proposed branch remains unsafe to merge as-is because it has blocking correctness findings and mixes unrelated memory, gateway, and operator-tooling changes into the Telegram runtime change.

## What This Changes

The PR adds Telegram parent-chat future-thread model defaults for `/model`, support for additional Telegram DM topic thread-id payloads, optional live Telegram E2E scripts, and unrelated memory/gateway diagnostic changes.

## Best Possible Solution

Keep the PR open for a scoped maintainer revision: implement only Telegram model inheritance first, with parent defaults keyed by the real Telegram chat, first-turn seeding that tolerates metadata pre-creation, and focused tests for directive-only, inline directive, concurrency, main-scoped DM, and `direct_messages_topic` paths. Split memory, gateway, recovery, and live E2E tooling into separate PRs.

## Reproduction Assessment

Yes. The current-main code inspection gives a high-confidence reproduction path: set `/model` in a Telegram thread/topic, create a new thread in the same chat, and the new thread has no parent future-thread default to read because main has no schema field, persistence path, or seeding path for it.

## Solution Assessment

No. The proposed direction is useful, but this branch is not the best way to solve it because the Telegram runtime change is mixed with unrelated changes and the diff still has concrete scoping, seeding, stale-write, and gateway auth regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope main-thread defaults by Telegram chat:** `src/sessions/session-key-utils.ts:171`
  - body: For `agent:...:main:thread:<chatId>:<topicId>` keys, this helper collapses the root to `main`, so `/model` in one Telegram DM chat writes the future-thread default onto the shared `agent:...:main` parent. New DM topics in another chat can then inherit the wrong model; keep the chat id in the parent key or skip inheritance for this key shape.
  - confidence: 0.91
- **[P2] Seed defaults before metadata rows block inheritance:** `src/auto-reply/reply/session.ts:472`
  - body: `recordInboundSession` can create a metadata-only row for a brand-new Telegram thread before `initSessionState` runs. Because this block requires `!hasExistingSessionEntry`, that first real turn skips the parent default and the new thread starts without the inherited `/model` selection.
  - confidence: 0.86
- **[P2] Write directive parent defaults only after `/model` changes them:** `src/auto-reply/reply/directive-handling.impl.ts:428`
  - body: This callback writes the parent session whenever a Telegram parent entry exists, even for non-model directives. A concurrent `/queue`, `/think`, or similar directive can persist a stale parent snapshot and revert a `/model` default written by another request.
  - confidence: 0.84
- **[P2] Avoid stale parent writes for inline non-model directives:** `src/auto-reply/reply/directive-handling.persist.ts:236`
  - body: The inline path has the same stale-parent overwrite window: any updated inline directive writes `sessionStore[telegramParentSessionKey]` back if it was loaded, even when the parent was not changed by this request. Gate this write on the parent default actually being modified.
  - confidence: 0.82
- **[P2] Preserve device identity for remote authenticated probes:** `src/gateway/probe.ts:77`
  - body: For non-loopback gateways with `auth.token` or `auth.password`, `attachDeviceIdentity` becomes false and the probe sends `deviceIdentity: null`. That regresses paired-device authenticated probes that require device identity on remote gateways.
  - confidence: 0.8
