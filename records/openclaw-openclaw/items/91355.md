## What This Changes

The issue asks for Gateway restart shutdown to persist restart-recovery state for active main-session runs before aborting them after restart-drain timeout.

## Best Possible Solution

Land or revise the linked PR so the restart-drain timeout path captures active main-session refs and calls the existing marker best-effort before abort cleanup, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a current-main path where `close({ restartExpectedMs, drainTimeoutMs: 0 })` with an active `chatAbortControllers` entry reaches `abortActiveRunsForRestart` without any restart marker hook.

## Solution Assessment

Yes. The narrow bridge to `markRestartAbortedMainSessions` is the maintainable fix shape because the marker already owns main-session filtering and recovery state, while the close path owns the last chance before abort cleanup.
