## Label Justifications

- `P2`: The bug can interrupt an active main turn during managed restart, but the trigger is limited to restart-drain timeout rather than normal chat execution.
- `impact:session-state`: The missing `abortedLastRun` marker leaves restart recovery without durable state for the interrupted main session.
- `impact:message-loss`: The interrupted turn can produce no final user-facing conclusion or restart-recovery notice after the Gateway comes back.

## Summary

Keep open: current main still aborts active restart-drain chat runs in `src/gateway/server-close.ts` without calling the existing main-session restart-recovery marker, and the related message-final PR at https://github.com/openclaw/openclaw/pull/91333 explicitly excludes gateway restart/session continuation.

## What This Changes

The issue asks for restart shutdown to persist restart-recovery state for active main-session turns before aborting them after a managed restart drain timeout.

## Best Possible Solution

Reuse the existing main-session restart marker from the restart close/drain timeout path before aborting active chat runs, keep it best-effort with a shutdown warning on failure, and add focused regression coverage.

## Reproduction Assessment

Yes at source level: construct `createGatewayCloseHandler` with an active main-session `chatAbortControllers` entry and `drainTimeoutMs: 0`; current main aborts the run through `abortActiveRunsForRestart` without any marker bridge. I did not run a live Windows Gateway replay in this read-only review.

## Solution Assessment

Yes, the proposed narrow bridge is the best fit because `markRestartAbortedMainSessions` already owns the skip policy and durable `abortedLastRun` state. A separate recovery mechanism or Codex-specific path would be the wrong layer.
