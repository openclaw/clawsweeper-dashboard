## Summary

Keep open: current main still uses the JSON session store, while this PR adds a real SQLite-backed replacement but remains unmergeable and has blocking data-integrity defects in the SQLite fallback and migration paths.

## What This Changes

The PR adds opt-in `session.storeType: "sqlite"` support, JSON-to-SQLite migration and store-info commands, config schema/help updates, and SQLite session-store tests.

## Best Possible Solution

Repair and rebase the SQLite path so sqlite mode has one authoritative store, migration failures fail visibly, legacy rows use the same normalization as JSON loads, and the user-facing feature is documented in the active changelog.

## Reproduction Assessment

Yes for the PR defects: configure sqlite mode, then force a SQLite save or JSON-to-SQLite migration failure, or migrate a legacy JSON row containing `provider`/`lastProvider`. The source shows those paths can acknowledge JSON fallback writes that later reads ignore, hide migration failures as `no_data`, and import unnormalized legacy metadata.

## Solution Assessment

No. The SQLite direction may be useful for the reported scale problem, but this branch is not the best mergeable solution until it preserves source-of-truth semantics, surfaces migration errors, normalizes imported rows, and rebases cleanly onto the current session-store split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop writing JSON after SQLite save failures:** `src/config/sessions/store-facade.ts:152-154`
  - body: In sqlite mode this catch writes the updated store to `sessions.json` when SQLite throws, but later reads still prefer the SQLite DB. A transient `SQLITE_BUSY` or filesystem error can therefore acknowledge an update that disappears on the next load. Propagate the error or make any fallback source-of-truth switch explicit and durable.
  - confidence: 0.92
- **[P1] Surface JSON-to-SQLite migration failures:** `src/config/sessions/store-sqlite.ts:603-605`
  - body: `migrateJsonToSqlite` catches every parse/open/write failure and returns `0`, which callers report as a normal empty store. A failed first migration can leave users on an empty or stale SQLite store instead of failing the CLI command or falling back to JSON.
  - confidence: 0.91
- **[P2] Normalize legacy rows before importing them:** `src/config/sessions/store-sqlite.ts:574-589`
  - body: The current JSON loader applies `applySessionStoreMigrations` before use, but this migration inserts the raw parsed JSON entries. Older rows with legacy `provider` or `lastProvider` fields will get null indexed `channel` or `last_channel` columns, so routing/list metadata can be wrong until each session is rewritten.
  - confidence: 0.86
- **[P2] Memoize migration only after it succeeds:** `src/config/sessions/store-facade.ts:83-96`
  - body: `migrationAttempted` is set before migration work runs, and the return value from `ensureSqliteMigrated` is ignored by the load path. A transient first-run migration failure can be memoized for the rest of the process, causing later loads to skip the JSON import and read an empty or stale SQLite database.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/cli/program/register.status-health-sessions.ts:229-236`
  - body: This user-facing feature adds a new config key and `openclaw sessions migrate/store-info` commands, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to be recorded under the active version.
  - confidence: 0.95
