## Summary

Keep open: the PR is aimed at a valid cost-control bug, but the new inline-key cooldown branch is not reached by the common unprofiled billing failover paths, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials and checks that state during auth resolution and embedded-run failure handling.

## Best Possible Solution

Wire unprofiled inline-key billing/auth failures through prompt and assistant failover marking, preserve a non-secret cooldown identity, and show a redacted real inline-key billing/fallback run before merge.

## Reproduction Assessment

Yes, source-level: current main resolves inline `models.providers.<id>.apiKey` credentials without `profileId`, uses `[undefined]` as the auth candidate, and profile-id-guarded failure marking cannot persist `disabledUntil` for that path. The PR's new helper branch is similarly unreachable from the common guarded prompt and assistant failover calls.

## Solution Assessment

No: provider-level non-secret cooldown state is a reasonable direction, but this implementation does not pass unprofiled billing/auth failures into the new marker from the main failover paths. The safer fix is to remove or bypass the profile-id guards at those call sites while keeping existing profile behavior unchanged.
