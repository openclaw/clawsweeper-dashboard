## Summary

Keep open. The PR targets a valid cost-control bug, but the proposed runner integration still skips the real inline-key failure paths when there is no auth profile id, and the supplied terminal proof exercises the helper directly instead of the embedded-runner billing/auth failure flow.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, blocks disabled inline keys during auth resolution, and adds focused auth tests.

## Best Possible Solution

Keep the provider-level, non-secret cooldown identity, but route prompt, assistant, and incomplete-turn auth/billing failures through it when inline auth has no profile id, with runner-path coverage.

## Reproduction Assessment

Yes, source-level. Inline `models.providers.<id>.apiKey` credentials resolve without `profileId`, explicit inline auth uses no profile candidates, and the PR head still leaves the real failure call sites guarded on a profile id.

## Solution Assessment

No. The provider-level cooldown approach is maintainable, but this patch needs to connect the no-profile inline branch to the embedded-runner failure paths and prove that path after the fix.
