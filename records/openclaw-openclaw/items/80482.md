## Summary

Keep open: the PR is a focused fix for a real auth cooldown bug and has useful proof, but it still misses the documented env-SecretRef inline API-key path, so a common `models.providers.<id>.apiKey` setup can still bypass the new cooldown logic.

## What This Changes

The branch adds inline provider API-key cooldown usage IDs, blocks disabled inline keys in model auth, marks no-profile embedded-runner failures, and adds auth/runner regression tests.

## Best Possible Solution

Keep the provider-level cooldown approach, but make configured inline API-key provenance survive env fallback so literal keys and env SecretRefs are both marked and skipped without synthetic profiles or secret persistence.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: inline `models.providers.<id>.apiKey` auth resolves without a profile id, and the current failure marker returns before recording cooldown state when no profile id exists.

## Solution Assessment

No. The PR's provider-level cooldown identity is the right shape, but the implementation needs to handle configured env SecretRefs before ambient env auth wins so documented inline provider keys cannot bypass the cooldown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Apply inline cooldowns before env fallback:** `src/agents/model-auth.ts:779-781`
  - body: For documented configs such as MiniMax `apiKey: "${MINIMAX_API_KEY}"`, the resolver can return `env: MINIMAX_API_KEY` before this configured-key cooldown branch runs, and `hasAvailableAuthForProvider` also returns true on the ambient env check. In that path `isInlineProviderApiKeyAuth()` never sees a models.json source, so the runner does not record or block `inline-api-key:minimax` after a billing failure. Move the configured inline-key cooldown/provenance handling ahead of ambient env fallback, or otherwise distinguish configured SecretRefs from unrelated env auth.
