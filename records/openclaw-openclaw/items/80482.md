## Summary

Keep open: the PR is a focused fix for a real P1 auth-provider bug, but the latest head still leaves env-backed inline provider keys outside cooldown handling and does not invalidate warmed auth state after inline-key failures.

## What This Changes

The PR adds provider-scoped usage/backoff state for inline `models.providers.<id>.apiKey` credentials and wires embedded-runner auth/billing failures without a profile id to mark that inline key.

## Best Possible Solution

Keep the provider-level cooldown design, but apply inline credential provenance before generic env fallback, run cooldown checks in availability, invoke the auth failure hook for inline failures, and cover literal plus env SecretRef configs with focused proof.

## Reproduction Assessment

Yes, source-level: on PR head, configure `models.providers.<id>.apiKey` through a known env marker or env SecretRef, record an inline cooldown, and the resolver/availability paths can still return env auth before consulting that cooldown. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: provider-level inline cooldown state is the right direction, but the implementation must cover env-backed configured keys and reuse the existing auth failure invalidation hook before it is the best fix.
