## Summary

Keep open. The PR is aimed at a real source-reproducible billing cooldown bug, but the new inline marker is still not reached from the actual no-profile embedded-runner failure paths, and the supplied proof only exercises the helper directly.

## What This Changes

Adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, blocks disabled inline keys during auth resolution and availability checks, records inline failures from the embedded runner, and adds focused auth tests.

## Best Possible Solution

Keep the provider-level non-secret cooldown identity, but wire prompt, assistant, and incomplete-turn billing/auth failures through it even when the active inline credential has no profile id, then add runner-path coverage and proof.

## Reproduction Assessment

Yes, source-level. Inline configured provider API keys resolve without a profile id, and the PR's actual prompt, assistant, and incomplete-turn failure paths still guard failure marking behind `lastProfileId` or `failedPromptProfileId`.

## Solution Assessment

No. The provider-level cooldown approach is maintainable, but this patch needs to connect the no-profile inline branch to the real embedded-runner failure paths before it solves the reported issue.
