## Summary

Keep this PR open with changes requested: the provider-level cooldown identity is a good direction, but the patch does not yet mark the actual inline API-key prompt/assistant failure paths because those call sites still require a profile id.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, skips disabled inline keys in auth resolution/availability checks, records inline failures from the embedded runner, and adds focused auth tests.

## Best Possible Solution

Keep the provider-level, non-secret cooldown identity, but route prompt, assistant, and incomplete-turn auth/billing failures through it even when the active inline credential has no profile id.

## Reproduction Assessment

Yes, source-level. Inline `models.providers.<id>.apiKey` credentials resolve without `profileId`, and the current PR still leaves the real failure call sites guarded on a profile id before they can record cooldown state.

## Solution Assessment

No. The provider-level cooldown approach is maintainable, but this patch needs to connect the no-profile inline branch to the actual embedded-runner failure paths and add coverage for that path.
