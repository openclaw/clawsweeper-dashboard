## Summary

Keep open. The PR targets the right inline API-key billing cooldown bug, but the new profile-less `inline-api-key:<provider>` usage entry is still dropped by the existing auth store save path before `auth-state.json` is persisted or reloaded.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, checks that state during auth availability/resolution, wires no-profile runner failures into it, and adds focused auth and runner tests.

## Best Possible Solution

Preserve the non-secret inline cooldown entries through auth-state save/load, add a regression that fails on a fresh persisted store, then land the focused provider-level cooldown fix and close the linked issue.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main resolves inline `models.providers.<id>.apiKey` credentials without `profileId`, and the current runner failure marker skips failures with no profile id.

## Solution Assessment

No, not as currently written. The provider-level non-secret usage ID is the right design direction, but the persistence layer must keep those inline usageStats entries before this can solve repeated turns or fresh process loads.
