## Summary

Keep open. The provider-level cooldown direction is useful, but the PR still does not route real inline no-profile billing/auth failures from the embedded runner into the new marker, so the linked bug remains source-reproducible.

## What This Changes

The PR adds non-secret provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, skips disabled inline keys during auth resolution, and adds focused auth helper tests.

## Best Possible Solution

Keep the provider-level, non-secret cooldown identity, but call it from prompt, assistant, and incomplete-turn billing/auth failures even when the active inline credential has no profile id, then add runner-path coverage and proof.

## Reproduction Assessment

Yes, source-level. Inline `models.providers.<id>.apiKey` credentials resolve without `profileId`, and PR head still has prompt, assistant, and incomplete-turn failure paths that only invoke failure marking when a profile id exists.

## Solution Assessment

No. The provider-level cooldown approach is maintainable, but this patch needs to connect the no-profile inline branch to the actual embedded-runner failure paths and prove that path.
