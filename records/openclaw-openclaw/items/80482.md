## Summary

This is a focused fix for a real auth-provider failure loop, but it is not merge-ready: env-backed inline keys can still bypass the new cooldown and provider availability can stay stale after the inline key is marked failed.

## What This Changes

The PR adds provider-level cooldown state for inline provider API-key failures and wires auth resolution and embedded-runner failure paths to consult or mark that state.

## Best Possible Solution

Finish one provider-level inline-key cooldown path that preserves provenance for literal keys, env SecretRefs, env markers, runtime availability, and gateway auth-state invalidation, with focused regression coverage for each path.

## Reproduction Assessment

Yes, source-level. On PR head, configure models.providers.<id>.apiKey as an env ref or env marker with inline-api-key:<id> disabled in usageStats; resolver and availability paths can still return env auth before consulting the inline cooldown.

## Solution Assessment

No. The provider-level cooldown design is the right direction, but the current implementation needs to carry inline credential provenance through env-backed config and provider availability before it is the narrow maintainable fix.
