## Summary

Keep open. This PR targets the right inline API-key billing cooldown bug, but the patch still does not connect profile-less embedded-runner billing/auth failures to the new inline cooldown marker, and the supplied proof only exercises helper-level calls.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, blocks disabled inline keys during auth resolution/availability checks, records inline failures from the embedded runner, and adds focused auth tests.

## Best Possible Solution

Keep the provider-level, non-secret cooldown identity, connect it to the actual prompt, assistant, and incomplete-turn failure paths for profile-less inline auth, and preserve current explicit API-key precedence without creating synthetic auth profiles.

## Reproduction Assessment

Yes, source-level. Current main resolves inline configured provider API keys without `profileId`, uses an empty profile order for explicit inline auth, and the real failure paths still gate failure marking on a profile id.

## Solution Assessment

No. The provider-level cooldown direction is maintainable, but this patch needs to route no-profile prompt, assistant, and incomplete-turn auth/billing failures through the inline marker and then prove that runner path.
