## Summary

Keep open. The updated PR resolves the earlier no-profile call-site gap, but the new inline cooldown entry is stored under a non-profile id that the auth-store save path filters out, so the cooldown will not persist for later runs.

## What This Changes

The PR adds provider-level cooldown tracking for inline provider API keys, blocks disabled inline keys during auth resolution, wires embedded-runner auth/billing failures into that marker, and adds focused auth and runner tests.

## Best Possible Solution

Keep the provider-level non-secret cooldown identity, but preserve `inline-api-key:*` usage stats through auth-store save/load before relying on runner-path marking and auth-resolution checks.

## Reproduction Assessment

Yes, source-level. The PR writes `inline-api-key:<provider>` cooldown stats, while the save path only persists usage stats whose keys match `profiles`, so the new runner-path cooldown is dropped from persisted auth state.

## Solution Assessment

No, not yet. The provider-level cooldown direction is maintainable, but the patch must update the persistence contract so inline cooldown state survives reloads and the regression test covers that saved state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve inline cooldown stats when saving the auth store:** `src/agents/auth-profiles/usage.ts:820`
  - body: The new inline marker records cooldowns under `inline-api-key:<provider>`, but `saveAuthProfileStore()` persists a local store that drops every `usageStats` entry whose key is not present in `profiles`. Inline credentials intentionally do not create profiles, so the disabled entry is written to memory and then omitted from `auth-state.json`; the next run can reload no cooldown and retry the exhausted key.
