## Summary

Keep open: the provider-level cooldown direction is useful, but the latest head still lets env-backed inline provider keys bypass the cooldown path and leaves warmed auth availability uninvaldated after inline failure writes.

## What This Changes

The PR adds provider-level usage cooldown state for inline model provider API keys and wires embedded-runner auth/billing failures without an auth profile to mark that inline key.

## Best Possible Solution

Keep the provider-level cooldown design, but resolve configured inline API-key provenance before generic env fallback, call the existing failure hook after inline failure writes, and cover literal plus env SecretRef/marker configs with focused tests and refreshed proof.

## Reproduction Assessment

Yes, source-level. On PR head, a provider env var or env SecretRef-backed models.providers.<id>.apiKey can return through resolveConfigAwareEnvApiKey before the inline cooldown assertion and before availability checks consult inline cooldown state.

## Solution Assessment

No, not yet. Provider-level usage state is the right repair direction, but the implementation needs to preserve inline provenance across env-backed config paths and reuse the existing auth failure invalidation hook.
