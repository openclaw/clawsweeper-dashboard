## Summary

Keep open. The branch is now focused on the auth cooldown fix, but PR head still lets configured env-backed inline provider keys resolve through generic env auth before inline cooldown state is checked, so a common MiniMax-style setup can keep bypassing the new backoff path.

## What This Changes

Adds provider-level usage-state cooldowns for inline `models.providers.<id>.apiKey` credentials and wires embedded runner failures to record them, with focused auth usage/resolver tests.

## Best Possible Solution

Keep the provider-level cooldown design, but make configured inline-key provenance win before generic env fallback in resolution and availability paths, then add regression proof for literal and env-backed inline keys.

## Reproduction Assessment

Yes, source-level. Configure an inline provider API key through an env marker such as MiniMax `apiKey: "${MINIMAX_API_KEY}"`, mark the inline usage id disabled, and PR head still reaches the generic env-auth return before the inline cooldown guard.

## Solution Assessment

No. The provider-level cooldown approach is appropriate, but the implementation needs to preserve configured inline provenance for env-backed keys before generic env fallback and availability checks can bypass it.
