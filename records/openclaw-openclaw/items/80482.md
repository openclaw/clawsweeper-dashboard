## Summary

Keep open. The branch targets the right auth bug, but the latest diff still lets MiniMax-style env/SecretRef inline credentials bypass cooldown marking and availability checks, so it does not fully fix the linked 402 retry loop.

## What This Changes

The PR adds provider-scoped cooldown state for inline `models.providers.<id>.apiKey` credentials and wires embedded-runner billing/auth failures without auth profiles to mark the inline provider key.

## Best Possible Solution

Keep the provider-level cooldown design, but preserve configured inline provenance before generic env fallback, apply cooldowns in resolution and availability checks, fire the existing auth-failure hook, and cover literal plus env SecretRef MiniMax-style configs.

## Reproduction Assessment

Yes, source-level. Current main and the PR diff show MiniMax/provider env vars can resolve through generic env auth before inline provenance and cooldown checks; I did not run tests because this review is read-only.

## Solution Assessment

No. The provider-level cooldown tracker is the right direction, but the implementation needs to cover env SecretRef/provider-env provenance and warmed auth invalidation before it is the maintainable fix.
