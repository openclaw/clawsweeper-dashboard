## Summary

Keep open: the PR targets a real linked billing-cooldown bug, but the actual embedded-runner no-profile failure paths still do not reliably invoke the new inline cooldown marker, and the supplied proof exercises the helper directly rather than the failing runner path.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, blocks disabled inline keys in auth resolution/availability checks, and records inline failures from the embedded runner.

## Best Possible Solution

Keep the provider-level non-secret cooldown identity, but wire prompt, assistant, fallback, and incomplete-turn auth/billing failures through it even when the active inline credential has no profile id.

## Reproduction Assessment

Yes, source-level. Configure a provider only with `models.providers.<id>.apiKey`: current main resolves it without `profileId`, and the relevant failure-marking call sites still depend on a profile id before cooldown state can be recorded.

## Solution Assessment

No. The provider-level cooldown approach is maintainable, but this patch needs to connect the no-profile inline branch to the actual embedded-runner failure paths and add runner-path coverage/proof.
