## Summary

Keep open. The PR targets the right auth/fallback path for the linked inline API-key billing cooldown bug and I found no definite blocking code finding, but this external PR still lacks after-fix real behavior proof from a real setup.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, skips disabled inline keys during auth resolution/availability checks, records inline key failures from the embedded runner, and adds focused tests.

## Best Possible Solution

Land a focused fix that persists only non-secret provider-level cooldown state for inline API keys and proves a real billing/auth failure is skipped or falls through without retrying the exhausted key.

## Reproduction Assessment

Yes, source-level. Current main resolves inline configured API keys without a `profileId`, makes explicit-config runs use an empty profile order, and returns before recording failures when `profileId` is missing; I did not perform a live MiniMax/402 run.

## Solution Assessment

Yes, directionally. A provider-level non-secret cooldown identity is safer than creating synthetic auth profiles with credential implications, and this PR applies that approach, but it still needs real runtime proof before merge.
