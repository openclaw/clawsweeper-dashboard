## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, checks that state during auth resolution/availability, records inline billing/auth failures from the embedded runner, and adds focused tests.

## Best Possible Solution

Land the provider-level non-secret cooldown identity approach after codeowner/maintainer review, keeping inline secrets out of auth profiles and preserving normal auth/fallback behavior.

## Reproduction Assessment

Yes, source-level. Current main resolves inline configured API keys without a `profileId`, the embedded runner uses an empty profile order for explicit inline API-key auth, and the cooldown writer returns when no profile exists.

## Solution Assessment

Yes. A provider-level non-secret usageStats identity is narrower and safer than creating synthetic auth profiles, and this PR wires it into failure recording plus auth resolution/availability checks.
