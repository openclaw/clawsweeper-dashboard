## Summary

Keep open. The PR targets a real current-main bug and has useful proof, but the current patch still lets documented env SecretRef inline API keys bypass the new cooldown path.

## What This Changes

The PR adds provider-level cooldown state for inline `models.providers.<id>.apiKey` credentials, blocks disabled inline keys during auth checks, and records no-profile embedded-runner auth/billing failures with regression coverage.

## Best Possible Solution

Keep the provider-level cooldown design, but preserve configured inline API-key provenance for literal keys, known env markers, and env SecretRefs before ambient env auth can win.

## Reproduction Assessment

Yes, source-level. Current main resolves inline `models.providers.<id>.apiKey` credentials without a profile id, and the runner only records auth/billing cooldowns when a failing profile id exists.

## Solution Assessment

No. The provider-level cooldown identity is the right shape, but the PR needs to handle configured env SecretRefs before ambient env fallback so documented inline keys cannot bypass marking and blocking.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Apply inline cooldowns before env fallback:** `src/agents/model-auth.ts:779-781`

Move configured inline SecretRef handling ahead of ambient env fallback, or otherwise tag configured env credentials with inline provenance.
