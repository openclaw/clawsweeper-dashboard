## Summary

Current main still has a source-visible false-success path: blocked/error metadata is carried on terminal events and wait snapshots, but the subagent completion bridge can still normalize that terminal state to an ok outcome and announce success.

## What This Changes

The issue asks OpenClaw to report blocked/context-overflow subagent terminals as failed or blocked instead of successful no-output completions.

## Best Possible Solution

Normalize blocked/error terminal metadata before parent completion synthesis, surface the error text to the requester, and add regression coverage for lifecycle end and agent.wait snapshots that contain livenessState "blocked" or context_overflow metadata.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only sweep. Source inspection gives a high-confidence current-main path: blocked metadata can be emitted and carried, then later mapped to ok and announced as completed successfully.

## Solution Assessment

Yes, a narrow bridge/status-normalization fix is the best direction. It should not change provider compaction behavior; it should classify terminal blocked/error metadata before the parent announce is generated.
