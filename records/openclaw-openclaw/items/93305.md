The PR changes `scanEmptyAllowlistPolicyWarnings` to skip the top-level empty group allowlist warning when every configured account has `groupAllowFrom` or `allowFrom`.

## Best Possible Solution

Land one canonical scanner fix that suppresses both shared and plugin parent-scope group warnings only when every active real account scope has an effective group allowlist, with focused regression coverage and real terminal proof.

## Reproduction Assessment

Yes. Current main checks the top-level channel record before iterating account records, and the linked issue provides the config shape that makes that parent-scope warning false; I did not run a live doctor command because this review is read-only.

Full review comments:

- **[P2] Gate plugin warnings with the top-level skip:** `src/commands/doctor/shared/empty-allowlist-scan.ts:66`
  - body: The new skip override only reaches `collectEmptyAllowlistPolicyWarningsForAccount`; `extraWarningsForAccount` still runs for the top-level record. Telegram suppresses the generic warning and emits its group warning from that extra hook, so the reported parent warning can still appear.
  - confidence: 0.93
- **[P2] Require active accounts before skipping:** `src/commands/doctor/shared/empty-allowlist-scan.ts:101-103`
  - body: The predicate treats disabled account entries as covered and only checks that the accounts map has keys. With only disabled accounts, the top-level warning is suppressed even though no enabled account supplies allowlist coverage.
  - confidence: 0.9

  - body: Some channels keep an active implicit or default account at the channel root while named accounts also exist. Skipping the top-level warning solely from named account coverage can hide the root/default account's empty group allowlist diagnostic.
  - confidence: 0.87
