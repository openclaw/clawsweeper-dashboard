## Summary

Keep open: the reported bug is valid and the PR's new external-browser-proxy direction is appropriate, but the current branch still leaves existing-session post-action and tab-list result checks on direct-mode DNS pinning, so the fix is partial.

## What This Changes

The PR adds an external-browser-proxy browser navigation mode for non-OpenClaw driver profiles, updates the navigation guard to skip Node DNS pinning in that mode while preserving hostname policy gates, and adds focused browser SSRF tests.

## Best Possible Solution

Thread the resolved browser proxy mode through initial navigation, post-action URL checks, new-tab checks, and tab listing, with route-level regression tests for split-DNS existing-session behavior.

## Reproduction Assessment

Yes. Source inspection of current main shows existing-session profiles are classified as direct and DNS-pinned, and the PR body gives both a live split-DNS setup and a deterministic /etc/hosts reproduction path; I did not run a live browser in this read-only review.

## Solution Assessment

No, not yet. The new external-browser-proxy mode is the narrow maintainable fix, but it is incomplete until every existing-session navigation/result policy check receives the same profile-derived mode.
