## Summary

Keep open: the PR addresses a source-reproducible existing-session false positive and includes real terminal proof, but the new external-browser-proxy branch bypasses configured strict SSRF hostname policy, so it should not merge as-is.

## What This Changes

The PR marks non-openclaw browser profiles as `external-browser-proxy`, skips Node DNS pinning for that mode, exports the SSRF hostname helper, and adds split-DNS navigation guard tests.

## Best Possible Solution

Keep the external-browser DNS-skip design, but preserve configured SSRF policy gates and add regressions for default split-DNS, explicit strict mode, hostname allowlists, and private-network opt-in.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main classifies existing-session as `direct`, then `assertBrowserNavigationAllowed` DNS-pins the hostname and rejects a lookup result like `192.168.42.1`. The PR body also includes copied terminal/live output for the same trigger condition.

## Solution Assessment

No. The proposed direction is plausible, but the implementation returns before enforcing explicit strict-mode and hostname-allowlist policy; the safer fix is to skip only Node DNS resolution for external browsers while retaining configured hostname/IP-literal gates.
