## Summary

Keep open. The PR addresses a real existing-session split-DNS false positive, but the branch is not ready because it changes the default browser SSRF trust model for all non-OpenClaw profiles and still leaves downstream existing-session result checks on direct DNS pinning.

## What This Changes

The PR adds an `external-browser-proxy` browser navigation mode for non-OpenClaw profiles, skips Node DNS pinning in that mode, exports the SSRF hostname helper, and adds focused proxy-mode/navigation-guard tests.

## Best Possible Solution

Decide and document the existing-session SSRF trust model, then either gate external browser DNS skipping behind an explicit trusted signal or consistently thread the accepted mode through initial navigation, post-action checks, tab listing, and tests.

## Reproduction Assessment

Yes. Current main clearly routes existing-session through direct DNS pinning, and the PR body supplies matching real macOS split-DNS before/after evidence.

## Solution Assessment

No. The approach fixes the initial false positive, but it broadly removes default DNS private-address pinning for all non-OpenClaw profiles and leaves post-action/tab-list result checks inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Gate external mode on an explicit trusted browser signal:** `extensions/browser/src/browser/browser-proxy-mode.ts:60-61`
  - body: Returning `external-browser-proxy` for every non-OpenClaw profile makes the default existing-session path skip `resolvePinnedHostnameWithPolicy` even when there is no PAC/VPN/proxy signal. That removes the current DNS-answer check that blocks public-looking hostnames resolving to private IPs, which conflicts with the documented fail-closed browser SSRF default. Please gate this mode on an explicit trusted external-browser/proxy signal, or get the default trust model and docs changed before removing DNS pinning.
  - confidence: 0.78
- **[P2] Propagate external mode to result checks:** `extensions/browser/src/browser/browser-proxy-mode.ts:60-61`
  - body: If maintainers accept the external-browser mode, it also needs to be carried into existing-session result checks. `assertExistingSessionPostInteractionNavigationAllowed` and `/tabs` redaction still build policy without `browserProxyMode`, so click/type/evaluate rechecks and tab listing can still DNS-pin and fail or redact the same split-DNS public URL after initial navigation succeeds.
  - confidence: 0.88
