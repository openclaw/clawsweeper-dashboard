## Summary

Keep open. The latest push addresses the earlier strict-policy bypass and the real behavior proof is sufficient, but the new external-browser mode is still not applied to all existing-session URL rechecks, so the split-DNS false positive can remain after a successful navigation; the user-facing fix also needs a changelog entry.

## What This Changes

The PR marks non-openclaw browser profiles as `external-browser-proxy`, skips Node-side DNS pinning for that mode while preserving hostname and strict-policy gates, exports the SSRF hostname helper, and adds focused browser tests.

## Best Possible Solution

Land a version that consistently applies the external-browser proxy mode to existing-session navigation, post-action, tab-list, and current-URL checks while preserving explicit strict SSRF policy gates and documenting the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path on current main: `existing-session` is classified as `direct`, then browser navigation DNS-pins the hostname and rejects a private lookup result such as `192.168.42.1`; the PR body adds matching terminal/live evidence.

## Solution Assessment

No, not yet. The main navigation-guard direction is maintainable after `fc479cee`, but the new mode is not propagated to all existing-session result checks, so the fix should be extended before merge.
