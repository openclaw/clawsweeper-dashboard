## Summary

Keep open: the branch fixes the initial existing-session navigation path, but it does not carry the new external-browser-proxy mode into all downstream result checks, so the reported split-DNS false positive can still occur after interactions and in tab listing.

## What This Changes

The PR adds an external-browser-proxy browser navigation mode for non-OpenClaw driver profiles and updates SSRF navigation-guard tests for split-DNS existing-session browser use.

## Best Possible Solution

Thread the resolved browser proxy mode through every existing-session navigation/result policy check so the initial navigation, post-action URL checks, new-tab checks, and tab listing all share the same SSRF trust boundary.

## Reproduction Assessment

Yes. Source inspection shows current main classifies existing-session as direct and DNS-pins hostnames, and the PR body gives a deterministic split-DNS or /etc/hosts reproduction path for the false positive.

## Solution Assessment

No, not yet. The new external-browser-proxy mode is the right narrow direction, but the patch remains partial until the mode is propagated to the existing-session post-action and tab-list result checks.
