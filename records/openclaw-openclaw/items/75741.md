## Summary

Close as clawhub. This is an optional Codex runtime integration that existing plugin, ACP runtime backend, Gateway method, CLI, and command seams can host outside core, while OpenClaw's vision and plugin docs explicitly direct optional plugin work to ClawHub/npm distribution. The proposed bundled branch also has blocking correctness and security findings, so it should not be merged into core as-is.

## What This Changes

The PR adds a bundled `codex-sdk` native plugin with Codex SDK ACP runtime support, Gateway/CLI/chat/UI surfaces, session/proposal state, MCP backchannel plumbing, docs, smoke coverage, package metadata, and lockfile entries.

## Best Possible Solution

Keep OpenClaw core lean and move this Codex SDK runtime to a standalone ClawHub/npm code plugin that uses `openclaw/plugin-sdk/acp-runtime-backend` plus existing Gateway, CLI, command, and plugin docs surfaces; open a separate narrow core PR only if the external plugin proves a missing typed SDK seam.

## Reproduction Assessment

Yes for the review blockers: static inspection shows the imported `openclaw/plugin-sdk/acpx` subpath is not in the current package exports/entrypoint list, and the Codex SDK contract confirms that omitting `env` inherits `process.env`. No live runtime reproduction was needed for the product-boundary decision.

## Solution Assessment

No. Bundling an XL optional Codex runtime into core is not the narrowest maintainable path under current OpenClaw policy; the safer path is an external plugin with tight env redaction and only narrowly justified core SDK additions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Strip inherited secrets before spawning Codex:** `extensions/codex-sdk/src/runtime-client.ts:178-185`
  - body: The new runtime defaults `inheritEnv` to true, and this branch either returns `undefined` or copies every key from `process.env` into the Codex SDK options. The SDK inherits the parent environment when `env` is omitted, so enabling this plugin exposes Gateway/provider secrets such as API keys or GitHub tokens to the Codex subprocess, unlike the existing ACP client path which strips provider auth env vars. Build a minimal allowlist or reuse the provider-env stripping helpers before launching Codex.
  - confidence: 0.88
- **[P2] Use the exported ACP backend SDK subpath:** `extensions/codex-sdk/index.ts:1`
  - body: This plugin imports `openclaw/plugin-sdk/acpx`, but that subpath is not published in the current package exports or SDK entrypoint list. The existing public backend registration surface is `openclaw/plugin-sdk/acp-runtime-backend`, so this import will fail typecheck/package resolution before the plugin can register.
  - confidence: 0.93
- **[P3] Fix the changelog entry format and credit:** `CHANGELOG.md:9-12`
  - body: The added user-facing changelog entry is wrapped over multiple lines and has no contributor credit. OpenClaw changelog entries are single-line bullets and external user-facing changes need a `Thanks @...` attribution, so this will fail the changelog policy checks.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete secret-exposure risk by passing the Gateway process environment through to the Codex SDK subprocess by default.

Concerns:

- **[high] Codex subprocess inherits Gateway secrets:** `extensions/codex-sdk/src/runtime-client.ts:178`
  - body: `inheritEnv` defaults to true and `resolveEnv` either omits the SDK `env` option or copies all of `process.env`, which means the Codex CLI can receive provider/API/GitHub tokens from the Gateway process. Existing ACP child-process code strips provider auth env vars, so this path needs equivalent redaction before it is safe to enable.
  - confidence: 0.88
