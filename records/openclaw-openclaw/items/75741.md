## Summary

Close as `clawhub`. The existing ClawSweeper comment's scope-fit analysis is supported by current main: this is an optional Codex runtime integration that can live as a standalone plugin, and the current SDK/API surfaces already cover ACP backend registration, plugin services, CLI commands, chat commands, and Gateway methods. The proposed bundled branch also has concrete correctness and security blockers, so it should not be merged into core as-is.

## What This Changes

The PR adds a bundled `codex-sdk` plugin backed by `@openai/codex-sdk`, with ACP backend registration, Gateway/CLI/chat/UI surfaces, session replay/export, proposal inbox, MCP backchannel, docs, smoke script, and tests.

## Best Possible Solution

Keep OpenClaw core lean and move this Codex SDK runtime to a standalone ClawHub/npm plugin, using the current plugin API plus `openclaw/plugin-sdk/acp-runtime-backend`; open a focused core SDK PR only if the external plugin proves a missing typed seam.

## Reproduction Assessment

Yes for the review blockers. Static inspection reproduces the unpublished SDK subpath problem, and the upstream Codex SDK source/types confirm the environment inheritance behavior without needing a live Codex run.

## Solution Assessment

No. Bundling an XL optional Codex runtime into core is not the narrowest maintainable solution under the current plugin policy; a standalone plugin with explicit env redaction and only narrowly justified SDK additions is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Switch to a public SDK subpath:** `extensions/codex-sdk/index.ts:1`
  - body: This plugin imports `openclaw/plugin-sdk/acpx`, but current package exports and the curated SDK entrypoint list do not expose that subpath, and the contract test explicitly denies it. Installing or linking this plugin through package exports can fail; use `openclaw/plugin-sdk/acp-runtime-backend` plus exported core/plugin-entry types, or add a documented public seam.
  - confidence: 0.9
- **[P1] Redact env before spawning Codex:** `extensions/codex-sdk/src/runtime-client.ts:171-181`
  - body: `inheritEnv` defaults to true and `resolveEnv` either omits the SDK `env` option or copies all of `process.env`. The Codex SDK inherits `process.env` when `env` is not provided, so the Codex subprocess can receive Gateway provider/API/GitHub tokens; build an allowlist or redaction path before passing env.
  - confidence: 0.9
