## Summary

Keep open. Current main still lacks the proposed Discord post-reset startup marker, but this PR branch is not merge-ready because its embedded-steer path regresses current active-session routing/queue options and its clear logic is not tied to actual owner-steer consumption.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session field, sets it for Discord channel/thread resets, and wraps queued post-reset owner prompts in `runReplyAgent`.

## Best Possible Solution

Consolidate this PR with #49001 into one durable Discord reset-window implementation that applies only to the first qualifying owner turn, clears on every successful consumption path, preserves current active-steering behavior, and adds channel plus thread regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction for the PR defects: a non-owner Discord run can clear the marker before the owner turn, and a successful embedded steer enqueue returns before any clear path runs.

## Solution Assessment

No. The direction is useful, but this PR is not the best final fix until it is rebased onto current main, preserves active steering contracts, and consolidates the overlapping thread-only work in #49001.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear the window only after consuming an owner steer:** `src/auto-reply/reply/agent-runner.ts:850-857`
  - body: `startedRun` is set for any run, but the startup steer only applies to owner Discord turns. In a mixed Discord channel, a non-owner message can start a run first and clear `postRotationStartupUntilMs`, so the owner's first post-reset message never receives the recovery prompt. Gate the clear on actual steer consumption instead of any started run.
  - confidence: 0.88
- **[P2] Clear the window on successful embedded steer enqueue:** `src/auto-reply/reply/agent-runner.ts:273-277`
  - body: When `queueEmbeddedPiMessage` succeeds, this early return exits before the later `finally` block is reached. The marker remains active, so subsequent owner messages inside the startup window can be wrapped again as the first post-reset message. Clear the window before returning from this successful enqueue path.
  - confidence: 0.86
- **[P2] Preserve active session routing for embedded steers:** `src/auto-reply/reply/agent-runner.ts:273`
  - body: Current main resolves the live session id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This PR queues with `followupRun.run.sessionId`, which can target a stale pre-reset id after session rotation. Keep the active-id resolution while swapping in the wrapped prompt.
  - confidence: 0.78
- **[P2] Forward queue steering options when wrapping prompts:** `src/auto-reply/reply/agent-runner.ts:273`
  - body: Current main forwards `steeringMode` and optional `debounceMs` into `queueEmbeddedPiMessage`; this replacement call drops those options. That changes active steering drain mode and can lose configured debounce behavior. Preserve the existing third-argument options when using the wrapped prompt.
  - confidence: 0.8
