## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the proposed Discord post-reset startup marker, so the underlying channel/thread reset problem remains relevant, but the PR branch has blocking one-shot and active-run steering regressions in `agent-runner.ts`.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session field and uses it to wrap Discord channel/thread post-reset queued owner messages with a startup-recovery prompt.

## Best Possible Solution

Land one consolidated Discord reset-window implementation that marks explicit channel and thread resets, applies the startup-recovery steer only to the first qualifying owner turn, clears the marker on every successful consumption path, preserves current active session-id and queue-option behavior, and adds focused channel plus thread regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: a non-owner Discord run after reset can clear the marker before the owner's first qualifying turn, and a successful embedded steer enqueue returns before the marker is cleared.

## Solution Assessment

No. The proposed direction is useful, but this branch is not the best final fix until it rebases onto current main, preserves active-run routing and queue options, and makes the marker truly one-shot on steer consumption.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear the window only after an owner steer is consumed:** `src/auto-reply/reply/agent-runner.ts:854`
  - body: `startedRun` is set for any agent turn, but the startup steer only applies to owner Discord messages. If a non-owner channel message starts a run before the owner's first post-reset turn, this `finally` block clears `postRotationStartupUntilMs` and the owner turn never receives the recovery prompt. Gate the clear on actual steer consumption instead of any started run.
  - confidence: 0.88
- **[P2] Clear the window before returning from embedded steer enqueue:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: When `queueEmbeddedPiMessage` succeeds, this early return exits before the later clear path runs. The marker remains active, so subsequent owner messages inside the 30 second window can be wrapped again as the first post-reset message. Clear the marker on this successful enqueue path too.
  - confidence: 0.86
- **[P2] Preserve the active session id when queueing the steer prompt:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main resolves the active session id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This PR queues with `followupRun.run.sessionId`, which can target a stale pre-reset id after session rotation; keep the current active-id resolution while swapping in the wrapped prompt.
  - confidence: 0.78
- **[P2] Preserve steering queue options when wrapping the prompt:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main forwards `steeringMode` and optional `debounceMs` into `queueEmbeddedPiMessage`; this PR's replacement call drops those options. That changes `queue` mode back to the default `all` drain mode and loses configured debounce behavior for active embedded steering.
  - confidence: 0.8
