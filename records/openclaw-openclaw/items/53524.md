## Summary

Keep open: current main still lacks the proposed post-reset startup marker, and the PR patch has blocking one-shot and active-steering regressions that would keep the Discord channel/thread reset race partly broken.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session window and uses it to steer Discord post-reset queued owner messages in reply/session runtime code.

## Best Possible Solution

Land one consolidated Discord reset-window implementation that marks explicit channel and thread resets, applies only to the first qualifying owner turn, clears after every successful consumption path, preserves current active-session routing and queue options, and adds focused channel plus thread regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: a non-owner Discord run can clear the marker before the owner's first qualifying turn, and a successful embedded steer enqueue returns before the marker is cleared.

## Solution Assessment

No. The direction is useful, but this PR is not the best final fix until it is rebased onto current main, consolidated with #49001, and made one-shot without regressing active steering behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear the window only after an owner steer is consumed:** `src/auto-reply/reply/agent-runner.ts:854`
  - body: `startedRun` is set for any run, but the startup steer only applies to owner Discord messages. If a non-owner channel message starts a run before the owner's first post-reset turn, this `finally` block clears `postRotationStartupUntilMs` and the owner turn never receives the recovery prompt. Gate the clear on actual steer consumption instead of any started run.
  - confidence: 0.88
- **[P2] Clear the window before returning from embedded steer enqueue:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: When `queueEmbeddedPiMessage` succeeds, this early return exits before the later clear path runs. The marker remains active, so subsequent owner messages inside the startup window can be wrapped again as the first post-reset message. Clear the marker on this successful enqueue path too.
  - confidence: 0.86
- **[P2] Preserve the active session id when queueing the steer prompt:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main resolves the active session id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This PR queues with `followupRun.run.sessionId`, which can target a stale pre-reset id after session rotation; keep the current active-id resolution while swapping in the wrapped prompt.
  - confidence: 0.78
- **[P2] Preserve steering queue options when wrapping the prompt:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main forwards `steeringMode` and optional `debounceMs` into `queueEmbeddedPiMessage`; this PR's replacement call drops those options. That changes active steering drain mode and can lose configured debounce behavior.
  - confidence: 0.8
