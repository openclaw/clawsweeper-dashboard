## Label Justifications

- `P2`: This is a normal priority context-management/model-routing bug with a direct workaround: use the canonical provider/model id in `compaction.model`.
- `impact:session-state`: Failed auto-compaction prevents the intended long-session context-management behavior from completing.
- `impact:auth-provider`: The failure is in model choice/provider routing for a configured compaction override alias.

## Summary

Keep open: current main still treats a bare `agents.defaults.compaction.model` override as a raw model id on the active provider, so a configured alias such as `gpt54mini` can still flow toward compaction as `openai/gpt54mini` instead of the canonical configured model.

## What This Changes

The issue requests resolving `agents.defaults.compaction.model` aliases before embedded auto-compaction dispatch.

## Best Possible Solution

Resolve compaction model overrides through the shared model alias resolver before provider/model dispatch, preserving exact canonical refs and existing auth-profile drop behavior when the resolved provider changes.

## Reproduction Assessment

Yes at source level: a focused test with `compaction.model: "gpt54mini"` and an alias entry for `openai/gpt-5.4-mini` would currently return provider `openai` and model `gpt54mini`. I did not run a live 2026.6.1 session in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing alias/model-ref resolver in the compaction target helper, then add regression coverage for alias, canonical provider/model, and provider-change auth-profile behavior.
