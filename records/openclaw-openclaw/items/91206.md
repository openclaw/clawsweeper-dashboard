## Summary

Keep open: the PR targets the right bug, but it is not merge-ready because `allowModelOverride: true` is added to request params while the gateway authorizes model overrides from client/internal scope, so forwarded `model` can be rejected instead of applied. The supplied proof also shows params/log shape, not an accepted child run after the gateway authorization gate.

## What This Changes

The PR adds `model: resolvedModel || undefined` and `allowModelOverride: true` to the `spawnSubagentDirect` gateway `agent` call.

## Best Possible Solution

Forward the resolved model through a gateway authorization path that is actually honored, preserving existing least-privilege behavior for no-model subagent calls and adding focused coverage for websocket-style plus in-process dispatch.

## Reproduction Assessment

Yes at source level: current main resolves and stores a subagent model but does not pass it into the child gateway `agent` run. I did not run a live gateway reproduction because this review is read-only.

## Solution Assessment

No. Passing `model` is necessary, but `allowModelOverride` in request params is not the gateway authorization contract; the fix should authorize the internal subagent call at the client/options seam.
