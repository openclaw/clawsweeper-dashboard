## Summary

Keep this PR open: the runtime-provider direction addresses a real source-reproducible auth-routing bug, but the patch introduces a blocking test expectation that contradicts OpenRouter model-id normalization.

## What This Changes

The PR adds a simple-completion runtime provider override so logical `openai/gpt-*` plugin completions can prepare and execute through `openai-codex` when Codex runtime policy applies, plus tests and a changelog entry.

## Best Possible Solution

Keep the runtime-provider seam for Codex-backed OpenAI completions, restore the OpenRouter normalized model-id expectation, and then re-run the focused simple-completion/runtime-LLM tests plus changed checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main prepares plugin LLM completions with the logical provider, and the PR-introduced test regression follows directly from OpenRouter's manifest `prefixWhenBare` normalization.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes, but not as submitted; routing only auth/transport through `runtimeProvider` is narrow, while changing the OpenRouter expectation should be reverted because it contradicts existing provider model-id normalization.
