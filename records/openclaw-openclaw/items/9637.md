## Summary

Keep open: current main still lacks a TUI plain-text/accessibility config option, strict config validation would reject the proposed key, and the main TUI/tool rendering paths still emit emoji or Unicode glyphs unconditionally. The related implementation PR was closed unmerged, and the issue discussion includes both the original screenreader report and a second affected user, so this remains valid core TUI/config work.

## What This Changes

Add a UI accessibility/plain-text config option that disables emoji and Unicode symbols in the terminal UI for screenreader-friendly output.

## Best Possible Solution

Implement this as core TUI accessibility support: add a documented `ui.accessibility.plainText`-style config field, validate it in the strict config schema, load it once at TUI startup, and pass it through TUI/tool rendering seams so tool cards, summaries, selectors, settings, waiting/status text, and docs all have deterministic ASCII output when enabled while preserving existing output by default.

## Reproduction Assessment

Yes. A high-confidence reproduction path is source-level: on current main, `ui.accessibility.plainText` is absent from `OpenClawConfig` and strict zod validation, while `resolveToolDisplay()`, `ToolExecutionComponent`, and selector components still render emoji/Unicode glyphs unconditionally.

## Solution Assessment

Yes. A core TUI config flag is the maintainable direction because the problematic glyphs are emitted by OpenClaw's own config-validated TUI and tool-display render paths; this cannot be solved cleanly as an external plugin without a missing core rendering seam.
