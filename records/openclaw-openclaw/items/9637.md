## Summary

Keep open: current main still does not provide a TUI plain-text/accessibility mode, the proposed config key is still absent from the exported config type and strict zod schema, and TUI/tool rendering still emits emoji or Unicode glyphs unconditionally. The linked implementation PR #9685 was closed unmerged, and later comments include another screenreader user affected by the same behavior, so this remains actionable core TUI/config work.

## What This Changes

Request to add a documented TUI accessibility/plain-text config option that replaces emoji and Unicode symbols with screenreader-friendly ASCII output.

## Best Possible Solution

Implement this as core TUI accessibility support: add a documented ui.accessibility.plainText-style config field, validate it, load it once at TUI startup, and pass it through TUI/tool rendering so ASCII output is deterministic when enabled while preserving current rich output by default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the config key is absent from the exported type and strict zod schema, while tool summaries and TUI selector components still render emoji/Unicode glyphs without a plain-text branch.

## Solution Assessment

Yes. A core TUI config flag is the narrow maintainable direction because the problematic output is emitted by OpenClaw's own config-validated TUI/tool render paths and cannot be solved cleanly by an external plugin alone.
