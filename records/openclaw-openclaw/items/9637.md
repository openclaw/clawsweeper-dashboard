## Summary

Keep open: current main still lacks a documented TUI plain-text/accessibility mode, strict config validation would reject the proposed key, and the related implementation PR at https://github.com/openclaw/openclaw/pull/9685 is closed unmerged while another screen-reader user confirmed the problem.

## What This Changes

The issue requests a TUI accessibility option that disables decorative emoji and Unicode symbols in favor of ASCII/plain-text output.

## Best Possible Solution

Ship a default-off core TUI plain-text mode documented in config/help/docs and applied consistently across tool summaries, selectors, status/waiting glyphs, settings rendering, and focused tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The exact key name, ASCII replacements, and pi-tui coverage still need maintainer direction.
