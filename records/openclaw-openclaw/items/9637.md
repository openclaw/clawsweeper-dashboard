## Summary

Keep open: current main and the latest release still lack a documented TUI plain-text/accessibility mode, while strict config validation and TUI render paths continue to allow only emoji and Unicode glyph behavior. The related implementation attempt at https://github.com/openclaw/openclaw/pull/9685 is closed unmerged, and the remaining choices are product/config scope rather than a safe auto-close.

## What This Changes

Add a default-off TUI accessibility config option to disable emoji and Unicode symbols in terminal UI output.

## Best Possible Solution

Ship a documented default-off core TUI plain-text mode applied consistently across config validation/help/docs, tool summaries, selector/theme rendering, status/waiting glyphs, and focused TUI/config tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. Maintainers still need to choose the exact key name, ASCII replacements, and selector/pi-tui coverage.
