Keep open: current main still lacks a TUI plain-text/accessibility mode, config validation would reject the proposed key, and TUI/tool rendering still emits emoji and Unicode glyphs without an ASCII branch. The related implementation PR #9685 was closed unmerged, so the requested accessibility work remains outstanding.

## What This Changes

Add a documented TUI accessibility config option that disables emoji and Unicode symbols in terminal output.

## Best Possible Solution

Ship core TUI accessibility support with default rich output unchanged and deterministic ASCII output when the new option is enabled.

## Reproduction Assessment

Yes. Source inspection on current main shows the accessibility/plainText config key is absent from exported types and strict zod validation while tool summaries and TUI selectors still render emoji or Unicode glyphs without an alternate branch.

## Solution Assessment

Yes. A core TUI config flag is the narrow maintainable direction because OpenClaw itself emits these glyphs in config-validated TUI/tool render paths, which an external plugin cannot reliably override.
