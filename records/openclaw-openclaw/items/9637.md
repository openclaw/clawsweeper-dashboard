## Summary

Keep open: current main still has no accepted or documented TUI plain-text/accessibility mode, and the relevant config, tool-display, selector, and dependency-backed TUI paths still emit emoji or Unicode glyphs unconditionally. The linked implementation attempt at https://github.com/openclaw/openclaw/pull/9685 is closed unmerged, while maintainer review already marked this as needing product/UX follow-up.

## What This Changes

Add a documented TUI accessibility setting that suppresses emoji and Unicode glyphs in terminal rendering.

## Best Possible Solution

Ship a documented default-off core TUI plain-text mode threaded from config through TUI startup and applied consistently to tool summaries, selectors, settings cursors, status/waiting glyphs, generated config surfaces, public docs, and focused render/config tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs unconditionally. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The option name, replacement strings, and pi-tui wrapper scope should be decided before coding.
