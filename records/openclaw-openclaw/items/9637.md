## Summary

Keep open: current main still lacks a TUI plain-text/accessibility mode, strict config validation would reject the proposed key, and the related implementation PR was closed unmerged.

## What This Changes

Request to add a documented TUI accessibility/plain-text config mode that replaces emoji and Unicode glyphs with screen-reader-friendly ASCII output.

## Best Possible Solution

Ship core TUI plain-text support with rich output unchanged by default and deterministic ASCII alternatives when the documented config flag is enabled.

## Reproduction Assessment

Yes for the current behavior, though this is a feature request rather than a broken contract. Source inspection on current main shows no accepted config key and multiple OpenClaw-owned render paths still emit emoji or Unicode glyphs unconditionally.

## Solution Assessment

Yes. A core TUI config flag is the narrow maintainable path because OpenClaw-owned config validation and TUI render code emit these glyphs; an external plugin cannot reliably override that output.

## Repair Work Prompt

Implement the TUI plain-text accessibility mode requested by https://github.com/openclaw/openclaw/issues/9637. Treat https://github.com/openclaw/openclaw/pull/9685 as prior art, not as merged code. Add a documented config key, preferably ui.accessibility.plainText with default false, to OpenClawConfig, strict zod validation, generated help/labels/schema surfaces, and public TUI/config docs. Load the option once at TUI startup or an existing config boundary, thread it through TUI components without render-time config reads, and provide ASCII alternatives for OpenClaw-owned emoji/Unicode glyphs in tool summaries/cards, selectors, settings cursors, separators, waiting/status text, and related TUI output. Keep rich Unicode output as the default. Add focused regression tests for config acceptance/rejection and TUI/tool rendering in both default and plain-text modes. Do not change unrelated channel reaction emoji behavior, plugin metadata emoji, terminal table border policy outside this TUI mode, dependency versions, or broad UI styling.
