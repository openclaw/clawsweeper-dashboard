## Summary

Keep open: current main still lacks an accepted or documented TUI plain-text/accessibility mode, and the relevant render paths continue to emit emoji or Unicode glyphs unconditionally. The linked implementation attempt is closed unmerged, and another screen-reader user confirmed the same problem in the discussion.

## What This Changes

Requests a default-off TUI accessibility/plain-text config option that suppresses emoji and Unicode symbols for screen-reader-friendly terminal output.

## Best Possible Solution

Ship a documented core TUI plain-text mode read once at startup and applied consistently across config validation/docs plus tool, selector, settings, waiting/status, and TUI component glyph renderers.

## Reproduction Assessment

Yes, for source-level verification: current main lacks an accepted/documented plain-text config key, and inspected TUI/tool render paths still emit emoji or Unicode glyphs unconditionally. I did not run a live TUI or screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core config flag is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The option name, ASCII replacements, and dependency wrapper strategy should be decided before coding.
