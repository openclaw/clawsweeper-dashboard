## Summary

Current main still emits decorative emoji and Unicode throughout the TUI, no supported plain-text mode or canonical replacement exists, and the only implementation PR closed unmerged. The accessibility need is valid, but maintainers should first choose a comprehensive presentation policy rather than accepting the originally proposed narrow config patch.

## What This Changes

Add a TUI accessibility mode that replaces emoji and decorative Unicode symbols with screen-reader-friendly ASCII or plain text.

## Best Possible Solution

Define one core TUI presentation mode with a complete glyph inventory and ASCII alternatives, extend or replace the upstream selector seam for a configurable prefix, and add a documented config option only after maintainers choose explicit opt-in versus automatic or accessible-default behavior.

## Reproduction Assessment

Yes, at source level with high confidence: current main unconditionally emits tool emoji and box-drawing characters, upstream hardcodes the selection arrow, and two users report the resulting screen-reader difficulty. This read-only review did not run a live screen-reader session.

## Solution Assessment

No, not exactly as originally scoped: the accessibility objective is correct, but a 20–30 line patch across two or three files would miss sibling glyph paths and the upstream hardcoded arrow. A shared TUI presentation contract is the narrower maintainable solution at the product boundary.
