## Summary

Keep open: current main and the latest release still lack a documented TUI plain-text/accessibility mode, strict config validation would reject the proposed key, TUI/tool render paths still emit emoji and Unicode glyphs, and the related implementation PR is closed unmerged.

## What This Changes

The issue requests a TUI accessibility config option that renders plain ASCII instead of emoji and Unicode symbols for screen-reader users.

## Best Possible Solution

Ship a default-off core TUI plain-text mode documented in config/help/docs and applied consistently across tool summaries, selectors, status/waiting glyphs, settings rendering, and focused tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The exact key name, ASCII replacements, and selector/pi-tui coverage still need maintainer direction.
