## Summary

Current main and the latest release still lack a TUI plain-text accessibility mode, the only linked implementation PR closed unmerged, and two screen-reader users describe the impact. This remains useful core TUI work, but it needs maintainer product judgment because current code has a dedicated `tui` config namespace, the glyph surface is broader than the original estimate, and pi-tui 0.78.0 hard-codes its selection arrow without a theme override.

## What This Changes

Add a terminal UI accessibility mode that replaces decorative emoji, arrows, box drawing, and other non-ASCII symbols with screen-reader-friendly text or ASCII alternatives.

## Best Possible Solution

Define one terminal-specific text-style policy under the current TUI ownership boundary, prepare it once at startup, provide meaningful ASCII/text alternatives across all OpenClaw glyph paths, add or upstream a pi-tui selection-prefix seam, preserve the current default, and document and test both modes.

## Reproduction Assessment

Yes at source level with high confidence: current main has no opt-out, OpenClaw and pi-tui render the cited glyphs unconditionally, and two users report screen-reader difficulty. This read-only review did not run a live screen-reader session.

## Solution Assessment

No, the originally proposed two- or three-file `ui.accessibility.plainText` patch is no longer the best complete shape. The maintainable solution should use the current terminal-specific `tui` boundary, cover the full glyph inventory, and address the upstream SelectList contract.
