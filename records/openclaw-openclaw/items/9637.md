## Summary

Current main still lacks a plain-text TUI mode and continues to emit decorative emoji and Unicode unconditionally. The linked implementation PR was closed unmerged after the feature was deprioritized, so it does not supersede the issue; the remaining work needs a maintainer decision on the public config contract and substitution scope.

## What This Changes

Add a screen-reader-friendly TUI mode that replaces emoji and decorative Unicode symbols with consistent ASCII alternatives.

## Best Possible Solution

Define one `tui`-scoped accessibility mode, resolve it once at TUI startup, and apply a centralized ASCII symbol policy across tool rendering, statuses, selectors, separators, settings, and pi-tui theme or wrapper seams, with config validation, rendering tests, PTY proof, and public documentation.

## Reproduction Assessment

Yes at source level: current main has no accepted plain-text accessibility setting and still emits unconditional emoji and box-drawing characters in the reported TUI paths. A live screen-reader session was not run, so this is source reproduction rather than an end-to-end reproduction.

## Solution Assessment

No, the original `ui.accessibility.plainText` placement is no longer the best shape because current main has a dedicated `tui` configuration namespace. The maintainable solution is a single TUI rendering policy that covers all sibling and dependency-backed glyph surfaces rather than several local boolean branches.
