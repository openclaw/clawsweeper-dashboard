## Summary

Keep this issue open: current main still emits decorative emoji and Unicode throughout the TUI, no accessibility text mode exists, and the only implementation PR was closed unmerged after deprioritization rather than superseded by another solution.

## What This Changes

Add a TUI accessibility mode that replaces decorative emoji and Unicode symbols with screen-reader-friendly ASCII output.

## Best Possible Solution

Define one `tui`-owned accessibility text-mode contract, load it once during TUI startup, and apply a centralized ASCII symbol policy across tool titles, selectors, separators, status text, and pi-tui customization or wrapper seams, with config and PTY rendering coverage plus public TUI documentation.

## Reproduction Assessment

Yes, from source inspection: current main unconditionally emits decorative emoji and Unicode glyphs across multiple TUI render paths. A live screen-reader session was not run, so the reproduction is source-backed rather than an end-to-end accessibility test.

## Solution Assessment

No, the original narrow `ui.accessibility.plainText` sketch is not the best current solution because a dedicated `tui` config boundary now exists and the complete glyph surface needs one centralized policy. The underlying direction remains sound once maintainers choose that contract.
