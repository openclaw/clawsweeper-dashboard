## Summary

Keep open: current main and the latest release still lack a documented TUI plain-text/accessibility mode, the config schema would reject the proposed key, and TUI/tool render paths still emit emoji or Unicode glyphs unconditionally. The linked implementation attempt is closed unmerged, so the remaining work needs maintainer product/config direction rather than cleanup closure.

## What This Changes

The issue requests a TUI accessibility config option that replaces emoji and Unicode symbols with plain ASCII output for screen-reader and limited-terminal users.

## Best Possible Solution

Ship a documented, default-off core TUI plain-text mode that covers config validation/help/docs, tool summaries, selector/theme glyphs, waiting/status glyphs, and focused TUI/config tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. Maintainers still need to choose the exact key name, ASCII replacements, and pi-tui coverage.
