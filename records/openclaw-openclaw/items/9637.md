## Summary

Keep open: current main still has no TUI plain-text/accessibility mode, strict config validation would reject the proposed key, and the central TUI/tool render paths still emit emoji or Unicode glyphs unconditionally. Related PR #9685 was closed unmerged, while the discussion includes a second affected screenreader user, so this remains actionable core TUI/config work.

## What This Changes

Issue requests a TUI accessibility/plain-text config option that disables emoji and Unicode symbols for screenreader and limited-terminal users.

## Best Possible Solution

Implement this as core TUI accessibility support: add a documented ui.accessibility.plainText-style config field, validate it, load it once at TUI startup, and pass it through TUI/tool rendering so ASCII output is deterministic when enabled while preserving current rich output by default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the config key is absent from the exported type and strict zod schema, while tool summaries and TUI components still render emoji/Unicode glyphs without a plain-text branch.

## Solution Assessment

Yes. A core TUI config flag is the narrow maintainable direction because the problematic output is emitted by OpenClaw's own config-validated TUI/tool render paths and cannot be solved cleanly by an external plugin alone.
