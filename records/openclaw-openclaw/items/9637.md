## Summary

Keep open: current main and the latest shipped release still lack a documented TUI plain-text/accessibility mode, while tool summaries and selector/status rendering still emit emoji or Unicode glyphs. The linked implementation PR was closed unmerged, and the discussion includes another screen-reader user confirming the same TUI usability problem.

## What This Changes

Request a documented TUI plain-text accessibility config option that replaces emojis and Unicode glyphs with ASCII-friendly output.

## Best Possible Solution

Ship a default-off core TUI plain-text mode that is documented in config surfaces and consistently applied across tool summaries, selectors, status/waiting glyphs, and TUI rendering tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The exact option name, replacement strings, and pi-tui wrapper scope should be decided before coding.
