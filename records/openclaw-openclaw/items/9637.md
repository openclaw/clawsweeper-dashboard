## Summary

Current main still lacks an ASCII/plain-text TUI mode, and the only implementation PR was closed unmerged after deprioritization rather than replacement. This remains the canonical core accessibility request, but maintainers need to approve its permanent configuration and rendering semantics.

## What This Changes

Add a TUI accessibility mode that replaces decorative emoji and Unicode symbols with screen-reader-friendly ASCII alternatives.

## Best Possible Solution

Approve one terminal-specific accessibility contract, load it once during TUI construction, and pass a canonical ASCII-safe symbol set through selectors, statuses, tool summaries, spinners, and all other visible TUI rendering seams with focused config, component, and PTY coverage.

## Reproduction Assessment

Yes at source level: current main directly emits decorative emoji and Unicode in TUI rendering and exposes no accepted plain-text setting. A live screen-reader session was not run.

## Solution Assessment

No, the original narrow two-to-three-file proposal is not the best complete fix. The maintainable path is a single current-architecture TUI rendering policy covering every visible glyph source under an approved terminal configuration namespace.
