## Summary

Current main still emits decorative emoji and Unicode throughout the TUI, and the only linked implementation PR closed unmerged; the issue remains necessary, but the original `ui.accessibility.plainText` design should be reconsidered against the newer terminal-specific `tui` config boundary.

## What This Changes

Add a TUI accessibility mode that replaces decorative emoji and Unicode symbols with screen-reader-friendly plain text or ASCII alternatives.

## Best Possible Solution

Add one documented terminal-owned accessibility policy under `tui`, resolve it once at TUI startup, and use shared ASCII presentation tokens across selectors, themes, waiting/status output, tool headings, and previews without rewriting user or model content.

## Reproduction Assessment

Yes, from source: current-main TUI paths unconditionally emit decorative emoji and Unicode and expose no accessibility branch. A live screen-reader session would improve UX validation but is not needed to establish the missing behavior.

## Solution Assessment

No for the exact original implementation shape. The narrowest maintainable solution should use the newer `tui` configuration boundary and a shared rendering policy that covers all current decorative-glyph paths, rather than reviving the older partial `ui.accessibility` patch unchanged.
