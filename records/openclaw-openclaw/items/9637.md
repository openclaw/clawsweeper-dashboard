## Summary

Keep open: current main still lacks an accepted or documented TUI plain-text accessibility mode, strict config validation would reject the proposed key, and the linked implementation was closed unmerged.

## What This Changes

The issue requests a documented config option, such as `ui.accessibility.plainText`, to render TUI tool/status/selector output without emoji and Unicode glyphs.

## Best Possible Solution

Ship a documented core TUI plain-text mode that is read once from config, validated in schema/generated docs, and applied consistently across tool summaries, selectors, settings, waiting/status text, and relevant TUI component glyphs.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented TUI plain-text config key, and inspected TUI/tool render paths still emit emoji or Unicode glyphs. This is a feature/config request rather than a broken existing contract.

## Solution Assessment

Yes in broad direction: a default-off core config flag is the maintainable boundary because OpenClaw owns config validation, docs, and built-in TUI renderers. The option name and ASCII replacement set still need maintainer agreement before implementation.
