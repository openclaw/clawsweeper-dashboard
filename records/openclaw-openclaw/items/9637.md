- `P2`: The issue is a source-confirmed accessibility improvement with meaningful but bounded impact on the built-in TUI.
- `impact:other`: The concrete affected problem class is screen-reader accessibility, which is outside the narrower owned impact categories.

## Summary

Current main and the latest release still render emoji, Unicode cursors, and box-drawing separators without a plain-text mode, while the only linked implementation PR closed unmerged after deprioritization; this remains useful core TUI accessibility work, but the public config shape and complete ASCII presentation policy need maintainer approval.

## What This Changes

Add an opt-in plain-text rendering mode for the built-in terminal UI to reduce screen-reader noise from emoji and decorative Unicode symbols.

## Best Possible Solution

Add a TUI-owned setting such as `tui.accessibility.plainText`, load it once at startup, and route every visible TUI glyph through one ASCII-capable symbol/theme policy while preserving existing non-TUI display behavior.

## Reproduction Assessment

Yes at the source level: current main and `v2026.6.5` visibly emit emoji, arrows, and box-drawing characters without an alternative mode, and two screen-reader users report difficulty; no live assistive-technology session was run.

## Solution Assessment

The capability is appropriate, but the original `ui.accessibility.plainText` proposal is no longer the best exact shape because current config has a dedicated `tui` section and shared tool-display helpers serve non-TUI callers.
