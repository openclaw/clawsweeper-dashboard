## Summary

Current main and the latest release still lack a TUI plain-text mode, while OpenClaw and its pi-tui dependency emit screen-reader-hostile decorative glyphs unconditionally; the only implementation PR closed unmerged after deprioritization, so the accessibility request remains valid but needs maintainer UX/config direction.

## What This Changes

Add a TUI accessibility mode that replaces decorative emoji and Unicode symbols with screen-reader-friendly ASCII or plain text.

## Best Possible Solution

Adopt one documented core TUI plain-text accessibility mode, read it once at startup, pass a closed presentation profile through TUI renderers, provide consistent ASCII alternatives across tool summaries, selectors, settings, status/spinner surfaces, and either add a pi-tui cursor seam or use an OpenClaw-owned selector implementation.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 lack an accessibility setting and unconditionally render emoji, arrows, and box-drawing characters; two screen-reader users independently described the resulting usability problem. A live screen-reader session was not run during this read-only review.

## Solution Assessment

Unclear: a unified plain-text presentation profile is the narrowest maintainable direction, but the issue’s proposed boolean and replacement strings are product choices, and an emoji-only patch would not cover the full current glyph surface or pi-tui contract.
