## Summary

Keep open: current main still lacks a documented TUI plain-text/accessibility mode, the strict config schema would reject the proposed key, and TUI/tool rendering still emits emoji or Unicode glyphs. The linked implementation PR was closed unmerged, and another screen-reader user confirmed the same usability problem in the discussion.

## What This Changes

Requests a default-off TUI accessibility/plain-text config option to suppress emoji and Unicode glyphs in tool summaries, selectors, and related terminal UI output.

## Best Possible Solution

Ship a default-off core TUI plain-text mode that is documented in config surfaces and consistently applied across tool summaries, selectors, status/waiting glyphs, and TUI rendering tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The exact option name, replacement vocabulary, and pi-tui wrapper scope should be decided before coding.
