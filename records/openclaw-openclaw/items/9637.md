## Summary

Keep open: current main still lacks a documented TUI plain-text/accessibility mode, strict config validation would reject the proposed key, and TUI render paths still emit emoji or Unicode glyphs. The related implementation PR is closed unmerged, while another screen-reader user confirmed the same usability problem.

## What This Changes

Add a user-facing TUI accessibility config option that renders tool summaries and selector/status UI with ASCII/plain-text alternatives instead of emoji and Unicode glyphs.

## Best Possible Solution

Ship a default-off core TUI plain-text mode documented in config/help/docs and consistently applied across tool summaries, selectors, status/waiting glyphs, settings rendering, and focused tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The exact key name, ASCII replacements, and pi-tui/theme scope still need maintainer direction.
