## Summary

Keep open: current main and the latest release still do not have a documented TUI plain-text/accessibility mode, while config validation and TUI/tool rendering still emit or allow only emoji and Unicode glyph paths. The related implementation attempt at https://github.com/openclaw/openclaw/pull/9685 is closed unmerged, and another screen-reader user confirmed the issue in discussion.

## What This Changes

The issue requests a default-off TUI config mode that renders emoji and decorative Unicode as plain ASCII for screen-reader users.

## Best Possible Solution

Ship a default-off core TUI plain-text mode documented in config/help/docs and applied consistently across tool summaries, selectors, status/waiting glyphs, settings rendering, and focused tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. Maintainers still need to choose the exact key name, ASCII replacements, and selector/pi-tui coverage.
