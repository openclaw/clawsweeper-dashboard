## Summary

Keep open: current main still lacks a documented TUI plain-text/accessibility mode, strict config validation would reject the proposed key, and TUI/tool render paths still emit emoji or Unicode glyphs. The related implementation PR was closed unmerged, and another screen-reader user confirmed the same usability problem.

## What This Changes

The issue requests a TUI accessibility config option to render tool labels, selectors, separators, and status output with plain ASCII instead of emoji and Unicode glyphs.

## Best Possible Solution

Ship a default-off core TUI plain-text mode that is documented in config surfaces and consistently applied across tool summaries, selectors, status/waiting glyphs, and focused rendering tests.

## Reproduction Assessment

Yes, for source-level verification: current main has no accepted or documented plain-text config key, and inspected TUI/tool render paths still emit emoji and Unicode glyphs. I did not run a live screen-reader session in this read-only review.

## Solution Assessment

Yes in broad direction: a default-off core TUI config mode is the maintainable boundary because OpenClaw owns strict config validation, public docs, and built-in TUI renderers. The exact option name, ASCII replacement vocabulary, and pi-tui wrapper scope still need maintainer direction.
