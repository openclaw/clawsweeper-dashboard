## Summary

Keep open. Current main and the latest release still emit warn-level escaped skill-path messages, while the exact Markdown-link mechanism in the report is not proven by the inspected loader path; the closed related PR did not merge, so the noisy-warning behavior remains unresolved.

## What This Changes

The issue asks OpenClaw to avoid misleading startup warnings when intentional sibling skill references appear in `SKILL.md` documentation while preserving skill-root containment checks.

## Best Possible Solution

Keep the issue open and make a focused skills-loader/logging decision after confirming the exact sibling-reference shape. The desired end state is no misleading startup noise for intentional local sibling references, while true escaped skill roots remain rejected and visible through an appropriate warning, doctor check, or documented debug path.

## Reproduction Assessment

Unclear for the exact Markdown-link claim: the inspected loader scans filesystem candidates and reads frontmatter rather than enforcing Markdown body links. Yes for the underlying warning class, because current tests construct escaped workspace skill paths and assert the warn-level console output.

## Solution Assessment

Unclear. Downgrading every escaped-path warning is likely too broad because it weakens containment observability; a narrower suppression, non-enforceable reference convention, or doctor/debug split would be safer after maintainer confirmation.
