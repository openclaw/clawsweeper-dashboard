## Summary

Keep open: current main still lets malformed memory-wiki frontmatter throw through `wiki_apply` and `wiki_lint`, and two open PRs explicitly target this issue but neither has merged.

## What This Changes

The issue asks for `wiki_apply` and `wiki_lint` to tolerate or report malformed wiki YAML frontmatter instead of returning a vault-wide YAML parse error after a write/lint operation.

## Best Possible Solution

Review and land one focused memory-wiki fix that prevents unrelated-page bad frontmatter from aborting vault compilation, reports the offending page clearly, and keeps edit-target parsing strict enough to avoid metadata loss.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: malformed frontmatter reaches `parseWikiMarkdown` and `YAML.parse`, then `readPageSummaries` stops and rethrows through `compileMemoryWikiVault`. I did not run a local tool repro because this review was read-only.

## Solution Assessment

Yes, the best direction is a focused memory-wiki compile/summary parsing fix, not a new `ignore_yaml_errors` option. The linked PRs target the right surface; the better final shape should preserve edit-target metadata while turning unrelated bad pages into actionable lint output.
