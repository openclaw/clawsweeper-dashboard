## Summary

Keep open: current main and the latest release still have the source-level failure path for malformed memory-wiki frontmatter, while open candidate PRs already target the fix and should be reviewed or reconciled before this issue closes.

## What This Changes

The issue reports that malformed YAML frontmatter in one memory-wiki page can make `wiki_apply` and `wiki_lint` return a YAML parse error even after the requested write or lint work should succeed.

## Best Possible Solution

Land one summary/lint-scoped memory-wiki fix that degrades only unrelated bad pages into actionable lint output while keeping direct edit-target parsing strict, then close this issue after merge.

## Reproduction Assessment

Yes. Source inspection shows malformed frontmatter reaches `YAML.parse`, `readPageSummaries` stops and rethrows, and both reported tools can reach that compile path; I did not execute a live tool repro in this read-only review.

## Solution Assessment

Yes. The best fix is to catch malformed frontmatter at the vault-wide summary/lint boundary and report the bad page, not to add an `ignore_yaml_errors` option or make the shared parser silently return empty frontmatter for edit-target reads.
