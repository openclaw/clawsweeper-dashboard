## Summary

Keep open: current main and the latest release still let malformed memory-wiki frontmatter throw through vault compilation, and open candidate PRs already target the reported `wiki_apply`/`wiki_lint` failure.

## What This Changes

The issue asks for `wiki_apply` and `wiki_lint` to tolerate or report malformed frontmatter on unrelated wiki pages instead of returning a YAML parse error after the requested operation writes.

## Best Possible Solution

Land one focused memory-wiki scan-boundary fix that degrades unrelated bad frontmatter into an actionable lint issue while keeping edit-target parsing strict enough to avoid metadata loss.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create a vault page with invalid YAML frontmatter, then run `wiki_apply` or `wiki_lint`; current main calls `YAML.parse` without a catch and rethrows through `readPageSummaries`. I did not run the tool end-to-end in this read-only review.

## Solution Assessment

Yes, the best direction is a focused `toWikiPageSummary`/vault-scan fix that records invalid frontmatter for lint, not a new `ignore_yaml_errors` flag or blanket parser-level ignore that could affect edit paths.
