## Best Possible Solution

Land a focused memory-wiki fix that tolerates malformed frontmatter only during unrelated-page summary/lint scans, reports the offending page clearly, and keeps edit-target parsing strict enough to avoid metadata loss.

## Solution Assessment

Yes. The best direction is a summary/lint-scoped tolerance path, not a new `ignore_yaml_errors` option or a global parser catch that could erase edit-target metadata.
