## What This Changes

The issue reports that `wiki_apply` and `wiki_lint` fail when memory-wiki vault compilation encounters another page with malformed YAML frontmatter.

## Best Possible Solution

Review and land one focused memory-wiki fix that tolerates unrelated-page bad frontmatter during vault summary compilation, reports the offending page clearly, and keeps direct edit-target parsing strict enough to avoid metadata loss.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: malformed frontmatter reaches `YAML.parse`, `readPageSummaries` stops and rethrows, and both `wiki_apply` and `wiki_lint` call vault compilation; I did not run a local tool repro because this review was read-only.

## Solution Assessment

Yes. The best direction is a focused memory-wiki compile/summary parsing fix, not a new `ignore_yaml_errors` option; the open linked PR targets the right boundary by preserving strict direct parsing while surfacing unrelated bad pages as lint output.
