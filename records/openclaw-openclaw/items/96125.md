## Summary

Keep open: current main still has an uncaught memory-wiki YAML frontmatter parse path that can make both wiki_apply and wiki_lint fail after encountering malformed vault frontmatter.

## What This Changes

The issue reports that wiki_apply and wiki_lint fail while parsing malformed YAML frontmatter in a memory-wiki page, even though the wiki_apply write appears to succeed.

## Best Possible Solution

Keep the issue open and fix memory-wiki so malformed page frontmatter becomes a structured lint/compile diagnostic instead of an uncaught tool failure; avoid adding an ignore_yaml_errors option unless maintainers explicitly want that product surface.

## Reproduction Assessment

Yes, source-level: a wiki page with malformed YAML frontmatter can make parseWikiMarkdown throw, and both compileMemoryWikiVault callers used by wiki_apply and wiki_lint rethrow that path. I did not run the live tool because this checkout lacks installed dependencies.

## Solution Assessment

Yes, if the fix stays in memory-wiki parsing/compile/lint behavior and reports malformed page frontmatter as diagnostics. The requested ignore_yaml_errors parameter would be a less desirable new config/tool surface and should not be the default repair.
