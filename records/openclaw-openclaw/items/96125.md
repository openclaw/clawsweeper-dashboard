## What This Changes

The issue reports that `wiki_apply` and `wiki_lint` can return a YAML parse error from malformed memory-wiki frontmatter on another vault page after the requested operation has otherwise succeeded.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96177 or an equivalent memory-wiki fix that tolerates malformed unrelated-page frontmatter during vault summaries, reports an actionable lint issue, and keeps edit-target parsing strict.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: malformed frontmatter reaches `parseWikiMarkdown` and `YAML.parse`, then `readPageSummaries` rethrows through `compileMemoryWikiVault`; both `wiki_apply` and `wiki_lint` call that compile path.

## Solution Assessment

Yes for the linked summary/lint-scoped direction, not for a new ignore flag or a global parser catch. The narrow maintainable fix should tolerate bad unrelated pages during vault scans while preserving strict parsing for pages being edited.
