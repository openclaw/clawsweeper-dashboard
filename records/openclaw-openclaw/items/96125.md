## Summary

Keep open: current main and the latest checked release still parse memory-wiki frontmatter strictly during vault-wide summary reads, so malformed YAML on an unrelated page can still make `wiki_apply` and `wiki_lint` fail. The remaining work is already owned by the open candidate fix at https://github.com/openclaw/openclaw/pull/96177, so no new repair PR should be queued.

## What This Changes

The issue asks OpenClaw to stop `wiki_apply` and `wiki_lint` from failing when another memory-wiki page has malformed YAML frontmatter.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96177 or an equivalent memory-wiki fix that tolerates malformed unrelated-page frontmatter during vault-wide summary/lint reads while keeping direct edit-target parsing strict.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: malformed frontmatter reaches `parseWikiMarkdown`, `YAML.parse` can throw by dependency contract, and `readPageSummaries` rethrows through `compileMemoryWikiVault` used by both reported tools.

## Solution Assessment

Yes, the best direction is the linked summary/lint-scoped memory-wiki fix rather than a new `ignore_yaml_errors` option or a global parser catch. That shape preserves strict parsing for pages being edited while making unrelated malformed pages actionable lint findings.
