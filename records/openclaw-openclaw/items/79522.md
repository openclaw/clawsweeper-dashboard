## Summary

Keep open: current main still treats an ACP stream that reaches `done` as a successful background task, and the reported failure mode is specifically success after progress-only output. The closed related PR targeted this issue but was not merged and the author agreed the core text-prefix heuristic was the wrong direction.

## What This Changes

ACP/Codex spawned background runs should not be marked succeeded when the child produced only an initial progress message and no task result.

## Best Possible Solution

Capture a real redacted `sessions_spawn(runtime="acp", agentId="codex")` event stream, then fix the Codex ACP/acpx runtime boundary so premature close or adapter failure becomes `failed` or `timed_out` without rejecting legitimate text-only successful turns.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows current main will mark a parented ACP turn succeeded once a `done` event arrives, even if the visible output was only text progress, but the real Codex ACP event stream still needs capture.

## Solution Assessment

Unclear; the closed PR’s core progress-text heuristic is not the best fix because ACP success does not require tool calls. The safer solution is adapter/runtime-level failure signaling for premature termination, backed by a real Codex ACP trace.
