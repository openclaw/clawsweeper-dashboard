## Summary

Keep open: the report describes a current release regression, and current main still has source-level completion semantics where an ACP/Codex run can be recorded as successful after the runtime emits a terminal done event without proving that the requested work produced a substantive final result.

## What This Changes

Investigate ACP/Codex sessions that report success after emitting only an initial progress sentence and no final work result.

## Best Possible Solution

Add a regression-backed ACP/Codex completion contract so a one-shot Codex ACP run cannot be marked successful unless it produces a final answer or substantive tool/result evidence; otherwise surface a timeout or clear error.

## Reproduction Assessment

No high-confidence live reproduction in this review. The source path is clear and the reporter supplied two current-release observations, but I did not run a real ACP/Codex session in this shell.

## Solution Assessment

Unclear. A source-level guard in OpenClaw may be narrow, but maintainers should first confirm whether Codex ACP can expose final-versus-progress events or whether OpenClaw should classify progress-only/tool-less Codex ACP completions as failed.
