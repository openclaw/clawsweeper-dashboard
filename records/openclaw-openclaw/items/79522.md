## Summary

Keep open: the report is fresh, externally authored, and still matches a real ACP/Codex completion boundary on current main. OpenClaw still treats a terminal ACP `done` as success even when the only visible output was progress text, and the attempted core heuristic was closed as the wrong fix direction.

## What This Changes

ACP/Codex spawned runs should not report successful completion when the child session only produced initial progress text and no substantive result.

## Best Possible Solution

Capture the real Codex/acpx event stream on current main, then fix the adapter/runtime contract so premature progress-only termination surfaces as `failed` or `timed_out` while legitimate text-only completed turns still succeed.

## Reproduction Assessment

No high-confidence live reproduction is available yet. The source path is clear for a `text_delta` followed by `done` being marked successful, but the real current-main Codex ACP event sequence still needs a redacted adapter-level capture.

## Solution Assessment

Unclear until the real event stream is captured. The attempted core text-prefix/tool-call heuristic is not the best fix; the safer path is to make the Codex ACP/acpx completion signal authoritative or fail explicitly when it is not.
