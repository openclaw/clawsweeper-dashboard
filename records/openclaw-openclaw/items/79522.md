## Summary

Keep open: the report still describes an unresolved ACP/Codex false-success boundary, and the only attempted fix was closed after reviewers agreed its core heuristic could break valid text-only ACP turns.

## What This Changes

The issue asks OpenClaw to stop reporting spawned ACP/Codex runs as successful when the child session only emits an initial progress message and does not perform the requested work.

## Best Possible Solution

Capture a redacted real Codex/acpx event stream for this failure, then fix the adapter or runtime contract so premature close/error becomes a failed ACP turn while valid text-only completed turns still succeed.

## Reproduction Assessment

No high-confidence live reproduction against current main yet. The reporter provided two concrete 2026.5.7 observations, and source inspection shows a text_delta followed by done still reaches the succeeded path, but the real Codex ACP adapter stream needs to be captured.

## Solution Assessment

No, the closed PR's core text-prefix/no-tool-call heuristic is not the best fix because ACP success does not require tool calls. The safer path is adapter/runtime-level failure signaling once the real event stream proves where the premature success originates.
