## Summary

Keep open: current main still treats a terminal ACP `done` event as successful completion for parented/background tasks, and the only linked fix PR was closed unmerged after review found its core progress-text heuristic unsafe. The exact Codex/acpx event stream still needs owner-level reproduction before a narrow fix can be chosen.

## What This Changes

The issue asks ACP/Codex spawned runs not to report successful completion when the child session only emits an initial progress sentence and no substantive work or final result.

## Best Possible Solution

Capture a redacted real Codex/acpx event stream for this failure and fix the adapter or ACP runtime contract so premature close/error becomes `ACP_TURN_FAILED` while legitimate text-only completed turns still succeed.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection shows the core outcome if Codex/acpx emits progress text followed by `done`, but the real adapter event stream has not been captured.

## Solution Assessment

Unclear until the real Codex/acpx stream is known. The closed PR's core progress-text heuristic is not the best fix; the safer direction is adapter/runtime-level failure signaling for premature termination.
