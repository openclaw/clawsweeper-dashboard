## Summary

Keep open. The crash-hardening target is still relevant and current main only partially overlaps, but the PR is not merge-ready because the new uncaught-exception suppression can run before fatal/config-style precedence and needs reconciliation with current main's narrower benign-uncaught policy.

## What This Changes

The branch adds strict transient network/TLS suppression to both global uncaughtException handlers, attaches a raw gateway upgrade-socket error listener, and expands classifier unit coverage.

## Best Possible Solution

Preserve the crash-hardening goal, but implement it through the current uncaught-exception policy seam with explicit fatal/config precedence, narrow TLS/undici/socket signatures, and regression coverage for both CLI entrypoints plus the upgrade-socket path.

## Reproduction Assessment

No high-confidence deterministic reproduction is available. The linked reports provide detailed intermittent WSL2/concurrent-subagent crash evidence, but the PR itself still marks unstable-network gateway verification as manual and incomplete.

## Solution Assessment

No. The direction is plausible, but the current PR suppresses before fatal precedence and should be revised so fatal wrappers cannot be hidden by nested transient network causes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve fatal precedence before suppressing uncaught errors:** `src/cli/run-main.ts:164-169`
  - body: This handler returns for isStrictTransientNetworkError(error) before any fatal/config-style classification. Because the strict classifier walks nested causes, a fatal wrapper such as ERR_WORKER_UNCAUGHT_EXCEPTION with a transient network cause can be logged as non-fatal and leave the CLI or gateway running after a fatal exception. Keep fatal checks ahead of transient suppression or make the strict check ignore fatal wrappers.
  - confidence: 0.86
