## Summary

Keep open: the linked gateway crash is plausible and current main still has a narrower uncaught-exception allowlist, but this PR should not merge as-is because its suppression path can outrank fatal handling and the PR lacks after-fix real behavior proof.

## What This Changes

The PR adds strict transient network/TLS classification to uncaughtException handlers, installs a raw gateway upgrade socket error listener, and adds classifier tests.

## Best Possible Solution

Rebase the fix onto the current uncaught-exception policy seam, add narrow TLS/undici signatures while preserving fatal/config precedence, and include focused tests plus redacted real gateway proof.

## Reproduction Assessment

No high-confidence deterministic live reproduction is available. Source inspection confirms the current-main allowlist gap, and the linked reports provide detailed intermittent WSL2/concurrent-agent crash evidence.

## Solution Assessment

No. The direction is useful, but the PR should integrate with the current `isBenignUncaughtExceptionError`/handler seam and keep fatal wrappers ahead of transient suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve fatal precedence before suppressing uncaught errors:** `src/cli/run-main.ts:164-169`
  - body: This handler returns for `isStrictTransientNetworkError(error)` before fatal/config-style classification, and the strict classifier walks nested causes. A fatal wrapper such as `ERR_WORKER_UNCAUGHT_EXCEPTION` with a transient network cause can be logged as non-fatal and leave the CLI or gateway running after a fatal exception, so keep fatal wrappers ahead of transient suppression or make the strict check ignore them.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes global uncaught-exception handling and can keep the process running after a fatal wrapper when a nested transient cause matches.

  - body: Because the new uncaught-exception suppression checks nested transient causes before fatal handling, a fatal worker/runtime wrapper with a network cause can bypass process exit and leave the gateway in an undefined state.
