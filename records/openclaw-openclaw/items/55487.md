## Summary

Keep open. The PR targets a real remaining gateway reliability gap, but the branch should not merge as-is because its uncaught-exception suppression can run before fatal-wrapper precedence and it is stale against current main's newer handler seam.

## What This Changes

The PR adds strict transient network/TLS suppression to both uncaughtException handlers, a raw gateway upgrade-socket error listener, and classifier unit coverage.

## Best Possible Solution

Rebase onto current main and implement the TLS/undici uncaught-exception suppression through the existing uncaught handler policy seam, preserving fatal wrapper behavior and adding focused regression coverage plus the upgrade-socket guard if still needed.

## Reproduction Assessment

No high-confidence deterministic reproduction is available. The linked reports provide detailed intermittent WSL2/concurrent-subagent crash evidence, and current source inspection confirms gaps, but the PR's own unstable-network verification remains manual.

## Solution Assessment

No. The direction is plausible, but this patch is not the best current solution because it layers stale handler logic over a newer policy seam and can suppress nested fatal wrappers as non-fatal network errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve fatal precedence before suppressing uncaught errors:** `src/cli/run-main.ts:164-169`
  - body: This handler returns for `isStrictTransientNetworkError(error)` before any fatal/config-style classification, and the strict classifier walks nested causes. A fatal wrapper such as `ERR_WORKER_UNCAUGHT_EXCEPTION` with a transient network cause can be logged as non-fatal and leave the CLI or gateway running after a fatal exception, so keep fatal wrappers ahead of transient suppression or make the strict check ignore them.
  - confidence: 0.86
