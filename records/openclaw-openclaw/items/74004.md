## Summary

Keep open. Current main still lacks the Claude CLI write-back path this PR is trying to add, but the proposed patch has blocking correctness and security gaps around external OAuth secret writes that should be fixed before merge.

## What This Changes

The PR adds an optional OAuth-manager post-refresh hook and wires it to write refreshed `anthropic:claude-cli` OAuth credentials back to the Claude CLI file or macOS Keychain via the existing Claude CLI credential writer.

## Best Possible Solution

If bidirectional Claude CLI sync is accepted, keep the hook narrow: only write back validated `anthropic:claude-cli` / `claude-cli` OAuth credentials after a successful refresh, use redacted or omitted error details, cover the new boundary with focused tests, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes for the source-level behavior: current main can refresh an expired OAuth profile into OpenClaw's store, but there is no hook capable of writing the rotated token back to Claude CLI storage. I did not run a live OAuth/Keychain race because this review was read-only.

## Solution Assessment

No, not as written. The direction may be the narrow product fix, but the patch should guard on both profile id and provider, sanitize write-back logging, correct the documented scope, and cover the new external secret write boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard write-back by provider as well as profile:** `src/agents/auth-profiles/external-cli-sync.ts:170-171`
  - body: This writes to the Claude CLI credential store whenever the profile id is `anthropic:claude-cli`, but it does not verify that the refreshed credential is actually for provider `claude-cli`. A misconfigured or legacy profile using that id with another OAuth provider would copy unrelated OAuth tokens into Claude CLI storage; require the expected provider/profile pair before calling the writer.
  - confidence: 0.84
- **[P2] Redact write-back error messages:** `src/agents/auth-profiles/external-cli-sync.ts:184-185`
  - body: The new write-back catch logs `String(error)` while handling OAuth credential writes. Unexpected adapter or writer errors can include serialized credential payloads, so this should use the repo's redacting `formatErrorMessage()` path or omit the error message before adding this token-adjacent log sink.
  - confidence: 0.88
- **[P3] Correct the write-back scope comment:** `src/agents/auth-profiles/external-cli-sync.ts:151-152`
  - body: The new JSDoc says the helper writes to Claude CLI and Codex CLI stores, but the implementation only handles `CLAUDE_CLI_PROFILE_ID`. Since Codex is intentionally bootstrap-only on current main, the comment should name only Claude CLI unless Codex write-back is actually implemented.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/agents/auth-profiles/oauth.ts:165-168`
  - body: This is a user-facing auth fix, but the PR only changes auth TypeScript files and omits the required `CHANGELOG.md` entry. Add a single-line Fixes entry with contributor credit before merge.

## Security Review

Status: needs_attention

Summary: The diff introduces OAuth secret write-back to an external credential store and needs provider scoping plus redacted logging before it is safe to merge.

Concerns:

- **[medium] Profile-only guard can write wrong-provider tokens:** `src/agents/auth-profiles/external-cli-sync.ts:170`
  - body: The helper keys the external write solely on `CLAUDE_CLI_PROFILE_ID`; it should also require `credential.provider === "claude-cli"` before placing refreshed OAuth material into Claude CLI storage.
  - confidence: 0.84
