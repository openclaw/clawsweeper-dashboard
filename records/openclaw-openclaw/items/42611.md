## Summary

Keep this PR open for rework before merge. The enterprise proxy topic is useful, but the branch conflicts with current docs/i18n policy, current gateway service naming, managed proxy configuration, and includes raw diagnostics that can expose proxy credentials.

## What This Changes

This PR adds a corporate proxy and SSL inspection guide to `docs/vps.md` plus a localized `docs/zh-CN/vps.md` copy covering systemd environment variables, `NODE_EXTRA_CA_CERTS`, and troubleshooting commands.

## Best Possible Solution

Rework this as an English-source update centered on `docs/security/network-proxy.md`, add only a short `docs/vps.md` cross-link, use the current managed proxy/service contract, and keep diagnostics redaction-safe while leaving translation to the publish/i18n pipeline.

## Reproduction Assessment

Not applicable as a runtime bug. The review findings are high-confidence from comparing the PR patch with current main docs and service/proxy contract files.

## Solution Assessment

No. The topic is valid, but this branch is not the best solution because it targets stale service commands, bypasses the managed proxy configuration guidance, edits localized docs in the source repo, and asks users to print raw proxy diagnostics.

  - body: Current proxy docs and service environment code prefer `proxy.enabled` plus `proxy.proxyUrl`, or durable `OPENCLAW_PROXY_URL` for installed services.   - confidence: 0.9
