## Summary

Keep open: the PR targets a real memory-flush gap, but the helper-level trigger also broadens same-turn preflight-compaction behavior and the PR still lacks real-session proof for the in-turn compaction-to-memory-write path.

## What This Changes

The branch widens `shouldRunMemoryFlush` to flush when `compactionCount` is ahead of `memoryFlushCompactionCount` even below the token threshold, and adds helper-level regression tests.

## Best Possible Solution

Land a version that either scopes catch-up to pending in-turn compaction or explicitly accepts and tests the broader below-threshold unflushed-compaction semantics, with redacted real-session proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at the source-helper level: current main returns false below the token threshold before considering an unflushed compaction count, although I did not reproduce a full long-running OpenClaw session.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the helper-level catch-up is a plausible narrow fix for in-turn compaction, but current caller order makes it affect same-turn preflight compaction too, so it needs scoped gating or explicit maintainer acceptance.
