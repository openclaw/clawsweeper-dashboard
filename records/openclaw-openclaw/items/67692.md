## Summary

Keep open: current main still has the documented ACP runtime default gap for first-turn TUI sessions, but this PR is not merge-ready because real behavior proof is missing and the lazy-init path still has correctness and security-boundary defects.

## What This Changes

This PR adds lazy ACP session initialization for `runtime.type: "acp"` agents in auto-reply/TUI and CLI command paths, extends resolved agent config fields/tests, and adds a changelog entry.

## Best Possible Solution

Centralize ACP lazy initialization after policy and non-empty-turn checks, reuse the existing binding/runtime init contract, exclude raw runs, rebuild prompt state after re-resolve, and land with focused tests plus redacted real TUI ACP proof.

## Reproduction Assessment

Yes, source-reproducible. Current main returns `null` from `tryDispatchAcpReply` when ACP metadata is absent, even though docs expose `agents.list[].runtime.type="acp"` as an agent runtime default.

## Solution Assessment

No. Lazy initialization is the right direction, but this implementation is not the best fix until it shares the existing ACP init contract, preserves binding/default semantics, excludes raw runs, and proves the real TUI path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate lazy ACP init out of raw runs:** `src/agents/agent-command.ts:495`
  - body: This condition is not gated by `!isRawModelRun`, while the ACP `runTurn` branch below still is. A raw `modelRun` or `promptMode: "none"` request on an ACP-configured session can start an external ACP runtime and write metadata before continuing down the embedded/raw path.
  - confidence: 0.9
- **[P2] Rebuild the ACP turn body after re-resolve:** `src/agents/agent-command.ts:521`
  - body: `body` is computed before lazy initialization, when `acpResolution` is still `none`. After the re-resolve makes the session ready, `runTurn` still receives the stale embedded-format body instead of the ACP-specific prompt form for internal-event turns.
  - confidence: 0.88
- **[P2] Count inline images before skipping lazy init:** `src/auto-reply/reply/dispatch-acp.ts:391`
  - body: This guard returns before lazy init when there is no prompt text and no inbound channel media, but the normal ACP path also supports inline `params.images`. A first inline-image-only TUI/WebChat turn still falls back to embedded instead of initializing the ACP session.
  - confidence: 0.86
- **[P2] Use configured binding specs for binding sessions:** `src/auto-reply/reply/dispatch-acp.ts:394`
  - body: The lazy-init path rebuilds init input from only `agents.list[].runtime.acp`, so a configured ACP binding session can lose `bindings[].acp` overrides such as `mode`, `cwd`, and `backend`, or fail to initialize when the owning agent is not itself `runtime.type: "acp"`.
  - confidence: 0.82
- **[P2] Map global aliases to the configured default agent:** `src/auto-reply/reply/dispatch-acp.ts:86-95`
  - body: When `resolveMainSessionKey(cfg)` returns the literal `global`, this helper feeds that through `resolveAgentIdFromSessionKey`, which collapses it to `main`. In global-session configs whose default agent is not `main`, lazy init looks up the wrong agent config and can skip ACP.
  - confidence: 0.8
