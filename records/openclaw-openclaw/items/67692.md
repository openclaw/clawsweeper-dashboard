## Summary

Keep open: current main still has the ACP fallback gap, this PR is the only matching canonical item I found, and the latest PR head still has correctness/security blockers plus missing real TUI ACP proof.

## What This Changes

The PR adds lazy ACP session initialization for agent-configured ACP runtimes in TUI/auto-reply and CLI agent command paths, extends resolved agent config fields/tests, and adds a changelog entry.

## Best Possible Solution

Centralize lazy ACP initialization after policy and non-empty-turn checks, keep raw model runs side-effect free, preserve the established binding/backend/default init contract, cover TUI/CLI regressions, and require real TUI ACP proof.

## Reproduction Assessment

Yes by source inspection. Current main returns null from `tryDispatchAcpReply` when ACP metadata is absent, so a first TUI turn for an agent configured with `runtime.type="acp"` falls back before any ACP session can be initialized; I did not run a live ACP harness in this read-only review.

## Solution Assessment

No, not yet. Lazy initialization is the right direction, but this implementation should be revised to avoid raw-run side effects, recompute ACP prompt state after init, include inline images, preserve binding/default init semantics, and add real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate lazy ACP init out of raw runs:** `src/agents/agent-command.ts:495`
  - body: This condition runs before the later `!isRawModelRun` ACP execution gate. A raw `modelRun` or `promptMode: "none"` request on an ACP-configured session can start an external ACP runtime and write metadata, then continue down the embedded/raw path.
  - confidence: 0.88
- **[P2] Recompute the ACP prompt after lazy init:** `src/agents/agent-command.ts:521`
  - body: `body` is computed before lazy initialization, while `acpResolution` is still `none`. After the re-resolve makes the session ready, `runTurn` still receives the stale embedded-format body instead of the ACP-specific prompt body for internal-event turns.
  - confidence: 0.87
- **[P2] Include inline images in the empty-turn guard:** `src/auto-reply/reply/dispatch-acp.ts:391`
  - body: This guard returns before lazy init when there is no prompt text and no inbound channel media, but the normal ACP path also supports inline `params.images`. A first inline-image-only TUI/WebChat turn still falls back to embedded instead of initializing ACP.
  - confidence: 0.85
- **[P2] Preserve ACP binding init options:** `src/auto-reply/reply/dispatch-acp.ts:110`
  - body: The lazy-init helper only reads `agents.list[].runtime` and returns null unless that agent is ACP-configured. Configured ACP bindings can be owned by embedded agents and can carry their own mode/cwd/backend overrides, so this path can skip init or initialize with different inputs than the established binding lifecycle.
  - confidence: 0.78
- **[P2] Use the configured default for global aliases:** `src/auto-reply/reply/dispatch-acp.ts:87`
  - body: When `session.scope` is global, `resolveMainSessionKey(cfg)` returns `global`, and `resolveAgentIdFromSessionKey("global")` becomes `main` instead of the configured default agent. A global TUI session for a non-main default ACP agent can still look up the wrong config and skip lazy init.
  - confidence: 0.77
