## Summary

Keep open: current main still has the ACP TUI fallback gap, but the latest PR branch still has blocking correctness/security issues and lacks after-fix real behavior proof.

## What This Changes

The branch adds lazy ACP session initialization for agent-configured ACP runtimes in TUI/auto-reply and CLI agent command paths, extends resolved agent config fields, adds ACP dispatch tests, and adds a changelog entry.

## Best Possible Solution

Centralize lazy ACP initialization after policy and non-empty-turn checks, keep raw model runs embedded, recompute ACP prompt text after re-resolve, include inline-image first turns, and land with focused tests plus redacted real TUI ACP proof.

## Reproduction Assessment

Yes from source inspection: current main returns `null` when ACP metadata is absent, so a first TUI turn for an ACP-configured agent cannot initialize ACP. I did not run a live ACP harness.

## Solution Assessment

No. Lazy initialization is the right direction, but this PR still initializes ACP on raw CLI runs, keeps stale CLI prompt formatting after re-resolve, and misses inline-image-only first turns.

  - body: This condition is not gated by `!isRawModelRun`, while the ACP `runTurn` branch below still is. A raw `modelRun` or `promptMode: "none"` request on an ACP-configured session can start an external ACP runtime and write metadata before continuing down the embedded/raw path.
  - confidence: 0.88

  - confidence: 0.87

  - body: This guard returns before lazy init when there is no prompt text and no inbound channel media, but the normal ACP path also supports inline `params.images`. A first inline-image-only TUI/WebChat turn still falls back to embedded instead of initializing the ACP session.
  - confidence: 0.85
