## Summary

Keep open. Current main still enumerates `default` and the named iMessage account as separate startup candidates, and live GitHub state shows open implementation PRs targeting this exact duplicate-watcher bug, so closing before one lands would be premature.

## What This Changes

Prevent duplicate iMessage watchers and duplicate replies when `channels.imessage.accounts` contains both `default` and a named account that resolve to the same local Messages backend.

## Best Possible Solution

Land or replace one scoped iMessage startup dedupe that suppresses fallback-only `default` watchers for the same effective backend while preserving intentionally distinct `cliPath`/`dbPath` runtimes and adding focused regression coverage.

## Reproduction Assessment

Yes. The reporter supplied a concrete config, process-list evidence with two `imsg rpc` children, duplicate message history, and a working workaround; current source also deterministically traces that config into two startup account tasks.

## Solution Assessment

Yes, the iMessage-owned dedupe direction is the narrow maintainable solution. The fix should apply to startup account selection or an equivalent startup-only gate, not broadly change shared account semantics for other channels.
