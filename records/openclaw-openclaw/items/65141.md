## Summary

Keep open. Current main still preserves both `default` and the named iMessage account as startup candidates, and the issue is already paired with open fix PRs #65194 and #65383, so closing before one fix lands would be premature.

## What This Changes

Report requests iMessage startup to avoid launching duplicate watchers when `accounts.default` and a named account resolve to the same local Messages backend.

## Best Possible Solution

Land one scoped iMessage fix that suppresses or deduplicates fallback-only `default` watchers when a named account already covers the same backend, with regression coverage for both duplicate and distinct-backend cases.

## Reproduction Assessment

Yes. The reporter supplied concrete config, process-list evidence, message history, and a working workaround; current source also provides a deterministic static reproduction path through account enumeration, gateway startup, and `watch.subscribe`.

## Solution Assessment

Yes. A scoped iMessage account-enumeration or backend-dedupe fix is the narrowest maintainable path; changing the shared account helper broadly could alter valid `accounts.default` behavior for other channels.
