## Summary

Keep open. Current main and latest release still enumerate `default` plus the named iMessage account as separate startup account IDs, and the two target-repo PRs for this bug were closed with maintainer comments saying the bug is real but those branches were not the desired startup-only fix.

## What This Changes

The issue requests preventing duplicate iMessage watcher startup when `channels.imessage.accounts` contains both `default` and a named account pointing at the same local Messages backend.

## Best Possible Solution

Land a focused startup-only iMessage watcher dedupe that suppresses fallback-only `default` monitors for the same effective `imsg` backend while preserving intentionally distinct `cliPath`/`dbPath` runtimes and adding regression coverage.

## Reproduction Assessment

Yes. The reporter supplied real process/message evidence and current main deterministically traces the same config into two account startup tasks, although I did not run a live macOS/iMessage gateway during this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is the maintainer-stated startup-only iMessage watcher dedupe; the closed PRs show why filtering an unused helper or global enabled-state semantics is not the best fix.
