## Summary

Keep open. Current main still exposes both a configured `accounts.default` entry and a named iMessage account as startup account IDs, then starts one gateway account task per ID; each iMessage gateway account starts its own monitor and `watch.subscribe`. Open PRs #65194 and #65383 already target this exact duplicate-watcher bug, so maintainers should reconcile those implementation candidates instead of closing this issue early.

## What This Changes

The issue asks OpenClaw to avoid duplicate iMessage replies when `channels.imessage.accounts` contains both `default` and a named account that resolve to the same local Messages backend.

## Best Possible Solution

Land one narrow iMessage-plugin fix, or an equivalent owner-maintained change, that suppresses or deduplicates a fallback-only `default` watcher when a named account already covers the same local backend, while preserving intentionally distinct default backends and adding regression coverage for both cases.

## Reproduction Assessment

Yes. The reporter supplied a concrete config, process list, and message-history proof, and current main gives a deterministic static path: both account IDs are listed, gateway starts each, and each iMessage account subscribes to `watch.subscribe`.

## Solution Assessment

Yes, the best direction is a scoped iMessage account-enumeration/startup fix with tests. The open PRs appear to target the right owner boundary, while a shared core account-list change would be broader than necessary.
