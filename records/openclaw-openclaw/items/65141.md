## Summary

Keep open. Current main still exposes the duplicate iMessage watcher path, and two open PRs already target this exact bug, so closing the issue before one fix lands would be premature.

## What This Changes

The issue asks OpenClaw to avoid duplicate iMessage replies by not starting two watchers for the same local Messages backend when both `channels.imessage.accounts.default` and a named account are configured.

## Best Possible Solution

Land one narrow iMessage-plugin fix, or an equivalent maintainer-owned change, that suppresses or deduplicates a fallback-only `default` watcher when a named account already covers the same local backend, while preserving intentionally distinct default backends and adding regression coverage for both cases.

## Reproduction Assessment

Yes. The reporter supplied concrete runtime evidence, and current source provides a deterministic static reproduction path: the account helper returns both ids, gateway starts each id, and each iMessage account subscribes to `watch.subscribe`.

## Solution Assessment

Yes. The best path is a scoped iMessage account-enumeration/startup fix with tests; a broad shared account-helper change would risk changing valid `accounts.default` behavior for other channels.
