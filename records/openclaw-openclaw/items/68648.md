## Summary

Keep open: the PR has the right central idea for avoiding the 60s final-turn wait, but it drops the existing direct fallback when the accepted gateway run later fails during setup or dispatch. Current gateway/client contracts show `{ expectFinal: false }` resolves on the accepted frame while the server can still emit a terminal error for that same run afterward.

## What This Changes

The PR changes exec approval followup session resumes from final-result waiting to accepted-only waiting and updates focused tests to expect `{ expectFinal: false }`.

## Best Possible Solution

Keep the non-blocking accepted path, but preserve direct fallback when setup or dispatch fails after acceptance without waiting for a full successful agent turn.

## Reproduction Assessment

Yes, at source level. A focused test can mock an accepted `agent` response followed by a terminal run error, and current gateway/client source shows `{ expectFinal: false }` resolves before that terminal error can set `sessionError`.

## Solution Assessment

No. Accepted-only semantics fixes the long-wait symptom, but the patch should add a bounded terminal-error observer or equivalent fallback path before returning success.
