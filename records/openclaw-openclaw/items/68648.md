## Summary

Keep open. The reported freeze is plausible on current main, but the PR’s accepted-only change would skip the existing fallback path when the gateway accepts the resumed run and later reports a terminal dispatch/runtime error; current main also has a newer followup assertion the branch does not update.

## What This Changes

This PR changes exec approval session followups to return after the gateway accepts the resumed agent request and adds regression assertions for accepted-only behavior.

## Best Possible Solution

Revise the followup path so acceptance no longer blocks behind the full 60s final wait while a separate terminal observer, agent.wait path, or equivalent mechanism still triggers sanitized direct fallback when the resumed run fails after acceptance.

## Reproduction Assessment

Yes. A focused unit-level reproduction can mock the gateway call returning an accepted ack while the gateway code still emits a later terminal error; with this patch, sendExecApprovalFollowup returns true before fallback can run. Current main also has a newer expectation that would fail under the branch’s accepted-only behavior unless the PR is rebased and updated.

## Solution Assessment

No. The one-line expectFinal switch fixes the apparent freeze by treating acceptance as success, but that is not the narrowest maintainable fix because it discards the terminal failure signal currently used for fallback delivery.
