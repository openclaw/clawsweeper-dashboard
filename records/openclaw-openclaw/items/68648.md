## Summary

Keep open: the PR targets a real exec followup timeout, but the accepted-only change loses the existing fallback path when the gateway accepts the resumed run and later reports a terminal error.

## What This Changes

This PR changes exec approval session followups to return after the gateway accepts the resumed agent request and adds regression assertions for accepted-only behavior.

## Best Possible Solution

Revise this PR so followup delivery no longer blocks on the full resumed run but still observes terminal failures enough to trigger direct fallback, with targeted regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A focused unit-level reproduction can mock an accepted gateway response followed by a terminal error path; the current gateway and client code show that `expectFinal: false` resolves on the accepted frame before fallback can run.

## Solution Assessment

No. Accepted-only semantics addresses the timeout symptom, but the best fix needs a non-blocking terminal observer, `agent.wait` path, or equivalent bounded mechanism that preserves fallback on post-accept failures.

## Repair Work Prompt

Repair the exec approval followup behavior from https://github.com/openclaw/openclaw/pull/68648. The proposed accepted-only change avoids the 60s `expectFinal: true` wait, but it currently treats the accepted ack as final success and skips direct fallback if the resumed `agent` run later returns a terminal error. Revise `sendExecApprovalFollowup` so it does not block on the full resumed run while still observing post-accept terminal failure enough to call `sendDirectFollowupFallback` or otherwise surface the failure. Prefer accepted ack plus `agent.wait`/a bounded terminal observer using the returned runId if that fits the existing contracts. Preserve the idempotency key, delivery target behavior, direct mode, denied suppression, and exec approval enforcement semantics. Add regression tests for accepted-then-terminal-error fallback, accepted pending/no 60s final wait, and existing no-session/direct fallback behavior. Add a single-line `CHANGELOG.md` fix entry if the resulting change remains user-facing.
