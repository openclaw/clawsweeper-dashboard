## Summary

Keep open: the PR targets a real exec-followup timeout symptom, but the patch returns success immediately on the accepted ack and skips the existing direct fallback if the gateway later reports a terminal setup or dispatch failure for that same run.

## What This Changes

The PR switches exec approval followup session resumes to accepted-only gateway waiting and updates focused tests to expect `{ expectFinal: false }`.

## Best Possible Solution

Keep the non-blocking accepted path, but capture the accepted run id and use a bounded terminal observer so post-accept setup or dispatch failures still route through the existing direct fallback.

## Reproduction Assessment

Yes, at source level. A focused test can mock an accepted `agent` response followed by a terminal run error, and the current client/gateway source shows `{ expectFinal: false }` resolves before that terminal error can set `sessionError`.

## Solution Assessment

No. Accepted-only waiting fixes the long-wait symptom, but returning success immediately is not the best fix because it loses the existing fallback for terminal post-accept failures.
