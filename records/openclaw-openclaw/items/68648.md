## Summary

Keep this PR open: it targets a real async exec followup stall, but the current diff would treat the accepted ack as full success and skip the existing direct fallback if the resumed run later fails during setup or dispatch.

## What This Changes

The PR changes exec approval followup session resumes from `expectFinal: true` to `expectFinal: false` and updates focused tests to assert accepted-only behavior.

## Best Possible Solution

Keep the accepted-only direction, but capture the accepted run id and use a short bounded observer so post-accept terminal failures still trigger the existing direct fallback.

## Reproduction Assessment

Yes, at source level. Current main still waits with `expectFinal: true`, and the gateway client/server contracts show that `expectFinal: false` resolves on the accepted ack before deferred setup or dispatch can fail.

## Solution Assessment

No. Accepted-only waiting is the right direction for the freeze, but the current PR is incomplete because it drops fallback behavior for terminal failures emitted after the accepted ack.
