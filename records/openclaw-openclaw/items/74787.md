## Summary

Keep open. Current main still lacks acpx PID-level cleanup for orphaned Codex ACP/acpx process trees, and the PR is a valid acpx-local direction, but the current head still has one blocking ownership-root bug plus two narrow cleanup/noise defects before merge.

## What This Changes

The PR adds acpx-owned process-tree cleanup for generated Codex/Claude ACP wrappers and plugin-runtime-deps ACP processes on startup/cancel/close, hardens generated wrappers, and updates acpx tests, docs, and the changelog.

## Best Possible Solution

Keep the acpx-local reaper, but repair ownership-root selection, recorded-PID fallback, and unsupported-scan handling before landing; keep the cleanup limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands.

## Reproduction Assessment

Yes. Current main has no PID-level acpx cleanup in cancel/close and still documents manual stale-process cleanup; the PR discussion adds live v2026.4.29 evidence of PPID=1 `codex-acp` accumulation and recovery after manual SIGTERM.

## Solution Assessment

No, not as-is. The acpx plugin is the right boundary and the ownership-checked reaper is maintainable, but the current patch needs the three narrow repairs before it fully solves the issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so the PPID=1 root this cleanup is meant to clear survives.
  - confidence: 0.94
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: If `listProcesses` throws, the cleanup context has an empty process list. `cleanupOpenClawOwnedAcpxProcessTree` can still prove ownership from `rootCommand`, but it collects no root process and sends no signal, so close/cancel do no PID fallback on the stored process.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix process scanner and logs any thrown error as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
