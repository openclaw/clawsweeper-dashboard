## Summary

Keep open. Current main still has the ACP/acpx orphan cleanup gap, and the live installed-runtime evidence supports the PR direction, but the PR head is not merge-ready because its startup reaper can miss generated wrappers with custom state dirs and its best-effort cleanup gets noisy or incomplete in expected fallback cases.

## What This Changes

The PR adds an acpx-owned process reaper, wires cleanup into acpx runtime cancel/close and service startup, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx plugin fix that keeps ownership checks limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands, uses the actual generated wrapper root for startup reaping, preserves a direct recorded-PID fallback when enumeration fails, and keeps unsupported startup scans quiet.

## Reproduction Assessment

Yes. Current main source and docs reproduce the missing PID-level cleanup path, and the PR discussion adds installed v2026.4.29 evidence of PPID=1 codex-acp accumulation; the remaining PR defects are reproducible by source inspection of custom stateDir startup filtering, process-list failure handling, and unsupported ps behavior.

## Solution Assessment

No, not as-is. The acpx-local reaper is the right maintainable boundary, but the branch should repair the wrapper-root mismatch, direct recorded-PID fallback, and unsupported-scan warning behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx stateDir, stale generated wrappers under the service state directory are skipped, so startup reaping misses the process tree this PR is meant to clear.
  - confidence: 0.94
- **[P3] Keep a recorded PID cleanup fallback when listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: `resolveCleanupContextFromRecord` turns a process-list failure into `[]`, and cleanup then has no root process to signal. The stored PID and `rootCommand` can still prove an OpenClaw-owned root, so cancel/close cleanup is lost when process enumeration is unavailable or incomplete.
  - confidence: 0.88
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.81
