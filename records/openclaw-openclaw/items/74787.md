## Summary

Keep open. Current main still has the acpx orphan-process gap and the PR is aimed at the right plugin-owned boundary, but the latest head still misses custom-state startup reaping and recorded-PID fallback when process listing fails.

## What This Changes

The PR adds an acpx process reaper, cleans recorded harness process trees on cancel/close and startup, hardens generated ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land an acpx-local reaper after passing the generated-wrapper root into startup cleanup, adding a real stored-PID fallback when process listing fails, and keeping ownership checks limited to OpenClaw-generated wrappers and plugin-runtime-deps commands.

## Reproduction Assessment

Yes. Current main still documents manual stale acpx process cleanup, and the PR discussion provides installed v2026.4.29 evidence of 502 PPID=1 OpenClaw-owned codex-acp orphans that were cleared by SIGTERM.

## Solution Assessment

No, not as-is. The plugin-owned cleanup design is the narrow maintainable direction, but the current patch needs the state-root and recorded-PID fallback fixes before it reliably solves the reported process leak.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass the wrapper state root to startup reaping:** `extensions/acpx/src/service.ts:219`
  - body: Startup writes generated Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but the new reaper filters wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state dir are skipped, so the PPID=1 orphan this cleanup targets can survive startup.
  - confidence: 0.94
- **[P2] Signal the stored PID when process listing fails:** `extensions/acpx/src/runtime.ts:528`
  - body: `resolveCleanupContextFromRecord` turns process-list failures into an empty list, and this line passes that empty list through to cleanup. The reaper can still prove ownership from `rootCommand`, but it collects an empty tree and never signals the recorded PID, so close/cancel keep failing on hosts where `ps` is unavailable or errors.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:227-229`
  - body: `listUnixProcesses` always shells out to Unix `ps`, and startup logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate the scan or treat unsupported listing as an empty no-op.
  - confidence: 0.82
