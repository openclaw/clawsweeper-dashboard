## Summary

Keep open. Current main still lacks the PID-level acpx orphan cleanup this PR is trying to add, and the installed-release live evidence in the discussion supports the direction, but the latest PR head still has narrow correctness defects that can make startup or cancel/close cleanup miss the exact OpenClaw-owned process trees it is meant to reap.

## What This Changes

This PR adds an acpx-local process reaper, hardens generated Codex/Claude ACP wrappers, wires runtime cancel/close and startup orphan cleanup, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx plugin fix that preserves the OpenClaw-owned process boundary, passes the generated wrapper state root to startup reaping, keeps a recorded-PID fallback when process listing fails, and treats unsupported startup scans as quiet best-effort behavior.

## Reproduction Assessment

Yes. Current main source and docs reproduce the missing PID-level cleanup path, and the PR discussion adds installed v2026.4.29 live evidence of PPID=1 codex-acp accumulation; the remaining PR defects are reproducible by inspecting the latest PR head lines cited above.

## Solution Assessment

No, not as-is. The acpx-local reaper is the right maintainable boundary, but this branch should repair the wrapper-root mismatch, the list-failure fallback, and unsupported-scan warning behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx stateDir, stale generated wrappers under the service state directory are skipped, so startup reaping misses the process tree this PR is meant to clear.
  - confidence: 0.94
- **[P3] Keep a recorded PID cleanup fallback when listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: `resolveCleanupContextFromRecord` turns a process-list failure into `[]`, and cleanup then sees no recorded root process to signal. The stored PID and `rootCommand` can still prove an OpenClaw-owned root, so cancel/close cleanup is lost when process enumeration is unavailable or incomplete.
  - confidence: 0.88
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.81
