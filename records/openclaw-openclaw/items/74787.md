## Summary

Keep open. The PR is pointed at a real acpx orphan-process leak, but the current head still has narrow correctness blockers around startup ownership matching, recorded-PID fallback, and unsupported process scans.

## What This Changes

The branch adds an acpx-owned process reaper, wires runtime cancel/close and startup cleanup, hardens generated ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land the acpx-local reaper after fixing startup root selection, direct recorded-PID fallback, and unsupported-scan behavior while keeping cleanup limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands.

## Reproduction Assessment

Yes. Current main lacks PID-level acpx cleanup and still documents manual stale-process cleanup, while the PR discussion provides installed v2026.4.29 evidence of PPID=1 codex-acp accumulation and recovery after manual SIGTERM cleanup.

## Solution Assessment

No, not as-is. The acpx plugin is the right boundary, but the latest head needs the three narrow repairs before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so the PPID=1 root this cleanup is meant to clear survives.
  - confidence: 0.94
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: If `listProcesses` throws, `context.processes` becomes empty. `cleanupOpenClawOwnedAcpxProcessTree` can still prove ownership from `rootCommand`, but it collects no root process and sends no signal, so close/cancel do no PID fallback on the stored process.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
