## Summary

Keep open. Current main still has the acpx orphan-process cleanup gap, and the latest head is a useful acpx-local direction, but it still has three narrow cleanup correctness blockers before merge.

## What This Changes

The branch adds an acpx-owned process reaper, wires cancel/close and startup cleanup, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land the acpx-local reaper after repairing the startup ownership root, direct recorded-PID fallback, and unsupported-scan behavior, while keeping the cleanup limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters ownership with `pluginConfig.stateDir`. When acpx is configured with a custom stateDir, stale generated wrappers under the service state directory are skipped, so startup misses the orphan tree this PR is meant to clear.
  - confidence: 0.94
- **[P3] Add a direct PID fallback when listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: If `listProcesses` throws, `context.processes` is empty and `cleanupOpenClawOwnedAcpxProcessTree` collects no root process, so no signal is sent. The stored PID plus `rootCommand` can still prove an OpenClaw-owned root, so cleanup should at least signal that recorded PID.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
