## Summary

Keep open. The PR addresses a real current-main ACP process leak, but the latest head still misses generated-wrapper roots, generated Claude wrapper ownership, and recorded-PID fallback cleanup when process listing fails.

## What This Changes

The PR adds an acpx-local process reaper, cleans recorded harness process trees on cancel/close, hardens generated ACP wrappers, and updates tests, docs, and changelog.

## Best Possible Solution

Land an acpx-local, ownership-limited reaper after fixing wrapper-root filtering, generated Claude wrapper recognition, recorded-PID fallback cleanup, and quiet unsupported scans.

## Reproduction Assessment

Yes. Current main lacks PID-level cleanup on acpx cancel/close and still documents manual stale-process cleanup, while the PR discussion provides live PPID=1 orphan evidence; the PR-specific blockers reproduce by reading the latest head diff.

## Solution Assessment

No, not as-is. The acpx-local cleanup boundary is the right direction, but the current patch needs focused repairs before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Startup writes generated Codex/Claude wrappers under `ctx.stateDir/acpx`, but this call filters wrapper ownership with `pluginConfig.stateDir`. When those paths differ, stale generated wrappers are skipped by the startup cleanup this PR is meant to provide.
  - confidence: 0.94
- **[P2] Recognize generated Claude wrappers in ownership checks:** `extensions/acpx/src/process-reaper.ts:57-58`
  - body: OpenClaw also generates `claude-agent-acp-wrapper.mjs`, and this PR hardens the shared wrapper script, but the reaper only accepts `codex-acp-wrapper.mjs`. Recorded close/cancel cleanup and startup reaping therefore skip OpenClaw-generated Claude wrapper trees.
  - confidence: 0.91
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: When process listing fails, cleanup receives an empty process list. The cleanup path can prove ownership from `rootCommand`, but with no root entry it collects an empty tree and sends no signal to the stored PID.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:230-232`
  - body: Startup always reaches the Unix `ps` scanner through the default reaper path and logs failures as warnings. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported scans or treat them as an empty list.
  - confidence: 0.82
