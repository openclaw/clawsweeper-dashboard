## Summary

Keep open. The PR targets a valid acpx orphan-process leak with live v2026.4.29 evidence, but the latest head still has narrow correctness gaps in startup root selection, direct PID fallback, and unsupported startup scans.

## What This Changes

The PR adds an acpx-owned reaper for OpenClaw-generated ACP wrappers and plugin-runtime-deps processes, wires cleanup into startup/cancel/close, hardens generated wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land the acpx-local reaper after fixing the three blockers, keeping ownership checks limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands while leaving broader parent-reset cleanup tracked by #68916.

## Reproduction Assessment

Yes. Current main lacks PID-level AcpxRuntime close/cancel cleanup and still documents manual stale-process cleanup, and the PR discussion provides live v2026.4.29 evidence of PPID=1 codex-acp accumulation cleared by SIGTERM.

## Solution Assessment

No, not as-is. The acpx plugin is the right boundary, but startup root selection, recorded-PID fallback, and unsupported-scan handling need repair before this is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so the PPID=1 root this cleanup is meant to clear survives.
  - confidence: 0.94
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: If `listProcesses` throws, `context.processes` becomes empty. `cleanupOpenClawOwnedAcpxProcessTree` can still prove ownership from `rootCommand`, but it collects no root process and sends no signal, so close/cancel do no PID fallback on the stored process.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
