## Summary

Keep open. Current main still lacks PID-level acpx cleanup and still documents manual stale-process recovery, and the PR addresses a real orphan-process problem backed by the related issue and live v2026.4.29 evidence. The latest PR head is still not merge-ready because startup reaping filters the wrong generated-wrapper root, close/cancel cleanup drops the recorded-root kill path when process listing fails, and unsupported startup scans can warn on every startup.

## What This Changes

The PR adds an acpx process reaper, startup and cancel/close cleanup for OpenClaw-owned acpx/Codex ACP process trees, generated Codex/Claude wrapper signal hardening, tests, docs, and a changelog entry.

## Best Possible Solution

Land the acpx-local, ownership-limited cleanup after it uses the generated wrapper root for startup reaping, preserves a direct recorded-PID fallback when process enumeration is unavailable, keeps unsupported startup scans quiet, and retains focused tests plus the changed gate.

## Reproduction Assessment

Yes. Current main source and docs show no PID-level cancel/close cleanup and manual stale-process recovery, while #68916 and the PR discussion provide concrete orphan-process reproduction and live runtime evidence. The remaining PR blockers reproduce from source with custom acpx stateDir, process-list failure, and unsupported ps scanner paths.

## Solution Assessment

No, not as-is. The acpx plugin is the right boundary and the ownership checks are the right shape, but this PR needs the wrapper-root, recorded-root fallback, and unsupported-scan fixes before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx stateDir, stale generated wrappers under the service state directory are skipped; pass `ctx.stateDir` or an explicit generated-wrapper root instead.
  - confidence: 0.94
- **[P3] Kill the recorded root when process listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: When process listing throws, `resolveCleanupContextFromRecord` records an empty process list and cleanup receives that list. The helper can validate `rootCommand`, but `collectProcessTree` finds no root and sends no signal to the recorded PID, so close/cancel cleanup is lost on hosts where `ps` is unavailable or incomplete.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup invokes the Unix `ps` scanner unconditionally and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.82
