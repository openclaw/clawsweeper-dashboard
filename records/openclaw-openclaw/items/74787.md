## Summary

Keep open. Current main still lacks the acpx process-tree cleanup this PR targets, and the latest PR head is a valid acpx-boundary implementation direction but still has concrete correctness issues that should be repaired before merge.

## What This Changes

The PR adds an acpx process reaper, wires cleanup into startup/cancel/close, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land the acpx-local reaper after fixing startup ownership-root selection, direct recorded-PID fallback when process listing is unavailable, and quiet no-op behavior for unsupported startup scans, while keeping cleanup limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands.

## Reproduction Assessment

Yes. Current main has no process-reaper file, cancel/close do not perform PID-level cleanup, and docs still instruct manual stale-process cleanup; the PR discussion adds live v2026.4.29 evidence of PPID=1 codex-acp accumulation and recovery after manual SIGTERM.

## Solution Assessment

No, not as-is. The acpx plugin is the right implementation boundary, but the latest PR head needs the three narrow repairs before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so the PPID=1 roots this cleanup is meant to clear survive.
  - confidence: 0.94
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: If `listProcesses` throws, `context.processes` becomes empty. `cleanupOpenClawOwnedAcpxProcessTree` can still prove ownership from `rootCommand`, but it collects no root process and sends no signal, so close/cancel do no PID fallback on the stored process.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
