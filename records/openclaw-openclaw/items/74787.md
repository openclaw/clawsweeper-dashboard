## Summary

Keep open. The PR targets a real acpx orphan-process leak with live v2026.4.29 evidence, but the latest head still has narrow blockers around acpx test-option typing, wrapper-root filtering, list-failure fallback, and unsupported startup scans.

## What This Changes

The PR adds an acpx-local process reaper, wires it into runtime cancel/close and service startup, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land an acpx-local reaper after separating wrapper and plugin state roots, preserving recorded-PID fallback when process listing fails, making unsupported scans quiet, and keeping cleanup scoped to OpenClaw-generated wrappers/plugin-runtime-deps commands.

## Reproduction Assessment

Yes. Current main only delegates acpx cancel/close and still documents manual stale-process cleanup, and the PR discussion gives installed v2026.4.29 evidence of 502 PPID=1 codex-acp orphans cleared by SIGTERM.

## Solution Assessment

No, not as-is. The acpx plugin boundary is appropriate, but the latest patch needs the file-local repairs before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep BaseAcpxRuntime test options separate:** `extensions/acpx/src/runtime.ts:435`
  - body: `BaseAcpxRuntime` from acpx@0.6.1 only accepts `managerFactory` and `probeRunner` test deps, while `AcpxRuntimeTestOptions` only defines `openclawProcessCleanup`. This assignment has no overlapping properties with `ConstructorParameters<typeof BaseAcpxRuntime>[1]`, so the acpx extension typecheck will reject the branch; extend the upstream test-option type and omit `openclawProcessCleanup` before forwarding.
  - confidence: 0.84
- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so the PPID=1 root this cleanup is meant to clear survives.
  - confidence: 0.94
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: If `listProcesses` throws, `context.processes` becomes empty. `cleanupOpenClawOwnedAcpxProcessTree` can still prove ownership from `rootCommand`, but it collects no root process and sends no signal, so close/cancel do no PID fallback on the stored process.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
