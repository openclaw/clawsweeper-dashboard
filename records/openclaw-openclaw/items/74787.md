## Summary

Keep open. The PR addresses a real acpx orphan-process problem with useful live v2026.4.29 evidence, but the current head still has source-level cleanup gaps that would leave some OpenClaw-owned process trees unreaped or make best-effort startup cleanup noisy on unsupported hosts.

## What This Changes

This PR adds an acpx-local process reaper, startup orphan cleanup, recorded PID cleanup on cancel/close, generated Codex/Claude ACP wrapper hardening, focused tests, docs, and a changelog entry.

## Best Possible Solution

Land a scoped acpx plugin fix that keeps ownership checks limited to OpenClaw-managed acpx/Codex ACP paths, uses the generated wrapper root for startup reaping, preserves direct recorded-PID cleanup when enumeration fails, and treats unsupported startup scans as quiet no-op cleanup.

## Reproduction Assessment

Yes. Current main source and docs show no PID-level acpx cleanup, and the PR discussion provides installed v2026.4.29 evidence of PPID=1 codex-acp accumulation. The remaining PR defects are reproducible by tracing the custom stateDir, empty process-list, and startup warning paths in the PR head.

## Solution Assessment

No, not as-is. The acpx-local, ownership-limited reaper is the right boundary, but this branch should repair the wrapper-root mismatch, recorded-PID fallback, and unsupported-scan behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx stateDir, stale generated wrappers under the service state directory are skipped, so startup reaping misses the OpenClaw-owned process tree this PR is meant to clear.
  - confidence: 0.94
- **[P3] Keep a recorded PID fallback when process listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: When process listing fails, the cleanup context contains the stored PID and command but an empty process snapshot. Passing that empty list into cleanup means `collectProcessTree` finds no recorded root and sends no signal, so cancel/close cleanup is lost exactly when enumeration is unavailable or incomplete.
  - confidence: 0.88
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup invokes the Unix `ps` scanner unconditionally and logs failures as warnings. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.8
