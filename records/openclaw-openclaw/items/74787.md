## What This Changes

The PR adds an acpx-local process reaper, wires stale orphan cleanup into service startup and runtime cancel/close, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Track any remaining parent-reset child-session cleanup under #68916 if it is not covered by the final patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so the PPID=1 root this cleanup is meant to clear survives.
  - confidence: 0.94
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:508-512`
  - body: If `listProcesses` throws, `context.processes` becomes empty. `cleanupOpenClawOwnedAcpxProcessTree` can still prove ownership from `rootCommand`, but it collects no root process and sends no signal, so close/cancel do no PID fallback on the stored process.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or treated as an empty result.
  - confidence: 0.82
