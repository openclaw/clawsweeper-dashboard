## Summary

Keep open. The PR targets a real acpx/Codex ACP orphan-process leak that is still present on current main, but the current head still has one merge-blocking startup cleanup bug and two smaller robustness issues around process-list failures and unsupported startup scans.

## What This Changes

The PR adds an acpx-owned process reaper, wires recorded process-tree cleanup into cancel/close and startup, hardens generated ACP wrapper scripts, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the acpx plugin as the implementation boundary and land the reaper after repairing the generated-wrapper root, direct recorded-PID fallback, and unsupported-scan behavior while keeping ownership checks limited to OpenClaw-generated wrappers and plugin-runtime-deps ACP commands.

## Reproduction Assessment

Yes. Current main lacks PID-level acpx cleanup, the docs still describe manual stale-process cleanup, and the PR discussion includes live v2026.4.29 codex-acp orphan evidence; the remaining PR defects are reproducible by source inspection of the startup filter and fallback paths.

## Solution Assessment

No, not as-is. The acpx-local reaper is the right shape, but this head needs the three file-local repairs before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root into startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup writes the generated ACP wrappers under `ctx.stateDir/acpx`, but this call filters ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped, so startup misses the orphan tree this PR is intended to clear.
  - confidence: 0.94
- **[P3] Fall back to signaling the recorded PID when ps fails:** `extensions/acpx/src/runtime.ts:511-512`
  - body: If `listProcesses` throws, `context.processes` becomes empty and cleanup can verify the owned `rootCommand` but collects no root process, so no signal is sent. The stored PID should still be signaled as a best-effort fallback when the command proves OpenClaw ownership.
  - confidence: 0.9
- **[P3] Treat unsupported startup scans as empty instead of warning:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; unsupported scans should be gated or handled as an empty result.
  - confidence: 0.82
