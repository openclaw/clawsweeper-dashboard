## Summary

Keep open. The PR addresses a real acpx/Codex ACP orphan-process problem that current main still leaves to manual cleanup, but the latest head still has source-level blockers around generated-wrapper ownership, cleanup when process listing fails, and noisy unsupported startup scans.

## What This Changes

This PR adds an acpx-local process reaper and generated wrapper hardening for OpenClaw-owned Codex/Claude ACP process trees, wires startup/cancel/close cleanup, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx plugin fix that keeps ownership checks local to OpenClaw-managed acpx/Codex ACP paths, uses the generated wrapper root for startup reaping, preserves a recorded-PID kill path when process enumeration is unavailable, and covers custom stateDir plus unsupported scan behavior in focused tests.

Yes. The remaining PR blockers are reproducible by reading the custom stateDir and process-list failure paths in the PR head.

## Solution Assessment

No, not as-is. The acpx-local ownership-limited reaper is the right boundary, but the branch should repair the wrapper-root mismatch, list-failure fallback, and unsupported-scan warning behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:221-223`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx stateDir, stale generated wrappers under the service state directory are skipped, so startup reaping misses the OpenClaw-owned process tree this PR is meant to clear.
  - confidence: 0.94
- **[P3] Keep a recorded PID cleanup fallback when listing fails:** `extensions/acpx/src/runtime.ts:527-531`
  - body: `resolveCleanupContextFromRecord` turns a process-list failure into `[]`, and this line passes that empty list through to cleanup. The helper can validate `rootCommand`, but `collectProcessTree` sees no recorded root and sends no signal to the stored PID, so cancel/close cleanup is lost when process enumeration is unavailable or incomplete.
  - confidence: 0.88
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:229-231`
  - body: Startup invokes the Unix `ps` scanner unconditionally and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.8
