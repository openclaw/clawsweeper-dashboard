## Summary

Keep open. The acpx-local direction matches the live orphan-process evidence, but this head still misses generated-wrapper root alignment, generated Claude wrapper ownership, and recorded-PID fallback cleanup cases that can leave the same class of orphans behind.

## What This Changes

This PR adds an acpx-local process reaper, startup orphan cleanup, cancel/close recorded process-tree cleanup, generated ACP wrapper signal handling, tests, docs, and a changelog entry.

## Best Possible Solution

Land an ownership-limited acpx reaper after it uses the generated wrapper root, covers Codex and Claude wrappers, preserves recorded-PID fallback cleanup, and treats unsupported scans as best-effort.

Yes, source-reproducible. Current main cancel/close lacks PID-level cleanup and docs still describe manual stale-process cleanup, while the PR discussion adds live v2026.4.29 PPID=1 codex-acp orphan evidence.

## Solution Assessment

No, not as-is. The acpx-local cleanup boundary is the narrow maintainable fix, but this implementation still misses important ownership and fallback cases before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Pass the generated wrapper root to startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Startup writes generated Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters wrapper ownership with `pluginConfig.stateDir`. When those paths differ, stale generated wrapper roots are skipped and can survive the startup cleanup this PR is meant to provide.
  - confidence: 0.94
- **[P2] Recognize generated Claude wrappers in ownership checks:** `extensions/acpx/src/process-reaper.ts:57-58`
  - body: `prepareAcpxCodexAuthConfig` also writes `claude-agent-acp-wrapper.mjs`, and this PR hardens the shared wrapper script, but the reaper only accepts `codex-acp-wrapper.mjs`. Recorded close/cancel cleanup and startup reaping therefore skip OpenClaw-generated Claude ACP wrapper trees.
  - confidence: 0.91
- **[P3] Signal the recorded PID when process listing fails:** `extensions/acpx/src/runtime.ts:511-512`
  - body: When `listProcesses` throws, cleanup receives an empty process list. The cleanup path can prove ownership from `rootCommand`, but with no root entry it collects an empty tree and sends no signal to the stored PID.
  - confidence: 0.9
- **[P3] Keep unsupported startup scans quiet:** `extensions/acpx/src/service.ts:230-231`
  - body: Startup reaches the Unix `ps` scanner through the default reaper path and logs failures as warnings. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported scans or treat them as an empty process list.
  - confidence: 0.82
