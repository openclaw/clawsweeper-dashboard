## Solution Assessment

Full review comments:

- **[P1] Import HTTP helpers from provider-http:** `extensions/xiaomi/media-understanding-provider.ts:7-9`
  - body: This new provider imports `assertOkOrThrowHttpError`, `normalizeBaseUrl`, and `postJsonRequest` from `openclaw/plugin-sdk/media-understanding`, but current main exports those helpers from `openclaw/plugin-sdk/provider-http`.   - confidence: 0.94
- **[P1] Forward request policy to Xiaomi video calls:** `extensions/xiaomi/media-understanding-provider.ts:90-97`
  - body: The video request posts user video bytes with provider credentials but never resolves or forwards `params.request`, `allowPrivateNetwork`, or `dispatcherPolicy`. Existing video providers use `resolveProviderHttpRequestConfig` so configured proxy and private-network policy are honored for provider egress.
  - confidence: 0.92
- **[P1] Use MiMo V2 Omni instead of the active text model:** `extensions/xiaomi/media-understanding-provider.ts:65`
  - body: `describeXiaomiVideo` prefers `params.model`, and the video runner passes the active Xiaomi reply model through. A normal Xiaomi install uses `mimo-v2-flash`, which the catalog declares as text-only, so default video understanding can call the wrong model instead of MiMo V2 Omni.
  - confidence: 0.9
- **[P2] Declare Xiaomi media metadata in the manifest:** `extensions/xiaomi/index.ts:61`
  - body: Runtime registration alone does not make the provider available to manifest-backed media discovery or pre-runtime defaults. Add `contracts.mediaUnderstandingProviders` plus `mediaUnderstandingProviderMetadata` with MiMo V2 Omni defaults and priority so video selection works before plugin runtime loading.
  - confidence: 0.86
