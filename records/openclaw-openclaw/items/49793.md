## Summary

Keep open. Current main has Xiaomi model/catalog groundwork and generic image-capable provider fallback, but it still does not provide Xiaomi video media understanding; this PR is useful source work but is not mergeable as-is because it is stale against the current Xiaomi plugin shape and misses current SDK/provider HTTP policy and video model-selection requirements.

## What This Changes

The PR adds a Xiaomi media-understanding provider for image and video descriptions and registers it from the Xiaomi plugin entrypoint.

## Best Possible Solution

Land an updated Xiaomi plugin change that adds the missing video media-understanding path through the current single-provider plugin entry, shared OpenAI-compatible video helpers, provider-http request policy resolution, video-safe Xiaomi model selection, manifest metadata/defaults, focused Xiaomi media tests, and matching docs while preserving the current generic image-capable model fallback where it is sufficient.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The merge blockers are high-confidence by static inspection: current main lacks Xiaomi video registration, current SDK exports place HTTP helpers under provider-http, and the runner passes the active Xiaomi model into video providers.

## Solution Assessment

No, not as submitted. The direction is reasonable, but the implementation should be updated to current SDK/provider contracts, request-policy forwarding, manifest-backed discovery/defaults, and Xiaomi video-safe model selection before merge.

Overall confidence: 0.91

Full review comments:

- **[P1] Import provider HTTP helpers from provider-http:** `extensions/xiaomi/media-understanding-provider.ts:7-9`
  - body: On current main, `openclaw/plugin-sdk/media-understanding` exports media provider types and image/video body helpers, but not `assertOkOrThrowHttpError`, `normalizeBaseUrl`, or `postJsonRequest`. This added file will fail typechecking until those imports come from `openclaw/plugin-sdk/provider-http`.
  - confidence: 0.94
- **[P1] Forward request policy to video requests:** `extensions/xiaomi/media-understanding-provider.ts:90-97`
  - body: This call posts user video bytes with Xiaomi credentials but omits the resolved request policy, so configured proxy/private-network dispatcher behavior will not be applied. Match the existing Moonshot/Qwen video providers by using `resolveProviderHttpRequestConfig` and passing `allowPrivateNetwork` and `dispatcherPolicy` into `postJsonRequest`.
  - confidence: 0.91
- **[P1] Use the Xiaomi video model by default:** `extensions/xiaomi/media-understanding-provider.ts:63`
  - body: For automatic video understanding, the runner passes the active agent model as `params.model`; a normal Xiaomi install uses text-only `mimo-v2-flash`. Preferring that value here makes the default video path call the wrong model instead of MiMo V2 Omni unless the user manually configures a media video override.
  - confidence: 0.88
- **[P2] Declare Xiaomi media metadata in the manifest:** `extensions/xiaomi/media-understanding-provider.ts:117-122`
  - body: Runtime registration alone leaves manifest-backed media discovery and default metadata unaware of Xiaomi media support before the plugin runtime is loaded. Add the Xiaomi media-understanding contract and metadata/defaults to the manifest so auto fallback and capability discovery can see it.
  - confidence: 0.84
