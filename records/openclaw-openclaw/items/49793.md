## Summary

Keep open: the Xiaomi media provider is useful and not already on current main, but this patch is not ready to merge. Source review found that the runtime registration is not discoverable through the current manifest-owned media capability path, the new file imports HTTP helpers from the wrong SDK subpath on current main, video requests would use Xiaomi's text-only default model, and the PR still lacks real Xiaomi behavior proof.

## What This Changes

The PR adds a Xiaomi media-understanding provider and registers it with the Xiaomi plugin so image and video attachments can be described via MiMo V2 Omni.

## Best Possible Solution

Rework this as a current-main Xiaomi media provider following the Moonshot/Qwen pattern: manifest contract and metadata, current provider HTTP helpers, MiMo V2 Omni defaults for image/video, focused provider tests, and redacted real Xiaomi image/video proof.

## Reproduction Assessment

Yes for the review-blocking code issues: current source shows the manifest discovery path and video active-model path that make this patch fail or misroute after rebase. I did not reproduce with a live Xiaomi API request.

## Solution Assessment

No. Registering only the runtime provider is not enough for current media capability discovery, and the provider should use the current SDK HTTP/request helpers plus MiMo V2 Omni video selection before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Declare Xiaomi as a media provider in the manifest:** `extensions/xiaomi/index.ts:61`
  - body: The media registry cold-loads providers from `contracts.mediaUnderstandingProviders` and manifest metadata, but Xiaomi's manifest still only declares `speechProviders`. This runtime registration can be missed by normal media-understanding discovery, so add the manifest contract plus `mediaUnderstandingProviderMetadata` for Xiaomi's image/video capabilities and defaults.
  - confidence: 0.9
- **[P1] Import HTTP helpers from the provider HTTP subpath:** `extensions/xiaomi/media-understanding-provider.ts:1-9`
  - body: On current main, `openclaw/plugin-sdk/media-understanding` does not export `assertOkOrThrowHttpError`, `normalizeBaseUrl`, or `postJsonRequest`; those live under `openclaw/plugin-sdk/provider-http`. As written, the new provider will fail to typecheck after rebase before it can run.
  - confidence: 0.92
- **[P2] Route video descriptions to MiMo V2 Omni:** `extensions/xiaomi/media-understanding-provider.ts:57`
  - body: Automatic video understanding passes the active Xiaomi agent model into `describeVideo`, and the default Xiaomi model is the text-only `mimo-v2-flash`. Because `resolveModel` accepts that value, a default Xiaomi install would send video descriptions to the wrong model instead of MiMo V2 Omni.
  - confidence: 0.86
