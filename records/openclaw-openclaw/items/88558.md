## Summary

Keep open. The PR is a focused Gateway compatibility change: current main still rejects the two Chat Completions `tool_choice` forms, while the branch adds validation, response-path enforcement, docs, tests, and credible live API proof. I found no blocking line-level defect, but maintainers should decide whether this Chat Completions-only scope is acceptable while `/v1/responses` remains prompt-only for the same forms.

## What This Changes

This PR updates `/v1/chat/completions` to accept and enforce `tool_choice: "required"` and function-pinned requests, with Gateway docs and e2e coverage.

## Best Possible Solution

Land the Chat Completions boundary fix if maintainers accept the endpoint-specific scope, and keep broader agent-loop/provider-forced tool-choice work tracked in https://github.com/openclaw/openclaw/issues/45049 or a narrower follow-up.
