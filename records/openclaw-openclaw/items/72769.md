## Summary

Keep open. Current main does not implement the approval-decision audit log, and the PR is still useful in direction, but the branch is not merge-ready because it introduces a persisted approval-history surface with approval-scope, secret-redaction, blocking I/O, protocol-artifact, and changelog gaps.

## What This Changes

The PR adds an append-only exec approval decision JSONL audit log, registers an `exec.approval.auditLog` gateway RPC with filters, and adds custom persisted-argument redaction.

## Best Possible Solution

Keep the feature direction, but revise this branch so audit history is guarded by the same approval/admin boundary as live approval data, persisted argv uses the existing shared tool/config audit redaction coverage, reads are nonblocking and bounded, protocol docs and generated clients are regenerated, and a user-facing changelog entry is added before merge.

## Reproduction Assessment

Yes. A focused branch check can show `exec.approval.auditLog` is authorized by `operator.read`, and unit tests against the new redactor can show option-style, lowercase, bearer, and raw token argv forms persist unmasked.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive gateway feature, but this implementation is not the narrowest maintainable solution until it reuses the existing approval scope and redaction contracts and keeps the public protocol artifacts in sync.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:110`
  - body: This registers `exec.approval.auditLog` under `operator.read`, while current live approval methods and approval broadcasts require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only operator tokens would gain access to approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse the existing argv redaction path:** `src/infra/exec-approval-audit.ts:40-43`
  - body: The array redactor only masks uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped values can still be written to `approvals.jsonl`; reuse the existing audit/tool redaction helper or equivalent coverage before persisting argv.
  - confidence: 0.88
- **[P2] Avoid blocking the gateway on audit reads:** `src/infra/exec-approval-audit.ts:90`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived gateway can accumulate a large file, and one `exec.approval.auditLog` request would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.83
- **[P2] Register audit params in generated protocol artifacts:** `src/gateway/protocol/schema/exec-approvals.ts:153-161`
  - body: Adding `ExecApprovalAuditLogParamsSchema` in this file is not enough for a public gateway RPC. The new params/result surface is missing from the protocol schema registry, generated TS schema types, generated Swift models, and protocol docs, so generated clients will not learn the new contract.
  - confidence: 0.79
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/exec-approval.ts:399-418`
  - body: This adds a user-facing gateway audit log and RPC, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to include a changelog entry before merge.
  - confidence: 0.82
