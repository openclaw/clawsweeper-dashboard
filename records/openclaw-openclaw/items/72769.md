## Summary

Keep open. The feature direction is useful, but the current PR still has blocking security and correctness issues around approval-history authorization, persisted argv redaction, unbounded synchronous audit reads, protocol artifact drift, and the required changelog entry.

## What This Changes

The PR adds a JSONL exec approval decision audit sink, writes audit entries when approvals are resolved, registers a new `exec.approval.auditLog` RPC with filters, and exports new protocol validator/schema types.

## Best Possible Solution

Keep the approval-decision audit feature direction, but revise this branch so audit reads use the approval/admin authorization boundary, persisted argv goes through the shared redaction contract, reads are nonblocking and bounded, public protocol/docs/generated clients are synchronized, and the release notes document the user-facing feature.

## Reproduction Assessment

Yes. A focused check can show `exec.approval.auditLog` would authorize under `operator.read`, and unit tests for the new redactor can show `--token secret`, `--password=secret`, lowercase assignments, bearer headers, and raw token-shaped argv values persist unmasked.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive Gateway feature, but this implementation is not the best path until it reuses the existing approval scope and redaction contracts and keeps protocol artifacts in sync.

Overall confidence: 0.9

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:110`
  - body: This registers `exec.approval.auditLog` under `operator.read`, while current live approval methods and approval broadcasts require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only operator tokens would gain access to approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse the existing argv redaction path:** `src/infra/exec-approval-audit.ts:34-43`
  - body: The array redactor only masks uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped values can still be written to `approvals.jsonl`; reuse the existing audit/tool redaction helper or equivalent coverage before persisting argv.
  - confidence: 0.9
- **[P2] Avoid blocking the gateway on audit reads:** `src/infra/exec-approval-audit.ts:84`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived Gateway can accumulate a large file, and one `exec.approval.auditLog` request would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.85
- **[P2] Register audit params in generated protocol artifacts:** `src/gateway/protocol/schema/exec-approvals.ts:153-161`
  - body: Adding `ExecApprovalAuditLogParamsSchema` here is not enough for a public Gateway RPC. The new params/result surface is missing from the protocol schema registry, generated TS schema output, generated Swift models, and protocol docs, so generated clients will not learn the new contract.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/exec-approval.ts:423-426`
  - body: This adds a user-facing Gateway audit log and RPC, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to include a changelog entry before merge.
  - confidence: 0.82
