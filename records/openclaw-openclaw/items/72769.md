## Summary

Keep this PR open: the feature direction is reasonable, but the current diff introduces approval-audit access-control, secret-redaction, and availability regressions that should block merge.

## What This Changes

The PR adds append-only JSONL persistence for exec approval decisions plus an `exec.approval.auditLog` RPC, scope/list/protocol exports, and a new audit helper.

## Best Possible Solution

Keep the approval-audit feature direction, but revise the branch so the RPC is approval-scoped, persisted argv uses the shared redaction contract, reads are async and bounded, and protocol/docs/changelog artifacts are synchronized.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough to reproduce the blockers: `operator.read` authorizes the new audit RPC, the array redactor misses common secret argv forms, and the read path uses `fs.readFileSync`; no write-producing tests were run because this review is read-only.

## Solution Assessment

No. Persisting approval decisions is a maintainable additive feature, but this implementation is not the best fix until it follows the existing approval scope and shared redaction behavior and lands through the full Gateway protocol update path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:109`
  - body: This registers `exec.approval.auditLog` under `operator.read`, while live exec approval methods require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only or write-only operator tokens would gain access to approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse shared argv redaction before persisting:** `src/infra/exec-approval-audit.ts:35-43`
  - body: The array redactor only masks uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped values can still be written to `approvals.jsonl`; reuse the existing audit/tool redaction helper or equivalent coverage before persisting argv.
  - confidence: 0.88
- **[P2] Avoid blocking the gateway on audit-log reads:** `src/infra/exec-approval-audit.ts:90`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived gateway can accumulate a large file, and one `exec.approval.auditLog` request would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.84
- **[P2] Synchronize the new Gateway protocol surface:** `src/gateway/protocol/schema/exec-approvals.ts:156-169`
  - body: Adding params validation is not enough for a public Gateway RPC. The new method needs the full protocol/docs/generated-client path, including result typing where appropriate, generated client artifacts, protocol docs, and focused validator/client coverage so operator clients learn the contract.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/exec-approval.ts:424`
  - body: This is a user-facing Gateway feature that adds a persisted audit log and query RPC, but the PR does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-visible `feat` changes before merge.
  - confidence: 0.82
