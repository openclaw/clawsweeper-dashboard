## Summary

Keep this PR open. The feature direction is useful, but the current branch adds a security-sensitive persisted approval audit surface with blocking access-control and secret-redaction bugs, plus a synchronous unbounded read path and incomplete public protocol follow-through.

## What This Changes

The PR adds an append-only JSONL audit log for exec approval decisions, registers an `exec.approval.auditLog` gateway RPC with filters, exports a params validator, and persists redacted approval command args.

## Best Possible Solution

Keep the feature direction, but revise the branch so approval history is guarded by the existing approval/admin boundary, persisted argv uses the shared tool/config audit redaction coverage, audit reads are async and bounded, protocol registry/generated clients/docs are synchronized, and the user-facing feature has a changelog entry before merge.

## Reproduction Assessment

Yes. A focused branch check can verify that `exec.approval.auditLog` authorizes with `operator.read`, and unit tests against the new redactor can show `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped argv values persist unmasked.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive gateway feature, but this implementation is not the narrowest maintainable solution until it reuses the existing approval scope and redaction contracts and avoids synchronous full-log reads on the gateway request path.

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:109`
  - body: This registers `exec.approval.auditLog` under `operator.read`, while live exec approval methods require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only operator tokens would gain access to approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse the shared argv redaction path:** `src/infra/exec-approval-audit.ts:36-43`
  - body: The array redactor only masks uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped values can still be written to `approvals.jsonl`; reuse the existing audit/tool redaction helper or equivalent coverage before persisting argv.
  - confidence: 0.88
- **[P2] Avoid blocking the gateway on audit reads:** `src/infra/exec-approval-audit.ts:90`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived gateway can accumulate a large file, and one `exec.approval.auditLog` request would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.83
- **[P2] Register audit params in protocol artifacts:** `src/gateway/protocol/schema/exec-approvals.ts:156-163`
  - body: Adding `ExecApprovalAuditLogParamsSchema` in this file is not enough for a public gateway RPC.   - confidence: 0.79
