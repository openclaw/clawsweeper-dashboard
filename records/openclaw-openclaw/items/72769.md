## Summary

Keep open. The PR is a useful active feature branch, but it is not merge-ready because the new persisted approval history is exposed under the read scope, the custom argv redaction is narrower than existing audit/tool redaction, the query path synchronously reads an unbounded append-only file, and protocol/changelog surfaces are not yet complete.

## What This Changes

The PR adds append-only JSONL persistence for exec approval decisions, an `exec.approval.auditLog` RPC with filters, protocol validator exports, and custom persisted-argument redaction.

## Best Possible Solution

Keep the feature direction, but revise this branch so approval audit history uses the same approval/admin authorization boundary as live approval data, persisted argv goes through the shared audit/tool redaction path, reads are asynchronous and bounded, protocol artifacts/docs are regenerated, and a user-facing changelog entry is added before merge.

## Reproduction Assessment

Yes. Static review against the PR diff shows `exec.approval.auditLog` is authorized by `operator.read`, and focused redaction tests can demonstrate that option-style and token-shaped `commandArgv` entries are persisted unmasked.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive gateway feature, but this implementation is not the safest maintainable version until it reuses existing scope and redaction contracts and keeps the public protocol surface in sync.

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:110`
  - body: This registers `exec.approval.auditLog` under `operator.read`, while live approval methods and approval broadcasts require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only operator tokens would gain approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse shared argv redaction before persisting:** `src/infra/exec-approval-audit.ts:38-40`
  - body: The array redactor only masks uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, Bearer strings, and raw token-shaped values can still be written to `approvals.jsonl`; use the existing audit/tool redaction coverage before persisting argv.
  - confidence: 0.88
- **[P2] Avoid blocking audit-log reads:** `src/infra/exec-approval-audit.ts:97`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived gateway can accumulate a large file, and one audit-log RPC would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.83
- **[P2] Register audit params in generated protocol artifacts:** `src/gateway/protocol/schema/exec-approvals.ts:153-161`
  - body: Adding `ExecApprovalAuditLogParamsSchema` here is not enough for a public gateway RPC. The new params/result surface is missing from the protocol schema registry, generated schema types, generated client artifacts, and protocol docs, so generated clients will not learn the new contract.
  - confidence: 0.79
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/exec-approval.ts:399-418`
