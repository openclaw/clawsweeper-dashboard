## Summary

Keep this PR open. Current main does not already implement approval-decision audit logging, and the proposed branch is still not mergeable because it introduces a new persisted approval history surface with access-control, argv redaction, availability, protocol, and changelog gaps that need branch revision before maintainer acceptance.

## What This Changes

The PR adds an append-only exec approval decision JSONL audit file, writes approval resolutions to it, exposes an `exec.approval.auditLog` Gateway RPC with filters, and exports a new params schema/validator.

## Best Possible Solution

Keep the feature direction, but revise the branch so approval history is guarded by the existing approval/admin boundary, persisted argv uses the shared audit/tool redaction coverage, audit reads are async and bounded, protocol registry/generated clients/docs/tests are synchronized, and the user-facing feature has a changelog entry before merge.

## Reproduction Assessment

Yes. Static branch inspection verifies the audit RPC is registered under `operator.read`, the redactor leaves common argv secret forms unmasked, and the query helper synchronously reads the whole JSONL file; focused tests can cover each path without broad CI.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive Gateway feature, but this implementation is not the narrowest maintainable solution until it reuses the existing approval scope and redaction contracts and avoids synchronous full-log reads on the request path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:109`
  - body: This registers `exec.approval.auditLog` under `operator.read`, while live exec approval methods and approval broadcasts require `operator.approvals`. The returned history includes command text, args, agent/session IDs, and resolver identity, so read-only operator tokens would gain approval data they cannot see live.
  - confidence: 0.91
- **[P1] Reuse the shared argv redaction path:** `src/infra/exec-approval-audit.ts:36-43`
  - body: The array redactor only masks uppercase env-var style `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped values can still be written to `approvals.jsonl`; reuse the existing audit/tool redaction helper or equivalent coverage before persisting argv.
  - confidence: 0.88
- **[P2] Avoid blocking the Gateway on audit reads:** `src/infra/exec-approval-audit.ts:90`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived Gateway can accumulate a large file, and one `exec.approval.auditLog` request would block the Node event loop; use an async bounded or streaming read path instead.
  - confidence: 0.83
- **[P2] Register the audit schema in protocol artifacts:** `src/gateway/protocol/schema/exec-approvals.ts:156-163`
  - body: Adding `ExecApprovalAuditLogParamsSchema` here is not enough for a public Gateway RPC. The new params/result surface is missing from the protocol schema registry, generated schema types, generated Swift models, and protocol docs, so generated clients will not learn the new contract.
  - confidence: 0.79
- **[P3] Add the required changelog entry:** `src/gateway/server-methods/exec-approval.ts:424`
  - body: This adds a user-facing Gateway audit log and RPC, but the PR does not update `CHANGELOG.md`. Repo policy requires user-facing `feat` changes to include a changelog entry before merge.
  - confidence: 0.82
