## Summary

Keep open: current main does not implement this approval audit log, and the PR is still a plausible feature direction, but the branch is not mergeable because the new audit RPC broadens approval data access, argv redaction is incomplete, log reads are synchronous and unbounded, protocol artifacts are incomplete, and external real behavior proof is missing. The related open gateway tool-call audit PR is adjacent, not a merged replacement.

## What This Changes

The PR adds an append-only JSONL audit log for exec approval decisions, writes resolve events into it, and exposes an `exec.approval.auditLog` gateway query method.

## Best Possible Solution

Revise the branch into a focused approval-scoped audit feature that reuses shared redaction, performs bounded async reads, updates protocol/docs/generated artifacts, coordinates with the related gateway audit work, and includes redacted real behavior proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows `exec.approval.auditLog` in the read scope, incomplete argv redaction, and synchronous full-file reads. I did not run tests because this was a read-only review.

## Solution Assessment

No. Persisting approval decisions is a reasonable additive gateway feature, but this implementation is not the narrowest maintainable solution until access control, redaction, log-reading, and protocol surfaces are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate audit-log reads with approval scope:** `src/gateway/method-scopes.ts:109`
  - body: This registers `exec.approval.auditLog` under the read scope, but the response contains command text, args, agent/session IDs, and resolver identity. Live exec approval methods require `operator.approvals`, so read-only or write-only operator tokens would gain historical approval data they cannot access live.
  - confidence: 0.91
- **[P1] Reuse shared argv redaction before persistence:** `src/infra/exec-approval-audit.ts:36-43`
  - body: The array redactor only masks uppercase `KEY=value` entries. Common argv forms such as `--token secret`, `--password=secret`, lowercase `token=secret`, bearer headers, and raw token-shaped values can still be written to `approvals.jsonl`; use the shared argv/tool redaction path or equivalent coverage before persisting.
  - confidence: 0.88
- **[P2] Avoid synchronous full-log reads:** `src/infra/exec-approval-audit.ts:90`
  - body: `readFileSync` loads and parses the complete append-only audit file before applying `limit`. A long-lived gateway can accumulate a large file, so one audit-log request can block the Node event loop; use an async bounded or streaming read path.
  - confidence: 0.84
- **[P2] Register the new protocol contract end to end:** `src/gateway/protocol/schema/exec-approvals.ts:156-169`
  - body: Adding this params schema and validator is not enough for a public gateway RPC. The new method is missing from the protocol schema registry, generated schema types, generated Swift models, protocol docs, and a result schema, so generated clients will not learn the new contract.
  - confidence: 0.78
