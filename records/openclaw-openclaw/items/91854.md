## Summary

Keep open: current main and the latest release still route Dream Diary narrative generation through the normal plugin subagent path without a prose-only/no-tools profile, so the reported apply_patch exposure remains a real cleanup/security-boundary item.

## What This Changes

The issue asks memory-core Dream Diary narrative and REM runs to avoid file-mutation tools such as apply_patch so prose-only diary generation does not trigger patch-parser errors.

## Best Possible Solution

Run Dream Diary narrative generation through an explicit prose-only/no-tools runtime path while preserving the existing dreaming.model override behavior and adding focused regression coverage that apply_patch is unavailable for memory-core narrative lanes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main still routes Dream Diary through the normal plugin subagent path without a per-run tool allowlist; the exact intermittent weak-model apply_patch call still needs live reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the API shape; removing mutation tools from this prose lane is the right end state, but the clean implementation may be subagent tool scoping or a no-tools raw model path.
