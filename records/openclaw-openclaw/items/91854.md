## Summary

Keep open: current main and v2026.6.8 still route memory-core Dream Diary narrative generation through the plugin subagent runtime without a no-tools or prose-only control, so the reported apply_patch exposure remains a valid security-boundary cleanup item.

## What This Changes

The issue requests a prose-only/no-mutation tool profile for memory-core Dream Diary narrative subagent runs so apply_patch cannot be invoked during scheduled dreaming.

## Best Possible Solution

Run Dream Diary narrative generation through an explicit prose-only/no-tools runtime path while preserving dreaming.model override behavior and adding regression coverage that apply_patch is unavailable for memory-core narrative lanes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main still calls the plugin subagent runtime without a per-run no-tools control, though I did not live-reproduce the intermittent weak-model apply_patch call.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the API shape; removing mutation tools from this prose lane is the right end state, but the clean implementation may be subagent tool scoping or a no-tools raw model path.
