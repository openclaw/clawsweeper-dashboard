## Summary

Keep open: v2026.6.5 and current main both run Dream Diary narrative generation through the ordinary plugin subagent surface without tool scoping, so the reported least-privilege and log-noise problem remains; however, the permanent SDK contract needs maintainer judgment before implementation.

## What This Changes

Prevent memory dreaming narrative/REM generation from receiving and misusing file-mutation tools such as `apply_patch`.

## Best Possible Solution

Add an explicit opt-in no-tools or prose-only mode to the plugin subagent request contract, use it only for Dream Diary narrative phases, and test that mutation tools are unavailable while narrative extraction, cleanup, and fallback remain intact.

## Reproduction Assessment

Yes at source level: both v2026.6.5 and current main route Dream Diary prose through an unscoped normal subagent. The intermittent weak-model `apply_patch` call was not reproduced live, so this is not yet a high-confidence runtime reproduction.

## Solution Assessment

No, prompt tightening alone would not remove the capability and a global tool-deny change would be too broad; the narrow maintainable solution is an explicit, opt-in no-tools contract at the plugin subagent boundary.
