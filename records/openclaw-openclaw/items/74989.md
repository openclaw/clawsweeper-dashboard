## Summary

Keep open. Current main still has the undefined plugin-command result crash path reported by the linked Telegram issue, and the PR addresses that path; the remaining concrete blocker is the required changelog entry for a user-facing fix.

## What This Changes

The PR normalizes nullish and non-object plugin command results in the shared executor plus generic and Telegram dispatch paths, and adds regression coverage for undefined command results.

## Best Possible Solution

Land one narrow fix that normalizes plugin command results before reply dispatch reads fields, keep the regression tests, add the required single-line Unreleased Fixes entry, and then close #74800. If maintainers want no-payload handlers to become an explicit SDK guarantee, handle that as a follow-up docs/contract change rather than expanding this bug fix.

Yes. On current main, a plugin command handler that resolves `undefined` flows through `executePluginCommand` and then `handlePluginCommand` reads `result.continueAgent`; #74800 provides matching Telegram gateway logs for that TypeError.

## Solution Assessment

Yes, with the changelog fix. Normalizing at the shared executor boundary is the narrowest maintainable repair, and the generic/Telegram caller guards are reasonable defense for direct or mocked runtime paths.
