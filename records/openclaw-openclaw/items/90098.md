## Summary

## What This Changes

Make large Control UI attachment parsing and validation stack-safe and remove persisted queue entries that repeatedly fail with stack overflow.

## Best Possible Solution

Authorize one narrow PR limited to stack-safe data-URL parsing, iterative base64 validation, and exact-error stale queue cleanup, then prove a formerly affected upload through a real Control UI-to-gateway send without increasing limits or weakening media trust checks.

## Reproduction Assessment

No, not yet at high confidence on current main. Released gateway traces consistently identify isValidBase64 and the normalization path, but isolated large-string probes did not overflow, so a real affected Control UI upload remains necessary.

## Solution Assessment

Yes, conditionally. Replacing the two full-string data-URL/base64 regexes with index/slice and iterative validation, while pruning only queue failures with the exact historical error, is the narrowest maintainable repair; live gateway proof must confirm it addresses the runtime trigger.
