## Summary

Keep open: this maintainer-authored PR appears to address the prior missing explicit-timeout plumbing without a blocking code defect, but the longer no-output watchdog behavior is still an availability tradeoff for maintainers to accept before merge.

## What This Changes

The PR carries explicit and configured agent timeout intent into CLI-backed resumed runs so the no-output watchdog can use the longer fresh-run profile while preserving the inherited 180s resume guard.

## Best Possible Solution

Merge the narrow timeout-intent plumbing only after maintainers accept that configured CLI resumes should trade faster stale-process recovery for honoring user timeout intent.

## Reproduction Assessment

Yes. Source inspection of current main shows explicit `--timeout` is collapsed into `timeoutMs`, while resumed user CLI runs only use the fresh watchdog profile for cron, so a 600s user resume remains capped by the 180s resume profile.

## Solution Assessment

Yes, this is the best bounded implementation shape. It reuses the embedded runner's explicit-timeout signal pattern instead of trying to infer intent from numeric equality, with the remaining question limited to maintainer acceptance of longer silent waits.
