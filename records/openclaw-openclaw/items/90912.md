## Summary

Keep open: the change is useful but not merge-ready because normal user `--timeout` dispatch still does not set the new explicit timeout signal, and the configured-default path needs maintainer acceptance because it can leave silent resumed CLI processes running longer before abort.

## What This Changes

The PR changes resumed CLI-backed agent runs so configured/default and explicit run timeouts can raise the no-output watchdog above the 180s resume ceiling while inherited resumes keep the existing guard.

## Best Possible Solution

Wire the parsed per-run timeout through `agent-command` into `runAgentAttempt` and `runCliAgent`, cover both explicit and configured resumed CLI dispatch, and land after maintainers accept the longer configured watchdog behavior.

## Reproduction Assessment

Yes from source: current main uses the 180s resume watchdog for user-triggered resumed CLI runs unless the trigger is cron or backend watchdog config overrides it. The PR body includes a resolver probe, but not a live long-running CLI session.

## Solution Assessment

No: the resolver change is a plausible layer, but explicit per-run timeout support is incomplete until the parsed timeout is plumbed from command dispatch and covered by a dispatch-level test.
