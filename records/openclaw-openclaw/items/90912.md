## Summary

Keep open. The patch targets the right CLI timeout path, but it is not merge-ready because the new `runTimeoutOverrideMs` field is forwarded only after `runAgentAttempt` receives it, and the sole production user-command caller still collapses `--timeout` into `timeoutMs` without passing the explicitness signal.

## What This Changes

The PR changes resumed CLI-backed agent runs so configured/default and explicit run timeouts can lift the no-output watchdog above the existing 180s resume ceiling.

## Best Possible Solution

Before merge, carry an explicit run-timeout signal from `agent-command.ts` into `runAgentAttempt` and `runCliAgent`, while keeping inherited user resumes on the 180s ceiling when no timeout intent exists.

## Reproduction Assessment

Yes at source level: current main routes user-triggered resumed CLI runs through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000` unless the trigger is cron or a backend watchdog override exists. I did not run a live 180s CLI provider session in this read-only review.

## Solution Assessment

No, not yet: forwarding `runTimeoutOverrideMs` inside CLI dispatch is the right layer, but the PR does not pass that signal from the user command parser that knows `--timeout` was explicit. The safer fix is to compute it next to `timeoutSecondsRaw` and cover the full dispatch path.
