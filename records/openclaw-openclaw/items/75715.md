## Summary

Keep open, but do not merge as-is. Current main already builds the Play debug APK in Android CI; this PR starts publishing that debug APK as a workflow artifact, which does not resolve the canonical prebuilt Android release request and creates a supply-chain/product risk for untrusted PR-built APKs.

## What This Changes

This PR adds an `Upload Android APK` artifact step to the Android CI job for the `build-play` matrix task.

## Best Possible Solution

Handle Android APK distribution through the canonical prebuilt Android release tracker #9443 with a trusted release or maintainer-dispatched workflow, signed or clearly non-official artifacts, provenance/naming that prevents confusion, and matching docs; keep normal PR CI focused on validation unless maintainers explicitly want internal debug artifacts.

## Reproduction Assessment

Yes. Reading the workflow is enough to reproduce the behavior: CI runs on `pull_request`, the proposed step runs when `matrix.task == 'build-play'`, and the existing Gradle task produces debug APKs under `apps/android/app/build/outputs/apk`.

## Solution Assessment

No. Publishing a debug APK from normal Android CI is not the narrowest maintainable solution for prebuilt Android releases; a trusted release/manual path tied to #9443 is safer than generic PR artifact upload.

## Review Findings

- **[P2] Restrict APK artifacts to trusted release runs:** `.github/workflows/ci.yml:2146-2151`
