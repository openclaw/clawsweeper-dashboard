## Summary

Keep open: the PR addresses a real Google Chat message-loss bug, but it leaves the message-action media path uncovered, uses an overbroad upload-error predicate, and still lacks inspectable real app-auth Google Chat proof.

## What This Changes

The PR adds Google Chat remote-media text-link fallback after attachment-upload authorization errors in the outbound adapter and monitor reply delivery paths, plus focused tests and a runtime helper export.

## Best Possible Solution

Use a narrow insufficient-authentication-scope upload predicate and apply the same remote-link fallback to every affected Google Chat media caller, then provide redacted live app-auth proof or explicit maintainer acceptance.

## Reproduction Assessment

Yes at source-contract level: current main and the latest release route Google Chat remote media through `attachments:upload`, while Google's docs require user authentication for media upload and allow app-auth text message creation. I did not run a live Google Chat credentialed repro.

## Solution Assessment

No as written; the channel-local link fallback is the right layer, but it must cover the shared message-action caller and activate only for the specific insufficient-scope upload failure.
