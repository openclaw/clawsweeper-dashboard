## Summary

Keep this PR open: the reported Google prefix regression is valid, but the patch is incomplete because it only normalizes the synchronous resolver while the normal embedded runner path uses the async resolver, and the rewrite needs to preserve provider-local namespaced model IDs.

## What This Changes

The PR changes `resolveModel` to strip a same-provider prefix before registry lookup and adds resolver tests for Google inline models and OpenRouter pass-through IDs.

## Best Possible Solution

Land one shared resolver normalization path that covers sync and async resolution, strips only redundant outer same-provider refs, preserves proxy and provider-local nested IDs, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. Source inspection shows provider `google` plus model ID `google/gemini-2.5-pro` can still miss a bare configured `gemini-2.5-pro`, and the main embedded runner path reaches `resolveModelAsync`.

## Solution Assessment

No. The direction is right, but the current patch is not the best fix because it is sync-only and should use a shared, non-destructive normalization path that preserves namespaced local model IDs.
