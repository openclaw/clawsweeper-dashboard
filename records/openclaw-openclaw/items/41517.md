## Summary

Keep open and do not merge as-is: the Google prefix bug is valid, but this patch strips too broadly and can break current provider-literal model IDs such as NVIDIA catalog entries.

## What This Changes

The PR normalizes model IDs with `parseModelRef` before embedded model resolution, reports unknown-model errors with the normalized ID, and adds two `resolveModel` regression tests.

## Best Possible Solution

Fix request-side redundant Google prefixes at the matching boundary, while preserving nested and literal provider-local IDs for OpenRouter, NVIDIA, MLX-style, and other slash-bearing models.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call embedded `resolveModel` with provider `google`, model ID `google/gemini-2.5-pro`, and a config whose Google provider lists bare `gemini-2.5-pro`; current matching misses that bare configured row. The introduced regression is also source-reproducible with a provider-local NVIDIA ID such as `nvidia/nemotron-3-super-120b-a12b`.

## Solution Assessment

No. Pre-normalizing every same-provider slash ID with `parseModelRef` is broader than the bug requires; a safer solution is to match a request-side `provider/model` value against bare configured IDs without rewriting provider-local slash IDs globally.

## Review Findings

Scope the Google fix to matching redundant request prefixes instead of stripping all same-provider slash IDs.
