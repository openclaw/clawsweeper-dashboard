## Summary

Keep open. The PR addresses a real Google model-prefix bug, but the proposed patch only normalizes the synchronous resolver while current embedded agent turns use the asynchronous resolver, so the reported 404 path can remain. The fix also needs the required user-facing changelog entry before merge.

## What This Changes

The branch changes `resolveModel` to strip redundant same-provider prefixes before registry lookup and adds resolver regression tests for Google configured models and OpenRouter pass-through IDs.

## Best Possible Solution

Land one narrow resolver fix that shares same-provider outer-prefix normalization between `resolveModel` and `resolveModelAsync`, preserves OpenRouter and provider-native nested IDs such as NVIDIA refs, adds focused sync and async regressions, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The source-level reproduction is provider `google` with model id `google/gemini-2.5-pro` and a config that lists bare `gemini-2.5-pro`; current main’s matcher and async runner path show why the prefixed request can miss the configured model.

## Solution Assessment

No. The direction is right, but this patch is incomplete because it changes only the synchronous resolver; the maintainable fix should centralize the normalization and use it in both resolver paths without broadening transport or provider-native ID behavior.
