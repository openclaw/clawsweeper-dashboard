## Summary

Keep open. Current main still lacks the requested same-provider Google prefix stripping, and this PR remains useful but incomplete because it only changes the synchronous resolver while the embedded runner’s main path uses the asynchronous resolver. The overlapping #41398 is still open, so there is no merged superseding fix to justify cleanup closure.

## What This Changes

The PR changes `resolveModel` to parse provider/model refs and strip a redundant active-provider prefix before resolver lookup, with regression tests for Google same-provider prefixes and OpenRouter cross-provider pass-through ids.

## Best Possible Solution

Consolidate this PR with #41398 into one narrow resolver fix that shares the same-provider prefix normalization between `resolveModel` and `resolveModelAsync`, strips only genuinely redundant active-provider prefixes for matching/transport, preserves pass-through and provider-native nested ids, and adds regression tests for Google configured-model matching plus OpenRouter/NVIDIA preservation.

## Reproduction Assessment

Yes. The source-level reproduction is `resolveModel("google", "google/gemini-2.5-pro", ...)` or the async embedded-runner path with a bare configured `gemini-2.5-pro`; current main’s matching logic and Google manifest show why the prefixed request misses the bare configured model.

## Solution Assessment

No. The PR is directionally plausible, but it is not the best complete fix because it only changes the synchronous resolver while the live runner uses `resolveModelAsync`; a shared helper with sync/async coverage and provider-native preservation tests is safer.
