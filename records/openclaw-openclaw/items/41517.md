## Summary

Keep open: the PR targets a real Google model-prefix regression, but the branch is not mergeable as-is because it only changes the sync resolver, applies generic same-provider stripping that can break valid provider-local namespaced IDs, and still lacks real behavior proof.

## What This Changes

The branch strips same-provider prefixes inside `resolveModel` and adds resolver tests for a Google prefixed model plus OpenRouter cross-provider pass-through.

## Best Possible Solution

Consolidate this into one provider-owned Google prefix-normalization fix through the shared manifest/static normalization and native Google transport paths, with coverage for sync, async, configured-model, and sub-agent flows while preserving valid nested provider IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows provider `google` plus model ID `google/gemini-*` is not stripped by the Google manifest/static normalizer, and related issue comments show configured Google models with prefixed IDs still producing 404s.

## Solution Assessment

No. The PR identifies the right regression, but a sync-only generic strip in `resolveModel` is not the narrowest maintainable fix; provider-scoped normalization shared by sync and async resolution is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Normalize through the shared async path:** `src/agents/pi-embedded-runner/model.ts:257-263`
  - body: This only rewrites `resolveModel`, but normal embedded runs also use `resolveModelAsync`. A prefixed Google model can still reach the async lookup/fallback path unchanged, so put the provider-owned strip in the shared static or manifest normalization path used by both resolvers.
  - confidence: 0.91
- **[P1] Preserve provider-local namespaced model IDs:** `src/agents/pi-embedded-runner/model.ts:257-263`
  - body: The new generic same-provider strip treats every `<provider>/...` as redundant. Providers can have valid local IDs like `groq/compound`, so resolving that as provider `groq` would rewrite it to `compound` and can miss configured/catalog matches.
  - confidence: 0.9
