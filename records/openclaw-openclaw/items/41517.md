## Summary

Keep open. This PR targets a real Google model-prefix regression, but the branch only patches the sync resolver, can over-strip valid provider-local namespaced model IDs, and lacks real after-fix behavior proof; the remaining work should be consolidated with the related Google prefix threads rather than closed as stale or implemented.

## What This Changes

The PR strips same-provider model prefixes inside `resolveModel` and adds resolver tests for Google prefix stripping plus OpenRouter cross-provider pass-through IDs.

## Best Possible Solution

Land one focused Google-owned prefix-normalization fix through the shared manifest/static normalization and native Google transport paths, preserving wrapper/provider-local namespaced IDs and covering sync, async, configured-model, and sub-agent paths.

## Reproduction Assessment

Yes, source-reproducible. Current main has a clear path where provider `google` plus model ID `google/gemini-*` is not stripped by the Google manifest/static normalizer, and related issue comments show configured Google models with prefixed IDs still producing 404s.

## Solution Assessment

No. The PR identifies the right regression, but a sync-only generic strip in `resolveModel` is not the narrowest maintainable fix; the safer solution is provider-scoped normalization shared by sync and async resolution plus Google transport coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the shared async resolver normalization path:** `src/agents/pi-embedded-runner/model.ts:257-263`
  - body: This only rewrites `resolveModel`, but normal embedded runs also use `resolveModelAsync`. A prefixed Google model can still reach the async lookup/fallback path unchanged, so put the provider-owned strip in the shared static/manifest normalization path used by both resolvers.
  - confidence: 0.91
- **[P1] Preserve provider-local namespaced model IDs:** `src/agents/pi-embedded-runner/model.ts:257-263`
  - body: The new generic same-provider strip treats every `<provider>/...` as redundant. Providers can have valid local IDs like `groq/compound`; resolving that as provider `groq` would rewrite it to `compound` and miss the configured/catalog model. Scope stripping to explicit provider policy instead.
  - confidence: 0.9
