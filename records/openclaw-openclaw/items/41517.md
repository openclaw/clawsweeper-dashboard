## Summary

Keep open: the PR targets a real Google model-prefix bug, but the branch is incomplete for current main and should not merge as-is because it normalizes only the legacy sync resolver path, risks over-stripping provider-local namespaced IDs, and lacks the required changelog entry.

## What This Changes

The branch rewrites `resolveModel` to parse same-provider model refs before lookup and adds sync resolver tests for Google prefix stripping and OpenRouter pass-through IDs.

## Best Possible Solution

Use the existing provider-owned manifest/static normalization path to strip Google’s redundant `google/` prefix for both sync and async resolution, preserve provider-native nested IDs, add regression tests, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Source inspection shows the reported `provider=google` plus `modelId=google/gemini-2.5-pro` shape can miss bare configured Google models unless the provider prefix is stripped before resolver matching, and current main’s normal runner path reaches `resolveModelAsync`.

## Solution Assessment

No. The direction is right, but the PR’s sync-only parser is not the narrowest maintainable current-main fix; the existing manifest/static normalization seam should carry the provider-owned strip policy and cover both resolver paths.
