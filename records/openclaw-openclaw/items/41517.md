## Summary

Keep open. The branch addresses the right resolver area, but it only normalizes the synchronous resolver while current main routes normal embedded agent turns through the async resolver, so the reported Google prefixed-model failure can still survive the patch.

## What This Changes

The PR changes `resolveModel` to strip redundant same-provider prefixes before lookup and adds resolver regression tests for Google same-provider refs and OpenRouter cross-provider pass-through ids.

## Best Possible Solution

Land one narrow resolver fix that shares same-provider prefix normalization between sync and async model resolution, preserves cross-provider and provider-native nested ids, covers Google configured-model matching plus OpenRouter/NVIDIA preservation, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The source-level reproduction is resolving provider `google` with model id `google/gemini-2.5-pro` against a config that lists bare `gemini-2.5-pro`; current main's matcher and manifest normalization show why the prefixed request misses the bare configured model.

## Solution Assessment

No. The proposed direction is reasonable, but the patch is incomplete because it only changes `resolveModel`; the maintainable fix should use a shared normalization path that also covers `resolveModelAsync` and its live runner call sites.
