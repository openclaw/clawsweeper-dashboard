## Summary

Keep open. Current main still keeps Slack thread participation in memory only, so the restart bug remains valid, but the PR should not merge as-is because the custom JSON persistence still has a non-regular-file hydration risk and a focused test defect, and the newly merged plugin state store changes the preferred storage direction.

## What This Changes

The PR adds JSON-backed persistence for the Slack thread participation cache, expands restart and persistence tests, and keeps only the production cache functions exported from the Slack API barrel.

## Best Possible Solution

Land one owner-approved Slack restart-survival implementation that preserves the fast in-memory participation check, stores account/channel/thread participation durably for the existing 24-hour policy, preferably uses `api.runtime.state.openKeyedStore` where feasible, and keeps restart, TTL, clear, corrupt-state, cap, and split-module behavior covered by focused Slack tests.

## Reproduction Assessment

Yes. Record Slack participation in a thread, restart the gateway so the process-global cache is empty, then send a non-mention reply in that same thread; current main has no durable state for `prepare.ts` to classify it as `bot_thread_participant`.

## Solution Assessment

No. The branch targets the right user-visible bug, but the custom JSON persistence is not the best merge shape while `openKeyedStore` exists and while the patch still has non-regular-file, failed-rename cleanup, and isolated-test defects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject non-regular state files before hydration:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` follows the configured path with `statSync` and then `readFileSync` without verifying it is a regular file. If the state path is replaced with a FIFO, device, or symlink, first Slack message handling can block or read unintended local content before falling back; validate with `lstat`/`fstat` or move this persistence onto the keyed store before reading.
  - confidence: 0.87
- **[P3] Remove temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: After writing the UUID-named temp JSON file, `renameSync` can throw and the scheduled persist path swallows the error. That leaves Slack thread identifiers in leftover `.tmp` files, so wrap the rename and unlink `tmpPath` on failure.
  - confidence: 0.82
- **[P3] Initialize tempDir before the directory-mode test:** `extensions/slack/src/sent-thread-cache.test.ts:311-312`
  - body: This test builds `path.join(tempDir, ...)` without calling `setup()`, so running it by itself starts with `tempDir` undefined and fails before exercising the directory-mode assertion. Create the temp dir in this test or call the shared setup helper first.
  - confidence: 0.79

## Security Review

  - body: The persisted state path is read with `statSync` and `readFileSync` without rejecting symlinks, FIFOs, devices, or other non-regular files, which can turn local state-path tampering into a gateway event-loop block or unintended local file read before Slack message handling continues.
