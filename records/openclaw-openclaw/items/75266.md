## Summary

Keep open: the PR targets a real config-surface request, but the current patch is not mergeable because it regresses existing heartbeat defaults, drops a shipped schema field, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds `heartbeat: false` support to agent heartbeat config schemas/types and threads that disabled state through heartbeat runtime, prompt, ACP, WhatsApp, config validation, generated schema output, and tests.

## Best Possible Solution

If maintainers want the new shorthand, add it through one centralized disabled-state resolver that preserves existing `0m`, global defaults, `skipWhenBusy`, prompt, ACP, summary, runner, docs, schema, and tests.

## Reproduction Assessment

Yes for the review blockers by source inspection: the PR diff deterministically collapses literal `false` into missing config in prompt and ACP paths, drops `skipWhenBusy` from the strict schema branch, and changes global default semantics covered by current tests. I did not run tests because this review is read-only.

## Solution Assessment

No. Accepting `heartbeat: false` may be a reasonable product direction, but this patch needs a central disabled-state contract and must preserve existing heartbeat object fields and default inheritance before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve global heartbeat defaults for all agents:** `src/infra/heartbeat-summary.ts:47-51`
  - body: With no explicit per-agent heartbeat entries, `agents.defaults.heartbeat: { every: "30m" }` currently enables heartbeats for every listed agent. This change only checks for literal `false` and then falls back to default-agent-only behavior, so non-default agents stop inheriting global heartbeat defaults.
  - confidence: 0.93
- **[P2] Keep false disabled in heartbeat prompt policy:** `src/agents/heartbeat-system-prompt.ts:17-25`
  - body: Returning `undefined` for `defaults === false` or `overrides === false` makes disabled heartbeat look the same as absent heartbeat. The cadence check then falls back to `DEFAULT_HEARTBEAT_EVERY`, so `heartbeat: false` can still inject heartbeat system-prompt guidance.
  - confidence: 0.88
- **[P2] Treat heartbeat false as disabled in ACP policy:** `src/agents/acp-spawn.ts:325-329`
  - body: `resolveHeartbeatConfigForAgent` returns `undefined` for explicit `false`, but `isHeartbeatEnabledForSessionAgent` then uses `heartbeat?.every ?? DEFAULT_HEARTBEAT_EVERY`. That can leave ACP heartbeat relay decisions enabled after a user disables heartbeat with `false`.
  - confidence: 0.84
- **[P2] Preserve skipWhenBusy in the heartbeat schema:** `src/config/zod-schema.agent-runtime.ts:45-64`
  - body: The new strict object branch omits `skipWhenBusy`, even though current main accepts, documents, and consumes that heartbeat field. Existing valid configs using busy-lane deferral would fail validation after this change.
  - confidence: 0.94
- **[P3] Remove the hard-coded local gate path:** `gates/openclaw-heartbeat-false-disable.sh:5`
  - body: The checked-in gate changes into `/Users/karlkarl/Documents/vibe_coding/openclaw-upstream`, so it only works on one contributor machine. Remove it or resolve the repository root portably before keeping a gate script.
  - confidence: 0.91
