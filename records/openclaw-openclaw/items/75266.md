## Summary

Keep this PR open: the requested `heartbeat: false` config mode is useful, but the current patch still regresses existing heartbeat object behavior and leaves concrete P2/P3 review blockers before merge.

## What This Changes

The PR adds support for configuring `agents.defaults.heartbeat` and per-agent `heartbeat` as `false`, then threads that state through heartbeat schemas, runtime helpers, generated schema output, and tests.

## Best Possible Solution

Keep the `heartbeat: false` capability, but carry an explicit disabled state through schema, summaries, prompt policy, ACP policy, runner helpers, generated docs, tests, and changelog while preserving the existing object-config contract.

## Reproduction Assessment

Yes for the review blockers: current-main source and tests define the existing heartbeat object semantics, and the PR diff shows false being collapsed into absent/default config in several runtime paths. I did not run tests because this review was read-only.

## Solution Assessment

No. Accepting `false` is a reasonable direction, but this patch needs a false-aware helper or sentinel, must preserve global object defaults and `skipWhenBusy`, and should remove the local validation artifact before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve per-agent false in the runner:** `src/infra/heartbeat-runner.ts:198-200`
  - body: Returning `undefined` for `overrides === false` loses the explicit disabled state. With `agents.defaults.heartbeat: { every: "30m" }` and the default agent set to `heartbeat: false`, `resolveHeartbeatIntervalMs` falls back to the global default and the runner can still schedule that agent. Carry a disabled sentinel through this helper instead of treating false as absent.
  - confidence: 0.9
- **[P2] Preserve global heartbeat defaults for all agents:** `src/infra/heartbeat-summary.ts:47-51`
  - body: The no-explicit-agent path now falls through to only `resolveDefaultAgentId`, so `agents.defaults.heartbeat: { every: "30m" }` stops enabling heartbeats for other listed agents. Current tests treat defaults as global until explicit per-agent heartbeat blocks opt into a subset; keep object defaults enabled for all agents and handle only literal `false` as disabled.
  - confidence: 0.93
- **[P2] Keep false disabled in heartbeat prompt policy:** `src/agents/heartbeat-system-prompt.ts:19-27`
  - body: Returning `undefined` for `defaults === false` or `overrides === false` collapses disabled heartbeat into absent heartbeat, and the policy then falls back to `DEFAULT_HEARTBEAT_EVERY`. A default or per-agent `heartbeat: false` can still inject heartbeat system-prompt guidance, so carry an explicit disabled state into this check.
  - confidence: 0.88
- **[P2] Treat heartbeat false as disabled in ACP policy:** `src/agents/acp-spawn.ts:329`
  - body: `resolveHeartbeatConfigForAgent` returns `undefined` for false, but `isHeartbeatEnabledForSessionAgent` falls back to `DEFAULT_HEARTBEAT_EVERY` when `heartbeat?.every` is absent. That can leave ACP heartbeat relay decisions enabled after a user disables heartbeat with `false`.
  - confidence: 0.84
- **[P2] Preserve skipWhenBusy in the heartbeat schema:** `src/config/zod-schema.agent-runtime.ts:42-47`
  - body: The new strict object branch omits `skipWhenBusy`, even though current main accepts, documents, generates schema for, and consumes that heartbeat field. Existing valid configs using busy-lane deferral would fail validation, so keep `skipWhenBusy` in the object branch and exported type while adding the false arm.
  - confidence: 0.94
- **[P3] Remove the hard-coded local gate path:** `gates/openclaw-heartbeat-false-disable.sh:5`
  - body: The added gate changes into `/Users/karlkarl/Documents/vibe_coding/openclaw-upstream`, so it only works on the contributor's machine. Remove the checked-in local harness or resolve the repository root portably before keeping a gate script.
  - confidence: 0.91
