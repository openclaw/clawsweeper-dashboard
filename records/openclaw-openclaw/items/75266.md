## Summary

Keep open: the PR addresses a real missing capability on current main, but the current patch is not mergeable because the new `false` union is not carried consistently through type consumers or heartbeat policy paths, and it drops an existing heartbeat field from the schema contract.

## What This Changes

The PR widens heartbeat config schemas/types to allow `false`, updates heartbeat runtime, prompt, delivery, provider/default, WhatsApp, memory, and model-pricing helpers, regenerates the base config schema, adds regression tests, and adds a local gate script.

## Best Possible Solution

Keep the `heartbeat: false` capability, but implement it with a clear disabled sentinel or helper that distinguishes disabled from absent config across schema, exported types, scheduler, prompt, ACP, cron wake, doctor, model lookup, and delivery helpers while preserving all existing heartbeat object fields and docs.

## Reproduction Assessment

Yes. The schema regression reproduces by parsing an existing supported config such as `heartbeat: { timeoutSeconds: 45, skipWhenBusy: true }` against the PR object branch, and the false-vs-absent regressions follow from source inspection where `false` becomes `undefined` before callers fall back to `DEFAULT_HEARTBEAT_EVERY`.

## Solution Assessment

No, not as written. Adding `false` support is the right narrow direction, but the patch needs false-aware narrowing across all consumers, preservation of the existing heartbeat object contract, removal of the local-only gate, and a changelog entry before it is the maintainable fix.
