## Summary

Keep this PR open. The requested `heartbeat: false` config support is still missing from current main, but the proposed patch is not merge-ready because it regresses the existing `skipWhenBusy` heartbeat object contract, collapses `false` into `undefined` on prompt/ACP paths that then fall back to the default heartbeat cadence, and adds a local-only gate script.

## What This Changes

The PR allows `agents.defaults.heartbeat` and per-agent `heartbeat` to be set to `false` across config schemas, runtime helpers, prompt/summary/target resolution, WhatsApp heartbeat handling, generated schema output, and regression tests.

## Best Possible Solution

Keep the `heartbeat: false` capability, but represent disabled heartbeat distinctly from absent heartbeat everywhere policy code can fall back to `DEFAULT_HEARTBEAT_EVERY`; preserve every existing heartbeat object field, especially `skipWhenBusy`, across Zod schema, exported types, generated schema, docs/help, and tests; remove the local-only gate script and add a required changelog entry for the user-facing config fix.

## Reproduction Assessment

Yes. The schema regression reproduces by parsing current valid config like `heartbeat: { timeoutSeconds: 45, skipWhenBusy: true }` against the PR’s strict object branch, and the prompt/ACP regressions follow from source inspection because `false` is converted to `undefined` before code that defaults absent heartbeat to `DEFAULT_HEARTBEAT_EVERY`.

## Solution Assessment

No, not as written. Adding `false` is the right direction, but the implementation needs a false-aware disabled sentinel or helper instead of treating disabled and absent config the same, plus preservation of the full existing heartbeat object contract.
