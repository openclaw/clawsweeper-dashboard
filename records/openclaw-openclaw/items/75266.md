## Summary

This PR should close because the underlying user need, disabling heartbeats via config, is already covered by the shipped `every: "0m"` contract and the branch adds a new boolean alias with correctness regressions and no real behavior proof.

## What This Changes

The PR adds `heartbeat: false` as an accepted agent-default and per-agent config value, then threads that disabled state through heartbeat runtime, prompt, ACP, WhatsApp, schema, generated schema, and tests.

## Best Possible Solution

Keep the shipped `every: "0m"` disable contract; any future boolean alias should start from a maintainer-approved config proposal and one centralized heartbeat resolver.

## Reproduction Assessment

Yes for the PR blockers: source inspection of PR head shows disabled `false` is collapsed into absent config before default cadence fallback, global default inheritance regresses, and `skipWhenBusy` is dropped from the strict schema branch. I did not run tests because this review is read-only.

## Solution Assessment

No. The supported disable behavior is already shipped via `every: "0m"`, while this patch adds a new config alias with regressions and without maintainer approval of the config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve global heartbeat defaults for all agents:** `src/infra/heartbeat-summary.ts:47-51`
  - body: With no explicit per-agent heartbeat entries, `agents.defaults.heartbeat: { every: "30m" }` currently enables heartbeats for every listed agent. This change only checks for literal `false` and then falls through to default-agent-only behavior, so non-default agents stop inheriting global heartbeat defaults.
  - confidence: 0.93
- **[P2] Keep false disabled in heartbeat prompt policy:** `src/agents/heartbeat-system-prompt.ts:16-25`
  - body: Returning `undefined` for `defaults === false` or `overrides === false` makes disabled heartbeat look the same as absent heartbeat. The cadence check then falls back to `DEFAULT_HEARTBEAT_EVERY`, so `heartbeat: false` can still inject heartbeat system-prompt guidance.
  - confidence: 0.88
- **[P2] Treat heartbeat false as disabled in ACP policy:** `src/agents/acp-spawn.ts:325-330`
  - body: `resolveHeartbeatConfigForAgent` returns `undefined` for explicit `false`, but `isHeartbeatEnabledForSessionAgent` then uses `heartbeat?.every ?? DEFAULT_HEARTBEAT_EVERY`. That can leave ACP heartbeat relay decisions enabled after a user disables heartbeat with `false`.
  - confidence: 0.84
- **[P2] Preserve skipWhenBusy in the heartbeat schema:** `src/config/zod-schema.agent-runtime.ts:45-47`
  - body: The new strict object branch omits `skipWhenBusy`, even though current main accepts, documents, and consumes that heartbeat field. Existing valid configs using busy-lane deferral would fail validation after this change.
  - confidence: 0.94
- **[P3] Remove the hard-coded local gate path:** `gates/openclaw-heartbeat-false-disable.sh:5`
  - body: The checked-in gate changes into `/Users/karlkarl/Documents/vibe_coding/openclaw-upstream`, so it only works on one contributor machine. Remove it or resolve the repository root portably before keeping a gate script.
  - confidence: 0.91
