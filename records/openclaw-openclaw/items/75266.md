## Summary

Keep open: current main still lacks `heartbeat: false` support, but this PR head regresses existing heartbeat behavior by collapsing disabled and unset config, dropping current `skipWhenBusy` support, adding a local-only gate script, and missing user-facing release/docs updates.

## What This Changes

The PR adds `heartbeat: false` as an accepted default/per-agent config value and threads it through heartbeat runtime, prompt, outbound target, WhatsApp, generated schema, and tests.

## Best Possible Solution

Land a false-aware heartbeat config helper that distinguishes disabled from unset while preserving object defaults for all agents and existing heartbeat fields, then update generated schema, docs/changelog, and focused tests.

## Reproduction Assessment

Yes. The regressions are reproducible from deterministic source paths: the PR head converts `false` to `undefined` before helpers that fall back to `DEFAULT_HEARTBEAT_EVERY`, and current tests/docs define global defaults and `skipWhenBusy` as existing behavior.

## Solution Assessment

No. Adding `heartbeat: false` is a good direction, but the patch needs an explicit disabled state rather than treating disabled and absent config the same, plus preservation of the current heartbeat object contract.
