## Summary

Keep open: current main does not yet support `heartbeat: false`, and the PR addresses a useful config capability, but the proposed patch still introduces source-reproducible regressions in existing heartbeat defaults, disabled-state handling, schema coverage, and checked-in validation artifacts.

## What This Changes

The PR extends heartbeat config schemas, types, runtime helpers, generated schema output, and tests so `agents.defaults.heartbeat` and per-agent `heartbeat` can be set to `false`.

## Best Possible Solution

Keep the `heartbeat: false` capability, but represent disabled heartbeat explicitly through schema, summaries, prompt policy, ACP policy, runner helpers, and generated docs while preserving the existing object-config contract.

## Reproduction Assessment

Yes for the review blockers: current-main source and tests define the existing heartbeat object semantics, and the provided PR diff deterministically collapses literal false into the same state as absent config in prompt/ACP paths. I did not run tests because this was a read-only review.

## Solution Assessment

No. Accepting `false` is a reasonable direction, but this patch needs a false-aware helper or sentinel, must preserve global object defaults and `skipWhenBusy`, and should remove the local validation artifact before merge.
