## Summary

Keep open: the PR proposes a new `heartbeat: false` config surface that current main does not support, but the branch still has correctness blockers and no contributor-provided real behavior proof. Current main already documents and tests `every: "0m"` as the shipped disable path, so accepting the boolean shorthand needs maintainer product judgment.

## What This Changes

The PR adds `heartbeat: false` support across agent defaults and per-agent schemas, heartbeat runtime/prompt/ACP/WhatsApp paths, generated schema output, and regression tests.

## Best Possible Solution

If maintainers want boolean disable, implement it through one shared heartbeat disable and precedence resolver that preserves `0m`, default inheritance, `skipWhenBusy`, prompt/ACP/scheduler behavior, docs, and tests; otherwise keep the documented `0m` contract.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the diff shows literal `false` is converted to absent config before prompt/ACP cadence defaults, global defaults are no longer preserved for non-default agents, and `skipWhenBusy` is dropped from the strict schema branch. I did not run tests because this review is read-only.

## Solution Assessment

No. Accepting `heartbeat: false` may be reasonable, but this patch needs a centralized disabled-state contract and must preserve existing heartbeat object fields and default inheritance before it is maintainable.

- **[P2] Preserve global heartbeat defaults for all agents:** `src/infra/heartbeat-summary.ts:47-51`
  - body: With no explicit per-agent heartbeat entries, `agents.defaults.heartbeat: { every: "30m" }` currently enables heartbeats for every listed agent.   - confidence: 0.93
- **[P2] Keep false disabled in heartbeat prompt policy:** `src/agents/heartbeat-system-prompt.ts:17-25`
  - body: Returning `undefined` for `defaults === false` or `overrides === false` makes disabled heartbeat look the same as absent heartbeat. The cadence check then falls back to `DEFAULT_HEARTBEAT_EVERY`, so `heartbeat: false` can still inject heartbeat system-prompt guidance.
  - confidence: 0.88

  - body: `resolveHeartbeatConfigForAgent` returns `undefined` for explicit `false`, but `isHeartbeatEnabledForSessionAgent` then uses `heartbeat?.every ?? DEFAULT_HEARTBEAT_EVERY`. That can leave ACP heartbeat relay decisions enabled after a user disables heartbeat with `false`.
  - confidence: 0.84
- **[P2] Preserve skipWhenBusy in the heartbeat schema:** `src/config/zod-schema.agent-runtime.ts:45-64`
  - body: The new strict object branch omits `skipWhenBusy`, even though current main accepts, documents, and consumes that heartbeat field. Existing valid configs using busy-lane deferral would fail validation after this change.
