## Summary

Keep open. Current main still lacks `heartbeat: false` support, but this PR is not ready because it regresses existing heartbeat object defaults, disabled-state handling, and `skipWhenBusy`, and it lacks after-fix real behavior proof.

## What This Changes

The PR changes heartbeat configuration to accept `false` and threads that disabled state through schemas, runtime helpers, prompt policy, delivery target resolution, WhatsApp heartbeats, generated schema output, and tests.

## Best Possible Solution

Add `heartbeat: false` through one explicit disabled-state helper or sentinel, preserve object-default and `skipWhenBusy` behavior, update the current schema/docs surfaces, and cover prompt, ACP, summary, runner, and config parsing paths with focused tests.

## Reproduction Assessment

Yes for the review blockers by source inspection: current tests define global heartbeat object semantics, and the PR head deterministically collapses literal `false` into missing config in prompt and ACP paths. I did not run tests because this review was read-only.

## Solution Assessment

No. Accepting `heartbeat: false` is a reasonable direction, but this patch needs to preserve existing heartbeat object fields and carry disabled state explicitly instead of falling back to default heartbeat behavior.

- **[P2] Preserve global heartbeat defaults for all agents:** `src/infra/heartbeat-summary.ts:47-51`
  - body: With no explicit per-agent heartbeat entries, `agents.defaults.heartbeat: { every: "30m" }` currently enables heartbeats for every listed agent.   - confidence: 0.93
- **[P2] Keep false disabled in heartbeat prompt policy:** `src/agents/heartbeat-system-prompt.ts:17-25`
  - body: Returning `undefined` for `defaults === false` or `overrides === false` makes disabled heartbeat look the same as absent heartbeat. The cadence check then falls back to `DEFAULT_HEARTBEAT_EVERY`, so `heartbeat: false` can still inject heartbeat system-prompt guidance.
  - confidence: 0.88
- **[P2] Treat heartbeat false as disabled in ACP policy:** `src/agents/acp-spawn.ts:325-329`
  - body: `resolveHeartbeatConfigForAgent` returns `undefined` for explicit `false`, but `isHeartbeatEnabledForSessionAgent` then uses `heartbeat?.every ?? DEFAULT_HEARTBEAT_EVERY`. That can leave ACP heartbeat relay decisions enabled after a user disables heartbeat with `false`.
  - confidence: 0.84

  - body: The new strict object branch omits `skipWhenBusy`, even though current main accepts, documents, and consumes that heartbeat field. Existing valid configs using busy-lane deferral would fail validation after this change.
