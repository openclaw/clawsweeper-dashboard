## Summary

Keep open: current main does not implement `heartbeat: false`, and the PR is pointed at a real config gap, but the proposed patch still has blocking behavior and contract regressions before it can merge.

## What This Changes

The PR allows `agents.defaults.heartbeat` and per-agent `heartbeat` to be set to `false` across config schemas, runtime heartbeat helpers, prompt/target resolution, WhatsApp heartbeat handling, generated schema, and tests.

## Best Possible Solution

Keep the `heartbeat: false` capability, but carry an explicit disabled state through policy helpers, preserve existing heartbeat object fields, remove non-portable validation artifacts, and update generated schema plus changelog together.

## Reproduction Assessment

Yes. The regressions are reproducible from deterministic source paths: object defaults now enable only the default agent, and false config is converted to `undefined` before prompt/ACP helpers that fall back to `DEFAULT_HEARTBEAT_EVERY`.

## Solution Assessment

No. Adding `false` is the right direction, but the implementation needs a false-aware helper or sentinel rather than treating disabled and absent config the same, and it must preserve the current heartbeat object contract.
