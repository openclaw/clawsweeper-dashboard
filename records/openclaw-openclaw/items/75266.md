## Summary

Keep this PR open. Current main does not yet accept `heartbeat: false`, so the contribution is not obsolete, but the branch is not mergeable as written because it breaks an existing heartbeat config field, leaves the system-prompt/HEARTBEAT.md path enabled for the false default case, and adds a non-portable gate script. The already-posted ClawSweeper review's `skipWhenBusy` and gate findings are confirmed, with one additional prompt-policy blocker from this pass.

## What This Changes

The PR adds `false` support for default and per-agent heartbeat config and threads that disabled state through schemas, generated config, runtime and prompt helpers, WhatsApp heartbeat handling, tests, and a new gate script.

## Best Possible Solution

Keep the PR open and repair the branch by preserving the new `heartbeat: false` capability without weakening the existing heartbeat object contract. The maintainable end state is a schema/type/generated-schema update that keeps all current fields, explicit false handling in runner and system-prompt/bootstrap paths, focused regression tests for both default and per-agent false semantics, and no checked-in machine-local gate script.

## Reproduction Assessment

Yes. The `skipWhenBusy` regression is reproducible by comparing current main's accepted `heartbeat: { timeoutSeconds: 45, skipWhenBusy: true }` schema/test path with the PR's strict object branch that omits `skipWhenBusy`; the prompt regression is reproducible by tracing `agents.defaults.heartbeat: false` through the PR's system-prompt helper, where `undefined` falls back to the default cadence.

## Solution Assessment

No, not as written. Supporting `false` is the right direction, but the implementation should preserve all existing heartbeat object fields, make `false` a real disabled state in the prompt/bootstrap policy, and remove or replace the local-only gate before merge.
