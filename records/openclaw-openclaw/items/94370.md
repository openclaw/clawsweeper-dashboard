## Summary

Keep this issue open: current `main` still has a source-reproducible Docker rw sandbox path where the workspace bind is created before the read-only materialized skills overlay, and the nested `/workspace/.openclaw/sandbox-skills/skills` destination is not explicitly pre-created in the agent workspace.

## What This Changes

The issue reports that Docker can create `.openclaw/sandbox-skills` as `root:root` because OpenClaw does not pre-create the materialized skills mount destination before starting a rw sandbox container.

## Best Possible Solution

Add a narrow sandbox layout step that pre-creates the nested `.openclaw/sandbox-skills` destination under the writable agent workspace before Docker/browser container creation, while preserving the current state-backed materialized skills source.

## Reproduction Assessment

Yes, by source inspection: rw Docker sandboxes bind the agent workspace at `/workspace` and then bind a read-only materialized skills source to `/workspace/.openclaw/sandbox-skills/skills`, while current layout setup only creates the workspace root. I did not run a live Docker ownership repro in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow pre-creation step for the nested destination under the writable agent workspace, plus a regression test. Pre-creating only `skillsWorkspaceDir` is weaker because that path is the state-backed source that `syncSkillsToWorkspace` already prepares.
