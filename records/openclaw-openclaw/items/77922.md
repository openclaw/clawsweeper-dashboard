## Summary

Keep open: current main and the latest release still drop the Comfy local DNS-hostname case before building the private-network SSRF policy, so the reported private-DNS/LAN Comfy workflow remains source-reproducible.

## What This Changes

The issue asks for the bundled Comfy plugin's local/private-network mode to allow a configured DNS baseUrl that resolves to a private IP, not only IP-literal or localhost base URLs.

## Best Possible Solution

Preserve SSRF default-deny behavior, remove the DNS-hostname exclusion only for Comfy's existing explicit local/private-network opt-in, and add focused regression coverage for default-deny plus configured-host opt-in.

## Reproduction Assessment

Yes, by source inspection: a local Comfy config with a DNS baseUrl sets allowPrivateNetwork true, but resolveComfyNetworkPolicy returns no policy because the hostname is not an IP literal or localhost. I did not run a live LAN DNS plus ComfyUI setup.

## Solution Assessment

Yes as a direction: the maintainable fix is to honor the existing Comfy local/private-network opt-in while keeping the configured hostname allowlist, not to add a new config key or global SSRF bypass.
