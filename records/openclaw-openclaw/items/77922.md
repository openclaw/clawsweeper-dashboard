## Summary

Keep open: current main and v2026.6.10 still drop Comfy DNS hostnames before building the private-network SSRF policy, so the reported local-DNS path remains source-reproducible. The direct Comfy fix PR is closed unmerged, and the open xAI PR is adjacent provider work rather than a Comfy fix.

## What This Changes

The issue requests that the Comfy plugin allow explicit local/private-network access when `baseUrl` is a DNS hostname that resolves to a private IP.

## Best Possible Solution

Keep this open for a narrow Comfy plugin fix that allows explicitly trusted local DNS hostnames through a scoped SSRF policy, with regression coverage and maintainer/security acceptance.

## Reproduction Assessment

Yes, at source level. A DNS hostname such as `comfy.local.example.com` leaves `resolveComfyNetworkPolicy` with no policy even when local mode sets private-network intent, and the shared guard then rejects private DNS answers.

## Solution Assessment

Yes as a direction, but it needs maintainer/security acceptance. The maintainable fix is plugin-owned and should reuse a scoped SSRF policy rather than adding a new config key or broad private-network bypass.
