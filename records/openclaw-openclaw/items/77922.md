## Summary

Keep open: current main and v2026.6.8 still drop Comfy DNS hostnames before building the private-network SSRF policy, so the reported local-DNS path remains source-reproducible. The matching fix PR was closed unmerged, so it is useful context but not a safe superseding implementation.

## What This Changes

The issue asks for Comfy local mode/private-network opt-in to allow a configured DNS `baseUrl` that resolves to a private/LAN IP, not only IP literal hostnames.

## Best Possible Solution

Update the Comfy plugin to pass an origin-scoped private-network SSRF policy for explicit local/private `baseUrl` DNS hostnames, with regression coverage for DNS success, different-port blocking, and metadata/link-local rebinding blocking.

## Reproduction Assessment

Yes. Source inspection shows a DNS hostname such as `comfy.local.example.com` leaves `resolveComfyNetworkPolicy` with no policy even when local mode has private-network intent, and the shared fetch guard then rejects private DNS answers.

## Solution Assessment

Yes as a direction: the best fix is narrow and plugin-owned, reusing the existing exact-origin SSRF policy shape rather than adding a new config key or broad bypass. The final patch still needs maintainer/security review because it relaxes an SSRF allow path.
