## Summary

Keep open: current main and latest v2026.6.6 still build the Comfy private-network SSRF policy only when the configured hostname string is already private or loopback, so DNS hostnames that resolve to private IPs remain source-reproducible. The related broad media-provider PR is closed unmerged and is useful context, but it is not a superseding landing path.

## What This Changes

The issue reports that Comfy local mode with `allowPrivateNetwork: true` blocks DNS hostnames resolving to private IPs even though equivalent private IP literal base URLs work.

## Best Possible Solution

Update the Comfy plugin to use the existing origin-scoped SSRF policy for the configured local/private `baseUrl`, with regression coverage for private-DNS success, different-port blocking, and metadata/link-local rebinding blocking.

## Reproduction Assessment

Yes. Source inspection shows a DNS hostname such as `comfy.local.example.com` leaves `resolveComfyNetworkPolicy` with no policy even when local mode makes `allowPrivateNetwork` true, so the lower fetch guard can still block private DNS answers.

## Solution Assessment

Yes as a direction, but it needs security review. The narrow maintainable solution is to fix Comfy’s policy builder to reuse the existing configured-origin SSRF contract, not to add a new config option or broad private-network bypass.
