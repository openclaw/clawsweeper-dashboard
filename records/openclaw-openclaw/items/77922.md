## Summary

Keep open. Current main still has the reported Comfy local-mode SSRF gap, and the linked fix PR was closed unmerged rather than landing a current-main fix.

## What This Changes

The issue reports that Comfy local mode with `allowPrivateNetwork: true` blocks DNS hostnames that resolve to private/LAN IPs, while IP-literal LAN URLs work.

## Best Possible Solution

Patch the Comfy-owned network policy to build a hostname-scoped private-network policy for the configured `baseUrl` when local mode or explicit `allowPrivateNetwork` is active, with focused regression coverage and no core-wide SSRF default change.

## Reproduction Assessment

Yes. Source inspection on current main shows a Comfy local DNS `baseUrl` returns no private-network policy, then strict SSRF resolution blocks the private DNS answer; I did not run a live LAN Comfy setup.

## Solution Assessment

Yes. The narrow maintainable fix is in the Comfy policy builder: keep the configured hostname allowlist and allow private DNS answers only for that opted-in endpoint, rather than changing shared SSRF defaults.
