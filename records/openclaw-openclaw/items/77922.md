## Summary

Keep open: current main and latest v2026.6.6 still gate Comfy's private-network SSRF policy on an IP-literal/private-host classifier, so DNS hostnames that resolve to private IPs remain source-reproducible. The paired Comfy fix PR was closed unmerged, and the remaining fix touches an SSRF boundary that should get maintainer/security review.

## What This Changes

The issue requests that Comfy local mode honor `allowPrivateNetwork: true` for configured DNS hostnames whose DNS answers are private/LAN IPs.

## Best Possible Solution

Land a narrow Comfy plugin policy fix that lets explicit local/private opt-in apply to the configured DNS hostname while preserving hostname allowlisting and shared SSRF protections, with regression coverage and real Comfy/DNS proof.

## Reproduction Assessment

Yes, from source inspection: local mode sets `allowPrivateNetwork` true, DNS hostnames fail the Comfy `isPrivateOrLoopbackHost` precheck, and the shared SSRF guard then rejects private DNS answers. I did not run a live LAN DNS plus ComfyUI setup.

## Solution Assessment

Yes, the best fix shape is the narrow Comfy policy change: build the configured-host private-network policy when local or explicit opt-in is active, without broadening the shared SSRF guard. Because this changes an SSRF exception, it should receive maintainer/security review before automation or merge.
