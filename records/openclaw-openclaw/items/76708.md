## Summary

Keep open. Current `main` still has a credible source-level path where embedded Telegram tools are scoped to the sandbox/policy session key, and `session_status({sessionKey:"current"})` can still choose that stale default-direct row before the live `agent:main:main` row.

## What This Changes

Requests that `session_status({sessionKey:"current"})` in Telegram direct conversations report the live current session/model instead of a stale Telegram default-direct session.

## Best Possible Solution

Thread the live run session key separately from the sandbox/policy key, or otherwise make explicit `current` resolve to the live run identity before any stale default-direct requester fallback.

## Reproduction Assessment

Yes. The source path is high-confidence: build embedded tools with `sessionKey` set to `agent:main:telegram:default:direct:<id>` while the live run is `agent:main:main`, then call `session_status` with `sessionKey:"current"` against a store containing both rows.

## Solution Assessment

Yes. The narrow maintainable fix is to separate user-visible current-session identity from sandbox/tool-policy identity, with regression coverage for the stale Telegram default-direct row plus live main row.
