## Summary

Keep open. Current `main` still has a credible source-level path where embedded Telegram tools are constructed with the sandbox/runtime-policy session key, so `session_status({ sessionKey: "current" })` can resolve an existing default direct Telegram row instead of the live run session.

## What This Changes

The issue reports that `session_status({ sessionKey: "current" })` can identify a stale Telegram direct session and model instead of the live `agent:main:main` run.

## Best Possible Solution

Thread the live run session key separately from the sandbox/runtime-policy key so `session_status current` reports the same session/model as `/status` while sandbox visibility and tool policy remain keyed to the policy session.

## Reproduction Assessment

Yes, at source level. Build embedded tools with `sessionKey` set to a Telegram default direct sandbox key while the live run is `agent:main:main`, then call `session_status` with `sessionKey: "current"` against a store containing both entries; I did not execute a live Telegram run.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity and add a regression covering a stale Telegram direct row plus a live `agent:main:main` row.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76708. `session_status({ sessionKey: "current" })` in an embedded Telegram direct run must report the live run session, e.g. `agent:main:main`, not the sandbox/runtime-policy/default-direct session key. Keep `sandboxSessionKey` for sandbox resolution, tool policy, visibility restrictions, and process/tool isolation, but thread a separate live/current requester session key through the OpenClaw tool construction path used by `session_status`. Add focused regression coverage with a store containing a live `agent:main:main` GPT-5.5 entry and a stale/done `agent:main:telegram:default:direct:<id>` DeepSeek entry; when tools are built with the Telegram policy key but the live run key is main, `session_status current` should return `agent:main:main`. Also cover the embedded runner or factory wiring so future refactors cannot collapse the two keys again. Add a user-facing changelog fix entry. Do not weaken session visibility, agent-to-agent access, sandbox policy, or explicit default-direct lookups.
