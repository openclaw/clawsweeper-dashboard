## Summary

Keep open. Current main still has a credible source-level path where `session_status({ sessionKey: "current" })` follows the tool requester key, while embedded Telegram runs can build tools with the derived default-direct sandbox/policy key instead of the live `agent:main:main` run key.

## What This Changes

The issue reports that `session_status({ sessionKey: "current" })` in a Telegram direct conversation can report a stale default-direct policy session instead of the live main session.

## Best Possible Solution

Thread live current-session identity separately from sandbox/runtime-policy identity so `session_status current` reports the active run while policy enforcement continues to use the derived Telegram key.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: build `session_status` with `agentSessionKey` set to `agent:main:telegram:default:direct:<id>` while the store also contains live `agent:main:main`, then call it with `sessionKey: "current"`. I did not execute a live Telegram run in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to separate the live requester session key from the sandbox/policy key and add a regression test for explicit `current` with stale default-direct and live main rows.
