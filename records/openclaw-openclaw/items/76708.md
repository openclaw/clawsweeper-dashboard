## Summary

Keep open. Current main still has a credible source-level path where `session_status({sessionKey:"current"})` resolves through the Telegram sandbox/policy key instead of the live canonical run session key.

## What This Changes

The issue reports that `session_status({sessionKey:"current"})` in Telegram direct chats can report a stale channel session/model instead of the live `agent:main:main` run.

## Best Possible Solution

Thread the live run session key separately from the sandbox/tool-policy session key so `session_status current` reports the same session/model as `/status` while sandbox enforcement remains unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: construct tools for a live `agent:main:main` run with a Telegram default direct `sandboxSessionKey`, then call `session_status` with `sessionKey: "current"` against a store containing both rows.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity and add a regression test for a stale Telegram direct row plus a live `agent:main:main` row.
