## Summary

Keep open. Current main still has a credible source-level path where Telegram direct runs use a direct runtime-policy key to build tools, and `session_status({sessionKey:"current"})` can resolve through that requester key instead of the live `agent:main:main` run.

## What This Changes

The issue asks for `session_status({sessionKey:"current"})` in Telegram direct conversations to report the live current session/model rather than a stale Telegram direct runtime-policy session.

## Best Possible Solution

Thread the live run session key separately from the sandbox/runtime-policy key into tool construction, and make `session_status current` use that live key while policy and sandbox checks keep using the derived policy key.

## Reproduction Assessment

Yes, source-level: build `session_status` with `agentSessionKey` set to a Telegram default direct runtime-policy key while the store contains both that stale direct row and a live `agent:main:main` row, then call `{sessionKey:"current"}`. I did not execute a live Telegram run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity, with regression coverage for explicit `current` in a Telegram direct main-session run.
