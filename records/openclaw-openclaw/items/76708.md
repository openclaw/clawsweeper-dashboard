## Summary

Keep open. Current `main` still has a credible source-level path where `session_status({sessionKey:"current"})` follows the tool requester key, while embedded runs can build tools with the Telegram sandbox/policy key instead of the live canonical session key.

## What This Changes

The issue reports that `session_status({sessionKey:"current"})` in a Telegram direct conversation can report a stale direct-session model instead of the live `agent:main:main` run.

## Best Possible Solution

Thread the live run session key separately from the sandbox/policy key so `session_status current` reports the same current session/model as `/status` while sandbox and tool-policy enforcement remain unchanged.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: build tools with `sessionKey` set to a Telegram default direct sandbox key while the run session is `agent:main:main`, then call `session_status` with `sessionKey: "current"` against a store containing both entries. I did not execute a live Telegram run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity, with a regression test for a stale Telegram direct row plus a live `agent:main:main` row.
