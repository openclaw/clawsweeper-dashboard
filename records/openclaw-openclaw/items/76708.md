## Summary

Keep open. Current `main` still has a source-level path where explicit `session_status({sessionKey:"current"})` can fall back to the Telegram default direct requester key even when the live route session is `agent:main:main`.

## What This Changes

`session_status` should report the live current Telegram direct run session/model instead of a stale Telegram direct sandbox or policy session.

## Best Possible Solution

Keep the separate sandbox policy key, but pass or derive a distinct live current-session key for status/current identity so `/status`, `openclaw status`, and `session_status current` agree for Telegram direct runs.

## Reproduction Assessment

Yes, at source level: build `session_status` with requester `agent:main:telegram:default:direct:<id>`, keep both that stale row and a live `agent:main:main` row in the session store, then call with `sessionKey: "current"`. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity and add a regression test for the stale Telegram direct row plus live main row case.
