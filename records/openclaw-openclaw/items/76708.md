## Summary

Keep open. Current main still has a credible status-routing bug: embedded OpenClaw tools are built with the sandbox/default Telegram direct key as the requester identity, so an explicit `session_status({sessionKey:"current"})` can resolve the stale direct row before the live `agent:main:main` session.

## What This Changes

`session_status({sessionKey:"current"})` should report the live Telegram direct run on `agent:main:main` instead of a stale default-account direct session.

## Best Possible Solution

Thread the live run session key separately from the sandbox/policy key so `session_status current` reports the same current session/model as `/status`, with regression coverage for a stale Telegram direct row plus a live `agent:main:main` row.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: build tools with a Telegram default direct sandbox key while the session store contains both that row and a live `agent:main:main` row, then call `session_status({sessionKey:"current"})`; I did not run a live Telegram turn in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity, without changing Telegram route keys or startup-time session migration behavior.
