## Summary

Keep open. Current `main` still has a source-level path where `session_status({ sessionKey: "current" })` can resolve through the Telegram default direct sandbox/requester key instead of the live `agent:main:main` run session.

## What This Changes

The issue reports that `session_status({ sessionKey: "current" })` can report a stale Telegram direct session/model instead of the live `agent:main:main` run shown by `/status` and `openclaw status`.

## Best Possible Solution

Thread live run session identity separately from sandbox/tool-policy identity so `session_status current` reports the same session/model as `/status` without broadening visibility or changing sandbox policy.

## Reproduction Assessment

Yes, source-level. Build tools with `sessionKey` set to a Telegram default direct sandbox key while the live run session is `agent:main:main`, then call `session_status` with `sessionKey: "current"` against a store containing both entries.

## Solution Assessment

Yes. The narrow maintainable fix is to separate live current-session identity from sandbox/tool-policy identity, with regression coverage for a stale Telegram direct row plus a live `agent:main:main` row.

## Repair Work Prompt

Fix `session_status({ sessionKey: "current" })` so a Telegram direct run whose sandbox/tool-policy key is `agent:main:telegram:default:direct:<peer>` reports the live `agent:main:main` session when that is the actual run session. Keep sandbox and tool-policy decisions scoped to the sandbox key; only separate the user-visible current-session identity used by session tools/status. Add regression coverage for an explicit `sessionKey: "current"` call when the session store contains both a stale/completed `agent:main:telegram:default:direct:<peer>` row and a live `agent:main:main` row. Do not broaden session-tool visibility, add config, or change Telegram routing semantics. Add a single-line changelog fix entry for the user-facing behavior change.
