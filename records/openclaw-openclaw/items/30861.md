## Summary

The issue remains necessary on current main. The Dashboard breadcrumb now persistently identifies the active agent, but it still omits that agent’s configured workspace, and the separate workspace rail is hidden at widths of 1120px or less; the discussion’s warning about deriving the label from canonical backend target state is therefore relevant to the remaining implementation.

## What This Changes

Add an always-visible Dashboard indication of the active agent and workspace, synchronized across switches and reconnects.

## Best Possible Solution

Extend the existing Dashboard header context to include the active agent’s configured workspace, truncate it responsively with a full-path tooltip, and prove switch plus reconnect behavior without adding a parallel target store or new setting.

## Reproduction Assessment

Yes. Current main deterministically renders only the active agent in the persistent header, while workspace is confined to a rail hidden at viewport widths of 1120px or less.

## Solution Assessment

Yes, with a narrower scope than the full proposal. Extending the existing header from canonical session and agents.list state is preferable to adding message prefixes, another persisted setting, or a second source of routing truth.
