## Label Justifications

- `P2`: Most supporting infrastructure now exists, but common narrower Dashboard layouts still lack the requested persistent workspace safety context.
- `impact:session-state`: The remaining gap is ambiguity about which agent workspace the active Dashboard session represents after switching or reconnecting.
- `impact:message-loss`: Ambiguous active context can cause an instruction to be submitted to the wrong agent target.

## Summary

Keep open: current main now persistently identifies the active agent and provides switch feedback, but the Dashboard header remains agent-only and the separate workspace rail disappears at widths of 1120px or less. The discussion’s warning about proving the backend-selected agent/workspace tuple reinforces that the remaining responsive workspace label should derive from canonical session state, not a separate visual selection.

## What This Changes

Add an always-visible active-agent and workspace indicator to the Dashboard, synchronized with agent switching, reloads, and reconnects.

## Best Possible Solution

Add a compact, responsive workspace label beside the existing Dashboard agent context, sourced from the active session’s existing Gateway agent summary and retained when the workspace file rail is hidden.

## Reproduction Assessment

Yes at source level. The header context returns only an agent label, and responsive CSS deterministically removes the only visible workspace path at widths of 1120px or less.

## Solution Assessment

Yes, after narrowing the proposal: reuse the current agent header, canonical session-derived target, and existing workspace metadata rather than adding another switcher, persisted state, colors, or message-prefix configuration.
