## What This Changes

The PR adds a shared cron expression humanizer and uses it in the Control UI Cron Jobs and Dreaming phase schedule displays.

## Best Possible Solution

Keep the Cron Jobs formatter only if maintainers want that UI enhancement, but revise or drop the Dreaming phase schedule wording until it reflects the single managed sweep contract or a real per-phase scheduling API exists.

## Reproduction Assessment

Yes for the PR-introduced Dreaming display problem: current source sets all phase cron strings and next-run status from the one managed sweep, and the screenshot shows identical phase schedule labels. No high-confidence live repro exists for the original linked report's `*/6` rendering as `3:00` on current main.

## Solution Assessment

No for the Dreaming half of the PR: a shared cron formatter is narrow for Cron Jobs, but using the unified sweep cron as per-phase copy does not solve the claimed distinguishability problem. The safer direction is to keep Dreaming copy tied to sweep status unless real per-phase schedule data is introduced.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep the sweep cron out of per-phase labels:** `ui/src/ui/views/dreaming.ts:451`
  - body: Current status data gives light, deep, and REM the same `dreaming.frequency` cron and the same managed `nextRunAtMs`, so this call can render all phase cards with identical text such as `Daily at 3:00 AM · next 3:00 AM`. That does not fix the claimed distinguishability problem and makes the one managed sweep look like three independent phase schedules; keep the old next-run/status copy here or add a real per-phase schedule contract first.
  - confidence: 0.9
- **[P3] Guard fixed-minute hourly cron labels:** `ui/src/ui/format.ts:162-164`
  - body: The `hour === "*"` branch humanizes any numeric minute without the 0-59 guard used by the daily and weekly branches, so a malformed persisted expression like `60 * * * *` would display as `Every hour at :60` instead of falling back to raw cron text. Add the same minute range check and a regression assertion.
  - confidence: 0.78
