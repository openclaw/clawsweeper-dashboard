## Summary

Keep open: current main still has a source-visible mismatch where private-LAN QR setup is accepted but the iOS bootstrap handoff persistence gate only trusts `wss` or loopback, so the reported token-loss and disconnected chat flow remains actionable.

## What This Changes

The issue reports that iOS QR pairing can approve a LAN Gateway node but lose or fail to use the post-pairing Gateway/operator token, leaving mobile chat disconnected.

## Best Possible Solution

Align iOS bootstrap handoff token persistence with the existing private-LAN/.local setup-code trust boundary, add regression coverage for node plus operator token persistence, and keep public/tailnet plaintext fail-closed.

## Reproduction Assessment

Yes, at source level: private-LAN QR setup is accepted, iOS clears old tokens before pairing, and current main persists bootstrap handoff tokens only for `wss` or loopback. I did not establish a live iPhone reproduction in this review.

## Solution Assessment

No, current main is not the best state because the setup-code trust predicate and the handoff persistence predicate disagree for private-LAN QR pairing. The narrow maintainable fix is to reuse the established local-network trust boundary for persistence and cover the iOS reconnect/chat loop.
