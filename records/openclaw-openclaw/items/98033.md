## Summary

Keep open: current main still appears to accept private-LAN plaintext setup codes while the Swift client only persists QR/bootstrap handoff tokens on `wss://` or loopback, matching the reported LAN HTTP token-loss and disconnected-chat path. The related merged PRs repaired adjacent LAN, QR handoff, and scope issues but do not prove this private-LAN persistence case fixed.

## What This Changes

The issue requests that iOS QR pairing over LAN HTTP persist the gateway/operator token and keep mobile chat connected after pairing.

## Best Possible Solution

Confirm the trusted LAN bootstrap contract, then repair Swift token persistence and focused Swift/docs tests plus live iOS/Safari LAN proof without weakening public-host plaintext fail-closed behavior.

## Reproduction Assessment

No live reproduction was run, but the current source gives a medium-confidence reproduction path: pair iOS over private LAN plaintext and the Swift handoff persistence gate skips tokens that docs and setup-code parsing otherwise allow.

## Solution Assessment

Unclear until the trust boundary is confirmed; the narrow likely fix is to align handoff-token persistence with the documented local-network setup-code predicate while keeping public plaintext hosts fail-closed.
