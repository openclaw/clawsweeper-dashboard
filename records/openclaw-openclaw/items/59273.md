## Summary

Keep open. Current main still reuses ACP cached runtime handles without comparing active runtime config or restart generation, and Discord queued jobs still carry the preflight `cfg` snapshot into later processing. The related #56597 remains open for persisted `/exec` override precedence, but this report describes a distinct ACP/runtime-cache and queued-config restart boundary.

## What This Changes

The issue requests that Discord-bound ACP sessions stop reusing stale runtime handles or old exec configuration after gateway restarts and `tools.exec`/`safeBins` config changes.

## Best Possible Solution

Add a generic ACP freshness boundary, preferably a stable runtime config or restart-generation fingerprint stored in `CachedRuntimeState` and compared in `ensureRuntimeHandle()`, or an explicit ACP manager cache reset at gateway/plugin restart. Separately, keep `/exec` persisted override precedence aligned with #56597 rather than solving that policy decision in Discord-only code.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: keep a Discord-bound ACP session cached, change exec-related runtime config, restart through the in-process boundary, then process a turn whose cached handle still satisfies the current reuse checks; queued Discord jobs can also carry the old `cfg` snapshot into dispatch.

## Solution Assessment

Unclear. The report’s fingerprint and forced-reinitialize options are both plausible, but the best maintainable fix should be a generic ACP lifecycle/cache contract plus focused Discord queued-config handling only if jobs can survive the restart boundary with old policy.
