## Summary

Keep open. Current main still reuses cached ACP runtime handles without a runtime-config or restart-generation freshness check, and Discord queued jobs still preserve the preflight config snapshot; the related open report covers a separate persisted `/exec` override precedence path.

## What This Changes

Ensure Discord ACP sessions refresh cached runtime handles and queued config after gateway restarts so updated `tools.exec.security` and `safeBins` take effect.

## Best Possible Solution

Define a generic ACP runtime freshness boundary, then compare a stable config or restart-generation fingerprint in cached runtime state and refresh Discord queued config only where jobs can cross that boundary.

## Reproduction Assessment

Yes, at source level. Keep a Discord-bound ACP session cached, change exec-related config, restart through the in-process gateway boundary, and the current reuse checks can still accept the old runtime handle while queued Discord jobs can retain the old `cfg` payload.

## Solution Assessment

Unclear. The report's fingerprint and forced-reinitialize options are plausible, but the maintainable fix should be a generic ACP cache lifecycle contract coordinated with the separate persisted `/exec` precedence discussion at https://github.com/openclaw/openclaw/issues/56597.
