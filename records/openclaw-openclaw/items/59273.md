## Summary

Keep open. Current main still reuses ACP runtime handles without comparing active runtime config or a restart generation, and Discord queued jobs still carry the preflight config snapshot into later processing; the related exec override issue remains open but tracks a distinct persisted-session precedence problem.

## What This Changes

The issue reports that Discord ACP sessions can retain stale exec/security configuration across gateway restarts instead of honoring updated `tools.exec` and `safeBins` settings.

## Best Possible Solution

Define a generic ACP runtime freshness boundary, then store and compare a stable runtime-config or restart-generation fingerprint in cached runtime state, with Discord queue handling adjusted only if jobs can still cross a restart with old policy.

## Reproduction Assessment

Yes, at source level. Keep a Discord-bound ACP session cached, change exec-related runtime config, restart through the in-process SIGUSR1 boundary, then process a later turn whose cached handle still satisfies the current reuse checks; queued Discord jobs can also carry the old `cfg` snapshot into dispatch.

## Solution Assessment

Unclear. The report's fingerprint and forced-reinitialize options are both plausible, but the maintainable fix should be a generic ACP cache lifecycle contract coordinated with the separate persisted `/exec` precedence discussion at https://github.com/openclaw/openclaw/issues/56597.
