## Summary

Keep open: current main still has provider-response and replay paths that can throw on non-array assistant `content`, and both repair PRs linked from the issue were closed unmerged.

## What This Changes

The issue asks OpenClaw to stop repeated 500 `v.content is not iterable` failures when a GLM-4-Air/OpenAI-compatible provider returns malformed assistant content during Feishu agent runs.

## Best Possible Solution

Add provider-agnostic malformed assistant-content normalization before Responses output conversion and before replayed assistant turns reach transport transforms, with regression coverage for string, object, null, and undefined shapes.

## Reproduction Assessment

Yes from source inspection, not from live provider verification. A message output item with object-shaped `content` can hit `.some(...)`, and replayed assistant turns with object/null `content` can pass through normalization into `for...of`.

## Solution Assessment

Yes. The narrowest maintainable fix is generic hardening at the provider-response conversion and replay/transport boundaries, not a GptGod-specific workaround or broad transcript migration.

## Repair Work Prompt

Fix issue #43795 by hardening malformed assistant `content` handling in the agent/provider replay paths. Normalize provider-supplied assistant content before `buildAssistantMessageFromResponse` uses `.some` or `for...of`, and normalize/drop/placeholder replayed assistant turns with object, null, or undefined `content` before they reach `transformTransportMessages` or transport-specific conversion. Keep this provider-agnostic; do not add GptGod-specific policy, new config, broad transcript migration, or unrelated gateway changes unless a focused failing test proves that path. Add regression tests for string, single object, null, and undefined content shapes, preserve existing well-formed array behavior, and add a single-line user-facing `CHANGELOG.md` fix entry.
