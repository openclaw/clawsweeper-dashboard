## Summary

Keep open: current main still has provider-agnostic Responses and replay paths that can read or iterate malformed assistant `content` before proving it is array-shaped, and the two linked repair PRs for this report were closed unmerged.

## What This Changes

The issue reports repeated embedded-agent 500 errors when a third-party OpenAI-compatible `glm-4-air` provider returns malformed assistant `content`.

## Best Possible Solution

Add a generic malformed assistant-content normalization boundary before OpenAI Responses output conversion and before replayed assistant turns reach transport transforms, with regression tests for string, object, null, and undefined shapes.

## Reproduction Assessment

Yes. A focused unit can pass a Responses output message with object-shaped `content` into `buildAssistantMessageFromResponse` to hit `.some(...)`, or replay an assistant turn with object/null `content` through `normalizeMessagesForLlmBoundary` and then `transformTransportMessages` to hit `for...of`.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-agnostic hardening at the Responses conversion and replay normalization boundaries, not a GptGod-specific workaround or a broad transcript storage migration.
