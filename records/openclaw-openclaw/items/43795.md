## Summary

Keep open: current main still has source-level crash paths for malformed assistant `content`, and the two linked repair PRs for this report were closed unmerged.

## What This Changes

The issue reports repeated `500 v.content is not iterable` failures when a third-party OpenAI-compatible GLM provider returns malformed assistant content during embedded Feishu agent runs.

## Best Possible Solution

Add provider-agnostic malformed assistant-content normalization before Responses output conversion and before replayed assistant turns reach transport transforms, with regression coverage for string, object, null, and undefined shapes.

## Reproduction Assessment

Yes from source inspection. A Responses output message with object-shaped `content` can hit `.some(...)`, and a replayed assistant turn with object/null `content` can pass through replay normalization into `for...of`.

## Solution Assessment

Yes. The narrowest maintainable solution is generic hardening at the Responses conversion and replay normalization boundaries, not a GptGod-specific workaround or broad transcript migration.
