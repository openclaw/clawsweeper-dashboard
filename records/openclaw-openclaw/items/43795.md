## Summary

Keep open. Current main still has provider-agnostic Responses and replay paths that can read or iterate assistant `content` before proving it is array-shaped, and the two linked repair PRs for this report were closed unmerged.

## What This Changes

This issue reports repeated `500 v.content is not iterable` embedded-agent failures when an OpenAI-compatible provider returns assistant `content` in a non-array shape.

## Best Possible Solution

Add a generic malformed assistant-content normalization boundary before OpenAI Responses output and replayed assistant turns are inspected. Preserve valid arrays, convert strings into text blocks where replayable, wrap only safe block-shaped objects, and drop or sentinel null/unrecognized values according to existing error-turn policy.

## Reproduction Assessment

Yes. A focused unit can pass a Responses output message with object-shaped `content` into `buildAssistantMessageFromResponse`, or replay an assistant turn with object/null `content`, and current main reaches `.some(...)` or `for...of` without proving the value is an array.

## Solution Assessment

Yes. The narrowest maintainable solution is provider-agnostic hardening at the Responses conversion and replay normalization boundaries, with regression tests for string, object, null, and undefined shapes instead of a GptGod-specific workaround.
