## Best Possible Solution

Yes, from source inspection. Mock `bot.api.setMyCommands` to reject with a grammY/Telegram 429 carrying `parameters.retry_after`; current main reaches the non-`BOT_COMMANDS_TOO_MUCH` throw path and logs `Telegram command sync failed` instead of waiting and retrying.

## Solution Assessment

No, not as currently rebased. The retry direction is right, but the implementation must wrap the current scoped command-registration helper so both default and all-group menu syncs preserve the same retry behavior.
