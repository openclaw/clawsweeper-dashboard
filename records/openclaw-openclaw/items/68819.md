## Summary

Keep open: the Windows claude-cli spawn regression is still present on current main, and this PR is aimed at the right surface, but the latest head is not merge-ready because its resolver can diverge from Node/libuv command selection and it still needs current-main integration validation.

## What This Changes

This PR adds a Windows `.cmd` shim argv resolver for supervisor child spawns and unit tests covering `.cmd` to `.exe`/`.js` rewrites, PATH lookup, and non-Windows fallthrough.

## Best Possible Solution

Land a rebased process fix that resolves npm-style Windows `.cmd` shims for supervisor spawns without shelling out, while matching Node's env/PATH and `.com`/`.exe` lookup semantics and preserving current child-adapter OOM/output handling.

## Reproduction Assessment

Yes. The reporter supplied Windows 11 steps and logs, and current main still routes `claude-cli` through bare `claude` while the supervisor child adapter lacks a `.cmd` shim-to-entrypoint resolver for that command path.

## Solution Assessment

No, not as written. Shim unwrapping is the right direction, but the implementation should first match Node/libuv command selection, add the required changelog entry, and be validated on a current-main merge result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Select PATH the same way Node will:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` walks `Object.entries` in insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawning. With duplicate `PATH`/`Path` keys, OpenClaw can inspect a shim from one PATH while the child receives another, so sort and dedupe PATH keys the same way before resolving.
  - confidence: 0.88
- **[P2] Honor .com before cmd shim resolution:** `src/process/windows-command.ts:86-87`
  - body: For extensionless Windows spawns, libuv checks `.com` and then `.exe`, and it does not resolve `.cmd` without a shell. This branch only checks for `<name>.exe` before selecting `<name>.cmd`, so a `tool.com` on PATH can be bypassed in favor of a shim target.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `src/process/supervisor/adapters/child.ts:38-40`
  - body: This wires a user-visible Windows fix, but the PR only changes process code and tests. OpenClaw policy requires user-facing `fix` changes to include a single-line `CHANGELOG.md` entry, so the release notes would miss the claude-cli Windows repair.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff avoids shelling through `cmd.exe`, but the new resolver can still choose a different Windows executable than Node/libuv would spawn.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong binary:** `src/process/windows-command.ts:26`
  - body: Because the resolver uses insertion order while Node sorts duplicate Windows env keys, an env object with both `PATH` and `Path` can make OpenClaw resolve a shim from a different PATH than the spawned child actually receives.
  - confidence: 0.84
- **[low] .com precedence is not preserved:** `src/process/windows-command.ts:86`
  - body: The extensionless-command branch only checks `.exe` before rewriting to `.cmd`, but libuv tries `.com` before `.exe`; this can bypass a direct `.com` executable and run a shim target instead.
  - confidence: 0.8
