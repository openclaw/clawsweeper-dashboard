## Summary

Keep this PR open: the Windows Claude CLI spawn bug is still source-reproducible on current main and this PR is the canonical fix path, but the current patch still has command-resolution correctness/security blockers and lacks inspectable real Windows proof.

## What This Changes

The PR adds a Windows supervisor helper that rewrites npm-style `.cmd` shims to underlying `.exe` or `node <script.js>` targets and extends process command tests.

## Best Possible Solution

Land a rebased core process fix that unwraps npm-style Windows `.cmd` shims without shelling out, while matching Node/libuv command selection and adding focused regression coverage plus redacted Windows runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main registers `claude-cli` as bare `claude`, sends it through the supervisor child adapter, and that adapter only suffixes npm/pnpm/yarn/npx. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style shims is the right direction, but the resolver must match Node's Windows env-key handling and libuv direct-spawn precedence before this command-execution patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH de-duplication:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` scans env entries in insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawn. With both `Path` and `PATH`, this can inspect a different PATH than the child actually receives, causing wrong-binary execution or missed shim resolution.
  - confidence: 0.88
- **[P2] Preserve direct-spawn precedence before .cmd fallback:** `src/process/windows-command.ts:85-89`
  - body: The extensionless branch only checks for `<name>.exe` on PATH before unwrapping `<name>.cmd`. libuv checks the current directory and `.com` before `.exe`, and direct spawn never uses `.cmd`, so this can unwrap a shim when `tool.com` or a CWD `tool.exe` is the program Node would have executed.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff touches command execution and can select a different Windows program than Node/libuv direct spawn would select.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong command:** `src/process/windows-command.ts:26`
  - body: The resolver scans env keys in insertion order while Node sorts and deduplicates Windows env keys before spawn, so OpenClaw can inspect one PATH while the child receives another.
  - confidence: 0.86
- **[medium] Direct-spawn precedence is not fully preserved:** `src/process/windows-command.ts:85`
  - body: The `.cmd` fallback does not rule out current-directory executables or `.com` candidates that libuv would prefer, so a shim target can run instead of the command Node would execute.
