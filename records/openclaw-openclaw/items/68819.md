## Summary

Keep open: current main still has the Windows `claude-cli` supervisor spawn bug and this PR is the canonical active fix path, but the proposed command-resolution patch still has blocking Windows command-selection issues and lacks inspectable real behavior proof.

## What This Changes

This PR adds a Windows supervisor helper that rewrites npm-style `.cmd` shims to an underlying `.exe` or `node <script.js>` target and adds unit coverage in `src/process/windows-command.test.ts`.

## Best Possible Solution

Land a rebased core process fix that unwraps npm-style Windows `.cmd` shims for Claude CLI without shelling out, while matching Node/libuv command selection and adding focused regression coverage plus redacted Windows runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main registers `claude-cli` as `claude`, sends that argv through the supervisor child adapter, and that adapter only suffixes npm/pnpm/yarn/npx. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style shims is the right direction, but the resolver must match Node env-key handling and libuv direct-spawn precedence before this command-execution patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH key selection:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` scans `Object.entries` in insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawn. With both `Path` and `PATH`, this can inspect a different PATH than the child actually receives, causing wrong-binary execution or missed shim resolution.
  - confidence: 0.88
- **[P2] Preserve direct-spawn candidates before .cmd fallback:** `src/process/windows-command.ts:85-89`
  - body: The extensionless branch only checks for `<name>.exe` on PATH before unwrapping `<name>.cmd`. libuv checks the current directory and `.com` before `.exe`, and direct spawn never uses `.cmd`, so this can unwrap a shim when `tool.com` or a CWD `tool.exe` is the program Node would have executed.
  - confidence: 0.84

## Security Review
