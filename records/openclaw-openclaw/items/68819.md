## What This Changes

The PR adds Windows supervisor child-spawn logic to resolve `.cmd` shims to an underlying `.exe` or `node.exe <cli.js>` before spawning, with focused resolver tests.

## Best Possible Solution

Land a rebased process-layer fix that reuses or moves the existing Windows spawn resolver into the supervisor path, preserves current child-adapter protections, handles cwd/env lookup consistently, and includes focused tests plus redacted Windows runtime proof.

## Reproduction Assessment

Yes. Current main still routes bare `claude` through the supervisor child adapter without a `claude` shim allowlist entry, and the related issue logs match Node's documented Windows `.cmd` execution limitation.

## Solution Assessment

No. The no-shell shim-to-entrypoint direction is right, but this patch should not introduce a second resolver with different lookup semantics when current main already has a shared Windows spawn resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Reuse the shared Windows spawn resolver:** `src/process/windows-command.ts:84-89`
  - body: Current main already has `resolveWindowsSpawnProgram` for PATH/PATHEXT lookup and `.cmd` wrapper entrypoint policy, and multiple runtime call sites use it. This PR adds a second resolver with different lookup order and fail-open behavior, so supervisor spawns can execute a different program than ACP/Codex/LSP paths for the same command.
  - confidence: 0.86
- **[P2] Mirror Node PATH lookup for custom envs:** `src/process/windows-command.ts:24-31`
  - body: `resolveWindowsPathEnv({ FOO: "bar" })` returns an empty PATH, so a supervisor child with a sanitized env that omits PATH will skip `.cmd` unwrapping even though Node's command lookup on Windows still falls back to the parent PATH for this case. That leaves the reported Windows spawn failure in place for custom-env runs.
  - confidence: 0.78
- **[P2] Resolve relative shims from the child cwd:** `src/process/windows-command.ts:75-76`
  - body: When `argv[0]` contains a relative path such as `node_modules/.bin/tool.cmd`, this resolver reads it relative to the gateway process cwd because no child `cwd` is passed in. `spawn` resolves that path relative to `options.cwd`, so the helper can miss the intended shim or inspect a same-named file in the wrong directory.
  - confidence: 0.74
