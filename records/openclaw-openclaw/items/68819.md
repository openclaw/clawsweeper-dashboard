## Summary

Keep open. Current main still lacks a working Windows `.cmd` shim-to-target resolver for the `claude-cli` supervisor spawn path, so the PR addresses a real regression, but this head is not ready to merge because its resolver can choose a different executable than Node/libuv would and the live PR is currently unmergeable against main.

## What This Changes

This PR adds a Windows `.cmd` shim argv resolver, wires supervisor child spawns through it, and adds unit tests for `.cmd` to `.exe`/`.js` rewrites, PATH lookup, and fallthrough cases.

## Best Possible Solution

Land a rebased core process fix that resolves npm-style Windows `.cmd` shims for supervisor child spawns while matching Node/libuv Windows command lookup semantics, preserving current child-adapter OOM and output-decoding behavior, and covering the `claude-cli` path with focused tests.

## Reproduction Assessment

Yes. Current main still invokes bare `claude` for `claude-cli`, and the child adapter has no resolver for that `.cmd` shim path; Node documentation/source also confirms `.cmd` files are not directly executable without a shell on Windows.

## Solution Assessment

No, not as written. Resolving npm-style `.cmd` shims to direct executable/Node invocations is the right direction, but this implementation must first match Node's Windows PATH-key selection and libuv `.com`/`.exe` precedence before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Select PATH the same way Node will:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` walks env entries in insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate. With duplicate `PATH`/`Path` values, this resolver can parse a shim from one PATH while the spawned child uses another, leading to wrong-binary execution. Sort and dedupe PATH keys the same way Node does before resolving shims.
  - confidence: 0.88
- **[P2] Preserve .com precedence before cmd shim resolution:** `src/process/windows-command.ts:86-87`
  - body: For extensionless Windows spawns, libuv tries `.com` and then `.exe`, and does not resolve `.cmd` without a shell. This branch checks only for `<name>.exe` before selecting `<name>.cmd`, so a direct `tool.com` on PATH can be bypassed in favor of a shim target. Treat `.com` as a higher-precedence direct executable too.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff avoids `cmd.exe`, but its custom command resolver can diverge from Node's Windows lookup and execute an unintended PATH target.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong binary:** `src/process/windows-command.ts:26`
  - body: Because the resolver uses insertion order while Node sorts duplicate Windows env keys, a merged env object with both `PATH` and `Path` can make OpenClaw resolve a shim from a different PATH than the child process actually receives.
  - confidence: 0.84
- **[low] .com precedence is not preserved:** `src/process/windows-command.ts:86`
  - body: The extensionless-command branch only checks `.exe` before rewriting to `.cmd`, but libuv tries `.com` before `.exe`; this can bypass a direct `.com` executable and run a shim target instead.
  - confidence: 0.8
