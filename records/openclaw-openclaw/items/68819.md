## Summary

Keep open: current main still has the Windows `claude-cli` supervisor spawn gap, and this PR is the canonical fix path, but the patch is not safe to merge as written because its command resolver can diverge from Node/libuv Windows command selection and the contributor still has not provided inspectable after-fix real behavior proof.

## What This Changes

This PR adds Windows supervisor logic to unwrap npm-style `.cmd` shims to underlying `.exe` or `node <script.js>` targets and adds focused unit coverage in `src/process/windows-command.test.ts`.

## Best Possible Solution

Land a rebased core process fix that unwraps npm-style Windows `.cmd` shims for Claude CLI without shelling out, while matching Node/libuv command selection and preserving current child-adapter OOM/output handling.

## Reproduction Assessment

Yes, source-reproducible: current main registers `claude-cli` as bare `claude`, routes it through the supervisor child adapter, and that adapter only suffixes npm/pnpm/yarn/npx. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style shims is the right direction, but the resolver must match Node env-key handling and libuv direct-spawn precedence before this command-execution patch is safe to land.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH key selection:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` scans `Object.entries` in insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawn. With both `Path` and `PATH`, this can inspect a different PATH than the child actually receives, causing wrong-binary execution or missed shim resolution.
  - confidence: 0.88
- **[P2] Preserve libuv direct-spawn candidates before .cmd fallback:** `src/process/windows-command.ts:85-89`
  - body: The extensionless branch only checks for `<name>.exe` on PATH before unwrapping `<name>.cmd`. libuv checks the current directory and `.com` before `.exe`, and it never uses `.cmd` for direct spawn, so this can unwrap a shim when `tool.com` or a CWD `tool.exe` is the program Node would have executed.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff avoids shelling through `cmd.exe`, but the new resolver can still select a different Windows program than Node/libuv would spawn.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong command:** `src/process/windows-command.ts:26`
  - body: The resolver scans env keys in insertion order while Node sorts and deduplicates Windows env keys before spawn, so OpenClaw can inspect one PATH while the child receives another.
  - confidence: 0.86

  - body: The `.cmd` fallback does not rule out CWD executables or `.com` candidates that libuv would prefer, so a shim target can be run instead of the command Node would execute.
