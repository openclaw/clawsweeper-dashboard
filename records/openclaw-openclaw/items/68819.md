## Summary

Keep open: the Windows Claude CLI spawn bug is real and still present, but this PR is not merge-ready because proof is not inspectable, the branch conflicts with current process-adapter work, and it duplicates command-resolution policy in a security-sensitive spawn path.

## What This Changes

The PR adds a Windows supervisor-child shim resolver that unwraps `.cmd` shims to `.exe` or `node.exe <cli.js>` and adds unit coverage in `src/process/windows-command.test.ts`.

## Best Possible Solution

Land a rebased process-layer fix that reuses or moves the shared Windows spawn resolver into supervisor child spawns, preserves current child-adapter protections, and includes focused tests plus inspectable Windows proof.

## Reproduction Assessment

Yes. Source inspection shows current main still spawns the Anthropic CLI as bare `claude` through a child adapter that does not resolve that command, and a 2026-05-22 Windows report confirms ENOENT/EINVAL on release 2026.5.20.

## Solution Assessment

No. The direct executable or Node entrypoint direction is right, but this branch should reuse or move the existing Windows spawn resolver instead of adding a second policy in the supervisor path.

Full review comments:

- **[P1] Reuse the shared Windows spawn resolver:** `src/process/windows-command.ts:84-89`
  - body: Current main already has `resolveWindowsSpawnProgram` with PATH/PATHEXT lookup and wrapper-entrypoint policy. This new resolver scans and fails open differently, so supervisor spawns can execute or reject a different program than ACP, Codex, LSP, sandbox, and memory-host paths for the same command.
  - confidence: 0.86
- **[P2] Match the existing PATH fallback policy:** `src/process/windows-command.ts:24-31`
  - body: `resolveWindowsPathEnv({ FOO: "bar" })` returns an empty PATH, while the existing shared resolver falls back to the parent PATH for missing PATH values. Custom-env supervisor runs can therefore skip shim unwrapping and keep the same Windows ENOENT/EINVAL failure this PR is meant to fix.
  - confidence: 0.78
- **[P2] Resolve relative shims from the child cwd:** `src/process/windows-command.ts:75-76`
  - body: When `argv[0]` is a relative path such as `node_modules/.bin/tool.cmd`, this resolver reads it relative to the gateway process cwd because `params.cwd` is not passed in. `spawn` resolves that path relative to `options.cwd`, so the helper can inspect the wrong file or miss the intended shim.
  - confidence: 0.74
