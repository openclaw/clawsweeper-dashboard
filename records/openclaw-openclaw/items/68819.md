## Summary

Keep open: current main still has a source-reproducible Windows claude-cli spawn gap, but this PR is not merge-ready because after-fix proof is not inspectable and the resolver can diverge from Node/libuv command selection.

## What This Changes

The PR adds a Windows supervisor resolver that unwraps npm-style `.cmd` shims to `.exe` or `node <script.js>` targets and wires it into child process spawning with unit coverage.

## Best Possible Solution

Land a rebased supervisor fix that unwraps Claude CLI/npm-style Windows shims without shelling out while preserving Node/libuv command selection and current child-adapter OOM/output behavior.

## Reproduction Assessment

Yes, source-reproducible: current main registers `claude-cli` as bare `command: "claude"`, and the child adapter only suffixes npm/pnpm/yarn/npx before spawn. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style shims is the right direction, but the resolver must match Node env-key handling and libuv direct-spawn precedence before this command-execution patch is safe to land.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH selection:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` scans `Object.entries` insertion order, but Node sorts Windows env keys and deduplicates case-insensitive duplicates before spawn. With both `Path` and `PATH`, this can inspect a different PATH than the child receives, causing wrong-binary execution or missed shim resolution.
  - confidence: 0.88
- **[P2] Preserve libuv direct-spawn candidates before .cmd fallback:** `src/process/windows-command.ts:85-89`
  - body: The extensionless branch only checks for `<name>.exe` on PATH before unwrapping `<name>.cmd`. libuv checks the current directory and `.com` before `.exe`, and it never uses `.cmd` for direct spawn, so this can run a shim target when `tool.com` or a CWD `tool.exe` is what Node would have executed.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff avoids `cmd.exe`, but its command-resolution logic can still choose a different Windows program than Node/libuv would spawn.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong command:** `src/process/windows-command.ts:26`
  - body: The resolver scans env keys in insertion order while Node sorts and deduplicates Windows env keys before spawn, so OpenClaw can inspect one PATH while the child receives another.
  - confidence: 0.86
- **[medium] Direct-spawn precedence is not fully preserved:** `src/process/windows-command.ts:85`
  - body: The `.cmd` fallback does not rule out CWD executables or `.com` candidates that libuv would prefer, so a shim target can be run instead of the command Node would execute.
  - confidence: 0.82
