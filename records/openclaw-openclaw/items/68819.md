## Summary

Keep open. Current main still has the Windows Claude CLI spawn failure path, and this PR is the canonical active fix path, but the patch is not merge-ready because it lacks inspectable real behavior proof and its resolver can choose a different Windows program than Node/libuv would spawn.

## What This Changes

The PR adds Windows supervisor logic to unwrap npm-style `.cmd` shims to an underlying `.exe` or `node <cli.js>` target before child-process spawn, with focused resolver tests.

## Best Possible Solution

Land a rebased supervisor fix that unwraps npm-style Windows `.cmd` shims for Claude CLI without shelling out, while matching Node/libuv command selection and preserving current child-adapter process handling.

## Reproduction Assessment

Yes, source-reproducible: current main registers `claude-cli` as bare `claude`, sends it through the supervisor child adapter, and only suffixes npm/pnpm/yarn/npx before spawn. I did not run a native Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style shims is the right direction, but the resolver must match Node's Windows env-key handling and libuv's direct-spawn precedence before this command-execution patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH key selection:** `src/process/windows-command.ts:24-28`
  - body: `resolveWindowsPathEnv` scans insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawning. With both `Path` and `PATH`, this resolver may inspect a different PATH than the child receives, so it can unwrap or miss a different shim.
  - confidence: 0.88
- **[P2] Preserve libuv direct-spawn candidates before .cmd fallback:** `src/process/windows-command.ts:84-89`
  - body: The extensionless branch checks only `<name>.exe` on PATH before unwrapping `<name>.cmd`. libuv checks the current directory and `.com` before `.exe`, and does not use `.cmd` for direct spawn, so this can run the shim target when Node would have launched another binary.
  - confidence: 0.85
