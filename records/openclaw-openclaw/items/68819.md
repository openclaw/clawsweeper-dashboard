## Summary

This PR should stay open as the canonical fix path: current main still has the Windows `claude-cli` supervisor spawn gap, but the branch is conflicted and still has command-selection issues in the new Windows resolver.

## What This Changes

The PR adds a Windows supervisor resolver that unwraps `.cmd` shims to underlying `.exe` or `node <script.js>` targets before spawning them.

## Best Possible Solution

Land a rebased core process fix that resolves npm-style Windows `.cmd` wrappers on supervisor child spawns without shelling out, while matching Node/libuv command selection and preserving current OOM/output handling.

## Reproduction Assessment

Yes, source-reproducible: current main sends bare `claude` through the supervisor child path, and that path neither allowlists `claude` nor unwraps `.cmd` shims. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style shims is the right direction, but the resolver must match Node/libuv env and direct-spawn lookup semantics before this command-execution patch is safe to land.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH key selection:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` returns the first case-insensitive PATH key in insertion order, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawn. With both `Path` and `PATH`, the resolver can inspect a shim from one PATH while the child receives another, leading to wrong-binary execution or missed shim resolution.
  - confidence: 0.88
- **[P2] Preserve libuv direct-spawn candidates before .cmd fallback:** `src/process/windows-command.ts:85-89`
  - body: The extensionless branch falls back to `<name>.cmd` after checking only for `<name>.exe` on PATH. libuv checks the current directory and `.com` before `.exe`, and it never uses `.cmd` for direct spawn, so this can unwrap a shim when `tool.com` or a CWD `tool.exe` is the program Node would have executed.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch avoids shelling through `cmd.exe`, but the new resolver can still select a different Windows executable than Node/libuv would spawn.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong command:** `src/process/windows-command.ts:26`
  - body: Because the resolver scans env keys in insertion order while Node sorts and deduplicates Windows env keys before spawn, OpenClaw can inspect one PATH while the child receives another.
  - confidence: 0.86
- **[medium] Direct-spawn precedence is not fully preserved:** `src/process/windows-command.ts:85`
  - body: The `.cmd` fallback does not rule out CWD executables or `.com` candidates that libuv would prefer, so a shim target can be run instead of the command Node would execute.
  - confidence: 0.82
