## Summary

Keep open: the Windows claude-cli spawn bug is still source-reproducible on current main, but this PR is not merge-ready because proof is missing, the branch conflicts with current main, and the new resolver duplicates and diverges from the existing Windows spawn policy.

## What This Changes

The PR adds a Windows supervisor child-spawn resolver that unwraps npm-style `.cmd` shims to `.exe` or `node` entrypoints and adds focused unit coverage.

## Best Possible Solution

Land a rebased process-layer fix that routes supervisor child spawns through the shared Windows spawn policy or one equivalent shared helper, preserves current child-adapter behavior, adds regression coverage, and includes redacted Windows runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main registers claude-cli as bare `claude`, sends it through the supervisor child adapter, and that adapter still omits `claude` from Windows shim handling. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping Windows npm-style shims before spawn is the right direction, but this PR should be rebased and aligned with the current shared Windows spawn resolver instead of adding a second partial resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use the shared Windows spawn resolver:** `src/process/windows-command.ts:59-61`
  - body: Current main already has `resolveWindowsSpawnProgram()` with PATH/PATHEXT lookup, package metadata fallback, and fail-closed wrapper policy. Adding a second resolver here creates drift and already misses parts of that contract, so wire supervisor child spawns through the shared helper or factor one shared process helper instead.
  - confidence: 0.86
- **[P2] Match Node's Windows PATH key selection:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` scans `Object.entries()` order, but Node sorts env keys and passes only the first case-insensitive duplicate on Windows. With duplicate `PATH`/`Path` keys, this helper can inspect a different PATH than the child process receives and rewrite to the wrong binary.
  - confidence: 0.88
- **[P2] Preserve libuv's direct-spawn precedence:** `src/process/windows-command.ts:47-52`
  - body: `hasExeOnPath` only checks PATH for `<name>.exe`; libuv checks the current directory and tries `.com` before `.exe` when resolving direct spawns. This can unwrap a later `.cmd` shim even when a cwd executable or `.com` is what direct spawn would have run.
  - confidence: 0.84
