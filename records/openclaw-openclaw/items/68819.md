## Best Possible Solution

Land a rebased core process fix that unwraps npm-style Windows `.cmd` shims without shelling out, matches Node/env and Windows command selection, preserves current child-adapter behavior, and includes focused tests plus redacted Windows proof.

## Reproduction Assessment

Yes, source-reproducible: current main registers `claude-cli` as bare `claude`, passes it to the supervisor child adapter, and that adapter does not resolve `claude` or unwrap `.cmd` shims. I did not run a Windows E2E in this read-only review.

## Solution Assessment

No, not as written. Unwrapping npm-style `.cmd` shims without `cmd.exe` is the right direction, but the resolver must match Node's Windows env and command selection semantics before this command-execution patch is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match Node's Windows PATH selection:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` returns the first case-insensitive PATH entry from `Object.entries`, but Node sorts Windows env keys and keeps the first case-insensitive duplicate before spawn. With both `Path` and `PATH`, this can search a different PATH than the child receives, causing missed shim resolution or wrong-binary execution.
  - confidence: 0.88
- **[P2] Preserve Windows direct-spawn precedence:** `src/process/windows-command.ts:85-89`
  - body: The extensionless branch only rules out `<name>.exe` on PATH before unwrapping `<name>.cmd`. Windows direct spawn can prefer current-directory executables and `.com` candidates, so this can run the shim target when Node would have run a different binary.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The patch touches command execution and can choose a different Windows program than Node/libuv direct spawn would choose.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong command:** `src/process/windows-command.ts:26`
  - body: The resolver scans PATH-like env keys in insertion order while Node sorts and deduplicates Windows env keys before spawn, so OpenClaw can inspect one PATH while the child receives another.
