## Summary

Keep this PR open as a plausible fix attempt for a real doctor/config-path bug, but the current patch is not merge-ready because it can warn on the wrong path, recommend stale configs, and gives destructive deletion advice without verifying the active service config.

## What This Changes

The branch adds a doctor preflight warning for root and service-user config path drift by checking `/root`, `/home/node`, and `/home/openclaw` config file existence.

## Best Possible Solution

Rework this into a resolver-backed doctor/service config audit that compares doctor’s actual write path with a verified managed-service or running-gateway config path, emits non-destructive guidance, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-backed. The linked bug report gives concrete root-shell/systemd `HOME` mismatch steps, and current main source shows doctor reads through the invoking environment without a doctor-time service config-path comparison; I did not run the destructive root/systemd flow in this read-only review.

## Solution Assessment

No. A warning or guard is the right narrow direction, but this implementation compares hard-coded home-directory probes rather than doctor’s actual config resolver path and a verified service or running-gateway config path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve doctor target through the config resolver:** `src/commands/doctor-config-preflight.ts:144`
  - body: This preflight passes `path.join(resolveHomeDir(), ".openclaw")` as the doctor target even though doctor reads and writes through the config resolver. With `HOME=/root` and `OPENCLAW_CONFIG_PATH=/home/node/.openclaw/openclaw.json`, doctor is already targeting the service-user file, but this line still treats the target as `/root/.openclaw` and emits a false mismatch warning.
  - confidence: 0.92
- **[P2] Verify the active service config before recommending overrides:** `src/commands/doctor-config-preflight.ts:71-81`
  - body: The check treats the first existing `/home/node` or `/home/openclaw` config as the gateway config without proving the running service loaded it. A stale file at either path can make doctor recommend `OPENCLAW_CONFIG_PATH` for the wrong config, recreating the drift this warning is meant to prevent.
  - confidence: 0.88
- **[P2] Drop the unconditional root-config deletion advice:** `src/commands/doctor-config-preflight.ts:101-102`
  - body: This message tells operators to remove `/root/.openclaw/openclaw.json` whenever doctor is using another config path, but the patch does not verify that a legitimate root-managed gateway is not using that file. Following the advice can remove a live service config on multi-instance or root-managed hosts.
  - confidence: 0.84
