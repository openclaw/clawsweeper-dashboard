## Summary

Keep open. The underlying doctor/config drift report is still valid on current main, and this PR is useful context, but the branch is not merge-ready because it can warn on the wrong file and recommend unsafe cleanup without proving the active gateway config.

## What This Changes

The PR adds a doctor preflight warning that checks for `/root`, `/home/node`, and `/home/openclaw` config files before reading the doctor config snapshot.

## Best Possible Solution

Implement a doctor/service config-path audit that reuses the real config resolver and installed service environment, compares doctor’s actual target path with the gateway service’s effective config path, and emits non-destructive guidance unless it can prove a candidate config is stale and inactive.

## Reproduction Assessment

Yes. The linked report gives concrete root/systemd steps, and source inspection shows current main still has no doctor preflight guard; the PR’s false-positive path follows from `HOME=/root` plus `OPENCLAW_CONFIG_PATH=/home/node/.openclaw/openclaw.json`.

## Solution Assessment

No. A hard-coded existence scan is not the best fix because it does not prove which config the gateway loaded; reusing the resolver plus service/status metadata is narrower and less misleading.

  - body: This branch treats any existing `/home/node` or `/home/openclaw` config as the gateway config without checking the installed service environment or running daemon context. A stale file at either path can trigger a warning that tells the operator to use the wrong `OPENCLAW_CONFIG_PATH`.
  - confidence: 0.86

  - body: The warning tells users to run `sudo rm /root/.openclaw/openclaw.json` solely because that file exists. On root-managed or multi-instance hosts, that can delete an active gateway config; the preflight should stick to non-destructive investigation unless it proves the file is stale.
  - confidence: 0.82
