## Summary

Keep open: the linked bug is still source-reproducible and current main lacks a doctor-vs-service config-path guard, but this PR is not merge-ready because the patch can misidentify the active config path and lacks inspectable real behavior proof.

## What This Changes

The PR adds a doctor preflight warning for `/root/.openclaw/openclaw.json` versus `/home/node` or `/home/openclaw` config drift, with guidance to set `OPENCLAW_CONFIG_PATH` or remove the root config.

## Best Possible Solution

Rework this into a resolver-backed doctor/service config audit that compares doctor’s actual write path with a verified managed-service or running-gateway config path, emits non-destructive guidance, and adds focused regression coverage.

## Reproduction Assessment

Yes, source-backed. The linked report gives a concrete root-shell/systemd `HOME` mismatch flow, and current main shows doctor uses process-env config resolution and writes the selected config without a doctor-time service config-path comparison; I did not run the destructive root/systemd flow.

## Solution Assessment

No. A warning or guard is the right narrow direction, but this implementation compares hard-coded home-directory probes rather than doctor’s actual resolver path and a verified service or running-gateway config path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve doctor target through the config resolver:** `src/commands/doctor-config-preflight.ts:144`
  - body: This call passes `path.join(resolveHomeDir(), ".openclaw")` as the doctor target even though doctor reads and writes through the config resolver. With `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR` pointing at the service-user file, doctor can already target that file while this preflight still treats `/root/.openclaw` as the target and emits a misleading mismatch warning.
  - confidence: 0.92
- **[P2] Verify the active service config before recommending overrides:** `src/commands/doctor-config-preflight.ts:71`
  - body: The check treats the first existing `/home/node` or `/home/openclaw` config as the gateway config without proving the running service loaded it. A stale file at either path can make doctor recommend `OPENCLAW_CONFIG_PATH` for the wrong config, recreating the drift this warning is meant to prevent.
  - confidence: 0.88
- **[P2] Remove the unconditional root-config deletion advice:** `src/commands/doctor-config-preflight.ts:102`
  - body: This warning tells operators to remove `/root/.openclaw/openclaw.json` whenever doctor is using another config path, but the patch does not verify that a legitimate root-managed gateway is not using that file. Following the advice can remove a live service config on multi-instance or root-managed hosts.
  - confidence: 0.84
