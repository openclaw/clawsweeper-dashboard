## Summary

Keep open. The PR targets a real doctor/config drift bug that current main still does not guard against, but the patch is not merge-ready because it compares hard-coded home-directory guesses instead of doctor’s actual resolved config path or the managed gateway’s effective service config path, and it includes destructive cleanup advice that can be wrong on legitimate root-managed installs.

## What This Changes

The PR adds a doctor config preflight in `src/commands/doctor-config-preflight.ts` that checks `/root`, `/home/node`, and `/home/openclaw` config file locations and emits warning/remediation text before reading the config snapshot.

## Best Possible Solution

Implement a doctor/service config audit that compares doctor’s actual resolved write path with a verified managed gateway config path from service metadata or another runtime signal, respects `OPENCLAW_CONFIG_PATH`, `OPENCLAW_STATE_DIR`, `OPENCLAW_HOME`, and legacy candidates, emits non-destructive guidance, and includes focused tests plus a changelog entry.

## Reproduction Assessment

Yes. By source inspection, `HOME=/root` plus `OPENCLAW_CONFIG_PATH=/home/node/.openclaw/openclaw.json` means config IO targets the override while the PR still compares `path.join(resolveHomeDir(), '.openclaw')`, so it can emit a false mismatch warning after the documented workaround is already applied.

## Solution Assessment

No. A warning is the right narrow product direction, but this implementation is not the best fix because hard-coded existence checks can identify stale files, miss supported config paths, and recommend the wrong override or deletion; a resolver-backed service/runtime comparison is safer.
