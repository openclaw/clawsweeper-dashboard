Full review comments:

- **[P2] Use the real config resolver for the doctor target:** `src/commands/doctor-config-preflight.ts:139-141`
  - body: This preflight computes the doctor target from `resolveHomeDir()` and `path.join(home, ".openclaw")`, but the command reads through `readConfigFileSnapshot()`/`createConfigIO()`/`resolveConfigPath()`, which honors `OPENCLAW_CONFIG_PATH`, `OPENCLAW_STATE_DIR`, and legacy candidates. If the operator already applies the recommended `OPENCLAW_CONFIG_PATH=/home/node/...` fix while running as root, this branch can still warn that doctor is using `/root`, so the warning contradicts the actual write path.
  - confidence: 0.9
- **[P2] Verify the active service path before recommending a config:** `src/commands/doctor-config-preflight.ts:72-78`
  - body: The new warning selects `/home/node` or `/home/openclaw` only because the file exists, without checking whether the running gateway or installed service actually loaded that file. On hosts with stale service-user configs or multiple instances, this can point `OPENCLAW_CONFIG_PATH` at the wrong config and send operators away from the live gateway state.
  - confidence: 0.88
- **[P1] Remove the root config deletion advice:** `src/commands/doctor-config-preflight.ts:110`
  - body: The warning tells users to run `sudo rm /root/.openclaw/openclaw.json` whenever doctor is not targeting `/root` and that file exists, but it does not verify that no root-managed gateway is actively using it. On multi-instance or root-owned deployments, following this advice can delete the live config and cause an outage.
  - confidence: 0.92
