## Summary

Keep open. The PR targets a valid doctor/config-clobbering bug and is explicitly paired with the still-open report, but the current patch is not merge-ready because it uses hard-coded file-existence heuristics instead of doctor’s actual resolved config path or the managed gateway’s effective service environment, and it includes unsafe deletion guidance.

## What This Changes

The PR adds a doctor config preflight in `src/commands/doctor-config-preflight.ts` that checks `/root`, `/home/node`, and `/home/openclaw` config file locations and emits warning/remediation text before reading the config snapshot.

## Best Possible Solution

Implement a doctor/service config audit that compares doctor’s actual resolved write path with the managed gateway’s effective config path from service metadata or another verified runtime signal, respects `OPENCLAW_CONFIG_PATH`, `OPENCLAW_STATE_DIR`, `OPENCLAW_HOME`, and legacy config candidates, and emits non-destructive guidance unless stale inactive config can be proven.

## Reproduction Assessment

Yes. By source inspection, `HOME=/root` plus `OPENCLAW_CONFIG_PATH=/home/node/.openclaw/openclaw.json` means config IO targets the override while the PR still passes `path.join(resolveHomeDir(), '.openclaw')`, so a service-user file can trigger a false mismatch warning. Current main also still lacks a doctor preflight comparison for the original report.

## Solution Assessment

No. The PR’s hard-coded existence scan is not the best fix because it can warn on stale files, miss supported config paths, and recommend the wrong override or deletion; the safer path is a resolver-backed service/runtime config comparison with non-destructive output.

  - confidence: 0.88
