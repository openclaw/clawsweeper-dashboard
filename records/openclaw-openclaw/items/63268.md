## What This Changes

Adds a doctor preflight warning that checks `/root`, `/home/node`, and `/home/openclaw` config locations and prints `OPENCLAW_CONFIG_PATH` or deletion guidance when doctor appears to target a different config.

## Best Possible Solution

Rework this into a resolver-backed doctor/service config audit that compares doctor’s actual write path with a verified managed-service or running-gateway config path, gives non-destructive guidance, adds focused regression coverage, and includes redacted real-run proof.

## Reproduction Assessment

Yes, source-backed. The paired issue gives a concrete root-shell/systemd `HOME` mismatch flow, and current source shows doctor reads and writes through process-env config resolution without a doctor-time service config-path comparison.

## Solution Assessment

No. A focused warning or guard is the right direction, but this implementation guesses from hard-coded files instead of comparing doctor’s resolver-backed target to verified service or running-gateway state.
