## Summary

Keep open: current main and the latest release still only partially mitigate stale Control UI protocol mismatches. Service-worker build-id registration, bounded cache rotation, strict protocol rejection, and fragment-token hydration are present, but the reported cloud stale-service-worker recovery path remains manual and not token-safe for non-technical hosted deployments.

## What This Changes

The issue requests a user-friendly, token-safe recovery path for cloud-hosted Control UI protocol mismatches caused by stale browser service-worker assets after Gateway updates.

## Best Possible Solution

Add a token-safe in-browser stale-asset recovery path for Control UI protocol mismatch, with browser proof, while keeping Gateway protocol negotiation strict and avoiding default storage wipes.

## Reproduction Assessment

No for a high-confidence full live repro: I did not reproduce the Zeabur/Chrome stale service-worker loop. Yes at source level: stale Control UI protocol ranges are rejected by the Gateway, and current UI/docs still provide manual-only recovery.

## Solution Assessment

No, current main is not the full solution. Existing worker versioning and #token parsing are useful partial mitigations, but the missing piece is a user-safe browser recovery action rather than relaxing Gateway protocol compatibility.
