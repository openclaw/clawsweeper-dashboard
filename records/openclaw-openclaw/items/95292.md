## Summary

Keep open: current main and the latest release still provide service-worker versioning, strict protocol rejection, and fragment-token hydration, but they do not provide the requested token-safe, user-friendly stale-Control-UI recovery path for hosted cloud deployments.

## What This Changes

The issue asks for Control UI protocol-mismatch recovery after cloud updates so stale service-worker assets can be cleared without DevTools, manual cache clearing, or losing token-auth access.

## Best Possible Solution

Add a token-safe Control UI stale-asset recovery path for protocol mismatch, backed by browser proof for an old service worker and a token-auth cloud-style dashboard, while keeping Gateway protocol negotiation strict.

## Reproduction Assessment

Yes at source level, but not as a full live reproduction: current source rejects stale protocol ranges and the UI/docs still expose manual-only recovery. I did not reproduce the Zeabur/Chrome stale-service-worker loop in a browser.

## Solution Assessment

No, current main is not the full solution: build-id service-worker rotation and #token parsing are useful, but the requested non-technical recovery path remains missing. A browser-side recovery action is safer than relaxing the gateway protocol contract.
