## Summary

Keep open: current main and v2026.6.8 already include the earlier service-worker cache rotation, navigation bypass, shared protocol constants, and fragment-token hydration, but the reported cloud stale-service-worker recovery gap remains. The protocol-mismatch UI and docs still provide manual recovery steps rather than a token-safe in-browser recovery path for non-technical cloud deployments.

## What This Changes

The issue requests reliable Control UI recovery after cloud gateway updates when an old browser service worker keeps serving a protocol-v3 bundle against a protocol-v4 gateway.

## Best Possible Solution

Add a token-safe Control UI stale-asset recovery path for protocol mismatch, with browser-oriented proof, while keeping gateway protocol negotiation strict and avoiding Clear-Site-Data storage wipes by default.

## Reproduction Assessment

Yes at source level, but not as a full live reproduction: stale Control UI protocol ranges are rejected by the gateway, and current protocol-mismatch UI/docs still provide manual-only recovery. I did not reproduce the Zeabur/Chrome stale-service-worker loop in this review.

## Solution Assessment

No, current main is not the full solution: the existing worker versioning and #token parsing are useful, but they are already present in v2026.6.8 while the user-friendly recovery path remains absent. A narrow browser-side recovery action is safer than relaxing the gateway protocol.
