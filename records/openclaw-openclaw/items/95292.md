## Summary

Keep open: current main and v2026.6.8 already have build-id service-worker cache rotation and #token hydration, but the protocol-mismatch UI/docs still offer manual recovery only. The older desktop version-skew issue overlaps, but it does not own this cloud browser service-worker plus token-auth recovery gap.

## What This Changes

The issue reports stale Control UI service-worker assets causing protocol mismatch lockouts after cloud gateway updates, especially on token-based Zeabur deployments.

## Best Possible Solution

Add a token-safe Control UI recovery path for protocol mismatch caused by stale service-worker/browser assets, while keeping gateway protocol negotiation strict and avoiding new config or destructive Clear-Site-Data behavior by default.

## Reproduction Assessment

Yes at source level, not live: stale Control UI protocol ranges are rejected by the gateway, and current protocol-mismatch UI/docs still provide manual-only recovery. I did not reproduce the full Zeabur/Chrome stale-service-worker loop.

## Solution Assessment

No, current main is not the full fix: service-worker versioning and #token parsing are useful but already present in v2026.6.8, while the requested user-friendly recovery path remains absent. The narrow maintainable fix is a Control UI/browser-cache recovery action, not relaxing the gateway protocol.
