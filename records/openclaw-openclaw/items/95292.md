## Summary

Keep open: current main and v2026.6.9 already include service-worker build-id rotation, navigation bypass, and fragment-token hydration, but the protocol-mismatch UI and docs still provide manual recovery only, with no token-safe in-browser stale-cache repair for cloud Control UI deployments.

## What This Changes

The issue requests automatic or one-click Control UI recovery when stale service-worker assets keep speaking an older gateway protocol after a cloud update.

## Best Possible Solution

Add a token-preserving stale-Control-UI recovery path for protocol mismatch, with focused UI/SW tests and browser proof, while keeping strict gateway protocol negotiation.

## Reproduction Assessment

Yes at source level, but not as a live Zeabur reproduction: a stale Control UI protocol range is rejected by the gateway, and current UI/docs still provide manual-only recovery. The intermittent Chrome service-worker loop was not independently reproduced here.

## Solution Assessment

No, current main is not the full solution: existing service-worker versioning and #token parsing are useful but already present while the user-friendly recovery path remains absent. A narrow browser-side recovery action is safer than relaxing gateway protocol negotiation or clearing storage by default.
