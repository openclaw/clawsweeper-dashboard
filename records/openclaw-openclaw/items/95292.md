## Summary

Keep open: current main still has strict gateway protocol rejection and build-scoped service-worker cache rotation, but it does not provide a token-safe in-browser recovery path when a stale Control UI bundle keeps reconnecting with an old protocol range. The only linked implementation attempt, https://github.com/openclaw/openclaw/pull/95559, is closed unmerged and was not a safe canonical fix because review found service-worker activation/message sequencing defects and missing real browser proof.

## What This Changes

The issue requests a user-friendly, token-safe recovery path for cloud-hosted Control UI deployments where stale browser service-worker assets cause protocol mismatch after gateway updates.

## Best Possible Solution

Add browser-owned stale Control UI asset recovery for protocol mismatch that preserves or rehydrates token auth, keeps gateway protocol negotiation strict, and includes real stale-service-worker upgrade proof.

## Reproduction Assessment

Yes at source level, but not as a full live reproduction: stale Control UI protocol ranges are rejected by the gateway, and current UI/docs still provide manual-only recovery. I did not reproduce the Zeabur/Chrome stale-service-worker loop in this read-only review.

## Solution Assessment

No current solution is complete. The safest direction is a token-safe browser recovery path; the closed activation-reload PR is only a plausible starting point and needs corrected service-worker sequencing plus real browser proof.
