## Summary

Keep open: current main and v2026.6.8 already include service-worker build-id rotation, top-level navigation bypass, strict protocol mismatch reporting, and #token hydration, but the reported cloud stale-service-worker plus token-auth lockout still lacks a user-friendly in-browser recovery path.

## What This Changes

The issue asks OpenClaw to recover from stale Control UI service-worker assets after Gateway protocol updates on cloud deployments without requiring manual browser cache or token workarounds.

## Best Possible Solution

Add a token-safe in-browser stale Control UI recovery path for protocol mismatch, backed by browser proof, while leaving Gateway protocol negotiation strict.

## Reproduction Assessment

Yes at source level, but not as a full live Zeabur reproduction: stale Control UI protocol ranges are rejected by the Gateway, and current UI/docs still provide manual-only recovery.

## Solution Assessment

No, current main is not the full solution: the existing worker versioning and #token parsing are useful but already shipped in v2026.6.8. A narrow token-safe browser recovery action is safer than relaxing Gateway protocol checks or clearing all storage by default.
