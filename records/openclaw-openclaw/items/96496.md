## Summary

Keep open: the patch is focused and appears correct. Current main still leaves the six targeted speech success JSON reads on `response.json()`, while this PR wires them to the shared bounded provider JSON reader with positive CI and terminal proof; the remaining maintainer call is accepting the 16 MiB cap for these provider responses.

## What This Changes

The PR routes six speech TTS/STT voice-list or transcription success JSON reads through `readProviderJsonResponse` and updates the OpenRouter test mock for that SDK helper.

## Best Possible Solution

Merge after maintainer acceptance that the shared 16 MiB provider JSON cap is the correct speech-provider boundary, with any legitimate larger-response need handled by a separate documented contract.

## Reproduction Assessment

Yes. Current main has direct `response.json()` calls at the six targeted speech success-response paths, and the PR proof shows the old unbounded path consumes the full oversized body while the bounded reader stops near the cap.

## Solution Assessment

Yes. Reusing the existing SDK `readProviderJsonResponse` is the narrow owner-boundary fix; adding provider-local readers or new config knobs would duplicate the shared response-limit contract.
