## Summary

Keep open. Current main and the latest tagged release still hardcode memory-core inline dreaming output to `memory/{YYYY-MM-DD}.md` and expose no schema, normalizer, docs, or test support for an alternate inline target; this is a real feature/config request that needs maintainer API and path-validation judgment.

## What This Changes

Request configurable memory-core dreaming inline output so Obsidian daily-note layouts can target paths such as `memory/daily-memory/{date}.md` while preserving managed markers and documentation.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to the existing path, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and focused tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes for source-level verification, though this is a feature request rather than a broken existing contract. Current main resolves inline dreaming only to `workspaceDir/memory/YYYY-MM-DD.md`, and the config type plus manifest schema expose no alternate target.

## Solution Assessment

Unclear. A configurable memory-core inline target is the right solution family, but maintainers should choose the public API shape and write-path validation policy before implementation.
