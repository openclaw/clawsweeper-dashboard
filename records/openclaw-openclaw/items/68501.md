## Summary

Keep open. Current main and v2026.6.6 still hardcode inline light/REM dreaming output to `memory/YYYY-MM-DD.md`, expose no accepted inline target config, and no canonical issue or PR now owns the specific `daily-memory` target request.

## What This Changes

Request to make memory-core inline dreaming output configurable so managed blocks can target Obsidian-style `memory/daily-memory/YYYY-MM-DD.md` instead of only `memory/YYYY-MM-DD.md`.

## Best Possible Solution

Define the memory-core inline target contract first, then implement a narrow schema/runtime/docs/test change that preserves the current default and managed markers.

## Reproduction Assessment

Yes. Source inspection shows inline mode still writes through `resolveDailyMemoryPath` to `memory/YYYY-MM-DD.md`, while the manifest and runtime config expose no inline path or target field.

## Solution Assessment

Unclear. A path or preset config would solve the reported workflow, but the best permanent contract needs maintainer judgment because OpenClaw treats config surface area and memory-core layout as compatibility-sensitive.
