## Summary

Keep open. Current main still hardcodes memory-core inline light/REM dreaming output to `memory/{YYYY-MM-DD}.md`, while the manifest schema, host config resolver, docs, and tests still expose only storage mode/separate-report controls. The reporter's workaround and the existing ClawSweeper comment are consistent with the current source; related #69536 is adjacent daily-file discovery work, not a configurable inline dreaming write target.

## What This Changes

The issue requests a configurable memory-core inline dreaming write destination so managed light/REM blocks can target Obsidian-style `memory/daily-memory/{date}.md` instead of hardcoded `memory/{date}.md`.

## Best Possible Solution

Add a memory-core-owned storage option that defaults to `memory/{date}.md`, safely resolves a workspace-relative date template or explicit target enum, passes that value through the host config to the inline writer, preserves the existing managed block markers, and updates schema/docs/tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure `dreaming.storage.mode: "inline"` and the current writer/test path resolves to `memory/2026-04-05.md`, with no schema or resolver field that can redirect it to `memory/daily-memory/2026-04-05.md`.

## Solution Assessment

Unclear. A configurable path is the right solution family, but the exact public API shape, validation rules, and docs contract need maintainer judgment before an automated implementation should pick `inlineRelativePath` versus `inlineTarget`.
