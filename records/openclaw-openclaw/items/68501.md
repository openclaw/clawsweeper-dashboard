## Summary

Keep open. Current main and the latest release still hardcode inline memory-core dreaming output to `memory/{YYYY-MM-DD}.md`, while the requested Obsidian `daily-memory` alignment would add a new public config contract and path-validation policy.

## What This Changes

Request adding memory-core dreaming config to redirect inline daily dreaming output from `memory/{date}.md` to an Obsidian-style daily-memory path while preserving managed markers and docs.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and focused tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: inline storage calls a resolver that only returns `memory/{YYYY-MM-DD}.md`, and neither the host config type nor manifest schema accepts an alternate inline target.

## Solution Assessment

Unclear. A configurable memory-core target is the right solution family, but maintainers should choose the public API and validation boundary before an implementation picks `inlineRelativePath`, `inlineTarget`, or another shape.
