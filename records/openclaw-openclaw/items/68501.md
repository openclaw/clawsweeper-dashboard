## Summary

Keep open. Current main still hardcodes inline light/REM dreaming output to `memory/{YYYY-MM-DD}.md`, while the manifest schema, host config resolver, docs, and tests expose no supported target-path option. The related daily-note discovery work is adjacent; it does not add a configurable inline dreaming write target.

## What This Changes

The issue requests a memory-core dreaming configuration option that redirects inline light/REM dreaming blocks to layouts such as `memory/daily-memory/{date}.md` while preserving managed markers and documenting Obsidian daily-note interaction.

## Best Possible Solution

Add a memory-core-owned storage option that defaults to the current `memory/{date}.md`, safely resolves only workspace-relative date templates or a constrained target enum, preserves existing managed block markers, and updates schema, docs, and tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `dreaming.storage.mode` as `inline`, and current main resolves/writes the inline path to `memory/2026-04-05.md` with tests pinning that destination and no schema/resolver field to redirect it.

## Solution Assessment

Unclear. A configurable target is the right solution family, but maintainers should choose the public config shape and validation contract before an automated repair picks `inlineRelativePath` or `inlineTarget`.
