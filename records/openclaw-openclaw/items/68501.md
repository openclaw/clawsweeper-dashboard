## Summary

Keep open. Current main and the latest release still preserve explicit inline dreaming mode as `memory/YYYY-MM-DD.md`, while the memory-core runtime/schema/docs do not expose an alternate `daily-memory` target; this remains a product/config-contract decision, not an implemented cleanup.

## What This Changes

The issue requests a configurable memory-core inline dreaming output path so managed dreaming blocks can target Obsidian-style `memory/daily-memory/YYYY-MM-DD.md` files instead of only `memory/YYYY-MM-DD.md`.

## Best Possible Solution

Choose a narrow memory-core inline target contract, then update schema, normalization, markdown writing, docs, and tests while preserving the current default and managed markers.

## Reproduction Assessment

Yes. Source inspection of current main shows explicit inline dreaming still resolves to `memory/YYYY-MM-DD.md`, and the manifest/runtime config expose no accepted inline target field.

## Solution Assessment

Unclear. A path or preset knob would solve the reported workflow, but the best permanent shape needs maintainer judgment because OpenClaw has a high bar for new config surface.
