## Summary

Keep open. Current main and the latest shipped tag still hardcode inline memory-core dreaming output to `memory/{YYYY-MM-DD}.md`, and accepting an Obsidian `daily-memory` target would add a new public config and validation contract rather than closeable cleanup.

## What This Changes

The issue requests a configurable memory-core inline dreaming output target so vaults can write managed dreaming blocks under `memory/daily-memory/{date}.md` instead of `memory/{date}.md`.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes for the current behavior: source inspection shows inline storage resolves only `memory/{YYYY-MM-DD}.md`, and the schema/normalizer expose no accepted redirect field. This is a feature/config gap rather than a broken existing contract.

## Solution Assessment

Unclear. A configurable memory-core inline target is the right solution family, but maintainers should choose the public API and validation boundary before an implementation picks `inlineRelativePath`, `inlineTarget`, or another shape.
