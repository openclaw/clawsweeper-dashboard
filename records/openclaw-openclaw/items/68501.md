## Summary

Keep open. Current main and the latest release still hardcode memory-core inline dreaming output to `memory/{YYYY-MM-DD}.md`; supporting `daily-memory` would add a new public config surface and write-path validation policy.

## What This Changes

Request configurable memory-core inline dreaming output so Obsidian-style daily notes can live under `memory/daily-memory/{YYYY-MM-DD}.md`.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and focused tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes for source proof, though this is a feature request rather than a broken existing contract. Current main resolves inline dreaming only to `workspaceDir/memory/YYYY-MM-DD.md`, and the config type plus manifest schema expose no alternate target.

## Solution Assessment

Unclear. A configurable memory-core inline target is the right solution family, but maintainers should choose the public API shape and path-validation policy before implementation.
