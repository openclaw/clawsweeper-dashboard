## Summary

Keep open. Current main and v2026.5.7 still hardcode inline memory-core dreaming output to `memory/{YYYY-MM-DD}.md`, and supporting `memory/daily-memory/{date}.md` needs a new public config and path-validation decision.

## What This Changes

Request configurable memory-core inline dreaming output so managed dreaming blocks can be written under `memory/daily-memory/{date}.md` instead of only `memory/{date}.md`.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to the current path, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and focused tests.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows inline storage always resolves to `workspaceDir/memory/YYYY-MM-DD.md`, and the config schema/type expose no redirect field. I did not run runtime tests because this review is intentionally read-only.

## Solution Assessment

Unclear. A configurable memory-core destination is the right solution family, but maintainers should choose the public API and validation boundary before an implementation picks `inlineRelativePath`, `inlineTarget`, or another shape.
