## Summary

Keep open. Current main and the latest release still hardcode inline memory-core dreaming output to `memory/{YYYY-MM-DD}.md`, while the requested configurable inline target would add a new memory-core config surface that needs maintainer product/API judgment.

## What This Changes

Request to make memory-core inline dreaming output configurable so it can write managed light/REM blocks under layouts such as `memory/daily-memory/{date}.md` instead of only `memory/{date}.md`.

## Best Possible Solution

Decide the memory-core inline target API first, then implement it in the plugin schema, config resolver, dreaming markdown writer, docs, and regression tests while preserving the default `memory/{date}.md` behavior.

## Reproduction Assessment

Yes for the current limitation: source and tests show inline dreaming writes to `memory/{YYYY-MM-DD}.md`. I did not run a live dreaming sweep because the reported problem is a missing config capability rather than an already-defined failing behavior.

## Solution Assessment

Unclear. A path knob is plausible, but the best permanent shape needs maintainer judgment because OpenClaw’s config surface is intentionally high-bar and memory-core owns the daily-note layout contract.
