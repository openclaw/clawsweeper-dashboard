## Summary

Keep open. Current main and the latest release still hardcode memory-core inline dreaming output to `memory/{YYYY-MM-DD}.md`, and the requested Obsidian `daily-memory` alignment would add a new public config contract plus write-path validation policy.

## What This Changes

Request configurable memory-core inline dreaming output so managed dreaming blocks can target layouts such as `memory/daily-memory/{date}.md`.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and focused tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. Source inspection on current main shows inline storage calls a resolver that only returns `memory/{YYYY-MM-DD}.md`, while the host type and manifest schema expose no alternate inline target.

## Solution Assessment

Unclear. A configurable memory-core target is the right solution family, but maintainers should choose the public API and validation boundary before an implementation picks `inlineRelativePath`, `inlineTarget`, or another shape.
