## Summary

Keep open. Current main and the latest shipped release still hardcode inline memory-core dreaming output to `memory/{YYYY-MM-DD}.md`, and adding an Obsidian-style `daily-memory` redirect requires a new public config and path-validation contract rather than cleanup that can be closed.

## What This Changes

The issue requests configurable inline dreaming output so memory-core can write managed dreaming blocks to Obsidian-style `memory/daily-memory/{date}.md` files instead of only `memory/{date}.md`.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: inline storage calls `writeDailyDreamingPhaseBlock`, which resolves only `memory/{YYYY-MM-DD}.md`, and the schema/host resolver expose no accepted redirect field.

## Solution Assessment

Unclear. A configurable memory-core target is the right solution family, but maintainers should choose the public API and validation boundary before an implementation picks `inlineRelativePath`, `inlineTarget`, or another shape.
