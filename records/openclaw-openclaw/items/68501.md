## Summary

Keep open. Current `main` still hardcodes inline memory-core dreaming output to `memory/{YYYY-MM-DD}.md`, and the config schema, resolver, tests, and docs do not expose a supported redirect to `memory/daily-memory/{date}.md`; this is a feature/config contract decision, not an already-implemented cleanup.

## What This Changes

The issue requests a configurable memory-core inline dreaming output path so Obsidian daily-note vaults can use `memory/daily-memory/{date}.md` instead of the hardcoded `memory/{date}.md`.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: inline storage calls `writeDailyDreamingPhaseBlock`, which resolves `memory/{YYYY-MM-DD}.md`, and current tests/schema expose no accepted redirect field.

## Solution Assessment

Unclear. A configurable memory-core target is the right solution family, but maintainers should choose the public config shape and path-validation contract before an implementation picks `inlineRelativePath` or `inlineTarget`.
