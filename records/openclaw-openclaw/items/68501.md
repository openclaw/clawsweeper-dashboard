## Summary

Keep open. Current main still hardcodes inline dreaming output to `memory/{YYYY-MM-DD}.md`, and the manifest schema, host resolver, tests, and docs expose no supported inline target-path option; the remaining work is a public memory-core storage/config decision rather than an already-implemented fix.

## What This Changes

Request to add a memory-core dreaming config option that redirects inline light/REM managed blocks from `memory/{date}.md` to a daily-memory-compatible path.

## Best Possible Solution

Add a memory-core-owned storage option that defaults to `memory/{date}.md`, safely resolves only constrained workspace-relative targets, preserves the existing managed block markers, and updates schema, docs, and tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: inline storage calls `writeDailyDreamingPhaseBlock`, which resolves `memory/{YYYY-MM-DD}.md`, and current tests/schema pin that behavior with no accepted redirect field.

## Solution Assessment

Unclear. A configurable memory-core target is the right solution family, but maintainers should choose the public config shape and path-validation contract before an implementation picks `inlineRelativePath` or `inlineTarget`.
