## Summary

Keep open. Current main still hardcodes inline light/REM dreaming output to `memory/{YYYY-MM-DD}.md` and exposes no accepted path or target option in the memory-core schema or host resolver; related daily-note subdirectory recognition is adjacent but does not redirect inline writes.

## What This Changes

Request a memory-core dreaming config option to write inline dreaming blocks under a daily-memory-style path while preserving managed markers and documenting Obsidian daily-note interactions.

## Best Possible Solution

Add a memory-core-owned storage option that defaults to the current `memory/{date}.md`, safely resolves a constrained workspace-relative target or enum, preserves the existing managed blocks, and updates schema, docs, and focused tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: set `dreaming.storage.mode` to `inline`, and current main resolves/writes the inline path to `memory/2026-04-05.md` with tests pinning that destination and no schema/resolver field to redirect it.

## Solution Assessment

Unclear. A configurable target is the right solution family, but maintainers should choose whether this is a relative date template, a constrained target enum, or another validated API before implementation.
