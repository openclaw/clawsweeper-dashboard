## Summary

Keep open. Current main and the latest release still hardcode inline light/REM dreaming output to `memory/YYYY-MM-DD.md`, and the memory-core schema/runtime do not expose a target path or `daily-memory` preset; this remains a product/config-contract decision rather than an implemented cleanup.

## What This Changes

The issue requests a configurable memory-core dreaming inline output path so Obsidian-style daily-memory vaults can receive managed dreaming blocks under `memory/daily-memory/` instead of the hardcoded `memory/YYYY-MM-DD.md` path.

## Best Possible Solution

Choose a narrow memory-core inline target contract, then update schema, normalization, markdown writing, docs, and tests while preserving the current default and managed markers.

## Reproduction Assessment

## Solution Assessment
