## Summary

Keep open. Current main and the latest release still hardcode inline dreaming phase output to `memory/YYYY-MM-DD.md`, while the requested Obsidian-style daily-memory target requires a new memory-core config contract and write-path validation policy.

## What This Changes

Request configurable memory-core dreaming inline output so managed Light/REM blocks can target Obsidian-style `memory/daily-memory/{date}.md` paths while preserving markers and documenting Obsidian daily-note interaction.

## Best Possible Solution

Add a memory-core-owned inline storage target option that defaults to `memory/{date}.md`, validates constrained workspace-relative destinations, preserves managed markers, and updates schema, docs, and focused tests for Obsidian daily-note layouts.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: inline storage calls `writeDailyDreamingPhaseBlock`, which resolves only `memory/YYYY-MM-DD.md`, and the config type/schema expose no alternate target. I did not execute tests or a live dreaming run because this review was read-only.

## Solution Assessment

Unclear. A configurable memory-core inline target is the right solution family, but maintainers should choose the public API and validation boundary before an implementation picks `inlineRelativePath`, `inlineTarget`, or another shape.
