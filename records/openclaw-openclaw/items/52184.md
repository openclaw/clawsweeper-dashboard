## Label Justifications

- `impact:crash-loop`: A LaunchAgent pinned to a removed Volta Node image path can fail to restart or repeatedly fail after Node upgrades.

## Summary

Keep open: current main and the latest release still stabilize Homebrew Cellar Node paths only, so a Volta `tools/image/node/<version>/bin/node` path remains version-pinned when no supported system Node is available. The related target-repo PR was closed unmerged, and the remaining question is a maintainer policy choice about whether launchd services should use the Volta shim fallback.

## What This Changes

Prefer the Volta `~/.volta/bin/node` shim over `~/.volta/tools/image/node/<version>/bin/node` when generating macOS gateway LaunchAgent Node paths.

## Best Possible Solution

Maintainers should choose the LaunchAgent Node policy; if accepted, add a narrow Volta image-to-shim normalization behind the existing system-Node preference with focused unit coverage and live macOS LaunchAgent proof.

## Reproduction Assessment

Yes, source-reproducible: current main returns non-Homebrew paths unchanged in `resolveStableNodePath`, and `resolvePreferredNodePath` falls back to that unchanged path when no supported system Node exists. I did not run launchd because this review is read-only.

## Solution Assessment

Unclear: the proposed Volta shim fallback is plausible and narrow, but it changes daemon runtime policy where current docs prefer system Node and warn about version-manager paths. The best fix needs maintainer confirmation plus live macOS LaunchAgent proof.
