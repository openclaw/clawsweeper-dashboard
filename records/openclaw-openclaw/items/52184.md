## Label Justifications

- `P2`: The issue affects a bounded macOS gateway LaunchAgent runtime-selection workflow that can break after Volta Node upgrades, but it is not a core-runtime emergency.
- `impact:crash-loop`: A LaunchAgent pinned to a removed Volta tools/image Node binary can fail to start or restart after a Node upgrade.

## Summary

Keep open: current main and latest release still do not normalize Volta tools/image Node paths, and the remaining choice is whether OpenClaw should support Volta shim fallback for macOS LaunchAgent generation instead of continuing to prefer system/Homebrew Node policy.

## What This Changes

Request that macOS gateway LaunchAgent generation prefer the stable Volta shim path over a version-pinned Volta tools/image Node path when no better system Node is available.

## Best Possible Solution

Adopt a maintainer-approved service runtime policy, then either land a narrow resolver and regression-test change for Volta shim fallback with macOS proof, or explicitly keep system/Homebrew Node as the supported service path.

## Reproduction Assessment

Yes, source inspection is enough: with a supported Volta tools/image `execPath` and no supported system Node, current main returns the unchanged current executable because only Homebrew Cellar paths are stabilized.

## Solution Assessment

Unclear. Resolver-level canonicalization is the narrow plausible boundary, but maintainers still need to decide whether Volta shim fallback is a supported service policy or whether system/Homebrew Node remains the intended answer.
