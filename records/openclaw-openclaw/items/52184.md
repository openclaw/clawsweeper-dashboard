## Label Justifications

- `P2`: This is a bounded macOS gateway service-install reliability issue for Volta-managed Node users, with medium upgrade fragility but no emergency blast radius.
- `impact:crash-loop`: A LaunchAgent pinned to a removed Volta `tools/image` Node binary can fail to start or recover after Node upgrades.

## Summary

Keep open: current main and v2026.6.10 still leave Volta `tools/image` Node paths unchanged in the no-system-Node fallback path, and the remaining action is maintainer policy review plus the open fork PR that references this issue with closing syntax.

## What This Changes

The issue requests that macOS gateway LaunchAgent generation prefer the stable Volta shim Node path over a version-pinned Volta `tools/image` Node path.

## Best Possible Solution

Keep this issue as the narrow Volta shim policy point, then review, land, or reject the linked fork PR after maintainer approval and real macOS LaunchAgent proof.

## Reproduction Assessment

Yes, source-level: `resolveStableNodePath` returns non-Cellar paths unchanged, and `resolvePreferredNodePath` returns that unchanged value when no supported system Node exists. I did not run a live macOS launchd install in this read-only review.

## Solution Assessment

Unclear. Resolver-level canonicalization is the narrow plausible fix, but current docs and audit policy still prefer system Node and warn on version-managed daemon runtimes, so maintainers need to confirm the Volta shim fallback policy.
