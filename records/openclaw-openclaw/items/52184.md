## Label Justifications

- `P2`: The issue affects a bounded macOS gateway service install path and can leave LaunchAgents pinned to brittle Node paths, but it is not an emergency.
- `impact:crash-loop`: A stale version-managed Node path can make the supervised gateway fail to start after Node upgrades or service regeneration.

## Summary

Keep open: current main and the latest release still leave Volta tools/image Node paths unchanged in the no-system-Node fallback path, and the only exact implementation candidate is an open cross-repo PR that has not landed in the target repository.

## What This Changes

Request to prefer the stable Volta Node shim over a version-pinned Volta tools/image Node path when generating the macOS gateway LaunchAgent.

## Best Possible Solution

Review the linked Volta resolver candidate against launchd and non-interactive Volta behavior, then either land a target-repo resolver update with focused tests or document why the version-pinned fallback remains intentional.

## Reproduction Assessment

Yes. Source inspection is enough: with a supported Volta tools/image execPath and no supported system Node, resolvePreferredNodePath returns resolveStableNodePath(execPath), and resolveStableNodePath only normalizes Homebrew Cellar paths.

## Solution Assessment

Unclear. Resolver-level canonicalization is the narrow plausible boundary, but maintainers still need to decide whether the Volta shim is the safe fallback for macOS launchd when no supported system Node exists.
