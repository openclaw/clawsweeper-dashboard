## Summary

Keep open: the reported embedded-runner hook bug is still present on current main, but this PR is not merge-ready because the guard uses session-level output state, it carries a separate parser change with a content-dropping edge case, and after-fix real behavior proof is still missing.

## What This Changes

The PR adds a `lastAssistant` guard and usage fallback around embedded-runner `llm_output` emission, plus JSON-stringified chat-content parsing and helper tests.

## Best Possible Solution

Land a focused current-main fix that suppresses embedded-runner `llm_output` unless the current attempt produced assistant output, with regression coverage; keep JSON-stringified content parsing in https://github.com/openclaw/openclaw/pull/39583.

## Reproduction Assessment

Yes, by source inspection: current main computes `currentAttemptAssistant` but emits `llm_output` whenever hooks exist and the prompt-error gate passes. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No, not as submitted. The hook guard should use current-attempt output rather than session-level `lastAssistant`, and the JSON-stringified parser change should be removed from this PR or handled through https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2007`
  - body: A timeout after an earlier assistant message can leave session-level `lastAssistant` truthy, so this condition can still emit `llm_output` even though the current model attempt produced no output. Gate on `currentAttemptAssistant` or current-attempt text instead of the session's last assistant message.
  - confidence: 0.87
- **[P2] Keep unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The new string path recurses when every parsed item has `type` and `text`, but the array extractor only keeps blocks where `type === "text"`. A serialized block such as `[{"type":"output_text","text":"hello"}]` now parses and returns `null` instead of preserving the original string, so restrict recursion to supported text block shapes or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
