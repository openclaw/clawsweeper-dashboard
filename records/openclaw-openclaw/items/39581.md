## Summary

Keep open: current main still lacks a safe embedded-runner `llm_output` no-output guard, but this PR is not ready because the new guard keys off transcript-wide `lastAssistant` and the branch also mixes in an unrelated chat-content parser change that can regress serialized non-`text` blocks.

## What This Changes

The PR adds a `lastAssistant` guard and usage coercion around embedded `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded `llm_output` emission on current-attempt assistant output or non-empty current `assistantTexts`, preserve normal output hooks, add focused regression coverage, and remove or split the parser change.

## Reproduction Assessment

Yes, by source inspection. A current-main embedded run with a prior assistant message and no current-attempt assistant output can still reach `runLlmOutput` because `lastAssistant` is transcript-wide while the hook emission has no current-output guard.

## Solution Assessment

No. The PR targets the right bug, but `lastAssistant` is the wrong boundary; the maintainable fix should use `currentAttemptAssistant` or current `assistantTexts`, and the JSON-string parser work should be removed or handled separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check looks at the last assistant anywhere in the transcript. If a run times out before producing output after a previous assistant turn, this remains truthy and `llm_output` can still fire for a no-output current attempt. Guard on `currentAttemptAssistant` or non-empty current `assistantTexts` instead.
  - confidence: 0.9
- **[P2] Remove or split the JSON string parser change:** `src/shared/chat-content.ts:18-24`
  - body: This unrelated parser path accepts serialized blocks with any `type` plus `text`, then recurses into logic that only keeps `type === "text"`. A serialized `[{"type":"output_text","text":"hello"}]` can become `null` instead of preserving the original string, so this regression should not ride along with the hook timeout fix.
  - confidence: 0.86
