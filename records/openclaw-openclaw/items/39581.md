## Summary

Keep open: the timeout hook bug still looks real on current main, but this PR’s transcript-wide `lastAssistant` guard can still fire after a no-output attempt when earlier assistant output exists. The branch also includes unrelated chat-content parser work already covered by https://github.com/openclaw/openclaw/pull/39583 and lacks real behavior proof.

## What This Changes

The branch adds a `lastAssistant` guard and usage coercion around embedded `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded `llm_output` emission on current-attempt assistant output or non-empty current assistant text, add focused regression coverage, and leave stringified chat-content parsing to the separate parser PR.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main can reach `runLlmOutput` with a prior transcript assistant and no current-attempt assistant output because the guard does not check current output.

## Solution Assessment

No. The PR targets a real bug, but `lastAssistant` is the wrong guard boundary and the unrelated parser change should be removed or handled in https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check uses the last assistant anywhere in the transcript. If a later attempt times out before producing output after an earlier assistant turn, this remains truthy and `llm_output` can still fire for a no-output attempt; guard on `currentAttemptAssistant` or non-empty current `assistantTexts` instead.
  - confidence: 0.9
- **[P2] Remove or split the JSON string parser change:** `src/shared/chat-content.ts:18-24`
  - body: This parser change is unrelated to the hook timeout fix and can regress existing string handling: a serialized `[{"type":"output_text","text":"hello"}]` is parsed, then dropped by the array extractor that only keeps `type === "text"`. Keep this PR focused and handle that parser work separately.
  - confidence: 0.86
