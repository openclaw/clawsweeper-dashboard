## Summary

Keep open: current main still has the embedded-runner `llm_output` no-output bug, but this PR is not correct as submitted because the guard can still use a stale `lastAssistant`, the branch carries unrelated parser work with a regression, and contributor real-behavior proof is missing.

## What This Changes

The PR guards embedded-runner `llm_output` hook emission on `lastAssistant`, coerces missing usage to `undefined`, and adds JSON-stringified chat-content parsing with tests.

## Best Possible Solution

Land a current-main patch that gates embedded-runner `llm_output` on current-attempt assistant output with focused regression coverage, and leave JSON-stringified content parsing to the dedicated parser PR or a corrected separate patch.

## Reproduction Assessment

Yes, by source inspection: current main tracks `currentAttemptAssistant` separately but emits `llm_output` with the session-level `lastAssistant` whenever hooks and the prompt-error gate pass. A live timeout scenario was not run because this review is read-only.

## Solution Assessment

No, not as submitted. The hook guard is in the right area, but it should gate on current-attempt output rather than session-level `lastAssistant`, and the unrelated parser change should be removed or fixed separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate the hook on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2007`
  - body: A timeout in a session that already has an earlier assistant message can still leave `lastAssistant` truthy, so this condition can still emit `llm_output` even though the current model attempt produced no output. Gate on the current-attempt assistant or current attempt text instead of the session's last assistant message.
  - confidence: 0.84
- **[P2] Keep unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The new string path recurses when every parsed item has `type` and `text`, but the array extractor only keeps blocks where `type === "text"`. A serialized block such as `[{"type":"output_text","text":"hello"}]` now parses and returns `null` instead of preserving the original string, so restrict recursion to supported text block shapes or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
