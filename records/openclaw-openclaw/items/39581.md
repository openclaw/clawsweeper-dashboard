## What This Changes

## Best Possible Solution

Land a focused fix that gates embedded-runner `llm_output` on current-attempt assistant output with regression coverage, and keep JSON-stringified parser work in https://github.com/openclaw/openclaw/pull/39583.

## Reproduction Assessment

Yes, at source level: current main computes `currentAttemptAssistant` but emits `llm_output` without requiring that current-attempt output exists. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No. The intended fix is narrow, but this patch uses session-level `lastAssistant` and bundles unrelated parser work; the safer path is a focused current-attempt guard plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: A timeout after an earlier assistant message can leave session-level `lastAssistant` truthy, so this condition can still emit `llm_output` even when the current model attempt produced no output. Gate on `currentAttemptAssistant` or current-attempt assistant text instead.
  - confidence: 0.9
- **[P2] Preserve unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The stringified-array path recurses for items with `type` and `text`, but the array extractor only accepts `type === "text"`. Payloads like `[{"type":"output_text","text":"hello"}]` can now collapse to `null` instead of preserving the original string, so only parse supported text blocks or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
