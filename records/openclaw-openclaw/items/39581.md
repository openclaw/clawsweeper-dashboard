## Summary

Keep open: the central embedded-runner timeout guard is still a valid source-reproducible bug fix, but this PR is not mergeable as submitted because it carries an unrelated chat-content parser regression and lacks after-fix real behavior proof.

## What This Changes

The PR guards embedded-runner `llm_output` hook emission on `lastAssistant`, coerces missing usage to `undefined`, and adds JSON-stringified chat-content parsing plus tests.

## Best Possible Solution

Land a narrow current-main patch that suppresses embedded-runner `llm_output` only when no assistant output exists, preserves the current prompt-error hook gate, adds focused regression coverage where feasible, and handles chat-content parsing separately.

## Reproduction Assessment

Yes. Source inspection shows `lastAssistant` is optional and current main still passes it to `runLlmOutput` without an output-presence guard, so an embedded-runner timeout before any assistant message can reproduce the bad hook payload.

## Solution Assessment

No, not as submitted. The hook guard is the right direction, but the branch should be rebased onto current main, keep the existing prompt-error gate, and remove or separately fix the unrelated chat-content parser change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate serialized blocks by extractable text:** `src/shared/chat-content.ts:18-24`
  - body: The new JSON-string path recurses when every parsed item merely has `type` and `text`, but the array extractor only keeps blocks where `type === "text"`. Serialized blocks like `[{"type":"output_text","text":"hello"}]` now parse and collapse to `null`, while mixed text/tool arrays still fall through as raw JSON; gate on extractable `type: "text"` blocks or keep the string unchanged.
  - confidence: 0.88
