## Summary

Keep open: the reported hook bug is still source-reproducible on current main, but this PR does not fix it narrowly enough and also carries an unrelated parser change with its own correctness risk. The branch needs contributor real-behavior proof and human-reviewed cleanup before merge.

## What This Changes

The PR adds a `lastAssistant` guard and usage fallback around embedded-runner `llm_output` emission, plus JSON-stringified chat-content parsing and helper tests.

## Best Possible Solution

Land a focused current-main patch that suppresses `llm_output` unless the current attempt produced assistant output, with regression coverage; keep JSON-stringified content parsing in its dedicated PR.

## Reproduction Assessment

Yes, by source inspection: current main computes `currentAttemptAssistant` separately but emits `llm_output` whenever the hook and prompt-error gates pass, without requiring current-attempt assistant output. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No, not as submitted. The hook guard should use current-attempt output rather than session-level `lastAssistant`, and the JSON-stringified parser change should be removed or handled through https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2007`
  - body: A timeout in a session that already has an earlier assistant message can leave `lastAssistant` truthy, so this condition can still emit `llm_output` even though the current model attempt produced no output. Gate on `currentAttemptAssistant` or current attempt text instead of the session's last assistant message.
  - confidence: 0.86
- **[P2] Keep unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The new string path recurses when every parsed item has `type` and `text`, but the array extractor only keeps blocks where `type === "text"`. A serialized block such as `[{"type":"output_text","text":"hello"}]` now parses and returns `null` instead of preserving the original string, so restrict recursion to supported text block shapes or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
