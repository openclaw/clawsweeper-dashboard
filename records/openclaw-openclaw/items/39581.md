## Summary

Keep open: the reported hook bug is valid, but this PR does not fully fix the current-attempt no-output case and also includes an unrelated chat-content parser change with a concrete regression risk.

## What This Changes

The PR adds a `lastAssistant` guard and usage fallback around embedded-runner `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded-runner `llm_output` emission on current-attempt assistant output or visible current `assistantTexts`, preserve normal output hook behavior, add focused regression coverage, and remove or split the parser change.

## Reproduction Assessment

Yes, by source inspection. Current main computes `lastAssistant` from the full transcript and emits `llm_output` whenever hooks exist, so a timeout or rejection after a prior assistant turn can still emit a no-current-output event.

## Solution Assessment

No. The PR targets the right bug, but the guard should use current-attempt output rather than transcript-wide `lastAssistant`, and the unrelated parser change should be removed or split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check looks at the last assistant anywhere in the transcript. If an embedded run times out before producing output after a previous assistant turn, this remains truthy and `llm_output` can still fire for a no-output current attempt. Guard on `currentAttemptAssistant` or non-empty current `assistantTexts` instead, and cover both timeout/no-output and normal-output cases.
  - confidence: 0.9
- **[P2] Remove or narrow the JSON-string parser recursion:** `src/shared/chat-content.ts:18-24`
  - body: This new branch parses any JSON array item that has `type` and `text`, but the existing array extractor only keeps blocks with `type === "text"`. A serialized payload like `[{"type":"output_text","text":"hello"}]` can now return `null` instead of preserving the original string, and the parser behavior is unrelated to the timeout hook fix.
  - confidence: 0.86
