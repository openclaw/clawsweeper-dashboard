## Label Justifications

- `P2`: This is a normal-priority agent hook bug with limited blast radius, not an emergency runtime outage.
- `merge-risk: 🚨 message-delivery`: Merging the PR as written can still mis-emit or suppress `llm_output` hook events in timeout/no-output attempts and can drop serialized content-block text.
- `merge-risk: 🚨 compatibility`: The PR changes shared chat-content parsing behavior and predates current coercion hardening, so existing payload handling can change during upgrade.

## Summary

Source review confirms the timeout hook problem is plausible, but the PR is not merge-ready because its guard uses session-level `lastAssistant` instead of current-attempt output and it bundles a flawed parser change that overlaps the separate vLLM parser PR.

## What This Changes

The PR adds a truthy `lastAssistant` guard around embedded-runner `llm_output` hook emission, coerces hook usage to `undefined`, and adds JSON-stringified chat-content parsing plus tests.

## Best Possible Solution

Land a focused fix that gates embedded-runner `llm_output` on current-attempt assistant output with regression coverage, and leave JSON-stringified chat-content parsing to the separate parser PR.

## Reproduction Assessment

Yes at source level: current main tracks `currentAttemptAssistant` separately but emits `llm_output` without requiring current-attempt assistant output. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No. The hook fix should gate on `currentAttemptAssistant` or current-attempt assistant text, and the unrelated JSON parser work should stay in https://github.com/openclaw/openclaw/pull/39583.

Overall correctness: patch is incorrect

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: A timeout after an earlier assistant message can leave session-level `lastAssistant` truthy, so this condition can still emit `llm_output` even when the current model attempt produced no output. Gate on `currentAttemptAssistant` or current-attempt assistant text instead.
  - confidence: 0.9

Payloads like `[{"type":"output_text","text":"hello"}]` can now collapse to `null` instead of preserving the original string, so only parse supported text blocks or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
