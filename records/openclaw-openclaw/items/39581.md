## Summary

Keep open: the underlying hook bug is source-reproducible on current main, but this PR is not mergeable as submitted because its guard uses session-level `lastAssistant`, it carries unrelated parser work with a regression, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds a `lastAssistant` truthy guard around embedded-runner `llm_output` hook emission, coerces missing usage to `undefined`, and adds JSON-stringified chat-content parsing with tests.

## Best Possible Solution

Land a current-main patch that suppresses `llm_output` unless the current attempt produced assistant output, with focused regression coverage; keep JSON-stringified content parsing in the dedicated parser PR or a corrected separate patch.

## Reproduction Assessment

Yes, by source inspection: current main tracks `currentAttemptAssistant` separately but emits `llm_output` with session-level `lastAssistant` whenever hooks and the prompt-error gate pass. A live timeout scenario was not run in this read-only review.

## Solution Assessment

No, not as submitted. The hook guard should be based on current-attempt output, and the unrelated JSON-stringified parser change should be removed or handled through https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2007`
  - body: A timeout in a session that already has an earlier assistant message can still leave `lastAssistant` truthy, so this condition can still emit `llm_output` even though the current model attempt produced no output. Gate on the current-attempt assistant or current attempt text instead of the session's last assistant message.
  - confidence: 0.84
- **[P2] Keep unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The new string path recurses when every parsed item has `type` and `text`, but the array extractor only keeps blocks where `type === "text"`. A serialized block such as `[{"type":"output_text","text":"hello"}]` now parses and returns `null` instead of preserving the original string, so restrict recursion to supported text block shapes or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
