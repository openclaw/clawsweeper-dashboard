## Summary

Keep open: current main still lacks the documented output guard for embedded-runner `llm_output`, but this PR is not mergeable as submitted because it gates on session-level `lastAssistant`, mixes in parser work tracked separately, and lacks after-fix real timeout proof.

## What This Changes

The PR adds a `lastAssistant` guard and usage normalization around embedded-runner `llm_output`, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Land a focused current-main fix that suppresses embedded-runner `llm_output` unless the current attempt produced assistant output, with regression coverage; leave serialized content parsing to the dedicated parser PR.

## Reproduction Assessment

Yes, by source inspection: current main has a helper for current-attempt assistant output but the hook emission condition does not use it. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No, not as submitted. The guard should use `currentAttemptAssistant` or current-attempt text rather than session-level `lastAssistant`, and the serialized chat-content parser change should be removed or handled through https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2007`
  - body: A timeout after an earlier assistant message can leave session-level `lastAssistant` truthy, so this condition can still emit `llm_output` even though the current model attempt produced no output. Gate on `currentAttemptAssistant` or current-attempt text instead of the session's last assistant message.
  - confidence: 0.86
- **[P2] Keep unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The new string path recurses when every parsed item has `type` and `text`, but the array extractor only keeps blocks where `type === "text"`. A serialized block such as `[{"type":"output_text","text":"hello"}]` now parses and returns `null` instead of preserving the original string, so restrict recursion to supported text block shapes or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
