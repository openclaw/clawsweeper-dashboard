## Summary

Keep open: the PR targets a real embedded `llm_output` timeout bug, but the proposed guard is not sufficient after prior assistant turns, the branch includes an unrelated chat-content parser change with a concrete regression risk, and external real-behavior proof is missing.

## What This Changes

The PR adds a `lastAssistant` guard and usage coercion to embedded `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded `llm_output` emission on current-attempt assistant output or non-empty current assistant text, add focused regression coverage, and leave stringified chat-content parsing to https://github.com/openclaw/openclaw/pull/39583.

## Reproduction Assessment

Yes, source-level. Current main computes transcript-wide `lastAssistant` separately from `currentAttemptAssistant` and still emits `llm_output` without a current-output guard, so a timeout after a prior assistant turn can reach the hook path.

## Solution Assessment

No. The PR targets the right bug, but `lastAssistant` is the wrong guard boundary and the unrelated parser change should be removed or handled in https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check uses the last assistant anywhere in the transcript. If a later attempt times out before producing output after an earlier assistant turn, this remains truthy and `llm_output` can still fire for a no-output attempt; guard on current-attempt output or non-empty current assistant text instead.
  - confidence: 0.9
- **[P2] Avoid dropping stringified output_text blocks:** `src/shared/chat-content.ts:18-24`
  - body: The new JSON string branch treats any item with `type` and `text` as extractable, then recurses into an extractor that only keeps `type === "text"`. A serialized `[{"type":"output_text","text":"hello"}]` now becomes `null` instead of preserving the text/string, so this parser change should be split out or fixed before merging.
  - confidence: 0.86
