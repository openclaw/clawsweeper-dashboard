## Summary

Keep open: this PR targets a real embedded `llm_output` timeout bug, but the patch still guards on transcript-wide `lastAssistant`, mixes in unrelated chat-content parsing, and lacks contributor real-behavior proof.

## What This Changes

The branch adds a `lastAssistant` guard and usage coercion to embedded `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded `llm_output` emission on current-attempt assistant output or non-empty current assistant text, add focused regression coverage, and leave JSON-stringified chat-content parsing to https://github.com/openclaw/openclaw/pull/39583.

## Reproduction Assessment

Yes, by source inspection. Current main can reach `runLlmOutput` with hooks enabled even when the current attempt produced no assistant output, because `lastAssistant` is transcript-wide and `currentAttemptAssistant` is computed separately.

## Solution Assessment

No. The PR targets the right bug class, but the guard should use current-attempt output rather than transcript-wide `lastAssistant`, and the unrelated parser change should be split out.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check uses the last assistant anywhere in the transcript. If a later attempt times out before producing output after an earlier assistant turn, this remains truthy and `llm_output` can still fire for a no-output attempt; guard on `currentAttemptAssistant` or non-empty `assistantTexts` instead.
  - confidence: 0.9

  - confidence: 0.86
