## Summary

Keep open: current main still lacks a current-attempt output guard for embedded `llm_output`, and this PR's `lastAssistant` check is the wrong boundary. The branch also carries unrelated chat-content parser work that is already tracked separately.

## What This Changes

The PR adds a `lastAssistant` guard and usage coercion to embedded `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded `llm_output` emission on current-attempt assistant output or non-empty current assistant text, add focused regression coverage, and leave stringified chat-content parsing to the separate PR.

## Reproduction Assessment

Yes, by source inspection. A current-main embedded run with a prior assistant message and no current-attempt assistant output can still reach `runLlmOutput` because `lastAssistant` is transcript-wide and the hook condition lacks a current-output guard.

## Solution Assessment

No. The PR targets a real bug, but `lastAssistant` is not the right guard boundary and the unrelated parser change should be removed or handled in https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check uses the last assistant anywhere in the transcript. If a later attempt times out before producing output after an earlier assistant turn, this remains truthy and `llm_output` can still fire for a no-output attempt; guard on `currentAttemptAssistant` or non-empty current `assistantTexts` instead.
  - confidence: 0.9
- **[P2] Remove or split the JSON string parser change:** `src/shared/chat-content.ts:18-24`
  - body: This parser change is unrelated to the hook timeout fix and can regress existing string handling: a serialized `[{"type":"output_text","text":"hello"}]` is parsed, then dropped by the array extractor that only keeps `type === "text"`. Keep this PR focused and handle that parser work separately.
  - confidence: 0.86
