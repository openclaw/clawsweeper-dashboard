## Summary

Keep open. The embedded-runner bug still exists on current main, but this PR is not safe to merge as-is: the proposed guard keys off transcript-wide `lastAssistant`, so it can still emit `llm_output` after a no-output timeout when an earlier assistant message exists, and the PR also bundles an unrelated chat-content parser change with concrete data-loss edge cases.

## What This Changes

The PR adds a `lastAssistant` guard around embedded-runner `llm_output` hook emission, coerces hook usage to `undefined`, and adds JSON-stringified content-block parsing plus tests for shared chat content extraction.

## Best Possible Solution

Update the PR to guard `llm_output` on current-attempt assistant output, not transcript-wide history, preserve normal-output hook emission, add focused embedded-runner regression coverage for timeout/no-output and normal-output cases, and remove or split the chat-content JSON parsing change unless it is fixed and justified separately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run an embedded attempt with a prior assistant message in the transcript, make `activeSession.prompt(...)` reject or time out before adding a current assistant message, register an `llm_output` hook, and assert the hook is not called; current main and this PR's `lastAssistant` guard both fail that prior-history variant.

## Solution Assessment

No. The proposed guard is not the narrowest maintainable fix because it uses `lastAssistant`, which is full-history state; the safer solution is to guard on `currentAttemptAssistant` or non-empty current `assistantTexts`, with regression tests, and keep unrelated content parsing out of this PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check looks at the last assistant anywhere in the transcript. If a run times out before producing output after a previous assistant turn, this remains truthy and `llm_output` still fires with stale/no current output. Use `currentAttemptAssistant` or non-empty current `assistantTexts` for the guard and cover both timeout/no-output and normal-output cases.
  - confidence: 0.9
- **[P2] Avoid parsing non-text content blocks as text arrays:** `src/shared/chat-content.ts:18-24`
  - body: The JSON-string path recurses for any array item that has `type` and `text`, but the extractor only emits blocks whose type is exactly `text`. A serialized Responses-style `[{"type":"output_text","text":"hello"}]` would now parse and return null instead of preserving the string, so this unrelated parser change can drop output. Drop it or define supported block types with tests.
  - confidence: 0.86
