## Summary

Keep open: the PR targets a real source-reproducible `llm_output` hook bug, but the proposed guard still uses session-level `lastAssistant`, bundles an unrelated parser change with its own correctness risk, and has no after-fix real timeout proof.

## What This Changes

The PR adds a `lastAssistant` guard and usage fallback to embedded-runner `llm_output` emission, and adds JSON-stringified chat-content parsing with tests.

## Best Possible Solution

Land a focused fix that gates embedded-runner `llm_output` on current-attempt assistant output with regression coverage, and leave JSON-stringified chat-content parsing to the separate parser PR.

## Reproduction Assessment

Yes, at source level: current main tracks `currentAttemptAssistant` separately but emits `llm_output` without requiring current-attempt assistant output. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No. The hook fix should gate on `currentAttemptAssistant` or current-attempt assistant text while preserving current main's prompt-error guard, and the unrelated parser work should stay in https://github.com/openclaw/openclaw/pull/39583.
