## Summary

Keep open because the documented bug remains present on current main, but this PR is not mergeable as submitted: it lacks real timeout proof, its guard still uses session-level state, and it mixes in a parser change with a known edge-case regression.

## What This Changes

The PR guards embedded-runner `llm_output` hook emission with `lastAssistant`, coerces usage to `undefined`, and adds JSON-stringified chat-content parsing plus tests.

## Best Possible Solution

Land a focused current-main fix that suppresses embedded-runner `llm_output` unless the current attempt produced assistant output, with regression coverage; keep JSON-stringified content parsing in https://github.com/openclaw/openclaw/pull/39583.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main computes `currentAttemptAssistant` separately but emits `llm_output` whenever hooks exist and the prompt-error gate passes. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No, not as submitted. The hook guard should use current-attempt output rather than session-level `lastAssistant`, and the JSON-stringified parser change should be removed or handled through https://github.com/openclaw/openclaw/pull/39583.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate llm_output on current-attempt output:** `src/agents/pi-embedded-runner/run/attempt.ts:2007`
  - body: A timeout after an earlier assistant message can leave session-level `lastAssistant` truthy, so this condition can still emit `llm_output` even though the current model attempt produced no output. Gate on `currentAttemptAssistant` or current-attempt assistant text instead.
  - confidence: 0.87
- **[P2] Keep unsupported serialized blocks as text:** `src/shared/chat-content.ts:18-24`
  - body: The stringified-array path recurses for any item with `type` and `text`, but the array extractor only accepts `type === "text"`. Payloads like `[{"type":"output_text","text":"hello"}]` now collapse to `null` instead of preserving the original string, so only parse supported text blocks or leave unsupported serialized arrays unchanged.
  - confidence: 0.88
