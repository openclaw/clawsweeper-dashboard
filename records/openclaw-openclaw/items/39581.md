## Summary

Keep open: current main still lacks the embedded-runner `lastAssistant` guard, so the primary bug remains source-reproducible, but this PR is not mergeable as-is because it includes an unrelated parser change with a P2 regression and lacks real timeout proof.

## What This Changes

The PR guards embedded-runner `llm_output` hook emission on `lastAssistant`, coerces missing usage to `undefined`, and adds JSON-stringified chat-content parsing with tests.

## Best Possible Solution

Land a current-main patch that adds the `lastAssistant` guard to the existing hook predicate, adds focused regression coverage where feasible, and handles JSON-stringified chat content in the separate dedicated parser PR.

## Reproduction Assessment

Yes, by source inspection: current main declares `lastAssistant` optional, passes it to `llm_output` without requiring assistant output, and docs say the hook runs only after assistant output exists. I did not run a live timeout scenario because this review is read-only.

## Solution Assessment

No, not as submitted. The hook guard is the right direction, but the branch is stale/conflicting and should preserve the current prompt-error gate while removing or separately fixing the unrelated parser change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Restrict serialized block parsing before recursing:** `src/shared/chat-content.ts:18-24`
  - body: The new JSON-string path recurses whenever every parsed item has `type` and `text`, but the extractor then keeps only `type === "text"`. A serialized payload like `[{"type":"output_text","text":"hello"}]` now parses and collapses to `null` instead of preserving the original string, so require a supported text block before recursing or leave the string unchanged.
  - confidence: 0.88
