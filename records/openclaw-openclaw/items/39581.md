## Summary

Keep open. Current `main` still has the embedded-runner `llm_output` no-output hook path, but this PR is not merge-ready because it guards on transcript-wide `lastAssistant` and bundles an unrelated shared chat-content parser change with edge-case regressions.

## What This Changes

The PR adds a `lastAssistant` guard and `usage ?? undefined` around embedded-runner `llm_output` hook emission, plus JSON-stringified content-block parsing and tests for shared chat content extraction.

## Best Possible Solution

Guard embedded-runner `llm_output` emission on current-attempt assistant output, not transcript history, and add focused regression coverage for timeout/no-output and normal-output hook behavior. Keep JSON-stringified chat-content parsing out of this PR unless it is split into a separately justified and tested parser change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an embedded run with a prior assistant message in the transcript, an `activeSession.prompt(...)` timeout or rejection before any current assistant output is added, and an `llm_output` hook registered; current `main` still reaches hook emission, and this PR's `lastAssistant` guard would still be truthy in that prior-history case.

## Solution Assessment

No. The PR targets a real bug, but the proposed guard is not the best fix because it checks full-transcript `lastAssistant`; the safer fix is to guard on `currentAttemptAssistant` or non-empty current `assistantTexts` and remove or split the unrelated content parser change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check looks at the last assistant anywhere in the transcript. If an embedded run times out before producing output after a previous assistant turn, this remains truthy and `llm_output` can still fire for a no-output current attempt. Guard on `currentAttemptAssistant` or non-empty current `assistantTexts` instead, and cover both timeout/no-output and normal-output cases.
  - confidence: 0.9
- **[P2] Avoid parsing non-text content blocks as text arrays:** `src/shared/chat-content.ts:18-24`
  - body: The JSON-string path recurses when every item merely has `type` and `text`, but the downstream extractor only emits blocks whose type is exactly `text`. A serialized payload like `[{"type":"output_text","text":"hello"}]` would now parse and return `null` instead of preserving the original string, so this unrelated parser change can drop output. Remove it from this PR or define and test the supported block types precisely.
  - confidence: 0.86
