## Summary

Keep open. The timeout-hook bug is plausible and current main does not fully solve the embedded-runner `llm_output` case, but this PR should not merge as-is because the added JSON-string parser can drop serialized text-bearing content and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a `lastAssistant` guard around the embedded runner `llm_output` hook and also adds JSON-stringified content-block parsing plus tests for `extractTextFromChatContent`.

## Best Possible Solution

Rebase onto current main and implement the no-output hook guard in the current embedded runner hook path with focused timeout-hook regression coverage; leave JSON-string content parsing out unless it is fixed, justified, and tested separately.

## Reproduction Assessment

Yes from source inspection, though I did not run a live repro. A prompt timeout before any assistant message can leave `lastAssistant` undefined while the current embedded hook predicate still permits `llm_output`, so a handler that enumerates `event.lastAssistant` can throw.

## Solution Assessment

No, not as submitted. The `lastAssistant` guard is directionally right, but the PR needs to be rebased onto the current helper path and the unrelated JSON-string parser regression should be removed or repaired with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve serialized non-text content blocks:** `src/shared/chat-content.ts:23`
