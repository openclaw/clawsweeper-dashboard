## Summary

Keep open: current main still appears to violate the documented `llm_output` hook contract on no-output attempts, but this PR is not merge-ready because it lacks real timeout proof, is stale against the current hook predicate, and includes an unrelated parser regression.

## What This Changes

The PR guards embedded-runner `llm_output` hook emission on `lastAssistant`, coerces missing usage to `undefined`, and adds JSON-stringified chat-content parsing plus tests.

## Best Possible Solution

Land a current-main patch that adds the `lastAssistant` guard to the existing hook predicate, keeps the prompt-error gate, adds focused regression coverage where feasible, and handles serialized chat-content parsing separately.

## Reproduction Assessment

Yes, by source inspection rather than execution. Current main declares `lastAssistant` as optional and still calls `runLlmOutput` without requiring assistant output, while the docs say the hook runs only after assistant output exists.

## Solution Assessment

No, not as submitted. The hook guard is the right direction, but the PR should be rebased onto current main, preserve the existing prompt-error gate, and remove or separately fix the unrelated chat-content parser change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate serialized blocks by extractable text:** `src/shared/chat-content.ts:18-24`
  - body: The new JSON-string path recurses whenever every parsed item has `type` and `text`, but the downstream extractor only keeps `type === "text"`. A serialized payload like `[{"type":"output_text","text":"hello"}]` now parses and then collapses to `null`, losing text that current main would have preserved as a string; require an extractable `text` block or leave the string unchanged.
