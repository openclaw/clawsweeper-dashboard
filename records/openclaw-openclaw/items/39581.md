## Summary

Keep open: current main still can emit `llm_output` with an undefined `lastAssistant` despite documented output-only semantics, so the hook guard is useful, but this PR lacks external real-behavior proof and carries unrelated parser logic with a correctness regression.

## What This Changes

The PR guards embedded-runner `llm_output` emission on `lastAssistant`, coerces missing usage to `undefined`, and adds JSON-stringified chat-content parsing with tests.

## Best Possible Solution

Land a current-main patch that combines the `lastAssistant` guard with the existing prompt-error gate and focused regression coverage, while leaving JSON-stringified content parsing to https://github.com/openclaw/openclaw/pull/39583 or a corrected dedicated patch.

## Reproduction Assessment

Yes, by source inspection: current main declares `lastAssistant` optional, then sends it in `runLlmOutput` whenever hooks exist and the prompt-error gate passes; docs say `llm_output` runs only after assistant output exists. A live timeout scenario was not run because this review is read-only.

## Solution Assessment

No, not as submitted. The hook guard is the right direction, but the PR should preserve the current prompt-error gate and remove or separately fix the unrelated parser change.

## Review Findings
