## What This Changes

The PR adds a `lastAssistant` guard and `usage ?? undefined` to embedded-runner `llm_output` emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Land a focused hook fix that gates `llm_output` on current-attempt assistant output with regression coverage, and keep JSON-stringified content parsing in the dedicated parser PR or a separate narrow change.

## Reproduction Assessment

Yes, at source level: current main derives `lastAssistant` from the full transcript, derives `currentAttemptAssistant` only after `prePromptMessageCount`, and emits `llm_output` without requiring current-attempt output. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No. The hook fix should gate on current-attempt assistant output, and the unrelated JSON parser work should be split out to https://github.com/openclaw/openclaw/pull/39583 or another dedicated parser change.
