## Summary

Keep open: current main still does not require current-attempt assistant output before emitting `llm_output`, but this PR is not merge-ready because its guard can still use stale session output, it bundles overlapping parser changes, and it lacks real timeout proof.

## What This Changes

The PR guards embedded-runner `llm_output` hook emission with `lastAssistant`, normalizes hook usage to `undefined`, and adds JSON-stringified chat-content parsing plus tests.

## Best Possible Solution

Land a focused rebased hook fix that gates `llm_output` on current-attempt assistant output with regression coverage and real timeout proof, while leaving JSON-stringified chat-content parsing to https://github.com/openclaw/openclaw/pull/39583 or a dedicated parser PR.

## Reproduction Assessment

Yes, at source level: current main selects `lastAssistant` from the full session and `currentAttemptAssistant` from only post-prompt messages, then emits `llm_output` without requiring current-attempt output. I did not run a live timeout scenario in this read-only review.

## Solution Assessment

No, as proposed. The hook fix should gate on current-attempt assistant output, and the unrelated JSON parser work should be removed from this PR and handled through https://github.com/openclaw/openclaw/pull/39583.
