## Summary

Keep this PR open, but it still needs changes: the reported hook bug is real on current main, while the proposed guard uses transcript-wide history and the PR also carries an unrelated chat-content parser change with a text-dropping edge case.

## What This Changes

The PR adds a `lastAssistant` guard and usage coercion around embedded-runner `llm_output` hook emission, plus JSON-stringified chat-content parsing and tests.

## Best Possible Solution

Guard embedded-runner `llm_output` emission on current-attempt assistant output or visible current `assistantTexts`, preserve normal output hook behavior, add focused regression coverage, and leave JSON-stringified content parsing to a separate justified change.

## Reproduction Assessment

Yes, by source inspection. A current-main embedded run with a previous assistant message, a timeout or rejection before current assistant output, and a registered `llm_output` hook can still reach hook emission because `lastAssistant` is transcript-wide.

## Solution Assessment

No. The PR targets the right bug, but the guard should use current-attempt output rather than transcript-wide `lastAssistant`, and the parser change should be removed or split.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard on current-attempt output instead:** `src/agents/pi-embedded-runner/run/attempt.ts:2009`
  - body: The new `&& lastAssistant` check looks at the last assistant anywhere in the transcript. If an embedded run times out before producing output after a previous assistant turn, this remains truthy and `llm_output` can still fire for a no-output current attempt. Guard on `currentAttemptAssistant` or non-empty current `assistantTexts` instead, and cover both timeout/no-output and normal-output cases.
  - confidence: 0.9
- **[P2] Remove or split the JSON-string parser change:** `src/shared/chat-content.ts:18-24`
  - body: This PR now parses JSON arrays whose items have `type` and `text`, but the downstream extractor only keeps blocks with `type === "text"`. A serialized payload such as `[{"type":"output_text","text":"hello"}]` can therefore return `null` instead of preserving the original string, and this parser behavior is unrelated to the timeout hook fix.
  - confidence: 0.86
