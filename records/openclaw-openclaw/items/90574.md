## Summary

Keep open: the PR is a focused and useful bug fix, but the current branch still gates the GPT-5.5 Azure tool-payload workaround on a literal provider id, so custom provider ids that point at Azure hosts can keep sending the rejected `reasoning_effort` shape. The supplied live proof is also still too weak for merge because it shows a gateway model run, not this branch/build exercising a Chat Completions tool-call payload.

## What This Changes

The PR changes OpenAI Chat Completions request construction to omit `reasoning_effort` for Azure GPT-5.5 tool payloads and adds regression tests for direct ids, deployment aliases, custom opt-ins, and no-tool payloads.

## Best Possible Solution

Use the existing endpoint classification to detect Azure GPT-5.5 tool payloads, add a regression for a custom provider id on an Azure host, and pair it with redacted branch-level tool-call proof.

## Reproduction Assessment

Yes at source level: current main only skips GPT-5.4-mini, and the PR body provides the Azure 400 for GPT-5.5 Chat Completions tools. I did not live-test Azure credentials in this read-only review.

## Solution Assessment

No. The fix is in the right builder, but the Azure detector should use the already-resolved endpoint class rather than `model.provider`, and proof should show the branch's tool-call path.
