## Summary

Keep open: current main and the latest shipped tag still have partial source-specific external-content hardening, but the shared trust-boundary contract for non-owner/current message text, generic tool results, and stable model guidance remains unimplemented. Related work at https://github.com/openclaw/openclaw/pull/78521, https://github.com/openclaw/openclaw/pull/64720, and https://github.com/openclaw/openclaw/pull/39231 is open, partial, and blocked by security/proof or design issues, so it is not a shipped replacement.

## What This Changes

The issue requests structural prompt-injection defenses and stable trust guidance for untrusted tool results, incoming messages, web/file/API content, and persisted transcript content.

## Best Possible Solution

Adopt a maintainer/secops-approved design that composes existing external-content wrappers with delimiter-safe generic tool-result handling, channel/message trust semantics, stable prompt guidance, and the open provenance/CaMeL decisions, then land focused PRs with cross-provider tests and real behavior proof.

## Reproduction Assessment

Yes. Source inspection against current main and v2026.5.12 shows non-owner/current message text and generic toolResult text still flow through shared transport/provider conversion as ordinary text apart from invalid-surrogate sanitization.

## Solution Assessment

Unclear. Structural data/instruction separation is directionally right, but maintainers should choose the exact design because non-owner semantics, delimiter spoofing, provider payload shapes, and open CaMeL/provenance work all affect the maintainable fix.
