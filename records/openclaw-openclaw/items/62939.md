## Summary

Keep open. Current main has useful targeted hardening for web fetches, channel metadata, media/file extraction, and external-content marker spoofing, but it still lacks the requested shared prompt-boundary contract for non-owner current messages, generic/open-world tool results, and stable model guidance; the narrow linked implementation PR is closed unmerged and the broader CaMeL PR remains open, not shipped.

## What This Changes

Request to add structural trust boundaries, stable prompt guidance, source classification, and injection-pattern handling for untrusted tool results and inbound messages.

## Best Possible Solution

Adopt or explicitly supersede this with a core-owned prompt-boundary design covering stable guidance, non-owner current-message marking, open-world tool-result classification, delimiter-safe framing, and cross-provider regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is to pass non-owner prompt text or a toolResult containing injection text through the current transport/provider conversion paths; current main preserves it as ordinary provider text without the requested shared boundary.

## Solution Assessment

Unclear. Structural data/instruction separation is a reasonable direction, but the exact wrapper layer, delimiter format, escaping rules, and relationship to the open CaMeL work need maintainer/secops design before implementation.
