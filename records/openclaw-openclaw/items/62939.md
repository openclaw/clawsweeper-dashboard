## Summary

Keep open. Current main has useful partial hardening for external web/file/hook content and inbound system-marker spoofing, but it still lacks the requested shared trust-boundary contract for non-owner current messages, generic tool results, and stable prompt guidance; related #62973 is closed unmerged and #39231 is open rather than shipped.

## What This Changes

Add structural trust boundaries and optional validation/config around untrusted tool results, external fetches, file/API content, and non-owner messages so models treat them as data rather than instructions.

## Best Possible Solution

Adopt or explicitly supersede this with a core-owned prompt-boundary design covering stable pre-cache guidance, non-owner current-message marking, generic tool-result classification, delimiter-safe framing, and provider regression coverage while preserving existing external-content wrappers.

## Reproduction Assessment

Yes. A source-level reproduction is to pass a non-owner prompt or a generic `toolResult` containing injection text through the current embedded runner and transport/provider conversion paths; current main preserves it as ordinary provider text without the requested shared boundary.

## Solution Assessment

Unclear. Structural data/instruction separation is directionally right, but the specific wrapper/tag layer is not yet proven as the best fix because delimiter escaping, provider payload shape, opt-out policy, and #39231 overlap need maintainer/security direction.
