## Summary

Keep open: current main and the latest shipped release have useful external-content wrappers, but they still do not implement the requested shared trust-boundary contract for non-owner current messages, generic tool-result replay, and stable system-prompt guidance; the narrow linked PR is closed unmerged and the broader CaMeL PR remains open with review blockers.

## What This Changes

Requested core prompt-injection hardening at tool-result and message boundaries with structural untrusted-content framing, stable system-prompt guidance, trust classification, validation, and an opt-out config path.

## Best Possible Solution

Define or explicitly supersede a core prompt-boundary design that covers stable pre-cache guidance, escape-safe trust framing or taint provenance, non-owner current-message handling, generic tool-result/provider replay coverage, and focused regression tests while preserving existing external-content sanitizers.

## Reproduction Assessment

Yes by source inspection. Current main still sends non-owner current prompt text through buildCurrentInboundPrompt to activeSession.prompt, and replayed user/toolResult content through transformTransportMessages into OpenAI/Anthropic provider payloads without a shared trust-boundary wrapper.

## Solution Assessment

Unclear. Structural separation is directionally reasonable, but XML-style delimiters alone are not enough until maintainers decide escape safety, provider payload shape, stored replay behavior, and the relationship to the open CaMeL approach.
