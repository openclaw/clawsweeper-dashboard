## Summary

Keep open. Current main has useful source-specific wrappers for web/file/hook content, but it still lacks the requested shared trust-boundary contract for non-owner message bodies, generic tool results, and stable system-prompt guidance; the related narrow PR is closed unmerged and the broader CaMeL PR remains open with unresolved security-review blockers.

## What This Changes

The issue requests structural prompt-injection defenses that mark externally sourced message, tool-result, web-fetch, and file content as untrusted data rather than model instructions.

## Best Possible Solution

Define or explicitly supersede a core prompt-boundary design that includes stable guidance, trust classification, escape-safe framing, provider payload coverage, replay/current-turn handling, and focused regression tests.

## Reproduction Assessment

Yes. A current-main source repro traces non-owner prompt text through `buildCurrentTurnPrompt` to `activeSession.prompt`, and traces user/toolResult replay through `transformTransportMessages` into OpenAI/Anthropic payload conversion without a shared trust boundary.

## Solution Assessment

Unclear. Structural separation is directionally appropriate, but the exact XML-style delimiter approach is not enough until maintainer/secops decide escape/nonce safety, storage/replay behavior, provider payload shape, and the relationship to the open CaMeL plan.
