## Summary

Keep open: current main has useful source-specific external-content hardening, but the requested shared trust-boundary contract for non-owner message text, generic tool results, and stable model guidance is still not implemented.

## What This Changes

Request to add structural untrusted-content boundaries and stable model guidance for tool results, inbound messages, web/file/API content, plus optional validation/config behavior.

## Best Possible Solution

Adopt or explicitly supersede this with a maintainer-approved prompt-boundary design covering stable guidance, delimiter-safe framing, non-owner current-message semantics, generic tool-result/source classification, and cross-provider regression coverage while preserving existing external-content wrappers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass non-owner prompt text or a generic toolResult containing instruction-looking content through transformTransportMessages and the OpenAI/Anthropic conversion paths, and it remains ordinary provider text apart from invalid-surrogate sanitization.

## Solution Assessment

Unclear. Structural data/instruction separation is directionally right, but maintainers/security should choose the exact design because non-owner semantics, delimiter safety, provider payload shapes, and the open CaMeL proposal all affect the maintainable fix.
