## Summary

Keep open. Current main has useful partial external-content hardening for web fetch/search, browser, and file-input paths, but the broader requested shared trust-boundary contract is still missing for non-owner current-message semantics, generic/open-world tool results, and cache-stable model guidance; the narrow implementation PR was closed unmerged and the broader CaMeL architecture PR remains open.

## What This Changes

Request to add structural prompt-injection defenses that classify and frame untrusted messages, tool results, web/file/API content, plus stable prompt guidance and optional configuration.

## Best Possible Solution

Adopt or explicitly supersede this with a maintainer-approved prompt-boundary design that covers stable guidance, delimiter-safe framing, non-owner current-message semantics, generic tool-result/source classification, and cross-provider regression coverage while preserving existing external-content wrappers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass non-owner prompt text or a generic toolResult containing instruction-looking content through the current transform and OpenAI/Anthropic conversion paths, and it remains ordinary provider text apart from invalid-surrogate sanitization.

## Solution Assessment

Unclear. Structural data/instruction separation is directionally right, but the exact implementation should be decided by maintainers/security because non-owner message semantics, delimiter escaping, provider payload shapes, and the open CaMeL proposal all affect the safest design.
