## Summary

Keep open. Current main has partial prompt-injection hardening for specific external-content paths, but it still does not implement the requested shared trust-boundary contract for non-owner message bodies, generic tool results, or stable system-prompt guidance. The narrow linked implementation PR is closed unmerged, and the broader CaMeL PR remains open rather than shipped.

## What This Changes

The issue requests structural trust delimiters, stable prompt guidance, source-trust classification, and injection-pattern handling for externally sourced tool results, non-owner messages, web/API content, and persisted context.

## Best Possible Solution

Implement or explicitly supersede this with a core-owned prompt-injection boundary design: cache-stable trust guidance, tested non-owner current-message marking, shared classification/wrapping for open-world tool results, delimiter escaping, and provider-shape regression coverage across OpenAI, Anthropic, and Google while preserving existing external-content sanitizers.

## Reproduction Assessment

Yes for the structural gap: a source-level reproduction is to pass a user message or toolResult containing injection text through transformTransportMessages and the OpenAI/Anthropic converters, where current main preserves it as plain provider text. A live model exploit is nondeterministic, but the missing boundary behavior is directly observable in code.

## Solution Assessment

Unclear. The requested direction is valid, but the exact XML-like wrapper approach is not sufficient by itself until delimiter escaping, provider transcript shape, prompt-cache placement, and the relationship to the open CaMeL PR are settled by maintainers/security reviewers.
