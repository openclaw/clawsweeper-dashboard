Full review comments:

- **[P1] Extend the strict model schema:** `src/config/types.agents-shared.ts:39`
  - body: `fallbackOnErrors` is added to `AgentModelConfig`, but `AgentModelSchema` is still strict and only accepts `primary`, `fallbacks`, and `timeoutMs`. A user following the new config examples would hit an unrecognized-key validation error before the fallback runner can read the setting.
  - confidence: 0.92
- **[P1] Honor defaults.model fallbackOnErrors:** `src/agents/agent-scope.ts:259-263`
  - body: This resolver reads only `resolveAgentConfig(...).model`. Current `resolveAgentConfig` exposes only the per-agent entry model, so the documented `agents.defaults.model.fallbackOnErrors` setting is ignored for agents without an explicit model object.
  - confidence: 0.89
- **[P2] Fail fast in image fallback for disallowed errors:** `src/agents/model-fallback.ts:834-839`
  - body: The image path passes `fallbackOnErrors` into `runFallbackAttempt`, but after a rejected status comes back as an error, the surrounding loop records it and continues. A custom allowlist like `[429]` would still try later image models on a 400 or 401.
  - confidence: 0.8
- **[P2] Preserve fallbackOnErrors during model scans:** `src/config/types.agents-shared.ts:39`
  - body: After this field is persisted, `openclaw models scan` rebuilds model defaults from `primary` and `fallbacks` only. Running the scan can silently drop the new policy unless the rewrite path preserves the field or the exposed config surface is narrowed.
  - confidence: 0.76
