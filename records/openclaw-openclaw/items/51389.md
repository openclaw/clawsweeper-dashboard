  - body: `fallbackOnErrors` becomes a public `AgentModelConfig` key here, but `AgentModelSchema` is still `.strict()` and only accepts `primary` and `fallbacks`. Users who follow the PR body's JSON example will get an unknown-key config validation error before fallback code can run.
  - confidence: 0.92

  - body: The PR body documents `agents.defaults.model.fallbackOnErrors`, but this resolver reads only the selected agent's `model`. Agents that inherit `agents.defaults.model` will never pass the setting into `runWithModelFallback`, so the documented default-level config silently has no effect.
  - confidence: 0.88
- **[P2] Pass fallbackOnErrors through queued follow-ups:** `src/auto-reply/reply/followup-runner.ts:198-209`
  - body: Queued follow-up replies call `runWithModelFallback` directly with only `fallbacksOverride`, so a reply that runs through the follow-up queue will ignore the new setting and use different fallback rules from a normal turn for the same agent config.
  - confidence: 0.83
- **[P2] Stop image fallback on disallowed statuses:** `src/agents/model-fallback.ts:834-839`
  - body: `runWithImageModelFallback` accepts and forwards `fallbackOnErrors`, but after an attempt returns an error the loop always records it and advances to the next image model. With `fallbackOnErrors: [429]`, a 400/401 image failure still burns through later candidates instead of failing fast.
  - confidence: 0.8
- **[P2] Preserve fallbackOnErrors when rewriting defaults:** `src/commands/models/scan.ts:355-362`
  - body: Making this a persisted model config means config rewrite paths must keep it. `openclaw models scan` rebuilds `agents.defaults.model` from `primary` and `fallbacks` only, so after the schema accepts the key, running scan can silently delete the user's fallback policy.
  - confidence: 0.76
