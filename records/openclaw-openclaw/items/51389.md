## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Extend the strict model schema for fallbackOnErrors:** `src/config/types.agents-shared.ts:37`
  - body: This exposes `fallbackOnErrors` on `AgentModelConfig`, but `AgentModelSchema` remains strict with only `primary`, `fallbacks`, and `timeoutMs`. A user following the new examples would hit an unrecognized-key config validation error before the fallback runner can read the setting, so the schema and generated metadata need to be updated with the type.
  - confidence: 0.92
- **[P1] Resolve fallbackOnErrors from default model config:** `src/agents/agent-scope.ts:257-261`
  - body: This helper reads only `resolveAgentConfig(...).model`, which is the per-agent model config. The documented `agents.defaults.model.fallbackOnErrors` example is ignored for agents without an explicit per-agent object override, including the default agent and string-shorthand agents.
  - confidence: 0.89
- **[P2] Fail fast for disallowed image fallback errors:** `src/agents/model-fallback.ts:831-836`
  - body: Passing `fallbackOnErrors` into the image fallback attempt is not enough if the surrounding loop still advances after a rejected status. A custom allowlist like `[429]` should surface a 400/401 immediately on this path, or the option should not be exposed for image/PDF fallback until it is supported.
  - confidence: 0.8
- **[P2] Preserve fallbackOnErrors during model rewrites:** `src/config/types.agents-shared.ts:37`
  - body: Once this field is persisted, existing model-scan rewrites still rebuild `agents.defaults.model` and `imageModel` from only `primary` and `fallbacks`. Running `openclaw models scan` can silently drop the new policy, so preserve the field in those rewrites or narrow the exposed config surface.
  - confidence: 0.76
