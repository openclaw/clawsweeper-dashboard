## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Extend the strict model schema for fallbackOnErrors:** `src/config/types.agents-shared.ts:39`
  - body: This adds `fallbackOnErrors` to `AgentModelConfig`, but `AgentModelSchema` is still a strict object that only accepts the existing model keys. A user following the new config examples would hit an unrecognized-key validation error before the fallback runner can read the setting.
  - confidence: 0.92
- **[P1] Resolve fallbackOnErrors from default model config:** `src/agents/agent-scope.ts:259-263`
  - body: This helper reads only `resolveAgentConfig(...).model`, which is the per-agent model entry. The documented `agents.defaults.model.fallbackOnErrors` setting is ignored for agents without an explicit per-agent object override, including default and string-shorthand agents.
  - confidence: 0.89
- **[P2] Fail fast for disallowed image fallback errors:** `src/agents/model-fallback.ts:834-839`
  - body: Passing `fallbackOnErrors` into the image fallback attempt is not enough because the surrounding loop still advances after a rejected status. A custom allowlist like `[429]` should surface a 400 or 401 immediately on this path, or the option should not be exposed for image fallback yet.
  - confidence: 0.8
- **[P2] Preserve fallbackOnErrors during model rewrites:** `src/config/types.agents-shared.ts:39`
  - body: Once this field is persisted, existing model-scan rewrites rebuild model defaults from only `primary` and `fallbacks`. Running `openclaw models scan` can silently drop the new policy, so the rewrite path needs to preserve the field or the exposed config surface should be narrowed.
  - confidence: 0.76
