## What This Changes

This PR adds a public `fallbackOnErrors` model config option and threads it into model fallback decisions so configured HTTP statuses can control fallback behavior.

## Best Possible Solution

Land a rebased, chat-model-scoped config contract that validates and documents `fallbackOnErrors`, resolves default and per-agent policy consistently, preserves it during config rewrites, covers default/all/custom fail-fast behavior with tests, and includes redacted runtime proof.

## Reproduction Assessment

Yes for the blocking PR defects by source inspection: the strict zod schema rejects the new field, the resolver only reads per-agent model config, and the image fallback loop still advances after disallowed errors. I did not run tests because this was a read-only review.

## Solution Assessment

No. The current PR is directionally useful, but the maintainable solution needs a rebased and complete public config contract rather than exposing a key that validation, defaults, rewrites, and all fallback paths do not honor consistently.
