## Summary

Keep open. Current main still does not implement `fallbackOnErrors`, and the PR is not merge-ready because it exposes a public model config field without the matching strict schema/config metadata, misses default-level resolution, and only partially applies the policy to media fallback paths.

## What This Changes

The PR branch adds a `FallbackOnErrorCodes` type, a `fallbackOnErrors` model config resolver, and partial chat, cron, and image fallback plumbing to control which HTTP errors trigger model fallback.

## Best Possible Solution

Land an updated current-main implementation only after the public config contract is complete: strict schema, schema labels/help/docs, config rewrite preservation, default and per-agent resolution, explicit text-vs-media scope, and focused fallback tests for `default`, `all`, custom lists, cron, queued follow-ups, and any supported media path.

## Reproduction Assessment

Yes. A source-level reproduction is enough: the PR adds `fallbackOnErrors` to the type but not to the strict `AgentModelSchema`, so the documented config key cannot validate before runtime sees it.

## Solution Assessment

No. The implementation direction is plausible, but this branch is not the best mergeable fix until schema/default resolution, rewrite preservation, media scope, and regression coverage are aligned with the new public config surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Extend the strict model schema for fallbackOnErrors:** `src/config/types.agents-shared.ts:37`
  - body: This exposes `fallbackOnErrors` on `AgentModelConfig`, but `AgentModelSchema` remains `.strict()` with only `primary`, `fallbacks`, and `timeoutMs`. A user following the new examples will hit an unrecognized-key config validation error before the fallback runner can read the setting, so the schema and generated config metadata need to be updated with the type.
  - confidence: 0.92
- **[P1] Resolve fallbackOnErrors from defaults too:** `src/agents/agent-scope.ts:257-261`
  - body: This helper reads only `resolveAgentConfig(...).model`, which is the per-agent model config. The documented `agents.defaults.model.fallbackOnErrors` example is ignored for agents without an explicit per-agent object override, including the default agent and string-shorthand agents, leaving them on legacy retry rules.
  - confidence: 0.89
- **[P2] Fail fast for disallowed image fallback errors:** `src/agents/model-fallback.ts:831-836`
  - body: Passing `fallbackOnErrors` into `runFallbackAttempt` is not enough on the image path: when a custom allowlist rejects a status, the loop still records the error and advances to the next image model. Mirror the text fallback fail-fast check or avoid exposing this option on image/PDF paths until it is supported.
  - confidence: 0.8
- **[P2] Preserve fallbackOnErrors during model rewrites:** `src/config/types.agents-shared.ts:37`
  - body: Once this field is persisted, existing model-scan rewrites still rebuild `agents.defaults.model` and `imageModel` from only `primary` and `fallbacks`. Running `openclaw models scan` can silently drop the new policy, so preserve the field in those rewrites or narrow the exposed config surface.
  - confidence: 0.76
