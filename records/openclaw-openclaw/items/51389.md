## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Extend the strict model schema:** `src/config/types.agents-shared.ts:39`
  - body: `fallbackOnErrors` is added to `AgentModelConfig`, but `AgentModelSchema` is strict and still accepts only the existing model keys. A user following the new config examples would hit an unrecognized-key validation error before the fallback runner can read the setting.
  - confidence: 0.92
- **[P1] Honor defaults.model fallbackOnErrors:** `src/agents/agent-scope.ts:259-263`
  - body: This resolver only reads `resolveAgentConfig(...).model`, so `agents.defaults.model.fallbackOnErrors` from the documented example is ignored for agents without a per-agent model object. The option needs the same effective-default resolution as the fallback list, or the docs/config surface should be narrowed.
  - confidence: 0.89
- **[P2] Fail fast in image fallback for disallowed errors:** `src/agents/model-fallback.ts:834-839`
  - body: The image fallback path passes `fallbackOnErrors` into each attempt, but after an error comes back the loop records it and continues to the next candidate. With a custom allowlist like `[429]`, a 400 or 401 would still burn through later image models instead of surfacing the disallowed error.
  - confidence: 0.8
- **[P2] Preserve fallbackOnErrors during model scans:** `src/config/types.agents-shared.ts:39`
  - body: After this field becomes persisted config, `openclaw models scan` still rebuilds model defaults from only `primary` and `fallbacks`. Running the scan can silently delete the new policy unless the rewrite path preserves the field or this public config surface is narrowed.
  - confidence: 0.76
