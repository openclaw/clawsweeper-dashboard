Full review comments:

- **[P1] Add fallbackOnErrors to the config schema:** `src/config/types.agents-shared.ts:39`
  - body: This declares `fallbackOnErrors` as a user config key, but `AgentModelSchema` is still a strict object with only `primary` and `fallbacks`. A user following the PR body's `agents.defaults.model.fallbackOnErrors` example will hit config validation before the runtime resolver can read the value.
  - confidence: 0.94
- **[P1] Read the default model fallbackOnErrors setting:** `src/agents/agent-scope.ts:259-263`
  - body: `resolveRunModelFallbackOnErrors` only inspects the selected agent's own `model` object. For the documented `agents.defaults.model.fallbackOnErrors` setting, agents without a per-agent object override get `undefined` and continue using the legacy retry policy.
  - confidence: 0.91
- **[P2] Stop image fallback on disallowed statuses:** `src/agents/model-fallback.ts:838`
  - body: The image fallback path passes `fallbackOnErrors` into the attempt helper, but after an error it always records the failure and advances to the next candidate. With `fallbackOnErrors: [429]`, a 400 from the first image model would still burn through later image models instead of failing immediately.
  - confidence: 0.86
- **[P2] Preserve fallbackOnErrors when rewriting defaults:** `src/config/types.agents-shared.ts:39`
  - body: Once this field is part of persisted model config, `openclaw models scan` still rebuilds `agents.defaults.model` from only `primary` and `fallbacks`. Running that command after setting `fallbackOnErrors` would silently remove the configured retry policy.
  - confidence: 0.84
