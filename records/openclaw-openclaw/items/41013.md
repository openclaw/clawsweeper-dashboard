## Summary

Keep this PR open: it targets a real user-visible sanitizer bug still present on current main, but the proposed patch can delete malformed user text and can alter mid-message content that should remain unchanged.

## What This Changes

This PR adds a `session-recap` fast-path/helper to strip leading recap blocks from inbound user text plus sanitizer regression tests.

## Best Possible Solution

Update this PR or a replacement branch so the shared sanitizer strips only complete leading `<session-recap>` and `<session_recap>` blocks, preserves malformed and mid-message literals, keeps existing metadata/timestamp behavior intact, and adds the required changelog entry.

## Reproduction Assessment

Yes. From current-main source inspection, a complete leading `<session-recap>...</session-recap>` string has no fast-path match, so both exported strip functions return it unchanged before Control UI and TUI display paths render user text.

## Solution Assessment

No, not as-is. The shared sanitizer is the right boundary, but the patch must require a matching close tag, return the original text when no leading recap block is removed, and rebase onto current main without dropping newer sanitizer behavior.

## Review Findings

- **[P1] Require a closing recap tag before stripping:** `src/auto-reply/reply/strip-inbound-meta.ts:132`
  - body: If the first non-blank line is a recap opener but no matching close tag exists, this loop advances to EOF and returns `lines.slice(i)`, deleting the whole user message. Only strip after locating the closing line; otherwise preserve the original lines.
