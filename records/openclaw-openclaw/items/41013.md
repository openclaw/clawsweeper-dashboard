## Summary

Keep open: current main still lacks `session-recap` stripping, and this PR targets the right shared sanitizer, but the patch is not correct or mergeable as-is because it can delete malformed leading content, trims user text when no block is stripped, and is dirty against current `main`.

## What This Changes

The PR adds a `session-recap` fast path/helper in `src/auto-reply/reply/strip-inbound-meta.ts` plus six colocated tests for leading, blank-line, mid-message, and underscore-tag cases.

## Best Possible Solution

Update this PR or a replacement branch on current `main` so the shared sanitizer strips only complete leading `<session-recap>` and `<session_recap>` blocks, preserves malformed and mid-message literals, keeps existing metadata/timestamp behavior intact, adds regression coverage, and includes the required changelog entry.

## Reproduction Assessment

Yes. From current-main source inspection, a complete leading `<session-recap>...</session-recap>` string has no `SENTINEL_FAST_RE` match, so both exported strip functions return it unchanged before the Control UI and TUI display paths render user text.

## Solution Assessment

No, not as-is. The shared sanitizer is the right boundary, but the patch must require a matching close tag, return the original text when no leading recap block is removed, rebase against current main, and add the required changelog entry.

## Review Findings

- **[P1] Require a closing recap tag before stripping:** `src/auto-reply/reply/strip-inbound-meta.ts:132`
  - body: If the first non-blank line is a recap opener but no matching close tag exists, this loop advances to EOF and returns `lines.slice(i)`, deleting the whole user message. Only strip after locating the closing line; otherwise preserve the original lines.
