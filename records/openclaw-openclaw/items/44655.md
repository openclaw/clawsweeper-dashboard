## Summary

Keep open. Current main has improved Z.AI endpoint detection, a refreshed static GLM catalog, and GLM-5 forward-compat resolution, but it still does not implement provider-backed `/models` discovery or dynamic population for the Z.AI model list.

## What This Changes

Add dynamic model discovery for the bundled Z.AI provider by querying the provider model-list endpoint and falling back to stable defaults or custom model IDs when discovery fails.

## Best Possible Solution

Implement this inside the bundled `zai` provider as a provider-owned catalog/discovery path that uses the configured Z.AI endpoint and credential, preserves the existing static catalog as a fallback, keeps GLM-5 forward compatibility, and documents the behavior on the Z.AI/GLM provider pages after upstream endpoint semantics are confirmed.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The requested behavior absence is reproducible by source inspection: current Z.AI catalog/listing code is static, and targeted searches found no `/models` request in the Z.AI provider path.

## Solution Assessment

No, the current static catalog plus GLM-5 synthesis is not the best long-term solution for this request. The narrow maintainable direction is a provider-local dynamic catalog hook with timeout, auth, parsing, tests, and static fallback, not new core hardcoding.
