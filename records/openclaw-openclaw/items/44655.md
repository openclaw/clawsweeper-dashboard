## Summary

Keep open. Current main still ships Z.AI as a manifest-backed static catalog with GLM-5 forward-compat synthesis, but it does not fetch the Z.AI/Zhipu `/models` endpoint; the remaining work is a provider-owned feature that needs endpoint and Coding Plan behavior confirmed before implementation.

## What This Changes

The issue requests live Z.AI/Zhipu model discovery from `/models`, dynamic model picker population, static fallback, and a custom model ID escape hatch.

## Best Possible Solution

Implement this inside the bundled `zai` provider as an authenticated catalog hook with guarded `/models` fetch, tolerant schema parsing, static manifest fallback, preserved GLM-5 forward compatibility, and updated Z.AI docs/tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The absence is source-verifiable: current Z.AI code uses a static manifest catalog plus GLM-5 synthesis and has no Z.AI `/models` request path.

## Solution Assessment

No, the current static catalog plus GLM-5 synthesis is not the best long-term solution for the requested provider behavior. The maintainable path is a provider-owned live catalog hook with static fallback after Z.AI endpoint semantics are confirmed.
