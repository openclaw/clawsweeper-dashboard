## Summary

Keep open. Current main has improved Z.AI coverage with `glm-4.7-flash`, `glm-5.1`, and GLM-5 forward-compat synthesis, but it still does not fetch Z.AI/Zhipu `/models`; this remains a provider-owned feature needing endpoint and catalog-refresh semantics confirmed.

## What This Changes

The issue requests dynamic Zhipu/Z.AI model discovery via a provider `/models` endpoint instead of relying only on a static bundled catalog.

## Best Possible Solution

Implement this inside the bundled `zai` provider as authenticated model catalog discovery with tolerant response parsing, static manifest fallback, preserved GLM-5 forward compatibility, and updated Z.AI docs/tests after confirming Z.AI endpoint behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. The missing feature is source-verifiable: current Z.AI code has static manifest rows plus GLM-5 synthesis and no `/models` request path.

## Solution Assessment

No, the current static catalog plus GLM-5 synthesis is only a partial mitigation for fast-moving Z.AI model lists. The maintainable path is a provider-owned live catalog/cache hook with static fallback once endpoint semantics are confirmed.
