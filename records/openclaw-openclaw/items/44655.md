## Summary

Keep open. Current main still uses a manifest-backed Z.AI catalog plus GLM-5 forward-compat synthesis, but it does not fetch a live Z.AI `/models` catalog; the existing provider catalog hooks make a provider-local implementation feasible after endpoint semantics are confirmed.

## What This Changes

Add Z.AI provider-backed model discovery from `/models`, populate model choices dynamically, and retain static/custom-model fallbacks.

## Best Possible Solution

Implement this in the bundled `zai` provider as an authenticated provider catalog discovery path against the configured Z.AI endpoint, with timeout/guarded fetch, schema-tolerant parsing, static manifest fallback, preserved GLM-5 forward compatibility, and updated Z.AI docs.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The requested behavior absence is source-verifiable: current Z.AI code has a static manifest catalog and GLM-5 synthesis, with no Z.AI `/models` request path.

## Solution Assessment

No, the current static catalog plus GLM-5 synthesis is not the best long-term solution for this request. The maintainable direction is a provider-owned live catalog hook with static fallback after Z.AI endpoint behavior is confirmed.
