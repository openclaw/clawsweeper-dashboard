## Summary

Close as duplicate/superseded by the open canonical dynamic model-discovery issue at https://github.com/openclaw/openclaw/issues/10687. Current main and the latest release still show Z.ai as a static manifest-backed catalog with only a narrow GLM-5 forward resolver, while the remaining work is the same provider-backed live catalog, picker/list, cache, offline, and allowlist policy already tracked there.

## What This Changes

The issue requests dynamic Zhipu/Z.ai model discovery from the provider /models endpoint instead of relying on static model IDs.

## Best Possible Solution

Keep the provider-agnostic dynamic discovery design in https://github.com/openclaw/openclaw/issues/10687, then implement Z.ai through the provider-owned live catalog seam after caching, list-size, offline, and allowlist semantics are settled.

## Reproduction Assessment

Yes. Source inspection shows Z.ai still advertises static discovery and only has a narrow GLM-5 forward resolver, while unauthenticated endpoint probes confirm the provider /models routes exist but require credentials.

## Solution Assessment

Yes for cleanup: the best repository state is to close this provider-specific duplicate and keep the canonical dynamic model-discovery issue open. The implementation should be a provider-owned live catalog path, not a one-off Z.ai core fallback or separate product policy.
