## Label Justifications

- `P2`: The missing feature limits access to newly released Z.AI models but existing catalog models remain usable.
- `impact:auth-provider`: The request concerns provider catalog discovery, model availability, and selection routing.

## Summary

The Z.AI gap is real on current main, but it is a provider-specific instance of the broader, still-open dynamic model discovery architecture tracked in https://github.com/openclaw/openclaw/issues/10687; keeping a second design thread would split decisions about catalog hooks, caching, picker visibility, allowlists, and unknown-model runtime behavior.

## What This Changes

Add authenticated Z.AI `/models` discovery so newly available models appear in selection surfaces, with fallback defaults and custom model support.

## Best Possible Solution

Preserve the Z.AI endpoint and Coding Plan requirements on https://github.com/openclaw/openclaw/issues/10687, then implement a provider-owned Z.AI catalog adapter through the shared discovery contract so runtime, CLI, chat commands, and Control UI use one model source.

## Reproduction Assessment

Yes at source level: current main uses manifest-backed Z.AI models and only synthesizes unknown `glm-5*` IDs, with no authenticated `/models` enumeration path.

## Solution Assessment

No as a separate provider-specific design thread; the best solution is to handle Z.AI within the canonical cross-provider discovery architecture while keeping endpoint logic in the Z.AI plugin.
