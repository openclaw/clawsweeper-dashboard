## Summary

Keep open. Current main has a static Z.AI manifest catalog and GLM-5 forward compatibility, but it still does not implement authenticated Z.AI `/models` discovery or dynamic picker population; the prior dynamic-discovery PR was closed unmerged, so real provider/product work remains.

## What This Changes

Request dynamic Zhipu/Z.AI model discovery from the provider `/models` endpoint with fallback/default behavior instead of relying only on a static model list.

## Best Possible Solution

Add a Z.AI-owned live catalog provider using `registerModelCatalogProvider`/`liveCatalog`, authenticated fetch, defensive response parsing, static fallback, GLM-5 forward-compat preservation, and focused tests/docs for refresh and failure behavior.

## Reproduction Assessment

No high-confidence bug reproduction is applicable because this is a feature request. Source inspection on current main shows the missing behavior: Z.AI uses static manifest discovery, chat-completion endpoint probing, and no live catalog hook.

## Solution Assessment

Unclear until provider owners confirm the Z.AI endpoint contract and fallback UX. The maintainable direction is provider-owned live catalog discovery with static fallback, not a frontend-only dropdown patch.
