## Summary

Close as duplicate/superseded: the current and shipped Z.AI provider still uses a static manifest catalog plus narrow GLM-5 forward resolution, and the same provider-backed dynamic catalog, picker/list, cache, offline, and allowlist policy work is already tracked by the open canonical dynamic discovery issue.

## What This Changes

Request dynamic Zhipu/Z.AI model discovery from provider `/models` endpoints instead of relying on the bundled static Z.AI catalog.

## Best Possible Solution

Keep the provider-agnostic dynamic discovery design in https://github.com/openclaw/openclaw/issues/10687, then implement Z.AI through the provider-owned live catalog seam after picker, cache, offline, and allowlist semantics are decided.

## Reproduction Assessment

Yes. Source inspection shows Z.AI still advertises static discovery and only has narrow GLM-5 forward resolution, while live endpoint probes show the provider `/models` routes exist but require credentials.

## Solution Assessment

Yes for cleanup. The best repository state is to close this provider-specific duplicate and keep the canonical dynamic-discovery policy and implementation discussion in the broader open issue.
