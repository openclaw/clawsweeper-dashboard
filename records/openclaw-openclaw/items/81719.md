## Summary

Keep open: the fail-closed direction is plausible security hardening, but the patch changes the runtime omitted-host default without updating the shared exec-default reporting path, so no-config sessions can still be reported as auto/gateway/node-capable while actual exec would fail closed. The external PR also has only test-command validation and no real after-fix OpenClaw behavior proof.

## What This Changes

The branch makes omitted exec host config default to sandbox/fail closed without a sandbox runtime, preserves explicit host:auto gateway fallback, and updates exec runtime/config tests.

## Best Possible Solution

Adopt the fail-closed default only if maintainers accept the default-policy change, then centralize the omitted-host contract so runtime execution, `/exec` status, node eligibility, docs, and regression tests all agree while keeping explicit `host:auto` as the opt-in fallback.

## Reproduction Assessment

Yes for the review finding by source inspection: with omitted `tools.exec.host` and no sandbox runtime, the patched exec tool would select sandbox/fail closed, while `resolveExecDefaults` still reports auto/gateway and node eligibility. I did not run tests because this review is read-only.

## Solution Assessment

No; changing only `resolveExecTarget` is too narrow because OpenClaw already has a shared exec-defaults/reporting path for user-visible state. The safer fix is to update the central omitted-host contract and its tests/docs together.
