## Summary

Keep open. Current main still has the reported deep code-safety scan shape: plugin roots, skill roots, and files inside each root are awaited serially, while normal CLI output is emitted only after the audit finishes; related closed audit-hang reports fixed plugin-runtime and backup-directory boundaries, not this remaining scan-throughput and progress request.

## What This Changes

The issue requests bounded parallelism, incremental caching, progress/timing output, per-collector controls, and optional collector selection for `openclaw security audit --deep`.

## Best Possible Solution

Add a bounded, deterministic deep-audit scan plan with limited concurrency and visible phase progress/timing first, then treat persistent cache, per-collector limits, and `--only` as scoped follow-ups after maintainer UX decisions.

## Reproduction Assessment

Yes for a source-level reproduction: current main still awaits plugin roots, skill roots, and files within each root serially, and the CLI does not emit normal output until `runSecurityAudit` returns. I did not live-reproduce the exact 548s timing from the reporter's many-file macOS environment.

## Solution Assessment

Unclear as a single patch: bounded parallel scanning plus progress/timing is likely the narrow first solution, but persistent cache, per-collector limits, and `--deep --only` add CLI/config policy that should be scoped by a maintainer before implementation.
