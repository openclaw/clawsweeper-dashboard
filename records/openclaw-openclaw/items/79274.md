## Summary

Keep open. Current main still has the reported deep code-safety scan shape: deep audit reaches the plugin/skill collectors, collector groups and roots are awaited serially, each directory scans files serially, and normal CLI output is built only after `runSecurityAudit` returns; the linked fix PR was closed unmerged.

## What This Changes

Request improves `openclaw security audit --deep` performance and observability for large plugin/skill trees by adding bounded scan concurrency, persistent caching, progress/timing output, collector limits, and optional collector selection.

## Best Possible Solution

Land a bounded, deterministic deep-audit scan plan with limited concurrency and visible phase/timing feedback first, then split persistent cache, per-collector limits, and collector selection into scoped follow-ups.

## Reproduction Assessment

Yes for a source-level reproduction: current main still awaits plugin roots, skill roots, and files within each root serially, and CLI output is delayed until `runSecurityAudit` returns. I did not live-reproduce the reporter's exact 548s timing against a large macOS plugin/skill workspace.

## Solution Assessment

Unclear as a single change. Bounded concurrency is the narrowest first fix for the bottleneck, but persistent caching, collector-specific limits, timing/report output, and `--deep --only` need maintainer UX and contract decisions.
