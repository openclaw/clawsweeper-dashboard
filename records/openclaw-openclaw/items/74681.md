## Summary

Keep open. Current main still defers non-main isolated `agentTurn` cron timeouts until the runner reports execution start, and the existing #41783 regression/docs/changelog show that execution-only budget is intentional. The requested hard wall-time deadline is a real product bug report, but it conflicts with that current contract, so this needs maintainer timeout-semantics judgment rather than cleanup closure or a narrow automated fix.

## What This Changes

The issue asks for isolated cron `agentTurn` `timeoutSeconds` to be enforced as a hard wall-time deadline from job start instead of from the runner execution-start callback.

## Best Possible Solution

Define the cron timeout contract explicitly, then implement separate or clearly named budgets for total wall time, queue/preparation time, and execution time. The final behavior should preserve #41783’s lane-wait protection while giving operators a deterministic upper bound for isolated cron runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` delays `onExecutionStarted` beyond `timeoutSeconds`; current regression coverage already asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No. A direct hard wall-time swap is not clearly the best fix because it would contradict the current #41783 execution-only timeout contract; a split or renamed budget contract is the safer maintainable path.
