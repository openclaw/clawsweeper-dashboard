## Summary

Keep open. Current main still defers isolated cron `agentTurn` timeout arming until the execution-start callback, and the requested hard wall-time behavior conflicts with the documented and tested #41783 execution-budget contract.

## What This Changes

The issue requests hard wall-time enforcement of isolated cron `agentTurn` `timeoutSeconds` from job start, independent of `onExecutionStarted` timing or async callback state.

## Best Possible Solution

Define separate timeout semantics for total wall time versus execution budget, then document and test both so isolated cron jobs can have a reliable hard deadline without reintroducing #41783 queue-wait failures.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a fake-timer isolated `agentTurn` run that delays `onExecutionStarted` past `timeoutSeconds`; current regression coverage already asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for a direct replacement of the current timer start. The safer maintainable solution is an additive or split timeout contract, such as a total wall-time cap plus the existing execution-start budget.
