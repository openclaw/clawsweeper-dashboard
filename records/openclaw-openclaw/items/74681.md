## Summary

Keep open. Current main still arms isolated cron `agentTurn` timeouts through the execution-start callback, and existing docs/tests intentionally preserve the #41783 execution-budget behavior that this report wants to change.

## What This Changes

The issue requests hard wall-time enforcement for isolated cron `agentTurn` `timeoutSeconds` jobs, independent of delayed execution-start callbacks.

## Best Possible Solution

Define the cron timeout contract explicitly, then add separate or additive deadline handling so isolated agent-turn jobs can have a reliable total wall-time cap without reintroducing #41783 queue-wait failures.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer run where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; current regression coverage already asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for a direct hard wall-time replacement. The maintainable fix needs an explicit split or renamed contract so total-deadline behavior does not silently change the existing execution-budget semantics.
