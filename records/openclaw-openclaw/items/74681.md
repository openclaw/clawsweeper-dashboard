## Summary

Keep open. Current main still arms the isolated non-main `agentTurn` cron timeout from the runner's `onExecutionStarted` callback, and the existing #41783 regression coverage intentionally protects excluding runner-lane wait from the execution timeout. The requested hard wall-time deadline from dispatch/start remains unresolved and needs an explicit maintainer contract decision.

## What This Changes

The issue requests enforcing cron isolated `agentTurn` `timeoutSeconds` as a hard wall-time deadline from job start or dispatch rather than a callback-deferred execution timer.

## Best Possible Solution

Define the cron timeout contract explicitly and implement it across code, tests, docs, and changelog. A maintainable end state is likely separate total wall-clock and execution-budget semantics so pre-execution delays are bounded without silently undoing the #41783 fix.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` waits longer than `timeoutSeconds` before calling `onExecutionStarted`; current regression coverage already demonstrates the abort signal remains un-aborted during that wait.

## Solution Assessment

No. A direct automated change that starts every isolated `agentTurn` timeout at dispatch would match this report narrowly but conflicts with the documented and tested #41783 behavior, so the safer solution needs maintainer-approved split or renamed timeout budgets.
