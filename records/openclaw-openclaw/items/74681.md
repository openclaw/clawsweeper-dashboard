## Summary

Keep open. Current main fixed the related cold-start/pre-model stall class through https://github.com/openclaw/openclaw/pull/80130, but it still intentionally treats isolated cron `agentTurn` `timeoutSeconds` as an execution-start budget; changing that into a dispatch-time hard wall deadline conflicts with the documented and tested queue-wait contract from https://github.com/openclaw/openclaw/issues/41783.

## What This Changes

The issue requests hard wall-time enforcement for isolated cron `agentTurn` `timeoutSeconds` from job dispatch instead of arming the timeout after execution starts.

## Best Possible Solution

Define a separate total wall-time budget, or a documented split between queue/setup wall time and execution budget, without reusing `timeoutSeconds` in a way that regresses the queue-wait fix.

## Reproduction Assessment

Yes, at source level. Current regression coverage delays `onExecutionStarted` past the configured timeout and asserts the abort signal remains un-aborted, while current source still starts the configured timeout from that callback rather than `startedAt`.

## Solution Assessment

No for directly moving the existing timer to dispatch time. The maintainable path is an additive or split timeout contract so hard wall-time enforcement can coexist with the existing execution-budget semantics.
