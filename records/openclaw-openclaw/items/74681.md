## Summary

Keep open. Current main still implements isolated cron agentTurn timeouts as an execution-start budget rather than a hard wall-time deadline from cron dispatch, and the existing docs/tests intentionally preserve that behavior for the related #41783 lane-wait contract.

## What This Changes

The issue requests enforcing isolated cron agentTurn `timeoutSeconds` as a hard wall-time timer from job start, independent of delayed execution-start callbacks or async preparation work.

## Best Possible Solution

Define the cron timeout contract explicitly, then implement separate or additive budgets so isolated agentTurn jobs get a reliable total deadline without silently reintroducing #41783 queue-wait failures. The final change should update the timer wrapper, isolated-agent preparation/executor wiring, embedded/CLI runner notifications, docs, changelog, and regression coverage for both timeout semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated agentTurn fake-timer run where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; current regression coverage already asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for a direct hard wall-time swap. It would match this report but conflict with the protected #41783 execution-budget behavior, so the maintainable solution is an explicit split or renamed contract rather than silently changing one meaning into the other.
