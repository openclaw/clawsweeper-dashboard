## Summary

Keep open. Current main still defers isolated non-main `agentTurn` cron timeout arming until the runner calls `onExecutionStarted`, and the existing regression coverage intentionally protects waiting longer than `timeoutSeconds` before that callback. The report is valid, but a direct hard wall-time change would conflict with the shipped #41783 execution-budget contract, so this needs maintainer contract judgment rather than cleanup closure.

## What This Changes

The issue requests enforcing isolated cron `agentTurn` `timeoutSeconds` as a hard wall-time deadline from job start or dispatch instead of from the runner execution-start callback.

## Best Possible Solution

Define the cron timeout contract explicitly, then implement either separate named budgets or an additive total deadline plus execution timeout. The change should cover `executeJobCoreWithTimeout`, isolated-agent preparation/executor wiring, embedded and CLI runner start callbacks, docs, changelog, and regression tests for both the requested hard-deadline case and the existing #41783 lane-wait case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; current tests already show the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated swap to start every isolated `agentTurn` timeout at dispatch. That would match this issue but contradict the currently tested #41783 behavior, so the safer path is an explicit split or renamed timeout contract.
