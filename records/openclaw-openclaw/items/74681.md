## Summary

Keep open. Current main still enforces isolated cron agentTurn `timeoutSeconds` as an execution-start budget rather than a hard wall-time deadline from cron dispatch, and changing it directly would conflict with the documented/tested #41783 lane-wait contract.

## What This Changes

The issue asks for isolated cron agentTurn `timeoutSeconds` to be enforced as a hard wall-time deadline from job dispatch rather than being deferred until execution-start callbacks fire.

## Best Possible Solution

Define separate timeout semantics, then implement an additive total deadline or renamed budget so isolated agentTurn jobs can have a reliable wall-time cap without silently reintroducing #41783 queue-wait failures.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer run where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; current coverage already asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for a direct hard wall-time replacement. The maintainable fix is an explicit split or contract change with regression coverage for both total-deadline and execution-budget behavior.
