## Summary

Keep open. Current main still arms isolated non-main `agentTurn` cron timeouts from the runner's `onExecutionStarted` callback, not from cron dispatch or the recorded run start, and the existing regression test intentionally protects waiting longer than `timeoutSeconds` before that callback. The report is valid, but the requested hard wall-time behavior conflicts with the shipped #41783 execution-budget semantics, so this needs maintainer contract judgment rather than a conservative cleanup close.

## What This Changes

The issue requests that cron isolated `agentTurn` `timeoutSeconds` be enforced as a hard wall-time deadline starting at job dispatch/start instead of when the isolated runner calls `onExecutionStarted`.

## Best Possible Solution

Define the cron timeout contract explicitly and then implement it with separate, named budgets or an additive total deadline plus execution timeout. The update should cover `executeJobCoreWithTimeout`, isolated-agent preparation/executor wiring, embedded/CLI runner start callbacks, docs, changelog, and regression tests for both this hard-deadline case and the existing #41783 lane-wait case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; current tests already show the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated swap to start every isolated `agentTurn` timeout at dispatch. That would match this issue but contradict the currently tested #41783 behavior, so the safer solution is an explicit split or renamed timeout contract.
