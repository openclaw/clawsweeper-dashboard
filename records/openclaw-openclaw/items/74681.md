## Summary

Keep open. Current main fixes related pre-model stall and false-watchdog cases, but this issue still asks to make isolated cron `agentTurn` `timeoutSeconds` a dispatch-time hard wall deadline, which conflicts with the documented and tested execution-start timeout contract.

## What This Changes

The issue requests hard wall-time enforcement of isolated cron `agentTurn` `timeoutSeconds` from job start, independent of `onExecutionStarted` callback timing.

## Best Possible Solution

Define an additive total wall-time budget or a documented split between setup, queue, execution, and hard-ceiling deadlines so hard wall enforcement can coexist with the existing execution-budget semantics.

## Reproduction Assessment

Yes, at source level. Current source and regression coverage show `timeoutSeconds` intentionally starts from `onExecutionStarted`, and fake timers past the configured timeout do not abort while the runner lane is still waiting.

## Solution Assessment

No for moving the existing `timeoutSeconds` timer to dispatch time. The safer maintainable path is an additive or split timeout contract so operators can request a hard ceiling without undoing the queue-wait fix.
