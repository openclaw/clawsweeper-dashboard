## Summary

Keep open. Current main still defers isolated non-main `agentTurn` cron timeouts until the runner reports `onExecutionStarted`, while the issue asks for a hard wall-time deadline from job start; the existing regression coverage and changelog show this is tied to the intentional #41783 execution-budget behavior, so it needs maintainer contract judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to enforce `timeoutSeconds` for isolated cron `agentTurn` jobs as a hard wall-time deadline from job start, independent of delayed runner callbacks or async operations.

## Best Possible Solution

Define the cron timeout contract explicitly, preferably with separate named budgets or an additive total deadline plus execution timeout, then update `executeJobCoreWithTimeout`, isolated-agent executor wiring, embedded/CLI runner callbacks, docs, changelog, and regression tests for both this hard-deadline case and the existing #41783 lane-wait case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to delay `runIsolatedAgentJob`/`onExecutionStarted` past `timeoutSeconds` under fake timers and observe that the current abort signal remains un-aborted until the callback fires; the existing #41783 test already demonstrates that path.

## Solution Assessment

No, not as a direct automated swap to start every isolated `agentTurn` timeout at dispatch. That would satisfy this report but conflict with the current execution-timeout semantics, so the safer solution is an explicit split or renamed timeout contract.
