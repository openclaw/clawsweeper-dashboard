## Summary

Keep open. Current main still arms isolated non-main `agentTurn` cron timeouts only after `onExecutionStarted`, and the docs plus regression coverage preserve the #41783 execution-start contract, so the requested hard wall-time deadline remains an unresolved timeout-contract decision rather than a config or provider API timeout issue.

## What This Changes

Request to enforce cron isolated `agentTurn` `timeoutSeconds` as a hard wall-clock deadline from job start/dispatch instead of waiting for the execution-start callback.

## Best Possible Solution

Define the cron timeout contract explicitly, then update implementation, regression tests, docs, and changelog together. The most maintainable end state is likely separate total-deadline and execution-budget semantics, or a clearly named queue/preparation budget, so pre-execution delays are bounded without making useful execution inherit lane wait again.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a fake isolated `agentTurn` runner that receives the abort signal but delays `onExecutionStarted` beyond `timeoutSeconds`; the existing #41783 test already demonstrates that current main leaves the signal un-aborted during that wait.

## Solution Assessment

No, not as a direct change that starts the existing isolated execution timeout at dispatch. That would satisfy this report narrowly but contradict current #41783 behavior, so the safer solution needs an explicit maintainer decision on split or renamed budgets.
