## Summary

Keep open. Current main still defers isolated cron `agentTurn` timeout arming until the execution-start callback, and the requested hard wall-time deadline conflicts with the documented and tested #41783 execution-budget contract.

## What This Changes

Request that isolated cron `agentTurn` jobs enforce `timeoutSeconds` as a hard wall-time deadline from job start instead of from the execution-start callback.

## Best Possible Solution

Define separate total wall-time and execution-budget semantics for isolated cron runs, then document and test both so jobs get a reliable hard deadline without regressing #41783 queue-wait protection.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a fake-timer isolated `agentTurn` run that delays `onExecutionStarted` past `timeoutSeconds`; current regression coverage explicitly asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for directly replacing the current timer start. The safer solution is an additive or split timeout contract that preserves the existing execution-start budget while adding a hard total wall-time cap where needed.
