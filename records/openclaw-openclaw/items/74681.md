## Summary

Keep open. Current main still starts isolated non-main `agentTurn` cron timeouts from the runner's `onExecutionStarted` callback rather than from cron dispatch or the recorded run start, and current regression coverage intentionally preserves the opposite #41783 execution-budget behavior. The report is valid, but changing `timeoutSeconds` into a hard wall-clock deadline needs maintainer contract judgment before implementation.

## What This Changes

The issue requests that isolated cron `agentTurn` jobs enforce `timeoutSeconds` as a hard wall-time deadline from job start, independent of runner callbacks or async preparation state.

## Best Possible Solution

Define the cron timeout contract explicitly and implement it with separate named budgets or an additive total deadline plus execution timeout. The final change should update `executeJobCoreWithTimeout`, isolated-agent preparation/executor wiring, embedded/CLI runner start callbacks, docs, changelog, and regression coverage for both this hard-deadline case and the existing #41783 lane-wait case.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; the existing regression test currently expects the abort signal to remain un-aborted during that wait.

## Solution Assessment

No, not as a direct automated change to start every isolated `agentTurn` timeout at cron dispatch. That would satisfy this report but contradict the currently tested and shipped #41783 execution-budget behavior, so the safer solution is an explicit split or renamed timeout contract.
