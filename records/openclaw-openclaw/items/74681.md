## Summary

Current main still matches the report's timeout-contract problem: isolated cron agentTurn timeouts are armed from the execution-start callback rather than from cron dispatch, while current tests and docs intentionally protect the #41783 execution-budget behavior. This should remain open for maintainer judgment because the requested hard wall-time deadline is not already implemented and conflicts with an existing documented contract.

## What This Changes

The issue requests that isolated cron `agentTurn` jobs enforce `timeoutSeconds` as a hard wall-time deadline from job start instead of starting the timer when execution-start notification fires.

## Best Possible Solution

Define the timeout contract explicitly and add a separate total-deadline or preparation budget for isolated cron agentTurn jobs while preserving the existing execution-budget behavior where maintainers still want it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer run where `runIsolatedAgentJob` delays `onExecutionStarted` past `timeoutSeconds`; current regression coverage asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for a direct replacement of the current timer semantics. The maintainable fix is to split or rename the budget contract so this report can get a reliable total deadline without silently reintroducing #41783 queue-wait failures.
