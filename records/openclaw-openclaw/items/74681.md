## Summary

Keep open. Current main still intentionally defers isolated cron agentTurn timeout arming until execution starts, so the requested hard wall-time deadline remains unresolved and overlaps a maintainer-level timeout-contract decision from #41783.

## What This Changes

The issue requests enforcing isolated cron agentTurn timeoutSeconds as a hard wall-time deadline from job dispatch/start rather than from the execution-start callback.

## Best Possible Solution

Add explicit split semantics, such as a total wall-time cap plus the existing execution-start budget, and document/test both so cold-start and queue-wait behavior are handled deliberately.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a fake-timer isolated agentTurn run that delays `onExecutionStarted` past `timeoutSeconds`; current regression coverage already asserts the abort signal remains un-aborted during that wait.

## Solution Assessment

No for a direct replacement of the current timer start. The safer maintainable path is an additive or split timeout contract that preserves #41783 execution-budget behavior while adding a hard deadline where needed.
