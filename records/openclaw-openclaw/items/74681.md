## Summary

Keep open. Current main still arms isolated cron agentTurn timeouts from the execution-start callback, while docs and regression coverage intentionally protect that behavior for the earlier lane-wait timeout fix; the remaining work needs maintainer timeout-contract judgment rather than cleanup closure.

## What This Changes

Request to enforce isolated cron agentTurn timeoutSeconds as a hard wall-time deadline from job dispatch instead of execution-start notification.

## Best Possible Solution

Define the cron timeout contract explicitly, then implement separate or additive budgets so isolated agentTurn jobs have a reliable total deadline without silently removing the existing execution-budget semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a fake-timer isolated agentTurn run that delays onExecutionStarted past timeoutSeconds and observes that the abort signal remains un-aborted until execution is reported started.

## Solution Assessment

No for a direct hard wall-time replacement. The maintainable fix is an explicit split or renamed/additive timeout contract with regression coverage for both total deadline and lane-wait behavior.
