## Summary

Keep this issue open: current main and the latest release still expose only `open`, `disabled`, and `allowlist`, while the closed `members` PRs and the chat-only-trigger community plugin are useful prior art/workaround context rather than a shipped bundled Telegram access policy.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created groups by trusted user membership instead of manually allowlisting each group chat ID.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, security-audit semantics, docs, and focused regression tests while preserving existing policies and failing closed on uncertain membership checks.

## Reproduction Assessment

Yes for the missing current behavior: source inspection shows current main rejects `groupPolicy: "members"` at the config type/schema layer and Telegram runtime paths have no membership verification branch. No live Telegram repro was run because this is a new feature request rather than a broken shipped policy.

## Solution Assessment

Unclear until maintainers choose the access-control contract. If accepted, the narrow maintainable solution belongs in the bundled Telegram group-policy path rather than relying on the adjacent chat-only-trigger plugin workaround, because inbound and native-command authorization own this safety boundary.
