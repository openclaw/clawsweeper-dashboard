## Summary

Keep open: current main and the latest release still only support `open`, `disabled`, and `allowlist`; the prior `members` implementation PRs are closed unmerged, and the chat-only-trigger plugin is useful workaround context rather than the requested bundled Telegram access policy.

## What This Changes

Request a Telegram `groupPolicy: "members"` mode that authorizes newly created groups based on trusted participant membership instead of manually allowlisting each group chat ID.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, security-audit semantics, docs, and focused tests while preserving existing policies and failing closed on uncertain membership checks.

## Reproduction Assessment

Yes for the missing capability: source inspection shows current main rejects `groupPolicy: "members"` at the config type/schema layer and the Telegram runtime paths have no membership-verification branch. This is a feature request rather than a broken shipped policy, so no live Telegram repro was run.

## Solution Assessment

Unclear until maintainers choose the access-control contract. If accepted, the maintainable solution belongs in the bundled Telegram group-policy path rather than relying on the adjacent chat-only-trigger plugin workaround, because inbound and native-command authorization own this safety boundary.
