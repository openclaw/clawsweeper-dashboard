## Summary

Keep open: current main and the latest release still lack the requested Telegram policy, and the remaining work changes group access control semantics that need maintainer/security approval before implementation.

## What This Changes

Request a Telegram `groupPolicy: "members"` mode that authorizes newly created groups when all current participants are allowlisted users rather than requiring each group chat ID to be configured.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with config/schema/generated metadata, inbound and native-command authorization, fail-closed Bot API membership verification, security-audit behavior, docs, and focused tests after maintainer approval of the access-control contract.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection of current main and `v2026.5.12` gives high confidence that `members` is absent from config, schema, docs, Telegram inbound authorization, native-command authorization, and security audit.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked chat-only-trigger plugin is useful prior art but not equivalent.
