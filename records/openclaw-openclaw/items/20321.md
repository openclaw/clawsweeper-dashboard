## Summary

Keep open: current main and latest release still lack the requested bundled Telegram `groupPolicy: "members"` behavior. The closed implementation PRs and chat-only-trigger plugin are useful prior art/workarounds, but this remains a security-sensitive Telegram authorization product decision rather than completed core work.

## What This Changes

The issue requests a Telegram group access policy that authorizes newly created groups based on trusted member IDs instead of per-group chat ID allowlisting.

## Best Possible Solution

Have maintainers define the fail-closed membership contract, then land a bundled Telegram policy with aligned config/schema metadata, inbound and native-command authorization, security-audit behavior, docs, and focused tests.

## Reproduction Assessment

Yes, source-level: current main's config type, schema, Telegram runtime helper, inbound handler, and native-command path only support `open`, `disabled`, and `allowlist`. This is a feature/policy gap rather than a failing existing behavior reproduction.

## Solution Assessment

Unclear until maintainers define the security contract. If accepted, the best path is a bundled Telegram channel policy rather than a ClawHub-only workaround because it changes channel authorization and audit semantics.
