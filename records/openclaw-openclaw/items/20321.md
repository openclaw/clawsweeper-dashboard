## Summary

Keep open because current main still lacks the requested bundled Telegram members-based group authorization, and a recent maintainer review explicitly left this for maintainer/security follow-up rather than stale cleanup.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created groups based on trusted group members instead of per-group chat IDs.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed member verification, docs, security-audit behavior, and focused tests while preserving existing policies.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static current-main inspection shows the requested `members` value is absent from config, schema, generated metadata, Telegram runtime authorization, native-command authorization, and docs.

## Solution Assessment

Unclear until maintainers define the authorization contract. The durable implementation belongs in the bundled Telegram plugin plus shared config/docs/security surfaces; the external chat-only-trigger plugin is useful prior art but not equivalent to all-current-members authorization.
