## Summary

Keep open because current main still lacks the requested bundled Telegram members-based group authorization; the closed implementation PRs and chat-only-trigger plugin are useful prior art/workarounds, not shipped equivalent behavior.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes new groups based on trusted current members instead of allowlisting each group chat ID.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata support, Telegram inbound and native-command authorization, fail-closed all-current-members verification, security-audit behavior, documentation, and focused regression tests while preserving existing `open`, `disabled`, and `allowlist` semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The missing behavior is high-confidence from static current-main verification across exported config, strict schema, generated metadata, Telegram runtime authorization, native commands, docs, and tests.

## Solution Assessment

Unclear until maintainers define the authorization contract. The durable solution belongs in the bundled Telegram plugin plus shared config/docs/security surfaces; the external chat-only-trigger workaround is useful but not equivalent to the requested all-current-members policy.
