## Summary

Keep open: current main still supports only `open`, `disabled`, and `allowlist` group policies, while the linked `members` implementation attempts are closed unmerged and the remaining work changes Telegram group authorization semantics.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes new groups by trusted current members instead of manually allowlisting each group chat ID.

## Best Possible Solution

Define and land a bundled Telegram `members` policy with config/schema/generated metadata, inbound and native-command authorization, fail-closed membership verification, security-audit behavior, docs, and focused tests after maintainer approval of the access-control contract.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection of current main gives high confidence that `members` is absent from config, schema, docs, Telegram inbound authorization, native-command authorization, and security audit.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked chat-only-trigger plugin is useful prior art but not equivalent.
