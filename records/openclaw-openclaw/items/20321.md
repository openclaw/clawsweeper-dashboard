## Summary

Keep open: current main and the latest release still expose only `open`, `disabled`, and `allowlist`, and the linked `members` implementation PRs are closed unmerged. The community chat-only-trigger plugin is useful adjacent prior art, but it does not provide the requested bundled Telegram all-current-members authorization policy.

## What This Changes

Request a Telegram `groupPolicy: "members"` mode that allows new groups when all current members are trusted users instead of manually allowlisting each group chat ID.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with config/schema/generated metadata, inbound and native-command authorization, fail-closed Bot API membership verification, security-audit behavior, docs, and focused tests after maintainer approval of the access-control contract.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection of current main and `v2026.5.12` gives high confidence that `members` is absent from config, schema, docs, Telegram inbound authorization, native-command authorization, security audit, and tests.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked chat-only-trigger plugin is useful prior art but not equivalent.
