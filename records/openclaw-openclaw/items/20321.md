## Summary

Keep open: current main and v2026.5.19 still expose only `open`, `disabled`, and `allowlist`, while the `members` implementation attempts are closed unmerged. The request is valid but changes Telegram group authorization and security-audit semantics, so maintainers should approve the fail-closed contract before implementation.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that allows newly created groups when every current group member is trusted, instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Define and land a bundled Telegram `members` policy with config/schema/generated metadata, inbound and native-command authorization, fail-closed membership verification, security-audit behavior, docs, and focused tests after maintainer approval of the access-control contract.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection of current main and v2026.5.19 gives high confidence that `members` is absent from config, schema, docs, Telegram inbound authorization, native-command authorization, and security audit.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked chat-only-trigger plugin is useful prior art but not equivalent.
