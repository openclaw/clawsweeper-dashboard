## Summary

Keep open: current main still lacks a bundled Telegram `groupPolicy: "members"` path, and the linked implementation attempts plus the chat-only-trigger plugin are prior art/workarounds rather than shipped all-current-members authorization.

## What This Changes

Request to add a Telegram group policy that authorizes new groups when every current participant is in a trusted user allowlist.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed Bot API member verification, security-audit behavior, docs, and focused tests while preserving existing `open`, `disabled`, and `allowlist` semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection on current main gives high confidence that `members` is absent from config, schema, docs, Telegram runtime authorization, and native-command authorization.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus shared config/docs/security updates; the chat-only-trigger plugin is useful prior art but not equivalent.
