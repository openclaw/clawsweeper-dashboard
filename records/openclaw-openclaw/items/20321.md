## Summary

Keep open: current main still does not support the requested Telegram `groupPolicy: "members"`, and the closed implementation PRs plus the chat-only-trigger community plugin are useful prior art/workarounds rather than a shipped bundled all-members authorization path.

## What This Changes

Add a Telegram `groupPolicy: "members"` mode that allows newly created groups when every current group member is trusted by the configured user allowlist.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed member verification, docs, security-audit behavior, and focused tests while preserving existing policies.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static current-main inspection gives high confidence that `members` is absent from config, schema, generated metadata, Telegram runtime authorization, native commands, docs, and security audit handling.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the durable implementation belongs in the bundled Telegram plugin plus shared config/docs/security surfaces; the chat-only-trigger plugin is helpful prior art but not equivalent.
