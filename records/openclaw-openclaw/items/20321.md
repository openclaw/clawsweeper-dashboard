## Summary

Keep open: current main still supports only `open`, `disabled`, and `allowlist`, and the requested Telegram `members` policy remains a security-sensitive feature needing maintainer contract decisions; the chat-only-trigger plugin is useful related prior art but not equivalent to bundled all-members verification.

## What This Changes

Add a Telegram `groupPolicy: "members"` mode that authorizes newly created groups when every current group member is trusted by user allowlist instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed member verification, docs, security-audit behavior, and focused tests while preserving existing policies.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static current-main inspection shows the requested `members` value is absent from config, schema, generated metadata, Telegram runtime authorization, native-command authorization, tests, and docs.

## Solution Assessment

Unclear until maintainers define the authorization contract. The durable implementation belongs in the bundled Telegram plugin plus shared config/docs/security surfaces; the external chat-only-trigger plugin is helpful prior art but not equivalent to all-current-members authorization.
