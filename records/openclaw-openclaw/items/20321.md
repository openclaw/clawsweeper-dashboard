## Summary

Keep open because current main still lacks the requested bundled Telegram members-based group authorization; the closed implementation PRs and the chat-only-trigger plugin are useful prior art/workarounds, not shipped equivalent behavior.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes new groups based on a trusted member allowlist instead of per-group chat IDs.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed member verification, docs, security-audit behavior, and focused tests while preserving existing policies.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static current-main checks show the requested `members` value is absent from config, schema, generated metadata, runtime group authorization, native commands, docs, and tests.

## Solution Assessment

Unclear until maintainers define the authorization contract. The durable solution belongs in the bundled Telegram plugin plus shared config/docs/security surfaces; the external chat-only-trigger workaround is useful but not equivalent to the requested all-current-members policy.
