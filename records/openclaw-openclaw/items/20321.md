## Summary

Keep open: current main still lacks a bundled Telegram `groupPolicy: "members"` implementation, while the closed implementation attempts and the chat-only-trigger plugin are useful prior art rather than a shipped all-current-members authorization policy.

## What This Changes

Request to add a Telegram `groupPolicy: "members"` mode that authorizes new groups by trusted user membership instead of pre-registering every group chat ID.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed member verification, security-audit behavior, docs, and focused tests while preserving existing policy semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection on current `origin/main` gives high confidence that `members` is absent from config, schema, generated metadata, docs, Telegram inbound authorization, and native-command authorization.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked chat-only-trigger plugin is useful prior art but not equivalent.
