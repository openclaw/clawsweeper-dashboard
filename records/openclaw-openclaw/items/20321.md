## Summary

Keep open: current main still does not implement a bundled Telegram `groupPolicy: "members"`, and the related implementation PRs are closed unmerged. The linked chat-only-trigger community plugin is useful prior art/workaround, but it is not equivalent to all-current-members Telegram authorization in OpenClaw.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes new groups based on trusted group members instead of allowlisting each group chat ID.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy with fail-closed Bot API membership verification, config/schema/metadata, inbound and native-command authorization, security-audit behavior, docs, and focused tests while preserving existing policies.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection on current main gives high confidence that `members` is absent from config, schema, docs, Telegram runtime authorization, and native-command authorization.

## Solution Assessment

Unclear until maintainers define the fail-closed authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus shared config/docs/security updates; the chat-only-trigger plugin is useful prior art but not equivalent.
