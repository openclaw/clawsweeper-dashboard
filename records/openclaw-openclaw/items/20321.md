## Summary

Keep open: current main and v2026.6.10 still expose only `open`, `disabled`, and `allowlist`; the closed implementation PRs and the chat-only-trigger plugin are useful prior art/workarounds, but they do not provide the requested bundled Telegram `members` access policy.

## What This Changes

Request a Telegram `groupPolicy: "members"` mode that authorizes new groups when every current participant is trusted by user allowlist rather than requiring each group chat ID to be configured.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy across config schema/metadata, inbound and native-command authorization, security audit semantics, docs, and focused tests while preserving existing policies.

## Reproduction Assessment

Yes for the missing capability: current-source inspection shows current main rejects `groupPolicy: "members"` at the type/schema layer and the Telegram runtime path has no membership-verification branch. This is a feature request rather than a broken shipped policy, so no live Telegram repro was run.

## Solution Assessment

Unclear until maintainers choose the access-control contract. If accepted, the maintainable solution belongs in the bundled Telegram group-policy path rather than relying on the adjacent chat-only-trigger plugin workaround, because inbound and native-command authorization own this safety boundary.
