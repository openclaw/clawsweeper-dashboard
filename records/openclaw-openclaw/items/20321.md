## Summary

Keep open. Current main still lacks the requested bundled Telegram `groupPolicy: "members"` in the public config contract, schema, generated metadata, Telegram inbound/native-command authorization, docs, and security-audit semantics. The closed implementation PRs and constansino's chat-only-trigger plugin are useful prior art/workarounds, but they do not ship the requested OpenClaw Telegram access-control behavior.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that lets the bot respond in newly created groups only when every current member is trusted via the user allowlist, avoiding per-group chat ID allowlisting.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata support, Telegram inbound and native-command authorization, fail-closed all-current-members verification, security-audit behavior, documentation, and focused regression tests while preserving existing `open`, `disabled`, and `allowlist` semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The missing behavior is high-confidence from static current-main verification across exported config, strict schema, generated metadata, Telegram inbound/native-command authorization, docs, and security-audit-adjacent code.

## Solution Assessment

Unclear until maintainers define the authorization contract. The durable solution belongs in the bundled Telegram plugin plus shared config/docs/security surfaces; the external chat-only-trigger workaround is useful but not equivalent to the requested all-current-members policy.
