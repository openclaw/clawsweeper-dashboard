## Summary

Keep open: current main still lacks the requested Telegram `members` group policy, and the remaining work changes Telegram group authorization rather than fixing a broken existing contract. The closed implementation attempts and the chat-only-trigger community plugin are useful prior art, but they are not a shipped bundled implementation.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created groups from trusted user membership instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Define and land a bundled Telegram `members` policy with config/schema/generated metadata, inbound and native-command authorization, fail-closed membership verification, security-audit behavior, docs, and focused tests after maintainer approval of the access-control contract.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection of current main gives high confidence that `members` is absent from config, schema, docs, Telegram inbound authorization, native-command authorization, and security audit.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked community plugin is useful prior art but not equivalent.
