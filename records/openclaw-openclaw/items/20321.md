## Summary

Keep this issue open: current main and the latest release still expose only `open`, `disabled`, and `allowlist`, while the closed implementation PRs and the chat-only-trigger plugin are useful prior art rather than a shipped bundled Telegram `members` policy.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created groups when all current participants are trusted users instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy, including config type/schema/generated metadata, inbound and native-command authorization, security-audit semantics, docs, and focused regression tests while preserving existing `open`, `disabled`, and `allowlist` behavior.

## Reproduction Assessment

Yes for the missing current behavior: source inspection shows current main rejects `groupPolicy: "members"` at the config type/schema layer and Telegram runtime paths have no membership verification branch. No live Telegram repro was run because this is a new feature request rather than a broken shipped policy.

## Solution Assessment

Unclear until maintainers choose the access-control contract; if accepted, the right solution is a bundled Telegram policy rather than relying on the adjacent chat-only-trigger plugin workaround because current Telegram inbound/native-command authorization owns this safety boundary.
