## Summary

Keep open: current main and the latest release still support only `open`, `disabled`, and `allowlist`; the same-purpose `members` implementation PRs are closed unmerged, and the chat-only-trigger plugin is useful workaround context rather than the requested bundled Telegram access policy.

## What This Changes

The issue requests a Telegram `groupPolicy: "members"` mode that authorizes newly created groups based on trusted participants rather than pre-registered group chat IDs.

## Best Possible Solution

Define and land an explicit bundled Telegram `members` policy across config schema/metadata, inbound and native-command authorization, security audit semantics, docs, and focused tests while preserving existing policies.

## Reproduction Assessment

Yes for the missing capability: current-source inspection shows current main rejects `groupPolicy: "members"` at the config type/schema layer and the Telegram runtime path has no membership-verification branch. This is a feature request rather than a broken shipped policy, so no live Telegram repro was run.

## Solution Assessment

Unclear until maintainers choose the access-control contract. If accepted, the maintainable solution belongs in the bundled Telegram group-policy path rather than relying on the adjacent chat-only-trigger plugin workaround, because inbound and native-command authorization own this safety boundary.
