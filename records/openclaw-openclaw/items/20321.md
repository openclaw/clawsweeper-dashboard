## Summary

Keep open: current main still does not implement a bundled Telegram `groupPolicy: "members"` path, and the prior implementation PRs are closed unmerged. The linked chat-only-trigger plugin is useful prior art and a workaround, but it is not equivalent to shipped all-current-members Telegram authorization in OpenClaw.

## What This Changes

Request a Telegram `groupPolicy: "members"` mode that allows new groups when every current participant is in a trusted user allowlist instead of allowlisting each group chat ID.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed Bot API membership verification, security-audit behavior, docs, and focused tests while preserving existing `open`, `disabled`, and `allowlist` semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection on current main gives high confidence that `members` is absent from config, schema, docs, Telegram runtime authorization, and native-command authorization.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus shared config/docs/security updates; the chat-only-trigger plugin is useful prior art but not equivalent.
