## Summary

Keep open: current main and the latest release still expose only `open`, `disabled`, and `allowlist`, while this requests a new Telegram authorization mode whose fail-closed semantics need maintainer/security approval.

## What This Changes

Add a Telegram `groupPolicy: "members"` mode that authorizes new groups based on trusted current participants instead of per-group chat IDs.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config/schema/generated metadata, inbound and native-command authorization, fail-closed Bot API membership verification, security-audit behavior, docs, and focused tests after maintainer approval of the access-control contract.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection of current main and `v2026.5.12` gives high confidence that `members` is absent from config, schema, docs, Telegram inbound authorization, and native-command authorization.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus config/docs/security updates; the linked chat-only-trigger plugin is useful prior art but not equivalent.
