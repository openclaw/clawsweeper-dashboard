## Summary

Keep open: current `main` still lacks a bundled Telegram `groupPolicy: "members"` path, and the closed implementation PRs plus the chat-only-trigger plugin are useful prior art rather than a shipped all-current-members authorization feature.

## What This Changes

The issue requests a Telegram group policy that lets new groups work when every current group member is trusted, instead of requiring each group chat ID to be allowlisted.

## Best Possible Solution

Land an explicit bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, fail-closed member verification, docs, security-audit behavior, and focused tests while preserving existing policies.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Static inspection on current `origin/main` gives high confidence that `members` is absent from config, schema, generated metadata, Telegram runtime authorization, native commands, docs, and security audit handling.

## Solution Assessment

Unclear until maintainers define the authorization contract. If accepted, the maintainable path is a bundled Telegram policy plus shared config/docs/security updates; the chat-only-trigger plugin is helpful prior art but not equivalent.
