## Summary

Keep this issue open: current main and the latest release still expose only `open`, `disabled`, and `allowlist`, and the Telegram inbound/native-command paths still have no all-current-members authorization branch. The related `members` PRs are closed unmerged prior art, while the community plugin is an adjacent workaround rather than a bundled Telegram access policy.

## What This Changes

Request a new Telegram `groupPolicy: "members"` mode that allows newly created groups when every participant is trusted by user allowlist membership rather than by group chat ID.

## Best Possible Solution

Define and land a first-class bundled Telegram `members` policy with config type/schema/generated metadata, inbound and native-command authorization, security-audit semantics, docs, and focused tests while preserving existing policies and failing closed on uncertain membership checks.

## Reproduction Assessment

Yes for the missing current behavior: source inspection shows current main rejects `groupPolicy: "members"` at the config type/schema layer and the Telegram runtime paths have no membership verification branch. No live Telegram reproduction was run because this is a new feature request rather than a broken shipped policy.

## Solution Assessment

Unclear until maintainers choose the access-control contract. If accepted, the narrow maintainable solution belongs in the bundled Telegram group-policy path rather than relying on the adjacent chat-only-trigger plugin workaround, because inbound and native-command authorization own this safety boundary.
