## Summary

Keep open. Current main and the latest release still call `saveMediaBuffer` for Discord attachments and stickers without passing an original filename, and this issue is already paired with open PR #59783 using closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue asks Discord inbound attachment filenames to be preserved in saved media paths and agent metadata instead of being reduced to UUID-based filenames.

## Best Possible Solution

Land a Discord-owned fix that passes `attachment.filename` and sticker candidate filenames into `saveMediaBuffer`, updates Discord media tests, and confirms the desired metadata parity with WhatsApp.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: an `APIAttachment.filename` reaches Discord media resolution, but the saved inbound media path is built without that filename because the fifth `saveMediaBuffer` argument is omitted.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing Discord filename values into the existing media-store API; adding a distinct metadata field is a follow-up product choice rather than a blocker for preserving saved filenames.
