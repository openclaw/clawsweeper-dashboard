## Label Justifications

- `P2`: The PR addresses a bounded browser startup diagnostic bug, but the patch has a normal-priority semantic regression rather than an emergency failure.
- `merge-risk: 🚨 compatibility`: `ensurePortAvailable` is exported through core/library surfaces, and the PR changes how non-busy bind failures are reported to existing callers.
- `merge-risk: 🚨 availability`: Treating unknown port-probe failures as available can let browser startup proceed into a later CDP readiness failure instead of failing at the preflight.

## Summary

Keep open: the PR targets a real linked bug and uses the right shared probe direction, but it should not merge until it preserves non-busy probe failures and the external PR supplies real after-fix behavior proof.

## What This Changes

The PR exports the existing multi-address port probe, routes `ensurePortAvailable` through it, and adds a regression test for an IPv4-loopback port occupant.

## Best Possible Solution

Keep the multi-family busy detection, but make the preflight reject or rethrow when availability is unknown, with focused coverage for both IPv4-only occupancy and non-`EADDRINUSE` probe failures.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main uses a hostless Node listen while the linked issue and Node docs explain why that can miss an IPv4-loopback occupant on dual-stack hosts. I did not run a live macOS/Chrome reproduction because this review was read-only.

## Solution Assessment

No, not as submitted: delegating to the existing multi-family probe is the right direction, but `ensurePortAvailable` must not treat `unknown` probe results as a free port. The safer fix preserves current non-`EADDRINUSE` failure behavior while adding the IPv4-loopback busy detection.
