## Summary

Keep open: current main still accepts approved hook session keys, but the isolated-run path unconditionally passes `forceNew: true` before `resolveCronSession` can reuse a fresh session, and PR #70895 remains an open closing implementation candidate.

## What This Changes

Webhook-triggered agent runs should reuse fresh sessions when an approved stable `sessionKey` is supplied instead of forcing a new session for every isolated run.

## Best Possible Solution

Review and land PR #70895, or an equivalent narrow change, so approved webhook-sourced isolated runs can reuse fresh sessions while ordinary scheduled isolated cron runs continue to start fresh sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: repeated approved `/hooks/agent` calls with the same `sessionKey` flow into an isolated cron job, and current main passes `forceNew: true` before `resolveCronSession` can reuse the stored session.

## Solution Assessment

Yes. The best fix direction is a narrow `forceNew` decision based on hook/webhook provenance, not a broad change that makes all isolated cron jobs reusable or loosens hook session-key policy.
