## Summary

Keep open. Current main still routes approved `/hooks/agent` requests into isolated cron execution and then forces a fresh session before the session resolver can reuse the stored `sessionId`; the issue is also paired with open PR #70895 using closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that webhook-triggered agent runs with an approved stable `sessionKey` preserve conversation history instead of always starting a fresh isolated session.

## Best Possible Solution

Review and land PR #70895, or an equivalent narrow change, so approved webhook-sourced isolated runs can reuse fresh session entries while ordinary scheduled isolated cron runs continue to start new transcript/session IDs; add a focused regression test for the webhook path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: repeated approved `/hooks/agent` calls with the same `sessionKey` flow through hook dispatch into an isolated cron job, and current main still passes `forceNew: true` before `resolveCronSession` can reuse the stored session.

## Solution Assessment

Yes. The best fix direction is a narrow `forceNew` decision based on isolated cron target plus hook/webhook provenance, not making all isolated cron jobs reusable or weakening hook session-key policy.
