## Label Justifications

- `P2`: This is a concrete hook/session continuity bug with production workflow impact, but it is bounded to webhook agent sessions and has open implementation candidates.
- `impact:session-state`: Stable-key webhook runs lose transcript/session continuity because the hook path is forced into a new isolated session id.
- `impact:data-loss`: Prior webhook conversation history becomes unavailable to later runs despite the stable session-key mapping, and old transcript files can be orphaned.
- `impact:security`: The fix changes when hook-selected session keys can reuse existing transcript context, so hook session-key policy gates matter.

## Summary

Current main and the latest release still take the same source path: approved stable-key `/hooks/agent` requests become isolated cron jobs, and isolated cron jobs still pass `forceNew: true` before the session reuse branch can run. Because this issue has an open same-author closing PR and there is also an open broader `sessionMode` implementation, it should stay open until maintainers land or reject one fix path.

## What This Changes

The issue asks for webhook-triggered agent runs with an approved stable `sessionKey` to reuse conversation session state instead of always creating a fresh session.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/70895 or the explicit persistent hook-session design in https://github.com/openclaw/openclaw/pull/75918, while keeping current request-session-key and prefix policy gates intact and ordinary isolated cron runs fresh by default.

## Reproduction Assessment

Yes, source-level. Approved stable-key `/hooks/agent` requests still become `sessionTarget: "isolated"` cron jobs, and current main passes `forceNew: true` before `resolveCronSession` can reuse the existing session id.

## Solution Assessment

Unclear as the final product choice. The narrow same-author PR fixes automatic hook reuse, while the broader `sessionMode` PR is safer if maintainers want persistent webhook sessions to require explicit opt-in.
