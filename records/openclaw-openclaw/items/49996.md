The PR changes shared reasoning-tag stripping and tests to recover answer text after malformed unclosed thinking tags, while the current branch also includes unrelated gateway, session, hook, UI, config, and generated backup-tree changes.

## Best Possible Solution

Keep the shipped shared-text implementation on main and close this PR as superseded by v2026.4.29; any remaining Telegram HTML escaping or strict whitespace-boundary hardening should be handled in a narrow follow-up rather than by merging this broad branch.

Yes. The deterministic input `<think>Answer after malformed opening tag` exercises the former empty-answer path, and current main has a focused test proving the recovered answer text; Telegram's strict split path then routes that recovered text to the answer lane.

## Solution Assessment

Yes for closing this PR, because the current main solution is the narrow maintainable path for the reported strict-strip recovery. No for merging the PR branch as-is, because it mixes the text change with unrelated gateway/session/hook/UI/generated changes and a shell-execution concern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep strict recovery from treating whitespace as zero capture:** `src/shared/text/reasoning-tags.ts:91`
  - body: The strict-mode guard still uses trimmed output, so an input like `"\n<think>secret"` can recover `secret` even though strict mode already captured a prefix. Callers use strict stripping as a safety boundary, so whitespace-formatted model output can expose an unclosed reasoning tail instead of failing closed.
  - confidence: 0.92
- **[P2] Restore the sessions.changed broadcast after reset:** `src/gateway/server-methods/sessions.ts:998`
  - body: This removes the `emitSessionsChanged` call after a successful `sessions.reset`. Other connected clients rely on `sessions.changed` to reload the updated session ID and metadata, so resets from one tab can leave other tabs stale until a manual refresh or another mutation happens.
  - confidence: 0.87
- **[P2] Add a timeout around awaited transcribed hooks:** `src/auto-reply/reply/message-preprocess-hooks.ts:31-38`
  - body: `message:transcribed` is now awaited before the agent reply starts, but the await has no timeout. A slow or stuck hook handler can block the inbound reply path indefinitely for that message, unlike the previous fire-and-forget behavior.
  - confidence: 0.84
- **[P2] Use execFile for workspace memory extraction:** `src/hooks/bundled/session-memory/handler.ts:186-188`
  - body: The new session-memory path builds a shell command with interpolated `extractScript` and `sessionId`. If either value contains shell metacharacters, the shell can execute more than the intended Python script; use `execFile("python3", [extractScript, "--session", sessionId], ...)` instead.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR branch introduces unrelated shell execution and a large generated backup tree, even though the closing decision avoids merging it.

Concerns:

- **[medium] Shell interpolation in bundled memory hook:** `src/hooks/bundled/session-memory/handler.ts:186`
  - body: `execAsync` is called with a shell command assembled from `extractScript` and `sessionId`; this can execute unintended shell syntax if path or session values contain metacharacters. Use `execFile` with an argv array if this feature remains.

  - body: The PR adds a large `dist.bak/*` generated JavaScript tree unrelated to the Telegram fix. Even if not imported, committing generated backup code makes the review surface too large and can hide executable changes from normal source review.
