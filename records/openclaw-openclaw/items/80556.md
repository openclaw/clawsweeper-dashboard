## Summary

Keep open. Current main still projects ACP live output into block deliveries and routes those through normal channel delivery, while Telegram draft editing exists on the native Telegram delivery path only. The request is valid, but the fix is an additive channel/plugin SDK live-preview feature that needs maintainer design before implementation.

## What This Changes

Route ACP `acp.stream.deliveryMode: "live"` progress for bound channel sessions into a single edited channel draft/live preview instead of separate block messages.

## Best Possible Solution

Add a generic, additive channel live-preview/draft contract for ACP dispatch under the existing `acp.stream.deliveryMode: "live"` gate, implement Telegram first using the existing send/edit preview semantics, and document fallback behavior for channels without the capability.

## Reproduction Assessment

Yes for a source-level reproduction: current main projects ACP live text into `block` deliveries and dispatches them through `routeReply` or `sendBlockReply`. I did not establish a live Telegram/ACP harness reproduction in this read-only review.

## Solution Assessment

Unclear as an exact implementation, but the direction is sound. The safer solution should reuse or extend the existing channel-message live-preview/finalizer contract if it can support ACP, rather than adding a parallel ad hoc Telegram-only method.
