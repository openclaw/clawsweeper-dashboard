## Summary

Keep open. Current main still sends ACP `deliveryMode: "live"` progress as normal block replies, while Telegram's edited preview stream is only wired into the native Telegram delivery path; solving this needs an additive channel/plugin SDK design rather than cleanup closure.

## What This Changes

Request ACP bound sessions with `acp.stream.deliveryMode: "live"` to feed channel draft/live-preview streams instead of delivering each progress block as separate messages.

## Best Possible Solution

Add a backwards-compatible, typed channel live-preview/draft contract that ACP live delivery can use, implement Telegram first, and document fallback behavior for unsupported channels.

## Reproduction Assessment

Yes for a source-level current-main path: ACP live deltas are projected as `block` replies and then delivered through `routeReply` or `sendBlockReply`. I did not run a live Telegram/ACP harness, so this is not a full end-to-end reproduction.

## Solution Assessment

Unclear as an exact implementation, but the direction is sound. The narrowest maintainable fix should extend the existing channel streaming/live-preview contracts instead of adding a Telegram-only ACP shortcut.
