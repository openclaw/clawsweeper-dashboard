## Summary

Keep open. Current `main` still sends ACP `deliveryMode: "live"` output as block deliveries rather than feeding a channel draft/live-preview stream; the remaining work is a public channel/plugin contract decision, not conservative cleanup.

## What This Changes

Requests that ACP live-mode bound sessions update a channel draft/live preview instead of emitting incremental block messages, while treating `streamTo: "parent"` as separate parent-session relay work.

## Best Possible Solution

Wire ACP live delivery through the existing typed channel live-preview/finalization contract if it can support this path, adding only the minimum backwards-compatible SDK extension needed, with Telegram first and documented fallback behavior for channels without draft-preview support.

## Reproduction Assessment

Yes for a source-level reproduction: current `main` projects ACP live deltas into `block` deliveries and routes those through `routeReply` or `sendBlockReply`. I did not run a live Telegram ACP harness, so this is not a full platform reproduction.

## Solution Assessment

Unclear as the exact proposed method. The direction is valid, but the narrowest maintainable solution should reuse or extend the existing channel message live-preview contract before adding a parallel ACP-specific channel method.
