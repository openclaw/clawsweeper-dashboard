## Summary

Keep open. Current main still turns ACP live output into block deliveries routed as normal channel messages, while Telegram draft editing remains on the native Telegram delivery path; the remaining work is a channel live-preview contract decision, not stale cleanup.

## What This Changes

Request ACP bound-session live output to update a channel draft/progress preview instead of emitting separate block messages.

## Best Possible Solution

Wire ACP live delivery through the existing channel message live-preview/finalizer contract where possible, implement Telegram first, and document fallback behavior for unsupported channels.

## Reproduction Assessment

Yes, at source level: ACP live chunks are projected as block deliveries and routed through routeReply or sendBlockReply on current main. I did not run a live Telegram/ACP harness, so this is not a full transport reproduction.

## Solution Assessment

Unclear as proposed. The corrected direction is right, but the best implementation should reuse or deliberately extend the typed channel message live-preview contract rather than expose streamTo="parent" or add an ACP-only duplicate draft-stream method.
