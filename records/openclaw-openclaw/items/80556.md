## Summary

Keep open. Current main still routes ACP live chunks through normal block delivery, while Telegram draft editing remains in the native Telegram delivery pipeline; this needs a public channel/plugin live-preview contract decision rather than conservative cleanup.

## What This Changes

The issue requests ACP live-mode progress chunks to update a channel draft stream instead of appearing as separate channel messages.

## Best Possible Solution

Add an additive typed channel live-preview/draft capability that ACP dispatch can use when `acp.stream.deliveryMode` is `"live"`, implement Telegram first, and document fallback behavior for unsupported channels.

## Reproduction Assessment

Yes for a source-level current-main reproduction: ACP live chunks are projected as `block` deliveries and then routed through `routeReply` or `sendBlockReply`. I did not run a live Telegram/ACP harness, so this is not a full end-to-end reproduction.

## Solution Assessment

Unclear as an exact implementation, but the direction is plausible. The safest fix should reuse or extend a typed channel live-preview contract rather than expose only `streamTo: "parent"`, because that relay targets parent-session context and not channel draft editing.
