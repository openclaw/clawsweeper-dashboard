## Summary

Keep open. Current main still routes ACP live chunks through normal block delivery while Telegram draft editing remains in the native Telegram dispatch path, and solving this needs an additive channel/plugin live-preview contract rather than only exposing `streamTo: "parent"`.

## What This Changes

The issue asks for ACP `deliveryMode: "live"` progress chunks in bound channel sessions to update a channel draft stream, with Telegram first, instead of emitting separate block messages.

## Best Possible Solution

Add an additive typed channel live-preview/draft capability that ACP dispatch can use when `acp.stream.deliveryMode` is `"live"`, implement Telegram first, and keep `streamTo: "parent"` as a separate parent-session relay feature.

## Reproduction Assessment

Yes for a source-level current-main reproduction: ACP live chunks are projected as block deliveries and then routed through `routeReply` or `sendBlockReply`. I did not run a live Telegram/ACP harness, so this is not full end-to-end transport proof.

## Solution Assessment

Unclear as an exact implementation, but the direction is plausible. The maintainable solution should introduce or reuse a typed channel live-preview capability rather than treating `streamTo: "parent"` as a channel draft mechanism.
