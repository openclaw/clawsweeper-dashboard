## Summary

Keep open: this is an active member PR with the protected `maintainer` label, and the current head still has a concrete P2 correctness blocker in the new policy-refresh fast path. The branch handles terminal CLI toggles, but no-id `policy-changed` callers can still take the persisted-only path without proving the changed plugin is indexed.

## What This Changes

The PR adds a persisted installed-plugin-index policy-refresh fast path, threads `policyPluginIds` through terminal `plugins enable` and `plugins disable`, extends enable results with the resolved plugin id, updates related tests, and adds a changelog entry.

## Best Possible Solution

Keep the policy-refresh optimization, but make the fast path target-aware for every policy mutation path: omitted or empty `policyPluginIds` should fall back to the full rebuild, or the chat `/plugins enable/disable` path should pass the resolved `plugin.id`. Add regression coverage for no-id fallback and the auto-reply caller before merge.

## Reproduction Assessment

Yes. The PR head shows the no-id fast path at `src/plugins/installed-plugin-index-store.ts:197` and the chat toggle caller omitting target ids at `src/auto-reply/reply/commands-plugins.ts:520`; a focused test can seed a persisted index missing a target and call `policy-changed` without ids to observe the full rebuild being skipped.

## Solution Assessment

No, not as currently proposed. The optimization is the right shape for terminal CLI toggles, but the maintainable fix needs either a strict target-id guard in the store or target ids from all policy mutation callers.
