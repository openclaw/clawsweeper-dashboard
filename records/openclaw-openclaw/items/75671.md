## Summary

Keep open. This is a draft, maintainer-labeled PR from a repository member, and current main does not yet contain the proposed policy-refresh fast path. The direction matches the plugin control-plane laziness goal, but the diff has one blocking correctness issue: the new fast path treats omitted `policyPluginIds` as safe while an existing `/plugins enable/disable` auto-reply caller still omits the changed id, so the intended missing-target fallback can be bypassed.

## What This Changes

The PR adds a persisted installed-plugin-index policy refresh fast path, passes changed plugin ids from `plugins enable`/`plugins disable`, extends enable results with the resolved plugin id, and updates related tests and the changelog.

## Best Possible Solution

Keep the policy-refresh optimization, but make the fast path target-aware across every policy mutation caller: either require `policyPluginIds` before using the persisted-only refresh or pass the resolved plugin id from the auto-reply `/plugins enable/disable` path, with regression coverage that omitted ids fall back to a full rebuild.

## Reproduction Assessment

Yes. The source path is clear without mutating the checkout: current main routes policy toggles through a full `refreshInstalledPluginIndex`, while the PR head can fast-path with `policyPluginIds` omitted and the existing auto-reply policy mutation caller still omits that field.

## Solution Assessment

No, not as currently proposed. The optimization is the right shape for terminal CLI toggles, but it is not yet the narrowest maintainable fix until all `policy-changed` mutation paths either pass the changed id or deliberately fall back to a full rebuild when no id is supplied.
