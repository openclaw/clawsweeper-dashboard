## Summary

Keep open. Current main still lacks the requested `includedWorkDirs` config/runtime capability, so the PR is not obsolete, but the branch is not merge-ready because it changes security-sensitive filesystem boundaries and still has concrete correctness issues in root validation, tool factory propagation, edit parameter handling, and private QA loading behavior.

## What This Changes

The PR adds `agents.list[].includedWorkDirs` to agent config/schema and threads extra agent work roots through path guards, `apply_patch`, PI file tools, image/PDF media handling, tests, plus private QA loader and release-check workflow adjustments.

## Best Possible Solution

Land the capability only after included roots reuse the same root-directory rejection semantics as workspaces, all tool factories derive the agent's included roots consistently from config/session context, edit semantics are preserved, private QA/workflow changes are either justified or split out, and focused docs/changelog plus regression tests cover the new public config field.

## Reproduction Assessment

Yes. The root-boundary issue is reproducible by configuring `tools.fs.workspaceOnly=true` with `includedWorkDirs: ["/"]`; the PR's resolver accepts the root and the boundary matcher treats arbitrary absolute paths as inside that allowed root.

## Solution Assessment

No. Adding an explicit per-agent allowlist is a reasonable core direction, but this implementation is not the narrowest maintainable fix until it rejects filesystem roots, resolves roots uniformly at each entrypoint, and avoids unrelated private QA/workflow behavior changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Reject filesystem-root included work dirs:** `src/agents/agent-scope.ts:191-197`
  - body: `resolveAgentIncludedWorkDirs` accepts `/` or a drive root and passes it into the allowed-root boundary list. With that root configured, every absolute host path is considered inside the allowed roots, so `tools.fs.workspaceOnly` no longer contains read/write/edit/apply_patch for that agent. Reject filesystem roots before adding them to the allowlist.
  - confidence: 0.9
- **[P2] Derive included roots for image and PDF tools:** `src/agents/openclaw-tools.ts:148`
  - body: `createOpenClawTools` forwards included roots only when callers pass `options.includedWorkDirs`. Existing gateway and inline-command paths already pass config plus session/agent identity but not this new option, so configured included roots still do not work for image/PDF tools outside `createOpenClawCodingTools`. Resolve the agent's included roots here when the option is omitted.
  - confidence: 0.86
- **[P2] Preserve whitespace-only edit targets:** `src/agents/pi-tools.read.ts:665-668`
  - body: The canonical edit adapter drops replacements where `oldText.trim().length === 0`, which silently discards valid edits targeting indentation or blank-line blocks. When all edits are filtered, it falls through with an incompatible `{ path, edits }` payload. Accept whitespace-only strings as valid `oldText` values.
  - confidence: 0.88
- **[P2] Keep the private QA loader behind its flag:** `src/cli/program/private-qa-cli.ts:82-86`
  - body: When the env flag is disabled, the source resolver returns null, but the new no-params path still tries runtime QA imports and can load the private QA CLI if a built module is present. That bypasses the explicit `OPENCLAW_ENABLE_PRIVATE_QA_CLI` gate and makes behavior depend on local layout.
  - confidence: 0.82
- **[P2] Only fall back when the candidate module is missing:** `src/cli/program/private-qa-cli.ts:87-90`
  - body: The runtime import loop treats every `ERR_MODULE_NOT_FOUND` as a missing candidate. Node uses the same code when a dependency inside the candidate module is missing, so this can mask a broken QA package and load another candidate instead of failing on the real error. Restrict fallback to errors for the candidate specifier itself.
  - confidence: 0.78
