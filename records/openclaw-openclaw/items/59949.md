## Summary

Keep open: current main does not implement `includedWorkDirs`, and this PR is a plausible feature branch, but the latest head still has blocking review issues in Windows path normalization and private QA module loading. The supplied proof is also too indirect because it only shows a downstream checkout-marker fallback, not real OpenClaw tool access allowing an included root while rejecting an outside root.

## What This Changes

The PR adds `agents.list[].includedWorkDirs` config/schema support and threads per-agent extra filesystem roots through path policy, `apply_patch`, PI file tools, image/PDF path guards, and related tests while also changing private QA loader behavior.

## Best Possible Solution

Land a narrowed, maintainer-approved per-agent extra-root design after fixing the path and QA regressions, choosing how it relates to `tools.fs.allowedRoots`, and showing redacted real tool proof for included-root allow plus outside-root rejection.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head bypasses single-root Windows sandbox input normalization and lets private QA calls without explicit params try runtime imports. The `includedWorkDirs` request itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No. A per-agent allowlist may be a reasonable core design, but this branch should not merge until the Windows path regression and private QA fallback are corrected, maintainer direction is clear against the overlapping allowed-roots proposal, and real behavior proof is supplied.

- **[P2] Preserve Windows sandbox input normalization:** `src/agents/path-policy.ts:72-74`
  - body: The new win32 branch resolves relative candidates directly with `path.win32.resolve`, so existing single-root callers of `toRelativeWorkspacePath`/`toRelativeSandboxPath` no longer run `resolveSandboxInputPath` before boundary comparison.   - body: After the source checkout probe fails, calls without explicit params fall through to `PRIVATE_QA_RUNTIME_SPECIFIERS` even when `OPENCLAW_ENABLE_PRIVATE_QA_CLI` is unset. That changes the existing `qa` command gate so a runtime `qa-lab` module can load based on file layout rather than the explicit private QA flag.
  - confidence: 0.86

  - body: This catches every `ERR_MODULE_NOT_FOUND` from importing a candidate. Node reports the same code for missing dependencies inside `qa-lab`, so a broken private QA module can be hidden while the loop tries another specifier; continue only when the missing module is the candidate itself.
