## Summary

Keep open: current main does not implement `includedWorkDirs`, and this recent feature PR still has source-visible correctness and security blockers plus insufficient real behavior proof.

## What This Changes

The PR adds an `agents.list[].includedWorkDirs` config field and threads explicit extra work roots through agent path guards, `apply_patch`, PI file tools, image/PDF media guards, schema/help, and tests while also changing private QA CLI loading.

## Best Possible Solution

Land a narrowed branch that preserves existing path and private-QA contracts, keeps included roots explicit and per-agent, retains targeted boundary coverage, and supplies redacted real tool proof for included-root allow plus outside-root rejection.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head still bypasses single-root Windows sandbox input normalization and lets private QA loading fall through to runtime imports when the env/source gate is absent. The `includedWorkDirs` request itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No. A per-agent allowlist is a plausible core direction, but this branch should not merge until the unrelated private QA fallback is corrected, Windows single-root path normalization is preserved, and real included-root behavior is shown in a real setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve Windows sandbox input normalization:** `src/agents/path-policy.ts:71-74`
  - body: The Windows branch now resolves `params.candidate` directly with `path.win32.resolve`, so single-root callers of `toRelativeWorkspacePath`/`toRelativeSandboxPath` no longer get the existing `resolveSandboxInputPath` normalization. Inputs like `~/file.txt` can be treated as `<cwd>\~\file.txt` instead of the normalized target when included roots are not involved; keep the resolved-input step before the platform branch.
  - confidence: 0.84
- **[P2] Keep private QA runtime imports behind the opt-in:** `src/cli/program/private-qa-cli.ts:78-86`
  - body: When source resolution returns null and this helper is called without explicit params, it now falls through to runtime specifier imports. A present `qa-lab` runtime module can then load the private command without the existing env/source-checkout gate, so preserve the current error unless `OPENCLAW_ENABLE_PRIVATE_QA_CLI=1` enables it.
  - confidence: 0.86
- **[P2] Only ignore missing candidate modules during QA fallback:** `src/cli/program/private-qa-cli.ts:88-90`
  - body: This catches every `ERR_MODULE_NOT_FOUND` while importing a candidate. Node also uses that code for dependencies missing inside `qa-lab`, so a broken private QA module can be hidden while the loop tries another specifier; only continue when the missing module is the candidate itself.
  - confidence: 0.78
