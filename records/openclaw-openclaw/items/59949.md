## Summary

Keep open and do not merge as-is. The feature is still absent from current main and the PR implements the right general contract, but the current head has a security-sensitive filesystem containment gap plus two concrete runtime regressions that should block merge.

## What This Changes

Adds `agents.list[].includedWorkDirs` config/schema support and threads extra agent work roots through workspace inference, path policy, `apply_patch`, PI file tools, image/PDF guards, tests, and related CI/QA tweaks.

## Best Possible Solution

Land the capability only after included roots reject filesystem roots, every tool entrypoint derives the configured roots consistently, canonical edit semantics are preserved, and unrelated workflow/QA changes are split or explicitly reviewed.

## Reproduction Assessment

Yes. On the PR head, configure `tools.fs.workspaceOnly=true` with `includedWorkDirs: ["/"]`; the resolver returns `/`, and the boundary matcher then treats arbitrary absolute host paths as inside the allowed roots.

## Solution Assessment

No. A per-agent allowlist is a reasonable core design, but this implementation is not the safest maintainable version until containment, entrypoint consistency, and canonical edit behavior are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject filesystem-root included work dirs:** `src/agents/agent-scope.ts:191-197`
  - body: `resolveAgentIncludedWorkDirs` accepts `/` or a drive root and forwards it into the allowed-root list. In `tools.fs.workspaceOnly` mode, that makes arbitrary absolute host paths fall inside the agent boundary for read/write/edit/apply_patch, so reject filesystem roots before returning included roots.
  - confidence: 0.9
- **[P2] Resolve included roots in OpenClaw tool factories:** `src/agents/openclaw-tools.ts:148`
  - body: `createOpenClawTools` only forwards included roots when callers pass `options.includedWorkDirs`. Gateway and inline-command paths already pass config plus session/agent context but not this new option, so configured included roots still do not work for image/PDF tools outside the coding-tools path.
  - confidence: 0.86
- **[P2] Preserve whitespace-only edit targets:** `src/agents/pi-tools.read.ts:667`
  - body: The canonical edit adapter drops replacements whose `oldText` is whitespace-only. Valid edits that target indentation or blank-line blocks are silently discarded, and all-whitespace edit batches fall back to the legacy tool with the wrong payload shape.
  - confidence: 0.88
