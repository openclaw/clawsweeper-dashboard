## Summary

Keep open. Current main still lacks the requested `includedWorkDirs` config/runtime support, and the PR is useful source work, but the latest head is not merge-ready because it expands filesystem containment while still accepting filesystem roots and has two remaining behavioral gaps in tool entrypoints and edit handling.

## What This Changes

Adds `agents.list[].includedWorkDirs` and threads extra agent work roots through config/schema, path policy, `apply_patch`, PI file tools, image/PDF path guards, tests, and related CI/QA adjustments.

## Best Possible Solution

Land the capability after included roots reject filesystem roots, every tool entrypoint derives the agent's configured roots consistently, canonical edit semantics are preserved, unrelated changes are split or justified, and docs/changelog plus focused regression tests cover the public field.

## Reproduction Assessment

Yes. On the PR head, configure `tools.fs.workspaceOnly=true` with `includedWorkDirs: ["/"]`; the resolver returns `/`, and the boundary matcher then treats arbitrary absolute paths as inside the allowed roots.

## Solution Assessment

No. A per-agent allowlist is a reasonable core design, but this implementation is not the safest maintainable version until the containment and entrypoint gaps are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject filesystem-root included work dirs:** `src/agents/agent-scope.ts:191-197`
  - body: `resolveAgentIncludedWorkDirs` accepts `/` or a drive root and forwards it into the allowed-root list. In `tools.fs.workspaceOnly` mode, that makes arbitrary absolute host paths fall inside the agent boundary for read/write/edit/apply_patch. Reject filesystem roots before returning included roots.
  - confidence: 0.9
- **[P2] Resolve included roots in OpenClaw tool factories:** `src/agents/openclaw-tools.ts:148`
  - body: `createOpenClawTools` only forwards included roots when callers pass `options.includedWorkDirs`. Gateway and inline-command paths already pass config plus session/agent context but not this new option, so configured included roots still do not work for image/PDF tools outside the coding-tools path. Derive the agent roots here when the option is omitted.
  - confidence: 0.86
- **[P2] Preserve whitespace-only edit targets:** `src/agents/pi-tools.read.ts:665-668`
  - body: The canonical edit adapter drops replacements whose `oldText` is whitespace-only. Valid edits that target indentation or blank-line blocks are silently discarded, and all-whitespace edit batches fall back to the legacy tool with the wrong payload shape. Accept whitespace-only strings as valid edit targets.
  - confidence: 0.88
