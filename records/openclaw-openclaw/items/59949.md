Keep open. Current main does not implement `includedWorkDirs`, and the PR is an active feature branch, but the latest head still has source-inspected correctness/security blockers plus insufficient real behavior proof for the filesystem-access change.

## What This Changes

The PR adds `agents.list[].includedWorkDirs` and threads extra allowed filesystem roots through agent scope, path guards, `apply_patch`, PI file tools, and image/PDF tools, while also changing private QA CLI module fallback behavior.

## Best Possible Solution

Land a narrowed branch after restoring existing path and private-QA gates, keeping included roots explicit and per-agent, preserving root-bound mutation checks, and adding redacted real proof of included-root allow plus outside-root rejection.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head still bypasses single-root Windows sandbox input normalization and lets private QA loading fall through to runtime imports when the source/env gate is absent. The `includedWorkDirs` request itself is a new feature, not a current-main bug reproduction.

## Solution Assessment

No, not yet. A per-agent allowlist is a plausible core direction, but this branch should not merge until the unrelated private QA fallback is corrected, Windows single-root normalization is preserved, and actual included-root allow/reject behavior is shown in a real setup.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve Windows sandbox input normalization:** `src/agents/path-policy.ts:71-74`
  - body: The Windows branch now resolves `params.candidate` directly with `path.win32.resolve`, so single-root callers of `toRelativeWorkspacePath`/`toRelativeSandboxPath` no longer run `resolveSandboxInputPath`. Inputs like `~/file.txt` or `@...` can be interpreted as literal paths under the cwd instead of using the existing normalization, so keep the resolved-input step before the platform branch.
  - confidence: 0.86
- **[P2] Keep private QA runtime imports behind the opt-in:** `src/cli/program/private-qa-cli.ts:78-86`
  - body: When source resolution returns null and the helper is called without explicit params, this falls through to runtime specifier imports. A present `qa-lab` runtime module can then load the private command without the existing `OPENCLAW_ENABLE_PRIVATE_QA_CLI=1` source-checkout gate, so preserve the current error unless the flag enables it.
  - confidence: 0.86
- **[P2] Only ignore missing candidate modules during QA fallback:** `src/cli/program/private-qa-cli.ts:88-90`
  - body: This catches every `ERR_MODULE_NOT_FOUND` while importing a candidate. Node also uses that code for dependencies missing inside `qa-lab`, so a broken private QA module can be hidden while the loop tries another specifier; only continue when the missing module is the candidate itself.
  - confidence: 0.78
