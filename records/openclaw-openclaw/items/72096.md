## Summary

Keep open. Current main still builds light Dream Diary input from a lookback window and appends every successful narrative without a body-level duplicate guard; the existing dedupe path remains timestamp-sensitive. Open PR #70332 is already the active fix candidate, so this should not be closed or sent to a second repair lane unless that PR is rejected or replaced.

## What This Changes

The issue asks for light-phase Dream Diary work summaries to be incremental or deduplicated so identical commit-list summaries do not repeat across repeated dreaming cycles.

## Best Possible Solution

Memory-core should suppress unchanged automatic Dream Diary output before append, or track an incremental light-summary emission boundary, with regression coverage proving repeated light sweeps over unchanged input do not create duplicate diary bodies while genuinely new or changed narrative prose still appends.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append the same Dream Diary narrative twice with different timestamps, or run repeated light dreaming sweeps over unchanged deterministic input; current main can preserve both dated entries because append has no body guard and dedupe includes the timestamp.

## Solution Assessment

Unclear. The issue's `lastEmittedAt` cutoff is plausible, but PR #70332's normalized body-level skip may be the narrower user-visible repair if it prevents repeated DREAMS.md bodies without dropping legitimately new diary prose.
