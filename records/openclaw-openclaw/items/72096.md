## Summary

Keep open. Current main still takes light-phase Dream Diary input from a lookback window and appends every successful narrative, while the existing diary dedupe path treats the timestamp as part of the duplicate key. The prior ClawSweeper comment remains directionally correct: open PR #70332 is the active fix candidate for duplicate diary bodies, so this should stay open until that PR lands or is replaced.

## What This Changes

The issue asks memory-core dreaming to stop repeating identical light-phase Dream Diary work summaries across cycles by using incremental emission state or duplicate-body suppression.

## Best Possible Solution

Memory-core should prevent unchanged automatic Dream Diary output from being appended, either by suppressing normalized duplicate bodies before append or by tracking a light-phase emission boundary, with regression coverage proving repeated unchanged light sweeps do not add duplicate diary entries while genuinely changed prose still appends.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append the same Dream Diary narrative twice with different timestamps, or run repeated light dreaming over unchanged deterministic input; current main can preserve both dated entries because append has no body guard and dedupe includes the timestamp.

## Solution Assessment

Unclear. The issue's lastEmittedAt cutoff could fix repeated light summaries, but PR #70332's normalized body-level skip may be the narrower user-visible repair if it avoids repeated DREAMS.md bodies without adding phase state.
