## Summary

Keep open. Current main still builds light Dream Diary input from lookback-filtered short-term recall and appends each successful narrative without a duplicate-body guard; the existing diary dedupe remains timestamp-sensitive. Open PR #70332 is already the active fix candidate for body-level Dream Diary duplicate suppression, so this should stay open for that PR to be reviewed, landed, or replaced rather than closed or sent to a second repair lane.

## What This Changes

The issue asks memory-core dreaming to stop repeating identical light-phase Dream Diary/work-summary content across hourly cycles by tracking incremental emission or deduping repeated output.

## Best Possible Solution

Memory-core should suppress unchanged automatic Dream Diary output before append, with regression coverage proving repeated light sweeps over unchanged input do not create duplicate diary bodies while genuinely new snippets or changed narrative prose still append.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append the same Dream Diary narrative twice with different timestamps, or to run two light dreaming sweeps over stable recall input with deterministic narrative output; current main can preserve both dated entries because the append path lacks a body guard and dedupe includes the timestamp.

## Solution Assessment

Unclear. The issue's `lastEmittedAt` cutoff is plausible for light summaries, but PR #70332's normalized body-level skip may be the narrower user-visible guard if it preserves legitimate new diary prose.
