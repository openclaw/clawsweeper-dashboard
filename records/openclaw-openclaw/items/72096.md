## Summary

Keep open. Current main still builds light dreaming input from a rolling lookback window and appends every successful Dream Diary narrative without a body-level duplicate guard; the existing dedupe includes the formatted timestamp, so identical prose emitted at different times is preserved. Related open PR #70332 is already the active fix candidate for body-level diary dedupe, so this issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

Request memory-core make light-phase Dream Diary/work-summary output incremental or deduplicated so repeated overnight cycles do not append identical content to DREAMS.md.

## Best Possible Solution

Memory-core should suppress unchanged automatic Dream Diary output before append, or track an incremental light-summary emission boundary, with regression coverage proving repeated light sweeps over unchanged deterministic input do not create duplicate diary bodies while genuinely new or changed diary prose still appends.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append the same Dream Diary narrative twice with different timestamps, or run repeated light dreaming sweeps over unchanged deterministic input; current main can preserve both dated entries because append has no body guard and dedupe includes the timestamp.

## Solution Assessment

Unclear. The requested lastEmittedAt cutoff is plausible, but PR #70332's normalized body-level skip may be the narrower user-visible repair if it prevents repeated DREAMS.md bodies without dropping legitimately new diary prose.
