## Summary

Keep open. Current main still builds light Dream Diary input from a lookback window and appends every successful narrative without a duplicate-body guard, while the existing diary dedupe remains timestamp-sensitive. The existing discussion correctly points at open PR #70332 as the active fix candidate, so this issue should stay open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue requests that light-phase Dream Diary work summaries stop repeating identical commit-list content across dreaming cycles, either by tracking an incremental cutoff or by deduping unchanged diary output.

## Best Possible Solution

Memory-core should suppress unchanged automatic Dream Diary output before append, or otherwise track an incremental light-summary emission boundary, with regression coverage proving repeated light sweeps over stable input do not create duplicate diary bodies while genuinely new or changed narrative prose still appends.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append the same Dream Diary narrative twice with different timestamps, or run repeated light dreaming sweeps over unchanged deterministic input; current main can preserve both dated entries because append has no body guard and dedupe includes the timestamp.

## Solution Assessment

Unclear. The issue's `lastEmittedAt` cutoff is a plausible phase-level fix, but PR #70332's normalized body-level skip may be the narrower user-visible repair if it prevents repeated DREAMS.md bodies without dropping legitimately new diary prose.
