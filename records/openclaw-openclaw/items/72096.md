## Summary

Keep open. Current main still feeds light Dream Diary generation from a lookback-filtered candidate set and appends each generated narrative without a duplicate-body guard, while the existing dedupe path intentionally treats the timestamp as part of the fingerprint. Related PR #70332 is already the active fix candidate for body-level Dream Diary duplicate suppression, so the issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue asks memory-core dreaming to stop repeating identical light-phase work-summary Dream Diary content across hourly overnight cycles, either by emitting only new work or by deduping repeated diary bodies.

## Best Possible Solution

Memory-core should suppress unchanged automatic Dream Diary output before append, with regression coverage proving repeated light sweeps over stable input do not create duplicate diary bodies while changed narrative prose or genuinely new snippets still append.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to append the same Dream Diary narrative twice with different timestamps, or run two light dreaming sweeps over stable recall input with deterministic narrative output; current main preserves both dated entries because append lacks a body guard and dedupe includes the timestamp.

## Solution Assessment

Unclear. The issue's `lastEmittedAt` cutoff is plausible, but PR #70332's normalized body-level skip may be the narrower user-visible fix if it preserves legitimate new diary prose.
