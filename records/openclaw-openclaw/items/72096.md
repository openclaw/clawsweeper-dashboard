## Summary

Keep open. Current main still rebuilds light Dream Diary input from the configured lookback window and appends each successful narrative without a body-level duplicate guard; the existing diary dedupe intentionally treats the same body at different timestamps as distinct. Open PR #70332 already targets duplicate diary-body appends, so this should stay open for that PR to be reviewed, landed, or reworked rather than closed or sent to a second repair lane.

## What This Changes

The issue asks memory-core dreaming to avoid emitting identical light-phase Dream Diary work-summary content across repeated overnight cycles, either by tracking an emitted cutoff or deduping prior emitted content.

## Best Possible Solution

Memory-core should suppress unchanged automatic Dream Diary output before append, with regression coverage for repeated light sweeps over unchanged input and for genuinely new snippets or changed prose still producing a new entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed stable daily memory or short-term recall input, run two light dreaming sweeps inside the same lookback window with deterministic narrative text, and observe that current append/dedupe logic can keep the same body under different timestamps.

## Solution Assessment

Unclear. The issue's lastEmittedAt cutoff is plausible for light summaries, but PR #70332's normalized body-level skip may be the narrower central guard if it preserves legitimate new diary prose.
