## Summary

Keep open: current main still stores workspace setup state in the nested `.openclaw/workspace-state.json` location and this PR is the canonical follow-up for the closed nested-workspace report, but the branch is not mergeable as-is because it has blocking fallback, migration, collision, and proof gaps.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Revise this PR around the same goal: keep workspace setup state out of dot-prefixed workspace entries, use a collision-resistant non-dot state filename or detect foreign files before writing, make legacy dot-path probes best-effort, promote loaded legacy state to the new location, rebase over current bootstrap-state repairs, and add focused regression coverage plus real setup proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main resolves and tests the state file under `.openclaw`, and the PR-specific failure paths follow from legacy reads rethrowing non-ENOENT errors and root `workspace-state.json` being parsed as empty state. I did not run a live TigerFS mount in this read-only review.

## Solution Assessment

No. The direction is right, but this patch is not the best mergeable fix until it handles legacy I/O errors, foreign-file collisions, migration promotion, current-main rebasing, tests, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-ENOENT errors. On the dot-restricted filesystems this PR targets, first-run/status checks can fail before the new root state is created, so legacy fallback reads need to ignore or safely handle those I/O errors.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Moving OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing project file with the same generic name. A non-OpenClaw JSON file is parsed as empty state and later overwritten when setup markers are persisted, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on `.openclaw/workspace-state.json`; persist fallback-loaded state to complete migration.
  - confidence: 0.86
