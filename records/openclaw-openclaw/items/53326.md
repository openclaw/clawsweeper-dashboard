## Summary

Keep this PR open: current main and the latest release still store workspace setup state under the nested `.openclaw` path, and the closed related issue points here as the canonical implementation, but the branch still has blocking correctness issues plus missing real behavior proof.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, add legacy `.openclaw` fallback reads, and stop creating the legacy state directory.

## Best Possible Solution

Revise this PR around the same goal: keep setup state out of dot-prefixed workspace entries, use a collision-resistant non-dot filename or foreign-file guard, make legacy probes best-effort, promote legacy state, and add focused tests.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.5.12` still resolve and test workspace setup state under `<workspace>/.openclaw/workspace-state.json`; I did not run a live TigerFS mount.

## Solution Assessment

No. The direction is appropriate, but the current PR is not the best complete fix until it handles legacy probe failures, root-file collision risk, legacy-state promotion, and current-main rebase fallout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-ENOENT errors. On dot-restricted filesystems this can make setup/status fail before the new root state is created, so legacy lookup errors need to be ignored or treated as unavailable.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers if that directory becomes unavailable.
  - confidence: 0.86
