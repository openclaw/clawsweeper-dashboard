## Summary

Keep open: current main and the latest release still store workspace setup state under a dot-prefixed `.openclaw` workspace subdirectory, and the linked nesting issue was closed in favor of this PR, but the PR head still has blocking migration/collision issues and lacks real behavior proof.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read `<workspace>/.openclaw/workspace-state.json`, and stop creating the legacy state directory.

## Best Possible Solution

Revise this PR around the same goal: use a collision-safe non-dot state filename or foreign-file guard, make legacy dot-path probes best-effort, promote legacy state to the new location, and add focused current-main tests.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.5.12` still resolve and test workspace setup state under `<workspace>/.openclaw/workspace-state.json`; I did not run a live TigerFS mount.

## Solution Assessment

No. The direction is appropriate, but the current PR is not the best complete fix until it handles legacy probe failures, root-file collision risk, legacy-state promotion, and current-main rebase fallout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root state has no markers, including first run. On the dot-restricted filesystems this PR targets, that legacy probe can throw non-ENOENT errors and abort setup/status before the new root state is created, so the fallback needs to ignore unavailable legacy paths.
  - confidence: 0.91
- **[P2] Use a collision-safe state filename:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so choose a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on `.openclaw/workspace-state.json` and can lose completion markers if that directory is unavailable, so persist fallback-loaded state to the new path.
  - confidence: 0.86
