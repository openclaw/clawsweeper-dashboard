## Summary

Keep open. Current main still stores workspace setup state under `.openclaw/workspace-state.json`, but this PR has blocking fallback, migration, and data-integrity issues that should be revised before merge.

## What This Changes

The PR changes `src/agents/workspace.ts` to write setup state at `<workspace>/workspace-state.json`, fallback-read legacy `.openclaw/workspace-state.json`, and stop creating the state subdirectory.

## Best Possible Solution

Revise this PR around the same compatibility goal with a collision-resistant non-dot state location or foreign-file guard, best-effort legacy probing, explicit legacy-state promotion, and focused workspace tests.

## Reproduction Assessment

Yes. The diff supports targeted reproductions for a legacy dot-path `EPERM`/`EINVAL` probe failure, a foreign root `workspace-state.json` overwrite, and a legacy-only completed state that is read without being migrated.

## Solution Assessment

No. Moving state out of the dot directory is a reasonable direction, but this implementation is not the narrowest safe fix until it handles legacy probe errors, root filename collisions, and persistence of migrated legacy state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-`ENOENT` errors. On the dot-restricted filesystems this PR targets, first-run setup and status checks can still fail before the new root state is written; catch and ignore legacy-probe I/O errors in the fallback path.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing workspace file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Persist fallback-loaded legacy state at the new path:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on `.openclaw/workspace-state.json`; mark fallback-loaded state dirty or explicitly write it to complete migration.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, release, or secrets changes, but it introduces a concrete local data-integrity risk with a generic root state filename.

Concerns:

- **[medium] Generic root state file can clobber workspace data:** `src/agents/workspace.ts:207`
  - body: Moving OpenClaw state to `<workspace>/workspace-state.json` without namespacing or a foreign-file guard can overwrite an existing workspace file with the same name, which is a data-integrity regression for user-managed workspaces.
  - confidence: 0.88
