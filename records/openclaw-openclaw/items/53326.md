## Summary

Keep open. Current main still writes workspace setup state under the nested dot-prefixed path, while this PR’s branch has unresolved correctness, data-integrity, and proof blockers before it should merge.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Revise this PR around the same goal with a collision-resistant non-dot filename or foreign-file guard, best-effort legacy probes, explicit legacy-state promotion, focused workspace tests, and redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main still writes the dot-prefixed `.openclaw/workspace-state.json` path, and the PR diff gives high-confidence source paths for the remaining fallback, promotion, and collision failures.

## Solution Assessment

No. Moving state out of the dot directory is the right direction, but this implementation is not the safest fix until it handles legacy probe errors, migration persistence, and root filename collision policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-`ENOENT` errors. On dot-restricted filesystems this can make first-run setup or status checks fail before root state is written; catch and ignore legacy-probe I/O errors in the fallback path.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign content before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on the dot-prefixed path and can lose completion markers if that directory becomes unavailable; persist fallback-loaded state to the new path.
  - confidence: 0.86
