## Summary

Keep open: current main and the latest release still use the nested workspace setup-state path, and the closed related issue points remaining work here. The PR is useful but not mergeable as-is because the head still has blocking fallback, migration, and root-file collision issues, plus missing real behavior proof.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the state subdirectory.

## Best Possible Solution

Revise this PR around the same bug fix: use a collision-resistant non-dot state location or foreign-file detection, make legacy dot-path probes best-effort, promote legacy state to the new location, preserve current atomic-write safety without dot-prefixed target entries, and add focused workspace tests.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.5.7` still resolve workspace setup state to `<workspace>/.openclaw/workspace-state.json`; I did not run filesystem writes because this review is read-only.

## Solution Assessment

No. The direction is appropriate, but the current PR head can fail while probing the legacy dot path, can clobber a generic root `workspace-state.json`, and does not promote fallback-loaded legacy state to the new location.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy fallback reads best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-`ENOENT` I/O errors. On the dot-restricted filesystems this PR targets, that can make setup/status checks fail before the new root state is written, so legacy probes should be caught and ignored when unavailable.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can overwrite an existing project file with the same generic name. If that JSON has no OpenClaw markers, setup treats it as empty state and later persists over it, so the new root filename needs namespacing or foreign-file detection before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:400-401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers when that directory is unavailable.
  - confidence: 0.86
