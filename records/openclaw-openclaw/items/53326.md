## Summary

Keep open: the underlying workspace-state path problem is still present on current main and in the latest release, and this PR remains the canonical focused attempt, but the branch is currently conflicting and has blocking correctness issues before it can merge.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Rebase and revise the PR around the same goal: keep workspace setup state out of dot-prefixed workspace entries, avoid generic root-file clobbering, make legacy probes best-effort, promote legacy state to the new location, adapt the current atomic writer so it does not require dot-prefixed temp entries on the target filesystem, and add focused workspace tests.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.5.7` still write state under `<workspace>/.openclaw/workspace-state.json`; I did not run filesystem writes because this review was required to keep the checkout read-only.

## Solution Assessment

No. The goal is the right fix direction, but this PR as written can throw while probing the legacy dot path, can overwrite a generic root `workspace-state.json`, does not promote fallback-loaded legacy state, and must be rebased onto current main's workspace writer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: The fallback read calls `readWorkspaceSetupState()` on `.openclaw/workspace-state.json`, and that helper rethrows non-`ENOENT` errors. On the dot-restricted filesystems this PR targets, the first-run fallback can raise `EPERM`/`EINVAL` and abort setup or status checks before the new root state is written.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to a generic `<workspace>/workspace-state.json` can clobber an existing project file. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so the root filename should be namespaced or foreign files should be detected before writing.
  - confidence: 0.88
- **[P2] Promote legacy state after fallback reads:** `src/agents/workspace.ts:401-402`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces can keep depending on the dot-prefixed location and lose completion markers when that path is unavailable.
  - confidence: 0.86
