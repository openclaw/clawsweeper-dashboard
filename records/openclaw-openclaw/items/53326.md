## Summary

Keep open. Current main still stores workspace setup state under the dot-prefixed workspace subdirectory, so the goal remains valid, but this PR is not merge-ready because the fallback reads can still fail on the target filesystems, legacy state is not promoted, the new root filename can clobber a foreign file, and real behavior proof is missing.

## What This Changes

The PR changes `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, fallback-read legacy `<workspace>/.openclaw/workspace-state.json`, and stop creating the dot-prefixed state directory.

## Best Possible Solution

Revise this PR around the same goal: use a collision-resistant non-dot state name or foreign-file guard, make legacy dot-path probes best-effort, promote legacy state to the new location, rebase over current bootstrap repairs, and add targeted tests plus real proof.

## Reproduction Assessment

Partly. Current main's nested state path is source-reproducible from `src/agents/workspace.ts` and tests, but the dot-restricted filesystem failure was not live-tested in this read-only review.

## Solution Assessment

No. Moving setup state out of a dot-prefixed workspace entry is a reasonable direction, but this patch still probes the legacy dot path as a fatal read, does not complete migration, and uses a generic root filename without a collision guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` falls back to `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-`ENOENT` errors. On the dot-restricted filesystems this PR targets, that fallback can still raise `EPERM`/`EINVAL` and abort first-run/status checks before the new root state is written, so legacy probes should ignore non-`ENOENT` lookup failures.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing project file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates `<workspace>/workspace-state.json`. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers if that directory becomes unavailable; persist fallback-loaded state to the new path.
  - confidence: 0.86
