## Summary

Keep open. Current main still supports MCP config editing and session-time tool inventory, but it does not expose the requested per-server connectivity/error state or refresh/reconnect workflow in the Gateway or Control UI.

## What This Changes

Add a Control UI MCP connectivity section that shows each configured MCP server's connection status, exposed tools, manual refresh/reconnect action, and error details.

## Best Possible Solution

Design an additive Gateway operator diagnostics surface for configured MCP servers, backed by the MCP runtime/config loader, with redacted per-server state, tool names/counts after a successful handshake, last safe error details, and a scoped refresh action; then render it under Control UI Infrastructure > MCP and document the behavior.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification is high-confidence: the UI only exposes MCP config/counts, docs state config commands do not connect to servers, and no Gateway MCP health or reconnect method exists.

## Solution Assessment

Yes in principle: an additive Gateway diagnostics API plus Control UI panel is the maintainable path. Reusing `tools.catalog` or `tools.effective` alone is not sufficient because those surfaces cannot represent configured-but-failed MCP servers or trigger a refresh/reconnect.
