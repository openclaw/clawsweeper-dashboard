## Summary

Keep open. Current main still exposes MCP as saved config plus session-scoped tool inventory, but not the requested per-server connectivity/error state or refresh/reconnect workflow in the Gateway or Control UI.

## What This Changes

Add a Control UI MCP Connectivity panel that shows per-server status, tool names/counts, safe error details, and refresh/reconnect actions for configured `mcp.servers`.

## Best Possible Solution

Design an additive operator-only Gateway MCP diagnostics surface backed by MCP config/runtime state, with redacted per-server status, tool names/counts after handshake, safe last-error details, and a scoped refresh action rendered under Control UI Infrastructure > MCP with docs.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence: the Gateway has no MCP diagnostics method, the Control UI renders only config/count surfaces, and docs state saved MCP commands do not validate reachability.

## Solution Assessment

Yes in principle: an additive Gateway diagnostics API plus Control UI panel is the maintainable direction. Reusing `tools.catalog` or `tools.effective` alone is not enough because those surfaces cannot show configured-but-failed MCP servers or trigger reconnect.
