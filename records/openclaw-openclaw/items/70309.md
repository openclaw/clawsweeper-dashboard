## Summary

Keep open. Current main still exposes MCP as saved configuration plus session-scoped tool inventory, not a per-server Control UI connectivity panel with failure details or refresh/reconnect controls; the related open MCP issues cover agent notification and fast degradation, but not this UI/operator diagnostics surface.

## What This Changes

Request to add a Control UI MCP Connectivity panel showing per-server connection state, exposed tools, manual refresh/reconnect, and redacted error details.

## Best Possible Solution

Design an additive operator-only Gateway MCP diagnostics surface backed by config and runtime state, then render it in Control UI with redacted per-server status, safe error summaries, successful tool names/counts, and a scoped refresh action.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence: the Gateway has no MCP diagnostics method, Control UI shows only config/count surfaces, and the runtime drops failed-server state after logging it.

## Solution Assessment

Yes in principle: an additive Gateway diagnostics API plus Control UI panel is the maintainable direction. It should not be attempted as a narrow automated fix until maintainers settle redaction, authorization, reconnect lifecycle, and overlap with the related MCP health issues.

## Risks / Open Questions

- MCP startup errors can include secret-bearing URLs, headers, env-derived values, or stderr, so any UI/API response needs strict redaction before display.
- A refresh/reconnect control can spawn or dispose transports, so operator authorization, rate limiting, lifecycle ownership, and session impact need maintainer design.
- Related open issues cover agent notification and runtime degradation, so maintainers should decide whether this remains a separate Control UI diagnostics issue or becomes part of a broader MCP health contract.
