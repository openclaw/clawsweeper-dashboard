## Summary

Keep open. Current main still treats configured MCP servers as config plus session-time successful tool inventory, but it does not expose a Control UI or Gateway diagnostics surface for configured-but-failed servers, per-server connected/error state, redacted last errors, tool names by server, or manual refresh/reconnect.

## What This Changes

Add a Control UI MCP Connectivity panel for `mcp.servers` that shows per-server connection state, tool counts/names, refresh or reconnect controls, and redacted error details.

## Best Possible Solution

Design an additive Gateway operator diagnostics API for OpenClaw-managed MCP servers, backed by the existing MCP config/runtime loader, and render it in Control UI under an Infrastructure or MCP area with configured server name, transport summary, state, tool count/names, redacted last error, and a scoped refresh action.

## Reproduction Assessment

Not applicable as a feature request. The current-main inspection is high confidence: existing UI surfaces only show MCP configuration/counts, docs explicitly say config commands do not connect, and no Gateway MCP health or reconnect method exists.

## Solution Assessment

Yes, the requested direction is best solved as a small core Gateway protocol/server-method plus Control UI panel. Reusing `tools.catalog` or `tools.effective` alone is insufficient because they cannot show configured-but-failed MCP servers or drive a reconnect action.
