## Summary

Keep open. Current main still exposes MCP through saved config and session-scoped tool inventory, but it does not provide the requested per-server connectivity state, safe error details, or refresh/reconnect workflow in the Gateway or Control UI.

## What This Changes

Add a Control UI MCP Connectivity panel showing per-server status, exposed tool names/counts, error details, and refresh/reconnect actions for configured `mcp.servers`.

## Best Possible Solution

Ship an additive operator-only Gateway MCP diagnostics API and Control UI Infrastructure > MCP panel with redacted per-server state, tool inventory, safe errors, and scoped refresh/reconnect.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken defined behavior. Current-main source verification is high-confidence: Gateway has no MCP diagnostics method, Control UI renders only config/count surfaces, and docs say saved MCP commands do not validate reachability.

## Solution Assessment

Yes in principle: an additive Gateway diagnostics API backing a Control UI panel is the maintainable direction. Reusing `tools.catalog` or `tools.effective` alone would miss configured-but-failed MCP servers and cannot safely trigger reconnect.
