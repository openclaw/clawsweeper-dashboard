## Summary

Keep open: current main and the latest release still leave the subagent list/control path dependent on exact requester/controller key equality, while the canonical fix candidate at https://github.com/openclaw/openclaw/pull/75679 is open and unmerged. This should remain the canonical issue until that PR is refreshed, repaired, and landed or replaced.

## What This Changes

The issue asks for spawned subagents to appear in `/subagents list` and the agent-facing subagents list/control surface when the requester is represented by equivalent main-session aliases.

## Best Possible Solution

Make subagent spawn registration and every list/control reader share one canonical owner-key resolver, then verify both the agent-tool list and Telegram `/subagents list` against the same spawned child.

## Reproduction Assessment

Yes at source level: current main still resolves slash-command requester keys without full main-session alias canonicalization, then filters subagent runs by exact stored owner-key equality. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes, the maintainable fix direction is narrow: use one canonical owner-key resolver for both registration and list/control reads. The existing linked PR is the right candidate lane, but it needs refresh and the writer-side/read-side symmetry proven before the issue can close.
