## Summary

Keep open: current main and v2026.6.10 still resolve `/subagents` requester keys through a non-canonical alias path and then filter subagent runs by exact owner-key equality, while the linked closing PR remains open and unmerged.

## What This Changes

The issue reports that spawned subagents can be visible from the agent tool surface but absent from `/subagents list` when equivalent parent-session aliases do not compare equal.

## Best Possible Solution

Refresh, review, and land or replace the linked fix so spawn registration plus list/control readers share one canonical controller-key resolver, then verify both agent-tool list and Telegram `/subagents list` against the same child run.

## Reproduction Assessment

Yes at source level: current main preserves non-literal requester aliases and then filters controlled subagent runs by exact key equality. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Yes in direction: the maintainable fix is a shared canonical owner-key resolver across spawn registration and list/control lookup. The open PR is the right candidate path, but it still needs refresh, review, and real behavior proof before this issue should close.
