## Summary

Keep open: current main still records the fast main-session `next-heartbeat` handoff as success without a ghost-run warning or persisted warning marker, and the same-author closing implementation PR is still open rather than merged.

## What This Changes

The issue requests detection and logging or run-log annotation for very fast successful main-session `systemEvent` cron handoffs that may not have reached an agent turn.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72677, or an equivalent scoped warning, run-log marker, config, docs, and regression-test change, while leaving stronger delivery-confirmation semantics to a separate scheduler design decision.

## Reproduction Assessment

Yes by source inspection. Current main still has a `main` `systemEvent` `next-heartbeat` path that enqueues the event, requests a heartbeat, and returns success without confirmed processing or the requested warning.

## Solution Assessment

Yes for the narrow observability problem, with a maintainer decision on the public threshold config. A warning plus persisted run-log marker is smaller than changing cron to synchronously confirm agent processing.
