## Summary

Keep open. Current main still has the fast-return main-session cron handoff and no ghost-run warning, and #72677 is an open implementation PR that uses closing syntax for this issue, so the issue should stay open until that PR lands or maintainers choose another fix path.

## What This Changes

Main-session `systemEvent` cron jobs using `wakeMode: "next-heartbeat"` can be recorded as fast `ok` runs without evidence that an agent turn processed the event, and the request asks for a configurable ghost-run warning/run-log signal.

## Best Possible Solution

Review and land #72677 or an equivalent maintainer-approved change that adds a scoped ghost-run warning/run-log annotation, exposes the threshold through cron config/schema/help, and keeps the warning distinct from confirmed agent-processing semantics.

## Reproduction Assessment

Yes. The source-level path is clear: a main-session `systemEvent` cron job with `wakeMode: "next-heartbeat"` enqueues the event, requests a heartbeat, returns `ok`, and is logged without a warning or threshold check.

## Solution Assessment

Yes for the narrow observability request. A scoped warning plus run-log annotation is a maintainable fix for silent fast handoffs; changing cron to await confirmed agent processing would be larger behavioral work and should be a separate product decision if needed.
