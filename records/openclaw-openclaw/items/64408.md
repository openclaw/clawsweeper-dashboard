## Summary

Keep open. Current main still emits the malformed `read` warning for pathless embedded tool-start events without run/session/agent correlation, and the related alias issue/PR only covers `file`/`filePath` false positives rather than identifying a genuinely pathless caller.

## What This Changes

The issue asks OpenClaw to diagnose and fix embedded/background `read` tool calls that reach `handleToolExecutionStart` without a valid `path`, preferably by adding run/session correlation first.

## Best Possible Solution

Add correlation fields to the existing pathless-read diagnostic with focused regression coverage, then use those logs to patch the specific retry or recovery caller while preserving the canonical `path` validation contract.

## Reproduction Assessment

Yes for the handler-level behavior: current source and tests show a `read` tool-start event with `{}` args reaches the warning path. No for the full Feishu/reconnect/failover trigger, which still lacks enough runtime data to identify the caller.

## Solution Assessment

Yes for the first repair step: enriching the existing warning with already-available identity is narrower than changing read aliases or guessing at the embedded caller. The actual caller fix should follow after correlated logs identify it.
