## Summary

Keep open: the PR targets the right Discord REST and gateway surfaces for the IPv4-only regression, but the proposed lookup helper does not honor Node's all-address lookup callback shape and the direct REST path would bypass OpenClaw's configured global undici dispatcher.

## What This Changes

The PR adds a Discord DNS lookup helper, wires it into Discord REST and gateway WebSocket agents, extends focused proxy tests, and adds a changelog fix entry.

## Best Possible Solution

Keep the Discord-owned IPv4 preference, but make the lookup function compatible with Node autoSelectFamily/all-address callers and preserve existing global dispatcher and env proxy behavior.

## Reproduction Assessment

Yes for the blocking PR regression: a targeted Node HTTPS Agent probe shows custom lookups are called with { all: true }, and the PR helper's single-address callback shape fails that path. The linked IPv4-only production issue is source-reproducible from current main, but I did not live-test an affected ISP/network.

## Solution Assessment

No. The Discord extension is the right boundary, but this implementation must honor all:true lookup callbacks and avoid bypassing OpenClaw's global EnvHttpProxyAgent/dispatcher policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor all:true DNS lookup callbacks:** `extensions/discord/src/network-config.ts:71`
  - body: Node's HTTPS/TLS connector calls custom lookup functions with `{ all: true }` when auto-select-family is active. This helper still calls back with a single string address, which Node treats as an invalid address list and fails before connecting, so the new direct gateway and REST agents can break startup instead of fixing it. Return the reordered address array when `lookupOptions.all` is set, or explicitly disable that caller mode.
  - confidence: 0.94
- **[P2] Preserve env proxy routing for direct REST fetches:** `extensions/discord/src/monitor/rest-fetch.ts:61-64`
  - body: On current main, the no-Discord-proxy path returns the global `fetch`, so the CLI-installed undici global dispatcher still honors `HTTP_PROXY`/`HTTPS_PROXY` via `EnvHttpProxyAgent`. This branch always supplies a new direct `Agent` dispatcher for Discord REST calls, bypassing that configured proxy and global dispatcher policy for users behind outbound proxies. Compose the IPv4 lookup with the existing global policy or keep the global dispatcher when env proxy mode is active.
  - confidence: 0.88
