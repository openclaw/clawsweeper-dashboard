## Summary

Keep open. The PR targets a real Discord regression path, but the new DNS lookup helper violates Node's `lookup` callback contract under `autoSelectFamily`, so the proposed direct REST and gateway agent paths can fail before connecting.

## What This Changes

The PR adds Discord-local IPv4-first DNS lookup wiring for REST and gateway networking, plus focused proxy/gateway tests and a changelog entry.

## Best Possible Solution

Keep the Discord-local transport fix, but make the lookup helper preserve Node's `all: true` callback shape and add regression coverage for that contract before merging.

## Reproduction Assessment

Yes for the PR defect: Node's connector calls custom lookups with `all: true` when auto-family selection is enabled, and a scalar callback result reproduces `ERR_INVALID_IP_ADDRESS`. The original Discord regression is source-backed by current main and the linked two-machine report, but I did not live-test the affected Windows network.

## Solution Assessment

No. The PR uses the right owner-local surface, but the helper must honor the `LookupFunction` overloads before it can safely solve Discord REST and gateway startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Return arrays for all:true DNS lookups:** `extensions/discord/src/network-config.ts:64`
  - body: The helper always calls the lookup callback as `(err, address, family)` after internally resolving all addresses. Node's connector passes `options.all: true` when `autoSelectFamily` is enabled, and in that mode the callback must receive a `LookupAddress[]`; returning a scalar produces `ERR_INVALID_IP_ADDRESS`, so the new direct Agent/HttpsAgent paths can fail before opening Discord connections. Preserve the caller's `all` contract and add a focused regression test.
  - confidence: 0.94
