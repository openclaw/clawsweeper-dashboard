## Summary

Keep open: this PR is aimed at a real Discord IPv4-only regression, but the current diff still leaves the monitor client's normal no-proxy REST client on global fetch and is missing the required user-facing changelog entry.

## What This Changes

The PR adds a Discord-specific IPv4-first DNS lookup helper, wires it into selected Discord REST and gateway transports, and updates focused Discord monitor tests.

## Best Possible Solution

Thread the Discord DNS-aware fetch/lookup through the actual Discord monitor RequestClient path, keep the gateway WebSocket lookup wiring, cover the startup REST client with focused tests, and add a single-line changelog entry.

## Reproduction Assessment

No, not as a high-confidence live reproduction: I did not recreate the IPv4-only network. The source path is reproducible with medium confidence because current main leaves no-proxy Discord REST and gateway WebSocket paths without a scoped lookup, and #77398 reports the IPv4-first workaround restoring READY and inbound messages.

## Solution Assessment

No, not yet: wiring a Discord-scoped lookup in the owner plugin is the right direction, but this diff does not install it on the monitor client's normal REST RequestClient path. The narrower maintainable fix is to reuse the new fetcher where client.rest is constructed or replaced, then add regression coverage and the changelog line.
