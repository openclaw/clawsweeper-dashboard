## Summary

Keep open. Current main still routes generic Telegram callback turns with the callback query id as the turn message id, and the related open metadata PR does not yet fix all of these `MessageSid` paths.

## What This Changes

The issue requests using the originating Telegram callback message `message_id` for `messageIdOverride` instead of `callback.id`.

## Best Possible Solution

Update the generic Telegram callback synthetic-message paths to preserve `callback.id` for callback answering/metadata while deriving `messageIdOverride` from `callbackMessage.message_id`, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. A `callback_query` whose `id` differs from `message.message_id` reaches the generic callback paths, and current main passes the query id into `messageIdOverride`.

## Solution Assessment

Yes. The narrow maintainable fix is to change the callback synthetic-message overrides to the source Telegram message id while leaving callback answering, dedupe, and plugin callback metadata semantics intact.
