## Summary

Keep open. Current main and the latest release still route generic Telegram callback turns through `callback.id` as the turn `MessageSid`, and the correct fix is not the one-line swap because dedupe still needs a distinct callback event identity.

## What This Changes

The issue asks Telegram callback handling to stop using `CallbackQuery.id` as `messageIdOverride` and preserve the originating Telegram `message_id` for callback turns.

## Best Possible Solution

Carry both identities through the Telegram callback path: keep `callback.id` for answering and dedupe, and expose the source `message_id` plus chat id where actions, prompt metadata, abort-cutoff, and threading need the real Telegram message.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` with distinct `id` and `message.message_id` values reaches current generic callback paths that pass `callback.id` into `messageIdOverride`, which becomes `MessageSid`.

## Solution Assessment

No for the literal one-line swap. The maintainable fix must preserve callback query identity for dedupe and `answerCallbackQuery` while carrying the real source message id separately.
