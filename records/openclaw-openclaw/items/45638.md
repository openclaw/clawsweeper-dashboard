## Summary

Current main still contains the reported Telegram callback bug: the generic callback_query synthetic-message path passes the callback query id into `MessageSid`. The related open PR #55548 is adjacent but not a complete fix for this issue because its patch keeps the bad `messageIdOverride` and adds a separate callback message id field instead.

## What This Changes

The issue asks Telegram callback_query handling to use the originating callback message's `message_id` as `messageIdOverride` instead of the opaque `callback.id`.

## Best Possible Solution

Use `String(callbackMessage.message_id)` only for the generic callback synthetic-message `messageIdOverride`, keep `callback.id` for `answerCallbackQuery` and callback identity, and add a focused regression plus changelog entry.

## Reproduction Assessment

Yes, by source inspection: a callback_query with data that reaches the generic synthetic-message branch still passes `callback.id` into `messageIdOverride`, and context construction maps that directly to `MessageSid`. I did not run live Telegram or Vitest because this review was constrained to read-only inspection.

## Solution Assessment

Yes. The narrow maintainable fix is the one proposed by the issue: change only the callback synthetic-message override to `String(callbackMessage.message_id)` and add regression coverage that proves `MessageSid` stays numeric while `answerCallbackQuery` still uses `callback.id`.
