## Summary

Keep open. Current main and the latest release still pass the Telegram callback query id as `messageIdOverride` on generic callback paths, while the Telegram type contract makes `callback.id` the callback query identifier and `callback.message.message_id` the message identifier.

## What This Changes

Telegram callback query handling should use the originating Telegram `message_id` for `messageIdOverride` instead of the callback query id.

## Best Possible Solution

Update the generic Telegram callback synthetic-message paths to keep `callback.id` only for callback answering/metadata and use `String(callbackMessage.message_id)` for `messageIdOverride`, with focused regression coverage.

## Reproduction Assessment

Yes, source-level. A `callback_query` with `id: "cbq-1"` and `message.message_id: 10` reaches generic callback handling and currently sets `MessageSid` from `cbq-1` via `messageIdOverride`.

## Solution Assessment

Yes. The narrow maintainable fix is to change the three generic callback `messageIdOverride` assignments to the source message id, while preserving `callback.id` for `answerCallbackQuery`, callback ids, and dedupe semantics.
