## Summary

Keep open. Current main still routes generic Telegram callback turns with the callback query id as the turn `MessageSid`, and prior related PR review shows the literal one-line swap would regress callback dedupe, so this needs a scoped fix rather than cleanup closure.

## What This Changes

The issue asks Telegram callback_query handling to use the originating `message.message_id` instead of `callback.id` for callback-derived message identity.

## Best Possible Solution

Split callback event identity from the source Telegram message id, preserve `callback.id` for answering and dedupe, expose the originating `callbackMessage.message_id` where agents and abort-cutoff/action flows need a real Telegram message id, and update all generic callback paths with regression coverage.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` whose `id` differs from `message.message_id` reaches current generic callback paths that still pass `callback.id` into `messageIdOverride`, which becomes `MessageSid`.

## Solution Assessment

No, not as a one-line `messageIdOverride` swap. The maintainable fix needs a separate callback event/dedupe identity while carrying the real source message id to agent-visible and abort/action paths.
