## Summary

Keep open: current `main` still sends Telegram callback fallback messages through `processMessage` with `messageIdOverride: callback.id`, and that override becomes `MessageSid`. The related open #55548 is adjacent to callback metadata but leaves this `MessageSid` path unchanged.

## What This Changes

The issue asks Telegram inline button callbacks to use the callback message's Telegram `message_id` as `messageIdOverride` instead of the callback query id.

## Best Possible Solution

Update only the Telegram callback fallback to set `messageIdOverride` from `String(callbackMessage.message_id)`, keep `callback.id` for callback acknowledgement/plugin callback metadata, and add regression coverage for `MessageSid`.

## Reproduction Assessment

Yes, source-reproducible: invoke the registered Telegram `callback_query` handler with `id: "cbq-1"` and `message.message_id: 10`, then inspect the inbound payload. Current `main` will route `MessageSid` from the callback query id because the handler passes that value as `messageIdOverride`.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. The callback query id should remain available for `answerCallbackQuery` and plugin callback metadata, while the synthetic inbound message id should come from Telegram `message.message_id`.
