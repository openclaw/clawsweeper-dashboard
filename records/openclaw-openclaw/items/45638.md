## Summary

Keep open. Current main still routes generic Telegram callback turns with `callback.id` as `MessageSid`, and prior related PR discussion shows the literal one-line swap would regress callback dedupe, so this needs a scoped fix rather than cleanup closure.

## What This Changes

The issue requests that Telegram `callback_query` handling use the originating callback message's `message_id` instead of `callback.id` for callback turn message identity.

## Best Possible Solution

Carry both identities through the Telegram callback path: keep `callback.id` for answering/dedupe/event identity, expose `callbackMessage.message_id` and chat id where actions, prompt metadata, abort-cutoff, and threading need a real Telegram message id, and cover all generic callback paths with regression tests.

## Reproduction Assessment

Yes, source-level. A Telegram `callback_query` whose `id` differs from `message.message_id` reaches current generic callback paths that still pass `callback.id` into `messageIdOverride`, which becomes `MessageSid`.

## Solution Assessment

No, not as a one-line `messageIdOverride` swap. The maintainable fix needs a separate callback event/dedupe identity while carrying the real source message id to agent-visible, abort/action, and threading paths.
