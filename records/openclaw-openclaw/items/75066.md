## Summary

Keep open. Current main still has the timestamp-refresh behavior reported by the linked bug, and this PR is a narrow unmerged implementation candidate that looks directionally correct but still needs two small pre-merge cleanups: the new test fixture should not hard-code `/tmp`, and the user-facing fix needs a changelog entry.

## What This Changes

The PR adds a preserve-activity option to session-store entry updates, applies it to compaction and memory-flush metadata writes, removes compaction bookkeeping `updatedAt` updates, and adds regression tests for legacy daily/idle reset behavior.

## Best Possible Solution

Land the narrow preserve-activity fix after the test fixture uses the platform temp directory and the changelog records the user-facing session reset fix. The reset policy itself should stay unchanged; only metadata-only compaction and memory-flush writers should opt into activity preservation.

## Reproduction Assessment

Yes. The linked bug gives high-confidence legacy daily and idle session-store scenarios, and current main source confirms the reproduction path: metadata writes advance `updatedAt` while freshness falls back to `updatedAt` when explicit lifecycle timestamps are missing.

## Solution Assessment

Mostly yes. Reusing the existing preserve-activity merge policy for selected compaction and flush metadata writers is the narrowest maintainable fix; the safer mergeable version also fixes the cross-platform fixture and satisfies changelog policy.
