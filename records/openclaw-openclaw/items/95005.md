## Summary

Keep this PR open: the direction is useful for the linked sandbox outage issue, and Mantis now shows the chat-only Telegram path improving, but the patch is not merge-ready because `needed` still leaves the Codex app-server native process/filesystem surface able to treat the run as unsandboxed.

## What This Changes

The PR adds `agents.defaults.sandbox.mode: "needed"`, run-vs-tool sandbox activation, lazy embedded `exec` sandbox resolution, and matching sandbox explain, audit, test, and docs updates.

## Best Possible Solution

Before merge, make `needed` lazily sandbox or fail-close every sandbox-bound surface, including Codex app-server native tools, then prove both chat-only Docker-off success and sandbox-bound failure behavior.

## Reproduction Assessment

Yes, from source inspection rather than a live Docker-off run: `needed` reports unsandboxed for run activation while the Codex app-server native surface derives host/native capability from that run-level sandbox context.

## Solution Assessment

No. Lazy embedded exec is a plausible part of the fix, but the PR is too narrow because Codex app-server native process and filesystem execution also need the same fail-closed sandbox invariant.
