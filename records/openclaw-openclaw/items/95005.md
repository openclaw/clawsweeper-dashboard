## Summary

Keep this draft PR open for maintainer/security review because the new `needed` sandbox mode is useful but the current patch leaves a Codex native-execution path using run-level sandbox state, so users can believe sandbox-bound execution is isolated while Codex-native shell/filesystem/process surfaces remain host-capable.

## What This Changes

The PR adds `agents.defaults.sandbox.mode: "needed"`, lazy sandbox resolution for embedded exec, updated sandbox explain/audit messaging, and focused tests for the new activation mode.

## Best Possible Solution

Finish the `needed` contract across embedded exec and Codex-native execution so sandbox-bound tools either lazily create the OpenClaw sandbox or fail closed, with fresh-install and upgrade proof.

Yes by source inspection: the PR makes `needed` unsandboxed for run activation, while current Codex app-server native execution policy still keys off run-level sandbox state.

## Solution Assessment

No; the embedded exec lazy path is plausible, but the fix is incomplete until Codex-native execution follows the same fail-closed sandbox-bound invariant or is explicitly excluded.
