## Summary

Keep open. Current main still lacks the requested BlueBubbles-owned pre-dispatch audio transcript enricher: there is no BlueBubbles audio-transcript endpoint client, no `channels.bluebubbles.inboundAudioEnricher` config, no UTI-aware transcript rewrite before dispatch, and no scoped public `message_transcribed` hook. The existing generic media-understanding path can transcribe audio through configured STT providers, but that is not the same as using BlueBubbles' on-device transcript endpoint before the agent sees the message.

## What This Changes

Add a BlueBubbles inbound audio enricher that fetches Apple's BlueBubbles audio transcript before agent dispatch, rewrites placeholder audio bodies, emits a scoped transcription hook, and exposes opt-out configuration.

## Best Possible Solution

Implement the behavior in the BlueBubbles plugin after maintainer approval of the config and hook contract: add a guarded client method for the transcript endpoint, detect audio by MIME and UTI, rewrite empty/audio-placeholder bodies before dispatch, avoid transcript logging, and document/test opt-out defaults plus the scoped event payload.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. Source-level verification is high confidence: an attachment-only BlueBubbles audio webhook currently follows the placeholder-plus-media path, not a BlueBubbles transcript rewrite path.

## Solution Assessment

Unclear until maintainers approve the public hook/config shape. The generic `tools.media.audio` transcription path is useful fallback behavior, but it is not the best solution for this request because it uses configured STT providers later in the shared reply pipeline instead of BlueBubbles' on-device transcript endpoint before dispatch.
