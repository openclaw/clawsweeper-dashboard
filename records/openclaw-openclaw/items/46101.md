## Summary

Keep open. Current main still implements and documents the mixed per-agent heartbeat scoping rule, checked-in tests reproduce that defaulted agents are disabled when any listed peer has an explicit heartbeat block, and the doctor preview warning collector still has no heartbeat-shadowing diagnostic. The broader behavior question remains related to open #41879, but this issue is a narrower diagnostics gap that can be fixed without changing scheduler semantics.

## What This Changes

The issue asks for `openclaw doctor` to warn when `agents.defaults.heartbeat` is configured but mixed `agents.list[].heartbeat` blocks cause listed agents without explicit heartbeat blocks to stop running heartbeats.

## Best Possible Solution

Add a non-mutating doctor config-health warning that detects `agents.defaults.heartbeat` plus at least one explicit `agents.list[].heartbeat`, then names listed agents missing explicit heartbeat blocks and explains that only explicit heartbeat agents currently run. Keep runtime scoping unchanged in this issue and leave any inheritance-policy change to #41879.

## Reproduction Assessment

Yes. A high-confidence reproduction is a config with `agents.defaults.heartbeat`, `main` listed without `heartbeat`, and `ops` listed with `heartbeat`; current source and checked-in tests show `main` is disabled while `ops` runs.

## Solution Assessment

Yes. A doctor warning is the narrowest maintainable solution for this report because the mixed scoping rule is implemented and documented today, while changing that rule would be a broader policy fix tracked separately.
