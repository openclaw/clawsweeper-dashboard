## Summary

Keep open: current main still direct-delivers ACP task terminal messages when a requester origin exists, and this PR’s intended fix is useful but misses real `:group:` session keys while carrying unrelated release/plugin changes.

## What This Changes

The PR routes channel-scoped ACP task completions through the parent session, adds task-registry coverage and docs, and also includes many unrelated release, plugin, workflow, and test changes.

## Best Possible Solution

Land a narrow task-registry fix that classifies both group and channel owner sessions, queues ACP terminal/state updates through the parent session, adds focused regression coverage and docs, and splits unrelated release/plugin work.

## Reproduction Assessment

Yes, from source inspection: current main direct-sends ACP terminal updates whenever `requesterOrigin` has a deliverable channel/to pair, and existing tests cover that generic direct-delivery behavior. I did not execute tests because this review is read-only.

## Solution Assessment

No: the intended parent-session routing is the right direction, but the PR should cover documented `:group:` keys and be narrowed to the task/docs/test surface before merge.
