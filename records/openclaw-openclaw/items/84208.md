## Summary

Keep open: the click-coords guard addresses a real current-main validation bug and the PR now includes copied CLI output, but the resize hunk duplicates the existing shared validation used by both resize and set viewport.

## What This Changes

The PR adds local `Number.isFinite` guards to browser CLI `click-coords` and `resize` positional numeric arguments before request helpers run.

## Best Possible Solution

Keep the click-coords CLI guard, drop or centralize the resize validation in `runBrowserResizeWithOutput`, and add focused regression coverage if feasible.

## Reproduction Assessment

Yes, from source inspection: current main converts non-numeric `click-coords` arguments with `Number()` and sends the resulting values into the `/act` body. The resize half is already source-reproducible as covered by the shared helper, not by the PR’s added navigation guard.

## Solution Assessment

No, not as written. The `click-coords` guard is the right boundary, but the resize validation should remain in the shared helper or be centralized there to avoid duplicate behavior.
