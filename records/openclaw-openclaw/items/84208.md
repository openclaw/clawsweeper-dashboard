## Summary

Keep open: the click-coords fix is valid and the PR body now includes copied CLI output, but the resize hunk duplicates current shared validation and would leave `browser resize` with a separate alias-specific error path.

## What This Changes

The PR adds finite-number guards to the browser CLI `click-coords` and `resize` handlers before dispatching browser actions.

## Best Possible Solution

Keep the `click-coords` CLI-boundary guard, remove or centralize the resize validation in `runBrowserResizeWithOutput`, and add focused CLI regression coverage when practical.

## Reproduction Assessment

Yes. Source inspection shows `browser click-coords abc 200` flows through `Number('abc')` with no guard before the `/act` body is built, and a Node probe confirms that NaN serializes to JSON null; the resize half is already rejected by the shared helper on current main.

## Solution Assessment

No, not as written. The `click-coords` guard is the right boundary, but the resize guard should be removed or moved into the existing shared helper so `browser resize` and `browser set viewport` stay aligned.
