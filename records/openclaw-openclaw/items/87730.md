## Summary

Keep open: current main still has a source-checkout path where refresh can persist mixed built/source bundled plugin roots, and the persisted-registry validator then rejects source-root bundled entries as a different bundled tree.

## What This Changes

The issue reports that `openclaw plugins registry --refresh` writes a manual registry but `openclaw plugins list --json` still reports `persisted-registry-stale-source` in a source-checkout install.

## Best Possible Solution

Make a manual registry refresh produce a persisted index that the next cold registry read accepts for source checkouts with mixed built/source bundled plugin roots, and cover that with a focused regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: in a source checkout with a usable `dist/extensions` tree plus source-only bundled plugins, refresh can persist source-root bundled records and the next list rejects them as outside the resolved bundled root. I did not execute the reporter's macOS CLI commands in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow plugin-registry repair that aligns refresh and persisted validation for source-checkout bundled roots. It should not add a new user option or weaken stale checks for real missing or moved plugin files.
