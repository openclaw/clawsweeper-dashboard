## Summary

Keep open. Current main still has only DM and group inbound paths for WhatsApp, so a Baileys message whose `remoteJid` ends in `@newsletter` is treated as non-group and dropped when phone/LID normalization returns null. The related open PR is explicitly outbound message-tool support and does not close this inbound newsletter processing request.

## What This Changes

Add inbound WhatsApp Channel/Newsletter message support with a newsletter-specific access-control/config path for JIDs ending in `@newsletter`.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin, not core: explicitly detect `@newsletter` JIDs, preserve the newsletter JID as the conversation identity, add conservative newsletter-specific config and account overrides, route allowed newsletter messages through normal inbound processing, and document the new policy surface with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A code-level reproduction is clear: feed a `messages.upsert` notify message with user content and `key.remoteJid` ending in `@newsletter`; current main treats it as non-group, `resolveJidToE164` returns null, and the inbound message is discarded before `onMessage`. Tests were not run because this review was required to be read-only.

## Solution Assessment

Yes, the requested separate newsletter access path is the maintainable direction. Reusing `dmPolicy` would keep conflating public broadcast channels with direct chats and would not preserve a distinct newsletter identity or conservative default.
