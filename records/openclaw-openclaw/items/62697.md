## Summary

Keep open. Current main still has only direct and group WhatsApp inbound paths, so `@newsletter` messages are not preserved or authorized separately, and the related open PR is outbound message-tool work rather than a fix for inbound newsletter processing.

## What This Changes

Add WhatsApp inbound support for Channel/Newsletter messages from `@newsletter` JIDs with a newsletter-specific access policy.

## Best Possible Solution

Implement this inside the bundled WhatsApp plugin: detect `@newsletter` JIDs, keep the newsletter JID as the conversation identity, add conservative newsletter-specific config and account overrides, route allowed inbound messages through the normal reply pipeline, and update docs, tests, and changelog.

## Reproduction Assessment

Yes. A code-level reproduction is clear: emit a Baileys `messages.upsert` notify message with user content and `key.remoteJid` ending in `@newsletter`; current main treats it as non-group, fails JID-to-phone resolution, and returns before `onMessage`.

## Solution Assessment

Yes. A separate newsletter access path is the narrow maintainable direction because public broadcast channels should not inherit DM pairing approvals or be collapsed into phone-number direct-chat identity.
