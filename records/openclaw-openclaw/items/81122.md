## Summary

Keep open: current main and the latest release still mark an inbound Twilio call active when stream TwiML is generated, but there is no pending-stream timeout for the case where Twilio never sends a media-stream start, so the stale state can still route a later caller to the hold queue.

## What This Changes

The issue reports that Twilio inbound voice-call stream TwiML can leave stale `activeStreamCalls` state after a failed/no-stream attempt, causing later inbound calls to hear hold music.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve queueing for genuinely active streams.

## Reproduction Assessment

Yes, at source level: one inbound webhook can add a CallSid to `activeStreamCalls`, and without WebSocket start, unregister, hangup, or terminal status cleanup, a later inbound webhook follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by `registerCallStream` and existing cleanup paths is the narrowest maintainable fix; making queueing configurable would be a separate product choice.
