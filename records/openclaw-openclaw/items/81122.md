## Summary

Keep open: current main still marks an inbound Twilio call active as soon as stream TwiML is returned, but has no pending-stream timeout if the WebSocket start never arrives; that can leave stale state and route later inbound calls to the hold queue.

## What This Changes

Request clearing Twilio inbound active-stream state when stream TwiML is issued but no media stream connects, so later callers do not get hold music from stale state.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve queueing for genuinely active streams.

## Reproduction Assessment

Yes, at source level: one inbound webhook can add a call SID to activeStreamCalls, and without register/unregister/status cleanup a later inbound webhook follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by registerCallStream and existing cleanup paths is the narrowest maintainable fix; making queueing configurable would be a separate product choice.
