## Summary

Keep open: current main still has the reported no-start stale-state window, because Twilio inbound stream TwiML marks a call active before any media stream registers and there is no pending-stream timeout cleanup.

## What This Changes

The issue asks for Twilio inbound streaming to clear stale `activeStreamCalls` entries when a stream TwiML response never leads to a connected media stream.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for call SIDs activated by inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve the queue behavior for genuinely active streams.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a first inbound webhook that returns stream TwiML adds the call SID to `activeStreamCalls`, and without unregister/status cleanup a later inbound webhook follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes, a provider-owned pending-stream timeout canceled by `registerCallStream()` and existing cleanup paths is the narrowest maintainable fix. Making queueing configurable is a separate product choice and should not be required to fix the stale no-stream bug.

## Repair Work Prompt

Fix the Twilio inbound streaming stale-active-call bug reported in https://github.com/openclaw/openclaw/issues/81122. In `extensions/voice-call/src/providers/twilio.ts`, when inbound TwiML generation adds `activateStreamCallSid` to `activeStreamCalls`, start a short unref'd pending-stream timer that deletes that call SID if no media stream has registered. Cancel and clear that timer when `registerCallStream()` succeeds, and in all existing cleanup paths: `unregisterCallStream()`, terminal call status handling, and `hangupCall()`. Preserve the existing call-waiting queue behavior for genuinely active streams; do not add a queue config option or change TwiML policy semantics unless the narrow fix cannot work. Add focused Vitest coverage in `extensions/voice-call/src/providers/twilio.test.ts` for: first inbound returns stream TwiML, no stream registers, timer expires, next inbound gets stream TwiML rather than hold queue; timer is canceled on successful `registerCallStream()`; terminal status cleanup still clears state/timer. Add a `CHANGELOG.md` Fixes entry because this is user-visible voice-call behavior. Validate with the targeted provider tests, then `pnpm check:changed` if the touched surface remains narrow.
