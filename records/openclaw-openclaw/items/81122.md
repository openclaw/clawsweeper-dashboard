## Summary

Keep open: current main and the latest release still have a source-proven Twilio path where stream TwiML marks a call active before any media stream starts, and no pending-stream timeout or duplicate fix was found.

## What This Changes

The issue requests clearing Twilio inbound stream-active state when stream TwiML is issued but the WebSocket media stream never starts.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve queueing for genuinely active streams.

## Reproduction Assessment

Yes, at source level: one inbound webhook can add a call SID to activeStreamCalls, and a later inbound webhook queues while no register/unregister/status cleanup has occurred. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by registerCallStream and terminal cleanup is the narrowest maintainable fix; making queueing configurable would be a separate product decision.
