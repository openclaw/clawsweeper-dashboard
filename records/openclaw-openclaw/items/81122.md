## Summary

Keep open: current main still has the reported Twilio no-start stale-state window, because inbound stream TwiML marks a call active before any media stream registers and there is no pending-stream timeout cleanup. No duplicate or linked fix PR currently owns this exact `activeStreamCalls` failure.

## What This Changes

The issue asks for Twilio inbound stream call SIDs to be cleared when stream TwiML never becomes a connected media stream, so later calls are not sent to hold music by stale state.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for call SIDs activated by inbound stream TwiML, cancel it on successful stream registration and terminal cleanup paths, and keep queue behavior only for genuinely active streams.

## Reproduction Assessment

Yes, by source inspection: a first inbound webhook that returns stream TwiML adds the call SID to `activeStreamCalls`, and without a media-stream connect, unregister, or terminal status cleanup, a later inbound webhook follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by `registerCallStream()` and existing cleanup paths is the narrowest maintainable fix; making queueing configurable is a separate product choice, not required for the stale no-stream bug.
