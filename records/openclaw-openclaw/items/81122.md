## Summary

Keep open: current main still has the no-start stale-state window where inbound Twilio stream TwiML marks a call active before any media stream registers, so a later inbound call can be sent to the hold queue.

## What This Changes

Fix Twilio inbound voice-call cleanup so a failed or never-started media stream cannot leave `activeStreamCalls` stale and route later calls to hold music.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve queueing for genuinely active streams.

## Reproduction Assessment

Yes, at source level: one inbound webhook can add a call SID to `activeStreamCalls`, and without a WebSocket start/register or terminal cleanup, a later inbound webhook follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by `registerCallStream` and existing cleanup paths is the narrow maintainable fix; making queueing configurable would be a separate product decision.
