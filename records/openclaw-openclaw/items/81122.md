## Label Justifications

- `P1`: A real user-reported Twilio voice-call workflow can leave later inbound calls stuck in hold music with no stream or transcript.
- `impact:message-loss`: Later inbound calls can be accepted by Twilio but fail to reach OpenClaw streaming/transcript handling.
- `impact:session-state`: The root cause is stale provider call state in activeStreamCalls after a no-start media stream attempt.

## Summary

Keep open: current main still marks an inbound Twilio call active when stream TwiML is returned, and there is no provider-side pending-stream timeout for the case where Twilio never opens or starts the media WebSocket.

## What This Changes

The issue reports that a failed or no-stream Twilio inbound voice-call can leave stale active stream state and route later inbound calls to hold music.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve queueing for genuinely active streams.

## Reproduction Assessment

Yes, at source level: parse one inbound Twilio webhook to add the first call SID, do not register a media stream, then parse a second inbound webhook and it follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by registerCallStream and existing cleanup paths is the narrowest maintainable fix; making queueing configurable would be a separate product decision.
