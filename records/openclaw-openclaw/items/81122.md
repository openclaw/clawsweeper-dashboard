## Summary

Keep open: current main still has the reported Twilio stale-state path. Inbound stream TwiML marks a call active before any media stream registers, later inbound calls queue when any active call exists, and there is no pending-stream timeout to clear a no-start stream SID.

## What This Changes

This issue requests a Twilio voice-call fix so failed inbound media-stream startup cannot leave `activeStreamCalls` stale and route later callers to hold music.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for SIDs activated by inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and preserve queue behavior for genuinely active streams.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction path: one inbound webhook returns stream TwiML and adds the call SID to `activeStreamCalls`, then a later inbound webhook queues because no stream registration or terminal cleanup cleared that SID. I did not run a live Twilio call in this read-only review.

## Solution Assessment

Yes, a provider-owned pending-stream timeout canceled by `registerCallStream()` and terminal cleanup is the narrowest maintainable fix. Making queueing configurable is a separate product choice and should not be required for this stale-state bug.
