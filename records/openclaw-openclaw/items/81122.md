## Summary

This should stay open: current main still has a source-reproducible stale-state path where an inbound Twilio call can be marked active when stream TwiML is generated, but no timeout clears that state if the media stream never reaches the register/unregister lifecycle.

## What This Changes

The issue requests cleanup for stale Twilio `activeStreamCalls` entries when an inbound call is given stream TwiML but no media stream ever connects.

## Best Possible Solution

Keep the queue feature, but add a narrow pending-stream cleanup path that clears only unregistered inbound stream activations after a short timeout and cancels that cleanup on real stream registration, stream teardown, terminal status, or hangup.

## Reproduction Assessment

Yes: source inspection gives a high-confidence path by sending one inbound Twilio webhook that returns stream TwiML without any register/unregister or terminal callback, then sending a second inbound webhook and observing the queue decision. I did not run a live Twilio call or local tests because this review was read-only.

## Solution Assessment

Yes: a provider-local pending activation timeout is the narrowest maintainable fix; disabling queueing or adding a new queue configuration would broaden the behavior change beyond the reported stale-state bug.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81122. In the Twilio voice-call provider, prevent stale `activeStreamCalls` entries when inbound stream TwiML is issued but the WebSocket media stream never reaches `registerCallStream()`. Add a provider-local pending stream activation cleanup: when `activateStreamCallSid` is added, start a short conservative timeout; if that call SID still has no registered stream when the timeout fires, remove it from `activeStreamCalls` and clean up any pending timer. Cancel/clear the pending timer on `registerCallStream()`, `unregisterCallStream()`, terminal status callbacks, and `hangupCall()`. Do not disable the existing queue behavior, do not add a new config option unless maintainers explicitly ask, and avoid a broad Twilio queue redesign. Add focused Vitest coverage in the Twilio provider tests for: first inbound returns stream, immediate second inbound still queues, no-stream timeout allows a later inbound call to stream, real `registerCallStream()` before the timeout keeps the active-call queue behavior until unregister, and terminal status cleanup cancels pending state. Add a changelog entry because this is user-visible voice-call behavior.
