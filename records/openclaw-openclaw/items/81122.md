## Summary

Keep open: current main still has the stale no-start window because Twilio inbound stream TwiML marks a call active before any media stream registers, with no pending-stream timeout cleanup.

## What This Changes

Fix Twilio inbound voice-call handling so a failed or never-started media stream does not leave stale active state that routes later calls to hold music.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for SIDs activated by inbound stream TwiML, cancel it on successful stream registration and existing cleanup paths, and keep queueing only for genuinely active streams.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a first inbound webhook returns stream TwiML and adds the call SID to activeStreamCalls, and a later inbound webhook queues while that stale SID remains. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes, a provider-owned pending-stream timeout canceled by registerCallStream and terminal cleanup paths is the narrow maintainable fix. Making queue behavior configurable is a separate product choice and should not be required for this bug.
