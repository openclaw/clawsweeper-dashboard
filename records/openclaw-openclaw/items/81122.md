## Summary

Keep open: current main still marks an inbound Twilio call active as soon as stream TwiML is returned, but there is no pending-stream timeout for the case where the media WebSocket never starts; the later inbound call can still be sent to the hold queue.

## What This Changes

The issue requests clearing stale Twilio inbound active-stream state when a stream response is issued but no media stream connects.

## Best Possible Solution

Add a Twilio-provider pending-stream TTL for inbound stream TwiML, cancel it on successful stream registration and existing terminal cleanup paths, and cover the no-start case with focused regression tests.

## Reproduction Assessment

Yes, at source level: one inbound webhook can add a call SID to activeStreamCalls, and without register/unregister/status cleanup a later inbound webhook follows the queue path. I did not run a live Twilio repro in this read-only review.

## Solution Assessment

Yes. A provider-owned pending-stream timeout canceled by registerCallStream and existing cleanup paths is the narrowest maintainable fix; making queueing configurable would be a separate product choice.
