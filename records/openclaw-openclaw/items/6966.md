## Summary

Keep open. Current main has ordered model fallback, model-scoped auth-profile cooldowns, cooldown probes, and fallback observability, but it still does not implement the requested rate-limit-driven model strategy, persisted model health, automatic downgrade selection, or upgrade-back policy; the linked aashari fork remains useful prior art rather than upstream behavior.

## What This Changes

Add automatic model downgrade and later upgrade-back behavior when API rate-limit signals indicate the current model tier is overloaded.

## Best Possible Solution

Define a first-class core model-routing contract for health-aware failover or downgrade policies, with persisted recovery state, observable downgrade/recovery events, config schema/docs updates, and regression coverage for rate-limit scenarios.

## Reproduction Assessment

Not applicable. This is a feature/design request; current-main source and docs inspection gives high-confidence implementation-gap evidence rather than a failing runtime reproduction path.

## Solution Assessment

Unclear until maintainers choose the policy contract. The capability likely belongs in core model routing because core owns fallback candidate selection, auth cooldown state, session override persistence, and observability, but the issue's Phase 1 approach is only one possible design.
