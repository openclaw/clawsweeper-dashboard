## Summary

Keep open: current main and v2026.5.12 have ordered error-driven fallback, auth-profile rotation, model-scoped cooldowns, and cooldown probes, but they still lack the requested dynamic routing policy with strategy/health configuration, durable model-health or round-robin selection, and explicit downgrade/failback semantics. The aashari fork and open upstream PRs are useful prior art or partial work, not a shipped resolution.

## What This Changes

Request automatic model downgrade and recovery based on API rate limits instead of only static default/fallback configuration.

## Best Possible Solution

Define a core model-routing contract that preserves ordered failover by default while adding explicit provider/model health state, downgrade and failback semantics, observability, config schema, docs, and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature/design request; source inspection gives high confidence that current main lacks a strategy, durable health state, downgrade selector, and explicit failback policy.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product/config contract; the maintainable path is an explicit routing policy rather than hidden behavior in the existing ordered fallback loop or fragmented session-scoped fixes.
