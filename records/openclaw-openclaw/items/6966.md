## Summary

Keep open. Current main has ordered model fallback, rate-limit profile rotation, model-scoped cooldown metadata, and cooldown probes, but it still lacks the requested dynamic health-aware downgrade and upgrade-back routing policy; the aashari fork in the discussion is useful prior art, not shipped upstream behavior.

## What This Changes

Requests automatic downgrade to cheaper or faster fallback models when API rate-limit errors spike, plus gradual upgrade-back after recovery.

## Best Possible Solution

Define and document a first-class core model-routing policy that can choose between ordered failover and health-aware downgrade/round-robin behavior, with recovery semantics and regression tests for rate-limit scenarios.

## Reproduction Assessment

Not applicable. This is a feature/design request, and source/docs inspection on current main gives a high-confidence comparison rather than a failing bug reproduction.

## Solution Assessment

No, current ordered fallback is not the full solution. The narrow maintainable path is an explicit config/schema/docs/tested routing policy rather than another hidden cooldown tweak.
