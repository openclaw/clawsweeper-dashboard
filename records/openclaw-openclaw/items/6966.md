## Summary

Keep open. Current main has ordered model fallback, rate-limit classification, model-scoped auth-profile cooldowns, cooldown probes, and fallback observability, but it still does not implement the requested rate-limit-driven model strategy, persisted model-health selector, automatic downgrade, or upgrade-back policy; the linked aashari fork remains useful prior art rather than upstream behavior.

## What This Changes

The issue requests automatic model downgrade and later recovery when API rate-limit errors spike, instead of relying only on a static fallback chain and auth-profile rotation.

## Best Possible Solution

Define a first-class core model-routing contract for health-aware failover or downgrade policies, backed by persisted health/recovery state, observable downgrade and recovery events, config schema/docs updates, and regression coverage for rate-limit scenarios.

## Reproduction Assessment

Not applicable. This is a feature/design request, and current-main source plus docs inspection gives high-confidence implementation-gap evidence rather than a failing runtime reproduction path.

## Solution Assessment

Unclear until maintainers choose the product contract. The capability likely belongs in core model routing because core owns fallback candidate selection, auth cooldown state, session override behavior, and observability, but the issue's Phase 1 approach is only one possible design.
