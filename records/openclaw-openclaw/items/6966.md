## Summary

Keep open. Current main has strong adjacent ordered fallback, auth-profile rotation, model-scoped cooldowns, and cooldown probing, but it still lacks the requested dynamic rate-limit-driven model routing policy: no model strategy config, no persistent per-model health selector, no round-robin/health-aware routing, and no automatic downgrade plus upgrade-back behavior. The aashari fork linked in the discussion is useful prior art, not shipped upstream behavior.

## What This Changes

Request automatic rate-limit-driven switching from an expensive primary model tier to cheaper/faster fallbacks, then gradual recovery back to the primary.

## Best Possible Solution

Keep this as a core model-routing design issue and define a documented health-aware policy with config/schema, provider/model/profile state, observable downgrade and recovery events, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection gives high-confidence proof that current main has reactive ordered fallback but not the requested dynamic downgrade and upgrade-back policy.

## Solution Assessment

Unclear until maintainers choose the product/config contract. The narrowest maintainable direction is a core model-routing policy layered before candidate execution, while preserving the existing reactive fallback and cooldown behavior.
