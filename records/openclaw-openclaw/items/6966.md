## Summary

Keep open: current main has adjacent ordered fallback, model-scoped cooldowns, and sticky auto-fallback behavior, but it still lacks the requested dynamic rate-limit-driven model-routing policy with strategy, model health state, downgrade selection, and upgrade-back semantics.

## What This Changes

Request automatic downgrade of the default model tier when API rate-limit errors spike and gradual upgrade back when rate limits subside.

## Best Possible Solution

Define a core model-routing policy that layers health state and rate-limit recovery onto the existing ordered fallback path, with documented config, observable downgrade/recovery events, and tests for rate-limit downgrade plus safe upgrade-back behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement/design request. Source inspection gives high confidence that current main lacks the requested strategy, durable health state, automatic downgrade selector, and upgrade-back policy while supporting adjacent ordered fallback.

## Solution Assessment

Unclear until maintainers choose the product contract. The maintainable direction is a core model-routing policy with explicit health and recovery state, not an ad hoc cache layer or hidden change to the existing fallback loop.
