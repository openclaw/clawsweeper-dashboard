## Summary

Keep open. Current main has ordered model fallback, model-scoped cooldown metadata, and a rate-limit profile-rotation cap, but it still lacks the requested dynamic rate-limit-driven routing policy with explicit strategy, health state, downgrade thresholds, and upgrade-back behavior; the aashari fork in the discussion is useful prior art, not shipped upstream behavior.

## What This Changes

Request automatic downgrading and later upgrading of configured default model tiers when API rate-limit errors spike.

## Best Possible Solution

Define an upstream model-routing policy that distinguishes ordered failover from health-aware downgrade or round-robin routing, with compatible config schema, persisted health/recovery state, observable downgrade/upgrade events, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/design request. Source inspection gives a high-confidence verification path because the schema, docs, fallback runner, and tests show ordered fallback primitives but no dynamic strategy or health-routing contract.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The current cooldown and fallback implementation is a reasonable foundation, but the requested behavior needs an explicit compatible routing policy rather than a one-off selector change.
