## Summary

Keep open. Current main has model-scoped cooldowns, ordered model fallback, and a rate-limit profile-rotation cap, but it still lacks the requested dynamic model-routing policy with strategy/health state, downgrade semantics, thresholds, and upgrade-back behavior; the aashari fork in the discussion is useful external prior art, not shipped upstream behavior.

## What This Changes

The issue requests automatic model downgrade and later upgrade-back behavior when API rate limits are detected, rather than only static ordered fallback.

## Best Possible Solution

Define an upstream model-routing policy that distinguishes ordered failover from health-aware downgrade or round-robin routing, with compatible config schema, persisted health/recovery state, observable downgrade/upgrade events, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/design request. Current-main source inspection gives a high-confidence verification path because the schema, docs, fallback runner, and tests show ordered fallback primitives but no dynamic strategy/health routing contract.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The current cooldown and fallback implementation is a reasonable foundation, but the requested behavior needs an explicit compatible routing policy rather than a one-off selector change.
