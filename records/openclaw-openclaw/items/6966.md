## Summary

Keep open: current main has strong adjacent rate-limit fallback, model-scoped cooldown, and sticky auto-fallback behavior, but it still lacks the requested dynamic model-routing policy with strategy/health state, downgrade selection, and upgrade-back semantics.

## What This Changes

Request automatic model downgrade and recovery routing based on API rate-limit signals instead of only static primary/fallback configuration.

## Best Possible Solution

Define a core model-routing policy that layers health state and rate-limit recovery onto the existing ordered fallback path, with documented config, observable downgrade/recovery events, and tests for rate-limit downgrade plus safe upgrade-back behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement/design request. Source inspection gives high confidence that current main lacks the requested strategy, health state, and automatic upgrade-back policy while supporting adjacent ordered fallback.

## Solution Assessment

Unclear until maintainers choose the product contract. The maintainable direction is a core model-routing policy with explicit health/recovery state, not an ad hoc cache layer or hidden change to the existing fallback loop.
