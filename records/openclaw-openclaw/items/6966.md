## Summary

Keep open: current main has ordered reactive fallback, model-scoped auth-profile cooldowns, cooldown probing, and sticky auto-fallback overrides, but it still lacks the requested dynamic rate-limit-driven model routing policy with strategy selection, durable model health, downgrade choice, and upgrade-back semantics. The linked aashari fork is useful prior art, not shipped upstream behavior.

## What This Changes

The issue requests automatic downgrade and later upgrade-back of configured LLM models when API rate-limit pressure changes.

## Best Possible Solution

Define a core model-routing contract that layers provider/model health and recovery state onto the existing fallback path, documents config and events, preserves current failover behavior by default, and adds regression coverage for downgrade plus safe upgrade-back scenarios.

## Reproduction Assessment

Not applicable as a bug reproduction because this is an enhancement/design request. Source inspection gives high confidence that current main lacks the requested strategy, durable health state, automatic downgrade selector, and upgrade-back policy while supporting adjacent ordered fallback behavior.

## Solution Assessment

Unclear until maintainers choose the product contract. The maintainable direction is a core model-routing policy with explicit health, recovery, persistence, and observability semantics rather than an ad hoc cache or hidden change to the existing fallback loop.
