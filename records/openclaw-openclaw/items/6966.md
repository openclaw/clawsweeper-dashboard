## Summary

Keep open. Current main has ordered rate-limit fallback, cooldown probing, and model-scoped auth-profile cooldowns, but it still lacks the requested dynamic health-aware model downgrade and upgrade-back policy; the aashari fork in the discussion is useful prior art, not shipped upstream behavior.

## What This Changes

The issue requests automatic downgrade and later recovery of the default model tier when API rate-limit errors spike.

## Best Possible Solution

Define and document a core model-routing policy that supports explicit failover versus health-aware downgrade/round-robin behavior, with recovery semantics and tests for rate-limit scenarios.

## Reproduction Assessment

Not applicable. This is a feature/design request, and source inspection gives a high-confidence current-main comparison rather than a failing bug reproduction.

## Solution Assessment

No, current ordered fallback is not the full solution. The narrow maintainable path is a designed model-routing policy and config surface rather than another hidden cooldown tweak.
