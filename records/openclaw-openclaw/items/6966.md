## Summary

Keep open. Current main has ordered model fallback, rate-limit/auth-profile cooldowns, and model-scoped cooldown metadata, but it still lacks the requested dynamic rate-limit-driven model routing policy with downgrade, model health, and upgrade-back behavior; the aashari fork linked in discussion is useful prior art, not shipped upstream behavior.

## What This Changes

The issue requests automatic model tier switching when API rate limits spike, with downgrade on failures and gradual recovery to the preferred model later.

## Best Possible Solution

Define an upstream model-routing policy that distinguishes ordered failover from health-aware downgrade routing, persists the right health state, records recovery, emits observable downgrade/upgrade events, and updates schema, docs, and regression tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/design request. Current-main source inspection gives a high-confidence verification path because the schema, docs, fallback runner, and tests show the requested dynamic routing policy is absent.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The current cooldown and fallback implementation is a reasonable foundation, but the requested behavior needs an explicit compatible routing policy rather than a one-off selector change.
