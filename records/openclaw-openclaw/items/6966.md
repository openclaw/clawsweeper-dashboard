## Summary

Keep open. Current main has useful adjacent rate-limit handling, ordered fallback, cooldown probes, and model-scoped auth-profile cooldowns, but it still lacks the requested rate-limit-driven model routing policy with model strategy config, persisted model health, downgrade selection, and upgrade-back behavior; the linked aashari fork is useful prior art, not shipped OpenClaw behavior.

## What This Changes

The issue requests automatic model downgrading on rate-limit pressure and gradual recovery back to higher-tier models.

## Best Possible Solution

Define a first-class core model-routing contract for health-aware failover/downgrade policies, backed by persisted recovery state, observable downgrade/recovery events, config schema/docs updates, and regression tests for rate-limit scenarios.

## Reproduction Assessment

Not applicable. This is a feature/design request; current-main source and docs inspection gives high-confidence implementation-gap evidence rather than a failing runtime reproduction path.

## Solution Assessment

Unclear until maintainers choose the policy contract. The capability likely belongs in core model routing because core owns candidate selection, fallback persistence, cooldown recovery, and status/observability, but the issue’s specific Phase 1 approach is only one possible design.
