## Summary

Keep open: current main and the latest release have strong adjacent ordered fallback, auth-profile rotation, model-scoped cooldown metadata, cooldown probing, and auto fallback overrides, but they still do not implement dynamic rate-limit-driven model routing with strategy selection, durable model health, downgrade choice, and upgrade-back semantics. The aashari fork and open model-exhaustion PR are useful prior art or partial work, not a shipped upstream resolution.

## What This Changes

Add a policy to automatically downgrade to cheaper or faster fallback models when API rate limits spike, then upgrade back when errors subside.

## Best Possible Solution

Define a core model-routing contract that preserves ordered failover by default while adding explicit provider/model health state, downgrade and recovery semantics, observability, config schema, docs, and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is an enhancement/design request; source inspection gives high confidence that current main lacks the requested strategy, durable health state, downgrade selector, and upgrade-back policy.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product contract; the requested direction is plausible, but the maintainable solution should be an explicit routing policy rather than hidden behavior in the existing ordered fallback loop or only the partial session-scoped model-exhaustion work in https://github.com/openclaw/openclaw/pull/81054.
