## Summary

Keep open. Current main has ordered fallback, rate-limit classification, model-scoped profile cooldowns, cooldown probes, and fallback observability, but it still lacks the requested rate-limit-driven model routing strategy, persisted model-health selector, automatic downgrade policy, and explicit upgrade-back semantics.

## What This Changes

The issue requests automatic model downgrade and later recovery when API rate-limit errors spike, instead of relying only on static default model configuration and ordered fallbacks.

## Best Possible Solution

Define a first-class health-aware model-routing contract, then implement persisted model/provider health state, downgrade and recovery semantics, observability, config schema/docs updates, and regression coverage for rate-limit scenarios.

## Reproduction Assessment

Not applicable. This is a feature/design request, and current-main source plus docs inspection gives high-confidence implementation-gap evidence rather than a failing runtime reproduction path.

## Solution Assessment

Unclear until maintainers choose the product contract. The capability likely belongs in core model routing because core owns fallback candidate selection and cooldown handling, but the proposed Phase 1 is only one possible policy design.
