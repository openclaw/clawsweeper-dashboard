## Summary

Keep open. Current main has adjacent rate-limit handling, auth-profile rotation, model-scoped cooldowns, cooldown probing, and ordered fallback, but it still lacks the requested dynamic rate-limit-driven model routing policy with model health, downgrade, and upgrade-back behavior.

## What This Changes

The issue requests automatic model downgrade and later recovery to stronger models when API rate-limit errors spike or subside.

## Best Possible Solution

Define and implement an upstream model-routing policy that can distinguish ordered failover from health-aware or downgrade routing, persists the right provider/model/profile health state, records success recovery, exposes observable downgrade and upgrade-back events, and updates config schema, docs, and regression tests around rate-limit scenarios.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/design request. Current-main inspection gives a high-confidence verification path: the schema, docs, fallback runner, and tests show the requested dynamic routing policy is absent.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The current cooldown and fallback implementation is the right foundation, but not the complete solution; a maintainable fix should be explicit, configurable, and compatible with existing fallback and session override rules.
