## Summary

Keep open. Current main has ordered fallback, rate-limit profile rotation, model-scoped cooldowns, and cooldown probes, but it still lacks the requested dynamic health-aware downgrade and upgrade-back model-routing policy; the aashari fork in the discussion is useful prior art, not shipped upstream behavior.

## What This Changes

The issue requests dynamic model routing that downgrades or rotates models when API rate limits are detected, then recovers when limits subside.

## Best Possible Solution

Define a first-class core model-routing policy with explicit config/schema/docs, persisted health or cooldown state, downgrade and recovery semantics, observable events, and regression coverage for rate-limit scenarios.

Not applicable. This is a feature/design request, and source/docs inspection on current main gives a high-confidence comparison rather than a failing bug reproduction.

## Solution Assessment

Unclear. Rate-limit-aware model routing likely belongs in the core fallback path, but maintainers still need to choose the routing policy and config/state shape before implementation.
