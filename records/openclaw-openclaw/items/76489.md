## Summary

Current main and the latest shipped tag already route manual cron runs through the job's configured agent identity, including agent-scoped cron session keys, so the reported wrong-agent manual-run behavior is no longer necessary to track as open.

## What This Changes

The issue reports that manual cron triggers via CLI or Dashboard ignored a cron job's configured `agentId` and ran under another agent.

## Best Possible Solution

Keep the current single cron execution path where manual and scheduled runs both resolve the job's configured agent before building run/session identity.

## Reproduction Assessment

No current-main failure was reproduced live; the original report provides a concrete old-version path, and source inspection now shows manual cron runs resolve and pass `job.agentId` into the run/session path.

## Solution Assessment

Yes. The best fix shape is the current shared cron path that resolves agent identity once and uses it for isolated execution and session-key construction, rather than adding a separate manual-run workaround.
