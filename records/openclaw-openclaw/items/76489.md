## Summary

Current main already routes manual cron runs through the same stored job execution path and resolves isolated agent turns from the job's configured `agentId`, with regression coverage for a forced manual run using a non-default agent.

## What This Changes

The issue requests that CLI and Dashboard manual cron runs honor the cron job's configured `agentId` instead of routing to the wrong agent.

## Best Possible Solution

Keep the current gateway cron routing behavior and direct affected 2026.4.29 installs to upgrade to `v2026.5.2` or newer.

## Reproduction Assessment

No current-main failure reproduction: source inspection shows manual runs execute the stored job and the isolated runner receives `job.agentId`. The current regression test covers a forced manual run with an explicit non-default agent.

## Solution Assessment

Yes. Resolving the agent at the gateway cron runner boundary from `job.agentId` is the narrowest maintainable fix because both CLI and Dashboard manual triggers share `cron.run`/`enqueueRun`.
