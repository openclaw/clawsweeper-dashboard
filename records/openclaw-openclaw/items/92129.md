## Label Justifications

- `P1`: The issue describes a real Telegram isolated-ingress workflow silently losing inbound messages during failed turns or gateway recreation.
- `impact:message-loss`: The affected spool path can accept a Telegram update and then delete the durable copy without replaying or recording the message after processing failure.
- `impact:session-state`: The failure path can leave no session transcript entry for a user message that Telegram already delivered to the gateway.

## Summary

Keep open. Current main still deletes a claimed Telegram isolated-ingress spool entry when `bot.handleUpdate()` resolves, while OpenClaw's Telegram message paths can log a dispatch failure and return normally, violating the documented retryable-handler contract and leaving a credible silent message-loss path.

## What This Changes

The issue reports that Telegram isolated polling can clean-delete a spooled update after a failed turn because message-handler errors are swallowed before the spool consumer sees a rejection.

## Best Possible Solution

Keep this as a Telegram plugin bugfix: preserve the spooled update until a real per-update completion signal says dispatch succeeded, otherwise release or dead-letter it with focused regression coverage.

## Reproduction Assessment

Yes, at source level: a spooled update is deleted after `bot.handleUpdate()` resolves, while current Telegram message-processing paths log and swallow dispatch failures. I did not run live Docker/Telegram proof in this read-only review.

## Solution Assessment

Yes, the best direction is to make the Telegram spool consumer depend on a real per-update completion/failure signal or a scoped spooled-replay error path. Relying only on the `handleUpdate()` promise is too weak because OpenClaw handlers can intentionally resolve after failed dispatch.
