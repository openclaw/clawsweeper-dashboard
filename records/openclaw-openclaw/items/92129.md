## Label Justifications

- `P1`: The issue describes a credible user-facing Telegram workflow where an inbound message can be accepted, lose its durable spool copy, and never reach a session.
- `impact:message-loss`: The affected isolated-ingress path can delete the only durable copy of a Telegram update after a dispatch failure that returned normally.
- `impact:session-state`: The failure mode can leave no transcript/session record for a Telegram message that the gateway already accepted.

## Summary

Keep open. Current main and the latest release still have a source-reproducible Telegram isolated-ingress path where dispatch failure can resolve normally and let the spooled update be deleted; the linked fix PR is open but unmerged, so this issue should remain open behind that review path.

## What This Changes

Fix Telegram isolated polling ingress so failed spooled-update processing releases or dead-letters the update instead of clean-deleting it after a swallowed dispatch failure.

## Best Possible Solution

Review and land a narrow Telegram-plugin fix through https://github.com/openclaw/openclaw/pull/92151, or replace it with an equally focused PR if that branch cannot be made narrow and mergeable.

## Reproduction Assessment

Yes, at source level: current main deletes the spooled update after handleUpdate resolves, and the dispatch-failure path catches and returns normally. I did not run a live Docker/Telegram repro in this read-only review.

## Solution Assessment

Yes, the best fix direction is to make failed Telegram dispatch propagate or otherwise mark the spooled replay as failed before spool cleanup. The open linked PR is the current implementation candidate, but this issue should not close until that PR lands or is replaced.
