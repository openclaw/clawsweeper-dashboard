## Label Justifications

- `P1`: Default isolated Telegram ingress can silently discard an accepted user message when its turn fails.
- `impact:message-loss`: The durable inbound update can be deleted despite unsuccessful processing, leaving nothing to replay.
- `impact:session-state`: The lost turn can leave no session transcript or completion artifact.

## Summary

The issue remains necessary: current main still deletes a claimed Telegram spool entry whenever `handleUpdate()` resolves, while the message processor and bot error handler can consume turn failures. Two open PRs claim the issue with materially different fixes, so the issue should remain the canonical tracker until maintainers select and merge the correct approach.

## What This Changes

Ensure failed Telegram isolated-ingress turns release or dead-letter their durable spool entries instead of deleting them as completed.

## Best Possible Solution

Use a single explicit per-update terminal outcome shared by Telegram processing, update tracking, and isolated spool cleanup, deleting only confirmed-success entries and applying bounded release or durable dead-letter handling for failures.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction chain: dispatch failure is consumed, `handleUpdate()` can resolve, and the isolated spool consumer then deletes the claimed update; this read-only review did not run a live Telegram reproduction.

## Solution Assessment

No proposed PR is established as the best fix yet. The maintainable direction is an explicit closed processing outcome rather than relying only on exception propagation through middleware that intentionally handles errors.
