## Summary

Keep open. Current main and the latest release still have the source-level Telegram isolated-ingress message-loss path, and an open linked fix PR now owns the implementation path.

## What This Changes

The issue reports that Telegram isolated polling ingress can delete a claimed spooled update after a failed turn because Telegram message-processing errors are caught before the spool consumer can retry or dead-letter the update.

## Best Possible Solution

Review and land a focused Telegram fix through https://github.com/openclaw/openclaw/pull/92151 or replace it with an equally narrow PR, then close this issue after the merged code proves retry or dead-letter behavior for failed spooled turns.

## Reproduction Assessment

Yes, at source level: the spooled update is deleted after `bot.handleUpdate()` resolves, while the Telegram message processor catches dispatch failures and returns `true`. I did not run live Docker/Telegram proof in this read-only review.

## Solution Assessment

Yes, the maintainable fix is a narrow Telegram plugin repair that preserves the spooled update until a real per-update success signal exists, otherwise releasing or dead-lettering it. The active PR should be reviewed against that boundary.
