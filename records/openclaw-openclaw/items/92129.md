## Label Justifications

- `P1`: The issue describes a credible Telegram channel workflow where an accepted inbound message can be silently lost during a failed turn.
- `impact:message-loss`: The affected spool path can delete the durable Telegram update without retrying or recording the user message after dispatch failure.
- `impact:session-state`: The failure can leave no session transcript entry for a Telegram message that was already accepted into isolated ingress.

## Summary

Keep open. Current main and v2026.6.5 still show the source-level dispatch-failure path that can resolve successfully and let the isolated-ingress spool delete the claimed update; the open fixing PR must merge before this issue should close.

## What This Changes

The issue asks Telegram isolated polling ingress to preserve, retry, or dead-letter spooled updates when message dispatch fails instead of deleting them after a swallowed error.

## Best Possible Solution

Review and land the open Telegram fix PR or an equivalent focused repair so failed dispatch leaves the spooled update retryable or dead-lettered instead of clean-deleted.

## Reproduction Assessment

Yes, at source level: dispatch failures currently resolve through `processMessage`, and the spool consumer deletes after a resolved `handleUpdate`. I did not run a live Docker/Telegram reproduction in this read-only sweep.

## Solution Assessment

Yes, the best direction is a focused Telegram plugin fix that turns failed dispatch into a real per-update failure signal for the spool consumer. The open fixing PR is the current implementation candidate, but the issue should stay open until a reviewed fix lands.
