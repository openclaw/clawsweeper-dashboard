## Label Justifications

- `P1`: The issue describes silent loss of real Telegram inbound messages during a normal gateway/channel workflow.
- `impact:message-loss`: A failed Telegram turn can make the durable spooled update disappear without retry, replay, or user-visible delivery.
- `impact:session-state`: The reported failure leaves no session transcript entry or durable turn artifact for the lost inbound message.

## Summary

Keep open. Current `main` and the latest release still have the actionable failure mode where Telegram isolated ingress deletes a claimed spool entry after `handleUpdate()` resolves, while normal message dispatch failures are caught and reported as a successful message-processing return; the related timeout/replay fixes do not cover this swallowed dispatch-error path.

## What This Changes

Stop Telegram isolated ingress from cleanly deleting a spooled update when downstream turn processing failed; preserve it for retry or dead-letter instead.

## Best Possible Solution

Make isolated ingress completion depend on an explicit per-update processing outcome and preserve failed updates as retryable claims or `.failed` tombstones without weakening Telegram replay dedupe.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: make `dispatchTelegramMessage` reject under isolated ingress, and `processMessage` resolves `true`, letting `#handleClaimedSpooledUpdate` delete the claim. I did not run the live Telegram/Docker repro in this read-only review.

## Solution Assessment

Unclear. The direction of not cleanly deleting failed spooled updates is correct, but maintainers should choose the durable failure policy and account for dispatch replay dedupe before automation changes the semantics.
