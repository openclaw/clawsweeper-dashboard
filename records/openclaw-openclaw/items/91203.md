## Summary

Keep open: the narrowed PR addresses a real misleading progress-label path, but the current patch also turns no-exit `command-output` failures into `finished`, and live GitHub now reports the branch as conflicting with current main.

## What This Changes

The PR changes shared compact progress formatting and Matrix monitor tests so ambiguous command-progress failure statuses render neutral lifecycle text instead of visible failure labels.

## Best Possible Solution

Limit neutral wording to ambiguous item/tool command progress, preserve concrete command-output failure status when no exit code is available, then rebase and get maintainer acceptance for the shared wording change.

## Reproduction Assessment

Yes. Source inspection shows current main can pass `status: "failed"` through the shared formatter for command progress without exit-code evidence; I did not run tests because this review is read-only.

## Solution Assessment

No as-is. The shared formatter is the right owner for the Matrix-visible symptom, but command-output failures without exit codes should not be neutralized as finished.
