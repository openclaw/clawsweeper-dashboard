## Summary

Keep this PR open, but the current patch is a bit too broad: it correctly targets ambiguous command-progress labels, yet it also normalizes failed command-output events without numeric exit codes to `finished`, which can hide real terminal failures.

## What This Changes

The PR changes shared channel progress formatting so ambiguous command progress statuses render neutral lifecycle text, and adds shared formatter plus Matrix monitor regressions.

## Best Possible Solution

Keep the neutral wording for ambiguous command item/tool progress, but preserve real failed command-output status when no numeric exit code exists, then rebase and land after maintainer acceptance of the shared wording change.

## Reproduction Assessment

Yes. Current main can pass a command item with status `failed` into the shared compact-progress renderer, where the status can render without exit-code evidence.

## Solution Assessment

No. The shared formatter is the right layer for ambiguous command-progress wording, but the current patch also changes terminal command-output failure statuses with null exit codes to `finished`, which is too broad.
