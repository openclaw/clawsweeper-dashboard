## Summary

Current main already handles the reported memory index failure by opening node:sqlite with extension loading allowed and degrading sqlite-vec failures to a warning instead of touching a missing chunks_vec table.

## What This Changes

The issue asks for Windows memory indexing to stop failing with `no such table: chunks_vec` when sqlite-vec cannot load through the node:sqlite backend.

## Best Possible Solution

Keep the current degraded sqlite-vec path and ship it in the next release so packaged installs warn and continue indexing when vector acceleration is unavailable.

## Reproduction Assessment

No current-main reproducer remains: source inspection shows node:sqlite is opened with extension loading allowed, and sqlite-vec load failure now leaves vector unavailable instead of preparing statements against a missing chunks_vec table. The v2026.5.2 report is credible, but it no longer matches current main behavior.

## Solution Assessment

Yes; the current main behavior is the narrow maintainable fix because it preserves sqlite-vec acceleration when available and degrades to indexed chunks/FTS with an explicit warning when it is not.
