## What This Changes

The issue asks OpenClaw to fix Windows memory indexing failures where vector readiness and `chunks_vec` table state get out of sync after the 2026.5.2 memory backend changes.

## Best Possible Solution

Make memory-core treat vector table existence as part of readiness before preparing `chunks_vec` statements, and use the requested deep status logs to decide whether this belongs under #70559, #66746, or a separate 2026.5.2 regression.

## Reproduction Assessment

Unclear for the reporter's exact steps. Source inspection shows a credible `chunks_vec` prepare failure when unsafe reindex drops the table while `vector.available` remains true, but normal full reindex only enters that path with specific env vars.

## Solution Assessment

Unclear: switching backends or adding a backend config option is not supported by the current evidence because node:sqlite does support extension loading when opened with `{ allowExtension: true }`. The narrower maintainable direction is to harden vector-table readiness and then route by diagnostics.
