## Summary

Keep open: current main disproves the broad `node:sqlite` load-extension diagnosis, but the reported `chunks_vec` missing-table failure remains plausible from current source and overlaps unresolved Windows/vector-state reports.

## What This Changes

The issue requests that `openclaw memory index --force` stop failing with `no such table: chunks_vec` on Windows and report sqlite-vec/vector readiness accurately when vector indexing is unavailable.

## Best Possible Solution

Fix the memory-core vector-state/table-existence invariant so `vector.available` cannot imply `chunks_vec` exists unless the table has actually been created, while keeping Windows loader fixes tracked separately.

## Reproduction Assessment

No, not for the full Windows CLI report. Source inspection does show a current-main path where `chunks_vec` can be prepared after being dropped while vector state still says available, especially in the unsafe reindex path from #70559.

## Solution Assessment

No, a wholesale switch back to `better-sqlite3` or a backend selector is not the best first fix because current main already enables `node:sqlite` extension loading. The narrower maintainable fix is to make vector readiness depend on the actual `chunks_vec` table lifecycle and leave Windows loader suffix handling to the related loader work.
