## Summary

Close as implemented on current main: package entry resolution now honors `runtimeExtensions` and `runtimeSetupEntry` before TypeScript source fallback, with regression coverage and docs showing the source fields are intentional rather than stale. The proof commit is not in a tagged release I could identify, so this is fixed on main only.

## What This Changes

The issue asks OpenClaw to stop Feishu package doctor warnings when `@openclaw/feishu@2026.5.3` includes source `extensions`/`setupEntry` fields alongside built runtime entries.

## Best Possible Solution

Keep the documented source-plus-runtime entry contract, ship the current runtime-entry resolution and Feishu 2026.5.4 package path, and avoid stripping source fields unless maintainers intentionally change the plugin entrypoint contract.

## Reproduction Assessment

No for a current-main failure: I did not run a live 2026.5.3 doctor path, and source inspection shows current main would prefer the declared runtime entries instead of warning on the TypeScript source fields. The issue body still gives a credible release-artifact reproduction for OpenClaw 2026.5.3.

## Solution Assessment

Yes: honoring `runtimeExtensions` and `runtimeSetupEntry` while preserving source fields is the narrowest maintainable fix because it matches the current docs and keeps workspace/git plugin development working.
