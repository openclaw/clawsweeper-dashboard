## Summary

Keep open: current main still checks for the bare "401" substring before transient Telegram sendChatAction classification, and an open candidate fix already targets this issue at https://github.com/openclaw/openclaw/pull/94791.

## What This Changes

The issue asks OpenClaw to stop treating transient Telegram sendChatAction 429/5xx/network errors that contain the text "401" as real 401 Unauthorized failures.

## Best Possible Solution

Land a narrow Telegram handler fix that prefers structured error_code === 401, keeps a safe Unauthorized fallback for non-structured errors, and adds focused regression coverage for retry_after=401.

## Reproduction Assessment

Yes, source-reproducible: current main checks message.includes("401") before transient classification, and grammY formats Telegram errors with both error_code and description in the message. I did not execute the repro test in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to classify real Telegram 401s from structured error_code and avoid numeric substring matching before the transient branch. The open PR at https://github.com/openclaw/openclaw/pull/94791 appears to target that boundary, but it still needs PR review before merge.
