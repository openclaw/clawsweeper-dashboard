## Summary

Keep open: current main still checks a broad `401`/`unauthorized` message match before the structured transient-error branch, so a grammY-shaped 429 or 5xx message containing `401` can still take the invalid-token suspension path. No open or merged related item appears to own this narrower false-401 bug yet.

## What This Changes

Fix Telegram `sendChatAction` error classification so structured transient errors whose message contains `401` do not trigger the invalid-token suspension path.

## Best Possible Solution

Narrow the Telegram handler so structured 429/5xx/recoverable errors are classified as transient before any broad string fallback, while preserving true 401 suspension semantics and adding focused regression coverage.

## Reproduction Assessment

Yes at source level: current main checks `is401Error` before the transient branch, and grammY formats structured Telegram errors with `error_code` plus the human description in the Error message. I did not run a live Telegram API reproduction in this read-only pass.

## Solution Assessment

Yes: a narrow Telegram-owned classifier/order fix in `sendchataction-401-backoff.ts` is the best path, using existing `network-errors.ts` helpers and tests rather than adding config or changing message delivery.
