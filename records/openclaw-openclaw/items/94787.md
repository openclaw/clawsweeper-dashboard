## Label Justifications

- `P2`: This is a narrow but real Telegram behavior bug with limited blast radius, causing false account-wide sendChatAction suspension and a misleading invalid-token alarm when triggered.
- `impact:other`: The impact is operator-visible false credential alarm noise and suppressed typing indicators, which is meaningful but does not fit the more specific owned impact labels.

## Summary

Keep open: current main still classifies any rendered sendChatAction error message containing `401` as an invalid-token 401 before the transient 429/5xx/network branch can run, and multiple open PRs with closing syntax now own the narrow repair. This should close only after maintainers choose and land one focused classifier fix.

## What This Changes

Fix Telegram sendChatAction 401 detection so structured transient errors whose rendered message contains `401` do not trigger account-wide 401 suspension or the invalid-token alarm.

## Best Possible Solution

Choose and land one focused structured-error-code classifier fix with regression coverage, then close this issue and decline the overlapping PRs that were not selected.

## Reproduction Assessment

Yes, at source level. Current main evaluates a bare `401` message substring before the transient branch, so a structured 429 grammY error with `retry_after: 401` follows the invalid-token suspension path; I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to trust structured Telegram `error_code` for true 401s and keep only a careful unstructured unauthorized fallback, while preserving the existing real invalid-token guard.
