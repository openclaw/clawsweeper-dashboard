## Label Justifications

- `P2`: The issue is a narrow but real Telegram behavior bug with a misleading operator alarm and account-wide sendChatAction suspension when a transient error message contains `401`.
- `impact:other`: The impact is an operator-visible false invalid-token alarm and suppressed typing actions, which is meaningful but does not fit the more specific owned impact labels.

## Summary

Keep open: current main still appears to route any error message containing `401` into the Telegram sendChatAction invalid-token suspension path before transient handling, and an open linked fix PR now owns the narrow repair, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that Telegram sendChatAction 429/5xx transient errors whose rendered message contains `401` are misclassified as 401 authentication failures and can suspend chat actions with a false token-deletion alarm.

## Best Possible Solution

Review and land the linked fix PR, or replace it with the same narrow structured-error-code classifier and regression coverage if maintainers choose a different implementation.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a structured 429 error with `retry_after: 401` reaches the broad `is401Error` check before transient handling. I did not run the repro because this review is read-only, but the current code path matches the issue's minimal Vitest case.

## Solution Assessment

Yes, the best fix is the narrow one already proposed by the linked PR: classify true 401s by structured Telegram `error_code` when present and preserve only a careful unauthorized fallback for unstructured errors. That keeps the prior real-token-protection behavior without adding config or changing Telegram product policy.
