## Label Justifications

- `P2`: This is a narrow Telegram behavior bug with limited blast radius, but it can cause account-wide sendChatAction suspension and a misleading invalid-token alarm when triggered.
- `impact:other`: The issue has a concrete operator-visible impact through false token-deletion alarms and suppressed typing indicators, but it is not message loss, crash-loop, session-state, or auth-provider routing.

## Summary

Keep open: current main still classifies any sendChatAction error message containing `401` as an invalid-token failure before transient 429/5xx/network handling can run, and several open PRs already target the narrow repair.

## What This Changes

The issue reports that Telegram `sendChatAction` misclassifies transient errors whose rendered message contains `401` as true 401 invalid-token failures, suspending account-wide chat actions and logging a false token-deletion alarm.

## Best Possible Solution

Land one focused structured-error-code classifier fix, preferably the proof-sufficient ready candidate if maintainers choose it, then close this issue and resolve the overlapping PRs.

## Reproduction Assessment

Yes. Source inspection on current main shows a structured 429 whose rendered message contains `401` reaches `is401Error` before transient handling; I did not run the repro because this review is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to classify Telegram API failures by structured `error_code` when present and preserve only a careful unstructured unauthorized fallback, without adding config or changing Telegram policy.
