## Summary

Keep open: current `main` and the latest release still allow credential-bearing browser page-navigation URLs through the shared navigation guard, and the related cron webhook PR plus CDP SecretRef issue cover adjacent surfaces rather than this browser input path.

## What This Changes

Reject or redact browser tool page-navigation URLs that include embedded URL credentials before they reach Chromium, Playwright, or CDP target creation.

## Best Possible Solution

Add a narrow userinfo rejection with redacted `InvalidBrowserNavigationUrlError` text in the shared browser page-navigation guard, cover the route/Playwright/CDP callers, and leave remote CDP `cdpUrl` auth intact.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: send `/navigate` or `/tabs/open` a URL such as `https://user:pass@example.com`, and current `main` parses it as `http(s)` without checking `URL.username` or `URL.password` before forwarding the original string.

## Solution Assessment

Yes. The shared page-navigation guard is the narrowest maintainable fix point; changing remote CDP endpoint auth handling would solve the wrong surface and risk breaking documented `cdpUrl` authentication.
