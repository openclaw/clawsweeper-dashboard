## Summary

Keep this open: current main still allows credential-bearing browser navigation URLs through the shared navigation guard and forwards the original URL into Chrome MCP, Playwright, and CDP page-opening paths. The related cron webhook PR is a separate open surface and does not address this browser tool issue.

## What This Changes

Reject or redact browser tool navigation URLs that contain embedded credentials before they reach Chromium, Playwright, or CDP page-navigation calls.

## Best Possible Solution

Add a narrow page-navigation userinfo rejection in the browser plugin’s shared navigation guard with redacted, actionable error text and regression coverage across `/navigate`, `/tabs/open`, Playwright, and CDP target creation, while leaving remote CDP endpoint auth intact.

## Reproduction Assessment

Yes. Source inspection is high confidence: a URL such as `https://user:pass@example.com` parses as `https:` and the current shared guard never checks `URL.username` or `URL.password` before callers forward the original string onward.

## Solution Assessment

Yes. The narrowest maintainable solution is a shared browser page-navigation guard plus boundary tests; changing remote CDP URL auth would solve the wrong surface because that auth is explicitly documented and used separately.
