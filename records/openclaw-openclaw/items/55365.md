## Label Justifications

- `P2`: This is a normal-priority browser hardening issue with limited blast radius but real credential-exposure risk.
- `impact:security`: The reported path can expose embedded URL credentials through browser runtime errors or logs.

## Summary

Keep open: current `main` and `v2026.5.18` still do not reject URL userinfo in browser page-navigation inputs, and the open closing PR at https://github.com/openclaw/openclaw/pull/82490 should remain the implementation path before this issue closes.

## What This Changes

Browser navigation inputs with embedded URL credentials should be rejected early and surfaced with redacted diagnostics instead of being passed to browser runtimes.

## Best Possible Solution

Land a narrow browser guard/redaction fix through https://github.com/openclaw/openclaw/pull/82490 or a replacement PR, with tests and real browser/proxy proof that credentials are not forwarded or echoed.

## Reproduction Assessment

Yes. Source inspection shows HTTP(S) navigation URLs are parsed and passed through without checking `URL.username` or `URL.password`, while browser tool and route/runtime callers forward the original URL.

## Solution Assessment

Yes. The shared browser navigation guard is the right fix point, as long as the implementation preserves remote CDP `cdpUrl` credential support and only blocks credentialed page-navigation URLs.
