## Summary

Keep this issue open: current main and the latest release still lack a browser navigation userinfo guard, and the open linked fix at https://github.com/openclaw/openclaw/pull/82490 is the active path to review or land.

## What This Changes

The issue requests that browser open/navigate URLs with embedded credentials be rejected before browser dispatch and that credential-bearing diagnostics be redacted.

## Best Possible Solution

Review and land, or explicitly replace, https://github.com/openclaw/openclaw/pull/82490 so the shared browser navigation path rejects credentialed URLs and redacts diagnostics before browser dispatch.

## Reproduction Assessment

Yes from source inspection: the current shared browser navigation guard accepts parsed HTTP(S) URLs without checking `username` or `password`, so a credential-bearing URL can continue when the host passes policy. I did not run a live Chromium repro in this read-only review.

## Solution Assessment

Yes. The best fix is the linked PR's shape: reject userinfo at the model-facing browser URL input and shared navigation guard, while leaving separate CDP profile credential support and explicit browser credentials controls intact.
