## Summary

Keep this issue open: current main and the latest release still pass credential-bearing browser open/navigate URLs through, and the active fix is the open linked PR at https://github.com/openclaw/openclaw/pull/82490.

## What This Changes

Reject or redact browser tool open/navigate URLs that contain embedded username/password userinfo before they reach browser runtime or logs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82490, or a narrow equivalent shared browser-navigation guard, with redacted real browser/proxy proof and maintainer acceptance that URL userinfo is unsupported.

## Reproduction Assessment

Yes, at source level: current main reads browser open/navigate target URLs and forwards them before any username/password userinfo rejection. I did not run a live Chromium reproduction in this read-only review.

## Solution Assessment

Yes, the best code shape is a shared navigation guard plus browser-tool pre-dispatch check so local and proxy paths fail before runtime dispatch. The remaining question is maintainer acceptance of the fail-fast compatibility change and proof on a real browser/proxy path.
