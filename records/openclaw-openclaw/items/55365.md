## Summary

Keep open: current main and the latest release still lack a browser navigation userinfo guard, and https://github.com/openclaw/openclaw/pull/82490 is an open, unmerged fix candidate that explicitly targets this issue.

## What This Changes

The issue asks OpenClaw to reject browser navigation URLs containing embedded username/password credentials and redact those credentials from surfaced errors.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82490, or an equivalent browser-owned shared guard, so local, proxy, Playwright, and Chrome MCP navigation paths reject credentialed HTTP(S) URLs before dispatch and redact diagnostics.

## Reproduction Assessment

Yes. Source inspection shows current main accepts and forwards browser `open`/`navigate` URLs without checking URL `username` or `password`, and the current invalid-URL diagnostic uses the raw input.

## Solution Assessment

Yes for the direction, with maintainer review still required. The narrowest maintainable end state is a shared browser navigation guard plus model-facing tool preflight, using existing browser credentials/cookies/headers mechanisms instead of embedded URL credentials.
