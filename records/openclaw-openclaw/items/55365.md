## Summary

Keep this open: current main still accepts credential-bearing browser navigation URLs through the shared guard and forwards the original URL to Chrome MCP, Playwright, or CDP paths; the related cron webhook PR is a separate open surface.

## What This Changes

The issue asks the browser tool to reject page-navigation URLs containing embedded username/password credentials and redact any surfaced error text.

## Best Possible Solution

Add a narrow userinfo rejection in the browser page-navigation guard with redacted InvalidBrowserNavigationUrlError text, cover /navigate, /tabs/open, Playwright, and CDP page-opening paths, and leave remote CDP cdpUrl credential handling intact.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: send /navigate or /tabs/open a URL like https://user:pass@example.com, and current main parses it as http(s) without checking URL.username/password before passing the original string onward.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared page-navigation check in extensions/browser/src/browser/navigation-guard.ts plus boundary tests; changing remote CDP endpoint auth handling would solve the wrong surface.
