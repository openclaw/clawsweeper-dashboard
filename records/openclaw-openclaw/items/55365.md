## Label Justifications

- `P2`: This is a normal-priority browser hardening issue with limited blast radius but a real credential-exposure risk.
- `impact:security`: The reported path can expose embedded URL credentials through browser runtime errors or logs.

## Summary

Keep open: current `main` and latest shipped `v2026.5.18` still do not reject browser navigation URL userinfo, and the open closing PR at https://github.com/openclaw/openclaw/pull/82490 should remain the implementation path before this issue closes.

## What This Changes

Request browser open/navigate inputs to reject URLs with embedded credentials early and redact credential-bearing URL text before browser runtime errors or logs can expose it.

## Best Possible Solution

Land a narrow shared browser navigation guard and redaction fix through https://github.com/openclaw/openclaw/pull/82490 or a replacement PR, preserving remote CDP auth behavior while blocking credentialed page-navigation URLs.

## Reproduction Assessment

Yes, at source level. Current main parses and forwards HTTP(S) browser navigation URLs without checking `URL.username` or `URL.password`, though I did not run a live Chromium reproduction in this read-only review.

## Solution Assessment

Yes. The shared browser navigation guard is the right fix point, provided the implementation keeps remote CDP `cdpUrl` credential support separate from page-navigation URL rejection.
