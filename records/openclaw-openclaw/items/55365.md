## Summary

Keep this issue open: current main still has no browser-navigation guard that rejects credentialed HTTP(S) URLs before the browser runtime, and the open closing PR is the active implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests early rejection and redaction for browser tool navigation URLs that contain embedded username or password credentials.

## Best Possible Solution

Review and land a browser-owned fix that rejects credentialed HTTP(S) navigation URLs at every browser navigation entry point, redacts userinfo in surfaced diagnostics, and records the compatibility decision before closing the issue.

## Reproduction Assessment

Yes from source inspection: browser open/navigate accepts raw URL strings and forwards them to local or proxy browser navigation paths without checking URL username/password before runtime errors can include the original URL.

## Solution Assessment

Unclear until the linked PR is reviewed for compatibility and proof, but the correct shape is a browser-boundary guard plus redacted diagnostics rather than letting Chromium/Playwright reject credentialed URLs later.
