## Summary

Keep this open: current main still has the browser page-navigation gap. The shared browser navigation guard does not reject URL userinfo, and the route/runtime callers continue forwarding the original credential-bearing URL toward Chrome MCP, Playwright, or CDP navigation paths; the related cron webhook PR tracks a separate surface.

## What This Changes

The issue requests that browser navigation reject credential-bearing URLs before handing them to browser runtimes and redact any surfaced credential text.

## Best Possible Solution

Add a shared page-navigation userinfo guard with redacted error text and supported-auth guidance, covering /navigate, /tabs/open, Playwright, and CDP page-opening paths while preserving remote CDP endpoint auth.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: send /navigate or /tabs/open a public http(s) URL like https://user:pass@example.com, and current main parses it as a normal network URL without checking username/password before forwarding the original string.

## Solution Assessment

Yes. The narrow maintainable fix is one shared page-navigation check in extensions/browser/src/browser/navigation-guard.ts plus boundary coverage; changing remote CDP endpoint auth handling would solve the wrong surface.
