## Summary

Keep this open. Current main still parses browser navigation URLs without rejecting username/password userinfo, then forwards the original URL to Chrome MCP, Playwright, or CDP tab creation paths; the related cron webhook PR is separate, still open, and does not cover the browser tool surface.

## What This Changes

The issue requests that browser navigation reject URLs containing embedded username or password credentials, redact surfaced error text, and direct users to supported auth mechanisms.

## Best Possible Solution

Add a narrow browser-navigation guard in extensions/browser/src/browser/navigation-guard.ts after URL parsing: for http: and https: page-navigation URLs, reject parsed.username or parsed.password with a redacted InvalidBrowserNavigationUrlError that points to browser set credentials, cookies, or headers. Cover the guard, /navigate, /tabs/open, Playwright create/navigate paths, and preserve the intentional CDP endpoint userinfo behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: submit /navigate or /tabs/open with a URL like https://user:pass@example.com; current main's guard does not check URL.username/password and the route code forwards the same URL downstream.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared page-navigation check in the existing navigation guard, with tests at the guard and route/runtime boundaries; changing CDP endpoint auth handling would be the wrong layer.
