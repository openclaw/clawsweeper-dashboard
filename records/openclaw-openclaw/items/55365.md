## Summary

Keep open: current main and the latest release still allow browser page-navigation URLs with embedded credentials to reach browser runtime paths, and the open closing PR at https://github.com/openclaw/openclaw/pull/82490 should remain the implementation path before this issue closes.

## What This Changes

Reject browser open/navigate URLs containing embedded username or password before Chromium/Playwright and redact any surfaced error or log text.

## Best Possible Solution

Land a narrow shared browser navigation guard and redaction fix through https://github.com/openclaw/openclaw/pull/82490 or a replacement PR, preserving documented remote CDP auth and adding real browser/proxy proof.

## Reproduction Assessment

Yes, at source level. Current main parses and forwards HTTP(S) browser navigation URLs without checking `URL.username` or `URL.password`, though I did not run a live Chromium reproduction in this read-only review.

## Solution Assessment

Yes. The shared browser navigation guard is the right fix point, provided the implementation keeps remote CDP `cdpUrl` credential support separate from page-navigation URL rejection.
