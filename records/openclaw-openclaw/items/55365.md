## Summary

Keep open: current main still accepts credential-bearing browser page-navigation URLs through the shared guard and forwards the original string into browser runtime paths; the related cron webhook PR and CDP SecretRef issue are adjacent but do not resolve this browser input surface.

## What This Changes

The issue requests rejecting browser tool navigation URLs that contain embedded username or password credentials and redacting any surfaced error text.

## Best Possible Solution

Add a narrow userinfo rejection in the shared browser page-navigation guard with redacted InvalidBrowserNavigationUrlError text, cover /navigate, /tabs/open, Playwright, and CDP page-opening paths, and leave remote CDP cdpUrl auth intact.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: send /navigate or /tabs/open a URL like https://user:pass@example.com, and current main parses it as http(s) without checking URL.username/password before passing the original string onward.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared page-navigation check in extensions/browser/src/browser/navigation-guard.ts plus boundary tests; changing remote CDP endpoint auth handling would solve the wrong surface.
