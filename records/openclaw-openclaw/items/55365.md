## Summary

Keep open: current main and latest release still lack a browser navigation userinfo guard, and the linked closing PR is still open rather than merged.

## What This Changes

The issue asks the browser tool to reject page-navigation URLs containing embedded username/password credentials and avoid echoing those credentials in browser errors or logs.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/82490 or an equivalent narrow browser guard/redaction fix, while preserving documented remote CDP credential handling.

## Reproduction Assessment

Yes, at source level. Current main parses and forwards HTTP(S) browser navigation URLs without checking `URL.username` or `URL.password`, though I did not run a live Chromium reproduction in this read-only review.

## Solution Assessment

Yes. The shared browser navigation guard is the right fix point, provided the implementation keeps remote CDP `cdpUrl` credential support separate from page-navigation URL rejection.
