## Summary

Keep open for maintainer review: this is a focused, evidence-backed bug fix with no blocking code findings, but it intentionally turns node removal into a device-credential mutation path and needs maintainer acceptance plus normal merge gating.

## What This Changes

The PR changes gateway `node.pair.remove`, docs, and tests so removing a displayed node can revoke a device-backed `node` role, clear matching legacy node records, and block queued requests after credential invalidation.

## Best Possible Solution

Land this PR after maintainers accept the node-removal credential semantics and normal checks finish; no ClawSweeper repair is indicated.

## Reproduction Assessment

Yes. Current main and v2026.6.6 can list device-backed node-role pairings through `createKnownNodeCatalog`, while `node.pair.remove` still calls only `removePairedNode` and returns `unknown nodeId` when the row is only in `devices/paired.json`.

## Solution Assessment

Yes, with maintainer acceptance of the semantics. The PR fixes the gateway/storage owner boundary by resolving the device-backed row in `node.pair.remove`, reusing device-management authz, removing only the `node` role, clearing any legacy node row, and extending the existing credential barrier.
