## Summary

Keep open: current main and the latest release still queue accepted Discord reactions as system events without requesting a heartbeat wake, and an open linked PR already tracks the specific implementation. A broader shared notification-wake PR is also open, so this should stay open until maintainers choose and merge the fix path.

## What This Changes

Discord reaction notifications should wake the routed agent session after an accepted reaction system event is queued.

## Best Possible Solution

Keep this issue as the Discord-specific tracker until maintainers land either the linked narrow PR or the broader shared notification-wake policy, preserving existing gates and wake-after-success semantics.

## Reproduction Assessment

Yes, source-reproducible: an accepted Discord reaction reaches emitReaction and enqueueSystemEvent, but current main does not request a heartbeat wake on that path. I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the product/API path. The narrow implementation shape is otherwise clear: request a heartbeat only after enqueueSystemEvent returns true and keep all existing reaction notification, sender, channel, and guild gates intact.
