## Label Justifications

- `P2`: The issue affects Discord reaction-only operator signals but is scoped to a channel behavior rather than a core outage.
- `impact:message-loss`: Accepted reaction-only inputs can remain queued without promptly waking the routed agent, so operator signals may be effectively missed until another wake occurs.

## Summary

Keep open: current main and v2026.5.18 still queue accepted Discord reaction notifications as system events without requesting a heartbeat wake, and the live implementation PR that closes this issue remains open and conflicting.

## What This Changes

The issue requests that accepted Discord reaction notification events wake the routed agent session after the system event is successfully queued.

## Best Possible Solution

Review, repair, and land or reject https://github.com/openclaw/openclaw/pull/83169; if accepted, preserve existing Discord reaction gates and make wake behavior default-safe with focused queued, wake, off, blocked, and deduped tests.

## Reproduction Assessment

Yes, source-reproducible: in current main an accepted Discord reaction reaches emitReaction, resolves a route, and calls enqueueSystemEvent, with no requestHeartbeat/notificationWake match in the listener or focused tests. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear until maintainers decide policy; the likely maintainable shape is the linked PR's shared opt-in notificationWake policy rather than unconditional reaction wakes.
