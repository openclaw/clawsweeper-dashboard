## Summary

Keep open: current main still queues accepted Discord reaction notifications as system events without requesting a heartbeat wake, and an open closing PR already owns an implementation path that needs maintainer/product review.

## What This Changes

Request accepted Discord reaction notification system events to wake the routed agent session after successful enqueue while preserving existing reaction notification gates.

## Best Possible Solution

Choose the canonical shared notification wake policy path, preserve queue-only behavior unless maintainers approve opt-in wake semantics, and close this issue after the selected PR lands with focused Discord coverage.

## Reproduction Assessment

Yes, source-reproducible: accepted Discord reactions reach enqueueSystemEvent without any requestHeartbeat call on current main. I did not run a live Discord scenario in this read-only review.

## Solution Assessment

Unclear until maintainers choose the reaction-wake policy and canonical branch. The narrow safe shape is to wake only after successful enqueue and keep existing access/reaction gates intact, preferably behind an explicit notification wake policy.
