This issue should remain the tracker until https://github.com/openclaw/openclaw/pull/83169 is either landed or rejected.

## What This Changes

The issue requests immediate routed session wake behavior for accepted Discord reaction notification events after successful enqueue, while preserving existing Discord reaction notification gates.

## Best Possible Solution

Review one canonical notification-wake policy path, preferably by reconciling https://github.com/openclaw/openclaw/pull/83169 with the shared policy work, then close this issue only after the chosen implementation lands or is explicitly rejected.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted Discord reactions reach `emitReaction`, resolve a route, and call `enqueueSystemEvent`, while the inspected current-main path has no heartbeat wake call or notification-wake policy.

## Solution Assessment

Unclear until maintainers choose the canonical behavior. A policy-controlled queue/wake/off path is safer than an unconditional Discord-only wake, but this issue should track the linked implementation PR until that decision is made.
