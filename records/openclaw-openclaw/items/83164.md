## Summary

Keep open: current main still queues accepted Discord reaction notifications as passive system events without requesting a heartbeat wake, and the linked closing PR plus broader shared-policy PR are both still open.

## What This Changes

The issue requests that accepted Discord reaction-only notifications wake the routed agent session after the system event is successfully queued.

## Best Possible Solution

Review and land either the narrow Discord PR or the shared notification wake policy, then close this issue only after accepted Discord reactions wake sessions or maintainers explicitly keep them queued-only.

## Reproduction Assessment

Yes, source-reproducible: accepted Discord reactions reach enqueueSystemEvent on current main, but no heartbeat wake is requested on that path. I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the reaction-wake policy. The narrow code shape is clear after approval: wake only after enqueueSystemEvent returns true and preserve existing reactionNotifications and access gates.
