## Summary

Keep open: current main already queues accepted Discord reaction notifications as session system events, but the Discord reaction path does not request a heartbeat wake. The requested behavior is plausible and narrowly scoped, but it changes when reaction-only Discord inputs can start agent work and should wait for maintainer/product direction while the related Telegram wake work remains open.

## What This Changes

Request accepted Discord reaction notification events to wake the routed agent session after the existing system event is successfully queued.

## Best Possible Solution

If maintainers approve reaction wake semantics, implement the wake inside the Discord monitor after enqueueSystemEvent returns true, preserving all existing reactionNotifications and access gates and adding focused monitor tests for queued, blocked, and deduped reactions.

## Reproduction Assessment

Yes, source-reproducible: an accepted Discord reaction reaches emitReaction and enqueueSystemEvent, but no heartbeat wake is requested on that path. I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the reaction-wake policy; the narrow implementation shape is otherwise straightforward: use the existing heartbeat-runtime API only after a successful enqueue and keep all existing gates intact.
