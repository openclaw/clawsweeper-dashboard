## Summary

Keep open: current main and the latest release still queue accepted Discord reaction notifications as passive system events without a notification wake policy, and the open linked PRs are implementation candidates that still need a maintainer product/API choice before this issue can close.

## What This Changes

Request accepted Discord reaction notifications to wake the routed agent session after enqueue while preserving reaction notification gating and access controls.

## Best Possible Solution

Land one maintainer-approved notification-wake policy that keeps queue as the safe default, makes wake/off explicit, runs only after existing Discord gates, and closes this issue after the chosen implementation merges.

## Reproduction Assessment

Yes at source level: current main routes accepted Discord reactions to enqueueSystemEvent and targeted search found no notificationWake or heartbeat request in the reaction path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the canonical policy path. A default-safe queue/wake/off policy after existing Discord gates is safer than unconditional wake, but this issue should stay open until that direction merges or is rejected.
