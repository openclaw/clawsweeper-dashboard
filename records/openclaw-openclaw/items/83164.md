## Summary

Keep open: current main still queues accepted Discord reaction notifications as session system events without requesting a heartbeat wake, and https://github.com/openclaw/openclaw/pull/83169 is an open closing implementation candidate. The broader shared-policy discussion in https://github.com/openclaw/openclaw/issues/17840 and https://github.com/openclaw/openclaw/pull/83611 also means this should wait for maintainer behavior choice rather than be closed as cleanup.

## What This Changes

Request accepted Discord reaction notifications to wake the routed agent session after successful system-event enqueue while preserving existing notification modes and access gates.

## Best Possible Solution

Review and land either the scoped Discord PR or the shared notification-wake policy, keeping passive/default behavior safe and waking only after accepted reaction events successfully enqueue.

Yes. By source inspection, an accepted Discord reaction reaches emitReaction and enqueueSystemEvent on current main, while no requestHeartbeat call exists in that listener path; I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrow implementation is straightforward, but https://github.com/openclaw/openclaw/pull/83611 may be the more maintainable route if the intended outcome is a shared reaction wake policy.
