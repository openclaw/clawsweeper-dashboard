## Summary

Keep open: current main can enqueue accepted Discord reaction notifications as system events, but it does not wake the routed session afterward. A same-author open implementation PR already closes this issue, so the issue should stay open until that PR is reviewed, landed, or explicitly superseded.

## What This Changes

Request Discord reaction notifications that pass existing filters to wake the routed agent session after the system event is successfully queued.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/83169 against the product decision; if approved, wake only after enqueueSystemEvent returns true while preserving all existing Discord access and reactionNotification gates.

## Reproduction Assessment

Yes, source-reproducible: an accepted Discord reaction can reach emitReaction, resolve a route, and enqueue a system event, but no heartbeat wake is requested on that path. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear until maintainers decide the reaction-wake policy. The narrow technical shape is straightforward: use the existing heartbeat API only after successful enqueue and keep the existing filters intact.
