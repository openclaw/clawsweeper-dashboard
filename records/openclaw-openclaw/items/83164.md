## Summary

Keep open: current main and v2026.6.8 still queue accepted Discord reaction notifications as session system events without a reaction wake policy or heartbeat request, and the linked implementation PR is open and unmerged.

## What This Changes

Requested behavior is for accepted Discord reaction notifications to enqueue the existing routed system event and wake the routed agent session immediately after successful enqueue.

## Best Possible Solution

Land one maintainer-approved notification-wake policy that keeps queue as the default, makes wake/off explicit, runs only after existing Discord access and reaction filters, and closes this issue after the chosen implementation merges.

## Reproduction Assessment

Yes at source level: current main routes accepted Discord reactions to enqueueSystemEvent, while targeted search found no reaction wake policy or heartbeat request in the Discord reaction path. I did not run a live Discord repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the canonical policy path. A default-safe queue/wake/off policy after existing Discord gates is safer than unconditional wake, but this issue should track the linked implementation until that direction is merged or rejected.
