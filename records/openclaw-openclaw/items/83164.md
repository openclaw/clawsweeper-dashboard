## Label Justifications

- `P2`: This is a normal-priority channel behavior improvement with clear user friction but limited blast radius and no core outage.
- `impact:message-loss`: Accepted reaction-only inputs are queued as context but may not reach the agent until a later wake, making actionable reaction signals effectively delayed or missed.

## Summary

Keep open: current main still queues accepted Discord reaction notifications as passive session system events without a heartbeat wake, and this issue is paired with the open closing PR https://github.com/openclaw/openclaw/pull/83169 plus a broader open policy PR at https://github.com/openclaw/openclaw/pull/83611.

## What This Changes

Request accepted Discord reaction notifications to wake the routed agent immediately after successful system-event enqueue while preserving existing notification modes and access gates.

## Best Possible Solution

Choose the canonical implementation path, then land wake-after-success behavior with existing Discord gates intact and close this issue after the fixing PR merges.

## Reproduction Assessment

Yes. Source inspection on current main shows accepted Discord reactions reach enqueueSystemEvent, while the wake API remains separate and unused on that path.

## Solution Assessment

Unclear until maintainers choose the product path. The narrow wake-after-success implementation is maintainable if the policy is accepted, but the broader shared notification-wake PR may be the preferred canonical contract.
