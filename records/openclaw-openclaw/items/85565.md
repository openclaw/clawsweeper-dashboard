## Summary

This is a concrete feature request tied to a real shared-gateway reliability gap: current main exposes a global embedded-agent concurrency cap and same-session lanes, but no per-agent lane or per-agent maxConcurrent schema. It should stay open because the requested policy requires a new config surface and maintainer design choice rather than being solved by the related session-lock bug reports.

## What This Changes

Request per-agent embedded-run concurrency controls so one shared gateway can allow different agents to run concurrently while serializing overlapping runs for each individual agent.

## Best Possible Solution

Add a documented per-agent concurrency contract in the agents config schema and enforce it between the existing session lane and global embedded-run cap across gateway and embedded entry points, while leaving the compaction lock bug tracked separately.

## Reproduction Assessment

Yes for the feature gap: current source inspection shows only global embedded-run and same-session lane controls, with no per-agent maxConcurrent schema or lane. The reported field lock loop itself is covered by related live reports, but this item is primarily a new configuration capability.

## Solution Assessment

Unclear until maintainers choose the config contract. The proposed lane composition is plausible, but the best implementation should align schema defaults, per-agent overrides, gateway behavior, docs, and tests before adding a new public knob.
