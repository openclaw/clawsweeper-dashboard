## Summary

Keep open. The PR targets a real BlueBubbles cron transcript gap, but the submitted patch still constructs mirror session keys through generic raw routing instead of the plugin-aware outbound session route, so DM `chat_guid` targets can mirror to the wrong or unknown transcript; it also needs the required user-facing changelog entry.

## What This Changes

The PR adds BlueBubbles-only transcript mirroring to isolated cron direct delivery plus a focused regression test file for group, DM, fallback, and multi-payload mirror cases.

## Best Possible Solution

Use the existing outbound route contract for BlueBubbles cron mirroring, ensure the outbound session entry exists before transcript append, keep non-BlueBubbles direct cron delivery unchanged, and ship the behavior with focused regression coverage plus a single-line `CHANGELOG.md` Fixes entry.

## Reproduction Assessment

Yes. Current main has a static reproduction for the original gap because direct cron delivery calls `deliverOutboundPayloads` without a mirror, and the PR's remaining bug is reproducible from the diff: a DM `chat_guid:iMessage;-;+15551234567` is passed as a raw generic peer id instead of the BlueBubbles-normalized `+15551234567`.

## Solution Assessment

No. The PR direction is right, but duplicating partial BlueBubbles target parsing in cron core bypasses the established plugin-aware route and session-entry seam; reusing `resolveOutboundSessionRoute` or `prepareOutboundMirrorRoute` is the narrower maintainable fix.
