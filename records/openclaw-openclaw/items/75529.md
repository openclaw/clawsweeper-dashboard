## Summary

Keep this PR open. The BlueBubbles route-aware mirror is the right narrow boundary, but the current patch still drops media-only or mixed media context, can mirror failed best-effort batch content, and duplicates the changelog entry.

## What This Changes

The PR adds a BlueBubbles-only isolated cron delivery transcript mirror through outbound session routing, a focused regression test file, and active changelog entries.

## Best Possible Solution

Land a BlueBubbles-only cron mirror that derives text and media from the shared outbound payload projection, mirrors only content consistent with cron delivered semantics, and keeps one changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: media-only BlueBubbles cron payloads pass current delivery filtering but the PR returns no mirror, and best-effort partial success can append a full-batch mirror before cron marks the batch undelivered.

## Solution Assessment

No. The BlueBubbles outbound route seam is the right narrow solution boundary, but this patch needs shared payload projection and success-bound mirroring before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve media in cron transcript mirrors:** `src/cron/isolated-agent/delivery-dispatch.ts:135-141`
  - body: The helper builds the mirror only from `payload.text` and returns before route resolution when that text is empty. Current delivery filtering accepts `mediaUrl` and `mediaUrls`, and the shared mirror projection already carries media, so media-only BlueBubbles cron deliveries can still send without any target transcript mirror and mixed text+media deliveries lose media context. Build the mirror from the shared outbound payload projection and add a media-only regression.
  - confidence: 0.94
- **[P2] Mirror only successfully delivered cron content:** `src/cron/isolated-agent/delivery-dispatch.ts:723-724`
  - body: This passes one full-batch mirror into `deliverOutboundPayloads` before best-effort delivery knows whether every payload succeeded. That helper appends the mirror whenever any result succeeds, while cron treats partial best-effort failures as not delivered, so a multi-payload run can record failed payload text or media in the target transcript. Gate the mirror on all-success semantics or mirror only the successfully delivered projection.
  - confidence: 0.88
- **[P3] Remove the duplicate changelog entry:** `CHANGELOG.md:153-154`
  - body: The active changelog now has two adjacent bullets for the same BlueBubbles cron fix, one with this PR reference and one without. Keep a single user-facing entry so release notes do not duplicate the change.
  - confidence: 0.98
