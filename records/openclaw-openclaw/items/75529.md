## Summary

Keep open. The PR is aimed at a real cron/BlueBubbles transcript gap, but the current head still builds the mirror session through generic routing with the raw BlueBubbles target, so DM and group `chat_guid` deliveries can be appended to a transcript key that normal BlueBubbles replies and message-tool sends will not use. The user-facing fix is also missing the required changelog entry.

## What This Changes

The PR adds BlueBubbles-specific mirror construction to isolated cron delivery dispatch and a new regression test file covering group, DM, fallback, and multi-payload mirroring.

## Best Possible Solution

Mirror cron deliveries through the same plugin-owned outbound session route contract used by normal BlueBubbles sends, keep non-BlueBubbles behavior unchanged unless a generic product decision expands mirroring, cover real DM/group route normalization in tests, and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a DM `chat_guid:iMessage;-;+15551234567` target is normalized to `+15551234567` by the BlueBubbles route seam, while this PR passes the raw `chat_guid:...` id into generic routing and `deliverOutboundPayloads` appends to that supplied key unchanged.

## Solution Assessment

No as submitted. The direction is right, but the maintainable fix should use `resolveOutboundSessionRoute` or an equivalent BlueBubbles-owned route seam rather than duplicating partial target parsing and route construction in cron core.
