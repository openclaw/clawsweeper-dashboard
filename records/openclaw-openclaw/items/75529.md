## Summary

Keep open. The PR targets a real BlueBubbles cron transcript gap, but the submitted patch can still mirror to the wrong transcript because it resolves the mirror route from the raw outbound target instead of the plugin-normalized BlueBubbles peer id; it also lacks the required changelog entry.

## What This Changes

The PR adds BlueBubbles-only transcript mirror construction to isolated cron delivery dispatch and adds a focused regression test file for group, DM, fallback, and multi-payload cases.

## Best Possible Solution

Mirror BlueBubbles cron deliveries through the same plugin-owned outbound/inbound session route contract used by normal BlueBubbles traffic, preserve non-BlueBubbles delivery behavior, cover DM/group target shapes with real route assertions, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR routes the mirror with the raw chat_guid target, while inbound BlueBubbles replies and plugin-owned outbound routing use normalized peer ids, and delivery writes to the supplied mirror session key unchanged.

## Solution Assessment

No as submitted. The direction is reasonable, but the maintainable fix should use resolveOutboundSessionRoute or an equivalent plugin-owned BlueBubbles normalization seam instead of duplicating partial target parsing in cron core.
