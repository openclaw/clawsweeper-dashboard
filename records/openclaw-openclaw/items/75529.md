## Summary

Keep open. The intended bug fix is valid, but the current PR is still not mergeable because the mirror route bypasses the existing BlueBubbles outbound routing contract, so `chat_guid` DM deliveries can be mirrored under a different or unknown transcript key than normal sends and replies. The PR also still needs the required user-facing changelog entry.

## What This Changes

The PR adds BlueBubbles-only transcript mirror construction to isolated cron delivery dispatch plus a focused regression test file for group, DM, fallback, and multi-payload delivery mirroring.

## Best Possible Solution

Route BlueBubbles cron delivery mirroring through the plugin-aware outbound session contract, ensure outbound session metadata exists before transcript append, keep non-BlueBubbles behavior unchanged, and ship the fix with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR passes `chat_guid:iMessage;-;+15551234567` as the generic route peer id, while current BlueBubbles outbound routing normalizes that same DM target to `+15551234567` before building the session key.

## Solution Assessment

No. The direction is right, but the submitted implementation duplicates partial BlueBubbles parsing in cron core and bypasses the existing route/metadata seam; reusing `resolveOutboundSessionRoute` or `prepareOutboundMirrorRoute` is the narrower maintainable fix.
