## Summary

This should stay open: current main still has the generic runtime-only continuation path and a silent memory-flush path that can plausibly produce the reported `Continue the OpenClaw runtime event.` plus `NO_REPLY` behavior, but I do not have a high-confidence live reproduction of the full WebSocket-failure/compaction sequence.

## What This Changes

The issue requests reliable recovery for user-facing turns interrupted by runtime failure or compaction so the pending task is resumed or a visible failure notice is delivered instead of a silent `NO_REPLY`.

## Best Possible Solution

Add a narrow recovery contract for interrupted user-facing turns: runtime-only maintenance events must not consume or silence a pending user-visible objective, and unrecoverable interrupted turns should produce a visible stall/failure notice to the originating channel.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; the source shows a plausible current-main path for the generic runtime event and silent `NO_REPLY`, but the exact WebSocket failure, compaction timing, and reporter session state were not reproduced locally.

## Solution Assessment

Is this the best way to solve the issue? Unclear; the right fix should be in the shared session recovery/runtime-only continuation policy, but maintainers should choose whether to resume the pending objective, block `NO_REPLY`, or send a visible stall notice for unsafe resumes.
