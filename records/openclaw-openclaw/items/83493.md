## Label Justifications

- `P1`: The report describes a real Telegram channel workflow becoming functionally dead during cold-start until manual operator intervention.
- `impact:message-loss`: Incoming Telegram updates are spooled and retried indefinitely instead of being processed into replies.

## Summary

Current main already addresses the reported Telegram cold-start spool wedge: isolated polling ingress now waits for grammY `bot.init()` and retries recoverable init failures before starting the worker or draining spooled updates. The fix is not in v2026.5.12, so affected users need a build from current main or the next release.

## What This Changes

The issue reports a Windows cold-start race where Telegram `getMe`/`bot.init()` times out and isolated polling ingress keeps retrying spooled updates with `Bot not initialized`.

## Best Possible Solution

Keep the current Telegram-owned initialization gate and let the next release carry it to affected Windows Scheduled Task installs.

## Reproduction Assessment

Yes for the affected release: the v2026.5.12 source starts isolated ingress and drains the spool without first initializing the bot, matching the provided logs. I did not run a Windows cold-boot repro, but current main no longer follows that source path.

## Solution Assessment

Yes. Gating isolated ingress on grammY initialization and retrying recoverable init failures is the narrow owner-boundary fix for this failure mode without adding core startup policy.
