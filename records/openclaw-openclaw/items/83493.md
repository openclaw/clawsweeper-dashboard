## Summary

Close as implemented: current main prevents the reported stuck Telegram spool loop by initializing the grammY bot before isolated ingress starts draining, with retry handling for recoverable init failures.

## What This Changes

The issue reports Windows cold-start Telegram `getMe` timeouts that let isolated polling ingress start with an uninitialized bot and retry spooled updates indefinitely.

## Best Possible Solution

Keep the init-before-drain implementation and ship it in the next stable release; if Windows `getMe` timeouts still occur on a build containing the fix, track that remaining transport/startup latency separately from the fixed spool loop.

## Reproduction Assessment

No current-main failure path remains: current source initializes the bot before isolated ingress drains and the regression test asserts that order. The original `v2026.5.12` failure is credible from the report and prior duplicate source analysis.

## Solution Assessment

Yes. Initializing the existing Telegram bot before isolated worker startup is the narrow maintainable fix because it satisfies grammY's initialization contract and reuses the existing recoverable retry path.
