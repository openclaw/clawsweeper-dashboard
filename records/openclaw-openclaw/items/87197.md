## Summary

Keep open: this PR appears to target a real Brave web-search startup gap, but it is not merge-ready because after-fix real behavior proof is missing and the patch adds another local copy of shared web-search compatibility logic instead of reusing the existing SDK helper.

## What This Changes

The PR updates Brave's local `mergeScopedSearchConfig` to define runtime-injected legacy search provider entries as non-enumerable.

## Best Possible Solution

Reuse the shared plugin-SDK web-search config helper in Brave, delete the local compatibility copy, and require Brave-specific startup or validation proof for the linked crash-loop before merge.

## Reproduction Assessment

Yes from source and linked report, but I did not run the container path. Current main's Brave provider still injects an enumerable scoped search config object, while the runtime validator rejects enumerable legacy provider objects and the linked report gives a fresh-container beta crash-loop path.

## Solution Assessment

No. Hiding runtime-injected Brave config is the right direction, but copying the core legacy-provider key list into the plugin is not the narrowest maintainable fix when the SDK already exports the shared helper.
