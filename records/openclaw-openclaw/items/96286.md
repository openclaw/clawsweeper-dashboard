## Summary

The latest head fixes the earlier broad provider-error downgrade, but the new recovery predicate still clears fatality when any error payload is a tool warning and it does not also select the recovered final answer for no-preference delivery channels. This should stay open for author revision, proof, and maintainer consolidation with the canonical cron bug rather than being closed or merged as-is.

## What This Changes

The PR adds a `hasRecoveredByFinalAnswer` cron outcome gate and updates helper tests so tool-warning-shaped recovered cron errors with final assistant text can be treated as non-fatal while plain fatal errors remain fatal.

## Best Possible Solution

Land one canonical cron payload-outcome fix that uses a single guarded recovered-warning predicate for both fatality and final-answer delivery selection, preserves mixed/fatal error controls, and includes real cron delivery proof.

Yes, source-level: current main still returns before normal delivery when recovered warning payloads remain fatal, and the canonical issue has a reduced scheduled cron repro. I did not run a live cron/channel reproduction in this read-only review.

## Solution Assessment

No. The helper is the right layer, but this implementation needs a stricter recovered-warning predicate and matching final-text payload selection for no-preference channels before it is the best fix.
