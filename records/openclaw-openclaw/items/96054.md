## Summary

Keep open: current main and the latest release still contain an agent-facing bundled skill helper that writes live `SKILL.md` files directly while Skill Workshop prompt/docs require generated durable skill work to go through proposals. Resolving whether to remove, quarantine, or bridge that helper touches a security/product boundary, so maintainer review is still needed.

## What This Changes

The issue asks for bundled `skill-creator` scaffolding to stop directly writing `SKILL.md` and instead route durable skill creation through Skill Workshop or a documented fallback.

## Best Possible Solution

Make Skill Workshop the only agent-facing path for durable generated skill creation, then retire `init_skill.py` or clearly document and test it as a non-agent fallback outside the proposal workflow.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main shows `init_skill.py` writing `SKILL.md` directly while the current prompt and docs route generated durable skill work through `skill_workshop`. I did not execute the helper because this review is read-only and that path writes files.

## Solution Assessment

Unclear as a final implementation. Removing the agent-facing direct write is the right outcome, but maintainers should choose whether the helper is deleted, converted to a proposal bridge, or retained only as a documented non-agent fallback.
