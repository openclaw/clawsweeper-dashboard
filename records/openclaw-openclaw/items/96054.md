## Summary

Keep open: current main and the latest release still contain the direct skill-creator writer beside the governed Skill Workshop proposal path, so the reported security-boundary mismatch is real and not already fixed.

## What This Changes

The issue asks OpenClaw to stop skill-creator from bypassing the Skill Workshop proposal workflow when creating or updating durable skills.

## Best Possible Solution

Make Skill Workshop the only agent-facing durable skill creation path, and either remove init_skill.py or clearly quarantine it as a documented non-agent/manual fallback with tests.

## Reproduction Assessment

Yes. Source inspection on current main shows init_skill.py writing SKILL.md directly while the prompt, docs, and service define Skill Workshop as the governed proposal path; I did not execute the helper because this review is read-only and that path writes files.

## Solution Assessment

Unclear as a final implementation. The current direct initializer is not the right agent-facing shape, but maintainers should choose whether to remove it, convert it to a proposal-producing CLI/helper, or retain a narrow non-agent fallback.
