## Summary

Keep open, but this PR should not merge as-is. The bind-side bug is still source-reproducible on current main and is also covered by the open listener-side PR at https://github.com/openclaw/openclaw/pull/40247, while the auth/local-service direction is under active RFC discussion at https://github.com/openclaw/openclaw/issues/69066; this branch is conflicting, lacks real behavior proof, and introduces blocking client-target and auth-hint regressions.

## What This Changes

The PR changes `src/gateway/call.ts` so local Gateway calls derive their host from `gateway.bind` and add a close-error auth hint when no token/password was resolved.

## Best Possible Solution

Keep implicit local callers on loopback, resolve non-loopback bind breakage through the approved listener-side or service-identity design, and add mode-specific auth diagnostics with focused tests and redacted real tailnet/custom proof.

## Reproduction Assessment

Yes, source-level for the bind half: current main local clients target `127.0.0.1` while non-loopback listeners return only the configured bind host. I did not live-run a tailnet/custom gateway or reproduce the silent auth close path in this read-only review.

## Solution Assessment

No. The client-side bind-host change conflicts with current loopback-local caller tests and the default plaintext WebSocket security policy; the safer direction is an approved listener-side dual-bind or service-identity path plus mode-specific diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the loopback client target:** `src/gateway/call.ts:188-189`
  - body: For `gateway.bind=tailnet` or `custom` with TLS disabled, this builds `ws://<non-loopback-host>:<port>`. The same connection-details path still rejects plaintext non-loopback WebSockets unless the private-network break-glass env var is set, so the default setup this PR tries to unblock still fails before connecting. Keep implicit local callers on loopback and fix the listener side, or require a TLS-backed remote URL explicitly.
  - confidence: 0.91
- **[P2] Tailor auth hints to the configured mode:** `src/gateway/call.ts:936-937`
  - body: The new hint fires for every auth mode except `none`, but it always tells users to pass `--token` or configure `gateway.auth.token`. In `password` mode the accepted credential is a password, and in `trusted-proxy` mode token paths are intentionally ignored, so this sends operators toward credentials the gateway will not use.
  - confidence: 0.9
