## Summary

Keep open: the branch still conflicts with current Gateway URL security policy and mode-specific auth guidance, and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The branch changes `src/gateway/call.ts` to derive implicit CLI Gateway targets from tailnet/custom bind config and append an authentication hint on reasonless close errors.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback, resolve tailnet/custom reachability through an approved listener or service-identity design, and land a small mode-aware close-error hint with focused tests.

## Reproduction Assessment

Yes. By source inspection, with `gateway.bind="tailnet"`, TLS disabled, and a discovered `100.64.0.1` address, this branch builds a plaintext non-loopback `ws://` target that current Gateway URL policy rejects by default.

## Solution Assessment

No. The bind-derived client target is not the best fix while current policy intentionally keeps plaintext same-host calls on loopback; the safer path is https://github.com/openclaw/openclaw/pull/40247 or https://github.com/openclaw/openclaw/issues/69066, plus a narrower auth-hint patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and default TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing URL guard rejects plaintext non-loopback WebSockets by default, so the new path still fails unless users enable the private-WS break-glass opt-in; keep implicit same-host calls on loopback or solve the listener side instead.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-938`
  - body: The new hint fires for every auth mode except `none` and always tells users to pass `--token` or set `gateway.auth.token`. That is wrong for password mode and misleading for trusted-proxy, where token fallback is intentionally unsupported, so build the message from the active mode and exclude trusted-proxy token advice.
  - confidence: 0.91
- **[P3] Keep bind-derived source labels accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify, so derive the label from the selected bind mode if bind-derived targets remain.
  - confidence: 0.82
