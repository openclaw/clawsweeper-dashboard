## Summary

Keep this PR open. Current main still lacks the mode-aware silent-close auth diagnostic, and the bind-host problem is only partly covered by the related open #40247; however, this PR is not safe to merge as written because its client-side bind-host change conflicts with the plaintext WebSocket guard and its auth hint gives wrong advice for password/trusted-proxy modes.

## What This Changes

The PR changes `src/gateway/call.ts` so local CLI gateway URLs are derived from `gateway.bind` for tailnet/custom binds and reasonless gateway close errors can include an authentication hint.

## Best Possible Solution

Keep the useful diagnostic goal, but do not merge the branch unchanged. Let #40247 or an equivalent server-side change handle local loopback reachability for tailnet/custom binds, and narrow this PR to a tested close-error diagnostic that chooses token, password, or trusted-proxy guidance based on the active auth mode while preserving the current plaintext WebSocket policy.

## Reproduction Assessment

Yes. The current-main behavior is reproducible from source and existing tests: local gateway connection details still use loopback for tailnet/LAN configs, and close errors have no auth-mode hint. The PR regression is also reproducible by combining its diff with the existing `isSecureWebSocketUrl()` tests for `ws://100.64.0.1:18789`.

## Solution Assessment

No. The user-facing problem is real, but client-side non-loopback URL selection is not the narrowest maintainable fix while the gateway intentionally rejects plaintext non-loopback WebSocket URLs; a server-side loopback bind fix plus a mode-aware auth diagnostic is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:187-188`
  - body: With `gateway.bind=tailnet` or `custom` and TLS disabled, this builds a local URL such as `ws://100.64.0.1:18789`. The existing `isSecureWebSocketUrl()` check rejects plaintext non-loopback URLs by default, so the default setup still fails before it can connect. Keep local calls on loopback, or redesign this path around the existing TLS/break-glass policy.
  - confidence: 0.94
- **[P2] Tailor auth hints to the configured mode:** `src/gateway/call.ts:933-936`
  - body: The hint fires for every auth mode except `none` and always tells users to pass `--token` or configure `gateway.auth.token`. For `password` mode that points at the wrong credential, and for `trusted-proxy` mode it can suggest a token path the server will not accept. Build the hint from the active auth mode and exclude trusted-proxy unless there is a real local password fallback to explain.
  - confidence: 0.91
- **[P3] Keep the source label accurate for bind-derived URLs:** `src/gateway/call.ts:187-188`
  - body: After changing the fallback URL to a tailnet/custom host, the connection details still report `Source: local loopback`. That makes diagnostics misleading exactly when users are debugging non-loopback bind failures. Derive the source label from the selected bind mode if this approach stays.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR changes default local gateway targeting toward plaintext non-loopback WebSocket URLs for tailnet/custom binds, which conflicts with the current remote/VPN security guard and needs redesign before merge.

Concerns:

- **[medium] Plaintext non-loopback gateway URL conflict:** `src/gateway/call.ts:188`
  - body: For non-TLS tailnet/custom configs, the PR constructs `ws://<non-loopback-host>:<port>` as the local target. Current policy rejects that by default because credentials and chat data would traverse a non-loopback plaintext WebSocket, so the fix either fails securely or pressures users toward the break-glass env var.
