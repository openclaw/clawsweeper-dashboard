## Label Justifications

- `P1`: The PR targets a broken Gateway CLI/internal-call workflow for tailnet/custom deployments that can block real management commands.
- `merge-risk: 🚨 compatibility`: The diff changes the implicit local Gateway target for existing tailnet/custom configurations and overlaps with an open listener-side fix.

- `merge-risk: 🚨 availability`: The bind-derived client target can still miss the actual listener when the server falls back from the configured tailnet/custom bind host.

## Summary

Keep open, but this PR is not merge-ready: the tailnet/custom Gateway mismatch is real at source level, yet the branch uses a stale client-side connection path, diverges from server bind fallback behavior, and gives token-only guidance for password/trusted-proxy deployments without real behavior proof.

## What This Changes

The branch changes Gateway CLI connection setup in `src/gateway/call.ts` to choose a local WebSocket host from `gateway.bind` and append an auth hint on reasonless close.

## Best Possible Solution

Adopt one Gateway local-call model, reconcile it with https://github.com/openclaw/openclaw/pull/40247, and make any close-error auth guidance mode-specific with focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes for the bind half at source level: current main local callers target `127.0.0.1` while non-loopback listeners return only the configured bind host. I did not establish a live tailnet/custom run or a high-confidence reproduction of the silent auth-close path.

## Solution Assessment

No, not as proposed: the branch is stale against the current connection-detail split, creates a competing Gateway locality rule, and gives token-only advice for modes that use password or trusted-proxy semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Mirror the server bind fallback before changing the client target:** `src/gateway/call.ts:153-154`
  - body: `resolveGatewayBindHost()` only uses a tailnet/custom address after checking it can actually bind, and otherwise falls back to loopback or `0.0.0.0`. This new client resolver calls `pickPrimaryTailnetIPv4()` directly, so a host with a visible but unbindable Tailnet address can still make the CLI connect to an address where the server is not listening.
  - confidence: 0.87
- **[P2] Tailor the auth hint to the active mode:** `src/gateway/call.ts:936-937`
  - body: The new reasonless-close hint fires for every non-`none` auth mode but always tells users to pass a token or configure `gateway.auth.token`. In `password` and `trusted-proxy` setups that points operators at the wrong credential path, so the diagnostic should be mode-specific or suppressed when token auth cannot win.
  - confidence: 0.92
