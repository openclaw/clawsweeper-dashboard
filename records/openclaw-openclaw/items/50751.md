## Summary

Keep open: the reported CLI/gateway pain is real, but this patch conflicts with current Gateway security policy by redirecting default local calls to plaintext non-loopback URLs, and its auth hint gives incorrect guidance for non-token modes.

## What This Changes

The PR changes `src/gateway/call.ts` so local CLI Gateway URLs follow `gateway.bind` for tailnet/custom binds and reasonless close errors can include an authentication hint.

## Best Possible Solution

Preserve the strict plaintext WebSocket policy, resolve the locality problem through the server/listener model or the linked Gateway auth design, and land a tested mode-aware close hint.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence path: with `gateway.bind="tailnet"` and TLS disabled, this PR would build a `ws://100.64.0.1:<port>` local target while current `isSecureWebSocketUrl()` rejects that URL by default.

## Solution Assessment

No. Client-side non-loopback targeting is not the narrowest maintainable fix while plaintext private Gateway WebSockets are intentionally blocked; a server-side loopback listener fix plus mode-aware auth diagnostics is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing WebSocket security guard rejects plaintext non-loopback URLs by default, so the default setup still fails unless users set the break-glass env var. Keep same-host local calls on loopback or solve the listener side without bypassing the private-WS policy.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-937`
  - body: The hint fires for every auth mode except `none` and always tells users to pass `--token` or configure `gateway.auth.token`. That is wrong for `password` mode and misleading for `trusted-proxy`, where token fallback is intentionally unsupported. Build the hint from the active auth mode and exclude trusted-proxy token advice.
  - confidence: 0.91
- **[P3] Keep the source label accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the non-loopback bind failures this PR is trying to clarify. If bind-derived targets stay, derive the source label from the selected bind mode too.
  - confidence: 0.82
