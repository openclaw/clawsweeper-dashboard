## Summary

Keep open. The PR addresses real CLI gateway pain, but the branch is not merge-ready: its bind-derived local URL path conflicts with the existing plaintext non-loopback WebSocket guard, and its silent-disconnect auth hint gives wrong guidance for password and trusted-proxy modes.

## What This Changes

The PR changes `src/gateway/call.ts` so CLI gateway URLs follow `gateway.bind` for tailnet/custom binds and reasonless gateway close errors can include an authentication hint.

## Best Possible Solution

Keep the diagnostic goal, but do not merge this branch unchanged. The maintainable direction is to keep same-host internal calls on loopback, let #40247 or an equivalent server-side listener change restore loopback reachability for tailnet/custom binds, and add a tested auth diagnostic that chooses token, password, or trusted-proxy guidance from the active auth mode.

## Reproduction Assessment

Yes. Source and tests provide a high-confidence reproduction path: current main still targets loopback for local tailnet/LAN calls, while the PR would build `ws://100.64.0.1:<port>` for default non-TLS tailnet binds, and current `isSecureWebSocketUrl()` tests reject that URL by default.

## Solution Assessment

No. The user-facing problem is real, but client-side non-loopback URL selection is not the best fix while the gateway intentionally blocks plaintext non-loopback WebSocket clients; a server-side loopback bind fix plus a mode-aware auth hint is narrower and safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing WebSocket security guard rejects plaintext non-loopback URLs by default, so the default setup still fails unless users set the break-glass env var. Keep same-host local calls on loopback or redesign this path around the existing TLS/private-WS policy.
  - confidence: 0.94
- **[P2] Tailor auth hints to the configured mode:** `src/gateway/call.ts:933-937`
  - body: The hint fires for every auth mode except `none` and always tells users to pass `--token` or configure `gateway.auth.token`. That is wrong for `password` mode and actively misleading for `trusted-proxy`, where token fallback is intentionally unsupported. Build the hint from the active auth mode and exclude trusted-proxy unless there is a local password fallback to explain.
  - confidence: 0.91
- **[P3] Keep the source label accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, the connection details still report `Source: local loopback`. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify. If bind-derived targets stay, derive the source label from the selected bind mode too.
  - confidence: 0.82
