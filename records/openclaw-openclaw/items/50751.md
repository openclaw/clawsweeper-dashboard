## Summary

Keep open: the PR has a useful auth-diagnostic idea, but the bind-host change conflicts with current loopback/plaintext WebSocket policy and the contributor has not provided real behavior proof.

## What This Changes

The PR changes `src/gateway/call.ts` to derive implicit CLI Gateway hosts from `gateway.bind` for tailnet/custom and add an auth hint on reasonless 1000/1008 disconnects.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback, solve tailnet/custom reachability through the approved listener or service-identity path, and land a separate mode-aware auth hint with focused tests and real setup proof.

## Reproduction Assessment

Yes. By source inspection, with `gateway.bind="tailnet"`, default TLS disabled, and a discovered `100.64.0.1`, this branch builds a plaintext non-loopback `ws://` local target that the current URL guard rejects by default.

## Solution Assessment

No. Changing the implicit client target away from loopback is not the safest fix under the current Gateway URL policy; the better direction is the listener-side approach in https://github.com/openclaw/openclaw/pull/40247 or the service-identity design in https://github.com/openclaw/openclaw/issues/69066, plus a smaller mode-aware auth hint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and default TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing URL guard rejects plaintext non-loopback WebSockets by default, so the new path still fails unless users enable the private-WS break-glass opt-in.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-938`
  - body: The new hint fires for every auth mode except `none` and always tells users to pass `--token` or set `gateway.auth.token`. That is wrong for password mode and misleading for trusted-proxy setups where token fallback is intentionally unsupported, so build the message from the active mode.
  - confidence: 0.91
- **[P3] Keep bind-derived source labels accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify, so derive the label from the selected bind mode if bind-derived targets remain.
  - confidence: 0.82
