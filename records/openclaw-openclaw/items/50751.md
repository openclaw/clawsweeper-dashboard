## Summary

Keep open: this targets a real Gateway CLI/auth problem, but the branch is not merge-ready because it changes the local loopback trust boundary, gives mode-wrong auth guidance, lacks real behavior proof, and overlaps an open Gateway locality/auth RFC.

## What This Changes

The PR changes Gateway CLI connection setup to derive the implicit WebSocket host from `gateway.bind` and adds a token-oriented hint when the Gateway closes silently under auth.

## Best Possible Solution

Adopt one canonical Gateway local-call model, then land a rebased patch with mode-specific auth diagnostics, focused tests, docs alignment, and redacted runtime proof.

## Reproduction Assessment

Yes, at source level for the bind/locality half: current main builds implicit local URLs with `127.0.0.1` while non-loopback listener hosts are single-address. I did not run a live tailnet/custom Gateway or reproduce the silent auth-close path.

## Solution Assessment

No, not as proposed. Moving implicit local callers to tailnet/custom addresses changes the local trust boundary; the safer path is an approved Gateway locality/auth contract plus mode-specific diagnostics.
