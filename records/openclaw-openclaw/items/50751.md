## Summary

Keep open: the PR targets a real Gateway CLI/local-call failure, but the current patch is stale against `main`, changes the loopback trust boundary, gives token-only auth guidance for non-token modes, and still lacks inspectable real behavior proof.

## What This Changes

The PR changes Gateway local connection setup to derive the fallback WebSocket host from `gateway.bind` and adds an authentication hint on reasonless close events.

## Best Possible Solution

Adopt one canonical Gateway local-call model, keep implicit same-host calls aligned with the approved loopback/auth contract, and add mode-specific close diagnostics with focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level for the bind half: current `main` local callers target `127.0.0.1` while non-loopback listeners return only the configured bind host. I did not establish a live tailnet/custom run or a high-confidence reproduction of the silent auth-close path.

## Solution Assessment

No, not as proposed. The branch needs to be reconciled with the current `connection-details.ts` split and the Gateway locality/auth RFC, and the auth hint should be mode-specific rather than token-only.
