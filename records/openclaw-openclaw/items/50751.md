## Summary

Keep this PR open, but not mergeable as written: the reported tailnet/custom Gateway reachability problem is real, yet the branch solves it by moving implicit local calls to non-loopback plaintext WebSocket URLs that current policy rejects by default. The auth hint idea is useful, but the implementation needs mode-specific wording and real after-fix setup proof.

## What This Changes

The PR modifies `src/gateway/call.ts` to derive implicit CLI Gateway hosts from `gateway.bind` and add an auth hint for reasonless Gateway close codes.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback, fix tailnet/custom reachability at the listener or service-identity layer, and land a separate mode-aware auth diagnostic with focused tests and redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection shows the original tailnet/custom loopback reachability problem on current main, and also shows that this branch’s non-TLS tailnet/custom fix path would build a plaintext non-loopback WebSocket URL rejected by current policy.

## Solution Assessment

No. The branch is not the best fix because it moves local clients onto non-loopback plaintext targets; the safer maintainable direction is the linked listener-side loopback bind or the service-identity design, plus a narrower mode-aware close-error hint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and default TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing URL guard rejects plaintext non-loopback WebSockets by default, so the new path still fails unless users enable the private-WS break-glass opt-in.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-936`
  - body: The new hint fires for every auth mode except `none` and always tells users to pass `--token` or set `gateway.auth.token`. That is wrong for password mode and misleading for trusted-proxy setups where token fallback is intentionally unsupported, so build the message from the active mode.
  - confidence: 0.91
- **[P3] Keep bind-derived source labels accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify, so derive the label from the selected bind mode if bind-derived targets remain.
  - confidence: 0.82
