## Summary

Keep open: the PR targets real Gateway CLI pain, but the client-side host change conflicts with the current strict plaintext WebSocket policy and the auth hint is mode-insensitive. The existing Greptile and ClawSweeper comments are supported by current main source, tests, and the linked open server-side alternative.

## What This Changes

The PR modifies `src/gateway/call.ts` to derive implicit CLI Gateway targets from `gateway.bind` and append auth hints to reasonless close errors.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback or solve tailnet/custom reachability at the listener side, then land a small mode-aware close-error hint with targeted tests.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence path: with `gateway.bind="tailnet"` and TLS disabled, this PR builds a `ws://100.64.0.1:<port>` implicit local target while current policy rejects plaintext non-loopback WebSockets by default.

## Solution Assessment

No. Client-side non-loopback targeting is not the narrowest maintainable fix while plaintext private Gateway WebSockets are intentionally blocked; a server-side loopback listener fix plus mode-aware auth diagnostics is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing WebSocket security guard rejects plaintext non-loopback URLs by default, so the default setup still fails unless users set the break-glass env var. Keep implicit same-host calls on loopback or solve the listener side instead.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-938`
  - body: The hint fires for every auth mode except `none` and always tells users to pass `--token` or configure `gateway.auth.token`. That is wrong for `password` mode and misleading for `trusted-proxy`, where token fallback is intentionally unsupported. Build the hint from the active auth mode and exclude trusted-proxy token advice.
  - confidence: 0.91
- **[P3] Keep the source label accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the non-loopback bind failures this PR is trying to clarify. If bind-derived targets stay, derive the source label from the selected bind mode too.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete transport-policy conflict by constructing plaintext non-loopback Gateway URLs for implicit local calls.

Concerns:
