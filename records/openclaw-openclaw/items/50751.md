## Summary

Keep open. The branch targets a real Gateway usability problem, but its bind-derived host strategy conflicts with the current plaintext non-loopback WebSocket guard, the new auth hint is mode-incorrect, and the external PR does not include after-fix real behavior proof.

## What This Changes

The branch changes CLI Gateway connection details to target tailnet/custom bind hosts and adds an auth hint for reasonless Gateway close errors.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback, fix tailnet/custom reachability at the listener or service-identity layer, and land a small mode-aware close-error hint with targeted tests.

## Reproduction Assessment

Yes, by source inspection and current tests. With `gateway.bind="tailnet"`, TLS disabled, and a discovered `100.64.0.1` address, this branch builds `ws://100.64.0.1:<port>` while current Gateway URL policy rejects plaintext non-loopback WebSockets by default.

## Solution Assessment

No. The bind-host change is not the best fix while current policy intentionally keeps plaintext same-host calls on loopback; the safer direction is https://github.com/openclaw/openclaw/pull/40247 or the design path in https://github.com/openclaw/openclaw/issues/69066 plus a narrower auth-hint patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and default TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing URL guard rejects plaintext non-loopback WebSockets by default, so the new path still fails unless users enable the private-WS break-glass env var; keep implicit same-host calls on loopback or solve the listener side instead.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-938`
  - body: The new hint fires for every auth mode except `none` and always tells users to pass `--token` or set `gateway.auth.token`. That is wrong for `password` mode and misleading for `trusted-proxy`, where token fallback is intentionally unsupported, so build the message from the active mode and exclude trusted-proxy token advice.
  - confidence: 0.91
- **[P3] Keep bind-derived source labels accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify, so derive the label from the selected bind mode if bind-derived targets remain.
  - confidence: 0.82
