## Summary

Keep open: the PR addresses real Gateway CLI/local auth pain, but the branch conflicts with current plaintext WebSocket policy, gives mode-inaccurate auth guidance, lacks real after-fix proof, and overlaps with an open listener/service-identity direction.

## What This Changes

The branch changes `src/gateway/call.ts` to choose implicit local Gateway hosts from `gateway.bind` for tailnet/custom and append an auth hint on reasonless 1000/1008 closes.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback, solve tailnet/custom reachability at the listener or service-identity layer, and land a narrow mode-aware auth hint with focused tests.

## Reproduction Assessment

Yes. By source inspection, with `gateway.bind="tailnet"`, default TLS disabled, and a discovered `100.64.0.1` address, this branch builds a plaintext non-loopback `ws://` target that current Gateway URL policy rejects by default.

## Solution Assessment

No. The bind-derived client target is not the best fix while current policy intentionally keeps plaintext same-host calls on loopback; the safer direction is https://github.com/openclaw/openclaw/pull/40247 or the service-identity design in https://github.com/openclaw/openclaw/issues/69066, plus a smaller mode-aware auth-hint patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and default TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing URL guard rejects plaintext non-loopback WebSockets by default, so the new path still fails unless users enable the private-WS break-glass opt-in.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-938`
  - body: The new hint fires for every auth mode except `none` and always tells users to pass `--token` or set `gateway.auth.token`. That is wrong for password mode and misleading for trusted-proxy, where token fallback is intentionally unsupported, so build the message from the active mode and exclude token advice where it cannot work.
  - confidence: 0.91
- **[P3] Keep bind-derived source labels accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify, so derive the label from the selected bind mode if bind-derived targets remain.
  - confidence: 0.82
