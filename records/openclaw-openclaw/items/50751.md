## Summary

This PR should stay open for maintainer review, but it is not merge-ready: the branch is conflicting, missing real behavior proof, and mixes a client-side Gateway locality change with a generic auth hint that is wrong for some configured auth modes.

## What This Changes

The PR changes Gateway CLI connection setup to derive the local WebSocket host from `gateway.bind` and adds a reasonless-close authentication hint when no credentials were resolved.

## Best Possible Solution

Adopt one canonical Gateway local-call model, preferably by reconciling this branch with the listener-side work, and make any close-error hint mode-specific with focused tests plus redacted terminal/log proof.

## Reproduction Assessment

Yes at source level for the bind half: current main local callers target `127.0.0.1` while non-loopback listeners return only the configured bind host. I did not establish a live tailnet/custom run or a high-confidence reproduction of the silent auth-close path.

## Solution Assessment

No, not as proposed: the branch conflicts with the current connection-details split, creates a competing client-side Gateway locality rule, and gives token-only advice for modes that use password or trusted-proxy semantics.
