## Summary

Keep open for maintainer review: the reported Gateway CLI problem is source-visible on current main, but this PR is not the right merge shape because it moves implicit local calls onto non-loopback plaintext URLs, gives token-only auth guidance for other auth modes, lacks real behavior proof, and overlaps an open listener-side PR plus a service-identity RFC.

## What This Changes

The PR changes `src/gateway/call.ts` to derive implicit CLI Gateway targets from `gateway.bind` and append an auth hint for reasonless Gateway close codes.

## Best Possible Solution

Keep implicit same-host Gateway calls on loopback, fix tailnet/custom reachability at the listener or approved service-identity layer, and land a separate mode-aware close-error hint with focused tests and redacted real setup proof.

## Reproduction Assessment

Yes, source-level. Current main still builds implicit local Gateway URLs as `127.0.0.1` while non-loopback listeners return only the requested bind host, and this PR branch would then hit the plaintext non-loopback URL guard when TLS is off.

## Solution Assessment

No. The PR's client-side tailnet/custom URL rewrite conflicts with loopback-only plaintext policy and gives token-only auth guidance; the safer maintainable path is listener-side loopback or approved service identity plus mode-aware diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and default TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The current URL guard rejects plaintext non-loopback WebSockets by default, so the new path still fails unless users enable the private-WS break-glass opt-in.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-936`
  - body: The new hint fires for every auth mode except `none` and always tells users to pass `--token` or set `gateway.auth.token`. Password mode needs password guidance, and trusted-proxy relies on proxy identity rather than token fallback, so this gives wrong operator instructions on the failure path.
  - confidence: 0.91
- **[P3] Keep bind-derived source labels accurate:** `src/gateway/call.ts:188-189`
  - body: If bind-derived targets remain, connection details still report `Source: local loopback` even when the target is a tailnet or custom host. That makes diagnostics misleading in the exact non-loopback bind failures this PR is trying to clarify.
  - confidence: 0.82
