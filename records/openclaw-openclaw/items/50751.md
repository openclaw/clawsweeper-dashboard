## Summary

Keep open: the bind-host half of this PR conflicts with the current Gateway transport policy, and the auth-hint half is still useful but needs mode-aware wording before merge. The related open #40247 is the safer server-side track for tailnet/custom local calls, while this PR could still inform a narrower diagnostic fix.

## What This Changes

The PR changes `src/gateway/call.ts` to derive the local Gateway target host from `gateway.bind` for tailnet/custom binds and append an authentication hint to reasonless Gateway close errors.

## Best Possible Solution

Keep the strict plaintext WebSocket policy, solve tailnet/custom same-host reachability on the listener side through #40247 or its successor, and land a separate tested auth diagnostic that is correct for token, password, and trusted-proxy modes.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence reproduction path: this PR can build `ws://100.64.0.1:<port>` for a non-TLS tailnet local target, while current `isSecureWebSocketUrl()` and `net.test.ts` reject that URL by default.

## Solution Assessment

No. The client-side host change is not the narrowest maintainable fix because it conflicts with the plaintext private-WS guard; a server-side loopback listener fix plus mode-aware close diagnostics is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve loopback for plaintext local calls:** `src/gateway/call.ts:188-189`
  - body: With `gateway.bind=tailnet` or `custom` and TLS disabled, this builds a local target like `ws://100.64.0.1:18789`. The existing WebSocket security guard rejects plaintext non-loopback URLs by default, so the setup still fails unless users set the break-glass env var. Keep implicit local calls on loopback and solve the listener side instead.
  - confidence: 0.94
- **[P2] Tailor auth hints to the active mode:** `src/gateway/call.ts:933-937`
  - body: The new hint fires for every auth mode except `none` but always tells users to pass `--token` or configure `gateway.auth.token`. That is wrong for password mode and misleading for trusted-proxy, where token fallback is intentionally not the credential path. Build the hint from the active auth mode and avoid token advice where it cannot work.
  - confidence: 0.91
- **[P3] Keep the source label accurate:** `src/gateway/call.ts:188-189`
  - body: After changing the fallback URL to a tailnet/custom host, connection details still report `Source: local loopback`. That makes diagnostics misleading in the non-loopback bind failures this PR is trying to clarify. If bind-derived targets stay, derive the source label from the selected bind mode too.
  - confidence: 0.82
