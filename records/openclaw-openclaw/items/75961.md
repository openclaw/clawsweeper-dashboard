## Summary

This PR should stay open: the Discord deploy-mode direction is useful and targets a real gap from the linked report, but the current patch still has blocking correctness issues in cache semantics, generated config metadata, and documented native-off cleanup behavior.

## What This Changes

The PR adds a Discord `slashCommandDeploy` config mode with persistent command fingerprints, serialized background deploy state, docs/changelog/config surfaces, Discord extension tests, and a Vitest 4 runner import update.

## Best Possible Solution

Land a Discord-local deploy-mode implementation after the new modes own cache semantics end to end, generated config metadata is refreshed, and native-off clearing is either implemented with regression coverage or removed from user-facing docs.

## Reproduction Assessment

Yes at source level: current main has no `slashCommandDeploy` config surface and still schedules native Discord command reconcile through the existing startup path. I did not establish a live multi-account Discord 429 reproduction in this read-only review.

## Solution Assessment

No. The overall direction is maintainable, but this patch is not the best current solution until it removes the legacy shared-cache interaction, refreshes generated config metadata, and aligns native-off docs with wired behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bypass the shared deploy cache for deploy modes:** `extensions/discord/src/monitor/provider.startup.ts:143-151`
  - body: The monitor startup path still passes `commandDeployHashStorePath` while also adding `commandDeployInitialHashes`. Because the deployer loads that legacy shared file before comparing hashes, a previous account/application can still make another account skip REST, and `slashCommandDeploy: "always"` can still skip instead of reconciling on every restart. Remove or bypass the legacy cache for this path and force deploys for `always`.
  - confidence: 0.9
- **[P2] Regenerate the bundled config metadata:** `src/config/zod-schema.providers-core.ts:644-653`
  - body: This adds `slashCommandDeploy` to the source schema/types and UI hints, but the PR does not update the generated bundled channel metadata that config schema/UI assembly consumes. Without regenerating the tracked metadata outputs, bundled channel consumers can miss the new key even though the source zod schema changed.
  - confidence: 0.89
- **[P2] Wire native-off clearing before documenting it:** `docs/channels/discord.md:627-633`
  - body: The docs now say `commands.native=false` explicitly clears existing Discord native commands and resets fingerprints, but the PR does not add a monitor call site for `clearDiscordNativeCommands`. Either implement the explicit native-off clear path with coverage or keep the existing documented behavior that startup registration is skipped without cleanup.
  - confidence: 0.87
