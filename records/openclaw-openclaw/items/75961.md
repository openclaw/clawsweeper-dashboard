## Summary

Keep open: current main still lacks the requested Discord deploy-mode config, but this PR remains incorrect because the mode-aware path still reads the legacy shared cache, generated channel metadata is missing, native-off clearing is documented without being wired, and real Discord proof is still mock-only.

## What This Changes

The PR adds Discord `slashCommandDeploy` modes, account/application-scoped fingerprint persistence, serialized background deploys, docs/changelog/config schema updates, Discord tests, and a Vitest 4 runner import update.

## Best Possible Solution

Land a Discord-local deploy policy after cache ownership, generated metadata, and native-off behavior are aligned, with redacted live gateway/Discord proof.

## Reproduction Assessment

Yes at source level: current main has no `slashCommandDeploy` schema/type/runtime branch and still schedules background deploy when native commands are enabled. I did not reproduce the live multi-account Discord 429 path.

## Solution Assessment

No. The direction is maintainable, but this patch is not the best current solution until it bypasses or scopes the legacy cache for deploy modes, regenerates bundled metadata, and either wires native-off clearing with coverage or restores the docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bypass the shared deploy cache for deploy modes:** `extensions/discord/src/monitor/provider.startup.ts:143-151`
  - body: The new mode path still passes `commandDeployHashStorePath` while also adding account-scoped `commandDeployInitialHashes`. Because the deployer loads that legacy shared file before comparing hashes, a prior deploy can still make another account skip REST, and `slashCommandDeploy: "always"` can skip instead of reconciling every restart.
  - confidence: 0.9

  - body: This adds `slashCommandDeploy` to the source schema/types and UI hints, but the tracked bundled channel metadata is not updated. Config schema/UI assembly reads that generated file, so bundled channel consumers can miss the new key even though the source zod schema changed.
  - confidence: 0.89
- **[P2] Wire native-off clearing before documenting it:** `docs/channels/discord.md:627-633`
  - body: The docs now say `commands.native=false` clears existing Discord native commands, but the monitor path never calls `clearDiscordNativeCommands` and the existing provider test still asserts no clear log. Either implement the explicit native-off clear path with regression coverage or keep the prior documented behavior.
  - confidence: 0.88
