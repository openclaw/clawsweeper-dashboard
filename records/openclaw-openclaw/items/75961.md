## Summary

Keep open. The PR addresses the right Discord deploy-policy gap, but it is not merge-ready because generated config consumers still miss the new key and the new per-account fingerprint path is still combined with the legacy shared deploy cache.

## What This Changes

The PR adds Discord `slashCommandDeploy` modes with persisted fingerprints, serialized background deploy, native-off cleanup, docs/changelog/tests, and a Vitest 4 runner import update.

## Best Possible Solution

Land a Discord-local implementation after generated config artifacts are refreshed and deploy modes use account/application-scoped fingerprints without the legacy shared-cache skip path.

## Reproduction Assessment

Yes. Current main source lacks `slashCommandDeploy` and still schedules background reconcile for native-enabled Discord accounts; exact Discord 429 timing depends on live API state.

## Solution Assessment

No. The direction is maintainable, but the patch needs generated config artifacts and deploy-cache semantics fixed before it is the best solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Disable the legacy shared deploy cache:** `extensions/discord/src/monitor/provider.startup.ts:148-150`
  - body: The new per-account `commandDeployInitialHashes` are added while the client still receives the old shared `commandDeployHashStorePath`. Since `DiscordCommandDeployer` loads that shared file before checking hashes, a cold multi-account startup can let account A write the legacy hash, account B skip Discord REST, and then record B as successfully deployed; it also makes `always` subject to an old hash skip. Use only account/application-scoped fingerprints for `changed-only`, and bypass cached hashes for `always`.
  - confidence: 0.88
- **[P2] Regenerate generated config surfaces:** `src/config/zod-schema.providers-core.ts:622-631`
  - body: This adds `slashCommandDeploy` to the source Discord schema, but PR head still leaves the generated base schema and bundled channel metadata without that key. `src/config/schema.ts` serves schema/UI consumers from those generated artifacts, so the new option will not appear for generated config consumers until the tracked outputs are regenerated and committed.
  - confidence: 0.93
