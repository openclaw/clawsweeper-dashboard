## Summary

Keep open: the PR targets the right Discord slash-command deploy problem, but the current head still has blocking generated-config drift and keeps the old shared deploy cache in a way that can preserve the multi-account skip bug it is trying to fix.

## What This Changes

The PR adds a Discord `slashCommandDeploy` config mode, persisted slash-command deploy fingerprints, queued background deploy, native-off fingerprint cleanup, docs/changelog updates, Discord tests, SDK config type exports, and a Vitest 4 runner import update.

## Best Possible Solution

Keep the Discord-local deploy policy, remove or safely migrate the old shared cache path, regenerate all tracked config schema/metadata artifacts, then land after focused Discord and config validation.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested `slashCommandDeploy` config surface is absent and Discord startup still schedules native command reconcile; the PR-head regressions are also source-reproducible from the raw generated files and cache load order.

## Solution Assessment

No. The implementation direction is maintainable, but the current patch is incomplete because generated config consumers miss the new option and the old shared cache can still suppress per-account deploys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Stop loading the shared deploy cache:** `extensions/discord/src/monitor/provider.startup.ts:148-150`
  - body: The new per-account `commandDeployInitialHashes` path is still combined with the old un-namespaced `discord/command-deploy-cache.json`. Because the deployer loads that shared cache after seeding the new hashes, a prior or other-account `global:reconcile` hash can skip REST for this account and then get written as a successful namespaced fingerprint. Drop, migrate, or namespace the old cache before making `changed-only` the default.
  - confidence: 0.91
- **[P2] Regenerate generated config schemas:** `src/config/zod-schema.providers-core.ts:622-631`
  - body: This adds `slashCommandDeploy` to the source Discord schema, but the PR head still leaves `src/config/schema.base.generated.ts` and `src/config/bundled-channel-config-metadata.generated.ts` without the new key. `src/config/schema.ts` serves those generated artifacts to config/UI consumers, so the new option remains hidden until the tracked generated outputs are refreshed.
  - confidence: 0.93
