## Summary

Keep open. The patch is still incorrect: PR head leaves the new deploy policy mixed with the legacy shared cache, omits generated channel config metadata, documents native-off clearing without wiring it, and the supplied real-behavior proof is only test/typecheck output.

## What This Changes

The PR adds Discord `slashCommandDeploy` modes, persisted deploy fingerprints, serialized background deploy coordination, docs/config/schema/test updates, and a Vitest 4 runner import update.

## Best Possible Solution

Land a Discord-local deploy policy only after cache ownership, generated metadata, and native-off clearing semantics are aligned, with redacted live Discord gateway proof.

## Reproduction Assessment

Yes at source level: current main lacks the requested `slashCommandDeploy` schema/type/runtime branch and still schedules native command deploy when native commands are enabled. I did not exercise the live multi-account Discord 429 timing.

## Solution Assessment

No. The direction is maintainable, but this patch is not ready until the new policy bypasses or scopes the legacy cache, regenerated config metadata is included, and documented native-off clearing is wired or reverted.

  - confidence: 0.9

  - body: The docs now say `commands.native=false` clears previously registered Discord native commands, but PR head never calls `clearDiscordNativeCommands()` from the monitor path; the existing provider test still asserts no clear log. Either implement and test that startup clear path or keep the prior documented behavior.
