## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bypass the shared deploy cache for new deploy modes:** `extensions/discord/src/monitor/provider.startup.ts:145-149`
  - body: The client still receives the legacy `commandDeployHashStorePath` while this adds account/application-scoped `commandDeployInitialHashes`. Because `DiscordCommandDeployer.loadPersistedHashes()` loads that shared file before comparing hashes, a previous account/application can make another account skip REST, and `slashCommandDeploy: "always"` can still skip. Remove or bypass the legacy shared cache for monitor startup and force deploys for `always`.
  - confidence: 0.9
- **[P2] Regenerate the bundled config metadata:** `src/config/zod-schema.providers-core.ts:641-650`
  - body: This adds `slashCommandDeploy` to the source zod/type surface, but the PR does not update the generated bundled channel metadata or generated baselines. `src/config/schema.ts` and config validation seed bundled channel consumers from `src/config/bundled-channel-config-metadata.generated.ts`, so generated config UI/schema consumers will miss the new key until the tracked outputs are refreshed.
  - confidence: 0.91
- **[P2] Wire native-off clearing before documenting it:** `docs/channels/discord.md:627-633`
  - body: The PR says `commands.native=false` clears existing Discord commands and stored fingerprints, but the monitor path still only disables background deploy and `clearDiscordNativeCommands` has no call site. Either call it on explicit native-off with regression coverage, or keep the existing docs/behavior that no cleanup REST is attempted.
  - confidence: 0.88
