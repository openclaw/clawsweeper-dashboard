## Summary

Keep open. This PR is a plausible implementation for the linked Discord slash-command deploy policy request, but the current branch still has blocking mode semantics, generated metadata, native-off behavior, and real-behavior-proof gaps.

## What This Changes

The PR adds a Discord `slashCommandDeploy` config surface with deploy modes, persisted slash-command fingerprints, serialized background deploy, docs/changelog/schema/test updates, and a Vitest 4 runner import change.

## Best Possible Solution

Land a Discord-local deploy policy after mode semantics use one account/application-scoped cache, generated config metadata is refreshed, native-off clearing is wired and tested or the docs are restored, and redacted live Discord proof is supplied.

## Reproduction Assessment

Yes at source level: current main lacks `slashCommandDeploy` in config types/schema/docs and still schedules native deploy when native commands are enabled. I did not live-reproduce the Discord 429 burst.

## Solution Assessment

No. The direction is maintainable, but this patch is not the best current solution until it fixes the shared-cache interaction, regenerates bundled metadata, and aligns native-off clearing behavior with docs/tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bypass the shared deploy cache for deploy modes:** `extensions/discord/src/monitor/provider.startup.ts:143-150`
  - body: The mode-aware client still passes the old shared `commandDeployHashStorePath` alongside account-scoped `commandDeployInitialHashes`. Because `DiscordCommandDeployer.loadPersistedHashes()` loads that shared file before comparing hashes, a prior account can make `changed-only` skip from the wrong cache, and `slashCommandDeploy: "always"` can skip instead of reconciling every restart.
  - confidence: 0.9
- **[P2] Regenerate the bundled config metadata:** `src/config/zod-schema.providers-core.ts:641-653`
  - body: The source schema/types add `slashCommandDeploy`, but the generated bundled channel metadata is absent from this PR and still lacks that property. Runtime config validation and schema/UI assembly read that generated metadata, so bundled Discord config consumers can still miss or reject the new key.
  - confidence: 0.89
- **[P2] Wire native-off clearing before documenting it:** `docs/channels/discord.md:627`
  - body: The docs now say `commands.native=false` clears previously registered Discord commands, but `monitorDiscordProvider` never calls `clearDiscordNativeCommands`, and the provider test still asserts no clear log when native commands are disabled. Either implement the clear path with coverage or keep the prior documented behavior.
  - confidence: 0.88
