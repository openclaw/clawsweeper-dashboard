## Summary

Keep open. The deploy-policy direction is useful, but the current PR is still incorrect because it mixes the new account-scoped fingerprint state with the old shared deploy cache, misses generated config metadata, documents native-off clearing that is not wired, and still lacks real Discord gateway proof.

## What This Changes

The PR adds Discord `slashCommandDeploy` modes with persisted slash-command fingerprints, serialized background deploys, schema/docs/UI hint/test updates, and a Vitest runner import update.

## Best Possible Solution

Land a Discord-local deploy policy after it uses one account/application-scoped cache source, updates generated metadata, wires native-off clearing or revises the docs, and includes redacted live Discord proof.

## Reproduction Assessment

Yes at source level: current main lacks the `slashCommandDeploy` schema/type/runtime surface, and the current Discord startup path still relies on background reconcile behavior. I did not reproduce the live Discord 429 timing against a real multi-account gateway.

## Solution Assessment

No. The proposed direction is maintainable, but this patch is not the best current solution until cache ownership, generated metadata, native-off behavior, and real Discord proof are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Scope the shared deploy cache for deploy modes:** `extensions/discord/src/monitor/provider.startup.ts:148-151`
  - body: The new mode path still passes the legacy `commandDeployHashStorePath` while adding account-scoped `commandDeployInitialHashes`. Since the deployer loads that shared file before comparing hashes, a prior deploy can make another account/application skip REST, and `slashCommandDeploy: "always"` can still skip instead of reconciling every restart.
  - confidence: 0.91
- **[P2] Regenerate bundled config metadata:** `src/config/zod-schema.providers-core.ts:644-653`
  - body: This adds `slashCommandDeploy` to the source schema/types and UI hints, but the tracked bundled channel metadata is not updated and still lacks the key. Config schema/UI assembly consumes that generated metadata, so bundled channel consumers can miss or reject the new option even though the source zod schema changed.
  - confidence: 0.89
- **[P2] Wire native-off clearing before documenting it:** `docs/channels/discord.md:627-633`
  - body: The docs now say `commands.native=false` clears existing Discord native commands, but the monitor path never calls `clearDiscordNativeCommands`, and the provider test still asserts no clear log. Either implement the explicit native-off clear path with regression coverage or keep the prior documented behavior.
  - confidence: 0.88
