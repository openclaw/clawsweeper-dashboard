## Summary

Keep open: the PR is directionally useful and has live Discord proof, but the shipped sidecar-cache migration still cannot reliably seed the new `changed-only` path after upgrade because it imports legacy hash keys unchanged while the PR-head deployer checks app-scoped keys.

## What This Changes

Adds Discord `slashCommandDeploy` modes, plugin-state-backed startup deploy fingerprints, legacy sidecar migration, serialized background deploys, and matching Discord docs/schema/metadata/tests.

## Best Possible Solution

Keep the Discord-owned plugin-state design, but make the doctor migration translate every released legacy cache shape into the same app-scoped key shape the current deployer reads before accepting the new default.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: feed the migration a `v2026.6.5`-style `{ hashes: { "global:reconcile": sha } }` cache and compare its imported value with the PR-head deployer key `app:<applicationId>:global:reconcile`; I did not run a mutating test in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not yet; plugin-state persistence and a doctor migration are the right boundary, but the migration must translate shipped cache keys and env-token application IDs before this is the safest upgrade path.
