## Summary

Keep this PR open: current main still lacks the requested Discord deploy-mode config, but the proposed patch has blocking runtime/docs/generated-metadata issues and the supplied proof is test-only rather than real gateway/Discord behavior proof.

## What This Changes

The PR adds Discord `slashCommandDeploy` modes with persisted slash-command fingerprints, queued background deploys, config/docs/changelog updates, Discord tests, and a Vitest runner import update.

## Best Possible Solution

Land a Discord-local deploy-mode implementation after the cache ownership semantics, generated config metadata, and native-off clearing docs/wiring are aligned, with contributor-supplied redacted real gateway/Discord proof.

## Reproduction Assessment

Yes at source level: current main has no `slashCommandDeploy` config surface and still schedules Discord deploy when native commands are enabled. I did not establish a live multi-account Discord 429 reproduction in this read-only review.

## Solution Assessment

No. The direction is maintainable, but this patch is not the best current solution until it removes the legacy shared-cache interaction, refreshes generated metadata, and either wires native-off clearing with coverage or removes that claim from the docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bypass the shared deploy cache for deploy modes:** `extensions/discord/src/monitor/provider.startup.ts:143-151`
  - body: The new mode path still passes `commandDeployHashStorePath` while also adding account-scoped `commandDeployInitialHashes`. Because the deployer loads that legacy shared file before comparing hashes, a prior deploy can still make another account skip REST, and `slashCommandDeploy: "always"` can skip instead of reconciling every restart.
  - confidence: 0.9
- **[P2] Regenerate the bundled config metadata:** `src/config/zod-schema.providers-core.ts:644-653`
  - body: This adds `slashCommandDeploy` to the source schema/types and UI hints, but the tracked bundled channel metadata is not updated. Config schema/UI assembly reads that generated file, so bundled channel consumers can miss the new key even though the source zod schema changed.
  - confidence: 0.89
- **[P2] Wire native-off clearing before documenting it:** `docs/channels/discord.md:627-633`
  - body: The docs now say `commands.native=false` clears existing Discord native commands, but the monitor path never calls `clearDiscordNativeCommands`. Either implement the explicit native-off clear path with regression coverage or keep the prior documented behavior that startup registration is skipped without cleanup.
  - confidence: 0.87
