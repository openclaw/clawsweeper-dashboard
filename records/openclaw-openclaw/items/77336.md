## Summary

Keep open: current main already has generic strict OpenAI-compatible turn validation, and the SGLang/custom path can use it, but the bundled Mistral provider owns its replay policy and currently opts into only strict9 tool-call IDs, so the Mistral-family strict-alternation failure remains unresolved.

## What This Changes

The issue requests built-in handling for Mistral/SGLang backends that reject consecutive same-role chat messages with strict role-alternation errors.

## Best Possible Solution

Extend the Mistral provider replay policy to enable the existing strict turn validators while preserving strict9 tool-call IDs, then add regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, source-reproducible: `resolveTranscriptPolicy` prefers the Mistral provider-owned policy, and that policy does not enable the validators that merge consecutive user/assistant turns. I did not run the live Mistral/SGLang HTTP path.

## Solution Assessment

Yes: the narrow maintainable solution is to reuse the existing provider replay-policy flags and validators rather than adding a new config knob or requiring the external proxy.
