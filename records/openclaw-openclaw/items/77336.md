## Summary

Keep open: current main already handles unowned/custom `openai-completions` SGLang-style providers with strict turn validators, but the bundled Mistral provider still owns its replay policy and only opts into strict9 tool-call IDs, so the reported Mistral-family role-alternation failure remains source-reproducible.

## What This Changes

The issue requests built-in strict role alternation handling for Mistral/SGLang-style backends so consecutive same-role turns are merged before provider requests.

## Best Possible Solution

Extend the bundled Mistral replay policy to opt into the existing strict turn validators while preserving strict9 tool-call ID handling, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, from source inspection: the Mistral provider has a provider-owned replay policy that bypasses the generic strict OpenAI-compatible fallback and does not enable the validators that merge same-role turns. I did not run a live Mistral/SGLang HTTP reproduction.

## Solution Assessment

Yes: reusing the existing provider replay-policy flags and validators is the narrowest maintainable fix; adding a new config knob or keeping a side-car proxy would be broader than necessary.

On current main, the bundled Mistral provider's `buildReplayPolicy` only enables strict9 tool-call ID sanitization; because provider-owned replay policy bypasses the generic strict `openai-completions` fallback, Mistral-family strict role alternation does not use `validateGeminiTurns` or `validateAnthropicTurns`.
