## Summary

Keep open: current main still leaves the bundled Mistral provider with a provider-owned replay policy that only sanitizes strict9 tool-call IDs, so the generic OpenAI-compatible turn validators are bypassed while Mistral chat templates reject non-alternating turns. The existing SGLang-owned replay path shows the intended helper shape, but it does not fix the Mistral provider path. ([docs.mistral.ai](https://docs.mistral.ai/resources/cookbooks/concept-deep-dive-tokenization-chat_templates?utm_source=openai))

## What This Changes

Requested built-in handling for strict user/assistant role alternation with Mistral-family SGLang/OpenAI-compatible backends instead of relying on an external proxy.

## Best Possible Solution

Extend the bundled Mistral replay policy to reuse the existing strict-turn validators, with focused tests for consecutive user turns while preserving strict9 IDs and reasoning replay semantics.

## Reproduction Assessment

Yes, source-reproducible: current main shows Mistral's provider-owned replay policy lacks the validators that merge consecutive user turns, and that policy bypasses the generic OpenAI-compatible fallback. I did not run a live SGLang/Mistral server.

## Solution Assessment

Yes: the narrow maintainable fix is a Mistral provider replay-policy update that reuses the existing OpenAI-compatible turn normalization. A new config flag or external proxy is not needed for the reported default provider behavior.
