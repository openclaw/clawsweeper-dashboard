## Summary

Keep open: the SGLang-owned path now has strict OpenAI-compatible replay handling, but current main still leaves the bundled Mistral provider with only strict9 tool-call ID sanitization and no turn validators, so the reported Mistral-family role alternation failure remains source-reproducible.

## What This Changes

The issue requests built-in strict role alternation handling for Mistral/SGLang-style OpenAI-compatible backends when OpenClaw would otherwise send consecutive same-role messages.

## Best Possible Solution

Extend the bundled Mistral replay policy to add the existing strict-turn validators while preserving Mistral-specific strict9 tool-call IDs and current reasoning replay semantics, then cover it with focused provider tests.

## Reproduction Assessment

Yes, source-reproducible on current main: Mistral owns a replay policy that bypasses the generic strict OpenAI-compatible fallback, and upstream Mistral templates reject non-alternating user/assistant messages. I did not run a live SGLang server.

## Solution Assessment

Yes: the maintainable fix is a narrow Mistral provider replay-policy update that reuses existing strict-turn normalization rather than adding a new config knob or requiring an external proxy.
