## Summary

Keep open: current main still leaves the bundled Mistral provider on a provider-owned replay policy that only applies `strict9` tool-call IDs, while the generic OpenAI-compatible and SGLang paths already carry the turn validators needed for strict role alternation. The reporter’s linked side-car workaround and follow-up comment align with the narrow in-repo fix, and I found no open PR superseding this issue.

## What This Changes

Add built-in provider-side normalization for consecutive same-role messages when Mistral/SGLang strict role alternation rejects OpenAI-compatible chat histories.

## Best Possible Solution

Extend the Mistral provider replay hook to reuse the existing strict-turn validators while preserving `strict9` IDs, then add focused provider and transcript-policy regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: Mistral is `openai-completions`, but its provider-owned policy keeps both turn validators false, so consecutive user messages are not merged before reaching strict Mistral/SGLang templates. I did not run a live SGLang server.

## Solution Assessment

Yes: the narrow maintainable solution is to compose the existing OpenAI-compatible turn validators in the Mistral provider and keep the existing `strict9` Mistral tool-call ID behavior, rather than adding a new config knob or keeping an external proxy.
