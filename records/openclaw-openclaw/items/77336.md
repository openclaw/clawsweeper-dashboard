## Summary

Keep open: current main already handles strict turn merging for generic `openai-completions` and the bundled SGLang provider path, but the bundled Mistral provider owns its replay policy and still only enables strict9 tool-call ID sanitization, leaving the Mistral-family part of the report unresolved.

## What This Changes

The issue requests built-in merging or replay-policy handling for providers that reject consecutive same-role messages, especially Mistral models and Mistral models served through SGLang.

## Best Possible Solution

Extend the Mistral provider replay policy to opt into the existing strict turn validators while preserving strict9 tool-call IDs, then add regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes, from source inspection: the Mistral provider has a provider-owned replay policy that bypasses the generic strict OpenAI-compatible fallback and does not enable the validators that merge consecutive user/assistant turns. I did not run a live Mistral/SGLang HTTP reproduction.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing provider replay-policy flags and validators for Mistral rather than adding a new config knob or side-car proxy behavior.
