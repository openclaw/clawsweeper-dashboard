## Summary

Keep open for maintainer review before merge. The patch addresses a real shipped media-deletion path and has sufficient host screenshot proof, but it also changes the unset `media.ttlHours` behavior from implicit write-time cleanup to no automatic cleanup, which is compatibility-sensitive.

## What This Changes

The PR removes write-time media pruning from `saveMediaSource`, maps explicit non-recursive cleanup to root-only depth, adds a media-store regression test, and updates `media.ttlHours` help text.

## Best Possible Solution

Land after maintainers explicitly accept the opt-in cleanup default with release-note context, or preserve a non-destructive compatibility cleanup path while still removing the silent first-level deletion behavior.

## Reproduction Assessment

Yes for a source-level reproduction: current main and `v2026.6.6` call `cleanOldMedia(DEFAULT_TTL_MS, { recursive: false })` in `saveMediaSource`, and fs-safe descends into first-level directories when `maxDepth: 1`. I did not run a fresh local repro, but the inspected before/after host screenshots match that path.

## Solution Assessment

Unclear until maintainer policy review. Removing write-time pruning and relying on the configured timer is the clean owner-boundary fix for silent deletion, but the unset-default retention change is compatibility-sensitive.
