## Summary

Keep this PR open for maintainer review: it fixes a real shipped media-loss path, but it intentionally changes default retention behavior for deployments with `media.ttlHours` unset.

## What This Changes

The PR removes `saveMediaSource` write-time pruning, maps non-recursive media cleanup to root-only pruning, adds a regression test, and updates `media.ttlHours` help text.

## Best Possible Solution

Land this only after maintainers explicitly accept the opt-in cleanup default, or adjust the patch to preserve a non-destructive compatibility cleanup path with clear release-note context.

## Reproduction Assessment

Yes, from source inspection and supplied real-host screenshots: current shipped code calls `saveMediaSource` cleanup with a two-minute TTL, and fs-safe descends into first-level media directories when `maxDepth: 1`. I did not run a local live reproduction in this read-only review.

## Solution Assessment

Unclear until maintainer policy review: removing write-time pruning and leaving cleanup to the configured timer is the clean owner-boundary fix for silent deletion, but the unset-default retention change is compatibility-sensitive.
