## Summary

The PR should stay open: current main and v2026.6.1 still perform write-time media cleanup from `saveMediaSource`, and the fs-safe 0.3.0 prune contract explains why the old `recursive:false` mapping can delete first-level media subdirectories. The branch is a plausible and narrow fix with sufficient real host proof, but it deliberately changes shipped default retention for users with `media.ttlHours` unset, so maintainer acceptance of that compatibility risk remains the gating decision.

## What This Changes

The PR removes write-time media pruning from `saveMediaSource`, changes non-recursive `cleanOldMedia` to root-only cleanup, and updates the `media.ttlHours` help text.

## Best Possible Solution

Keep cleanup owned by the configured gateway maintenance timer if maintainers accept the retention compatibility change; otherwise require an explicit upgrade-compatible cleanup policy before merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to create an old file under `media/inbound`, then call `saveMediaSource`; current main and v2026.6.1 call two-minute cleanup on write, and fs-safe descends into first-level directories when `maxDepth` is 1.

## Solution Assessment

Likely yes, subject to maintainer acceptance of the compatibility change. Removing write-time cleanup puts retention under the configured gateway timer and the PR keeps recursive timer cleanup behavior unchanged; the safer alternative is only needed if implicit cleanup is considered a shipped contract.
