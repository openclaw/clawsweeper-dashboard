## Label Justifications

- `impact:data-loss`: Oversized `model.completed` events can silently drop the recorded per-turn `usage` object from persisted trajectory data.

## Summary

Keep open: current main and the latest release still replace the entire oversized trajectory `data` object, and the issue has an open closing fix candidate at https://github.com/openclaw/openclaw/pull/96811.

## What This Changes

The issue asks for oversized `model.completed` trajectory truncation to preserve small accounting fields such as `usage` while dropping large payload fields like `messagesSnapshot`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96811 or an equivalent narrow fix that preserves accounting fields under the existing event cap.

## Reproduction Assessment

Yes, at source level: current main records `usage` inside `model.completed.data` and then replaces that whole data object when the serialized row exceeds 262144 bytes. I did not run a live heavy-turn replay in this read-only review.

## Solution Assessment

Yes in direction: preserving small structured accounting fields inside the truncation stub is the narrow runtime-layer fix. The open linked PR covers both the core and bundled Codex recorder paths and should be reviewed or replaced before this issue closes.
