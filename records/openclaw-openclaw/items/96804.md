## Label Justifications

- `P2`: The bug affects token and cost accounting for oversized trajectory events, but the blast radius is limited to trajectory capture rather than core agent availability.
- `impact:data-loss`: Oversized `model.completed` trajectory events can silently drop `data.usage`, losing per-turn accounting data.

## Summary

The issue is still necessary: current main and the latest release still replace oversized trajectory event data with a truncation stub, so `model.completed.data.usage` is not preserved. An open closing PR already owns the fix path, so the issue should stay open until https://github.com/openclaw/openclaw/pull/96811 lands or is replaced.

## What This Changes

The issue asks for oversized `model.completed` trajectory events to preserve small accounting fields such as `data.usage` while dropping only large payload fields.

## Best Possible Solution

Review and land the open closing PR, or replace it with an equivalent narrow fix that preserves accounting-critical fields in both trajectory recorders while keeping the event byte cap enforced.

## Reproduction Assessment

Yes from source inspection: current main stores usage inside `model.completed.data`, then replaces the entire data object when the serialized event exceeds the 256 KiB cap. I did not run a writable trajectory reproduction because this review is read-only.

## Solution Assessment

Yes in direction: preserving small structured accounting fields while dropping only large payload fields is the narrow maintainable fix. The open closing PR is the active implementation path and should be reviewed on its exact head before this issue closes.
