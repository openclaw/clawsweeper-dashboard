## Summary

Keep open: current main and the latest release do not emit or advertise the requested wake completion event, and the remaining work is a new wake lifecycle telemetry contract tied to still-open wake injection decisions.

## What This Changes

The issue requests a durable `wake.response_sent` gateway/TUI telemetry event after a wake-injected live-session turn finishes responding.

## Best Possible Solution

Define a first-class wake lifecycle contract after the live injection path is approved, then add the scoped gateway event, completion emitter, TUI/event-log handling, docs, and tests for success/failure semantics with identifier redaction.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a new telemetry event request. Source inspection confirms the event is absent from current gateway event advertising, broadcast guards, wake hook handling, and live Claude turn completion.

## Solution Assessment

Unclear until maintainers settle the broader wake injection contract. A wake-specific completion event may be useful, but the maintainable path should avoid duplicating `session.message` semantics unless wake-id-scoped completion is explicitly needed.
