## Summary

Keep open. Current main does not implement this catalog, and this PR still needs maintainer security/product direction because its documented matcher and tests do not match current exec approval semantics.

## What This Changes

The PR adds a `src/security/destructive-commands.ts` string catalog of HIL-worthy destructive command patterns plus tests for those pattern arrays.

## Best Possible Solution

Treat this catalog as design input, then implement a first-class exec command/argv HIL or denylist policy with config, docs, and production-path tests coordinated with #71097 and #75101.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to evaluate `kubectl delete *` through current `matchAllowlist` with a `kubectl` executable resolution; the matcher compares path/basename semantics rather than full command text, so the PR's advertised match does not occur.

## Solution Assessment

No. An unused command-string catalog with a non-working `matchAllowlist` example is not the safest solution; the maintainable path is a real command/argv policy hook or a coordinated denylist-mode design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use real command semantics for destructive patterns:** `src/security/destructive-commands.ts:27-34`
  - body: `ALL_DESTRUCTIVE_PATTERNS` is built from full command prefixes such as `kubectl delete *`, but the documented consumer is `matchAllowlist`, which evaluates executable paths or PATH-invoked basenames. A caller following this catalog will compare the pattern to `kubectl`, so destructive commands can fall through instead of requiring HIL approval. Model executable and argv separately or add a dedicated command-pattern matcher wired into policy code.
  - confidence: 0.9
- **[P2] Fix the documented matcher call:** `src/security/destructive-commands.ts:153`
  - body: The example calls `matchAllowlist(command, [{ pattern }])`, but the exported function takes allowlist entries first and an executable resolution second. This example will not compile against the real API and cannot initialize a working approval policy, so update or remove it after choosing the actual matcher surface.
  - confidence: 0.92
- **[P2] Test through the production approval matcher:** `src/security/destructive-commands.test.ts:19-31`
  - body: The tests use `startsWith` plus ad hoc string splitting/replacement instead of `matchAllowlist` or the exec allowlist evaluation path. That lets the suite pass even when the catalog cannot work with the approval system, so the tests should exercise the production matching semantics for the policy that will consume these patterns.
  - confidence: 0.88
- **[P3] Align the pattern invariant test name:** `src/security/destructive-commands.test.ts:61-65`
  - body: This test says every pattern ends with `*`, but it only checks that each value is a non-empty string, and existing patterns such as `git push *--force*` do not satisfy the stated invariant. Rename the test or assert the invariant that is actually required.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/security/destructive-commands.ts:1`
  - body: This is a user-facing `feat(security)` change, but the PR only adds the catalog and tests. Add a `CHANGELOG.md` entry for the new exec security surface before merge.
  - confidence: 0.86
