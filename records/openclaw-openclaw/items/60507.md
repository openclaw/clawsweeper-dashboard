## Summary

Keep open. The PR is useful security input, but the diff is not mergeable as written because it documents command-text destructive patterns as inputs to the existing executable allowlist matcher, which does not evaluate command verbs or full argv strings; related exec deny/HIL work is still open in #71097 and #75101.

## What This Changes

Adds a new `src/security` destructive-command pattern catalog plus tests covering kubectl, Terraform, cloud, filesystem, git, database, process, and SSH patterns.

## Best Possible Solution

Implement a first-class exec command/argv policy hook or denylist mode with config, docs, and production-path tests, coordinated with #71097 and #75101, instead of routing command text through the executable allowlist matcher.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to evaluate a pattern like `kubectl delete *` through current `matchAllowlist` using a resolved `kubectl` executable; current main compares that pattern to the executable path or basename, so the advertised destructive-command match does not occur.

## Solution Assessment

No. An unused command-string catalog with a non-compiling/non-semantic matcher example is not the safest solution; the safer path is a dedicated command/argv policy surface with tests against the actual node-host approval flow.

  - body: `ALL_DESTRUCTIVE_PATTERNS` is built from full command prefixes such as `kubectl delete *`, but the documented consumer is the executable allowlist matcher.   - confidence: 0.92

  - body: The tests use `startsWith` after splitting or replacing `" *"`, so they prove only the local string catalog shape. They do not exercise `matchAllowlist`, `evaluateExecAllowlist`, or the node-host policy path that would consume these patterns, which lets the suite pass while the advertised security behavior remains broken.
  - confidence: 0.88

  - body: The test title says every pattern ends with `*`, but the body only checks that values are non-empty strings, and existing entries such as `git push *--force*` do not satisfy the stated invariant. Rename the test or assert the invariant that is actually required.
