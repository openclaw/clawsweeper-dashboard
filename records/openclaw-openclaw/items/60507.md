## Summary

Keep open; this PR is not mergeable as-is. The proposed catalog uses full command-text patterns, but current exec approvals match executable paths/basenames plus optional argv regexes, and the documented example calls the matcher with the wrong API shape. Existing review discussion and related open denylist work confirm this needs a real security-policy design rather than a catalog-only merge.

## What This Changes

The PR adds a new destructive command pattern catalog under `src/security` plus tests for those pattern arrays.

## Best Possible Solution

Treat the catalog as design input and implement a first-class exec command/argv deny or HIL policy with config, docs, UI/protocol coverage, and production-path tests coordinated with the open denylist work.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: evaluate a command-text pattern such as `kubectl delete *` through current `matchAllowlist` and it is compared against executable path/basename semantics, not the full command string.

## Solution Assessment

No. A standalone unused catalog with a broken matcher example is not the safest solution; the maintainable path is a real command/argv policy hook or denylist mode that is wired into the exec approval runtime.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Model executable and argv semantics before matching:** `src/security/destructive-commands.ts:27-34`
  - body: `ALL_DESTRUCTIVE_PATTERNS` is built from full command prefixes such as `kubectl delete *`, but the documented consumer is `matchAllowlist`, which evaluates executable paths or PATH-invoked basenames plus optional argv regexes. A caller following this catalog will not match the advertised destructive commands, so add a real command/argv policy matcher or change the data shape before wiring this into HIL behavior.
  - confidence: 0.9
- **[P2] Correct the documented matcher call:** `src/security/destructive-commands.ts:153`
  - body: The example calls `matchAllowlist(command, [{ pattern }])`, but the exported function takes allowlist entries first and an executable resolution second. This example will not compile against the real API and cannot initialize a working approval policy.
  - confidence: 0.92
- **[P2] Exercise production approval matching in tests:** `src/security/destructive-commands.test.ts:19-31`
  - body: These tests use `startsWith` and ad hoc string splitting instead of the exec approval matcher or policy evaluation path. They can pass even when the catalog misses commands in real approval checks, so the tests should cover the production semantics for whatever policy consumes these patterns.
  - confidence: 0.88
- **[P3] Align the pattern invariant test name:** `src/security/destructive-commands.test.ts:61-65`
  - body: The test title says every pattern ends with `*`, but the body only checks string type and non-empty length, and existing patterns such as `git push *--force*` do not satisfy that invariant. Rename the test or assert the actual invariant.
  - confidence: 0.9
