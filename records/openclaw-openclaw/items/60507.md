## Summary

Keep open for maintainer security/product review. The PR's catalog uses full command-text patterns and documents wiring through the current executable allowlist matcher, which does not model those patterns as command-plus-argv deny/HIL rules; the related native denylist design is still open.

## What This Changes

Adds a new `src/security/destructive-commands.ts` catalog of destructive command-text patterns and colocated tests.

## Best Possible Solution

Define additive exec approval deny/HIL rules that evaluate executable and argv semantics before broad allow decisions, with config, protocol, UI/CLI, docs, and production-path tests updated together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: feed a pattern like `kubectl delete *` through current `matchAllowlist`, and it is interpreted as an executable selector plus optional argv regex rather than a full command-text rule.

## Solution Assessment

No. A standalone unused catalog with a broken matcher example is not the safest solution; the maintainable path is a first-class command/argv deny or HIL policy coordinated with the open native denylist work.

Full review comments:

- **[P1] Model executable and argv semantics before matching:** `src/security/destructive-commands.ts:27-34`
  - body: `ALL_DESTRUCTIVE_PATTERNS` is built from full command prefixes such as `kubectl delete *`, but the documented consumer matches executable path/basename patterns plus optional `argPattern`. A caller following this catalog will not reliably match the advertised destructive commands, so this needs a real command/argv policy shape before integration.
  - confidence: 0.9
- **[P2] Correct the matcher call example:** `src/security/destructive-commands.ts:153`
  - body: The example calls `matchAllowlist(command, [{ pattern }])`, but the exported function takes allowlist entries first and an executable resolution second. This example will not compile against the real API and cannot initialize a working approval policy.
  - confidence: 0.92
- **[P2] Test the production approval path:** `src/security/destructive-commands.test.ts:19-31`
  - body: The tests use `startsWith` and string splitting instead of the exec approval matcher or policy evaluation path. They can pass even when the catalog misses commands in real approval checks, so tests should cover the production semantics for whatever policy consumes these patterns.
  - confidence: 0.88
- **[P3] Align the invariant test name:** `src/security/destructive-commands.test.ts:61-65`
  - body: The test title says every pattern ends with `*`, but the body only checks type and non-empty length, while existing patterns such as `git push *--force*` do not satisfy that invariant. Rename the test or assert the actual invariant.
  - confidence: 0.9
