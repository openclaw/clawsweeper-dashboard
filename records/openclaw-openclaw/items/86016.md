## Summary

Keep open: current main does not provide durable TTS audio archiving. Message TTS writes a temporary delivery file that is scheduled for cleanup, while `talk.speak` returns base64 audio from the in-memory buffer, so `saveAudioTo` would be a new opt-in config and file-write feature rather than an implemented behavior.

## What This Changes

Requested behavior is to optionally save synthesized TTS audio to disk while streaming talk/messages audio so users can archive generated speech without paying for a second synthesis call.

## Best Possible Solution

Design an explicit opt-in synthesized-audio archive contract, either in the speech/TTS pipeline or as a plugin SDK hook, with safe path resolution, deterministic filenames, retention expectations, docs, and focused tests that keep temporary delivery media separate from durable archives.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken documented behavior. Source inspection confirms current main lacks `saveAudioTo`, uses temporary message TTS files, and returns Talk audio as base64 rather than a durable archive path.

## Solution Assessment

Unclear pending maintainer decision: the requested direct `fs.writeFile` idea is feasible, but the safer maintainable solution should define the archive boundary, security model, and whether this belongs in core speech runtime or a plugin/SDK hook.
