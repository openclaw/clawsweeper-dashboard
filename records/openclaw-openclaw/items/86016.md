## Summary

Keep open: current main and the latest release still lack durable TTS audio archiving, and adding it is a new config, storage, and privacy surface that needs maintainer product direction.

## What This Changes

Request to add an opt-in TTS/Talk setting that persists synthesized audio files locally during generation instead of only streaming or temporary delivery.

## Best Possible Solution

Define a generic TTS archival or export contract with fs-safe path handling, retention and privacy semantics, and docs, then implement it consistently for Talk and message TTS.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source and release checks show current OpenClaw has temporary delivery files and Talk base64 responses, but no durable archive option.

## Solution Assessment

Unclear: saving the existing audio buffer is plausible, but the proposed provider-local `saveAudioTo` field may not be the narrowest maintainable boundary. Maintainers should choose the generic archive/export or plugin-hook contract first.
