## Summary

Keep open: current main and the latest release still do not expose durable TTS audio archiving, and the requested `saveAudioTo` behavior is a new opt-in retention/config/API decision rather than a strict bug fix.

## What This Changes

Request to add a `saveAudioTo`-style option for Talk and messages TTS so synthesized audio can be archived locally without a second provider call.

## Best Possible Solution

Maintainers should decide whether durable TTS archiving belongs as a plugin/provider wrapper or a generic TTS archive hook/policy, with safe path handling, opt-in retention, docs, and tests across messages and Talk.

## Reproduction Assessment

Not applicable; this is a feature request, not broken existing behavior. Source inspection confirms current main and v2026.6.8 use temporary TTS delivery files and Talk base64 responses, not durable archives.

## Solution Assessment

Unclear; provider-local `saveAudioTo` is plausible but not proven as the best owner boundary. A maintainable solution should first decide between a plugin/provider wrapper and a generic TTS archive hook/policy.
