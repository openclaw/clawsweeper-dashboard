## Summary

Keep open. Current main still lacks an opt-in sender/channel identity contract for tool hooks, shell exec environment construction, and bundled MCP tool calls; the related PRs are still open implementation candidates rather than merged fixes.

## What This Changes

Expose channel sender and channel identity to skill/tool execution so downstream integrations can attribute actions to the requester.

## Best Possible Solution

Add an additive, opt-in SDK/runtime contract that carries normalized requester identity into trusted hook context, guarded exec env injection, and MCP metadata where appropriate, with sanitizer reuse and privacy docs.

## Reproduction Assessment

Yes. Source inspection on current main traces `sessionCtx.SenderId` into run state, then shows `before_tool_call`, exec env construction, and bundled MCP `client.callTool()` do not receive sender/channel identity.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but it should cover hook context, exec env, MCP metadata, sanitizer reuse, tests, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1617`
  - body: Any exec-env hook should reuse or extend the current host-env validation boundary so plugins cannot inject blocked loader, path, interpreter-startup, or runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.9
