## Summary

Keep open. Current main still lacks an opt-in sender/channel identity contract for `before_tool_call`, shell exec environment construction, and bundled MCP tool calls; related #72796 and #72797 are still open implementation candidates, not merged fixes.

## What This Changes

Expose channel requester identity to skill/tool execution so downstream APIs and MCP servers can attribute actions to the triggering user.

## Best Possible Solution

Add a versioned, opt-in plugin SDK/runtime contract that passes normalized requester identity to trusted hook context, validates env contributions through the existing exec env sanitizer, threads MCP metadata where appropriate, and documents privacy controls.

## Reproduction Assessment

Yes. Source inspection on current main shows `SenderId` reaches run state, then drops before `before_tool_call`, exec env construction, and bundled MCP `client.callTool()`.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but it should cover hook context, exec env sanitization, MCP metadata, tests, and privacy documentation together.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose requester identity to executable tools and third-party MCP surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1617`
  - body: Any exec-env hook should reuse or extend the current host-env validation boundary so plugins cannot inject blocked loader, path, interpreter-startup, or runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.9
- **[medium] Treat MCP requester metadata as sensitive:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: Bundled MCP servers may proxy to third-party or tenant services, so requester identity metadata should be passed only through an explicit trusted/opted-in path rather than silently attaching it to every `callTool` request.
  - confidence: 0.88
