## Summary

Keep open. Current main has a partial requester identity surface for plugin-owned tool factories, but it still does not expose sender/channel identity to before_tool_call hooks, shell exec environment construction, or bundled MCP tool calls; the linked implementation PRs remain open candidates rather than merged fixes.

## What This Changes

Expose channel requester identity, such as SenderId and channel context, to skill/tool execution so downstream tools and APIs can attribute actions to the initiating user.

## Best Possible Solution

Define an additive, opt-in SDK/runtime contract that carries normalized requester identity into hook context, guarded exec env injection, and MCP metadata where appropriate, with sanitizer reuse, tests, and privacy docs.

## Reproduction Assessment

Yes for source-level reproduction: current main carries SenderId into run state, but before_tool_call, exec env construction, and bundled MCP callTool do not receive sender/channel identity. No live test was run because this is a feature request and the gap is visible in the current source.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in hook/context contract is safer than automatic global env injection, but it needs exec sanitizer, node/sandbox/MCP routing, and docs/tests aligned before it is the maintainable fix.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose channel requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Define identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Preserve exec env reserved-key filtering:** `src/agents/bash-tools.exec.ts:1456`
  - body: Any hook-provided exec env values must pass through the same host/node/sandbox validation boundary as request env so plugins cannot reintroduce blocked PATH, loader, interpreter-startup, proxy, TLS, or runtime-control pivots.
  - confidence: 0.9
- **[medium] Define MCP requester metadata boundaries:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: Forwarding requester identity to MCP servers would extend the trust boundary beyond local shell execution, so the contract should specify what identity is sent, when, and how plugin/server authors should treat it.
  - confidence: 0.84
