## Summary

Keep open. Current main still carries sender identity into run/tool-policy state, but it does not expose sender/channel identity through `before_tool_call`, guarded exec environment construction, or bundled MCP tool calls; the related PRs are still open implementation candidates rather than merged fixes.

## What This Changes

Expose channel requester identity to plugin hooks, shell/skill execution, and MCP tool execution so downstream systems can attribute actions to the triggering channel user.

## Best Possible Solution

Add an additive, opt-in SDK/runtime contract that carries normalized requester identity into trusted hook context, guarded exec env injection, and MCP metadata where appropriate, with tests and privacy/security documentation.

## Reproduction Assessment

Yes. A code-inspection reproduction traces `sessionCtx.SenderId` into run state, then verifies `before_tool_call`, exec env construction, and bundled MCP calls omit sender/channel identity.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but it should cover hooks, exec env, MCP, sanitization, and docs together.
