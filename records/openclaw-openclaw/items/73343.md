## Summary

Keep open. Current main still only passes requester identity to narrower plugin-owned tool/message discovery paths, while `before_tool_call`, shell exec environment construction, and bundled MCP tool calls lack the requested sender/channel identity seam; related PRs #72796 and #72797 are open candidates, not merged fixes.

## What This Changes

Expose channel sender and channel identity to skill/tool execution surfaces so downstream integrations can attribute user-triggered actions.

## Best Possible Solution

Add an additive, opt-in SDK/runtime contract that carries normalized requester identity into trusted hook context, validates plugin-contributed exec env, supports MCP metadata where approved, and documents the privacy model.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing `SenderId` into run state and plugin-owned tool context, then verifying that `before_tool_call`, shell exec env construction, and bundled MCP calls do not receive it.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but the final fix should cover hook context, exec env, MCP metadata, tests, and privacy docs together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it proposes exposing requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers may be personal, tenant-scoped, or auth-relevant data. They should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.95
- **[medium] Validate plugin-provided exec env keys:** `src/agents/bash-tools.exec.ts:1615`
  - body: Any exec-env hook should reuse or extend the current host-env sanitizer so plugins cannot inject blocked loader, path, interpreter-startup, or runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.9
- **[medium] Gate MCP requester metadata deliberately:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: If requester identity is threaded through MCP `_meta` or a similar channel, it should be an explicit trust boundary because MCP servers can be third-party integrations outside OpenClaw's process.
  - confidence: 0.86
