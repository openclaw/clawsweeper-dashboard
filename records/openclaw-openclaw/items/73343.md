## Summary

Keep open. Current main still carries requester identity into run state, but it does not expose sender/channel identity through `before_tool_call`, shell exec environment construction, or bundled MCP tool calls; related PRs #72796 and #72797 remain open implementation candidates rather than merged fixes.

## What This Changes

Expose channel sender/channel identity to plugin hooks, shell or skill execution environments, and MCP tool calls so downstream integrations can attribute user-triggered actions.

## Best Possible Solution

Add an additive, opt-in SDK/runtime contract that carries normalized requester sender/channel identity into trusted hook context, guarded exec env injection, and MCP metadata where appropriate, with sanitizer reuse and privacy docs.

## Reproduction Assessment

Yes. A code-inspection reproduction traces `sessionCtx.SenderId` into run options, then verifies `before_tool_call`, exec env construction, and bundled MCP `callTool` do not receive sender/channel identity.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but it should cover hook context, exec env, MCP metadata, docs, and privacy controls coherently.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1617`
  - body: Any exec-env hook should reuse or extend the current host-env sanitization boundary so plugins cannot inject blocked loader, path, interpreter-startup, or runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.9
- **[medium] Scope MCP identity metadata deliberately:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: Passing sender identity through MCP metadata would expose it to external servers, so the contract needs explicit trust semantics, opt-in behavior, and tests proving untrusted/default MCP calls do not receive identity.
  - confidence: 0.86
