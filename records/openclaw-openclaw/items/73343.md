## Summary

Keep open. Current main still only partially carries requester identity: sender identity reaches run state and plugin-owned tool context, but the public before-tool-call contract, shell exec environment builder, and bundled MCP call path still lack the requested sender/channel identity seam. Related PRs #72796 and #72797 are useful implementation candidates, but this remains a public SDK/runtime and privacy-boundary design issue rather than conservative cleanup.

## What This Changes

Expose channel sender and channel identity to skill, shell tool, and MCP tool execution so downstream integrations can attribute actions to the initiating user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester identity into trusted tool context, supports explicit exec-env contribution through the existing env validation boundary, threads caller metadata into MCP where appropriate, and documents privacy/redaction behavior with focused coverage for hooks, exec hosts, plugin tools, and bundled MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing MsgContext.SenderId into run state and plugin-owned tool context, then observing that before_tool_call types/wiring, exec env construction, and bundled MCP callTool do not carry sender/channel identity further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global OPENCLAW_* env injection, but the durable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it requests exposing requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant data. Passing them through env vars or MCP metadata should require explicit plugin/admin opt-in, clear redaction guidance, and a documented trust boundary so untrusted tools do not receive identity by default.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1611`
  - body: Any exec-env hook must reuse or extend the current host-env validation boundary so plugins cannot inject blocked loader, path, interpreter-startup, or runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.88
