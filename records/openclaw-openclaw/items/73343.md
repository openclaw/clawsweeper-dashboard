## Summary

Keep open. Current main still has the sender identity in run state and in some plugin-owned tool contexts, but it does not expose sender/channel identity to `before_tool_call`, shell exec environment construction, or bundled MCP calls; the related implementation PRs remain open rather than merged.

## What This Changes

Expose channel sender and channel identity to plugin hooks and shell/MCP tool execution so downstream integrations can opt in to per-user attribution.

## Best Possible Solution

Add an additive, opt-in SDK/runtime contract that carries normalized requester identity into trusted hook context, guarded exec env injection, and MCP metadata where appropriate, with sanitizer reuse, tests, and privacy docs.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction traces `sessionCtx.SenderId` into run state, then verifies the `before_tool_call` contract, exec env construction, and bundled MCP call path omit sender/channel identity.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but the final fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers may be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented privacy expectations.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1617`
  - body: Any exec-env hook should reuse or extend the existing host env validation boundary so plugins cannot inject blocked loader, path, interpreter-startup, proxy, TLS, or runtime-control variables into subprocesses.
  - confidence: 0.89
- **[medium] Scope MCP requester metadata deliberately:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: If requester identity is threaded through MCP `_meta`, it should be server-scoped and auditable so untrusted or unrelated MCP servers do not receive channel identity by default.
  - confidence: 0.84
