## Summary

Keep open. Current main still lacks the requested opt-in sender/channel identity seam for `before_tool_call`, shell exec environment construction, and bundled MCP tool calls; related PRs #72796 and #72797 are open implementation candidates but are not merged.

## What This Changes

Expose trusted channel sender and channel identity to plugin hooks, shell/skill exec environments, and MCP tool execution so downstream tools can attribute user actions.

## Best Possible Solution

Add an additive, opt-in SDK/runtime contract that carries normalized requester identity into trusted hook context, guarded exec env injection, and MCP metadata where appropriate, with sanitizer reuse and privacy docs.

## Reproduction Assessment

Yes. A code-inspection reproduction traces `sessionCtx.SenderId` into run state, then verifies `before_tool_call`, exec environment construction, and bundled MCP calls omit sender/channel identity.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but it should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1617`
  - body: Any exec-env hook should reuse or extend the current host-env validation boundary so plugins cannot inject blocked loader, path, interpreter-startup, or runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.89
