## Summary

Keep open. Current main still only partially exposes requester identity: plugin-owned tool factories can see requesterSenderId, but before_tool_call hooks, shell exec environment construction, and bundled MCP tool calls still do not receive sender identity; https://github.com/openclaw/openclaw/pull/72796 and https://github.com/openclaw/openclaw/pull/72797 remain open implementation candidates rather than merged fixes.

## What This Changes

Request to expose channel sender and channel identity to skill/tool execution, hook, shell exec, and MCP paths so downstream tools can attribute actions to the initiating channel user.

## Best Possible Solution

Define an additive, opt-in SDK/runtime contract that carries normalized requester identity into hook context, guarded exec env injection, and MCP metadata where appropriate, with sanitizer reuse, regression tests, API baseline/docs, and privacy guidance aligned.

## Reproduction Assessment

Yes, for source-level reproduction: current main normalizes SenderId into run state, but before_tool_call, exec env construction, and bundled MCP callTool do not receive sender/channel identity. No live test was run because this is a feature/API request and the gap is visible in current source.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in hook/context contract is safer than automatic global env injection, but it must preserve exec sanitizer behavior and define node, sandbox, and MCP forwarding semantics before implementation.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would expose channel requester identity to executable tool and third-party MCP surfaces.

Concerns:

- **[medium] Define identity exposure as opt-in:** `src/plugins/hook-types.ts:404`
  - body: Sender and channel identifiers can be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Preserve exec env reserved-key filtering:** `src/agents/bash-tools.exec.ts:1456`
  - body: Any hook-provided exec env values must pass through the same host/node/sandbox validation boundary as request env so plugins cannot reintroduce blocked PATH, loader, proxy, TLS, or runtime-control pivots.
  - confidence: 0.91
- **[medium] Define MCP requester metadata boundaries:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: Forwarding requester identity to MCP servers would extend the trust boundary beyond local shell execution, so the contract should specify what identity is sent, when it is sent, and how plugin/server authors should treat it.
  - confidence: 0.86
