## Summary

Keep open. Current main still omits sender/channel identity from the generic `before_tool_call` contract, exec environment construction, and bundled MCP tool calls; related PRs #72796 and #72797 are open candidates but are not merged.

## What This Changes

Expose inbound channel sender identity to plugin hook and tool/skill/MCP execution surfaces for per-user attribution, audit logging, and access control.

## Best Possible Solution

Land or replace the related PRs with one additive, opt-in SDK/runtime contract that threads requester identity through hook context, guarded exec env injection, and MCP metadata, with privacy docs and focused tests.

## Reproduction Assessment

Yes. A code-inspection reproduction traces `SenderId` into run state, then verifies it is absent from `before_tool_call`, exec env construction, and bundled MCP `callTool` metadata.

## Solution Assessment

Unclear pending maintainer design. The issue's plugin-driven opt-in direction is the maintainable boundary, but the final solution should cover hooks, exec env, MCP metadata, validation, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The requested change is security-sensitive because it would expose requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Keep identity exposure opt-in:**
  - body: Sender and channel identifiers may be personal or tenant-scoped data, so they should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documentation.
  - confidence: 0.94
- **[medium] Validate injected exec env keys:** `src/agents/bash-tools.exec.ts:1617`
  - body: Any hook-provided or built-in identity env must go through the existing host-env validation boundary so plugins cannot inject blocked loader, interpreter-startup, PATH, or runtime-control variables into subprocesses.
  - confidence: 0.9
