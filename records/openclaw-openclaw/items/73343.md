## Summary

Keep open. Current main still only partially carries requester identity: it reaches agent-run state and plugin-owned tool/message contexts, but the public `before_tool_call` contract, shell exec environment builder, and bundle MCP `callTool` path still lack a sender/channel identity seam. The related open PRs are useful adjacent work, but the remaining change is a public SDK/runtime plus privacy/security design decision rather than conservative cleanup.

## What This Changes

Expose trusted channel requester identity to plugin, skill, shell-exec, and MCP tool execution environments so downstream systems can attribute tool calls to the initiating channel user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester identity into trusted tool context, supports explicit exec-env contribution with reserved-key validation, threads caller metadata into MCP where appropriate, documents privacy/redaction behavior, and includes focused coverage for hooks, exec hosts, plugin tools, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing `MsgContext.SenderId` into the run and plugin-owned tool context, then observing that generic hook context/result types, exec env construction, and bundle MCP `callTool` do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global `OPENCLAW_*` env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it requests exposing requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant data. Passing them through env vars or MCP metadata should require explicit plugin/admin opt-in, clear reserved-key handling, and redaction guidance so untrusted tools do not receive identity by default.
  - confidence: 0.94
- **[medium] Validate hook-provided exec env keys:** `src/agents/bash-tools.exec.ts:1611`
  - body: Any exec-env hook must reuse or extend the existing host-env validation boundary so plugins cannot inject blocked loader/path/runtime-control variables into gateway, node, sandbox, or MCP subprocess paths.
  - confidence: 0.88
