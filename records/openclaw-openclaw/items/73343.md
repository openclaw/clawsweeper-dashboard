## Summary

Keep open. Current main carries sender identity into the agent run and selected plugin/message-tool contexts, but the generic `before_tool_call` contract, exec environment construction, and bundle MCP call path still omit trusted sender/channel identity. Related PRs #72796 and #72797 are useful adjacent implementation work, but they are open and the remaining contract is security-sensitive.

## What This Changes

Expose trusted channel sender and channel identity to plugin, skill, shell exec, and MCP tool execution so downstream integrations can attribute calls to the initiating user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester identity into tool execution context, supports explicit exec-env contribution and MCP caller metadata, validates reserved env keys, documents privacy/redaction behavior, and includes focused regression coverage for hooks, exec hosts, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by following `MsgContext.SenderId` into `run.senderId` and `createOpenClawCodingTools`, then observing that `before_tool_call`, exec env construction, and `SessionMcpRuntime.callTool` do not carry sender/channel identity further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global `OPENCLAW_*` injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose trusted channel requester identity to executable subprocess and third-party MCP surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant data. Passing them through environment variables or MCP metadata can expose them to subprocess logs, node hosts, sandbox runtimes, and third-party MCP servers, so any implementation should require explicit plugin/admin opt-in, documented fields, reserved-key validation, and redaction guidance.
