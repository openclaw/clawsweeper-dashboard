## Summary

Keep open. Current main still only partially carries requester identity: it reaches the reply run and some plugin-owned tool context, but not the public `before_tool_call` contract, shell exec environment construction, or bundle MCP `callTool` metadata path. The related open PRs are relevant implementation work, but the remaining shape crosses public SDK/runtime and privacy boundaries.

## What This Changes

The issue requests an opt-in way for plugins, skills, shell tools, and MCP calls to receive channel requester identity such as `SenderId` during execution.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester identity into trusted tool context, allows explicit exec-env contribution with reserved-key validation, threads caller metadata into MCP where appropriate, documents privacy/redaction behavior, and adds focused regression coverage for hooks, exec hosts, plugin tools, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing `MsgContext.SenderId` into `run.senderId` and plugin-owned tool context, then observing that generic tool hook types, hook wiring, exec env construction, and bundle MCP `callTool` do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global `OPENCLAW_*` env injection, but the maintainable fix should cover hook context, explicit exec env contribution, MCP metadata, docs, tests, and privacy controls together.
