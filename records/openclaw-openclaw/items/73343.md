## Summary

Keep open. Current main still lacks the requested sender/channel identity seam for `before_tool_call`, shell exec env construction, and bundled MCP `callTool`; PRs #72796 and #72797 remain open implementation candidates rather than merged proof.

## What This Changes

Request to expose channel sender and channel identity to skill/tool shell execution and MCP/tool hook paths for downstream per-user attribution.

## Best Possible Solution

Add an additive opt-in SDK/runtime contract that threads normalized requester identity into trusted hook context and approved exec/MCP metadata paths, with sanitizer reuse and public privacy/security docs.

## Reproduction Assessment

Yes. A high-confidence code-inspection path exists by tracing `MsgContext.SenderId` into run state and plugin-owned message tool context, then observing that `before_tool_call`, shell exec env construction, and bundled MCP calls do not receive it.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global env injection, but the final design should cover hook context, exec env validation, MCP metadata, and docs together.

## Security Review

Status: needs_attention

Summary: The requested change is security-sensitive because it would expose requester/channel identity to executable tools and third-party MCP servers.

Concerns:

- **[medium] Make identity exposure opt-in:**
  - body: Sender and channel identifiers can identify people, tenants, or workspaces. They should not be exposed to arbitrary tools, env vars, or MCP servers by default without explicit plugin/admin opt-in and documented redaction expectations.
  - confidence: 0.94
- **[medium] Reuse exec env validation:** `src/agents/bash-tools.exec.ts:1615`
  - body: Any hook-provided exec env must pass through the existing sanitizer and blocked-key boundary so plugins cannot override PATH, loader, interpreter-startup, or runtime-control variables.
  - confidence: 0.88
- **[low] Define the MCP metadata trust boundary:** `src/agents/pi-bundle-mcp-runtime.ts:358`
  - body: If requester identity is forwarded via MCP `_meta`, the runtime needs a clear opt-in and documentation because MCP servers may be third-party processes outside OpenClaw's normal plugin trust boundary.
  - confidence: 0.82
