## Summary

Keep open. Current main has partial sender plumbing for agent runs and plugin-owned tools, but it still does not expose requester/channel identity to the generic `before_tool_call` contract, shell exec environment construction, or bundle MCP `callTool` metadata path. The related open PRs are useful adjacent implementation work, but the remaining fix is a public SDK/runtime and privacy/security design decision rather than conservative cleanup.

## What This Changes

Expose trusted channel sender and channel identity to plugin/skill shell execution and MCP tool calls so downstream tools can perform per-user attribution.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester identity into trusted tool context, supports explicit exec-env contribution, threads caller metadata into MCP where appropriate, validates reserved env keys, documents privacy/redaction behavior, and includes focused regression coverage across hooks, exec hosts, plugin tools, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing MsgContext.SenderId into the run and plugin-tool context, then observing that generic hook context/result types, exec env construction, and bundle MCP callTool do not carry it further.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global OPENCLAW_* env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose requester identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers can be personal or tenant data. Passing them through environment variables or MCP metadata can expose them to subprocess logs, node hosts, sandbox runtimes, and third-party MCP servers, so any implementation should require explicit plugin/admin opt-in, reserved-key validation, and redaction guidance.
