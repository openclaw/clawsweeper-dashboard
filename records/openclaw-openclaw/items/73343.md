## Summary

Keep open. Current main has partial sender identity plumbing for agent runs and plugin-owned tools, but it still does not expose requester/channel identity through the generic `before_tool_call` SDK contract, shell exec environment construction, or bundle MCP `callTool` metadata path. The related open PRs are useful adjacent work, but the remaining fix changes public plugin/runtime contracts and needs maintainer security/product design rather than conservative cleanup.

## What This Changes

Expose channel requester identity, such as SenderId and channel context, to skill/tool execution surfaces so downstream exec scripts and MCP/API integrations can attribute actions to the triggering user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK/runtime design that carries normalized requester and channel identity into trusted tool context, supports validated exec-env contribution, threads caller metadata into MCP where appropriate, documents privacy/redaction behavior, and includes focused regression coverage across hooks, exec hosts, plugin tools, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing `sessionCtx.SenderId` into `run.senderId` and plugin-owned tool context, then confirming that generic `before_tool_call`, exec env construction, and bundle MCP `callTool` do not receive it.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global `OPENCLAW_*` env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.
