## Summary

Keep open. Current main has an internal Discord `fetchMessage` runtime path, but the public `message` tool still does not expose or map a `fetch` action, so the requested single-message-by-ID-or-URL surface remains missing.

## What This Changes

Request a Discord `message` tool action that fetches one specific message by `channelId` plus `messageId`, or by parsing a Discord message URL.

## Best Possible Solution

Expose a public Discord `message` action named `fetch` that reuses the existing `fetchMessageDiscord` runtime, supports `channelId` plus `messageId` and Discord message URLs, returns the normalized single-message shape, and updates docs and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high confidence that the requested public `message` tool action is missing while the lower-level Discord REST helper already exists.

## Solution Assessment

Yes. The narrow maintainable solution is to wire the existing Discord `fetchMessage` runtime through the public message-tool contract instead of adding a separate Discord-specific tool or changing unrelated action-gate policy.
