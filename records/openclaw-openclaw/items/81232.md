## Summary

Keep open: current main and the latest release still lack the public Discord `message` `fetch`/`get` action, and an open same-author implementation PR explicitly closes this issue, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests a Discord message-tool action that fetches one specific message by `channelId` plus `messageId` or by parsing a Discord message URL.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81243, or a narrow equivalent, so the existing Discord fetch backend is exposed through the public message tool with direct-ID and URL coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the public action list, Discord discovery, and public Discord handler do not expose `fetch`, while unsupported actions fall through to the provider unsupported-action error.

## Solution Assessment

Yes. The narrow maintainable solution is to expose the existing Discord `fetchMessage`/`Routes.channelMessage` path through the public message tool and preserve the existing Discord messages gate; the linked PR is the active implementation path.
