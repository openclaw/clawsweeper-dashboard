## Summary

Keep open: current main and the latest release still lack the public Discord `fetch` message action, and the open linked PR is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests a Discord message-tool action to retrieve one specific message by `channelId` plus `messageId` or by Discord message URL.

## Best Possible Solution

Review and land a focused implementation like the open PR after it wires the shared action contract, Discord discovery, handler mapping, URL/direct-ID normalization, docs, and focused tests cleanly.

## Reproduction Assessment

Yes, by source inspection: current main omits `fetch` from the public action enum, Discord discovery, and agent-facing handler while retaining an internal `fetchMessage` runtime path. No live Discord setup is needed to prove that tool-surface gap.

## Solution Assessment

Yes, the narrow maintainable solution is to expose the existing Discord `fetchMessage` backend through the public message tool instead of duplicating REST logic or broadening `read` semantics. The linked PR is the current review path for that solution.
