## Summary

Keep open: current main still has the internal Discord single-message fetch backend, but the public message tool does not expose a `fetch` or `get` action, and two open PRs currently reference this issue as their closing target.

## What This Changes

Expose a Discord message-tool action that fetches one specific message by channelId plus messageId or by parsing a Discord message URL.

## Best Possible Solution

Review and land one canonical PR that exposes the existing Discord `fetchMessage` runtime through the public message tool, including URL and direct-ID inputs, messages-gate preservation, gateway routing, schema coverage, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the public action list, Discord discovery, and agent-facing handler do not expose `fetch` or `get`, while the internal runtime and REST helper already support single-message fetch.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing Discord `fetchMessage` backend and wire it through the public message-tool contract, rather than duplicating REST logic or changing `read` semantics.
