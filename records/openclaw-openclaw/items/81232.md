## Summary

Keep open: current main has an internal Discord `fetchMessage` runtime path, but the agent-facing message tool still does not expose a public `fetch`/`get` action or URL alias, so the requested feature remains actionable.

## What This Changes

Requested Discord message tool support for fetching one specific message by channelId and messageId or by Discord message URL.

## Best Possible Solution

Expose the existing Discord fetchMessage backend as a public Discord message action, accepting channelId+messageId or a Discord message URL while preserving existing messages gating and permission behavior.

## Reproduction Assessment

Yes, by source inspection: the public message action enum, Discord discovery, and agent-facing handler do not expose fetch/get, while the internal fetchMessage runtime already exists. No live Discord setup is needed to confirm the tool-surface gap.

## Solution Assessment

Yes, the narrow maintainable fix is to map a public Discord message-tool fetch action onto the existing fetchMessage REST helper instead of duplicating REST logic or changing broad read semantics.
