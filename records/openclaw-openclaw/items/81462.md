## Summary

Keep open: the report is recent and concrete, and current main still has a clear source path where a mentioned Telegram group turn is processed while normal final text stays private under the default group visible-reply policy. The latest comment points to the supported `visibleReplies: "automatic"` workaround, but it does not prove that the reported Telegram client incompatible-message placeholder is only configuration or that current main fixed the payload path.

## What This Changes

The issue asks OpenClaw to deliver generated replies for Telegram group @mentions with `requireMention: true` instead of producing a Telegram incompatible-message placeholder while `/send` to the same chat works.

## Best Possible Solution

Capture a live Telegram group mention with redacted config and `sendMessage` parameters, then either document the `visibleReplies: "automatic"` configuration as the answer or patch the confirmed Telegram payload bug without changing the group visibility policy.

## Reproduction Assessment

No: source inspection gives a clear processed-but-private group-final path under the default visible-reply policy, but it does not reproduce the Telegram client incompatible-message placeholder. A live current-main private group probe with redacted send parameters is still needed.

## Solution Assessment

Unclear: setting `messages.groupChat.visibleReplies` to `automatic` is a supported workaround for visible final replies, but treating every @mention as automatic delivery would be a policy change. If captured payloads show a Telegram quote/thread/draft problem, the fix should stay in the Telegram delivery path.
