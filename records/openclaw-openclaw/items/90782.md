## Summary

Keep open: this maintainer/protected PR is still useful and current main does not already move the embedded TUI runtime-plugin cold load out of first send, but the patch warms a workspace that can differ from the actual send path and carries an explicit startup-latency tradeoff.

## What This Changes

The branch prewarms embedded local TUI runtime plugins during history loading and reports a best-effort warmup status to the TUI, with tests for warmup success and failure.

## Best Possible Solution

Align embedded TUI prewarm with the exact workspace/runtime registry used by the first local send, preserve best-effort history rendering on failure, and have maintainers explicitly accept the startup-for-first-send latency tradeoff before merge.

## Reproduction Assessment

Yes for the code path: current main first send still reaches ensureRuntimePluginsLoaded in the embedded runner, and the PR body provides before/after local TUI profile numbers. I did not rerun the live profile in this read-only review.

## Solution Assessment

No, not yet: warming during embedded history is a plausible narrow approach, but the spawnedWorkspaceDir branch does not match the workspace contract used by the first send path. The best fix is to make prewarm and send share the same workspace decision, then re-profile the intended local TUI flow.
