## Summary

Keep open: this maintainer-labeled PR is still necessary because current main has no embedded TUI runtime prewarm, and the branch targets the measured first-submit cold plugin activation path. I found no blocking correctness or security issue, but the change intentionally moves plugin activation into initial TUI history load, so maintainers should accept that startup tradeoff before merge.

## What This Changes

The branch adds an optional embedded TUI backend runtime-prewarm hook, starts it during the first history load, blocks submits with a `warming runtime` busy state, and adds focused TUI tests.

## Best Possible Solution

Land this only if maintainers accept the startup-for-first-submit latency tradeoff, keep the prewarm scoped to embedded local TUI, and align the blocked-submit message with the warmup state.

## Reproduction Assessment

Yes, source inspection shows current main reaches `ensureRuntimePluginsLoaded` during embedded agent run startup, and the PR body includes before/after local TUI profile timings for that path. I did not rerun the local CPU profile in this read-only review.

## Solution Assessment

Yes, with a small UX caveat: using the existing `ensureRuntimePluginsLoaded` helper inside the embedded TUI backend is the narrowest owner-boundary fix for the measured local runtime-plugin spike, while the related harness prewarm path does not cover the built-in runtime case described here.
