## Summary

Current main does not already solve the measured local TUI first-send runtime-plugin spike, and the related prewarm stack is not a viable superseding merge target because it remains open/draft on a closed base. Keep this PR open for maintainer review because it is a coherent small patch, but it intentionally moves runtime plugin activation into initial history load and carries an availability tradeoff.

## What This Changes

The PR adds an optional embedded TUI backend runtime-prewarm hook, starts runtime plugin warmup during first history load, blocks submits while `warming runtime` is active, and adds focused TUI tests.

## Best Possible Solution

Keep the direct embedded TUI prewarm path only if maintainers accept the startup-readiness tradeoff, and fix the warmup-specific blocked-submit copy before or during merge if they want a polished UX.
