## Summary

Keep this open: current main still exposes `appServer.codeModeOnly`, forwards it as `features.code_mode_only=true`, and only bridges OpenClaw dynamic tools through `item/tool/call`; the merged compatibility work proves a nested dynamic-tool path, but this report uniquely covers native Codex command execution closing before `turn/completed`.

## What This Changes

The issue reports native Codex app-server command execution closing before completion when `features.code_mode_only=true` is enabled.

## Best Possible Solution

Land a narrow Codex app-server fix that preserves the compatible default while proving `appServer.codeModeOnly=true` either completes native command execution or is clearly gated as unsupported for command-capable turns.

## Reproduction Assessment

Yes. The issue gives a focused `openclaw agent` smoke that fails with `appServer.codeModeOnly=true` and succeeds when only `features.code_mode_only` is stripped, and current source confirms the flag still reaches Codex; I did not run the live Codex app-server command in this read-only pass.

## Solution Assessment

No patch is proposed here. The workaround of stripping `code_mode_only` is reasonable for affected users, but the repository fix should first verify the Codex app-server contract and then add live command-execution proof, not only dynamic-tool bridge proof.
