## Summary

Keep this open: current main still treats `appServer.codeModeOnly` as a supported Codex app-server option, forwards it into Codex thread config, and has no handling for the raw `custom_tool_call` path described by the reporter. The report is fresh, detailed, and points at a real advanced runtime path rather than stale or already-implemented behavior.

## What This Changes

The issue reports native Codex app-server command turns ending with `codex app-server client closed before turn completed` when OpenClaw sends `features.code_mode_only=true`.

## Best Possible Solution

Keep the issue open and land a narrow Codex app-server fix that preserves `code_mode=true` command execution while either correctly bridging raw `custom_tool_call` in `codeModeOnly` or gating that option with clear docs and live proof.

## Reproduction Assessment

Yes, there is a high-confidence reproduction path in the issue: run the provided `openclaw agent` smoke with `appServer.codeModeOnly=true` and compare it to the same run with only that flag stripped. I did not execute a live Codex app-server probe in this read-only review, so the status is source-reproducible rather than locally reproduced.

## Solution Assessment

No patch is proposed here. The best fix is likely narrow, but the implementation should first verify the Codex app-server `code_mode_only` contract and then add regression coverage for command execution, not only dynamic-tool registration.
