## Label Justifications

- `impact:message-loss`: The report says Telegram turns that need command/tool execution fail instead of delivering the expected reply.
- `impact:session-state`: The failure occurs inside an app-server thread/turn completion path, leaving the session outcome incomplete.

## Summary

Current main fixes the reported Codex app-server failure: the compatible default no longer forces code-mode-only, and the explicit code-mode-only path has a live harness probe showing OpenClaw dynamic tools complete and return a final assistant reply.

## What This Changes

The issue reports Codex app-server tool turns closing before completion when OpenClaw sends `features.code_mode_only=true`, with a wrapper that strips only that flag restoring command execution.

## Best Possible Solution

Keep the current main Codex harness behavior: code mode remains enabled, code-mode-only is opt-in and still uses the OpenClaw dynamic-tool bridge, and users should upgrade to a build containing the fix.

## Reproduction Assessment

Yes for the affected beta path: this issue gives a concrete command, error, wrapper workaround, and direct app-server probe, and the related closed report has matching live evidence. I did not rerun a live Codex turn in this read-only review, but current main changes the implicated source path and adds live harness coverage.

## Solution Assessment

Yes. Current main preserves the compatible default while proving explicit code-mode-only dynamic-tool calls through the existing app-server `item/tool/call` bridge instead of relying on the local wrapper workaround.
