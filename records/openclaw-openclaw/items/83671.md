## Label Justifications

- `P2`: The failure affects an advanced Codex app-server opt-in path with a clear workaround and limited blast radius, but it breaks command/tool turns when enabled.
- `impact:crash-loop`: The reported symptom is the Codex app-server client closing before turn completion, leaving the command-execution path unavailable for that run.

## Summary

Keep this open: current main fixed the default forced-`code_mode_only` regression, but it still exposes an explicit `appServer.codeModeOnly` opt-in, forwards that flag into Codex thread config, and has no source-visible handler for the raw `custom_tool_call` command path described by the reporter.

## What This Changes

Investigate or fix native Codex app-server command turns closing before completion when `features.code_mode_only=true` is enabled.

## Best Possible Solution

Keep this as the canonical deeper code-mode-only command-execution follow-up while preserving the current compatible default of `codeModeOnly=false`; support opt-in code-mode-only only after live command proof or clearly gate/document the unsupported path.

## Reproduction Assessment

Yes, at source level: enabling `appServer.codeModeOnly` still forwards `features.code_mode_only=true`, and the reporter gives a live before/after command smoke plus direct app-server probe. I did not run a live current-main Codex app-server command test in this read-only review.

## Solution Assessment

Unclear; stripping `code_mode_only` while keeping `code_mode` is a reasonable compatibility workaround, but the maintainable fix should first verify the upstream Codex app-server raw `custom_tool_call` contract and then either bridge it or gate/document the opt-in mode.
