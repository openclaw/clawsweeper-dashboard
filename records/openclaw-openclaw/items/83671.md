- `P2`: The issue affects an advanced explicit Codex app-server setting with a clear workaround, but it can break command/tool turns for users who opt into that mode.
- `impact:crash-loop`: The reported failure is the Codex app-server/client turn closing or stalling before completion, leaving the command path unavailable for that run.

## Summary

Keep this open: current `main` fixed the default regression and has live proof for code-mode-only OpenClaw dynamic tools, but this report is a newer, narrower failure for explicit `codeModeOnly` native command execution. The source still forwards the opt-in flag, and the current bridge evidence covers `item/tool/call` plus raw `custom_tool_call_output` diagnostics, not execution of the raw `custom_tool_call` path the reporter observed.

## What This Changes

Ensure Codex app-server native command execution completes when `features.code_mode_only` is enabled, or clearly gate/document that unsupported opt-in behavior.

## Best Possible Solution

Preserve the compatible default, then either make explicit `appServer.codeModeOnly=true` complete real native command/tool turns with live proof or narrow the config/docs so users do not opt into a broken command surface.

## Reproduction Assessment

Yes. The issue gives an exact live smoke command and before/after wrapper result, and current source still forwards explicit `codeModeOnly` while lacking an evident raw `custom_tool_call` execution bridge; I did not run the live Codex probe in this read-only sweep.

## Solution Assessment

No patch is proposed here. The best fix needs Codex harness owner judgment after verifying the upstream code-mode-only command contract, then focused unit coverage plus a live command smoke rather than only dynamic-tool coverage.
