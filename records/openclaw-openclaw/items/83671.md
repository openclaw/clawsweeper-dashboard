## Summary

Keep this open as the follow-up to the earlier compatibility hotfix: current main no longer forces `code_mode_only` by default, and the merged opt-in work proves an OpenClaw dynamic-tool path, but it does not prove the reporter's native Codex command smoke (`pwd`/bash-style execution) completes when `features.code_mode_only=true`. The report is fresh, detailed, and still points at a real supported opt-in surface rather than stale behavior.

## What This Changes

The issue asks whether Codex app-server command execution should complete with `features.code_mode_only=true`, and reports that stripping only that flag makes the same `pwd` smoke test pass.

## Best Possible Solution

Make `appServer.codeModeOnly=true` either complete native command/tool turns with live proof covering a shell command, or clearly gate/document that unsupported path while preserving the compatible default where code-mode-only stays off.

## Reproduction Assessment

Yes, with source-level confidence rather than a local live run: the issue provides a concrete command smoke, before/after wrapper behavior, and direct app-server probe, and current main still has an explicit path to send `features.code_mode_only=true`. I did not execute a live Codex 0.130.0 command turn in this read-only review.

## Solution Assessment

Unclear because no patch is proposed and the remaining behavior was not covered by the merged dynamic-tool live proof. The maintainable fix should first reproduce the native command path, then either support it under `codeModeOnly` or narrow the documented support contract.
