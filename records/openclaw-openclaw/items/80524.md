## Summary

Keep open: current main and v2026.5.7 still take the ack-only path for bare chat `/reset`, while older history shows the startup-turn behavior existed before it was replaced by documented and tested ack-only behavior. Restoring the prior Telegram UX needs maintainer product direction because it now conflicts with current docs and regression tests.

## What This Changes

Restore or make configurable the pre-v2026.5.7 Telegram bare `/reset` behavior that launched a startup agent turn instead of only acknowledging the reset.

## Best Possible Solution

Decide the intended bare chat reset contract; if startup turns should return, add a compatible reset-startup path or explicit opt-in and update reset docs plus shared, native, and Telegram coverage together.

## Reproduction Assessment

Yes. Telegram native `/reset` source-reproduces through the shared command pipeline, where an empty reset tail returns `shouldContinue: false` and the ack before prepared reply/model execution.

## Solution Assessment

Unclear. The suggested fallthrough is mechanically plausible, but it would contradict current docs and tests that deliberately make bare chat resets ack-only; a maintainer-approved contract change or opt-in is safer.
