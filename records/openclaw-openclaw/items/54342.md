## Summary

Keep this open. Current main has adjacent task/status tracking and hook `deliver: false`, but the central channel-less persistent ACP workflow is still blocked: `mode: "session"` still requires `thread: true`, accepted spawn results still do not expose a stable ACP resume identity, and `/hooks/agent` still runs through isolated cron handling that drops ACP metadata before the normal ACP `runTurn` dispatch path can apply.

## What This Changes

Request to let hook-triggered agents track, resume, and message ACP child sessions without requiring channel thread bindings or duplicate hook auto-delivery.

## Best Possible Solution

Define a first-class headless/unbound persistent ACP contract with explicit list/status/close/retention behavior, return an authorized stable resume identity or opaque resume token from spawn/status surfaces, and document whether follow-up turns should use `sessions_send` or hook-side ACP dispatch under the existing hook session-prefix and ownership policies. Keep `deliver: false` as the hook auto-delivery suppression mechanism.

## Reproduction Assessment

Yes. A hook-triggered agent that calls `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still hits the `thread_required` guard, and a `/hooks/agent` request aimed at an ACP session key still goes through isolated cron session creation that drops ACP metadata before `runTurn` can be selected.

## Solution Assessment

No, simply removing the `thread: true` guard is not the best fix. The maintainable fix needs an explicit headless ACP lifecycle and a safe resume-identity/dispatch authorization model before implementation.

## Risks / Open Questions

- Allowing unbound persistent ACP sessions without a defined lifecycle could leave sessions orphaned, invisible, or retained beyond the intended idle policy.
- Exposing ACP backend or agent resume identifiers needs an ownership boundary so one agent cannot infer or resume unrelated harness sessions.
- Routing hook-supplied ACP session keys into `runTurn` must respect hook token, session-prefix, target-agent, and requester-ownership policy; otherwise a webhook could become a cross-session injection path.
