## Summary

Keep open: current main partially addresses ACP run tracking and hook auto-delivery suppression, but the channel-less persistent ACP workflow is still blocked by the explicit `mode: "session"` thread guard and hook dispatch still uses isolated cron rather than a guarded ACP turn path.

## What This Changes

The issue requests ACP child-session tracking, headless persistent ACP sessions for hook-triggered agents, and hook delivery control to avoid duplicate Telegram/topic messages.

## Best Possible Solution

Define a first-class headless persistent ACP lifecycle with explicit status/list/close/retention semantics, a safe resume identity or opaque token, and an authorized follow-up path through `sessions_send` or guarded hook ACP dispatch.

## Reproduction Assessment

Yes. `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still reaches the `thread_required` guard without `thread: true`, and `/hooks/agent` still enters isolated cron execution rather than a direct ACP `runTurn` path.

## Solution Assessment

No. Removing the thread guard alone is not the best complete fix; the maintainable path needs explicit headless ACP lifecycle semantics plus safe resume-identity and hook/session-key authorization rules.
