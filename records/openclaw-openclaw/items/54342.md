## Summary

Keep open: current main has partial progress for ACP task tracking and hook delivery suppression, but the channel-less persistent ACP workflow is still blocked by the explicit `mode: "session"` thread guard and hook runs still go through the isolated cron/embedded-agent path.

## What This Changes

The issue requests hook-triggered ACP child sessions that can be tracked, resumed, followed up without channel thread bindings, and delivered without duplicate hook auto-delivery.

## Best Possible Solution

Define a first-class headless persistent ACP lifecycle with explicit status/list/close/retention behavior, a safe resume identity or opaque token, and authorized follow-up delivery through `sessions_send` or guarded hook ACP dispatch.

## Reproduction Assessment

Yes, source-reproducible. `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still reaches the `thread_required` guard without `thread: true`, and hook agent dispatch still enters isolated cron execution rather than a direct ACP turn path.

## Solution Assessment

No, simply removing the thread guard is not the best complete fix. The maintainable path needs explicit headless ACP lifecycle semantics plus safe resume-identity and hook/session-key authorization rules.
