## Summary

Keep this open. Current main has added adjacent ACP task tracking, `session_status` task visibility, `sessions_send`, and hook `deliver: false`, but the central headless persistent ACP workflow is still blocked by the `mode: "session"` thread requirement, accepted spawn results still do not expose a harness resume identity, and `/hooks/agent` still runs through isolated cron execution rather than directly dispatching ACP session keys through ACP `runTurn`.

## What This Changes

The issue requests reliable tracking and continuation for ACP sessions spawned by hook-triggered agents, including poll/status visibility, unbound persistent ACP sessions, ACP resume identity exposure, hook-side ACP follow-up dispatch, and suppression of duplicate hook delivery.

## Best Possible Solution

Define an explicit headless/unbound persistent ACP contract, expose stable ACP resume identity through spawn/status surfaces with access controls, document whether follow-up turns should use `sessions_send` or hook-side ACP dispatch, and add focused regression coverage for hook-triggered channel-less ACP workflows while retaining `deliver: false` for hook auto-delivery suppression.
