## Summary

Keep open: current main and the latest release cover ACP task/status tracking and hook fallback-delivery suppression, but they still reject channel-less persistent ACP session spawns. The broader decoupling work in https://github.com/openclaw/openclaw/issues/53548 and https://github.com/openclaw/openclaw/pull/39372 is relevant, but it does not fully settle this issue's hook-triggered ACP lifecycle, resume identity, and authorized follow-up routing concerns.

## What This Changes

The issue requests trackable ACP spawns from hook-triggered agents, persistent ACP sessions without channel thread binding, and a way to avoid duplicate hook delivery when the agent uses the message tool.

## Best Possible Solution

Define a first-class headless persistent ACP lifecycle with explicit status/list/close/retention semantics, safe opaque resume identity handling, and an authorized follow-up path through `sessions_send` or guarded hook ACP dispatch.

## Reproduction Assessment

Yes. Source inspection shows `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still reaches the `thread_required` guard without `thread: true`, and hook-triggered runs still go through isolated cron execution rather than a first-class hook-to-ACP dispatch path.

## Solution Assessment

No as a simple guard removal. The maintainable solution needs explicit headless ACP lifecycle, delivery, ownership, and resume-identity rules rather than only making `mode: "session"` bypass the current thread check.
