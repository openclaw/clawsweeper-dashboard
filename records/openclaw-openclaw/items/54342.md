## Summary

Keep open: current main has adjacent progress for ACP task tracking, `sessions_send`, and hook `deliver: false`, but the channel-less persistent ACP workflow remains blocked by the `mode: "session"` thread guard and hook dispatch still runs through isolated cron rather than a guarded ACP turn path.

## What This Changes

This issue requests tracked/resumable ACP child sessions from hook-triggered agents, unbound persistent ACP sessions, and hook delivery control to avoid duplicate or misrouted Telegram/topic messages.

## Best Possible Solution

Define a first-class headless persistent ACP lifecycle with explicit status/list/close/retention behavior, return an authorized stable resume identity or opaque token, and route follow-up turns through `sessions_send` or a guarded hook ACP-dispatch path without requiring channel thread bindings.

## Reproduction Assessment

Yes, from source inspection. `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still reaches the `thread_required` guard without `thread: true`, and `/hooks/agent` still enters isolated cron execution rather than an ACP `runTurn` dispatch path.

## Solution Assessment

No, removing the guard alone is not the best fix. The maintainable solution needs an explicit headless ACP lifecycle plus safe resume-identity and hook/session-key authorization semantics.
