## Summary

Keep open: current main and v2026.5.18 still reject threadless persistent ACP spawns from `sessions_spawn`, while newer code only partially addresses task/status visibility, resume ownership, `sessions_send` ownership handling, and hook fallback-delivery suppression.

## What This Changes

The issue requests tracked, persistent, threadless ACP sessions from hook-triggered agents, plus safe follow-up routing and duplicate-delivery control.

## Best Possible Solution

Define a first-class headless persistent ACP lifecycle with explicit status/list/close/retention semantics, safe opaque resume identity handling, and an authorized follow-up path through `sessions_send` or guarded hook ACP dispatch.

## Reproduction Assessment

Yes. Source inspection shows `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still reaches the `thread_required` guard without `thread: true`, while the latest release contains the same guard.

## Solution Assessment

No as a simple guard removal. The maintainable fix needs explicit headless ACP lifecycle, delivery, ownership, and resume-identity rules rather than only bypassing the current thread check.
