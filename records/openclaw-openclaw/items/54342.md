## Summary

Keep open: current main has addressed ACP run-mode task completion/status and hook fallback-delivery suppression, but the requested channel-less persistent ACP workflow is still blocked by the explicit `mode="session"` thread guard and needs a deliberate lifecycle/API decision.

## What This Changes

The issue requests ACP spawn tracking for hook-triggered agents, unthreaded persistent ACP sessions with resume identity, and hook delivery controls that avoid duplicate Telegram topic posts.

## Best Possible Solution

Define a first-class headless persistent ACP lifecycle with explicit status/list/close/retention semantics, safe opaque resume identity handling, and an authorized follow-up path through `sessions_send` or guarded hook ACP dispatch.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `sessions_spawn({ runtime: "acp", mode: "session", streamTo: "parent" })` still reaches the `thread_required` guard without `thread: true`, and `/hooks/agent` still runs through isolated cron rather than an ACP `runTurn` route.

## Solution Assessment

No. Removing the guard alone is not the best maintainable fix; the safer solution needs explicit headless ACP lifecycle semantics plus resume-identity and hook/session-key authorization rules.
