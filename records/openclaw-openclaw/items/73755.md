## Summary

Keep open: this PR is a useful, scoped fix for the serve-mode Funnel overwrite path, but it still needs real Tailscale behavior proof and has one concrete matching gap for a Tailscale-supported Proxy scheme.

## What This Changes

The PR adds opt-in `gateway.tailscale.preserveFunnel` config, Tailscale Funnel status parsing, gateway startup wiring, docs/schema/changelog updates, and unit coverage for preserving external Funnel routes.

## Best Possible Solution

Land this opt-in preserveFunnel direction only after it recognizes all relevant Tailscale loopback Proxy schemes and the contributor adds after-fix live or terminal proof from a real Tailscale setup.

## Reproduction Assessment

Yes by source inspection. Current main's serve-mode startup always calls `enableTailscaleServe`, and that helper runs `tailscale serve --bg --yes <port>`; I did not run a live tailnet restart because it would mutate host Tailscale state.

## Solution Assessment

No, not yet. The opt-in status precheck is the right narrow shape, but the current Proxy normalization misses Tailscale's supported `https+insecure` scheme and the external PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Normalize Tailscale https+insecure Proxy targets:** `src/infra/tailscale.ts:441`
  - body: Tailscale accepts `https+insecure://localhost:<port>` as an HTTP(S) Serve/Funnel proxy target, but this normalization strips only `http://` and `https://`. For an externally managed Funnel route using that supported scheme, `preserveFunnel` returns false and startup still runs `tailscale serve`, overwriting the route this PR is meant to preserve.
  - confidence: 0.87
