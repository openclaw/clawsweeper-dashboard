## Summary

Keep open: current main still lacks a preserve-Funnel path, and this PR is aimed at the right linked bug, but the latest head still misses a documented Tailscale proxy target form and the after-fix proof is shim-based rather than a real Tailscale setup.

## What This Changes

The PR adds optional `gateway.tailscale.preserveFunnel` config, schema/docs/changelog updates, and gateway/infra tests so serve mode can skip reapplying Serve when an existing Funnel route covers the gateway port.

## Best Possible Solution

Normalize Tailscale proxy target parsing for all documented loopback target schemes, keep config/docs/tests aligned, and land after real Tailscale proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, by source inspection. Current main's serve-mode startup always calls `enableTailscaleServe`, and that helper runs `tailscale serve --bg --yes <port>`; I did not run a live tailnet restart because this review is read-only and that would mutate host Tailscale state.

## Solution Assessment

No, not yet. The opt-in precheck is a narrow maintainable shape, but the parser misses a documented `https+insecure` target form and the PR still needs real Tailscale behavior proof or maintainer override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize https+insecure proxy targets:** `src/infra/tailscale.ts:441`
  - body: Tailscale documents `https+insecure://localhost:<port>` as a valid Serve/Funnel target, but this normalization strips only `http://` and `https://`. For that stored proxy value, `stripped` becomes `https+insecure:` before host/port parsing, so `preserveFunnel` returns false and startup still runs `tailscale serve`, overwriting the route this PR is meant to preserve. ([tailscale.com](https://tailscale.com/docs/reference/tailscale-cli/serve))
  - confidence: 0.9
