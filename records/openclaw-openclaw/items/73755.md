## Summary

Keep open for maintainer review: current main still reapplies Tailscale Serve on every serve-mode startup, and this PR appears to address the external Funnel clobber path without a discrete blocker found in this read-only pass.

## What This Changes

The PR adds opt-in `gateway.tailscale.preserveFunnel` config, schema/docs/changelog updates, and gateway/Tailscale tests so serve-mode startup can skip `tailscale serve` when an external Funnel route already covers the gateway port.

## Best Possible Solution

Land a scoped preserveFunnel fix after maintainer review and exact-head checks, while keeping OpenClaw-managed Funnel password-only behavior separate from the broader token-auth Funnel policy discussion.

## Reproduction Assessment

Yes. Current main source-reproduces the overwrite path through `startGatewayTailscaleExposure` to `enableTailscaleServe`, which runs `tailscale serve --bg --yes <port>` without checking existing Funnel state.

## Solution Assessment

Yes. The opt-in status precheck is a narrow maintainable fix for externally managed Funnel routes because it only skips reapplying Serve when a public Funnel route already covers the gateway port and leaves managed Funnel auth policy unchanged.
