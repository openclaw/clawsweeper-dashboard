## Summary

Keep open. Current main still reapplies Tailscale Serve on every serve-mode startup and has no preserve-Funnel path, while this PR addresses the linked report with a focused opt-in startup check, config/schema/docs updates, and regression coverage; I found no blocking correctness issue in the latest diff.

## What This Changes

The branch adds opt-in `gateway.tailscale.preserveFunnel`, wires it through Gateway startup and config/docs surfaces, adds Tailscale Funnel status parsing with unit tests, and updates the changelog.

## Best Possible Solution

Land the opt-in preserve-Funnel behavior with the config/schema/docs/test coverage intact, while keeping OpenClaw-managed Funnel password-only and leaving externally managed Funnel routes owned by the operator.

## Reproduction Assessment

Yes. Current main statically reproduces the overwrite path because serve-mode startup always calls `enableTailscaleServe(params.port)`, and that helper runs `tailscale serve --bg --yes <port>`; the linked issue supplies the live restart scenario and this PR adds focused unit coverage for the preserve-hit and preserve-miss cases.

## Solution Assessment

Yes. The opt-in serve-mode precheck is the narrowest maintainable fix for token-auth users who manage Funnel externally, because it preserves an already-public Funnel route without relaxing the password-only policy for OpenClaw-managed Funnel mode.
