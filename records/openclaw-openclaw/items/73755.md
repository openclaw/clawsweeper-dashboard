## Summary

Keep this PR open. Current main still re-applies Tailscale Serve in serve mode with no existing-Funnel precheck, and the PR is a narrow opt-in fix for the linked regression; the remaining issue I found is a small public-docs gap for the new config key.

## What This Changes

The PR adds opt-in `gateway.tailscale.preserveFunnel` handling, a Tailscale Funnel status parser/precheck, config schema/help/baseline updates, regression tests, a changelog entry, plus a few incidental formatting changes and an added `trusted: true` system-event argument.

## Best Possible Solution

Land the PR after adding the public docs note for the new opt-in and passing the normal focused gateway/config checks, keeping OpenClaw-managed Funnel password-only while allowing token-auth operators to preserve externally managed Funnel routes.

## Reproduction Assessment

Yes. The high-confidence reproduction is `gateway.tailscale.mode="serve"`, an externally configured Funnel route for the gateway port, then a gateway restart; current main statically re-runs `tailscale serve --bg --yes <port>` with no preservation check.

## Solution Assessment

Yes, with the docs follow-up. The opt-in precheck is narrower and safer than changing Funnel auth policy, and it preserves default behavior for existing deployments.
