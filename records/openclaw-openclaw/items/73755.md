## Summary

Keep open. Current main still reapplies Tailscale Serve on every serve-mode startup, and this PR is the right narrow feature shape, but the latest parser still misses a Tailscale-supported proxy target form, so it can fail to preserve some external Funnel routes.

## What This Changes

The PR adds opt-in `gateway.tailscale.preserveFunnel`, Tailscale Funnel-status parsing, config schema/docs, changelog coverage, and regression tests so serve-mode startup can skip reapplying Serve when an external Funnel route already covers the gateway port.

## Best Possible Solution

Make the Funnel-status parser scheme-aware for all Tailscale-supported loopback HTTP handler proxy targets, add the missing regression fixture, and then land the opt-in preserve-Funnel path without changing the password-only OpenClaw-managed Funnel policy.

## Reproduction Assessment

Yes. Current main statically reproduces the overwrite path through `startGatewayTailscaleExposure` calling `tailscale serve --bg --yes <port>`; the remaining PR bug reproduces with a `Proxy: "https-insecure://127.0.0.1:18789"` Funnel-status fixture.

## Solution Assessment

No, not quite. The opt-in serve-mode precheck is the narrow maintainable solution, but the current proxy matching should parse Tailscale-supported schemes instead of only stripping `http` and `https`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle Tailscale https-insecure proxy targets:** `src/infra/tailscale.ts:441-454`
  - body: The matcher only strips `http://` and `https://` before extracting host and port. Tailscale supports HTTP handler proxy targets like `https-insecure://localhost:3000/foo`, so that value leaves `https-insecure://127.0.0.1` as the host, fails the loopback check, and makes `preserveFunnel` rerun `tailscale serve`, clobbering the external Funnel route. Normalize all supported proxy schemes or parse the target as a URL, and add a regression fixture for this form.
  - confidence: 0.86
