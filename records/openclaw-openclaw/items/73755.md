## Summary

Keep this PR open: current main still unconditionally reapplies Tailscale Serve in serve mode, but the proposed fix is not merge-ready because its proxy normalization still misses a supported Tailscale Funnel backend form and the after-fix proof uses a local Tailscale shim rather than a real tailnet setup.

## What This Changes

The PR adds opt-in `gateway.tailscale.preserveFunnel`, checks `tailscale funnel status --json` before `tailscale serve` in serve mode, wires config/schema/docs/changelog, and adds gateway/infra unit coverage.

## Best Possible Solution

Update the parser to recognize every Tailscale loopback proxy form the CLI can persist, add regression coverage for the missed scheme, and land only after real Tailscale proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, by source inspection. Current main's serve-mode startup always calls `enableTailscaleServe`, and that helper runs `tailscale serve --bg --yes <port>`; I did not run a live tailnet restart because this review is read-only and that would mutate host Tailscale state.

## Solution Assessment

No, not yet. The opt-in precheck is the right narrow shape, but the current normalization misses `https+insecure` proxy targets and the external PR still needs real Tailscale behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Normalize https+insecure proxy targets:** `src/infra/tailscale.ts:441`
  - body: Tailscale can persist Funnel/Serve backends as `https+insecure://localhost:<port>`, but this normalization strips only `http://` and `https://` before trimming at the first slash. That leaves `https+insecure:` instead of `localhost:<port>`, so `preserveFunnel` returns false and startup still runs `tailscale serve`, overwriting the route this PR is meant to preserve.
  - confidence: 0.88
