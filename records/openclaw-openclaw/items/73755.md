## Summary

Keep open: the PR targets a real current-main gateway/Tailscale overwrite path, but it is not merge-ready because the parser misses a documented Tailscale target scheme, the required changelog entry is absent, proof is shim-based rather than real Tailscale, and the new public Funnel preservation knob needs maintainer policy review.

## What This Changes

The PR adds `gateway.tailscale.preserveFunnel`, startup logic to skip `tailscale serve` when Funnel already covers the port, Tailscale status parsing, config/docs updates, and unit tests.

## Best Possible Solution

Land a guarded opt-in preservation implementation after normalizing documented Tailscale target schemes, adding the changelog entry, and getting real-tailnet proof or an explicit maintainer proof override for the exposure-policy choice.

## Reproduction Assessment

Yes, by source inspection. Current main always runs `tailscale serve --bg --yes <port>` in serve mode, and Tailscale documents that the most recent Serve/Funnel command controls same-port public/private state.

## Solution Assessment

No, not yet. The opt-in precheck is a narrow maintainable direction, but it must handle all documented loopback target schemes, include the changelog, and get real Tailscale proof or maintainer override before merge.
