## Summary

Keep this PR open: the approval-flow polish is plausible, but the current branch still has a source-reproducible event-history regression, is conflicted with current main, and lacks required real behavior proof.

## What This Changes

The PR renames Control UI approval buttons, filters resolved approvals by the server-returned id, logs a local approval-resolution event, and refreshes Overview after resolving an approval.

## Best Possible Solution

Rebase onto current main, keep the useful approval-flow polish, record resolution history through the canonical gateway event buffer or existing resolved broadcasts, route label changes through the English locale source, and add real runtime proof.

## Reproduction Assessment

Yes. Source inspection shows the PR writes `approval-resolved` only to `eventLog`, while current main later syncs `eventLog` from `eventLogBuffer`, dropping that local-only entry.

## Solution Assessment

No. The current patch bypasses the established event-history source; the narrower maintainable fix is to rely on canonical `*.approval.resolved` gateway events or append through the shared capped buffer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Write approval logs through the canonical event buffer:** `ui/src/ui/app.ts:760-764`
  - body: `eventLog` is not the durable event-history source: gateway events are stored in `eventLogBuffer`, and Debug/Overview sync paths copy that buffer back into `eventLog`. This new `approval-resolved` entry is pushed only into `eventLog`, so it can disappear on the next event-log sync. Route it through the buffer path too, or rely on the existing gateway `*.approval.resolved` broadcasts.
