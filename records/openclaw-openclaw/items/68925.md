## Summary

Keep open: the approval-flow polish is plausible, but the current PR patch still writes its new synthetic approval log only to visible `eventLog`, so the entry can disappear when the UI syncs from the canonical `eventLogBuffer`. The PR also has no after-fix real behavior proof beyond a local build note.

## What This Changes

The PR branch renames Control UI approval buttons, removes resolved approval entries after a resolve request, adds a local `approval-resolved` log/DOM event, and refreshes overview state after a decision.

## Best Possible Solution

Keep the approval-flow polish only if it is rebased onto current main, logs through the canonical gateway event buffer or existing resolved broadcasts, and routes any copy changes through locale keys with focused UI coverage.

## Reproduction Assessment

Yes, from source inspection. With the PR patch, resolve an approval and then trigger a Debug or Overview event-log sync; the synthetic `approval-resolved` entry is absent from `eventLogBuffer` and can be overwritten.

## Solution Assessment

No. The patch bypasses the established event-history source; the narrower maintainable solution is to rely on canonical `*.approval.resolved` gateway events or append through a shared capped buffer path, then rebase the button copy through i18n.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Write approval logs to the canonical event buffer:** `ui/src/ui/app.ts:760-764`
  - body: `eventLog` is not the durable event history: gateway events are stored in `eventLogBuffer`, and Debug/Overview sync paths copy that buffer back into `eventLog`. This new `approval-resolved` entry is pushed only into `eventLog`, so it can disappear on the next event-log sync. Route it through the buffer path too, or rely on the existing gateway `*.approval.resolved` broadcasts.
