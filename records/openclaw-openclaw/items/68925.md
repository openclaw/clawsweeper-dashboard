## Summary

Keep open: the branch still has a source-reproducible event-log correctness issue, lacks contributor-supplied real approval-flow proof, and is currently conflicted with main.

## What This Changes

The PR changes Control UI approval handling by renaming approval action labels, filtering resolved approvals by the returned id, appending a local resolution log/event, and refreshing overview after a decision.

## Best Possible Solution

Keep the approval-flow polish only after rebasing, recording resolution history through the canonical gateway event buffer or existing resolved broadcasts, and routing copy changes through the English locale source with focused UI coverage.

## Reproduction Assessment

Yes. From source inspection, resolving an approval with the PR patch adds a synthetic entry only to `eventLog`, then a Debug or Overview event-log sync copies `eventLogBuffer` over it and drops that entry.

## Solution Assessment

No. The patch bypasses the established event-history source; the maintainable path is to rely on canonical `*.approval.resolved` gateway events or append through the shared capped buffer path, then rebase button copy through i18n.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Write approval logs through the canonical event buffer:** `ui/src/ui/app.ts:760-764`

Route it through the buffer path too, or rely on the existing gateway `*.approval.resolved` broadcasts.
