The PR updates Control UI approval decisions by changing action labels, filtering resolved approvals with the RPC result, adding a local resolution log/event, refreshing overview state, and dispatching an approval-resolved DOM event.

## Best Possible Solution

Keep the approval-flow polish after rebasing onto current i18n/modal UI, recording resolution history through canonical gateway resolved events or the shared event buffer, and adding real approval-resolution proof.

## Reproduction Assessment

Yes, from source inspection: the PR head writes `approval-resolved` only to `eventLog`, while current main treats `eventLogBuffer` as canonical and overwrites `eventLog` during sync paths. I did not run local tests because this was a read-only review.

## Solution Assessment

No. The maintainable path is to preserve the current localized approval view and route resolution history through existing gateway resolved events or the shared event buffer instead of adding a local-only log entry.
