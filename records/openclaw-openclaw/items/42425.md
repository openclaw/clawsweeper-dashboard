## What This Changes

The PR adds a startup-specific multi-workspace internal hook loader, wires Gateway startup to enumerate agent workspaces, and expands hook loader regression tests.

## Best Possible Solution

Land a rebased replacement in src/gateway/server-startup-post-attach.ts that enumerates agent workspaces while reusing current hook discovery, opt-in, cleanup, source precedence, plugin hook, and fail-closed path contracts.

## Reproduction Assessment

Yes, source-reproducible: the docs define per-agent workspace hooks, while current main's live Gateway startup still calls loadInternalHooks once with params.defaultWorkspaceDir. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

No, not as-is. Multi-workspace loading is the right direction, but this branch must be rebased or replaced against the live post-attach startup path while preserving current hook discovery, opt-in, cleanup, precedence, plugin hook, and path-safety contracts.

  - confidence: 0.9

  - body: Suppressing shared handlers whenever a workspace-local hook has the same name lets workspace code override managed, plugin, or bundled hooks. Current policy only allows workspace hooks to override workspace hooks, so this can disable trusted shared behavior.
  - confidence: 0.89

  - confidence: 0.88

  - confidence: 0.88

  - body: safeRealpathOrResolve falls back to path.resolve before opening and importing hook handlers, whereas current main skips unreadable hook roots after realpath failure and opens under a checked root. Keep the fail-closed path before loading trusted hook code.
