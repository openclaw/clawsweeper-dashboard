## Summary

Keep this PR open, but do not land it as-is. Current main still loads internal hooks from only the default workspace during Gateway startup, so the intended bug fix is not implemented; however, the PR branch is stale against the current startup split and its loader changes would regress current hook configuration and extra-directory policy.

## What This Changes

The PR adds a startup-specific multi-workspace internal hook loader, scopes workspace-local hook handlers by agent workspace, updates Gateway startup to call it, and adds hook-loader regression tests.

## Best Possible Solution

Refresh the fix against `src/gateway/server-startup-post-attach.ts`: enumerate configured agent workspaces with `listAgentWorkspaceDirs`, load shared hooks once, load explicitly enabled workspace-local hooks per workspace, scope local handlers to matching event/session/agent workspace, isolate per-workspace failures, preserve current extraDirs/managed precedence and `hasConfiguredInternalHooks` behavior, and add focused Gateway startup coverage.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: configure two agents with distinct `<workspace>/hooks/` entries, start the Gateway, and only the default workspace can be discovered because startup calls `loadInternalHooks(cfg, defaultWorkspaceDir)` and the loader scans one workspace hooks directory.

## Solution Assessment

No. The intended direction is plausible, but this PR is not the best current solution as-is because it targets the old startup file shape and does not preserve current configured-hook and extraDirs contracts; a rebased implementation in the active post-attach startup path is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve configured hook startup eligibility:** `src/hooks/loader.ts:48-50`
  - body: The new startup loader returns early unless `hooks.internal.enabled` is exactly true. Current main uses `hasConfiguredInternalHooks`, so named enabled entries, hook installs, extraDirs, and legacy handlers can load without that top-level flag; keep that contract or existing hook setups stop loading at startup.
  - confidence: 0.88
- **[P2] Keep extra hook dirs at managed precedence:** `src/hooks/loader.ts:312`
  - body: `hooks.internal.load.extraDirs` is loaded here as `openclaw-workspace`, but current loader/docs treat extra dirs as managed/shared hooks. Marking them as workspace hooks makes them explicit-opt-in and removes their managed override behavior, so configured extra hook directories can stop loading or lose precedence.
  - confidence: 0.9
- **[P2] Wire the active startup module:** `src/gateway/server-startup.ts:113-117`
  - body: This hunk updates the old monolithic startup file, but current main's `src/gateway/server-startup.ts` is only a re-export and active hook loading lives in `src/gateway/server-startup-post-attach.ts`. A rebase needs to move the startup hook call there or the intended behavior will not be present.
  - confidence: 0.86
