## Summary

Keep open: the underlying bug is still source-reproducible on current main, but this PR is conflicting, targets an obsolete startup file, lacks required real Gateway proof, and regresses several current hook loading contracts.

## What This Changes

The PR adds a multi-workspace internal hook loader, wires Gateway startup to it, and expands hook loader regression tests for non-default agent workspace hooks.

## Best Possible Solution

Land a rebased replacement in src/gateway/server-startup-post-attach.ts that enumerates agent workspaces while reusing current hook discovery, policy, registration cleanup, and fail-closed path checks.

## Reproduction Assessment

Yes, by source inspection: the docs define per-agent workspace hooks, while current main's live Gateway startup still calls loadInternalHooks once with params.defaultWorkspaceDir. I did not run a live Gateway reproduction in this read-only review.

## Solution Assessment

No, not as-is. Multi-workspace loading is the right direction, but this branch needs to target the live post-attach startup module and preserve current hook discovery, opt-in, cleanup, precedence, and path-safety contracts.

Overall confidence: 0.91

Full review comments:

- **[P2] Retarget the live post-attach startup path:** `src/gateway/server-startup.ts:22`
  - body: Current main no longer runs this startup module; Gateway post-attach startup now goes through src/gateway/server-startup-post-attach.ts, which still calls loadInternalHooks with only params.defaultWorkspaceDir. A rebase of this patch would not wire the fix into the live path.
  - confidence: 0.95
- **[P2] Preserve configured hook opt-ins:** `src/hooks/loader.ts:54-55`
  - body: The new startup loader returns unless hooks.internal.enabled is truthy, but current main treats enabled entries, installs, extraDirs, and legacy handlers as valid opt-ins through hasConfiguredInternalHooks. Setups that enable hooks without the global flag would stop loading at Gateway startup.
  - confidence: 0.93
- **[P2] Track loader-owned registrations on reload:** `src/hooks/loader.ts:185`
  - body: registerHookEntries calls registerInternalHook without adding the registered wrapper to loadedHookRegistrations. Current loader cleanup depends on that list, so repeated loads can leave duplicate workspace and legacy handlers registered.
  - confidence: 0.9
- **[P2] Keep current hook source precedence:** `src/hooks/loader.ts:442-443`
  - body: Suppressing shared handlers whenever a workspace-local hook has the same name lets workspace code override managed, plugin, or bundled hooks. Current policy and docs only allow workspace hooks to override workspace hooks, so this can disable trusted shared behavior.
  - confidence: 0.89
- **[P2] Keep extra hook dirs at managed precedence:** `src/hooks/loader.ts:312-316`
  - body: hooks.internal.load.extraDirs are currently discovered as openclaw-managed sources and share managed enablement and collision policy. Loading them as openclaw-workspace changes default enablement and precedence for configured extra directories.
  - confidence: 0.88
- **[P2] Include plugin hook dirs in shared discovery:** `src/hooks/loader.ts:329`
  - body: The shared startup discovery rebuilds the source list from extra, bundled, and managed directories but omits resolvePluginHookDirs. Plugin-provided internal hooks would stop loading during Gateway startup.
  - confidence: 0.88
- **[P2] Preserve fail-closed hook path checks:** `src/hooks/loader.ts:203`
  - body: safeRealpathOrResolve falls back to path.resolve before opening and importing hook handlers, whereas current main skips unreadable hook roots after realpath failure and opens under a checked root. Keep the fail-closed path before loading trusted hook code.
  - confidence: 0.84
