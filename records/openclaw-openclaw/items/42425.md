## Summary

Keep this PR open: it targets a real documented bug, but the branch is stale against current main and would regress current hook startup, discovery, and loader contracts before it can merge.

## What This Changes

The PR adds a startup hook loader for all configured agent workspaces, scopes workspace-local handlers, switches Gateway startup to call it, and adds hook loader regression tests.

## Best Possible Solution

Land a rebased or replacement fix in the current post-attach startup path that enumerates configured agent workspaces while preserving hasConfiguredInternalHooks, plugin/managed/extra/workspace precedence, and loader cleanup.

## Reproduction Assessment

Yes, from source inspection. Current main documents per-agent workspace hooks, but Gateway startup still calls loadInternalHooks with only params.defaultWorkspaceDir, so non-default workspace hook directories are not discovered.

## Solution Assessment

No, not as-is. Multi-workspace startup loading is the right narrow direction, but this stale branch targets the old startup file and loses current hook opt-in, discovery, and reload-cleanup contracts.

  - confidence: 0.92

  - confidence: 0.86
- **[P2] Keep extra hook dirs as managed hooks:** `src/hooks/loader.ts:312-316`
  - body: hooks.internal.load.extraDirs currently load with source openclaw-managed and share managed-hook precedence. Marking them openclaw-workspace changes both trust classification and override behavior for configured extra hook directories.
  - confidence: 0.88
- **[P2] Include plugin hook dirs in shared startup loading:** `src/hooks/loader.ts:329`
  - body: The shared startup loader rebuilds discovery from extra, bundled, and managed dirs only. Current hook discovery also includes plugin hook dirs, so switching startup to this helper would stop plugin-bundled internal hooks from loading.
