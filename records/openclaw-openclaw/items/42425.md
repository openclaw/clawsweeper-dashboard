## Summary

Keep open. The underlying bug is still source-reproducible on current main, but this external PR is not mergeable as-is because it targets the old startup module, weakens current hook-loading contracts, touches trusted local code loading, and still lacks real Gateway behavior proof.

## What This Changes

The PR adds a multi-workspace internal-hook startup loader, wires Gateway startup to it, scopes workspace-local handlers, and expands hook-loader tests.

## Best Possible Solution

Land a rebased or replacement fix in src/gateway/server-startup-post-attach.ts that enumerates agent workspaces while reusing current hook discovery, policy, loading, cleanup, and path-safety contracts.

## Reproduction Assessment

Yes, by source inspection. Current main still loads internal hooks once from params.defaultWorkspaceDir while the docs define workspace hooks as per-agent <workspace>/hooks entries.

## Solution Assessment

No, not as-is. Multi-workspace loading is the right direction, but this branch needs to be retargeted to the live startup module and rebuilt around current hook loading contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Retarget the live post-attach startup path:** `src/gateway/server-startup.ts:22`
  - body: Current main no longer has this startup module; Gateway sidecars now start from src/gateway/server-startup-post-attach.ts, which still calls loadInternalHooks with only params.defaultWorkspaceDir. As written, a rebase will not wire the fix into the code that runs.
  - confidence: 0.95
- **[P2] Preserve configured hook opt-ins:** `src/hooks/loader.ts:54-55`
  - body: The new startup loader returns unless hooks.internal.enabled is true, but current main treats enabled entries, installs, extraDirs, and legacy handlers as opt-ins through hasConfiguredInternalHooks. Setups that enable hooks without the global flag would stop loading at Gateway startup.
  - confidence: 0.93
- **[P2] Track loader-owned registrations on reload:** `src/hooks/loader.ts:184-185`
  - body: registerHookEntries calls registerInternalHook without adding the wrapped handler to loadedHookRegistrations. Current loader clears those registrations before reload, so repeated loads would leave duplicate workspace and legacy handlers registered.
  - confidence: 0.9
- **[P2] Keep current hook source precedence:** `src/hooks/loader.ts:442-443`
  - body: Suppressing shared handlers whenever a workspace-local hook has the same name lets workspace code override managed, plugin, or bundled hooks. Current policy and docs only allow workspace hooks to override workspace hooks, so this can disable trusted shared behavior.
  - confidence: 0.89
- **[P2] Keep extra hook dirs at managed precedence:** `src/hooks/loader.ts:312-316`
  - body: hooks.internal.load.extraDirs are currently discovered as openclaw-managed sources and share managed enablement and collision policy. Loading them as openclaw-workspace changes default enablement and precedence for configured extra directories.
  - confidence: 0.88
- **[P2] Include plugin hook dirs in shared discovery:** `src/hooks/loader.ts:329`
  - body: The shared startup discovery rebuilds the source list from extra, bundled, and managed directories but omits resolvePluginHookDirs. Plugin-provided internal hooks would stop loading during Gateway startup.
  - confidence: 0.88
- **[P2] Preserve fail-closed hook path checks:** `src/hooks/loader.ts:203`
  - body: safeRealpathOrResolve falls back to path.resolve before open/import, whereas current main skips unreadable hook roots after realpath failure and opens under a checked root. Keep the fail-closed path before loading trusted hook code.
  - confidence: 0.84
