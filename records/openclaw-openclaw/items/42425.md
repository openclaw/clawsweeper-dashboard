## Summary

Keep open: the documented bug still exists on current main, but this PR is not mergeable as-is because it targets an obsolete startup file and regresses current hook loader contracts.

## What This Changes

The PR adds a startup hook loader for all configured agent workspaces, wires it into Gateway startup, and adds hook loader regression tests.

## Best Possible Solution

Land a rebased or replacement fix in src/gateway/server-startup-post-attach.ts that enumerates configured agent workspaces while preserving current hook opt-in, discovery precedence, plugin hook, trust-source, and reload-cleanup contracts.

## Reproduction Assessment

Yes, by source inspection. Current main documents per-agent workspace hooks, but Gateway startup still calls loadInternalHooks only with params.defaultWorkspaceDir, so non-default workspace hook directories are not discovered.

## Solution Assessment

No, not as-is. Multi-workspace startup loading is the right direction, but this branch targets a removed startup surface and would regress current hook opt-in, plugin discovery, source classification, and reload cleanup behavior.

  - confidence: 0.88

  - body: hooks.internal.load.extraDirs currently load with source openclaw-managed and share managed-hook precedence. Marking them openclaw-workspace changes both trust classification and override behavior for configured extra hook directories.
  - confidence: 0.88
- **[P2] Include plugin hook dirs in shared startup loading:** `src/hooks/loader.ts:329`
  - body: The shared startup loader rebuilds discovery from extra, bundled, and managed dirs only. Current hook discovery also includes plugin hook dirs, so switching startup to this helper would stop plugin-bundled internal hooks from loading.
