## Summary

Keep open: the underlying documented bug is still valid and this PR is the canonical fix thread, but the branch is now conflicting and the patch regresses current hook startup contracts while still lacking real behavior proof.

## What This Changes

The PR adds a multi-workspace internal hook startup loader, wires it into Gateway startup, and expands hook loader tests for non-default agent workspace hooks.

## Best Possible Solution

Land a rebased or replacement fix in src/gateway/server-startup-post-attach.ts that enumerates configured agent workspaces while preserving current hook opt-in, plugin/managed precedence, source classification, startup event semantics, and reload cleanup.

## Reproduction Assessment

Yes, by source inspection. Current main documents per-agent workspace hooks, but Gateway startup still calls loadInternalHooks only with params.defaultWorkspaceDir, so non-default workspace hook directories are not discovered.

## Solution Assessment

No, not as-is. Multi-workspace startup loading is the right direction, but this branch targets a removed startup surface and would regress hook opt-in, plugin discovery, extraDir source classification, and reload cleanup behavior.

  - confidence: 0.88

  - body: hooks.internal.load.extraDirs currently load as openclaw-managed and share managed-hook precedence. Marking them openclaw-workspace makes configured extra dirs disabled by default unless individually enabled and changes collision behavior.
  - confidence: 0.88
- **[P2] Include plugin hook dirs in shared startup loading:** `src/hooks/loader.ts:329`
  - body: The new shared startup discovery rebuilds the list from extra, bundled, and managed dirs only. Current discovery also includes plugin hook dirs, so plugin-bundled internal hooks would stop loading at startup.
