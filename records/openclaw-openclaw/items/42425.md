## Summary

Keep open: the underlying documented bug still exists on current main, but this PR is not mergeable as-is because it targets an obsolete startup file, conflicts with current main, and regresses current hook loader policy and path-hardening contracts. The external PR also lacks real behavior proof from a Gateway setup.

## What This Changes

The PR adds a startup-specific internal hook loader that scans multiple agent workspaces, scopes workspace-local handlers, rewires Gateway startup, and expands hook loader tests.

## Best Possible Solution

Land a rebased or replacement fix in src/gateway/server-startup-post-attach.ts that enumerates configured agent workspaces while preserving hook opt-ins, plugin/managed precedence, reload cleanup, fail-closed path checks, and startup event semantics.

## Reproduction Assessment

Yes, by source inspection. Current docs define per-agent workspace hooks, while current Gateway startup still calls loadInternalHooks with only params.defaultWorkspaceDir.

## Solution Assessment

No, not as-is. Multi-workspace startup loading is the right direction, but this branch must be rebased onto the live post-attach startup path and preserve newer hook loader policy, reload, plugin, and security contracts.

  - confidence: 0.89

  - body: Current main loads hooks.internal.load.extraDirs as openclaw-managed, sharing managed-hook enablement and collision behavior. Marking those entries as openclaw-workspace changes default enablement and precedence for configured extra hook directories.
  - confidence: 0.88

  - confidence: 0.88
