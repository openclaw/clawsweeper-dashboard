## Summary

Keep open: current main still has the non-default workspace hook startup bug, but this PR is stale against the active startup path and would regress current hook configuration and extra directory contracts if merged as-is.

## What This Changes

The PR adds a startup-specific multi-workspace internal hook loader, scopes workspace-local handlers by agent workspace, updates Gateway startup to enumerate configured agent workspaces, and adds hook-loader regression tests.

## Best Possible Solution

Refresh or replace the PR on current main by wiring multi-workspace hook loading into src/gateway/server-startup-post-attach.ts while preserving existing hook enablement, precedence, opt-in, failure isolation, and legacy-handler contracts.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence path: configure two agents with distinct <workspace>/hooks directories, start the Gateway, and current main only passes the default workspace into loadInternalHooks, whose discovery builds one workspace hooks path.

## Solution Assessment

No. The intended direction is reasonable, but this PR is not the best current solution as-is because it targets the old startup file shape and does not preserve current configured-hook and extraDirs contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve configured hook startup eligibility:** `src/hooks/loader.ts:48-50`
  - body: The new startup loader returns early unless hooks.internal.enabled is exactly true. Current main uses hasConfiguredInternalHooks, so named enabled entries, hook installs, extraDirs, and legacy handlers can load without that top-level flag; keep that contract or existing hook setups stop loading at startup.
  - confidence: 0.88
- **[P2] Keep extra hook dirs at managed precedence:** `src/hooks/loader.ts:312`
  - body: hooks.internal.load.extraDirs is loaded here as openclaw-workspace, but current loader and docs treat extra dirs as managed/shared hooks. Marking them as workspace hooks changes opt-in and collision/precedence behavior, so linked hook packs can stop loading or lose their managed semantics.
  - confidence: 0.9
- **[P2] Wire the active startup module:** `src/gateway/server-startup.ts:113-117`
  - body: This hunk updates the old monolithic startup file, but current main's src/gateway/server-startup.ts is only a re-export and active hook loading lives in src/gateway/server-startup-post-attach.ts. A rebase needs to move the startup hook call there or the intended behavior will not be present.
  - confidence: 0.86
