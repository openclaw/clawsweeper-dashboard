## Summary

Keep open: current main still has the reported single-workspace startup hook loading behavior, and this PR contains useful source work, but the branch is stale against the current startup split and introduces blocking hook-contract regressions around configured hook eligibility and extra hook directory precedence.

## What This Changes

The PR adds a startup-specific multi-workspace internal hook loader, scopes workspace-local handlers by agent workspace, updates Gateway startup to enumerate configured agent workspaces, and adds hook-loader regression tests.

## Best Possible Solution

Refresh the implementation against the active post-attach startup path: enumerate configured agent workspaces with listAgentWorkspaceDirs, load shared hooks once, load explicitly enabled workspace-local hooks per workspace, scope local handlers to matching event/session/agent workspace, preserve hasConfiguredInternalHooks and managed extraDirs precedence, isolate per-workspace failures, and cover the Gateway startup path with focused tests.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: configure two agents with distinct <workspace>/hooks entries, start the Gateway, and only the default workspace can be discovered because startup still calls loadInternalHooks with params.defaultWorkspaceDir and the loader scans one workspace hooks directory.

## Solution Assessment

No. The intended direction is plausible, but this PR is not the best current solution as-is because it targets the old startup file shape and does not preserve current configured-hook and extraDirs contracts; a rebased implementation in the active post-attach startup path is safer.

## Review Findings

  - body: The new startup loader returns early unless hooks.internal.enabled is exactly true. Current main uses hasConfiguredInternalHooks, so named enabled entries, hook installs, extraDirs, and legacy handlers can load without that top-level flag; keep that contract or existing hook setups stop loading at startup.
  - confidence: 0.88

  - body: hooks.internal.load.extraDirs is loaded here as openclaw-workspace, but current loader and docs treat extra dirs as managed/shared hooks. Marking them as workspace hooks makes them explicit opt-in and removes their managed override behavior, so configured extra hook directories can stop loading or lose precedence.
  - confidence: 0.9
- **[P2] Wire the active startup module:** `src/gateway/server-startup.ts:113-117`
  - body: This hunk updates the old monolithic startup file, but current main's src/gateway/server-startup.ts is only a re-export and active hook loading lives in src/gateway/server-startup-post-attach.ts. A rebase needs to move the startup hook call there or the intended behavior will not be present.
  - confidence: 0.86
