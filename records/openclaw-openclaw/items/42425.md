## Summary

Keep open. The PR targets a real source-reproducible bug and the discussion shows useful follow-up fixes for earlier review feedback, but the branch is stale against current main and the proposed startup loader would miss current hook configuration and plugin-hook contracts.

## What This Changes

The PR branch adds a startup hook loader for all configured agent workspaces, calls it from Gateway startup, and adds regression tests for multi-workspace loading, scoping, and override behavior.

## Best Possible Solution

Land a rebased or replacement implementation that loads workspace-local hooks for every configured agent workspace while preserving current hook gating, plugin hook discovery, extra-dir precedence, loader registration cleanup, and startup-post-attach structure.

## Reproduction Assessment

Yes, from source inspection. Current main documents per-agent workspace hooks but Gateway startup calls loadInternalHooks only with the default workspace directory, so non-default workspace hook directories are not discovered at startup.

## Solution Assessment

No, not as-is. The multi-workspace startup direction is the right narrow fix, but the PR needs to be adapted to current main and must preserve current hook configuration, plugin-hook, and extra-dir semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use configured-hook detection at startup:** `src/hooks/loader.ts:54-55`
  - body: The new startup loader returns unless hooks.internal.enabled is explicitly true. Current main intentionally uses hasConfiguredInternalHooks so enabled entries, installs, extraDirs, or legacy handlers can opt in without that flag; this would skip those configured hooks at Gateway startup.
  - confidence: 0.91
- **[P2] Keep plugin hook dirs in the shared startup loader:** `src/hooks/loader.ts:329`
  - body: loadSharedHookEntries rebuilds discovery from extra, bundled, and managed dirs only. Current hook discovery also includes plugin hook dirs, so switching Gateway startup to this helper would stop plugin-bundled internal hooks from loading.
  - confidence: 0.88
- **[P2] Keep extra hook dirs as managed hooks:** `src/hooks/loader.ts:315`
  - body: The PR marks hooks.internal.load.extraDirs as openclaw-workspace. Current main treats those dirs as openclaw-managed; changing the source makes them explicit-opt-in workspace hooks and changes their documented precedence.
  - confidence: 0.86
