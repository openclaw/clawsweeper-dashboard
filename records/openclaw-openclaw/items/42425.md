## Summary

This PR still targets a real documented hook-loading gap, but it should not merge as-is: current main moved the live Gateway startup path and strengthened hook loader policy after the branch was written, so the branch now misses the running startup module and regresses current hook opt-in, reload, plugin, precedence, and trusted-path behavior. It also lacks after-fix proof from a real Gateway setup, which is required before an external PR can merge.

## What This Changes

The PR adds a startup-specific multi-workspace internal hook loader, wires Gateway startup to enumerate configured agent workspace directories, and expands hook loader regression tests.

## Best Possible Solution

Land a rebased or replacement fix in src/gateway/server-startup-post-attach.ts that loads per-agent workspace hooks while preserving the current hook source, opt-in, precedence, reload cleanup, and trusted-path contracts.

## Reproduction Assessment

Yes, by source inspection. The docs define per-agent workspace hooks, while current main still calls loadInternalHooks with only params.defaultWorkspaceDir in the live Gateway startup path.

## Solution Assessment

No, not as-is. Multi-workspace loading is the right fix direction, but this branch is stale and must be retargeted to the live startup module while preserving newer hook loader contracts.
