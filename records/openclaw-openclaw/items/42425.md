## Summary

Keep open: current main still has the documented non-default workspace hook gap, but this PR is not mergeable as-is because it targets a deleted startup module, drops newer hook loader contracts, weakens hook path hardening, and lacks real behavior proof from a real Gateway setup.

## What This Changes

The PR adds a startup-only multi-workspace internal hook loader, wires Gateway startup to enumerate configured agent workspaces, scopes workspace-local handlers, and expands hook loader tests.

## Best Possible Solution

Land a rebased or replacement fix in src/gateway/server-startup-post-attach.ts that enumerates configured agent workspaces while preserving current hook opt-ins, plugin/managed precedence, reload cleanup, fail-closed path checks, and startup event semantics.

## Reproduction Assessment

Yes, by source inspection. Current main documents per-agent workspace hooks, but the live Gateway startup path still calls loadInternalHooks only with params.defaultWorkspaceDir.

## Solution Assessment

No, not as-is. Multi-workspace startup loading is the right direction, but this branch must be rebased onto the post-attach startup path and must preserve newer hook loader policy, reload, plugin, and security contracts.

- **[P2] Retarget the live post-attach startup path:** `src/gateway/server-startup.ts:22`
  - body: Current main no longer has this startup module; the live internal-hook startup path is `src/gateway/server-startup-post-attach.ts` and still calls `loadInternalHooks` with `params.defaultWorkspaceDir`.   - body: This startup helper returns unless `hooks.internal.enabled` is explicitly true. Current main also treats enabled entries, installs, extraDirs, and legacy handlers as configured hooks, so supported setups where the global flag is omitted would stop loading.
  - confidence: 0.93
- **[P2] Track loader-owned registrations on reload:** `src/hooks/loader.ts:184-185`
  - body: The refactor registers directory handlers without tracking them for unregistering, and `loadInternalHooks` no longer resets loader-owned registrations. Reloading hooks would duplicate workspace and legacy handlers that current main replaces cleanly.
  - confidence: 0.9
- **[P2] Keep current hook source precedence:** `src/hooks/loader.ts:442-443`
  - body: Current main prevents workspace hooks from overriding bundled, plugin, or managed hook code with the same name. Suppressing shared handlers whenever a same-name workspace hook exists changes that policy and can disable managed/plugin behavior for a workspace.
  - confidence: 0.89
- **[P2] Keep extra hook dirs at managed precedence:** `src/hooks/loader.ts:312-316`
  - body: Current main loads `hooks.internal.load.extraDirs` as `openclaw-managed`, sharing managed-hook enablement and collision behavior. Marking those entries as `openclaw-workspace` changes default enablement and precedence for configured extra hook directories.
  - confidence: 0.88
- **[P2] Include plugin hook dirs in shared startup loading:** `src/hooks/loader.ts:329`
  - body: Current discovery includes plugin hook directories via `resolvePluginHookDirs`. The new shared startup discovery rebuilds the shared list from extra, bundled, and managed dirs only, so plugin-bundled internal hooks would stop loading at Gateway startup.
  - confidence: 0.88
- **[P2] Preserve fail-closed hook path checks:** `src/hooks/loader.ts:203`
  - body: Current main requires readable real paths and open-root checks before importing hook handlers. This branch falls back from realpath failures before opening hook code, weakening the hardening around trusted local hook execution.
  - confidence: 0.84
