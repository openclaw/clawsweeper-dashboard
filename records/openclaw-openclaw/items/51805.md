## Summary

Keep open. Current main still has the feature gap: identity linking is a direct-message-only session-key behavior, while group/channel turns remain isolated, and the issue discussion adds privacy-filtering requirements that are not covered by the broader shared-channel trackers.

## What This Changes

Add an opt-in way for a linked user’s group-chat and direct-message turns to share one conversational session without including other group participants’ messages.

## Best Possible Solution

Keep this as a core session-routing design item. The end state should be an additive, opt-in generic contract that maps authorized sender identities from group/channel and direct turns to a shared canonical session, filters out other participants, preserves origin-specific reply delivery and runtime policy, and updates schema, docs, and cross-channel routing tests in one coherent model.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: build a direct route with identityLinks and a group/channel route for the same user context, and current main resolves them to different session keys; the existing continuity test codifies that separation.

## Solution Assessment

Unclear. The proposed config names are useful directionally, but the best fix needs maintainer design around privacy filtering, runtime posture, reply routing, and alignment with the broader shared-session trackers before implementation.

## Risks / Open Questions

- A naive shared key could leak messages from unrelated group participants into a private DM transcript.
- Group/channel sessions are documented as non-main and can have different sandbox/tool posture from DMs, so merging sessions changes a safety boundary.
- Reply delivery and last-route metadata must still route responses back to the originating group or DM even when several origins share one canonical transcript.
- The design overlaps #19929 and #7524; implementing a separate one-off option could create incompatible shared-session contracts.
