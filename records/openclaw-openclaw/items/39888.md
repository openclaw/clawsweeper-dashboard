## Summary

Keep open: the reported CLI diagnostics gap is still valid on current main, but this PR is not mergeable as-is because it patches the old command implementation shape and misses current trusted/context semantics. The prior Greptile and ClawSweeper comments identified useful direction; the repair should be a narrow port to the active agent command path.

## What This Changes

The PR adds CLI-path `model.usage` diagnostic emission to `src/commands/agent.ts`, computing usage, cost, context, and `channel: "cli"` after session-store updates.

## Best Possible Solution

Port the instrumentation to `src/agents/agent-command.ts` or a shared helper, matching reply-runner trusted emission, prompt-context accounting, current context-limit resolution, focused tests, and changelog coverage.

## Reproduction Assessment

Yes, source-level. Current main persists CLI agent usage through `updateSessionStoreAfterAgentRun` but has no `model.usage` emission in the active CLI path, while the reply-runner path emits that event.

## Solution Assessment

No. The goal is right, but this PR is not the best current fix because it targets the obsolete wrapper file and does not match trusted emission or prompt-context accounting semantics.

## Review Findings

Full review comments:

- **[P1] Port the emission to the active command path:** `src/commands/agent.ts:1101-1148`
  - body: Current main moved `agentCommandInternal` to `src/agents/agent-command.ts`, while `src/commands/agent.ts` is now a one-line re-export. This hunk targets the old implementation shape, so after rebase it will not affect the active CLI path and `openclaw agent` runs will still miss `model.usage`.
  - confidence: 0.96
- **[P2] Use the trusted diagnostic emitter:** `src/commands/agent.ts:1129`
  - body: This is a core-owned usage event, but the PR calls `emitDiagnosticEvent`, which marks it untrusted. Protected consumers such as diagnostics-prometheus drop untrusted events; match reply-runner and use `emitTrustedDiagnosticEvent`.
  - confidence: 0.9
- **[P2] Report prompt-context usage, not provider totals:** `src/commands/agent.ts:1144-1147`
  - body: The PR sets `context.used` to provider `totalTokens`, which can include output, cache, and multi-call tool-loop usage. Docs and reply-runner code define this field as the live prompt/context snapshot, so derive it from `promptTokens` or `lastCallUsage` instead.
  - confidence: 0.89
- **[P2] Resolve the current context limit:** `src/commands/agent.ts:1117-1122`
  - body: The fallback chain bypasses `resolveContextTokensForModel` and runtime `agentMeta.contextTokens` used by session persistence. Configured context windows, provider-qualified discovery values, and long-context settings can therefore emit a stale `context.limit`.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/commands/agent.ts:1101`
  - body: This is a user-facing diagnostics/telemetry fix, but the PR only changes code. Add a single-line `CHANGELOG.md` fix entry when porting the implementation so release notes cover CLI usage visibility.
  - confidence: 0.82
