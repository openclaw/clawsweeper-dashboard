## Best Possible Solution

Port the emitter to `src/agents/agent-command.ts` or a shared helper that matches reply-runner trust, trace, context-limit, prompt-context, cost, and identifier semantics, then add focused coverage plus redacted real CLI diagnostics proof.

- **[P1] Port the emission to the active CLI command:** `src/commands/agent.ts:1101-1148`
  - body: This block is added to the old `src/commands/agent.ts` implementation, but current `main` leaves that file as a re-export to `src/agents/agent-command.ts`. After rebase, this either conflicts or fails to instrument the active `openclaw agent` path.

- **[P2] Use the trusted diagnostic emitter:** `src/commands/agent.ts:1125-1129`
  - body: Core-owned `model.usage` events need to match reply-runner and use `emitTrustedDiagnosticEvent`; this call uses `emitDiagnosticEvent`, which marks the event untrusted and causes trusted-only consumers such as Prometheus to drop it.
  - confidence: 0.92
- **[P2] Report prompt-context usage:** `src/commands/agent.ts:1141-1144`
  - body: The patch reports `context.used` as provider total tokens, which can include output, cache, and accumulated tool-loop calls. Current docs define this field as the live prompt/context snapshot, so derive it the same way reply-runner does.
  - confidence: 0.9
- **[P2] Reuse the resolved context limit:** `src/commands/agent.ts:1115-1122`
  - body: The fallback chain bypasses runtime `agentMeta.contextTokens` and `resolveContextTokensForModel`, so provider-qualified config and runtime context metadata can emit stale limits. Reuse the same resolution semantics as session persistence and reply-runner.
  - confidence: 0.87
