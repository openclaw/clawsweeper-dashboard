## Summary

Keep open. Current main still has the CLI usage-telemetry gap, so the PR is not obsolete, but the branch is not merge-ready: it is dirty against main, targets the old `src/commands/agent.ts` implementation that is now only a re-export, and its emitted event does not match the trusted/context-accounting behavior of the existing reply-runner path.

## What This Changes

The PR adds `model.usage` diagnostic event emission to CLI agent runs by importing usage, cost, context, and diagnostic helpers in `src/commands/agent.ts` and emitting after session-store persistence.

## Best Possible Solution

Port the useful idea into current main as a small change in `src/agents/agent-command.ts` or a shared usage-diagnostics helper. Emit a trusted `model.usage` event after `updateSessionStoreAfterAgentRun`, reuse the same resolved provider/model/context/cost accounting as session persistence, derive `context.used` like the reply-runner path, and add a focused CLI regression test.

## Reproduction Assessment

Yes. Static current-main inspection is enough: the CLI path updates usage in the session store without emitting `model.usage`, while the reply-runner path emits the event; a live OTEL collector is not required to reproduce the missing emission point.

## Solution Assessment

No. The requested direction is right, but this PR is not the best current fix because it targets an obsolete wrapper, uses the untrusted emitter, and reports context usage/limits differently from the established reply-runner and session-store accounting.
