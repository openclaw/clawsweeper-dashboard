## Summary

Keep open: this PR targets a real CLI telemetry gap, but the patch is stale against current main and would not add the event to the active command implementation without a port.

## What This Changes

The PR adds `model.usage` diagnostic event emission to the CLI agent flow in `src/commands/agent.ts` after session-store persistence.

## Best Possible Solution

Port the instrumentation into `src/agents/agent-command.ts` or a shared usage-diagnostics helper, using trusted emission, resolved context-limit accounting, prompt-context usage, focused tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows CLI agent runs persist token/cost data but do not emit `model.usage`, while the reply-runner path does emit that diagnostic event.

## Solution Assessment

No. The requested direction is right, but this PR is not the best current fix because it targets the obsolete wrapper file and diverges from trusted emission and current context accounting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Port the emission to the active command path:** `src/commands/agent.ts:1104-1151`
  - body: Current main moved `agentCommandInternal` to `src/agents/agent-command.ts`, while `src/commands/agent.ts` is now only a re-export. This hunk targets the old implementation, so after rebase it will not patch the active CLI path and CLI runs will still miss `model.usage`.
  - confidence: 0.95
- **[P2] Use the trusted diagnostic emitter:** `src/commands/agent.ts:1128`
  - body: This core-owned event uses `emitDiagnosticEvent`, which marks it untrusted. Protected consumers such as diagnostics-prometheus skip untrusted events, and OTEL trusted parentage is lost; match reply-runner and use `emitTrustedDiagnosticEvent`.
  - confidence: 0.9
- **[P2] Report live context usage, not provider total:** `src/commands/agent.ts:1144-1146`
  - body: `context.used` is set to `totalTokens`, but provider totals can include output, cache, and multi-call/tool-loop usage. Docs define this field as the current prompt/context snapshot, so derive it from `promptTokens` or `lastCallUsage` like reply-runner.
  - confidence: 0.88
- **[P2] Reuse resolved context limit accounting:** `src/commands/agent.ts:1117-1121`
  - body: The fallback chain bypasses `resolveContextTokensForModel` and runtime `agentMeta.contextTokens` used by session persistence. Config-driven long-context settings can therefore emit a stale `context.limit`; reuse the resolved session-store/reply-runner accounting instead.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/commands/agent.ts:1104`
  - body: This is a user-facing diagnostics/telemetry fix, but the PR only changes code. Add a single-line `CHANGELOG.md` fix entry when porting the implementation so release notes cover the CLI usage visibility change.
  - confidence: 0.82
