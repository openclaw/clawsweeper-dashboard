## Summary

Keep open: the requested telemetry gap still exists on current main, but this branch cannot merge as-is because it edits the obsolete wrapper, emits an untrusted event, reports context incorrectly, and lacks real after-fix proof.

## What This Changes

The branch adds a `model.usage` diagnostic emission block to `src/commands/agent.ts` after CLI agent session-store updates.

## Best Possible Solution

Port the instrumentation into `src/agents/agent-command.ts` or a small shared helper, matching reply-runner trusted emission, prompt-context accounting, current context-limit resolution, regression tests, and changelog coverage.

## Reproduction Assessment

Yes by source inspection: current main's active CLI command path persists usage after the run but has no `model.usage` diagnostic emission, while the reply-runner emits that trusted event. I did not run a live OTEL collector in this read-only review, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. The goal is correct, but this PR edits the obsolete wrapper file and does not match current trusted-emitter, context-limit, or prompt-context semantics; the safer fix belongs in the active agent command path or a shared helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Port the emission to the active CLI path:** `src/commands/agent.ts:1101-1148`
  - body: This hunk edits the old in-file command implementation in `src/commands/agent.ts`, but current main leaves that file as a one-line re-export and runs `openclaw agent` from `src/agents/agent-command.ts`. After rebase, this either conflicts or does not instrument the active path.
  - confidence: 0.96
- **[P2] Use the trusted diagnostic emitter:** `src/commands/agent.ts:1129`
  - body: The new core-owned usage event calls `emitDiagnosticEvent`, which marks it untrusted. Protected consumers such as diagnostics-prometheus discard untrusted events, while the reply-runner uses `emitTrustedDiagnosticEvent` for `model.usage`.
  - confidence: 0.9
- **[P2] Report prompt-context usage:** `src/commands/agent.ts:1144-1147`
  - body: `context.used` is set from accumulated `totalTokens`, which can include output, cache, and multi-call tool-loop totals. Current docs and reply-runner semantics define this field as the live prompt/context snapshot, so it should be derived from `lastCallUsage` or prompt-token context data.
  - confidence: 0.89
- **[P2] Reuse the resolved context limit:** `src/commands/agent.ts:1115-1121`
  - body: The fallback chain now includes `sessionEntry?.contextTokens`, but it still bypasses runtime `agentMeta.contextTokens` and `resolveContextTokensForModel`. Configured long-context and provider-qualified model values can therefore emit stale context limits.
  - confidence: 0.84
