## Summary

Keep open: the CLI diagnostics gap is valid, but this PR cannot merge as-is because it patches the retired command body, is now conflicting with current main, emits an untrusted diagnostic event, and lacks required real behavior proof.

## What This Changes

The branch adds a `model.usage` diagnostic block to `src/commands/agent.ts` that computes token, cost, and context fields after CLI agent session-store updates.

## Best Possible Solution

Port the emission to `src/agents/agent-command.ts` or a shared helper that matches reply-runner trust, context-limit resolution, prompt-context accounting, trace handling where applicable, focused regression coverage, and redacted real CLI diagnostics proof.

## Reproduction Assessment

Yes. Source inspection shows current main persists CLI usage through `updateSessionStoreAfterAgentRun()` but has no active CLI `model.usage` emitter, while reply-runner already emits the trusted event.

## Solution Assessment

No. The requested direction is right, but this patch is not the best current solution because it edits the retired command body and diverges from trusted emitter, context-limit, and prompt-context semantics.

  - body: This block is added to the old in-file command implementation, but current main runs `openclaw agent` through `src/agents/agent-command.ts` and leaves this file as a re-export. After rebase, this either conflicts or fails to instrument the active CLI path.

  - confidence: 0.9

  - body: The fallback chain bypasses runtime `agentMeta.contextTokens` and `resolveContextTokensForModel`, so provider-qualified config and runtime context metadata can emit stale limits. Reuse the same resolution semantics as session persistence and reply-runner.
  - confidence: 0.85
