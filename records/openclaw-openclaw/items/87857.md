## Summary

Keep open. Current main injects a fixed workspace bootstrap-file set and a bounded reset/new daily-memory prelude, but it still lacks a runtime-enforced ordered startup-action contract or arbitrary per-agent first-turn file loading; the closed startupCommands report is a duplicate while the open related items cover narrower slices.

## What This Changes

The issue requests runtime-enforced startup context loading or ordered startup actions before an agent can answer the first user message.

## Best Possible Solution

Define a bounded startup-context contract with explicit diagnostics and privacy controls, route custom file injection through the bootstrap/directory-context API work, and avoid treating arbitrary AGENTS.md prose as executable policy.

## Reproduction Assessment

Yes, at source level. Current main has no per-agent startupCommands/init hook schema, rejects arbitrary extra bootstrap basenames, and gates the daily-memory startup prelude to bare /new or /reset handling.

## Solution Assessment

Unclear as a final product shape. The maintainable direction is a bounded startup-context/config contract with diagnostics and privacy controls, not automatic execution of arbitrary AGENTS.md text.
