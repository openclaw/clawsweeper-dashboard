## Summary

Keep open. Current main still chains ordinary same-chat Feishu messages through a per-chat Promise queue that awaits the full agent dispatch/reply lifecycle; related #54409 and the closed unmerged #43512 do not cover ordinary P2P starvation.

## What This Changes

The issue asks Feishu to prevent one slow or stuck same-chat P2P dispatch from indefinitely holding later messages behind the plugin queue.

## Best Possible Solution

Move later ordinary Feishu same-chat messages into the shared active-run/followup path instead of holding them behind the plugin Promise tail, while preserving dedupe, control-command, and `/btw` lanes.

## Reproduction Assessment

Yes. Hold the first same-key Feishu `handleMessage` Promise and enqueue a second normal P2P event; current main leaves the second task behind the unresolved `previous.then(task, task)` chain until the first settles.

## Solution Assessment

Yes. A Feishu-owned queue-boundary handoff is the narrowest maintainable direction; a blanket channel-local timeout around the full agent turn conflicts with the documented session/tool/runtime ownership model.

## Repair Work Prompt

Fix the Feishu per-chat queue starvation reported in https://github.com/openclaw/openclaw/issues/70133. Current main serializes ordinary same-chat P2P messages through `createSequentialQueue`, and the queued task awaits the full `handleFeishuMessage`/`core.channel.turn.run` reply lifecycle before releasing later messages. Implement a Feishu-owned queue-boundary release or bypass so later ordinary same-chat messages can reach the shared channel active-run/followup path once a run is active, while preserving first-message ordering, dedupe handling, control-command behavior, and the `/btw` out-of-band lane. Coordinate the shape with #54409 so collect mode is not worsened, but keep this PR focused on starvation/release behavior. Do not add a blanket channel-local timeout around `core.channel.turn.run`. Add or update deterministic tests that hold the first same-key dispatch and prove a later normal P2P message is not stuck behind the unresolved first task; keep existing sequential-key coverage for control and `/btw`. Add a user-facing `CHANGELOG.md` fix entry under the active version.
