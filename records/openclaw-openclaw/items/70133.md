## Summary

Keep open. Current main still serializes ordinary Feishu P2P messages behind a same-chat Promise chain that awaits the full agent dispatch/reply lifecycle, so a slow or stuck dispatch can continue starving later normal messages.

## What This Changes

Request a Feishu per-chat queue release or timeout guard so one stuck P2P dispatch cannot hold later same-chat messages indefinitely.

## Best Possible Solution

Release or bypass later ordinary Feishu same-chat messages into the shared channel active-run/followup path once a run is active, while preserving control-command and /btw lanes and coordinating any collect-mode fallout with #54409.

## Reproduction Assessment

Yes. Hold the first same-key Feishu handleMessage Promise and enqueue a second normal P2P event; current main leaves the second task behind previous.then(task, task) until the first settles.

## Solution Assessment

Yes, the requested release guard is the right narrow direction if implemented in the Feishu plugin queue boundary. A blanket per-channel agent-turn timeout would be less maintainable because routed runs are already governed by session, tool, and runtime lifecycle.
