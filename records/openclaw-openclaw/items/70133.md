## Summary

Keep open. Current main still has the reported Feishu same-chat starvation path for ordinary P2P messages: normal messages are chained behind a per-chat promise queue before they can reach the shared active-run/followup queue, and the latest release tag still contains the same behavior. The related #54409 is overlapping but distinct, and open PR #43512 only addresses control-command bypasses, not ordinary P2P messages queued behind a long dispatch.

## What This Changes

Feishu same-chat P2P dispatch should not let one long-running task hold the per-chat queue indefinitely and starve later messages.

## Best Possible Solution

Add a Feishu-owned stale-release or bounded bypass for ordinary same-chat dispatch so later messages can reach the shared gateway active-run/followup queue while preserving control-command and /btw lanes. Keep broader collect-mode semantics coordinated with #54409 if the final design also changes batching behavior.

## Reproduction Assessment

Yes. Hold the first same-key Feishu dispatch task and enqueue a second normal P2P event; current main keeps the second task behind previous.then(task, task) until the first dispatch Promise settles.

## Solution Assessment

Yes, the requested timeout or release guard is the right product direction, but the narrow maintainable fix should stay in the Feishu plugin unless multiple channel owners need a generic core seam. PR #43512 is not sufficient because it leaves non-command messages on the existing serialized path.
