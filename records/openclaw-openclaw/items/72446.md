## Summary

Keep this PR open: the guardrail is the right short-term direction, but the current patch still misses Claude deployments when Azure omits `modelName`, and the PR lacks required real Azure Foundry proof.

## What This Changes

The PR filters Anthropic deployments out of Microsoft Foundry onboarding and persisted catalogs, adds guardrail tests, and adds a changelog entry.

## Best Possible Solution

Keep the provider-local guardrail, update the classifier to fall back from deployment `modelName` to deployment `name`, add a regression test for missing `modelName`, rebase, and require redacted Azure Foundry proof before merge; first-class Azure Claude onboarding remains separately tracked at https://github.com/openclaw/openclaw/pull/47181.

## Reproduction Assessment

Yes for source-level reproduction: current main can offer and persist Claude deployments while Microsoft Foundry runtime paths are OpenAI-compatible `/openai/v1` requests. I did not run a live Azure Foundry account.

## Solution Assessment

No, not as currently patched: the guardrail is the narrow maintainable fix, but the classifier must use the same model-name fallback as runtime config so Claude deployment names are not missed when Azure omits `modelName`.

## Review Findings
