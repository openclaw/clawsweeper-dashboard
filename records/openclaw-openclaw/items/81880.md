## Summary

Keep open: this is a plausible gateway command-surface hardening PR, but the patch currently blocks existing deviceFamily fallback for protocol-accepted versioned mobile metadata and the external PR has only unit-test proof. The protected `maintainer` label also means this needs explicit maintainer handling before merge.

## What This Changes

The PR narrows gateway node command defaults to exact canonical platform IDs and maps the built-in node host's runtime platform values to those IDs.

## Best Possible Solution

Keep the exact-match hardening for desktop host command defaults, but preserve safe mobile deviceFamily fallback or first update producers, protocol/docs, and tests so every node client reliably sends canonical platform IDs.

## Reproduction Assessment

Yes by source inspection: current main maps noncanonical desktop aliases through prefix rules to desktop defaults, and the PR diff shows the intended narrowing path. I did not run a live gateway/node pairing flow.

## Solution Assessment

No. Exact canonical matching is a reasonable hardening direction, but returning `unknown` before consulting `deviceFamily` is too broad for current protocol-compatible mobile metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve safe deviceFamily fallback:** `src/gateway/node-command-policy.ts:162-164`
  - body: This early `unknown` return means any non-empty, noncanonical `platform` skips `deviceFamily`. Existing protocol-compatible clients can send values like `platform: "iOS 18.4.0"` with `deviceFamily: "iPhone"`; after this change they lose iOS defaults such as `device.info`, `contacts.search`, and `motion.activity`. Keep the desktop alias hardening, but preserve the safe family fallback for mobile metadata or update the producer/protocol contract before removing it.
