## Summary

Keep open. Current main still reuses a fresh stored agent main session when `openclaw agent --agent <id>` is run without a real `--session-id`, and there is still no fresh-run flag, Gateway protocol intent, resume banner, or docs coverage. The adjacent open PR #72932 is useful context for explicit session-id routing, but the provided context shows it does not fully cover this issue's implicit-resume UX.

## What This Changes

The issue asks for `openclaw agent --agent <id>` runs without `--session-id` to stop silently resuming the latest agent main session, either by starting fresh, adding an explicit fresh-session flag, or printing a resume banner.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run control such as `--new-session` with `--fresh` as an alias, plumb an additive fresh-session intent through local and Gateway paths, skip stored-session reuse only when that intent is set, emit visible resume metadata or a stderr banner when reuse happens, and document/test both local and Gateway behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway resolvers select the stored session id. I did not run a live agent invocation because this review is read-only, but the code path directly shows the reuse decision.

## Solution Assessment

Yes. The backward-compatible flag-plus-banner direction is the narrowest maintainable fix: flipping the default to fresh would break existing multi-turn scripts, while an explicit fresh-run knob and visible resume signal address the operator surprise without changing established session semantics.
