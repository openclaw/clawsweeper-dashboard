## Summary

Keep open. Current main and the latest release still reuse the agent main session for `openclaw agent --agent <id>` when no nonblank session selector is supplied, with no fresh/no-resume flag or resume disclosure; the narrower `--new-session` issue overlaps but does not fully own the banner/default-policy question.

## What This Changes

Requests a safer `openclaw agent` resume UX so repeated `--agent` invocations do not silently append to an existing main session without an explicit fresh-session control or disclosure.

## Best Possible Solution

Choose one backward-compatible fresh/no-resume contract, then implement it consistently across CLI parsing, local execution, Gateway protocol/handler, JSON metadata, docs, and focused regression tests while coordinating the narrower flag request.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: run `openclaw agent --agent <id>` twice without a nonblank `--session-id`, and the CLI/Gateway/local paths resolve the agent main session key and reuse a fresh stored session id. I did not run a live CLI repro in this read-only review.

## Solution Assessment

Unclear pending maintainer UX direction. A backward-compatible fresh/no-resume flag plus resumed-session disclosure looks safer than flipping the default immediately, but the exact flag, banner, JSON metadata, and suppression policy need a maintainer call.
