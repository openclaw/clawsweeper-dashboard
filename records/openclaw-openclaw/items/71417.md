## Summary

Keep open. Current main still has the reported no-`--session-id` path: `openclaw agent --agent <id> ...` resolves the agent main session key and reuses a fresh stored session id without a fresh-session flag, resume banner, or structured resumed indicator. The adjacent explicit `--session-id` remapping fix is useful context, but it does not cover this issue’s implicit-resume UX.

## What This Changes

Add an observable fresh-session control for `openclaw agent` so invocations without `--session-id` do not silently resume persisted agent main sessions without warning.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run control such as `--new-session` with `--fresh` as an alias, plumb an additive fresh-session intent through local CLI and Gateway paths, and make implicit reuse visible through stderr and structured metadata with matching docs and regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`; the local resolver and Gateway handler both select the stored session id.

## Solution Assessment

Yes, the reporter’s backward-compatible Option B is the best narrow path. Flipping the default to fresh would be a broader product change that could break existing multi-turn scripts and orchestrator workflows.
