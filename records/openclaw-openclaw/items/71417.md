## Summary

Keep open. Current main and v2026.6.8 still reuse a fresh stored agent main session without a `--new-session` or `--fresh` mode, resumed-session banner, or documented resumed metadata; the remaining work needs a maintainer CLI/session UX decision rather than conservative cleanup.

## What This Changes

The issue asks for `openclaw agent` to make implicit session resume observable or provide a clear fresh-session mode when no `--session-id` or `--channel` is passed.

## Best Possible Solution

Settle one backward-compatible CLI session-resume contract covering this issue and https://github.com/openclaw/openclaw/issues/54864, then implement the chosen flag, resume signal, Gateway/local plumbing, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection shows `--agent` maps to the agent main session key and both Gateway and local paths reuse a fresh stored `sessionId`; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as a final product direction. A backward-compatible fresh-session flag plus an explicit resume signal appears safer than flipping the default, but maintainers still need to choose the public CLI and JSON/banner contract.
