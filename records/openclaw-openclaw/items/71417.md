## Summary

Keep open. Current main still lacks a fresh/no-resume control or observable resume banner for `openclaw agent --agent <id>` without `--session-id`; both local and Gateway resolver paths can reuse a fresh stored agent main session, docs expose only main-session/resume behavior, and tests still assert reuse for absent or blank session ids. The adjacent explicit `--session-id` remapping problem has been partly improved, but the reporter's follow-up comments add concrete reproduction detail and the remaining CLI/Gateway UX gap is still real.

## What This Changes

Request that `openclaw agent --agent <id>` without an explicit fresh-session selector stop silently reusing the most recent/main session, or provide a documented fresh-session flag plus visible resume metadata/banner.

## Best Possible Solution

Keep resume-by-default compatibility for existing automation, add an explicit `--new-session` flag with `--fresh` as an alias, plumb an additive fresh-session intent through CLI, local embedded options, Gateway protocol, and Gateway handler, bypass stored-session reuse only for that flag, and make implicit resume observable through stderr diagnostics plus structured JSON metadata. Update docs, changelog, and regression tests for omitted, blank, explicit existing, explicit new, and forced-fresh session cases across both Gateway and `--local`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed or create a fresh `agent:<id>:main` session entry, run `openclaw agent --agent <id> -m ...` without a real `--session-id`, and both current resolver paths can select the stored session id. I did not run the live CLI because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The reporter's Option B is the narrowest maintainable direction: add a backwards-compatible fresh-session knob and make implicit resume visible, while leaving a default-to-fresh behavior change for a separate product/deprecation decision.
