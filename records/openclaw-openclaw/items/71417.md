## Summary

Keep open. Current main still silently reuses a fresh stored `agent:<id>:main` session when `openclaw agent --agent <id>` is run without a nonblank `--session-id`, and the CLI, Gateway protocol, Gateway handler, local resolver, and docs still lack an explicit fresh/no-resume control or resume disclosure.

## What This Changes

Request an explicit or observable fresh-session path for `openclaw agent --agent <id>` invocations that currently resume the last agent-main session by default.

## Best Possible Solution

Preserve backward-compatible resume-by-default behavior, add an explicit fresh/no-resume CLI and Gateway intent through local and Gateway paths, surface resumed-session metadata or a stderr banner, and document/test both resumed and fresh flows.

## Reproduction Assessment

Yes. A high-confidence source-level path is available: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` without a nonblank `--session-id`; current local and Gateway resolution choose the stored session id.

## Solution Assessment

Yes at the implementation-shape level: a backward-compatible fresh-session intent plus resume disclosure is narrower than flipping the default. The final flag names and banner/metadata policy still need maintainer approval.
