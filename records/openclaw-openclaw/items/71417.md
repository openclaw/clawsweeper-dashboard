## Summary

Keep open. Current main and v2026.6.10 still reuse a fresh stored agent main session for `openclaw agent --agent <id>` without a fresh-session flag, resume banner, or JSON `resumed` signal; the explicit `--session-id` sibling is fixed, but this broader CLI session UX still needs a maintainer product decision.

## What This Changes

The issue requests a visible fresh-session or resume contract for `openclaw agent --agent <id>` so repeated CLI invocations do not silently reuse the agent main session.

## Best Possible Solution

Maintainers should choose one backward-compatible CLI session contract, likely a fresh-session flag plus an explicit resume signal, then implement it consistently in Gateway and local paths with docs and focused tests.

## Reproduction Assessment

Yes, source-level. Current main maps `--agent` without `--session-id` to the agent main session key and reuses a fresh stored session entry in both embedded and Gateway paths; this read-only review did not run the CLI.

## Solution Assessment

Unclear as a final product direction. A backward-compatible fresh-session flag plus explicit resume signal is plausible, but maintainers still need to choose the flag name, banner or JSON shape, suppression policy, and any future default change.
