## Summary

Keep open. Current main still has the reported source-level behavior: `openclaw agent --agent <id>` routes to the agent main session, reuses a fresh stored session without a fresh-session flag, and does not expose a resume banner or resumed JSON signal; the remaining work needs a maintainer product call rather than cleanup closure.

## What This Changes

The issue requests safer `openclaw agent` session behavior when an invocation without an explicit session silently resumes the agent's existing main session.

## Best Possible Solution

Choose one backward-compatible CLI session contract across this issue and the narrower fresh-session proposal, then implement it consistently through CLI parsing, Gateway schema/handler, embedded local execution, output, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current CLI and Gateway/local session resolution still reuse a fresh stored main session when no session id is requested, and there is no fresh-session flag or resumed output contract. I did not run a live CLI reproduction because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the public contract; a backward-compatible explicit fresh-session flag plus resume indicator/banner appears safer than flipping the default because current multi-turn scripts may rely on auto-resume.
