## Summary

Keep open. Current main and v2026.6.10 still silently reuse a fresh `agent:<id>:main` session for `openclaw agent --agent <id>` without a fresh-session flag or resume banner, and the remaining choice is a public CLI/session UX contract that needs maintainer direction.

## What This Changes

The issue requests making `openclaw agent --agent ...` session reuse observable or adding an explicit fresh-session control because omitted session selectors reuse the agent main session silently.

## Best Possible Solution

Choose one backward-compatible agent CLI session contract, likely preserving default resume while adding an explicit fresh/no-resume selector plus resume observability in stderr and/or JSON, then cover both Gateway and `--local` paths in docs and tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `openclaw agent --agent foo` resolves to the agent main session key, and the gateway/local paths reuse a fresh stored session id. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as final product direction. A backward-compatible fresh-session flag plus resume banner is plausible, but maintainers should choose the public flag, alias, output, and default-resume policy before implementation.
