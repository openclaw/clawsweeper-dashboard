## Summary

Keep open. Current main still resumes the fresh stored `agent:<id>:main` session for `openclaw agent --agent <id>` when no real `--session-id` is supplied, and there is still no CLI flag, Gateway protocol field, resume banner, or docs path for an operator-facing fresh run. The adjacent open PR #72932 is useful context, but its diff targets explicit `--session-id` routing and Gateway startup, not this issue's implicit-resume UX.

## What This Changes

Request an explicit or visible fresh-session control for `openclaw agent` so omitted `--session-id` no longer silently resumes the most recent agent main session.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run control such as `--new-session` with `--fresh` as an alias, plumb an additive `forceNewSession` intent through local and Gateway paths, skip stored-session reuse only when that intent is set, emit visible resume metadata/banner when reuse happens, and document/test both local and Gateway behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway resolvers select the stored session id. No live run was performed because this review was read-only, but the code path directly shows the reuse decision.

## Solution Assessment

Yes. The backward-compatible flag-plus-banner direction is the best narrow fix: flipping the default to fresh would break existing multi-turn scripts, while an explicit fresh-run knob and visible resume signal solve the operator surprise without changing established session semantics.
