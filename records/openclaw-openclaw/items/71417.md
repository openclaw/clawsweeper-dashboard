## Summary

Keep open. Current main still routes `openclaw agent --agent <id>` with no real `--session-id` to the agent main session key and reuses the stored session id while it is fresh, with no fresh-session flag, no Gateway protocol field, no resume banner, and no docs for an operator-facing fresh-run control. The adjacent explicit `--session-id` work and open PR #72932 are useful context, but they do not cover this issue's implicit-resume UX.

## What This Changes

The issue requests that `openclaw agent --agent <id>` without `--session-id` stop silently resuming the previous agent main session, either by starting fresh by default or by adding an explicit fresh-session control and visible resume notice.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run control such as `--new-session` with `--fresh` as an alias, plumb an additive fresh-session intent through both local embedded and Gateway request paths, make implicit reuse visible through stderr and structured metadata, and update docs plus regression tests for omitted, blank, explicit, and fresh-session cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` with no real `--session-id`; both local and Gateway resolvers select the stored session id. No live run was needed for this read-only review because the current code path directly shows the reuse decision.

## Solution Assessment

Yes. The reporter's backward-compatible Option B is the narrowest maintainable fix because flipping the default to fresh could break existing multi-turn scripts, while a flag plus visible resume metadata solves the discoverability problem without changing established session semantics.
