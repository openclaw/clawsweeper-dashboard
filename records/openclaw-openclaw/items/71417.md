## Summary

Keep open. Current main still reuses a fresh stored `agent:<id>:main` session for `openclaw agent --agent <id> -m ...` when no real `--session-id` is supplied, and there is still no `--new-session`/`--fresh` flag, resume banner, or Gateway protocol field to force a fresh run. The discussion adds a concrete VPS/MAS repro and correctly distinguishes this from the adjacent explicit-session-id fix.

## What This Changes

Request to make `openclaw agent --agent <id>` without `--session-id` avoid or clearly expose implicit resume of the existing agent main session.

## Best Possible Solution

Keep resume-by-default compatibility, add an explicit `--new-session` flag with `--fresh` as an alias, plumb an additive fresh-session intent through CLI, local embedded execution, Gateway protocol, and Gateway handling, and make implicit resume observable through stderr plus structured result metadata and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed or create a fresh `agent:<id>:main` session, then run `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway code paths select the stored session id.

## Solution Assessment

Yes. The reporter's Option B is the narrowest maintainable direction because it gives operators an explicit fresh-run control and visible resume signal without silently breaking existing multi-turn workflows that depend on continuity.
