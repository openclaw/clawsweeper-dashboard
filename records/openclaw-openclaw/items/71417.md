## Summary

Keep open. Current main still reuses a fresh stored `agent:<id>:main` session when `openclaw agent --agent <id> -m ...` omits a nonblank `--session-id`, and there is still no fresh-session flag, Gateway protocol field, resume banner, or docs coverage for that path. The adjacent explicit `--session-id` issue was fixed separately, but it does not resolve this no-session-id resume behavior.

## What This Changes

Request that `openclaw agent` stop making implicit session resume silent, either by defaulting fresh, adding a fresh-session flag, or printing an observable resume banner.

## Best Possible Solution

Keep resume-by-default compatibility for existing multi-turn workflows, add an explicit fresh-session control such as `--new-session` with `--fresh` as an alias, plumb that intent through CLI, local embedded execution, Gateway protocol, and Gateway handling, and make implicit resume visible through stderr, structured metadata, tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: seed or create a fresh `agent:<id>:main` session entry, then run `openclaw agent --agent <id> -m ...` without a nonblank `--session-id`; both local and Gateway paths select the stored session id. I did not run the live CLI because this review was constrained to read-only inspection.

## Solution Assessment

Yes. The backward-compatible flag-plus-observability approach is the narrowest maintainable fix because it lets operators force fresh runs and notice implicit resume without breaking scripts and orchestrator workflows that intentionally rely on continuity.
