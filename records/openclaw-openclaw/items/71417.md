## Summary

Keep open. Current main still has the reported silent-resume shape for `openclaw agent --agent <id>` without a nonblank `--session-id`: it resolves the agent main session, reuses a fresh stored `sessionId`, and exposes no fresh/no-resume flag or resume disclosure; the reporter’s follow-up also confirms the linked Gateway PR is no longer a candidate fix.

## What This Changes

Request to make `openclaw agent --agent <id>` fresh-session behavior explicit by changing the default, adding a fresh/no-resume flag, or warning when a stored main session is resumed.

## Best Possible Solution

Add a backward-compatible fresh/no-resume intent through CLI, local execution, Gateway protocol, and Gateway handler paths, surface resumed-session state, and coordinate the flag-only portion with https://github.com/openclaw/openclaw/issues/54864.

## Reproduction Assessment

Yes, source-level. On current main, omitting a nonblank `--session-id` with `--agent <id>` resolves the agent main session and both local and Gateway paths can reuse a fresh stored `sessionId`; I did not run a live CLI turn in this read-only review.

## Solution Assessment

Unclear pending maintainer UX direction. A backward-compatible fresh/no-resume control plus resume disclosure looks safer than flipping the default immediately, but the exact flag, metadata/banner, and suppression policy need a maintainer call.
