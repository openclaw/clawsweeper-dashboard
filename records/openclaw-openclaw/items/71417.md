## Summary

Keep open. Current main and v2026.6.10 still reuse a fresh stored agent main session for `openclaw agent --agent <id>` without a fresh-session flag, resume banner, or JSON resumed signal; #71415 fixed the explicit `--session-id` sibling bug, and #54864 overlaps with one possible flag shape, but this broader CLI resume UX still needs a maintainer product decision.

## What This Changes

Request a safer `openclaw agent` session UX so omitted `--session-id` does not silently resume stale agent context, or at least exposes and controls that resume.

## Best Possible Solution

Choose one backward-compatible CLI/session contract across this issue and #54864, then implement it through the parser, Gateway schema and handler, embedded local path, JSON/text output, docs, and focused tests.

## Reproduction Assessment

Yes at source level: current CLI and Gateway session resolution reuse a fresh stored session when no session id is requested, and current parser/docs expose no fresh-session flag, resume banner, or resumed JSON field; I did not run a live CLI repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract; a backward-compatible explicit fresh-session flag plus resume indicator/banner appears safer than flipping the default because current multi-turn scripts may rely on auto-resume.
