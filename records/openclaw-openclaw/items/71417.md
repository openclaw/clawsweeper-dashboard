## Summary

Keep open. Current main still routes `openclaw agent --agent <id>` without an explicit session selector to the agent main session and reuses a fresh stored session without a `--new-session`/`--fresh` flag, resume banner, or JSON `resumed` signal; the related `--session-id` remap bug is fixed, but this broader CLI session UX still needs a maintainer product decision.

## What This Changes

The issue asks OpenClaw to make `openclaw agent --agent <id>` fresh-session intent observable or explicit instead of silently reusing that agent's main session.

## Best Possible Solution

Adopt one backward-compatible CLI session-resume contract across this issue and https://github.com/openclaw/openclaw/issues/54864, then implement it consistently in CLI parsing, Gateway dispatch, embedded/local resolution, docs, and focused session tests.

## Reproduction Assessment

Yes, source-level. Current main maps `--agent` without `--session-id` to the agent main session key and reuses a fresh stored session entry in both embedded and Gateway paths; this read-only review did not run the CLI.

## Solution Assessment

Unclear as a final product direction. A backward-compatible fresh-session flag plus explicit resume signal is plausible, but maintainers still need to choose the flag name, banner or JSON shape, suppression policy, and any future default change.
