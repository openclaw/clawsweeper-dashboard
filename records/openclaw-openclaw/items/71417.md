## Summary

Keep open. Current main still reuses a fresh stored agent main session for repeated `openclaw agent --agent <id>` runs and has no `--new-session`, resume banner, or resumed metadata contract; the narrower fresh-session request remains open and the referenced PR no longer addresses this UX.

## What This Changes

Make `openclaw agent` fresh-session and resume behavior explicit instead of silently reusing the agent main session when no explicit fresh-session control is provided.

## Best Possible Solution

Choose one backward-compatible CLI session-resume contract for this issue and https://github.com/openclaw/openclaw/issues/54864, then implement it consistently through CLI parsing, Gateway/local execution, docs, and focused session tests.

## Reproduction Assessment

Yes, source-level. Current main maps `--agent` without `--session-id` to the agent main session key and reuses a fresh stored session entry in both embedded and Gateway paths; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as a final product direction. A backward-compatible fresh-session flag plus an explicit resume signal is plausible, but maintainers still need to choose the flag name, banner or JSON shape, suppression policy, and any future default change.
