## Summary

Keep open. Current main fixed the adjacent explicit `--session-id` remapping problem, but the reported no-`--session-id` path still silently reuses a fresh stored `agent:<id>:main` session and lacks a fresh-session flag, resume banner, or structured resumed indicator.

## What This Changes

The issue asks for `openclaw agent --agent <id>` runs without `--session-id` to either start fresh, expose a fresh-session flag, or visibly report that an existing session is being resumed.

## Best Possible Solution

Keep resume-by-default compatibility for existing multi-turn workflows, add an explicit `--new-session` flag with `--fresh` as an alias, plumb an additive fresh-session intent through CLI, local embedded execution, Gateway protocol, and Gateway handling, and make implicit resume observable in stderr plus structured metadata and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create or seed a fresh `agent:<id>:main` session entry, then run `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway paths select the stored session id. I did not run the live CLI because this review was constrained to read-only inspection.

## Solution Assessment

Yes. The reporter’s Option B is the narrowest maintainable fix because it adds an explicit fresh-run control and makes resume visible without breaking scripts and orchestrator workflows that intentionally rely on continuity. Flipping the default to fresh should be a separate product or deprecation decision.
