## Summary

Keep open. Current main and v2026.6.6 still auto-select the agent main session and reuse a fresh stored session without a force-new flag or resume banner; https://github.com/openclaw/openclaw/issues/54864 is narrower, and https://github.com/openclaw/openclaw/pull/72932 no longer addresses this session UX.

## What This Changes

Requests a safer `openclaw agent` session UX so repeated runs without an explicit fresh-session selector do not silently continue the last agent session.

## Best Possible Solution

Decide one core CLI session-resume contract covering this issue and https://github.com/openclaw/openclaw/issues/54864, then implement the chosen flag, banner or JSON metadata, Gateway/local plumbing, docs, and focused tests while preserving compatibility unless maintainers choose a deprecation path.

## Reproduction Assessment

Yes, source-level: current main maps `--agent` without `--session-id` to the agent main session key and reuses a fresh stored session entry. I did not run the CLI because this review is read-only, but the path is visible in the CLI, local resolver, Gateway handler, docs, and release tag.

## Solution Assessment

Unclear as a final product direction. A backward-compatible fresh-session flag plus an explicit resume signal is plausible, but maintainers still need to choose the flag name, banner/JSON shape, suppression policy, and whether the default should ever flip.
