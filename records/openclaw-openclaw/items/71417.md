## Summary

Keep open. Current main still resumes a fresh stored agent-main session for `openclaw agent --agent <id>` when no real `--session-id` is supplied, and there is still no CLI flag, Gateway protocol field, resume banner, or docs path for forcing or observing a fresh run.

## What This Changes

The issue asks for `openclaw agent --agent <id>` to avoid silently resuming the most recent agent session by defaulting fresh, adding a fresh-session flag, or showing a resume banner.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run control such as `--new-session` with `--fresh` as an alias, plumb additive fresh-session intent through local and Gateway paths, skip stored-session reuse only for that intent, expose resumed-session metadata or a stderr banner when reuse happens, and document/test both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway resolvers select the stored session id.

## Solution Assessment

Yes. The backward-compatible flag-plus-disclosure direction is the narrowest maintainable fix; flipping the default to fresh would break scripts that intentionally rely on multi-turn agent-main sessions.
