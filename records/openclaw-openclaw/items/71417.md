## Summary

Keep open. Current main still silently resumes a fresh stored agent-main session for `openclaw agent --agent <id>` without `--session-id`, and it lacks a CLI flag, Gateway protocol field, resume banner, or docs path for forcing or observing a fresh run.

## What This Changes

The issue asks for `openclaw agent` to make implicit session resume discoverable or provide an explicit fresh-session control when no session id is supplied.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run control such as `--new-session` with `--fresh` as an alias, plumb additive intent through local and Gateway paths, expose resumed-session metadata or a stderr banner, and document/test both behaviors.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: seed a fresh `agent:<id>:main` session entry, then run `openclaw agent --agent <id> -m ...` without a real `--session-id`; the local and Gateway resolvers both select the stored session id.

## Solution Assessment

Yes. The backward-compatible flag-plus-disclosure path is the narrowest maintainable fix; flipping the default to fresh would break scripts and multi-turn workflows that intentionally rely on agent-main session continuity.
