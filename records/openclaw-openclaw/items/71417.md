## Summary

Keep open. Current main still has the implicit resume path for `openclaw agent --agent <id>` without a real `--session-id`, and it still lacks a fresh-session flag, Gateway protocol intent, resume banner, or docs for forcing/observing a fresh run.

## What This Changes

Requests that `openclaw agent --agent <id>` without `--session-id` stop silently reusing the last stored main session by adding a fresh-session control and/or visible resume disclosure.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit fresh-run flag such as `--new-session` with `--fresh` as an alias, plumb that additive intent through local and Gateway paths, disclose resumed sessions, and document/test both modes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed a fresh `agent:<id>:main` session entry, then invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway resolvers select the stored session id.

## Solution Assessment

Yes. A backward-compatible fresh flag plus resume disclosure is the narrowest maintainable direction; flipping the default to fresh would break scripts and multi-turn workflows that intentionally rely on the main agent session.
