## Summary

Keep this open: current main and the latest release still allow WebChat streaming updates to re-pin the transcript after a small upward user scroll, and the open closing PR already owns the focused repair.

## What This Changes

The issue asks for WebChat sticky-bottom behavior that preserves backscroll when new messages or streamed chunks arrive after the user scrolls up.

## Best Possible Solution

Land the narrow WebChat follow-lock repair in https://github.com/openclaw/openclaw/pull/92622, or replace it with an equivalent focused fix that preserves manual, near-bottom, always, and off modes with regression coverage.

## Reproduction Assessment

Yes. From source inspection, a small upward scroll can remain inside the 450px near-bottom threshold, leave auto-follow eligible, and the next chatStream update calls scheduleChatScroll back to the bottom.

## Solution Assessment

Yes. The best current path is the linked focused repair because it changes the central scroll helper and preserves existing scroll modes without broader UI restructuring.
