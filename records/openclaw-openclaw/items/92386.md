## Label Justifications

- `P1`: The issue reports a current user-facing WebChat regression that disrupts reading backscroll while an agent is actively streaming.
- `impact:other`: The impact is meaningful WebChat usability breakage, but it is not data loss, message loss, session state, auth, security, or availability.

## Summary

This should stay open: current main and v2026.6.6 still have the near-bottom WebChat scroll path that can re-stick after an upward user scroll, and the open closing PR at https://github.com/openclaw/openclaw/pull/92622 already owns the focused repair.

## What This Changes

WebChat should preserve the user's backscroll when new messages or streaming chunks arrive unless the user is already following the bottom.

## Best Possible Solution

Review and land one narrow WebChat follow-lock repair, preferably the focused open closing PR if maintainer review stays positive, while preserving the existing manual, near-bottom, always, and off scroll modes.

## Reproduction Assessment

Yes from source inspection: a small upward scroll can remain within the 450px near-bottom threshold and then the next chatStream update can call scheduleChatScroll back to the bottom. I did not run a browser repro in this read-only cleanup review.

## Solution Assessment

Yes, the best current path is the linked focused repair because it changes the central scroll helper, preserves existing scroll modes, and adds regression proof without broad UI restructuring.
