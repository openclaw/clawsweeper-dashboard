## Summary

Keep this open: current main and the latest release still allow WebChat streaming updates to re-stick near-bottom scroll state after a user scrolls up, and the open proof-supplied repair already owns the narrow fix.

## What This Changes

The issue reports that WebChat forces the conversation view to the bottom on new or streamed messages even after the user has scrolled up to read earlier content.

## Best Possible Solution

Land one focused WebChat scroll follow-lock fix, preferably the open proof-supplied repair if maintainer review remains positive, while preserving the existing manual, near-bottom, always, and off scroll modes.

## Reproduction Assessment

Yes from source inspection: a small upward scroll can remain inside the 450px near-bottom threshold or leave chatUserNearBottom true, then the next chatStream update calls scheduleChatScroll back to the bottom. I did not run a browser repro in this read-only review.

## Solution Assessment

Yes, the best current path is the linked focused repair because it changes the central scroll helper, preserves existing scroll modes, and adds targeted regression proof without broad UI restructuring.
