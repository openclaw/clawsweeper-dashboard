## Summary

This is a valid current-release WebChat regression, and current main still contains the scroll-interruption race described by the report. An existing open, proof-positive repair PR addresses that exact race, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

Fix WebChat so incoming and streaming messages preserve deliberate backscroll unless the user is already near the bottom.

## Best Possible Solution

Review and land a focused interrupt-aware scroll-state fix, preferably the existing canonical PR, that immediately releases auto-follow on deliberate upward input and proves streaming cannot reacquire follow until the user returns to the bottom.

## Reproduction Assessment

Yes from source inspection: upward input during the programmatic-scroll guard can fail to release near-bottom follow state, after which streaming updates schedule another scroll. The reporter's concrete v2026.6.5 steps and the canonical PR's real Control UI smoke match that path, although this review did not execute the UI itself.

## Solution Assessment

Yes in direction: explicit user-intent and follow-lock handling at the central scroll helper addresses the state race more directly than changing the existing 450-pixel threshold. The open repair should remain the canonical path if its current branch still preserves all three configured auto-scroll modes.
