## Label Justifications

- `P1`: The issue reports a current user-facing WebChat regression that disrupts reading backscroll while an agent is actively streaming.
- `impact:other`: The impact is meaningful WebChat usability breakage, but it is not data loss, message loss, session state, auth, security, or availability.

## Summary

This should stay open: current main and v2026.6.6 still have the WebChat scroll helper path that can re-pin the transcript after an upward user scroll during streaming, and the open repair at https://github.com/openclaw/openclaw/pull/92622 already owns the narrow fix.

## What This Changes

WebChat should preserve the user's scroll position when new messages or streamed chunks arrive after the user has deliberately scrolled up.

## Best Possible Solution

Land one narrow WebChat scroll follow-lock fix, preferably https://github.com/openclaw/openclaw/pull/92622 if maintainer review remains positive, while preserving the existing manual, near-bottom, always, and off scroll modes.

## Reproduction Assessment

Yes from source inspection: a small upward scroll can remain within the 450 px near-bottom threshold, leave `chatUserNearBottom` true, and let the next `chatStream` update call `scheduleChatScroll()` back to the bottom. I did not run a browser repro because this cleanup review was read-only.

## Solution Assessment

Yes, the best current path is the linked focused repair because it changes the central scroll helper, preserves existing scroll modes, and adds regression proof without broad UI restructuring.
