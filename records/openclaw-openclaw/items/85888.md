## Repair Work Prompt

Fix the recurring cron retry scheduling bug reported in https://github.com/openclaw/openclaw/issues/85888. Current docs say recurring cron transient errors retry with exponential backoff, but src/cron/service/timer.ts treats backoff as a lower bound against the natural cron slot, so daily jobs can skip the near retry after overloaded/server_error failures. Implement a narrow scheduler repair in the cron result path so retryable recurring transient failures schedule the backoff retry when it is earlier than the natural next run, while preserving normal cadence after success or when retry is no longer useful. Do not add MiniMax-specific logic, do not add new public CLI/API/config retry flags, and do not duplicate the broader design in https://github.com/openclaw/openclaw/pull/51001 unless maintainers explicitly choose that path. Add focused regression coverage for daily cron overloaded/server_error behavior, keep one-shot retry behavior intact, and update docs only if the final behavior changes the existing documented contract. Changelog is expected only if the repair is user-visible under repo policy.
