## Summary

Keep open. Current main and the latest release still show a source-backed cron retry mismatch: recurring cron failures compute a backoff window, but daily cron schedules can store the later natural cron slot instead of the near retry, which matches the report that manual reruns can succeed after scheduled failures.

## What This Changes

Scheduled cron runs that hit MiniMax overload/server errors should retry according to the documented recurring backoff behavior instead of waiting for the next natural cron slot.

## Best Possible Solution

Repair recurring cron error handling so retryable provider failures schedule the next attempt at the documented backoff slot, then resume the normal cron schedule after success or exhausted retry policy.

## Repair Work Prompt

Fix the recurring cron retry mismatch reported at https://github.com/openclaw/openclaw/issues/85888. Current main computes a backoff retry after a recurring cron error but, for daily cron schedules, stores the later natural cron slot instead of the near backoff retry; MiniMax overloaded_error/503 is already classified as retryable. Implement the narrow scheduler repair in the cron service so retryable recurring provider failures schedule the backoff attempt before the next natural cron slot, then return to the normal cron schedule after success or exhausted retry policy. Add regression coverage for a daily Asia/Shanghai cron job with an overloaded/server_error failure where the first retry runs at the backoff time, plus a success path that resumes the normal cron schedule. Do not change MiniMax provider code, global provider failover policy, or add a new config knob. Include a changelog fix entry if this becomes a user-facing fix PR.
