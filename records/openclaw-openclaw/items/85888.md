## Repair Work Prompt

Fix the recurring cron retry scheduling bug reported in https://github.com/openclaw/openclaw/issues/85888. Current source classifies MiniMax 503/overloaded errors as retryable and docs describe recurring backoff, but recurring cron errors currently compute backoffNext and then schedule max(naturalNext, backoffNext), so daily jobs can wait until the next natural daily slot instead of retrying in the backoff window. Keep the fix generic to cron retry semantics, not MiniMax-specific. Preserve normal cron cadence after success or retry exhaustion, do not add new CLI/protocol/config retry flags for this narrow bug, and be aware of related broader work in https://github.com/openclaw/openclaw/issues/49740 and https://github.com/openclaw/openclaw/pull/51001. Add regression coverage for a recurring daily cron job that fails with a retryable overload/server error and schedules the next attempt at the backoff time instead of the next natural daily slot, plus coverage that successful retry returns to the normal schedule.
