## Summary

Keep open. Current main still has no Codex pre-turn exact token accounting or estimate marker on the context-engine projection path, and the remaining work needs a Codex runtime/tokenizer contract decision rather than conservative cleanup.

## What This Changes

Request a Codex projection accounting seam that reports exact or explicitly estimated pre-turn token counts and distinguishes frontier, projected prompt, and provider-observed usage.

## Best Possible Solution

Add a Codex-owned projection stats and accounting seam that records rendered projection size, marks estimates explicitly, reports exact input tokens when the runtime can provide them, and feeds status/afterTurn telemetry without adding a new config knob.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature and observability gap. Source inspection confirms current main has no exact pre-turn token accounting or estimate marker on the Codex projection path.

## Solution Assessment

Unclear until the Codex runtime contract is checked. The maintainable direction is to use an authoritative app-server/tokenizer preflight if one exists, and otherwise expose bounded projection stats with an explicit estimated status.
