## Summary

Keep open: current main still has a source-proven sender-blind compaction dedup path, and the remaining fix should coordinate with the existing sender-metadata persistence work rather than close this as already handled.

## What This Changes

The issue reports that compaction duplicate-user-message deduplication can drop a distinct group participant's identical-text message because the dedup signature ignores sender identity.

## Best Possible Solution

Land or revise the focused sender-metadata persistence path, then make compaction duplicate-user dedup use that trusted sender discriminator while preserving same-sender retry pruning and adding focused regression coverage.

## Reproduction Assessment

Yes, at source level: current main's duplicateSignature keys only on normalized user text and timestamp, and both compaction consumers use that signature. A full live group-channel compaction run was not performed, but the helper boundary and persistence gap are clear from source.

## Solution Assessment

Partly. A sender-aware key is the right compaction behavior, but the durable sender discriminator and privacy semantics should line up with the existing open sender-metadata work rather than inventing a parallel metadata contract.
