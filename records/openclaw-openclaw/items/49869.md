## Summary

Keep open: current main still lacks Nextcloud Talk rich-object media parsing, and this PR is a plausible fix, but it needs MIME propagation, stronger real file-share proof, and coordination with overlapping structured-body work before merge.

## What This Changes

The PR parses Nextcloud Talk JSON rich-object webhook content, extracts file links into inbound media URLs, passes them through inbound context, relaxes media-only drops, and adds parser/webhook tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text plus aligned media URLs and MIME types, preserves malformed/plain-text fallbacks, and is proven with a real file-share webhook before merge.

## Reproduction Assessment

Yes, at source level: a signed Nextcloud Talk `Create` webhook with JSON rich-object file content currently becomes raw text on main and has no media URL/type path. I did not establish a live file-share repro, and the supplied live report says real file-share webhooks did not reach the patched code.

## Solution Assessment

No. The plugin boundary is the right place to fix this, but the current PR needs MIME propagation, parser coordination, and real file-share proof before it is the best maintainable solution.

Full review comments:

- **[P2] Forward MIME types with media URLs:** `extensions/nextcloud-talk/src/inbound.ts:281`
  - body: The parser reads each file parameter's `mimetype`, but the changed inbound context only passes `MediaUrls`. Nextcloud share links are often extensionless, and downstream attachment classification uses `MediaTypes` before falling back to URL extensions, so these attachments can still be treated as unknown or text/plain and skipped. Carry aligned MIME data, such as `MediaTypes` or media facts, beside the URLs.
  - confidence: 0.85
- **[P3] Return parsed text for valid rich objects:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: For a valid rich-object body whose `message` has no matching placeholders and no file links, `parseRichObjectContent` returns `null`, so the caller still exposes the raw JSON string to the agent. Once parsing succeeds and the object has the expected rich-object shape, return the trimmed `message` so future Talk object types remain human-readable.
  - confidence: 0.74
