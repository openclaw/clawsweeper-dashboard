## Summary

Keep this PR open: current main and the latest release still pass Nextcloud Talk rich-object `object.content` through as raw text without inbound media URLs, so the attachment fix remains useful, but the branch is not merge-ready because URL-only forwarding can miss extensionless media, promoted links need an explicit trust rule, and the live proof still does not show a real file/image share reaching the agent.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing for inbound file shares, threads extracted media URLs through inbound context, relaxes the media-only guard, and adds signed webhook parsing tests.

## Best Possible Solution

Land one coordinated Nextcloud Talk structured-content parser that emits clean text plus aligned URL/MIME metadata, validates promoted file links against the configured backend trust boundary, and includes redacted live file-share proof.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.10` copy Nextcloud Talk `object.content` directly into `text` and have no inbound `MediaUrls`, while the official bot docs define `object.content` as structured JSON with rich-object parameters. I did not establish a live current-main file-share reproduction in this read-only review.

## Solution Assessment

No as-is. Plugin-local rich-object parsing is the right layer, but the best fix must preserve MIME metadata, constrain promoted file links to a trusted Nextcloud origin, and coordinate with the adjacent structured mention parser.
