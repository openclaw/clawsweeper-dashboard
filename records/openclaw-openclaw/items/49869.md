## Summary

Keep this PR open: it targets a real Nextcloud Talk attachment-loss bug, but the branch still drops MIME hints, leaves a valid structured-body raw-JSON fallback, and still lacks after-fix proof that a real file share reaches the patched path.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing for file-share webhook content, forwards extracted media URLs into inbound context, relaxes media-only drops, and adds signed webhook parser tests.

## Best Possible Solution

Unify the Nextcloud Talk structured-body parser so it emits display text plus aligned MediaUrls and MediaTypes, coordinates with the mention and non-message webhook work, and lands with focused tests plus redacted live file-share proof.

## Reproduction Assessment

Yes. A high-confidence source path exists: a signed Nextcloud Talk Create webhook whose object.content contains the documented rich-object file-share JSON remains raw text on current main and produces no media facts; I did not run a live Nextcloud reproduction in this read-only review.

## Solution Assessment

No. Plugin-local parsing is the right boundary, but this patch should preserve MIME types, return parsed display text for all valid structured bodies, coordinate with the active structured parser, and include real file-share proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Forward MIME types with rich-object media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: This only sends MediaUrls into the inbound context. The rich-object file parameters already include mimetype, but without aligned MediaTypes finalizeInboundContext pads application/octet-stream, and extensionless Nextcloud share URLs like /f/42 can remain unknown before image/audio/video handling. Preserve each rich-object mimetype and pass MediaTypes alongside MediaUrls.
  - confidence: 0.91
- **[P3] Return parsed text for valid rich objects:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: When JSON parses as a structured Talk body but has no file URL and no matched placeholder, such as {"message":"hello","parameters":[]}, this returns null and exposes the raw JSON envelope to the agent. Return displayText for valid structured bodies and reserve null for malformed or non-structured input.
  - confidence: 0.87
