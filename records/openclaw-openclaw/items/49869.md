## Summary

Keep open: this PR addresses a real source-reproducible Nextcloud Talk attachment bug, but it is not merge-ready because URL-only media propagation drops MIME hints, actual file-share delivery is still unproven, and an open structured-body parser PR changes the same boundary.

## What This Changes

The branch adds a rich-object parser in Nextcloud Talk webhook handling, threads `mediaUrls` through inbound context, relaxes the empty-body guard, and adds webhook/parser tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text plus aligned media URLs and MIME types, preserves malformed/plain-text fallbacks, and is proven with a real file-share webhook before merge.

## Reproduction Assessment

Yes, at source level. A signed Nextcloud Talk `Create` webhook whose `object.content` is a rich-object file share currently flows through `payloadToInboundMessage()` as raw text on main and has no path to produce `MediaUrls` for the agent.

## Solution Assessment

No. The plugin boundary is the right place to fix this, but the current PR needs MIME propagation, parser coordination, and real file-share proof before it is the best maintainable solution.
