## Summary

Keep open: this PR addresses a real Nextcloud Talk attachment gap, but it is not merge-ready because it drops rich-object MIME data, leaves some valid structured bodies as raw JSON, lacks end-to-end file-share proof, and overlaps another open structured-body parser.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing for file-share webhook content, forwards extracted media URLs into inbound context, relaxes media-only drops, and adds signed-webhook parser tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text, mention facts, media URLs, and aligned MIME types, preserves fallback behavior, respects the channel's network boundary, and is proven with a real file-share webhook.

## Reproduction Assessment

Yes, at source level. A signed Nextcloud Talk Create webhook whose object.content is a rich-object file share currently flows through payloadToInboundMessage as raw text on main and has no path to produce MediaUrls for the agent.

## Solution Assessment

No. The plugin boundary is right, but the current patch needs MIME propagation, parser coordination, and real file-share proof before it is the best maintainable solution.

- **[P1] Propagate MIME types with media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: Forwarding only `MediaUrls` lets `finalizeInboundContext` synthesize `application/octet-stream` for each attachment. Downstream selection keys off MIME or filename extension, and Nextcloud share links like `/f/<id>` are extensionless, so image/file shares can still be classified as unknown and skipped; carry each rich-object `mimetype` beside its URL.
  - confidence: 0.91
- **[P2] Return parsed rich-object text without placeholders:** `extensions/nextcloud-talk/src/monitor.ts:211-213`
  - body: This guard returns `null` for valid rich-object content with no files or placeholder substitutions, so the caller keeps the raw JSON body. Nextcloud Talk docs and live reports show ordinary messages can arrive as `{"message":"hello","parameters":[]}`, so return the parsed `message` for any valid rich object while keeping malformed/plain-text fallback behavior.
  - confidence: 0.84
