## Summary

Keep open: the underlying Nextcloud Talk attachment bug is valid and the PR is in the right plugin boundary, but the current patch still has merge-blocking URL trust, MIME propagation, and rich-object fallback issues.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing, extracts file links into `mediaUrls`, forwards those URLs through inbound handling, and adds webhook parsing tests.

## Best Possible Solution

Revise the plugin patch so valid rich objects always produce parsed display text, promoted file links are constrained to the verified Nextcloud backend or constructed from trusted metadata, MIME hints travel with each URL, media-only delivery is covered, and the changelog records the user-facing fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a signed Create webhook with JSON rich-object `object.content` reaches `payloadToInboundMessage` on current main, which copies that JSON into text and never populates media URLs.

## Solution Assessment

No. The PR targets the right Nextcloud Talk plugin boundary, but it is not the best mergeable fix until URL promotion uses the existing trusted backend boundary, MIME hints are forwarded, and parsed rich-object text does not fall back to raw JSON.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain promoted file links to the trusted backend:** `extensions/nextcloud-talk/src/monitor.ts:193-198`
  - body: The parser accepts any http(s) `parameters[*].link` from rich-object content and forwards it as `MediaUrls`. A signed webhook can still carry user-controlled chat content, so only promote links whose origin matches the verified backend/configured `baseUrl`, or construct the file URL from trusted Nextcloud metadata before downstream media fetching sees it.
  - confidence: 0.87
- **[P2] Forward MIME hints with extracted media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: The rich-object file parameter includes `mimetype`, but the inbound context only receives `MediaUrls`. `finalizeInboundContext` pads missing types as `application/octet-stream`, so extensionless Nextcloud links like `/f/42` can remain unknown to image/audio/video routing.
  - confidence: 0.9
- **[P2] Return parsed text for rich objects without substitutions:** `extensions/nextcloud-talk/src/monitor.ts:209-213`
  - body: For a valid rich object whose `message` has no matching placeholder, `parseRichObjectContent` returns `null`, causing the caller to expose the raw JSON string. Once content is recognized as a rich object, return the trimmed parsed message even when no placeholder was replaced.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `extensions/nextcloud-talk/src/monitor.ts:150`
  - body: This is a user-facing Nextcloud Talk fix, but the PR does not update `CHANGELOG.md`. Add a single-line fix entry with appropriate contributor credit so release notes include the attachment parsing behavior change.
  - confidence: 0.78
