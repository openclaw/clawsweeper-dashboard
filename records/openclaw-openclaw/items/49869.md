## Summary

Keep open: current main and v2026.6.6 still do not parse Nextcloud Talk rich-object file-share payloads into inbound media, and this PR contains useful unique attachment work. It is not merge-ready because it drops MIME hints needed by downstream media classification, promotes links without origin-aware validation, lacks live file-share proof, and overlaps the structured-message parser in https://github.com/openclaw/openclaw/pull/79397.

## What This Changes

The PR updates Nextcloud Talk inbound webhook parsing to turn rich-object file share payloads into display text plus media URLs, with focused webhook parsing tests.

## Best Possible Solution

Consolidate Nextcloud Talk structured payload parsing so mentions and attachments share one decoded rich-object boundary, then pass media URLs and MIME types through the existing inbound context with origin-aware URL validation and live file-share proof.

## Reproduction Assessment

Yes. Source and the official Nextcloud Talk bot contract show current main copies documented JSON `object.content` into text and never populates inbound media URLs; I did not run a live Nextcloud file-share reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Plugin-local parsing is the right owner boundary, but the best fix should consolidate with the adjacent structured parser, preserve MIME metadata, validate promoted links against the trusted Nextcloud origin, and include live file-share proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Forward MIME hints with extracted media URLs:** `extensions/nextcloud-talk/src/monitor.ts:194`
  - body: The parser reads `mimetype` from rich-object file parameters but only forwards the link. `finalizeInboundContext` will then pad these attachments as `application/octet-stream`, and media-understanding classifies image/audio/video attachments by MIME or filename/URL extension; Nextcloud share links like `/f/<id>` can stay `unknown` and never reach vision/audio/video handling. Carry the per-file MIME values through `MediaTypes` alongside `MediaUrls`.
  - confidence: 0.88
- **[P1] Constrain promoted file links to the trusted Nextcloud origin:** `extensions/nextcloud-talk/src/monitor.ts:192`
  - body: This check promotes any `http(s)` rich-object file link into `MediaUrls`, including cleartext HTTP and hosts unrelated to the signed webhook backend. Downstream SSRF guards reduce private-IP exposure, but this path still bypasses the normal media text parser's stricter URL validation and should only promote links from the trusted Nextcloud backend/account origin, or use an explicit channel URL policy with tests.
  - confidence: 0.79
