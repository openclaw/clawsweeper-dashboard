## Summary

Keep this PR open. The bug is still source-reproducible on current main, but the branch is blocked by missing real Nextcloud file-share proof and two narrow correctness findings around MIME propagation and rich-object text fallback.

## What This Changes

The PR parses Nextcloud Talk rich-object file-share webhook content into display text and inbound media URLs, adds an optional inbound media field, relaxes media-only drops, and adds webhook parsing tests.

## Best Possible Solution

Land a repaired Nextcloud Talk plugin fix that parses file-share rich objects into human text plus aligned URL/MIME media facts, preserves parsed text for non-file rich objects, and includes real file-share proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a file-share `object.content` JSON string is passed through as message text, and the inbound context has no media URL/type facts for Nextcloud Talk.

## Solution Assessment

No. Plugin-local parsing is the right boundary, but this patch should carry MIME types with URLs, avoid raw-JSON fallback for valid rich objects, and include real after-fix Nextcloud Talk proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Forward MIME types with rich-object URLs:** `extensions/nextcloud-talk/src/inbound.ts:281`
  - body: This only forwards share URLs into the inbound context. `finalizeInboundContext` pads missing types to `application/octet-stream`, and media selection relies on MIME or URL extension before fetch; extensionless Nextcloud links like `/f/42` can still be classified as unknown. Preserve each rich-object `mimetype` and pass aligned `MediaTypes` with `MediaUrls`.
  - confidence: 0.91
- **[P3] Return parsed rich-object messages without placeholders:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: For a valid rich object whose `message` has no matching placeholders and no files, this returns `null`, so the caller surfaces the raw JSON string instead of the human-readable `message`. Once JSON validates as rich-object content, return the trimmed `displayText`; malformed or non-rich text already exits earlier.
  - confidence: 0.84
