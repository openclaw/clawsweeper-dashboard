## Summary

Keep open: current main still forwards Nextcloud Talk rich-object content as plain text, and this PR targets the right plugin boundary but still has security and media-contract blockers before merge.

## What This Changes

This PR adds Nextcloud Talk rich-object parsing for webhook file shares, threads extracted media URLs into inbound context, relaxes media-only drops, and adds webhook parsing tests.

## Best Possible Solution

Revise the plugin patch so valid rich objects always produce parsed display text, promoted file links are constrained to the trusted Nextcloud origin, MIME hints are forwarded with URLs, media-only inbound delivery is covered, and the user-facing fix is recorded in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a signed Create webhook with object.content set to a Nextcloud rich-object file payload reaches payloadToInboundMessage on main, where the JSON becomes text and no media URL can be produced.

## Solution Assessment

No. The PR is aimed at the correct Nextcloud Talk plugin boundary, but it is not mergeable until URL promotion is tied to the trusted backend, MIME hints are carried through, and parsed text is returned for every valid rich object.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain promoted file links to the trusted backend:** `extensions/nextcloud-talk/src/monitor.ts:193-198`
  - body: The parser accepts any http(s) `parameters[*].link` from rich-object content and forwards it as `MediaUrls`. A signed Nextcloud message can contain user-controlled rich-object data, so only promote links whose origin matches the verified backend/configured `baseUrl`, or construct the URL from trusted Nextcloud file metadata before downstream media fetching sees it.
  - confidence: 0.86
- **[P2] Forward MIME hints with extracted media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: This forwards `MediaUrls` without aligned `MediaTypes`, so `finalizeInboundContext` pads media as `application/octet-stream` and extensionless Nextcloud share links such as `/f/42` can be classified as unknown. Preserve each file parameter's `mimetype` and pass aligned `MediaTypes` with the URLs.
  - confidence: 0.9
- **[P2] Return parsed text for rich objects without substitutions:** `extensions/nextcloud-talk/src/monitor.ts:209-213`
  - body: For a valid rich object whose `message` has no matching placeholders, `parseRichObjectContent` returns `null`, causing the caller to surface the raw JSON string. Once the content has been validated as a rich object, return the trimmed `message` text even when no placeholder was replaced.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `extensions/nextcloud-talk/src/monitor.ts:150`
  - body: This is a user-facing Nextcloud Talk fix, but the PR does not update `CHANGELOG.md`. Add a single-line fix entry with appropriate contributor credit so release notes include the attachment parsing behavior change.
  - confidence: 0.78

## Security Review

Status: needs_attention
