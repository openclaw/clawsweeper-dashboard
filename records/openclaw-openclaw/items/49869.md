## Summary

Keep open; this PR targets a real Nextcloud Talk attachment parsing bug in the right plugin boundary, but the current head is not mergeable because it promotes arbitrary rich-object links into downstream media fetching, drops MIME hints, and still exposes raw JSON for some valid rich objects.

## What This Changes

The PR adds Nextcloud Talk webhook rich-object parsing, a mediaUrls field on inbound messages, passes URLs into inbound context, and adds attachment parsing tests.

## Best Possible Solution

Revise the plugin patch so valid rich objects produce parsed display text, file links are promoted only from the trusted Nextcloud origin, MIME hints are forwarded with URLs, media-only delivery works, and the user-facing fix is captured in the changelog.

## Reproduction Assessment

Yes. Source inspection plus the Nextcloud Talk bot contract gives a high-confidence path: a signed Create webhook whose `object.content` is rich-object JSON reaches `payloadToInboundMessage`, where current main stores the raw JSON as text and cannot populate media URLs.

## Solution Assessment

No. The PR is in the right Nextcloud Talk plugin boundary, but it is not the best mergeable fix until URL promotion is constrained to the trusted backend, MIME hints are propagated, and every valid rich object returns parsed text instead of raw JSON.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain promoted file links to the trusted backend:** `extensions/nextcloud-talk/src/monitor.ts:193-198`
  - body: The parser accepts any http(s) `parameters[*].link` from rich-object content and forwards it as `MediaUrls`. A signed webhook body can still carry user-controlled file metadata, so only promote links whose origin matches the verified backend/configured `baseUrl`, or construct the download URL from trusted Nextcloud metadata before downstream media fetching sees it.
  - confidence: 0.86
- **[P2] Forward MIME hints with extracted media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: The rich-object file parameter includes `mimetype`, but the inbound context only receives `MediaUrls`. `finalizeInboundContext` will pad `MediaTypes` as `application/octet-stream`, so extensionless Nextcloud links like `/f/42` can remain unknown to image/audio/video routing.
  - confidence: 0.9
- **[P2] Return parsed text for rich objects without substitutions:** `extensions/nextcloud-talk/src/monitor.ts:209-213`
  - body: For a valid rich object whose `message` has no matching placeholder, `parseRichObjectContent` returns `null`, causing the caller to expose the raw JSON string. Once the content is recognized as a rich object, return the trimmed parsed message even when no placeholder was replaced.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `extensions/nextcloud-talk/src/monitor.ts:150`
  - body: This is a user-facing Nextcloud Talk fix, but the PR does not update `CHANGELOG.md`. Add a single-line fix entry with appropriate contributor credit so release notes include the attachment parsing behavior change.
  - confidence: 0.78
