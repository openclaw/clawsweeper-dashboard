## Summary

Keep open: this is a plausible Nextcloud Talk attachment bug fix, but the PR is not merge-ready because the proposed path lacks real file-share proof and still misses MIME propagation, structured-body coordination, and channel URL trust handling.

## What This Changes

The PR parses Nextcloud Talk rich-object webhook bodies, extracts file links into inbound media URLs, threads them through inbound context, and adds parser/webhook tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text plus aligned media URLs and MIME types, validates file-link origin against the channel trust boundary, preserves malformed/plain-text fallbacks, and is proven with a real file-share webhook before merge.

## Reproduction Assessment

## Solution Assessment

No. The plugin boundary is right, but the implementation needs MIME propagation, channel-origin validation, parser coordination, and real file-share proof before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Propagate MIME types with media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: The parser extracts `parameters[*].link` but drops each file parameter's `mimetype`, and this line forwards only `MediaUrls`. `finalizeInboundContext` will then pad `MediaTypes` from the webhook object's `mediaType`, which is commonly `text/plain` for the JSON envelope, so extensionless Nextcloud share URLs like `/f/42` are classified as unknown/text instead of image/audio/video. Return aligned MIME values from the rich object and pass them as `MediaTypes`.
  - confidence: 0.9
- **[P2] Return parsed rich-object text without placeholders:** `extensions/nextcloud-talk/src/monitor.ts:211-213`
  - body: For valid Nextcloud Talk rich objects whose `message` has no substituted placeholder, including the live-reported `{"message":"hello","parameters":[]}` shape for plain text, this guard returns `null` and the caller still surfaces the raw JSON envelope. Once JSON parsing succeeds, return the parsed `message` text unless the structure is not a rich object at all.
  - confidence: 0.82
