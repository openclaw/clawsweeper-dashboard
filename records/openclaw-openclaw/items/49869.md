## Summary

Keep open: current main still lacks Nextcloud Talk rich-object attachment parsing, and this PR is aimed at the right plugin boundary, but the patch is not mergeable yet because it promotes untrusted URLs, drops MIME hints, can still surface raw JSON for valid rich objects, and lacks the required changelog entry.

## What This Changes

This PR adds Nextcloud Talk webhook rich-object parsing, threads extracted media URLs through inbound handling, and adds attachment parsing tests.

## Best Possible Solution

Revise the Nextcloud Talk plugin patch so trusted rich-object file shares produce parsed display text, origin-constrained media URLs, aligned MIME types, media-only dispatch, regression tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a signed Create webhook with object.content set to Nextcloud rich-object file JSON reaches payloadToInboundMessage() on current main, where it is copied into text and no media URL fields exist.

## Solution Assessment

No. The PR targets the correct owner plugin, but the mergeable fix should constrain promoted links to the trusted Nextcloud origin, preserve MIME hints, and return parsed text for every valid rich object rather than relying on scheme-only URL checks and placeholder matches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain promoted file links to the trusted backend:** `extensions/nextcloud-talk/src/monitor.ts:193-198`
  - body: The parser accepts any http(s) parameters[*].link from rich-object content and forwards it as MediaUrls. A signed webhook body can still contain user-controlled chat content, so only promote links tied to the verified Nextcloud backend/configured baseUrl, or construct them from trusted Nextcloud metadata before downstream media fetching sees them.
  - confidence: 0.86
- **[P2] Forward MIME hints with extracted media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: The rich-object file parameter includes mimetype, but the patch forwards only MediaUrls. finalizeInboundContext then pads these as application/octet-stream, and extensionless share links such as /f/42 can be classified as unknown instead of image/audio/video, so pass aligned MediaTypes with the URLs.
  - confidence: 0.9
- **[P2] Return parsed text for rich objects without substitutions:** `extensions/nextcloud-talk/src/monitor.ts:209-213`
  - body: For a valid rich object whose message has no matching placeholders, parseRichObjectContent returns null and the caller falls back to the raw JSON string. Once the content validates as a rich object, return the trimmed parsed message even when no placeholder was replaced.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `extensions/nextcloud-talk/src/monitor.ts:150`
  - body: This is a user-facing Nextcloud Talk fix, but the PR does not update CHANGELOG.md. Add a single-line Fixes entry for the attachment parsing behavior so release notes capture it.
  - confidence: 0.78

## Security Review
