## Summary

Keep open: the attachment parsing fix is still useful and is not implemented on current main, but the branch is not merge-ready because the real file-share path is unproven and the diff forwards incomplete/trust-loose media metadata into downstream media fetching.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing for inbound file shares, threads extracted links through `mediaUrls`, relaxes the media-only inbound guard, and adds signed webhook parsing tests.

## Best Possible Solution

Land one coordinated Nextcloud Talk structured-content parser that emits clean text plus aligned URL/MIME metadata, validates promoted file links against the configured backend trust boundary, and includes redacted live file-share proof.

## Reproduction Assessment

Yes, source-level: current main and `v2026.6.10` copy Nextcloud Talk `object.content` directly into `text` and have no inbound `mediaUrls` field, while upstream docs define `object.content` as structured JSON with rich-object parameters. I did not establish a live current-main file-share reproduction in this read-only review.

## Solution Assessment

No as-is. Plugin-local rich-object parsing is the right layer, but the best fix must preserve MIME metadata, constrain promoted file links to a trusted Nextcloud origin, and coordinate with the adjacent structured mention parser.
