## Summary

Keep open: current main still lacks Nextcloud Talk rich-object file-share parsing, but this PR is not merge-ready because it forwards media links without MIME hints, leaves some structured bodies as raw JSON, needs a trusted-origin media boundary, and still lacks real file-share proof.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing for webhook file shares, forwards extracted media URLs into inbound context, allows media-only inbound messages, and adds parser/webhook tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text, mention facts, aligned media URLs/MIME types, and trusted-origin media facts, then prove it with a real file-share webhook.

## Reproduction Assessment

Yes, at source level: current main passes Nextcloud Talk `object.content` through as raw text, never emits `MediaUrls`, and upstream docs define `object.content` as structured JSON. I did not reproduce a live file-share webhook end to end, and the available live report says file-share webhooks did not reach the patched path.

## Solution Assessment

No, not as currently proposed. The plugin boundary is right, but the branch should propagate MIME types, validate file-link origin, coordinate with the structured mention parser, and include real file-share proof before merge.
