## Summary

Keep open: this PR targets a real Nextcloud Talk attachment-delivery bug that current main still has, but it is not merge-ready because real file-share proof is insufficient and the branch drops MIME context needed for extensionless share links.

## What This Changes

The branch adds a Nextcloud Talk rich-object parser for webhook content, threads extracted media URLs through inbound context, relaxes the empty-body guard for media-only messages, and adds parser/webhook tests.

## Best Possible Solution

Land one coordinated Nextcloud Talk structured-body/media parser that emits resolved text plus aligned media URL/type facts, preserves malformed/plain-text fallbacks, and is proven with a redacted live file-share-to-agent trace.

## Reproduction Assessment

Yes, at source level: a signed Nextcloud Talk `Create` webhook with JSON rich-object file content currently becomes raw text on main and has no media URL/type path. I did not establish a live file-share repro, and the supplied live report says that real file-share webhooks did not reach the patched code.

## Solution Assessment

No, not as currently proposed: parsing rich-object file shares is the right direction, but the branch should carry MIME/media facts, return parsed text for valid rich objects, and coordinate with the overlapping structured mention parser before merge.

## Review Findings

Full review comments:

- **[P2] Forward MIME types with media URLs:** `extensions/nextcloud-talk/src/inbound.ts:281`
  - body: The parser reads each file parameter's `mimetype`, but the changed inbound context only passes `MediaUrls`. Nextcloud share links are often extensionless, and downstream attachment classification uses `MediaTypes` before falling back to URL extensions, so these attachments can still be treated as unknown and skipped. Carry aligned MIME data, such as `MediaTypes` or media facts, beside the URLs.
  - confidence: 0.85
- **[P3] Return parsed text for valid rich objects:** `extensions/nextcloud-talk/src/monitor.ts:211-212`

Once parsing succeeds and the object has the expected rich-object shape, return the trimmed `message` so future Talk object types remain human-readable.
  - confidence: 0.74
