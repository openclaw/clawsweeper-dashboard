## Summary

Keep open: current main still has the reported Nextcloud Talk rich-object attachment gap, and this PR is a plausible plugin-local fix, but it still needs contributor real file-share proof plus two focused repairs before merge.

## What This Changes

The PR parses Nextcloud Talk rich-object webhook content, extracts file-share links into inbound media URLs, relaxes media-only inbound filtering, and adds signed webhook parser tests.

## Best Possible Solution

Land one Nextcloud Talk structured-body path that emits display text plus aligned MediaUrls and MediaTypes, preserves plain-text fallback, coordinates with https://github.com/openclaw/openclaw/pull/79397, and includes redacted real file-share proof.

## Reproduction Assessment

Yes, as a source-level path: a signed Nextcloud Talk Create webhook whose object.content is the documented rich-object file-share JSON still becomes raw text on current main with no MediaUrls or MediaTypes. I did not run a live Nextcloud file-share reproduction in this read-only review.

## Solution Assessment

No. Plugin-local parsing is the right boundary, but this branch should forward MIME types, return parsed display text for every valid structured body, and coordinate with the active structured mention parser before merge.
