## Summary

Keep this PR open: current main still leaves Nextcloud Talk rich-object content as raw text, but this branch needs stronger real behavior proof and fixes for MIME propagation, structured text fallback, and URL trust before merge.

## What This Changes

The PR adds a Nextcloud Talk rich-object parser for webhook content, threads extracted file links as inbound media URLs, relaxes media-only drops, and adds webhook parser tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text plus aligned media URLs and MIME types, validates file-link origin against the trusted backend, and is proven with a real file-share webhook before merge.

## Reproduction Assessment

No live high-confidence reproduction is established for the file-share path: source inspection and upstream docs show current main leaves object.content raw, but the live tester could not make actual file-share webhooks reach the patched code.

## Solution Assessment

No, not yet. The plugin-local boundary is right, but the implementation should share one structured-body parser, preserve MIME hints, return parsed text for valid structured bodies, and validate file-link origin before merge.

- **[P1] Propagate MIME types with media URLs:** `extensions/nextcloud-talk/src/inbound.ts:281`
  - body: The rich-object file parameters include `mimetype`, but the branch only forwards `MediaUrls`. `finalizeInboundContext` then pads `MediaTypes` with `application/octet-stream`, and extensionless Nextcloud links such as `/f/42` are classified as unknown instead of image/audio/video, so the central file-share path can still be skipped by media understanding.
  - confidence: 0.89
- **[P2] Return parsed text for structured bodies without placeholders:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: Nextcloud documents `object.content` as a JSON dictionary with `message` and `parameters`, and live testing reported plain messages with `parameters: []`. When no placeholder is substituted, this helper returns `null`, so callers keep showing the raw JSON envelope instead of the human-readable `message`.
  - confidence: 0.84
- **[P2] Constrain extracted links to the trusted backend:** `extensions/nextcloud-talk/src/monitor.ts:196`
  - body: This scheme check accepts any public HTTP(S) URL found in a rich-object file parameter and promotes it into automatic media fetch context. Core SSRF guards block private/internal targets, but this channel should still only treat links from the configured Nextcloud backend or an explicit trusted origin as file-share media.
  - confidence: 0.76
