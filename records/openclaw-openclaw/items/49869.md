## Summary

Keep this PR open: current main still leaves Nextcloud Talk rich-object file shares as raw text, but the branch needs repair before merge because it drops MIME metadata for extracted media URLs and lacks contributor real behavior proof.

## What This Changes

This PR adds Nextcloud Talk rich-object parsing for file-share webhook content, threads extracted media URLs into inbound dispatch, and adds webhook parsing tests.

## Best Possible Solution

Land a repaired Nextcloud Talk plugin fix that parses rich-object file shares into human text plus aligned URL/MIME media context, preserves parsed rich-object text, and coordinates with the structured mention parser if that PR lands first.

## Reproduction Assessment

Yes. Source inspection on current main shows `object.content` is passed through as plain text and the inbound context has no media URL/type handoff, matching the file-share payload described in the PR.

## Solution Assessment

No. Plugin-local rich-object parsing is the right boundary, but this patch should preserve aligned MIME types, avoid raw-JSON fallback for valid rich objects, and include real after-fix Nextcloud Talk proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Forward MIME types with rich-object URLs:** `extensions/nextcloud-talk/src/inbound.ts:281`
  - body: This only forwards share URLs into the inbound context. `finalizeInboundContext` pads missing types to `application/octet-stream`, and media selection relies on MIME or URL extension before fetch; extensionless Nextcloud links like `/f/42` can still be classified as unknown. Preserve each rich-object `mimetype` and pass aligned `MediaTypes` with `MediaUrls`.
  - confidence: 0.91

Once JSON validates as rich-object content, return the trimmed `displayText`; malformed or non-rich text already exits earlier.
  - confidence: 0.84
