Full review comments:

- **[P1] Forward MIME hints with rich-object media:** `extensions/nextcloud-talk/src/inbound.ts:281`
  - body: This only forwards the share links, but Nextcloud file-share links are commonly extensionless (`/f/<id>`). `finalizeInboundContext` will default the media type to `application/octet-stream`, and `resolveAttachmentKind` falls back to the URL extension, so image/audio/video understanding can still skip the attachment this PR is meant to expose. Thread the rich-object `mimetype` through the inbound message and pass matching `MediaTypes` with `MediaUrls`.
  - confidence: 0.9
- **[P2] Return parsed text for valid structured bodies:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: For a valid structured body with no matched placeholders, such as the live-reported `{"message":"hello","parameters":[]}` shape, this returns `null` and the caller sends the raw JSON envelope to the agent. Once JSON parsing succeeds and the shape is a Nextcloud rich object, return the trimmed `message` text even when no substitutions or media links were found.
  - confidence: 0.84
- **[P2] Constrain promoted file links to the trusted media boundary:** `extensions/nextcloud-talk/src/monitor.ts:196-198`
  - body: The parser accepts any `http(s)` file link from the rich-object parameters and forwards it as media. Core media fetches have SSRF guards, but this channel should still prove the link belongs to the trusted Nextcloud backend or reconstruct a known WebDAV/share URL before converting webhook content into a remote fetchable attachment.
  - confidence: 0.8
