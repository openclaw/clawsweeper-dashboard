## Summary

Keep open: current main still does not parse Nextcloud Talk rich-object file shares into inbound media, but this PR is not merge-ready because the patch misses MIME propagation, link-origin hardening, non-placeholder rich-object text handling, current-main alignment, and sufficient real file-share proof.

## What This Changes

The PR adds a Nextcloud Talk rich-object parser that resolves file placeholders into text, forwards file links as inbound `MediaUrls`, permits media-only inbound messages, and adds webhook parsing tests.

## Best Possible Solution

Carry attachment parsing into the current Nextcloud Talk structured-body path, preserving per-file MIME types, handling all valid rich-object text, constraining promoted links to the trusted Nextcloud origin, and requiring live file-share proof before merge.

## Reproduction Assessment

## Solution Assessment

No, not as submitted. Parsing at the plugin boundary is the right layer, but this implementation needs MIME propagation, trusted-origin URL handling, non-placeholder rich-object text handling, and coordination with the current structured mention parser.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve MIME hints for rich-object media:** `extensions/nextcloud-talk/src/monitor.ts:198`
  - body: The rich-object parameter includes `mimetype`, but the parser only returns `mediaUrls`. `finalizeInboundContext` will then fill `MediaTypes` with `application/octet-stream`, and media selection falls back to URL extensions; common Nextcloud share links like `/f/42` have no image extension, so the attachment can still be skipped by vision/audio/video handling. Forward an aligned `MediaTypes` array with the URLs.
  - confidence: 0.88
- **[P1] Constrain promoted links to the trusted Nextcloud origin:** `extensions/nextcloud-talk/src/monitor.ts:196-198`
  - body: This accepts any `http(s)` value from a rich-object file parameter and forwards it as agent media. Even though the downstream fetch guard blocks private IPs, this creates a new arbitrary public-URL fetch path from webhook content; validate the URL against the signed backend/account base URL or construct the download URL from trusted Nextcloud config before promoting it.
  - confidence: 0.79
- **[P2] Return parsed text for rich objects without placeholders:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: For a valid rich-object body such as `{"message":"hello","parameters":[]}`, this guard returns `null`, so the caller falls back to the raw JSON string. The live test comment says plain messages can arrive with exactly that shape, so the patch still leaks JSON into agent prompts instead of using the parsed `message`.
  - confidence: 0.82
