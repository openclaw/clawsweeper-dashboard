## Summary

This PR should stay open because current main and the latest release still copy Nextcloud Talk rich-object `object.content` as raw text and do not expose inbound media URLs, and no merged replacement owns file-share media extraction. The branch remains useful but is not merge-ready until the attachment path has real file-share proof plus aligned MIME metadata and a tighter link trust boundary.

## What This Changes

The PR adds Nextcloud Talk rich-object content parsing in the webhook monitor, threads optional inbound `mediaUrls` into the finalized context, and adds signed webhook parser tests.

## Best Possible Solution

Land a coordinated Nextcloud Talk structured-content parser that emits clean text plus aligned URL and MIME metadata, validates file links against the configured backend origin, and includes live file-share proof.

Yes, source-level: current main copies Nextcloud Talk `object.content` directly into `text` and has no inbound `mediaUrls` field, while the upstream docs define that content as JSON with rich-object parameters. ## Solution Assessment

No as-is. Plugin-local rich-object parsing is the right layer, but the best fix must preserve MIME metadata, tighten file-link trust, and coordinate with the adjacent structured mention parser instead of landing a partial parallel parser.
