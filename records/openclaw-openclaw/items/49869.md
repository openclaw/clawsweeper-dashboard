## Summary

This PR still targets a real Nextcloud Talk media-loss bug, but it should stay open because the branch has missing after-fix file-share proof and a concrete attachment-routing defect: rich-object MIME types are dropped before media understanding, so common extensionless Nextcloud share links can still be classified as unknown. The recent live test comment is useful evidence for startup and plain-text compatibility, but it explicitly did not validate the file-share webhook path end to end.

## What This Changes

The branch parses Nextcloud Talk rich-object webhook content, extracts file-share links into inbound media URLs, allows media-only inbound messages, forwards those URLs into inbound context, and adds parser/webhook tests.

## Best Possible Solution

Land one coordinated Nextcloud Talk structured-body parser that emits resolved text plus aligned media URLs and MIME types, preserves plain-text fallback behavior, and is proven with a redacted live file-share-to-agent trace.

## Reproduction Assessment

Yes, at source level: a signed Nextcloud Talk `Create` webhook whose `object.content` is a JSON rich-object file share currently has no path on main to produce media URLs for the agent. The available live report did not reproduce the file-share path end to end because the webhook never reached the patched code.

## Solution Assessment

No, not as currently proposed: parsing rich-object file shares is the right direction, but the best fix should carry MIME types with URLs and coordinate with the overlapping structured mention parser before merge.
