## Summary

Keep open: this is a valid Nextcloud Talk attachment bug, but the PR is not merge-ready because the changed path still lacks target file-share proof, drops MIME hints needed for media understanding, and overlaps an open structured-body parser change.

## What This Changes

The branch adds Nextcloud Talk rich-object parsing in the webhook monitor, forwards extracted media URLs into inbound context, relaxes empty-body handling, and adds signed webhook/parser tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text, mention facts, media URLs, and aligned MIME types, then prove it with a real file-share webhook before merge.

## Reproduction Assessment

Yes, at source level. A signed Nextcloud Talk Create webhook whose object.content is a rich-object file share currently flows through payloadToInboundMessage as raw text on main and has no path to produce MediaUrls for the agent.

## Solution Assessment

No. The plugin boundary is the right place to fix this, but this PR should first propagate MIME types, share the structured-body parser with mention handling, and provide real file-share proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Propagate MIME types with rich-object media URLs:** `extensions/nextcloud-talk/src/inbound.ts:278`
  - body: The new parser drops each file parameter's `mimetype` and the inbound context only sends `MediaUrls`. `normalizeAttachments` needs `MediaTypes`/`MediaType` or a filename extension to classify image/audio/video attachments; Nextcloud share links like `/f/42` are commonly extensionless, so the shared file can still be treated as unknown and skipped by media understanding. Carry aligned MIME types with the URLs.
  - confidence: 0.86
- **[P2] Return parsed text for rich objects without replacements:** `extensions/nextcloud-talk/src/monitor.ts:211-213`
  - body: When JSON parses and has a `message` field but no media URL and no matched placeholder, the helper returns `null`, so the caller surfaces the raw JSON string instead of the human-readable message. Nextcloud's bot docs show structured message bodies and live testing reported plain messages with `parameters: []`; return the parsed message for valid rich objects while keeping malformed JSON on the raw fallback path.
  - confidence: 0.78
