## Summary

Keep open: current main and v2026.6.6 still leave Nextcloud Talk file-share rich objects as raw text with no inbound media fields, and this PR contains useful unique bug-fix work. It is not merge-ready because it promotes rich-object links without a trusted Nextcloud-origin policy, drops MIME metadata needed for extensionless share links, lacks real file-share proof, and overlaps the open structured-body parser work in https://github.com/openclaw/openclaw/pull/79397.

## What This Changes

The PR adds Nextcloud Talk rich-object parsing in the webhook monitor, threads `mediaUrls` through inbound context, and adds signed webhook tests for file-share payloads.

## Best Possible Solution

Land a coordinated Nextcloud Talk structured-body parser that feeds mention and attachment handling, restricts or reconstructs trusted file links, forwards MIME metadata, and includes live file-share proof.

## Reproduction Assessment

Yes. Current main and v2026.6.6 source copy Nextcloud Talk `object.content` into `text` and never pass inbound `MediaUrls`, matching the reported raw rich-object attachment loss; I did not run a live Nextcloud file-share server in this read-only review.

## Solution Assessment

No, not as submitted. Plugin-local structured parsing is the right area, but the best fix needs one canonical parser boundary, trusted file-link policy, MIME propagation, and live file-share proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Constrain promoted file-share links:** `extensions/nextcloud-talk/src/monitor.ts:196-198`
  - body: `parseRichObjectContent` accepts any `http(s)` `parameters[*].link` and forwards it as `MediaUrls`. That lets a signed webhook body turn ordinary chat content into arbitrary public remote media fetches; constrain the link to the configured Nextcloud origin or reconstruct a trusted file-download URL before promoting it.
  - confidence: 0.84
- **[P1] Forward MIME hints with media URLs:** `extensions/nextcloud-talk/src/monitor.ts:198`
  - body: The rich-object file parameter includes `mimetype`, but the PR only pushes the link. Downstream context then defaults `MediaTypes` to `application/octet-stream`, and media selection relies on MIME or URL extension, so extensionless Nextcloud shares like `/f/42` can still be skipped by image/audio/video handling.
  - confidence: 0.87
- **[P2] Return parsed text for valid rich objects:** `extensions/nextcloud-talk/src/monitor.ts:211-212`
  - body: When a valid rich-object body has no file URL and no placeholder substitution, this returns `null` and the caller sends the raw JSON envelope to the agent. Real Nextcloud Talk traffic can encode even plain messages as `{message, parameters: []}`, so successful parsing should use the human-readable `message` field instead of falling back to raw JSON.
  - confidence: 0.75
