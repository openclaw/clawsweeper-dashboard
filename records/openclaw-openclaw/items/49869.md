## Label Justifications

- `merge-risk: 🚨 compatibility`: The PR changes structured Nextcloud Talk body handling and overlaps another open structured-body parser path.

## Summary

This PR is still useful because current main and the latest release still pass Nextcloud Talk rich-object content through as raw text and do not forward inbound media, but it is not merge-ready as written because the parser leaves real structured text cases raw, omits MIME data needed by downstream media selection, and promotes file links without a channel-origin trust check.

## What This Changes

The PR adds rich-object parsing to Nextcloud Talk webhook messages, forwards extracted `mediaUrls` to inbound context, allows media-only inbound messages, and adds signed webhook parser tests.

## Best Possible Solution

Use one Nextcloud Talk structured-body parser that returns display text, mention facts, aligned media URLs and MIME types, validates media origin against the trusted Nextcloud source, and is proven with a real file-share webhook.

## Reproduction Assessment

Yes, at source level: current main passes Nextcloud Talk `object.content` through as raw text, drops empty bodies, and never emits `MediaUrls`. Live end-to-end proof for an actual file-share webhook reaching this patched path is still missing.

## Solution Assessment

No, not as currently proposed. The plugin boundary is right, but the PR should carry MIME types, validate media-link origin, coordinate with the structured mention parser, and include real file-share proof before merge.
