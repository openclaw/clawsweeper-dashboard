## Summary

Keep open. Current main still keeps the relevant-memory stripper on assistant-visible and iMessage-specific paths, while normal outbound reply normalization and streaming block replies do not apply it; #74436 is an open closing PR, not a merged fix.

## What This Changes

Strip reflected `<relevant-memories>` blocks from assistant text before delivery through outbound messaging channels.

## Best Possible Solution

Land #74436 or a replacement that reuses the existing code-fence-aware relevant-memory sanitizer at every outbound delivery boundary, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` path and observe that current main does not apply the memory-tag stripper there.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing shared sanitizer at central outbound boundaries rather than adding a Telegram-only regex; maintainers should make sure streaming/preview paths are covered too.
