## Summary

Keep open. Current main still has `<relevant-memories>` stripping in the shared assistant-visible/UI sanitizer, but not at all outbound delivery boundaries; the linked open PR #74436 targets this issue with closing syntax and is still unmerged, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` memory-scaffolding blocks from assistant responses before they are delivered through messaging channels such as Telegram and Discord.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible sanitizer at all assistant-origin outbound delivery boundaries, including reply normalization and streaming/block-reply paths, then add focused regression coverage for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the streaming `emitBlockChunk` path and observe that current main does not apply the shared memory-tag stripping there.

## Solution Assessment

Yes. The maintainable direction is to reuse the existing code-fence-aware sanitizer at shared delivery boundaries rather than adding channel-specific regexes or relying on providers not to echo the context.
