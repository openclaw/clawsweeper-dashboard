## Summary

Keep open. Current main still sanitizes this scaffold only on assistant-visible/UI-style paths, while shared outbound channel normalization uses `sanitizeUserFacingText` without relevant-memory stripping; #74436 is open and explicitly targets the gap, so this issue should stay open until that fix or a replacement merges.

## What This Changes

The issue asks OpenClaw to strip reflected `<relevant-memories>...</relevant-memories>` scaffolding from assistant text before delivery through outbound messaging channels.

## Best Possible Solution

Review and land #74436 or a replacement that applies the existing code-fence-aware memory-tag stripper at the shared outbound delivery boundary, with regression coverage for normalized replies and any streaming/preview paths that can reach external channels.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: memory auto-recall can prepend `<relevant-memories>` context, and current outbound normalization routes reflected assistant text through `sanitizeUserFacingText` without calling the memory-tag stripper. I did not establish a live Telegram/Discord run during this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing code-fence-aware relevant-memory stripper at shared outbound sanitizer boundaries rather than adding a channel-specific regex. #74436 follows that direction, but the final merged fix should cover any streaming or partial-preview delivery path that can reach users.
