## Summary

Keep open. Current main still has a valid source-level outbound disclosure path for reflected `<relevant-memories>` blocks, and the linked closing PR #74436 is still an open implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to strip reflected `<relevant-memories>` blocks from outbound assistant messages before delivery to Telegram, Discord, Slack, iMessage, and similar channels.

## Best Possible Solution

Land #74436 or a replacement that reuses the code-fence-aware memory-tag stripper at all outbound delivery boundaries, including streaming or partial reply paths, with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` path and current main has no memory-tag stripping step there. I did not run a live Telegram/Discord integration because this is a read-only review and `pnpm` is not available in the checkout.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing code-fence-aware stripper at shared outbound delivery boundaries; a Telegram-only regex would under-cover the reported cross-channel leak.
