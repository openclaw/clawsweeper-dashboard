## Summary

Keep open. Current main still has a code-fence-aware `<relevant-memories>` stripper only on the assistant-visible sanitizer path, while normal outbound channel delivery and streaming block replies still use paths that do not strip these memory blocks; #74436 is an open, unmerged closing PR rather than a shipped fix.

## What This Changes

Request to strip reflected `<relevant-memories>` memory-context blocks from assistant text before delivery through outbound messaging channels.

## Best Possible Solution

Land #74436 or a replacement that reuses the code-fence-aware relevant-memory sanitizer at normal and streaming outbound channel boundaries, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` block-reply path and observe that current main does not apply the shared memory-tag stripper there.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing code-fence-aware assistant-visible sanitizer at central outbound delivery boundaries; a Telegram-only regex would under-cover the reported cross-channel leak.
