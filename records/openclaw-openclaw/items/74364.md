## Summary

Keep open. Current main still strips `<relevant-memories>` only on the shared assistant-visible/UI sanitizer path, while normal outbound reply normalization and streaming block replies do not apply that sanitizer; the direct closing PR #74436 is open and unmerged.

## What This Changes

The issue asks OpenClaw to strip reflected `<relevant-memories>` blocks from outbound channel messages before delivery to Telegram, Discord, Slack, iMessage, and similar channels.

## Best Possible Solution

Land #74436 or a replacement that reuses the existing code-fence-aware memory-block sanitizer at central normal and streaming outbound channel boundaries, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` path and observe that current main does not call the relevant-memory stripper there.

## Solution Assessment

Partly. Reusing the existing code-fence-aware sanitizer at `sanitizeUserFacingText` is the right normal-delivery boundary, but the best fix should also account for streaming block replies before maintainers close this issue.
