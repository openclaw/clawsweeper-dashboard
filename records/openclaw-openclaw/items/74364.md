## Summary

Keep open. Current main still has outbound channel delivery paths that call `sanitizeUserFacingText` or the streaming block-reply cleaner without applying the existing `<relevant-memories>` stripper, and #74436 is an open, unmerged closing PR rather than a shipped fix.

## What This Changes

Strip reflected `<relevant-memories>` blocks from assistant text before it is delivered through outbound messaging channels such as Telegram, Discord, Slack, and iMessage.

## Best Possible Solution

Review and land #74436 or a replacement that reuses the code-fence-aware sanitizer at all shared outbound channel delivery boundaries, with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` block-reply path and observe that current main does not apply the shared memory-tag stripper there.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing code-fence-aware assistant-visible sanitizer at central outbound delivery boundaries; a Telegram-only regex would under-cover the reported cross-channel leak.
