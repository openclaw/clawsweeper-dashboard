## Summary

Keep open. Current main still leaves reflected `<relevant-memories>` blocks outside the normal outbound channel sanitizer and streaming block-reply path, and #74436 is an open, unmerged closing PR rather than a shipped fix.

## What This Changes

Strip reflected `<relevant-memories>` blocks from assistant responses before outbound delivery to messaging channels.

## Best Possible Solution

Land #74436 or a replacement that applies the existing code-fence-aware memory-tag sanitizer at every outbound channel delivery boundary, with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` block-reply path and observe that current main does not apply the memory-tag stripper there.

## Solution Assessment

Yes for the direction, but the complete fix should cover all outbound delivery boundaries. Reusing the existing code-fence-aware sanitizer is narrower and safer than adding a Telegram-only regex.
