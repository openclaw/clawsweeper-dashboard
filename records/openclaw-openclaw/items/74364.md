## Summary

Keep open. Current main still permits reflected memory scaffolding to pass through shared outbound normalizers and block-reply streaming, and #74436 is an open closing PR rather than a merged fix.

## What This Changes

The issue asks OpenClaw to strip reflected `<relevant-memories>` blocks from assistant replies before sending them through channel transports.

## Best Possible Solution

Shared outbound delivery sanitizers should reuse the code-fence-aware memory-tag stripper across normal and streaming channel paths, with regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: reply text containing `<relevant-memories>...</relevant-memories>` reaches `normalizeReplyPayload`/`sanitizeUserFacingText` or `emitBlockChunk` without memory-tag stripping before channel delivery.

## Solution Assessment

Yes. The maintainable direction is a shared outbound-boundary sanitizer using the existing code-fence-aware stripper, not a Telegram-only regex; maintainers should ensure the streaming path is covered before closing.
