## Summary

Keep open. Current main still has outbound delivery paths that do not strip reflected `<relevant-memories>` blocks, and the linked closing PR #74436 remains open and unmerged in the provided context, so this issue should stay open until that fix is reviewed, expanded if needed, and landed.

## What This Changes

Strip reflected `<relevant-memories>` plugin context from assistant responses before they are delivered through outbound messaging channels while preserving legitimate literals such as code-fenced examples.

## Best Possible Solution

Land or replace #74436 with a fix that reuses the existing code-fence-aware assistant-visible sanitizer at normal outbound delivery and streaming/block-reply boundaries, with regression tests for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` path and observe that current main does not apply the shared memory-tag stripper there.

## Solution Assessment

Yes. Reusing the shared sanitizer at outbound delivery boundaries is the narrow maintainable direction; a channel-specific regex would be more brittle and could miss other channel paths.
