## Summary

Keep open. Current main still strips `<relevant-memories>` only in the assistant-visible sanitizer path, while normal outbound channel reply normalization and streaming block replies do not apply that memory-tag sanitizer; #74436 is an open, unmerged closing PR rather than a shipped fix.

## What This Changes

Strip reflected `<relevant-memories>` blocks from assistant text before delivery to Telegram, Discord, Slack, iMessage, and other messaging channels.

## Best Possible Solution

Land #74436 or a replacement that reuses the existing code-fence-aware sanitizer at every outbound channel delivery boundary, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` block-reply path and observe that current main does not apply the shared memory-tag stripper there.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing code-fence-aware sanitizer at central outbound delivery boundaries; a Telegram-only regex would under-cover the reported cross-channel leak.
