## Summary

Keep open: current main still routes normal reply text through `sanitizeUserFacingText` without stripping `<relevant-memories>` blocks. Some channel adapters now call `sanitizeAssistantVisibleText`, but the shared/fallback outbound paths remain uncovered and the direct fix PRs are closed unmerged.

## What This Changes

The issue requests stripping `<relevant-memories>` prompt-memory scaffolding from assistant text before delivery through Telegram, Discord, Slack, iMessage, and other channel surfaces.

## Best Possible Solution

Land or sponsor a shared outbound/user-facing sanitizer fix with focused regression tests and redacted real channel proof, or deliberately fold this exact memory-tag case into the broader sanitizer boundary tracked by https://github.com/openclaw/openclaw/issues/90684.

## Reproduction Assessment

Yes, source-level: current main has a code-fence-aware memory-tag stripper, but `sanitizeUserFacingText` does not call it while normal reply normalization does call `sanitizeUserFacingText`. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Yes on direction, but current main is not complete. The shared outbound/user-facing sanitizer boundary is the narrow maintainable fix for this tag leak; per-channel sanitizer hooks are only partial coverage.
