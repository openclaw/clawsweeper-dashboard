## Summary

Keep open. Current main still has the memory-tag stripping helper on the shared assistant-visible path, but the outbound user-facing sanitizer and streaming block-reply path do not apply it; open PR #74436 also targets this issue with closing syntax and is still unmerged in the provided context.

## What This Changes

Strip memory plugin `<relevant-memories>` scaffolding from assistant text before outbound messaging-channel delivery.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible memory sanitizer at every assistant-origin outbound delivery boundary, including normal reply normalization and text-end/block-reply streaming, then add focused regression coverage for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the text-end block-reply path and observe that current main does not apply the shared memory-tag sanitizer there.

## Solution Assessment

Yes. The maintainable fix is to reuse the existing code-fence-aware sanitizer rather than adding a channel-specific regex, but the active PR should be reviewed or expanded so streaming/block-reply delivery is covered before this issue closes.
