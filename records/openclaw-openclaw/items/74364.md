## Summary

Keep open. Current main has a shared code-fence-aware sanitizer for assistant-visible memory scaffolding, but the outbound user-facing sanitizer and non-phase block-reply streaming path still do not consistently apply it; open PR #74436 already targets this issue and should be reviewed before any cleanup close.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` / `<relevant_memories>` blocks from outbound channel messages across Telegram, Discord, Slack, iMessage, and similar messaging surfaces while preserving literal code-fence content.

## Best Possible Solution

Review and adjust #74436, or a replacement, so the existing code-fence-aware assistant-visible sanitizer is applied at every assistant-origin outbound delivery boundary, including `sanitizeUserFacingText`, non-phase `text_end` block replies, `message_end` fallbacks, and preview/partial delivery where applicable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send assistant text containing `<relevant-memories>private</relevant-memories>` through a non-phase `text_end` block-reply path; current main appends the raw chunk and `emitBlockChunk` strips think/final/tool scaffolding but not memory scaffolding.

## Solution Assessment

Yes. Reusing the shared code-fence-aware sanitizer at outbound delivery boundaries is the narrow maintainable direction; the current open PR may need expansion before it fully closes the streaming/channel leak.
