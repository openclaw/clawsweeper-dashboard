## Summary

Keep open. Current main still has a source-level outbound disclosure path for reflected `<relevant-memories>` blocks, and the only direct fix PR was closed unmerged after missing real behavior proof.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` scaffolding from outbound channel replies before Telegram, Discord, and similar channels deliver assistant text.

## Best Possible Solution

Land a shared outbound sanitizer fix that reuses the code-fence-aware memory-tag stripper at normal and streaming delivery boundaries, preserves SDK API boundaries, and validates the result with focused regression tests plus real Telegram or channel-runtime proof.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the streaming `emitBlockChunk` path; current main still lacks memory-tag stripping at those delivery boundaries.

## Solution Assessment

Yes, the narrow maintainable direction is a shared outbound sanitizer rather than a Telegram-only regex. The earlier PR used the right boundary in spirit, but a replacement should avoid accidental public SDK export drift, cover streaming/Telegram, and include real behavior proof.
