## Summary

Keep open. Current main still has `<relevant-memories>` stripping on the shared assistant-visible/UI path, but not consistently on outbound user-facing and streaming block-reply delivery paths; open PR #74436 explicitly targets this issue with closing syntax and is still unmerged in the provided context.

## What This Changes

Strip memory plugin `<relevant-memories>` scaffolding from assistant responses before outbound channel delivery while preserving literal code-fenced content.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible sanitizer at all assistant-origin outbound delivery boundaries, including normal reply normalization and text-end/block-reply streaming, then add focused regression coverage for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the text-end block-reply path and observe that current main does not apply the shared memory-tag stripping there.

## Solution Assessment

Yes, the maintainable direction is to reuse the existing code-fence-aware sanitizer instead of adding a channel-specific regex. The linked PR should be reviewed or expanded so streaming/block-reply delivery is covered before this issue closes.
