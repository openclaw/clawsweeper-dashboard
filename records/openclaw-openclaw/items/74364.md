## Summary

Keep open. Current main still has the reported outbound disclosure path: the memory plugin can inject `<relevant-memories>` through `prependContext`, the shared assistant-visible sanitizer can strip it, but the normal channel reply sanitizer and streaming block-reply path do not call that sanitizer. PR #74436 is an open closing PR for this issue, so the issue should remain open until that PR or a successor lands.

## What This Changes

Strip reflected `<relevant-memories>...</relevant-memories>` blocks from assistant responses before they are delivered to outbound messaging channels such as Telegram and Discord.

## Best Possible Solution

Reuse the existing code-fence-aware assistant-visible sanitizer at every assistant-origin outbound delivery boundary, including normal reply normalization and streaming/block replies, then land PR #74436 or a successor with focused regression tests for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the block-reply chunk path and observe that current main does not call `stripRelevantMemoriesTags` there.

## Solution Assessment

Yes, the best fix direction is to reuse the shared code-fence-aware sanitizer at outbound delivery boundaries. A channel-specific regex would be easier to miss on another channel path, and the linked PR should be reviewed for full delivery-path coverage before landing.
