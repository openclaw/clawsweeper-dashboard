## Summary

Keep open. Current `main` still strips `<relevant-memories>` only on the shared assistant-visible sanitizer path, while normal outbound reply normalization and streaming block replies do not apply that sanitizer; the linked fix PR is still open and unmerged.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` blocks from assistant responses before delivery through messaging channels.

## Best Possible Solution

Land #74436 or a replacement that reuses the code-fence-aware memory sanitizer at central normal and streaming outbound delivery boundaries, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` path and observe that current `main` does not call the memory-tag stripper there.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing code-fence-aware sanitizer at shared outbound boundaries; a Telegram-only regex would under-cover the reported cross-channel leak.
