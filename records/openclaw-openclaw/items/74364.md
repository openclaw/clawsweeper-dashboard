## Summary

Keep open: current main only partially addresses the report. The assistant-visible final/history sanitizer strips `<relevant-memories>`, but outbound block-reply and streaming delivery still pass through `sanitizeUserFacingText`, which does not remove those blocks; the related fix PR was closed unmerged and no canonical replacement was found.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` blocks from outbound messaging-channel assistant text so internal memory context is not shown to users.

## Best Possible Solution

Land a maintainer-approved narrow fix at the shared outbound delivery sanitizer/block-reply boundary that reuses the existing code-fence-aware memory stripper and adds regression coverage for final and streaming channel replies.

## Reproduction Assessment

Yes, as a source-level reproduction: a reflected `<relevant-memories>...</relevant-memories>` block can pass through block-reply delivery because `emitBlockChunk` and `sanitizeUserFacingText` do not strip that tag family. I did not run a live channel or test command because the review was constrained to read-only inspection.

## Solution Assessment

Yes, the best fix direction is to reuse the existing code-fence-aware relevant-memory stripping at the shared outbound delivery boundary. A channel-specific regex would be more brittle and would leave sibling delivery paths at risk.
