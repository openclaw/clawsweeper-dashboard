## Summary

Keep open. Current main still has code-fence-aware `<relevant-memories>` stripping on the shared assistant-visible path, but the outbound user-facing sanitizer and raw streaming block-reply paths do not consistently apply it; the open linked PR #74436 uses closing syntax and should be reviewed or expanded before this issue closes.

## What This Changes

Request to strip memory-plugin `<relevant-memories>` scaffolding from assistant responses before those responses are delivered through outbound messaging channels.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible sanitizer at every assistant-origin outbound delivery boundary that can reach channels, including normal reply payload normalization and streaming/block-reply delivery, then add focused regression tests for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `sanitizeUserFacingText` or a normal `normalizeReplyPayload` path, and separately through the raw streaming block-reply chunk path; current main does not apply the shared memory-tag sanitizer at those boundaries.

## Solution Assessment

Yes, the best direction is a shared delivery-boundary sanitizer rather than channel-specific regexes. The linked PR should be reviewed and likely expanded so streaming/block-reply paths are covered before the issue closes.
