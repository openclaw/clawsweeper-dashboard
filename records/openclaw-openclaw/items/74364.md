## Summary

Keep open. Current main still has outbound channel delivery paths that can pass reflected `<relevant-memories>` blocks through unchanged, and the linked closing PR #74436 is still open in the provided context rather than merged.

## What This Changes

The issue asks OpenClaw to strip reflected `<relevant-memories>...</relevant-memories>` scaffolding from assistant text before delivering replies to messaging channels such as Telegram and Discord.

## Best Possible Solution

Land or replace #74436 with a shared, code-fence-aware outbound sanitizer applied at the normal channel reply and block-reply delivery boundaries, with regression tests for stripped reflected memory blocks and preserved code-fenced literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk` path and observe that current main does not call the shared memory-tag stripper there.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse the existing code-fence-aware assistant-visible sanitizer at outbound delivery boundaries; a Telegram-only or channel-specific regex would leave other channel paths exposed.
