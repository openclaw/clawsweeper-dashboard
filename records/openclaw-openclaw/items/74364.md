## Summary

Keep open. Current main still strips `<relevant-memories>` on the shared assistant-visible path, but the normal outbound channel sanitizer and streaming block-reply path do not consistently apply that sanitizer; open PR #74436 explicitly targets this issue and remains unmerged in the provided context.

## What This Changes

Request to strip model-echoed `<relevant-memories>...</relevant-memories>` blocks before assistant replies are delivered through messaging channels.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible sanitizer at all assistant-origin outbound delivery boundaries, including normal reply normalization and streaming/block replies, then add focused regression coverage for stripped memory blocks and preserved code-fence literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the `emitBlockChunk`/`emitBlockReply` path and observe that current main does not apply the shared memory-tag stripper there.

## Solution Assessment

Yes. Reusing the shared code-fence-aware sanitizer at outbound delivery boundaries is the narrow maintainable fix; a channel-specific regex would be more brittle and easier to miss on another channel path.
