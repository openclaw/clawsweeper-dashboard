## Summary

Keep open. Current main still has a source-level outbound disclosure path for reflected `<relevant-memories>` blocks, the direct fix PR was closed unmerged, and the broader registry RFC is complementary rather than a replacement for this delivery-boundary bug.

## What This Changes

The issue asks OpenClaw to strip reflected `<relevant-memories>` blocks from assistant responses before sending them through outbound messaging channels.

## Best Possible Solution

Land a shared, code-fence-aware outbound sanitizer at both normal and streaming delivery boundaries, keep helper/API exposure intentional, and validate it with focused regression tests plus real Telegram or equivalent channel proof.

## Reproduction Assessment

Yes. A high-confidence source-level path is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `normalizeReplyPayload` or the streaming `emitBlockChunk` path; current main does not remove that block before delivery.

## Solution Assessment

Yes in direction, but it needs maintainer review. The narrow maintainable fix is a shared outbound delivery-boundary sanitizer, not a Telegram-only regex and not waiting for the broader registry RFC to land first.
