## Summary

Keep open. Current main still decodes and validates `WizardStep.sensitive`, but the bundled macOS `openclaw-mac wizard` text prompt ignores it and routes sensitive text through an echoing `readLine()` path that can also print bracketed initial values.

## What This Changes

Honor `WizardStep.sensitive` in the bundled macOS `openclaw-mac wizard` CLI so credential text prompts do not echo input or show initial secret values.

## Best Possible Solution

Implement the fix in `apps/macos/Sources/OpenClawMacCLI/WizardCommand.swift`: branch on `step.sensitive == true`, use a termios-based no-echo reader that preserves long inputs, suppress sensitive initial-value display, and add Swift/manual proof for the macOS CLI surface.

## Reproduction Assessment

Yes, at source level: a `WizardStep` with `type: "text"`, `sensitive: true`, and an initial value reaches `promptAnswer(for:)`, which ignores `sensitive` and calls the echoing `readLineWithPrompt`. I did not run an interactive macOS TTY reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in the macOS CLI renderer, because the protocol field already exists and the SwiftUI app already treats it as a secure input signal.
