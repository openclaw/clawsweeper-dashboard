Keep open. This is an active protected-label PR, and the patch still leaves an approval-boundary gap for unquoted `$*` shell positional carriers while the PR body does not include actual after-fix runtime output.

## What This Changes

The PR adds shell positional-carrier traversal to inline command risk detection and extends command-analysis unit coverage.

## Best Possible Solution

Align the detector with the existing positional-carrier allowlist policy by failing closed or conservatively detecting unquoted `$*` carriers, then add a regression that exercises strictInlineEval through the allowlist policy path.

## Reproduction Assessment

Yes, from source inspection: `sh -lc '$0 $*' find . -exec id {} ';'` is accepted as a direct positional carrier by current allowlist matching, while the PR’s new detector returns null for `star`. I did not execute a runtime probe to keep the checkout read-only.

## Solution Assessment

No. The direction is right, but the implementation should cover or fail closed on unquoted `$*` and prove the strictInlineEval policy path, not only direct detector calls.
