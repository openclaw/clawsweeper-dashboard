## Summary

Keep open. Current main narrows some causes, but source and tests still show REM phase artifacts can be written while timeout, non-ok, or empty narrative generation leaves `DREAMS.md` absent, and the Dreams UI still has no fallback reader beyond the diary RPC.

## What This Changes

The issue asks memory-core dreaming to create or update `DREAMS.md` after REM writes phase output so the Control UI Dreams tab is populated.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always append a safe fallback diary entry when REM has material but narrative generation fails, or show phase reports in the Dreams UI when no diary exists.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: run REM with eligible entries and make the narrative subagent return `timeout`, another non-ok status, or no assistant text. Current tests assert the timeout path leaves `DREAMS.md` absent.

## Solution Assessment

Unclear. The request identifies a real gap, but the narrowest maintainable fix needs a maintainer decision between fallback diary writes and a phase-report UI fallback.
