## Summary

Current main still allows REM phase reports and phase signals to be written while Dream Diary generation remains detached and best-effort, so `DREAMS.md` can still be absent and the Dreams UI will show no diary content. This should stay open until maintainers decide whether `DREAMS.md` is guaranteed after REM has usable material or whether the UI/docs should explicitly represent phase-report-only runs.

## What This Changes

The issue asks for successful memory-core REM runs to create or update `DREAMS.md` so the Control UI Dreams diary view is not empty when REM report files exist.

## Best Possible Solution

Define the Dream Diary contract explicitly. If visible diary output is guaranteed after REM produces usable material, memory-core should append a deterministic fallback entry for timeout, empty narrative, and unavailable narrative paths; otherwise docs and the Dreams UI should clearly surface phase-report-only runs instead of presenting an absent diary as empty content.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with separate report output and a diary generation path that has no snippets, times out, has no subagent surface, or yields no assistant narrative; current tests already assert `DREAMS.md` remains absent for no-snippet and timeout paths.

## Solution Assessment

Unclear. A narrow fallback diary write is maintainable if `DREAMS.md` is intended as guaranteed output, but current docs and tests encode best-effort no-write behavior, so maintainers should choose the contract before changing implementation behavior.
