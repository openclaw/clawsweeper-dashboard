## Summary

Keep open. Current main handles the request-scoped fallback class, but material REM output can still leave `DREAMS.md` absent when the narrative run times out, fails, or produces no assistant text, and the Dreams UI still has no phase-report fallback.

## What This Changes

The issue reports that REM writes `memory/dreaming/rem/YYYY-MM-DD.md` but does not create `DREAMS.md`, leaving the Control UI Dreams diary empty.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either guarantee a safe fallback diary entry for material REM output, or show phase reports when no diary exists.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: run REM with eligible entries while the narrative subagent times out, returns a non-ok status, or produces no assistant text. Current tests still assert `DREAMS.md` remains absent on the timeout path.

## Solution Assessment

Unclear. Current main fixed the direct request-scoped fallback case, but the remaining behavior needs a maintainer choice between guaranteed fallback diary writes and a phase-report fallback in the UI.
