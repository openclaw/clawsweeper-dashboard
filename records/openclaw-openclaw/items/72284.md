## Summary

Keep open. Current main still allows REM phase output and phase signals to be written while the optional Dream Diary path returns without creating `DREAMS.md`; the Gateway and Control UI then treat the missing diary as an empty Dreams tab despite docs still presenting `DREAMS.md` as the human-readable diary surface.

## What This Changes

Ensure successful REM dreaming creates or exposes Dream Diary content so the Control UI Dreams tab is not empty after a material REM run.

## Best Possible Solution

Define and enforce one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always provide a safe fallback diary artifact for material REM output, or explicitly surface phase reports when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while making the narrative subagent time out, return a non-ok status, or produce no assistant text; current tests already assert `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the visible gap, but maintainers should first decide whether `DREAMS.md` is guaranteed output for material REM runs or whether phase reports are a separate UI surface.
