## Summary

Keep open. Current main has useful Dream Diary mitigations, but the source and tests still allow a successful REM phase report and phase signals to exist while `DREAMS.md` is absent, which keeps the Control UI diary empty.

## What This Changes

The issue asks for successful REM dreaming runs to create or update `DREAMS.md` so the Control UI Dreams diary has content.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always write a safe fallback diary artifact for material REM output, or explicitly surface phase reports when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while the narrative subagent times out, returns a non-ok status, or produces no assistant text; current tests still assert `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. The current 60-second timeout, request-scoped fallback, and detached concurrency cap are useful mitigations, but they do not settle whether `DREAMS.md` is guaranteed output or whether the UI should read phase reports when the diary is missing.
