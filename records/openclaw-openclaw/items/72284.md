## Summary

Keep open. Current main still allows REM phase reports and phase signals to be written while `DREAMS.md` remains absent, and the Control UI still reads only the diary file; recent shipped mitigations reduce timeout causes but do not settle the missing-diary contract.

## What This Changes

The requested fix is to ensure a successful REM run that creates phase output also creates or surfaces Dream Diary content in the Control UI.

## Best Possible Solution

Define and implement one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always create a safe fallback `DREAMS.md` entry for material REM output, or show phase reports when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while the narrative subagent times out, returns non-ok, or produces no assistant text; current tests still assert `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. The shipped timeout and concurrency mitigations are useful but not a complete fix because they do not decide fallback diary writes versus a phase-report UI fallback.
