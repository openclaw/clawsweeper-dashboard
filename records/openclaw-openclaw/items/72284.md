## Summary

Keep open. Current main still allows the reported user-visible state: REM phase reports and signals are written first, while Dream Diary generation remains optional/best-effort and can return without creating `DREAMS.md`; the Gateway and Control UI then read only the diary file and show an empty diary when it is absent.

## What This Changes

The issue asks for successful REM dreaming runs that write `memory/dreaming/rem/YYYY-MM-DD.md` to also create or update `DREAMS.md` so the Control UI Dreams diary is populated.

## Best Possible Solution

Define the Dream Diary contract explicitly. If a usable REM run should always be visible in the Dreams UI, memory-core should write a deterministic fallback diary entry for timeout, empty-text, unavailable-narrative, and phase-report-only paths; otherwise docs and UI should make phase-report-only runs visible instead of treating an absent diary as empty content.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with separate report output and usable entries, then force the narrative subagent to time out or produce no assistant text; current tests already assert `DREAMS.md` remains absent for those no-write paths.

## Solution Assessment

Unclear. A narrow fallback diary write could solve the visible gap, but current docs and tests still encode Dream Diary generation as best-effort, so maintainers should choose the contract before automation changes behavior.
