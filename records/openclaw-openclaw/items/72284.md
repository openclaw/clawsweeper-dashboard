## Summary

Keep open because current main still has source-backed paths where a material REM phase writes reports/signals but no Dream Diary file, and the Control UI still treats the missing diary as empty; recent mitigations reduce likely causes but do not close the contract gap.

## What This Changes

Ensure material REM dreaming creates or exposes Dream Diary content so the Control UI Dreams tab is not empty after a successful REM run.

## Best Possible Solution

Define one product contract and implement it consistently: either synthesize a safe fallback diary entry whenever REM has material, or let the Dreams tab render the REM phase report when the diary is unavailable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while the narrative subagent returns `timeout`, a non-ok status, or no assistant text; current tests still assert `DREAMS.md` remains absent on the timeout path.

## Solution Assessment

Unclear. Current main’s request-scoped fallback, longer timeout, and concurrency cap are useful mitigations, but the narrow maintainable solution still depends on whether maintainers want guaranteed diary output or phase-report fallback in the UI.
