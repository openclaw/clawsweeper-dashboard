## Summary

Keep open. Current main still has a real product-contract gap: REM phase report writing succeeds independently from Dream Diary generation, while the Control UI Dreams reader only displays `DREAMS.md`/`dreams.md`; current docs and tests still encode the diary path as best-effort rather than guaranteed.

## What This Changes

The issue reports that a successful memory-core REM run writes a REM phase report but does not create or update `DREAMS.md`, leaving the Control UI Dreams tab empty.

## Best Possible Solution

Define the Dream Diary contract explicitly. If a usable REM run must always be visible in Dreams UI, memory-core should write a deterministic fallback diary entry for timeout, empty narrative, unavailable narrative, and no-material paths or the UI should read phase reports as a visible report-only state; if the diary remains best-effort, docs and UI copy should stop implying that an absent `DREAMS.md` means no dreaming output exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with usable entries and force the narrative subagent to time out, return no assistant text, or fail outside the request-scoped fallback path; current tests already assert `DREAMS.md` remains absent in those paths while the Gateway/UI treat the missing diary as empty content.

## Solution Assessment

Unclear. A narrow fallback diary write could solve the reported empty Dreams tab, but current docs/tests still preserve best-effort diary semantics, so maintainers should choose the contract before automation changes behavior.
