## Summary

Keep open. Current main still has source-visible paths where REM phase output and phase signals are written but `DREAMS.md` can remain absent, and the Control UI only reads the diary RPC rather than falling back to phase reports.

## What This Changes

The reporter asks for successful REM dreaming output to create or update `DREAMS.md` so the Control UI Dreams tab can display diary content.

## Best Possible Solution

Define one Dream Diary contract across memory-core, gateway, UI, and docs: either always append a safe fallback diary entry when REM has material, or surface phase reports in the Dreams UI when the diary is missing.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: REM writes its phase artifact before narrative generation, and current tests still assert a timeout leaves `DREAMS.md` absent. I did not run a live Windows/npm reproduction because this review is read-only and `pnpm` is unavailable in the checkout environment.

## Solution Assessment

Unclear. Current request-scoped fallback and timeout-concurrency mitigations address some causes, but the remaining fix needs a maintainer choice between guaranteed fallback diary writes and a UI/gateway fallback to phase reports.
