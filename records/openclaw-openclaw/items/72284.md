## Summary

Keep open. Current main still permits a successful REM phase to write the phase artifact and phase signals while `DREAMS.md` remains absent when the optional Dream Diary narrative path has no material, times out, returns a non-ok status, produces no assistant text, or fails outside the request-scoped fallback case; the Control UI then treats the missing diary as empty content.

## What This Changes

The issue reports that successful REM dreaming can create `memory/dreaming/rem/YYYY-MM-DD.md` without creating or updating `DREAMS.md`, leaving the Control UI Dreams diary empty.

## Best Possible Solution

Define the Dream Diary contract and enforce it consistently: either write a deterministic fallback diary entry from material REM output whenever the narrative subagent cannot produce text, or make the Dreams UI display phase-report-only REM runs and update docs so missing `DREAMS.md` is not presented as an empty diary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while making the narrative subagent time out, return a non-ok status, or produce no assistant text; current tests already assert that `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the reported visible gap, but maintainers should first decide whether `DREAMS.md` is guaranteed output for successful REM material or whether phase reports remain separate with clearer UI/docs handling.
