## Summary

Keep open. Current main still has source-visible paths where REM phase reports and phase signals are written, but `DREAMS.md` remains absent and the Control UI has no fallback beyond the diary RPC.

## What This Changes

Ensure successful REM dreaming runs create or expose human-readable Dream Diary content instead of leaving only `memory/dreaming/rem/YYYY-MM-DD.md` and an empty Dreams tab.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always append a safe fallback diary entry for material REM output, or show phase reports when no diary exists.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: run REM with eligible entries while the narrative subagent times out, returns non-ok, or produces no assistant text. Current tests still assert `DREAMS.md` remains absent on at least the timeout path.

## Solution Assessment

Unclear. The current request-scoped fallback, model retry, timeout, and concurrency mitigations help specific causes, but the remaining fix needs a maintainer decision between guaranteed fallback diary writes and a phase-report UI fallback.
