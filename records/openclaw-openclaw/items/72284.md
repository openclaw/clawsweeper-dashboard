## Summary

Keep open. Current main still allows a successful REM phase to write the REM phase block and phase signals while leaving `DREAMS.md` absent when the optional Dream Diary narrative path times out, returns no assistant text, has no usable snippets, or hits a non-request-scoped runtime failure. The Control UI reads only `DREAMS.md`/`dreams.md` for the Dreams diary view and treats a missing diary as empty content, so the reported user-visible gap remains possible despite recent timeout, fallback, and detached-concurrency hardening.

## What This Changes

The issue requests that successful REM dreaming runs create or update `DREAMS.md` so the Control UI Dreams tab can display the Dream Diary content.

## Best Possible Solution

Define the Dream Diary contract explicitly. If REM output should always be visible in the Dreams UI, memory-core should write a deterministic safe fallback diary entry for successful REM runs with material when the narrative subagent times out or yields no text; otherwise the UI and docs should show phase-report-only runs instead of presenting an empty diary as if nothing happened.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with live entries and make the narrative subagent return `timeout` or no assistant text; current tests already assert that `DREAMS.md` remains absent for those paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the visible gap, but maintainers should first choose whether `DREAMS.md` is guaranteed output for successful REM material or whether Dream Diary remains best-effort with clearer UI/docs handling.
