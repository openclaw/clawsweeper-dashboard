## Summary

Keep open. Current main still allows REM phase reports and phase signals to be written before Dream Diary generation succeeds, while existing tests preserve paths where `DREAMS.md` remains absent and the Dreams UI renders the empty diary state.

## What This Changes

The issue reports that a completed REM dreaming phase can create `memory/dreaming/rem/YYYY-MM-DD.md` without creating root `DREAMS.md`, leaving the Control UI Dream Diary empty.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always leave a safe human-readable diary artifact for material REM output, or surface phase reports in the Dreams UI when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible material while the narrative subagent times out, returns a non-ok status, or produces no assistant text; current tests assert `DREAMS.md` remains absent and the UI maps that to the empty state.

## Solution Assessment

Unclear. The shipped timeout and concurrency-cap mitigations reduce common failure causes, but they do not decide whether `DREAMS.md` is guaranteed output or whether the UI should fall back to phase reports.
