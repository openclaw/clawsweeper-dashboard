## Summary

Keep open. Current main has narrowed some likely failure modes with request-scoped fallback, a longer narrative timeout, and detached-run concurrency caps, but it still deliberately permits material REM phase output to exist without a `DREAMS.md` entry, and the Gateway/UI still show an empty diary when that file is absent.

## What This Changes

Ensure successful REM dreaming creates or exposes Dream Diary content so the Control UI Dreams tab is not empty after a material REM run.

## Best Possible Solution

Define one cross-surface Dream Diary contract: either always write a safe fallback diary artifact for material REM runs, or explicitly expose REM phase reports in the Dreams UI and docs when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while the narrative subagent times out, returns a non-ok status, or produces no assistant text; current tests assert `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. A narrow fallback diary write would fix the visible gap, but the maintainable choice depends on whether `DREAMS.md` is meant to be guaranteed output or whether separate phase reports should be the fallback UI surface.
