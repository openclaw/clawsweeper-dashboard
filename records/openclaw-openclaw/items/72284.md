## Summary

Keep open. Current main still allows a successful REM phase to write phase output and phase signals while the optional Dream Diary path returns without creating `DREAMS.md`; the Gateway and Control UI then render the missing diary as empty even though docs still describe `DREAMS.md` as the human-readable Dream Diary surface.

## What This Changes

Ensure successful REM dreaming creates or updates `DREAMS.md` so the Control UI Dreams tab can display diary content instead of an empty state.

## Best Possible Solution

Define the Dream Diary contract and enforce it consistently: either write a deterministic fallback diary entry from material REM output when narrative generation times out, returns no text, or otherwise fails safely, or change the Dreams UI and docs so phase-report-only REM runs are visible and not treated as an empty diary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while making the narrative subagent time out, return a non-ok status, or produce no assistant text; current tests already assert that `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the reported visible gap, but maintainers should first confirm whether `DREAMS.md` is guaranteed output for material REM runs or whether the UI should display phase reports when no diary exists.
