## Summary

Keep open. Current main has shipped mitigations for short Dream Diary timeouts and detached-narrative fan-out, but it still permits a REM phase report and phase signals to exist while `DREAMS.md` is absent, leaving the Control UI diary reader empty.

## What This Changes

Ensure successful REM dreaming creates or exposes Dream Diary content so the Control UI Dreams tab is not empty after a material REM run.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always write a safe fallback diary artifact for material REM output, or explicitly surface phase reports when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while the narrative subagent times out, returns a non-ok status, or produces no assistant text; current tests still assert `DREAMS.md` remains absent on those paths.

## Solution Assessment

Unclear. The shipped 60-second timeout and concurrency cap are useful mitigations, but they do not decide whether `DREAMS.md` is guaranteed output or whether the UI should read phase reports when the diary is missing.
