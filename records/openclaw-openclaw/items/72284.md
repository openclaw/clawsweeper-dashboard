## Summary

Keep open. Current main has improved the Dream Diary pipeline, including request-scoped fallback, configured-model retry, longer timeout, and detached narrative throttling, but a successful REM phase can still write its phase report/signals while leaving `DREAMS.md` absent. The remaining question is the product contract: guaranteed diary visibility in the Dreams UI versus best-effort diary generation with clearer phase-report handling.

## What This Changes

The issue requests that a successful REM dreaming run create or update `DREAMS.md` so the Control UI Dreams tab displays diary content instead of an empty state.

## Best Possible Solution

Define the Dream Diary contract explicitly. If usable REM output should always be visible in the Dreams UI, memory-core should write a deterministic fallback diary entry for successful REM runs with material when the narrative path times out, returns no text, or is unavailable; otherwise the Control UI and docs should surface phase-report-only runs instead of showing an empty diary as if nothing happened.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with live entries and force the narrative subagent to time out or return no assistant text; current tests already assert `DREAMS.md` remains absent for those no-write paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the user-visible gap, but it must not silently mask trust/allowlist/auth failures, and current docs/tests still treat the narrative subagent as best-effort.
