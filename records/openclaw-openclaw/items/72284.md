## Summary

Keep open. Current main still separates REM phase report writes from Dream Diary generation: request-scoped fallback now creates `DREAMS.md`, but timeout, non-ok run, no assistant text, missing subagent, and no-material paths can still leave the diary absent while the Control UI reads only the diary file.

## What This Changes

The issue asks memory-core to create or update `DREAMS.md` after a successful REM phase so the Control UI Dreams tab is populated.

## Best Possible Solution

Define the Dream Diary contract explicitly. If a usable REM run should always be visible in the Dreams UI, memory-core should write a deterministic fallback diary entry for timeout, empty narrative, and unavailable-narrative paths; otherwise docs and UI should surface phase-report-only runs instead of treating an absent diary as empty content.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM in separate-report mode with usable entries and force the narrative subagent to time out or return no assistant text; current tests already assert `DREAMS.md` remains absent for those no-write paths.

## Solution Assessment

Unclear. A narrow fallback diary write could solve the user-visible gap, but current docs/tests still encode Dream Diary generation as best-effort, so maintainers should choose the contract before an automated repair changes behavior.
