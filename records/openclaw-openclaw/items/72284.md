## Summary

Keep open. Current main can still write REM phase output and phase signals before optional Dream Diary generation, and the diary path can return without creating `DREAMS.md`; Gateway and Control UI then treat the missing diary as empty Dreams content while docs still present `DREAMS.md` as the human-readable diary surface.

## What This Changes

The issue asks memory-core REM dreaming to create or update `DREAMS.md` after successful REM output so the Control UI Dreams tab has diary content.

## Best Possible Solution

Define one Dream Diary contract across memory-core, Gateway, Control UI, and docs: either always write a safe fallback diary artifact for material REM output, or explicitly surface phase reports when no diary exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with eligible entries while the narrative subagent times out, returns a non-ok status, or produces no assistant text; current source writes REM artifacts first and tests assert `DREAMS.md` can remain absent on those paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the visible gap, but maintainers should first decide whether `DREAMS.md` is guaranteed output for material REM runs or whether phase reports are the intended fallback UI surface.
