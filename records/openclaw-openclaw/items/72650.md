## Summary

Keep open: current main still rejects symlinked `~/.openclaw`, while the branch still has a correctness bug and lacks contributor real-behavior proof.

## What This Changes

The PR changes exec approvals path validation and tests to allow a current-user-owned, non-group/other-writable first-level `~/.openclaw` symlink while retaining deeper symlink rejection.

## Best Possible Solution

Land one maintained implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while preserving deeper-symlink, chained-symlink, hard-link, and final-file write protections.

## Reproduction Assessment

Yes, from source inspection. `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current main rejects a symlinked `.openclaw` approvals directory under the resolved home directory.

## Solution Assessment

No, not as currently submitted. The one-hop trusted symlink direction is narrow and maintainable, but symlink resolution and validation errors must not be swallowed, and the final patch should align with the current fs-safe wrapper.
