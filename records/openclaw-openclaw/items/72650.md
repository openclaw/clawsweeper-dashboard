## Summary

Keep open: the PR targets a real current-main bug, but its head still swallows trusted-symlink resolution errors and lacks after-fix real behavior proof before merge.

## What This Changes

This PR updates exec approvals path validation and store tests to allow a current-user-owned, non-group/other-writable first-level `~/.openclaw` symlink under the trusted home directory.

## Best Possible Solution

Land one maintained, security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while preserving current fs-safe, chmod, deeper-symlink, final-file symlink, and hard-link protections.

## Reproduction Assessment

Yes, from source inspection. `applyLocalExecPolicy()` calls `saveExecApprovals()`, and current main's approvals-directory guard rejects a symlinked `.openclaw` path below the resolved home root.

## Solution Assessment

No, not as currently submitted. The one-hop trusted symlink direction is maintainable, but symlink resolution and target-validation errors must be rethrown and the final patch should align with current main's fs-safe wrapper.
