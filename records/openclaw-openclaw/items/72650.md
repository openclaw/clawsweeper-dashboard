## Summary

Keep open: the PR targets a valid current-main exec-policy regression, but the latest patch still has a dangling-symlink error-handling bug and lacks the required changelog entry; overlapping #72713 also needs maintainer scope direction.

## What This Changes

The PR changes exec approvals path validation and store tests to allow an owner/mode-checked first-level `~/.openclaw` symlink while keeping deeper symlink traversal rejected.

## Best Possible Solution

Land one maintained, security-reviewed fix that accepts only a validated first-level `.openclaw` symlink, preserves deeper traversal and destination-file guards, and documents the user-facing change.

## Reproduction Assessment

Yes, from source inspection. `exec-policy preset` calls `saveExecApprovals()`, and current main's path walk throws on a symlinked `.openclaw` component before the approvals write.

## Solution Assessment

No, not as written. The one-hop trusted-symlink direction is narrow and maintainable, but symlink resolution errors must not be swallowed and the changelog entry is missing; maintainers also need to choose this branch versus #72713.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Rethrow trusted symlink resolution errors:** `src/infra/exec-approvals.ts:282`
  - body: `fs.realpathSync(current)` now runs inside the same `try` block that intentionally swallows `ENOENT` from `lstatSync` for missing path components. If `.openclaw` is a dangling symlink, trusted-target validation is skipped and the command fails later with a lower-level mkdir/stat error; scope the `ENOENT` swallow to `lstatSync`, or rethrow once the code has identified a symlink.
