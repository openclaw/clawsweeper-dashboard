## Summary

Keep open: the PR targets a real current-main regression, but the latest branch still has a correctness issue in trusted-symlink error handling and lacks the required user-facing changelog entry. ## What This Changes

The PR changes exec approvals path validation and store tests to allow an owner/mode-checked immediate `.openclaw` symlink under the trusted home while preserving deeper symlink rejection.

## Best Possible Solution

Land one maintained fix for #72572 that accepts the trusted dotfile-managed `.openclaw` symlink shape, rejects unsafe or deeper symlink traversal, includes a changelog entry, and resolves whether #72713 supersedes this narrower path.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `~/.openclaw` as a symlink below a real home, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, whose current-main directory validation rejects the `.openclaw` component before writing.

## Solution Assessment

No, not as written. The one-hop trusted-symlink direction is plausible, but symlink resolution errors must not be swallowed, the user-facing fix needs a changelog entry, and maintainers should choose between this PR and the broader #72713 design.
