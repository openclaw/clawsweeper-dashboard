## Summary

Keep this PR open: current main still has the reported `.openclaw` symlink gap, and the PR is a useful narrow fix path, but the latest head still has a blocking catch-scope bug in the new trusted-symlink branch plus a missing required changelog entry. The broader draft #72713 overlaps the same issue, so maintainers should decide whether to patch this branch or consolidate on that replacement before merge.

## What This Changes

The PR changes exec approvals directory validation to allow one owner/mode-checked `.openclaw` symlink hop under the trusted home root and adds regression coverage for GNU Stow/chezmoi-style layouts.

## Best Possible Solution

Land one fail-closed implementation that accepts an immediate, trusted `.openclaw` symlink for dotfile-managed setups while preserving deeper symlink rejection, with the symlink-resolution catch scope fixed, regression coverage retained, and the user-facing changelog updated. If maintainers want macOS/node-host parity in the same change, consolidate this PR into #72713 instead of merging both paths separately.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, and `assertNoSymlinkPathComponents()` rejects `.openclaw` as soon as `lstatSync()` reports it is a symlink below the trusted home root. The PR finding also has a clear focused reproduction with a dangling `.openclaw` symlink causing `realpathSync()` to throw `ENOENT` inside the swallowed catch.

## Solution Assessment

No, not as written. The one-hop, owner/mode-gated allowance is the narrow maintainable direction for the reported GNU Stow/chezmoi layout, but symlink resolution errors must not share the missing-component `ENOENT` catch and the user-facing fix needs a changelog entry.

## Review Findings
