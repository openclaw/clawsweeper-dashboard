## Summary

Keep open: the PR targets a valid current-main exec-policy regression, but the latest patch still has a real dangling-symlink error-handling bug and is missing the required changelog entry; maintainers also need to choose this narrow approach or the broader draft #72713.

## What This Changes

The PR changes exec approvals path validation and store tests to allow an owner/mode-checked immediate `~/.openclaw` symlink under the trusted home directory while keeping deeper symlinks rejected.

## Best Possible Solution

Land one maintained fix for #72572 that accepts a trusted first-level `.openclaw` symlink after clear ownership/permission validation, keeps deeper traversal and symlinked destination files rejected, and documents the user-facing change.

## Reproduction Assessment

Yes. The source path is high confidence: with `~/.openclaw` as a symlink below a real home, `openclaw exec-policy preset yolo` calls `saveExecApprovals()`, and current main rejects the symlinked `.openclaw` component before writing.

## Solution Assessment

No, not as written. The one-hop trusted-symlink direction is narrow and maintainable for the reported GNU Stow/chezmoi layout, but symlink resolution errors must not be swallowed and the changelog needs the user-facing fix entry.
