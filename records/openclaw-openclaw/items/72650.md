## Summary

Keep open: the reported regression is valid and this PR is close, but the current head still swallows trusted symlink resolution errors and lacks the required changelog entry. The broader #72713 is still draft and is not a clear terminal superseder for this narrower branch.

## What This Changes

The PR updates exec approvals path validation and store tests to allow an owner/mode-checked immediate `.openclaw` symlink under the trusted home directory.

## Best Possible Solution

Land one maintained fix for #72572 that accepts a trusted first-level `.openclaw` symlink after explicit ownership and permission validation, keeps deeper traversal and symlinked destination files rejected, and documents the shipped behavior.

## Reproduction Assessment

Yes. On current main, a real home with `.openclaw` as a symlink reaches `saveExecApprovals()` through `openclaw exec-policy preset yolo`, and `assertNoSymlinkPathComponents()` rejects that symlinked component before writing.

## Solution Assessment

No, not as written. The one-hop trusted symlink direction is narrow and maintainable, but symlink resolution errors must not be swallowed and the user-facing fix needs a changelog entry.
