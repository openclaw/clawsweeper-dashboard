## Summary

Keep open: current main still rejects the reported symlinked `~/.openclaw` layout, and this PR is a plausible narrow fix, but the latest head still has one correctness issue plus the required changelog gap and overlaps the broader draft #72713.

## What This Changes

The branch changes exec approvals path validation and store tests to allow an owner/mode-checked immediate `.openclaw` symlink under the home directory while keeping deeper symlink traversal rejected.

## Best Possible Solution

Land one maintained fix for #72572 that accepts a trusted first-level `.openclaw` symlink after clear ownership/permission validation, keeps deeper traversal and symlinked destination files rejected, documents the user-facing behavior, and resolves the competing draft path.

## Reproduction Assessment

Yes. The source-level path is high confidence: with `~/.openclaw` as a symlink under a real home, `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, whose current-main directory validation rejects the `.openclaw` component before the approvals file can be written.

## Solution Assessment

No, not as written. The one-hop trusted-symlink direction is narrow and maintainable for the reported GNU Stow/chezmoi layout, but symlink resolution errors must not be swallowed and the user-facing fix needs a changelog entry; maintainers should also choose whether #72713 is the preferred broader implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Rethrow trusted symlink resolution errors:** `src/infra/exec-approvals.ts:282`

Scope the `ENOENT` swallow to `lstatSync`, or rethrow once the code has identified a symlink.
