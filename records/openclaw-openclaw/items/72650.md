## What This Changes

The PR changes exec approvals path validation and store tests to allow a current-user-owned, non-group/other-writable first-level `~/.openclaw` symlink under the trusted home directory.

## Best Possible Solution

Land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink under the trusted home boundary while preserving fs-safe, chmod, deeper-symlink, final-file symlink, and hard-link protections.

## Reproduction Assessment

Yes, from source inspection. `applyLocalExecPolicy()` calls `saveExecApprovals()`, and current main's approvals-directory guard rejects a symlinked `.openclaw` component before writing.

## Solution Assessment

No, not as currently submitted. The trusted one-hop direction is maintainable, but PR-head symlink resolution and target-validation errors must be rethrown and the final patch must be reconciled with current main's fs-safe wrapper.
