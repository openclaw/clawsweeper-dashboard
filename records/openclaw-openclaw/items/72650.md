## Summary

Keep open: the underlying bug still exists on current main and this PR is a plausible narrow fix, but the latest head still swallows trusted-symlink resolution errors and the external PR lacks real behavior proof before merge.

## What This Changes

The branch modifies `src/infra/exec-approvals.ts` and `src/infra/exec-approvals-store.test.ts` to allow an owner- and mode-checked immediate `~/.openclaw` symlink under the trusted home directory.

## Best Possible Solution

Land one maintained implementation that accepts only a validated first-level `.openclaw` symlink while preserving deeper-symlink, chained-symlink, hard-link, and final-file write protections.

## Reproduction Assessment

Yes. From source inspection, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current main still rejects `.openclaw` when it is a symlink under the resolved home directory.

## Solution Assessment

No, not as currently submitted. The one-hop trusted symlink direction is narrow and maintainable, but symlink resolution errors must not be swallowed and maintainers should reconcile this narrow branch with the broader open draft.
