## Summary

Keep open. Current main still rejects the symlinked `~/.openclaw` layout reported in #72572, and this PR is a relevant narrow fix, but the latest patch still has a concrete error-handling bug in the new trusted-symlink branch and is missing the required changelog entry. There is also an overlapping draft replacement in #72713, so maintainers should choose whether to patch this branch or consolidate on the broader approach.

## What This Changes

The branch updates exec approvals directory validation to allow an owner/mode-checked immediate `.openclaw` symlink under the home directory and adds store tests for accepted and unsafe symlink layouts.

## Best Possible Solution

Land one maintained fix for #72572 that allows a trusted first-level POSIX `.openclaw` symlink after owner/permission validation, keeps deeper traversal and symlinked destination files rejected, documents the user-facing behavior in the changelog, and either closes or explicitly supersedes the competing draft path.

## Reproduction Assessment

Yes. The source-level path is high confidence: with `~/.openclaw` as a symlink under a real home, `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, whose current-main directory validation rejects the `.openclaw` component before the approvals file can be written.

## Solution Assessment

No, not as written. The one-hop trusted-symlink direction is narrow and maintainable for the reported GNU Stow/chezmoi layout, but symlink resolution errors must not be swallowed by the missing-component catch and the user-facing fix needs a changelog entry; maintainers should also decide whether #72713 is the preferred broader implementation.
