## Summary

Keep open: current main and the latest release still run direct preflight compaction before memory flush, while the PR merge ref puts memory flush first and has focused terminal proof. I found no blocking line-level defect, but the session-state ordering and sibling-scope tradeoff still need maintainer acceptance before merge.

## What This Changes

The PR reorders `runReplyAgent` pre-run maintenance so `runMemoryFlushIfNeeded` runs before `runPreflightCompactionIfNeeded`, with regression coverage for that call order.

## Best Possible Solution

Land this direct-run ordering fix if maintainers accept flush-first timing, while keeping queued-followup expansion and in-turn catch-up in separate canonical work.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.6` shows direct `runReplyAgent` preflight compaction runs before memory flush, while the memory docs promise the silent flush before compaction; I did not run a live long-context provider turn.

## Solution Assessment

Yes for the direct pre-turn path: moving the existing memory-flush call before the existing preflight-compaction call is the narrow owner-boundary fix. It is not the full answer for queued followups or in-turn auto-compaction, which remain separate maintainer decisions.
