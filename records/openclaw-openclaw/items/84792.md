## Summary

Keep open for maintainer review: current main still runs preflight compaction before memory flush, while this PR moves the direct reply path to the documented flush-before-compaction order with focused coverage and terminal proof. The remaining concerns are deliberate session-state timing and sibling-surface scope, not a close reason.

## What This Changes

The PR reorders `runReplyAgent` so `runMemoryFlushIfNeeded` runs before `runPreflightCompactionIfNeeded` and adds a direct-runtime regression assertion.

## Best Possible Solution

Land the direct-run flush-before-compaction order after maintainer acceptance of the session-state timing, while keeping in-turn catch-up and any queued-followup expansion in their canonical follow-up work.

## Solution Assessment

Yes for the direct pre-turn `runReplyAgent` path: moving the existing memory-flush call before the existing preflight-compaction call is narrow and matches the documented before-compaction memory behavior. It is not the complete answer for in-turn auto-compaction or queued followup compaction, which remain separate maintainer decisions.
