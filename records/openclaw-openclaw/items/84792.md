## Summary

This PR is still useful and should remain open: current `main` and the latest release still run preflight compaction before memory flush, while the PR changes the direct `runReplyAgent` path to the documented flush-before-compaction order and adds a focused ordering regression test. The remaining question is maintainer acceptance of the session-state timing and sibling-scope tradeoff, not a reason to close the PR.

## What This Changes

The PR reorders `runReplyAgent` pre-run maintenance so automatic memory flush runs before preflight compaction and updates direct-runtime tests to assert that order.

## Best Possible Solution

Land the direct-run order fix after maintainer acceptance of the session-state timing, while keeping in-turn catch-up and any queued-followup expansion in their canonical follow-up work.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.6` shows `reply.preflight_compaction` runs before `reply.memory_flush`, while the docs promise automatic memory flush before compaction; I did not run a live long-context Telegram/provider turn.

## Solution Assessment

Yes for the direct pre-turn path: moving the existing memory-flush call before the existing preflight-compaction call is narrow and matches the documented before-compaction memory behavior. It is not the complete answer for queued followups or in-turn auto-compaction, which remain separate maintainer decisions.
