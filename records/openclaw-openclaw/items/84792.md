## Summary

Keep open: this PR is still useful because current main and the latest release still run direct preflight compaction before the automatic memory flush, while the patch narrowly changes that direct-run ordering. The remaining blocker is maintainer acceptance of the session-state timing and sibling-scope tradeoff before merge.

## What This Changes

The PR reorders `runReplyAgent` so `runMemoryFlushIfNeeded` runs before `runPreflightCompactionIfNeeded`, and adds regression coverage for that call order.

## Best Possible Solution

Land this direct-run ordering fix if maintainers accept flush-first timing, while keeping queued-followup expansion and in-turn catch-up in separate canonical work.

## Reproduction Assessment

Yes. Source inspection of current main and `v2026.6.8` shows direct `runReplyAgent` preflight compaction runs before memory flush, while the memory docs promise the silent flush before compaction; I did not run a live long-context provider turn.

## Solution Assessment

Yes for the direct pre-turn path: moving the existing memory-flush call before the existing preflight-compaction call is the narrow owner-boundary fix. It is not the full answer for queued followups or in-turn auto-compaction, which remain separate maintainer decisions.
