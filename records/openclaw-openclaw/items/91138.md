## Summary

Keep open for maintainer security/product review: current main still intentionally keeps safety-boundary tool and exec display redaction active, while the requested opt-out would add new credential-exposure behavior and the report still does not include an exact command/output trace proving shell-input mutation before execution.

## What This Changes

The issue requests a global or per-agent way to disable or narrow exec command credential masking so inline secret-looking strings can reach shell commands unchanged.

## Best Possible Solution

Keep secure default redaction, collect an exact shell-input mutation reproduction, and only add a narrow maintainer-approved trusted credential path or parser/display exemption if the security model supports it.

## Reproduction Assessment

No high-confidence end-to-end reproduction proves that current main mutates the shell command before execution. Source inspection does reproduce the active event/display/approval redaction path and also shows the raw exec execution path is separate.

## Solution Assessment

No: a global `credentialGuard: false` switch is not the best fix without maintainer security approval. A narrower trusted credential-passing path or heredoc/display exemption would better preserve the existing safety boundary if the reported shell mutation is confirmed.
