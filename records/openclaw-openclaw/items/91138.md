## Summary

Keep open for maintainer security/product review: current main and v2026.6.9 intentionally keep tool/UI/approval redaction active, while normal exec execution is a separate raw-command path and the report still lacks an exact command/output pair proving shell-input mutation.

## What This Changes

The issue requests a global or per-agent way to disable exec credential masking, or to skip masking inside heredocs/quoted strings that are not shell credential assignments.

## Best Possible Solution

Keep secure default redaction, collect an exact command/output reproduction for any real shell-input mutation, then design a narrow trusted credential-passing or display exemption only if maintainers approve the security boundary.

## Reproduction Assessment

No. Source inspection clearly reproduces safety-boundary redaction on tool/UI/approval surfaces, but the issue does not include an exact command/output pair proving that current main mutates the command before shell execution.

## Solution Assessment

No. A global `tools.exec.credentialGuard: false` switch is too broad for the current security boundary; if maintainers approve the direction, a targeted trusted credential-passing path or narrowly scoped display exemption would be safer.
