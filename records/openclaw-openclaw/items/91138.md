## Summary

Keep open for maintainer security/product review: current main and v2026.6.8 still intentionally redact safety-boundary tool/UI payloads, while the source path does not prove the stronger report that redacted command text reaches the shell.

## What This Changes

The issue requests a global or per-agent option to disable exec credential masking, or a narrower exemption for credential-like text inside quoted or heredoc strings.

## Best Possible Solution

Keep safety-boundary redaction enabled by default, request an exact repro for any alleged shell mutation, and only add a narrow trusted debugging or credential-passing path if maintainers approve that security tradeoff.

## Reproduction Assessment

No. Source inspection reproduces the display/event redaction behavior, but this issue still lacks an exact command and observed output proving that redacted text reaches the shell on current main.

## Solution Assessment

No. A global `credentialGuard=false` switch is too broad for the current security boundary; the safer route is either a targeted execution-path bugfix if pre-shell mutation is proven, or a maintainer-approved trusted credential/debugging design.
