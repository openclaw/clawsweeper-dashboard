## Summary

Keep open for maintainer security/product review: current main still treats exec redaction as a safety-boundary display/logging concern, while the requested opt-out would add a new credential-exposure path and the report still lacks an exact command/output repro proving pre-shell command mutation.

## What This Changes

The issue requests a global or per-agent way to disable exec credential masking, or quote/heredoc-aware behavior that lets inline API keys reach shell commands unchanged.

## Best Possible Solution

Keep secure redaction on by default and, if maintainers accept the tradeoff, design a narrow operator-controlled credential-passing path or display-only parsing fix with focused security tests.

## Reproduction Assessment

No. The report says the behavior is reproducible on v2026.6.1, but it lacks an exact command/output pair, and current-main source inspection shows display redaction separately from normal shell launch text.

## Solution Assessment

No. A global `credentialGuard=false` option is too broad without maintainer approval; the safer direction is a narrow trusted credential path or a targeted fix if a repro proves display redaction is crossing into execution.
