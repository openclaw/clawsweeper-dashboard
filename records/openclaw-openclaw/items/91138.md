## Summary

Keep open for maintainer product/security review: current main and the latest release intentionally keep exec/tool UI redaction active, while the requested opt-out would change a credential-exposure boundary and the report still needs a concrete current-main trace proving shell input is mutated before execution.

## What This Changes

The issue requests a global or per-agent way to disable exec credential masking, or a parser refinement that avoids replacing secret-like text inside heredocs or quoted strings.

## Best Possible Solution

Preserve redaction on shared UI/diagnostic safety surfaces by default, verify whether any current path actually mutates shell input, then consider only a narrow trusted diagnostic mode or parser refinement if maintainers approve the security tradeoff.

## Reproduction Assessment

No for the claimed shell-input mutation: this review found source evidence that raw `params.command` reaches process launch, but did not establish a failing current-main command trace. Yes from source for the documented display/event behavior: tool args and exec approval command text are redacted before UI/event emission.

## Solution Assessment

Unclear: a global credential-guard disable switch is not clearly the best solution because it weakens a documented safety boundary. A narrower trusted diagnostic option or parser fix would be safer if maintainers decide the workflow needs an escape hatch.
