## Summary

Keep open. Current main still applies the same weighted vector/BM25 merge to every builtin memory result, and multimodal chunks still reach search as short label text without persisted image/audio modality metadata that ranking can use. The prior ClawSweeper comment remains accurate against the newer main SHA, and the related open PR covers a different keyword-fallback threshold bug rather than this non-text vector scoring problem.

## What This Changes

The issue asks for builtin hybrid memory search to avoid BM25 dampening for image/audio chunks indexed through multimodal memory search.

## Best Possible Solution

Carry reliable image/audio modality metadata from multimodal indexing into persisted chunks and search rows, then have the builtin hybrid merge score non-text vector candidates without BM25 dampening while preserving existing weighted BM25 + vector behavior for text chunks.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with multimodal chunks stored as label text and vector-only candidates receiving textScore 0, default 0.7/0.3 hybrid scoring lowers image/audio vector hits below comparable text chunks; the reported Gemini/extraPaths setup matches the current code path.

## Solution Assessment

Yes, the requested direction is the right narrow fix, but it should first add a reliable modality signal through the memory schema/result path rather than inferring from filenames at the final ranking step only.

Fix issue #44540 in the builtin memory engine. Current behavior: multimodal image/audio files are indexed with structured embeddings, but stored chunks/search results do not carry reliable modality metadata; mergeHybridResults applies vectorWeight/textWeight to all candidates, so image/audio vector-only hits get dampened by BM25 textScore 0 under default 0.7/0.3 hybrid weights.
