## Summary

Keep open: the reported duplicate-send path is plausible on current main, but this PR is not merge-ready because it leaks raw idempotency keys and can drop partially failed best-effort deliveries during recovery.

## What This Changes

Adds a persistent outbound idempotency ledger and deterministic keys for Discord-originated/tool-driven WhatsApp sends, threading keys through reply routing, sendMessage, and delivery-queue recovery with new tests.

## Best Possible Solution

Implement a generic outbound idempotency path that stores redacted durable keys with queue entries, commits only after complete delivery success, and covers Discord/tool-originated WhatsApp sends with regression tests and a changelog entry.

## Reproduction Assessment

Source-reproducible: current main enqueues before sending and recovery replays pending entries, while direct sends do not pass the existing idempotency key into outbound delivery. I did not live-crash a WhatsApp gateway in this read-only review.

## Solution Assessment

No. Persistent outbound idempotency is the right direction, but this patch should hash/redact keys, avoid committing a batch after partial failure, and be ported onto the current split queue files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Hash idempotency keys before persisting or logging them:** `src/infra/outbound/outbound-dedupe.ts:97`
  - body: The key builders include recipient/account data, and `buildToolOutboundIdempotencyKey` includes outbound message/media JSON. Storing the raw key as the JSON property and logging it on suppression can leave message text, media URLs, and phone/JID targets in `~/.openclaw/outbound-dedupe/global.json` and logs; store/log a stable hash or redacted token instead.
  - confidence: 0.88
- **[P2] Don't suppress recovery after partial delivery failures:** `src/infra/outbound/deliver.ts:815-816`
  - body: This records the whole batch as completed whenever any result exists, but the wrapper still calls `failDelivery` when `bestEffort` hit `onError`. On recovery the same key returns an empty result and the queue entry is acked, dropping unsent payload/media units; commit only after the whole queued delivery succeeds or record per-unit keys.
  - confidence: 0.83
