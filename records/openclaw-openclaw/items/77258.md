## Summary

Keep open: the duplicate-send problem looks valid, but this patch should not merge yet because it persists/logs raw recipient and message data, records the completion marker too late to close the crash window, and is stale against current main's split delivery-queue files.

## What This Changes

The PR adds a persistent outbound dedupe ledger and deterministic idempotency keys for Discord-originated and tool-driven WhatsApp sends, threading the key through follow-up routing, message sends, delivery queue recovery, and tests.

## Best Possible Solution

Implement the fix against current main's split queue files with opaque hashed keys, commit dedupe at the smallest safe post-provider-send boundary or via provider-level idempotency, and add regression coverage for crash recovery plus Discord replay paths.

## Reproduction Assessment

Yes, source-level: current main writes a delivery queue entry before sending, acks it only after delivery returns, and recovery replays pending entries with skipQueue. I did not run a live WhatsApp crash/restart test in this read-only review.

## Solution Assessment

No. The PR moves in the right direction, but the durable key must not contain raw content/recipients, the completion marker must close the provider-success crash window, and the queue changes must be ported to the current split storage/recovery files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Hash idempotency material before storing or logging it:** `src/infra/outbound/outbound-dedupe.ts:168-180`
  - body: The key persisted to `~/.openclaw/outbound-dedupe` includes the target recipient plus a JSON fingerprint containing the outbound message text and media URLs, and duplicate suppression logs the full key. This leaks message content, phone numbers, and platform IDs into state and logs; build an opaque hash from canonical material and log only a short non-sensitive identifier.
  - confidence: 0.93
- **[P2] Commit dedupe before the crash-replay window reopens:** `src/infra/outbound/deliver.ts:815-816`
  - body: Recording the key after the whole delivery loop finishes still allows the reported duplicate: if the process dies after `handler.sendText`/`sendMedia` succeeds but before this record call, recovery sees the queued entry with no completed ledger record and sends it again. The completion should be committed at the smallest safe post-provider-send boundary, with tests that simulate a crash between provider success and queue ack.
  - confidence: 0.86
- **[P2] Port queue persistence through the split queue files:** `src/infra/outbound/delivery-queue.ts:46`
  - body: Current main no longer keeps the queue payload and recovery implementation in this monolithic file; `delivery-queue.ts` is a barrel over `delivery-queue-storage.ts` and `delivery-queue-recovery.ts`. As written this branch is stale/unmergeable, and leaving the split files unchanged would drop the idempotency key from persisted entries and recovery replay.
  - confidence: 0.82
- **[P2] Do not collapse separate same-content tool sends:** `src/infra/outbound/outbound-send-service.ts:125-132`
  - body: The generated tool-send key is based on the source message and payload, but not the actual tool call id or per-call sequence. Two intentional `message` tool sends with the same text/media to the same recipient in one agent turn would share a key and the second would be suppressed for the TTL.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete privacy regression by persisting and logging raw outbound content, recipient identifiers, and platform message IDs as idempotency keys.

Concerns:

- **[medium] Raw message data in dedupe keys:** `src/infra/outbound/outbound-dedupe.ts:168`
  - body: `buildToolOutboundIdempotencyKey` embeds message text and media URLs, while the Discord key embeds recipients and message IDs; those keys are written to disk and logged on duplicate suppression. Use a keyed or stable hash over canonical fields and keep logs redacted.
  - confidence: 0.94
