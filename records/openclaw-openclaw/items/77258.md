## Summary

Keep open. The PR targets a valid duplicate outbound-delivery bug, but the current diff is stale against main and still leaves correctness and privacy blockers before merge.

## What This Changes

The PR adds a persistent outbound dedupe ledger and deterministic idempotency keys for Discord-originated and tool-driven WhatsApp sends, threading keys through follow-up routing, direct sends, delivery queue recovery, and tests.

## Best Possible Solution

Port a smaller fix to current main's split queue/recovery files using opaque hashed keys, record completion at the smallest safe post-provider-send boundary, include per-tool-call or per-payload discriminators, and add crash-recovery plus replay regression coverage.

## Reproduction Assessment

Yes, source-level. Current main persists an outbound queue entry before sending, acks only after delivery returns, and recovery replays pending entries with skipQueue, which recreates the duplicate window if the process dies after provider success but before ack.

## Solution Assessment

No. The PR moves in the right direction, but it is not the best mergeable fix because it persists/logs sensitive raw keys, records completion too late for the reported crash window, and is stale against the split queue implementation on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Hash idempotency material before storing or logging it:** `src/infra/outbound/outbound-dedupe.ts:137-180`
  - body: The key persisted to the outbound dedupe file includes recipient identifiers, Discord message IDs, message text, and media URLs, and duplicate suppression logs the full key. Build an opaque stable hash from canonical fields and log only a short non-sensitive identifier.
  - confidence: 0.93
- **[P2] Commit dedupe before the crash-replay window reopens:** `src/infra/outbound/deliver.ts:815-816`
  - body: Recording the key after the whole delivery loop finishes still allows the reported duplicate: a crash after provider send success but before this record call leaves recovery with no completed ledger entry, so it sends again.
  - confidence: 0.86
- **[P2] Port queue persistence through the split queue files:** `src/infra/outbound/delivery-queue.ts:46`
  - body: Current main no longer keeps queue payload and recovery implementation in this monolithic file; delivery-queue.ts is a barrel over storage and recovery modules. As written, the branch is stale and would not preserve the idempotency key through current persisted entries and recovery replay.
  - confidence: 0.82
- **[P2] Include a per-tool-call discriminator in tool-send keys:** `src/infra/outbound/outbound-send-service.ts:125-132`
  - body: The tool-send key is based on source message and payload, but not the actual tool call id or per-call sequence. Two intentional message tool sends with the same text/media to the same recipient in one turn would share a key and suppress the second send for the TTL.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete privacy regression by persisting and logging raw outbound content, recipient identifiers, and platform message IDs as idempotency keys.

Concerns:

- **[medium] Raw message data in dedupe keys:** `src/infra/outbound/outbound-dedupe.ts:137`
  - body: The Discord and tool key builders embed recipients, platform message IDs, message text, and media URLs; those keys are written to disk and logged on duplicate suppression. Use a stable opaque hash and redact logs.
  - confidence: 0.94
