## Summary

Keep open: current main and v2026.6.6 still define `maxActiveTranscriptBytes` as an active-JSONL byte trigger, while the reporter’s follow-up shows the affected transcript was below that threshold; the remaining request is a token/context-budget and status-accounting product decision, not an already-implemented fix.

## What This Changes

The issue asks for automatic compaction to trigger more usefully in long sessions configured with `midTurnPrecheck` and `maxActiveTranscriptBytes`, and for status to distinguish manual from automatic compactions.

## Best Possible Solution

Define the durable compaction-status contract and fold token-trigger policy into the model-aware context-budget/defaults work, while preserving an explicit follow-up for manual-versus-automatic status clarity if it is not handled there.

## Reproduction Assessment

Partly yes: the reporter’s byte-size follow-up and current docs/source reproduce why a 15 MB byte guard would not fire for a 1.1 MB transcript. The token-based trigger expectation is not a broken existing contract; it is a requested behavior change tied to context-budget policy.

## Solution Assessment

Unclear: a narrow status-accounting split is plausible, but changing trigger thresholds or adding token-aware defaults should be decided with the open model-aware context-budget and long-running-agent defaults work instead of patched as a one-off lower byte threshold.
