## Summary

Keep open: current main and v2026.6.8 still treat `maxActiveTranscriptBytes` as an active-JSONL byte guard, while the reporter’s follow-up shows the affected transcript never reached that byte threshold; the remaining token/context-budget trigger and manual-versus-auto status split need maintainer product direction.

## What This Changes

The issue asks for automatic compaction and user notification to trigger during long sessions configured with `maxActiveTranscriptBytes` and `midTurnPrecheck`, plus clearer status accounting for manual versus automatic compactions.

## Best Possible Solution

Define a model-aware compaction trigger and status-accounting contract that preserves the byte guard’s local-file purpose while adding explicit token/context-budget behavior and clear manual-versus-auto reporting.

## Reproduction Assessment

Yes, source-level: the reporter’s byte-size data plus current source show why a 15 MB byte guard would not fire for a 1.1 MB active transcript, and current status code shows one combined compaction counter. The exact long-running Webchat production path was not live-reproduced in this read-only review.

## Solution Assessment

Unclear as a one-off fix: lowering the byte threshold or silently redefining `maxActiveTranscriptBytes` would be compatibility-sensitive. The safer path is a maintainer-approved token/context-budget and status contract with schema, docs, runtime, and regression coverage.
