## Summary

Keep open: current main and v2026.6.9 still implement the active-transcript trigger as a raw JSONL byte threshold and the mid-turn precheck as a context-fit recovery check, while status still exposes one compaction counter that can include manual compactions.

## What This Changes

The issue asks for automatic compaction to fire in long sessions with `maxActiveTranscriptBytes` and `midTurnPrecheck` configured, plus clearer status accounting for manual versus automatic compactions.

## Best Possible Solution

Maintainers should define the supported auto-compaction contract first, then align trigger semantics, status accounting, docs, and focused regression tests around that decision.

## Reproduction Assessment

Yes, at source level: with an active transcript below the configured byte threshold and estimated prompt tokens below the context-fit threshold, current main will not auto-compact, and `/status` reads one shared compaction counter. I did not run a live overnight WebChat session.

## Solution Assessment

No, not yet: a direct code patch would have to choose new byte-vs-token trigger semantics or new status accounting, while current code matches the documented byte/context-fit contract. The best path is a maintainer product decision followed by a focused implementation.
