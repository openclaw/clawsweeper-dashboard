## Summary

Keep open: current main and v2026.6.8 still implement `maxActiveTranscriptBytes` as an active-JSONL byte guard and `midTurnPrecheck` as a context-fit recovery precheck, while `/status` still renders one persisted `compactionCount` that can include manual compactions.

## What This Changes

The issue asks for automatic compaction to trigger during long sessions despite `midTurnPrecheck` and `maxActiveTranscriptBytes`, and for status output to stop conflating manual and automatic compactions.

## Best Possible Solution

Define the intended compaction contract first, then land a focused change that either documents the current byte/context-fit behavior more clearly or adds a maintainer-approved token/context trigger and separate status accounting.

## Reproduction Assessment

Yes for the clarified source behavior: current main uses a byte-size transcript guard and context-fit prechecks, so the reported 1.1 MB transcript below a 15 MB byte threshold and 118k tokens below a 1M window would not auto-compact. No live long-session reproduction was run.

## Solution Assessment

No. The issue points at a real gap, but changing byte-vs-token trigger semantics or splitting the status counter needs a maintainer product/config decision before implementation.
