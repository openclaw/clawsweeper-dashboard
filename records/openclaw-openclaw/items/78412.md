## Summary

Keep open: the PR addresses valid source-level macOS launchd lifecycle bugs, but latest head still has a correctness blocker in the new `--disable` path and the supplied macOS proof is prose-only rather than inspectable runtime output.

## What This Changes

The branch changes macOS gateway stop to use `launchctl bootout` by default with a new `--disable` flag, skips repair kickstart for already-running LaunchAgents, and updates launchd tests, docs, and changelog.

## Best Possible Solution

Keep the bootout-by-default direction only after making `gateway stop --disable` persistently disable even when the LaunchAgent is currently not loaded, then require redacted latest-head launchd proof and maintainer acceptance of the macOS stop semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main unconditionally calls `launchctl disable` in `stopLaunchAgent` and unconditionally kickstarts the already-loaded repair branch. I did not live-run a macOS launchd scenario in this read-only review.

## Solution Assessment

No, not as latest head stands. The launchd direction is plausible, but `--disable` is bypassed after the not-loaded branch and the changed lifecycle contract still needs inspectable live macOS proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Honor --disable before the not-loaded return:** `src/cli/daemon-cli/lifecycle-core.ts:417`
  - body: After a default bootout, launchd no longer lists the LaunchAgent, so a later `openclaw gateway stop --disable` takes the not-loaded branch and returns after `onNotLoaded`; the new `params.service.stop(...disable...)` call is never reached. The documented persistent-disable command can therefore report success without running `launchctl disable`, leaving KeepAlive/RunAtLoad enabled.
