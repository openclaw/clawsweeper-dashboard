## Summary

Keep open. The PR fixes the active runtime provider path, but it does not expose the same thinking profile through the cold bundled provider-policy artifact, so opencode-go cold validation paths can still reject or downgrade `xhigh`/`max`.

## What This Changes

The PR adds an opencode-go DeepSeek V4 thinking-policy module and runtime provider hooks for `xhigh` and `max` thinking levels.

## Best Possible Solution

Keep the runtime hook, add `extensions/opencode-go/provider-policy-api.ts` that exports the same DeepSeek V4 thinking profile, and cover both active runtime and cold policy resolution with focused regression tests.

## Reproduction Assessment

Yes, source-level. Current main lacks an opencode-go thinking profile, and core fallback does not add `max`, so `opencode-go/deepseek-v4-*` `max` support is rejected unless a provider profile is visible.

## Solution Assessment

No, not yet. The runtime hook is the right active-session fix, but the maintainable fix also needs the bundled cold provider-policy artifact used by cron and other lightweight policy paths.
