## What This Changes

This PR makes `nodes` `describe` accept an omitted `node` parameter and resolve it through the existing default-node path before calling `node.describe`.

## Best Possible Solution

Merge or replace the narrow resolver change after adding focused omitted-node `describe` coverage and contributor-supplied live proof, while preserving explicit-node requirements for targeted node actions.

## Reproduction Assessment

Yes. Source inspection on current main shows the schema permits omitted `node`, while the `describe` branch hard-requires it and throws `node required` before default-node resolution can run.

## Solution Assessment

Yes if omitted-node `describe` should mean the current/default node: reusing `resolveNodeId(..., true)` is the narrowest maintainable code change. The remaining question is whether maintainers instead want list/discovery or hybrid semantics.
