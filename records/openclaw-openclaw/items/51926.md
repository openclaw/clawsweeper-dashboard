## Summary

Current main still has the schema/implementation mismatch, and this PR is a narrow active fix candidate, but it still needs contributor real-behavior proof and maintainer confirmation that omitted-node `describe` should default to a selected/current node rather than return discovery output.

## What This Changes

The PR changes `src/agents/tools/nodes-tool.ts` so `describe` reads `node` as optional and calls `resolveNodeId` with default-node fallback enabled.

## Best Possible Solution

Land or replace the narrow resolver change after maintainers confirm omitted-node `describe` semantics, with focused regression coverage and redacted real-behavior proof.

## Reproduction Assessment

Yes, at source level. Current main accepts omitted `node` in the schema but the `describe` branch calls `readStringParam` with `{ required: true }`, which throws `node required` before the default resolver can run.

## Solution Assessment

Mostly yes if maintainers accept default/current-node semantics. Reusing `resolveNodeId(..., true)` is the narrowest code change, but a list/discovery response would be a different product decision.
