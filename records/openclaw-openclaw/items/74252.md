## Summary

Keep open. Current main still reports `reset` during fast log growth, and the PR's semantic split plus regression tests are the right narrow fix, but this user-visible `openclaw logs --follow` behavior change still needs the required changelog entry before merge.

## What This Changes

The PR changes log tailing so fast growth beyond `maxBytes` reports `truncated` without `reset`, preserves reset behavior for true file shrink/rotation, and adds two `readConfiguredLogTail` regression tests.

## Best Possible Solution

Land the semantic split and regression tests after adding one Unreleased/Fixes changelog entry, keeping the broader `logs --follow` disconnect/retry topic tracked separately from this focused false-rotation notice fix.

## Reproduction Assessment

Yes. Against current main, read a log once, append more than `maxBytes` while the previous cursor is still within the file, then read with that cursor; `src/logging/log-tail.ts:91-94` sets both `truncated` and `reset`, and the CLI turns `reset` into the rotation notice.

## Solution Assessment

Yes for the code path. Removing `reset` only from the fast-growth branch is the narrowest maintainable fix because true file shrink still resets the cursor, skipped bytes still report `truncated`, and the public payload shape stays unchanged; the remaining issue is release-note hygiene.
