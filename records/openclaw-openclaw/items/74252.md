## Summary

Keep open for normal maintainer review: current main still reports a valid-cursor fast-growth tail as `reset`, while the PR's latest described patch is the narrow semantic fix with regression coverage and the changelog blocker already addressed.

## What This Changes

The PR narrows log-tail `reset` semantics so fast-growth truncation no longer reports rotation, adds unit tests for fast-growth and true-shrink cases, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the reset/truncated semantic split with the new tests and changelog once the branch is mergeable and CI is green.

## Reproduction Assessment

Yes. On current main, read a log once, append more than `maxBytes` while the prior cursor remains within the file, then read again with that cursor; `readLogSlice` enters the fast-growth branch and sets `reset`, which the CLI reports as rotation.

## Solution Assessment

Yes. Removing `reset` only from the fast-growth branch is the narrowest maintainable fix because true file shrink still reports reset, skipped bytes still report truncation, and the payload shape stays unchanged.
