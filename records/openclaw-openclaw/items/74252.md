## Summary

Keep this PR open: it fixes a real false CLI rotation notice, but the current patch changes the shared `logs.tail.reset` semantics without preserving the Control UI's existing re-anchor behavior for skipped fast-growth windows.

## What This Changes

The PR narrows `readLogSlice` so fast-growth truncation returns `truncated: true` without `reset: true`, and adds log-tail plus CLI tests for truncated-only and rotation cases.

## Best Possible Solution

Preserve a clear non-contiguous-window signal for consumers while removing the misleading CLI rotation wording, either by keeping `reset` as the re-anchor signal plus adding/using a rotation-specific field, or by updating the Control UI contract and tests before merge.

## Reproduction Assessment

Yes from source inspection and supplied terminal proof. Current main sets `reset: true` during fast-growth truncation, and the CLI renders every reset payload as a file-rotation notice.

## Solution Assessment

No, not as submitted. Narrowing `reset` fixes the CLI symptom at the source, but it also changes the shared `logs.tail` contract and Control UI re-anchor behavior without an additive signal or UI coverage.
