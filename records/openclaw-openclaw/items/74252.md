## Summary

Keep open. Current main still conflates fast log growth with rotation, and this PR is a narrow, well-covered fix; it should proceed through rebase/mergeability and CI rather than be closed as implemented.

## What This Changes

The PR narrows log-tail reset semantics, adds fast-growth and real-rotation unit tests, and adds a changelog entry for the user-facing logs fix.

## Best Possible Solution

Rebase the branch onto current main, keep the reset/truncated semantic split and regression tests, then land after targeted log-tail tests and the changed gate pass.

## Reproduction Assessment

Yes. On current main, read a log once, append more than maxBytes while the prior cursor is still valid, then read again with that cursor; readLogSlice enters the fast-growth branch and sets reset, which the CLI reports as rotation.

## Solution Assessment

Yes. Removing reset only from the fast-growth branch is the narrowest maintainable fix because true file shrink still reports reset, skipped bytes still report truncation, and the payload shape stays unchanged.
