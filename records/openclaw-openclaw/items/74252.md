## What This Changes

This PR narrows log-tail reset semantics so fast log growth reports truncation without a false rotation reset, and adds unit coverage for fast growth versus real file shrink.

## Best Possible Solution

Add one active `CHANGELOG.md` Fixes entry credited to @BSG2000, then land this semantic split and regression coverage while keeping broader `logs --follow` disconnect work separate.

## Reproduction Assessment

Yes. On current main, read a log once, append more than `maxBytes` while the previous cursor is still within the file, then read again with that cursor; `readLogSlice` sets both flags and the CLI renders the false rotation notice.

## Solution Assessment

Yes for the code path. Removing `reset` only from the fast-growth branch is the narrow maintainable fix because true file shrink still reports reset, skipped bytes still report truncation, and the payload shape stays unchanged.
