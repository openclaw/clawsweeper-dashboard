## Summary

## What This Changes

The PR changes log tailing so fast-growth truncation no longer sets the reset flag, adds regression tests for fast growth and true rotation, and adds a changelog entry.

## Best Possible Solution

Land the narrow semantic correction once maintainers accept that `reset` means cursor invalidation by shrink/rotation and `truncated` means skipped bytes; richer skipped-byte UI or protocol documentation can be additive follow-up work.

## Solution Assessment

Yes. The patch removes only the incorrect `reset` assignment from the fast-growth branch, preserves the true rotation path, and adds focused coverage for both cases; any richer UI gap marker should be additive follow-up rather than keeping a false rotation signal.
