## Summary

This PR remains useful and should not be closed: current main still conflates fast-growth truncation with file rotation in `readLogSlice`, and the PR directly fixes that false CLI notice. The remaining blocker is maintainer judgment on the existing `logs.tail.reset` compatibility/UI contract, because the Control UI currently uses `reset` to replace rather than append log rows.

## What This Changes

The PR removes `reset = true` from the fast-growth branch of `readLogSlice` and adds log-tail plus CLI tests for truncated-only tails and real file shrink/rotation.

## Best Possible Solution

Land this shape only if maintainers accept `reset` as a true file-shrink/rotation signal; otherwise preserve gap-aware UI behavior with an additive skipped-window signal or a Control UI controller update before merge.

## Reproduction Assessment

Yes from source inspection and supplied terminal proof. Current main sets `reset: true` in the fast-growth branch and the CLI prints the rotation notice for any reset payload; the PR proof exercises the same real-file sequence before and after the change.

## Solution Assessment

Unclear as a final merge path. Narrowing `reset` fixes the false CLI rotation notice at the source, but it also changes the public `logs.tail` contract and Control UI append behavior, so maintainer acceptance or a distinct skipped-window signal is the safer decision point.
