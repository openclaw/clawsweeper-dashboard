## Summary

Keep this PR open. Current main still conflates fast log growth with real rotation, while the PR narrows `reset` to the cursor-past-EOF case and adds focused tests; the only remaining maintainer-policy blocker I found is the missing changelog entry for a user-visible logs CLI behavior fix.

## What This Changes

The PR removes the fast-growth `reset = true` assignment from `readLogSlice` and adds log-tail tests for fast growth versus genuine file shrink rotation.

## Best Possible Solution

Land the semantic split and regression tests after adding a single `CHANGELOG.md` Fixes entry credited to the contributor; keep the broader `logs --follow` disconnect reports separate from this focused false-rotation notice fix.

## Reproduction Assessment

Yes. Against current main, read the log once, append more than `maxBytes` while the previous cursor is still inside the file, then read with that cursor; `src/logging/log-tail.ts:91-94` sets both `reset` and `truncated`, and the CLI renders `reset` as a rotation notice.

## Solution Assessment

Yes for the code path. Removing `reset` only from the fast-growth branch is the narrowest maintainable fix because real file shrink still reports reset, skipped bytes still report truncation, and the payload shape stays unchanged; the PR still needs the required changelog entry before merge.
