## Summary

Keep open: this is a focused, well-proven CLI/logging bug fix, but it is not merge-ready as submitted because it narrows the shared `logs.tail.reset` result semantics. Current Control UI log rendering uses `reset` to replace entries instead of appending, so fast-growth skipped windows would change visible behavior unless maintainers accept that contract change or the PR adds a distinct signal.

## What This Changes

The PR changes log tail slicing so fast-growth windows report `truncated` without `reset`, and adds log-tail plus CLI tests for truncated-only and real-rotation cases.

## Best Possible Solution

Preserve a distinct consumer-visible signal for non-contiguous/skipped log windows while removing the misleading CLI rotation wording, preferably with additive rotation or skipped-window metadata and focused CLI plus Control UI coverage.

## Reproduction Assessment

Yes. Source inspection and the supplied terminal proof show current main reports `reset: true` for fast-growth truncation, and the CLI maps that flag to the misleading file-rotation notice.

## Solution Assessment

No, not as submitted. Narrowing `reset` fixes the CLI symptom, but it also changes a shared `logs.tail` contract that the Control UI already uses for replace-vs-append rendering.
