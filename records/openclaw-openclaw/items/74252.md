## Summary

Keep open. Current main still reports fast log growth as both truncation and reset, while this PR narrowly fixes that behavior with focused tests; the remaining blocker is the required changelog entry for a user-visible logs fix.

## What This Changes

The PR removes the fast-growth `reset` flag from `readLogSlice` and adds log-tail unit tests for fast-growth truncation versus true file-shrink rotation.

## Best Possible Solution

Land the semantic split and regression tests after adding a single `CHANGELOG.md` Fixes entry credited to @BSG2000, while keeping broader `logs --follow` disconnect work separate.

## Reproduction Assessment

Yes. On current main, read a log once, append more than `maxBytes` while the prior cursor is still valid, then read again with that cursor; `src/logging/log-tail.ts:91` enters the fast-growth branch and sets both `reset` and `truncated`, which the CLI renders as a false rotation notice.

## Solution Assessment

Yes for the code path. Removing `reset` only from the fast-growth branch is the narrowest maintainable fix because true file shrink still reports reset, skipped bytes still report truncation, and the payload shape stays unchanged; the changelog entry is the remaining merge requirement.
