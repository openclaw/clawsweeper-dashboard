## Summary

Keep open: the corrected `sessions.usage` direction matches current Gateway contracts, but the PR head still has a concrete `--output` path bug and lacks inspectable real behavior proof for PNG generation from a cached session.

## What This Changes

The branch adds a core `context` CLI group with a `map` subcommand, Gateway session usage lookup, existing treemap renderer reuse, CLI loader/help registration, and command policy metadata.

## Best Possible Solution

Keep the CLI-command approach if it creates missing output directories before copying, keeps using `sessions.usage`, and is backed by focused CLI validation plus redacted real behavior proof from a cached session run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after a cached report exists, `openclaw context map --output missing-dir/ctx.png` reaches `copyFile` without creating the destination parent directory, so Node will raise `ENOENT`.

## Solution Assessment

No. Reusing `sessions.usage` and `renderContextTreemapPng` is maintainable, but the patch should create `path.dirname(outputPath)` recursively before copying and include proof that the CLI path works in a real setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Create the output directory before copying:** `src/cli/context-cli/register.ts:83`
  - body: When `--output` points inside a directory that does not already exist, `copyFile` throws `ENOENT`, so the advertised custom output path can fail. Create `path.dirname(outputPath)` with `{ recursive: true }` before copying, matching the existing CLI output-copy pattern.
