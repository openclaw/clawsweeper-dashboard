## Summary

Keep open: the PR is recent and useful, but the branch is not mergeable because the new command calls unsupported `sessions.list` params while current main exposes the desired report through `sessions.usage`, and the contributor has not provided real gateway/session proof yet.

## What This Changes

Adds an `openclaw context map` CLI subcommand that fetches cached session context data through gateway RPC, renders the existing treemap PNG, and wires the command into CLI registration/help/policy catalogs.

## Best Possible Solution

Keep the CLI wrapper direction, but fetch reports through `sessions.usage` or deliberately extend `sessions.list` with schema, handler, and regression coverage, then require redacted real gateway/session proof that a PNG is rendered.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: every invocation sends `includeContextWeight` to `sessions.list`, and `--session` also sends `key`; current main rejects both before rendering.

## Solution Assessment

No. Reusing `renderContextTreemapPng` is a good direction, but the gateway fetch should use the supported `sessions.usage` report path or an explicitly added `sessions.list` contract with tests; the output copy should also follow existing mkdir/copy patterns if nested paths are supported.
