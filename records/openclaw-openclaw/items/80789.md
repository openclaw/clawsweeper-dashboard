## Summary

Keep open: the new CLI feature is useful in direction, but this PR currently calls the wrong gateway RPC contract and still lacks real after-fix behavior proof.

## What This Changes

Adds an `openclaw context map` CLI subcommand that fetches a cached session context report through the gateway and writes the existing treemap PNG renderer output to disk.

## Best Possible Solution

Keep the CLI wrapper only if maintainers accept the new surface, fetch cached reports through the supported `sessions.usage` contract, add focused regression coverage, and require real gateway proof that a PNG is written.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: every invocation sends `includeContextWeight` to `sessions.list`, and current-main validation rejects that field before any report can be returned.

## Solution Assessment

No. Reusing `renderContextTreemapPng` is a good direction, but the PR should use `sessions.usage` or intentionally extend the gateway contract instead of sending unsupported params to `sessions.list`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the report-bearing sessions usage RPC:** `src/cli/context-cli/register.ts:27`
  - body: `sessions.list` validates against `SessionsListParamsSchema`, which does not allow `includeContextWeight` or `key` and has `additionalProperties: false`. Since this command always sends `includeContextWeight`, the gateway rejects `openclaw context map` before it can render; call `sessions.usage` or add a supported list contract with tests.
  - confidence: 0.94
- **[P3] Create the output directory before writing:** `src/cli/context-cli/register.ts:83`
  - body: When `--output` points under a missing parent directory, `writeFile(outputPath, ...)` throws `ENOENT` after rendering the temp PNG. Match other CLI output paths by creating `path.dirname(outputPath)` and using `copyFile` for the generated PNG.
  - confidence: 0.82
