## Summary

Current main already routes usage-cost cache writes through the shared atomic file writer, which cleans temporary files on failed write/rename paths; the reported v2026.5.6 direct-temp writer is no longer present on main.

## What This Changes

The issue asks OpenClaw to stop leaving orphaned `.usage-cost-cache.json.*.tmp` files in agent session stores after failed or interrupted usage-cost cache writes.

## Best Possible Solution

Keep the current main implementation that uses the shared atomic writer; any one-time cleanup for files already leaked by v2026.5.6 can be handled separately as a doctor or maintenance enhancement if maintainers want it.

## Reproduction Assessment

Yes for the reported release by source inspection: v2026.5.6 directly wrote and renamed the usage-cost temp file without a cleanup `finally`. I did not run a live disk-failure repro, and current main no longer has that vulnerable write path.

## Solution Assessment

Yes. Reusing the shared atomic writer is the narrowest maintainable fix for failed write/rename cleanup and avoids another bespoke temp-file implementation in the usage-cost cache path.
