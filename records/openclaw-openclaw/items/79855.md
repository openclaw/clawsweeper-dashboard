## Summary

Keep open: the branch adds a plausible new SDK/Discord capability, but it is not mergeable because it imports new hook types from a public SDK subpath that does not export them, the typing activity path still lacks Discord typing gateway intents, and the external contributor has not supplied real Discord behavior proof.

## What This Changes

The PR adds an `inbound_activity` plugin hook with debouncer extend/cancel controls, wires Discord message/typing/delete events to it, and documents the new hook.

## Best Possible Solution

Land a maintainer-approved public hook only after the SDK export, Discord intent bits, focused coverage, docs, and real Discord smoke proof all line up.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR imports hook types from a subpath that does not export them, and Discord's gateway contract shows `TYPING_START` requires typing intents that current main omits. No live Discord reproduction was run.

## Solution Assessment

No. The hook direction may be reasonable, but this PR should first expose the new types through the public SDK subpath, request typing intents, add focused coverage, and provide redacted live Discord proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Export the inbound activity types from this SDK subpath:** `extensions/discord/src/monitor/message-handler.ts:5-9`
  - body: `openclaw/plugin-sdk/plugin-runtime` does not export `PluginHookInboundActivityContext` or `PluginHookInboundActivityEvent`; this PR only adds those exports to `src/plugins/hooks.ts`. As written, this import will fail typechecking before the Discord hook code can build, so either export the new types through the public subpath or import them from a public SDK module that actually exposes them.
  - confidence: 0.95

  - confidence: 0.9
