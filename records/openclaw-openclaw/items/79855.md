## Summary

Keep open. The hook direction is plausible and the PR now has sufficient live Discord proof, but the patch still has two merge-blocking issues: it can retain Discord message IDs for non-debounced traffic, and it changes the public Plugin SDK surface without refreshing the tracked API baseline.

## What This Changes

The branch adds a plugin `inbound_activity` hook, debouncer extend/cancel plumbing, Discord message/typing/delete emitters and intents, and hook documentation.

## Best Possible Solution

Restrict activity-key bookkeeping to real pending buffers, refresh the SDK API baseline, and land only after maintainers accept the new hook semantics.

## Reproduction Assessment

Not applicable as a feature PR; current main lacks an `inbound_activity` hook rather than having broken existing behavior. The two review blockers are source-inspection reproducible from the PR diff and current debounce defaults.

## Solution Assessment

No, not merge-ready yet. The proposed owner boundary is plausible, but the PR should first fix the pending-message key retention path and regenerate the public SDK API baseline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid retaining non-debounced message keys:** `extensions/discord/src/monitor/message-handler.ts:189`
  - body: `buildActivity` stores every message id in `pendingMessageKeys` before the debouncer knows whether a buffer will exist. When inbound debounce is absent/0, the message is a command/media item, or the key map is saturated, `createInboundDebouncer` takes the immediate path and calls `runFlush` directly, so Discord's `onFlush`/`onCancel` cleanup never runs. That leaves normal Discord traffic retaining message ids for the handler lifetime, even when no `inbound_activity` hook is registered.
  - confidence: 0.88
- **[P2] Regenerate the plugin SDK API baseline:** `src/plugins/hook-types.ts:83`
  - body: This adds a public hook name and exported hook types that flow through `openclaw/plugin-sdk/plugin-runtime`, but the PR does not update `docs/.generated/plugin-sdk-api-baseline.sha256`. The repo's `plugin-sdk:api:check` script is the guard for intentional SDK surface changes, so the baseline hash needs to be refreshed with the hook change.
  - confidence: 0.86
