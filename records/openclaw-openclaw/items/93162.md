- `P2`: The item affects WebChat session continuity with a bounded but real session-state impact.

## Summary

Keep open: current source supports the reporter's observation that Control UI can create `agent:<id>:dashboard:<uuid>` sessions, but current docs and tests also treat dashboard sessions as intentional, so this needs maintainer product direction rather than conservative cleanup closure.

## What This Changes

The issue asks for Control UI WebChat sessions to honor `session.dmScope: "main"` and share `agent:main:main` instead of creating isolated dashboard session keys.

## Best Possible Solution

Maintainers should decide whether WebChat is an operator/dashboard multi-session surface or a direct-channel participant for `dmScope`, then align Gateway session creation, Control UI behavior, and docs around that decision.

## Reproduction Assessment

Partly: source inspection confirms the dashboard-key creation path, and current docs confirm that dashboard sessions are intentional for New Chat. I did not establish a live current-main repro that ordinary initial WebChat tabs create fresh UUID sessions without user New Chat action.

## Solution Assessment

Unclear: unconditional WebChat-to-main aliasing is not obviously the best fix because current dashboard sessions are documented behavior. The safer end state needs a maintainer decision, likely either an explicit WebChat session mode/binding design or clearer docs preserving the current operator-session model.
