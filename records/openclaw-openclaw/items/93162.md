## Label Justifications

- `P2`: The report concerns a normal-priority session-context isolation problem with limited blast radius and source-backed current behavior.
- `impact:session-state`: The item is about whether WebChat conversation context is shared or isolated under the configured direct-message session scope.

## Summary

Current main already covers the reported WebChat main-session behavior, and the same handling is present in the latest release: fresh WebChat aliases normalize to the agent main session, and WebChat New Chat from the agent main session resets `agent:main:main` in place under `session.dmScope: "main"` instead of creating a dashboard UUID child.

## What This Changes

The issue reports that WebChat creates isolated dashboard session keys even when direct-message scope is configured to use the shared main session.

## Best Possible Solution

Keep the shipped WebChat main-session normalization and in-place main reset behavior; treat any remaining dashboard UUID case as a new report only if it occurs without explicit New Chat or user-selected child-session intent.

## Reproduction Assessment

No. Source and regression-test inspection show that fresh WebChat main aliases resolve to `agent:main:main`, and WebChat New Chat from the agent main session resets `agent:main:main` in place under `session.dmScope: "main"`.

## Solution Assessment

Yes. The current implementation keeps session ownership at the Gateway/session-routing boundary and preserves explicit dashboard child sessions only when the user creates or selects one, which is narrower than adding a new WebChat-specific config path.
