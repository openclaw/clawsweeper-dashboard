## Summary

Keep open: this PR targets a real current-main Codex app-server tool-policy bypass, but as written it still leaves `disableTools` no-tool runs with native code mode enabled and only provides synthetic test proof.

## What This Changes

The PR gates Codex app-server native code mode, dynamic tool filtering, and plugin app config on restricted runtime `toolsAllow` policies, with focused app-server regression tests.

## Best Possible Solution

Update the Codex app-server native-surface gate to fail closed for all no-tool runtime states, then land after redacted live Telegram/EVE proof shows denied senders get no native tools, dynamic tools, MCP servers, or plugin apps.

## Reproduction Assessment

Yes, source-reproducible: current main treats `toolsAllow: []` as unrestricted for Codex dynamic tools and always enables native code mode in thread config. I did not execute a live Telegram/Codex run in this read-only review.

## Solution Assessment

No, not as written: gating the Codex app-server native and plugin surfaces is the right boundary, but the patch must also honor `disableTools` before it fully closes the no-tool policy bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
