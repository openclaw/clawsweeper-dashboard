## Summary

Keep this PR open: it targets a real heartbeat source-reply privacy bug and is paired with the open linked issue, but the patch is not merge-ready because the new suppression still lets no-tool media payloads through and the supplied proof is mock-only.

## What This Changes

The PR changes the heartbeat runner so plain text heartbeat replies are suppressed when message-tool mode expected `heartbeat_respond` but the model did not call it, and adds a focused regression test for the text case.

## Best Possible Solution

Land one canonical fix that suppresses the whole no-tool heartbeat main payload under message-tool-only mode, except explicit internal notices and any source-proven relayable exec-completion exception, with regression coverage and real Telegram or recorder-backed runner proof.

## Reproduction Assessment

Yes, the issue has a high-confidence source reproduction path: configure `messages.visibleReplies: "message_tool"`, return a no-tool heartbeat final payload, and current main falls through to durable channel delivery when `shouldSkipMain` is false. I did not execute it because this review is read-only.

## Solution Assessment

No, this is only a partial fix. The right shape is the same runner-level guard, but it must suppress the whole main reply payload in message-tool-only mode and include real behavior proof, not only a text-only unit case.
