## Summary

This issue should stay open: current main still registers read-only discovery context-engine factories and immediately invokes the selected factory during resolution, while the latest lossless-claw package intentionally rejects engine initialization in read-only registration. I found no merged/current-main fix or canonical newer issue that covers this v2026.6.10 regression.

## What This Changes

Investigate a v2026.6.10 regression where the external lossless-claw context engine loads but is quarantined during read-only plugin registration, causing OpenClaw to fall back to the legacy context engine.

## Best Possible Solution

Keep this as the active bug and fix OpenClaw so read-only discovery can collect context-engine registrations without constructing stateful plugin engines, while full gateway/agent runtime resolution still selects `lossless-claw` after normal plugin activation.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection is reproducible: OpenClaw invokes the selected factory during context-engine resolution, and lossless-claw 0.13.1 throws the reported error when that factory runs in read-only registration.

## Solution Assessment

Unclear until a fix PR traces the exact startup path, but the narrow direction is to separate read-only registration from stateful context-engine factory initialization. Adding a new config option or telling users to select `legacy` would not be the best fix because the existing context-engine slot contract should keep working.
