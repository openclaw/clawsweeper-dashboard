## Summary

Current main has strong generic defenses for malformed tool-call/result history and user-visible provider conversation-state errors, but the report’s specific 25-result trigger, silent retry loop, and session/queue accumulation are not proven fixed. The issue should remain open until a focused current-main reproduction separates transcript pairing from retry and session-lifecycle behavior.

## What This Changes

Fix long-session provider request failures attributed to histories with many tool results, surface terminal errors, and address resulting session accumulation or document safe cleanup.

## Best Possible Solution

Add a focused long-history regression that captures the provider-bound request after more than 25 valid tool-call/result pairs, then fix only the failing canonical boundary; ensure unrecoverable conversation-state errors terminate retries visibly, and track session retention cleanup separately if accumulation reproduces.

## Reproduction Assessment

Partly yes: current source and tests clearly define the invalid tool-use/result pairing failure class, but there is no high-confidence current-main reproduction of the claimed 25-result threshold or the full retry, file-growth, and queue-lock cascade.

## Solution Assessment

No, an arbitrary tool-result count cap is not yet the best-supported fix. The maintainable path is pairing-preserving transcript normalization plus terminal retry handling, with session pruning considered separately only if retention behavior independently reproduces.
