## Label Justifications

- `P2`: The official Play release serves most users, while trusted sideload distribution remains a meaningful but non-emergency product improvement.
- `impact:security`: The issue concerns trusted executable provenance and users have reported encountering unofficial Android listings and packages in the absence of an official sideload artifact.

## Summary

Current main and v2026.6.5 now provide an official Google Play install path, resolving the earlier confusion and unsafe search for unofficial apps, but the requested signed APK/GitHub-release path remains absent and materially distinct because the third-party flavor retains capabilities excluded from Play. The issue also carries explicit maintainer/product-decision gates, so it should remain the canonical tracker for whether OpenClaw will support official sideload distribution.

## What This Changes

Request official prebuilt Android APK downloads through OpenClaw releases or another supported distribution channel.

## Best Possible Solution

Keep Google Play as the canonical supported install path, then either publish a signed, provenance-verifiable third-party APK with explicit capability and update documentation or explicitly document that official sideload distribution is not supported.

## Reproduction Assessment

Not applicable; this is a distribution capability request, and current docs, scripts, and release assets directly show that Play distribution exists but official APK downloads do not.

## Solution Assessment

No, the original suggestion to upload both debug and release APKs on every release is not sufficiently safe; the best path requires a maintainer-approved signed flavor, provenance, update policy, and release-readiness gate.

## Risks / Open Questions

- An official sideload channel distributes executable packages outside Play’s review and update mechanisms, so signing-key custody, reproducible provenance, update behavior, and compromise response need explicit ownership.
- The Play and third-party flavors expose different capability sets; publishing the wrong artifact or documenting them ambiguously could surprise users about SMS, Call Log, and photo access.
- The Android README still marks the app extremely alpha with full end-to-end QA and release hardening incomplete, so artifact automation alone would overstate release readiness.
