## Label Justifications

## Summary

Current main and v2026.6.5 provide an official Google Play install path, resolving the earlier discoverability and impersonation confusion, but they still do not publish an official APK in GitHub Releases. This issue remains the canonical request for a supported sideload channel, and the repository’s explicit product-decision label plus signing, flavor, update, and support choices make it unsuitable for automatic implementation.

## What This Changes

Request official prebuilt Android APK assets in OpenClaw releases.

## Best Possible Solution

Keep Google Play as the supported default, then have Android/release owners decide whether to add one signed third-party APK channel with stable signing identity, checksums/provenance, an update policy, release automation, and a documented official download path.

## Reproduction Assessment

Not applicable; this is a new distribution-channel request, and current source plus the v2026.6.5 asset inventory directly establish that GitHub Releases contain no official APK.

## Solution Assessment

No, the original suggestion to upload both debug and release APKs is not yet the best solution because it leaves signing identity, flavor selection, upgrades, provenance, and long-term support undefined; one explicitly supported signed artifact is safer.

## Risks / Open Questions

- Publishing a release APK without a chosen signing identity and key-custody process could create an unsafe or non-upgradable sideload channel.
- Play and third-party flavors have different capabilities and distribution expectations, so choosing the wrong artifact could create user confusion or an unsupported permanent contract.
- A GitHub APK channel needs an update and support policy; otherwise sideloaded installations may silently lag the Gateway protocol, as the related open report https://github.com/openclaw/openclaw/issues/85971 illustrates for version skew.
