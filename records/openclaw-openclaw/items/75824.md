## Summary

Keep open. The report is fresh, has multiple independent confirmations across macOS, WSL2, and Linux, and current main still matches the visible failure path: OpenAI Responses runs can end in the embedded-runner idle-timeout path while the Control UI clears the in-flight assistant stream on chat error events instead of adding a visible assistant transcript message. The related open PRs address UI error surfacing, but they are not merged and do not prove the OpenAI Responses transport/runtime root cause is fixed.

## What This Changes

The issue reports that OpenAI-backed Web UI, CLI, and channel runs time out and surface no assistant reply on 2026.4.29/2026.4.30 even though direct OpenAI API calls from the same environments succeed.

## Best Possible Solution

Keep the issue open and fix the OpenAI Responses embedded-runner path so generated output is consumed and surfaced across Web UI, CLI, and channel sessions; separately land or replace the WebChat inline error-surfacing work so genuine provider failures remain visible in the transcript.

## Reproduction Assessment

Yes for the observable failure class: the issue and comments give concrete version, provider, model, and prompt steps, and three environments report the same idle-timeout/no-output behavior while direct OpenAI calls work. The exact source-level root remains unclear until a diagnostics or trajectory bundle confirms which transport path failed in a live OpenClaw run.

## Solution Assessment

Not applicable as a patch review. The best path is to coordinate with the open UI error-surfacing PRs, collect diagnostics for the OpenAI Responses timeout, and then make a narrow provider/runtime fix instead of closing this as a UI-only duplicate.
