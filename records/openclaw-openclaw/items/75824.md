## Summary

Keep open. Multiple independent reports now corroborate the OpenAI idle-timeout/no-output path, while current main still routes direct OpenAI Responses auto traffic through the WebSocket-first path and the Control UI still clears live error state without appending a visible assistant error bubble.

## What This Changes

The issue asks OpenClaw to return a normal Web UI assistant response for direct OpenAI API models instead of timing out and removing the in-flight assistant bubble while direct OpenAI API calls succeed.

## Best Possible Solution

Diagnose the direct OpenAI Responses transport path with live failing-run artifacts, fix the stream-consumption or fallback behavior if confirmed, and make failed Web UI turns remain visible with an inline error while preserving the existing history-reconciliation guarantees.

## Reproduction Assessment

Yes for the user-observed scenario: configure the direct OpenAI API provider, send a simple Web UI prompt such as `Reply only: OK`, and compare OpenClaw’s idle-timeout/empty-bubble result against a direct OpenAI Responses call that succeeds. The exact source-level root still needs a diagnostics export or trajectory bundle from an affected run.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable direction is provider/runtime investigation first, with #40277 and #55923 treated as related UI symptom work rather than a complete fix for the OpenAI timeout root.
