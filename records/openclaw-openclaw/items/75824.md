## Summary

Keep open. The report is fresh, independently corroborated across macOS, WSL2, and Debian, and current main still has an OpenAI Responses transport path plus Web UI error handling that match the observed timeout and disappearing-bubble symptoms without proving the root is fixed.

## What This Changes

The issue reports OpenAI-backed web chat creating a transient assistant bubble that disappears after the embedded OpenAI run times out, despite direct OpenAI API calls succeeding.

## Best Possible Solution

Fix or rule out the OpenAI Responses transport/stream-consumption path with live diagnostics, then land the Web UI inline-error and history-reconciliation work so failed turns remain visible while the provider/runtime root is resolved.

## Reproduction Assessment

Yes for the user-observed scenario: configure the direct OpenAI API provider, send a simple Control UI or channel prompt, and compare OpenClaw's idle-timeout/empty-assistant result against a direct OpenAI call that succeeds. The exact source-level failure still needs a diagnostics export or trajectory bundle from an affected run.
