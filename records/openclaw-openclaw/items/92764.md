## Summary

Keep this PR open: the state and extraction changes follow the existing typed progress contract, but the proposed render line is in a helper that the normal live tool-message shell bypasses, so source review shows the main webchat path can still fail to display the progress text. The external real behavior proof gate is also still test/build/source-only.

## What This Changes

The PR adds Control UI handling for `stream:"item"` tool progress events, emits `toolprogress` content, stores it on `ToolCard`, and tries to render it until the final tool result arrives.

## Best Possible Solution

Update the UI so `ToolCard.progressText` is rendered in the actual tool-shell/shared render path, add focused grouped-render coverage, then provide real webchat proof or an explicit maintainer proof override before merge.

## Reproduction Assessment

Yes, at source level: current main emits typed `stream:"item"` tool `progressText`, while the current UI handler returns before non-tool streams can update tool cards. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No, not yet: the PR handles the right state path but renders progress only through `renderToolCard`, while the normal live tool-shell path uses its own renderer. The safer fix is to render the progress in the shared or tool-shell path and prove it in real webchat.
