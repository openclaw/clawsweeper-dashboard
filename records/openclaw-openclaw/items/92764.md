Keep this PR open: the underlying WebChat tool-progress bug is source-reproducible, but the branch still has blocking patch issues. ## What This Changes

This PR adds Control UI handling for `stream:"item"` tool progress events, emits/extracts a `toolprogress` content item, renders `progressText` on tool cards, and adds handler-level tests.

## Best Possible Solution

Move the progress display into the shared or tool-shell render path, fix the added test type error, add focused grouped-render coverage for a `toolprogress` content item, then provide real WebChat proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes, at source level: current main emits typed public tool progress as `stream:"item"` data, while the current UI handler returns before non-tool streams can update tool cards. I did not run a live browser reproduction in this read-only review.

## Solution Assessment

No. The PR handles the right event state, but the visible rendering is added only to `renderToolCard` while normal live tool messages use the grouped tool-shell path; the safer fix is to render progress in that shared path and prove it in WebChat.
