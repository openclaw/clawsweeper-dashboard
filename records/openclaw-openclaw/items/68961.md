## Summary

Keep open. Current main does not contain the claude-sdk runtime dispatch/adapter, and the PR is still an active implementation candidate, but the branch is not mergeable as-is because the SDK bridge drops existing tool-policy and Claude Code runtime-contract behavior on security-sensitive agent execution paths.

## What This Changes

The PR adds an opt-in Claude Agent SDK runtime adapter, native OpenClaw tool bridging through an in-process MCP server, config/plugin SDK surfaces, docs/tests, and generated `.claude/agents` files while retaining the legacy embedded runtime default after reverting the proposed default flip.

## Best Possible Solution

Keep the work as an explicit opt-in runtime candidate, but require the SDK path to preserve embedded-runtime tool allow/deny semantics, Claude Code prompt/settings behavior, transcript/tool-result error semantics, and live Anthropic smoke proof before considering any default flip or legacy-runtime migration.

## Reproduction Assessment

Yes for the blocking PR defects: the static path is enough to reproduce them. A cron or isolated run with `toolsAllow: ["read"]` reaches `runClaudeSdkAgent`, which builds the full native MCP tool inventory without applying the embedded allow-list filter; live end-to-end behavior still needs a real SDK smoke.

## Solution Assessment

No. The direction is plausible, but this implementation is not the best mergeable solution until it applies runtime tool filtering to both native MCP and SDK built-ins, loads the intended Claude Code settings/prompt contract, and preserves failed tool status in MCP results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter native MCP tools by toolsAllow:** `src/agents/claude-sdk/run.ts:233-267`
  - body: `toolsAllow` is the runtime allow-list for cron/isolated runs, and the embedded runner filters the full tool inventory with it. The SDK path builds `nativeMcpServers` from the unfiltered `createOpenClawCodingTools()` inventory, while the later `tools` option only controls SDK built-ins, so a run constrained to `toolsAllow: ["read"]` can still expose OpenClaw MCP tools such as `message`, `cron`, or `sessions_spawn`.
  - confidence: 0.93
- **[P1] Map OpenClaw deny IDs to SDK built-ins:** `src/agents/claude-sdk/sdk-tool-policy.ts:56-61`
  - body: `collectSdkDisallowedTools` forwards deny entries verbatim, but OpenClaw config commonly uses canonical IDs like `exec`, `read`, and `edit`, while the SDK built-ins are named `Bash`, `Read`, and `Edit`. With `tools.deny: ["exec"]`, the OpenClaw exec tool can be filtered while SDK `Bash` remains available, leaving command execution exposed under claude-sdk.
  - confidence: 0.9
- **[P1] Load filesystem settings explicitly:** `src/agents/claude-sdk/run.ts:552-563`
  - body: The SDK options never set `settingSources`; for `@anthropic-ai/claude-agent-sdk@0.1.77`, omitted or empty `settingSources` loads no filesystem settings and must include `project` to load `CLAUDE.md`. That means claude-sdk runs ignore checked-in `.claude` settings and project instructions that are expected to govern Claude Code behavior.
  - confidence: 0.84
- **[P2] Always pass the Claude Code preset:** `src/agents/claude-sdk/run.ts:490-496`
  - body: When `extraSystemPrompt` is absent, `systemPromptOption` is `undefined`, so ordinary SDK runs use the minimal SDK prompt instead of the Claude Code preset. Pass `{ type: "preset", preset: "claude_code" }` by default and add `append` only when there is extra prompt text.
  - confidence: 0.88
- **[P2] Return MCP errors for failed tool results:** `src/agents/claude-sdk/native-tools-adapter.ts:183-203`
  - body: `agentResultToMcp` always returns `isError: false`, but OpenClaw tools often report failures via `details.status` or JSON payloads without throwing. The model will see failed exec/session/message results as successful MCP calls and may continue from an invalid state.
  - confidence: 0.86
