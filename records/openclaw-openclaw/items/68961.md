## Summary

Keep open. Current main does not implement the claude-sdk runtime or public runAgent dispatch, so this PR is still the active implementation candidate, but the branch is not merge-ready because the new SDK path still bypasses existing tool policy and Claude Code runtime contracts in security-sensitive paths.

## What This Changes

The PR adds an opt-in Claude Agent SDK runtime driver with native OpenClaw MCP tool bridging, runtime dispatch/API exports, config schema support, generated Claude sub-agent files, docs, and tests while leaving the embedded runtime as the default after reverting the default flip.

## Best Possible Solution

Keep the branch as an explicit opt-in runtime candidate, but require it to preserve embedded-runtime allow/deny semantics for native and SDK tools, load the intended Claude Code prompt/settings contract, preserve transcript/tool-result error semantics, and pass targeted plus live smoke validation before any default flip.

## Reproduction Assessment

Yes for the blocking PR defects: static inspection reproduces the tool-policy and SDK-option gaps. A constrained cron or isolated run with `toolsAllow: ["read"]` would still build an OpenClaw MCP server from the full native inventory, and SDK source confirms omitted `settingSources` loads no project settings.

## Solution Assessment

No. The direction is plausible, but the current implementation is not the narrowest maintainable solution until it routes all runtime policy through the same effective tool inventory and explicitly sets the SDK prompt/settings behavior OpenClaw depends on.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter native MCP tools by toolsAllow:** `src/agents/claude-sdk/run.ts:233-267`
  - body: `toolsAllow` is the runtime allow-list used by cron and isolated runs, but this branch builds the native MCP server from the full `createOpenClawCodingTools()` inventory and only passes `toolsAllow` to the SDK built-in `tools` option. A run restricted to `read` can still expose OpenClaw MCP tools such as message, cron, or sessions tools.
  - confidence: 0.93
- **[P1] Normalize OpenClaw deny IDs for SDK built-ins:** `src/agents/claude-sdk/sdk-tool-policy.ts:56-61`
  - body: The helper forwards deny entries verbatim to `disallowedTools`, but OpenClaw policy commonly uses canonical IDs such as `exec`, `read`, and `edit` while Claude SDK built-ins are `Bash`, `Read`, and `Edit`. With `tools.deny: ["exec"]`, SDK Bash can remain available.
  - confidence: 0.9
- **[P1] Load project settings for SDK runs:** `src/agents/claude-sdk/run.ts:550-564`
  - body: The SDK options never set `settingSources`, and the SDK defaults that field to an empty list. That means claude-sdk runs ignore project `.claude` settings and CLAUDE.md instructions unless the adapter explicitly includes the project setting source.
  - confidence: 0.86
- **[P2] Always pass the Claude Code preset:** `src/agents/claude-sdk/run.ts:490-496`
  - body: When `extraSystemPrompt` is absent, `systemPromptOption` is `undefined`, so ordinary runs use the SDK's minimal prompt instead of the Claude Code preset. Pass the `claude_code` preset by default and add `append` only when extra prompt text exists.
  - confidence: 0.88
- **[P2] Return MCP errors for failed tool results:** `src/agents/claude-sdk/native-tools-adapter.ts:183-203`
  - body: `agentResultToMcp` marks every non-throwing OpenClaw tool result as successful. OpenClaw tools can report failure via structured result metadata without throwing, so Claude may continue as if a failed native tool call succeeded.
  - confidence: 0.86
- **[P2] Honor ACP runtime entries in runAgent:** `src/agents/runtime-dispatch.ts:46-55`
  - body: `runAgent` is documented as dispatching across claude-sdk, embedded, and ACP, but `selectAgentRuntime` collapses `runtime.type: "acp"` into the default branch and `runAgent` then calls the embedded runner. Callers using the new public dispatch with an ACP-configured agent would bypass the ACP path.
  - confidence: 0.8
