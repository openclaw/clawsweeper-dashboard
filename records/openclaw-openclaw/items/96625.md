## Summary

Keep this PR open for maintainer review: the SQLite storage direction matches the tracked Path 3 work, but the branch currently breaks storage-neutral plugin transcript reads by returning synthetic `sqlite:` targets to helpers that still stream real files.

## What This Changes

The branch makes per-agent SQLite the runtime store for session entries and transcript events, adds a doctor session-SQLite migration submode, updates lifecycle cleanup, and ratchets the old session-file SDK surface.

## Best Possible Solution

Land only after storage-neutral transcript SDK helpers use SQLite-aware readers/writers, the SDK export removal is explicitly accepted or preserved, and the linked Path 3 tracker remains the canonical coordination point.

## Reproduction Assessment

Yes from source inspection, though I did not run tests in this read-only review. A storage-neutral scope now returns a `sqlite:` pseudo-path, and the existing SDK reader still streams that value as a filesystem path that returns no lines.

## Solution Assessment

No. The SQLite flip is the right broad direction for Path 3, but this branch must route public transcript SDK reads/writes through SQLite-aware APIs and settle SDK compatibility before merge.
