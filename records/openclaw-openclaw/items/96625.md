## Summary

Keep this PR open. The SQLite storage direction matches the Path 3 tracker, and the submitted proof is useful, but PR head still routes storage-neutral public SDK transcript reads through filesystem JSONL helpers after the accessor starts returning synthetic `sqlite:` targets; the draft/protected PR also needs maintainer sign-off for the SDK and upgrade contract.

## What This Changes

The PR flips session metadata and transcript runtime access to per-agent SQLite, adds doctor session-SQLite migration modes, bumps the agent DB schema, and ratchets session store SDK compatibility.

## Best Possible Solution

Land only after storage-neutral SDK transcript reads and write-lock helpers use SQLite-aware accessors, the removed SDK export is preserved or explicitly accepted with an upgrade rationale, and the Path 3 tracker signs off the migration behavior.

## Reproduction Assessment

Yes, source inspection gives a high-confidence PR-head reproduction path: append transcript rows through the new SQLite accessor, then read by public SDK identity without `sessionFile`; the SDK still streams the synthetic `sqlite:` value as a filesystem path and returns no events.

## Solution Assessment

No. The SQLite flip is the right broad direction, but this branch is not the best mergeable solution until public SDK transcript helpers are storage-aware and the SDK/export compatibility decision is explicit.
