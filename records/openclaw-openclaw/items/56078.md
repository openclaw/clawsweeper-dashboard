## Summary

Keep this issue open. Current main still defaults Control UI-created and backend-created isolated agent-turn cron jobs to announce delivery, and the implicit no-channel path still reaches the same "Channel is required" error unless the job explicitly selects delivery.mode="none" or bestEffort. The related PRs and discussion are useful context, but the direct cron default regression is not merged or superseded.

## What This Changes

The issue requests that local/no-channel cron jobs continue producing prompt-specified saved output instead of failing cron delivery resolution with "Channel is required".

## Best Possible Solution

Fix the cron product path so implicit local/webchat-only isolated cron jobs do not fail merely because no external delivery channel exists. A conservative implementation should either default Control UI/local implicit jobs to delivery.mode="none" or downgrade only unresolved implicit announce delivery non-fatally, while keeping explicit channel/to delivery strict and documenting the no-deliver workaround.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Control UI default isolated agent-turn cron job in a config with no deliverable channels: the UI submits announce delivery, cron resolves channel "last", and target resolution returns the reported no-channel error. A live WSL2/LM Studio run was not performed.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a proposed patch. The narrow maintainable solution is to handle the implicit no-channel cron path directly in cron/UI defaults or target-resolution downgrade logic, with tests proving explicit delivery still fails when misconfigured.

## Repair Work Prompt

Fix the cron no-channel regression reported in https://github.com/openclaw/openclaw/issues/56078. Current main defaults Control UI and backend-created isolated agent-turn cron jobs to announce delivery; in local/webchat-only configs with no deliverable channel, implicit delivery.channel="last" resolves to "Channel is required" and can turn a successful agent run into a delivery-target failure. Implement the narrow cron-owned fix: implicit/no-target local cron jobs should not fail solely because no external delivery channel exists, but explicit delivery targets such as a configured channel/to/accountId must remain strict. Acceptable directions include defaulting Control UI-created local/no-channel isolated agent-turn jobs to delivery.mode="none" where the UI/backend can prove no deliverable channel, or downgrading only unresolved implicit announce delivery non-fatally while preserving explicit announce/webhook failures. Add regression tests for: Control UI default cron add payload/path in the no-channel shape, backend-created isolated agentTurn jobs without explicit delivery, delivery-plan/target behavior for implicit last with no configured channels, and explicit announce targets still failing when unresolved. Update docs with the no-deliver workaround if behavior/docs change, and add a user-facing changelog Fixes entry crediting @nobbywfc and prior related contributors if their work is reused; do not credit bots. Do not broaden cron output routing or implement the larger file/channel/session target feature from #49078 in this fix.
