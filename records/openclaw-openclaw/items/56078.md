## Summary

Keep open: current main and the latest release still default UI and backend-created isolated agent-turn cron jobs to strict `announce` delivery against `last`, and a local/no-channel setup still reaches the reported `Channel is required` delivery-target error. The directly related fixes are closed unmerged or cover adjacent multi-channel/webchat diagnostics rather than this exact implicit no-channel cron output failure.

## What This Changes

Cron jobs in local or webchat-only setups should preserve generated output instead of failing delivery resolution when no external channel is configured.

## Best Possible Solution

Cron-owned implicit/no-channel isolated agent-turn runs should avoid mandatory fallback announce delivery unless the user explicitly chooses a deliverable target, while explicit announce and webhook delivery remain strict and tested.

## Reproduction Assessment

Yes, source-level. A default Control UI or backend-created isolated `agentTurn` cron job still requests strict announce delivery to `last`, and with no prior/configured channel the resolver reaches the reported `Channel is required` path before dispatch treats it as fatal.

## Solution Assessment

Yes. The narrow maintainable fix is to handle implicit no-channel cron delivery in the UI/default/runtime path while preserving strict behavior for explicit delivery targets; the current related PRs do not yet solve this exact output failure.
