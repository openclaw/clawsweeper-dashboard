## Summary

Keep this issue open. Current main and the latest shipped tag still route default Control UI and backend isolated agent-turn cron jobs through implicit announce delivery, so a local/no-channel setup can still hit the reported delivery-target error unless the job explicitly selects no delivery or a valid delivery target.

## What This Changes

Cron jobs in local/no-channel setups should still save requested output files instead of failing because implicit delivery cannot find a configured channel.

## Best Possible Solution

Cron-owned implicit/no-channel isolated runs should avoid required fallback announce delivery unless the user explicitly chooses a deliverable target, while explicit announce and webhook delivery remain strict.

## Reproduction Assessment

Yes, at source level. A default Control UI isolated agent-turn cron job still submits announce delivery on channel "last", and with no previous or configured channel the target resolver reaches the reported no-channel error path.

## Solution Assessment

Not applicable as a patch review. The narrow maintainable fix is to handle implicit no-channel cron delivery in the cron/UI default path or target-resolution downgrade path while preserving strict behavior for explicit delivery targets.
