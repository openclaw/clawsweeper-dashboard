- `impact:data-loss`: The reported user-visible failure is that generated cron report files are not created or preserved at the requested path.

## Summary

Keep open: current main and v2026.5.18 still have the source-level no-channel cron path reported here, and the related fix PRs are closed without merging. Default UI/backend isolated agent-turn cron jobs still request strict announce delivery to `last`, so local/webchat-only setups with no configured channel can still hit the reported resolver error.

## What This Changes

The issue asks for cron jobs in local/webchat-only setups to save or complete their requested output without failing on missing external delivery channels.

## Best Possible Solution

Cron-owned implicit/no-channel isolated runs should avoid mandatory fallback announce delivery unless the user explicitly chooses a deliverable target, while explicit announce and webhook delivery remain strict.

## Reproduction Assessment

Yes, at source level. A default Control UI or backend-created isolated agent-turn cron job still requests announce delivery to `last` with best-effort disabled, which reaches the no-channel target resolver failure when no previous or configured channel exists.

## Solution Assessment

Yes, the narrow maintainable fix is to handle implicit no-channel cron delivery in the UI/default path or target-resolution downgrade path while preserving strict behavior for explicit delivery targets.

## Repair Work Prompt

Fix the no-channel implicit cron delivery regression reported in https://github.com/openclaw/openclaw/issues/56078. A default Control UI or backend-created isolated agentTurn cron job currently defaults to strict announce/last delivery with bestEffort false, so local/webchat-only setups without configured external channels fail with the reported Channel is required path even when the job prompt should save output locally. Keep explicit announce and webhook delivery strict, preserve channel-backed cron behavior, and avoid adding a new config option. Likely fix boundary: adjust the implicit/no-channel default or downgrade path for cron-owned implicit delivery only, then add regression coverage for UI-created jobs, backend-created jobs, delivery planning, and unresolved target dispatch. Include a changelog entry if the fix changes user-visible cron behavior. Do not broaden into warning-status modeling; https://github.com/openclaw/openclaw/issues/49190 owns that separate feature.
