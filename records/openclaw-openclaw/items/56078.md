## Summary

Keep this issue open. Current main still defaults Control UI-created and backend-created isolated agent-turn cron jobs to announce delivery, so local/no-channel cron runs can still reach the reported delivery-target failure unless the job explicitly selects no delivery or best-effort delivery.

## What This Changes

The issue requests that local/webchat-only cron jobs save prompt-generated output instead of failing delivery resolution when no channel is configured.

## Best Possible Solution

Fix the cron-owned local/no-channel path so implicit isolated cron jobs do not fail solely because no external channel exists, while keeping explicit announce/webhook delivery strict and documenting the no-deliver workaround if behavior changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Control UI default isolated agent-turn cron job in a config with no deliverable channels: the UI submits announce delivery, cron resolves channel "last", and target resolution returns the reported no-channel error.

## Solution Assessment

Not applicable as a patch review because this is an issue, not a proposed patch. The narrow maintainable fix is to handle implicit no-channel cron delivery in cron/UI defaults or target-resolution downgrade logic, with regression tests proving explicit delivery still fails when misconfigured.

## Repair Work Prompt

Fix the cron no-channel regression reported in https://github.com/openclaw/openclaw/issues/56078. Current main defaults Control UI and backend-created isolated agent-turn cron jobs to announce delivery; in local/webchat-only configs with no deliverable channel, implicit delivery.channel="last" resolves to "Channel is required" and can turn a successful agent run into a delivery-target failure. Implement the narrow cron-owned fix: implicit/no-target local cron jobs should not fail solely because no external delivery channel exists, but explicit delivery targets such as configured channel/to/accountId must remain strict. Acceptable directions include defaulting Control UI-created local/no-channel isolated agent-turn jobs to delivery.mode="none" where the UI/backend can prove no deliverable channel, or downgrading only unresolved implicit announce delivery non-fatally while preserving explicit announce/webhook failures. Add regression tests for the Control UI default cron add payload/path in the no-channel shape, backend-created isolated agentTurn jobs without explicit delivery, delivery-plan/target behavior for implicit last with no configured channels, and explicit announce targets still failing when unresolved. Update docs with the no-deliver workaround if behavior/docs change, and add a user-facing CHANGELOG.md Fixes entry crediting @nobbywfc and prior related contributors if their work is reused; do not credit bots. Do not broaden cron output routing or implement the larger file/channel/session target feature from #49078 in this fix.
