## Label Justifications

- `P1`: This regression breaks scheduled/manual cron output workflows for local or webchat-only users with no configured external channel.
- `impact:data-loss`: The reported user-visible failure is that generated cron report output is not created or preserved at the requested path.

## Summary

Keep open: current main and the latest release still default Control UI and backend-created isolated agent-turn cron jobs to strict `announce` delivery against `last`, so a local/no-channel setup can still hit the reported `Channel is required` delivery-target failure. Related closed PRs were not merged, and the open adjacent work covers multi-channel ambiguity or webchat-specific diagnostics rather than this exact implicit no-channel cron output regression.

## What This Changes

Cron jobs in local/no-channel setups should save or preserve agent output instead of failing delivery resolution with `Channel is required`.

## Best Possible Solution

Cron-owned implicit/no-channel isolated agent-turn runs should avoid mandatory fallback announce delivery unless the user explicitly chooses a deliverable target, while explicit announce and webhook delivery remain strict and tested.

## Reproduction Assessment

Yes, source-level. A default Control UI or backend-created isolated `agentTurn` cron job still requests strict announce delivery to `last`, and with no prior/configured channel the resolver reaches the reported `Channel is required` path before dispatch treats it as fatal.

## Solution Assessment

Yes. The narrow maintainable fix is to handle implicit no-channel cron delivery in the UI/default/runtime path while preserving strict behavior for explicit delivery targets; the current related PRs do not yet solve this exact output failure.
