## Label Justifications

- `P1`: This regression can break scheduled/manual cron output workflows for local or webchat-only users with no configured external channel.
- `impact:data-loss`: The reported user-visible failure is that generated cron report output is not created or preserved at the requested path.

## Summary

Keep open: current `main` still maps default UI and backend isolated cron agent-turn jobs to strict `announce` delivery against `last`, and the no-configured-channel path still becomes a fatal delivery-target error. The directly related attempted fixes are closed unmerged, while the open adjacent work at https://github.com/openclaw/openclaw/pull/80438, https://github.com/openclaw/openclaw/pull/80985, and https://github.com/openclaw/openclaw/issues/80984 covers multi-channel validation, explicit webchat rejection, or diagnostics rather than this local/no-channel cron output failure.

## What This Changes

The issue reports a regression where local or webchat-only cron jobs run but do not preserve the expected output because cron delivery fails with `Channel is required (no configured channels detected)`.

## Best Possible Solution

Cron-owned implicit/no-channel isolated agent-turn runs should avoid mandatory fallback announce delivery unless the user explicitly chooses a deliverable target, while explicit announce and webhook delivery remain strict and covered by regression tests.

## Reproduction Assessment

Yes, source-level. A default Control UI or backend-created isolated `agentTurn` cron job still requests strict announce delivery to `last`, and with no configured channel the resolver reaches the reported `Channel is required` path before dispatch treats it as fatal.

## Solution Assessment

Yes. The narrow maintainable fix is to handle implicit no-channel cron delivery in the UI/default/runtime path while preserving strict behavior for explicit delivery targets; the current related PRs do not yet solve this exact output failure.
