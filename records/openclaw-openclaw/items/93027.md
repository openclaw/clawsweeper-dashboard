## Summary

Keep open. The bug and narrow intent are valid, but the patch is not merge-ready because the state-dir dotenv path can still re-enter `loadCliDotEnv()` and throw `process.cwd()` after the new catch, and the supplied proof does not run the real deleted-cwd scenario.

## What This Changes

The PR wraps `shouldLoadCliDotEnv()`'s cwd-local `.env` probe in a try/catch so a deleted cwd falls through to state-dir `.env` detection.

## Best Possible Solution

Resolve cwd once during CLI dotenv startup, skip workspace `.env` when cwd is unavailable, still load the state-dir `.env`, and add regression plus live proof for a deleted cwd with a state-dir `.env`.

## Reproduction Assessment

Yes, source-level. Current `runCli()` reaches `shouldLoadCliDotEnv()` for non-help commands, and after the PR's catch a state-dir `.env` still sends execution into `loadCliDotEnv()`, which directly calls `process.cwd()`.

## Solution Assessment

No. Guarding only the preflight predicate is too narrow; the maintainable fix needs the same safe-cwd decision carried through the loader path that actually reads dotenv files.

- **[P1] Guard the dotenv loader path too:** `src/cli/run-main.ts:303-306`
  - body: When `process.cwd()` throws and a state-dir `.env` exists, this catch falls through to the state-dir predicate and `runCli()` then calls `loadCliDotEnv()`.
