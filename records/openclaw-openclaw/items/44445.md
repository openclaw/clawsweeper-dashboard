## Summary

Keep open. Current main still does not sanitize user text in the session-memory transcript extraction path, but this PR is not merge-ready because it also changes scheduled reset hook semantics and introduces concrete fidelity regressions around slash-prefixed and message-id-looking user content.

## What This Changes

This PR strips leading inbound metadata from user turns before session-memory stores them, extends inbound metadata stripping for Feishu message-id envelope lines, adds scheduled reset hook dispatch, and adds regression tests.

## Best Possible Solution

Land a narrow session-memory sanitizer change that stays in the transcript extraction layer, reuses the existing inbound metadata/envelope/message-id cleanup semantics, and adds focused tests for Feishu blocks, timestamp/envelope variants, slash-prefixed real content, and user-authored message-id-looking lines. Keep daily/idle reset hook dispatch in #61675 or a dedicated lifecycle PR.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a JSONL user message containing Feishu-style inbound metadata or a standalone `[message_id: ...]` line will be read by `getRecentSessionContent()` and appended to memory without sanitizer cleanup on current main.

## Solution Assessment

No. Sanitizing session-memory user text is the right direction, but this PR is not the best form because it bundles scheduled reset hook dispatch and its cleanup can drop valid user-authored content; the safer solution is a narrower sanitizer-only patch plus targeted regression tests.

  - body: When `scheduledResetTriggered` is true, this emits `command:new` hooks for the first ordinary message after a stale daily reset even though no `/new` or `/reset` command was sent, and it bypasses the authorized-sender guard used by the manual path. That can run reset automations and plugin hooks on normal traffic; keep the lazy reset hook work in the dedicated #61675 path instead of bundling it here.
  - confidence: 0.88
- **[P2] Preserve slash-prefixed user text after cleanup:** `src/hooks/bundled/session-memory/handler.ts:83-84`
  - body: The slash-command filter now runs after metadata stripping, so a legitimate channel message whose real content starts with `/` is dropped from session-memory. For example, a Feishu user asking about `/var/log` would be kept before this change because the stored text started with metadata, but would now disappear from the memory artifact.
  - confidence: 0.86
- **[P2] Preserve user-authored message_id-looking lines:** `src/auto-reply/reply/strip-inbound-meta.ts:240-245`
  - body: This loop removes any standalone `[message_id: ...]` line immediately after the metadata blocks, even when that line is the user's actual first line. The cleanup should only remove a known transport hint, such as one matched to injected metadata, rather than treating all message_id-looking user text as metadata.
  - confidence: 0.8
