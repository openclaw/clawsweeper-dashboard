## Summary

Keep open. The PR targets a real session-memory contamination problem, but the current diff has blocking correctness and security concerns around synthetic scheduled-reset hook dispatch, slash-prefixed user text loss, and over-stripping `[message_id: ...]` user lines; external real behavior proof is also missing.

## What This Changes

The PR strips inbound channel metadata from user turns before session-memory writes and adds scheduled/daily reset hook emission plus regression tests.

## Best Possible Solution

Move session-memory sanitization into the current transcript formatting helper, preserve legitimate user text, add durable untrusted historical framing or the accepted reset policy, and handle automatic daily/idle rollovers with explicit lifecycle semantics rather than synthetic `command:new`.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the PR's scheduled rollover path calls reset command hooks without a command, and the session-memory formatter would filter slash-prefixed real user text after stripping metadata. I did not run a live Feishu session in this read-only review.

## Solution Assessment

No. The metadata stripping direction is useful, but this implementation mixes it with synthetic command hook dispatch and loses valid user content; the safer fix is transcript-local sanitization plus an explicit reset/memory trust-boundary decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not fire command hooks for automatic rollovers:** `src/auto-reply/reply/get-reply.ts:356-357`
  - body: When `scheduledResetTriggered` is true, this emits `action: "new"` on the first ordinary post-expiry message and skips the existing sender authorization guard. `emitResetCommandHooks` can route hook replies and run `before_reset` plugins, so automatic rollovers can trigger reset automations even though no `/new` or `/reset` command was authorized.
  - confidence: 0.9
- **[P2] Keep stripped slash-prefixed user text in memory:** `src/hooks/bundled/session-memory/handler.ts:83-84`
  - body: The PR now strips inbound metadata before checking `!text.startsWith("/")`. For channels that prepend metadata, a real user turn such as `/var/log is missing` or a quoted slash command changes from being preserved to being dropped from session-memory output.
  - confidence: 0.83
- **[P2] Preserve user-authored message_id lines:** `src/auto-reply/reply/strip-inbound-meta.ts:236`
  - body: This loop removes any standalone `[message_id: ...]` line after OpenClaw metadata blocks. The inbound metadata contract explicitly says user text must not be treated as metadata just because it resembles `[message_id: ...]`, so a real first line with that text would be lost.
  - confidence: 0.82
