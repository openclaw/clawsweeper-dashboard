## Summary

Keep open: the sanitizer goal is valid and still not implemented on current main, but this PR is not ready to merge because it bundles overlapping scheduled-reset hook behavior, introduces sanitizer regressions, and still lacks external contributor real behavior proof.

## What This Changes

The branch sanitizes session-memory user text with `stripLeadingInboundMetadata`, extends message-id stripping, adds scheduled reset hook dispatch, and adds regression tests.

## Best Possible Solution

Land a focused sanitizer in `src/hooks/bundled/session-memory/transcript.ts` using the established user-display cleanup semantics, and leave daily/idle reset hook dispatch to https://github.com/openclaw/openclaw/pull/61675 or a dedicated lifecycle PR.

## Reproduction Assessment

Yes, at source level. A JSONL user message containing Feishu-style inbound metadata or a standalone message-id line is read by `getRecentSessionContent()` and appended to memory without sanitizer cleanup on current main.

## Solution Assessment

No. Sanitizing session-memory user text is the right direction, but this PR is not the best form because it bundles scheduled reset hook dispatch and introduces cleanup regressions; a focused sanitizer patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep automatic rollovers out of /new hook dispatch:** `src/auto-reply/reply/get-reply.ts:356-368`
  - body: When `scheduledResetTriggered` is true, this emits `command:new` hooks for the first ordinary message after a stale daily reset even though no `/new` or `/reset` command was sent, and it bypasses the authorized-sender guard used by the manual path. Keep lazy reset hook work in the dedicated daily/idle reset path instead of bundling it here.
  - confidence: 0.88
- **[P2] Preserve slash-prefixed user text after cleanup:** `src/hooks/bundled/session-memory/handler.ts:83-84`
  - body: The slash-command filter now runs after metadata stripping, so a legitimate channel message whose real content starts with `/` is dropped from session memory. For example, a Feishu user asking about `/var/log` would be kept before this change because the stored text started with metadata, but now disappears from the memory artifact.
  - confidence: 0.86
- **[P2] Preserve user-authored message_id-looking lines:** `src/auto-reply/reply/strip-inbound-meta.ts:240-245`
  - body: This loop removes any standalone `[message_id: ...]` line immediately after metadata blocks, even when that line is the user's actual first line. The cleanup should only remove known transport hints rather than treating all message-id-looking user text as metadata.
  - confidence: 0.8
- **[P2] Use the full envelope sanitizer for timestamped turns:** `src/hooks/bundled/session-memory/handler.ts:83-84`
  - body: `stripLeadingInboundMetadata()` only strips when the first non-empty line is an inbound metadata sentinel, so timestamp-prefixed transcript content can still persist the metadata blocks. Reuse the same runtime/inbound/envelope/message-id cleanup semantics as the display sanitizer instead of this narrower helper.
  - confidence: 0.74
