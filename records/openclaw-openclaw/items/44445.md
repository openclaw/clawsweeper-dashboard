## What This Changes

This PR changes session-memory transcript extraction to strip inbound channel metadata from user turns and adds a scheduled-reset flag that dispatches reset hooks for stale daily rollovers.

## Best Possible Solution

Close this branch and consolidate the remaining work into the newer automatic-rollover hook PRs plus the canonical session-memory trust-boundary/sanitizer track, using a single policy for metadata stripping, reset semantics, and durable untrusted framing.

## Reproduction Assessment

Yes. Source inspection shows current main still extracts raw transcript text and writes it directly into session-memory Markdown; the PR diff also exposes high-confidence introduced regressions around synthetic command:new dispatch, message_id over-stripping, and slash filtering.

## Solution Assessment

No. The PR mixes a useful metadata cleanup with a superseded scheduled-reset design; the safer solution is to land automatic rollover hooks through the newer stale-session-path PRs and handle transcript sanitization/framing under the canonical session-memory issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Separate automatic rollover hooks from command:new dispatch:** `src/auto-reply/reply/get-reply.ts:360`
  - body: When `scheduledResetTriggered` is true, this branch emits `command:new` even though no user command was sent, faning out normal command:new hooks and before_reset plugins on an ordinary post-expiry message while bypassing the manual sender-authorization guard. Automatic rollovers need their own explicit hook semantics instead of pretending a `/new` command happened.
  - confidence: 0.9
- **[P2] Preserve user-authored message_id lines:** `src/auto-reply/reply/strip-inbound-meta.ts:242`
  - body: The new loop drops a standalone `[message_id: ...]` line after inbound metadata even when that line is the first real user-authored line. That regresses transcript fidelity and conflicts with the existing prompt invariant that user text should not become metadata merely because it resembles an envelope tag.
  - confidence: 0.86
- **[P2] Do not drop slash-prefixed user content after stripping:** `src/hooks/bundled/session-memory/handler.ts:85`
  - body: The PR strips metadata before the existing `!text.startsWith("/")` filter, so a legitimate channel message whose real body starts with `/` is now omitted from session-memory. That drops non-command content such as paths, command examples, or questions about `/new` for metadata-prefixed channels.
  - confidence: 0.88
- **[P2] Handle timestamp-prefixed inbound metadata:** `src/hooks/bundled/session-memory/handler.ts:85`
  - body: `stripLeadingInboundMetadata` only strips when the first non-empty line is a metadata sentinel. If `BodyForAgent` has an injected timestamp before the metadata block, this cleanup returns the full metadata-prefixed text and leaves the reported pollution in memory files.
  - confidence: 0.82
