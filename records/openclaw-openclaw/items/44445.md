## Summary

Keep open. Current main still writes raw user transcript text into session-memory artifacts, so the core bug is real, but this PR is not merge-ready because it mixes the metadata cleanup with reset-hook lifecycle behavior and has blocking fidelity regressions in the cleanup path.

## What This Changes

The PR sanitizes session-memory user turns with inbound metadata stripping, extends inbound metadata cleanup for Feishu message-id envelope lines, adds scheduled reset hook dispatch, and adds regression tests around those paths.

## Best Possible Solution

Keep the fix focused in the session-memory transcript extraction layer: reuse or factor the existing display sanitizer so user turns have inbound metadata, timestamp prefixes, channel envelopes, and injected message-id hints removed without dropping legitimate slash-prefixed or message-id-looking user text. Leave daily/idle reset hook semantics to #61675 and broader historical-memory trust framing to #68751.

## Reproduction Assessment

Yes. A JSONL transcript containing a user message with Feishu-style `Conversation info`, `Sender`, and `[message_id: ...]` prefixes will flow through `getRecentSessionContent()` on current main and be appended raw to the memory markdown artifact.

## Solution Assessment

No. Sanitizing user text before session-memory persistence is the right direction, but this PR is not the best solution because it also changes scheduled reset hook dispatch and its cleanup path can drop valid user-authored content; a narrower sanitizer plus focused tests is safer.

  - body: When `scheduledResetTriggered` is true, this emits `command:new` hooks for the first ordinary message after a stale daily reset even though no `/new` or `/reset` command was sent, and it bypasses the authorized-sender guard used by the manual path.   - body: The slash-command filter now runs after metadata stripping, so a legitimate channel message whose real content starts with `/` is dropped from session-memory. For example, a Feishu user asking about `/var/log` would now disappear from the memory artifact.
  - confidence: 0.86

  - body: This loop removes any standalone `[message_id: ...]` line immediately after metadata blocks, even when that line is the user's actual first line. The cleanup should only remove a known transport hint, not every user-authored line that resembles one.
  - confidence: 0.8
