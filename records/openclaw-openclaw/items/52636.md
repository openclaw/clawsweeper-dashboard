## Summary

Keep open for maintainer/security review. Current main already covers the config.openFile command-injection slice, but the PR is broad, unmergeable against current main, and still has concrete correctness and security-policy blockers across loopback auth throttling, browser evaluate hardening, webhook verification, and Shield runtime state.

## What This Changes

The PR adds a new `src/security/shield` runtime subsystem and changes gateway config file opening, auth rate limiting, default HTTP security headers, and browser Playwright evaluate filtering.

## Best Possible Solution

Split the contribution into focused security PRs. Keep current main's `config.openFile` hardening as the canonical solution for that slice, then review loopback throttling, browser evaluate restrictions under `extensions/browser`, global response headers, and any Shield-style runtime subsystem separately with secops ownership, provenance, threat-model rationale, and targeted regression tests.

## Reproduction Assessment

Yes. The blockers are reproducible by static inspection of the PR head and current main: the loopback lockout path cannot reach the claimed 10x ceiling, malformed webhook signatures can throw, the browser change targets a removed path, and Shield state can remain stale after recovery or metric expiry.

## Solution Assessment

No. This is not the best way to land the work because it mixes shipped hardening, stale-path changes, new runtime security policy, and third-party-attributed code into one broad branch; the maintainable solution is a split, rebased, secops-reviewed set of narrow changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce the loopback cap before applying normal lockout:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still sets `lockedUntil` after the normal `maxAttempts`, but `check()` returns through the relaxed loopback exemption before honoring that lockout while attempts are below `maxAttempts * 10`. Once locked, later failures stop being recorded, so the intended 10x loopback cap is never reached.
  - confidence: 0.9
- **[P2] Move evaluate filtering to the active browser plugin path:** `src/browser/pw-tools-core.interactions.ts:305-306`
  - body: Current main no longer has `src/browser`; the active `evaluateViaPlaywright()` implementation lives under `extensions/browser/src/browser`. Adding the blocked-pattern filter here either conflicts on rebase or leaves the active browser evaluate surface unchanged.
  - confidence: 0.88
- **[P2] Decode signatures before timing-safe comparison:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature decodes to a shorter buffer and throws instead of returning `false`, which turns a bad signature into an exception path.
  - confidence: 0.86
- **[P2] Refresh escalation after circuit recovery:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit state but never recomputes `this.escalation`. If an open circuit closes, `getStatus()` and `getSummary()` can keep reporting `PARTIAL_PAUSE` or `FULL_PAUSE` until another circuit opens.
  - confidence: 0.84
- **[P2] Drop stale health scores when metrics expire:** `src/security/shield/gateway-shield.ts:263-265`
  - body: `computeHealthScores()` filters to recent metrics but only updates functions present in the new aggregate. Functions with no recent metrics keep prior `CRITICAL` or `CIRCUIT_OPEN` health, so `shouldThrottle()` can keep throttling recovered or idle functions.
  - confidence: 0.82

## Security Review

  - body: The proposed 10x loopback cap is undercut by the normal `maxAttempts` lockout path, which stops recording further failures while `check()` still allows loopback requests below the 10x ceiling.
  - confidence: 0.9

  - confidence: 0.88
- **[medium] Malformed webhook signatures can throw:** `src/security/shield/webhook-dispatch.ts:63`
  - body: Signature verification can throw for same-length malformed hex input instead of failing closed with `false`, which is an unsafe exception path in security verification code.
  - confidence: 0.86

  - body: Escalation and health-score state can persist after recovery or after metrics age out, which is risky for a subsystem that may throttle or pause gateway behavior.
  - confidence: 0.82

  - body: The new Shield files state they are adapted from the Kairos Shield Protocol and by Kairos Lab, but the PR does not add local license/provenance evidence for the adapted code.
