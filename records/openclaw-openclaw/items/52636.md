## Summary

Keep open. Current main already covers the `config.openFile` hardening slice, but this PR still combines broad security-policy work with a new Shield runtime subsystem, an obsolete browser path edit, and concrete correctness/security blockers.

## What This Changes

The PR adds a new `src/security/shield` runtime subsystem and modifies gateway config opening, auth rate limiting, default HTTP security headers, and browser Playwright evaluate filtering.

## Best Possible Solution

Split the work into focused, separately reviewable security PRs: keep the shipped `config.openFile` hardening as canonical, then review loopback throttling, browser evaluate policy, headers, and any Shield-style runtime layer with secops ownership and targeted tests.

## Reproduction Assessment

Yes. The main blockers are reproducible by static inspection of PR head and current main: the loopback cap is bypassed by the lockout path, the evaluate filter targets a removed path, malformed webhook signatures can throw, and Shield state can remain stale.

## Solution Assessment

No. This PR is not the best merge shape because it mixes a shipped hardening fix, stale-path edits, new runtime security policy, and a large new subsystem; narrower secops-reviewed patches are safer and easier to validate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce the loopback 10x threshold in lockout state:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still sets the normal `maxAttempts` lockout for loopback clients, but `check()` returns from `isExempt()` before honoring `lockedUntil` while the count is below `maxAttempts * 10`. Once locked, later failures stop being recorded, so the counter never reaches the intended 10x ceiling and localhost failures remain allowed.
  - confidence: 0.9
- **[P2] Move evaluate filtering to the active browser plugin path:** `src/browser/pw-tools-core.interactions.ts:305-306`
  - body: Current main no longer has `src/browser`; the active `evaluateViaPlaywright()` implementation lives under `extensions/browser/src/browser`. Adding the blocked-pattern filter here either conflicts on rebase or leaves the active browser evaluate surface unchanged.
  - confidence: 0.88
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature decodes to a shorter buffer and can throw instead of returning `false`, turning a bad signature into an exception path.
  - confidence: 0.86
- **[P2] Recompute escalation after circuit recovery:** `src/security/shield/gateway-shield.ts:175`
  - body: `recordSuccess()` updates the circuit state but never recomputes `this.escalation`. If an open circuit closes, `getStatus()` and `getSummary()` can keep reporting an escalated pause until another circuit opens.
  - confidence: 0.84
- **[P2] Expire health scores when metrics age out:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters to the last five minutes but only updates functions present in the new aggregate. Functions with no recent metrics keep previous `CRITICAL` or `CIRCUIT_OPEN` health, so `shouldThrottle()` can continue throttling idle or recovered functions from stale state.
  - confidence: 0.82
- **[P2] Collapse health buckets before storing scores:** `src/security/shield/gateway-shield.ts:287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this loop stores them in `healthScores` keyed only by function name. If metric insertion order is not chronological, an older minute can overwrite a newer health score and make throttling depend on arrival order.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes gateway/browser security policy and adds a new runtime protection subsystem while concrete bypass, exception, and stale-state concerns remain.

Concerns:

- **[high] Loopback throttling remains bypassable:** `src/gateway/auth-rate-limit.ts:214`
  - body: The proposed 10x loopback cap is undercut by the normal `maxAttempts` lockout path, which stops recording further failures while `check()` still allows loopback requests below the 10x ceiling.
  - confidence: 0.9
- **[high] Browser evaluate hardening targets an obsolete path:** `src/browser/pw-tools-core.interactions.ts:305`
  - body: The PR adds filtering under `src/browser`, but current main's active browser implementation is under `extensions/browser`, so the claimed evaluate mitigation is not applied to the active surface until rebased and moved.
  - confidence: 0.88
- **[medium] Malformed webhook signatures can throw:** `src/security/shield/webhook-dispatch.ts:63`
  - body: Signature verification can throw for same-length malformed hex input instead of failing closed with `false`, which is an unsafe exception path in security verification code.
  - confidence: 0.86
- **[medium] Shield runtime state can remain stale:** `src/security/shield/gateway-shield.ts:175`
  - body: Escalation and health-score state can persist after recovery or after metrics age out, which is risky for a subsystem intended to drive runtime throttling and pauses.
  - confidence: 0.84
