## Summary

Keep open. Current main only covers the config file opener hardening slice, while the PR still proposes broader gateway security policy, browser evaluate filtering, headers, and a new Shield runtime subsystem with merge-blocking correctness and security concerns.

## What This Changes

This PR adds a `src/security/shield` runtime subsystem and changes gateway config file opening, auth rate limiting, default HTTP security headers, and browser Playwright evaluate filtering.

## Best Possible Solution

Split the work into focused, separately reviewable security changes: keep the shipped `config.openFile` hardening as canonical, then evaluate loopback throttling, browser evaluation restrictions, headers, and any Shield-style runtime subsystem against OpenClaw's trust model with secops review and targeted tests.

## Reproduction Assessment

Yes. The blockers are reproducible by static inspection of PR head and current main: loopback lockout short-circuits the intended 10x cap, malformed webhook signatures can throw, the browser change targets a removed path, and Shield state can remain stale.

## Solution Assessment

No. This is not the best way to land the work because it combines a shipped hardening fix, stale-path edits, new runtime security policy, and third-party-attributed code in one broad branch; narrower secops-reviewed PRs are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce the loopback 10x threshold in the lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still sets the normal `maxAttempts` lockout for loopback clients, but `check()` returns from `isExempt()` before honoring `lockedUntil` while the count is below `maxAttempts * 10`. Once the normal lockout is set, further `recordFailure()` calls stop appending attempts, so the counter never reaches the intended 10x ceiling and localhost failures remain allowed indefinitely.
  - confidence: 0.9
- **[P2] Move evaluate filtering to the active browser plugin path:** `src/browser/pw-tools-core.interactions.ts:305-306`
  - body: Current main no longer has `src/browser`; the active `evaluateViaPlaywright()` implementation lives under `extensions/browser/src/browser`. Adding the blocked-pattern filter here either conflicts on rebase or leaves the active browser evaluate surface unchanged.
  - confidence: 0.88
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature decodes to a shorter buffer and makes verification throw instead of returning `false`, which can turn a bad signature into an exception path for callers.
  - confidence: 0.86
- **[P2] Refresh escalation state when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit state but never recomputes `this.escalation`. If a previously open circuit closes, `getStatus()` and `getSummary()` can keep reporting `PARTIAL_PAUSE` or `FULL_PAUSE` until another circuit opens.
  - confidence: 0.84
- **[P2] Expire stale health scores without recent metrics:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters metrics to the last five minutes but only updates functions present in the new aggregate. Functions with no recent metrics keep their previous `CRITICAL` or `CIRCUIT_OPEN` health score, so `shouldThrottle()` can continue throttling idle or recovered functions from stale state.
  - confidence: 0.82
- **[P2] Collapse health metrics by function before storing scores:** `src/security/shield/gateway-shield.ts:287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this loop writes them into `healthScores` keyed only by function name. If metrics are not already inserted in chronological bucket order, an older minute can overwrite a newer health score and make throttling decisions depend on arrival order.
  - confidence: 0.78

  - body: The proposed 10x loopback cap is undercut by the normal `maxAttempts` lockout path, which stops recording further failures while `check()` still allows loopback requests below the 10x ceiling.
  - confidence: 0.9

  - body: The PR adds filtering under `src/browser`, but current main's active browser implementation is under `extensions/browser`, so the claimed evaluate mitigation is not applied to the active surface until rebased and moved.
  - confidence: 0.88
- **[medium] Malformed webhook signatures can throw:** `src/security/shield/webhook-dispatch.ts:63`
  - body: Signature verification can throw for same-length malformed hex input instead of failing closed with `false`, which is an unsafe exception path in security verification code.
  - confidence: 0.86
- **[medium] Shield runtime state can remain stale:** `src/security/shield/gateway-shield.ts:174`
  - body: Escalation and health-score state can persist after recovery or after metrics age out, which is risky for a subsystem that may throttle or pause gateway behavior.
  - confidence: 0.82
