Keep open: current main only covers the config.openFile hardening slice, while this PR still proposes broad security policy and a new Shield runtime subsystem with concrete branch-level blockers in loopback throttling, browser evaluate hardening, webhook verification, and Shield state freshness.

## What This Changes

The PR adds a new `src/security/shield` runtime subsystem and modifies gateway config file opening, auth loopback rate limiting, default HTTP security headers, and Playwright evaluate filtering.

## Best Possible Solution

Split the work into focused security PRs: keep current main's shipped config.openFile hardening, then separately design and review loopback throttling, browser evaluate policy, global headers, and any Shield-style subsystem with secops ownership, provenance, config contract, tests, docs, and changelog.

## Reproduction Assessment

Yes. Static inspection of the PR diff against current main reproduces the blockers: the loopback counter cannot reach the advertised 10x cap, the evaluate filter targets an obsolete path, malformed webhook signatures can throw, and Shield status can remain stale after recovery or metric expiry.

## Solution Assessment

No. This is not the best way to land the work because it combines a shipped hardening fix, stale-path edits, new runtime security policy, and externally attributed code in one broad branch; narrower secops-reviewed changes are safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce the loopback cap before the normal lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still applies the normal `maxAttempts` lockout to loopback entries, then stops recording while locked. Since `check()` returns from `isExempt()` before honoring `lockedUntil` until the count reaches `maxAttempts * 10`, the counter never reaches the advertised relaxed cap and localhost failures can keep proceeding.
  - confidence: 0.9
- **[P2] Move evaluate filtering to the active browser plugin path:** `src/browser/pw-tools-core.interactions.ts:305-306`
  - body: Current main no longer has this `src/browser` implementation; active `evaluateViaPlaywright()` lives under `extensions/browser/src/browser`. Adding the blocked-pattern filter here either conflicts on rebase or leaves the active browser evaluate surface unchanged.
  - confidence: 0.88
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature decodes to a shorter buffer and can throw instead of returning `false`, turning bad input into an exception path.
  - confidence: 0.86
- **[P2] Refresh escalation state when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit but never recomputes `this.escalation`. After an open circuit closes, `getStatus()` and `getSummary()` can continue reporting `PARTIAL_PAUSE` or `FULL_PAUSE` until a later failure opens another circuit.
  - confidence: 0.84
- **[P2] Drop stale health scores when metrics leave the window:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters to recent metrics but only updates functions present in the new aggregate. Functions with no recent metrics keep prior `CRITICAL` or `CIRCUIT_OPEN` scores, so `shouldThrottle()` can continue throttling idle or recovered functions from stale state.
  - confidence: 0.82
- **[P2] Collapse health buckets before storing function scores:** `src/security/shield/gateway-shield.ts:287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this loop writes each bucket into `healthScores` keyed only by function name. If insertion order is not chronological, an older minute can overwrite a newer score and make throttling depend on arrival order.
  - confidence: 0.78
