## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce the loopback cap through the lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: The new loopback policy still trips the normal `maxAttempts` lockout and then stops recording failures while `lockedUntil` is active. Because `check()` returns from `isExempt()` before honoring that lockout until attempts reach `maxAttempts * 10`, localhost failures stay allowed and the intended 10x cap never becomes enforceable.
  - confidence: 0.9
- **[P2] Wire the Shield runtime into real gateway paths:** `src/security/shield/gateway-shield.ts:78`
  - body: `GatewayShield` is only exported/tested; no gateway HTTP, WebSocket, auth, or plugin runtime path instantiates it or calls its checks. That means the circuit breaker, session monitor, health scoring, escalation, and WS validation modules have no production effect despite being presented as runtime protection.
  - confidence: 0.86
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature decodes to a shorter buffer and throws `RangeError` instead of returning `false`, which can turn bad input into an exception path.
  - confidence: 0.88
- **[P2] Refresh escalation when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit map but never recomputes `this.escalation`. Once a circuit closes or leaves an open state, `getStatus()` and `getSummary()` can keep reporting a previous pause/escalation state until another failure opens a circuit.
  - confidence: 0.84
- **[P2] Expire stale health scores without recent metrics:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters metrics to the last five minutes but only updates functions present in the new aggregate. Functions that drop out of the window keep old `CRITICAL` or `CIRCUIT_OPEN` scores, so `shouldThrottle()` can keep throttling recovered or idle functions.
  - confidence: 0.82
- **[P2] Collapse health buckets before storing scores:** `src/security/shield/gateway-shield.ts:287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this loop stores each bucket under only the function name. If metrics arrive out of chronological order, an older minute can overwrite a newer score and make throttling depend on insertion order.
  - confidence: 0.78
