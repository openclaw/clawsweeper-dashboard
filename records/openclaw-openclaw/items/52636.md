## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce the loopback 10x threshold in the lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still sets `lockedUntil` at normal `maxAttempts`. Once that lock is active, further loopback failures stop being recorded while `check()` keeps allowing loopback until `attempts.length` reaches `maxAttempts * 10`, so the intended localhost cap is never reached.
  - confidence: 0.9
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: A same-length non-hex signature passes the string-length check but decodes to a shorter buffer, so `timingSafeEqual()` throws instead of returning `false`. Bad signatures should stay on the normal rejection path.
  - confidence: 0.86
- **[P2] Refresh escalation state when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` writes the recovered circuit but never recomputes `this.escalation`. `getStatus()` and `getSummary()` can keep reporting a pause state after the last open circuit has closed.
  - confidence: 0.84
- **[P2] Drop stale health scores when metrics leave the window:** `src/security/shield/gateway-shield.ts:263`
  - body: `computeHealthScores()` filters to recent metrics, but functions missing from the new aggregate keep their old `CRITICAL` or `CIRCUIT_OPEN` score. That lets stale health state continue throttling idle or recovered functions.
  - confidence: 0.82
- **[P2] Collapse health metrics by function before storing scores:** `src/security/shield/gateway-shield.ts:287`
  - body: The aggregate is keyed by `function|minute`, but the loop stores scores by function name. Multiple minute buckets for one function can overwrite each other in metric arrival order, making throttling depend on stale bucket order.
  - confidence: 0.78
