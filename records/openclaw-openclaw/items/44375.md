## Label Justifications

- `P1`: A normal config edit can silently reroute all unmatched traffic away from the user's established main agent and affect active real workflows.
- `impact:message-loss`: Unmatched DMs, group topics, heartbeats, or cron messages can be delivered to the new ACP default agent instead of the intended main agent.
- `impact:session-state`: The fallback agent id changes session keys and can strand existing `main` history, memory index, and auth-related state.
- `impact:security`: Private or scheduled traffic can enter an unintended ACP runtime and workspace after implicit default-agent displacement.

## Summary

Keep open. Current main still uses the documented first-entry fallback for a non-empty `agents.list`, unmatched traffic still routes through that configured default, and raw `openclaw config set agents.list ...` still has no specific warning for displacing implicit `main`; the related fallback-routing issue is adjacent but does not cover this first-time default displacement path.

## What This Changes

The issue requests safer handling when adding an ACP agent to `agents.list` would replace the implicit `main` default for unmatched traffic.

## Best Possible Solution

Add a compatible warning, doctor diagnostic, or guarded config-set diagnostic when `agents.list` omits both `main` and `default: true` while existing `main` state is present; reserve fallback-contract changes for explicit maintainer approval and migration tests.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: set `agents.list` to a single non-main ACP entry without `default: true`, then unmatched routing resolves through `resolveDefaultAgentId` to that entry after binding tiers miss.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. A warning or diagnostic is the narrowest compatible fix, while auto-including `main` or requiring explicit defaults changes documented routing semantics and needs explicit owner approval.
