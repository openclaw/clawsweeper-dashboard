## Label Justifications

- `P1`: A common ACP setup can silently reroute real user traffic, heartbeats, and cron jobs to the wrong agent.
- `impact:session-state`: Existing `main` sessions, memory, and transcript continuity can be orphaned when the default agent id changes silently.
- `impact:message-loss`: Unmatched DMs, topics, heartbeats, and crons can be delivered to an unintended agent instead of the prior main route.
- `impact:security`: Private channel traffic may reach an unintended ACP runtime or persona when fallback routing changes without confirmation.

## Summary

Keep open: current main and the latest release still use the first `agents.list` entry as the default when no `default: true` entry exists, so the reported silent routing takeover remains source-reproducible. The adjacent implicit-main binding issue and open PR are related but narrower and do not settle the default-agent policy for unmatched traffic.

## What This Changes

The issue requests safer handling when adding the first ACP-bound agent to `agents.list` would silently move fallback routing from implicit `main` to that ACP agent.

## Best Possible Solution

Make the default-agent transition explicit and safe across config writes, route resolution, validation, doctor, docs, and tests without creating hidden or conflicting fallback policies.

## Reproduction Assessment

Yes. Source-level: `resolveDefaultAgentId` chooses the first listed agent when no `default: true` exists, and unmatched routes call that helper; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Unclear. Warning, auto-preserving `main`, and requiring explicit defaults are all plausible repairs, but the routing fallback/security tradeoff needs maintainer product direction before an automated fix.
