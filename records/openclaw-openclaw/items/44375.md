## Label Justifications

- `P1`: A normal config edit can silently reroute all unmatched traffic away from an established main agent and affect active real workflows.
- `impact:session-state`: The default agent id changes session keys and can strand existing `main` history, memory index, and auth-related state.
- `impact:security`: Private or scheduled traffic can enter an unintended ACP runtime and workspace after implicit default-agent displacement.
- `impact:message-loss`: Unmatched DMs, topics, heartbeats, or cron messages can be delivered to the new ACP default agent instead of the intended main agent.

## Summary

Keep open. Current main still follows the reported path: a non-empty `agents.list` without `default: true` makes the first entry the configured default, unmatched routing uses that agent, and raw `config set agents.list` still emits only a generic restart message; changing this needs an explicit routing/security compatibility decision.

## What This Changes

The issue asks for safer handling when adding an ACP agent to `agents.list` would displace the implicit `main` agent as the default for unmatched traffic.

## Best Possible Solution

Add a compatible warning, doctor diagnostic, or guarded config-set diagnostic when `agents.list` omits both `main` and `default: true` while existing `main` state is present; reserve fallback-contract changes for explicit maintainer approval and migration tests.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: set `agents.list` to a single non-main ACP entry without `default: true`, then unmatched routing resolves through `resolveDefaultAgentId` to that entry after binding tiers miss.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. A warning or diagnostic is the narrowest compatible path, while auto-including `main` or requiring explicit defaults changes documented routing semantics.
