## Label Justifications

- `P1`: The source-proven behavior can reroute unmatched real traffic and session state to a newly listed ACP agent without an explicit default.
- `impact:session-state`: The reported path changes session keys and default workspace ownership from `main` to the first configured agent.
- `impact:message-loss`: The issue is about channel traffic being misrouted to the wrong agent instead of staying with the implicit main agent.
- `impact:security`: Unmatched private messages can route to an unintended ACP-backed agent or workspace when the operator expected only one bound topic to use it.

## Summary

Keep open: current main and the latest release still use the first `agents.list` entry as the default for unmatched routing and bare session keys when no entry is marked default, and raw `config set agents.list ...` still has no special warning or guard for displacing implicit `main`.

## What This Changes

The issue asks for safer handling when adding an ACP agent to `agents.list` would silently replace the implicit `main` agent as the default route.

## Best Possible Solution

Add a maintainer-approved safety path around first-time or non-default `agents.list` writes, such as warning or requiring an explicit default when existing implicit `main` state would be displaced, while preserving a valid explicit-main binding path.

## Reproduction Assessment

Yes. Source inspection on current main shows that a populated `agents.list` with no explicit default makes the first list entry the default for unmatched routes and bare session keys; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear as a final policy. Warning or blocking unsafe first-time raw `agents.list` writes looks safer than silently changing runtime fallback, but maintainers need to choose the compatibility behavior.
