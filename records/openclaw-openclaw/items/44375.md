## Summary

Keep open: current main still implements the documented first-entry fallback, but the report identifies a real routing-safety foot-gun that has not been addressed with warnings or guardrails.

## What This Changes

Request to warn or otherwise protect users when adding an ACP agent to `agents.list` without an explicit default displaces the implicit `main` agent for unmatched traffic.

## Best Possible Solution

Add a shared default-agent guardrail on config writes and align ACP/multi-agent examples so ACP-only entries do not silently become the fallback; any resolver-semantic change should be an explicit compatibility decision.

## Reproduction Assessment

Yes from source inspection. A config with only `agents.list: [{id:"codex-cubby", runtime:{type:"acp"}}]` and no matching binding will make `resolveDefaultAgentId` return `codex-cubby`, and `resolveAgentRoute` will use that for unmatched traffic.

## Solution Assessment

Unclear. A warning on config writes is the narrowest low-compatibility-risk direction, while auto-adding `main` or requiring explicit defaults would change documented routing behavior and needs maintainer choice.
