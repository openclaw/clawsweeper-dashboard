## Label Justifications

- `P1`: A normal config edit can reroute all unbound traffic and split session continuity for real multi-agent users.
- `impact:message-loss`: Messages not matching the intended topic binding can be delivered to the new ACP default agent instead of the previous main agent.
- `impact:session-state`: The fallback agent id changes session keys and can leave existing `main` history, memory, and auth state unused.
- `impact:security`: Private or scheduled traffic can enter an unintended ACP runtime and workspace after the default displacement.

## Summary

Keep open. Current `main` still matches the reported source path: a non-empty `agents.list` without `default: true` makes the first list entry the configured default, unmatched routing falls through to that agent, and raw `config set agents.list` still prints only the generic restart message. The related open routing issue covers stale or missing binding targets, not this implicit-main displacement path.

## What This Changes

The issue asks OpenClaw to prevent or warn about a raw `agents.list` change that makes a newly added ACP agent the default route for unbound traffic instead of preserving the implicit `main` agent.

## Best Possible Solution

Add a compatible warning or doctor/startup diagnostic when `agents.list` is populated without `main` or an explicit default while `main` state exists; reserve fallback-contract changes for explicit maintainer approval.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: set `agents.list` to one non-main ACP entry without `default: true`; `resolveDefaultAgentId` selects that entry and `resolveAgentRoute` uses it after binding misses.

## Solution Assessment

Unclear until maintainers choose the contract. A warning or diagnostic is the narrow compatible fix, while auto-including `main` or requiring explicit defaults changes documented routing behavior.
