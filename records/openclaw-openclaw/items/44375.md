## Label Justifications

- `P1`: Silent default-agent displacement can reroute live channel, heartbeat, and cron traffic away from the user's expected primary agent.
- `impact:session-state`: Changing the fallback agent changes agent-scoped session keys and can strand existing `main` history, memory, and auth state.
- `impact:security`: Unmatched traffic can be delivered into an unintended ACP runtime or workspace with different tools, auth, and sandbox policy.
- `impact:message-loss`: Channel traffic that lacks a matching binding can be delivered to a different agent than the user intended after adding one ACP agent.

## Summary

Keep open: current main and v2026.6.10 still route unmatched traffic to `agents.list[].default` or the first listed agent, so adding one ACP agent can still displace the implicit `main` agent. The related open PR preserves explicit `main` bindings, but it does not resolve this default-selection policy issue.

## What This Changes

The issue asks OpenClaw to warn, preserve `main`, or require an explicit default when populating `agents.list` would silently change the default agent.

## Best Possible Solution

Maintainers should choose the default-routing compatibility policy, then update config write/doctor warnings or routing behavior plus docs and focused regression coverage consistently.

## Reproduction Assessment

Yes, source-level: a non-empty `agents.list` with no `default: true` resolves the first entry as the default, and unmatched route fallback uses that agent. I did not run a live gateway because this review is read-only.

## Solution Assessment

Unclear as a final implementation: warning, auto-preserving `main`, and requiring an explicit default each change documented routing semantics differently. The right fix needs a maintainer compatibility choice before code is changed.
