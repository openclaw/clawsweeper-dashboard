## Summary

Keep this open. Current main still has the reported config/routing foot-gun: once `agents.list` is non-empty, the first listed agent becomes the default when no entry has `default: true`, unmatched traffic uses that default, and `openclaw config set agents.list ...` still only prints the generic restart message. The related open #60712 is adjacent routing/privacy work, but it does not supersede this narrower config-write warning request.

## What This Changes

The issue requests a warning or guard so adding an ACP agent to `agents.list` without an explicit default or `main` entry does not silently make that agent the default for all unbound routing.

## Best Possible Solution

Add a conservative config UX guard for `agents.list` mutations: when the resulting list is non-empty and has neither `default: true` nor `id: "main"`, warn that unbound traffic will move away from implicit `main`. Keep the existing fallback semantics and document the warning path.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `resolveDefaultAgentId({ agents: { list: [{ id: "codex-cubby" }] } })` resolves to `codex-cubby`, unmatched `resolveAgentRoute` traffic uses that default, and the config writer has no special warning for this mutation.

## Solution Assessment

Yes, the narrow warning/guard is the best next fix. Auto-including `main` or requiring explicit defaults would change the documented routing contract and should be separate maintainer product work.
