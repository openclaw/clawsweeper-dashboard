## Label Justifications

- `P1`: The source-proven behavior can reroute all unmatched real traffic and session state to a newly listed ACP agent without an explicit default.
- `impact:session-state`: The reported path changes session keys and default workspace ownership from `main` to the first configured agent, orphaning active continuity unless manually repaired.
- `impact:security`: Unmatched private messages can route to an unintended ACP agent/workspace when the operator expected only one bound topic to use it.
- `impact:message-loss`: The issue is about channel traffic being misrouted to the wrong agent rather than staying with the implicit main agent.

## Summary

Keep open: current main still selects the first `agents.list` entry as the default for unmatched traffic, and the remaining safety behavior is not fully covered by the narrower open explicit-main binding issue.

## What This Changes

The issue asks OpenClaw to prevent a newly added ACP-bound `agents.list` entry from silently becoming the default route for all otherwise-unmatched traffic.

## Best Possible Solution

Add a maintainer-approved safety path around first-time or non-default `agents.list` writes, such as warning or requiring an explicit default when existing implicit `main` state would be displaced, while preserving a valid explicit-main binding path.

## Reproduction Assessment

Yes. Source inspection on current main shows a config with one `agents.list` entry and no matching binding resolves all fallback routes and bare session keys to that first entry; I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear as a final policy. Warning or blocking unsafe first-time `agents.list` writes looks safer than silently auto-adding state, but changing the fallback rule itself needs maintainer product and upgrade judgment.
