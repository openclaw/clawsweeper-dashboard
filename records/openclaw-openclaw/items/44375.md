## Label Justifications

- `P1`: The issue can silently reroute live channel, heartbeat, and cron traffic away from the user's expected default agent.
- `impact:session-state`: Changing the fallback agent changes agent-scoped session keys and can strand existing `main` history and memory state.
- `impact:message-loss`: Unmatched channel traffic can be delivered to a different agent than the user expected after adding one ACP agent.
- `impact:security`: Silent fallback can expose messages to an unintended ACP/runtime workspace with different tool and sandbox policy.

## Summary

Keep open: current main and v2026.6.10 still make the first `agents.list` entry the default when no entry has `default: true`, and unmatched route fallback uses that default. The linked open PR only addresses explicit `main` binding preservation, not this issue's default-selection foot-gun.

## What This Changes

The issue requests warning or upgrade-safe behavior when adding an ACP agent to `agents.list` would displace the implicit `main` agent as the fallback route target.

## Best Possible Solution

Maintainers should choose the default-routing compatibility policy, then update config write/doctor warnings or routing behavior plus docs and regression coverage consistently.

## Reproduction Assessment

Yes, source-level: a non-empty `agents.list` with no `default: true` resolves the first entry as the default, and unmatched routes use that value. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Current behavior matches docs but is unsafe for this reported upgrade/setup path; the best fix needs a maintainer compatibility choice rather than a narrow automatic code change.
