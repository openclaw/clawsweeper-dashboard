## Label Justifications

- `P1`: Silent default-agent reassignment can redirect all unbound traffic and disrupt established agent workflows without immediate operator visibility.
- `impact:session-state`: The behavior changes agent-scoped session keys and disconnects new traffic from the established main-agent history and memory state.
- `impact:security`: A silent default change can expose private or privileged traffic to an unintended agent persona or ACP runtime.
- `impact:message-loss`: Messages intended for the established default agent are instead delivered to another configured agent.

## Summary

Current main and v2026.6.5 still use the first agents.list entry as the default when no entry has default=true, so the reported silent routing takeover remains necessary to address; related issues cover missing-agent fallback and implicit-main binding validation, not this compatibility-sensitive default-selection policy.

## What This Changes

Prevent adding a scoped ACP agent from silently replacing the implicit main agent as the default for unrelated traffic.

## Best Possible Solution

Adopt one explicit, upgrade-safe default-agent contract that preserves existing main-agent state, warns or migrates before first-entry takeover, and aligns routing, config mutation, doctor, setup, regression tests, and multi-agent documentation.

## Reproduction Assessment

Yes, from current-main source: populate agents.list with one unmarked ACP agent and any route that misses a binding selects that first entry instead of main. This read-only review did not start a live gateway, so the classification is source-reproducible rather than runtime-reproduced.

## Solution Assessment

Unclear: the report identifies a real foot-gun, but warning-only, automatic main insertion, and mandatory explicit defaults have different upgrade consequences. Maintainers should choose the canonical default contract before implementation and require fresh-install plus upgrade coverage.
