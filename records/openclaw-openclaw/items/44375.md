## Label Justifications

- `P1`: The issue can silently reroute live channel, heartbeat, and cron traffic away from the user's expected default agent.
- `impact:message-loss`: Unmatched channel traffic can be delivered to a different agent than the user expected after adding one ACP agent.
- `impact:session-state`: Changing the fallback agent changes agent-scoped session keys and strands existing `main` history and memory state.
- `impact:security`: Silent fallback can expose messages to an unintended ACP/runtime workspace with different tool and sandbox policy.

## Summary

Keep open: current main and the latest release still let a non-empty `agents.list` with no `default: true` make the first listed agent the default, so adding one ACP agent can silently move unmatched traffic away from implicit `main`. The linked open work only addresses explicit `main` binding preservation, not the default-selection foot-gun reported here.

## What This Changes

The issue requests safer multi-agent default routing when adding an ACP-bound agent so unbound traffic keeps using implicit `main` or gets an explicit warning/default choice.

## Best Possible Solution

Maintainers should choose the default-routing policy, then implement one consistent upgrade-safe path covering config writes, doctor guidance/migration, ordinary route fallback, configured ACP bindings, docs, and regression tests.

## Reproduction Assessment

Yes, source-level: with `agents.list` containing one entry and no `default: true`, current main resolves that entry as the default and unmatched routes use it. I did not run a live gateway because this review is read-only.

## Solution Assessment

No, current behavior is not the best end state for safety; however the exact fix shape needs maintainer policy because docs currently define first-entry default fallback. A safer solution must make default displacement explicit and cover both normal routing and persistent ACP binding consumers.
