## Label Justifications

- `P1`: The report describes a real user workflow where a config change can reroute all unbound traffic and split session continuity.
- `impact:message-loss`: Unbound messages can be delivered to an unintended agent after `agents.list` default displacement.
- `impact:session-state`: Existing `main` sessions, auth state, and memory can become unused when the fallback agent id changes.
- `impact:security`: Private or scheduled traffic can enter an unintended ACP runtime or workspace.

## Summary

Keep open. Current `main` still resolves a non-empty `agents.list` to `default: true` or the first configured agent, raw `config set agents.list` still has only generic restart output, and the related open routing privacy issue covers stale binding fallback rather than this implicit-main displacement path.

## What This Changes

The issue requests a warning, migration, or stricter default-agent behavior when adding a single ACP agent to `agents.list` would displace the implicit `main` agent for unbound traffic.

## Best Possible Solution

Add a compatible `config set`, startup, or `doctor` diagnostic when `agents.list` is populated without `main` or an explicit default while `main` state exists; reserve semantic fallback changes for an explicit maintainer product/security decision.

## Reproduction Assessment

Yes, at source level: set `agents.list` to one non-main ACP entry without `default: true`; `resolveDefaultAgentId` picks that entry and `resolveAgentRoute` uses it after binding misses. I did not run a live Telegram/ACP gateway in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract. A warning or doctor diagnostic is the narrowest compatible direction, while auto-including `main` or requiring explicit defaults would change documented routing behavior.
