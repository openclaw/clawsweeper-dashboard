## Label Justifications

- `P1`: A normal config edit can silently reroute all unmatched traffic away from the user's established main agent and affect active real workflows.
- `impact:message-loss`: Unmatched DMs, group topics, heartbeats, or cron messages can be delivered to the new ACP default agent instead of the intended main agent.
- `impact:session-state`: The fallback agent id changes session keys and can strand existing `main` history, memory index, and auth-related state.
- `impact:security`: Private or scheduled traffic can enter an unintended ACP runtime and workspace after implicit default-agent displacement.

## Summary

Keep open. Current main and v2026.5.22 still use the documented first-entry default for a non-empty `agents.list`, raw `openclaw config set agents.list ...` still has no displacement warning, and the adjacent open routing issue covers stale binding fallback rather than this first-time implicit-main displacement path.

## What This Changes

The issue asks for a warning, guard, or explicit-default behavior when adding an ACP agent to `agents.list` would displace the implicit `main` agent for unmatched routing.

## Best Possible Solution

Add a compatible config-set, startup, or doctor diagnostic when `agents.list` omits both `main` and `default: true` while existing `main` state is present; reserve default-contract changes for explicit maintainer approval and migration tests.

## Reproduction Assessment

Yes, at source level. Set `agents.list` to a single non-main ACP entry without `default: true`; after binding tiers miss, routing resolves through `resolveDefaultAgentId` to that entry, and bare session keys canonicalize to the same configured default.

## Solution Assessment

Unclear until maintainers choose the compatibility contract. A warning or diagnostic is the narrowest compatible fix, while auto-including `main` or requiring explicit defaults changes documented routing semantics and needs explicit approval.
