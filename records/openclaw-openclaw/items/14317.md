## Summary

Keep open for explicit maintainer and security review: current main still lacks the requested Slack admission mode, no canonical issue or PR supersedes it, and the proposal introduces a new access-control contract rather than fixing broken documented behavior.

## What This Changes

Add a Slack `groupPolicy: "private"` mode that accepts invited private channels while rejecting non-allowlisted public channels.

## Best Possible Solution

If maintainers approve the trust model, add a Slack-specific private-channel admission mode while retaining `allowlist` as the default, use one canonical decision across messages and slash commands, preserve explicit public-channel allowlisting, and document and audit Slack Connect exposure accurately.

## Reproduction Assessment

Yes at source level: current configuration and both Slack authorization routes accept only `open`, `allowlist`, and `disabled`, so the requested mode is demonstrably absent. This is a feature gap rather than a failure of documented existing behavior.

## Solution Assessment

Unclear pending the product decision: a Slack-scoped policy is a maintainable implementation shape, but its trust semantics and security-audit treatment must be approved before coding, and the implementation must cover both message and command routes.
