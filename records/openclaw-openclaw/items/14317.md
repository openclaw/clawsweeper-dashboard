## Summary

Keep open: current main and the latest release still lack Slack `groupPolicy: "private"`, and the remaining work changes Slack channel authorization semantics rather than fixing a documented regression.

## What This Changes

The issue requests a Slack-only `groupPolicy: "private"` mode that allows private channels while ignoring public channels unless explicitly allowlisted.

## Best Possible Solution

If maintainers approve the trust boundary, add a Slack-scoped private-channel admission mode across config/schema/metadata, normal and slash routing, Slack read/action authorization, security audit guidance, docs, and focused tests while preserving existing defaults unless separately approved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main and `v2026.6.8` accept only `open`, `disabled`, and `allowlist`, and Slack routing has no privacy-aware policy branch.

## Solution Assessment

Is this the best way to solve the issue? Unclear pending product/security approval: a Slack-scoped private policy is plausible, but maintainers must first decide whether private-channel membership is an acceptable authorization boundary and how to avoid widening the shared group-policy contract accidentally.
