## Summary

Keep open: current main and the latest release still lack Slack `groupPolicy: "private"`, and the remaining work changes Slack authorization semantics rather than fixing a documented regression.

## What This Changes

The issue requests a Slack-only `groupPolicy` value that allows private channels while dropping public channels unless allowlisted.

## Best Possible Solution

If maintainers approve the trust boundary, add a Slack-scoped private-channel policy across config/schema/metadata, normal and slash routing, Slack read/action authorization, security audit guidance, public docs, and focused tests while preserving existing defaults unless separately approved.

## Reproduction Assessment

Yes at source level: current main accepts only `open`, `disabled`, and `allowlist`, and Slack routing has no privacy-aware policy branch. A live Slack workspace reproduction was not run because the schema and routing gap already proves the requested feature is absent.

## Solution Assessment

Unclear pending product/security approval: a Slack-scoped private policy is plausible, but maintainers must decide whether Slack private-channel membership is an acceptable authorization boundary and how to avoid widening shared group-policy behavior accidentally.
