## Summary

Keep open: current main and the latest release still lack Slack `groupPolicy: "private"`, and the remaining work changes Slack channel authorization semantics, so it needs maintainer product/security approval before implementation.

## What This Changes

The issue requests a Slack-only `groupPolicy: "private"` mode that would allow private channels by membership while dropping public channels unless explicitly allowlisted.

## Best Possible Solution

If maintainers approve the trust boundary, add a Slack-scoped private-channel admission mode across config/schema/metadata, normal and slash routing, Slack read/action authorization, security audit guidance, docs, and focused tests while preserving existing defaults unless separately approved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main and `v2026.6.8` accept only `open`, `disabled`, and `allowlist`, and Slack routing has no privacy-aware policy branch.

## Solution Assessment

Is this the best way to solve the issue? Unclear pending product/security approval: a Slack-scoped policy is plausible, but maintainers need to decide whether private-channel membership, including Slack Connect private channels, is an acceptable authorization boundary.
