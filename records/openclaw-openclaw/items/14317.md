## Summary

Keep open: current main and v2026.6.6 still lack Slack `groupPolicy: "private"`, and the remaining work is a security-sensitive Slack config/access-control feature that needs maintainer product approval before implementation.

## What This Changes

Add a Slack `groupPolicy: "private"` option that allows private channels while blocking public channels unless explicitly allowlisted.

## Best Possible Solution

If maintainers approve the trust model, add a Slack-scoped private-channel admission mode across config/schema/metadata, normal and slash routing, Slack action/read authorization, security audit guidance, docs, and focused tests while preserving existing defaults unless separately approved.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 accept only `open`, `disabled`, and `allowlist`, and Slack routing has no privacy-aware policy branch. This is a feature gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear pending product/security approval: a Slack-scoped policy is plausible, but maintainers need to decide whether private-channel membership, including Slack Connect private channels, is an acceptable authorization boundary.
