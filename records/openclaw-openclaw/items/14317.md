## Summary

Keep open. Current main still rejects Slack `groupPolicy: "private"` in the shared config contract and Slack routing has no privacy-aware branch; prior review comments remain accurate, and this access-control feature needs maintainer/security approval before implementation.

## What This Changes

Request adds a Slack `groupPolicy: "private"` mode that permits invited private channels while blocking public channels unless explicitly allowlisted.

## Best Possible Solution

If approved, add a Slack-scoped private policy across the public config contract, Slack normal and slash routing, generated metadata/docs, exposure audit classification, and focused tests, with explicit Slack Connect caveats.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection on current main shows `groupPolicy: "private"` is rejected by schema and has no private-channel routing branch.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the Slack trust model; if approved, the maintainable path is Slack-specific policy handling with privacy metadata and audit/docs coverage, not a generic enum-only change.
