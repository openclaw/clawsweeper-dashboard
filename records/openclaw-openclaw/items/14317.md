Request to add a Slack `groupPolicy: "private"` mode that admits invited private channels while dropping public channels unless explicitly allowlisted.

## Best Possible Solution

If maintainers approve the trust model, add a Slack-scoped private-channel admission mode that preserves explicit public-channel allowlisting, keeps `allowlist` as the safe default, and documents Slack Connect caveats.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` accept only `open`, `disabled`, and `allowlist`, and Slack routing has no privacy-aware branch. This is a feature gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear pending product/security approval: a Slack-scoped policy is plausible, but maintainers must first decide whether Slack private-channel membership is an acceptable authorization boundary and how to treat Slack Connect.
