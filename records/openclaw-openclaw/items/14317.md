## Summary

Keep open. Current main still does not accept or enforce Slack `groupPolicy: "private"`, and the prior keep-open review is still accurate; this is an access-control feature that needs maintainer/security product approval before implementation.

## What This Changes

Add a Slack `groupPolicy: "private"` mode that allows private channels by membership, blocks public channels unless allowlisted, and adjusts audit guidance.

## Best Possible Solution

If maintainers approve the trust boundary, implement a Slack-scoped private policy through config schema/types, metadata, normal and slash routing, security audit guidance, docs, and focused tests while preserving fail-closed missing-provider behavior and documenting Slack Connect caveats.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence current main lacks the requested mode because config validation rejects `private` and both Slack routing paths only implement open/disabled/allowlist.

## Solution Assessment

Not yet for automation. The implementation shape is clear, but this changes Slack authorization semantics, so the best path starts with maintainer/security approval before coding the config and routing changes.
