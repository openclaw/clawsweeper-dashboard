## Summary

Keep open. Current main still does not implement Slack `groupPolicy: "private"`: the config contract rejects it, Slack channel and slash-command routing have no private-channel branch, docs/metadata list only the existing policies, and the exposure audit has no classification for the proposed trust boundary.

## What This Changes

The issue requests a Slack `groupPolicy: "private"` mode that allows invited private channels while requiring explicit allowlisting for public channels.

## Best Possible Solution

After maintainer/security approval, implement a Slack-scoped private policy end to end: config types/schema/metadata, normal and slash-command authorization that carries Slack privacy metadata, public-channel allowlist fallback, exposure-audit classification, docs with Slack Connect caveats, and focused regression tests while preserving fail-closed behavior when Slack config is missing.

## Reproduction Assessment

Yes. A source-level reproduction is enough: current main rejects `groupPolicy: "private"` in the shared schema and has no private-channel branch in normal or slash-command Slack routing, so no live Slack workspace is needed to verify the missing feature.

## Solution Assessment

Unclear until maintainers approve the trust model. If approved, the narrow maintainable solution is a Slack-specific `private` policy with explicit audit/docs coverage, not a generic enum addition that relies on existing policy fallthrough.
