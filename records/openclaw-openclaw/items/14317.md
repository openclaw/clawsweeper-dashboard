## Summary

Keep open. Current main still lacks Slack `groupPolicy: "private"` in the shared config contract and Slack routing paths, and the requested behavior changes Slack authorization and audit semantics rather than fixing a broken existing contract.

## What This Changes

Add a Slack `groupPolicy: "private"` mode that allows invited private channels by default while requiring explicit allowlist entries for public channels.

## Best Possible Solution

If maintainers approve the trust boundary, implement a Slack-scoped private policy across config, normal and slash routing, audit/docs, generated metadata, and focused tests with Slack Connect caveats.

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature request; source inspection against current main shows `groupPolicy: "private"` is rejected by config and has no privacy-aware Slack routing path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the Slack trust model; if approved, the narrow maintainable path is a Slack-specific policy branch with privacy metadata, audit classification, docs, and tests, not an enum-only change.
