## Summary

Keep open. Current main still lacks Slack `groupPolicy: "private"`, and the request changes Slack authorization and security-audit semantics rather than being obsolete cleanup.

## What This Changes

Add a Slack `groupPolicy: "private"` mode that allows invited private channels while blocking public channels unless explicitly allowlisted.

## Best Possible Solution

If maintainers approve the trust boundary, implement a Slack-scoped private policy across config types/schema/metadata, normal and slash-command routing, exposure audit/docs, and focused regression tests with a Slack Connect caveat.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature request; source inspection gives high-confidence verification that current main rejects `groupPolicy: "private"` and has no private-channel authorization path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the Slack trust model. If approved, the maintainable path is a Slack-specific policy branch with audit and docs coverage, not a generic enum addition that falls through existing policy logic.
