## Summary

Keep open. Current main still does not support Slack `groupPolicy: "private"` in the shared config contract, Slack docs, normal message routing, slash-command routing, or exposure-audit logic, and the requested behavior changes Slack channel authorization semantics.

## What This Changes

Request to add a Slack `groupPolicy: "private"` value that allows invited private channels while requiring explicit allowlist entries for public channels.

## Best Possible Solution

Approve or reject the Slack private-channel trust model; if approved, add a Slack-scoped policy covering config schema/types/metadata, routing, audit, docs, and focused tests with explicit caveats.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection shows `groupPolicy: "private"` is not accepted by the type/schema contract and Slack routing has no privacy-aware branch.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the Slack private-channel trust model; if approved, the maintainable path is a Slack-aware policy with audit/docs/tests, not an enum-only change.
