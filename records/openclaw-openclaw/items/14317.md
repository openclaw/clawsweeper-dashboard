## Summary

Keep open. Current main still lacks Slack `groupPolicy: "private"` in the config contract, docs, normal routing, slash-command routing, and exposure audit; the request changes Slack authorization semantics and needs maintainer/security product approval before implementation.

Request to add a Slack-only `groupPolicy: "private"` that allows invited private channels while blocking public channels unless explicitly allowlisted.

## Best Possible Solution

Approve or reject the Slack trust model; if approved, add a Slack-scoped policy covering config schema/types/metadata, routing, audit, docs, and focused tests with Slack Connect caveats.

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection of current main shows `groupPolicy: "private"` is not accepted by the type/schema contract and Slack routing has no privacy-aware branch.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the Slack private-channel trust model; if approved, the narrow maintainable path is a Slack-specific policy with privacy metadata, audit, docs, and tests, not an enum-only change.
