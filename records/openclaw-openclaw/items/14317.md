## Summary

Keep open. Current main still does not implement Slack `groupPolicy: "private"` in the config contract, Slack docs, normal routing, slash-command routing, or exposure audit, and the requested behavior changes a Slack authorization boundary that needs maintainer/security product approval before implementation.

## What This Changes

Requested a Slack `groupPolicy: "private"` mode that allows invited private channels, optionally allowlisted public channels, and ignores other public channels.

## Best Possible Solution

If approved, implement a Slack-scoped private policy across config types/schema/metadata, normal and slash routing, exposure audit, docs, and focused tests with explicit Slack Connect caveats.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection of current main shows `groupPolicy: "private"` is not in the schema/type contract and Slack routing has no privacy-aware branch.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve the Slack trust model; if approved, the narrow maintainable path is a Slack-specific policy with privacy metadata and audit/docs coverage, not a generic enum-only change.
