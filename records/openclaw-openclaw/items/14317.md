## Summary

Keep open. Current main still does not implement Slack `groupPolicy: "private"`, and the requested change would alter Slack authorization and security-audit semantics rather than clean up an obsolete report.

## What This Changes

The issue requests a Slack-only `groupPolicy: "private"` option that allows invited private channels while requiring explicit allowlisting for public channels.

## Best Possible Solution

If maintainers approve the trust boundary, implement a Slack-scoped private policy end to end across config contract, normal and slash-command routing, exposure audit/docs, and focused tests, with a documented Slack Connect caveat and explicit public-channel allowlist fallback.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high-confidence verification that current main rejects `groupPolicy: "private"` and has no private-channel authorization path.

## Solution Assessment

Unclear until maintainers approve the Slack trust model. If approved, the maintainable solution is a Slack-specific policy branch with audit and docs coverage, not a generic enum addition that falls through existing policy logic.
