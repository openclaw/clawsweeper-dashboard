## Summary

Keep open: current main and the latest release still lack Slack `groupPolicy: "private"`, and the remaining work is a security-sensitive Slack access-control feature that needs maintainer product/security approval before implementation.

## What This Changes

The issue requests a Slack-only `groupPolicy: "private"` mode that admits invited private channels while keeping public channels blocked unless explicitly allowlisted.

## Best Possible Solution

If maintainers approve the trust model, add a Slack-scoped private-channel admission mode across config/schema/metadata, normal and slash routing, Slack action/read authorization, security audit guidance, docs, and focused tests while preserving existing defaults unless separately approved.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` accept only `open`, `disabled`, and `allowlist`, and Slack routing has no privacy-aware policy branch. This is a feature gap rather than broken existing documented behavior.

## Solution Assessment

Unclear pending product/security approval: a Slack-scoped policy is plausible, but maintainers need to decide whether private-channel membership, including Slack Connect private channels, is an acceptable authorization boundary.
