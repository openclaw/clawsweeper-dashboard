## What This Changes

The PR adds an exec preflight guard for raw global OpenClaw package-manager self-updates, regression coverage, and a changelog entry.

## Best Possible Solution

Finish the parser so it covers package-manager global options wherever the real CLI accepts them, add regression cases for those forms, then require real after-patch exec or Gateway proof before landing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the linked field report gives concrete raw npm self-update logs followed by stale chunk failures, and current main source inspection shows no exec-side guard for that command class.

## Solution Assessment

Is this the best way to solve the issue? Not yet. Exec preflight is the narrow maintainable boundary, but the parser must recognize npm's pre-subcommand global config forms and the PR still needs real after-fix behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle global options before npm subcommands:** `src/agents/bash-tools.exec.ts:1172-1182`
  - body: `parseNpmLikeOpenClawGlobalInstall` assumes `argv[1]` is the subcommand and only scans for global flags after it. npm also accepts global config before the command, so `npm --global install openclaw@latest` and `npm --location global update openclaw` still bypass this guard and can leave the stale-gateway failure mode reachable. Parse leading npm config flags before selecting the subcommand and add regression coverage for those forms.
  - confidence: 0.88
