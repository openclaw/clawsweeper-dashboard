## Summary

Keep open: current main and latest release still have the pre-guard plugin service metadata read, and this PR is a focused fix attempt, but the branch still starts registrations whose owner metadata is unreadable and overlaps with https://github.com/openclaw/openclaw/pull/89898. Because this is a draft, maintainer-labeled MEMBER PR, maintainers should choose the canonical branch instead of cleanup-closing it.

## What This Changes

The PR adds guarded plugin service metadata preparation in `startPluginServices` and regression tests for unreadable service metadata, explicit `unknown-service` ids, and dynamic stop handlers.

## Best Possible Solution

Adopt one canonical plugin service metadata guard, preferably the sibling or an updated version of this branch that treats unreadable owner metadata as a failed registration, preserves receiver and dynamic-stop behavior, and passes normal pre-merge validation.

## Reproduction Assessment

Yes by source inspection: current main and `v2026.6.6` read plugin service registration metadata before the per-service guard, so a throwing getter can abort startup before healthy services run. I did not run tests in this read-only review.

## Solution Assessment

No, not as this branch currently stands. Preparing registration reads inside `startPluginServices` is the right boundary, but unreadable owner metadata should fail the registration instead of starting it under fallback identity.
