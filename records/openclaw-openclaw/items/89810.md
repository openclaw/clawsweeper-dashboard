## What This Changes

The PR adds prepared plugin service registration reads, per-service failure logging, and regression tests for unreadable service metadata, explicit `unknown-service` ids, and stop handlers installed during startup.

## Best Possible Solution

Land one canonical plugin service metadata guard that skips any unreadable owner or service metadata, preserves receiver-bound start/stop behavior, covers dynamic stop handlers, and retires the overlapping branch.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main dereferences service registration metadata before the per-service `try`, and the PR body includes a focused red repro. I did not run tests in this read-only review.

## Solution Assessment

No, not as written. Preparing registrations inside `startPluginServices` is the right boundary, but unreadable owner metadata should fail the registration instead of starting the service under fallback identity; the sibling PR already demonstrates that narrower shape.
