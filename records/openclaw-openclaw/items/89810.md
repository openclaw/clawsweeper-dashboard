## What This Changes

The PR guards plugin service registrations before startup, logs unreadable metadata as per-service failures, and adds service lifecycle regression tests.

## Best Possible Solution

Land this after maintainer review and the normal broad changed gate, keeping malformed plugin service registrations isolated to per-service failures without changing the public service API.

## Reproduction Assessment

Yes. Source inspection of current main shows `entry.service` is dereferenced before the per-service `try`, and the PR body includes a focused red repro for a throwing service getter.

## Solution Assessment

Yes. Preparing each service registration before startup is the narrow runtime-boundary fix; moving the check only to registration would not cover already-assembled bad registry rows or metadata becoming unreadable at service-start time.
