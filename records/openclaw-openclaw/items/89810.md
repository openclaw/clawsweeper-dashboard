## Summary

Keep open: current main still has the plugin-service row isolation failure, but this draft is not merge-ready because unreadable owner metadata is still downgraded to `unknown-plugin` and an open sibling draft covers the same lifecycle guard with owner-metadata coverage.

## What This Changes

The branch pre-reads plugin service registration data in `startPluginServices`, logs and skips unreadable service records, preserves explicit service IDs, and adds service lifecycle regression tests.

## Best Possible Solution

Land one canonical plugin service metadata guard that treats unreadable owner and service metadata as failed registrations, preserves receiver and dynamic-stop behavior, and then retire the overlapping draft.

## Reproduction Assessment

Yes by source inspection. Current main reads service registration metadata before the per-service `try`, so a throwing getter can abort the loop before later healthy services start.

## Solution Assessment

No, not as written. Preparing registration reads inside `startPluginServices` is the right boundary, but unreadable owner metadata should fail the registration or this branch should be replaced by the sibling that covers that path.
