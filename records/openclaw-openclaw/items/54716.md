## Summary

Keep open. Current main still under-counts sessions for literal per-agent store paths, and this PR is useful but not merge-ready because it needs a current-main port plus fixes for resolved path propagation, disk-derived agent identity, and the required changelog entry.

## What This Changes

The PR changes gateway session aggregation so a literal `agents/<id>/sessions/sessions.json` store discovers sibling per-agent stores and adds regression coverage for that layout.

## Best Possible Solution

Land an equivalent current-main fix in the session config loader and target resolver: literal agent-layout stores should discover sibling stores only under the same agents root, preserve the resolved literal store path, derive agent identity from disk layout, and keep true shared literal stores on the single-store path.

## Reproduction Assessment

Yes. Configure `session.store` as a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store on disk; current main returns before sibling discovery, and the PR discussion includes API-backed proof changing `sessions.list` from `count=1` to `count=2`.

## Solution Assessment

No, not as written. The direction is right, but the patch needs to be ported to the current `src/config/sessions` implementation and must fix path preservation plus literal-layout agent identity before it is the narrow maintainable solution.
