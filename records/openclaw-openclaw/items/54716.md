## Summary

Keep open: the bug and proof are valid, but this PR is stale against current main because the active `sessions.list` loader now lives under `src/config/sessions`; the diff still edits the old gateway-local implementation and leaves concrete path and agent-id defects.

## What This Changes

This PR changes gateway session aggregation so literal `agents/<id>/sessions/sessions.json` `session.store` paths discover same-root sibling agent stores and adds regression coverage.

## Best Possible Solution

Port the fix into `src/config/sessions/combined-store-gateway.ts` and `src/config/sessions/targets.ts` so literal agent-layout stores discover only same-root sibling stores while non-agent-layout literal stores remain single-store.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source still returns after loading only the literal default-agent store, and the linked proof shows the before-fix API count as 1.

## Solution Assessment

No, not as written. The direction is right, but the implementation must move to the active `src/config/sessions` loader/resolver and preserve the resolved literal path plus the agent id derived from the on-disk store layout.
