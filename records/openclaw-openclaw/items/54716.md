## Summary

Keep open: the PR targets a real gateway session-listing bug and has usable API-backed before/after proof, but the branch is not merge-ready because current main moved the active loader and the latest patch can still canonicalize a literal per-agent store under the wrong agent id.

## What This Changes

The PR changes gateway session-store loading so literal per-agent `session.store` paths can discover sibling agent stores while preserving single-store behavior for non-layout literal paths.

## Best Possible Solution

Port or replace this PR in the current combined-store loader so literal per-agent paths discover same-root siblings, non-layout literal stores remain single-file stores, and canonicalization derives the agent namespace from the on-disk path.

## Reproduction Assessment

Yes, at source level. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store; current main returns from the literal branch before discovery, and the linked proof artifacts show the API count changing from 1 to 2 on the fix branch.

## Solution Assessment

No, not as currently shaped. The requested direction is right, but the implementation must be ported to the current loader and must derive the canonical agent id from the literal store path before merge.
