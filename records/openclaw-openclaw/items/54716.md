## Summary

Keep open. The reported bug is real and the linked proof is sufficient, but this PR is not correct against current main because it patches the old gateway-local loader instead of the active config/sessions loader, and it still has resolved-path and agent-id semantics issues.

## What This Changes

The PR branch updates gateway session utilities and regression tests so literal `agents/<id>/sessions/sessions.json` store paths discover sibling agent stores while preserving non-agent literal paths as single-store.

## Best Possible Solution

Implement the literal agent-layout handling in `src/config/sessions` so same-root sibling stores are discovered, true shared literal stores remain single-store, resolved paths are preserved, and on-disk agent IDs drive key canonicalization.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source returns from the literal-store branch before sibling discovery, and the linked proof shows count 1 before versus count 2 after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the maintainable fix needs to move into the active `src/config/sessions` loader and resolver while preserving resolved-path and on-disk-agent-id semantics.
