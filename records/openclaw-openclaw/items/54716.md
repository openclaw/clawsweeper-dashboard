## Summary

Keep this PR open: the linked `sessions.list` bug is still present on current main, and the PR has useful real before/after proof, but the branch now needs a current-main port because the active loader moved into `src/config/sessions`.

## What This Changes

The PR changes gateway session aggregation so literal `agents/<id>/sessions/sessions.json` store paths discover sibling per-agent session stores and adds regression coverage for that layout.

## Best Possible Solution

Land an equivalent current-main fix in the config/sessions loader and target resolver, preserving resolved paths and disk-derived agent identity while leaving true shared literal stores on the single-store path.

## Reproduction Assessment

Yes. Source inspection shows a literal `.../agents/main/sessions/sessions.json` config enters the active loader's single-store return before sibling discovery, matching the PR's API-backed before-proof.

## Solution Assessment

No, not as currently written. The direction is right, but the patch must be ported to `src/config/sessions/combined-store-gateway.ts` and `src/config/sessions/targets.ts` before it is the narrow maintainable fix.

  - confidence: 0.84

Legacy keys can then be canonicalized under the wrong agent namespace; derive the agent id from the literal/discovered store path for this layout.
  - confidence: 0.82
