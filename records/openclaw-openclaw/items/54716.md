## Summary

Keep open: current main still takes the single-store path for literal `session.store` values, while this PR is useful source work but needs to be ported to the current `src/config/sessions` loader and corrected before merge.

## What This Changes

The PR changes gateway session aggregation so literal `agents/<id>/sessions/sessions.json` store paths discover sibling per-agent session stores instead of returning only the configured file.

## Best Possible Solution

Land an equivalent current-main fix in the session config loader and target resolver so literal agent-layout stores discover siblings only under the same agents root, preserve resolved paths, derive agent identity from disk layout, and leave true shared literal stores on the single-store path.

## Reproduction Assessment

Yes. A literal `.../agents/main/sessions/sessions.json` config with a sibling agent store enters the current single-store return, and `sessions.list` consumes that combined store, so sibling sessions are omitted without needing a new product assumption.

## Solution Assessment

No, not as written. The direction is right, but the patch must be ported to the current `src/config/sessions` implementation and fix resolved path propagation plus literal-layout agent identity before it is the narrow maintainable solution.
