## Summary

Keep this PR open: it targets a real `sessions.list` bug with useful proof, but the branch is stale against current `main` and patches the old gateway-local implementation instead of the active `src/config/sessions` loader.

## What This Changes

The PR changes session-store aggregation so a literal `agents/<id>/sessions/sessions.json` path discovers sibling per-agent session stores and adds regression coverage for that case.

## Best Possible Solution

Port the literal agent-layout handling into `src/config/sessions` so scoped sibling discovery, resolved store paths, and on-disk agent IDs are handled by the active loader while true shared literal stores remain single-store paths.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source still returns early for literal stores, and the linked proof reports `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
