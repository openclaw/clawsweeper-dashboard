## Summary

Keep open: the linked bug is real and the PR has useful before/after proof, but the branch is now conflict-stale and the patch still risks mis-associating literal per-agent stores when configured agent order differs from the path owner.

## What This Changes

The PR changes gateway session-store loading so literal `session.store` paths matching `agents/<id>/sessions/sessions.json` discover sibling agent stores, with regression tests for sibling discovery and custom-root scoping.

## Best Possible Solution

Land a rebased fix in `src/config/sessions/combined-store-gateway.ts` that treats literal agent-layout paths as scoped multi-agent roots while deriving ownership from the on-disk `agents/<id>` path and preserving regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main still treats literal store paths as a single file, and the attached before/after artifacts show the real `sessions.list` handler returning one session before and two after for the literal per-agent path case.

## Solution Assessment

No, not as submitted. The direction is right, but the fix should be ported into the current combined-store owner module and derive agent ownership from the on-disk `agents/<id>` path before key canonicalization.
