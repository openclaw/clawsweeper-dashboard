## Summary

Keep open. The bug and PR direction are valid, but current main still short-circuits literal non-template `session.store` paths through a single default-agent store, and this PR branch is now conflicting and patches the older gateway-local loader instead of the active config/sessions loader.

## What This Changes

The branch changes gateway session aggregation and adds regression tests so literal `agents/<id>/sessions/sessions.json` store paths discover sibling agent stores while non-agent-layout literal stores remain single-store.

## Best Possible Solution

Land an equivalent current-main repair in `src/config/sessions/combined-store-gateway.ts` and `src/config/sessions/targets.ts`, with regression coverage for literal `agents/main/sessions/sessions.json` configs and same-root sibling discovery.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source still returns after loading only the literal default-agent store, and the linked proof shows `count=1` before the branch fix.

## Solution Assessment

No, not as written. The fix direction is right, but it needs to be ported to the active `src/config/sessions` loader and resolver while preserving resolved literal paths and on-disk agent IDs.
