## Summary

Keep open. The bug and intended direction are valid, and the linked proof is sufficient, but this branch is not correct against current main because it modifies a stale gateway-local loader while `sessions.list` now uses the active loader under `src/config/sessions`, with two additional path/canonicalization issues still unresolved.

## What This Changes

The branch modifies gateway session-store aggregation and tests so literal `agents/<id>/sessions/sessions.json` store paths discover sibling stores scoped to the same agents root.

## Best Possible Solution

Port the literal agent-layout handling into `src/config/sessions` so scoped sibling discovery, resolved store paths, and on-disk agent IDs are handled by the active loader while true shared literal stores remain single-store paths.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source shows the literal branch returns one store before sibling discovery, and the linked proof reports `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
