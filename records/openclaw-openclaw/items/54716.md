## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Return the resolved literal store path:** `src/gateway/session-utils.ts:1050-1052`
  - body: When a literal `session.store` uses `~` or a relative path and also matches the agent-layout pattern, this branch falls through to the multi-store return and reports `storeConfig.trim()` instead of the already resolved path. Callers such as sessions usage pass that `storePath` into transcript path resolution, so after this change usage/transcript lookup can miss existing files even though sessions.list found the row. Keep the resolved literal path, or return `(multiple)` only when the result truly spans multiple concrete roots.
  - confidence: 0.89
- **[P2] Canonicalize rows with the store-path agent id:** `src/gateway/session-utils.ts:1039`
  - body: `resolveAllAgentSessionStoreTargetsSync` dedupes by store path and preserves the first configured target, so a literal `.../agents/main/sessions/sessions.json` can be paired with the wrong `agentId` when config order starts with another agent. This loop then canonicalizes legacy keys and spawnedBy values under the wrong namespace, making older rows unreachable by their expected session keys; derive the owner from the literal store path or filter configured duplicates before merging rows.
  - confidence: 0.85
