## Summary

Keep open. The PR targets a real `sessions.list` bug and has sufficient linked before/after proof, but current `main` moved the active combined-store loader into `src/config/sessions`, so this branch's gateway-local fix is stale, conflicting, and still misses resolved-path and on-disk agent-id edge cases.

## What This Changes

The PR changes gateway session-store aggregation so literal `agents/<id>/sessions/sessions.json` paths discover sibling agent stores, while preserving single-store behavior for other literal paths, and adds regression coverage.

## Best Possible Solution

Move literal agent-layout handling into `src/config/sessions` so same-root sibling stores are discovered, true shared literal stores remain single-store, resolved paths and on-disk agent IDs are preserved, and regression coverage follows the active loader.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source still returns from the literal-store branch before sibling discovery, and the linked proof shows one row before versus two after.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader and resolver while preserving resolved-path and on-disk-agent-id semantics.
