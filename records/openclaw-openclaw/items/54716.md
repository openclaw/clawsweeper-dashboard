## Summary

Keep open. Current main still takes the single-store path for literal `session.store` values, and this PR is useful but not merge-ready because the branch is stale against the current session-store split and still has path propagation, agent identity, and changelog blockers.

## What This Changes

The PR changes gateway session aggregation so literal `agents/<id>/sessions/sessions.json` stores discover sibling per-agent session stores and adds regression coverage for that layout.

## Best Possible Solution

Land an equivalent current-main fix in the session config loader and target resolver: literal agent-layout stores should discover sibling stores only under the same agents root, preserve the resolved literal store path, derive agent identity from disk layout, and keep true shared literal stores on the single-store path.

## Reproduction Assessment

Yes. Configure `session.store` as a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store on disk; source inspection shows current main returns after loading the literal file, and the PR discussion includes API proof changing `sessions.list` from `count=1` to `count=2`.

## Solution Assessment

No, not as written. The direction is right, but the patch needs to be ported to the current `src/config/sessions` implementation and must fix resolved-path preservation plus literal-layout agent identity before it is the narrow maintainable solution.
