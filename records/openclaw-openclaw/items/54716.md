## Summary

Keep open: the PR is aimed at a valid sessions.list bug and has sufficient real behavior proof, but current main still has the literal-store early return and the branch patches a stale gateway-local loader instead of the active src/config/sessions implementation.

## What This Changes

The PR changes gateway session-store aggregation and regression tests so literal agents/<id>/sessions/sessions.json store paths discover sibling agent session stores.

## Best Possible Solution

Land a current-main fix in src/config/sessions that treats literal agent-layout stores as scoped multi-agent roots while preserving true shared literal stores as single-store paths.

## Reproduction Assessment

Yes. Configure session.store to a literal agents/main/sessions/sessions.json path with a sibling agent store, then call sessions.list; current-main source still returns from the literal single-store branch before sibling discovery, and the linked artifact shows count improving from one to two after the branch change.

## Solution Assessment

No, not as currently patched. The direction is right, but the fix needs to move into the active src/config/sessions loader and resolver, preserve the resolved literal path, and canonicalize with the on-disk agent id.
