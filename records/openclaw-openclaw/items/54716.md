## Summary

Keep open. The PR targets a real current-main bug and has useful proof, but the branch is conflicting and still patches a stale gateway-local implementation instead of the active `src/config/sessions` loader used by `sessions.list`.

## What This Changes

The PR changes literal `session.store` handling for per-agent session-store paths and adds regression coverage for sibling agent discovery.

## Best Possible Solution

Port the proven behavior into `src/config/sessions/combined-store-gateway.ts` and `src/config/sessions/targets.ts`, preserving true shared literal stores as single-store paths while literal agent-layout stores discover only same-root sibling stores with regression coverage.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` with a sibling agent store, then call `sessions.list`; current-main source returns from the single-store branch before sibling discovery, and the linked artifacts show the API improving from one returned session to two after the branch fix.

## Solution Assessment

No, not as currently patched. The direction is right, but the implementation needs to move into the active `src/config/sessions` loader and resolver while preserving resolved store paths and on-disk agent-id canonicalization.
