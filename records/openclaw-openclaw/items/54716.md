## Summary

Keep open. The PR addresses a real `sessions.list` bug and has useful before/after proof, but the current branch is conflicting and patches the stale gateway-local loader instead of the active `src/config/sessions` implementation on current main.

## What This Changes

The PR changes session-store loading so literal `agents/<id>/sessions/sessions.json` paths discover sibling agent stores, and adds gateway regression coverage for that scenario.

## Best Possible Solution

Land an equivalent current-main fix in `src/config/sessions` that treats literal agent-layout stores as same-root multi-agent roots while preserving true shared literal stores as single-store paths, with focused regression coverage.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` with a sibling agent store, then call `sessions.list`; current-main source returns from the single-store branch before sibling discovery, and the linked artifacts show the same scenario improving from one row to two after the branch fix.

## Solution Assessment

No, not as currently patched. The direction is right, but the implementation needs to move into the active `src/config/sessions` loader and resolver while preserving resolved store paths and on-disk agent-id canonicalization.
