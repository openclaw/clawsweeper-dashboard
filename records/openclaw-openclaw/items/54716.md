## Summary

Keep open: the PR targets a real `sessions.list` bug and has useful before/after proof, but the current branch does not patch the active current-main loader and still has path/canonicalization gaps.

## What This Changes

The branch updates gateway session-store aggregation and regression tests so literal `agents/<id>/sessions/sessions.json` paths discover sibling agent stores under the same agents root.

## Best Possible Solution

Land a current-main fix in `src/config/sessions` that treats literal agent-layout stores as scoped multi-agent roots, keeps true shared literal stores single-store, and covers sibling discovery, scoping, resolved paths, and legacy key canonicalization.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store, then call `sessions.list`; current-main source shows the non-template branch returns one store before sibling discovery, and the linked artifact reports `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
