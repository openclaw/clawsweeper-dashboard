## Summary

Keep open. The reported bug and proof are valid, but this PR cannot merge as-is because its production change is on the stale gateway utility surface while current `sessions.list` uses the active `src/config/sessions` loader, and the PR head still has resolved-path and agent-id correctness gaps.

## What This Changes

The PR changes session-store aggregation so literal `agents/<id>/sessions/sessions.json` store paths discover sibling per-agent session stores and adds regression coverage for that case.

## Best Possible Solution

Port literal agent-layout handling into `src/config/sessions` so same-root sibling discovery, resolved store paths, and on-disk agent IDs are handled by the active loader while true shared literal stores remain single-store paths.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source still returns from the literal-store branch before sibling discovery, and the linked proof shows `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended fix direction is right, but it needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
