## Summary

Keep open. The bug and proof are credible, but the PR is not correct against current main because it patches a stale gateway-local loader while the active `sessions.list` path now lives in `src/config/sessions`, and the remaining path/id issues are concrete merge blockers.

## What This Changes

The PR changes literal `session.store` handling in `src/gateway/session-utils.ts` so per-agent layout paths discover sibling session stores, with regression tests in `src/gateway/session-utils.subagent.test.ts`.

## Best Possible Solution

Port the fix to the active `src/config/sessions` loader and target resolver, preserving true shared literal stores as single-store paths while agent-layout literal stores discover same-root siblings with resolved paths and correct on-disk agent IDs.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source still returns from the literal-store branch before sibling discovery, and the linked proof shows `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended fix direction is right, but it needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
