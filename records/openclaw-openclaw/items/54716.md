## Summary

Keep open: the PR addresses a real `sessions.list` bug with sufficient before/after proof, but the current patch is still aimed at stale gateway-local loader code and misses current-main path/id semantics needed before merge.

## What This Changes

The PR changes session-store aggregation and regressions so literal `agents/<id>/sessions/sessions.json` store paths discover sibling agent stores while non-agent-layout literal stores remain single-store paths.

## Best Possible Solution

Land a current-main fix in `src/config/sessions` that treats literal agent-layout stores as scoped multi-agent roots, preserves true shared literal stores as single-store paths, and adds regressions for sibling discovery, scoping, resolved paths, and legacy key canonicalization.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store, then call `sessions.list`; current-main source shows the active combined loader returns before sibling discovery, and linked proof shows `apiCount` moving from 1 to 2 after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
