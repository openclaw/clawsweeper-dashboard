## Summary

Keep open: the reported bug is real and the proof is useful, but this PR still patches the stale gateway-local loader while current main uses the `src/config/sessions` loader, so the branch is not merge-ready as-is.

## What This Changes

The PR changes gateway session aggregation and tests so literal `agents/<id>/sessions/sessions.json` `session.store` paths discover sibling agent stores instead of only the configured default store.

## Best Possible Solution

Port the proof-backed behavior into `src/config/sessions`, keeping true shared literal stores single-store while literal `agents/<id>/sessions/sessions.json` paths discover same-root sibling stores with resolved paths and on-disk agent IDs.

## Reproduction Assessment

Yes, source-reproducible. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current main returns from the literal-store branch before sibling discovery, and the linked proof shows count 1 before versus count 2 after.

## Solution Assessment

No, not as currently patched. The behavior is the right narrow fix, but the implementation must move to the active `src/config/sessions` loader/resolver and preserve same-root scoping, resolved store paths, and on-disk agent IDs.
