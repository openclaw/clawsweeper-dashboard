## Summary

Keep open: the reported bug is still present on current main and the PR has useful real before/after proof, but this branch patches the old gateway-local loader while current main uses the active `src/config/sessions` loader and the PR is currently conflicting.

## What This Changes

The PR changes literal `session.store` handling so `agents/<id>/sessions/sessions.json` paths discover sibling agent stores, and adds gateway regression tests for that case.

## Best Possible Solution

Port the proven fix into `src/config/sessions/combined-store-gateway.ts` and `src/config/sessions/targets.ts`, preserving true shared literal stores as single-store paths while literal agent-layout stores discover only same-root sibling stores with regression coverage.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` with a sibling agent store, then call `sessions.list`; current-main source returns from the single-store branch before sibling discovery, and the linked artifacts show the API improving from one returned session to two after the branch fix.

## Solution Assessment

No, not as currently patched. The direction is right, but the implementation needs to move into the active `src/config/sessions` loader and resolver while preserving resolved store paths and on-disk agent-id canonicalization.
