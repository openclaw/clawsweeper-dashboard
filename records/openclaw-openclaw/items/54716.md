## Summary

Keep open. The PR targets a real sessions.list bug with sufficient before/after proof, but current main now routes the active loader through src/config/sessions, so the branch is conflicting and patches a stale gateway-local implementation.

## What This Changes

This PR changes gateway session-store aggregation and regression tests so literal agents/<id>/sessions/sessions.json store paths discover sibling agent session stores under the same root.

## Best Possible Solution

Land a current-main fix in src/config/sessions that treats literal agent-layout stores as scoped multi-agent roots while preserving true shared literal stores as single-store paths.

## Reproduction Assessment

Yes. Configure session.store to a literal agents/main/sessions/sessions.json path with a sibling agent store, then call sessions.list; current-main source still returns from the literal single-store branch before sibling discovery, and the linked API artifact shows the count improving from one to two after the branch change.

## Solution Assessment

No, not as currently patched. The direction is right, but the fix needs to move into the active src/config/sessions loader and resolver, preserve the resolved literal path, and canonicalize with the on-disk agent id.
