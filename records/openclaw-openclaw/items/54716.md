## Summary

Keep this PR open: it is a useful, proof-backed candidate for the linked sessions listing bug, but it is not merge-ready because listed sibling sessions can still be unresolved by key-based preview/history paths and the live branch is conflicting with main.

## What This Changes

The PR changes Gateway session-store discovery so literal `session.store` paths under `agents/<id>/sessions/sessions.json` enumerate sibling agent stores, derive agent ownership from paths, and add regression tests.

## Best Possible Solution

Land a refreshed fix where `sessions.list`, preview/history, and key-based session lookup share one scoped literal agent-tree discovery path, while true shared literal stores remain single-store.

## Solution Assessment

No, not yet. The listing-side direction is right, but the best fix also updates key-based lookup so newly listed sibling sessions can be previewed, opened, and read from the same backing store.
