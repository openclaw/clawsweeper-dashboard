## Summary

Keep this PR open. The linked session-listing bug remains valid on current main, and the contributor provided useful before/after proof, but the branch is not merge-ready because the proposed implementation still has concrete correctness blockers and now targets an older file layout.

## What This Changes

The PR changes gateway session-store aggregation so literal `agents/<id>/sessions/sessions.json` paths discover sibling per-agent stores under the same agents root, with regression coverage for that layout.

## Best Possible Solution

Land an equivalent current-main fix in the session config loader/target resolver: literal agent-layout stores should discover sibling stores only under the same agents root, preserve the resolved literal store path, derive the agent ID from each disk layout, and keep true shared literal stores on the existing single-store path.

## Reproduction Assessment

Yes. Configure `session.store` as a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store on disk; current main returns before discovery, and the PR discussion reports API-backed `sessions.list` proof changing from `count=1` to `count=2`.

## Solution Assessment

No, not as written. The direction is right, but the patch needs to be adapted to the current-main session config files and must fix path preservation plus literal-layout agent identity before it is the narrow maintainable solution.
