## Summary

Keep open. The PR targets a valid sessions.list bug and includes useful before/after proof, but current main still has the literal-store early return and this branch is conflicting while patching a gateway-local loader implementation that current main no longer uses.

## What This Changes

The PR changes gateway session-store aggregation and tests so literal `agents/<id>/sessions/sessions.json` store paths discover sibling agent stores under the same agents root while leaving shared literal stores as single-store paths.

## Best Possible Solution

Land a current-main fix in `src/config/sessions` that treats literal agent-layout stores as scoped multi-agent roots, keeps true shared literal stores single-store, and adds regression coverage for sibling discovery, scoping, resolved paths, and legacy key canonicalization.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store, then call `sessions.list`; current-main source shows the non-template branch returns one store before sibling discovery, and the linked artifact reports `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
