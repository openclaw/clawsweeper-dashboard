## Summary

Keep open. The bug and after-fix proof are credible, but this PR is conflicting/outdated and still patches the old gateway-local loader while current main routes `sessions.list` through the active `src/config/sessions` loader.

## What This Changes

The PR attempts to make literal `session.store` paths under `agents/<id>/sessions/sessions.json` discover sibling agent stores and adds gateway regression coverage.

## Best Possible Solution

Land a current-main fix in `src/config/sessions` that treats literal agent-layout stores as scoped multi-agent roots while preserving true shared literal stores as single-store paths.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` with a sibling agent store, then call `sessions.list`; current-main source still returns from the literal single-store branch before sibling discovery, and the linked artifacts show the API count improving from one to two after the branch change.

## Solution Assessment

No, not as currently patched. The direction is right, but the implementation must move into the active `src/config/sessions` loader and resolver, preserve resolved literal paths, and canonicalize with the on-disk agent id.
