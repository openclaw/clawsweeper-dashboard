## Summary

Keep open. The linked bug still exists on current main, and the branch is useful but not merge-ready because the active loader has moved and the proposed multi-store path still needs path-preservation, agent-id, and changelog fixes.

## What This Changes

The PR changes gateway session-store aggregation so literal `agents/<id>/sessions/sessions.json` paths discover sibling per-agent stores under the same agents root, with regression tests for the reported layout.

## Best Possible Solution

Land an equivalent current-main fix in the session loader and target resolver that treats literal agent-layout stores as scoped multi-agent roots while preserving true shared literal-store behavior.

## Reproduction Assessment

Yes. Configure `session.store` as a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store on disk; current main returns from the literal single-store branch before sibling discovery, and the PR discussion includes API-backed `count=1` to `count=2` proof.

## Solution Assessment

No, not as written. The direction is right, but the maintainable fix should be applied to the current `src/config/sessions` implementation, preserve resolved literal store paths, scope discovery to the literal agents root, and derive agent ids from the disk layout.
