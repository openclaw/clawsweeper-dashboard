## Summary

Keep open: the reported `sessions.list` bug is valid and the linked before/after proof is sufficient, but this PR is not ready because it patches a stale gateway-local loader surface and still misses resolved-path plus on-disk-agent-id behavior needed by current main.

## What This Changes

The PR updates gateway session-store aggregation and tests so literal `agents/<id>/sessions/sessions.json` store paths discover same-root sibling agent stores while shared literal stores stay single-store paths.

## Best Possible Solution

Port the literal agent-layout handling into `src/config/sessions` so same-root sibling stores are discovered, true shared literal stores remain single-store, resolved store paths are preserved, and legacy keys are canonicalized with the on-disk agent id.

## Reproduction Assessment

Yes, by source inspection: configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store, then call `sessions.list`; current main returns from the literal-store branch before sibling discovery. The linked artifact also shows the real API count changing from 1 before the branch to 2 after it.

## Solution Assessment

No, not as currently patched. The intended behavior is the right narrow fix, but it needs to land in the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
