## Summary

Keep this PR open: current main still has the literal `session.store` early-return bug, and the PR targets the right user problem, but the patch is not merge-ready because it misses resolved-path preservation, can canonicalize literal-layout stores under the wrong agent id, lacks the required changelog entry, and now needs adaptation to the current `src/config/sessions/*` split.

## What This Changes

The PR changes gateway session-store aggregation so literal `agents/<id>/sessions/sessions.json` paths discover sibling per-agent stores and adds regression coverage for that layout.

## Best Possible Solution

Land an equivalent current-main fix in the session combined-store/target resolver path: literal agent-layout stores should discover siblings only under the same real agents root, preserve the resolved literal path for downstream lookups, derive agent identity from the disk layout, and keep true shared literal stores on the single-store path.

## Reproduction Assessment

Yes. Configure `session.store` as a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store on disk, then call `sessions.list`; current main reads only the literal default-agent store and returns before sibling discovery, and the PR reports API-backed `count=1` to `count=2` proof.

## Solution Assessment

No, not as written. The direction is narrow and maintainable, but the patch needs to be adapted to current `src/config/sessions/*` files and must fix resolved-path propagation, literal-layout agent identity, and changelog coverage before merge.
