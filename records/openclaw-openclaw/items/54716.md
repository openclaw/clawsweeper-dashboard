## Summary

Keep open: the intended bug fix is valid and has useful before/after proof, but the branch still patches the old gateway-local loader while current main drives `sessions.list` through `src/config/sessions`, and the PR has concrete path/canonicalization gaps that should block merge.

## What This Changes

The PR changes gateway session-store aggregation and regression tests so literal `agents/<id>/sessions/sessions.json` store paths discover same-root sibling agent stores while shared literal stores remain single-store paths.

## Best Possible Solution

Land a current-main fix in `src/config/sessions` that treats literal agent-layout stores as scoped multi-agent roots, preserves true shared literal stores as single-store paths, and adds regressions for sibling discovery, scoping, resolved paths, and legacy key canonicalization.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store, then call `sessions.list`; current-main source shows the non-template branch returns one store before sibling discovery, and the linked artifact reports `count=1` before versus `count=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to move into the active `src/config/sessions` loader/resolver and preserve resolved-path plus on-disk-agent-id semantics.
