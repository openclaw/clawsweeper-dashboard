## Summary

Keep open. Current main still short-circuits literal non-template session stores before sibling agent discovery, so the linked bug remains valid; this PR is a useful narrow fix attempt, but it is currently dirty against main and has concrete P2 correctness issues before it can merge.

## What This Changes

The PR changes gateway session-store aggregation so literal `agents/<id>/sessions/sessions.json` paths discover sibling per-agent stores under the same agents root, with regression tests for the literal-path case.

## Best Possible Solution

Land an equivalent current-main fix in `src/config/sessions/combined-store-gateway.ts` and, if needed, `src/config/sessions/targets.ts`: literal paths matching `agents/<id>/sessions/sessions.json` should discover sibling stores under that same agents root, preserve the resolved literal store path for downstream file lookup, derive literal-layout agent IDs from on-disk paths, and keep true shared literal stores on the existing single-store path.

## Reproduction Assessment

Yes. A high-confidence reproduction is to configure `session.store` as a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store on disk; current main's loader returns before discovery, while the PR discussion reports API-backed `sessions.list` proof changing from `count=1` to `count=2`.

## Solution Assessment

No, not as-is. The direction is right, but the PR needs to be adapted to the current-main files and must fix the resolved-path and agent-ID canonicalization issues before it is the narrow maintainable solution.
