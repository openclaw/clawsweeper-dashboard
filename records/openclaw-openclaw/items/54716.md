## Summary

Keep open: the reported `sessions.list` bug is real and the contributor proof is sufficient, but this PR still patches the stale gateway-local implementation while current `main` uses the active `src/config/sessions` loader that still short-circuits literal stores.

## What This Changes

The PR changes gateway session aggregation and regression tests so literal `session.store` paths under `agents/<id>/sessions/sessions.json` discover same-root sibling agent stores while preserving single-store behavior for other literal paths.

## Best Possible Solution

Land an equivalent current-main repair in `src/config/sessions/combined-store-gateway.ts` and `src/config/sessions/targets.ts` so literal agent-layout stores discover only same-root sibling stores while true shared literal stores remain single-store.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; source inspection shows current `main` returns after reading only the literal default-agent store, and the linked proof shows the API count improving from 1 to 2 on the branch.

## Solution Assessment

No, not as written. The direction is right, but the fix must move to the active `src/config/sessions` loader and resolver while preserving resolved paths and disk-derived agent IDs.
