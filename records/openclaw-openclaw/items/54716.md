## Summary

Keep open: the reported bug still exists on current main, but this PR is not merge-ready because it patches the stale gateway-local loader instead of the active `src/config/sessions` implementation and still needs path/agent-id correctness repairs.

## What This Changes

The PR changes gateway session-store aggregation so literal per-agent `session.store` paths discover sibling agent stores under the same on-disk agents root.

## Best Possible Solution

Port the literal agent-layout handling into `src/config/sessions/combined-store-gateway.ts` and the target resolver so same-root sibling stores are discovered, true shared literal stores remain single-store paths, resolved store paths are preserved, and legacy keys use the on-disk agent id.

## Reproduction Assessment

Yes from source inspection. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store, then call `sessions.list`; current main returns from the literal-store branch before sibling discovery, and the linked proof shows `apiCount=1` before versus `apiCount=2` after the branch.

## Solution Assessment

No, not as currently patched. The intended behavior is correct, but the fix needs to move into the active `src/config/sessions` loader/target resolver and preserve resolved-path plus on-disk-agent-id semantics.
