## Summary

Keep open: the bug is valid and the intended behavior is useful, but this PR is still incorrect against current main because it edits the old gateway-local loader while the active runtime loader now lives under `src/config/sessions`, and the remaining path/agent-id edge cases need repair before merge.

## What This Changes

The PR changes session-store aggregation so literal `agents/<id>/sessions/sessions.json` paths discover sibling agent stores, with regression tests for that layout.

## Best Possible Solution

Move literal agent-layout handling into `src/config/sessions` so same-root sibling stores are discovered, true shared literal stores remain single-store, resolved paths are preserved, and agent IDs come from the on-disk store path when appropriate.

## Reproduction Assessment

Yes. Configure `session.store` to a literal `.../agents/main/sessions/sessions.json` path with a sibling agent store and call `sessions.list`; current-main source returns from the literal-store branch before sibling discovery, and the linked proof shows one row before versus two after.

## Solution Assessment

No, not as currently patched. The intended behavior is right, but the fix needs to target the active `src/config/sessions` loader and resolver while preserving resolved-path and on-disk-agent-id semantics.
