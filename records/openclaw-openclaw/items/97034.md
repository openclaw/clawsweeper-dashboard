## Summary

Keep open: this is a focused, still-useful bug-fix PR for a current-main gap in the session-memory sanitizer. I found no blocking patch findings; the broader related issue should remain open because this PR only prevents new orphan role-line persistence and does not handle already-poisoned memory files or the full structural validation policy.

## What This Changes

The PR adds a session-memory sanitizer rule and regression tests so role-only `user`, `assistant`, or `system` lines are dropped before transcript content is persisted.

## Best Possible Solution

Land the narrow sanitizer and tests if CI remains clean, while keeping the canonical issue open for recovery and broader persistence-boundary policy decisions.

## Reproduction Assessment

Yes, at source level. Current main runs the sanitizer before memory persistence, and the existing replacement pipeline has no plain role-only line rule, so a transcript message containing `user` or `assistant` on its own line survives into `sessionContent`.

## Solution Assessment

Yes for this PR's scope. The shared transcript sanitizer is the right narrow boundary because the handler and reset fallback consume `getRecentSessionContent`; it is not a complete solution for existing poisoned memory files.
