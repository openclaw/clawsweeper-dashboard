## Summary

Keep this PR open: the reported memory-core failure is plausible and the PR now includes real after-fix proof, but the current patch can reopen the target sqlite DB while safe reindex is in its close-before-swap window.

## What This Changes

The PR adds closed-sqlite detection and reopen logic to the memory-core index manager plus a regression test that closes the underlying handle before status/search.

## Best Possible Solution

Keep the closed-handle recovery, but make it sync-aware so status/search/sync cannot reopen the target DB during atomic reindex, then cover both the recovery path and the reindex race with tests and a changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: I did not run a failing current-main status/search scenario in this read-only review. Source inspection shows the gap, and the PR adds a regression that closes the handle before status/search.

## Solution Assessment

Is this the best way to solve the issue? No: reopen-on-use is the right narrow direction for a closed cached handle, but the reopen must be coordinated with the sync lifecycle so it cannot interfere with atomic file swaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Gate sqlite reopen during atomic reindex:** `extensions/memory-core/src/memory/manager.ts:276`
  - body: `runSafeReindex` closes the temp and original handles before `runMemoryAtomicReindex` renames the target files, then the sync owner reopens the target after the swap. This helper can run from status/search/sync in that gap and reopen `settings.store.path`, leaving a live handle on the target during rename and leaking it when the sync owner assigns a fresh handle. Gate recovery while `this.syncing` is active or keep reopening in the sync owner after the swap.
