## Summary

Keep open: the PR targets a real memory-core closed-sqlite failure and includes live after-fix output, but the current patch can reopen the target database during the existing atomic reindex swap and also lacks the required user-facing changelog entry.

## What This Changes

The PR adds closed-sqlite detection and reopen-on-use logic to the memory-core manager, evicts closed cached managers, and adds a regression test for status/search after closing the underlying handle.

## Best Possible Solution

Keep the closed-handle recovery, but make it sync-aware so status/search/sync cannot reopen the target DB during atomic reindex, then cover the recovery and race paths with tests plus a changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: I did not run a failing current-main status/search scenario, but source inspection shows current main can keep using a cached DatabaseSync handle and the PR adds a regression that closes that handle before status/search.

## Solution Assessment

Is this the best way to solve the issue? No: reopen-on-use is the right narrow direction, but it must be coordinated with the sync lifecycle so recovery cannot interfere with the atomic file swap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Gate sqlite reopen during atomic reindex:** `extensions/memory-core/src/memory/manager.ts:273`
  - body: `runSafeReindex` deliberately closes the temp and original handles before renaming the target files, then reopens the target after the swap. This helper can run from status/search/sync in that gap and reopen `settings.store.path`, leaving a live handle on the target during rename and leaking it when the sync owner assigns a fresh handle. Gate closed-handle recovery while `this.syncing` is active, or keep the reopen solely in the sync owner after the swap.
  - confidence: 0.86
