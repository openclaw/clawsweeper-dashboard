## Summary

Keep open: the PR targets a real memory-core closed-sqlite failure and includes real behavior proof, but the current patch can reopen a target sqlite handle during the atomic reindex close/swap window and still needs a changelog entry.

## What This Changes

The PR adds memory-core closed-sqlite-handle detection and reopen logic around cached manager status/search/sync paths, plus a regression test that closes the manager DB handle.

## Best Possible Solution

Land a sync-aware memory-core closed-handle recovery that avoids reopening during atomic reindex, keeps the regression coverage, and documents the user-facing fix in the changelog.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at the source level: current main caches MemoryIndexManager instances and uses their DatabaseSync handles directly, and a closed node:sqlite handle on Node v24.14.1 throws ERR_INVALID_STATE with `database is not open`. I did not run a full current-main OpenClaw memory_search scenario in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: reopen-on-use is the right narrow direction, but the implementation should gate recovery while `this.syncing` owns the atomic reindex lifecycle or let the sync owner perform the reopen after the swap.

## Review Findings
