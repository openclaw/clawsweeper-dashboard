## Summary

Keep open: the PR addresses a plausible memory-core closed-sqlite bug and includes real live proof, but it is not merge-ready because the reopen hook can race safe reindex, the branch edits a generated schema file removed on current main, and the user-facing fix lacks a changelog entry.

## What This Changes

The PR adds closed-sqlite detection and reopen-on-use logic for memory-core status/search/sync, a regression test, and a generated config schema version bump.

## Best Possible Solution

Land a sync-aware closed-handle recovery in memory-core that defers or gates reopen during atomic reindex, keeps current runtime-computed schema behavior, and documents the user-facing fix in the changelog.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at the source level: a closed node:sqlite DatabaseSync handle throws `database is not open`, and current main status/search use the cached handle without an open check. I did not run a full live OpenClaw failure scenario in this read-only sweep.

## Solution Assessment

Is this the best way to solve the issue? No: reopen-on-use is the right narrow direction, but the implementation must respect the sync/atomic-reindex lifecycle and be rebased onto current main without the obsolete generated schema edit.
