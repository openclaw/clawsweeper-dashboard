## Summary

Keep open: the PR targets a plausible memory-core closed-handle failure, but it is draft, lacks real after-fix behavior proof, and the patch introduces a concrete atomic-reindex race risk.

## What This Changes

The PR adds closed-sqlite detection and reopen logic to the memory-core `MemoryIndexManager`, plus a regression test that closes the underlying handle before status/search.

## Best Possible Solution

Recover closed cached builtin memory DB handles from the manager owner path without reopening during an in-flight atomic reindex, then add regression coverage and real status/search proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: source inspection plus a Node `DatabaseSync` probe show the current-main failure mode is plausible, but I did not run the full memory-core scenario or the PR's regression test.

## Solution Assessment

Is this the best way to solve the issue? No: the direction is narrow, but synchronous reopen from status/search can race with safe reindex's close-before-rename swap; recovery should be coordinated with the sync lifecycle or manager replacement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid reopening during atomic reindex swaps:** `extensions/memory-core/src/memory/manager.ts:273`
  - body: `runSafeReindex` closes the temp and original handles before `runMemoryAtomicReindex` renames the target DB/WAL/SHM files, then reopens the target after the swap. If `status()` or `search()` hits this new helper in that await gap, it will reopen the target path and leave that handle open during the rename path, defeating the close-before-swap behavior and making transient `EBUSY`/`EPERM` swap failures or leaked handles likely. Gate this recovery while `this.syncing` is active, or perform the reopen only in the sync owner after the atomic swap.
  - confidence: 0.86
