## Summary

Keep open: current main has an intentional same-target foreground freshness fence that can cancel final reply delivery before Slack send, and Slack then returns without an error when no visible reply was delivered. That matches the reported transcript-composed/no-Slack-post symptom, but the fix needs a maintainer policy call because it overlaps with recently merged stale-reply suppression behavior.

## What This Changes

The issue requests Slack delivery reliability so completed channel and DM replies are posted, fail visibly, or are preserved instead of silently disappearing when new inbound Slack messages arrive mid-turn.

## Best Possible Solution

Define and implement a non-silent foreground delivery policy: Slack same-target finals should either drain before the queued follow-up, emit an explicit suppressed/failed delivery diagnostic with preserved context, or be folded into the next turn under a documented stale-reply rule.

## Reproduction Assessment

Yes, source-level: current main already has a regression test that suppresses an older same-target final after a newer inbound starts, and Slack routes through that buffered dispatcher. I did not replay the reporter's private Slack workspace/session.

## Solution Assessment

Unclear as currently implemented: the existing fence is the right mechanism for stale-reply suppression, but using it to silently cancel Slack finals after transcript completion is not an acceptable delivery outcome. The safer direction is to make cancellation observable and preserve or serialize the reply according to a maintainer-chosen policy.
