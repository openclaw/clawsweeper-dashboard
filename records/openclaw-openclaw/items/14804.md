## Summary

Keep open. Current main still lacks a first-class ephemeral or tmpfs-backed sandbox workspace lifecycle, and open PR #73738 directly references this issue with closing syntax, so the right path is maintainer/secops review of that PR rather than cleanup closure.

## What This Changes

Issue requests an opt-in ephemeral or tmpfs-backed sandbox workspace so isolated sessions start cleanly and do not persist file changes across runs.

## Best Possible Solution

Review and either land or close PR #73738 after confirming a backwards-compatible opt-in lifecycle contract, persistent defaults, cleanup constrained to run-owned ephemeral paths, aligned config/schema/generated metadata/docs/tests, and an explicit decision on tmpfs-backed workspace scope.

## Reproduction Assessment

Yes for the current-main feature gap; this is not a runtime crash reproduction. The focused check is that workspaceLifecycle is absent from the config type and strict schema while sandbox workspace resolution still derives stable per-scope directories that preserve existing contents.

## Solution Assessment

Unclear until PR #73738 is reviewed. The maintainable direction is an opt-in lifecycle setting with persistent defaults and tightly bounded cleanup semantics; the safer next step is review of the linked PR, not a separate automated fix.
