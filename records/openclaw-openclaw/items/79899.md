## Summary

Keep open: current main has a partial resource-loader fix, but source and dependency inspection show `DefaultResourceLoader.reload()` still calls Pi's package-manager discovery before the `noSkills`/`noThemes` style flags can suppress loading, so the reported synchronous filesystem scan is not proven fixed.

## What This Changes

The issue reports that Pi's `DefaultResourceLoader.reload()` performs synchronous resource discovery on Windows and blocks the OpenClaw Gateway event loop during embedded agent startup.

## Best Possible Solution

Keep this open and either add an OpenClaw-owned embedded resource-loader path or coordinate an upstream Pi option that skips package-manager discovery entirely when OpenClaw supplies inline extensions and its own resource prompts, then validate on Windows with event-loop timing proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The issue has concrete Windows timing logs, and source/dependency inspection shows the current embedded path still reaches synchronous Pi package-manager discovery during `reload()`.

## Solution Assessment

No: the current main fix is a useful step, but passing `noSkills`/`noThemes` style flags is not sufficient because the dependency resolves package resources before those flags filter the results. The maintainable fix should remove or bypass that discovery in the embedded OpenClaw path, or add an upstream skip-discovery contract.
