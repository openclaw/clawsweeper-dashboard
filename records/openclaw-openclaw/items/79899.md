## Summary

Keep open: current main and the latest release still route embedded attempts through `DefaultResourceLoader.reload()` and synchronous resource discovery, while the related fast-path PR was closed unmerged.

## What This Changes

The issue reports Windows/NTFS event-loop stalls during `DefaultResourceLoader.reload()` in embedded agent prep because resource discovery still performs synchronous filesystem scanning.

## Best Possible Solution

Land a loader-owned fast path for embedded attempts and compaction that avoids ambient discovery while preserving settings reload semantics, compaction guard reapplication, inline extension factories, diagnostics/source metadata, and Windows large-tree timing proof.

## Reproduction Assessment

No, not for the exact Windows/NTFS timing in this review. Source inspection shows the current-main blocking-prone path still runs through `resourceLoader.reload()` and synchronous package discovery, while the reporter supplied concrete timing logs.

## Solution Assessment

Unclear. Avoiding unnecessary embedded-run discovery is the right direction, but the best fix should be a supported loader-owned fast path rather than user-facing skip config or private-field shortcuts.
