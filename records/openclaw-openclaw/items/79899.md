## Summary

Keep open: current main still awaits Pi `DefaultResourceLoader.reload()` on embedded run and compaction startup, and the pinned Pi package source still performs broad synchronous filesystem discovery before OpenClaw can filter resource types.

## What This Changes

The issue reports synchronous Pi resource discovery blocking the OpenClaw Gateway event loop on Windows during embedded agent startup.

## Best Possible Solution

Design an OpenClaw-owned embedded resource-loader boundary, or coordinate an upstream Pi cached/async discovery fix, while preserving inline extension factories, OpenClaw skills prompt behavior, and compaction guards with Windows proof.

## Reproduction Assessment

Unclear for live reproduction: the issue includes concrete timing logs and current source confirms the dependency path, but this review did not rerun a Windows/NTFS large-tree scenario. Source inspection does provide a credible reproduction path through embedded agent startup and compaction startup.

## Solution Assessment

Unclear: exposing Pi skip flags alone is probably not the best fix because dependency discovery runs before those filters. The narrower maintainable direction is an OpenClaw embedded loader boundary or upstream Pi cache/async change, validated against the reported Windows Gateway responsiveness problem.
