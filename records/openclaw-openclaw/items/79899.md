## Summary

Keep open: current main still constructs the embedded Pi resource loader and awaits `reload()` in the normal attempt and compaction paths, while the pinned Pi dependency still runs synchronous package/resource discovery before the OpenClaw `no*` flags filter results. The linked fix attempt at https://github.com/openclaw/openclaw/pull/82536 is relevant, but it is still open and previously reviewed as needing real Windows proof plus a safer public-contract loader boundary.

## What This Changes

The issue reports multi-second Windows event-loop stalls from Pi `DefaultResourceLoader.reload()` during embedded agent startup and asks for caching, async discovery, or exposed skip controls.

## Best Possible Solution

Keep this as the canonical resource-loader stall issue and fix it with a public-contract fast path or upstream Pi change that avoids package discovery while preserving settings reload, inline extensions, source info, and real Windows large-tree timing proof.

## Reproduction Assessment

No high-confidence live reproduction was established on current main. The source path is clear and the reporter supplied fresh 2026.5.12 Windows timing, so this is source-reproducible but still needs Windows/large-tree proof for the final fix.

## Solution Assessment

Unclear as proposed: config skip flags alone do not avoid the dependency `packageManager.resolve()` call, and the current open PR's private-internal shortcut is not the best maintainable shape. The safer direction is a public fast path or upstream Pi contract with real timing proof.
