## Summary

Keep open: current main still awaits Pi's `DefaultResourceLoader.reload()` in the embedded run and compaction startup paths, and the pinned Pi dependency still performs synchronous recursive resource discovery. The existing ClawSweeper comment matches this re-check; related open performance issues do not specifically own this Windows `session-resource-loader` scan.

## What This Changes

The issue asks OpenClaw to stop Pi `DefaultResourceLoader.reload()` from blocking agent startup on Windows by caching or asyncing discovery, or by exposing resource skip controls.

## Best Possible Solution

Design a narrow embedded loader or upstream Pi cached/async discovery fix that preserves OpenClaw extension factories and skills prompt behavior, then validate it on Windows/NTFS with a large resource tree.

## Reproduction Assessment

Unclear for live reproduction: the report has concrete Windows timing logs, and source inspection confirms current main still calls the synchronous Pi discovery path. I did not run a current-main Windows/NTFS reproduction with the reporter's large resource tree.

## Solution Assessment

Unclear: exposing Pi skip flags alone is probably not the best fix because `packageManager.resolve()` runs before those filters. The safer direction is an OpenClaw-owned embedded loader boundary or an upstream Pi cache/async discovery change with Windows proof.
