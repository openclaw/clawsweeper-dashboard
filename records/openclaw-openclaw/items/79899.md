## Summary

Keep open: current main has a partial embedded-loader skip-flag wrapper, but embedded runs still await Pi `DefaultResourceLoader.reload()`, and the pinned Pi 0.74.0 source still performs synchronous auto-discovery before those flags filter resources. I did not find a merged upstream/OpenClaw fix or live Windows proof showing the reported 12-15s stalls are gone.

## What This Changes

The issue reports Windows event-loop stalls during embedded agent startup because Pi `DefaultResourceLoader.reload()` performs synchronous resource discovery over large OpenClaw and skills trees.

## Best Possible Solution

Keep this open and design a narrow loader-boundary or upstream Pi discovery fix that avoids synchronous auto-discovery for embedded runs while preserving OpenClaw inline extensions, then validate with real Windows timing proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The source path is clear on current main and the issue includes concrete Windows timing logs, but the stall was not re-run on a large Windows/NTFS tree here.

## Solution Assessment

Unclear: the current skip-flag wrapper is not enough by itself because Pi resolves and auto-discovers resources before applying those flags. The safer solution is an upstream cached/async discovery change or an OpenClaw-owned embedded loader boundary with Windows proof.
