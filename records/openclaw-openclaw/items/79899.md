## Summary

Keep open: current main still calls Pi's `DefaultResourceLoader.reload()` in embedded agent startup, and the dependency source confirms broad synchronous filesystem discovery that can plausibly block the Gateway event loop on large Windows trees.

## What This Changes

The issue asks OpenClaw to avoid or mitigate slow synchronous Pi resource discovery during every embedded agent run on Windows.

## Best Possible Solution

Keep this open and design a narrow OpenClaw-owned embedded resource-loader path, or coordinate an upstream Pi cached/async discovery fix, with Windows proof that Gateway responsiveness and inline compaction/context-pruning extensions are preserved.

## Reproduction Assessment

Unclear for live reproduction: the issue includes concrete timing logs and the source path is clear, but I did not reproduce the Windows/NTFS stall on current main. Source inspection confirms current main still calls the dependency reload path and the dependency uses synchronous broad filesystem discovery.

## Solution Assessment

Unclear: a config-only skip-flag approach is probably not the best fix because upstream discovery scans before those flags filter resources. The safer maintainable direction is an OpenClaw embedded loader boundary or upstream Pi discovery cache/async change, validated against real Windows Gateway behavior.
