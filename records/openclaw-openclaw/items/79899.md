## Summary

Keep open: current main and the latest release still route embedded runs through `DefaultResourceLoader.reload()`, and `reload()` still calls `DefaultPackageManager.resolve()` before later filtering disabled resource types. The related open PR at https://github.com/openclaw/openclaw/pull/82536 is not a resolved or safe canonical fix yet because it is stale against current paths and still needs proof plus loader-contract review.

## What This Changes

The issue reports that embedded agent runs block the Windows event loop because resource-loader reload performs synchronous filesystem discovery, and asks for caching, async scanning, or skip behavior.

## Best Possible Solution

Add a maintainer-approved embedded resource-loader fast path that avoids ambient discovery while preserving settings reload semantics, inline extension factories, compaction guards, diagnostics, and Windows large-tree timing proof.

## Reproduction Assessment

No, not for the exact Windows/NTFS timing. Source inspection gives a current-main path through `resourceLoader.reload()` and `packageManager.resolve()`, but a live Windows large-tree benchmark is still missing.

## Solution Assessment

Unclear. Avoiding unnecessary embedded-run discovery is the right direction, but the durable solution should be a supported loader fast path rather than a broad new config surface or private-state shortcut.
