## Summary

Keep open: current main still awaits Pi's `DefaultResourceLoader.reload()` during embedded run and compaction startup, and the published dependency source confirms broad synchronous filesystem discovery before OpenClaw can filter resource types.

## What This Changes

This issue asks OpenClaw to prevent Pi resource discovery from blocking the Gateway event loop during embedded agent startup on large Windows resource trees.

## Best Possible Solution

Design a narrow embedded resource-loader boundary or coordinate an upstream Pi discovery cache/async fix, then prove it on Windows with large resource trees while preserving OpenClaw extension and skill behavior.

## Reproduction Assessment

Unclear for high-confidence live reproduction: the issue includes concrete Windows timing logs, and source inspection confirms the current blocking path, but I did not reproduce the 12-15s Windows/NTFS stall on current main.

## Solution Assessment

Unclear: exposing skip flags alone is likely not the best fix because Pi resolves and scans resources before those flags filter loaded resource types. A safer maintainable path is a cached/async upstream loader or an OpenClaw-owned embedded loader boundary with targeted compatibility tests.
