## Summary

Keep open: current main and the latest release still route embedded attempts and compaction through `DefaultResourceLoader.reload()`, and that reload still resolves resource/package discovery before the disabled-resource filters take effect. The linked open PR is a candidate direction, but it is unmerged, conflicting, and still needs loader-contract review plus Windows timing proof.

## What This Changes

The issue requests eliminating Windows event-loop stalls from embedded `DefaultResourceLoader.reload()` resource discovery during agent runs.

## Best Possible Solution

Add a maintainer-approved embedded resource-loader fast path that avoids ambient discovery while preserving settings reload semantics, inline extension factories, compaction guards, diagnostics, and Windows large-tree timing proof.

## Reproduction Assessment

No high-confidence live reproduction was run for the exact Windows/NTFS timing. Source inspection gives a clear current-main path through embedded `resourceLoader.reload()` and synchronous package discovery.

## Solution Assessment

Unclear. Avoiding unnecessary embedded discovery is the right direction, but the maintainable fix should be a supported loader fast path rather than broad new config or a private-state shortcut.
