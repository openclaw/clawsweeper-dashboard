## Summary

Keep open: current main has an attempted embedded-loader mitigation, but it still awaits Pi `DefaultResourceLoader.reload()`, and the pinned Pi loader still runs synchronous package/resource discovery before the `no*` filters take effect.

## What This Changes

The issue reports Windows event-loop stalls from synchronous Pi resource discovery during embedded agent startup.

## Best Possible Solution

Keep this canonical issue open and either add an OpenClaw-owned embedded loader path that avoids upstream package/resource discovery when all Pi resources are disabled, or coordinate an upstream Pi fix, with Windows before/after timing proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection confirms current main still reaches a dependency path that synchronously scans the reported resource directories before filtering them, and the issue includes concrete Windows timing logs.

## Solution Assessment

Unclear: the current shipped `no*` wrapper is a useful mitigation but not sufficient because Pi `PackageManager.resolve()` still scans first. A config-only skip-flag surface is not the best fix unless the loader or upstream dependency stops discovery before the synchronous scan.
