## Summary

Keep open: current main still awaits Pi's `DefaultResourceLoader.reload()` during embedded agent startup, and the published Pi dependency still performs broad synchronous filesystem discovery before resource skip flags are applied. The report is credible, but I did not live-reproduce the Windows/NTFS stall, and the fix crosses the OpenClaw/Pi resource-loader contract.

## What This Changes

The issue reports Windows event-loop stalls and Gateway timeouts from synchronous Pi `DefaultResourceLoader` filesystem discovery during embedded agent startup.

## Best Possible Solution

Design a narrow OpenClaw-owned embedded resource-loader path, or coordinate an upstream Pi cached/async discovery fix, with Windows proof that Gateway responsiveness and intentional Pi resources are preserved.

## Reproduction Assessment

Unclear: the issue includes concrete Windows timing logs and current source confirms the blocking reload path, but I did not run the Windows/NTFS scenario. Source inspection makes the reported root cause credible rather than fully reproduced.

## Solution Assessment

Unclear: exposing skip flags alone is likely not the best fix because Pi resolves and scans resources before those flags filter them. A safer solution is an OpenClaw/Pi loader-boundary change or upstream cached/async discovery, validated against real Gateway behavior.
