## Summary

Keep open: current main still awaits Pi's `DefaultResourceLoader.reload()` in embedded run and compaction startup, and the pinned dependency still performs broad synchronous resource discovery, so the reported Windows event-loop blocking remains plausible and unresolved.

## What This Changes

The issue requests reducing `DefaultResourceLoader.reload()` startup blocking on Windows by avoiding repeated synchronous Pi resource discovery scans, possibly through caching, async discovery, or skip controls.

## Best Possible Solution

Design a narrow OpenClaw embedded resource-loader boundary, or coordinate an upstream Pi cached/async discovery fix, that preserves OpenClaw skills prompt behavior and inline Pi extensions while proving Gateway responsiveness on Windows.

## Reproduction Assessment

Unclear for live reproduction: the issue includes concrete timing logs, and source inspection confirms the current-main call path and synchronous dependency scan. I did not reproduce the Windows/NTFS stall in a real Windows setup during this read-only review.

## Solution Assessment

Unclear: a config-only skip-flag approach is likely insufficient because Pi resolves packages before those flags filter resources. The safer maintainable direction is an OpenClaw-owned loader seam or upstream Pi cache/async change with Windows proof.
