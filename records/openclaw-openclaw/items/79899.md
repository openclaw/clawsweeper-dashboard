## Summary

Keep open: current main still awaits Pi `DefaultResourceLoader.reload()` in embedded attempt and compaction, and the pinned dependency still calls `packageManager.resolve()` before the no-discovery filters take effect. The open fix path is now https://github.com/openclaw/openclaw/pull/82536, but it is unmerged and already needs real behavior proof plus resource-loader contract review.

## What This Changes

The issue reports Windows event-loop stalls from synchronous Pi resource discovery during OpenClaw embedded agent startup.

## Best Possible Solution

Review, land, or replace https://github.com/openclaw/openclaw/pull/82536 with a public-contract fast path or upstream API that avoids `packageManager.resolve()` while preserving inline extension factories and Windows timing proof.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection does reproduce the current-main path: embedded runs still await `DefaultResourceLoader.reload()`, and the pinned dependency performs synchronous discovery before filtering.

## Solution Assessment

No, skip flags alone are not the best fix because the dependency resolves and auto-discovers resources before those flags discard results. The maintainable fix should avoid that discovery through a stable OpenClaw/Pi contract and prove the result on Windows.
