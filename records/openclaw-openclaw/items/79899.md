## Summary

Keep open: current main still awaits Pi `DefaultResourceLoader.reload()` in embedded agent startup and compaction, and the pinned Pi dependency still performs broad synchronous filesystem discovery before an agent session starts.

## What This Changes

The issue asks OpenClaw to avoid 12-15 second Windows event-loop stalls caused by synchronous Pi resource discovery during embedded agent startup.

## Best Possible Solution

Add a narrow embedded resource-loader boundary or coordinate an upstream Pi cache/async discovery fix that avoids broad synchronous scans while preserving OpenClaw skills prompt behavior and inline extension factories.

## Reproduction Assessment

Unclear: the issue includes concrete timing logs and the current source path is clear, but I did not reproduce the Windows/NTFS stall on current main. Source inspection does confirm the reload path still exists and uses synchronous broad dependency discovery.

## Solution Assessment

Unclear: a config-only skip-flag solution is probably not sufficient because the dependency resolves resources before applying those filters. The better fix is a deliberate OpenClaw embedded-loader boundary or upstream Pi discovery change with Windows Gateway responsiveness proof.
