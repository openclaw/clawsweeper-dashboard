## Summary

Keep open: current main has a partial mitigation that passes Pi's skip flags, but the upstream loader still calls synchronous package/resource discovery before those flags filter resources, so the reported Windows event-loop stall is not proven fixed.

## What This Changes

The issue reports that Pi's `DefaultResourceLoader.reload()` synchronously scans large resource trees on Windows and blocks OpenClaw agent startup.

## Best Possible Solution

Keep this open and fix the OpenClaw/Pi embedded loader boundary so disabled Pi discovery avoids `packageManager.resolve()` scanning entirely, or coordinate an upstream Pi option that skips discovery before synchronous filesystem traversal.

## Reproduction Assessment

Unclear for high-confidence live reproduction: the issue includes concrete Windows timing logs and current source still reaches the dependency reload path, but I did not reproduce the Windows/NTFS stall on current main. Source inspection does confirm that the dependency still performs synchronous discovery before OpenClaw's current skip flags filter resources.

## Solution Assessment

No: the current skip-flag helper is a useful partial mitigation, but it is not the best complete fix because upstream `DefaultResourceLoader.reload()` resolves packages and auto-discovered resources before those flags are applied. A safer fix is an OpenClaw-owned embedded loader path or upstream Pi pre-resolve skip/caching contract with Windows proof.
