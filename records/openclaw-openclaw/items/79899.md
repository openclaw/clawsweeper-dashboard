## Summary

Keep open: this remains the canonical issue for the embedded Pi resource-loader stall. Current main shipped a mitigation that disables Pi resource types, but it still awaits `DefaultResourceLoader.reload()`, and the pinned dependency still runs synchronous package/resource discovery before those flags filter results.

## What This Changes

The issue requests eliminating Windows event-loop stalls caused by synchronous Pi `DefaultResourceLoader.reload()` filesystem discovery during embedded agent startup.

## Best Possible Solution

Keep this issue as the canonical tracker and design a public-contract embedded Pi loader fast path, or upstream Pi change, that avoids `packageManager.resolve()` for OpenClaw embedded runs while preserving inline extension factories and settings semantics, then prove it with Windows large-tree timing logs.

## Reproduction Assessment

No high-confidence live current-main reproduction was run here. The reporter provided concrete Windows timing logs, and source/dependency inspection confirms current main still reaches the synchronous Pi reload/discovery path.

## Solution Assessment

Unclear: the shipped skip-flag wrapper is a useful mitigation but not enough evidence for closure because upstream reload still resolves packages first. A maintainable fix should avoid that public reload path or add an upstream-supported fast path rather than rely on private loader fields.
