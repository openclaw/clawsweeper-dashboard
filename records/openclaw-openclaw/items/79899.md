## Summary

Keep open: current main contains a shipped partial mitigation that passes Pi `no*` discovery flags for embedded runs, but the dependency contract still calls `packageManager.resolve()` before those flags filter resources, and that resolver still uses synchronous recursive filesystem discovery across the reported resource directories.

## What This Changes

The issue asks OpenClaw to stop Pi `DefaultResourceLoader.reload()` from synchronously scanning resource directories on every embedded agent run on Windows.

## Best Possible Solution

Finish the embedded Pi resource-loader boundary so OpenClaw preserves inline extension factories while bypassing Pi package/resource auto-discovery for embedded runs, then validate the startup path on Windows/NTFS.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live Windows reproduction was established here, but current source plus the Pi dependency shows the reported synchronous discovery path still runs before session creation.

## Solution Assessment

Is this the best way to solve the issue? The shipped skip-flag wrapper is not sufficient by itself because Pi resolves and scans before applying those flags; a narrower embedded resource-loader path or upstream no-discovery hook is the maintainable fix.

Keep the change scoped to the embedded runner/resource-loader boundary; do not expose new user config, change global Pi CLI behavior, or remove compaction/context-pruning extension factories. Add regression coverage that would fail if embedded `reload()` calls Pi package/resource discovery, and preserve the existing tests that pass the resource loader into normal and compaction session creation.
