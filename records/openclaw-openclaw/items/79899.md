## Summary

Keep open: current main still awaits Pi's resource-loader reload on embedded agent and compaction startup, and upstream Pi source still performs broad synchronous filesystem discovery before OpenClaw can create the session.

## What This Changes

The issue asks OpenClaw to stop `DefaultResourceLoader.reload()` from blocking Windows Gateway runs with synchronous Pi filesystem discovery.

## Best Possible Solution

Keep this open and design a narrow embedded resource-loader boundary, or coordinate an upstream Pi cached/async discovery fix, with Windows proof that Gateway responsiveness improves and inline OpenClaw extension behavior is preserved.

## Reproduction Assessment

Unclear for high-confidence live reproduction: the reporter provided concrete Windows timing logs, and current source still reaches the synchronous dependency path, but I did not run a Windows/NTFS Gateway setup with a large `.openclaw` tree. The source-level path is clear enough to keep investigating.

## Solution Assessment

Unclear: exposing skip flags alone is probably not the best fix because Pi resolves package/resource paths before those flags filter loaded resources. The maintainable direction is an OpenClaw-owned embedded loader boundary or upstream cached/async Pi discovery, validated on the reported Windows path.
