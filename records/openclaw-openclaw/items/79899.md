## Summary

Keep open: current main still awaits Pi `DefaultResourceLoader.reload()` in embedded run prep, and the pinned dependency still performs synchronous package/resource discovery before the OpenClaw `no*` filters discard those resources. The open replacement PR at https://github.com/openclaw/openclaw/pull/82536 targets the right area, but it is unmerged, has review blockers, and lacks after-fix real behavior proof.

## What This Changes

The issue asks OpenClaw to stop Windows event-loop stalls caused by synchronous `DefaultResourceLoader.reload()` filesystem discovery during embedded agent prep.

## Best Possible Solution

Keep this as the canonical issue and land a reviewed fix that avoids upstream package discovery through a public or owned resource-loader boundary, with Windows large-tree timing proof and preserved inline extension behavior.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection and the reporter's logs are credible: current main still awaits `reload()`, and the pinned dependency still performs synchronous discovery in that path.

## Solution Assessment

No, current main is not a complete solution because `no*` flags do not stop upstream package discovery. The open PR direction needs owner review, a public/owned contract, and real Windows timing proof before it can close the issue.
