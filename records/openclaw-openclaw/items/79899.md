## Summary

Keep open: current main and the latest release still route embedded runs through `DefaultResourceLoader.reload()` and `packageManager.resolve()`, so the reported synchronous resource-discovery stall is not fully implemented away. The issue is actionable, but the safest fix needs maintainer design review and live Windows/NTFS timing proof because the related open PR is conflicting and lacks after-fix real behavior proof.

## What This Changes

The issue asks OpenClaw to stop `DefaultResourceLoader.reload()` from synchronously scanning resource directories on every Windows agent run, where it causes multi-second event-loop stalls and channel timeouts.

## Best Possible Solution

Add a maintainer-approved embedded resource-loader fast path that avoids ambient package/resource discovery when embedded runs already pass explicit resources, while preserving settings reload semantics, compaction guards, inline extensions, source info, and real Windows timing proof.

## Reproduction Assessment

No, not for the exact Windows timing. Source inspection shows current main still awaits `reload()` and `packageManager.resolve()` on embedded runs, but reproducing the reported 12-15s stall needs a Windows/NTFS setup with a large resource tree.

## Solution Assessment

Unclear. The likely direction is to avoid unnecessary embedded-run discovery, but maintainers still need to choose the contract shape and reject config-surface churn unless the fast path cannot solve the runtime issue.
