## Summary

Keep open: current main and the latest release still route embedded runs through `DefaultResourceLoader.reload()` and `DefaultPackageManager.resolve()`, so the reported resource-discovery stall is not implemented away. The related open PR is conflicting and not proof-positive, so it should remain maintainer-owned rather than being treated as a resolved or superseding fix.

## What This Changes

This issue reports Windows event-loop blocking during embedded agent runs because `DefaultResourceLoader.reload()` performs synchronous resource and package discovery on every run.

## Best Possible Solution

Add a maintainer-approved embedded resource-loader fast path that avoids ambient discovery while preserving settings reload semantics, inline extension factories, compaction guards, metadata/conflict diagnostics, and Windows large-tree timing proof.

## Reproduction Assessment

No, not for the exact Windows/NTFS 12-15s timing. Source inspection gives a high-confidence current-main path through `reload()` and `packageManager.resolve()`, but a live Windows large-tree benchmark is still missing.

## Solution Assessment

Unclear. Avoiding unnecessary embedded-run discovery is the right direction, but the best fix should be a supported loader fast path rather than a broad new user config surface or private-state shortcut.
