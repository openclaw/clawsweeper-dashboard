## Summary

Keep open: current main and latest release still route embedded attempts and compaction through `DefaultResourceLoader.reload()`, and `reload()` still runs synchronous package/resource discovery before the disabled-resource filters take effect. The related open PR at https://github.com/openclaw/openclaw/pull/82536 is a candidate direction, but it is not merged and still needs loader-contract review plus Windows timing proof.

## What This Changes

The issue asks OpenClaw to stop `DefaultResourceLoader.reload()` from blocking embedded agent runs on Windows/NTFS through synchronous resource discovery.

## Best Possible Solution

Add a maintainer-approved embedded resource-loader fast path that avoids ambient discovery while preserving settings reload semantics, inline extension factories, compaction guards, diagnostics, and Windows large-tree timing proof.

## Reproduction Assessment

No, not for the exact Windows/NTFS timing. Source inspection gives a high-confidence current-main path through embedded `resourceLoader.reload()` and synchronous package discovery, but a live large-tree Windows benchmark is still missing.

## Solution Assessment

Unclear. Avoiding unnecessary embedded-run discovery is the right direction, but the durable fix should be a supported loader fast path rather than broad new config or a private-state shortcut.
