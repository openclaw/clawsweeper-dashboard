## Summary

Keep this PR open: it is a focused fix for a source-reproducible Chrome MCP session-cache cancellation bug, and I found no discrete blocking patch defect. The remaining decision is maintainer acceptance of the Chrome MCP lifecycle risk and proof depth.

## What This Changes

The branch changes Chrome MCP cached session attachment to use waiter-counted pending-session leases with an internal abort controller, and adds regression tests for cancellation, ready, stale-transport, and close-cleanup races.

## Best Possible Solution

Land the waiter-counted pending-session design after a browser owner accepts the targeted lifecycle proof, or add one live Chrome/CDP smoke if maintainers want end-to-end proof before merge.

## Reproduction Assessment

Yes from source inspection, though I did not run the repro. Current main shares one pending Chrome MCP attach promise created with the first caller's signal, so aborting that caller can reject unrelated waiters that joined the same promise.

## Solution Assessment

Yes, this is an appropriate fix shape: the ownership belongs in the Chrome MCP session cache, and the PR gives the shared attach its own abort controller with per-caller leases. The only unresolved question is proof depth for live Chrome/CDP lifecycle behavior.
