## Summary

Keep this PR open: it addresses a real Chrome MCP pending-session cancellation bug that is still present on current main, but the proposed last-waiter cleanup leaves a closing session visible in the cache long enough for a fresh caller to reuse it.

## What This Changes

The PR changes Chrome MCP cached-session pending attach handling so caller aborts release only that caller while the shared attach is aborted or closed only after the last waiter leaves.

## Best Possible Solution

Evict the cached session before any awaited last-waiter close, keep the waiter-isolation design, and add a regression test for a fresh caller arriving during that cleanup window.

## Reproduction Assessment

Yes. Current main shares one pending Chrome MCP attach promise created with the first caller's AbortSignal, so the linked two-waiter abort scenario is source-reproducible without needing a new product decision.

## Solution Assessment

No, not yet. The waiter-isolation direction is the right narrow fix, but the last-waiter cleanup must remove the cached session before awaiting close to avoid introducing a fresh-caller race.
