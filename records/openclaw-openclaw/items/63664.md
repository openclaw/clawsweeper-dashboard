## Summary

Keep open. Current main still enforces a single exact memory-flush write target, and relaxing that guard overlaps the intentional sandbox hardening from the merged memory-flush protection work.

## What This Changes

The issue asks for memory-flush compaction to stop blocking unrelated non-memory file writes during the flush turn.

## Best Possible Solution

Keep the strict single-target guard unless maintainers approve an explicit extra-target contract; if approved, implement it with normalized allowlists, mode-specific behavior, docs, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. A memory-flush run passes one `memoryFlushWritePath`, and the wrapper rejects any requested write path that does not resolve exactly to that target.

## Solution Assessment

No, not as a soft limit, magic word, or prompt-derived bypass. The maintainable path is an explicit write-policy contract, or keeping the current strict guard if that risk is not accepted.
