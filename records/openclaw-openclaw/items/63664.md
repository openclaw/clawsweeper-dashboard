## Summary

Keep open. Current `main` and the latest release still enforce the exact single-target memory-flush write sandbox, the permissive linked fix PR was closed unmerged, and the remaining work is a product/security decision about whether to add an explicit expanded write-target contract.

## What This Changes

The issue asks OpenClaw to stop blocking unrelated file writes while a memory-flush compaction turn is active, or to provide an explicit bypass/configuration for that restriction.

## Best Possible Solution

Keep this canonical policy issue open until maintainers choose either to preserve the strict single-target sandbox or to add a validated memory-flush write-target contract with schema/docs/tests and no prompt-derived permission.

## Reproduction Assessment

Yes, at source level. A memory-triggered run forwards one `memoryFlushWritePath`, wraps the write tool, and rejects any resolved path other than that target; I did not run an end-to-end live flush.

## Solution Assessment

No, not as a broad passthrough, magic word, or prompt-derived permission. The maintainable path is an explicit configured or plugin-provided allowlist with validated paths and per-target write modes, or an explicit decision to keep the strict guard.
