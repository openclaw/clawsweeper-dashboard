## Summary

Keep open. Current main and the latest release still enforce a single exact memory-flush write target, and the newer related report was closed as a duplicate pointing back here, so this remains the canonical thread but needs product/security review before changing write authority.

## What This Changes

The issue asks OpenClaw to allow non-memory file writes, or an explicit bypass/configuration, while a pre-compaction memory-flush turn is active.

## Best Possible Solution

Decide the memory-flush write policy once, then either preserve the strict guard or add explicit validated additional write targets with per-target modes, docs/schema alignment, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. A memory-triggered flush forwards one `memoryFlushWritePath`, and the wrapped write tool rejects any resolved path other than that target; I did not run an end-to-end live flush.

## Solution Assessment

No, not as a soft limit, magic word, or broad passthrough. The safer maintainable path is an explicit approved write-target contract, or keeping the strict guard if maintainers do not accept the sandbox expansion.
