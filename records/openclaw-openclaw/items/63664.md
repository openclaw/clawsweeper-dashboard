## Summary

Keep open. Current main still enforces a single exact memory-flush write target, the permissive fix PR was closed unmerged, and this remains the canonical broader product/security decision for whether memory-flush turns may write outside the planned daily memory file.

## What This Changes

The issue asks for memory-flush compaction writes to stop blocking unrelated non-memory file writes or to provide an explicit bypass/permission mechanism.

## Best Possible Solution

Keep this as the canonical policy item, then either preserve the strict single-target sandbox or implement an explicit validated memory-flush write-target contract with docs and focused regression coverage.

## Reproduction Assessment

Yes, at source level. A memory-triggered run forwards one `memoryFlushWritePath`, and the wrapped write tool rejects every resolved path other than that target; I did not run a live compaction turn.

## Solution Assessment

No, not as broad passthrough, prompt-derived permission, or a magic-word unlock. The safer maintainable path is an explicit approved write-target contract, or keeping the strict guard if maintainers reject the sandbox expansion.
