## Summary

Keep open: current main and the latest release still enforce a single-target append-only memory-flush write guard, the direct fix PR is closed unmerged, and a duplicate report was already closed into this issue as the canonical thread. The remaining work is a product/security decision about whether memory-triggered housekeeping turns may write non-memory files.

## What This Changes

The issue asks for memory-flush compaction to stop blocking unrelated non-memory file writes during an active session.

## Best Possible Solution

Maintainers should choose a narrow write-authority policy for memory-triggered runs: keep the append-only memory target as the default, and only allow non-memory writes through an explicit, tested escape path if that is the intended security tradeoff.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current code wraps memory-triggered writes and rejects every path except the exact memory flush file, with tests asserting the same error. I did not run a live concurrent normal-write reproduction, so that narrower race claim remains unproven.

## Solution Assessment

Unclear. The closed pass-through PR is a plausible fix for productivity, but the best solution needs maintainer approval for the security boundary before allowing non-memory writes during silent compaction housekeeping.
