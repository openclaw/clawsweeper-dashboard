## Summary

Keep open: current main and v2026.6.10 still enforce the exact single-file memory-flush write guard, recent user reports show the same failure remains visible, and relaxing it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop blocking unrelated write-tool file edits while a memory-flush compaction turn is active.

## Best Possible Solution

Define and land a maintainer-approved memory-flush write-authority contract that either preserves the single-target append guard or allows narrowly validated non-memory exceptions with tests, docs, and explicit security/upgrade reasoning.

## Reproduction Assessment

Yes, at source level: current main routes memory-triggered tool calls through `wrapToolMemoryFlushAppendOnlyWrite`, and any path other than the configured `memoryFlushWritePath` throws the reported error. I did not run a live long-session compaction in this read-only review.

## Solution Assessment

Unclear: the bug is real, but the suggested soft limit, config option, or magic unlock would relax a security-sensitive write boundary. The best path is a maintainer-approved narrow contract rather than an automatic broad passthrough.
