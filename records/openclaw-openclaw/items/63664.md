## Summary

Keep open. Current main and the latest release still enforce a single exact memory-flush write target, the attempted fix PR was closed unmerged, and relaxing this guard is a product/security policy decision rather than conservative cleanup.

## What This Changes

The issue requests allowing non-memory writes, or an explicit bypass, while session memory flush compaction is active.

## Best Possible Solution

Decide the memory-flush write policy once, then either keep the strict guard or add an explicit validated write-target contract with per-target modes, docs/schema alignment, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. A memory-triggered flush forwards one `memoryFlushWritePath`, and the wrapped write tool rejects any resolved path other than that target; I did not run an end-to-end live flush.

## Solution Assessment

No, not as a broad passthrough, magic word, or prompt-derived permission. The safer maintainable path is an explicit approved write-target contract, or keeping the strict guard if maintainers reject the sandbox expansion.
