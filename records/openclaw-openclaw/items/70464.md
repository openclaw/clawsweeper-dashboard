## Summary

Keep open. The PR targets a real memory-core dreaming session leak that current main still has, but the current patch is not merge-ready because it narrows the documented generic plugin-owned `deleteSession` cleanup path to a memory-core-only exception and the latest PR head still has failing core CI checks.

## What This Changes

The PR stabilizes memory-core dreaming narrative session keys, moves the timestamp into the idempotency key, adds pre-run cleanup, and changes gateway subagent deletion authorization/tests around memory-core dreaming cleanup.

## Best Possible Solution

Keep the memory-core stable per-workspace/per-phase session key, timestamped idempotency key, and pre-run cleanup direction, but preserve current main's generic owner-bounded plugin cleanup seam instead of adding a memory-core-specific gateway exception. Add regression coverage for third-party plugin-owned cleanup and the required changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a non-admin plugin runtime calling `api.runtime.subagent.deleteSession()` for its own plugin-created session: current main forces an owner-tagged synthetic admin client, while this PR only does that for memory-core `dreaming-narrative-*` keys.

## Solution Assessment

No. The memory-core stable-key direction is the right narrow product fix, but narrowing core authorization to a bundled plugin id conflicts with the documented generic plugin runtime contract; the safer solution is to keep the generic owner check and fix memory-core on top of it.
