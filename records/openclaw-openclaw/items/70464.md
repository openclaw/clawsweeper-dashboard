## Summary

Keep this PR open. The memory-core change still addresses real dreaming-session pollution that current main has not fully fixed, but the latest diff would regress the documented generic plugin-owned `deleteSession` contract by making the fallback admin cleanup path memory-core-specific, and the latest head has failing core CI checks.

## What This Changes

The PR stabilizes memory-core dreaming narrative session keys, uses timestamped idempotency keys, adds pre-run session cleanup, and changes gateway subagent deletion authorization/tests around memory-core dreaming cleanup.

## Best Possible Solution

Preserve current main's generic `pluginOwnerId`-bounded cleanup seam, implement the stable per-workspace/per-phase dreaming key plus timestamped idempotency and pre-run cleanup inside `extensions/memory-core`, add regression coverage that third-party plugin-owned cleanup remains generic, and add the required changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a non-admin plugin runtime calling `api.runtime.subagent.deleteSession()` for its own plugin-created session: current main grants a synthetic admin client and relies on `pluginOwnerId`, while this PR only grants that path for memory-core `dreaming-narrative-*` keys.

## Solution Assessment

No. The memory-core stable-key direction is the right narrow product fix, but changing core to a memory-core-specific authorization exception is not the best maintainable solution because it conflicts with the documented generic plugin runtime contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep plugin-owned cleanup generic:** `src/gateway/server-plugins.ts:491-494`
  - body: This makes the fallback admin delete path depend on `syntheticScopes`, which is only set for memory-core `dreaming-narrative-*` sessions. Preserve the generic synthetic admin path and rely on the owner check.
