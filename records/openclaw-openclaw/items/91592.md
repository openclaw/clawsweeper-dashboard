## Summary

Keep open: current main and v2026.6.6 still have the memory-core identity, live-manager, and safe-reindex shapes that can explain the reported post-rebuild `memory_search` failures, and this issue remains a canonical scope/provider evidence thread rather than a safely closeable duplicate.

## What This Changes

The issue reports builtin `memory_search` returning `index scope changed` or `index metadata is missing` after `openclaw memory index --force`, especially with non-default embedding providers and multi-agent setups.

## Best Possible Solution

Coordinate one memory-core lifecycle fix that preserves true provider/model/settings fail-closed behavior, publishes or reloads rebuilt indexes for live managers, and proves non-default provider scopeHash plus second-search gateway behavior.

## Reproduction Assessment

Yes, at source level: current main still has the CLI/status versus cached live-manager split, search-triggered sync, safe-reindex temp DB exposure, and post-search paused-identity gate that match the reported failure class; no live gateway reproduction was run in this read-only review.

## Solution Assessment

The report should stay open, but the best fix is not simply to ignore scope mismatches; maintainers need one manager-owned publication/reload and identity-validation contract with live proof across non-default providers.
