Keep open. Current main still pauses memory_search when the stored scopeHash differs from the configured scope; the reporter’s file-table hash comparison is not the implemented contract, but the post-rebuild paused-tool symptom is plausible and not proven fixed.

## What This Changes

The issue reports memory_search returning index scope changed after openclaw memory index --force because the persisted memory index scopeHash does not match the active runtime scope.

## Best Possible Solution

Keep scopeHash as a configured-scope identity, then add a focused repro and fix the live gateway recovery path for scope-only mismatches or CLI reindex handoff without auto-rebuilding unrelated provider/model mismatches.

Yes from source for the paused mismatch path: a mismatched index identity makes search return no results and the tool returns unavailable before retrying sync. I did not establish the full WSL2/Ollama/sqlite-vec after-force live reproduction locally.

## Solution Assessment

Unclear. Changing scopeHash to a files-table hash is not the best fix because current code defines it as configured scope; the maintainable fix is likely a scope-aware gateway repair or reopen path coordinated with the related memory-index recovery report.
