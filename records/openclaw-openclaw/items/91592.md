## Label Justifications

## Summary

Current main addresses the observable post-rebuild failure: the reported files-table hash is not the scopeHash contract, while the sticky missing-metadata behavior after a force rebuild was traced to the SQLite swap’s absent-file window and fixed by preserving a continuously present live database. The confirming reports are against v2026.6.5 or earlier, before the June 11, 2026 main-only fix.

## What This Changes

Fix `memory_search` remaining paused with missing or mismatched index identity after a forced builtin-memory SQLite rebuild.

## Best Possible Solution

Keep the current-main no-gap SQLite swap and no-auto-create reopen behavior; reopen or file a new focused report only if the two-search sequence still fails on a build containing commit 865fdab07501b5d85b4057d06b22e0ce907300e7, with resolved workspace, database path, and full index-identity fields.

## Reproduction Assessment

Yes. The underlying current-release failure has a deterministic source reproduction: an open during the old atomic-swap absent-file window auto-created an empty SQLite database without index metadata, pinning the gateway to the reported paused state.

## Solution Assessment

Yes. Removing the absent live-database window and refusing silent empty-database creation fixes the storage lifecycle at its source without weakening provider/model/scope identity validation.
