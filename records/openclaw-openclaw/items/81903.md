## Summary

Keep this open: current main still has a plausible source-level path where a Slack draft that was finalized into the assistant answer can later be cleared by a non-finalizable `isError` tool-warning payload, which matches the reported “answer replaced by Apply Patch failed” symptom.

## What This Changes

The issue reports that Slack partial streaming can replace an in-flight assistant draft with an `Apply Patch failed` tool-status message even when `streaming.preview.toolProgress` is disabled.

## Best Possible Solution

Keep the Slack assistant draft/final answer lifecycle separate from later tool-warning error payloads: after a preview is finalized into assistant text, later non-finalizable/error finals must not clear or edit that finalized message.

## Reproduction Assessment

Yes, source-level: a Slack streamed text final can be finalized in place, then a later `isError` tool-warning final follows the generic non-finalizable path that calls `draft.clear()` on the same draft id. I did not run a live Slack repro in this read-only review.

## Solution Assessment

Yes, a narrow lifecycle fix is the best path: mark or forget the draft after successful preview finalization, or otherwise prevent later non-finalizable finals from clearing the finalized assistant answer. Hiding all mutating tool warnings would be a broader policy change and is not required to fix the reported overwrite.
