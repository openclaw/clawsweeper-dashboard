## Summary

Keep this open: current main still has a source-level path where a Slack assistant draft can be finalized in place and then a later non-finalizable `isError` tool-warning payload clears the same draft, matching the reported replacement of the draft with an `Apply Patch failed` status.

## What This Changes

The issue reports that Slack partial streaming can replace an in-flight assistant draft with a tool failure status even when Slack preview tool progress is disabled.

## Best Possible Solution

Keep Slack assistant draft/final answer state separate from later tool-warning/error finals so disabled tool progress cannot clear or replace the assistant draft, with focused regression coverage for Slack finalization and the generic live-preview lifecycle.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: finalize a Slack streamed assistant draft, then emit a later non-finalizable `isError` tool-warning payload on the same draft, which reaches `draft.clear()`. I did not run a live Slack workspace repro in this read-only review.

## Solution Assessment

Yes, the best fix is narrow lifecycle isolation: after a Slack assistant preview is finalized into user-visible text, later non-finalizable tool-warning finals should not clear that finalized draft and should use a separate allowed progress/error surface if shown.
