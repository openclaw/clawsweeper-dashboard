## Summary

Keep open. Current main still has a source-reproducible Slack path where `streaming.preview.toolProgress=false` suppresses progress callbacks, but terminal tool-error warning payloads are handled separately and can clear the active draft preview while delivering the warning as a normal Slack error message. The related channel-leak issue is broader and remains open, but this report has a narrower Slack draft/finalization failure mode worth fixing directly.

## What This Changes

The issue requests that Slack streaming preserve the live assistant draft when a tool failure occurs while preview tool progress is disabled.

## Best Possible Solution

Keep this open and make a narrow Slack/message-lifecycle fix so tool-error warning payloads cannot delete, overwrite, or reuse the assistant draft preview when preview tool progress is disabled.

## Reproduction Assessment

Yes for source-level reproduction, but not a live Slack run: current main can emit a separate `isError` tool warning after assistant text, and Slack finalization clears the draft for non-finalizable error finals. A focused unit test can exercise this path without needing real Slack credentials.

## Solution Assessment

Yes, the best fix is narrow: preserve the assistant draft/finalized preview lane and either suppress or separate tool-failure status delivery under the existing progress/error-warning contracts. This should not be folded into the broader explicit-send-semantics work tracked by https://github.com/openclaw/openclaw/issues/25592.
