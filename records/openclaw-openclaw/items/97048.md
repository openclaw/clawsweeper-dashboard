## Summary

Keep open: current main and v2026.6.10 still let the session model registry reject generated plugin catalog rows containing `audio` or `video` inputs, even though adjacent config and catalog surfaces already preserve those modalities. The issue is not fixed on main and the older related reports/PRs are closed or only partially cover the remaining generated-catalog path.

## What This Changes

The issue reports that OpenClaw-generated provider catalog entries with `audio` or `video` input modalities fail model catalog validation and get dropped, causing configured primary models to fall back.

## Best Possible Solution

Align the generated plugin catalog validator and model input types with the broader `audio`/`video` contract, then add a focused regression for a generated plugin catalog shard so OpenClaw-owned refreshes cannot produce rows the registry rejects.

## Reproduction Assessment

Yes from source inspection: a generated catalog row with `input: ["text", "image", "video"]` or `audio` reaches `validateModelsConfig`, whose schema only allows `text` and `image`. I did not run a live provider fetch in this read-only review.

## Solution Assessment

Yes, the best fix is a generic schema/type alignment plus generated-catalog regression coverage. A provider-specific clamp for MiniMax or NVIDIA would leave the split validator behavior in place.
