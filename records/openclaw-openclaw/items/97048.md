## Summary

Keep open: current main and the latest release still reject generated plugin catalog rows whose `input` includes `audio` or `video`, and the issue is already paired with open fixing PRs. The strongest current path is to review and land the canonical proof-positive PR, then close this issue after merge.

## What This Changes

The issue asks OpenClaw to stop dropping generated provider catalog models like MiniMax-M3 and phi-4-multimodal when their input modalities include audio or video.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/97252 or an equivalent contract-alignment fix, then close this issue as implemented after the merge.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a generated catalog row with `audio` or `video` reaches the session registry schema, which still only accepts `text` and `image`, and invalid shards return no models. I did not run a live provider fetch.

## Solution Assessment

Yes, the best fix shape is generic registry/schema/type alignment with focused generated-catalog regression coverage. Provider-specific catalog clamping would discard metadata and leave config/catalog contracts split.
