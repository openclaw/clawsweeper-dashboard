## Summary

Keep open: current main still has a model-registry validator split where generated plugin catalog shards are validated with a `text`/`image`-only schema even though current config and catalog surfaces accept or preserve `audio` and `video`, so the reported dropped catalog rows and fallback behavior remain source-reproducible.

## What This Changes

Fix generated plugin model catalog validation so OpenClaw-owned catalog rows containing `audio` or `video` input modalities are accepted or normalized instead of being dropped and triggering model fallback.

## Best Possible Solution

Align the generated plugin catalog validator and runtime model input contract with the current config/catalog modality contract, then add a focused generated-catalog regression so OpenClaw-owned refreshes cannot write rows the registry drops.

## Reproduction Assessment

Yes, at source level: a generated plugin catalog shard with `input: ["text", "image", "video"]` or `audio` reaches `ModelRegistry` validation, whose TypeBox model schema only allows `text` and `image`. I did not run a live MiniMax/NVIDIA catalog fetch in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is generic registry schema/type alignment with generated-catalog regression coverage; hardcoding MiniMax or NVIDIA rows would leave the split validator behavior in place.
