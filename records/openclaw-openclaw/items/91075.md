## Label Justifications

- `P1`: The report describes shipped MCP runtime process accumulation with multi-GB memory growth during cron reuse, which can break real gateway workflows.
- `impact:crash-loop`: Leaked MCP child processes and SSE/HTTP runtime handles create process-level availability and memory-pressure risk.
- `impact:session-state`: The stale session-scoped MCP runtime remains cached past the intended idle boundary because its session timestamp is refreshed on release.

## Summary

Keep open: current main still refreshes the session MCP runtime idle timestamp when a materialized-tool lease is released, so a recently completed run can postpone the idle sweep even though the documented TTL is meant to reap idle runtimes.

## What This Changes

The issue asks OpenClaw to stop treating MCP runtime lease release as runtime use so `mcp.sessionIdleTtlMs` can evict idle session runtimes on time.

## Best Possible Solution

Keep the existing idle TTL design, remove the release-time `lastUsedAt` refresh, and add a focused regression test that sweeps immediately after an over-TTL active lease is released.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: acquire a runtime lease, let the TTL elapse while active, release the lease, then sweep before another TTL; current main refreshes `lastUsedAt` on release and skips eviction.

## Solution Assessment

Yes. Removing the release-time timestamp update is the narrowest maintainable fix because runtime creation, materialization, and tool execution already call `markUsed()` for actual use.
