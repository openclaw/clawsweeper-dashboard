## Summary

Keep open: current main still has no `src/verify` implementation, and the branch remains stale against the live policy/catalog/pipeline contracts with insufficient after-fix proof, so it is neither obsolete nor safe to merge as-is.

## What This Changes

The PR adds a parser-only `src/verify` module with TypeScript AST parsers, README docs, Vitest coverage, and a standalone runner for future SMT/Z3 tool-policy verification.

## Best Possible Solution

Land a maintainer-approved verification lane that parses current catalog, owner-only approval-class, subagent, and sender-policy contracts fail-closed and documents only supported parser/generator surfaces.

## Reproduction Assessment

Yes for the review findings: source comparison shows current main uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps, while the PR targets the old fallback set, 25 tools, and seven steps. I did not run the PR tests because this review is read-only and the source mismatch is sufficient.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and keep README/proof aligned with files actually included.

  - confidence: 0.86

  - body: Current main returns eight policy pipeline steps including the sender-policy step, but this parser still expects seven. Running the parser after rebase would fail even if the runtime pipeline is valid.
  - confidence: 0.95
