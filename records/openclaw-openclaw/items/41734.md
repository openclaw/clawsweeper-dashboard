## Summary

Keep open: the verification-parser idea is still unique work, but this branch is not merge-ready against current main because it targets stale policy/catalog shapes and can produce misleading future verification inputs.

## What This Changes

Adds a new `src/verify` parser-only module with TypeScript AST parsers, parsed-policy types, README documentation, a Vitest parser suite, and a standalone parser runner for future SMT/Z3 tool-policy verification.

## Best Possible Solution

If maintainers want this verifier lane in core, land a rebased parser-only change that targets current catalog and policy contracts, fails closed on unsupported AST forms, and documents only included functionality.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is comparing the PR parser/tests with current main: the PR searches for `OWNER_ONLY_TOOL_NAME_FALLBACKS` while main uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS`, and the PR asserts 25 tools while main has 33.

## Solution Assessment

No. Parser-only tooling is a plausible incremental direction, but this implementation should be rebased onto live contracts, reject partial extraction, and remove or add the documented generator workflow before merge.

  - body: Current main has 33 core tool definitions, but this test still asserts 25 and hard-codes the older tool id set. The parser fixtures and standalone runner need to be refreshed or derived from the current catalog after rebase.
  - confidence: 0.95
