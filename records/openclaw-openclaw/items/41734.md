## Summary

Keep open. The PR still adds unique verification tooling, but it is stale against current `main` policy/catalog contracts, has blocking parser-soundness issues, and the submitted terminal proof does not show the corrected parser running against current `main`.

## What This Changes

The PR adds a parser-only `src/verify` module for extracting tool catalog, policy, and pipeline facts for future SMT/Z3 policy verification.

## Best Possible Solution

Land only a maintainer-approved verification lane rebased to current policy/catalog contracts, fail closed on partial AST extraction, and keep docs matched to the code actually included.

## Reproduction Assessment

Yes. Source comparison reproduces the blockers: current `main` uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps while the PR targets an old fallback set, 25 tools, and seven steps.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and remove or add the documented SMT generator before merge.

  - body: Current `main` no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` plus `resolveOwnerOnlyToolApprovalClass`. After rebase, this parser cannot extract the live owner-only boundary.

  - body: `parseStringArray` filters unsupported entries and callers only check that parsed lists are non-empty. A future constant, spread, or template entry could be silently omitted from the verification model while runtime still enforces it.
  - confidence: 0.88
- **[P1] Fail on unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the model instead of stopping extraction.
  - confidence: 0.86
- **[P2] Refresh the catalog fixtures for current main:** `src/verify/__tests__/parse-tools.test.ts:12-44`
  - body: Current `main` has 33 core tool definitions, but the test still asserts 25 and hard-codes the older id set. The parser fixtures and standalone runner need current catalog expectations before this can merge.
  - confidence: 0.96
- **[P2] Refresh the pipeline step contract:** `src/verify/parse-pipeline.ts:127-128`
  - body: Current `main` returns eight policy pipeline steps including the sender policy step, but this parser still expects seven. Running the parser after rebase would fail even if the runtime pipeline is valid.
  - confidence: 0.95
