## Summary

Keep open: the new verification parser is not implemented on current main and the branch still has blocking stale-contract and security-adjacent extraction issues that need contributor updates plus maintainer/security review before merge.

## What This Changes

Adds a new `src/verify` AST parser module, parser tests, shared types, standalone runner, and README for extracting tool catalog, policy, and pipeline data for future formal verification.

## Best Possible Solution

Land a maintainer-approved verification lane that parses the current catalog, approval-class owner-only policy, subagent arrays, and sender-policy pipeline fail-closed, with docs and proof limited to implemented surfaces.

## Reproduction Assessment

Yes for review purposes from source inspection; I did not execute the branch because this is a read-only review. Current main has 33 tools, an 8-step sender-policy pipeline, and approval-class owner-only policy while this branch targets 25 tools, 7 steps, and a retired owner-only symbol.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should first be rebased to current contracts, fail closed on unparsed AST data, and remove generator docs until generator files exist.
