## Summary

Keep open for maintainer/security review: the PR is meaningful internal verification work, but it is incorrect against current main and still lacks sufficient after-fix real behavior proof.

## What This Changes

Adds a new `src/verify` parser module, README, Vitest tests, and a standalone runner to extract tool catalog, policy, and pipeline metadata for a future SMT/Z3 verification lane.

## Best Possible Solution

Land only a maintainer-approved verification lane that parses current owner-only approval classes, catalog tools, and pipeline steps fail-closed, with docs and proof matching the parser-only scope or a real SMT generator.

## Reproduction Assessment

Yes. Source comparison is high confidence: current main uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps, while the PR targets the old fallback set and 25-tool/7-step expectations.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and remove or add the documented SMT generator before merge.

  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` plus `resolveOwnerOnlyToolApprovalClass`. After rebase, this parser cannot extract the live owner-only boundary.
  - confidence: 0.97

  - body: `parseStringArray` filters unsupported entries and callers only check that each parsed list is non-empty. A future constant, spread, or template entry could be silently omitted from the verification model while runtime still enforces it.
  - confidence: 0.89
- **[P1] Fail on unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the model instead of stopping extraction.
  - confidence: 0.86
- **[P2] Refresh the catalog fixture for current main:** `src/verify/__tests__/parse-tools.test.ts:12-44`
  - body: Current main has 33 core tool definitions, but the test still asserts 25 and hard-codes the older ID set. The parser fixtures and standalone runner need current catalog expectations before this can merge.
  - confidence: 0.96
- **[P2] Refresh the pipeline step count:** `src/verify/parse-pipeline.ts:127-128`
  - body: Current main returns eight policy pipeline steps including the sender policy step, but this parser still expects seven. Running the parser after rebase would fail even if the runtime pipeline is valid.
  - confidence: 0.95
