Keep open for maintainer/security review rather than cleanup close: current main does not implement `src/verify`, and the PR still contains unique verification work, but the branch is not merge-ready because it targets retired policy shapes, stale catalog/pipeline counts, and stale proof.

## What This Changes

The PR adds a new `src/verify` parser module, README, Vitest tests, and a standalone runner for extracting tool catalog, policy, and pipeline metadata for future SMT/Z3 verification.

## Best Possible Solution

Land only a maintainer-approved verification lane that parses current owner-only approval classes, current catalog tools, and current pipeline steps fail-closed, with docs and proof matching either parser-only scope or a real SMT generator.

## Reproduction Assessment

Yes. Source comparison is high confidence: current main uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps, while the PR targets the old fallback set and 25-tool/7-step expectations.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and remove or add the documented SMT generator before merge.
