## Summary

Keep open: this PR is not implemented on current main and still proposes unique security-verification tooling, but the branch is stale against live tool-policy contracts and has blocking correctness, security-review, and real-proof gaps.

## What This Changes

The PR adds a new `src/verify` AST-parser module, README, Vitest tests, and standalone runner to extract tool catalog, policy, and pipeline metadata for future SMT/Z3 verification.

## Best Possible Solution

Land a maintainer-approved verification lane that parses current owner-only approval classes, current catalog tools, and current pipeline steps fail-closed, with docs and proof matching either parser-only scope or an added generator.

## Reproduction Assessment

Yes. Source comparison against current main is high-confidence: live code has 33 catalog tools, an approval-class owner-only map, and eight pipeline steps, while the PR targets 25 tools, the retired fallback symbol, and seven steps.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch must be rebased to live contracts, made fail-closed for unparsed data, and aligned with the documented/proven scope before merge.

## Review Findings

Full review comments:

- **[P1] Target the current owner-only approval map:** `src/verify/parse-policies.ts:112-116`
  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` plus `resolveOwnerOnlyToolApprovalClass`. After rebase, this parser cannot extract the live owner-only boundary.
  - confidence: 0.97
- **[P1] Fail on every unparsed policy array entry:** `src/verify/parse-policies.ts:92-94`
  - body: `parseStringArray` filters unsupported entries and callers only check that each parsed list is non-empty. A future constant, spread, or template entry could be silently omitted from the verification model while runtime still enforces it.
  - confidence: 0.89
- **[P1] Fail on unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the model instead of stopping extraction.
  - confidence: 0.86
- **[P2] Refresh the catalog fixture for current main:** `src/verify/__tests__/parse-tools.test.ts:12-14`
  - body: Current main has 33 core tool definitions, but the test still asserts 25 and hard-codes the older id set. The parser fixtures and standalone runner need current catalog expectations before this can merge.
  - confidence: 0.96
- **[P2] Refresh the pipeline step count:** `src/verify/parse-pipeline.ts:127-128`
  - body: Current main returns eight policy pipeline steps including the sender policy step, but this parser still expects seven. Running the parser after rebase would fail even if the runtime pipeline is valid.
  - confidence: 0.95
- **[P2] Remove docs for absent SMT generator files:** `src/verify/README.md:17-45`
  - body: The README documents `generate-smt.ts`, `emit-smt/`, generated model files, and Z3 commands, but this branch only adds parser code. Users following these docs would hit missing files.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch is inert at runtime, but its security-verification parser can target stale or partial policy data and produce misleading future proof inputs.

Concerns:

- **[medium] Verification parser targets a retired policy shape:** `src/verify/parse-policies.ts:112`
  - body: The parser looks for an old owner-only set while current main uses an approval-class map, so it cannot verify the current owner-only boundary without a rebase and parser update.
  - confidence: 0.94

  - body: Profile extraction filters unparsed array elements, so the verification model can lose profile memberships after a source refactor instead of failing closed.
