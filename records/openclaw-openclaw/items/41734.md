## Summary

Keep open: this PR is not implemented on current main and is not merge-ready because it targets stale policy/catalog contracts, has correctness findings in the verification parser, and still needs stronger real behavior proof.

## What This Changes

The PR adds a parser-only `src/verify` module with TypeScript AST parsers, parsed-policy types, README docs, a Vitest suite, and a standalone runner for future SMT/Z3 tool-policy verification.

## Best Possible Solution

Land only a maintainer-approved verification lane rebased to current policy/catalog contracts, fail closed on partial AST extraction, and keep docs matched to the code actually included.

## Reproduction Assessment

Yes. Source comparison is high confidence: current main uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps, while the PR targets an old fallback set, 25 tools, and seven steps.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and remove or add the documented SMT generator before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Target the current owner-only approval map:** `src/verify/parse-policies.ts:112-116`
  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` plus `resolveOwnerOnlyToolApprovalClass`. After rebase, this parser throws before extracting the live owner-only boundary.
  - confidence: 0.97
- **[P1] Fail on every unparsed policy array entry:** `src/verify/parse-policies.ts:92-94`
  - body: `parseStringArray` filters unsupported entries and callers only check that each parsed list is non-empty. A future constant, spread, or template entry could be silently omitted from the verification model while runtime still enforces it.
  - confidence: 0.88
- **[P1] Fail on unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the model instead of stopping extraction.
  - confidence: 0.86
- **[P2] Refresh the catalog fixtures for current main:** `src/verify/__tests__/parse-tools.test.ts:12-44`
  - body: Current main has 33 core tool definitions, but the test still asserts 25 and hard-codes the older id set. The parser fixtures and standalone runner need current catalog expectations before this can merge.
  - confidence: 0.96
- **[P2] Refresh the pipeline step contract:** `src/verify/parse-pipeline.ts:127-128`
  - body: Current main returns eight policy pipeline steps including the sender policy step, but this parser still expects seven. Running the parser after rebase would fail even if the runtime pipeline is valid.
  - confidence: 0.95
- **[P2] Remove docs for absent SMT generator files:** `src/verify/README.md:17-45`
  - body: The README documents `generate-smt.ts`, `emit-smt/`, generated model files, and Z3 commands, but this branch only adds parser code. Users following these docs would hit missing files.
  - confidence: 0.9
