## Summary

The PR is useful source work, but it is not merge-ready against current main because the parser and tests target stale tool-policy contracts and can silently produce partial verification inputs.

## What This Changes

The PR adds a parser-only `src/verify` module with AST parsers, shared parsed-policy types, README documentation, a Vitest parser suite, and a standalone runner for future SMT/Z3 tool-policy verification.

## Best Possible Solution

If maintainers want this verifier lane in core, land it as a rebased parser-only change that targets the current catalog, approval-class owner-only map, and subagent policy shapes, fails closed on unsupported AST forms, and documents only functionality included in the diff.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is comparing the PR parser/tests with current main: the PR searches for `OWNER_ONLY_TOOL_NAME_FALLBACKS` while main uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS`, and the PR asserts a 25-tool catalog while main has 33 tool definitions.

## Solution Assessment

No. Parser-only tooling is a reasonable incremental direction, but this implementation is not the best current solution until it is rebased onto live contracts, rejects partial extraction, and removes docs for absent SMT generator workflows.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Target the current owner-only approval map:** `src/verify/parse-policies.ts:112-116`
  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` plus `resolveOwnerOnlyToolApprovalClass`. After rebase, this parser throws before extracting the owner-only boundary, so the verifier cannot cover the current policy shape.
  - confidence: 0.96
- **[P1] Reject unparsed policy array entries:** `src/verify/parse-policies.ts:92-94`
  - body: `parseStringArray` filters unsupported entries and callers only check that each parsed list is non-empty. If a deny list later uses a constant, spread, or template entry, extraction can silently omit that denied tool and reason about a weaker policy than runtime enforces.
  - confidence: 0.88
- **[P1] Reject unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the verification model instead of stopping extraction.
  - confidence: 0.84
- **[P2] Refresh current catalog expectations:** `src/verify/__tests__/parse-tools.test.ts:12-44`
  - body: Current main has 33 core tool definitions, but this test still asserts 25 and hard-codes the older tool id set. The parser fixtures need to be refreshed or derived from the current catalog after rebase.
  - confidence: 0.95
- **[P2] Remove references to absent SMT generator files:** `src/verify/README.md:17-45`
  - body: The README documents `generate-smt.ts`, `emit-smt/`, generated model files, and Z3 commands, but this branch only adds parser code. Users following these docs would hit missing-module or missing-output errors.
  - confidence: 0.9
