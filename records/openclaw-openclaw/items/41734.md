## Summary

Keep open: the parser module is not on current main, but the PR is stale against current tool-policy contracts, has fail-open extraction risks, and lacks sufficient after-fix real behavior proof.

## What This Changes

Adds a new `src/verify` parser module, README, Vitest coverage, and standalone runner for extracting tool catalog, policy, and pipeline metadata for future SMT/Z3 verification.

## Best Possible Solution

Land only a maintainer-approved verification lane rebased to current policy/catalog/pipeline contracts, with fail-closed extraction and docs/proof matching the actual parser-only scope.

## Reproduction Assessment

Yes. Source inspection shows current main uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps, while the PR still targets the old fallback symbol, 25 tools, and seven steps.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and align README/proof with the files actually included.

  - confidence: 0.86

  - confidence: 0.96

  - confidence: 0.95
