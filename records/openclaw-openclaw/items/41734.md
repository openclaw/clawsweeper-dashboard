## Summary

Keep this PR open, but it should not merge as-is. It is not implemented on current main and remains a plausible internal verification feature, but the branch targets retired policy/catalog/pipeline contracts and needs maintainer/security review plus stronger real behavior proof.

## What This Changes

The PR adds a parser-only `src/verify` module with TypeScript AST parsers, parsed-policy types, README docs, a Vitest suite, and a standalone runner for future SMT/Z3 tool-policy verification.

## Best Possible Solution

Land only a maintainer-approved verification lane rebased to live policy/catalog contracts, fail closed on partial AST extraction, and keep README/proof aligned with the files actually included.

## Reproduction Assessment

Yes. Source comparison is high confidence: current main uses an approval-class owner-only map, 33 catalog tools, and eight pipeline steps, while the PR targets the old fallback set, 25 tools, and seven steps.

## Solution Assessment

No. Parser-only tooling is a plausible increment, but this branch should be rebased onto live contracts, fail closed on every unparsed policy/catalog element, and remove or add the documented SMT generator before merge.

  - confidence: 0.88

  - confidence: 0.86

  - body: Current main has 33 core tool definitions, but the test still asserts 25 and hard-codes the older id set. The parser fixtures and standalone runner need current catalog expectations before this can merge.
  - confidence: 0.96

  - body: Current main returns eight policy pipeline steps including the sender policy step, but this parser still expects seven. Running the parser after rebase would fail even if the runtime pipeline is valid.
  - confidence: 0.95
