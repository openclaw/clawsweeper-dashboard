## Summary

Keep this PR open, but it is not mergeable as-is: the verifier parser branch is stale against current tool-policy contracts and still has fail-open extraction paths that could make future security proofs misleading.

## What This Changes

Adds a new `src/verify` parser-only module with TypeScript AST parsers, parsed-policy types, README documentation, a Vitest suite, and a standalone parser runner for future SMT/Z3 tool-policy verification.

## Best Possible Solution

If maintainers want this verifier in core, land a rebased parser-only change that targets current catalog, owner-only, subagent, and pipeline contracts; fails closed on unsupported AST forms; and documents only functionality included in the PR.

## Reproduction Assessment

Yes. A high-confidence source reproduction path is comparing PR head ca6a97a1a15f77d0d0181432beda78329bca3b44 with current main 309f000341528a07e556561adcbdb98f1ec29a7e: the PR searches for an old owner-only symbol and expects 25 tools and 7 pipeline steps while main uses a map, 33 tools, and 8 steps.

## Solution Assessment

No. Parser-only verification tooling is a plausible incremental direction, but this patch should be rebased onto the live contracts, made fail-closed for partial extraction, and have README claims aligned with the files actually added.

Full review comments:

- **[P1] Target the current owner-only approval map:** `src/verify/parse-policies.ts:112-116`
  - body: Current main no longer declares `OWNER_ONLY_TOOL_NAME_FALLBACKS`; it uses `OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS` plus `resolveOwnerOnlyToolApprovalClass`. After rebase, this parser throws before extracting the owner-only boundary, so the verifier cannot cover the live policy shape.
  - confidence: 0.96
- **[P1] Update the expected pipeline step count:** `src/verify/parse-pipeline.ts:127-156`
  - body: Current main returns eight default tool-policy pipeline steps, including the sender policy step, but this parser still hard-codes `EXPECTED_PIPELINE_STEPS = 7`. A rebased parser will fail even when it has correctly extracted the live pipeline.
  - confidence: 0.94
- **[P1] Reject unparsed policy array entries:** `src/verify/parse-policies.ts:92-94`
  - body: `parseStringArray` filters unsupported entries and callers only check that each parsed list is non-empty. If a deny list later uses a constant, spread, or template entry, extraction can silently omit that denied tool and reason about a weaker policy than runtime enforces.
  - confidence: 0.88
- **[P1] Reject unparsed catalog profile entries:** `src/verify/parse-tools.ts:80-85`
  - body: `extractStringArray` filters profile entries it cannot decode and returns the truncated array. A catalog refactor through constants, spreads, or templates would silently remove profile memberships from the verification model instead of stopping extraction.
  - confidence: 0.84
- **[P2] Refresh current catalog expectations:** `src/verify/__tests__/parse-tools.test.ts:12-44`
  - body: Current main has 33 core tool definitions, but this test still asserts 25 and hard-codes the older tool id set. The parser fixtures and standalone runner need to be refreshed or derived from the current catalog after rebase.
  - confidence: 0.95
- **[P2] Remove references to absent SMT generator files:** `src/verify/README.md:38-45`
  - body: The README documents `generate-smt.ts`, `emit-smt/`, generated model files, and Z3 commands, but this branch only adds parser code. Users following these docs would hit missing-module or missing-output errors.
  - confidence: 0.9
