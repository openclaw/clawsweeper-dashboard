## Summary

Keep open: current main still lacks typing feedback for inline tool-dispatched skills, but this PR starts typing with a direct controller call that bypasses existing typing policy, and the external PR has no real Telegram proof yet.

## What This Changes

The PR adds an awaited typing-loop start before executing tool-dispatched skill commands in the inline action reply path.

## Best Possible Solution

Wire inline tool-dispatch typing through the existing policy-aware typing semantics, keep typing-start failures non-fatal, and add focused coverage for allowed and suppressed cases before validating with real Telegram proof.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the patch calls `typing.startTypingLoop()` directly in an inline branch that can return before normal typing policy is resolved. The intended visible Telegram behavior still lacks live proof.

## Solution Assessment

No. A direct controller call is not the best fix because it can bypass `typingMode: "never"`, explicit suppression, heartbeat, system-event, and internal-webchat policy; the safer path is to reuse or pass a policy-aware typing signaler for inline tool dispatch.
