## Summary

Keep open. The requested UX is plausible and still not implemented on current main, but this PR is not mergeable as-is because the PR-head change can turn typing failures into skill failures, bypasses existing typing policy, and lacks required real Telegram proof.

## What This Changes

The branch adds an awaited `typing.startTypingLoop()` call before executing `dispatch.kind === "tool"` skill slash commands in `handleInlineActions`.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing policy-aware typing signaler, keep typing-start failures non-fatal, add focused regression coverage, and validate with redacted Telegram proof.

## Reproduction Assessment

Yes for the PR defects: source inspection of the PR head shows the raw awaited typing start inside the tool execution `try` block and outside policy-aware signaling. The intended visible Telegram improvement itself is not proven in a real setup.

## Solution Assessment

No. A direct controller call is not the best fix because it can bypass typing policy and, on the PR head, typing failures can stop the skill before execution; the safer approach is the existing policy-aware signaler plus non-fatal typing-start handling.

  - body: On the PR head, `startTypingLoop()` can reject via the typing start guard, and this new await sits in the same `try` block as `tool.execute()`. A transient channel typing error would return the error reply before the skill tool runs.

  - confidence: 0.9
