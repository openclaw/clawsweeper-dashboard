## Summary

Keep open: current main still has the tool-dispatched skill typing gap, but this PR cannot merge as-is because it starts typing outside the existing suppression policy and lacks real Telegram behavior proof. The older concern that typing-start failures abort tool execution appears superseded on current main by the typing controller’s non-fatal start path.

## What This Changes

The PR adds an awaited typing-loop start before executing a tool-dispatched skill command in the inline action path.

## Best Possible Solution

Wire inline tool-dispatch typing through the existing policy-aware typing semantics, keep typing starts non-fatal, and add focused coverage for allowed and suppressed typing cases.

## Reproduction Assessment

Yes. Source inspection is enough for the PR defect: the patch starts typing unconditionally in the inline tool branch, while current main resolves typing suppression outside that early-return path.

## Solution Assessment

No. A direct controller call is not the best fix because it bypasses `typingMode: "never"`, explicit suppression, heartbeat, system, and internal-webchat policy; the fix should reuse or pass a policy-aware typing signaler.
