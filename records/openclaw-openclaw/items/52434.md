## Summary

Keep this PR open, but do not merge it as-is: the user-visible feature is still missing on current main, while the proposed implementation can start typing in paths where OpenClaw explicitly suppresses typing indicators.

## What This Changes

The PR adds an awaited `typing.startTypingLoop()` before `tool.execute()` in the tool-dispatched skill branch of `handleInlineActions`.

## Best Possible Solution

Inline tool-dispatch typing should reuse the existing resolved typing policy/signaler semantics, so tool-start feedback appears for allowed user runs while `typingMode: "never"`, heartbeat, system-event, internal-webchat, and explicit suppression paths stay silent.

## Reproduction Assessment

Yes. A focused unit test can exercise a tool-dispatched skill with `opts.suppressTyping` or `typingMode: "never"` and assert that the tool still executes while `onReplyStart` is not called.

## Solution Assessment

No. The direct `startTypingLoop()` call is not the best fix because inline actions return before the normal run typing policy is resolved; the safer fix is to resolve or pass policy-aware tool-start signaling into this branch.
