## Summary

Keep open: current main still lacks typing feedback for tool-dispatched skill commands, but the PR's direct `startTypingLoop()` call bypasses existing typing suppression policy, so it should not merge until routed through policy-aware tool-start signaling and covered by tests.

## What This Changes

The PR adds an awaited `typing.startTypingLoop()` before `tool.execute()` in the tool-dispatched skill branch of `handleInlineActions`.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing resolved typing policy/signaler semantics and add focused coverage for allowed and suppressed tool-dispatch runs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a tool-dispatched skill returns from `handleInlineActions` before the normal run path resolves typing policy, so a config or option that suppresses typing can still emit a typing signal with this PR.

## Solution Assessment

No. The direct `startTypingLoop()` call is not the best fix because this inline branch bypasses the normal policy resolver; a policy-aware tool-start signal is the narrower maintainable solution.
