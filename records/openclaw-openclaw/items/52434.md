## Summary

Keep open: current main still lacks inline tool-dispatch typing, but this PR is not merge-ready because the branch is conflicting, lacks required real Telegram proof, and starts typing without the existing policy-aware signaler path.

## What This Changes

The PR adds an awaited `typing.startTypingLoop()` before executing tool-dispatched skill commands in `src/auto-reply/reply/get-reply-inline-actions.ts`.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing typing policy/signaler semantics, keep typing-start failures non-fatal, add narrow regression coverage, and attach redacted Telegram proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head adds a raw awaited typing start inside the tool execution `try`, and current main resolves typing policy only after inline actions. The intended visible Telegram improvement itself is not proven in a real setup.

## Solution Assessment

No. A raw `typing.startTypingLoop()` call in `handleInlineActions` bypasses the existing typing policy/signaler contract; the maintainable fix should reuse that contract and keep typing failures off the tool-execution critical path.

## Review Findings

Overall correctness: patch is incorrect

  - body: On the PR head, `startTypingLoop()` can reject through the typing start guard, and this new await is inside the same `try` block as `tool.execute()`. A transient channel typing failure would return the error reply before the skill tool runs.
