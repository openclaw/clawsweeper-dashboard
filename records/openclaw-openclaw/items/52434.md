## Summary

Keep open: current main still lacks typing feedback for inline tool-dispatched skill commands, but this branch is not mergeable as-is because the PR-head change starts typing through a raw controller call outside typing policy, can abort tool execution on that head, and has no required real Telegram proof.

## What This Changes

The PR adds an awaited `typing.startTypingLoop()` before `tool.execute()` in the inline `dispatch.kind === "tool"` skill command path.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing typing policy/signaler semantics, keep typing-start failures non-fatal to skill execution, add narrow tests for allowed and suppressed typing, and attach redacted Telegram proof.

## Reproduction Assessment

Yes for the PR defects: live PR-head source shows the raw awaited typing start inside the tool execution `try` block and before any inline typing-policy resolution. The intended visible Telegram improvement itself is not proven in a real setup.

## Solution Assessment

No. A raw `typing.startTypingLoop()` call is not the best fix because inline actions run before normal typing policy resolution; use the policy-aware typing signaler and keep typing-start failures non-fatal.
