## Summary

Keep open: current main still lacks inline tool-dispatch typing, but this PR’s direct controller call bypasses the existing typing policy and the external Telegram-visible change still lacks real behavior proof.

## What This Changes

The PR inserts an awaited `typing.startTypingLoop()` before `tool.execute()` in the inline `dispatch.kind === "tool"` skill-command path.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing policy-aware typing signaler semantics, add focused coverage for allowed and suppressed cases, then validate with redacted real Telegram proof.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the patch starts the raw typing controller in an inline branch that returns before normal typing policy is resolved. The intended visible Telegram behavior itself is not proven in a real setup.

## Solution Assessment

No. A direct controller call is not the best fix because it can bypass `typingMode: "never"`, explicit suppression, heartbeat/system/internal-webchat policy, and source-reply suppression; the safer path is to reuse policy-aware typing signaler semantics for inline tool dispatch.
