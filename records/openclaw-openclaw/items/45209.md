## Summary

Keep this PR open, but it needs changes before merge: the core toggle approach matches the existing Feishu tool-category pattern, yet the new config key is not propagated to the checked-in channel config metadata that global validation uses.

## What This Changes

The PR adds a Feishu `tools.bitable` flag and a Bitable registration guard so `feishu_bitable_*` tools stay enabled by default but can be disabled by config.

## Best Possible Solution

Land the toggle after syncing generated channel config metadata and proving `tools.bitable:false` is accepted and removes `feishu_bitable_*` tools while default behavior stays enabled.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a high-confidence path for the review finding: global validation reads the generated channel schema, and that schema still rejects unknown Feishu `tools` keys such as `bitable`.

## Solution Assessment

Is this the best way to solve the issue? No, not yet; the runtime guard follows the right local pattern, but the solution is incomplete until generated metadata and proof are added.
