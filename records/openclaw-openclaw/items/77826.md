## Summary

Keep open: current main still has a source-level path where a config refresh with a stripped source config can activate an empty web-tools snapshot over a populated runtime snapshot, and there is no regression coverage for that transition.

## What This Changes

The issue asks OpenClaw to preserve active plugin-provided web and web-adjacent tools across runtime config writes instead of replacing their metadata with an empty fast-path snapshot.

## Best Possible Solution

Add regression coverage for active snapshot refresh, then make the refresh/fast-path path preserve or recompute prior populated webTools when the next source config is a stripped view while still honoring explicit web disablement or removal.

## Reproduction Assessment

Yes at source level: activate a snapshot with a configured web/plugin web surface, then refresh with a source config lacking `tools.web` and plugin `webSearch`/`webFetch`; current main prepares an empty fast-path `webTools` snapshot and activates it. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the runtime snapshot refresh or secrets fast-path boundary, because patching individual config writers or tool factories would leave other write paths able to clobber active web metadata.
