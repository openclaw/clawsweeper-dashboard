## Summary

Keep open. Current main and v2026.6.6 mitigate normal config writes by rereading canonical source config before final refresh, but the secrets runtime still has the fast-path/activation shape that can overwrite populated web-tool metadata if a stripped refresh source reaches it.

## What This Changes

Issue reports that runtime web/plugin tool metadata is cleared after config writes when the secrets runtime fast path sees a stripped config view.

## Best Possible Solution

Add a focused runtime-level regression guard so stripped or partial refreshes preserve populated web-tool metadata, while explicit empty web/plugin containers still clear it intentionally.

## Reproduction Assessment

Partly: source inspection shows the fast-path can still prepare empty web-tool metadata and activation still writes it unconditionally when handed a stripped source config. I did not run a live gateway reproduction, and current file-write APIs now reread canonical source before final refresh.

## Solution Assessment

No: current main mitigates normal config writes but does not enforce the runtime invariant directly. The best fix is a small, regression-tested guard in the secrets runtime preparation/activation boundary.
