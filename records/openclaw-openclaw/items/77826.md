## Label Justifications

- `P1`: The issue reports a regression that can remove web, memory, media, and plugin tools from subsequent agent sessions after ordinary config writes.
- `impact:session-state`: The failure is active runtime metadata state drift across config refreshes, causing fresh sessions to see a different tool surface than the resolved gateway state.
- `impact:auth-provider`: The affected metadata carries runtime web-search/web-fetch provider selection and SecretRef-derived provider state, which can be lost after refresh.

## Summary

Keep open: current main and the latest release still have the source-level path where a stripped config-write refresh can take the secrets fast path and replace populated runtime web-tool metadata with an empty metadata object. The linked repair PRs are all closed unmerged, and the open related plugin-tool lifecycle issue is adjacent rather than a safe canonical replacement for this config-refresh bug.

## What This Changes

Fix a runtime bug where config-write refreshes can replace populated plugin web-tool metadata with an empty fast-path snapshot, causing web and plugin tools to disappear from fresh sessions.

## Best Possible Solution

Add a runtime-boundary preservation guard so ambiguous fast-path stripped refreshes do not clear populated active web-tool metadata, while full resolver results and explicit empty web/plugin containers still clear it.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: activate populated runtime web-tool metadata, then process a config write whose `nextSourceConfig` omits web/plugin containers; the refresh can take the fast path and activation installs empty metadata.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable fix belongs at the secrets runtime snapshot boundary, preserving only ambiguous fast-path refreshes and leaving explicit clears and full resolver results authoritative.
