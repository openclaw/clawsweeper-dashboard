## Summary

Keep open. Current main and the latest release still allow a stripped no-web/no-SecretRef refresh to prepare empty web-tool metadata and activate it unconditionally, even though normal config-write paths now reduce exposure by rereading canonical source config before final refresh.

## What This Changes

The issue reports that plugin-provided web tools disappear after runtime config writes when a stripped config refresh reaches the secrets runtime fast path.

## Best Possible Solution

Add a focused runtime-boundary guard so ambiguous fast-path refreshes preserve populated web-tool metadata, while full resolver results and explicit empty web/plugin containers remain authoritative clears.

## Reproduction Assessment

Yes, source-level: activate populated web-tool metadata, then refresh with a no-web/no-SecretRef source config and the current fast path can prepare empty metadata that activation installs. I did not run a live gateway session in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is at the secrets runtime preparation/activation boundary, preserving ambiguous fast-path refreshes without changing config schemas or normal canonical-source write handling.
