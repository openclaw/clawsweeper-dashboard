## Label Justifications

- `P1`: The issue reports a regression that can remove web, memory, media, and plugin tools from subsequent agent sessions after ordinary config writes.
- `impact:session-state`: The failure is active runtime metadata state drift across config refreshes, so fresh sessions can see a different tool surface than the resolved gateway state.
- `impact:auth-provider`: The affected metadata carries runtime web-search/web-fetch provider selection and SecretRef-derived provider state that can be lost after refresh.

## Summary

Keep open: current main and the latest release still have the source-level path where a stripped config-write refresh can take the secrets fast path and replace populated web-tool metadata with an empty metadata object. The linked repair PRs were closed unmerged, and the open adjacent plugin-tool lifecycle issue is not a safe canonical replacement for this config-refresh bug.

## What This Changes

The issue reports that config-write refreshes can strip runtime plugin web-tool metadata, making web/search/fetch and plugin-provided tools disappear from later sessions until restart.

## Best Possible Solution

Add a durable runtime-boundary provenance guard so ambiguous stripped fast-path refreshes preserve populated active web-tool metadata, while full resolver results and explicit web/plugin containers remain authoritative clears.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: activate populated runtime web-tool metadata, then process a config write whose nextSourceConfig omits web/plugin containers; the refresh can take the fast path and activation installs empty metadata.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable fix belongs at the secrets runtime snapshot boundary, preserving only ambiguous fast-path refreshes and leaving explicit clears and full resolver results authoritative.
