## Summary

Keep open. Current main still has the source-level stripped-refresh path: a no-web/no-SecretRef config can prepare empty web-tool metadata, and activation still publishes that empty metadata over the active runtime state; the related merged plugin-registry PR fixed an adjacent registry-retention problem, not this secrets-runtime refresh invariant.

## What This Changes

The issue reports that runtime config writes can replace populated plugin web-tool metadata with an empty snapshot, causing web and plugin tools to disappear from fresh sessions.

## Best Possible Solution

Add a focused secrets-runtime guard that preserves populated web-tool metadata only for ambiguous stripped fast-path refreshes, while keeping full resolver runs and explicit empty web/plugin containers authoritative clears.

## Reproduction Assessment

Yes for source reproduction: current main still lets a no-web/no-SecretRef source config prepare empty web-tool metadata and activation installs it unconditionally. No live gateway repro was run in this read-only review.

## Solution Assessment

No, current main is only partially mitigated by canonical-source rereads for normal writes. The maintainable fix belongs at the secrets runtime preparation/activation boundary so stripped refreshes cannot clobber active metadata while explicit clears still work.
