## Summary

Keep open: current main still has a source-level path where a config-write refresh can replace populated runtime web-tool metadata with an empty fast-path snapshot, matching the reported tool loss.

## What This Changes

The issue asks OpenClaw to preserve web and plugin tool availability across runtime config writes.

## Best Possible Solution

Keep this issue open and fix the refresh path so incomplete non-web config writes cannot clear populated runtime web-tool metadata, while still honoring explicit web/plugin tool removal or disablement.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: activate a snapshot with populated webTools, then refresh from a source config without `tools.web` or plugin web config so the fast path returns empty metadata and activation overwrites the active state. I did not run a live gateway session in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow runtime/config-refresh guard that preserves or recomputes prior populated webTools when the incoming source config is an incomplete writer view, without adding a config knob or plugin-specific policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77826. Current main can drop active web/plugin tool metadata after config writes: `activateSecretsRuntimeSnapshot` registers a refresh handler, `finalizeRuntimeSnapshotWrite` passes `nextSourceConfig`, `prepareSecretsRuntimeSnapshot` may take `canUseSecretsRuntimeFastPath`, and activation then writes empty `webTools` into active runtime state. Implement a narrow guard in the secrets runtime/config-refresh path so an incomplete writer source config cannot clear previously populated runtime webTools; still honor explicit removal/disablement of `tools.web.search`, `tools.web.fetch`, or plugin `webSearch`/`webFetch` config. Add focused regression coverage around activated snapshots plus refresh/finalize behavior. Do not add a config option, plugin-specific IDs, broad startup migrations, or unrelated plugin-loader changes. Add a single-line `CHANGELOG.md` Fixes entry because this is user-facing.
