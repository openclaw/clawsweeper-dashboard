## Summary

Keep this PR open: the browser proxy hardening is addressing a real authorization mismatch and the terminal proof is useful, but the current patch still exposes a generic plugin-runtime scope elevation path that can mint an admin-scoped synthetic client from `runtime.nodes.invoke` params.

## What This Changes

The PR adds an admin-scope check before direct `node.invoke(browser.proxy)` dispatch and updates Browser, Google Meet, plugin runtime, CLI runtime, and regression tests to request admin-scoped browser proxy dispatch.

## Best Possible Solution

Keep the direct `browser.proxy` admin gate, but route first-party Browser and Google Meet through a private owner-approved internal dispatch or command-specific policy without adding arbitrary requested scopes to `PluginRuntime.nodes.invoke`.

## Reproduction Assessment

Yes. Source inspection shows `browser.request` is admin-scoped while current `node.invoke` is write-scoped, and the PR body supplies before/after terminal output for a scoped Gateway device-token test.

## Solution Assessment

No. The direct node guard is the right direction, but the current solution exposes generic plugin runtime scope elevation instead of keeping the admin browser proxy dispatch private or explicitly authorized.
