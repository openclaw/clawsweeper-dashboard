## Summary

Keep open: the PR targets a real gateway authorization mismatch, but the current patch would reject existing first-party node-backed browser proxy flows that still call `browser.proxy` through write-scoped `node.invoke`.

## What This Changes

The PR adds an `operator.admin` check before forwarding direct `node.invoke(browser.proxy)` calls and adds gateway tests for write-scoped rejection and admin-scoped allow.

## Best Possible Solution

Keep the admin-only direct `browser.proxy` boundary, but update official browser and Google Meet/runtime proxy paths to use an admin-scoped route and add regression coverage that proves official flows still work.

## Reproduction Assessment

Yes, by source inspection and supplied terminal proof: current main has `node.invoke` at `operator.write` and lacks a non-mutating `browser.proxy` admin gate, while the PR body shows before/after gateway behavior. This read-only review did not run the test command.

## Solution Assessment

No, not as-is. The admin boundary is the right direction, but the patch also needs to preserve existing official browser and Google Meet proxy routes that still go through write-scoped `node.invoke` defaults.
