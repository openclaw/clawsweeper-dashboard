## Summary

Keep this PR open: the core authorization hardening is well-motivated, but the latest patch introduces a security-boundary defect by allowing the plugin CLI runtime to forward arbitrary requested operator scopes, and the external-PR proof is still test-only rather than current-head real runtime proof.

## What This Changes

The PR makes direct `node.invoke(browser.proxy)` require `operator.admin`, adds a `scopes` request field to plugin node invocation, and updates bundled browser and Google Meet callers/tests to request admin scope.

## Best Possible Solution

Keep the admin-scope alignment, but gate or remove CLI-runtime scope elevation for untrusted plugins, add focused regression coverage for that boundary, then require gateway/security owner acceptance plus current-head real runtime proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main registers `browser.request` as admin-scoped while direct `node.invoke` is write-scoped and only blocks persistent browser-profile mutations. I did not run a live Gateway repro in this read-only review.

## Solution Assessment

No, not as currently patched: the direct admin gate is the right shape, but the CLI plugin runtime should not forward requested admin scopes without the same trusted-plugin gate used by the in-process path. The safer fix keeps the hardening and narrows the new scope request contract before merge.

Full review comments:

- **[P1] Gate CLI node-invoke scope requests to trusted plugins:** `src/plugins/cli-gateway-nodes-runtime.ts:56`
  - body: This forwards `params.scopes` from every plugin CLI runtime call into `callGateway`, while the in-process path only honors requested scopes after checking that the plugin is bundled or trusted-official. Because `api.runtime.nodes.invoke` is also available from plugin CLI commands, a third-party plugin can request `operator.admin` here and bypass the trusted-plugin gate added in `server-plugins.ts`. Please either ignore requested scopes in the generic CLI runtime or thread plugin identity/trust through this path and add coverage that an untrusted plugin CLI cannot self-mint admin for `browser.proxy`.
