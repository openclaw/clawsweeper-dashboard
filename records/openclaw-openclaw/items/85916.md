## Summary

Keep open: the admin gate is directionally correct and the current-head terminal proof improved, but the branch still forwards requested node-invoke scopes from every plugin CLI command without the trusted-official gate used for in-process plugins, leaving a security-boundary and plugin API decision unresolved.

## What This Changes

The branch makes direct `node.invoke(browser.proxy)` require `operator.admin`, adds an optional `scopes` field to plugin node invocation, and updates bundled browser and Google Meet callers/tests to request admin scope.

## Best Possible Solution

Land only after the CLI scope request path is either removed or gated by trusted plugin identity, with regression coverage for bundled and third-party in-process and CLI callers plus explicit gateway/security owner acceptance for the compatibility break.

## Reproduction Assessment

Yes, from source: current main registers `browser.request` as `operator.admin`, while `node.invoke` is `operator.write` and only blocks persistent browser profile mutations before forwarding other `browser.proxy` calls. I did not run the tests in this read-only review.

## Solution Assessment

No: the direct admin gate is the right shape, but the current branch leaves the CLI plugin runtime able to forward requested admin scopes without the trusted-plugin check used in-process. The safer solution is to gate or remove CLI scope forwarding and add focused third-party CLI coverage.
