## Summary

Keep open: current main and the latest release still route the bundled Claude CLI command through the supervisor without adding the Windows `.cmd` shim, and this issue already has an open closing PR that should be reviewed rather than cleanup-closed.

## What This Changes

Windows Claude CLI provider should spawn the npm-installed Claude Code command on Windows instead of failing with `spawn claude ENOENT`.

## Best Possible Solution

Land a narrow supervisor Windows shim fix for bundled CLI backend commands, with focused regression tests and redacted Windows proof, then close this issue through the merged PR.

Yes at source level: current main passes bare `claude` into the supervisor and the supervisor Windows shim allowlist does not include it. I did not run a live Windows reproduction in this read-only review, but the reporter's Windows logs match the current code path.
