## Label Justifications

- `P1`: The regression makes the Claude CLI agent workflow unusable for affected Windows users on every request.
- `impact:auth-provider`: The configured Claude CLI provider fails before model execution because its executable cannot be launched.

## Summary

Current main still contains the Windows command-resolution defect, and this issue is paired with open implementation PRs. Keep it open until maintainers select and land a fix that handles both npm shim discovery and Node’s Windows batch-file execution requirements safely.

## What This Changes

Fix Windows Claude CLI provider launches that fail with `spawn claude ENOENT` even though the npm-installed CLI is available from the user’s shell.

## Best Possible Solution

Resolve bundled npm CLI shims in the core supervisor, invoke Windows batch shims through a narrowly controlled mechanism, add focused Claude and Gemini regression coverage, and prove a complete Claude agent turn on a supported Windows and Node setup.

## Reproduction Assessment

Yes. Current main omits the bundled bare `claude` command from Windows shim resolution, and the reporter supplied a matching, repeatable Windows ENOENT path; this review did not independently execute it on Windows.

## Solution Assessment

No, neither proposed direction is fully proven yet. The supervisor is the correct boundary, but the whitelist-only implementation does not complete Node’s documented `.cmd` launch requirements, while the shell-based alternative needs explicit argument-safety review and live Windows proof.
