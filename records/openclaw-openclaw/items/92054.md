## Summary

Keep this open: current main and the latest release still route the bundled Claude CLI bare command through a supervisor Windows shim whitelist that excludes `claude`, and the issue is already paired with open fix PRs that should be reviewed rather than replaced by another cleanup action.

## What This Changes

Windows 11 report that the Claude CLI provider fails before model execution with `spawn claude ENOENT` even though Claude Code is installed and available on `PATH`.

## Best Possible Solution

Review and land the narrowest Windows command-resolution fix that restores the shipped Claude CLI provider on Windows, with regression coverage and enough Windows runtime proof to settle the `.cmd` execution shape.

## Reproduction Assessment

Yes, at source level: the bundled Claude backend passes bare `claude` into the process supervisor, and current Windows shim resolution does not append `.cmd` for that command. I did not run a live Windows repro in this read-only review.

## Solution Assessment

Unclear until the linked fix PRs are reviewed: a supervisor whitelist fix directly addresses the reported ENOENT, but maintainers should decide whether the broader Windows `.cmd` shell behavior is also required.
