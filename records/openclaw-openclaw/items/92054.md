## Summary

Current `main` and the latest release still have the failing supervisor path: the Claude backend supplies bare `claude`, the CLI runner sends it to the process supervisor, and the supervisor Windows shim whitelist does not include `claude`. This should stay open because an open linked fix PR already uses closing syntax for this issue, and the canonical sibling issue remains open for the same Windows `.cmd` spawn failure.

## What This Changes

The issue requests fixing Windows Claude CLI provider launches that fail with `spawn claude ENOENT` even though the Claude CLI is installed and works from a shell.

## Best Possible Solution

Land one narrow supervisor Windows shim fix with focused regression coverage and redacted Windows proof, then close this issue and the canonical sibling as implemented.

## Reproduction Assessment

Yes, at source level: current `main` passes the bundled backend's bare `claude` command into a supervisor Windows shim resolver that does not whitelist `claude`. I did not run a live Windows machine in this read-only cleanup pass.

## Solution Assessment

Yes on the boundary, but maintainers still need to choose the linked implementation: a narrow supervisor shim fix with tests is the right path, while shell-mode behavior needs explicit security approval.
