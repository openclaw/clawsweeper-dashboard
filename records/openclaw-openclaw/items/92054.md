## Summary

This is a fresh, well-supported reproduction of the same unresolved Windows Claude CLI shim bug already tracked by https://github.com/openclaw/openclaw/issues/91489, so keeping a second issue open would split the fix and security discussion.

## What This Changes

Fix the Windows 11 Claude CLI backend so an npm-installed `claude` command launches successfully instead of failing with `spawn claude ENOENT`.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/91489 as the canonical report, preserve this issue’s Windows 11 and Node 24 reproduction there, and implement a shared shell-safe Windows shim resolver for both supervisor-backed Claude execution modes.

Yes, at source level with high confidence: the reporter shows direct spawn failing while shell resolution succeeds, and current main still sends bare `claude` through an allowlist that omits it. A live Windows run was not performed in this read-only review.

Closing this duplicate is the best tracking outcome, while the code fix should use the canonical issue’s shared, shell-safe Windows invocation direction rather than enabling generic `shell: true`.
