## Summary

Keep open. Current main and the latest release have fixed and documented the `api.on(...)` typed-hook path, but `api.registerHook("message_received", ...)` still silently registers an internal hook entry that the typed `message_received` runtime dispatch never reads, so the central SDK compatibility problem remains necessary.

## What This Changes

The issue asks for plugin hook registrations for inbound messages and prompt-building to dispatch reliably so plugins can observe messages or inject context.

## Best Possible Solution

Choose and implement one explicit SDK contract for typed names passed to `registerHook`, preferably a tested diagnostic or compatibility bridge that prevents silent success while keeping `api.on(...)` as the documented typed-hook surface.

## Reproduction Assessment

Yes, source-based: current main still stores `api.registerHook("message_received")` in the internal hook registry while `message_received` runtime dispatch checks `typedHooks`. I did not run a live gateway plugin under the read-only review constraints.

## Solution Assessment

Unclear. The supported `api.on(...)` path is now documented and wired, but maintainers still need to choose whether `registerHook` typed names should be bridged, rejected, or warned as an internal-hook-only surface.
