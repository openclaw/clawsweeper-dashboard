## Label Justifications

- `P2`: The remaining hook-dispatch mismatch affects non-bundled plugin workflows but is not a crash, outage, or emergency core-runtime failure.
- `impact:message-loss`: Plugin handlers for documented message events can register without receiving the corresponding message event.
- `impact:session-state`: The reported use case includes plugin context capture and next-turn prompt injection, so missed hooks can leave plugin-maintained session context stale.
- `impact:security`: The fix path crosses non-bundled plugin trust gates and conversation-content access, and users may rely on hooks for audit or outbound policy enforcement.

## Summary

Keep open. Current main has addressed the earlier `api.on("before_prompt_build")` no-op and now documents the conversation-access gate, but `registerHook("message_received")` still lands in the internal-hook map under the raw underscore key while inbound dispatch checks typed `message_received` hooks and internal `message:received` listeners separately.

## What This Changes

The issue asks OpenClaw to make plugin hook handlers registered for `message_received` and prompt/context injection dispatch reliably for non-bundled plugins instead of registering silently and never firing.

## Best Possible Solution

Keep `api.on(...)` as the typed plugin-hook API, then make `registerHook(...)` safe by maintainer decision: either add explicit compatibility handling for typed hook names or emit a clear diagnostic/error, while leaving outbound hook coverage to the existing canonical umbrella.

## Reproduction Assessment

Yes, source-based. A plugin calling `api.registerHook("message_received", ...)` is stored under raw internal key `message_received`, while current inbound dispatch checks typed `message_received` and internal `message:received` separately.

## Solution Assessment

Unclear until maintainers choose the API compatibility policy. The narrow maintainable options are to preserve `api.on(...)` as the typed hook path and either alias typed names passed to `registerHook(...)` or warn/error clearly instead of accepting an inert registration.
