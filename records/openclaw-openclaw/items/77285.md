## Summary

Keep open. Current main wires the typed `api.on` path for `message_received` and `before_prompt_build`, but `api.registerHook("message_received")` still registers a raw internal-hook key that inbound message dispatch never looks up; the remaining fix needs maintainer direction on compatibility aliasing versus diagnostics/docs.

## What This Changes

The issue asks OpenClaw to dispatch plugin `registerHook("message_received")` handlers and `api.on("before_prompt_build")` listeners so plugins can capture inbound messages and inject prompt context.

## Best Possible Solution

Keep `api.on` as the documented typed hook API, then either add a backward-compatible `registerHook` alias/diagnostic for typed hook names or update validation/docs so `registerHook` users are steered to `message:received` for internal hooks.

## Reproduction Assessment

Yes for the remaining `registerHook` half by source inspection: `registerHook` stores `message_received`, while internal inbound dispatch only matches `message` and `message:received`. No for the `api.on("before_prompt_build")` half on current main, where `api.on` is wired and prompt assembly dispatches the typed hook.

## Solution Assessment

Unclear. The narrow fix depends on maintainer API direction: compatibility aliasing would preserve the reporter's call shape, while validation/docs would preserve the existing split between typed plugin hooks and internal hooks.
