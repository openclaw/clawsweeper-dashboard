## Summary

Keep open. Current main has fixed and documented the `api.on("before_prompt_build")` typed-hook path, but `registerHook("message_received")` still registers an underscore event key that the inbound internal-hook dispatcher does not check.

## What This Changes

The issue asks for `registerHook("message_received")` and `api.on("before_prompt_build")` plugin handlers to dispatch reliably so third-party plugins can observe messages and inject prompt context.

## Best Possible Solution

Keep `api.on(...)` as the typed plugin-hook API, and add explicit compatibility handling or diagnostics for typed hook names passed to `registerHook(...)` with matching docs and regression coverage.

## Reproduction Assessment

Yes, source-based. Current main stores `registerHook("message_received")` under `message_received`, while inbound dispatch checks typed `message_received` separately and internal hooks as `message` or `message:received`.

## Solution Assessment

Unclear. The current `api.on(...)` implementation is the maintainable typed-hook path, but the remaining `registerHook(...)` behavior needs a maintainer decision between compatibility aliasing, diagnostics, or docs-only clarification.
