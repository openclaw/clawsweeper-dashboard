## Summary

Keep open. Current main fixes the typed `api.on` path for `message_received` and `before_prompt_build`, but the reported `registerHook("message_received")` call still registers an internal-hook key that inbound message dispatch will not trigger, so the remaining behavior needs maintainer API-direction judgment.

## What This Changes

The issue asks OpenClaw to dispatch plugin handlers registered for inbound messages and prompt-build context injection instead of accepting them silently and never calling them.

## Best Possible Solution

Keep `api.on` as the documented typed plugin hook API, then either add an explicit compatibility alias or diagnostic for `registerHook("message_received")` so plugin authors are not left with a silent dead registration.

## Reproduction Assessment

Yes for the `registerHook("message_received")` half by source inspection: it stores the raw `message_received` key, while inbound internal dispatch checks `message` and `message:received`. The `api.on("before_prompt_build")` half is not reproduced on current main because `api.on` is wired and prompt assembly calls the typed hook runner.

## Solution Assessment

Unclear. The narrowest code change depends on whether maintainers want `registerHook` to accept typed-hook aliases for compatibility or to warn/reject those names and steer authors to `api.on` and `message:received` internal-hook syntax.
