## Summary

Keep open. Current main and v2026.6.10 still keep `api.registerHook(...)` on the internal hook path while typed `message_received` dispatch reads only `registry.typedHooks`, so the silent SDK mismatch remains despite `api.on(...)` now being documented and wired as the supported typed-hook path.

## What This Changes

The issue asks OpenClaw to stop silently accepting plugin `registerHook("message_received", ...)` handlers that never dispatch, and to clarify or fix the related typed hook path for prompt/context injection.

## Best Possible Solution

Choose and document one SDK contract: either bridge typed hook names passed to `registerHook(...)` into typed dispatch, or emit a visible warning/rejection so typed-name registrations cannot silently no-op.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source: `registerHook("message_received")` registers an internal event key, while runtime `message_received` dispatch only consults `registry.typedHooks`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the SDK contract; the supported `api.on(...)` path works, but `registerHook(...)` typed-name calls still need either compatibility or visible diagnostics.
