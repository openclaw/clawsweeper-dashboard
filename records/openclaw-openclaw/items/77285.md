## Summary

Keep open. Current main has fixed and documented the `api.on(...)` typed-hook path, but the legacy `registerHook(...)` path still silently records typed names as internal hook keys, and Telegram direct streaming still has a source-visible path that can finalize a reply without the hook-bearing send path.

## What This Changes

The issue reports silent non-dispatch of plugin message hooks registered through `registerHook(...)`, plus related typed-hook dispatch gaps for non-bundled plugins.

## Best Possible Solution

Keep `api.on(...)` as the documented typed hook API, add maintainer-approved compatibility or diagnostics for typed names passed to `registerHook(...)`, and make Telegram direct/streaming delivery pass through one hook-bearing pre-send path.

## Reproduction Assessment

Yes. Source inspection on current main shows `registerHook("message_received")` still registers a raw internal key that the `message:received` dispatcher will not check, and the Telegram streaming path has a source-visible pre-send hook bypass.

## Solution Assessment

Unclear. The maintainable end state is clear, but the exact SDK compatibility behavior for `registerHook(...)` and the canonical Telegram/outbound delivery path need maintainer judgment before an automated fix should choose one.
