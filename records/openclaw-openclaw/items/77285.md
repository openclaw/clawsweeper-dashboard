## Summary

Keep open. Current main and the latest release still split `registerHook` into the internal hook registry while typed `message_received` dispatch reads only `typedHooks`; `api.on(...)` is now wired and documented, but the silent-success behavior for typed names passed to `registerHook` still needs maintainer compatibility direction.

## What This Changes

The issue reports that third-party plugin hooks registered with `registerHook("message_received")` never dispatch, and that `api.on("before_prompt_build")` was previously a no-op for prompt injection.

## Best Possible Solution

Keep `api.on(...)` as the documented typed-hook surface and add an explicit, tested contract for typed hook names passed to `registerHook`, ideally a clear diagnostic or compatibility bridge that prevents silent success.

## Reproduction Assessment

Yes, source-level. On current main, a plugin using `api.registerHook("message_received", ...)` lands in the internal hook registry, while inbound `message_received` dispatch checks only typed-hook registrations; I did not run a live gateway repro under the read-only review constraint.

## Solution Assessment

Unclear. `api.on(...)` is now the narrow documented typed-hook path, but the best remaining fix depends on maintainer direction for the shipped `registerHook` API contract: bridge typed names, reject/warn on them, or deprecate the confusing usage.
