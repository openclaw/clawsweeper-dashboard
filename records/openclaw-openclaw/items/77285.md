## Summary

Keep open. Current main fixes the `api.on("before_prompt_build")` typed-hook path and dispatches typed `message_received` hooks through `api.on(...)`, but `registerHook("message_received")` still registers a raw internal-hook key that current internal dispatch will not match.

## What This Changes

The issue asks for plugin handlers registered through `registerHook("message_received")` and `api.on("before_prompt_build")` to dispatch instead of being silently accepted and never run.

## Best Possible Solution

Keep `api.on(...)` as the documented typed hook API, and add an explicit compatibility alias or diagnostic for typed names passed to `registerHook(...)` so this no longer succeeds silently without dispatch.

## Reproduction Assessment

Yes, source-based for the remaining issue: `registerHook("message_received")` stores the raw underscore key, while message receive dispatch checks typed hooks separately and internal hooks as `message` / `message:received`. A live gateway run was not performed.

## Solution Assessment

Unclear. The current `api.on(...)` typed-hook path is the best documented path, but the remaining fix needs maintainer API judgment between backward-compatible aliasing, warning/rejecting typed names in `registerHook(...)`, or docs-only clarification.
