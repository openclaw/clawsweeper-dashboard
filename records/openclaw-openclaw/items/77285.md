## Summary

Keep open. Current main fixes and documents the `api.on(...)` typed-hook path, but `registerHook("message_received")` still stores an underscore event key that the inbound internal-hook dispatch path does not check, so the remaining silent non-dispatch behavior is still source-proven.

## What This Changes

The issue asks for plugin `registerHook("message_received")` and `api.on("before_prompt_build")` registrations to dispatch instead of silently accepting handlers that never run.

## Best Possible Solution

Keep `api.on(...)` as the typed plugin-hook API, and add explicit compatibility handling or diagnostics for typed hook names passed to `registerHook(...)` so authors are not left with a silently non-dispatching registration.

## Reproduction Assessment

Yes, source-based: current main stores `registerHook("message_received")` under the raw underscore key, while inbound dispatch checks typed `message_received` separately and internal hooks as `message` / `message:received`. A live gateway run was not performed.

## Solution Assessment

Unclear. The current `api.on(...)` implementation is the maintainable typed-hook path, but the remaining `registerHook(...)` behavior needs maintainer judgment between compatibility aliasing, diagnostics, or docs-only clarification.
