## Summary

Keep open. Current main has the typed `api.on` path wired for `message_received` and `before_prompt_build`, but the exact `registerHook("message_received")` call still registers an internal-hook key that current message dispatch will not trigger; the remaining fix needs maintainer API judgment around compatibility aliasing versus validation/docs.

## What This Changes

A plugin developer asks for `registerHook("message_received")` and `api.on("before_prompt_build")` handlers to dispatch so plugins can capture inbound messages and inject prompt context.

## Best Possible Solution

Keep the typed `api.on` path as the documented plugin hook API, then either add a backward-compatible `registerHook` alias/diagnostic for typed hook names or clarify the docs so `registerHook` users choose `message:received` for internal hooks.

## Reproduction Assessment

Yes for the `registerHook("message_received")` half: source inspection shows that raw key is registered but internal dispatch only checks `message` and `message:received`. No for the `api.on("before_prompt_build")` half on current main, where `api.on` is wired to typed hooks and prompt construction calls the hook runner.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on whether maintainers want `registerHook` to accept typed hook aliases or to reject/diagnose them and steer plugin authors to `api.on`.
