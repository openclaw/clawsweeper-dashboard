## Summary

Keep open. Current main has fixed and documented the `api.on(...)` typed-hook path, and the later Telegram `message_sending` bypass is now tracked by a narrower open issue, but the remaining `registerHook(...)` silent typed-name mismatch still exists in source and needs a maintainer SDK compatibility decision.

## What This Changes

The issue reports that plugin `registerHook("message_received")` succeeds but never dispatches, and originally also reported `api.on("before_prompt_build")` as a runtime no-op.

## Best Possible Solution

Keep this issue as the canonical `registerHook(...)` typed-name compatibility tracker, while routing the newer Telegram `message_sending` delivery-path problem to https://github.com/openclaw/openclaw/issues/92374 and the conversation-access diagnostics problem to https://github.com/openclaw/openclaw/issues/79421.

## Reproduction Assessment

Yes, source-based. Current main still stores `api.registerHook("message_received")` in the internal hook registry, while inbound runtime dispatch checks typed `registry.typedHooks` for `message_received` and internal listeners under `message` or `message:received`.

## Solution Assessment

Unclear. The `api.on(...)` typed-hook path is now the maintainable supported path, but the remaining `registerHook(...)` silent-success behavior needs a maintainer decision between a compatibility bridge, explicit diagnostics, or a docs-only contract boundary.
