## Summary

Keep open. Current main has the typed `api.on` path wired for `message_received` and `before_prompt_build`, but `api.registerHook("message_received")` still registers the raw name into the internal hook system, whose message dispatch only checks `message` and `message:received`.

## What This Changes

The issue reports that plugin `registerHook("message_received")` and `api.on("before_prompt_build")` registrations succeed but never dispatch, blocking plugin-driven prompt context injection.

## Best Possible Solution

Keep `api.on` as the documented typed-hook API, then decide whether `registerHook` should reject typed hook names with diagnostics or support explicit compatibility aliases with tests and docs.

## Reproduction Assessment

Yes for the remaining `registerHook("message_received")` behavior: source inspection shows the raw key is registered but inbound internal dispatch only looks for `message` and `message:received`. The `api.on("before_prompt_build")` half is not reproducible on current main because the typed hook is wired and dispatched.

## Solution Assessment

Unclear. The narrowest maintainable repair depends on whether maintainers want `registerHook` to accept typed hook aliases, reject them with a diagnostic, or remain internal-only with clearer docs.
