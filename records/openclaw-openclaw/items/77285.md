## Summary

Keep open. Current main wires the typed `api.on` paths, but `registerHook("message_received")` still records an inert internal-hook key, so the remaining report is a real API/docs compatibility problem rather than something already implemented.

## What This Changes

The issue reports that plugin `registerHook("message_received")` handlers never dispatch and that `api.on("before_prompt_build")` silently no-ops at runtime.

## Best Possible Solution

Preserve `api.on` as the typed plugin-hook contract and add an explicit compatibility, validation, or documentation path so known underscore typed names cannot register silently as inert internal events.

## Reproduction Assessment

Yes for the remaining `registerHook("message_received")` path: source inspection shows it registers the raw key, while dispatch emits `message` / `message:received`. No for the `api.on("before_prompt_build")` half on current main, where the registry and prompt builder are wired.

## Solution Assessment

Unclear. The narrow fix is straightforward, but maintainers should choose whether compatibility aliasing, a warning/error, or docs cleanup is the intended SDK contract.
