## Label Justifications

- `P2`: This is a source-proven plugin SDK/runtime bug with a real workaround and limited blast radius, but it needs an API-contract decision before implementation.
- `impact:message-loss`: Handlers registered through the wrong public-looking hook surface can silently miss inbound or outbound hook events.
- `impact:session-state`: Plugins using these hooks for context capture or prompt injection can fail to keep session context aligned with received messages.

## Summary

Keep open. Current main fixes the `api.on(...)` typed-hook path and documents the conversation-access gate, but `api.registerHook("message_received", ...)` still registers an internal hook key that the typed plugin message dispatcher will not call.

## What This Changes

Plugin authors report that `api.registerHook("message_received", ...)` registers without error but never dispatches, leaving typed plugin hook registration and prompt-injection paths confusing or inert.

## Best Possible Solution

Keep this issue as the canonical SDK compatibility tracker for typed hook names passed to `registerHook(...)`; resolve it with a maintainer-approved compatibility bridge or explicit diagnostic/docs boundary, and route broader outbound hook coverage to the existing outbound lifecycle trackers.

## Reproduction Assessment

Yes, source-based. Current main still stores `registerHook("message_received")` as a raw internal event key while the inbound plugin dispatcher checks typed `message_received` registrations in `registry.typedHooks`.

## Solution Assessment

Unclear. `api.on(...)` is the maintainable typed-hook path, but the remaining silent `registerHook(...)` behavior needs a maintainer API decision between compatibility aliasing, diagnostics, or docs-only clarification.
