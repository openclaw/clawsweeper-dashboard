## Summary

Keep open. Current `main` appears to have the typed `api.on` paths wired, but `registerHook("message_received")` still registers a raw internal-hook event name that inbound dispatch never matches; the remaining fix needs maintainer API judgment around aliasing, diagnostics, or docs.

## What This Changes

The issue asks OpenClaw to dispatch plugin handlers registered with `registerHook("message_received")` and to make `api.on("before_prompt_build")` invoke typed prompt hooks.

## Best Possible Solution

Keep `api.on` as the documented typed plugin hook API, then make the `registerHook` boundary explicit by either adding a safe compatibility alias/diagnostic for typed hook names or documenting rejection clearly.

## Reproduction Assessment

Yes for the `registerHook("message_received")` half: source inspection shows the raw key is registered but dispatch only checks `message` and `message:received`. No for the `api.on("before_prompt_build")` half on current `main`, where `api.on` is wired to typed hooks and prompt construction calls the hook runner.

## Solution Assessment

Unclear. The narrowest maintainable fix depends on whether maintainers want backward-compatible `registerHook` aliases, a warning/error for typed names on the internal-hook API, or docs-only clarification.
