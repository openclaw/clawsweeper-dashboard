## Summary

Keep open. Current main fixes and documents the `api.on(...)` typed-hook path, but `registerHook("message_received")` still registers a raw underscore key that the internal message dispatch path does not check, so the remaining silent non-dispatch behavior is source-proven and needs an API compatibility decision.

## What This Changes

The issue asks OpenClaw to dispatch plugin `registerHook("message_received")` handlers and avoid silent no-op behavior for prompt-related plugin hooks.

## Best Possible Solution

Keep `api.on(...)` as the typed plugin-hook API, and add explicit compatibility handling or diagnostics for typed hook names passed to `registerHook(...)` so plugin authors are not left with a successful but non-dispatching registration.

## Reproduction Assessment

Yes, source-based: current main stores `registerHook("message_received")` under the raw underscore key, while inbound internal hook dispatch checks `message` and `message:received`. A live gateway run was not performed.

## Solution Assessment

Unclear. The current `api.on(...)` implementation is the maintainable typed-hook path, but the remaining `registerHook(...)` behavior needs maintainer judgment between compatibility aliasing, diagnostics, or docs-only clarification.
