## Summary

Keep open. Current main already wires the documented `api.on(...)` typed hook path for `message_received` and `before_prompt_build`, but `api.registerHook("message_received", ...)` still registers a raw internal-hook key that inbound message dispatch will not trigger.

## What This Changes

The issue asks for plugin `registerHook("message_received")` and `api.on("before_prompt_build")` handlers to dispatch reliably so plugins can capture message context and inject prompt context.

## Best Possible Solution

Keep `api.on(...)` as the documented typed plugin hook API, then add either a compatibility alias/diagnostic for typed names passed to `registerHook` or clearer validation/docs that steer internal-hook users to `message:received`.

## Reproduction Assessment

Yes for the remaining `registerHook("message_received")` path: current source shows it registers a raw internal key while inbound dispatch emits `message`/`received`. No for the `api.on("before_prompt_build")` half on current main, where `api.on` is wired and prompt assembly calls the typed hook runner.

## Solution Assessment

Unclear. The narrow fix depends on maintainer API policy: alias typed hook names through `registerHook`, warn/reject them, or rely on docs and `api.on(...)` only.
