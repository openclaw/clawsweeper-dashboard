## Summary

Keep open. Current main wires the typed `api.on` paths for `message_received` and `before_prompt_build`, but `registerHook("message_received")` still registers a raw internal-hook key that inbound message dispatch will not trigger.

## What This Changes

The issue requests a working plugin path for inbound message hooks and before-prompt context injection, especially `registerHook("message_received")` and `api.on("before_prompt_build")`.

## Best Possible Solution

Keep `api.on` as the documented typed plugin-hook API, then make `registerHook` non-silent for typed hook names through either a backward-compatible alias or a clear diagnostic/docs path.

## Reproduction Assessment

Yes for the `registerHook("message_received")` half by source inspection: it registers `message_received`, while current inbound internal dispatch only matches `message` and `message:received`. The `api.on("before_prompt_build")` half is not reproduced on current main because prompt assembly now calls the typed hook runner.

## Solution Assessment

Unclear. The current `api.on` path is the documented plugin-hook solution, but maintainers need to decide whether `registerHook` should accept typed hook aliases or reject/diagnose them and steer authors to `api.on`/`message:received`.
