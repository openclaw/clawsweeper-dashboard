- `impact:session-state`: Skipped prompt and message hooks can prevent plugins from capturing or injecting session context as intended.
- `impact:security`: The fix path touches plugin hook trust boundaries, especially conversation-content access and outbound guardrail hooks.

## Summary

Keep open. Current main fixes the `api.on("before_prompt_build")` path and now documents `api.on(...)` as the typed plugin-hook API, but `api.registerHook("message_received", ...)` still records a raw internal-hook key that inbound dispatch will not match, so the original silent-registration trap remains a real API compatibility decision.

## What This Changes

The issue reports that plugin hook registrations can appear accepted while `message_received`, prompt-build, and related runtime hook handlers are never invoked.

## Best Possible Solution

Keep `api.on(...)` as the documented typed hook API, then make `api.registerHook(...)` fail visibly or intentionally route recognized typed hook names through the same typed-hook policy path with regression coverage.

## Reproduction Assessment

Yes, source-based. Current main still stores `registerHook("message_received")` under the raw key `message_received`, while inbound dispatch checks typed `message_received` and internal `message:received` through separate systems.

## Solution Assessment

Unclear. The maintainable direction is to keep `api.on(...)` as the typed hook path, but maintainers need to choose whether `registerHook(...)` gets compatibility aliasing, a deprecation diagnostic, or stricter docs/errors.
