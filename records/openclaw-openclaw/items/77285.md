## Summary

Keep open. Current main supports the documented typed `api.on(...)` path, including `message_received` and `before_prompt_build`, but `registerHook("message_received")` still stores the raw key in the internal-hook registry where inbound dispatch only triggers `message` and `message:received`; that remaining mismatch is real but needs maintainer API-direction judgment.

## What This Changes

The issue asks for plugin `registerHook("message_received")` and `api.on("before_prompt_build")` registrations to dispatch instead of silently doing nothing.

## Best Possible Solution

Keep `api.on(...)` as the supported typed hook path, then add either a clear diagnostic for typed hook names passed to `registerHook(...)` or a carefully documented compatibility alias that preserves hook policy and handler signatures.

## Reproduction Assessment

Yes for the `registerHook("message_received")` half: source inspection shows the raw key is registered but inbound internal dispatch only checks `message` and `message:received`. No high-confidence reproduction for the `api.on("before_prompt_build")` no-op half because current main wires `api.on` and prompt assembly calls the typed hook runner.

## Solution Assessment

Unclear. The narrowest maintainable solution depends on whether maintainers want backward-compatible aliasing from `registerHook` to typed hooks or a warning/docs fix that steers plugin authors to `api.on(...)`.
