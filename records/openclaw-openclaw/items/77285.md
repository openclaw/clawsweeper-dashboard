## Label Justifications

- `P2`: The issue is a real plugin SDK dispatch bug affecting non-bundled hook workflows, but it is not a core runtime outage.
- `impact:message-loss`: The affected `message_received` and related message hooks can be registered yet suppressed or bypassed, leaving message-flow plugins blind to channel messages.
- `impact:session-state`: The reported use case is plugin context capture and prompt injection, which can leave session context and next-turn state out of sync.
- `impact:security`: The fix touches plugin trust boundaries around conversation access and outbound message mutation or veto hooks.

## Summary

Keep open. Current main and the latest release still leave `api.registerHook("message_received", ...)` on the internal-hook path while inbound `message_received` dispatch reads typed hooks, so the original silent no-dispatch bug remains even though `api.on("before_prompt_build")` is now wired and documented.

## What This Changes

The issue asks OpenClaw to dispatch plugin handlers registered with `registerHook("message_received")` and to make `api.on("before_prompt_build")` usable for prompt-context injection.

## Best Possible Solution

Keep `api.on(...)` as the typed plugin-hook path, then add maintainer-approved compatibility handling or a loud diagnostic for typed names passed to `registerHook(...)`, with docs and regression coverage; route the newer channel-specific `message_sending` evidence to the canonical outbound-hook work if it remains source-proven.

## Reproduction Assessment

Yes, source-based. Current main still stores `registerHook("message_received")` under the raw underscore key while inbound dispatch reads typed `message_received` hooks and internal `message:received` hooks instead.

## Solution Assessment

Unclear. The maintainable API direction appears to be `api.on(...)` for typed hooks, but maintainers still need to choose whether `registerHook(...)` should compat-route typed names, reject them, or warn with a deprecation path.
