## Summary

Keep open. Current main and v2026.6.10 document and wire `api.on(...)` as the typed-hook path, but `api.registerHook("message_received", ...)` still records an internal hook key that neither typed dispatch nor internal `message:received` dispatch reads, so the remaining SDK compatibility or diagnostic decision is still necessary.

## What This Changes

The issue asks for plugin hook registrations such as `registerHook("message_received")` and `api.on("before_prompt_build")` to dispatch or fail visibly instead of silently accepting inert handlers.

## Best Possible Solution

Choose one explicit SDK contract for typed names passed to `registerHook`, then add focused tests and docs so typed hook registrations either dispatch through `api.on(...)` semantics or fail visibly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-based: `registerHook("message_received")` records an internal key while runtime `message_received` dispatch only checks `registry.typedHooks`, so the handler will not run through the typed hook path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the SDK contract: the supported `api.on(...)` path works, but `registerHook` typed-name calls still need either a compatibility bridge or an explicit warning/rejection.
