## Summary

Keep open: the report is fresh, detailed, and current source still has a plausible retained-resource path where Active Memory invokes memory search and the local embedding provider keeps the node-llama-cpp model/context alive without a provider close contract.

## What This Changes

Investigate retained gateway RSS after Telegram Active Memory preflight loads or touches the local memory-search embedding model and then times out.

## Best Possible Solution

Define a memory embedding provider lifecycle that can release or bound local node-llama-cpp model mappings, then wire Active Memory timeout/completion behavior to that contract with live Linux Telegram RSS proof.

## Reproduction Assessment

No high-confidence local reproduction from this review: the provided live steps are detailed and current source matches the reported path, but I did not recreate the Linux Telegram/node-llama-cpp RSS profile. Source inspection is enough to keep it open, not enough to claim reproduced.

## Solution Assessment

Unclear: closing over the Active Memory timeout alone may miss the real ownership boundary because memory-core lacks a provider close/dispose contract. The maintainable solution should first decide whether local embedding models are intentionally cached, then implement bounded release semantics and proof.
