## Label Justifications

- `P2`: The issue describes a scoped but plausible gateway memory-pressure bug in Telegram Active Memory rather than a total outage or security emergency.
- `impact:session-state`: The retained resource belongs to Active Memory and memory-search runtime state after a preflight turn completes or times out.

## Summary

Keep open: current main still has the reported source path where Telegram-eligible Active Memory runs a blocking embedded recall, that recall can call memory_search, and the local embedding provider retains node-llama model/context objects without a provider cleanup contract. The report is fresh, consolidated from the two closed predecessor issues, and the reporter explicitly asked that duplicate fix work not be started while they prepare a fix.

## What This Changes

Report a live Linux Telegram Active Memory timeout that leaves a local memory-search GGUF embedding model mapping resident in the gateway parent after idle.

## Best Possible Solution

Define an explicit memory embedding provider lifecycle or bounded retention policy, wire Active Memory timeout/completion through that ownership boundary, and prove the Linux Telegram RSS/smaps behavior before and after the fix.

## Reproduction Assessment

No high-confidence local reproduction from this review: the live steps and profiling are detailed, and current source matches the Active Memory to memory_search to local embedding path, but I did not recreate the Linux Telegram/node-llama-cpp RSS profile.

## Solution Assessment

Unclear: an Active Memory-only timeout cleanup may miss the ownership boundary because the embedding provider contract currently lacks disposal semantics. The maintainable fix should decide whether local embedding model caches are intentional, then add bounded release behavior with live proof.
