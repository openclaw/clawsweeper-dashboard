## Summary

Keep open: current main still lacks the full-requester lookup for WebChat `session_status({ sessionKey: "current" })`, and this PR is the focused candidate fix with sufficient runtime proof and no blocking patch findings.

## What This Changes

The PR updates `session_status` current-session resolution to try the full effective requester key before fallback, and adds WebChat regressions for persisted and unpersisted `agent:admin:main` lookups.

## Best Possible Solution

Land this narrow resolver fix, or an equivalent rebased patch, so semantic `current` checks the full active requester before falling back to alias/main while preserving literal-current, runSessionKey, sessionId, and visibility behavior.

## Reproduction Assessment

Yes, at source level: current main can construct `session_status` with `agentSessionKey: "agent:admin:main"` and `sessionKey: "current"`, then fall through to alias fallback that can select bare `main`. I did not run tests or a live WebChat server in this read-only review.

## Solution Assessment

Yes, this is the best narrow fix shape I found: keep the change in `session-status-tool.ts`, resolve semantic `current` to the full effective requester before alias fallback, and cover both persisted and unpersisted WebChat cases. A broader gateway or client-alias change would touch more surface, and the merged adjacent resolver PR did not replace this missing lookup.
