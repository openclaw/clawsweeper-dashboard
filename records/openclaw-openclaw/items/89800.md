## Best Possible Solution

## Reproduction Assessment

Yes, at source level: current main can leave WebChat `sessionKey: "current"` unresolved when no `runSessionKey` exists, then fall through to alias/main lookup instead of the full effective requester key. I did not run a live WebChat scenario in this read-only pass.

## Solution Assessment

Yes, this is the best narrow fix shape: resolve semantic current against the effective requester before alias fallback while preserving literal `current`, sessionId resolution, and visibility handling. The whitespace edge raised in review is covered by `readStringParam` trimming tool inputs before the comparison.
