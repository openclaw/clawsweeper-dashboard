## Summary

Keep open: the shared SQLite execution-layer fix is the right direction, but the latest head regressed the fallback by treating every `RawNode` as non-row-returning when `StatementSync.columns()` is absent, so supported Node 23.0-23.10 raw selects/PRAGMAs/RETURNING SQL can still route through `run()`.

## What This Changes

The PR centralizes OpenClaw's node:sqlite Kysely execution helper and adds a `StatementSync.columns()` feature-detection fallback for supported Node runtimes where that method is absent.

## Best Possible Solution

Keep the shared executor approach, restore safe raw SQL row-returning and raw insert classification for the `columns()`-absent path, add hidden-columns regression coverage for raw SELECT/PRAGMA/RETURNING, then refresh the Node 23 proof against the current head.

## Reproduction Assessment

Yes from source for the review finding: with `statement.columns` hidden, Kysely raw SQL compiles as a `RawNode`, `queryNodeReturnsRows` returns false, and `executeCompiledQuerySync` calls `run()` instead of `all()`. I did not run a live Node 23.9 repro in this read-only review.

## Solution Assessment

No. The shared SQLite/Kysely executor is the right layer, but the current head is not the best fix until it preserves raw row-returning SQL in the `columns()`-absent fallback or explicitly narrows the supported contract.
