## Summary

Keep open: the PR targets the right stuck-session/context-overflow delivery bug, but the recovery reason is passed into the active handle’s timeout-flag slot and no after-fix real behavior proof is present.

## What This Changes

The PR threads abort reasons through the embedded Pi runner and changes overflow fallback handling so stuck or context-overflowed sessions return a user-visible error payload.

## Best Possible Solution

Keep the user-visible recovery direction, but make timeout state and abort reason separate in the active handle contract, add focused reason-to-payload coverage, and prove one visible channel error for overflow or stuck recovery.

## Reproduction Assessment

Yes for the PR defect from source inspection: current main’s active handle abort takes a timeout flag first and a reason second, while the PR passes `stuck_recovery` as the first argument. I did not run tests because this review is read-only.

## Solution Assessment

No as written; the product direction is reasonable, but the implementation must preserve the abort reason without reclassifying the run as a timeout and needs real behavior proof before merge.
