## Summary

Needs changes before merge: the updated branch fixes the reported over-masking for selected word-based kebab identifiers, but it also removes fail-closed masking for explicit Apple/iCloud app-password fields when the 4x4 token happens to be made of allowed words.

## What This Changes

The PR replaces the broad app-password field gate with an unconditional 4-4-4-4 sweep that skips tokens whose four segments are all in a large benign wordlist, and adds redaction regression tests.

## Best Possible Solution

Land one canonical redaction fix that keeps explicit Apple/iCloud/application-password fields fail-closed, narrows generic/freeform fields enough to preserve benign identifiers, and covers both paths with regression tests plus real output proof.

## Solution Assessment

No. The word-based discriminator is useful for generic text false positives, but explicit Apple/iCloud/app-specific-password fields should remain fail-closed instead of relying on a large allowlist.
