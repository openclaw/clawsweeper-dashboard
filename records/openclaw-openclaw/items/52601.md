## Summary

Keep open. The FreeBSD report appears valid against current main, but this PR is not merge-ready because routing FreeBSD through the legacy fallback extends weaker crash semantics in a hardened write path instead of fixing the pinned-helper identity mismatch directly.

## What This Changes

The PR changes `src/infra/fs-safe.ts` so `writeFileWithinRoot` and `copyFileWithinRoot` use the legacy fallback path on FreeBSD as well as Windows.

## Best Possible Solution

Land a tested FreeBSD-compatible fix that normalizes or compares pinned-helper device identities safely while preserving atomic new-target materialization, hardlink rejection, symlink/root-boundary checks, and post-write verification.

## Reproduction Assessment

Yes for triage: the PR body gives FreeBSD 15/ZFS jail steps and logs, and current main still compares Python helper `st_dev` values strictly against Node stats. I did not live-run FreeBSD locally.

## Solution Assessment

No. Reusing the legacy fallback is narrow, but a safer maintainable solution is to fix the pinned-helper identity flow so FreeBSD keeps the stronger write/copy semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve atomic writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write; when the destination is missing, that creates the final inode before rename. If the process is killed in that window, FreeBSD can retain a zero-byte target where the pinned helper would not materialize the final path until replace, so this should normalize the pinned-helper identity mismatch instead.
  - confidence: 0.84
