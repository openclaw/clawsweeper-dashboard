## Summary

Keep open. The FreeBSD/ZFS helper mismatch is plausible from source and prior discussion, but this branch is not merge-ready because it routes a hardened write/copy path through legacy missing-target semantics, lacks after-fix real behavior proof, and no longer matches current main's package-backed fs-safe wrapper.

## What This Changes

The PR adds FreeBSD to the write and copy platform guards so those operations use the legacy fs-safe fallback instead of the Python pinned-write helper.

## Best Possible Solution

Land a current-main fix in the shared fs-safe path that handles FreeBSD helper identity without weakening new-target atomicity, then verify helper-enabled writes/copies and plugin install in a FreeBSD/ZFS environment.

## Reproduction Assessment

No high-confidence live reproduction was established here. The source path is clear for helper-enabled FreeBSD/ZFS from the strict helper identity comparison, while current main defaults the helper off unless explicitly configured.

## Solution Assessment

No. The fallback route avoids the Python-vs-Node device mismatch, but it extends legacy missing-target semantics and no longer matches current main's package-backed wrapper; a targeted helper identity fix is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:554`
  - body: Routing FreeBSD into the legacy fallback calls `openWritableFileWithinRoot` before staging the temp file; when the destination is missing, that creates the final inode before `rename`. A crash in that window can leave an empty target where the pinned helper would not publish the path until replace, so please either normalize the helper identity flow or preserve new-target atomicity on FreeBSD.
  - confidence: 0.82
