## Summary

Keep open: the FreeBSD failure is credible on current main, but this PR should not merge as-is because it extends the legacy write path to FreeBSD and weakens new-target all-or-nothing behavior.

## What This Changes

The PR adds FreeBSD to the platform guards that route `writeFileWithinRoot` and `copyFileWithinRoot` through the legacy fs-safe fallback in `src/infra/fs-safe.ts`.

## Best Possible Solution

Land a tested FreeBSD-compatible pinned-helper identity fix that avoids the Python-vs-Node device mismatch while preserving atomic replace behavior and existing hardlink, symlink, and root-boundary checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the PR provides FreeBSD 15/ZFS jail logs, and current main still compares Python helper identity against Node stats without FreeBSD normalization; I did not live-run FreeBSD here.

## Solution Assessment

Is this the best way to solve the issue? No. The fallback is narrow, but it extends legacy new-target crash semantics to FreeBSD; normalizing or safely comparing the pinned-helper identity is the more maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write, so a missing destination is created at the final path before the later rename. If the process is killed in that window, FreeBSD can retain a zero-byte target where the pinned helper would not publish the final path until replace; fix the identity mismatch without sending FreeBSD through this legacy path.
  - confidence: 0.84
