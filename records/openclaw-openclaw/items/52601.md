## Summary

Keep open. The FreeBSD failure appears real, but this PR trades the pinned helper's stronger new-target atomicity for the legacy fallback path on another platform, so it is not merge-ready without maintainer judgment or a safer identity-normalization fix.

## What This Changes

The PR changes `writeFileWithinRoot` and `copyFileWithinRoot` so FreeBSD uses the legacy fs-safe write/copy fallback instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible identity-normalization or pinned-helper comparison fix that preserves pinned write/copy semantics, with focused coverage for device identity and fs-safe write/copy behavior.

## Reproduction Assessment

Yes for triage: the PR body gives FreeBSD 15/ZFS jail steps and logs, and current main still compares Python helper `st_dev` values strictly against Node stats. I did not live-run FreeBSD/ZFS locally.

## Solution Assessment

No. Reusing the legacy fallback is narrow, but the safer maintainable solution is to fix the pinned-helper identity flow so FreeBSD keeps the stronger write/copy semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes on FreeBSD:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write; when the destination is missing, that creates the final inode before rename. A crash in that window can leave a zero-byte target where the pinned helper would not materialize the final path until `os.replace`, so this should normalize the pinned-helper identity mismatch or use a FreeBSD path that preserves new-target atomicity.
  - confidence: 0.84
