## Summary

Keep open for maintainer handling: the FreeBSD failure signal is plausible, but this branch is now conflicted against the current package-backed fs-safe surface, lacks after-fix real behavior proof, and extends legacy write semantics in a hardened filesystem path.

## What This Changes

The PR adds FreeBSD to the write/copy fallback platform guards in `src/infra/fs-safe.ts` so those operations avoid the Python pinned-write helper.

## Best Possible Solution

Land a current-main fix in the shared fs-safe path that handles FreeBSD helper identity without weakening new-target atomicity, then verify helper-enabled writes/copies and plugin install in a FreeBSD/ZFS environment.

## Reproduction Assessment

No high-confidence live reproduction was established here. The source path is clear for helper-enabled FreeBSD/ZFS because the package still compares Python helper identity strictly outside Windows, while current OpenClaw avoids that path by defaulting the helper off.

## Solution Assessment

No. The PR’s fallback route avoids the signed/unsigned device mismatch, but it extends legacy missing-target semantics and no longer matches the current package-backed wrapper; a targeted helper identity fix is safer.
