## Summary

Keep open: the FreeBSD failure is plausible for helper-enabled writes, but this PR is not safe to merge as-is and current main has since moved the default path to the shared fs-safe package with the Python helper off by default.

## What This Changes

The branch adds FreeBSD to the legacy write/copy fallback guards in `src/infra/fs-safe.ts`.

## Best Possible Solution

Fix or explicitly document the helper-enabled FreeBSD behavior in the shared `@openclaw/fs-safe` path while preserving atomic publication and existing identity checks; do not merge this stale platform-guard workaround as-is.

## Reproduction Assessment

No high-confidence end-to-end reproduction was established here. The source path is clear for helper-enabled FreeBSD/ZFS because the dependency still compares helper identity strictly outside Windows, while current OpenClaw defaults the helper off.

## Solution Assessment

No. Routing FreeBSD through the legacy fallback avoids the signed/unsigned device mismatch, but it extends weaker missing-target interruption semantics; a targeted helper identity fix in `@openclaw/fs-safe` is the safer direction.
