## Summary

Keep open: current main still sends FreeBSD writes/copies through the Python pinned-write helper, while this PR's legacy fallback avoids the reported mismatch by accepting a weaker new-target crash behavior in a hardened filesystem path.

## What This Changes

The PR adds FreeBSD to the existing Windows legacy fallback guards for `writeFileWithinRoot` and `copyFileWithinRoot` in `src/infra/fs-safe.ts`.

## Best Possible Solution

Land a FreeBSD-compatible pinned-helper identity fix, with regression coverage, while preserving atomic replace behavior and the existing hardlink, symlink, and root-boundary checks.

## Reproduction Assessment

Yes at source level, but not live-reproduced here: the PR body provides FreeBSD 15/ZFS jail steps and logs, and current main still strictly compares Python helper identity against Node stats. I did not execute a FreeBSD environment during this read-only review.

## Solution Assessment

No. The fallback is narrow, but it extends legacy new-target crash semantics to FreeBSD; normalizing or safely comparing the pinned-helper identity is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing FreeBSD writes:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into `writeFileWithinRootLegacy` calls `openWritableFileWithinRoot` before the temp write; missing destinations are created at the final path before `rename`, so a crash can leave an empty target where the pinned helper would only publish at `os.replace`. Please fix the Python/Node device identity mismatch without sending FreeBSD through this legacy path.
  - confidence: 0.84
