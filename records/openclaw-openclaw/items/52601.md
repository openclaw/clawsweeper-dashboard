## Summary

Keep this PR open, but not merge-ready as-is. The FreeBSD/ZFS failure is plausible and current main still has the Python-helper identity mismatch path, but the proposed fallback extends legacy write semantics to FreeBSD and can leave a zero-byte final target if interrupted before the temp-file rename.

## What This Changes

The PR changes `src/infra/fs-safe.ts` so FreeBSD follows the same legacy write/copy fallback path currently used on Windows.

## Best Possible Solution

Land a FreeBSD-compatible fs-safe fix that keeps pinned-helper all-or-nothing target materialization while normalizing or comparing the Python and Node device identities safely, without weakening hardlink rejection, symlink/root-boundary checks, or post-write verification.

## Reproduction Assessment

Yes for triage: the PR body gives FreeBSD 15/ZFS jail steps and logs, and current main still compares Python helper `st_dev` values strictly against Node stats. I did not live-run a FreeBSD jail in this read-only review.

## Solution Assessment

No. Routing FreeBSD to the legacy fallback is narrow, but it extends a known final-path precreation window; a safer maintainable fix is to normalize the pinned-helper identity flow or add a FreeBSD path that preserves atomic new-target materialization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: When the destination is missing, the legacy path calls `openWritableFileWithinRoot`, which creates the final inode before the temp write and later rename. Adding FreeBSD to this guard means an interrupted plugin install or config write can leave a zero-byte final target where the pinned helper would not have materialized the final path yet. Please keep FreeBSD on a pinned/atomic path and normalize the device identity mismatch instead.
  - confidence: 0.84
