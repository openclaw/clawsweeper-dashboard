## Summary

Keep open: current main still has the FreeBSD pinned-helper identity mismatch, but this PR moves FreeBSD onto legacy fallback semantics that can regress new-target atomicity, so it should not merge as-is.

## What This Changes

The PR adds FreeBSD to the existing Windows legacy fallback guards for `writeFileWithinRoot` and `copyFileWithinRoot` in `src/infra/fs-safe.ts`.

## Best Possible Solution

Implement a FreeBSD-compatible pinned-helper identity comparison or normalization with regression coverage while keeping atomic replace, hardlink rejection, symlink, and root-boundary checks intact.

## Reproduction Assessment

Yes at source level, but not live-reproduced here: the PR body gives FreeBSD 15/ZFS jail steps/logs, and current `main` still compares Python helper `st_dev|st_ino` against Node stats with strict non-Windows device equality.

## Solution Assessment

No. Reusing the legacy fallback is narrow, but it extends its pre-created-target crash window to FreeBSD; normalizing or safely comparing the pinned-helper identity is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing FreeBSD writes:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into `writeFileWithinRootLegacy`/`copyFileWithinRootLegacy` calls `openWritableFileWithinRoot` before the temp write; for a missing destination, that creates the final path before the later rename. A crash in that window can leave an empty target where the pinned helper only publishes at `os.replace`, so fix the Python/Node device identity mismatch without moving FreeBSD onto this legacy path.
  - confidence: 0.84
