## Summary

Keep open: the FreeBSD failure appears valid, but the patch should not merge as-is because it expands legacy write semantics onto a hardened filesystem path instead of repairing the pinned-helper identity mismatch.

## What This Changes

The PR changes `writeFileWithinRoot` and `copyFileWithinRoot` so FreeBSD uses the existing legacy write/copy fallback instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible pinned-helper identity fix that preserves new-target atomic replace behavior, hardlink rejection, symlink boundaries, and post-write verification.

## Reproduction Assessment

Yes at source level, but not live-reproduced here: the PR provides FreeBSD 15/ZFS jail steps and logs, while current main still compares Python helper identity against Node stats with no FreeBSD normalization. A maintainer with FreeBSD/ZFS access should verify the end-to-end `openclaw plugins install` path.

## Solution Assessment

No. The fallback is narrow, but it extends legacy new-target crash semantics to FreeBSD; normalizing or safely comparing the pinned-helper identity is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing FreeBSD writes:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into `writeFileWithinRootLegacy`/`copyFileWithinRootLegacy` opens or creates the final destination before the temp write. If the process is killed in that window, a missing target can be left as an empty file, whereas the pinned helper publishes the target only at replace time. Please fix the Python/Node device identity mismatch without sending FreeBSD through the legacy path, or preserve the new-target atomicity property in the FreeBSD path.
  - confidence: 0.84
