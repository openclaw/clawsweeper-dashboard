## Summary

Keep open: current main still lacks `/hooks/agent` thread routing and this PR is useful, but it should not merge until the parser preserves numeric thread values, the public `thread`/`threadId` API naming is settled, the branch is rebased, and real threaded-delivery proof is added.

## What This Changes

The PR adds optional `/hooks/agent` `thread` payload handling, forwards it to cron `delivery.threadId`, scopes idempotency replay by thread, and updates docs, changelog, and gateway tests.

## Best Possible Solution

Land a rebased hook-threading change that feeds `delivery.threadId`, accepts string and finite numeric thread values, scopes replay by thread, documents the chosen field or alias, and includes redacted live threaded-delivery proof.

## Reproduction Assessment

Yes, at source level for the PR defect: the PR normalizes `payload.thread` with `normalizeOptionalString` while the downstream `delivery.threadId` contract accepts strings and finite numbers. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is narrow and useful, but the implementation should preserve the existing string-or-number thread contract, settle `thread` versus `threadId`, and provide real threaded-delivery proof before merge.
