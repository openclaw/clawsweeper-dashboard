## Summary

Keep open: this is useful active work that current main still lacks, but the PR head drops numeric thread values and still has no after-fix real threaded-delivery proof, so it is not ready to merge.

## What This Changes

The PR adds optional `/hooks/agent` `thread` parsing, maps it to `CronJob.delivery.threadId`, scopes idempotency replay by thread, and updates gateway tests, docs, and changelog.

## Best Possible Solution

Land a hook-threading change that preserves string and finite numeric thread values, scopes replay by thread, documents the chosen `thread`/`threadId` compatibility, and includes redacted live threaded-delivery proof.

## Reproduction Assessment

Yes, at source level: current main lacks hook thread parsing and dispatch, while the PR defect is visible because `payload.thread` is normalized as a string even though `CronDelivery.threadId` accepts strings or finite numbers. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but the parser should reuse the numeric-aware thread normalizer, type the hook value consistently with delivery, and settle `thread` versus `threadId` compatibility before merge.
