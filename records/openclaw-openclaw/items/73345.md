## Summary

Keep open: the PR addresses a real `/hooks/agent` threading gap, but the current patch still drops finite numeric thread values accepted by the delivery contract and lacks required real threaded-delivery proof before merge.

## What This Changes

The PR adds a `/hooks/agent` `thread` payload field, forwards it to `CronJob.delivery.threadId`, scopes idempotency replay by thread, and updates docs, changelog, and gateway tests.

## Best Possible Solution

Land a hook-threading change that preserves the `string | number` delivery thread contract, scopes replay by normalized thread, documents the chosen public field or alias, and includes redacted real Telegram/topic or equivalent threaded-delivery proof.

## Reproduction Assessment

Yes, at source level for the PR defect: the PR normalizes `payload.thread` with `normalizeOptionalString` while downstream `delivery.threadId` accepts strings and finite numbers. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is useful, but the implementation should reuse the numeric-aware thread normalizer, keep replay scoping aligned to the normalized thread value, settle `thread` versus `threadId`, and provide real threaded-delivery proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Accept numeric hook thread values:** `src/gateway/hooks.ts:425`
