## Summary

Keep open. Current main does not implement `/hooks/agent` thread delivery, and this PR is useful but incomplete: the new delivery target is not included in idempotency replay scope, and the public docs/changelog are not updated. The existing ClawSweeper concern about replay scope is confirmed by current source inspection.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent`, forwards it as `CronJob.delivery.threadId`, and adds gateway hook coverage for thread-present and thread-absent payloads.

## Best Possible Solution

Revise this PR so `thread` is treated as part of the hook delivery destination end to end: normalize it, forward it to cron delivery, include it in `/hooks/agent` replay scope, test same-idempotency-key/different-thread behavior, and update the public docs and changelog before merge.

## Reproduction Assessment

Yes. On the PR branch, send two `POST /hooks/agent` requests with the same idempotency key and identical payload except `thread`; because replay scope omits thread, the second request can return the first runId instead of dispatching to the new thread.

## Solution Assessment

No. Parsing and forwarding `thread` is the right direction, but the maintainable fix must also scope replay dedupe by normalized thread and document/release-note the new webhook field.
