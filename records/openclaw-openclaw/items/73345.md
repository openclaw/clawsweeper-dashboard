## What This Changes

The PR branch adds optional `/hooks/agent` `thread` payload parsing, forwards it to `CronJob.delivery.threadId`, scopes hook replay by thread, and updates gateway tests, docs, and changelog.

## Best Possible Solution

Update the PR to preserve the existing `string | number` thread contract, add numeric-thread regression coverage, rebase, and attach redacted live threaded-delivery proof.

## Reproduction Assessment

Yes for the PR defect from source inspection: PR head normalizes `payload.thread` with `normalizeOptionalString`, while the delivery contract accepts finite numbers. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is narrow and aligned with existing delivery contracts, but the implementation should preserve numeric thread IDs and provide real Telegram/topic delivery proof before merge.
