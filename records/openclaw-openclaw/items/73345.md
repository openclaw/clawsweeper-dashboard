## Summary

Keep open. The PR is useful and matches the intended `/hooks/agent` thread-delivery plumbing, but the current patch treats `thread` as a delivery destination without adding it to hook replay scope, and it omits the public docs and changelog updates required for a new webhook field.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent`, forwards it as `CronJob.delivery.threadId`, and adds gateway hook tests for payloads with and without `thread`.

## Best Possible Solution

Revise this PR so `thread` is handled as part of the hook delivery destination everywhere it matters: normalize and forward it, include the normalized value in `/hooks/agent` replay scope, add same-idempotency-key/different-thread regression coverage, and update the public webhook docs plus changelog before merge.

## Reproduction Assessment

Yes. On the PR branch, two `/hooks/agent` requests with the same idempotency key and identical payloads except `thread` can reproduce the bug: the second request can reuse the first replay cache entry because the replay scope does not include thread.

## Solution Assessment

No. Parsing and forwarding `thread` is the right direction, but the current patch is not the narrow complete fix because replay dedupe and public documentation/changelog are still out of sync with the new delivery field.
