## Summary

Keep open. Current main does not yet implement hook `thread` input, and the PR is useful but not merge-ready because the new delivery destination is not included in `/hooks/agent` replay scope, so same-idempotency-key requests that only change `thread` can replay the wrong run. The PR also needs the public webhook docs and changelog updated before merge.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent`, forwards it as `CronJob.delivery.threadId`, and adds gateway hook tests for thread-present and thread-absent payloads.

## Best Possible Solution

Revise this PR so `thread` is part of the hook delivery destination end to end: normalize it, forward it to cron delivery, include it in `/hooks/agent` replay scope, test same-idempotency-key/different-thread behavior, and update the public docs and changelog.

## Reproduction Assessment

Yes. On the PR branch, send two `POST /hooks/agent` requests with the same idempotency key and identical payload except `thread`; because replay scope omits thread, the second request can return the first runId instead of dispatching to the new thread.

## Solution Assessment

No. Parsing and forwarding `thread` is the right direction, but the maintainable fix must also scope replay dedupe by normalized thread and document/release-note the new webhook field.
