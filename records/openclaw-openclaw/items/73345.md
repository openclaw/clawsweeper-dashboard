## Summary

Keep open. Current main still lacks this hook payload surface and the PR is useful, but latest head drops numeric thread IDs even though cron delivery accepts them, and the external PR only provides test-based proof.

## What This Changes

Adds `/hooks/agent` `thread` payload support, propagates it into `CronJob.delivery.threadId` and idempotency replay scope, and updates gateway tests, docs, and changelog.

## Best Possible Solution

Finish the branch by preserving the existing `string | number` thread contract end-to-end, adding a numeric-thread regression, and attaching real threaded-delivery proof before merge.

## Reproduction Assessment

Yes for the PR defect. The latest head normalizes `payload.thread` with `normalizeOptionalString`, so a JSON numeric `thread` value is dropped before `CronJob.delivery.threadId` is built; I did not run tests in this read-only review.

## Solution Assessment

No. The direction is narrow and useful, but the parser should reuse the existing numeric-aware thread normalizer and `string | number` typing, then cover numeric input and provide real Telegram/threaded delivery proof.
