## Summary

Keep open: current main still lacks the requested hook-thread delivery path, and this PR is the narrow active successor to the closed earlier attempts, but it should not merge until numeric thread values are preserved, the public field name or alias is settled, and real threaded-delivery proof is added.

## What This Changes

The branch adds optional `/hooks/agent` `thread` parsing, maps it into cron `delivery.threadId`, includes `thread` in replay scope, and updates gateway tests, docs, and changelog.

## Best Possible Solution

Land a hook-threading change that feeds `delivery.threadId`, accepts string and finite numeric thread values, scopes replay by thread, documents the chosen `thread`/`threadId` API, and includes redacted live threaded-delivery proof.

## Reproduction Assessment

Yes, at source level: current main lacks hook thread parsing, and the PR defect is visible because `payload.thread` is normalized as a string while `delivery.threadId` accepts strings or finite numbers. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but the implementation should reuse the numeric-aware thread normalizer, settle `thread` versus `threadId` compatibility, and include real threaded-delivery proof before merge.
