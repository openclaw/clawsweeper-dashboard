## Summary

Keep open. Current main still runs the unverified Bun installer in the root Dockerfile, while this PR removes that build-time remote shell execution and uses a digest-pinned official Bun image stage; the branch is useful and CI is green, but it is not merged into main yet and the new digest test helper should model Docker ARG scope more tightly.

## What This Changes

The PR replaces the root Dockerfile Bun curl-pipe installer with a digest-pinned `oven/bun` source stage and extends Docker image digest tests to check ARG-backed `FROM` stages.

## Best Possible Solution

Land a reviewed Docker hardening change that removes build-time execution of the remote Bun installer, keeps the Bun source image digest-pinned and aligned with CI, tightens the new ARG-backed digest helper, and lets the linked Docker supply-chain issue close through the merge after Docker build proof passes.

## Reproduction Assessment

Yes. The issue is reproducible by static inspection of target main: `Dockerfile:47` still pipes `https://bun.sh/install` into `bash`.

## Solution Assessment

Yes, the proposed Dockerfile direction is the narrow maintainable fix: copy Bun from a digest-pinned official image stage instead of executing a network script during the build. The safer final shape is the same patch with the test helper limited to Dockerfile-global ARG scope and a root Docker build or install-smoke proof.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Stop collecting ARG defaults after the first FROM:** `src/docker-image-digests.test.ts:38-43`
  - body: Docker only lets `FROM` use ARGs declared before the first `FROM`, but this helper scans ARG defaults from every stage. A future stage-local ARG could make an ARG-backed `FROM` look digest-pinned in the test even though Docker would not resolve that reference as written; limit defaults to global scope or model ARG scope explicitly.
  - confidence: 0.74
