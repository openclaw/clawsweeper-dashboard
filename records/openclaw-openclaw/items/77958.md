## Summary

This should stay open: current main still publishes plugin-shipped skill directories with a Windows directory symlink, while the documented behavior says enabled plugin skills should load and the report gives a concrete stock-Windows EPERM failure.

## What This Changes

Request that plugin-skill publication use Windows junctions or an EPERM fallback for directory targets so plugin-shipped skills register without admin rights or Developer Mode.

## Best Possible Solution

Update plugin-skill publishing to use a Windows junction for directory targets, or fall back to one on EPERM/ENOTSUP, while preserving plugin-root validation and stale-link cleanup semantics.

## Reproduction Assessment

Yes, as source-reproducible: current main publishes every plugin skill directory with `fs.symlinkSync(..., "dir")`, and the report includes Windows 11 standard-user EPERM output from `openclaw doctor`. I did not perform a live no-Developer-Mode Windows run in this review.

## Solution Assessment

Yes. The narrow maintainable fix is to handle Windows directory links with `junction` in this helper, matching existing repository practice, rather than adding a new config knob or changing plugin-skill discovery policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77958. Current main publishes plugin-shipped skills under `~/.openclaw/plugin-skills/` with `fs.symlinkSync(target, linkPath, "dir")`, which fails with EPERM on standard Windows accounts without Developer Mode/admin and leaves browser and memory-wiki plugin skills undiscovered. Keep the fix scoped to plugin-skill publication: use `junction` for Windows directory targets or fall back to `junction` on EPERM/ENOTSUP, preserve existing plugin-root escape validation, idempotency, and stale-link cleanup, and do not broaden into the unrelated #77942 stderr/Ollama issue. Add focused regression coverage in `src/agents/skills/plugin-skills.test.ts` by stubbing/mocking the filesystem path enough to prove Windows EPERM on `"dir"` retries with `"junction"` and still logs/continues for non-recoverable failures. Add a single-line user-facing changelog fix entry. Validate with the targeted plugin-skill test and the changed gate in Testbox per repo policy; if a Windows Crabbox/Testbox lane is available, include a standard-user `openclaw doctor` / `openclaw skills list --json` smoke showing the plugin skills appear without EPERM.
