## Summary

Keep open: current main still has a source-reproducible storeless replay gap, and the linked open fix PR is the active implementation path that should be reviewed or landed before this issue closes.

## What This Changes

Custom OpenAI Responses-compatible providers that strip Responses store support should omit stale prior Responses item ids on follow-up/tool-turn requests.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89729, or an equivalent small patch, so replay ids are disabled whenever the effective Responses payload is storeless while store-enabled OpenAI Responses replay remains intact.

## Reproduction Assessment

Yes. Source inspection shows the reported `compat.supportsStore=false` path sets a store-stripping policy while current replay gating still defaults from `explicitStore !== false`.

## Solution Assessment

Yes. The best fix is the narrow provider-payload correction already proposed by the linked PR: make replay-id permission follow the effective storeless policy in both the builder and wrapper paths.
