## Summary

Keep open: current main still advertises the official packaged `diagnostics-otel` install path, but the service runtime grants `internalDiagnostics` only to bundled diagnostics services and the exporter exits without that capability. The remaining fix crosses an internal diagnostics trust boundary, so it needs maintainer policy rather than cleanup closure.

## What This Changes

Allow externally installed official `diagnostics-otel` packages to receive diagnostics events, or document/provide a trusted opt-in path.

## Best Possible Solution

Define and test a trusted external-official diagnostics exporter path, or an explicit operator trust opt-in, so the documented packaged install works without granting internal diagnostics to arbitrary global plugins.

## Reproduction Assessment

Yes, from source inspection. Installed ClawHub/npm plugins become `origin: "global"`, while `createServiceContext` only injects `internalDiagnostics` for `origin: "bundled"`, and the exporter returns without subscribing when that capability is absent.

## Solution Assessment

Unclear for automation. The narrow fix could trust the official external package, but maintainers need to choose the provenance or opt-in policy because the capability exposes internal diagnostic events.
