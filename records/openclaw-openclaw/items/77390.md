## Summary

The report is valid on current main: the public docs and package metadata present `diagnostics-otel` as an official ClawHub/npm install, but the service runtime only injects `internalDiagnostics` for bundled origins, so an externally installed official exporter can start and still never subscribe to diagnostics events. This should stay open because the fix crosses the internal diagnostics trust boundary rather than being an already-implemented cleanup.

## What This Changes

The issue requests that externally installed diagnostics-otel plugins either receive internal diagnostics events, get a documented trusted opt-in, or use a public hook surface instead of the bundled-only internal diagnostics gate.

## Best Possible Solution

Make the packaged official diagnostics exporter path match the documented behavior while preserving the internal diagnostics trust model, then update the service test and docs to lock that policy.

## Reproduction Assessment

Yes, from source inspection. A non-bundled `diagnostics-otel` service fails the `origin === "bundled"` grant in `src/plugins/services.ts`, and the exporter immediately returns when `ctx.internalDiagnostics` is absent.

## Solution Assessment

Unclear for automation. The narrowest code change might trust the official external package, but that is a policy/security decision because the same capability subscribes to internal diagnostic events.
