## Summary

Keep open: current main still matches the report, and the documented official ClawHub/npm install path can load `diagnostics-otel` without receiving the internal diagnostics capability it needs. The fix should not be automated before maintainers choose the trust policy for external official packages.

## What This Changes

Externally installed official `diagnostics-otel` should receive diagnostics events or have a documented opt-in trust path instead of silently running without telemetry.

## Best Possible Solution

Make the official packaged diagnostics exporter path match the public docs while preserving a verifiable internal diagnostics trust model, then update service tests and docs to lock that policy.

## Reproduction Assessment

Yes, from source inspection. Installed plugins are discovered as `global`, `createServiceContext` grants `internalDiagnostics` only to bundled diagnostics services, and the exporter returns before subscribing when that capability is missing.

## Solution Assessment

Unclear for automation. The expected behavior is documented, but the narrowest maintainable fix depends on a maintainer policy choice between verified official-package provenance, explicit operator trust, or a public diagnostics hook.
