## Summary

Keep open. Current main still documents and catalogs `diagnostics-otel` as an official ClawHub/npm install, but the runtime only injects `internalDiagnostics` for bundled service origins, so the reported external-install path remains source-reproducible after the recent catalog-install trust work.

## What This Changes

Allow externally installed official `diagnostics-otel` packages, or an explicit trust/public-hook path, to receive the internal diagnostics events needed for OTel export.

## Best Possible Solution

Make the documented packaged exporter path work while preserving the diagnostics trust boundary, then update the service tests and OpenTelemetry docs to lock that policy.

## Reproduction Assessment

Yes, from source inspection. Installed plugins are global-origin candidates, `createServiceContext` grants `internalDiagnostics` only to bundled diagnostics exporters, and the OTel exporter exits when that capability is absent.

## Solution Assessment

Unclear for automation. A narrow fix could use official catalog/install provenance or explicit operator trust, but a broad id-only exception would weaken the internal diagnostics boundary.
