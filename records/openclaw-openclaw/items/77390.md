## Summary

The issue still reproduces from current-main source: externally installed `diagnostics-otel` plugins are discovered with a non-bundled origin, while the service context grants `internalDiagnostics` only to bundled diagnostics exporters. It should stay open because fixing the documented install path requires an explicit trust policy for an internal diagnostics capability.

## What This Changes

Allow the official externally installed `diagnostics-otel` plugin to receive internal diagnostics events, or document and provide a trusted opt-in alternative.

## Best Possible Solution

Make the official packaged diagnostics exporter path match the documented behavior while preserving the internal diagnostics trust boundary, then update tests and docs to state the chosen policy.

## Reproduction Assessment

Yes. From current-main source, a ClawHub/npm install is discovered as `origin: "global"`, fails the `origin === "bundled"` grant in `src/plugins/services.ts`, and the exporter exits when `ctx.internalDiagnostics` is missing.

## Solution Assessment

Unclear for automation. The narrowest fix may be to trust the official external package, but that needs maintainer policy on package provenance or explicit trust because the capability exposes internal diagnostics events.
