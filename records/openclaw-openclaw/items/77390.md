## Summary

This should stay open: current main still grants `internalDiagnostics` only to bundled diagnostics exporter services, while the docs and package metadata advertise `diagnostics-otel` as an official ClawHub/npm install path. The remaining decision is the trusted external-package policy, not stale cleanup.

## What This Changes

The issue asks for externally installed official `diagnostics-otel` packages to receive internal diagnostics events, or for OpenClaw to document/provide a trusted opt-in or public-hook alternative.

## Best Possible Solution

Define a trusted external official diagnostics exporter path, or an explicit operator trust opt-in, that matches the documented install flow while preserving denial for spoofed or arbitrary third-party plugins.

## Reproduction Assessment

Yes, from source inspection. Installed plugins are discovered as `global`, `createServiceContext` only grants `internalDiagnostics` to bundled diagnostics exporters, and `diagnostics-otel` exits before subscribing when that context is absent.

## Solution Assessment

Unclear for automation. The observable bug is real, but the maintainable fix needs an explicit provenance or trust policy rather than granting the internal diagnostics capability by plugin id alone.
