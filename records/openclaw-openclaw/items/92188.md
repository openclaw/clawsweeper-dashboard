## Summary

This PR should stay open because the proposed cleanup is broader than the reported failed-recovery leak: it can delete legitimate processing or waiting state before resumed activity refreshes it. The contributor also supplied only fake-timer unit-test evidence rather than real Gateway behavior proof.

## What This Changes

The PR makes diagnostic-session pruning delete non-idle entries older than 30 minutes and adds a regression test that directly seeds a stale processing entry.

## Best Possible Solution

Represent failed-recovery orphaning explicitly and prune only entries whose owner is known to be gone, while preserving ordinary long-running processing and waiting state.

## Reproduction Assessment

Yes at source level: failed recovery can leave a non-idle entry untouched indefinitely. The submitted test does not reproduce that coordinator path or demonstrate it in a running Gateway.

## Solution Assessment

No. Deleting every stale non-idle entry generalizes beyond failed-recovery ghosts and does not preserve the lifecycle invariant for legitimate long-running work.
