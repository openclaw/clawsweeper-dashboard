## Summary

Keep open: current main still uses the value-only 80+ character base64-like warning, the focused sanitizer test still expects that warning for a non-credential key, and the related fix PRs are closed unmerged. The remaining decision is security-sensitive because different tighter heuristics reduce false positives with different false-negative tradeoffs.

## What This Changes

Reduce the sandbox env sanitizer's base64-like credential warning so legitimate long alphanumeric or encoded values are not flagged solely by value length and character class.

## Best Possible Solution

Land a secops-reviewed sanitizer update with regression coverage for long alphanumeric, hex-like, and encoded non-credential values while preserving null-byte blocking, maximum-length handling, explicit credential-name blocking, and useful credential warnings.

## Reproduction Assessment

Yes at source level. Current main still has the exact broad regex, the call path lacks key context, the existing test expects the warning on `SAFE_TEXT`, and an exact regex probe matches the representative false-positive values.

## Solution Assessment

Unclear. Reducing false positives is the right direction, but the exact rule needs secops/product judgment because structural-only, signal-character, and key-name-gated approaches trade off warning noise against missed encoded secret visibility.
