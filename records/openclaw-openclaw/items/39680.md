## Summary

Keep open: current main still uses the value-only 80+ character base64-like warning, related fix PRs are closed unmerged, and the replacement rule is a security-sensitive heuristic choice rather than a settled cleanup.

## What This Changes

The issue asks to reduce false-positive sandbox env-var warnings caused by `validateEnvVarValue` treating any 80+ character base64-character string as credential-like data.

## Best Possible Solution

Land a secops-reviewed sanitizer update with regression coverage for long alphanumeric, hex-like, and encoded non-credential values while preserving null-byte blocking, maximum-length handling, explicit credential-name blocking, and useful credential warnings.

## Reproduction Assessment

Yes, at source level. Current main still has the broad regex, the call path lacks key context, the focused test expects the warning on `SAFE_TEXT`, and a read-only regex probe matches the representative false-positive values.

## Solution Assessment

Unclear. Reducing false positives is the right direction, but the exact rule needs secops/product judgment because the closed PRs show different false-positive and false-negative tradeoffs.
