## Summary

Keep open: current main still uses the reported value-only base64 heuristic, still calls value validation without env-key context, and still has a test that expects the false-positive warning on a non-credential key. Related fix PRs are closed unmerged, and the affected sandbox env sanitizer is security-sensitive, so the heuristic should be chosen by maintainers/secops before another automated patch.

## What This Changes

The issue asks to reduce false-positive sandbox sanitizer warnings caused by flagging any 80+ character base64-character env value as credential data.

## Best Possible Solution

Keep the issue open and land a focused sanitizer change that makes the base64 warning structurally aware and/or credential-key aware, with regression tests for long alphanumeric, hex-like, JWT-segment-like, and genuine credential-shaped values while preserving null-byte blocking, maximum length handling, explicit secret-name blocking, and existing sandbox warning behavior.

## Reproduction Assessment

Yes. Current main reproduces the report from source and tests: the regex matches the reported legitimate long values, and the sanitizer test still expects SAFE_TEXT to receive the credential-data warning.

## Solution Assessment

No implemented solution is present on current main. The narrow technical fix is clear, but because this is a sandbox security warning boundary the exact heuristic trade-off should be reviewed by sandbox/secops owners before landing.
