## Summary

Keep open: current main and the latest release still use the value-only 80+ base64-like regex, and the linked fix PRs are closed unmerged. The remaining work is a security-sensitive heuristic choice that needs secops/product review before implementation.

## What This Changes

Tighten the sandbox env-var sanitizer so legitimate long alphanumeric or encoded values are not warned as credential data solely by character class and length.

## Best Possible Solution

Land a secops-reviewed sanitizer update with regression coverage for long alphanumeric, hex-like, and encoded non-credential values while preserving null-byte blocking, maximum-length handling, explicit credential-name blocking, and useful credential warnings.

## Reproduction Assessment

Yes, at source level. Current main still has the broad regex, the call path lacks key context, the focused test preserves the warning path, and a read-only regex probe matches representative false-positive values.

## Solution Assessment

Unclear. Reducing false positives is the right direction, but the exact rule needs secops/product judgment because the closed PRs show different false-positive and false-negative tradeoffs.
