## Summary

Keep open: current main still has the reported value-only base64 heuristic, existing tests preserve the warning on a non-credential key, and no related fix PR has merged. The remaining decision is the sandbox/security tradeoff for tightening that warning without weakening credential detection.

## What This Changes

The issue requests reducing false-positive sandbox warnings from the long base64-like environment variable value heuristic.

## Best Possible Solution

Land a focused secops-reviewed sanitizer change with regression coverage for long alphanumeric, hex-like, and JWT-segment-like non-credential values while preserving null-byte blocking, max-length handling, explicit credential-name blocking, and warnings for accepted credential-shaped values.

## Reproduction Assessment

Yes. Source inspection on current main plus a read-only probe of the exact regex reproduces the reported false-positive shapes, and sanitizeEnvVars still applies that warning path without key context.

## Solution Assessment

No implemented solution is present on current main. The narrow implementation likely belongs in src/agents/sandbox/sanitize-env-vars.ts and its colocated test, but secops should choose the heuristic because it changes a security warning boundary.
