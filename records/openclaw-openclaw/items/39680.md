## Summary

Keep open: current main still uses the value-only 80+ character base64-like regex, the focused sanitizer test still expects a warning on a non-credential key, and the related fix PRs were all closed unmerged. The remaining choice is how to reduce false positives without weakening a sandbox credential-warning boundary.

## What This Changes

The issue asks OpenClaw to tighten sandbox environment-variable value validation so legitimate long alphanumeric or encoded values are not warned as base64 credential data.

## Best Possible Solution

Land a secops-reviewed sanitizer update with regression coverage for long alphanumeric, hex-like, and encoded non-credential values while preserving null-byte blocking, maximum-length handling, explicit credential-name blocking, and meaningful credential warnings.

## Reproduction Assessment

Yes. Source inspection plus a read-only probe of the exact current regex reproduces warnings for the reported long alphanumeric and encoded-looking values, and the current test suite explicitly expects that warning on `SAFE_TEXT`.

## Solution Assessment

Unclear. Reducing false positives is the right direction, but the exact rule should be chosen by secops because structural-only, key-name-only, and signal-character approaches have different false-negative tradeoffs.
