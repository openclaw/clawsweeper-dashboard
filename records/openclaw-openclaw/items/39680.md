## Summary

Keep open: current main still uses the value-only 80+ character base64-like regex, current tests still expect a warning on a non-credential key, and the related fix PRs are all closed unmerged. The remaining decision is the safer sanitizer heuristic for a security-sensitive sandbox warning path.

## What This Changes

The issue asks to reduce false-positive sandbox warnings from the env-var base64 credential heuristic on long legitimate alphanumeric values.

## Best Possible Solution

Land a secops-reviewed sanitizer update with regression coverage for long non-credential alphanumeric, hex-like, and encoded-looking values while preserving null-byte blocking, maximum-length handling, explicit credential-name blocking, and meaningful credential warnings.

## Reproduction Assessment

Yes. Current source still calls the exact value-only regex without key context, and a read-only probe of that regex reproduces matches for representative long alphanumeric, hex, JWT-like, and padded values.

## Solution Assessment

Unclear. Reducing false positives is the right direction, but structural validation, key-name gating, and signal-character gating have different security tradeoffs that should be chosen by secops before implementation.
