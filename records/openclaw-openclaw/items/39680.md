## Summary

Keep open: current main still has the reported value-only base64 heuristic, no accepted fix has landed, and the sanitizer is a secops-owned sandbox boundary where the exact false-positive/security tradeoff needs maintainer review.

## What This Changes

The issue requests tightening sandbox environment-variable credential warnings so legitimate long alphanumeric, hex-like, or encoded config values are not flagged as base64 credential data.

## Best Possible Solution

Land a focused secops-reviewed sanitizer change with regression coverage for long alphanumeric, hex-like, and JWT-segment-like values while preserving null-byte blocking, max-length handling, explicit secret-name blocking, and warnings for genuinely credential-shaped values.

## Reproduction Assessment

Yes. A read-only probe against the current regex reproduces the reported false-positive shapes, and source inspection shows sanitizeEnvVars still applies that value-only warning path to non-credential keys.

## Solution Assessment

No implemented solution is present on current main. The likely fix is narrow, but the best heuristic should be chosen by sandbox/secops owners because it changes a security warning boundary.
