## Summary

Keep open: current main and the latest release still use the value-only 80+ base64-character warning heuristic, and the linked fix attempts are closed unmerged. The bug is source-reproducible, but the right replacement rule is a security/product tradeoff rather than a safe autonomous cleanup close.

## What This Changes

The issue requests tightening the sandbox env-var credential warning heuristic so legitimate long alphanumeric values are not flagged as base64 credential data.

## Best Possible Solution

Land a secops-reviewed sanitizer update with regression coverage for long alphanumeric, hex-like, and JWT-like non-credential values while preserving null-byte blocking, maximum-length handling, explicit credential-name blocking, and useful encoded-secret warnings.

## Reproduction Assessment

Yes, at source level. Current main still has the broad regex, the warning call path lacks key context, the focused test preserves the warning, and a read-only regex probe matches representative false-positive values.

## Solution Assessment

Unclear. Reducing false positives is the right direction, but the exact replacement heuristic needs secops/product judgment because the closed PRs make different false-positive and false-negative tradeoffs.
