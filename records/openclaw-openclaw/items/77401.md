## Repair Work Prompt

Fix the Telegram partial-streaming ordering race from https://github.com/openclaw/openclaw/issues/77401. Current main fires embedded `onAgentEvent`/`onPartialReply` progress callbacks with `void`, while Telegram only serializes progress after callbacks reach `enqueueDraftLaneEvent`; final delivery can therefore overtake slow late progress callbacks. Add a shared awaited drain/ack path for delivery-affecting progress callbacks so final delivery waits for all earlier progress work. Start with a deterministic regression where slow `onItemEvent`/`onCommandOutput` or `onPartialReply` callbacks would otherwise enqueue after final dispatch; expected order is progress before final. Keep the fix generic to the embedded/reply pipeline when possible, with Telegram using the existing draft queue; do not add sleeps, network timing workarounds, config options, or Telegram-specific policy changes. Add a user-facing changelog entry if behavior changes are shipped. Preserve existing chunked-send behavior from #62036.
