## Summary

Keep open: current main serializes Telegram delivery after progress reaches the draft queue, and the older chunked-send fix is still present, but embedded progress callbacks are still fired without awaiting or draining them before final delivery can begin. That leaves the reported final-before-last-progress race actionable.

## What This Changes

The issue asks for Telegram `streaming.mode: "partial"` turns to keep tool progress messages before the final assembled reply on tool-heavy runs.

## Best Possible Solution

Add a shared drain or acknowledgement path for delivery-affecting progress callbacks so final delivery waits for all progress work already emitted by the agent run, without adding sleeps or Telegram-specific network timing workarounds.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection on current main shows async progress callbacks are fire-and-forget before Telegram can enqueue them, which is a credible reproduction path for the intermittent ordering race.

## Solution Assessment

No patch is under review. The narrow maintainable fix is to drain delivery-affecting progress callbacks before final dispatch, not to add Telegram sleeps or rely on network timing.

## Repair Work Prompt

Fix the Telegram partial-streaming ordering race reported in https://github.com/openclaw/openclaw/issues/77401. On current main, embedded agent handlers fire `onAgentEvent`/`onPartialReply` callbacks with `void`, while Telegram final delivery only drains work that has already entered `enqueueDraftLaneEvent`; add a shared callback-drain or acknowledgement path so final delivery waits for all delivery-affecting progress callbacks emitted before finalization. Keep the fix narrow: do not add sleeps, Telegram network timing hacks, new config, or broad channel redesign. Add deterministic regression coverage for a tool-heavy turn where a slow progress callback would otherwise enqueue after final delivery. Include a single-line `CHANGELOG.md` fix entry because this is user-facing Telegram behavior.
