## Summary

Keep open: current main serializes Telegram draft/final delivery after progress callbacks enter the queue, but embedded tool and assistant progress callbacks are still invoked fire-and-forget, leaving a plausible late-enqueue race that matches the reported final-before-progress ordering.

## What This Changes

The issue requests chronological Telegram delivery so partial streaming progress/tool messages finish before the final assembled reply.

## Best Possible Solution

Add a shared callback-drain or acknowledgment path for channel-visible progress events, then cover Telegram partial streaming with a deterministic late-callback regression.

## Reproduction Assessment

No high-confidence live reproduction was established. The issue has concrete steps and screenshots, and source inspection shows a current fire-and-forget progress path that can enqueue Telegram work after final delivery begins.

## Solution Assessment

No proposed patch is under review. The maintainable fix is a narrow shared ordering seam for delivery-affecting progress callbacks, not Telegram-specific sleeps or network timing assumptions.

## Repair Work Prompt

Fix the Telegram partial/progress ordering race reported in https://github.com/openclaw/openclaw/issues/77401. Preserve chronological delivery so channel-visible tool/progress callbacks that have been emitted by the embedded agent cannot enqueue after Telegram final delivery starts. Do not add sleeps, Telegram API timing guesses, or broad blocking of tool execution. Prefer a narrow shared callback-drain/ack seam for delivery-affecting `onAgentEvent`/`onPartialReply` progress work, then make Telegram final delivery wait on that seam before final send/edit. Add deterministic regression coverage for fire-and-forget tool/item/command progress callbacks resolving after final delivery begins. Keep plugin/core boundaries intact. Add a user-facing `CHANGELOG.md` fix entry crediting @SheraSnow.
