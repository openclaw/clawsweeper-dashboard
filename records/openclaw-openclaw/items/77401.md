Add a shared callback-drain/ack path for channel-visible progress events so Telegram can flush all pending progress work before final delivery, with a race regression covering partial streaming and tool-heavy turns.

## Repair Work Prompt

Fix the Telegram partial-streaming final-before-progress race reported in https://github.com/openclaw/openclaw/issues/77401. Current main serializes progress/final only after progress reaches Telegram's draft/dispatcher queues, but embedded subscription code calls `onAgentEvent` and `onPartialReply` with `void`, so a callback that awaits typing/progress setup can enqueue after final delivery starts. Implement a narrow shared ordering seam so channel-visible progress callbacks for assistant partials, tool/item events, command output, plan/patch/approval events are tracked and drained before final reply delivery. Prefer a shared agent-runner/subscription drain over Telegram-only sleeps; keep core extension-agnostic and do not add config knobs or arbitrary delays. Add regression tests that simulate a slow progress callback followed immediately by final delivery and assert Telegram progress/update/send completes before final send/edit. Update `CHANGELOG.md` under Unreleased fixes if user-visible behavior changes.
