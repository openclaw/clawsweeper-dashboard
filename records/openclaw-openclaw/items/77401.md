## Summary

Keep open: current main still has fire-and-forget embedded progress callbacks that can let Telegram final delivery overtake late tool-progress preview work, while Telegram only serializes work after it has reached its draft/final queues.

## What This Changes

The issue requests chronological Telegram partial-streaming delivery so tool progress updates always appear before the final assembled reply on tool-heavy turns.

## Best Possible Solution

Add a narrow shared drain/ack path for channel-visible progress callbacks before final dispatch, with deterministic Telegram partial-streaming regression coverage and no sleep-based timing workaround.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a deterministic test shape: delay the embedded onAgentEvent/tool-progress callback before it reaches Telegram, then allow final delivery to drain the current queue first.

## Solution Assessment

No patch is under review. The best fix is a narrow callback-drain/ack ordering seam before final delivery, not Telegram-specific sleeps or network timing assumptions.
