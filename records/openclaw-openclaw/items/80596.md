## Summary

Keep this PR open because current main and the latest release still lack the requested plugin runtime emitter, but the branch is not merge-ready: the new public wrapper still forwards untyped/object payloads into the internal transcript emitter and the PR is currently conflicting with main.

## What This Changes

The PR adds `runtime.events.emitSessionTranscriptUpdate(sessionFile)` to the plugin runtime surface, documents the durable-write expectation, updates the plugin runtime mock, and adds a happy-path runtime test.

## Best Possible Solution

Keep the path-only plugin API only if the wrapper enforces string input at runtime, adds negative coverage for object/message-bearing payloads, and refreshes cleanly against current main before maintainer API acceptance.

## Reproduction Assessment

Yes, by source inspection: PR head forwards the plugin-facing value directly to an internal emitter that accepts full transcript update objects, and gateway code broadcasts message-bearing updates as `session.message`. I did not execute a live JavaScript plugin repro in this read-only pass.

## Solution Assessment

No: a path-only reload notifier is a plausible API shape, but the current implementation only narrows the TypeScript type and leaves the runtime plugin boundary unenforced. The safer narrow fix is to validate or reject non-string values before calling the internal emitter and test that negative path.
