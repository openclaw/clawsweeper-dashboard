## Summary

Keep this PR open, but the current head is not merge-ready: the wrapper narrows the TypeScript signature but still forwards non-string runtime values to the internal message-capable emitter, so the earlier synthetic transcript-message concern remains for JavaScript plugins or casts.

## What This Changes

The PR adds a plugin runtime event method `runtime.events.emitSessionTranscriptUpdate(sessionFile)`, documents it, updates the runtime mock/type surface, and adds a focused runtime test.

## Best Possible Solution

If maintainers want this API, merge only after the emitter is runtime-enforced as path-only, a negative test proves object payloads cannot broadcast messages, and the public plugin SDK contract is approved.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the new wrapper forwards its raw argument to an internal emitter that accepts message-bearing objects. A JavaScript plugin or `any` cast can call the new method with `{ sessionFile, message }` and reach the gateway message broadcast path.

## Solution Assessment

No: the path-only API shape is reasonable if maintainers want this surface, but the current implementation only narrows the TypeScript type and does not enforce the contract at runtime. The safer repair is a runtime string guard plus a negative regression test.
