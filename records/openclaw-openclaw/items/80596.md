## Summary

Keep open: the latest head addresses the direct-emitter safety concern with a path-only wrapper and now has sufficient terminal proof, with no blocking correctness or security finding found; the remaining decision is whether maintainers want this new public plugin runtime API.

## What This Changes

This PR adds `runtime.events.emitSessionTranscriptUpdate(sessionFile)` to the plugin runtime surface, updates the runtime type and SDK mock, adds focused runtime coverage, and documents the durable-write expectation.

## Best Possible Solution

If maintainers accept this API, keep the path-only wrapper and documentation while leaving the message-capable internal emitter private.

## Reproduction Assessment

Not applicable: this is an additive plugin runtime API PR, not a bug report. Current main exposes only transcript listeners through `createRuntimeEvents()`, while the PR head adds a path-only emit method.

## Solution Assessment

Mostly yes, if maintainers want the API: wrapping the internal emitter as `sessionFile: string` is the narrow safe shape and avoids exposing synthetic message payload emission. The remaining question is API ownership, not a code defect in the patch.
