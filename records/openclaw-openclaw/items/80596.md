## Summary

Keep open: the positive durable-write proof is sufficient, but the implementation still lets the plugin-facing wrapper forward non-string payload objects to the internal message-capable transcript emitter, which contradicts the documented path-only contract and can create non-persistent UI-visible messages.

## What This Changes

The PR adds a plugin runtime events method for plugins to emit transcript update notifications by session file path, documents the durable-write contract, updates the runtime mock/type surface, and adds a focused runtime test.

## Best Possible Solution

Expose the API only after the wrapper enforces the path-only contract at runtime, a negative regression test proves message payloads cannot pass through, and maintainers accept the new plugin runtime contract.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: a JS or unknown-casted plugin can call the new wrapper with an object containing sessionFile and message, and the wrapper forwards it to the internal emitter that preserves message fields for gateway broadcast; I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. The path-only wrapper is the right narrow shape, but it needs runtime enforcement, negative coverage for object payloads, and maintainer approval before becoming a supported plugin API.
