## Summary

Keep open: the PR is useful and now has real behavior proof, but the current wrapper does not enforce the path-only contract at runtime, so untyped plugins can still forward message-capable payloads into the internal transcript emitter.

## What This Changes

The PR adds a path-only plugin runtime transcript refresh emitter, documents durable-write usage, updates the runtime mock, and adds focused runtime coverage.

## Best Possible Solution

Keep the additive API path-only if maintainers accept the surface, but enforce that contract at runtime and cover object-payload misuse before merge.

## Reproduction Assessment

Yes: source inspection gives a high-confidence reproduction path for the PR-introduced bug by calling `runtime.events.emitSessionTranscriptUpdate` through an untyped cast with `{ sessionFile, message, messageId }` and observing the listener receive the message payload. The requested feature itself is additive, not a bug report against current main.

## Solution Assessment

No: the path-only wrapper is the right direction if this API is accepted, but the implementation only narrows TypeScript and must add a runtime guard or construct a fresh string-only update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Guard non-string transcript updates before forwarding:** `src/plugins/runtime/runtime-events.ts:9`
  - body: This wrapper only narrows the TypeScript signature. A JS or `unknown` plugin caller can still pass `{ sessionFile, message, messageId }`, and this line forwards that object to the internal emitter, re-enabling the transient UI message path the review was trying to avoid. Validate `typeof sessionFile === "string"` or construct a fresh string-only update before calling the internal emitter, and add a regression test for an object payload cast through the plugin boundary.
  - confidence: 0.95
