## Summary

Close: current main and the v2026.6.6 release now implement active cron task cancellation through the cron runtime/Gateway path, with focused regression coverage for the reported task-ledger cancel flow.

## What This Changes

The issue asks for running cron task records shown by the task ledger to be cancellable via `openclaw tasks cancel <taskId>`, or to give clear operator guidance when no cancellation handle exists.

## Best Possible Solution

Keep the shipped Gateway-owned cron cancellation implementation, and treat any remaining Gmail MCP cleanup symptom as a separate follow-up if it still reproduces.

## Reproduction Assessment

Yes for the original failure: the pre-fix task registry source fell through to the unsupported-runtime branch for cron tasks. Current main no longer reproduces that source path because cron cancellation is handled before the fallback and is covered by focused tests.

## Solution Assessment

Yes. The current implementation uses the live cron/Gateway-owned abort handle and only reports a no-handle result when the cron wrapper does not own cancellable work, which is safer than a registry-only status flip.
