## Summary

Keep this open. Current `main` and the latest release still preserve exact recorded npm specs for id-based plugin updates, the closed linked fix PR was not merged, and the active bulk-update PR only covers trusted-official `--all` sync while leaving this targeted exact-pin dry-run/latest behavior unresolved.

## What This Changes

The issue asks for `openclaw plugins update <id> --dry-run` to report a newer npm `latest` release instead of saying an exact-pinned installed plugin is up to date.

## Best Possible Solution

Make an explicit maintainer decision for targeted exact-pin update semantics, then align code, tests, and docs around either sticky pins plus a clearer unpin/latest-check path, or an intentionally documented latest-advisory mode.

## Reproduction Assessment

Yes, source-reproducible: current `main` preserves exact recorded npm specs for id-based dry-run probes, and live npm metadata confirms exact specs resolve to the pinned versions while bare packages have newer `latest` dist-tags. I did not run the full OpenClaw CLI flow because this review is read-only and should not create local plugin state.

## Solution Assessment

No, the narrow “strip every exact selector” fix is not the best solution by itself because it contradicts current docs and tests that treat exact pins as sticky. The safer path is a maintainer-approved contract that distinguishes intentional pins from stale exact records or adds an explicit latest-check/unpin flow.
