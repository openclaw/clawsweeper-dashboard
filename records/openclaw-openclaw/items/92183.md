## Summary

This issue is still necessary: current main can reproduce the false-negative path from source when a tracked install record contains an exact version. The Feishu case appears to expose a gap introduced by exact-pinning managed official plugin records, while the third-party Weixin case may instead be correct pin behavior, so maintainers must define which exact records are intentional pins before changing update semantics.

## What This Changes

Make plugin update dry-runs discover newer npm releases for unpinned installs instead of resolving a stale exact recorded spec back to itself.

## Best Possible Solution

Give install records an unambiguous distinction between user-requested pins and OpenClaw-managed exact provenance, then have both dry-run and real updates resolve managed/unpinned records through their tracked moving package or catalog target while preserving explicit exact pins.

## Reproduction Assessment

Yes, at source level with high confidence: an exact stale record is reused as the registry probe target, resolves to the installed version, and satisfies the unchanged comparison. This review did not execute a live npm update against the reporter's mirrored registry.

## Solution Assessment

No, comparing every tracked install directly with dist-tags.latest is not the best general fix because it would defeat documented exact pins. The narrow maintainable repair is to preserve explicit pin intent while restoring moving resolution for managed or historically accidental exact records.
