## Summary

Keep this open: current main and the latest release still preserve exact npm install specs during id-based plugin updates, and the open fixing PR has not merged. The remaining decision is product/compatibility-sensitive because current docs and tests say exact pins stay pinned unless the user passes an unversioned package spec.

## What This Changes

The issue asks for `openclaw plugins update <id> --dry-run` to detect newer npm `latest` versions even when the tracked plugin record is pinned to an exact package version.

## Best Possible Solution

Resolve the exact-pin contract explicitly, then align code, docs, tests, and real CLI proof so id-based updates either preserve pins with a clearer latest-advisory path or intentionally move exact pins to the registry default line.

## Reproduction Assessment

Yes, at source level: current main passes the recorded exact npm spec into the update probe, and live npm queries show exact specs resolve to the pinned version while bare package names resolve to newer latest releases. I did not run a full `openclaw plugins update ... --dry-run` CLI repro in this read-only review.

## Solution Assessment

No final fix direction yet. The linked PR is a plausible root-cause patch, but it is not clearly the best solution until maintainers decide whether exact pins should stay sticky or whether OpenClaw should add an explicit latest-check/advisory behavior.
