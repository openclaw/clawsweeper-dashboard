## What This Changes

The issue asks for `openclaw plugins update <name> --dryrun` to report an available npm update when an installed plugin record is pinned to an older exact version but the package `latest` dist-tag is newer.

## Best Possible Solution

Define the exact-pin/latest-check contract once, then land or replace the linked PR with code, docs, and regression coverage that preserves intentional pins while surfacing stale or unintended exact records.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main feeds exact recorded npm specs into the dry-run probe, and live npm metadata shows the exact examples resolve lower than their package latest dist-tags. I did not run a full `openclaw plugins update` CLI repro because this review is read-only.

## Solution Assessment

No, not as a literal always-float-to-latest change: that conflicts with documented exact-pin semantics. The best fix needs maintainer-approved behavior that distinguishes intentional pins from stale or unintended exact records.
