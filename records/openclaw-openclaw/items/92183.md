## Summary

Keep this open: the reported mechanics are source-reproducible, but an open linked fix PR already owns this exact issue and the remaining decision is how latest-check behavior should coexist with documented exact-pin semantics.

## What This Changes

The issue requests that `openclaw plugins update <name> --dry-run` report a newer npm latest version instead of treating an exact-version installed plugin record as up to date.

## Best Possible Solution

Resolve this through the linked PR or the broader plugin-update contract issue by defining an explicit downgrade-safe/latest-check behavior that preserves intentional exact pins, then align code, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main reuses exact recorded npm specs for id-based dry-run update probes and reports unchanged when the probe resolves to the installed version. I did not run a live npm CLI reproduction because this review is read-only.

## Solution Assessment

No, the literal “always compare exact pins to latest” path is not clearly the best fix because current docs intentionally preserve exact pins. The maintainable solution needs an explicit contract that distinguishes intentional pins from stale or unintended exact records.
