## Summary

The report identifies a real CLI clarity problem, but the observed result comes from tracked exact-version selectors rather than a missing npm metadata lookup. Current behavior deliberately honors pins, while the output does not explain that the plugin is only “up to date” relative to its recorded selector.

## What This Changes

Clarify and improve plugin dry-run update behavior when a tracked exact-version selector prevents discovery of a newer npm default release.

## Best Possible Solution

Keep exact versions and explicit tags pinned, but make dry-run say that no update exists for the recorded selector and show the command needed to switch to the default release line; optionally report a newer `latest` as informational without treating it as the update target.

## Reproduction Assessment

Yes. With an installed version matching a tracked exact npm spec, current main resolves that same exact target and emits “up to date”; the reporter's missing install records prevent proving why those records were exact, but the output path itself is clear from source.

## Solution Assessment

No. Unconditionally comparing against `dist-tags.latest` is not the best fix because it conflicts with deliberate exact pins and explicit tags; the narrow fix is selector-aware output and an explicit opt-in path back to the default release line.
