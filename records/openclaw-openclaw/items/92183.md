## Summary

This issue is a well-supported second manifestation of the recorded-version replay already tracked by the older canonical report: current main and v2026.6.6 reuse an exact stored npm spec for id-based updates, causing both false-negative dry-runs and real downgrades.

## What This Changes

Make plugin dry-run updates detect newer npm releases instead of treating an older exact-version install record as the update target.

## Best Possible Solution

Consolidate both manifestations under https://github.com/openclaw/openclaw/issues/84256 and define one update contract that distinguishes intentional pins from stored resolution provenance, covering dry-run and real updates without silently downgrading packages.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction: an exact stored npm spec remains the stable update target, and dry-run compares the installed version against that same exact version; the reporter also supplied two real CLI captures.

## Solution Assessment

No, an unconditional comparison with `dist-tags.latest` is not the complete solution because explicit pins and non-default tags must remain meaningful. The canonical issue should define how intentional pins differ from historical install provenance, then apply that rule consistently to dry-run and actual updates.
