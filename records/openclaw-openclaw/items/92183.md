## Summary

Keep open: current `main` and latest release still route targeted plugin-id dry-runs through the recorded exact npm spec, while current docs/tests intentionally keep exact pins sticky. The direct fix PR was closed unmerged, and the open related PR only covers bulk official-plugin sync, not this targeted third-party latest-check case.

## What This Changes

The issue requests that `openclaw plugins update <name> --dryrun` detect and report a newer npm `latest` dist-tag instead of saying an exact-version-pinned installed plugin is up to date.

## Best Possible Solution

Define the exact-pin/latest-check contract first, then align targeted update behavior, user-facing messaging, docs, and regression tests so intentional pins remain understandable while stale exact install records are discoverable.

## Reproduction Assessment

Yes, source-reproducible: current main sends recorded exact npm specs into dry-run probes, and current npm registry metadata still shows newer defaults than the exact versions in the report. I did not run a mutating or cache-writing live `openclaw plugins update` command in this read-only checkout.

## Solution Assessment

No, not as originally stated: always comparing targeted exact pins to `latest` would conflict with the current documented pin contract. The maintainable solution needs a maintainer-approved distinction between intentional exact pins, stale recorded exact specs, and explicit requests to move back to the registry default line.
