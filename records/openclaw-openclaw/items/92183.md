## Summary

This is a valid current and shipped plugin-manager bug, not a duplicate and not fixed on `main`. The report’s two real CLI captures match the npm dry-run path, while the documented contract says moving package specs should follow registry releases and only explicit versions or tags stay pinned.

## What This Changes

Fix `openclaw plugins update <name> --dry-run` so moving npm plugin specs report newer registry-resolved versions instead of falsely reporting the installed version as current.

## Best Possible Solution

Keep one canonical npm resolution path: make dry-run outcomes use the version resolved for a moving tracked spec, preserve explicit version/tag pins, and add focused real-registry plus unit coverage for both behaviors.

## Reproduction Assessment

Yes. The reporter provides two concrete real CLI reproductions, and current source exposes the exact registry-probe and unchanged-outcome path that can produce the false negative; this read-only review did not rerun the command itself.

## Solution Assessment

Yes, with one constraint: fix the existing npm installer/update resolution boundary and preserve explicit pins, rather than adding a parallel `npm view dist-tags.latest` client or changing all exact specs to moving updates.
