## Summary

Keep open: current main and the latest release still reuse the recorded exact npm spec for id-based plugin updates, and the linked open PR is the active implementation candidate but needs maintainer judgment because the requested latest-check behavior conflicts with documented exact-pin semantics.

## What This Changes

The issue requests that `openclaw plugins update <name> --dry-run` surface a newer npm `latest` version instead of reporting an exactly pinned installed plugin as up to date.

## Best Possible Solution

Define a maintainer-approved plugin update contract that preserves intentional exact pins by default while adding a clear latest-advisory, unpin, or downgrade-safe path with aligned docs, regression tests, and real CLI proof.

## Reproduction Assessment

Yes, at source level: current main feeds the recorded exact npm spec into the dry-run update probe, and live npm metadata shows exact specs resolve to the pinned version while bare package names resolve to the newer default. I did not run a mutating local plugin setup reproduction in this read-only review.

## Solution Assessment

No, not as an unqualified default change: stripping exact selectors makes the reported false negative disappear, but it also silently changes documented exact-pin behavior. The safer solution is a maintainer-approved latest-advisory or unpin path that keeps intentional pins sticky.
