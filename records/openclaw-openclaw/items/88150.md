## Summary

Keep open for maintainer review: the patch appears correct and proof-sufficient for the focused cache-retention bug, but it changes default cache residency for large existing stores and still needs maintainer acceptance plus a green gate.

## What This Changes

The branch adds a 16 MiB runtime session-store cache cap, an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override/test hook, and regression coverage proving oversized stores are reread instead of retained.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the default cap, while keeping broader sessions.json pruning, bootstrap-cache, and long-soak work tracked separately.

## Reproduction Assessment

Yes, at source/proof level: current main can retain parsed session-store object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being reread instead of cached. I did not reproduce a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cap: the cache module is the right boundary because loaders and writers already pass file size metadata into cache reads and writes. A gateway-call-site-only fix would duplicate cache policy and be easier to miss.
