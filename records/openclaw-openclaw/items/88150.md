## Summary

Keep open for maintainer review: the patch is focused and I found no blocking code defect, but current main still lacks the runtime cache cap and the PR changes default cache residency plus adds an environment override that maintainers should explicitly accept or tune.

## What This Changes

The branch adds a 16 MiB default runtime byte cap, an OPENCLAW_SESSION_CACHE_MAX_BYTES override, a test-only override, and regression coverage for oversized session-store cache retention.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the default cap, while keeping broader sessions.json pruning and long-soak memory work on their separate trackers.

## Reproduction Assessment

Yes, at source/proof level: current main can cache parsed session-store objects and snapshots without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cap: the cache module is the right boundary because callers already pass file size metadata into cache reads and writes. A gateway-call-site-only fix would duplicate policy and be easier to miss.
