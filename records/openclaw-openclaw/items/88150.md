## What This Changes

The PR adds a 16 MiB runtime session-store cache cap with an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override and tests that oversized stores are re-read instead of retained in object, snapshot, or serialized caches.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the default cap, while keeping broader sessions.json pruning, prompt-report bloat, bootstrap-cache, and long-soak work on their own trackers.

## Reproduction Assessment

Yes, at source/proof level: current main can retain parsed session-store object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cap: the cache module is the narrowest maintainable boundary because callers already pass file size metadata into cache reads and writes. A gateway-call-site-only fix would duplicate policy and be easier to miss.
