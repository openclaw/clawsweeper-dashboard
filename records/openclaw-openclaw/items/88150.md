## Summary

Keep open: current main and v2026.6.10 still lack the runtime session-store cache byte cap, and this PR remains the focused candidate fix with sufficient terminal proof but a compatibility-sensitive default that needs maintainer acceptance before merge.

## What This Changes

The PR adds a 16 MiB runtime byte cap, an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override, and regression tests so oversized session stores are re-read instead of retained in live caches.

## Best Possible Solution

Land this size-aware cache boundary after maintainers accept or tune the default cap, while keeping broader sessions.json pruning, schema, bootstrap-cache, and long-soak work on their own trackers.

## Reproduction Assessment

Yes, at source/proof level: current main can retain parsed session-store object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cap: the cache module is the narrowest maintainable boundary because callers already pass file size metadata into cache reads and writes. A caller-only fix would duplicate cache policy and be easier to miss.
