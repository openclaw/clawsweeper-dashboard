## Summary

Keep open: the patch appears to address a real current-main session-store cache retention bug, but it changes default runtime cache policy and adds an operator env override, so the remaining action is maintainer acceptance or tuning before merge.

## What This Changes

The PR adds a 16 MiB runtime session-store cache cap with an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override and regression tests for oversized stores being re-read instead of retained.

## Best Possible Solution

Land the size-aware session-store cache boundary once maintainers accept or tune the 16 MiB default, keeping disk as source of truth and tracking bootstrap, webgate, or long-soak memory work separately.

## Reproduction Assessment

Yes, at source/proof level: current main retains parsed session-store objects and snapshots without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM reproduction in this read-only review.

## Solution Assessment

Yes, with a maintainer decision on the default: the session-store cache is the right owner boundary because its callers already pass `sizeBytes` and the patch gates all runtime cache surfaces there. A gateway-call-site-only fix would be narrower but one-sided.
