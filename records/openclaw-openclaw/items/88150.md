## Summary

Keep open: current main and v2026.6.8 still lack the runtime byte cap for parsed session-store object and snapshot caches, and this focused PR remains the live implementation path. ## What This Changes

The PR adds a 16 MiB runtime byte cap and test override for session-store object, snapshot, and serialized caches, plus regression coverage for oversized stores being re-read instead of retained.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the default cap, while keeping broader sessions.json pruning, bootstrap-cache, and long-soak work tracked separately.

## Reproduction Assessment

Yes, at source/proof level: current main can retain parsed session-store object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cap: the cache module is the right boundary because callers already pass file size metadata into cache reads and writes. A gateway-call-site-only fix would duplicate cache policy and be easier to miss.
