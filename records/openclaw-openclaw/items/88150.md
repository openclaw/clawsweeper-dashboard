## Summary

Keep this PR open: current main and the latest release still lack the runtime session-store cache byte cap, and the patch is a focused candidate fix, but it remains compatibility-sensitive and needs maintainer acceptance plus fresh green checks before merge.

## What This Changes

The PR adds a 16 MiB runtime session-store cache retention cap, an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override, and regression tests proving oversized stores are re-read instead of retained.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the default cap, while keeping broader sessions.json pruning and long-soak memory work tracked separately.

## Reproduction Assessment

Yes, at source/proof level: current main can retain parsed session-store object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the cap: the cache module is the right boundary because callers already pass file size metadata into cache reads and writes. A gateway-call-site-only fix would duplicate cache policy and be easier to miss.
