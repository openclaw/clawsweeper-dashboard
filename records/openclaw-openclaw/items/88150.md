## Summary

Keep open: the PR is a focused and plausible fix for the session-store cache-retention bug, but it still needs maintainer acceptance or tuning of the compatibility-sensitive 16 MiB default and env override before merge.

## What This Changes

Adds a runtime byte cap and `OPENCLAW_SESSION_CACHE_MAX_BYTES` override for session-store object, snapshot, and serialized caches, with regression tests for oversized stores being re-read instead of retained.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the default cap, and keep broader session-store OOM, bootstrap-cache, and long-soak work tracked separately.

## Reproduction Assessment

Yes at source/proof level: current main retains parsed object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the default: the cache module is the right owner boundary because callers already pass file size metadata into cache reads and writes. A gateway-call-site-only fix would be easier to miss and would duplicate cache policy.
