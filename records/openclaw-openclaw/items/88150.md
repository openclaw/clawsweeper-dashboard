## Summary

Keep open: the PR is still necessary because current main lacks a runtime byte cap for retained session-store object and snapshot caches, and the patch looks correct but still needs maintainer acceptance of the new 16 MiB default and environment override.

## What This Changes

The PR adds a size-aware runtime session-store cache cap, an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override, and regression tests proving oversized stores are re-read instead of retained in runtime caches.

## Best Possible Solution

Land the size-aware session-store cache boundary once maintainers accept or tune the 16 MiB default, keeping disk as the source of truth and tracking any remaining bootstrap, webgate, or long-soak memory work separately.

## Reproduction Assessment

Yes, at source/proof level: current main retains parsed session-store objects and snapshots without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM reproduction in this read-only review.

## Solution Assessment

Yes, with a maintainer decision on the default: the session-store cache is the right owner boundary because its callers already pass `sizeBytes` and the patch gates all runtime cache surfaces there. A gateway-call-site-only fix would be narrower but one-sided.
