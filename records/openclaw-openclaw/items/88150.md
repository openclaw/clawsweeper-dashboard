## What This Changes

The PR adds a 16 MiB runtime session-store cache cap with an `OPENCLAW_SESSION_CACHE_MAX_BYTES` override and tests that oversized stores are re-read instead of retained.

## Best Possible Solution

Land the size-aware session-store cache boundary after maintainers accept or tune the 16 MiB default, and track bootstrap-cache, webgate, or long-soak memory work separately.

## Reproduction Assessment

Yes, at source/proof level: current `main` retains parsed object and snapshot caches without a runtime byte cap, and the PR body plus tests show oversized stores being re-read instead of cached. I did not run a live OOM reproduction in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the default: the cache module is the narrow owner boundary because callers already pass `sizeBytes` into the cache API. A gateway-call-site-only fix would be more one-sided and easier to miss.
