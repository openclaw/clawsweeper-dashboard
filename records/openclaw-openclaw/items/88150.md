## Summary

Keep open: current main and v2026.6.8 still lack the runtime session-store cache cap, and this PR remains the focused implementation path; merge still needs maintainer acceptance or tuning of the compatibility-sensitive default and env override.

## What This Changes

The PR adds a 16 MiB runtime byte cap and `OPENCLAW_SESSION_CACHE_MAX_BYTES` override for session-store object, snapshot, and serialized caches, with regression tests for oversized stores re-reading from disk.

## Best Possible Solution

Land the size-aware cache boundary after maintainers accept or tune the 16 MiB default/env override, while keeping broader session-store OOM cleanup tracked separately.

## Reproduction Assessment

Yes, at source/proof level: current main caches parsed object and snapshot stores without a runtime byte cap, and the PR proof/tests demonstrate oversized stores re-reading from disk. I did not run a live OOM soak in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the default: the cache module is the narrow owner boundary because callers already pass `sizeBytes` into cache reads and writes. A caller-only fix would be more one-sided and easier to miss.
