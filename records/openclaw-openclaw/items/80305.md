## Summary

The requested BlueBubbles suppression knob is reasonable for the old released plugin, but current main has removed the bundled BlueBubbles channel and the remaining supported/core path is iMessage via imsg. The channel-plugin surface already gives an external BlueBubbles plugin ownership over config, outbound delivery, threading, and receipt metadata, so this belongs on ClawHub rather than re-expanding OpenClaw core.

## What This Changes

The issue asks for a BlueBubbles channel config option to suppress outbound message-id system events while preserving reply-cache state.

## Best Possible Solution

Keep the core-supported path on iMessage/imsg and let any BlueBubbles-backed continuation publish as a ClawHub channel plugin that owns this config knob.

## Reproduction Assessment

Not applicable: this is a request for a new config option on a removed/deprecated BlueBubbles channel, and current main has no BlueBubbles runtime to exercise. The old released behavior is described clearly, but it is not a current-main bug path.

## Solution Assessment

No: adding `channels.bluebubbles.*` back to core would re-expand a removed channel surface. The maintainable solution is iMessage migration or an external BlueBubbles channel plugin published through ClawHub.
