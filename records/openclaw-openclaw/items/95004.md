## Summary

Keep open: current main and the latest release still intentionally suppress short first Telegram previews and answer-lane deltas in progress mode, and maintainer notes make this a Telegram UX/product decision rather than a safe cleanup close.

## What This Changes

The issue asks for Telegram preview streaming to show a predictable lightweight draft or progress placeholder whenever upstream draft/progress state exists.

## Best Possible Solution

Have maintainers choose and document a Telegram-owned policy: preserve the first-preview debounce by default, add a tested single-preview placeholder or delayed materialization only if it satisfies the no-spam intent, or document the Web/Telegram tradeoff clearly.

## Reproduction Assessment

Yes at source level: current tests and dispatch wiring prove short first previews below `minInitialChars` are not materialized, and progress mode suppresses answer-lane partial updates. I did not run live Telegram proof in this read-only review.

## Solution Assessment

No as an automatic code change: the request includes several acceptable behavior directions, and existing Telegram notes intentionally keep the first-preview debounce. The narrow maintainable solution needs maintainer choice before implementation.
