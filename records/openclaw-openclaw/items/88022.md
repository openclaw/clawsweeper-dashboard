## Summary

Keep open: current main still emits the Gateway auth SecretRef warning from doctor when CLI-side resolution fails, and the warning path does not first distinguish a healthy running Gateway that already resolved the token. The report is source-reproducible and touches Gateway auth diagnostics, so it should stay open for maintainer review rather than conservative cleanup.

## What This Changes

The issue asks doctor to stop warning that Gateway token SecretRefs are unavailable when the running Gateway has resolved the token and is connected.

## Best Possible Solution

Teach doctor Gateway-auth diagnostics to distinguish CLI-local SecretRef resolution failure from a healthy running Gateway runtime, preferably by using active runtime snapshot or Gateway health/auth facts while preserving warnings for genuinely unresolved startup/auth paths.

## Reproduction Assessment

Unclear for the exact macOS file/exec environment because no live SecretRef setup was run. Source inspection gives a clear current-main path where doctor warns on CLI-local SecretRef resolution failure without first using running-Gateway state.

## Solution Assessment

Unclear: the requested direction is valid, but the safest fix needs a maintainer choice between active-runtime-snapshot use, connected-Gateway suppression, or clearer diagnostics so real unresolved auth failures remain visible.
