## Summary

Keep open: current main and the latest release fixed SecretRefs that doctor can resolve locally, but the remaining request is different. The `doctor:gateway-auth` warning still runs before Gateway health proof and still warns when CLI-local token resolution fails, so the runtime-healthy/CLI-unavailable case needs maintainer auth-policy direction.

## What This Changes

The issue asks doctor to avoid a false `gateway.auth.token` SecretRef-unavailable warning when the Gateway is running and using that SecretRef successfully, even if the CLI doctor process cannot resolve it.

## Best Possible Solution

Classify this warning through Gateway-auth-specific runtime proof or an active runtime snapshot path, while preserving fail-closed diagnostics when the configured SecretRef cannot be proved usable.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible but not live-reproduced: current code will emit the warning when CLI-local `resolveGatewayAuthToken` cannot resolve the configured ref, and that path runs before Gateway health proof is collected.

## Solution Assessment

Is this the best way to solve the issue? Not yet; the current implementation is only the best fix for CLI-resolvable refs, while the remaining runtime-healthy case needs a maintainer-approved token-specific proof path rather than a broad reachability suppression.
