## Label Justifications

- `P1`: The report describes a live Discord group workflow becoming uncontrollable because old queued messages were processed as fresh agent turns after long delays.
- `impact:message-loss`: The failure mode is channel delivery/order drift where stale Discord messages reach the agent pipeline as current turns while current steering is delayed.
- `impact:session-state`: The stale backlog is appended as new user turns and interacts with `/stop`, session history, and active-run state.
- `maturity:stable`: The affected Discord Conversation Routing and Delivery surface maps to the Discord M4 Stable scorecard entry.

## Summary

Keep open: current main still carries Discord source timestamps into session context, but accepted Discord group jobs are serialized and dispatched without source-age, stale-event, or backlog coalescing policy, and no merged or open canonical fix now owns this exact stale-backlog problem.

## What This Changes

Add Discord group backlog freshness handling so delayed stale messages are logged, marked/coalesced/dropped before normal agent dispatch, and `/stop` handles queued stale work for the session.

## Best Possible Solution

Define a maintainer-approved Discord/group-channel freshness policy that carries source age through receive, enqueue, append, and dispatch, then add `/stop` drain or stale-mark semantics and regression coverage for delayed group backlog.

## Reproduction Assessment

Yes at source level, not live. The issue provides timestamp deltas, current Discord code preserves source timestamps, and the existing queue test proves delayed queued jobs still dispatch after the earlier job completes.

## Solution Assessment

Unclear until maintainers choose the policy. A Discord-only age check is plausible, but the best fix likely needs shared freshness metadata plus Discord group policy and `/stop` queue semantics.
