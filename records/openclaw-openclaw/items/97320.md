## Label Justifications

- `P1`: The report describes a real Discord group workflow becoming uncontrollable because old queued messages are processed as fresh agent turns after long delays.
- `impact:message-loss`: The failure mode is channel delivery/order drift where stale Discord messages reach the agent pipeline as current turns while current steering is delayed.
- `impact:session-state`: Stale backlog is appended as new user turns and interacts with /stop, session history, and active-run state.
- `maturity:stable`: The affected Discord Conversation Routing and Delivery surface is listed as M4 Stable in the maturity shortlist.

## Summary

Keep open: current main still preserves Discord source timestamps but accepts and serializes inbound Discord jobs without an age guard, stale marker, or queue-age policy before dispatching them as normal turns.

## What This Changes

Request stale/backlog handling for Discord group messages so old queued events are logged, coalesced, dropped, or marked stale before normal agent dispatch.

## Best Possible Solution

Define a Discord/group-channel freshness policy that carries source age through receive, enqueue, append, and dispatch, handles stale low-value room events without normal full turns by default, and gives /stop explicit queued-work drain or stale-mark semantics.

## Reproduction Assessment

Yes at source level, not live. Discord retains the source timestamp, then accepted jobs serialize by session/channel with no age guard; the existing queue test proves delayed jobs still dispatch after an earlier run completes.

## Solution Assessment

Unclear until maintainers choose policy. A Discord-only age check would reduce stale backlog, but the best fix likely needs shared freshness metadata plus Discord group policy and /stop drain semantics so it does not conflict with broader scheduler work.
