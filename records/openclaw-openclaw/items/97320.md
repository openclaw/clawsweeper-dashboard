## Label Justifications

- `P1`: The report describes a real Discord group workflow becoming uncontrollable because old queued messages are processed as fresh turns after long delays.
- `impact:message-loss`: The failure mode is channel delivery/order drift: stale Discord messages are delivered into the agent turn pipeline as if they were current, while current steering is delayed.
- `impact:session-state`: Stale backlog is appended as new user turns and interacts with `/stop`, session history, and active-run state.
- `maturity:stable`: The affected Discord Conversation Routing and Delivery surface is scored M4 Stable in `qa/maturity-scores.yaml` with quality 73 and completeness 87.

## Summary

Keep open: current main preserves Discord source timestamps but accepts and serializes inbound jobs per session/channel without an age guard, stale marker, or Discord-owned timeout, and existing queue tests show later messages can wait behind long runs before dispatch.

## What This Changes

The issue requests Discord group backlog/age handling so stale source messages are logged, marked/coalesced/dropped before normal agent dispatch, and `/stop` prevents queued stale work from replaying as fresh turns.

## Best Possible Solution

Define a Discord/group-channel freshness policy that carries source age through receive, enqueue, append, and dispatch; logs age at each stage; drops or observe-only handles stale low-value room events by default; and makes `/stop` drain, cancel, or stale-mark queued work for the affected session with regression coverage.

## Reproduction Assessment

Yes at source level, not live. Discord carries the source message timestamp into `ctx.Timestamp`, then accepted jobs serialize by session/channel with no age guard; current tests show later jobs can wait behind long runs and still dispatch afterward.

## Solution Assessment

Unclear until maintainers choose policy. A Discord-only age check would reduce stale backlog, but the best fix likely needs shared freshness metadata plus Discord group policy and `/stop` drain semantics so it does not conflict with broader same-session scheduler work.
