## Label Justifications

- `P1`: The report describes a real Discord group workflow becoming uncontrollable because old queued messages are processed as fresh agent turns after long delays.
- `impact:message-loss`: The failure mode is channel delivery/order drift where stale Discord messages reach the agent pipeline as current turns while current steering is delayed.
- `impact:session-state`: Stale backlog is appended as new user turns and interacts with `/stop`, session history, and active-run state.
- `maturity:stable`: The affected Discord channel surface is scored M4 Stable in `qa/maturity-scores.yaml`.

## Summary

Keep open: current main still carries Discord source timestamps but accepts and serializes inbound Discord jobs without a queue-age or stale-event policy before normal agent dispatch.

## What This Changes

Request stale/backlog handling for Discord group messages so old queued events are logged, coalesced, dropped, or marked stale instead of becoming fresh agent turns after long delays.

## Best Possible Solution

Define a Discord/group-channel freshness policy that carries source age through receive, enqueue, append, and dispatch, suppresses stale low-value group events by default, and gives `/stop` explicit queued-work drain or stale-mark semantics.

## Reproduction Assessment

Yes, at source level. Discord retains the source timestamp, accepted jobs serialize by session/channel with no age guard, and existing queue coverage proves delayed jobs still dispatch after earlier work completes.

## Solution Assessment

Unclear until maintainers choose policy. A Discord-only age check would reduce stale backlog, but the best fix likely needs shared freshness metadata plus Discord group policy and `/stop` drain semantics.
