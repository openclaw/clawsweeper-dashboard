## Summary

Close: the actionable fix is in the separately published WeCom plugin, not OpenClaw core. Current main and v2026.6.9 only catalog and pin `@wecom/wecom-openclaw-plugin@2026.5.7`, and that published package contains the disabled per-chat queue that matches the reported separated file-plus-text race and the upstream plugin issue.

## What This Changes

The issue reports a WeCom separated text-plus-file callback deadlock where model output is produced internally but the plugin WebSocket stream never sends final reply frames.

## Best Possible Solution

Fix and release the WeCom runtime in `WecomTeam/wecom-openclaw-plugin`, then update OpenClaw’s external catalog pin and integrity only after a fixed package is published.

## Reproduction Assessment

Yes, at source level. I did not run a live WeCom setup, but the reporter’s steps match the published plugin’s disabled per-chat queue and the upstream plugin’s open file-plus-text race report.

## Solution Assessment

No, keeping this as OpenClaw core work is not the best solution. The narrow repair belongs in the external WeCom plugin; OpenClaw should only consume a fixed release through its catalog.
