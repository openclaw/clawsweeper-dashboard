## Summary

Keep open: current main still has the reported `memory-lancedb` packaging gap, because the plugin declares `@lancedb/lancedb` but no `apache-arrow` peer while the documented `ltm` CLI commands lazily load LanceDB on first database access.

## What This Changes

The issue asks OpenClaw to fix bundled `memory-lancedb` packaging so `openclaw ltm` does not hang or fail when LanceDB's `apache-arrow` peer is missing.

## Best Possible Solution

Keep the fix inside `memory-lancedb` packaging and dependency contracts so LanceDB has its Arrow peer in packaged installs without moving LanceDB ownership into core.

## Reproduction Assessment

No live high-confidence CLI reproduction was established in this read-only environment. Source inspection still gives a clear path: `openclaw ltm` reaches `MemoryDB.doInitialize`, imports `@lancedb/lancedb`, and the plugin package omits the peer dependency that the lockfile records for LanceDB.

## Solution Assessment

Yes. The narrow maintainable fix is to declare and test the missing plugin-owned runtime dependency, with package-path validation; core Gateway memory behavior should stay unchanged unless validation exposes a generic plugin dependency-contract gap.
