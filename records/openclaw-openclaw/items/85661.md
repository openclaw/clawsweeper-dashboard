## Summary

Current main and the shipped v2026.5.20 release already implement the requested GitHub Copilot live model catalog path: the plugin fetches `/models`, marks the manifest catalog refreshable, and startup/runtime discovery writes those rows into the model catalog when Copilot is configured.

## What This Changes

The issue asks for GitHub Copilot model availability to track the live Copilot `/models` endpoint or expose a user-facing refresh/documented workaround.

## Best Possible Solution

Keep the shipped provider-owned live catalog discovery and static fallback, and ask users on 2026.5.12 to upgrade to v2026.5.20 or newer before reopening with fresh evidence.

## Reproduction Assessment

No current-main failing path remains: source and release-tag inspection show Copilot `/models` discovery is now wired into the provider catalog and the static fallback is refreshed. The original 2026.5.12 behavior is covered by the merged live-catalog PR history.

## Solution Assessment

Yes. The shipped implementation keeps Copilot discovery inside the GitHub Copilot plugin, preserves static fallback behavior, and avoids adding a separate core-specific refresh command.
