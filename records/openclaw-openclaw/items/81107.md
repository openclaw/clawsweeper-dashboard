## Summary

Keep open: current main still runs Discord native skill command discovery before Discord client lifecycle startup, and duplicate skill names can still be uniqued/logged before the final by-skillName dedupe removes them from the returned list. The broader READY reports are useful context, but this issue has a distinct startup-load path that is not clearly resolved by the existing readiness fixes.

## What This Changes

The issue reports that Discord native skill command discovery repeatedly de-duplicates `/discord` commands on v2026.5.7, saturating CPU and blocking Discord gateway readiness.

## Best Possible Solution

Collapse duplicate skill identities before native command-name uniquing/deploy work on Discord startup, while preserving per-agent allowlists and the generic `/skill` fallback.

## Reproduction Assessment

No, not for the full live WSL2 Discord CPU saturation and READY hang. Source inspection does show a plausible current-main path where duplicate skill identities are uniqued and logged before final `skillName` dedupe on Discord startup.

## Solution Assessment

Yes, the narrow maintainable fix is to avoid building/registering duplicate native skill command specs before Discord reaches client/gateway lifecycle work. A config workaround or readiness retry alone would not remove the startup CPU/logging work.

## Repair Work Prompt

Fix the Discord skill command duplicate startup-load bug reported in https://github.com/openclaw/openclaw/issues/81107. Current main builds workspace skill command specs with a shared reserved-name set and only dedupes by `skillName` after command-name uniquing/logging; Discord calls this before `createDiscordMonitorClient`, so duplicate plugin skills can inflate native command work before gateway lifecycle startup. Move or add a pre-dedupe step so duplicate skill identities are collapsed before per-workspace command-name uniquing/deploy work, while preserving per-agent allowlists, shared-workspace filter merging, command authorization, and the generic `/skill` fallback. Do not change Discord dependencies, plugin install/update behavior, or add a config flag. Add focused regression coverage that duplicate skill names across agents/workspaces do not produce suffixed command specs/logging or an inflated Discord command count. Add a user-facing changelog fix entry if behavior changes.
