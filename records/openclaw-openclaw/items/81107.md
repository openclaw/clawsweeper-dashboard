## Summary

Keep open: current main and the latest release still run Discord native skill discovery through the global skill-command lister on startup, and duplicate skill names are still uniqued/logged before the final by-skillName de-dupe removes them. The broader Discord READY/1006 trackers are useful context, but I did not find an open or merged item that specifically owns this skill-command startup-load path.

## What This Changes

Issue reports Discord startup CPU saturation and readiness failure when repeated external-plugin skill commands generate many duplicate slash-command names.

## Best Possible Solution

Collapse duplicate skill identities before slash-command name uniquing and native Discord command generation, while preserving per-agent allowlists, shared-workspace merging, and the generic `/skill` fallback.

## Reproduction Assessment

No for the full live CPU-saturation plus Discord WebSocket hang on current main. Source inspection does show the current-main path where duplicate skill names are uniqued and logged before final by-skillName de-dupe removes them.

## Solution Assessment

Yes: the maintainable fix direction is a narrow source change that avoids duplicate skill-command work before Discord startup deploys native commands. A config workaround or broader gateway lifecycle change would not directly address the duplicate command generation path.
