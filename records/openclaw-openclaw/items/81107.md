## Summary

Keep open: current main and the latest release still build Discord native skill commands before Discord client lifecycle work, and duplicate skill names are still removed only after per-workspace command-name uniquing/logging has already run. The related readiness issues are useful context, but this report describes a distinct startup-load path that is not fixed or superseded.

## What This Changes

The issue reports a v2026.5.7 Discord startup hang where repeated skill command name de-duplication saturates CPU before Discord gateway readiness completes.

## Best Possible Solution

Collapse duplicate canonical skill identities before native Discord command-name uniquing and deploy work, while preserving per-agent allowlists, workspace grouping, command limits, and the generic `/skill` fallback.

## Reproduction Assessment

No, not for the full live CPU-saturation plus WebSocket hang. Source inspection does show a clear duplicate skill-command generation path before final de-duplication, which is enough to keep the issue actionable.

## Solution Assessment

Yes, the narrow maintainable fix is to avoid building/registering duplicate native skill command specs before Discord startup reaches client/gateway lifecycle work. A readiness-only patch or config workaround would not address the repeated pre-dedupe work.
