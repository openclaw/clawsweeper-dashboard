Keep open: current main still resolves Discord native skill commands on the startup path, still performs slash-command name uniquing before the final by-skillName de-dupe, and no related READY fix owns this remaining skill-command startup-load path.

## What This Changes

The issue reports a Discord external-plugin skill command de-duplication loop that can saturate CPU and leave Discord stuck awaiting gateway readiness.

## Best Possible Solution

Collapse duplicate skill identities before slash-command name uniquing and native Discord command generation, while preserving per-agent allowlists, canonical workspace grouping, and the generic `/skill` fallback.

## Reproduction Assessment

No for the full live CPU-saturation plus Discord WebSocket hang on current main. Source inspection does show the current-main path where duplicate skill names are uniqued and logged before final by-skillName de-dupe removes them.

## Solution Assessment

Yes: the narrow maintainable direction is to avoid duplicate skill-command work before Discord startup deploys native commands. A config workaround or broader gateway lifecycle change would not address the duplicate command generation path directly.
