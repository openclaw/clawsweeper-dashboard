## Summary

Keep open: current main still resolves Discord native skill commands before Discord client startup, and the current source only drops duplicate skill names after per-workspace command-name uniquing/logging has already run. The related readiness reports are useful context, but this issue adds a distinct skill-command startup-load path that is not clearly fixed by https://github.com/openclaw/openclaw/issues/60559 or https://github.com/openclaw/openclaw/issues/61703.

## What This Changes

The issue reports a v2026.5.7 Discord startup hang where repeated skill command deduplication appears to saturate the event loop and prevent gateway readiness handling.

## Best Possible Solution

Collapse duplicate skill identities before native Discord command-name uniquing/deploy work on startup, while preserving per-agent allowlists and the generic `/skill` fallback.

## Reproduction Assessment

No, not for the full CPU-saturation plus Discord WebSocket hang in a real Discord setup. Source inspection does show a plausible duplicate skill-command inflation path: duplicate skill names can be uniqued and logged before the final by-skill-name dedupe removes them from the returned list.

## Solution Assessment

No current fix is evident. The narrow maintainable direction is to avoid building/registering duplicate native skill command specs before the Discord startup path reaches client/gateway lifecycle work, not to add a new config workaround.
