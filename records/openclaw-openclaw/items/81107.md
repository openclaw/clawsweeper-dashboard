## Summary

Keep open: current main still does Discord native skill-command discovery before Discord client startup, and duplicate skill names across distinct agent workspaces can still be uniqued/logged before the final by-skill-name dedupe drops them. The older gateway-readiness race reports are useful context, but this issue has a remaining startup-load path that is not fully covered by the closed fixes.

## What This Changes

The issue reports that Discord native skill-command de-duplication loops during startup on v2026.5.7, saturating CPU and preventing Discord gateway readiness from completing.

## Best Possible Solution

Deduplicate by canonical skill identity before per-workspace slash-command uniquing/native Discord command deploy work, while preserving per-agent allowlists, shared-workspace merging, reserved command collisions, and the generic `/skill` fallback.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81107 on current main. The source bug is that Discord native skill command resolution runs before Discord client startup, and `listSkillCommandsForAgents` currently builds per-workspace command specs with a shared reserved-name set before `dedupeBySkillName` drops duplicate skill names. That means bundled/plugin skills like `discord` can still be suffix-uniqued and logged many times across distinct agent workspaces before the returned list is clean. Move or add canonical skill identity dedupe before command-name uniquing/native Discord deploy work, while preserving per-agent allowlists, same-workspace allowlist merging, reserved chat command collisions, user-invocable filtering, plugin-published skills, and the generic `/skill` fallback. Add focused regression coverage proving duplicate skill names across distinct agent workspaces do not perform pre-dedupe suffix generation/logging and that Discord command-spec resolution receives one native skill command. Do not add a config knob, change dependency versions, or change unrelated Discord gateway lifecycle behavior. Add a changelog fix entry if the repair is user-visible.
