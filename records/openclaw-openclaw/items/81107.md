## Summary

Keep open: current main still runs Discord native skill-command discovery before the Discord client/lifecycle starts, and duplicate skill names can still be uniqued/logged per workspace before the final by-skill-name dedupe removes them. The older Discord readiness and duplicate native-command reports are useful context, but they do not cover this distinct startup-load path.

## What This Changes

Fix Discord gateway startup CPU saturation caused by repeated native skill-command de-duplication when the external Discord plugin discovers duplicate skill commands.

## Best Possible Solution

Deduplicate duplicate skill identities before command-name uniquing and native Discord command preparation, while preserving per-agent allowlists, shared-workspace semantics, and the generic `/skill` fallback.

## Reproduction Assessment

No, not for the full CPU-saturation plus Discord WebSocket hang in a real Discord environment. Source inspection does show a plausible current-main path where duplicate skill names are uniqued and logged before final dedupe removes them.

## Solution Assessment

Yes, the narrow maintainable direction is to collapse duplicate skill identities before native command-name uniquing/deploy work, rather than adding a config workaround or changing Discord gateway readiness policy.
