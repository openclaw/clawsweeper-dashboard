## Summary

Keep open: current main still builds Discord native command specs by listing skills before Discord lifecycle startup, and duplicate skill names can still be unique-suffixed/logged before the final de-duplication step drops them. The broader Discord READY reports are related, but this issue adds a distinct skill-command startup-load path that is not clearly owned by the existing READY tracker.

## What This Changes

The issue reports a v2026.5.7 Discord startup hang where duplicate `discord` skill command discovery loops through suffixed names and blocks gateway readiness.

## Best Possible Solution

Move duplicate skill identity collapse ahead of per-workspace command-name uniquing/deploy for native command consumers, while preserving agent skill filters and the generic `/skill` fallback.

## Reproduction Assessment

No, not for the full live Discord CPU-saturation and WebSocket readiness hang. Source inspection does show the duplicate skill-name work can still happen before the final de-duplication step, which matches the reported log shape.

## Solution Assessment

Yes, a focused pre-generation de-dupe is the best maintainable direction. The current late de-dupe keeps the returned list clean but is too late to avoid the startup uniquing/logging work reported here.
