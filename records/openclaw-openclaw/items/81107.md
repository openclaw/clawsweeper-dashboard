## Summary

Keep open: current main still has the reported duplicate skill-command generation path on Discord startup, and the broader Discord readiness trackers do not own this specific native skill-command de-dupe workload. The issue has strong shipped-version logs plus source-reproducible current-main behavior, but the full live CPU-saturation and WebSocket hang were not reproduced in this read-only review.

## What This Changes

The issue reports Discord startup CPU saturation from repeated native skill-command de-duplication after installing the external Discord plugin on v2026.5.7.

## Best Possible Solution

Collapse duplicate skill identities before slash-command name uniquing and Discord native command generation, while preserving per-agent allowlists, per-workspace discovery, and the generic `/skill` fallback.

## Reproduction Assessment

No, not for the full live CPU-saturation plus Discord WebSocket hang on current main. Source inspection does show a clear current-main path where duplicate skill names are uniqued and logged before the final by-skillName de-dupe removes them.

## Solution Assessment

Yes, the best fix direction is a narrow skill-command discovery change that avoids duplicate command-name work before Discord startup deploys native commands. A config workaround or broader gateway lifecycle change would not directly address the duplicate generation path.
