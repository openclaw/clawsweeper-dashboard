## Summary

Keep open: this is a focused fix for a real memory-wiki tool-wrapper bug, but the current diff still uses platform-specific `path.relative`, so Windows output can violate the promised stable `reports/lint.md` tool path.

## What This Changes

The PR changes the memory-wiki `wiki_lint` agent tool to return a vault-relative report path and bounded details payload, with a focused tool regression test.

## Best Possible Solution

Keep the wrapper-only fix, normalize the returned relative report path to slash separators, preserve the lint core/CLI/gateway absolute-path contract, and land with focused memory-wiki tests.

## Reproduction Assessment

Yes for the review finding: direct Node runtime probing shows `path.win32.relative` returns `reports\lint.md`, so the PR's current tool output is platform-specific. For the linked runtime failure, current source confirms the raw absolute tool result shape, but I did not reproduce the full bridge-mode failure path.

## Solution Assessment

Mostly yes: changing only `createWikiLintTool` is the narrow maintainable fix because lint core and CLI callers keep their absolute report path. The implementation still needs slash normalization after `path.relative` before it is the best version of that fix.
