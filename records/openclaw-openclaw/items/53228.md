## Summary

Keep this issue open. Current main still lacks the requested shell-facing notification command, a typed admin-scoped Gateway notification RPC/event, and Control UI handling for visible connected-session notification bubbles. The linked Web Push work (#44590), broader WebUI notification tracker (#20237), sessions_broadcast PR (#47486), and Notifications settings-tab PR (#74715) are adjacent but do not supersede the requested no-agent-turn UI broadcast path.

## What This Changes

Request a first-class `openclaw notify <message>` command that broadcasts a visible notification bubble to all connected Control UI sessions without enqueueing an agent turn.

## Best Possible Solution

Define an intentional notification contract: an admin-scoped Gateway RPC plus allowlisted broadcast event, a shell entry point such as `openclaw notify <message>`, Control UI rendering for connected-session notifications, and docs/tests covering authorization, delivery semantics, and its relationship to Web Push and existing chat/system-event surfaces.

## Reproduction Assessment

Not applicable as a feature request. The current-main source/docs check is high-confidence: the relevant CLI, Gateway, broadcast, and Control UI surfaces do not contain the requested command or notification event path.

## Solution Assessment

Unclear until maintainer product/API review. The requested shape is plausible, but the maintainable solution should first settle whether this is core behavior or a new plugin/API seam and define the auth, scope, UI, and abuse-control semantics.
