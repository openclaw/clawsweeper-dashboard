## Label Justifications

- `P2`: The issue describes a real QMD memory-indexing robustness failure in a bounded broad-workspace setup rather than a whole-runtime outage.
- `impact:session-state`: A failed QMD update can leave durable memory indexing unhealthy and degrade memory context used by agent sessions.

## Summary

Keep open. Current main and v2026.6.6 still leave the QMD traversal gap unresolved: OpenClaw resolves QMD collections as plain path/pattern entries and delegates `qmd update` to the configured QMD binary, while the docs still describe hidden temp directories or paths outside indexed roots as the workaround.

## What This Changes

The issue asks for QMD-backed memory indexing to avoid failing on workspace-visible temporary monorepos with symlink loops and to keep docs aligned with runtime traversal behavior.

## Best Possible Solution

Keep this as active QMD traversal robustness work: choose the maintained boundary, then add regression coverage and either require or diagnose a safe QMD scanner version or add OpenClaw-side pre-update pruning before `qmd update` can traverse loops.

## Reproduction Assessment

Yes at source level: the reporter supplied a controlled symlink-loop scanner shape, and current OpenClaw still delegates QMD collection traversal to `qmd update`. I did not run a live end-to-end QMD binary repro in this read-only review.

## Solution Assessment

Unclear. The runtime gap is real, but a maintainer needs to choose whether OpenClaw should enforce a QMD version/diagnostic, add pre-update pruning/exclusion, or coordinate upstream before implementation.
