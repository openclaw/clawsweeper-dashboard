## Summary

Keep open. The original QMD symlink-loop failure may now be mitigated by current upstream QMD scanner defaults, but OpenClaw still passes plain path/pattern collections to QMD, still documents older-QMD compatibility, and has not landed a version gate, OpenClaw-side traversal guard, or regression test that settles the reported workspace temp-monorepo failure.

## What This Changes

Ensure QMD-backed memory indexing cannot recurse through symlinked workspace-visible temporary monorepos and break indexing with ENAMETOOLONG.

## Best Possible Solution

Decide the supported QMD scanner contract, add the controlled symlink-loop regression case, then either require/probe a safe QMD version or add OpenClaw-owned pre-update pruning/ignore wiring before relaxing the workaround docs.

## Reproduction Assessment

Yes, at source level with a caveat: the issue comments provide a concrete fixture and current OpenClaw still delegates plain QMD collections to qmd update. I did not run a live QMD update, and current upstream QMD source appears to use no-follow scanner defaults.

## Solution Assessment

Unclear. The report is valid, but maintainers need to choose whether the durable fix is a supported QMD version requirement/probe, OpenClaw pre-update pruning, QMD ignore wiring, or a combined contract.
