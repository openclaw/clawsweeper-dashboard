## Summary

Keep open: current OpenClaw main and v2026.6.8 still document this QMD symlink-loop case as a live limitation, while the code has no OpenClaw-side traversal/version contract that would make the issue cleanup-closeable. Current upstream QMD appears safer now, so the remaining work is a maintainer decision about supported QMD versions, docs, and whether OpenClaw needs its own guard.

## What This Changes

The issue asks for QMD memory indexing to be robust against symlink-looping temporary monorepos and for docs/runtime behavior to agree on symlink traversal.

## Best Possible Solution

Decide the supported QMD traversal contract, then either update docs/doctor/version guidance for the current safe QMD behavior or add an OpenClaw-side pre-update guard with focused symlink-loop regression coverage.

## Reproduction Assessment

Partly. The issue discussion provides a controlled scanner repro and OpenClaw still has no min-version or traversal gate, but the currently published QMD package now appears to set `followSymbolicLinks: false`; a latest-QMD live repro was not established.

## Solution Assessment

No, cleanup-closing or auto-fixing is not the best path yet. The narrowest maintainable solution needs maintainer agreement on whether OpenClaw should rely on current QMD behavior or own a local guard/docs/version contract.
