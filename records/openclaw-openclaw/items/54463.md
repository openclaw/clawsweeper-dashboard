## Summary

Keep this open. The docs-only part was corrected, but current main still delegates QMD collection traversal to the installed `qmd update` path while only the builtin memory backend has OpenClaw-side symlink skipping; the remaining fix needs a maintainer decision on QMD version enforcement versus OpenClaw-side pruning or exclusions.

## What This Changes

The issue reports that QMD-backed memory indexing can follow symlink loops in workspace-visible temporary monorepos and fail with `ENAMETOOLONG`, making memory indexing and retrieval unhealthy.

## Best Possible Solution

Keep this as active QMD traversal robustness work: prove the controlled symlink-loop fixture against the QMD version OpenClaw actually runs, then either require/probe a safe QMD traversal contract or add OpenClaw-owned pre-update pruning before broad workspace collections update.

## Reproduction Assessment

Partially. The reporter supplied a concrete source-level symlink-loop scanner shape and current OpenClaw still delegates QMD collection traversal to `qmd update`; I did not run a live QMD repro here, and latest upstream QMD suggests behavior may depend on the installed QMD version.

## Solution Assessment

Unclear. A docs-only workaround is not the best fix, but the final repair point still needs maintainer choice between relying on a fixed QMD version, adding OpenClaw-side pre-update pruning or exclusions, or combining both.
