## Summary

Keep open. The docs mismatch was already corrected, but current main still documents this QMD symlink-loop case as a live runtime limitation and the QMD manager still delegates plain path/pattern collections to `qmd update` without an OpenClaw-side no-follow, exclude, or realpath-cycle guard.

## What This Changes

The issue asks OpenClaw to prevent QMD-backed memory indexing from recursing through workspace-visible symlink-loop temp monorepos until it fails with ENAMETOOLONG.

## Best Possible Solution

Add the controlled symlink-loop temp-monorepo repro as regression coverage, then make QMD updates traversal-safe through a no-follow/realpath-cycle guard, a safe QMD version contract, or an explicitly chosen exclusion layer, with docs updated when runtime behavior changes.

## Reproduction Assessment

Yes. The comments provide a concrete minimal workspace shape and direct scanner observation, and current main still registers plain path/pattern collections before running `qmd update`; I did not run a live QMD scan, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. A traversal-safe QMD update path is the right end state, but maintainers still need to choose whether the narrow fix belongs upstream in QMD, in an OpenClaw pre-update guard, or in an exclusion/version contract.
