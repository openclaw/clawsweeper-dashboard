## Summary

Keep this open. Current main still documents the QMD symlink-loop case as a live limitation, and the source still hands plain path/pattern collections to `qmd update` without an OpenClaw pre-traversal guard or a proven safe QMD scanner contract.

## What This Changes

The issue asks for QMD memory indexing to stay healthy when broad workspace indexing sees temporary monorepos with recursive symlink structures, and for docs/runtime symlink behavior to agree.

## Best Possible Solution

Keep this as an active QMD traversal robustness bug, add the controlled symlink-loop fixture as regression proof, then require a safe QMD scanner contract or add an OpenClaw-side pre-update no-follow/cycle guard before relaxing the docs workaround.

## Reproduction Assessment

Yes, by source-level review rather than a live run. The issue comments give a controlled temp-monorepo symlink-loop fixture, and current main still maps QMD paths and extraPaths into plain QMD collections before running `qmd update`.

## Solution Assessment

Unclear until maintainers choose the boundary. A maintainable fix should either rely on a proven safe QMD scanner version or add OpenClaw-side pre-update cycle-safe traversal/exclusion, with docs updated to match the chosen runtime contract.
