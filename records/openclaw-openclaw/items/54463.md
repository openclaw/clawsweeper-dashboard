## Summary

Keep open. The docs-only mismatch was corrected, but current main still treats QMD collections as plain path/pattern inputs and delegates traversal to the installed `qmd update` without an OpenClaw-side no-follow, ignore, version-gate, or cycle guard for this temp-monorepo symlink-loop case.

## What This Changes

The issue requests QMD memory indexing to avoid `ENAMETOOLONG` when workspace-visible temporary monorepos contain symlink loops, and to keep symlink docs aligned with runtime behavior.

## Best Possible Solution

Keep this as an active QMD robustness item: add the controlled symlink-loop regression case, then align OpenClaw with a cycle-safe QMD contract through version detection, QMD ignore metadata, or an OpenClaw-owned pre-update guard before updating the workaround docs.

## Reproduction Assessment

Yes, with one caveat. The issue comments provide a concrete temp-monorepo symlink-loop shape and current main still delegates QMD traversal to the installed `qmd update`; I did not establish a live end-to-end failure with the latest QMD package.

## Solution Assessment

Unclear. A directory-exclusion feature alone is not enough unless pruning happens before QMD traversal, and maintainers need to choose between a QMD version/ignore contract and an OpenClaw-owned guard.
