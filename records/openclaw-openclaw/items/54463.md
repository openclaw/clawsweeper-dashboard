## Summary

Keep this issue open: current main still has no landed QMD traversal guard or exclusion contract for symlink-looping workspace temp repos, and maintainer discussion explicitly kept the runtime problem open after the docs-only correction.

## What This Changes

The issue asks OpenClaw to prevent QMD memory indexing from recursing through workspace-visible monorepo symlink loops and failing with ENAMETOOLONG while keeping docs and runtime behavior aligned.

## Best Possible Solution

Add a controlled symlink-loop regression case and land a QMD-safe traversal contract, either by making upstream QMD traversal cycle-safe/no-follow for OpenClaw indexing inputs or by pruning/excluding loop-prone paths before qmd update runs, then update docs from workaround to shipped behavior.

## Reproduction Assessment

Yes, by source-level review with a caveat. The issue comments provide a concrete temp-monorepo symlink-loop fixture, and current main still delegates plain path/pattern collections to qmd update without an OpenClaw traversal guard; I did not run a live QMD update.

## Solution Assessment

Unclear. The bug is real, but maintainers still need to choose the durable boundary between upstream QMD traversal behavior, OpenClaw-side pruning/exclusions, and any user-facing config contract.
