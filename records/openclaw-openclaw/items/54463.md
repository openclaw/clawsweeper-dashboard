## Summary

Keep this open, but narrow the next step. Current OpenClaw still documents the QMD symlink-loop case as a live limitation and has no OpenClaw-side traversal guard or QMD minimum-version check, while the latest inspected QMD package appears to have moved to no-follow scanning; maintainers should verify the versioned runtime behavior before closing or choosing an OpenClaw-side fix.

## What This Changes

The issue asks OpenClaw to prevent QMD memory indexing from recursing through symlinked workspace-visible temp monorepos and to align runtime and docs on symlink behavior.

## Best Possible Solution

Verify a real QMD-versioned symlink-loop repro against current OpenClaw; if upstream QMD is fixed, update OpenClaw docs and doctor/min-version guidance, otherwise add a cycle-safe/no-follow or pre-update exclusion guard at the QMD collection boundary.

## Reproduction Assessment

No high-confidence current-main plus current-QMD reproduction is established in this review. The historical report and maintainer comments are credible, but latest @tobilu/qmd source now appears to scan with symlink following disabled, so the next proof needs to be versioned.

## Solution Assessment

Unclear. A custom OpenClaw traversal layer may be unnecessary if supported QMD versions already fixed scanning, but OpenClaw still needs a maintainer decision on min-version/docs/doctor guidance versus an OpenClaw-side guard.
