## Summary

Keep this open. Current main still treats the reported QMD symlink-loop case as a live runtime limitation: OpenClaw creates plain QMD path/pattern collections and runs `qmd update` without a no-follow, exclusion, or realpath-cycle guard that would prevent the reported `ENAMETOOLONG` failure mode.

## What This Changes

The issue asks OpenClaw to prevent QMD memory indexing from recursing through workspace-visible symlink-loop temp monorepos and failing with `ENAMETOOLONG`, or otherwise align docs and config with the actual traversal contract.

## Best Possible Solution

Keep this as an active QMD traversal robustness item. Add the controlled symlink-loop temp-monorepo repro as regression coverage, then make traversal cycle-safe before QMD can expand looped paths, either by requiring/detecting a safe QMD scanner or by adding an OpenClaw-side pre-update no-follow/exclusion/realpath-cycle guard; update the QMD docs once runtime behavior is fixed.

## Reproduction Assessment

Yes. The issue comments give a concrete minimal workspace shape and direct scanner output showing repeated symlink-expanded Markdown paths; current main still routes plain QMD collections into `qmd update` without an OpenClaw traversal guard.

## Solution Assessment

Unclear. There is no merged patch to assess, and the related open #40438 exclusion work is adjacent rather than a proven QMD symlink-loop fix unless maintainers decide that OpenClaw should own the QMD exclusion layer.
