## Summary

Keep this open. Current main still delegates QMD indexing to the external `qmd update` traversal path, the docs still describe workspace-visible temp repos as a live limitation, and maintainer discussion explicitly kept the runtime traversal/exclusion issue open after the docs-only correction.

## What This Changes

The issue asks OpenClaw to prevent QMD-backed memory indexing from failing when broad workspace indexing reaches symlink-looping temporary monorepos.

## Best Possible Solution

Define the QMD indexing contract in one place, either by enforcing a safe QMD version/doctor check or by passing a supported exclusion/cycle-safe traversal policy through QMD collections, then keep the docs aligned with that behavior.

## Reproduction Assessment

Yes at source level: the reporter provided a controlled scanner shape where symlink following repeatedly yields longer aliases before OpenClaw filters can help. I did not establish an end-to-end current-main failure with a pinned affected QMD binary, so this is not marked as fully reproduced.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. A blind OpenClaw filesystem crawl is probably not the best fix if the supported QMD contract can be enforced or configured directly.
