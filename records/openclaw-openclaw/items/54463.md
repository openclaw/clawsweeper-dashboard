## Summary

Keep open. The stale docs mismatch was corrected, but current main still has no OpenClaw-side QMD traversal guard, no resolved exclusion/no-follow metadata, and no realpath-cycle handling before delegating `qmd update`; the related open pull request is adjacent but is not a landed or sufficient canonical fix.

## What This Changes

The issue asks OpenClaw to keep QMD memory indexing from failing on workspace-visible temporary monorepos with recursive symlink/package-manager structures.

## Best Possible Solution

Add the controlled symlink-loop regression case, then make QMD traversal cycle-safe through a chosen upstream QMD contract or an OpenClaw-owned pre-update guard before relaxing the workaround docs.

## Reproduction Assessment

Yes, with a caveat. The issue comments give a concrete temp-monorepo symlink-loop fixture, and current main still delegates plain path/pattern collections to `qmd update`; I did not run a live QMD update locally.

## Solution Assessment
