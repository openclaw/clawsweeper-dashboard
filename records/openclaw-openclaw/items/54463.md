## Summary

Keep open. The docs-only mismatch was corrected, but current main still delegates QMD indexing to plain path/pattern collections and `qmd update` without a no-follow, exclusion, realpath-cycle, or safe-QMD-version guard for the reported symlink-loop temp-monorepo case.

## What This Changes

Prevent QMD-backed memory indexing from recursing through symlink-loop temporary monorepos and keep the docs/runtime symlink contract aligned.

## Best Possible Solution

Add the controlled symlink-loop temp-monorepo repro as regression coverage, then either require/detect a cycle-safe QMD scanner or add an OpenClaw-owned pre-update no-follow/exclusion/realpath-cycle guard before `qmd update`; update QMD docs when runtime behavior is fixed.

Yes. The comments provide a concrete minimal workspace shape and direct scanner observation, and current main still passes plain path/pattern collections to `qmd update`; I did not run a live QMD/Bun scan in this read-only review, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. There is no merged patch, and maintainers still need to choose whether the durable fix belongs upstream in QMD, in an OpenClaw pre-update guard, or in a narrower exclusion/version contract; #40438 is not enough by itself unless that boundary is explicitly chosen.
