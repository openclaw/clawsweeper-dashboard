## Summary

Keep open. Current main and the latest release still treat the QMD symlink-loop case as unresolved, while OpenClaw only passes plain path/pattern collections to external QMD; upstream QMD now appears safer, but OpenClaw has not validated or enforced that as the product contract.

## What This Changes

QMD-backed memory indexing can fail when broad workspace indexing reaches a symlink-looping temporary monorepo, producing repeated paths or ENAMETOOLONG.

## Best Possible Solution

Keep #54463 as the canonical QMD traversal robustness item until maintainers verify the supported QMD version and land one owner-approved runtime/config/docs/test change that makes broad QMD indexing cycle-safe or declares the enforced QMD contract.

## Reproduction Assessment

Yes at source/discussion level, but not as a fresh live current-main run. The issue includes a controlled symlink-loop scanner repro, and current OpenClaw still delegates QMD update traversal to external QMD.

## Solution Assessment

Unclear. Current upstream QMD v2.5.3 appears to avoid symlink following and supports ignore patterns, but OpenClaw still needs a maintainer choice on version floor, ignore plumbing, or OpenClaw-side pruning.
