## Summary

Keep open. Current main still documents the QMD symlink-loop case as a live runtime limitation, resolves QMD inputs to plain path/pattern collections, and delegates `qmd update` without an OpenClaw-side pre-traversal guard; the adjacent exclusion PR is still open and not a landed canonical fix.

## What This Changes

The issue asks OpenClaw to keep QMD-backed memory indexing from failing on workspace-visible temporary monorepos with symlink cycles, and to keep runtime and docs aligned on symlink handling.

## Best Possible Solution

Add the controlled symlink-loop regression case, then make QMD traversal cycle-safe through the chosen upstream QMD contract or an OpenClaw-owned pre-update pruning/exclusion layer before relaxing the workaround docs.

## Reproduction Assessment

Yes, by source-level review with a caveat. The issue comments provide a concrete temp-monorepo symlink-loop fixture, and current main still maps extra paths to plain QMD collections before delegating `qmd update`; I did not run a live QMD update.

## Solution Assessment

Unclear. The bug is valid, but maintainers still need to choose whether the durable fix belongs in upstream QMD traversal/versioning, OpenClaw pre-update pruning, explicit exclusions, or a combined contract.
