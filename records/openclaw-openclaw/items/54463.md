## Summary

Keep this open. Current main still documents the workspace-visible temp repo symlink-loop case as an active QMD runtime limitation, and the implementation still resolves plain QMD path/pattern collections before invoking `qmd update` without an OpenClaw-side no-follow, exclusion, safe-version, or realpath-cycle contract. The maintainer discussion also explicitly kept the runtime investigation open after the docs-only part was corrected.

## What This Changes

The issue requests that QMD-backed memory indexing avoid recursing through symlink loops in workspace-visible temporary monorepos, preventing `ENAMETOOLONG` and aligning docs with runtime behavior.

## Best Possible Solution

Keep this as an active QMD traversal robustness item. Add the controlled symlink-loop repro as regression coverage, then make the runtime contract explicit by requiring/detecting a safe QMD scanner or adding an OpenClaw-side pre-update pruning/exclusion guard before `qmd update` can recurse through workspace-visible temp monorepos; update the QMD docs once that runtime behavior ships.
