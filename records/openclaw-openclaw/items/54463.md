## Summary

Keep open. Current main still documents this as a live QMD traversal limitation, and the QMD manager still hands plain path/pattern collections to `qmd collection add` and `qmd update` without a no-follow, realpath-cycle, exclusion, or safe-QMD-version guard for the temp-monorepo symlink-loop case.

## What This Changes

Prevent QMD memory indexing from recursing through symlink-looped workspace-visible temp monorepos and align the runtime behavior with the documented symlink contract.

## Best Possible Solution

Add the controlled symlink-loop temp-monorepo repro as regression coverage, then make QMD traversal cycle-safe/no-follow upstream or add an OpenClaw-owned pre-update pruning/exclusion guard before `qmd update`; keep the current docs workaround until the runtime fix ships.

## Reproduction Assessment

Yes. The issue comments provide an exact minimal workspace shape and direct scanner observation, and current source still delegates plain path/pattern collections to QMD update; I did not run a live QMD/Bun scan, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Unclear. The report's acceptable directions are reasonable, but maintainers still need to choose whether the durable fix belongs upstream in QMD traversal, in an OpenClaw pre-update guard/exclusion layer, or in a narrower version/config contract.
