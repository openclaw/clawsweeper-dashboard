## Summary

Keep this open. The docs-only mismatch was corrected, but current main still treats the QMD symlink-loop case as a live runtime limitation and still delegates indexing to `qmd update` with plain path/pattern collections; I found no OpenClaw-side cycle guard, no QMD version/safety gate, no configurable no-follow/exclusion contract, and no regression test covering the reported temp-monorepo symlink loop.

## What This Changes

The issue asks for QMD-backed memory indexing to avoid failing on workspace-visible temporary monorepos with symlink loops that can expand paths until ENAMETOOLONG.

## Best Possible Solution

Keep this as an active QMD traversal robustness item: add the controlled symlink-loop temp-monorepo reproduction as regression coverage, then make indexing cycle-safe through a supported QMD scanner/exclusion contract or an OpenClaw pre-update pruning layer, with docs updated once runtime behavior is actually safe.

## Reproduction Assessment

Yes. The issue comments provide a concrete workspace shape with `tmp/openclaw-pr/packages/moltbot/node_modules/openclaw -> ../../..` and repeated symlink-expanded Markdown paths, and current main still has no guard before `qmd update` traverses configured collections.

## Solution Assessment

Unclear. There is no proposed patch in this item, and the open #40438 excludeDirs work is adjacent but not a complete QMD update traversal fix unless maintainers decide that OpenClaw should own a QMD collection exclusion layer.
