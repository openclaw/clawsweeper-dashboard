## Summary

Keep open. Current main still delegates QMD collection traversal to the external `qmd update` path without an OpenClaw-side pre-traversal symlink/cycle guard or ignore-pattern contract, and the maintainer discussion explicitly narrowed this to a remaining runtime/upstream fix rather than docs cleanup.

## What This Changes

The issue asks OpenClaw to prevent QMD-backed memory indexing from breaking on workspace-visible temp monorepos with recursive symlink structures and to keep docs/runtime behavior aligned.

## Best Possible Solution

Reproduce against the supported QMD version, then harden the QMD collection path with one owner-approved approach that keeps runtime, docs, and any config contract aligned.

## Reproduction Assessment

Yes, source-level evidence is strong but not a fresh local current-main runtime repro: the issue discussion includes a controlled scanner reproduction, and current OpenClaw still delegates QMD update traversal to the QMD subprocess.

## Solution Assessment

Unclear. The best fix depends on whether maintainers want OpenClaw to require the fixed QMD scanner behavior, pass QMD ignore patterns, or add its own collection/exclusion hardening before QMD traversal.
