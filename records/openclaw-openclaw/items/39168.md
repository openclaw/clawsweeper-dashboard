## Summary

Keep open. The reported `mcp_exec`/`mcp_read` alias problem still appears valid on current `main`, but this PR is not merge-ready: it patches an obsolete dependency version, omits lockfile patch metadata, and would mishandle legitimate `mcp__...` MCP-style tool names if applied to the active dispatch path.

## What This Changes

The PR adds a root pnpm patchedDependency and compiled patch for `@mariozechner/pi-agent-core@0.55.3` that strips a leading `mcp_` from `toolCall.name` before dependency-level tool lookup.

## Best Possible Solution

Implement this in the existing embedded-runner tool-call normalizer on current `main`: exact registered names should win first, one leading `mcp_` alias should be stripped only when the suffix is non-empty, not `mcp__...`, and resolves unambiguously to an allowed canonical tool; keep dependency patching out of this fix.

## Reproduction Assessment

Yes. The PR gives live reproduction steps and current source inspection gives a focused unit path: a streamed `toolCall` named `mcp_exec` with allowed tools containing only `exec` remains `mcp_exec` through the current normalizer and reaches exact dependency lookup unchanged.

## Solution Assessment

No. A compiled pnpm patch against an obsolete `pi-agent-core` version is not the narrowest maintainable fix; the safer path is a guarded OpenClaw-side normalizer change with exact-first matching and regression coverage for both `mcp_` aliases and legitimate `mcp__...` names.
