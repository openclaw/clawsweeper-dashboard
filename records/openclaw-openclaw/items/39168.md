## Summary

Keep open: the reported `mcp_exec` dispatch bug still appears source-reproducible on current main, but this PR is not a correct merge shape because it patches a stale compiled dependency version, omits lockfile patch metadata, and rewrites exact `mcp__...` tool names too broadly.

## What This Changes

The PR adds a pnpm patch for `@mariozechner/pi-agent-core@0.55.3` that strips a leading `mcp_` prefix from `toolCall.name` before dependency-level tool lookup.

## Best Possible Solution

Extend OpenClaw's first-party embedded-runner normalizer to try exact allowed names first, then map non-empty single-underscore `mcp_` aliases only when the suffix uniquely matches an allowed tool, with regression tests.

## Reproduction Assessment

Yes. A source reproduction is a streamed `toolCall` named `mcp_exec` with `allowedToolNames` containing `exec`; current main's normalizer has no single-underscore `mcp_` alias branch, so the raw name remains unresolved before exact dispatch.

## Solution Assessment

No. Patching compiled `pi-agent-core@0.55.3` is stale and overbroad; the safer fix belongs in OpenClaw's existing first-party dispatch normalizer with exact-name precedence and regression coverage.
