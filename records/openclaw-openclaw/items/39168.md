## Summary

The underlying `mcp_exec`/`mcp_read` dispatch problem still looks plausible on current main, but this PR should not merge as written because it patches a stale dependency package, omits reproducible patch metadata, and rewrites valid `mcp__...` tool names too broadly.

## What This Changes

The PR adds a root pnpm patched dependency for `@mariozechner/pi-agent-core@0.55.3` plus a patch file that strips a leading `mcp_` before dependency-level tool lookup.

## Best Possible Solution

Add a guarded first-party embedded-runner normalizer that preserves exact registered names first, then maps a non-empty single-underscore `mcp_` suffix only when that suffix uniquely matches an allowed tool, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a tool call named `mcp_exec` with `allowedToolNames` containing `exec` is not normalized by current main before exact dependency dispatch, though I did not run a live repro in this read-only review.

## Solution Assessment

No. Patching a stale compiled dependency is not the best fix; the narrow maintainable solution belongs in OpenClaw's existing stream-normalization seam with exact-name precedence.
