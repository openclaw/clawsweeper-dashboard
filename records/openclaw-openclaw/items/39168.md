## Summary

Keep open: the reported `mcp_exec` to `exec` dispatch gap still appears source-reproducible on current main, but this PR is not a safe merge shape because it patches a stale dependency version, lacks lockfile patch metadata, rewrites valid `mcp__...` tool names, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a root pnpm patch for `@mariozechner/pi-agent-core@0.55.3` that strips a leading `mcp_` from tool-call names before dependency-level lookup.

## Best Possible Solution

Add a guarded first-party embedded-runner normalizer: exact registered names first, then map a non-empty single-underscore `mcp_` suffix only when that suffix uniquely matches an allowed tool, with regression tests and no compiled dependency patch.

## Reproduction Assessment

Yes. A source reproduction is a streamed or final tool call named `mcp_exec` with `allowedToolNames` containing `exec`; current main has no single-underscore `mcp_` branch, so the raw name remains unresolved before exact dependency dispatch.

## Solution Assessment

No. Patching compiled `pi-agent-core@0.55.3` is stale and overbroad; the narrower maintainable solution is in OpenClaw's existing stream normalization seam with exact-name precedence and focused regression coverage.
