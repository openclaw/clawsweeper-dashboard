## Summary

Keep open. The underlying `mcp_` alias bug still appears valid on current `main`, but this PR is not merge-ready because it patches an obsolete `pi-agent-core` version, omits required lockfile patch metadata, and can corrupt legitimate `mcp__...` tool names.

## What This Changes

The PR adds a root pnpm patchedDependency and compiled patch for `@mariozechner/pi-agent-core@0.55.3` that strips a leading `mcp_` from dependency-level tool-call lookup names.

## Best Possible Solution

Implement the fix in OpenClaw's existing embedded-runner tool-call normalizer: exact registered names should win first, legitimate `mcp__...` names should be preserved, and one leading `mcp_` alias should be stripped only when the suffix is non-empty and resolves unambiguously to an allowed canonical tool. Keep dependency patching out of this change and add focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A focused reproduction is a streamed `toolCall` named `mcp_exec` with allowed tools containing `exec`; current main leaves the name as `mcp_exec`, and upstream pi-agent-core then performs exact lookup. The PR body also provides live Feishu/macOS reproduction notes for the same `Tool mcp_exec not found` failure.

## Solution Assessment

No. The submitted dependency patch is not the best solution because it targets a stale package version, lacks lockfile metadata, and over-strips valid MCP-style names. The safer fix is a guarded OpenClaw-side normalizer change in the existing runner seam.
