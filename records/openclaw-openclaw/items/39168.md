## Summary

Keep open: the reported `mcp_exec` dispatch failure is still source-reproducible on current main, but this PR does not fix the active runtime path and lacks inspectable real behavior proof. The submitted patch targets an obsolete `@mariozechner/pi-agent-core@0.55.3` package while current main uses `@earendil-works/pi-agent-core@0.74.1`, and its blanket prefix rewrite would also break valid exact `mcp__...` tool names.

## What This Changes

The PR adds package-level pnpm patch metadata and a compiled patch for `@mariozechner/pi-agent-core@0.55.3` to strip a leading `mcp_` prefix before tool lookup.

## Best Possible Solution

Implement guarded alias normalization in `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts`, preserving exact allowed names first, covering `mcp_exec`/`mcp_read`/`mcp_write` to canonical names, and adding focused regression tests for both alias recovery and exact `mcp__...` preservation.

## Reproduction Assessment

Yes, source-reproducible. Current main does not map single-underscore `mcp_exec` to allowed `exec`, and the active pi-agent-core dependency exact-matches `toolCall.name` before returning `Tool mcp_exec not found`.

## Solution Assessment

No. Patching obsolete compiled dependency code is not the best fix; the narrow maintainable solution is first-party stream normalization with exact-name precedence and regression coverage.
