## Summary

Keep open: the reported `mcp_exec` dispatch failure is source-reproducible and not clearly implemented on current `main`, but this PR should not merge as written because it patches a stale dependency/version, lacks reproducible lockfile patch metadata, and applies an overbroad prefix rewrite in compiled dependency code.

## What This Changes

Adds a root pnpm patched dependency and patch file for `@mariozechner/pi-agent-core@0.55.3` that strips a leading `mcp_` from tool-call names before dependency-level lookup.

## Best Possible Solution

Add a guarded first-party normalizer in the embedded-runner stream normalization path that preserves exact registered names first, then maps a non-empty single-underscore `mcp_` suffix only when it uniquely matches an allowed canonical tool, with focused regression coverage.

Yes, source inspection gives a high-confidence path: `mcp_exec` with allowed `exec` is not resolved by the current exact, ID-inference, or structured dot/slash suffix normalization before dependency dispatch. I did not run a live provider repro in this read-only review.

## Solution Assessment
