## Summary

Close: current main and the latest shipped tag treat this as expected Gateway security behavior, not a core bug. Device-less custom WebSocket clients do not receive operator scopes from trusted-proxy or shared-token auth; supported paths are device identity/pairing, the exact internal loopback `gateway-client` backend path, or an HTTP/admin surface designed for host automation.

## What This Changes

The issue asks Gateway WebSocket backend/trusted-proxy authentication to grant operator scopes from a valid master token or trusted proxy session so `models.list` succeeds.

## Best Possible Solution

Keep the current safe default: custom WebSocket clients should bind scopes through device identity/pairing or use a documented HTTP/admin automation surface rather than receiving full scopes from a device-less trusted-proxy/backend handshake.

## Reproduction Assessment

Yes from source inspection: the current handshake defaults missing scopes to `[]` and clears unbound scopes for device-less trusted-proxy/shared-secret clients outside the narrow supported exceptions. I did not run a live Java/Docker repro because the source and shipped tag already match the reported outcome.

## Solution Assessment

No, granting full operator scopes to arbitrary device-less backend/trusted-proxy WebSocket clients is not the best fix because it weakens the documented device-binding boundary. The maintainable solution is to use device identity/pairing, the reserved `gateway-client` direct-loopback helper path, or an HTTP/admin surface intended for host automation.
