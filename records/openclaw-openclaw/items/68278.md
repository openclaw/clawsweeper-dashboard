## Summary

Keep this PR open: the intended Discord account-routing hardening is valid, but the current patch still applies strict token resolution too broadly and lacks required real Discord behavior proof.

## What This Changes

The PR makes explicit non-default Discord account token resolution return `source: none`, threads an `explicit`/enumeration distinction through some account callers, and adds Discord token/account regression tests.

## Best Possible Solution

Make strict token resolution opt-in from outbound or cron delivery boundaries, preserve inherited channel-token behavior for config/status/setup/enumeration paths, then add focused regression coverage and real Discord proof.

## Reproduction Assessment

Yes, by source inspection. Current main falls back to the channel token for a named Discord account without its own token, while the PR-head resolver returns `none` for supplied non-default account ids and leaves generic config/setup callers on that strict default.

## Solution Assessment

No. The strict explicit-send guard is the right direction, but putting strictness in the generic token/account resolver default is broader than the reported cron-delivery boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep config account resolution implicit:** `extensions/discord/src/accounts.ts:110-111`
  - body: `tokenExplicit` only opts out callers that pass `enumerateAccounts`, but Discord still exposes this resolver through `discordConfigAdapter.resolveAccount`. Gateway startup/status pass each account id and then check `account.token`, so a named account that inherits `channels.discord.token` can be treated as unconfigured; keep this path implicit and apply strictness only at send/cron boundaries.
  - confidence: 0.92
- **[P2] Preserve setup token lookups:** `extensions/discord/src/token.ts:70-74`
  - body: Because supplied account ids are strict unless `explicit: false` is passed, setup helpers that resolve Discord allowlist users or groups for a selected account stop using the channel-level token for named accounts without their own token. That degrades inherited-token setup from name resolution to unresolved entries; pass the implicit flag from setup/discovery callers or make strict resolution opt-in.
  - confidence: 0.84
