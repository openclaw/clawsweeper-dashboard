## Summary

Keep open: the PR targets a real Discord account-routing bug, but the current patch changes a compatibility-sensitive inherited-token fallback for explicit named accounts, lacks real Discord behavior proof, and is currently conflicting with main.

## What This Changes

The PR changes Discord token/account resolution so explicit non-default account IDs without their own token return `source: none`, while adding tests and preserving some implicit enumeration/default-account fallback paths.

## Best Possible Solution

Keep the wrong-identity protection, but make it compatibility-safe by enforcing strict account-token resolution only at explicit delivery boundaries or by adding an approved migration/docs path for changing inherited named-account behavior.

## Reproduction Assessment

Yes, by source inspection: current main falls back to `channels.discord.token` for `resolveDiscordToken(cfg, { accountId: "work" })` when `accounts.work` has no token, while the PR changes that explicit path to `source: none`. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No. The bug is real, but changing the generic token resolver is broader than the explicit cron/outbound delivery boundary and silently replaces an existing inherited-token config behavior.
