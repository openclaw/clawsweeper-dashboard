## Label Justifications

- `P1`: The item concerns Discord cron/account routing that can use the wrong bot identity, and the current patch has merge-blocking account/token compatibility risk.
- `merge-risk: 🚨 compatibility`: The generic resolver change can break existing named Discord accounts that inherit a channel-level token.
- `merge-risk: 🚨 message-delivery`: Merging as-is can fail or suppress Discord sends for account-scoped workflows outside the intended cron boundary.
- `merge-risk: 🚨 auth-provider`: The PR changes credential selection for Discord account IDs from fallback token resolution to fail-closed behavior.

## Summary

Keep open: the Discord account-token bug is plausible and unique, but this PR still makes strict resolution too broad for generic config/setup account lookups and lacks real Discord behavior proof.

## What This Changes

The branch adds strict Discord token resolution for supplied non-default account IDs without account tokens, plus account/listing fallbacks and token/account regression tests.

## Best Possible Solution

Make strict token resolution opt-in at outbound/cron delivery boundaries, preserve channel-token inheritance for config/status/setup/enumeration paths, then add focused regression tests and redacted real Discord proof.

## Reproduction Assessment

Yes, by source inspection: current main falls back from a named Discord account without its own token to channels.discord.token, while the PR head returns none for supplied non-default account IDs. I did not run tests because this review is read-only.

## Solution Assessment

No. The strict explicit-send guard is the right direction, but applying strictness through the generic resolver is broader than the reported cron-delivery boundary.
