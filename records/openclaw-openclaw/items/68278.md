## Summary

Keep open. The PR targets a real Discord identity-routing bug that current main still has, but the proposed boundary is still too broad: it treats any `resolveDiscordAccount({ accountId })` call as an explicit user target, while framework/runtime/setup paths also pass account IDs during internal account resolution.

## What This Changes

The PR changes Discord token resolution so explicit non-default account IDs without per-account tokens return `source: "none"`, threads an explicitness flag through account resolution, and adds token/account regression tests.

## Best Possible Solution

Land a rebased Discord-owned fix that distinguishes externally targeted account IDs from internal/default account resolution, keeps explicit cron/outbound sends strict, preserves channel-token inheritance for implicit/default/runtime/setup account scans, and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A static reproduction is clear: current main returns the top-level token for `resolveDiscordToken(cfg, { accountId: "work" })` when `accounts.work` has no token, while the PR can make internal `resolveDiscordAccount({ accountId: "work" })` calls lose that inherited token even when the caller is not an explicit send target.

## Solution Assessment

No. The strict token guard is the right direction for explicit non-default sends, but presence of an `accountId` inside the generic account resolver is not a reliable caller-intent signal; the safer fix is an explicit intent option only at the outbound/delivery boundary or an equivalent separated resolver path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep internal account resolution implicit:** `extensions/discord/src/accounts.ts:67`
  - body: `callerProvidedAccountId` is true for any non-empty `params.accountId`, but the plugin framework and runtime helpers also pass account IDs while enumerating or starting configured accounts. With `channels.discord.token` plus `accounts.work` that has no token, those internal calls now resolve `work` with `tokenSource: "none"` instead of inheriting the channel token, so a valid named-account config can lose its runtime token. Thread explicit send intent from the outbound/cron path instead of treating every account resolver call as user-targeted.
  - confidence: 0.9
