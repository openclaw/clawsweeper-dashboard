## Summary

Keep this PR open because the Discord identity-routing bug is plausible and not fixed on current main, but the current patch still treats internal account-resolution calls as explicit send intent and needs revision before merge.

## What This Changes

The PR changes Discord token resolution so explicit non-default account IDs without per-account tokens return no token instead of inheriting the top-level Discord token, with token/account regression tests.

## Best Possible Solution

Land a rebased Discord fix that passes strict account-token intent only from outbound/cron delivery boundaries, preserves channel-token inheritance for internal runtime/setup discovery, and keeps the current SecretRef tokenStatus contract.

## Reproduction Assessment

Yes. Source inspection shows current main falls back from a named Discord account without its own token to channels.discord.token, and the PR's resolver-level explicitness makes internal account scans lose that inherited token when accountId is passed internally.

## Solution Assessment

No. The strict explicit-send guard is the right direction, but deriving intent inside the generic account resolver is too broad; the safer solution is a dedicated explicit outbound/delivery option or separated resolver path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep internal account scans implicit:** `extensions/discord/src/accounts.ts:67`
  - body: `callerProvidedAccountId` treats every non-empty `params.accountId` as explicit send intent, but internal paths such as account enumeration also pass account IDs. With `channels.discord.token` plus `accounts.work` without a token, those internal calls now resolve `work` with `tokenSource: "none"` instead of inheriting the valid channel token, so runtime/setup discovery can drop configured accounts. Pass strict intent only from outbound/cron send boundaries.
