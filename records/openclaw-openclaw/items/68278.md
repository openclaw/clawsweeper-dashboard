## Summary

Keep open: the latest PR head is still not mergeable because strict Discord accountId token resolution remains the default for config/gateway and setup paths, and the external PR still lacks real Discord after-fix proof.

## What This Changes

The PR changes Discord token resolution so explicit non-default account targets without their own token do not inherit the channel-level token, with regression tests for explicit and implicit paths.

## Best Possible Solution

Preserve inherited channel-token behavior in non-send resolution paths, and pass strict token intent only from outbound or cron delivery boundaries with focused regression coverage.

## Reproduction Assessment

Yes, by source inspection against current main and the PR head. Current main falls back from a named Discord account without its own token to `channels.discord.token`, while PR-head gateway/config and setup token lookups still pass an accountId through the strict default.

## Solution Assessment

No. The strict explicit-send guard is the right direction, but making supplied accountId strict by default inside the generic token/account resolver is too broad for gateway/config/setup callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep config/startup account resolution implicit:** `extensions/discord/src/accounts.ts:110-111`
  - body: `tokenExplicit` only opts out callers that pass `enumerateAccounts`, but Discord still exposes this resolver through `discordConfigAdapter.resolveAccount`. Gateway startup/status pass each account id and then check `account.token`, so a named default account that inherits `channels.discord.token` can be treated as unconfigured instead of starting; keep this path implicit and apply strictness only at send/cron boundaries.
  - confidence: 0.92
- **[P2] Preserve setup token lookups:** `extensions/discord/src/token.ts:72`
  - body: Because any supplied accountId is strict unless `explicit: false` is passed, setup helpers that resolve Discord allowlist users or groups for a selected account stop using the channel-level token for named accounts without their own token. That degrades inherited-token setup from name resolution to unresolved/numeric-only entries; pass the implicit flag from setup/discovery callers or make strict resolution opt-in.
  - confidence: 0.84
