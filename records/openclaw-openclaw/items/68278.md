## Summary

Keep open: the branch addresses the earlier account-scan blocker and restores the shared token-resolution type shape, but the shared resolver still makes setup/direct non-outbound account token lookups strict by default, and the PR still lacks real Discord after-fix proof.

## What This Changes

The PR adds an explicit flag to Discord token resolution, treats direct non-default accountId lookups as strict, preserves enumeration fallback, and adds Discord token/account regression tests.

## Best Possible Solution

Land a rebased Discord fix that passes strict explicit-account intent only from outbound/cron send boundaries, keeps runtime/setup/discovery token inheritance implicit where appropriate, and adds setup regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main falls back from a named Discord account without its own token to channels.discord.token, and the PR diff shows direct setup token lookups would become strict unless callers pass explicit:false.

## Solution Assessment

No. The explicit option is a reasonable mechanism, but defaulting all direct non-default account token lookups to strict in the shared resolver is too broad; strict intent should be passed from outbound/cron delivery boundaries or through a separate send resolver.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Preserve setup token inheritance:** `extensions/discord/src/token.ts:93-95`
  - body: The new guard makes every direct non-default resolveDiscordToken(cfg, { accountId }) strict unless the caller opts out. Keep strict intent on outbound/cron sends, or opt setup/internal callers into implicit fallback.
