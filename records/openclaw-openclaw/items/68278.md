## Summary

Keep open: the Discord account-routing bug is valid and current main still falls back to the channel token, but this PR keeps strict token resolution in generic config/status/setup paths and still lacks real Discord behavior proof.

## What This Changes

The PR changes Discord token/account resolution so explicitly targeted non-default accounts without their own token return `source: none`, with regression tests and enumeration fallbacks.

## Best Possible Solution

Make strict Discord token resolution opt-in from outbound/cron send boundaries, preserve inherited channel-token behavior for config/status/setup/enumeration paths, then add focused regression coverage and redacted real Discord proof.

## Reproduction Assessment

Yes, by source inspection. Current main falls back to the channel token for a named Discord account without its own token, while the PR-head resolver returns `none` for supplied non-default account ids and still affects generic config/setup callers.

## Solution Assessment

No. The strict explicit-send guard is the right direction, but keeping strictness as the default in the generic resolver is broader than the reported cron-delivery boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep config account resolution implicit:** `extensions/discord/src/accounts.ts:110-111`
  - body: `resolveDiscordAccount` now treats every caller-supplied id as strict unless `enumerateAccounts` is set, but the exported `discordConfigAdapter.resolveAccount` still wraps it without that flag. Gateway startup/status call `plugin.config.resolveAccount(cfg, id)` and then `isConfigured`, so a named account that intentionally inherits `channels.discord.token` is reported unconfigured.
  - confidence: 0.92
- **[P2] Preserve inherited-token setup lookups:** `extensions/discord/src/token.ts:71-73`
  - body: Strict-by-default token resolution also catches setup callers that pass an account id for the selected setup account. Named accounts without per-account tokens lose the channel-level token and can no longer resolve usernames/channels during setup; make strictness opt-in for send/cron paths or pass `explicit: false` from setup.
  - confidence: 0.84
