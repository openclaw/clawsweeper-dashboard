## Summary

Keep open: the PR targets a real Discord multi-account routing bug, but the current patch still leaks strict token resolution into config/status/setup paths and lacks real Discord behavior proof.

## What This Changes

The branch changes Discord token/account resolution so explicitly targeted non-default accounts do not inherit the channel token, with regression coverage for explicit and implicit account flows.

## Best Possible Solution

Keep inherited channel-token behavior for generic config/status/setup/enumeration resolution, and make strict token resolution opt in from explicit outbound or cron delivery call sites with focused tests and real Discord proof.

## Reproduction Assessment

Yes, by source inspection. Current main falls back to channels.discord.token for a named account without its own token, while the PR-head resolver returns source:none for supplied non-default account IDs unless callers opt out.

## Solution Assessment

No. Strict explicit-send behavior is the right direction, but putting strictness in the generic token/account resolver is broader than the cron/outbound boundary and still breaks established inherited-token paths.
