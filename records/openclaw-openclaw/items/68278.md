## Summary

This PR targets a real Discord account-routing bug on current main, but it should stay open because the latest branch still broadens strict account-token handling into setup token-resolution paths and lacks after-fix proof from a real Discord setup.

## What This Changes

The PR adds explicit Discord token-resolution intent so user-targeted non-default account IDs without their own token do not inherit the channel-level token, while preserving fallback for some implicit account-enumeration paths and adding resolver tests.

## Best Possible Solution

Land a rebased Discord fix that applies strict token scope only where a user or delivery explicitly targets an account, keeps inherited-token behavior for setup/discovery paths, and includes focused tests plus redacted real Discord proof.

## Reproduction Assessment

Yes. Source inspection against current main shows named Discord accounts without their own token fall back to `channels.discord.token`, and the current test suite still pins that fallback for direct token resolution.

## Solution Assessment

No. The branch has the right safety direction for explicit sends, but the current resolver default still affects setup token-assisted username/channel resolution; the safer fix is to pass strict intent only from outbound delivery boundaries and mark setup/discovery calls implicit.
