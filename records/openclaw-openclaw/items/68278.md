## Summary

Keep open: the PR is a plausible Discord account-routing fix, but the latest branch still applies strict token resolution through generic config/setup paths and still has no real Discord behavior proof.

## What This Changes

The branch adds strict Discord token resolution for explicit non-default account IDs, preserves some implicit enumeration fallbacks, and adds token/account regression tests.

## Best Possible Solution

Limit strict resolution to explicit outbound/cron send boundaries, keep generic config/status/setup/enumeration resolution compatible with inherited channel tokens, then add focused regression tests and redacted real Discord proof.

## Reproduction Assessment

Yes, by source inspection. Current main falls back from a named Discord account without its own token to channels.discord.token, while the PR-head resolver returns source:none for supplied non-default account IDs unless callers opt out.

## Solution Assessment

No. The strict explicit-send behavior is the right direction, but making strictness the default for generic token/account resolution is broader than the cron-delivery boundary and breaks established inherited-token paths.
