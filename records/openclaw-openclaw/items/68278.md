## Summary

Keep open: current main still has the explicit Discord account token fallback behavior this PR is trying to harden, but the PR head still treats internal account enumeration as user-targeted account selection, which can hide Discord message actions for valid top-level-token configs. The branch also needs to be brought current and a user-facing changelog entry added before merge.

## What This Changes

The PR adds strict Discord token resolution for explicit non-default account IDs, threads an `explicit` flag through account resolution, and adds Discord token/account regression tests.

## Best Possible Solution

Land a rebased Discord-owned fix that separates user-targeted account IDs from internal/default/list-discovery account resolution, keeps explicit sends strict, preserves implicit channel-token fallback for scans/defaults, and covers token, account, action-discovery, and changelog surfaces.

## Reproduction Assessment

Yes. With `channels.discord.token`, `channels.discord.defaultAccount: "work"`, and `channels.discord.accounts.work` present without a token, unscoped message-tool discovery scans `work`, marks it explicit, receives `tokenSource: "none"`, and drops Discord actions.

## Solution Assessment

No. The strict token guard is the right direction for explicitly targeted sends, but using mere presence of an `accountId` inside `resolveDiscordAccount` is not the right boundary because internal scans also pass account IDs; the resolver needs explicit caller intent or a separate implicit scan path.
