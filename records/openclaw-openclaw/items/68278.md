## Summary

Keep open: current main still has the explicit Discord account-token fallback bug, but this PR’s latest resolver boundary remains too broad for gateway/config/setup paths that should continue inheriting channel tokens, and real behavior proof is still absent.

## What This Changes

The PR adds an explicit Discord token-resolution mode and regression tests so non-default Discord accountId sends without account tokens return no token instead of falling back to the channel token.

## Best Possible Solution

Land a Discord-owned fix that passes strict token intent only from outbound or cron send boundaries while gateway/config/setup enumeration and name-resolution paths keep inherited channel-token behavior.

## Reproduction Assessment

Yes by source inspection, not by an executed test in this read-only sweep. Current main falls back from a named account without its own token to channels.discord.token, while the PR head would make plugin config/gateway and setup token lookups lose that inherited token unless every internal caller opts out.

## Solution Assessment

No. The strict explicit-send guard is the right direction, but making supplied accountId strict by default inside the generic resolver is still too broad; strictness should be opt-in from outbound/cron delivery or routed through a separated resolver path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep gateway account resolution implicit:** `extensions/discord/src/accounts.ts:110-111`
  - body: `tokenExplicit` only opts out for callers that pass the new `enumerateAccounts` flag, but the Discord plugin still exports `resolveDiscordAccount` through `discordConfigAdapter.resolveAccount`. Gateway startup calls `plugin.config.resolveAccount(cfg, id)` for each listed account and then `isConfigured`, so a named account that intentionally inherits `channels.discord.token` now resolves with an empty token and will not start. Move strict intent to outbound/cron sends or make the plugin config/runtime resolver use the implicit path.
  - confidence: 0.92
- **[P2] Preserve setup token lookups:** `extensions/discord/src/token.ts:72`
  - body: Because `resolveDiscordToken(cfg, { accountId })` is strict unless `explicit: false` is supplied, existing setup helpers that resolve Discord allowlist users and groups for a selected account will stop using the channel-level token for named accounts without their own token. That degrades a supported inherited-token setup path to unresolved/numeric-only entries. Pass the implicit flag from setup/discovery callers or make strict resolution opt-in.
  - confidence: 0.82
