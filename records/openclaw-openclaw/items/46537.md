## Summary

Keep open. The PR addresses a real ACP identity mismatch, but the current patch makes ensure-derived IDs authoritative whenever they are marked resolved, which can replace fresher persisted status/event IDs with stale ensure IDs before status reconciliation repairs them.

## What This Changes

The PR marks ACP ensure-derived identities as resolved when stable session IDs are present, adds focused identity-construction tests, and updates an ACP spawn fixture expectation.

## Best Possible Solution

Keep the intended resolved-on-ensure behavior, but make identity merge authority source-aware: ensure results with stable IDs may resolve new or still-pending identities, while already-resolved status/event IDs should not be replaced by later ensure-derived stale IDs. Add a regression test for resolved current IDs plus stale ensure IDs, including the status-refresh-failure path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: merge an already-resolved status/event identity with a later ensure identity carrying stale stable IDs, then let the status refresh path return early with `failOnStatusError: false`.

## Solution Assessment

No. Marking ensure identities as resolved is directionally useful, but using resolvedness as merge authority is too broad; the safer fix is explicit precedence for ensure versus status/event identities.
