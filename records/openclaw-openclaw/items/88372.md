## Summary

Keep open: current main and v2026.6.6 still do not provide a provider-switch transaction or automatic stale config/session cleanup for this Windows repro; the code intentionally preserves explicit or legacy session selections and only clears narrower auto-owned stale route state.

## What This Changes

The issue requests that Windows provider/model switching update or diagnose stale config and session route state instead of leaving visible and actual provider/model routes disagreeing.

## Best Possible Solution

Define one provider-switch/reset contract that preserves explicit user pins, clears auto-owned stale route state, diagnoses remaining stale refs, and aligns status, docs, onboarding, and doctor behavior.

## Reproduction Assessment

Yes, partially from source: current code shows model-default updates do not clear session stores and resets preserve user or legacy pins. The original Windows Groq/Google/LiteLLM sequence still needs live confirmation.

## Solution Assessment

Unclear: a provider-switch transaction or reset path is plausible, but the best fix must distinguish stale automatic state from intentional session pins and avoid changing compatibility semantics by accident.
