## What This Changes

The PR changes Control UI settings storage to accept legacy settings only when the stored gateway URL matches the current page scope, stops writing the unscoped settings key, and adds storage regression tests.

## Best Possible Solution

Land a storage-boundary fix that blocks stale same-origin different-base-path legacy gatewayUrl pollution while preserving documented custom/remote gatewayUrl persistence and safe legacy settings migration, with focused tests and real network proof.

Yes. Source inspection shows the linked bug path, and the PR regression can be reproduced by seeding legacy localStorage with a custom remote gatewayUrl, loading from a page whose derived URL differs, and observing the new fallback gate return defaults instead of the saved gatewayUrl.

No. The owner boundary is right, but the current condition treats valid custom Gateway URLs as pollution; the safer fix should preserve custom/remote migration while only neutralizing stale sibling-base-path legacy gatewayUrl values.

Full review comments:

- **[P1] Preserve custom gateway URL fallbacks:** `ui/src/ui/storage.ts:278-280`
  - body: This gate rejects any legacy settings record whose `gatewayUrl` scope differs from the page-derived URL, but the Control UI explicitly supports remote/custom `gatewayUrl` values that differ from the HTTP origin and stores them in localStorage. Since `loadSettings()` initially looks up the scoped key for the page default, a profile that saved `?gatewayUrl=ws://remote...` or edited the login URL can no longer rediscover that custom scoped record after reload and will connect to the page-derived default instead. Keep legacy data as a migration source, but only replace the stale sibling-base-path `gatewayUrl` case.
