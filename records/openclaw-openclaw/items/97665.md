## Summary

Keep open: this PR targets the right Control UI storage boundary and adds relevant regression tests, but the latest patch still breaks first-load upgrade for existing custom or remote `gatewayUrl` records and still lacks real browser/network proof.

## What This Changes

The PR changes Control UI settings load/save behavior to reject cross-base-path legacy fallback data, stop writing the unscoped settings key, add a page-derived index for custom Gateway URLs, and extend storage tests.

## Best Possible Solution

Land a storage-boundary fix that blocks stale same-origin different-base-path legacy `gatewayUrl` pollution while preserving first-load migration or discovery for intentional custom Gateway URLs, with focused regression tests and real two-base-path proof.

## Reproduction Assessment

Yes, this is source-reproducible. Seed the legacy global settings key with Gateway A's `gatewayUrl`, load Gateway B with no B-scoped settings key, and current main can return A's URL, which `connectGateway()` then uses as the WebSocket target.

## Solution Assessment

No. The owner boundary is right, but the current patch is not the best fix because it still drops first-load legacy custom or remote Gateway URL discovery before the new page-derived index can be created.
