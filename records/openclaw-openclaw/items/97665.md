## Summary

Keep open: this PR targets the right Control UI storage boundary for the linked wrong-Gateway bug, but it is not merge-ready while the new fallback heuristic can drop shipped same-origin custom Gateway URL settings and the branch still lacks real browser or network proof.

## What This Changes

This PR gates Control UI legacy settings fallback by Gateway scope, stops writing the unscoped legacy settings key, writes a page-derived scoped index for custom Gateway URLs, and adds storage regression tests.

## Best Possible Solution

Land a storage-boundary fix that blocks stale sibling base-path legacy URLs, preserves or explicitly migrates intentional custom Gateway URLs, and includes real two-base-path WebSocket proof.

Yes. By source inspection, seed the legacy global settings key with Gateway A's gatewayUrl, load Gateway B with no B-scoped settings key, and current main can return A's URL, which connectGateway then uses as the WebSocket target.
