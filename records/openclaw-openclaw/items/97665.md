## Summary

Keep open: this is a valid Control UI storage bug fix path, but the current fallback predicate would silently drop released legacy custom same-origin Gateway URL settings, and the contributor proof is still test-harness output rather than real behavior proof.

## What This Changes

The PR updates Control UI localStorage load/save logic to isolate gateway settings by base path, preserve some remote legacy migrations, stop writing the shared legacy key, and add storage regression tests.

## Best Possible Solution

Keep the storage-boundary approach, but narrow the legacy fallback logic so stale same-origin sibling records are rejected while intentional same-origin custom Gateway endpoints survive upgrade and reload, then add focused tests plus real browser/network proof.

## Reproduction Assessment

Yes by source inspection: seed the legacy key with a same-origin custom endpoint such as wss://host/ws while loading a page-derived path like /openclaw/chat, and PR head rejects the saved endpoint because origins match but paths differ. I did not run a live browser or reverse-proxy setup in this read-only review.

## Solution Assessment

No. The storage layer and page-scoped save are the right direction, but the current predicate is too broad because it treats every same-origin different-path legacy URL as sibling pollution instead of preserving documented custom Gateway URL settings.
