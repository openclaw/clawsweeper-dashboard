## Summary

Keep open: this PR targets the right Control UI storage bug and adds useful regression coverage, but it is not merge-ready because the fallback gate can drop shipped legacy custom Gateway URL settings before the new page-derived index exists, and the PR still has unit-test output only for a real two-base-path WebSocket behavior change.

## What This Changes

This PR changes Control UI localStorage settings load/save behavior to gate legacy fallback by basePath, stop writing the unscoped settings key, add a page-derived index for custom Gateway URLs, and extend storage tests.

## Best Possible Solution

Land a storage-boundary fix that blocks stale same-origin different-base-path legacy gatewayUrl pollution while preserving first-load migration or discovery for intentional custom Gateway URLs, with focused regression tests and real two-base-path proof.

## Reproduction Assessment

Yes. By source inspection, seed the legacy global settings key with Gateway A's gatewayUrl, load Gateway B with no B-scoped settings key, and current main can return A's URL, which connectGateway then uses as the WebSocket target.

## Solution Assessment

No. The owner boundary is right, but the current patch is not the best fix because it can drop first-load legacy custom or remote Gateway URL discovery before the new page-derived index can be created.
