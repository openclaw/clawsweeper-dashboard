## Summary

Keep open. Current main still returns memory corpus supplements in registration order, while this PR is a focused deterministic-ordering fix with matching regression coverage and no blocking review findings; the remaining decision is maintainer acceptance of `pluginId` as the tie-break before rebase/landing.

## What This Changes

This PR sorts memory corpus supplement registrations by `pluginId` and adds unit tests for out-of-order registration and same-plugin re-registration stability.

## Best Possible Solution

Land this focused change after maintainer review if `pluginId` ordering is the accepted deterministic tie-break; otherwise define an explicit corpus supplement priority/order contract and keep equivalent regression coverage.

## Reproduction Assessment

Yes. On current main, registering corpus supplements as `plugin-z`, `plugin-a`, and `plugin-m` then calling `listMemoryCorpusSupplements()` follows registration order because the function only copies the backing array; overlapping `get()` results can also be reproduced by two supplements where the first non-null result wins.

## Solution Assessment

Yes. Sorting the returned corpus supplement registrations by `pluginId` is the narrowest maintainable fix for the stated cache/order problem and matches the existing prompt supplement pattern; a priority field would only be preferable if maintainers want a richer conflict-resolution policy.
