## Summary

Keep this PR open. Current main still exposes corpus supplements in registration order, while the adjacent prompt supplement path already uses pluginId ordering for cache stability, and the corpus get path can make ordering behaviorally visible. I found no blocking correctness or security finding in the supplied PR diff.

## What This Changes

This PR sorts `listMemoryCorpusSupplements()` by `pluginId` and adds unit coverage for reverse registration order and same-plugin re-registration stability.

## Best Possible Solution

Land this focused change after maintainer review if `pluginId` ordering is the accepted deterministic tie-break; if not, define the preferred explicit ordering contract for corpus supplements and keep the same style of regression coverage around registration and re-registration order.

## Reproduction Assessment

Yes. On current main, registering corpus supplements as `plugin-z`, `plugin-a`, and `plugin-m` then calling `listMemoryCorpusSupplements()` follows registration order because the function only copies the backing array; overlapping `get()` results can also be reproduced by two supplements where the first non-null result wins.

## Solution Assessment

Yes. Sorting the returned corpus supplement registrations by `pluginId` is the narrowest maintainable fix for the stated cache/order problem and matches the existing prompt supplement pattern; a priority field would only be preferable if maintainers want an explicit conflict-resolution policy beyond deterministic ordering.
