## Summary

Keep open: this is a focused Telegram ingress bug fix, but the current diff makes every malformed-JSON polling/webhook error recoverable regardless of HTTP status, so it needs a status-aware retry guard before merge.

## What This Changes

The PR wraps the Telegram ingress worker getUpdates JSON.parse in a descriptive error, adds malformed-json recoverability for polling/webhook contexts, and adds network-errors regression coverage.

## Best Possible Solution

Land the parse guard after making malformed-body recovery status-aware so transient 2xx/5xx/proxy bodies retry while client-status misconfiguration or auth responses remain fatal/status-visible.

## Reproduction Assessment

Yes from source inspection: current main calls JSON.parse on the getUpdates body and the polling catch rethrows when SyntaxError is not classified recoverable. I did not run the PR author's temporary proof script locally because this review is read-only.

## Solution Assessment

No: catching the parse error in fetchJson is the right layer, but the retry classification should be status-aware instead of a broad message snippet that also catches malformed 4xx responses.
