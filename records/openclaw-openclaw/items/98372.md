## Summary

Keep open: the reported Telegram ingress-worker bug is valid, but this PR currently broadens malformed-json recovery so malformed 4xx Bot API responses can retry indefinitely instead of surfacing as fatal/status-visible failures.

## What This Changes

The branch wraps Telegram ingress-worker Bot API JSON parsing in a descriptive error, adds a malformed-json recoverability snippet, and adds network-errors regression coverage.

## Best Possible Solution

Land the parse guard after making malformed-body recovery status-aware so transient 2xx/5xx/proxy bodies retry while client-status misconfiguration or auth responses remain fatal/status-visible.

## Solution Assessment

No: catching the parse error in fetchJson is the right layer, but the retry classification should be status-aware instead of a broad message snippet that also catches malformed 4xx responses.
