## Summary

Keep open: current main and v2026.6.8 still implement Voice Wake, push-to-talk, and Talk as speech-to-transcript or wake-word pipelines with no enrolled-speaker profile or speaker-match gate. ## What This Changes

The issue requests enrolled-speaker recognition for OpenClaw voice modes so only a trusted speaker can trigger or contribute speech input.

## Best Possible Solution

Define one speaker-gating contract for native voice input that covers enrollment UX, local state ownership, provider egress policy, fallback semantics, and plugin/API boundaries before implementation.

- Provider-based diarization could send raw microphone audio to third parties, requiring consent, SecretRef/auth handling, provider capability checks, and egress policy.
- The owner boundary is unresolved across macOS Voice Wake/PTT, native mobile Talk, Gateway realtime Talk, and plugin/provider extension points.
