## Summary

Keep open: current main and v2026.6.6 still implement Voice Wake, push-to-talk, and Talk as speech-recognition/transcript pipelines without an enrolled-speaker profile or speaker-match gate. Related raw-audio, custom-ASR, and identity-routing items do not cover rejecting non-enrolled voices, and this needs product/security design before implementation.

## What This Changes

Request speaker enrollment and speaker-identity gating so Voice Wake, push-to-talk, and Talk ignore voices that do not match the enrolled user.

## Best Possible Solution

Define one speaker-gating contract for native voice input that covers enrollment UX, local state storage, provider egress policy, fallback semantics, and extension boundaries before adding code.

## Risks / Open Questions

- The owner boundary is unresolved across native app capture, Gateway Talk sessions, provider diarization, and possible plugin/API extension points.
