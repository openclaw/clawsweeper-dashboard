## Summary

Keep this issue open. Current main still has the reported cron announce gap: normal reply/ACP delivery applies the TTS transform, but isolated cron announce text is normalized and sent through outbound delivery without calling the TTS directive pipeline, so `[[tts]]` tags can remain visible.

## What This Changes

Cron isolated announce delivery should process model-emitted `[[tts]]` and `[[tts:text]]` directives like normal replies, producing voice-note media instead of sending raw tags.

## Best Possible Solution

Add a lazy shared pre-delivery TTS transform in isolated cron announce delivery before `deliverOutboundPayloads`, using `cfgWithAgentDefaults`, resolved channel/account, agent id, final reply kind, and the cron session's preserved `ttsAuto` value. Cover tagged cron text so visible tags are stripped and synthesized audio is delivered as media/voice.

## Reproduction Assessment

Yes. The issue has a high-confidence static reproduction path: isolated cron announce text becomes a plain `ReplyPayload`, then `dispatchCronDelivery` sends it without the TTS helper that normal replies use.

## Solution Assessment

Yes. Reusing the existing TTS payload transform in the cron announce path is the narrowest maintainable fix; a Telegram-specific workaround would leave other channels and future TTS providers inconsistent.
