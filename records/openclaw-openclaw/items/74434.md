## Summary

Keep this PR open, but do not merge the current patch as-is. The central fix direction is plausible because current main still leaves pi-coding-agent auto-retry enabled while OpenClaw has its own outer retry paths, but this implementation uses a persistent Pi settings setter and can permanently change an agent's global Pi retry setting in the trusted project-settings path.

## What This Changes

The PR disables pi-coding-agent auto-retry from `createPreparedEmbeddedPiSettingsManager` and adds a Vitest assertion that the prepared settings manager reports retry disabled.

## Best Possible Solution

Land a narrow retry fix that disables pi-coding-agent agent-level auto-retry only for OpenClaw embedded sessions, preserves user/global Pi settings, reapplies the runtime override after any `DefaultResourceLoader.reload()` that can discard in-memory overrides, and adds targeted coverage for the trusted file-backed path plus a user-facing changelog entry.

## Reproduction Assessment

Yes for the PR-introduced defect: configure embedded Pi project settings policy as `trusted` with no plugin settings, then this patch's `setRetryEnabled(false)` runs on a file-backed manager whose upstream setter saves global `retry.enabled=false`. The original #73781 loop has plausible user reproduction steps, but this read-only review did not run a live model loop.

## Solution Assessment

No. Disabling the inner Pi auto-retry is likely the right boundary, but this patch should implement it as a session-scoped runtime override rather than a persistent settings write, and it should be coordinated with the broader #74503 loop-detection proposal.
