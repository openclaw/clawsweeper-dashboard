## Summary

Keep open: the patch is focused and I did not find a blocking correctness bug, but it intentionally broadens the `apply_patch` file-write boundary and needs explicit maintainer/security policy approval before merge.

## What This Changes

The PR updates tool construction, tests, docs, and schema help so `apply_patch` can inherit `security=full` plus `ask=off` exec authority when its own `workspaceOnly` setting is unset.

## Best Possible Solution

If maintainers accept the trusted-exec inheritance policy, merge the narrow implementation with its regression tests and docs; otherwise keep requiring the explicit `tools.exec.applyPatch.workspaceOnly=false` opt-out.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main passes `workspaceOnly: true` to `apply_patch` unless `tools.exec.applyPatch.workspaceOnly` is explicitly false, so full/off exec with that setting unset remains contained. The PR body also supplies after-fix live output for the changed behavior.

## Solution Assessment

Likely yes if the policy is accepted: the patch changes the single tool-construction decision and adds focused guardrail tests for the important containment overrides. The remaining question is product/security approval for the permission inheritance itself.
