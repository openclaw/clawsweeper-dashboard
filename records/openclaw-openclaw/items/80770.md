## Summary

Keep open: this is an active, focused permission-boundary PR with sufficient real behavior proof and passing CI, but it still needs maintainer/security approval and one small docs consistency fix before merge.

## What This Changes

The PR lets `apply_patch` inherit configured `security=full`/`ask=off` exec authority when `workspaceOnly` is unset, with regression tests plus config-help/type and docs updates.

## Best Possible Solution

If maintainers accept the trusted-exec inheritance policy, merge the narrow runtime change with coherent docs; otherwise keep requiring explicit `tools.exec.applyPatch.workspaceOnly=false`.

## Reproduction Assessment

Yes, source-reproducible: current main computes `applyPatchWorkspaceOnly` as workspace-contained unless `workspaceOnly` is explicitly false, so configured full/off exec with `applyPatch.workspaceOnly` unset remains blocked. The PR body supplies after-fix live output for the changed behavior.

## Solution Assessment

Likely yes if the policy is accepted: the implementation changes the single tool-construction decision and keeps approval-required exec plus explicit workspace guards contained. The apply_patch section in `docs/tools/exec.md` should be updated so the public docs do not contradict the new behavior.
