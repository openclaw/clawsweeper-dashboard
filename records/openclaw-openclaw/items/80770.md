## Summary

Keep open: this is a focused PR with sufficient proof, but it intentionally expands a security-sensitive file-write boundary and still needs maintainer/security approval; I also found one docs consistency issue to fix before merge.

## What This Changes

The PR lets `apply_patch` inherit configured `tools.exec.security=full` plus `tools.exec.ask=off` authority when `applyPatch.workspaceOnly` is unset, with tests and docs/schema-help updates.

## Best Possible Solution

If maintainers accept the trusted-exec inheritance policy, merge the narrow implementation after fixing the duplicate docs note; otherwise keep the explicit `tools.exec.applyPatch.workspaceOnly=false` opt-out.

## Reproduction Assessment

Yes, source-reproducible: current main passes `workspaceOnly: true` to `apply_patch` unless `tools.exec.applyPatch.workspaceOnly` is explicitly false. The PR body also supplies after-fix live output from a local tool-construction run showing the escaped file write under configured full/off exec.

## Solution Assessment

Not as-is: the code change is narrow if maintainers accept the policy, but the same exec docs page still contains the old default statement. The security boundary change also needs explicit maintainer/security approval before merge.
