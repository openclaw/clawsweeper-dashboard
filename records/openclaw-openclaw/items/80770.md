## Summary

Keep open for maintainer/security judgment: the patch is narrow and has convincing proof, but it intentionally changes the `apply_patch` file-write boundary for trusted full/off exec configurations.

## What This Changes

The PR changes `apply_patch` policy, tests, and docs so unset `tools.exec.applyPatch.workspaceOnly` can inherit configured no-approval full-exec host-write authority.

## Best Possible Solution

If maintainers accept trusted-exec inheritance, merge the narrow implementation with its tests/docs; otherwise keep the current explicit `tools.exec.applyPatch.workspaceOnly=false` opt-out requirement.

## Reproduction Assessment

Yes, source-reproducible: current main passes `workspaceOnly: true` to `apply_patch` unless the applyPatch flag is explicitly false, while the PR body supplies after-fix live output for the changed full/off case.

## Solution Assessment

Likely yes if maintainers accept the policy: the implementation changes the single tool-construction decision and preserves approval-required exec, explicit applyPatch guards, and `tools.fs.workspaceOnly`. The unresolved part is product/security approval for the new default inheritance.
