## Summary

Keep open for maintainer/security review: this PR is focused and has real after-fix proof, but it intentionally broadens `apply_patch` host-write scope under trusted full/off exec and leaves one duplicate docs note stale.

## What This Changes

This PR makes `apply_patch` inherit explicit no-approval full-exec authority when `applyPatch.workspaceOnly` is unset, with focused regression tests and docs/schema help updates.

## Best Possible Solution

If maintainers accept trusted-exec inheritance, merge the narrow implementation after updating the duplicate exec doc note and required checks; otherwise keep the explicit `tools.exec.applyPatch.workspaceOnly=false` requirement.

## Reproduction Assessment

Yes. Current main source shows the unset `applyPatch.workspaceOnly` path is still workspace-contained, and the PR body supplies after-fix live output for the intended full/off behavior; I did not run tests because this review is read-only.

## Solution Assessment

Likely yes if maintainers accept the policy: the code change is localized to tool construction and the tests cover the important guards. The safer alternative is to keep requiring the explicit dangerous `applyPatch.workspaceOnly=false` flag.
