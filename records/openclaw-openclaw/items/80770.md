## Summary

Keep open. Current main still enforces the existing workspace-only default, and this PR is an active, focused behavior change that broadens a tool permission boundary; it needs maintainer/security policy approval rather than cleanup closure.

## What This Changes

The PR changes apply_patch tool construction so unset `tools.exec.applyPatch.workspaceOnly` inherits configured `security=full` plus `ask=off` exec authority, with focused tests and docs/config-help updates.

## Best Possible Solution

Have maintainers decide the trusted-exec permission policy; if accepted, merge the narrow implementation with its regression tests and docs, otherwise keep requiring the explicit dangerous opt-out.

## Reproduction Assessment

Yes, source-reproducible. Current main passes workspace-only behavior to `apply_patch` unless `tools.exec.applyPatch.workspaceOnly` is explicitly false, while the PR body supplies after-fix live output for the changed full/off case.

## Solution Assessment

Likely yes if the policy is accepted. The PR changes the single tool-construction decision and keeps approval-required exec, `tools.fs.workspaceOnly`, and explicit `applyPatch.workspaceOnly=true` contained; the remaining question is maintainer/security approval of the default inheritance.
