## Summary

Keep open: the patch is aimed at a real friction point, but it does not use the same effective exec defaults as the exec tool, so a documented no-approval full-exec path can still leave `apply_patch` workspace-contained unless the extra dangerous knob is set.

## What This Changes

The PR changes `apply_patch` to inherit unrestricted path authority when trusted full exec with approvals off is configured, adds regression coverage, and updates tool docs/config help.

## Best Possible Solution

Keep the PR open and update the implementation to derive `apply_patch` containment from the same resolved exec defaults used by `exec`, then align both tool docs sections.

## Reproduction Assessment

Yes. Source inspection shows current main keeps `apply_patch` workspace-only unless `applyPatch.workspaceOnly=false`, while the exec tool can be no-approval full exec through defaulted `ask=off`; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The direction is plausible, but the patch should resolve effective exec defaults before computing `applyPatchWorkspaceOnly`, otherwise it misses documented trusted exec paths where `ask` is omitted.
