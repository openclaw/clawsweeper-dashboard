## Summary

Keep open: the PR addresses a real current-main limitation, but it intentionally changes a file-write permission boundary and leaves one conflicting exec-docs statement, so it needs maintainer/security review plus a small docs fix before merge.

## What This Changes

The PR makes `apply_patch` inherit explicit `tools.exec.security=full` plus `tools.exec.ask=off` when `tools.exec.applyPatch.workspaceOnly` is unset, with tests and docs updates.

## Best Possible Solution

If maintainers accept trusted-exec inheritance, update the remaining exec apply_patch doc note and merge after security approval; otherwise keep requiring explicit `tools.exec.applyPatch.workspaceOnly=false`.

## Reproduction Assessment

Yes, source-reproducible: current main passes `workspaceOnly: true` to `apply_patch` unless `tools.exec.applyPatch.workspaceOnly` is explicitly false. The PR body supplies after-fix live output for the changed full/off behavior.
