## Summary

Keep open: the PR is a narrow implementation with sufficient proof and no discrete correctness finding, but it intentionally expands the `apply_patch` file-write boundary and needs maintainer/security approval before merge.

## What This Changes

The PR makes `apply_patch` inherit `security=full` plus `ask=off` exec authority when `applyPatch.workspaceOnly` is unset, with docs, schema help, type comments, and regression tests.

## Best Possible Solution

If maintainers accept the trusted-exec policy, merge the narrow inheritance with the included tests and docs; otherwise keep requiring explicit `tools.exec.applyPatch.workspaceOnly=false`.

## Reproduction Assessment

Yes, source-reproducible: current main keeps `apply_patch` workspace-contained unless `applyPatch.workspaceOnly=false`, while the PR body supplies after-fix live output for the full/off case. I did not run a live Gateway session.

## Solution Assessment

Likely yes if the policy is accepted: the patch changes the single tool-construction decision and keeps approval-required exec, `tools.fs.workspaceOnly`, and explicit `applyPatch.workspaceOnly=true` contained. The unresolved part is maintainer/security approval of the policy change, not a narrower code repair.
