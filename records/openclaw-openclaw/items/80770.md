## Summary

Keep open: the patch is narrow and has sufficient after-fix proof, but it intentionally broadens `apply_patch` host-write authority under trusted full exec and needs explicit maintainer/security approval before merge.

## What This Changes

The PR lets `apply_patch` inherit no-approval full exec authority when `tools.exec.applyPatch.workspaceOnly` is unset, with focused tests plus docs and config-help updates.

## Best Possible Solution

If maintainers accept this trusted-exec policy, merge the narrow implementation with its regression tests and docs; otherwise keep requiring the explicit `tools.exec.applyPatch.workspaceOnly=false` opt-out.

## Reproduction Assessment

Yes, source-reproducible: current main passes `workspaceOnly: true` to `apply_patch` unless `tools.exec.applyPatch.workspaceOnly` is explicitly false, so full/off exec with the setting unset remains blocked today. The PR body supplies after-fix live output for the changed behavior.

## Solution Assessment

Likely yes if the policy is accepted: the PR changes the single tool-construction decision and keeps approval-required exec, `tools.fs.workspaceOnly`, and explicit `applyPatch.workspaceOnly=true` contained. The unresolved question is whether maintainers want this permission inheritance by default.
