## Summary

Keep open. Current main still leaves the reported source-level behavior unresolved: ACP backend registration remains local to the process that starts the acpx plugin service, while the explicit agent JSON path still returns the raw run result without a descendant-drain reconciliation that would correct stale `sessions_yield` liveness/replay metadata.

## What This Changes

The issue asks for source-level fixes and regression coverage for explicit `openclaw agent --json` ACP runs that fail to see the embedded ACPX backend and can return stale final JSON metadata after `sessions_yield`.

## Best Possible Solution

Patch the source path with a policy-aware ACPX backend registration or activation seam for explicit embedded ACP runs, then add explicit final-result reconciliation for `agent --json` so descendant completion after `sessions_yield` updates final assistant text and terminal metadata only when completion is proven.

## Reproduction Assessment

Yes for a focused source-level reproduction: run `openclaw agent --json` against an ACP session in an embedded fallback/local process without preloaded acpx, and use an agent turn that spawns ACP work then calls `sessions_yield` before final text. I did not run the live Homebrew SSD smoke in this read-only review.

## Solution Assessment

Yes, the issue's proposed direction is the right shape, but the backend fix should be policy-aware instead of broadly starting all plugins from the CLI. The final JSON fix should reuse the existing run-wait/assistant-reply helpers or an equivalent narrow gateway seam rather than special-casing only the reported transcript text.
