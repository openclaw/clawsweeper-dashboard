## Summary

Close as ClawHub/plugin work: the custom portal use case is useful, but current core already exposes plugin-owned session state plus prompt-hook system-context seams, and VISION.md explicitly routes optional integration capability out of core when those seams are sufficient.

## What This Changes

The issue requests a session-persistent `/sticky` command or WebSocket/API surface that appends dynamic context to the agent system prompt instead of adding context to visible message history.

## Best Possible Solution

Build the Jarvis portal behavior as a plugin published or documented through ClawHub: store trusted per-session portal rules in plugin-owned session state and inject them via `before_prompt_build` system-context hooks, rather than adding a generic core `/sticky` command.

## Reproduction Assessment

Not applicable. This is a feature/API request, and current source/docs already show `/btw`, `chat.inject`, plugin session state, and prompt hooks are behaving according to their documented contracts.

## Solution Assessment

No, adding a built-in core `/sticky` command is not the best default path. The narrower maintainable solution is a trusted plugin/ClawHub implementation that owns portal authorization, per-session state, and prompt-hook injection.
