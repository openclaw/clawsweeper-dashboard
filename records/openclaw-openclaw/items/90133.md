## Label Justifications

- `P3`: This is a useful but speculative feature/API request with limited immediate blast radius and no current broken documented behavior.
- `impact:session-state`: The request is about durable session-scoped prompt context that would affect future turns without appearing as normal chat history.
- `impact:security`: The proposed surface would let external clients inject higher-authority instructions, so auth and prompt-authority boundaries are central to the design.

## Summary

Keep open. Current main and the latest release already make `/btw` ephemeral rather than persistent transcript history, but custom WebSocket/chat clients still lack a deliberate session-scoped system-prompt overlay; adding that would create a new prompt-authority and session-state surface that needs maintainer product and security review.

## What This Changes

Request a `/sticky` command or `/btw --system` flag that stores session-scoped instructions in system-prompt space instead of visible message history.

## Best Possible Solution

Keep `/btw` ephemeral and design a separate explicit session prompt-overlay/API, or document the supported per-run and plugin-hook paths, with clear auth, persistence, visibility, reset, and cache semantics.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs show `/btw` is already ephemeral, while `chat.send` lacks a persistent system-prompt overlay field for the requested WebSocket client use case.

## Solution Assessment

No for `/btw --system`; `/btw` is a side-question contract. A separate session-scoped prompt-overlay or explicit API/SDK capability is likely the cleaner maintainable design.
