## Summary

Keep open. Current main still lacks the live active-job producer signal for startup catch-up and manual cron runs, while task maintenance treats that signal as the cron runtime authority; the PR appears narrowly correct, but the branch is currently dirty/unmergeable and needs maintainer rebase/review rather than closure.

## What This Changes

The PR imports active-job helpers into `src/cron/service/ops.ts`, marks and clears job IDs around manual run execution, marks and clears startup catch-up execution in `src/cron/service/timer.ts`, and adds `src/cron/active-jobs-symmetry.test.ts` regression coverage.

## Best Possible Solution

Land a rebased version of the narrow active-job symmetry fix for startup catch-up and manual cron runs, while leaving durable restart/recovery policy to the broader cron reconciliation work tracked separately.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction: startup catch-up and manual cron runs can create `runtime="cron"` task rows and run longer than the 5-minute grace window while `isCronJobActive(jobId)` remains false on those producer paths.

## Solution Assessment

Yes. Mirroring `markCronJobActive`/`clearCronJobActive` onto the two missing producer paths is the narrowest maintainable fix for this live in-process ownership signal; durable restart recovery remains a separate design path.
