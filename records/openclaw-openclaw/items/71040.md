## Summary

Keep open. Current main still lacks active-job mark/clear coverage for startup catch-up and manual cron runs, so the PR addresses a real remaining bug, but the proposed startup catch-up clear happens before batched outcomes are applied and can still let earlier catch-up task rows become `lost` while later missed jobs are running.

## What This Changes

The PR adds cron active-job mark/clear bookkeeping around manual and startup catch-up cron execution paths and adds regression tests for active-job symmetry.

## Best Possible Solution

Land a rebased narrow fix that marks manual and startup catch-up cron jobs active for the full period where their task rows are non-terminal, keeps startup catch-up active until each outcome is applied or applies outcomes per candidate, and adds focused regression plus changelog coverage without changing durable restart recovery policy.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction: a startup catch-up or manual cron task can remain `running` past the 5-minute reconciliation grace while `isCronJobActive(jobId)` stays false and no terminal durable run history exists yet.

## Solution Assessment

No, not as written. Mirroring active-job bookkeeping is the right approach, but the startup catch-up clear should not run inside `runStartupCatchupCandidate` before `applyStartupCatchupOutcomes` finalizes batched results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep startup jobs active until their outcomes are applied:** `src/cron/service/timer.ts:1079-1080`
  - body: `executeStartupCatchupPlan` collects all missed-job outcomes and only later calls `applyStartupCatchupOutcomes`, so clearing `activeJobIds` in the candidate `finally` leaves an earlier completed-but-unapplied catch-up task inactive while later missed jobs run. If that later run exceeds the 5-minute grace, maintenance can still mark the first task `lost`, and the later success update will not downgrade `lost`. Let `applyOutcomeToStoredJob` perform the clear or apply each outcome before starting the next candidate.
  - confidence: 0.87
