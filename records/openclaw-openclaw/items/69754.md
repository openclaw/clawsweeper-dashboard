## Summary

Keep open: current main still has source-reproducible Feishu approval fallback gaps where degraded document-comment delivery drops actionable button commands and shared approval fallback text still shows literal `/approve <id>` placeholders.

## What This Changes

Request clearer Feishu approval fallback guidance when interactive approval buttons are unavailable or degrade to text.

## Best Possible Solution

Keep native Feishu cards for supported chat surfaces, but make degraded Feishu text fallbacks explicitly say buttons are unavailable in the current surface and include exact allowed `/approve` commands from the approval descriptors.

## Reproduction Assessment

Yes, at source level. Current main routes Feishu document-comment approval payloads through a fallback renderer that omits button values, and shared approval builders still emit literal `/approve <id>` text despite the documented exact-command fallback contract.

## Solution Assessment

Yes. A focused fallback-rendering and copy repair is the narrowest maintainable solution because it can reuse existing approval button values, allowed-decision descriptors, and the documented `/approve` contract without adding a new config option or native Feishu feature.
