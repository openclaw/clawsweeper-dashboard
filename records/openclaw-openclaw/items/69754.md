## Summary

Keep open. Current main still has source-reproducible Feishu approval fallback gaps: Feishu document-comment payloads degrade approval controls to label-only text, and the shared exec/plugin fallback builders still render literal `/approve <id>` placeholders instead of copy-pasteable commands.

## What This Changes

The issue asks OpenClaw to explain Feishu approval fallback/degraded states clearly and provide exact manual approval instructions when interactive buttons are unavailable.

## Best Possible Solution

Keep native Feishu cards where supported, and make degraded Feishu text fallbacks explicitly say buttons are unavailable on that surface while including exact allowed `/approve <id> <decision>` commands.

## Reproduction Assessment

Yes, at source level. The current Feishu document-comment path converts approval-style interactive payloads to label-only text, and the shared fallback builders still emit literal `/approve <id>` placeholders despite the docs requiring exact commands.

## Solution Assessment

Yes. A focused fallback-rendering and copy repair is the narrowest maintainable path because it can reuse existing approval button values, allowed-decision descriptors, and the documented `/approve` contract without adding a new Feishu transport or config knob.
