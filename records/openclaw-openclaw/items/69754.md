## Summary

Keep open. Current main fixed the literal `<id>` fallback bug from a related PR, but Feishu still has source-reproducible approval fallback clarity gaps: document-comment delivery can strip approval buttons to label-only text, and Feishu has no native exec approval client/config surface while the generic pending text still implies interactive approvals are available.

## What This Changes

Request clearer Feishu approval fallback copy and documentation when interactive approval buttons/cards are unavailable or degrade to manual `/approve` text.

## Best Possible Solution

Keep this open and land a narrow Feishu fallback repair that preserves exact `/approve` commands when buttons/cards cannot render, explains why manual approval is required on that surface, and updates Feishu approval docs without adding a new config surface.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Feishu comment-target delivery strips interactive approval buttons to label-only text, and direct exec fallback copy still presents manual `/approve` as an interactive approval path. I did not run a live Feishu setup in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow fallback-rendering and copy repair, not a full Feishu native exec approval implementation. The broader native-card feature can continue separately in https://github.com/openclaw/openclaw/pull/60328.
