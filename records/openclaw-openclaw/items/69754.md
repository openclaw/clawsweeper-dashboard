## Summary

Keep open. Current main still has source-reproducible Feishu approval fallback gaps: Feishu document-comment payloads degrade approval buttons to label-only text, and shared approval fallback builders still render literal `/approve <id>` placeholders instead of concrete commands.

## What This Changes

Requests clearer Feishu approval fallback guidance when interactive buttons are unavailable, including why the approval degraded and the exact manual approval path.

## Best Possible Solution

Keep native Feishu cards for supported chats, but make degraded Feishu text fallbacks explain that buttons are unavailable on the current surface and include exact allowed `/approve <id> <decision>` commands from existing approval descriptors.

## Reproduction Assessment

Yes, at source level. The Feishu document-comment path converts approval-style interactive payloads to label-only text, and the generic fallback builders still emit literal `/approve <id>` placeholders despite the docs requiring exact commands.

## Solution Assessment

Yes. A focused fallback-rendering and copy repair is the narrowest maintainable path because it can reuse existing approval button values, allowed-decision descriptors, and the documented `/approve` contract without adding a new config knob or native Feishu approval transport.
