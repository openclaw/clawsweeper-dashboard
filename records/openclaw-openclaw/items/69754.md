## Summary

Keep open. Current main still has source-reproducible Feishu approval fallback gaps: document-comment delivery strips interactive/card data into label-only text, and the shared approval fallback builders still render literal `/approve <id>` placeholders instead of concrete commands; related open PRs cover only part of the remaining work or a broader Feishu native-approval feature.

## What This Changes

The issue asks for clearer Feishu approval fallback guidance when interactive approval buttons are unavailable or degraded to text.

## Best Possible Solution

Keep native Feishu cards for supported chats, but make degraded Feishu text fallbacks explicitly say buttons are unavailable on the current surface and include exact allowed `/approve <id> <decision>` commands from the approval descriptors.

## Reproduction Assessment

Yes, at source level. Current main routes Feishu document-comment approval payloads through a fallback renderer that omits button values, and the shared approval builders still emit literal `/approve <id>` text despite the docs requiring exact commands.

## Solution Assessment

Yes. A focused fallback-rendering and copy repair is the narrowest maintainable path because it can reuse existing approval button values, allowed-decision descriptors, and the documented `/approve` contract without adding a new config knob or native Feishu transport.
