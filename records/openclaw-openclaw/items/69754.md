## Summary

Keep open. Current main covers the happy path by rendering Feishu approval buttons as native cards, but the degraded/text path still uses generic or label-only fallback copy and the Feishu docs do not explain how to finish approvals when buttons are unavailable, so the reported UX gap remains actionable.

## What This Changes

The issue requests clearer Feishu approval fallback guidance when interactive approval buttons are unavailable or degrade to manual `/approve ...` text.

## Best Possible Solution

Keep the existing native Feishu card path, but add a narrow fallback-copy and docs update so any Feishu text-only downgrade explains that interactive buttons are unavailable for that surface/context, includes exact `/approve <id> <decision>` commands for the allowed decisions, and tells the user whether same-chat Feishu, Web UI, or terminal approval is the right completion path.

## Reproduction Assessment

Yes. A code-level reproduction exists: Feishu normal chat sends interactive buttons as cards, while the Feishu comment-target fallback strips interactivity and the current tests assert label-only text; the generic forwarded approval test also asserts the placeholder `/approve <id>` copy.

## Solution Assessment

No. Current main is a partial solution because it fixes normal Feishu button rendering, but it does not fully solve the reported fallback guidance problem when the approval payload becomes text-only.
