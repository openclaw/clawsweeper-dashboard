## Summary

Keep open. Current main still has a source-reproducible Feishu approval fallback gap: document-comment delivery strips interactive/card data into text, the shared renderer drops non-URL button values, and the generic approval fallback text still contains literal `/approve <id>` placeholders despite docs and prompts expecting exact commands.

## What This Changes

The issue asks for Feishu approval fallback messages to clearly explain unavailable interactive buttons and provide exact manual approval instructions when approval degrades to text.

## Best Possible Solution

Keep native Feishu cards where supported, but make degraded text fallbacks explicitly say interactive buttons are unavailable on the current surface and include exact `/approve <request id> <decision>` commands derived from existing approval descriptors or button values.

## Reproduction Assessment

Yes, at source level. Current main routes Feishu document-comment approval payloads through a fallback renderer that omits button values, and shared approval builders still emit literal `/approve <id>` text despite docs requiring exact commands.

## Solution Assessment

Yes. A focused fallback-rendering and copy repair is the narrowest maintainable path because it can reuse existing approval button values, allowed-decision descriptors, and the documented `/approve` contract without adding a config knob or native Feishu transport.
