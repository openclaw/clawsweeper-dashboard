## Summary

Keep open. Current main still has Feishu degraded approval paths where interactive approval payloads become label-only text, and the generic forwarded approval text still uses a literal `/approve <id>` placeholder; the open interpolation PR helps one piece but does not cover Feishu document-comment degradation or channel-specific fallback guidance.

## What This Changes

Request clearer Feishu approval fallback copy when interactive buttons are unavailable, including why the UX degraded and the exact manual approval path.

## Best Possible Solution

Keep native Feishu cards for supported chats, but make degraded Feishu text fallbacks explain that buttons are unavailable on the current surface and include exact allowed `/approve <id> <decision>` commands.

## Reproduction Assessment

Yes. At source/test level, sending an interactive approval-style payload to a Feishu document-comment target produces label-only text, and the generic fallback builder still emits a literal `/approve <id>` placeholder; I did not run tests because this was a read-only review.

## Solution Assessment

Yes. A focused fallback-rendering/copy repair is the narrowest maintainable path; it should reuse the existing approval payload values and docs contract without adding a new approval transport, config knob, or authorization model.
