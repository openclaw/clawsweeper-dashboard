## Summary

Keep open. Current main has useful mitigations for pricing startup, model prewarm, Telegram control calls, proxy dispatchers, and event-loop diagnostics, but there is no proof that the broader native Windows gateway-process HTTP failure across Telegram REST, Discord REST, pricing/catalog fetches, and local handshakes is fixed or fully superseded.

## What This Changes

The issue reports native Windows 10 plus Node 22 losing outbound HTTP/HTTPS inside the running Gateway process while WebSocket paths continue, and requests a model-prewarm skip workaround.

## Best Possible Solution

Reproduce on native Windows with Node 22 and Node 24 under multiple Discord plus Telegram providers, instrument fetch/undici dispatcher state, DNS/connect/TLS timing, active handles, event-loop delay, and startup concurrency, then land a targeted Gateway or plugin transport fix that preserves proxy and SSRF boundaries.

## Reproduction Assessment

Unclear: the issue includes detailed field steps and same-machine standalone fetch controls, but this review did not run a native Windows Gateway reproduction against current main. High-confidence reproduction needs native Windows 10/11, Node 22/24, multiple Discord and Telegram providers, configured default primary model, and live Gateway HTTP probes.

## Solution Assessment

Yes for keeping this open. The current implementation provides scoped mitigations and diagnostics, but the narrow maintainable fix still requires maintainer-led Windows root-cause isolation before code changes can be safely queued.
