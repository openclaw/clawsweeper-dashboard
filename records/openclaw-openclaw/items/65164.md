## Summary

Keep open. Current main has partial mitigations for the startup pain, including a non-blocking bounded primary-model prewarm, a startup-prewarm skip env var, optional model-pricing bootstrap, and scoped Telegram timeout/fallback work. Those do not prove the broader native Windows gateway-process HTTP failure is fixed across Telegram REST, Discord REST, model/catalog HTTP, and local Gateway client/approval handshakes under the reported multi-provider load.

## What This Changes

The issue asks OpenClaw to fix a native Windows gateway-process failure where outbound HTTP/HTTPS calls hang or fail while WebSocket surfaces keep working, and requests a model-prewarm skip workaround.

## Best Possible Solution

Reproduce on native Windows with Node 22/24 and multiple Discord plus Telegram providers, instrument global undici dispatcher state, active handles, DNS/connect/TLS timing, and channel startup concurrency, then fix the process-wide HTTP failure without weakening SSRF or proxy boundaries. Keep the existing startup-prewarm and pricing-disable mitigations as bounded workarounds.
