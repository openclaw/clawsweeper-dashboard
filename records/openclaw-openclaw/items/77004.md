## Summary

Keep open: the Control UI `/new` regression is real, but the PR currently makes every `sessions.create` call with `parentSessionKey` look like a webchat `/new` command, which is broader than the documented hook contract.

## What This Changes

The PR adds parent-session hook emission to `sessions.create`, exports the before-reset helper, adds gateway regression tests, and records a changelog fix for the Control UI `/new` hook regression.

## Best Possible Solution

Constrain the restored hook emission to the actual Control UI new-session transition, while leaving generic SDK/programmatic parent-session creation as normal session creation unless it explicitly opts into command/reset semantics.

## Reproduction Assessment

Yes, source-level. Current main routes Control UI typed `/new` into `sessions.create` with a `parentSessionKey`, and that handler records the parent but does not emit the documented command or reset/lifecycle hooks.

## Solution Assessment

No, not as currently patched. Emitting the reset-style hooks from the gateway is the right boundary, but `parentSessionKey` alone is too broad because the SDK exposes it for ordinary session creation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate hook emission on the actual webchat /new path:** `src/gateway/server-methods/sessions.ts:1011-1021`
  - body: This block runs for any `sessions.create` request that supplies `parentSessionKey`, but `parentSessionKey` is exposed on the SDK create params and is not itself proof that the caller issued a webchat `/new` command. As written, programmatic child-session creates will fire `command:new`, `before_reset`, and session lifecycle hooks with `commandSource: "webchat"`, causing session-memory/custom hooks to run for non-command creates. Please gate this on the Control UI new-session source, or add an explicit create reason/emit-hooks signal, instead of using `parentSessionKey` alone.
  - confidence: 0.86

## Repair Work Prompt

On PR #77004, preserve the fix for #76957 but narrow the hook emission boundary. `sessions.create` should emit `command:new`, `before_reset`, `session_end`, and `session_start` only for the actual Control UI/new-session transition, not for every API/SDK call that passes `parentSessionKey`. Prefer using existing request/client context such as the Control UI webchat client identity if sufficient; otherwise add the smallest explicit internal create reason/emit-hooks signal from the Control UI call path. Keep generic SDK/programmatic `oc.sessions.create({ parentSessionKey })` behavior as ordinary session creation. Update or add regression tests covering both the Control UI hook path and a non-webchat/programmatic parent create that must not emit `command:new`. Keep the existing changelog entry if the user-facing fix remains, and do not broaden session API semantics beyond this bug fix.
