## Summary

Keep open: the regression is valid and the PR is close, but the current diff keys hook emission on `parentSessionKey` alone, which would treat public SDK/dashboard branched session creates as webchat `/new` resets.

## What This Changes

The PR exports the before-reset hook helper, emits `command:new`, `before_reset`, `session_end`, and `session_start` hooks from `sessions.create` when a parent session is provided, adds gateway regression tests, and records a changelog entry.

## Best Possible Solution

Preserve `/new` hook delivery for Control UI fresh-session creates while keeping generic `sessions.create` parent linkage side-effect-free unless the caller explicitly requests `/new` semantics.

## Reproduction Assessment

Yes, source-level: current main routes typed Control UI `/new` to `sessions.create` with a `parentSessionKey`, while `sessions.create` only records that parent link and does not run the documented `/new` hook path.

## Solution Assessment

No. The PR restores the missing hooks, but using `parentSessionKey` as the only trigger is broader than the public SDK/API contract; the safer fix is a scoped Control UI new-session signal plus a negative generic-parent-create test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope `/new` hooks to actual new-session commands:** `src/gateway/server-methods/sessions.ts:1008-1026`
  - body: This block runs whenever `sessions.create` has a `parentSessionKey`, but that field is exposed through the SDK and used as branch/thread metadata, not only for Control UI `/new`. Non-`/new` parent-linked creates would now fire `command:new`, `before_reset`, and lifecycle hooks with `commandSource: "webchat"`, causing session-memory or custom hooks to snapshot the parent even when no `/new` command was issued. Please gate this on an explicit new-session/reset intent, and add a negative test for generic parent creates.
