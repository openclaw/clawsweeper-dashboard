## Summary

Keep open: current main still has the Control UI typed `/new` hook regression, but this PR over-corrects by treating every `sessions.create` request with `parentSessionKey` as a webchat `/new` command.

## What This Changes

The PR exports gateway reset/lifecycle helpers, emits `command:new`, `before_reset`, `session_end`, and `session_start` hooks from `sessions.create` when `parentSessionKey` is present, adds gateway regression tests, and records a changelog fix.

## Best Possible Solution

Restore the Control UI typed `/new` hook contract through an explicit command/source signal while leaving ordinary SDK and programmatic parent-linked session creation as normal session creation.

## Reproduction Assessment

Yes, source-level. Current main routes typed Control UI `/new` into `sessions.create` with `parentSessionKey`, and that handler records the parent but emits none of the documented command or reset/lifecycle hooks.

## Solution Assessment

No, not as currently patched. Emitting reset-style hooks from the gateway is the right boundary, but `parentSessionKey` alone is broader than the documented `command:new` contract and the public SDK create surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate reset hooks on an explicit /new source:** `src/gateway/server-methods/sessions.ts:1008`
  - body: This block runs for any `sessions.create` request that supplies `parentSessionKey`, but that parameter is public SDK/protocol state and is not proof that a webchat `/new` command was issued. Programmatic parent-linked creates will now fire `command:new`, `before_reset`, `session_end`, and `session_start` with `commandSource: "webchat"`, causing session-memory/custom hooks to run for non-command creates. Please carry an explicit Control UI typed-`/new` signal, or otherwise restrict this hook path to real command dispatch.
  - confidence: 0.9
