## Summary

Keep open: the regression is real and the PR is useful, but the current patch still treats any `sessions.create` with `parentSessionKey` as a Control UI `/new` command, which would misfire command/reset hooks for ordinary SDK or programmatic parent-session creation.

## What This Changes

This PR adds command:new, before_reset, session_end, and session_start hook emission around parent-session `sessions.create`, exports the before-reset helper, adds gateway reset-hook tests, and updates the changelog.

## Best Possible Solution

Restore the Control UI typed-`/new` hook contract while leaving generic SDK and programmatic parent-session creation free of command/reset semantics unless it explicitly opts in.

## Reproduction Assessment

Yes, source-level. Current main routes Control UI typed `/new` through `sessions.create` with `parentSessionKey`, and that handler records the parent without emitting the documented command/reset hooks.

## Solution Assessment

No, not as currently patched. Emitting reset-style hooks from the gateway is the right boundary, but `parentSessionKey` alone is too broad because it is a public create parameter used outside the Control UI `/new` command path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87
