## Summary

Keep open: current main has the source-level regression this PR targets, and the PR is directionally right, but the patch emits `before_reset` before `sessions.create` is known to succeed, which can notify reset plugins for a failed create.

## What This Changes

This PR exports the gateway before-reset hook helper, emits `command:new` plus reset lifecycle hooks from `sessions.create` when `parentSessionKey` is present, adds gateway regression tests, and updates the changelog.

## Best Possible Solution

Land a gateway-side fix that restores the successful parent-session create hook contract with focused coverage, including a failure-path test that prevents reset plugin hooks from firing when create fails.

## Reproduction Assessment

Yes, source-level. Current main routes Control UI typed `/new` into `sessions.create` with a `parentSessionKey`, and current `sessions.create` does not emit the documented command/reset lifecycle hook contract there.

## Solution Assessment

No as written. The proposed direction is the right fix boundary, but `before_reset` should move to the successful-create path before the response, with a regression test for rejected parent creates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Move before_reset after create succeeds:** `src/gateway/server-methods/sessions.ts:1036`
  - body: `emitGatewayBeforeResetPluginHook` runs before `applySessionsPatchToStore` and transcript creation have succeeded. Those later steps can reject the request, for example on duplicate or invalid labels, unavailable or invalid models, or transcript-file failure, so a failed `sessions.create` with `parentSessionKey` would still notify `before_reset` plugins even though no new session was created; existing reset coverage keeps `before_reset` quiet on failed reset cleanup. Move this emission to the successful-create path before responding and add a failure-path regression test.
  - confidence: 0.82
