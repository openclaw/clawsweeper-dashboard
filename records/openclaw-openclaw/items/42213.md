## Label Justifications

- `P2`: This is a normal-priority Control UI reliability bug with limited blast radius but real stuck-session and queued-message impact.
- `impact:message-loss`: Queued user messages can remain unsent behind a stale busy state until another prompt or recovery signal arrives.
- `impact:session-state`: The failure is caused by local UI run state drifting from gateway run state after missed terminal or recovery events.

## Summary

Keep open. Current main and the latest release still have a source-reproducible stale `chatRunId` path that can block queued Control UI chat messages, and the same-author closing PR remains open rather than shipped.

## What This Changes

Request automatic Control UI recovery when a stale chat run keeps chat busy and prevents queued messages from flushing after a long run.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent current-main Control UI recovery path that uses authoritative run status and preserves session-row reconciliation.

## Reproduction Assessment

Yes, at source level. Keep `chatRunId` non-null after a missed terminal or session recovery event, submit another normal message, and current main queues it while `flushChatQueue` returns because the UI still considers the host busy.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path that consults the existing `agent.wait` contract is the narrow maintainable fix, but it should land through the linked PR or a current-main replacement before this issue closes.
