## Summary

Keep open. Current main still has a source-reproducible stale `chatRunId` path that blocks queued Control UI chat messages, and the same-author closing PR is open and unmerged.

## What This Changes

The issue requests automatic Control UI recovery and queued-message flushing when a long-running `chat.send` run has already ended but the UI missed its terminal event.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent Control UI `agent.wait` recovery path that clears stale run state and drains the queue only after authoritative terminal results.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after a missed terminal chat event, submit a normal message, and the message queues while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

Yes, directionally. A focused Control UI `agent.wait` recovery path is the narrow maintainable solution, but it should be reviewed through the linked PR or an equivalent replacement branch before this issue closes.
