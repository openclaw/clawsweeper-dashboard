## Summary

Keep open. Current main still has the stale busy-latch path: `chatRunId` keeps the Control UI busy, queued normal messages wait behind that latch, and queue flushing still depends on a matching terminal chat event or reconnect cleanup rather than an `agent.wait` recovery path. The same-author implementation candidate #42258 remains open and uses closing syntax, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for Control UI chat to recover automatically when a long-running `chat.send` run has finished on the gateway but the UI missed the terminal event, so queued messages flush without another manual prompt.

## Best Possible Solution

Review and land #42258, or an equivalent narrow Control UI recovery path that probes `agent.wait` after inactivity and only clears busy state from authoritative terminal results.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: leave `chatRunId` non-null after the gateway-side run is already done, submit a normal message, and the UI enqueues it while `flushChatQueue` returns because the busy latch never clears without a matching terminal event.

## Solution Assessment

Yes. The linked `agent.wait` watchdog direction is the narrowest maintainable fix because the backend already has a wait contract for completed `chat.send` runs and the UI can recover without changing queue semantics or the chat protocol.
