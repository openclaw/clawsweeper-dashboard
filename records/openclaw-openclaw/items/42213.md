- `P2`: This is a normal-priority Control UI reliability bug with limited blast radius but real stuck-session and queued-message impact.
- `impact:message-loss`: Queued user messages can remain unsent behind a stale busy state until another prompt or recovery signal arrives.

## Summary

Keep open. Current main still has a source-reproducible stale `chatRunId` path that can block queue flushing, and the same-author closing PR is open and unmerged rather than shipped.

## What This Changes

The issue reports that Control UI chat can retain a stale busy run id after a long run, causing later user messages to stay queued until another prompt nudges the session.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent current-main Control UI recovery path that uses authoritative run status and preserves existing session-row reconciliation.

## Reproduction Assessment

Yes, at source level. Keep `chatRunId` non-null after a missed terminal or session recovery event, submit another normal message, and current main queues it while `flushChatQueue` returns because the UI still considers the host busy.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path that consults the existing `agent.wait` contract is the narrow maintainable fix, but it should land through the linked PR or a current-main replacement before this issue closes.
