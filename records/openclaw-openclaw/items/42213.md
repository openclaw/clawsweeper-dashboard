## Label Justifications

- `P2`: This is a normal-priority Control UI reliability bug with limited blast radius but real stuck-session and queued-message impact.
- `impact:message-loss`: Queued user messages can remain unsent behind a stale busy state until another manual prompt nudges the session.
- `impact:session-state`: The failure is caused by stale UI run state (`chatRunId`) drifting from the gateway's actual run state.

## Summary

Keep open. Current main still has the source-reproducible stale Control UI chat run path, and the same-author closing PR remains open rather than merged, so the issue should stay tracked until that fix or an equivalent replacement lands.

## What This Changes

The issue reports that Control UI chat can leave `chatRunId` busy after a missed terminal event, preventing queued messages from flushing until the user sends another prompt.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent Control UI `agent.wait` recovery path that clears stale run state and drains queued messages only after authoritative terminal status.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after a missed terminal chat event, submit another message, and the code queues it while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path using the existing `agent.wait` contract is the narrow maintainable fix, but it should land through the linked PR or an equivalent replacement before this issue closes.
