## Summary

Keep open. Current main still has a source-reproducible stale busy latch path, and the same-author implementation candidate at https://github.com/openclaw/openclaw/pull/42258 is still open rather than merged.

## What This Changes

The issue asks the Control UI chat to recover automatically when a stale `chatRunId` keeps queued messages blocked after the gateway-side `chat.send` run has already finished.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent watchdog that uses authoritative `agent.wait` terminal results before clearing busy state and flushing the queue.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after a missed terminal chat event, submit a normal message, and the message is queued while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

Yes. A focused Control UI `agent.wait` recovery path is the narrow maintainable fix because the backend already exposes terminal run state without changing chat protocol or queue semantics.
