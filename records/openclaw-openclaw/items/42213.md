## Summary

Keep open: current main and the latest release still have the stale `chatRunId` busy gate without a Control UI `agent.wait` watchdog, and the open closing PR at https://github.com/openclaw/openclaw/pull/90618 is the canonical fix path.

## What This Changes

The issue reports that Control UI chat can retain a stale active run after a missed terminal event, leaving queued follow-up messages blocked until another user action.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90618 after real Control UI proof, or replace it with an equivalent narrow UI-side recovery that uses the existing `agent.wait` authority before closing this issue.

## Reproduction Assessment

Yes, at source level: current main keeps chat busy while `chatRunId` is set, queues new sends while busy, and only flushes once that state clears. I did not run a live long-running browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes in direction: using the existing gateway `agent.wait` contract to reconcile local Control UI run state is the narrowest maintainable repair. The issue should stay open until the linked implementation lands or maintainers choose an equivalent recovery path.
