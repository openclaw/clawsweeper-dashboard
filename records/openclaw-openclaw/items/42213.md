## Summary

Keep open: current main and v2026.6.6 still have the stale `chatRunId` busy gate without a Control UI `agent.wait` recovery path, and the open closing PR at https://github.com/openclaw/openclaw/pull/90618 is the canonical fix path.

## What This Changes

The issue requests automatic recovery for Control UI chat runs whose stale busy state blocks queued messages after a long run.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90618 once real Control UI proof covers the missed-terminal recovery path, or close that PR and choose an equivalent narrow UI-side recovery before closing this issue.

## Reproduction Assessment

Yes, at source level: current main keeps chat busy while `chatRunId` is set, queues new sends while busy, and only flushes when the busy state clears. I did not run a live long-running browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes in direction: using the existing gateway `agent.wait` authority to reconcile local Control UI run state is the narrow repair. The issue should stay open until the linked implementation lands or maintainers abandon that path.
