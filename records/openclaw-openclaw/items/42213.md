## Summary

Keep open. Current main still has a source-reproducible stale `chatRunId` path that can block queue flushing, and the same-author closing PR remains open, conflicting, and unmerged.

## What This Changes

The issue reports that Control UI chat can stay locally busy after a missed terminal run event, causing later user messages to queue without flushing automatically.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent current-main Control UI recovery path that uses authoritative run status while preserving current session-row reconciliation.

## Reproduction Assessment

Yes, at source level. Keep `chatRunId` non-null after a missed terminal or session recovery event, submit another normal message, and current main queues it while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path that consults the existing `agent.wait` contract is the narrow maintainable fix, but it should land through the linked PR or a rebased current-main replacement before this issue closes.
