## Summary

Keep open: current main and the latest release still have the stale `chatRunId` busy gate without a Control UI `agent.wait` watchdog, and the open same-repo fix candidate already has closing linkage for this issue.

## What This Changes

The issue requests automatic Control UI recovery when a long-running chat run misses its terminal event so queued follow-up messages flush once the gateway-side run is no longer active.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/90618 lands or maintainers replace it with an equivalent missed-terminal recovery design and proof.

## Reproduction Assessment

Yes at source level: keep `chatRunId` non-null after the gateway run is terminal, and current main keeps `isChatBusy` true so `flushChatQueue` will not drain queued sends. I did not run a live browser/gateway repro in this read-only pass.

## Solution Assessment

Yes for triage: the best current path is to keep this issue paired with the open same-repo watchdog PR rather than queueing a duplicate repair. The proposed `agent.wait` watchdog is a bounded fit for missed terminal events, but it still needs merge review and proof on the PR.
