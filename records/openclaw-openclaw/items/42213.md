## Label Justifications

- `P2`: This is a normal-priority Control UI reliability bug with limited blast radius but real stuck-session and queued-message impact.
- `impact:message-loss`: Queued user messages can remain unsent behind a stale busy state until another manual prompt or recovery signal nudges the session.
- `impact:session-state`: The failure is caused by local UI run state drifting from gateway run state after missed terminal or recovery events.

## Summary

Keep open. Current main still has a source-reproducible stale `chatRunId` path that blocks queue flushing, and the same-author closing PR is open rather than merged.

## What This Changes

The issue asks the Control UI chat to recover from stale busy state and automatically flush queued messages after long `chat.send` runs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent current-main Control UI recovery path that uses authoritative run status and preserves session-row cleanup.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after missed terminal or session recovery events, submit another message, and current main queues it while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path using the existing `agent.wait` contract is the narrow maintainable fix, but it should land through the linked PR or an equivalent replacement before this issue closes.
