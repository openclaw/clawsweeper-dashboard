## Label Justifications

- `P2`: The issue is a normal-priority Control UI reliability bug with bounded session/message impact rather than a core runtime outage.
- `impact:session-state`: The stale `chatRunId` leaves the Control UI in an incorrect busy state after the backend run is no longer active.
- `impact:message-loss`: Queued follow-up messages can remain unflushed behind the stale busy state until another manual prompt is sent.

## Summary

Keep open: current main and the latest release still block queued sends while `chatRunId` is non-null and do not contain the watchdog recovery path, while https://github.com/openclaw/openclaw/pull/90618 is the open candidate fix with closing linkage.

## What This Changes

The issue reports that Control UI chat can stay busy after a long-running `chat.send` run misses its terminal event, leaving follow-up messages queued until another manual prompt unsticks the session.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/90618 lands as the replacement fix or a maintainer chooses a different Control UI recovery path.
