The issue asks for Control UI chat to recover from stale busy runs and flush queued messages once the gateway run is no longer active.

Yes, at source level: leave `chatRunId` non-null after a missed terminal event, submit another message, and current main queues it while `flushChatQueue` returns because `isChatBusy` remains true. I did not run a live browser/gateway reproduction in this read-only review.

## Solution Assessment

Yes, likely: a UI-side `agent.wait` watchdog is a bounded fix at the local state boundary. The remaining path is to prove and review the existing open PR rather than create another repair branch.
