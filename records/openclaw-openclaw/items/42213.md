## Summary

Keep open. Current main still has the reported stale-busy shape in the Control UI: a non-null `chatRunId` keeps the composer busy, normal follow-ups enqueue behind it, queue flushing exits while busy, and there is no UI-side `agent.wait` recovery path for a missed terminal chat event. The same-author implementation candidate #42258 is still open and unmerged, so this issue should stay open until that PR, or an equivalent fix, is resolved.

## What This Changes

The issue requests automatic Control UI chat recovery when a stale `chatRunId` leaves queued messages blocked after a long-running `chat.send` run misses its terminal event.

## Best Possible Solution

Review and land #42258 or an equivalent narrow Control UI recovery that probes `agent.wait` for the active run after inactivity, clears busy state only on authoritative terminal results, and includes regression coverage for stale-finished recovery, still-active timeout behavior, watchdog cancellation/reset, and queued-message flushing.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: leave `host.chatRunId` non-null, submit another normal message, and the current UI enqueues it while `flushChatQueue` returns because `isChatBusy` remains true; without a matching terminal chat event or UI `agent.wait` recovery, that queue does not flush.

## Solution Assessment

Yes for the requested direction, pending PR review. A client-side watchdog that uses the existing backend `agent.wait` contract is the narrowest maintainable fix because it recovers the UI without changing backend chat protocol or queue semantics.
