## Summary

Keep open. Current main and the latest release still have a source-reproducible stale `chatRunId` path that can block queue flushing, and the same-author closing PR is open and unmerged rather than implemented on main.

## What This Changes

The issue reports that Control UI chat can remain busy after a long run, causing later user messages to queue without flushing until another prompt or recovery signal arrives.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent current-main Control UI recovery path that uses authoritative run status and preserves existing session reconciliation.

## Reproduction Assessment

Yes, at source level. Keep `chatRunId` non-null after a missed terminal or session recovery event, submit another normal message, and current main queues it while `flushChatQueue` returns because the UI still considers the host busy.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path that consults the existing `agent.wait` contract is the narrow maintainable fix, but it should land through the linked PR or a current-main replacement before this issue closes.
