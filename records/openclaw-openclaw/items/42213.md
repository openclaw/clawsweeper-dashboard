## Label Justifications

- `P2`: This is a normal-priority Control UI reliability bug with limited blast radius but real stuck-session and queued-message impact.
- `impact:message-loss`: Queued user messages can remain unsent behind a stale busy state until another manual prompt or recovery signal nudges the session.
- `impact:session-state`: The failure is caused by local UI run state drifting from gateway run state after missed terminal or recovery events.

## Summary

Keep open. Current main still has a source-reproducible path where a stale local `chatRunId` keeps the Control UI busy and blocks queue flushing, and the same-author fixing PR is still open rather than merged.

## What This Changes

The issue asks for Control UI chat to recover automatically from stale busy run state and flush queued messages after the gateway run has finished.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/42258, or replace it with an equivalent Control UI recovery path that uses authoritative run status and preserves the newer session-row cleanup behavior.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after missed terminal or session recovery events, submit another message, and current main queues it while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

Yes, directionally. A focused Control UI recovery path using the existing `agent.wait` contract is the narrow maintainable fix, but this should land through the linked PR or an equivalent replacement before this issue closes.
