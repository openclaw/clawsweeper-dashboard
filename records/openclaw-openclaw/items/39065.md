## Summary

Keep open: this is a useful security-hardening direction, but the PR is not merge-ready because the new config surface is split across incompatible schema layers and several pairing paths still do not honor the requested silent/code-only modes.

## What This Changes

Adds an optional `unpairedResponse` mode (`silent`, `code-only`, `branded`) to DM pairing config, shared pairing reply helpers, and many channel pairing call sites.

## Best Possible Solution

Rebase on current main, centralize the config contract, wire every advertised channel schema and runtime through one response-mode owner, keep or explicitly decide the branded default, and then add focused tests plus successful live Telegram proof.

## Reproduction Assessment

Yes for the main blockers from source inspection: the PR-head strict schemas omit fields that changed types expose, and several changed `buildPairingReply` calls omit the configured mode. I did not run a live transport reproduction.

## Solution Assessment

No. The PR is a plausible mitigation, but it is not the best merge shape until the config owner is unified, all advertised channel paths are wired, and the security/default tradeoff is explicitly decided.

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the new option into every strict config schema:** `src/config/types.discord.ts:224`
  - body: `unpairedResponse` is now exposed in channel config types/help, but the PR-head strict schemas still omit it for surfaces such as Discord, Slack, LINE, IRC, and BlueBubbles. Users who follow the new config can fail validation/startup instead of getting the advertised behavior, so the field needs one complete schema owner or the unsupported channel exposure should be removed.
  - confidence: 0.91
- **[P1] Pass the configured mode to extension pairing replies:** `extensions/googlechat/src/monitor-access.ts:321-325`
  - body: This changed call still invokes `buildPairingReply` without a `mode`, so it will always use the helper default and send the branded text even when the channel is configured for `silent` or `code-only`. The same pattern appears in multiple extension call sites, so the account/channel config needs to be threaded through before advertising the option.
  - confidence: 0.9
- **[P2] Suppress repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: The `created === false` branch still sends the hard-coded “Pairing already requested” response even when `unpairedResponse` is `silent`. A sender with an existing pending pairing row can therefore still get an automatic DM/component reply, which violates the new suppression contract.
  - confidence: 0.88
- **[P3] Move outbound status updates behind the send guard:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: `statusSink` is updated after the `if (replyText)` guard, so silent mode can record `lastOutboundAt` even when no reply was sent. Keep the status update inside the send block so channel status reflects real outbound delivery.
  - confidence: 0.82
