## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the no-mode reply helper contract as string:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through the plugin/runtime SDK, so changing its return type to `string | null` makes existing plugin callers handle `null` even when they never pass `mode: "silent"`. Preserve the current no-mode/branded return contract with overloads or move silent suppression behind a new helper so external callers do not get a breaking type change.
  - confidence: 0.84
- **[P1] Wire exposed response keys through strict validators:** `src/config/types.discord.ts:224`
  - body: The public Discord, Slack, and LINE config types/runtime paths now expose `unpairedResponse`, but the strict validators on this branch do not accept that field. Operators who add the newly advertised key can fail config validation/startup instead of getting the new behavior.
  - confidence: 0.9
- **[P1] Do not advertise plugin modes without plugin wiring:** `src/config/schema.help.ts:1498-1499`
  - body: The help/labels advertise `channels.bluebubbles.unpairedResponse` and the old core schema adds IRC, but the plugin-local schemas do not accept the key and their pairing reply paths still call `buildPairingReply` without a mode. That means the setting is either rejected or ignored for those channels.
  - confidence: 0.88
- **[P2] Suppress repeat Discord pairing replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: In the Discord component path, `mode: unpairedResponse` is only used for newly created pairing requests. When a request already exists, the hardcoded repeat message is still sent, so `silent` mode continues to fingerprint the bot for repeat attempts.
  - confidence: 0.91
- **[P3] Move the status update inside the send guard:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: Nextcloud Talk updates `lastOutboundAt` even when `replyText` is null and no message was sent. Keep that status update inside the `if (replyText)` block so silent mode does not record a fake outbound send.
  - confidence: 0.78
