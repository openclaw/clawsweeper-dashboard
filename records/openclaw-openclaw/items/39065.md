Full review comments:

- **[P1] Add the response key to every strict schema:** `src/config/types.discord.ts:223-224`
  - body: `unpairedResponse` is exposed as public channel config here and in help/label metadata, but the strict Discord, Slack, and LINE schemas on this branch still do not accept it. Users following the new config would fail validation before the runtime can honor the setting, so either wire the strict schemas and tests for every advertised channel or remove those advertised surfaces.
  - confidence: 0.92
- **[P2] Pass the response mode into plugin reply builders:** `extensions/irc/src/inbound.ts:217-221`
  - body: This IRC path now guards a nullable reply, but it still calls `buildPairingReply` without a mode, and BlueBubbles has the same shape. Because the PR advertises `channels.irc.unpairedResponse` and `channels.bluebubbles.unpairedResponse`, `silent` and `code-only` are ignored and the default branded reply is still sent.
  - confidence: 0.9
- **[P2] Suppress pending component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: For Discord component interactions, the configured mode is used only when a pairing row is newly created. If the sender already has a pending pairing request, `silent` mode still receives the hardcoded `Pairing already requested` reply, so the same unpaired user can still fingerprint the bot through this DM surface.
  - confidence: 0.91
- **[P3] Record outbound status only after sending:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: `lastOutboundAt` is updated even when `buildPairingReply` returns `null` and no Nextcloud Talk message is sent. That makes silent mode look like a successful outbound send and diverges from the Google Chat path, which records status only inside the send guard.
  - confidence: 0.84
