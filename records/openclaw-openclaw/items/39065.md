## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the plugin-facing reply helper contract:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through the plugin SDK/runtime, so changing it from `string` to `string | null` can break existing plugin consumers that pass the result directly to send APIs. Keep this helper string-returning and add a separate response-mode helper/result type, or provide an explicit compatibility/deprecation path with SDK tests.
  - confidence: 0.88
- **[P1] Wire the advertised config through strict schemas:** `src/config/types.discord.ts:224`
  - body: This exposes `unpairedResponse` in public config types/help for channels such as Discord, Slack, and LINE, but their strict schemas still do not accept the key; IRC and BlueBubbles also have plugin-local schemas/runtime paths that do not fully participate. Users who set the documented key can fail validation or get the default branded reply instead of the configured mode.
  - confidence: 0.9
- **[P2] Suppress the repeat-pairing Discord reply in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When a Discord component interaction hits an already-pending pairing request, this branch still sends the hardcoded `Pairing already requested` response even if `unpairedResponse` is `silent`. Gate the repeat path on the same mode or route it through the shared Discord DM decision helper.
  - confidence: 0.92
- **[P2] Only mark outbound activity after sending a reply:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: `statusSink` is updated outside the `if (replyText)` guard, so Nextcloud Talk records `lastOutboundAt` even when silent mode sends no message. Move the status update inside the send guard to keep channel activity state accurate.
  - confidence: 0.84
