- **[P1] Align the new config key with validators:** `src/config/types.discord.ts:224`
  - body: This exposes `unpairedResponse` on the Discord config type, but the strict Discord account schema in `src/config/zod-schema.providers-core.ts` still has no matching field; Slack and LINE have the same type/runtime exposure without strict schema acceptance.   - confidence: 0.9
- **[P1] Pass the response mode into plugin callers:** `extensions/bluebubbles/src/monitor-processing.ts:606-610`
  - body: BlueBubbles gets a schema-level `unpairedResponse` field, but this runtime call still invokes `buildPairingReply` without `mode`, so `silent` and `code-only` are ignored and the default branded reply is sent. Wire the parsed channel/account setting into the pairing helper for BlueBubbles and the sibling plugin callers such as IRC, or do not expose the config there.
  - confidence: 0.88
- **[P2] Suppress repeat Discord component replies:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When a Discord component user already has a pending pairing request, this branch always sets the fallback text even if `unpairedResponse` is `silent`. That means repeat attempts still disclose the bot and pairing state, so gate this fallback on the mode or route the component path through the shared DM decision helper.
  - confidence: 0.92
