## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the pairing helper's string contract:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exposed through `core.channel.pairing`, so changing its general return type to `string | null` can break plugin callers that use it as a string formatter without opting into silent mode. Keep the existing string contract for ordinary calls, use overloads, or move nullable behavior behind a narrower response-mode API.
  - confidence: 0.88
- **[P1] Wire the advertised config through strict schemas:** `src/config/types.discord.ts:224`
  - body: This adds `unpairedResponse` to public config types, but strict schemas for Discord, Slack, LINE, IRC, and BlueBubbles do not all accept the key, and IRC/BlueBubbles still do not pass the mode to the reply builder. Users following the new config can fail validation or keep receiving branded replies.
  - confidence: 0.9
- **[P2] Suppress repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When a pairing row already exists, this branch skips `buildPairingReply(..., mode)` and always sends `Pairing already requested...`, so `unpairedResponse: "silent"` still emits an identifying Discord response on repeat component interactions.
  - confidence: 0.87
- **[P3] Record outbound activity only after sending:** `extensions/nextcloud-talk/src/inbound.ts:192`
  - body: `statusSink` is updated even when `replyText` is null, so silent mode can mark Nextcloud Talk as having sent an outbound reply that was intentionally suppressed. Move the activity update inside the `if (replyText)` block.
  - confidence: 0.78
