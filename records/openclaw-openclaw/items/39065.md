## What This Changes

The PR adds optional `unpairedResponse` modes (`silent`, `code-only`, `branded`) for DM pairing replies and wires them through pairing helpers, config types/help, tests, and many channel implementations.

## Best Possible Solution

Keep the PR open and repair it into one coherent response-mode contract with aligned schemas, docs/help, runtime call sites, and real transport proof, or replace it with a narrower maintainer-approved implementation tied to the open security issue.

## Solution Assessment

No. The response-mode idea is useful, but the implementation spreads ownership across types, zod schemas, plugin schemas, help/labels, and channel call sites, so a shared contract or narrower fully-wired scope is safer.
