Full review comments:

- **[P1] Suppress repeat component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When `unpairedResponse` is `silent`, this branch still sends the hardcoded "Pairing already requested" reply for users with an existing pending pairing row because the mode is only applied on the `created` path. That breaks the advertised silent privacy posture for Discord component DMs, so gate the repeat message on the same mode or route this through the shared Discord DM pairing decision helper.
  - confidence: 0.94
- **[P1] Wire the advertised config through strict schemas:** `src/config/types.discord.ts:223-224`
  - body: This exposes `unpairedResponse` on the Discord config type/help surface, but the strict Discord and Slack validators are still missing the key, and LINE has the same schema gap in review context. Operators who configure the documented key can therefore fail validation instead of getting the new behavior; add the field to every strict owner or remove the advertised surface until it is fully supported.
  - confidence: 0.9
- **[P2] Pass the selected mode into plugin pairing replies:** `extensions/bluebubbles/src/monitor-processing.ts:606-610`
  - body: The BlueBubbles path now guards a nullable reply, but it still calls `buildPairingReply` without the configured mode, so `silent` and `code-only` would keep producing the default branded reply even after schema validation is fixed. IRC has the same pattern, so these plugin paths need to pass the configured response mode or stop advertising support for them.
  - confidence: 0.88
- **[P3] Only mark outbound activity after a reply is sent:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: In silent mode `replyText` is null, but `lastOutboundAt` is still updated after the guarded send block. Move this status update inside the `if (replyText)` branch so channel status does not claim an outbound message was sent when the PR intentionally suppressed it.
  - confidence: 0.86
