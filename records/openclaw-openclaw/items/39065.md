  - body: `code-only` still appends the `openclaw pairing approve ...` command, so the supposedly de-branded mode continues fingerprinting OpenClaw to unknown senders. Keep the CLI hint in `branded` only, or rename/redefine the mode explicitly before exposing it.

  - body: This adds `unpairedResponse` to the Discord config type, but the strict Discord schema still omits the key, and Slack has the same type/schema split.   - confidence: 0.88
- **[P2] Register LINE unpairedResponse in the schema:** `src/line/types.ts:25`
  - body: The LINE runtime now reads `context.account.config.unpairedResponse`, but the LINE config schema remains strict without this property. Add the field to the schema/default path or remove the runtime/type exposure so the setting is actually usable.
  - confidence: 0.86
- **[P2] Suppress Discord repeat pairing replies in silent mode:** `src/discord/monitor/agent-components.ts:540`
  - body: For an already-pending pairing request, `replyText` becomes a hard-coded message even when `unpairedResponse` is `silent`, so repeat component interactions still answer unknown users.   - confidence: 0.92
- **[P2] Do not mark silent replies as outbound:** `extensions/nextcloud-talk/src/inbound.ts:192`
  - body: When `replyText` is null, no Nextcloud-Talk reply is sent, but `lastOutboundAt` is still updated. Move the status update inside the `if (replyText)` block so silent mode does not look like a real outbound send.
  - confidence: 0.82

  - body: IRC still calls `buildPairingReply` without a mode, and its plugin-local schema does not expose `unpairedResponse`, so `silent` and `code-only` cannot take effect for this advertised channel. Wire the config through the plugin schema/account object and pass `mode`, or drop the field for unsupported channels.
  - confidence: 0.84
