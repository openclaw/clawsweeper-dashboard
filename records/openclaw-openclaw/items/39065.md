  - body: `buildPairingReply` is exposed through the plugin runtime, so changing its return type to `string | null` makes existing plugin callers handle a nullable value even when they only use the old branded behavior.   - confidence: 0.9
- **[P1] Add the option to every strict config schema:** `src/config/types.discord.ts:223-224`
  - body: The PR exposes `unpairedResponse` in Discord, Slack, and LINE config types/help, but the corresponding strict validators still omit the field. Users who configure the advertised key for those channels will fail validation instead of getting silent or code-only pairing replies.
  - confidence: 0.89
- **[P1] Wire plugin-local schemas and runtimes before advertising modes:** `src/config/schema.help.ts:1498-1499`
  - body: The help text advertises `channels.bluebubbles.unpairedResponse`, and core schema adds IRC/BlueBubbles fields, but the plugin-local schemas/runtime paths do not consistently accept the key or pass `mode` into `buildPairingReply`. Those channels can reject, ignore, or keep sending branded replies despite the new documented setting.
  - confidence: 0.86
- **[P2] Suppress repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When a Discord DM component user already has a pending pairing request, this branch still returns the hardcoded "Pairing already requested" response instead of honoring `unpairedResponse: "silent"`. That leaves the same sender state behaving differently depending on which Discord DM surface they hit.
  - confidence: 0.91
- **[P2] Record outbound status only after sending a reply:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: This hunk guards the Nextcloud reply send when `replyText` is falsy, but `lastOutboundAt` is still updated immediately afterward. Under the new nullable reply contract, silent mode would record outbound activity even when no message was sent.
  - confidence: 0.74
