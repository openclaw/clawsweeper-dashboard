## Summary

Keep this PR open for maintainer review: the privacy/security direction is useful and tied to open reports, but the branch is conflicting with current main and still has confirmed config-contract and channel-behavior defects that can leave the leak active or make the new option unusable.

## What This Changes

The PR adds an optional `unpairedResponse` DM pairing reply mode (`silent`, `code-only`, `branded`) and threads it through shared pairing helpers plus many channel call sites.

## Best Possible Solution

Update the current pluginized channel surfaces with one consistent config/schema/runtime contract for unpaired pairing replies, then prove fresh config, upgrade behavior, and representative live DM delivery.

## Reproduction Assessment

Yes for the branch defects by source inspection: the Discord component path hardcodes a repeat-pairing reply, and several added config fields are not accepted by matching strict schemas or are not passed to `buildPairingReply`. I did not run live channel delivery in this read-only review.

## Solution Assessment

No. The direction is useful, but the best fix is to put the mode behind one canonical channel config/schema contract and route every DM pairing surface through the shared helper on current main's pluginized channel paths.

- **[P1] Suppress repeat pairing replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When `unpairedResponse` is `silent`, this path still sends the hardcoded repeat-pairing message for users with an existing pending pairing row.   - confidence: 0.95
- **[P1] Wire the advertised config key through strict schemas:** `src/config/types.discord.ts:223-224`
  - body: This adds `unpairedResponse` to public config types, but the strict validators for channels like Discord, Slack, and LINE do not accept the field in the PR head. Operators who configure the new key can fail startup or reload instead of getting the advertised response mode, so every exposed channel needs matching schema coverage or the exposed field should be removed.
  - confidence: 0.91
- **[P2] Pass the mode where IRC and BlueBubbles advertise it:** `src/config/zod-schema.providers-core.ts:1081`
  - body: The PR exposes `unpairedResponse` for IRC, but the plugin-local schema lacks the key and the IRC/BlueBubbles runtime calls still invoke `buildPairingReply` without `mode`. That means `silent` and `code-only` are either rejected or ignored for those advertised channels.
  - confidence: 0.9
- **[P3] Move the outbound timestamp inside the send guard:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: In silent mode `replyText` is null and no message is sent, but this path still records `lastOutboundAt`. Move the status update inside the `if (replyText)` block so channel activity does not claim an outbound send that did not happen.
  - confidence: 0.82
