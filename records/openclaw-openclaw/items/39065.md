- **[P1] Preserve the string-returning pairing helper:** `src/pairing/pairing-messages.ts:10-13`
  - body: `buildPairingReply` is exported through plugin runtime and SDK surfaces, so changing it to return `null` in silent mode makes existing callers handle a new nullable formatter contract.   - confidence: 0.9

  - body: This adds `unpairedResponse` to visible config/type/help surfaces, but strict owning schemas and runtime call sites are not consistently updated, so users can configure an advertised key and get validation failures or unchanged branded replies. Every advertised channel needs one schema/runtime owner or the field should be removed for unsupported channels.
  - confidence: 0.88
- **[P1] Refresh Discord changes onto plugin-owned paths:** `src/discord/monitor/native-command.ts:1372-1403`
  - body: The branch still edits `src/discord/monitor/**`, but current main has moved Discord runtime code under `extensions/discord/**`. These changes need to be reapplied to the current plugin-owned DM preflight, native-command, and component-auth paths before the PR can prove the behavior it intends to change.
  - confidence: 0.93
- **[P2] Suppress repeat component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: On the component path, `silent` only suppresses newly created pairing replies; repeat attempts still get the hardcoded `Pairing already requested` response. That keeps a visible bot fingerprint for pending users even when the configured mode says no reply should be sent.
  - confidence: 0.86
- **[P3] Record outbound activity only after a send:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: `statusSink` is updated even when `replyText` is absent and no Nextcloud Talk message was sent. Move the activity update inside the send guard so silent mode does not look like an outbound reply happened.
  - confidence: 0.8
