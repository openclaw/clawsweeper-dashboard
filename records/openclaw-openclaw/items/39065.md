  - body: `DiscordAccountConfig` now exposes `unpairedResponse`, and Slack/LINE have the same pattern, but their strict zod schemas on this branch do not declare the key.   - confidence: 0.9
- **[P2] Pass the configured mode through plugin reply paths:** `extensions/irc/src/inbound.ts:217-221`
  - body: The branch accepts `unpairedResponse` in core schema for IRC and BlueBubbles, but the plugin-local schemas omit it and these runtime calls still call `buildPairingReply` without a mode. That makes `silent` and `code-only` rejected or ignored for advertised channels, leaving the branded reply in place.
  - confidence: 0.88
- **[P2] Suppress repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When the pairing request already exists, this path unconditionally returns the hardcoded “Pairing already requested” reply. With `channels.discord.unpairedResponse="silent"`, repeat component interactions still produce an observable DM response, so silent mode does not actually suppress pairing replies for this Discord surface.
  - confidence: 0.92
- **[P3] Only mark outbound activity after a reply is sent:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: `statusSink` still records `lastOutboundAt` after `buildPairingReply` returns null in silent mode. That reports an outbound send that did not happen and can skew channel status or health decisions; move the status update inside the `if (replyText)` block.
