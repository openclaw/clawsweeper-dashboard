Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the pairing formatter SDK contract:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through the plugin runtime and SDK subpaths, so changing the formatter's return type to `string | null` breaks existing plugin callers that use it as a string formatter without opting into silent mode. Keep this API string-returning and put nullable suppression behind an explicit challenge/response-mode API or overload.
  - confidence: 0.89
- **[P1] Wire the new key into strict schemas:** `src/config/types.discord.ts:224`
  - body: This adds `unpairedResponse` to public Discord/Slack/LINE types and runtime reads, but their strict zod schemas still do not accept the key. An operator who configures the documented field can fail validation before the new behavior is usable, so the schemas/defaults need to be added or the advertised fields removed.
  - confidence: 0.9
- **[P2] Suppress repeat Discord component replies:** `src/discord/monitor/agent-components.ts:533-540`
  - body: In silent mode the `created` branch returns `null`, but the already-pending branch still sends `"Pairing already requested..."` to the unpaired user. That keeps the fingerprinting reply for repeat component interactions, so this branch should share the same silent-mode guard or reuse the common Discord DM decision helper.
  - confidence: 0.91
- **[P2] Pass response mode to advertised plugin channels:** `extensions/bluebubbles/src/monitor-processing.ts:606-610`
  - body: BlueBubbles is advertised with `channels.bluebubbles.unpairedResponse`, but this pairing reply call does not pass any mode to `buildPairingReply`, so `silent` and `code-only` still behave as branded. Wire the parsed account config through this path, and check the same pattern in IRC before advertising those keys.
  - confidence: 0.84
