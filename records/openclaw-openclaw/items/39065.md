## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve the plugin-facing reply helper contract:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through plugin runtime surfaces, so changing it from `string` to `string | null` can break plugin consumers that pass the result directly to send APIs. Keep this helper string-returning and add a separate response-mode helper/result type, or provide an explicit compatibility path with SDK tests.
  - confidence: 0.9
- **[P1] Wire unpairedResponse through every strict config owner:** `src/config/types.discord.ts:224`
  - body: This exposes `unpairedResponse` in public config types/help for channels such as Discord, Slack, and LINE, but strict schemas and plugin-local schemas do not all accept the key. Users who set the documented key can fail validation or get the default branded reply instead of the configured mode.
  - confidence: 0.9
- **[P2] Suppress repeat Discord replies in silent mode:** `src/discord/monitor/agent-components.ts:540`
  - body: When a Discord component interaction hits an already-pending pairing request, this branch still sends the hardcoded pending-pairing response even if `unpairedResponse` is `silent`. Gate the repeat path on the same mode or route it through the shared Discord DM decision helper.
  - confidence: 0.92
- **[P2] Only mark outbound activity after sending a reply:** `extensions/nextcloud-talk/src/inbound.ts:192`
  - body: `statusSink` is updated after a nullable pairing reply even when `replyText` is empty, so silent mode can record outbound activity that never happened. Move the status update inside the send guard anywhere this pattern remains.
  - confidence: 0.84
