  - body: `buildPairingReply` is exported through plugin/runtime surfaces, so changing it to return `null` for silent mode can break callers that treat it as a string formatter. Keep this helper string-returning and model suppression at the call site or through a deliberately versioned API.

  - body: This adds `unpairedResponse` to public channel types/help, but strict owning schemas still omit it for Discord, Slack, LINE, and some plugin-local surfaces. Users can hit startup validation failures or unchanged branded replies when trying the newly advertised mitigation.
  - confidence: 0.86
- **[P1] Refresh Discord changes onto plugin-owned paths:** `src/discord/monitor/native-command.ts:1372-1403`
  - body: Current main moved Discord runtime ownership to `extensions/discord/**`, but this branch still changes removed `src/discord/monitor/**` files. The Discord behavior must be ported to the current plugin-owned paths before the fix can affect the real runtime.
  - confidence: 0.9
- **[P2] Suppress repeat component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When the pairing request already exists, this branch still uses the hardcoded repeat message instead of honoring `unpairedResponse: "silent"`. That lets repeat component interactions continue fingerprinting the bot even after the operator selects silent mode.
