## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve the pairing helper's SDK contract:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through plugin runtime surfaces, so widening its general return type to `string | null` can break plugin callers that use it as a string formatter without opting into silent mode. Keep the existing string contract for ordinary calls, use overloads, or move the nullable behavior behind a narrower response-mode API.
  - confidence: 0.84
- **[P1] Register the new option in every strict schema:** `src/config/types.discord.ts:224`
  - body: This exposes `unpairedResponse` on public config types, but strict validators for Discord, Slack, and LINE still omit the key, while IRC and BlueBubbles also have plugin-local schema/runtime gaps. Users configuring the advertised option can fail startup or get branded replies where silent/code-only was requested.
  - confidence: 0.9
- **[P2] Gate repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: For Discord component DMs, `silent` only suppresses the newly-created pairing reply; repeat pending requests still send the hardcoded "Pairing already requested" response. Apply the same response-mode decision to the repeat path or route this through the shared DM decision helper.
  - confidence: 0.88
- **[P3] Only mark outbound status after sending:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: Nextcloud Talk updates `lastOutboundAt` even when `buildPairingReply` returns no text, so silent mode records outbound activity without actually sending a message. Move the status update inside the `if (replyText)` branch, matching the Google Chat path.
  - confidence: 0.77
