Overall confidence: 0.9

Full review comments:

- **[P1] Keep the pairing formatter string-safe for SDK callers:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through current main's plugin SDK runtime subpath, so changing the general formatter to return `string | null` can break callers that use it as a string formatter without opting into silent mode. Keep the existing string contract with overloads or move nullable behavior behind a narrower helper before rebasing this branch.
  - confidence: 0.84
- **[P1] Wire strict schemas before exposing the config key:** `src/config/types.discord.ts:224`
  - body: This branch exposes `unpairedResponse` in public config types/help, but strict schemas such as Discord, Slack, and LINE still reject the field. An operator following the new config surface can fail validation/startup instead of getting the advertised response mode, so add the schema fields and tests or remove the exposed key for those channels.
  - confidence: 0.91
- **[P2] Pass the response mode through plugin call sites:** `extensions/irc/src/inbound.ts:217-221`
  - body: IRC and BlueBubbles add nullable guards around `buildPairingReply`, but their local schemas do not accept `unpairedResponse` and these call sites never pass a mode, so configured `silent` or `code-only` behavior cannot work there. Either fully wire the local schema/runtime path or stop advertising the key for those plugins.
  - confidence: 0.89
- **[P2] Suppress repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:533-540`
  - body: When a pairing request already exists, this path ignores `unpairedResponse` and sends the hardcoded "Pairing already requested" response. That breaks the `silent` security posture for repeat DM component interactions, so gate this fallback on the mode or route the path through the shared Discord DM decision helper.
  - confidence: 0.93
- **[P3] Only record outbound status when a reply is sent:** `extensions/nextcloud-talk/src/inbound.ts:187-192`
  - body: `statusSink` still updates after `replyText` is skipped, so silent mode records `lastOutboundAt` even though no Nextcloud message was delivered. Move the status update inside the send guard to keep delivery state truthful.
  - confidence: 0.87
