  - body: `buildPairingReply` is exported through the plugin runtime/SDK, so changing it from always returning `string` to returning `string | null` can break existing plugin callers that pass its result directly to a send API.   - confidence: 0.86

  - body: The PR exposes `unpairedResponse` on Discord/Slack/LINE type surfaces, but the strict validators shown in the diff do not add those keys, so users can configure the documented option and fail validation or have a channel ignore it. Add the field to every owning schema or remove the exposed type/help surface until the runtime can honor it.
  - confidence: 0.9
- **[P1] Suppress repeat Discord component replies in silent mode:** `src/discord/monitor/agent-components.ts:530-539`
  - body: When `created` is false, this path still sends `Pairing already requested...` even if `unpairedResponse` is `silent`, so repeat component interactions leak that a bot/pairing flow exists. Gate the fallback on the mode or route this through the shared Discord DM decision helper.
  - confidence: 0.92
