## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve the pairing formatter's string contract:** `src/pairing/pairing-messages.ts:10`
  - body: `buildPairingReply` is exported through the plugin runtime pairing surface, so changing its general return type to `string | null` can break plugin callers that use it as a plain string formatter. Keep the existing formatter string-only and put silent-mode behavior behind a narrower helper or overload that callers opt into.
  - confidence: 0.85
- **[P1] Register the new option in every strict schema:** `src/config/types.discord.ts:224`
  - body: This advertises `unpairedResponse` on channel config types, but the strict schemas for Discord, Slack, and LINE still omit that property. Operators who configure the advertised key can fail validation instead of getting the promised behavior, so the schema source each channel uses needs to accept the field or the advertised surface should be removed.
  - confidence: 0.89
- **[P2] Suppress repeat Discord component replies:** `src/discord/monitor/agent-components.ts:540`
  - body: When `unpairedResponse` is `silent` and a Discord component user already has a pending pairing request, this fallback still sends `Pairing already requested...`. That leaks the same bot/pairing signal silent mode is meant to suppress, so gate the repeat message on the mode or route the component path through the shared Discord DM decision helper.
  - confidence: 0.9
- **[P3] Record outbound status only after a reply is sent:** `extensions/nextcloud-talk/src/inbound.ts:192`
  - body: In silent mode `replyText` is falsy and no Nextcloud Talk message is sent, but `lastOutboundAt` is still updated after the guard. Move the status update inside the send block so channel status reflects actual outbound delivery.
  - confidence: 0.86
