## What This Changes

The branch adds an optional `unpairedResponse` mode (`silent`, `code-only`, `branded`) for unpaired DM pairing replies across shared pairing helpers, config surfaces, tests, and channel call sites.

## Best Possible Solution

Rebase onto current plugin paths, centralize a response-mode or denial-response contract without breaking the public pairing helper API, and prove default, silent, and code-only behavior across representative channels before merge.

## Reproduction Assessment

Yes for the PR blockers. Diff and source inspection show the nullable public helper contract, missing schema/runtime participation, and the Discord repeat path that still sends a reply; I did not run live channel delivery in this read-only review.

## Solution Assessment

No. The response-mode direction is plausible, but this patch is stale and fragmented; the safer solution is a rebased, single-owner contract with explicit maintainer approval for the default posture and plugin API compatibility.
