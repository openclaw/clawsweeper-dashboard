## Summary

Keep open: the UX goal is useful, but the PR is not merge-ready because it adds ordering and persistence to a stale helper path instead of the active Control UI picker path, has model-choice correctness issues, and still lacks real behavior proof.

## What This Changes

The PR adds browser-local recent chat-model storage, recent/custom-provider ordering, successful-selection persistence, and related tests for the Gateway/Control UI model picker.

## Best Possible Solution

Port the UX into the active picker modules, keep Default pinned, rank resolvable recents before true custom-provider entries, preserve catalog order for the rest, and persist only successful non-default canonical selections best-effort.

## Reproduction Assessment

Yes. Source inspection against current main shows the visible select is rendered and switched through ui/src/ui/chat-model-select-state.ts and ui/src/ui/chat/session-controls.ts, while the PR adds the behavior to ui/src/ui/app-render.helpers.ts.

## Solution Assessment

No. The requested UX is reasonable, but the maintainable fix should be rebased into the active picker path and should derive true custom-provider status from current provider/catalog metadata rather than a stale hard-coded allowlist.
