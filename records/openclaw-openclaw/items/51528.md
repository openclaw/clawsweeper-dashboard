## Summary

Keep open. The feature is useful and still absent from current main, but this PR is not merge-ready because it wires ranking and persistence into the older app-render helper path while the active Control UI picker now renders and switches through the split chat session controls.

## What This Changes

The PR branch adds browser-local recent chat model storage, sorts picker options as recent models then custom-provider models then remaining models, persists successful selections, and adds UI unit coverage.

## Best Possible Solution

Land a narrow port in the current Control UI architecture: keep Default pinned, rank resolvable recent selections first, rank truly custom-provider entries before remaining configured/catalog entries, ignore stale recents, and persist only successful non-default selections as canonical resolved model refs with best-effort storage access.

## Reproduction Assessment

Yes. A focused unit check against resolveChatModelSelectState with a catalog containing recent/custom candidates would show current main still returns catalog order, and the active switchChatModel source shows no successful-selection MRU persistence path.

## Solution Assessment

No, not as-is. The intended behavior is narrow and maintainable, but the PR should move the implementation from app-render.helpers.ts into chat-model-select-state.ts and chat/session-controls.ts so it affects the live picker.
