## Summary

Keep open: the proposed UX is plausible, but the branch is currently incorrect because it applies the model ordering and persistence to an older helper path instead of the active Control UI picker modules, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The branch adds browser-local recent chat-model storage, custom-provider-aware model ordering, and tests for the gateway/control UI model picker.

## Best Possible Solution

Port the UX into ui/src/ui/chat-model-select-state.ts and ui/src/ui/chat/session-controls.ts: keep Default pinned, rank resolvable recents first, then true custom-provider entries, then catalog order, and persist only successful non-default canonical selections best-effort with current-path tests.

## Reproduction Assessment

Yes for the PR defects by source inspection. Current main renders and switches the picker through ui/src/ui/chat/session-controls.ts and ui/src/ui/chat-model-select-state.ts, while this PR changes app-render.helpers.ts and does not update the active path.

## Solution Assessment

No, not as-is. The intended UX is narrow, but the implementation should move to the active picker modules, skip Default MRU writes, and classify bundled providers before custom sorting.

  - confidence: 0.9
- **[P2] Skip recording Default selections as recents:** `ui/src/ui/app-render.helpers.ts:676`
  - body: rememberResolvedChatModelRecent runs after every sessions.patch, including when nextModel is empty for Default. The gateway resolves that back to the default concrete model, so clearing an override would pollute the MRU list with a model the user did not explicitly select.
  - confidence: 0.87

  - body: The built-in provider set omits bundled providers currently present on main, including byteplus, volcengine, google-gemini-cli, and google-vertex. Those first-party catalog entries would be floated as custom providers even without user-configured custom providers.
  - confidence: 0.87
