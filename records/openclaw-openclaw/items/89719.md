## Summary

Keep open: the linked cron retention bug is valid, but this PR’s one-line reaper gate is too broad and would make every stale cron-prefixed session key eligible for deletion, not just disposable isolated or orphaned base sessions.

## What This Changes

The PR imports `isCronSessionKey` and lets the cron session reaper prune stale entries matching either cron run keys or any agent-scoped cron key.

## Best Possible Solution

Refresh or replace this with a guarded reaper/timer fix that prunes stale isolated or orphaned base cron keys while preserving known persistent, named, and main cron session state plus transcript archival safeguards.

## Reproduction Assessment

Yes. Source inspection shows isolated cron jobs resolve to base `agent:<id>:cron:<jobId>` keys while current main’s reaper only accepts `isCronRunSessionKey`; the linked issue also includes live OpenClaw reproduction evidence.

## Solution Assessment

No. The broad `isCronSessionKey` gate is only a plausible fix; the maintainable fix needs to distinguish disposable isolated or orphaned base keys from durable main or named cron session state.
