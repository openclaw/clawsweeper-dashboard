## Summary

Keep open: the PR targets a valid cron retention bug, but the one-line predicate broadening is not safe enough to merge because it makes every stale cron-prefixed session entry deletion-eligible, including stable base entries that current code, docs, and tests treat as preserved cron session state.

## What This Changes

The PR changes `src/cron/session-reaper.ts` so stale cron entries are pruned when they match either `isCronRunSessionKey` or the broader `isCronSessionKey`.

## Best Possible Solution

Define the cron session lifecycle contract, then prune expired run artifacts and deleted-job leftovers without deleting active-job base metadata or carried user overrides.

## Reproduction Assessment

Yes for source-level reproduction: isolated cron resolves a stable `agent:<id>:cron:<job>` base key, while current main's reaper only prunes `cron:<job>:run:<run>` keys. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No: this is a plausible small fix for the predicate mismatch, but it is broader than the documented/current retention contract and can delete cron base state that current code preserves. The safer path is a narrow lifecycle-aware cleanup for expired run artifacts or deleted-job leftovers.
