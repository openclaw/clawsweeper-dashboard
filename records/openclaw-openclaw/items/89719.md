## Summary

This PR should stay open for maintainer review: it targets a real source-reproducible cron retention gap, but the patch changes the reaper from pruning run-shaped cron records to deleting any stale cron-prefixed session entry, including durable base cron state.

## What This Changes

The PR changes `sweepCronRunSessions` to prune stale entries matching either `isCronRunSessionKey` or the broader `isCronSessionKey`.

## Best Possible Solution

Define the retention contract for stable cron base sessions, then prune only ephemeral run records or explicitly safe legacy/orphaned base records without deleting durable preferences or heartbeat state.

## Reproduction Assessment

Yes from source inspection: current main persists isolated cron metadata under a stable `agent:<id>:cron:<job>` base key while the reaper only matches `cron:<job>:run:<runId>` keys. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No: using `isCronSessionKey` in the deletion gate is broader than the reported bug and can delete durable cron base state. The safer solution needs an explicit base-session retention policy and tests for preserved preferences, heartbeat state, and the intended cleanup target.
