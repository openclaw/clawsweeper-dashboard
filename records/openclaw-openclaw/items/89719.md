## Summary

Keep open: the underlying cron retention bug is source-reproducible, but this PR is not merge-ready because it broadens deletion of cron base session rows and is currently conflicting with main; the newer guarded PR is useful context but is also conflicting, so it is not a clean superseding landing path.

## What This Changes

The PR changes the cron session reaper to treat both cron run keys and broader cron session keys as retention-prunable candidates.

## Best Possible Solution

Land one refreshed cron lifecycle fix that prunes stale isolated or orphaned base rows while preserving durable main/named cron session metadata and transcript archival safeguards.

## Reproduction Assessment

Yes. Source inspection shows isolated cron base keys resolve to `agent:<id>:cron:<jobId>`, while current main's reaper only accepts `:run:` cron keys, so stale isolated base rows skip the retention cutoff.

## Solution Assessment

No. The one-line predicate fix is plausible but too broad because it makes all cron-prefixed base rows retention-deletable instead of distinguishing disposable isolated or orphaned rows from durable persistent cron state.
