## Summary

Keep open: the PR targets a real source-reproducible cron retention gap, but the current one-line gate makes every cron-prefixed session key retention-deletable, which is broader than the existing base-session preservation/state-carrying contract. It also still lacks after-fix real gateway/session-store proof.

## What This Changes

The PR adds `isCronSessionKey` to the cron reaper gate so stale cron-prefixed session-store entries without `:run:` become eligible for retention pruning.

## Best Possible Solution

Keep durable base cron entries preserved unless maintainers explicitly choose otherwise; add a narrow expired-isolated-cron cleanup path with tests and real proof for stale, fresh, heartbeat-like, and preference-carrying cron entries.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: isolated cron jobs resolve to `agent:<id>:cron:<job>` base keys, while current main's reaper only matches `:run:` keys. I did not run a live gateway/session-store reproduction in this read-only review.

## Solution Assessment

No, this PR is only a plausible fix. The safer solution must either preserve durable base cron state while pruning disposable records, or make an explicit maintainer decision that base cron sessions are retention-deletable and prove the upgrade behavior.
