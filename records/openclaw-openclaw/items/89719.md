## What This Changes

The PR changes `src/cron/session-reaper.ts` so stale entries are pruned when they match either `isCronRunSessionKey` or the broader `isCronSessionKey` predicate.

## Best Possible Solution

Define and land a cron-owned lifecycle fix that prunes stale isolated or orphaned base rows while explicitly preserving durable main/named cron base metadata and transcript archival safeguards.

## Reproduction Assessment

Yes. Source inspection shows isolated cron builds `agent:<id>:cron:<job>` base keys while current main's reaper accepts only `:run:` cron keys, so stale isolated base rows skip the retention cutoff.

## Solution Assessment

No. This is a plausible predicate fix, but it is broader than the current retention contract; the better fix should distinguish disposable isolated or orphaned base rows from durable base cron session metadata.
