## Summary

Keep this PR open, but the current patch is not merge-ready: it fixes the reaper by widening a shared helper whose existing callers deliberately distinguish durable cron base sessions from ephemeral `:run:` sessions.

## What This Changes

The PR widens `isCronRunSessionKey` so `agent:<id>:cron:<jobId>` base keys match, and updates the session reaper key-shape test expectation.

## Best Possible Solution

Keep `isCronRunSessionKey` scoped to `:run:` keys and add a reaper-specific retention predicate or key classification that prunes stale isolated base entries without changing routing/listing semantics.

## Reproduction Assessment

Yes, from source inspection: isolated cron metadata is persisted under the stable `agent:<id>:cron:<job>` key while the current reaper only matches `:run:` keys. I did not run tests because this review is read-only.

## Solution Assessment

No. The requested bug fix is valid, but changing the shared `isCronRunSessionKey` contract is too broad; a reaper-specific matcher is the narrower maintainable solution.
