## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist rollback state before applying staged files:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1217`
  - body: `applyDreamingMaintenance` writes each staged file before `last-apply.json` exists. If the process exits after `MEMORY.md` or `current.json` is written, the workspace is changed but rollback has no restore record, and re-apply can then conflict on the SHA check.
  - confidence: 0.91
- **[P1] Reject staged paths outside the workspace:** `extensions/memory-core/src/dreaming-maintenance.ts:1121-1128`
  - body: Staged plans are mutable JSON under the workspace, but apply and rollback trust `change.path` and join it with `workspaceDir`. A tampered plan can write or unlink outside the workspace unless paths are resolved and checked against an allowed root before read, write, or unlink.
  - confidence: 0.9
- **[P1] Reject daily signal paths outside the workspace:** `extensions/memory-core/src/dreaming-phases.ts:1070`
  - body: `dailySignalFiles` accepts arbitrary strings and this path only strips `./` before calling `path.join`. A value containing `../` can make Dreaming stat and read files outside the workspace and ingest their contents as memory evidence.
  - confidence: 0.88
- **[P1] Pass allowed paths for configured daily-signal records:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: Configured files like `memory/daily-log.md` are not date-named, and this primary ingestion path calls `recordShortTermRecalls` without the `allowPaths` whitelist used by the fallback path. Once fallback skips an already tracked path, later edits can be dropped from short-term recall evidence.
  - confidence: 0.84
