## Summary

Keep open. Current main still uses the direct Dreaming promotion path, while this PR is a useful but broad replacement that still has concrete correctness and security blockers before merge.

## What This Changes

The PR changes memory-core Dreaming from direct durable promotion into staged maintenance artifacts with `/dreaming apply` and rollback, adds daily signal file configuration and tests/docs, and also changes Amazon Bedrock Mantle provider discovery behavior.

## Best Possible Solution

Revise this into a focused memory-core PR: keep stage-first Dreaming maintenance and docs/tests together, persist rollback records before file writes, validate staged file-change paths against workspace-owned maintenance targets, pass daily signal allowlists through ingestion, and split the Bedrock Mantle change into its own provider PR.

## Reproduction Assessment

Yes. Current main behavior is reproducible from source and tests that read `MEMORY.md` after Dreaming promotion; the PR regressions are directly visible in the diff paths. I did not run tests because this was a read-only review.

## Solution Assessment

No. The stage-first direction is plausible, but this patch is not the best mergeable form until the rollback ordering, path safety, daily-signal whitelist, and unrelated provider changes are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist the rollback record before applying file changes:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1218`
  - body: `applyDreamingMaintenance` writes staged file changes before saving `last-apply.json`. If the process exits after `MEMORY.md` or `current.json` changes but before that record is persisted, rollback has no restore source and re-apply will conflict against the changed hashes. Build the restore record and save it before writing the file changes.
  - confidence: 0.95
- **[P1] Pass allowPaths when recording daily-signal batches:** `extensions/memory-core/src/dreaming-phases.ts:1262-1266`
  - body: `recordShortTermRecalls` now filters non-date memory files unless they are explicitly whitelisted, but this collected daily-signal batch path does not pass the whitelist. After the fallback creates the first tracked entry, later edits to `memory/daily-log.md` can be skipped and stop refreshing the evidence chain.
  - confidence: 0.9
- **[P1] Constrain staged file-change paths to the workspace:** `extensions/memory-core/src/dreaming-maintenance.ts:1121-1122`
  - body: `writeFileChange` trusts `change.path` from staged workspace JSON and joins it to the workspace path. Absolute paths or `..` segments can escape the workspace during `/dreaming apply` or rollback, so staged file changes need normalized allowlisted relative paths before hash checks or writes.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete workspace path-safety issue in the new staged apply/rollback path.

Concerns:

- **[medium] Staged file paths are not workspace-bound:** `extensions/memory-core/src/dreaming-maintenance.ts:1122`
  - body: `readStagedPlan` trusts staged JSON file changes, and `writeFileChange` writes `path.join(workspaceDir, change.path)` without rejecting absolute paths or parent traversal. A modified staged plan or rollback record could cause writes outside the intended memory maintenance files.
  - confidence: 0.87
