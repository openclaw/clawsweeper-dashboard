## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain staged paths before writing maintenance files:** `extensions/memory-core/src/dreaming-maintenance.ts:1122-1128`
  - body: `writeFileChange` accepts `change.path` from parsed staged/rollback JSON and joins it with the workspace without rejecting parent traversal or unexpected targets. Because `/dreaming apply` and rollback consume workspace-writable artifacts, a tampered plan can write outside the intended Dreaming maintenance files; normalize and validate every path before hashing or writing.
  - confidence: 0.9
- **[P1] Persist rollback metadata before writing staged files:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1218`
  - body: `applyDreamingMaintenance` writes planned file changes before saving `last-apply.json`. If the process exits after MEMORY.md or current.json is written but before the record is persisted, rollback has no restore source and a later apply will conflict against the changed `beforeSha1`; build and write the restore record first.
  - confidence: 0.95
- **[P1] Pass daily signal allowPaths when recording recalls:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: This `recordShortTermRecalls` call omits the configured daily-signal whitelist even though non-date memory files are filtered unless they are allowed. After `memory/daily-log.md` has already been tracked by the fallback, later edits can stop refreshing the short-term evidence chain.
  - confidence: 0.9
