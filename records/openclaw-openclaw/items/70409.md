## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Persist the rollback record before writing files:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1217`
  - body: `applyDreamingMaintenance` builds `restoreFiles` but writes each file before `last-apply.json` is persisted. If the process exits after a file write and before the record write, the workspace is changed with no rollback record and re-apply can conflict against the changed SHA. Persist the rollback record first, then perform the file writes and cleanup.
  - confidence: 0.92
- **[P1] Constrain staged file paths to the workspace:** `extensions/memory-core/src/dreaming-maintenance.ts:1122-1128`
  - body: `writeFileChange` accepts paths from staged JSON and writes `path.join(workspaceDir, change.path)` without checking the resolved path stays under the workspace. A modified staged plan or rollback record can use `../` segments to overwrite files outside the intended workspace, so apply and rollback need a shared containment check before reading or writing.
  - confidence: 0.9
- **[P2] Pass allowPaths for configured daily signals:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: Configured files such as `memory/daily-log.md` are only accepted by `recordShortTermRecalls` when they are passed through `allowPaths`, but the main collected-batch path calls it without that whitelist. The fallback covers a first-seen file, then skips already tracked paths, so later edits can be silently ignored and durable maintenance works from stale evidence.
  - confidence: 0.87
