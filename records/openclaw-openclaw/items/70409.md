## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Persist the rollback record before writing files:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1217`
  - body: `applyDreamingMaintenance` writes each planned file before `last-apply.json` exists. If the process exits after a file write but before the record is persisted, rollback has no restore data and a retry can conflict against the changed file hash.
  - confidence: 0.93
- **[P1] Constrain staged file paths to the workspace:** `extensions/memory-core/src/dreaming-maintenance.ts:1122-1128`
  - body: `writeFileChange` joins `change.path` directly with `workspaceDir`, and both apply and rollback consume paths from mutable JSON records. A staged plan containing `../` can make the maintenance action write or delete outside the workspace unless the resolved path is checked against the workspace root.
  - confidence: 0.91
- **[P2] Pass allowPaths for configured daily signals:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: The PR makes non-date files eligible only through `allowPaths`, but this main ingestion loop records configured daily-signal batches without passing those paths. After the fallback has tracked the file once, later edits can be skipped as non-short-term paths and leave durable maintenance driven by stale evidence.
  - confidence: 0.86
