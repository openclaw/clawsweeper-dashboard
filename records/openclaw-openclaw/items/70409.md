## Summary

Keep open. Current main still uses direct deep-sweep promotion into MEMORY.md, so this PR is not obsolete, but the proposed branch has blocking correctness and security issues in the new apply/rollback path plus an unrelated Bedrock Mantle provider edit that should be split or explicitly reviewed.

## What This Changes

This PR changes memory-core Dreaming from direct deep-sweep MEMORY.md promotion to staged maintenance artifacts with `/dreaming apply|rollback`, adds daily signal file config/docs/tests/a verification script, and also edits Amazon Bedrock Mantle discovery behavior.

## Best Possible Solution

Revise this into a focused memory-core change: keep stage-first Dreaming maintenance scoped to the memory-core and memory host SDK contract, validate all staged/rollback file paths as workspace-relative maintenance targets before hashing or writing, persist rollback metadata before applying changes, pass configured daily signal paths through the recall whitelist, add the required changelog entry, and split the Bedrock Mantle provider edit into a separate PR.

## Reproduction Assessment

Yes. Current main reproduction is source-backed: deep Dreaming still calls applyShortTermPromotions, which writes MEMORY.md directly. The PR regressions are also directly reproducible from the PR head diff paths identified above; I did not run tests because this review was read-only.

## Solution Assessment

No. Stage-first Dreaming maintenance with explicit apply/rollback is a plausible direction, but this implementation is not the best merge candidate until the rollback ordering, path validation, daily-signal ingestion, changelog, and unrelated provider scope are corrected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Constrain staged paths before writing maintenance files:** `extensions/memory-core/src/dreaming-maintenance.ts:1122`
  - body: `writeFileChange` accepts `change.path` from parsed staged/rollback JSON and joins it without rejecting parent traversal. Because `/dreaming apply` and rollback consume workspace artifacts, a tampered plan can write outside the intended memory maintenance files. Normalize and validate every file-change path as workspace-relative before hash checks or writes.
  - confidence: 0.9
- **[P1] Persist the rollback record before applying file changes:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1218`
  - body: `applyDreamingMaintenance` writes planned file changes before saving `last-apply.json`. If the process exits after MEMORY.md or current.json is written but before that record is persisted, rollback has no restore source and a later apply will conflict against the changed `beforeSha1`. Build the restore record and write it before applying file changes.
  - confidence: 0.95
- **[P1] Pass daily signal allowPaths into recall recording:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: `recordShortTermRecalls` now filters out non-date memory files unless they are in `allowPaths`, but this daily-signal ingestion call omits the whitelist. After the fallback has already tracked `memory/daily-log.md`, later edits to that configured signal file can stop refreshing the short-term evidence chain.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `extensions/memory-core/src/dreaming-command.ts:82`
  - body: This PR adds user-facing `/dreaming apply|rollback` behavior and changes deep Dreaming's default persistence flow, but the branch does not update CHANGELOG.md. Repo policy requires a user-facing feature entry under Unreleased so release notes do not omit the behavior change.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The new staged maintenance apply/rollback path can write outside the intended workspace targets when a staged plan or last-apply record is tampered.

Concerns:

- **[medium] Staged file paths are not workspace-bound:** `extensions/memory-core/src/dreaming-maintenance.ts:1122`
  - body: `readStagedPlan` trusts `report.fileChanges`, and `writeFileChange` joins `change.path` with the workspace without rejecting parent traversal. Because apply and rollback consume workspace-writable JSON artifacts, this can overwrite files outside the intended memory maintenance surface.
