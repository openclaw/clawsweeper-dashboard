Overall confidence: 0.9

Full review comments:

- **[P1] Preserve existing direct-promotion behavior on upgrade:** `src/memory-host-sdk/dreaming.ts:21`
  - body: On current main and the latest release, enabled deep Dreaming writes promoted entries to `MEMORY.md` automatically. With `autoApply` defaulting to false, existing users can silently move to stage-only output after upgrade unless maintainers explicitly approve that break or the PR keeps the current behavior as the default with an opt-in stage-only mode.
  - confidence: 0.9
- **[P1] Persist the rollback record before writing files:** `extensions/memory-core/src/dreaming-maintenance.ts:1214-1217`
  - body: `applyDreamingMaintenance` writes staged files before `last-apply.json` exists. If the process exits after a file write but before the record is persisted, rollback has no restore data and retry can conflict against the changed file hash.
  - confidence: 0.93
- **[P1] Constrain staged file paths to the workspace:** `extensions/memory-core/src/dreaming-maintenance.ts:1122-1128`
  - body: `writeFileChange` joins a mutable staged-plan path directly with `workspaceDir`, and rollback reuses the same helper for JSON restore records. A tampered `../` path can write or unlink outside the workspace unless the resolved target is checked against the workspace root.
  - confidence: 0.91
- **[P2] Pass allowPaths for configured daily signals:** `extensions/memory-core/src/dreaming-phases.ts:1263-1272`
  - body: The PR makes non-date files eligible only through `allowPaths`, but this main ingestion loop records configured daily-signal batches without passing those paths. After the fallback has tracked the file once, later edits can be skipped as non-short-term paths and leave durable maintenance driven by stale evidence.
  - confidence: 0.86
