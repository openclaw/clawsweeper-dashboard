## Summary

Keep this PR open. The proposed partial-image forwarding has a blocking ordering bug: it compacts surviving images into an all-inline prefix, which can mis-associate or duplicate images when an earlier attachment fails, and the PR still has only unit-test proof for a user-visible media-delivery path.

## What This Changes

The PR changes `resolveCurrentTurnImages` to return successfully loaded disk-backed current-turn images when some image attachments fail, and adds a focused regression test for that partial-success case.

## Best Possible Solution

Preserve partial image delivery only with an ordering/index-aware contract that keeps prompt refs and native image blocks aligned; otherwise keep the current fallback for gapped batches and let the broader Telegram media-hydration work proceed separately.

## Reproduction Assessment

Yes for the PR-introduced defect: current source, shipped runner coverage, and PR status show the partial-current-media contract failing against the proposed behavior. The linked live Telegram user flow was not reproduced in this read-only review.

## Solution Assessment

No. The PR touches the right shared auto-reply boundary, but compacting a partial batch into `imageOrder: ["inline"]` is not the best fix because it loses original attachment positions.
