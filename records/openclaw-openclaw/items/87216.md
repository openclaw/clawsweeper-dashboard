## Summary

Keep open: current main and latest release still do not resolve the central manual full-URL input problem; host-only private-LAN input is covered, but URL-shaped `ws://...` Host input is still routed through the host/port/TLS composer instead of a clear WebSocket endpoint contract.

## What This Changes

The issue asks Android Manual/Advanced setup to stop parsing `ws://...` as host `ws` and either accept full WebSocket URLs or reject them clearly while resolving host/port/TLS to a valid WebSocket endpoint.

## Best Possible Solution

Make Android Manual setup unambiguous end to end: either accept and normalize full `ws://`/`wss://` endpoint input, or reject URL-shaped Host values while previewing and connecting through the actual `ws://`/`wss://` transport derived from host/port/TLS.

## Reproduction Assessment

Partially: source inspection gives a concrete current-main path because manual input is composed from Host/Port/TLS through `http` or `https` before parsing, so a pasted full `ws://...` endpoint is not preserved. I did not run a live Android device reproduction of the original v2026.4.5 screen.

## Solution Assessment

Unclear: accepting full WebSocket URLs is a plausible small fix, but keeping separate Host/Port/TLS fields with explicit URL rejection and a WebSocket-accurate preview may be the safer product contract.
