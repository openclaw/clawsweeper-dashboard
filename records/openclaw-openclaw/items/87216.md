## Summary

Keep open: current main and v2026.6.9 still leave the Android manual setup contract unresolved for URL-shaped `ws://...` input, even though host-only private-LAN input is now partly covered.

## What This Changes

The issue requests fixing Android manual setup so private-LAN WebSocket endpoints are not parsed as host `ws` or previewed as `http://ws:<port>`.

## Best Possible Solution

Make Android manual setup unambiguous end to end: either accept and normalize full `ws://`/`wss://` endpoint input, or reject URL-shaped Host values while previewing the actual `ws://`/`wss://` transport endpoint for host/port/TLS entries.

## Reproduction Assessment

Partially: source inspection gives a concrete current-main path for the remaining mismatch because manual input is composed from host/port/TLS through HTTP/HTTPS before parsing. I did not establish a live Android reproduction of the original v2026.4.5 screen.

## Solution Assessment

Unclear. Accepting full WebSocket URLs is plausible, but keeping host-only manual fields with explicit URL rejection and an accurate WebSocket preview may be the safer product contract.
