## Summary

Keep open: current main and v2026.6.8 fixed the private-LAN host-only path, but Android manual setup still does not fully match the report because manual input is composed through HTTP/HTTPS and the visible resolved endpoint for cleartext WebSocket routes remains HTTP-shaped.

## What This Changes

The issue asks Android manual LAN setup to treat `ws://` as a WebSocket scheme, not as host text, or to make host-only manual entry unambiguous and preview the actual `ws://` endpoint.

## Best Possible Solution

Make Android manual setup unambiguous end to end: either accept full WebSocket URLs directly, or keep host-only fields with validation that rejects pasted URLs and previews the actual `ws://` or `wss://` transport endpoint.

## Reproduction Assessment

Yes for a source-level remaining mismatch: current manual setup still builds an HTTP/HTTPS URL from host/port/TLS fields before shared parsing, and cleartext display URLs remain HTTP-shaped. I did not run a live Android device repro for the old `http://ws:18789` screen.

## Solution Assessment

No. Current main is only a partial fix because host-only private-LAN input now works, but full `ws://` manual entry and the displayed resolved endpoint are still not aligned with the WebSocket transport contract.
