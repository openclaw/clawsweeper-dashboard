## Summary

Keep open: current main and v2026.6.8 cover the private-LAN host-only path, but the Android manual flow still does not make full `ws://...` manual input first-class and still previews cleartext WebSocket routes as HTTP-shaped endpoints, so the central UX/contract mismatch remains.

## What This Changes

The issue requests Android manual setup to stop treating `ws://` as hostname text and either preserve full WebSocket URLs or clearly require host-only input.

## Best Possible Solution

Make Android manual setup unambiguous end to end: either accept full WebSocket URLs directly, or keep host-only fields with validation that rejects pasted URLs and previews the actual `ws://` or `wss://` transport endpoint.

## Reproduction Assessment

Partially: source inspection gives a concrete current-main reproduction for the remaining mismatch, because manual input is composed from host/port/TLS through HTTP/HTTPS before parsing. I did not establish a live Android reproduction of the original v2026.4.5 `http://ws:18789` screen.

## Solution Assessment

Unclear. Accepting full WebSocket URLs is plausible, but keeping host-only manual fields with explicit URL rejection and an accurate `ws://`/`wss://` preview may be the safer product contract.
