## Summary

Keep open: current main and the latest release fixed the private-LAN host-only path, but the remaining manual-entry behavior is still not fully aligned with the report. The manual UI still composes raw Host text through an HTTP/HTTPS display helper before parsing, and the preview shown as “Resolved endpoint” still renders cleartext WebSocket endpoints as `http://...` rather than the actual `ws://...` transport.

## What This Changes

Android manual LAN setup should not treat a pasted `ws://` scheme as hostname text or show a misleading resolved endpoint.

## Best Possible Solution

Make Android manual setup unambiguous end to end: either accept full `ws://`/`wss://` URLs directly as manual endpoint input and preview the WebSocket URL, or keep host-only fields with scheme-specific validation that rejects pasted URLs and previews the actual transport scheme.

## Reproduction Assessment

Yes for source-level reproduction of the remaining mismatch: current manual setup still composes Host text into an HTTP/HTTPS display URL before parsing, while tests only prove the host-only private-LAN path. I did not run a live Android device repro for the old `http://ws:18789` screen.

## Solution Assessment

No, current main is only a partial fix for this issue. The best solution should settle the manual URL-vs-host UX and make the displayed resolved endpoint match the actual WebSocket transport.
