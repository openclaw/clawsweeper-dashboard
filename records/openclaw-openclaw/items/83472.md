## Summary

Keep open. The bug is real and source-reproducible, but this PR adds a separate hard-delete fallback where current main already has an fs-safe Trash helper used by the Gateway path, so the branch should not merge as-is.

## What This Changes

This PR adds an `fs.rm({ recursive: true, force: true })` fallback and regression tests for `moveToTrash` when spawning `trash` fails.

## Best Possible Solution

Update the CLI/reset cleanup helper to use the same `movePathToTrash` fs-safe Trash behavior as the Gateway, preserving documented Trash semantics while removing the PATH `trash` dependency.

## Reproduction Assessment

Yes by source inspection and provided issue/proof data: current main calls a PATH `trash` binary in the offline cleanup path, and the Docker runtime package list does not install it. I did not run a live Docker reproduction in this read-only review.

## Solution Assessment

No. The branch fixes the missing-binary symptom, but the maintainable fix is to share the existing fs-safe Trash helper rather than add a separate hard-delete fallback that diverges from Gateway and docs.
