## Summary

Keep open: the bug is real and the supplied production proof is strong, but the patch should not merge as-is because the empty-provider matching path can let lower-precedence defaults veto an agent-specific runtime policy, and the CLI shortcut bypasses the provider/runtime pairing check that the normal path enforces.

## What This Changes

The PR updates agent runtime-policy lookup so provider-prefixed model entries can match persisted bare-model sessions, adds empty-provider ambiguity handling, returns CLI runtime aliases for that path, and adds focused tests.

## Best Possible Solution

Land the narrow runtime-policy fix after preserving agent/default precedence and validating CLI runtime aliases against the provider from the matched model key.

## Solution Assessment

No. The central direction is right, but the implementation should preserve model-map precedence and use the matched provider when deciding whether a CLI runtime alias is compatible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve model-map precedence before ambiguity:** `src/agents/model-runtime-policy.ts:158-179`
  - body: The new empty-provider collector walks both the agent-specific map and `agents.defaults.models` before deciding ambiguity. If an agent has `anthropic/foo` configured but defaults also contain `openai/foo`, this returns `{}` even though per-agent exact model policy is documented to win over defaults, leaving the legacy session on the wrong/default runtime. Detect ambiguity only within the first model map that has matches, then return that scope before considering lower-precedence maps.
  - confidence: 0.86
- **[P2] Keep CLI alias validation provider-scoped:** `src/agents/model-runtime-aliases.ts:292-293`
  - body: When `provider` is empty, returning any known CLI alias skips the existing `${provider}:${runtime}` compatibility check. A single provider-prefixed policy with a mismatched CLI alias can now route to that CLI even though the non-empty provider path would reject it; carry the matched entry provider forward and use the same provider/runtime map before returning.
  - confidence: 0.76
