## Summary

Keep open. Current main has per-agent exec allowlists, durable exact-command approvals, safe bins, and node-host `autoAllowSkills` for trusted skill binaries, but it does not have a skill-scoped readonly exec declaration or a per-skill command allowlist. Because the request changes host exec approval semantics, it needs maintainer/security design rather than conservative cleanup closure.

## What This Changes

Add skill-level readonly exec approval metadata and a bounded command allowlist so healthcheck/canary skills can run approved introspection commands without repeated prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit command/argv rules, keeps mutating and outbound actions gated regardless of metadata, and updates skill metadata parsing, exec approval evaluation, protocol/config/UI/docs, and focused security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema/docs, durable approval matching, and node-host `autoAllowSkills`; replaying the exact WSL2 canary loop is not needed to confirm the requested API is absent.

## Solution Assessment

Unclear. The proposed `metadata.openclaw.execScope` direction is plausible, but the exact contract should be settled by maintainers/security because current `autoAllowSkills` is intentionally broad implicit trust rather than per-skill readonly command approval.
