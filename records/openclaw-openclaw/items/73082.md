## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract, and the requested path would introduce a new host exec trust boundary that needs maintainer/security design before implementation.

## What This Changes

Request a SKILL.md `metadata.openclaw.execScope: readonly` plus per-skill readonly allowlist so healthcheck/canary skills can auto-approve bounded read-only `openclaw` introspection commands without exact command-hash prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity and explicit command/argv rules, preserves approval gates for outbound or mutating actions, and updates metadata parsing, approval evaluation, config/protocol/UI/docs, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is high-confidence statically reproducible on current main by inspecting skill metadata parsing, exec approval schema/evaluation, durable approval matching, docs, and the absence of the requested metadata symbols.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact design should be settled by maintainers/security because current `autoAllowSkills` is broad binary trust rather than per-skill readonly command authorization.
