## Summary

Keep open. Current main still lacks a skill-scoped readonly exec approval contract; the existing mechanisms are per-agent host allowlists, durable exact-command or segment allow-always approvals, safe bins, and broad `autoAllowSkills` binary trust. Because the request would add a new host exec auto-approval path, it needs maintainer/security design before implementation.

## What This Changes

The issue requests a `metadata.openclaw.execScope: readonly` plus bounded readonly command allowlist for healthcheck/canary skills to avoid repeated exec approval prompts for read-only status commands.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to a trusted skill identity plus explicit command/argv rules, preserves normal prompts for anything outside that contract, and updates metadata parsing, approval evaluation, protocol/config/docs/UI, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema/evaluation, docs, and the absence of `execScope`/`readonlyAllowlist` symbols.

## Solution Assessment

Unclear. The proposed metadata shape is plausible, but the exact trust boundary should be settled by maintainers/security because existing `autoAllowSkills` is broad binary trust rather than per-skill readonly command authorization.
