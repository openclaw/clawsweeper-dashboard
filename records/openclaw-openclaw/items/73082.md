## Summary

Keep open. Current main still lacks a skill-scoped readonly exec approval contract; the existing paths cover per-agent allowlists, durable exact-command approvals, safe bins, and broad `autoAllowSkills` skill-binary trust, but not the requested per-skill readonly command allowlist. Because this would change host exec auto-approval semantics, it needs maintainer/security design before implementation.

## What This Changes

The issue requests a `metadata.openclaw.execScope: readonly` plus per-skill readonly allowlist so healthcheck/canary skills can auto-approve declared read-only OpenClaw introspection commands while keeping mutating and outbound actions gated.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to a trusted skill identity plus explicit command/argv rules, keeps outbound and mutating operations approval-gated regardless of metadata, and updates skill metadata parsing, approval evaluation, protocol/config/UI/docs, and focused security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema/evaluation, durable approval matching, docs, and the absence of `execScope`/`readonlyAllowlist` symbols.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact implementation should be settled by maintainers/security because the existing `autoAllowSkills` path is broad implicit binary trust rather than per-skill readonly command authorization.
