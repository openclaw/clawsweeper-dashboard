## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract, and the request would create a new host-exec trust boundary that needs maintainer/security design before implementation.

## What This Changes

Add a skill-scoped readonly exec approval mechanism so healthcheck/canary skills can auto-approve bounded read-only OpenClaw commands without per-invocation prompts.

## Best Possible Solution

Design an additive trusted-skill exec scope contract with identity binding, explicit argv rules, fail-closed side-effect exclusions, protocol/schema support, docs/UI, and security tests before implementation.

## Reproduction Assessment

Not applicable as a feature request. Static verification is high-confidence: current main lacks the requested metadata and still routes host exec approval through agent-level allowlists plus broad `autoAllowSkills`.

## Solution Assessment

Unclear. The proposed metadata shape is plausible, but the best maintainable path depends on a maintainer-approved trust model for skill identity, third-party skills, argv matching, and side-effect classification.
