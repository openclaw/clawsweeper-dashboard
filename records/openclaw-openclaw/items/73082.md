## Summary

Current main still lacks a skill-scoped readonly exec approval contract. The existing implementation covers durable exact-command approvals, per-agent allowlist patterns, safe bins, and broad `autoAllowSkills` skill-binary trust, but not a trusted-skill identity plus readonly command allowlist; because this would create a new host exec auto-approval boundary, it should remain open for maintainer/security design.

## What This Changes

The issue requests a skill metadata field such as `execScope: readonly` plus a per-skill readonly command allowlist so canary and healthcheck skills can run bounded introspection commands without repeated exec approval prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to a trusted skill identity plus explicit command/argv rules, keeps outbound and mutating operations approval-gated regardless of metadata, and updates metadata parsing, approval evaluation, config/protocol/UI/docs, and focused security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema/evaluation, durable approval matching, docs, and the absence of the requested metadata symbols.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact design should be settled by maintainers/security because current `autoAllowSkills` is broad binary trust rather than per-skill readonly command authorization.
