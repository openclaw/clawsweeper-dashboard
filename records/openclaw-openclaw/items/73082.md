## Summary

Keep open. Current main has adjacent exec-approval machinery, but it does not implement a per-skill read-only exec scope or SKILL.md allowlist, and the requested direction changes a security-sensitive host-exec trust boundary that needs maintainer design review before implementation.

## What This Changes

The issue requests a per-skill read-only exec approval scope so healthcheck or canary skills can auto-approve bounded introspection commands without accumulating brittle command-hash approvals.

## Best Possible Solution

Design a generic core approval seam for trusted skill-scoped read-only command intent: bind it to skill identity/source, use explicit argv or command-pattern matching, keep mutating/outbound operations fail-closed, and cover gateway/node exec paths plus docs and security audit warnings before enabling canary automation.

## Reproduction Assessment

Yes. As a capability gap, current main is reproducible by source inspection: the exec approvals and skill metadata contracts lack execScope/readonlyAllowlist fields, and approval checks consult only per-agent allowlists, safeBins, durable approvals, and autoAllowSkills skill-bin trust.

## Solution Assessment

Unclear. The proposed SKILL.md shape is plausible, but SKILL.md self-attestation alone is not obviously the safest solution for a host-exec approval boundary; a maintainer-approved policy model tied to trusted skill identity and explicit operator opt-in would be safer.
