## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract. The existing surfaces are agent-level allowlists plus broad `autoAllowSkills`, and the requested change would create a new host-exec trust boundary that needs maintainer/security design before implementation.

## What This Changes

Request to add `SKILL.md` metadata for readonly exec scope and a per-skill command allowlist for canary/healthcheck skills.

## Best Possible Solution

Design an additive approval contract bound to trusted skill identity plus explicit argv rules, then update metadata parsing, protocol/schema, approval evaluation, docs/UI, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. Static verification is high confidence: the requested metadata names are absent and current main routes exec approval through agent-level allowlists and `autoAllowSkills`.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but maintainers should settle the trust model first because it would authorize host exec without per-run approval.
