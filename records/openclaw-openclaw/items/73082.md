## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract. The report is concrete, but it would create a new host-exec trust boundary, so it needs maintainer/security design before a repair PR is queued.

## What This Changes

The issue requests skill-level readonly exec approval metadata so bounded canary and healthcheck commands can run without repeated per-command approvals.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv rules, then update metadata parsing, protocol/schema, approval evaluation, docs, UI, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection gives a high-confidence verification path: the requested metadata symbols are absent and current main uses agent-level allowlists plus broad autoAllowSkills.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact trust model should be settled by maintainers because current autoAllowSkills trusts skill binaries broadly rather than authorizing per-skill readonly commands.
