## Summary

Keep open. Current main still lacks a skill-level readonly exec scope or bounded readonly command allowlist, and the existing `autoAllowSkills` path is documented and implemented as broad skill-bin trust rather than a per-skill readonly approval contract. Because this would change host exec approval semantics, it needs maintainer/security design before implementation.

## What This Changes

Request to add `metadata.openclaw.execScope: readonly` plus a per-skill `readonlyAllowlist` so healthcheck/canary skills can auto-approve bounded readonly commands while keeping mutating and outbound actions approval-gated.

## Best Possible Solution

Define an additive core approval contract that binds readonly auto-approval to trusted skill identity and explicit argv rules, keeps outbound delivery, filesystem writes, service restarts, cron mutations, and other side effects fail-closed, and updates skill metadata parsing, gateway protocol/schema, node-host evaluation, CLI/UI/docs, and tests together.

## Reproduction Assessment

Not applicable as a normal bug reproduction. The missing surface is statically reproducible on current main by inspecting skill metadata parsing/types, exec approval schema, node-host allowlist evaluation, and docs; replaying the exact WSL2 canary loop is not required to confirm the API is absent.

## Solution Assessment

Unclear. The proposed `execScope` direction is plausible, but the exact contract should be settled by maintainers/security because it creates a new host exec trust boundary and current `autoAllowSkills` is intentionally broader implicit trust.
