## Summary

Keep open: current main still has per-agent exec allowlists, durable allow-always entries, safe bins, and broad skill-binary auto-allow, but no skill-scoped readonly exec approval contract. The requested behavior would add a new host exec trust boundary, so it needs maintainer/security design before implementation.

## What This Changes

Add a `metadata.openclaw.execScope: readonly` skill marker with a per-skill readonly command allowlist so canary/healthcheck skills can avoid repeated approvals for known read-only status commands.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv rules, keeps side-effecting operations gated, and updates metadata parsing, protocol/schema, approval evaluation, docs, UI, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection of current main gives a high-confidence verification path: the requested metadata symbols are absent, and the existing approval model is agent-level allowlist plus broad `autoAllowSkills`.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact design should be settled by maintainers because current `autoAllowSkills` is broad binary trust rather than per-skill readonly command authorization.
