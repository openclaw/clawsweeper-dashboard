## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract, and the requested behavior would add a new host-exec auto-approval boundary that needs maintainer/security design before implementation.

## What This Changes

Add a skill-scoped readonly exec approval mechanism so healthcheck/canary skills can auto-run explicit read-only OpenClaw commands without per-invocation prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit command/argv rules, keeps outbound and mutating operations approval-gated regardless of metadata, and updates metadata parsing, approval evaluation, config/protocol/UI/docs, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema/evaluation, durable approval matching, docs, and the absence of the requested metadata symbols.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact trust model should be settled by maintainers/security because current `autoAllowSkills` is broad skill-binary trust rather than per-skill readonly command authorization.
