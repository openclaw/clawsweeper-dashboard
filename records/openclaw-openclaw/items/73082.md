## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract; the existing controls are agent-level allowlists, safe bins, and broad autoAllowSkills, while the request would add a security-sensitive host-exec trust boundary.

## What This Changes

Request adding skill frontmatter metadata so readonly healthcheck/canary skills can auto-approve a bounded list of read-only OpenClaw CLI commands.

## Best Possible Solution

Design an additive, versioned core exec approval contract that binds trusted skill identity to strict argv rules, then update metadata parsing, protocol schema, runtime evaluation, docs/UI, and security regression tests together.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken documented behavior. Static inspection gives high confidence that current main lacks a skill-scoped readonly exec approval contract.

## Solution Assessment

Unclear. The proposed metadata shape targets the real pain point, but the safest solution needs maintainer-approved trust semantics instead of directly trusting arbitrary skill frontmatter.
