## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract. Existing behavior covers per-agent allowlists, exact durable command approvals, safe bins, and broad skill-binary auto-allow, but the requested skill metadata boundary is still missing and needs maintainer/security design.

## What This Changes

Request adds SKILL.md metadata for a readonly exec scope and bounded readonly command allowlist for canary/status skills.

## Best Possible Solution

Define an additive approval contract that binds trusted skill identity to explicit argv rules, keeps mutating/outbound operations gated, and updates metadata parsing, protocol/schema, approval evaluation, docs, UI, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection gives a high-confidence verification path: the requested metadata fields are absent and the current approval model has no skill-scoped readonly command authorization.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact trust model should be specified by maintainers because current autoAllowSkills is broad binary trust rather than per-skill readonly command authorization.
