## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract, and the requested change would create a new host-exec trust boundary that needs maintainer/security design before implementation.

## What This Changes

The issue requests a skill-level readonly exec approval scope and bounded command allowlist for healthcheck/canary skills.

## Best Possible Solution

Define an additive approval contract that binds trusted skill identity to explicit argv rules, then update metadata parsing, protocol/schema, node-host evaluation, docs/UI, and security tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection gives a high-confidence verification path: the requested metadata symbols are absent, and current main uses agent-level allowlists plus broad autoAllowSkills.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but the exact trust model should be settled by maintainers because current skill metadata is a load/config contract while host exec auto-approval is a security boundary.
