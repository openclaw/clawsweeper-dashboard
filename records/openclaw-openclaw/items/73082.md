## Summary

Keep open. Current main still lacks a per-skill readonly exec scope or bounded readonly command allowlist, and the requested behavior would change host exec approval trust semantics, so it needs maintainer/security design rather than conservative cleanup closure.

## What This Changes

Request adding `metadata.openclaw.execScope: readonly` plus a bounded `readonlyAllowlist` in `SKILL.md` so healthcheck/canary skills can auto-approve specific read-only commands while mutating and outbound actions remain gated.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv/pattern rules, keeps outbound delivery, filesystem writes, service restarts, cron mutations, and other side effects fail-closed, and updates protocol, config/UI, docs, and targeted approval tests together.

## Reproduction Assessment

Yes, for the missing requested capability. Static inspection of current main reproduces the gap: skill metadata parsing, the exec approvals schema, runtime allowlist evaluation, and docs have no per-skill readonly exec scope or scoped readonly command allowlist.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact `execScope` contract should be decided by maintainers/security because current `autoAllowSkills` is intentionally broader and documented as implicit trust rather than per-skill readonly approval.
