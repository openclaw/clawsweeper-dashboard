## Summary

Keep open: current main still lacks a skill-scoped readonly exec approval contract, and the request would introduce a new host exec trust boundary rather than a conservative cleanup close.

## What This Changes

Add a `metadata.openclaw.execScope: readonly` skill marker with a per-skill readonly command allowlist so canary and healthcheck skills can avoid repeated approvals for known status commands.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv rules, with protocol/schema, docs, UI, and security tests updated together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection gives a high-confidence verification path: the requested metadata symbols are absent, and current approval evaluation is agent-level allowlist plus broad skill-binary auto-allow.

## Solution Assessment

Unclear. The proposed metadata direction is plausible, but maintainers should settle the exact design because it creates a host exec auto-approval boundary based on skill metadata.
