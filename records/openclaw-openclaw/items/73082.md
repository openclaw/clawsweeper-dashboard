## Summary

Keep open: current main still has agent-level exec approvals and `autoAllowSkills`, but no skill-scoped readonly command approval contract. The request would add a new host-exec trust boundary, so it needs maintainer/security design rather than cleanup closure or an automated fix.

## What This Changes

Request a `metadata.openclaw.execScope: readonly` skill metadata field with a per-skill readonly command allowlist for automatic healthcheck/canary skills.

## Best Possible Solution

Design an additive skill-scoped exec approval contract tied to trusted skill identity and explicit argv rules, then update metadata parsing, protocol/schema, runtime evaluation, docs, UI, and security regression coverage together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection of current main is high confidence that OpenClaw has agent-level exec approvals plus `autoAllowSkills`, but no skill-scoped readonly approval contract.

## Solution Assessment

Unclear. The proposed metadata shape is plausible, but the narrow maintainable solution needs maintainer-approved trust semantics for skill identity, argv matching, and fail-closed side-effect exclusions first.
