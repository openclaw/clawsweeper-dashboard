## Summary

Keeping this open: current main still has manual/per-agent exec allowlists, durable command approvals, safe bins, and broad `autoAllowSkills`, but no skill-scoped readonly exec approval contract. Because the request would add a new host-exec auto-approval boundary, it needs maintainer/security design before a repair PR is safe.

## What This Changes

Add a `metadata.openclaw.execScope: readonly` skill metadata contract with a bounded readonly command allowlist for canary/healthcheck skills.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to a trusted skill identity plus explicit argv rules, keeps outbound and mutating operations approval-gated, and updates metadata parsing, approval evaluation, protocol/UI/docs, and focused security tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by searching for the proposed metadata symbols and inspecting skill metadata parsing plus exec approval schema/evaluation.

## Solution Assessment

Unclear. The metadata direction is plausible, but the exact trust model should be settled by maintainers/security because current `autoAllowSkills` is broad binary trust rather than per-skill readonly command authorization.
