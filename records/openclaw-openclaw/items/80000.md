## Summary

Keep open. Current main still has a source-level fallback delivery gap, and the linked implementation candidate is still open rather than merged.

## What This Changes

The issue reports that a successful fallback model retry after a primary timeout is not delivered back to WebChat/TUI sessions.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/80009; after it merges, this issue can close as implemented if the delayed fallback response is delivered without making the optional notification a blocker.

## Reproduction Assessment

Yes from source inspection, not from a live timeout run. Current main can finalize a chat/TUI lifecycle error before a later successful fallback emits final output, matching the reported delayed fallback timeline.

## Solution Assessment

Yes for the current direction. The narrow maintainable fix is to preserve or restore delivery across same-run fallback recovery; an unconditional user-facing recovery notice can remain a separate UX decision.
