## Summary

Keep open. Current main has fallback lifecycle handling, but the source path still appears vulnerable when a per-attempt timeout/error is followed by a fallback success after the 15s lifecycle-error grace; that matches the report's roughly 52s gap between primary timeout and fallback success.

## What This Changes

The issue requests that successful model fallback after an LLM timeout still delivers the final response to WebChat/TUI sessions, with recovery visibility for the user.

## Best Possible Solution

Keep the issue open and add a regression that simulates a fallback success after the lifecycle-error grace, then make chat-linked delivery retain or restore the run mapping until fallback success can emit the final response without duplicating Pi transcript persistence.

## Reproduction Assessment

Yes from source, but not from a live run in this review. The current gateway code finalizes lifecycle errors after 15s while the report's fallback success arrived about 52s later, which is a concrete source-level path to losing the chat-linked final delivery.

## Solution Assessment

Unclear for the exact patch because none is proposed. The best direction is a narrow gateway/agent lifecycle fix and regression test, not a broad new notification system; verbose fallback notices can remain a separate UX decision.
