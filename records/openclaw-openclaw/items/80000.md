## Summary

Keep open. Current main still has a source-level delivery gap for delayed fallback recovery, and an open implementation PR already references this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or rejected.

## What This Changes

Ensure successful fallback model responses after a primary LLM timeout are delivered to WebChat/TUI sessions with recovery visibility.

## Best Possible Solution

Review and land or close https://github.com/openclaw/openclaw/pull/80009; the durable fix should preserve final fallback response delivery and add delayed-fallback regression coverage without making a broad new notification contract mandatory.

## Reproduction Assessment

Yes, source-reproducible. Current main can finalize lifecycle errors before the reported delayed fallback success arrives, and embedded TUI currently treats the first lifecycle error as terminal; no live provider timeout run was performed.

## Solution Assessment

Yes for the current direction. The narrow maintainable solution is to use the existing fallback observation/lifecycle path to keep the run non-terminal until fallback success can deliver the final response; unconditional recovery notices can be a separate UX choice.
