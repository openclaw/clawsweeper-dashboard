## Summary

Keep open: the PR targets a valid route-persistence bug, but the current diff is broader than the reported non-direct case and can bypass the existing direct main-DM route-pinning safeguard.

## What This Changes

The PR adds fallback delivery-context persistence in `recordInboundSession`, rewrites session regression tests around store-backed fallback cases, and adds a changelog entry.

## Best Possible Solution

Land a version that persists fallback delivery context for non-direct inbound sessions while preserving explicit update behavior, same-session context isolation, `createIfMissing`, and direct main-DM pinning coverage.

## Reproduction Assessment

Yes. Source inspection shows current main returns before route persistence when `updateLastRoute` is omitted, and Mattermost non-direct turns omit it while carrying `OriginatingTo`; the proposed patch’s direct-message overreach is also source-reproducible from Feishu/Google Chat/Zalo-style callers.

## Solution Assessment

No. The fallback route-persistence direction is appropriate for the scoped non-direct bug, but the implementation should guard direct contexts or otherwise preserve the main-DM pin policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep fallback route updates out of direct main sessions:** `src/channels/session.ts:56-84`
  - body: This fallback runs whenever a channel omits `updateLastRoute`, including direct-message paths that still provide `OriginatingTo`. With `session.dmScope: "main"`, a non-owner direct message can now write the shared main session `lastRoute` without the existing `mainDmOwnerPin` safeguard, contradicting the documented route-pinning behavior. Please restrict the fallback to non-direct contexts or carry the pin policy into this path.
  - confidence: 0.86
