## Summary

Keep this PR open, but do not merge as-is: the feature is not implemented on current main, and the proposed branch still has concrete config-contract, group-keying, resource-bound, security-disclosure, and real-behavior-proof blockers.

## What This Changes

Adds an opt-in WhatsApp `composingDebounceMs` config that subscribes to Baileys composing presence events and extends inbound debounce buffers via new debouncer touch helpers.

## Best Possible Solution

Land a rebased WhatsApp update only if maintainers accept the opt-in presence-subscription behavior, with schema, generated metadata/help, docs, tests, bounded subscription/cache state, group-key-compatible debounce touching, and real WhatsApp proof completed together.

Yes for the review blockers: source inspection shows the new option is not wired into the strict schema, and the group-key mismatch follows from current debounce key construction versus the PR's E.164 presence prefix. No live WhatsApp proof is present for the feature behavior itself.

## Solution Assessment

No. The direction is plausible, but this is not the best mergeable solution until the config contract, presence subscription bounds/disclosure, group key matching, tests, docs, and real behavior proof are handled together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:79-83`
  - body: The patch adds `composingDebounceMs` to the public WhatsApp type and runtime path, but not to `WhatsAppSharedSchema`. Because the root and account schemas are strict, configs using the new option are rejected as unknown before the feature can run.
  - confidence: 0.95
- **[P2] Use a debounce key compatible with group buffers:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches buffers by an E.164 prefix, but group keys are built from comparable identity or participant JID before E.164. Normal group composing events therefore will not reset the pending buffer.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `presenceSubscribe(remoteJid)` runs for every inbound message while the feature is enabled. Keep per-chat subscription state, and retry on failure, to avoid repeated WhatsApp subscription traffic in active chats.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` stores every distinct remote JID indefinitely even though it only needs to bridge short-lived presence events. Add a TTL or small max-size cap so long-running sessions do not grow this map without bound.
  - confidence: 0.82
