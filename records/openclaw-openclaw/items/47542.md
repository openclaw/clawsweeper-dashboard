## Summary

Keep open. The feature direction is plausible, but the patch is not mergeable as-is: the new config option is not accepted by the strict runtime schema, group composing events can miss the debounce buffer, presence subscription/cache state is unbounded, and the external PR lacks real WhatsApp behavior proof.

## What This Changes

This PR adds a WhatsApp `composingDebounceMs` config path, Baileys presence handling, and `touch`/`touchByPrefix` helpers to extend inbound debouncing while a sender is composing.

## Best Possible Solution

Land a rebased WhatsApp-owned implementation that wires schema/types/docs/tests, matches debounce keys for DMs/groups/LIDs, bounds and deduplicates presence state, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes, source-level for the review blockers. The strict WhatsApp schema lacks `composingDebounceMs`, and the PR branch builds group touch prefixes with a key shape that can differ from the actual debounce buffer key; I did not run a live WhatsApp session.

## Solution Assessment

No. The opt-in composing debounce direction is reasonable, but this implementation needs schema wiring, key matching, bounded presence bookkeeping, a rebase over current monitor lifecycle code, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the option into the strict WhatsApp schema:** `src/config/types.whatsapp.ts:85`
  - body: `composingDebounceMs` is added to the TypeScript config type, but `buildWhatsAppCommonShape` in the strict WhatsApp zod schema still only accepts `debounceMs`. A user setting `channels.whatsapp.composingDebounceMs` or an account override will be rejected as an unknown key before this runtime path can be enabled.
  - confidence: 0.96
- **[P2] Match group buffers with the actual sender key:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: The presence handler touches group buffers with a prefix ending in the sender's E.164 value, but group debounce keys are built from `senderJid` before `senderE164` on this branch. In normal group traffic where the participant JID is present, the composing event will not touch the buffered message, so the feature silently fails for those group senders.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:402`
  - body: When composing debounce is enabled, this calls `presenceSubscribe(remoteJid)` for every inbound message in the chat. Baileys sends a subscription request for the JID, so active conversations will issue redundant network work and can add avoidable rate-limit/noise risk; track subscribed JIDs and retry only after failure or reconnect.
  - confidence: 0.84
- **[P2] Bound the LID to phone mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: `jidToE164Cache` is populated from inbound messages and never expires or caps entries. It only needs to bridge the short gap until a presence event arrives, so long-running gateways with many WhatsApp contacts can accumulate stale mappings unless this gets a TTL or max-size eviction policy.
  - confidence: 0.82
