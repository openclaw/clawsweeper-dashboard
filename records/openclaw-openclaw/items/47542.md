## Summary

Keep open. The feature is not implemented on current main and is not superseded, but this PR cannot merge as written because the new config is rejected by the strict runtime schema, group composing events miss the buffered keys, and real WhatsApp behavior proof is absent.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, subscribes to Baileys composing presence updates, and extends the shared inbound debouncer with touch helpers.

## Best Possible Solution

Keep the PR open for follow-up, but require a rebased WhatsApp-owned implementation that wires schema/types/account/docs/tests, uses the correct direct/group debounce keys, bounds subscriptions and caches, and includes redacted live WhatsApp proof; keep timestamp coalescing in the separate open issue.

## Reproduction Assessment

Not applicable for the feature request itself. The review findings are source-reproducible from the PR head: the strict WhatsApp schema rejects the new config key, and the group presence prefix does not match the branch’s group debounce key format.

## Solution Assessment

No. The direction is plausible, but this is not the best mergeable solution until the runtime schema, group keying, subscription/cache bounds, tests, rebase, and real WhatsApp proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wire the setting into the runtime schema:** `src/config/types.whatsapp.ts:85`
  - body: The new option is only added to TypeScript types and account plumbing, but `WhatsAppConfigSchema` and `WhatsAppAccountSchema` are strict and `WhatsAppSharedSchema` still only declares `debounceMs`. Any user who sets `channels.whatsapp.composingDebounceMs` or an account override will get an unknown-key validation error before the feature can run.
  - confidence: 0.94
- **[P2] Use keys that match group debounce buffers:** `extensions/whatsapp/src/inbound/monitor.ts:490`
  - body: For group messages this branch keys buffers as `account:groupJid:senderJid` when `participantJid` exists, but the presence handler touches `account:groupJid:e164`. That means composing events will not extend the pending group sender buffer for normal group traffic.
  - confidence: 0.89
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:402-404`
  - body: With composing debounce enabled, every accepted inbound message calls `sock.presenceSubscribe(remoteJid)`. Baileys sends a subscribe request for that JID, so active conversations can generate redundant network calls; track subscribed JIDs and retry only after a failed subscribe.
  - confidence: 0.82
- **[P3] Bound the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: `jidToE164Cache` is never evicted, even though it only needs to bridge the short window between a message and presence update. Long-running WhatsApp sessions with many distinct senders can retain stale entries indefinitely; add a TTL or max-size cap.
  - confidence: 0.78
