## Summary

Keep open. Current main does not implement composing-aware WhatsApp inbound debounce, but this external feature PR is not merge-ready because the opt-in config is not wired through strict schemas, group composing touches can miss active buffers, presence subscription/cache handling is unbounded, the security section understates new network behavior, and no real WhatsApp proof is provided.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` config, account plumbing, Baileys presence-update handling, and shared inbound debouncer touch helpers.

## Best Possible Solution

Land a rebased WhatsApp-owned opt-in implementation only after strict schema/config metadata/docs, group/direct key matching, bounded and deduplicated presence handling, focused tests, accurate security notes, and redacted real WhatsApp proof are complete.

## Reproduction Assessment

Yes for the PR defects: source and PR-diff inspection show the strict schema gap, group-key mismatch, repeated subscription path, and unbounded cache. No live WhatsApp runtime behavior is proven because the PR supplies no real setup proof.

## Solution Assessment

No. Presence-aware debounce is a plausible WhatsApp-owned direction, but this patch is not the narrowest maintainable merge path until schema support, group matching, bounded and deduplicated presence handling, security disclosure, tests, and real-behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire composingDebounceMs into strict schemas:** `src/config/types.whatsapp.ts:82-85`
  - body: Adding only the TypeScript config field leaves the strict WhatsApp zod schemas unaware of `composingDebounceMs`. User config that sets the root or per-account key will be rejected as an unknown property, so the opt-in feature cannot be enabled.
  - confidence: 0.97
- **[P2] Match group composing touches to debounce keys:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For groups, debounce buffers prefer identity or participant JID before E.164, but the presence handler touches only an E.164 prefix. Normal group traffic with a participant JID or LID will not match the active buffer, so composing indicators do not extend the debounce window for that sender.
  - confidence: 0.91
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:402`
  - body: This calls `presenceSubscribe(remoteJid)` from the inbound path whenever `composingDebounceMs` is set, including repeated messages from the same chat. Active conversations can generate unnecessary WhatsApp network requests, so guard by active debounce state and subscribed JID, with retry after failure.
  - confidence: 0.88
- **[P2] Bound the JID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` keeps one entry per distinct JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running listeners.
  - confidence: 0.84
