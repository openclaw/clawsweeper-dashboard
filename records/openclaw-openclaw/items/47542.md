## Summary

Keep open: current main does not implement composing-aware WhatsApp inbound debounce, and this external feature PR still has blocking schema, group-key, presence-subscription, cache-bounding, security-disclosure, and real-behavior-proof gaps.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, account/runtime plumbing, Baileys presence subscription/update handling, and shared inbound debouncer touch helpers.

## Best Possible Solution

Land a rebased WhatsApp-owned opt-in implementation only after schema/config/docs support, direct and group key matching, bounded and deduplicated presence handling, focused tests, accurate security notes, and redacted real WhatsApp proof are complete.

## Reproduction Assessment

Yes for the PR defects: source and diff inspection show the strict schema gap, group key mismatch, repeated subscription path, and unbounded cache. No live WhatsApp runtime behavior is proven because the PR supplies no real setup proof.

## Solution Assessment

No. Presence-aware debounce is a plausible WhatsApp-owned direction, but this patch is not the narrowest maintainable merge path until schema support, key matching, bounded/deduplicated presence handling, security disclosure, tests, and real-behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire composingDebounceMs into strict schemas:** `src/config/types.whatsapp.ts:82-85`
  - body: The new config field is added to the WhatsApp TypeScript type but not to `WhatsAppSharedSchema`. Because both root and per-account WhatsApp schemas are strict, user config that sets `composingDebounceMs` is rejected as unknown, so the opt-in feature cannot be enabled.
  - confidence: 0.97
- **[P2] Match group composing touches to debounce keys:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For groups, active debounce buffers prefer primary identity or `senderJid` before E.164, but the presence handler touches only an E.164 prefix. Normal group traffic with participant JIDs or LIDs will not match the active buffer, so composing indicators do not extend that sender's debounce window.
  - confidence: 0.91
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:397-404`
  - body: This calls `presenceSubscribe(remoteJid)` from the inbound path whenever composing debounce is enabled, including repeated messages from the same chat. Active conversations can generate unnecessary WhatsApp network requests and rate-limit noise, so subscription attempts need a per-chat guard with retry after failure.
  - confidence: 0.88
- **[P2] Bound the JID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` keeps one entry per distinct JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running listeners.
  - confidence: 0.84
