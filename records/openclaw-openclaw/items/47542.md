## Summary

Keep open. Current main still lacks composing-aware WhatsApp inbound debounce, but this PR is not merge-ready because the new config cannot pass strict schema validation, group composing touches can miss active debounce buffers, presence subscription/cache handling is unbounded, real WhatsApp proof is missing, and the security section understates changed presence-subscription traffic.

## What This Changes

The branch adds an opt-in WhatsApp `composingDebounceMs` setting, propagates it into the Web monitor, subscribes to Baileys presence updates, and adds shared inbound debouncer touch helpers.

## Best Possible Solution

Land a rebased WhatsApp-owned opt-in implementation only after schema/config/docs/test support, direct and group key matching, bounded and deduplicated presence handling, accurate network disclosure, and redacted real WhatsApp proof.

## Reproduction Assessment

Yes for the PR blockers: source and diff inspection show the strict schema gap, group key mismatch, repeated subscription path, and unbounded cache. No live WhatsApp after-fix behavior is proven because the Human Verification section is blank.

## Solution Assessment

No. Presence-aware debounce is a plausible WhatsApp-owned direction, but this patch is not the narrowest maintainable merge path until schema support, key matching, bounded presence handling, disclosure, tests, and real proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire composingDebounceMs into strict schemas:** `src/config/types.whatsapp.ts:82-85`
  - body: The new config field is added only to the WhatsApp TypeScript type. Because the root and per-account WhatsApp schemas are strict and still expose only `debounceMs`, user config that enables `composingDebounceMs` is rejected as an unknown key, so the feature cannot be enabled.
  - confidence: 0.97
- **[P2] Use debounce-compatible keys for group composing touches:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For group messages, active debounce buffers prefer `senderJid` before E.164, but the presence handler touches only an E.164-based prefix. Normal group composing updates therefore do not extend the matching sender's debounce buffer.
  - confidence: 0.91
- **[P2] Deduplicate presence subscriptions:** `extensions/whatsapp/src/inbound/monitor.ts:397-404`
  - body: The inbound path calls `presenceSubscribe(remoteJid)` whenever composing debounce is configured, including repeated messages from the same chat. Guard it by per-chat subscription tracking with retry after failure to avoid repeated WhatsApp subscription traffic.
  - confidence: 0.89
- **[P2] Bound the LID-to-E164 cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: `jidToE164Cache` grows one entry per distinct mapped JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running listeners.
  - confidence: 0.84
