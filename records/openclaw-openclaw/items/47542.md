## Summary

Keep open: current main does not implement composing-aware inbound debounce, and the PR is still useful, but this patch is not mergeable as written because the opt-in config is not accepted by the runtime schema, group composing touches can miss active buffers, presence subscription/cache handling is unbounded or repeated, the security disclosure is inaccurate, and contributor real-behavior proof is missing.

## What This Changes

The PR adds a WhatsApp `composingDebounceMs` option, Baileys presence subscription/update handling, and shared debouncer touch helpers to extend inbound debounce while a sender is composing.

## Best Possible Solution

Land a WhatsApp-owned opt-in composing debounce only after schema/config metadata, group key matching, bounded and deduped presence handling, focused tests, accurate security notes, and redacted real WhatsApp proof are complete.

## Reproduction Assessment

Yes for the PR defects: source and PR-diff inspection show the missing strict schema entry, group key mismatch, repeated subscription path, and unbounded cache. No live WhatsApp runtime behavior was reproduced because the PR supplies no real setup proof.

## Solution Assessment

No. Presence-aware debounce is a plausible direction, but this implementation is not the narrowest maintainable merge path until schema support, group matching, bounded/deduped presence handling, security disclosure, tests, and real-behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire composingDebounceMs into strict WhatsApp schemas:** `src/config/types.whatsapp.ts:82-85`
  - body: Adding only the TypeScript config field leaves the strict WhatsApp zod schemas unaware of `composingDebounceMs`. A user setting the new root or account key will be rejected as an unknown property, so the opt-in feature cannot be enabled from config.
  - confidence: 0.97
- **[P2] Match group composing touches to queued sender keys:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For groups, debounce buffers prefer an identity/JID sender key before E.164, but the presence handler touches only an E.164 prefix. Normal group traffic with a participant JID or LID will not match the active buffer, so composing indicators do not extend the debounce window for that sender.
  - confidence: 0.91
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:402`
  - body: This calls `presenceSubscribe(remoteJid)` on every inbound message while composing debounce is configured. Active chats can generate repeated WhatsApp network requests and possible rate limiting; track subscribed JIDs and retry after failures.
  - confidence: 0.86
- **[P2] Bound the JID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` keeps one entry per distinct JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running processes.
  - confidence: 0.84
