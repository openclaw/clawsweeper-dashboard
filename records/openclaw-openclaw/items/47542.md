## Summary

Keep open: current main does not implement the requested behavior, and the PR still has merge-blocking config validation, group debounce-key, unbounded subscription/cache, and security disclosure issues.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, wires it into account resolution, subscribes to Baileys composing presence events, and extends inbound debounce buffers with new touch helpers.

## Best Possible Solution

Land a rebased WhatsApp update only after maintainers accept the opt-in presence-subscription behavior and the PR wires schema/help/docs/changelog/tests with bounded subscription/cache state and direct/group debounce-key matching.

## Reproduction Assessment

Yes for the review blockers. Source inspection shows strict config validation rejects the new option, the group presence prefix misses current group buffer keys, and the PR patch repeats presence subscriptions while keeping an unbounded JID cache.

## Solution Assessment

No. The feature direction is plausible, but this implementation is not yet the narrowest maintainable solution because it omits config/docs/test follow-through and adds unbounded, under-disclosed presence subscription behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:79-83`
  - body: The patch adds `composingDebounceMs` to the public WhatsApp type and runtime path, but not to `buildWhatsAppCommonShape`. Because the root and account schemas are strict, configs using the new option are rejected as unknown before the feature can run.
  - confidence: 0.95
- **[P2] Use the existing group debounce key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches buffers by an E.164 prefix, but group keys are built from comparable identity or participant JID before E.164. Normal group composing events therefore will not reset the pending buffer.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `presenceSubscribe(remoteJid)` runs for every inbound message while the feature is enabled. Keep per-chat subscription state, and retry on failure, to avoid repeated WhatsApp subscription traffic in active chats.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` stores every distinct remote JID indefinitely even though it only needs to bridge short-lived presence events. Add a TTL or small max-size cap so long-running sessions do not grow this map without bound.
  - confidence: 0.82
- **[P3] Add the user-facing changelog entry:** `src/config/types.whatsapp.ts:79-83`
  - body: This adds a new WhatsApp config option and behavior, but `CHANGELOG.md` is untouched. Add a single active-version entry with appropriate human contributor credit if the feature is accepted.
  - confidence: 0.84

## Security Review
