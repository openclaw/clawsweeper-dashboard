## Summary

Keep open: current main does not implement composing-aware WhatsApp inbound debounce, and this feature PR still has merge blockers around strict config validation, group debounce-key matching, unbounded presence subscription/cache behavior, and user-facing config follow-through. Existing Greptile and ClawSweeper review comments correctly identify important operational blockers; this review confirms those blockers against current main and the PR diff. PR URL: https://github.com/openclaw/openclaw/pull/47542

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting that subscribes to Baileys composing presence events and extends inbound debounce timers via new debouncer touch helpers.

## Best Possible Solution

Land a rebased WhatsApp update only if maintainers accept the opt-in presence-subscription behavior, with schema, generated config metadata/help, docs, changelog, tests, bounded subscription/cache state, and direct/group debounce-key matching completed together.

## Reproduction Assessment

Yes for the review blockers. Source inspection shows the strict schema omission rejects the new option, and the group-key mismatch follows from current group buffer construction versus the PR's E.164 presence prefix.

## Solution Assessment

No. The feature direction is plausible, but this PR is not yet the narrowest maintainable solution because it omits required config/docs/test follow-through and introduces unbounded, undisclosed presence subscription behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:79-83`
  - body: The patch adds `composingDebounceMs` to the public WhatsApp type and runtime path, but not to `WhatsAppSharedSchema`. Because the root and account schemas are strict, configs using the new option are rejected as unknown before the feature can run.
  - confidence: 0.95
- **[P2] Use the existing group debounce key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches buffers by an E.164 prefix, but group keys are built from comparable identity or participant JID before E.164. Normal group composing events therefore will not reset the pending buffer.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `presenceSubscribe(remoteJid)` runs for every inbound message while the feature is enabled. Keep per-chat subscription state, and retry on failure, to avoid repeated WhatsApp subscription traffic in active chats.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` stores every distinct remote JID indefinitely even though it only needs to bridge short-lived presence events. Add a TTL or small max-size cap so long-running sessions do not grow this map without bound.
  - confidence: 0.82
- **[P3] Add the user-facing changelog entry:** `src/config/types.whatsapp.ts:79-83`
  - body: This adds a new WhatsApp config option and behavior, but `CHANGELOG.md` is untouched. Add a single active-version entry with appropriate human contributor credit if the feature is accepted.
  - confidence: 0.84
