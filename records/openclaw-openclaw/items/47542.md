## Summary

Keep open; this PR is not safe to merge as-is. Current main does not already provide the composing-aware WhatsApp debounce, and the proposed branch has blocking config/runtime defects plus missing real behavior proof.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, account/runtime plumbing, Baileys presence subscriptions, and shared debouncer touch helpers to extend inbound batching while a sender is composing.

## Best Possible Solution

Land a WhatsApp-owned opt-in composing debounce only after the config schema, account/runtime plumbing, group sender-key matching, bounded/deduped presence handling, focused tests, and redacted real WhatsApp proof are all in place.

## Reproduction Assessment

Yes for the blocking PR defects: source inspection shows the new config key is absent from the strict WhatsApp zod schema, and the group touch prefix does not match the current group debounce key order. Live WhatsApp behavior was not reproduced because the PR provides no real setup proof.

## Solution Assessment

No. The direction is plausible, but this implementation is not the best mergeable form until schema support, group key matching, bounded presence subscription/cache behavior, and real WhatsApp proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire composingDebounceMs into the runtime schema:** `src/config/types.whatsapp.ts:83`
  - body: Adding only the TypeScript config field leaves `WhatsAppConfigSchema` unaware of it. Because the root and account WhatsApp schemas are strict, any user setting `channels.whatsapp.composingDebounceMs` or `accounts.*.composingDebounceMs` will be rejected as an unknown key, so the opt-in feature cannot be enabled from config.
  - confidence: 0.97
- **[P2] Match group composing touches to the buffered sender key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: For groups, the current debounce key prefers the sender's primary identity or `senderJid` before `senderE164`, but the presence handler touches only `account:chat:e164`. In normal group traffic with a participant JID, that prefix will not match the active buffer, so composing indicators will not extend the debounce window for group senders.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:397-404`
  - body: This calls `presenceSubscribe(remoteJid)` on every inbound message while composing debounce is configured, even though a chat only needs one active subscription. Active chats can generate repeated WhatsApp network requests and possible rate limiting; track subscribed JIDs and retry after subscription failures.
  - confidence: 0.84
- **[P2] Bound the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:76`
  - body: The new `jidToE164Cache` grows for every distinct JID that sends a message and is never expired. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap so long-running WhatsApp processes do not accumulate stale contacts indefinitely.
  - confidence: 0.83
