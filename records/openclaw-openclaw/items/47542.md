## Summary

Keep open. The PR is a plausible WhatsApp feature direction, but the current patch is not mergeable as-is: user config would be rejected by the strict schemas, group composing updates can miss the active debounce buffer, presence subscriptions are not bounded/deduplicated, real WhatsApp proof is absent, and the security section does not disclose the new subscription traffic.

## What This Changes

Adds an opt-in WhatsApp `composingDebounceMs` setting, Baileys presence handling, account plumbing, and shared inbound debouncer touch helpers.

## Best Possible Solution

Land a rebased WhatsApp-owned opt-in implementation after schema/config metadata/docs/test support, direct and group key matching, bounded and deduplicated presence handling, accurate network disclosure, and redacted real WhatsApp proof.

## Reproduction Assessment

Yes for the PR defects: source and diff inspection show the strict schema gap, group key mismatch, repeated subscription path, and unbounded cache. No live WhatsApp behavior is proven because the Human Verification section is blank.

## Solution Assessment

No. Presence-aware debounce is a plausible WhatsApp-owned direction, but this patch is not the narrowest maintainable merge path until schema support, key matching, bounded presence handling, security disclosure, tests, and real behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wire composingDebounceMs into strict schemas:** `src/config/types.whatsapp.ts:82-85`
  - body: The new setting is only added to the WhatsApp TypeScript type. Because the root and per-account WhatsApp schemas are strict and still expose only `debounceMs`, user config that sets `channels.whatsapp.composingDebounceMs` or an account override is rejected as an unknown key, so the feature cannot be enabled.
  - confidence: 0.97
- **[P2] Use debounce-compatible keys for group composing touches:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For group messages, active debounce buffers are keyed with `senderJid` before E.164, but the presence handler touches only an E.164-based prefix. Normal group composing updates therefore do not extend the matching sender's debounce buffer.
  - confidence: 0.91
- **[P2] Deduplicate presence subscriptions:** `extensions/whatsapp/src/inbound/monitor.ts:397-402`
  - body: The inbound path calls `presenceSubscribe(remoteJid)` whenever composing debounce is configured, including repeated messages from the same chat and cases where no debounce listener is registered because `debounceMs` is zero. Guard this by active debounce state and per-chat subscription tracking with retry after failure.
  - confidence: 0.89
- **[P2] Bound the LID-to-E164 cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: `jidToE164Cache` grows one entry per distinct mapped JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running listeners.
  - confidence: 0.84
