## Summary

Keep open. Current main does not implement this WhatsApp composing-aware debounce feature, and the PR direction is plausible, but the current head still has merge-blocking schema/runtime defects, missing real WhatsApp proof, and an under-disclosed presence subscription network path.

## What This Changes

The PR adds a WhatsApp `composingDebounceMs` option, account/runtime plumbing, Baileys presence subscriptions, and debouncer touch helpers to extend inbound batching while a sender is composing.

## Best Possible Solution

Land a WhatsApp-owned opt-in composing debounce only after schema/config metadata, group key matching, bounded and deduped presence handling, focused tests, security notes, and redacted real WhatsApp proof are complete.

## Reproduction Assessment

Yes for the PR defects: PR-head source shows strict zod schemas omit `composingDebounceMs`, group touch prefixes use E.164 while group buffers prefer sender JIDs, and the presence subscription/cache path is unbounded. No live WhatsApp runtime behavior was reproduced because the PR supplies no real setup proof.

## Solution Assessment

No. Presence-aware debounce is a plausible direction, but this implementation is not mergeable until schema support, group matching, bounded/deduped presence handling, accurate security disclosure, and real-behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire composingDebounceMs into the runtime schema:** `src/config/types.whatsapp.ts:82-85`
  - body: Adding only the TypeScript config field leaves the strict WhatsApp zod schemas unaware of `composingDebounceMs`. Any user setting the new root or account key will be rejected as an unknown property, so the opt-in feature cannot be enabled from config.
  - confidence: 0.97
- **[P2] Match group composing touches to queued sender keys:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For groups, the buffer key prefers `senderJid` before `senderE164`, but the presence handler touches only an E.164 prefix. Normal group traffic with a participant JID will not match the active buffer, so composing indicators do not extend the debounce window for that sender.
  - confidence: 0.91
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:402`
  - body: This calls `presenceSubscribe(remoteJid)` on every inbound message while composing debounce is configured. Active chats can generate repeated WhatsApp network requests and possible rate limiting; track subscribed JIDs and retry after failures.
  - confidence: 0.86
- **[P2] Bound the JID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` keeps one entry per distinct JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running processes.
  - confidence: 0.84
