## Summary

Keep open. Current main still lacks composing-aware WhatsApp debounce, and the PR has real pre-merge blockers: strict config validation is not wired, group presence touches can miss the existing debounce key, and the new presence subscription path is currently unbounded.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, account/runtime propagation, Baileys presence handling, and `touch`/`touchByPrefix` helpers on the inbound debouncer.

## Best Possible Solution

Land a rebased WhatsApp update that wires `composingDebounceMs` through schema, generated metadata/help, docs, changelog, account resolution, runtime handling, and tests while bounding presence subscription/cache behavior and matching the existing direct/group debounce keys.

## Reproduction Assessment

Yes for the review blockers. Source inspection shows the strict schema omission would reject the new option, and the group-key mismatch follows from the current monitor's sender-key construction versus the PR's E.164 presence prefix.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not yet the narrowest maintainable solution because it omits required config/docs/test follow-through and introduces unbounded presence subscription behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:79-83`
  - body: The PR adds `composingDebounceMs` to the public WhatsApp type and runtime reads, but not to `WhatsAppSharedSchema`. Since the root and account schemas are strict, configs using the new option are rejected before the feature can run.
  - confidence: 0.95
- **[P2] Match group buffers with the existing sender key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches buffers by an E.164 prefix, but group debounce keys prefer the comparable sender identity or participant JID before E.164. For normal group messages, composing events will not extend the buffered message window.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `sock.presenceSubscribe(remoteJid)` is called for every inbound message while the feature is enabled. Keep per-chat subscription state and retry on failure to avoid unnecessary WhatsApp presence traffic and rate-limit exposure.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:76`
  - body: The new JID-to-E.164 map stores every distinct sender indefinitely, but it only needs to bridge short-lived presence events. Add a TTL or max-size cap so long-running sessions do not grow this cache without bound.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/types.whatsapp.ts:79-83`
  - body: This is a user-facing WhatsApp feature with a new config option, but the PR does not update `CHANGELOG.md`. Add a single-line entry in the active version section with appropriate human contributor credit.
  - confidence: 0.84
