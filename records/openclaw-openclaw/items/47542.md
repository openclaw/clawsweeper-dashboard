## Summary

Keep open. Current main still lacks the requested WhatsApp composing-aware debounce behavior, and the PR is a useful but incomplete implementation with blocking config, keying, subscription-bounding, and cache-lifetime issues. The provided PR state also shows the branch is dirty against main, whose WhatsApp inbound monitor has since been refactored.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, passes it through account and auto-reply setup, subscribes to Baileys presence updates, and extends the inbound debouncer with `touch` and `touchByPrefix` helpers.

## Best Possible Solution

Land a rebased, focused WhatsApp update that adds `composingDebounceMs` consistently to types, strict schema, generated/help metadata, docs, changelog, account resolution, and runtime wiring; adds debouncer touch tests; matches presence events to the same direct and group keys used by the monitor; deduplicates presence subscriptions; and expires the short-lived JID mapping cache.

## Reproduction Assessment

Yes for the review blockers. Source inspection shows a config with `composingDebounceMs` would hit strict WhatsApp schema validation before runtime, and the group-key mismatch follows directly from the PR's E.164 prefix versus the current sender-identity/JID debounce key.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not yet the narrowest maintainable solution because it omits schema/docs/changelog/test coverage and introduces unbounded presence subscription/cache behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:79-83`
  - body: The PR adds `composingDebounceMs` to the public WhatsApp config type and runtime reads, but not to the strict WhatsApp zod schema. A user config that sets the option will be rejected as an unknown key before the feature can run.
  - confidence: 0.95
- **[P2] Match group buffers with the same sender key:** `extensions/whatsapp/src/inbound/monitor.ts:486-487`
  - body: The presence handler touches group buffers using an E.164 prefix, while WhatsApp group debounce keys prefer the sender identity or participant JID before E.164. In normal group traffic, composing events will not match the buffered message key, so the debounce window is not extended.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `sock.presenceSubscribe(remoteJid)` is called for every inbound message while the feature is enabled. Keep a per-chat subscription guard and retry on failure to avoid unnecessary WhatsApp presence traffic and rate-limit exposure.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:76`
  - body: `jidToE164Cache` records every distinct JID indefinitely, but the mapping only needs to bridge the short interval between an inbound message and its presence events. Add a TTL or max-size cap so long-running sessions do not grow this map without bound.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/types.whatsapp.ts:79-83`
  - body: This is a user-facing WhatsApp feature with a new config option, but the PR does not update `CHANGELOG.md`. Add a single-line entry in the active version section with contributor credit.
  - confidence: 0.86
