## Summary

Keep open: current main does not contain the composing-aware WhatsApp debounce feature, and this PR still has blocking schema, group-key matching, presence subscription/cache, security disclosure, and real-behavior-proof gaps.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, Baileys presence update handling, and shared debouncer `touch` helpers to extend inbound debounce while a sender is composing.

## Best Possible Solution

Land this only after schema/config metadata, group sender matching, bounded and deduped presence handling, focused coverage, accurate security notes, and redacted real WhatsApp proof are complete.

## Reproduction Assessment

Yes for the PR defects: source and PR-diff inspection show the strict schema gap, group-key mismatch, repeated subscription path, and unbounded cache. No live WhatsApp runtime behavior is proven because the PR supplies no real setup proof.

## Solution Assessment

No. Presence-aware debounce is a plausible WhatsApp-owned direction, but this implementation is not the narrowest maintainable merge path until schema support, group matching, bounded/deduped presence handling, security disclosure, tests, and real-behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire composingDebounceMs into strict WhatsApp schemas:** `src/config/types.whatsapp.ts:82-85`
  - body: Adding only the TypeScript config field leaves the strict WhatsApp zod schemas unaware of `composingDebounceMs`. A user setting the root or per-account key will be rejected as an unknown property, so the opt-in feature cannot be enabled from config.
  - confidence: 0.97
- **[P2] Match group composing touches to queued sender keys:** `extensions/whatsapp/src/inbound/monitor.ts:489-490`
  - body: For groups, debounce buffers prefer an identity or participant JID sender key before E.164, but the presence handler touches only an E.164 prefix. Normal group traffic with a participant JID or LID will not match the active buffer, so composing indicators do not extend the debounce window for that sender.
  - confidence: 0.91
- **[P2] Gate and deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:402`
  - body: This calls `presenceSubscribe(remoteJid)` from the inbound path whenever `composingDebounceMs` is set, including repeated messages from the same chat and cases where debounce is not active. Active chats can generate unnecessary WhatsApp network requests; guard it by active debounce state and subscribed JID, with retry after failure.
  - confidence: 0.88
- **[P2] Bound the JID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: The new `jidToE164Cache` keeps one entry per distinct JID and never expires entries. Since the mapping only needs to bridge nearby composing events, use a short TTL or size cap to avoid stale contact growth in long-running processes.
  - confidence: 0.84
