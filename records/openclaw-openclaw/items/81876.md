## Summary

Keep open: current main still implements and documents the post-bootstrap `pairing` default, and this is a valid security-default/product request rather than an implemented, duplicate, or stale cleanup item.

## What This Changes

The issue requests changing implicit DM channel defaults after first-owner bootstrap so unknown direct senders stop receiving pairing replies unless pairing was explicitly configured.

## Best Possible Solution

Decide the post-bootstrap security default explicitly, then implement the chosen policy centrally with per-channel/account tests and docs updates.

## Reproduction Assessment

Yes for source-level reproduction: unset `dmPolicy` resolves to `pairing`, and an unallowlisted direct sender with `mayPair` reaches `dm_policy_pairing_required`. I did not run a live channel because current source and docs directly show the behavior.

## Solution Assessment

No for an automated fix: the requested direction is plausible hardening, but it changes documented cross-channel authorization defaults and needs maintainer agreement on owner mapping and group-policy semantics.
