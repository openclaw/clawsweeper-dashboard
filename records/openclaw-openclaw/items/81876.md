## Summary

Keep open: current main still follows the documented pairing default after first-owner bootstrap, and the requested auto-flip is a cross-channel security-default/product decision rather than cleanup that can be closed or safely patched automatically.

## What This Changes

Request auto-flipping unset channel DM policy from `pairing` to an owner-derived allowlist, with groups disabled by default, once `commands.ownerAllowFrom` exists.

## Best Possible Solution

Decide the post-bootstrap security default explicitly, then implement it centrally in shared ingress, setup, or doctor policy code with per-channel/account tests and docs updates.

## Reproduction Assessment

Yes for source-level reproduction: unset `dmPolicy` resolves to `pairing`, and an unallowlisted direct sender with `mayPair` reaches the pairing-required path. I did not run a live channel because the current source and docs directly show the behavior.

## Solution Assessment

No for an automated fix: the requested direction is plausible hardening, but it changes a documented cross-channel default and needs maintainer agreement on owner mapping and group-policy semantics.
