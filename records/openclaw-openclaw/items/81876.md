## Summary

Keep open: current main still documents and implements pairing as the default DM policy, and the requested owner-derived auto-flip is a cross-channel default-policy/security-hardening change rather than an already implemented cleanup.

## What This Changes

Request to treat omitted channel DM policy as an owner-derived allowlist after first pairing bootstraps `commands.ownerAllowFrom`, while preserving explicit `dmPolicy` settings.

## Best Possible Solution

After maintainer approval, implement a shared effective-policy layer that derives same-channel owner allowlist entries only when `dmPolicy` is truly omitted, keeps explicit pairing intact, and adds regression coverage plus docs for the post-bootstrap default.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-reproducible: current main defaults omitted DM policy to `pairing` and maps unknown direct senders to `pairing-required`; I did not run a live channel bridge.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer product/security review decides whether `commands.ownerAllowFrom` should implicitly become a DM allowlist and how group policy should behave after bootstrap.

## Risks / Open Questions

- Changing this default affects every DM-capable channel and may surprise operators who intentionally rely on post-bootstrap pairing without explicitly setting `dmPolicy`.
- The implementation must distinguish an omitted `dmPolicy` from schema/defaulted `"pairing"`; several channel schemas currently apply `optional().default("pairing")`.
- Deriving `allowFrom` from `commands.ownerAllowFrom` needs per-channel normalization so entries like `telegram:123` do not accidentally authorize the wrong channel or account.
