## Summary

Keep open: current main and the latest release still default unset channel DM policy to `pairing` after first-owner bootstrap, so this is a valid security-default/product request rather than stale, implemented, or safely duplicate cleanup.

## What This Changes

The issue requests an automatic post-bootstrap effective default from `dmPolicy: pairing` to owner-derived `allowlist` for DM-capable channels when `commands.ownerAllowFrom` is populated and the channel has no explicit `dmPolicy`.

## Best Possible Solution

Make a maintainer product/security decision for the shared channel ingress model, then implement one cross-channel path with explicit-DM-policy precedence, first-run pairing preserved, owner-derived allowlist behavior tested, and docs updated.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: leave a DM-capable channel without explicit `dmPolicy`, approve the first pairing so `commands.ownerAllowFrom` is populated, then an unknown direct sender still follows the shared `pairing` default and receives the pairing-required path. I did not run a live channel because this review is read-only.

## Solution Assessment

Unclear; the requested auto-flip is a plausible security-default direction, but the best fix needs maintainer agreement on cross-channel precedence, owner ID derivation, multi-account behavior, and whether `groupPolicy` should also change implicitly.
