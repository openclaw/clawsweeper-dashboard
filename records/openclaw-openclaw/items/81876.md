## Summary

Keep open: current main still implements and documents the pairing default after first-owner bootstrap, and the requested auto-flip is a cross-channel security-default/product decision rather than conservative cleanup.

## What This Changes

The issue requests that omitted DM policies auto-flip after first-owner bootstrap from pairing to an owner-derived allowlist, with groups disabled unless explicitly configured.

## Best Possible Solution

Make an explicit maintainer decision on the post-bootstrap DM security default, then implement any accepted change centrally in shared ingress/setup/doctor policy with channel/account coverage and docs updates.

## Reproduction Assessment

Yes for a source-level path: omitted `dmPolicy` resolves to `pairing`, and an unknown direct sender reaches the pairing-required branch when pairing is allowed. I did not run a live channel because the shared resolver and docs directly show the behavior.

## Solution Assessment

No for an automated fix: the requested hardening is plausible, but it changes a documented cross-channel default and needs maintainer agreement on owner mapping and group-policy semantics.
