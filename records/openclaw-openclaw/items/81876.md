## Summary

Keep open: current main and v2026.6.6 still resolve unset channel DM policy to `pairing` after first-owner bootstrap, while bootstrap only writes `commands.ownerAllowFrom`. The related Telegram and one-line-bootstrap issues are still open and narrower/broader respectively, so they do not supersede this cross-channel security-default decision.

## What This Changes

The issue requests changing implicit DM defaults after first-owner bootstrap so unset channel DM policy behaves like an owner allowlist instead of continuing to offer pairing replies to unknown DM senders.

## Best Possible Solution

Make a maintainer product/security decision for the shared channel ingress model, then implement one explicit-policy-preserving path with owner-derived allowlist behavior, upgrade handling, docs, and cross-channel regression tests.

## Reproduction Assessment

Yes, at source level: leave a DM-capable channel without explicit `dmPolicy`, approve the first pairing so `commands.ownerAllowFrom` is populated, and the shared resolver still uses `pairing` while the sender gate returns `dm_policy_pairing_required` for a new unknown DM. I did not run a live channel path in this read-only review.

## Solution Assessment

Unclear: the requested auto-flip is a plausible security-default direction, but the best fix needs maintainer agreement on cross-channel precedence, owner ID derivation, multi-account behavior, explicit `dmPolicy: "pairing"`, and whether group policy should change implicitly.
