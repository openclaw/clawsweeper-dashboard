## Summary

Keep open: current main still treats unset DM policy as `pairing`, and changing that after owner bootstrap would be a cross-channel default-policy change rather than a broken-contract bug fix.

## What This Changes

Request auto-flipping unset DM-capable channel defaults from pairing to owner allowlist after first-owner bootstrap.

## Best Possible Solution

Decide the post-bootstrap security default explicitly, then implement it centrally in shared channel ingress or setup/doctor policy code with per-channel/account tests and docs.

## Reproduction Assessment

Yes for current source behavior: unset DM policy resolves to `pairing`, and unallowlisted direct senders reach the pairing-required path. I did not run a live channel because the source path and docs are direct enough for triage.

## Solution Assessment

No as an automated bug-fix lane: the requested direction is plausible hardening, but it changes a documented cross-channel default and needs maintainer agreement on owner-to-channel mapping and group-policy semantics.
