## Summary

Keep open: current main still implements and documents `pairing` as the implicit DM default after first-owner bootstrap, so this is a valid security-default/product request rather than an implemented cleanup item.

## What This Changes

Request that DM-capable channels treat omitted `dmPolicy` as owner-derived `allowlist` after first-owner pairing bootstrap, while preserving explicit `dmPolicy` settings.

## Best Possible Solution

Decide the post-bootstrap security default explicitly, then implement the chosen policy centrally with per-channel/account tests and docs updates.

## Reproduction Assessment

Yes. Source inspection on current main shows omitted `dmPolicy` resolves to `pairing`, and an unallowlisted direct sender with `mayPair` reaches the pairing-required path.

## Solution Assessment

No for an automated fix. The requested hardening is plausible, but it changes documented cross-channel authorization defaults and needs maintainer agreement on owner mapping and group-policy semantics.
