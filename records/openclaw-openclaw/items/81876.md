## Summary

Keep open: current main still documents and implements `pairing` as the unset DM default, so the requested post-bootstrap auto-flip is a cross-channel security-default/product change rather than an already-defined bug fix.

## What This Changes

Request auto-flipping unset channel DM policy from `pairing` to an owner-derived allowlist after first-owner pairing bootstrap.

## Best Possible Solution

Decide the post-bootstrap security default explicitly, then implement the chosen policy centrally in shared channel ingress or setup/doctor code with per-channel/account tests and docs.

## Reproduction Assessment

Yes for current source behavior: unset DM policy resolves to `pairing`, and unallowlisted direct senders with `mayPair` reach the pairing-required path. I did not run a live channel because the shared source path and docs directly establish the behavior.

## Solution Assessment

No as an automated bug-fix lane: the requested direction is plausible hardening, but it changes a documented cross-channel default and needs maintainer agreement on owner-to-channel mapping and group-policy semantics.
