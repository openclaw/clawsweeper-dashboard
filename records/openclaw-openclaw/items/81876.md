## Summary

Keep open: current main and the latest shipped release still implement/document `pairing` as the unset DM default, so the requested post-bootstrap auto-flip is a real cross-channel security-default change that needs maintainer policy review rather than cleanup closure.

## What This Changes

Request auto-flipping unset channel DM defaults from `pairing` to an owner-derived allowlist after `commands.ownerAllowFrom` is populated by first-owner bootstrap.

## Best Possible Solution

Make an explicit maintainer decision on the post-bootstrap DM security default, then implement it centrally with owner-to-channel mapping rules, explicit-unset handling, docs, and per-channel regression coverage.

## Reproduction Assessment

Yes for current source behavior: unset `dmPolicy` resolves to `pairing`, and unallowlisted direct senders with pairing enabled reach `dm_policy_pairing_required`. I did not run a live channel because the shared resolver, sender gate, docs, and release-tag source are direct enough for triage.

## Solution Assessment

Unclear until maintainers decide the default-policy direction. The proposed owner-derived allowlist is plausible hardening, but it should not be automated before the owner mapping, account scoping, and group-policy semantics are agreed.
