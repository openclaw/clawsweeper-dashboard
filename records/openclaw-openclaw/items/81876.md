Keep open: current main and v2026.6.10 still resolve omitted DM policy to `pairing` after first-owner bootstrap, while the related issues are partial overlaps rather than a merged canonical fix. ## What This Changes

The issue requests changing implicit DM-capable channel defaults after first-owner bootstrap so omitted `dmPolicy` behaves like an owner allowlist instead of continuing pairing replies to strangers.

## Best Possible Solution

Make a maintainer product/security decision for the shared channel ingress default, then implement one explicit-policy-preserving owner-derived allowlist path with upgrade handling, docs, and cross-channel regression coverage.

## Reproduction Assessment

Yes, at source level: approve the first pairing so `commands.ownerAllowFrom` is populated, leave a DM-capable channel without explicit `dmPolicy`, and the shared resolver still uses `pairing` while unknown direct senders return the pairing-required path. ## Solution Assessment

Unclear: the requested auto-flip is a plausible security-default direction, but maintainers need to choose precedence, owner ID derivation, multi-account behavior, explicit `dmPolicy: "pairing"` compatibility, and whether group policy should change implicitly.
