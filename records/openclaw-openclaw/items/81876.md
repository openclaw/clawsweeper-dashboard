## Summary

Keep open: current main still documents and implements unset channel DM policy as pairing, so the requested post-bootstrap auto-flip is a cross-channel security-default decision rather than an already-implemented cleanup or narrow broken-contract fix.

## What This Changes

Request auto-deriving post-bootstrap channel DM defaults from `commands.ownerAllowFrom` when a channel has no explicit `dmPolicy`.

## Best Possible Solution

Make an explicit maintainer decision on the post-bootstrap security default, then implement it centrally in shared channel ingress or setup/doctor policy code with broad channel coverage and docs.

## Reproduction Assessment

Yes for current source behavior: unset DM policy resolves to `pairing`, and unallowlisted direct senders enter the pairing-required path. I did not run a live channel because the source and docs directly establish the behavior.

## Solution Assessment

No as an automated bug-fix lane: the requested direction is plausible hardening, but it changes a documented cross-channel default and needs maintainer agreement on owner-handle derivation and group-policy semantics.
