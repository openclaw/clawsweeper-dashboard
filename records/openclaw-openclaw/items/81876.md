## Summary

Keep open: current main and v2026.6.8 still leave omitted channel `dmPolicy` as `pairing` after first-owner bootstrap, and the related Telegram/bootstrap issues are open partial overlaps rather than superseding this cross-channel request. The work is valid security-default cleanup, but it needs maintainer product/security direction because it changes implicit channel authorization defaults and upgrade behavior.

## What This Changes

Request auto-flipping DM-capable channels with unset `dmPolicy` from `pairing` to owner-derived `allowlist`, plus disabled group policy, after `commands.ownerAllowFrom` is first populated.

## Best Possible Solution

Make a maintainer product/security decision for the shared channel ingress default, then implement one explicit-policy-preserving owner-derived allowlist path with upgrade handling, docs, and cross-channel regression coverage.

## Reproduction Assessment

Yes, at source level: approve the first pairing so `commands.ownerAllowFrom` is populated, leave a DM-capable channel without explicit `dmPolicy`, and the shared resolver still uses `pairing` while unknown direct senders return `dm_policy_pairing_required`. I did not run a live channel path in this read-only review.

## Solution Assessment

Unclear: the requested auto-flip is a plausible security-default direction, but the best fix needs maintainer agreement on cross-channel precedence, owner ID derivation, multi-account behavior, explicit `dmPolicy: "pairing"`, and whether group policy should change implicitly.
