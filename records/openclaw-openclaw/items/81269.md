## Summary

Keep this open: current main and the latest release still render the Runtime `agent=` value from the technical slot id, and an open linked PR with closing syntax already owns the active fix path.

## What This Changes

The issue requests that Runtime prompt metadata use the configured agent identity name instead of the technical slot id when `agents.list[].identity.name` is set.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81308 or an equivalent narrow fix that resolves `identity.name` through the agent config, displays it first, and keeps the stable slot id for diagnostics.

## Reproduction Assessment

Yes, as a source-level reproduction: a config with slot id `main` and `identity.name` `Runt` still reaches current main with only `agentId`, so the Runtime line can only render `agent=main`. I did not run a live Telegram drift session in this read-only review.

## Solution Assessment

Yes. Threading the existing configured identity name into runtime metadata and retaining the slot id as diagnostic metadata is the narrow maintainable fix; adding prompt prefix/suffix config would be a broader follow-up, not required for this bug.
