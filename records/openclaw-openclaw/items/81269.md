Keep this open: current main and the latest release still render the Runtime `agent=` value from the technical slot id, and the only directly linked fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to use configured `agents.list[].identity.name` in the Runtime system-prompt line instead of exposing only the stable agent slot id.

## Best Possible Solution

Resolve the configured identity name during system-prompt parameter assembly, render it as the visible Runtime agent label, preserve the stable slot id separately for diagnostics, and cover both formatter and parameter plumbing with focused tests.

## Reproduction Assessment

Yes for the source-level prompt mismatch: current main passes only the slot `agentId` into Runtime metadata and renders it as `agent=...`. I did not live-reproduce the intermittent model drift in Telegram.

## Solution Assessment

Yes. The narrow maintainable fix is to thread the already-configured `identity.name` into runtime metadata and keep `agentId` as separate diagnostic metadata; adding prompt prefix/suffix config is a broader optional follow-up.
