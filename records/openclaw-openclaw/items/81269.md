## Summary

Keep this open: current main and the latest release still render the Runtime `agent=` value from the technical slot id, and the linked fix PR had the right narrow direction but was closed unmerged.

## What This Changes

The issue requests that Runtime prompt metadata use `agents.list[].identity.name` for the visible agent identity instead of always using the technical agent slot id.

## Best Possible Solution

Thread the configured `identity.name` into runtime prompt params, render it as the visible Runtime identity when present, and keep the technical slot id separately for diagnostics when it differs.

## Reproduction Assessment

Yes by source inspection: configure an agent whose slot id differs from `identity.name`, and current main still builds the Runtime line from `runtimeInfo.agentId` because no identity-name field is available. I did not run a live Telegram/chat session, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the configured identity name into runtime metadata and retain the stable slot id as a separate diagnostic field; adding prompt prefix/suffix config would be broader follow-up work, not the needed default fix.
