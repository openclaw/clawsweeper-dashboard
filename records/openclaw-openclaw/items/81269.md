## Summary

Keep this open: current main still renders the Runtime agent label from the technical slot id and does not carry the configured identity name into runtime prompt metadata. The linked open fix PR already references this issue and covers the narrow implementation path, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests using `agents.list[].identity.name` in Runtime prompt metadata instead of presenting only the technical agent slot id as the agent label.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81308 so runtime prompt metadata carries the resolved identity name as the visible agent label while preserving the stable slot id separately for diagnostics.

## Reproduction Assessment

Yes, source-level: a config with `agentId` distinct from `identity.name` still produces Runtime metadata from `agentId` only because current main has no identity-name field in runtime prompt params. I did not establish the intermittent Telegram model-drift behavior live.

## Solution Assessment

Yes, the linked approach is the narrow maintainable fix: thread the already-configured identity name into runtime metadata and keep the slot id as a separate diagnostic field. Adding prompt prefix/suffix config would be a broader feature, not required for this bug.
