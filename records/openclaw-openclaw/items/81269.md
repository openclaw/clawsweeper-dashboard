## Summary

Keep this open: current main still renders the Runtime `agent=` field from the technical slot id and does not thread `agents.list[].identity.name` into runtime prompt metadata. The prior fix PR for the same bug is closed unmerged, and the current release still has the same source behavior.

## What This Changes

The issue requests that Runtime prompt metadata use the configured agent identity name instead of the slot id to avoid model identity drift.

## Best Possible Solution

Thread the existing configured identity name into shared runtime prompt params, render it first in the Runtime line, preserve the slot id separately for diagnostics, and add focused regression coverage.

## Reproduction Assessment

Yes for the source-level prompt mismatch: current main passes only `agentId` into runtime metadata and renders it as `agent=...`. I did not run a live Telegram session to reproduce the intermittent model-level drift.

## Solution Assessment

Yes. Using the existing `agents.list[].identity.name` in runtime metadata while retaining the technical slot id is narrower than adding a new prompt prefix/suffix or identity-line config surface.
