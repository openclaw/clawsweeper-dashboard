## What This Changes

Fix the 2026.4.29 request-latency regression causing slow embedded-run startup/prep, stuck sessions, MCP timeouts, failovers, and event-loop blocking in multi-agent/channel installs.

## Best Possible Solution

Land a targeted gateway/session hot-path fix that removes repeated full combined-store loads from transcript-update broadcasts while preserving correct transcript-to-session routing, then use current-main profiling to decide whether remaining prep/runtime costs belong in #75290 or a narrower follow-up.

## Solution Assessment

A bare unconditional cache early return is only safe if stale mappings are prevented through invalidation, sessionKey propagation, or an equivalent correctness guard.

## Repair Work Prompt

Fix the transcript update broadcast/session-key hot path from https://github.com/openclaw/openclaw/issues/76123. Current main resolves transcript-only updates in `src/gateway/session-transcript-key.ts` by calling `loadCombinedSessionStoreForGateway` before any cache benefit, and `src/gateway/server-session-events.ts` resolves a session key before dropping updates with no message. Implement a narrow gateway/session fix that avoids full combined-store reloads on repeated transcript updates while preserving correct routing for shared transcript paths, stale mappings, subagent sessions, and file-only refresh behavior. Do not add channel/provider-specific behavior, do not change bundled plugin policy, and do not claim to fix the separate `core-plugin-tools` prep-stage issue tracked in #75290 unless the patch directly proves it. Add regression tests that fail on current main, including repeat lookups not reloading the combined store when safe, message-less file-only updates not reverse-mapping unnecessarily, and existing stale/shared transcript-path behavior staying correct. Add a `CHANGELOG.md` fix entry because this is user-facing performance behavior. Validate with targeted gateway/session tests and the changed gate in Testbox per repo policy.
