## Summary

Keep open. Current main and v2026.6.8 still route ordinary foreground channel runs through a per-session lane and then the shared `main` global lane, with no supported browser, workload, or per-channel lane routing surface; this remains a valid scheduler/product design item rather than conservative cleanup.

## What This Changes

The issue requests dedicated browser/workload or per-channel command lane routing so slow browser automation does not starve fast inbound channel replies.

## Best Possible Solution

Define one scheduler-owned lane isolation model for slow foreground work that preserves per-session ordering, keeps a gateway-wide cap, uses trusted lane selection, and ships schema, docs, upgrade behavior, and focused fairness tests.

## Reproduction Assessment

Yes, at source level: current main still sends ordinary foreground channel runs through a per-session lane and then the shared `main` global lane, matching the reporter's `lane wait exceeded` logs. I did not run a live multi-channel browser workload in this read-only review.

## Solution Assessment

Unclear as proposed. A dedicated browser lane is plausible, but the best fix should be a maintainer-approved scheduler policy for workload, channel, or agent isolation with trusted lane ownership and a preserved global cap.
