## Summary

Keep open. Current main and the latest release still route ordinary inbound agent runs through the shared foreground `main` lane by default, while browser/workload or per-channel lane routing would add scheduler config semantics that need maintainer product review.

## What This Changes

Request a browser/workload or per-channel lane-routing config so long browser automation cannot monopolize the shared foreground lane used by quick channel replies.

## Best Possible Solution

Define a bounded, server-owned lane-pool routing contract for browser, workload, channel, or agent traffic, with allowlisted lane names, aggregate caps, fairness rules, config/docs, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary inbound reply runs to `CommandLane.Main` and exposes only global foreground/subagent caps, so long browser turns share the same foreground pool as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. A bounded server-owned lane-pool routing API is safer than browser-tool auto-detection or arbitrary lane strings alone because lane selection affects resource caps and fairness before tool use begins.
