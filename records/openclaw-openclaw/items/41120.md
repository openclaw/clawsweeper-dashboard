## Summary

Keep open. Current main still has arbitrary lane primitives but no validated browser/workload/per-channel routing config, and inbound reply execution continues to enter the process-wide main lane before browser/tool usage can affect scheduling.

## What This Changes

Request a dedicated browser lane or per-channel/workload lane routing config so long browser automation does not consume all main lane concurrency.

## Best Possible Solution

Define a validated lane-pool and routing contract, apply it at gateway startup/reload, and route inbound primary plus follow-up runs into the selected lane before the embedded runner enters the global queue.

## Reproduction Assessment

Yes for the current scheduling behavior: source inspection shows inbound reply execution uses CommandLane.Main, absent embedded-run lanes resolve to main, and gateway startup configures only built-in lane pools. I did not establish an end-to-end live browser crash reproduction.

## Solution Assessment

Unclear until maintainers choose the public config/API contract. Explicit per-channel or workload lane routing looks more maintainable than browser-tool auto-detection alone because the default global lane is chosen before tool usage is known.
