## Summary

Keep open: current main and the latest release still route ordinary non-cron embedded agent work through a per-session lane and then the shared `main` global lane, so the browser/workload or per-channel lane-isolation request remains unresolved and needs maintainer scheduler/config direction.

## What This Changes

The issue requests dedicated browser or per-channel lane routing so browser-heavy agent turns do not consume the shared main lane used by quick channel replies.

## Best Possible Solution

Design a core-owned lane/concurrency model that separates slow workload classes or channel/agent groups while preserving global resource caps, schema/docs consistency, and foreground embedded-run coverage.

## Reproduction Assessment

Yes from source for the core contention shape: current non-cron embedded runs still compose a per-session lane with a process-wide `main` global lane. I did not run a live browser-heavy multi-channel reproduction.

## Solution Assessment

Unclear: dedicated browser lanes and per-channel routing are plausible, but maintainers need to choose between browser/workload classification, per-channel lanes, per-agent lanes, and deadline/fairness policy before implementation.
