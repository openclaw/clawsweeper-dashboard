## Label Justifications

- `P2`: Long-running workloads can delay unrelated channel replies for minutes, but the item is a bounded scheduling improvement requiring product design rather than an emergency outage.
- `impact:message-loss`: Inbound messages can remain effectively suppressed in the shared queue long enough that users perceive them as lost or the agent as unavailable.

## Summary

This remains necessary: released and current code still lets legitimate long-running inbound turns occupy every process-wide main-lane slot, while priority ordering and task timeouts only improve queued ordering or hung-task recovery. The requested implementation needs a maintainer scheduling decision because per-channel lanes, browser classification, reserved foreground capacity, and the broader user-versus-background design have different ownership and compatibility implications.

## What This Changes

Add workload- or channel-aware scheduling so long browser sessions cannot consume all inbound agent capacity and delay unrelated fast messages.

## Best Possible Solution

Define one scheduler-owned foreground/background or workload-pool contract that reserves responsive inbound capacity while preserving global provider limits and per-session serialization, then use that same contract for browser-heavy turns and background follow-up work.

## Reproduction Assessment

Yes, from source with high confidence: if three legitimate long inbound runs occupy a main lane configured for concurrency three, a later fast message cannot start until one active run releases its slot. The reporter's exact multi-channel deployment was not executed, so this is source-reproducible rather than a live reproduction.

## Solution Assessment

No, not yet. Arbitrary per-channel lanes and post-browser lane migration are plausible but introduce parallel scheduling contracts; a scheduler-level reserved-capacity or foreground/background design is the safer canonical direction.
