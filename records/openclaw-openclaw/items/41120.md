## Summary

Keep open. Current main still has the reported product gap: inbound channel and follow-up runs resolve to the process-wide main global lane unless an explicit caller supplies a lane, and there is still no validated browser/workload/per-channel lane-pool configuration surface.

## What This Changes

The issue requests a dedicated browser workload lane or configurable per-channel/per-workload lane routing so long browser automation does not starve normal inbound replies.

## Best Possible Solution

Add a validated custom scheduler lane-pool config surface, apply it on gateway startup and reload, and route inbound primary plus follow-up runs into a selected lane before runEmbeddedPiAgent enters the global queue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: inbound auto-reply and follow-up runs do not derive a lane, absent params.lane resolves to main, and gateway startup only configures built-in lane pools.

## Solution Assessment

Unclear until maintainers choose the public config/API contract. Explicit per-channel or workload lane routing looks like the right first direction, while browser-tool auto-detection alone is weaker because the default global lane is chosen before tool usage is known.
