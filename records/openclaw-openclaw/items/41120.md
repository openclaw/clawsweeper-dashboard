## Summary

Keep open: current main and v2026.6.10 still route ordinary foreground channel agent turns through the shared `main` global lane, and the remaining browser/workload isolation request needs a scheduler/config product decision rather than cleanup closure.

## What This Changes

The issue requests a dedicated browser lane or broader per-channel/per-workload lane routing so long browser sessions do not consume all shared foreground agent slots.

## Best Possible Solution

Design a maintainer-approved scheduler model that derives workload lanes server-side from trusted config/run metadata, preserves safe defaults and a gateway-wide cap, and covers browser/workload, per-agent, and follow-up fairness without overlapping knobs.
