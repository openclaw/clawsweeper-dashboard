## Summary

Keep open: current main still routes ordinary foreground channel agent turns through the shared `main` global lane, so the browser/workload or per-channel lane-isolation request remains unresolved and needs maintainer scheduler/config direction.

## What This Changes

The issue requests a dedicated browser/workload lane or per-channel lane routing so long browser automation turns do not consume all shared main-lane capacity.

## Best Possible Solution

Design a core-owned scheduler/config model for workload or channel lane isolation that preserves global caps, avoids arbitrary client lane IDs, updates schema/docs, and covers all foreground embedded-run entry points.

## Reproduction Assessment

Yes, at source level: ordinary channel runs enter a per-session lane and then the shared `main` global lane, and current config/docs expose only `agents.defaults.maxConcurrent` for that cap. I did not run a live multi-channel browser workload.

## Solution Assessment

Unclear. A dedicated browser lane or per-channel lane is plausible, but the best fix should be unified with per-agent lane isolation and follow-up fairness so OpenClaw does not grow conflicting scheduler knobs.
