## Summary

Keep open: current main and v2026.6.10 still route ordinary foreground agent turns through the shared main lane unless a caller explicitly supplies a lane, and this issue asks for a new browser/workload or channel-routing scheduler surface that has not been designed or implemented.

## What This Changes

The issue requests a dedicated browser lane or per-channel lane routing so slow browser-heavy channel sessions do not starve fast text replies on the shared main lane.

## Best Possible Solution

Design a maintainer-approved workload or channel lane-routing model that preserves existing defaults, enforces a gateway-wide cap, and isolates browser-heavy turns without relying on arbitrary client-supplied lane ids.

## Reproduction Assessment

Yes at source level: ordinary foreground reply execution still selects CommandLane.Main and embedded runs fall back to main when no lane is supplied. A live browser-heavy multi-channel run was not performed in this read-only review.

## Solution Assessment

Unclear as proposed: a dedicated browser lane or per-channel routing is plausible, but maintainers still need to choose the scheduler/config contract, global cap behavior, and browser-workload classification boundary.
