## Summary

Keep open. Current main still sends ordinary channel/browser foreground agent work through the shared `main` lane with only global foreground concurrency, and the related per-agent lane PR is still open with unresolved resource-control concerns rather than a merged fix.

## What This Changes

Request dedicated browser/workload or per-channel command-lane routing so slow browser automation does not occupy the same shared foreground lane as quick channel replies.

## Best Possible Solution

Define a server-owned, bounded lane-pool routing contract for browser/workload/channel or agent traffic, with allowlisted lane names, global caps, and fairness guarantees.

## Reproduction Assessment

Yes, at source level. Current main hard-codes normal channel runs to `CommandLane.Main` and exposes only global foreground concurrency, so browser-heavy turns can occupy the same foreground slots as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler API. A server-owned routing contract with bounded lane names and global caps is safer than browser-tool auto-detection alone because foreground lane selection happens before tool use and arbitrary lane names have resource-control concerns.
