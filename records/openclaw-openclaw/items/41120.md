## Summary

Keep open. Current main and the latest release still route ordinary foreground channel work through a per-session lane and then the shared `main` lane, with no browser, per-channel, or workload-class lane config; the remaining fix needs maintainer scheduler/product design rather than stale cleanup.

## What This Changes

The issue requests dedicated browser or per-channel/workload lane routing so long browser automation does not consume all shared main-lane capacity for fast channel replies.

## Best Possible Solution

Define one scheduler-owned lane isolation model that preserves per-session serialization, keeps a real gateway-wide cap, uses server-controlled lane ids, and supports browser/workload, channel, or agent grouping with aligned schema, docs, fresh-install tests, and upgrade behavior.

## Reproduction Assessment

Yes, at source level: current main still sends ordinary foreground runs through a per-session lane and then the shared `main` global lane, and the reporter's logs match that path. I did not run a live 5-10 minute browser workload in this read-only review.

## Solution Assessment

Unclear. A dedicated browser lane is plausible, but the best maintainable fix should be a scheduler-owned workload/channel/agent isolation design that preserves global caps and avoids arbitrary network-facing lane overrides.
