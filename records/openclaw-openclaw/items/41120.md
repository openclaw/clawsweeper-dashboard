## Label Justifications

- `P2`: Browser-heavy multi-channel deployments can see multi-minute delays for unrelated text replies, but this is a bounded scheduler/config gap rather than an emergency outage.
- `impact:message-loss`: The reported impact is inbound channel messages waiting behind long browser runs until replies are delayed or effectively suppressed.

## Summary

Keep open: current main and v2026.6.6 still route normal inbound agent replies through the process-wide `main` lane, and the related per-agent lane tracker does not cover browser/workload/per-channel routing. The remaining work needs a maintainer-owned scheduler and configuration decision.

## What This Changes

The issue requests browser-workload or per-channel lane routing so slow browser automation does not occupy all global main-lane capacity for fast channel replies.

## Best Possible Solution

Design one scheduler-owned routing contract that preserves per-session ordering, allows approved workload/channel/agent isolation, and still enforces a global total cap with documented config semantics.

## Reproduction Assessment

Yes, at source level: current main routes normal inbound replies through `CommandLane.Main`, the docs define `main` as process-wide for inbound runs, and the reporter supplied matching `lane=main` wait logs. I did not run a live multi-channel browser load test in this read-only review.

## Solution Assessment

Unclear. A dedicated browser or per-channel lane is directionally plausible, but the best fix needs maintainer design around workload classification, explicit config, session ordering, and global caps rather than a narrow automatic lane move.
