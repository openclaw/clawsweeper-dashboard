## Summary

Keep open. Current main and the latest release still route normal inbound/follow-up agent turns through the shared `main` global lane unless a caller supplies an explicit lane, and the remaining request is a real product/config gap rather than an implemented fix or a clean duplicate.

## What This Changes

Request a dedicated browser/workload lane or per-channel lane-routing config so long browser automation does not consume all shared main-lane capacity.

## Best Possible Solution

Add a validated, server-owned lane-pool and routing config that can isolate browser/workload/channel traffic while preserving shared `main` defaults and global resource caps.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: inbound and follow-up runs default to the process-wide `main` lane when no explicit lane is supplied, and there is no config surface to route browser or per-channel work elsewhere. I did not run a live multi-channel browser stress test.

## Solution Assessment

Unclear until maintainers choose the public API. Explicit server-owned per-channel/workload lane routing looks more maintainable than browser-tool auto-detection alone, because the global lane is selected before tool use is known.
