## Label Justifications

- `P2`: The issue reports multi-minute delays for ordinary channel replies under browser-heavy workloads, but the remaining work is a bounded scheduler/config design rather than an emergency outage.
- `impact:message-loss`: Fast inbound channel messages can wait behind long browser work long enough to appear suppressed from the user's workflow.

## Summary

Keep open. Current main still routes ordinary foreground channel work through a per-session lane and then the shared `main` global lane, and the remaining browser/workload or per-channel isolation request needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests lane isolation for browser-heavy or channel-specific work so long browser automation does not starve ordinary inbound channel replies.

## Best Possible Solution

Define one scheduler-owned lane isolation model that preserves per-session serialization, keeps a real gateway-wide cap, uses validated server-controlled lane ids, and supports browser/workload, channel, or agent grouping with aligned schema, docs, tests, and upgrade behavior.

## Reproduction Assessment

Yes, at source level: ordinary runs still enter `session:<key>` and then the shared `main` global lane when no explicit lane is supplied. I did not run a live multi-channel browser workload in this read-only review.

## Solution Assessment

Unclear as proposed. A dedicated browser lane is plausible, but the best maintainable fix should be a scheduler-owned workload/channel/agent isolation design that preserves global caps and avoids arbitrary lane overrides.
