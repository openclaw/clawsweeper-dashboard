## Label Justifications

- `P2`: The issue reports multi-minute delays for ordinary channel replies under browser-heavy workloads, but the remaining work is a product/config scheduler design rather than an emergency outage.
- `impact:message-loss`: Fast inbound channel messages can wait behind long browser work long enough to appear suppressed or unprocessed.

## Summary

Keep open. Current main and the latest release still route ordinary foreground embedded-agent work through a per-session lane and then the shared `main` global lane, with no browser/workload or per-channel lane config; related issues and PRs overlap but do not provide a merged or exact canonical replacement.

## What This Changes

Issue requests dedicated browser/workload lanes or per-channel lane routing so slow browser automation does not starve fast channel replies.

## Best Possible Solution

Define one scheduler-owned lane isolation model that preserves per-session ordering, keeps a gateway-wide resource cap, uses validated server-selected lane ids, and aligns schema, docs, and regression tests across browser, channel, and agent workloads.

## Reproduction Assessment

Yes, at source level: current main resolves ordinary embedded-agent runs with no explicit lane to `main`, then queues through a per-session lane followed by the shared global lane. I did not run a live multi-channel browser workload in this read-only review.

## Solution Assessment

Unclear as proposed; a dedicated browser lane is plausible, but the maintainable fix should be a server-owned workload/channel/agent isolation design that preserves global caps and avoids arbitrary client-controlled lane expansion.
