## Summary

Keep open. Current main and the latest shipped release still route ordinary inbound agent turns through the shared foreground `main` lane by default, while the requested browser/workload or per-channel routing surface would add new scheduler config/API semantics that need maintainer product review.

## What This Changes

Request a dedicated browser/workload or per-channel command-lane routing surface so slow browser sessions stop consuming the shared `main` lane used by quick channel replies.

## Best Possible Solution

Define a bounded, server-owned lane-pool routing contract for browser, workload, channel, or agent traffic, with allowlisted lane names, aggregate caps, fairness rules, config/docs, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary inbound reply runs to `CommandLane.Main` and only exposes global foreground/subagent caps, so long browser turns share the same foreground pool as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. A bounded server-owned lane-pool routing API is safer than browser-tool auto-detection or arbitrary lane strings alone because lane selection affects resource caps and fairness before tool use begins.
