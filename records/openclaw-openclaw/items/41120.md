## Summary

Keep open. Current main still routes ordinary inbound agent turns through the shared foreground `main` lane, and the existing overlap only covers adjacent per-agent isolation rather than the requested browser, workload, or per-channel routing contract.

## What This Changes

The issue requests dedicated browser or per-channel lane routing so long browser automation does not starve fast inbound channel replies.

## Best Possible Solution

Define a bounded server-owned lane-pool routing contract for browser, workload, channel, or agent traffic, with allowlisted lane names, aggregate caps, fairness rules, config/docs, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary inbound reply runs to the shared `main` foreground lane and only exposes global foreground/subagent caps, so long browser turns can consume the same pool as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. A bounded lane-pool routing API is safer than browser-tool auto-detection or arbitrary lane strings alone because lane selection affects resource caps and fairness before tool use begins.
