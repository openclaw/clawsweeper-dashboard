## Summary

Keep open. Current main still routes ordinary inbound and follow-up agent runs to the shared `main` global lane unless a caller explicitly provides another lane, and there is still no browser, workload, or per-channel lane-routing config surface; https://github.com/openclaw/openclaw/pull/73991 covers only the per-agent subset and has unresolved design/security review, while https://github.com/openclaw/openclaw/issues/54488 tracks related session fairness rather than superseding this request.

## What This Changes

The issue requests a dedicated browser lane or configurable per-channel/workload lane routing so long browser automation runs do not starve quick inbound channel replies.

## Best Possible Solution

Define a validated, server-owned lane-pool and routing contract that can isolate browser, workload, channel, or agent traffic while preserving shared defaults, global resource caps, and coordination with https://github.com/openclaw/openclaw/issues/43235 and https://github.com/openclaw/openclaw/issues/54488.

## Reproduction Assessment

Yes, at source level. Normal inbound runs without an explicit lane still default to the process-wide `main` global lane, and current config/schema/docs have no browser, workload, or per-channel lane routing surface; I did not run the live multi-channel browser load scenario.

## Solution Assessment

Unclear until maintainers choose the public API. A server-owned per-channel or workload lane routing contract looks more maintainable than browser-tool auto-detection alone because the global lane is selected before tool use is known.
