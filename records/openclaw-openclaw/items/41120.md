## Summary

Keep open. Current main still routes ordinary channel/browser foreground runs through the shared `main` lane with no browser, workload, or per-channel routing config; the related per-agent lane PR remains open and blocked, and an independent user comment supports the workload-classified lane direction.

## What This Changes

Request a dedicated browser/workload lane or per-channel lane-routing config so slow browser automation does not consume all foreground main-lane capacity.

## Best Possible Solution

Define a validated server-owned lane-pool and routing contract that can isolate browser, workload, channel, or agent traffic while preserving bounded lane names, global caps, and session fairness semantics.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary channel/browser foreground runs to `CommandLane.Main` unless a caller supplies a lane, and current config/docs expose no browser, workload, or per-channel lane-routing surface.

## Solution Assessment

Unclear until maintainers choose the public scheduler API. A server-owned routing contract with bounded lane names and global caps is safer than browser-tool auto-detection alone because foreground lane selection currently happens before tool use is known.
