## Label Justifications

- `P2`: The issue describes real multi-channel latency and starvation with source-backed missing scheduler isolation, but it requires a bounded product/config design rather than an emergency fix.
- `impact:message-loss`: Fast channel messages can sit behind long browser turns long enough to appear dropped or ignored even though they were accepted into the queue.

## Summary

Keep open. Current main and v2026.6.8 still route ordinary foreground channel work through a per-session lane and then the shared `main` lane, with no browser, workload-class, or per-channel lane configuration; the remaining work is a real scheduler/product design issue, not stale cleanup.

## What This Changes

Request dedicated browser/workload lanes or per-channel lane routing so long browser automation does not consume all shared main-lane capacity needed for fast channel replies.

## Best Possible Solution

Adopt one maintainer-approved, server-owned workload/channel/agent lane policy that isolates slow browser automation while preserving per-session ordering, global resource caps, strict schema/docs, upgrade-safe defaults, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level: current main still sends ordinary foreground embedded runs through a session lane and then the process-wide `main` lane, and the latest release has the same central behavior. I did not run the live 5-10 minute browser workload reproduction in this read-only pass.

## Solution Assessment

Unclear as proposed. A browser or per-channel lane is plausible, but the best fix should be a maintainer-approved scheduler policy that also accounts for per-agent lane overlap, followup fairness, global caps, and trusted lane ownership.
