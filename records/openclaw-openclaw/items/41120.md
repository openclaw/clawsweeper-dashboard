## Summary

Keep open. Current main still lacks a public browser/workload/per-channel lane routing surface; inbound and follow-up agent runs still land on the shared main lane unless an explicit internal caller supplies a lane.

## What This Changes

Request a dedicated browser/workload lane or per-channel lane routing config so slow browser automation does not starve fast inbound replies.

## Best Possible Solution

Add a validated custom lane-pool config plus routing policy, apply it on startup/reload, and pass the resolved lane into inbound primary and follow-up runs before global queue enqueue.

## Reproduction Assessment

Yes for the product gap by source inspection: absent lane resolves to main, the follow-up runner uses main, and gateway lane setup only configures built-in pools. I did not run a live browser workload in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public config/API contract. Explicit per-channel or workload lane routing looks like the right first direction, while browser-tool auto-detection alone is weaker because the global queue lane is selected before tool usage is known.
