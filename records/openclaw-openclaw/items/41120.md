## Summary

Keep open. Current main still lacks browser/workload/per-channel foreground lane routing: ordinary auto-reply work is still funneled through the shared `main` lane unless a caller supplies a lane, and the related per-agent lane PR is open but does not fully cover this browser/workload routing request.

## What This Changes

The issue requests a dedicated browser lane or per-channel/workload lane routing so long browser automation does not starve fast channel replies.

## Best Possible Solution

Define one validated, server-owned lane-pool and routing contract that can isolate browser/workload/channel/agent traffic while preserving bounded lane names, global caps, session serialization, and fair handling of fast user messages.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary foreground work to `main` when no explicit lane is supplied and exposes no browser, workload, or per-channel lane-routing config.

## Solution Assessment

Unclear until maintainers choose the public API. A server-owned routing contract with bounded lane pools looks safer than browser-tool auto-detection alone because the foreground lane is selected before tool use is known.
