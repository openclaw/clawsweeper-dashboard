## Label Justifications

- `P2`: Browser-heavy multi-channel deployments can see multi-minute delays for unrelated text replies, but this is a bounded feature/design gap rather than an emergency outage.
- `impact:message-loss`: The reported impact is channel messages sitting behind long browser runs until users experience delayed or effectively suppressed replies.

## Summary

Keep open: current main and v2026.6.5 still route ordinary inbound agent work through the process-wide `main` lane, and the related per-agent tracker/PR does not cover browser/workload/per-channel routing. The remaining work needs a maintainer-owned scheduler and configuration decision.

## What This Changes

Request a dedicated browser/workload lane or per-channel lane routing so slow browser automation does not consume all foreground `main` lane slots.

## Best Possible Solution

Design one scheduler-owned routing contract that preserves per-session ordering, adds validated workload/channel/agent isolation where maintainers approve it, and still enforces a global total cap, with docs and upgrade-safe config semantics.

## Reproduction Assessment

Yes, at source level. Current main sets foreground runs to `CommandLane.Main`, documents that session runs enter the process-wide `main` lane by default, and the reporter supplied matching `lane=main` wait logs; I did not run a live multi-channel browser load test.

## Solution Assessment

Unclear. A dedicated browser/per-channel lane is directionally plausible, but the best solution needs maintainer design around workload classification, explicit config, session ordering, and a global cap rather than a narrow automatic lane move.
