## Summary

Keep open. Current main still routes ordinary channel/browser foreground runs through the shared `main` lane with no browser, workload, or per-channel lane-routing config, and the related per-agent lane PR is still open and blocked on resource-control review concerns.

## What This Changes

The issue asks for browser/workload or per-channel command lane routing so long browser automation does not starve quick channel replies on the shared main lane.

## Best Possible Solution

Define a server-owned lane-pool and routing contract for browser, workload, channel, or agent traffic, with bounded lane names, ingress allowlisting, global caps, and session fairness guarantees.

## Reproduction Assessment

Yes, at source level. Current main hard-codes ordinary channel runs to `CommandLane.Main` and exposes only global foreground concurrency, so browser-heavy turns can occupy the same foreground slots as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler API. A server-owned routing contract with bounded lane names and global caps is safer than browser-tool auto-detection alone because foreground lane selection happens before tool use and arbitrary lane names have resource-control concerns.
