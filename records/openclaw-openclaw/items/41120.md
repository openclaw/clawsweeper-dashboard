## Summary

Keep open. Current main still sends ordinary channel foreground runs through the shared `main` lane with no browser, workload, or per-channel lane-routing config, and the related per-agent lane PR remains open/conflicting with review and security blockers.

## What This Changes

The issue asks for browser-heavy channel work to be isolated with a dedicated browser lane or configurable per-channel/per-workload lane routing.

## Best Possible Solution

Define a validated server-owned lane-pool and routing contract that can isolate browser, workload, channel, or agent traffic while preserving bounded lane names, global caps, and session fairness semantics.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary channel/browser foreground runs to `CommandLane.Main` unless a caller supplies a lane, and current config/docs expose no browser, workload, or per-channel lane-routing surface.

## Solution Assessment

Unclear until maintainers choose the public scheduler API. A bounded, server-owned routing contract is safer than browser-tool auto-detection alone because foreground lane selection currently happens before browser use is known.
