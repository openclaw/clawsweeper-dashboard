## Summary

Keep open. Current main and the latest shipped release still route ordinary inbound agent work through the shared foreground `main` lane by default; browser/workload or per-channel routing remains a new scheduler/config contract that needs maintainer product review.

## What This Changes

The issue requests isolating browser-heavy work from ordinary inbound channel replies through a dedicated browser lane or configurable per-channel lane routing.

## Best Possible Solution

Define a bounded, server-owned lane-pool routing contract for browser, workload, channel, or agent traffic, with allowlisted lane names, aggregate caps, fairness rules, config/docs, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary inbound reply runs to `CommandLane.Main`, preserves only caller-supplied lane strings, and exposes global foreground/subagent caps rather than browser or per-channel routing.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. A bounded server-owned lane-pool routing API is safer than browser-tool auto-detection or arbitrary lane strings alone because lane selection affects resource caps and fairness before tool use begins.
