## Summary

Keep open. Current main still lacks browser/workload/per-channel foreground lane routing; ordinary channel runs without a configured lane still share `main`, and the open per-agent lane PR is only a partial, unresolved design/security path.

## What This Changes

Request a browser, workload, or per-channel lane-routing config so long browser automation does not consume every shared main lane slot.

## Best Possible Solution

Define a validated server-owned lane-pool and routing contract that can isolate browser, workload, channel, or agent traffic while preserving bounded lane names, global caps, and session fairness semantics.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary foreground runs without an explicit lane to the process-wide `main` lane, and current config/docs have no browser, workload, or per-channel lane-routing surface.

## Solution Assessment

Unclear until maintainers choose the public API. A server-owned routing contract with bounded lane names and global caps looks safer than browser-tool auto-detection alone because the foreground lane is selected before tool use is known.
