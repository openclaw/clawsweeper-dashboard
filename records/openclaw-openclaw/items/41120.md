## Summary

Keep open. Current main still sends ordinary inbound and follow-up agent turns through the shared `main` global lane unless a caller explicitly supplies another lane, and the repo still lacks a validated browser, workload, or per-channel lane routing config surface.

## What This Changes

Request a dedicated browser/workload lane or per-channel lane routing configuration so long browser sessions stop occupying all shared `main` lane slots.

## Best Possible Solution

Add a validated, server-owned lane-pool and routing contract that can isolate browser, workload, or channel traffic while preserving safe shared defaults and global resource caps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: normal runs default to the process-wide `main` global lane when no explicit lane is supplied, and there is no config surface to route browser or channel work elsewhere; I did not run a live multi-channel browser load test.

## Solution Assessment

Unclear until maintainers choose the public API. A server-owned per-channel or workload lane routing contract looks more maintainable than browser-tool auto-detection alone because the global lane is selected before tool use is known.
