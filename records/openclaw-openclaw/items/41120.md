## Summary

Keep open. Current main still lacks a browser/workload/per-channel lane routing surface for inbound channel runs; the queue can run arbitrary lanes, but normal auto-reply and follow-up paths still enter the default global lane unless a trusted caller explicitly passes one.

## What This Changes

The issue requests dedicated browser or per-channel/workload lane routing so long browser-heavy sessions do not starve normal inbound channel replies.

## Best Possible Solution

Add a validated custom lane-pool config, apply it on gateway startup and reload, and route inbound primary plus follow-up runs into an explicit channel/workload lane before embedded execution enters the global queue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: inbound auto-reply/follow-up runs do not derive a browser or per-channel lane, absent params.lane resolves to main, and gateway startup configures only built-in lane pools.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. Explicit per-channel or workload lane routing looks like the maintainable direction; browser-tool auto-detection alone is weaker because the global lane is selected before tool use is known.
