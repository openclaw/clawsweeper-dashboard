## Summary

Keep open. Current main still has the reported product gap: inbound channel and follow-up agent runs can only enter the configured process-wide main global lane unless an explicit caller supplies a lane, and there is no validated browser/workload/per-channel lane-pool config surface for those inbound paths.

## What This Changes

The issue requests dedicated browser or per-channel/workload lane routing so slow browser automation does not starve ordinary inbound channel replies on the shared main lane.

## Best Possible Solution

Add a validated custom scheduler lane-pool config surface, apply it on gateway startup and reload, and route inbound primary plus follow-up runs into a selected lane before runEmbeddedPiAgent enters the global queue. The first maintainable version should prefer explicit channel/account/binding/agent/workload routing precedence; browser-tool auto-detection and preemptive lane release can follow once the pre-run routing contract exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: inspect the inbound auto-reply and follow-up run callsites, observe that they omit params.lane, then trace runEmbeddedPiAgent through resolveGlobalLane to the main global lane. No live browser session is needed to prove the missing config/routing surface.

## Solution Assessment

Yes for explicit per-channel or workload lane routing as the first fix; that isolates slow browser work before it enters the global queue. Browser-tool auto-detection alone is not the best first step because the run has already acquired the default global lane by the time tool usage is known.
