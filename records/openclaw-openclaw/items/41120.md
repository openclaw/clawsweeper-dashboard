## Summary

Keep open. Current main and the latest release still route ordinary foreground agent runs to the shared `main` lane unless a caller supplies an explicit lane, and there is no browser, workload, or per-channel lane-routing config surface; the related per-agent PR only covers part of the problem and still has unresolved design/security review.

## What This Changes

Request a dedicated browser lane or configurable per-channel/workload lane routing so long browser automation does not occupy all shared main-lane capacity.

## Best Possible Solution

Define a validated, server-owned lane-pool and routing contract that can isolate browser, workload, channel, or agent traffic while preserving shared defaults, global resource caps, and coordination with the per-agent and session-fairness tracks.

## Reproduction Assessment

Yes, at source level. Ordinary foreground runs without an explicit lane still fall back to the process-wide `main` global lane, and current config/schema/docs have no browser, workload, or per-channel lane-routing surface; I did not run the live multi-channel browser load scenario.

## Solution Assessment

Unclear until maintainers choose the public API. A server-owned routing contract with bounded lane names and global caps looks safer than browser-tool auto-detection alone because the foreground lane is selected before tool use is known.
