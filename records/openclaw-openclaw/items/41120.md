## Summary

Keep open. Current main still sends ordinary inbound agent work through the shared foreground `main` lane, and the closest open per-agent lane work does not fully settle the browser/workload/per-channel routing contract requested here.

## What This Changes

The issue requests a dedicated browser lane or configurable per-channel/workload lane routing so long browser automation does not starve fast channel replies.

## Best Possible Solution

Define a bounded server-owned lane-pool routing contract for browser, workload, channel, or agent traffic, with allowlisted lane names, aggregate caps, fairness rules, config/docs, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level. Current main routes ordinary inbound reply runs to `CommandLane.Main` and only exposes global foreground/subagent caps, so long browser turns share the same foreground pool as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. A bounded lane-pool routing API is safer than browser-tool auto-detection or arbitrary lane strings alone because lane selection affects resource caps and fairness before tool use begins.
