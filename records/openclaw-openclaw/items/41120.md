## Summary

Keep open: current main and v2026.6.6 still route ordinary auto-reply/browser-triggered agent work through the shared `main` global lane, and the related per-agent lane issue/closed PR do not yet provide a safe canonical browser or per-channel routing solution.

## What This Changes

Request dedicated browser/workload or per-channel lane routing so slow browser automation does not consume all shared main-lane slots for fast channel replies.

## Best Possible Solution

Adopt one maintainer-approved scheduler/config design that isolates slow browser or workload-classified channel work while preserving per-session ordering, a real process-wide resource cap, trusted lane selection, docs, and upgrade-safe tests.

## Reproduction Assessment

Yes, at source level: current main still routes auto-reply agent work through the shared `main` global lane, and the latest release shows the same lane composition. I did not run a live 5-10 minute browser workload across multiple messaging channels in this read-only pass.

## Solution Assessment

Unclear. A dedicated browser lane or per-channel routing is plausible, but the best fix needs a maintainer decision on workload classification, trusted config shape, global cap composition, and interaction with per-agent/session fairness work.
