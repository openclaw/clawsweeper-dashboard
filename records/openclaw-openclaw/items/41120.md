## Summary

Keep open. Current main and the latest release still route normal inbound agent work through the shared foreground `main` lane, while the closest linked implementation path remains open and still needs maintainer resource-control/product review.

## What This Changes

The issue requests dedicated browser or per-channel/workload lane routing so long browser-heavy agent turns do not starve ordinary channel replies on the shared main lane.

## Best Possible Solution

Define a server-owned, bounded lane-pool routing contract for workload, browser, channel, or agent traffic, with allowlisted lane names, global caps, fairness guarantees, config/docs, and focused scheduler tests.

## Reproduction Assessment

Yes, at source level. Current main hard-codes ordinary auto-reply execution to `CommandLane.Main` and exposes only global foreground concurrency, so browser-heavy turns can occupy the same foreground slots as quick text replies.

## Solution Assessment

Unclear until maintainers choose the public scheduler contract. A bounded, server-owned lane routing API is safer than browser-tool auto-detection or arbitrary lane strings alone because lane selection affects resource caps and fairness before tool use begins.
