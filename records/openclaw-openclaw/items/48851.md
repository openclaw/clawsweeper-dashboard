## Summary

Keep open: current `main` still lacks the requested call-count display, and this PR contains useful work, but the head patch still has correctness blockers plus missing real behavior proof.

## What This Changes

The PR adds per-run API call counting through the embedded runner, persists it on session entries, and displays it in `/status` plus the response usage footer.

## Best Possible Solution

Land a rebased implementation that counts actual model requests independently from token metadata, replaces or clears the latest persisted count deterministically, and includes redacted real-session proof for `/status` plus the response footer.

## Reproduction Assessment

Yes for review purposes: current `main` lacks the feature, and the PR’s remaining defects are source-reproducible from the head diff. I did not run a live provider session in this read-only review.

## Solution Assessment

No, not as written. The feature direction is reasonable, but the patch should count model requests outside usage metadata filtering and should not preserve stale latest-run counts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count calls outside the usage guard:** `src/agents/pi-embedded-subscribe.ts:275-278`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so successful provider responses that omit usage or report zero usage are excluded even though a model request occurred. Count the assistant/model request independently, and keep token aggregation behind the usage guard.
  - confidence: 0.86
- **[P2] Overwrite the latest call count deterministically:** `src/auto-reply/reply/session-usage.ts:104-106`
  - body: `callCount` represents latest-run data, but this only writes it when the value is greater than zero. A later zero-count or unknown-count update can leave an older value visible in `/status`, so numeric values should replace the prior value or the field should be cleared when unknown.
  - confidence: 0.9
