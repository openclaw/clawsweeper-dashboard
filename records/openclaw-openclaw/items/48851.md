## Summary

Keep open: the feature is still useful and current main does not implement it, but the PR head still has correctness blockers around usage-gated call counting and stale latest-run state, and it lacks contributor-provided real behavior proof.

## What This Changes

The branch adds per-run API call count tracking through embedded runner/session metadata and displays it in `/status` plus the response usage footer with unit tests.

## Best Possible Solution

Land a rebased implementation that tracks per-turn model/API request counts at the model-response boundary, stores latest-run state deterministically, renders it in the footer and status output, and proves the behavior in a real channel.

## Reproduction Assessment

Yes for review purposes: current main can be inspected to show no call-count state/display, and the PR diff shows the new counter and persistence defects without requiring a live provider run. Contributor real behavior proof is still missing.

## Solution Assessment

No, not as written. The direction is reasonable, but the implementation should count model responses independently from token usage metadata, deterministically replace or clear latest-run state, rebase against current status/usage files, and include redacted real-session proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Count calls outside usage metadata guards:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: The new counter increments only after `hasNonzeroUsage(usage)` passes, so a successful LLM response that omits token usage or reports zero usage is not counted even though the PR advertises API-call counts. Move the count to the model-response boundary and keep token aggregation conditional.
  - confidence: 0.88
- **[P2] Replace stale latest-run call counts:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: This only writes `callCount` when it is greater than zero. Because session entries are merged and this field represents latest-run state, a later zero or unknown-count update can leave an older value visible in `/status`; persist numeric zero when known or explicitly clear the field when unknown.
  - confidence: 0.9
