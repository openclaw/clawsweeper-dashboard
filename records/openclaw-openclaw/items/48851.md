## Summary

Keep open: current main still lacks the requested API-call-count display, but this PR is not merge-ready because the branch still undercounts calls without usage metadata, can leave stale latest-run counts, and lacks contributor real behavior proof.

## What This Changes

The PR adds per-run API call count plumbing through the embedded runner and session persistence, then displays Calls in `/status` and the response usage footer with unit tests.

## Best Possible Solution

Rebase and land a narrow implementation that counts model/API responses independently from token metadata, persists or clears latest-run call state deterministically, renders in the current status/footer helpers, and includes redacted real-session proof.

## Reproduction Assessment

Yes for review purposes: current main can be inspected to show no call-count state/display, and the PR diff shows the remaining counter and persistence defects. Contributor after-fix real behavior proof is still missing.

## Solution Assessment

No, not as written. The direction is reasonable, but the patch should count calls independently of usage-token metadata, replace or clear latest-run call state, and include redacted real-session proof.

  - body: The new counter increments only after `hasNonzeroUsage(usage)` passes, so successful model responses that omit token usage or report zeros are not counted even though the PR advertises API-call counts.   - confidence: 0.88

  - confidence: 0.9
