## Summary

Keep this PR open. Current main still lacks the requested call-count persistence and display, so the work is not obsolete, but the branch is currently unmergeable/stale and still has correctness gaps around stale values and usage-missing model calls.

## What This Changes

The branch adds per-run API/model call count tracking through embedded runner metadata, session persistence, `/status`, the response usage footer, and related unit tests.

## Best Possible Solution

Land a rebased implementation on the active main paths that records a numeric per-run model/API call count independent of token usage, clears stale latest-run values, renders the count in `/status` and the response usage footer when known, and updates docs/changelog coverage.

## Reproduction Assessment

Yes. Static current-main inspection is enough to reproduce the missing feature: the active status builder, footer formatter, session type, persistence helper, and embedded metadata contain no call-count path. The remaining PR bugs are also reproducible from the diff and review comments: zero/missing usage can leave stale or undercounted values.

## Solution Assessment

No. The product direction is reasonable, but this branch is not the best current solution until it is ported to the active files and counts actual model/API requests rather than only usage-bearing responses; persisting numeric zero or clearing the field is the safer latest-run contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear stale call counts on every persisted run:** `src/auto-reply/reply/session-usage.ts:104-105`
  - body: `callCount` is meant to describe the latest/current run, but this only writes it when the new value is `> 0`. If a later run has zero counted calls or a path omits the field, the merged session entry keeps the previous nonzero value and `/status` can report that stale count as current. Persist numeric `0` or explicitly clear the field when the current run has no count.
  - confidence: 0.9
- **[P2] Count model requests without requiring usage tokens:** `src/agents/pi-embedded-subscribe.ts:278`
  - body: The new counter increments only after nonzero usage is accepted, so providers or error paths that return a model response without usage metadata are excluded. That breaks the advertised API-call accounting for providers where token usage is unavailable; count at the model request/assistant-response boundary and keep token aggregation conditional.
  - confidence: 0.86
- **[P3] Add release notes for the visible usage change:** `src/auto-reply/status.ts:665-666`
  - body: This is a user-facing feature that changes `/status` and the response usage footer, but the branch does not update `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing `feat` changes, credited to the contributor.
