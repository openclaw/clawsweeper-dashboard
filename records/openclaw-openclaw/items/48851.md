## Summary

Keep open. Current main still lacks the API-call count surface, and this PR remains the canonical implementation path, but the branch is not merge-ready because it can undercount calls without usage metadata, preserve stale persisted counts, and lacks contributor-provided real behavior proof.

## What This Changes

The branch adds per-turn API-call count metadata through the embedded runner/session store and renders Calls in `/status` plus the response usage footer with unit coverage.

## Best Possible Solution

Land a rebased implementation that tracks actual per-turn model requests, stores the latest count without stale carry-over, and renders it through the shared status and response-usage helpers with focused tests and real proof.

## Reproduction Assessment

Yes for review purposes. Current main lacks the requested display/persistence, and the two blocking PR defects are visible from source inspection at the branch head without needing a live provider run.

## Solution Assessment

No, not as written. The feature direction is useful, but the patch should count model requests independently from token metadata and replace or clear the latest persisted call count deterministically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count calls before filtering usage metadata:** `src/agents/pi-embedded-subscribe.ts:275-278`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so a provider/model response that omits usage or reports zero usage is excluded even though a model request occurred. Count the assistant/model request independently and keep token aggregation behind the usage guard.
  - confidence: 0.86
- **[P2] Replace stale call counts with the latest value:** `src/auto-reply/reply/session-usage.ts:104-106`
  - body: `callCount` represents the latest run, but this only writes it when the value is greater than zero. A later zero-count or no-count update can leave an older session value in place, making `/status` report stale current-turn data.
  - confidence: 0.9
