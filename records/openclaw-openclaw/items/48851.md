## Summary

Keep open. Current main still lacks the requested call-count feature and the older related issues were closed in favor of this PR, but the PR still needs real behavior proof and has correctness defects in usage-less response counting and stale session persistence.

## What This Changes

The PR adds per-run API-call count metadata from the embedded runner through session persistence and displays it in `/status` and response usage footers with unit tests.

## Best Possible Solution

Land a rebased implementation that defines model-request count semantics, records the per-run count at the request or assistant-response boundary, persists or clears the latest count deterministically, and displays it in `/status` and the footer with focused tests and real proof.

## Reproduction Assessment

Yes for review purposes. Source inspection shows current main lacks the feature, and the PR diff gives a deterministic path to the usage-less undercount and stale-persistence defects without needing a live provider run.

## Solution Assessment

No. The product direction is reasonable, but this patch should count model requests independently of usage metadata, overwrite or clear the persisted latest count, and adapt to current main's formatter/status layout before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count model requests without requiring usage tokens:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so providers or error paths that return an assistant response without nonzero usage are excluded. That breaks the advertised API-call accounting for providers where token usage is unavailable; count at the model request or assistant-response boundary and keep token aggregation conditional.
  - confidence: 0.86
- **[P2] Clear stale call counts on every persisted run:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: `callCount` is meant to describe the latest run, but this only writes it when the new value is greater than zero. If a later run has zero counted calls or a path omits the field, the merged session entry keeps the previous value and `/status` can report stale data.
  - confidence: 0.9
