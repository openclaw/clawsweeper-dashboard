## Summary

Keep this PR open: current main still lacks API/model call-count display, and this PR is the active implementation for the closed canonical issue, but it still has correctness blockers and missing real behavior proof before merge.

## What This Changes

The PR adds per-turn API call-count tracking to embedded agent runs, persists the latest count on session entries, and displays it in `/status` and response usage footers with tests.

## Best Possible Solution

Land a rebased implementation on current main that records model request count independently of usage metadata, updates or clears the latest persisted count deterministically, and displays it in `/status` and the response footer with focused coverage.

## Reproduction Assessment

Yes for review purposes. Current main lacks the requested display/persistence, and the PR diff exposes the usage-metadata undercount and stale-persistence paths without needing a live provider run.

## Solution Assessment

No. The direction is reasonable, but this patch should count requests at the model/assistant-response boundary rather than after usage metadata exists, and it should overwrite or clear the latest stored count on every persisted run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count model requests even without usage tokens:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so providers or error paths that return an assistant response without nonzero usage are excluded. That breaks the advertised API-call accounting for providers where token usage is unavailable; count at the model request or assistant-response boundary and keep token aggregation conditional.
  - confidence: 0.86
- **[P2] Overwrite stale call counts on every run:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: `callCount` is meant to describe the latest run, but this only writes it when the new value is greater than zero. If a later run has zero counted calls or a path omits the field, the merged session entry keeps the previous value and `/status` can report stale data.
  - confidence: 0.9
