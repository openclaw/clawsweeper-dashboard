## What This Changes

The PR adds per-turn API call count tracking in embedded runs, stores it on session entries, and renders it in `/status` plus the response usage footer with unit tests.

## Best Possible Solution

Rebase onto the current status/footer/session-accounting layout, count model-response calls independently from token usage metadata, replace or clear latest-run call-count state deterministically, and add redacted real-session proof for `/status` plus the full response footer.

## Reproduction Assessment

Yes for the review findings: source inspection shows the counter is gated by nonzero usage and persistence writes only values greater than zero. I did not run live proof, and the contributor has not supplied real behavior evidence.

## Solution Assessment

No. The feature direction is useful, but the current patch should be rebased and should count model responses independently from token usage while replacing latest-run session state deterministically.
