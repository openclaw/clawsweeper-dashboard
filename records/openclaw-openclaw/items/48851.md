## What This Changes

The PR adds per-turn API call-count tracking through the embedded runner, session entries, `/status`, response usage footer output, and related tests.

## Best Possible Solution

Land a rebased implementation on current main that records per-run model request count at the request or assistant-response boundary, persists the latest count deterministically, and displays it in `/status` and the response footer with focused coverage.

## Reproduction Assessment

Yes for review purposes. Source inspection shows current main lacks the feature, and the PR diff exposes the usage-less undercount and stale-persistence paths without needing a live provider run.

## Solution Assessment

No. The product direction is reasonable, but this patch should count model requests independently of usage metadata, overwrite or clear the latest stored count, and adapt to current main's formatter/status layout before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count model requests without requiring usage tokens:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so providers or error paths that return an assistant response without nonzero usage are excluded. That breaks the advertised API-call accounting for providers where token usage is unavailable; count at the model request or assistant-response boundary and keep token aggregation conditional.
  - confidence: 0.86
- **[P2] Clear stale call counts on every persisted run:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: `callCount` is meant to describe the latest run, but this only writes it when the new value is greater than zero. If a later run has zero counted calls or a path omits the field, the merged session entry keeps the previous value and `/status` can report stale data.
  - confidence: 0.9
