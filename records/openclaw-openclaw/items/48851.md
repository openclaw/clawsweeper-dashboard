The PR adds per-run API call count plumbing through embedded runner metadata and session entries, then renders the count in `/status` and the response usage footer with unit tests.

## Best Possible Solution

Rebase onto the current status/footer helper layout, count model responses independently from usage-token metadata, make `callCount` deterministic latest-run state, and add redacted real-session proof.

## Reproduction Assessment

Yes for review purposes: current main source lacks the call-count output, and PR-head source inspection shows the two remaining incorrect code paths. I did not run tests because this is a read-only review and contributor after-fix real behavior proof is still absent.

## Solution Assessment

No, not as written. The direction is reasonable, but the implementation should count calls outside usage-token guards, replace or clear latest-run call state deterministically, and then prove the real `/status` and footer behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count calls before the usage guard:** `src/agents/pi-embedded-subscribe.ts:278`
  - body: `callCount` increments only after `hasNonzeroUsage` passes, so a successful provider response that omits usage or reports zero usage still made an API call but will not be counted. Move the counter to the model-response boundary and keep token aggregation conditional.
  - confidence: 0.88
- **[P2] Replace stale latest-run call counts:** `src/auto-reply/reply/session-usage.ts:104-105`
  - body: This only writes `callCount` when it is greater than zero. Because session entries are patched and the field represents latest-run state, a later zero or unknown-count update can leave an older value visible in `/status`; persist numeric zero when known or clear the field when unknown.
  - confidence: 0.9
