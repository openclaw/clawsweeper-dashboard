## Summary

Keep this PR open because current main still lacks API-call count display and the linked feature issue was already closed in favor of this implementation PR, but the branch is conflicting, lacks real behavior proof, and still has call-count semantics bugs to resolve before merge.

## What This Changes

The PR adds per-turn API call counting through the embedded runner/session store and displays the count in `/status` plus the response usage footer with focused unit tests.

## Best Possible Solution

Rebase onto current main, preserve the newer status and usage-line helpers, count calls at the assistant response/API-call boundary or clearly rename the metric to usage-bearing calls, persist or clear the current-turn count consistently, and add real runtime proof plus focused tests.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR increments and persists the new count only on nonzero usage paths, so usage-less provider responses or count-unavailable turns can produce missing or stale `Calls` output. This is not a current-main bug reproduction because the feature is not on current main yet.

## Solution Assessment

No, not as-is. The feature direction is reasonable, but the implementation should be rebased and should make the displayed metric match the advertised API-call semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Count calls even when usage is missing:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: This counter only increments after `recordAssistantUsage` sees a nonzero usage object. Providers can return successful assistant responses with missing or zero usage, and those are still API calls; in that case the new `/status` and footer metric reports no call even though the PR advertises actual per-call accounting. Track the call at the assistant/API response boundary, or change the label and docs to make this explicitly usage-bearing calls only.
  - confidence: 0.84
- **[P2] Clear stale counts when a turn has no count:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: This only writes `callCount` when the new value is greater than zero. Session entries are merged, so a later turn that persists usage without a numeric count, such as a non-embedded or count-unavailable run, will keep showing the prior turn's `Calls` value in `/status` even though the field is described as current-turn data. Persist `0` or clear the field on usage updates where the count is unavailable.
  - confidence: 0.78
