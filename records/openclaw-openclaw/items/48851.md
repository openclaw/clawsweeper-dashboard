## Summary

Keep this PR open: current main still lacks the requested API-call count in both `/status` and the response usage footer, and the closed related feature requests point back to this PR as the active implementation. The branch is not merge-ready because it still undercounts usage-less model calls, can preserve stale persisted counts, and has no contributor-provided real behavior proof.

## What This Changes

The PR adds per-turn API call-count tracking through embedded runner metadata/session state and displays `Calls` in `/status` plus response usage footers with tests.

## Best Possible Solution

Land a rebased implementation that counts model requests independently of token usage metadata, overwrites or clears the latest persisted count deterministically, and renders it through the current status and usage-footer helpers with focused coverage and real proof.

## Reproduction Assessment

Yes for review purposes. Current main lacks the requested display/persistence, and the PR diff exposes the two correctness issues from source inspection without needing a live provider run.

## Solution Assessment

No. The direction is reasonable, but the patch should count model requests at the request or assistant-response boundary and persist the latest count deterministically instead of tying both behaviors to nonzero token usage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count model requests even without usage tokens:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so providers or error paths that return an assistant response without token usage are excluded. Count the model request or assistant response independently, while keeping token aggregation conditional.
  - confidence: 0.86
- **[P2] Overwrite stale call counts on every run:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: `callCount` describes the latest run, but this only writes it when the new value is greater than zero. A later zero or omitted count can leave an older session value in place, making `/status` report stale data.
  - confidence: 0.9
