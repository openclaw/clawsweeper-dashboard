## Best Possible Solution

Land a rebased implementation that counts model/API responses at the response boundary, stores latest-run call state deterministically, renders it in status and usage footers, and proves it in a real session.

## Reproduction Assessment

## Solution Assessment

No, not as written. The direction is reasonable, but the patch should count calls independently of usage-token metadata, replace or clear latest-run call state, and include redacted real-session proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Count calls outside usage metadata guards:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: The new counter increments only after `hasNonzeroUsage(usage)` passes, so a successful model response that omits token usage or reports zeros is not counted even though the PR advertises API-call counts. Move the count to the model-response boundary and keep token aggregation conditional.
  - confidence: 0.88
- **[P2] Replace stale latest-run call counts:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: This only writes `callCount` when it is greater than zero. Because session entries are patched and this field represents latest-run state, a later zero or unknown-count update can leave an older value visible in `/status`; persist numeric zero when known or explicitly clear the field when unknown.
  - confidence: 0.9
