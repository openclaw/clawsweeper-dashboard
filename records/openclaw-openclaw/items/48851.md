## Summary

Keep open: current main still lacks API-call count display/persistence, but this PR has source-reproducible correctness defects and only unit-test proof, so it should not merge as-is.

## What This Changes

The PR adds per-run `callCount` tracking through embedded runner metadata/session entries and displays it in `/status` plus the response usage footer, with unit tests.

## Best Possible Solution

Land a rebased implementation that counts model requests independently from token metadata, deterministically replaces or clears latest-run call count, and includes redacted real-session proof for `/status` plus the footer.

## Reproduction Assessment

Yes for review purposes: current main source lacks the requested call-count field/display, and the PR head defects are visible from the diff. I did not run a live provider/channel session in this read-only review.

## Solution Assessment

No, not as written. The feature direction is reasonable, but the patch should count actual model requests outside usage metadata filtering and should avoid stale latest-run values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count calls outside usage metadata:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: `callCount` is incremented only after `hasNonzeroUsage(usage)` passes, so providers or responses that omit usage fields can make a real model request while the new counter stays at zero. Count the model request independently, and keep token aggregation behind the usage guard.
  - confidence: 0.86
- **[P2] Replace stale call counts deterministically:** `src/auto-reply/reply/session-usage.ts:101-105`
  - body: `callCount` is latest-run state, but this only writes values greater than zero. A later zero-count or unknown-count update can leave an older value visible in `/status`; persist numeric zero counts or clear the field when the count is unknown.
  - confidence: 0.9
