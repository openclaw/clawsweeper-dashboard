## Summary

Keep open: current main still lacks API-call count display/persistence, but this external PR is conflicting with main, still has source-level correctness gaps, and still lacks after-fix real behavior proof.

## What This Changes

The branch adds per-turn API-call count metadata from the embedded runner, persists it on sessions, and renders Calls in /status and response usage footers with tests.

## Best Possible Solution

Land a rebased implementation that counts model requests independently from token usage metadata, deterministically replaces the latest persisted count, and renders it through the current status and response-usage helpers with focused coverage and real proof.

## Reproduction Assessment

Yes for review purposes. Current main lacks the requested display/persistence, and the PR diff exposes the undercount and stale-persistence paths from source inspection without needing a live provider run.

## Solution Assessment

No, not as written. The direction is useful, but the patch should count actual model requests and update the persisted latest-run count deterministically instead of tying both behaviors to nonzero usage metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count model requests even without usage tokens:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: callCount is incremented only after hasNonzeroUsage(usage) passes, so providers or error paths that return an assistant response without token usage are excluded even though the feature is advertised as API-call accounting. Count the model request or assistant response independently while keeping token aggregation conditional.
  - confidence: 0.86
- **[P2] Overwrite stale call counts on every run:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: callCount describes the latest run, but this only writes it when the new value is greater than zero. A later zero or omitted count can leave an older session value in place, making /status report stale data.
  - confidence: 0.9
