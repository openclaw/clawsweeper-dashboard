## Summary

Keep open. This PR is a useful implementation candidate for #55801, but the current patch would misattribute deny removals under allow-labeled pipeline steps and can double-log sandbox denials because the audit side effect lives in formatting.

## What This Changes

The branch adds INFO-level audit logs to tool-policy pipeline filtering and sandbox blocked-tool formatting, plus pipeline logging tests.

## Best Possible Solution

Keep the audit feature, but route policy evaluation through provenance-aware helpers that report the matched allow/deny source and rule, emit sandbox decisions once, and ship with regression tests plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `tools.allow`-labeled pipeline step can include `deny`, and shared assistant-error formatting can be reached from several runtime renderers.

## Solution Assessment

No. Logging removals is the right direction, but this patch should attribute the matched allow/deny rule and move sandbox audit logging to a single policy/enforcement boundary or add explicit dedupe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Report the matched policy rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The audit line uses `step.label`, but default global/agent/group steps are labeled with allow-oriented keys even when `policy.deny` removes a tool. A config with both `allow` and `deny` under a `tools.allow` step would still log `tools.allow` for a deny block, so operators cannot tell which rule matched.
  - confidence: 0.91
- **[P2] Log sandbox denials outside the formatter:** `src/agents/sandbox/runtime-status.ts:160-162`
  - body: `formatSandboxToolPolicyBlockedMessage` is reached through `formatAssistantErrorText`, which is used by runner payload, failover, and lifecycle formatting paths. Putting `logInfo` here can emit multiple audit entries for one blocked call; emit once at the enforcement boundary or dedupe per run/tool.
  - confidence: 0.87

  - body: This adds an operator-visible INFO audit feature, but `CHANGELOG.md` is not updated. Add a single Unreleased Changes entry with appropriate human contributor credit before merge.
