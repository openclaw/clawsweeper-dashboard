## Summary

Keep open: the PR addresses a valid operator-audit gap, but it should not merge as-is because the pipeline audit can misattribute deny-rule removals to allow-labeled config steps, the sandbox audit side effect is placed in shared error formatting, and the user-facing feature lacks the required changelog entry.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline removals and sandbox blocked-tool messages, plus pipeline audit-log unit tests.

## Best Possible Solution

Implement rule-aware audit at the policy decision boundary, emit sandbox audit once per blocked call, and document the operator-visible logging in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a `tools.allow`-labeled pipeline step can include `deny`, and the PR logs only that broad label; the sandbox formatter path is also reachable from multiple assistant-error rendering call sites.

## Solution Assessment

No. The direction is useful, but this implementation does not yet provide reliable rule attribution and puts sandbox audit side effects in formatting instead of a single audit boundary.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Report the matched allow or deny rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The new audit line uses `step.label`, but default global, agent, and group steps are labeled `*.allow` even when `policy.deny` removes a tool. A normal deny rule can therefore be logged as `tools.allow`, sending operators to the wrong config key; derive the matched allow/deny key and rule from policy evaluation instead.
  - confidence: 0.91

Logging here can emit repeated identical INFO audit entries for one blocked tool call, so emit once at the enforcement boundary or add explicit per-run/tool dedupe.
  - confidence: 0.87
