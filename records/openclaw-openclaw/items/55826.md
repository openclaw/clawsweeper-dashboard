## Summary

Keep open. The feature is useful and tied to https://github.com/openclaw/openclaw/issues/55801, but the current diff does not actually report the matched allow/deny rule in the pipeline, can duplicate sandbox audit logs from shared error formatting, and lacks required real behavior proof.

## What This Changes

The branch adds INFO-level audit logs for tool-policy filtering and sandbox-blocked tools, plus pipeline tests for emitted and suppressed audit log cases.

## Best Possible Solution

Carry matched allow/deny rule metadata through policy evaluation and emit one sanitized audit record at enforcement boundaries, with focused tests and redacted real audit-output proof.

## Reproduction Assessment

Yes, source-reproducible. Current main uses deny-capable policy objects under allow-labeled pipeline steps, and the PR diff logs only the broad step label while the sandbox formatter is reachable from multiple error-rendering paths.

## Solution Assessment

No. The direction is right, but the PR should report the matched rule/type/path and move sandbox audit logging to a single enforcement point or add explicit per-run/tool dedupe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Report the matched allow or deny rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The audit line uses `step.label`, but default global/agent/group policy steps are labeled `*.allow` even when `policy.deny` removes a tool. In an allow+deny policy under `tools.allow`, this will log the allow key for a deny-list block and point operators at the wrong rule; carry the matched rule type/path from policy evaluation instead.
  - confidence: 0.91
- **[P2] Emit sandbox audit logging once at enforcement:** `src/agents/sandbox/runtime-status.ts:160-162`
  - body: The new `logInfo` call is inside `formatSandboxToolPolicyBlockedMessage`, which is reached through shared assistant-error rendering paths. One blocked tool call can therefore produce repeated identical INFO audit entries, so emit at a single enforcement boundary or dedupe per run/tool.
  - confidence: 0.87
