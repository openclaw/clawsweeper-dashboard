## Summary

Keep open. The audit-logging idea addresses a real operator visibility gap, but this branch is not merge-ready because proof is missing and the proposed logs can misattribute or duplicate security-relevant tool-policy decisions.

## What This Changes

The PR adds INFO-level logs for tool-policy pipeline removals and sandbox blocked-tool decisions, plus three pipeline audit-log tests.

## Best Possible Solution

Carry sanitized matched allow/deny rule metadata through policy evaluation and emit one audit record at enforcement boundaries, with focused tests and redacted real-output proof.

## Reproduction Assessment

Yes, source-level. Current main returns only filtered tools or boolean allow/deny results, and the PR diff shows the new logs infer attribution after matched-rule metadata is lost.

## Solution Assessment

No. The PR is in the right area, but broad step labels are not matched-rule attribution and formatter-side sandbox logging can overcount; enforcement-side logging with explicit metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Report the matched policy rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: This log is derived by diffing the filtered tool list, so it can only report the broad pipeline step. For policies with allow entries, deny entries, wildcards, or group expansion, operators still cannot tell which rule matched even though the feature is rule attribution; carry matcher metadata instead of inferring it after filtering.
  - confidence: 0.87
- **[P2] Emit sandbox audit logs once at enforcement:** `src/agents/sandbox/runtime-status.ts:160-162`
  - body: The new log is inside `formatSandboxToolPolicyBlockedMessage`, which is reached through `formatAssistantErrorText` from run, payload, lifecycle, and failover rendering paths. A single blocked tool can therefore produce repeated INFO audit lines; emit at the enforcement point or dedupe per run/tool.
  - confidence: 0.84
