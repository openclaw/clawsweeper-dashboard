## Summary

Keep open: current main still lacks matched-rule audit attribution, and this branch is useful but currently misattributes some deny removals and can duplicate sandbox audit logs from an error formatter path.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox-blocked tool calls, plus pipeline audit-log tests.

## Best Possible Solution

Carry matched allow/deny rule metadata through policy evaluation, emit one audit record at the policy/enforcement boundary, and keep the linked issue open until that implementation lands.

## Reproduction Assessment

Yes. Source inspection shows a tools.allow-labeled pipeline step can contain deny rules, and the sandbox formatter is reachable from multiple error-rendering paths; I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but the current diff should report the matched allow/deny rule instead of a broad step label and should move sandbox audit logging out of formatting or explicitly dedupe it.
