## Summary

Keep open. The PR is a useful implementation candidate for tool-policy audit logging, but the current branch still misattributes some deny removals, adds an audit side effect to a shared formatter, and lacks after-fix real behavior proof from a real setup.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox-blocked tool calls, plus pipeline audit-log tests and log-spy cleanup.

## Best Possible Solution

Carry matched allow/deny rule metadata through policy evaluation and emit one sanitized audit record at the embedded and gateway enforcement boundaries, with focused tests and real audit-output proof.

## Reproduction Assessment

Yes. Source inspection shows a tools.allow-labeled pipeline step can contain deny rules, and the sandbox formatter is reachable from multiple error-rendering paths; I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but the current diff should report the matched allow/deny rule instead of a broad step label and should move sandbox audit logging out of formatting or explicitly dedupe it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Report the matched allow or deny rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The audit line uses `step.label`, but default global/agent/group policy steps are labeled `*.allow` even when `policy.deny` removes a tool. That can tell operators to fix `tools.allow` for a deny-list block, so carry the matched allow/deny rule and config path from policy evaluation instead.
  - confidence: 0.91

  - body: `formatSandboxToolPolicyBlockedMessage` is reached through shared assistant-error rendering paths. Logging here can emit repeated identical INFO audit entries for one blocked tool call, so emit once at the enforcement boundary or add explicit per-run/tool dedupe.
  - confidence: 0.87
