## Summary

Keep open. The requested audit direction is useful, but current `main` still lacks matched-rule audit attribution and this PR introduces two blocking design bugs plus lacks real behavior proof.

## What This Changes

The PR adds INFO-level audit logging for tool-policy pipeline filtering and sandbox-denied tools, plus pipeline audit-log tests.

## Best Possible Solution

Carry matched allow/deny rule metadata through policy evaluation and emit one sanitized audit record at enforcement boundaries, with focused tests and redacted real audit-output proof.

## Reproduction Assessment

Yes. Source inspection on current `main` shows policy evaluation returns filtered tools or booleans without matched-rule audit attribution, and the PR diff exposes the attribution and duplicate-logging problems without a live run.

## Solution Assessment

No. The direction is useful, but this patch should report matched allow/deny rule metadata and move sandbox audit logging to a single enforcement boundary or explicit per-run/tool dedupe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Report the matched allow or deny rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The audit line uses `step.label`, but default steps such as `tools.allow` can also contain `deny` rules. An allow+deny policy can therefore report a deny removal as `tools.allow`; carry the matched rule type/path from policy evaluation instead.
  - confidence: 0.91
- **[P2] Emit sandbox audit logging once at enforcement:** `src/agents/sandbox/runtime-status.ts:160-162`
  - body: This log is inside `formatSandboxToolPolicyBlockedMessage`, which is reached through shared error-rendering paths via `formatAssistantErrorText`. Rendering the same blocked tool error in multiple places can produce duplicate INFO audit lines, so emit at enforcement or dedupe per run/tool.
  - confidence: 0.87
