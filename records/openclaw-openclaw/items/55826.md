## Summary

Keep open: the linked operator problem is real and still not implemented on current main, but this PR does not yet provide reliable rule attribution and would add repeated sandbox audit side effects from a shared formatter path.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox blocked-tool messages, plus pipeline audit-log tests.

## Best Possible Solution

Expose matched allow/deny rule metadata from policy evaluation, audit pipeline and sandbox decisions once at the policy boundary, and document the operator-visible feature in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a `tools.allow`-labeled pipeline step can include `deny`, and shared assistant-error formatting can be reached from several runtime paths.

## Solution Assessment

No. The direction is useful, but the current implementation logs broad step labels instead of matched rules and places sandbox audit logging in formatting rather than a single enforcement or deduped audit point.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Report the matched allow or deny rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The audit line uses `step.label`, but default global/agent/group steps are labeled `*.allow` even when `policy.deny` removes a tool. That can tell operators to fix `tools.allow` for a deny-list block, so derive the matched allow/deny key and pattern from policy evaluation instead.
  - confidence: 0.91

  - confidence: 0.87
