## Summary

Keep open. The PR targets a real operator/audit gap tracked by #55801, and the earlier mock-restore review note appears addressed by the second commit, but the current patch is not merge-ready because pipeline logs can attribute a deny removal to an allow key, sandbox audit logging is emitted from a shared formatter path, and the user-facing feature lacks the required changelog entry.

## What This Changes

The PR adds INFO-level audit logs for tool-policy pipeline filtering and sandbox blocked-tool decisions, plus unit coverage for pipeline audit log emission.

## Best Possible Solution

Implement audit logging at the policy decision boundary with matched allow/deny rule and source-key attribution, keep user-facing error formatting side-effect-free or explicitly deduped per blocked call, and document the operator-visible change in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a default pipeline step with globalPolicy.deny can be labeled tools.allow, and the unknown-tool sandbox formatter path is reachable from multiple assistant-error rendering call sites.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because attribution needs to come from the matched allow/deny rule rather than the broad step label, and sandbox audit emission should happen once at enforcement time or through explicit dedupe.
