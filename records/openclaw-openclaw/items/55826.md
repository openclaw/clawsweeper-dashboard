## Summary

Keep open: the feature direction is useful and current main still lacks tool-policy audit logs, but this PR should not merge as-is because it can misattribute deny removals as allow-list steps, emits sandbox audit logs from a shared formatter path, and is missing the required changelog entry.

## What This Changes

The PR adds INFO-level tool-policy audit logging for pipeline filtering and sandbox blocked-tool decisions, plus pipeline audit-log unit tests.

## Best Possible Solution

Implement rule-aware audit at the policy decision boundary, return or derive the matched allow/deny key and pattern, emit sandbox audit once per blocked call, and document the operator-visible feature in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a default `tools.allow`-labeled pipeline step can contain `deny`, and the PR logs only the step label; the sandbox formatter path is also reachable from multiple assistant-error rendering call sites.

## Solution Assessment

No. The proposed direction is right, but this implementation does not yet provide reliable rule attribution and places sandbox audit side effects in formatting instead of a single enforcement or deduped audit boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Report the matched allow or deny rule:** `src/agents/tool-policy-pipeline.ts:161`
  - body: The new audit line uses `step.label`, but default global/agent/group steps are labeled `*.allow` even when `policy.deny` removes a tool. A normal deny rule can therefore be logged as `tools.allow`, sending operators to the wrong config key; derive the matched allow/deny key and rule from policy evaluation instead.
  - confidence: 0.91

  - body: `formatSandboxToolPolicyBlockedMessage` is called through shared assistant-error rendering paths, including runner, payload, failover, and lifecycle handling. Logging here can emit repeated identical INFO audit entries for one blocked tool call, so emit once at the enforcement boundary or add explicit per-run/tool dedupe.
  - confidence: 0.87

  - body: This is an operator-visible feature for audit logs, but the PR does not update `CHANGELOG.md`. Add a single Unreleased entry with appropriate contributor credit before merge.
