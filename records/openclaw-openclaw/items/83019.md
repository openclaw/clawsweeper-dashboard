## Summary

Keep open: the feature direction is plausible, but PR-head source writes raw web provider identifiers into secrets diagnostics despite the promised sanitized boundary, and external real behavior proof is still insufficient.

## What This Changes

The PR adds secrets diagnostics spans, gateway startup trace logging, web-tool diagnostics metadata, and startup benchmark coverage for secret-backed config cases.

## Best Possible Solution

Keep the diagnostics span direction, but hash or classify every provider-like value before timeline emission, add web-tool leak coverage, and attach redacted runtime proof for the benchmark and leak audit.

## Reproduction Assessment

Yes for the review finding: PR-head source passes raw configured and selected web provider ids into `secrets.webTools` span attributes when diagnostics are enabled. I did not execute the branch in this read-only review.

## Solution Assessment

No. The instrumentation belongs in the right area, but provider-like attributes should follow the hash/class/count pattern used by `secrets.resolve.provider` before merge.
