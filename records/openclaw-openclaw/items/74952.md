## Summary

Keep open. The feature direction is plausible, but the PR still has blocking scheduler regressions against current main contracts and an unrelated CI supply-chain change that needs separate maintainer/security review before merge.

## What This Changes

This PR adds `cron.maintenance` config, cron and heartbeat maintenance-window gating with deferred replay, status/protocol/UI/config/docs updates, focused tests, and a changed-file/metadata secret scan in CI.

## Best Possible Solution

Revise the PR so maintenance windows preserve existing cron heartbeat override and busy-retry contracts, arm deferred cron replay independently of ordinary future wake times, add the required changelog entry, and move the CI secret-scanning workflow change to a separately reviewed security/CI PR.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by code inspection against current-main tests and contracts: cron main-session wake tests require `heartbeat.target = "last"`, heartbeat wake tests require all busy reasons to retry, and a deferred run plus any unrelated future wake exercises the `armTimer` starvation path.

## Solution Assessment

No. The feature should be implemented in this area, but this PR is not the best mergeable solution until it preserves the existing scheduler contracts and separates the unrelated CI runtime install.

  - body: Current main merges per-wake heartbeat overrides such as `{ target: "last" }` into targeted heartbeat runs so cron output routes back to the last chat. The new helper always passes only the agent's configured heartbeat, so targeted and replayed cron wakes can lose their delivery target.
  - confidence: 0.9

  - confidence: 0.86

  - body: Deferred maintenance runs are only considered when `nextWakeAtMs(state)` is null. If another job has any future `nextRunAtMs`, a deferred run that is already allowed waits for that ordinary wake instead of replaying promptly when the maintenance phase allows it.
  - confidence: 0.88
