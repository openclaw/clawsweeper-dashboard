## Label Justifications

- `P1`: The reported release failure breaks every Windows turn for an affected agent runtime, and the same source-level gap remains for current official CLI backends.
- `impact:crash-loop`: The issue is about process launch failure/ENOENT at agent-turn runtime, which is a process availability problem.

## Summary

This should stay open: current main has moved Codex agent routing away from the released `codex-cli` CLI backend, but the underlying Windows npm-shim spawn gap still exists for official CLI backends and should be fixed in the generic CLI launch path.

## What This Changes

Windows npm-installed CLI backends should spawn `.cmd` shims through the existing Windows spawn resolver instead of failing with ENOENT.

## Best Possible Solution

Keep this as the canonical Windows CLI-backend spawn bug and fix the generic CLI launch path for official npm shim-backed backends while preserving the current Codex app-server route.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: on Windows, an npm-installed CLI backend such as `claude` or `gemini` can resolve to only a `.cmd` shim, while the current supervisor spawns the bare backend command without the Windows spawn helper. I did not run the Windows repro live.

## Solution Assessment

Yes, the suggested helper is the right mechanism, but the maintainable fix should be generic and security-conscious: route official CLI backend commands through `resolveWindowsSpawnProgram`/`materializeWindowsSpawnProgram` with package metadata, not blanket `shell: true`.
