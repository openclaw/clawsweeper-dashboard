## Label Justifications

- `P1`: The shipped-release report describes a Windows runtime failure that blocks every Codex CLI agent turn until users move to a fixed build.
- `impact:crash-loop`: The symptom is a repeated `spawn codex ENOENT` process launch failure for every affected turn.
- `impact:auth-provider`: The affected surface is provider/model runtime routing through `agentRuntime.id` and the Codex runtime alias path.

## Summary

Current main already resolves the Codex-specific failure: the bundled `codex-cli` backend was removed, legacy Codex CLI refs/runtime pins migrate to the Codex app-server runtime, and the app-server stdio launch path uses the Windows spawn resolver with regression coverage.

## What This Changes

The issue asks OpenClaw to stop raw-spawning the legacy Codex CLI command on Windows and route Codex runtime launches through the Windows spawn helper.

## Best Possible Solution

Keep the current main direction: Codex uses the app-server runtime with the Windows spawn helper, legacy `codex-cli` config is repaired by doctor, and broader non-Codex CLI shim handling remains on the separate Windows process fix path.

## Reproduction Assessment

Yes for the reported shipped behavior: the issue and prior Windows Codex reports identify the raw `spawn codex` failure, and current history shows that path was removed/migrated. The same failure is not expected on current main because Codex app-server spawning now resolves Windows shims before `spawn`.

## Solution Assessment

Yes: removing the obsolete bundled `codex-cli` backend and migrating Codex routes to the app-server runtime is more maintainable than wiring a Windows helper into a backend that current main no longer supports.
