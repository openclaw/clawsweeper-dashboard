## Summary

Current main and v2026.6.6 still permit a successful abort-and-drain to return `released=0` when the same pre-existing command-lane task remains active. The open closing PR https://github.com/openclaw/openclaw/pull/92303 targets that primary defect, so this issue should remain paired with it rather than spawning a parallel repair.

## What This Changes

Fix stuck-session recovery so an aborted embedded run actually releases its blocked lane, with follow-up handling for stale target binding, ineffective-recovery reporting, queue-empty visibility, and threshold configuration.

## Best Possible Solution

Prove and land a lane-owner-safe version of the linked fix, then preserve stale-target binding, queue-empty diagnostics, and configurable thresholds as separate focused follow-ups before closing this umbrella report.

## Reproduction Assessment

Yes, at high confidence from source and production evidence: current recovery can complete abort-and-drain while retaining the same active lane task and returning `released=0`. This read-only review did not execute an unreleased build against the reporter's host.

## Solution Assessment

The linked lane-reset condition is a plausible narrow repair for the primary defect, but it is not yet proven to be the best safe fix because clearing a still-executing lane may detach valid work and does not address stale target selection.
