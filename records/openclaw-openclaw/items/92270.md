## Summary

Current main and `v2026.6.6` still allow an abort/drain recovery to report `status=aborted` with `released=0` when the same pre-existing command-lane task remains active and no queue counter triggers a reset. The open closing PR https://github.com/openclaw/openclaw/pull/92303 targets that exact primary gap, so this issue should stay open until that PR is reviewed, proven, and landed or replaced.

## What This Changes

The issue requests reliable stuck-session recovery when `abort_embedded_run` reports success without freeing the affected session lane.

## Best Possible Solution

Land or replace the focused recovery fix for pre-existing active lane tasks, then split any remaining target-binding, observability, and reset-registry behavior into narrower follow-up issues if they still reproduce.

## Reproduction Assessment

Yes at source level: current main can return `status=aborted` with `released=0` after abort/drain when the same pre-existing lane task remains active and no queue counter triggers reset. I did not run the live multi-hour gateway scenario in this read-only review.

## Solution Assessment

Unclear as a finished solution: the linked PR is the narrowest visible fix for the primary lane-release defect, but maintainers still need real behavior proof and a decision on whether the issue's other asks are separate follow-ups.
