## Summary

Current main and v2026.6.6 now cover the queued-session-work form of this recovery failure, but the zero-queue active-task path is still not implemented on main and is owned by the open closing PR; this issue should stay open until that PR is reviewed, proven, and landed or replaced.

## What This Changes

The issue asks for stuck-session recovery to actually free the lane after `abort_embedded_run` reports success, with follow-up handling for stale target binding, recovery verification, reset registry consistency, empty-queue visibility, and thresholds.

## Best Possible Solution
