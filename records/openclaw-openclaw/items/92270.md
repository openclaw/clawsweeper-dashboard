## Label Justifications

- `P1`: The issue documents multi-hour agent-lane outages where recovery repeatedly reports success but users remain wedged until reset or process restart.
- `impact:session-state`: The embedded-run registry and command-lane state can disagree after abort, leaving the session stuck active or bound to the wrong target.
- `impact:message-loss`: User messages stay queued and unanswered while recovery logs `status=aborted` without freeing the lane.
- `impact:crash-loop`: The affected session becomes unavailable indefinitely and observed recovery requires gateway restart or manual reset.

## Summary

Keep open. Current main still has a source-reproducible `aborted`/`released=0` lane-release gap for the primary failure, and the direct closing fix is still the open pull request https://github.com/openclaw/openclaw/pull/92303, so this issue should remain paired with that review rather than being swept closed.

## What This Changes

The issue asks for stuck-session recovery to actually release or clearly mark failed `abort_embedded_run` attempts, bind recovery to the stalled run, keep reset/session registries coherent, and improve recovery observability/configuration.

## Best Possible Solution

Review, prove, and either land or replace https://github.com/openclaw/openclaw/pull/92303 for the primary lane-release defect, then split any remaining target-binding and observability requests into narrower follow-up issues if they still reproduce.
