## Summary

Current main still permits stuck-session recovery to report a successful abort with no lane release, and the open closing PR addresses only the primary release path; keep this issue open until that fix lands and the remaining target-binding and recovery-verification gaps are resolved or separately tracked.

## What This Changes

Fix stuck-session recovery so aborting a stalled run releases the blocked lane, targets the triggering run reliably, and reports when recovery is ineffective.

## Best Possible Solution

Land a narrowly proven fix that resets only a surviving pre-abort lane generation without interrupting newly started work, then add explicit target-mismatch handling and post-recovery progress verification; track idle-queue observability and configurable thresholds separately if they cannot fit safely in the same change.

## Reproduction Assessment

Yes at source level: current main has a concrete abort-and-drain path where the pre-existing lane task can remain active while the release predicate returns zero. The reporter also supplies repeated production evidence, although this review did not execute a live multi-hour wedge reproduction.

## Solution Assessment

Partly. Resetting a still-active pre-abort lane task is the narrowest maintainable fix for the primary defect, but the proposed PR does not solve stale target selection, queued-reset registry consistency, or verification that recovery restored progress.
