## Summary

Current main addresses the central failure reported from v2026.5.28: after aborting stale active work with queued session work, recovery now resets the wedged session lane instead of leaving `released=0`; it also avoids resetting a lane when fresh replacement work started during the abort. The fix is not in v2026.6.5, so this is current-main-only.

## What This Changes

Fix stuck-session recovery that reports an aborted run without releasing the affected session lane, can target stale sessions after reset, and hides idle-queue stalls.

## Best Possible Solution

Keep the focused current-main lane-release implementation and validate it in the next release; file narrower follow-ups only for any independently reproducible reset-binding or diagnostics gaps that remain afterward.

## Reproduction Assessment

Yes. The v2026.5.28 event sequence maps directly to the pre-fix source path where an abort could drain while queued session work did not reset the lane, producing `released=0`.

## Solution Assessment

Yes for the central outage: resetting the owning session lane after abort when queued work exists, while preserving any fresh task that starts during the abort, is the narrow maintainable correction.
