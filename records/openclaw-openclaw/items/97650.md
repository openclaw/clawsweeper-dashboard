## Summary

Keep open: current main removed the obsolete `sessions.json.lock` store-lock path, but session transcript lock timeouts still exist and the global unhandled-rejection handler still exits for unclassified lock-timeout rejections, so the crash-loop policy question is not safely cleanup-closeable.

## What This Changes

The issue reports a session-store lock timeout that can surface as an unhandled rejection and crash-loop a supervisor-managed gateway.

## Best Possible Solution

Keep this open until maintainers decide the current policy for `SessionWriteLockTimeoutError`: caller-scoped graceful degradation for non-critical writes, a narrow non-fatal unhandled-rejection classification, or consolidation into the open stale-lock lifecycle tracker.

Unclear. A global non-fatal exemption is only a plausible mitigation; the best fix likely needs maintainer policy on which session-lock writes can degrade safely versus which should remain fatal or visibly fail.
