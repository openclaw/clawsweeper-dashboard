## Summary

Keep open: current main still throws typed session write-lock timeout errors and still exits on any unhandled non-transient rejection, but the report's exact `withSessionStoreLock` / `sessions.json.lock` stack does not map cleanly to current source, so this needs maintainer scoping rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to handle session-store lock timeouts gracefully so stale lock contention does not become an unhandled rejection and gateway restart loop under launchd.

## Best Possible Solution

Keep the issue open until maintainers decide the correct owner boundary for `SessionWriteLockTimeoutError`: either catch and degrade specific non-critical background writes, or add a narrowly tested non-fatal classification for known session-lock contention without masking real write failures.

## Reproduction Assessment

Yes at source level for the fatal classification: a session write-lock timeout that reaches `unhandledRejection` exits the process on current main. No for the exact report stack yet, because `withSessionStoreLock` and `sessions.json.lock` do not map cleanly to the current source paths I inspected.

## Solution Assessment

Unclear. Making all lock timeouts globally non-fatal is only a plausible mitigation; the maintainable solution should distinguish non-critical background writes from truly failed session persistence and coordinate with the existing lock-lifecycle trackers.
