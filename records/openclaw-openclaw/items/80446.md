## Summary

Keep open: current `main` still sends Telegram Bot API requests through the Telegram transport without stamping a User-Agent, and undici v8.2.0 supplies a low-cardinality default when the header is absent. The ISP throttling itself is environment-specific, but the reported header path is source-backed and narrow enough for a focused fix.

## What This Changes

The issue requests stamping Telegram Bot API outbound requests with a versioned OpenClaw User-Agent instead of allowing Node/undici defaults.

## Best Possible Solution

Add a Telegram-plugin-owned fetch wrapper that stamps a stable versioned OpenClaw User-Agent on every Bot API outbound path while preserving proxy, dispatcher, abort, fallback, and caller-header semantics.

## Reproduction Assessment

No, not for the ISP throttling end-to-end: this environment cannot reproduce a UK consumer-ISP layer-7 throttle. The source path is reproducible with high confidence: current Telegram transport calls undici fetch without adding a User-Agent, and undici supplies the default when the header is missing.

## Solution Assessment

Yes for the core problem: stamping the header at the Telegram transport boundary is the narrow maintainable fix because that module owns Bot API fetch, proxy, dispatcher, and fallback behavior. Header stripping should not be bundled unless it is separately justified because it changes more request metadata than needed for the reported fingerprint.
