## Summary

Keep open: current main still sends Telegram Bot API requests through the Telegram transport without stamping a User-Agent, and the linked closing PR is the right implementation path to review before this issue closes.

## What This Changes

Request that the Telegram plugin stamp a stable OpenClaw User-Agent on outbound Telegram Bot API requests to avoid ISP throttling of the bare undici fingerprint.

## Best Possible Solution

Review and land or close the linked Telegram fix PR, keeping the final change at the Telegram transport boundary with caller-provided headers, proxy behavior, dispatchers, aborts, and fallback semantics preserved.

## Reproduction Assessment

No for the ISP throttling end-to-end; that requires the affected network path. Yes at source level: current main routes Telegram outbound requests through undici without a Telegram-owned User-Agent stamp, and undici supplies the bare default when the header is absent.

## Solution Assessment

Yes, stamping the User-Agent in the Telegram transport is the narrow maintainable fix because that module owns Bot API fetch, proxy, dispatcher, and fallback behavior; broader header stripping should be separate follow-up if needed.
