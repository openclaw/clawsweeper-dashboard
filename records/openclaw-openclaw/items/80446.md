## Summary

Keep open: current main still routes Telegram Bot API requests through the Telegram transport without stamping a User-Agent, and the only related fix PR was closed unmerged. The ISP throttling is environment-specific, but the header path and undici default are source-backed enough for a focused repair.

## What This Changes

Telegram outbound Bot API requests should avoid undici's default User-Agent by using a stable versioned OpenClaw User-Agent at the Telegram transport boundary.

## Best Possible Solution

Add a Telegram-owned fetch wrapper that stamps a real versioned OpenClaw User-Agent on every Bot API path, preserves caller headers and proxy/dispatcher behavior, and proves the patched transport with focused tests plus live or equivalent redacted transport output.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the ISP throttling end-to-end because it depends on a specific network path; yes at source level because current main sends Telegram transport requests without a User-Agent and undici supplies a default when absent.

## Solution Assessment

Is this the best way to solve the issue? Yes for the direction: stamping at the Telegram transport boundary is the narrow maintainable fix. The prior PR's version-source and proof gaps should be addressed, and optional header stripping should stay out unless separately justified.
