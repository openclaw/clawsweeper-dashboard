## Summary

Keep open: current main still routes Telegram Bot API requests through the selected undici/proxy fetch without a User-Agent wrapper, and the only linked fix PR was closed unmerged, so the reported fingerprint remains unresolved.

## What This Changes

Request that Telegram Bot API outbound fetches use a stable versioned OpenClaw User-Agent instead of relying on undici's default header.

## Best Possible Solution

Add a Telegram-plugin-owned fetch wrapper that stamps a stable `OpenClawBot/<runtime version>` User-Agent on every Bot API outbound path while preserving caller custom headers, Request/init semantics, proxy dispatchers, and fallback behavior.

## Reproduction Assessment

No for the end-to-end ISP throttle, because that depends on a specific network path. Yes at source level: current main omits a User-Agent in the Telegram transport and undici appends its low-cardinality default when absent.

## Solution Assessment

Yes for the main direction: the Telegram transport boundary owns these Bot API fetches and is the narrow place to stamp the header. The safer implementation should preserve caller-provided User-Agent values and use OpenClaw's service/runtime version fallback rather than broad header stripping.
