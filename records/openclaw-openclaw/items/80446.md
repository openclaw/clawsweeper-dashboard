## Summary

Keep open: current `main` still sends Telegram Bot API traffic through the Telegram transport without stamping a stable OpenClaw `User-Agent`, and the only linked fix PR was closed unmerged. The ISP throttling is environment-specific, but the missing-header path is source-backed and narrow.

## What This Changes

The issue requests stamping a stable OpenClaw User-Agent on outbound Telegram Bot API requests so they do not use undici's low-cardinality default header.

## Best Possible Solution

Add a Telegram-plugin-owned fetch wrapper that stamps a stable versioned OpenClaw User-Agent on all Bot API outbound paths while preserving caller-provided headers, proxy/dispatcher behavior, abort handling, and fallback retries.

## Reproduction Assessment

No for the ISP throttling end-to-end, because that needs a specific UK consumer ISP path. Yes at source level: current `main` still sends Telegram transport requests through undici/proxy fetches without adding a `User-Agent`.

## Solution Assessment

Yes, stamping the header at the Telegram transport boundary is the narrow maintainable fix. The optional `Sec-Fetch-Mode` or `Accept-Language` stripping is broader than the proven bug and should be left out unless separately justified.
