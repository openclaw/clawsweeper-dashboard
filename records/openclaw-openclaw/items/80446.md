## Summary

Keep open: current main still routes Telegram Bot API traffic through the Telegram transport without adding a User-Agent, so the reported undici default-header path remains source-backed. The ISP throttling is environment-specific, but the repair boundary is narrow and belongs in the Telegram plugin transport.

## What This Changes

Request to stamp Telegram Bot API outbound fetches with a stable versioned OpenClaw User-Agent so they do not rely on undici's bare default header.

## Best Possible Solution

Add a Telegram-plugin-owned transport wrapper that sets a stable versioned OpenClaw User-Agent on every Bot API fetch path while preserving proxy, dispatcher, abort, retry, fallback, and caller-header behavior.

## Reproduction Assessment

No, not for the ISP throttling end-to-end: that requires the affected consumer ISP path. The source path is high-confidence: current main calls undici-backed Telegram fetch without adding User-Agent, and undici fills a default when the header is absent.

## Solution Assessment

Yes for the core issue: the Telegram transport owns Bot API fetch selection, dispatcher fallback, and proxy handling, so stamping the header there is the narrowest maintainable fix. Header stripping should be treated as a separate decision rather than bundled into the first repair.
