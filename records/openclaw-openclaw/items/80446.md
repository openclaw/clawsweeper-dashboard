## Summary

Keep open: current main still routes Telegram Bot API traffic through the Telegram transport without stamping a User-Agent, while the imported undici package appends its own low-cardinality default when none is present. The linked fix PR was closed unmerged, so the issue still has a narrow remaining code path to repair.

## What This Changes

Request to stamp a stable versioned User-Agent on Telegram Bot API outbound fetches so undici's default does not trigger ISP throttling.

## Best Possible Solution

Add a Telegram-plugin-owned fetch wrapper that stamps a real versioned OpenClaw User-Agent on every Bot API path while preserving caller headers, proxy dispatchers, abort handling, and fallback behavior.

## Reproduction Assessment

No for the ISP throttling end-to-end because it depends on a specific network path. Yes at source level: current main calls undici-backed Telegram transport fetches without a User-Agent, and undici appends its own default when absent.

## Solution Assessment

Yes, stamping at the Telegram transport boundary is the narrow maintainable solution because that module owns Bot API fetch, proxy, dispatcher, and fallback behavior. Do not bundle `Sec-Fetch-Mode` or `Accept-Language` stripping without a separate maintainer decision.
