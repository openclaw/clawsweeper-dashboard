## Summary

Keep open: current main still routes Telegram Bot API requests through the Telegram transport without stamping a User-Agent, and the only linked fix PR is closed unmerged, so the source-level problem remains.

## What This Changes

The issue asks OpenClaw to stamp outbound Telegram Bot API fetches with a stable versioned OpenClaw User-Agent instead of relying on undici's default header.

## Best Possible Solution

Add a Telegram-plugin-owned fetch wrapper that stamps a versioned OpenClaw User-Agent on every Bot API outbound path while preserving proxy, dispatcher, abort, fallback, Request/init, and caller-header semantics.

## Reproduction Assessment

No end-to-end for the ISP throttling, because that depends on a specific network path. The source path is high-confidence: current main calls undici-backed Telegram fetches without adding a User-Agent, and undici appends a default when none is present.

## Solution Assessment

Yes: stamping the User-Agent at the Telegram transport boundary is the narrow maintainable fix. Stripping `Sec-Fetch-Mode` or `Accept-Language` should be left out unless separately justified.

Fix https://github.com/openclaw/openclaw/issues/80446.
