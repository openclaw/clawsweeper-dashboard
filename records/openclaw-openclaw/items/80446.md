## Summary

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83254 if maintainers accept the narrow Telegram transport header policy, then close this issue once the fix reaches main.

## Reproduction Assessment

No, we do not have a high-confidence end-to-end reproduction of the ISP throttling behavior. The source path is reproducible from current main: Telegram fetches do not set a User-Agent, and undici adds its default when absent.

## Solution Assessment

Yes, the linked transport-boundary wrapper is the narrow maintainable fix direction because extensions/telegram/src/fetch.ts owns Telegram fetch selection, proxy dispatchers, fallback, and media sourceFetch. The issue should remain open until that PR or an equivalent fix lands.
