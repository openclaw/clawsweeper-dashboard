## Summary

Keep open: current main still routes Telegram Bot API calls through the Telegram plugin's undici-backed transport without any User-Agent stamping, and no duplicate or merged fix owns the remaining work. The exact ISP throttling was not reproduced here, but the source path and dependency default-header behavior make this a valid, narrow Telegram transport bug candidate.

## What This Changes

The issue asks OpenClaw to stamp Telegram API requests with a less fingerprintable, versioned User-Agent instead of allowing Node/undici's default User-Agent to reach api.telegram.org.

## Best Possible Solution

Add a Telegram plugin transport-layer wrapper that stamps a versioned OpenClaw User-Agent on all Telegram Bot API fetch paths, including dispatcher fallback and exported sourceFetch paths, with focused regression coverage for proxy and custom transport behavior.

## Reproduction Assessment

No, not for the full ISP throttling symptom because this environment does not provide the reported UK consumer ISP path. Yes for the source-level trigger: current main has no Telegram User-Agent override and dependency/default fetch behavior supplies low-cardinality headers when none are provided.

## Solution Assessment

Yes, the maintainable fix seam is the Telegram plugin's resolveTelegramTransport path, not core fetch defaults or a new config knob. The first pass should stamp a versioned User-Agent and avoid optional stripping of other headers unless a maintainer explicitly wants that broader policy change.
