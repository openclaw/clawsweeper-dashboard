## Summary

Keep open: current main still sends Telegram Bot API transport requests through the selected `sourceFetch` without stamping a Telegram-owned `User-Agent`, and the active fix path is the open closing PR at https://github.com/openclaw/openclaw/pull/83254. This issue should close from the merge if that PR lands, not before.

## What This Changes

Request that Telegram Bot API outbound fetches avoid the bare undici default User-Agent by stamping a versioned OpenClawBot header to reduce ISP throttling.

## Best Possible Solution

Review and land or reject the open Telegram transport PR; if it lands, close this issue from the merge and keep optional header stripping as separate follow-up only if proven necessary.

## Reproduction Assessment

No for the end-to-end ISP throttle, because it depends on a specific network and sustained Telegram traffic. Yes at source level: current `resolveTelegramTransport` calls `sourceFetch` without a Telegram-owned `User-Agent`, and undici supplies a default when it is absent.

## Solution Assessment

Yes: stamping a versioned User-Agent inside the Telegram transport boundary is the narrow maintainable fix because that module owns Bot API fetch, proxy dispatchers, fallback retries, and media `sourceFetch` callers. The open PR is the active solution path, so this issue should remain open until it merges or is rejected.
