## Summary

Keep open: current main still selects Telegram Bot API fetch implementations and calls them without stamping a User-Agent, while undici 8.2.0 supplies the bare default when no header is present. The linked fix PR was closed unmerged, so the issue remains a narrow actionable Telegram transport bug rather than an implemented cleanup item.

## What This Changes

The issue asks for Telegram Bot API outbound requests to stamp a stable versioned OpenClaw User-Agent instead of relying on undici's bare default fingerprint.

## Best Possible Solution

Add a Telegram-plugin-owned fetch wrapper in resolveTelegramTransport that stamps a stable versioned OpenClaw User-Agent on every Bot API outbound path while preserving caller-provided User-Agent, proxy, dispatcher, abort, fallback, and custom apiRoot behavior.

## Reproduction Assessment

No for the ISP throttling end-to-end because it depends on a specific network path; yes at source level because current main calls undici-backed Telegram fetch without adding User-Agent, and undici appends its bare default when absent.

## Solution Assessment

Yes for the core direction: stamping the header at the Telegram transport boundary is the narrow maintainable fix. Optional stripping of other headers is broader than the proven issue and should be left out unless separately justified.
