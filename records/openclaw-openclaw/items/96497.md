## Summary

Keep open: current main and the latest release still contain the JSON-to-SSE wrapping path that can produce `data: data: ...`, and https://github.com/openclaw/openclaw/pull/96503 is the viable candidate fix rather than a reason to close before merge.

## What This Changes

The issue reports that OpenAI-compatible streaming responses with an SSE body mislabeled as JSON are rewrapped as `data: data: ...`, causing JSON parse failures before an agent reply.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96503, folding in any needed defense-in-depth there while preserving true JSON fallback behavior and avoiding a new strict-mode config option.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.10` can feed an already-SSE `application/json` body into the JSON-to-SSE fallback, producing `data: data: ...` before the OpenAI SDK parses it.

## Solution Assessment

Yes. The narrow maintainable solution is provider transport normalization that distinguishes true JSON bodies from already-SSE bodies before synthesis; adding a new strict-mode config option is not needed for the reported regression.
