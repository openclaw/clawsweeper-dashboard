## Summary

Keep open: current main and the latest release still contain the JSON-to-SSE synthesis path that can turn an already-SSE `application/json` stream into `data: data: ...`; the linked fix PR is still open, so the issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue reports a shipped `openai-completions` regression where custom OpenAI-compatible streaming gateways can fail before producing a reply because OpenClaw re-prefixes SSE frames before the OpenAI SDK parses them.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96503 or an equivalent narrow `provider-transport-fetch` fix that distinguishes true JSON from already-SSE bodies without adding a new config option.

## Reproduction Assessment

Yes, at source level. A streamed SDK request receiving an already-SSE body with `application/json` content type reaches the JSON-to-SSE synthesis branch and can produce `data: data: ...`; I did not run a live gateway in this read-only review.

## Solution Assessment

Yes, the best fix is the narrow transport normalization repair already proposed by the linked PR: sniff or classify JSON-labeled bodies before synthesis, preserve the true-JSON fallback, and avoid the requested strict-mode config option.
