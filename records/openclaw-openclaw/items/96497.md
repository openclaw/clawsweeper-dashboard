## Summary

Keep open: current main and the latest release still contain the JSON-labeled streaming body wrap path that can produce `data: data: ...`, and the issue is already paired with an open proof-positive fix candidate at https://github.com/openclaw/openclaw/pull/96503.

## What This Changes

The issue requests that OpenAI-compatible streaming responses labeled as JSON not be re-wrapped into `data: data: ...` before OpenAI SDK parsing.

## Best Possible Solution

Land one narrow provider-transport fix that preserves true JSON fallback, treats already-SSE JSON-labeled bodies as SSE, and closes this issue after the fix reaches main.

## Reproduction Assessment

Yes. A streaming OpenAI SDK request whose response has `Content-Type: application/json` but an already-SSE body flows through the current JSON-to-SSE synthesis branch and can become `data: data: ...`; I did not run a live gateway in this read-only review.

## Solution Assessment

Yes. The best fix is content-aware provider-transport normalization, not a new config option; the open canonical PR at https://github.com/openclaw/openclaw/pull/96503 is the appropriate landing path to review.
