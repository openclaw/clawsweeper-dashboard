## Summary

Keep open: current main and the latest release still have the JSON-labeled streaming path that can wrap an already-SSE body into `data: data: ...`, and open fix PRs with closing syntax need maintainer review before this issue can close.

## What This Changes

The issue reports a regression where OpenAI-compatible streaming responses labeled as JSON can be rewrapped as `data: data: ...`, causing OpenAI SDK JSON parse failures before an agent reply.

## Best Possible Solution

Land the narrow provider-transport fix, preferably through the canonical open PR at https://github.com/openclaw/openclaw/pull/96503, and fold in any proven defense-in-depth coverage there if maintainers want it.

## Reproduction Assessment

Yes, at source level: a streaming request receiving an already-SSE body with `application/json` content type reaches the JSON synthesis branch and can be emitted as `data: data: ...`. I did not run live or local tests because this review was read-only.

## Solution Assessment

Yes for the narrow transport repair; adding a new config option is not the best default fix. The maintainable path is to distinguish true JSON bodies from already-SSE bodies in `provider-transport-fetch` while preserving true JSON synthesis and invalid-response rejection.
