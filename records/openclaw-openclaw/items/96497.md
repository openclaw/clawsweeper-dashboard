## Summary

Keep open: current main still has a source-reproducible OpenAI-compatible streaming regression where a streaming response labeled as JSON is blindly wrapped as SSE, so an upstream `data: {...}` frame can become `data: data: {...}` before the OpenAI SDK parses it. The same code is present in the latest release, so this needs a narrow provider-transport fix rather than cleanup.

## What This Changes

The issue reports that custom `openai-completions` gateways can fail before replying when OpenClaw double-prefixes streamed SSE data before JSON parsing.

## Best Possible Solution

Teach `provider-transport-fetch` to distinguish true JSON bodies from already-SSE bodies before JSON-to-SSE synthesis, keep bad-response rejection intact, and cover both true JSON fallback and mislabeled SSE with focused tests.

## Reproduction Assessment

Yes at source level, but not with a live gateway in this read-only review. A streaming request that receives an already-SSE body with an `application/json` content type will be wrapped as `data: data: ...`, and the OpenAI SDK then parses `data: {...}` as JSON.

## Solution Assessment

Yes, the narrow best fix is in the guarded OpenAI SDK transport sanitizer, not a new `sse.strictMode` config option. The repair should sniff or classify the body before synthesis and add regression coverage for mislabeled SSE plus true JSON fallback.
