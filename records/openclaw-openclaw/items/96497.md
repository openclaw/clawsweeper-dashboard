## Summary

Keep open: current main and the latest release still contain a plausible OpenAI-compatible stream fallback path that can wrap an already-SSE body as `data: data: ...`, matching the reported OpenAI SDK JSON parse failure.

## What This Changes

The issue reports a regression where `openai-completions` streaming responses from custom OpenAI-compatible gateways can be double-prefixed as `data: data: ...`, causing JSON parsing to fail before an assistant reply is produced.

## Best Possible Solution

Fix the provider transport fallback so already-SSE bodies are detected before JSON-to-SSE synthesis, while preserving true JSON-body streaming fallback and invalid HTML/body rejection.

## Reproduction Assessment

No live high-confidence reproduction was established in this worker. Source inspection plus an OpenAI SDK 6.39.1 scratch proof reproduces the exact parser failure when an already-SSE body is routed through the JSON-to-SSE synthesis fallback, but the reporter's response headers are still missing.

## Solution Assessment

Yes, the best path is a narrow transport fix rather than a new `sse.strictMode` configuration option. The repair should classify an already-SSE body before wrapping JSON responses and add focused regression coverage for both the SSE-body and true-JSON fallback cases.

Fix the `openai-completions` double-prefixed SSE parser failure reported in https://github.com/openclaw/openclaw/issues/96497. In `src/agents/provider-transport-fetch.ts`, before JSON-content-type body synthesis wraps content as SSE, detect whether the response body is already SSE-formatted and pass it through or normalize the content type instead of emitting `data: data: ...`.
