## Summary

Keep open. Current main still contains the reconnect cap this PR targets, the linked Slack bug has source-backed reproduction evidence, and the one-line branch matches the provider's existing zero-as-unlimited retry logic; the remaining action is maintainer and CI review, not cleanup closure.

## What This Changes

The PR changes the Slack Socket Mode reconnect policy constant from 12 attempts to 0 so the existing provider loop retries indefinitely.

## Best Possible Solution

Review and land this narrow policy change if maintainers agree Slack should reconnect indefinitely by default, with focused Slack monitor coverage if requested. Track provider supervision and reconnect-window delivery hardening separately so this PR stays scoped to reconnect exhaustion.

## Reproduction Assessment

Yes. Repeated Slack Socket Mode disconnects or start failures on current main increment `reconnectAttempts` until the hardcoded 12-attempt cap throws, matching the linked #72808 field logs and the provider source path.

## Solution Assessment

Yes for the permanent reconnect-exhaustion bug. The patch uses the existing `maxAttempts <= 0` convention in the provider loop, keeps non-recoverable auth failures fail-fast, and avoids expanding config or gateway supervision policy in this small Slack fix.
