## Summary

Keep open. The Slack source change is a plausible fix for the capped-reconnect failure behind the linked message-loss report, but this PR is not merge-ready because it adds release-owned changelog content, lacks sufficient after-fix live Slack proof, and changes recovery policy in an area with an open maintainer decision about unlimited retry versus supervisor-visible escalation.

## What This Changes

The PR changes Slack Socket Mode reconnect policy from 12 attempts to unlimited retries and adds a CHANGELOG entry for that behavior.

## Best Possible Solution

Decide the Slack recovery policy, remove the CHANGELOG edit, and land a narrow Slack-owned fix with live proof that reconnect continues past the old cap and delivers messages after recovery.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main throws after 12 recoverable Slack socket reconnect/start failures, and linked field logs show users hitting capped Slack retry behavior. I did not run a live Slack disconnect scenario in this read-only review.

## Solution Assessment

Unclear whether this is the best final fix. Removing the cap is a clean mitigation for the capped-exhaustion bug, but maintainers still need to choose between unlimited in-process retry and explicit escalation for stuck Slack providers.
