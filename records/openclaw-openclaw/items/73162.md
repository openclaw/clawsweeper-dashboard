## Summary

Keep this PR open for maintainer review: current main still has the Slack Socket Mode reconnect cap, while the PR is a narrow one-line change that uses the existing `maxAttempts <= 0` unlimited convention and preserves the non-recoverable auth-error fast-fail path. I did not find a blocking correctness or security issue in the proposed diff.

## What This Changes

The PR changes Slack Socket Mode reconnect policy by setting `SLACK_SOCKET_RECONNECT_POLICY.maxAttempts` from `12` to `0` in `extensions/slack/src/monitor/reconnect-policy.ts` so the existing reconnect loop retries indefinitely.

## Best Possible Solution

Land the one-line policy change if maintainers agree Slack Socket Mode should behave like a long-running transport by default, ideally with a focused regression assertion for unlimited reconnects. Keep disconnected-status hardening and broader provider supervision as separate follow-up work so this PR stays scoped to reconnect exhaustion.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: repeated Slack Socket Mode disconnect or start-failure events increment `reconnectAttempts` until the `12` attempt cap throws, and #72808 provides matching field logs for the capped retry behavior.

## Solution Assessment

Yes. The PR uses the narrowest existing seam because the provider already interprets non-positive `maxAttempts` as unlimited, logs that as infinity, and still fails fast for known permanent Slack auth errors before the cap check.
