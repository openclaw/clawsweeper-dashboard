Keep open for normal PR handling: current main still has the 12-attempt Slack Socket Mode reconnect cap, while the PR now applies the narrow policy change and includes the required changelog entry. I found no blocking code or security findings in the proposed diff.

## What This Changes

The PR changes Slack Socket Mode reconnects to unlimited retries by setting `SLACK_SOCKET_RECONNECT_POLICY.maxAttempts` to `0` and adds a changelog entry for #72808.

## Best Possible Solution

Land the policy change and changelog after normal CI/mergeability gates, and track disconnected-status or supervisor hardening separately if maintainers want that broader reliability work.

## Reproduction Assessment

Yes from source inspection and field logs, though I did not run a live Slack disconnect loop in this read-only pass. Current main increments reconnect attempts until the 12-attempt guard throws, and the linked field report/comments match that exhaustion path.

## Solution Assessment

Yes. Setting `maxAttempts: 0` is the narrowest maintainable fix because the existing loop already treats zero as unlimited, keeps abortable capped backoff, and still fails fast for permanent Slack auth errors.
