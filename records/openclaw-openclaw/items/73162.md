## Summary

Keep open. The Slack reconnect-cap bug is real and this PR targets the right policy, with linked field logs showing the mitigation working, but the patch should not merge until the non-recoverable Slack Socket Mode start-error matcher covers the remaining terminal errors from the Slack SDK.

## What This Changes

This PR changes Slack Socket Mode reconnects from a 12-attempt cap to unlimited retries and adds a changelog entry for the linked Slack outage bug.

## Best Possible Solution

Keep the unlimited reconnect behavior for recoverable Slack socket failures, but first extend the non-recoverable Slack error matcher and tests to cover the upstream terminal start errors, then land the focused policy change.

## Reproduction Assessment

Yes. Current main has a source-visible path where repeated recoverable Slack disconnect or start-failure events hit the 12-attempt guard and throw, and the linked issue contains matching field logs plus a patched mitigation log.

## Solution Assessment

No, not quite. Setting `maxAttempts: 0` is the right narrow direction, but the safe patch should first add Slack SDK terminal errors such as `user_removed_from_team` and `team_disabled` to the non-recoverable matcher and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle terminal Slack start errors before unlimited retries:** `extensions/slack/src/monitor/reconnect-policy.ts:12`
  - body: The upstream `@slack/socket-mode` 2.0.7 source marks `user_removed_from_team` and `team_disabled` as unrecoverable Socket Mode start failures, but `SLACK_AUTH_ERROR_RE` does not match either. After this line makes retries unlimited, those permanent workspace/app states would no longer fail out after 12 attempts and can loop/log forever. Please extend the non-recoverable matcher and tests before removing the cap.
  - confidence: 0.88
