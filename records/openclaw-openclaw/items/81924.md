## Summary

Keep open. Current main still matches the reported source path: suppressed control replies can drop the visible `session.message`, while the follow-up `sessions.changed` message-phase payload has no marker that tells clients the transcript update was intentionally hidden.

## What This Changes

Request an additive `sessions.changed` marker when a silent/control reply suppresses the corresponding `session.message` broadcast.

## Best Possible Solution

Define and document an additive `sessions.changed` suppressed-update contract, then implement it at the transcript broadcast/projection boundary with focused websocket regression coverage.

## Reproduction Assessment

Yes, by source inspection: a text-only assistant `NO_REPLY` update is filtered by display projection, causing no `session.message`, while the subsequent `sessions.changed` payload has no suppression marker. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear. Adding a marker in `handleTranscriptUpdateBroadcast` is a narrow likely direction, but treating every falsy projection as `silent_reply` would be too broad without an explicit drop-reason contract.
