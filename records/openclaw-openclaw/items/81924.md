## Summary

Keep open. Current main still has the reported client-observability gap, and the requested marker would add a protocol-visible event payload contract that needs maintainer/API confirmation before implementation.

## What This Changes

The issue requests an additive suppression marker on `sessions.changed` message-phase events when a silent-reply update is filtered from `session.message`.

## Best Possible Solution

Add an additive, typed Gateway event payload contract for suppressed transcript updates, document it, and cover the silent-token path with a focused session-event regression test.

## Reproduction Assessment

Yes, by source inspection: a text-only assistant `NO_REPLY` update is filtered by display projection, causing no `session.message`, while the following `sessions.changed` payload has no suppression marker. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear. Reusing the existing `sessions.changed` message-phase broadcast looks like the narrowest implementation point, but the protocol-visible marker and reason taxonomy should be decided before treating every falsy projection as `silent_reply`.
