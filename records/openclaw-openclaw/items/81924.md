## Summary

Keep open. Current main and v2026.6.10 still filter silent-control assistant transcript updates out of `session.message` and emit only a bare `sessions.changed` `phase: "message"` payload without `suppressed` or `suppressedReason`; no canonical related PR exists, but the fix needs maintainer event-contract approval.

## What This Changes

Add an explicit suppression marker and reason to the `sessions.changed` message-phase event when a silent-control transcript update is filtered out of `session.message`.

## Best Possible Solution

Have a maintainer approve the additive `sessions.changed` suppression semantics, then update the broadcaster, protocol docs or schema surface if needed, and focused gateway tests for suppressed control replies and visible-message non-duplication.

## Reproduction Assessment

Yes, from source inspection: an exact silent-control assistant message without non-text content is display-filtered, and the fallback `sessions.changed` message-phase event has no explicit suppression metadata. I did not run a live gateway test because this read-only review cannot create runtime artifacts in the checkout.

## Solution Assessment

Unclear until maintainers choose the public event contract. The narrow implementation likely belongs in `src/gateway/server-session-events.ts`, but field names, reason values, docs, and protocol typing need API/product confirmation first.
