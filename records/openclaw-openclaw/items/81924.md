## Summary

Keep open. Current main and the latest release still drop silent-control transcript updates down to a bare `sessions.changed` message-phase event without a suppression marker, and the requested fix is a Gateway event-payload contract change that needs maintainer/API confirmation rather than stale cleanup.

## What This Changes

Request adds a `suppressed` marker and optional reason to `sessions.changed` when a silent-control transcript update is filtered instead of broadcasting `session.message`.

## Best Possible Solution

Add an additive, documented Gateway `sessions.changed` message-phase suppression contract with an explicit reason taxonomy, SDK/docs coverage, and a focused regression test once maintainers approve the API shape.

## Reproduction Assessment

Yes from source inspection: a subscribed session event client can receive a `sessions.changed` message-phase event after `projectChatDisplayMessage` filters a `NO_REPLY`-only assistant row, and current main has no marker on that fallback payload. I did not run a live agent turn in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public payload semantics. The narrow implementation likely belongs in `src/gateway/server-session-events.ts`, but the reason taxonomy and docs/SDK contract should be confirmed first.
