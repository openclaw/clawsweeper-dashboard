## Summary

Keep open: current `main` still has the reported unmarked suppressed-reply path, and the requested fix is a new additive gateway protocol field rather than cleanup-close material.

## What This Changes

The issue requests a `sessions.changed` payload marker when a silent/control assistant reply is filtered out before `session.message` broadcast.

## Best Possible Solution

Define an additive gateway event contract for suppressed transcript updates, then implement it in the session-event broadcaster with focused websocket coverage and protocol docs.

## Reproduction Assessment

Yes, from source: a text-only `NO_REPLY` assistant message is filtered by `projectChatDisplayMessage`, so no `session.message` is sent while an unmarked `sessions.changed` message-phase event still broadcasts. I did not run a live gateway websocket repro.

## Solution Assessment

Unclear until maintainers accept the protocol shape. Adding an additive marker to the existing `sessions.changed` event looks narrow, but the field names and reason semantics should be decided as part of the gateway wire contract.
