## Summary

Keep open. Current main and the latest release still drop text-only `NO_REPLY` updates from `session.message` while emitting `sessions.changed` without a suppression marker, so the client-observability gap remains and the protocol-visible payload contract needs maintainer agreement.

## What This Changes

The issue requests an additive suppression marker on `sessions.changed` message-phase events when a silent control reply is intentionally dropped from `session.message`.

## Best Possible Solution

Define an additive Gateway event payload contract for suppressed transcript updates, then implement it at the broadcast/projection boundary with protocol docs and regression coverage.

## Reproduction Assessment

Yes, by source inspection: a text-only assistant `NO_REPLY` update projects to no display message, so no `session.message` is sent, while the following `sessions.changed` message-phase payload has no marker. I did not run a live gateway.

## Solution Assessment

Unclear. Adding a marker to the existing `sessions.changed` frame is likely narrow, but treating every falsy projection as `silent_reply` would be too broad because the projection layer also hides other non-visible transcript updates.
