## Summary

Keep open: this is a focused fix for a real Nextcloud Talk webhook bug, but the patch turns every strict payload parse miss into `ignore`, so malformed JSON and malformed Note-shaped payloads would be acknowledged with 200/no dispatch. The contributor's added proof comment is descriptive and still does not show output from the patched webhook path.

## What This Changes

The PR changes the Nextcloud Talk webhook decoder to return `ignore` for strict payload parse failures instead of writing HTTP 400 `Invalid payload format`.

## Best Possible Solution

Use a two-stage webhook decoder that preserves 400 for malformed JSON and malformed message-shaped payloads, returns 200/no dispatch only for valid non-message Activity Streams events, and includes signed webhook tests plus redacted real-path proof.

## Reproduction Assessment

Yes, source-level. Current main sends signed webhook bodies through a strict Note schema and writes HTTP 400 on any schema miss before ignore handling; I did not run a live Nextcloud Talk server in this read-only review.

## Solution Assessment

No. The handler is the right layer, but this implementation collapses malformed JSON, malformed message-shaped payloads, and valid non-message events into the same 200/no-dispatch result.
