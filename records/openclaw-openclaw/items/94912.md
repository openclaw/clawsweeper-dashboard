## Summary

Keep open: this PR targets a real Nextcloud Talk webhook bug, but the patch is not merge-ready because it turns every strict parse miss into `ignore`, including malformed JSON and malformed Note-shaped payloads that current behavior and the linked issue expect to keep on HTTP 400. The supplied proof is descriptive only and does not show after-fix output from the patched webhook path.

## What This Changes

The PR changes `extensions/nextcloud-talk/src/monitor.ts` so `decodeWebhookCreateMessage` returns `ignore` instead of writing HTTP 400 when the strict webhook payload schema does not parse.

## Best Possible Solution

Use a two-stage webhook decoder that preserves 400 for malformed JSON and malformed message-shaped payloads, returns 200/no dispatch only for valid non-message Activity Streams events, and adds signed webhook regression coverage plus redacted real-path proof.

## Reproduction Assessment

Yes, source-level. Current main sends signed webhook bodies through a strict Note schema and writes HTTP 400 on any schema miss before ignore handling; I did not run a live Nextcloud Talk server in this read-only review.

## Solution Assessment

No. The handler is the right layer, but this implementation collapses malformed JSON, malformed message-shaped payloads, and valid non-message events into the same 200/no-dispatch result.
