## Summary

Keep open. Current main still has a plausible duplicate/echo path for the reported native persistent sub-agent flow: the implemented skip only covers parent-owned ACP sessions, while waited native `sessions_send` still returns the child reply inline and then starts the A2A reply/announce flow using the target session's stored delivery context. Related #53145 is useful broader context but remains open and does not terminally supersede this narrower native persistent sub-agent report.

## What This Changes

Fix duplicate parent-visible replies when `sessions_send` targets a persistent native sub-agent session that has parent streaming or inherited announce delivery context.

## Best Possible Solution

Fix the native parent-owned persistent sub-agent/session-send contract so a parent sending to its own native child has one parent-visible result path. The likely maintainable shape is a generic ownership/target-kind guard that suppresses duplicate A2A announce/reply reinjection for parent-owned native sub-agent targets while preserving peer A2A, thread-bound user follow-ups, and the existing ACP-specific skip.

Yes. The issue provides concrete steps, and current source gives a high-confidence code-level reproduction path: a parent-owned native child can keep a stored delivery context, `sessions_send` can return its reply inline, and the A2A flow can then announce through the target session's delivery context.

## Solution Assessment

No, current main is not the best solution yet. It fixed the related ACP background-child slice, but the narrower remaining fix should detect parent-owned native persistent sub-agent sends and choose a single delivery path without disabling unrelated A2A behavior.
