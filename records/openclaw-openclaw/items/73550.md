## Summary

Keep open. Current main still has a plausible native persistent sub-agent duplicate/echo path: the only skip guard is ACP-specific, native sub-agent sessions can retain a stored delivery context, and waited `sessions_send` can return the child reply inline while also starting the A2A announce flow. Related #53145 is useful broader context but remains open and does not terminally supersede this narrower native sub-agent report.

## What This Changes

Fix `sessions_send` so parent-to-native persistent sub-agent replies are delivered once and do not re-enter the parent conversation via the stored delivery-context announce path.

## Best Possible Solution

Add a generic parent-owned native sub-agent/session-send guard so a parent sending to its own native child has one parent-visible result path. The fix should suppress duplicate A2A announce/reply reinjection for that owner-child pair while preserving peer A2A, unrelated senders with broad visibility, existing ACP handling, and thread-bound completion delivery.

## Reproduction Assessment

Yes. The issue provides concrete reproduction steps, and current source gives a high-confidence code-level path: a native child can keep a stored `deliveryContext`, `sessions_send` can return its reply inline, and the A2A flow can then announce through that target session context.

## Solution Assessment

No, current main is not the best solution yet. The existing ACP-only skip fixes one related slice, but the narrower maintainable fix should detect parent-owned native persistent sub-agent sends and choose a single delivery path rather than globally clearing delivery context.
