## Summary

Keep open. Current main still has the native parent-owned sub-agent duplication risk described by the report: the only skip guard is ACP-specific, waited `sessions_send` returns the child reply inline while also starting the A2A announce flow, and announce target resolution can use the child session's stored `deliveryContext`. The issue is narrow enough for a focused repair PR rather than a close or duplicate cleanup.

## What This Changes

The issue requests that persistent sub-agent replies sent through `sessions_send` be delivered to the parent conversation exactly once instead of 2-3 times via overlapping inline reply, A2A announce, and stored delivery-context paths.

## Best Possible Solution

Parent-owned native child sessions contacted through `sessions_send` should have one explicit parent-visible result path, while preserving stored delivery context for legitimate thread-bound completion routing and preserving normal peer A2A behavior for unrelated sessions.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current source shows a high-confidence code-level path: native child sessions can retain `deliveryContext`, the ACP-only skip does not apply, and waited `sessions_send` can both return inline and start A2A announce work.

## Solution Assessment

No, current main is not the best solution yet. The narrow maintainable fix should add a native parent-owned sub-agent/session-send guard or single-delivery-path decision rather than globally clearing `deliveryContext` or disabling peer A2A.
