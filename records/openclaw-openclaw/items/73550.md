## Summary

Keep open. Current main still has a high-confidence native parent-owned sub-agent duplication path: the existing single-delivery guard is ACP-specific, waited `sessions_send` returns the child reply inline and then starts the A2A announce flow, and the announce target can be hydrated from the child session's stored `deliveryContext`. Recent main hardening covers ACP child echo loops and inter-session provenance, but it does not cover the native persistent sub-agent case reported here.

## What This Changes

The issue requests that replies from persistent native sub-agent sessions sent via `sessions_send` be delivered once to the parent conversation, without announce-mode feedback loops from inherited delivery context.

## Best Possible Solution

Parent-owned native child sessions contacted through `sessions_send` should have one explicit parent-visible result path. The likely maintainable fix is a native parent-owned child/session-send guard or delivery-path decision that skips A2A announce when the parent already receives the inline or completion result, while preserving normal peer A2A and stored delivery context for valid thread-bound routing.

## Reproduction Assessment

Yes. The issue provides concrete steps, and current source shows the reproducing path: a native sub-agent can retain `deliveryContext`, the ACP-only skip does not apply, and waited `sessions_send` can return an inline reply while also starting A2A announce work.

## Solution Assessment

No, current main is not the best solution yet. The issue's workaround of clearing `deliveryContext` is too broad; the safer fix is a targeted native parent-owned sub-agent guard or single-delivery-path rule that keeps legitimate announce routing intact.
