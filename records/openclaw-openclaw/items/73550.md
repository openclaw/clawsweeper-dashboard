## Summary

Keep open. Current main still has a credible duplicate native sub-agent delivery path: native `sessions_spawn` ignores `streamTo`, native child sessions can retain a parent chat `deliveryContext`, and `sessions_send` only skips the follow-up A2A announce flow for parent-owned ACP sessions, not native parent-owned sub-agents.

## What This Changes

The issue reports duplicate parent-visible replies when a persistent native sub-agent is messaged via `sessions_send`, with the reporter pointing to stored child-session `deliveryContext` overlapping with the reply/announce path.

## Best Possible Solution

Add a generic parent-owned native sub-agent ownership guard so a parent sending to its own native persistent child has one parent-visible result path, while preserving peer A2A messaging, thread-bound follow-ups, ACP `streamTo` behavior, and channel announce delivery for actual detached work.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current source gives a high-confidence code-level reproduction path: native child sessions can keep parent delivery context, waited `sessions_send` returns the child reply inline, and the non-skipped A2A flow can also inject/re-announce the reply.

## Solution Assessment

No, current main is not the best solution yet. Clearing all native child `deliveryContext` would be too broad because native sub-agent announces need it; the safer fix is a narrow ownership/target-kind guard that selects a single delivery path only for parent-owned native child sends.
