## Summary

Keep open. Current `main` still has the reported duplicate-delivery shape for native persistent sub-agent sends: the existing skip is ACP-specific, waited sends return the child reply inline, and the normal A2A announce path can still hydrate the target session's stored delivery context and deliver another parent-visible message. The issue is recent, actionable, and not terminally covered by the broader open #53145 or the already-merged ACP-only #69817 slice.

## What This Changes

The issue requests that `sessions_send` to persistent native sub-agent sessions deliver each sub-agent reply to the parent conversation only once and avoid feedback through inherited announce delivery context.

## Best Possible Solution

Add a narrow native parent-owned sub-agent/session-send rule so a parent messaging its own persistent native child has exactly one parent-visible result path. The implementation should suppress duplicate same-run transcript/UI injection and avoid using the child's inherited `deliveryContext` for parent-owned sends, while preserving unrelated peer A2A and the existing ACP parent-owned skip semantics.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current source provides a high-confidence code-level reproduction path: a waited native `sessions_send` can return the target reply inline and still schedule A2A reply/announce work that resolves delivery through the target session's stored `deliveryContext`.

## Solution Assessment

No. Current `main` only skips A2A for parent-owned ACP sessions; the reported flow is a native persistent sub-agent session, so the narrower maintainable fix is to detect the parent-owned native child case and choose one delivery path without disabling peer A2A globally.
