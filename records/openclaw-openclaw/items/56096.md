## Summary

Keep open. Current main still has the reported Telegram sendChatAction transient-failure gap: only 401 errors drive the dedicated backoff/suspension state, while network, rate-limit, 5xx, and overload-style failures still rethrow into the typing keepalive path; the issue is also paired with open implementation candidates that should be reviewed or replaced before closure.

## What This Changes

The issue requests bounded retry/backoff and cleanup for Telegram sendChatAction typing loops after transient failures.

## Best Possible Solution

Land or replace the existing candidate work with a Telegram plugin fix that treats sendChatAction as best-effort, adds bounded transient cooldown/backoff for network, 429, 5xx, and overload-style failures, preserves the 401 auth breaker, and covers repeated keepalive behavior.

## Reproduction Assessment

Yes, at source level. Current main rethrows non-401 sendChatAction failures outside the handler breaker, and the current test suite asserts repeated 500 errors do not suspend the handler.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix belongs in the Telegram sendChatAction handler and focused typing/sendChatAction tests, without adding core policy or new configuration.
