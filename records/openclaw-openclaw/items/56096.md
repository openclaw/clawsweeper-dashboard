## Summary

Keep this open. Current main still only applies Telegram sendChatAction backoff/suspension to 401 Unauthorized errors, current tests explicitly preserve non-401 non-suspension, and open implementation candidates already cover the same transient failure path for maintainer review.

## What This Changes

The issue asks for Telegram sendChatAction typing failures to be bounded with retry/backoff or suspension so transient failures cannot keep the Telegram channel unresponsive indefinitely.

## Best Possible Solution

Keep the issue open and make one canonical Telegram-owned fix: treat transient sendChatAction failures as best-effort with bounded retries or cooldown, preserve 401 invalid-token suspension semantics, avoid rearming the shared typing failure guard while a keepalive is already active or tripped, and add regression tests for repeated non-401 failures through the Telegram handler and typing callback path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: a focused unit reproduction can make sendChatAction reject repeatedly with a non-401/network-style error and observe that current main never suspends it; the existing test already asserts that behavior for repeated 500 failures.

## Solution Assessment

Is this the best way to solve the issue? No for current main: the 401-only circuit breaker is too narrow for the reported transient failure mode. The narrower maintainable path is to review and land or supersede the existing Telegram cooldown PR work with coverage rather than close the issue now.
