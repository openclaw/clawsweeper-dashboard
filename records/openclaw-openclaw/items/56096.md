## Summary

Keep open. Current main still limits the dedicated Telegram sendChatAction breaker to 401 Unauthorized errors, so transient network, 429, 5xx, and overload-style failures can still re-enter the typing keepalive path; the provided timeline also shows open implementation candidates that should be reviewed or replaced before this is closed.

## What This Changes

The issue requests bounded retry, backoff, and cleanup for Telegram sendChatAction typing failures so Telegram does not wedge the channel during model/network overloads.

## Best Possible Solution

Land or replace the existing candidate work with a Telegram plugin fix that treats sendChatAction as best-effort, adds transient failure cooldown/backoff for network, 429, 5xx, and overload-style failures, preserves the 401 auth breaker, and covers repeated reply-start keepalive behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows non-401 sendChatAction failures still rethrow outside the handler breaker, and the current test suite explicitly asserts repeated 500 errors do not suspend the handler.

## Solution Assessment

Yes for the requested direction. The narrow maintainable fix belongs in the Telegram-owned sendChatAction handler and typing callback tests, without adding core policy or new configuration.
