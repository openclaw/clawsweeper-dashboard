## Summary

Keep open. Current main still parses custom-scheme Origin headers through `URL.origin`, which becomes the opaque string `null`, so explicit `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` cannot match; open PR #73511 is the active implementation candidate and still references this issue.

## What This Changes

Explicit custom-scheme Control UI origins such as `tauri://localhost` should match `gateway.controlUi.allowedOrigins` without requiring wildcard origins.

## Best Possible Solution

Land PR #73511 or an equivalent narrow gateway parser fix that preserves exact hosted custom-scheme origin matching while keeping literal `Origin: null`, malformed origins, host-header fallback, and local-loopback behavior unchanged.

## Reproduction Assessment

Yes. The source path is deterministic: Node returns `URL.origin === "null"` for `tauri://localhost` and `electron://app`, and current `checkBrowserOrigin` compares only that parsed origin against the exact allowlist.

## Solution Assessment

Yes. The narrowest maintainable solution is the parser-level fix with colocated `tauri://` and `electron://` regression tests, rather than broadening wildcard behavior or relying on Host-header fallback.
