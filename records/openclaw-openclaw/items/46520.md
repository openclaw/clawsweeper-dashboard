## Summary

Keep this open: current main and v2026.6.6 still normalize custom-scheme browser origins through `URL.origin`, so exact `gateway.controlUi.allowedOrigins` entries cannot match; https://github.com/openclaw/openclaw/pull/73511 is the open closing fix path.

## What This Changes

The issue requests exact `gateway.controlUi.allowedOrigins` matching for custom-scheme Control UI origins such as `tauri://` and `electron://` without falling back to wildcard origins.

## Best Possible Solution

Review and land one consolidated Gateway origin-normalization fix, likely via https://github.com/openclaw/openclaw/pull/73511, that preserves exact custom-scheme allowlist matching without broadening wildcard semantics.

## Reproduction Assessment

Yes, at source level. Current main compares configured entries against `URL.origin`, and a Node probe shows the reported custom schemes serialize that origin as `null` before the exact allowlist check.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/gateway/origin-check.ts` with regression coverage in `src/gateway/origin-check.test.ts`; the open closing PR follows that boundary, so a separate fix lane is unnecessary.
