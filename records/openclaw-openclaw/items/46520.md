## Summary

Keep open: current main and v2026.6.6 still compare exact allowlist entries against `URL.origin`, which is `null` for the reported custom schemes, and the linked fix PR has not merged.

## What This Changes

Specific `gateway.controlUi.allowedOrigins` entries should accept non-standard desktop-app schemes like `tauri://` and `electron://` instead of forcing wildcard origins.

## Best Possible Solution

Land or replace the parser-local fix that preserves exact `scheme://host` matching for hosted opaque origins while still rejecting literal `Origin: null` and unlisted custom origins.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.6 compare configured entries against `URL.origin`, and a Node probe shows the reported custom schemes serialize that origin as `null` before exact allowlist matching.

## Solution Assessment

Yes, the narrow fix belongs in `src/gateway/origin-check.ts` with focused allow/reject tests. Broader wildcard or Host-header fallback changes would unnecessarily expand a Gateway security boundary.
