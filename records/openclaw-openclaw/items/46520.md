- `impact:security`: The issue affects the Gateway Control UI browser-origin trust boundary and can push operators toward wildcard allowlisting.

## Summary

Keep open: current main and the latest release still collapse hosted custom-scheme Origins to `"null"` before exact allowlist matching, and an open closing PR already owns the fix path with an overlapping Gateway origin PR to consolidate.

## What This Changes

The issue reports that exact `gateway.controlUi.allowedOrigins` entries for custom schemes such as `tauri://localhost` and `electron://...` are rejected.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73511 or a consolidated equivalent with https://github.com/openclaw/openclaw/pull/38290, preserving exact allowlist semantics and leaving scheme-wildcard policy to https://github.com/openclaw/openclaw/issues/39739.

## Reproduction Assessment

Yes, at source level. Current `parseOrigin` stores `URL.origin`, and Node reports hosted custom schemes such as `tauri://localhost` and `electron://app` as opaque `origin:"null"`, so exact configured entries cannot match.

## Solution Assessment

Yes. A parser-level fallback that preserves hosted custom-scheme `scheme://host` request origins while keeping configured allowlist entries exact is the narrow maintainable fix; wildcard or prefix matching would be a separate policy decision.
