## Summary

Keep open. Current main still collapses hosted custom-scheme Origin values to `null` before the explicit allowlist comparison, and https://github.com/openclaw/openclaw/pull/73511 is an open fix PR that references this issue with closing syntax.

## What This Changes

The issue reports that `gateway.controlUi.allowedOrigins` rejects explicit custom-scheme origins such as `tauri://localhost` and `electron://...` unless operators use the wildcard allowlist.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73511 or a consolidated equivalent with https://github.com/openclaw/openclaw/pull/38290, preserving exact allowlist semantics and adding focused allow/reject coverage for hosted custom-scheme origins.

## Reproduction Assessment

Yes, at source level. Node reports hosted custom schemes with `URL.origin === "null"`, and current `checkBrowserOrigin` compares that parsed value against explicit `gateway.controlUi.allowedOrigins` entries.

## Solution Assessment

Yes. Preserving the exact `scheme://host` origin before the existing allowlist comparison is the narrow maintainable fix; wildcard, prefix, or Host-header fallback changes would broaden policy.
