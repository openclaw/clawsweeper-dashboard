## Summary

Keep this open: current `main` and the latest release still normalize custom-scheme browser origins to the opaque `null` value before exact allowlist comparison, and an open linked PR specifically closes this issue.

## What This Changes

The issue asks for exact `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` and `electron://...` to be accepted without falling back to wildcard origins.

## Best Possible Solution

Review and land one consolidated gateway origin-normalization fix that preserves exact configured allowlist semantics while accepting explicitly listed opaque custom-scheme origins.

## Reproduction Assessment

Yes. Source inspection plus a Node URL check show current main normalizes `tauri://localhost` and `electron://app` to `null`, so exact configured entries cannot match the allowlist branch.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/gateway/origin-check.ts` with regression coverage in `src/gateway/origin-check.test.ts`, and the open linked PR already targets that layer.
