## Summary

Keep open: current main and the latest release still normalize custom-scheme browser origins to `null`, while the issue is already paired with an open focused closing PR that should be reviewed or replaced before the issue closes.

## What This Changes

The issue reports that exact `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` and `electron://...` are rejected because custom URL schemes normalize to an opaque origin.

## Best Possible Solution

Review and land, or replace, the open focused PR that fixes `parseOrigin` for custom-scheme origins while preserving exact allowlist matching and literal `Origin: null` rejection.

## Reproduction Assessment

Yes, at source level. Current main passes `tauri://localhost` through `new URL(...).origin`, and the Node probe shows that value becomes `null`, so exact allowlist entries cannot match.

## Solution Assessment

Yes, the narrow parser-local fix proposed by the linked PR is the right repair shape. The issue should stay open until that PR is reviewed and merged or replaced.
