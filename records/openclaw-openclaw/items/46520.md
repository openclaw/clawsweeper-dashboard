## Summary

Keep open. Current main and the latest release still reduce hosted custom-scheme origins to `null` before exact allowlist matching, and https://github.com/openclaw/openclaw/pull/73511 is an open closing fix path while https://github.com/openclaw/openclaw/pull/38290 overlaps the same parser behavior.

## What This Changes

The issue reports that exact custom-scheme Control UI origins such as `tauri://localhost` and `electron://app` are rejected even when configured in `gateway.controlUi.allowedOrigins`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73511 or a consolidated equivalent with https://github.com/openclaw/openclaw/pull/38290, preserving exact allowlist semantics and adding focused allow/reject coverage for hosted custom-scheme origins.

## Reproduction Assessment

Yes, at source level. Node reports hosted custom schemes with `URL.origin === "null"`, and current `checkBrowserOrigin` compares only that parsed origin against explicit `gateway.controlUi.allowedOrigins` entries.

## Solution Assessment

Yes. Preserving the exact incoming `scheme://host` value for hosted custom schemes before the existing exact allowlist comparison is the narrow maintainable fix; wildcard, prefix, Host-header fallback, or parsing configured entries would broaden policy.
