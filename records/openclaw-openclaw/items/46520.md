## Summary

Keep open. Current main still rejects exact custom-scheme Control UI origins because the parser normalizes them to the opaque `null` origin, and https://github.com/openclaw/openclaw/pull/73511 is the active open implementation candidate with closing syntax for this issue.

## What This Changes

The issue asks for `gateway.controlUi.allowedOrigins` to accept exact custom URL scheme origins such as `tauri://localhost` and `electron://app` without requiring wildcard origin allowlisting.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73511 or an equivalent narrow parser-level fix with regression coverage for allowlisted and unlisted custom-scheme origins while preserving literal `Origin: null`, malformed-origin, host-header fallback, and local-loopback behavior.

## Reproduction Assessment

Yes, at source level. Node parses `tauri://localhost` and `electron://app` with `URL.origin === "null"`, and current `checkBrowserOrigin` compares that parsed value against explicit allowlist entries.

## Solution Assessment

Yes. Preserving hosted custom-scheme origins during parser normalization is the narrow maintainable fix because it feeds the existing explicit allowlist gate instead of broadening wildcard or Host-header fallback behavior.
