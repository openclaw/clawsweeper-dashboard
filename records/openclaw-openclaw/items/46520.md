## Summary

Keep open: current main and latest release v2026.6.8 still compare exact allowlist entries against `URL.origin`, which is `null` for the reported custom schemes, and two open linked PRs already claim the narrow fix path.

## What This Changes

The issue asks for exact `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` and `electron://localhost` to match browser `Origin` headers without falling back to `"*"`.

## Best Possible Solution

Land one narrow origin-check fix that preserves exact hosted custom-scheme origins while still rejecting literal `Origin: null`, malformed origins, unlisted custom origins, and unchanged host-header fallback behavior.

## Reproduction Assessment

Yes. Current source compares configured entries against `URL.origin`, and a Node probe shows the reported custom schemes serialize that origin as `null` before exact allowlist matching.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/gateway/origin-check.ts` with focused allow/reject coverage, while broader wildcard or Host-header fallback policy should stay out of this issue.
