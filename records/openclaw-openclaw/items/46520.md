## Summary

Keep open. Current main still parses custom-scheme Origin headers through `URL.origin`, which becomes the opaque string `null`, so explicit `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` cannot match. Open PR #73511 already references this issue with closing syntax and is the active implementation candidate; earlier PRs #46523 and #46555 were closed unmerged.

## What This Changes

Allow `gateway.controlUi.allowedOrigins` to accept explicit custom-scheme Control UI origins such as `tauri://localhost` and `electron://app`.

## Best Possible Solution

Land PR #73511 or an equivalent narrow gateway parser fix that preserves exact `scheme://host` matching for hosted non-special schemes, keeps literal `Origin: null` and malformed origins rejected, leaves host-header/local-loopback fallback behavior unchanged, and adds focused `tauri://` / `electron://` regression coverage.

## Reproduction Assessment

Yes. The source path is deterministic: Node returns `URL.origin === "null"` for `tauri://localhost` and `electron://app`, and current `checkBrowserOrigin` compares only that parsed origin against the exact allowlist.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/gateway/origin-check.ts` with colocated regression tests, rather than broadening wildcard behavior or relying on Host-header fallback.
