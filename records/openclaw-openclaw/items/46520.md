## Summary

Keep open. Current main still normalizes hosted custom-scheme Origin headers through `URL.origin`, which becomes `null`, and the open linked pull request is the active closing implementation candidate for this issue.

## What This Changes

The issue reports that exact `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` and `electron://app` fail while wildcard allowlisting works.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73511 or an equivalent narrow parser-level fix with regression tests for allowlisted and unlisted custom-scheme origins while preserving literal `Origin: null`, malformed-origin, host-header fallback, and local-loopback behavior.

## Reproduction Assessment

Yes. The source path is deterministic: Node parses `tauri://localhost` with `URL.origin === "null"`, and current `checkBrowserOrigin` compares that parsed origin against the exact allowlist entry.

## Solution Assessment

Yes. Preserving hosted custom-scheme origins during parser normalization is the narrowest maintainable fix because it feeds the existing explicit allowlist gate instead of broadening wildcard or Host-header fallback behavior.
