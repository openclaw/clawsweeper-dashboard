Current main still reproduces the custom-scheme allowlist failure, and an open PR explicitly closes this issue. Keep the issue open until maintainers select, verify, and merge one canonical fix; the stale label does not reflect the active implementation work.

## What This Changes

Accept explicitly allowlisted Tauri and Electron-style custom origins without requiring the wildcard origin setting.

## Best Possible Solution

Consolidate on one narrow PR that reconstructs only validated opaque `scheme://host` origins, preserves exact matching and all current rejection and fallback rules, and proves both allowlisted success and unlisted rejection through a real gateway WebSocket handshake.

## Reproduction Assessment

Yes, at source level with high confidence: current Node serializes these valid custom URLs to opaque `null`, and current main compares that value against the explicit custom-scheme allowlist entry. The issue also includes a real failed handshake log matching this path.

## Solution Assessment

Yes, the proposed narrow normalization direction is the right fix shape, provided it validates `scheme://host`, preserves literal-`null` and malformed-origin rejection, and adds real handshake proof; a new config option or wildcard exception would be worse.
