## Summary

Keep this open: current main and v2026.6.6 still route custom-scheme browser origins through `URL.origin`, so exact `gateway.controlUi.allowedOrigins` entries cannot match, and https://github.com/openclaw/openclaw/pull/73511 is the open closing fix path.

## What This Changes

The issue reports that exact `gateway.controlUi.allowedOrigins` entries using custom schemes like `tauri://localhost` and `electron://app` are rejected during Gateway Control UI WebSocket origin validation.

## Best Possible Solution

Land one consolidated current-main Gateway origin-normalization fix that preserves explicit exact matching for custom schemes without broadening configured allowlist semantics.

## Reproduction Assessment

Yes, at source level. Current main compares configured entries against `URL.origin`, and a Node probe shows the reported custom schemes serialize that origin as `null` before the exact allowlist check.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/gateway/origin-check.ts` with regression coverage in `src/gateway/origin-check.test.ts`, while preserving fail-closed handling for unsafe origins.
