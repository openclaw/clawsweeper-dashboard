## Summary

Current main still rejects explicitly allowlisted opaque custom-scheme origins, and an open PR specifically closes this issue. Keep the issue open until maintainers consolidate the overlapping origin fixes and land one security-reviewed implementation.

## What This Changes

Allow exact `gateway.controlUi.allowedOrigins` matching for custom desktop URL schemes such as `tauri://` and `electron://`.

## Best Possible Solution

Consolidate on one parser fix that preserves a validated exact `scheme://host` only when `URL.origin` is opaque, covers desktop and extension schemes, and proves both accepted and rejected real WebSocket handshakes.

## Reproduction Assessment

Yes. Current-main source deterministically serializes an opaque custom-scheme origin as `null` before exact allowlist comparison, and the issue includes a real rejected gateway handshake log.

## Solution Assessment

Yes in principle. A narrow opaque-origin normalization repair is the maintainable layer, but maintainers should consolidate the two overlapping open PRs and require strict rejection plus handshake proof.
