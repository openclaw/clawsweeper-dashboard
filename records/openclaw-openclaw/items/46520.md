## Label Justifications

- `P2`: This is a normal-priority Gateway origin allowlist bug with a limited but real affected desktop-wrapper surface.
- `impact:security`: The issue affects a Gateway browser-origin trust boundary and can push operators toward wildcard origin allowlisting.

## Summary

Keep open: current main still collapses hosted custom-scheme Origins to the opaque `null` value before exact allowlist matching, and an open closing PR already owns the fix path.

## What This Changes

Allow exact `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` and `electron://app` to pass Control UI browser-origin validation without using wildcard origins.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73511 or a consolidated equivalent with https://github.com/openclaw/openclaw/pull/38290, preserving exact allowlist semantics and adding focused allow/reject coverage for hosted custom-scheme origins.

## Reproduction Assessment

Yes, at source level. Node reports hosted custom schemes with `URL.origin === "null"`, and current `checkBrowserOrigin` compares only that parsed origin against explicit `gateway.controlUi.allowedOrigins` entries.

## Solution Assessment

Yes. Preserving the exact incoming `scheme://host` value for hosted custom schemes before the existing exact allowlist comparison is the narrow maintainable fix; wildcard, prefix, Host-header fallback, or parsed-config semantics would broaden policy.
