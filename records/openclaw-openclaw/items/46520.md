## Summary

Keep open. Current main still rejects exact custom-scheme Control UI origins because `new URL()` normalizes hosted non-special schemes to the opaque `null` origin, and an open linked PR already references this issue with closing syntax.

## What This Changes

The issue reports that Control UI `gateway.controlUi.allowedOrigins` rejects exact allowlist entries for custom desktop URL schemes such as `tauri://` and `electron://`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/73511 or an equivalent consolidated Gateway parser fix with focused regression coverage for allowlisted and unlisted hosted custom-scheme origins.

## Reproduction Assessment

Yes, at source level. Node reports `tauri://localhost` and `electron://app` with `URL.origin === "null"`, and current `checkBrowserOrigin` compares that parsed origin against explicit allowlist entries.

## Solution Assessment

Yes. Preserving hosted custom-scheme origins before the existing explicit allowlist comparison is the narrow maintainable fix; wildcard, prefix, or Host-header fallback changes would broaden policy.
