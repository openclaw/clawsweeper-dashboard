## Label Justifications

- `P2`: The bug affects a limited class of desktop and extension Control UI clients but forces affected operators toward the unsafe wildcard workaround.
- `impact:security`: The issue concerns the browser-origin allowlist that protects Gateway WebSocket access, and the documented workaround broadens that boundary.

## Summary

Current main still rejects exactly allowlisted custom-scheme origins because the URL parser produces the opaque origin string `null`; the issue should remain open while the linked implementation PR is reviewed and consolidated with the overlapping extension-origin PR.

## What This Changes

Allow exact gateway Control UI origin entries for custom desktop-app URL schemes without requiring the wildcard origin setting.

## Best Possible Solution

Land one canonical fix that reconstructs only a structurally valid `scheme://host` value for opaque custom schemes, retains exact allowlist semantics and existing rejection paths, and proves both unit-level matching and an authenticated WebSocket handshake before closing this issue and the redundant PR.

## Reproduction Assessment

Yes, at high confidence from current source and the runtime URL contract: a custom-scheme header becomes the opaque origin `null`, which cannot equal its explicit `allowedOrigins` entry. The submitted log also shows this exact `origin-mismatch` path in a real deployment.

## Solution Assessment

Yes, a narrow custom-origin normalization repair is the right boundary, but the best landing path should combine the focused current PR with the broader PR’s authenticated handshake coverage and explicit malformed-origin security cases.
