## Summary

Keep open: current main and v2026.5.20 still collapse hosted custom-scheme browser origins to the opaque `null` origin before exact allowlist matching, and an open closing pull request already owns the targeted fix path.

## What This Changes

Fix Gateway Control UI origin validation so explicit custom-scheme origins such as `tauri://localhost` and `electron://app` match `gateway.controlUi.allowedOrigins` without falling back to wildcard origins.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/73511 or a consolidated equivalent with https://github.com/openclaw/openclaw/pull/38290, preserving exact allowlist semantics and adding focused allow/reject coverage for hosted custom-scheme origins.

## Reproduction Assessment

Yes, at source level. Node reports hosted custom schemes with `URL.origin === "null"`, and current `checkBrowserOrigin` compares only that parsed origin against explicit `gateway.controlUi.allowedOrigins` entries.

## Solution Assessment

Yes. Preserving the exact incoming `scheme://host` value for hosted custom schemes before the existing exact allowlist comparison is the narrow maintainable fix; wildcard, prefix, or Host-header fallback changes would broaden policy.
