- `impact:message-loss`: The reported behavior ends the run with an error instead of delivering a response via a configured fallback model.

## Summary

Keep open: the report has concrete trajectory evidence, and current main/latest release still contain a harness-owned timeout decision that can surface a timeout before model fallback; the exact Ollama path needs maintainer review because current source also suggests normal Ollama runs should use the OpenClaw harness, not the plugin-harness timeout branch.

## What This Changes

The issue asks OpenClaw to attempt configured model fallbacks when an Ollama or other timeout occurs before a prompt completes instead of surfacing `LLM request timed out.` immediately.

## Best Possible Solution

Keep Codex/plugin timeout isolation, but make fallback-eligible local/provider prompt timeouts enter model fallback when no output or side effects were committed, with tests that distinguish OpenClaw-owned Ollama/OpenRouter runs from plugin-harness lifecycle timeouts.

## Reproduction Assessment

Yes at source level for harness-owned prompt timeouts: current main returns `surface_error` before `fallback_model`, and tests assert that behavior. It is still unclear whether the reporter's Ollama run should have selected that harness-owned path, so a live/config-backed Ollama repro remains the missing check.

## Solution Assessment

Unclear: allowing configured fallback on prompt timeout is likely the right outcome for local provider request timeouts, but changing the harness-owned branch broadly would conflict with the Codex timeout boundary from PR 86476. The best fix should split those cases instead of applying the one-line change everywhere.
