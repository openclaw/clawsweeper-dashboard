## Summary

Keep open. Current main still lacks the requested declarative registry for plugin-injected context blocks: plugin manifests do not declare injected tags, prompt hook results remain raw string fields, and downstream sanitizers still carry local hardcoded knowledge. This is a public plugin manifest/SDK plus privacy-sensitive sanitizer contract, so it needs maintainer API review rather than autoclose, ClawHub routing, or a narrow automated fix PR.

## What This Changes

The issue requests an additive plugin manifest/SDK contract and aggregated registry so plugin-injected context tags or blocks can be declared once and stripped by display, compaction, session export, and memory/dreaming ingestion paths.

## Best Possible Solution

Keep this open for maintainer API/security review. The likely end state is an additive, documented manifest/SDK contract for plugin-owned injected context blocks, a normalized registry getter built from plugin metadata before runtime loads, and staged consumer adoption with tests for existing active-memory and memory-lancedb tags, user-authored lookalikes, and fenced code preservation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a design RFC, but current-main verification is high confidence: source/docs search shows no registry and file inspection shows hardcoded sanitizer lists. A concrete end-to-end feedback-loop fixture is still missing.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer API review, but a manifest-backed registry is the narrowest maintainable direction visible from current code because manifests already carry static plugin contracts before runtime execution. The design should be reconciled with open context-provenance work in #54373 rather than rushed as an isolated patch.
