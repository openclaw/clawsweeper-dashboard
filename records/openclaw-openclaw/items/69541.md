## Summary

Keep open. Current main still lacks a declarative injected-context tag registry: producer plugins inject XML-like blocks through raw hook strings, while downstream display and memory sanitizers keep local hardcoded strip rules. This is a public plugin manifest/SDK plus privacy-sensitive sanitizer contract and should get maintainer API review before implementation.

## What This Changes

Request an additive plugin manifest/SDK registry for producer-declared injected context tags so downstream display, compaction, dreaming, and memory consumers can strip them consistently.

## Best Possible Solution

Keep this open for maintainer API/security review, then implement an additive manifest/SDK contract with a normalized registry getter and staged consumer adoption across display, session/dreaming ingestion, compaction, and memory-retain paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as an end-to-end bug fixture, because this is a design RFC; current-main source verification is high confidence that no registry exists and consumers/producers still carry hardcoded tag knowledge.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer API review, but a manifest-backed registry is the narrowest maintainable direction visible from current code because manifests are already the metadata-first plugin contract surface. It should be reconciled with #54373 rather than implemented as an isolated sanitizer patch.
