## Summary

Keep open. Current main still lacks the requested manifest/SDK registry and shared strip/sanitize/ingest contract, while the issue discussion now includes concrete external adopter evidence from Crystal/MemPalace showing plugins are hand-rolling this boundary.

## What This Changes

Add a declarative plugin manifest/runtime registry for plugin-injected context tags so downstream consumers can strip, sanitize, and ingest them from one shared contract.

## Best Possible Solution

Keep this open for maintainer API/security review, then implement a staged manifest-backed registry and migrate core sanitizers plus downstream consumers to that contract.

## Reproduction Assessment

Yes for the contract gap: current main lacks the requested registry symbols and still relies on hardcoded producer/consumer strings in the inspected paths. No single end-to-end fixture covers every leakage mode.

## Solution Assessment

Unclear pending maintainer review. A manifest-backed registry fits OpenClaw's manifest-first plugin architecture, but strip, sanitize, ingest, runtime registration, duplicate-tag handling, and provenance boundaries need product and security decisions before implementation.
