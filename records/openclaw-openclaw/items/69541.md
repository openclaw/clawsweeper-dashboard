## Summary

Keep open. Current main still lacks the requested manifest/SDK registry, and the discussion now has concrete adopter evidence from Crystal/MemPalace showing external plugins are hand-rolling this strip/sanitize/ingest contract.

## What This Changes

The issue requests a manifest-backed registry for plugin-injected context tags so downstream consumers can strip, sanitize, and ingest those blocks consistently.

## Best Possible Solution

Keep this open for maintainer API/security review, then implement a staged manifest-backed registry and migrate core sanitizers plus downstream consumers to that contract.

## Reproduction Assessment

Yes for the contract gap: current main has no registry symbols and still relies on hardcoded producer/consumer strings in the inspected paths. No end-to-end leakage fixture was established for every downstream consumer.

## Solution Assessment

Unclear pending maintainer review. A manifest-backed registry fits OpenClaw's manifest-first plugin architecture, but strip, sanitize, ingest, runtime registration, and provenance boundaries need product and security decisions before implementation.
