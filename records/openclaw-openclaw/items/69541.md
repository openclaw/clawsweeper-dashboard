## Summary

Keep open. Current main still lacks a declarative injected-context tag registry: plugin prompt producers return raw tagged strings, while display, recall, and memory-ingestion consumers still maintain local hardcoded strip behavior; the Crystal/MemPalace adopter comment is fresh positive evidence that third-party plugins are hitting the same contract gap.

## What This Changes

The issue requests a declarative plugin manifest/SDK registry for prompt-injected context tags so downstream consumers can strip, sanitize, and ingest plugin context consistently.

## Best Possible Solution

Keep this open for maintainer API/security review, then implement an additive manifest/SDK registry with a normalized getter and staged consumer adoption across display, replay, compaction, memory ingestion, and docs; coordinate with https://github.com/openclaw/openclaw/issues/54373 rather than treating this as a one-off sanitizer patch.

## Reproduction Assessment

No high-confidence end-to-end bug fixture was run because this is an RFC, but source inspection gives high-confidence proof of the current contract gap: manifests and hook results still have no registry metadata while consumers maintain hardcoded strip rules.

## Solution Assessment

Unclear until maintainer API/security review, but a manifest-backed registry is the most maintainable direction visible from current code because manifests are already the static plugin metadata contract surface. The final design should align strip/sanitize/ingest semantics with context provenance, replay hygiene, and third-party plugin compatibility.
