## Summary

Keep open. Current main still lacks a declarative injected-context tag registry: plugin prompt producers return raw tagged strings, while display and memory consumers still rely on local hardcoded strip rules, and the recent Crystal/MemPalace adopter comment is positive evidence that third-party plugins are hitting the same contract gap.

## What This Changes

This issue requests a plugin manifest/SDK registry for prompt-injected context tags so downstream display, compaction, memory, and ingestion paths can strip, sanitize, and ingest those blocks consistently.

## Best Possible Solution

Keep this open for maintainer API/security review, then implement an additive manifest/SDK contract with a normalized registry getter and staged consumer adoption, coordinated with https://github.com/openclaw/openclaw/issues/54373 rather than as a one-off sanitizer patch.

## Reproduction Assessment

No high-confidence end-to-end bug fixture is established here because this is an RFC, but source inspection gives high-confidence proof of the current contract gap: no registry symbols exist, hook outputs are plain strings, and consumers hardcode tag lists.

## Solution Assessment

Unclear until maintainer API review, but a manifest-backed registry is the most maintainable direction visible from current code because manifests are already the static plugin metadata contract surface. The final design should align strip/sanitize/ingest semantics with the broader context-provenance discussion.
