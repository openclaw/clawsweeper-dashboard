## Summary

Keep open for normal maintainer merge handling: the PR addresses a valid current-main CLI task liveness bug, includes the matching regression/docs/changelog updates, and I found no blocking review findings.

## What This Changes

The PR makes all CLI task liveness depend on the active embedded run context, updates the task audit expectation, adds an exec-approval-followup regression test, and documents the new CLI backing-state rule.

## Best Possible Solution

Land this PR so CLI task reconciliation uses live run ownership consistently, while leaving the separate bundle-mcp disposal race tracked by #76233 for a distinct lifecycle fix.

## Reproduction Assessment

Yes. Current main can reproduce with a grace-expired CLI task whose `sourceId`/`runId` has no live run context while `childSessionKey` points at a persistent `agent:main:main` session row; that session row keeps the task `running`.

## Solution Assessment

Yes. Returning `hasActiveCliRun(task)` for all CLI tasks is the narrow maintainable fix because the task-specific run context, not the reusable session row, is the authoritative liveness signal for the affected gateway and media CLI task paths.
