## Summary

Keep open: current main intentionally treats cron `payload.model` as a per-job primary model, while current OpenAI routing docs say `openai/*` is now the canonical Codex agent route rather than an API-key-only namespace. The remaining useful request is a new diagnostic or migration policy for cron model overrides that may not match the user's intended auth route; the linked harness-registration failure is related but already tracked separately at https://github.com/openclaw/openclaw/issues/81326.

## What This Changes

Issue requests migration, doctor/status detection, or UI warning for cron jobs whose `payload.model` pins can keep scheduled runs on a different OpenAI model/auth route than the default Codex OAuth session.

## Best Possible Solution

Add an owner-approved diagnostic, likely in `openclaw doctor` plus cron list/status surfaces, that reports cron model overrides and their effective runtime/auth route without weakening intentional per-job model selection.

## Reproduction Assessment

Partially: current source gives a high-confidence path for `payload.model` overriding the default cron model selection. It does not prove the reported API-key billing route on current main, especially because current docs define `openai/*` as the Codex agent route by default.

## Solution Assessment

Unclear: removing or rewriting stored cron `payload.model` pins could break intentional per-job model choices. A doctor/status warning looks safer, but the repo needs a maintainer decision on the desired migration and UX policy.
