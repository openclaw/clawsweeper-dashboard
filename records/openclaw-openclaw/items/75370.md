## Summary

Keep open. The PR targets a valid document-extract bug and belongs in the owner plugin, but it is not merge-ready because the pdftoppm filename handling fails for multi-digit page counts, the availability probe is unbounded, and the user-facing behavior change lacks docs/changelog coverage.

## What This Changes

The PR adds a pdftoppm-based PDF page image rendering fallback in the document-extract plugin when @napi-rs/canvas cannot be loaded.

## Best Possible Solution

Keep the fallback in the document-extract plugin, but make it a bounded and tested single-page Poppler render path, then update the PDF docs and changelog before merge.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction for the original bug: low-text PDFs that need image rendering return empty images when @napi-rs/canvas cannot load. The PR regression is also reproducible from Poppler's source contract: a 10+ page PDF page 1 writes a padded output filename that this patch does not read.

## Solution Assessment

No. A pdftoppm fallback is a plausible owner-module fix, but this implementation needs filename handling, process-boundary hardening, tests, docs, and changelog updates before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Read the Poppler filename that pdftoppm writes:** `extensions/document-extract/document-extractor.ts:191`
  - body: `pdftoppm` pads output page numbers to the digit width of the whole PDF, so page 1 of a 10+ page PDF is written like `root-01.png`. This code always reads `root-1.png`, so the fallback returns null and the image loop breaks for common multi-page PDFs. Use `-singlefile` for the one-page render or read the padded output name.
  - confidence: 0.94
- **[P2] Bound the pdftoppm availability probe:** `extensions/document-extract/document-extractor.ts:151`
  - body: The availability check runs `execSync("pdftoppm -v")` through a shell and without a timeout, so a stuck wrapper or binary on PATH can hang PDF extraction before the 30-second render timeout applies. Probe with `execFileSync("pdftoppm", ["-v"], { timeout: ... })` or fold availability into the timed render call.
  - confidence: 0.82
- **[P3] Document the new Poppler fallback:** `extensions/document-extract/document-extractor.ts:324-325`
  - body: This changes the public PDF fallback dependency story, but the PDF docs still say image rendering fallback depends on `@napi-rs/canvas`. Update the PDF docs so users know when `pdftoppm` is used and what system package they need on headless Linux.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `extensions/document-extract/document-extractor.ts:324-325`
  - body: This is a user-facing feature/fix for PDF extraction, but the branch only changes the extractor file. Repository policy requires a single-line `CHANGELOG.md` entry for user-facing `feat` and `fix` changes.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds external Poppler execution for user-supplied PDFs and leaves the availability probe unbounded.

Concerns:

- **[medium] Unbounded shell probe before timed renderer:** `extensions/document-extract/document-extractor.ts:151`
  - body: `execSync("pdftoppm -v")` runs before the timed `execFileSync` render path, uses a shell, and has no timeout. A stuck or wrapped `pdftoppm` binary can hang the PDF extraction request before the intended 30-second bound applies.
