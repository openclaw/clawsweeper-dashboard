## Summary

Keep open. Current main still has the canvas-missing PDF image path, and this draft PR is a plausible owner-module fix, but it needs maintainer review before merge because it adds external Poppler execution for untrusted PDFs and still lacks release-note coverage.

## What This Changes

The PR modifies the bundled `document-extract` PDF extractor to call `pdftoppm` when `@napi-rs/canvas` cannot load, adding temp-file rendering helpers and refactoring text/image extraction.

## Best Possible Solution

Land a bounded `pdftoppm` fallback in the `document-extract` plugin only after maintainer review confirms the external-renderer boundary, with regression tests and an Unreleased changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a low/no-text PDF in extraction fallback mode reaches image rendering, a failed `@napi-rs/canvas` import is caught, and the extractor returns text with `images: []`; the linked issue supplies the affected Linux setup.

## Solution Assessment

Unclear as-is. The plugin is the right owner boundary and Poppler is a plausible fallback, but this draft still needs changelog/test coverage and a small process-boundary cleanup before it is the best maintainable fix.
