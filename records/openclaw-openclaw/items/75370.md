## Summary

Keep open and do not merge as-is. Current main still lacks a Poppler/pdftoppm PDF image-rendering fallback, but this PR introduces correctness, security, lint, docs, and changelog blockers that need revision and maintainer/security review before the linked bug can be closed.

## What This Changes

The PR modifies the bundled `document-extract` PDF extractor to add a `pdftoppm` fallback when `@napi-rs/canvas` cannot load, with temp-file rendering helpers and refactored text/image extraction loops.

## Best Possible Solution

If maintainers accept Poppler as an optional renderer, keep the fix in the `document-extract` plugin but revise it to preserve short extracted text, use bounded argv-based async process execution, enforce actual pixel/output limits, surface renderer failures appropriately, and add focused tests plus PDF docs and changelog coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: a low/no-text PDF in extraction fallback mode reaches image rendering, a failed `@napi-rs/canvas` import is caught, and the extractor returns text with `images: []`; the linked report provides a concrete Ubuntu setup with Poppler installed.

## Solution Assessment

No. The owner-module direction is plausible, but this PR is not the best implementation until it fixes the text-preservation regression, resource-limit bypass, synchronous shell execution, lint failure, and user-facing docs/changelog gaps.

Full review comments:

- **[P2] Preserve short extracted PDF text:** `extensions/document-extract/document-extractor.ts:241`
  - body: `extractTextFromPages()` now returns `""` whenever extracted text is below `minTextChars`, so low-text PDFs lose text that current main preserves while still deciding whether page images are needed. Keep the extracted text in the final result and use the threshold only for image-rendering decisions.
  - confidence: 0.94
- **[P2] Remove the 72 DPI floor from Poppler rendering:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` chooses a scale below 1 to satisfy `maxPixels`, this `Math.max(72, ...)` still renders through Poppler at 72 DPI. Large or low-budget pages can exceed the request pixel budget while the loop subtracts only the smaller planned count.

- **[P2] Avoid blocking shell execution for Poppler:** `extensions/document-extract/document-extractor.ts:187-189`
  - body: This runs `pdftoppm` through `execSync()` with a command string for each page. A malformed or expensive PDF can block the Gateway event loop for up to the timeout per page; use bounded argv-based `execFile` or `spawn` instead.
  - confidence: 0.9
- **[P2] Fix the Node builtin lint failure:** `extensions/document-extract/document-extractor.ts:162`
  - body: The new fallback `require("fs")` branch is flagged by the PR head's lint checks for missing the `node:` protocol, which currently leaves `check-lint` and bundled extension checks failing. Remove the fallback or keep the builtin reference compliant.
  - confidence: 0.88

  - body: The fallback only works when `pdftoppm` is installed and on `PATH`, but the PR does not update the PDF docs that currently describe the canvas-only image-rendering path. Add user-facing install and behavior notes for this optional renderer.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces external rendering of untrusted PDFs and needs safer process execution plus hard resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:187`
  - body: The patch invokes `pdftoppm` through `execSync()` with a command string. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of generated temp paths.
  - confidence: 0.9
- **[medium] Pixel budget bypass amplifies PDF rendering work:** `extensions/document-extract/document-extractor.ts:175`
  - body: The Poppler fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger memory or prompt payloads for attacker-supplied PDFs.
  - confidence: 0.92
