## Summary

Keep open. The linked bug is valid on current main, but this PR is not merge-ready because the refactor changes the existing trimmed-text threshold semantics and still lacks the required changelog/test coverage for the new user-visible fallback.

## What This Changes

The PR adds a pdftoppm-based PDF page rendering fallback in `extensions/document-extract/document-extractor.ts` when `@napi-rs/canvas` cannot be loaded.

## Best Possible Solution

Land a tested document-extract fallback that uses pdftoppm when canvas is unavailable while preserving current text-threshold behavior and recording the user-visible fix in the changelog.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: low-text PDFs attempt canvas rendering, and a canvas import failure returns `images: []`, matching the linked headless Linux report.

## Solution Assessment

No, not as-is. The fallback belongs in `document-extract`, but the PR should preserve the existing trimmed-text threshold, add focused fallback tests, and add the required changelog entry before merge.
