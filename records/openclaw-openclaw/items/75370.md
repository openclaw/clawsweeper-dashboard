## Summary

Keep open. The PR targets a real linked #75358 document-extract bug, but the current branch is not merge-ready because it changes the text-threshold semantics and lacks the required tests, docs, and changelog coverage for the new fallback.

## What This Changes

The PR adds a `pdftoppm` PDF page-rendering fallback in `extensions/document-extract/document-extractor.ts` when `@napi-rs/canvas` cannot be loaded.

## Best Possible Solution

Land a focused document-extract fallback that preserves existing text-threshold behavior, covers the canvas-missing/pdftoppm path, and updates user-facing docs and changelog before merge.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: a low-text PDF with a missing `@napi-rs/canvas` import reaches the catch block and returns no images, and #75358 supplies the affected Ubuntu setup.

## Solution Assessment

No, not as-is. A `pdftoppm` fallback belongs in the document-extract plugin and the fixed-argv execution shape is reasonable, but the branch should preserve trimmed-threshold semantics and add docs, changelog, and focused tests first.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve the trimmed text threshold:** `extensions/document-extract/document-extractor.ts:235`
  - body: Current main skips image rendering only when `text.trim().length >= request.minTextChars`, but this branch computes `meetsThreshold` from the untrimmed accumulated length while returning `text.trimEnd()`. Whitespace-heavy text layers can now satisfy the threshold, skip both renderers, and return effectively empty or low-value text with no images. Base the threshold on the joined trimmed text to preserve existing fallback behavior.

Add focused coverage for canvas import failure plus available/unavailable `pdftoppm` so output naming, fallback selection, cleanup, and pixel-budget behavior can regress visibly.
  - confidence: 0.82
