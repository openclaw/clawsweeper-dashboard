## Summary

Keep open. The linked bug is real and current main still has no Poppler/pdftoppm fallback, but this PR is not merge-ready: the current diff still drops sub-threshold extracted text, bypasses the render pixel budget with a 72-DPI floor, uses synchronous shell execution for untrusted PDF rendering, and lacks the required docs/changelog follow-through.

## What This Changes

The PR refactors the bundled `document-extract` PDF extractor and adds a `pdftoppm`/Poppler image-rendering fallback when `@napi-rs/canvas` cannot load.

## Best Possible Solution

If maintainers accept Poppler as an optional renderer, keep the change inside the `document-extract` plugin and revise it to preserve extracted text, use bounded argv-based async process execution, enforce actual output and pixel limits, and update focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: a low-text PDF reaches image rendering, `loadCanvasModule()` throws when `@napi-rs/canvas` is absent, and the extractor returns no images; the linked #75358 report adds a concrete Ubuntu 24.04 setup with `pdftoppm` present.

## Solution Assessment

No. The owner-module direction may be reasonable, but this PR is not the best implementation until it fixes the text-preservation regression, process execution model, resource bounds, and user-facing docs/changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve low-volume extracted text:** `extensions/document-extract/document-extractor.ts:241`
  - body: `extractTextFromPages()` now returns `""` whenever extracted text is below `minTextChars`, so PDFs with a little text lose that text before image rendering and also return empty text when no renderer is available. Keep the extracted text in the final result and use the threshold only to decide whether images are needed.
  - confidence: 0.94
- **[P2] Honor the render pixel budget:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` chooses `scale < 1`, this `Math.max(72, ...)` still renders through Poppler at full 72 DPI. Large or low-budget pages can produce more pixels than `request.maxPixels` while the loop subtracts only the smaller planned pixel count.
  - confidence: 0.93
- **[P2] Avoid blocking shell execution for Poppler:** `extensions/document-extract/document-extractor.ts:187-189`
  - body: This invokes `pdftoppm` through `execSync()` for every rendered page. A malformed or expensive PDF can block the Gateway event loop for up to 30 seconds per page, and argv-based `execFile` or `spawn` would avoid shell parsing of generated paths.
  - confidence: 0.9
- **[P3] Document the optional Poppler dependency:** `extensions/document-extract/document-extractor.ts:307-308`
  - body: The fallback only works when `pdftoppm` is installed and on `PATH`, but the PR does not update the PDF docs that currently describe the canvas-only image-rendering path. Add user-facing install and behavior notes for the optional renderer.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `extensions/document-extract/document-extractor.ts:307-308`
  - body: This is a user-facing PDF extraction behavior change, but the PR only changes plugin source. Add an entry under the active `CHANGELOG.md` section and credit the contributor per repo policy.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces external rendering of untrusted PDFs and still needs safer process execution plus hard resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:187`
  - body: The patch invokes `pdftoppm` through `execSync()` with a command string. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of generated temp paths.
  - confidence: 0.9
- **[medium] Pixel budget bypass amplifies PDF rendering work:** `extensions/document-extract/document-extractor.ts:175`
  - body: The Poppler fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger memory or prompt payloads for attacker-supplied PDFs.
  - confidence: 0.92
