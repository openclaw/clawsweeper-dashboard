## Summary

Keep open. The PR addresses a valid current-main gap from the linked bug, but it is not merge-ready because the new pdftoppm path reads an unpadded output filename, while Poppler pads page suffixes to the document page-count width; common 10+ page PDFs can still produce no images for early pages.

## What This Changes

The PR adds a pdftoppm/poppler-utils fallback renderer in the bundled document-extract plugin when @napi-rs/canvas cannot be loaded, with temp-file rendering and PDF image-loop refactoring.

## Best Possible Solution

Keep the contributor direction, but fix pdftoppm output discovery, add regression coverage for canvas-missing fallback on a 10+ page PDF, update PDF docs and changelog, then review the draft PR normally.

## Reproduction Assessment

Yes. Current main reproduces the linked bug by loading a low-text PDF with @napi-rs/canvas unavailable, and the PR-specific regression can be reproduced with pdftoppm on a PDF whose total page count requires padded output suffixes.

## Solution Assessment

No, not as-is. The fallback approach is narrow and maintainable, but it should use `-singlefile` for per-page renders or otherwise read Poppler's padded output name, then document and test the new host-tool fallback.
