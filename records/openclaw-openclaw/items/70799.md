## Summary

Keep this PR open, but it needs changes before merge. The reported BlueBubbles failure is credible, current main still has the best-effort read/typing calls on the affected path, and the PR’s added `=== false` outer guards do not address the `null` cached-status path described as the root failure; related open PR #69932 overlaps part of the typing-call-site fix but does not make this PR implemented on main.

## What This Changes

The PR adds early cached Private API status guards to the BlueBubbles mark-read and typing helper functions in `extensions/bluebubbles/src/chat.ts`.

## Best Possible Solution

Make BlueBubbles read receipts and typing indicators best-effort against the resolved account capability: enabled servers keep the current behavior, disabled or still-unknown status must not call Private-API-only endpoints or surface a user-facing fallback, and focused tests should cover false and null/fresh-boot paths while coordinating the overlap with #69932.

## Reproduction Assessment

Yes. The PR gives live reproduction steps with a BlueBubbles server reporting `private_api: false`, and the source-level reproduction is to have `getCachedBlueBubblesPrivateApiStatus` return `null` for read or typing; current main and this PR still call the endpoint in that state.

## Solution Assessment

No. The proposed outer guard is not the best fix because it handles only known `false` status while preserving the `null` fallthrough named in the PR as the production failure; the safer repair is to skip or refresh capability before these best-effort calls.

## Repair Work Prompt

Fix the BlueBubbles Private-API-disabled/unknown fallback around read receipts and typing indicators, using PR #70799 as source context and coordinating overlap with PR #69932. The observable bug is that `markBlueBubblesChatRead` and `sendBlueBubblesTyping` or their inbound call sites can call Private-API-only `/api/v1/chat/:id/read` and `/typing` endpoints when `getCachedBlueBubblesPrivateApiStatus(accountId)` is `null` or `false`, causing BlueBubbles HTTP 500 errors and user-visible fallback warnings. Keep the fix inside the BlueBubbles plugin; do not change core channel/runtime APIs. For these best-effort actions, either skip unless Private API is known enabled or perform a bounded capability refresh before calling and skip when refresh remains unknown/false. Preserve behavior for known-enabled Private API servers and avoid changing outbound send/reply-threading semantics. Add focused tests for helper-level false and null paths plus inbound typing/read call paths. If incorporating both source PRs, add a user-facing CHANGELOG entry crediting the non-bot contributors, for example `Thanks @boatworkco` and `Thanks @Kerr1st`, following repo changelog policy.
