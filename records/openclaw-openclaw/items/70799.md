## Summary

Keep open. The PR targets a real BlueBubbles Private API-off failure, but the proposed false-only outer guard does not address the stated fresh-boot/null cache path and overlaps the still-open typing-call-site work in #69932.

## What This Changes

The PR adds early cached Private API disabled checks and explanatory comments to the BlueBubbles mark-read and typing helper functions.

## Best Possible Solution

Resolve the effective BlueBubbles account and Private API capability once for inbound best-effort actions, skip or refresh before read receipts and typing indicators when capability is false or unknown, and add focused coverage for false, null/fresh-boot, and multi-account/default-account behavior while coordinating with #69932.

## Reproduction Assessment

Yes. The PR gives concrete live steps against a BlueBubbles server with Private API off, and the source-level reproduction is to make getCachedBlueBubblesPrivateApiStatus return null for mark-read or typing; this patch's false-only guard still lets the request proceed.

## Solution Assessment

No. The proposed outer guard duplicates the existing false-only guard and preserves the null fallthrough described as the production failure; a safer fix should resolve or refresh capability for the effective account before these best-effort calls.

## Review Findings
