## Summary

Keep this PR open for revision. The reported BlueBubbles Private API-off failure is credible, but the patch does not fix the PR's own null-cache failure mode because the new outer guards still return only when the cached status is exactly false; an unknown status still reaches the Private-API-only endpoints. The work also overlaps open PR #69932, which is trying to address the same typing failure from the inbound call sites.

## What This Changes

The PR adds early cached Private API status guards to the BlueBubbles mark-read and typing helper functions in `extensions/bluebubbles/src/chat.ts`.

## Best Possible Solution

Make BlueBubbles read receipts and typing indicators best-effort based on the resolved account capability: skip or refresh before calling when Private API status is unknown or disabled, and add focused coverage for false, null/fresh-boot, and inbound typing/mark-read call paths while coordinating the overlap with #69932.

## Reproduction Assessment

Yes. The PR gives live steps using a BlueBubbles server with `private_api: false`, and the source-level reproduction is to make `getCachedBlueBubblesPrivateApiStatus` return null for a mark-read or typing call; this patch still calls the endpoint in that state.

## Solution Assessment

No. The proposed outer guard is not the best fix because it handles only known false status while preserving the null fallthrough described as the root failure; a safer fix should gate unknown status for these best-effort actions or refresh capability before sending.
