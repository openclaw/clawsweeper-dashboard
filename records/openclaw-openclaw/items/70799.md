## Summary

Keep this PR open for revision. The BlueBubbles Private API-off failure is credible, but the patch misses the PR's own null-cache failure mode: unknown status still reaches the Private-API-only read and typing endpoints, and the work overlaps open PR #69932 on the same inbound typing surface.

## What This Changes

The PR adds early cached Private API status guards to the BlueBubbles mark-read and typing helper functions in `extensions/bluebubbles/src/chat.ts`.

## Best Possible Solution

Make BlueBubbles read receipts and typing indicators best-effort based on the resolved account capability: handle disabled and unknown capability before hitting Private-API-only endpoints, add focused false/null/inbound coverage, and reconcile the overlap with #69932 in one reviewed fix.

## Reproduction Assessment

Yes. The PR gives live steps using a BlueBubbles server with `private_api: false`, and the source-level reproduction is to make `getCachedBlueBubblesPrivateApiStatus` return null for a mark-read or typing call; this patch still calls the endpoint in that state.

## Solution Assessment

No. The proposed outer guard is not the best fix because it handles only known false status while preserving the null fallthrough described as the root failure; a safer fix should gate unknown status for these best-effort actions or refresh capability first.
