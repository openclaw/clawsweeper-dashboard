## Summary

Keep open for merge review. Current main still drops the short explicit tagged TTS case, and this PR is a narrow, correct runtime/test fix; the only actionable gap I found is the missing required changelog entry for a user-facing fix.

## What This Changes

The PR changes speech-core TTS payload handling so non-empty explicit `[[tts:text]]...[[/tts:text]]` content can bypass untagged short-text suppression, adds an empty post-markdown guard, and adds regression tests for tagged short, untagged short, and markdown-only tagged text.

## Best Possible Solution

Add the missing single-line changelog fix entry, keep the focused runtime/test patch, and land this PR or an equivalent narrow change so linked #73758 can close after merge; keep #74955 separate because it covers message-tool send-path parity rather than the short-text guard.

## Reproduction Assessment

Yes. On current main, `[[tts:text]]hello[[/tts:text]]` is captured by the directive parser, but `maybeApplyTtsToPayload` returns at the unconditional under-10-character guard before `textToSpeech`; the PR adds direct regression coverage for that path.

## Solution Assessment

Yes for the code path. Distinguishing explicit tagged TTS text from untagged auto-TTS at the existing short-text guards is the narrowest maintainable fix, and the post-markdown empty guard handles the reviewed edge case; the remaining fix is a changelog entry, not a runtime redesign.
