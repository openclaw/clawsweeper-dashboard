## Summary

Keep this PR open: current main still skips short explicit tagged TTS, while the PR implements the right narrow runtime/test fix but still needs docs and changelog cleanup before merge.

## What This Changes

The PR changes speech-core TTS handling so explicit `[[tts:text]]...[[/tts:text]]` content can synthesize under the short-text threshold, adds an empty post-markdown guard, and adds regression tests.

## Best Possible Solution

Land the focused runtime/test fix after adding a release-note bullet and aligning the TTS docs so explicit non-empty tagged speech is documented as an exception to the untagged short-text skip.

## Reproduction Assessment

Yes. On current main, `[[tts:text]]hello[[/tts:text]]` reaches `directives.ttsText` but then hits the unconditional under-10-character guard in `maybeApplyTtsToPayload` before synthesis.

## Solution Assessment

Yes, with small docs and changelog cleanup. Separating explicit tagged speech from untagged auto-TTS is the narrowest maintainable fix, and the added empty-string guard covers the markdown-only edge case raised in review.
