## Summary

Keep open. The PR fixes a real current-main TTS bug and its exact head has green CI, but the user-facing behavior change still needs the required changelog entry and docs clarification before merge.

## What This Changes

This PR changes speech-core TTS payload handling so explicit `[[tts:text]]...[[/tts:text]]` speech can bypass untagged short-text skips, adds a post-markdown empty guard, and adds regression coverage.

## Best Possible Solution

Land this PR after a docs/changelog-only follow-up, preserving the explicit-tag bypass and empty post-markdown guard while leaving #74955 as the separate message-tool send-path parity review.

## Reproduction Assessment

Yes. On current main, `[[tts:text]]hello[[/tts:text]]` is parsed as tagged TTS text, but `maybeApplyTtsToPayload` returns at the unconditional under-10-character guard before synthesis.

## Solution Assessment

Yes, with small docs and release-note cleanup. Separating explicit tagged speech from untagged auto-TTS is the narrowest maintainable fix, and the added empty-string guard covers the reviewed markdown-only edge case.
