## Summary

Keep open. Current main still short-circuits explicit tagged speech under the 10-character guard, and the PR is a narrow implementation candidate, but it needs small docs/changelog cleanup before merge.

## What This Changes

The PR changes speech-core TTS payload handling so explicitly tagged speech text can bypass untagged short-text guards, adds an empty post-markdown guard, and adds regression coverage for tagged short, untagged short, and markdown-only tagged text.

## Best Possible Solution

Land this PR or an equivalent narrow patch after adding the required release note and docs clarification, preserving the untagged short auto-TTS guard while allowing non-empty explicit tagged speech; keep #74955 as the separate message-tool send-path parity review.

## Reproduction Assessment

Yes. On current main, `[[tts:text]]hello[[/tts:text]]` is parsed into `directives.ttsText`, but `maybeApplyTtsToPayload` returns at the unconditional under-10-character guard before `textToSpeech` can run.

## Solution Assessment

Yes, with small documentation and release-note cleanup. The PR distinguishes explicit tagged speech from untagged auto-TTS and keeps the post-markdown empty guard, which is the narrowest maintainable fix for the reported runtime bug.
