## Summary

Keep this PR open. Current main still has the reported unconditional short-text TTS guards, while the branch is a focused runtime/test fix for short explicit tagged speech; the remaining blockers are the required user-facing changelog entry and docs clarification.

## What This Changes

The PR updates speech-core TTS payload handling so explicit `[[tts:text]]` content can bypass untagged short-text guards, adds an empty post-markdown guard, and adds speech-core regression tests for tagged short, untagged short, and markdown-only tagged text.

## Best Possible Solution

Land this PR or an equivalent narrow patch after adding the release note and docs clarification, preserving the untagged short auto-TTS guard while allowing non-empty explicit tagged speech; keep #74955 as the separate outbound send-path parity review.

## Reproduction Assessment

Yes. On current main, `[[tts:text]]hello[[/tts:text]]` is parsed into `directives.ttsText`, but `maybeApplyTtsToPayload` returns at the under-10-character guard before `textToSpeech` can run.

## Solution Assessment

Yes, with small docs and changelog cleanup. Distinguishing explicit tagged speech from untagged auto-TTS is the narrowest maintainable fix, and the branch also keeps an empty post-markdown guard for markdown-only tagged content.
