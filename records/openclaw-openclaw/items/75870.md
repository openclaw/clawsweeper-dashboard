## Summary

Keep open. Current main still deliberately drops Signal own-account messages and all `syncMessage` envelopes, while the linked Note-to-Self report provides a concrete destination/source guard that is not implemented or superseded elsewhere.

## What This Changes

The issue asks for Signal channel inbound handling to work for single-number "Note to Self" usage by safely processing Signal `syncMessage.sentMessage` payloads only when the sent-message destination matches the sender's own Signal identity.

## Best Possible Solution

Add a Signal-plugin-owned, default-off Note-to-Self/self-messaging mode that preserves current behavior by default, extracts `syncMessage.sentMessage` only when the sync destination is the sender's own Signal identity, ignores ordinary sent-to-friend transcripts, and documents the safer single-number setup path without changing the separate-bot-number recommendation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by feeding `createSignalEventHandler` a Signal receive event with `syncMessage.sentMessage` for a Note-to-Self payload; current main returns on own-account sender and on `syncMessage` before dispatch.

## Solution Assessment

No, not exactly as pasted. The destination/source check is the right safety guard, but the maintainable fix should add an opt-in config surface, verify the `signal-cli` payload contract, keep normal sent transcripts ignored, and add regression tests plus docs.
