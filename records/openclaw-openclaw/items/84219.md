## Label Justifications

- `P1`: The PR targets silently dropped group/channel agent replies, which is an active broken channel workflow for real users.
- `merge-risk: 🚨 message-delivery`: The patch changes when group/channel replies are retried, suppressed, or surfaced after app-server bridge failures.

## Summary

Keep open; this is a plausible reliability fix, but it is not merge-ready because the retry guard can still replay after a failed mutating tool result, and the proof is mock/local-only for a transport-visible reply-delivery change.

## What This Changes

The PR retries early Codex app-server bridge client-close and turn-completion idle failures once, then surfaces repeated or unsafe failures visibly instead of letting group/channel replies disappear silently.

## Best Possible Solution

Keep the narrow pre-output bridge retry, but make every started mutating or messaging side-effect attempt a retry blocker even when it ends in an error, then require redacted live app-server/Telegram proof before merge.

## Reproduction Assessment

Yes for the PR-introduced bug: source inspection shows failed mutating tool results clear the pending retry blocker, and existing mutation contracts treat those failed mutating actions as unsafe. For the original dropped-reply bug, only source/test reproduction is present, not a live current-main transport repro.

## Solution Assessment

No, not as written. The retry direction is maintainable, but the side-effect boundary must include failed mutating tool completions and the transport-visible behavior needs real proof before this is the best fix.
