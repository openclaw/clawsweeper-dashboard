## Summary

Keep open. The PR addresses a plausible P1 app-server message-drop path, but the current head still permits retry after a failed mutating tool result and only supplies mocked/local proof for the live Telegram/app-server failure mode.

## What This Changes

The PR adds a one-time retry for selected Codex app-server client-close or turn-completion idle failures before visible output or side effects, surfaces skipped/repeated failures, and adds focused auto-reply tests plus a changelog entry.

## Best Possible Solution

Keep the pre-output retry design, but make the retry blocker fail closed after any mutating tool start/result and land only with focused regression coverage plus redacted live Telegram/app-server proof.

## Reproduction Assessment

Yes for the PR-introduced replay-safety bug: source inspection shows a mutating tool start, failed result, then app-server bridge error clears pending state without setting the side-effect latch before retry. The original live Telegram dropped-session symptom is supported by PR context but not live-reproduced here.

## Solution Assessment

No. Retrying only before output is a reasonable direction, but the implementation must fail closed for failed mutating tool results and needs real Telegram/app-server proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Latch failed mutating tool results before retrying:** `src/auto-reply/reply/agent-runner-execution.ts:1990-1991`
  - body: The retry gate clears `pendingPotentialToolSideEffectCallIds` on every tool result, but only sets `didObservePotentialToolSideEffect` when `evt.data.isError !== true`. A failed `exec`, `bash`, or `apply_patch` can still have changed the workspace before reporting failure; if the app-server bridge then throws one of the retried errors, the blocker is false and the whole turn replays. Treat any completed mutating tool attempt as retry-blocking and add a failed-result regression.
