## Summary

Keep this PR open: the retry direction targets a real dropped-reply surface, but the current head still can replay after a failed mutating tool result and the PR only supplies mock/unit proof for the live Telegram/app-server behavior.

## What This Changes

The PR modifies auto-reply execution and block reply delivery to retry selected Codex app-server bridge failures before visible output or side effects, adds focused tests, and updates the changelog.

## Best Possible Solution

Keep the retry feature, but make any started or completed mutating tool replay-unsafe regardless of failed result status, add regression coverage for that case, and require redacted live Telegram/app-server proof before merge.

## Reproduction Assessment

Yes for the PR-introduced replay-safety bug: source inspection shows failed mutating tool results clear the pending blocker without latching a side-effect flag, then the app-server bridge catch path can retry. The original dropped Telegram symptom is supported by PR context and tests, but this review did not live-reproduce it.

## Solution Assessment

No: the overall retry direction is plausible, but the implementation is not safe until failed mutating tool completions are treated as replay-unsafe and the live transport path is proven.
