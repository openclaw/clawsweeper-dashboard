## Summary

This PR should stay open: the retry direction addresses a real dropped-reply failure, but the current patch still allows a Codex app-server bridge retry after an errored mutating tool result, which can replay commands, file edits, or message sends that may already have had side effects. The supplied Telegram screenshot is useful context, but it does not make the new retry/unsafe-failure behavior inspectable.

## What This Changes

The PR adds one-time Codex app-server bridge retry and visible failure handling in auto-reply, latches visible-output and side-effect blockers, updates reply-delivery callback plumbing, tests, and the changelog.

## Best Possible Solution

Keep the pre-output app-server retry, but fail closed after any mutating tool starts or completes, including errored results, and require redacted runtime or transport proof before merge.

## Reproduction Assessment

Yes for the PR-introduced replay-safety bug: source inspection shows an errored mutating tool result clears the pending blocker without latching a side effect before the bridge retry decision. ## Solution Assessment

No. The retry approach is maintainable only if the replay boundary is fail-closed for errored mutating tool results; otherwise the fix can trade silent drops for duplicated side effects.
