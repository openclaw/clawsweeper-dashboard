## Summary

Keep this open. Current main still has no SQLite or FTS-backed wiki index in the bundled memory-wiki plugin: `wiki_search` uses compiled digest candidate selection plus Markdown file reads, the plugin config/docs expose only `shared` and `local` search backends, and write paths refresh JSON/JSONL digest artifacts rather than a database cache. The cross-reference to #75312 is related but tracks a separate shared-memory runtime failure, not this performance/indexing feature.

## What This Changes

The issue requests a SQLite FTS5-backed index for memory-wiki `wiki_search`, refreshed by wiki write/compile paths and used before the existing Markdown scan fallback.

## Best Possible Solution

Add the FTS cache inside the bundled memory-wiki plugin as a read-optimized artifact under the wiki cache area, refresh it from compile/apply/ingest paths, query it before the existing local Markdown search, and keep digest/file-scan behavior as the compatibility fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from current source because memory-wiki has no SQLite symbols or config/docs surface and `wiki_search` still reads digest candidates plus Markdown files.

## Solution Assessment

Yes. The narrow maintainable direction is plugin-local indexing in memory-wiki; reusing the active memory database would blur the current split where active memory owns recall/indexing and memory-wiki owns compiled wiki pages and provenance-specific retrieval.
