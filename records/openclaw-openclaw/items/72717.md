## Summary

Keep open. Current main still implements `wiki_search` through the bundled `memory-wiki` plugin's compiled-digest and Markdown read path, with no SQLite/FTS wiki index or freshness/rebuild logic. The issue discussion adds useful implementation constraints around stale-index drift, and the related #75312 report is adjacent shared-memory runtime fallout rather than a duplicate of this indexing/performance request.

## What This Changes

Add an internal SQLite FTS5-backed cache for `memory-wiki` wiki content so `wiki_search` can query ranked indexed results, refresh the cache after compile/apply writes, and fall back to file scanning when the cache is missing or stale.

## Best Possible Solution

Implement this inside the bundled `memory-wiki` plugin as a read-optimized local SQLite FTS cache with Markdown remaining the source of truth, mtime/hash validation before trusting rows, rebuild support from `wiki compile`, refresh coverage for `wiki_apply` and other write/import paths, file-scan fallback when unavailable or stale, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Partly. This is a feature/performance request rather than a crash report, but current main gives a high-confidence source-level reproduction of the missing behavior: `wiki_search` reads digests and Markdown files and never queries a SQLite FTS index; a large-vault benchmark would quantify the performance delta.

## Solution Assessment

Yes, if scoped narrowly. A local FTS read-through cache is the maintainable next step for wiki search performance; the safer alternative is to avoid adopting the companion DB's broader durable decision/exchange schema in the first fix.
