## Summary

Keep open: current main and v2026.6.10 still use compiled digest prefiltering plus Markdown reads for memory-wiki search, not a SQLite FTS-backed wiki index. The request remains meaningful, but the next step is maintainer design for index freshness, rebuild/doctor behavior, and whether preserved decision context belongs in a cache or durable data model.

## What This Changes

Add a plugin-owned SQLite FTS5-backed memory-wiki search index, refreshed by compile/apply, so `wiki_search` can avoid full Markdown scans and improve durable decision retrieval.

## Best Possible Solution

Define the memory-wiki source-of-truth and freshness contract, then implement a plugin-owned SQLite FTS path with rebuild/doctor support and focused compile/apply/search coverage if maintainers accept the data-model direction.

## Reproduction Assessment

No high-confidence live benchmark was reproduced for the reported scale/performance loss. Source inspection does confirm current main and v2026.6.10 still search wiki content through digest-prefiltered Markdown reads rather than SQLite FTS.

## Solution Assessment

Unclear. SQLite FTS is a plausible implementation, but the best fix needs maintainer decisions on cache freshness, rebuild/doctor behavior, and whether preserved decision exchanges belong in wiki Markdown, plugin SQLite state, or a broader memory model.
