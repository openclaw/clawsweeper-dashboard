## Summary

Keep open: current main and v2026.6.10 still implement memory-wiki search as digest-prefiltered Markdown reads plus optional shared memory search, not as a SQLite FTS-backed wiki index. The request remains useful, but it needs maintainer design for cache-vs-durable source of truth, stale-index detection, rebuild/doctor behavior, and how it relates to broader memory storage work.

## What This Changes

The issue requests an internal SQLite FTS5-backed wiki_search index that is updated by wiki compile/apply paths and used for ranked wiki retrieval.

## Best Possible Solution

Define a plugin-owned wiki search-index contract first, then implement SQLite FTS as a rebuildable or deliberately durable memory-wiki layer with explicit mtime/hash freshness, compile/apply/refresh integration, doctor validation, and focused search/ranking tests.

## Reproduction Assessment

No high-confidence live benchmark was reproduced for the reported scale/performance loss. Source inspection does give a high-confidence reproduction of the missing requested behavior: current main and v2026.6.10 still search wiki content through digest-prefiltered Markdown reads rather than SQLite FTS.

## Solution Assessment

Unclear. SQLite FTS is a plausible implementation direction, but the best solution needs maintainer decisions on cache freshness, rebuild/doctor behavior, and whether preserved decision exchanges belong in Markdown, plugin SQLite state, or a broader memory model.
