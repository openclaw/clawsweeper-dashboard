## Summary

Keep open: current main and the latest release still implement memory-wiki search as digest-assisted Markdown reads, not a SQLite FTS cache, and the linked shared-memory crash report is a separate open bug rather than a replacement for this performance/storage feature.

## What This Changes

The issue requests an internal SQLite FTS5-backed index for memory-wiki content that `wiki_search` can query with ranked results and Markdown fallback.

## Best Possible Solution

Design a memory-wiki-owned derived FTS cache with drift detection, rebuild tooling, compile/apply/import integration, and Markdown fallback, or explicitly move the work into a broader shared-memory indexing design.

## Reproduction Assessment

Not applicable as a bug reproduction, but the missing capability is source-reproducible: current `wiki_search` reads digest-selected Markdown files and memory-wiki has no SQLite/FTS implementation.

## Solution Assessment

Yes directionally, with a design caveat. A derived cache with stale detection, rebuild, and fallback is the maintainable shape; making the companion DB the durable source of truth would be a larger product decision.

## Risks / Open Questions

- A derived SQLite index can silently drift after crashed `wiki_apply` writes or manual Markdown edits unless hash/mtime validation and a rebuild path are designed up front.
- Node/Bun and platform-specific SQLite/FTS availability need a clear fallback contract so `wiki_search` does not become dependent on an optional storage layer.
- The request overlaps conceptually with the built-in memory engine's FTS index, so maintainers should choose whether memory-wiki owns a plugin-local cache or a shared indexing seam is needed.
