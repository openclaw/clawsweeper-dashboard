## Summary

Keep open. Current main still uses the memory-wiki plugin's local Markdown/digest search path for wiki results, not a SQLite FTS-backed wiki index, and the related #75312 report is a separate shared-memory runtime failure rather than a duplicate of this performance/indexing request.

## What This Changes

Add an internal SQLite FTS5-backed index for memory-wiki pages so `wiki_search` can query ranked indexed wiki results with stale/missing-index fallback to file scanning.

## Best Possible Solution

Implement this in the bundled `memory-wiki` plugin as a read-optimized SQLite FTS cache for wiki pages, with Markdown remaining the source of truth, mtime/hash validation before trusting indexed rows, `wiki compile` rebuild support, `wiki_apply` refresh coverage, and file-scan fallback whenever the index is missing, stale, or unavailable.

## Reproduction Assessment

Partly. This is a feature/performance request rather than a crash report, but the current-main code path is clear: `wiki_search` reads Markdown pages after an optional compiled-digest prefilter and does not query a SQLite FTS table; a benchmark with a 50+ page vault would quantify the reported latency/token cost.

## Solution Assessment

Yes, if scoped narrowly. A local FTS cache is a maintainable way to improve `wiki_search` performance, but it should include the freshness/rebuild safeguards raised in the discussion and should not turn the companion DB's durable decision/exchange schema into the first fix.

Add an internal SQLite FTS5-backed cache under the memory-wiki vault metadata/cache area for queryable wiki pages, including title, path/id, body/search text, claim text/metadata needed for existing result metadata, and a freshness marker such as mtime plus content hash. `wiki_search` should prefer the FTS index for wiki corpus ranking when the index is present and fresh, then fall back to the current file-scan/digest path when SQLite/FTS is unavailable, the index is missing, or freshness checks fail. `wiki compile` should rebuild/refresh the FTS cache, and `wiki_apply` should leave the cache fresh via the existing compile path or a narrow update hook.
