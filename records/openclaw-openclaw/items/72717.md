## Summary

Keep open: current main still uses digest/file-backed Markdown search for memory-wiki, with no SQLite FTS cache, update path, stale-index guard, or fallback contract. The related shared-memory crash report is separate and has its own open fix PR, so it does not supersede this storage/performance feature.

## What This Changes

Request an internal SQLite FTS5-backed index for wiki_search that updates on wiki writes, ranks results from the index, and falls back to Markdown scanning when stale or missing.

## Best Possible Solution

Implement a plugin-owned derived FTS cache with drift detection, rebuild tooling, compile/apply/import integration, and Markdown fallback, or explicitly choose a broader shared-memory indexing design before coding.

## Reproduction Assessment

Yes for the missing capability, though this is not a bug reproduction. Source inspection on current main shows wiki_search still reads compiled digest candidates and Markdown files, with no SQLite/FTS branch or index refresh path.

## Solution Assessment

Yes directionally, with a design caveat. A derived cache with stale detection and fallback is the narrow maintainable direction; making the companion database the durable source of truth would be a larger product decision.
