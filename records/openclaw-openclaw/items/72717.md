## Summary

Keep open: current main still uses compiled JSON digests plus Markdown reads for memory-wiki search/get, while compile/apply paths do not maintain a SQLite FTS cache; no canonical PR supersedes this, and the remaining work needs cache-drift design.

## What This Changes

The issue requests a SQLite FTS5-backed index for `wiki_search`, refreshed by wiki writes and used before falling back to Markdown scanning.

## Best Possible Solution

Add a plugin-owned derived FTS cache with drift detection, rebuild tooling, compile/apply/import integration, and Markdown fallback, or explicitly route the work into a broader shared-memory indexing design.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main verifies the requested capability is absent from the memory-wiki query, compile, and apply paths.

## Solution Assessment

Yes with a design caveat: a derived SQLite FTS cache with stale-index detection and fallback is a plausible maintainable direction. Treating the companion database as the durable source of truth would be a larger product change.

## Risks / Open Questions

- A derived FTS cache could silently drift after crashed `wiki_apply` writes or manual edits unless hash/mtime validation and rebuild behavior are specified.
- SQLite/FTS availability must stay optional across Node and Bun runtime paths, with Markdown fallback preserved.
- The design overlaps conceptually with memory-core FTS, so maintainers need to choose between a plugin-owned cache and a new shared indexing seam.
