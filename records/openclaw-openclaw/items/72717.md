## Summary

Keep open: current main and the latest release still use memory-wiki compiled digests plus Markdown reads for wiki search, not a SQLite FTS-backed wiki index. The request is valid, but it needs maintainer design around persisted cache freshness and source-of-truth behavior before implementation.

## What This Changes

Request to add a SQLite FTS5-backed `wiki_search` index that updates on wiki compile/apply and falls back when stale or missing.

## Best Possible Solution

Design a memory-wiki-owned SQLite search layer with explicit hash/mtime validation and rebuild or doctor recovery, keeping either Markdown or SQLite as the clear canonical source.

## Reproduction Assessment

Not applicable as a feature/performance request. Source inspection gives high confidence that current main still lacks the requested SQLite FTS wiki index, but there is no broken existing contract to reproduce as a bug.

## Solution Assessment

Unclear as stated. A SQLite FTS cache is plausible, but the best solution must first settle stale-index detection, rebuild/doctor behavior, and whether the durable wiki layer remains Markdown-first or becomes database-first.
