## Summary

Keep open: current main and v2026.6.8 still use digest-prefiltered Markdown reads for memory-wiki search, not a SQLite FTS-backed wiki index. The request remains valid, but it needs maintainer design around index freshness, rebuild behavior, and whether SQLite is only a cache or part of the durable memory-wiki data model.

## What This Changes

Request to add a SQLite FTS5-backed index for memory-wiki `wiki_search` so large wiki vault searches are faster and more reliable than Markdown scanning.

## Best Possible Solution

Define the memory-wiki source-of-truth and freshness contract, then add a plugin-owned SQLite FTS path with rebuild/doctor repair and focused compile/apply/search coverage if maintainers accept the data-model direction.

## Reproduction Assessment

No high-confidence live benchmark was established for the reported scale/performance loss. Source inspection does confirm current main and v2026.6.8 still search wiki content through digest-prefiltered Markdown reads rather than SQLite FTS.

## Solution Assessment

Unclear. SQLite FTS is plausible, but the best fix needs maintainer decisions on cache freshness, rebuild/doctor behavior, and whether preserved decision exchanges belong in wiki Markdown, plugin SQLite state, or a broader memory model.
