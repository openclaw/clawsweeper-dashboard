Keep open: current main and v2026.6.8 still use digest-prefiltered Markdown reads for memory-wiki search, not a SQLite FTS-backed wiki index. ## What This Changes

Request an internal SQLite FTS5 index for memory-wiki so wiki_search can query ranked indexed wiki content and refresh it from wiki compile/apply operations.

## Best Possible Solution

Define the memory-wiki source-of-truth and freshness contract, then add a plugin-owned SQLite FTS path with rebuild/doctor repair and focused compile/apply/search coverage if maintainers accept the data-model direction.

## Reproduction Assessment

No high-confidence live reproduction or benchmark was established for the reported scale/performance loss. Source inspection does confirm current main still searches wiki content through digest-prefiltered Markdown reads rather than SQLite FTS.

## Solution Assessment

Unclear. SQLite FTS is a plausible implementation, but the best fix requires maintainer decisions on cache freshness, rebuild/doctor behavior, and whether preserved decision exchanges belong in wiki Markdown, plugin SQLite state, or a broader memory model.
