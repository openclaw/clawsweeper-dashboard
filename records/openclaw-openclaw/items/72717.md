## Summary

Keep open. Current main still implements `wiki_search` as compiled-digest prefiltering plus markdown file reads/scoring, not a SQLite FTS-backed wiki index; the request remains valid and the follow-up comment identifies a real design constraint around stale index drift.

## What This Changes

Add a memory-wiki SQLite FTS5 index for `wiki_search`, refresh it from compile/apply writes, rank through the index first, and fall back to markdown scanning when the index is missing or stale.

## Best Possible Solution

Implement the index inside the bundled `memory-wiki` plugin as a read-optimized cache: build or refresh it during compile/apply/import flows, validate page hashes or mtimes on read, expose a rebuild/status/doctor path, preserve existing ranking/provenance fields, and fall back to the current markdown scan whenever the cache cannot be trusted.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. Source inspection gives a high-confidence verification path that current `wiki_search` still scans markdown after digest prefiltering, and a synthetic 50+ page vault could benchmark the reported latency/token concern.

## Solution Assessment

Yes, an internal plugin-owned FTS cache is a plausible narrow solution if it is treated as a cache rather than a durable source of truth. The safer shape is the one @m13v described: mtime/hash validation plus a rebuild command, not a silent dual-write-only index.
