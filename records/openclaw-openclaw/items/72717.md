## Summary

Keep open: current main still lacks a SQLite FTS-backed memory-wiki search cache, and the existing implementation remains digest/Markdown-backed. The related #75312 report is a separate shared-memory runtime bug, while the May 1 drift comment is a real implementation requirement for this feature.

## What This Changes

Add a SQLite FTS5-backed index for memory-wiki `wiki_search`, refreshed by compile/apply paths with fallback to file scanning when the cache is missing or stale.

## Best Possible Solution

Keep Markdown/digests as the durable wiki source of truth and add a memory-wiki-owned derived FTS cache with stale detection, rebuild behavior, and file-scan fallback.

## Reproduction Assessment

Yes for the missing capability: source inspection of current main shows `wiki_search` still reads digest candidates and Markdown files, with no SQLite/FTS path. This is a feature/performance request rather than a broken existing contract.

## Solution Assessment

Yes, if implemented as a derived cache. A plugin-local FTS cache with mtime/hash drift checks and fallback is narrower than adopting the companion database as the durable wiki store.

## Repair Work Prompt

Implement a memory-wiki-owned SQLite FTS5 cache for wiki pages. Keep Markdown pages plus `.openclaw-wiki/cache/agent-digest.json` and `claims.jsonl` as the source of truth; the SQLite database must be a derived read cache only. Add a small plugin-local cache module, likely under `extensions/memory-wiki/src/`, that stores page path/title/body/metadata needed by `wiki_search`, validates freshness with mtime/hash or equivalent per-page state, and falls back to the current file-scan path when SQLite is unavailable, missing, corrupt, or stale. Refresh or rebuild the cache from `wiki compile`, `wiki_apply`, ingest/import paths that call compile, and provide a rebuild path through existing compile behavior rather than adding a broad new product surface. Preserve current search modes, claim/provenance metadata, shared-memory corpus behavior, and local/shared backend semantics. Do not import core `src/**` or memory-core internals; use public plugin SDK/host storage helpers or local code consistent with bundled plugin boundaries. Add focused regression tests for indexed search, stale/manual-edit fallback or rebuild, apply/compile refresh, SQLite-unavailable fallback, and docs/changelog updates. Treat #75312 as related context only; do not mix that shared-memory manager bug into this PR.
