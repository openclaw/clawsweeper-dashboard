## Summary

Keep this open. Current main still implements `wiki_search` in the bundled memory-wiki plugin with compiled JSON/JSONL digest prefiltering plus Markdown file reads, and there is no SQLite/FTS-backed wiki index, cache path, stale-index fallback policy, or config/docs surface for it. The request is recent, concrete, and narrow enough for a focused memory-wiki feature PR.

## What This Changes

The issue requests an internal SQLite FTS5 cache for memory-wiki so `wiki_search` can query ranked indexed wiki content first, refresh the cache after compile/apply/write operations, and fall back to the existing file scan when the cache is stale or unavailable.

## Best Possible Solution

Add the FTS cache inside the bundled memory-wiki plugin, preferably under `.openclaw-wiki/cache`, populate it from parsed wiki page metadata plus body/claim text during compile/apply/ingest/import refresh paths, query it before the current local digest/Markdown path, and keep the existing digest/file scan as the fallback when SQLite is unavailable or the cache is stale.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from source and docs because memory-wiki has no SQLite/FTS search path or config surface and still reads digest candidates plus Markdown files.

## Solution Assessment

Yes, the requested direction is the best narrow solution if it stays inside memory-wiki as a read-optimized generated cache. Reusing `memory/main.sqlite` would blur ownership between active memory recall and wiki-specific ranking/provenance behavior.
