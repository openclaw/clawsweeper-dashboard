## Summary

Keep open: current main still lacks a SQLite FTS-backed `memory-wiki` search cache, and the current code remains digest/file-backed with no SQLite path or index refresh behavior. The drift concern raised in discussion is a real implementation requirement, while #75312 is a separate shared-memory runtime bug rather than a duplicate of this performance feature.

## What This Changes

Request to add a SQLite FTS5-backed cache for `memory-wiki` `wiki_search`, refreshed by compile/apply flows with ranked search and file-scan fallback.

## Best Possible Solution

Implement a memory-wiki-owned read-through FTS cache derived from Markdown, with hash/mtime stale checks, rebuild through compile/apply/import paths, file-scan fallback, docs, tests, and a changelog entry.

## Reproduction Assessment

Yes for the missing capability: source inspection of current main shows `wiki_search` still reads digest candidates and Markdown files, with no SQLite/FTS index path. This is a feature/performance request rather than a broken existing contract.

## Solution Assessment

Yes, if kept as a derived cache. A plugin-local FTS cache with drift detection and fallback is narrower and safer than turning the companion database pattern into the durable wiki source of truth.

## Repair Work Prompt

Implement a narrow memory-wiki FTS cache for this issue. Add a plugin-local SQLite FTS5-backed read-through cache for wiki Markdown pages so `wiki_search` queries the cache first for ranked wiki results, detects stale/missing cache state with page mtime/hash metadata, and falls back to the existing file-scan path if SQLite/FTS is unavailable or the cache cannot be trusted. Keep Markdown as the source of truth; do not convert the wiki or companion decision DB idea into a durable database source of truth. Refresh/rebuild the cache from existing `wiki compile`, `wiki_apply`, and relevant import paths, and include a manual rebuild path only if it naturally fits the existing `openclaw wiki compile` surface. Do not import core `src/**` from extension production code; use plugin-local code, `node:sqlite`, or an existing public SDK/runtime seam. Do not conflate this with #75312, which is a separate shared-memory manager bug. Add focused regression tests for cache creation/query ranking, stale/manual-edit detection, missing/failed SQLite fallback, and compile/apply refresh behavior. Update memory-wiki docs, CLI docs if user-visible behavior changes, and add a user-facing changelog entry.
