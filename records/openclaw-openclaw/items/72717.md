## Summary

Keep this open. Current main still has no SQLite/FTS-backed wiki index in memory-wiki; `wiki_search` is implemented with compiled digest prefiltering plus Markdown reads, and the request is recent, concrete, plugin-owned, and suitable for one focused feature PR.

## What This Changes

The issue requests a memory-wiki SQLite FTS5 cache for `wiki_search`, refreshed by compile/apply/write paths and used before falling back to Markdown scanning.

## Best Possible Solution

Add the FTS cache inside the bundled memory-wiki plugin as a generated read cache under `.openclaw-wiki/cache`, populate it from parsed page metadata/body/claim text during compile/apply/ingest/import refresh paths, query it before the current digest/Markdown path, and keep the existing file scan as the fallback when SQLite is unavailable, missing, or stale.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from source and docs because current main has no memory-wiki SQLite/FTS branch and still reads digest candidates plus Markdown files.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution if it stays inside memory-wiki as a generated cache and preserves Markdown as the source of truth. Reusing or mutating `memory/main.sqlite` would mix wiki-specific ranking/provenance with active memory search ownership.
