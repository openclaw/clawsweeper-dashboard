## Summary

Keep open: current main still lacks a SQLite FTS-backed wiki index; `wiki_search` uses the memory-wiki plugin's compiled digest prefilter and Markdown file reads, while `wiki_apply` and `wiki compile` only refresh generated Markdown/digest artifacts. The related #75312 report is adjacent shared-memory runtime breakage, not a duplicate of this indexing/performance request.

## What This Changes

Add a SQLite FTS5 cache for memory-wiki pages so `wiki_search` can query ranked indexed results, refresh on compile/apply writes, and fall back to file scanning when the cache is missing or stale.

## Best Possible Solution

Implement the cache inside the bundled `memory-wiki` plugin with Markdown as source of truth, mtime/hash validation before trusting indexed rows, rebuild support, compile/apply refresh coverage, and file-scan fallback whenever SQLite is unavailable, stale, or missing.

## Reproduction Assessment

Partly. We have a high-confidence source-level reproduction for the missing behavior: current `wiki_search` reads Markdown after an optional compiled-digest prefilter and has no SQLite/FTS path; quantifying the reported latency/token cost still needs a benchmark vault with roughly 50 or more pages.

## Solution Assessment

Yes, if scoped narrowly. A local read-through FTS cache is the maintainable path for the reported search performance gap; the safer boundary is to defer the companion DB's durable decision/exchange schema until a separate product discussion.

## Repair Work Prompt

Implement a narrow `memory-wiki` SQLite FTS cache for this issue. Keep Markdown pages as the source of truth. Add an internal local SQLite/FTS5 index for wiki page title/body/search metadata, query it first from `wiki_search`, rank results, and fall back to the current Markdown/digest scan whenever the index is missing, stale, corrupt, or SQLite is unavailable. Refresh or rebuild the cache from `wiki compile` and after `wiki_apply` mutations; include mtime or content-hash validation on read so manual edits and crashed writes cannot silently return stale results. Add an explicit rebuild path through the existing wiki compile/CLI flow if needed. Do not implement the companion DB durable decisions/exchange schema in this PR, do not add networked storage, and do not change active-memory FTS behavior except where integration tests require keeping wiki corpus supplements working. Update user-facing docs and `CHANGELOG.md` because this changes `wiki_search` behavior.
