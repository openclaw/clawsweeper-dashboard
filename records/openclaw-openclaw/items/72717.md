## Summary

Keep open: current main still lacks a SQLite FTS-backed `memory-wiki` search cache; `wiki_search` still reads Markdown via digest-selected candidates and file-scan fallback, while `wiki compile` and `wiki_apply` only refresh Markdown/digest artifacts. The linked #75312 report is adjacent shared-memory runtime breakage, not a duplicate of this indexing/performance request.

## What This Changes

Request to add a SQLite FTS5-backed index for `wiki_search` so memory-wiki can rank wiki pages without scanning Markdown and refresh the cache on compile/apply writes.

## Best Possible Solution

Implement the cache inside the bundled `memory-wiki` plugin with Markdown as source of truth, content-hash or mtime stale detection, an explicit rebuild path, and file-scan fallback when SQLite is unavailable or stale; keep the companion durable decision/exchange schema as a separate product discussion.

## Reproduction Assessment

Yes for the missing implementation: source inspection on current main shows `wiki_search` still reads Markdown after optional digest prefiltering and has no SQLite/FTS path. The exact performance impact needs a benchmark vault with about 50+ pages.

## Solution Assessment

Yes, if scoped narrowly to a read-through wiki search cache. A local FTS cache is the maintainable fix for the search-performance gap, while the broader companion DB durability model should not be folded into the same change.

## Repair Work Prompt

Implement a narrow SQLite FTS5-backed wiki search cache for the bundled memory-wiki plugin for https://github.com/openclaw/openclaw/issues/72717. Keep Markdown wiki pages as the source of truth. Add a plugin-local cache module, likely under `extensions/memory-wiki/src/`, that indexes page path/id/title/body plus enough metadata for ranking/snippets. `wiki_search` should query the FTS cache first when present and fresh, then fall back to the existing Markdown scan path if SQLite is unavailable, the cache is missing, or stale state is detected. `wiki compile` and `wiki_apply` should refresh or invalidate the cache after successful page/digest updates, and there should be an explicit rebuild path through the existing compile flow rather than a startup migration. Include mtime or content-hash validation so manual edits and interrupted writes cannot silently hide pages. Do not implement the broader companion decision/exchange database schema from the issue body, and do not fold in the separate #75312 shared-memory `.search is not a function` bug. Update memory-wiki docs/CLI docs and add a user-facing changelog entry. Validate with targeted memory-wiki tests, formatter check, and the changed gate in Testbox per repo policy.
