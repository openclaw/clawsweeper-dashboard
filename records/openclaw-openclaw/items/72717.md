## Summary

Keep this open. Current main still has no SQLite/FTS-backed wiki index in memory-wiki: `wiki_search` uses compiled digest prefiltering plus Markdown reads, compile/apply only refresh JSON/JSONL digest artifacts, and the docs/tests describe that digest-backed behavior rather than an indexed cache. The request is recent, unprotected, plugin-owned, and the maintainer comment about mtime/hash drift is a concrete acceptance criterion rather than a reason to close or defer.

## What This Changes

The issue requests a SQLite FTS5-backed cache for memory-wiki `wiki_search`, updated by wiki compile/apply/write paths and used before falling back to Markdown scanning.

## Best Possible Solution

Add the cache inside the bundled memory-wiki plugin as derived local state under `.openclaw-wiki/cache`, keep Markdown and compiled digests as the source of truth, validate cache freshness on read, provide a rebuild path through compile, and preserve the current digest/Markdown scan fallback whenever SQLite is missing, stale, locked, or unavailable. Keep the distinct shared-memory runtime bug in #75312 separate unless a small guard is naturally needed to preserve existing behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from current source and docs because memory-wiki has no SQLite/FTS branch and still reads digest candidates plus Markdown files.

## Solution Assessment

Yes. A generated FTS cache owned by memory-wiki is the narrowest maintainable direction, provided it includes stale-cache detection and fallback; reusing `memory/main.sqlite` or making the database the durable source of truth would mix ownership with memory-core and broaden the feature beyond the request.
