## Summary

Keep open. Current main still implements `wiki_search` through the memory-wiki plugin's Markdown/digest read path, not a SQLite FTS-backed wiki index, and the related #75312 report is an adjacent shared-memory runtime failure rather than a duplicate of this indexing/performance request.

## What This Changes

Add an internal SQLite FTS5-backed index for memory-wiki pages so `wiki_search` can return ranked indexed wiki results with stale or missing index fallback to file scanning.

## Best Possible Solution

Implement this inside the bundled `memory-wiki` plugin as a read-optimized SQLite FTS cache for wiki pages, with Markdown remaining the source of truth, mtime or content-hash validation before trusting indexed rows, `wiki compile` rebuild support, `wiki_apply` refresh coverage, and file-scan fallback whenever the index is missing, stale, or unavailable.

## Reproduction Assessment

Partly. This is a feature/performance request rather than a crash report, but the current-main code path is clear: `wiki_search` reads Markdown pages after an optional compiled-digest prefilter and does not query a SQLite FTS table; a 50+ page vault benchmark would quantify the latency/token impact.

## Solution Assessment

Yes, if scoped narrowly. A local read-through FTS cache is a maintainable way to improve `wiki_search`, but the first fix should include freshness and rebuild safeguards and should not convert the companion DB's durable decision/exchange schema into the primary wiki store.

Add an internal SQLite FTS5-backed cache under the memory-wiki vault metadata/cache area for queryable wiki pages, including title, path/id, body/search text, claim text/metadata needed for existing result metadata, and a freshness marker such as mtime plus content hash. `wiki_search` should prefer the FTS index for wiki corpus ranking when the index is present and fresh, then fall back to the current file-scan/digest path when SQLite/FTS is unavailable, the index is missing, or freshness checks fail. `wiki compile` should rebuild/refresh the FTS cache, and `wiki_apply` should leave the cache fresh via the existing compile path or a narrow update hook. Include tests for indexed search, stale/missing index fallback, manual file edit detection, `wiki_apply`/compile refresh, and SQLite unavailable fallback.
