## Summary

Keep open: current `main` still implements `memory-wiki` search as digest/file-backed Markdown scanning with no SQLite/FTS index, no index refresh path, and no stale-index fallback behavior. The related https://github.com/openclaw/openclaw/issues/75312 is a separate shared-memory runtime bug, not a duplicate of this performance feature.

## What This Changes

Add an internal SQLite FTS-backed index for `wiki_search` over memory-wiki pages, refreshed by compile/apply/write paths and falling back to file scan when stale or unavailable.

## Best Possible Solution

Implement a memory-wiki-owned derived FTS cache over Markdown pages, with stale detection, rebuild through compile/apply/import paths, file-scan fallback, docs, tests, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current `main` proves the missing capability: `wiki_search` still reads compiled digest candidates and Markdown files, with no SQLite/FTS index path or refresh behavior; this is a feature/performance gap, not a broken existing contract.

## Solution Assessment

Yes, if implemented as a derived cache. A plugin-local FTS cache with hash/mtime drift detection and Markdown fallback is narrower than making the companion database pattern the durable source of truth.
