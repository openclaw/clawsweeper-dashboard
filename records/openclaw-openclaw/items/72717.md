## Summary

Keep this open. Current `main` still implements `wiki_search` in the bundled memory-wiki plugin with compiled JSON/JSONL digest prefiltering plus Markdown reads, and there is no SQLite/FTS-backed wiki index, cache path, stale-index fallback policy, or config/docs surface for it. The request is recent, concrete, plugin-owned, and not superseded by the related shared-memory search bug.

## What This Changes

Request to add a SQLite FTS5-backed cache for `wiki_search` in memory-wiki so large wiki vault searches avoid repeated Markdown scanning and can refresh after wiki writes.

## Best Possible Solution

Add the FTS cache inside the bundled memory-wiki plugin, preferably under `.openclaw-wiki/cache`, populate it from parsed wiki metadata plus body/claim text during compile and write/import refresh paths, query it before the current digest/Markdown path, and keep the existing local search as the fallback when SQLite is unavailable or the cache is stale.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from source and docs because current `main` has no SQLite/FTS path and still reads digest candidates plus Markdown files.

## Solution Assessment

Yes. A memory-wiki-owned generated FTS cache is the narrow maintainable direction; reusing `memory/main.sqlite` would blur ownership between active memory recall and wiki-specific ranking/provenance behavior.
