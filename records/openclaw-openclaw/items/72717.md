## Summary

Keep open: current main and the latest release still implement memory-wiki search as digest/file-backed Markdown scanning, with no SQLite FTS cache, refresh path, stale-index detection, or fallback contract. The related shared-memory crash report is a separate runtime bug, not a duplicate of this performance/storage feature.

## What This Changes

Add a memory-wiki-owned SQLite FTS5 cache for `wiki_search` that updates on compile/apply and falls back to file scanning when stale or missing.

## Best Possible Solution

Implement a plugin-owned derived FTS cache with drift detection, rebuild tooling, compile/apply/import integration, and Markdown fallback, or explicitly choose a broader shared-memory indexing design before coding.

## Reproduction Assessment

Yes for the missing capability, though this is not a bug reproduction. Source inspection on current main shows `wiki_search` still reads compiled digest candidates and Markdown files, with no SQLite/FTS branch or index refresh path.

## Solution Assessment

Yes with a design caveat. A derived cache with stale detection and fallback is the narrow maintainable direction; making the companion database the durable source of truth would be a larger product change.

## Risks / Open Questions

- A SQLite FTS cache must not silently drift after `wiki_apply` crashes or manual edits; hash/mtime validation, rebuild, and Markdown fallback need to be part of the design.
- SQLite/FTS availability differs across runtimes and platforms, so Node/Bun paths need an optional fallback rather than making wiki search depend on FTS being present.
- The feature overlaps conceptually with memory-core FTS behavior, but current public plugin boundaries mean memory-wiki needs a plugin-owned cache or an explicit new shared seam.
