## Summary

Keep open: the stack-safe attachment parsing and stale queue cleanup are still needed and well proven, but the PR head is not merge-ready because its inline extraction-limit resolver bypasses the capped helper already on current main and in v2026.6.9.

## What This Changes

The branch makes gateway and Control UI attachment data URL/base64 parsing stack-safe, moves chat attachment size resolution into media config, routes inbound file extraction through existing media/PDF settings, drops stale stack-overflow failed queue items, and updates docs/tests.

## Best Possible Solution

Rebase or narrow the PR onto current main, keep the stack-safe parser and stale-queue cleanup, and preserve `resolveFileExtractionLimits` caps unless maintainers explicitly accept broader extraction semantics.

Yes, at source level: current main and v2026.6.9 still use full-string regex parsing over multi-megabyte gateway/UI attachment payloads, and the discussion includes matching production logs. I did not run a fresh live current-main repro in this read-only pass.

## Solution Assessment

Partly. The stack-safe parser and stale-queue cleanup are the right fix surface, but the extraction-limit portion should be rebased onto the capped current-main helper or explicitly approved as a policy change.
