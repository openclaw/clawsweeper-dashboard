## Summary

Keep open. Current main still has the split-singleton risk this draft PR targets, and the proposed patch is a narrow, relevant fix with regression coverage and no blocking review findings; it still needs maintainer review plus CI/Testbox validation before merge.

## What This Changes

The PR moves memory plugin registry storage into `createPluginRuntimeStore` and adds mirrored-runtime tests for memory-state, Memory Wiki bridge import, and Memory Wiki status artifact counts.

## Best Possible Solution

Merge a narrow, backward-compatible fix that makes the active memory capability registry visible across canonical and mirrored runtime module instances, keeps the existing memory plugin SDK API intact, and leaves any host-owned capability-service redesign as a separate follow-up.

## Reproduction Assessment

Yes. #74981 gives concrete packaged Gateway steps with `memory-core` plus Memory Wiki bridge returning zero artifacts, and current main source shows the module-local registry that can split across mirrored runtime modules.

## Solution Assessment

Yes for the immediate regression. Reusing `createPluginRuntimeStore` is the narrowest existing process-wide duplicate-module sharing mechanism, while a host-owned service would be a larger durability redesign rather than a prerequisite for this fix.
