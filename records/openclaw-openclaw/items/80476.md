## Summary

Kept open because current main still does not contain the requested `openai-compatible` memory embedding provider, and a same-author implementation PR is open with closing syntax for this issue. The remaining decision is whether that optional provider belongs bundled in OpenClaw or should live as an external plugin/ClawHub package.

## What This Changes

The issue requests a bundled `openai-compatible` memory embedding provider for self-hosted OpenAI-compatible embedding servers such as llama.cpp, Ollama `/v1`, vLLM, TGI, LocalAI, and llamafile.

## Best Possible Solution

Resolve the provider boundary through the linked implementation PR: if accepted, merge the provider with focused docs and tests; if not, keep the current workaround documented and route the adapter to an external plugin/ClawHub package.

## Reproduction Assessment

Not applicable as a bug reproduction because this requests a new provider adapter. Source inspection confirms current main lacks that provider and the described `local`, `lmstudio`, and `openai` alternatives remain the available paths.

## Solution Assessment

Unclear. A dedicated adapter directly targets the reported operator pain and can use existing plugin/provider seams, but bundling a new optional provider is a maintainer product decision already represented by the open implementation PR.
