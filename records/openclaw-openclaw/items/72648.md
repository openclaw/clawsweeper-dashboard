## Summary

Keep open. The PR targets the real #71581 dmScope/session-routing gap, but the latest head still regresses established exec completion and heartbeat routing contracts on current main, including trusted-system-event handling, delivery-context preservation, exec-event wake classification, and heartbeat session canonicalization/ownership checks.

## What This Changes

The PR adds a shared event-routing-key normalizer and rewrites exec notify-on-exit plus heartbeat session resolution so background exec/system events can route toward an agent main session.

## Best Possible Solution

Repair this branch narrowly: add dmScope=main single-owner routing after the existing session-key canonicalization and ownership checks, while preserving notifyOnExit background opt-in, notifyDeliveryContext, trusted:false, structured output summaries, exec-event wake reason, and coalesceMs behavior.

## Reproduction Assessment

Yes. The regressions are reproducible by static inspection of the latest PR diff against current main and by the existing targeted tests that assert notifyOnExit trust, delivery context, output payload, and exec-event heartbeat classification; I did not run tests because this review is read-only.

## Solution Assessment

No. The requested direction is valid, but this implementation rewrites too much of the existing session/exec notification contract; the safer solution is a small dmScope-aware routing seam layered onto the current canonicalization and security checks.

Full review comments:

- **[P1] Preserve heartbeat session canonicalization:** `src/infra/heartbeat-runner.ts:315-324`
  - body: The new resolver returns raw forced/configured keys and uses `store[candidateKey] || mainEntry`, removing the existing alias canonicalization and agent ownership check. A mixed-case alias or another agent's key can now select or drain the wrong heartbeat session instead of being rejected or normalized.
  - confidence: 0.95
- **[P2] Keep notify-on-exit background opt-in:** `src/agents/bash-tools.exec-runtime.ts:328`
  - body: The current guard only emits notify-on-exit events for backgrounded sessions with `notifyOnExit` enabled. This replacement only suppresses `notifyOnExit === false`, so foreground sessions or unset notify settings can enqueue system events and wake heartbeat unexpectedly.
  - confidence: 0.91
- **[P1] Use the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:362`
  - body: `ProcessSession` stores the origin route as `notifyDeliveryContext`, but this reads a loose `deliveryContext` property instead. Delayed exec completions will lose the pinned channel/topic from the run that started them and can fall back to mutable session routing.
  - confidence: 0.94
- **[P1] Wake exec completions as exec-event:** `src/agents/bash-tools.exec-runtime.ts:403-406`
  - body: The notify-on-exit wake now uses `reason: "exec-exit"`, and the immediate-wake option is misspelled as `coalesMs`. Heartbeat only classifies `exec-event` as the exec completion path and `requestHeartbeatNow` only reads `coalesceMs`, so these completions lose the exec-specific prompt path and zero-delay wake.
  - confidence: 0.96
- **[P1] Keep exec completions untrusted:** `src/agents/bash-tools.exec-runtime.ts:396-400`
  - body: The enqueue call no longer passes `trusted: false`. `enqueueSystemEvent` defaults missing trust to true, so exec completion events enter the prompt as privileged System context instead of the existing untrusted completion path.
  - confidence: 0.92
- **[P2] Preserve exec completion details:** `src/agents/bash-tools.exec-runtime.ts:365-368`
  - body: The new event text drops the exec id and output tail. Existing behavior and tests rely on those fields so the heartbeat has the completed command result context it is supposed to process.
  - confidence: 0.89
- **[P1] Read session.dmScope for routing normalization:** `src/agents/bash-tools.exec-runtime.ts:387`
  - body: The normalizer is fed `cfg.session?.scope` in the system-event paths, but the requested feature is controlled by `session.dmScope`. With the default storage scope, `dmScope=main` deployments can miss the single-owner main-session rewrite this PR is meant to add.
  - confidence: 0.9
