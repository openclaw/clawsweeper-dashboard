## Summary

Keep open: the PR targets a real dmScope/exec-event routing gap, but the current head introduces blocking exec-notification, heartbeat canonicalization, and prompt-trust regressions and still lacks real behavior proof.

## What This Changes

The PR adds a shared dmScope-based event-routing helper and rewires bash exec completion plus heartbeat session resolution to route internal events toward an agent main session.

## Best Possible Solution

Repair or replace the branch with a narrow dmScope=main single-owner routing seam that normalizes enqueue and wake keys together while preserving exec trust, delivery context, payloads, subagent suppression, and heartbeat canonicalization.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by comparing PR head 69d7317930b5cfa62b027125a42a166a1a752be1 with current main and tests: the branch removes contracts around notifyOnExit opt-in, trusted:false, delivery context, exec-event wake metadata, and heartbeat key canonicalization.

## Solution Assessment

No. The direction is valid, but this implementation rewrites too much of the existing exec and heartbeat contract; the safer solution is a small dmScope-aware routing seam layered after current canonicalization and ownership checks.

Full review comments:

- **[P1] Preserve heartbeat session canonicalization:** `src/infra/heartbeat-runner.ts:323-327`
  - body: This branch returns raw forced/configured keys and falls back to mainEntry without the current toAgentStoreSessionKey, canonicalizeMainSessionAlias, or same-agent ownership checks. A main alias or another agent's session key can now run against the wrong or missing heartbeat session instead of being normalized or rejected.
  - confidence: 0.96
- **[P2] Keep notify-on-exit behind the background opt-in:** `src/agents/bash-tools.exec-runtime.ts:330-333`
  - body: Current main only notifies after the process was backgrounded and notifyOnExit is true. This branch only suppresses notifyOnExit=false, so foreground sessions or sessions with an unset notify flag can enqueue system events and wake heartbeat on ordinary exec completion.
  - confidence: 0.93
- **[P1] Use the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:360-361`
  - body: ProcessSession stores the immutable origin route in notifyDeliveryContext, but this reads a generic deliveryContext property that the session type does not carry. Delayed exec completions will lose the pinned channel/topic and can fall back to stale session route metadata.
  - confidence: 0.95
- **[P1] Keep exec completion events untrusted:** `src/agents/bash-tools.exec-runtime.ts:389-393`
  - body: enqueueSystemEvent defaults missing trusted to true. Omitting trusted:false here makes exec completion text and command output enter the prompt as privileged System context; the same omission appears in emitExecSystemEvent.
  - confidence: 0.96
- **[P1] Wake exec completions as exec-event:** `src/agents/bash-tools.exec-runtime.ts:397-400`
  - body: Heartbeat classifies exec completions by the exec-event source/reason path, and the scheduler reads coalesceMs. This call uses exec-exit and misspells coalesceMs as coalesMs, so completions lose the exec-specific prompt path and zero-delay wake behavior.
  - confidence: 0.96
- **[P2] Preserve structured exec completion payloads:** `src/agents/bash-tools.exec-runtime.ts:364-366`
  - body: Current completion events include the exec id, exit label, and compact output tail. This replacement emits only an exit-code or signal string, so output-bearing background completions no longer carry the result payload that heartbeat prompts and tests rely on.
  - confidence: 0.9
