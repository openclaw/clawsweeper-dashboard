## Summary

Keep open. The underlying #71581 direction is valid, but this PR is still unsafe to merge because the current diff breaks existing session-routing, exec completion, and prompt-trust contracts that main and tests already protect.

## What This Changes

The PR adds a dmScope-aware event-routing helper and rewrites exec completion notifications plus heartbeat session resolution to route system-event wakes toward an agent main session.

## Best Possible Solution

Repair the branch narrowly by applying dmScope=main routing after existing canonicalization and agent-ownership checks, re-keying queue and wake paths consistently, and preserving exec payload, trust, delivery-context, and changelog requirements.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by comparing the PR diff with current main and the focused tests for notifyOnExit, trusted:false, delivery context, exec-event wake coalescing, and heartbeat session-key resolution.

## Solution Assessment

No. The requested behavior belongs in this area, but this implementation replaces too many established contracts; the safer solution is a small dmScope-aware routing seam around the existing resolver and system-event enqueue paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Preserve heartbeat session canonicalization:** `src/infra/heartbeat-runner.ts:319-327`
  - body: This returns raw forced/configured keys without toAgentStoreSessionKey, canonicalizeMainSessionAlias, or the resolveAgentIdFromSessionKey ownership check. A `main` alias, mixed-case key, or another agent's session key can now run against the wrong or missing heartbeat session.
  - confidence: 0.96
- **[P1] Keep exec completion events untrusted:** `src/agents/bash-tools.exec-runtime.ts:389-392`
  - body: enqueueSystemEvent defaults missing `trusted` to true. Omitting `trusted: false` here makes exec completion text and command output enter the prompt as privileged System context; the same omission remains in emitExecSystemEvent.
  - confidence: 0.96
- **[P1] Use the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:360-361`
  - body: ProcessSession stores the immutable origin route in `notifyDeliveryContext`, but this reads a generic `deliveryContext` property that the session does not carry. Delayed exec completions will lose the pinned channel/topic and can fall back to stale route metadata.
  - confidence: 0.95
- **[P1] Wake notify-on-exit as an exec-event:** `src/agents/bash-tools.exec-runtime.ts:397-399`
  - body: Heartbeat only classifies `exec-event` as an exec completion, and the wake scheduler reads `coalesceMs`. This call uses `exec-exit` and misspells `coalesceMs` as `coalesMs`, so completions lose the exec-specific prompt path and zero-delay wake behavior.
  - confidence: 0.96
- **[P2] Keep notify-on-exit behind the background opt-in:** `src/agents/bash-tools.exec-runtime.ts:331`
  - body: Current main only notifies after the process has been backgrounded and notifyOnExit is true. This branch only suppresses `notifyOnExit === false`, so foreground sessions with the default true setting can enqueue system events and wake heartbeat on ordinary exec completion.
  - confidence: 0.93
- **[P2] Preserve structured exec completion payloads:** `src/agents/bash-tools.exec-runtime.ts:364-366`
  - body: Current completion events include the exec id, exit label, and compact output tail. This replacement emits only a generic exit-code string, so output-bearing background completions no longer carry the result payload that heartbeat prompts and tests rely on.
  - confidence: 0.9
