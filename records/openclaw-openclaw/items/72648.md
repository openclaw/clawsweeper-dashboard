## Summary

Keep open. The linked feature request is valid, but the current PR head still introduces blocking regressions in exec completion routing, heartbeat session canonicalization, and prompt-trust handling before it can merge.

## What This Changes

The PR adds a shared event-routing-key normalizer and rewrites exec completion plus heartbeat session routing so background system events can be routed toward an agent main session for single-owner DM setups.

## Best Possible Solution

Repair this branch narrowly by adding dmScope=main single-owner routing after existing canonicalization and ownership checks, while preserving notifyOnExit background opt-in, notifyDeliveryContext, trusted:false, structured exec output summaries, exec-event wake reason, and coalesceMs behavior.

## Reproduction Assessment

Yes. Static comparison of the PR head against current main and the existing targeted tests gives a high-confidence reproduction path; I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The requested direction is reasonable, but this implementation rewrites core exec and heartbeat contracts too broadly; the safer fix is a small dmScope-aware routing seam layered onto the current session-key, trust, and delivery-context behavior.

  - confidence: 0.92
- **[P1] Use the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:360-361`
  - body: ProcessSession stores the immutable origin route in notifyDeliveryContext, but this reads an unrelated deliveryContext property. Delayed exec completions will lose the pinned channel/topic and can fall back to the session's later route.
  - confidence: 0.95
- **[P1] Keep exec completion events untrusted:** `src/agents/bash-tools.exec-runtime.ts:389-393`
  - body: enqueueSystemEvent defaults missing trusted to true. Dropping trusted:false makes exec completion text enter the prompt as privileged System context instead of the existing untrusted completion path.
  - confidence: 0.94
- **[P1] Wake exec completions as exec-event:** `src/agents/bash-tools.exec-runtime.ts:397-399`
  - body: Heartbeat only classifies reason: "exec-event" as an exec completion, and requestHeartbeatNow reads coalesceMs. This uses exec-exit and coalesMs, so completions lose the exec-specific prompt path and zero-delay wake.
  - confidence: 0.96
- **[P2] Preserve structured exec completion payloads:** `src/agents/bash-tools.exec-runtime.ts:364-366`
  - body: Current completion text includes the exec id, exit label, and output tail, which the exec-event prompt parser and tests rely on. This replacement emits only the exit code text, so output-bearing completions lose the result payload.
  - confidence: 0.9
- **[P1] Honor dmScope with effective allowlist inputs:** `src/agents/bash-tools.exec-runtime.ts:375-383`
  - body: The feature is controlled by session.dmScope and the channel's effective single-owner allowlist, but this uses a dmScope field that ProcessSession does not carry and falls back to cfg.session?.scope, the storage scope.
