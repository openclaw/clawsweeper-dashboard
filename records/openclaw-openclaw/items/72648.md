## Summary

Keep open. Current main still does not implement the linked dmScope heartbeat-routing feature, but this PR should not merge as-is because the latest head still regresses heartbeat session canonicalization, notifyOnExit opt-in behavior, exec-event trust handling, delivery context preservation, and exec-event wake classification.

## What This Changes

The PR adds a shared DM event-routing-key normalizer and rewrites exec completion plus heartbeat session resolution to route background system events toward an agent main session under single-owner DM setups.

## Best Possible Solution

Repair this branch narrowly by layering dmScope=main single-owner routing after existing canonicalization and agent-ownership checks, using effective channel/account allowlist inputs, and preserving the current exec completion trust, delivery-context, summary, and exec-event wake contracts.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by comparing the PR head to current main and existing tests: the branch removes guards and fields that current tests assert for notifyOnExit, delivery context, trusted:false, and exec-event wakes. I did not execute tests because the review was read-only.

## Solution Assessment

No. The requested direction is valid, but this implementation rewrites core exec and heartbeat contracts too broadly; the safer fix is a small dmScope-aware routing seam around the existing session-key normalization and system-event queue behavior.

- **[P1] Preserve heartbeat session canonicalization:** `src/infra/heartbeat-runner.ts:319-327`
  - body: This returns the raw forced/configured key and falls back to mainEntry without toAgentStoreSessionKey, canonicalizeMainSessionAlias, or the resolveAgentIdFromSessionKey ownership check. A forced alias like `main` or another agent's key can now run against the wrong or missing heartbeat session instead of being normalized or rejected.
  - confidence: 0.96

  - body: Current main only notifies after the process has been backgrounded and notifyOnExit is true. This branch only suppresses `notifyOnExit === false`, so foreground sessions or sessions with an unset notify flag can enqueue system events and wake heartbeat on ordinary exec completion.
  - confidence: 0.93
- **[P1] Use the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:360-361`
  - body: ProcessSession stores the immutable origin route in `notifyDeliveryContext`, but this reads a generic `deliveryContext` property that the session does not carry. Delayed exec completions will lose the pinned channel/topic and can fall back to stale session route metadata.
  - confidence: 0.95
- **[P1] Keep exec completion events untrusted:** `src/agents/bash-tools.exec-runtime.ts:388-393`
  - body: `enqueueSystemEvent` defaults missing `trusted` to true. Omitting `trusted: false` here makes exec completion text and command output enter the prompt as privileged System context; the same omission appears in `emitExecSystemEvent` below.
  - confidence: 0.96
- **[P1] Wake exec completions as exec-event:** `src/agents/bash-tools.exec-runtime.ts:396-400`
  - body: Heartbeat only classifies `exec-event` as an exec completion, and the wake scheduler reads `coalesceMs`. This call uses `exec-exit` and misspells `coalesceMs` as `coalesMs`, so completions lose the exec-specific prompt path and zero-delay wake behavior.
  - confidence: 0.96
- **[P2] Preserve structured exec completion payloads:** `src/agents/bash-tools.exec-runtime.ts:364-366`
  - body: Current completion events include the exec id, exit label, and compact output tail. This replacement emits only an exit-code string, so output-bearing background completions no longer carry the result payload that the heartbeat prompt and tests rely on.
  - confidence: 0.9
- **[P2] Use effective dmScope and allowlist inputs:** `src/agents/bash-tools.exec-runtime.ts:380-385`
  - body: The routing decision should be gated by `session.dmScope` and the effective single-owner DM allowlist for the channel/account. This code uses storage `session.scope` fallback and root/default allowlist fields, so the #71581 configuration can still miss the intended main-session rewrite or apply it inconsistently.
  - confidence: 0.88
