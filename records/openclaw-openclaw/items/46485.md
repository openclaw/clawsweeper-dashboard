## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks a plugin-side Gateway call API and related API request #39000 remains open, but exposing the raw Gateway dispatcher directly crosses a public Plugin SDK and Gateway authorization boundary without a documented capability, method allowlist, caller identity contract, docs, or complete test-helper updates.

## What This Changes

This PR exports the internal Gateway dispatcher, adds `runtime.gateway.dispatchMethod<T>(method, params)` to `PluginRuntime`, and wires a Gateway runtime fallback/options path through plugin runtime creation.

## Best Possible Solution

Define a narrow, documented Plugin SDK seam for sanitized history and inter-plugin calls, with explicit allowed methods or capabilities, plugin caller identity propagation, fallback behavior, SDK docs, typed mocks, and contract tests coordinated with #39000.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a bug report. The high-confidence static check is that current main lacks a plugin-side Gateway call API while the PR adds a broad required runtime member around the internal dispatcher.

## Solution Assessment

No. Directly exposing the internal dispatcher is not the narrowest maintainable solution; a scoped helper, capability-checked dispatcher, or purpose-built sanitized-history/inter-plugin service API would preserve the Gateway trust model better.
