## Summary

Keep open: the PR addresses a real plugin runtime API gap, but the proposed implementation exposes a broad raw Gateway dispatcher to plugins and lacks required after-fix real behavior proof.

## What This Changes

The PR exports the internal Gateway dispatcher and adds `runtime.gateway.dispatchMethod()` to the plugin runtime API.

## Best Possible Solution

Define a narrow documented plugin Gateway-call seam with explicit method policy, plugin caller attribution, SDK types/docs/tests, and then update or supersede this PR.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the API gap on current main, but this is a new plugin runtime capability rather than broken existing behavior.

## Solution Assessment

No. Exposing arbitrary `runtime.gateway.dispatchMethod()` is broader than the maintainable boundary; the safer solution is a policy-limited Gateway-call API with plugin caller attribution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Do not expose raw Gateway dispatch to plugins:** `src/gateway/server-plugins.ts:192-193`
  - body: This wires the raw dispatcher into plugin runtime, but fallback calls on this branch use a synthetic operator client with admin, approvals, and pairing scopes.
