## Summary

Keep open: current main still lacks the requested plugin Gateway caller, but this PR should not merge as-is because the exposed dispatcher misses plugin Gateway handlers, lacks an explicit caller/policy design, and has no real behavior proof.

## What This Changes

The PR exposes `api.runtime.gateway.dispatchMethod()` by exporting the Gateway dispatcher, wiring it into plugin runtime options, and adding runtime types and fallbacks.

## Best Possible Solution

Define a narrow documented plugin interop or Gateway-call seam that routes through active plugin handlers, carries plugin identity, enforces method policy, and includes SDK docs plus contract coverage.

## Reproduction Assessment

Yes for the main review finding: source inspection shows the proposed dispatcher calls `handleGatewayRequest` without `extraHandlers`, while plugin-owned Gateway methods are only attached through those handlers. I did not run tests because this was a read-only review.

## Solution Assessment

No. A raw arbitrary dispatcher is not the narrowest maintainable solution until it can reach plugin handlers and has explicit method-scope, caller-identity, audit, docs, and test contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route runtime dispatch through plugin gateway handlers:** `src/gateway/server-plugins.ts:190-192`
  - body: `dispatchGatewayMethod` still invokes `handleGatewayRequest` without `extraHandlers`, but plugin RPCs live in `pluginRegistry.gatewayHandlers` and are only attached to normal requests through those handlers. As exposed here, `runtime.gateway.dispatchMethod('stability.getState', ...)` will fail as an unknown method, so the inter-plugin use case described by the PR is not implemented.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff exposes a broad in-process Gateway RPC surface to plugins without a method policy, caller attribution, or real behavior proof.

Concerns:

- **[medium] Avoid arbitrary Gateway RPC access from plugins:** `src/gateway/server-plugins.ts:190`
  - body: Wiring `runtime.gateway.dispatchMethod` directly to an arbitrary method string lets plugin code choose any Gateway RPC. This needs an explicit capability or allowlist contract instead of relying on existing subagent-specific synthetic dispatch behavior.
  - confidence: 0.88

  - body: The proposed runtime method accepts only `method` and `params`, so Gateway policy and audit code cannot identify the calling plugin even though the existing request-scope bridge can carry plugin identity for narrower runtime helpers.
  - confidence: 0.82
