## What This Changes

The PR exports the internal Gateway dispatcher and wires `runtime.gateway.dispatchMethod()` into PluginRuntime types and runtime factory fallbacks.

## Best Possible Solution

Let https://github.com/openclaw/openclaw/issues/39000 own the API decision, then land a narrow documented plugin Gateway-call seam with handler routing, method policy, caller identity, and tests.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection shows the proposed dispatcher would call `handleGatewayRequest` without `extraHandlers`, while plugin Gateway methods live in `registry.gatewayHandlers` and the normal WebSocket path passes that table explicitly.

## Solution Assessment

No. A raw arbitrary dispatcher is not the best way to solve the issue; the safer path is a maintainer-approved SDK/API contract tracked by the canonical issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route runtime dispatch through plugin handlers:** `src/gateway/server-plugins.ts:191`
  - body: The exposed `dispatchMethod` points at `dispatchGatewayMethod`, but that helper calls `handleGatewayRequest` without `extraHandlers`.
