## Summary

Keep open for maintainer review: current main still lacks a plugin-side Gateway caller, but this PR is not merge-ready because it exposes a broad internal dispatcher without an approved plugin authorization and caller-identity contract, is stale/conflicting against main, and lacks real after-fix behavior proof.

## What This Changes

The PR exports the internal Gateway dispatcher and wires it into plugin runtimes as `runtime.gateway.dispatchMethod()` with an unavailable fallback outside Gateway contexts.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call seam with explicit method/capability policy, plugin caller attribution, SDK mocks/contract tests, docs, and coordination with https://github.com/openclaw/openclaw/issues/39000.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API PR. Source inspection gives high confidence that current main lacks the requested plugin-side Gateway caller while supporting `registerGatewayMethod`.

## Solution Assessment

No. Exposing the raw internal dispatcher is not the narrowest maintainable solution because it skips an explicit plugin authorization policy and caller-identity contract.

  - body: Adding `runtime.gateway.dispatchMethod` lets plugin code pass any Gateway method name and params into the internal dispatcher. ## Security Review

Status: needs_attention

Summary: The diff widens plugin access to the Gateway RPC bus without an explicit authorization and caller-attribution contract.

Concerns:

  - body: The proposed pass-through dispatcher does not show how generic Gateway calls preserve `pluginRuntimeOwnerId`, while current runtime ownership handling is namespace-specific. Caller attribution is needed for ownership checks and auditability.
