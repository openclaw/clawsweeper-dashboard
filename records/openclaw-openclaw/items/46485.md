## Summary

Keep open, but this patch should not merge as-is: it exposes arbitrary Gateway RPC dispatch across a security-sensitive boundary, lacks plugin caller policy/attribution, and has no after-fix real behavior proof.

## What This Changes

The PR adds `runtime.gateway.dispatchMethod()` to `PluginRuntime` and wires the existing internal Gateway dispatcher through Gateway plugin loading.

## Best Possible Solution

Define a narrow documented plugin Gateway-call seam with explicit method policy, plugin identity propagation, SDK types/docs/tests, and then update or supersede this PR.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the API gap on current main, but this is a new plugin runtime capability rather than broken existing behavior.

## Solution Assessment

No. Exposing arbitrary `runtime.gateway.dispatchMethod()` is broader than the maintainable boundary; the safer solution is a policy-limited Gateway-call API with plugin caller attribution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not expose admin-scoped gateway dispatch to plugins:** `src/gateway/server-plugins.ts:192-193`
  - body: This wires the raw dispatcher into plugin runtime, but fallback calls use a synthetic operator client that carries admin-class scopes on this branch. That would let any loaded plugin call arbitrary Gateway RPCs, including admin methods, without a method allowlist or plugin caller attribution. Please make the public seam policy-limited and identity-aware instead of passing through the generic dispatcher.
  - confidence: 0.91
