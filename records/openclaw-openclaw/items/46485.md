## Summary

Keep open: current main still lacks the requested plugin-side Gateway caller, but this PR exposes raw internal Gateway dispatch as a public runtime API without a complete authorization, attribution, docs, or test-helper contract.

## What This Changes

The PR exports the internal Gateway method dispatcher and adds `runtime.gateway.dispatchMethod()` to plugin runtimes.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call seam with explicit method/capability policy, plugin identity propagation, contract tests, docs, and changelog coverage, coordinated with #39000.

## Reproduction Assessment

Not applicable; this is a feature/API PR rather than a bug report. Source inspection is high-confidence that current main lacks `runtime.gateway`, while the PR adds a broad dispatcher around the internal Gateway method bus.

## Solution Assessment

No. Exposing the raw dispatcher is not the narrowest maintainable solution; a scoped helper or capability-checked dispatcher would better preserve the Gateway trust model.
