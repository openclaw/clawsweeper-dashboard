## Summary

Keep open: this is a real SDK feature request, but it changes the plugin/Gateway authorization boundary and lacks after-fix real behavior proof. Current main does not already expose a plugin-side Gateway caller, and the related feature issue remains open for design review.

## What This Changes

The branch adds a `runtime.gateway.dispatchMethod(method, params)` PluginRuntime surface and wires it to `dispatchGatewayMethod` during Gateway plugin loading.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call or interop seam with explicit method policy, plugin caller attribution, focused tests, and SDK docs, then update or supersede this PR.

## Reproduction Assessment

Not applicable as a feature PR: there is no broken existing contract to reproduce. Source inspection does confirm the current API gap because plugins can register Gateway methods but cannot call them through a supported plugin-side caller.

## Solution Assessment

No. Exposing the raw Gateway dispatcher is broader than the need; a safer design should constrain callable methods, preserve plugin identity for audit/policy, and document the supported contract.
