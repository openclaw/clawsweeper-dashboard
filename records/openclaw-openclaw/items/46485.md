## Summary

Keep open: the PR targets a real plugin API gap, but the current patch does not yet implement the inter-plugin Gateway method use case and broadens a security-sensitive runtime surface without a settled policy or real behavior proof.

## What This Changes

The PR adds `api.runtime.gateway.dispatchMethod` by exporting the internal Gateway dispatcher, wiring it into plugin runtime options, and adding the runtime type/fallback.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call or interop seam that routes through active plugin handlers, preserves method scopes and plugin caller identity, adds SDK/docs/contract coverage, and then supersedes or updates this PR.

## Reproduction Assessment

Yes for the main review finding: source inspection shows the proposed dispatcher would call `handleGatewayRequest` without `extraHandlers`, while plugin-owned Gateway methods are only attached through those handlers. I did not run tests because this was a read-only review.

## Solution Assessment

No. A raw arbitrary dispatcher is not the best final shape until it can reach plugin handlers and has an explicit method-scope, caller-identity, and documentation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route runtime dispatch through plugin gateway handlers:** `src/gateway/server-plugins.ts:190`
  - body: `dispatchGatewayMethod` still invokes `handleGatewayRequest` without the Gateway `extraHandlers`, but plugin RPCs live in `pluginRegistry.gatewayHandlers` and are only attached to normal requests through those handlers. As exposed here, `runtime.gateway.dispatchMethod('stability.getState', ...)` will fail as an unknown method, so the inter-plugin use case described by the PR is not implemented.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff exposes a new broad Gateway RPC authority to plugin code and needs an explicit policy before merge.

Concerns:

- **[medium] Avoid arbitrary Gateway RPC access from plugins:** `src/gateway/server-plugins.ts:323`
  - body: The dispatcher uses Gateway operator scopes for authorization, and synthetic fallback dispatch currently mints write-scoped requests for internal plugin runtime calls. Exposing this as a public arbitrary method string needs a capability or allowlist contract instead of relying on the existing subagent-specific helper behavior.
  - confidence: 0.88
- **[medium] Preserve plugin caller attribution:** `src/gateway/server-plugins.ts:319`
  - body: A generic Gateway caller needs to propagate the calling plugin id for audit and policy checks; the proposed `dispatchMethod` shape does not include caller attribution even though existing subagent cleanup paths now carry plugin ownership explicitly.
  - confidence: 0.82
