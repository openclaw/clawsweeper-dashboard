## Summary

Keep open: current main still has no public plugin-side Gateway caller, but this PR exposes arbitrary internal Gateway RPC dispatch with an admin-class fallback on the PR head and no method policy or caller attribution, so it needs SDK/security redesign rather than cleanup closure.

## What This Changes

The branch exports the internal Gateway dispatcher and adds `runtime.gateway.dispatchMethod()` to `PluginRuntime` via gateway plugin runtime options.

## Best Possible Solution

Define a narrow documented plugin interop or Gateway-call seam with explicit method policy, plugin caller attribution, SDK types/docs, and focused tests, then rework or supersede this PR.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the current API gap, but this is a new plugin runtime capability rather than broken existing behavior.

## Solution Assessment

No. Exposing arbitrary `runtime.gateway.dispatchMethod()` is broader than the maintainable boundary; the safer path is a policy-limited, identity-aware plugin Gateway-call API.
