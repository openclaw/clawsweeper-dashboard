## Summary

Keep open: current main still lacks the requested plugin Gateway caller, but this PR exposes arbitrary in-process Gateway dispatch without a settled SDK policy/caller-attribution contract and lacks real behavior proof.

## What This Changes

The PR adds `PluginRuntime.gateway.dispatchMethod`, wires it to the existing in-process gateway dispatcher, and adds a non-gateway fallback error.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call seam with explicit method policy, plugin caller attribution, SDK types/baselines, docs, and focused regression tests, then land or supersede this PR.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the requested plugin-side Gateway caller is absent on current main, so this is a feature/API gap rather than broken existing behavior.

## Solution Assessment

No for the exact proposed shape. A maintainable solution should not expose arbitrary Gateway RPC dispatch until the SDK contract, method policy, and plugin identity propagation are explicit.
