## Summary

Keep open: the PR adds a new public plugin runtime capability, but the proposed raw Gateway dispatcher crosses the Gateway authorization boundary without a method policy or plugin caller attribution, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch exports `dispatchGatewayMethod` and wires it into `PluginRuntime` as `runtime.gateway.dispatchMethod()`.

## Best Possible Solution

Define a narrow documented plugin Gateway-call or interop seam with explicit method policy, plugin caller attribution, SDK types/docs, and focused tests, then rework or supersede this PR.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection confirms the current API gap, but this is a new plugin runtime capability rather than broken existing behavior.

## Solution Assessment

No. Exposing arbitrary `runtime.gateway.dispatchMethod()` is broader than the maintainable boundary; the safer solution is a policy-limited, identity-aware plugin Gateway-call API.
