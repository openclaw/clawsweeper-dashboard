## Summary

Keep open: this PR addresses a real plugin API gap that is also tracked by #39000, but the current patch exposes a broad Gateway dispatcher without a settled capability, authorization, or caller-attribution contract.

## What This Changes

The PR exports the internal Gateway dispatcher and adds `runtime.gateway.dispatchMethod<T>(method, params)` to the plugin runtime so plugins can invoke Gateway RPCs in-process.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call or inter-plugin seam with explicit capabilities, reliable caller attribution, regression tests, docs, and changelog coverage, coordinated with #39000.

## Reproduction Assessment

Not applicable; this is a feature/API PR rather than a bug report. Static inspection is high-confidence: current main lacks `runtime.gateway`, while the PR adds a broad method-string dispatcher around the internal Gateway method bus.

## Solution Assessment

No. Directly exposing the raw internal dispatcher is broader than the likely maintainable contract; a scoped or capability-checked dispatcher, or a purpose-built sanitized history/inter-plugin API, would fit the Gateway trust model better.
