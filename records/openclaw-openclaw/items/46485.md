## Summary

Keep this PR open, but do not merge it as-is: the implementation exposes a broad public plugin runtime Gateway dispatcher without a capability/allowlist contract, plugin caller attribution, SDK test-helper updates, or docs/changelog coverage.

## What This Changes

The PR exports the internal Gateway dispatcher and adds a required `runtime.gateway.dispatchMethod<T>(method, params)` surface to plugin runtimes.

## Best Possible Solution

Define a narrow, documented plugin Gateway-call seam with explicit allowed methods or capabilities, reliable plugin caller attribution, contract tests, typed mocks, docs, and changelog coverage, coordinated with #39000.

## Reproduction Assessment

Not applicable; this is a feature/API PR rather than a bug report. The static check is high-confidence: current main lacks `runtime.gateway`, while the PR adds a broad required dispatcher around the internal Gateway method bus.

## Solution Assessment

No. Directly exposing the raw internal dispatcher is not the narrowest maintainable solution; a scoped helper, capability-checked dispatcher, or purpose-built sanitized-history/inter-plugin API would preserve the Gateway trust model better.
