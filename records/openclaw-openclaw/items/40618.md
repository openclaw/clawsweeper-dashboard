## Summary

Keep open. Current main still lacks a configurable session-memory output directory, and the live user comment confirms the proposed direction solves a real workspace collision, but this PR is not mergeable as submitted because it is stale/conflicting, regresses newer handler behavior unless rebased, creates a filesystem side effect before root-bound validation, edits localized docs that repo policy excludes, and lacks real workspace proof.

## What This Changes

The branch adds optional `hooks.internal.entries.session-memory.outputDir` handling, unit coverage, and docs so bundled session-memory snapshots can be written to a configured workspace-relative directory instead of always under `memory/`.

## Best Possible Solution

Rebase `outputDir` support onto the current handler, create configured paths only through alias-safe workspace-root helpers, drop localized docs edits, coordinate scope with https://github.com/openclaw/openclaw/issues/60034, and land only after real workspace proof.

## Reproduction Assessment

Yes, at source level. Current main still hardcodes `<workspace>/memory` in `saveSessionMemoryNow`, and the PR body plus the live user comment describe the custom-directory scenario clearly; I did not run a live gateway workspace.

## Solution Assessment

No, not as submitted. The `outputDir` direction is plausible, but this branch needs a rebase, root-safe directory creation/write handling, localized-doc removal, maintainer scope review, and real workspace proof.
