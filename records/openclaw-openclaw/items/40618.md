## Summary

Keep open. Current main still lacks configurable session-memory output directories and the discussion shows a real use case, but this branch is not safe to merge as submitted because it is stale against current handler behavior, has a filesystem safety issue, edits localized docs against policy, and lacks real behavior proof.

## What This Changes

The PR adds optional `hooks.internal.entries.session-memory.outputDir` handling, tests, and docs so session-memory snapshots can be written to a configured workspace-relative directory instead of always under `memory/`.

## Best Possible Solution

Rebase output-dir support onto the current handler, validate and create configured paths through root-aware filesystem helpers before any side effect, drop localized docs edits, and land only after maintainer config-scope review plus real workspace proof.

## Reproduction Assessment

Yes, at source level. Current main hardcodes `session-memory` writes under `<workspace>/memory`, and the PR body plus live user comment give a concrete custom-directory scenario; I did not run a live gateway workspace.

## Solution Assessment

No, not as submitted. The `outputDir` direction is plausible, but this patch should be rebased onto current behavior, avoid raw mkdir before root-bound validation, remove localized docs edits, and get maintainer scope approval.
