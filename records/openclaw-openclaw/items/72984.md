## Summary

Keep this PR open: it targets a real current-main subagent model-resolution bug and includes sufficient real behavior proof, but the current patch has a blocking alias-resolution regression after it changes the active subagent default provider, and the branch is currently conflicting with main.

## What This Changes

The PR updates subagent model selection across shared selection, gateway session resolution, reply startup, stored parent override handling, isolated cron selection, tests, and changelog text.

## Best Possible Solution

Rebase onto current main, fix the alias-index lifetime bug while preserving the subagent selector and parent-override guard, rerun the focused model/session gates, then land this as the canonical broader fix if maintainers accept the documented precedence.

## Reproduction Assessment

Yes, at source level. Current main still lets the agent primary shadow `agents.defaults.subagents.model`, lacks a subagent-aware gateway fallback, and lets parent stored overrides apply to subagent child sessions.

## Solution Assessment

No as-is. The overall direction is the right maintainable fix, but the patch must rebuild or carry the alias index with the updated subagent default provider before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Rebuild aliases after changing the subagent default:** `src/auto-reply/reply/get-reply.ts:473-477`
  - body: When this block rewrites `defaultProvider`/`defaultModel` for a subagent, `aliasIndex` remains the one built earlier from the parent defaults. Later reset, channel, and `/model` directive paths receive the new default provider but the old alias map, so aliases backed by bare `agents.defaults.models` keys can still parse under the parent provider and route subagent turns to the wrong model. Rebuild the alias index when the active default changes, or keep the active defaults and alias index together.
  - confidence: 0.84
