## Summary

Keep open. The PR targets a real subagent model-selection gap that current main still has, but the proposed fallback helpers bypass the alias-aware resolution path already used by subagent spawning, so the patch should not merge as-is.

## What This Changes

The PR changes subagent configured model precedence, adds gateway and reply-runtime fallback helpers so subagent sessions use configured subagent defaults, adds tests, and includes small config-validation and qa-lab type-export adjustments.

## Best Possible Solution

Keep the PR open and revise it so both gateway and reply-runtime fallback paths share the same alias-aware subagent model resolution contract as spawning, with regression coverage for configured aliases and a changelog entry for the user-visible precedence/runtime behavior.

## Reproduction Assessment

Yes. Configure a model alias such as `gpt` in `agents.defaults.models`, set `agents.defaults.subagents.model` to that alias, then exercise the PR's reply helper or an empty-model subagent `resolveSessionModelRef` fallback; the patch resolves `<defaultProvider>/gpt` instead of the alias target.

## Solution Assessment

No. The direction is useful, but the current patch is not the best implementation because it duplicates model parsing in two fallback paths and skips the existing alias-aware resolver used by the spawn path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Resolve aliases in the reply-time subagent default:** `src/auto-reply/reply/directive-handling.defaults.ts:56`
  - body: `resolveSubagentConfiguredModelSelection` can return a configured alias such as `gpt`; the existing spawn path resolves that through the alias index, but this helper feeds the raw value to `parseModelRef`. In that configuration a subagent can boot and persist `<defaultProvider>/gpt` instead of the configured provider/model. Reuse the alias-aware spawn resolver or a shared alias-aware helper here.
  - confidence: 0.91
- **[P2] Resolve aliases in the gateway fallback:** `src/gateway/session-utils.ts:1149-1150`
  - body: The new `resolveSessionModelRef` fallback has the same alias gap when `entry.model` is empty: a configured subagent alias is parsed as a bare model under the default provider rather than resolved through the alias index. That makes the race fallback disagree with the spawn-time model selection already covered by current tests.
  - confidence: 0.89
