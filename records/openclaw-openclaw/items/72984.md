## Summary

Keep open. Current main supersedes the PR's write-side clobber concern by persisting child-session model overrides before the child run starts, but main still has the old subagent-default precedence and lacks the PR's defensive fallback work. The PR is not mergeable as-is because both new fallback paths bypass the existing alias-aware model resolver, and the user-facing model behavior change needs a changelog entry while overlapping #58822/#58823.

## What This Changes

The PR changes subagent model precedence, adds gateway and reply-runtime fallback helpers/tests for subagent default model resolution, includes config-validation cleanup, and makes the qa-lab Command export type-only.

## Best Possible Solution

Keep the current-main child-session modelOverride persistence behavior, repair this PR's remaining defensive fallback work to share the alias-aware model resolver, and coordinate the subagent-default precedence change with #58822/#58823 so native sessions and isolated cron semantics stay consistent.

## Reproduction Assessment

Yes. Configure agents.defaults.models with an alias such as gpt, set a subagent default to that alias, then exercise the PR's new reply helper or empty-model gateway fallback; the PR parses the alias as <defaultProvider>/gpt instead of the configured provider/model target.

## Solution Assessment

No. The direction is useful, but the current patch is not the best solution because it bypasses the existing alias-aware resolver and leaves overlapping precedence work uncoordinated with the open canonical precedence PR.

  - body: `resolveSubagentConfiguredModelSelection` can return a configured alias such as `gpt`; the existing spawn path resolves that through the alias index, but this helper feeds the raw value to `parseModelRef`. That makes the race fallback disagree with the spawn-time model selection already covered by current tests.
  - confidence: 0.89
