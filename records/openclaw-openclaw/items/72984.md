## Summary

Keep this PR open: it fixes the documented subagent-default precedence in several places, but the reply-runtime path can still let a parent stored model override replace the configured subagent default.

## What This Changes

The PR changes subagent model selection so configured subagent defaults win over parent primary models across gateway session resolution, reply runtime defaults, isolated cron selection, tests, and the changelog.

## Best Possible Solution

Keep the unified subagent-default precedence, but make the reply-runtime selection immune to inherited parent stored overrides when a configured subagent default applies.

## Reproduction Assessment

Yes. Current main docs and source show the precedence bug, and the PR body supplies a concrete local `sessions_spawn` proof; the remaining PR blocker is source-reproducible because the subagent default is applied before parent stored override inheritance.

## Solution Assessment

No. The shared selector and alias-aware fallback direction is maintainable, but the reply-runtime path still needs to run the subagent default after parent override inheritance or suppress parent inheritance for configured subagent-default sessions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep parent overrides from replacing subagent defaults:** `src/auto-reply/reply/get-reply.ts:404-416`
  - body: The new subagent default is applied before the stored parent override pass. For a subagent entry without its own `providerOverride`/`modelOverride`, `resolveStoredModelOverride` still inherits the parent session override and later replaces the configured subagent default, so this edge still runs on the parent model. Apply the subagent-default selection after that pass or skip parent inheritance when a configured subagent default applies, and add a regression for that shape.
