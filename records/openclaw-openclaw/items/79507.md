## Summary

Keep this issue open: current main has some cron/session corpus filters, but it still lacks a provenance-aware way to suppress runtime-only cron delivery-report turns and their paired assistant report, and the linked fixing PR is still open and unmerged.

## What This Changes

The issue asks Dreaming/memory-core to stop promoting runtime-generated cron delivery reports and one-way system announcements into MEMORY.md.

## Best Possible Solution

Land a rebased provenance-aware fix equivalent to https://github.com/openclaw/openclaw/pull/79547 for runtime-only delivery reports, while keeping broader configurable Dreaming exclusions with https://github.com/openclaw/openclaw/issues/72611.

## Reproduction Assessment

Yes, at source level: current main can still represent runtime-only events with the visible marker prompt, memory-host export lacks a provenance-aware paired assistant-report skip, and Dreaming consumes that export. I did not run a live cron plus Dreaming sweep in this read-only review.

## Solution Assessment

Yes for the narrow bug direction: provenance-aware filtering is the maintainable fix because it can distinguish runtime-generated delivery reports from a real user typing similar text. A new operator config surface is broader and should stay with the separate configurable-exclusions issue.
