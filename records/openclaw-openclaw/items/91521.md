## Summary

Keep open: the reported Tailscale Serve + token-mode Control UI media failure still has a clear current-source path, and current main does not appear to have changed the assistant-media auth behavior since v2026.6.1.

## What This Changes

The issue requests that Control UI chat render generated local assistant images when accessed through Tailscale Serve in token mode with `gateway.auth.allowTailscale` enabled.

## Best Possible Solution

Keep the issue open until maintainers choose a narrow auth design that lets verified Tailscale Control UI sessions fetch assistant media without opening unrelated HTTP APIs, then cover it with focused gateway and UI tests plus a live Tailscale proof if feasible.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not with live runtime proof yet; source inspection shows that tokenless Tailscale Control UI sessions can authenticate the WebSocket while assistant-media metadata still requires HTTP bearer/device/trusted-proxy auth.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the auth boundary; the best fix should be scoped to Control-UI-owned media/avatar reads or operator device-token issuance, not a broad HTTP Tailscale bypass.
