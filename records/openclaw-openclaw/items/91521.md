## Label Justifications

- `P2`: The issue affects generated media rendering for a specific Control UI Tailscale Serve setup without making the core runtime unusable.
- `impact:message-loss`: Generated image content reaches the session but cannot render in the Control UI chat surface.
- `impact:auth-provider`: The failure is an Unauthorized response at the gateway auth boundary under token mode plus Tailscale Serve.

## Summary

Keep open: current main and v2026.6.8 still leave tokenless Tailscale Control UI assistant-media metadata on the normal HTTP auth path, and the Tailscale Control UI pairing skip does not create the paired-device record needed to mint a device token for an unpaired browser. The remaining fix is auth-boundary-sensitive and needs maintainer/security direction.

## What This Changes

The issue reports that Control UI token mode with Tailscale Serve lets the UI connect, but assistant-media requests return Unauthorized so generated images do not render in chat.

## Best Possible Solution

Give verified Tailscale Control UI sessions a narrow scoped credential or media-ticket bootstrap for assistant-media metadata while keeping Tailscale identity-header auth out of general HTTP APIs.

## Reproduction Assessment

No live Tailscale Serve reproduction was run. Source inspection gives a high-confidence reproduction path: tokenless Tailscale WebSocket auth can open Control UI, but assistant-media metadata uses HTTP auth and has no bearer credential when the browser has not received a paired device token.

## Solution Assessment

Unclear until maintainers choose the security boundary. The reporter's trusted-proxy workaround is valid, but the in-core fix should be narrower than accepting Tailscale identity for all HTTP routes.
