## Label Justifications

- `P2`: The issue affects generated media rendering for a specific Tailscale Serve plus token-auth Control UI setup.
- `impact:message-loss`: Generated image content is present but fails to render in the Control UI chat surface.
- `impact:auth-provider`: The observed Unauthorized response is caused by the gateway auth boundary under token mode and Tailscale Serve.

## Summary

Keep open: current main and the latest release still require normal HTTP auth for assistant-media ticket minting, while Tailscale identity auth remains WebSocket-only, so the reported tokenless Tailscale Serve media failure remains source-reproducible and auth-sensitive.

## What This Changes

The issue requests that generated assistant images render in the Control UI when accessed through Tailscale Serve under token auth with `allowTailscale` enabled.

## Best Possible Solution

Add a narrow, security-reviewed credential or ticket path for verified Tailscale Control UI media access without enabling Tailscale identity-header auth for general HTTP APIs.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection still makes the path reproducible: assistant-media meta requests require HTTP auth, while Tailscale identity-header auth remains WebSocket-only.

## Solution Assessment

Unclear until maintainers choose the security boundary. The reporter's trusted-proxy workaround is valid, but the in-core fix should be narrower than accepting Tailscale identity for all HTTP routes.
