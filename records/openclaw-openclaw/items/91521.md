## Summary

Keep open: current main still routes Control UI assistant media through HTTP auth while Tailscale identity-header auth is intentionally limited to the Control UI WebSocket surface, so the reported tokenless Tailscale Serve media failure is still plausible and not covered by the existing Tailscale service-name change.

## What This Changes

The issue reports that Control UI in token auth mode can be opened through Tailscale Serve, but generated assistant images under `/__openclaw__/assistant-media` return Unauthorized and render as unavailable.

## Best Possible Solution

Add a narrow, security-reviewed Control UI media/avatar credential path for verified Tailscale Control UI sessions, with live Tailscale proof and focused regression tests, without enabling Tailscale identity-header auth for general HTTP APIs.

## Reproduction Assessment

No live high-confidence reproduction was run, but the current-source path is source-reproducible: assistant media is an HTTP route that needs bearer/query token, device token, trusted-proxy auth, or a prior media ticket, while Tailscale identity-header auth is WebSocket-only.

## Solution Assessment

Unclear until maintainers choose the security boundary. The reporter's trusted-proxy workaround is valid, but the best in-core fix should be narrower than enabling Tailscale identity headers for all HTTP routes.
