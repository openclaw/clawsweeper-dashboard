## Label Justifications

- `P2`: The bug affects generated image visibility in a specific Tailscale Serve plus token-mode Control UI setup, but it is not a core outage.
- `impact:message-loss`: Assistant-generated image attachments are effectively suppressed in the Control UI chat when the media route returns unauthorized.
- `impact:auth-provider`: The root cause is the mismatch between gateway token mode, Tailscale identity auth, and Control UI HTTP media authorization.
- `impact:security`: The fix must preserve authorization boundaries around local assistant media and avoid leaking long-lived shared secrets in URLs.

## Summary

Keep open: current main and v2026.6.9 still keep Tailscale tokenless auth on the Control UI websocket surface only, while assistant-media uses HTTP read auth, so the reported Tailscale Serve token-mode media failure remains real and security-sensitive.

## What This Changes

The issue requests that Control UI assistant media images load when accessed through Tailscale Serve with default token-mode gateway auth.

## Best Possible Solution

Define a safe Control UI media-auth path for Tailscale Serve, preferably scoped media tickets from the authenticated Control UI session or a narrowly authorized Tailscale HTTP read path, without relying on long-lived `?token=` URLs.

## Reproduction Assessment

Yes at source level. Tailscale tokenless auth is explicitly websocket-only, while assistant-media metadata/file reads use HTTP auth and require a bearer/query token or preexisting media ticket, so a tokenless Tailscale Control UI session cannot mint the ticket.

## Solution Assessment

No final fix is selected yet. The issue is valid, but simply enabling query-token media auth is not the best solution because it preserves long-lived URL secrets; maintainers should choose a scoped ticket/session or narrowly reviewed Tailscale HTTP read design.
