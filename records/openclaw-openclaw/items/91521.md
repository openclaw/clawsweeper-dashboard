## Summary

Keep open: current main and v2026.6.9 still require HTTP read auth for assistant-media metadata while Tailscale Serve identity only authorizes the Control UI WebSocket path, so tokenless Tailscale sessions can open chat but cannot mint media tickets for generated images. The linked open PR https://github.com/openclaw/openclaw/pull/92584 is a partial-overlap hardening change for query-token auth, not a fix for this Tailscale media gap.

## What This Changes

Fix Tailscale Serve token-mode Control UI so generated local images render instead of failing with unauthorized assistant-media requests.

## Best Possible Solution

Give verified Tailscale Control UI sessions a narrow scoped way to request assistant-media metadata or media tickets, or document that shared-secret/trusted-proxy HTTP auth is required for local media previews; do not broadly enable Tailscale identity for all HTTP APIs.

## Reproduction Assessment

Yes at source level: tokenless Tailscale Serve can authorize the Control UI WebSocket, but assistant-media metadata uses HTTP auth and the UI sends a bearer header only when it has a token-like credential. A live Tailscale Serve repro was not run in this read-only review.

## Solution Assessment

No broad HTTP-auth change is safe as the default fix. The best solution is a narrow maintainer-approved media credential or ticket path for verified Tailscale Control UI sessions, with tests for the existing HTTP boundary.
