- `impact:message-loss`: Generated image content reaches the session but cannot render in the Control UI chat surface.

## Summary

Keep open: current main and the latest release still keep tokenless Tailscale Serve auth on the Control UI WebSocket surface while assistant-media metadata remains an HTTP read-auth route, so the reported image-rendering failure is still source-reproducible and security-sensitive.

## What This Changes

The issue reports that Control UI token-mode access through Tailscale Serve can load the UI while assistant-media image requests return Unauthorized.

## Best Possible Solution

Give verified Tailscale Control UI sessions a narrow scoped way to obtain assistant-media metadata/media tickets without making Tailscale identity a general HTTP API credential.

## Reproduction Assessment

Yes at source level, but not live: tokenless Tailscale WebSocket auth can open Control UI, while assistant-media metadata is an HTTP route and the UI sends bearer auth only when a token-like credential exists.

## Solution Assessment

No automatic broad HTTP-auth change is safe here; the reporter's trusted-proxy workaround is valid, but the product fix should be a narrower maintainer-approved media credential or ticket path.
