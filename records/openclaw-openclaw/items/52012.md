## Summary

Keep open. Current main does not implement compaction-specific `modelFallbacks`, and this PR still contains blocking regressions from a large stale rewrite of the direct compaction path: it drops current stream/auth wrapping, checkpoint persistence, workspace-scoped auth resolution, and small-context reserve capping.

## What This Changes

The PR adds an `agents.defaults.compaction.modelFallbacks` config field and changes direct Pi compaction to try fallback models during model resolution, auth, and runtime compaction failures.

## Best Possible Solution

Rebase and reduce the change to an additive fallback resolver/retry layer on top of current main's compaction helpers, preserving existing stream preparation, workspace auth scope, settings budget capping, checkpoint/rotation behavior, provider prompt transforms, and hooks; then add focused tests for resolution/auth/runtime fallback plus docs for the new config option.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by comparing PR head `compact.ts` with current main: provider-owned streams are assigned without the current wrapper, fallback compaction metrics read the original session after `fbSession.compact()`, and checkpoint persistence is absent from the PR success path.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best implementation because it achieves fallback retries by carrying an outdated copy of the direct compaction flow; the safer path is a smaller current-main patch that reuses the existing compaction session preparation and post-success pipeline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Wrap provider streams before assigning compaction streams:** `src/agents/pi-embedded-runner/compact.ts:958-960`
  - body: The PR assigns `providerStreamFn` directly to `session.agent.streamFn`. Current compaction goes through `resolveEmbeddedAgentStreamFn`, which injects resolved runtime credentials and transport wrappers; bypassing it can make provider-owned or OAuth-backed compaction calls fail even after auth prep succeeds.
  - confidence: 0.93
- **[P1] Persist checkpoints after successful compaction:** `src/agents/pi-embedded-runner/compact.ts:1238-1242`
  - body: The success path now runs post-compaction side effects and returns without calling `persistSessionCompactionCheckpoint`. The legacy context engine delegates into this direct path, so successful compactions would stop producing checkpoints for compaction list/restore workflows.
  - confidence: 0.91
- **[P1] Pass the context window into prepared compaction settings:** `src/agents/pi-embedded-runner/compact.ts:906-910`
  - body: `createPreparedEmbeddedPiSettingsManager` is now called without `contextTokenBudget`. The reserve-floor safety cap in `src/agents/pi-settings.ts` only runs when that value is present, so small-context compaction models can inherit an oversized reserve floor and repeatedly overflow instead of compacting.
  - confidence: 0.9
- **[P2] Use the compacted fallback session for metrics and hooks:** `src/agents/pi-embedded-runner/compact.ts:1244-1251`
  - body: When primary `session.compact()` fails and a fallback `fbSession.compact()` succeeds, the code disposes the fallback session and then computes `tokensAfter`, `messageCountAfter`, diagnostics, and hooks from the original `session.messages`. That reports pre-compaction state and can trigger incorrect follow-up compaction decisions.
  - confidence: 0.89
- **[P2] Keep workspace auth scope in compaction auth resolution:** `src/agents/pi-embedded-runner/compact.ts:431-436`
  - body: The new `tryAuthForModel` helper omits `workspaceDir` when calling `getApiKeyForModel`, although current main passes the resolved workspace. Workspace-scoped provider auth evidence and env/secret resolution are therefore invisible for primary and fallback compaction models.
  - confidence: 0.86
