## Label Justifications

- `P1`: This blocks the Windows Claude CLI agent runtime path for users whose Claude install is exposed as an npm `.cmd` shim.
- `impact:auth-provider`: The affected behavior is provider/runtime routing for the `claude-cli` backend, causing configured Claude CLI model runs to fail before execution.

## Summary

Keep open: current main and the latest release still route the bundled Claude CLI backend through the supervisor child-process path, but that path only appends Windows `.cmd` shims for npm/pnpm/yarn/npx and does not wrap `.cmd`/`.bat` execution. The report is source-reproducible, but the suggested `shell: true` fix needs care because existing exec code explicitly avoids shell-based argv execution for injection hardening.

## What This Changes

The issue requests Windows-safe spawning for the bundled Claude CLI backend when `claude` resolves to an npm-style `.cmd` shim.

## Best Possible Solution

Use a shared, safe Windows batch-command invocation for supervisor child spawns, cover both one-shot and live Claude CLI backend paths, and keep argv/shell hardening intact.

## Reproduction Assessment

Yes from source inspection, but not by live Windows execution. Configure the bundled `claude-cli` backend on Windows with only `claude.cmd` on PATH and launch a CLI-backed turn; current main sends `claude` through a child adapter that neither maps it to `claude.cmd` nor wraps `.cmd` execution.

## Solution Assessment

No, not exactly as suggested. Adding `claude` to shim resolution is necessary, but the maintainable fix should avoid generic `shell: true` and instead share the safer Windows `cmd.exe` wrapper or equivalent shim materialization used by the exec path.
