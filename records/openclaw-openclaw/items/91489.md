## Summary

Current main and the latest release still leave the Windows Claude CLI supervisor path source-reproducible. This should stay open because it is paired with open implementation PRs that reference the same defect; the next maintainer action is to choose and land the canonical fix rather than close the issue during cleanup.

## What This Changes

The issue reports that Windows spawning of the bundled Claude CLI fails because the supervisor does not resolve bare `claude` to a Windows shim and lacks safe `.cmd`/`.bat` handling.

## Best Possible Solution

Land a narrow Windows-safe supervisor spawn fix with regression coverage and Windows proof, preferably reusing the existing no-shell wrapper resolution pattern rather than adding a broad shell toggle without maintainer approval.

## Reproduction Assessment

Yes, at source level: current main passes the bundled `claude` command into the supervisor child adapter, and the adapter's Windows shim allowlist omits `claude`. I did not run a live Windows repro in this read-only review.

## Solution Assessment

Unclear which linked PR is best as-is: adding `claude` addresses the ENOENT half, but the requested `shell: true` path needs security-conscious Windows proof because current process code intentionally avoids shell-mediated argv execution.
