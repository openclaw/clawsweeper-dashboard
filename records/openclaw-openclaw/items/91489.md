## Label Justifications

- `P1`: The defect makes the bundled Claude CLI provider unusable for affected Windows users before model execution begins.
- `impact:auth-provider`: The selected provider route cannot launch its configured CLI executable, so provider/model execution is blocked.

## Summary

The issue remains necessary: current main still sends the bundled Claude CLI’s bare command through a Windows supervisor whitelist that omits it, matching the reported provider-blocking failure. Two open PRs are attempting the repair, so the issue should remain open until one lands; the discussion usefully identifies Gemini as the same sibling surface, but the final fix must satisfy both Node’s batch-file execution contract and the supervisor’s command-injection boundary.

## What This Changes

Fix Windows supervisor spawning so npm-installed Claude CLI shims resolve and launch correctly.

## Best Possible Solution

Use one generic, narrowly scoped Windows batch-shim launch path for bundled CLI backends: resolve Claude and Gemini npm shims, invoke them through an explicitly controlled and correctly quoted Windows command wrapper rather than broad shell mode, add supervisor-level metacharacter and sibling-backend regression coverage, and prove a real Claude turn on supported Windows/Node.

## Reproduction Assessment

Yes. Current main passes bare `claude` from the bundled backend into a supervisor whitelist that does not resolve it, and the reported ENOENT/EINVAL sequence matches Node’s Windows batch-file execution contract; a live current-main run was not performed in this Linux checkout.

## Solution Assessment

No, the issue’s two-part suggestion is not yet the best complete fix: adding Claude resolution is necessary, but broad `shell: true` is unsafe, while suffix-only resolution lacks proof against direct `.cmd` EINVAL. The narrow maintainable solution is an explicit quoted Windows batch-shim wrapper covering Claude and Gemini.
