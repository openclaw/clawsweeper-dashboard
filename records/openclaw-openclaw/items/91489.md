## Summary

The current main tree and latest release still leave the Windows supervisor path source-reproducible: bundled Claude runs pass bare `claude` into the supervisor, and the supervisor shim whitelist does not include it. The issue should stay open because it is already paired with open implementation PRs that need maintainer review rather than another cleanup close.

## What This Changes

The issue requests Windows-safe spawning for npm-installed Claude CLI shims so OpenClaw does not fail with ENOENT/EINVAL when launching the Claude provider.

## Best Possible Solution

Land a narrow supervisor Windows shim fix for the shipped Claude CLI path, with regression coverage and a maintainer-approved answer on `.cmd` execution that avoids unsafe shell parsing unless explicitly justified.

## Reproduction Assessment

Yes, at source level: bundled Claude CLI execution passes bare `claude` to the supervisor and current shim resolution does not append `.cmd` for that command. I did not run a live Windows machine in this read-only review.

## Solution Assessment

Unclear which linked PR is best as-is: adding `claude` to the supervisor shim list directly addresses ENOENT, but the broader `shell:true` proposal conflicts with existing shell-avoidance patterns unless maintainers intentionally accept that risk.
