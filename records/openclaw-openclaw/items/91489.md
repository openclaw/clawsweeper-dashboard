## Summary

Keep open: current main and the latest release still leave the supervisor Windows shim whitelist without `claude`, and this issue is already paired with open fix PRs that maintainers should review rather than closing the report early.

## What This Changes

Windows Claude CLI provider spawning fails because the process supervisor passes bare `claude` on Windows without resolving the npm `.cmd` shim.

## Best Possible Solution

Land a narrow supervisor Windows command-resolution fix for shipped CLI backends with regression coverage and Windows after-fix proof, then close this issue and the duplicate report.

## Reproduction Assessment

Yes, at source level: current main passes the Anthropic backend's bare `claude` command into the supervisor, and the Windows shim resolver does not append `.cmd` for that command. I did not run a live Windows reproduction in this read-only review.

## Solution Assessment

Partly: adding `claude` to the supervisor shim list directly addresses the reported ENOENT, but the suggested `shell: true` shape needs maintainer scrutiny because the sibling process helper avoids shell mode for argv-based execution.
