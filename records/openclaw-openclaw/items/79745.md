## Summary

Keep open: this active PR targets a real QMD/mcporter failure, but the submitted fix replaces the configured mcporter server definition and still lacks real after-fix behavior proof from a QMD/mcporter setup.

## What This Changes

The branch writes an agent-scoped mcporter config for QMD calls, appends `--config` to mcporter invocations, parses JSON after daemon log noise, adds qmd-manager tests, and updates the changelog.

## Best Possible Solution

Preserve configured mcporter server definitions while isolating qmd-only state, then require redacted real multi-agent QMD/mcporter proof before merge.

## Reproduction Assessment

Yes, from source inspection and the mcporter v0.10.2 contract: current main sends qmd-scoped XDG env to mcporter, and this PR's explicit generated config would bypass configured mcporter server definitions. I did not run a live gateway/QMD daemon reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Passing an explicit per-agent config is a plausible isolation handle, but the generated single-server config must preserve or intentionally migrate existing custom mcporter setups before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the configured mcporter server:** `extensions/memory-core/src/memory/qmd-manager.ts:2057-2058`
  - body: This always writes an explicit mcporter config containing only a local `qmd` stdio server, then passes it via `--config`. mcporter uses only that file when `--config` is explicit, while OpenClaw's current contract lets `serverName` point at a configured custom target, including remote qmd MCP setups. Those users will have their server definition ignored and replaced by `this.qmd.command`, so `memory_search` can regress for the setup described in https://github.com/openclaw/openclaw/issues/79847.
