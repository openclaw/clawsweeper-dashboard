## Summary

Keep open: this PR targets a real QMD/mcporter integration bug, but the latest branch still has a blocking code-path issue where configured stdio qmd servers can lose mcporter keep-alive lifecycle in the generated agent config.

## What This Changes

The branch makes memory-core QMD mcporter calls use agent-scoped mcporter configs, preserve configured server definitions, strip QMD XDG env from mcporter spawns, parse noisy JSON output, add tests, and update the changelog.

## Best Possible Solution

Keep the agent-scoped config and cleaned mcporter environment approach, but reapply keep-alive for stdio qmd entries when mcporter's serialized config output lacks lifecycle and add focused coverage for that exact output shape.

## Reproduction Assessment

Yes. Source inspection shows current main passes qmd-scoped XDG variables into mcporter, and the PR finding is reproducible from mcporter 0.10.2 source because config get JSON omits lifecycle while qmd is not default keep-alive.

## Solution Assessment

No, not as submitted. Passing an explicit per-agent mcporter config and stripping QMD XDG vars is the right direction, but configured stdio qmd servers must remain daemon-capable before merge.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Restore keep-alive for serialized qmd servers:** `extensions/memory-core/src/memory/qmd-manager.ts:2133-2136`

Reapply lifecycle for stdio qmd entries when it is absent while preserving the configured fields.
