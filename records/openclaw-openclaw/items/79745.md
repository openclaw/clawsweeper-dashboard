## Summary

Keep this PR open: the agent-scoped mcporter config direction matches the current source problem and mcporter 0.10 config discovery, and the later hclsys comment positively corroborates the approach, but the patch is not merge-ready because exact-head lint fails and real QMD/mcporter behavior proof is still missing.

## What This Changes

The PR writes an agent-scoped QMD mcporter config, passes `--config` to mcporter calls, keys daemon startup by that config path, parses noisy mcporter JSON output, adds qmd-manager tests, and updates the changelog.

## Best Possible Solution

Land a narrow version after the lint fix, redacted real multi-agent QMD/mcporter proof, and green exact-head checks; treat stripping qmd XDG vars from mcporter's own spawn env as optional defense-in-depth unless maintainers want it rolled in now.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-reproducible: current main scopes QMD state per agent but invokes mcporter with shared qmd XDG env and no explicit `--config`, while mcporter 0.10 resolves config through XDG when no explicit config is supplied; hclsys also reported an independent reproduction. I did not run a live daemon setup in this read-only pass.

## Solution Assessment

Is this the best way to solve the issue? Mostly yes, but not merge-ready: explicit agent-scoped `--config` short-circuits mcporter's XDG config discovery and keys daemons by config path, which is a narrow maintainable fix. It still needs the lint fix, real behavior proof, and green checks before merge; stripping qmd XDG vars from mcporter's own spawn env is a reasonable follow-up or small roll-in.
