## Best Possible Solution

Keep the generated per-agent mcporter config path for unauthenticated/local QMD servers, keep env-bearing or authenticated mcporter servers on the original external config, and add regression coverage for refresh-auth and query-token remote definitions before merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep refresh-auth remotes external:** `extensions/memory-core/src/memory/qmd-manager.ts:193-198`
  - body: `mcporter config get --json` serializes refreshable bearer auth as a top-level `refresh` object, but this predicate only treats keys containing auth/header/secret/token as auth material and only detects URL username/password credentials. A remote QMD server configured with mcporter refresh auth, or credentials carried in URL query parameters, will fall through to generated mode and be written into OpenClaw's per-agent `mcporter.json`, despite the nearby comment saying authenticated remotes should keep using the original mcporter config. Treat `refresh` and credential-bearing URL parameters as auth material, with focused tests, before merging.
  - confidence: 0.86
