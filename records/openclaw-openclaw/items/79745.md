## Summary

Keep open: the PR addresses a source-proven QMD/mcporter XDG leak and includes useful terminal proof, but the latest diff still creates a new agent-scoped mcporter config that can duplicate configured stdio server env/auth material into OpenClaw state. That security and compatibility policy needs resolution before merge.

## What This Changes

The PR changes memory-core QMD/mcporter runtime and tests to generate per-agent mcporter configs, split qmd versus mcporter process envs, preserve configured server definitions, parse log-prefixed JSON, and wait for one agent MCP test log asynchronously.

## Best Possible Solution

Keep the per-agent --config isolation, but land only after configured stdio env/auth preservation is made non-duplicating or explicitly approved with focused upgrade and secret-bearing-config coverage.

## Reproduction Assessment

Yes, by source inspection: current main passes qmd-scoped XDG_CONFIG_HOME/XDG_CACHE_HOME to mcporter, and upstream mcporter 0.10 uses XDG_CONFIG_HOME for config discovery. The linked report and PR proof add real-world confirmation, but I did not rerun the live mcporter setup locally.

## Solution Assessment

No, not as written. The --config isolation and mcporter/qmd env split are the right shape, but configured stdio server env/auth material needs a safer preservation model before this is the best fix.
