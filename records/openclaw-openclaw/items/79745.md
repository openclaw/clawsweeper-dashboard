## Summary

Keep this PR open but do not merge it yet: the QMD/mcporter isolation work is still useful, but the current head rewrites the generated mcporter config before daemon startup and again before each mcporter call, which can invalidate mcporter 0.10.x daemon freshness checks.

## What This Changes

The branch updates memory-core QMD mcporter search to use agent-scoped generated mcporter configs, preserve configured servers, strip qmd XDG env from mcporter process env, parse log-prefixed JSON, and add qmd-manager regressions.

## Best Possible Solution

Keep the per-agent explicit mcporter config and clean process env approach, but make generated mcporter.json writes content-idempotent and add a repeated-search mtime regression before merge.

## Reproduction Assessment

Yes. Source inspection plus the related reports provide a high-confidence path: current main leaks qmd-scoped XDG env to mcporter and raw-parses log-prefixed stdout, while the PR head's remaining mtime issue follows from mcporter 0.10.2 config freshness logic.

## Solution Assessment

No, not as written. The explicit per-agent config and clean mcporter env are the right direction, but the config writer must skip unchanged content so mcporter's daemon freshness metadata is not invalidated by OpenClaw itself.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid rewriting unchanged mcporter config:** `extensions/memory-core/src/memory/qmd-manager.ts:2076`
  - body: `ensureMcporterConfig()` writes `mcporter.json` every time, and both daemon startup and each mcporter call route through it. mcporter v0.10.2 compares config layer mtimes with daemon metadata, so rewriting identical content after `daemon start` can make the just-started per-agent daemon stale on the following search call. Please skip the write when the generated content is unchanged and add a repeated-search mtime regression.
  - confidence: 0.9
