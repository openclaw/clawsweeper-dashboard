## Summary

Keep open. The PR fixes a real QMD/mcporter XDG leak, but current head still routes documented absolute-path QMD stdio servers through the external mcporter config path, dropping the generated per-agent config and QMD env that provide isolation. The supplied real proof also conflicts with that code path, so the PR needs a fix and fresh proof before merge.

## What This Changes

The PR rewrites memory-core QMD mcporter handling to generate per-agent mcporter configs, split QMD and mcporter environments, preserve selected configured servers externally, parse log-prefixed mcporter JSON, and expand focused tests.

## Best Possible Solution

Keep the generated per-agent mcporter config path for uncustomized QMD stdio servers, including documented absolute or basename qmd executables, and leave only clearly user-owned auth/custom wrappers external with explicit maintainer-approved tradeoff and fresh real proof.

## Reproduction Assessment

Yes, by source inspection: a configured stdio server with command "/absolute/path/to/qmd" and args ["mcp"] is supported by docs and shown in the PR proof, but current head rejects it from generated mode and then runs mcporter without the generated config or QMD env.

## Solution Assessment

No. The generated per-agent config approach is the right direction for uncustomized qmd stdio, but the current classifier is too narrow and the remaining external configured-server behavior needs an explicit maintainer tradeoff plus proof.
