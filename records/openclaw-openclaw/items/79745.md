## Summary

Keep this PR open, but not mergeable as submitted. The related issue and follow-up comments support using an explicit mcporter config as a way to bypass mcporter 0.10.x XDG config discovery, but this diff also replaces the user's configured mcporter server target with a generated local `qmd mcp` entry and still lacks after-fix real QMD/mcporter proof.

## What This Changes

The PR writes an agent-scoped mcporter config, passes it via `--config`, keys daemon starts by config path, tolerates noisy mcporter JSON output, adds QMD manager tests, and updates the changelog.

## Best Possible Solution

Keep the fix focused on isolating per-agent QMD state while preserving configured mcporter server definitions, then require redacted real mcporter 0.10.x `memory_search` proof before merge.

## Reproduction Assessment

Yes, from source inspection and dependency contract: current main passes qmd-scoped XDG variables into `runMcporter`, and mcporter v0.10.2 uses explicit `--config` and XDG-aware config discovery. I did not run the live gateway/QMD daemon scenario in this read-only review.

## Solution Assessment

No, not as submitted. Passing an agent-scoped `--config` is a plausible isolation mechanism, but the generated single-server config must preserve or explicitly replace existing custom mcporter setups before this is the narrow maintainable fix.
