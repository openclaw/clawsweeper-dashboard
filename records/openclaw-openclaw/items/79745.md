## Summary

Keep open: current head still has P1 merge blockers. The PR fixes useful QMD/mcporter paths, but configured stdio qmd servers can now run through the original mcporter config without OpenClaw's per-agent QMD env, remote `env` blocks can still be copied into OpenClaw-managed config, and the PR-body real proof no longer matches the current head behavior.

## What This Changes

The PR rewrites memory-core QMD mcporter handling to add per-agent generated mcporter configs and daemon keys, externalize some configured mcporter servers, strip QMD XDG env from mcporter, parse log-prefixed JSON, and expand focused tests plus one MCP-runtime test wait.

## Best Possible Solution

Merge only after configured stdio qmd servers either keep per-agent QMD state or have an explicit maintainer-approved external-config tradeoff, and after remote env/auth material is never copied into OpenClaw-managed mcporter config.

## Reproduction Assessment

Yes, from source inspection: current head classifies all configured stdio servers as external and its tests assert no generated config, while mcporter stdio children inherit the mcporter process env. I did not run a live QMD/mcporter setup in this read-only review.

## Solution Assessment

No. The generated `--config` approach is a good direction for mcporter config discovery, but the current stdio-external branch and remote-env handling are not yet the narrowest safe fix.

- **[P1] Preserve agent QMD env for configured stdio qmd:** `extensions/memory-core/src/memory/qmd-manager.ts:2367`
  - body: This branch sends every configured stdio server down external mode.
