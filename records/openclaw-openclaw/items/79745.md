## Best Possible Solution

Land the per-agent QMD mcporter isolation after the generated config path is backed by mcporter's actual JSON contract or avoids rewriting configured server definitions that cannot be faithfully preserved.

## Reproduction Assessment

Yes. Current main passes qmd-scoped XDG env into mcporter, and mcporter 0.10.2 resolves config from XDG_CONFIG_HOME unless an explicit config path is supplied, matching the reported failure path.

## Solution Assessment

No, not yet. The direction is right, but the generated-config preservation logic relies on mcporter config get JSON fields that mcporter 0.10.2 does not emit, so it needs a compatibility repair before this is the best merge shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve configured mcporter lifecycle fields:** `extensions/memory-core/src/memory/qmd-manager.ts:2345-2346`
  - body: This branch builds generated configs from `mcporter config get --json`, but mcporter 0.10.2's serializer does not include `lifecycle` or logging fields. For a configured stdio server without `env`, the new path writes a generated config and either drops the user's lifecycle or replaces it with a default keep-alive lifecycle when `startDaemon` is true, changing existing mcporter behavior on upgrade.
  - confidence: 0.88
- **[P2] Ignore mcporter's default Accept header for auth detection:** `extensions/memory-core/src/memory/qmd-manager.ts:209`
  - body: mcporter 0.10.2 serializes HTTP servers with a default `headers.accept` even when the raw remote has no auth. Because this classifier treats any `headers` key as auth-like, ordinary unauthenticated remotes never take the generated per-agent path that this PR claims and tests, so that remote-isolation coverage is not exercising a real dependency output shape.
  - confidence: 0.84
