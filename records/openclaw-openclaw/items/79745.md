## Summary

Keep open for maintainer review: current main still has the QMD-to-mcporter env leak, and the latest branch appears to address the prior ClawSweeper blockers with real terminal proof, but the config/daemon behavior remains compatibility-sensitive enough for explicit maintainer acceptance before merge.

## What This Changes

The PR updates memory-core QMD/mcporter runtime and tests to generate per-agent mcporter configs, split qmd versus mcporter process environments, preserve configured server definitions, parse log-prefixed mcporter JSON, and stabilize one agent MCP runtime test wait.

## Best Possible Solution

Land the generated-config plus external-fallback design once required checks pass and maintainers accept the compatibility tradeoff, keeping any broader remote-live coverage as follow-up if desired.

## Reproduction Assessment

Yes. Source inspection shows current main passes QMD-scoped XDG env to mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery unless an explicit config path is supplied; I did not run a live mcporter repro in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility-sensitive behavior. The branch uses explicit per-agent --config for generated QMD stdio configs, strips QMD XDG from mcporter spawns, caches config-mode resolution, and keeps env/auth-bearing configs external instead of overwriting user-owned state.
