## Summary

Keep this PR open, but not merge-ready: the central QMD/mcporter isolation fix is useful and has real proof, yet the latest diff still has compatibility and security-boundary blockers around mcporter config discovery, copied auth metadata, and lifecycle preservation.

## What This Changes

The PR changes memory-core QMD mcporter handling to generate per-agent mcporter configs, pass those configs to mcporter calls, strip QMD-scoped XDG env from mcporter spawns, preserve configured mcporter server definitions, and parse log-prefixed JSON responses.

## Best Possible Solution

Land a narrow QMD/mcporter fix that keeps QMD-scoped XDG only on the QMD server process, preserves user mcporter config discovery including XDG/MCPORTER_CONFIG/project config, and has maintainer-approved handling for remote auth material before writing generated agent config.

## Reproduction Assessment

Yes. Source inspection plus the linked issue show current/shipped QMD manager passes QMD-scoped XDG env to mcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery; I did not run a live repro in this read-only review.

## Solution Assessment

No. The per-agent --config direction addresses the main leak, but the current implementation is not the safest merge-ready shape until it preserves original mcporter discovery and resolves the remote-auth storage model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve user mcporter XDG config discovery:** `extensions/memory-core/src/memory/qmd-manager.ts:2271-2274`
  - body: `runMcporterCommand(..., { includeGeneratedConfig: false })` still uses `buildMcporterProcessEnv()`. Because that env is copied from `this.env` after the constructor replaced the user's `XDG_CONFIG_HOME` with the agent QMD dir, deleting it here also removes a legitimate user mcporter config location; mcporter 0.10 documents `$XDG_CONFIG_HOME/mcporter/mcporter.json[c]`, so those users can fall through to the generated local `qmd mcp` entry instead of their configured server.
  - confidence: 0.91
- **[P1] Avoid copying remote auth headers into agent state:** `extensions/memory-core/src/memory/qmd-manager.ts:2237-2239`
  - body: For HTTP servers this branch returns the serialized mcporter server and `ensureMcporterConfig()` writes it under the agent's QMD state. Upstream `mcporter config get --json` includes `headers`, and static `bearerToken` normalizes to `headers.Authorization`, so this can duplicate sensitive auth material into OpenClaw-managed agent state without an explicit storage decision.
  - confidence: 0.88
- **[P2] Preserve configured mcporter lifecycle values:** `extensions/memory-core/src/memory/qmd-manager.ts:2226-2227`
  - body: The tests assume `mcporter config get --json` returns `lifecycle`, but mcporter 0.10.2's serializer omits it. That means every configured stdio server reaches this fallback as `undefined` and gets rewritten to `keep-alive` with a 300s timeout, overriding any existing lifecycle or idle timeout.
  - confidence: 0.84
