## Summary

Keep open: this PR addresses a real QMD/mcporter isolation bug and has useful live terminal proof, but it still has a blocking security and compatibility concern because configured stdio server env/auth material can be copied into OpenClaw-managed per-agent state.

## What This Changes

The PR changes memory-core QMD/mcporter runtime and tests to generate per-agent mcporter configs, split qmd versus mcporter process environments, preserve configured server definitions, parse log-prefixed JSON, and stabilize one agent MCP test log wait.

## Best Possible Solution

Land the per-agent mcporter isolation approach only after configured stdio servers no longer duplicate user-owned secret-bearing env/auth material, or after maintainers explicitly approve and test that compatibility/security tradeoff.

## Reproduction Assessment

Yes for the reported bug from source inspection: current main sends QMD-scoped XDG env into mcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME during config discovery. I did not run a live current-main repro in this read-only review.

## Solution Assessment

No, not yet: the per-agent --config approach is a good core fix, but the configured stdio path still needs a maintainer-approved security/compatibility policy before it is the best merge shape.
