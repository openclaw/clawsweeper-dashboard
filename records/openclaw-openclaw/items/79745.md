## Summary

This PR should stay open for maintainer review: it addresses a real QMD/mcporter 0.10 failure and is not obsolete on current main, with sufficient redacted terminal proof and focused tests, but it changes compatibility-sensitive mcporter config and daemon behavior.

## What This Changes

The branch updates memory-core QMD runtime and tests to isolate mcporter config/daemon state per agent, split qmd and mcporter environments, preserve configured mcporter servers, parse log-prefixed mcporter JSON, and stabilize one MCP runtime test wait.

## Best Possible Solution

Land the generated-config plus external-fallback design after maintainers accept the compatibility tradeoff, keeping the preservation tests and redacted proof as the merge baseline.

## Reproduction Assessment

Yes. Source inspection shows current main passes QMD-scoped XDG env to mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery unless an explicit config path is supplied; I did not run a live mcporter repro in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility-sensitive behavior. The branch uses explicit per-agent --config for generated QMD stdio configs, strips QMD XDG from mcporter spawns, caches config-mode resolution, and keeps env/auth-bearing configs external instead of overwriting user-owned state.
