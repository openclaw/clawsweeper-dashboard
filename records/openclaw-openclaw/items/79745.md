## Summary

Keep open: this PR fixes a real mcporter 0.10 config-discovery problem for generated configs, but the current head still risks breaking configured stdio qmd servers by running them outside the agent-scoped QMD state. The supplied real behavior proof is also stale for that now-external configured-stdio path.

## What This Changes

The PR adds QMD mcporter generated/external config selection, per-agent generated mcporter configs, separated qmd versus mcporter spawn environments, log-prefixed JSON parsing, and expanded QMD/mcporter tests.

## Best Possible Solution

Land after configured stdio qmd servers preserve per-agent QMD state by default, or after maintainers explicitly accept and document external-state semantics with fresh real behavior proof.

## Reproduction Assessment

Yes, source-reproducible: a configured stdio server without env is classified as external, then mcporter is spawned without generated config and without the qmd-scoped state env. I did not run the live QMD/mcporter path in this read-only review.

## Solution Assessment

No. The generated-config path is a good direction for the mcporter 0.10 XDG discovery bug, but the current configured-stdio branch is not the narrowest safe fix because it loses agent-scoped QMD state for a normal `qmd mcp` server.
