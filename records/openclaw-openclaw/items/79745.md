## Summary

Keep this PR open for normal maintainer review: the latest head addresses the source-reproducible QMD/mcporter bug, includes real terminal proof, and I did not find a blocking correctness issue in the changed runtime path.

## What This Changes

The PR changes QMD memory mcporter calls to use per-agent generated mcporter configs, clean mcporter process env, configured-server preservation, log-prefixed JSON parsing, focused tests, and a changelog entry.

## Best Possible Solution

Land the per-agent mcporter config approach after maintainer review and green CI, keeping clean mcporter env, configured-server preservation, and keep-alive reapplication.

## Reproduction Assessment

Yes. Current main source still passes QMD-scoped XDG variables into mcporter, mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery when no explicit config is supplied, and the linked report includes concrete `memory_search` failure logs.

## Solution Assessment

Yes. Explicit per-agent `--config` paths align with mcporter's config-path contract, and stripping QMD XDG variables from mcporter's process env avoids the current leak while preserving configured stdio and remote server definitions.
