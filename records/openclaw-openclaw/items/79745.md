## Summary

Keep open: the PR targets a real QMD/mcporter session-state bug and has useful proof, but the branch still has a merge-blocking daemon warmness regression because it rewrites the generated mcporter config before every call.

## What This Changes

The PR adds agent-scoped mcporter config generation for QMD, preserves configured mcporter server definitions, strips QMD-scoped XDG env from mcporter process launches, parses log-prefixed JSON, and updates tests plus changelog.

## Best Possible Solution

Keep the per-agent explicit mcporter config and clean mcporter env design, but write the generated config only when content changes and add a repeated-search regression that proves the daemon stays warm.

## Reproduction Assessment

Yes, at source level: current main passes qmd-scoped XDG variables into mcporter and raw-parses mcporter stdout, matching the related reports' mcporter 0.10 and log-prefixed JSON failures. I did not run a live QMD/mcporter gateway in this read-only review.

## Solution Assessment

No, not as written. The explicit per-agent --config approach is maintainable, but the generated config write must be idempotent so mcporter's mtime-based daemon freshness check does not cold-restart keep-alive on repeated searches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid rewriting the mcporter config before every call:** `extensions/memory-core/src/memory/qmd-manager.ts:2076`
  - body: `ensureMcporterConfig()` always rewrites `mcporter.json`, and `runMcporter()` calls it before each mcporter command. mcporter records config layer mtimes in daemon metadata and treats any mtime change as stale, so same-content searches can repeatedly restart the per-agent daemon instead of reusing QMD keep-alive. Please skip the write when the generated content is unchanged and add a repeated-search regression.
  - confidence: 0.87
