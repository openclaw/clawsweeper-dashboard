## Summary

Keep open: the PR fixes the reported XDG leak direction and now includes real multi-agent QMD/mcporter proof, but the updated configured-server preservation path can drop the mcporter keep-alive lifecycle required for `startDaemon=true`.

## What This Changes

The PR writes agent-scoped mcporter configs for QMD, cleans mcporter's own process env, preserves configured server definitions, parses noisy JSON output, adds focused QMD manager tests, and updates the changelog.

## Best Possible Solution

Keep the agent-scoped config approach, but preserve or intentionally reapply keep-alive lifecycle for configured stdio qmd servers before merging.

## Reproduction Assessment

Yes. Source inspection plus mcporter 0.10.2 source shows current main leaks QMD-scoped XDG env into mcporter, and the PR's configured-server path can drop keep-alive because `config get --json` omits lifecycle.

## Solution Assessment

No, not as submitted. Passing an explicit per-agent `--config` is the right isolation handle, but the generated config must keep configured stdio qmd entries daemon-capable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep generated qmd servers keep-alive:** `extensions/memory-core/src/memory/qmd-manager.ts:2133-2136`
  - body: When a configured stdio qmd server is found, this returns the `mcporter config get --json` serialization. mcporter 0.10.2 does not serialize `lifecycle`, so the agent-scoped config drops the required keep-alive setting and `mcporter daemon start --config ...` sees no qmd server to keep warm. Please restore or reapply `lifecycle: keep-alive` for stdio qmd entries while preserving the rest of the configured server.
  - confidence: 0.89
