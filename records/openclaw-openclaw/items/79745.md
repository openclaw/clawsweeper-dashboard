## Summary

Keep open: current main still has the qmd/mcporter env coupling this PR is trying to fix, but the latest branch is not merge-ready because it can still rewrite configured `qmd mcp` stdio servers from mcporter's lossy JSON summary and drop existing lifecycle/logging behavior.

## What This Changes

This PR splits QMD and mcporter child environments, generates per-agent mcporter configs and daemon keys for QMD sidecars, handles configured or remote mcporter definitions, tolerates log-prefixed JSON, and localizes Skill Workshop mode labels.

## Best Possible Solution

Land a version that splits QMD and mcporter envs while preserving existing mcporter stdio definitions, including raw lifecycle/logging metadata, before generating per-agent sidecar config.

## Reproduction Assessment

Yes, source-reproducible. mcporter v0.10.2 serializes `config get --json` without lifecycle/logging, and the PR classifies visible `qmd mcp` stdio summaries as generated, so existing raw-only mcporter metadata would be dropped.

## Solution Assessment

No, not yet. Splitting qmd and mcporter envs is the right direction, but the best fix must preserve configured stdio metadata or keep those servers external instead of reconstructing them from lossy mcporter output.
