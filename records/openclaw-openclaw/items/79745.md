## Summary

Keep open: this active external bug-fix PR addresses a real QMD/mcporter failure path, but the submitted approach can bypass configured custom mcporter servers and still lacks after-fix real behavior proof.

## What This Changes

The branch writes an agent-scoped mcporter config for QMD, passes it with `--config`, keys daemon starts by config path, tolerates daemon log lines before JSON responses, adds qmd-manager tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the fix focused on preventing qmd-only XDG state from breaking mcporter while preserving configured/custom mcporter server targets, then require redacted real `memory_search` proof with mcporter 0.10.x and two agents.

## Reproduction Assessment

Yes, source-reproducible: current main passes qmd-scoped XDG variables into `runMcporter`, and mcporter 0.10.2 uses XDG config discovery unless an explicit config/env override changes that path. I did not run a live gateway/QMD daemon scenario in this read-only review.

## Solution Assessment

No, not as submitted. Passing an explicit per-agent config is a plausible isolation handle, but replacing the user's configured mcporter server with a generated local `qmd mcp` entry is not the narrow maintainable fix unless maintainers intentionally change that contract.
