## Summary

Keep open: this is a plausible fix for a real QMD/mcporter isolation bug, but the branch is not merge-ready because it lacks real behavior proof and its generated `--config` file can bypass users' configured mcporter server definitions.

## What This Changes

The PR writes an agent-scoped mcporter config for QMD, passes it via `--config`, keys daemon starts by that config path, tolerates noisy mcporter JSON output, adds QMD manager tests, and updates the changelog.

## Best Possible Solution

Keep the QMD/mcporter fix focused on isolating OpenClaw's qmd-only state while preserving configured mcporter server definitions, then require redacted real multi-agent QMD/mcporter proof before merge.

## Reproduction Assessment

Yes, from source inspection and the mcporter v0.10.2 contract: current main sends qmd-scoped XDG env to mcporter, and the PR's explicit generated config would bypass the user's configured mcporter server. I did not run a live gateway/QMD daemon reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Passing an explicit per-agent config is a plausible isolation handle, but the generated single-server config needs to preserve or intentionally replace existing custom mcporter setups before this is the narrow maintainable fix.
