## Summary

Keep this PR open for human review: it targets a real source-reproducible QMD/mcporter bug, but the submitted generated-config path bypasses configured mcporter server definitions, the branch has a concrete lint blocker, and after-fix real behavior proof is still absent.

## What This Changes

The PR writes an agent-scoped mcporter config for QMD, passes it to mcporter calls with `--config`, keys daemon starts by config path, tolerates daemon log lines before JSON output, adds QMD manager tests, and updates the changelog.

## Best Possible Solution

Land a focused QMD/mcporter fix that isolates qmd-only state while preserving configured mcporter targets, with regression tests and redacted live proof.

## Reproduction Assessment

Yes, source-reproducible: current main passes qmd-scoped XDG variables into mcporter, and mcporter v0.10.2 uses those variables for config discovery unless an explicit config is supplied. I did not run a live gateway/QMD daemon reproduction in this read-only review.

## Solution Assessment

No, not as submitted. Explicit `--config` is a plausible isolation handle, but this implementation must preserve configured mcporter server definitions or intentionally update the public config contract, fix the lint blocker, and supply real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve the configured mcporter server:** `extensions/memory-core/src/memory/qmd-manager.ts:2057-2058`
  - body: This generated config contains only a local `qmd mcp` stdio server and is passed with `--config`, which mcporter treats as the only config file. That bypasses the existing `serverName` contract for user-configured mcporter targets, including the custom or remote qmd server setup described in the linked bug, so those searches can regress even though XDG discovery is bypassed.
  - confidence: 0.87
- **[P2] Fix the redundant unknown union:** `extensions/memory-core/src/memory/qmd-manager.ts:152`
  - body: The new `extractFirstJsonValue(raw: string): unknown | null` signature trips the extension-bundled oxlint rule because `unknown` absorbs `null`, and CI reports this exact error. Use a concrete JSON-ish type plus `null`, or a discriminated result, so the lint lane can pass.
  - confidence: 0.96
