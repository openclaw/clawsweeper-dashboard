## Summary

Keep this PR open: it targets a real QMD/mcporter isolation bug, but it is not merge-ready because it lacks real behavior proof, has a CI-blocking lint error, and appears to replace existing configured mcporter server setups with a generated local `qmd mcp` entry.

## What This Changes

The PR writes an agent-scoped mcporter config for QMD searches, passes it via `--config`, keys daemon startup by config path, tolerates noisy mcporter JSON output, adds QMD manager tests, and updates the changelog.

## Best Possible Solution

Land a revised PR that keeps explicit per-agent mcporter config/daemon isolation, preserves supported configured-server semantics or intentionally updates the contract, fixes lint, and includes redacted live multi-agent QMD/mcporter proof.

## Reproduction Assessment

Yes, from source inspection and the linked report: current main passes QMD-scoped XDG variables into mcporter, and mcporter 0.10.x honors XDG for config discovery unless `--config` is explicit. I did not run the live daemon scenario in this read-only review.

## Solution Assessment

No as-is: explicit per-agent `--config` is the right direction, but the patch needs to fix the lint failure, avoid silently dropping existing mcporter server configurations, and add real behavior proof before it is the best merge candidate.
