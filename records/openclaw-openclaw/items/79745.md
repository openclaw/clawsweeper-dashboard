## Summary

Keep this PR open: it addresses a real QMD/mcporter isolation regression with useful proof, but the current patch still has a compatibility blocker for configured stdio mcporter servers whose raw lifecycle or logging settings are not preserved through `mcporter config get --json`.

## What This Changes

This PR splits QMD and mcporter spawn environments, writes per-agent mcporter config for generated QMD sidecars, preserves some configured or remote mcporter definitions, tolerates log-prefixed mcporter JSON, and localizes Skill Workshop switcher text.

## Best Possible Solution

Keep generated per-agent mcporter config for OpenClaw-owned default QMD sidecars, but leave user-configured stdio mcporter servers on their original config unless the raw entry can be copied losslessly.

## Reproduction Assessment

Yes, source inspection and dependency inspection give a high-confidence path: current main sends QMD-scoped XDG env to mcporter, and the PR's configured-stdio branch can regenerate from a lossy mcporter JSON summary. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. The explicit `--config` and env split are the right shape for generated QMD sidecars, but configured stdio servers need a compatibility-preserving path before this is the best fix.
