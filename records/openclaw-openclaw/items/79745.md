## Summary

Keep open: the fix direction is well-supported and the PR supplies real QMD/mcporter behavior proof, but the current diff still rewrites the generated mcporter config before every mcporter invocation, which conflicts with mcporter's mtime-based stale-config detection and can restart warm daemons on repeated searches.

## What This Changes

The PR updates memory-core's QMD manager to generate per-agent mcporter configs, pass them with --config, strip QMD XDG env from mcporter spawns, preserve configured stdio/remote server definitions, parse log-prefixed mcporter JSON, and expand qmd-manager tests.

## Best Possible Solution

Land the per-agent --config isolation with clean mcporter env only after generated config writes are stable across repeated searches and the custom/remote server preservation path remains covered.

## Reproduction Assessment

Yes, from source inspection and the linked report: current main passes agent-scoped QMD XDG env into mcporter, and mcporter's own source shows explicit config mtimes drive stale-daemon restarts. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. The per-agent --config approach is the right direction, but the implementation needs an idempotent generated-config write before it is the safest maintainable fix.
