## Summary

Keep open: the memory/QMD direction is useful, but the latest head still externalizes configured stdio servers whose `command` is an absolute `qmd` path, which contradicts the PR's own proof and can drop per-agent QMD state isolation for documented setups.

## What This Changes

The PR rewrites QMD mcporter handling to generate per-agent mcporter configs, split QMD and mcporter environments, preserve user-owned configured servers externally, parse log-prefixed mcporter JSON, expand focused tests, and localize Skill Workshop mode-switch labels.

## Best Possible Solution

Generate the per-agent mcporter config for plain `qmd mcp` servers regardless of whether `command` is a bare `qmd` or an absolute qmd executable, while keeping custom/env/auth wrappers and authenticated remotes external.

## Reproduction Assessment

Yes. Source inspection and the linked reports show current main passes qmd-scoped env to mcporter, and the PR head's classifier path can be followed without running it; I did not execute the live mcporter scenario in this read-only review.

## Solution Assessment

No, not yet. The generated/external split is the right shape, but the current classifier is too narrow for documented absolute qmd commands and should be fixed before merge.
