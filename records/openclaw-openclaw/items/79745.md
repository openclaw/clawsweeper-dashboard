## Summary

Keep this PR open: the core fix direction is right and the real behavior proof is adequate, but the latest head still rewrites the generated mcporter config before every daemon/call path, which can make mcporter treat the warm daemon as stale immediately.

## What This Changes

The PR changes QMD mcporter search to generate an agent-scoped mcporter config, pass `--config` to mcporter calls, strip QMD XDG vars from mcporter's own environment, preserve configured server definitions, parse log-prefixed JSON, and add qmd-manager tests.

## Best Possible Solution

Keep the per-agent explicit mcporter config approach, but only update `mcporter.json` when content changes so daemon metadata remains fresh across repeated searches.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main passes QMD-scoped XDG variables into mcporter and raw-parses log-prefixed stdout, and the related reports include concrete failure logs. I did not run a live repro because the review was read-only.

## Solution Assessment

No, not as written. The `--config` and clean mcporter process env approach is the right shape, but generated config writes need to be idempotent to preserve mcporter's mtime-based daemon freshness check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid rewriting unchanged mcporter config:** `extensions/memory-core/src/memory/qmd-manager.ts:2076`
  - body: `ensureMcporterConfig()` writes `mcporter.json` every time, and both daemon startup and each mcporter call route through it. mcporter v0.10.2 uses config layer mtimes to decide whether the daemon is stale, so rewriting the same content after `daemon start` can make the just-started per-agent daemon cold-restart on the following search call. Please skip the write when the generated content is unchanged and add a repeated-search/mtime regression.
  - confidence: 0.9
