## Summary

Keep open: the QMD/mcporter bug is real and the PR is still useful, but the latest head still rewrites the generated mcporter config before every mcporter command, which conflicts with mcporter daemon stale-config detection and can restart or cold-start the daemon on repeated searches.

## What This Changes

The PR changes memory-core's QMD manager to generate per-agent mcporter configs, pass them via --config, strip QMD XDG env from mcporter spawns, preserve configured server definitions, parse log-prefixed mcporter JSON, and expand focused tests.

## Best Possible Solution

Keep the PR open and land the per-agent --config approach only after generated mcporter config writes are content-idempotent and the repeated-search daemon regression is covered.

## Reproduction Assessment

Yes. Current main still passes QMD-scoped XDG env into mcporter and raw-parses mcporter stdout, and mcporter v0.10.2 source shows why explicit --config and stable config mtimes matter.

## Solution Assessment

No, not yet. The per-agent --config direction is the maintainable fix, but the implementation should avoid rewriting unchanged generated config before it can be considered safe to merge.
