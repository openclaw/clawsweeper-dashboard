## What This Changes

The PR changes QMD memory mcporter routing to add generated per-agent mcporter configs, split qmd versus mcporter environments, tolerate log-prefixed mcporter JSON, and expand QMD/mcporter tests.

## Best Possible Solution

Land a QMD/mcporter fix that keeps mcporter config discovery out of QMD's XDG dirs while preserving agent-scoped QMD env for qmd stdio servers, then add fresh redacted terminal or live proof on the current head.

## Reproduction Assessment

Yes, from source: current main passes agent-scoped XDG/QMD env into mcporter, and mcporter 0.10 source uses XDG for config discovery unless --config or MCPORTER_CONFIG is explicit. The PR-introduced stdio-state issue is also source-reproducible because the latest test asserts configured stdio servers run without --config while mcporter stdio children inherit process.env plus server env.

## Solution Assessment

No, not yet: the generated-config approach is a good direction for the XDG leak, but routing all configured stdio servers externally leaves a common qmd stdio setup outside agent-scoped memory state. The safer solution is to preserve qmd's per-agent env for qmd stdio entries or get explicit maintainer acceptance of that break with current-head proof.
