## Summary

Keep open. Current main still has the plugin-update diagnostics gap: npm metadata probe failures fall through to the installer path without retry or no-output watchdog, while the exact `acpx@0.5.3` invalid-spec outcome is not fully proven because current parser logic accepts exact versions and `@beta` tags.

## What This Changes

The issue asks for plugin update failures after transient npm metadata lookup errors to report the registry problem clearly instead of surfacing a misleading unsupported npm spec error.

## Best Possible Solution

Add focused regression coverage for incomplete/transient npm metadata lookup failures, then either retry those failures with a no-output watchdog or preserve the registry-failure context so valid npm specs are not reported as spec-validation failures.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a clear mockable path through metadata probe failure and installer fallback, but the specific `acpx@0.5.3` invalid-spec result does not directly follow from current main.

## Solution Assessment

Yes, the best direction is a narrow resolver/update diagnostics repair. Keep registry npm spec validation strict, and avoid changing ClawHub/catalog migration behavior unless a regression test proves that path is involved.
