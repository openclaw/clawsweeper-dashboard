## Summary

Keep open. Current main and the latest stable release still have the plugin-update diagnostics gap: npm metadata probing is a single `npm view` call without retry/no-output watchdog, metadata failure falls through to installer update, and existing tests cover the fallback but not preserving transient registry context over misleading spec-validation output. The exact reported `acpx@0.5.3` unsupported-spec transition is not fully proven from current source, so the fix should be narrow and test-driven.

## What This Changes

The issue requests that plugin update handle transient npm metadata failures without surfacing a misleading unsupported-spec error for valid npm specs.

## Best Possible Solution

Add focused regression coverage for transient or incomplete npm metadata lookup failures, then retry those failures with the existing no-output watchdog or preserve the registry-failure context so valid npm specs are not reported as spec-validation failures.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a clear mockable path through metadata probe failure and installer fallback, but the specific `acpx@0.5.3` invalid-spec result does not directly follow from current main.

## Solution Assessment

Yes, the best direction is a narrow resolver/update diagnostics repair. Keep registry npm spec validation strict, and avoid changing ClawHub/catalog migration behavior unless a regression test proves that path is involved.
