## Summary

Keep open. Current main still has a narrow plugin-update diagnostics gap: npm metadata probe failures are not retried and the updater falls through to the installer path, but the exact `acpx@0.5.3` invalid-spec transition is not fully proven from source because that spec parses as valid.

## What This Changes

The issue asks plugin update checks to handle transient npm metadata failures with retry or clear registry-failure messaging instead of surfacing a misleading unsupported npm spec error.

## Best Possible Solution

Add focused regression coverage around npm metadata probe failure plus fallback failure, then either retry incomplete/transient `npm view` results with the existing no-output watchdog or preserve the registry-failure context so valid specs are not reported as spec-validation failures.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection gives a clear mockable path through metadata probe failure and installer fallback, but the specific `acpx@0.5.3` invalid-spec result does not follow directly from current main's parser.

## Solution Assessment

Yes, a narrow repair in the metadata resolver/update formatter is the best direction. Keep registry-only npm validation unchanged and avoid changing ClawHub/catalog migration behavior.

## Repair Work Prompt

Fix the plugin-update diagnostics reported in this issue. In current main, transient `resolveNpmSpecMetadata` failures during npm plugin updates warn and fall through to `installPluginFromNpmSpec`, which can leave users with a misleading installer/spec error instead of the transient npm metadata failure. Keep registry-only npm spec validation unchanged. Add or update tests in `src/plugins/update.test.ts`, `src/infra/install-source-utils.test.ts`, and `src/infra/npm-registry-spec.test.ts` to cover valid exact specs such as `acpx@0.5.3`, `@beta` dist-tags, metadata probe failure, and fallback failure. Implement a small retry/no-output timeout in `resolveNpmSpecMetadata` using existing `runCommandWithTimeout` support, or preserve the first transient metadata error when fallback fails for metadata reasons. Do not allow URL/git/file specs or semver ranges, do not add config, and do not change plugin trust/provenance rules. Add a changelog entry if user-facing output changes.
