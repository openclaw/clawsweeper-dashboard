## Summary

Keep open. Current main still has a localized plugin-update diagnostics gap: npm metadata probing is a single `npm view` call and the updater falls through to the installer after probe failure, while the exact `acpx@0.5.3` unsupported-spec result is not proven because current parsing accepts exact versions and dist-tags.

## What This Changes

The issue asks plugin npm update checks to handle transient npm metadata failures without surfacing a misleading unsupported npm spec error.

## Best Possible Solution

Add focused regression coverage and adjust the metadata resolver or updater so transient/incomplete `npm view` results retry or preserve the registry-failure cause, while leaving registry-only npm spec validation intact.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a mockable path through metadata probe failure and installer fallback, but the specific `acpx@0.5.3` unsupported-spec output does not follow from current main's parser.

## Solution Assessment

Yes, the best direction is a narrow plugin-update/metadata resolver repair with regression tests. Avoid changing ClawHub migration policy or broadening npm spec validation unless a test proves that parser layer is actually involved.

## Repair Work Prompt

Fix the plugin npm update diagnostics gap reported in https://github.com/openclaw/openclaw/issues/77616. Add focused regression coverage for `resolveNpmSpecMetadata` returning incomplete/transient npm metadata and for `updateNpmInstalledPlugins` when the precheck falls through and the installer also fails. The fix should either retry incomplete/transient `npm view` results with the existing command timeout/no-output timeout seam or preserve the original registry-failure context so valid specs such as `acpx@0.5.3` are not reported as unsupported npm specs.
