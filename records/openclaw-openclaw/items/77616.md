## What This Changes

The issue asks OpenClaw to stop reporting a spec-validation error when plugin update metadata checks hit transient `npm view` failures, and to retry or surface a clearer transient registry error.

## Best Possible Solution

Add bounded retry/no-output handling around npm metadata lookups and regression coverage ensuring valid npm specs keep transient registry failures from being reclassified as invalid-spec errors.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows the current metadata-failure fallback path, but I did not establish a live transient npm registry repro or reproduce the exact `unsupported npm spec` string on current main.

## Solution Assessment

Is this the best way to solve the issue? A targeted retry/error-preservation fix in `resolveNpmSpecMetadata` and the plugin update fallback path is the narrow maintainable direction; the exact timeout/backoff values should be covered by focused tests rather than a new config surface.
