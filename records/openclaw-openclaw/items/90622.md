## Summary

This PR is useful but not merge-ready: it adds the intended batched multi-port listener path, yet it also changes existing single-port Unix `inspectPortUsage()` calls to run an all-listener `lsof` scan, including restart-health and diagnostic callers that the PR says are out of scope.

## What This Changes

The PR adds `inspectPortUsages()` for batched Unix listener snapshots, wires daemon status to use it for distinct daemon/CLI ports, exports the helper, and adds focused regression tests.

## Best Possible Solution

Land the batch helper only after single-port inspections keep the targeted `-iTCP:<port>` command shape and tests lock both the targeted one-port path and the batched multi-port path.

## Reproduction Assessment

Yes for the PR-introduced command-scope regression by source inspection: `inspectPortUsage(port)` reaches the all-LISTEN lsof snapshot on the PR head, while restart-health still calls `inspectPortUsage` one port at a time. I did not live-reproduce the disruptive launchservicesd/WindowServer saturation scenario.

## Solution Assessment

No. The batch helper is the right direction for same-cycle multi-port status checks, but the best fix keeps existing one-port callers targeted and uses all-LISTEN lsof only for the explicit batch helper.
