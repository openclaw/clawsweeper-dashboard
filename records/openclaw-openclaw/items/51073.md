## Summary

Keep open. Current main has adjacent mitigations, including tool-shaped-text logging, proxy `tool_choice: auto`, incomplete-turn retries, trajectory export, and diagnostics, but it still does not verify that final claims about web/property/source research are backed by matching `web_search`, browser, or tool telemetry.

## What This Changes

The issue requests mandatory source verification so agents cannot fabricate property research or falsely claim Google Hotels/search activity.

## Best Possible Solution

Keep this as a core trust/integrity issue and design a configurable source/claim verification layer that cross-checks final claims about searches, browser activity, and cited sources against actual `web_search`, `web_fetch`, browser, MCP, or channel/tool telemetry, then warns or blocks delivery according to policy. Coordinate the tool-call enforcement side with #45049 and fail-closed delivery behavior with #49876.

## Reproduction Assessment

Unclear for the exact property dataset without replaying the reporter's OpenRouter/Google Hotels setup. There is high-confidence source-level reproduction of the missing guarantee: current payload code and tests still allow visible final-answer text to be delivered without matching tool evidence.

## Solution Assessment

No, the current mitigations are not the best complete solution for this issue. Diagnostics, tool-shaped-text warnings, and planning-only retries are useful but do not verify that a claimed search/source action actually occurred; a telemetry-backed claim verifier is the narrower maintainable direction.
