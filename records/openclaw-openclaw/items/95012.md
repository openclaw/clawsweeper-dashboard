## Summary

Keep open: this is a focused fix attempt for a real P1 cron completion-delivery bug, but it is not merge-ready because the lease is skipped when an existing session entry is empty and it stores raw cron delivery config instead of the resolved route needed by the reported channel-only path.

## What This Changes

The PR adds a SQLite-backed task-route lease table/module and wires isolated cron delivery-target resolution to consult a run-id keyed lease fallback.

## Best Possible Solution

Store the concrete resolved delivery target in a shared background-task route lease and consult it when higher-precedence session sources are missing or unroutable, with explicit retirement/pruning coverage.

## Reproduction Assessment

Yes, from source inspection: a cron session entry that exists without deliveryContext still wins over the lease, and a channel-only cron delivery config stores no `to` in the lease. I did not run a live cron/channel send in this read-only review.

## Solution Assessment

No. The requested direction is valid, but this implementation needs to persist the resolved route and fall back based on routability rather than object presence.
