## Summary

The PR should stay open for repair: it targets a real isolated-cron completion delivery bug, but the proposed persistent lease uses raw runId as a global identity even though current task lifecycle code treats runId as scope-local, so it can overwrite or settle another task route and reintroduce wrong-target or dropped completion delivery.

## What This Changes

The PR adds a SQLite-backed task route lease table and wires cron/task execution, delivery-target resolution, and repro scripts so isolated cron completion delivery can recover an explicit requester route from runId.

## Best Possible Solution

Before merge, key task route leases by the same scoped identity used by task lifecycle delivery, or narrow the stored lease to an unambiguous cron-owned identity, then prove the repaired route on the real completion path.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: issue #92460 describes an explicit isolated cron channel being lost on the completion announcer path, and the current code path lacks a persisted completion route after the isolated session context is empty.

## Solution Assessment

No, this is not yet the best fix because the route lease identity is broader than the existing task scope contract; the design must preserve scoped runId collisions before the persistent lease approach is acceptable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope route leases beyond raw runId:** `src/tasks/task-route-lease.ts:137-146`
  - body: The new lease store is keyed only by `runId` and overwrites on conflict, but current task lifecycle code intentionally allows the same runId in different runtime/session scopes and scopes delivery updates by that owner/session identity. With two scoped tasks sharing a runId, the later acquire can replace the earlier requester route and a terminal update can settle the wrong lease, so completion delivery may go to the wrong target or disappear. Key this by `taskId` or the same scoped identity used by the task registry, and carry that key through lookup and settlement.
  - confidence: 0.91
