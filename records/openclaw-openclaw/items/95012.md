## Summary

Keep open: the proposed runtime direction is plausible, but current head still has a blocking proof/test isolation defect. The repro scripts open a temp SQLite handle, while the lease helpers they call use `process.env` and can reset/write the default shared state DB instead of the advertised temp fixture; the new persistent lease table also needs maintainer acceptance before merge.

## What This Changes

Adds a SQLite-backed task-route lease table/module plus cron resolver, manual-run, GC, tests, and repro wiring to preserve isolated cron completion delivery targets.

## Best Possible Solution

Bind all lease repros/tests to the intended temp `OPENCLAW_STATE_DIR` or add env/path-aware lease helper options, rerun current-head proof, then merge only after maintainers accept the persistent route-lease schema, retention, and stacked ordering with https://github.com/openclaw/openclaw/pull/95352.

  - confidence: 0.91
- **[P2] Set OPENCLAW_STATE_DIR for the lease unit tests:** `src/tasks/task-route-lease.test.ts:52`
  - body: The test opens an isolated DB handle, but the lease helpers under test reopen the shared state DB from `process.env`, so this handle is unused. In Vitest that falls back to the worker-scoped default DB rather than this test's `stateDir`, which means cases can share rows and the tests do not prove the per-test isolation claimed in the comments. Run the helper calls inside `withEnvAsync({ OPENCLAW_STATE_DIR: stateDir })` or thread an explicit DB/env option through the lease API.
