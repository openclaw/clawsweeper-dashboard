## Label Justifications

- `P1`: The PR targets P1 completion message loss and currently introduces a blocking regression in active manual cron cancellation.
- `merge-risk: 🚨 compatibility`: The diff adds a new shared SQLite state table and changes run-id semantics used by existing task operations.
- `merge-risk: 🚨 message-delivery`: The PR changes fallback precedence for cron completion delivery, where a mistake can drop or misroute user-visible messages.
- `merge-risk: 🚨 session-state`: The new lease stores delivery-route identity across evicted or retargeted sessions and affects session-origin recovery.

## Summary

Keep open: the route-lease fix has current-head terminal proof for the reported cron route-id path, but the patch is not merge-ready because the manual run-id change rekeys active cron cancellation away from the task ledger run id and the persistent route-state model still needs maintainer acceptance.

## What This Changes

The PR adds a SQLite-backed task-route lease table/module and wires cron/task execution to use run-id-based delivery-route fallback for isolated cron completion delivery, with repro scripts and regression tests.

## Best Possible Solution

Preserve task cancellation by separating the delivery-route lookup id from the task-ledger/cancellation id, and merge the persistent route-lease model only after maintainer acceptance of its retention and upgrade semantics.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction for the introduced regression: a manual cron run with a caller manual: run id creates the task row with the cron: run id, but the active cancellation handle is registered under the manual: id.

## Solution Assessment

No. The route-lease lookup direction is plausible, but using executeJobCoreWithTimeout's single runId for both delivery-route lookup and cancellation is not the best fix; those identities need to be split or both registered.

- **[P1] Keep the cron task run id available for cancellation:** `src/cron/service/ops.ts:1010`
  - body: `tryCreateManualTaskRun` still stores the task ledger row with `runId: taskRunId`, and `cancelTaskById` cancels cron runs by that stored `task.runId`. Passing the caller's `manual:` id into `executeJobCoreWithTimeout` makes `registerActiveCronTaskRun` register the abort handle under a different key, so cancelling the visible cron task returns `Cron task has no active cancellation handle` while the manual run keeps running. Split the resolver/lease key from the cancellation task-run key, or register both ids.
