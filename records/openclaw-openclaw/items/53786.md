## Summary

Keep open. Current main still treats Control UI/WebChat as an internal surface and prevents it from inheriting Feishu delivery routes, while Feishu ingress records and dispatches in the resolved Feishu session. This remains the canonical Feishu + Control UI sync/product-boundary issue after #46315 was closed as a duplicate; #36640 does not cover the Feishu behavior.

## What This Changes

The issue requests real-time bidirectional message and task-status synchronization between the local Control UI dashboard and the Feishu bot channel.

## Best Possible Solution

Keep this as the canonical Feishu + Control UI sync item. Maintainers should decide whether OpenClaw should support an explicit opt-in Feishu/WebChat mirror or delivery mode; if yes, implement it with route-safety and live-session tests, and if no, document the boundary clearly in Control UI, WebChat, channel-routing, and Feishu docs.

## Reproduction Assessment

Yes for the code-level message desync boundary: current main shows Control UI sends are internal-only, WebChat clients cannot inherit external routes, and dashboard refresh follows only the selected session key. The full live Feishu delay/task-status portion still needs Feishu DM, group, and topic evidence with gateway logs.

## Solution Assessment

Unclear. Exact always-on mirroring is not obviously the safest fix because the code and docs intentionally route replies back to the originating surface; the maintainable solution is either an explicit opt-in mirror/delivery mode or clear documentation that Feishu and WebChat are not identical synchronized clients.
