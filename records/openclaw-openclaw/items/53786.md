## Summary

Keep this open. Current main still preserves the WebChat/internal delivery boundary for Control UI sends, while Feishu ingress writes to the resolved Feishu session and only refreshes the dashboard when that exact session is selected. Related #46315 was closed as a duplicate of this issue, and #36640 only proves the Telegram/WebChat active-session update path, not Feishu bidirectional mirroring.

## What This Changes

The issue requests real-time bidirectional message and task-status synchronization between the local Control UI dashboard and the Feishu/Lark bot channel.

## Best Possible Solution

Keep this as the canonical Feishu + Control UI sync item. Maintainers should decide whether OpenClaw should support an explicit opt-in Feishu/WebChat mirror or delivery mode; if yes, implement it with route-safety and live-session tests, and if no, document the boundary clearly in Control UI, WebChat, channel-routing, and Feishu docs.

## Reproduction Assessment

Yes for the core behavior boundary: current main shows Control UI sends use `deliver: false`, WebChat clients cannot inherit Feishu routes, and dashboard refresh only follows the selected session key. Live Feishu DM/group/topic testing is still needed to determine whether a separate transport-delay bug is also present.

## Solution Assessment

Unclear. Always-on exact mirroring is not obviously the safest solution because the code intentionally keeps WebChat internal; the maintainable path is either an explicit opt-in mirror/delivery mode or clear docs that Feishu and WebChat are not intended to be identical surfaces.
