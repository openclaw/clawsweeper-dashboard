## Summary

Keep open: this is a useful doctor diagnostic with sufficient copied CLI proof, but the patch can miss valid routed agents because it matches `binding.agentId` to `agents.list[].id` by raw string instead of the runtime-normalized agent id. The linked issue also has a separate prompt/introspection half, so it should not be auto-closed by this doctor-only PR unless that remainder is tracked elsewhere.

## What This Changes

The PR adds a doctor preview warning and tests for channel-bound agents whose resolved tool policy does not include the `message` tool.

## Best Possible Solution

Keep the doctor warning, normalize route-bound agent resolution to match runtime behavior, and adjust the linked issue handling so the prompt/introspection remainder stays tracked.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence repro for the PR bug: `agents.list[].id: "Commander"` with `binding.agentId: "commander"` routes at runtime, but the PR's raw map lookup misses the agent-specific allowlist and can suppress the doctor warning.

## Solution Assessment

No. The doctor warning is the right narrow surface, but it should resolve agent ids through the same normalized route/agent path as runtime instead of raw string lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve bindings through normalized agent ids:** `src/commands/doctor/shared/preview-warnings.ts:140`
  - body: The new warning looks up `agentById.get(agentId)` using raw config strings, but runtime route selection normalizes binding agent ids before matching configured agents. A valid config such as `id: "Commander"` with a binding to `"commander"` still routes to that agent, while this check falls back to global tools and can leave doctor silent for a routed agent missing `message`.
