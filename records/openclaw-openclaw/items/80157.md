## Summary

Keep open: the doctor warning is useful and the PR has sufficient live-output proof, but the current patch can miss the same runtime route when the configured agent id and binding id differ only by normalization.

## What This Changes

The PR adds a doctor preview warning and tests for channel-bound agents whose resolved tool policy lacks the `message` tool.

## Best Possible Solution

Keep the doctor warning, resolve route binding agent ids with the same normalization runtime uses, add the missing mixed-case regression, and leave the runtime self-introspection half tracked separately.

## Reproduction Assessment

Yes. Source inspection shows a concrete case: `agents.list[].id: "Commander"` with a binding to `commander` resolves at runtime, but the PR's raw map lookup reads no per-agent tools and can miss the warning.

## Solution Assessment

No. The doctor diagnostic is the right narrow surface, but it needs runtime-equivalent normalized agent resolution before checking the bound agent's tool policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize binding IDs before checking agent tools:** `src/commands/doctor/shared/preview-warnings.ts:140`
  - body: The new check looks up tools with the raw binding `agentId`, but route resolution normalizes binding ids before selecting the effective agent. With a configured agent like `Commander` and a binding to `commander`, runtime selects that agent while doctor reads no per-agent tools, falls back to global policy, and can miss the missing-`message` warning.
