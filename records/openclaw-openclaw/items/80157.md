## Summary

Keep open: the PR adds a useful doctor warning for the linked channel/tool-policy bug, but the new check does not resolve binding agent ids the same way runtime routing does, so a supported mixed-case or fallback binding can still miss the warning.

## What This Changes

Adds a doctor preview warning and focused tests for route-bound agents whose resolved tool policy excludes the `message` tool.

## Best Possible Solution

Keep the doctor warning, but resolve each route-bound agent through the same normalized agent-id path as runtime before checking message-tool availability.

## Reproduction Assessment

Yes, source-reproducible: current main filters `message` out of an explicit allowlist and the existing doctor code has no route-bound explicit-action warning. The PR body also gives copied after-fix doctor output for the CLI path.

## Solution Assessment

No: the warning is the right narrow surface, but the implementation should reuse normalized route/agent resolution rather than raw `agentId` string lookup.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Resolve bindings through normalized agent ids:** `src/commands/doctor/shared/preview-warnings.ts:140`
  - body: The warning looks up `agentById.get(agentId)` using raw config strings, but runtime routing normalizes binding ids with `pickFirstExistingAgentId`/`normalizeAgentId`.
