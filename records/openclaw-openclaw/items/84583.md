- `impact:session-state`: The failure is a session transcript mutation tripping the embedded session-file takeover fence.

## Summary

Close as duplicate: this report is a concrete Telegram cron-announce instance of the already-open shared-session `EmbeddedAttemptSessionTakeoverError` race, and the canonical issue already covers legitimate cron/channel co-tenant writes during the released prompt-lock window.

## What This Changes

An isolated cron job with explicit Telegram announce delivery can append to the same user's active session transcript and trip the embedded prompt-lock takeover fence.

## Best Possible Solution

Track and verify this through https://github.com/openclaw/openclaw/issues/84071 and the active session-fence fix PR, with an explicit regression for isolated cron announce delivery to an active chat target.

## Reproduction Assessment

Yes, at source level. Current main resolves explicit cron announce delivery to a destination session and mirrors into that transcript, while the embedded runner still throws if the same session file changes during the released prompt-lock window.

## Solution Assessment

No, not as a separate tracking issue. The maintainable path is to resolve the canonical co-tenant session-fence race and verify this cron announce variant there, instead of splitting another overlapping fix lane.
