## Summary

Keep open: current main and latest v2026.6.8 still mirror routed peer cron deliveries into the resolved session transcript, and an open linked PR now owns the narrow fix path.

## What This Changes

Cron announce delivery can append a transcript mirror to an active Telegram peer session during an embedded prompt lock release, causing EmbeddedAttemptSessionTakeoverError and a failed user turn.

## Best Possible Solution

Keep this issue open until the linked cron mirror fix is reviewed and either landed or rejected; the desired end state is direct delivery without unsafe routed-peer transcript writes during active embedded turns.

## Reproduction Assessment

Yes. Source inspection shows isolated cron direct delivery can resolve to a per-channel-peer session, append a transcript mirror, and trip the embedded prompt-release session fence; I did not run a live Telegram cron scenario.

## Solution Assessment

Yes at the issue level. The best fix is to stop or defer the unsafe routed-peer cron mirror path, not to weaken the session takeover fence globally.
