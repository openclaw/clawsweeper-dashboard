## Label Justifications

- `P2`: The bug has real user-visible failed turns, but the affected surface is a bounded concurrent cron/announce delivery workflow rather than a core runtime outage.
- `impact:session-state`: The reported failure is caused by an unsafe transcript/session-file mutation during an embedded attempt lock-release window.
- `impact:message-loss`: When the race fires, the in-flight channel response aborts and the user sees an error instead of the expected reply.

## Summary

Keep open: current main still resolves direct cron delivery to a session key and can append a delivery-mirror transcript into that resolved session, while the newer fix is an open PR rather than merged code. The latest issue discussion also reports the same failure on v2026.6.8, so this is still active maintainer work.

## What This Changes

The issue asks OpenClaw to prevent isolated cron or announce deliveries from mutating an active Telegram or per-channel-peer session transcript while an embedded attempt has released its prompt lock.

## Best Possible Solution

Review and land the current routed-peer cron delivery fix, or replace it with an equivalent narrow PR, then close this issue only after the fix reaches main.

## Reproduction Assessment

Yes for source reproduction: current main still has a cron delivery mirror path into the resolved destination session, and the session-lock tests show file changes during prompt release trigger `EmbeddedAttemptSessionTakeoverError`. I did not run a live Telegram cron race in this read-only review.

## Solution Assessment

Yes, the best fix direction is the narrow cron delivery-dispatch path already represented by the linked PR: keep the direct channel send, but skip the unsafe routed-peer transcript mirror rather than weakening the embedded session takeover fence.
