Keep open. ## What This Changes

Cron announce delivery to an active routed chat session should avoid mutating the active session transcript or otherwise avoid triggering EmbeddedAttemptSessionTakeoverError.

## Best Possible Solution

Keep this issue open as the canonical cron announce delivery-mirror tracker, then review and land a proof-backed narrow cron mirror guard while leaving broader storage/fallback work to the separate canonical trackers.

## Reproduction Assessment

Yes at source level. Current main and v2026.6.8 still resolve routed peer deliveries and mirror them into that peer session, and recent production comments report the same latest-release failure, though I did not run a live Telegram cron reproduction.

## Solution Assessment

Yes for the cron path: avoiding the unsafe immediate peer transcript mirror is the narrow owner-boundary fix and is better than weakening the embedded session fence. The exact PR still needs real behavior proof and maintainer acceptance of the transcript-history tradeoff.
