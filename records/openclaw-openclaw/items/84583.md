## What This Changes

The issue reports that cron or announce delivery can write into an active Telegram-routed session while an embedded prompt lock is released, causing `EmbeddedAttemptSessionTakeoverError` and a failed visible reply.

## Best Possible Solution

Have maintainers retire or close the cron-specific suppression PR if v2026.6.11's shared session-lock behavior is accepted as the fix, then close this issue with the release proof; otherwise land a narrow, proven PR that explicitly owns the routed-peer transcript persistence tradeoff.

Yes. The source path and v2026.6.11 regression test cover a `delivery-mirror` append while the prompt lock is released and verify no takeover; I did not run a live Telegram cron race in this read-only review.

## Solution Assessment
