## What This Changes

The issue reports that isolated cron announce delivery to the same Telegram user can mutate the active chat session during an embedded prompt window and trigger `EmbeddedAttemptSessionTakeoverError`.

## Best Possible Solution

Review and either land or close https://github.com/openclaw/openclaw/pull/84603; the fix should preserve the visible direct channel send while avoiding immediate transcript mutation of active routed peer sessions.

## Reproduction Assessment

Yes, at source level: current main can resolve explicit cron delivery to a per-channel-peer destination session, append a mirror to that session, and then the embedded lock rejects the changed session file during the released prompt window. I did not run a live Telegram cron reproduction.

## Solution Assessment

Yes, the narrowest maintainable fix is in the cron delivery mirror/awareness path rather than weakening the embedded takeover guard. The linked PR is the active implementation candidate and should be reviewed for proof and persistence semantics before merge.
