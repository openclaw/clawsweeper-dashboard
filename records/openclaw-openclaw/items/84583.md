## Label Justifications

- `P2`: The issue is a concrete cron-plus-Telegram session-state bug with a narrow affected workflow rather than a total core outage.
- `impact:session-state`: The reported failure is caused by a session transcript changing while the embedded runner is preserving session state across a released prompt lock.
- `impact:message-loss`: The takeover error can abort or suppress the active in-flight chat response after cron delivery races the same session.

## Summary

Keep open: current main still has a source-traceable path where isolated cron announce delivery can append a delivery mirror into the resolved Telegram destination session while an embedded prompt has released its lock. A broader open fix for the same same-process session-write contention exists at https://github.com/openclaw/openclaw/pull/84250, so this should stay open until that PR or an equivalent fix is merged and verified against the cron announce case.

## What This Changes

The issue reports that isolated cron announce delivery to the same Telegram user can mutate the active chat session and trigger `EmbeddedAttemptSessionTakeoverError`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84250 or an equivalent ownership-aware session-write fix, then verify that cron announce mirroring no longer trips active-session takeover; add a cron-specific deferral only if maintainers reject the broader fence fix.

## Reproduction Assessment

Yes, by source inspection: isolated cron announce delivery can resolve an explicit Telegram DM target to a per-channel-peer session and append a delivery mirror there, while the embedded runner throws when that session file changes during the released prompt window. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the best current path is to preserve the visible channel send while making same-process owned transcript writes safe for active embedded prompts, as in https://github.com/openclaw/openclaw/pull/84250. Weakening the takeover guard or adding Telegram-only policy would be less maintainable.
