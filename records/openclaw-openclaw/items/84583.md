## Summary

Keep open: current main still has a plausible source path where explicit isolated cron announce delivery can append a delivery-mirror message into a resolved per-channel-peer Telegram session, while the embedded runner treats any external transcript change during the released prompt lock as a takeover.

## What This Changes

The issue reports that isolated cron announce delivery to a Telegram user can modify the same active session transcript and trigger EmbeddedAttemptSessionTakeoverError during an in-flight chat turn.

## Best Possible Solution

Keep the direct channel send, but defer or queue the cron completion awareness for the resolved active destination session instead of immediately appending a delivery-mirror transcript when an isolated cron job targets a chat session that may be in flight.

## Reproduction Assessment

Yes, by source inspection: an isolated cron job with explicit Telegram announce delivery can resolve to a per-channel-peer user session and append a delivery-mirror transcript, while the embedded runner explicitly throws if that session file changes during releaseForPrompt. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the best fix is likely in the cron delivery mirror/awareness path, not in Telegram sending or by weakening the embedded session takeover guard. The fix should preserve the visible channel send while avoiding immediate transcript mutation of a possibly active destination session.
