## Summary

Keep open: current main still has a source-traceable cron delivery mirror path into routed peer sessions, and the linked closing PR remains open and conflicting rather than merged. The broader owned-write session fence fix is on main, but it does not remove the cron-specific mirror into the active destination session, so this issue should stay open until https://github.com/openclaw/openclaw/pull/84603 or an equivalent fix lands.

## What This Changes

Cron announce delivery to a Telegram per-channel-peer session can append a delivery-mirror transcript while that same session is active, triggering the embedded session takeover fence.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84603 or an equivalent cron-owned fix that preserves direct channel delivery while avoiding immediate delivery-mirror writes to active routed peer sessions, with coverage for routed peer, main-session, and threaded-main behavior.

## Reproduction Assessment

Yes by source inspection: direct cron delivery can resolve to a per-channel-peer session and append a delivery-mirror transcript, while the embedded attempt fence still rejects unaccepted prompt-window session file changes. I did not run a live Telegram concurrent cron reproduction in this read-only review.

## Solution Assessment

Yes: the best fix belongs in cron delivery mirroring or awareness, preserving the visible channel send while avoiding immediate transcript mutation of a possibly active routed peer session. Weakening the embedded takeover guard globally would be a broader and riskier solution.
