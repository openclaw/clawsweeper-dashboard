## Summary

Keep open: current main and the latest release still contain the routed-peer cron delivery mirror path, and the open closing pull request is a candidate fix rather than merged implementation.

## What This Changes

The issue reports that cron announce delivery can write into the same active Telegram session transcript while an embedded prompt lock is released, causing `EmbeddedAttemptSessionTakeoverError` and aborting the user turn.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94563 or an equivalent narrow cron dispatcher guard that preserves direct delivery and main-session awareness while preventing unsafe routed-peer transcript mirrors; keep broader non-cron takeover recovery in its own trackers.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 can resolve explicit cron delivery into routed peer sessions and still append transcript mirrors there. I did not live-run the concurrent Telegram race in this read-only review, but the issue has multiple release-version log reports matching the source path.

## Solution Assessment

Yes for the central cron path: the narrowest maintainable fix is to suppress or defer unsafe routed-peer transcript mirrors while preserving direct delivery and main-session awareness. Maintainers still need to accept the resulting transcript-history tradeoff, and broader non-cron announce variants should remain separately tracked.
