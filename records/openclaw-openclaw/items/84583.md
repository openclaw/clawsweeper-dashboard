## Label Justifications

- `P2`: The bug can abort in-flight user turns under concurrent cron/announce delivery, but the trigger is bounded to routed peer delivery races rather than a core-wide outage.
- `impact:session-state`: The failure is caused by a session transcript write racing the embedded prompt-release session fence.
- `impact:message-loss`: The active user turn can surface a generic failure instead of delivering the expected response.

## Summary

Keep open: both current `main` and the latest release still mirror successful routed peer cron deliveries into the resolved peer transcript, and https://github.com/openclaw/openclaw/pull/94563 is an open closing PR rather than merged implementation.

## What This Changes

The issue reports that isolated cron announce delivery can write into the same routed Telegram peer session while an embedded prompt lock is released, causing `EmbeddedAttemptSessionTakeoverError` and a failed active user turn.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/94563 or an equivalent narrow cron mirror guard lands, preserving direct delivery and main-session awareness while avoiding routed peer transcript writes during active embedded sessions.

## Reproduction Assessment

Yes. Source inspection on current `main` and `v2026.6.8` shows isolated cron delivery can resolve to a routed peer session and append a transcript mirror there; recent issue comments also report the same signature on `v2026.6.8`.

## Solution Assessment

Yes. The best fix is a narrow cron dispatcher change that skips unsafe routed-peer transcript mirrors while leaving the direct channel send and main/threaded-main awareness behavior intact.
