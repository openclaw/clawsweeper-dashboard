- `P2`: The issue is a concrete cron-plus-Telegram session-state bug with a narrow affected workflow rather than a core-wide outage.

## Summary

Keep open: current main still has the cron delivery mirror path that can write into a routed peer session, and an open linked implementation PR specifically targets this issue but has not merged.

## What This Changes

The issue asks for cron announce delivery to avoid mutating the active Telegram user session file while an embedded prompt lock is released.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84603 or an equivalent cron-owned fix that preserves direct channel delivery while avoiding immediate transcript mutation of active routed peer sessions.

## Reproduction Assessment

Yes, by source inspection: current main can resolve a cron announce delivery to a per-channel-peer destination session, append a transcript mirror there, and then the embedded runner throws if that same session file changes during the released prompt-lock window. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the best fix shape is in cron delivery mirroring or awareness, not in weakening Telegram send or the embedded takeover guard. The issue should remain open until the linked PR or an equivalent fix merges.
