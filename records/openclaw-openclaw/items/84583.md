## Summary

Keep open: current main and the latest release still mirror routed peer-session cron delivery into the destination transcript, recent field reports say the error still reproduces on v2026.6.8, and an open closing PR now owns the likely fix path.

## What This Changes

Cron announce delivery should avoid mutating an active routed peer-session transcript in a way that trips the embedded attempt session-takeover fence.

## Best Possible Solution

Review and land a narrow cron delivery fix that skips or defers routed peer-session transcript mirrors while preserving direct channel delivery, main-session awareness, and main/threaded-session mirror behavior.

## Reproduction Assessment

Yes for source-level reproduction: current main still resolves a routed peer session and appends the cron delivery mirror to that session, and the issue includes v2026.6.8 logs for the same error pattern. I did not run a live Telegram concurrency repro in this read-only review.

## Solution Assessment

Yes, keeping this issue open behind the linked fix PR is the right current path. The narrow code direction is to stop the unsafe routed peer-session mirror write rather than weakening the session takeover fence globally.
