## Label Justifications

- `P2`: The issue is a concrete cron-plus-Telegram session-state race with a clear but limited affected workflow rather than a whole-runtime outage.
- `impact:session-state`: The reported failure is caused by a same-process transcript write changing the session file while the embedded runner is preserving prompt-release session state.
- `impact:message-loss`: The takeover error can abort or suppress the user's active in-flight chat response after cron delivery races the same session.

## Summary

Keep open: current main still has a source-traceable path where isolated cron announce delivery sends to Telegram and then appends a delivery-mirror assistant message into the resolved per-peer session, while the embedded runner still treats any session-file fingerprint change during prompt release as takeover.

## What This Changes

Cron announce delivery should avoid mutating the active chat session transcript when an isolated job announces to the same Telegram user who is already chatting.

## Best Possible Solution

Preserve the direct channel send, but defer, queue, or skip the immediate delivery-mirror transcript append when an isolated cron announce resolves to an active destination session, while keeping the embedded takeover guard fail-closed for unowned external edits.

## Reproduction Assessment

Yes, by source inspection: an isolated cron announce can resolve to a Telegram per-peer session and append a delivery-mirror transcript while another embedded attempt would throw on any session-file fingerprint change during releaseForPrompt. I did not run a live Telegram reproduction.

## Solution Assessment

Yes, the best fix is in the cron delivery mirror or awareness path, not in Telegram sending and not by broadly weakening the embedded takeover guard. The direct channel send should stay, while the conflicting destination transcript mutation should be deferred or avoided.
