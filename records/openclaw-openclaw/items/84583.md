## Summary

Keep open: current main and v2026.6.8 still contain the routed peer cron transcript-mirror path, and two open pull requests still use closing references for this issue rather than merged implementation.

## What This Changes

The issue requests preventing isolated cron announce/direct delivery from mutating an active Telegram or per-channel-peer session in a way that trips `EmbeddedAttemptSessionTakeoverError`.

## Best Possible Solution

Review and land or replace the narrow cron dispatcher guard from the open candidate PR, preserving direct channel delivery plus main/threaded-main awareness while avoiding unsafe routed peer transcript writes.

## Reproduction Assessment

Yes. Current source and v2026.6.8 both show explicit cron delivery can resolve to a routed peer session and append a transcript mirror there; recent comments add live release logs, but I did not run a live Telegram race in this read-only review.

## Solution Assessment

Yes for the narrow cron path: the best fix is to change the cron dispatcher so routed peer deliveries do not mutate the active peer transcript while direct delivery and main-session awareness continue. The open PRs still need maintainer review for proof and transcript-history tradeoff before this issue closes.
