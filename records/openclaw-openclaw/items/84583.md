## Summary

Keep open: current main still has the cron peer-session delivery-mirror path, and an open conflicting fix PR already references this issue with closing syntax. Recent comments also add same error-class variants on stable, so closing now would get ahead of the implementation path.

## What This Changes

Cron announce delivery should avoid mutating the same active Telegram session while an embedded attempt is released for prompt or tool I/O.

## Best Possible Solution

Keep this issue open until a cron-specific fix, likely route-aware suppression or deferral of delivery-mirror writes for active routed peer sessions, lands; handle the newer compaction self-write variant separately if it proves distinct.

## Reproduction Assessment

Yes, by source inspection: current main still resolves explicit direct cron delivery to a routed peer session and appends a delivery-mirror transcript there. I did not run a live Telegram cron race, and #84250 means the exact runtime failure should still be validated against current main.

## Solution Assessment

For the cron subcase, yes: the maintainable boundary is in cron delivery mirror/awareness routing, not Telegram transport or weakening the takeover guard globally. The newer auto-compaction/inode rewrite report may need a separate session-lock repair if it is not covered by the cron PR.
