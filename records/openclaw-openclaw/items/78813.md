## Summary

Keep open: the PR addresses a real Discord delivery need and now includes convincing after-fix proof, but the current diff still exposes raw Discord components as a shared Gateway `send` field against the existing presentation/channel boundary.

## What This Changes

The PR adds top-level Gateway `send.components`, forwards it into Discord outbound payload data, wraps raw Discord component JSON for serialization, forwards component context through the runtime send proxy, re-exports `ChannelOutboundContext`, and adds a changelog entry.

## Best Possible Solution

Keep raw provider payloads out of shared `send` params; either route this through a Discord-owned request shape such as `channelData.discord.components` or extend the semantic presentation contract after maintainer API direction.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level Gateway API gap: current main has no top-level `components` field, disallows additional send params, and does not strip unknown fields. I did not live-reproduce the claimed silent Discord drop on current main.

## Solution Assessment

Is this the best way to solve the issue? No. The current PR proves the Discord behavior but puts a Discord API-shaped field into the shared Gateway protocol; a Discord-owned input shape or maintainer-approved generic presentation extension is the narrower maintainable path.
