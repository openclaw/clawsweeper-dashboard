## Summary

Keep this PR open for maintainer direction: it proves a useful Discord button delivery path, but the proposed public API shape still conflicts with the current shared presentation/channel-data boundary.

## What This Changes

The PR adds a top-level Gateway `send.components` schema field, forwards it into Discord raw component delivery, updates runtime forwarding, re-exports `ChannelOutboundContext`, regenerates Swift models, and adds a changelog entry.

## Best Possible Solution

Keep the Discord button delivery goal, but land it behind a maintainer-approved channel-owned input shape or the semantic presentation contract instead of a Discord-native field on shared `send`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level gap: a current-main Gateway `send` request with top-level `components` is outside the schema and is rejected by the closed Ajv validator; the contributor also supplied live Discord before/after proof for the desired behavior.

## Solution Assessment

Is this the best way to solve the issue? No. The Discord behavior is useful, but adding Discord-native `components` to the shared Gateway `send` contract conflicts with the documented presentation/channel-owned payload boundary.
