## Summary

Keep open: the real Discord proof is sufficient, but the patch still adds a Discord-native `components` field to the shared Gateway protocol/runtime surface, which conflicts with the current presentation boundary and needs maintainer API direction before merge.

## What This Changes

The PR adds a top-level Gateway `send.components` field, forwards it through Discord outbound payload/runtime paths as raw Discord components, re-exports `ChannelOutboundContext`, regenerates Swift gateway models, and adds a changelog entry.

## Best Possible Solution

Keep the end-to-end Discord component delivery work behind a maintainer-approved channel-owned or semantic presentation API, without adding provider-native fields to the shared Gateway `send` contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level current-main gap: `send` lacks top-level `components` and rejects additional properties. I did not run a live Gateway reproduction, but the PR body supplies live before/after Discord proof for the proposed path.

## Solution Assessment

Is this the best way to solve the issue? No. The PR proves the Discord behavior, but the public input shape is Discord-native in a shared Gateway protocol; a channel-owned request shape or maintainer-approved semantic presentation extension is narrower.
