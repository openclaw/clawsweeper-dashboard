## Summary

Keep open. This PR is the active narrow fix for the linked Telegram polling self-conflict: current main still uses fire-and-forget stale transport close before the next polling cycle, while the PR serializes that teardown at the Telegram plugin lifecycle boundary; I found no blocking functional or security issue.

## What This Changes

This PR makes Telegram polling dirty-rebuild transport acquisition async, awaits stale transport close before returning the replacement transport, updates related tests, and adds a changelog fix entry.

## Best Possible Solution

Land the Telegram plugin-scoped teardown fix after required checks finish, keeping stale transport coordination inside polling-transport-state/session rather than adding core channel policy.

## Reproduction Assessment

Yes. The linked #75852 report has a high-confidence macOS stop/start reproduction, and current main source shows the fire-and-forget close path that lets a replacement getUpdates race stale socket teardown.

## Solution Assessment

Yes. Awaiting TelegramTransport.close() in the transport state is the narrowest maintainable fix because close already owns undici dispatcher teardown and the only production caller needs to await bot creation.
