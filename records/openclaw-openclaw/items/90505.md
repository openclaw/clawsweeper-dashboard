## Summary

Keep open: this member-authored PR is the active fix candidate for the linked killed-subagent task-row bug and the runtime approach looks sound, but it still needs a narrow docs update for the new `lost` to `succeeded` exception before merge.

## What This Changes

The PR marks killed subagent task delivery as not applicable immediately, defers task-row finalization to maintenance using terminal subagent-run state, and allows late same-run completion to upgrade maintenance-inferred lost rows to succeeded.

## Best Possible Solution

Land the runtime repair only after documenting the maintenance-inferred `lost` to same-run `succeeded` exception and maintainer acceptance of the bounded session-state tradeoff.

## Reproduction Assessment

Yes at source level: the linked issue describes killed subagent task rows stuck in `running`, and the PR targets the kill path plus maintenance reconciliation paths that own those rows. I did not run a live gateway race in this read-only review.

## Solution Assessment

Yes with a docs update and maintainer risk acceptance: maintenance-owned finalization avoids the direct kill-path failed write race while preserving a path for authoritative late success. The simpler direct kill-path failure update is less safe because current run-scoped terminal guards would block `failed` to `succeeded`.
