## Summary

Keep open: the wake belongs in this Slack handler conceptually, but the proposed implementation reads `runtime.system` from a production Slack monitor runtime that only exposes logger-style methods, so the new wake path is a no-op outside the test mock.

## What This Changes

The PR changes Slack block-action handling to request an immediate session wake after enqueueing a reply interaction event and adds regression coverage.

## Best Possible Solution

Wake the resolved Slack session through a runtime surface that is actually present when the gateway starts Slack accounts, with a regression test whose mock context matches that startup path.

## Reproduction Assessment

Yes from source inspection: current Slack interactions enqueue system events, and this PR’s attempted wake is absent in the real gateway-started Slack runtime because `runtime.system` is not provided there.

## Solution Assessment

No. The handler location and session key are reasonable, but the wake call must be wired through an actual Slack monitor/gateway runtime surface rather than a test-only `runtime.system` mock.

## Review Findings
