## Summary

Keep open: the validation direction is useful, but the patch would reject an existing accepted runtime alias because it compares raw lowercased ids against manifest ids instead of using the same normalization path as runtime selection. This external PR also still has only test/check proof rather than real after-fix behavior output.

## What This Changes

The PR adds plugin-aware config validation for provider/model-scoped `agentRuntime.id` values and regression tests for provider ids such as `anthropic` and unknown runtime ids.

## Best Possible Solution

Keep the early validation, but normalize runtime ids with the same runtime-selection helper before checking manifest-owned ids and add coverage for accepted aliases such as `codex-app-server`.

## Reproduction Assessment

Yes, from source inspection: current validation accepts arbitrary `agentRuntime.id` strings while harness selection fails closed for unknown non-PI runtimes. The PR's new regression is also source-reproducible because `codex-app-server` normalizes to `codex` at runtime but is not in the PR's raw known-runtime set.

## Solution Assessment

No as submitted. The maintainable solution is the same validation boundary, but it must share runtime-id normalization with selection so diagnostics do not reject currently accepted aliases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve normalized runtime aliases:** `src/config/validation.ts:636-638`
  - body: This check compares the raw lowercased `agentRuntime.id` against manifest-owned ids. Current runtime selection normalizes `codex-app-server` to `codex`, and existing config-shaped tests use that alias; because the Codex manifest only declares `codex`, this patch would reject a config the runtime currently accepts. Normalize with the same runtime helper, or include the supported aliases, before reporting an unknown runtime.
  - confidence: 0.88
