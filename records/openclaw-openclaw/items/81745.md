## Summary

Keep open: the validation gap is real, but the PR is not mergeable as submitted because it would reject an existing normalized runtime alias and the attached proof is only tests/checks rather than real config-validation behavior.

## What This Changes

Adds plugin-aware config validation for provider/model-scoped `agentRuntime.id` values, plus regression tests for provider ids such as `anthropic` and unknown runtime ids.

## Best Possible Solution

Keep the early plugin-aware validation, but normalize runtime ids with the same selection contract and add coverage for accepted aliases such as `codex-app-server` before requesting real CLI/config-validation proof.

## Reproduction Assessment

Yes, from source inspection: current schema accepts arbitrary `agentRuntime.id` strings while explicit unknown runtimes fail later during harness selection. The review finding is also source-reproducible because current selection normalizes `codex-app-server` to `codex` but the PR compares the raw id.

## Solution Assessment

No as submitted. The validation boundary is the right place, but the implementation must share runtime-id normalization with selection so it does not reject accepted aliases.
