## Label Justifications

- `P1`: The issue describes privacy-sensitive cross-session chat routing with duplicate or wrong-surface replies in a real Android plus WhatsApp workflow.
- `impact:security`: Native chat notification text can still enter an unrelated node-session prompt, which is a sensitive cross-context privacy boundary.
- `impact:message-loss`: The reported failure mode includes duplicate and misrouted visible replies to the wrong WhatsApp group.
- `impact:session-state`: The root behavior is native-channel content being associated with a node session rather than the originating WhatsApp session.

## Summary

Keep open. Current main has reduced the worst outbound failure by keeping the message tool out of node-originated runs, but Android-forwarded native chat notifications can still become node-session prompt events and wake runs.

## What This Changes

Request to stop Android notification forwarding from feeding native-channel app notifications into node sessions and causing duplicate or wrong-surface replies.

## Best Possible Solution

Keep the node-safe tool restriction, then add owner-approved native-provider notification filtering before Android or Gateway turns those notifications into node-session prompt events, with tests, docs, and any Gateway config contract aligned.

## Reproduction Assessment

Yes for the remaining forwarding path by source inspection: with Android notification forwarding enabled in empty blocklist mode, native chat app packages are allowed and the Gateway enqueues any valid notifications.changed event. The original wrong-group send is partially mitigated on current main because node-originated runs no longer expose the message tool.

## Solution Assessment

No. Current main is a partial solution: the node-safe tool subset is the right outbound defense, but native-provider notifications should be filtered before they wake or prompt a node session.
