## Label Justifications

- `P1`: The issue describes privacy-sensitive cross-session chat handling that can duplicate or misroute visible replies.
- `impact:security`: Native chat notification text can cross into an unrelated node-session prompt, which is a sensitive context-boundary failure.
- `impact:message-loss`: The reported behavior includes duplicate and wrong-surface WhatsApp replies rather than only internal logging noise.
- `impact:session-state`: The root behavior associates WhatsApp notification content with a node session instead of the originating WhatsApp session.

## Summary

Keep open. Current main has partially reduced the blast radius by removing outbound message-tool access from node-originated runs, but enabled Android notification forwarding can still turn WhatsApp-class notifications into node-session system events and wake the agent.

## What This Changes

The issue reports Android-forwarded native chat notifications entering node sessions and causing duplicate or wrong-target WhatsApp replies.

## Best Possible Solution

Keep the node-safe outbound restriction and add owner-approved native-provider notification filtering before Android or the Gateway turns those notifications into node-session prompt events.

## Reproduction Assessment

Yes, at source level. With Android notification forwarding enabled in blocklist mode, an unlisted WhatsApp-class package passes Android policy, is emitted as notifications.changed, and is enqueued by the Gateway as a node-session system event.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset blocks the reported wrong-group message-tool send, but native-provider notifications should be filtered before they wake or prompt a node session.
