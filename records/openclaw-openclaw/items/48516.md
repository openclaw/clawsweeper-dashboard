## Label Justifications

- `P1`: The issue describes a real channel workflow that can duplicate replies and send output to the wrong WhatsApp group if reproduced.
- `impact:session-state`: The failure centers on notification events entering the wrong node session context and affecting later reply routing.
- `impact:security`: A reply derived from one conversation can be sent into another conversation, which is a privacy and authorization boundary concern.
- `impact:message-loss`: The reported outcome is duplicated or misrouted outbound chat messages.

## Summary

Keep open. Current main has useful mitigations, but allowed Android `notifications.changed` events still become prompt-visible node-session system events, and heartbeat runs can still inherit heartbeat or persisted channel routing rather than the node-only tool policy, so the reported cross-session outbound risk is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded native-channel notifications from waking node sessions that can duplicate or misroute WhatsApp replies.

## Best Possible Solution

Make native-channel notification events have one owner and enforce outbound channel, peer, and session binding before node-originated notification wakes can produce visible sends.

## Reproduction Assessment

Yes, source-reproducible in part: current code still allows accepted Android notifications to become queued system events in a node session, but the exact WhatsApp wrong-group send needs live Android plus WhatsApp confirmation.

## Solution Assessment

No: the shipped Android controls and node direct-command allowlist are useful mitigations, but they do not prove notification-triggered heartbeat runs are bound to node-safe outbound policy or that native-channel packages are suppressed.
