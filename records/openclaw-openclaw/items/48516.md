## Summary

Keep open. Current main and v2026.6.10 still allow accepted Android `notifications.changed` events to become node-session system events and heartbeat wakes, so the reported notification-to-node-session path is not proven fixed even though Android forwarding controls now reduce exposure.

## What This Changes

The issue reports duplicate WhatsApp handling when Android notification forwarding turns native-channel notifications into node-session events, allowing replies from the wrong session or group.

## Best Possible Solution

Make native-channel notification ownership explicit and enforce channel, peer, and session binding before notification-triggered node sessions can produce visible sends.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full live wrong-group WhatsApp outcome; yes at source level for the remaining path where allowed Android notifications become node-session system events and heartbeat wakes.

## Solution Assessment

Is this the best way to solve the issue? Unclear: Android forwarding controls are useful mitigations, but the best fix likely needs a maintainer decision about native-channel package ownership, gateway rejection, prompt handling, and outbound tuple binding.
