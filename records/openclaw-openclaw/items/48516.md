- `impact:security`: The issue is about cross-session leakage where private channel content can lead to an unauthorized reply in another group.

- `impact:session-state`: The failure depends on WhatsApp session context and Android node-session context drifting for the same inbound message.

## Summary

Current main has partial mitigations, but this should stay open: Android notification forwarding is configurable and off by default, yet enabled blocklist mode still allows native chat-app packages unless the operator manually blocks them, and the gateway still accepts any `notifications.changed` payload into a node-scoped system-event wake.

## What This Changes

The issue asks OpenClaw to prevent Android companion notification forwarding from duplicating native channel messages into node sessions and causing replies in the wrong chat.

## Best Possible Solution

Keep this issue open until maintainers choose and validate a canonical boundary that default-excludes or server-rejects native-channel app notifications and proves notification-triggered node sessions cannot send through unrelated channel routes.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. The injection path is source-reproducible: allowed Android notifications can still become `notifications.changed` system events in a node session, but the final wrong-group send needs real Android plus WhatsApp proof.

## Solution Assessment

Unclear as an implementation shape. Android default exclusions, gateway defense-in-depth, and node-session delivery isolation are all plausible, and maintainers need to pick the security boundary before code work proceeds.
