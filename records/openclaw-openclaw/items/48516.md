## Summary

Keep open. Current main mitigates the most dangerous wrong-group outbound path by excluding the `message` tool from node-originated runs, but Android and the gateway still allow native messaging-app notifications to enter node-session prompt context when forwarding is enabled.

## What This Changes

The issue requests suppression/filtering of Android-forwarded notifications for apps already handled by native channel providers, plus gateway-side package filtering and node-session send isolation.

## Best Possible Solution

Keep the node-safe tool restriction, then suppress native-provider notifications before they become node-session prompt events through Android defaults/settings, gateway filtering, regression tests, docs, and changelog.

## Reproduction Assessment

Yes for the remaining forwarding path by source inspection: with Android notification forwarding enabled in empty blocklist mode, `com.whatsapp`-style packages are allowed and the gateway enqueues any valid `notifications.changed` event. The original wrong-group send is no longer source-reproducible because node-originated runs do not expose the `message` tool.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset is the right outbound defense, but native-provider notifications should be suppressed before they wake or prompt a node session.
