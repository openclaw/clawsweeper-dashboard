## Label Justifications

- `P1`: The reported workflow can produce public replies in the wrong WhatsApp group when Android notification forwarding and native WhatsApp handling both process the same message.
- `impact:security`: The issue concerns cross-session leakage where native chat content can influence a reply outside the intended channel boundary.
- `impact:message-loss`: The failure mode duplicates and misroutes outbound chat replies instead of producing one response to the intended target.
- `impact:session-state`: The bug depends on WhatsApp session context and Android node-session context drifting for the same inbound message.

## Summary

Keep open. Current main and v2026.6.6 still have an enabled Android notification-forwarding path that can pass allowed third-party app notifications into node-session system events, so this is still a security/product-boundary issue rather than safe cleanup.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded native-channel notifications, especially WhatsApp, from creating duplicate node-session events and misrouted outbound replies.

## Best Possible Solution

OpenClaw should default-suppress or server-reject notifications from natively owned channel apps and enforce outbound channel/session binding so node notification events cannot send into unrelated WhatsApp targets.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the Android notification-to-node-system-event path still exists when forwarding is enabled, but the wrong-group WhatsApp send needs real Android plus WhatsApp proof on current main.

## Solution Assessment

Unclear until maintainers choose the boundary. A combined Android default suppression, gateway defense, and outbound session-binding fix is safer than only adding an operator config knob.
