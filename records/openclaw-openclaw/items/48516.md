## Label Justifications

- `P1`: The issue describes a privacy-sensitive cross-session chat path that can duplicate or misroute visible replies.
- `impact:security`: Native chat notification text can cross into an unrelated node-session prompt, which is a sensitive context-boundary failure.
- `impact:message-loss`: The reported behavior includes duplicate and wrong-surface WhatsApp replies rather than only internal logging noise.
- `impact:session-state`: The root behavior associates WhatsApp notification content with a node session instead of the originating WhatsApp session.

## Summary

Keep open. Current main and latest release partially mitigate the worst wrong-group send by excluding the message tool from node-originated runs, but Android and Gateway still allow policy-permitted native chat notifications to become node-session system events.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded notifications from native channel apps such as WhatsApp from entering node sessions and causing duplicate or misrouted replies.

## Best Possible Solution

Keep the node-safe outbound restriction and add owner-approved native-provider notification filtering before Android or Gateway routes those notifications into node-session prompt events.

## Reproduction Assessment

Yes, at source level. With Android notification forwarding enabled in blocklist mode and a WhatsApp-class package not configured as blocked, the listener emits `notifications.changed` and Gateway enqueues it as a node-session system event that wakes the agent.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset blocks the reported wrong-group message-tool send, but native-provider notifications should be filtered before they wake or prompt a node session.
