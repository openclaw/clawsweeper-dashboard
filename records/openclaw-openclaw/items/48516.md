- `impact:security`: The issue is about cross-session leakage where native chat content can trigger an unauthorized reply in another group.
- `impact:message-loss`: The reported behavior duplicates and misroutes outbound channel messages instead of sending exactly one response to the intended chat.
- `impact:session-state`: The failure depends on WhatsApp session context and Android node-session context drifting for the same inbound message.

## Summary

Keep this open: current main and the latest release still only partially mitigate the report, because enabled Android notification forwarding can still pass native chat app packages unless manually blocked and the gateway still turns `notifications.changed` payloads into node-session heartbeat system events without a native-channel package boundary.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded native chat notifications from creating duplicate cross-session replies, especially WhatsApp replies sent to the wrong group.

## Best Possible Solution

Keep this open until maintainers choose and validate a canonical boundary that default-excludes or server-rejects native-channel app notifications and proves notification-triggered node/system-event sessions cannot send through unrelated channel routes.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. The injection path is source-reproducible on current main: allowed Android notifications can still become `notifications.changed` system events in a node session, but the final wrong-group WhatsApp send needs real Android plus WhatsApp proof.

## Solution Assessment

Unclear as an implementation shape. Android default exclusions, gateway defense-in-depth, and node/session delivery isolation are all plausible, and maintainers need to pick the security boundary before code work proceeds.
