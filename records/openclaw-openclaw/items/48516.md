## Summary

Keep open. Current main and v2026.6.6 include safer Android notification-forwarding controls, but accepted third-party notifications can still become `notifications.changed` system events in a node session, so the reported native-channel collision and wrong-session outbound risk is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded notifications from native channel apps from triggering duplicate cross-session WhatsApp replies.

## Best Possible Solution

Make native-channel inbound messages have one owner: suppress or reject Android notification events for channel-owned packages and add session/outbound binding proof before node notification wakes can send across channels.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source and tests show allowed Android notifications still enqueue and wake a node session, but the WhatsApp wrong-group outbound send still needs a real Android plus WhatsApp live repro on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: the shipped Android controls are useful mitigation, but the best complete fix needs maintainer choice on native-channel package ownership and outbound session binding rather than only relying on operator-selected package filters.
