## Label Justifications

- `P1`: The reported path can produce public replies in the wrong WhatsApp group when Android notification forwarding and native WhatsApp handling both process the same message.
- `impact:security`: The issue concerns cross-session leakage where native chat content can influence a reply outside the intended channel boundary.
- `impact:message-loss`: The failure mode duplicates and misroutes outbound chat replies instead of producing one response to the intended target.
- `impact:session-state`: The bug depends on WhatsApp session context and Android node-session context drifting for the same inbound message.

## Summary

Keep this open. Current main and v2026.6.6 still allow enabled Android notification forwarding to pass native-channel app notifications into node-session system events, and the remaining fix needs security/product boundary review plus live Android and WhatsApp proof.

## What This Changes

The issue requests preventing Android-forwarded native chat notifications from creating duplicate node-session replies, especially WhatsApp replies sent to the wrong group.

## Best Possible Solution

OpenClaw should default-suppress or server-reject notifications from natively owned channel apps and enforce outbound channel/session binding so node notification events cannot send into unrelated WhatsApp targets.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows allowed Android notifications can still become node-session system events, but the wrong-group WhatsApp send needs real Android plus WhatsApp proof on current main.

## Solution Assessment

Unclear until maintainers choose the boundary. A combined Android default suppression, gateway defense, and outbound session-binding fix is safer than only adding an operator config knob.
