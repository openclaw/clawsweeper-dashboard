## Label Justifications

- `P1`: The issue describes privacy-sensitive cross-session chat routing and duplicate or wrong-surface replies in a real Android plus WhatsApp workflow.
- `impact:security`: Native chat notification text can enter an unrelated node-session prompt, crossing a sensitive context boundary.
- `impact:message-loss`: The reported failure mode includes duplicate and misrouted visible replies to the wrong WhatsApp group.
- `impact:session-state`: The root behavior associates native-channel content with a node session instead of the originating WhatsApp session.

## Summary

Keep open. Current main mitigates the worst outbound wrong-group send by removing the message tool from node-originated runs, but Android-forwarded WhatsApp-class notifications can still become node-session system events and heartbeat wakes; the remaining default/filter shape needs maintainer product and security review.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded notifications from native chat apps from creating duplicate node-session prompts and cross-session outbound replies.

## Best Possible Solution

Keep the node-safe tool restriction, then add owner-approved native-provider notification filtering before Android or Gateway turns those notifications into node-session prompt events, with tests and docs/config contract updates aligned.

## Reproduction Assessment

Yes. Source inspection shows that with Android notification forwarding enabled in blocklist mode, unlisted native chat packages are allowed, emitted as notifications.changed, and enqueued by the Gateway as node-session system events; I did not run a live Android/WhatsApp E2E.

## Solution Assessment

No. Current main is only a partial solution: the node-safe tool subset is the right outbound defense, but native-provider notifications should be filtered before they wake or prompt a node session.
