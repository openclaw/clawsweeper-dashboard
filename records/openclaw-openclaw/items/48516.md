## Summary

Keep open: current main has Android notification forwarding controls, but enabled blocklist mode with no selected packages still forwards native chat app notifications, and the gateway still turns `notifications.changed` into node-session events. This remains a security/session-routing issue that needs maintainer policy choice and live repro before a fix path is selected.

## What This Changes

The issue reports Android notification forwarding duplicating native WhatsApp handling and allowing a node session to send replies into the wrong WhatsApp group.

## Best Possible Solution

Pick one owner-approved safety boundary, then implement native-chat notification exclusion and/or node-session outbound restrictions with Android, gateway, and outbound routing regression coverage.

## Reproduction Assessment

Partly yes: source inspection reproduces the duplicate ingress path when Android notification forwarding is enabled and no package block is configured. No live current-main WhatsApp wrong-group send was established in this read-only review.

## Solution Assessment

Unclear: Android-side default exclusion, gateway-side rejection, and outbound peer/session locks are all plausible, but the best permanent fix needs maintainer security-policy approval.
