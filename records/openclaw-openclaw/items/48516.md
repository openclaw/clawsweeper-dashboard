## Summary

Current main has useful mitigations, but the central routing hazard is not closed: once Android notification forwarding is enabled, native chat app notifications are not excluded by default and the gateway still accepts `notifications.changed` events into node sessions. Because the report is security- and session-isolation-sensitive, this should stay open for a maintainer decision rather than be swept as stale or implemented.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded notifications from native channel apps from creating duplicate node-session replies to the wrong WhatsApp group.

## Best Possible Solution

Keep one canonical issue open until OpenClaw either default-excludes native-channel app notifications or rejects them server-side, and also proves notification-triggered node sessions cannot send through unrelated channel routes.

## Reproduction Assessment

Yes from source inspection, but not from a live device run. The Android listener can still forward allowed WhatsApp notifications, and the gateway still converts `notifications.changed` into a node-session heartbeat event.

## Solution Assessment

Unclear as an implementation shape. Android-side package defaults, gateway defense-in-depth, and node-session tool isolation are all plausible, but the security-sensitive owner boundary needs maintainer selection before code work proceeds.
