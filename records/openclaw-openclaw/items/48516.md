## Label Justifications

- `P1`: The issue describes privacy-sensitive cross-session chat handling that can duplicate or misroute visible replies.
- `impact:security`: Native chat notification text can cross into an unrelated node-session prompt boundary.
- `impact:message-loss`: The reported behavior includes duplicate and wrong-surface WhatsApp replies.
- `impact:session-state`: The root behavior associates WhatsApp notification content with a node session instead of the originating WhatsApp session.

## Summary

Keep open. Current main and the latest release still allow enabled Android notification forwarding to turn WhatsApp-class notifications into node-session system events; the later node-safe tool restriction mitigates the reported wrong-group send path, but does not stop native chat notification text from entering and waking the node session.

## What This Changes

The issue requests filtering Android-forwarded native chat notifications and preventing node-session replies from duplicating or misrouting WhatsApp messages.

## Best Possible Solution

Keep the node-safe outbound restriction and add maintainer-approved native-provider notification filtering before Android or the Gateway turns those notifications into node-session prompt events.

## Reproduction Assessment

Yes, at source level. With Android notification forwarding enabled in blocklist mode, an unlisted WhatsApp-class package passes Android policy, is emitted as notifications.changed, and is enqueued by the Gateway as a node-session system event.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset blocks the reported wrong-group message-tool send, but native-provider notifications should be filtered before they wake or prompt a node session.
