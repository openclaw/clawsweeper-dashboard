## Summary

Keep this open: current main has partial Android-side controls, but the central security/session-routing hazard remains when notification forwarding is enabled because native chat app notifications are not excluded by default and the gateway still converts `notifications.changed` payloads into node-session heartbeat work.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded notifications from native-handled chat apps, especially WhatsApp, from creating duplicate node-session replies to the wrong chat target.

## Best Possible Solution

Keep one canonical issue open until OpenClaw chooses and proves a combined boundary: native-channel app notifications are excluded or rejected by default, and notification-triggered node sessions cannot send through unrelated channel routes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows the enabled forwarding path still accepts allowed WhatsApp-like notifications into node-session heartbeat work, but the full wrong-group send needs a live Android plus WhatsApp proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the boundary; Android-only filtering is a useful mitigation, but gateway rejection and outbound session/tool isolation may also be needed for a complete fix.
