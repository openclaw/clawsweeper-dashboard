## Summary

Keep open. Current main has partially reduced the dangerous outbound-send path by excluding the `message` tool from node-originated runs, but the reported native-notification forwarding gap remains: enabled Android blocklist mode still forwards WhatsApp/Telegram/Discord/Signal-style packages unless manually blocked, and the gateway still accepts any valid `notifications.changed` package as an untrusted system event that wakes heartbeat.

## What This Changes

The issue requests preventing Android-forwarded notifications from native channel apps from creating duplicate node-session processing and cross-session outbound replies.

## Best Possible Solution

Keep the node-originated `message` tool restriction, then add layered native-provider suppression: Android should ship default denies or a narrow allowlist posture for native OpenClaw channel packages, the gateway should support and enforce notification package filtering before enqueueing/waking heartbeat, and regression tests should cover WhatsApp-class packages plus generic allowed packages.

## Reproduction Assessment

Yes for the remaining forwarding bug: with Android notification forwarding enabled in empty blocklist mode, the source-level policy allows a `com.whatsapp` notification and the gateway enqueues any valid `notifications.changed` package. No live device run was performed, and the original wrong-group send is no longer reproduced from source because node-originated runs do not expose the `message` tool.

## Solution Assessment

No, current main is only a partial fix. The node-safe tool subset is the right defense for outbound leakage, but it does not satisfy the requested native-provider notification suppression or the server-side filtering defense-in-depth.
