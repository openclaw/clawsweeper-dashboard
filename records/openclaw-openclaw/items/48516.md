## Summary

Keep open. Current main has useful Android notification-forwarding controls and node-safe tool filtering, but accepted `notifications.changed` events can still become prompt-visible node-session system events and the remaining cross-session outbound boundary is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded native app notifications from causing duplicate or wrong-session WhatsApp replies.

## Best Possible Solution

Make native-channel notification ownership explicit and enforce channel, peer, and session binding before notification-triggered node sessions can produce visible sends.

## Reproduction Assessment

Yes, at source level: current main still lets allowed Android `notifications.changed` events become queued system events and heartbeat wakes. The exact WhatsApp wrong-group send still needs live Android plus WhatsApp proof.

## Solution Assessment

Unclear: Android forwarding controls are useful mitigations, but the best fix likely needs a maintainer decision about native-channel package ownership, gateway rejection, prompt handling, and outbound tuple binding.
