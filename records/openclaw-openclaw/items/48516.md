## Label Justifications

- `P1`: The reported workflow can produce public replies in the wrong WhatsApp group when Android notification forwarding and native WhatsApp handling both process the same message.
- `impact:security`: The issue concerns cross-session leakage where native chat content can influence a reply outside the intended channel boundary.
- `impact:message-loss`: The failure mode duplicates and misroutes outbound chat replies instead of producing one response to the intended target.
- `impact:session-state`: The bug depends on WhatsApp session context and Android node-session context drifting for the same inbound message.

## Summary

Keep open. Current main has mitigations, but Android forwarding can still be enabled for third-party app notifications and the gateway still turns any accepted `notifications.changed` payload into a node-session heartbeat without native-channel package ownership checks.

## What This Changes

Prevent Android-forwarded notifications from native messaging apps from entering node-session auto-reply paths and causing cross-session WhatsApp replies.

## Best Possible Solution

Define one canonical ownership rule: native-channel app notifications should be suppressed before node-session prompting, and any remaining node-originated reply must be bound to its source session/channel.

## Reproduction Assessment

No live reproduction was run; source inspection shows enabled Android forwarding can still pass allowed third-party app notifications into node-session system events, but current wrong-group WhatsApp delivery needs live Android plus WhatsApp proof.

## Solution Assessment

Partially implemented but not complete; Android controls and node-safe filtering reduce risk, but relying on operator package choices is not the best fix for a native-provider ownership conflict.
