## Summary

Keep open. Current main has shipped a partial mitigation for the dangerous outbound path by excluding the `message` tool from node-originated runs, but the remaining native-notification forwarding gap is still present in Android and the gateway.

## What This Changes

The issue requests preventing Android notification forwarding from duplicating native provider messages and causing node-session cross-channel replies.

## Best Possible Solution

Keep the node-safe tool restriction, then add layered native-provider suppression before notifications become node-session prompt events: Android defaults or allowlist posture, gateway package filtering before enqueue/wake, regression tests, docs, and changelog.

## Reproduction Assessment

Yes for the remaining forwarding path: with Android notification forwarding enabled in empty blocklist mode, source inspection shows `com.whatsapp`-style packages are allowed and the gateway enqueues any valid `notifications.changed` package. The original wrong-group send is no longer source-reproducible because node-originated runs do not expose the `message` tool.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset is the right outbound defense, but native-provider notifications should still be suppressed in Android and defensively filtered in the gateway before they wake a node session.
