## Label Justifications

- `P1`: The issue describes a privacy-sensitive real channel workflow where duplicate node-session handling can misroute visible chat replies.
- `impact:security`: Native chat notification text can cross into an unrelated node-session prompt, which is a sensitive context-boundary failure.
- `impact:message-loss`: The reported failure mode includes duplicate or wrong-surface WhatsApp replies rather than a purely internal diagnostic problem.
- `impact:session-state`: The root behavior associates native WhatsApp content with a node session instead of the originating WhatsApp session.

## Summary

Keep open. Current main mitigates the worst wrong-group outbound send by limiting node-originated tool access, but Android notification forwarding and the Gateway still allow policy-approved native chat notifications to become node-session system events and wake the agent.

## What This Changes

The issue asks OpenClaw to prevent Android-forwarded native chat notifications, especially WhatsApp, from being processed as separate node-session events that can duplicate or misroute replies.

## Best Possible Solution

Keep the reduced node tool surface and add owner-approved native-provider notification filtering before Android or the Gateway turns those notifications into node-session prompt events, with tests and docs/config contracts aligned.

## Reproduction Assessment

Yes, at source level. With Android notification forwarding enabled in blocklist mode, unlisted WhatsApp-class packages pass the Android policy, are emitted as notifications.changed, and are enqueued by the Gateway as node-session system events; I did not run a live Android/WhatsApp E2E.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset is the right outbound defense, but native-provider notifications should be filtered before they wake or prompt a node session.
