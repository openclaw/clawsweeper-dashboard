## Label Justifications

- `P1`: The issue describes a privacy-sensitive cross-session chat path that can duplicate or misroute visible replies.
- `impact:security`: Native chat notification text can cross into an unrelated node-session prompt, which is a sensitive context-boundary failure.
- `impact:message-loss`: The reported behavior includes duplicate and wrong-surface WhatsApp replies rather than only internal logging noise.
- `impact:session-state`: The root behavior associates WhatsApp notification content with a node session instead of the originating WhatsApp session.

## Summary

Keep open. Current main and the latest release partially mitigate the worst wrong-group send by keeping the message tool out of node-originated runs, but enabled Android notification forwarding can still turn WhatsApp-class notifications into node-session system events and wake the node session.

## What This Changes

The issue requests preventing native-channel mobile notifications, especially WhatsApp, from entering node-session prompt handling and causing duplicate or cross-session replies.

## Best Possible Solution

Keep the node-safe outbound restriction and add owner-approved native-provider notification filtering before Android or the Gateway turns those notifications into node-session prompt events.

## Reproduction Assessment

Yes, at source level. With Android notification forwarding enabled in blocklist mode and a WhatsApp-class package not configured as blocked, the listener emits notifications.changed and the Gateway enqueues it as a node-session system event that wakes the agent.

## Solution Assessment

No, current main is only a partial solution. The node-safe tool subset blocks the reported wrong-group message-tool send, but native-provider notifications should be filtered before they wake or prompt a node session.
