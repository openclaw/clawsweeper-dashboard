## Summary

Keep open. Current main mitigates the highest-risk wrong-group outbound send by keeping the `message` tool out of node-originated runs, but source inspection still shows native chat-app notifications can enter node-session system events when Android notification forwarding is enabled.

## What This Changes

The issue asks OpenClaw to stop Android notification forwarding from turning native-channel WhatsApp, Telegram, Discord, and Signal notifications into node-session events and cross-session replies.

## Best Possible Solution

Keep the node-safe tool restriction, then suppress native-provider notifications before they become node-session prompt events, with Android defaults/settings, any Gateway filter contract, tests, docs, and changelog aligned.

## Reproduction Assessment

Yes for the remaining forwarding path by source inspection: with Android notification forwarding enabled in empty blocklist mode, `com.whatsapp`-style packages are allowed and the Gateway enqueues any valid `notifications.changed` event. The original wrong-group send is no longer source-reproducible because node-originated runs do not expose the `message` tool.

## Solution Assessment

No. Current main is only a partial solution: the node-safe tool subset is the right outbound defense, but native-provider notifications should be filtered before they wake or prompt a node session.
