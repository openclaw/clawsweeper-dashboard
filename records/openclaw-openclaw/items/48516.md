## Summary

Keep this open: current main and the latest release have partial Android notification-forwarding controls, but enabled forwarding can still pass native chat-app notifications into a node-scoped system event and the gateway still has no package/native-provider filter for `notifications.changed`.

## What This Changes

The issue requests preventing Android companion notification forwarding from duplicating natively handled chat messages and enabling cross-session replies to the wrong WhatsApp group.

## Best Possible Solution

Keep this as a security-sensitive routing bug until maintainers choose a canonical boundary, likely default native-provider notification exclusions plus gateway-side defense and a stricter node-originated message-send fence.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection does show the reported path remains possible when Android notification forwarding is enabled and the package is allowed.

## Solution Assessment

No, current main is only a partial mitigation. The safest solution should remove the duplicate ownership path by default and fence node-originated message sends, rather than relying only on manual package selection.
