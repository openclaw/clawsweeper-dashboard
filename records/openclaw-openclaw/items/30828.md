## Summary

Keep open: current main still has only reactive rate-limit handling and auth-profile cooldown/fallback behavior, while the requested manual token/request interval budgets need a new public model scheduling and config design.

## What This Changes

Allow users to configure model rate limits for tokens and requests over a time interval.

## Best Possible Solution

Add a documented core model request budget system for per-provider/per-model, likely per-auth-profile, request and token windows that schedules before shared model calls and reconciles actual usage after responses or streams.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main shows no config/schema/docs surface or model-path scheduler that can express token/request interval budgets.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduling semantics. A shared core model-request budget layer integrated with fallback, auth-profile cooldowns, streaming usage reconciliation, and optional provider header feedback is the maintainable direction.
