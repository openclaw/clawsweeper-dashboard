## Summary

Keep open: current main still handles model rate limits reactively through SDK retry caps and auth-profile cooldown/fallback behavior, but it does not expose a configurable token/request interval budget or preflight scheduler.

## What This Changes

The issue requests configurable model rate limits for tokens per interval and requests per interval.

## Best Possible Solution

Add a documented core model-request budget layer that can key limits by provider/model and likely auth profile, preflight queues requests, and reconciles observed token/request usage after responses or streams.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence absence check: the config type, strict schema, docs, and model transport do not expose token/request interval budgets.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduling semantics. A shared budget layer with optional provider-header feedback is the maintainable direction; the current cooldown/fallback behavior is not enough.
