## Summary

Keep open: current main still has reactive rate-limit handling and auth-profile cooldown/fallback behavior, but no public model scheduler or config surface for manual token/request interval budgets.

## What This Changes

Request to let users configure model token-per-interval and request-per-interval limits so OpenClaw can throttle model calls before providers reject them.

## Best Possible Solution

Add a documented core model-request budget layer that can key limits by provider/model and likely auth profile, schedules before shared model calls, and reconciles actual token/request usage after responses or streams.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main shows no config/schema/docs surface or model-path scheduler that can express token/request interval budgets.

## Solution Assessment

Unclear until maintainers choose the public config contract and scheduling semantics. A shared model-request budget layer with optional provider-header feedback is the maintainable direction; the current reactive cooldown/fallback logic is not enough.
