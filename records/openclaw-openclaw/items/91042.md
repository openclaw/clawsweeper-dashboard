## Summary

Keep open: current main and the latest release still send ReplyChain bodies and fallback ReplyToBody through the generic JSON sanitizer, so long bodies are prefix-truncated at 2,000 chars and the reported tail-loss path remains. The related open pull request adds a long-body sanitizer for chat_window and Telegram inline quotes, but its head still leaves the two paths from this issue on sanitizePromptBody before JSON serialization.

## What This Changes

The issue asks OpenClaw to apply body-aware head+tail truncation to ReplyChain entries and fallback ReplyToBody JSON blocks instead of prefix-only JSON string truncation.

## Best Possible Solution

Land one shared body-aware truncation path for reply-context bodies, preferably folded into https://github.com/openclaw/openclaw/pull/87909 before it merges or as a narrow follow-up with ReplyChain and fallback ReplyToBody regression tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: ReplyChain and fallback ReplyToBody bodies flow into formatUntrustedJsonBlock and then truncateUntrustedJsonString, which preserves only the prefix beyond 2,000 chars. I did not execute a runtime test because this review is read-only.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse or introduce one body-aware sanitizer at the reply-context boundary and cover both JSON paths with sentinel-tail tests. A line-based rewrite is possible but broader than necessary for this bug.
