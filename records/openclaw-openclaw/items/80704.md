## Summary

Close: current main already re-resolves WhatsApp runtime config for each inbound Web message, so group gating now sees updated `channels.whatsapp.groups` instead of the listener startup snapshot; the fix is included in `v2026.5.10-beta.4`.

## What This Changes

Hot-reload changes to `channels.whatsapp.groups` should either update the active WhatsApp group allowlist or be surfaced as restart-required instead of silently dropping newly allowed groups.

## Best Possible Solution

Keep the current per-message config loader in the WhatsApp listener path so policy-only WhatsApp config changes remain hot-reloadable without forcing a channel or gateway restart.

## Reproduction Assessment

Yes, by source inspection and reporter logs: the failing release path used a startup listener config while the current path loads runtime config per inbound message before group gating. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes. Refreshing the policy input at the active listener/message boundary is the narrow maintainable fix here, and it avoids marking all `channels.whatsapp` policy-only writes as restart-required.
