## Summary

Current main already implements the requested hot-reload behavior: the active WhatsApp Web listener now reloads the runtime config for each inbound message and passes that fresh config through group gating, including `channels.whatsapp.groups`. The fix shipped in `v2026.5.10-beta.4`; `v2026.5.7` predates it.

## What This Changes

The issue asks for WhatsApp `channels.whatsapp.groups` changes made through `config.patch` to affect the active group allowlist without silently dropping messages, or to be marked restart-required.

## Best Possible Solution

Keep the fresh-runtime-config implementation and treat any remaining non-hot-reload WhatsApp group-dispatch symptoms as a separate report, such as https://github.com/openclaw/openclaw/issues/80655.

## Reproduction Assessment

Yes for the historical release path: `v2026.5.7` used startup `params.cfg` for group gating after only route lookup read fresh config, matching the stale allowlist report. Current main no longer reproduces from source because the active listener passes fresh runtime config into the group gate.

## Solution Assessment

Yes. Current main implements the narrower maintainable option: hot-reload by re-reading runtime config on inbound WhatsApp messages, instead of adding a restart-required classifier for `channels.whatsapp.groups`.
