## Summary

Current main already fixes the reported stale WhatsApp group allowlist path: the active WhatsApp listener now reloads runtime config for each inbound message and passes that refreshed config into group gating, so newly added `channels.whatsapp.groups` entries are evaluated without a full process restart. The fix is present on main after v2026.5.7 and does not appear to have shipped in a release yet.

## What This Changes

The issue reports that WhatsApp group allowlist changes under `channels.whatsapp.groups` are not reflected after SIGUSR1 hot reload, causing newly allowed groups to be dropped until a full restart.

## Best Possible Solution

Keep the current-main hot-reload implementation and ship it in the next release; no new config option or restart-only policy is needed for this specific allowlist path.

## Reproduction Assessment

Yes. The v2026.5.7 source path passed the startup `params.cfg` into group gating, while current main passes a freshly loaded runtime config through the same path; I did not need a live WhatsApp group to identify the stale-config boundary.

## Solution Assessment

Yes. The current-main fix is the narrow maintainable solution because it refreshes the existing runtime config at the active inbound boundary instead of adding a new operator-facing restart rule for `channels.whatsapp.groups`.
