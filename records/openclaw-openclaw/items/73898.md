## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: backend TTS/media delivery exists, but assistant audio still has only manual playback in the browser UI and no persisted opt-in preference.

## What This Changes

Add a Control app or Control UI setting that auto-plays newly generated assistant TTS audio after explicit user opt-in.

## Best Possible Solution

Add a browser-local, default-off assistant audio playback preference, normalize gateway rich audio blocks into renderable attachments, auto-play only newly arrived assistant audio after an explicit user gesture, and show a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable `messages.tts.auto`, let the gateway produce assistant audio media, then observe that Control UI has no playback-consent setting and renders available audio only through manual controls.

## Solution Assessment

Yes for the requested browser Control UI direction. A local opt-in playback setting is the narrow maintainable solution because backend TTS generation already exists and browser autoplay policy requires explicit user consent.
