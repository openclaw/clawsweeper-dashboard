## Summary

Keep open. Current main still renders assistant TTS audio as a manual `<audio controls>` attachment and has no persisted Control UI playback-consent setting or blocked-autoplay recovery prompt.

## What This Changes

Add a default-off Control app or Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Add a browser-local playback-consent preference that defaults off, persists per browser/profile, plays only newly arrived assistant audio attachments when ready, and shows a compact enable/retry prompt when browser autoplay blocks playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-TTS can attach audio to assistant replies, but the Control UI render path only creates a manual audio control and has no persisted opt-in playback setting or rejected `play()` handling.

## Solution Assessment

Yes for the browser Control UI portion. A local explicit opt-in is the narrowest maintainable fix because backend TTS generation already works and backend-driven autoplay cannot satisfy browser sound autoplay policy or user expectations.
