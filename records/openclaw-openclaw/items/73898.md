## Summary

Keep open. Current main still lacks an opt-in playback-consent setting: browser Control UI audio attachments render as manual media controls, and the native chat surfaces still treat assistant audio as generic or unsupported attachment content.

## What This Changes

The issue requests a default-off Control app / Control UI setting that auto-plays newly generated assistant TTS audio after explicit user consent.

## Best Possible Solution

Add a UI-owned, default-off playback-consent preference that persists locally, auto-plays only newly arrived assistant audio when ready, keeps manual controls visible, and shows a retry prompt when play() is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS attaches audio to replies, browser rendering is a manual audio-control path with no persisted playback consent, and native chat surfaces do not implement assistant audio playback.

## Solution Assessment

Yes for a UI-level opt-in setting. Backend-driven autoplay would not satisfy browser or platform consent expectations; native app parity should be an explicit scoped addition rather than mixed into Talk mode playback.

## Repair Work Prompt

Implement the browser Control UI portion of https://github.com/openclaw/openclaw/issues/73898. Add a default-off, browser-local setting such as "Auto-play assistant TTS audio" that is enabled only by explicit user interaction, persists in UiSettings/local storage, and is wired into chat rendering. When enabled, play only newly arrived assistant audio attachments after their media is ready; keep normal manual audio controls visible. If HTMLMediaElement.play() rejects, show a compact "Tap to enable voice playback" style prompt/retry affordance instead of looping or producing surprise sound. Do not change backend TTS generation, provider selection, assistant-media transcript insertion, or the separate duplicate assistant-media bug tracked by #73956. Update focused browser tests for the setting default/persistence, chat prop wiring, successful play attempt, and rejected play handling. Add user-facing docs and a CHANGELOG.md entry if the UI setting ships. If native Control app audio playback is included, keep it as an explicit scoped addition to apps/shared/OpenClawKit/Sources/OpenClawChatUI and Android chat UI rather than mixing in unrelated Talk mode playback changes.
