## Summary

Keep open. Current main still lacks the requested default-off Control UI auto-play preference: generated TTS audio is available to chat, but browser rendering remains a manual audio-control path and there is no persisted playback-consent setting or blocked-autoplay recovery prompt.

## What This Changes

The issue requests an opt-in Control app / Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user consent.

## Best Possible Solution

Add a Control UI-owned playback-consent setting that defaults off, is enabled by explicit user action, persists locally per browser/profile, plays only newly arrived assistant audio attachments when ready, and surfaces a compact enable/retry prompt when play() is blocked. If native Control app parity is in scope, handle the shared Swift Chat UI audio-player gap deliberately rather than coupling it to backend TTS generation.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS can attach audio to assistant replies, the gateway can expose audio blocks, and the browser chat renderer only produces manual audio controls with no persisted auto-play preference or blocked-play handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrowest maintainable solution because backend TTS generation and media embedding already exist, while global or backend-driven autoplay would not satisfy browser autoplay policy or user expectations around unexpected sound.
