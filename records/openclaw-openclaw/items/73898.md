## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: backend TTS/media delivery exists, but the browser UI has no persisted auto-play preference, does not call `play()` for new assistant audio, and still does not normalize gateway rich `audio` blocks into renderable attachments.

## What This Changes

Add an explicit Control app/Control UI setting to automatically play newly generated assistant TTS audio attachments after user opt-in.

## Best Possible Solution

Add a Control UI-owned, default-off playback-consent preference that normalizes rich audio blocks, auto-plays only newly arrived assistant audio after a user gesture, and shows a compact retry prompt when the platform blocks playback.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: enable auto-TTS so assistant audio media is emitted, then observe that the Control UI has no playback-consent setting and the render path either drops rich `audio` blocks or renders manual `<audio controls>` only.

## Solution Assessment

Yes for the browser Control UI direction. The narrow maintainable solution is an explicit default-off playback setting plus rich-audio normalization; native Control app parity should be confirmed as part of the UX scope before implementation.
