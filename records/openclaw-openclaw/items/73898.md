## Summary

Keep open. Current main still has the reported Control UI gap: auto-TTS creates assistant audio, but the browser UI has no persisted opt-in playback setting, no auto-play or blocked-playback recovery path, and the newer assistant-media audio block shape is not rendered as a visible player.

## What This Changes

Add an opt-in Control app / Control UI setting to automatically play newly generated assistant TTS audio and improve the loading or blocked-playback state.

## Best Possible Solution

Implement this as a focused Control UI enhancement: render assistant audio content blocks and MEDIA audio attachments consistently, add a browser-local default-off auto-play preference, trigger playback only for new assistant audio after explicit user opt-in, and show a compact retry/enable prompt when `HTMLMediaElement.play()` is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS returns audio payloads, the assistant-media path can emit `type: "audio"` blocks, and the Control UI either renders only a manual `<audio controls>` attachment or does not normalize the raw audio block at all.

## Solution Assessment

Yes. A browser-local opt-in plus UI renderer support is the narrowest maintainable solution; changing backend TTS generation or channel media semantics would not satisfy browser autoplay policy and could surprise users on other surfaces.
