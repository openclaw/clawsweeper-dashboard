## Summary

Keep open. Current main still lacks the requested default-off Control UI setting and playback-consent flow: backend TTS/audio delivery exists, but the browser chat path either renders manual audio controls or does not normalize gateway audio blocks into visible playable attachments, with no persisted auto-play preference or blocked-play recovery.

## What This Changes

Add an opt-in Control app / Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user consent.

## Best Possible Solution

Add a UI-owned, default-off playback-consent setting that normalizes assistant audio blocks into renderable audio attachments, plays only newly arrived assistant audio after an explicit user gesture, and shows a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-TTS can create assistant audio, gateway code emits audio content blocks, and the Control UI has no persisted auto-play setting or play/blocked-play handling for assistant audio.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrow maintainable fix because backend TTS generation already works and browser autoplay must be initiated and recovered by the UI.
