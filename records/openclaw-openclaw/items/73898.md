## Summary

Keep open. Current main still lacks a Control UI playback-consent setting or assistant-audio autoplay path, and the related WebChat media-delivery work only targets whether TTS audio arrives, not whether it plays automatically.

## What This Changes

The issue requests a default-off Control app setting to auto-play newly generated assistant TTS audio after explicit user consent.

## Best Possible Solution

Add a Control UI-owned, browser-local playback-consent setting after the media delivery contract is stable, then auto-play only newly arrived assistant audio after a user gesture and show a compact retry prompt when play() is blocked.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main renders assistant audio manually and has no persisted autoplay-consent setting or assistant-audio play() path.

## Solution Assessment

Yes directionally, but it needs maintainer UX scope. A default-off local playback-consent setting is the narrow maintainable direction because browser autoplay policy requires explicit user consent, and it should coordinate with the open WebChat media-delivery work.
