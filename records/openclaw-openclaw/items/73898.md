## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: backend TTS/media delivery and manual audio controls exist, but there is no persisted auto-play preference, no assistant-audio `play()` path, and gateway rich `audio` blocks are not a normal Control UI attachment type.

## What This Changes

Add a Control app/UI option that auto-plays newly generated assistant TTS audio after explicit user consent, with a blocked-autoplay retry prompt.

## Best Possible Solution

Add a Control UI-owned, default-off playback-consent setting that normalizes gateway rich audio blocks into renderable audio attachments, auto-plays only newly arrived assistant audio after an explicit user gesture, and shows a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes at source level. Enable auto-TTS so the gateway emits assistant audio media, then current main still lacks a persisted playback-consent setting and the Control UI renderer only exposes manual audio controls; I did not run a live browser session in this read-only review.

## Solution Assessment

Yes directionally, but it needs maintainer UX scope. A default-off local playback-consent setting is the narrow maintainable direction because backend TTS generation already works and browser autoplay policy requires explicit user consent.
