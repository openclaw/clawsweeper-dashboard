## Summary

Keep open. Current main still lacks the requested default-off Control UI auto-play preference: generated TTS audio can reach the chat media path, but the browser renderer remains a manual audio-control flow with no persisted playback consent or blocked-autoplay recovery prompt.

## What This Changes

Add an opt-in Control app or Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user consent.

## Best Possible Solution

Add a browser-local playback-consent setting that defaults off, plays only newly arrived assistant audio when ready, keeps manual controls visible, and surfaces a compact retry prompt when play() is blocked; split native app parity if it needs separate shared Chat UI work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS can attach audio to assistant replies, the gateway can expose audio blocks, and the browser chat renderer only produces manual audio controls with no persisted auto-play preference or blocked-play handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrowest maintainable solution because backend TTS generation already works and backend-driven autoplay would not satisfy browser autoplay policy or user expectations around unexpected sound.
