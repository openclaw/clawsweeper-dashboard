## Summary

Keep open. Current main still lacks the requested default-off Control UI auto-play preference: backend TTS/audio delivery exists, but browser chat rendering remains a manual or missing audio-rendering path with no persisted playback-consent setting or blocked-play recovery.

## What This Changes

Add an opt-in Control app/Control UI setting to automatically play newly generated assistant TTS audio attachments after explicit user consent.

## Best Possible Solution

Add a UI-owned, default-off playback-consent setting that renders assistant audio blocks consistently, plays only newly arrived assistant audio after an explicit user gesture, and shows a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-TTS can attach audio to replies, gateway can expose audio content blocks, and the browser/native chat renderers lack a persisted auto-play setting plus blocked-play handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrow maintainable fix because backend TTS generation already works and backend-driven autoplay cannot satisfy browser autoplay policy or user expectations around unexpected sound.

Add a default-off, browser-local setting such as `Auto-play assistant TTS audio` that persists with `UiSettings`, is surfaced in the appropriate chat/quick settings UI, and is passed into the chat renderer.
