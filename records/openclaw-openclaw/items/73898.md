## Summary

Keep open. Current main still lacks the requested default-off Control UI auto-play preference and blocked-autoplay recovery; available audio is rendered as a manual player only on the MEDIA attachment path, while current assistant-media audio content blocks are not converted into a visible player.

## What This Changes

Add an opt-in Control app/Control UI setting that automatically plays newly generated assistant TTS audio when it arrives.

## Best Possible Solution

Add a browser-local, default-off assistant audio playback consent setting, render current structured assistant audio blocks as playable attachments, auto-play only newly arrived assistant audio after opt-in, and show a compact retry prompt when play() is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS attaches audio, the Control UI lacks a persisted auto-play setting or play() handler, and structured assistant-media audio blocks are not converted into visible audio attachments.

## Solution Assessment

Yes for the browser Control UI path. A local opt-in renderer setting is narrower and more maintainable than backend-driven autoplay because browser autoplay policy requires a user gesture and explicit consent.
