## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: backend TTS/media delivery exists, but browser chat either renders audio as manual controls or drops gateway audio blocks, with no persisted preference or blocked-play recovery.

## What This Changes

Request a default-off Control app/Control UI option that automatically plays newly generated assistant TTS audio after explicit user consent and handles blocked or loading playback gracefully.

## Best Possible Solution

Add a browser-local, default-off playback-consent preference that converts assistant audio blocks into renderable audio, plays only newly arrived assistant audio after a user gesture, and shows a compact retry prompt when playback is blocked; handle native parity separately if it cannot share that UI path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: gateway TTS can create audio content blocks, while Control UI lacks persisted autoplay consent and does not render generic rich `audio` blocks as playable attachments.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrow maintainable solution because backend TTS generation already works and browser autoplay policy requires explicit user consent.
