## Summary

Keep open. Current main still has the requested Control UI gap: generated assistant TTS/audio reaches the browser as an audio attachment, but the chat renderer exposes only a manual audio control and there is no default-off playback preference or rejected-autoplay recovery path.

## What This Changes

Add a default-off Control app/Control UI option that automatically plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

The end state should be a Control UI-owned, browser-local playback-consent setting that defaults off, is enabled by explicit user action, plays only new assistant audio attachments when ready, and surfaces a compact retry/enable prompt when the browser blocks playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS attaches reply audio, the gateway can expose it to Control UI, and the browser chat renderer only emits a manual audio control with no playback preference or autoplay handling.

## Solution Assessment

Yes. A browser-local opt-in is the narrowest maintainable fix because backend TTS generation already works and global/channel-level autoplay would not satisfy browser autoplay policy or user expectations around unexpected sound.
