## Summary

Keep open. Current main still lacks the requested default-off Control UI auto-play preference: assistant audio can reach the chat media path, but the browser renderer remains a manual `<audio controls>` flow with no persisted playback-consent setting or blocked-autoplay recovery prompt.

## What This Changes

The issue requests a default-off Control app/Control UI setting that auto-plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Add a browser Control UI-owned playback-consent setting that defaults off, persists locally per browser/profile, auto-plays only newly arrived assistant audio attachments when ready, and shows a compact enable/retry prompt when play() is blocked; handle native app parity as a separate scoped follow-up unless maintainers explicitly include it.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: Auto-TTS can attach audio to replies, gateway media helpers can surface audio blocks, and the Control UI renderer only creates manual audio controls with no persisted auto-play preference or blocked-play handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrowest maintainable solution because backend TTS generation already works and backend-driven autoplay would not satisfy explicit user consent or browser media behavior.
