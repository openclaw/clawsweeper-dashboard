## Summary

Keep open. Current main still has the requested Control UI gap: auto-TTS audio is generated and attached, but the browser chat renderer only emits a manual audio control and there is no persisted, default-off playback preference or blocked-autoplay recovery path.

## What This Changes

Add a Control app / Control UI option to automatically play new assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Implement this as a focused Control UI enhancement: add a browser-local, default-off assistant audio auto-play preference, expose it in the existing settings surface, pass it into chat rendering, play only newly arrived assistant audio after explicit opt-in, and show a compact retry/enable prompt when `HTMLMediaElement.play()` is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS attaches audio to assistant replies, and current Control UI rendering emits only a manual audio element with no playback preference or autoplay handling.

## Solution Assessment

Yes. The requested explicit browser-local opt-in is the narrowest maintainable solution; changing backend TTS generation or generic channel media semantics would not satisfy browser autoplay policy and could cause surprising sound on other surfaces.
