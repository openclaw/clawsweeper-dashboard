## Summary

Keep open. Current main still has the reported Control UI UX gap: auto-TTS can generate assistant audio, but the UI renders it only as a manual audio attachment and has no browser-local auto-play opt-in, media play path, or blocked-autoplay recovery prompt.

## What This Changes

Add a default-off Control app/Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user opt-in and handles blocked autoplay/loading states.

## Best Possible Solution

Implement this as a focused Control UI enhancement: add a browser-local, default-off assistant audio auto-play preference, expose it through existing settings UI, pass it into chat rendering, auto-play only newly arriving assistant audio attachments after explicit opt-in, and show a compact retry/enable prompt if HTMLMediaElement.play() is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS returns assistant audio media, and current Control UI chat rendering emits only a manual audio control with no persisted playback opt-in or autoplay handling.

## Solution Assessment

Yes. The requested explicit browser-local opt-in is the narrowest maintainable direction; changing backend TTS generation or channel media semantics would not satisfy browser autoplay policy and could create surprising sound on other surfaces.
