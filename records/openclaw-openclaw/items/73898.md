## Summary

Keep open. Current main still lacks the requested opt-in Control UI playback-consent setting and automatic assistant TTS playback path; the backend can attach audio and the browser can render some manual audio attachments, but no persisted preference, play() flow, or blocked-play prompt exists.

## What This Changes

Add a default-off Control app/Control UI setting to auto-play newly generated assistant TTS audio after explicit user opt-in.

## Best Possible Solution

Add UI-owned, default-off playback consent that renders assistant audio blocks consistently, plays only newly arrived assistant audio after a user gesture, and shows a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-TTS can produce assistant audio content, but the browser settings/rendering path lacks any persisted auto-play preference, play() call, or blocked-play handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrow maintainable solution because browser autoplay policy requires a user gesture and backend-driven autoplay cannot reliably satisfy that contract.
