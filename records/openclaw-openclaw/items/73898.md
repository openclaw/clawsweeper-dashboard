## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: backend TTS/media delivery exists, but the browser UI does not persist any assistant-audio auto-play preference, still renders available audio as a manual `<audio controls>` path, and does not handle gateway rich `audio` content blocks as renderable attachments.

## What This Changes

Requests a default-off Control app/UI setting that auto-plays newly generated assistant TTS audio after explicit user consent and handles blocked/loading playback gracefully.

## Best Possible Solution

Add a browser Control UI-owned, default-off playback-consent setting that normalizes gateway rich audio blocks into renderable audio attachments, auto-plays only newly arrived assistant audio after an explicit user gesture, and shows a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: enable auto-TTS so the gateway emits assistant audio media, then observe that Control UI has no playback-consent setting and the render path either drops rich `audio` blocks or renders manual `<audio controls>` only.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrow maintainable solution because backend TTS generation already works and browser autoplay policy requires explicit user consent; native app parity can be handled separately if it cannot share the browser chat path.
