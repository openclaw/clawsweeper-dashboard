## Summary

Keep open. Current main still has the reported UX gap: auto-TTS generates assistant audio, but the Control UI renders it as a manual audio attachment and has no browser-local auto-play opt-in, media `play()` path, or blocked-autoplay recovery prompt.

## What This Changes

The issue requests a Control app / Control UI setting that auto-plays generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Implement this as a focused Control UI enhancement: add a browser-local, default-off assistant audio auto-play preference, expose it through existing settings UI, pass it into chat rendering, auto-play only new assistant audio attachments after explicit opt-in, and show a compact retry/enable prompt when `HTMLMediaElement.play()` is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS attaches generated audio to assistant replies, and current Control UI chat rendering emits only a manual audio control with no persisted playback opt-in or autoplay handling.

## Solution Assessment

Yes. The requested explicit browser-local opt-in is the narrowest maintainable solution; changing backend TTS generation or channel media semantics would not satisfy browser autoplay policy and could create surprising sound on other surfaces.
