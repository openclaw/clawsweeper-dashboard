## Summary

Keep open. Current main still renders assistant TTS audio as a manual audio control and has no persisted Control UI autoplay-consent setting or assistant-audio `play()` path; the related WebChat media threads address delivery/deduplication, not autoplay after opt-in.

## What This Changes

Add a default-off Control app/Control UI option that automatically plays newly generated assistant TTS audio attachments after explicit user consent.

## Best Possible Solution

Add a Control UI-owned browser-local playback-consent setting after the media delivery contract is stable, then play only newly arrived assistant audio and show a compact retry prompt when `play()` is blocked.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main has manual assistant audio rendering and no persisted playback-consent or autoplay path.

## Solution Assessment

Yes directionally, but it needs maintainer UX scope. A default-off browser-local consent setting is the narrow maintainable direction because autoplay with sound requires explicit user action and a graceful blocked-playback fallback.
