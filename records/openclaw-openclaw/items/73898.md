## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: TTS audio can be generated and embedded, but the browser chat path still has manual audio controls, no persisted auto-play preference, and no blocked-play recovery prompt.

## What This Changes

Add a default-off Control app/Control UI setting that auto-plays newly generated assistant TTS audio after explicit user consent and handles blocked playback/loading gracefully.

## Best Possible Solution

Add a UI-owned playback-consent preference that normalizes assistant audio blocks into playable attachments, auto-plays only newly arrived assistant audio after an explicit user gesture, and surfaces a compact retry prompt when the browser blocks playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: Auto-TTS can create assistant audio, while Control UI has no persisted playback-consent setting and the current audio render path is a manual `<audio controls preload="metadata">` element.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in setting plus render-time playback handling is the narrow maintainable solution because backend TTS generation already exists and browser autoplay policy requires explicit user consent.
