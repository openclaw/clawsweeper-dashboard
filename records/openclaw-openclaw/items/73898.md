## Summary

Keep open. Current main has backend TTS/audio delivery and a manual browser audio player, but it still lacks a persisted Control UI playback-consent setting, autoplay/retry wiring, and reliable rendering for gateway `type: "audio"` content blocks; related #73956 only fixed duplicate text from stale assistant-media supplements.

## What This Changes

Add a default-off Control app/UI setting that automatically plays newly generated assistant TTS audio after explicit user opt-in.

## Best Possible Solution

Add a UI-owned, default-off playback-consent setting that normalizes assistant audio blocks into visible players, auto-plays only newly arrived assistant audio after a user gesture, and shows a compact retry prompt when the platform blocks playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: enable `messages.tts.auto`, receive an assistant audio reply, and the gateway can produce audio blocks while the browser/native chat renderers lack persisted autoplay consent and blocked-play recovery.

## Solution Assessment

Yes. The requested opt-in UI setting is the narrow maintainable direction because backend TTS generation already works, and browser autoplay rules require a user gesture and client-side playback handling.
