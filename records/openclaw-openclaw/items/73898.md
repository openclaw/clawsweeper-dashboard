## Summary

Keep open. Current main still lacks the requested default-off Control UI playback-consent setting: assistant audio can be delivered, but browser rendering is still a manual or incomplete audio path with no persisted auto-play preference or blocked-play recovery.

## What This Changes

The issue requests a Control UI or Control app option to automatically play newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Add a UI-owned, default-off playback-consent setting that renders assistant audio blocks consistently, plays only newly arrived assistant audio after an explicit user gesture, and shows a compact retry prompt when playback is blocked; decide native app parity separately if it cannot share the same chat UI path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: backend TTS can produce assistant audio media, while browser chat lacks a persisted auto-play setting and drops generic rich `audio` blocks instead of turning them into playable attachments.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrow maintainable fix because backend TTS generation already works and browser autoplay policy requires explicit user consent; native Control app parity may need a separate scoped follow-up.
