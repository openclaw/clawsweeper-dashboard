## Summary

Keep open. Current main still lacks a default-off Control UI playback-consent setting: assistant TTS audio is generated and delivered into chat media, but the browser renderer remains a manual `<audio controls>` flow with no persisted preference, `play()` handling, or blocked-autoplay recovery prompt.

## What This Changes

The issue requests a Control app/Control UI option that auto-plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Add a Control UI-owned local playback-consent setting that defaults off, plays only newly arrived assistant audio attachments when ready, and surfaces an enable/retry prompt when `play()` is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS can create assistant audio media, but the chat UI only renders manual audio controls and has no persisted auto-play preference or playback failure handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in setting is the narrowest maintainable path because backend TTS generation already works and backend-driven autoplay cannot satisfy browser user-gesture policy.
