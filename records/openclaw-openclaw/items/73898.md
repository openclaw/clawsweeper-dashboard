## Summary

Keep open. Current main still lacks the requested opt-in Control UI playback setting: TTS media generation exists, but assistant audio is either a manual audio attachment path or an unhandled rich audio block, with no persisted consent state or blocked-playback recovery.

## What This Changes

Requests a default-off Control app/UI setting that auto-plays newly generated assistant TTS audio after explicit user consent and handles blocked or loading playback gracefully.

## Best Possible Solution

Add a browser Control UI-owned, default-off playback consent setting that normalizes gateway rich audio blocks into renderable audio attachments, auto-plays only newly arrived assistant audio after a user gesture, and shows a compact retry prompt when playback is blocked.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable auto-TTS so the gateway emits assistant audio media, then observe that Control UI has no playback-consent setting and either drops rich `audio` blocks or renders manual `<audio controls>` only.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in setting is the narrow maintainable direction because backend TTS generation already works and browser autoplay policy requires explicit user consent.
