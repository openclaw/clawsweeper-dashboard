## Summary

Keep open. Current main still lacks the requested default-off Control UI auto-play preference: TTS audio reaches the chat media path, but browser rendering remains a manual audio-control flow with no persisted playback-consent setting or blocked-autoplay recovery prompt.

## What This Changes

The issue requests an opt-in Control app / Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user consent.

## Best Possible Solution

Add a Control UI-owned playback-consent setting that defaults off, persists locally per browser/profile, plays only newly arrived assistant audio attachments when ready, and surfaces a compact enable/retry prompt when play() is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: auto-TTS can attach audio to assistant replies, the gateway can expose audio blocks, and the browser chat renderer only produces manual audio controls with no persisted auto-play preference or blocked-play handling.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in playback setting is the narrowest maintainable solution because backend TTS generation already works and backend-driven autoplay would not satisfy browser autoplay policy or user expectations around unexpected sound.
