## Summary

Keep open. Current main still has manual Control UI audio playback and no persisted playback-consent setting or assistant-audio autoplay path; the remaining work is a user-facing feature/UX decision rather than an already-fixed bug.

## What This Changes

Add a default-off Control app setting that auto-plays newly generated assistant TTS audio after explicit user opt-in.

## Best Possible Solution

Add a Control UI-owned, default-off playback-consent setting after the audio delivery contract is stable, then auto-play only newly arrived assistant audio after a user gesture and show a compact retry prompt when play() is blocked.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection does confirm the current state: audio is rendered with manual controls and there is no persisted autoplay setting or assistant-audio play() path.

## Solution Assessment

Yes directionally, but it needs maintainer UX scope. A default-off local playback-consent setting is the narrow maintainable direction because browser autoplay policy requires explicit user consent.
