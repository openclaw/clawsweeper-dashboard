## Summary

Keep open. Current main still lacks the requested default-off Control UI or Control app auto-play preference: TTS audio is generated and exposed, but playback remains a manual UI attachment flow without persisted consent or blocked-autoplay recovery.

## What This Changes

Add an opt-in Control app/Control UI setting that automatically plays newly generated assistant TTS audio attachments and handles blocked autoplay/loading states.

## Best Possible Solution

Add a local, default-off playback-consent setting for newly arrived assistant audio in the browser Control UI, then decide whether the same behavior should be implemented in the shared native Chat UI surface.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-TTS can attach audio, but the browser Control UI renders only manual audio controls and has no persisted auto-play setting.

## Solution Assessment

Yes for the browser Control UI portion. A local opt-in setting with play() rejection handling is narrower and safer than changing backend TTS generation or attempting platform autoplay workarounds.

## Repair Work Prompt

Implement the browser Control UI portion of https://github.com/openclaw/openclaw/issues/73898. Add a default-off, browser-local setting for assistant TTS/audio attachment auto-play, persist it in UiSettings, expose a compact toggle in the appropriate chat/settings surface, and wire it into assistant audio rendering so only newly arrived assistant audio attachments play after explicit user opt-in. Handle browser autoplay failures by catching rejected play() promises and surfacing a small retry/enable prompt. Do not change backend TTS generation, provider defaults, or channel delivery policy. Do not attempt a broad native app rewrite unless the shared OpenClawChatUI audio-player change is small and separately tested. Update docs/changelog if the UI behavior changes.
