## Summary

Keep open. Current main still has the Control UI gap: TTS audio reaches the browser as assistant audio content, but chat rendering only exposes a manual audio control and there is no persisted, default-off playback preference or blocked-autoplay recovery path.

## What This Changes

Add a Control app / Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Implement this as a focused Control UI enhancement: add a browser-local, default-off assistant audio auto-play preference, expose it in the existing settings surface, pass it into chat rendering, play only newly arrived assistant audio after opt-in, and show a compact retry/enable prompt when playback is blocked.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS attaches audio to assistant replies, the gateway emits audio blocks for Control UI, and the browser chat renderer only creates a manual audio control with no playback preference or autoplay handling.

## Solution Assessment

Yes. The requested explicit browser-local opt-in is the narrowest maintainable solution; changing backend TTS generation or generic channel media semantics would not satisfy browser autoplay policy and could surprise users on other surfaces.
