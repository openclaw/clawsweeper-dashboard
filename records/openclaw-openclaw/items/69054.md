## Summary

Keep this open. Current main still has no `openclaw cron lint` command and still accepts announce-mode cron jobs without `delivery.accountId`; #69055 is complementary delivery logging, not a replacement for config-time lint and mutation validation.

## What This Changes

Add cron linting and validation so announce-mode cron jobs targeting multi-account channel configs must set `delivery.accountId`.

## Best Possible Solution

Implement a shared cron lint helper backed by channel plugin account lists, expose `openclaw cron lint`, warn at gateway startup, reject invalid cron add/update mutations, document the migration path, and keep #69055 scoped to structured delivery attribution logs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to add or update an isolated announce cron with `delivery.channel: "telegram"`, no `delivery.accountId`, and a Telegram config with multiple accounts; current gateway validation checks only channel compatibility/configuration and then accepts the job.

## Solution Assessment

Yes, with staged enforcement. The narrow maintainable fix is a cron-owned lint/validation helper using existing channel plugin account contracts; startup fail-closed behavior should follow after the warning-only migration window.

## Repair Work Prompt

Implement the cron account-id lint/enforcement slice for openclaw/openclaw#69054. Add a shared cron lint helper that reports `CRON_ANNOUNCE_ACCOUNT_ID_REQUIRED` when an announce-mode cron delivery resolves to a channel whose plugin/config account list has more than one account and `delivery.accountId` is missing; single-account channels should be a no-op. Expose `openclaw cron lint [--warn-only]` through the cron CLI, return exit code 1 for findings and 2 for parse/execution failures, and format findings with job id/name/channel/agent/message. Add warning-only gateway startup logging after cron jobs load and before scheduler start, but do not flip startup to fail-closed in this PR. Enforce the rule on cron create/update mutations in the gateway path so `cron add`/`cron edit` cannot create or edit invalid announce jobs; use existing channel/plugin account contracts rather than hardcoding Telegram/Discord ids. Update docs and CHANGELOG for the migration playbook. Keep #69055 structured delivery logging out of scope, and do not infer `accountId` from `agentId` as a hidden migration.
