## Summary

Keep this issue open. Current main still carries optional cron `delivery.accountId` through the delivery path, but it has no `openclaw cron lint` command, no multi-account account-id rule in cron add/update validation, and no startup lint warning for existing jobs. The linked #69055 remains complementary forensic logging work, not a duplicate of this config-time validation request.

## What This Changes

Requests a cron lint command plus startup and cron add/edit validation requiring explicit `delivery.accountId` for announce-mode cron jobs that target multi-account channel configurations.

## Best Possible Solution

Implement a cron-owned validation surface: reusable lint findings for announce jobs missing `delivery.accountId` only when the resolved channel has multiple configured accounts, a read-only `openclaw cron lint` command with the requested exit codes and `--warn-only`, startup warnings before scheduling, add/update rejection for new invalid jobs, migration docs, and focused tests. Keep explicit `accountId: "default"` valid for intentional default-bot delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to call `cron.add` or `cron.update` with `delivery.mode: "announce"`, a concrete configured channel that has multiple accounts, and no `delivery.accountId`; current validation reaches `assertValidCronAnnounceDelivery`, checks only channel ambiguity, and then accepts the job.

## Solution Assessment

Yes. The shared cron lint/validation helper is the narrow maintainable direction if it uses existing channel plugin account contracts rather than provider-specific branches. The safer rollout is lint, warning-only startup, and add/edit validation first, with fail-closed startup deferred to explicit maintainer approval after migration.
