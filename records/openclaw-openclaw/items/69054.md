## Summary

Keep open. Current main still lacks the requested `openclaw cron lint` command and account-aware validation for multi-account announce delivery; the linked logging issue remains complementary rather than a replacement.

## What This Changes

The issue requests a cron lint command plus validation that announce-mode cron jobs specify `delivery.accountId` when targeting a multi-account channel.

## Best Possible Solution

Add a shared cron lint helper backed by the channel plugin account-list contract, expose `openclaw cron lint`, warn at gateway startup, reject invalid new/edited announce jobs, and keep fail-closed rollout plus the linked telemetry issue separate.

## Reproduction Assessment

Yes. Source inspection shows a multi-account announce job with an explicit channel and no `delivery.accountId` reaches current validation because it checks channel ambiguity only, while the requested lint command and rule symbols are absent.

## Solution Assessment

Yes, for the staged version. A cron-owned lint/validation helper is the narrow maintainable path; strict startup failure and any opt-out should be a later maintainer-controlled rollout choice.

Add a shared cron lint helper, likely `src/cron/lint.ts`, that reports `CRON_ANNOUNCE_ACCOUNT_ID_REQUIRED` when an announce-mode cron job targets a channel whose plugin config lists more than one account and the job lacks `delivery.accountId`; single-account channels must remain a no-op.
