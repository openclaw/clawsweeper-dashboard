## Summary

Keep open. Current main still lacks the requested `openclaw cron lint` surface and still validates announce cron delivery by channel/target only, not by missing `delivery.accountId` on multi-account channels; the related open PR at https://github.com/openclaw/openclaw/pull/69562 is partial and does not supersede the lint/startup/missing-account rollout requested here.

## What This Changes

Request adds a read-only cron lint command plus startup warning and cron add/edit validation requiring `delivery.accountId` for announce-mode jobs targeting multi-account channels.

## Best Possible Solution

Add a cron-owned lint helper backed by configured-account metadata, expose `openclaw cron lint --warn-only`, warn at gateway startup, reject new or edited multi-account announce jobs missing `delivery.accountId`, and leave strict startup failure plus waiver/opt-out policy to maintainer-directed follow-up.

## Reproduction Assessment

Yes. Source inspection on current main shows no cron lint command/rule symbols, and gateway cron validation only checks announce channel ambiguity/target prefixes before persistence, so an explicit-channel announce job with missing `delivery.accountId` is not rejected.

## Solution Assessment

Yes for the staged lint plus mutation guard, but not for immediate fail-closed startup or waiver/env policy without maintainer direction. The implementation should be a shared cron validation helper using read-only configured-account metadata, not runtime plugin loading or agentId inference.
