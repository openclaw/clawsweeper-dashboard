## Summary

Keep this open. Current main still lacks the requested `openclaw cron lint` command and still allows announce-mode cron delivery without `delivery.accountId`; #69055 is complementary logging work, not a replacement for this config-time validation.

## What This Changes

Request a cron-owned lint and validation path that requires explicit `delivery.accountId` for announce-mode cron jobs targeting multi-account channel configurations.

## Best Possible Solution

Implement a shared cron lint/validation helper that reports missing announce `delivery.accountId` only for resolved multi-account channels, expose it through `openclaw cron lint`, warn at startup, reject invalid add/update mutations, and document the migration path while keeping #69055 scoped to delivery attribution logs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create or update an isolated agent-turn cron job with `delivery.mode: "announce"`, an explicit multi-account channel such as `telegram`, and no `delivery.accountId`; current validation reaches `assertValidCronAnnounceDelivery` and accepts it after checking only channel compatibility/configuration.

## Solution Assessment

Yes, with scope control. The maintainable fix is a cron-owned helper using existing channel/plugin account contracts; startup fail-closed behavior should remain a later maintainer-approved rollout after warning-only lint and mutation enforcement land.

Add a reusable lint helper for `CRON_ANNOUNCE_ACCOUNT_ID_REQUIRED` that flags cron jobs whose effective delivery mode is `announce`, whose resolved channel is configured with more than one account, and whose `delivery.accountId` is missing.
