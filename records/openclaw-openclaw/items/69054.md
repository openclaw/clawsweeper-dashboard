## Summary

Keep open: current main still lacks the requested `openclaw cron lint` command and does not require `delivery.accountId` when an announce-mode cron targets a multi-account channel. The related open PR overlaps only part of the validation surface and still does not cover the lint/startup rollout requested here.

## What This Changes

The issue requests a cron lint command plus validation requiring explicit `delivery.accountId` for announce-mode cron jobs targeting multi-account channel configurations.

## Best Possible Solution

Implement a shared cron-owned lint/account validation helper using read-only channel account metadata, expose warn-only CLI and startup reporting first, and promote hard enforcement only after an explicit rollout decision.

## Reproduction Assessment

Yes, for the current gap. Source inspection on current main shows no cron lint command/rule symbols and the gateway add/update path still validates announce channel/target compatibility without requiring missing `delivery.accountId` on multi-account channel configs.

## Solution Assessment

Partly yes. A shared lint helper plus mutation guard is the maintainable boundary, but startup fail-closed behavior, deprecation timing, waivers, and opt-out policy need maintainer direction before the full proposal should land.
