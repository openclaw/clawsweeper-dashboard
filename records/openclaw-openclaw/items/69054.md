## Summary

Keep open: current main and the latest release still lack the requested cron lint surface and only validate announce channel/target selection, while docs/source confirm omitted accountId can fall back to a default account in multi-account setups.

## What This Changes

Add an `openclaw cron lint` command plus startup and add/edit validation requiring explicit `delivery.accountId` for announce-mode cron jobs targeting multi-account channels.

## Best Possible Solution

Implement a shared cron-owned account lint helper with warn-only CLI/startup reporting first, add focused add/edit validation, and make any fail-closed rollout an explicit maintainer decision.

## Reproduction Assessment

Yes, for the missing behavior. Source inspection shows current `cron.add`/`cron.update` validate announce channel selection but allow announce delivery without `delivery.accountId`, and the CLI only passes `--account` when supplied.

## Solution Assessment

Partly yes. A shared lint helper plus mutation guard is the maintainable boundary, but startup fail-closed behavior, rollout timing, waivers, and opt-out policy need maintainer direction before the full proposal should land.
