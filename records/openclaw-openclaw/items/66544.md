## Summary

Keep this PR open: it targets a real Gateway row display-name path, but the current patch is overbroad because it suppresses every `origin.label` equal to `heartbeat`, and contributor real behavior proof is still missing.

## What This Changes

The PR changes `buildGatewaySessionRow` so `origin.label` is ignored when it equals the heartbeat fallback sender string.

## Best Possible Solution

Land a scoped Gateway row-projection guard that ignores only synthetic or stale heartbeat origin metadata, preserves legitimate labels named `heartbeat`, and includes focused coverage plus real behavior proof.

## Reproduction Assessment

Yes, at source level: current main still falls back from `displayName` to `origin.label`, so a stored row with no explicit display fields and `origin.label: "heartbeat"` will display that value. I did not run live WebChat or tests in this read-only review.

## Solution Assessment

No. The shared row projection is the right surface for stale display metadata, but matching only the raw string `heartbeat` also hides legitimate direct-chat labels; the guard should be tied to synthetic heartbeat provenance or stale metadata shape.
