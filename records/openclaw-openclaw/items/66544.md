## Summary

Keep this PR open, but it needs changes before merge. The branch targets a real stale display-name fallback path, but the current predicate suppresses any real session/contact label exactly equal to `heartbeat`; current main also already prevents new heartbeat system-event origin writes, so the remaining fix should be scoped to stale/synthetic origin metadata rather than a global label string.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `buildGatewaySessionRow` skips `origin.label` when it equals the heartbeat fallback sender string.

## Best Possible Solution

Use a shared row-projection helper in `buildGatewaySessionRow` that suppresses only known synthetic/stale heartbeat origin metadata, preserves explicit `entry.displayName` and `entry.label`, keeps legitimate direct labels named `heartbeat`, and covers both list and live session snapshot callers with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. A static reproduction is a direct/main session entry with no explicit `displayName` or `label` and `origin.label: "heartbeat"`, which current `buildGatewaySessionRow` will expose as `displayName`; the PR also creates a regression case for a legitimate direct label named `heartbeat`.

## Solution Assessment

No. The PR is in the right shared function, but a global `origin.label !== "heartbeat"` check is not the best fix because it loses valid human labels; the safer solution scopes suppression to synthetic heartbeat provenance or stale main-session metadata and proves both cases in tests.
