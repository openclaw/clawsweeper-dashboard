## Summary

Keep open: the PR addresses a real stale session-row display path, but the current patch suppresses any legitimate origin label named `heartbeat` and still lacks contributor real behavior proof. Current main already shipped the broader new-write guard for heartbeat/system-event origin pollution, so this branch is now a narrower hardening change rather than the canonical fix for the linked user report.

## What This Changes

The PR changes Gateway session-row projection so `origin.label` is ignored when it equals the heartbeat fallback sender string.

## Best Possible Solution

Land only a scoped Gateway row-projection guard that ignores synthetic or stale heartbeat-origin metadata while preserving legitimate labels named `heartbeat`, with focused coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: current main still falls back from `displayName` to stored `origin.label`, and the Control UI selector can render that projected row value. I did not run a live WebChat session, and current main already prevents new heartbeat system events from writing this metadata.

## Solution Assessment

No, not as currently written. The row-projection surface is reasonable for stale metadata, but suppressing the raw string `heartbeat` should be replaced with a predicate tied to synthetic heartbeat provenance or stale origin shape.
