## Label Justifications

## Summary

Keep open: current main and latest release only rotate generated reset transcript names when the basename still matches the stored previous session id, but this issue is about historically polluted rows where the stored `sessionId` has already drifted from the UUID-shaped generated filename. The remaining work is already covered by the open, conflicting related PR at https://github.com/openclaw/openclaw/pull/80251, so this issue should stay open until maintainers rebase/review that PR or land a narrow equivalent.

## What This Changes

The issue reports that TUI `/reset` can keep writing a fresh logical session to a stale generated transcript filename left behind by older reset behavior.

## Best Possible Solution

Reset should classify UUID-shaped generated transcript basenames as generated drift even when they no longer match the stored `sessionId`, rotate them to the new session id, and continue preserving true custom `sessionFile` paths.

## Reproduction Assessment

Yes, from source inspection: seed a session entry whose `sessionId` differs from a UUID-shaped generated `sessionFile` basename, then call `sessions.reset`; current main falls back to that stale `sessionFile`. I did not run the mutating test path in this read-only checkout.

## Solution Assessment

No, current main is only a partial fix because it covers generated names matching the current entry session id, not the historically polluted UUID basename in the report. The maintainable fix belongs in the gateway/session path boundary, not in a TUI-only workaround.
