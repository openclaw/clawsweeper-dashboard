## Summary

Current main still appears vulnerable by source inspection: TUI `/reset` reaches `sessions.reset`, and the reset service can carry a stored generated `sessionFile` into the new session instead of synthesizing `<nextSessionId>.jsonl`. This is narrow, source-reproducible, and suitable for a focused fix PR.

## What This Changes

The issue asks for TUI `/reset` to stop reusing an old generated transcript filename after upgrading from older reset behavior.

## Best Possible Solution

Teach reset path selection to preserve intentional custom transcript files but synthesize a fresh `<nextSessionId>.jsonl` when the previous `sessionFile` is an OpenClaw-generated filename for a different session, then cover that migration case with a regression test.

## Reproduction Assessment

Yes, via source inspection: a session-store row with `sessionId` set to a new value and `sessionFile` still pointing at an older generated UUID `.jsonl` will be reset through `sessions.reset`, which currently keeps the stale filename. I did not execute the TUI flow in this read-only review.

## Solution Assessment

Yes, the narrowest fix is in reset transcript path selection, not in TUI command handling. The fix should reuse the existing generated-vs-custom filename classification pattern and avoid breaking explicit custom transcript paths.

## Repair Work Prompt

Fix the TUI/Gateway `/reset` stale transcript filename bug from https://github.com/openclaw/openclaw/issues/77770. Current main mints a new `sessionId` in `src/gateway/session-reset-service.ts` but passes `currentEntry.sessionFile` into `resolveSessionFilePath`, so an old OpenClaw-generated UUID `.jsonl` path can be archived and then recreated as the active transcript for the new session. Preserve intentional custom transcript paths, including non-generated custom names and existing custom/fork semantics, but when the previous `sessionFile` is an OpenClaw-generated transcript filename for a different session id, choose the default path for the new `nextSessionId`. Prefer reusing or extracting the generated-transcript classification logic near `src/gateway/session-transcript-files.fs.ts` rather than duplicating fragile filename parsing. Add regression coverage for `sessions.reset` with an old generated UUID `sessionFile`: assert the returned/stored `entry.sessionId` changes, `entry.sessionFile` basename is `<new-session-id>.jsonl`, the old transcript is archived as `.jsonl.reset.<timestamp>`, and the new transcript header id matches the new session. Keep explicit custom `sessionFile` preservation tests green. Add a single-line user-facing `CHANGELOG.md` fix entry for the active version. Do not change TUI `/new` behavior or broad session-store migration policy.
