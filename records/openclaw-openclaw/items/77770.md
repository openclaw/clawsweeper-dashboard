## Summary

Keep open: current main fixed the clean generated/topic reset path, but the historically polluted case still appears source-reproducible because a UUID-shaped generated filename that no longer matches the stored `sessionId` is preserved as an explicit `sessionFile`. The earlier closing PR is closed unmerged, and the newer related PR that covers drifted UUID basenames is still open and conflicting.

## What This Changes

The issue reports that TUI `/reset` can keep writing fresh-session transcripts to an older generated `.jsonl` filename after older reset behavior left session metadata polluted.

## Best Possible Solution

Rebase and review https://github.com/openclaw/openclaw/pull/80251 or land a narrow equivalent that treats UUID-shaped generated transcript basenames as generated even when they have drifted from the stored `sessionId`, while preserving explicit custom transcript paths.

## Reproduction Assessment

Yes, at source level. A historically polluted entry with `sessionId=B` and `sessionFile=A.jsonl` reaches reset code that cannot rewrite `A.jsonl` because it only matches `B`, then `resolveSessionFilePath` preserves `A.jsonl` as the active file.

## Solution Assessment

Yes for the repair direction, but not through a TUI-only workaround. The maintainable fix belongs in the gateway/session path boundary: classify OpenClaw-generated UUID basenames as generated drift, rotate them to the new session id, and continue preserving true custom paths.
