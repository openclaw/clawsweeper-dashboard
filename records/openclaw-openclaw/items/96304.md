## Summary

Keep this PR open for normal maintainer review: the patch appears correct and focused, with no blocking code-review findings, but it intentionally changes future memory `conceptTags` and promotion scoring when snippets mention standalone `kv` or `s3`.

## What This Changes

The PR changes memory-core concept tag extraction so short protected glossary entries such as `kv` and `s3` bypass the generic minimum-length filter only when matched as standalone glossary terms.

## Best Possible Solution

Land the focused glossary fix after maintainers accept the bounded future-memory scoring change and required checks are green.

## Reproduction Assessment

Yes. Source inspection on current main shows `kv` and `s3` in the protected glossary while the shared normalizer rejects Latin tokens shorter than 3; the contributor also provided direct before/after `deriveConceptTags` output.

## Solution Assessment

Yes. The patch fixes the owning extraction pipeline, bypasses only the min-length gate for glossary matches, and limits whole-token matching to the short entries that need the bypass; lowering the global minimum or changing all Latin glossary entries to whole-word matching would be broader and riskier.
