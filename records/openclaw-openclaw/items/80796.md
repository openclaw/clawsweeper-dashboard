## Summary

Keep this PR open: the doctor-side historical scrub is the right direction and has convincing real behavior proof, but the current head still has a concrete race that can drop audit records appended after the guard and before the final rename.

## What This Changes

The PR adds `scrubConfigAuditLog`, a doctor health contribution, and tests to detect and rewrite historical `config-audit.jsonl` `argv`/`execArgv` values through the existing config-audit redactor.

## Best Possible Solution

Keep the doctor-side scrub, but make the replacement step detect or serialize appends through the whole temp-write-to-rename interval while preserving `0600` permissions and avoiding plaintext backups.

## Reproduction Assessment

Yes. Source inspection shows current main redacts only new snapshots and has no existing-log scrub path, and the supplied logs show historical pre-redactor entries being detected and scrubbed by the PR.

## Solution Assessment

No, not as-is. Doctor-side scrubbing through the existing redactor is the narrow maintainable solution, but the PR must close the post-stat/pre-rename append-loss window before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Recheck appends after writing the scrub file:** `src/config/io.audit.ts:578`
  - body: The guard compares `config-audit.jsonl` size before writing the temp file, then renames that transformed snapshot over the live log. Any audit record appended after this stat but before `rename` is not detected and will be dropped by the replacement; recheck immediately before rename after the temp write or coordinate with append writers, and add coverage for that window.
  - confidence: 0.9

## Security Review

Status: needs_attention
