## Summary

Keep open: this is a focused bug/security-hardening PR with real proof and a valid fix direction, but the latest patch still has a concrete audit-log record-loss race during the atomic replacement step.

## What This Changes

The PR adds a doctor health contribution and `scrubConfigAuditLog` helper to rewrite historical `config-audit.jsonl` `argv` and `execArgv` values through the existing config-audit redactor.

## Best Possible Solution

Keep the doctor-side scrub, but make the replacement step detect appends that happen after temp-file creation or otherwise coordinate with audit writers before landing it for the linked bug.

## Reproduction Assessment

Yes, by source inspection and supplied logs. Current main redacts newly captured argv fields but has no historical scrub path, while the linked issue and PR logs show live historical entries being found and scrubbed by the patch.

## Solution Assessment

No, not as-is. Doctor-side scrubbing through the existing redactor is the narrow maintainable direction, but the final rename must be guarded against appends that occur after the current size check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recheck the audit log after writing the temp file:** `src/config/io.audit.ts:574-578`
  - body: The append guard stats `config-audit.jsonl` before `writeFile(tmp)`, then renames the transformed snapshot over the live log. An audit entry appended while the temp file is being written, or just after it returns, is not detected and will be lost by the rename, so the scrub can still erase append-only audit history.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch improves credential scrubbing, but the remaining replacement race can drop forensic audit records.

Concerns:

- **[medium] Prevent audit-log record loss during scrub:** `src/config/io.audit.ts:578`
  - body: Because the scrub only checks the live log size before writing the temp file, a concurrent append after that check can be overwritten by the rename, weakening audit-history integrity.
  - confidence: 0.9
