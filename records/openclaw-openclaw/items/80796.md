## Summary

Keep open: the bug is valid and the PR is pointed at the right doctor-side scrub path with sufficient real behavior proof, but current head still has a replacement race that can drop append-only audit records.

## What This Changes

The PR adds a config-audit scrub helper, a doctor health contribution, and tests to detect and rewrite historical `config-audit.jsonl` `argv`/`execArgv` values through the existing redactor.

## Best Possible Solution

Keep the doctor-side scrub, but make the replacement step detect appends through the whole temp-write-to-rename interval or coordinate with audit writers before landing.

## Reproduction Assessment

Yes, by source inspection and supplied logs. Current main redacts newly captured argv fields but has no historical scrub path, while the PR proof shows the new doctor path finding and scrubbing real historical residue.

## Solution Assessment

No, not as-is. Doctor-side scrubbing through the existing redactor is the narrow maintainable direction, but the final replacement still needs a guard that covers appends after temp-file writing and before rename.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Recheck the audit log after writing the temp file:** `src/config/io.audit.ts:574-578`
  - body: The append guard stats `config-audit.jsonl` before writing the temp file, then renames the transformed snapshot over the live log.
