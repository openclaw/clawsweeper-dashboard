## Summary

Keep this PR open: the doctor-side scrub is the right focused fix for the historical config-audit credential residue, but the current replacement sequence can still overwrite audit records appended after the size check and before the final rename.

## What This Changes

The PR adds a `scrubConfigAuditLog` helper, a doctor health contribution, and config-audit tests to preview and rewrite historical `config-audit.jsonl` `argv`/`execArgv` values through the existing redactor.

## Best Possible Solution

Keep the doctor-side scrub, but guard the final replacement against appends after temp-file creation, or use an equivalent coordination strategy, with a regression test for that exact window.

## Reproduction Assessment

Yes. Current main source shows forward redaction but no existing-log rewrite path, and the PR/linked issue provide real logs showing historical plaintext entries before the patch and scrubbed entries after it.

## Solution Assessment

No, not as-is. Reusing the existing redactor from a doctor `--fix` path is the right maintainable direction, but the final replacement step must not be able to drop audit records appended during the scrub.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recheck the audit log after writing the temp file:** `src/config/io.audit.ts:574-578`
  - body: The append guard stats `config-audit.jsonl` before `writeFile(tmp)`, then renames the transformed snapshot over the live log. An audit entry appended while the temp file is being written, or just after it returns, is not detected and will be lost by the rename, so the scrub can still erase append-only audit history.
  - confidence: 0.9
