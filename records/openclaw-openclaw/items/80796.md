## Summary

Keep this PR open: the bug is real and the proposed doctor-side scrub is the right direction, but the latest head still has a concrete append-loss race before merge.

## What This Changes

The PR adds a doctor health contribution and `scrubConfigAuditLog` helper to rewrite historical `config-audit.jsonl` `argv` and `execArgv` values through the existing redactor, with focused tests.

## Best Possible Solution

Keep the doctor-side historical scrub, but make the replacement step coordinate with audit writers or otherwise prove the live log is unchanged at replacement time, with a regression test for the post-stat append window.

## Reproduction Assessment

Yes. Current main is source-reproducible because it redacts new audit snapshots but has no historical scrub path, and the supplied logs show the PR scrubbing real pre-redactor entries.

## Solution Assessment

No, not as-is. Doctor-side scrubbing through the existing redactor is the narrow maintainable fix, but the final rename needs a stronger concurrent-append guard before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard the final rename against concurrent appends:** `src/config/io.audit.ts:574-578`
  - body: The append guard stats `config-audit.jsonl` before writing the temp file, then renames the transformed snapshot over the live log. An audit entry appended while `writeFile(tmp)` is running, or just before the rename, is not detected and will be lost, so the scrub can still erase append-only audit history.
  - confidence: 0.9
