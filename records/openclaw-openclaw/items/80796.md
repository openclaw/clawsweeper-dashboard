## Summary

Keep open: this PR targets a real historical config-audit credential residue bug, but it needs after-fix real behavior proof and a fix for the live append-only audit-log rewrite race before merge.

## What This Changes

The PR adds a doctor health contribution and `scrubConfigAuditLog` helper to redact historical `argv`/`execArgv` values in `config-audit.jsonl`, plus focused config-audit tests.

## Best Possible Solution

Land a doctor `--fix` scrub that reuses the forward redactor, preserves permissions, guards or merges concurrent audit records, and includes redacted real CLI proof plus regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main redacts only new snapshots and appends audit records without any historical rewrite path. I did not execute the repro because this review was read-only, but the linked issue also provides real before-state grep evidence.

## Solution Assessment

No as-is: a doctor-side scrub is the right narrow direction, but the implementation must not replace a live append-only audit log in a way that can drop concurrent records.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid overwriting concurrent audit appends:** `src/config/io.audit.ts:558`
  - body: The scrub reads `config-audit.jsonl` once and later renames that snapshot over the live file. Existing `appendConfigAuditRecord` callers can still append to the same path during that window, so any config write/observe record added after `readFile` and before `rename` is deleted by the swap. Because this is a tamper/security audit log, `doctor --fix` should detect, abort, or merge file changes instead of replacing the live append-only log unconditionally.
  - confidence: 0.86
