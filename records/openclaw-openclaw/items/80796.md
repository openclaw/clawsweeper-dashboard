## Summary

Keep this PR open: the fix direction and real behavior proof are solid, but latest head still has a narrow audit-integrity race where a config-audit append after the size check can be overwritten by the final rename.

## What This Changes

The PR adds a doctor config-audit scrub that rewrites historical `config-audit.jsonl` `argv`/`execArgv` values through the existing redactor, wires it into doctor health, and adds filesystem-based unit coverage.

## Best Possible Solution

Keep the doctor-side scrub, but make the replacement step avoid losing appends that happen after temp-file creation or coordinate the scrub with audit writers before landing.

## Reproduction Assessment

Yes. The linked issue and PR proof show live historical entries being scrubbed, and the remaining PR-head race is source-reproducible from the stat/write/rename ordering in `scrubConfigAuditLog`.

## Solution Assessment

No, not as-is. Doctor-side scrubbing through the existing redactor is the narrow maintainable direction, but the final replacement must not be able to drop append-only audit records.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard the final rename against late appends:** `src/config/io.audit.ts:574-578`
  - body: The append guard stats `config-audit.jsonl` before writing the temp file, then renames the transformed snapshot over the live log. Any audit entry appended while the temp file is being written, or just after it returns, is not detected and can be lost by the rename, so the scrub can still erase append-only audit history.
  - confidence: 0.9
