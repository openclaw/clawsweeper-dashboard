## Summary

Keep open, but this PR is not ready to merge: the useful goal is plausible, yet the diff conflates Telegram private-chat `message_thread_id` topics with Bot API direct messages chat `direct_messages_topic_id`, and the external PR has no real Telegram proof.

## What This Changes

The PR adds a Telegram helper that falls back from `message_thread_id` to `direct_messages_topic.topic_id` and uses it in inbound context, native commands, runtime handlers, sequential keys, and regression tests.

## Best Possible Solution

Model BotFather private-chat topics and Bot API direct messages chat topics as distinct Telegram thread specs, emit the matching send parameter for each, and add focused tests plus real Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: the PR maps `direct_messages_topic.topic_id` into the existing DM thread path, and current main then emits `message_thread_id`. I did not run a live Telegram account in this read-only review.

## Solution Assessment

No: the shared fallback hides two distinct Bot API surfaces behind one id. The safer fix is an explicit typed path that preserves `message_thread_id` for private-chat topics and uses `direct_messages_topic_id` for direct messages chat topics.
