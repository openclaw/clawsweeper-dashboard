## Summary

Keep open: this is a focused Telegram bug-fix PR, but it still needs real Telegram proof before merge and the diff has a topic-id contract issue that should be resolved first.

## What This Changes

The PR adds a Telegram helper that falls back from `message_thread_id` to `direct_messages_topic.topic_id` and wires it through inbound context, native commands, runtime handlers, sequential keys, and regression tests.

## Best Possible Solution

Preserve separate handling for `message_thread_id` and direct-message topic identifiers, keep focused regression tests, and land only after live Telegram proof shows replies stay in the originating DM topic.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection shows current main only reads `message_thread_id` on the relevant paths, but the `direct_messages_topic` behavior still needs a real Telegram/BotFather threaded-mode probe.

## Solution Assessment

No: centralizing thread-id resolution is a good shape, but this implementation should not treat direct-message topic ids as `message_thread_id`. The safer fix keeps the API concepts distinct and proves the chosen mapping in a real Telegram setup.
