## Summary

Keep open: this is a plausible Telegram DM-topic routing bug area, but the PR lacks required real Telegram proof and the patch appears to collapse a distinct Bot API direct-message-topic field into the existing `message_thread_id` path.

## What This Changes

The PR adds a Telegram helper that falls back from `message_thread_id` to `direct_messages_topic.topic_id` and uses it across inbound context, native commands, runtime handlers, sequential keys, and regression tests.

## Best Possible Solution

Model direct-message-topic metadata separately from forum/private-chat `message_thread_id`, route it through the Telegram send/reply parameter builders with focused regression tests, and land only after redacted live Telegram proof shows replies stay in the intended topic.

## Reproduction Assessment

No live reproduction was established in this read-only review, but the issue is source-reproducible: current main ignores `direct_messages_topic` on inbound updates and has no send/reply parameter path for `direct_messages_topic_id`. The related prior report includes live Telegram probes, but this PR still needs after-fix proof.

## Solution Assessment

No, not as currently patched: centralizing thread-id resolution is a reasonable shape, but collapsing `direct_messages_topic.topic_id` into `message_thread_id` is not the safest API-boundary fix. The safer solution is to carry direct-message-topic IDs separately through the Telegram routing and send/reply paths, then prove the real Telegram behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve direct-message topic parameters separately:** `extensions/telegram/src/bot/helpers.ts:101-102`
  - body: The helper turns `direct_messages_topic.topic_id` into the existing thread id, which later flows through `buildTelegramThreadParams` and reply/send paths that only emit `message_thread_id`. Keep this field separate through the send/reply path and prove it with live Telegram output.
