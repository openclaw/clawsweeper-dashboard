## Summary

Keep open: the PDF-side bug is still real on current main, but this branch is not merge-ready because it lacks real behavior proof, still has a blocking PDF fallback regression, and overlaps a narrower open PDF fix that does not fully cover the broader registry-backed baseUrl override target.

## What This Changes

The PR exports the unknown-model error helper, removes the registry-only media resolver, and switches image/PDF model resolution toward config-aware provider overrides with PDF/image regression tests.

## Best Possible Solution

Refresh the PDF side around the approved config-aware resolver boundary, preserve current main’s image behavior, honor both registry-backed provider overrides and config-defined custom providers, and keep PDF text extraction working for resolved text-only models.

## Reproduction Assessment

Yes, source-reproducible. Current main’s PDF execution path still uses the cfg-less registry helper before `model.baseUrl`, and the linked PDF runtime report shows the same resolver family failing for configured providers.

## Solution Assessment

No, not as-is. The resolver direction is valid, but this branch should be rebased/reduced to the remaining PDF behavior and the PDF guard should treat the resolved model’s actual text-only capability as sufficient while still rejecting synthesized unknown fallbacks.
