## Summary

Keep open: current main already implements the image-side config-aware path, but the PDF path still uses registry-only resolution, and this PR still has a concrete PDF fallback bug while overlapping the narrower open #65746.

## What This Changes

The PR switches image and PDF media model lookup to config-aware resolution, exports the unknown-model helper, removes the registry-only media resolver, and adds image/PDF regression tests.

## Best Possible Solution

Choose one PDF resolver boundary: land the narrower #65746 if maintainers prefer the local media helper, or rebase and reduce this PR to PDF-only config-aware resolution that preserves dynamic/text-only fallback behavior and the image implementation already on main.

## Reproduction Assessment

Yes. Source inspection on current main shows the PDF tool still calls the registry-only helper with no config input, and the PR finding is source-reproducible with a plugin-dynamic text-only model plus a PDF containing both images and extractable text.

## Solution Assessment

No. The direction is valid, but this branch is not the best landing shape yet because the image side is already implemented on main, the PDF guard rejects a valid dynamic-model fallback case, and #65746 offers a narrower competing PDF fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow dynamic PDF models through text fallback:** `src/agents/tools/pdf-tool.ts:239-240`
  - body: `resolveModelWithRegistry` can return a plugin-dynamic text-only model that is not declared in `models.providers` and is not found by `modelRegistry.find`. This guard then throws before the existing extracted-text fallback, so PDFs with images plus extractable text can fail instead of returning a text-only summary. Treat the resolved model's known input/source as sufficient while still rejecting synthesized fallback models.
