## Summary

Keep open: current main already covers the image side, but the PDF tool still uses a registry-only resolver that ignores `models.providers` overrides; the PR is directionally valid, but it is not merge-ready because its PDF guard introduces a source-reproducible fallback regression and it overlaps an open narrower PDF fix.

## What This Changes

The PR exports the unknown-model helper, removes the shared registry-only media resolver, switches image/PDF model lookup toward `resolveModelWithRegistry`, and adds image/PDF regression tests for configured provider routing and text-only fallback behavior.

## Best Possible Solution

Land a refreshed PDF-only config-aware resolver that applies provider overrides to registry-backed/native PDF models and configured custom models, preserves text-only extraction fallback for real resolved models, and adds focused PDF regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows the PDF tool still resolves through a cfg-less registry helper, and the PR regression is source-reproducible from its new guard with a plugin-dynamic text-only model plus a PDF that has both images and extractable text.

## Solution Assessment

No. The direction is valid, but this branch is not the best landing shape because current main already implements the image side, the PDF guard rejects a valid dynamic-model text fallback case, and the open narrower PDF PR covers only part of the remaining behavior.
