## Summary

Keep this PR open for maintainer decision rather than cleanup-close: current main already covers the image-side resolver path, but the PDF path still has the reported registry-only behavior, and this branch has an unresolved PDF fallback finding plus overlap with the narrower open #65746.

## What This Changes

The PR exports the unknown-model error helper, removes the registry-only media resolver, switches image and PDF model lookup toward `resolveModelWithRegistry`, and adds image/PDF regression tests for configured provider behavior.

## Best Possible Solution

Pick one canonical PDF path: either land the narrower #65746 approach if maintainers want a local media helper, or rework this PR so it preserves the image behavior already on main and updates only PDF execution-time model resolution through an approved config-aware resolver that handles registry-backed, config-defined, and plugin-dynamic models.

## Reproduction Assessment

Yes. The original PDF problem is reproducible from the #65742 custom `models.providers` setup, and current main still routes PDF execution through the registry-only helper. The PR finding is reproducible with a plugin-dynamic text-only resolved model and a PDF containing both images and extractable text.

## Solution Assessment

No. The direction is valid, but this branch is not currently the best landing shape because the image half is already implemented on main, the PDF guard rejects plugin-dynamic resolved text-only models, and #65746 offers a narrower competing PDF-only boundary.

- **[P2] Allow plugin-resolved PDF models through text fallback:** `src/agents/tools/pdf-tool.ts:239-240`
