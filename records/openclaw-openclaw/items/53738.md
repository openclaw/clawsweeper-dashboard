## Summary

Keep open: the remaining PDF bug is real on current main, but this branch is missing contributor real-behavior proof, is merge-conflicting, overlaps a narrower open PDF PR, and still introduces a PDF text-only fallback regression.

## What This Changes

The PR exports `buildUnknownModelError`, removes the registry-only media resolver, routes image and PDF model resolution through `resolveModelWithRegistry`, and adds image/PDF regression tests for configured provider handling.

## Best Possible Solution

Keep the current image implementation and land one PDF resolver boundary that honors `models.providers` overrides for both registry-backed and config-defined models while preserving text-extraction fallback behavior.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main's PDF execution path still uses a cfg-less registry resolver while the config contract defines `models.providers.*.baseUrl`; the image side no longer reproduces because main is already config-aware.

## Solution Assessment

No, not as-is. The resolver direction is valid for the remaining PDF bug, but the branch is partly superseded by main, lacks real proof, overlaps a narrower open PDF fix, and needs the plugin dynamic text-only fallback regression fixed before merge.
