## Summary

Keep open: this PR targets a real media/PDF provider-override gap, but current main already covers the image side and the PDF branch is not ready because it lacks real behavior proof and introduces a text-only fallback regression. The narrower open PDF PR overlaps the config-defined custom-provider miss, but it does not cover registry-backed provider baseUrl overrides, so this PR is not safely closeable as a duplicate.

## What This Changes

The PR exports the unknown-model helper, removes the registry-only media resolver, and switches image/PDF media model lookup toward `resolveModelWithRegistry` with regression tests.

## Best Possible Solution

Land a refreshed PDF-specific fix that uses the existing config-aware model resolution contract for registry-backed provider overrides and config-defined custom providers, preserves current main image behavior, and keeps PDF text-extraction fallback for resolved text-only models.

## Reproduction Assessment

Yes. Source inspection on current main shows the PDF tool still resolves through a cfg-less registry helper before using `model.baseUrl`, while docs define `models.providers` as the override surface; the linked PDF runtime report also demonstrates the same resolver family failing for configured providers.

## Solution Assessment

No, not as-is. The intended resolver direction is valid, but this branch should be rebased and reduced to the remaining PDF behavior, and the PDF guard should treat the resolved model's actual text-only capability as sufficient instead of only trusting cfg or registry entries.

## Review Findings
