## Summary

Keep this open. Current main still leaves the core artifact class unresolved: Telegram inbound voice/media reaches the shared buffer media store, and that store still writes buffers directly to the final media path instead of staging to a temp file and renaming after a complete write.

## What This Changes

The issue requests safer Telegram inbound media handling for ENOSPC so failed downloads do not leave zero-byte or partial voice files that later produce opaque transcription EOF errors.

## Best Possible Solution

Finish this in the shared media store: stage buffer writes in the same directory, remove the temp file on failure, and rename into place only after the full write succeeds. Keep the existing Telegram failure return and tiny-audio transcription guard, and add a focused media-store regression test for failed writes leaving no final artifact.

## Reproduction Assessment

Yes. The issue has a plausible live disk-full repro, and a deterministic test can mock the shared buffer write path to fail with ENOSPC and assert no final media id path exists afterward.

## Solution Assessment

Yes. The narrowest maintainable fix is in `writeSavedMediaBuffer`, because Telegram inbound media and other plugin buffer saves share that helper; fixing Telegram alone would leave the same final-path artifact risk for other buffer callers.
