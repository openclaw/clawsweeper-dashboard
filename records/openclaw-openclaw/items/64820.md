This PR extracts Feishu message parsing into `message-parser.ts`, lazy-loads Feishu message/card handlers from the account monitor, updates sequential-key imports, and adds parser/import-graph tests.

## Best Possible Solution

Land a current-main Feishu plugin fix that preserves parser semantics, breaks eager handler imports through real `*.runtime.ts` boundaries, covers message/reaction/menu/card paths, and includes bundle plus real Feishu proof.

Yes. The linked bug report gives concrete multi-bot Feishu group mention steps and logs, and current main still carries the implicated static import graph; I did not live-run the Feishu setup in this read-only review.

## Solution Assessment

No. The parser split and lazy-loading direction is right, but this branch must preserve current parser fields, use runtime boundaries, fix path handling in tests, rebase, and provide real behavior proof before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve reply-target fields in the extracted parser:** `extensions/feishu/src/message-parser.ts:61-75`
  - body: Current main carries `replyTargetMessageId` and `suppressReplyTarget` into `FeishuMessageContext`, but the extracted parser builds `ctx` without them. A rebase that keeps this copy would drop reply threading and suppression metadata for Feishu replies and synthetic messages.
  - confidence: 0.9
- **[P2] Route lazy handlers through runtime boundaries:** `extensions/feishu/src/monitor.account.ts:410-422`
  - body: The lazy loaders import `./bot.js` and `./card-action.js` directly. This bypasses the repo's runtime-boundary pattern for lazy production imports, so the bundler can inline the same graph and leave the TDZ crash unresolved; use real `*.runtime.ts` boundaries and build-check for `[INEFFECTIVE_DYNAMIC_IMPORT]`.
  - confidence: 0.86
- **[P2] Resolve source paths with fileURLToPath:** `extensions/feishu/src/message-parser.test.ts:154`
  - body: The source-reading tests derive paths from `new URL(import.meta.url).pathname`, which leaves URL encoding and Windows path quirks in place. Use `fileURLToPath(import.meta.url)` before `path.dirname(...)` so these tests do not fail in supported paths containing spaces or on Windows.
  - confidence: 0.82
