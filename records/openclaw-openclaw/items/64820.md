Full review comments:

- **[P2] Preserve reply-target fields in the extracted parser:** `extensions/feishu/src/message-parser.ts:61-75`
  - body: Current main carries `replyTargetMessageId` and `suppressReplyTarget` through `FeishuMessageContext` and later uses them for reply targeting. The extracted parser builds `ctx` without those fields, so rebasing this branch as-is would drop reply threading and suppression metadata.
  - confidence: 0.9
- **[P2] Avoid loading bot.js at top level in Feishu tests:** `extensions/feishu/src/message-parser.test.ts:15`
  - body: The Feishu test lane runs non-isolated, so this top-level import can cache the real `bot.js` before other tests install their `vi.mock("./bot.js")` stubs. That can make handler mocks miss calls in adjacent monitor tests.
  - confidence: 0.86
- **[P2] Route lazy handlers through runtime boundaries:** `extensions/feishu/src/monitor.account.ts:405-422`
  - body: The lazy loaders import `./bot.js` and `./card-action.js` directly. Repo policy requires a narrow `*.runtime.ts` lazy boundary plus build verification for dynamic imports, otherwise bundling can pull the handler graph back into the same chunk and leave the TDZ crash unresolved.
  - confidence: 0.86
- **[P3] Resolve source paths with fileURLToPath:** `extensions/feishu/src/message-parser.test.ts:154-155`
  - body: The source-reading tests derive paths from `new URL(import.meta.url).pathname`, which keeps URL encoding and Windows drive quirks. Use `fileURLToPath(import.meta.url)` before `path.dirname(...)` so these tests work in supported paths containing spaces and on Windows.
  - confidence: 0.82
