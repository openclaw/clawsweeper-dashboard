## Label Justifications

## Summary

Keep open: current main and the latest release still preserve user-selected `/model` overrides across `/new`/`/reset`; only auto fallback pins are cleared. The remaining request conflicts with current tests/docs and should be handled as a cross-channel product decision, likely alongside the open WebChat model-inheritance report.

## What This Changes

The issue requests that `/new` in channel-bound sessions reset a persisted session model override back to the configured default model.

## Best Possible Solution

Make one maintainer decision for reset/default model semantics across channel-bound sessions, WebChat children, and parent-inherited sessions, then implement the shared behavior with gateway, auto-reply, UI, tests, and docs aligned.

## Reproduction Assessment

Yes, source-level: current main preserves user-driven `modelOverride` state in the reset path, and gateway tests assert that explicit overrides survive `sessions.reset`. I did not run a live Feishu DM, but the reported channel-bound key flows through the same reset/session-store logic.

## Solution Assessment

Unclear/no: automatic clearing on every bare `/new` is not obviously the best fix because it conflicts with the current strict session-selection contract. A safer direction is a maintainer-approved shared reset/default policy or explicit default affordance that covers Feishu and the related WebChat inheritance case together.
