## What This Changes

## Best Possible Solution

## Reproduction Assessment

Yes, from source inspection. Current main and v2026.6.10 parse explicit invalid cron add no-output timeout values with a permissive helper, so the invalid value is omitted before cron.add instead of rejected.

## Solution Assessment

Yes. Validating inside cron add payload construction is the narrow owner-boundary fix and matches the existing cron edit contract; moving this to gateway normalization would not give the same immediate CLI error.
