## Summary

Keep open for normal maintainer review. Source review confirms current main and the latest release still silently omit an explicitly invalid cron add no-output timeout, while this PR applies the narrow CLI validation fix with focused regression coverage and sufficient terminal proof.

## What This Changes

The PR changes cron add command-payload parsing to reject explicitly invalid --no-output-timeout-seconds values and adds regression coverage for invalid values.

## Best Possible Solution

Land the focused CLI validation fix if maintainers accept the invalid-value compatibility change; keep broader numeric-option parity as a separate follow-up.

## Reproduction Assessment

Yes, from source inspection. Current main and v2026.6.10 parse explicit invalid cron add no-output timeout values with a permissive helper, so the invalid value is omitted before cron.add instead of rejected.

## Solution Assessment

Yes. Validating inside cron add payload construction is the narrow owner-boundary fix and matches the existing cron edit contract; moving this to gateway normalization would not give the same immediate CLI error.
