## Summary

Keep open. Current main has better diagnostics and several vision-routing improvements, but it does not prove the reported combined ARM64 packaged-install failure is fixed; the linked merged PR was explicitly diagnostics-only and the sharp/runtime-dependency half is still tied to open follow-up context.

## What This Changes

The issue reports ARM64 vision-capable models being rejected as text-only and image optimization failing in packaged installs, including macOS where sharp appears available and Linux where it is missing.

## Best Possible Solution

Reproduce both failure modes independently on current main or the next package: configured image-capable providers should resolve to an image-capable media-understanding provider, and packaged ARM installs should stage or load media-understanding-core native dependencies or report the exact dependency failure.

## Reproduction Assessment

Unclear: the issue has plausible 2026.4.26 ARM64 npm-global steps and logs, but current main has newer diagnostics and runtime-dependency behavior. A high-confidence reproduction needs current-main or current-package logs showing resolved provider/model/input plus the sharp staging/load result.

## Solution Assessment

No: closing on diagnostics and partial routing improvements would be premature. The maintainable path is to split model capability resolution from packaged native dependency staging, prove which current-main path still fails, and fix that narrower surface.
