## Summary

Keep this issue open. Current main has better diagnostics and several related routing improvements, but it does not prove the reported combined ARM64 packaged-install regression is fixed: the image runtime still rejects a resolved text-only model, image optimization still depends on `media-understanding-core` loading `sharp`, and current docs/postinstall policy say bundled plugin dependency staging is not performed.

## What This Changes

The issue reports that OpenClaw 2026.4.26 rejects configured vision-capable models and then fails image optimization on macOS/Linux ARM64 packaged installs, including a macOS case where `sharp` appears available in the shell.

## Best Possible Solution

Keep the issue as the current tracker until maintainers reproduce the two failure modes independently on current main: provider/model resolution should choose an image-capable media-understanding path for the reported configs, and packaged ARM installs should either load image-ops dependencies reliably or surface a precise dependency/action error. Any fix should land in the owner surface, with routing changes in media-understanding/model resolution and dependency policy changes in the packaged plugin/runtime path.

## Reproduction Assessment

Unclear: the issue provides plausible 2026.4.26 ARM64 packaged-install steps and logs, and current code still contains the relevant failure paths. A high-confidence reproduction needs current-main or current-package logs showing the resolved provider/model/input diagnostics plus whether `media-understanding-core` can resolve `sharp` from the packaged install.

## Solution Assessment

No current solution is proven complete. PR #74567 was the right narrow diagnostics step, but the best full solution is to use those diagnostics to isolate provider capability resolution from native image-ops packaging and then fix only the path that still fails.
