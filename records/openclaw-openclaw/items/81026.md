## Summary

Keep open: this targets a real Codex app-server MCP gap, but the patch still appears to miss a packaged dist entry for the new private SDK subpath, and the supplied proof stops at helper/test output rather than a live app-server thread.

## What This Changes

The PR projects user-configured `mcp.servers` into Codex app-server `mcp_servers` thread config for new and resumed threads, with helper tests, app-server wiring tests, SDK alias plumbing, a generated baseline hash, and a changelog entry.

## Best Possible Solution

Keep the app-server MCP projection, but make the private helper follow the existing packaged Codex SDK build contract or avoid a new private subpath, then add live app-server proof showing user MCP tools are available.

## Reproduction Assessment

Yes. Source inspection shows current main never projects `cfg.mcp.servers` into Codex app-server `thread/start` or `thread/resume`, while the existing Codex CLI path does project normalized servers as `mcp_servers`; I did not run a live app-server session in this read-only review.

## Solution Assessment

No, not yet. The projection approach is narrow and maintainable, but the new private SDK subpath needs packaged dist/build plumbing or a different already-routed helper surface before this can merge safely.
