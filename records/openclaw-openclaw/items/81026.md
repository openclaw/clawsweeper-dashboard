## Summary

Keep open: the reported Codex app-server MCP bug and the fix direction are valid, but this patch adds a new local-only plugin SDK subpath without wiring it into the Codex-private alias/build paths, so bundled Codex can fail to load before the projection runs. The supplied proof is useful, but it does not yet show the real app-server runtime after the fix.

## What This Changes

The PR maps user `mcp.servers` into Codex app-server `mcp_servers` config for new and resumed threads, with a private SDK helper, regression tests, generated baseline hash, and changelog entry.

## Best Possible Solution

Keep the app-server MCP projection, but route and build the private Codex helper through the existing bundled-Codex local-only SDK machinery, then verify it with a real app-server runtime path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main never projects `cfg.mcp.servers` into Codex app-server `thread/start` or `thread/resume`, while the CLI runtime does project `mcp_servers`. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

No, not as written. The projection approach is narrow and maintainable, but the new private SDK subpath must be added to the bundled-Codex alias/build contract or replaced with an already-routed surface before this can merge.
