## Summary

This PR should stay open: it targets a real, source-reproducible Codex app-server MCP projection gap, but the current patch still routes bundled Codex through a new private SDK subpath without adding the matching packaged dist/build entry.

## What This Changes

The PR projects user `mcp.servers` into Codex app-server thread `mcp_servers`, adds a private SDK facade and alias allowlist, adds focused regression tests, refreshes the generated SDK baseline hash, and adds a changelog entry.

## Best Possible Solution

Keep the MCP projection approach, but route the helper through an emitted package entrypoint or an already-built runtime surface, then add redacted live Codex app-server proof showing user MCP tools are available.

## Reproduction Assessment

Yes, source-reproducible: current `main` lacks a Codex app-server path that projects `cfg.mcp.servers` into `thread/start` or `thread/resume`, while the Codex CLI path already emits normalized `mcp_servers`. I did not run a live app-server session in this read-only review.

## Solution Assessment

No, not yet. The projection itself is the right narrow fix, but this implementation must either emit the new private SDK subpath in packaged builds or use an existing built helper surface before merge.
