## Label Justifications

- `P2`: Provider usage retrieval is broken in proxy-only environments, but the affected surface is limited to status/quota display rather than core agent execution.
- `merge-risk: 🚨 compatibility`: Existing installs with proxy env configured will route provider-usage requests through that proxy path instead of using direct global fetch first.

## Summary

Keep open for maintainer review: current main and v2026.6.8 still have the provider-usage proxy bug, and this PR is the focused fix candidate with no blocking code or security findings.

## What This Changes

The PR changes provider-usage loading to preserve explicit fetch injection, otherwise use the existing env proxy-aware fetch helper before falling back to global fetch, adds provider-usage proxy coverage, and removes an unused MCP type.

## Best Possible Solution

Land this loader-level fix after maintainer acceptance of standard env-proxy precedence for provider usage, preserving explicit fetch injection and falling back to global fetch only when no proxy fetch resolves.

## Reproduction Assessment

Yes. Current main and v2026.6.8 resolve provider usage with plain fetch, then the OpenAI/Codex hook uses that fetch for the ChatGPT WHAM usage endpoint; I did not run a live proxy-only OpenClaw command in this read-only review.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of the fetch function passed to all usage hooks; putting proxy logic in the Codex hook or adding a new config knob would be broader and easier to drift.
