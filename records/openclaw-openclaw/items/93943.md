## Summary

Keep open: current main and the latest release still bypass proxy env for provider-usage fetches, while this PR is a focused fix candidate with sufficient local proxy-routing proof and no concrete review findings.

## What This Changes

The PR changes provider-usage loading to prefer an explicit fetch, otherwise use the existing env proxy-aware fetch before global fetch, adds plugin-boundary regression coverage, and removes one unused private MCP type.

## Best Possible Solution

Land the loader-level fix after the PR body includes redacted real proof that loadProviderUsageSummary or openclaw status --usage --json routes the Codex usage request through HTTPS_PROXY while preserving explicit fetch precedence.

## Reproduction Assessment

Yes from source: current main and v2026.6.8 resolve provider usage with plain resolveFetch(opts.fetch), and the OpenAI/Codex hook passes that fetch into the chatgpt.com usage endpoint. I did not run a live proxy-only OAuth status command in this read-only review.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of the fetchFn shared by provider usage hooks; duplicating proxy handling inside the Codex hook or adding a new config knob would be broader.
