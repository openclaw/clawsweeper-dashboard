## Summary

Keep open: this PR is the active focused fix candidate for the linked provider-usage proxy bug. Current main and the latest release still choose plain fetch at the provider-usage loader, while the PR reuses the existing env-proxy fetch helper with focused plugin-boundary coverage and supplied terminal proof.

## What This Changes

The PR changes provider-usage loading to preserve explicit fetch first, otherwise use env proxy fetch before global fetch, adds plugin-boundary tests, and removes an unused MCP helper type.

## Best Possible Solution

Land this loader-level fix, or an equally narrow replacement, so provider usage preserves explicit fetch precedence and otherwise honors existing proxy env settings before using global fetch.

## Reproduction Assessment

Yes from source: current main resolves provider-usage fetch with plain resolveFetch(opts.fetch), and the OpenAI/Codex hook passes that fetch into the chatgpt.com usage request. I did not run a live proxy-only OpenClaw status command in this read-only review.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of the shared fetchFn passed to provider usage hooks; duplicating proxy logic inside the Codex hook, mutating the global dispatcher, or adding a config knob would be broader.
