## Summary

Keep open: current main and v2026.6.8 still use plain fetch for provider-usage loading, while this PR is the active focused fix candidate for the linked proxy-only Codex usage bug and I found no blocking patch defect.

## What This Changes

The PR makes provider-usage loading prefer an explicit fetch, otherwise use the existing env proxy-aware fetch helper before falling back to global fetch, and adds proxy precedence coverage plus a now-main-equivalent MCP type cleanup.

## Best Possible Solution

Land this loader-level fix, or an equivalent narrow replacement, so provider usage preserves explicit fetch precedence and otherwise honors existing proxy env settings before global fetch.

## Reproduction Assessment

Yes from source: current main and v2026.6.8 choose plain resolveFetch(opts.fetch) for provider usage, then pass that fetch into the OpenAI/Codex wham usage request. I did not run a live proxy-only OpenClaw status command in this read-only review.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of fetchFn fan-out; duplicating proxy handling inside the Codex hook or adding a new config knob would be broader than needed.
