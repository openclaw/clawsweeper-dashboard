## Summary

Keep open: this PR is the active focused fix candidate for a real provider-usage proxy bug, current main and v2026.6.8 still bypass env proxy at the loader boundary, and I found no blocking code defect. Maintainer review should focus on accepting the intentional proxy-env precedence change.

## What This Changes

The PR changes provider-usage loading to prefer an explicit fetch, otherwise use the existing env-proxy-aware fetch helper before falling back to global fetch, with plugin-boundary tests for proxy and explicit-fetch precedence.

## Best Possible Solution

Land this loader-level fix, or an equivalent narrow replacement, after maintainer acceptance of env-proxy precedence for provider usage.

## Reproduction Assessment

Yes from source: current main and v2026.6.8 resolve provider usage fetch with plain resolveFetch(opts.fetch), while the linked issue shows that direct fetch fails and the existing proxy helper succeeds in a proxy-only environment. I did not run a live proxy-only OAuth command in this read-only review.

## Solution Assessment

Yes. The loader boundary owns the fetchFn passed to provider plugins, and Codex/OpenAI usage consumes that injected fetch for the same upstream usage endpoint; adding a config knob or duplicating proxy policy inside the provider hook would be broader than needed.
