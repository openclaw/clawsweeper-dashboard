## Summary

Keep open: current main and v2026.6.8 still use plain fetch for provider-usage loading, and this PR is the active fix candidate for the canonical proxy-only Codex usage bug. The patch looks correct, with the main maintainer decision being whether to accept env-proxy precedence for this status/usage path.

## What This Changes

The PR changes provider-usage loading to prefer explicit fetch, otherwise use the existing env-proxy-aware fetch helper before global fetch, adds plugin-boundary proxy precedence tests, and removes an unused MCP Claude permission type already absent on current main.

## Best Possible Solution

Land this loader-level fix, or an equivalent narrow replacement, after maintainer acceptance of env-proxy precedence for provider usage.

## Reproduction Assessment

Yes from source, but not from a live proxy-only OpenClaw command in this review. Current main and v2026.6.8 choose plain resolveFetch(opts.fetch) and pass that fetch into the OpenAI/Codex wham usage request, while the linked issue provides a proxy-only environment where direct fetch fails and proxy fetch succeeds.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of fetchFn fan-out; preserving explicit fetch first and using the existing resolveProxyFetchFromEnv(env) before global fetch avoids duplicating proxy policy inside the OpenAI/Codex hook or adding a broader config knob.
