## Summary

Keep open: this is the active, focused fix candidate for the linked provider-usage proxy bug, and current main plus the latest release still use plain fetch at the loader boundary. The patch looks correct, with sufficient terminal routing proof, but the env-proxy precedence change is compatibility-sensitive and should get maintainer acceptance before merge.

## What This Changes

The PR changes provider-usage loading to prefer an explicit fetch, then an env-proxy-aware fetch, then global fetch, adds provider-usage proxy precedence tests, and removes an unused MCP Claude permission type.

## Best Possible Solution

Land this loader-level fix, or an equivalent narrow replacement, after maintainer acceptance of env-proxy precedence for provider usage.

## Reproduction Assessment

Yes from source and linked report evidence: current main still passes plain fetch into the Codex usage request, while the issue documents a proxy-only environment where direct fetch fails and proxy fetch succeeds. I did not run a live OAuth status command in this read-only review.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of fetchFn fan-out; preserving explicit fetch first and reusing resolveProxyFetchFromEnv(env) avoids duplicating proxy policy inside the OpenAI/Codex hook or adding a new config knob.
