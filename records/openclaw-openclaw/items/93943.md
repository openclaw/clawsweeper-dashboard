## What This Changes

The PR changes provider usage loading to use the existing environment proxy-aware fetch helper when no explicit fetch is supplied, adds plugin-boundary tests for that behavior, and removes one unused MCP type.

## Best Possible Solution

Land this loader-level fix, or an equally narrow replacement, so provider usage preserves explicit fetch precedence and otherwise honors existing proxy env settings before using global fetch.

## Reproduction Assessment

Yes from source: current main still resolves provider usage with plain fetch and passes that fetch into the Codex usage request. I did not run a live proxy-only openclaw status command in this read-only review.

## Solution Assessment

Yes. The provider-usage loader is the narrow owner of the fetchFn that fans out to provider usage hooks; duplicating proxy logic in the Codex hook, mutating the global dispatcher, or adding a new config knob would be broader than needed.
