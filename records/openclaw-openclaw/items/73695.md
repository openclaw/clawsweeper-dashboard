## Summary

Keep open. Current main still lacks WhatsApp live commentary, and this PR is the right general direction; the earlier source-delivery concern appears covered by current policy because `message_tool_only` makes `suppressDelivery` true, but one helper bug can still duplicate already-delivered long commentary text.

## What This Changes

The PR adds opt-in WhatsApp `commentaryDelivery: "live"`, agent commentary tracking and dedupe, bounded delivery callbacks, SDK reply-runtime exports, docs, changelog, and regression tests.

## Best Possible Solution

Keep the opt-in WhatsApp-only, text-only live-commentary design, but fix delivered-length accounting so final dedupe remains correct for long segments before normal maintainer validation.

## Reproduction Assessment

Yes. A focused helper test can store an 8,000-character delivered prefix with a larger `deliveredTextLength`, append new text to the same segment, and show that `resolveAssistantCommentaryDeltaText` currently returns already-delivered text from the truncated boundary.

## Solution Assessment

No as-is. The proposed seam is maintainable and scoped, but the long-segment dedupe helper must honor the recorded full delivered length or equivalent verification metadata before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor the recorded delivered commentary length:** `src/agents/pi-embedded-commentary.ts:43-46`
  - body: `rememberDeliveredCommentarySegment` stores only a bounded prefix while separately recording the full delivered length. This branch discards any recorded length greater than the stored prefix, so when a long commentary segment later appends more text, final payload assembly slices from the prefix length and resends text that was already delivered. Use the recorded length when the current segment still matches the stored prefix, or store enough verification metadata to avoid the duplicate.
  - confidence: 0.88
