## Summary

Keep open. The PR is useful and current main still lacks WhatsApp live commentary, but the patch is not merge-ready because it can bypass message-tool-only source-delivery suppression and can resend part of long already-delivered commentary segments.

## What This Changes

The PR adds opt-in WhatsApp live assistant commentary by threading an `onCommentaryReply` callback through agent/reply dispatch, adding config, text-only delivery, dedupe, abort bounds, docs, changelog, SDK docs, and tests.

## Best Possible Solution

Keep the opt-in, WhatsApp-only, text-only design, but gate commentary exactly like other automatic source output and make delivered-commentary tracking append-safe for long segments before normal maintainer validation.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by source inspection of the commentary-phase early return, and the PR bugs are reproducible by combining WhatsApp `commentaryDelivery: "live"` with `sourceReplyDeliveryMode: "message_tool_only"`, plus a helper case where a delivered segment stores an 8,000-character prefix with a longer recorded length.

## Solution Assessment

No as-is. The proposed seam is a maintainable direction, but the callback must honor `suppressAutomaticSourceDelivery` and the long-segment delta calculation must avoid resending already-delivered commentary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Suppress commentary under message-tool-only delivery:** `src/auto-reply/reply/dispatch-from-config.ts:1080`
  - body: This forwards `onCommentaryReply` whenever `suppressDelivery` is false, but `message_tool_only` suppresses automatic source replies through `suppressAutomaticSourceDelivery` while still allowing processing. With WhatsApp live commentary enabled, that leaks intermediate commentary to the source chat even though visible output is supposed to come only from the message tool. Gate this callback the same way as the other automatic source-output callbacks and add a `message_tool_only` regression test.
  - confidence: 0.92
- **[P2] Honor recorded lengths for truncated commentary:** `src/agents/pi-embedded-commentary.ts:43-46`
  - body: `rememberDeliveredCommentarySegment` stores only an 8,000-character prefix while separately recording the full delivered length. This helper ignores any recorded length greater than the stored prefix, so appending to a long already-delivered segment resends the old text after character 8,000 before the new suffix. Use the recorded length when the current text still matches the stored prefix, or store enough verification metadata to avoid the duplicate.
  - confidence: 0.86
