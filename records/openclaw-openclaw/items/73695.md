## Summary

Keep open. Current main still suppresses assistant commentary and lacks the WhatsApp config surface, so the PR is not redundant and is useful follow-through from closed PR #57484; however, the current PR head still has a long-commentary dedupe bug that can resend already-delivered text after the stored 8,000-character prefix.

## What This Changes

The PR adds an opt-in WhatsApp `commentaryDelivery: "live"` path by threading an agent commentary callback through reply dispatch, delivering text-only WhatsApp commentary, filtering delivered commentary from final payloads, and updating config, docs, changelog, SDK runtime docs, and tests.

## Best Possible Solution

Keep the opt-in, WhatsApp-only, text-only design, but repair the delivered-commentary delta helper so recorded delivered lengths are honored for truncated stored prefixes, add a focused long-commentary regression test, then run the targeted agent/WhatsApp tests and normal maintainer validation before merge.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by source inspection of the commentary suppression path, and the PR bug is reproducible at helper level with a delivered segment whose stored prefix is 8,000 characters but whose recorded delivered length is larger.

## Solution Assessment

No as-is. The proposed seam and WhatsApp-only opt-in are a reasonable maintainable direction, but the delta calculation must be fixed before this is safe to merge.
