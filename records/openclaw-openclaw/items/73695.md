## Summary

Keep open. Current main still lacks the opt-in WhatsApp live-commentary behavior, and the PR is a real implementation candidate carrying forward the closed unmerged #57484 work, but PR head still has a blocking long-commentary dedupe bug that can resend already-delivered text after the stored 8,000-character prefix.

## What This Changes

The PR adds opt-in WhatsApp `commentaryDelivery: "live"` support with account overrides, live text-only commentary delivery, final-reply dedupe, abort/timeout bounds, SDK runtime exports, docs, changelog, and targeted tests.

## Best Possible Solution

Keep the WhatsApp-only, default-off, text-only callback design, but repair the delivered-commentary delta helper so truncated stored prefixes still honor the recorded delivered length, add the long-segment regression, then rerun the targeted agent and WhatsApp validation before merge.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by source inspection: commentary-phase stream text returns early and no WhatsApp config field exists. The PR bug is reproducible at helper level with an 8,000-character stored delivered prefix, a larger recorded delivered length, and a later same-segment extension.

## Solution Assessment

No as-is. The proposed opt-in seam is a narrow maintainable direction, but the delta calculation must be fixed before merge so long live commentary is not partially resent.
