## Summary

Keep open. Current main still lacks the requested Feishu Bitable list-records query parameters, and the PR is useful, but it is not merge-ready because the new tool schema advertises the Search API object-array sort shape while the wired `appTableRecord.list` GET endpoint expects a string query value like `["Field DESC"]`.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters to `feishu_bitable_list_records` and forwards them to the Lark SDK list-records call.

## Best Possible Solution

Keep the PR open and repair it by preserving the useful pass-through parameters, correcting or translating the `sort` input to the list-records endpoint contract, and adding focused regression coverage plus the required user-facing changelog entry.

## Reproduction Assessment

Yes for the PR finding: static source inspection plus the Lark SDK types and Feishu/Lark list-records docs reproduce the mismatch between the advertised object-array `sort` input and the string query format accepted by the endpoint. Current main also clearly reproduces the missing-capability part because it only forwards pagination parameters.

## Solution Assessment

No, not as submitted. The narrow maintainable fix is either to document/accept the list-records string sort syntax or explicitly translate object syntax before forwarding, then cover the forwarding behavior in Feishu Bitable tests.
