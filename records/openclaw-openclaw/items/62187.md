## Summary

Keep open: current main still lacks these Feishu Bitable list-record parameters, and the PR is useful but not merge-ready because the new `sort` schema advertises search-endpoint object syntax for a list-endpoint passthrough.

## What This Changes

The PR extends `feishu_bitable_list_records` with optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters and forwards them to the Lark Bitable list-records call.

## Best Possible Solution

Land a corrected Feishu plugin change that forwards supported list-record query params with valid list sort syntax or explicit translation, plus targeted test coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an agent following the PR's advertised object-array `sort` example would call `appTableRecord.list` with syntax that the SDK separates from the list-record params and models on the search endpoint instead.

## Solution Assessment

No, not as currently proposed. The Feishu-owner pass-through boundary is narrow and maintainable, but the schema guidance must match `appTableRecord.list` or translate the object form before forwarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use valid list-record sort syntax:** `extensions/feishu/src/bitable.ts:496`
  - body: The new `sort` parameter description tells callers to send `[{"field_name":"Created Time","desc":true}]`, but this path forwards the value directly to `appTableRecord.list`; the Lark SDK models object-array sorting on `appTableRecord.search`, while list params take `sort?: string`. Agents following the schema example will generate invalid list-record sort requests, so update the schema to the list API syntax or translate the object form before forwarding.
