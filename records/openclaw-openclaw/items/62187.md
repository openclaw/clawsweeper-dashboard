## Summary

Keep open: the Feishu Bitable feature is useful and is not implemented on current main, but this PR still advertises an invalid list-record sort shape and has no real Feishu behavior proof.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters to the Feishu Bitable list-records tool and forwards them to the Lark SDK list call.

## Best Possible Solution

Land a corrected Feishu plugin update that uses the list API's sort string contract or translates structured input, adds focused forwarding coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection of the PR schema and forwarding path, compared with the locked Lark SDK list/search types, shows that an agent following the advertised object-array `sort` example would send the search-endpoint shape to the list endpoint.

## Solution Assessment

No. Passing through the supported list query params in the Feishu plugin is the right narrow direction, but the sort schema must match the list API or translate structured input before forwarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use valid list-record sort syntax:** `extensions/feishu/src/bitable.ts:496`
  - body: The new `sort` description tells agents to send object-style JSON, but this value is forwarded directly to `appTableRecord.list`; the locked SDK and list API expose `sort` as a string query value, while the object-array shape is for `appTableRecord.search`. Tool calls generated from this schema will produce invalid list-record sort requests, so update the schema to the list syntax or translate the object form before forwarding.
  - confidence: 0.93
