## Summary

Keep open. Current main still lacks these Feishu Bitable list-record query options, and the PR is directionally useful, but it is not merge-ready because the new tool schema advertises the search-endpoint object sort format while forwarding it unchanged to the list endpoint’s string `sort` query parameter.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters to `feishu_bitable_list_records` and forwards them to Lark’s Bitable list API.

## Best Possible Solution

Keep the pass-through feature, but align the tool schema with the list endpoint’s actual string query format or translate object-style sort input before forwarding, then add focused tests and the required changelog entry.

## Reproduction Assessment

Yes. Source inspection reproduces the current-main gap, and the PR-branch schema plus Feishu/SDK list-record contract provide a high-confidence reproduction path for invalid generated sort calls without needing repository mutation.

## Solution Assessment

No. Forwarding optional list query parameters is the right narrow direction, but the PR currently mixes list and search sort formats; the safer fix is to correct the schema or implement a deliberate translation layer with tests.
