## Summary

Keep open. Current main still lacks these Feishu Bitable list-record query controls, and the PR is directionally useful, but it is not merge-ready because the new `sort` tool schema advertises the search-endpoint object shape while the list-records query parameter expects string entries like `Field DESC`/`Field ASC`.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters to `feishu_bitable_list_records` and forwards them to the Lark list-records call.

## Best Possible Solution

Land a corrected Feishu plugin change that forwards the supported list-record query params, advertises valid list-record sort syntax or translates object input, and covers the new params with a narrow test plus changelog entry.

## Reproduction Assessment

Yes. Comparing the PR schema to the Lark list-records contract gives a high-confidence reproduction path: an agent following the advertised object-array `sort` example will send syntax that belongs to the search endpoint rather than the list query parameter.

## Solution Assessment

No, not as currently proposed. The pass-through boundary is the right narrow owner-module fix, but the schema guidance must match the list-records API or the tool should translate a structured sort object before forwarding.
