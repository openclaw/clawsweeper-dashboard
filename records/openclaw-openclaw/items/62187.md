## Summary

Keep open: current main does not yet expose the requested Feishu Bitable list-record parameters, and the PR is directionally useful, but it should not merge yet because the advertised `sort` shape is wrong for the list endpoint and the external PR still lacks real behavior proof.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters to the Feishu Bitable list-records tool and forwards them to the Lark SDK list call.

## Best Possible Solution

Land a corrected Feishu plugin update that uses the list API's sort string contract or translates structured input, adds focused forwarding coverage, and includes redacted real behavior proof.

## Reproduction Assessment

Yes for the review finding. Source inspection of the PR schema and forwarding path, compared with the locked Lark SDK list/search types, shows that an agent following the advertised object-array `sort` example would send the search-endpoint shape to the list endpoint.

## Solution Assessment

No. Passing through the supported list query params in the Feishu plugin is the right narrow direction, but the sort schema must match the list API or translate structured input before forwarding.

## Review Findings
