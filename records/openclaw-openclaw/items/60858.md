## Summary

Keep open. Current `main` still has the empty-real-content skip paths plus positive-only compaction token bookkeeping, the issue thread has multiple independent repro reports after earlier mitigations, and the related implementation candidate #69270 is still open rather than merged or shipped.

## What This Changes

The issue reports that compaction can repeatedly skip real sessions when empty or unpopulated message arrays reach the real-conversation guards, leaving stale trigger/accounting state instead of compacting.

## Best Possible Solution

Keep the no-provider-call guard for truly empty preparations, but make compaction trigger state and bookkeeping agree so fallback or zero-message outcomes refresh stale usage/trigger state across manual, auto, context-engine, and preflight compaction paths.

## Reproduction Assessment

Yes. The issue discussion provides concrete repros across Telegram/lossless-claw, TUI/omlx, idle main-agent runs, and WhatsApp/Baileys, and current source still contains the relevant empty-preparation skip and positive-only accounting paths; I did not run a live multi-channel session in this read-only review.

## Solution Assessment

No for the report's workaround: removing the guards would risk provider calls on empty compaction input. The narrow maintainable fix is to preserve the guard while repairing the upstream trigger/accounting disagreement, either through #69270 or an equivalent owned patch.
