## Label Justifications

- `P1`: Multiple real user deployments report compaction failing or looping across channels, causing context loss, resets, or channel degradation.
- `impact:data-loss`: The reported empty fallback path can replace real conversation history with a placeholder summary, losing usable session context.
- `impact:session-state`: The issue centers on compaction/session transcript state drifting from persisted real messages and stale token accounting.

## Summary

Keep open: current main still has the empty-content compaction skips and positive-only token accounting paths that match the multi-channel runtime reports, while the broader invariant fix remains in an open conflicting PR.

## What This Changes

The issue asks OpenClaw to compact real persisted conversations at threshold instead of skipping when prepared `session.messages` or compaction inputs arrive empty or fail the real-content classifier.

## Best Possible Solution

Keep the empty-input guard for genuinely idle sessions, but make compaction preparation, direct compaction, and token bookkeeping agree on the persisted transcript so real turns summarize and zero-token outcomes clear stale trigger state.

## Reproduction Assessment

Yes, source-backed but not live: the comments provide multiple concrete runtime repros, and current main still contains the direct empty-content skip, safeguard fallback boundary, and positive-only accounting paths. I did not run a live current-main multi-channel session.

## Solution Assessment

No to the report's workaround of deleting the guards, because true empty sessions still need to avoid provider dispatch. The maintainable fix is the invariant/accounting repair represented by https://github.com/openclaw/openclaw/pull/69270 or an equivalent focused patch with proof.
