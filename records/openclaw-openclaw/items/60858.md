## Summary

Keep open. Current main still contains the empty-real-content skip and fallback-boundary behavior, and the fresh issue discussion has multiple independent repros showing recurring compaction-safeguard attempts after the earlier loop mitigation. The related implementation PR is open/unmerged and not available as implemented-on-main evidence, so this is not ready for cleanup close.

## What This Changes

The issue reports that compaction is skipped or retriggered when `session.messages` or compaction preparation arrives empty despite real persisted conversation history, causing repeated fallback boundaries or context resets instead of threshold compaction.

## Best Possible Solution

Keep the empty-preparation safety boundary, but reconcile trigger and bookkeeping state so fallback or zero-message compactions reset stale usage and do not keep scheduling compaction. The final fix should cover manual `/compact`, auto-compaction, context-engine-owned compaction, and preflight transcript-size paths with focused regression coverage.

## Reproduction Assessment

Yes. The comments provide multiple concrete runtime reproductions, and current tests already exercise the empty-preparation fallback-boundary path; I did not run live tests in this read-only sweep.

## Solution Assessment

No, removing the guards is not the best fix. The safer direction is to preserve the guard as a no-provider-call safety check while fixing the upstream trigger/accounting disagreement, with #69270 looking like the current candidate path but still unmerged.
