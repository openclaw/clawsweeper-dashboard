## Summary

Keep open. Current main still contains the empty-real-content skip and fallback-boundary behavior, current accounting still ignores zero-token compaction outcomes in the relevant paths, and the issue discussion has multiple independent reproductions after earlier mitigations. The active related implementation candidate, #69270, is still open rather than merged/shipped evidence.

## What This Changes

The issue reports that compaction guards treat sessions as having no real conversation content because `session.messages` or prepared message arrays arrive empty, causing compaction to skip or repeatedly write fallback boundaries instead of reconciling state.

## Best Possible Solution

Keep the no-provider-call guard for truly empty preparations, but make compaction trigger and bookkeeping state agree: fallback or zero-message compaction outcomes should refresh stale usage/trigger state so manual `/compact`, auto-compaction, context-engine-owned compaction, and preflight transcript-size compaction do not keep scheduling empty work.

## Reproduction Assessment

Yes. The issue comments provide multiple concrete reproductions across Telegram/lossless-claw, TUI/omlx, idle main-agent runs, and WhatsApp/Baileys, and current tests directly exercise the empty-preparation fallback-boundary path; I did not run live sessions in this read-only review.

## Solution Assessment

No for the workaround described in the report: removing the guards would risk provider calls on empty compaction input. The maintainable direction is to preserve the safety guard while fixing the upstream trigger/accounting disagreement; #69270 appears aligned but is still unmerged.
