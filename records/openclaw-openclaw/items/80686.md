## Summary

Keep this PR open: the contended acquire-path change is directionally useful, but the patch is incomplete because the startup/doctor cleanup path can still remove a fresh payload-less lock without the new grace, and the supplied real behavior proof does not show the changed runtime path after the fix.

## What This Changes

The PR raises `ORPHAN_LOCK_PAYLOAD_GRACE_MS` to 30 seconds, updates payload-less lock boundary tests, and adds a changelog entry.

## Best Possible Solution

Use one shared payload-less-lock freshness policy for both contended acquisition and cleanup removal, then land with focused regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes, by source inspection. Current main creates an empty sidecar between fs-safe `open("wx")` and the payload write, then the contended reclaim path treats a 10-second empty lock as reclaimable because the payload-less grace is 5 seconds.

## Solution Assessment

No, not complete as written. Raising the constant covers contended acquisition, but the cleanup path still deletes payload-less locks without consulting the mtime grace, so the same race can remain during gateway startup or doctor repair.
