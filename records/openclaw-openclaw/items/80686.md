## Summary

Keep open: the reported lock bug is source-reproducible, but this PR should not merge because it replaces current fs-safe-backed lock semantics, drops existing cleanup behavior, and the supplied proof is from unpatched release logs rather than patched payload-less-grace behavior.

## What This Changes

This PR raises the payload-less session write-lock orphan grace from 5s to 30s, rewrites the lock acquisition/release implementation away from @openclaw/fs-safe, updates session-write-lock tests, and adds a changelog entry.

## Best Possible Solution

Make a narrow patch that preserves createFileLockManager, non-OpenClaw owner cleanup, and snapshot-safe release semantics while only widening the orphan payload grace and adding focused regression coverage.

## Reproduction Assessment

Yes from source: current main treats an empty lock payload as missing pid/invalid createdAt and reclaims it by mtime after the 5s grace when staleMs is larger. I did not run the focused test because this review is read-only.

## Solution Assessment

No for this PR as written. Raising the grace is a narrow maintainable fix, but replacing the fs-safe-backed lock manager and deleting cleanup coverage is not the safest way to solve it.
