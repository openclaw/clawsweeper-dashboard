## Summary

Keep open. The reported lock-grace bug is source-backed and still present on current main and the latest release, but this PR should not merge as written because it drops current false-live cleanup and snapshot-safe release semantics, and the supplied proof does not demonstrate patched behavior.

## What This Changes

The branch raises the payload-less session-write-lock orphan grace to 30 seconds while replacing fs-safe-backed lock handling with a custom in-process lock map, deleting owner-argv cleanup coverage, and adding a changelog entry.

## Best Possible Solution

Land a narrow patch that only adjusts the orphan payload grace/comment and focused tests while preserving the fs-safe manager, snapshot-safe release behavior, and false-live owner cleanup, then require patched-build proof before merge.

## Reproduction Assessment

Yes, at source level: create an empty `.jsonl.lock` older than 5 seconds but younger than `staleMs`, and current main reclaims it through the mtime fallback; the existing test covers a 10-second empty lock. I did not run the test because this review is read-only.

## Solution Assessment

No for this PR as written. Raising the grace is a narrow maintainable fix, but replacing the fs-safe-backed lock manager and deleting owner cleanup is broader than necessary and regresses existing behavior.
