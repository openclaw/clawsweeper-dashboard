## Summary

Keep open: the reported lock race is plausible from current source, but this PR should not merge as written because it removes owner-aware session-lock cleanup/API, bypasses the fs-safe snapshot checks that protect lock removal, and the supplied proof does not show patched payload-less grace behavior.

## What This Changes

The PR raises the session-write-lock payload-less orphan grace from 5s to 30s, replaces the fs-safe-backed session lock manager with a local implementation, trims owner-cleanup tests, and adds a changelog entry.

## Best Possible Solution

Make the grace/comment and focused tests change while preserving the fs-safe lock manager, owner-aware doctor cleanup, and snapshot-safe lock removal; merge only after patched-build proof.

## Reproduction Assessment

Yes from source, though not live-run in this read-only review: current main treats an empty lock payload as missing pid/invalid createdAt and reclaims by mtime after the 5s grace when staleMs is larger. The current focused test also documents reclaiming a 10-second-old empty lock.

## Solution Assessment

No for this PR as written. Raising the grace can be the narrow fix, but replacing the fs-safe-backed manager and removing owner-aware cleanup is not the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve owner-aware cleanup API:** `src/agents/session-write-lock.ts:513-522`
  - body: Removing the exported owner-argv reader type and readOwnerProcessArgs option leaves src/commands/doctor-session-locks.ts importing and passing symbols that no longer exist, so the branch will fail TypeScript. If the caller were simply stripped to compile, doctor --fix would also stop clearing fresh live .jsonl.lock files owned by non-OpenClaw processes, which current main tests cover.
  - confidence: 0.95
- **[P2] Keep snapshot-safe stale removal:** `src/agents/session-write-lock.ts:670-671`
  - body: After deciding a contended lock is stale, this removes lockPath directly. Two contenders can both decide the old file is stale; if one removes and recreates the lock before the other reaches this line, the second rm can delete the fresh holder's lock. fs-safe avoids this by checking the same file snapshot before removal.
  - confidence: 0.89
- **[P2] Keep snapshot-safe lock release:** `src/agents/session-write-lock.ts:183-194`
  - body: releaseHeldLock closes the handle and then deletes held.lockPath by path, with no file-identity or payload snapshot check. A contender that acquires a fresh lock at the same path during that window can have its new lock removed by the old release; the current fs-safe-backed release removes only if the observed lock is unchanged.
  - confidence: 0.91
