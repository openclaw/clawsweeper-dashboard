## Summary

Keep open: the underlying 5s orphan-grace problem is source-reproducible, but this PR is not mergeable as written because it replaces the fs-safe-backed lock manager, drops false-live owner cleanup, and still lacks patched-build proof of the changed behavior.

## What This Changes

The PR raises `ORPHAN_LOCK_PAYLOAD_GRACE_MS` from 5s to 30s, replaces the fs-safe-backed session lock manager with local file-handle state, updates tests, and adds a changelog entry.

## Best Possible Solution

Preserve the current fs-safe-backed lock manager and owner cleanup behavior, then land a minimal grace/comment/test change only after patched-build real behavior proof.

## Reproduction Assessment

Yes, from source: current main treats an empty lock payload as missing pid/invalid-createdAt and reclaims it by mtime after the 5s grace when `staleMs` is larger. I did not run tests because this review is read-only.

## Solution Assessment

No for this PR as written. Raising the grace can be a narrow maintainable fix, but the patch should not replace `createFileLockManager`, remove false-live owner cleanup, or bypass snapshot-safe release semantics.
