## Summary

Keep open: the underlying acquire-path bug is real and the proposed constant change helps, but the patch appears incomplete for startup/doctor cleanup and the supplied logs do not prove after-fix behavior from a patched build.

## What This Changes

The PR raises the session write-lock payload-less orphan grace from 5 seconds to 30 seconds, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Extend the payload-less mtime grace consistently across contended acquire and stale cleanup/doctor paths, then merge after contributor-supplied patched-run proof and focused validation.

## Reproduction Assessment

Yes from source: current main creates an empty sidecar lock before payload write, fs-safe passes a null payload to `shouldReclaim` on contention, and OpenClaw reclaims by mtime after 5 seconds when `staleMs` is larger. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted: the acquire-path constant/test change is narrow, but cleanup still removes payload-less `.jsonl.lock` files immediately, and the PR lacks patched-build proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Apply the orphan grace to cleanup too:** `src/agents/session-write-lock.ts:52-58`
  - body: Raising this constant only affects contended acquisition. `cleanStaleLockFiles()` still treats an empty `.jsonl.lock` as `missing-pid`/`invalid-createdAt` and removes it immediately during gateway startup cleanup or `openclaw doctor --fix`, so a writer paused between `open("wx")` and metadata write can still lose its transcript lock through that path. Reuse the same mtime grace before marking/removing payload-less locks in cleanup.
  - confidence: 0.84
