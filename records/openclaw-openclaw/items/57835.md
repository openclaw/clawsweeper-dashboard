## Summary

Keep open. Current main still lacks the requested top-level gateway agent-name fallback in both helper paths, while this PR applies the narrow `agents.list[].name` then `agents.list[].identity.name` precedence and adds focused regression coverage. ## What This Changes

The PR updates gateway agent list/session summary helpers to populate top-level agent `name` from `agents.list[].name` or fall back to `agents.list[].identity.name`, with tests for fallback, explicit-name precedence, and missing-name cases.

## Best Possible Solution

Land a refreshed version of this PR after adding the required changelog entry and rerunning the targeted gateway tests, unless a maintainer explicitly decides that top-level gateway agent `name` should remain config-only and consumers should always read nested `identity.name`.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: pass a config whose `agents.list[]` entry has `identity.name` but no `name`; both gateway helpers currently leave top-level `name` unset, and the PR's tests model that exact case.

## Solution Assessment

Yes, with a changelog addition. The proposed `configuredName ?? identity.name` fallback is the narrowest maintainable change because it preserves explicit configured names while aligning both gateway summary helpers with the nested identity data already present in the response.
