## Summary

Keep open: current main still has a credible per-run embedded auth preparation path for explicit API-key providers, while the existing model discovery cache only addresses part of the reported startup cost. The report is recent, specific, and not covered by a duplicate or shipped implementation based on the available context and source review.

## What This Changes

The issue asks OpenClaw to reduce embedded Pi per-run auth/model preparation latency, especially by fast-pathing explicitly configured inline API-key providers before channel dispatch.

## Best Possible Solution

Add a narrow embedded-run auth fast path or cache for explicit config API keys only when no provider runtime auth owner must participate, preserving OAuth, token, AWS SDK, plugin-owned transport, profile rotation, and runtime request override behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. The issue provides credible stage traces, and current-main source inspection shows the relevant per-run auth preparation path, but I did not run the reported Windows/QQBot/DashScope setup.

## Solution Assessment

Yes, the suggested first step is the narrow maintainable direction if guarded by provider-runtime ownership checks. Broader process-level model/auth caching can follow, but the first fix should avoid changing config schema or plugin auth semantics.
