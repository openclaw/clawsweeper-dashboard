## Summary

Keep open: current main still has a credible per-run embedded auth preparation path for explicit API-key providers, and the existing model discovery cache only addresses part of the reported startup cost. The issue is recent, source-backed, unpaired with an open fix PR, and no canonical duplicate was found in targeted searches.

## What This Changes

The issue requests reducing embedded Pi per-run model/auth preparation latency, especially for explicit inline API-key providers before gateway/channel dispatch.

## Best Possible Solution

Add a narrow embedded-run auth fast path or cache for explicit configured API keys only when no provider runtime-auth owner needs to participate, while preserving OAuth, token, AWS SDK, plugin-owned transport, profile failover, and runtime request override semantics.

## Reproduction Assessment

No high-confidence live reproduction was established. The issue provides credible startup-stage traces, and current-main source inspection shows the relevant per-run model/auth preparation path, but the reported Windows/QQBot/DashScope setup was not run here.

## Solution Assessment

Yes, the suggested first step is the narrow maintainable direction if it is guarded by provider-runtime ownership checks. Broader process-level model/auth caching can follow later, but the first fix should avoid config schema changes and plugin auth semantic changes.
