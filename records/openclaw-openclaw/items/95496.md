## Summary

Keep open: the PR targets the right fallback-classifier surface, but the new generic-failure classification still runs after the GPT-5-only guard, so the linked `claude-cli/claude-sonnet-4-6` case can still return `null` instead of advancing the fallback chain. The PR also still has only unit-test/build/CI proof, not redacted runtime proof of an exhausted Claude CLI primary advancing to a configured fallback.

## What This Changes

The PR imports the generic external-run failure matcher, bypasses visible-payload detection when normal text payloads all match it, and adds classifier regression tests for generic-failure payloads.

## Best Possible Solution

Move exact generic external-run failure classification before the GPT-5-only guard or shape the CLI output as fallback-safe earlier, add a regression using `claude-cli/claude-sonnet-4-6`, and provide redacted runtime proof of fallback advancing.

## Reproduction Assessment

Yes, at source level: current main wraps CLI output text as a normal visible payload and the classifier returns `null` before empty-result classification for non-GPT models. I did not run a live exhausted Claude Code account.

## Solution Assessment

No. The PR patches a plausible surface, but the new branch is below the non-GPT guard, so the reported model is still missed; the narrow maintainable fix is to classify the exact generic failure before that guard or shape the CLI result earlier.
