## Summary

Keep this PR open, but do not merge the current branch: the patch still misses the production WhatsApp media path, can corrupt inbound dedupe on split failures, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a WhatsApp inbound `onFlush` branch that sends media-bearing debounce batches entry-by-entry before text coalescing.

## Best Possible Solution

Implement a WhatsApp-owned media burst path that preserves all rapid forwarded images as `MediaPaths`/`MediaTypes`, or explicitly extend active-run steering to carry media, while keeping text-only debounce behavior unchanged.

## Reproduction Assessment

Yes at source level: current main routes WhatsApp media around debounce, and active-run steering carries only text, matching the linked rapid-image loss report. I did not run a live WhatsApp forwarded-image burst in this read-only review.

## Solution Assessment

No. The PR changes a multi-entry debounce branch that normal WhatsApp media does not reach, and split dispatch needs per-entry dedupe or an all-or-nothing wrapper before it is safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276`
  - body: This branch only runs for multi-entry debounce batches, but the production WhatsApp monitor returns `false` from `shouldDebounce` whenever `mediaPath` or `mediaType` is present. Rapid image forwards therefore still flush as single media entries and can still hit text-only steering instead of this new branch.
  - confidence: 0.93
- **[P2] Finalize split media entries individually:** `extensions/whatsapp/src/inbound/monitor.ts:278-280`
  - body: If one `onMessage(entry)` succeeds and a later split entry throws, the outer catch finalizes the whole `entries` array. Retryable failures can redeliver already-sent images, while non-retryable failures can commit keys for images that were never sent.
  - confidence: 0.91
