## Summary

Keep open: the PR targets a credible WhatsApp media-loss bug, but the proposed debounce flush branch misses the normal production media path and can duplicate or drop messages on partial failure.

## What This Changes

The PR adds a WhatsApp inbound debounce flush branch that dispatches each media-bearing batch entry separately before text coalescing.

## Best Possible Solution

Preserve every forwarded WhatsApp image in the actual runtime path, with per-delivered-item dedupe semantics and redacted live proof that all images reach the agent.

## Reproduction Assessment

Yes at source level, but not live: current main saves WhatsApp media, bypasses debounce for media, and active-run steering carries only text. The linked report includes gateway logs for the lost image, but I did not run a real WhatsApp forward-burst scenario.

## Solution Assessment

No. The proposed branch only handles multi-entry debounce batches, while production media messages bypass that path, and the split loop needs per-entry dedupe handling to avoid duplicate or dropped messages after partial failure.

## Review Findings

Rapid forwarded images still flush as single-entry media messages and can still hit active-run steering without attachment context.
  - confidence: 0.93

  - body: This loop can deliver an earlier entry and then throw on a later one while the outer catch finalizes the original batch. Retryable failures can release already-delivered keys for duplicate retry, and non-retryable failures can commit keys for entries that never reached onMessage.
  - confidence: 0.91
