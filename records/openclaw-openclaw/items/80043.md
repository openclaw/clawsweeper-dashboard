## Summary

Keep open: this is a plausible WhatsApp media-loss bug fix attempt, but the patch does not cover the production path where WhatsApp media bypasses debounce, and it introduces unsafe split-delivery dedupe semantics.

## What This Changes

The PR adds a media-bearing branch in the WhatsApp inbound debounce flush that dispatches each batch entry separately before finalizing dedupe.

## Best Possible Solution

Preserve every rapid WhatsApp image in a media-aware inbound or steering path, with per-delivered-item dedupe semantics and redacted real WhatsApp proof that all images reach the agent.

## Reproduction Assessment

Yes, at source level: current main saves WhatsApp media, then the production monitor refuses to debounce media and steer queues only followup text. I did not run a live WhatsApp forwarded-image burst, so this is not a full real-environment reproduction.

## Solution Assessment

No. The PR handles only media entries that already made it into a multi-entry debounce flush, while production WhatsApp media bypasses that flush, and the per-entry dispatch loop leaves dedupe finalization all-or-nothing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276`
  - body: The new media branch only runs after entries have already formed a multi-entry debounce batch, but production WhatsApp returns false from shouldDebounce for mediaPath/mediaType and the shared debouncer flushes those messages singly. Rapid forwarded images can still miss this branch and then hit text-only steer behavior without attachment context.
  - confidence: 0.94
- **[P2] Finalize split media entries per delivered item:** `extensions/whatsapp/src/inbound/monitor.ts:278-281`
  - body: This loop can deliver an earlier entry and then throw on a later one, while the outer catch still finalizes the original entries array as one unit. Retryable failures can re-release already delivered keys, and non-retryable failures can commit keys for entries that never reached onMessage.
  - confidence: 0.92
