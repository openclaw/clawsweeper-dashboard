## Summary

Keep open: the PR targets a real WhatsApp media-loss report, but the latest diff still changes a debounce batch path that normal WhatsApp media does not reach, retains the split-delivery dedupe risk, and lacks required after-fix real behavior proof.

## What This Changes

The PR changes WhatsApp inbound flush handling to send any media-bearing debounce batch entry-by-entry instead of merging the batch into one text-coalesced message.

## Best Possible Solution

Implement a WhatsApp-owned media burst or media-capable steering path that preserves all rapid forwarded images as `MediaPaths`/`MediaTypes`, handles dedupe per delivered unit, keeps text-only debounce behavior unchanged, and includes live redacted WhatsApp proof.

## Reproduction Assessment

Yes at source level: current main saves WhatsApp media, then the media debounce predicate forces single-item flushes, and active-run steering passes only text. I did not run a live WhatsApp forwarded-image burst in this read-only review.

## Solution Assessment

No. The PR changes a multi-entry debounce branch that ordinary WhatsApp media bypasses, and the split dispatch can finalize dedupe incorrectly after partial delivery; the fix should target the actual media burst or steering path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276`
  - body: This branch only runs after a multi-entry debounce batch exists, but the production WhatsApp monitor returns `false` from `shouldDebounce` for messages with `mediaPath` or `mediaType`. Rapid image forwards therefore still flush as single media entries and can still hit the text-only steering path instead of this new branch.
  - confidence: 0.93
- **[P2] Finalize split media entries individually:** `extensions/whatsapp/src/inbound/monitor.ts:278-280`
  - body: If one `onMessage(entry)` succeeds and a later split entry throws, the outer catch still finalizes the whole `entries` array. Retryable failures can redeliver already-sent images, while non-retryable failures can commit keys for images that were never sent.
  - confidence: 0.91
