## Summary

Keep open: this PR changes a debounced multi-entry media batch path, but current WhatsApp media is excluded from debounce before that branch, and the new split loop has unsafe batch-level dedupe behavior if delivery partially fails.

## What This Changes

The PR adds a WhatsApp inbound debounce branch that sends each debounced entry with a mediaPath separately before text coalescing.

## Best Possible Solution

Implement a WhatsApp-owned rapid media burst path or explicit media-aware steering that preserves every attachment, keeps text debounce behavior intact, and finalizes dedupe per delivered message.

## Reproduction Assessment

Yes, at source level: current main excludes WhatsApp media from debounce, flushes those entries singly, and active steering queues only text. I did not run a live WhatsApp forward-burst scenario.

## Solution Assessment

No. The proposed fix handles multi-entry debounced media batches that production WhatsApp media normally never enters, and the new split loop can corrupt dedupe state after partial delivery.
