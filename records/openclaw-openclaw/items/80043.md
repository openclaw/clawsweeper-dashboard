## Summary

Keep open: the linked WhatsApp media-loss bug is plausible, but this PR still targets a debounce branch that normal WhatsApp media bypasses and introduces a partial-delivery dedupe hazard; it also lacks after-fix proof from a real WhatsApp setup.

## What This Changes

The PR adds a media-detection branch to WhatsApp inbound debounce flushing so media-bearing batches dispatch each entry individually instead of building one coalesced message.

## Best Possible Solution

Preserve every WhatsApp forwarded image in the actual media path, either by coalescing related media before dispatch or by carrying media attachments through active-run steering, while finalizing dedupe per delivered unit and keeping text-only debounce unchanged.

## Reproduction Assessment

Yes at source level, but not live: current main saves WhatsApp media, opts media messages out of text debounce, and active-run steering queues only prompt text. The PR still needs a redacted real WhatsApp forwarded-image burst showing the after-fix result.

## Solution Assessment

No. The proposed branch does not cover the production media path described by the linked issue and needs safer dedupe handling if any split delivery remains.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276`
  - body: The new `hasMedia` branch only runs after a multi-entry debounce batch exists, but production WhatsApp returns false from `shouldDebounce` for messages with `mediaPath` or `mediaType`. Rapid forwarded images still flush as single-entry media messages and can still hit the active-run steering path without attachment context.
  - confidence: 0.93
- **[P2] Finalize split media entries per delivered item:** `extensions/whatsapp/src/inbound/monitor.ts:278-281`
  - body: This loop can deliver an earlier entry and then throw on a later one while the outer catch finalizes the original batch. Retryable failures can release already-delivered keys for duplicate retry, and non-retryable failures can commit keys for entries that never reached `onMessage`.
  - confidence: 0.91
