## Summary

Keep open: the linked WhatsApp multi-image loss is credible, but this PR does not cover the production media path and adds a split-delivery branch with unsafe batch-level dedupe handling. It also still needs after-fix proof from a real WhatsApp setup before merge.

## What This Changes

The PR changes WhatsApp inbound debounce flushing to send each debounced media entry separately instead of coalescing the batch into one message.

## Best Possible Solution

Implement a WhatsApp-owned media burst or media-aware steering path that preserves every forwarded image, keeps text-only debounce behavior intact, and finalizes dedupe per delivered message or with clear all-or-nothing semantics.

## Reproduction Assessment

Yes, at source level: current main routes WhatsApp media messages around inbound debounce and active steering queues only text, matching the linked report's rapid-image loss path. I did not run a live WhatsApp forward-burst scenario in this read-only review.

## Solution Assessment

No. The proposed change handles multi-entry debounced media batches, but production WhatsApp media normally flushes as single entries, and the new split loop can corrupt dedupe state after a partial failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276-277`
  - body: This branch is inside multi-entry debounce flushing, but the production WhatsApp monitor returns `false` from `shouldDebounce` for any `mediaPath` or `mediaType`, so media messages are flushed as single entries before this code can run. The reported rapid-image flow with `queue.mode: "steer"` still sends separate image turns and can still lose the later attachment; the fix needs to cover the real media path, such as WhatsApp media-burst coalescing before debounce filtering or media-aware steering.
  - confidence: 0.93
- **[P2] Finalize split media entries individually:** `extensions/whatsapp/src/inbound/monitor.ts:279`
  - body: After the loop partially succeeds, any later thrown error is handled by the outer catch with `finalizeInboundDedupe(entries, error)`. A retryable error releases every dedupe key and can redeliver already-sent images; a non-retryable error commits keys for entries that were never sent. Finalize each delivered entry separately, or make the split dispatch all-or-nothing.
  - confidence: 0.91
