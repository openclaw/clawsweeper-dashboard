## Summary

Keep open: the linked WhatsApp media-loss problem looks real, but this PR still misses the production path and introduces batch-dedupe edge cases; it also lacks after-fix real behavior proof.

## What This Changes

The PR changes WhatsApp inbound debounce flushing to send any media-bearing batch entries individually instead of merging them into one coalesced message.

## Best Possible Solution

Target the actual WhatsApp media burst or active-run media steering path so all forwarded images are preserved, dedupe is finalized per delivered unit, text-only debounce remains unchanged, and live redacted WhatsApp proof is attached before merge.

## Reproduction Assessment

Yes at source level: current main saves WhatsApp media, opts media messages out of text debounce, and then active-run steering passes only prompt text. I did not establish a live WhatsApp forwarded-image burst reproduction in this read-only review.

## Solution Assessment

No. The PR changes a multi-entry debounce branch that production WhatsApp media normally bypasses, and the split dispatch can finalize dedupe incorrectly after partial delivery; the safer fix should target the actual media burst or steering path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276`
  - body: The new `hasMedia` branch only runs after a multi-entry debounce batch exists, but production WhatsApp passes `shouldDebounce` that returns false for messages with `mediaPath` or `mediaType`. Rapid forwarded images still flush as single media entries and can still hit the existing active-run text-only steering path.
  - confidence: 0.93
- **[P2] Finalize split media entries per delivered item:** `extensions/whatsapp/src/inbound/monitor.ts:278-280`
  - body: This loop can successfully deliver an earlier entry and then throw on a later one, while the outer catch still finalizes the whole batch. Retryable failures can duplicate already-sent entries, and non-retryable failures can commit keys for entries that never reached `onMessage`.
  - confidence: 0.91
