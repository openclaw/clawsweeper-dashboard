## Summary

Keep open: the underlying WhatsApp image-loss bug is credible, but this patch does not cover the production path where WhatsApp media bypasses debounce and it leaves split-delivery dedupe all-or-nothing; the PR also lacks real WhatsApp after-fix proof.

## What This Changes

The PR adds a media-bearing branch to `extensions/whatsapp/src/inbound/monitor.ts` so a debounced batch containing media dispatches each entry separately instead of coalescing the batch.

## Best Possible Solution

Preserve every rapid WhatsApp image through a WhatsApp-owned media-burst path or an explicit media-capable steering contract, with per-delivered-item dedupe semantics and redacted real WhatsApp proof.

## Reproduction Assessment

Yes, at source level: rapid WhatsApp image messages can be traced through media debounce rejection, single-item flushes, and text-only active-run steering. I did not run a live forwarded-image burst, so this is not a full real-environment reproduction.

## Solution Assessment

No. The patch handles only media entries already inside a multi-entry debounce flush, while current production WhatsApp media normally skips that flush, and the split loop lacks per-entry dedupe/failure handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Cover the production media path:** `extensions/whatsapp/src/inbound/monitor.ts:276`
  - body: The new media branch only runs after entries have formed a multi-entry debounce batch, but production WhatsApp returns false from `shouldDebounce` for `mediaPath`/`mediaType` and the shared debouncer flushes those items singly. Rapid forwarded images can still miss this branch and then reach active steering without structured attachment context.
  - confidence: 0.94
- **[P2] Finalize split media entries per delivered item:** `extensions/whatsapp/src/inbound/monitor.ts:278-281`
  - body: This loop can deliver an earlier entry and then throw on a later one while the outer catch still finalizes the original entries array as one unit. Retryable failures can re-release already delivered keys, and non-retryable failures can commit keys for entries that never reached `onMessage`.
  - confidence: 0.92
