## Summary

Keep open: the docs repair is useful and targets the linked Notion allowlist bug, but PR head still documents a copied helper that can attach the Notion token to a malformed non-Notion URL.

## What This Changes

The branch updates the bundled Notion skill docs and changelog to replace shell-wrapped curl examples with `NOTION_API_KEY` setup and a file-backed `python3` helper flow.

## Best Possible Solution

Add a small guard that rejects helper paths outside `/v1/...`, keep the change docs-only, then land this PR as the active fix for https://github.com/openclaw/openclaw/issues/59235.

## Reproduction Assessment

Yes. Source inspection on current main shows the allowlist-hostile `NOTION_KEY=$(cat ...)` plus multiline `curl` examples, and the exec allowlist docs/tests explain why those shell constructs miss strict allowlists.

## Solution Assessment

No, not as written. The docs-only direction is the narrow maintainable fix, but the runnable helper should validate a `/v1/...` API path before constructing the request and attaching the bearer token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject non-Notion helper paths:** `skills/notion/SKILL.md:71`
  - body: The helper appends arbitrary `api_path` text to `https://api.notion.com` and then adds the bearer token. Inputs like `.evil.example/v1/search` or `@evil.example/v1/search` can change the destination host, so reject anything that does not start with `/v1/` before constructing the `Request`.
  - confidence: 0.94
