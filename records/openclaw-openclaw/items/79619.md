## Summary

Keep open: this PR is a focused docs repair for a real current-main Notion skill problem, but the new copied helper still needs a small path validation fix before merge because it can attach the Notion token to a malformed host.

## What This Changes

The PR updates `skills/notion/SKILL.md` and `CHANGELOG.md` to replace shell-local multiline `curl` examples with `NOTION_API_KEY` setup guidance and a file-backed `python3 notion_request.py` helper flow.

## Best Possible Solution

Land the docs repair after adding a small `/v1/...` path guard to the helper snippet, without changing runtime exec policy or Notion API behavior.

## Reproduction Assessment

Yes, source-level. Current main still contains the `NOTION_KEY=$(cat ...)` setup and multiline `curl` examples, while current exec docs/tests show those shell constructs are rejected in strict allowlist paths before a Notion API request is made.

## Solution Assessment

No, not as written. The docs-only direction is the narrowest maintainable fix, but the helper should reject paths that do not start with `/v1/` before constructing the request and attaching the token.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Validate the helper path before sending the token:** `skills/notion/SKILL.md:71`
  - body: The helper appends arbitrary `api_path` text to `https://api.notion.com` and then adds the bearer token. If the copied helper is called with a malformed value such as `.evil.example/v1/search`, Python targets `api.notion.com.evil.example` while still sending `NOTION_API_KEY`; reject paths that do not start with `/v1/` before constructing the request.
  - confidence: 0.88

## Security Review
