## Summary

Keep open. The reporter’s follow-up removes the earlier config uncertainty, and current main does not show a trajectory-sidecar cap or a shipped fix for post-`stream-ready` turn loss when the sidecar grows while the primary transcript remains below the configured byte thresholds.

## What This Changes

The issue reports WhatsApp turns silently wedging after `stream-ready` when the runtime trajectory sidecar grows past about 3 MB despite configured primary transcript compaction and memory-flush byte thresholds.

## Best Possible Solution

Treat this as active session reliability work: confirm the failing step from reporter logs/artifacts, then either add a bounded/configurable active trajectory-sidecar policy plus recovery for stuck turns, or fold the evidence into a canonical transcript/trajectory growth issue.

## Reproduction Assessment

Unclear. The reporter provides concrete config, file sizes, model/channel, workaround, and two observed incidents, but this review only verified static source paths and did not reproduce a live WhatsApp turn stalling after `stream-ready`.

## Solution Assessment

Unclear. The existing `maxActiveTranscriptBytes` and `forceFlushTranscriptBytes` paths are the right primary-transcript mechanisms, but they are not sufficient evidence for a trajectory-sidecar-triggered wedge; the safer direction is trajectory/run-lifecycle triage before patching.
