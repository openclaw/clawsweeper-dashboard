## Summary

Keep open. Current main implements byte guards for the primary session transcript, but the reporter corrected the config and showed the failing sessions had sub-2 MB primary JSONL files while the separate trajectory sidecars were 3.1-4.3 MB, and the current trajectory recorder still has only a separate 50 MiB writer cap plus a 10s cleanup flush timeout.

## What This Changes

Investigate WhatsApp turns that silently die after `stream-ready` when the trajectory sidecar grows past roughly 3 MB, and clarify or enforce the intended byte-cap behavior for trajectory files versus primary session transcripts.

## Best Possible Solution

Keep this issue open until maintainer triage uses the reporter's sanitized trajectory files and journal logs to decide whether OpenClaw needs a trajectory-sidecar cap, faster bounded trajectory replay/export handling, or stuck-turn recovery tied to the embedded runner lifecycle.

## Reproduction Assessment

Unclear. The reporter gives concrete session sizes, config, model/channel, and a recovery workaround, but this review did not reproduce a live WhatsApp turn stalling after `stream-ready` and the needed trajectory files/logs are not in the repo.

## Solution Assessment

Unclear. The existing primary-transcript compaction path is not sufficient for the corrected trajectory-sidecar report; the safer path is to inspect the sidecar read/write/flush lifecycle before adding a cap or recovery policy.
