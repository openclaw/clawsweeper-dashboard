## Summary

Keep open. Current main has active-transcript byte compaction and transcript-size memory flush logic, but this report is about silent post-`stream-ready` turn loss correlated with `trajectory.jsonl` growth plus a config shape that would not enable the active-transcript cap, so the issue is not fixed or disproven.

## What This Changes

Investigate whether long-lived WhatsApp sessions can silently lose turns after `stream-ready` when large trajectory sidecars accumulate and byte-based compaction settings are not taking effect.

## Best Possible Solution

Confirm the accepted config, primary session JSONL size, trajectory sidecar size, and verbose `memoryFlush check` / `preflightCompaction check` lines from an affected run, then fix the specific layer that still allows the turn to stall.

## Reproduction Assessment

Unclear. The report gives concrete production observations, but current source checks the primary session transcript while the reproduction threshold is based on `trajectory.jsonl`, and the pasted config does not match the documented active-byte-cap shape.

## Solution Assessment
