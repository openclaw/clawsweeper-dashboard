## Summary

Keep open: current main and the latest release still have the unbounded vec0 widen fallback, and this PR is a narrow source-backed fix for sqlite-vec's 4096 KNN ceiling. The patch looks correct, but it should not be closed as cleanup; before merge the external PR still needs concrete real-behavior proof and preferably a focused overflow regression test.

## What This Changes

The PR adds a `MAX_VECTOR_KNN_K = 4096` constant and clamps memory-core's vec0 widen fallback so it no longer calls sqlite-vec KNN with the full vector-table row count.

## Best Possible Solution

Close this PR as superseded by https://github.com/openclaw/openclaw/pull/83758.

## Reproduction Assessment

Yes from source inspection: current main widens from `limit * 8` to the full vector table count when filtered rows are short, while sqlite-vec v0.1.9 rejects KNN `k` above 4096. I did not run a live oversized-index reproduction in this read-only review.

## Solution Assessment

Yes, the clamp is the narrowest maintainable fix for the dependency ceiling because the normal first pass is already capped below 4096 by the manager's candidate limit; a regression test would improve confidence but does not change the fix shape.
