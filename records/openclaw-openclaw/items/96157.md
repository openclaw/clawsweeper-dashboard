## Summary

Keep open: current main and v2026.6.10 still have the unbounded sqlite-vec widen query, and this PR is a narrow source-backed fix with focused regression tests. The earlier supersession note does not hold after review because https://github.com/openclaw/openclaw/pull/83758 fixed the fallback scan/yielding path, not sqlite-vec's k > 4096 failure; the remaining merge blocker is concrete real-behavior proof from an oversized index.

## What This Changes

The PR clamps memory-core's sqlite-vec widen-fallback KNN `k` value to 4096 and adds regression tests for above- and below-ceiling widen queries.

## Best Possible Solution

Land the narrow clamp after the contributor adds redacted live or terminal proof that an oversized sqlite-vec-backed `memory_search` no longer throws or times out, while keeping the new regression tests.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.10 widen sqlite-vec KNN from `limit * 8` to the full vector table count, while sqlite-vec v0.1.9 rejects k values above 4096. I did not run a live >4096-vector store reproduction in this read-only review.

## Solution Assessment

Yes, the code shape is the narrowest maintainable fix for the dependency ceiling because it preserves the existing fast path and only bounds the invalid k value. A broader full-scan fallback could improve sparse-filter recall later, but it is not required to stop the current sqlite-vec error.
