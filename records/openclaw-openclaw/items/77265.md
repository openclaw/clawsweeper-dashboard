## Summary

Keep open: current main has a source-backed path where Gateway-backed CLI agent delivery can be downgraded to best-effort for main-session runs, so a delivery failure can still return an ok agent result with media payloads.

## What This Changes

The issue reports a Windows Telegram direct-chat case where `openclaw agent --deliver --json` returns `mediaUrl`/`mediaUrls` in a successful result but the media is not sent.

## Best Possible Solution

Make explicit CLI `agent --deliver` Gateway requests strict by default while preserving best-effort behavior for chat/UI main-session RPCs, then add regression coverage for surfaced delivery failures.

## Reproduction Assessment

Yes, from source inspection for the silent-success failure mode: a Gateway-backed main-session `openclaw agent --deliver --json` request without an explicit `bestEffortDeliver` override is converted to best-effort and delivery errors are swallowed. I did not run the reporter's live Telegram account path.

## Solution Assessment

No, the current behavior is not the best way to solve this CLI issue because the public command promises delivery but does not expose or set strict delivery intent. The safer fix is to make explicit CLI delivery fail closed without changing Gateway best-effort defaults for UI/chat callers.
