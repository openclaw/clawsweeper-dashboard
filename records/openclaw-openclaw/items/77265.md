## Summary

Keep open: current main still lets explicit CLI Gateway delivery inherit main-session best-effort behavior, so a failed outbound media send can be logged and returned as an ok agent result instead of failing the command.

## What This Changes

The issue asks for `openclaw agent --deliver` to deliver final `mediaUrl`/`mediaUrls` payloads to Telegram or return an explicit delivery error.

## Best Possible Solution

Make explicit CLI `agent --deliver` Gateway requests strict by default while preserving intentional best-effort behavior for chat/UI and internal callers.

## Reproduction Assessment

Yes, source-reproducible: a Gateway-backed main-session CLI `--deliver --json` request leaves `bestEffortDeliver` undefined, the Gateway flips it to true, and outbound send errors are swallowed. I did not run a live Telegram delivery.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because a public CLI delivery command should fail closed on delivery failure. The narrow fix is to pass strict delivery intent for explicit CLI `--deliver` while leaving best-effort RPC paths opt-in.
