## Summary

Keep open: current main still has a source-backed path where explicit CLI `agent --deliver` Gateway runs can inherit main-session best-effort delivery, so outbound media send failures can be swallowed while the agent result stays `ok`.

## What This Changes

The issue reports that `openclaw agent --deliver` can return an ok payload containing Telegram media URLs without actually sending the media or surfacing a delivery error.

## Best Possible Solution

Make explicit CLI Gateway `agent --deliver` requests strict by default while preserving best-effort delivery for UI/chat main-session callers, then add regression coverage for surfaced delivery failures.

## Reproduction Assessment

Yes. The source path is high-confidence: CLI `--deliver` leaves `bestEffortDeliver` undefined, the Gateway turns that into best-effort for main sessions, and outbound media delivery then swallows send errors.

## Solution Assessment

No, the current behavior is not the best way to solve this CLI issue. The narrow maintainable fix is to make explicit CLI delivery fail closed while keeping chat/UI best-effort fallback semantics intact.
