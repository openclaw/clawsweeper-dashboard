## Summary

Keep open: current main still has a source-backed path where public `agent --deliver` Gateway runs can use best-effort delivery and return an `ok` agent result after outbound media send errors are swallowed.

## What This Changes

The issue reports that `openclaw agent --deliver` can return a successful Telegram media payload without actually delivering the media or surfacing a delivery error.

## Best Possible Solution

Make explicit CLI `agent --deliver` Gateway requests strict by default while preserving best-effort behavior for chat/UI main-session RPCs, then add mocked regression coverage for surfaced delivery failures.

## Reproduction Assessment

Yes. Source inspection shows a current-main reproduction path for the silent-success class: public CLI leaves `bestEffortDeliver` undefined, Gateway main sessions default it to true, and outbound media send errors are swallowed under best-effort delivery.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because a command that explicitly requests delivery should fail closed or return a clear delivery error. The narrow fix is to send strict delivery intent from the CLI Gateway path without changing best-effort defaults for UI/chat callers.
