## Summary

Keep open: the report matches a real strict-delivery gap where media-bearing `agent --deliver` payloads can still be returned as an ok result if outbound delivery produces no send result.

## What This Changes

The issue asks `openclaw agent --deliver` to deliver `mediaUrl`/`mediaUrls` payloads to Telegram or return an explicit delivery error.

## Best Possible Solution

Make strict agent delivery fail when sendable media payloads produce no outbound delivery result, while preserving explicit `bestEffortDeliver` behavior.

## Reproduction Assessment

Yes, at source level: a focused test can make `deliverOutboundPayloads` return `[]` for a media-bearing payload and observe `deliverAgentCommandResult` still returning ok payloads. I did not establish a live Telegram reproduction.

## Solution Assessment

Yes, the narrowest maintainable fix is generic agent delivery result accounting, not a Telegram-specific workaround.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77265. For `openclaw agent --deliver`, when a sendable final payload contains text/media and strict delivery is requested, do not return an ok result if outbound delivery produces no send result. Keep the fix generic in `src/agents/command/delivery.ts`; do not add Telegram-specific policy unless a Telegram adapter bug is proven. Preserve `bestEffortDeliver` semantics. Add regression coverage for a media-bearing payload where `deliverOutboundPayloads` resolves to `[]`, and cover that normal media delivery still passes media URLs through. Add a user-facing CHANGELOG entry because this changes CLI delivery error behavior.
