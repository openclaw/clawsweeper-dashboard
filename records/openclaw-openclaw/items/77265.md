## Summary

Keep open: current main still has a source-backed path where explicit CLI `agent --deliver` requests sent through the Gateway can inherit main-session best-effort delivery, so outbound media send failures can be logged/swallowed while the agent result remains ok with `mediaUrl` payloads.

## What This Changes

The issue asks for `openclaw agent --deliver` to either deliver final `mediaUrl`/`mediaUrls` payloads to Telegram when reply routing is provided or return an explicit delivery error.

## Best Possible Solution

Make explicit CLI `agent --deliver` Gateway requests strict by default while preserving best-effort behavior for chat/UI main-session RPC callers, then add regression coverage for surfaced delivery failures.

## Reproduction Assessment

Yes, from source inspection: a Gateway-backed main-session `openclaw agent --deliver --json` request with no explicit `bestEffortDeliver` value is converted to best-effort, and outbound delivery errors are swallowed. I did not run the reporter's live Telegram account path.

## Solution Assessment

No, the current behavior is not the best way to solve the CLI contract because an explicit delivery command can report success after failed media delivery. The narrow fix is to mark CLI-initiated `--deliver` requests as strict while leaving UI/chat main-session defaults alone.
