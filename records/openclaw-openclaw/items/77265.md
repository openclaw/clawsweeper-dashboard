## Summary

Keep open: current main still has a source-backed path where Gateway-backed CLI `agent --deliver` can return an ok result with media payloads while delivery failures are treated as best-effort and swallowed.

## What This Changes

Make `openclaw agent --deliver` deliver returned Telegram media payloads or report an explicit delivery error instead of silent success.

## Best Possible Solution

Make explicit CLI Gateway `--deliver` requests strict by default while preserving best-effort behavior for UI/chat or other callers that opt into it, then add mocked media-delivery regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a normal Gateway-backed `openclaw agent --deliver --json` main-session request leaves `bestEffortDeliver` undefined, the Gateway turns it on, and outbound send errors no longer fail the command. I did not run a live Telegram account reproduction.

## Solution Assessment

No, the current behavior is not the best way to solve this CLI contract because `--deliver` can report success while media delivery failed. The narrow fix is to pass strict delivery intent for explicit CLI delivery without changing Gateway best-effort defaults for chat/UI callers.
