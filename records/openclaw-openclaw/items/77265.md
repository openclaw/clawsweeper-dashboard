## Summary

Keep this issue open. Current `main` and the latest release still exclude ordinary `mediaUrl`/`mediaUrls` payloads from the payload-aware `sendPayload` routing guard, and the linked closing PR remains open with proof/review work still outstanding.

## What This Changes

The issue reports that `openclaw agent --deliver` can return a successful Telegram-bound media payload without delivering the media or surfacing an explicit delivery error.

## Best Possible Solution

Land the linked PR or a narrowed successor with focused regression coverage and redacted live Telegram proof, then close this issue as fixed.

## Reproduction Assessment

Partly: the reporter provided a concrete Telegram command and current source still shows media-only payloads bypassing the payload-aware Telegram `sendPayload` guard. I did not run a live Telegram reproduction, and the generic media fallback path means live proof is still needed for the exact failure boundary.

## Solution Assessment

Yes for the narrow end state: media-only `agent --deliver` payloads should either use the payload-aware Telegram media path or fail explicitly. The broader text-plus-media channel contract belongs with https://github.com/openclaw/openclaw/issues/91867 rather than being decided accidentally here.
