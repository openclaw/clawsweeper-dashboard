## Label Justifications

- `P2`: This is a valid, bounded runtime bug in TTS credential resolution with a clear affected config surface but limited blast radius.
- `impact:security`: The documented workaround is to inline an API key in config, weakening the intended SecretRef credential boundary.
- `impact:auth-provider`: The failure is a provider credential/SecretRef resolution path that prevents the ElevenLabs provider from receiving its configured apiKey.

## Summary

Keep open: current main still resolves TTS SecretRefs only from top-level and agent provider maps, while schema/docs allow persona provider configs that are merged later as raw provider config. A persona-level ElevenLabs `apiKey` SecretRef can therefore remain unresolved and make the provider appear not configured.

## What This Changes

Resolve file-source SecretRefs used as ElevenLabs TTS persona provider `apiKey` values before runtime configuration checks and synthesis.

## Best Possible Solution

Extend the shared TTS SecretRef target discovery and runtime collector to resolve `personas.*.providers.*.apiKey` before provider readiness/synthesis, with regression coverage for file JSON and single-value SecretRefs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a schema-valid persona provider `apiKey` SecretRef is never collected by the TTS command/runtime secret targets and remains a non-string value when ElevenLabs checks configuration. I did not run the live conversion because this review is read-only.

## Solution Assessment

Yes, the best fix is in the shared TTS SecretRef collector/target registry, not an ElevenLabs-specific fallback. That keeps the documented persona provider merge semantics consistent for all speech providers.
