## Summary

## What This Changes

## Best Possible Solution

Keep `memory_search` from starting QMD full-manager lifecycle resources, but make the transient manager lifetime explicit with a close-safe wrapper or restrict the mode to true one-shot CLI paths, then prove a qmd-backed local-json turn exits cleanly.

Yes, source-level. On PR head, `memory_search` requests `purpose: "cli"`, and current manager code treats that purpose as transient/cache-bypassing while the shared tool has no close path; the original 10k Docker/qmd repro was not rerun.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Close transient memory managers after search:** `extensions/memory-core/src/tools.ts:408-412`
  - body: This changes the shared `memory_search` tool to acquire a `purpose: "cli"` manager, but `cli` managers are explicitly transient and bypass the manager cache, while this tool never closes the manager it receives. That means every memory_search call can leave an uncached QMD, QMD-fallback, or builtin manager alive after the tool returns; in particular builtin/local-embedding fallback managers can keep provider or worker resources around, which is exactly the class of process-lifetime bug this PR is trying to fix. Please either keep using the cached default manager where appropriate, or wrap the `cli` acquisition/search/retry in a `try/finally` that closes the transient manager.
  - confidence: 0.92
