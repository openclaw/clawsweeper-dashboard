## Summary

Keep open. Current main still lacks the requested opt-in Responses flag and still only emits `function_call` output for caller-provided client tools, while built-in tool start events exist on the agent event stream but are not forwarded into JSON or SSE output. The issue is paired with open closing PR #75075, so the correct cleanup action is to review that PR rather than close this issue early or queue a duplicate repair branch.

## What This Changes

Add a default-off `/v1/responses` config flag that surfaces built-in tool invocations as Responses `function_call` output items in JSON and SSE responses.

## Best Possible Solution

Review and land PR #75075, or an equivalent narrow implementation, with the behavior default-off and audit-only. The final change should cover config types/schema/generated docs, JSON and SSE output ordering, client-tool dedupe, focused gateway tests, public docs, changelog, and explicit review of argument exposure; tool results, compact summaries, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and the response output will contain assistant text plus only caller-provided `pendingToolCalls`, while built-in `stream: "tool"` events are emitted separately.

## Solution Assessment

Unclear until the paired PR receives normal maintainer review. A default-off in-band `function_call` audit surface is a plausible narrow fix, but maintainers should verify redaction/security behavior, SSE indices, output ordering, and client-tool dedupe before merge.
