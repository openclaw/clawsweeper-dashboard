## Summary

Keep open. Current main still lacks the requested default-off `/v1/responses` flag and still only emits `function_call` output items for caller-provided client tools, while built-in tool starts are available only on the separate agent event stream. The issue is also paired with open closing PR #75075, so the right cleanup path is to review or land that PR rather than close this issue early.

## What This Changes

The issue requests an opt-in `/v1/responses` configuration flag that surfaces built-in agent tool calls as JSON and SSE `function_call` output items for eval, UI trace, and audit/replay use cases.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with a default-off audit-only flag, aligned config type/schema/generated docs, JSON and SSE coverage, client-tool dedupe coverage, public docs, changelog, and focused security review of argument exposure. Tool results, compact trace digests, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run `/v1/responses` with an agent turn that invokes a built-in tool: current main has no config flag and no `evt.stream === "tool"` capture path, so saved output remains assistant-only unless caller-provided `pendingToolCalls` exist.

## Solution Assessment

Unclear until #75075 is reviewed. A default-off Responses-shaped `function_call` audit item is a plausible narrow fix, but maintainers should verify the security and wire-format details before merge.
