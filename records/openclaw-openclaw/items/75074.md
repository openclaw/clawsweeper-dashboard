## Summary

Keep open. Current main still lacks the requested opt-in built-in tool-call exposure path, and the issue is paired with open closing PR #75075, so the right cleanup outcome is to keep this issue open until that PR or an equivalent fix lands. The discussion adds useful eval, UI, and compliance context, and the closed parallel PR #75107 is relevant background but not a shipped fix.

## What This Changes

Add an opt-in `gateway.http.endpoints.responses.exposeBuiltInToolCalls` setting so `/v1/responses` JSON and SSE output can include built-in agent tool calls as Responses-shaped `function_call` items.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with the default-off config/schema/docs surface, JSON and SSE coverage, client-tool collision coverage, changelog entry, and explicit review of argument exposure. Tool results, compact digests, and built-in delegate mode should stay separate follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run `/v1/responses` on current main with an agent turn that invokes a built-in tool: tool-start events are emitted upstream, but the HTTP handler has no config flag or capture branch, so saved output remains assistant-only unless caller-provided client `pendingToolCalls` exist.

## Solution Assessment

Unclear until #75075 is reviewed. A default-off Responses-shaped `function_call` audit item is a plausible narrow fix, but the argument-exposure and streaming-order details still need maintainer review before this issue can be closed.
