## Summary

Keep open. Current main still lacks the requested opt-in `/v1/responses` flag and still only emits `function_call` output items for caller-provided client tools, while built-in tool start events remain on the separate agent event stream. The issue is also paired with open closing PR #75075, so the right cleanup path is to review or land that PR rather than close this issue early.

## What This Changes

Add a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` option so `/v1/responses` can include built-in tool invocations as Responses-shaped `function_call` output items in JSON and SSE responses.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with the flag defaulting off, config type/schema/generated-doc alignment, JSON and SSE ordering coverage, client-tool dedupe coverage, docs, changelog, and a dedicated security review for tool argument exposure. Tool results, compact trace digests, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to run `/v1/responses` with an agent turn that invokes a built-in tool; current main emits built-in `stream: "tool"` start events separately but only converts `pendingToolCalls` from caller-provided tools into `function_call` output items.

## Solution Assessment

Unclear until #75075 is reviewed. A default-off, Responses-shaped `function_call` audit item is a plausible narrow solution, but maintainers should verify redaction/security behavior, output ordering, SSE indices, and client-tool dedupe before merge.
