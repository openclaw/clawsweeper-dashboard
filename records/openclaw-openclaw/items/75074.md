## Summary

Keep open. Current main still lacks the requested opt-in `/v1/responses` flag and still surfaces `function_call` output only for caller-provided client tools, while built-in tool start events are emitted on the separate agent event stream. The issue is also paired with open closing PR #75075, so the correct cleanup path is to review or land that PR rather than close the issue early.

## What This Changes

Request an opt-in `gateway.http.endpoints.responses.exposeBuiltInToolCalls` setting that adds built-in tool invocations to `/v1/responses` JSON and SSE output as Responses-style `function_call` items.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with a default-off audit-only flag, config type/schema/generated docs alignment, JSON and SSE output coverage, client-tool dedupe coverage, public docs, changelog, and explicit security review of argument exposure. Tool results, compact trace digests, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. A high-confidence reproduction path is static and behavioral: run `/v1/responses` with an agent turn that invokes a built-in tool, and current main has no config flag or `tool` event capture branch, so the saved response output contains only assistant text unless caller-provided `pendingToolCalls` exist.

## Solution Assessment

Unclear until the paired PR is reviewed. A default-off Responses-shaped `function_call` audit item is a plausible narrow solution, but maintainers should verify security/redaction behavior, output ordering, SSE indices, and client-tool dedupe before merge.
