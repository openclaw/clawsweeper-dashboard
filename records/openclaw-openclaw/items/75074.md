## Summary

Keep open. Current main still lacks the requested default-off `/v1/responses` flag and still only returns `function_call` output items for caller-provided client tools, while built-in tool starts remain available only on the agent event stream. The issue is paired with open closing PR #75075, and the later comments add useful real-world eval/replay context rather than making this obsolete.

## What This Changes

The issue requests an opt-in `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag that surfaces built-in tool starts as Responses-shaped `function_call` output items in JSON and SSE `/v1/responses` output.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with a default-off audit-only flag, aligned config type/schema/generated docs, JSON and SSE coverage, client-tool dedupe behavior, public docs, changelog, and explicit review of argument exposure. Tool results, compact trace digests, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is to call `/v1/responses` for an agent turn that invokes a built-in tool: current main has no opt-in config key, no `tool` stream branch in the OpenResponses HTTP handler, and only creates `function_call` output from `pendingToolCalls` for caller-provided client tools.

## Solution Assessment

Unclear until #75075 is reviewed. A default-off Responses-shaped audit item is a plausible narrow solution, but maintainers should verify the privacy/security and wire-format details before merge.
