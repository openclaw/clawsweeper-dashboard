## Summary

Keep open. Current main still lacks the requested opt-in Responses flag and has no built-in tool-event capture path; built-in tool starts are emitted on the agent event stream, but `/v1/responses` only turns caller-provided client tools from `pendingToolCalls` into `function_call` output items. The issue is also paired with open closing PR #75075, while the parallel #75107 was closed after its improvements were folded into #75075, so the right cleanup path is review or land that PR rather than close this issue early.

## What This Changes

The issue requests a default-off `/v1/responses` configuration flag that exposes built-in agent tool invocations as `function_call` output items in JSON and SSE responses.

## Best Possible Solution

Review and land #75075, or an equivalent narrow implementation, with a default-off audit-only flag, JSON and SSE coverage, client-tool dedupe coverage, config type/schema/generated-doc alignment, public docs, changelog, and explicit review of built-in tool argument exposure. Tool results, compact trace digests, and built-in delegate mode should remain separate follow-ups.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to run `/v1/responses` for a turn that invokes a built-in tool: current main emits the tool start on the agent event stream but has no config key or handler branch that copies that event into the saved Responses `output`.

## Solution Assessment

Unclear until the paired PR is reviewed. A default-off Responses-shaped `function_call` audit item is a plausible narrow fix, but maintainers should verify redaction, ordering, SSE indices, and client-tool dedupe before merge.
