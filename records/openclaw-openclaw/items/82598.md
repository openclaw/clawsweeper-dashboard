## Summary

Keep open: the PR fixes the central CLI web-search SecretRef path and addressed the earlier provider-override blocker, but the latest target set still resolves the Google model API key for non-Google web-search providers.

## What This Changes

The PR resolves CLI web-search SecretRefs before local execution, adds web-search-specific secret targets, updates SearXNG SecretInput metadata and SecretRef docs, and adds focused CLI/secret-target tests.

## Best Possible Solution

Keep the PR direction, but make command-secret target selection match the actual selected web-search provider, including scoping the Google model fallback to Gemini/Google searches only.

## Reproduction Assessment

Yes. Current main passes raw config into runWebSearch, and the PR body shows the unresolved Exa file SecretRef error before the patch; the remaining blocker is source-reproducible from the unconditional Google target and active model-provider SecretRef classification.

## Solution Assessment

No. The current approach is close, but the target set still needs provider-aware scoping so a non-Google web-search command does not resolve unrelated Google model credentials.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope Google fallback secrets to Gemini searches:** `src/cli/command-secret-targets.ts:38`
  - body: `getWebSearchCommandSecretTargetIds()` now always includes `models.providers.google.apiKey`. Model provider SecretRefs are active whenever that provider is not disabled, so `openclaw infer web search --provider exa ...` can still call `secrets.resolve` or local fallback for a Google model key and fail or fetch that unrelated secret before Exa runs. Please include this fallback only when the selected or auto-detected web-search provider can use Gemini/Google, and cover the explicit non-Google override path.
  - confidence: 0.86
