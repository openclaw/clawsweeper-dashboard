## Summary

Keep open: the fix is well scoped and the supplied terminal proof covers the reported SecretRef failure, but the patch needs one correction before merge because it can corrupt `--json` output when secret resolution emits diagnostics.

## What This Changes

The PR resolves web-search CLI SecretRefs before local web search execution and adds scoped web-search secret-target tests.

## Best Possible Solution

Merge this direction after preserving pure JSON output for `--json` and keeping the SecretRef target set scoped to web-search credentials.

## Reproduction Assessment

Yes. Current main passes raw config into `runWebSearch`, and the PR body includes before/after terminal evidence for an Exa file-backed SecretRef; the remaining review finding is also source-reproducible through the existing local fallback diagnostic path.

## Solution Assessment

No, not quite. Resolving web-search SecretRefs at the CLI boundary is the right narrow fix, but diagnostics need to be suppressed for `--json` or returned inside the JSON envelope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep secret diagnostics out of JSON output:** `src/cli/capability-cli.ts:1535`
  - body: The new call passes `defaultRuntime` into `resolveCommandConfigWithSecrets` even when the user requested `--json`. That helper logs every diagnostic with `runtime.log`, and the gateway helper deliberately emits a diagnostic when it falls back to local web-search SecretRef resolution. In the supported gateway-unavailable local fallback path, stdout becomes a `[secrets]` line followed by JSON, so machine-readable `openclaw infer web search --json` output is invalid. Thread the JSON mode through this command and suppress stdout diagnostics, or return them inside the envelope.
  - confidence: 0.87
