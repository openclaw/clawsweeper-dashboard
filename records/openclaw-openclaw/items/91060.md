## Summary

Current main provides the generic plugin seam for prompt- and attachment-aware per-turn model selection, and the discussion links an external declarative routing plugin using it; adding a broad core routing language would duplicate plugin-owned policy and conflict with the repository’s lean-core direction.

## What This Changes

Requests declarative per-task or per-capability model selection within one agent session.

## Best Possible Solution

Keep declarative task classification and routing policy in external plugins, and open a narrow plugin-SDK issue only if a concrete lifecycle signal cannot be expressed through the existing typed hooks.

## Reproduction Assessment

Not applicable: this is a feature request, and source inspection establishes an existing extension mechanism rather than a current-main failure.

## Solution Assessment

No for a new core `modelRouting` schema; plugin-owned policy using `before_model_resolve` is the narrower maintainable solution, with generic SDK additions reserved for demonstrated gaps.
