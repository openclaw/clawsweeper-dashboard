## Summary

Keep open: current main still lacks node run/install WebSocket header support, and this PR is the canonical implementation path for the closed node-specific report, but the final PR head still has credential-handling blockers around env-secret persistence and install-time precedence.

## What This Changes

The PR adds repeatable node run/install `--header` support, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after env-only headers remain runtime-only, explicit install headers cannot be overridden by ambient CF env, and docs/tests cover the chosen precedence while coordinating the broader Gateway header contract in https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation: current main has no node `--header` option, no node header config field, and no GatewayClient header forwarding. The PR body includes copied mock-proxy output showing the missing-header 302 becoming a header-present 200, but I did not run live Cloudflare.

## Solution Assessment

No as currently patched. Passing headers into `wsOptions.headers` is the right low-level hook, but the env/config boundary and install-time precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object to `node.json`. The docs present env vars as the path for keeping secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into persisted config.
  - confidence: 0.9
- **[P2] Do not let ambient CF env override install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with exported `CF_ACCESS_CLIENT_*`, this code still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. Runtime precedence applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9
