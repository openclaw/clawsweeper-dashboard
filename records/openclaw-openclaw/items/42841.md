## Summary

Keep open: current main still lacks node WebSocket upgrade header support, and the PR is useful but not merge-ready because the installed service can still let ambient Cloudflare env vars override explicit `node install --header` credentials.

## What This Changes

The PR adds repeatable `--header` support for `openclaw node run/install`, persists and forwards node gateway headers/env to `GatewayClient`, redacts status output, and documents Cloudflare Zero Trust usage.

## Best Possible Solution

Land a current-main node-only implementation where explicit `--header` wins over env/config for foreground and installed daemons, secret-bearing values stay out of status output, and broader non-node CLI header support remains tracked by #49599.

## Reproduction Assessment

Yes. Source inspection shows current main lacks node header plumbing, and the PR-head blocker follows directly from `buildNodeInstallPlan()` copying `CF_*` into the service env while runtime applies those env values after `OPENCLAW_NODE_HEADERS`.

## Solution Assessment

No. The implementation direction is maintainable, but this PR should not merge until the explicit-header versus ambient-`CF_*` precedence bug is fixed and the branch is rebased or replaced on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stop ambient CF env from overriding install headers:** `src/commands/node-daemon-install-helpers.ts:74-81`
  - body: When `params.headers` comes from explicit `openclaw node install --header`, this still copies ambient `CF_ACCESS_CLIENT_ID` and `CF_ACCESS_CLIENT_SECRET` into the installed service. The daemon later applies those `CF_*` env vars after `OPENCLAW_NODE_HEADERS`, so stale exported credentials can replace the headers the user just supplied and reintroduce the 302/auth failure; skip copying `CF_*` when explicit headers are set, or preserve explicit serialized headers at runtime.
  - confidence: 0.91
