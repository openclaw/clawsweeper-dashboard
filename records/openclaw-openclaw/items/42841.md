## Summary

Keep open: current main still lacks node WebSocket upgrade header support and this PR is a plausible implementation path, but the patch still has install-time credential precedence and documented storage-contract issues, plus insufficient installed-service proof.

## What This Changes

The PR adds repeatable node `--header` parsing, node header env/config handling, GatewayClient WebSocket header forwarding, daemon status redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land a rebased implementation that threads headers through `GatewayClient`, defines one secret-safe precedence and persistence contract for node headers, redacts status surfaces, updates docs to match runtime behavior, and proves the installed service path.

## Reproduction Assessment

Yes for the source-level limitation and PR defects: current main has no node header option/config/client seam, and PR head shows the install/env precedence and docs/storage mismatch. I did not run a live Cloudflare or installed-service scenario in this read-only review.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but the secret precedence, storage semantics, and installed-service proof need correction before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid overriding install --header with exported CF vars:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When explicit install headers are provided, this still copies ambient `CF_ACCESS_CLIENT_ID` and `CF_ACCESS_CLIENT_SECRET` into the service environment. The daemon later resolves `OPENCLAW_NODE_HEADERS` before CF env vars, so stale exported CF credentials can replace the values the user explicitly installed.
  - confidence: 0.92
- **[P2] Align env header docs with config persistence:** `docs/cli/node.md:72`
  - body: The docs tell users env vars keep header secrets out of config, but `runNodeHost` persists resolved env-derived headers into `config.gateway.headers` before saving `node.json`. Either the runtime should stop persisting env-only secrets, or the docs need to state that env-derived headers are written to node config.
  - confidence: 0.88

## Security Review
