## Summary

Keep open: current main still lacks node WebSocket header support and this PR targets a real limitation, but the current head should not merge because env-derived secrets are persisted and explicit install headers can still be overridden by ambient Cloudflare env vars.

## What This Changes

The PR adds repeatable node run/install WebSocket upgrade headers, node header env/config handling, GatewayClient header forwarding, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after header resolution has explicit precedence, env-only secrets remain runtime-only, status/config redaction stays covered, and the broader Gateway CLI header work stays coordinated with https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation: current main has no node --header option, no node header config field, and no GatewayClient header forwarding. I did not run the mock proxy, but the PR body includes before/after live output for the missing-header 302 path.

## Solution Assessment

No as currently patched. Passing headers through wsOptions.headers is the right low-level hook, but env-only persistence and install-time Cloudflare env precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: resolvedHeaders includes values from OPENCLAW_NODE_HEADERS and CF_ACCESS_CLIENT_SECRET, then the merged object is assigned to config.gateway.headers and saved. The docs present env vars as the way to keep secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into node.json.
  - confidence: 0.9
- **[P2] Avoid overriding install --header with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When node install --header runs with exported CF_ACCESS_CLIENT_* values, this still copies those env vars into the service environment alongside explicit OPENCLAW_NODE_HEADERS. At node-host startup CF env wins over OPENCLAW_NODE_HEADERS, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9
