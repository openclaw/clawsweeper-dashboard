## Summary

Keep open: the PR targets a real missing node WebSocket-header capability, but current head still has two blocking credential-handling defects around env-secret persistence and install-time precedence. The branch is also merge-conflicting, so it needs human handling before any landing path.

## What This Changes

The PR adds repeatable `--header` support for `openclaw node run/install`, threads headers into `GatewayClient` WebSocket options, adds node header env/config handling, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after maintainers settle the secret-safe precedence and persistence contract: env-only secrets should remain runtime-only, explicit install headers should win over ambient CF env, status redaction should stay covered, and the broader Gateway CLI header request should remain coordinated.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main has no node `--header` option, no node header config field, and no `GatewayClient` header forwarding. I did not run the mock proxy, but the PR body includes before/after copied live output for the 302 path.

## Solution Assessment

No as currently patched. Passing headers through `wsOptions.headers` is the right low-level hook, but env-only persistence and install-time Cloudflare env precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then writes that merged object to `config.gateway.headers`. The docs present env vars as the way to keep secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into `node.json`.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with exported `CF_ACCESS_CLIENT_*` values, this still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. At node-host startup CF env wins over `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9
