## Summary

Keep this PR open: the node-scoped header support addresses a real gap on current main, but the latest PR head still has a merge-blocking install precedence bug where ambient Cloudflare env vars can override explicit `node install --header` credentials.

## What This Changes

The PR adds repeatable `--header` support for `openclaw node run/install`, persists and resolves node gateway headers from CLI/config/env, forwards them through `GatewayClient`, redacts status env, and updates node docs/tests/changelog.

## Best Possible Solution

Land the node-only header support with explicit `--header` values winning over env/config for both foreground and installed daemons, keep secret-bearing values out of status/process output, and leave broader CLI header support to https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Current main lacks the node header path, and the PR-head install precedence bug is source-reproducible from `buildNodeInstallPlan()` copying `CF_*` env while runtime applies those after `OPENCLAW_NODE_HEADERS`.

## Solution Assessment

No. The node-scoped direction is maintainable, but this PR should not merge until explicit install headers cannot be overridden by ambient `CF_*` env values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit install headers over ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-74`
  - body: When users run `openclaw node install --header` while stale `CF_ACCESS_CLIENT_ID` or `CF_ACCESS_CLIENT_SECRET` are exported, this still copies those ambient values into the service environment alongside the serialized explicit headers. The daemon later resolves `OPENCLAW_NODE_HEADERS` before `CF_*` and has no opts headers, so stale env can override the requested flags and keep hitting the same auth/302 failure.
  - confidence: 0.92
