## Summary

Keep open: current main still lacks node custom-header support, so the PR is not obsolete, but PR head still has blocking install precedence and secret-storage documentation issues plus incomplete real install-path proof.

## What This Changes

The PR adds repeatable node `--header` flags, node header env/config handling, `GatewayClient` WebSocket upgrade headers, daemon status redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land a rebased implementation that threads custom headers through `GatewayClient` for node run/install, defines explicit secret-safe precedence and storage semantics, documents them accurately, covers run/install/env/config cases, and includes real service proof.

## Reproduction Assessment

Yes for the current limitation and the patch defects: source inspection shows current main has no node `--header` option, no node config headers field, and no `GatewayClient` header option; PR head source shows the install precedence and env persistence mismatches. I did not run a live Cloudflare or service-install repro in this read-only review.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but install-time precedence and env/config secret-storage semantics need correction or maintainer approval before this is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid letting ambient CF env override install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with `CF_ACCESS_CLIENT_*` already exported, the install plan writes both the explicit headers into `OPENCLAW_NODE_HEADERS` and the ambient CF vars into the service env. On service startup there are no CLI opts, so `resolveNodeHostHeaders` applies `CF_*` after the JSON headers and can reconnect with stale credentials instead of the values the user just installed.
  - confidence: 0.93
- **[P2] Align env secret docs with config persistence:** `docs/cli/node.md:72`
  - body: The docs say using env vars keeps header secrets out of config, but `runNodeHost` persists the resolved headers into `config.gateway.headers` before saving `node.json`. Users who use `OPENCLAW_NODE_HEADERS` or `CF_*` would still have those secret-bearing headers written to config, so either the code or the documented storage contract needs to change.
  - confidence: 0.86
