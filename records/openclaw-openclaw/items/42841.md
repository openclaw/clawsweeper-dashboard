## Summary

Keep open: current main still lacks node WebSocket upgrade header support, and this PR is useful but currently incorrect because it can persist env-only header secrets to node config and let ambient Cloudflare env vars override explicit install headers.

## What This Changes

The branch adds repeatable `openclaw node run/install --header` support, node config/env header plumbing, GatewayClient WebSocket header forwarding, docs, changelog, and focused tests.

## Best Possible Solution

Land corrected node run/install header support that keeps env-only secrets runtime-only, makes explicit install headers authoritative, and coordinates shared GatewayClient semantics with https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation. Source inspection shows current main lacks node `--header` CLI options and GatewayClient custom-header forwarding, and the PR body includes copied mock-proxy output showing the missing-header 302 becoming a header-present 200.

## Solution Assessment

No as currently patched. Passing headers through `ws` is the right lower-level mechanism, but this implementation needs a secret-safe env/config boundary and install-time precedence that cannot let ambient CF env override explicit headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_*`, then this block saves the merged object as `config.gateway.headers`. The docs present env vars as the path for keeping secrets out of config, so env-only service-token headers should be used at runtime without being copied into `node.json`.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with `CF_ACCESS_CLIENT_*` exported, this code stores the explicit headers in `OPENCLAW_NODE_HEADERS` but also copies the ambient CF vars. The installed service has no CLI `optsHeaders`, so runtime applies those CF values after `OPENCLAW_NODE_HEADERS` and can use stale shell credentials instead of the headers the user explicitly installed.
  - confidence: 0.9
