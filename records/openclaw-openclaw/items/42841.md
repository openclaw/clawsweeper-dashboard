## Summary

Keep open: current main still lacks node run/install custom WebSocket header support, and this PR is the active implementation path, but the current branch still has blocking header-precedence and secret-handling issues before it should merge.

## What This Changes

The PR adds repeatable `--header` support to `openclaw node run` and `openclaw node install`, forwards those headers into `GatewayClient`, persists/propagates node header env vars, and documents Cloudflare Zero Trust usage.

## Best Possible Solution

Land a rebased implementation that threads node custom headers through `GatewayClient`, resolves headers with an explicit and documented precedence order, avoids unintended ambient secret persistence, and includes focused tests plus real install-path proof.

## Reproduction Assessment

Yes for the current limitation and the PR regression: source inspection shows current main has no node `--header` or GatewayClient header path, and PR head source shows stale `CF_*` env can override explicit install headers. I did not run commands because this was a read-only checkout review.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right core seam, but install-time precedence and the documented secret-storage behavior need correction before this is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not copy CF env when --header is explicit:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install` is run with explicit `--header` values while old `CF_ACCESS_CLIENT_*` vars are exported, this serializes both sources into the service environment. `runNodeHost` applies `CF_*` after `OPENCLAW_NODE_HEADERS`, so the installed daemon can ignore the values the user just provided and keep failing Zero Trust auth with stale credentials.
  - confidence: 0.93
- **[P2] Correct the env-only storage promise:** `docs/cli/node.md:72`
  - body: The docs say env vars keep secrets out of config, but the runner persists resolved `OPENCLAW_NODE_HEADERS` and `CF_*` headers back into node config before connecting. Operators choosing env vars to avoid config persistence will still write the secret there, so either stop persisting env-derived headers or document the owner-only config storage clearly.
  - confidence: 0.86

## Security Review

Status: needs_attention

  - body: `buildNodeInstallPlan` copies `CF_ACCESS_CLIENT_SECRET` from the install shell even when explicit `--header` values were provided, which can persist an unrelated or stale secret in the service definition and override the requested header at runtime.
