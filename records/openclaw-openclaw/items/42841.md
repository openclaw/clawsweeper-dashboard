## Summary

Keep open: current main still lacks node WebSocket upgrade header support, and this PR is the active implementation candidate, but the latest patch still has secret-handling and install-time precedence bugs that should block merge.

## What This Changes

The PR adds repeatable `openclaw node run/install --header` support, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after env-only headers remain runtime-only, explicit install headers cannot be overridden by ambient CF env, and docs/tests reflect the chosen precedence while coordinating the broader Gateway CLI header surface in https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main has no node `--header` option, no node header config field, and no GatewayClient header forwarding. I did not run a live Cloudflare setup in this read-only review.

## Solution Assessment

No as currently patched. Passing headers into `wsOptions.headers` is the right low-level hook, but the env/config persistence and install-time precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object to `node.json`. The new docs present env vars as the path for keeping service-token secrets out of config and command lines, so env-only headers should be used for the current run without being copied into persisted config.
  - confidence: 0.9
- **[P2] Do not install ambient CF env with explicit headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with exported `CF_ACCESS_CLIENT_*`, this code still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. Runtime precedence applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a secret-bearing header surface and still exposes two credential-at-rest or credential-precedence risks before merge.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and increasing credential-at-rest exposure.
  - confidence: 0.9
- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: The install plan copies `CF_ACCESS_CLIENT_SECRET` into the service environment even when explicit `--header` values were provided, and runtime precedence lets that ambient value override the serialized explicit header.
  - confidence: 0.9
