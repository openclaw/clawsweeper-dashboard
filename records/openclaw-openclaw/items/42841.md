## Summary

Keep open: the PR addresses a valid missing node-header path, but current head still has security-relevant env/config defects and is merge-conflicting with current main.

## What This Changes

The PR adds repeatable node run/install `--header` support, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after env-only headers remain runtime-only, explicit install headers cannot be overridden by ambient CF env, docs/tests cover the chosen precedence, and the broader Gateway header contract is coordinated in https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation: current main has no node `--header` option, no node header config field, and no GatewayClient header forwarding. I did not run a live Cloudflare setup; the PR body includes local mock-proxy output showing the missing-header 302 becoming a header-present 200.

## Solution Assessment

No as currently patched. Passing headers through `wsOptions.headers` is the right low-level hook, but the env-only persistence and install-time CF precedence issues need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object to `node.json`. The docs present env vars as the path for keeping secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into persisted config.
  - confidence: 0.9
- **[P2] Avoid overriding install --header with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with exported `CF_ACCESS_CLIENT_*`, this code still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. Runtime precedence applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a secret-bearing header surface and current head still persists env-only header secrets while allowing ambient CF env to override explicit install headers.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and increasing credential-at-rest exposure.
