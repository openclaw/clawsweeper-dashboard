## Summary

Keep open: current main still lacks node custom-header support and this PR is the active implementation path, but the current head still has two blocking secret-handling and precedence defects before merge.

## What This Changes

The PR adds repeatable node `--header` support, node header env/config handling, WebSocket header forwarding, docs, changelog, and focused tests for Cloudflare Zero Trust-style WebSocket upgrades.

## Best Possible Solution

Land node header support after maintainers settle the secret-safe persistence and precedence contract, keeping env-only secrets runtime-only and coordinating the shared Gateway CLI header work.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main has no node `--header` option, no node header config field, and no `GatewayClient` header forwarding. I did not run the mock proxy, but the PR body includes copied before/after live output for the 302 path.

## Solution Assessment

No as currently patched. Passing headers through `wsOptions.headers` is the right low-level hook, but env-only persistence and install-time Cloudflare env precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then writes that merged object to `config.gateway.headers`. The docs present env vars as the way to keep secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into `node.json`.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with exported `CF_ACCESS_CLIENT_*` values, this still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. At node-host startup CF env wins over `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a secret-bearing WebSocket header surface and current head still has credential persistence and precedence defects.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and increasing credential-at-rest exposure.
  - confidence: 0.9
- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: The install plan copies `CF_ACCESS_CLIENT_SECRET` into the service environment even when explicit `--header` values were provided, and runtime precedence lets that ambient value override the serialized explicit header.
  - confidence: 0.9
