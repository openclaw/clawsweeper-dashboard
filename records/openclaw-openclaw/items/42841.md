## Summary

This PR should stay open: the node-specific gap is still real on current main, but the current PR head has blocking secret-handling and precedence bugs and is dirty/conflicting against main.

## What This Changes

The PR adds repeatable custom WebSocket upgrade headers to `openclaw node run` and `openclaw node install`, plus node config/env plumbing, docs, changelog, and focused tests.

## Best Possible Solution

Port the node run/install header support onto current main, preserve the current node-host/gateway-client behavior, and fix secret persistence plus install-time header precedence before maintainer review.

## Reproduction Assessment

Yes for the requested node-specific gap: current main exposes no node run/install header option and has no custom header option passed into the WebSocket upgrade. I did not run the CLI because this is a read-only review, but the path is source-reproducible.

## Solution Assessment

No. The design direction is reasonable, but this implementation stores env-only header secrets despite documenting that env vars avoid config storage, still lets exported CF_* env vars override explicit install headers, and needs a current-main port.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep env-only header secrets out of node config:** `src/node-host/runner.ts:199-217`
  - body: The docs tell users they can set env vars so header secrets are not stored in config, but `runNodeHost` resolves `OPENCLAW_NODE_HEADERS` / `CF_*` into `gateway.headers` and immediately saves `node.json`. That means an env-only Cloudflare secret is silently persisted on disk, which defeats the documented secret-handling path.
  - confidence: 0.94
- **[P2] Do not let CF env override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs in a shell that also has `CF_ACCESS_CLIENT_ID` or `CF_ACCESS_CLIENT_SECRET`, the install plan writes both `OPENCLAW_NODE_HEADERS` and the CF env vars into the service. At daemon startup `resolveNodeHostHeaders` applies `OPENCLAW_NODE_HEADERS` first and then overwrites those keys from `CF_*`, so stale exported service tokens can replace the explicit header values the user just installed.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The patch can silently persist env-supplied auth headers to `node.json` despite documenting env vars as a way to avoid config storage.

Concerns:

- **[medium] Env-only header secrets are written to config:** `src/node-host/runner.ts:199`
  - body: `runNodeHost` merges `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET` into resolved headers, stores them under `gateway.headers`, and saves node config. Users following the documented env-var path can therefore end up with Cloudflare service-token material persisted to disk unexpectedly.
  - confidence: 0.94
