Full review comments:

- **[P2] Avoid overriding installed headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When explicit `node install --header` values are serialized into `OPENCLAW_NODE_HEADERS`, this still copies `CF_ACCESS_CLIENT_ID` and `CF_ACCESS_CLIENT_SECRET` into the installed service environment. At startup those CF vars are applied after the JSON headers, so stale exported Cloudflare credentials can replace the values the user just installed and recreate the auth/302 failure.
  - confidence: 0.93
- **[P2] Do not persist env-only header secrets unexpectedly:** `src/node-host/runner.ts:199-217`
  - body: `runNodeHost` saves the fully resolved header set into `config.gateway.headers`, including headers sourced only from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET`. That contradicts the docs' promise that env vars keep secrets out of config and creates unexpected credential-at-rest behavior for operators choosing env-based storage.
  - confidence: 0.9
