## Summary

This PR should stay open: current main still lacks node WebSocket upgrade header support, but the latest branch has two narrow correctness/security blockers around installed-service credential precedence and secret-persistence docs.

## What This Changes

The branch adds repeatable `openclaw node run/install --header` support, node header config/env plumbing, GatewayClient WebSocket headers, daemon status redaction, tests, docs, and a changelog entry.

## Best Possible Solution

Land the node-scoped header support after fixing install-time credential precedence and aligning the reverse-proxy docs with actual config persistence; keep broader CLI WebSocket header support in https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Current main source inspection shows no `node run/install --header` path, and the remaining PR-head install bug is source-reproducible by combining `buildNodeInstallPlan()` with `resolveNodeHostHeaders()` precedence.

## Solution Assessment

No. The node-scoped direction is maintainable, but this PR should not merge until explicit install headers cannot be overridden by ambient CF env and the docs accurately describe where secret-bearing headers are persisted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve install --header over ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-74`
  - body: When `openclaw node install --header` runs with stale `CF_ACCESS_CLIENT_ID` or `CF_ACCESS_CLIENT_SECRET` exported, the install plan still copies those ambient vars beside `OPENCLAW_NODE_HEADERS`. The daemon applies CF env after `OPENCLAW_NODE_HEADERS`, so the installed service can use stale ambient credentials instead of the explicit install flags.
  - confidence: 0.94
- **[P2] Align env-header docs with config persistence:** `docs/cli/node.md:72`
  - body: This says env vars keep header secrets out of config, but `runNodeHost()` persists resolved headers, including env-derived headers, back into node config on startup. Either avoid persisting env-derived secret headers or document that they may be stored in the 0600 node config file.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds secret-bearing header/env handling and still has concrete credential precedence plus persistence-documentation issues.

Concerns:

- **[medium] Ambient CF env can override explicit install credentials:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: Explicit install headers are serialized into `OPENCLAW_NODE_HEADERS`, but the installed service environment also receives ambient `CF_ACCESS_CLIENT_ID` and `CF_ACCESS_CLIENT_SECRET`; runtime precedence then lets those ambient values replace explicit header credentials.
  - confidence: 0.92
- **[medium] Docs misstate where env-provided header secrets persist:** `docs/cli/node.md:72`
  - body: The docs say env vars avoid config storage, but node-host startup persists resolved env headers into node config, which can surprise operators relying on environment-only secret storage.
  - confidence: 0.86
