## Summary

Keep this PR open: the node-scoped header support is still useful and is not on current main, but PR head still has a merge-blocking install precedence bug and a credential-handling docs mismatch.

## What This Changes

The PR adds repeatable `--header` support for node run/install, resolves headers from CLI/config/env, forwards them through `GatewayClient`, redacts daemon status env, and documents reverse-proxy auth.

## Best Possible Solution

Land the node-only header support after fixing install header precedence and credential-persistence docs, while leaving broader CLI WebSocket header support to https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Current main lacks the node header path, and the remaining PR-head install precedence bug is source-reproducible from `buildNodeInstallPlan()` copying `CF_*` env while runtime applies those after `OPENCLAW_NODE_HEADERS`.

## Solution Assessment

No. The node-scoped direction is maintainable, but this PR should not merge until explicit install headers cannot be overridden by ambient CF env and the docs accurately describe where secret-bearing headers are persisted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve install --header over ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-74`
  - body: When `openclaw node install --header` runs with stale `CF_ACCESS_CLIENT_ID` or `CF_ACCESS_CLIENT_SECRET` exported, this still copies those ambient vars into the installed service beside `OPENCLAW_NODE_HEADERS`. The daemon then applies `OPENCLAW_NODE_HEADERS` before `CF_*`, so the service can use stale ambient credentials instead of the explicit install flags.
  - confidence: 0.94
- **[P2] Align env-header docs with config persistence:** `docs/cli/node.md:72`
  - body: This promises users that env vars keep header secrets out of config, but `runNodeHost()` persists resolved headers, including env-derived headers, back into node config on startup. Either stop persisting env-derived secret headers or document that they may be stored in the 0600 node config file.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces secret-bearing header env handling, and PR head still has concrete credential precedence and persistence-documentation issues.

Concerns:

- **[medium] Ambient CF env can override explicit install credentials:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: Explicit install headers are serialized into `OPENCLAW_NODE_HEADERS`, but the installed service environment also receives ambient `CF_ACCESS_CLIENT_ID` and `CF_ACCESS_CLIENT_SECRET`; runtime precedence then lets those ambient values replace explicit header credentials.
  - confidence: 0.92

  - body: The docs say env vars avoid config storage, but node-host startup persists resolved env headers into node config, which can surprise operators relying on environment-only secret storage.
  - confidence: 0.86
