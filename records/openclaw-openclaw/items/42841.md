## Summary

Keep open. Current main still lacks node WebSocket header support, but the PR head still has a concrete install-time precedence bug where explicit installed headers can be overridden by ambient Cloudflare env vars.

## What This Changes

This PR adds repeatable `openclaw node run/install --header` support, node header config/env resolution, GatewayClient WebSocket header forwarding, daemon status redaction, and related docs/tests/changelog coverage.

## Best Possible Solution

Land the node-scoped header support once explicit install headers cannot be overridden by ambient env/config, secret-bearing values remain out of process/status output, and broader CLI header support stays tracked by https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Current main source lacks any node `--header` path, and the PR-head blocker is source-reproducible by combining explicit install headers with stale `CF_ACCESS_CLIENT_*` env values.

## Solution Assessment

No. The node-scoped approach is maintainable, but the PR should not merge until `buildNodeInstallPlan()` preserves explicit `--header` credentials over ambient Cloudflare env values for installed services.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve explicit install headers over ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-74`
  - body: When users run `openclaw node install --header` while stale `CF_ACCESS_CLIENT_ID` or `CF_ACCESS_CLIENT_SECRET` are exported, the install plan writes both explicit serialized headers and ambient CF env vars into the service. Runtime then applies `CF_*` after `OPENCLAW_NODE_HEADERS`, so stale env can override the requested install flags and keep the daemon hitting the same auth/302 failure.
  - confidence: 0.92
