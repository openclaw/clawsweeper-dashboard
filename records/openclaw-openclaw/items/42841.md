## Summary

Keep this PR open: current main still lacks custom WebSocket upgrade headers for node run/install, and the PR is the focused implementation attempt for the linked Zero Trust use case. The latest PR head is not merge-ready because it still has credential-storage and env-precedence issues, plus the branch is currently dirty against main.

## What This Changes

The PR adds repeatable node `--header` flags, threads node header config/env values into `GatewayClient` WebSocket options, redacts service env in node status output, updates node CLI docs/changelog, and adds targeted tests.

## Best Possible Solution

Land this PR or an equivalent narrow core change after rebasing onto current main and making header source semantics explicit: CLI headers should win, env-only secret headers should stay runtime-only unless persistence is deliberately documented, node status/logs/args must not expose secrets, docs should match the storage model, and the changelog should credit the contributor/reporter.

## Reproduction Assessment

Yes. The linked report and PR body provide a mock reverse-proxy path that returns 302 without Cloudflare headers, and current main source shows no node CLI or `GatewayClient` header path. The PR-head regressions are statically reproducible from the inspected env merge and service install code.

## Solution Assessment

No, not as currently written. The `GatewayClientOptions.headers` seam plus repeatable node CLI flags is the right narrow mechanism, but the credential persistence and install-time CF env precedence need to be fixed before this is the best maintainable solution.
