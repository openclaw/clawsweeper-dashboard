## Summary

Keep open: current main still lacks node WebSocket upgrade header support, and this PR is the canonical node run/install implementation path, but the current head still has two secret-handling defects that should block merge.

## What This Changes

The PR adds repeatable node run/install `--header` support, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after env-only headers remain runtime-only, explicit install headers cannot be overridden by ambient CF env, docs/tests cover the chosen precedence, and broader remote Gateway CLI headers stay coordinated in https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows no node `--header` option, no node header config field, and no GatewayClient header forwarding. The PR body also includes copied mock-proxy output showing the missing-header 302 becomes a header-present 200 after the patch.

## Solution Assessment

No as currently patched. Passing headers through `wsOptions.headers` is the right low-level hook, but env-only persistence and install-time CF env precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object into `node.json`. The docs describe env vars as the path for keeping secrets out of config and command lines, so env-only service-token headers should be used for the run without being copied into persisted config.
  - confidence: 0.9
- **[P2] Avoid overriding install --header with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with exported `CF_ACCESS_CLIENT_*`, this code still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. Runtime precedence applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a secret-bearing header surface and current head still has two concrete credential-at-rest/precedence concerns.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and increasing credential-at-rest exposure.
  - confidence: 0.9
- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: The install plan copies `CF_ACCESS_CLIENT_SECRET` into the service environment even when explicit `--header` values were provided, and runtime precedence lets that ambient value override the serialized explicit header.
  - confidence: 0.9
