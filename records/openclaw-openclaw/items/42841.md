## Summary

Keep open: the underlying node header support is useful and current main still lacks it, but this PR head still has two merge-blocking secret-handling regressions around env-derived header persistence and install-time env precedence.

## What This Changes

The branch adds repeatable node `--header` parsing, node run/install config and env plumbing, GatewayClient WebSocket header forwarding, docs, changelog, and focused tests.

## Best Possible Solution

Land corrected node run/install header support that keeps env-only secrets out of `node.json`, makes explicit install headers authoritative, and leaves broader remote Gateway CLI header support to https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Source inspection shows current main lacks both node `--header` CLI options and `GatewayClient` custom-header forwarding, and the PR body includes copied mock-proxy output showing the 302-to-200 behavior change.

## Solution Assessment

No. Passing headers through `ws` is the right lower-level mechanism, but this patch still needs to separate env-only secrets from persisted config and prevent ambient CF env from overriding explicit install headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-sourced headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_*`, then this block saves that merged object as `config.gateway.headers`. The docs present env vars as the path for keeping secrets out of config, so env-only Cloudflare service-token headers should be used at runtime without being copied into `node.json`.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with `CF_ACCESS_CLIENT_*` exported, this code stores explicit headers in `OPENCLAW_NODE_HEADERS` but also copies the ambient CF env vars. The installed service has no CLI `optsHeaders`, so runtime applies those CF values after `OPENCLAW_NODE_HEADERS` and can use stale shell credentials instead of the headers the user explicitly installed.
  - confidence: 0.9

## Security Review
