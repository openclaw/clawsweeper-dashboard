## Summary

Keep open. The feature is still missing from current main and the PR has a plausible implementation path, but the latest head still has a source-proven install-time header precedence bug, a secret-storage docs mismatch, and insufficient real install-path proof.

## What This Changes

The PR adds repeatable node `--header` support, node header env/config plumbing, GatewayClient WebSocket header forwarding, redaction/status tests, and Cloudflare Zero Trust docs.

## Best Possible Solution

Land a rebased implementation that threads custom headers through `GatewayClient`, defines and tests explicit precedence for CLI/env/config sources, avoids unintended ambient secret overrides, and documents the real persistence model.

## Reproduction Assessment

Yes, at source level. Current main has no node `--header` or GatewayClient header path, and PR head shows the install precedence regression by carrying both explicit header JSON and ambient CF env into the daemon environment.

## Solution Assessment

No, not as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but install-time precedence, secret-persistence documentation, and real install proof need correction first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid overriding explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67-74`
  - body: When `openclaw node install --header ...` runs in a shell that already exports `CF_ACCESS_CLIENT_ID` or `CF_ACCESS_CLIENT_SECRET`, the service environment contains both `OPENCLAW_NODE_HEADERS` and the ambient CF vars. `runNodeHost` applies CF vars after `OPENCLAW_NODE_HEADERS`, so the installed daemon can use stale exported credentials instead of the explicit headers the user just passed.
  - confidence: 0.92
- **[P2] Match docs to header persistence:** `docs/cli/node.md:72`
  - body: This says environment variables keep secrets out of config, but `runNodeHost` persists the resolved headers into `node.json` before connecting. Operators choosing env vars to avoid config storage would still write the secrets there, so either change persistence or document the owner-only config storage accurately.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch still has concrete secret-handling concerns around ambient Cloudflare credentials and the documented persistence model.

Concerns:

- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: `buildNodeInstallPlan` copies `CF_ACCESS_CLIENT_SECRET` from the install shell even when explicit `--header` values were provided, and runtime precedence can then prefer the stale ambient secret over the requested header JSON.
  - confidence: 0.9

  - body: The docs promise env vars keep secrets out of config, but the runner persists resolved env-derived headers into node config; users can rely on the wrong storage boundary for service-token material.
  - confidence: 0.84
