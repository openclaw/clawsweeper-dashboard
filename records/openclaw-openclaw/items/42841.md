## Summary

Keep open: current main still lacks node WebSocket header support and this PR targets a real gap, but the PR head is not merge-safe because it still mishandles secret-bearing header persistence and install-time precedence.

## What This Changes

The PR adds repeatable node `--header` parsing, node run/install env and config plumbing, GatewayClient WebSocket header forwarding, docs, changelog, and focused tests.

## Best Possible Solution

Land corrected node run/install header support that keeps env-only secrets runtime-only, makes explicit install headers authoritative, and coordinates shared GatewayClient semantics with https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Source inspection shows current main lacks both node `--header` CLI options and GatewayClient custom-header forwarding, and the PR body includes copied mock-proxy output showing the missing-header 302 becoming a header-present 200.

## Solution Assessment

No. Passing headers through `ws` is the right lower-level mechanism, but this patch still needs to keep env-sourced secrets out of `node.json` and prevent ambient CF env from overriding explicit install headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-sourced headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_*`, then this block saves the merged object as `config.gateway.headers`. The docs present env vars as the path for keeping secrets out of config, so env-only service-token headers should be used at runtime without being copied into `node.json`.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with `CF_ACCESS_CLIENT_*` exported, this code stores the explicit headers in `OPENCLAW_NODE_HEADERS` but also copies the ambient CF vars. The installed service has no CLI `optsHeaders`, so runtime can apply those CF values after `OPENCLAW_NODE_HEADERS` and use stale shell credentials instead of the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces concrete credential-handling regressions in where Cloudflare header secrets are stored and which source wins at daemon runtime.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and broadening credential-at-rest exposure.
  - confidence: 0.9
