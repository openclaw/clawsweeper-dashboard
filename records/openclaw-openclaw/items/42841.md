## Summary

Keep open: current main still lacks node WebSocket upgrade header support, and this PR is the canonical implementation candidate, but the branch still has two concrete credential-handling regressions that should block merge.

## What This Changes

The branch adds repeatable node `--header` support, node header env handling, GatewayClient WebSocket header forwarding, config/docs/changelog updates, and focused tests.

## Best Possible Solution

Land a corrected node run/install implementation that keeps explicit headers authoritative, keeps env-only secrets out of persisted config, and leaves broader remote Gateway CLI header support to https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Source inspection shows current main has no node `--header` option and no custom-header path into `GatewayClient`, and the PR body includes mock-proxy output for the 302-to-200 behavior change.

## Solution Assessment

No. Passing headers through `ws` is the right lower-level approach, but this patch needs to separate runtime env headers from persisted config and keep explicit install headers ahead of ambient Cloudflare env values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not persist env-sourced header secrets:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes `OPENCLAW_NODE_HEADERS` and `CF_*` values and is saved into `config.gateway.headers`. The docs tell users env vars keep secrets out of config, so installed-service credentials can be silently copied to `node.json`; keep runtime env headers separate from persisted config.
  - confidence: 0.88
- **[P2] Keep explicit install headers ahead of CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with `CF_ACCESS_CLIENT_*` exported, the plan stores both `OPENCLAW_NODE_HEADERS` and the CF env vars. Runtime then applies CF env after `OPENCLAW_NODE_HEADERS` for the service, so stale shell credentials can override the headers the user explicitly installed.
  - confidence: 0.9
