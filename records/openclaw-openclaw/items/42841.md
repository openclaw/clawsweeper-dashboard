## Summary

Keep open: this PR addresses a real node Gateway gap and current main still lacks custom WebSocket upgrade headers, but the branch is not merge-ready because current head still mishandles secret-bearing env/header precedence and persistence.

## What This Changes

The PR adds repeatable `openclaw node run/install --header`, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after headers are resolved with explicit precedence, env-only secrets remain runtime-only, status/config redaction is covered, and the broader Gateway header request stays coordinated with https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation: current main has no node `--header` option, no node header config field, and no GatewayClient header forwarding. The PR body also includes mock-proxy output showing the missing-header 302 changing to a header-present 200.

## Solution Assessment

No as currently patched. Passing headers through `wsOptions.headers` is the right low-level hook, but env-only persistence and install-time CF precedence need correction before this is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object to `node.json`. The docs present env vars as the way to keep secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into persisted config.
  - confidence: 0.9
- **[P2] Do not let ambient CF env override install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with exported `CF_ACCESS_CLIENT_*`, this code still copies those env vars into the service environment alongside explicit `OPENCLAW_NODE_HEADERS`. Runtime precedence applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention
