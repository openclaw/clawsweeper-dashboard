## Summary

Keep open. The PR addresses a real node-host gap that current main still lacks, but it should not merge as-is because env-sourced header secrets are saved into node config despite the docs, and explicit install headers can still be overridden by ambient Cloudflare env vars.

## What This Changes

The PR adds repeatable `--header` support for `openclaw node run` and `openclaw node install`, threads headers into the WebSocket client, adds node header env-var handling, docs, changelog, and tests.

## Best Possible Solution

Land a corrected node run/install implementation that keeps explicit headers authoritative, avoids undocumented secret persistence, and leaves broader gateway CLI header support to the separate remote-CLI feature issue.

## Reproduction Assessment

Yes. Current main has no node `--header` option and does not pass custom headers into `GatewayClient`, while source inspection of the PR head shows the remaining credential precedence and persistence regressions without needing to execute the branch.

## Solution Assessment

No. Passing headers through `ws` is the right direction, but the current patch needs to separate runtime env headers from persisted config and keep explicit install headers ahead of ambient `CF_*` env values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not persist env-sourced header secrets:** `src/node-host/runner.ts:209-214`
  - body: The env-var path is documented as keeping secrets out of config, but `resolvedHeaders` includes `OPENCLAW_NODE_HEADERS` and `CF_*` values and is saved into `node.json` before the client starts. A service installed with env-only Cloudflare credentials will silently copy `CF-Access-Client-Secret` to disk, so the runtime headers need to stay separate from the persisted config contract.
  - confidence: 0.88
- **[P2] Keep explicit install headers ahead of CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs from a shell with `CF_ACCESS_CLIENT_*` set, the install plan writes both `OPENCLAW_NODE_HEADERS` and the CF env vars. Runtime precedence then applies `CF_*` after `OPENCLAW_NODE_HEADERS`, so stale shell credentials override the explicit flags and the installed node can still fail behind Zero Trust.
  - confidence: 0.9

## Security Review
