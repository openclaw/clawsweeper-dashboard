## Summary

Keep open: current main still lacks node custom WebSocket header support and this PR targets the right surface, but the current head remains blocked by install-time secret precedence, env/config storage documentation, and incomplete real install-path proof.

## What This Changes

The PR adds repeatable node `--header` options, node header env handling, node config persistence, `GatewayClient` WebSocket headers, daemon status redaction, tests, docs, and a changelog entry.

## Best Possible Solution

Land a rebased implementation that threads custom headers through `GatewayClient` for node run/install, defines explicit secret-safe precedence and storage semantics, documents them, covers run/install/env/config cases, and includes real service proof.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows no node `--header` option, no node config headers field, and no `GatewayClient` header option. I did not run a live Cloudflare or service-install repro because this review was read-only.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but install-time precedence and env/config secret-storage semantics need correction or maintainer approval before this is maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs while `CF_ACCESS_CLIENT_*` is exported, the install plan writes both the explicit headers into `OPENCLAW_NODE_HEADERS` and the ambient CF vars into the service env. The service starts without CLI opts, so `resolveNodeHostHeaders` applies CF env after the JSON headers and can reconnect with stale credentials instead of the values the user just provided.
  - confidence: 0.93
- **[P2] Align env secret docs with config persistence:** `docs/cli/node.md:72`
  - body: The docs say env vars keep header secrets out of config, but `runNodeHost` persists resolved headers into `config.gateway.headers` before saving `node.json`. Users who use `OPENCLAW_NODE_HEADERS` or `CF_*` would still get those secret-bearing headers written to config, so either the code or docs need to change.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR adds secret-bearing header handling and current head still has concrete precedence and storage-contract concerns.

Concerns:

- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: `buildNodeInstallPlan` copies `CF_ACCESS_CLIENT_SECRET` from the install shell even when explicit `--header` values are provided, so the installed service can prefer an unrelated or stale secret at runtime.
  - confidence: 0.9

  - body: `runNodeHost` saves resolved headers into node config while the docs tell users env vars keep secrets out of config, creating a mismatch for secret-bearing header storage.
  - confidence: 0.84
