## Summary

Keep open: the PR addresses a real missing node WebSocket header path, but the current patch still has credential-handling defects around env-sourced header persistence and install-time precedence, so it is not safe to merge as-is.

## What This Changes

The branch adds repeatable `openclaw node run/install --header` support, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support only after env-only secrets remain runtime-only, explicit install headers cannot be overridden by ambient CF env, and the broader Gateway header surface is coordinated with https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes for the current limitation: current main has no node `--header` option, no node header config field, and no GatewayClient header forwarding. The PR body also includes copied mock-proxy output showing the missing-header 302 becoming a header-present 200; I did not run a live Cloudflare setup.

## Solution Assessment

No as currently patched. Passing headers into `ws` is the right low-level hook, but the implementation needs a secret-safe env/config boundary and install-time precedence that cannot let ambient CF env override explicit headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env headers out of saved node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values parsed from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object to `node.json`. The new docs present env vars as the way to keep secrets out of config and command lines, so env-only service-token headers should be used for this run without being copied into `config.gateway.headers`.
  - confidence: 0.9
- **[P2] Do not copy CF env beside explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with `CF_ACCESS_CLIENT_*` exported, this code writes the explicit headers to `OPENCLAW_NODE_HEADERS` but also copies the ambient CF vars. The installed service has no CLI opts, and runtime applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces a secret-bearing CLI/env/config surface and still has credential-at-rest and credential-precedence issues.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and increasing credential-at-rest exposure.
  - confidence: 0.9
- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: The install plan copies `CF_ACCESS_CLIENT_SECRET` into the service environment even when explicit `--header` values were provided, and runtime precedence lets that ambient value override the serialized explicit header.
  - confidence: 0.9
