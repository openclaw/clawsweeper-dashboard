## Summary

This PR addresses a real missing node-to-gateway capability, but it is not merge-ready: the branch conflicts with current main after the WebSocket client moved into `packages/gateway-client`, and its credential precedence/persistence design can copy environment-only Cloudflare secrets into `node.json`. The linked broader request remains open at https://github.com/openclaw/openclaw/issues/49599, but this PR still contains useful node-specific work rather than being safely superseded.

## What This Changes

Adds repeatable WebSocket upgrade headers to `openclaw node run` and `node install`, persists and resolves header config/environment sources, forwards headers to the gateway client, redacts daemon status output, and documents the workflow.

## Best Possible Solution

Rebase onto current main, add a narrow generic header option at the shared gateway-client boundary, keep runtime environment credentials ephemeral rather than writing them into node config, define and test one explicit precedence/rotation contract, and provide redacted installed-service WebSocket proof before merge.

## Reproduction Assessment

Yes for the underlying limitation: current main has no node CLI/config path to attach required headers, so an authenticating reverse proxy will reject or redirect the upgrade. The submitted after-fix proof is not a complete WebSocket or installed-service reproduction.

## Solution Assessment

No. Threading generic headers through the current shared gateway client is reasonable, but persisting resolved environment secrets and adding overlapping Cloudflare-specific sources is not the narrowest safe design.
