## Summary

Keep open: the node-header support is still useful and current main does not provide it, but the latest PR head still has two concrete credential-handling regressions before it can merge.

## What This Changes

The branch threads repeatable node WebSocket upgrade headers through the node run/install CLI, node config/env handling, GatewayClient, docs, changelog, and focused tests.

## Best Possible Solution

Land corrected node run/install header support that keeps env-only secrets out of persisted config, makes explicit install headers authoritative, and leaves broader CLI remote-header support to https://github.com/openclaw/openclaw/issues/49599.

## Reproduction Assessment

Yes. Current main lacks both the node `--header` CLI surface and the `GatewayClient` custom-header path, and the PR body includes copied mock-proxy output showing the 302-to-200 behavior change.

## Solution Assessment

No. Passing headers through `ws` is the right lower-level mechanism, but this patch must separate runtime env headers from persisted config and prevent ambient CF env from overriding explicit install headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-sourced headers out of node config:** `src/node-host/runner.ts:199-217`
  - body: `resolvedHeaders` includes `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_*` values, then gets saved into `config.gateway.headers`. The docs present env vars as the path for keeping secrets out of config, so this silently broadens credential persistence into `node.json`; only explicitly configured/persisted headers should be saved.
  - confidence: 0.9
- **[P2] Keep ambient CF env from overriding install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header` runs with `CF_ACCESS_CLIENT_*` exported, the plan stores explicit headers in `OPENCLAW_NODE_HEADERS` but also copies the ambient CF env vars. The installed service has no `optsHeaders`, so runtime applies those CF vars after `OPENCLAW_NODE_HEADERS` and can use stale shell credentials instead of the values the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff still has concrete secret-handling regressions around credential persistence and env precedence.

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
