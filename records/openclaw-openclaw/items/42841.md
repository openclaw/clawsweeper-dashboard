## Summary

Keep open: this PR addresses a real missing node WebSocket-header surface, but the latest diff still has credential persistence and precedence bugs before it should merge. Current main and the latest release do not implement the requested node run/install header support, and the broader Gateway header surface remains tracked at https://github.com/openclaw/openclaw/issues/49599.

## What This Changes

The PR adds repeatable node run/install `--header` support, node header env/config resolution, GatewayClient WebSocket header forwarding, daemon env redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land node header support only after env-only secrets remain runtime-only, explicit install headers cannot be overridden by ambient CF env, and the node-specific surface is coordinated with the broader Gateway header contract.

## Reproduction Assessment

Yes for the current limitation: source inspection shows current main and v2026.5.7 have no node `--header` option, no node header config field, and no GatewayClient header forwarding. The PR body includes copied mock-proxy output showing the missing-header 302 becoming a header-present 200; I did not run live Cloudflare.

## Solution Assessment

No as currently patched. Passing headers into `ws` is the right low-level hook, but the implementation needs a secret-safe env/config boundary and install-time precedence that cannot let ambient CF env override explicit headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env headers out of saved node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then this assignment saves the merged object to `node.json`. The docs present env vars as the path for keeping secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into persisted config.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with exported `CF_ACCESS_CLIENT_*`, this unconditionally copies those env vars alongside the explicit serialized headers. The installed service later applies `CF_*` after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the values the user explicitly installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces a secret-bearing CLI/env/config surface and still has credential-at-rest and credential-precedence issues.

Concerns:

- **[medium] Env-sourced header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and increasing credential-at-rest exposure.
  - confidence: 0.9
- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: The install plan copies `CF_ACCESS_CLIENT_SECRET` into the service environment even when explicit `--header` values were provided, and runtime precedence lets that ambient value override the serialized explicit header.
  - confidence: 0.9
