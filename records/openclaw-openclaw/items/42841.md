## Summary

Keep open: the PR is useful and current main still lacks node WebSocket header support, but the current branch still has credential-handling defects around env-sourced header persistence and explicit-header precedence.

## What This Changes

The PR adds repeatable `node run`/`node install` `--header` support, node header env handling, GatewayClient header forwarding, docs, changelog, and targeted tests.

## Best Possible Solution

Land a corrected node run/install header implementation that forwards headers through `ws`, keeps explicit flags authoritative, and separates runtime env secrets from persisted config/status.

## Reproduction Assessment

Yes. Current main has no node `--header` option and no GatewayClient custom-header path, and the two PR regressions are source-reproducible from the PR head without running the branch.

## Solution Assessment

No. Passing headers through the `ws` options is the right low-level mechanism, but the current patch needs to avoid env-secret persistence and prevent ambient CF env from overriding explicit install headers before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not persist env-sourced header secrets:** `src/node-host/runner.ts:209-214`
  - body: The env-var path is documented as keeping secrets out of config, but `resolvedHeaders` includes `OPENCLAW_NODE_HEADERS` and `CF_*` values and is saved into `node.json`. A service installed with env-only Cloudflare credentials will silently copy `CF-Access-Client-Secret` to disk, so runtime env headers need to stay separate from persisted config.
  - confidence: 0.88
- **[P2] Keep explicit install headers ahead of CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs from a shell with `CF_ACCESS_CLIENT_*` set, the install plan writes both `OPENCLAW_NODE_HEADERS` and the CF env vars. Runtime precedence then applies `CF_*` after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can override the explicit flags and the installed node can still fail behind Zero Trust.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces concrete credential-handling regressions for secret-bearing WebSocket headers.

Concerns:

- **[medium] Env header secrets are written to node config:** `src/node-host/runner.ts:209`
  - body: Headers resolved from `OPENCLAW_NODE_HEADERS` or `CF_ACCESS_CLIENT_SECRET` are saved into `node.json`, contradicting the documented env-var path for keeping secrets out of config and broadening where those credentials rest on disk.
  - confidence: 0.88
- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: The install plan copies `CF_ACCESS_CLIENT_SECRET` into the service environment even when explicit `--header` values were provided, and runtime precedence then lets that ambient secret override the serialized explicit header.
  - confidence: 0.9
