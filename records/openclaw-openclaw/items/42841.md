## Summary

Keep open: current main still lacks node WebSocket upgrade header support, and this PR remains the canonical implementation path for https://github.com/openclaw/openclaw/issues/42792. It is not merge-ready because the latest head still has secret-bearing install precedence/storage concerns and lacks real after-fix proof for the installed service path.

## What This Changes

The PR adds repeatable `--header` support for `openclaw node run/install`, node header env/config handling, GatewayClient WebSocket headers, daemon status redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land a rebased implementation that threads custom node headers through `GatewayClient`, defines explicit secret-safe precedence and persistence semantics, documents the contract accurately, redacts status surfaces, and proves the installed service restart path.

## Reproduction Assessment

Yes for the current limitation and PR defects: current main has no node `--header` option, node config header field, or `GatewayClient` header option, and PR-head source shows the install precedence and env persistence issues. I did not run a live Cloudflare or installed-service repro in this read-only review.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but install-time precedence and env/config secret-storage semantics need correction or maintainer approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Avoid persisting stale CF env with install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs with `CF_ACCESS_CLIENT_*` already exported, the install plan writes explicit headers into `OPENCLAW_NODE_HEADERS` but also copies the ambient CF vars into the service environment. On startup there are no CLI opts, so `resolveNodeHostHeaders` applies `CF_*` after the JSON headers and can reconnect with stale credentials instead of the values the user just installed.
  - confidence: 0.93
- **[P2] Align env header storage docs with runtime persistence:** `docs/cli/node.md:72`
  - body: The docs say env vars keep secrets out of config, but `runNodeHost` persists resolved headers into `config.gateway.headers` before saving `node.json`. Users who choose env vars for secret storage would still have those header values written to config, so either the code or the documented storage contract needs to change.
  - confidence: 0.86
