## Summary

Keep open: current main still lacks node custom WebSocket header support, but this PR still has concrete install-path precedence and secret-storage contract blockers plus insufficient real behavior proof for the installed service path.

## What This Changes

The PR adds repeatable node `--header` flags and header env support, persists node gateway headers, forwards them into `GatewayClient`, redacts daemon status env, and documents Cloudflare Zero Trust usage.

## Best Possible Solution

Land a rebased implementation that threads node custom headers through `GatewayClient`, defines explicit precedence and secret-storage semantics, covers run/install paths in tests, and includes real install-path proof.

## Reproduction Assessment

Yes for source-level behavior: current main lacks the node `--header` and `GatewayClient` header path, and PR head shows the remaining install precedence and config-persistence problems. I did not run a live proxy or service install because this review was read-only.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but the install-time env precedence and documented secret-storage behavior need correction before this is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not let CF env override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs while old `CF_ACCESS_CLIENT_*` vars are exported, the install plan writes both sources into the service environment. The installed daemon then has no CLI opts, so `CF_*` overrides the explicit `OPENCLAW_NODE_HEADERS` values and can keep using stale Cloudflare credentials.
  - confidence: 0.93
- **[P2] Align env secret docs with config persistence:** `docs/cli/node.md:72`
  - body: The docs say env vars keep header secrets out of config, but `runNodeHost` persists the resolved env-derived headers into `node.json` before connecting. Either stop saving env-derived headers or document that owner-only node config stores them.
  - confidence: 0.86

## Security Review
