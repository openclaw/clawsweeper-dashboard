## Summary

Keep open. Current main still lacks node WebSocket upgrade header support, and this PR is useful, but the current head still has blocking secret-handling defects around env-header persistence and install-time precedence.

## What This Changes

This PR adds repeatable `--header` support to `openclaw node run/install`, persists/propagates node gateway headers through config/env/service setup, forwards them to `GatewayClient`, and updates docs, changelog, and focused tests.

## Best Possible Solution

Land node header support after the secret-safe persistence and precedence contract is corrected, keeping env-only secrets out of node config and making explicit install headers win over ambient CF env.

## Reproduction Assessment

Yes for the current limitation: source inspection on current main shows no node `--header` option, no node header config field, and no `GatewayClient` header forwarding. I did not run the mock proxy, but the PR body includes copied before/after output for the 302-to-200 path.

## Solution Assessment

No as currently patched. Passing headers through `wsOptions.headers` is the right low-level hook, but env-only secrets should not be persisted into node config and ambient CF env should not override explicit install headers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep env-only headers out of node config:** `src/node-host/runner.ts:209-214`
  - body: `resolvedHeaders` includes values from `OPENCLAW_NODE_HEADERS` and `CF_ACCESS_CLIENT_SECRET`, then saves that merged object into `config.gateway.headers`. The new docs present env vars as the way to keep secrets out of config and command lines, so env-only service-token headers should be used for that run without being copied into `node.json`.
  - confidence: 0.9
- **[P2] Avoid overriding install headers with ambient CF env:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When explicit `node install --header` values are provided, this still copies exported `CF_ACCESS_CLIENT_*` values into the service environment next to `OPENCLAW_NODE_HEADERS`. The node-host resolver applies CF env after `OPENCLAW_NODE_HEADERS`, so stale shell credentials can replace the headers the user explicitly installed.
  - confidence: 0.9
