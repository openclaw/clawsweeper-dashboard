## Summary

Keep open: current main still lacks the node-specific custom header path, and this PR is a plausible implementation, but the latest head still has blocking secret-bearing install precedence and env/config storage-contract issues plus insufficient real install-path proof.

## What This Changes

The PR adds repeatable node `--header` parsing, node header env/config handling, GatewayClient WebSocket header forwarding, daemon status redaction, docs, changelog, and focused tests.

## Best Possible Solution

Land a rebased implementation that threads headers through `GatewayClient`, defines explicit secret-safe precedence and persistence semantics, redacts status surfaces, documents the true storage contract, and proves the installed service restart path.

## Reproduction Assessment

Yes, for the source-level limitation and PR defects: current main has no node header option/config/client seam, and PR head shows the install precedence and storage-contract mismatches. I did not run a live Cloudflare or installed-service reproduction in this read-only review.

## Solution Assessment

No as currently proposed. Forwarding headers through `GatewayClient` is the right seam, but the install-time precedence and env/config secret-storage semantics need correction or explicit maintainer approval before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not copy ambient CF vars with explicit headers:** `src/commands/node-daemon-install-helpers.ts:67-73`
  - body: When `node install --header ...` runs in a shell that already has `CF_ACCESS_CLIENT_*` exported, the service environment gets both the explicit headers in `OPENCLAW_NODE_HEADERS` and the ambient CF vars. On daemon startup `resolveNodeHostHeaders` applies CF vars after the JSON headers, so stale shell credentials can override the values the user explicitly installed.
  - confidence: 0.93
- **[P2] Align env header docs with config persistence:** `docs/cli/node.md:72`
  - body: The docs say env vars keep header secrets out of config, but `runNodeHost` persists resolved headers, including env-derived values, into `config.gateway.headers` before saving `node.json`. Users following the env-var guidance for secret storage would still have those values written to config, so either the runtime or the documented contract needs to change.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch adds a secret-bearing CLI/env/config path and still has concrete precedence and storage-contract concerns.

Concerns:

- **[medium] Ambient CF secret can override explicit install headers:** `src/commands/node-daemon-install-helpers.ts:67`
  - body: `buildNodeInstallPlan` copies `CF_ACCESS_CLIENT_SECRET` from the install shell even when explicit `--header` values are provided, and runtime precedence can then prefer that stale env secret over the just-installed header JSON.
