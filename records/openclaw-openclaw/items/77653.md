## Summary

Keep open: the PR targets a real Feishu bug with the right local normalization approach, but it leaves the default typing-indicator reaction API path using the synthetic ID and still needs the required user-facing changelog entry before merge.

## What This Changes

This PR adds a Feishu message-ID normalizer and applies it to reply, read, edit, card update, media reply, reaction, pin, and streaming-card API calls, with regression coverage for replies, lookups, and edits.

## Best Possible Solution

Keep the Feishu-local helper, apply it to every Feishu API boundary that accepts an Open Message ID including typing indicators, add focused regression coverage and a single-line Unreleased/Fixes changelog entry, and avoid moving this extension-owned behavior into core.

Yes, from source inspection: current main creates `:reaction:` synthetic message IDs and passes those IDs unmodified into Feishu `message_id` API fields. The linked issue also provides a concrete Feishu 400 error log, though I did not run a live Feishu account in this read-only review.

## Solution Assessment

No, not yet. The normalizer is the right narrow Feishu-plugin fix, but it should also cover typing indicator add/remove and include the required changelog before the PR is ready to merge.
