## Summary

Keep open: the PR targets a real memory status bug, but the proposed chunk-row fallback is too broad and can report vector storage as ready for chunks-only or degraded FTS indexes. It also still needs real CLI behavior proof before merge.

## What This Changes

The PR makes builtin memory status infer vector-store readiness from existing chunk rows and adds a regression test for reopening an indexed DB in status mode without initializing embeddings.

## Best Possible Solution

Use a vector-specific persisted readiness signal, gated by `vector.enabled`, keep semantic probing unchanged, add FTS-only/degraded regression coverage, and attach real CLI proof.

## Reproduction Assessment

Yes by source inspection. Plain status skips vector probing, current main renders undefined store availability as `unknown`, and the PR's fallback can be checked against the existing FTS-only chunk write path.

## Solution Assessment

No. The status fallback should use a vector-specific persisted signal, not any row in `chunks`, because chunks are also written for FTS-only and degraded indexes.
