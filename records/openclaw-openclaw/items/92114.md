## What This Changes

The PR changes memory-core status to report builtin vector-store readiness from persisted `vectorDims` metadata when vector search is enabled, with regressions for indexed, chunks-only, and disabled-vector states.

## Best Possible Solution

Land one canonical memory-core fix that uses `vectorDims` as the fast status signal, keeps `--deep` for live sqlite-vec probing, and closes the linked issue after merge.

## Reproduction Assessment

Yes. Source inspection shows plain `memory status` uses a status-purpose manager, skips vector probing unless deep/index is requested, and current main renders undefined `storeAvailable` as `Vector store: unknown`; I did not run a live failing CLI repro because this review is read-only.

## Solution Assessment

Yes, this is the best current fix shape: `vectorDims` is a vector-specific persisted signal, the branch rejects chunks-only and disabled-vector false positives, and `--deep` remains the live readiness probe. The older chunks-row fallback is broader and more likely to misreport FTS-only indexes.
