## Summary

Keep open: current main and the latest release still have the memory-status fast path that can render `Vector store: unknown`, and this PR is the open canonical fix candidate for the linked issue. The patch looks correct, but merging it changes plain CLI/JSON status semantics from live-probe-only to metadata-backed readiness, so maintainers should explicitly accept that `--deep` remains the live sqlite-vec probe.

## What This Changes

The PR changes memory-core status to report builtin vector-store readiness from persisted `vectorDims` metadata when vector search is enabled, and adds regressions for indexed, chunks-only, and disabled-vector states.

## Best Possible Solution

Land one canonical memory-core fix that uses `vectorDims` as the fast status signal, keeps `--deep` for live sqlite-vec probing, and closes the linked issue after merge.

## Reproduction Assessment

Yes by source inspection. Plain `memory status` uses a status-purpose manager, skips vector probing unless `--deep` or `--index` is requested, and current main renders undefined `storeAvailable` as `Vector store: unknown`.

## Solution Assessment

Yes, this is the best current fix shape if maintainers accept metadata-backed fast status. `vectorDims` is a vector-specific persisted signal, the branch rejects chunks-only and disabled-vector false positives, and `--deep` remains the live readiness probe.
