## Summary

Keep this PR open: the patch is a focused, well-proven candidate for the linked memory-status bug, but it still needs maintainer acceptance of metadata-backed fast-status semantics before merge.

## What This Changes

The PR updates memory-core builtin status to derive vector-store readiness from persisted `vectorDims` metadata and adds regressions for indexed, chunks-only, and disabled-vector states.

## Best Possible Solution

Land one canonical memory-core fix that uses `vectorDims` as the fast status signal, keeps `--deep` as the live sqlite-vec probe, and closes the linked issue after merge.

## Reproduction Assessment

Yes, as a source-level reproduction: current main uses a status-purpose manager, skips vector probing unless `--deep` or `--index` is used, and renders undefined `storeAvailable` as `Vector store: unknown`. I did not run a live failing CLI command in this read-only review.

## Solution Assessment

Yes, this is the best current fix shape if maintainers accept the status semantics: `vectorDims` is a vector-specific persisted signal, and the branch avoids the broader chunks-row false positive. The open decision is whether plain status may call that state ready without a live probe.
