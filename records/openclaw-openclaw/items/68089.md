## Summary

Keep open. Current main still wires the Mattermost slash-command callback route through the sync bundled-entry loader while activation runs through the native ESM monitor graph, so this PR is not obsolete and the affected `/api/channels/mattermost/command` behavior still has a plausible dual-instance failure path. The patch is a focused candidate, but maintainers should decide whether this static import is the final shape or whether to use a narrower shared-state/runtime seam with regression coverage.

## What This Changes

The PR changes `extensions/mattermost/index.ts` to statically import `registerSlashCommandRoute` from `./slash-route-api.js` instead of loading it with `loadBundledEntryExportSync`, with an explanatory comment about sharing the `slash-state` module instance.

## Best Possible Solution

Land one Mattermost-local fix that guarantees slash route registration and monitor activation share the same account state across supported loader paths, ideally with a focused dual-load regression test and an explicit decision on preserving the lightweight channel-entry boundary.

## Reproduction Assessment

Yes. The PR body and linked bug give concrete live reproduction steps, and current main source confirms that an empty route-side `accountStates` map returns the exact 503 response after activation logs could have populated a different module instance.

## Solution Assessment

Unclear. The PR is a narrow functional fix for module identity, but the static import may not be the best long-term shape if it pulls operational Mattermost runtime dependencies into the channel entry; a shared state anchor or thin runtime seam plus regression test may be safer.
