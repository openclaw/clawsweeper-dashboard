## Summary

Keep open. Current main still has the explicit `context1m` gate, so the requested behavior is not implemented, but this PR is not safe to merge as written because it changes session/context planning without updating the Anthropic request beta-header path that makes 1M context real.

## What This Changes

The PR changes `src/agents/context.ts` and `src/agents/context.test.ts` so Anthropic or Claude CLI Opus/Sonnet 4 models resolve to 1,048,576 context tokens by default unless `params.context1m` is explicitly false.

## Best Possible Solution

Land a coordinated Anthropic context-window fix that keeps resolver output, request beta injection, OAuth/legacy-token behavior, discovered/configured context windows, session/status limits, compaction thresholds, tests, and provider docs in agreement. If no-flag 1M is the product direction for eligible API-key traffic, the provider wrapper should request the 1M beta on that same path and the fallback behavior should be explicit for auth modes that cannot use it.

## Reproduction Assessment

Yes. The current-main gap is statically reproducible from `src/agents/context.test.ts`, which expects no-flag `anthropic/claude-opus-4-6` to resolve to `200_000`; the PR regression is also reproducible by following the no-flag path through `resolveAnthropicBetas()`, which returns no 1M beta header.

## Solution Assessment

No. A resolver-only default is not the best fix because it changes planning/status behavior without proving the request path has 1M capability; the safer solution is to align the resolver with Anthropic beta-header injection and auth-specific fallback behavior in the same change.
