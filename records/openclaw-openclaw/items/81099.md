## Summary

Keep open. Current main still treats `claude-cli` `AskUserQuestion` as an ordinary tool record: the Claude stream parser ignores non-text/non-result tool-use events, live session stdin only sends normal user messages, and Control UI renders generic input/output cards with no picker or submit/result path.

## What This Changes

Render and answer Claude CLI `AskUserQuestion` tool calls in Control UI instead of showing inert input and stalling the turn.

## Best Possible Solution

Define a small generic HITL tool-result contract for Claude CLI sessions, parse `AskUserQuestion`, render a Control UI picker, submit selected answers back to the live turn, and cover it with parser/UI tests plus live Claude CLI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a Control UI session backed by `claude-cli` and cause Claude Code to emit `AskUserQuestion`; current main has no parser, picker, or result-submission branch for that tool.

## Solution Assessment

No as a narrow automated UI patch. The maintainable fix needs a structured cross-boundary HITL result path, not only a different rendering of the existing tool card.
