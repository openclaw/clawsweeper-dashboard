## Summary

Keep open: current main can render the Claude `AskUserQuestion` payload as a generic tool card, but I found no picker UI or result-submission path for answering it. This is distinct from the prior skill-level ClawHub request and is not solved by the adjacent Claude control-request or tool-preview PRs.

## What This Changes

Render and answer Claude CLI `AskUserQuestion` prompts in the Control UI instead of showing only raw tool input.

## Best Possible Solution

Add a narrow Claude CLI user-input bridge that recognizes `AskUserQuestion`, renders a Control UI single/multi-select picker, persists the pending prompt, and sends the selected answer back through the correct Claude CLI control/defer/result channel with regression coverage.

## Reproduction Assessment

Yes, source-level. Current main normalizes Claude `tool_use` input into a generic tool card and has no parser, UI, or stdin/result path that could turn an `AskUserQuestion` payload into an answered Claude tool result.

## Solution Assessment

Mostly yes. The requested picker-and-result flow is the right user outcome, but the implementation should reuse the Claude CLI control/defer and existing Control UI streaming boundaries rather than only adding display logic.
