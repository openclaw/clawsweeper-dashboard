## Summary

Keep open: current main fixed part of the old unconditional direct-restart path, but the canonical browser-profile problem remains because the bundled browser plugin still classifies all `browser.*` changes as restart-required and the reload planner follows that broad rule.

## What This Changes

Request `config.patch` and reload planning to avoid SIGUSR1/full gateway restarts for hot-reloadable browser profile changes such as `browser.profiles.*.cdpUrl`.

## Best Possible Solution

Add granular browser reload metadata so safe `browser.profiles.*` and likely `browser.defaultProfile` changes avoid gateway restarts, preserve restart behavior for browser service settings, and cover the matrix in reload-plan plus config.patch tests.

## Reproduction Assessment

Yes, source-level: `browser.profiles.sandbox.cdpUrl` matches the browser plugin’s broad `browser` restart prefix, `buildGatewayReloadPlan()` marks it restart-required, and config.patch/reloader paths still turn that plan into a restart. I did not execute a live gateway run in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to narrow the browser plugin’s reload metadata/rules rather than hard-code browser prefixes in the generic config.patch handler. The existing open PR should be reviewed or split if it implements that boundary cleanly.
