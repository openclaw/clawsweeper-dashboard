## Summary

Keep open. Current main no longer unconditionally restarts for every config.patch, but browser profile changes still match the bundled browser plugin's broad browser restart prefix, so the hot-mode SIGUSR1 path remains source-reproducible; open PR #46405 is the related implementation candidate.

## What This Changes

config.patch should avoid SIGUSR1 restarts for hot-reloadable browser profile changes such as browser.profiles.<name>.cdpUrl.

## Best Possible Solution

Make browser-owned reload metadata granular so profile-only edits are no-op or hot-reloadable while structural browser control changes still require restart, then add regression coverage for the hot-mode config.patch path.

## Reproduction Assessment

Yes. A source-level reproduction is a config.patch changing browser.profiles.<name>.cdpUrl while gateway.reload.mode is hot; current main classifies that path through the browser restart prefix and the direct SIGUSR1 scheduler remains reachable.

## Solution Assessment

Yes on the requested direction, but the maintainable fix should be browser-owned granular reload metadata or a specificity-aware reload plan rather than a hard-coded core allowlist.
