## Summary

Keep open: current main still classifies `browser.profiles.*` under the browser plugin’s broad restart prefix, and the linked open PR is the right implementation candidate rather than closing the issue early.

## What This Changes

The issue asks for `config.patch` browser profile changes to avoid scheduling a SIGUSR1/full gateway restart when the reload planner can hot-reload those paths.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84410; if it lands, close this issue as implemented by granular browser profile reload metadata and planner coverage.

## Reproduction Assessment

Yes, source-level. On current main, `browser.profiles.<name>.*` matches the broad `browser` restart prefix, `buildGatewayReloadPlan()` sets `restartGateway`, and the direct config-write flow can schedule SIGUSR1 from that restart-required plan.

## Solution Assessment

Yes on direction, but the best path is to review the active linked PR rather than queue duplicate repair work. The maintainable fix belongs in granular browser plugin reload metadata plus planner tests, not a hard-coded allowlist inside the RPC handler.
