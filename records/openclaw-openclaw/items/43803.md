## Summary

Keep open: current main and the latest release still classify `browser.profiles.*` through the browser plugin's broad restart rule, so the hot-mode `config.patch` path can still schedule SIGUSR1 instead of treating profile-only browser config as reloadable. The two closest fix PRs are closed unmerged and live search found no open replacement PR owning the remaining work.

## What This Changes

`config.patch` requests that change browser profile config should not send SIGUSR1 when the reload planner can handle those profile-only paths without restarting the gateway.

## Best Possible Solution

Land a narrow browser-plugin reload metadata fix that classifies `browser.profiles.*` as hot while preserving restart behavior for `browser.defaultProfile` and service-level browser settings.

## Reproduction Assessment

Yes, at source level: `browser.profiles.sandbox.cdpUrl` matches the broad browser restart prefix, and hot-mode config writes schedule SIGUSR1 when the reload plan says `restartGateway`. I did not execute a live gateway repro in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned browser reload metadata plus planner/browser regression tests; a core `BASE_RELOAD_RULES` override would fix the symptom mechanically but is a weaker owner-boundary fit.
