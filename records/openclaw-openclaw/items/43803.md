## Summary

Keep open. Current main and the latest release still classify `browser.profiles.*` under the browser plugin's broad restart prefix, so a profile-only `config.patch` can still reach the SIGUSR1 restart path instead of staying hot/no-op reloadable.

## What This Changes

The issue requests that `config.patch` avoid scheduling gateway restarts for browser profile paths that the browser runtime can refresh without a full process restart.

## Best Possible Solution

Make the browser plugin reload metadata granular so browser profile and default-profile changes avoid gateway restarts while structural browser service settings still restart, then add focused reload-plan and `config.patch` regression coverage.

## Reproduction Assessment

Yes, source-level. `browser.profiles.<name>.cdpUrl` still matches the broad `browser` restart prefix, `buildGatewayReloadPlan` marks it restart-required, and `config.patch` can schedule SIGUSR1 when hot mode sees that restart-required plan.

## Solution Assessment

Yes on direction, with one adjustment: the maintainable fix should use plugin-owned granular reload metadata through the existing `restartPrefixes`/`hotPrefixes`/`noopPrefixes` contract, not a hard-coded core allowlist for browser profile paths.
