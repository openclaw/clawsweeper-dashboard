## Summary

Keep open: current main and the latest release still route `browser.profiles.*` through the browser plugin's broad restart rule, so `config.patch` can still restart the gateway instead of treating profile-only browser config as hot-reloadable. The useful related fix PRs are closed unmerged, and live search found no open replacement PR owning the remaining work.

## What This Changes

Reports that `config.patch` of `browser.profiles.*` still schedules a SIGUSR1 gateway restart even though profile-only browser changes should be handled by hot reload.

## Best Possible Solution

Land a narrow browser-plugin reload metadata fix that classifies `browser.profiles.*` as hot while preserving restart behavior for `browser.defaultProfile` and service-level browser settings.

## Reproduction Assessment

Yes. From source inspection, `browser.profiles.sandbox.cdpUrl` matches the broad browser restart prefix, and current config write/reload paths can schedule a restart from that plan.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned reload metadata plus planner/browser tests; changing only core base rules is mechanically possible but a weaker owner-boundary fit.
