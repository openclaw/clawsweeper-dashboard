## Summary

Keep open. Current main still lets a browser-profile config.patch reach the direct SIGUSR1 path because browser config is registered as a broad restart prefix, even though browser profile resolution has hot-refresh coverage.

## What This Changes

The issue asks for config.patch to skip SIGUSR1 restarts when changed paths such as browser.profiles.* can be handled by hot reload.

## Best Possible Solution

Make the browser plugin reload metadata granular so browser.profiles.* and likely browser.defaultProfile avoid gateway restarts while structural browser service settings still restart, then add focused reload-plan/config.patch coverage and align the browser docs.

## Reproduction Assessment

Yes, at source level. On current main, browser.profiles.<name>.cdpUrl matches the broad browser restart prefix, buildGatewayReloadPlan marks it restartGateway, and hot-mode config.patch can still schedule SIGUSR1.

## Solution Assessment

Yes on direction, but the maintainable fix should be plugin-owned granular reload metadata rather than a hard-coded core allowlist for browser profile paths.
