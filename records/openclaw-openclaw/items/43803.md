## Summary

Keep open: current main and the latest release still classify `browser.profiles.*` through the browser plugin's broad restart rule, so a hot-mode `config.patch` can still schedule SIGUSR1. The related fix PRs are closed unmerged, and live search did not find an open replacement owning the work.

## What This Changes

The issue asks for `config.patch` to skip SIGUSR1/full gateway restart when changes only touch hot-reloadable `browser.profiles.*` config paths.

## Best Possible Solution

Land a narrow browser-plugin reload metadata and regression-test fix that classifies `browser.profiles.*` as hot while preserving restart behavior for `browser.defaultProfile` and service-level browser settings.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `browser.profiles.*` matches the broad browser plugin restart prefix, and the config write path schedules SIGUSR1 when the reload plan says restart. I did not run a live gateway process in this read-only pass.

## Solution Assessment

Yes, the best fix is a narrow plugin-owned reload metadata change plus planner tests. A core `BASE_RELOAD_RULES` override can work mechanically, but it is a less clean owner-boundary fit than changing the browser plugin's reload declaration.
