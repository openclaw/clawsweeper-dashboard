## Summary

Keep open: current main and latest release still classify browser profile config changes under the browser plugin's broad restart prefix, and the open closing PR at https://github.com/openclaw/openclaw/pull/84410 already owns the targeted fix.

## What This Changes

The issue asks for `config.patch` changes under `browser.profiles.*` to avoid scheduling a SIGUSR1 gateway restart when the reload planner can handle the change as hot reload.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84410; if it lands, close this issue as implemented by granular browser profile reload metadata and planner regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `browser.profiles.<name>.*` matches the browser plugin's broad `browser` restart prefix, and `config.patch` restart scheduling is derived from `buildGatewayReloadPlan(...).restartGateway`.

## Solution Assessment

Yes on direction. The maintainable fix is to make the browser plugin reload metadata granular and cover it in planner tests, as the open linked PR does, rather than adding a hard-coded allowlist inside the RPC handler.
