## Summary

Keep open: current `main` and the latest release still classify `browser.profiles.*` through the browser plugin's broad restart rule, so a hot-mode `config.patch` can still schedule SIGUSR1. The related fix PRs are closed unmerged, and live search found no open replacement owning the work.

## What This Changes

The issue reports that `config.patch` restarts the gateway for hot-reloadable `browser.profiles.*` changes instead of using reload planning.

## Best Possible Solution

Land a narrow browser-plugin reload metadata and regression-test fix that classifies `browser.profiles.*` as hot while preserving restart behavior for `browser.defaultProfile` and service-level browser settings.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: `browser.profiles.*` matches the broad browser plugin restart prefix, and the config write path schedules SIGUSR1 when the reload plan says restart.

## Solution Assessment

Yes. The best fix is a narrow plugin-owned reload metadata change plus planner tests; a core `BASE_RELOAD_RULES` override can work mechanically, but it is a weaker owner-boundary fit.
