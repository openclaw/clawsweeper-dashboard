## Summary

Keep open. Current main still routes `browser.profiles.*` through the browser plugin's broad restart prefix, so hot-mode `config.patch` can still schedule SIGUSR1; the focused related PR attempts are closed and no merged/current fix owns the remaining behavior.

## What This Changes

`config.patch` should avoid a direct SIGUSR1 restart when only hot-reloadable `browser.profiles.*` paths change.

## Best Possible Solution

Make the browser plugin reload metadata granular so profile/default-profile changes avoid gateway restarts while structural browser service settings still restart, then add focused reload-plan and `config.patch` regression coverage.

## Reproduction Assessment

Yes, at source level. `browser.profiles.<name>.cdpUrl` still matches the broad `browser` restart prefix, `buildGatewayReloadPlan` marks it restart-required, and hot-mode `config.patch` can schedule SIGUSR1 from that plan.

## Solution Assessment

Yes on direction, but the maintainable fix should be plugin-owned granular reload metadata rather than a hard-coded core allowlist for browser profile paths.
