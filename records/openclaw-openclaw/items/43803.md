## Summary

Keep open: current main and the latest release still classify `browser.profiles.*` through the browser plugin's broad restart rule, so the reported `config.patch` restart path remains source-reproducible. The novita-sandbox ClawHub workflow is a concrete use case, but the missing behavior is in OpenClaw's bundled browser/gateway reload boundary; the focused linked fix PRs are closed unmerged rather than canonical merged work.

## What This Changes

`config.patch` should avoid scheduling a SIGUSR1 gateway restart when the changed paths are hot-reloadable browser profile config such as `browser.profiles.<name>.cdpUrl`.

## Best Possible Solution

Land a narrow plugin-owned reload metadata fix: make `browser.profiles` hot-reloadable while keeping service-level `browser.*` and `browser.defaultProfile` restart-backed until the default-profile selection order is fixed.

## Reproduction Assessment

Yes for a source-level repro: `config.patch` feeds changed paths into the direct restart gate, and current browser reload metadata classifies `browser.profiles.*` through the broad `browser` restart rule. I did not run a live gateway process in this read-only review.

## Solution Assessment

Yes, the narrow plugin-owned metadata shape from https://github.com/openclaw/openclaw/pull/84410 is the best fix direction. A core `BASE_RELOAD_RULES` override like https://github.com/openclaw/openclaw/pull/92435 can work mechanically, but it is a weaker owner-boundary fit because browser reload policy already belongs to the browser plugin metadata.
