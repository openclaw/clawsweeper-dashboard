## Summary

Keep open: current main and the latest release still classify `browser.profiles.*` under the broad browser restart rule, and the open closing PR is the active fix candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that `config.patch` changes to hot-reloadable browser profile paths still trigger a SIGUSR1 gateway restart instead of staying on the hot-reload path.

## Best Possible Solution

Review and either land or explicitly reject https://github.com/openclaw/openclaw/pull/93827, preserving restart behavior for service-level browser settings while making `browser.profiles.*` hot-reloadable with config.patch coverage.

## Reproduction Assessment

Yes from source inspection: current main registers only the broad `browser` restart prefix, the reload planner returns that restart rule for `browser.profiles.*`, and config writes schedule SIGUSR1 in hot mode when the plan requires a gateway restart. I did not run a live gateway repro in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to classify `browser.profiles.*` as hot while preserving restart classification for broader browser settings and adding config.patch regression coverage. The open closing PR is already attempting that path, so the issue should stay open until it merges or maintainers reject it.
