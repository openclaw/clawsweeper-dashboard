## Summary

Keep open: current main still preserves the reported browser-profile restart path. `config.patch` now routes direct restarts through the reload planner, but `browser.profiles.*` still falls under the browser plugin's broad `browser` restart prefix, so hot mode can still schedule SIGUSR1 for the reported profile-only change.

## What This Changes

The issue asks for `config.patch` to skip SIGUSR1 restarts when all changed paths, especially `browser.profiles.*`, can be handled by hot reload.

## Best Possible Solution

Land a focused gateway/browser reload fix, either by splitting/reviewing the gateway portion of #46405 or by adding plugin-owned granular reload metadata. `browser.profiles.*` and likely `browser.defaultProfile` should no longer require a full gateway restart, structural browser settings should remain restart-required, and regression tests plus browser docs should be updated together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `config.patch` that changes `browser.profiles.<name>.cdpUrl` while `gateway.reload.mode` is `hot`: current main classifies that changed path under the broad `browser` restart rule, so the direct restart scheduler remains reachable.

## Solution Assessment

Yes in direction, but the best implementation is narrower than a hard-coded core prefix list. The maintainable path is plugin-owned granular reload metadata or specificity-aware reload planning so browser profile writes are hot/no-op while browser service-affecting keys still restart.
