## Summary

Keep open. Current main and the latest release still classify `browser.profiles.*` under the browser plugin's broad `browser` restart prefix, so hot-mode `config.patch` can still schedule SIGUSR1; the linked `novita-sandbox` skill is useful real-world context, but this remains a core/browser-plugin reload metadata bug rather than ClawHub-only work.

## What This Changes

The issue asks `config.patch` to avoid scheduling SIGUSR1 for hot-reloadable browser profile changes such as `browser.profiles.<name>.cdpUrl`.

## Best Possible Solution

Make the browser plugin reload metadata granular so profile/default-profile changes avoid gateway restarts while structural browser service settings still restart, then add focused reload-plan and `config.patch` regression coverage.

## Reproduction Assessment

Yes, source-level. `browser.profiles.<name>.cdpUrl` still matches the broad `browser` restart prefix, `buildGatewayReloadPlan` marks it restart-required, and hot-mode `config.patch` can schedule SIGUSR1 from that plan.

## Solution Assessment

Yes on direction, but the maintainable fix should be plugin-owned granular reload metadata using the existing `hotPrefixes`/`noopPrefixes` contract rather than a hard-coded core allowlist for browser profile paths.
