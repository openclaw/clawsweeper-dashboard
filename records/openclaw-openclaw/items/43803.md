## Summary

Keep open. Current main no longer has the old unconditional restart for every patch, but `browser.profiles.*` still falls under the bundled browser plugin's broad restart prefix, so a hot-mode `config.patch` can still reach direct SIGUSR1 scheduling; open PR #46405 is the existing implementation candidate.

## What This Changes

The issue asks for `config.patch` to skip SIGUSR1 restarts when all changed paths are hot-reloadable, especially `browser.profiles.*` updates used for sandbox CDP profiles.

## Best Possible Solution

Make browser-owned reload metadata granular enough that profile-only edits are no-op or hot-reloadable while structural browser-control changes still restart, then cover hot-mode `config.patch` on `browser.profiles.*` with a regression test.

## Reproduction Assessment

Yes, at source level. A current-main `config.patch` changing `browser.profiles.<name>.cdpUrl` with `gateway.reload.mode` set to `hot` flows through the reload planner; the broad browser restart prefix sets `restartGateway`, making direct SIGUSR1 scheduling reachable.

## Solution Assessment

Yes on the requested behavior. The maintainable fix should be browser/plugin-owned granular reload metadata or specificity-aware reload planning, not a hard-coded core allowlist for browser paths.
