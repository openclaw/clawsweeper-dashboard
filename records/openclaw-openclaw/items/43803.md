## Summary

Keep open. Current main no longer schedules every config.patch restart unconditionally, but browser profile patches still match the browser plugin's broad restart prefix, so a hot-mode config.patch for browser.profiles.<name>.cdpUrl can still reach the direct SIGUSR1 scheduler.

## What This Changes

The issue asks for config.patch to avoid SIGUSR1 restarts when all changed paths, especially browser.profiles.*, are hot-reloadable.

## Best Possible Solution

Make the browser plugin's reload metadata granular so browser.profiles.* and likely browser.defaultProfile avoid gateway restarts while structural browser service settings still restart, then add focused reload-plan/config.patch coverage and align the browser docs.

## Reproduction Assessment

Yes, at source level. On current main, a config.patch that changes browser.profiles.<name>.cdpUrl with gateway.reload.mode set to hot is classified by the broad browser restart prefix, so the direct SIGUSR1 scheduler remains reachable.

## Solution Assessment

Yes on direction, but the maintainable fix should be plugin-owned granular reload metadata rather than a hard-coded core allowlist for browser profile paths.
