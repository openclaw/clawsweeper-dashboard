## Summary

Keep open. Current main removed unconditional `config.patch` restarts, but `browser.profiles.*` still falls under the browser plugin's broad `browser` restart prefix, so hot-mode `config.patch` can still schedule SIGUSR1 for the reported profile-only change; open PR #46405 is related but unmerged.

## What This Changes

The issue asks for `config.patch` to skip SIGUSR1/full gateway restart when all changed paths are browser-profile fields that can be handled by hot reload.

## Best Possible Solution

Update the browser plugin reload metadata or planner specificity so `browser.profiles.*` profile-only edits avoid gateway restart while structural browser settings remain restart-required, with regression coverage for `config.patch` in hot mode.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `config.patch` that changes `browser.profiles.<name>.cdpUrl` while `gateway.reload.mode` is `hot`; current main classifies that changed path under the broad browser restart rule and keeps the direct SIGUSR1 scheduler reachable.

## Solution Assessment

Yes on direction, but not as a hard-coded core prefix list. The maintainable fix is plugin-owned granular reload/no-op metadata or specificity-aware reload planning for browser profile fields.
