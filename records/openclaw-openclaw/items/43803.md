## Summary

Keep open. Current main removed the earlier unconditional `config.patch` restart behavior, but `browser.profiles.*` still matches the bundled browser plugin's broad `browser` restart prefix, so a hot-mode profile-only patch can still reach the direct SIGUSR1 scheduler; related PR #46405 is open and unmerged.

## What This Changes

The issue asks for `config.patch` to avoid SIGUSR1 restarts when changed paths such as `browser.profiles.*` are already hot-reloadable.

## Best Possible Solution

Make browser-owned reload metadata granular enough that profile-only edits are no-op or hot-reloadable while structural browser control changes still require a restart, then cover the hot-mode `config.patch` path with a regression test.

## Reproduction Assessment

Yes. A source-level reproduction is a `config.patch` changing `browser.profiles.<name>.cdpUrl` while `gateway.reload.mode` is `hot`; current main classifies that path through the broad browser restart prefix, so the direct SIGUSR1 scheduler remains reachable.

## Solution Assessment

Yes on the requested direction, but the maintainable fix should be plugin-owned granular reload metadata or specificity-aware reload planning rather than a hard-coded core prefix allowlist.
