## Summary

Keep open. Current `main` and the latest release still route `browser.profiles.*` through the browser plugin's broad `browser` restart prefix, so a profile-only `config.patch` can still be classified as restart-required and reach the direct SIGUSR1 scheduling path.

## What This Changes

Allow `config.patch` changes under browser profile config, such as `browser.profiles.<name>.cdpUrl`, to hot-reload without forcing a gateway restart.

## Best Possible Solution

Make the browser plugin's reload metadata granular so profile/default-profile changes avoid gateway restarts while structural browser service settings still restart, then update focused tests and the browser docs wording.

## Reproduction Assessment

Yes, source-level. `browser.profiles.<name>.cdpUrl` still matches the broad `browser` restart prefix, the planner marks that as `restartGateway`, and hot-mode `config.patch` schedules SIGUSR1 when the plan says a restart is required.

## Solution Assessment

Yes on direction. The maintainable fix is plugin-owned granular reload metadata, with rule-order coverage, rather than a hard-coded browser allowlist in the config.patch write flow.
