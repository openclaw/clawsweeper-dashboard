## Summary

Keep open. Current main still lets a `config.patch` that changes `browser.profiles.*` reach the direct SIGUSR1 scheduler in hot reload mode because browser plugin reload metadata still classifies the whole `browser` subtree as restart-required; related PR #46405 is open and unmerged.

## What This Changes

The issue asks for `config.patch` to avoid SIGUSR1 restarts when changed paths such as `browser.profiles.*` are hot-reloadable.

## Best Possible Solution

Make browser-owned reload metadata granular enough that profile-only edits are no-op or hot-reloadable while structural browser control changes still restart, then cover the hot-mode `config.patch` path with regression tests.

## Reproduction Assessment

Yes, at source level. A `config.patch` that changes `browser.profiles.<name>.cdpUrl` while `gateway.reload.mode` is `hot` matches the browser plugin's broad `browser` restart prefix, so `plan.restartGateway` remains true and direct SIGUSR1 scheduling is reachable.

## Solution Assessment

Yes on the requested direction, but the maintainable fix should be plugin-owned granular reload metadata or equivalent specificity-aware reload planning rather than a hard-coded core allowlist.
