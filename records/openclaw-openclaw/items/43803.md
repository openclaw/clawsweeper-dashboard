## Summary

Keep open. Current main and the latest release still register all `browser.*` config as restart-required, so `browser.profiles.<name>.cdpUrl` is not treated as a hot/no-op browser profile update and can still drive a gateway restart path; the linked sandbox skill is useful real-world context, but this remains a core/browser-plugin reload metadata bug.

## What This Changes

The issue requests that `config.patch` avoid scheduling a SIGUSR1 gateway restart for hot-reloadable browser profile config changes such as `browser.profiles.<name>.cdpUrl`.

## Best Possible Solution

Make the browser plugin reload metadata granular so browser profile/default-profile changes avoid gateway restarts while structural browser service settings still restart, then add focused reload-plan and `config.patch` regression coverage.

## Reproduction Assessment

Yes, source-level. `browser.profiles.<name>.cdpUrl` still matches the broad browser restart prefix, `buildGatewayReloadPlan` marks that path restart-required, and `config.patch` can schedule SIGUSR1 when hot mode sees a restart-required plan.

## Solution Assessment

Yes on direction, with one adjustment: the maintainable fix should be plugin-owned granular reload metadata using the existing `restartPrefixes`/`hotPrefixes`/`noopPrefixes` contract, not a hard-coded core allowlist for browser profile paths.
