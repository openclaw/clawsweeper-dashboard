## Summary

Keep open. Current main no longer unconditionally restarts for generic hot/no-op `config.patch`, but the reported `browser.profiles.*` path is still covered by the browser plugin's broad restart prefix, so `gateway.reload.mode="hot"` can still reach the direct SIGUSR1 scheduler. The narrower duplicate at https://github.com/openclaw/openclaw/issues/64204 was closed to this issue, and the related implementation PR at https://github.com/openclaw/openclaw/pull/46405 is still open and unmerged.

## What This Changes

`config.patch` should avoid queuing SIGUSR1 for browser profile config changes that the browser runtime can pick up without a full gateway restart.

## Best Possible Solution

Make browser-owned reload metadata granular so `browser.profiles.*` and likely `browser.defaultProfile` avoid gateway restart while structural browser control settings still require restart, with focused reload-plan, config.patch, and browser documentation coverage.

## Reproduction Assessment

Yes. A source-level reproduction is a `config.patch` that changes `browser.profiles.<name>.cdpUrl` with `gateway.reload.mode` set to `hot`; current main classifies that path through the broad browser restart prefix, so the direct SIGUSR1 scheduler remains reachable.

## Solution Assessment

Yes on the requested direction, but the maintainable fix should be plugin-owned granular reload metadata and specificity-aware tests rather than a hard-coded core allowlist.
