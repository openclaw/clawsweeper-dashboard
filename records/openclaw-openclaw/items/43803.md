## Summary

This issue is still necessary: current `main` still classifies `browser.profiles.*` through the browser plugin's broad `browser` restart prefix, and the linked fix PRs are closed unmerged rather than implemented on `main`.

## What This Changes

The issue requests that `config.patch` avoid SIGUSR1/full gateway restart for hot-reloadable `browser.profiles.*` changes so browser sandbox/profile updates do not disrupt sessions.

## Best Possible Solution

Land a narrow browser-owned reload metadata fix that makes `browser.profiles.<name>.*` hot-reloadable while leaving service-level browser config restart-required, with regression coverage for the planner and config write path.

## Reproduction Assessment

Yes, at source level: current `main` maps `browser.profiles.*` through the browser plugin's broad `browser` restart prefix, and both direct hot-mode scheduling and default hybrid watcher reload use `plan.restartGateway`. I did not run a live gateway in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow plugin-owned reload metadata change for `browser.profiles.*`, not a new config option or a broad core no-op rule for all browser config.
