## Summary

Keep open. Current main has fixed the broader unconditional `config.patch` restart bug, but this browser-profile case still falls through the browser plugin's broad restart prefix, so `browser.profiles.*` remains classified as gateway-restart-required and can still reach the direct SIGUSR1 path in hot mode.

## What This Changes

The issue asks for `config.patch` to skip SIGUSR1/full gateway restart when patching browser profile paths that the browser runtime can refresh without a process restart.

## Best Possible Solution

Make browser-owned reload metadata granular so `browser.profiles.*` and likely `browser.defaultProfile` avoid gateway restart while structural browser service settings still require restart, then cover the path with focused reload-plan/config.patch tests and docs updates.

## Reproduction Assessment

Yes. A source-level reproduction is a `config.patch` that changes `browser.profiles.<name>.cdpUrl` with `gateway.reload.mode` set to `hot`: current main classifies that path through the broad browser restart prefix, so the direct SIGUSR1 scheduler remains reachable.

## Solution Assessment

Yes on direction, but the maintainable fix should be plugin-owned granular reload metadata and specificity-aware tests rather than a hard-coded core allowlist.
