## Summary

Keep open. Current `main` still routes `browser.profiles.*` through the browser plugin's broad restart prefix, so a hot-mode `config.patch` can still reach the direct SIGUSR1 restart path; the same remaining work is also visible in the still-open related implementation PR at https://github.com/openclaw/openclaw/pull/46405.

## What This Changes

The issue asks for `config.patch` to avoid scheduling SIGUSR1 when changed paths are hot-reloadable browser profile settings.

## Best Possible Solution

Make browser-owned reload metadata granular so `browser.profiles.*` and `browser.defaultProfile` avoid gateway restart while structural browser-control settings still require restart, then update focused gateway/browser tests and browser docs.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `config.patch` with `gateway.reload.mode: "hot"` and a changed `browser.profiles.<name>.cdpUrl` still reaches the direct SIGUSR1 scheduler because the browser plugin marks the whole `browser` prefix as restart-required.

## Solution Assessment

Yes on the requested direction. The narrow maintainable fix is plugin-owned granular reload metadata with specificity-aware regression coverage, not a hard-coded core allowlist.
