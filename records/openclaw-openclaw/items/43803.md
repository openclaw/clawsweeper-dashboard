## Label Justifications

- `P2`: This is a confirmed, bounded gateway stability bug that disrupts browser-profile configuration workflows without making the core runtime broadly unusable.
- `impact:session-state`: The unnecessary full gateway restart can terminate or disrupt active agent sessions.
- `impact:crash-loop`: Foreground gateways exit on the scheduled restart, and repeated profile writes can produce restart-loop behavior.

## Summary

Current `main` and `v2026.6.5` still register all `browser.*` changes as restart-required, while the config write flow schedules SIGUSR1 directly from that planner result. The focused linked fix was closed unmerged for inactivity on June 10, 2026, so the bug remains actionable and should stay open for a replacement focused fix.

## What This Changes

Make `config.patch` avoid a full gateway restart when only hot-reloadable `browser.profiles.*` settings change.

## Best Possible Solution

Change the browser plugin reload metadata to classify only `browser.profiles.*` as hot, retain restart behavior for `browser.defaultProfile` and other service-level browser settings, and add plugin-registration plus planner tests for hot-only, restart-only, and mixed changes.

## Reproduction Assessment

Yes at source level with high confidence: `browser.profiles.*` matches the current broad browser restart prefix, and the config write flow schedules SIGUSR1 from that planner result. The linked PR also supplied focused planner output for the before/after classification, though this review did not run a live gateway reproduction.

## Solution Assessment

Yes, the scoped plugin metadata fix is the best approach because it corrects the owner-provided reload contract consumed by all config write paths instead of adding browser-specific logic to the generic RPC handler.
