## Summary

Keep open. Current main still classifies `browser.profiles.*` through the browser plugin's broad restart prefix, so a hot-mode `config.patch` can still reach the direct SIGUSR1 scheduler; the narrower duplicate was closed to this issue and the related implementation PR remains open and unmerged.

## What This Changes

Request that `config.patch` avoid SIGUSR1 restarts for browser profile changes that the browser runtime can already refresh without a full gateway restart.

## Best Possible Solution

Make browser-owned reload metadata granular so `browser.profiles.*` and `browser.defaultProfile` are no-op or hot-reloadable while structural browser control settings still require restart, then cover the hot-mode `config.patch` path and browser docs with focused regression updates.

## Reproduction Assessment

Yes. A source-level reproduction is a `config.patch` changing `browser.profiles.<name>.cdpUrl` with `gateway.reload.mode` set to `hot`; current main classifies that path through the broad browser restart prefix, so the direct SIGUSR1 scheduler remains reachable.

## Solution Assessment

Yes on the requested direction, but the maintainable fix should be plugin-owned granular reload metadata and specificity-aware tests rather than a hard-coded core allowlist.
