## Summary

Keep open: current main and the latest release still classify `browser.profiles.*` through the browser plugin's broad restart rule, so hot-mode `config.patch` can still schedule a SIGUSR1 restart. The related fix PRs are closed unmerged and live search found no open replacement owning the work.

## What This Changes

The issue requests that `config.patch` stop sending SIGUSR1 for hot-reloadable `browser.profiles.*` changes.

## Best Possible Solution

Land a narrow browser-plugin reload metadata fix that classifies `browser.profiles.*` as hot while preserving restart behavior for `browser.defaultProfile` and service-level browser settings.

## Reproduction Assessment

Yes, from source inspection rather than a live run. `browser.profiles.sandbox.cdpUrl` matches the broad browser plugin restart prefix, and hot-mode `config.patch` schedules SIGUSR1 when the reload plan says `restartGateway`.

## Solution Assessment

Yes. The narrowest maintainable fix is plugin-owned reload metadata plus planner tests; a core `BASE_RELOAD_RULES` override can work mechanically but is a weaker owner-boundary fit.
