## Summary

Keep open. Current main and v2026.5.12 still classify `browser.profiles.*` through the browser plugin's broad `browser` restart prefix, so the linked novita-sandbox browser-profile use case remains source-reproducible and not covered by the closed duplicate or closed unmerged fix PR.

## What This Changes

The issue asks OpenClaw to stop `config.patch` from scheduling SIGUSR1 when the changed paths are hot-reloadable browser profile config such as `browser.profiles.<name>.cdpUrl`.

## Best Possible Solution

Make browser-owned reload metadata granular so profile/default-profile changes avoid full gateway restarts while service-affecting browser fields still restart, then cover planner, config.patch, browser profile refresh, and docs behavior in focused tests.

## Reproduction Assessment

Yes, source-level. `browser.profiles.<name>.cdpUrl` still matches the broad `browser` restart prefix, the planner marks it as `restartGateway`, and hot-mode `config.patch` schedules SIGUSR1 when the plan says a restart is required.

## Solution Assessment

Yes on direction. The maintainable fix is plugin-owned granular browser reload metadata with prefix-order coverage, not a hard-coded browser allowlist inside the config.patch write flow.
