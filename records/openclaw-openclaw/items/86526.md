## Summary

Keep open: this is a useful focused bug fix, but the patch relaxes a security-sensitive SSRF policy less narrowly than the existing provider fake-IP policy, and the external PR still needs inspectable real behavior proof beyond assertion-only text.

## What This Changes

The PR adds an RFC2544 fake-IP SSRF policy opt-in to the OpenAI Realtime client-secret fetch and asserts that policy in two voice-provider tests.

## Best Possible Solution

Keep the fix in the OpenAI plugin, but align the Realtime secret fetch policy with the existing host-scoped provider fake-IP contract and require redacted live proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main calls fetchWithSsrFGuard for Realtime client secrets without an RFC2544 policy, while the SSRF guard blocks 198.18.0.0/15 unless that policy is set. I did not run a live Clash or Surge setup in this read-only review.

## Solution Assessment

No as submitted: adding the policy on the shared Realtime fetch is the right boundary, but the policy should be host-scoped and aligned with existing provider fake-IP semantics before merge.
