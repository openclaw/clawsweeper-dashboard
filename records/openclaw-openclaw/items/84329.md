## Summary

Keep open: current main still has no OpenClaw-owned iMessage send-transport setting or pass-through, but upstream imsg 0.9.0 already defaults omitted RPC send transport to auto, so the remaining decision is whether OpenClaw should expose a forceable bridge/applescript policy rather than only sending explicit auto.

## What This Changes

Request normal iMessage outbound replies to prefer or expose an imsg send transport selector for auto, bridge, or applescript delivery.

## Best Possible Solution

Add an iMessage-plugin-owned, account-inherited send transport enum only after maintainers choose the default policy; pass it to RPC send with older-imsg compatibility, document the behavior, and verify both unit coverage and live macOS delivery.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection confirms OpenClaw lacks a transport selector, while upstream imsg shows omitted transport already means auto, so logs or live proof are still needed to explain the reported AppleScript timeout path.

## Solution Assessment

Unclear until maintainers choose the compatibility policy. A config-backed selector is safer than only passing explicit auto, because imsg 0.9.0 already defaults to auto and forced bridge changes failure behavior.
