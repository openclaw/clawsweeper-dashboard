- `impact:message-loss`: The issue is about normal outbound iMessage replies timing out or failing when the AppleScript send path stalls.

## Summary

Keep open: current main still sends normal iMessage replies through RPC `send` without an `imsg` transport selector, while upstream `imsg` supports `auto`, `bridge`, and `applescript`. The referenced transport-fix commits are not contained in current main or any tag, so the remaining work still needs maintainer direction and live macOS proof.

## What This Changes

Request normal iMessage sends to prefer or expose an `imsg` transport selector so IMCore delivery can be used when available.

## Best Possible Solution

Add an iMessage-plugin-owned account-level transport enum with documented inheritance/default behavior, pass it into RPC `send`, preserve older `imsg` compatibility, and verify with focused tests plus live macOS bridge proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection does show current main omits the `imsg` transport selector, and upstream `imsg` has a transport contract that would exercise the requested behavior.

## Solution Assessment

Unclear until maintainers choose the default policy. The maintainable direction is a plugin-owned transport setting with tests and live proof, rather than relying on the current implicit AppleScript path.
