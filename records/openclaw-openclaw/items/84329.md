## Summary

Keep open: current OpenClaw still has no iMessage send-transport config or explicit selector on the normal RPC send path, but upstream imsg v0.9.0 already treats an omitted transport as auto, so the remaining work is a maintainer product/config decision rather than a straightforward missing-parameter bug.

## What This Changes

Request configurable iMessage outbound send transport selection so operators can prefer or force IMCore bridge, auto fallback, or AppleScript delivery.

## Best Possible Solution

Add an iMessage-plugin-owned send-transport enum only after maintainers choose the default policy, inherit it through account config, pass it to RPC send when supported, and document upgrade behavior plus live macOS proof.

## Reproduction Assessment

No live high-confidence macOS reproduction was established. Source inspection confirms OpenClaw has no explicit transport selector/config, but imsg v0.9.0 already defaults omitted transport to auto, so the actual timeout still needs live logs or proof.

## Solution Assessment

Unclear until maintainers choose policy. The best path is not merely adding transport: auto, because that matches the upstream default; the useful fix is a documented plugin config option with safe compatibility and forced-bridge semantics.
