## Label Justifications

- `P2`: This is a bounded iMessage delivery reliability improvement with a real reported deployment failure but a remaining config/default product decision.
- `impact:message-loss`: The report is about normal outbound iMessage replies timing out or failing when the AppleScript send path stalls.

## Summary

Keep open: current main still omits an iMessage send transport selector on the normal JSON-RPC send path and exposes no config/docs knob, while upstream imsg supports auto/bridge/applescript routing; the remaining blocker is a maintainer decision on default and compatibility policy.

## What This Changes

Add an iMessage send-transport selector or default so normal outbound replies can use imsg's IMCore bridge/auto transport when available.

## Best Possible Solution

Add a plugin-owned sendTransport enum with account-level inheritance, pass it into RPC send when supported, keep older imsg compatibility clear, and document/test the chosen default or opt-in behavior.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection does confirm current main never passes an imsg transport selector on the normal outbound path.

## Solution Assessment

Unclear until maintainers choose the default policy. The safer maintainable path is an explicit plugin config enum with account inheritance and compatibility handling, with any default-to-auto behavior documented and tested as intentional.
