## Summary

Keep open: current main and v2026.5.18 still expose no OpenClaw iMessage send-transport setting or explicit RPC transport, while imsg v0.9.0 supports transport selection. Because omitted transport already defaults to auto upstream, the unresolved work is a product/config decision plus live validation of the reported AppleScript fallback.

## What This Changes

Request normal iMessage outbound sends to prefer or expose imsg IMCore transport selection instead of relying on the AppleScript-backed path.

## Best Possible Solution

Add an iMessage-plugin-owned account-inherited `sendTransport` enum only if maintainers want operator control; preserve compatible auto behavior by default, pass configured values to RPC `send`, document bridge fail-closed semantics, and prove older-imsg plus live macOS behavior.

## Reproduction Assessment

No. Source inspection confirms OpenClaw has no send-transport selector, but I did not reproduce the macOS AppleScript timeout/IMCore-success split against a live Messages/imsg setup.

## Solution Assessment

Unclear until maintainers choose the default policy. An account-inherited plugin enum that preserves current auto behavior unless configured is safer than unconditionally forcing bridge or assuming the missing field is the whole cause.
