## Summary

Keep open: current main still sends normal iMessage replies through RPC `send` without a transport selector, and the repo has no iMessage config/docs surface for choosing `auto`, `bridge`, or `applescript`. Upstream `imsg` supports that selector, so the remaining work is a product/config decision rather than cleanup.

## What This Changes

This issue requests a configurable or defaulted IMCore-aware transport choice for normal outbound iMessage sends.

## Best Possible Solution

Add an iMessage-plugin-owned send transport enum with account-level inheritance, pass it to RPC `send` when supported, document the default/upgrade behavior, and cover older-`imsg` compatibility plus live macOS proof.

## Reproduction Assessment

No, there is not a high-confidence live reproduction in this review. Source inspection does show current main has no normal-send transport selector in the iMessage RPC payload, config schema, or docs.

## Solution Assessment

Unclear until maintainers choose the compatibility default. The maintainable direction is a plugin-owned config enum with account inheritance and explicit older-`imsg` handling, rather than an unconditional behavior change hidden in the send path.
