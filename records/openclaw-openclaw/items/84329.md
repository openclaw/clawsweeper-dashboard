- `impact:message-loss`: The report is about normal outbound iMessage replies failing or timing out when the AppleScript path stalls.

## Summary

Keep open: current main still has no iMessage send-transport config or explicit RPC transport selector. Upstream imsg 0.9.0 defaults omitted RPC transport to auto, so the remaining question is explicit operator control, compatibility behavior, and live proof of the reported stall.

## What This Changes

Request that normal iMessage outbound sends support choosing an IMCore-capable imsg transport, either by defaulting safely or by adding a configurable transport selector.

## Best Possible Solution

Add an iMessage-plugin-owned, account-inheriting sendTransport enum only if maintainers want explicit operator control; otherwise document that imsg 0.9.0 RPC send already defaults omitted transport to auto and route remaining delivery failures to narrower bugs such as https://github.com/openclaw/openclaw/issues/83365.

## Reproduction Assessment

No high-confidence live reproduction of the AppleScript timeout was established. Source inspection does confirm current main has no OpenClaw transport selector and relies on imsg's omitted-transport default.

## Solution Assessment

Unclear until maintainers choose the policy. Defaulting to auto is already upstream imsg 0.9.0 behavior, so the maintainable remaining path is an explicit config enum with compatibility handling and live proof if force-bridge behavior is desired.
