## Summary

Keep open: current main and v2026.5.18 still have no OpenClaw-level iMessage send-transport setting, but upstream imsg v0.9.0 already defaults omitted RPC `transport` to `auto`, so the remaining useful work is a maintainer choice about an explicit operator override and compatibility behavior.

## What This Changes

Request to let normal iMessage outbound replies prefer or configure IMCore-backed send transport instead of relying on AppleScript delivery.

## Best Possible Solution

Expose an iMessage-plugin-owned, account-inheritable `sendTransport` enum only after maintainers choose the default policy, preserve current dependency-default compatibility unless intentionally changing it, and document/test older-imsg behavior plus live macOS delivery proof.

## Reproduction Assessment

No high-confidence live reproduction of the timeout was established. Source inspection does confirm the absence of an OpenClaw transport config/explicit param, while upstream imsg v0.9.0 already treats an omitted transport as `auto`.

## Solution Assessment

Unclear until maintainers choose the product policy. The maintainable path is an explicit plugin config enum with account inheritance and compatibility handling, not an unconditional behavior change that silently removes AppleScript fallback.
