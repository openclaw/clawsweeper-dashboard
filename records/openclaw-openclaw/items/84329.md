## Summary

Keep open: current main and the latest release still omit an iMessage RPC send transport selector, while upstream imsg supports it; the remaining work needs a maintainer default/config policy decision.

## What This Changes

The issue requests configurable or automatic imsg transport selection for normal outbound iMessage replies.

## Best Possible Solution

Add an iMessage-plugin-owned send transport enum with account-level inheritance, pass it to RPC `send` when supported, document the default/upgrade behavior, and prove it against a supported imsg build on macOS.

## Reproduction Assessment

No, not as a high-confidence live reproduction: this review did not run macOS Messages/imsg. Source inspection does clearly show that current main never sends an `imsg` transport selector on the normal outbound path.

## Solution Assessment

Unclear until maintainers choose the default policy. A plugin-owned config enum with account inheritance and compatibility handling is the safer maintainable direction than silently changing all existing installs without upgrade proof.
