## Label Justifications

- `P2`: This is a bounded iMessage delivery reliability improvement with a config/default decision, not a core-runtime emergency.
- `impact:message-loss`: The report is about normal outbound iMessage replies timing out or failing when the AppleScript send path stalls.

## Summary

Keep open: current main still sends normal iMessage replies through RPC `send` without a transport selector, and the referenced transport patch exists only as an unmerged commit. The remaining work needs a maintainer decision on the default/config contract plus live macOS proof.

## What This Changes

The issue requests that normal iMessage outbound sends use or expose the imsg RPC send transport selector so operators can prefer auto, bridge, or applescript.

## Best Possible Solution

Add an iMessage-plugin-owned account-level transport enum, pass it into RPC `send` with older-imsg compatibility handling, update docs/tests, and require live macOS proof for the selected default.

## Reproduction Assessment

No high-confidence live macOS reproduction is available from this review. Source inspection does reproduce the missing selector: the normal outbound path builds RPC `send` params without `transport` even though upstream imsg supports it.

## Solution Assessment

Unclear until maintainers choose the default policy. The narrow maintainable path is a plugin-owned config enum with account inheritance, compatibility handling, docs, and tests; unconditional defaulting to `auto` should be an explicit product choice.
