## Summary

Keep open: current main and the latest shipped tag still omit an iMessage normal-send transport selector, while upstream `imsg rpc send` supports `auto`, `bridge`, and `applescript`; the remaining work is a maintainer decision on the config/default policy.

## What This Changes

The issue requests a configurable iMessage send transport so normal outbound replies can prefer IMCore/auto instead of always relying on the AppleScript-backed path.

## Best Possible Solution

Add an iMessage-plugin-owned send-transport enum with account inheritance and compatibility handling, then document and test the chosen default/upgrade behavior.

## Reproduction Assessment

No high-confidence live reproduction was established for the macOS AppleScript timeout. Source inspection does show current main and the latest shipped tag never pass the upstream-supported `transport` selector on the normal iMessage outbound path.

## Solution Assessment

Unclear until maintainers choose the default policy. The safest maintainable direction is an explicit plugin config enum with account-level inheritance and older-`imsg` compatibility, with any default-to-`auto` change documented as intentional.
