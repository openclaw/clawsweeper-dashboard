## Summary

Keep open: current OpenClaw still has no iMessage send-transport config surface, but imsg 0.9.0 already defaults omitted RPC `send.transport` to `auto`, so this is a product/config decision rather than a proven current-main bug.

## What This Changes

The issue requests normal iMessage outbound replies to prefer or allow configuring imsg's `auto`, `bridge`, or `applescript` send transport.

## Best Possible Solution

Decide whether to add an account-level iMessage `sendTransport` enum that defaults to imsg's existing `auto`, documents the fail-closed `bridge` tradeoff, and passes the value only to imsg builds that support it.

## Reproduction Assessment

No high-confidence live reproduction was performed. Source inspection does confirm the missing OpenClaw config knob, while imsg v0.9.0 shows omitted transport already defaults to `auto`.

## Solution Assessment

Unclear. Exposing an explicit transport override is plausible, but unconditionally changing OpenClaw's default is not the safest fix because imsg already owns `auto` fallback semantics and private API bridge use is operationally fragile.
