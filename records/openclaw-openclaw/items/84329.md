## Summary

Keep open: current main and v2026.5.18 still lack an OpenClaw iMessage send-transport setting, but `imsg` v0.9.0 already defaults omitted RPC transport to `auto`, so the remaining work is a maintainer decision about explicit `bridge`/`applescript` control, compatibility, and live proof.

## What This Changes

The issue requests configurable iMessage outbound send transport selection so normal replies can use or force IMCore/private API transport when available.

## Best Possible Solution

Add an iMessage-plugin-owned, account-inheritable `sendTransport` enum only after maintainers choose the default policy; preserve current compatibility unless intentionally changing it, document the SIP/private-API tradeoff, and prove the chosen path on a real Mac.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does reproduce the missing OpenClaw transport config/param path, while upstream `imsg` source shows omitted transport already defaults to `auto` on v0.9.0.

## Solution Assessment

Unclear until maintainers choose the product policy. A config enum with account inheritance and compatibility handling is the maintainable path if operator control is desired; blindly defaulting to `bridge` or assuming explicit `auto` fixes the report would be premature.
