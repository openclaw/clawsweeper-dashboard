- `impact:message-loss`: The report is about normal outbound iMessage replies timing out or failing when the AppleScript send path stalls.

## Summary

Keep open: current main still sends normal iMessage replies through JSON-RPC `send` without a transport selector, while current `imsg` supports `auto`, `bridge`, and `applescript`; the remaining decision is how OpenClaw should expose/default that behavior.

## What This Changes

Request configurable/default iMessage JSON-RPC send transport so normal outbound replies can use IMCore when available instead of being stuck on AppleScript.

## Best Possible Solution

Add an iMessage-plugin-owned send-transport setting with account-level inheritance, pass it into RPC `send` when supported, document the upgrade/default behavior, and cover older `imsg` compatibility.

## Reproduction Assessment

No live high-confidence macOS reproduction was established in this review. Source inspection does clearly show current main never sends an `imsg` transport selector on the normal outbound path.

## Solution Assessment

Unclear until maintainers choose the default policy. The safer direction is an explicit plugin config enum with account inheritance and compatibility handling, with any move to default `auto` documented and tested as an intentional behavior change.
