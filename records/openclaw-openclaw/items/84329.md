## Summary

Keep open: current main and v2026.5.18 still lack an OpenClaw-owned iMessage send-transport setting or explicit RPC selector, while the remaining behavior needs a maintainer default/config decision and live macOS proof.

## What This Changes

The issue requests normal iMessage outbound sends to prefer or expose configurable `imsg` IMCore/bridge versus AppleScript transport selection.

## Best Possible Solution

Add an iMessage-plugin-owned account-level `sendTransport` enum with documented default behavior, pass it into RPC `send` only when supported, and prove both compatibility and the bridge-preferred path on a real macOS `imsg` setup.

## Reproduction Assessment

No, we do not have a high-confidence live reproduction of the reported AppleScript stall on current main. Source inspection does show the missing OpenClaw transport selector and missing config surface for normal outbound sends.

## Solution Assessment

Unclear until maintainers choose the compatibility default. The maintainable direction is an explicit plugin config enum with account inheritance, version/support gating, docs, and tests rather than silently changing all existing sends.

## Risks / Open Questions
