## Summary

Keep open. Current main still cancels supervisor-managed exec and CLI-agent runs through immediate `SIGKILL`, while tests continue to assert that behavior; the related reports show this remains a real root-cause tracker rather than stale cleanup.

## What This Changes

The issue requests changing process-supervisor cancellation for exec and CLI-managed runs to SIGTERM first, bounded SIGKILL escalation, and reliable drain timeout behavior.

## Best Possible Solution

Implement a focused core supervisor change that sends SIGTERM on cancellation, waits a bounded unref'ed grace period, escalates to SIGKILL only if the managed run has not settled, and preserves the existing child/PTY wait fallback and process-tree safety contracts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: any `supervisor.spawn` run cancelled by manual cancel, no-output timeout, or overall timeout reaches `cancelAdapter` and sends `SIGKILL`; the current tests assert that behavior with fake timers.

## Solution Assessment

Yes. A supervisor-level SIGTERM-first grace period with bounded SIGKILL fallback is the narrowest maintainable fix; it should not be broadened into session-lock redesign or provider-specific signal handling.
