## Summary

Keep open. Current main still cancels supervised exec and CLI-agent runs by calling `adapter.kill("SIGKILL")` directly, while the discussion and related reports show this remains the canonical root-cause tracker for graceful timeout/cancel cleanup.

## What This Changes

Request SIGTERM-first cancellation with a bounded SIGKILL escalation and drain timeout for supervised exec processes.

## Best Possible Solution

Land a focused core supervisor fix that sends SIGTERM first, waits a short unref'ed grace period, then escalates through the existing SIGKILL adapter path if the run has not settled, keeping session-lock redesign and diagnostic improvements in separate issues.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: supervisor no-output and overall timeout paths call `requestCancel`, and current tests assert that cancellation calls `adapter.kill("SIGKILL")`.

## Solution Assessment

Yes. SIGTERM-first supervisor cancellation with bounded SIGKILL escalation is the narrowest maintainable fix; it should preserve existing adapter process-tree safety instead of becoming a session-lock or provider redesign.
