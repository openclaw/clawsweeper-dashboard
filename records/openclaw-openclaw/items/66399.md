## Summary

Keep open. Current main still routes supervisor timeout and cancellation through `adapter.kill("SIGKILL")`, and the child adapter immediately follows `killProcessTree(...)` with `child.kill("SIGKILL")`, so the direct child still does not get a bounded graceful shutdown window.

## What This Changes

Add SIGTERM-first graceful escalation and a bounded drain/settlement path for exec-tool timeout and cancellation.

## Best Possible Solution

Implement a focused supervisor cancellation policy that sends SIGTERM, waits an unref'ed grace period, then escalates through the existing hard-kill/wait fallback with updated child, PTY, kill-tree, and exec regression tests.

## Reproduction Assessment

Yes, at source level. Timeout/no-output/scoped replacement paths call `requestCancel`, current supervisor tests assert `SIGKILL`, and exec runs pass `timeoutMs` into the same supervisor path.

## Solution Assessment

Yes. SIGTERM first with a bounded SIGKILL escalation is the narrow maintainable direction, and it avoids the SIGTERM-only liveness regression called out on the prior broad PR.
