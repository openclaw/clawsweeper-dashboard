## Summary

Keep open. Current main still routes supervisor timeout/cancel through `adapter.kill("SIGKILL")`; the adapter has some wait-fallback and process-tree grace behavior, but the direct supervised child still does not receive a bounded SIGTERM-first grace window before hard kill.

## What This Changes

Request SIGTERM-first supervisor cancellation with bounded SIGKILL escalation and drain-timeout handling for exec-managed subprocesses.

## Best Possible Solution

The end state should be a focused supervisor cancellation policy that gives the supervised process a short graceful window, then reuses the existing adapter SIGKILL and wait-fallback path with updated regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: supervisor timeout/cancel paths call `requestCancel`, and current tests still assert that the adapter receives `SIGKILL` for those paths.

## Solution Assessment

Yes. SIGTERM first with an unref'ed grace timer and escalation through the existing adapter hard-kill path is the narrowest maintainable fix; it avoids the SIGTERM-only liveness regression noted on the prior broad PR.

Fix the process-supervisor timeout/cancel behavior tracked by https://github.com/openclaw/openclaw/issues/66399. Implement SIGTERM-first cancellation in `src/process/supervisor/supervisor.ts`: on timeout/manual cancellation, send `adapter.kill("SIGTERM")`, wait a short unref'ed grace period, and if the run has not settled, escalate through the existing `adapter.kill("SIGKILL")` path so current process-tree cleanup and wait-fallback behavior remain intact. Do not make the fix SIGTERM-only. Preserve Windows and PTY semantics, and do not redesign session locks, gateway CLI signal handling, or SIGKILL diagnostics in this PR. Add/update focused regression coverage for SIGTERM first, no SIGKILL when the process exits during grace, SIGKILL after grace when it remains running, and existing child/PTY wait fallback behavior as needed.
