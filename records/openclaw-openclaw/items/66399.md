## Summary

Keep open. Current main still routes supervisor cancellations for exec and CLI-agent runs through immediate `SIGKILL`, and the issue discussion plus related reports show this is the canonical remaining root-cause tracker rather than stale cleanup.

## What This Changes

Request graceful SIGTERM-first cancellation with bounded SIGKILL escalation and drain behavior for supervisor-managed exec/agent processes.

## Best Possible Solution

Land a focused core supervisor fix that sends SIGTERM first on cancellation, waits a short unref'ed grace period, then escalates to SIGKILL through the existing adapter path if the run has not settled, while preserving adapter wait fallbacks and process-tree safety.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: supervisor no-output and overall timeout tests trigger cancellation and currently assert `adapter.kill("SIGKILL")`, and both exec and CLI-agent surfaces spawn through that supervisor path.

## Solution Assessment

Yes. A supervisor-level SIGTERM-first grace period with bounded SIGKILL fallback is the narrowest maintainable fix; it should not be broadened into session-lock redesign or provider-specific signal handling.
