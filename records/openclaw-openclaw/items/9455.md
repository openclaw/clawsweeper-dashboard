## Summary

Keep open: the merged Cron Jobs filter bar covers most of the request, but current main still misses the explicit job-ID search path and an Unknown/missing last-run-status filter path already narrowed by prior review comments.

## What This Changes

This issue requests a compact Cron Jobs UI filter bar with search by job name/jobId, enabled status, schedule kind, last-run status, reset controls, and optional URL persistence.

## Best Possible Solution

Finish the existing filter bar by adding `job.id` to `cron.list` search and normalizing `lastRunStatus ?? lastStatus ?? 'unknown'` into an Unknown status option with focused tests.

## Reproduction Assessment

Yes, at source level. A query that appears only in `job.id` cannot match because `cron.list` excludes `job.id`, and missing last-run status cannot be selected because there is no Unknown filter value or missing-status mapping.

## Solution Assessment

Yes. Extending the existing service search haystack and Jobs last-status filter is the narrow maintainable path; adding a second UI flow or extra RPC would be broader than needed.
