Keep open: current main and the latest release still have the Cron Jobs filter bar, but two explicit acceptance paths remain incomplete: job-id search and filtering jobs with absent/unknown last-run status. No newer canonical issue or PR appears to own those leftovers.

## What This Changes

The issue requests a compact Cron Jobs UI filter bar with search, enabled, schedule-kind, last-run-status, and reset controls.

## Best Possible Solution

Complete the existing filter implementation in place: include job.id in cron.list search, add an Unknown last-run-status option, normalize lastRunStatus and lastStatus with missing status mapped to unknown, and add focused tests.

## Reproduction Assessment

Yes, at source level. A cron.list query that appears only in job.id cannot match because the service haystack excludes job.id, and jobs without a last-run status cannot be selected because the UI/controller expose no Unknown bucket.

## Solution Assessment

Yes. Extending the existing cron.list haystack and current Jobs last-status filter is the narrow maintainable path; adding a second UI search path or new RPC would be broader than needed.
