## What This Changes

Add a compact Cron Jobs UI filter bar with search, enabled, schedule-kind, last-run-status, and reset controls.

## Best Possible Solution

Complete the existing filter implementation in place by adding job-id search to `cron.list` and a normalized Unknown last-run-status bucket in the controller, view, and tests.

## Reproduction Assessment

Yes, at source level. A query that appears only in `job.id` cannot match because `cron.list` excludes `job.id`, and jobs with no last-run status cannot be selected because the Jobs status filter has no Unknown value or missing-status mapping.

## Solution Assessment

Yes. Extending the existing `cron.list` haystack and current Jobs last-status filter is the narrow maintainable path; adding a second UI flow or new RPC would be broader than needed.
