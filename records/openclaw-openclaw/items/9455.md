## What This Changes

Request a compact Cron Jobs UI filter bar for status, schedule kind, enabled state, and job name/job ID search.

## Best Possible Solution

Finish the existing filter implementation in place by adding job.id to cron.list search and exposing an Unknown status filter normalized from lastRunStatus, lastStatus, and missing state.

## Reproduction Assessment

Yes, at source level. A cron.list query that appears only in job.id cannot match because the service haystack excludes job.id, and jobs without a last-run status cannot be selected because the UI/controller expose no Unknown bucket.

## Solution Assessment

Yes. Extending the existing cron.list haystack and Jobs last-status filter is the narrow maintainable path; a second UI-only search path would be broader and would not handle unloaded pages cleanly.
