## Summary

Keep open: current main has the Cron Jobs filter bar, but two explicit acceptance paths remain incomplete: search by job ID and filtering jobs with missing or unknown last-run status.

## What This Changes

Add a compact Cron Jobs UI filter bar for search, enabled state, schedule kind, last-run status, and reset behavior.

## Best Possible Solution

Complete the existing filter implementation in place: include job.id in the cron.list search haystack, add an Unknown last-run-status option, normalize lastRunStatus and lastStatus with missing status mapped to unknown, and cover those paths with focused tests.

## Reproduction Assessment

Yes, at source level. A cron.list query that appears only in job.id cannot match because the service haystack excludes job.id, and jobs without a last-run status cannot be selected because the UI/controller expose no Unknown bucket.

## Solution Assessment

Yes. Extending the existing cron.list haystack and current Jobs last-status filter is the narrow maintainable path; adding a second UI search path or new RPC would be broader than needed.
