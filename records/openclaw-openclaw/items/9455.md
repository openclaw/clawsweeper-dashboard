## Summary

Keep open. The filter bar exists, but current main and the latest release still miss two explicit acceptance paths: search by job ID and filtering jobs whose last-run status is missing or unknown.

## What This Changes

The issue requests a compact Cron Jobs UI filter bar with search, enabled, schedule-kind, last-run-status, and reset controls.

## Best Possible Solution

Finish the existing filter bar by adding job.id to cron.list search and adding a normalized Unknown last-status bucket based on lastRunStatus ?? lastStatus.

## Reproduction Assessment

Yes, at source level. A search string that appears only in job.id cannot match because cron.list excludes job.id from its haystack, and a job with no last-run status cannot be selected because there is no Unknown filter value or missing-status mapping.

## Solution Assessment

Yes. Extending the existing cron.list haystack and existing Jobs last-status filter is the narrow maintainable path; adding a second UI flow or extra RPC would be broader than needed.
