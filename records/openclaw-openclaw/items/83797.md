## Summary

Keep open. Source inspection supports the main report: Telegram gateway message sends can complete as tool results without creating a normal assistant transcript entry, while Codex history projection elides tool payloads by default, so the sent human-visible text can be hidden from Control UI and later context.

## What This Changes

Requests that Telegram `message(action="send")` replies be preserved as visible assistant transcript/context instead of only appearing as message tool cards, with aborted sends terminalizing session state.

## Best Possible Solution

Canonicalize successful source-conversation `message(action="send")` deliveries into the OpenClaw transcript/projection boundary, then let Control UI render that assistant row while preserving generic tool-payload elision for unrelated tools.

## Reproduction Assessment

Yes for the main UI/context symptom from source inspection: Telegram gateway `message.action` returns a tool result and Codex projection elides tool payloads. I did not establish a live Telegram/Codex run or fully prove the stuck running-state branch.

## Solution Assessment

Yes on direction, but the fix should live at the transcript/projection boundary for channel-visible send results rather than only synthesizing a Control UI bubble.
