## Summary

Keep open. Current main still supports the reported source-level path: Telegram `message(action="send")` can complete through the gateway plugin action path without creating a normal assistant transcript row, while Codex’s default projection elides tool payloads and Control UI main lacks a message-send assistant-bubble projection.

## What This Changes

The issue reports that Telegram-visible `message(action="send")` replies delivered through the Codex runtime are shown as tool cards and omitted from later context instead of being preserved as assistant-visible transcript/context.

## Best Possible Solution

Canonicalize successful current-source `message(action="send")` deliveries at the transcript/projection boundary, then let Control UI render that assistant row while preserving generic tool-payload elision for unrelated tools.

## Reproduction Assessment

Yes for the main UI/context symptom from source inspection: Telegram gateway `message.action` returns a tool payload, the gateway plugin path returns before core mirror setup, and Codex projection elides tool payloads by default. I did not establish a live Telegram/Codex run or fully trace the stuck running-state branch.

## Solution Assessment

Yes on direction, but the maintainable fix should live at the transcript/projection boundary for channel-visible sends rather than only synthesizing a Control UI bubble. The related UI PR is useful context but is not a full fix for later Codex context preservation.
