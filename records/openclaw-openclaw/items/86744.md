## Summary

The latest head no longer has the two earlier quarantine correctness defects: it closes a partially opened SQLite handle and rolls sidecars back when quarantine fails. This remains a maintainer merge decision because the recovery deliberately trades corrupted durable task and delivery state for availability by quarantining the old database and starting the live registry empty.

## What This Changes

The PR adds task-registry SQLite corruption detection that quarantines corrupted `runs.sqlite` plus WAL/SHM sidecars, opens a fresh registry, logs recovery, and adds corruption recovery tests.

## Best Possible Solution

Land the narrow quarantine-and-recreate path if maintainers accept that corrupted task state is preserved only in `.corrupted.*` files while the live registry restarts empty; otherwise redesign toward recover-in-place semantics before merge.

## Reproduction Assessment

Yes. Source inspection and the linked report show a malformed `tasks/runs.sqlite` reaches `DatabaseSync`/`ensureSchema()` on current main and the restore boundary only logs `Failed to restore task registry`; I did not run a live gateway repro in this read-only review.

## Solution Assessment

Yes, for the reported task registry DB. The latest PR head makes quarantine fail-closed, closes a partial handle before rename, recreates a fresh DB, and covers main/WAL sidecar cases; maintainers still need to accept the availability-over-state-recovery semantics.
