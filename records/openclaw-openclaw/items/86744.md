## Summary

Keep open. Current main and v2026.6.1 still lack shared-state SQLite corruption recovery, and this PR is the active, proof-backed fix path; the remaining blocker is maintainer acceptance of availability-first recovery for the whole shared state database.

## What This Changes

This PR adds corruption recovery to the shared OpenClaw state DB opener by closing partial SQLite/WAL/Kysely state, quarantining malformed DB files and sidecars, reopening a fresh DB, and adding state/task registry regression tests.

## Best Possible Solution

Land this shared-opener recovery only after maintainers explicitly accept availability-first quarantine/fresh-DB semantics for all shared-state tables, or replace it with an operator/doctor-driven recovery policy.

## Reproduction Assessment

Yes, at source level: corrupt the current shared SQLite state database and call `loadTaskRegistryStateFromSqlite()`, which routes through `openOpenClawStateDatabase()`. I did not execute a full gateway repro in this read-only review, but the current-main opener still rethrows malformed DB errors.

## Solution Assessment

Yes, if maintainers accept the availability-first policy: the shared opener is the right layer now that task tables live in `state/openclaw.sqlite`. The safer alternative is a doctor/operator-driven recovery path if automatic fresh-state startup is not acceptable.
