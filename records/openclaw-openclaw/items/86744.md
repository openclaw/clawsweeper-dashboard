## Summary

Keep this PR open: the shared-state recovery code is a plausible and well-scoped fix for the linked task-registry corruption bug, but it intentionally changes corrupted `state/openclaw.sqlite` startup from fail-closed to availability-first recovery, so maintainer acceptance is still needed before merge.

## What This Changes

The PR adds SQLite corruption detection, quarantine, handle cleanup, and fresh shared-state DB retry in the shared state opener, with regression coverage for task-registry recovery and stale file descriptors.

## Best Possible Solution

Merge only after maintainers explicitly accept availability-first shared-state recovery, or redirect the branch to an operator-driven doctor/backup recovery path if fail-closed semantics should remain the default.

## Reproduction Assessment

Yes at source level: current main routes task-registry restore through the shared SQLite opener, and malformed `node:sqlite` files can throw during WAL setup before any recovery path exists. I did not run a full current-main gateway reproduction.

## Solution Assessment

Mostly yes: moving corruption handling to the shared state opener is the right code boundary because task storage now lives in `state/openclaw.sqlite`. The unresolved part is product policy, since availability-first recovery affects all shared-state consumers.
