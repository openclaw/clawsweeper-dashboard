## Summary

Keep open. The PR is still useful and not implemented on current main, but the current diff has a concrete startup-recovery bug: WAL setup can throw before the new close-and-quarantine guard owns the open SQLite handle, which can leave the corrupted database locked before the quarantine rename path runs.

## What This Changes

The PR adds corrupted shared-state SQLite recovery by detecting corrupt database-open/schema errors, quarantining `state/openclaw.sqlite` plus sidecars, recreating a fresh database, logging the event, and adding task-registry regression tests.

## Best Possible Solution

Fix the WAL-setup cleanup gap, then merge the availability-first shared-state quarantine only after maintainers explicitly accept the empty-fresh-database recovery semantics and keep deeper salvage/root-cause work on the linked issue.

## Reproduction Assessment

Yes for source-level reproduction: current main routes task restore through `openOpenClawStateDatabase()` and direct SQLite schema/WAL setup without a corruption recovery path. I did not run a local malformed-DB gateway because this review is read-only, but the linked issue and PR body provide concrete malformed-image evidence.

## Solution Assessment

No for the current diff as merge-ready: quarantine-and-recreate is an acceptable availability-first direction, but WAL setup can throw before the new handle-close guard runs. The best fix is to close the handle for every early SQLite setup error, then get maintainer acceptance for shared-state quarantine semantics.

- **[P2] Close the database when WAL setup throws:** `src/state/openclaw-state-db.ts:744-747`
