## Summary

Keep open for maintainer review: current main and v2026.6.5 still lack corrupted shared-state SQLite recovery, and this PR is a viable implementation with sufficient real proof, but merging it intentionally changes corrupted shared-state startup semantics from fail-closed to quarantine plus fresh empty state.

## What This Changes

The PR adds corruption detection, quarantine, fresh shared-state DB retry, and regression tests around `openOpenClawStateDatabase()` and task-registry restore.

## Best Possible Solution

Land this shared-opener recovery only after maintainers explicitly accept availability-first quarantine/fresh-DB semantics for all shared-state tables, or replace it with an operator/doctor-driven recovery policy.

## Reproduction Assessment

Yes, at source level: corrupt `state/openclaw.sqlite` and call `loadTaskRegistryStateFromSqlite()`, which routes through the shared DB opener that current main and v2026.6.5 still rethrow from. I did not execute a live gateway repro in this read-only review.

## Solution Assessment

Yes if maintainers accept availability-first recovery: the shared opener is the right layer now that task rows live in `state/openclaw.sqlite`. The safer alternative is a doctor/operator-driven recovery path if automatic fresh-state startup is not acceptable.
