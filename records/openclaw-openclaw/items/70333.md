## Summary

Current main implements the central fix: gateway startup auto-enable is runtime-only and no longer persists inferred enabled flags that would trigger self-induced reload churn. The doctor behavior now follows the merged config policy: explicit repair commands can write outside Nix mode, while startup/read paths do not mutate config automatically.

## What This Changes

The issue requests that auto-enable-derived channel/provider enabled flags remain runtime-only so startup and doctor do not rewrite openclaw.json and trigger gateway restart churn.

## Best Possible Solution

Keep the merged runtime-only startup behavior, preserve explicit doctor repair writes under the documented config mutation policy, and track the distinct WhatsApp hot-enable wedge in the separate open issue.

## Reproduction Assessment

No, not against current main. Source and regression tests show startup auto-enable returns a runtime overlay with wroteConfig false and does not call replaceConfigFile.

## Solution Assessment

Yes. The merged fix removes the automatic startup write that caused restart churn while leaving explicit repair commands under the documented config policy.
