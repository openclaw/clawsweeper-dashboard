## Summary

Current main implements the central fix: gateway startup auto-enable now stays runtime-only and no longer writes inferred `enabled` flags back to `openclaw.json`, so the self-induced reload/restart churn path is gone. `doctor --fix` remains an explicit repair writer outside Nix mode, matching the merged config policy rather than an automatic startup write.

## What This Changes

The issue asks for auto-enabled channel/provider activation to remain in memory instead of writing legacy `enabled` flags that can make the gateway react to its own config write.

## Best Possible Solution

Keep the current startup runtime-only auto-enable behavior and preserve `doctor --fix` as an explicit repair operation outside immutable Nix-managed installs.

## Reproduction Assessment

No, not against current main. Source and regression tests show startup auto-enable no longer writes config, so the old remove-keys-and-start-gateway churn path should not reproduce on current main.

## Solution Assessment

Yes. The merged fix removes the automatic startup write that caused restart churn while leaving explicit repair commands under the documented config mutation policy.
