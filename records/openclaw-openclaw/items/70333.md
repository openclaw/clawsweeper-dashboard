## Summary

Keep open: current main still persists auto-enable-derived `enabled: true` flags during gateway startup and still treats auto-enable as a doctor repair mutation. The related restart-loop fix in #67557 shipped and mitigates the watcher churn path, but it does not implement the requested runtime-only canonical config behavior.

## What This Changes

The issue asks for plugin/channel auto-enable to activate configured built-ins at runtime without persisting legacy `enabled` flags into `openclaw.json` during gateway startup or `doctor --fix`.

## Best Possible Solution

Decide the canonical config contract explicitly. If runtime-only auto-enable is the desired contract, split auto-enable into a runtime activation overlay plus read-only doctor diagnostics, then update the startup, doctor, and regression tests so inferred `enabled` flags are not written back to `openclaw.json`.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-backed: remove the reported `enabled` keys while leaving channel/provider config present, then current startup and doctor paths still run `applyPluginAutoEnable()` and persist the returned config when it reports changes.

## Solution Assessment

No. The current main solution only suppresses the self-write watcher event with the startup hash guard; it does not solve the issue's requested non-persistent, runtime-only auto-enable behavior.
