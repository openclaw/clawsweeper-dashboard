## Summary

Keep open: current main still persists auto-enable-derived `enabled: true` flags during gateway startup and doctor repair; #67557 fixed the self-reload guard, but not the requested runtime-only canonical config behavior.

## What This Changes

The issue asks for plugin/channel auto-enable to activate built-in channels/providers at runtime without rewriting inferred `enabled` flags into `openclaw.json` during startup or doctor repair.

## Best Possible Solution

Decide the canonical auto-enable contract explicitly; if runtime-only is desired, split inferred activation into an in-memory overlay plus read-only doctor diagnostics and update startup, doctor, and regression tests accordingly.

## Reproduction Assessment

Yes. The source-backed reproduction is to remove the reported `enabled` keys while leaving channel/provider config present; current startup and doctor paths still apply auto-enable and persist the returned config when it reports changes.

## Solution Assessment

No. Current main’s hash guard is a good fix for the watcher churn, but it is not the narrowest solution for the requested runtime-only config behavior because it still rewrites inferred `enabled` flags.
