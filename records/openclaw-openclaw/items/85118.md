## Summary

Keep open. The issue is recent, source inspection supports a real auth-provider lock-contention path around malformed/non-OpenClaw sidecar locks, and the current docs do not spell out the external-writer contract for `auth-profiles.json`; however the exact fcntl-only reproduction still needs runtime confirmation before a core recovery change is chosen.

## What This Changes

The issue asks OpenClaw to clarify or harden `auth-profiles.json` OAuth lock handling when external scripts use incompatible locking or write behavior.

## Best Possible Solution

Document `auth-profiles.json` as an OpenClaw-owned write store with safe snapshot reads for external scripts, then consider a narrow diagnostic or recovery change for malformed sidecars only if auth owners accept the race tradeoff.

## Reproduction Assessment

No high-confidence runtime reproduction was established in this read-only review. Source inspection does show a reproducible variant where malformed/non-OpenClaw `.lock` sidecars are preserved until timeout, but the fcntl-only part still needs real-process confirmation.

## Solution Assessment

Unclear as a code fix: documenting the lock contract is the narrow maintainable step, while self-healing removal or helper packages need an auth-owner product/security decision.
