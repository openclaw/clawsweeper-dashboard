## Summary

Keep open: current main still has a source-visible exec approval follow-up path that can fall back to direct transport delivery after session resume failure, while the linked timeout-attribution work only partially covers parent-facing timeout reporting and explicitly leaves physical child cleanup, approval cancellation, and late direct-output suppression unresolved.

## What This Changes

The issue reports that a timed-out subagent can leave a live Claude CLI child whose late exec-approval output is delivered directly to the user transport outside the parent agent.

## Best Possible Solution

Implement one core invariant for hard-timed-out subagents: the timeout path should abort the child run, clear pending approval/follow-up queues, and prevent any later child output from bypassing the parent delivery path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully from this read-only review: the issue gives concrete live steps and current source still shows the late direct-followup surface, but I did not run the Claude CLI plus delayed approval path against current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the fail-closed policy; the best fix should connect hard-timeout cleanup, approval queue cancellation, and late-output suppression rather than only removing the visible fallback prefix.
