## Summary

Keep open: current main still has the timeout/output-routing shape described in the report, so there is no safe close reason. The subagent timeout path records a terminal timeout, while async exec approval followups can still fall back to direct external delivery after a failed session resume.

## What This Changes

The issue asks for timed-out subagents to terminate their physical Claude CLI child process, discard late subagent output after the parent has finished, and cancel pending exec approval work owned by the timed-out child.

## Best Possible Solution

Fix the core runtime boundary so subagent timeout aborts the owning run/process, expires or cancels owned approval waits, and suppresses or session-confines any late child followups after the parent/session is terminal.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the output-routing path at source level: current main can record a subagent timeout while the exec-approval followup code still has a direct-send fallback after session resume failure. I did not establish the live systemd/claude-cli process-table reproduction.

## Solution Assessment

Is this the best way to solve the issue? No, current main is not sufficient because timeout bookkeeping and direct followup delivery remain separate. The narrower maintainable fix is to bind timeout cancellation to the child run/process and gate exec followups by the owning subagent/session terminal state.
