## Summary

Keep open: current `main` and latest release still show the core lifecycle gap. Timeout is recorded as a terminal subagent outcome without using the explicit abort and queue-cleanup path, while late exec approval follow-up can still resume or direct-send non-denied child output after a session-follow-up failure.

## What This Changes

The issue reports that timed-out Claude-backed subagents can keep running after `runTimeoutSeconds`, then leak late child output directly to the user transport after exec approval completion.

## Best Possible Solution

Bind subagent timeout to explicit abort and queue-clear semantics, expire child-owned exec approval continuations, and make late child-session follow-up fail closed once the owning subagent run is terminal.

## Reproduction Assessment

No live high-confidence reproduction was established for the physical Claude CLI process in this read-only checkout. Source inspection does show the chain: documented timeout, timeout completion without kill cleanup, detached exec approval continuation, and direct external fallback after resume failure.

## Solution Assessment

Yes for the maintainer direction, but the exact fail-closed routing policy needs owner/security review. The narrow maintainable fix is lifecycle cleanup parity with explicit kill plus a terminal-child guard in approval follow-up delivery.
