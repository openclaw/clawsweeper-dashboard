Keep open. Current main still has the source-level Slack/CLI transcript-vs-visible-delivery drift, and the protected `maintainer` label plus product-choice labels make this unsuitable for cleanup closure or a new automated fix PR.

## What This Changes

The issue requests that Slack/CLI final replies not leave normal transcript rows implying visible delivery when foreground freshness suppresses the channel final.

## Best Possible Solution

Add a maintainer-owned path that records or avoids transcript-visible assistant rows when a Slack/CLI final is cancelled by foreground freshness, while preserving the newer-visible freshness fence and existing queued-followup barrier.

## Reproduction Assessment

Yes at source level, but not as a live Slack replay in this read-only pass. Current main still has CLI transcript persistence before final delivery settlement, optional CLI steering, and foreground freshness cancellation without a suppressed-delivery transcript marker.

## Solution Assessment

Unclear. The requested invariant is right, but maintainers need to choose whether to preserve the final, delay persistence, or add explicit suppressed-delivery metadata; the safest narrow path appears to be metadata/marker bookkeeping that preserves the freshness fence.
