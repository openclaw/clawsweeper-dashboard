## Summary

Keep open. Current main still routes the macOS Skills settings install action through the gateway-backed `skills.install` path, still prefers a Homebrew install when `skills.install.preferBrew` is true and `brew` is present, and still returns raw installer failures as `UNAVAILABLE` without a macOS 26 / Apple CLT remediation classifier. Adjacent open work on broken skill package metadata does not cover this CLT-specific failure mode.

## What This Changes

The issue reports that installing skills from the macOS app fails on macOS 26 when Homebrew exits because the installed Apple Command Line Tools do not support that macOS release.

## Best Possible Solution

Add a narrow gateway skill-installer fix that classifies Homebrew/Xcode CLT unsupported-macOS failures, returns targeted remediation, suppresses or de-prioritizes brew options when the host toolchain is known unusable, and only retries an alternate installer spec when the skill metadata and existing installer policy make that safe.

## Reproduction Assessment

Yes. The high-confidence reproduction path is a macOS 26 host with `brew` present but Apple CLT unsupported, then install a brew-backed skill through the macOS Skills settings flow; current source shows that brew exit will be surfaced as generic `UNAVAILABLE`.

## Solution Assessment

No. Current main is not the best solution because it still reports the raw Homebrew failure; the maintainable fix belongs in the gateway skill installer/status selection layer with focused tests, not as a macOS-only UI string workaround.
