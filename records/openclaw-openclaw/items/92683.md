## Summary

Keep open: current main only partially addresses the status half with `storage`/`sqlitePath`, while the doctor cron check can still print a legacy-storage headline for SQLite-only maintenance states when no legacy `jobs.json` file exists.

## What This Changes

The issue asks for `openclaw doctor` and `openclaw cron status` to stop pointing operators at retired `cron/jobs.json` storage after cron jobs have migrated to SQLite.

## Best Possible Solution

Preserve the additive `storage`/`sqlitePath` status contract, keep deprecated `storePath` as a logical key, and change doctor cron output so legacy-storage wording appears only when legacy cron files or run logs actually exist.

## Reproduction Assessment

Yes, source-backed. Current main still has a doctor branch where SQLite-only maintenance preview lines reuse the legacy-storage headline, and v2026.6.6 still lacks the newer `cron.status` SQLite fields.

## Solution Assessment

Yes. The narrow maintainable fix is at the reporting boundary: split doctor legacy-file warnings from SQLite maintenance warnings, while preserving the additive status compatibility shape already landed on main.
