## Summary

Keep open. The PR targets a real current-main audit false positive, but the latest diff still misses the dual-field auth-profile case where an explicit `keyRef`/`tokenRef` is present beside an inline env ref, and it also lacks the required changelog entry.

## What This Changes

The PR adds bare `$VAR` env SecretRef parsing, changes auth-profile audit handling for inline SecretRefs, and adds parser/audit regression tests.

## Best Possible Solution

Update this PR so auth audit classifies the inline credential value itself before plaintext fallback, preserve explicit-ref plus real leftover plaintext findings, add a changelog entry, and then use it to resolve the linked secrets-audit false-positive reports.

## Reproduction Assessment

Yes. A static trace reproduces the remaining PR bug: a profile with `keyRef` plus `key: "$OPENAI_API_KEY"` or `key: "${OPENAI_API_KEY}"` makes `inlineRef` null and still reaches `PLAINTEXT_FOUND`.

## Solution Assessment

No. The direction is right, but using `inlineRef` after explicit refs suppress inline parsing is incomplete; the narrower maintainable fix is to inspect the inline value independently before the plaintext fallback.
