## Summary

Keep open. The PR addresses a valid current-main secrets-audit false positive, but the live diff still has a concrete correctness gap: auth profiles with an explicit ref plus an inline `$VAR` or `${VAR}` value can still be reported as `PLAINTEXT_FOUND` because inline parsing is suppressed when the explicit ref is present. The PR also needs the required changelog entry for a user-facing CLI fix.

## What This Changes

The PR adds bare `$VAR` env SecretRef parsing, adjusts auth-profile audit classification for inline env refs, and adds parser/audit regression tests.

## Best Possible Solution

Update this PR so auth audit classifies the inline credential value itself before plaintext fallback, while keeping explicit-ref plus real leftover plaintext findings and documenting the CLI fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. A static trace gives a high-confidence reproduction: a profile containing `keyRef` plus `key: "$OPENAI_API_KEY"` or `key: "${OPENAI_API_KEY}"` makes `inlineRef` null, then the plaintext fallback still emits `PLAINTEXT_FOUND`.

## Solution Assessment

No. The PR is directionally correct, but using `inlineRef` after explicit refs have suppressed inline parsing is not the complete fix; checking the inline value independently before plaintext classification is safer and narrower.
