## Summary

Keep open. Current main still has the false-positive path, and this PR is useful but incomplete because its `inlineRef` guard misses dual-field auth profiles where an explicit ref suppresses inline parsing before plaintext classification.

## What This Changes

The PR adds bare `$VAR` env SecretRef parsing, skips plaintext findings for inline auth-profile env refs, and adds parser/audit regression tests.

## Best Possible Solution

Update auth audit to classify the inline credential value independently before plaintext fallback, preserve explicit-ref plus real-plaintext findings, add dual-field regression coverage, and document the CLI fix in the changelog.

## Reproduction Assessment

Yes. A static trace reproduces the remaining bug: an auth profile with `keyRef` plus `key: "$OPENAI_API_KEY"` or `key: "${OPENAI_API_KEY}"` leaves `inlineRef` null and still reaches `PLAINTEXT_FOUND`.

## Solution Assessment

No. The PR is directionally correct, but checking `inlineRef` after explicit refs have suppressed inline parsing is not the complete fix; checking the inline value separately is the narrower maintainable path.
