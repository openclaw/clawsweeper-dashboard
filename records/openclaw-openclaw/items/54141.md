## Summary

Keep open. The PR is useful but still leaves an auth-profile false positive when an explicit `keyRef`/`tokenRef` exists beside an inline `$VAR` or `${VAR}` value, because the proposed guard only uses `inlineRef` after explicit refs have already suppressed it.

## What This Changes

The PR adds bare `$VAR` env SecretRef parsing, adjusts auth-profile secrets audit handling for inline env refs, and adds parser/audit regression tests.

## Best Possible Solution

Revise this PR so auth audit classifies the inline credential value itself before the plaintext fallback, even when a sibling explicit ref is present, while preserving `PLAINTEXT_FOUND` for real leftover credential strings and adding the required changelog entry.

## Reproduction Assessment

Yes. A profile containing `keyRef` plus `key: "$OPENAI_API_KEY"` or `key: "${OPENAI_API_KEY}"` follows the PR path where `inlineRef` is null, then `isNonEmptyString(entry.value)` still emits `PLAINTEXT_FOUND`.

## Solution Assessment

No. The PR is directionally right, but the current guard is not the best complete fix because it relies on `inlineRef` after explicit refs suppress inline parsing; the safer fix is to classify `entry.value` independently before treating it as plaintext.
