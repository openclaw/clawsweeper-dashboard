## Summary

Keep open: current main still collapses the documented code-mode timeout and runtime-unavailable cases into `internal_error`, so this remains a narrow valid bug.

## What This Changes

The issue requests that code-mode `exec` return documented `timeout` and `runtime_unavailable` error codes for known failure paths instead of `internal_error`.

## Best Possible Solution

Map known timeout and worker-runtime-unavailable failures to the documented code-mode error codes and add regression assertions for the structured `code` field.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `while (true) {}` reaches the QuickJS timeout branch and then the worker catch returns `internal_error`, while a missing worker reaches the parent worker `error` handler and also returns `internal_error`.

## Solution Assessment

Yes. The requested fix is the narrow maintainable path: preserve the existing failure behavior but classify known failures with the documented codes, leaving `internal_error` for unexpected cases.

In code mode, known timeout failures must return `code: "timeout"`, and worker load/runtime-unavailable failures must return `code: "runtime_unavailable"`; keep `internal_error` for invalid worker responses, nonzero worker exits, and unexpected exceptions. Update the worker result/code types in `src/agents/code-mode.ts` and `src/agents/code-mode.worker.ts` as needed, classify the QuickJS `didTimeout()` path before it is collapsed by the worker catch, classify parent worker load errors from missing/unloadable `code-mode.worker.js`, and add focused regression coverage in `src/agents/code-mode.test.ts` asserting the structured `code` for the hostile infinite-loop timeout and runtime-unavailable worker path. Changelog is expected if maintainers treat this as user-facing release work.
