## Summary

Keep open. This is an active maintainer-labelled Signal PR, so it should not be auto-closed by cleanup. Current main already disables the SSE header deadline on the production Signal monitor path with `timeoutMs: 0`, while this PR broadens that behavior to the internal `streamSignalEvents` default; I found no concrete correctness or security finding in the proposed patch.

## What This Changes

The PR removes the default 10s header deadline for Signal SSE streams when callers omit `timeoutMs`, preserves explicit SSE deadlines, adds quiet-stream regression coverage, and adds a changelog entry.

## Best Possible Solution

Maintainers should decide whether the internal Signal SSE helper should be long-lived by default, then either merge this small default-semantics cleanup or close it after confirming the existing monitor-level fix fully covers the linked inbound Signal bug; RPC/check deadlines should stay bounded.

## Reproduction Assessment

Yes for the direct client behavior: a test HTTP server that accepts `/api/v1/events` and never sends headers reproduces the current default-deadline path unless `timeoutMs: 0` is supplied. The production monitor path on current main already supplies that opt-out, so live reproduction of the linked report against current main is less certain.

## Solution Assessment

Unclear. The patch is narrow and consistent with long-lived SSE semantics, but current main already uses the more localized monitor-level opt-out, so the best final path depends on whether maintainers want to change the internal helper default for all direct callers.
