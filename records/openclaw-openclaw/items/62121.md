## Summary

Keep open. Current main has partial guardrails for phase-tagged commentary and Telegram cron announce delivery, but it still does not define a provider-agnostic final-output boundary for plain untagged DeepSeek preamble; the maintainer discussion explicitly rejected marker/emoji/DeepSeek-specific filtering and left this for a cleaner cron delivery design.

## What This Changes

The issue asks OpenClaw to prevent untagged DeepSeek preamble text from being included in Telegram isolated cron fallback delivery after upgrading from 2026.3.13 to 2026.4.5.

## Best Possible Solution

Define a generic isolated-cron final-output contract that channels can trust without model IDs, emojis, or prose guessing. It should preserve existing phase-aware suppression, specify how unphased provider output becomes final-only, and add regression coverage for DeepSeek-style preamble before a formatted Telegram cron report.

## Reproduction Assessment

Yes for triage: the reporter provided version range, model, channel, cron mode, and confirmed consistent leakage on 2026.4.5. The best automated reproduction is a synthetic isolated cron run where the assistant emits unphased preamble plus the intended report, then Telegram announce delivery must exclude the preamble under the new contract.

## Solution Assessment

No. The suggested outputStartsWith/emoji sanitizer is not the best general fix; the safer direction is the maintainer-requested provider-agnostic final-output boundary after tracing the 2026.3.13 to 2026.4.5 delivery-path change.
