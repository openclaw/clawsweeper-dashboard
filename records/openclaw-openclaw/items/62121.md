## Label Justifications

- `P1`: This is a reported upgrade regression in Telegram cron delivery where unwanted assistant preamble text consistently reaches real users for affected DeepSeek jobs.
- `impact:message-loss`: The intended Telegram cron report is corrupted by extra assistant preamble text being routed into the user-visible message.

## Summary

Keep open. Current main still treats unphased assistant text as visible output, Telegram cron announce delivery can select that text as the final outbound message, and the live discussion explicitly asks for a provider-agnostic cron final-output contract rather than a DeepSeek-specific sanitizer.

## What This Changes

The issue reports that DeepSeek cron runs after the 2026.3.13 to 2026.4.5 upgrade deliver untagged assistant preamble text to Telegram before the intended formatted report.

## Best Possible Solution

Define a generic isolated-cron final-output contract that channels can trust without model IDs, emoji markers, prose heuristics, or unsupported cron payload fields, then cover unphased preamble-plus-report Telegram delivery with regression and live transport proof.

## Reproduction Assessment

Yes. Source inspection on current main shows unphased assistant text remains visible, Telegram opts into finalAssistantVisibleText, and cron can collapse text-only announce delivery to that value; I did not run a live DeepSeek Telegram cron job.

## Solution Assessment

No for the suggested sanitizer or outputStartsWith path. The maintainable solution is the maintainer-requested provider-agnostic cron final-output boundary, with compatibility decisions for legacy unphased assistant text.
