## Label Justifications

- `P1`: This is a reported upgrade regression in Telegram cron delivery where unwanted assistant preamble text reaches real users consistently for the affected DeepSeek jobs.
- `impact:message-loss`: Internal/unphased assistant text is being routed into channel-visible Telegram messages, corrupting the intended user-facing delivery.

## Summary

Keep open. Current main still lacks a provider-agnostic isolated-cron final-output boundary for untagged assistant prose, and the maintainer discussion explicitly left this for a design pass rather than a DeepSeek-specific sanitizer or marker/emoji stripping.

## What This Changes

The issue reports a 2026.3.13 to 2026.4.5 regression where DeepSeek cron output with untagged preamble text is delivered to Telegram despite thinking/commentary filtering.

## Best Possible Solution

Define a generic isolated-cron final-output contract that channels can trust without model IDs, emojis, prose heuristics, or unsupported cron payload fields, then add regression coverage for an unphased preamble-plus-report Telegram cron output.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main still falls back to unphased assistant text and Telegram cron delivery collapses text-only output to that value, matching the reported DeepSeek preamble-plus-report shape.

## Solution Assessment

No, the suggested sanitizer/outputStartsWith direction is not the best fix. The maintainable path is the maintainer-requested generic cron final-output boundary after tracing the 2026.3.13 to 2026.4.5 delivery-path change.
