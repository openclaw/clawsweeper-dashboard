## Summary

Keep open. Current main has plausible post-v2026.4.29 hardening for runtime-deps convergence, provider-hook caching, and model-catalog responsiveness, but the discussion reports a distinct TUI-side bundled provider reload loop with a working TUI-only environment workaround and a second user confirmation. I do not see enough direct current-main proof to close it as implemented or duplicate it into the broader gateway runtime-deps issue.

## What This Changes

The issue reports that `openclaw tui` in v2026.4.29 hangs before rendering because the TUI-side bundled plugin loader repeatedly reloads provider plugins until the process runs at about 100% CPU.

## Best Possible Solution

Keep this open until a focused current-main regression or live startup proof shows that gateway-backed `openclaw tui` does not repeatedly load bundled provider plugins before first render. If the loop still reproduces, fix the narrow TUI gateway startup/provider-hook/catalog path instead of globally disabling bundled plugins.
