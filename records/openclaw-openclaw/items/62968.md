## Summary

Current main still matches the report: completion announce dispatch can return a successful queue or steer fallback while preserving the failed direct send only in phase data, and the caller/lifecycle paths skip diagnostics once the top-level delivery is marked delivered. The related open PR #75462 covers a different swallowed completion error in the run manager, so it does not supersede this issue.

## What This Changes

Request logging or otherwise surfacing a direct subagent announce failure when completion-mode queue fallback succeeds.

## Best Possible Solution

Keep successful queue fallback behavior, but add one narrow recovered-direct-failure diagnostic for completion announce delivery and cover the delivered fallback case in focused tests.

## Reproduction Assessment

Yes. The unit path is high-confidence: call runSubagentAnnounceDispatch with expectsCompletionMessage=true, direct returning delivered=false/path=direct/error and queue returning steered or queued, then observe a delivered top-level result with the error only in phases and no caller-level diagnostic.

## Solution Assessment

Yes. The narrow maintainable fix is to inspect delivery phases after delivery or add a small dispatch diagnostic seam; changing fallback success semantics or relying on PR #75462 would solve a different problem.
