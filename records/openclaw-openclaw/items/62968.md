## Summary

Keep open: current main and the latest release still preserve the direct-primary error only in `phases` when fallback steering delivers, while the caller only logs top-level failed direct delivery. The issue is valid, but a new automated fix should wait for maintainer direction because the open draft queue-first completion PR changes the same dispatch decision surface.

## What This Changes

The issue requests a warning when subagent completion direct-send fails but fallback delivery succeeds, so operators can diagnose the recovered direct-send failure.

## Best Possible Solution

Fold a narrow phase-aware gateway warning into the active subagent completion delivery work: if a failed `direct-primary` phase is recovered by fallback delivery, emit a normal warning without changing delivery ordering unless maintainers choose the queue-first PR direction.

## Reproduction Assessment

Yes, at source level: current main has a tested path where completion direct delivery fails, fallback steering succeeds, and the failed direct phase is only retained in `delivery.phases`. I did not run a live gateway or cron scenario.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to log failed direct-primary phases when fallback delivery recovers, or to make the overlapping queue-first completion PR explicitly own the behavior. A new automated PR should wait because that PR changes the same ordering surface.
