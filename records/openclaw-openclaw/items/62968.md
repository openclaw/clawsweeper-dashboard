## Summary

Keep open: current main and v2026.6.10 still preserve recovered direct-send failures only in delivery phases while the logging and registry-delivery callback paths treat the fallback as a clean success. The issue remains a valid narrow diagnostics bug, but the open queue-first completion PR and existing maintainer-review labels make this a coordination item rather than a new automated fix lane.

## What This Changes

The issue requests warning-level logging when a failed direct subagent announce attempt is recovered by fallback delivery.

## Best Possible Solution

Add a bounded warning for failed recovered direct-primary phases in the subagent announce path, or fold that diagnostic into the open queue-first completion announce work if maintainers choose that direction.

## Reproduction Assessment

Yes, at source level: current dispatch tests and source show a failed completion direct-primary phase can be followed by delivered fallback steering, while caller and registry callback paths only surface errors for non-delivered results. I did not run a live cron or gateway fallback scenario in this read-only review.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to warn when a failed direct-primary phase is recovered by fallback delivery. The exact landing path should be coordinated with the open queue-first completion announce PR because it changes the same ordering surface.
