## Summary

Keep open. Current main still matches the report: completion announce dispatch records the failed direct send only in phase data, returns the successful queue/steer fallback as the top-level delivered result, and the caller plus lifecycle callback only surface direct errors when the overall delivery failed.

## What This Changes

Report a warning or diagnostic when a subagent completion announce direct-send fails but queue fallback later recovers delivery.

## Best Possible Solution

Keep the current successful fallback delivery semantics, but add one narrow recovered-delivery diagnostic for completion dispatch when a failed direct-primary phase with an error is followed by a delivered queue-fallback phase.

## Reproduction Assessment

Yes. The unit-level path is clear from current source and tests: make completion dispatch return direct delivered=false/path=direct/error and queue return queued or steered, then observe delivered=true at the top level with the error only in phases.

## Solution Assessment

Yes. The narrowest maintainable solution is to inspect the returned phases after delivery or add a small dispatch logger/callback seam; changing fallback success semantics would solve the wrong problem.

Current completion-mode subagent announce dispatch can return delivered=true/path queued or steered after a direct-primary phase failed with an error, and the caller/lifecycle do not surface that recovered direct-send error. Add a narrow warning or diagnostic for this recovered path without marking the announce failed and without duplicating transient retry logs from runAnnounceDeliveryWithRetry. Prefer implementing near runSubagentAnnounceFlow after delivery is returned, or via a small dispatch callback/logger seam if that fits existing patterns. Do not change queue fallback semantics, delivery status, public plugin APIs, or unrelated announce routing behavior.
