## Summary

Keep open: current main still has the recovered direct-send diagnostic gap. Direct completion delivery can fail, fallback steering can succeed, and the caller/lifecycle paths treat the delivered fallback as success without surfacing the direct error; the related queue-first PR is open, draft/dirty, and not merged.

## What This Changes

Report that subagent announce direct-send failures should be logged even when fallback steering successfully delivers the completion.

## Best Possible Solution

Coordinate with https://github.com/openclaw/openclaw/pull/76927; if that direction does not land, add a narrow warning for recovered direct-primary errors while preserving successful fallback delivery semantics.

## Reproduction Assessment

Yes, source-level. Stub direct delivery to return delivered false, path direct, and an error, then stub fallback steering to return steered; current main returns delivered fallback success with the direct error only in phases.

## Solution Assessment

Yes for the current direct-first design: inspect the recovered direct-primary phase or log at the fallback decision point. If the queue-first PR lands first, that broader routing change may be the better fix path.
