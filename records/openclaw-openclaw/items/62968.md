Keep open: current main and latest release still preserve the recovered direct-send failure only in dispatch phases, while the caller logs only top-level failed direct delivery. ## What This Changes

The issue reports that failed direct subagent announce delivery is silently swallowed when fallback steering succeeds.

## Best Possible Solution

Add a bounded warning for failed recovered direct-primary phases in the subagent announce delivery path, or explicitly fold that diagnostic into the open queue-first completion announce PR if maintainers choose that direction.

## Reproduction Assessment

Yes at source level: the dispatch test and current code show a failed completion `direct-primary` phase can be followed by delivered fallback steering, while the caller only logs failed top-level direct results. I did not run a live cron/gateway fallback scenario in this read-only review.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is to warn when a failed direct-primary phase is recovered by fallback delivery. The timing/order question should stay coordinated with the open queue-first completion announce PR.
