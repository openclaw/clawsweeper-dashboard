## Summary

Keep open: current main still has the recovered direct-send diagnostic gap. The direct failure is preserved only in `phases` when fallback steering succeeds, and the caller/lifecycle paths treat the delivered fallback as success without surfacing that direct error; the open queue-first completion announce PR may supersede the narrow warning fix but is not merged.

## What This Changes

This issue asks OpenClaw to log or otherwise surface a failed direct subagent completion announce when fallback queue/steer delivery succeeds.

## Best Possible Solution

Either land the queue-first completion announce direction in https://github.com/openclaw/openclaw/pull/76927 or add a narrow warning for recovered `direct-primary` failures while preserving successful fallback delivery semantics.

## Reproduction Assessment

Yes, source-level. Stub direct delivery to return `delivered: false`, `path: "direct"`, and an error, then stub fallback steering to return `steered`; current main returns delivered fallback success with the direct error only in `phases`.

## Solution Assessment

Yes for the current direct-first design: inspect the recovered `direct-primary` phase or log at the fallback decision point. If https://github.com/openclaw/openclaw/pull/76927 lands first, the better solution may be the queue-first dispatch order instead of adding a separate warning.
