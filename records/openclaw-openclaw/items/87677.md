## Summary

Keep open: current main and the latest release still require API-key-style auth for OpenAI memory embeddings, and the earlier PR that attempted this OAuth fallback closed unmerged. This remains a product/auth-provider feature decision rather than an implemented or stale issue.

## What This Changes

Request routing `memorySearch.provider: "openai"` embedding calls through Codex OAuth/runtime for ChatGPT subscriber auth instead of requiring `OPENAI_API_KEY`.

## Best Possible Solution

Have maintainers decide whether Codex OAuth may cover embeddings; if approved, implement a narrow native-OpenAI-only embedding route with docs, API-key/custom-endpoint regression tests, and live redacted OAuth proof.

## Reproduction Assessment

Yes for the current limitation: the reporter provided a live Pi reproduction, and current source/docs show OpenAI memory embeddings require API-key-style credentials. I did not run a live OAuth credential flow in this read-only review.

## Solution Assessment

Unclear until maintainers approve the auth contract change. If approved, a host-gated OpenAI embedding adapter path is safer than broad generic auth-order expansion.
