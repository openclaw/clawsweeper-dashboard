## Summary

Keep open: current main and the latest release still implement OpenAI memory embeddings as direct API-key-backed `/v1/embeddings`, so Codex OAuth routing remains a real auth/provider feature gap that needs maintainer product approval.

## What This Changes

Route `memorySearch.provider: "openai"` embedding calls through Codex OAuth for ChatGPT subscriber profiles instead of requiring `OPENAI_API_KEY`.

## Best Possible Solution

Have maintainers decide the OAuth embedding contract first; if approved, implement a scoped OpenAI embedding OAuth route with API-key and OAuth regression tests plus docs for billing and cache/index behavior.

## Reproduction Assessment

Yes. The reporter live-reproduced the failure on v2026.5.27, and current main still shows the same API-key-required OpenAI embedding path and documentation.

## Solution Assessment

Unclear until product approval. Routing through the OpenAI provider/embedding seam is plausible, but upstream Codex currently exposes Responses/chat plumbing rather than an embeddings endpoint, so this is not a safe automated bug fix.
