## Label Justifications

- `P2`: This is a normal-priority memory/provider capability gap with documented workarounds through API-key or non-OpenAI embedding providers.
- `impact:session-state`: The affected feature is semantic memory search, which controls retrieval of prior memory and session context.
- `impact:auth-provider`: The issue is specifically about routing OpenAI embedding auth through Codex OAuth instead of direct API-key credentials.

## Summary

Keep open: current main and the latest release still document and implement OpenAI memory embeddings as API-key-style remote embedding calls, while the requested Codex OAuth route remains a product/auth-provider decision with no current upstream Codex embeddings proxy contract.

## What This Changes

The issue requests allowing `memorySearch.provider: "openai"` to use a user's OpenAI Codex OAuth profile for native OpenAI embedding requests instead of requiring `OPENAI_API_KEY`.

## Best Possible Solution

Have maintainers decide whether Codex OAuth may cover native OpenAI embeddings; if approved, implement a narrowly host-gated OpenAI embedding path with docs, regression tests for API-key and custom-endpoint behavior, and redacted live OAuth proof.

## Reproduction Assessment

Yes, at source level: the reporter supplied live Pi output, and current main still routes OpenAI memory embeddings through API-key-style bearer auth while docs exclude Codex OAuth. I did not run a live OAuth credential flow in this read-only review.

## Solution Assessment

Unclear until maintainers approve the auth-contract change. If approved, a native-OpenAI-only embedding adapter path guarded to `api.openai.com` is safer than broad generic auth-order expansion or sending OAuth credentials to custom endpoints.
