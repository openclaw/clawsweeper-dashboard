## Summary

Keep open: current main and the latest release still intentionally require API-key-style auth for OpenAI memory embeddings, so this is a real auth/provider feature gap rather than an implemented or stale request.

## What This Changes

Requests routing OpenAI memory embedding calls through Codex OAuth for ChatGPT subscriber profiles instead of requiring `OPENAI_API_KEY`.

## Best Possible Solution

Have maintainers decide whether Codex OAuth may cover embeddings; if approved, implement it in the OpenAI provider/embedding seam with explicit OAuth and API-key regression tests and updated docs.

## Reproduction Assessment

Yes, at source level: current docs, auth tests, and the embedding client path show OpenAI memory embeddings require API-key-style credentials. I did not run a live OAuth profile reproduction.

## Solution Assessment

Unclear until maintainers confirm that Codex OAuth may serve embeddings under subscription auth. If that contract is approved, the OpenAI provider embedding adapter is the narrowest likely implementation boundary.
