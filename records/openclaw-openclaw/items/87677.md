## Label Justifications

- `impact:session-state`: The affected behavior is semantic memory search, which controls retrieval of memory and session context.

## Summary

Keep open: current main and the latest release still make OpenAI memory embeddings an API-key-backed path, while Codex OAuth embeddings would change the documented OpenAI auth and billing contract.

## What This Changes

The issue requests routing `memorySearch.provider: "openai"` embedding calls through Codex OAuth so ChatGPT subscribers do not need a separate OpenAI API key for memory search.

## Best Possible Solution

Have maintainers decide whether Codex OAuth may cover native OpenAI embeddings; if approved, implement a narrowly host-gated OpenAI path with docs, regression tests, and redacted live OAuth proof.

## Reproduction Assessment

Yes. The reporter supplied live Pi output, and current source plus docs still show `memorySearch.provider: "openai"` using API-key-style bearer auth while Codex OAuth is excluded for embeddings.

## Solution Assessment

Unclear until maintainers approve the auth-contract change. If approved, a native-OpenAI-only embedding route guarded to `api.openai.com` is safer than broad auth-order expansion or sending OAuth credentials to custom endpoints.
