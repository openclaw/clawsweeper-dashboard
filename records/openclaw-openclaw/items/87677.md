## Summary

Keep open: current main and v2026.6.8 still require API-key-style auth for OpenAI memory embeddings, the public docs explicitly say Codex OAuth does not satisfy embedding requests, and upstream Codex source does not expose a generic embeddings proxy. This remains a product/auth-provider decision, not conservative cleanup.

## What This Changes

Route OpenAI memory embeddings through Codex OAuth for ChatGPT subscription profiles instead of requiring `OPENAI_API_KEY`.

## Best Possible Solution

Have maintainers decide whether Codex OAuth may cover native OpenAI embeddings; if approved, implement a narrowly host-gated OpenAI embedding path with docs, regression tests for API-key/custom-endpoint behavior, and redacted live OAuth proof.

## Reproduction Assessment

Yes, at source level with high confidence: the reporter provided live Pi output, and current main still routes OpenAI memory embeddings through API-key bearer auth while docs exclude Codex OAuth. I did not run a live OAuth credential flow in this read-only review.

## Solution Assessment

Unclear until maintainers approve the auth-contract change. If approved, a native-OpenAI-only embedding adapter path guarded to `api.openai.com` is safer than broad generic auth-order expansion or routing custom endpoints through Codex OAuth.
