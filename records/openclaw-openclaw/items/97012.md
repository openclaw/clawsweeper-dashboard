## Label Justifications

- `P2`: This is a normal-priority plugin SDK live catalog bug fix with bounded blast radius but real provider-discovery impact.
- `merge-risk: 🚨 availability`: The PR can extend live catalog discovery from one guarded request to many sequential requests, which green unit tests do not prove safe for slow or malformed providers.

## Summary

Keep open: the shared-helper direction is right and current main is source-reproducibly single-page, but the PR should not merge until max-page exhaustion cannot silently return and cache a partial catalog and the external contributor adds real behavior proof.

## What This Changes

The PR updates `src/plugin-sdk/provider-catalog-live-runtime.ts` and its tests so live provider catalog helpers follow `next_cursor`, `next`, `links.next`, and `nextPageToken` pagination hints.

## Best Possible Solution

Keep pagination in the shared helper, but land it only after cap exhaustion fails or falls back instead of returning partial rows, the live discovery path remains bounded, and after-fix real behavior proof is attached.

## Reproduction Assessment

Yes. Current main's `fetchLiveProviderModelRows` reads only the first JSON response, so a mocked paginated catalog with a second page would lose later rows without needing a broad runtime setup.

## Solution Assessment

No. The shared helper is the right layer, but the submitted shape should treat page-cap exhaustion as incomplete discovery and preserve a bounded total refresh path before it is the best mergeable fix.
