## Summary

Keep open for maintainer review: this is a real gateway/device-pairing API proposal, but the current PR head is syntactically invalid, lacks real behavior proof, and changes auth token denial contracts in a security-sensitive way.

## What This Changes

This PR changes device-pairing token helpers, gateway handlers, websocket handshake token issuance, and tests to return structured error objects for scope-escalation and missing-token cases instead of existing null or ok/reason results.

## Best Possible Solution

Keep the current public denial contracts; if richer diagnostics are desired, add non-breaking internal log context or design a versioned device-pairing API with maintainer approval.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough to reproduce the parse blockers, and current main source/tests show the stable contracts this PR changes.

## Solution Assessment

No. Replacing nullable and ok/reason contracts plus exposing differentiated revoke failures is not the narrowest maintainable path without an explicit API and security decision.
