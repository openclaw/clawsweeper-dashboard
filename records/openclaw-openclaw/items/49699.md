## Summary

Keep this PR open, but do not merge it as-is. Current main still uses the legacy device-token contracts, and the PR branch has parse-breaking TypeScript plus a security-sensitive revoke response regression that needs focused rework.

## What This Changes

The PR changes device-pairing token helpers, gateway handlers, WebSocket handshake code, and tests to return and handle structured error objects instead of null or ok/reason-only denial results.

## Best Possible Solution

Rework the branch around the existing token-result API: preserve ok/reason discriminators or add structured fields compatibly, keep public rotate/revoke denial responses generic, log structured scope details server-side, fix the malformed gateway edits, and update the focused infra/gateway tests for scope escalation and token denial paths.

## Reproduction Assessment

Yes. The downloaded PR diff is enough to reproduce the immediate failure because both changed gateway files contain invalid TypeScript syntax; current main source and tests also provide a clear baseline for the legacy token contracts.

## Solution Assessment

No. The structured context is useful, but replacing stable discriminators and exposing differentiated revoke messages is not the safest path; a compatible result shape plus server-side details is the narrower maintainable solution.
