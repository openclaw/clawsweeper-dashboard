## Summary

## What This Changes

## Best Possible Solution

## Reproduction Assessment

Yes, source-level reproduction is high confidence: register old and new connections for the same node id, close the old connection, and verify the new session plus invoke ownership remain intact. Current main codifies that path in src/gateway/node-registry.test.ts, so the old failure no longer reproduces on main.

## Solution Assessment

No, this PR is not the best way to solve the issue because it returns a node id for stale unregisters and skips old pending-invoke cleanup. The already-merged current-main fix is narrower and more complete because it keys pending invokes by connId and returns null for stale closes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return null for stale node closes:** `src/gateway/node-registry.ts:92-94`
  - body: The stale-connection branch returns `nodeId`, but `server/ws-connection.ts` treats any non-null unregister result as a current node disconnect and removes node runtime state. A stale old socket close should return `null`, as current main now does, so the replacement session is not announced as disconnected.
  - confidence: 0.9
- **[P2] Clean up old pending invokes before returning:** `src/gateway/node-registry.ts:92-94`
  - body: Returning early for a stale close bypasses the pending-invoke cleanup loop, leaving invokes sent on the old connection alive until timeout. The landed fix stores the pending `connId` and rejects only invokes owned by the closing connection.
  - confidence: 0.86
