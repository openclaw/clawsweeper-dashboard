## Summary

Keep open. Current main still has a plausible bundled-channel loader mismatch: Telegram declares and stages `grammy`, but the full bundled channel fallback loads Telegram with runtime-dependency staging disabled and then imports code that reaches `grammy`, matching the reported warning while the live plugin path can still work.

## What This Changes

The issue reports that after updating to OpenClaw 2026.4.29, Telegram works but bundled channel/status loading warns that `grammy` is missing because Telegram code resolves from the global package context instead of the staged plugin runtime-deps context.

## Best Possible Solution

Keep the issue open and make a focused loader or Telegram light-surface fix so full bundled channel fallbacks either resolve through the prepared staged runtime root or avoid importing heavy Telegram runtime code on status/discovery paths, with regression coverage for a packaged external-stage install where `grammy` exists only in the stage directory.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a packaged Telegram plugin with `OPENCLAW_PLUGIN_STAGE_DIR` containing `grammy` but no `grammy` in the package root, then calling a full bundled-channel fallback such as `getBundledChannelPlugin("telegram")`; current code passes `installRuntimeDeps: false` and imports Telegram code that reaches `grammy`.

## Solution Assessment

Yes in direction, but the maintainable fix should not add `grammy` to the global OpenClaw package tree. The safer fix is to keep lightweight entry discovery cold while ensuring full runtime fallback loads use the staged runtime context or a lighter Telegram status surface.

## Repair Work Prompt

Fix openclaw/openclaw#75733. In packaged installs, Telegram declares `grammy` as a staged bundled runtime dependency, but the current full bundled-channel fallback calls `entry.loadChannelPlugin({ installRuntimeDeps: false })` and then imports the Telegram channel graph, which reaches `grammy` from the package/global context and logs `[channels] failed to load bundled channel telegram` even when the runtime deps were staged successfully. Implement a narrow fix so full bundled channel plugin loads used by runtime/status/doctor fallbacks resolve from the prepared/staged runtime root, or split the Telegram light status/setup surface so those paths do not import heavy runtime deps. Do not add `grammy` to root/global OpenClaw dependencies, do not mutate `node_modules`, and keep bundled-plugin boundary rules intact. Add a regression fixture with an external `OPENCLAW_PLUGIN_STAGE_DIR` where the dependency exists only in the staged root and assert the affected fallback no longer warns/fails while entry-only discovery remains light. Add a single-line `CHANGELOG.md` Fixes entry crediting @coolmanns unless maintainers classify it as internal-only.
