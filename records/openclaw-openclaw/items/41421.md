## Summary

Keep open: current main still does not provide this CLI command and the linked request was closed in favor of this PR, but the branch is not mergeable as-is because it imports moved Telegram/Signal internals, duplicates live prompt assembly, and lacks contributor real-behavior proof.

## What This Changes

The PR adds an `openclaw agents view-system-prompt [agentId]` subcommand that renders a static system-prompt preview with `--model` and `--channel` options plus stderr metadata.

## Best Possible Solution

Expose prompt inspection through a shared maintained prompt-preview path that reuses embedded attempt assembly where possible, keeps channel behavior behind plugin helpers, adds focused CLI regression coverage, and documents any unavoidable static-preview limits.

## Reproduction Assessment

Yes for the PR findings: source inspection of the PR diff and current main shows missing core imports and prompt-assembly divergence. I did not run the command because this is a read-only review.

## Solution Assessment

No. Duplicating prompt assembly in the command is brittle here; a shared preview helper or explicit static-preview contract would be the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route channel prompt helpers through plugins:** `src/commands/agents.commands.view-system-prompt.ts:31-33`
  - body: These imports target `src/signal` and `src/telegram`, but those modules do not exist on current main and core command code should not import plugin internals directly. Use the generic channel plugin prompt helpers so the branch can build and stays behind the plugin boundary.
  - confidence: 0.97
- **[P2] Collect runtime channel capabilities for previews:** `src/commands/agents.commands.view-system-prompt.ts:78-99`
  - body: This path uses config-only capabilities plus hardcoded Telegram handling. Real embedded runs normalize the channel and call `collectRuntimeChannelCapabilities`, which also includes plugin prompt capabilities and thread-bound/native approval capabilities, so `--channel` previews can omit instructions the agent actually receives.
  - confidence: 0.91
- **[P2] Build runtime info from the agent workspace:** `src/commands/agents.commands.view-system-prompt.ts:101-106`
  - body: Passing `process.cwd()` makes repo/source resolution depend on where the operator launches the CLI. Embedded runs build prompt params with the effective agent workspace as `cwd`, so this can render a different Runtime section for the selected agent.
  - confidence: 0.88
- **[P2] Share attempt prompt assembly instead of rendering directly:** `src/commands/agents.commands.view-system-prompt.ts:164-187`
  - body: Calling `buildEmbeddedSystemPrompt` directly skips attempt-level handling for prompt overrides, provider transforms/contributions, source path, native command guidance, sandbox info, heartbeat policy, and effective tool filtering. Configs using those hooks will see a preview that is not the submitted prompt.
  - confidence: 0.87
