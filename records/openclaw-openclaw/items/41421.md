## Review Findings

Overall confidence: 0.92

Full review comments:

- **[P1] Route channel prompt helpers through plugins:** `src/commands/agents.commands.view-system-prompt.ts:31-33`
  - body: These imports point at `src/signal` and `src/telegram`, but those modules do not exist on current main and core command code should not import plugin internals directly. Use the generic channel plugin prompt helpers instead so the branch can build and stays behind the plugin boundary.
  - confidence: 0.97
- **[P2] Resolve channel capabilities through the runtime collector:** `src/commands/agents.commands.view-system-prompt.ts:78-99`
  - body: This path uses config-only capabilities plus hardcoded Telegram handling. Real embedded runs normalize the channel and call `collectRuntimeChannelCapabilities`, which also includes plugin prompt capabilities and thread-bound spawn/native approval capabilities, so `--channel` previews can omit instructions the agent actually receives.
  - confidence: 0.91
- **[P2] Use the agent workspace when building runtime info:** `src/commands/agents.commands.view-system-prompt.ts:101-106`
  - body: Passing `process.cwd()` makes repo/source resolution depend on where the operator launches the CLI. Embedded runs build prompt params with the effective agent workspace as `cwd`, so this can render a different Runtime section for the selected agent.
  - confidence: 0.88
- **[P2] Build through the attempt prompt path:** `src/commands/agents.commands.view-system-prompt.ts:164-187`
  - body: Calling `buildEmbeddedSystemPrompt` directly skips current attempt-level handling for prompt overrides, provider transforms/contributions, source path, native command guidance, sandbox info, heartbeat policy, and effective tool filtering. Configs using those hooks will see a preview that is not the submitted prompt.
  - confidence: 0.87
