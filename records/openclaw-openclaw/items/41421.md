## Summary

Keep open. Current main still does not implement the requested prompt-preview command, and the PR direction is useful, but the branch is not mergeable as-is because it targets stale channel internals and duplicates prompt assembly that now diverges from the embedded runtime path.

## What This Changes

The PR adds an `openclaw agents view-system-prompt [agentId]` CLI subcommand, a new command handler that assembles and prints a local system prompt, `--model` and `--channel` overrides, and stderr metadata.

## Best Possible Solution

Land a rebased implementation that reuses or extracts the current embedded-run prompt-preview composition path, then add focused CLI docs and parity tests for model override, channel prompt capabilities, workspace cwd, heartbeat policy, provider/plugin prompt contributions, sandbox/tool-policy limitations, and workspace notes.

## Reproduction Assessment

Yes. Static inspection is enough: current main has no command, while the PR diff imports modules absent from current main and assembles the prompt through a path that differs from `runEmbeddedAttempt`.

## Solution Assessment

No. Adding the command is a useful direction, but duplicating prompt assembly inside the CLI handler is not the narrowest maintainable solution unless it shares the same runtime seams used by actual embedded runs.

  - body: These imports target `src/signal` and `src/telegram` modules that no longer exist on current main, so the branch will fail after rebase. Use the generic channel/plugin prompt helpers or plugin API exports instead of direct channel internals.

  - body: This calls only `resolveChannelCapabilities`, but real runs use `collectRuntimeChannelCapabilities`, which also merges plugin-owned prompt capabilities. Channel previews can omit capabilities the real prompt includes.
  - confidence: 0.9
- **[P2] Use the agent workspace as cwd:** `src/commands/agents.commands.view-system-prompt.ts:102`
  - body: Passing `process.cwd()` makes repo/runtime fields depend on the shell launch directory. Embedded runtime passes the effective agent workspace as `cwd`, so this can print a different prompt than a real run.
  - confidence: 0.92
- **[P2] Preserve runtime workspace notes:** `src/commands/agents.commands.view-system-prompt.ts:127-130`
  - body: Injecting `Loaded context files: ...` through `workspaceNotes` adds prompt text production does not send and misses the runtime BOOTSTRAP/embedded-mode reminders production may send. The file contents already flow through `contextFiles`.
  - confidence: 0.87
- **[P2] Gate heartbeat guidance like runtime:** `src/commands/agents.commands.view-system-prompt.ts:151-153`
  - body: This injects heartbeat guidance for the default agent regardless of trigger and heartbeat policy. Normal-run previews can therefore include heartbeat instructions the agent would not receive.
  - confidence: 0.88

  - body: The command prints the direct `buildEmbeddedSystemPrompt` result, while runtime resolves provider prompt contributions and then applies provider/plugin system-prompt transforms. Provider overlays can be missing from the displayed prompt.
