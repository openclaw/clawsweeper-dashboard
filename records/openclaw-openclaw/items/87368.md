## Summary

Keep open for maintainer decision: current main and v2026.5.26 both suppress Discord room-event ack reactions, but current tests and ambient-room-event docs treat room events as quiet visible-delivery mode rather than an obvious broken code path.

## What This Changes

The issue asks for `messages.ackReactionScope: "all"` to send Discord ack reactions even when the inbound event is classified as `room_event`.

## Best Possible Solution

Settle the room-event reaction policy once across supported channels: either document that ambient room events stay reaction-silent despite `all`, or add a named opt-in/config contract with Discord, Slack, and Telegram coverage.

## Reproduction Assessment

Yes from source inspection: current Discord code short-circuits ack reactions for `room_event`, and the adjacent test expects no reactions with `ackReactionScope: "all"`. I did not run live Discord because this review is read-only and current tests already prove the current-main behavior path.

## Solution Assessment

Unclear: Option A would be mechanically small, but it would reverse the tested quiet-room-event contract shared by major channel surfaces. A maintainer should first choose whether this is a docs clarification or a new explicit reaction-visible room-event mode.
