## Summary

Keep open: current main and the latest release still suppress Discord ack reactions for `room_event` inbounds before the shared ack-scope helper can honor `ackReactionScope: "all"`; the open linked PR at https://github.com/openclaw/openclaw/pull/87433 is the active path for the policy/code decision.

## What This Changes

The issue reports that `messages.ackReactionScope: "all"` is documented and implemented in the shared helper as unconditional, but Discord ambient `room_event` messages are still gated out before that helper runs.

## Best Possible Solution

Use the linked PR or a maintainer-owned replacement to settle the Discord/Slack/Telegram policy, then land matching runtime behavior, focused tests, and public docs or explicitly document that room events remain reaction-silent.

## Reproduction Assessment

Yes from source inspection: current main gates Discord ack reactions on `!isRoomEvent`, and the adjacent test expects no reaction with `ackReactionScope: "all"` on a room event. I did not run a live Discord workspace reproduction in this read-only review.

## Solution Assessment

Unclear: the narrow code fix is to let the explicit `all` scope bypass the room-event ack gate, but that changes tested quiet-room behavior across sibling chat channels. The best path is maintainer review of the linked cross-channel PR rather than a separate automated fix.
