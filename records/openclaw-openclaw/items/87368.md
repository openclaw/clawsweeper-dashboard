## Summary

Keep open: current main and latest release still suppress configured ack reactions for `room_event` inbounds despite `ackReactionScope: "all"`, and the linked open draft PR is the active path for resolving the cross-channel policy.

## What This Changes

The issue asks OpenClaw to honor `messages.ackReactionScope: "all"` for ambient `room_event` messages instead of silently suppressing the ack reaction.

## Best Possible Solution

Use the linked PR or a maintainer-owned replacement to settle the cross-channel policy, then land matching runtime changes, tests, and public docs or explicitly document that room events stay reaction-silent.

## Reproduction Assessment

Yes, from source inspection: current main gates Discord ack reactions on `!isRoomEvent`, and Discord, Slack, and Telegram tests expect no ack reaction for room events even with `ackReactionScope: "all"`. I did not run tests because this review is read-only.

## Solution Assessment

Unclear: honoring `all` for room events is mechanically plausible and already proposed, but it reverses the current quiet-room-event contract across sibling chat plugins. Maintainers should decide the policy through the linked PR before merging or closing this issue.
