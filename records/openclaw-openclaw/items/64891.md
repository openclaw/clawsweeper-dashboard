## Summary

Keep open. Current main still sends Bedrock memory auto-selection through `hasAwsCredentials()`, and that helper still invokes the AWS SDK default credential chain that can fall through to IMDS; the direct no-IMDS PR was closed unmerged and no open PR currently owns this exact fix.

## What This Changes

The issue asks OpenClaw to stop producing repeated startup requests to AWS IMDS at `169.254.169.254` in sandboxed, non-AWS environments.

## Best Possible Solution

Add a passive, no-network credential eligibility check for Bedrock memory auto-selection and status surfaces, while preserving the full AWS SDK default credential chain for explicit Bedrock runtime use.

## Reproduction Assessment

Yes. The reporter provided concrete sandbox logs, and current source plus the AWS SDK credential-chain contract gives a high-confidence path from memory auto-selection to IMDS requests, though I did not live-run the NemoClaw setup.

## Solution Assessment

No current fix is in place. The narrow maintainable solution is to separate passive Bedrock auto-selection eligibility from actual Bedrock runtime authentication, not to disable the AWS default chain globally.

## Repair Work Prompt

Fix the Bedrock memory auto-selection IMDS regression from https://github.com/openclaw/openclaw/issues/64891. Current main calls `hasAwsCredentials()` from `extensions/amazon-bedrock/memory-embedding-adapter.ts` during passive memory provider auto-selection, and that helper invokes `@aws-sdk/credential-provider-node`'s default chain, which can probe IMDS. Implement a passive/no-network Bedrock credential eligibility path for auto-selection and any status/preflight surfaces that should not make network calls, while preserving the full AWS SDK default credential chain for explicit Bedrock runtime usage and real Bedrock invocations. Do not globally disable IMDS for explicit Bedrock use. Add regression coverage that proves auto-selection does not call the AWS SDK default provider/IMDS path when no explicit credentials or passive local config are present, and that explicit/provider runtime paths still support AWS SDK-authenticated setups. Review closed PR https://github.com/openclaw/openclaw/pull/64944 for context but avoid the global IMDS-disable approach called out there. Add a changelog fix entry because this is user-visible.
