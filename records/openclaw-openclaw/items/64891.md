## Summary

Keep open. Current main still lets Bedrock memory auto-selection call `hasAwsCredentials()`, and that helper still invokes the AWS SDK default credential chain that can fall through to IMDS; the merged auto-selection fix solved a neighboring failure mode, but the no-IMDS startup probe problem is not implemented on main.

## What This Changes

The issue asks OpenClaw to stop producing repeated AWS IMDS requests during startup in sandboxed non-AWS environments after Bedrock memory credential auto-detection was added.

## Best Possible Solution

Add a passive, no-network Bedrock credential eligibility path for memory auto-selection and status surfaces while preserving the full AWS SDK default credential chain for explicit Bedrock runtime use.

## Reproduction Assessment

Yes, at source level. The reporter supplied concrete NemoClaw/OpenShell logs, and current main plus the pinned AWS SDK source shows the Bedrock auto-selection path can still reach IMDS through `defaultProvider()` when no earlier credentials resolve.

## Solution Assessment

Yes, keeping this open for a narrow Bedrock memory credential auto-selection fix is the best path. A global IMDS-disable change is not the best solution because it can break EC2 instance-role auth; the safer alternative is a passive eligibility probe only for auto-selection/status paths.
