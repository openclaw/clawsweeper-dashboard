## Summary

Keep open. Current main still has the Bedrock memory auto-selection path calling a network-capable AWS SDK credential probe, so the reported startup IMDS traffic is still source-reproducible despite the merged adjacent auto-select gate.

## What This Changes

The issue reports repeated AWS IMDS requests to `169.254.169.254` during OpenClaw startup in NVIDIA OpenShell/NemoClaw after upgrading from `2026.3.11` to `2026.4.9`.

## Best Possible Solution

Split passive Bedrock memory eligibility detection from runtime Bedrock authentication so startup auto-selection never probes IMDS, while explicit Bedrock use continues to rely on the AWS SDK default credential chain.

## Reproduction Assessment

Yes, as a source-backed reproduction path. The reporter's NemoClaw/OpenShell steps show the live symptom, and current main still routes memory auto-selection through `hasAwsCredentials()` into the pinned AWS SDK chain that probes IMDS.

## Solution Assessment

No current fix is in place. The merged adjacent gate improves Bedrock skipping semantics but still uses the network-capable helper, so the narrow maintainable fix is a passive no-IMDS credential check for auto-selection only.

## Repair Work Prompt

Fix the Bedrock memory startup IMDS regression reported in https://github.com/openclaw/openclaw/issues/64891. In current main, `extensions/amazon-bedrock/memory-embedding-adapter.ts` calls `hasAwsCredentials()` during memory provider auto-selection, and `extensions/amazon-bedrock/embedding-provider.ts` then calls the AWS SDK default credential chain, which can probe IMDS at `169.254.169.254`.
