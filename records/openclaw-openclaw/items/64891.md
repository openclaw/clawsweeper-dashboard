## Summary

Keep open. Current main still lets Bedrock memory auto-selection call the AWS SDK default credential chain, and the pinned SDK path can still fall through to IMDS; the prior no-IMDS PR is closed unmerged and the merged auto-select gate fixed a different failure mode.

## What This Changes

The issue asks OpenClaw to stop repeated Bedrock/AWS IMDS requests to 169.254.169.254 during startup credential auto-detection in sandboxed environments.

## Best Possible Solution

Add a passive, no-network Bedrock credential eligibility check for memory auto-selection/status surfaces, while preserving the full AWS SDK default credential chain for explicit Bedrock runtime use.

## Reproduction Assessment

Yes at source level. The reporter provided concrete sandbox logs, and current source plus the AWS SDK source contract gives a high-confidence path from Bedrock memory auto-selection to IMDS requests, though I did not live-run the NemoClaw setup.

## Solution Assessment

No current fix is in place for the repeated IMDS probes. The narrow maintainable solution is to separate passive Bedrock auto-selection eligibility from actual Bedrock runtime authentication, not to disable the AWS default chain globally.
