## Summary

Keep open. Current main still routes Bedrock memory auto-selection through `hasAwsCredentials()`, and that helper still invokes the AWS SDK default credential chain that can fall through to IMDS. The direct fix PR is closed unmerged, while the merged adjacent PR added the current credential gate but did not remove the reported IMDS probe source.

## What This Changes

The issue asks OpenClaw to stop making repeated AWS IMDS requests to `169.254.169.254:80` during startup in sandboxed or non-AWS environments when Bedrock credential auto-detection runs.

## Best Possible Solution

Add a focused Amazon Bedrock plugin fix that separates passive memory-embedding credential eligibility from actual Bedrock runtime authentication, so startup auto-selection does not contact IMDS in non-AWS sandboxes while explicit Bedrock usage and real AWS SDK default-chain flows still work. Lock the behavior with unit tests for `AWS_EC2_METADATA_DISABLED`, no repeated IMDS/defaultProvider probes during auto-selection, env restoration if env is temporarily changed, and static/profile/SSO/role-compatible paths.

## Reproduction Assessment

Yes. The issue has a detailed live NemoClaw/openshell reproduction, and current source gives a high-confidence static reproduction path: memory auto-selection calls the Bedrock adapter, the adapter calls `hasAwsCredentials()`, and that helper invokes the AWS SDK default credential chain that can fall through to IMDS.

## Solution Assessment

No current solution is in place. The prior #64944 direction addressed the unwanted network calls but had EC2 instance-role regression risk; the safer fix should be scoped to passive eligibility detection or bounded/coalesced probing while preserving runtime Bedrock auth semantics.
