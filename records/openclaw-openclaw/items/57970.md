## Summary

Keep this PR open. Current main and the latest release still have the reported Mattermost nullish channel-type fallback, but the proposed patch is not ready because it can still drop recoverable posts when the WebSocket event sends a blank `channel_type`, and the branch needs maintainer/author follow-up before merge.

## What This Changes

The PR changes Mattermost nullish channel-type mapping to `unknown`, adds fail-closed guards in Mattermost post/button/reaction/session routing, updates related tests and auth types, and widens shared chat-type/routing-adjacent typing.

## Best Possible Solution

Keep this PR open, rebase it onto current main, carry a trusted trimmed non-empty event-or-API channel type through `handlePost` before dedupe/drop decisions, and decide whether `unknown` belongs in shared `ChatType` or should remain Mattermost-local before landing with targeted regression coverage.

## Reproduction Assessment

Yes. Current main is reproducible through the pure mapper test expectation for `undefined -> "channel"`, and the PR regression is statically reproducible with `payload.data.channel_type = ""` plus `resolveChannelInfo(channelId).type` returning a valid Mattermost type.

## Solution Assessment

No, not as currently patched. The fail-closed direction is correct, but the implementation should use the API fallback for blank event values and settle the public `ChatType` contract before merge.
