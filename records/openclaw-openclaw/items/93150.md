## Label Justifications

- `P2`: The issue affects memory recall availability in constrained runtimes but has a limited memory-core blast radius and a supported Node upgrade path.
- `impact:session-state`: The affected surface is `memory_search` recall over user memory files and session-adjacent memory state.

## Summary

Keep open: current main and v2026.6.6 still route missing `node:sqlite` through the SQLite-backed memory manager into an unavailable `memory_search` response, while related open PRs cover only error copy, diagnostics, or a broader backend swap.

## What This Changes

The issue requests a bounded keyword/file fallback for `memory_search` when `node:sqlite` is unavailable and SQLite-backed memory search cannot initialize.

## Best Possible Solution

Make an explicit maintainer-approved memory-core fallback decision, then implement either a bounded local corpus scan for missing SQLite setup or fold this case into the chosen SQLite backend strategy with docs and regression coverage.

## Reproduction Assessment

Yes at source level: `requireNodeSqlite()` throws when `node:sqlite` is missing, `MemoryIndexManager` cannot initialize, and `memory_search` returns an unavailable payload. I did not live-reproduce it in this checkout because local Node v24.16.0 has `node:sqlite` available.

## Solution Assessment

Unclear until maintainers choose the fallback policy. A bounded memory-core text scan is plausible, but repository policy treats runtime fallbacks as product decisions and related SQLite backend work is still open.
