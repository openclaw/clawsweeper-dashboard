## Summary

Keep open: current main and v2026.6.9 still return unavailable metadata when the builtin memory manager cannot load `node:sqlite`; https://github.com/openclaw/openclaw/pull/93260 is the exact open closing implementation candidate, and the remaining question is maintainer acceptance of degraded fallback behavior.

## What This Changes

Request to add a bounded markdown keyword fallback for `memory_search` when `node:sqlite` is unavailable, searching configured memory files instead of returning only unavailable metadata.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/93260 or explicitly reject degraded keyword fallback; if accepted, keep SQLite/vector search canonical when available and bound fallback reads to the existing memory corpus helpers.

## Reproduction Assessment

Yes, at source level: missing `node:sqlite` throws through the memory host SDK, `getBuiltinMemorySearchManager` catches it as an error, and `memory_search` formats an unavailable result. I did not run a downlevel Node binary in this read-only review.

## Solution Assessment

Unclear until maintainers accept the policy: the linked PR targets the right manager-resolution seam if degraded recall is desired, but fallback behavior is a product choice rather than a strict bug fix.
