## Summary

Keep open: current main and the latest release still return an unavailable builtin memory manager when SQLite initialization fails, and the exact keyword-fallback behavior is owned by an open closing PR that has not merged yet.

## What This Changes

The issue requests a bounded keyword fallback for memory_search over local memory markdown files when SQLite-backed memory search cannot initialize because node:sqlite is unavailable.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/93260 is merged, rejected, or replaced by a maintainer-approved fallback direction.

Yes at source level: current main routes builtin memory search through MemoryIndexManager, node:sqlite load failure becomes an error from the host SDK, and the manager resolver returns unavailable metadata instead of keyword results. I did not live-run an affected Node runtime in this read-only review.

Yes for triage: the best current path is to keep this canonical issue open and review the linked implementation candidate. The fallback behavior itself remains a maintainer product decision because it changes the missing-SQLite response from unavailable to degraded recall.
