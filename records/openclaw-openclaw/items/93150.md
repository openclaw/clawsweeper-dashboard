## Summary

Keep open: current main and v2026.6.9 still return unavailable metadata when the SQLite-backed memory manager cannot initialize, while the exact keyword-fallback implementation is already owned by the open linked PR.

## What This Changes

Add a bounded keyword fallback for `memory_search` over configured memory Markdown files when the builtin SQLite-backed manager cannot initialize because `node:sqlite` is unavailable.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/93260 as the fallback implementation or explicitly keep unavailable-message guidance as the supported behavior.

## Reproduction Assessment

Yes at source level: current main catches builtin manager initialization failures and returns unavailable metadata, and the issue plus linked PR provide the missing-`node:sqlite` failure path. I did not live-run a downlevel Node runtime in this read-only review.

## Solution Assessment

Yes as a candidate direction if maintainers accept the fallback policy: the linked PR targets the manager-resolution seam and keeps SQLite/vector search canonical when available. The remaining question is whether degraded keyword fallback should be supported behavior.
