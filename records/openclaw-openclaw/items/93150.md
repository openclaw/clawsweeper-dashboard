## Summary

Keep open: current main and v2026.6.8 still route builtin memory search through the SQLite-backed manager with no file keyword fallback, and https://github.com/openclaw/openclaw/pull/93260 is the open closing implementation candidate.

## What This Changes

Request a bounded keyword fallback over configured memory files when `memory_search` cannot initialize because `node:sqlite` is unavailable.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/93260 as the fallback implementation or explicitly choose the unavailable-message-only path instead.

## Reproduction Assessment

Yes at source level: current main catches builtin manager initialization failures and returns an unavailable error, and the issue plus linked PR provide the missing-`node:sqlite` reproduction. I did not live-run a downlevel Node build in this read-only review.

## Solution Assessment

Yes as a triage path: the open memory-core PR targets the manager-resolution seam and leaves SQLite/vector search canonical when available. The fallback policy still needs maintainer acceptance before the issue can close.
