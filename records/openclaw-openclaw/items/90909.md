## Summary

Keep open: current main and the latest release still do not populate the embedded Lobster `ctx.llmAdapters` bridge, and the open implementation PR remains unmerged with provider-contract and security-review blockers.

## What This Changes

Request to add a no-credential in-process LLM adapter bridge for embedded Lobster workflows so `llm.invoke` can run without Gateway URL or bearer-token environment injection.

## Best Possible Solution

Settle and land a maintainer-approved embedded Lobster LLM adapter contract that reuses OpenClaw's existing LLM policy path, avoids env credential injection, and documents the chosen provider name.

## Reproduction Assessment

Yes. Source and dependency inspection show current main does not populate `ctx.llmAdapters`, while Lobster's `llm.invoke` requires that field for direct in-process adapters and current docs still describe the embedded bridge as missing.

## Solution Assessment

Unclear until maintainer direction is settled. The in-process adapter direction is plausible and safer than credential env injection, but the permanent provider name, policy reuse, and security boundary need owner approval before implementation lands.
