## Summary

Keep open: current main still documents embedded Lobster `openclaw.invoke`/LLM steps as unsupported, and source plus the Lobster dependency contract show the requested `ctx.llmAdapters` bridge is not implemented. This should not close as a duplicate of the older docs-only fix or the closed unmerged bridge PR because the no-credential in-process adapter decision remains unresolved.

## What This Changes

The issue requests a host-provided in-process LLM adapter/registry context for embedded Lobster so workflows can run `llm.invoke` without Gateway URL or bearer-token env exposure.

## Best Possible Solution

Define a maintainer-approved embedded Lobster LLM adapter contract that uses `ctx.llmAdapters` and any minimal registry needed, reuses OpenClaw's embedded agent runtime, and avoids placing Gateway operator credentials in workflow env.

## Reproduction Assessment

Yes. Source and dependency inspection show current main does not populate `ctx.llmAdapters`, while Lobster's `llm.invoke` needs that field for in-process adapters and current docs still say the embedded bridge is missing.

## Solution Assessment

Unclear pending maintainer direction. The proposed adapter approach is safer than credential env injection, but the permanent provider name, registry exposure, and plugin boundary should be decided before implementation.
