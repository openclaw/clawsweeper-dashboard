## Label Justifications

- `P2`: This is a bounded embedded workflow feature gap with strong source evidence, but it needs product and security review rather than urgent regression handling.
- `impact:security`: The central design constraint is avoiding exposure of full-operator Gateway credentials to Lobster workflow shell steps.
- `impact:auth-provider`: The requested bridge changes how embedded workflows route LLM provider/model/auth execution from HTTP bearer auth to an in-process adapter.

## Summary

Keep open: current main and the latest release still lack the requested embedded Lobster `ctx.llmAdapters` bridge, and the active implementation path is the open PR at https://github.com/openclaw/openclaw/pull/91028, which still needs maintainer decisions around provider naming, policy reuse, credential boundaries, and live proof.

## What This Changes

Request to populate the embedded Lobster context with a host-provided in-process LLM adapter and minimal registry so workflows can call `llm.invoke` without a gateway URL or bearer token.

## Best Possible Solution

Define and land a maintainer-approved embedded Lobster LLM adapter contract that reuses the `llm-task`/embedded-agent policy boundary, avoids Gateway credential env injection, documents the provider name, and includes live redacted invocation proof.

## Reproduction Assessment

Yes. Source inspection shows current main does not populate `ctx.llmAdapters`, while the Lobster dependency requires that field for direct in-process `llm.invoke` adapters and keeps `openclaw.invoke` as an HTTP/env bridge.

## Solution Assessment

Unclear pending maintainer direction. The in-process adapter direction is safer than credential env injection, but provider naming, registry exposure, and reuse of `llm-task` policy need an explicit decision before implementation lands.
