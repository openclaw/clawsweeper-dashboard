## Summary

Keep open: current main and latest release still do not populate the embedded Lobster context with an in-process LLM adapter, and the docs still describe the embedded bridge as missing. The paired open implementation PR should remain the candidate path, but the adapter contract, provider naming, live proof, and security boundary still need maintainer judgment.

## What This Changes

Request to populate embedded Lobster `ctx.llmAdapters` and possibly `ctx.registry` with a no-credential in-process LLM adapter so workflows can use `llm.invoke` without Gateway URL/token env.

## Best Possible Solution

Define and land a maintainer-approved Lobster embedded LLM bridge in the Lobster plugin that uses the dependency's direct-adapter seam, preserves OpenClaw LLM auth/model policy, avoids Gateway credentials in workflow env, updates docs, and includes live redacted invocation proof.

## Reproduction Assessment

Yes, at source-contract level. Current main declares `ctx.llmAdapters`/`ctx.registry` but does not populate them, while the Lobster dependency only takes the in-process `llm.invoke` path when a matching direct adapter is present.

## Solution Assessment

Unclear pending maintainer direction. The `ctx.llmAdapters` approach matches the dependency seam and is safer than credential env injection, but provider naming, registry exposure, and reuse of host LLM policy still need a product/security decision.
