Keep open: current main and v2026.6.9 still do not populate the embedded Lobster runner's in-process LLM adapter context, and the active implementation PR is the right candidate path but remains unmerged and security/product-sensitive.

## What This Changes

The issue requests a host-provided in-process LLM adapter for embedded Lobster so workflows can use `llm.invoke` without injecting Gateway URL or bearer-token credentials.

## Best Possible Solution

Review and land or replace the active implementation PR with a maintainer-approved embedded LLM adapter contract that avoids credential env injection, preserves LLM policy, documents the provider name, and includes live embedded invocation proof.

## Reproduction Assessment

## Solution Assessment

Unclear until maintainers settle the permanent bridge contract. The in-process adapter direction is safer than putting Gateway credentials in env, but provider naming, credential masking, and LLM auth/model policy need explicit review.
