## Summary

## What This Changes

The issue requests an in-process LLM adapter bridge for embedded Lobster workflows so `llm.invoke` can run without gateway URL or bearer-token exposure.

## Best Possible Solution

Keep this issue as the canonical product/security discussion until https://github.com/openclaw/openclaw/pull/91028 either lands with a maintainer-approved adapter contract and live proof or is replaced by a narrower approved design.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` declare Lobster context adapter fields but do not populate them, while the Lobster dependency uses `ctx.llmAdapters` for direct `llm.invoke` adapters. I did not run a live model workflow.

## Solution Assessment

Unclear pending maintainer direction. The in-process adapter direction is safer than credential env injection, but the provider name, registry exposure, and llm-task auth/model policy need to be settled before this is the best final fix.
