## Summary

Keep open: current main and v2026.6.9 still do not populate the embedded Lobster runner's in-process LLM adapter context, and the open implementation candidate at https://github.com/openclaw/openclaw/pull/91028 remains unmerged with product/security and live-proof blockers.

## What This Changes

The issue requests host-provided in-process Lobster LLM adapters so embedded workflows can use `llm.invoke` without gateway URL or bearer-token environment injection.

## Best Possible Solution

Review and land or replace the active implementation PR with a maintainer-approved adapter contract, provider name, policy preservation, credential-safe environment handling, and redacted live embedded LLM proof.

## Reproduction Assessment

Yes, by source and dependency inspection: current main and v2026.6.9 do not populate `ctx.llmAdapters`, while the Lobster dependency only takes the direct in-process LLM path when that context field exists. I did not run a live embedded model workflow in this read-only review.

## Solution Assessment

Unclear as a final product contract. The in-process adapter direction is safer than injecting Gateway credentials, but provider naming, LLM policy preservation, credential masking, and live proof remain open in the active implementation PR.
