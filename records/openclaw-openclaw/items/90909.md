## Summary

Keep open: current main and the latest release still leave the embedded Lobster LLM bridge unimplemented, and the linked open implementation PR is the right place to resolve the remaining product/security contract before this issue closes.

## What This Changes

Request to populate the embedded Lobster runner with host-provided in-process LLM adapters so `llm.invoke` can run without gateway URL or bearer-token environment exposure.

## Best Possible Solution

Define and land a maintainer-approved embedded Lobster LLM adapter contract that uses the existing in-process model runtime, preserves `llm-task` policy or documents an intentional replacement, and never exposes gateway credentials to workflow steps.

## Reproduction Assessment

Yes. Source inspection shows the embedded runner passes a context without `llmAdapters` or `registry`, while docs and the Lobster dependency show those fields are the in-process path needed for `llm.invoke`.

## Solution Assessment

Unclear pending maintainer direction. The in-process adapter approach is plausible and safer than credential env injection, but provider naming, registry exposure, and `llm-task` auth/model policy need a product/security decision before implementation is the best fix.
