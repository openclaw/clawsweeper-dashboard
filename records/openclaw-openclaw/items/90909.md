## Summary

Keep open: current main and the latest stable release still do not populate the embedded Lobster runner's in-process LLM adapter context, and https://github.com/openclaw/openclaw/pull/91028 is the active open implementation path with product, security, and live-proof questions still unresolved.

## What This Changes

The issue requests an in-process LLM adapter bridge for embedded Lobster workflows so `llm.invoke` can run without a gateway URL or bearer token.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/91028 with a maintainer-approved adapter contract that avoids credential env injection, preserves LLM policy, documents the provider name, and includes live embedded invocation proof.

## Reproduction Assessment

Yes, as a source-level reproduction: current main does not populate `ctx.llmAdapters`, while `@clawdbot/lobster@2026.5.22` only uses the direct in-process LLM path when that context field is present. I did not run a live model workflow.

## Solution Assessment

Unclear until maintainers settle the permanent bridge contract. The in-process adapter direction is safer than putting Gateway credentials in env, but provider naming and LLM auth/model policy need explicit review.
