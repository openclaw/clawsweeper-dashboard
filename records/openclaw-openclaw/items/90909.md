## Summary

Keep open: current main and the latest release still leave embedded Lobster without a populated in-process LLM adapter bridge, while the linked implementation PR is open and still needs maintainer product/security decisions before it can resolve the issue.

## What This Changes

The issue requests a host-provided in-process LLM adapter for embedded Lobster workflows so `llm.invoke` can run without requiring Gateway URL or bearer-token credentials.

## Best Possible Solution

Keep this issue open as the canonical product/security decision until the embedded Lobster bridge lands through https://github.com/openclaw/openclaw/pull/91028 or a replacement that preserves llm-task policy, avoids credential env exposure, and documents the provider contract.

## Reproduction Assessment

Yes. Source inspection shows current main does not populate `ctx.llmAdapters` or `ctx.registry`, while the Lobster dependency needs those fields for direct in-process `llm.invoke` and otherwise uses HTTP/env provider paths.

## Solution Assessment

Unclear pending maintainer direction. The in-process adapter direction is plausible and safer than credential env injection, but the provider name, registry exposure, llm-task policy reuse, and live proof need maintainer resolution before it is the best fix.
