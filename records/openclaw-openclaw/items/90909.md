## Summary

Keep open: current main and v2026.6.8 still leave the embedded Lobster runner without a populated in-process LLM adapter, and the active implementation path is the open, unmerged https://github.com/openclaw/openclaw/pull/91028 with product/security/proof questions still unresolved.

## What This Changes

Request to populate the embedded Lobster runner with a host-provided in-process LLM adapter so workflows can use `llm.invoke` without a Gateway URL or bearer token.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/91028 with a maintainer-approved adapter contract that avoids credential env injection, preserves `llm-task` policy, documents the provider name, and includes live embedded invocation proof.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main does not populate `ctx.llmAdapters`, while `@clawdbot/lobster@2026.5.22` only uses the direct in-process LLM path when that context field is present; I did not run a live model workflow.

## Solution Assessment

Unclear until maintainers choose the permanent contract. The in-process adapter direction is safer than putting Gateway credentials in env, but provider naming, registry exposure, and `llm-task` policy preservation need review.
