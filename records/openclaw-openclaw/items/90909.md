## Summary

Keep open: current main and the latest release still leave the embedded Lobster runner without a populated in-process LLM adapter, and the open implementation PR is not merged and still needs maintainer product/security review and real behavior proof.

## What This Changes

Request to populate embedded Lobster with host-provided in-process LLM adapters so workflows can use `llm.invoke` without a Gateway URL or bearer token.

## Best Possible Solution

Review and either land or replace the open bridge PR with a maintainer-approved adapter contract that avoids Gateway credential env injection, preserves `llm-task` auth/model policy, documents the provider name, and includes live embedded invocation proof.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main does not populate `ctx.llmAdapters`, while `@clawdbot/lobster@2026.5.22` only uses direct in-process LLM adapters when that context field is present; I did not run a live model workflow.

## Solution Assessment

Unclear until maintainers choose the permanent contract. The in-process adapter direction is safer than putting Gateway credentials in env, but provider naming, registry exposure, and `llm-task` policy preservation need review.
