## Summary

Current main has per-agent model metadata/runtime overrides, but the actual selection allowlist is still built from the global catalog; this remains a valid product/API feature requiring maintainer-selected composition semantics.

## What This Changes

Add a per-agent model allowlist that limits which models family or other users can view and select for a specific agent.

## Best Possible Solution

Extend the existing per-agent model configuration with explicit restrictive semantics, calculate one effective allowed set with documented inheritance rules, and enforce it across every listing, selection, session, scheduling, and execution path.

## Reproduction Assessment

Yes, at high confidence from current-main source: per-agent `models` entries are metadata/runtime overrides, while the effective selection allowlist remains global. This is a deterministic capability gap rather than an intermittent bug.

## Solution Assessment

The per-agent configuration boundary is the right direction, but the proposed behavior needs explicit intersection/replacement and fallback semantics; one canonical effective-policy helper is safer than parallel gates in individual selectors.
