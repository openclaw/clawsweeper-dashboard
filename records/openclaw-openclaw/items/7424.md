## Summary

Keep open. Current main still has only a global model visibility allowlist based on `agents.defaults.models`; per-agent `models` exists as metadata/runtime override surface, but it is not an enforced per-agent model allowlist, and an open implementation PR still needs maintainer API/product review before this issue can close.

## What This Changes

The issue requests a per-agent model allowlist so a specific agent cannot expose or switch to costly or inappropriate models from the global catalog.

## Best Possible Solution

Define a documented effective per-agent model visibility policy that preserves existing runtime metadata semantics, then enforce it consistently in `models.list`, `/model` selection, session patching, UI/TUI selectors, runtime paths, docs, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but the feature gap is source-reproducible: current main parses visibility from `agents.defaults.models`, `models.list` has no `agentId`, and session model validation is not scoped to an effective per-agent allowlist.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the public API semantics; the maintainable direction is an explicit per-agent visibility policy rather than accidentally changing the existing runtime metadata meaning of `agents.list[].models`.
