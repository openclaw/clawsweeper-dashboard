## Summary

Keep open. Current main still uses a global `agents.defaults.models` visibility policy plus per-agent model metadata/defaults, while the requested per-agent model allowlist remains unresolved and is being explored in an open implementation PR that needs maintainer API/product review.

## What This Changes

The issue requests a per-agent model allowlist so specific agents cannot expose or select costly or inappropriate models.

## Best Possible Solution

Review and reshape the open implementation PR or a successor into one documented per-agent model visibility policy that preserves existing runtime metadata behavior and enforces the effective allowlist across listing, selection, session patching, UI/TUI, docs, and tests.

## Reproduction Assessment

Yes for the source-level feature gap: current main reads configured visibility from global `agents.defaults.models`, `models.list` has no `agentId`, and session patch validation has no agent-scoped policy input. This is a feature gap rather than a runtime bug reproduction.

## Solution Assessment

Unclear until maintainers settle the public API semantics. A shared effective policy resolver threaded through Gateway, model selection, session patching, and UI/TUI callers is the maintainable direction, but config compatibility needs an explicit decision.
