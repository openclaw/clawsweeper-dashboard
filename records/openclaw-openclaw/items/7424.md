## Summary

Current main still lacks an agent-scoped model visibility policy: the per-agent `models` field carries runtime/model metadata, while selection and listing remain governed by the global catalog. The issue remains the canonical focused request because the linked implementation PR closed unmerged and the broader account-policy issue also closed without resolving it.

## What This Changes

Requests a per-agent model allowlist that prevents individual agents from exposing or selecting models outside their assigned catalog.

## Best Possible Solution

Define one typed agent-scoped model visibility policy with explicit inheritance and fallback rules, then make model listing, aliases, selection validation, session patching, UI/TUI selectors, cron/subagent paths, and runtime execution consume that same policy.

## Reproduction Assessment

Yes, at source level. On current main and v2026.6.6, the allowlist builder reads the global catalog and the agent identifier only affects fallback resolution, so an agent cannot independently restrict the visible and selectable model set.

## Solution Assessment

No implementation is currently present to accept as the best fix. The best path is a centralized effective-policy boundary with maintainer-approved composition semantics, rather than separate checks added independently to every command, UI, and runtime caller.
