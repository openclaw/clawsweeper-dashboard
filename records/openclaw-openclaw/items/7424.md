## Summary

Current main still lacks per-agent model restriction: the existing per-agent `models` map is documented as metadata/runtime overrides, while visibility and validation remain rooted in the global catalog. The issue remains canonical, and the linked implementation PR is open but conflicting, so maintainers should decide the config/inheritance contract and reconcile that PR rather than close this request or queue duplicate work.

## What This Changes

Add a per-agent model allowlist so each configured agent can expose and select only operator-approved models.

## Best Possible Solution

Define one documented effective per-agent model policy with explicit replace-or-compose semantics against `agents.defaults.models`, then enforce it consistently in model catalogs, `/model`, session patching, UI/TUI selectors, fallbacks, and runtime execution.

## Reproduction Assessment

Yes. Current main provides a high-confidence source reproduction: allowed visibility is derived globally, the Gateway request has no agent scope, and the UI requests the same configured catalog for every agent.

## Solution Assessment

No final implementation direction is approved yet. The linked PR is a plausible starting point, but maintainers must first decide whether the existing metadata map should also become an allowlist, how empty and missing values inherit, and which server-side paths enforce it.
