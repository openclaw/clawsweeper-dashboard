- `impact:auth-provider`: The missing guardrail allows users of an agent to select provider/model routes that its operator intended to exclude.

## Summary

Current main still lacks per-agent model visibility restrictions: agent-level model maps provide metadata overrides, while the effective allowlist remains global. The linked implementation at https://github.com/openclaw/openclaw/pull/78290 closed without merging, so it does not resolve this issue.

## What This Changes

Add a per-agent model allowlist that prevents users of an agent from listing or selecting models outside that agent’s permitted set.

## Best Possible Solution

Add one typed agent-level visibility catalog with explicit inheritance semantics, derive a canonical effective policy per agent, and enforce it across browsing, commands, session patches, UI/TUI selectors, and live runtime model changes.

## Reproduction Assessment

Yes, at source level: configure two agents that should expose different models and current visibility policy still derives both sets from `agents.defaults.models`; `agentId` changes fallback and auth context, not the configured allowlist.

## Solution Assessment

Yes in principle: a centralized per-agent effective visibility policy is the narrowest maintainable solution. The exact public field and composition semantics still require maintainer product judgment.
