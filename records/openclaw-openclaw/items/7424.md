## Summary

The requested cost-control guardrail is still necessary: current main has a per-agent `models` map only for metadata/runtime overrides, while the effective visibility and selection allowlist remains global. The issue is the canonical request and should remain open while the existing implementation PR is reviewed rather than spawning duplicate work.

## What This Changes

Add an administrator-defined per-agent model allowlist that limits which models each agent exposes and can select.

## Best Possible Solution

Settle one documented effective-allowlist contract, then use a shared resolver to enforce it consistently in model catalogs, `/model`, session patching, UI/TUI selection, and live agent execution while preserving unchanged behavior for agents without an override.

## Reproduction Assessment

Yes, at source level. Current main defines the agent `models` map as metadata overrides and retains the effective configured allowlist around the global `agents.defaults.models`, demonstrating the missing restrictive per-agent policy without needing a runtime failure.

## Solution Assessment

Unclear until maintainers approve the contract. The open PR is the right implementation vehicle, but the best fix must unify visibility and server-side enforcement and explicitly define inheritance rather than treating the agent map as a UI-only filter.
