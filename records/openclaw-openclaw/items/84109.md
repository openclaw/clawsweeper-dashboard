- `impact:auth-provider`: The failure is in provider request formatting/routing for Azure/OpenAI Responses models.

## Summary

Keep open: current main and v2026.5.18 still emit system/developer/user Responses input items without the explicit `type: "message"` shape, and the linked open fix PR now owns the narrow repair.

## What This Changes

Azure AI Foundry project endpoints reject OpenClaw Responses payloads because shared OpenAI Responses input items lack explicit `type: "message"` and system/developer prompts are emitted as bare strings.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/84170; the desired end state is a shared Responses converter that emits explicit `message` items with `input_text` content for system/developer/user inputs while keeping Foundry on the existing generic Responses path.

## Reproduction Assessment

Yes for source-level reproduction: current `buildOpenAIResponsesParams()` produces system/developer/user input items without `type: "message"`, while the Microsoft schema describes message input items with that discriminator. I did not run a live Foundry request because no Azure credentials were available.

## Solution Assessment

Yes for the central bug: adding the stricter shared Responses message-item shape is the narrow maintainable fix and is already proposed in the linked open PR. A separate `azure-openai-responses` Foundry transport decision can wait unless maintainers want that API name to support project endpoints too.
