## Summary

Keep open: current main and the latest shipped tag still emit Responses system/developer/user input items without the explicit `type: "message"` discriminator, while the linked fix PR was closed unmerged after useful proof rather than landed. The issue remains a narrow provider payload-shape bug with a clear repair path.

## What This Changes

Azure AI Foundry Responses requests fail because OpenClaw's shared Responses payload builder omits explicit `type: "message"` on system/developer/user input items.

## Best Possible Solution

Land a narrow shared Responses converter fix that emits explicit message items with `input_text` content arrays, with focused tests for OpenAI/Azure/custom Responses and Codex fallback payloads.

## Reproduction Assessment

Yes for source-level reproduction: current main's shared Responses converter emits system/developer/user input objects without `type: "message"`, matching the Azure Foundry 400 report and Microsoft's typed message-item contract. I did not establish a live Azure failure path in this review.

## Solution Assessment

Yes. The narrowest maintainable fix is to update the shared Responses converter to emit the stricter typed message-item shape; the separate `azure-openai-responses` SDK `api-version` behavior can remain a follow-up provider decision if maintainers want that transport to support Foundry project endpoints too.
