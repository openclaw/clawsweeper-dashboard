## Summary

Keep open: current main still builds Azure/OpenAI Responses user and system/developer input items without the explicit `type: "message"` discriminator, and no linked or searched PR currently fixes the exact Foundry payload shape.

## What This Changes

The issue requests fixing Azure AI Foundry Responses payloads so system/developer/user input items include explicit `type: "message"` and Azure Foundry project endpoints stop returning HTTP 400.

## Best Possible Solution

Make the shared Responses converter emit schema-explicit message input items for system/developer/user content, keep Foundry project endpoints on the current generic Responses path unless owners choose a separate Azure SDK transport change, and add focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main constructs the reported Responses input objects without `type: "message"`, and the issue includes a live Foundry 400 response for that shape. I did not run a live Azure request because no Azure credentials were available.

## Solution Assessment

Yes for the central bug: adding explicit `type: "message"` and array `input_text` content in the shared converter is the narrowest maintainable fix and should remain compatible with direct OpenAI Responses. The Azure SDK `api-version` workaround should be treated as a separate transport-support decision if maintainers want that API name to support Foundry project URLs.
