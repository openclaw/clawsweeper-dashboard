## Summary

Keep open: current main and the latest release still emit system/developer/user Responses input items without the explicit `type: "message"` shape, and the open linked fix PR is the right implementation candidate to review rather than closing this issue before merge.

## What This Changes

The issue asks OpenClaw to make OpenAI Responses input items conform to Azure AI Foundry’s stricter message-item shape for Foundry project endpoints.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84170 if maintainers accept the shared Responses wire-shape compatibility risk, then close this issue after merge.

## Reproduction Assessment

Yes for source-level reproduction: current main’s shared Responses converter emits system/developer/user input items without `type: "message"`, while the Foundry contract requires that discriminator. I did not run a live Azure request because this read-only review has no Azure credentials.

## Solution Assessment

Yes: the narrow fix is to make the shared Responses payload conform to the stricter message-item schema, which the open linked PR implements. Any `AzureOpenAI` SDK `api-version` change should be handled as a separate follow-up if maintainers want that transport name to support Foundry project endpoints.
