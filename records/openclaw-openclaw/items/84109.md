## Summary

Keep open: current main and the latest shipped release still build Responses input items for system/developer/user messages without the explicit `type: "message"` discriminator that Azure AI Foundry documents as required for array input items. The existing Microsoft Foundry provider already routes GPT/o-series Foundry deployments through `openai-responses` and strips unsupported `store`, so the central remaining fix is narrow request-shape compatibility rather than a new core feature.

## What This Changes

The issue requests Azure AI Foundry-compatible Responses API request formatting, including explicit `type: "message"` input items and handling Foundry project endpoint differences.

## Best Possible Solution

Add explicit `type: "message"` and array `input_text` content for system/developer/user Responses input messages in the shared converter, keep Foundry project endpoints on the existing generic `openai-responses` path unless owners choose a separate Azure SDK transport change, and cover the payload shape with focused regression tests.

## Reproduction Assessment

Yes for source-level reproduction: current `buildOpenAIResponsesParams()` produces user and developer/system input items without `type: "message"`, and Azure's published schema requires that discriminator for array input items. I did not run a live Foundry request because this review had no Azure credentials.

## Solution Assessment

Yes for the central bug: the narrowest maintainable fix is to make the shared Responses payload conform to the stricter schema, which should be accepted by OpenAI direct and fixes Foundry without a new config flag. The Azure SDK `api-version` behavior should only be changed if maintainers explicitly want `azure-openai-responses` to support Foundry project endpoints rather than routing those through the Microsoft Foundry/generic Responses path.
