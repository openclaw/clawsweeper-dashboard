## Summary

Keep open: current main still builds Responses input items for system/developer/user messages without the explicit `type: "message"` discriminator that Azure AI Foundry requires for array input items. The existing Foundry provider and payload policy already cover the endpoint and `store` boundary, so this is a narrow provider payload-shape bug rather than a stale or duplicate item.

## What This Changes

The issue asks OpenClaw to make Azure AI Foundry project-endpoint Responses requests valid by emitting explicit message input items and avoiding incompatible Azure SDK request shaping.

## Best Possible Solution

Make the shared Responses converter emit schema-valid message input items for system/developer/user content, keep Foundry on the existing generic Responses route unless owners choose to expand the Azure SDK route, and cover the request shape with focused regression tests.

## Reproduction Assessment

Yes, at source level: current `buildOpenAIResponsesParams()` produces system/developer/user input objects without `type: "message"`, while the Foundry Responses schema uses typed `OpenAI.InputItem` entries. I did not run a live Foundry call because no Azure credentials were available in this read-only review.

## Solution Assessment

Yes for the central bug: updating the shared converter to emit explicit message input items is the narrowest maintainable fix and should not require a new config flag. The Azure SDK `api-version` path should be treated as a separate provider-routing decision if maintainers want that API alias to support project endpoints too.
