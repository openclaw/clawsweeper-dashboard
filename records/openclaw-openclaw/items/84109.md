## Summary

Keep open: current main and the latest release still omit the explicit Responses `type: "message"` discriminator for system/developer/user input items, and the linked fix PR is open but unmerged.

## What This Changes

The issue asks OpenClaw to emit Azure AI Foundry-compatible Responses input items by adding explicit `type: "message"` entries and `input_text` content arrays.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84170 if maintainers accept the shared Responses wire-shape change, then close this issue when the fix reaches main.

## Reproduction Assessment

Yes, source-level: current main's shared Responses converter emits system/developer/user input items without `type: "message"`, matching the reported Azure Foundry 400. I did not run a live Azure request in this read-only review.

## Solution Assessment

Yes for the central bug: adding explicit message item types and `input_text` arrays in the shared Responses converter is the narrowest maintainable fix. The Azure SDK `api-version` transport behavior can be handled separately if maintainers want that API name to support Foundry project endpoints.
