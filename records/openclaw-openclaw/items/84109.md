## Summary

Keep open: current main still emits Responses input messages without the explicit `type: "message"` discriminator, and an open linked fix PR already references this issue with closing syntax. The right cleanup is to review and land or close that PR, not close the issue early.

## What This Changes

The issue requests Azure AI Foundry-compatible Responses API input items with explicit `type: "message"` and normalized `input_text` content for system/developer/user messages.

## Best Possible Solution

Land or reject the linked narrow PR after maintainer review; the desired end state is a shared Responses payload shape that is accepted by Azure Foundry without adding a new provider config flag unless maintainers choose to preserve legacy proxy compatibility.

## Reproduction Assessment

Yes for source-level reproduction: current main builds system/developer/user Responses input items without `type: "message"`, while the Azure Foundry contract and the reporter's live error require the discriminator. I did not run a live Foundry request because no Azure credentials were available.

## Solution Assessment

Yes for the central bug: adding explicit message item types and `input_text` arrays in the shared converter is the narrowest maintainable fix. The Azure SDK `api-version` path should remain a separate provider decision unless maintainers want `azure-openai-responses` to support Foundry project URLs directly.
