## Summary

Source inspection confirms the reported request-shape bug remains on current main: user and system/developer Responses input items are built without the explicit `type: "message"` discriminator, while the supported Microsoft Foundry provider already routes GPT/o-series deployments through `openai-responses` and marks `store` unsupported. This should stay open as a narrow provider payload compatibility fix, not a new transport feature.

## What This Changes

The issue asks OpenClaw to make Azure AI Foundry Responses payloads include explicit message input item types and avoid Foundry-incompatible Azure SDK request shaping.

## Best Possible Solution

Update the shared Responses message converter to emit schema-valid message input items for system/developer/user turns, keep Foundry project endpoints on the existing generic `openai-responses` provider path, and add focused regression coverage for the payload shape.

## Reproduction Assessment

Yes for source-level reproduction: current `buildOpenAIResponsesParams()` produces system/developer/user Responses input items without `type: "message"`, matching the reported Azure 400 shape. I did not perform a live Foundry request because this review has no Azure credentials.

## Solution Assessment

Yes for the central bug: the narrowest maintainable fix is to make the shared Responses payload conform to the stricter message-item schema and cover it with regression tests. A separate `azure-foundry-responses` transport is not required unless maintainers choose to expand the Azure SDK route.
