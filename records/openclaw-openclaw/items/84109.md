## Summary

Keep open: current main and the latest shipped release still omit the explicit Responses input message discriminator, and an open linked PR is already attached as the implementation candidate.

## What This Changes

The issue requests Azure AI Foundry-compatible Responses payloads by adding `type: "message"` and `input_text` content arrays to system/developer/user input items.

## Best Possible Solution

Land or revise the linked shared Responses converter fix, then handle any separate Azure SDK transport behavior in a follow-up if maintainers want that API alias to support Foundry project endpoints.

## Reproduction Assessment

Yes for source-level reproduction: current main builds system/developer/user Responses input items without `type: "message"`, matching the Azure Foundry 400 described by the reporter. I did not run a live Azure request because no credentials were available in this review.

## Solution Assessment

Yes for the central bug: adding explicit message item types in the shared converter is the narrowest maintainable fix and the open linked PR implements that direction. The Azure SDK `api-version` path should be treated separately if maintainers decide it must support Foundry project endpoints.
