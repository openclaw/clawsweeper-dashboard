## Summary

Keep open: current main still emits stricter Azure AI Foundry Responses inputs without the explicit message discriminator, and the linked open fix PR is the active implementation path. Because https://github.com/openclaw/openclaw/pull/84170 uses closing syntax for this issue and is not merged, this issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests that OpenAI Responses input items include explicit `type: "message"` and `input_text` content where Azure AI Foundry requires the stricter shape.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84170 if maintainers accept the shared Responses payload-shape change, then close this issue as implemented; treat the Azure SDK `api-version` behavior as a separate follow-up only if owners want Foundry project endpoints under `azure-openai-responses`.

## Reproduction Assessment

Yes for source-level reproduction: current main's shared Responses converter emits system/developer/user input items without `type: "message"`, and Azure/OpenAI Responses examples use typed message items with `input_text` content. I did not run a live Azure request because no Foundry credentials were available. ([learn.microsoft.com](https://learn.microsoft.com/en-us/azure/ai-services/openai/how-to/responses))

## Solution Assessment

Yes for the central bug: adding explicit message item types and `input_text` arrays in the shared converter is the narrowest maintainable fix. The separate `AzureOpenAI` SDK `api-version` path should remain a follow-up provider decision unless maintainers want that API alias to support project endpoints too.
