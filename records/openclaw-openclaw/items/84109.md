## Summary

Keep open: current main and the latest release still emit Responses system/developer/user input items without the explicit `type: "message"` shape, and the linked closing PR at https://github.com/openclaw/openclaw/pull/84170 is still open rather than merged.

## What This Changes

The issue asks OpenClaw to make shared OpenAI Responses input payloads compatible with Azure AI Foundry by emitting explicit message item types and `input_text` content arrays.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84170 if maintainers accept the shared Responses payload-shape change, then close this issue as implemented; keep any separate `AzureOpenAI` SDK `api-version` behavior as a follow-up provider decision if needed.

## Reproduction Assessment

Yes for source-level reproduction: current main’s shared Responses converter emits system/developer/user input items without `type: "message"`, while Microsoft’s Foundry OpenAI v1 schema requires message input items to declare that type. I did not run a live Azure request because this review had no Azure credentials.

## Solution Assessment

Yes for the central bug: the narrowest maintainable solution is the linked shared-converter fix that emits spec-shaped message input items and covers it with focused tests. A dedicated `azure-openai-responses` Foundry transport is a separate provider decision, not required to fix this reported payload shape.
