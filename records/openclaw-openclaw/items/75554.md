## Best Possible Solution

Fix the taskKind create-matching isolation bug, rebase onto current main, add redacted real plugin/session create-progress-finalize proof, then merge only after SDK-contract approval.

## Reproduction Assessment

Not applicable for the requested feature itself. The PR-specific defect is source-reproducible by creating two lifecycle tasks with the same owner/runId/title but different taskKind and observing that create can reuse the first record.

## Solution Assessment

Mostly yes, but not as currently implemented: the additive owner-bound SDK seam is a maintainable direction, while the create de-duplication must treat a provided taskKind as part of the exact identity before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep taskKind from falling back to another task:** `src/tasks/task-registry.ts:717-723`
  - body: When taskKind is provided, this branch first looks for a matching taskKind but then falls through to the older label/title match. A lifecycle create for the same owner and runId with a different taskKind but the same title can reuse and mutate the first task instead of creating an isolated record, so later progress/finalize by taskKind will not represent the intended task. If taskKind is present and no taskKind match exists, create a new record or include taskKind in every fallback match.
  - confidence: 0.86
