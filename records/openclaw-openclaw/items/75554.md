## Summary

Keep open: current main still lacks the proposed plugin lifecycle API and no canonical duplicate owns it, but this PR is not merge-ready because a source-reproducible taskKind identity bug remains, real behavior proof is missing, and the branch is conflicting against current main.

## What This Changes

The PR adds a plugin SDK task-run lifecycle API, exposes taskKind in task DTOs, adds task-registry ID mutators, and updates docs, changelog, API baselines, and runtime task tests.

## Best Possible Solution

Keep the additive owner-bound SDK seam, make taskKind part of create identity, rebase onto current task runtime, and merge only after real plugin/session proof plus SDK-contract approval.

## Reproduction Assessment

Yes for the PR-specific defect by source inspection: create the same owner/runId/title/label twice with different taskKind values and the second create can fall through to the legacy exact match. Reproduction is not applicable to the requested feature on current main because the lifecycle API does not exist there.

## Solution Assessment

Mostly yes, but not as currently implemented: the owner-bound lifecycle runtime is a maintainable direction, while create de-duplication must treat taskKind as identity and the branch needs proof, rebase, and SDK-contract approval.
