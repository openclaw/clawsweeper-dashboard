## Summary

Keep open. The PR is not implemented on current main and has a concrete task-boundary regression: the new plugin runtime directly calls raw task mutators that the existing boundary test only permits inside task internals. It also misses the required changelog entry for a public Plugin SDK feature, and the draft SDK contract still needs maintainer review before landing.

## What This Changes

The PR adds a plugin-facing task run lifecycle API under `runtime.tasks.runs`, maps plugin-owned records onto `cli` task rows keyed by `taskKind` and `runId`, exposes `taskKind` in task DTOs, refreshes SDK docs/API baseline, and adds runtime/task-registry coverage.

## Best Possible Solution

Keep the PR open, have the plugin/task runtime owners approve the public lifecycle contract, then route lifecycle writes through an approved task-internal seam, add the Unreleased changelog entry with contributor credit, and land after the boundary test, focused runtime tests, SDK API check, docs check, and changed gate are green.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is an additive feature PR. The high-confidence review path is static: current main lacks `runs.lifecycle`, and the PR branch’s new `createTaskRecord` call in plugin runtime would be caught by `src/tasks/task-boundaries.test.ts`.

## Solution Assessment

No. The API direction may be reasonable, but the implementation is not merge-ready until it preserves the existing task-boundary contract and records the public SDK feature in the changelog.
