Keep open. The PR adds a real plugin task lifecycle API that current main does not provide, but the exact head is not merge-ready because it violates the existing task boundary guard by calling a raw task lifecycle mutator from plugin runtime code.

The PR adds plugin-facing `api.runtime.tasks.runs.lifecycle` create/progress/finalize helpers, exposes lifecycle SDK types and taskKind DTOs, updates task registry mutation support, and adds docs, tests, changelog, and SDK API baseline hash updates.

## Best Possible Solution

Keep the additive plugin lifecycle API direction, but move the create/progress/finalize mutations behind a task-owned owner-scoped access seam so plugin runtime code does not call raw task mutators directly, then rerun the focused boundary/runtime/SDK/docs checks on the exact head.
