## Summary

Keep open: this is a fresh, externally reported Ollama/Qwen regression with a plausible current-main source path and a second user corroborating related Qwen 3.5 behavior. Current main and v2026.5.12 can still derive `compat.supportsTools: false` from Ollama `/api/show` capabilities and then remove the OpenClaw tool catalog before the native Ollama chat request, but the exact Qwen3.5 GGUF `/api/show` response and `/api/chat` payload are still missing.

## What This Changes

The issue reports that local Ollama Qwen3.5 4B tool calls worked in OpenClaw v2026.5.7 but stopped in v2026.5.12.

## Best Possible Solution

Capture live `/api/show` capabilities and the redacted native `/api/chat` payload for the reported Qwen3.5 models, then make the smallest Ollama-provider compatibility or documentation change that restores real tool-capable models without weakening non-tool-model protection.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection gives a clear conditional path where Ollama metadata without `tools` makes OpenClaw remove the tool catalog, but the actual reported model's `/api/show` capabilities and request payload are still missing.

## Solution Assessment

Unclear. A blind change that ignores Ollama capabilities is not the best fix; the safer path is live evidence followed by a narrow compatibility rule or explicit model-config guidance.
