Keep open: current main still has the same Ollama capability gate that can strip tools when `/api/show` does not report `tools`, and the follow-up comment suggests the regression may still affect larger local Qwen 3.5 models. ## What This Changes

The issue asks OpenClaw to preserve tool calling for a local Ollama Qwen3.5 4B GGUF model that reportedly worked in v2026.5.7 and stopped working in v2026.5.12.

## Best Possible Solution

Capture live `/api/show` capabilities and `/api/chat` payloads for the reported model, then make a narrow Ollama compatibility fix or documented override path that preserves protection for models that truly cannot accept tools.

No high-confidence live reproduction yet. Source inspection gives a clear current-main path where an Ollama model whose `/api/show` capabilities omit `tools` is marked non-tool-capable and receives no OpenClaw tools, but the reported model's actual capabilities and chat payload are still missing.

## Solution Assessment

Unclear. A blind reversal of the capability gate is not the best fix because current tests and docs intentionally protect models that cannot handle tools; the safer solution needs live model evidence and a narrow compatibility decision.
