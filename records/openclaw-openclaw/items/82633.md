## Summary

Close as implemented on current main: the merged Ollama provider fix now makes unknown-capabilities discovered native Ollama models explicitly tool-capable, restoring the tool-catalog path identified in the discussion; the fix is not in v2026.5.18 yet.

## What This Changes

The issue reports a regression where small local Ollama models that worked with tools in 2026.5.7 lost tool calling and produced generic assistant replies in 2026.5.12.

## Best Possible Solution

Keep the merged provider fallback, ship it in the next release, and rely on the documented `compat.supportsTools: false` opt-out for Ollama-compatible servers that truly reject tool schemas.

## Reproduction Assessment

Yes for the fixed source path: the source PR reproduced an Ollama-compatible endpoint whose `/api/show` response lacked capabilities, and current main now resolves that fallback as tool-capable. I did not rerun the exact Qwen3.5 4B Modelfile live in this read-only review.

## Solution Assessment

Yes. `buildOllamaModelDefinition` is the narrow provider-owned fallback point, and explicit capability lists plus `compat.supportsTools: false` still preserve the protection for models or servers that cannot handle tools.
