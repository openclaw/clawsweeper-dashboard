## Summary

Keep open: current main still has the Ollama capability-based gate that can mark an Ollama model non-tool-capable when `/api/show` omits `tools`, and this fresh report plus a second Qwen user report describe a plausible v2026.5.12 regression. No merged fix, open fix PR, or exact canonical duplicate covers this model-specific tool disappearance; live Ollama proof is still needed before changing the policy.

## What This Changes

The issue asks OpenClaw to preserve tool calling for a local Ollama Qwen3.5 4B model that reportedly worked in v2026.5.7 but stopped using tools in v2026.5.12.

## Best Possible Solution

Capture live `/api/show` metadata and `/api/chat` payloads for the reported Qwen3.5 model, then choose a narrow compatibility path that restores confirmed tool-capable models without weakening the protection for truly unsupported Ollama models.

## Reproduction Assessment

No high-confidence end-to-end reproduction yet. Source inspection gives a clear conditional path: if the reported model’s `/api/show` capabilities omit `tools`, current main marks it non-tool-capable and removes tools before the provider request, but the actual live metadata and payload were not captured.

## Solution Assessment

Unclear. The safest next step is live proof followed by a narrow compatibility or documentation decision; blindly ignoring Ollama capabilities would conflict with current tests and the documented stability escape hatch.
