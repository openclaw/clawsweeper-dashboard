## Summary

Keep open: current main still lacks the cron-level opt-in, and the branch is a plausible implementation, but it intentionally changes SSRF private-network posture, conflicts with an open stricter security direction, and lacks real behavior proof.

## What This Changes

The PR adds `cron.webhookAllowPrivateNetwork`, forwards `{ allowPrivateNetwork: true }` into cron webhook POST paths, and updates config docs, schema metadata, and gateway cron tests.

## Best Possible Solution

Maintainers should choose one cron webhook network policy: land a tightly documented opt-in with proof and stale branch cleanup, or supersede it with the stricter reject-before-storage path.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config PR. Source inspection gives high confidence that current main lacks the cron-level key and policy forwarding while the lower-level SSRF guard already has an `allowPrivateNetwork` policy seam.

## Solution Assessment

Unclear. The branch uses the existing `SsrFPolicy` seam narrowly, but choosing an operator private-network opt-in over the stricter open reject-before-storage approach is a maintainer/security policy decision.
