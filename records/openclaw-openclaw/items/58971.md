## Summary

Keep open: current main still lacks this cron webhook private-network opt-in, but the PR adds a new security-sensitive config policy that needs maintainer/security review, real behavior proof, and reconciliation with the opposing reject-at-save direction in https://github.com/openclaw/openclaw/pull/49120.

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/schema/docs flag, forwards `{ allowPrivateNetwork: true }` through cron webhook delivery and failure webhook POST paths, adds cron tests, and includes a couple unrelated extension test typing edits.

## Best Possible Solution

Maintainers should explicitly choose the cron webhook network posture: land a tightly documented operator-only opt-in with final naming, warnings, coverage, and real proof, or supersede it with the stricter reject-at-save approach.

## Reproduction Assessment

Yes, by source inspection: current main sends cron webhook POSTs through `fetchWithSsrFGuard` without a private-network policy and the config schema has no cron opt-in key. I did not run a live cron webhook reproduction in this read-only review.

## Solution Assessment

Unclear. The code uses the existing `SsrFPolicy` seam narrowly, but exposing a cron-level private-network escape hatch is a maintainer/security policy choice and may be superseded by rejecting those URLs before persistence.
