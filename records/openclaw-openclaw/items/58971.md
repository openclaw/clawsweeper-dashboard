## Summary

Keep open: current main still lacks a cron webhook private-network opt-in and this PR wires the existing SSRF policy through the cron webhook paths, but the change is security-sensitive, lacks after-fix real behavior proof for this head, conflicts with the open reject-at-save direction, and still carries an unrelated Zalo user test edit.

## What This Changes

The PR adds `cron.webhookAllowPrivateNetwork` config/docs/schema plumbing, forwards `{ allowPrivateNetwork: true }` into cron webhook delivery and failure webhook paths, adds gateway cron tests, and changes one unrelated Zalo user test cast.

## Best Possible Solution

Maintainers should explicitly choose the cron webhook network posture: land a tightly documented operator-only opt-in with final naming, warnings, coverage, and real proof, or supersede it with the stricter reject-at-save approach in https://github.com/openclaw/openclaw/pull/49120.

## Reproduction Assessment

Yes by source inspection: current main forwards no SSRF policy from cron webhook POSTs, so private or loopback targets remain under the default guard. I did not run a live cron webhook reproduction in this read-only review.

## Solution Assessment

Unclear. The implementation uses the existing `SsrFPolicy` seam narrowly, but exposing a cron-level private-network escape hatch is a maintainer/security policy choice and may be superseded by rejecting those URLs before persistence.
