## Summary

Keep open. The patch is focused and current main does not already implement this opt-in, but it deliberately widens a security-sensitive SSRF boundary and needs maintainer/security policy review, especially because #49120 tracks the opposite reject-before-persistence direction.

## What This Changes

This PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema flag and forwards `{ allowPrivateNetwork: true }` to cron webhook delivery, failure alert, and failure destination POST paths with gateway cron tests.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly: either land a tightly documented operator-only opt-in with final naming and all webhook paths covered, or supersede it with the stricter reject-at-save hardening in #49120.

## Reproduction Assessment

Yes. Configure a cron job with `delivery.mode: "webhook"` and `delivery.to` pointing at `http://127.0.0.1:PORT`; current main forwards no SSRF policy, while the PR's tests assert the allow-private-network policy is forwarded when the new flag is enabled.

## Solution Assessment

Unclear. The implementation uses the existing `SsrFPolicy` seam and is narrow, but deciding whether cron should expose this escape hatch is a security/product call rather than a purely mechanical fix.
