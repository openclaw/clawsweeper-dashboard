## Summary

Keep open. Current main does not implement this cron webhook private-network opt-in, and the PR is a focused but security-sensitive config change that still needs maintainer/security policy review and contributor real behavior proof before merge.

## What This Changes

The PR adds `cron.webhookAllowPrivateNetwork`, documents it, wires it through cron webhook delivery/failure POST paths, and adds gateway cron tests plus a couple unrelated extension test typing edits.

## Best Possible Solution

Maintainers should choose one cron webhook network policy: a documented operator-only opt-in with real loopback/private proof, or the stricter reject-at-save validation path.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection confirms current main has no cron-level private-network opt-in and cron webhook POSTs call `fetchWithSsrFGuard` without a policy.

## Solution Assessment

Unclear. The PR uses the existing `SsrFPolicy` seam narrowly, but choosing a cron-level private-network escape hatch versus the reject-at-save approach in https://github.com/openclaw/openclaw/pull/49120 is a maintainer/security policy decision.
