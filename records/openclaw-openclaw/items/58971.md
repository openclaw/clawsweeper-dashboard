## Summary

Keep open: the PR is a plausible focused implementation, but it deliberately changes cron webhook SSRF posture, lacks contributor real behavior proof, and conflicts with an open stricter security direction in https://github.com/openclaw/openclaw/pull/49120.

## What This Changes

The branch adds `cron.webhookAllowPrivateNetwork` config/schema/docs support and forwards an `allowPrivateNetwork` SSRF policy through cron webhook delivery and failure POST paths, with gateway cron tests and unrelated extension test cleanups.

## Best Possible Solution

Maintainers should choose the cron webhook network posture explicitly, then either land a tightly documented operator opt-in with proof or supersede it with the stricter reject-at-save direction in https://github.com/openclaw/openclaw/pull/49120.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection shows current main has the default SSRF-guarded cron webhook path and no cron-level private-network opt-in.

## Solution Assessment

Unclear. The PR uses the existing `SsrFPolicy` seam narrowly, but exposing a cron-level private-network escape hatch versus rejecting private targets before storage is a maintainer/security policy decision.
