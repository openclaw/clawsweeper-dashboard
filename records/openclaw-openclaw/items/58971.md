## Summary

Keep this PR open for maintainer/security review: the code change is focused and mostly correct, but it deliberately adds a cron webhook private-network opt-in and still lacks contributor real behavior proof.

## What This Changes

The PR adds `cron.webhookAllowPrivateNetwork` to cron config/docs/schema and forwards an optional SSRF policy through cron webhook delivery and failure webhook POST paths, with gateway cron tests and a few unrelated extension test edits.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly: either land a tightly documented operator opt-in with proof and final naming, or supersede it with the stricter reject-at-save direction.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config request. Source inspection confirms current main lacks the requested config key and policy plumbing.

## Solution Assessment

Unclear. The PR uses the existing `SsrFPolicy` seam narrowly, but exposing a cron-level private-network escape hatch versus rejecting private targets before storage is a maintainer/security policy decision.
