## Summary

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema flag and forwards an allow-private-network SSRF policy through cron webhook delivery, failure-alert, failure-destination, and legacy webhook POST paths.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly, then either land a tightly documented and proven opt-in with a cleaned branch or supersede it with strict reject-at-save behavior.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config PR. Source inspection shows current main lacks the cron-level override and sends cron webhook POSTs through the default SSRF guard.

## Solution Assessment

Unclear until maintainers choose the security posture. The patch uses the existing `SsrFPolicy` seam narrowly, but choosing an opt-in bypass over strict pre-persistence rejection is a maintainer/security policy decision.
