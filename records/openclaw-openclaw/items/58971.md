## Summary

Keep open. The PR addresses a real missing cron webhook opt-in on current main, but it deliberately changes SSRF posture, conflicts with the open strict-rejection direction, lacks contributor real behavior proof, and is currently dirty with unrelated extension test edits.

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema flag, forwards `{ allowPrivateNetwork: true }` through cron webhook delivery and failure webhook paths, updates the generated config baseline hash, and includes gateway cron plus unrelated extension test edits.

## Best Possible Solution

Maintainers should choose the cron webhook network policy explicitly, then either land a tightly documented opt-in with proof and a cleaned branch or supersede it with the strict reject-before-storage direction in https://github.com/openclaw/openclaw/pull/49120.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/config PR. Source inspection shows current main lacks the cron-level key and policy forwarding while the lower-level SSRF guard already has an `allowPrivateNetwork` seam.

## Solution Assessment

Unclear. The code uses the existing `SsrFPolicy` seam narrowly, but choosing an operator private-network opt-in over strict pre-persistence rejection is a maintainer/security policy decision.
