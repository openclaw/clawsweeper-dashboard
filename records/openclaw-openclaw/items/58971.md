## Summary

Keep open. Current main has not implemented this cron webhook opt-in, and the PR is coherent, but the remaining blocker is maintainer/security policy judgment because it intentionally relaxes the cron webhook SSRF posture while #49120 proposes stricter reject-before-persistence behavior; the branch is also currently dirty against main.

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema field and forwards an allow-private-network SSRF policy through cron webhook delivery, failure alert, and failure destination POST paths with gateway cron test coverage.

## Best Possible Solution

Maintainers should explicitly decide the cron webhook network policy: either land a tightly documented operator-only opt-in with final naming, warnings, schema/docs/tests, and all webhook paths covered, or supersede this PR with the stricter reject-at-save hardening from #49120.

## Reproduction Assessment

Yes. The behavior can be reproduced by configuring a cron job with `delivery.mode: "webhook"` and `delivery.to` pointing at `http://127.0.0.1:PORT`; current main forwards no SSRF policy, while the PR's tests assert `{ allowPrivateNetwork: true }` is forwarded when the new flag is enabled.

## Solution Assessment

Unclear. The implementation is narrow and uses the existing `SsrFPolicy` contract, but whether exposing this cron-level opt-in is the best solution is a security/product decision because #49120 proposes rejecting the same targets before persistence.
