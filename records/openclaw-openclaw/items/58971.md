## Summary

Keep open: current main still lacks this cron webhook opt-in, but the PR deliberately relaxes SSRF behavior and needs maintainer/security policy review, especially with https://github.com/openclaw/openclaw/pull/49120 proposing the opposite reject-at-save direction.

## What This Changes

The PR adds `cron.webhookAllowPrivateNetwork` config/docs/schema plumbing and forwards an allow-private-network SSRF policy through cron webhook delivery, failure alert, and failure destination paths with gateway tests.

## Best Possible Solution

Maintainers should choose one cron webhook SSRF posture: a tightly documented operator-only opt-in with final naming, warnings, and proof, or the stricter reject-at-save approach in https://github.com/openclaw/openclaw/pull/49120.

## Reproduction Assessment

Yes. Source inspection shows current main forwards no SSRF policy from cron webhook delivery, so a cron webhook targeting `http://127.0.0.1:PORT` will remain under the default guard; the PR's focused tests exercise the intended opt-in forwarding path.

## Solution Assessment

Unclear. The implementation uses the existing `SsrFPolicy` seam narrowly, but deciding whether cron should expose this escape hatch is a maintainer/security policy choice rather than a mechanical fix.
