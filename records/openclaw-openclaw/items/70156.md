## What This Changes

This PR generalizes Swift kit ignore patterns and adds package scripts, additional-boundary wiring, and tests for tracked large-file and unjustified exec/shell-surface guards.

## Best Possible Solution

Keep the contribution open, rebase it onto current main, refresh the allowlists against the current exec surface, replace the import regex with parser-backed or statement-safe detection, make shell:true detection ignore comments and strings, and then validate the targeted script tests plus the additional-boundary/changed lane.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting .gitignore, package.json, scripts/run-additional-boundary-checks.mjs, and git ls-files; the blocking PR bug is reproducible with a semicolonless unrelated import followed by an exec import from node:child_process.

## Solution Assessment

No. The hygiene direction is useful, but the proposed exec guard is not the best mergeable solution until import detection is semicolon-independent and the shell:true scan is code-aware rather than raw-text based.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Detect semicolonless child_process imports:** `scripts/check-no-unjustified-exec.mjs:88`
  - body: The import matcher can start at a previous semicolonless import and consume through the risky child_process import. In that case classifyImportSpecifier inspects only the first brace block, so a new exec/execSync import can enter src/** without tripping this guard.
  - confidence: 0.94
- **[P3] Ignore comments and strings before counting shell: true:** `scripts/check-no-unjustified-exec.mjs:168-172`
  - body: findShellTrueViolations scans raw file text, so a comment or string containing shell: true in a non-allowlisted source file would fail CI even though no shell option is used. Strip comments/strings or use a parser-backed check before counting matches.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR adds a security-oriented exec-surface CI guard, but the current import matcher can miss semicolonless child_process exec imports.

Concerns:

- **[medium] Exec import guard can be bypassed:** `scripts/check-no-unjustified-exec.mjs:88`
  - body: Because CHILD_PROCESS_IMPORT_STMT_RE uses [^;]*? as the statement boundary, a semicolonless import before a child_process import can be consumed as one match and the risky exec/execSync binding can be missed. That weakens the proposed guard against new shell execution surfaces.
  - confidence: 0.94
