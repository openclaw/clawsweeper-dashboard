## Summary

This issue should stay open because current main handles some Vertex/Gemini context-overflow wording, but not the generic LiteLLM Vertex beta 400 body reported here. The failure is plausible, user-impacting, and distinct enough from the related open context/session recovery work to need a focused maintainer decision before repair.

## What This Changes

The issue asks OpenClaw to classify generic Vertex/LiteLLM beta `INVALID_ARGUMENT` failures in long Enterprise sessions, preserve redacted diagnostics, surface actionable recovery, and clear or recover the failed session lane.

## Best Possible Solution

Add a narrow classifier and recovery policy for the observed Vertex/LiteLLM beta response that keeps redacted provider detail, distinguishes context/request-shape failures from other bad requests, and only resets, compacts, or falls back when the signal is specific enough.

## Reproduction Assessment

Unclear for live reproduction: current source shows the reported generic `INVALID_ARGUMENT` body is not covered by the Vertex/Gemini overflow pattern, but I did not reproduce the long-session failure against a real LiteLLM/Vertex beta route.

## Solution Assessment

Unclear until maintainers choose the recovery policy; a narrow provider classifier plus lane cleanup and actionable messaging looks appropriate, while automatic reset or fallback for every generic `INVALID_ARGUMENT` would be too broad.
