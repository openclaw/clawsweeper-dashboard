## Summary

Keep this open: current main still has source-level paths where model special tokens can reach visible assistant output, and the related fix PR was closed unmerged. The later `openclaw agent --json` evidence usefully widens the fix boundary beyond the TUI-only patch.

## What This Changes

Apply model-special-token stripping to primary assistant-visible output paths so GLM/DeepSeek/Hermes `<|...|>` tokens do not reach TUI chat, JSON payloads, or transcripts.

## Best Possible Solution

Centralize shared assistant-visible sanitization at the final display/result/transcript boundaries, preserving error formatting, silent-token behavior, media/directive handling, and code-region preservation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: final/fallback TUI text containing `<|assistant|>` is returned unchanged by `resolveFinalAssistantText`, and internal/no-delivery agent command payload text can be projected to JSON or CLI transcripts without `stripModelSpecialTokens`.

## Solution Assessment

Yes. The best fix is to reuse the existing `sanitizeAssistantVisibleText` pipeline at the shared user-visible assistant boundaries rather than duplicating regexes or only patching the TUI resolver.
