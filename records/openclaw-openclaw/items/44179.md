## Summary

Keep open: current main still has the TUI final-output sanitizer gap, and the newer discussion shows the same root cause can affect `openclaw agent --json` payloads. The shared sanitizer exists and is tested, but the visible output boundaries do not consistently use it, and the related fix PR was closed unmerged.

## What This Changes

The issue requests applying OpenClaw's existing model-special-token sanitizer to the main assistant output paths so GLM-5, DeepSeek, Hermes, and similar `<|...|>` tokens do not leak to users or downstream consumers.

## Best Possible Solution

Route all user-visible assistant finalization surfaces through the shared assistant-visible sanitizer: TUI final/streamed fallback text, embedded agent reply payloads, and any session-history or JSON-result projection that exposes assistant text to downstream consumers. Preserve current error formatting, tool/reasoning block semantics, media directives, and code-region preservation.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass `finalText` or fallback `streamedText` containing `<|assistant|>` through `resolveFinalAssistantText` and current main returns it unchanged; for `openclaw agent --json`, streamed `assistantTexts` containing `<|end_of_text|>` can be selected before the sanitized final fallback.

## Solution Assessment

Yes. The best fix is broader than the closed TUI-only PR: reuse `sanitizeAssistantVisibleText` at the shared visible-output boundaries instead of duplicating regexes or importing the sessions helper, and add regression tests for both TUI and agent JSON/session-visible paths.
