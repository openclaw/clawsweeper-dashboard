## Summary

Keep this open: current main still routes TUI final and fallback streamed assistant text to the visible chat log without the shared model-special-token sanitizer, while the shared sanitizer itself exists and is tested. The related fix PR was closed unmerged, so there is no current-main implementation to close against.

## What This Changes

The issue requests applying GLM-5/DeepSeek model-special-token stripping to the main TUI chat output path for final and fallback streamed assistant text.

## Best Possible Solution

Wire the TUI finalization path through the shared assistant-visible sanitizer before visible chat finalization, preserving current error-message behavior and code-region handling. Add focused regression coverage for final text, streamed fallback finalization, ASCII/full-width model tokens, and inline/fenced code preservation, plus a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: pass `finalText` or fallback `streamedText` containing `<|assistant|>` or `<｜begin▁of▁sentence｜>` through `resolveFinalAssistantText`; current main returns that text unchanged and then forwards it to `ChatLog.finalizeAssistant`.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing shared assistant-visible sanitizer at the TUI visible-output boundary instead of duplicating token regexes or importing the sessions tool helper. The fix should avoid sanitizing formatted error text unless maintainers explicitly want to change error semantics.
