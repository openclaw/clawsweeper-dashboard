## Summary

Keep open. Current main has partial Dreaming hygiene fixes for source-side stopwords, classified cron transcripts, deleted archives, heartbeat chatter, and silent reply sentinels, but the remaining report is still valid: REM only has a narrow legacy-tag blacklist, light-dream narrative themes still pass raw persisted conceptTags, unclassified direct cron prompts can still preserve paired assistant narration, and there is still no user-facing Dreaming session/cron exclusion config.

## What This Changes

The issue asks memory-core Dreaming to stop surfacing stopword/template/role artifacts as themes and to exclude cron/NO_REPLY automation noise from session corpus ingestion.

## Best Possible Solution

Keep this open until memory-core has one coherent Dreaming cleanup path: a shared concept-tag noise sanitizer used by derivation, REM reflections, and light narrative themes; tests for decimal/template/path and legacy persisted tags; and spoof-safe cron/session exclusions based on session metadata or documented operator config coordinated with #72611.

## Reproduction Assessment

Yes. A source-level repro is high-confidence: feed REM or light dreaming a short-term entry with legacy `conceptTags` such as `theme`, `kept`, or `1.00`, or ingest an ordinary unclassified transcript with `User: [cron:...]` followed by assistant narration; current main still lets those remaining paths through.

## Solution Assessment

No, current main is not the complete best solution yet. The narrow maintainable direction is a shared read-path tag sanitizer plus metadata-backed or configurable session exclusions, rather than only expanding stopword arrays or adding cross-message regex drops.
