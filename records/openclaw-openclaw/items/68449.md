## Summary

Keep open: current main has fixed important parts of the original report, including broader concept stopwords, generated cron transcript skipping, and staged-candidate contamination guards, but source inspection and active related items still show unresolved Dreaming hygiene gaps around REM candidate-truth selection, empty REM placeholders, legacy concept tags, and the operator-facing session/cron exclusion policy.

## What This Changes

The issue asks memory-core Dreaming to stop surfacing common stopwords and generated cron/session artifacts as themes, corpus entries, REM truths, or promotion candidates.

## Best Possible Solution

Define one memory-core Dreaming hygiene boundary, land the narrow open fixes for generated REM/promotion artifacts, and keep any public session/cron exclusion knobs on the existing configurable-filter tracker.

## Reproduction Assessment

Yes, at source level for the remaining current-main gap: REM candidate truth selection and placeholder rendering still accept generated short-term entries without a consistent source/type hygiene guard. I did not run a live cron-heavy deployment replay.

## Solution Assessment

No, current main is only a partial solution. The maintainable fix is to coordinate the existing narrow contamination PRs with a maintainer-approved session/cron exclusion contract instead of adding another isolated stopword or snippet filter.
