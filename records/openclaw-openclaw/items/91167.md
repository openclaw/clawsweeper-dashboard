## Label Justifications

- `P1`: Independent Linux and macOS reports show built-in memory search can remain unavailable until the gateway is restarted.
- `impact:session-state`: The live gateway can retain stale or invalid memory-index state that diverges from the canonical on-disk database.

## Summary

The issue remains necessary: current main partially self-heals missing identity when the provider is available, but the verified stale SQLite handle and provider-unavailable cases remain open, and an open PR still explicitly owns the issue.

## What This Changes

Make a running gateway recover memory search after index identity metadata is lost or its SQLite database is atomically replaced beneath an open handle.

## Best Possible Solution

Land one canonical provider-safe missing-identity repair with manager-level coverage, then implement lifecycle-owned database replacement detection at a sync boundary and verify the gateway reopens the canonical SQLite file without hot-path polling.

## Reproduction Assessment

Yes. The report provides concrete missing-meta and atomic-swap steps, an independent macOS reproduction confirms the deleted-inode state, and current source contains the matching identity gate and swap/reopen behavior.

## Solution Assessment

No, not as one undifferentiated change. Missing identity should self-heal only when index data can be preserved, while stale file replacement needs a separate lifecycle-owned reopen mechanism rather than per-search freshness polling.
