## What This Changes

The issue asks memory-core to recover when a running gateway has indexed chunks but missing index identity and when an out-of-process reindex swaps the SQLite file under the live gateway.

## Best Possible Solution

Land or deliberately reject the narrower stale-handle and reindex-read-race PRs, then close this broad report once gateway memory_search after external reindex is proven on current main or a release.

## Reproduction Assessment

Yes at source level for the remaining path: current main still opens and uses a long-lived SQLite handle without DB file identity refresh, while an external atomic reindex can replace the file under that handle. I did not run the live lsof reproduction in this read-only review.

## Solution Assessment

No, current main is not yet the complete fix; it covers part of missing-identity recovery but leaves stale live handles and related safe-reindex read races to narrower open PRs. The maintainable path is to review those focused PRs rather than queue another broad replacement from this issue.
