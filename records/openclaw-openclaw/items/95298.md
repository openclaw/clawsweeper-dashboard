## What This Changes

## Best Possible Solution

Land one canonical provider-boundary fix in `openai-completions` that emits exactly one `thinking_end` before text or tool-call lanes, preserves tag-based reasoning, and includes redacted real DeepSeek/OpenClaw proof.

Yes. Source inspection shows an OpenAI-completions stream can emit `thinking_delta` followed by `text_delta` while `thinking_end` is delayed until final block finishing; I did not run a live DeepSeek reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible subscriber mitigation, but the best fix is at the OpenAI-completions provider boundary so every consumer receives `thinking_end` before text or tool-call events.
