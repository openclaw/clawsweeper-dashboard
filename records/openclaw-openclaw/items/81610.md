## Label Justifications

- `P2`: This is a bounded Signal channel hook-coverage bug with safety and audit impact, but it is not a full runtime outage or emergency.
- `impact:security`: Signal replies can bypass pre-send content gate, DLP, and audit plugins registered on the documented delivery hook.
- `impact:message-loss`: Plugin rewrite or cancellation decisions for outbound Signal replies can be silently dropped, causing unintended or unmodified messages to be delivered.

## Summary

Keep open: current main and v2026.6.8 still send Signal monitor replies directly through `sendMessageSignal`, so the documented `message_sending` plugin hook remains bypassed for this path. The open related PRs are useful context, but they target dispatcher/Telegram or internal-hook behavior and do not currently cover `extensions/signal/src/monitor.ts`.

## What This Changes

Signal monitor replies should invoke the existing `message_sending` plugin hook before `sendMessageSignal` so plugins can rewrite, cancel, or audit outbound Signal replies.

## Best Possible Solution

Wire the existing `message_sending` hook into Signal `deliverReplies` exactly once before text/media sends, with focused coverage for media metadata, cancel, rewrite, and empty-payload behavior, or explicitly cover this path in a broader approved hook PR.

## Reproduction Assessment

Yes. Current source shows inbound Signal replies flow through a prebuilt dispatcher into the monitor-local `deliverReplies`, which sends via `sendMessageSignal` without the documented `message_sending` hook; no live Signal account was run.

## Solution Assessment

Yes. The narrow maintainable fix is to call the existing hook in Signal's reply delivery path rather than add a new config surface or rely on dispatcher-only work that does not cover this prebuilt dispatcher.
