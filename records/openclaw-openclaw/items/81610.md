## Summary

Keep this open: current main still sends Signal monitor replies directly through `sendMessageSignal` without the documented `message_sending` hook, the latest release still has the same path, and the linked closing PR remains open rather than merged.

## What This Changes

Signal inbound reply delivery should invoke the existing `message_sending` plugin hook before sending text or media replies.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/81680 with a focused Signal monitor fix that honors `message_sending` cancel/rewrite semantics and preserves empty-reply suppression.

## Reproduction Assessment

Yes. Source inspection on current main shows inbound Signal replies flow through `deps.deliverReplies`, and that function sends directly via `sendMessageSignal` without the documented outbound hook.

## Solution Assessment

Yes. The narrow maintainable solution is to add the existing `message_sending` gate in Signal reply delivery, matching Telegram/shared outbound behavior; the linked PR should be reviewed and fixed before this issue closes.
