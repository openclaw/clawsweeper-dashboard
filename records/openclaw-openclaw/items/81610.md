## Summary

The issue should stay open because current main still has the Signal reply hook coverage gap: inbound Signal replies are delivered directly through `sendMessageSignal`, while the documented outbound hook contract and comparable delivery paths run `message_sending`.

## What This Changes

Request that Signal monitor reply delivery invoke the `message_sending` hook before sending agent replies to inbound Signal messages.

## Best Possible Solution

Teach Signal monitor reply delivery to honor `message_sending` cancel and rewrite semantics through the existing plugin hook surface, with focused regression coverage for text and media replies.

## Reproduction Assessment

Yes, by source inspection: inbound Signal reply payloads flow into `deliverReplies`, and that function sends directly without invoking the hook runner. I did not run a live Signal monitor reproduction in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to add a hook gate to Signal reply delivery, matching the existing Telegram and shared outbound semantics through plugin-facing SDK exports. It should avoid routing Signal plugin code through core internals or adding new policy/configuration.
