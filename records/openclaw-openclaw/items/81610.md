## Summary

Keep open. Current main and shipped v2026.5.7 still route Signal inbound replies through a direct `sendMessageSignal` path, while the documented outbound hook contract and the shared/Telegram delivery paths run `message_sending` before delivery.

## What This Changes

Signal agent reply delivery should invoke the documented `message_sending` hook before sending text or media reply chunks.

## Best Possible Solution

Route Signal monitor reply delivery through the existing hook-aware message lifecycle, or add an equivalent plugin-SDK hook gate in `deliverReplies`, so text and media replies honor rewrite and cancel semantics with focused Signal tests.

## Reproduction Assessment

Yes, by source inspection rather than a live Signal run. Inbound Signal replies flow from `createSignalEventHandler` into `deliverReplies`, and that function sends text/media through `sendMessageSignal` without any `message_sending` hook call.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing hook-aware channel-message/durable delivery path or the public plugin-runtime hook runner in the Signal reply path, with tests for rewrite and cancellation.
