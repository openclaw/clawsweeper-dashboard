## Summary

The report describes a real current-main hook coverage gap. Signal inbound replies reach `deliverReplies`, which sends via `sendMessageSignal` directly, while the documented outbound hook contract and the shared/Telegram delivery paths run `message_sending` before delivery.

## What This Changes

Signal monitor reply delivery should invoke the `message_sending` hook before sending agent replies.

## Best Possible Solution

Teach Signal monitor reply delivery to honor `message_sending` cancel/rewrite semantics through the existing plugin SDK hook surface, with focused regression coverage for text and media replies.

## Reproduction Assessment

Yes, by source inspection: an inbound Signal reply flows through `createSignalEventHandler` into `deliverReplies`, and that function sends directly with no hook invocation. I did not run a live Signal reproduction in the read-only review environment.

## Solution Assessment

Yes, the narrow maintainable fix is to add the hook gate in the Signal reply delivery path, matching the existing Telegram/shared outbound semantics. It should use plugin-facing SDK exports rather than importing core internals from the Signal plugin.
