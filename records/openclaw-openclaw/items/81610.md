## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed for common Signal DM replies, affecting real safety and delivery-policy plugins.
- `impact:security`: Content gates, DLP filters, and audit plugins using `message_sending` can fail open for Signal agent replies.
- `impact:message-loss`: Plugin rewrite or cancellation decisions can be ignored, causing messages to be delivered when the hook contract says they should be changed or suppressed.

## Summary

Keep open: current main and the latest release still send Signal monitor replies directly through `sendMessageSignal` without the documented `message_sending` hook, and the open dispatcher PR at https://github.com/openclaw/openclaw/pull/92440 does not touch the Signal monitor path.

## What This Changes

Signal monitor inbound replies should invoke the documented `message_sending` plugin hook before sending Signal text or media replies.

## Best Possible Solution

Wire Signal monitor reply delivery into the existing outbound hook contract with focused text/media/rewrite/cancel tests, keeping hook ordering and exactly-once semantics explicit.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: current Signal inbound agent replies delegate to `deliverReplies`, which sends through `sendMessageSignal` directly while the documented hook contract and sibling delivery paths run `message_sending`. I did not run a live Signal account in this read-only review.

## Solution Assessment

Yes, the best path is a narrow Signal-owned hook integration or reuse of an existing outbound hook helper, with regression tests for text, media metadata, rewrite, and cancel behavior. Closing as covered by the dispatcher PR would be too broad because that PR does not touch Signal monitor delivery.
