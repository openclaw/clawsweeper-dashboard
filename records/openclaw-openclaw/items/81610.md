## Summary

Keep this open: current main still has the Signal reply hook bypass, the latest release tag still shows the same direct send path, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks for Signal monitor reply delivery to invoke the message_sending hook before sending agent replies to inbound Signal DMs.

## Best Possible Solution

Land the focused Signal monitor fix, likely through https://github.com/openclaw/openclaw/pull/81680 after review, so Signal replies honor the same message_sending cancel/rewrite semantics as other outbound paths.

## Reproduction Assessment

Yes, by source inspection: current main routes inbound Signal reply payloads through deps.deliverReplies, and deliverReplies sends via sendMessageSignal without runMessageSending. I did not run a live Signal gateway reproduction in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to add the existing message_sending gate in the Signal reply delivery path, matching Telegram and shared outbound behavior without adding a new config option or plugin API.
