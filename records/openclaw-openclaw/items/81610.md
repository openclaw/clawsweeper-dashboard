## Summary

Keep this open: current `main` still has the Signal reply hook bypass, the latest release tag still shows the same direct send path, and the linked closing PR is open rather than merged.

## What This Changes

Teach Signal monitor `deliverReplies` to invoke `message_sending` hooks before sending reply chunks.

## Best Possible Solution

Land a focused Signal monitor fix, likely through https://github.com/openclaw/openclaw/pull/81680 after Signal-path proof and regression coverage, so replies honor `message_sending` cancel/rewrite semantics before `sendMessageSignal`.

## Reproduction Assessment

Yes, by source inspection: current `main` routes inbound Signal reply payloads through `deps.deliverReplies`, and `deliverReplies` sends via `sendMessageSignal` without `getGlobalHookRunner` or `runMessageSending`. I did not run a live Signal gateway reproduction in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to add the existing plugin SDK `message_sending` gate in the Signal reply delivery path, matching Telegram/shared outbound behavior. The issue should remain open until the linked closing PR merges or another equivalent fix lands.
