## Label Justifications

- `P2`: This is a bounded Signal channel hook-coverage bug with security and audit impact, but it is not a full runtime outage.
- `impact:security`: Signal replies can bypass documented pre-send content gate, DLP, and audit plugin hooks.

## Summary

Keep open: current main and the latest release still send Signal monitor replies directly through `sendMessageSignal`, so the documented `message_sending` plugin hook remains bypassed on this path. The related open dispatcher/Telegram/internal-hook PRs overlap with the hook family but do not currently cover `extensions/signal/src/monitor.ts`.

## What This Changes

The issue reports that Signal monitor reply delivery should invoke the documented `message_sending` plugin hook before sending text or media replies.

## Best Possible Solution

Wire the existing `message_sending` hook into Signal monitor reply delivery exactly once, with focused coverage for text, media metadata, rewrite, cancellation, and empty-payload behavior.

## Reproduction Assessment

Yes. Current source shows inbound Signal replies flow through the monitor-local `deliverReplies`, which sends text/media through `sendMessageSignal` without invoking `message_sending`; no live Signal account was exercised in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to call the existing pre-send hook in the Signal reply delivery path, or have a broader maintainer-approved hook PR explicitly cover this same Signal path without double-running hooks.
