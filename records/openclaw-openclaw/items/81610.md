## Label Justifications

- `P1`: The report concerns a documented outbound interception hook being silently bypassed on real channel agent replies.
- `impact:security`: Safety, DLP, audit, and content-gate plugins using `message_sending` can be blind to Signal replies.
- `impact:message-loss`: Plugin rewrite or cancellation decisions can be ignored, allowing unintended or unmodified messages to be delivered.

## Summary

The Signal-specific hook bypass is valid on current main, but the same remaining outbound-hook contract work is now tracked by the newer canonical issue at https://github.com/openclaw/openclaw/issues/92374, which already links this report and has active fix PRs.

## What This Changes

Signal monitor agent-reply delivery should invoke the documented `message_sending` hook before sending reply text or media through `sendMessageSignal`.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/92374 as the canonical repair target and make sure its eventual fix or follow-up explicitly covers this Signal `deliverReplies` evidence.

## Reproduction Assessment

Yes from source inspection: current main routes Signal inbound replies through `deliverReplies`, and that function calls `sendMessageSignal` directly without the documented `message_sending` hook. I did not run a live Signal account in this read-only review.

## Solution Assessment

Yes for cleanup: the best current action is to close this older Signal-specific report as superseded by the broader canonical issue. For the code fix, a shared exactly-once delivery hook pipeline is more maintainable than a one-off Signal-only gate.
