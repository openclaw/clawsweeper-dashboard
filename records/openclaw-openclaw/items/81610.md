## Summary

Keep this open: current main still sends Signal inbound replies through `sendMessageSignal` without the documented `message_sending` hook, and the linked closing PR is open rather than merged.

## What This Changes

The issue reports that Signal inbound reply delivery sends replies directly through `sendMessageSignal`, bypassing the documented `message_sending` plugin hook.

## Best Possible Solution

Land a focused Signal monitor fix, likely through https://github.com/openclaw/openclaw/pull/81680 or an equivalent patch, so reply delivery honors `message_sending` cancel, rewrite, and media metadata semantics before calling `sendMessageSignal`.

## Reproduction Assessment

Yes, by source inspection: inbound Signal reply payloads flow through `deps.deliverReplies`, and current `deliverReplies` sends text/media directly with `sendMessageSignal` without invoking `message_sending`. I did not run a live Signal gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to add the existing `message_sending` hook gate inside the Signal reply delivery path, matching Telegram/shared outbound behavior. The issue should stay open until the linked PR or another equivalent fix merges.
