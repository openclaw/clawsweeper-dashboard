## Summary

Keep open: current `main` and the latest release still route Signal monitor replies through direct `sendMessageSignal` calls, so the documented `message_sending` plugin hook remains bypassed for this path. The related dispatcher, Telegram, and internal-hook PRs are useful context but do not currently cover the Signal monitor `deliverReplies` implementation.

## What This Changes

The issue reports that Signal monitor agent replies call the Signal sender directly and bypass the documented `message_sending` plugin hook.

## Best Possible Solution

Keep this Signal-specific issue open and fix `deliverReplies` to invoke the existing `message_sending` hook exactly once before `sendMessageSignal`, preserving media, rewrite, cancel, and empty-payload behavior with focused regression tests.

## Reproduction Assessment

Yes. Current source shows inbound Signal replies call the monitor-local `deliverReplies`, and that function sends text/media through `sendMessageSignal` without the documented hook runner path; no live Signal account was exercised.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing `message_sending` hook into Signal's existing reply delivery function, matching the Telegram/core outbound contract instead of adding a new config or parallel hook surface.
