Review, land, request changes on, or replace https://github.com/openclaw/openclaw/pull/92872 for the QQBot scoped-media bridge while keeping the overlapping WebChat attachment path linked to https://github.com/openclaw/openclaw/issues/67915.

Yes at source level: render or send a sandbox-created local media path while scoped media roots/read callbacks are not propagated into QQBot, or while WebChat denies the local path before server metadata can authorize it. ## Solution Assessment

Yes, the best fix direction is to preserve the existing scoped `mediaAccess`/`localRoots`/`readFile` contract through QQBot-owned send paths and coordinate the WebChat UI symptom with the existing Control UI issue, not add a broad sandbox mount or config bypass.
