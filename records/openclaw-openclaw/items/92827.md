## Summary

Keep open: current main still lacks the QQBot scoped media-access propagation needed for sandbox-created files, and the linked closing PR is open rather than merged. The WebChat symptom overlaps a separate open Control UI issue, but this report remains the canonical combined sandbox/WebChat/QQBot user report.

## What This Changes

Sandbox-enabled agents reportedly cannot send generated workspace files through WebChat or QQBot, resulting in unavailable WebChat attachments and non-delivered QQBot media paths.

## Best Possible Solution

Keep this issue open until the QQBot scoped-media fix lands or is replaced, and let the WebChat-only unavailable-root behavior remain tied to the existing Control UI issue and PR unless a final fix intentionally covers both paths.

## Reproduction Assessment

Yes at source level: current main has QQBot media paths that pass raw media URLs without the existing scoped mediaAccess/readFile/workspaceDir contract, and the WebChat path can still hard-deny with `Outside allowed folders`. I did not run a live Docker/WebChat/QQBot reproduction in this read-only review.

## Solution Assessment

Yes: the best fix shape is to preserve the existing scoped media access contract through QQBot private outbound paths and keep WebChat availability delegated to the authenticated server path when appropriate. Broadening allowed folders or mounting host media directories into the sandbox would be a worse security boundary.
