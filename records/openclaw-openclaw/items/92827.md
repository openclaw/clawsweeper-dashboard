## What This Changes

The issue reports that sandbox-enabled agents with `workspaceAccess: "rw"` cannot deliver generated files to WebChat or QQBot.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/92872 for the QQBot scoped-media bridge, and keep the overlapping WebChat attachment behavior coordinated with https://github.com/openclaw/openclaw/issues/67915.

## Reproduction Assessment

No live sandbox run was executed in this read-only review, but the source-level reproduction path is high-confidence: render or send a sandbox-created local media path while scoped media roots/read callbacks are not propagated, and current main can return `Outside allowed folders` or reject the QQBot local path.

## Solution Assessment

Yes, the best fix direction is to preserve the existing scoped `mediaAccess`/`localRoots`/`readFile` contract through QQBot-owned send paths and coordinate WebChat with the existing Control UI attachment issue, not add a broad sandbox mount or file-access bypass.
