## What This Changes

The issue reports that sandbox-enabled agents cannot deliver generated files to WebChat or QQBot: WebChat marks files unavailable and QQBot leaves raw `<qqmedia>` paths visible instead of sending the file.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/92872 as the canonical fix, then close this issue only after merged code proves sandboxed WebChat/QQBot media delivery.

## Reproduction Assessment

Yes at source level: current core normalizes sandbox media and exposes scoped media roots, but current QQBot adapter/gateway/media resolution does not carry that context into delivery. I did not run a live Docker/WebChat/QQBot reproduction in this read-only review.

## Solution Assessment

Yes, the best current path is a bounded fix that reuses scoped media roots, workspaceDir, and readFile contracts in QQBot/WebChat delivery paths. Broad sandbox mounts or arbitrary host-path access would be a worse security boundary.
