The issue reports that sandbox-enabled agents cannot deliver generated workspace files through WebChat or QQBot, producing WebChat “Unavailable — Outside allowed folders” cards and QQBot `<qqmedia>` paths that never arrive.

## Best Possible Solution

Review, land, request changes on, or replace https://github.com/openclaw/openclaw/pull/92872 for the QQBot scoped-media bridge, while keeping the WebChat rendering overlap coordinated with https://github.com/openclaw/openclaw/issues/67915 and https://github.com/openclaw/openclaw/pull/78034.

## Reproduction Assessment

Yes, at source level: current main drops or fails to consistently propagate scoped media access into QQBot media resolution, and WebChat can hard-deny local attachments before server metadata authorization. I did not run the live Docker/WebChat/QQBot scenario in this read-only review.

## Solution Assessment

Yes for direction, pending maintainer review: preserving the existing scoped `mediaAccess`/`localRoots`/`readFile` contract through QQBot-owned send paths is safer than broad mounts, arbitrary local-file mirroring, or a new config bypass. The WebChat overlap should stay on its focused Control UI track.
