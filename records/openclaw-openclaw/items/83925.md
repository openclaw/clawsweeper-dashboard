The issue asks for `openclaw node status` hint lines to use a consistent output stream across not-loaded, missing-unit, and stopped service states.

## Best Possible Solution

Normalize plain-text `openclaw node status` hint output so follow-up recovery hints use the intended stream consistently, with regression coverage for missing-unit and stopped service states.

Fix the `openclaw node status` plain-text output stream inconsistency reported at https://github.com/openclaw/openclaw/issues/83925. In `src/cli/node-cli/daemon.ts`, keep the terse missing-unit/stopped status headers on their intended stream, but route the follow-up recovery hint lines consistently with the not-loaded hint behavior unless code review identifies a stronger existing convention. Add a focused `runNodeDaemonStatus` regression test that stubs the service as loaded with `runtime.missingUnit` and as loaded with `runtime.status === "stopped"`, captures stdout/stderr, and proves hint lines are visible on stdout while JSON output remains unchanged. Keep the fix scoped to the node status command; do not refactor gateway daemon status output or service lifecycle behavior. Add a `CHANGELOG.md` fixes entry if the repair lane is responsible for release notes.
