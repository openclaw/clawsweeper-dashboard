## Summary

Close: current main has no DingTalk channel implementation to change; DingTalk is documented as a community plugin, and the external plugin source shows the reported text-only dispatcher alongside its own media sender while OpenClaw’s SDK already preserves mediaUrl(s).

## What This Changes

The issue reports that DingTalk generated-image replies lose media metadata in the session reply dispatcher and only deliver text.

## Best Possible Solution

Keep OpenClaw core unchanged and fix/publish the DingTalk dispatcher in the external @largezhou/ddingtalk plugin using the existing mediaUrl/mediaUrls reply payload contract.

## Reproduction Assessment

Yes, at source level: the external DingTalk plugin dispatcher accepts only text and returns without delivery for media-only payloads, while OpenClaw’s SDK tests show mediaUrls are preserved to the channel-owned deliver callback. This is not reproducible as an OpenClaw current-main source bug because the DingTalk implementation is not in this repository.

## Solution Assessment

Yes for this repository: closing here and routing the fix to the community plugin is the narrowest maintainable path. A core change would only be appropriate if the plugin maintainer identifies a missing SDK API, and current source shows the media API already exists.
