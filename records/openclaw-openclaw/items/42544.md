## Summary

Keep open: the reported bug is still plausible on current main, but this PR is not safe to merge as written because the healthy-gateway path can still fall through into startup, the probe shells out to PATH-resolved curl, and the probe ignores the resolved gateway bind host.

## What This Changes

This PR changes `src/cli/gateway-cli/run.ts` to run a curl-based `GET /health` probe on `127.0.0.1` before service-marker stale PID cleanup and skip that cleanup when the probe reports HTTP 200.

## Best Possible Solution

Add the service-marker safety gate on current main using the existing in-process HTTP probe style after resolving the effective gateway health host. A healthy existing gateway should log and return successfully; an unhealthy or unreachable gateway should preserve the stale cleanup and startup path, with focused regression coverage for service-marker mode.

## Reproduction Assessment

Yes. Set `OPENCLAW_SERVICE_MARKER`, simulate or run a healthy gateway already listening on the resolved port, and current main reaches `cleanStaleGatewayProcessesSync(port)` before any liveness probe can protect that process.

## Solution Assessment

No. The PR targets the right bug, but the implementation is not the best solution because it shells out to curl, probes only loopback, and does not return after detecting a healthy existing gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: The healthy branch only logs that cleanup was skipped and then continues into normal gateway startup. Since the existing healthy gateway still owns the port, this can still fail with a gateway lock conflict instead of making service-mode startup a clean success path.
  - confidence: 0.9
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:236`
  - body: The new check always probes `127.0.0.1`, but gateway startup can bind to custom, tailnet, LAN, or container-auto hosts. A healthy gateway that is not reachable on loopback can still be classified as stale and killed.
  - confidence: 0.88
- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The safety gate now depends on a PATH-resolved `curl` executable. If curl is missing or shadowed, the probe resolves false and falls back to stale cleanup, which can kill the healthy gateway this change is meant to protect.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete supply-chain/code-execution concern by running a PATH-resolved curl subprocess during gateway service startup.

Concerns:

- **[medium] Avoid PATH-resolved curl in service startup:** `src/cli/gateway-cli/run.ts:234`
  - body: The health gate executes `curl` from `PATH` before deciding whether to kill gateway processes. That broadens the startup code-execution surface and makes the safety check fail open to stale cleanup when the executable is unavailable or replaced.
