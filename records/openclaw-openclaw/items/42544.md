## Summary

Keep this PR open because it targets a valid service-mode gateway cleanup bug, but the proposed patch still has merge-blocking correctness and security issues. Current main still runs stale gateway cleanup before any liveness gate, while the PR adds a fragile `curl` probe, hardcodes loopback, and does not return after detecting a healthy gateway; the adjacent #47776 work is related stale-process cleanup but does not supersede this health-gate fix.

## What This Changes

The PR adds a `curl`-based `/health` probe before service-mode stale gateway cleanup in `src/cli/gateway-cli/run.ts` and skips cleanup when that probe reports healthy.

## Best Possible Solution

Add the service-marker safety gate on current main using the existing in-process `/healthz` probe against the resolved gateway health host, return cleanly when the existing service is healthy, and keep stale cleanup only for unreachable listeners.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `OPENCLAW_SERVICE_MARKER` set and a healthy gateway already listening on the resolved port, current main reaches stale PID cleanup before any liveness gate can protect that process.

## Solution Assessment

No. The PR targets the right bug, but the implementation should reuse the existing in-process `/healthz` probe, probe the resolved `healthHost`, and return after a healthy existing gateway is detected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The health gate now depends on a PATH-resolved `curl` executable. If `curl` is missing or shadowed, the probe resolves false and falls back to stale cleanup, which can kill the healthy gateway this change is meant to protect.
  - confidence: 0.91
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:236`
  - body: The new check always probes `127.0.0.1`, but gateway startup can bind to custom, tailnet, LAN, or container-auto hosts. A healthy non-loopback gateway can still be treated as stale and killed.
  - confidence: 0.88
- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: The healthy branch only logs that stale cleanup was skipped and then continues into normal gateway startup. Since the existing healthy gateway still owns the port, this can still fail with a gateway lock conflict instead of making service-mode startup a clean no-op.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete supply-chain/code-execution concern by running a PATH-resolved `curl` subprocess during gateway service startup.

Concerns:

- **[medium] Avoid PATH-resolved curl in service startup:** `src/cli/gateway-cli/run.ts:234`
  - body: The health gate executes `curl` from `PATH` before deciding whether to kill gateway processes. That broadens startup code execution and makes the safety check fail open when the executable is unavailable or replaced.
  - confidence: 0.86
