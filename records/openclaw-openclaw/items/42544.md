## Summary

This PR addresses a real service-mode gateway bug, but it should not merge as written because the healthy-gateway path can still fall through into startup, the probe ignores the resolved bind host, and the safety check depends on a PATH-resolved curl subprocess.

## What This Changes

The PR changes `src/cli/gateway-cli/run.ts` to probe `GET /health` on `127.0.0.1` before service-marker stale-PID cleanup and skip that cleanup when the probe returns HTTP 200.

## Best Possible Solution

Add a service-marker safety gate on current main using an in-process `/healthz` probe against the resolved gateway health host before stale cleanup; healthy existing gateways should exit successfully, while unreachable gateways should keep the stale cleanup and startup path.

## Reproduction Assessment

Yes. With `OPENCLAW_SERVICE_MARKER` set and a healthy gateway already listening on the resolved port, current main reaches `cleanStaleGatewayProcessesSync(port)` before any liveness gate can protect that process.

## Solution Assessment

No. The PR targets the right bug, but the proposed implementation is not the best fix because it shells out to curl, probes only loopback, and does not stop startup after detecting a healthy existing gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: The healthy branch only logs that stale cleanup was skipped and then continues into normal gateway startup. Since the existing healthy gateway still owns the port, this can still fail with a gateway lock conflict instead of making service-mode startup a clean no-op.
  - confidence: 0.9
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:236`
  - body: The new check always probes `127.0.0.1`, but gateway startup can bind to custom, tailnet, LAN, or container-auto hosts. A healthy gateway that is not reachable on loopback can still be treated as stale and killed.
  - confidence: 0.88
- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The safety gate now depends on a PATH-resolved `curl` executable. If curl is missing or shadowed, the probe resolves false and falls back to stale cleanup, which can kill the healthy gateway this change is meant to protect.
  - confidence: 0.91
