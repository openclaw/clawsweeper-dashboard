## Summary

Keep this PR open: it targets a real service-mode gateway cleanup bug, but the patch is not mergeable because the new probe depends on PATH-resolved `curl`, probes the wrong host for non-loopback gateway binds, and continues into gateway startup after detecting a healthy existing gateway.

## What This Changes

The PR adds a `curl`-based `/health` check before service-mode stale gateway cleanup in `src/cli/gateway-cli/run.ts` and skips cleanup when that check reports healthy.

## Best Possible Solution

Move the service-marker cleanup behind a resolved in-process `/healthz` liveness gate using the active `healthHost`, exit with the existing supervisor semantics when the current gateway is healthy, and keep stale PID cleanup only for unreachable listeners.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `OPENCLAW_SERVICE_MARKER` set and a healthy gateway already listening on the target port, current main reaches stale PID cleanup before any liveness gate can protect that process.

## Solution Assessment

No. The PR points at the right bug, but the maintainable fix should reuse the existing in-process `/healthz` probe, use the resolved gateway health host, and stop startup cleanly after detecting a healthy existing gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Remove the curl subprocess probe:** `src/cli/gateway-cli/run.ts:234-238`
  - body: The new health gate depends on a PATH-resolved `curl` executable. If `curl` is missing or shadowed, the probe resolves false and falls back to stale cleanup, which can still kill the healthy gateway this change is meant to protect.
  - confidence: 0.91
- **[P2] Probe the resolved gateway host:** `src/cli/gateway-cli/run.ts:236`
  - body: The check always probes `127.0.0.1`, but gateway startup can bind to custom, tailnet, LAN, or container-auto hosts. A healthy non-loopback gateway can still be treated as stale and killed.
  - confidence: 0.88
- **[P2] Return after detecting a healthy gateway:** `src/cli/gateway-cli/run.ts:271-275`
  - body: The healthy branch only logs that stale cleanup was skipped and then continues into normal gateway startup. Since the existing healthy gateway still owns the port, this can still fail with a gateway lock conflict instead of making service-mode startup a clean no-op.
  - confidence: 0.9

## Security Review

Status: needs_attention

  - body: The health gate executes `curl` from `PATH` before deciding whether to kill gateway processes. That broadens startup code execution and makes the safety check fail open when the executable is unavailable or replaced.
