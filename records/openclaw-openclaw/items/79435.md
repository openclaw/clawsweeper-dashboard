## Summary

Keep open: current `main` still has the reported elevation channel-context gap. The exec-event path carries the real channel in `OriginatingChannel`, but elevated authorization still derives the provider key from `sessionCtx.Provider` or `ctx.Provider`, so a synthetic `exec-event` provider will not match `tools.elevated.allowFrom.telegram`; the related fix PR at https://github.com/openclaw/openclaw/pull/79464 was closed unmerged.

## What This Changes

The issue reports that Telegram-originated elevated exec calls and diagnostics do not honor `tools.elevated.allowFrom.telegram` because the elevation gate resolves the channel as unknown or synthetic instead of `telegram`.

## Best Possible Solution

Repair elevated provider-key derivation so system-event providers use the real originating channel, keep fail-closed allowlist semantics, and add focused coverage for exec-event Telegram turns plus `sandbox explain` channel diagnostics.

## Reproduction Assessment

Yes, by source inspection on current `main`: an exec-event turn has `OriginatingChannel: "telegram"` but the elevation gate still passes `ctx.Provider`/`sessionCtx.Provider` as the provider key. That makes `tools.elevated.allowFrom.telegram` unreachable for the reported path without needing a new product behavior.

## Solution Assessment

No, current `main` is not the best solution because it leaves the documented per-channel elevated allowlist disconnected from exec-event provider resolution. The narrow maintainable fix is the direction proposed by the closed related PR, with a corrected regression test and no broader config change.
