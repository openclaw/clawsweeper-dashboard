## Summary

Current main still has the reported channel-resolution gap: the diagnostic path can turn a Telegram session into `channel: (unknown)`, and the runtime elevated gate still prefers the synthetic `exec-event` provider over the originating channel. The related runtime PR was closed unmerged, and the open sandbox PR only covers part of the diagnostic path.

## What This Changes

Make elevated exec allowlist evaluation use the actual originating channel so `tools.elevated.allowFrom.telegram` can match Telegram exec calls.

## Best Possible Solution

Land one focused fix that uses existing bundled-channel/session normalization for `sandbox explain` and makes exec-event elevation evaluate the real originating channel while keeping elevated allowlists fail-closed.

## Reproduction Assessment

Yes. This is source-reproducible on current main: `agent:main:telegram:direct:<id>` reaches registry-only channel inference in `sandbox explain`, and exec-event runtime contexts still carry `Provider: "exec-event"` while the real channel is in `OriginatingChannel`.

## Solution Assessment

Not applicable to a submitted patch for this issue. The narrow maintainable direction is to preserve the actual channel into elevated permission checks and add regression coverage for both the diagnostic and runtime paths, without changing config shape or broadening defaults.
