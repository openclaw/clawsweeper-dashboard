## Summary

Current main still has source-visible runtime and diagnostic channel-resolution gaps for elevated exec allowlists, and the only related fix PR is closed unmerged with no canonical open replacement found.

## What This Changes

The issue reports that channel-keyed elevated exec allowlists do not match Telegram exec calls because the gate sees an unknown or synthetic channel instead of the originating Telegram channel.

## Best Possible Solution

Fix exec-event elevated checks to use the real originating channel, repair sandbox explain channel inference for built-in/session channel IDs, and add focused regression coverage for both paths.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: heartbeat builds exec-event contexts with OriginatingChannel set to telegram while resolveReplyDirectives passes the synthetic Provider into the elevated allowlist resolver, and sandbox explain can still normalize a Telegram session key to undefined.

## Solution Assessment

No current-main solution exists yet. The narrow maintainable direction is to reuse the existing system-event/originating-channel pattern for elevation and fix the CLI diagnostic inference, with tests that assert the actual resolver provider and sandbox explain output.
