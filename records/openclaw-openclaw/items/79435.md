## Summary

Current main still has the reported elevated channel-resolution gap: exec-event contexts carry the real channel separately, but elevated allowlist evaluation still uses the synthetic provider key, and the related runtime fix PR was closed unmerged.

## What This Changes

Fix elevated exec allowlist evaluation so Telegram-originated exec calls and sandbox diagnostics resolve the real channel instead of `(unknown)`.

## Best Possible Solution

Fix the runtime provider-key derivation to use the real originating channel for system-event exec flows, and reconcile the sandbox-explain diagnostic path with the existing open diagnostic PRs.

Yes, source-reproducible. Current main creates exec-event contexts with `OriginatingChannel` set to the real channel and `Provider` set to `exec-event`, then resolves elevated allowlists from that provider key before checking `tools.elevated.allowFrom.<provider>`.

## Solution Assessment

Yes, the narrow maintainable direction is to reuse the existing system-event/OriginatingChannel pattern without adding config or widening allowlists. Any sandbox diagnostic change should be reconciled with the open raw-session-key PRs rather than handled as unrelated churn.
