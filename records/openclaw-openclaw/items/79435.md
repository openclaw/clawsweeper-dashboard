## Summary

Current main still has the reported source-visible gap: exec-event elevated checks use the synthetic provider instead of the real originating channel, and the diagnostic path still relies on registry-only channel normalization. The closed related PR addressed the runtime direction but was not merged, while the open related PR covers only a complementary sandbox-explain path.

## What This Changes

The issue reports that Telegram-originated elevated exec requests do not match `tools.elevated.allowFrom.telegram` because channel context is evaluated as unknown or synthetic.

## Best Possible Solution

Land a focused fix that derives elevated provider context from the real originating channel for system-event exec followups and makes sandbox explain resolve built-in channel ids without requiring an initialized plugin registry.

## Reproduction Assessment

Yes, source-reproducible on current main: heartbeat exec completions carry `OriginatingChannel: telegram` while `Provider` is `exec-event`, and the elevated gate currently uses only the provider key. I did not run a live Telegram/Windows elevated exec flow.

## Solution Assessment

Yes for the narrow direction, but not as a close: the maintainable fix is to reuse existing origin/channel normalization and keep unknown providers fail-closed, with regression coverage for both runtime exec-event and sandbox explain diagnostics.
