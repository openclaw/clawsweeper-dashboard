## Summary

Current main still has source-visible gaps for this elevated allowlist path, so this should stay open as a narrow access-control bug fix candidate. The prior related runtime fix was closed unmerged, and the remaining open sandbox-explain PR only covers part of the diagnostic session-key problem.

## What This Changes

The issue reports that elevated exec allowlists keyed by Telegram do not match because exec flows resolve the channel as unknown or as a synthetic provider instead of the originating channel.

## Best Possible Solution

Land one reconciled narrow fix that resolves built-in channel inference for sandbox explain and uses the originating channel for exec-event elevated permission checks, with regression coverage for both paths.

## Reproduction Assessment

Yes, by source inspection: sandbox explain can still fail to resolve Telegram from session keys through registry-only normalization, and exec-event contexts carry OriginatingChannel while the elevated resolver provider key is still derived from Provider. I did not run a live Telegram/Windows elevated exec reproduction in this read-only review.

## Solution Assessment

Yes, the maintainable direction is a narrow fix in existing channel/session normalization and elevated provider-key selection, not a new config knob. The related open PR should be reconciled with the closed runtime PR rather than treated as a complete replacement.
