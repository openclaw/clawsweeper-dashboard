## Summary

Keep open. Current main and the latest release still route plain `openclaw memory status` through a transient status manager that skips the existing session startup catch-up scan, then reports `dirty` from in-process flags and index identity only, so the false-clean status path remains source-proven.

## What This Changes

Fix plain `openclaw memory status` so it reports effective dirty state when usage-counted session transcripts exist on disk but have no indexed session source or chunk rows.

## Best Possible Solution

Reuse the existing session file versus `memory_index_sources` comparison in a bounded status-safe path so plain status reports effective dirtiness for missing or stale session transcript rows without performing a full sync.

## Reproduction Assessment

Yes. I did not execute the CLI in this read-only review, but current source gives a high-confidence repro path: create a status-purpose memory manager with `sessions` enabled, an on-disk session JSONL missing from `memory_index_sources`, and observe that status skips catch-up and can leave `dirty` false.

## Solution Assessment

Yes. A bounded status-mode drift check is the narrowest maintainable fix because the repo already has the session file/index-row comparison helper; a full status reindex or new config knob would be heavier than the reported failure needs.
