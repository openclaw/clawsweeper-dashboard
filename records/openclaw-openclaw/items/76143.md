## Summary

Keep open. The PR targets a real gap in current `main`, but it should not merge as-is because the new shared tool events bypass the existing tool-argument/result sanitization path and would expose raw Codex dynamic-tool payloads to event subscribers.

## What This Changes

The PR adds Codex app-server dynamic-tool start/result events on the canonical `tool` stream, converts Codex dynamic-tool responses to a TUI-style result shape, and adds a regression test for that request path.

## Best Possible Solution

Add the Codex dynamic-tool progress events, but route args/results through the same redaction, truncation, and image-stripping semantics as the existing Pi tool stream, then cover success/error/media cases and add the required changelog entry.

## Reproduction Assessment

Yes. The focused reproduction is the Codex app-server `item/tool/call` path: current `main` handles the request without canonical `tool` stream events, and the PR test drives that request handler and expects start/result events.

## Solution Assessment

No. Emitting canonical tool events is the right direction, but the proposed payload contract is not the narrowest maintainable fix until it reuses the existing sanitized tool-event shape and documents the user-facing change.
