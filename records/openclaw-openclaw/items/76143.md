## Summary

Keep open. Current main still lacks canonical `stream: "tool"` events around the Codex app-server dynamic tool request path, so the PR addresses a real gap, but it should not merge as-is because the new event payloads forward raw tool arguments and results onto the global agent-event stream.

## What This Changes

The PR adds Codex app-server dynamic tool start/result events, converts dynamic tool responses into tool-result-shaped payloads, and adds regression coverage for the request path.

## Best Possible Solution

Rebase the branch onto current main and emit Codex dynamic tool events through the same sanitized, bounded, TUI-compatible payload contract used by the existing Pi tool stream, with coverage for redaction and image results.

## Reproduction Assessment

Yes. The PR's added harness test exercises the `item/tool/call` request path, and static inspection of current main shows that path returns the dynamic tool response without emitting `stream: "tool"` events.

## Solution Assessment

No as proposed. Emitting events from the Codex-owned request path is the right boundary, but the implementation should sanitize arguments/results and preserve the established tool-result shape before global emission.

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive event path that emits unsanitized Codex dynamic tool payloads.

Concerns:

- **[medium] Raw tool arguments reach event subscribers:** `extensions/codex/src/app-server/run-attempt.ts:1342`
  - body: The new start event sends `call.arguments` directly through `emitCodexAppServerEvent`, which also emits to the global agent-event bus. Existing tool events use `sanitizeToolArgs` before global emission, so this can expose sensitive tool inputs to subscribers.
