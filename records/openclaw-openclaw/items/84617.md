## What This Changes

The PR adds `controlPlaneWrite: true` to four core gateway device pairing/token RPC descriptors, adds registry coverage, and adds a changelog entry.

## Best Possible Solution

Land the descriptor flags with focused coverage plus redacted terminal or log proof that real gateway calls hit the existing 3-per-60s budget, and make the upgrade tradeoff explicit for bulk device-management workflows.

## Reproduction Assessment

Yes, from source inspection: on current main the affected device-management descriptors lack `controlPlaneWrite`, and the gateway only consumes the write budget when that descriptor flag is present. A live repro would call any affected RPC four times in a minute from the same device/IP and observe that current main still dispatches it.

## Solution Assessment

Yes, the descriptor change is the narrow maintainable fix because the registry and server already provide generic control-plane write enforcement. The remaining gap is proof quality and explicit acceptance of the new throttle for existing bulk workflows, not a different code shape.
