## Summary

Keep open. Current main still covers the adjacent OpenRouter `Key limit exceeded` spending-limit case, but the reporter’s exact `API key budget limit exceeded (monthly limit)` text is not billing-classified before the 401/403 auth fallback; #74120 is open and appears to be the right overlapping implementation path, but it is not merged.

## What This Changes

The issue requests that OpenRouter `403 API key budget limit exceeded (monthly limit)` errors advance to configured fallback models during `/new` instead of repeatedly retrying the exhausted primary.

## Best Possible Solution

Current main should classify the reported OpenRouter 403 budget-limit payload as billing before the 401/403 auth fallback, with an exact regression assertion for the raw string and fallback-chain behavior. The cleanest path is to review and, if appropriate, extend #74120 so it explicitly covers and closes this report.

## Reproduction Assessment

Yes. The issue provides a concrete provider/model/config and raw error text, and current main can be traced statically through the classifier to show that the exact string is not billing-classified before the 403 auth override.

## Solution Assessment

Yes. A narrow classifier/test change in the existing failover classification layer is the best way to solve this; broad retry-policy changes are unnecessary for this billing-vs-auth classification gap.
