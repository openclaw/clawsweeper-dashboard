## Summary

Keep open: current main still renders chat history with only the 200-message count cap, and this PR addresses a real Control UI freeze path, but the current PR head is not merge-ready because its new raw-content estimator can recursively walk nested payloads without a depth, node, cycle, or remaining-budget bound.

## What This Changes

The PR branch adds a Control UI chat-history character budget, updates the truncation notice, moves the history limit into a shared constant module, adds focused chat item-builder tests, and records the user-facing fix in the changelog.

## Best Possible Solution

Land a bounded UI-local history windowing fix that counts payloads the Control UI can render, excludes hidden tool messages consistently from budget and notice counts, short-circuits estimation once the budget is exhausted, and leaves the gateway `chat.history` contract unchanged.

## Reproduction Assessment

Yes. The linked #11890 report gives a credible long-session, large-tool-output freeze path, and current main shows the count-only render window; the PR adds deterministic unit scenarios for count, character, tool-result, and hidden-tool-message cases.

## Solution Assessment

No. The proposed layer is the right narrow place for immediate relief, but the estimator needs bounded traversal with a remaining-budget short circuit before this is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Bound the raw content estimator:** `ui/src/ui/chat/build-chat-items.ts:159-170`
  - body: `estimateRawContentChars` recursively walks arrays and nested `content` fields without any depth, node, cycle, or remaining-budget guard. A deeply nested transcript payload can still monopolize the UI thread or overflow the stack during history trimming before the render budget has a chance to help, so bound the walk and stop once the character budget is exhausted.
  - confidence: 0.78
