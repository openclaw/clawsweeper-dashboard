## Summary

Keep this PR open. Current main still uses only the 200-message history cap, and the PR is aimed at a valid Control UI freeze path, but the current patch adds an unbounded recursive content estimator that should be fixed before merge.

## What This Changes

This PR adds a Control UI chat history character budget beside the existing message-count cap, updates the truncation notice, adds focused buildChatItems coverage, and records the user-facing fix in the changelog.

## Best Possible Solution

Land the UI-local history windowing fix after the estimator is made budget-aware and bounded, while preserving the gateway chat.history contract, server truncation behavior, and the existing tests for tool-result content and hidden tool-message notice counts.

## Reproduction Assessment

Yes. The linked bug describes long Control UI sessions with large tool-result-heavy histories, current main confirms only a count cap, and the PR adds deterministic unit coverage for count caps, character caps, tool_result content, and hidden tool messages.

## Solution Assessment

Not yet. The PR targets the right layer for an immediate Control UI mitigation, but the raw-content traversal needs depth/node/remaining-budget bounds before it is the safest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

- **[P2] Bound the raw content estimator:** `ui/src/ui/chat/build-chat-items.ts:159-170`
  - body: `estimateRawContentChars` recursively walks arrays and nested `content` fields without a depth, node, cycle, or remaining-budget guard.
