## Summary

Keep open. The feature is not already on current main, and the PR is close, but it leaves ChatController's applied main-session marker stale when an agent chip is selected, which can desynchronize the active agent UI from the chat.send session after a later gateway/default sync.

## What This Changes

The PR exposes gateway agent summaries to Android chat state and renders selectable agent chips above the existing session selector.

## Best Possible Solution

Land the Android-only selector only after selected-agent changes update both NodeRuntime mainSessionKey and ChatController's applied main-session state, with focused coverage for select-then-gateway-sync behavior.

## Reproduction Assessment

Yes for the review finding: the source trace shows selectChatAgent() updates NodeRuntime state and calls chat.load(), while later syncMainSessionKey() uses ChatController.applyMainSessionKey() with a marker that the new path never updates. The original request is a feature, and its happy path is covered by the contributor's physical-device proof.

## Solution Assessment

No. The patch should reuse or update the existing main-session reconciliation path instead of only loading the resolved session key, so future gateway/default syncs keep the selected agent UI and chat.send target consistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Keep chat main-session state in sync:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:1350`
  - body: Selecting an agent updates NodeRuntime's mainSessionKey and then calls chat.load(), but it never updates ChatController's appliedMainSessionKey. On a later gateway/default sync, chat.applyMainSessionKey() treats the selected agent session as a custom thread, so the UI can revert to the default agent while chat.send still uses the previously selected session. Update the controller's main-session marker when selecting an agent and cover the select-then-sync case.
  - confidence: 0.86
