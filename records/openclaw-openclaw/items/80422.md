## Summary

Keep open: this is a real Android feature PR that is not on current main, and the branch still needs a small correctness fix before it is ready for maintainer/product review.

## What This Changes

The PR exposes gateway agent summaries to Android chat state and adds an agent selector row in the chat sheet that switches the active chat target through NodeRuntime.

## Best Possible Solution

Use one Android main-session switching path that updates NodeRuntime, TalkMode, ChatController alias state, and selector state together, then add focused regression coverage before maintainer UX review.

## Reproduction Assessment

Not applicable to the feature request itself. For the patch defect, source inspection gives a high-confidence path: select a non-default agent, then trigger a later home-canvas/gateway main-key sync, which can leave the UI active key and ChatController's send key divergent.

## Solution Assessment

No, not as written: the UI/data exposure is narrow, but selectChatAgent should reuse the existing syncMainSessionKey/chat.applyMainSessionKey contract so the chat controller alias stays aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep the chat main-session alias in sync:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:1347`
  - body: selectChatAgent updates _mainSessionKey and loads the resolved agent key, but it bypasses ChatController.applyMainSessionKey. If a later gateway/home-canvas refresh syncs the main key back to the gateway/default agent, the active chip can move while ChatController keeps sending on the previously selected session because its applied-main alias was never advanced. Route this selection through the same main-session sync path, or update the chat alias before loading the selected key.
  - confidence: 0.84
