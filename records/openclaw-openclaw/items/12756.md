## Summary

Keep open. Current main still exposes shared `message read`, but Telegram does not advertise, map, gate, dispatch, test, or document a Telegram read action; the existing observed-message cache makes an implementation plausible, but the remaining work needs explicit product and privacy decisions.

## What This Changes

The issue requests Telegram support for the existing `openclaw message read` action so agents can read recent Telegram group messages.

## Best Possible Solution

Add a Telegram-owned observed-message `read` action backed by the bounded message cache, with explicit platform-limit docs, target/topic semantics, retention policy, access gating, and focused tests.

## Reproduction Assessment

Yes, by source inspection. The shared CLI `read` path exists while Telegram discovery, mapping, config, docs, and runtime dispatch still omit `read`; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as originally phrased. The maintainable solution is not a full Telegram Bot API history read or secondary `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after maintainer decisions on privacy and retention.
