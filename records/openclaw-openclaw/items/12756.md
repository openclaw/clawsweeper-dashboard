## Label Justifications

- `P2`: The missing Telegram read action blocks useful automation workflows, but it is a bounded feature/security design item rather than an urgent runtime outage.
- `impact:session-state`: The requested behavior would expose recent Telegram message history to agents and monitoring flows, affecting session/context state.
- `impact:security`: Reading and retaining Telegram chat content raises authorization, privacy, retention, and cross-chat disclosure boundaries.

## Summary

Keep open: current main and v2026.6.6 still do not expose Telegram `message read` through action discovery, action mapping, runtime dispatch, config gates, tests, or docs. Current main now has an observed-message cache that may be the right implementation base, but exposing it as a read action needs maintainer product/security decisions first.

## What This Changes

Add Telegram support for the shared `message read` action so agents and CLI users can read recent Telegram chat or group messages like supported Slack, Discord, and Matrix reads.

## Best Possible Solution

Implement Telegram `message read` in the Telegram plugin by reusing or deliberately extending the existing observed-message cache, with approved account/chat/topic authorization, retention limits, action gating, docs, focused tests, and live Telegram proof; avoid secondary `getUpdates` polling.

## Reproduction Assessment

Yes, at source level. Current main has the shared `read` action, but Telegram does not advertise, map, gate, or dispatch it, so Telegram read requests remain unsupported.

## Solution Assessment

Unclear; a cache-backed Telegram-plugin implementation is the likely best direction, but maintainers still need to approve read scope, retention, authorization, and Bot API semantics before coding.
