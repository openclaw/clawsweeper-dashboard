## Summary

Current main still rejects Telegram `message read`, and neither the closed implementation PR nor the duplicate issue changed that behavior. The request fits OpenClaw’s channel roadmap, but it needs an explicit maintainer product/security decision because Telegram has no arbitrary bot chat-history API and any implementation must define bounded retention, authorization, topic scope, and restart semantics.

## What This Changes

Add Telegram support for the shared message-history `read` action so agents can query recent OpenClaw-observed messages for a target chat or topic.

## Best Possible Solution

Implement Telegram `read` in the Telegram plugin as a bounded query over one canonical OpenClaw-observed message store, with account/chat/topic authorization, explicit retention and restart semantics, no secondary `getUpdates` polling, end-to-end action discovery/dispatch, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes. Current main omits `read` from Telegram action discovery and mapping, and the adapter rejects unmapped actions as unsupported; this is a high-confidence source reproduction without running a live configured bot.

## Solution Assessment

The bounded observed-message-store direction is the best feasible approach, but the earlier PR was not complete and maintainers still need to choose the canonical state, retention, authorization, and config contract before implementation.

## Risks / Open Questions

- The user-visible contract could be misleading unless Telegram `read` is explicitly documented as bounded OpenClaw-observed history rather than arbitrary Telegram chat history.
- Retained inbound content needs a single canonical owner and clear restart, eviction, account, chat, and topic semantics; a second ad hoc cache would create state drift.
- Authorization must preserve Telegram group/topic allowlists and context-visibility rules so the action cannot expose messages the requesting agent or account should not read.
- A new action/config surface requires maintainers to decide whether reads are opt-in, default-enabled, or derived from existing history settings.
