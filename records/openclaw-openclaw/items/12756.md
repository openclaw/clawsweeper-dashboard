## Summary

Keep open. Current main and the latest release still expose `read` as a shared message action but do not advertise, map, gate, dispatch, test, or document Telegram `message read`; the only direct implementation PR was closed unmerged, and the remaining work needs maintainer product/security decisions.

## What This Changes

Request support for `openclaw message read --channel telegram` so agents can inspect recent Telegram chat messages similarly to Slack, Discord, and Matrix.

## Best Possible Solution

Add a Telegram-owned observed-message read action with explicit action gating, authorization, retention, target/topic semantics, docs, and tests, without secondary `getUpdates` polling.

## Reproduction Assessment

Yes. Source inspection shows the shared `read` action and CLI docs exist, while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support; I did not run live Telegram because this read-only review has no credentials and should not create runtime artifacts.

## Solution Assessment

Unclear as phrased. The user-facing action is valid, but the maintainable solution should expose bounded OpenClaw-observed messages only after privacy, authorization, retention, and topic semantics are decided.
