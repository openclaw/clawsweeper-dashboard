## Summary

Keep open. Current main still has shared `message read` plumbing, but Telegram does not advertise, map, gate, dispatch, test, or document a Telegram read action; the existing observed-message cache makes the direction clearer, but the user-facing read semantics still need maintainer decisions.

## What This Changes

Request support for `openclaw message read --channel telegram` to read recent Telegram chat or group messages.

## Best Possible Solution

Add a Telegram-owned observed-message `read` action backed by the existing bounded message cache, with explicit platform-limit docs, target/topic semantics, retention policy, access gating, and focused tests.

## Reproduction Assessment

Yes for the current limitation by source inspection: shared/CLI `read` exists, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit it. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as originally phrased. The maintainable solution is not a direct Bot API history read or a second `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after maintainer decisions on privacy and retention.
