## Summary

Keep open. Current main still wires `read` as a shared message action, but Telegram does not advertise, map, gate, dispatch, test, or document Telegram read support; the prior inbound-store PR was closed unmerged, and maintainer discussion already called out retention/privacy design work.

## What This Changes

Add `openclaw message read --channel telegram` support for recent Telegram chat or group messages, comparable to existing Discord and Slack read actions.

## Best Possible Solution

Implement Telegram-owned observed-message reads with bounded runtime storage, explicit retention/privacy limits, no secondary `getUpdates` polling, and aligned adapter, discovery, runtime, config, docs, and tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from current-main source: `openclaw message read` dispatches `read`, shared target plumbing passes `to`, and Telegram action mapping/runtime omit `read`, so the adapter rejects it before delivery.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers approve Telegram read semantics; the user-facing `message read` surface is appropriate, but a bounded observed-message cache is safer than trying to fetch chat history through Telegram polling.
