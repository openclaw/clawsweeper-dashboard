## Summary

Keep open. Current main still has shared message-read plumbing, but the Telegram plugin does not advertise, map, gate, dispatch, test, or document a Telegram `read` action; the closed inbound-store PR did not merge, and the remaining design needs maintainer policy review.

## What This Changes

Request support for `openclaw message read --channel telegram` and the shared message tool `read` action against Telegram chats/groups.

## Best Possible Solution

Implement Telegram-owned observed-message reads using the existing message cache or an equivalent bounded store, then align action discovery, mapping, runtime dispatch, config/schema/help, CLI topic parameters, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI target plumbing exist, while Telegram discovery, adapter mapping, runtime aliases, action gates, tests, and docs omit it, so Telegram read falls into the unsupported-action path.

## Solution Assessment

Unclear as originally phrased. The user-facing action is reasonable, but the maintainable implementation should use bounded observed-message reads from Telegram-owned runtime state rather than direct Bot API history reads or a second `getUpdates` poller.
