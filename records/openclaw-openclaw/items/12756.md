## Summary

Keep open. Current main and the latest release still expose the shared `read` action and CLI command, but Telegram does not advertise, map, gate, dispatch, test, or document a Telegram read path; the remaining work is security/product-sensitive because it would expose observed Telegram message content.

## What This Changes

The issue requests adding Telegram support to the shared `openclaw message read` action so agents can inspect recent Telegram group messages.

## Best Possible Solution

Implement Telegram-owned observed-message reads using the existing message cache or a bounded successor, with explicit action gating, authorization and retention policy, target/topic semantics, docs, and focused tests, without secondary update polling.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The user-facing action is valid, but the maintainable solution should be a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided.
