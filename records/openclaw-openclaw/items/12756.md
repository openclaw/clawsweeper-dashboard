## Summary

Keep open. Current main and v2026.5.22 still expose the shared `read` action and CLI command, but Telegram does not advertise, map, gate, dispatch, test, or document a Telegram read path; the related inbound-store PR was closed unmerged.

## What This Changes

Add Telegram support for the shared `message read` action so agents and the CLI can read recent Telegram chat messages.

## Best Possible Solution

Implement Telegram-owned observed-message reads using the existing message cache or a bounded successor, with explicit action gating, authorization and retention policy, target/topic semantics, docs, and focused tests, without secondary `getUpdates` polling.

## Reproduction Assessment

Yes, source-reproducible. Current main includes the shared `read` action and CLI command while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The user-facing action is valid, but the maintainable solution should expose bounded OpenClaw-observed/cache-backed messages after explicit privacy, authorization, retention, and topic semantics are decided.

- Group, topic, account, and allowlist boundaries need explicit semantics before this becomes a safe shared message action.
