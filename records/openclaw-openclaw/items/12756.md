## Summary

Keep open. Current main still has shared message-read plumbing, but the bundled Telegram plugin does not expose or implement Telegram `read`; the prior inbound-store PR is closed unmerged, and maintainer discussion already identified privacy/retention design as the remaining decision point.

## What This Changes

Request support for `openclaw message read --channel telegram` to return recent Telegram chat messages similar to existing read-capable channels.

## Best Possible Solution

Implement Telegram-owned observed-message reads with bounded retention, no secondary `getUpdates` polling, and aligned discovery, action mapping, runtime dispatch, config gating, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI target plumbing exist, while the Telegram adapter, discovery, runtime aliases, action gates, tests, and docs omit `read`, leading to the unsupported-action path.

## Solution Assessment

Unclear as written. The requested user-facing action is reasonable, but the maintainable solution should be a bounded Telegram-owned observed-message store with explicit retention/privacy semantics, not direct Bot API history reads or another `getUpdates` poller.
