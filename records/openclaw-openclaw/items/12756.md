## Summary

Current main still lacks Telegram message reads, and the closed implementation attempt did not land. The request remains valid, but maintainers must define the privacy, authorization, storage, retention, and completeness contract before implementation because Telegram bots receive updates rather than arbitrary per-chat history.

## What This Changes

Add Telegram support for the shared `message read` action so agents can query recent group messages.

## Best Possible Solution

Implement a Telegram-plugin-owned read capability explicitly defined as bounded access to OpenClaw-observed messages, using the approved SQLite owner, tight default gating, and consistent authorization, target/topic, retention, documentation, and testing contracts.

## Reproduction Assessment

Yes. Current main's Telegram action map and contract omit `read`, so the reported unsupported-action behavior is source-reproducible with high confidence.

## Solution Assessment

No, the earlier in-memory proposal is not the best current solution as written. The implementation must follow current plugin and SQLite ownership rules and first define security, retention, restart, completeness, and target/topic behavior.
