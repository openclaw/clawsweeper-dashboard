## Summary

Keep open: current main and the latest release still do not expose Telegram `message read`; the request is valid, but it needs maintainer product/security direction before implementation because it would define retained-message, authorization, and read-scope semantics.

## What This Changes

The issue requests Telegram support for the shared `openclaw message read` action so agents can inspect recent Telegram group messages.

## Best Possible Solution

Approve and implement a Telegram-plugin-owned `message read` action that returns only bounded, authorized, OpenClaw-observed messages, with explicit gating, docs, tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main has the shared `read` action, but Telegram does not advertise, map, gate, dispatch, test, or document it, so Telegram read requests remain unsupported.

## Solution Assessment

Unclear until maintainers choose the product contract. A cache-backed Telegram-plugin implementation is likely the right direction, but read scope, retention, authorization, and Bot API semantics need approval before coding.
