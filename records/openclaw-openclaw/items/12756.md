## Summary

Current main still lacks Telegram message-history reads, and the closed implementation attempt did not land. The request remains useful and vision-aligned, but it needs maintainer product and security decisions because Telegram has no normal history endpoint and a correct solution must retain sensitive inbound messages locally.

## What This Changes

Add Telegram support for the shared message-history `read` action used by the CLI and message tool.

## Best Possible Solution

Implement a Telegram-owned `read` action with clearly documented received-since-ingestion semantics, backed by a bounded SQLite store populated by the existing inbound pipeline, with explicit account/chat authorization, retention and deletion rules, opt-in configuration, topic-aware targeting, docs, and focused tests.

## Reproduction Assessment

Yes, at source level: current main exposes the shared `read` action while the Telegram action map, discovery, runtime dispatch, config, tests, and documentation omit it. A live bot invocation was not required to establish the unsupported current-main path.

## Solution Assessment

No final solution is approved yet. The earlier inbound-store direction is plausible, but an in-memory cache is not the best current design; maintainers should first approve a bounded SQLite retention and authorization contract with received-since semantics.

## Risks / Open Questions

- Telegram bots do not provide an ordinary per-chat history endpoint, so this capability would need to retain inbound group-message content locally and define authorization, retention, deletion, restart, and operator-control semantics.
- Using another `getUpdates` consumer could conflict with the running bot or consume updates before normal delivery, so reads must use already-ingested messages rather than independent polling.
- The previous in-memory proposal would lose retained history on restart and conflicts with the repository's current SQLite-first policy for runtime state.
