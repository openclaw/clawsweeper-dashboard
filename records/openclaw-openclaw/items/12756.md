## Summary

Keep open. Current main still has the shared read action and CLI command, but Telegram does not advertise, map, dispatch, gate, test, or document Telegram read support; the prior implementation PR is closed unmerged and the duplicate report was closed back to this issue.

## What This Changes

Request to make `openclaw message read --channel telegram` return recent Telegram messages for Telegram chats or groups.

## Best Possible Solution

Implement Telegram-owned observed-message reads backed by bounded cache state, explicit action gating, authorization/retention policy, target/topic semantics, docs, and focused tests, without secondary `getUpdates` polling.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The requested user-facing action is valid, but the maintainable solution is a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided.
