## Summary

Keep open: current main and the latest release still expose `read` as a shared message action, but the bundled Telegram plugin does not advertise, map, gate, dispatch, test, or document Telegram message-history reads. The existing Telegram message cache is useful implementation substrate, but exposing it through `message read` needs maintainer product/security approval for retained-content semantics.

## What This Changes

The issue requests Telegram support for the shared `openclaw message read` action so agents can inspect recent Telegram group messages.

## Best Possible Solution

Keep this open for a Telegram-plugin-owned design that exposes bounded observed-history reads through the message action path with explicit access gating, retention semantics, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main includes the shared `read` action, but Telegram discovery, action mapping, config gates, runtime aliases, contract tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear until maintainers choose the product/security boundary. A one-off secondary `getUpdates` reader is not the best fix; the safer shape is to reuse or extend the Telegram plugin's observed-message cache with explicit read authorization and retention policy.
