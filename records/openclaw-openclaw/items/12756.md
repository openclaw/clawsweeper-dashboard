## Summary

Keep open. Current main still lacks Telegram support for the shared message `read` action, and the remaining work needs maintainer decisions around observed-message scope, cached-content retention, and authorization.

## What This Changes

Request to add Telegram support for `openclaw message read` so agents can read recent Telegram chat or group messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by the existing persisted message cache, with explicit authorization, retention, action gating, target/topic semantics, docs, and focused tests.

## Reproduction Assessment

Yes, by source inspection. The shared `read` action and CLI command exist, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit a Telegram read action on current main.

## Solution Assessment

Unclear as originally phrased. A full Telegram history reader or secondary `getUpdates` poller is not the best fit; the maintainable path is a bounded observed-message read from Telegram-owned runtime state after privacy and retention decisions.
