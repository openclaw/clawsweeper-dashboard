## Summary

Keep open. Current main still has the shared `read` action and CLI plumbing, but the bundled Telegram plugin does not advertise, map, gate, dispatch, test, or document Telegram message-history reads; the prior inbound-store PR was closed unmerged, so no current implementation supersedes this request.

## What This Changes

The issue requests Telegram support for the shared `message read` action so recent Telegram group messages can be read like Discord, Slack, and Matrix.

## Best Possible Solution

Implement Telegram-owned observed-message reads with bounded storage, explicit retention/privacy limits, no secondary `getUpdates` polling, and aligned adapter, runtime, config, docs, and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows shared `read` and CLI target plumbing exist, while the Telegram adapter, discovery, runtime aliases, action gates, and docs omit `read`, leading to the unsupported-action path.

## Solution Assessment

Unclear. The requested surface is reasonable, but the maintainable implementation needs maintainer-approved retention and privacy semantics rather than direct `getUpdates` history reads.
