## Summary

Keep open: current main and the latest release still do not expose Telegram `message read` through action discovery, action mapping, runtime dispatch, config gates, tests, or docs. The existing observed-message cache is useful implementation groundwork, but the remaining work needs maintainer product and security review before a fix PR.

## What This Changes

Add Telegram support for `openclaw message read` so agents and automation can read recent Telegram chat messages similarly to supported channels.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin by reusing or deliberately extending the observed-message cache, with approved authorization, retention, action gating, docs, focused tests, and live Telegram proof; avoid secondary `getUpdates` polling.

## Reproduction Assessment

Yes, at source level. Current main has the shared `read` action, but Telegram does not advertise, map, gate, dispatch, test, or document it, so Telegram read requests remain unsupported.

## Solution Assessment

Unclear until maintainers choose the product contract. A cache-backed Telegram-plugin implementation is likely the right direction, but read scope, retention, authorization, and Bot API semantics need approval before coding.

## Risks / Open Questions

- Exposing Telegram reads would make retained chat content available to agents and CLI users, so account, chat, topic, sender, retention, and operator-control boundaries need explicit approval.

- The user-facing semantics need a product decision: likely only OpenClaw-observed messages can be read, not full Telegram group history.
