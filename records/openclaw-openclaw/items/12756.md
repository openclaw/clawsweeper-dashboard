## Summary

Keep open. Current main still lacks Telegram `message read` support end to end, and the remaining implementation needs maintainer decisions around Telegram Bot API limits, observed-message retention, and read-access gating before this can be closed or queued as an automated fix.

## What This Changes

Add support for `openclaw message read --channel telegram` so Telegram chats can expose recent observed messages similarly to other supported channels.

## Best Possible Solution

Implement Telegram-owned observed-message reads using a bounded runtime cache/store, then align discovery, action mapping, runtime dispatch, config/schema/help, CLI target/topic semantics, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows shared `read` and CLI plumbing exist, while Telegram discovery, mapping, runtime aliases, action gates, tests, and docs omit it, so Telegram read remains unsupported.

## Solution Assessment

Unclear as originally phrased. The user-facing action is reasonable, but the maintainable implementation should use bounded observed-message reads from Telegram-owned runtime state rather than a direct Bot API history read or a second `getUpdates` poller.
