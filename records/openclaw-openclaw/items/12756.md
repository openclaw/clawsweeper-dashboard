## Summary

Keep open: current main and the latest release still do not expose Telegram for the shared `message read` action, even though Telegram now has an observed-message cache that could become part of a future implementation. The remaining work is a product/security decision about exposing retained Telegram chat content, not conservative cleanup.

## What This Changes

The issue requests Telegram support for `openclaw message read` so agents can retrieve recent Telegram group messages similarly to Slack, Discord, and Matrix.

## Best Possible Solution

Implement this in the Telegram plugin by reusing or extending the existing observed-message cache with approved read semantics, explicit action gating, account/topic scoping, docs, focused tests, and redacted live Telegram proof; avoid a second `getUpdates` reader.

## Reproduction Assessment

Yes, at source level. Current main exposes the shared `read` action, but Telegram discovery, mapping, runtime aliases, config gates, tests, and docs all omit Telegram read support.

## Solution Assessment

Unclear as an implementation plan. The requested CLI behavior is reasonable, but the best fix needs maintainer approval for observed-history retention, Bot API limits, authorization boundaries, and whether to expose the existing cache through `message read`.
