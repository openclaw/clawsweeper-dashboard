## Summary

Current main still rejects Telegram message-history reads, and no merged PR or release implements them; the request remains valid but needs maintainer approval for retention, authorization, persistence, and Telegram topic semantics before implementation.

## What This Changes

Add Telegram support for the shared `message read` action so agents can query recently observed messages for a chat or topic.

## Best Possible Solution

Implement one Telegram-plugin-owned store of authorized, OpenClaw-observed inbound messages with bounded retention and precise chat/topic targeting, then wire discovery, mapping, dispatch, config, docs, and end-to-end tests after maintainers approve its privacy and persistence contract.

## Reproduction Assessment

Yes, at source level: current main neither advertises nor maps Telegram `read`, and the adapter throws for unmapped actions. This is a missing capability rather than a regression requiring a runtime failure reproduction.

## Solution Assessment

No approved solution exists yet. The closed inbound-store PR provides useful platform analysis, but its in-memory design and incomplete end-to-end wiring do not settle the repository's current persistence, privacy, authorization, and topic semantics.

## Risks / Open Questions

- Telegram bots do not provide arbitrary per-chat history retrieval, so a second `getUpdates` consumer could interfere with normal update delivery and still would not provide true chat history.
- Retaining inbound group messages creates sensitive-data, authorization, retention, deletion, restart, and topic-isolation decisions that are not defined today.
- An ephemeral store would lose monitoring state on restart, while persistent storage must follow the repository's canonical SQLite ownership and migration rules.
