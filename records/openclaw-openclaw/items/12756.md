## Summary

Keep open: current main and v2026.6.6 still expose `read` as a shared message action but the bundled Telegram plugin does not advertise, map, gate, dispatch, test, or document Telegram message-history reads. The earlier implementation PR was closed unmerged, and the remaining work needs maintainer product/security review before an automated fix lane is safe.

## What This Changes

The issue requests support for `openclaw message read --channel telegram` so agents can inspect recent Telegram chat messages for monitoring and notification workflows.

## Best Possible Solution

Implement this in the Telegram plugin as an approved, bounded read capability with clear observed-history semantics, an explicit action gate, topic/account scoping, privacy-conscious retention, docs, focused tests, and redacted live Telegram proof; avoid secondary `getUpdates` polling.

## Reproduction Assessment

Yes. Source inspection shows `read` is a shared message action, but Telegram discovery, mapping, runtime aliases, config gates, tests, and docs all omit it, so a Telegram read request remains unsupported on current main.

## Solution Assessment

Unclear as an implementation plan. The requested CLI behavior is reasonable, but the best fix needs a maintainer-approved Telegram plugin design for observed-history retention, Bot API constraints, and authorization boundaries before coding.
