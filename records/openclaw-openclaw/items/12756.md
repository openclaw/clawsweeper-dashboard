## Summary

Keep open. Current main still exposes the shared `read` action and CLI command, but the Telegram plugin does not advertise, map, dispatch, gate, test, or document Telegram read support; the existing observed-message cache gives a plausible implementation shape, but exposing cached Telegram content needs maintainer security and product decisions first.

## What This Changes

The issue requests Telegram support for the shared `message read` action so agents can inspect recent Telegram chat messages similarly to Slack and Discord.

## Best Possible Solution

Implement Telegram-owned observed-message reads using the existing message cache or a bounded successor, with explicit action gating, authorization and retention policy, target/topic semantics, docs, and focused tests, without secondary `getUpdates` polling.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The user-facing action is valid, but the maintainable solution should be a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided.

## Risks / Open Questions

- A Telegram read action would expose observed or persisted Telegram message content, so maintainers need explicit authorization, retention, persistence, and group/topic boundary decisions before implementation.
- The implementation should read OpenClaw-observed cache state, not arbitrary Telegram history; a secondary `getUpdates` poller can conflict with receiver semantics and mislead users about history coverage.
- Bot API 10.0 added a narrow personal-chat message read method, but it does not replace a plugin-owned design for configured Telegram groups, topics, and gateway-observed chats.
