## Summary

Keep open. Current main still does not expose Telegram support for the shared message read action, and the remaining work needs maintainer decisions around observed-message scope, cached-content retention, and authorization before implementation.

## What This Changes

The issue requests Telegram support for `openclaw message read` so agents can read recent Telegram group messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by the existing persisted message cache, with explicit authorization, retention, target/topic semantics, docs, and focused tests.

## Reproduction Assessment

Yes, by source inspection. The shared `read` action and CLI path exist, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit a Telegram read action on current main.

## Solution Assessment

Unclear as originally phrased. The best implementation is not a full Telegram history reader or secondary `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after privacy and retention decisions.
