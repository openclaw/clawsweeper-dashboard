## Summary

Keep open: current main and the latest release still do not advertise, map, gate, dispatch, test, or document Telegram support for the shared message `read` action. The closed duplicate and the closed unmerged inbound-store PR are useful context, but no merged/current implementation now solves the requested Telegram read behavior.

## What This Changes

The issue requests Telegram support for the shared message tool's `read` action so agents can retrieve recent Telegram group messages.

## Best Possible Solution

Implement Telegram `message read` inside the bundled Telegram plugin after maintainer approval of retention and access semantics, using the observed-message cache or plugin-state-backed equivalent, avoiding secondary `getUpdates` polling, and wiring discovery, mapping, runtime dispatch, config gating, CLI target/topic behavior, docs, and focused tests together.

## Reproduction Assessment

Yes, at source level: a configured Telegram channel still advertises no `read` action, has no `read` action mapping, and the runtime has no `read` alias before its unsupported-action fallthrough. I did not run a live CLI command because this review is read-only and the source path is decisive.

## Solution Assessment

No, making Telegram behave exactly like Discord/Slack history reads is not the best framing because the current Telegram integration only has observed-message context and active polling/webhook constraints. The safer direction is a bounded Telegram-plugin read over observed cached messages with explicit retention, authorization, and docs.
