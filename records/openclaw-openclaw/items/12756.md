## Summary

Keep open. Current main still has the shared `read` action and CLI command, but Telegram discovery, action mapping, config, runtime dispatch, tests, and docs still omit Telegram `read`; the remaining work also needs maintainer product/security policy for cached message retention and authorization.

## What This Changes

Add Telegram support for the shared `message read` action so agents and CLI automation can inspect recent Telegram messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by the existing persisted message cache, with explicit authorization, retention, action gating, target/topic semantics, docs, and focused tests while avoiding secondary `getUpdates` polling.

## Reproduction Assessment

Yes, by source inspection. The shared `read` action and CLI command exist, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit a Telegram read action on current main.

## Solution Assessment

Unclear as originally phrased. A full Telegram history reader or secondary `getUpdates` poller is not the maintainable path; a bounded observed-message read from Telegram-owned runtime state is the safer direction after privacy and retention decisions.
