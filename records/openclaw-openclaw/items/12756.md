## Summary

Keep open. Current main still has shared `read` plumbing, but the Telegram plugin does not advertise, map, gate, dispatch, test, or document Telegram message reads, and the related inbound-store PR was closed unmerged.

## What This Changes

Request Telegram support for `openclaw message read` so agents can retrieve recent Telegram chat messages for monitoring workflows.

## Best Possible Solution

Implement Telegram-owned observed-message reads with bounded runtime storage, explicit retention/privacy limits, and end-to-end adapter, runtime, config, docs, and test coverage.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: the CLI dispatches `read`, shared target plumbing maps it to `to`, and the Telegram adapter/runtime omit `read`, causing the unsupported-action path.

## Solution Assessment

Unclear until maintainers approve Telegram read semantics. The user-facing `message read` surface is appropriate, but a direct Bot API history fetch is not the maintainable path; an observed-only bounded cache is safer.
