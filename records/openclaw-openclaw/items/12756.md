## Summary

Keep open. Current main and v2026.5.20 still expose the shared `read` action and CLI command, but Telegram does not advertise, gate, map, dispatch, test, or document a Telegram `read` action; the closed duplicate and closed attempted implementation are useful context but do not resolve the remaining product/security decision.

## What This Changes

Request Telegram support for `openclaw message read` and message-tool `read` over recent Telegram chat or group messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded message-cache state, with explicit gating, authorization, retention, target/topic semantics, docs, and tests while avoiding a secondary `getUpdates` history reader.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The maintainable path is a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided; a full history reader or secondary polling path is not the safe default.
