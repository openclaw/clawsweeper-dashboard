## Summary

Keep open. Current main and the latest release still expose the shared `read` action and CLI command, but Telegram does not advertise, gate, map, dispatch, test, or document a Telegram `read` action; the remaining work is also product/security-sensitive because it would expose observed Telegram message content.

## What This Changes

Request Telegram support for the shared `message read` action so agents can inspect recent observed Telegram chat messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded existing message-cache state, with explicit action gating, authorization, retention, target/topic semantics, docs, and focused tests while avoiding secondary `getUpdates` polling.

## Reproduction Assessment

Yes, by source inspection. The shared `read` action and CLI command exist, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit a Telegram read action on current main.

## Solution Assessment

Unclear as originally phrased. The maintainable direction is a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided; a full history reader or secondary `getUpdates` poller is not the safe path.
