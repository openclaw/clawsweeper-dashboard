## Summary

Keep open. Current main still lacks end-to-end Telegram `message read` support: shared and CLI `read` plumbing exists, but the Telegram plugin does not advertise, map, gate, document, or dispatch the action.

## What This Changes

Request to add Telegram support for `openclaw message read` so agents can read recent Telegram chat/group messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded runtime state, with explicit docs for platform limits, retention, and read-access gating.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist, while Telegram discovery, action mapping, runtime aliases, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The user-facing action is reasonable, but the maintainable solution should use bounded observed-message reads from Telegram-owned runtime state rather than a direct Bot API history read or a second `getUpdates` poller.
