## Summary

Keep open. Current main still exposes the shared `read` action, but Telegram does not advertise, map, gate, dispatch, test, or document Telegram message reads; the prior inbound-store PR was closed unmerged and the remaining design needs maintainer approval for retention and privacy semantics.

## What This Changes

Add Telegram support for `openclaw message read` so agents can read recent messages from Telegram chats or groups.

## Best Possible Solution

Implement Telegram-owned observed-message reads with bounded runtime storage, explicit retention/privacy limits, no secondary `getUpdates` polling, and aligned adapter, discovery, runtime, config, docs, and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI dispatches shared `read` with target plumbing, while Telegram discovery, adapter mapping, runtime aliases, config gates, and docs omit `read`, leading to the unsupported-action path.

## Solution Assessment

Unclear. The requested `message read` surface is reasonable, but the maintainable implementation needs a Telegram-specific observed-message cache and maintainer-approved privacy/retention policy before code work proceeds.
