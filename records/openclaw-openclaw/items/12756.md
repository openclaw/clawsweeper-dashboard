## Summary

Keep open. Current main still has shared message-read plumbing, but the bundled Telegram plugin does not expose or implement Telegram `read`; the prior inbound-store PR is closed unmerged, and the remaining design needs Telegram retention/privacy review.

## What This Changes

The issue requests Telegram support for `openclaw message read` so agents can inspect recent Telegram chat or group messages.

## Best Possible Solution

Implement Telegram-owned observed-message reads with bounded retention, no secondary `getUpdates` polling, and aligned discovery, action mapping, runtime dispatch, config gating, CLI target/topic semantics, docs, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI target plumbing exist, while the Telegram adapter, discovery, runtime aliases, action gates, tests, and docs omit `read`, leading to the unsupported-action path.

## Solution Assessment

Unclear as written. The user-facing action is reasonable, but the maintainable implementation should be a bounded Telegram-owned observed-message store with explicit retention/privacy semantics, not direct Bot API history reads or another `getUpdates` poller.
