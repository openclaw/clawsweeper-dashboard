- `impact:session-state`: The requested action depends on retaining and querying inbound Telegram message state across tool calls.

## Summary

Current main and release v2026.6.6 still reject Telegram message-history reads, and the related implementation PR closed unmerged. The gap is valid, but maintainers must define Telegram's bot-observed-history, authorization, retention, and durable-storage contract before implementation.

## What This Changes

Add Telegram support for the shared message-tool read action so agents can query recent messages from Telegram chats.

## Best Possible Solution

Implement a Telegram-plugin-owned, bounded SQLite inbound-message ledger—or reuse a proven canonical ledger—then expose read through discovery, target and topic normalization, authorization, runtime dispatch, configuration, documentation, and restart-aware tests with clearly documented bot-observed-history semantics.

## Reproduction Assessment

Yes. Current main has a deterministic source reproduction: Telegram does not advertise or map the shared read action, so `openclaw message read --channel telegram` is rejected before any history query occurs.

## Solution Assessment

No approved best implementation exists yet. The closed standalone in-memory-store proposal was incomplete and conflicts with current canonical-state policy; the safer direction is bounded plugin-owned SQLite storage with explicit authorization and observed-history semantics.

## Risks / Open Questions

- Telegram bots cannot fetch arbitrary chat history, so the action would likely expose only messages observed and retained by OpenClaw rather than complete Telegram history.
- The read path needs explicit account, chat, forum-topic, allowlist, and agent/session authorization boundaries to prevent unintended disclosure.
- Retaining inbound content requires a deliberate SQLite owner, bounded retention, deletion and restart semantics, and privacy documentation; the closed PR's process-memory ring buffer is not compatible with current canonical-state policy.
