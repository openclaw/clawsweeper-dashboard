## Summary

Keep open. Current main still has the shared and CLI `read` action, but the Telegram plugin does not expose, map, gate, dispatch, test, or document Telegram message-history reads; the related inbound-store PR was closed unmerged, so there is no implementation to close against.

## What This Changes

Add Telegram support for `openclaw message read` and message-tool reads so agents can retrieve recent Telegram chat or group messages for monitoring workflows.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin as an explicit observed-only read feature backed by a bounded, privacy-reviewed runtime-owned inbound cache or equivalent store, then wire it through action discovery, adapter mapping, runtime dispatch, config/schema help, CLI/channel docs, and focused tests.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: the CLI dispatches `read`, shared target plumbing maps `--target` to `to`, and the Telegram adapter/runtime have no `read` mapping, so Telegram rejects the action as unsupported.

## Solution Assessment

Unclear until maintainers approve Telegram read semantics. The requested `message read` surface is the right user-facing target, but the maintainable implementation should be Telegram-owned observed-message storage with explicit retention and privacy limits rather than a direct Bot API history fetch.
