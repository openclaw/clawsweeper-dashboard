## Summary

Keep open: current main and v2026.6.6 still omit Telegram `message read` from discovery, action mapping, runtime dispatch, config gating, tests, and docs. The closed duplicate points back here, and the prior inbound-store PR was closed unmerged, so this remains the canonical feature/security design item.

## What This Changes

Request support for `openclaw message read --channel telegram` so agents can read recent Telegram chat messages for monitoring workflows.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin with bounded observed-message storage or an approved equivalent, explicit privacy/access semantics, no secondary `getUpdates` polling, action discovery/mapping/runtime/config/docs wiring, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main has a shared `read` action, but Telegram does not advertise, map, gate, or dispatch it, so the adapter still rejects Telegram read requests.

## Solution Assessment

No approved best implementation exists yet. An inbound observed-message store is plausible, but maintainers still need to decide retention, authorization, topic/account scoping, and Bot API semantics before code should proceed.
