## Summary

Keep open: current main and v2026.6.10 still do not expose Telegram `message read`; the repo now has Telegram message-cache primitives that could help, but no discovery, mapping, config gate, runtime action, CLI docs, or tests wire them into the shared read action.

## What This Changes

The issue requests `openclaw message read --channel telegram` support for reading recent Telegram group messages, similar to Slack and Discord.

## Best Possible Solution

Define Telegram `message read` in the Telegram plugin using the existing observed-message cache or another bounded retained-message contract, with explicit privacy/config gating, target/topic semantics, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes. Source inspection is enough: the shared `read` action exists, but Telegram discovery, mapping, runtime dispatch, config gating, tests, and docs still omit it on current main and v2026.6.10.

## Solution Assessment

Unclear until maintainers choose the retained-message contract. The best layer is the Telegram plugin, but exposing the current message cache as a user-facing read action needs explicit privacy, authorization, and scope decisions first.
