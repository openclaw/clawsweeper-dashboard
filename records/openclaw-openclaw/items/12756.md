## Summary

Keep open. Current main still has the shared `read` action and CLI command, but Telegram does not advertise, gate, map, dispatch, test, or document a Telegram `read` action; the related closed implementation PR was not merged, and the open `inbound_observed` PR is adjacent archive-hook work rather than `openclaw message read` support.

## What This Changes

Request to support `openclaw message read --channel telegram` for recent Telegram chat or group messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded existing message-cache state, with explicit action gating, authorization, retention, target/topic semantics, docs, and focused tests while avoiding secondary `getUpdates` polling.

## Reproduction Assessment

Yes, by source inspection. The shared `read` action and CLI command exist, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit a Telegram read action on current main.

## Solution Assessment

Unclear as originally phrased. A full Telegram history reader or secondary `getUpdates` poller is not the maintainable path; a bounded observed-message read from Telegram-owned runtime state is the safer direction after privacy and retention decisions.
