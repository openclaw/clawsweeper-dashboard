## Summary

Keep open. Current main still lacks Telegram support for the shared `message read` action, and the prior discussion correctly identifies the remaining retention, authorization, and platform-scope decisions.

## What This Changes

The issue requests `openclaw message read --channel telegram` support for recent Telegram group messages, comparable to Slack and Discord read behavior.

## Best Possible Solution

Add a Telegram-owned observed-message `read` action backed by the existing bounded message cache, with explicit platform-limit docs, retention policy, authorization gating, target/topic semantics, and focused tests.

## Reproduction Assessment

Yes, by source inspection. The shared CLI `read` path exists while Telegram discovery, mapping, config, docs, and runtime dispatch still omit a Telegram read action.

## Solution Assessment

Unclear as originally phrased. The maintainable solution is not a full Telegram group-history reader or a secondary `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after privacy and retention decisions. ([core.telegram.org](https://core.telegram.org/bots/api%20))
