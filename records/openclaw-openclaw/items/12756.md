## What This Changes

Add Telegram support for the shared message-history read action so agents can inspect recently received group messages.

## Best Possible Solution

Add a Telegram-plugin-owned, bounded SQLite history projection populated from already-authorized inbound updates, with per-account/chat/topic isolation, explicit retention and action gating, clear “observed while OpenClaw was running” semantics, and end-to-end CLI/tool tests; never poll `getUpdates` independently for history.

## Reproduction Assessment

Yes, at source level with high confidence: Telegram does not advertise or map `read`, and its adapter throws for unmapped actions. The same omission is present in current main and release `v2026.6.5`.

## Solution Assessment

No, the earlier process-local ring-buffer proposal is not the best current solution because it conflicts with the repository's SQLite-first state policy and leaves retention, authorization, and restart semantics underspecified; a plugin-owned bounded SQLite projection is safer.

## Risks / Open Questions

- Persisting inbound Telegram content creates a new privacy and authorization surface; account, chat, topic, sender, retention, deletion, and tool-access boundaries need an explicit design.
- Telegram bots cannot fetch arbitrary chat history through a normal history endpoint, so a second `getUpdates` consumer could conflict with polling/webhooks, consume pending updates, or return incomplete cross-chat data.
- Existing transient prompt history and the ingress queue have different lifecycle purposes; reusing either without a dedicated canonical history projection could expose deleted queue payloads or make restart behavior misleading.
