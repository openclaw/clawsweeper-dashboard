## Summary

Keep open. Current main still has the shared `read` action and CLI command, but Telegram does not advertise, map, dispatch, gate, test, or document Telegram read support; the attempted implementation was closed unmerged and the duplicate was closed back to this issue.

## What This Changes

Request to support `openclaw message read` and the shared message-tool `read` action for Telegram chats and groups with target, limit, and pagination-style parameters.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded cache state, explicit gating, authorization, retention, target/topic semantics, docs, and focused tests while avoiding secondary `getUpdates` polling.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist while Telegram discovery, mapping, runtime dispatch, config gates, tests, and docs still omit Telegram read support.

## Solution Assessment

Unclear as originally phrased. The user-facing action is valid, but the maintainable solution is a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided.
