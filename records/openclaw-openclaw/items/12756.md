## Summary

Keep open. Current main still has the shared `message read` command, but Telegram does not advertise, map, gate, dispatch, test, or document a read action; the existing observed-message cache gives a plausible backing store, but the user-facing semantics still need maintainer decisions.

## What This Changes

Request to add Telegram support for `openclaw message read` so agents can inspect recent Telegram chat or group messages.

## Best Possible Solution

Add a Telegram-owned observed-message `read` action backed by the existing bounded message cache, with explicit platform-limit docs, target/topic semantics, retention policy, access gating, and focused tests.

## Reproduction Assessment

Yes by source inspection. The shared CLI `read` path exists, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit `read`; I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as originally phrased. The maintainable solution is not a direct Bot API history read or secondary `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after maintainer decisions on privacy and retention.
