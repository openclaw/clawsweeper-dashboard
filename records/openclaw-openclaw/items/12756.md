## Summary

Keep open. Current main still does not expose Telegram `message read`; the newer observed-message cache provides useful groundwork but is only used for Telegram prompt/reply context today.

## What This Changes

The issue requests Telegram support for the shared `openclaw message read` action.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded message-cache state, with explicit platform-limit docs, target/topic semantics, and read-access gating.

## Reproduction Assessment

Yes for the current limitation: source inspection shows shared/CLI `read` plumbing exists, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit it. This is a feature gap rather than a broken documented Telegram behavior.

## Solution Assessment

Unclear as originally phrased. The maintainable direction is not a direct Bot API history read or a second `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after maintainer decisions on privacy and retention.
