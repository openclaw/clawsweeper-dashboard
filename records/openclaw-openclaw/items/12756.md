## Summary

Keep open. Current main still lacks end-to-end Telegram `message read` support, even though it now has an observed-message cache that could be a building block; the remaining work needs maintainer decisions on observed-only semantics, retention, and read-access gating.

## What This Changes

The issue requests `openclaw message read --channel telegram` support for recent Telegram chat messages.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded runtime state, with explicit docs for retention and Telegram platform limits.

## Reproduction Assessment

Yes. Source inspection on current main shows shared `read` and CLI plumbing exist, while Telegram discovery, mapping, runtime aliases, action gates, tests, and docs still omit it.

## Solution Assessment

Unclear as originally phrased. The user-facing action is reasonable, but the maintainable solution should use bounded observed-message reads from Telegram-owned runtime state rather than a direct Bot API history read or a second `getUpdates` poller.
