## Summary

Keep open. Current main and the latest release still have the shared `read` action and CLI command, but Telegram does not advertise, gate, map, dispatch, test, or document a Telegram `read` action; the remaining implementation also needs product/security policy for exposing observed Telegram message content.

## What This Changes

The issue requests support for `openclaw message read --channel telegram` so agents can read recent Telegram group messages for monitoring workflows.

## Best Possible Solution

Add a Telegram-owned observed-message read action backed by bounded message-cache state, with explicit action gating, authorization, retention, target/topic semantics, docs, and focused tests while avoiding a secondary polling/history reader.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared `read` action and CLI command exist, while Telegram discovery, mapping, config, docs, tests, and runtime dispatch still omit a Telegram read action.

## Solution Assessment

Unclear as originally phrased. The maintainable path is a bounded observed-message read from Telegram-owned cache state after privacy, retention, authorization, and topic semantics are decided; a full history reader or secondary polling path is not the safe default.
