## Summary

Keep open: current main and v2026.6.6 still do not advertise, map, gate, dispatch, test, or document Telegram `message read`, and the closed duplicate plus closed unmerged inbound-store PR leave this as the canonical remaining design item.

## What This Changes

Add Telegram support for the shared `message read` action so agents and CLI users can read recent Telegram chat messages for automation workflows.

## Best Possible Solution

Implement Telegram `message read` in the Telegram plugin with approved observed-message semantics, explicit authorization and retention boundaries, no secondary `getUpdates` polling, action discovery/mapping/runtime/config/docs wiring, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main has a shared `read` action, but Telegram does not advertise, map, gate, or dispatch it, so Telegram read requests remain unsupported.

## Solution Assessment

No approved best implementation exists yet. A bounded observed-message store is plausible, but maintainers still need to decide read scope, retention, authorization, and Telegram Bot API semantics before implementation.
