## Summary

Keep open. Current main and the latest release still expose shared `message read` plumbing but do not advertise, map, gate, dispatch, test, or document Telegram `read`; the remaining work is a real feature gap with privacy and retention semantics that need maintainer judgment before implementation.

## What This Changes

Add Telegram support for `openclaw message read` / message-tool `read` so agents can read recent Telegram chat messages for monitoring workflows.

## Best Possible Solution

Add a Telegram-owned observed-message `read` action backed by bounded message-cache state, with explicit platform-limit docs, target/topic semantics, retention policy, access gating, and focused tests.

## Reproduction Assessment

Yes for the current limitation by source inspection: shared/CLI `read` exists, while Telegram discovery, mapping, config, docs, and runtime dispatch still omit it. I did not run the CLI because this review is read-only.

## Solution Assessment

Unclear as originally phrased. The maintainable direction is not a direct Telegram Bot API history read or a second `getUpdates` poller; it should expose bounded observed-message reads from Telegram-owned runtime state after maintainer decisions on privacy and retention.
