## Label Justifications

- `P1`: Feishu topic session splitting breaks a real channel workflow and can lose conversation context for configured groups.
- `merge-risk: 🚨 compatibility`: The PR changes documented normal-group topic routing semantics for existing `group_topic` users.
- `merge-risk: 🚨 session-state`: Changing the Feishu peer key can orphan or split topic sessions and configured conversation bindings.
- `merge-risk: 🚨 security-boundary`: The PR expands authenticated Feishu message hydration before group and sender admission checks complete.

## Summary

Keep open: the Feishu session-splitting bug is still present on current main and latest release, and this branch has useful real Feishu proof, but the patch is not merge-ready because it can re-key existing normal-group sessions and expands authenticated Feishu message reads before admission checks.

## What This Changes

The PR changes Feishu topic-session hydration/routing so normal groups with Feishu topic message format and native topic groups prefer canonical `omt_*` topic IDs, adds Feishu regression tests, updates Feishu docs, and edits `CHANGELOG.md`.

## Best Possible Solution

Land one Feishu-plugin fix that reliably distinguishes topic-message-format normal groups or preserves existing `om_*` routing by default, moves new hydration behind admission gates, and keeps release-owned changelog generation out of this PR.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.8 shows normal `chat_type="group"` topic events can still choose `rootId` before `threadId`, and the PR body includes live before/after Feishu logs for the same split shape.

## Solution Assessment

No. The Feishu plugin is the right owner and the branch is a plausible fix, but it is not yet the safest fix because it changes existing session keys and performs broader pre-admission Feishu message reads.
