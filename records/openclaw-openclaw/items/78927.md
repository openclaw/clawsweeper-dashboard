## Summary

Keep open: the reported Feishu session split is plausible and the PR includes real Feishu logs, but the patch still conflates normal OpenClaw-created group threads with Feishu topic-message-format groups and expands authenticated message reads before admission gates.

## What This Changes

The PR broadens Feishu group-topic hydration to query `rootId` for canonical `omt_*` thread IDs, suppresses `rootId` during session routing when topic context is detected, and updates Feishu docs, changelog, and bot tests.

## Best Possible Solution

Keep the fix Feishu-local: reliably distinguish Feishu topic-message-format groups from OpenClaw-created normal reply threads, hydrate canonical `omt_*` IDs only after admission gates, and preserve existing root-keyed normal-thread behavior.

## Reproduction Assessment

Yes for source/log-level review: current main normal `group` topic scopes prefer `rootId` before `threadId`, while the PR body has before/after Feishu WebSocket logs for the reported split. I did not run a live Feishu tenant from this read-only checkout.

## Solution Assessment

No. The Feishu plugin is the right surface, but this patch uses `om_*` root presence as too broad a topic discriminator and performs the new lookup before access checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve normal-group root-keyed sessions:** `extensions/feishu/src/bot.ts:589`
  - body: This suppresses `rootId` whenever a normal group event carries an `omt_*` thread and `om_*` root. Current tests and docs still require normal group reply threads that OpenClaw creates to use the `om_*` root key, so without a reliable Feishu topic-message-format discriminator these sessions can move to `omt_*` and split existing conversations.
  - confidence: 0.9
- **[P2] Gate topic hydration behind admission:** `extensions/feishu/src/bot.ts:546-551`
  - body: The expanded hydration path can call `getMessageFeishu` for normal group events before disabled-group, group allowlist, sender allowlist, and mention checks return. Move the lookup behind those admission gates, or add a non-fetch discriminator, so ignored traffic does not trigger authenticated Feishu reads.
  - confidence: 0.84
