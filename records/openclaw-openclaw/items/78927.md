## Summary

Keep open: the PR addresses a real Feishu topic-session bug that current main does not fully cover, but the proposed patch still has two merge-blocking issues around normal-group topic discrimination and pre-admission Feishu API reads.

## What This Changes

The PR changes Feishu topic routing to hydrate canonical `omt_*` topic IDs from replied-to messages, suppress `rootId` when topic context is detected, and updates Feishu docs, tests, and changelog.

## Best Possible Solution

Land a Feishu plugin fix that normalizes real Feishu topic-message-format groups to canonical `omt_*`, preserves OpenClaw-created normal group threads on `om_*`, and performs extra message lookups only after admission.

## Reproduction Assessment

Yes for source/log-level review: current main still prefers `rootId` over `threadId` for normal `chat_type: "group"` topic scopes, and the PR body includes before/after live Feishu WebSocket logs. I did not rerun a live Feishu tenant from this read-only checkout.

## Solution Assessment

No. The direction is right for real Feishu topic-message-format groups, but this implementation uses `om_*` root presence as too broad a discriminator and should move broadened message hydration behind admission gates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep root-based sessions for normal Feishu threads:** `extensions/feishu/src/bot.ts:589`
  - body: This branch treats a normal-group `omt_*` thread plus `om_*` root as a native topic signal and suppresses `rootId`. Current docs/tests keep `rootId` ahead of `threadId` for normal group reply-created threads, so this can re-key an existing `om_*` session to `omt_*` and split it.
  - confidence: 0.9
- **[P2] Gate broadened hydration behind group admission:** `extensions/feishu/src/bot.ts:537-549`
  - body: The new normal-group hydration path still runs before disabled-group, group allowlist, and sender allowlist returns. With topic session scope enabled, a non-admitted normal group reply carrying an `om_*` root can now trigger an authenticated Feishu message read, so move this lookup behind admission gates while still running before session routing.
  - confidence: 0.84
