## Label Justifications

- `P1`: The PR targets a broken Feishu topic workflow affecting real users and changes session routing plus access-boundary behavior in a compatibility-sensitive channel path.
- `merge-risk: 🚨 compatibility`: The patch changes established Feishu normal-group topic routing priority from `rootId` to `threadId` for some existing event shapes.
- `merge-risk: 🚨 session-state`: Changing the peer key can orphan or split existing Feishu topic sessions and configured conversation bindings.
- `merge-risk: 🚨 security-boundary`: The expanded hydration path can fetch Feishu message data before normal group/sender/mention admission checks complete.

## Summary

Keep open: the Feishu bug is real and the PR has useful live proof, but the patch is not merge-ready because it rekeys current normal-group topic sessions and expands a message-content fetch before group/sender admission.

## What This Changes

This PR changes Feishu group-topic routing to hydrate thread IDs from the replied-to message, suppress root IDs when an `omt_*` topic is detected, updates Feishu docs, adds regression tests, and edits the changelog.

## Best Possible Solution

Preserve root-id routing for existing normal-group reply-created threads unless Feishu topic-message-format mode is positively identified or migrated, keep hydration behind admission gates, and leave `CHANGELOG.md` to release generation.

## Reproduction Assessment

Yes. The PR body includes before/after live Feishu WebSocket logs, and current source/tests show normal-group topic routing currently prefers `rootId`; I did not run tests because this review is read-only.

## Solution Assessment

No. The Feishu plugin is the right surface, but the prefix-based normal-group discriminator is too broad and the new message read should run after admission gates or use a metadata-only owner-approved path.
