## Summary

Keep open: current main and the latest release still forward MCP tool input records without dropping null values, and the unmerged linked PR is the active candidate fix rather than an implemented solution.

## What This Changes

The issue reports that OpenClaw mishandles optional MCP tool arguments set to JSON null, causing a streamable-http MCP tool call to fail where the same call succeeds in another MCP client.

## Best Possible Solution

Keep this issue open while maintainers review the linked candidate fix; the final fix should normalize optional-null MCP arguments at the bundle-MCP boundary with focused regression coverage and avoid changing meaningful required-null parameters.

## Reproduction Assessment

Yes, with medium confidence from source and reporter evidence: the reported release and current main both forward null-containing input records through the bundle-MCP call path. I did not run the awslabs EKS MCP server or capture a live wire payload.

## Solution Assessment

Unclear until the linked PR is reviewed: stripping optional nulls before MCP dispatch is a plausible narrow fix, but a schema-aware version may be safer than dropping every top-level null value.
