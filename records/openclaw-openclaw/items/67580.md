## Summary

Keep open. Current main still has the reported leak shape: inline `status: staged - Candidate:` snippets are not classified as contaminated, promotion rehydration has no managed-dreaming fence guard, and `buildPromotionSection` still writes the rehydrated raw snippet into `MEMORY.md`; the open implementation candidate is https://github.com/openclaw/openclaw/pull/68774.

## What This Changes

The issue reports that memory-core dreaming promotion can append raw Candidate/status/confidence/evidence artifacts from managed dreaming output into `MEMORY.md` instead of durable distilled memory.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/68774 or an equivalent memory-core fix for inline metadata detection and rehydration-time dreaming-fence exclusion, then handle broader distillation and recall-threshold policy through https://github.com/openclaw/openclaw/issues/67363 or a maintainer-directed follow-up.

## Reproduction Assessment

Yes, at source level. The current detector misses the reported inline metadata prefix, and code inspection confirms raw snippet writing plus no managed-fence check during promotion rehydration.

## Solution Assessment

No for current main. The narrow fix in https://github.com/openclaw/openclaw/pull/68774 is the best immediate repair for the Candidate artifact leak, while LLM distillation and recall-threshold changes need maintainer policy direction.
