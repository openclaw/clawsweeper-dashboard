## Summary

Keep open. Current main has adjacent risk/tags metadata, toolKind/toolInputKind, derivedPaths, trusted policies, and Security Matrix draft work, but it still lacks the requested optional behavioral capability descriptor contract on tool definitions/plugin metadata and does not project those descriptors to before_tool_call or Gateway tool inventories; this remains a security-sensitive Plugin SDK/API design item.

## What This Changes

Add optional self-declared behavioral capability descriptors to built-in and plugin tool definitions so policy plugins can reason about stable metadata instead of tool names.

## Best Possible Solution

Define an additive, maintainer-approved Plugin SDK and Gateway protocol descriptor contract, project it through built-in/plugin tool metadata, before_tool_call, catalog/effective inventories, docs, baselines, and contract tests, and coordinate the taxonomy with #48503, #62962, and #12678.

## Reproduction Assessment

Yes at source level: current tool definitions, plugin metadata, hook dispatch, and Gateway catalog/effective schemas can be inspected and show the descriptor fields are absent; a runtime failing repro is not applicable for this API feature request.

## Solution Assessment

The direction is plausible, but not ready as a one-off field addition because the best fix needs maintainer-approved taxonomy and integration with existing risk/tags, toolKind/toolInputKind, trusted policy, and related security proposals.
