## Summary

Keep open: current main and v2026.6.10 still lack stable behavioral capability descriptors for tools; existing surfaces expose tool name, params, identity hints, derived paths, risk, and tags, but not the requested reads/writes/sends/privileged/autonomous descriptor contract.

## What This Changes

Request to add optional behavioral capability descriptors to built-in and plugin tool definitions so policy and audit code can reason about tool actions without brittle name matching.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK and Gateway protocol descriptor contract, then project it through built-in and plugin tool metadata, trusted policies, before_tool_call events, tools.catalog/tools.effective, docs, API baselines, and focused contract tests.

## Reproduction Assessment

Yes, as a source-level API gap rather than an end-user bug: current main and v2026.6.10 can be inspected and do not expose the requested descriptor contract on central tool, plugin metadata, hook, trusted-policy, or Gateway inventory surfaces.

## Solution Assessment

Unclear as a final implementation shape: optional descriptor flags are plausible, but the best fix needs a maintainer-approved security taxonomy and trust model rather than a one-off field addition.
