## Summary

Keep open. Current main still lacks optional behavioral capability descriptors on tool definitions, plugin tool metadata, before_tool_call/trusted-policy payloads, and Gateway tool projections; the related open design items are complementary, not merged replacements.

## What This Changes

The issue requests optional tool capability descriptors so policy plugins can reason about tool behavior without brittle tool-name matching.

## Best Possible Solution

Design one additive, versioned Plugin SDK and Gateway contract for tool behavior descriptors, project it through catalog/effective inventory and hook policy surfaces, and document trust/unknown semantics with focused contract tests.

## Reproduction Assessment

No, not as an end-user bug reproduction because this is a feature/API request. The gap is source-reproducible on current main across the tool type, plugin metadata, hook dispatch, Gateway schemas, tests, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The proposal is plausible, but the maintainable solution should coordinate descriptor taxonomy, trust semantics, Gateway projection, and the related action/provenance and security-classification issues.
