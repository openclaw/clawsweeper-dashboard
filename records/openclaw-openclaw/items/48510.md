## Summary

Keep open. Current main and v2026.6.10 still lack stable behavioral capability descriptors on tool definitions, plugin tool metadata, before_tool_call/trusted-policy payloads, and Gateway tool inventory schemas; the related open items are complementary security/provenance/audit work rather than a merged implementation.

## What This Changes

The issue requests optional structured capability descriptors on tool definitions so policy, permission, documentation, and audit code can reason about tool behavior without brittle name matching.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK and Gateway descriptor contract, then project it through built-in/plugin tool metadata, trusted policies, before_tool_call events, catalog/effective inventory, docs, API baselines, and focused contract tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 can be inspected and do not expose the requested descriptor contract on the central tool, plugin metadata, hook, trusted-policy, docs, or Gateway inventory surfaces. A runtime failing reproduction is not applicable because this is a feature/API request.

## Solution Assessment

Unclear as a direct flag addition: optional descriptors are plausible, but the best fix needs a maintainer-approved security taxonomy and trust model rather than a one-off field. The safer path is one coordinated SDK/Gateway contract aligned with the related provenance and Security Matrix work.
