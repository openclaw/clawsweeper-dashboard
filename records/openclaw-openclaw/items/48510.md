## Summary

Keep open. Current main and the latest release still do not expose stable behavioral capability descriptors for tools, and the related open design issues are complementary rather than superseding fixes.

## What This Changes

The issue requests optional capability descriptors on tool definitions so built-in and plugin tools can declare behavioral metadata for policy, audit, onboarding, and catalog surfaces.

## Best Possible Solution

Design an additive Plugin SDK and Gateway protocol contract for behavioral tool descriptors, project it through catalog/effective inventory and tool-hook policy surfaces, and document trust, unknown, and enforcement semantics with contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is source-verifiable on current main and v2026.5.18 across tool types, plugin metadata, hook dispatch, Gateway projections, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The direction is plausible, but the maintainable solution should coordinate SDK, Gateway protocol, policy, and trust semantics rather than adding a one-off field.
