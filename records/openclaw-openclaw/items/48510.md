## Summary

Keep open. Current main still lacks the requested stable behavioral capability descriptor contract for tools; related host-hook and metadata work exposes policy hooks, risk, tags, and provenance surfaces, but not self-declared tool behavior flags.

## What This Changes

The issue requests optional behavioral capability descriptors on built-in and plugin tool definitions so policy, onboarding, catalog, and audit logic can classify tools without name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, project it through tools.catalog/tools.effective, and expose it to trusted policies plus before_tool_call hooks with docs and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The verification path is static inspection of current tool types, plugin metadata, hook payload/dispatch, Gateway schemas, docs, and related design items.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are plausible, but the exact taxonomy and relationship to #48503 should be decided before changing public SDK, Gateway protocol, and security-policy inputs.
