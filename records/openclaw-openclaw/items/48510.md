## Summary

Keep open. Current main still lacks stable behavioral capability descriptors on the tool type, plugin tool metadata, before_tool_call/trusted-policy payloads, and Gateway catalog/effective projections; related items are complementary or explicitly say they do not implement this request.

## What This Changes

Request adds optional behavioral capability descriptors to tool definitions so policy, onboarding, catalog, and audit surfaces can reason about tool behavior without name matching.

## Best Possible Solution

Define one additive Plugin SDK and Gateway protocol contract for behavioral tool descriptors, project it through catalog/effective and trusted/before-hook surfaces, document trust and unknown semantics, and cover it with SDK/protocol contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is high-confidence and source-verifiable on current main across the tool type, plugin metadata, before_tool_call dispatch, Gateway schemas, effective inventory, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable solution should coordinate descriptor taxonomy, trust semantics, Gateway projection, and the related provenance/security-classification items.
