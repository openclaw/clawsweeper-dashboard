## Summary

Keep open. Current main and v2026.5.19 still do not expose stable behavioral capability descriptors for tools across the core tool type, plugin tool metadata, before_tool_call/trusted-policy dispatch, or Gateway tool projections; the related open design items are complementary rather than merged superseding fixes.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so policy and audit code can reason about tool effects without hardcoded name matching.

## Best Possible Solution

Define an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool descriptors, project it through tools.catalog/tools.effective and hook policy surfaces, and document trust, unknown, and enforcement semantics with contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is high-confidence and source-verifiable on current main and v2026.5.19 across the inspected tool type, metadata, hook dispatch, Gateway schema, and docs surfaces.

## Solution Assessment

Unclear until maintainers choose the public contract. The direction is plausible, but the maintainable solution should coordinate descriptor taxonomy, trust semantics, Gateway projection, and the related action/provenance and tool-security-classification issues.
