## Summary

Keep open. Current main has adjacent `derivedPaths`, trusted tool policy, and plugin tool metadata surfaces, but it still lacks stable behavioral capability descriptors for tools; this is a public Plugin SDK/Gateway protocol and security-policy design item, not stale cleanup.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so policy plugins can reason about tool effects without hardcoded tool-name lists.

## Best Possible Solution

Define an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool descriptors, then project it consistently through tool metadata, tool catalogs, effective inventory, trusted policies, and before_tool_call payloads with docs and contract tests.

## Reproduction Assessment

Not applicable as a feature request. The gap is source-verifiable on current main across the hook type, tool metadata registration, trusted policy dispatch, gateway schemas, effective inventory, and public hook docs.

## Solution Assessment

Yes, keeping this open for maintainer API/security design is the best current path. A narrow implementation should be additive and versioned, not a one-off field added to only one projection path.
