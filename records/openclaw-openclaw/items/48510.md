## Summary

Keep open. Current main and latest release v2026.6.8 still do not expose optional behavioral capability descriptors on core tool definitions, plugin tool metadata, before_tool_call/trusted-policy payloads, or Gateway tool inventory schemas; the related items are complementary security/API design tracks, not merged replacements.

## What This Changes

Request adds optional behavioral capability descriptors to built-in and plugin tool definitions so policy, onboarding, documentation, and audit code can classify tool calls without hardcoded name lists.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK and Gateway protocol descriptor contract, then project it through built-in/plugin tool metadata, trusted policies, before_tool_call events, catalog/effective inventory, docs, API baselines, and focused contract tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 can be inspected and do not expose the requested capability descriptor contract on the central tool, plugin metadata, hook, trusted-policy, or Gateway inventory surfaces. A runtime bug reproduction is not applicable because this is a feature/API request.

Unclear as proposed: optional flags are a plausible building block, but the best fix needs a maintainer-approved security taxonomy and trust model rather than a one-off field addition.
