## Summary

Keep open. Current main and v2026.6.6 still lack optional behavioral capability descriptors on tool definitions, plugin tool metadata, before_tool_call/trusted policy payloads, and Gateway tool inventories; related open items are overlapping security/API design work, not merged replacements.

## What This Changes

The issue requests optional tool capability descriptors so policy plugins can reason about tool behavior without brittle tool-name matching.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK and Gateway protocol descriptor contract, project it through built-in/plugin tool metadata, before_tool_call, trusted policies, tools.catalog/tools.effective, docs, API baselines, and contract tests while coordinating the taxonomy with the related security-policy issues.

## Reproduction Assessment

Yes, at source level: the current central tool type, plugin metadata registration, hook event, trusted policy dispatch, and Gateway inventory schemas can be inspected and do not expose the requested descriptor contract. A runtime bug repro is not applicable because this is a feature/API request.

## Solution Assessment

Unclear as proposed. Descriptor flags are a plausible building block, but the best fix needs maintainer-approved security taxonomy and integration with action classification, grants, audit, and unknown-by-default semantics rather than a one-off field addition.
