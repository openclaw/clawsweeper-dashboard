## Summary

Keep open. Current main and v2026.5.22 still expose tool names, params, display metadata, risk, tags, and narrow tool identity hints, but not stable behavioral capability descriptors across the tool type, plugin metadata, before_tool_call/trusted-policy events, or Gateway projections.

## What This Changes

Request to add optional behavioral capability descriptors to tool definitions so policy plugins can reason about tool effects without brittle name matching.

## Best Possible Solution

Define one additive Plugin SDK and Gateway protocol contract for behavioral tool descriptors, project it through catalog/effective and trusted/before-hook surfaces, document trust and unknown semantics, and cover it with SDK/protocol contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is high-confidence and source-verifiable on current main and v2026.5.22 across tool types, plugin metadata, before_tool_call dispatch, Gateway schemas, effective inventory, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable solution should coordinate descriptor taxonomy, trust semantics, Gateway projection, and related provenance/security-classification work.
