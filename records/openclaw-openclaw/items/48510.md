## Summary

Keep open. Current main still lacks the requested stable behavioral capability descriptor contract for tools; adjacent host-hook/tool metadata work adds trusted policy stages, derived path hints, display/risk/tags metadata, and effective inventory projection, but not self-declared behavioral capabilities for policy decisions.

## What This Changes

Request to add optional behavioral capability descriptors to built-in and plugin tool definitions and surface them for policy, catalog, audit, and hook consumers.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capabilities, project it through catalog/effective inventory, and expose it to trusted policies plus before_tool_call events with docs and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The current gap is source-verifiable because the central tool type, plugin metadata registration, before_tool_call payload, dispatch path, docs, and Gateway tool schemas omit behavioral capability descriptors.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but taxonomy, unknown defaults, trust tier, and coordination with action/provenance proposals should be settled before implementation.
