## Summary

Keep open. Current main has adjacent trusted-policy and plugin tool metadata surfaces, but still does not define stable behavioral capability descriptors for tools; the related action/provenance and tool-security classification threads remain open complementary design work, not merged superseding implementations.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so policy plugins can reason about tool effects without brittle name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, then project it through tool definitions, plugin metadata, tools.catalog/tools.effective, trusted policies, before_tool_call, docs, API baselines, and contract tests with explicit trust and unknown semantics.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature/API request. The gap is source-verifiable: current main’s tool type, plugin metadata, hook event, dispatch path, Gateway schemas, effective inventory, and docs omit behavioral capability descriptors.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable path is a coordinated SDK/protocol/security design rather than a one-off capabilities field added to one type.
