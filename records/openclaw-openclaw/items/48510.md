## Summary

Keep open. Current main still lacks the requested stable behavioral capability descriptor contract for tools; adjacent trusted tool policies, plugin tool display/risk/tags metadata, `derivedPaths`, and tools catalog/effective inventory projections do not let tools self-declare `readsExternalData`, `sendsExternally`, privileged execution, persistent writes, or autonomous-action behavior.

## What This Changes

The issue requests optional tool capability descriptors so built-in and plugin tools can self-declare behavioral traits for policy, documentation, onboarding, and audit consumers.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capabilities, project it through catalog/effective inventory, and surface it to trusted policies plus hook events with docs and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The current gap is source-verifiable because the inspected tool types, plugin metadata, hook event, dispatch path, docs, and Gateway schemas omit the requested descriptor fields.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but taxonomy, unknown defaults, trust tier, and coordination with action/provenance proposals should be settled before implementation.
