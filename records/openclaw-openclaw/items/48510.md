## Summary

Keep open. Current main and the latest release still lack stable behavioral capability descriptors for tools; the existing host-hook and tool metadata surfaces cover names, params, identity hints, display metadata, risk, and tags, but not the requested reads/writes/sends/privileged/autonomous descriptor contract.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so policy plugins can classify tool behavior without brittle tool-name matching.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK and Gateway protocol descriptor contract, then project it through built-in/plugin tool metadata, trusted policies, before_tool_call events, catalog/effective inventory, docs, API baselines, and focused contract tests.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 can be inspected and do not expose the requested descriptor contract on the central tool, plugin metadata, hook, trusted-policy, or Gateway inventory surfaces. A runtime failing reproduction is not applicable because this is a feature/API request.

## Solution Assessment

Unclear as proposed: optional flags are plausible, but the best fix needs a maintainer-approved security taxonomy and trust model rather than a one-off field addition.
