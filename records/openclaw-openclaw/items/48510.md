## Summary

Keep open. Current main still has adjacent trusted-policy, tool metadata, derived path, and inventory surfaces, but it does not expose stable behavioral capability descriptors for tools; the related before_tool_call provenance/classification work remains open and the closed SDK follow-up explicitly did not implement this proposal.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so built-in and plugin tools can declare stable policy metadata such as external reads, persistent writes, outbound sends, privileged execution, and autonomous actions.

## Best Possible Solution

Define an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, let built-in and plugin tools declare it optionally, project it through catalog/effective inventory, and surface it to trusted policies plus before_tool_call with docs and contract tests.

## Reproduction Assessment

Not applicable as a feature request. The current gap is source-verifiable: current main’s hook, metadata, catalog, and effective inventory contracts omit behavioral capability descriptors.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but a one-off field addition would be weaker than an additive SDK/protocol design coordinated with the related action/provenance and tool-security classification issues.
