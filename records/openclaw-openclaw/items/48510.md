## Summary

Keep open. Current main has adjacent trusted-policy and tool-metadata surfaces, but still does not expose stable behavioral capability descriptors for tools; related design items such as https://github.com/openclaw/openclaw/issues/48503 and https://github.com/openclaw/openclaw/issues/62962 remain open rather than merged superseding implementations.

## What This Changes

Add optional behavioral capability descriptors to tool definitions and expose them to policy, catalog, inventory, and hook consumers so plugins do not rely on tool-name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, project it through catalog and effective inventory, surface it to trusted policies and before_tool_call, and cover it with docs plus contract tests.

## Reproduction Assessment

Not applicable as an end-user reproduction because this is a feature request. The gap is source-verifiable: current main’s hook event, dispatch path, plugin metadata, catalog schema, and effective inventory types omit behavioral tool capability descriptors.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable solution should be a coordinated SDK/protocol design rather than a one-off field addition.
