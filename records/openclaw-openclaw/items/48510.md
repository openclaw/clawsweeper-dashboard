## Summary

Keep open: current main and v2026.6.10 still do not provide per-tool behavioral capability descriptors, and the related policy/provenance work is adjacent rather than a merged replacement.

## What This Changes

The issue requests optional capability metadata on built-in and plugin-registered tool definitions so policy, audit, docs, and hook consumers can reason about tool behavior without brittle name matching.

## Best Possible Solution

Keep this as the focused per-tool descriptor design item and coordinate it with the broader capability-permission and hook-provenance work before adding a public SDK/protocol contract.

## Reproduction Assessment

Yes. This is source-reproducible as an API gap: current main and v2026.6.10 expose tool names, params, identity hints, risk, tags, and derived paths, but not declared behavioral capability descriptors.

## Solution Assessment

Unclear as a final implementation shape. Optional descriptors are plausible, but the best fix needs maintainer-approved taxonomy, unknown/default semantics, and coordination with adjacent capability and provenance proposals.
