## Summary

Keep open. Current main and v2026.6.8 still lack the requested behavioral capability descriptor contract on tool definitions, plugin tool metadata, hook/trusted-policy payloads, and Gateway tool inventory surfaces; related items are overlapping security/API design work, not merged replacements.

## What This Changes

The issue requests optional capability descriptors on tool definitions so security policy plugins can reason about tool behavior without brittle tool-name lists.

## Best Possible Solution

Define one additive, maintainer-approved Plugin SDK and Gateway protocol descriptor contract, then project it through built-in/plugin tool metadata, hooks, trusted policy, catalog/effective inventory, docs, API baselines, and contract tests.

## Reproduction Assessment

Yes, at source level: current main’s central tool, plugin metadata, hook, trusted policy, and Gateway inventory surfaces can be inspected and do not expose the requested capability descriptor contract. A runtime bug repro is not applicable because this is a feature/API request.

## Solution Assessment

Unclear as proposed: descriptor flags are a plausible building block, but the best fix needs a maintainer-approved security taxonomy and integration with grants, audit, provenance, and unknown-by-default semantics.
