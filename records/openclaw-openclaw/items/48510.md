## Summary

Keep open. Current main and the latest shipped release still lack stable behavioral capability descriptors on tool definitions, plugin tool metadata, before_tool_call policy events, and Gateway tool projections; the related security/API issues remain complementary rather than merged superseding fixes.

## What This Changes

The issue requests optional structured capability descriptors on tool definitions so policy, onboarding, audit, and docs surfaces can reason about tool behavior without brittle tool-name lists.

## Best Possible Solution

Define one additive Plugin SDK and Gateway protocol contract for behavioral tool descriptors, project it through catalog/effective and trusted/before hook surfaces, document trust and unknown semantics, and cover it with SDK/protocol contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is high-confidence and source-verifiable on current main and v2026.5.22 across tool types, plugin metadata, before_tool_call dispatch, Gateway schemas, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable solution should coordinate descriptor taxonomy, trust semantics, Gateway projection, and the related provenance/security-classification items.
