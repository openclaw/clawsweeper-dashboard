## Summary

Keep open. Current main still lacks a stable behavioral capability descriptor contract for tools: the core tool type, plugin tool metadata registration, before_tool_call payload, trusted-policy dispatch, tools.catalog schema, tools.effective schema, and effective inventory projection expose names, params, display metadata, risk, tags, source, and run/tool correlation, but not self-declared tool behavior flags. Related #48503 remains an open complementary design item, not a merged replacement.

## What This Changes

The issue requests optional self-declared behavioral capability metadata on tool definitions so policy plugins can reason about tool behavior without brittle tool-name lists.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, allow built-in and plugin-owned tools to declare it optionally, project it through tools.catalog and tools.effective, and surface it to trusted policies plus before_tool_call payloads with API baselines, docs, and focused contract tests updated together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of the current tool type, plugin metadata registration, hook payload, dispatch code, gateway schemas, docs, and the related open design context.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but the exact taxonomy and relationship to #48503 should be settled first because this becomes public SDK, Gateway protocol, and security-policy input.
