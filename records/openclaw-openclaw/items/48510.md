## Summary

Keep open. Current main and the latest shipped release still lack stable behavioral capability descriptors for tools; existing surfaces expose only names, params, narrow tool discriminators, derived path hints, risk, tags, and display metadata. The related open security/API items are complementary rather than merged superseding fixes.

## What This Changes

Request to add optional tool capability descriptors to core and plugin tool definitions and surface them to policy, catalog, documentation, and audit flows.

## Best Possible Solution

Define one additive Plugin SDK and Gateway protocol contract for behavioral tool descriptors, project it through catalog/effective and trusted/before hook surfaces, document trust and unknown semantics, and cover it with SDK/protocol contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is high-confidence and source-verifiable on current main across the tool type, plugin metadata, before_tool_call dispatch, Gateway schemas, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable solution should coordinate descriptor taxonomy, trust semantics, Gateway projection, and the related provenance/security-classification items.
