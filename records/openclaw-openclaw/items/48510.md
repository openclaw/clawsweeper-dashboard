## Summary

Keep open. Current main and the latest shipped release still lack stable behavioral capability descriptors for tools, and related design items at https://github.com/openclaw/openclaw/issues/48503 and https://github.com/openclaw/openclaw/issues/62962 remain complementary open work rather than merged superseding fixes.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so policy, audit, onboarding, and documentation can reason about tool behavior without brittle name matching.

## Best Possible Solution

Design an additive Plugin SDK and Gateway protocol contract for behavioral tool capability descriptors, project it through catalog/effective inventory, surface it to trusted policies and before_tool_call, and cover the trust semantics with docs and contract tests.

## Reproduction Assessment

Not applicable as an end-user bug reproduction because this is a feature/API request. The gap is source-verifiable on current main and v2026.5.12: central hook, metadata, catalog, and inventory surfaces omit behavioral tool descriptors.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the maintainable solution should coordinate SDK, Gateway protocol, policy, and trust-model semantics rather than adding a one-off field.
