## Summary

Keep open. Current main has adjacent trusted tool policy, display/risk/tag metadata, derived path hints, catalog projection, and effective inventory work, but it still lacks the requested self-declared behavioral capability descriptor contract for tools.

## What This Changes

The issue requests optional behavioral capability descriptors on tool definitions so policy plugins can reason about tool effects without brittle name matching.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capabilities, project it through catalog/effective inventory, and expose it to trusted policies plus `before_tool_call` with docs, API baselines, and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The current gap is source-verifiable because the central tool type, plugin metadata registration, hook payload, dispatch path, docs, and Gateway schemas omit behavioral capability descriptors.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but taxonomy, unknown defaults, trust tier, and coordination with the related action/provenance proposal should be settled before implementation.
