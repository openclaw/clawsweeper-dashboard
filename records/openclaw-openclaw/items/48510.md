## Summary

Keep open. Current main still lacks the requested stable behavioral capability descriptor contract for tools; the relevant SDK, hook, trusted-policy, catalog, and effective-inventory surfaces expose names, params, ownership, risk, tags, and correlation IDs, but not self-declared behavior flags.

## What This Changes

Add optional self-declared behavioral capability descriptors to built-in and plugin-registered tool definitions for policy, onboarding, documentation, and audit use cases.

## Best Possible Solution

Design an additive, versioned Plugin SDK and Gateway protocol contract for behavioral tool capabilities, project it through tools.catalog/tools.effective, and expose it to trusted policies and before_tool_call hooks with docs and contract tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The verification path is static inspection of the current tool type, plugin metadata registration, hook payload/dispatch, Gateway schemas, docs, and related open design items.

## Solution Assessment

Unclear pending maintainer design. Capability descriptors are a plausible direction, but the exact taxonomy and relationship to #48503 should be settled before implementation because this becomes public SDK, Gateway protocol, and security-policy input.
