## Label Justifications

- `P2`: The missing deterministic completion-delivery control affects real multi-agent and messaging-channel workflows but is a bounded product enhancement rather than an emergency.
- `impact:message-loss`: The requested policy directly controls whether sub-agent completion messages are delivered, suppressed, or routed to a different recipient.

## Summary

Current main still lacks a supported framework-level suppression control: top-level sub-agent completions announce externally by default, while deterministic no-delivery is only internal bookkeeping and public suppression remains model-token driven. The issue should remain open for a maintainer product decision because related requests broaden the desired contract to parent-session and per-channel routing rather than proving a single boolean/config shape is canonical.

## What This Changes

Add a deterministic config and/or `sessions_spawn` option that prevents sub-agent completion announcements from being posted directly to the requester channel.

## Best Possible Solution

Define one typed completion-delivery policy that can select normal channel delivery, parent-session handoff, or suppression, preserve current channel delivery as the default, and map intentional suppression to a successful not-required lifecycle state rather than a delivery failure.

## Reproduction Assessment

Yes, at source level. Current main has no public `sessions_spawn` suppression parameter or sub-agent announce policy, and its docs still require model-emitted `ANNOUNCE_SKIP` or `NO_REPLY` for suppression.

## Solution Assessment

No, the exact standalone boolean/config proposal is not yet the best established solution. A single typed delivery policy should first reconcile this issue with parent-session routing and per-channel suppression so OpenClaw does not create competing sources of truth.
