## Summary

Keep this issue open: current main still has no documented config option or public `sessions_spawn` parameter for deterministic suppression. The internal `expectsCompletionMessage` field is outside the public schema, while the related open requests cover broader parent routing and per-channel policy rather than replacing this narrower requirement.

## What This Changes

Request a deterministic framework-level control to suppress sub-agent completion announcements globally or per spawn.

## Best Possible Solution

Define one closed completion-routing policy with channel, parent, and skip states, then expose only the agreed global and per-spawn overrides and thread the policy through native and ACP registration, descendant gating, retry bookkeeping, cleanup, and delivery.

## Reproduction Assessment

Yes at the source level: current main has framework-driven completion announcements, while its public tool schema, strict config schema, and documentation provide no deterministic user suppression control. No live Telegram run was performed during this read-only review.

## Solution Assessment

No, the proposed standalone boolean or three-value config is not yet clearly the best permanent API. Suppression should be designed as one state of the canonical completion-routing contract shared with parent-routing and per-channel requirements.
