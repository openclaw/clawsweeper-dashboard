## Summary

Keep open. Current main has improved sub-agent completions by steering/reviewing through the requester agent in more cases, but it still does not expose a documented framework-level `sessions_spawn` or `agents.defaults.subagents` suppression control; the remaining work is a product/API decision, not conservative cleanup.

## What This Changes

Request a deterministic config option or per-spawn parameter to suppress sub-agent completion announces without relying on `ANNOUNCE_SKIP` or `NO_REPLY`.

## Best Possible Solution

Define one backward-compatible completion audience/suppression contract that defaults to current behavior, supports deterministic skip/internal-only delivery where approved, and updates schema, docs, and focused delivery tests together.

## Reproduction Assessment

Yes, at source level: current main exposes no public `sessions_spawn` announce/silent parameter and no `agents.defaults.subagents.announcePolicy`, while docs still document only exact-token suppression. I did not run a live Telegram reproduction because the source gap is direct and this review is read-only.

## Solution Assessment

Unclear pending maintainer direction. A deterministic skip control is plausible, but it should be coordinated with the broader routing and per-channel policy threads so OpenClaw does not grow competing completion-delivery knobs.
