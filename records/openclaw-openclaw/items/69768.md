## Label Justifications

- `P2`: This is a real provider/tool-schema bug with repeated cron failures, but the affected surface is limited and has a known workaround by switching models.
- `impact:message-loss`: The issue describes scheduled agent runs failing and delivering generic error output instead of the expected cron result.
- `impact:auth-provider`: The failure is tied to provider/model-specific Gemini tool-schema handling and fallback behavior.

## Summary

Keep open. Current main and the latest release still omit `if`, `then`, and `else` from the Gemini schema cleaner, so the source-level failure path remains; the focused fix is still represented by the open PR at https://github.com/openclaw/openclaw/pull/70497.

## What This Changes

The issue asks OpenClaw to investigate and fix Gemini Flash rejecting tool-heavy cron schemas, with follow-up guidance around model choice and cron failure notifications.

## Best Possible Solution

Land a refreshed, focused version of https://github.com/openclaw/openclaw/pull/70497 with the Gemini sanitizer change and regression coverage, then handle any broader cron production guidance separately.

## Reproduction Assessment

Yes for a source-level reproduction: pass a schema containing `if`, `then`, or `else` through `cleanSchemaForGemini` or `normalizeToolParameterSchema({ modelProvider: "google" })` and those keys remain on current main. I did not live-replay the provider HTTP 400.

## Solution Assessment

Yes. The narrow maintainable fix is to strip the three conditional JSON Schema keywords at the Gemini provider-boundary sanitizer; docs-only or failover-only changes would still allow invalid tool schemas through.
