## Label Justifications

- `P1`: The report describes real scheduled agent workflows failing repeatedly for tool-heavy Gemini cron runs, with a source-level root cause still present on main.
- `impact:message-loss`: Affected cron jobs lose the intended generated output and surface an error reply instead.
- `impact:auth-provider`: The failure is tied to provider/model-specific tool schema handling for Google Gemini models.

## Summary

Keep open. Current main and the latest release still omit `if`, `then`, and `else` from the Gemini schema sanitizer, while the focused fix remains in an open, conflicting PR rather than a merged implementation.

## What This Changes

The issue reports Gemini 2.5 Flash rejecting tool-heavy cron runs because conditional JSON Schema keywords can pass through OpenClaw’s Gemini-bound tool schema cleanup.

## Best Possible Solution

Land a refreshed, focused version of https://github.com/openclaw/openclaw/pull/70497 that only strips Gemini conditional schema keywords and adds regression coverage, then handle broader cron guidance separately if maintainers still want it.

## Reproduction Assessment

Yes for a source-level path: pass a schema containing `if`, `then`, or `else` through `cleanSchemaForGemini` or `normalizeToolParameterSchema({ modelProvider: "google" })` on current main and those keys are not stripped. I did not live-replay the Google HTTP 400.

## Solution Assessment

Yes, the narrowest maintainable fix is to strip the three conditional JSON Schema keywords at the Gemini provider-boundary sanitizer. Docs-only or failover-only changes would not stop invalid tool schemas from reaching Gemini.
