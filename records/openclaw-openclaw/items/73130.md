## Summary

Keep open. The hardening direction is plausible, but the latest PR head still changes security-sensitive tool approval flow in ways that bypass downstream policy checks, ignores report-only approval semantics, and lacks real behavior proof.

## What This Changes

The PR adds a credential-name blast-radius classifier plus a before_tool_call guard, tests, docs, and changelog text for blocking or approval-routing destructive tool calls.

## Best Possible Solution

Land a maintainer/security-reviewed guard integrated into the existing trusted-policy and before_tool_call stack while preserving report mode, veto metadata, downstream policy execution, and aligned docs/tests.

## Reproduction Assessment

Yes. Source inspection gives high-confidence PR-defect paths: approve a require-approval guard match and downstream policies are skipped, invoke without confirm and the guard opens approval instead of reporting, or hit a block-severity match and wrappers treat it as a tool error because kind is missing.

## Solution Assessment

No. The security-hardening goal is reasonable, but this patch is not the best form until it composes with existing approvalMode, veto, and downstream hook contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Continue through downstream policies after guard approval:** `src/agents/pi-tools.before-tool-call.ts:621-623`
  - body: When a require-approval guard match is allowed, this returns the guard outcome before trusted policies and plugin before_tool_call hooks run. That lets one guard approval skip existing downstream controls that could still block or adjust the same tool call.
  - confidence: 0.9
- **[P2] Mark guard hard blocks as vetoes:** `src/agents/pi-tools.before-tool-call.ts:420-423`
  - body: Block-severity matches return only blocked and reason. Current tool wrappers throw blocked outcomes whose kind is not "veto", so a hard guard block such as recursive root deletion becomes a tool execution error instead of the structured blocked result used by policy vetoes.
  - confidence: 0.88
- **[P2] Honor report-only approval mode:** `src/agents/pi-tools.before-tool-call.ts:426-443`
  - body: The guard always opens plugin.approval.request for require-approval matches. tools.invoke uses report mode unless the caller confirms, so a dry or noninteractive approval check would now wait on an approval request instead of returning the existing typed requires_approval refusal.
  - confidence: 0.86
- **[P3] Align credential gating docs with runtime:** `docs/tools/exec-approvals.md:189`
  - body: This line says the protection applies when high-blast-radius credentials are present, but the runtime uses credential inventory only as metadata and applies the guard whenever a destructive-action pattern matches. Either gate the guard on inventory or document the global behavior.
  - confidence: 0.78
