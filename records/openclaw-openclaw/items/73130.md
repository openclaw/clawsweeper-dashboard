## Summary

Keep open. Current main still lacks the credential blast-radius guard, so this PR is relevant, but the proposed patch is not merge-ready because the new destructive-command matcher has a concrete split-flag `rm` bypass and the new global preflight conflicts with documented exec-approval expectations unless docs/config are updated.

## What This Changes

The PR adds a names-only credential blast-radius classifier, focused tests, an audit note, and a before-tool-call guard that blocks or routes selected destructive tool calls through plugin approval metadata.

## Best Possible Solution

Land a maintainer-reviewed version that tokenizes or parses `rm` arguments enough to require a real recursive flag before protected root/home targets, adds split-flag regression coverage, handles malformed approval decisions accurately, updates exec-policy docs and changelog for the new runtime behavior, and keeps PR-local audit artifacts out of the source tree unless maintainers explicitly want that archive path.

## Reproduction Assessment

Yes. The main blocker reproduces by evaluating the proposed `shell/rm-rf-root` regex against `rm -r -f /`, `rm -v -r /`, or `rm -r -v /*`, which return false even though they recursively target root or root globs.

## Solution Assessment

No. The hardening direction is reasonable, but this branch is not the best mergeable solution until the destructive-command matcher covers common split-flag forms and the new global preflight is reconciled with the existing exec-approval policy/docs model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Catch split rm flags before root targets:** `src/agents/credential-blast-radius-classifier.ts:330`
  - body: The new root/home deletion guard misses common catastrophic commands when a non-recursive flag is split from the recursive flag before the target. For example, `rm -r -f /`, `rm -v -r /`, and `rm -r -v /*` do not match this pattern, so the guard allows exactly the class of root wipes it is meant to block. Parse the `rm` arguments or allow arbitrary flag groups while separately requiring at least one recursive flag before the protected target.
  - confidence: 0.91
- **[P3] Treat undefined approval decisions as unavailable:** `src/agents/pi-tools.before-tool-call.ts:452-453`
  - body: If the approval request response includes a `decision` property with value `undefined`, this path skips the no-route branch and later reports a timeout. That is still fail-closed, but it gives the wrong operator/debug signal for an unavailable or malformed approval route; handle `null` and `undefined` together here.
  - confidence: 0.8
- **[P3] Align docs with the new exec preflight:** `src/agents/pi-tools.before-tool-call.ts:613-619`
  - body: This hook now runs a heuristic destructive-action guard before the existing trusted/plugin hook pipeline, but the current exec approvals docs say YOLO mode has no separate heuristic command-obfuscation or script-preflight gate. Update the exec-policy docs and changelog, or make the guard policy-configured, so operator expectations match runtime behavior.
  - confidence: 0.78
