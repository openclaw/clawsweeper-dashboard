## Summary

Keep open. The hardening direction is useful, but this PR is not merge-ready because the proposed destructive-command matcher misses common recursive root-delete forms, the new preflight can create approval side effects during report-mode tool probes, and the core exec-policy behavior needs maintainer/security-owner review before landing.

## What This Changes

The PR adds a core credential-name inventory and destructive-action classifier, wires it into `runBeforeToolCallHook`, adds focused tests, and includes a PR-local audit note.

## Best Possible Solution

Land a maintainer-reviewed version that parses `rm` arguments instead of relying on this regex, preserves report-mode semantics, updates exec-policy docs and changelog, and keeps audit artifacts out of the repository.

## Reproduction Assessment

Yes. The main defect is reproducible by evaluating the PR matcher against split-flag recursive root deletion commands such as `rm -r -f /`, `rm -v -r /`, `rm -r -v /*`, and `rm -f -R /`.

## Solution Assessment

No. The direction is reasonable, but the current implementation is not the best fix because the matcher is bypassable and the preflight is not integrated with the existing approval-mode contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Catch split rm flags before root targets:** `src/agents/credential-blast-radius-classifier.ts:330-331`
  - body: The root/home deletion guard misses common catastrophic commands when a non-recursive flag is split from the recursive flag before the target. `rm -r -f /`, `rm -v -r /`, `rm -r -v /*`, and `rm -f -R /` do not match this pattern, so the guard allows the class of root wipes it is meant to block. Parse the `rm` arguments or allow arbitrary flag groups while separately requiring at least one real recursive flag before the protected target.
  - confidence: 0.92
- **[P2] Honor report mode before requesting approvals:** `src/agents/pi-tools.before-tool-call.ts:610-616`
  - body: `tools.invoke` uses `approvalMode: "report"` when the caller has not confirmed execution, and the current hook path reports approval-required work without opening a gateway approval. This new preflight runs before those branches and does not receive `approvalMode`, so a report-only destructive probe can create and wait on a real approval request. Thread `approvalMode` through this guard or return an approval-required report without side effects.
  - confidence: 0.87
- **[P3] Treat undefined approval decisions as unavailable:** `src/agents/pi-tools.before-tool-call.ts:450-453`
  - body: If `plugin.approval.request` returns an object with a `decision` property set to `undefined`, this path skips the no-route branch and later reports a timeout. That remains fail-closed, but it gives the wrong operator signal for a malformed or unavailable approval route; handle `null` and `undefined` together here.
  - confidence: 0.78
- **[P3] Document the new exec preflight:** `src/agents/pi-tools.before-tool-call.ts:610-616`
  - body: This hook now adds a heuristic destructive-action guard before the existing hook pipeline, but the current exec approvals docs say YOLO mode has no separate heuristic command-obfuscation or script-preflight gate. Update the exec-policy docs and changelog, or make the guard policy-configured, so operator expectations match runtime behavior.
  - confidence: 0.8
- **[P3] Remove the PR-local audit artifact:** `shared/projects/system/audit/2026-04-27-1845-credential-blast-radius-quoted-home-root-revision.md:4`
  - body: The added audit note is a PR execution artifact with a machine-specific worktree reference and stale draft-status text. Keep that proof in the PR discussion or an external artifact store instead of adding a new root `shared/projects/system/audit` tree to the product repository.
  - confidence: 0.74
