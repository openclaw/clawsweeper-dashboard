## Summary

Keep open. The latest PR head addresses the earlier split-flag and undefined-decision review items, but it still regresses current blocked-result and report-only approval semantics in a core exec-policy path.

## What This Changes

The PR adds a credential blast-radius classifier, before_tool_call guard, tests, docs, and changelog entry for blocking or approval-routing destructive tool calls with names-only credential metadata.

## Best Possible Solution

Land a rebased version that preserves approvalMode report behavior, returns hard guard blocks as structured vetoes, and aligns credential-triggering docs, tests, and runtime scope after owner review.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a block-severity match returns a non-veto blocked outcome, and current wrappers throw those; a report-only tools.invoke call would also enter the new approval request path instead of returning requiresApproval.

## Solution Assessment

No. The hardening direction is plausible, but the current patch is not the narrowest maintainable solution until it integrates with existing HookOutcome and approvalMode contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Return hard guard blocks as vetoes:** `src/agents/pi-tools.before-tool-call.ts:420-423`
  - body: Block-severity matches return `{ blocked: true, reason }` without `kind: "veto"`. Existing tool wrappers throw any blocked outcome that is not a veto, so a hard guard match like `rm -rf /` becomes a tool execution error instead of the structured blocked result and diagnostics used by other before_tool_call blocks.
  - confidence: 0.88
- **[P2] Respect report-only approval mode:** `src/agents/pi-tools.before-tool-call.ts:427-443`
  - body: The new guard calls `plugin.approval.request` whenever a require-approval destructive action is seen, but current `tools.invoke` uses `approvalMode: "report"` unless the caller confirms. A report-only DROP TABLE check would open and wait for an approval request instead of returning the typed requiresApproval response.
  - confidence: 0.86
- **[P3] Align credential gating with the docs:** `src/agents/pi-tools.before-tool-call.ts:412`
  - body: The docs say the guard applies when high-blast-radius credentials are present, but the runtime builds credential metadata and proceeds even when no high or critical credential exists. Gate on the classified inventory or make the unconditional destructive-action guard explicit in docs and tests.
  - confidence: 0.78
