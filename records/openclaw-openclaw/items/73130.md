## Summary

Keep open. Current main does not implement this guard, and the PR is useful security-hardening work, but the current patch is not merge-ready because the new destructive-command matcher has a concrete split-flag bypass and the new preflight changes exec-policy behavior without matching docs.

## What This Changes

The PR adds a credential blast-radius classifier, focused tests, a PR-local audit note, and a `runBeforeToolCallHook` preflight that blocks or approval-routes destructive tool calls with names-only credential metadata.

## Best Possible Solution

Land a maintainer-reviewed version that tokenizes or otherwise parses `rm` arguments enough to require a real recursive flag before protected root/home targets, adds split-flag regression coverage, treats malformed approval decisions accurately, and updates exec-policy docs plus changelog for the new runtime behavior.

## Reproduction Assessment

Yes. The main defect is reproducible by checking the PR regex against `rm -r -f /`, `rm -v -r /`, or `rm -r -v /*`; those commands are catastrophic recursive root/root-glob deletions but do not match the proposed `shell/rm-rf-root` pattern.

## Solution Assessment

No. The hardening direction is reasonable, but the current regex and docs integration are not the narrowest maintainable solution; a small shell-argument parser or tokenized matcher plus policy/docs alignment would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Catch split rm flags before root targets:** `src/agents/credential-blast-radius-classifier.ts:330`
  - body: The root/home deletion guard misses common catastrophic commands when a non-recursive flag is split from the recursive flag before the target. For example, `rm -r -f /`, `rm -v -r /`, and `rm -r -v /*` do not match this pattern, so the guard allows exactly the class of root wipes it is meant to block. Parse the `rm` arguments or allow arbitrary flag groups while separately requiring at least one recursive flag before the protected target.
  - confidence: 0.91
- **[P3] Treat undefined approval decisions as unavailable:** `src/agents/pi-tools.before-tool-call.ts:450-453`
  - body: If the approval request response includes a `decision` property with value `undefined`, this path skips the no-route branch and later reports a timeout. That is still fail-closed, but it gives the wrong operator/debug signal for an unavailable or malformed approval route; handle `null` and `undefined` together here.
  - confidence: 0.8
- **[P3] Document the new exec preflight:** `src/agents/pi-tools.before-tool-call.ts:613-619`
  - body: This hook now runs a heuristic destructive-action guard before the existing trusted/plugin hook pipeline, but the current exec approvals docs say YOLO mode has no separate heuristic command-obfuscation or script-preflight gate. Update the exec-policy docs and changelog, or make the guard policy-configured, so operator expectations match runtime behavior.
  - confidence: 0.78
