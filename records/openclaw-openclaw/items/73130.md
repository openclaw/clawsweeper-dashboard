## Summary

Keep open. The hardening direction is useful, but this PR is not merge-ready because the new destructive-command matcher has a concrete split-flag bypass, the malformed approval-decision path reports the wrong reason, and the new global exec preflight conflicts with current exec-approval docs unless the docs/policy are updated.

## What This Changes

The PR adds a credential blast-radius classifier, focused tests, an audit note, and a `runBeforeToolCallHook` preflight that blocks or approval-routes matched destructive tool calls with names-only credential metadata.

## Best Possible Solution

Land a maintainer-reviewed version that tokenizes or parses `rm` arguments enough to require a real recursive flag before protected root/home targets, adds split-flag regression coverage, handles malformed approval decisions accurately, and updates exec-policy docs plus changelog for the new runtime behavior.

## Reproduction Assessment

Yes. Without executing any destructive command, the proposed regex can be evaluated directly against command strings; it returns `false` for `rm -r -f /`, `rm -v -r /`, `rm -r -v /*`, and `rm -f -R /`.

## Solution Assessment

No. The hardening goal is reasonable, but a brittle regex is not the safest implementation for command policy; a small tokenized shell-argument matcher plus docs/changelog alignment is a narrower maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Catch split rm flags before protected targets:** `src/agents/credential-blast-radius-classifier.ts:330`
  - body: The root/home deletion guard misses catastrophic recursive deletes when a non-recursive flag appears between the recursive flag and the protected target. For example, `rm -r -f /`, `rm -v -r /`, and `rm -r -v /*` do not match this regex, so the new guard would allow the exact class of root wipes it is meant to block. Parse the `rm` arguments or accept arbitrary flag groups while separately requiring at least one recursive flag before a protected target.
  - confidence: 0.91
- **[P3] Treat undefined approval decisions as unavailable:** `src/agents/pi-tools.before-tool-call.ts:450-453`
  - body: If the approval request response includes a `decision` property with value `undefined`, this branch skips the no-route case and later reports a timeout. That is still fail-closed, but it gives the wrong operator/debug signal for an unavailable or malformed approval route; handle `null` and `undefined` together here.
  - confidence: 0.8
- **[P3] Document the new exec preflight:** `src/agents/pi-tools.before-tool-call.ts:613-619`
  - body: This hook now runs a heuristic destructive-action guard before the existing trusted/plugin hook pipeline, but the current exec-approval docs say YOLO mode has no separate heuristic command-obfuscation or script-preflight gate. Update the exec-policy docs and changelog, or make the guard policy-configured, so operator expectations match runtime behavior.
  - confidence: 0.78
