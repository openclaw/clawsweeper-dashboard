## Summary

Keep open. Current main still lacks the proposed credential blast-radius guard, and the updated PR fixes the earlier split-flag and audit-artifact issues, but it still has blocking integration defects with current before_tool_call behavior.

## What This Changes

The PR adds a credential blast-radius classifier, tests, docs/changelog entries, and a before_tool_call preflight that blocks or approval-routes matched destructive tool calls with names-only credential metadata.

## Best Possible Solution

Land a rebased, maintainer-reviewed version that preserves current veto/report approval semantics, returns structured blocked results for hard guard blocks, and keeps credential-triggering docs, tests, and runtime behavior aligned.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a PR hard-block outcome lacks kind, and current wrappers throw blocked outcomes unless kind is veto; a report-only tools.invoke call would also enter the new approval request path instead of returning requiresApproval.

## Solution Assessment

No. The hardening direction is reasonable, but the current patch is not the best maintainable solution until it integrates with the current approvalMode contract and structured block semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Return hard guard blocks as vetoes:** `src/agents/pi-tools.before-tool-call.ts:420-423`
  - body: Block-severity matches return `{ blocked: true, reason }` without `kind: "veto"`. The existing wrappers treat any blocked outcome that is not a veto as a failure and throw, so a matched hard block like `rm -rf /` surfaces as a tool execution error instead of the structured blocked result and diagnostic used by other before_tool_call blocks.
  - confidence: 0.88
- **[P2] Respect report-only approval mode:** `src/agents/pi-tools.before-tool-call.ts:427-443`
  - body: This new guard calls `plugin.approval.request` whenever it sees a require-approval destructive action, but current `tools.invoke` uses `approvalMode: "report"` unless the caller explicitly confirms. A report-only invocation such as a DROP TABLE check would open and wait for an approval request instead of returning `requiresApproval`, regressing the noninteractive SDK/HTTP path.
  - confidence: 0.84
- **[P3] Align credential gating with the docs:** `src/agents/pi-tools.before-tool-call.ts:411-412`
  - body: The docs added in this PR say credential blast-radius protection applies when high-blast-radius credentials are present, but the runtime only builds metadata and never checks whether any high or critical credential exists before blocking or routing approvals. Either gate on the classified inventory or update the docs/tests to make the global behavior explicit.
  - confidence: 0.78
