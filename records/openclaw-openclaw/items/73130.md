## Summary

Keep open. Current main does not implement this guard, and the latest PR is still not safe to merge because approved guard prompts short-circuit existing trusted policy and hook controls while hard-block and report-mode outcomes do not preserve current contracts.

## What This Changes

The PR adds a credential blast-radius classifier, docs/changelog entries, and a before_tool_call guard that blocks or approval-routes matched destructive tool calls with names-only credential metadata.

## Best Possible Solution

Land a maintainer-reviewed version that layers the guard into the existing policy stack, preserves report-mode and veto semantics, and makes the credential-present versus global-guard scope explicit in code, tests, and docs.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: approve a matched DROP/TRUNCATE operation and it returns before trusted policies/hooks, call tools.invoke without confirm for the same operation and it opens an approval request, or run a block-severity match through a wrapper and it is thrown as a tool error because kind is missing.

## Solution Assessment

No. The hardening direction is reasonable, but this implementation is not the narrowest maintainable fix until it composes with approvalMode, veto, and downstream hook contracts already present on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Continue through downstream policies after guard approval:** `src/agents/pi-tools.before-tool-call.ts:621-623`
  - body: When a require-approval guard match is allowed, applyCredentialBlastRadiusGuard returns { blocked: false } and runBeforeToolCallHook immediately returns it. That skips runTrustedToolPolicies and plugin before_tool_call hooks, so approving one guard prompt can bypass existing policy adjustments or blocks for the same tool call.
  - confidence: 0.9
- **[P2] Mark guard hard blocks as vetoes:** `src/agents/pi-tools.before-tool-call.ts:420-423`
  - body: Block-severity matches currently return only blocked and reason. Existing tool wrappers throw any blocked outcome whose kind is not "veto", so a hard guard block like recursive root deletion becomes a tool execution error instead of the structured blocked result and diagnostics used by other policy vetoes.
  - confidence: 0.88
- **[P2] Honor report-only approval mode:** `src/agents/pi-tools.before-tool-call.ts:426-443`
  - body: This path always calls plugin.approval.request for require-approval matches. tools.invoke uses approvalMode: "report" unless the caller confirms, so a noninteractive approval check would now open and wait on an approval request instead of returning the existing typed requiresApproval refusal.
  - confidence: 0.86
- **[P3] Align the credential condition with behavior:** `src/agents/pi-tools.before-tool-call.ts:412`
  - body: The added docs and PR body say this protection applies when high-blast-radius credentials are present, but the runtime only inventories credentials for metadata and applies the guard even when none are present. Gate the guard on the classified inventory or update docs/tests to state the behavior is global.
  - confidence: 0.78
