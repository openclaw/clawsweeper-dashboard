## Summary

Keep open. Current main still permits the reported loopback shared-token backend self-pairing exception, latest release v2026.4.29 contains the same behavior, and the linked closing PR #72419 remains open/unmerged rather than shipped evidence.

## What This Changes

The issue reports that loopback callers with a shared gateway token can self-declare the internal backend gateway-client identity and skip device pairing.

## Best Possible Solution

Keep this issue open until maintainers either land PR #72419 or a replacement that narrows backend identity to a server-verifiable/dedicated credential boundary, or explicitly accept the current loopback shared-token backend exception with matching docs and regression tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare gateway-client/backend, connect from loopback without an Origin header, authenticate with a valid shared token/password, and shouldSkipLocalBackendSelfPairing returns true; the message handler then preserves the skip path through scope clearing and not-paired checks.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity check looks safer than trusting client-declared id/mode, but maintainers should choose the final contract because current docs intentionally reserve this internal backend exception.
