## Summary

Keep open. Current main still permits the reported no-Origin loopback `gateway-client`/`backend` shared-token or password pairing skip, v2026.5.7 shipped the same behavior, and the linked fix PR at https://github.com/openclaw/openclaw/pull/72419 is closed unmerged. Current docs and tests preserve this trust boundary, so this needs secops/product judgment rather than conservative cleanup closure.

## What This Changes

The issue reports that loopback clients can self-declare the backend gateway client identity and use shared token/password auth to skip device pairing.

## Best Possible Solution

Have secops choose the backend identity contract, then update the helper, message-handler policy, tests, and gateway security docs so shared-token/password, device-token, bootstrap-token, auth-none, direct-local, container-local, browser-origin, and remote behavior are all explicitly covered.

## Reproduction Assessment

Yes. Source and current tests give a high-confidence path: send `gateway-client`/`backend` connect params from no-Origin loopback with `sharedAuthOk: true` and `authMethod: "token"` or `"password"`; the helper returns true and the missing-device policy allows the operator session.

## Solution Assessment

Unclear. Removing shared token/password from the backend skip is plausible, but the closed fix PR also introduced a bootstrap-token bypass concern and current docs reserve the shared-token exception, so maintainers/secops need to choose the contract before implementation.
