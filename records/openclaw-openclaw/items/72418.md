## Summary

Keep open. Current main still preserves the reported loopback `gateway-client`/`backend` shared-token/password pairing skip, latest release v2026.5.7 shipped the same central behavior, and the paired fix PR is closed unmerged after security-contract concerns.

## What This Changes

The issue requests narrowing the gateway backend self-pairing exception so loopback clients cannot self-declare `gateway-client`/`backend` and use shared token/password auth to skip device pairing.

## Best Possible Solution

Have secops choose the backend identity and credential contract, then update the helper, message-handler policy, regression tests, and gateway security docs consistently.

## Reproduction Assessment

Yes. Source and current tests give a high-confidence path: send `gateway-client`/`backend` connect params from no-Origin loopback with `sharedAuthOk: true` and `authMethod: "token"` or `"password"`; the helper returns true and the missing-device policy allows the operator session.

## Solution Assessment

Unclear. Removing shared token/password from the bypass may be right, but current docs reserve the exception and the closed linked PR added a bootstrap-token bypass concern, so secops/product should set the trust contract first.
