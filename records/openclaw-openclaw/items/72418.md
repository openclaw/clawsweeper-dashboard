## Summary

Keep open. Current main still preserves the reported shared-token loopback backend self-pairing exception, the latest release tag contains the same helper/test behavior, and the linked PR #72419 remains open and unmerged, so this issue should not be closed before maintainer security/product review resolves the active implementation path.

## What This Changes

The issue requests hardening the local backend self-pairing path so a loopback caller with the shared gateway token cannot self-declare `gateway-client`/`backend` identity to skip device pairing and preserve elevated scopes.

## Best Possible Solution

Keep this issue open until maintainers either land PR #72419 or a replacement that narrows backend identity, or explicitly accept the current internal backend exception with matching docs and regression coverage.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare `gateway-client`/`backend`, connect from loopback without an Origin header, authenticate with a valid token/password, and `shouldSkipLocalBackendSelfPairing` returns true while compat tests show scopes are preserved.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity looks safer than relying on client-declared `id`/`mode`, but the current docs intentionally reserve this internal exception and the open PR needs maintainer security/product review.
