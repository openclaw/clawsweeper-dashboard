## Summary

Keep open. Current main and the latest release still preserve the reported loopback backend self-pairing exception, and the linked closing PR is still open/unmerged rather than shipped evidence.

## What This Changes

The issue requests tightening gateway WebSocket auth so loopback clients cannot self-declare the gateway-client/backend identity with a shared token/password and bypass device pairing or scope binding.

## Best Possible Solution

Keep this issue open until maintainers either land PR #72419 or a replacement that narrows backend identity to a server-verifiable or dedicated credential boundary, or explicitly accept the current exception with matching docs and regression tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare gateway-client/backend, connect from loopback without an Origin header, authenticate with a valid shared token/password, and shouldSkipLocalBackendSelfPairing returns true.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity check looks safer than trusting client-declared id/mode, but the final contract needs maintainer secops/product review because current docs intentionally reserve this internal backend exception.
