## What This Changes

The issue asks Gateway to stop allowing loopback backend clients authenticated only with shared token/password to self-declare gateway-client backend identity and bypass device pairing/scope clearing.

## Best Possible Solution

Review and land the active fix PR after Gateway/security acceptance of the shared-token compatibility break, or explicitly document the trusted-loopback policy and replace or close that fix path.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.10 still return true for local backend token/password auth when sharedAuthOk is true, and the caller preserves scopes when that exception applies.

## Solution Assessment

Yes, conditionally. Removing shared token/password from the backend self-pairing exception is the narrowest code path if maintainers accept the compatibility break; otherwise the best solution is an explicit trusted-loopback policy decision.
