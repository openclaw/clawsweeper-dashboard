## Summary

Keep open. Current main still permits the reported no-Origin loopback `gateway-client`/`backend` shared-token pairing skip, the tests and docs preserve that behavior, and the linked closing PR #72419 is still open and unmerged.

## What This Changes

The issue asks the gateway to stop allowing no-Origin loopback clients that self-declare the backend gateway identity to skip device pairing with shared token/password auth.

## Best Possible Solution

Land a security-reviewed gateway auth change that binds backend self-pairing to a server-verifiable backend credential or identity, or explicitly accept and document the current single-operator trust model with matching tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence: use `gateway-client`/`backend` connect params, direct loopback or shared-secret-loopback locality, no browser Origin, `sharedAuthOk: true`, and `authMethod: "token"` or `"password"`; current tests assert that path returns true.

## Solution Assessment

Unclear. The linked PR’s dedicated bootstrap/device credential boundary is a plausible narrow fix, but maintainers/secops should choose the intended contract because current security docs reserve the shared-token loopback backend exception.
