## Summary

Keep open. Current main still permits the reported no-Origin loopback `gateway-client`/`backend` shared-token pairing skip, tests and docs preserve that behavior, and linked PR #72419 is open and unmerged.

## What This Changes

The issue asks the gateway to stop allowing no-Origin loopback clients that self-declare the backend gateway identity to skip device pairing with shared token/password auth.

## Best Possible Solution

Review and land PR #72419 or an equivalent secops-approved gateway auth change that binds backend self-pairing to a dedicated credential or server-verifiable backend identity, with docs and regression tests aligned to the chosen trust model.

## Reproduction Assessment

Yes. Source inspection and current helper tests give a high-confidence path: use `gateway-client`/`backend` connect params, no Origin, loopback locality, `sharedAuthOk: true`, and `authMethod: "token"` or `"password"`; the helper returns true and the message handler preserves the skip.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable identity is a plausible narrow fix, but maintainers/secops need to choose the intended contract because current security docs reserve this shared-token loopback backend exception.
