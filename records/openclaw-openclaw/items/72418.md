## Summary

Keep open. Current main still permits the reported no-Origin loopback gateway-client/backend self-pairing skip with shared token/password auth, the checked-in helper test expects that behavior, and the linked closing PR remains open and unmerged, so this is still a security-sensitive gateway auth trust-boundary decision for maintainers/secops.

## What This Changes

Harden gateway WebSocket backend self-pairing so loopback shared-token callers cannot skip device pairing solely by declaring gateway-client/backend identity.

## Best Possible Solution

Keep the issue open until maintainers land a secops-reviewed change that narrows the backend self-pairing exception, or explicitly accept and test the current loopback shared-token trust model.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare gateway-client/backend, connect over loopback without an Origin header, authenticate with a valid shared token/password, and shouldSkipLocalBackendSelfPairing returns true while message-handler preserves the skip through scope clearing and pairing checks.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity appears safer than trusting client-declared id/mode, but maintainers should choose the final contract because current docs reserve an internal backend exception.
