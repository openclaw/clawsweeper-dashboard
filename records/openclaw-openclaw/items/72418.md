## Summary

Keep open. Current main and latest release still allow the reported shared-token/password loopback backend self-pairing skip, while docs and tests describe that exception as intentional, so this needs a security/product boundary decision rather than cleanup closure.

## What This Changes

Narrow backend self-pairing so loopback clients cannot skip device pairing solely by self-declaring gateway-client/backend identity with shared token/password auth.

## Best Possible Solution

Have secops define the backend identity and credential contract, then update the helper, missing-device policy, regression tests, and gateway security docs to match that decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send gateway-client/backend connect params from no-Origin loopback or shared-secret loopback locality with sharedAuthOk true and authMethod token/password, and the helper returns true before the missing-device policy allows the operator session.

## Solution Assessment

Unclear. Removing shared token/password from the bypass may be the right hardening direction, but current docs reserve the exception and the closed related PR exposed bootstrap-token/container-local contract concerns that need secops/product review first.
