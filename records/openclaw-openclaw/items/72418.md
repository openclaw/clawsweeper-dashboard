## Summary

Keep open. Current main and the latest release still preserve the reported backend self-pairing exception, and the linked closing PR is open/unmerged. Because this is a gateway authentication trust-boundary question with an intentional docs carveout for internal backend clients, it should stay with maintainers/secops rather than be closed by cleanup.

## What This Changes

The issue reports that a loopback, no-Origin client with a valid shared gateway token can self-declare the gateway-client/backend identity and skip device pairing and scope binding.

## Best Possible Solution

Keep this issue open until maintainers land a security-reviewed change that binds backend self-pairing to a server-verifiable backend identity or dedicated backend credential, or explicitly reaffirm the current loopback shared-token backend exception with matching docs and regression tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare gateway-client/backend, connect from loopback without an Origin header, authenticate with a valid shared token/password, and shouldSkipLocalBackendSelfPairing returns true; the message handler then preserves that skip through scope clearing and not-paired checks.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity appears safer than trusting client-declared id/mode, but maintainers/secops should choose the final contract because current docs reserve an internal backend exception.
