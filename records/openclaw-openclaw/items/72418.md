## Summary

Keep open: current main and the latest release still preserve the reported shared token/password backend self-pairing path, and the open linked fix PR is the right place to resolve the compatibility-sensitive gateway auth decision.

## What This Changes

Prevent loopback backend clients from using shared gateway token/password auth plus self-declared gateway-client/backend identity to skip device pairing and retain privileged scopes.

## Best Possible Solution

Review and either land the linked fix PR after explicit gateway/security acceptance of the shared-token compatibility break, or close/supersede it with a documented trust-model decision.

## Reproduction Assessment

Yes for source reproduction: current main's helper still returns true for local backend token/password auth when sharedAuthOk is true, and the message handler preserves scopes when that exception applies. I did not run a live WebSocket probe in this read-only review.

## Solution Assessment

Yes, conditionally: removing shared token/password from the backend self-pairing exception is the narrow fix if maintainers accept the compatibility break. If the existing loopback trusted-operator model is intentional, the better solution is to document that decision and close or replace the PR.
