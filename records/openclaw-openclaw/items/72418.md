## Summary

Keep open. Current main still preserves the reported shared-token/password local backend self-pairing skip, current tests and gateway security docs describe that behavior as intentional, and the paired fix PR is closed unmerged after security-contract concerns.

## What This Changes

The issue reports that no-Origin loopback clients can self-declare the backend gateway client identity and use shared gateway token or password authentication to skip device pairing.

## Best Possible Solution

Have secops choose the backend identity and credential contract, then update the helper, missing-device policy, regression tests, and gateway security docs consistently.

## Reproduction Assessment

Yes. Source and current tests give a high-confidence path: send `gateway-client`/`backend` connect params from no-Origin loopback with `sharedAuthOk: true` and `authMethod: "token"` or `"password"`; the helper returns true and the missing-device policy allows the operator session.

## Solution Assessment

Unclear. Removing shared token/password from the backend skip may be the right direction, but current docs reserve the exception and the closed fix PR added a bootstrap-token bypass concern, so secops should choose the contract first.
