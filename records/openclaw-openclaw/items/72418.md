## Summary

Keep open. Current main and v2026.4.29 still allow the reported loopback gateway-client/backend shared-token pairing skip, and the linked closing PR #72419 is open/unmerged, so this remains a security-sensitive gateway auth trust-boundary decision for maintainers/secops.

## What This Changes

The issue reports that loopback clients can self-declare the internal gateway backend identity and skip device pairing when authenticating with a shared gateway token or password.

## Best Possible Solution

Keep this issue open until maintainers land a security-reviewed change that binds backend self-pairing to server-verifiable backend identity or a dedicated backend credential, or explicitly accept and test the current documented loopback shared-token exception.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare gateway-client/backend, connect over loopback without an Origin header, authenticate with a valid shared token/password, and shouldSkipLocalBackendSelfPairing returns true; the message handler then preserves the skip through scope clearing and not-paired checks.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity appears safer than trusting client-declared id/mode, but maintainers should choose the final contract because the current docs reserve an internal backend exception.
