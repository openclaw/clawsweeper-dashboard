## Summary

Keep open. Current main still permits the reported no-Origin loopback `gateway-client`/`backend` shared-token or password pairing skip, v2026.5.7 shipped the same behavior, and the linked fix PR is closed unmerged with unresolved security-contract concerns.

## What This Changes

The issue asks OpenClaw to stop no-Origin loopback clients from self-declaring `gateway-client`/`backend` and using shared token/password auth to skip device pairing.

## Best Possible Solution

Have secops choose the backend identity and credential contract, then update the helper, message-handler policy, regression tests, and gateway security docs consistently.

## Reproduction Assessment

Yes. Source and current tests give a high-confidence path: send `gateway-client`/`backend` connect params from no-Origin loopback with `sharedAuthOk: true` and `authMethod: "token"` or `"password"`; the helper returns true and the missing-device policy allows the operator session.

## Solution Assessment

Unclear. Removing shared token/password from the backend skip is plausible, but current docs reserve the exception and the closed fix PR added a bootstrap-token bypass concern, so secops needs to choose the contract first.
