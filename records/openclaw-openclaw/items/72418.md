- `P1`: This is a source-reproducible gateway auth and device-pairing boundary issue affecting privileged local backend sessions.

- `impact:auth-provider`: The affected behavior depends on gateway token/password, bootstrap-token, and device-token authentication method handling.

## Summary

Keep open. Current main and the latest shipped tag still allow the reported no-Origin loopback/shared-secret backend self-pairing path, and the behavior is documented and tested as intentional, so this needs a gateway security/product boundary decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop allowing loopback clients authenticated with the shared gateway token or password to self-declare the backend client identity and skip device pairing.

## Best Possible Solution

Have the gateway security owners define the backend credential contract, then update the helper, missing-device policy, regression tests, gateway security docs, and any container-local troubleshooting guidance to match that decision.

## Reproduction Assessment

Yes. Source inspection shows the reported path: a no-Origin direct_local or shared_secret_loopback_local client that declares gateway-client/backend and authenticates with token/password sets localBackendSelfPairingOk before missing-device policy allows the operator session.

## Solution Assessment

Unclear. Removing shared token/password from the bypass is a plausible hardening path, but the current docs and tests intentionally reserve that exception, so the best fix needs a gateway security contract decision first.
