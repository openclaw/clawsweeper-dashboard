## Summary

Keep open. Current main still permits the reported no-Origin loopback `gateway-client`/`backend` shared-token pairing skip, the behavior is covered by checked-in tests and docs, and the linked closing PR #72419 remains open/unmerged.

## What This Changes

The issue requests hardening gateway backend self-pairing so loopback clients cannot skip device pairing solely by self-declaring `gateway-client`/`backend` identity with shared-token auth.

## Best Possible Solution

Resolve this through a security-reviewed gateway auth change that binds backend self-pairing to server-verifiable backend identity or a dedicated backend credential, or explicitly document and test the accepted loopback shared-token exception.

## Reproduction Assessment

Yes. The helper-level path is high-confidence from current source and tests: declare `gateway-client`/`backend`, connect over loopback without an Origin header, authenticate with the shared token/password, and `shouldSkipLocalBackendSelfPairing` returns true while the message-handler skip preserves the pairing bypass path.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity appears safer than trusting client-declared id/mode, but maintainers/secops need to choose the intended contract because the current docs reserve an internal backend exception.
