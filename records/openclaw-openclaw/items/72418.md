## Summary

Keep open. Current main and the latest release still allow the reported helper path, the message handler still uses that skip to avoid scope clearing and not-paired approval, and the linked closing PR remains open/unmerged rather than shipped evidence. Because this is a gateway authentication trust-boundary question that docs currently describe as an intentional internal backend exception, it needs maintainer/secops review rather than cleanup closure.

## What This Changes

The issue reports that loopback callers with a valid shared gateway token can self-declare the backend gateway-client identity and skip device pairing and scope clearing.

## Best Possible Solution

Keep this issue open until maintainers land a security-reviewed change that binds backend self-pairing to a server-verifiable backend identity or dedicated backend credential, or explicitly accept the current loopback shared-token backend exception with matching docs and regression tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare gateway-client/backend, connect from loopback without an Origin header, authenticate with a valid shared token/password, and shouldSkipLocalBackendSelfPairing returns true; the message handler then preserves the skip path through scope clearing and not-paired checks.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity check appears safer than trusting client-declared id/mode, but maintainers should choose the final contract because the current docs reserve an internal backend exception.
