## What This Changes

Restrict local backend self-pairing so a loopback client cannot self-declare `gateway-client`/`backend` with shared token/password auth and skip device pairing while preserving privileged scopes.

## Best Possible Solution

Review and either land the linked fix PR after explicit gateway/security acceptance of the compatibility-sensitive auth boundary, or close/supersede it with a documented trust-model decision.

## Reproduction Assessment

Yes for source reproduction: current main's helper still returns true for local backend token/password auth when `sharedAuthOk` is true, and the message handler preserves scopes when that exception applies. I did not run a live WebSocket probe in this read-only review.

## Solution Assessment

Yes, conditionally: the linked PR's direction of removing shared token/password from the backend self-pairing exception is the narrowest current fix if maintainers accept the compatibility break. If maintainers decide shared-secret loopback remains fully trusted by policy, the best solution is to document that decision and close or replace the PR.
