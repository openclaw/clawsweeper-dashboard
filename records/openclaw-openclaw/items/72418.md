## Summary

Keep open. Current main and the latest tagged release still preserve the reported shared-token loopback backend self-pairing exception, and the linked closing PR #72419 is still open/unmerged, so this is not safe to close as implemented or obsolete.

## What This Changes

The issue reports that a loopback client with a valid shared gateway token can self-declare the `gateway-client` backend identity and bypass device pairing.

## Best Possible Solution

Keep this as an explicit gateway security review item until maintainers either land a narrowed backend identity boundary, likely via PR #72419 or a replacement, or consciously accept and document the current loopback shared-token backend exception with matching tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare `gateway-client`/`backend`, connect from loopback without an Origin header, authenticate with a valid token/password, and `shouldSkipLocalBackendSelfPairing` returns true.

## Solution Assessment

Unclear. A dedicated backend credential or server-side identity check may be the narrowest hardening path, but the final contract should be chosen by gateway security maintainers because current docs intentionally reserve an internal backend exception.
