## Summary

Keep open. Current main still allows the reported no-Origin loopback gateway-client/backend shared-token pairing skip, the behavior is preserved by checked-in tests and gateway security docs, and the linked closing PR #72419 is still open/unmerged.

## What This Changes

The issue requests hardening gateway backend self-pairing so loopback callers cannot skip device pairing solely by self-declaring the gateway-client/backend identity with shared-token auth.

## Best Possible Solution

Land or reject a security-reviewed change, likely via #72419 or a maintainer replacement, that binds backend self-pairing to server-verifiable backend identity or a dedicated backend credential, or explicitly accepts and tests the documented loopback shared-token exception.

## Reproduction Assessment

Yes. The helper-level path is high-confidence from current source and tests: declare gateway-client/backend, connect over loopback without an Origin header, authenticate with the shared token/password, and shouldSkipLocalBackendSelfPairing returns true while message-handler skips scope clearing and not-paired approval.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable backend identity appears safer than trusting client-declared id/mode, but the final contract should be chosen by maintainers/secops because the docs currently reserve an internal backend exception.
