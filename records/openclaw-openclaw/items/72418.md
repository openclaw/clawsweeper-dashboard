## Summary

Keep open. Current main still preserves the reported shared-token loopback backend self-pairing exception, the latest release tag contains the same behavior, and the linked closing PR #72419 remains an unmerged implementation candidate rather than shipped evidence.

## What This Changes

The issue reports that a loopback client with a valid shared gateway token/password can self-declare the `gateway-client` backend identity, skip device pairing, and preserve self-declared operator scopes without a registered device token.

## Best Possible Solution

Keep this issue open until maintainers either land a narrowed backend identity boundary, likely through PR #72419 or a replacement, or explicitly accept the current loopback shared-token backend exception with matching docs and regression tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: declare `gateway-client`/`backend`, connect from loopback without an Origin header, authenticate with a valid token/password, and `shouldSkipLocalBackendSelfPairing` returns true; the message handler then preserves scopes and skips pairing on that path.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable identity check looks safer than trusting client-declared `id`/`mode`, but maintainers should choose the final contract because current docs intentionally reserve an internal backend exception.
