## Summary

Keep open. Current main and the latest release still preserve the shared-token/password local backend self-pairing skip, the related fix pull request at https://github.com/openclaw/openclaw/pull/72419 is closed unmerged, and no newer canonical fix supersedes it; this remains a security-sensitive gateway auth trust-boundary decision.

## What This Changes

The issue asks OpenClaw to stop loopback backend clients from self-declaring `gateway-client`/`backend` identity and using shared token/password auth to skip device pairing.

## Best Possible Solution

Have secops choose the gateway backend identity contract, then land a focused change that removes shared token/password from the backend pairing bypass or binds it to a server-verifiable backend credential, with helper tests, message-handler comments, docs, and release notes aligned.

## Reproduction Assessment

Yes. Source inspection and existing tests reproduce the path: `gateway-client`/`backend`, no Origin, local locality, `sharedAuthOk: true`, and `authMethod: "token"` makes the helper return true and lets the missing-device path allow the session.

## Solution Assessment

Unclear. Removing shared token/password from this bypass is plausible, but the exact credential boundary needs secops/product approval because current docs reserve this exception and the closed proposed fix introduced a bootstrap-token bypass concern.
