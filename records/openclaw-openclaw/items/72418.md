## Summary

Keep open. Current main still permits the reported no-Origin loopback gateway-client/backend shared-token pairing skip, checked-in tests and docs preserve that exception, and the closing PR #72419 is still open/unmerged.

## What This Changes

The issue asks OpenClaw to stop loopback clients from self-declaring gateway-client/backend identity with shared token/password auth to skip device pairing.

## Best Possible Solution

A security-reviewed change should either bind backend self-pairing to server-verifiable backend identity or a dedicated backend credential while preserving internal control-plane RPCs, or explicitly accept and test the documented exception.

## Reproduction Assessment

Yes. The helper-level path is high-confidence from current source and tests: declare gateway-client/backend, connect over loopback without an Origin header, authenticate with token/password shared auth, and shouldSkipLocalBackendSelfPairing returns true while message-handler skips scope clearing and not-paired approval.

## Solution Assessment

Unclear. A dedicated backend-only credential or server-verifiable identity appears safer than trusting client-declared id/mode, but the final contract needs maintainer/secops judgment because current docs reserve this internal backend exception.
