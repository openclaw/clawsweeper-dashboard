## Summary

Keep open: current main and the latest release still preserve the reported shared token/password backend self-pairing path, and the open linked fix PR is the active place to resolve the compatibility-sensitive gateway auth decision.

## What This Changes

The issue reports that loopback backend clients can self-declare the gateway backend identity and use shared token or password auth to skip device pairing and retain privileged scopes.

## Best Possible Solution

Review and either land the active fix PR after explicit gateway/security acceptance of the shared-token compatibility break, or document the trusted-loopback decision and close or replace the fix path.

## Reproduction Assessment

Yes, by source inspection: current main's helper still returns true for local backend token/password auth when sharedAuthOk is true, and the message handler preserves scopes when that exception applies. I did not run a live WebSocket probe in this read-only review.

## Solution Assessment

Yes, conditionally: removing shared token/password from the backend self-pairing exception is the narrow fix if maintainers accept the compatibility break. If the documented loopback trusted-operator model is intentional, the better solution is to document that decision and close or replace the active PR.
