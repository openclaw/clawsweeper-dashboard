## Summary

Keep open: current main and the latest release still have a plausible source-level reconnect loop for direct `wss://` macOS node mode because the coordinator creates a fresh TLS-pinning session each pass, and `GatewayNodeSession` treats that new object identity as a required reconnect.

## What This Changes

The issue reports repeated successful macOS node-mode gateway connection logs during an apparently healthy remote direct `wss://` session.

## Best Possible Solution

Keep one stable TLS-pinning session identity for unchanged macOS node-mode `wss://` endpoint parameters, rebuild only when the semantic TLS/session inputs change, and add regression coverage for repeated equivalent direct-`wss://` loop iterations not reconnecting.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: in direct `wss://` mode the coordinator creates a new TLS session box each loop and the node session treats that object identity change as a reconnect trigger. A live macOS remote gateway replay was not run in this read-only review.

## Solution Assessment

No current fix is already present. The best fix is a narrow stable-session or semantic-session-key repair in the macOS node/TLS session boundary, not changing the low-level channel connected guard.

## Repair Work Prompt

Fix the macOS node-mode direct `wss://` reconnect churn reported in https://github.com/openclaw/openclaw/issues/90668. Current source creates a new `GatewayTLSPinningSession` in `MacNodeModeCoordinator.buildSessionBox()` on every run-loop pass, then `GatewayNodeSession.connect()` compares `ObjectIdentifier(sessionBox.session)` and treats each equivalent TLS session as a required reconnect. Implement a narrow repair that keeps session identity stable for unchanged URL/connection mode/TLS fingerprint inputs or otherwise compares a semantic session key, while preserving intentional reconnects when TLS params or a truly different injected session changes. Do not remove the existing low-level `GatewayChannelActor.connect()` guard or weaken TLS pinning/device-token retry behavior. Add focused Swift regression coverage showing repeated equivalent direct-`wss://` macOS node connection attempts do not rebuild/relog, and keep the existing changed-session-box rebuild behavior covered.
