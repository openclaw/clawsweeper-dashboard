## Summary

Keep open. The PR is aimed at a valid Slack capabilities bug, but the current patch still mixes Slack method-accepted scopes into the same list that OpenClaw prints as granted bot/user token scopes, which can make the diagnostic report capabilities the token does not actually have.

## What This Changes

The PR adds Slack `auth.test` as a third scope-discovery fallback, reads Slack response metadata in the scope extractor, adds focused Slack scope tests, and adds a changelog entry for modern granular bot-token capabilities discovery.

## Best Possible Solution

Land the Slack plugin fix after narrowing scope extraction to actual granted scopes: keep the legacy methods first, add `auth.test` as the modern-token fallback, read grants from `response_metadata.scopes`, and update the focused tests and changelog so accepted scopes are not presented as token grants.

## Reproduction Assessment

Yes. The linked bug gives concrete CLI steps, and current main still contains the exact two-method Slack scope lookup that produces the joined `unknown_method` output; the PR tests model the intended modern-token fallback path.

## Solution Assessment

No. The `auth.test` fallback plus `response_metadata.scopes` is the narrow maintainable fix, but `response_metadata.acceptedScopes` should not be merged into the granted-scope list.
