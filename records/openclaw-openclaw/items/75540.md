## Summary

Keep open. The PR fixes a real Slack capabilities bug, but it currently treats Slack method-accepted scopes as token-granted scopes, which can make the diagnostic overstate what a bot or user token actually has.

## What This Changes

The PR adds a Slack `auth.test` scope-discovery fallback, reads Slack response metadata during scope extraction, adds focused Slack scope tests, and updates the changelog.

## Best Possible Solution

Keep the Slack-owned fix in the plugin: preserve the legacy fallback order, add `auth.test` for modern granular bot tokens, use only `response_metadata.scopes` as the granted-scope source, and keep accepted scopes out of `Bot scopes` / `User scopes` unless they are displayed separately with a precise label.

## Reproduction Assessment

Yes. The linked issue has concrete CLI steps, and current main still shows the exact two-method fallback and joined error path; the PR tests also model the modern-token recovery path.

## Solution Assessment

No. The `auth.test` fallback plus `response_metadata.scopes` is the narrow maintainable fix, but including `acceptedScopes` in granted-scope discovery is not correct for this diagnostic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not report accepted scopes as granted scopes:** `extensions/slack/src/scopes.ts:75-77`
  - body: The Slack SDK fills `response_metadata.acceptedScopes` from `x-accepted-oauth-scopes`, which is the set accepted by the Web API method, not the scopes granted to this token. Because this diagnostic prints returned values as `Bot scopes` or `User scopes`, an acceptedScopes-only response can now report method requirements as token grants, and the new test locks that in. Keep the `auth.test` fallback, but only use `response_metadata.scopes` for granted-scope discovery or label accepted scopes separately.
  - confidence: 0.86
