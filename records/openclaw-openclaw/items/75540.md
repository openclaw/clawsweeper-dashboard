## Summary

Keep open. The PR targets a valid Slack capabilities bug and the `auth.test` fallback is the right narrow direction, but the patch should not merge as-is because it treats Slack SDK `response_metadata.acceptedScopes` as granted bot/user scopes even though the SDK fills that field from `x-accepted-oauth-scopes`, which is the method-accepted-scope header rather than the token-grant header.

## What This Changes

The branch adds a Slack `auth.test` scope-discovery fallback, reads Slack SDK response metadata in `extractScopes`, adds focused Slack scope tests, and updates the changelog.

## Best Possible Solution

Keep the Slack-owned fix in the plugin: preserve legacy fallback order, add `auth.test` for modern bot tokens, use `response_metadata.scopes` as the granted-scope source, and either drop `acceptedScopes` from this diagnostic or expose it separately with a label that does not imply token grants.

## Reproduction Assessment

Yes. The linked issue has CLI steps and a contributor reproduced the exact current-main fallback chain; the existing source still shows the two legacy attempts and joined error formatting. The PR's new tests also model the modern-token recovery path.

## Solution Assessment

No, not as currently written. The `auth.test` fallback plus `response_metadata.scopes` is the narrow maintainable fix, but folding `acceptedScopes` into granted-scope discovery can make `channels capabilities` report method requirements as bot/user scopes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not treat accepted scopes as granted Slack scopes:** `extensions/slack/src/scopes.ts:75-77`
  - body: The Slack SDK fills `response_metadata.acceptedScopes` from the `x-accepted-oauth-scopes` header, which describes scopes accepted by the Web API method, not the scopes granted to this token. With this branch, a successful call that lacks `x-oauth-scopes` but includes `x-accepted-oauth-scopes` will make `channels capabilities` report those method requirements as `Bot scopes`, and the new test locks in that false-positive behavior. Keep the `auth.test` fallback, but only use `response_metadata.scopes` for granted-scope discovery or label accepted scopes separately.
  - confidence: 0.86
