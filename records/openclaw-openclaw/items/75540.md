## Summary

Keep open. The PR fixes the right Slack-owned failure path by adding an `auth.test` fallback, but it also treats Slack `acceptedScopes` metadata as granted token scopes, which would make capability diagnostics report method requirements as bot/user grants.

## What This Changes

The PR adds a Slack `auth.test` scope-discovery fallback, reads Slack response metadata in the scope extractor, adds focused Slack scope tests, and updates the changelog for the linked modern bot-token bug.

## Best Possible Solution

Keep the Slack plugin fix, but only use `response_metadata.scopes` as the granted-scope fallback from `auth.test`; remove accepted-scope extraction from capability grants and update the focused tests and changelog wording accordingly.

## Reproduction Assessment

Yes. The linked issue has concrete CLI steps, current main still contains the exact two-method fallback and joined-error path, and the PR's tests model the modern-token recovery scenario.

## Solution Assessment

No. The `auth.test` fallback plus `response_metadata.scopes` is the narrow maintainable fix, but including `acceptedScopes` in the granted-scope list is not correct for this diagnostic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not report accepted scopes as granted scopes:** `extensions/slack/src/scopes.ts:75-77`
  - body: The Slack SDK fills `response_metadata.acceptedScopes` from `x-accepted-oauth-scopes`, which is the set accepted by the Web API method rather than scopes granted to this token. Because capability diagnostics print returned values as `Bot scopes` or `User scopes`, an acceptedScopes-only response can now falsely report method requirements as token grants. Keep the `auth.test` fallback, but use only `response_metadata.scopes` for granted-scope discovery or display accepted scopes under a separate precise label.
  - confidence: 0.86
