## Summary

Keep open. The PR is a useful Slack capability fix direction, but it should not merge as-is because it treats Slack `acceptedScopes` as granted token scopes, which can make capability diagnostics overstate what the bot or user token actually has.

## What This Changes

The PR adds an `auth.test` fallback to Slack scope discovery, extends scope extraction to Slack response metadata, adds focused Slack scope tests, and adds a changelog entry for modern granular bot tokens.

## Best Possible Solution

Keep the `auth.test` fallback after the two legacy methods, but treat only `response_metadata.scopes` as granted scopes for capability output; update the focused test and changelog wording so `acceptedScopes` is not presented as a token grant.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The linked bug gives concrete CLI steps and current main still contains the two-method fallback that produces the joined `unknown_method` error for modern granular bot tokens.

## Solution Assessment

Is this the best way to solve the issue? No. The fallback direction is narrow and maintainable, but including `acceptedScopes` in the granted-scope list is not correct for this diagnostic; using only `response_metadata.scopes` is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Do not report accepted scopes as granted scopes:** `extensions/slack/src/scopes.ts:75-77`
  - body: The added fallback pushes `response_metadata.acceptedScopes` into the same returned list as granted token scopes. Slack defines accepted scopes as the scopes the API method checks for, while this diagnostic prints returned values as `Bot scopes` or `User scopes`, so an acceptedScopes-only response can falsely claim the token has those grants. Use `response_metadata.scopes` for granted-scope discovery.
