## Summary

Keep open. The PR addresses a real Slack capabilities bug with a useful `auth.test` fallback, but the current diff still treats Slack SDK `acceptedScopes` as granted token scopes, which would make `openclaw channels capabilities` report method requirements as `Bot scopes` or `User scopes`.

## What This Changes

The PR adds a Slack `auth.test` fallback for scope discovery, extends extraction to Slack response metadata, adds focused Slack scope tests, and updates the changelog for the linked modern bot-token bug.

## Best Possible Solution

Keep the Slack plugin fix, but return granted scopes only from `response_metadata.scopes` for the `auth.test` fallback; remove accepted-scope extraction from the grant list and update the focused test and changelog wording to match.

## Reproduction Assessment

Yes. The linked issue has concrete CLI steps, current main still contains the exact two-method fallback and joined-error path, and the PR's new tests model the modern-token recovery scenario.

## Solution Assessment

No. The `auth.test` fallback plus `response_metadata.scopes` is the narrow maintainable fix, but including `acceptedScopes` in the granted-scope list is not correct for this diagnostic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not report accepted scopes as granted scopes:** `extensions/slack/src/scopes.ts:75-77`
  - body: The Slack SDK fills `response_metadata.acceptedScopes` from `x-accepted-oauth-scopes`, which is the set accepted by the Web API method rather than scopes granted to this token. Since capability diagnostics print returned values as `Bot scopes` or `User scopes`, an acceptedScopes-only response can now falsely report method requirements as token grants. Use only `response_metadata.scopes` for granted-scope discovery or label accepted scopes separately.
  - confidence: 0.88
