## Summary

Keep open. The PR is a useful narrow Slack plugin fix for the linked capabilities bug, but its current diff incorrectly treats Slack `response_metadata.acceptedScopes` as granted token scopes, which can make capability diagnostics falsely report API method requirements as `Bot scopes` or `User scopes`.

## What This Changes

The PR adds an `auth.test` fallback to Slack scope discovery, extends scope extraction to Slack response metadata, adds focused Slack scope tests, and adds a changelog entry for the modern bot-token capabilities bug.

## Best Possible Solution

Keep the Slack plugin fix direction, but return granted scopes only from `response_metadata.scopes` for the `auth.test` fallback; remove `acceptedScopes` from the grant list and align the focused test and changelog wording.

## Reproduction Assessment

Yes. The linked bug has concrete CLI steps, current main still contains the exact two-method fallback and joined-error path, and the PR's tests model the modern-token recovery scenario.

## Solution Assessment

No. The `auth.test` fallback plus `response_metadata.scopes` is the narrow maintainable solution, but including `acceptedScopes` in the granted-scope list is not correct for this diagnostic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Do not report accepted scopes as granted scopes:** `extensions/slack/src/scopes.ts:75-77`
  - body: The Slack SDK fills `response_metadata.acceptedScopes` from `x-accepted-oauth-scopes`, which is the set accepted by the Web API method rather than scopes granted to this token. Since capability diagnostics print returned values as `Bot scopes` or `User scopes`, an acceptedScopes-only response can now falsely report method requirements as token grants. Use only `response_metadata.scopes` for granted-scope discovery or label accepted scopes separately.
  - confidence: 0.88
