## Summary

Keep this open. Current main still renders the reported Control UI recommendation as static guidance, while the existing before/after diff and base-hash guarded config writes remain scoped to the Config surfaces and are not wired into Overview recommendation callouts. The related #38249 and #38420 items overlap, but both are open and do not supersede this narrower UI apply-flow request.

## What This Changes

Request the Control UI to let config-upgrade and hardening recommendations stage a config change, show a before/after diff, and apply it after explicit user acceptance instead of telling users to edit config files manually.

## Best Possible Solution

Keep this open until the Control UI has a shared recommendation-apply pattern that can stage a narrow config patch, show before/after values, require accept/dismiss, write through the existing base-hash guarded config path with current SecretRef/reload behavior, and confirm the saved value. Start with the insecure HTTP Overview hint, add focused UI/controller tests, and update the Control UI docs when behavior changes.

## Reproduction Assessment

Yes. A source-level reproduction is clear: in a non-secure browser context with a device-identity connection error, `renderOverview` reaches the insecure-context hint and renders only static guidance plus docs links, while no config write callback is available to that view.

## Solution Assessment

Yes, with guardrails. Reusing the existing base-hash guarded Config write/diff machinery is the maintainable direction, but the first implementation should be narrow and explicit because enabling `allowInsecureAuth` changes a security-sensitive compatibility toggle.
