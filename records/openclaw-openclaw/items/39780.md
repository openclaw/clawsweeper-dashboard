## Summary

The issue remains open: current main still presents the reported Control UI hardening/auth recommendation as manual text, and the existing Config apply path only submits a user-staged editor draft. Related work covers CLI upgrade scanning and the QuickStart allowInsecureAuth default, but none implements the requested Control UI before/after diff, accept/dismiss, and apply-confirmation flow.

## What This Changes

The issue requests Control UI config recommendations that can show a before/after diff and apply accepted config changes through the UI instead of instructing users to manually edit JSON.

## Best Possible Solution

Define a reusable Control UI recommendation-action model backed by existing config APIs, with an explicit safe-change allowlist, before/after diff, accept/dismiss, confirmation, and security-sensitive copy.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is a generic allowlisted recommendation-action flow over existing config APIs, not a hardcoded one-off allowInsecureAuth button.
