## Summary

Current main still lacks a recommendation-to-action workflow; the issue is a valid product proposal, but automatic security-sensitive config changes need explicit maintainer design approval rather than autonomous implementation.

## What This Changes

Add Control UI recommendation actions that show an exact before/after config diff and apply the accepted change through the existing config API.

## Best Possible Solution

Introduce a small typed catalog of maintainer-approved recommendation actions carrying an exact merge patch, rationale, security and restart metadata, then preview the computed diff, require explicit confirmation, apply with the current base-hash-protected config API, reload, and verify the resulting value.

## Solution Assessment

No, not as a generic mechanism: reusing the existing config API is correct, but executable recommendations should be typed, allowlisted, and carry explicit security/restart semantics rather than automatically applying arbitrary suggested edits.
