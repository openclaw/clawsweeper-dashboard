## Summary

Keep open: current main and v2026.6.10 still present the Control UI insecure-context/config hardening guidance as manual copy, while the existing Config tab apply path only submits a user-staged config draft. ## What This Changes

Request a Control UI recommendation-action flow that applies safe config hardening changes through existing config APIs after showing a before/after diff and getting user acceptance.

## Best Possible Solution

Define a reusable Control UI recommendation-action model backed by existing config RPCs, with an explicit safe-change allowlist, before/after diff, accept/dismiss, confirmation, and security-sensitive copy.

## Solution Assessment

Unclear until maintainers choose the product and security boundary. The maintainable direction is a generic allowlisted recommendation-action flow over existing config APIs, not a hardcoded one-off allowInsecureAuth button.
