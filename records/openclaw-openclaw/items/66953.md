## Summary

Keep open. Current main still allows a delivery-mirror route/session key and mirror agent id to diverge, and transcript append uses the supplied agent id to choose the session store. The report is recent, has concrete Feishu multi-account steps plus a corroborating comment, and the related reports cover adjacent duplicate/account-scope cases rather than this transcript-owner mismatch.

## What This Changes

Fix delivery-mirror transcript ownership so outbound sends from a writer Feishu account are written under the writer agent session instead of the default/main agent session.

## Best Possible Solution

Add a focused multi-agent, multi-account regression where the effective/default request agent differs from the outbound route session agent, then enforce a single invariant: delivery-mirror transcript store selection must follow the same owner as the route/session key, or the gateway must reject mismatched explicit agent/session inputs before delivery.

## Reproduction Assessment

Yes. The issue provides concrete Feishu multi-account steps, and current-main source inspection shows the same failure path: mirror.sessionKey can come from the outbound route while mirror.agentId remains the request/effective agent before transcript store selection.

## Solution Assessment

Unclear. Deriving mirror ownership from the outboundRoute.sessionKey looks like the narrow fix for transcript pollution, but the current gateway test suite explicitly asserts the opposite precedence for mismatched explicit agentId/sessionKey, so the ownership contract needs maintainer confirmation first.
