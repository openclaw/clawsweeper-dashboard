## Label Justifications

- `P2`: The bug affects multi-account transcript ownership and UI session state, but it is limited to a specific multi-agent routing configuration.
- `impact:session-state`: The reported failure writes transcript/session data under the wrong agent session, creating ghost sessions and state drift.
- `impact:security`: Cross-agent transcript writes can cross an intended isolation boundary when agent session stores are separated.

## Summary

Keep open. Current main and the latest release still let delivery-mirror transcript ownership diverge from the outbound route session owner, and the suggested fix touches an explicit-agent versus session-key precedence rule that needs maintainer direction.

## What This Changes

The issue reports that Feishu delivery-mirror transcript entries are written under the main agent instead of the writer agent in multi-account, multi-agent direct-message routing.

## Best Possible Solution

Define a single delivery-mirror transcript ownership invariant, then enforce it at route construction or transcript append with regression coverage for Feishu-style per-account direct sessions and the explicit-agent gateway send case.

## Reproduction Assessment

Yes for source-level reproduction: current main can route mirror.sessionKey to one agent while mirror.agentId selects another agent's transcript store. I did not run a live Feishu multi-account setup.

## Solution Assessment

Unclear. Deriving mirror.agentId from the routed session key is plausible for this report, but current gateway tests encode explicit agentId precedence, so maintainers should settle that invariant before a fix lands.
