## Summary

Keep open. Current main still permits delivery-mirror transcript ownership to diverge between the outbound session key and the mirror agent id, so the reported ghost-session behavior remains a source-level bug rather than stale cleanup.

## What This Changes

The issue requests that delivery-mirror transcript entries for multi-account outbound sends be written under the agent that owns the outbound session key instead of the default/main request agent.

## Best Possible Solution

Define one transcript ownership invariant for delivery-mirror writes, then enforce it at mirror construction or transcript append so sessionKey and agentId cannot target different agent stores, with regression coverage for message-action and gateway send paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main still lets mirror.sessionKey come from the outbound route while the transcript store is selected by mirror.agentId. I did not run a live Feishu setup, but the issue's steps map directly onto that code path.

## Solution Assessment

Unclear. Deriving mirror.agentId from outboundRoute.sessionKey is plausible for the reported ghost session, but current gateway tests encode explicit-agent precedence, so maintainers should first settle the ownership rule and then patch the narrow paths consistently.
