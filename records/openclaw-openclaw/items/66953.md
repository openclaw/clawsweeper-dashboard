## Summary

Keep open. Current main still allows the routed delivery session key and delivery-mirror transcript owner agent to diverge, so the reported cross-agent transcript pollution remains plausible on source inspection; the corroborating comment supports that this is not a one-off report.

## What This Changes

The issue requests that multi-account, multi-agent delivery-mirror transcript entries be written under the agent encoded by the routed session key instead of the default/request agent.

## Best Possible Solution

Add a focused multi-agent, multi-account regression and enforce one invariant: delivery-mirror transcript storage follows the route/session-key owner, or mismatched explicit agent/session inputs are rejected before delivery.

## Reproduction Assessment

Yes. The issue gives concrete Feishu multi-account steps, and current-main source inspection shows the reproducing shape: mirror.sessionKey can be route-owned while mirror.agentId remains request/effective-agent-owned before transcript store selection.

## Solution Assessment

Unclear. Deriving mirror.agentId from the outbound route session key is a plausible narrow fix, but it conflicts with an existing gateway test that says explicit agentId should override the sessionKey agent for mirror delivery.
