## Summary

Keep open: this is a focused fix for a real memory-core startup log-noise bug, but the current head is not merge-ready because the new `startup_retry` reason does not typecheck, the promised post-exhaustion warning path is still missing, and real behavior proof remains unit-test only.

## What This Changes

The PR adds a startup-retry reconciliation reason for memory-core dreaming cron retries and updates focused tests so transient retry-window cron-unavailable logs are debug-only while runtime failures still warn.

## Best Possible Solution

Use a typed startup-retry reason or flag only for scheduled retry callbacks, keep ordinary runtime warnings intact, and add a final warning plus coverage when the bounded retry budget is exhausted.

## Reproduction Assessment

Yes. Current main routes the bounded startup retry through the runtime reconciliation warning path, and the focused test currently expects the warn-level `cron service unavailable` log after the retry delay.

## Solution Assessment

No, not in the current form. A dedicated startup-retry reason is the right direction, but the implementation must typecheck and must warn after retry exhaustion while preserving genuine runtime warning coverage.
