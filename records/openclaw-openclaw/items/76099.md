## Summary

Keep this PR open. The explicit `startup_retry` reason is the right direction, but the current patch makes every scheduled startup retry debug-only and then the retry cap returns silently, so it does not preserve the promised warning when cron never becomes available after the startup window.

## What This Changes

The PR changes memory-core dreaming cron startup retry reconciliation so transient cron unavailability logs at debug level during the retry window while preserving warn-level runtime failures.

## Best Possible Solution

Keep the explicit `startup_retry` path, add a tested warn-through when the bounded retry window is exhausted, preserve normal runtime warnings, and attach redacted real logs or terminal output that show the after-fix startup behavior.

## Reproduction Assessment

Yes, from source inspection: current main schedules startup retries as runtime reconciliation, and the adjacent fake-timer test expects the cron-unavailable warning on the first retry tick.

## Solution Assessment

No, not in the current form. The explicit `startup_retry` reason is maintainable, but the patch also needs an exhausted-retry warning path and proof that the real startup behavior matches the intended operator signal.
