## Summary

Keep open: the PR fixes part of the reused CLI-session retry path, but it still does not clear the persisted session-store binding for timeout/unknown failures when the fresh retry also fails. The author’s fixup redacts the new log and fixes the changelog reference, but the caller-owned clear/persist path remains `session_expired`-only.

## What This Changes

The PR widens `runPreparedCliAgent` retry handling for reused-session `FailoverError`s, adds timeout/unknown regression tests, and adds a changelog entry for #77089.

## Best Possible Solution

Land a narrow fix that clears the persisted CLI session binding for any `FailoverError` from an active reused binding, retries once fresh, persists a replacement binding on success, and leaves non-resume failures unchanged.

## Reproduction Assessment

Yes. Source inspection shows watchdog failures become `FailoverError` reason `timeout`, while the current store-clearing recovery branch only handles `session_expired`; the missing timeout/unknown caller-level path is the reproducible gap.

## Solution Assessment

No, not as-is. Widening recovery is the right direction, but the durable clear/retry behavior belongs in `attempt-execution`, where the session store is available, or it needs an equivalent callback from the runner.
