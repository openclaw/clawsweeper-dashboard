## Summary

Keep open: the PR addresses a valid, source-reproducible CLI stale-session bug, but the latest diff still leaves a stale stored binding when the fresh retry fails with a different FailoverError reason, and the branch also needs merge-state cleanup before it can land.

## What This Changes

The PR broadens resumed CLI FailoverError recovery, adds timeout/unknown regression tests, widens caller-level store clearing, and adds a changelog entry for #77089.

## Best Possible Solution

Clear the persisted CLI binding in the store-aware path before starting the fresh retry, or pass an explicit store-clear callback into the runner, then persist any replacement binding on success while preserving existing fallback behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows resumed CLI sessions only clear/retry for `session_expired`, while the executor can throw `timeout` and `unknown`; the related incident report gives a concrete stale-session sequence.

## Solution Assessment

No, not as-is. Widening recovery is the right direction, but the stored binding must be cleared based on the original reused-session failure before the fresh retry can hide that original reason.
