## Summary

Keep open: the reported stale CLI-session failure is valid on current main, but this PR needs a tighter store-clearing fix before merge because the retry is widened in a layer that cannot guarantee the persisted stale binding is removed before fallback.

## What This Changes

The PR widens `runPreparedCliAgent` stale-session retry handling to any `FailoverError`, adds warning logging, adds timeout/unknown regression tests, and adds a changelog entry.

## Best Possible Solution

Move or augment the widened FailoverError handling at the session-store owner boundary so a resumed-session failure clears or replaces the persisted CLI binding before retry/fallback, with regression coverage for `timeout` and `unknown`.

## Reproduction Assessment

Yes, by source inspection: current main maps CLI watchdog and generic CLI failures to `FailoverError` reasons `timeout`/`unknown`, while both retry/clear branches are narrowed to `session_expired`. I did not run tests because this review is read-only and `pnpm` is unavailable in the environment.

## Solution Assessment

No. Widening the inner runner retry covers the fresh-retry success path, but the durable fix should prove the persisted stale binding is cleared or replaced before fallback, ideally in `src/agents/command/attempt-execution.ts` or via an explicit clear callback.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Clear the persisted binding before fallback:** `src/agents/cli-runner.ts:331`
  - body: This widened retry happens inside `runPreparedCliAgent`, which does not own `clearCliSessionInStore`. Please clear at the session-store boundary, or add an explicit clear callback, and cover the persisted-store case.
