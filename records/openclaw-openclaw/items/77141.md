## Summary

Keep open: the PR targets a valid CLI stale-session bug, but the latest patch still leaves a stale stored binding when a resumed-session timeout/unknown failure is followed by a fresh retry that fails with another FailoverError reason.

## What This Changes

The PR broadens CLI stale-session recovery for resumed sessions, adds timeout/unknown regression coverage in the runner and attempt-execution paths, and adds a changelog entry for the related CLI backend incident.

## Best Possible Solution

Clear the persisted CLI session binding based on the original reused-session failure before starting the fresh retry, then preserve the existing one-fresh-retry behavior and replacement binding persistence.

## Reproduction Assessment

Yes. Source inspection shows current main gates stale-session cleanup on session_expired even though the executor emits timeout and unknown, and the remaining PR bug follows from the runner rethrowing only the fresh retry's FailoverError reason.

## Solution Assessment

No, not as-is. Widening stale-session recovery is the right direction, but the cleanup must be tied to the original reused-session failure rather than the fresh retry's later reason.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear stale bindings before the fresh retry can fail:** `src/agents/cli-runner.ts:331-334`
  - body: The runner now retries fresh on any FailoverError before the store-aware caller clears the persisted binding.
