## What This Changes

The issue reports that forced memory indexing can persist providerKey and scopeHash metadata incompatible with the runtime embedding provider, disabling memory_search.

## Best Possible Solution

Review or replace https://github.com/openclaw/openclaw/pull/92079 only after the degradation-to-recovery path is proven, and keep any broader missing-metadata work aligned with https://github.com/openclaw/openclaw/issues/90361.

No high-confidence current-main CLI reproduction was established. Source inspection shows the providerKey and scopeHash mismatch gates, but the CLI force path currently initializes the provider before reindexing, so the stale-sentinel write needs a focused regression or live proof.

## Solution Assessment

Unclear. A narrow repair for the exact stale FTS-only sentinel may be acceptable if proven, but broad runtime auto-repair of provider metadata should not land without preserving genuine provider/model/settings mismatch protection.
