## Summary

Keep open, but the reported root-cause explanation is not supported by v2026.6.5 or current main: both initialize the configured provider before forced sync writes metadata, and both use the normalized multimodal byte limit. The observable provider mismatch still merits a focused real-config reproduction because it could arise from provider configuration, SecretRef, or cache-key differences not captured in the report.

## What This Changes

Fix forced CLI memory indexing allegedly writing provider and scope metadata that runtime memory search rejects.

## Best Possible Solution

Reproduce with a redacted OpenAI-compatible configuration, capture provider identity and index metadata before and after `memory index --force`, then fix the earliest canonical identity divergence rather than bypassing the runtime mismatch guard.

## Reproduction Assessment

No. Source inspection shows provider initialization occurs before metadata writes in both v2026.6.5 and current main, while the report lacks the redacted resolved configuration and before/after metadata needed to reproduce another provider-identity divergence.

## Solution Assessment

No, not as proposed. Loading the provider before CLI metadata writes is already the canonical behavior, while unconditional runtime auto-repair would weaken a valid index-compatibility guard; the fix should target a demonstrated difference in provider or scope identity construction.
