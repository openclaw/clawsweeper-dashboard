## Summary

Keep open. The reported runtime `memory_search` failure is plausible and recent, but current `main` does not provide enough proof to close it: the CLI force path, provider-key identity check, and openai-compatible cache-key behavior still need a focused regression around the exact providerKey/scopeHash mismatch.

## What This Changes

The issue reports that `openclaw memory index --force` writes a CLI-only memory index identity that does not match the runtime embedding provider, disabling runtime `memory_search`.

## Best Possible Solution

Add a focused regression for CLI force reindex with openai-compatible cacheKeyData and default multimodal scope, then fix the narrow identity-write path if it fails without adding broad runtime shims.

## Reproduction Assessment

No, not a high-confidence live reproduction. Source inspection identifies the CLI force entry point, providerKey identity gate, and openai-compatible cacheKeyData surface, but the current code path must be exercised with a focused regression to prove the exact DashScope mismatch.

## Solution Assessment

Unclear as proposed. Loading or aligning the CLI manager with the runtime provider identity is likely the narrow fix if the regression fails, while automatic runtime repair of mismatched provider metadata would be broader and should not be the first move without proof.
