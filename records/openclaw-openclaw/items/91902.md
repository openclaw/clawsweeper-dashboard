## Summary

Close as duplicate/superseded: the providerKey/meta-write part is already covered by the closed memory metadata issue, current source routes CLI force reindex through provider-initialized sync, and the remaining scopeHash symptom has its own open canonical report.

## What This Changes

The issue reports that `openclaw memory index --force` writes memory index metadata with a CLI-only provider key that is incompatible with the runtime embedding provider, causing `memory_search` to disable itself after reindexing.

## Best Possible Solution

Keep the current provider-initialized CLI sync and current-main missing-metadata repair, and route any remaining scopeHash-specific failure to https://github.com/openclaw/openclaw/issues/91592.

## Reproduction Assessment

Yes for the already-covered metadata failure class: source and regression coverage show how invalid/missing index identity pauses search and how current main repairs it. The exact claim that CLI force writes a `provider:"none"` providerKey is not present in current source because sync initializes the provider before reindexing.

## Solution Assessment

Yes. Closing this as duplicate/superseded is the best cleanup path because the providerKey/meta-write report matches the already closed canonical issue, while the separate scopeHash symptom remains tracked in the open canonical scopeHash issue.
