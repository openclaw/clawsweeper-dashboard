## What This Changes

The issue reports that `openclaw memory index --force` can write a stale or incompatible memory index `providerKey`, causing runtime `memory_search` to fail provider identity checks for openai-compatible embeddings.

## Best Possible Solution

Keep the issue open until the prerequisite provider-degradation behavior is decided and a narrow stale-sentinel providerKey repair has live CLI-to-runtime proof while preserving real provider/model/settings mismatch blocking.

## Reproduction Assessment

No live high-confidence reproduction is available. Source inspection shows stale providerKey metadata can fail identity and suppress search, but current `sync()` initializes the provider before CLI reindex, so the original `--force` path still needs focused proof.

## Solution Assessment

Unclear as proposed. A narrow stale FTS-only sentinel repair is plausible after the prerequisite lifecycle is proven, but broad automatic rewriting of arbitrary providerKey mismatches would weaken the current safety check.
