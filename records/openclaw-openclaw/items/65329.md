## Summary

Keep open. Current main still has the MS Teams personal-DM media gap described here: non-HTML DM images/files can fall through direct download, the Bot Framework fallback is still gated on HTML `<attachment>` IDs, and the caller passes a Graph-style conversation ID into the fallback path, so the documented RSC-only DM attachment support is not reliably delivered.

## What This Changes

The issue asks for MS Teams personal-DM inline images and file attachments to be resolved and passed to the agent instead of being silently dropped as unresolved media placeholders.

## Best Possible Solution

Fix the MS Teams plugin’s inbound personal-DM media resolver so RSC-only DMs can download real non-HTML image/file media after direct download fails, preferably by preserving the original Bot Framework conversation ID for Bot Framework attachment fetches, deriving eligible attachment IDs or candidates from real media/file attachments, and keeping mention-only HTML messages out of the fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `resolveMSTeamsInboundMedia` for a personal DM where `downloadMSTeamsAttachments()` returns `[]` and the activity contains non-HTML media/file attachments; current code and tests show neither Bot Framework nor Graph fallback runs unless HTML `<attachment>` IDs exist.

## Solution Assessment

No. Current main is not the best solution because it still ties the fallback to HTML attachment IDs and can hide the original Bot Framework DM identifier. The safer solution is a narrow DM fallback that only triggers for real media/file attachments and keeps the mention-only guard intact.
