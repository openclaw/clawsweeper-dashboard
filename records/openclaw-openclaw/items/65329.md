## Label Justifications

- `P1`: Personal-DM media is a broken channel workflow that can silently prevent real user attachments from reaching the agent.
- `impact:data-loss`: The issue is about user-sent image/file content being silently dropped before the agent receives it.
- `impact:message-loss`: The affected MS Teams channel path omits attachment payloads from inbound messages while leaving only placeholders.

## Summary

Keep open. Current main still leaves MS Teams personal-DM non-HTML media without a Bot Framework fallback after direct download returns no media, so pasted images or direct file attachments can still be represented only as unresolved media placeholders.

## What This Changes

Fix MS Teams personal DM inline images and file attachments so attachment bytes are downloaded and passed to the agent instead of silently leaving unresolved `msteams://media/...` placeholders.

## Best Possible Solution

Land a narrow MS Teams plugin fix that preserves the original Bot Framework DM id, attempts Bot Framework fallback only for real non-HTML DM media/file candidates after direct download fails, and adds regression coverage without broadening mention-only fallback behavior.

## Reproduction Assessment

Yes, source-level. A personal-DM call where direct attachment download returns `[]` and `extractMSTeamsHtmlAttachmentIds()` returns `[]` still skips both Bot Framework and Graph fallback paths on current main; no live Teams tenant was run.

## Solution Assessment

Yes. The narrowest maintainable fix is to keep the mention-only HTML guard but add a real non-HTML DM media fallback path using the original Bot Framework conversation id and focused regression tests.
