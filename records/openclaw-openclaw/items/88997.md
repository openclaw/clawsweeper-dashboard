## Summary

Keep open: the approach fits the existing delivery-mode seam and current source explains why Discord sends should go through Gateway hooks, but the PR still needs real Discord/Gateway behavior proof and maintainer acceptance of the compatibility change before merge.

## What This Changes

The PR changes Discord's outbound adapter from direct to gateway delivery and adds a regression assertion for the adapter delivery mode.

## Best Possible Solution

Land gateway-routed Discord delivery only after proof shows a real Discord CLI send goes through Gateway hooks/scopes and maintainers accept the no-direct-bypass behavior.

## Reproduction Assessment

Yes from source inspection: current main has `discordOutbound.deliveryMode` set to direct, so `sendMessage` takes the non-gateway branch and does not forward Gateway client scopes. I did not run a live Discord/Gateway reproduction in this read-only review.

## Solution Assessment

Likely yes: using the existing `deliveryMode: "gateway"` seam is the narrowest owner-boundary fix for core/CLI sends while the Gateway handler still performs final platform delivery through the adapter. The remaining question is upgrade safety, not a better code layer.
