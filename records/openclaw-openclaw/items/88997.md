## Summary

Keep open: current main does not route Discord CLI/core sends through gateway hooks, but this PR is not merge-ready because it changes the shared CLI owner default for every `openclaw message` action and still lacks real Discord behavior proof.

## What This Changes

## Best Possible Solution

Yes at source level: current main still declares Discord outbound delivery as direct, and the gateway-send path is the hook-running route the PR is trying to use. I did not establish a live Discord protected-scope reproduction in this read-only review.

## Solution Assessment

No, not as written. The Discord `deliveryMode: "gateway"` direction is plausible, but globally forcing CLI message actions to non-owner is broader than the bug and conflicts with current command semantics.
