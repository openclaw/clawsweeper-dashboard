## Summary

Keep open. The PR targets a real current-main security-sensitive behavior, but it is not mergeable as submitted because it changes only the validator and local net tests while current main still has caller, onboarding, and shared-test contracts that accept opted-in `ws://` hostnames from the Docker/Kubernetes remote-gateway workflow.

## What This Changes

The branch removes the `net.isIP(...) === 0` hostname fallback from `src/gateway/net.ts` and updates `src/gateway/net.test.ts` so `allowPrivateWs` permits private IP literals but rejects `ws://` hostnames.

## Best Possible Solution

Choose the hostname policy explicitly, then land one coordinated change that updates the validator, GatewayClient, connection-details, onboarding validation, shared mocks, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main reaches `net.isIP(hostForIpCheck) === 0` under `allowPrivateWs`, and a focused Node check shows ordinary hostnames return `0`; I did not run a full gateway flow in this read-only review.

## Solution Assessment

No, not as submitted. Removing the fallback may be the right hardening direction, but this PR only updates the validator and net tests while current caller/onboarding contracts, the shared mock, docs, and prior hostname-support history remain out of sync.
