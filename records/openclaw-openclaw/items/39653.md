## Summary

Keep open. The security concern is real on current main, but this external PR is incomplete: it changes only the validator and validator tests while current caller tests/shared mocks still encode hostname acceptance, and it lacks required after-fix real behavior proof.

## What This Changes

The PR removes the allowPrivateWs hostname fallback in `isSecureWebSocketUrl` and updates gateway net tests so opted-in plaintext WebSocket mode allows private IP literals but rejects hostnames.

## Best Possible Solution

Decide the private-network hostname policy explicitly, then land a complete fail-closed implementation or a safer hostname mechanism that updates the validator, callers, shared mocks, tests, docs, and maintainer-owned release notes together.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main’s validator returns `net.isIP(hostname) === 0` in the allowPrivateWs branch, and a focused Node check confirms ordinary hostnames return `0`.

## Solution Assessment

No, not as submitted. Removing the fallback is a narrow hardening step, but the PR must also update caller contracts/shared mocks and resolve the intentional hostname support policy before it is maintainable.
