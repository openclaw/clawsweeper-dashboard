## Summary

Keep open: this is a real gateway security-policy question, but the branch conflicts with current private-DNS support for Docker/Kubernetes remote gateways and still lacks real behavior proof.

## What This Changes

The PR removes hostname fallback acceptance from `isSecureWebSocketUrl(..., { allowPrivateWs: true })` and updates gateway net tests to reject `ws://` hostnames while keeping private IP literals allowed.

## Best Possible Solution

Make one explicit gateway plaintext WebSocket hostname policy decision, then update the validator, client/call/onboarding paths, docs, mocks, and regression tests together.

## Reproduction Assessment

Yes, for source-level reproduction. Current main accepts ordinary hostnames under `allowPrivateWs`, current tests lock in that behavior, and a focused Node check confirms `net.isIP()` returns `0` for hostnames.

## Solution Assessment

No, not as submitted. Removing the fallback may be the right hardening direction, but the isolated validator/test change reverses an intentional private-DNS workflow without aligning the rest of the gateway policy surface.
