## Summary

Keep open. The PR addresses a real source-level `allowPrivateWs` hostname bypass, but it only updates the validator unit test while current caller/onboarding tests and the shared gateway mock still encode hostname acceptance; it also lacks external real behavior proof and needs maintainer direction for the private-DNS hostname policy.

## What This Changes

The PR removes the `net.isIP(hostForIpCheck) === 0` hostname fallback from `isSecureWebSocketUrl(..., { allowPrivateWs: true })` and adds gateway net regression coverage for hostname rejection.

## Best Possible Solution

Land a complete security-policy change that explicitly chooses fail-closed hostnames or DNS/allowlist validation and updates the validator, callers, shared mocks, docs, and regression tests together.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main returns `net.isIP(hostname) === 0` in the `allowPrivateWs` branch, and a focused Node check confirms ordinary hostnames return `0`.

## Solution Assessment

No, not as submitted. Removing the fallback is a plausible hardening step, but the PR needs caller contracts, the shared mock, docs/tests, and hostname policy kept in sync before it is maintainable.
