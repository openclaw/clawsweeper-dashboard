## Summary

Keep open. The security concern is source-reproducible, but this PR would reject hostname `ws://` only in the validator while current main and the latest release still carry caller, onboarding, and test contracts that allow private-network hostname workflows under `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`; it also lacks contributor real behavior proof.

## What This Changes

The PR removes the hostname fallback from `isSecureWebSocketUrl` in `allowPrivateWs` mode and updates net tests to reject `ws://` hostnames while still allowing private IP literals.

## Best Possible Solution

Land one coordinated security-policy change that either validates private hostname targets safely or intentionally rejects hostnames across the validator, GatewayClient, connection-details, onboarding, docs, mocks, and regression coverage.

## Reproduction Assessment

Yes. Source inspection plus the focused Node `net.isIP` check show current main accepts ordinary hostnames under `allowPrivateWs`, and current tests lock that behavior in the validator, client, connection-details, and onboarding paths.

## Solution Assessment

No, not as submitted. Removing the validator fallback may be a valid hardening direction, but this PR leaves shipped caller contracts and operator documentation out of sync, so the policy change needs to be coordinated.
