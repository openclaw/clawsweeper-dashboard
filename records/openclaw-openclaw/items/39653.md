## Summary

Keep this PR open. Current main still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, so the hardening remains relevant, but the submitted two-file patch is not merge-ready because downstream gateway and onboarding tests still assert hostname acceptance under the same break-glass flag.

## What This Changes

The PR removes the hostname fallback from `isSecureWebSocketUrl(..., { allowPrivateWs: true })` and updates `src/gateway/net.test.ts` so the break-glass mode accepts private/loopback IP literals but rejects hostnames.

## Best Possible Solution

Land the narrow fail-closed validator change only after aligning all gateway caller tests and onboarding expectations with the chosen contract. If private DNS hostname support is required, track it separately with an explicit allowlist or verified-resolution design rather than keeping the current blanket hostname pass-through.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: current main returns true for non-IP hostnames in the `allowPrivateWs` branch because `net.isIP()` returns `0`, and current tests already assert hostname acceptance under the env flag; I did not run Vitest because this review is read-only and tests can create cache artifacts.

## Solution Assessment

No, not as submitted. The validator direction is the narrow maintainable fix for a synchronous URL check, but the PR should also update `GatewayClient`, connection-detail, and onboarding tests so the changed contract is consistently enforced and CI-ready.
