## Summary

Keep open. The PR is plausible security hardening, but current main still intentionally tests and documents hostname-based private-network `ws://` flows from https://github.com/openclaw/openclaw/issues/36930, so this needs a maintainer security/product decision and real behavior proof before merge.

## What This Changes

The PR removes the `allowPrivateWs` hostname fallback from `isSecureWebSocketUrl` and adds gateway net tests expecting `ws://` hostnames to be rejected while private IP literals remain allowed.

## Best Possible Solution

Land one coordinated security-policy change that either validates private hostname targets safely or intentionally rejects hostnames across the validator, GatewayClient, connection-details, onboarding, docs, mocks, and regression coverage.

## Reproduction Assessment

Yes. Source inspection plus a focused Node `net.isIP` check show current main accepts ordinary hostnames under `allowPrivateWs`, and current tests lock that behavior in the validator, client, connection-details, and onboarding paths.

## Solution Assessment

No, not as submitted. Removing the validator fallback may be a valid hardening direction, but this PR leaves shipped caller contracts and operator documentation out of sync, so the policy change needs to be coordinated.
