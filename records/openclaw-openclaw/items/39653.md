## Summary

Keep open. The PR targets a real current-main hostname fallback, but it is incomplete because caller/onboarding contracts and the shared gateway test mock still allow hostname `ws://` URLs, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR removes hostname pass-through from `isSecureWebSocketUrl(..., { allowPrivateWs: true })` and updates gateway net tests so private/loopback IP literals remain allowed while hostnames are rejected.

## Best Possible Solution

Decide the private-network hostname policy explicitly, then land a complete fail-closed or DNS/allowlist implementation that updates the validator, callers, shared mocks, docs, and regression tests together.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. Current main returns `net.isIP(hostname) === 0` in the `allowPrivateWs` branch, and a focused Node check confirms ordinary hostnames return `0`.

## Solution Assessment

No, not as submitted. Removing the fallback is a plausible hardening step, but this PR needs the caller contracts, shared mock, docs/tests, and hostname policy decision kept in sync before it is maintainable.
