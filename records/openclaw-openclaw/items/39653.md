## Summary

Keep open. The hardening direction may be valid, but this PR changes only the gateway URL validator and net tests while current main intentionally supports break-glass `ws://` hostnames across onboarding, runtime client checks, docs, and regression tests after https://github.com/openclaw/openclaw/issues/36930.

## What This Changes

The PR removes the `allowPrivateWs` hostname fallback in `src/gateway/net.ts` and updates gateway net tests so opted-in `ws://` hostnames are rejected while private IP literals remain accepted.

## Best Possible Solution

Make one explicit gateway plaintext WebSocket policy decision, then update the validator, clients, onboarding, mocks, docs, and regression tests together.

## Reproduction Assessment

Yes, source-level. Current main accepts ordinary hostnames under `allowPrivateWs`, current caller tests lock in `ws://openclaw-gateway.ai` with the env var set, and a focused Node check confirms `net.isIP()` returns `0` for hostnames.

## Solution Assessment

No, not as submitted. Removing the fallback may be the right hardening direction, but a validator-only/net-test change conflicts with current caller contracts and docs instead of making one coordinated gateway WebSocket policy change.
