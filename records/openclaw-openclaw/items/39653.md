## Summary

Keep open. The PR targets a real source-level security concern, but it reverses current hostname opt-in behavior that is still encoded in caller/onboarding tests, the shared gateway mock, docs, and the prior Docker/Kubernetes remote-gateway fix, while also lacking contributor real behavior proof.

## What This Changes

The branch removes the `allowPrivateWs` hostname fallback from `isSecureWebSocketUrl` and updates gateway net tests so opted-in `ws://` hostnames are rejected.

## Best Possible Solution

Choose the hostname policy explicitly, then land one coordinated change that updates the validator, GatewayClient, connection-details, onboarding validation, shared mocks, docs, and regression coverage together.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main reaches `net.isIP(hostForIpCheck) === 0` under `allowPrivateWs`, and a focused Node check shows ordinary hostnames return `0`. I did not run a full gateway flow in this read-only review.

## Solution Assessment

No, not as submitted. Removing the fallback is a plausible hardening direction, but this PR only updates the validator and net tests while current caller/onboarding contracts, the shared mock, docs, and prior hostname-support history remain out of sync.
