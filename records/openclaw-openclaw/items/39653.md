## Summary

Keep open. The current-main hostname allowance is real, but the PR conflicts with existing GatewayClient, connection-details, onboarding, and mock contracts that intentionally accept hostname `ws://` URLs under the break-glass flag, so it needs maintainer security/product direction and real behavior proof before merge.

## What This Changes

The PR removes the `allowPrivateWs` hostname fallback in `isSecureWebSocketUrl()` and updates net tests so private IP literals remain allowed while `ws://` hostnames are rejected.

## Best Possible Solution

Land one coordinated security-policy change that either keeps hostname support with a safer allowlist/DNS check or rejects hostnames everywhere, updating the validator, GatewayClient, connection-details, onboarding, mocks, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main's `allowPrivateWs` branch returns `net.isIP(hostForIpCheck) === 0`, Node returns `0` for ordinary hostnames, and caller tests assert hostname acceptance under the env override.

## Solution Assessment

No, not as submitted. Removing the fallback may be the right hardening direction, but this validator-only PR leaves current caller, onboarding, mock, docs, and shipped-workflow contracts out of sync.
