## Summary

Keep open. The PR targets a real hostname-acceptance security concern, but current `main` still treats `ws://` hostnames under the break-glass flag as an intentional Docker/Kubernetes private-DNS workflow, and this patch only updates the validator/test surface without coordinating the caller contracts or real behavior proof.

## What This Changes

The PR removes the hostname fallback from `isSecureWebSocketUrl(..., { allowPrivateWs: true })` and updates gateway net tests to reject `ws://` hostnames while private IP literals remain allowed.

## Best Possible Solution

Make one explicit gateway WebSocket policy decision: either safely validate private hostnames or reject them, then update the validator, clients, onboarding, docs, mocks, and tests together.

## Reproduction Assessment

Yes, for source-level reproduction. Current `main` accepts ordinary hostnames under `allowPrivateWs`, and current caller tests lock in that behavior while a Node `net.isIP` check explains why the fallback returns true.

## Solution Assessment

No, not as submitted. Removing the fallback may be the right hardening direction, but this PR changes only the validator and nearby test, leaving shipped caller contracts and private-DNS workflows out of sync.
