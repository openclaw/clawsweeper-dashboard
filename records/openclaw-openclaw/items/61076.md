## Summary

Keep open. Current main has an adjacent improvement that classifies malformed Anthropic SSE JSON as a stable malformed-fragment error and surfaces friendlier text, but the Anthropic transport still ends the stream with `stopReason: "error"`, so the reported mid-response termination is not solved. Related #68442 is closed and #59076/#59118 cover nearby chat-surface/tool-call error handling, but they do not supersede this Telegram pure-text streaming failure mode.

## What This Changes

The issue requests hardened handling for intermittent Anthropic streaming JSON parse failures in embedded agent runs so long emoji/non-ASCII responses do not abort the run.

## Best Possible Solution

Keep the issue open and add a narrow Anthropic Messages streaming fix: first codify the reported failure class with parser tests for raw control characters inside SSE JSON string values, valid emoji split across UTF-8 chunks, and multiline `data:` frames; then add scoped parser/caller recovery or retry behavior that prevents a raw JSON syntax error from silently aborting a user-visible Telegram reply.

## Reproduction Assessment

Partially yes: current source and tests give a high-confidence synthetic reproduction for malformed Anthropic SSE causing `stopReason: "error"`. The exact intermittent live pure-text emoji trigger has not been reduced to a deterministic Anthropic API reproduction.

## Solution Assessment

No: current main's stable `malformed_streaming_fragment` classification is useful but not the best complete solution because it still terminates the agent run. A narrower maintainable fix would harden the Anthropic SSE parser or its caller with focused recovery semantics and regression coverage for the reported control-character failure class.
