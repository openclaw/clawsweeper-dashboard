## Summary

Keep this open. Current source still makes provider-scoped configure updates derive their removable scope from the live/static catalog, so an existing allowlist key that disappeared from the catalog can be excluded from the multi-select scope and survive the write.

## What This Changes

The issue requests that `openclaw configure` prune or visibly flag stale/delisted model entries instead of leaving them in `openclaw.json` indefinitely.

## Best Possible Solution

Include existing same-provider allowlist/fallback keys that are missing from the catalog as stale/removable entries in the configure multi-select, and leave destructive `models.providers` pruning to an explicit doctor/cleanup policy if maintainers want it.

## Reproduction Assessment

Yes from source inspection: with a preferred provider and `loadCatalog` enabled, the removable scope is derived from catalog rows, and scoped deletion preserves keys outside that scope. I did not run an interactive live wizard session.

## Solution Assessment

Yes for the narrow allowlist bug: extend the provider-scoped picker to show existing same-provider keys missing from the catalog as `configured (not in catalog)` and let deselection remove them. Automatic pruning of `models.providers` should not be bundled into that fix without an explicit maintainer decision.
