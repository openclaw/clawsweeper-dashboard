## Summary

Current main implements the requested exact compaction and memory-flush model overrides, but no schema, runtime, docs, or tests implement flush-then-reset; that remaining session-lifecycle behavior is a maintainer product decision, and the open per-agent compaction work does not replace it.

## What This Changes

Request configurable models for compaction and memory flush, plus an optional automatic session reset after a successful flush.

## Best Possible Solution

Retain the current exact model overrides, then—only with explicit maintainer approval—add one canonical opt-in post-flush lifecycle policy that reuses session reset machinery, defines safe execution boundaries and preserved settings, and proves resume behavior plus failure handling.

## Reproduction Assessment

Not applicable: this is a request for new configuration and session-lifecycle behavior, not a failure of an existing documented contract. Source inspection does provide a high-confidence capability check: both model overrides exist and flush-then-reset does not.

## Solution Assessment

Partly yes: separate exact model overrides are already the narrow supported solution for housekeeping cost control. No for an immediate `resetAfterFlush` boolean without first defining a canonical reset boundary and preservation contract across active session work.

## Risks / Open Questions

- Automatic reset after flush can discard conversational continuity that was not selected for durable memory, so a successful flush is not proof that all user-relevant session state is safely reconstructible.

- The overlapping open per-agent compaction PR changes the same provider/model/auth and session-agent routing surfaces, so independent implementation would create conflict and duplicate contract decisions.
