## Summary

Keep open: current main implements the compaction and memory-flush model override parts, but the requested `memoryFlush.resetAfterFlush` mode is still absent from the strict config schema, docs, and runtime flow. The remaining work is a new session-lifecycle policy/config feature, not a conservative cleanup close.

## What This Changes

The issue requests cheaper maintenance-model controls for compaction and memory flush plus an optional flush-then-reset mode that skips compaction after durable memory is written.

## Best Possible Solution

Keep the shipped model overrides, then design a guarded reset-after-flush mode with explicit schema, docs, failure semantics, queue handling, and regression coverage around the existing session reset lifecycle.

## Reproduction Assessment

Not applicable. This is a feature/config request; current-main source, schema, docs, and targeted search show the model overrides exist and the reset-after-flush option does not.

## Solution Assessment

Unclear. The shipped model overrides are the narrow maintainable cost-control path already implemented, while reset-after-flush needs maintainer-selected lifecycle and state-safety semantics before code should be added.
