## Summary

Current main and the latest release already provide exact model overrides for both compaction and memory flush, but the central flush-then-reset capability remains intentionally unresolved: successful flush still feeds the normal compaction path, while session reset is owned by a separate gateway lifecycle service. The issue should remain open for a maintainer product decision, coordinated with the overlapping draft per-agent compaction PR rather than treated as a narrow bug or a new automated fix.

## What This Changes

Add independent model selection for compaction and memory flush, and an opt-in mode that starts a fresh session after a successful memory flush instead of compacting the transcript.

## Best Possible Solution

Keep the shipped exact model overrides, settle the overlapping per-agent config work first, then—only with explicit maintainer approval—add a default-off flush-success policy that delegates to the canonical gateway reset service, defines queued/subagent/model-setting preservation rules, and proves both successful handoff and flush-failure behavior.

## Risks / Open Questions

- A successful memory flush is a model-generated persistence attempt, not proof that every recent conversational detail, pending decision, or tool result was written durably; automatic reset can therefore cause intentional but surprising session-state loss.
- The canonical reset lifecycle clears and reconstructs multiple runtime surfaces; bypassing it from memory flush could leave queued follow-ups, subagents, embedded runs, process state, or channel/session metadata attached to the wrong session.
- The overlapping draft per-agent compaction PR changes config resolution and provider/model/auth-profile routing, so a separate reset/model patch could create duplicate resolution paths or unstable upgrade behavior.
