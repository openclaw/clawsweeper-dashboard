## Summary

Keep open: current main and v2026.6.8 cover the `compaction.model` and `memoryFlush.model` cost controls, but the requested successful `memoryFlush.resetAfterFlush` mode is still absent and remains a session-lifecycle product decision.

## What This Changes

The issue requests configurable cheaper models for compaction and memory flush, plus an optional flush-then-reset mode that skips compaction after durable memory flush.

## Best Possible Solution

Keep the shipped model overrides and design reset-after-flush only as an explicit default-off lifecycle contract coordinated with reset-memory work, with tests for active runs, queues, subagents, transcript archival, and provider/auth routing.

## Reproduction Assessment

Not applicable as a feature request, but the remaining gap is source-reproducible: current main and v2026.6.8 lack a `resetAfterFlush` schema key, docs entry, and successful memory-flush reset branch.

## Solution Assessment

Unclear until maintainers define the lifecycle contract. The existing shipped model overrides are the narrow cost-relief path, while reset-after-flush needs explicit state, safety, and upgrade boundaries.
