## Summary

Keep open. Current main implements the two model-override pieces, but the highest-impact `memoryFlush.resetAfterFlush` mode is still absent and needs a product decision around session continuity before implementation.

## What This Changes

The issue requests configurable cheaper models for compaction and memory flush, plus an optional flush-then-reset mode that skips compaction after durable memory flush.

## Best Possible Solution

Keep the shipped model overrides, then decide whether an explicit opt-in reset-after-flush policy belongs in core compaction/session lifecycle with safeguards and documentation.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken existing behavior. Source inspection gives high confidence that the remaining requested `resetAfterFlush` mode is absent from current main.

## Solution Assessment

No, current main is only a partial solution to the issue as written. The remaining reset-after-flush behavior needs maintainer design judgment before a narrow implementation would be safe.
