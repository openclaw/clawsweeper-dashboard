## Summary

Keep open. Current main covers the model-cost parts of the request with `compaction.model` and `compaction.memoryFlush.model`, but it still has no `resetAfterFlush`/flush-then-reset config surface or successful-flush runtime branch that skips compaction and rotates the session.

## What This Changes

Request configurable compaction and memory-flush housekeeping model overrides plus an opt-in mode that flushes durable memory and resets the session instead of compacting.

## Best Possible Solution

Add a maintainer-approved reset-after-flush policy with explicit config, schema, docs, and tests. It should reset only after a verified successful memory flush, skip redundant compaction only when configured, and reuse existing session-reset machinery with coverage for metadata, queues, subagents/ACP, and delivery behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. There is a high-confidence static verification path: the current schema and runtime contain the model override pieces but no reset-after-flush field or successful-flush reset branch.

## Solution Assessment

Unclear until maintainers approve lifecycle semantics. The narrow maintainable direction is a config-gated policy built on existing reset/session replay code, not an unconditional reset after every memory flush.
