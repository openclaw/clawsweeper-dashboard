## Summary

Keep open: current main and v2026.6.8 already cover the `compaction.model` and `memoryFlush.model` cost-control pieces, but the requested successful `memoryFlush.resetAfterFlush` mode is still absent and remains a session-lifecycle product decision.

## What This Changes

The issue requests configurable cheaper housekeeping models for compaction and memory flush, plus a mode that flushes durable memory and resets instead of compacting.

## Best Possible Solution

Keep the shipped model overrides, and design flush-then-reset only as an explicit default-off lifecycle contract coordinated with the reset-memory work, with tests for active runs, queues, subagents, transcript archival, and provider/auth routing.

## Reproduction Assessment

Not applicable as a feature request. Source inspection is high-confidence: current main has `compaction.model` and `memoryFlush.model`, while the strict schema and runtime have no `resetAfterFlush` or equivalent successful flush-then-reset path.

## Solution Assessment

Unclear until maintainers define the lifecycle contract. The shipped model overrides are the narrow cost-relief path, but automatic reset after memory flush needs explicit safety, state, and upgrade boundaries before implementation.
