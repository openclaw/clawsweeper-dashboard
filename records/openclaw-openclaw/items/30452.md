## Summary

Keep open: current main and the latest release cover the compaction and memory-flush model override cost controls, but the requested successful flush-then-reset mode is still absent and remains a product-sensitive session lifecycle change.

## What This Changes

The issue requests configurable compaction and memory-flush model overrides plus a `memoryFlush.resetAfterFlush` mode that flushes durable memory and resets instead of compacting.

## Best Possible Solution

Keep the shipped model overrides, then design reset-after-flush as an explicit default-off lifecycle contract coordinated with reset-memory work and covered by tests for active runs, queues, subagents, transcript archival, and provider/auth routing.

## Reproduction Assessment

Not applicable as a bug reproduction, but the remaining gap is source-reproducible: current main and v2026.6.8 lack a `resetAfterFlush` schema key, docs entry, and successful memory-flush reset branch.

## Solution Assessment
