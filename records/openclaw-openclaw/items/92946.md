## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Align legacy preview accounting with visible text:** `extensions/telegram/src/draft-stream.ts:139`
  - body: When rich previews fall back, `capTelegramLegacyTransportPreview` sends at most the first legacy-sized chunk, but the caller records the full requested text as delivered after success. A long reply under the rich limit can therefore make finalization skip the suffix; record only the visible slice or continue sending the remainder.
  - confidence: 0.94
- **[P2] Require method-not-found text before rich fallback:** `extensions/telegram/src/network-errors.ts:295`
  - body: This regex arm matches `Call to 'sendRichMessage' failed! (400:` before requiring method-not-found wording, so ordinary rich validation errors can be retried through legacy delivery as if the API method were missing. Restrict fallback to actual missing/unknown rich-method responses and add negative tests for non-method 400s.
  - confidence: 0.9
- **[P2] Define long rich-edit fallback behavior:** `extensions/telegram/src/send.ts:1682`
  - body: When rich edit is unavailable, the fallback sends the full rich-sized text through legacy `editMessageText`. Long rich edits can still fail on legacy Bot API roots unless the PR scopes edits to short text, truncates intentionally, or switches to a follow-up send path.
  - confidence: 0.86
