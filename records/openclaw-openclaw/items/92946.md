## Summary

Keep open: this PR addresses a real Telegram fallback gap that current main does not cover, but it is not merge-ready because the legacy fallback can still send rich-sized chunks through `sendMessage` and the PR has no real Telegram proof.

## What This Changes

Adds Telegram rich-method-unavailable fallbacks from raw rich send/edit/draft delivery to legacy `sendMessage`/`editMessageText`, with unit coverage for short-message fallback cases.

## Best Possible Solution

Keep the plugin-owned fallback, but re-plan rich-unavailable sends at legacy Telegram limits and require redacted live proof against a no-rich API root before merge.

## Reproduction Assessment

Yes for the patch defect: a no-rich Telegram API plus a reply over 4096 characters follows the new fallback into legacy `sendMessage` with a rich-sized chunk. I did not run a live Telegram endpoint, but the source path and repository Telegram limit note make the failure path clear.

## Solution Assessment

No. The fallback belongs in the Telegram plugin, but the current implementation needs legacy-limit chunking and real no-rich endpoint proof before it is the best maintainable fix.
