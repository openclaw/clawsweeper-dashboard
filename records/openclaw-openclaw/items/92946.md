- `merge-risk: 🚨 message-delivery`: Long messages can still fail in the fallback path, leaving visible Telegram delivery broken when rich methods are unavailable.

## Summary

Keep open: the compatibility direction is useful, but the fallback still sends rich-sized chunks through legacy Telegram methods that have a 4096-character text limit, and this external PR has no real behavior proof.

## What This Changes

The PR adds fallback handling for Telegram rich-message send, edit, and preview failures by retrying legacy `sendMessage` and `editMessageText` paths with unit coverage for unsupported rich methods.

## Best Possible Solution

Keep the rich-method fallback, but make every legacy fallback path re-chunk or cap text to the legacy Telegram 4096-character contract before calling `sendMessage` or `editMessageText`, then add focused long-message regression coverage and real API-root proof.

## Reproduction Assessment

Yes from source inspection: the PR sends rich-sized chunks through legacy Telegram methods, while @grammyjs/types documents a 4096-character text limit for those methods. I did not run a live custom `apiRoot` reproduction in this read-only review.

## Solution Assessment

No as-is. The fallback is the right owner-boundary direction, but the best fix must apply legacy-size chunking across send, delivery, and preview fallback paths before it can be merge-ready.
