## Summary

Keep open: the current head repairs parts of the earlier fallback work, but it still over-accounts capped legacy stream previews and the real Telegram proof is for an older short-message head, not the current long-fallback behavior.

## What This Changes

The PR adds Telegram rich Bot API fallbacks to legacy `sendMessage` and `editMessageText` paths, plus tests for method-unavailable errors and legacy text chunking.

## Best Possible Solution

Land a Telegram-plugin fallback that retries only true rich-method gaps, chunks legacy sends, does not over-account capped previews, handles the legacy edit limit deliberately, and includes current-head redacted Telegram proof.

## Reproduction Assessment

Yes, source-level. A Telegram setup with `richMessages: true` and a Bot API root that rejects `sendRichMessage` exercises the new fallback, and current-head source shows the long-preview suffix-loss path without needing a live token.

## Solution Assessment

No as-is. The plugin-local fallback is the right layer, but the current implementation must preserve suffix delivery after capped legacy previews and provide current-head real Telegram proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve suffix after capped legacy previews:** `extensions/telegram/src/draft-stream.ts:268`
  - body: When rich previews are unavailable, `capTelegramLegacyTransportPreview` can send only the first 4096 characters through legacy `sendMessage`/`editMessageText`, but the successful send still causes `lastDeliveredText` to be set to the full requested text. `stop()` then sees the final text as already delivered and can skip sending the suffix, so long streamed replies can be visibly truncated on non-rich Bot API roots.
  - confidence: 0.94
- **[P2] Bound the legacy edit fallback text:** `extensions/telegram/src/send.ts:1682`
  - body: The rich edit fallback now retries through `performLegacyTextEdit()`, which sends the full `htmlText` or `plainText` through legacy `editMessageText`. Telegram's non-rich edit text is limited to 4096 characters, so a long rich edit on a lagging Bot API root still fails instead of preserving delivery or choosing an explicit bounded behavior.
  - confidence: 0.86
