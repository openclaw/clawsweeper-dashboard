- **[P1] Keep preview accounting aligned with legacy caps:** `extensions/telegram/src/draft-stream.ts:139`
  - body: When rich previews fall back to legacy delivery, this cap can send or edit only the first 4096-character legacy chunk, but the stream later records the full requested text as delivered. A long preview under the rich limit can therefore stop with the suffix never sent; record only the visible capped text or force finalization to send the remaining suffix.
  - confidence: 0.94

  - body: This pattern matches `Call to 'sendRichMessage' failed! (400:` before requiring method-not-found text, so ordinary rich validation errors can be retried through legacy delivery as if the API method were missing. Restrict it to actual missing/unknown rich-method responses and add negative tests for non-method 400s.
  - confidence: 0.9
- **[P2] Define the long rich-edit fallback path:** `extensions/telegram/src/send.ts:1682`
  - body: When rich edit is unavailable, this fallback sends the full rich-sized text through legacy `editMessageText`, but Telegram's legacy text edit path is limited to 4096 characters after entity parsing. Long rich edits can still fail unless the PR scopes edits to short text, truncates intentionally, or switches to a follow-up send path.
  - confidence: 0.86
