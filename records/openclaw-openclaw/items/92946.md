## Summary

Keep open: the Telegram fallback is useful and belongs in the plugin boundary, but current head still has long-message delivery defects and the supplied real Telegram proof predates the force-pushed head that added the long fallback changes.

## What This Changes

The PR changes Telegram rich send/edit/stream-preview paths to fall back to legacy sendMessage/editMessageText when rich Bot API methods are unavailable, with legacy chunking coverage.

## Best Possible Solution

Keep the Telegram-plugin fallback, but track only actually delivered legacy preview text, safely handle long legacy edits, and refresh current-head real Telegram proof before merge.

## Reproduction Assessment

Yes. Source inspection at PR head shows a non-rich Bot API plus a legacy preview over 4096 characters sends a capped preview while marking the full text delivered; long rich edit fallback also calls legacy editMessageText without a legacy limit check.

## Solution Assessment

No as-is. The fallback belongs in the Telegram plugin boundary, but the best fix must preserve only actually delivered preview text, handle long legacy edits, and refresh real proof for the current head.

Full review comments:

- **[P1] Preserve unsent suffix after capped legacy previews:** `extensions/telegram/src/draft-stream.ts:187-191`
  - body: When rich previews are unavailable, this sends only the capped legacy preview, but the stream later records the full trimmed text as delivered and stop() skips finalization when the final text matches. A long preview can therefore deliver only the first legacy-sized slice and never send the suffix.
  - confidence: 0.92
- **[P2] Limit legacy editMessageText fallback to 4096 characters:** `extensions/telegram/src/send.ts:1516-1522`
  - body: The rich-edit fallback still passes the full HTML/plain text to legacy editMessageText. Telegram's text edit contract is 1-4096 characters when rich_message is not used, so a long edit on a non-rich API root can fail instead of preserving delivery.
  - confidence: 0.86
