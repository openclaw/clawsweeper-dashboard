Full review comments:

- **[P1] Keep preview accounting aligned with legacy preview payload:** `extensions/telegram/src/draft-stream.ts:127-143`
  - body: When rich previews fall back to legacy delivery, this helper caps the actual `sendMessage`/`editMessageText` payload to 4096 characters, but a successful send later records the full requested text as delivered. A long preview can therefore leave the suffix unsent and skip finalization because the stream believes the whole text is already visible.
  - confidence: 0.92
- **[P2] Narrow rich-method-unavailable matching:** `extensions/telegram/src/network-errors.ts:295`
  - body: This pattern matches `Call to 'sendRichMessage' failed! (400:` before requiring method-not-found text, so ordinary rich validation errors can be retried through legacy delivery as if the API method were missing. Restrict it to actual missing/unknown rich-method responses and add negative tests for non-method 400s.
  - confidence: 0.9
- **[P2] Define the long rich edit fallback policy:** `extensions/telegram/src/send.ts:1682`
  - body: When rich edit is unavailable, this fallback sends the full rich-sized text through legacy `editMessageText`, but Telegram's legacy text edit path is limited to 4096 characters after entity parsing. Long rich edits can still fail unless the PR scopes edits to short text, truncates intentionally, or switches to a follow-up send path.
  - confidence: 0.86
