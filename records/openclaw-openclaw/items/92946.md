## Summary

Keep this PR open: current main and the latest release still lack a legacy fallback when Telegram rich methods are unavailable, but this head is not merge-ready because long preview accounting can suppress visible text, the error classifier is too broad, long rich-edit fallback is undefined, and the real Telegram proof is stale for the current head.

## What This Changes

The PR adds Telegram rich-method-unavailable fallbacks from rich sends, edits, and draft previews to legacy sendMessage/editMessageText, with legacy text-limit chunk handling and tests.

## Best Possible Solution

Land one Telegram-plugin fallback that narrowly detects missing rich methods, keeps visible and recorded preview state aligned, defines legacy edit-limit behavior, and includes redacted current-head Telegram proof.

## Reproduction Assessment

Yes, at source level: enable richMessages and use a Bot API root that rejects sendRichMessage, and current main has no legacy fallback in the normal send, bot delivery, or preview paths. I did not run live Telegram in this read-only review.

## Solution Assessment

No as submitted. The Telegram plugin is the right boundary, but this head needs narrower error classification, corrected long-preview accounting, defined long-edit behavior, and current-head real Telegram proof before it is the best fix.
