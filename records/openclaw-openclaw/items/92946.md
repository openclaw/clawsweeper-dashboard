## Summary

Keep open: the fallback direction is useful, but current head can mark a capped legacy Telegram preview as fully delivered, still sends long rich edits through legacy editMessageText, and has no real Telegram proof yet.

## What This Changes

The PR adds Telegram fallbacks from rich Bot API send, edit, and preview failures to legacy sendMessage/editMessageText paths, with legacy chunking tests for final sends.

## Best Possible Solution

Keep the fallback approach, but make legacy preview/edit paths deliver or record only text actually accepted by legacy Telegram methods, preserve long final text without dropping suffixes, and add redacted real Telegram proof before merge.

## Reproduction Assessment

Yes from source inspection: simulate a non-rich Bot API by making raw rich preview fail, stream more than 4096 characters, then stop; the fallback sends capped text while lastDeliveredText records the full text. I did not run a live Telegram root in this read-only review.

## Solution Assessment

No as-is. The fallback belongs in the Telegram plugin boundary, but it must preserve long final text and long edits under the legacy 4096-character contract before it is the best fix.
