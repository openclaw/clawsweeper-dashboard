## Summary

Keep open. Current main still has no simple-completion path that dispatches `claude-cli` or `cli://claude` to the Claude CLI backend, and it also lacks a typed unsupported-runtime guard, so the reported `Connection error` remains actionable rather than implemented or obsolete.

## What This Changes

Request that Lossless/pi-ai simple completion either route `claude-cli`/`cli://claude` through OpenClaw’s Claude CLI transport or fail with a specific unsupported transport error.

## Best Possible Solution

Define the simple-completion contract for CLI runtime models: either add first-class one-shot Claude CLI completion through the existing CLI backend layer, or reject CLI runtime/baseUrl inputs with a typed unsupported-runtime error plus docs guidance for plugin authors.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue provides a concrete harness and controls, and current-main source inspection shows the same missing route/guard that would leave `cli://claude` on the pi-ai simple-completion path.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Passing `cli://claude` through pi-ai unchanged is not maintainable; a maintainer should choose explicit CLI backend support or a deliberate unsupported-runtime error before implementation.
