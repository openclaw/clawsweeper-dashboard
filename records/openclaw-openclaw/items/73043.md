## Summary

Keep open. Current main still does not route plugin simple-completion calls for `claude-cli`/`cli://claude` through the Claude CLI backend, and it also does not fail early with a typed unsupported-runtime error, so the reported behavior remains actionable.

## What This Changes

The issue asks OpenClaw to make Lossless/pi-ai simple completions using the synthetic `claude-cli` Sonnet transport either work through the Claude CLI backend or fail with a clear transport/provider error.

## Best Possible Solution

Define the simple-completion contract for CLI runtime models in the plugin-SDK/runtime path: either add a first-class Claude CLI one-shot helper that preserves CLI execution semantics, or detect CLI runtime/baseUrl inputs and fail closed with a typed unsupported-runtime error plus user-facing docs guidance.

## Reproduction Assessment

Yes. The issue provides a concrete standalone harness, and current-main source inspection shows the simple-completion path still has no Claude CLI backend branch or explicit guard for `claude-cli`/`cli://claude`.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai is not the maintainable solution; a maintainer should choose between explicit CLI one-shot support and a deliberate unsupported-runtime error before implementation.
