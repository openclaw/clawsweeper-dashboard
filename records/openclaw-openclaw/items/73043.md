## Summary

Keep open. Current `main` still sends prepared simple-completion models into pi-ai without a Claude CLI backend dispatch or a clear unsupported-runtime guard, so the reported `cli://claude` failure remains actionable.

## What This Changes

The issue asks for Lossless/pi-ai simple completion to work with, or clearly reject, OpenClaw’s synthetic `claude-cli/claude-sonnet-4-6` transport.

## Best Possible Solution

Define the simple-completion contract for CLI runtime models: either route one-shot text completions through the registered CLI backend or reject `cli://` runtime models before pi-ai receives them.

## Reproduction Assessment

Yes. The issue provides a concrete harness and controls, and current source plus the locked pi-ai package show `cli://claude` still has no simple-completion CLI dispatch or guard.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai is not the right long-term behavior, but maintainers should choose first-class CLI simple-completion support versus an explicit unsupported-runtime error before implementation.
