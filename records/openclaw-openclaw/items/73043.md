## Summary

Keep open. Current main still prepares the Claude CLI model for simple completion and then calls pi-ai without a Claude CLI dispatch path or a clear unsupported-runtime guard, so the reported `cli://claude` path can still fail as a generic connection error.

## What This Changes

Issue requests that Lossless/pi-ai simple completion either support OpenClaw's synthetic Claude CLI Sonnet transport or fail with a clear provider/transport error.

## Best Possible Solution

Define the SDK simple-completion contract for CLI runtimes, then either add a Claude CLI one-shot path or fail fast with a specific unsupported-runtime error covered by focused regression tests.

## Reproduction Assessment

Yes, by source inspection rather than a local live run. The reported harness targets `completeSimple` with `claude-cli/claude-sonnet-4-6`, and current main still has no simple-completion CLI dispatch or guard before calling pi-ai.

## Solution Assessment

Unclear. Passing `cli://claude` through the generic pi-ai simple-completion path is not the right end state, but maintainers should choose whether the contract is full CLI support or a clear unsupported-runtime failure.
