## Summary

Keep open. Current main still sends the SDK simple-completion path through pi-ai without dispatching `claude-cli` / `cli://claude` models to the Claude CLI backend or failing early with a typed unsupported-runtime error.

## What This Changes

Request that Lossless/pi-ai simple completions using OpenClaw's synthetic Claude CLI Sonnet model either run through the Claude CLI transport or fail with a specific unsupported-transport error.

## Best Possible Solution

Define the plugin-SDK simple-completion contract for CLI runtime models, then implement either a one-shot Claude CLI backend path or a clear unsupported-runtime failure with docs guidance.

## Reproduction Assessment

Yes. The issue provides a concrete macOS harness and controls, and current-main source inspection shows the same missing CLI backend route or guard; live verification was not feasible on this Linux host without a `claude` binary.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai unchanged is not the best solution, but maintainers should choose between explicit CLI support and deliberate fail-fast unsupported-runtime behavior before implementation.
