## Summary

Keep open. Current main still leaves the simple-completion path to delegate to pi-ai without routing `claude-cli` / `cli://claude` through OpenClaw’s CLI backend or failing early with a typed unsupported-runtime error.

## What This Changes

The issue asks for Lossless/pi-ai simple completions using OpenClaw’s synthetic Claude CLI Sonnet model to either work through the CLI transport or fail with a specific unsupported-transport error.

## Best Possible Solution

Define the plugin-SDK simple-completion contract for CLI runtime models, then implement either a one-shot Claude CLI backend path or a clear unsupported-runtime failure with docs guidance.

## Reproduction Assessment

Yes. The issue provides a concrete harness and controls, and current source plus the pinned pi-ai dependency show the reported `cli://claude` model is still handed to an HTTP-style Anthropic simple-completion provider.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai unchanged is not maintainable; a maintainer should choose between explicit one-shot CLI support and deliberate fail-fast unsupported-runtime behavior before implementation.
