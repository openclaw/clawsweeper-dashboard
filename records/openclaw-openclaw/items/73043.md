## Summary

Keep open. Current main still sends the prepared `claude-cli` simple-completion model into pi-ai’s Anthropic simple stream without a Claude CLI dispatch path or a clear unsupported-runtime guard, so the reported generic transport failure remains plausible on the current source path.

## What This Changes

The issue asks OpenClaw to either make Lossless/pi-ai simple completion work with the synthetic Claude CLI Sonnet transport or fail with a specific unsupported-transport error instead of `Connection error`.

## Best Possible Solution

Define the SDK simple-completion contract for CLI runtimes, then either add a Claude CLI one-shot path or reject CLI runtime models with a specific unsupported-runtime error and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live macOS run. The reported model still reaches pi-ai simple completion without a Claude CLI dispatch path or unsupported-runtime guard, while pi-ai’s Anthropic stream uses `model.baseUrl` as an HTTP SDK base URL.

## Solution Assessment

Unclear. Passing `cli://claude` through the generic pi-ai Anthropic path is not maintainable, but maintainers should choose whether the contract is real CLI simple-completion support or a clear unsupported-runtime failure.
