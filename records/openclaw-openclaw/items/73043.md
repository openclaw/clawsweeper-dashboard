## Summary

Keep open. Current main still lacks a Claude CLI dispatch or explicit unsupported-runtime guard in the plugin simple-completion path, while the docs and Anthropic plugin code treat `claude-cli` as a CLI backend/runtime rather than an HTTP-style pi-ai provider transport.

## What This Changes

The issue asks OpenClaw to make Lossless/pi-ai simple completion either work with the synthetic Claude CLI Sonnet path or fail with a specific unsupported-transport error.

## Best Possible Solution

Define the plugin simple-completion contract for CLI runtime models: either add a first-class, non-mutating Claude CLI one-shot helper, or detect `claude-cli`/`cli://claude` inputs and return a typed unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue provides a concrete harness, and current-main static inspection shows the same model shape still flows into pi-ai completion without a Claude CLI backend branch or explicit guard.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai is not the maintainable fix; maintainers should choose between a supported OpenClaw CLI runtime helper and a deliberate fail-closed error.
