## Summary

Keep open. Current main still leaves `claude-cli` / `cli://claude` outside the simple-completion transport path, so the reported Lossless/pi-ai failure is source-reproducible and needs a maintainer contract decision rather than cleanup closure.

## What This Changes

Request that Lossless/pi-ai simple completions using `claude-cli/claude-sonnet-4-6` either route through OpenClaw's Claude CLI backend or fail with a specific unsupported-transport error.

## Best Possible Solution

Define the simple-completion contract for CLI runtimes and implement either a Claude CLI one-shot bridge or a typed unsupported-runtime error with focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live macOS run. The reported `claude-cli/claude-sonnet-4-6` model still reaches pi-ai's Anthropic simple provider with `baseUrl: "cli://claude"` because current main has no CLI dispatch or fail-fast guard in the simple-completion preparation path.

## Solution Assessment

Unclear. Letting `cli://claude` fall through to pi-ai's Anthropic HTTP provider is not the right end state, but maintainers need to choose whether OpenClaw should support CLI-backed simple completions or reject them with a clear unsupported-runtime error.
