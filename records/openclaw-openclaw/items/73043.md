## Summary

Keep open. Current main still sends `claude-cli` / `cli://claude` simple-completion models through pi-ai without a Claude CLI backend route or a typed unsupported-runtime guard, so the reported `Connection error` remains actionable.

## What This Changes

The issue asks OpenClaw to support or clearly reject Lossless/pi-ai simple completions for the synthetic `claude-cli` Sonnet transport.

## Best Possible Solution

Define the SDK simple-completion contract for CLI runtime models, then either add first-class one-shot Claude CLI completion through the CLI backend layer or return a typed unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete harness and passing controls, and current-main source inspection shows the same missing CLI backend route or fail-fast guard.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai unchanged is not the right end state, but a maintainer should choose between support and explicit rejection before implementation.
