## Summary

Keep open. Current main still sends the simple-completion model into the pi-ai completion path without a Claude CLI backend dispatch or an early typed unsupported-runtime error, while the docs and plugin registration treat `claude-cli` as a separate CLI runtime/backend rather than an HTTP-style `baseUrl` transport.

## What This Changes

The issue requests that Lossless/pi-ai simple completions using OpenClaw's synthetic `claude-cli` Sonnet path either route through the Claude CLI transport or fail with a specific unsupported-provider/runtime error.

## Best Possible Solution

Define the plugin-SDK simple-completion contract for CLI runtime models: either add a first-class non-mutating Claude CLI one-shot helper that plugins can call, or detect CLI runtime/baseUrl inputs and return a typed unsupported-runtime error with docs guidance for summarizer plugins.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and current-main static inspection shows the same model shape still flows through pi-ai `complete(...)` without Claude CLI dispatch or an early unsupported-runtime guard.

## Solution Assessment

Unclear. Letting `cli://claude` pass through pi-ai is not the maintainable fix; the safer path is an explicit OpenClaw runtime contract, either implemented support or deliberate fail-closed rejection.
