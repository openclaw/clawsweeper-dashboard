## Summary

Keep open. Current main still leaves Claude CLI as a runtime/backend separate from the pi-ai simple-completion path, so the reported `cli://claude` / `completeSimple` failure remains a real contract gap rather than cleanup.

## What This Changes

The issue requests that Lossless/pi-ai simple completion either support OpenClaw's synthetic Claude CLI transport or fail with a specific unsupported-transport error.

## Best Possible Solution

Define and document the SDK simple-completion contract for CLI runtime models, with focused tests covering either first-class Claude CLI one-shot completion or a clear unsupported-runtime error.

## Reproduction Assessment

Yes. The issue provides a concrete harness and controls, and current-main source inspection shows the same missing CLI backend dispatch or guard; this host cannot run the macOS Claude CLI control live.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai unchanged is not the best path; maintainers should choose explicit CLI backend support or a typed unsupported-runtime contract before implementation.
