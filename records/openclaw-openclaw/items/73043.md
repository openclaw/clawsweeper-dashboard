## Summary

Keep open. Current main still lacks a simple-completion path that either routes `claude-cli`/`cli://claude` through the Claude CLI backend or fails early with a typed unsupported-runtime error, so the reported Lossless/pi-ai behavior remains actionable.

## What This Changes

The issue requests that Lossless/pi-ai simple completion for `claude-cli/claude-sonnet-4-6` either use OpenClaw's Claude CLI transport or return a specific unsupported transport/provider error.

## Best Possible Solution

Define the SDK simple-completion contract for CLI runtime models: either add first-class one-shot Claude CLI completion through the OpenClaw CLI backend layer, or detect CLI runtime/baseUrl inputs and return a typed unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness and controls, and current-main source inspection shows the same missing route/guard; live macOS verification was not feasible on this Linux host without `claude`.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai unchanged is not maintainable; a maintainer should choose between explicit CLI one-shot support and deliberate unsupported-runtime rejection before implementation.
