## Summary

Keep open. Current main still has the same source-level gap: Claude CLI auth and the CLI backend exist, but the simple-completion path still delegates to pi-ai without a `claude-cli` dispatch or explicit unsupported-runtime guard.

## What This Changes

Lossless/pi-ai simple completion should either work with OpenClaw's `claude-cli` / `cli://claude` transport or fail with a specific unsupported-transport error instead of a generic `Connection error`.

## Best Possible Solution

Define the SDK simple-completion contract for CLI runtimes, then add focused coverage for either Claude CLI one-shot completion or a clear unsupported-runtime error that plugin callers can surface.

## Reproduction Assessment

Yes. The issue provides a concrete macOS harness with passing controls, and current-main source inspection confirms the simple-completion path still has no Claude CLI dispatch or guard.

## Solution Assessment

Unclear. Letting `cli://claude` fall through pi-ai unchanged is not the best solution; the maintainable fix is either an owned CLI backend path or a deliberate fail-fast contract.
