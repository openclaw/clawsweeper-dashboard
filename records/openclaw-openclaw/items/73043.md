## Summary

Keep open. Current main still passes `claude-cli` / `cli://claude` simple-completion models into pi-ai without dispatching to the Claude CLI backend or failing early with a typed unsupported-transport error, so the reported `Connection error` path remains actionable.

## What This Changes

Request that Lossless/pi-ai simple completions using OpenClaw's synthetic `claude-cli/claude-sonnet-4-6` path either run through Claude CLI or fail with a specific unsupported-transport error.

## Best Possible Solution

Define the SDK simple-completion contract for CLI runtime models: either add first-class one-shot Claude CLI completion through the CLI backend layer, or detect CLI runtime/baseUrl inputs and return a typed unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete harness and passing controls, and current-main source inspection shows the same missing CLI backend route or guard; live macOS verification was not run because this Linux host has no `claude` binary or relevant Claude/OpenClaw env markers.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai unchanged is not maintainable; a maintainer should choose between explicit one-shot CLI support and deliberate fail-fast unsupported-runtime behavior before implementation.
