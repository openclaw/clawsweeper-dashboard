## Summary

Keep open. Current main still has no simple-completion path that maps `claude-cli`/`cli://claude` onto OpenClaw's Claude CLI backend, and it also does not fail early with a specific unsupported-runtime error for that combination.

## What This Changes

Make Lossless/pi-ai simple completion either work with OpenClaw's synthetic `claude-cli` Sonnet transport or return a specific unsupported-transport error.

## Best Possible Solution

Define the simple-completion contract for CLI runtime models, then implement either a first-class, non-mutating Claude CLI one-shot helper for plugin/context-engine use or a clear unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue provides a concrete standalone harness, and current-main source inspection confirms that the model would pass through pi-ai simple completion rather than OpenClaw's registered Claude CLI backend.

## Solution Assessment

Unclear. Passing `cli://claude` through raw pi-ai is not the maintainable contract; the safer direction is an explicit OpenClaw simple-completion runtime seam or an early unsupported-runtime failure.
