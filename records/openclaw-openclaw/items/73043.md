## Summary

Keep open. Current main still lacks a CLI-runtime simple-completion contract for `claude-cli` / `cli://claude`, and source plus upstream pi-ai dispatch show the reported fall-through remains plausible rather than implemented or obsolete.

## What This Changes

Request that OpenClaw's simple-completion/SDK path either route `claude-cli` Sonnet through the Claude CLI backend or fail with a clear unsupported-runtime error instead of pi-ai's generic connection error.

## Best Possible Solution

Define a plugin-SDK/simple-completion contract for CLI runtimes, then add focused coverage for either Claude CLI one-shot dispatch or a clear unsupported-runtime error that plugin callers can surface.

## Reproduction Assessment

Yes. The issue gives a complete macOS harness and passing controls; I could not run Claude CLI here, but current main deterministically has no `claude-cli` dispatch or guard before pi-ai.

## Solution Assessment

Unclear. Supporting direct pi-ai `cli://claude` is not the narrowest path because pi-ai dispatches by API only; maintainers should either bridge through OpenClaw's CLI backend or reject synthetic CLI providers before pi-ai with a typed error.
