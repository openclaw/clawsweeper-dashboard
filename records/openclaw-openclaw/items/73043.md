## Summary

Keep open. The prior ClawSweeper comment correctly identified this as still actionable, and current main still has no path that routes simple-completion calls for `claude-cli`/`cli://claude` through the Claude CLI backend or rejects them with a typed unsupported-runtime error.

## What This Changes

The issue asks for Lossless/pi-ai `completeSimple` to either support OpenClaw's `claude-cli`/`cli://claude` Sonnet transport or fail with a specific unsupported-transport error.

## Best Possible Solution

Define the simple-completion contract for CLI runtime models: either add first-class one-shot Claude CLI completion support through the OpenClaw plugin-SDK/runtime path, or detect CLI runtime/baseUrl inputs and return a typed unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and current-main source inspection shows the simple-completion path still lacks both Claude CLI backend routing and an explicit guard for `claude-cli`/`cli://claude`.

## Solution Assessment

Unclear. Passing `cli://claude` through pi-ai is not the maintainable solution; a maintainer should choose between explicit CLI one-shot support and deliberate unsupported-runtime rejection before implementation.
