## Summary

Keep this PR open. The metadata-aware heartbeat filtering direction is useful, but the latest head has a compile-blocking undefined symbol, still drops the effective prompt on split misses such as custom heartbeat prompts, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds metadata-aware heartbeat message filtering, a metadata-prefixed heartbeat filter test, and a heartbeat-specific runtime-context fallback for transcript marker/effective-prompt mismatches.

## Best Possible Solution

Preserve the compact heartbeat transcript marker, keep metadata-aware pair filtering, pass an explicit heartbeat signal or marker-based fallback that preserves the full effective body for default and custom heartbeat prompts, and require real heartbeat/channel proof before merge.

## Reproduction Assessment

Yes, at source level: current main uses the heartbeat transcript marker while effectivePrompt carries the full body, and current filtering misses metadata-prefixed heartbeat text. The PR findings are also source-reproducible from the raw head file: an undefined symbol is returned and split misses drop runtimeContext.

## Solution Assessment

No. The metadata stripping direction is narrow, but the runtime-context change must compile and must preserve the effective prompt for custom heartbeat prompts instead of recognizing only the default prompt text.
