## Summary

Keep open: the latest head resolves the earlier missing-import blocker and the metadata-aware filter direction looks appropriate, but the runtime-context fallback still drops custom heartbeat prompts and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR strips inbound metadata before heartbeat pair matching and adds a heartbeat-specific runtime-context fallback for transcript-marker/effective-prompt mismatches.

## Best Possible Solution

Preserve the compact heartbeat transcript marker, keep metadata-aware pair filtering, and carry the full effective heartbeat body for every configured heartbeat prompt with focused regression coverage and real runtime proof.

## Reproduction Assessment

Yes, at source level: current main uses a compact heartbeat transcript marker while the effective prompt carries the full body, and the PR head only preserves runtimeContext for default-prompt-shaped heartbeats. The custom-prompt failure follows directly from the documented configurable heartbeat prompt contract.

## Solution Assessment

No. The metadata-aware filter is narrow and useful, but the runtime fallback should identify heartbeat marker turns independent of the default prompt text and should include custom heartbeat prompt coverage.
