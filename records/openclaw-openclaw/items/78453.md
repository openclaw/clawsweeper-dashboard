## Summary

Keep open: the metadata-aware heartbeat filter is useful, but the PR still drops custom heartbeat prompt bodies in the split-marker runtime-context path and lacks required real behavior proof.

## What This Changes

The PR adds metadata-aware heartbeat prompt filtering and a heartbeat-specific runtime-context fallback when the compact transcript marker is not present in the effective prompt.

## Best Possible Solution

Preserve the stable heartbeat transcript marker and metadata-aware filtering, but make the runtime-context fallback use an explicit heartbeat signal or marker-based contract that also covers custom heartbeat prompts.

## Reproduction Assessment

Yes, at source level: with transcriptPrompt `[OpenClaw heartbeat poll]` and a custom heartbeat prompt as effectivePrompt, the PR-head detector leaves `isHeartbeatTurn` false and returns only the marker. I did not run a live heartbeat in this read-only review.

## Solution Assessment

No. The metadata-aware filter is narrow and useful, but the runtime fallback should not depend on the default prompt phrase when docs allow custom heartbeat prompt bodies.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Preserve custom heartbeat prompts when the marker is split:** `src/agents/pi-embedded-runner/run/runtime-context-prompt.ts:100-102`
