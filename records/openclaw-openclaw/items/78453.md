## Summary

Keep open: the PR still has a correctness blocker because its heartbeat fallback only recognizes the default heartbeat prompt text, so configured custom heartbeat prompts can lose the actual heartbeat body. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds metadata-aware heartbeat pair detection and a heartbeat-specific runtime-context fallback when the compact transcript marker is split from the effective prompt.

## Best Possible Solution

Keep the compact transcript marker and metadata-aware filtering, but make the heartbeat runtime-context fallback independent of the default prompt text and cover custom heartbeat prompts against current main.

## Reproduction Assessment

Yes, at source level: with transcriptPrompt `[OpenClaw heartbeat poll]` and a configured custom heartbeat prompt as the effective prompt, the PR-head detector leaves `isHeartbeatTurn` false and returns only the marker. I did not run a live heartbeat in this read-only review.

## Solution Assessment

No. The metadata-aware filter is narrow and useful, but the runtime fallback should key off the stable heartbeat marker or explicit heartbeat context rather than the default prompt phrase, then prove the real heartbeat path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve custom heartbeat prompts when the marker is split:** `src/agents/pi-embedded-runner/run/runtime-context-prompt.ts:100-102`
