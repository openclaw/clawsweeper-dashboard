## Summary

Keep open: the PR is aimed at the right Windows gateway bug, but the current patch sets the hidden-launcher flag in the generated service environment after the Scheduled Task action has already been resolved, so fresh and upgraded tasks can still point at `gateway.cmd`. It also still lacks after-fix Windows Task Scheduler proof.

## What This Changes

Adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER=1` to the gateway service environment builder and asserts that default in the service-env test.

## Best Possible Solution

Move the hidden-launcher default into the install/scheduler decision path and add fresh/update Scheduled Task coverage plus redacted Windows install/run proof before merge.

## Reproduction Assessment

Yes at source level: current scheduler code resolves `.cmd` versus `.vbs` from the install env, while this PR places the flag only in the generated service environment. I did not run a native Windows desktop repro in this read-only review.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right direction, but this patch applies the default too late; the maintainable fix should set the default before the task action is resolved and prove fresh/update behavior.
