## Summary

Keep open: the patch is not merge-ready because it sets the hidden-launcher flag in the generated gateway service environment, while the Scheduled Task action is resolved from the installer env before that generated environment is used.

## What This Changes

Adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER=1` to the generated gateway service environment and asserts it in `service-env` tests.

## Best Possible Solution

Set the hidden-launcher default on the scheduler/install env used to resolve `taskLaunchPath`, preserve explicit opt-out, add fresh/update task-action coverage, and require redacted native Windows no-console proof.

## Reproduction Assessment

Yes, at source level: current main selects `gateway.cmd` unless the scheduler installer env has a truthy hidden-launcher flag, and the linked issue supplies Windows 11 reproduction steps. I did not run a live Windows desktop reproduction in this read-only review.

## Solution Assessment

No. The PR sets the flag in the generated command-script environment, but the Scheduled Task action is chosen earlier from the installer env; the safer fix is to move the default into the scheduler decision path and test the actual task action.
