## Summary

Keep open: this is a focused attempt at the right Windows gateway bug, but the patch writes the hidden-launcher flag into the generated service environment while the Scheduled Task action decision still reads the install env, so the task can still point at `gateway.cmd`. It also needs real Windows behavior proof before merge.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER: "1"` to the generated gateway service environment and asserts that default in the service environment test.

## Best Possible Solution

Make the actual schtasks launcher-selection path default gateway installs and updates to `gateway.vbs`, preserve an explicit false opt-out, and prove the fresh/update task action plus no-console behavior on Windows.

## Reproduction Assessment

Yes at source level: current main defaults `shouldUseHiddenWindowsTaskLauncher()` to false unless the install env has a truthy flag, and gateway install callers pass `process.env` or `serviceInstallEnv` separately from the generated service environment. The linked issue provides live Windows reporter evidence, but I did not run a native Windows desktop reproduction in this read-only review.

## Solution Assessment

No: adding the flag only to `buildServiceEnvironment()` is not the best fix because schtasks chooses the task action from the install env before the generated service environment is written into `gateway.cmd`. The narrower maintainable fix is to default the schtasks launcher decision itself to hidden, or pass the same generated launcher-selection flag into the install env, with explicit false opt-out coverage.
