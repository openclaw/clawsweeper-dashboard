## Summary

Keep open: the branch still applies the hidden-launcher flag too late, inside the generated gateway service environment, while current Windows task installation resolves the Scheduled Task action from the separate install env before that environment is written. The PR also still lacks after-fix Windows Scheduled Task install/run proof from a real setup.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER: "1"` to the gateway service environment builder and asserts that default in the service-env test.

## Best Possible Solution

Set the gateway hidden-launcher default at the scheduler input boundary, or merge it before task-action resolution, then add fresh/update Scheduled Task coverage and redacted Windows install/run proof.

## Reproduction Assessment

Yes at source level: current install callers pass `process.env` as the scheduler control env while the generated service `environment` is separate, and the scheduler resolves `.cmd` versus `.vbs` from the control env. I did not run a native Windows desktop repro in this read-only review.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right direction, but this patch applies the default after the scheduler has already chosen the task action.
