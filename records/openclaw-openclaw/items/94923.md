## Summary

Keep open: the branch targets a real Windows gateway Scheduled Task bug, but it is not merge-ready because the added default lands in the generated service script environment rather than the installer env used to choose the Scheduled Task action, and it still lacks live Windows proof.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER=1` to `buildServiceEnvironment()` and asserts that default in the daemon service-env test.

## Best Possible Solution

Set the hidden-launcher default in the scheduler/install env before `writeScheduledTaskScript()`, preserve an explicit opt-out, assert fresh and update task actions use `gateway.vbs`, and add redacted Windows proof.

## Reproduction Assessment

Yes, at source level: current main selects `gateway.cmd` unless the scheduler installer env has a truthy hidden-launcher flag, and the linked issue supplies Windows 11 reproduction steps for the visible console. I did not run a live Windows desktop reproduction in this read-only review.

## Solution Assessment

No. The PR sets the flag in the generated command-script environment, but the Scheduled Task action is chosen earlier from the installer env; the safer fix is to move the default into the scheduler decision path and test the actual task action.
