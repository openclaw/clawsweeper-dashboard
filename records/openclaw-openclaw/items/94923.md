## Summary

Keep open: the PR is a useful attempt but is not merge-ready. It adds the hidden-launcher flag only to the generated service environment, while the Windows task action is selected from the scheduler env before that environment is written, and the PR still lacks real Windows Scheduled Task proof.

## What This Changes

The branch adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER: "1"` to the gateway service environment builder and asserts it in the service-env test.

## Best Possible Solution

Set the gateway hidden-launcher default at the scheduler/install env boundary, or make action resolution use the merged install environment, then prove fresh and existing tasks use `gateway.vbs`.

## Reproduction Assessment

Yes at source level: the scheduler selects `.cmd` versus `.vbs` from the install env, while this PR changes only the generated service env. I did not run a native Windows desktop reproduction in this read-only review.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right direction, but this patch applies the default after the scheduler has already chosen the task action.
