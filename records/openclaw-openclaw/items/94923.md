## Summary

Keep open: the PR targets a valid Windows Scheduled Task visible-console bug, but the patch sets the flag in the generated script environment after the scheduler action has already been chosen, so the task can still point at `gateway.cmd`.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER=1` to the generated gateway service environment and asserts that default in the daemon service-env test.

## Best Possible Solution

Set the hidden-launcher default in the scheduler action decision path or gateway install env before `writeScheduledTaskScript()`, preserve an explicit opt-out, assert fresh/update task actions use `gateway.vbs`, and add redacted Windows proof.

## Reproduction Assessment

Yes, at source level: current main selects `gateway.cmd` unless the scheduler installer env has a truthy hidden-launcher flag, and the linked issue supplies Windows 11 reproduction steps for the visible console. I did not run a live Windows desktop reproduction in this read-only review.

## Solution Assessment

No. The PR sets the flag in the generated command-script environment, but the Scheduled Task action is chosen earlier from the installer env; the safer fix is to move the default into the scheduler decision path and test the actual task action.
