## Summary

Keep open: the patch is not merge-ready because it writes the hidden-launcher flag to the generated service environment after the Scheduled Task action path has already been resolved, so gateway tasks can still point at `gateway.cmd`; native Windows proof is also missing.

## What This Changes

Adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER=1` to `buildServiceEnvironment()` and asserts that default in the service environment test.

## Best Possible Solution

Set the hidden-launcher default on the scheduler/install env used to resolve `taskLaunchPath`, preserve any intentional opt-out, add fresh/update task-action coverage, and require redacted native Windows no-console proof.

## Reproduction Assessment

Yes. At source level, current main selects `gateway.cmd` unless the scheduler installer env has a truthy hidden-launcher flag, and the linked issue supplies Windows 11 reproduction steps; I did not run a live Windows desktop reproduction in this read-only review.

## Solution Assessment

No. The PR sets the flag in the generated command-script environment, but the Scheduled Task action is chosen earlier from the installer env; the safer fix is to move the default into the scheduler decision path and test the actual task action.
