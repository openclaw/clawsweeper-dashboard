## Summary

Keep open: the PR is focused but does not yet fix the scheduler boundary. It adds the hidden-launcher flag to the generated gateway script environment, while Windows task action resolution still reads the install/control env and can keep fresh or upgraded tasks pointed at `gateway.cmd`.

## What This Changes

Adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER: "1"` to the gateway service environment builder and one service-env test assertion.

## Best Possible Solution

Set the hidden-launcher default at the scheduler/install input boundary, or make launcher resolution default-on with explicit opt-out, then add fresh and existing task action coverage plus redacted Windows proof.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 keep the gateway Scheduled Task hidden launcher opt-in, and install callers pass `process.env` separately from the generated service environment. I did not run a native Windows desktop reproduction in this read-only review.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right direction, but this patch applies the default after the scheduler has already chosen the task action.
