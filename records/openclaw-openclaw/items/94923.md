## Summary

Keep open: the linked Windows gateway hidden-launcher bug is still real, but this PR is not sufficient because it adds the flag only to the generated service environment after the scheduler has already selected the task action path.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER="1"` to the generated gateway service environment and asserts it in the service-env test.

## Best Possible Solution

Set the hidden-launcher default at the scheduler input boundary, or resolve the task action from the merged install environment, then prove fresh and existing task actions use `gateway.vbs` with redacted Windows evidence.

## Reproduction Assessment

Yes, at source level: `writeScheduledTaskScript()` resolves `taskLaunchPath` from `args.env`, while gateway install callers pass process/install env separately from the generated service environment changed by this PR. I did not run a native Windows desktop reproduction in this read-only review.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right direction, but adding the flag only in `buildServiceEnvironment()` is too late for Scheduled Task action selection.
