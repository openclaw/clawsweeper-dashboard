## Summary

Keep open: this is a focused bug-fix PR, but the patch sets the hidden-launcher flag only in the generated gateway service environment while Scheduled Task action selection still reads the separate install/control env. The supplied proof is macOS build/source/Vitest output, not real Windows Scheduled Task behavior proof.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER: "1"` to the gateway service environment builder and asserts it in the service-env test.

## Best Possible Solution

Set the gateway hidden-launcher default at the scheduler/install boundary or make launcher resolution use the merged install environment, then add fresh/update Scheduled Task action coverage and redacted Windows proof.

## Reproduction Assessment

Yes, at source level: the scheduler chooses `.cmd` versus `.vbs` from `args.env`, while this patch only changes the generated service environment. I did not run a native Windows desktop reproduction in this read-only review.

## Solution Assessment

No. Reusing the existing VBS launcher is the right direction, but setting the flag only in `buildServiceEnvironment()` is too late for Scheduled Task action resolution.
