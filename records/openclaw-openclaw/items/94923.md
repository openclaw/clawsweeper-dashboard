## Summary

Keep open: this PR is not merge-ready because it adds the hidden-launcher flag only to the generated service environment, while the Windows Scheduled Task action is resolved earlier from the separate install/control environment. The supplied proof is also source and Vitest output only, not real Windows Scheduled Task behavior proof.

## What This Changes

The PR adds `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER="1"` to the gateway service environment builder and asserts that value in the service-env test.

## Best Possible Solution

Set the gateway hidden-launcher default at the scheduler input boundary, or resolve the action from the merged install environment, then add fresh/update Scheduled Task action coverage and redacted Windows install/run proof.

## Reproduction Assessment

Yes, at source level: current install callers pass `process.env` as the scheduler control env while the generated service environment is separate, and the scheduler chooses `.cmd` versus `.vbs` from that control env. I did not run a native Windows desktop reproduction in this read-only review.

## Solution Assessment

No. Reusing the existing VBS hidden launcher is the right direction, but this patch applies the default after the scheduler has already chosen the task action.
