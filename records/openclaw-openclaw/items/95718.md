## Best Possible Solution

Keep the Signal plugin-local approach, but land it only after the adapter declares and proves reply support and quote fallback is limited to quote-specific rejection without duplicate-send risk.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Declare Signal reply support before forwarding replyToId:** `extensions/signal/src/channel.ts:118`
  - body: This PR starts passing `ctx.replyToId` into Signal sends, but the Signal message adapter still declares only `text` and `media` durable capabilities. Core marks any send with `replyToId` as requiring `replyTo`, so required durable Signal sends with a reply target will still fail capability preflight before reaching the new quote path. Add and prove `replyTo: true`, or avoid advertising this path through the durable adapter.
  - confidence: 0.9
- **[P2] Limit quote fallback to quote-specific failures:** `extensions/signal/src/send.ts:330-331`
  - body: The new fallback catches every `signalRpcRequest` failure and immediately retries the same message without quote metadata. Since the Signal HTTP wrapper can reject for timeouts and transport errors, an accepted quote send whose response is lost can be followed by a second ordinary send, duplicating the assistant reply. Only retry for classified quote-metadata rejection; otherwise preserve the original failure.
  - confidence: 0.86
