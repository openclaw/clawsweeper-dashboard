## Summary

Keep open. Current main still has a cron-only pre-resolution path where a slash-form `kimi-code/...` ref can fall through to legacy provider normalization as `kimi/...` unless the provider is declared in `models.providers`, while the reported provider lives in models.json/catalog state.

## What This Changes

The issue asks OpenClaw to resolve custom provider aliases such as `kimi-code/kimi-for-coding` consistently in cron-nested runs, preserve custom models.json additions, and document the supported alias shape.

## Best Possible Solution

Cron and embedded runs should share exact-provider preservation for configured or catalog-backed custom providers, while the built-in Kimi Coding alias remains `kimi/kimi-code`.

## Reproduction Assessment

Yes. The issue includes concrete v2026.4.29 failure logs, and current main statically shows a models.json/catalog-only `kimi-code/...` ref can be normalized to `kimi/...` before cron-nested dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve exact catalog/config provider refs in the shared resolver and to merge existing models.json custom model entries for regenerated provider keys, with focused cron and merge regression coverage.
