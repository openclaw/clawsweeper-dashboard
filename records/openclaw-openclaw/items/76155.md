## Summary

Keep open. Current main still appears to preserve only a narrow custom-provider case before falling back to provider normalization, and open PR #76163 already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests consistent cron-nested resolution for custom provider aliases such as `kimi-code/kimi-for-coding`, preservation of custom `models.json` additions, and docs for the canonical alias format.

## Best Possible Solution

Cron and embedded model resolution should share exact custom-provider preservation, and bootstrap/merge behavior should not drop user-added model entries; the open PR should be evaluated against both outcomes or split a narrow follow-up for the merge behavior.

Yes. The report includes concrete v2026.4.29 logs, and current main statically shows cron `payload.model` goes through a path that can normalize `kimi-code/...` to `kimi/...` when exact configured-provider preservation does not apply.
