## Summary

Keep open. Current main still has a cron payload resolver path that can normalize a catalog/models.json-only `kimi-code/...` ref to `kimi/...`, and the related fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve cron-nested custom provider refs such as `kimi-code/kimi-for-coding`, keep custom models.json additions across regeneration, and document the canonical alias format.

## Best Possible Solution

Cron and embedded model selection should preserve literal custom provider refs for configured or catalog-backed providers before legacy alias normalization, and regenerated models.json should merge user-added models while docs keep the built-in `kimi/kimi-code` path clear.

## Reproduction Assessment

Yes, by source inspection plus the issue's concrete logs. A cron `payload.model` flows through `resolveAllowedModelRef`; for `kimi-code/...`, the exact-provider guard can return null and `parseModelRef` then normalizes the provider to `kimi` before dispatch.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve literal alias-colliding provider keys in the shared resolver and add regression coverage for cron forwarding plus models.json merge behavior. #76163 is useful context, but it closed unmerged and appears narrower than the full report.
