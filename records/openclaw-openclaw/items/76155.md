## Summary

Keep open. Current main still has a cron pre-resolution path that can normalize a catalog/models.json-only `kimi-code/...` ref to `kimi/...`, and the related PR that attempted a partial fix was closed unmerged.

## What This Changes

Request cron-nested model resolution to preserve custom `kimi-code/kimi-for-coding` aliases, keep custom `models.json` additions across bootstrap, and document canonical custom-provider alias formats.

## Best Possible Solution

Cron and embedded runs should preserve exact custom provider refs from config or catalog state before legacy Kimi normalization, while keeping the built-in `kimi/kimi-code` alias intact and preserving custom `models.json` model entries during regeneration.

## Reproduction Assessment

Yes, by source inspection plus the issue's concrete v2026.4.29 logs. Current main still shows cron resolving `payload.model` through a path that can normalize `kimi-code/...` to `kimi/...` before dispatch when the exact provider is only catalog/models.json-backed or misses the narrow config-preservation branch.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve literal custom provider keys in shared model resolution and merge existing custom model entries on regenerated provider keys, with cron and merge regression coverage. The closed #76163 branch is useful context but did not land and appears narrower than the full report.
