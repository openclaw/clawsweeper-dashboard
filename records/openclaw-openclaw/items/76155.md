## Summary

Keep open. Current main still contains the resolver path that can normalize `kimi-code/...` to `kimi/...`, and open PR #76163 is an unmerged closing implementation candidate, so this issue should stay open until that PR is reviewed, landed, or rejected.

## What This Changes

The issue asks for cron-nested model selection to preserve custom provider aliases such as `kimi-code/kimi-for-coding`, avoid wiping custom `models.json` additions, and document the canonical alias format.

## Best Possible Solution

Review #76163 against both api-absent `models.providers` and models.json/catalog-only `kimi-code/kimi-for-coding` cases, then land it if cron and embedded lanes preserve the same exact provider ref.

## Reproduction Assessment

Yes. The issue includes concrete v2026.4.29 cron-nested failure logs, and current main statically shows `payload.model` flowing through a resolver path that can fall through to `normalizeProviderId("kimi-code") === "kimi"`.

## Solution Assessment

Yes, with a caveat. Preserving exact user-configured provider keys in the shared resolver with regression coverage is the narrow maintainable direction, but the active PR should also be checked against the catalog-only shape reported here.
