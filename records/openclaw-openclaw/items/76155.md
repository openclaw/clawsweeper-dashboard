## Summary

Keep open. Current main still lets cron pre-resolution normalize a catalog/models.json-only `kimi-code/...` ref to `kimi/...`, the related fix PR was closed unmerged, and regenerated provider-key merges still preserve only selected secrets/base URL rather than custom model additions.

## What This Changes

The issue asks for cron-nested model resolution to preserve custom provider aliases like `kimi-code/kimi-for-coding`, keep custom `models.json` additions across bootstrap, and document the canonical alias format.

## Best Possible Solution

Shared model resolution should preserve literal custom provider refs from config and persisted catalog state before legacy Kimi normalization, and provider regeneration should merge existing custom model entries without weakening built-in alias compatibility.

## Reproduction Assessment

Yes. The issue has concrete v2026.4.29 logs, and current-main source shows the cron `payload.model` path can still fall through from a `models.json`-only `kimi-code/...` ref to `parseModelRef`, where `kimi-code` normalizes to `kimi`.

## Solution Assessment

Yes. The narrow maintainable fix is in shared model-resolution and provider-merge helpers with cron/model-selection regression tests, rather than a cron-only special case or a new config knob.
