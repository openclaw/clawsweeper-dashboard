## Summary

Current main has an unreleased repair for malformed escaped and partial JSON tool arguments, but the issue’s central claim that large payloads are reliably dropped is not yet proven fixed or reproducible; keep it open as a diagnosis-first product decision rather than committing to a parallel streaming/file API.

## What This Changes

Add reliable support for large write, exec, and other tool arguments, potentially through streaming, file-backed input, automatic chunking, or more tolerant JSON handling.

## Best Possible Solution

Keep one canonical tool-argument reliability investigation, reproduce a large ASCII and CJK write/exec call on a named provider and model, then fix the smallest existing provider or shared parser boundary with end-to-end regression coverage; add a new transport only if a core payload ceiling is demonstrated.

## Reproduction Assessment

No. The relevant streamed parsing path is clear and related reports describe similar symptoms, but there is no deterministic current-main run with raw provider fragments proving a size-based failure.

## Solution Assessment

No. Streaming or file-backed arguments are broader than the evidence supports; the best first solution is to verify and repair the existing provider/shared tool-call path.
