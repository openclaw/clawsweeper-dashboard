## Summary

Keep open. Current main still has only inline JSON argument paths for tool invocation and Codex dynamic-tool calls, while related open issues cover narrower provider-specific argument loss rather than resolving this broader streamed or file-backed API decision.

## What This Changes

Add streamed or file-backed tool arguments so large exec/write payloads can avoid inline JSON serialization limits and fail with clearer recovery behavior.

## Best Possible Solution

Keep this as the product/API tracker for a staged or file-backed OpenClaw tool input contract, while provider-specific argument-loss bugs stay in their narrower reports.

## Reproduction Assessment

No high-confidence live reproduction of the exact >10KB JSON parse failure was run. Source inspection does confirm current main lacks the requested streamed, staged, or file-backed tool-argument API and still routes relevant payloads through inline JSON objects.

## Solution Assessment

Unclear until maintainers choose the API boundary. A staged-content or file-backed stdin contract at the tool/protocol layer looks safer than transparent shell chunking, while provider-specific empty-argument bugs should stay in narrower trackers.
