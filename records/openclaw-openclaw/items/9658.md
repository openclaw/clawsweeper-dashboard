## Summary

Keep open: current main still has inline JSON/string arguments for exec, write, Gateway tool invocation, and the SDK, and the related open report at https://github.com/openclaw/openclaw/issues/53408 covers a narrower provider/runtime argument-loss bug rather than this staged-payload or streaming contract.

## What This Changes

The request asks OpenClaw to support streaming or file-backed tool arguments so large exec/write/tool payloads can avoid JSON serialization limits.

## Best Possible Solution

Define one additive staged-payload or streaming contract and carry it consistently through tool schemas, Gateway HTTP/RPC invoke, SDK types, approvals/hooks, audit logging, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a new transport/API capability request. Source inspection on current main shows exec, write, Gateway invoke, and SDK paths still accept inline JSON/string payloads only.

## Solution Assessment

Unclear. The direction is valid, but maintainers should choose one explicit staged-payload or streaming contract rather than adding ad hoc shell rewriting, transparent command chunking, or only raising JSON limits.
