- `P2`: The request affects core tool reliability and automation, but it needs design review rather than emergency runtime cleanup.

## Summary

Keep open. Current main still passes dynamic tool-call arguments as JSON values and only has scoped malformed-argument repairs for selected transports; streaming or file-backed tool arguments remain a broader product/API design, not cleanup-safe implemented work.

## What This Changes

The issue requests streaming or file-backed tool arguments so large write/exec payloads can bypass inline JSON/tool-call serialization limits.

## Best Possible Solution

Keep this as the product/API tracker for large tool arguments, while using the narrower open reports for provider-specific argument-loss fixes and designing a documented staged/file-backed stdin contract before implementation.

## Reproduction Assessment

No high-confidence end-to-end current-main reproduction was run for the exact reporter scenario. Source inspection confirms the requested streamed/file-backed argument API is absent, and related open issues provide the concrete argument-dropping bug context.

## Solution Assessment

Unclear as an implementation until maintainers choose the API boundary. A staged/file-backed stdin or content-reference contract looks safer than automatic shell chunking, but it needs owner review across Codex app-server, Pi runner, tool schemas, docs, and sandbox policy.
