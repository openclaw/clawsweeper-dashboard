## Summary

Keep open: current main still exposes exec/write and Gateway/SDK tool invocation as inline string/JSON argument contracts only. The related open reports cover narrower provider/runtime argument-loss failures, but they do not supersede this broader staged-payload or streaming tool-argument design request.

## What This Changes

Request support for streaming or file-backed tool arguments so large exec/write payloads do not rely only on inline JSON serialization.

## Best Possible Solution

Define one additive staged-payload or streaming argument contract across tool schemas, Gateway HTTP/RPC, SDK types, approvals/hooks, audit logging, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this asks for a new streaming or file-backed argument contract. Source inspection on current main shows those carriers are still absent, while provider-specific failures are tracked by separate open reports.

## Solution Assessment

Unclear. The direction is valid, but maintainers should choose one explicit staged-payload or streaming contract instead of ad hoc shell rewriting, transparent chunking, or only raising JSON limits.
