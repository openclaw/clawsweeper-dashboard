## Summary

Keep open. Current main still exposes the affected exec, process, and write inputs as inline string arguments, and Codex-native argument mutation is explicitly unsupported, so this remains a broad core enhancement rather than cleanup closure.

## What This Changes

Support streaming or file-based tool argument payloads for exec/write/process so large inputs are not passed entirely through inline JSON strings.

## Best Possible Solution

Define and document a large-input tool contract spanning exec/process/write and the Codex harness, including stdin/file-reference semantics, sandbox/node path rules, clear size errors, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection against current main shows the requested stdin/file-reference escape hatch is absent, but I did not live-reproduce the provider JSON parse error.

## Solution Assessment

Unclear. Streaming or file references are plausible, but the maintainable path needs a versioned large-input contract across OpenClaw-owned tools plus a Codex-native boundary plan, not a one-off chunking shim.
