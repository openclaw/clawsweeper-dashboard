## Summary

This remains a valid reliability concern, but the issue does not establish its claimed 10 KB JSON serialization limit or show that a new transport API would address the failure. Current OpenClaw already repairs some malformed provider tool-call arguments, while the related reports point to provider-, model-, and context-dependent argument loss, so maintainers need a current-main raw-stream reproduction before choosing an API direction.

## What This Changes

Request streaming, file-backed, or automatically chunked tool arguments so large write and exec payloads do not fail or disappear.

## Best Possible Solution

Capture a current-main failure with redacted raw provider tool-call deltas and the resulting normalized call, then fix the earliest lossy provider/stream-normalization boundary; design a file or stdin tool contract only if complete payloads demonstrably reach OpenClaw and are then lost by its own transport.

## Reproduction Assessment

No. The prompts are plausible, but there is no high-confidence current-main reproduction with raw provider events showing whether arguments are truncated upstream, malformed during stream assembly, or lost inside OpenClaw.

## Solution Assessment

No. A new streaming or file-backed public tool contract is premature because it operates downstream of likely provider tool-call generation or assembly failures, and automatic chunking would alter execution semantics.
