## Summary

Keep open. Current main still exposes the relevant exec, process, and write inputs as inline JSON-shaped string arguments, and Codex-native argument rewriting remains explicitly unsupported, so this is a broad enhancement that needs maintainer design rather than cleanup closure.

## What This Changes

Request support for streaming or file-referenced tool inputs so large exec/process/write payloads do not have to be serialized inline as JSON strings.

## Best Possible Solution

Define a core large-input contract for OpenClaw-owned tools and the Codex bridge, including stdin/file-reference semantics, sandbox and node security rules, documented limits, and regression coverage for large payloads and clear errors.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that the requested streaming/file-reference escape hatch is absent, but the exact parser error was not live-reproduced.

## Solution Assessment

Unclear. The requested direction is reasonable, but the maintainable solution needs a design across tool schemas, process stdin lifecycle, sandbox/node boundaries, write semantics, docs, and possibly Codex-native app-server support.
