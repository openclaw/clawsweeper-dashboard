## Summary

Keep open. Current main and v2026.6.5 still use inline JSON object/string tool arguments; recent parsing repairs reduce malformed-argument loss, but they do not add the requested streaming or file-backed large-argument contract.

## What This Changes

The issue requests streaming or file-backed large tool arguments so `exec`, `write`, and similar tools can avoid inline JSON serialization limits.

## Best Possible Solution

Design one maintainer-approved staged-argument contract that preserves normal inline JSON args while adding an explicit large-payload path for exec/write and gateway invocation, with documented limits, cleanup, security review, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection does provide a source-level reproduction of the capability gap: current main, v2026.6.5, and sibling Codex still route tool arguments through inline JSON/string contracts.

## Solution Assessment

No, the issue body lists plausible options rather than one clearly best implementation. The best path is a maintainer-approved staged-argument design because streaming, file-backed args, automatic chunking, and higher JSON limits have different protocol and security tradeoffs.
