## Summary

Keep open. Current main still has inline JSON-valued tool arguments and scoped malformed-argument repairs, while streamed or file-backed arguments remain an unresolved product/API design rather than cleanup-safe implemented work.

## What This Changes

Support streamed or file-backed tool arguments so large write/exec payloads do not depend on one JSON-serialized argument object.

## Best Possible Solution

Keep this as the product/API tracker for large tool arguments, design a staged content or file-backed stdin contract across Codex app-server, embedded runner, tool schemas, docs, and sandbox policy, and leave provider-specific empty-argument bugs in their narrower linked reports.

## Reproduction Assessment

No, not for the exact reporter scenario. Source inspection confirms the requested streamed/file-backed API is absent, and related open reports provide concrete large-argument failure context, but this review did not run an end-to-end current-main reproduction.

## Solution Assessment

Unclear until maintainers choose the API boundary. A staged content or file-backed stdin contract looks safer than automatic shell chunking, but it needs owner review across protocol, runner, schema, docs, and sandbox behavior.
