## Summary

Keep open: current main still has JSON-only tool argument contracts, and the requested large-argument transport remains an unimplemented API/protocol design rather than a cleanup-close item.

## What This Changes

Add streaming or file-backed tool argument support so large exec/write/process payloads can avoid inline JSON serialization failures.

## Best Possible Solution

Define one explicit large tool-input contract, such as a staged file/token or streamed stdin primitive, then wire it through Gateway protocol, OpenClaw-owned tools, provider adapters, docs, and focused regression coverage without automatic shell-command rewriting.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection does show the current exec, write, process, and Gateway invoke paths still accept JSON string/object arguments only.

## Solution Assessment

Unclear which proposed option is best. A maintainer should choose a single explicit transport contract that preserves tool permission and audit boundaries instead of adding ad hoc automatic chunking.
