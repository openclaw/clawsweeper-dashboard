## Summary

Keep open: current main still routes exec, write, Gateway, and SDK tool invocation through inline JSON/string argument contracts, while streaming or file-backed arguments would be a new public contract with lifecycle and security design work. The narrower large-argument loss reports remain open and should stay separate from this broader feature tracker.

## What This Changes

Request support for streaming or file-backed large tool arguments so exec/write and other tool calls can avoid inline JSON serialization limits.

## Best Possible Solution

Define one additive staged-payload or streaming argument contract across tool schemas, Gateway HTTP/RPC, SDK types, approvals/hooks, audit logging, docs, and regression coverage, while keeping narrower provider-specific argument-loss fixes in their canonical bug reports.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this asks for a new streaming or file-backed argument contract. Source inspection shows current main still lacks those carriers in exec, write, Gateway, and SDK surfaces.

## Solution Assessment

Unclear. The direction is valid, but maintainers should choose one explicit staged-payload or streaming contract instead of ad hoc shell rewriting, transparent chunking, or only raising JSON/buffer limits.
