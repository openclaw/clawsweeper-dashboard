## Summary

Keep open. Current main still routes the relevant `exec`, `process`, and `write` surfaces through inline JSON-shaped string parameters, and there is no implemented streaming or file-reference escape hatch for large tool arguments. The exact reported parser threshold was not live-reproduced, but source and docs show the requested capability is still missing and broad enough to need maintainer design before an automated fix.

## What This Changes

Request support for streaming or file-based tool arguments so large `exec`, `write`, and subagent payloads can avoid inline JSON serialization failures.

## Best Possible Solution

Design a core-owned large-input contract before implementation: define how `exec`, `process`, and `write` can accept stdin or file references across host, node, and sandbox execution; document provider/schema limits; preserve security boundaries; and add focused tests for large payloads and clear limit errors. If the failing path is Codex-native, coordinate the required app-server hook/schema support instead of trying to rewrite unsupported native arguments inside OpenClaw.

## Reproduction Assessment

Unclear. The issue gives plausible examples, but this review did not run a live provider or subagent turn; static inspection does give a high-confidence path showing no current streaming/file-based argument alternative exists.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best maintainable fix is not a one-line schema addition because it spans tool schemas, process stdin lifecycle, sandbox/node file security, docs, and possible Codex-native API support.
