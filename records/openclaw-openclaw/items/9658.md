## Summary

Keep open. Current main and the latest release still pass exec, write, and Codex dynamic-tool inputs as inline JSON/object arguments; recent related repairs only cover narrower malformed or provider-specific argument symptoms, not the requested streamed or file-backed tool-input contract.

## What This Changes

The issue requests streaming or file-backed tool arguments so large exec/write inputs do not have to travel as inline JSON strings.

## Best Possible Solution

Keep this as the product/API tracker for staged, file-backed, or streamed OpenClaw tool input, with explicit sandbox, temp-file lifetime, audit, and redaction semantics; narrower provider-specific argument bugs should remain separate.

## Reproduction Assessment

No high-confidence live reproduction of the exact >10KB Bad escaped character failure was run. Source and docs do confirm current main lacks any streamed or file-backed tool-argument API and still validates exec/write through inline command/content fields.

## Solution Assessment

Unclear until maintainers choose the API boundary. A staged-content or file-backed stdin contract at the tool/protocol layer looks safer than transparent shell chunking, while provider-specific empty-argument bugs should stay in narrower trackers.
