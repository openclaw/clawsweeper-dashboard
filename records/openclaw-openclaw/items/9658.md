## Summary

Keep open: current main still exposes exec, write, Gateway HTTP/RPC tools.invoke, and SDK tool invocation as inline string/JSON argument contracts. Related open reports cover narrower provider/runtime argument-loss failures, but they do not supersede the requested staged-payload or streaming tool-argument design.

## What This Changes

The issue requests streamed or file-backed tool arguments so large exec/write payloads do not have to travel entirely as inline JSON strings.

## Best Possible Solution

Define one additive staged-payload or streaming tool-argument contract and carry it through tool schemas, Gateway HTTP/RPC, SDK types, approvals/hooks, audit logging, docs, and regression tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a new API/transport capability request. Source inspection on current main gives high confidence that streamed or file-backed tool arguments are still absent.

## Solution Assessment

Unclear. The direction is valid, but maintainers should choose a single explicit staged-payload or streaming contract instead of ad hoc shell rewriting, transparent chunking, or only raising JSON limits.
