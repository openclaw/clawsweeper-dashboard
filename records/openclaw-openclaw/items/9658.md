## Summary

The reported symptom is still worth investigating, but this feature request assumes an unproven core JSON size limit and proposes a parallel tool transport before isolating the failure. The active, more specific report at https://github.com/openclaw/openclaw/issues/53408 already owns the observable `write`/`exec` argument-loss investigation with provider, model, session-length, content, and reproduction details.

## What This Changes

Requests streaming, file-backed, chunked, or larger tool arguments to avoid reported JSON serialization failures on large `exec` and `write` calls.

## Best Possible Solution

Keep the observable argument-loss investigation consolidated in https://github.com/openclaw/openclaw/issues/53408, capture raw streamed tool-call events on current main, and repair the owning provider/parser boundary before considering any new tool payload contract.

## Reproduction Assessment

No. The example prompts may trigger the symptom, but there is no deterministic current-main trace showing where the arguments are lost or proving a 10 KB JSON serialization ceiling.

## Solution Assessment

No. Adding streaming or file-backed tool parameters is premature because current provider code already streams and accumulates tool JSON; the narrower solution is to diagnose and fix the provider/parser path tracked by the canonical report.
