## Summary

Keep open: current main and v2026.6.5 still use inline JSON object/string tool arguments, so the requested streamed or file-backed large-argument contract is not implemented. Recent parser repairs reduce some malformed streamed-argument failures, but they do not solve the broader large-payload design request.

## What This Changes

Request support for streamed or file-backed tool arguments so large exec/write payloads do not have to fit inside inline JSON serialization.

## Best Possible Solution

Design and land one maintainer-approved staged large-argument contract for exec/write and gateway invocation, preserving inline JSON args while documenting limits, cleanup, security boundaries, and regression coverage.

## Reproduction Assessment

No high-confidence live reproduction of the exact Bad escaped character failure was established. Source inspection does show the missing capability: current main and v2026.6.5 still accept only inline JSON object/string tool arguments.

## Solution Assessment

Unclear as proposed: streaming, file-backed args, automatic chunking, and higher limits have different protocol and security tradeoffs. The best path is a maintainer-approved staged-argument design rather than a narrow autonomous patch.
