## Summary

Keep open: current main still accepts tool inputs as inline JSON string/object parameters on the exec, process, write, RPC, and HTTP invoke paths, so the requested large-input transport remains unresolved core API/protocol work.

## What This Changes

The issue requests streaming or file-based tool arguments so large exec/write payloads do not have to be serialized inline as JSON.

## Best Possible Solution

Define one explicit large-input contract, such as staged file tokens or streamed stdin, then wire it through tool schemas, Gateway protocol/HTTP invoke, hooks/approvals, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that current main still only supports inline JSON string/object arguments on the affected paths.

## Solution Assessment

Unclear which proposed option is best. Maintainers should choose an explicit transport contract that preserves permission and audit boundaries rather than adding ad hoc automatic shell-command rewriting.
