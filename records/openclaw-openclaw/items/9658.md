## Summary

Keep open: current main still routes exec, process/write, file write, and Gateway tool invocation inputs through inline JSON string/object parameters, so the requested large-input transport remains unresolved API/product work.

## What This Changes

The issue requests streaming or file-backed tool arguments so large exec/write-style payloads do not have to be serialized entirely as JSON strings.

## Best Possible Solution

Define one explicit large-input contract, such as staged file tokens or a streamed stdin primitive, then wire it through tool schemas, Gateway protocol, permission hooks, audit/approval behavior, docs, and focused regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that current main still only supports inline JSON string/object arguments on the affected paths.

## Solution Assessment

Unclear which proposed option is best. Maintainers should choose an explicit transport contract that preserves permission and audit boundaries rather than adding ad hoc automatic shell-command rewriting.
