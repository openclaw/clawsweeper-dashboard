## Summary

Keep open: current main still routes exec/write/process/tool-invoke inputs through JSON string/object parameters, and the remaining request is a new large-argument transport/API design rather than a cleanup-close item.

## What This Changes

Request support for streaming or file-backed tool arguments so large exec/write inputs do not depend on inline JSON serialization.

## Best Possible Solution

Define one explicit large tool-input contract, likely a staged file/token or streamed stdin primitive, then wire it through Gateway protocol, dynamic tools, provider adapters, docs, and regression tests while keeping existing exec/write validation intact.

## Reproduction Assessment

No high-confidence failing run was established for the exact `Bad escaped character in JSON` error. Reproduction is mostly not applicable because this is a feature/API request, though source inspection and related #70033/#53408 confirm the current JSON-only argument shape and similar user-visible failures.

## Solution Assessment

Unclear whether streaming, file-backed args, or a staged-token contract is the best final shape. The safest next step is a maintainer design decision that avoids automatic shell command rewriting and preserves tool permission/audit boundaries.
