## Summary

Keep open. Current main and the latest release still use inline JSON/object tool arguments for Codex dynamic tools, exec, and write; the related open reports cover narrower argument-loss symptoms, not the broader streamed or file-backed tool-input contract requested here.

## What This Changes

The issue requests streamed or file-backed tool arguments so large write/exec payloads do not have to travel as inline JSON.

## Best Possible Solution

Keep this as the product/API tracker for staged, file-backed, or streamed OpenClaw tool input, while narrower provider-specific argument-loss bugs continue in https://github.com/openclaw/openclaw/issues/53408 and https://github.com/openclaw/openclaw/issues/70033.

## Reproduction Assessment

No high-confidence live reproduction of the exact >10KB JSON parse failure was run. Source inspection does confirm current main lacks the requested streamed, staged, or file-backed tool-argument API and still routes relevant payloads through inline JSON/object arguments.

## Solution Assessment

Unclear until maintainers choose the API boundary. A staged-content or file-backed stdin contract at the tool/protocol layer looks safer than transparent shell chunking, while provider-specific empty-argument bugs should stay in narrower trackers.
