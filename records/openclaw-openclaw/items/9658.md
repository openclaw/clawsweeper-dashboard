## Summary

Keep open. Current main still accepts OpenClaw/Codex dynamic tool inputs as inline JSON arguments and has no streamed, staged-content, or file-backed argument contract, while related large/empty argument reports remain open and point to the same product area.

## What This Changes

Request streamed or file-backed tool arguments so large `exec` and `write` payloads can avoid inline JSON serialization limits.

## Best Possible Solution

Keep this as the product/API tracker for staged or file-backed OpenClaw tool input, while provider-specific empty-argument failures stay in narrower linked reports.

## Solution Assessment

Unclear until maintainers choose the API boundary. A staged-content or file-backed stdin contract at the dynamic-tool/protocol layer looks safer than automatic shell chunking, while provider-specific empty-argument bugs should stay in their narrower trackers.
