## Summary

Keep open. Current main and the latest release still pass exec, write, Gateway `tools.invoke`, and Codex dynamic-tool inputs as inline JSON/object arguments; existing repairs only cover narrower streamed/malformed argument parsing symptoms, not the requested streamed or file-backed tool-input contract.

## What This Changes

Request support for streamed or file-backed tool arguments so large exec/write payloads do not have to be carried entirely inside JSON-serialized tool parameters.

## Best Possible Solution

Keep this as the product/API tracker for staged, file-backed, or streamed OpenClaw tool input while narrower provider-specific argument-loss and timeout reports continue separately.

## Reproduction Assessment

No high-confidence live reproduction of the exact `Bad escaped character in JSON` failure was run. Source and release inspection do confirm that the requested streamed or file-backed tool-argument API is absent and that related large-argument failures remain tracked separately.

## Solution Assessment

Unclear until maintainers choose the API boundary. A staged-content or file-backed stdin contract at the tool/protocol layer looks safer than transparent shell chunking, while provider-specific empty-argument bugs should stay in narrower trackers.
