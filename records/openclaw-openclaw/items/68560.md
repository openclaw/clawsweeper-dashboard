## Summary

Keep open for maintainer review. The PR addresses a real transcript-bloat problem, but its write-time empty-payload approach conflicts with current main's intentional recent-image replay window and would make follow-up turns lose image context; the added test also copies the helper instead of proving the production persistence path.

## What This Changes

The PR adds a `guardSessionManager` persistence transform that clears image block data in tool-result messages after plugin hooks, plus a colocated test file that duplicates the stripping helper behavior.

## Best Possible Solution

Implement the media-reference plus hydration direction tracked by #1210, or extend the existing replay-pruning model so long-lived base64 payloads are removed only after durable replay and display semantics are defined. The final fix should keep recent image context replayable and cover the production persistence, replay sanitization, and pruning path together.

## Reproduction Assessment

Yes. Persist a tool-result image through the PR transform, then reload or sanitize session history; `data: ""` is converted to an omitted-empty-payload text placeholder, so the image cannot be replayed on a follow-up turn.

## Solution Assessment

No. Emptying every persisted tool-result image is narrower than the media-reference design, but it is not the best fix because it removes the recent replay window current main deliberately keeps for follow-up vision context.
