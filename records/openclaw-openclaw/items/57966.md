## Summary

Keep open. Current main does not implement the central alias-normalization behavior, but this PR is not mergeable as-is because it targets the pre-split resolver surface and introduces fuzzy-alias precedence, same-provider collision, and alias-output sanitization problems.

## What This Changes

The PR changes model-selection and gateway session identity logic to fold spaces and underscores in model aliases, infer providers from configured models with matched model IDs, add alias hints to disallowed-model errors, and expand resolver tests.

## Best Possible Solution

Land a rebased fix on the current shared resolver that preserves exact configured/default-provider model ID precedence, treats normalized alias and model collisions as ambiguous, returns matched configured model IDs only when unambiguous, sanitizes alias hints and logs, and adds focused tests for the chosen separator policy.

## Reproduction Assessment

Yes. The PR patch itself provides high-confidence resolver-level reproductions: with an allowlisted exact model such as anthropic/my_model plus alias my-model -> openai/gpt-4o, providerless my_model is routed through the fuzzy alias path instead of the exact model ID.

## Solution Assessment

No. The requested direction is plausible for #57917, but this PR is not the best implementation until it is rebased onto the shared resolver and fixes precedence, ambiguity, and alias-output sanitization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve exact model IDs before fuzzy alias fallback:** `src/agents/model-selection.ts:360-364`
  - body: Folding `_` and spaces to `-` before alias lookup lets a providerless exact model ID such as `my_model` match an alias like `my-model` before the normal default-provider model path. With `anthropic/my_model` allowlisted and `my-model` pointing elsewhere, `/model my_model` can silently switch providers; exact configured/default-provider model IDs need to win before fuzzy alias fallback.
  - confidence: 0.92
- **[P2] Escape aliases before adding error hints:** `src/agents/model-selection.ts:765-767`
  - body: The new available-alias hint joins configured alias strings directly into `model not allowed` errors. Aliases come from config, so newlines, control characters, or ANSI escapes can be reflected into terminal or UI output; sanitize or escape each alias before joining.
  - confidence: 0.88
- **[P2] Reject normalized same-provider model collisions:** `src/agents/model-selection.ts:236-237`
  - body: `inferUniqueProviderFromConfiguredModels` overwrites `matchedModelId` for each normalized match but only treats multiple providers as ambiguous. If one provider configures both `gpt_5.4` and `gpt-5.4`, input like `gpt 5.4` resolves to whichever entry is visited last instead of rejecting an ambiguous match.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete output-sanitization issue by appending raw configured aliases to user-visible disallowed-model errors.

Concerns:

- **[medium] Raw aliases are reflected in error hints:** `src/agents/model-selection.ts:766`
  - body: Configured aliases can contain newlines, control characters, or ANSI escapes, and the PR returns them through the new available-alias hint in `model not allowed` errors. That can forge terminal or chat output unless each alias is sanitized or escaped first.
  - confidence: 0.88
