## Summary

Keep open: the linked Control UI bug is already fixed on current main, but this PR still proposes unique fuzzy alias/error-message behavior and the current head has concrete routing and output-safety regressions plus no real behavior proof.

## What This Changes

The PR normalizes spaces and underscores in model alias lookup, adds configured-provider inference and alias hints for disallowed models, updates session model identity resolution, and expands model-selection tests.

## Best Possible Solution

Keep the current-main fix for the linked provider-prefix bug; if fuzzy separator matching is still desired, rebase a narrower change into model-selection-shared.ts with exact-ID precedence, collision rejection, sanitized hints, focused tests, and real behavior proof.

## Reproduction Assessment

Yes for the PR-head defects by source inspection: alias lookup happens before exact model parsing, same-provider normalized matches overwrite the matched model id, and raw aliases flow into errors. I did not run tests because this is a read-only review.

## Solution Assessment

No. Current main already fixes the linked provider-prefix drift, while this PR adds fuzzy matching on a stale resolver surface and needs maintainer policy judgment plus safer precedence and output handling before it could be the right fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve exact model IDs before fuzzy alias fallback:** `src/agents/model-selection.ts:361-364`
  - body: Folding spaces and underscores before alias lookup lets a providerless exact model ID such as `my_model` match an alias like `my-model` before the normal model path. With `anthropic/my_model` allowlisted and `my-model` pointing elsewhere, `/model my_model` can silently switch providers.
  - confidence: 0.92
- **[P2] Treat same-provider model collisions as ambiguous:** `src/agents/model-selection.ts:236-237`
  - body: `inferUniqueProviderFromConfiguredModels` overwrites `matchedModelId` for every normalized match but only treats multiple providers as ambiguous. If one provider configures both `gpt_5.4` and `gpt-5.4`, input like `gpt 5.4` resolves to whichever entry is visited last.
  - confidence: 0.84
- **[P2] Sanitize aliases before adding error hints:** `src/agents/model-selection.ts:765-766`
  - body: The new available-alias hint joins configured aliases directly into `model not allowed` errors. Aliases can contain control characters or ANSI escapes, so each alias needs to be sanitized or escaped before it is returned to terminal or UI surfaces.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete output-injection risk by reflecting raw configured aliases in user-visible error text.

Concerns:

- **[medium] Raw aliases are reflected in error hints:** `src/agents/model-selection.ts:766`
  - body: Configured aliases can include newlines, control characters, or ANSI escapes, and the PR appends them directly to the `model not allowed` message. That can forge terminal or chat output unless each alias is sanitized or escaped.
