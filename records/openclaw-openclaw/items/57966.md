## Summary

Keep open. Current main does not implement this PR's fuzzy space/underscore alias matching or alias-hint errors, so it is not obsolete; however the latest PR branch is not merge-ready because it changes the pre-split resolver, introduces exact-model routing regressions, and reflects raw configured aliases in user-visible errors.

## What This Changes

The PR branch changes model selection to fold spaces and underscores into hyphens for alias/model inference, return matched configured model IDs from provider inference, append available aliases to disallowed-model errors, update session identity resolution, and add model-selection tests.

## Best Possible Solution

Keep the contributor PR open as source work, but require a rebase onto `src/agents/model-selection-shared.ts` and a narrower implementation that preserves exact model-id precedence, treats normalized collisions as ambiguous, returns matched configured model IDs only when unambiguous, sanitizes alias hints/log text, and tests the chosen separator policy.

## Reproduction Assessment

Yes. The PR patch itself gives a high-confidence reproduction: with `anthropic/my_model` allowlisted and alias `my-model -> openai/gpt-4o`, the added fuzzy normalization makes `/model my_model` resolve to the alias target instead of the exact allowlisted model.

## Solution Assessment

No. The direction is plausible, but this implementation is not the best fix because fuzzy aliases win before exact model IDs, raw alias strings are reflected in errors, same-provider normalized collisions are not rejected, and the branch no longer targets the current shared resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve exact model IDs before fuzzy aliases:** `src/agents/model-selection.ts:72-75`
  - body: Folding `_` and spaces to `-` in `normalizeAliasKey` lets a bare exact model ID such as `my_model` match an alias like `my-model` first. The PR adds a test expecting `anthropic/my_model` to route to `openai/gpt-4o`; exact configured model IDs should win before fuzzy alias fallback.
  - confidence: 0.92
- **[P2] Escape aliases before adding error hints:** `src/agents/model-selection.ts:765-767`
  - body: The new error hint joins configured alias strings directly into `model not allowed` errors. Aliases come from config, so control characters, newlines, or ANSI escapes can be reflected into terminal or UI output; sanitize or escape each alias before joining.
  - confidence: 0.88
- **[P2] Reject same-provider normalized model collisions:** `src/agents/model-selection.ts:212-219`
  - body: `inferUniqueProviderFromConfiguredModels` records only providers and overwrites `matchedModelId` for each match. If one provider configures both `gpt_5.4` and `gpt-5.4`, input like `gpt 5.4` resolves to whichever entry is visited last instead of reporting an ambiguous match.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete output-sanitization issue by appending raw configured alias strings to user-visible disallowed-model errors.

Concerns:

- **[medium] Raw aliases are reflected in error hints:** `src/agents/model-selection.ts:766`
  - body: Configured aliases can contain newlines, control characters, or ANSI escapes, and the PR returns them via `aliases.join(", ")` in `model not allowed` errors. That can forge terminal or chat output unless each alias is sanitized or escaped first.
  - confidence: 0.88
