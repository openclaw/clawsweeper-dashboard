## Summary

Keep open. The underlying alias-resolution bug still appears real on current main, but this PR should not merge as-is: current main has since split the resolver into shared files, and the proposed fuzzy normalization can override exact model IDs, leave normalized same-provider collisions ambiguous, and reflect raw aliases in user-visible errors.

## What This Changes

The PR changes model alias lookup to fold spaces and underscores, makes configured-model provider inference return matched model IDs, appends available aliases to disallowed-model errors, updates session identity resolution, and adds resolver tests.

## Best Possible Solution

Land a rebased fix on the current shared resolver that preserves exact model-id precedence over fuzzy aliases, treats normalized alias/model collisions as ambiguous, returns matched configured model IDs only when unambiguous, sanitizes alias hints and logs, and adds focused tests for the chosen separator policy.

## Reproduction Assessment

Yes. The PR patch and comments give a high-confidence reproduction: configure an exact allowlisted model such as `anthropic/my_model` plus alias `my-model -> openai/gpt-4o`; after separator folding, providerless `my_model` can resolve through the alias before the exact model ID.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best implementation until it is rebased onto the shared resolver and fixes the precedence, ambiguity, and alias-output sanitization issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve exact model IDs before fuzzy aliases:** `src/agents/model-selection.ts:72-75`
  - body: Folding `_` and spaces to `-` lets a bare exact model ID such as `my_model` match an alias like `my-model` before normal model parsing. With `anthropic/my_model` allowlisted and `my-model` pointing at another provider, `/model my_model` can silently switch providers; exact configured/default-provider model IDs need to win before fuzzy alias fallback.
  - confidence: 0.92
- **[P2] Escape aliases before adding error hints:** `src/agents/model-selection.ts:768`
  - body: The new error hint joins configured alias strings directly into `model not allowed` errors. Aliases come from config, so control characters, newlines, or ANSI escapes can be reflected into terminal or UI output; sanitize or escape each alias before joining.
  - confidence: 0.88
- **[P2] Treat same-provider normalized model collisions as ambiguous:** `src/agents/model-selection.ts:218-224`
  - body: `inferUniqueProviderFromConfiguredModels` tracks providers but overwrites the matched model ID for each match. If one provider configures both `gpt_5.4` and `gpt-5.4`, input like `gpt 5.4` resolves to whichever entry is visited last instead of reporting an ambiguous match.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete output-sanitization issue by appending raw configured aliases to user-visible disallowed-model errors.

Concerns:

- **[medium] Raw aliases are reflected in error hints:** `src/agents/model-selection.ts:768`
  - body: Configured aliases can contain newlines, control characters, or ANSI escapes, and the PR returns them through the new available-alias hint in `model not allowed` errors. That can forge terminal or chat output unless each alias is sanitized or escaped first.
  - confidence: 0.88
