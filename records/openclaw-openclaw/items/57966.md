## Summary

Keep open: the linked UI issue was closed separately, but this PR still proposes backend resolver changes and has blocking routing/security findings plus missing real behavior proof.

## What This Changes

The branch changes model alias normalization, provider inference, model-not-allowed alias hints, session model identity handling, and model-selection tests.

## Best Possible Solution

Rebase onto current main’s shared resolver, prefer exact configured model IDs before fuzzy alias matching, reject ambiguous normalized collisions, sanitize alias hints, and show real `/model` or `sessions.patch` proof.

## Reproduction Assessment

Yes, from source inspection rather than execution. Current main lowercases alias keys without folding separators, then falls back to the default provider when `gpt 5.4` does not match configured alias `gpt-5.4`.

## Solution Assessment

No. The PR targets the right resolver area, but its global separator folding can override exact model IDs, leaves same-provider normalized collisions ambiguous, and emits unsanitized aliases in errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve exact model IDs before fuzzy alias matches:** `src/agents/model-selection.ts:75`
  - body: Folding spaces and underscores into `-` before alias lookup makes exact providerless model IDs lose to unrelated aliases. With allowlisted `anthropic/my_model` plus alias `my-model -> openai/gpt-4o`, `/model my_model` now resolves to the alias target instead of the exact model under the default provider; exact configured model/allowlist matches should win before fuzzy alias matching.
  - confidence: 0.9
- **[P2] Reject same-provider normalized model collisions:** `src/agents/model-selection.ts:219`
  - body: `matchedModelId` is overwritten for every normalized match while ambiguity only checks provider count. If one provider configures both `gpt_5.4` and `gpt-5.4`, input like `gpt 5.4` resolves to whichever entry is visited last instead of reporting an ambiguous configured model match.
  - confidence: 0.86
- **[P2] Sanitize aliases before appending error hints:** `src/agents/model-selection.ts:766`
  - body: The new error hint joins raw configured aliases into the message returned by `sessions.patch`. Aliases come from config, so newline/control/ANSI content can forge terminal or chat output; sanitize or escape each alias before adding it to the error string.
  - confidence: 0.92
