## Label Justifications

- `P2`: A healthy restored provider can remain unusable for hours, but the failure is bounded to affected billing-disabled profiles and has workarounds such as waiting or switching providers.
- `impact:auth-provider`: The issue directly affects persisted auth-profile disable state, provider selection, and recovery after billing restoration.

## Summary

Current main and the latest release retain the reported billing-disable deadlock: the runner has a half-open probe seam, but it intentionally permits only transient reasons and excludes billing. The issue is valid and belongs in core auth-state recovery, but maintainers should choose the billing probe cadence or an explicit verified reset contract before implementation.

## What This Changes

Add a supported way to recover a provider auth profile early after credits are restored instead of waiting for the full persisted billing-disable interval.

## Best Possible Solution

Extend the existing auth-controller cooldown-probe seam with a persisted, bounded billing half-open cadence that clears state through markAuthProfileSuccess, and expose an operator-visible verified reset through the canonical agent auth store if automatic probing is not desired.

## Reproduction Assessment

Yes, from source with high confidence. An active billing disabledUntil is rejected before dispatch, billing is excluded from the existing half-open probe, and only a dispatched success reaches the state-clearing helper; the reporter also gives a concrete v2026.6.5 reproduction.

## Solution Assessment

Partly. Reusing the existing half-open probe seam is the narrowest architectural fix, but billing needs a persisted cadence or single-flight policy; a verified doctor/CLI reset is the safer alternative if maintainers reject automatic probes.
