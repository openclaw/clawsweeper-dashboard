## Summary

Current main does not fully solve the reported problem: billing-disabled single-provider setups receive throttled recovery probes, but configurations with fallback candidates suspend the billing provider until the cooldown is near expiry. The issue remains valid for that common fallback topology and needs a maintainer choice between earlier automatic billing probes and an explicit verified reset path.

## What This Changes

Allow a billing-disabled provider profile to recover before its five-hour persisted disable expires after provider credits are restored.

## Best Possible Solution

Define one canonical recovery contract in core: preferably a persisted, bounded half-open cadence for billing-disabled primaries even when fallbacks exist, with successful execution using the existing state reset and focused tests for restart persistence, throttling, fallback behavior, and permanent-auth exclusion.

## Reproduction Assessment

Yes, from current source with high confidence: configure a primary provider with at least one fallback, persist an active `disabledReason: billing` window far from expiry, and run the primary; `resolveCooldownDecision` returns `suspend_lanes` before dispatch, preventing success-based clearing.

## Solution Assessment

The suggested half-open direction is the best architectural fit, but the exact cadence and whether probe eligibility survives restarts require maintainer policy judgment; adding multiple doctor, CLI, and display paths at once would be broader than necessary.
