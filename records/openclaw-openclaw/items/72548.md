## Summary

Keep open: current main does not implement this feature, and the PR still has a blocking completeness issue because it opens/docs plugin-owned SecretRef sources while resolving only bundled secret-provider artifacts.

## What This Changes

The PR opens SecretRef sources and schemas to plugin-owned ids, adds a secret-provider SDK subpath, bundled GCP/keyring provider plugins, docs/tests/baselines, and resolver dispatch through a lazy bundled public-artifact loader.

## Best Possible Solution

Complete the SDK seam by resolving source ownership from the effective enabled bundled, installed, and workspace plugin set and loading only the enabled owner’s narrow secret-provider artifact, or explicitly scope this PR as bundled-only and leave the broader third-party seam tracked elsewhere.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: an installed or workspace plugin declaring contracts.secretProviders for a source such as aws can be accepted by the new schema/docs path, but resolution still calls the bundled-only resolver and then throws Unknown secret provider source.

## Solution Assessment

No. The manifest-first direction is maintainable, but the current PR is not the best complete solution while the advertised third-party plugin seam cannot load non-bundled providers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Load installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:67-84`
  - body: The SDK docs/schema accept arbitrary plugin-owned SecretRef sources and tell plugins to declare `contracts.secretProviders`, but runtime lookup only checks the bundled contract registry and loads bundled artifacts. An installed or workspace plugin declaring `contracts.secretProviders: ["aws"]` will validate and then fail as `Unknown secret provider source`, so the advertised third-party seam is unusable.
