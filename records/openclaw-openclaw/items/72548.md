## Summary

Keep open. Current main still has a closed env/file/exec SecretRef implementation, and this PR is not obsolete, but the proposed runtime seam still has blocking activation and third-party plugin loading gaps before it should merge.

## What This Changes

The PR adds a manifest-declared `contracts.secretProviders` SDK seam, opens SecretRef config and wire schemas for plugin-owned sources, adds bundled GCP/keyring secret-provider plugins, and updates resolver dispatch, docs, tests, exports, labeler, changelog, and generated baselines.

## Best Possible Solution

Keep the additive schema and SDK direction, but revise the runtime seam so source ownership is resolved from the effective enabled plugin set, including installed/workspace plugin manifests, and only the enabled owner’s narrow secret-provider artifact is executed.

## Reproduction Assessment

Yes. A static reproduction path is clear: in the PR diff, a config that disables all plugins, denies `secrets-gcp`, or leaves the default-off bundled provider disabled can still reach the bundled artifact loader when a SecretRef uses `source: "gcp"`; an installed plugin source such as `aws` can validate but cannot resolve.

## Solution Assessment

No. The manifest-first seam is the right direction, but the proposed implementation is not the best maintainable solution until it reuses activation-aware plugin resolution and supports the third-party plugin origins promised by the SDK/docs.

  - body: The resolver maps a SecretRef source directly to a bundled plugin and loads its artifact without receiving config or activation state. Because the new `secrets-gcp` and `secrets-keyring` manifests are default-off, configs that globally disable plugins, deny the plugin, disable its entry, or omit explicit enablement can still execute the provider when a SecretRef references its source.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The schemas and docs accept third-party secret-provider sources, but runtime lookup only calls the bundled contract registry and bundled public-artifact loader. An installed or workspace plugin can declare `contracts.secretProviders`, pass validation, and still fail as an unknown source, so AWS/Vault/Bitwarden-style providers still require core changes.
  - confidence: 0.88
