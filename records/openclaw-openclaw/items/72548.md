## Summary

Keep open: the latest head fixes the activation-gating security issue, but the advertised third-party SecretRef provider seam still resolves only bundled owners and artifacts, so installed/workspace providers accepted by the new schema fail at runtime.

## What This Changes

This PR opens SecretRef sources to plugin-owned strings, adds a manifest/SDK secret-provider contract, bundled GCP and keyring provider plugins, schema/protocol changes, docs, tests, generated baselines, labeler, and a changelog entry.

## Best Possible Solution

Keep the additive SecretRef/SDK direction, but resolve non-built-in source ownership through the effective enabled bundled, installed, and workspace plugin set, then load only the enabled owner’s narrow secret-provider artifact with activation and boundary checks.

## Reproduction Assessment

Yes. A static reproduction is clear: add an installed/workspace plugin declaring `contracts.secretProviders: ["aws"]`, configure `secrets.providers.ops = { source: "aws" }`, and the PR schema accepts it while runtime lookup returns no bundled owner and throws `Unknown secret provider source`.

## Solution Assessment

No. The manifest-first direction is maintainable, but this implementation only executes bundled providers; the safer complete fix is activation-aware installed/workspace owner lookup and public-artifact loading, or narrower docs/types that do not advertise a third-party seam.
