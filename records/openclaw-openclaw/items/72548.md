## Summary

Keep open. Latest head still promises third-party pluggable SecretRef sources, but runtime resolution only maps and loads bundled secret-provider artifacts, so installed/workspace plugins can validate but cannot resolve.

## What This Changes

The branch opens SecretRef source/config/protocol schemas, adds a manifest-declared secret-provider SDK contract with bundled GCP and OS keyring plugins, and updates docs, tests, generated baselines, changelog, labeler, and exports.

## Best Possible Solution

Keep the additive SecretRef/SDK direction, but resolve non-built-in source ownership through the effective enabled plugin set for bundled, installed, and workspace plugins, then load only the enabled owner’s narrow secret-provider artifact with the same boundary and activation checks.

## Reproduction Assessment

Yes. A static reproduction is clear: install or workspace-add a plugin declaring contracts.secretProviders for a source such as aws, configure secrets.providers with source aws, and the PR schema/docs accept it while resolveBundledSecretProviderForSource returns no owner and the runtime throws Unknown secret provider source.

## Solution Assessment

No. The manifest-first direction is maintainable, but the current PR only makes bundled secret-provider plugins executable; the advertised third-party seam needs an activation-aware installed/workspace owner lookup and public-artifact loader.

## Review Findings
