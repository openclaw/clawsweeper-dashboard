## Summary

Keep open. This recent feature PR still has a blocking correctness gap: the branch advertises plugin-owned SecretRef sources, including third-party plugins, but the latest resolver only loads bundled plugin owners/artifacts, and the PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds a SecretRef provider plugin SDK seam, open SecretRef config and Gateway wire schemas, bundled GCP and OS keyring provider plugins, docs, tests, labeler, changelog, and generated baseline updates.

## Best Possible Solution

Land only after the branch either loads enabled bundled, installed, and workspace secret-provider plugins through the plugin registry/public-artifact boundary, or intentionally narrows the user-facing contract and docs to a bundled-only first step with the third-party seam tracked separately.

## Reproduction Assessment

Yes for the review finding. Source inspection of the latest PR head shows arbitrary plugin-owned sources can validate through schema/docs, while runtime lookup only consults bundled ownership and bundled artifacts.

## Solution Assessment

No. The manifest-first direction fits the plugin architecture, but the proposed implementation is not the best complete solution while advertised third-party providers can validate but cannot load at runtime.
