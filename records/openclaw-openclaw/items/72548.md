## Summary

Keep open: the feature direction is useful, but the latest PR head still has blocking correctness and validation issues, failing dependency checks, merge conflicts, and no inspectable real behavior proof from the contributor.

## What This Changes

The PR adds a manifest-declared SecretRef provider contract, a public plugin SDK subpath, bundled GCP and OS keyring secret-provider plugins, schema/protocol/doc updates, generated baselines, tests, and type-guard updates across existing SecretRef callers.

## Best Possible Solution

Land a security-reviewed SecretRef provider seam that either genuinely supports enabled bundled, installed, and workspace plugin providers, or narrows the first step to bundled GCP/keyring providers with follow-up issues for third-party loading and broader SecretRef field coverage.

## Reproduction Assessment

Yes for the PR review blockers: CI reproduces the dependency and cron diagnostic failures, and source inspection shows the docs/schema advertise third-party sources while runtime lookup is bundled-only. This is not a current-main bug reproduction because the PR is additive feature work.

## Solution Assessment

No. The manifest-first direction fits OpenClaw's plugin architecture, but this diff over-advertises third-party support, changes an existing validation diagnostic contract, fails dependency checks, and lacks inspectable real behavior proof.

  - body: The docs and open SecretRef source shape now describe third-party plugin-owned sources, but runtime lookup only resolves bundled plugin contracts and bundled public artifacts.   - confidence: 0.9

  - confidence: 0.9

Preserve the old field-level error for objects that are not valid SecretRefs, or get maintainer approval to change that contract and tests.
