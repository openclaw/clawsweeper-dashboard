## Summary

Keep open: the feature direction is useful, but the latest head still advertises a third-party SecretRef provider seam while runtime lookup only loads bundled providers, and the external PR lacks inspectable real behavior proof.

## What This Changes

The branch opens SecretRef source/config/wire schemas to plugin-owned sources, adds a secret-provider SDK subpath and bundled GCP/keyring plugins, updates docs/tests/baselines, and routes non-built-in SecretRefs through a lazy bundled provider resolver.

## Best Possible Solution

Land this only after the seam either supports enabled bundled, installed, and workspace plugins through the plugin registry/public-artifact boundary, or the docs/types/changelog are narrowed to bundled-only with third-party support tracked separately.

## Reproduction Assessment

Yes for the review finding. Source inspection of the latest head shows plugin-owned sources validate through schema/docs, but resolution still calls only bundled owner and bundled artifact loaders, so a non-bundled plugin source can validate and then fail at runtime.

## Solution Assessment

No. The manifest-first direction fits the plugin architecture, but this is not the best complete solution while the advertised third-party provider seam cannot load installed or workspace plugins.
