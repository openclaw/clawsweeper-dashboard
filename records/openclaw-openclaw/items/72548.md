## Summary

Keep open. Current main has not implemented pluggable SecretRef sources, and the PR addresses a real plugin-SDK gap, but the proposed runtime path is not mergeable yet because it can load disabled/default-off bundled secret provider plugins and does not resolve installed or workspace secret-provider plugins despite the public SDK direction.

## What This Changes

This PR adds a manifest-declared SecretRef provider contract, a `plugin-sdk/secret-provider` subpath, open config and Gateway wire schemas, lazy secret-provider dispatch, bundled GCP and OS keyring secret provider plugins, and related docs/tests/changelog/labeler updates.

## Best Possible Solution

Revise the PR so secret-provider source ownership is resolved against the effective enabled plugin set, including installed/workspace/config plugins when the SDK docs promise third-party providers, and pass the needed config/activation context from the secrets resolver. Keep the additive schema/protocol shape and leave broader raw env/config field coverage to a separate tracked follow-up unless maintainers choose to expand scope.

## Reproduction Assessment

Yes. A static reproduction path is clear: with `plugins.enabled: false`, `plugins.deny: ["secrets-gcp"]`, or no explicit enablement for a default-off bundled secret plugin, a `{ source: "gcp" }` provider still reaches the bundled artifact loader; a third-party source such as `aws` is accepted by schema but cannot resolve because lookup is bundled-only.

## Solution Assessment

No. A manifest-first SecretRef provider seam is the right direction, but this patch is not the narrowest maintainable fix until it reuses activation-aware plugin resolution and supports the third-party plugin origins that the SDK/docs advertise.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin activation before loading providers:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The resolver loads the bundled owner for a referenced source without checking `plugins.enabled`, denylist entries, explicit `plugins.entries.<id>.enabled: false`, allowlist misses, or bundled default-off state. A disabled or denied secret plugin can still execute whenever a SecretRef source references it, bypassing the plugin activation controls used elsewhere.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The runtime lookup only calls the bundled contract registry and bundled public-artifact loader. A third-party plugin can declare `contracts.secretProviders` and the schemas will accept its source, but resolution still returns an unknown-source error, so AWS/Vault/Bitwarden-style providers still require core changes despite the advertised SDK seam.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a secret-provider code execution path that does not honor plugin disable/deny/default-off activation controls before loading bundled provider artifacts.

Concerns:

- **[medium] Secret providers bypass plugin activation controls:** `src/plugins/secret-provider-resolver.ts:41`

Operators who disable all plugins, deny a plugin, or leave a default-off secret plugin disabled would still execute that plugin when a SecretRef source references it.
