## Summary

Keep open: the activation-gate issue appears addressed, but the PR still advertises a third-party `contracts.secretProviders` seam while runtime lookup only resolves bundled plugin owners, so installed or workspace secret-provider plugins can validate but cannot resolve at runtime.

## What This Changes

The PR adds a `contracts.secretProviders` manifest and SDK contract, opens SecretRef config and gateway schemas, dispatches plugin-owned secret sources through a lazy public-artifact resolver, adds bundled GCP/keyring secret-provider plugins, and updates docs, tests, baselines, labeler, and changelog.

## Best Possible Solution

Keep the additive SecretRef/SDK direction, but resolve non-built-in source ownership through the effective enabled plugin metadata for bundled, installed, and workspace plugins, then lazy-load only the enabled owner’s narrow `secret-provider` artifact under the existing boundary checks.

## Reproduction Assessment

Yes. A static reproduction is clear: install or workspace-add a plugin declaring `contracts.secretProviders: ["aws"]`, configure `secrets.providers` with source `aws`, and the PR schema/docs accept it while the resolver only asks the bundled registry and then throws `Unknown secret provider source`.

## Solution Assessment

No. The manifest-first direction is maintainable, but the current implementation only makes bundled secret-provider plugins executable; the advertised third-party seam needs activation-aware installed/workspace owner lookup and artifact loading.
