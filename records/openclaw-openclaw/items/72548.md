## Summary

Keep open. Current main still has closed env/file/exec SecretRef support, and latest PR head fixes the activation bypass but still leaves the promised third-party plugin seam bundled-only at runtime.

## What This Changes

The PR opens SecretRef source/config/protocol schemas, adds a manifest-declared secret-provider SDK contract with bundled GCP and OS keyring provider plugins, and updates docs, tests, generated baselines, changelog, labeler, and exports.

## Best Possible Solution

Keep the additive SecretRef schema and SDK seam, but resolve non-built-in source ownership through the effective enabled plugin set for bundled, installed, and workspace plugins, then execute only the enabled owner’s narrow artifact.

## Reproduction Assessment

Yes. A static reproduction is clear: an installed or workspace plugin declaring contracts.secretProviders for a source such as aws can be indexed and schema-accepted, but resolution still calls the bundled-only resolver and fails as an unknown source.

## Solution Assessment

No. The manifest-first direction is maintainable, but the current implementation is not complete until the runtime path supports enabled installed/workspace owners as well as bundled owners.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Load installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:67-84`
  - body: The SDK/docs/schema now accept arbitrary plugin-owned sources, and installed plugin manifests are indexed with `contracts.secretProviders`, but runtime lookup only asks the bundled contract registry and loads bundled public artifacts. An installed or workspace plugin declaring `contracts.secretProviders: ["aws"]` will validate but still resolve as `Unknown secret provider source`, so the promised third-party seam is unusable.
