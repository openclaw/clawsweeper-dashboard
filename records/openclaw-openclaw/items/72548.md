## Summary

Keep open. Current main has not implemented pluggable SecretRef providers, and this PR is the active implementation path, but the proposed patch still has blocking correctness/security issues around activation-aware provider loading and third-party plugin resolution.

## What This Changes

The PR opens SecretRef source/config/protocol schemas, adds a manifest-declared `contracts.secretProviders` SDK seam, adds bundled GCP and OS keyring secret provider plugins, and updates resolver dispatch, docs, tests, changelog, labeler, exports, and generated baselines.

## Best Possible Solution

Revise the secret-provider resolver so source ownership is resolved against the effective enabled plugin set, including installed and workspace plugins when their manifests declare `contracts.secretProviders`, and execute only the enabled owner’s narrow artifact. Keep the additive schema/protocol shape and leave broader raw env/config field coverage as a separate follow-up unless maintainers choose to expand scope.

## Reproduction Assessment

Yes. A static reproduction path is clear: at PR head, `plugins.enabled: false`, `plugins.deny: ["secrets-gcp"]`, or leaving a default-off bundled provider disabled still allows `{ source: "gcp" }` to reach the bundled artifact loader; a third-party source such as `aws` is schema-accepted but cannot resolve because lookup is bundled-only.

## Solution Assessment

No. The manifest-first SecretRef provider seam is the right direction, but this patch is not the best maintainable implementation until it reuses activation-aware plugin resolution and supports the third-party plugin origins promised by the SDK/docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin activation before loading providers:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The resolver maps a SecretRef source directly to a bundled plugin and loads its artifact without seeing config or activation state. Because the new `secrets-gcp` and `secrets-keyring` manifests are default-off, configs that globally disable plugins, deny the plugin, disable its entry, or omit explicit enablement can still execute the provider when a SecretRef references its source.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The schemas and docs accept third-party secret-provider sources, but runtime lookup only calls the bundled contract registry and bundled public-artifact loader. An installed or workspace plugin can declare `contracts.secretProviders`, pass validation, and still fail as an unknown source, so AWS/Vault/Bitwarden-style providers still require core changes.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a secret-provider code execution path that can bypass plugin activation controls before loading bundled provider artifacts.

Concerns:

- **[medium] Secret providers bypass plugin activation controls:** `src/plugins/secret-provider-resolver.ts:41`
  - body: Secret provider artifacts can import cloud SDKs or spawn credential tools, but the new resolver loads them based only on bundled manifest ownership. Operators who disable all plugins, deny a plugin, or leave a default-off secret plugin disabled would still execute that plugin when a SecretRef source references it.
  - confidence: 0.86
