## Summary

Keep open. The PR should not merge yet because the advertised third-party SecretRef provider seam still resolves only bundled provider artifacts, and the external real-behavior proof is narrative-only rather than inspectable output or artifacts.

## What This Changes

The PR adds a SecretRef provider plugin SDK seam, opens config and Gateway schemas for plugin-owned source ids, and adds bundled GCP and OS keyring secret-provider plugins with docs and tests.

## Best Possible Solution

Route non-built-in SecretRef sources through the effective enabled plugin metadata and a public-artifact loader that supports bundled, installed, and workspace plugins, or narrow the contract/docs to bundled-only and leave the third-party seam tracked in a canonical follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an installed or workspace plugin can declare a plugin-owned source that the PR schemas and docs accept, but runtime lookup only asks the bundled resolver and then fails as an unknown source.

## Solution Assessment

No. The manifest-first direction is maintainable, but this implementation is not the best complete solution while non-bundled provider plugins cannot be resolved through the SDK seam it advertises.
