## Summary

Keep open. Current main still restricts SecretRef sources to env/file/exec, so the work is not obsolete, but the PR head has blocking runtime issues: provider artifact loading ignores plugin activation controls and only resolves bundled providers despite advertising third-party plugin sources.

## What This Changes

This PR adds a manifest-declared SecretRef provider contract, a public `plugin-sdk/secret-provider` subpath, open config and Gateway wire schemas, lazy secret-provider dispatch, bundled GCP and OS keyring secret provider plugins, and related docs/tests/changelog/labeler updates.

## Best Possible Solution

Revise the PR so SecretRef source ownership is resolved against the effective enabled plugin set, including installed/workspace/config plugins when the SDK documents third-party providers, and pass the needed config and activation context from the secrets resolver. Keep the schema/protocol opening additive and leave broader raw env/config field SecretRef coverage to a separate tracked follow-up unless maintainers expand scope.

## Reproduction Assessment

Yes. A static reproduction path is high-confidence: with the PR head, a config that disables or denies `secrets-gcp` can still reach the bundled artifact loader for `{ source: "gcp" }`, and a third-party source such as `aws` is accepted by the new schema/docs but cannot resolve because lookup is bundled-only.

## Solution Assessment

No. The manifest-first SecretRef provider seam is the right direction, but this implementation is not the best fix until it reuses activation-aware plugin resolution and supports the third-party plugin origins that the new public SDK surface advertises.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin activation before loading providers:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The resolver maps a SecretRef source directly to a bundled plugin id and loads that artifact without any config or activation context. Since `resolvePluginSecretRefs` passes only the source string, a disabled, denied, or default-off secret plugin can still execute when a config references its source, bypassing the plugin activation controls used elsewhere.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: Runtime lookup only calls the bundled contract registry and bundled public-artifact loader. A third-party plugin can declare `contracts.secretProviders` and the schemas/docs will accept its source, but resolution still returns unknown-source, so AWS/Vault/Bitwarden-style providers still need core changes despite the advertised SDK seam.
  - confidence: 0.88
