## Summary

Keep this PR open. The direction is useful, and prior discussion correctly identifies the SecretRef provider gap, but the current branch still advertises a third-party plugin seam while the runtime can only resolve bundled plugin owners; it also still needs inspectable real behavior proof before merge.

## What This Changes

The PR adds a SecretRef provider plugin SDK seam, open SecretRef config and Gateway wire schemas, bundled GCP and OS keyring provider plugins, docs, tests, and generated baselines.

## Best Possible Solution

Land this only after the branch either supports enabled bundled, installed, and workspace plugins through the plugin registry/public-artifact boundary, or deliberately narrows the docs/types/changelog to bundled-only and tracks third-party support separately.

## Reproduction Assessment

Yes for the review finding. Source inspection of the latest PR head shows a non-bundled plugin source can validate through the new open schema and docs, but runtime lookup only consults bundled contract and bundled artifact loaders.

## Solution Assessment

No. The manifest-first direction fits the plugin architecture, but this is not the best complete solution while the advertised third-party provider seam cannot load installed or workspace plugins.
