## Summary

Keep open. Current main still has a closed `env | file | exec` SecretRef contract, and this PR is a real implementation attempt, but the current head has blocking runtime/plugin-boundary issues: secret-provider resolution bypasses plugin activation controls and only loads bundled providers while the SDK/docs advertise third-party plugin sources. The live PR state is also dirty against main, so this needs author/maintainer follow-up before merge.

## What This Changes

This PR adds a manifest-declared secret-provider contract, a `plugin-sdk/secret-provider` subpath, open SecretRef config and Gateway wire schemas, lazy secret-provider dispatch, bundled GCP and OS keyring provider plugins, and related docs/tests/changelog/labeler updates.

## Best Possible Solution

Revise the PR so secret-provider resolution is activation-aware and resolves source ownership for the effective enabled plugin set, including installed/workspace/config plugins if the SDK docs continue to promise third-party providers. Keep the additive schema/protocol shape, and track the broader #71593 raw env/config field-coverage work separately unless maintainers explicitly decide to include it here.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is clear: with `plugins.enabled: false`, `plugins.deny: ["secrets-gcp"]`, or no `plugins.entries.secrets-gcp.enabled`, a `{ source: "gcp" }` provider still reaches the bundled artifact loader because no activation state is passed; a third-party source such as `aws` cannot resolve because lookup filters to bundled manifests only.

## Solution Assessment

No. The manifest-first secret-provider seam is the right general direction, but this implementation is not mergeable until runtime loading honors plugin activation controls and supports the plugin origins the SDK/docs advertise.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor plugin activation before loading providers:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The secret-provider resolver loads any bundled manifest owner for a referenced source without checking `plugins.enabled`, `plugins.deny`, `plugins.entries.<id>.enabled`, or the plugin's default-off state. A disabled or denied bundled plugin such as `secrets-gcp` can still execute whenever `secrets.providers` references its source, bypassing the plugin activation controls used elsewhere.
  - confidence: 0.9
- **[P2] Resolve installed secret-provider plugins:** `src/plugins/secret-provider-resolver.ts:41-52`
  - body: The runtime lookup only calls the bundled contract registry and bundled public-artifact loader. A third-party plugin can declare `contracts.secretProviders` and the config schema will accept its source, but resolution still returns `Unknown secret provider source`, so AWS/Vault/etc. providers still require core changes despite the advertised SDK seam.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a secret-provider code execution path that does not honor plugin disable/deny/default-off activation controls before loading bundled provider artifacts.

Concerns:

- **[medium] Secret providers bypass plugin activation controls:** `src/plugins/secret-provider-resolver.ts:41`
  - body: Secret provider artifacts can import cloud SDKs or spawn platform credential tools, but the new resolver loads them based only on bundled manifest ownership. Operators who disable all plugins, deny a plugin, or leave an `enabledByDefault: false` secret plugin disabled would still execute that plugin if a SecretRef source references it.
