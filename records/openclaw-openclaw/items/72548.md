## Summary

Keep open. Current main still has the closed built-in SecretRef resolver, and the PR’s latest diff still leaves the advertised third-party secret-provider seam incomplete because runtime lookup is bundled-only.

## What This Changes

This PR opens SecretRef source/config/wire schemas, adds a manifest-declared secret-provider SDK subpath with bundled GCP and keyring reference plugins, and updates docs, tests, generated baselines, package exports, and labeler/changelog metadata.

## Best Possible Solution

Complete the SDK seam by resolving enabled bundled, installed, and workspace owners for contracts.secretProviders, then lazy-load only the enabled owner’s narrow secret-provider artifact under the existing plugin boundary checks.

## Reproduction Assessment

Yes for the review finding. A static reproduction is clear from the PR diff: an installed/workspace plugin can declare contracts.secretProviders for a source such as aws, config/schema accepts that source, and runtime lookup only calls the bundled resolver path before throwing unknown source.

## Solution Assessment

No. The manifest-first direction is maintainable, but the current PR only makes bundled secret-provider plugins executable while advertising a third-party SDK seam.
