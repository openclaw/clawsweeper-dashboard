## Summary

Keep open: this is useful feature work, but the PR still has blocking correctness/CI issues and insufficient inspectable real behavior proof for a security-sensitive secrets path.

## What This Changes

The PR adds a SecretRef plugin-provider SDK seam, opens SecretRef/config/wire schemas to plugin-owned sources, adds GCP and OS keyring bundled plugins, and updates docs, tests, baselines, labels, and changelog.

## Best Possible Solution

Land a complete, additive secret-provider seam that either resolves enabled installed/workspace plugin public artifacts or clearly narrows the public contract to bundled providers until external support is implemented.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new resolver only checks bundled plugin contracts, and CI logs show check-dependencies failing on the new GCP dependency. Feature end-to-end proof from the contributor is not yet inspectable.

## Solution Assessment

No. The direction is plausible, but the current patch should not merge until the public SDK seam matches installed/workspace plugin expectations or the exposed contract is narrowed, and CI plus real behavior proof are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve installed plugin secret providers:** `src/plugins/secret-provider-resolver.ts:67-83`
  - body: This new SDK seam is documented and typed as plugin-facing, but resolution only calls the bundled contract registry and bundled artifact loader. An installed or workspace plugin can declare `contracts.secretProviders` and pass manifest/config validation, then every SecretRef for that source fails at runtime as unknown. Either resolve enabled installed/workspace public artifacts here or narrow the public contract/docs until that support exists.
  - confidence: 0.92
- **[P2] Make the GCP dependency visible to Knip:** `extensions/secrets-gcp/secret-provider.ts:36`
  - body: `check-dependencies` is red because Knip reports `extensions/secrets-gcp/package.json`'s `@google-cloud/secret-manager` dependency as unused. The runtime import is hidden behind `import(GCP_MODULE)`, so the dependency gate cannot see it; use a literal dynamic import or the repo's dependency-ignore pattern for this convention before merge.
  - confidence: 0.95
