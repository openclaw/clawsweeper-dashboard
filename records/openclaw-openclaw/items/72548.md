## Summary

Keep open. The branch moves main's closed SecretRef model toward a plugin-backed seam and now gates bundled artifact execution on plugin activation, but the advertised third-party secret-provider path still cannot run because runtime ownership and artifact loading are bundled-only.

## What This Changes

The PR opens SecretRef config/protocol schemas for plugin-owned sources, adds a secret-provider SDK/manifest contract, ships bundled GCP and OS keyring providers, and updates docs, tests, generated baselines, labeler, and changelog.

## Best Possible Solution

Keep the additive SecretRef/SDK direction, but resolve non-built-in source ownership through the effective enabled plugin set for bundled, installed, and workspace plugins, then load only the enabled owner's narrow secret-provider artifact with the same boundary checks.

## Reproduction Assessment

Yes. A static reproduction is clear: add an installed or workspace plugin declaring contracts.secretProviders for a source such as aws, configure secrets.providers with source aws, and the PR schemas accept it while the bundled-only resolver returns no owner and runtime throws Unknown secret provider source.

## Solution Assessment

No. The bundled activation-gated path is sound for the two new bundled plugins, but it is not the best complete solution for the advertised third-party SDK seam because non-bundled owner lookup and artifact loading are missing.
