## Summary

Current main already provides the headline Shift+Enter-newline and Enter-submit behavior through the pinned editor, but meaningful work remains: OpenClaw does not document the shortcut or terminal fallbacks and exposes no configurable send/newline preference. The closed, unmerged implementation PR does not supersede this work.

Likely related people:
- Vincent Koc — recent contributor across the current TUI wrapper, keybinding registry, CLI registration, and documentation paths; high confidence.
- steipete — prior reviewer who identified the remaining documentation and product-decision scope; medium confidence.

## What This Changes

Add reliable multiline TUI composition with Enter-to-send, document the newline shortcut, and optionally allow users to configure send versus newline keybindings.

## Best Possible Solution

Document current Shift+Enter and backslash-enter behavior in TUI help and public docs, then either expose `tui.input.submit` and `tui.input.newLine` through the existing keybinding architecture or explicitly decline configuration; track webchat composer policy separately.

## Reproduction Assessment

Yes. Current source and the pinned dependency prove that Shift+Enter already inserts a newline, while the missing TUI documentation and configurable keybinding surface are directly visible on current main.

## Solution Assessment

Not fully decided. Reusing the existing pi-tui/OpenClaw keybinding registry is the narrowest maintainable path; adding a standalone `enterSends` toggle or mixing webchat into this TUI issue would create parallel policy.
