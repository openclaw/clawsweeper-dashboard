## Summary

Keep open. Current main has dependency-backed multiline input support for Shift+Enter, but the OpenClaw TUI still omits the shortcut from docs and `/help`, and there is still no OpenClaw CLI/config bridge for choosing Enter/Ctrl+Enter/Shift+Enter behavior.

## What This Changes

Request TUI multiline composition where Enter sends, Shift+Enter inserts a newline, optional configurable send/newline keybindings are supported, and the behavior is documented in TUI help/docs.

## Best Possible Solution

Keep the existing dependency-backed multiline behavior, document Shift+Enter and fallback newline composition in `docs/web/tui.md` and TUI `/help`, then decide whether OpenClaw should expose a stable config/CLI keybinding bridge before adding preference wiring.

## Reproduction Assessment

Yes for the remaining gap: current-main source inspection shows docs and `/help` omit newline composition, and the CLI/runtime option surface has no send-keybinding preference. The input behavior itself is dependency-backed, so a live terminal check would mainly validate terminal-specific sequences rather than the missing OpenClaw surfaces.

## Solution Assessment

Unclear. Documenting the current Shift+Enter/backslash fallback is the narrow safe step, but adding configurable send/newline behavior should wait for a maintainer decision on the public config and CLI contract.
