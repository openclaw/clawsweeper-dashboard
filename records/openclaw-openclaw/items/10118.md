## Summary

Keep open: current main and v2026.6.5 have baseline Shift+Enter newline insertion through pi-tui, but the TUI docs and `/help` still omit the shortcut and OpenClaw still has no approved configurable send/newline keybinding surface. The later WebChat request in the discussion is already tracked separately at https://github.com/openclaw/openclaw/issues/8147.

## What This Changes

The issue requests TUI multiline composition with Enter kept as send, Shift+Enter or equivalent inserting newlines, optional configurable keybindings, and docs/help coverage.

## Best Possible Solution

Keep the pi-tui newline/submit implementation, document the existing shortcut and fallback behavior in the TUI guide and `/help`, and add a configurable keybinding bridge only after maintainers approve the public TUI policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a TUI UX/config/docs request. Source and dependency inspection show baseline Shift+Enter newline behavior is present via pi-tui, while docs/help and configurable keybinding support remain absent.

## Solution Assessment

Unclear until maintainers decide whether OpenClaw should expose a public keybinding preference. The maintainable path is docs/help coverage plus an approved pi-tui keybinding bridge, not a duplicate custom Shift+Enter handler.
