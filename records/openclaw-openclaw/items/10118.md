## Summary

Keep open: current main and v2026.6.10 already have dependency-backed default Shift+Enter newline support, but the requested TUI docs, /help coverage, and configurable send/newline keybinding policy remain unresolved.

## What This Changes

The issue requests TUI multiline composition with Enter sending by default, Shift+Enter inserting a newline, optional configurable send/newline keybindings, and help/docs coverage.

## Best Possible Solution

Document the current Shift+Enter and fallback newline behavior in the TUI guide and /help; if maintainers approve configurable shortcuts, reuse pi-tui's keybinding model while keeping Enter-to-send as the default.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.10 provide dependency-backed Shift+Enter newline behavior, but TUI docs, /help, CLI, and config types do not expose the requested documentation or configurable send/newline bridge. I did not run an interactive PTY session in this read-only review.

## Solution Assessment

Unclear until maintainers decide the public keybinding policy. The narrowest maintainable path is to document pi-tui's existing behavior and reuse its keybinding model if a preference is approved, rather than adding a one-off OpenClaw toggle.
