## Summary

Current main and shipped releases already provide the issue’s central behavior: the TUI delegates to pi-tui, where Shift+Enter inserts a newline before Enter’s submit path, with terminal-sequence and backslash-Enter fallbacks. The optional configurable-binding idea was never established as required core scope, and the remaining documentation omission is minor enough not to keep the broad implementation request open.

## What This Changes

Support Shift+Enter for multiline TUI composition while keeping Enter as send, with optional configurable bindings and shortcut documentation.

## Best Possible Solution

Keep the shipped pi-tui-native Shift+Enter behavior and terminal fallbacks; if maintainers want configurable submit bindings, track that as a narrower product decision rather than adding duplicate editor handling.

## Reproduction Assessment

Yes. Current source delegates to the exact published pi-tui dependency, whose editor maps Shift+Enter to newline and Enter to submit before the submit branch, including terminal fallbacks.

## Solution Assessment

Yes. Using pi-tui’s canonical keybinding and editor implementation is narrower and more maintainable than adding an OpenClaw-specific Shift+Enter handler; configurable bindings can be considered separately if desired.
