## Summary

Keep open: current main already gets Shift+Enter newline composition from the pinned `pi-tui` editor, but the user-facing TUI docs and `/help` still omit newline composition and there is no OpenClaw config/CLI surface for send/newline keybinding preferences.

## What This Changes

The issue requests TUI multiline composition with Enter sending, Shift+Enter inserting a newline, optional configurable keybindings, and docs/help coverage.

## Best Possible Solution

Document the current newline and fallback behavior in the TUI guide and `/help`, then expose a stable send/newline keybinding preference only after maintainers choose the config or CLI contract.

## Reproduction Assessment

Not applicable as a feature request; there is no broken current contract to reproduce. Source inspection establishes dependency-backed Shift+Enter support plus remaining docs/help/config gaps.

## Solution Assessment

Not yet. The dependency behavior is the right low-level path, but the maintainable completion is docs/help plus a deliberate keybinding configuration design rather than a redundant `CustomEditor`-only handler.
