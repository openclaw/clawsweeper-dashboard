## Summary

Keep this issue open. Current main and v2026.6.6 already provide the headline Shift+Enter-newline and Enter-submit behavior through `@earendil-works/pi-tui`, but OpenClaw still does not document that shortcut/fallback behavior in the TUI guide or `/help`, and it has no product-approved CLI/config bridge for configurable send/newline keybindings. The closed implementation PR at https://github.com/openclaw/openclaw/pull/43327 does not supersede that remaining work.

## What This Changes

The issue requests multiline TUI composition with Enter sending by default, Shift+Enter inserting a newline, optional configurable send/newline keybindings, and help/docs coverage.

## Best Possible Solution

Document the current Shift+Enter and fallback newline behavior in the public TUI guide and `/help`, then decide whether OpenClaw should expose the existing pi-tui keybinding registry as the canonical configurable TUI keybinding surface; track WebChat composer policy separately.

## Reproduction Assessment

Yes, for a source-level capability gap: current main and v2026.6.6 expose Shift+Enter newline through pi-tui but omit the requested TUI help/docs and configurable send/newline bridge. I did not run an interactive PTY session in this read-only review.

## Solution Assessment

Unclear until maintainers decide the public keybinding policy. Reusing the existing pi-tui/OpenClaw keybinding registry is the narrowest maintainable direction, while a separate `enterSends` boolean or mixing WebChat into this TUI item would create parallel policy.
