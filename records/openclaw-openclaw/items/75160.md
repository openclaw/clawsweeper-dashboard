## Summary

Keep open. Current main already has the Gateway tool-invoke fallback behavior, but it does not yet have this memory-specific regression coverage; the PR is a focused test-only addition with no blocking review findings and public head checks showing no failures.

## What This Changes

This PR modifies only `src/gateway/tools-invoke-http.test.ts` to add mocked memory tools, record tool-factory contexts, and assert HTTP `/tools/invoke` retries plugin-enabled resolution for `memory_search` while ordinary core tools stay plugin-disabled.

## Best Possible Solution

Merge this focused regression coverage after normal maintainer review, keeping the already-present runtime fallback unchanged. A real memory-plugin integration smoke can remain a separate follow-up if maintainers want broader end-to-end proof.

## Reproduction Assessment

Yes. The PR creates a deterministic unit reproduction by making `memory_search` a cataloged core ID that is absent while plugin tools are disabled, then verifying `/tools/invoke` retries with plugin tools enabled and succeeds.

## Solution Assessment

Yes. Since production fallback behavior is already on current main, targeted regression coverage is the narrowest maintainable solution; adding runtime changes here would increase risk without addressing the remaining test gap.
