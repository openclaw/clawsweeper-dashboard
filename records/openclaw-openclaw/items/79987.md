## Summary

Keep open: the latest PR head still adds the right cache-hit direction but shortcuts managed npm plugin validation, security scanning, and install-record provenance, so it should not merge as-is.

## What This Changes

The PR adds an install-mode fast path in the managed npm plugin installer plus focused tests to skip npm install when the existing package version is greater than or equal to the requested version.

## Best Possible Solution

Keep the cache-hit behavior, but validate the existing package through the normal installed-package and scan contract, return installed version/extension data, persist accurate installed npm provenance, and prove the real CLI or wizard path.

## Reproduction Assessment

Yes, from source inspection: current main resolves the managed npm install root and rejects an existing package directory in install mode before comparing installed and requested versions. I did not run a live Docker wizard reproduction in this read-only review.

## Solution Assessment

No. The cache-hit direction is maintainable, but this implementation is not the best fix because it accepts version-only package directories and records requested metadata instead of validating and recording the accepted installed package.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate cached packages before returning success:** `src/plugins/install.ts:482-489`
  - body: This branch returns success after reading only package.json.version. A preexisting target can lack OpenClaw extensions, have a mismatched manifest id, fail expectedPluginId, have bad lockfile provenance, or be blocked by the install scans, but install mode will still enable and record it. Reuse the normal installed-package validation/security path before treating the cache hit as installed.
  - confidence: 0.9
- **[P2] Record installed package metadata on cache hits:** `src/plugins/install.ts:484-488`
  - body: When a newer installed package satisfies an older requested spec, the result has no installed version/extensions and carries params.npmResolution for the older request. Onboarding records result.version and buildNpmResolutionInstallFields(result.npmResolution), so config can persist missing or stale provenance and later update checks can target the wrong artifact.
  - confidence: 0.86
