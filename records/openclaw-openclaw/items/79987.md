## Summary

Keep open: the linked bug is source-reproducible, but this branch still returns success from a version-only directory check, bypassing the managed npm validation/scan path and recording requested rather than installed metadata; contributor proof is still mock-only.

## What This Changes

The PR adds an install-mode fast path that reads an existing managed npm package version, skips npm install when installed is at least the requested version, and adds focused unit coverage.

## Best Possible Solution

Keep the version-satisfies skip direction, but route cache hits through the existing installed-package validation/scan path and return metadata from the accepted installed package before requiring live CLI or wizard proof.

## Reproduction Assessment

Yes. Current main source shows install mode reaches the duplicate-target guard before comparing the installed package version, matching the linked wizard report; I did not run a live Docker wizard reproduction in this read-only review.

## Solution Assessment

No. The narrow cache-hit idea fits the bug, but this implementation is not the best fix because it trusts package.json.version alone and returns requested metadata instead of validating and recording the installed package.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate cached packages before returning success:** `src/plugins/install.ts:482-489`
  - body: This branch reports a successful install after reading only package.json.version. A preexisting target can have the wrong plugin id, missing OpenClaw entries, incompatible host metadata, or scan-blocked contents, but install mode would still enable and record it. Reuse the normal installed-package validation and scan path before treating the directory as installed.
  - confidence: 0.9
- **[P2] Return installed package metadata on cache hits:** `src/plugins/install.ts:484-487`
  - body: When a newer installed version satisfies an older requested spec, the result keeps the requested npmResolution and empty extensions instead of the accepted package's version, extensions, and provenance. The wizard records these fields, so config can persist stale metadata and later update checks can target the wrong artifact.
  - confidence: 0.86
