## Label Justifications

- `P1`: The report describes an active Discord voice workflow becoming unresponsive because empty STT segments consume serialized agent turns.
- `impact:message-loss`: The affected behavior delays or suppresses useful Discord voice replies by filling the voice processing queue with empty transcript turns.

## Summary

Keep open: current main and latest release still allow empty sherpa JSON to fall back to raw stdout and reach the Discord voice agent turn, and https://github.com/openclaw/openclaw/pull/84667 is an open closing PR that should be reviewed or landed before this issue closes.

## What This Changes

The issue requests filtering empty sherpa/moonshine STT JSON before Discord voice sends it to the LLM, with a possible later queue-depth or stale-segment safeguard.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/84667 or an equivalent shared media-understanding parser fix that treats matched empty sherpa JSON as no transcript, then evaluate queue policy separately only if needed.

## Reproduction Assessment

Yes. From source inspection, sherpa stdout shaped like `{"text":""}` returns no parsed text, falls back to raw stdout, passes Discord's `!transcript` guard, and enters the serialized voice queue; I did not run a live Discord/moonshine session.

## Solution Assessment

Yes for the central bug. Fixing shared sherpa CLI normalization before `runCliEntry` emits an audio transcription is the narrowest maintainable path; queue-depth or stale-discard behavior should be a separate follow-up only if maintainers want it.
