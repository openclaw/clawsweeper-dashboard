## Summary

Keep open: current main still has the source-level empty sherpa JSON fallback, and the linked closing PR is open rather than merged.

## What This Changes

The issue asks OpenClaw to skip sherpa/moonshine STT outputs whose structured JSON `text` is empty before Discord voice queues an agent turn.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84667 or an equivalent shared media-understanding parser fix, then close this issue; keep any queue-depth policy as a separate follow-up.

## Reproduction Assessment

Yes, source-reproducible: sherpa stdout shaped like `{"text":""}` misses extraction, falls back to raw stdout, and passes Discord's `!transcript` guard. I did not run a live Discord/moonshine session.

## Solution Assessment

Yes for the central bug: fixing shared sherpa CLI normalization before an audio output is emitted is the narrowest maintainable path. Queue-depth or stale-discard policy should be considered separately if maintainers still want it.
