## Summary

Keep open: current main still has a source-reproducible path where sherpa/moonshine empty structured output can become a non-empty raw JSON transcript, pass the Discord voice falsy check, and enter the serialized agent-turn queue.

## What This Changes

The issue asks to filter empty moonshine/sherpa STT JSON before Discord voice sends a transcript to the LLM and to prevent empty segments from clogging the serialized voice processing queue.

## Best Possible Solution

Normalize sherpa/moonshine structured STT output so empty `text` fields produce no transcript before Discord voice builds an agent prompt, add regression tests for empty JSON and the Discord voice skip path, and treat queue-depth policy as a follow-up only if stalls remain.

## Reproduction Assessment

Yes for source-level reproduction: with sherpa stdout containing `{"text":""}`, the parser returns null, the CLI path falls back to raw stdout, and Discord voice treats that raw JSON as a non-empty transcript. I did not run a live moonshine/Discord voice session in this read-only sweep.

## Solution Assessment

Yes: the narrow maintainable fix is to make structured empty STT output normalize to no transcript before the Discord voice prompt path. Queue limits can remain a separate follow-up unless empty-transcript filtering proves insufficient.
