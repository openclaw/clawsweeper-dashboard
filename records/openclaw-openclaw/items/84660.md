## Label Justifications

- `P1`: The report describes an active Discord voice workflow becoming unresponsive because empty STT segments consume serialized agent turns.
- `impact:message-loss`: The affected behavior delays or suppresses useful Discord voice replies by filling the voice processing queue with empty transcript turns.

## Summary

Keep open: current main still has a source-level path where sherpa-onnx empty JSON output can become a non-empty transcript string and enter the serialized Discord voice processing lane.

## What This Changes

The issue requests filtering empty Discord voice STT transcripts from sherpa/moonshine before they are sent to the LLM, with queue-stall prevention as a possible follow-up.

## Best Possible Solution

Fix the shared media-understanding sherpa CLI normalization so structured outputs with an empty `text` produce no transcript before Discord voice queues an agent turn.

## Reproduction Assessment

Yes, from source inspection: mock sherpa stdout shaped like `{"text":""}` would miss `extractSherpaOnnxText`, fall back to raw stdout, pass Discord's `!transcript` guard, and be sent to the agent. I did not run a live Discord/moonshine reproduction.

## Solution Assessment

Yes for the central bug: the narrow maintainable fix is to make sherpa structured empty transcripts resolve to no CLI output and add regression coverage. Queue-depth or stale-discard policy should stay out of the first fix unless maintainers explicitly choose it.

## Repair Work Prompt

Fix the Discord stt-tts empty sherpa transcript bug reported at https://github.com/openclaw/openclaw/issues/84660. Current main parses sherpa JSON only when `text` is non-empty, then falls back to raw stdout, so stdout like `{"text":"","tokens":[]}` becomes a transcript and can block later voice segments in `entry.processingQueue`. Change media-understanding CLI output normalization so `sherpa-onnx-offline` JSON with an empty `text` field produces no output/null instead of raw JSON, while preserving non-JSON stdout fallback for other CLIs and non-empty sherpa JSON extraction. Add focused regression coverage for empty and non-empty sherpa JSON. Do not add queue-depth or stale-segment discard policy in this PR unless maintainers ask; mention it as a follow-up if still needed. Add a user-visible changelog fix at landing time if repo policy requires it.
