## Summary

Keep open. Current main and the latest shipped validator still only support agent-mediated cron payloads, while the related direct-exec tracker remains open and the closest implementation PR was closed unmerged for design/security validation concerns.

## What This Changes

The issue requests a `staticMessage` cron payload that sends fixed text directly to a channel without invoking an AI session.

## Best Possible Solution

Define an additive no-token fixed-text cron delivery contract, or explicitly fold fixed text into the canonical no-LLM cron design, then update protocol, runtime, CLI, Control UI, agent-tool guidance, docs, and focused tests together.

## Reproduction Assessment

Not applicable for a feature request. Source inspection gives high confidence current main cannot express the proposed payload because the cron contract and user-facing surfaces accept only `systemEvent` or `agentTurn`.

## Solution Assessment

Unclear until maintainers choose the public cron API. A direct static payload is plausible for fixed text, but it should be reconciled with https://github.com/openclaw/openclaw/issues/18160 and existing outbound delivery semantics before implementation.
