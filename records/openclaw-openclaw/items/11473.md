## Label Justifications

- `impact:message-loss`: The requested feature is about scheduled fixed channel messages being delivered directly instead of relying on agent-mediated delivery.

## Summary

Keep open: current main and v2026.5.12 still only support `systemEvent` and `agentTurn` cron payloads, and related no-LLM cron/direct-invoke work is open, closed unmerged, or on a different operator surface rather than a shipped fixed-text channel delivery contract.

## What This Changes

Request a `staticMessage` cron payload that sends fixed text directly to a channel without an AI/model turn.

## Best Possible Solution

Define an additive no-token fixed-text cron delivery contract, or fold fixed-text delivery into the broader no-LLM cron design, then update protocol, runtime, CLI, Control UI, agent-tool guidance, docs, and focused tests together.

## Reproduction Assessment

Not applicable for a feature request. Source inspection gives high confidence that current main and v2026.5.12 cannot express or dispatch `payload.kind="staticMessage"`.

## Solution Assessment

Unclear until maintainers choose the public cron API. A dedicated static payload is plausible for fixed messages, but it should be reconciled with https://github.com/openclaw/openclaw/issues/18160 and existing outbound delivery semantics before implementation.
