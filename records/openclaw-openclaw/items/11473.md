## Summary

Keep open: current main still lacks a token-free static cron message payload, and the related direct-exec cron discussion is broader and still open rather than a shipped replacement. This is valid product/API work, not a cleanup close.

## What This Changes

The issue requests a cron payload that sends fixed text directly to a channel without invoking an AI model.

## Best Possible Solution

Define an additive no-token fixed-text cron delivery contract, or explicitly fold fixed text into the broader no-LLM cron design, then update protocol, runtime, CLI, Control UI, agent-tool guidance, docs, and focused tests together.

## Reproduction Assessment

Not applicable for a feature request. Source inspection on current main gives high confidence that the proposed payload cannot be expressed because cron payload contracts and user-facing surfaces only accept systemEvent or agentTurn.

## Solution Assessment

Unclear until maintainers choose the public cron API. A dedicated staticMessage payload is plausible for fixed messages, but it should be reconciled with the broader no-LLM cron design and existing delivery semantics.
