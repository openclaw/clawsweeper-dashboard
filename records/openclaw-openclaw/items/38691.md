## Summary

Keep this PR open. It targets a real Android Voice Tab gap, but the submitted patch is stale against current main and removes the existing no-recognizer hard stop without adding an equivalent no-backend safety path, so it should not merge as-is.

## What This Changes

The PR changes Android `MicCaptureManager` to continue after `SpeechRecognizer.isRecognitionAvailable(context)` is false, read `Settings.Secure` `voice_recognition_service`, create a targeted `SpeechRecognizer`, and fall back to the default recognizer.

## Best Possible Solution

Implement the OEM recognizer fallback in the current Android manual mic path, keep it vendor-agnostic, verify any configured `RecognitionService` component before targeting it, add the package-visibility query if needed, and preserve a clean unsupported-device path that disables the mic once instead of retrying indefinitely.

## Reproduction Assessment

Yes. The current-main source path shows the Voice Tab manual mic exits before recognizer creation when `isRecognitionAvailable(context)` is false, and the PR body provides a concrete Xiaomi 14 Pro positive-device smoke for the intended behavior.

## Solution Assessment

No, not as submitted. The configured-service fallback is the right narrow direction, but the patch needs to be rebased to the active package path and must add a verified no-backend disable path before removing the existing hard stop.
