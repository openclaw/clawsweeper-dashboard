## Summary

Keep this PR open because current main still has the Android Voice-tab hard block the PR is trying to fix, and the PR body gives a plausible Xiaomi OEM reproduction. The patch is not merge-ready: it removes the unavailable-recognizer stop path without adding a no-backend safety valve, and it still needs real after-fix proof beyond a written test claim.

## What This Changes

The PR changes Android mic capture to continue when `SpeechRecognizer.isRecognitionAvailable` is false, read `Settings.Secure.voice_recognition_service`, and target the configured OEM speech recognizer before falling back to the default recognizer.

## Best Possible Solution

Land an updated Android voice fallback in the current `ai.openclaw.app` package with an explicit no-backend disable path, focused regression coverage where feasible, and redacted real-device proof from an affected OEM device.

## Reproduction Assessment

No live high-confidence reproduction was established in this review because no affected Chinese OEM Android device was exercised. Source inspection does show the reported condition: current main disables the Voice-tab mic whenever `isRecognitionAvailable(context)` is false, and the PR body provides a concrete Xiaomi 14 Pro reproduction claim.

## Solution Assessment

No, not as currently written. The OEM service lookup and targeted `createSpeechRecognizer` direction is reasonable, but the patch must preserve a clean disable path when no valid recognizer backend exists and should be updated against the current Android package path.

## Review Findings

- **[P2] Stop retrying when no recognizer backend exists:** `apps/android/app/src/main/java/ai/openclaw/android/voice/MicCaptureManager.kt:186-190`
  - body: After removing the `isRecognitionAvailable=false` return, devices with no valid OEM `voice_recognition_service` fall through to the default recognizer. Keep a no-backend stop path or disable on this unavailable-client failure before retrying.
