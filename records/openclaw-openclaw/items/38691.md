## Summary

Keep open. Current main and the latest release still hard-disable the Android Voice Tab manual mic path when Android reports no recognizer, while this PR is useful but not mergeable as-is because it targets the old Android package path and removes the hard stop without a no-backend retry safety valve.

## What This Changes

The PR changes Android `MicCaptureManager` to continue past `SpeechRecognizer.isRecognitionAvailable=false`, read the configured `voice_recognition_service`, and create a targeted `SpeechRecognizer` for that component before falling back to the default recognizer.

## Best Possible Solution

Implement the generic configured speech-service fallback in the current `apps/android/app/src/main/java/ai/openclaw/app/voice/MicCaptureManager.kt` path, but preserve graceful failure for devices with no usable recognizer by disabling the manual mic or avoiding retries when the configured component is absent, invalid, or immediately fails as a client/no-backend error. Keep it vendor-agnostic and add focused Android/Robolectric coverage for the creation/error seam if practical.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: when `SpeechRecognizer.isRecognitionAvailable(context)` is false, `start()` exits before recognizer creation, so a configured OEM recognizer can never be tried; the PR body adds Xiaomi manual proof for the positive device case.

## Solution Assessment

No, not as submitted. The configured-service fallback is the right narrow direction, but the patch should be rebased to the current package path and should add a no-backend safety path before removing the existing hard stop.
