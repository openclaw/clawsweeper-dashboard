## Summary

Keep open: current main still hard-disables the Voice-tab mic when Android reports no recognizer, so the central bug is not implemented. The PR remains a plausible fix direction, but it is merge-blocked by missing real-device proof and unsafe fallback behavior.

## What This Changes

The PR changes Android Voice-tab speech recognizer startup to continue after `isRecognitionAvailable=false` and try a configured OEM recognition service before falling back to the default recognizer.

## Best Possible Solution

Land a current-path Android Voice-tab fallback that verifies a configured OEM `RecognitionService`, preserves clean disable behavior when no usable recognizer exists, and includes redacted affected-device proof.

## Reproduction Assessment

No high-confidence affected-device reproduction is available. Source inspection shows current main disables the Voice-tab mic when `SpeechRecognizer.isRecognitionAvailable(context)` is false, but the OEM false-negative and after-fix result still need inspectable Xiaomi/OEM proof.

## Solution Assessment

No, not as written. The fallback direction is narrow and plausible, but it should target the current `ai.openclaw.app` path, validate the OEM component through `RecognitionService.SERVICE_INTERFACE`, and keep a no-backend disable path instead of retrying indefinitely.

Android's `SpeechRecognizer` contract says callers must first verify the component exists and provides `RecognitionService.SERVICE_INTERFACE`; stale or non-recognition settings can otherwise hit the same failure and retry path.
  - confidence: 0.84
