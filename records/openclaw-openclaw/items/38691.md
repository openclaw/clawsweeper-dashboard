## Summary

Keep open: current main still has the hard stop when Android reports no speech recognizer, so the reported Voice-tab bug is plausible, but this PR is not mergeable as written because it removes the no-backend stop path and lacks inspectable real-device proof.

## What This Changes

This PR changes Android Voice-tab mic startup to continue after `SpeechRecognizer.isRecognitionAvailable=false`, query the secure-settings recognizer component, and fall back to the default recognizer.

## Best Possible Solution

Land a current-path Android Voice-tab fallback that verifies a configured OEM `RecognitionService`, preserves clean disable behavior when no usable backend exists, and includes redacted affected-device proof.

## Reproduction Assessment

No high-confidence affected-device reproduction is available from this review. Source inspection does show the current-main failure path when `isRecognitionAvailable(context)` returns false, but the OEM-device premise and after-fix behavior still need real proof.

## Solution Assessment

No, not as written. The OEM recognizer lookup is a reasonable narrow direction, but the patch must preserve the no-backend stop path, verify the component path defensively, move onto the current package path, and include real-device proof before merge.
