## Summary

Keep this PR open because it addresses a real Android Voice Tab gap that current main still has, and the proposed generic OEM speech-service fallback is a plausible direction. It is not merge-ready: the branch targets the pre-rename Android package path, and the diff removes the existing no-recognizer hard stop without adding a no-backend safety path, so devices with no usable recognizer can fall into the existing retry loop.

## What This Changes

The PR changes Android `MicCaptureManager` to continue after `SpeechRecognizer.isRecognitionAvailable(context)` is false, read `Settings.Secure` `voice_recognition_service`, parse it as a `ComponentName`, and create a targeted `SpeechRecognizer` before falling back to the default recognizer.

## Best Possible Solution

Implement the OEM speech-service fallback in the current Android app voice path, keep it vendor-agnostic, verify the configured `RecognitionService` component before targeting it, add the Android package-visibility query if needed, and preserve a clean no-backend disable path so unsupported devices fail once instead of retrying indefinitely.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: the Voice Tab manual mic path exits at `SpeechRecognizer.isRecognitionAvailable(context) == false` before any recognizer can be created, and the PR body provides manual Xiaomi proof for the target positive case.

## Solution Assessment

No, not as submitted. The configured-service fallback is the narrow maintainable direction, but the patch needs to be rebased to the current package path and must add a no-backend safety path before removing the existing hard stop.
