## Summary

Keep open: current main still blocks manual mic capture when Android reports no recognizer, while this PR’s OEM fallback is useful but unsafe as submitted because unsupported devices can fall into the existing retry path.

## What This Changes

The PR changes Android manual mic capture to continue after `isRecognitionAvailable=false`, read the configured secure voice-recognition service, create a targeted `SpeechRecognizer`, and fall back to the default recognizer.

## Best Possible Solution

Port the generic OEM recognizer fallback into the current `ai.openclaw.app` manual mic path, preserve a one-shot unsupported-device disable path, and add the required changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main returns before recognizer creation when `isRecognitionAvailable(context)` is false, and the PR body supplies a concrete Xiaomi device where a configured OEM recognizer exists despite that false value.

## Solution Assessment

No, not as submitted. The OEM recognizer direction is maintainable, but the patch should require or validate a configured backend and keep unsupported devices from entering the `ERROR_CLIENT` retry loop.
