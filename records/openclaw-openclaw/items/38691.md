## Summary

Keep open: current main still blocks this device class, but this PR is not merge-ready because it removes the unsupported-device stop, can retry repeatedly on devices with no usable recognizer, lacks inspectable real-device proof, and needs porting to the current Android package path.

## What This Changes

This PR changes Android manual mic capture to continue after `isRecognitionAvailable=false`, select a configured secure-setting speech recognizer component, and fall back to the default recognizer.

## Best Possible Solution

Port the generic OEM component fallback into the current manual mic path, bypass the availability stop only when a configured recognizer component is usable, and keep unsupported devices on a one-shot disable path.

## Reproduction Assessment

Yes, source-reproducible for the current-main code path: when Android reports recognition unavailable, `MicCaptureManager.start()` disables the mic before trying any configured recognizer. The OEM-specific Xiaomi behavior is still contributor-supplied because no inspectable device proof was posted.

## Solution Assessment

No, not as submitted. The component-targeted OEM fallback is the right direction, but the patch should not fall through to default-recognizer retries when no valid backend exists and must be ported to the current package path.
