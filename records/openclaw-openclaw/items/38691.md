## Summary

Keep open: current main still hard-stops Android manual mic capture when platform speech availability is false, and this PR points at a useful generic OEM fallback, but the submitted patch is dirty against main and removes the unsupported-device stop without preserving a no-backend safety path.

## What This Changes

The PR changes Android manual mic capture to continue past `isRecognitionAvailable=false`, read `Settings.Secure` `voice_recognition_service`, and create a targeted `SpeechRecognizer` before falling back to the default recognizer.

## Best Possible Solution

Carry forward the generic OEM recognizer fallback in the current `ai.openclaw.app` manual mic path, validate a configured recognition service, and preserve a one-shot unsupported-device disable path.

## Reproduction Assessment

Yes, source-reproducible. Current main's Voice Tab mic flow returns immediately when `isRecognitionAvailable(context)` is false, and the PR body provides a concrete Xiaomi 14 Pro case where an OEM recognizer is configured despite that false value.

## Solution Assessment

No, not as submitted. The OEM recognizer fallback is the right direction, but it should validate a configured recognition service and keep a no-backend stop instead of falling through to repeated `ERROR_CLIENT` retries.
