## Summary

Keep open: current main still has the reported Android Voice Tab hard stop, and this PR is useful source work, but the patch needs changes before merge because it removes the unsupported-device stop without proving a valid recognizer backend exists.

## What This Changes

The PR changes Android manual mic startup to continue after `isRecognitionAvailable()` is false, read the configured OEM speech service from `Settings.Secure`, target that component, and fall back to the default recognizer.

## Best Possible Solution

Port the OEM recognizer fallback to the current manual mic path, add the required recognition-service visibility query, validate any configured `RecognitionService`, and preserve a one-shot unavailable-device disable path.

## Reproduction Assessment

Yes. Source inspection shows the current Voice Tab manual mic path returns before recognizer creation whenever `isRecognitionAvailable(context)` is false, and the PR body provides a concrete Xiaomi 14 Pro smoke case for the intended OEM path.

## Solution Assessment

No, not as submitted. The generic targeted-recognizer direction is narrow and maintainable, but the patch must validate the configured backend and keep a clean no-backend stop before removing the early guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stop when no recognizer backend exists:** `apps/android/app/src/main/java/ai/openclaw/android/voice/MicCaptureManager.kt:184-190`
  - body: Continuing after `isRecognitionAvailable=false` is only safe after finding a valid configured recognition service. As patched, a blank, unparseable, or unusable `voice_recognition_service` falls back to the default recognizer; Android reports `ERROR_CLIENT` when no selected service exists, and the current listener treats that as retryable with a 600 ms restart path. Keep a one-shot unavailable path after validation before removing the guard.
  - confidence: 0.9
