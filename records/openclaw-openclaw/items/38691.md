## Summary

Keep open: current main and the latest release still have the Android Voice-tab hard stop, and this PR is a plausible user-facing fix direction, but the branch still lacks inspectable real-device proof and introduces a no-backend retry regression that should block merge.

## What This Changes

The PR changes Android mic startup to continue after `SpeechRecognizer.isRecognitionAvailable=false`, read `Settings.Secure.voice_recognition_service`, and create a targeted `SpeechRecognizer` before falling back to the default recognizer.

## Best Possible Solution

Land a current-path Android Voice-tab fallback that verifies a configured OEM `RecognitionService`, preserves clean disable behavior when no usable backend exists, and includes redacted affected-device proof.

## Reproduction Assessment

No high-confidence affected-device reproduction is available. Source inspection confirms current main stops when `isRecognitionAvailable(context)` returns false, but the OEM-device premise and after-fix result still need inspectable proof.

## Solution Assessment

No, not as written. The OEM recognizer lookup is a reasonable narrow direction, but the patch must preserve the no-backend stop path, verify the component path defensively, move onto the current package path, and include real-device proof before merge.
