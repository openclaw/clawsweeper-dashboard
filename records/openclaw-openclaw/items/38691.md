## Summary

Keep open. Current main still has the reported Android Voice Tab hard stop, and the PR's OEM-recognizer direction is useful, but the submitted diff is not merge-ready because it removes the unsupported-device stop and can drive the existing ERROR_CLIENT retry path.

## What This Changes

The PR changes Android manual mic capture to continue past `isRecognitionAvailable=false`, read the configured secure `voice_recognition_service`, create a targeted `SpeechRecognizer`, and fall back to the default recognizer.

## Best Possible Solution

Land a current-main Android manual mic fix that targets the configured OEM recognition service when platform availability is false, avoids vendor hardcoding, and disables once when no backend exists.

## Reproduction Assessment

Yes, source-reproducible. Current main returns immediately when `SpeechRecognizer.isRecognitionAvailable(context)` is false, and the PR body provides a concrete Xiaomi 14 Pro case with a configured OEM recognition service despite that false result.

## Solution Assessment

No, not as submitted. The generic OEM fallback is the right narrow direction, but it must validate or require a configured backend and preserve a one-shot unsupported-device disable path instead of falling through to repeated `ERROR_CLIENT` retries.
