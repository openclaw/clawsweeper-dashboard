## Summary

Keep open: current main still hard-stops the Android Voice Tab mic when `SpeechRecognizer.isRecognitionAvailable(context)` is false, but this PR should not merge as submitted because unsupported devices can fall through into the existing retry loop and the branch is stale against the current app package path.

## What This Changes

The PR changes Android manual mic capture to continue after `isRecognitionAvailable=false`, read `Settings.Secure.voice_recognition_service`, and create a targeted OEM `SpeechRecognizer` before falling back to the default recognizer.

## Best Possible Solution

Port the generic OEM recognizer fallback into the current `ai.openclaw.app` manual mic path, only bypass the availability guard when a configured recognizer service can be used, preserve a one-shot unsupported-device disable path, and add the required changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main returns before recognizer creation when platform availability is false, and the PR body gives a concrete Xiaomi 14 Pro case where a configured OEM recognizer exists despite that false value.

## Solution Assessment

No, not as submitted. The OEM recognizer fallback is the right direction, but the patch should validate or require a configured backend and keep the unsupported-device stop instead of falling through to repeated `ERROR_CLIENT` retries.
