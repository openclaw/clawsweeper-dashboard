## Summary

Keep open. Current main still lacks the requested response-prefix context variables, and the PR is a relevant implementation candidate, but the patch is not merge-ready because it can overstate context usage from aggregate totals and does not thread the new usage callback through all existing channel delivery paths.

## What This Changes

The PR adds `{context}` and `{contextPercent}` response-prefix template variables, an `onContextUsage` callback, runtime usage population, and focused interpolation tests.

## Best Possible Solution

Revise the PR so `{context}` and `{contextPercent}` are derived from prompt-side context counters, the usage callback is centrally propagated anywhere `onModelSelected` is threaded, and the public config docs/type help/changelog describe the exact semantics before merge.

## Reproduction Assessment

Yes. On current main, configuring `messages.responsePrefix` with `{context}` or `{contextPercent}` leaves the placeholders unresolved; on the PR head, a mocked `lastCallUsage` with a large output-inclusive `total` reproduces an inflated context percentage.

## Solution Assessment

No. The direction is narrow and appropriate, but the implementation should reuse `deriveContextPromptTokens` or equivalent prompt-only math and sweep the reply option propagation boundary instead of relying on partial call-site wiring.
