## What This Changes

The PR persists the effective creator tool allowlist into agent-created cron `agentTurn` jobs, caps cron add/update payloads against it, restricts the standalone OpenClaw tools MCP cron surface, and adds focused regression tests.

## Best Possible Solution

Persist the creator’s effective tool boundary only when the creator was restricted, keep unrestricted cron creation unchanged, and require redacted runtime proof plus maintainer acceptance of the stricter boundary before merge.

## Solution Assessment

Likely yes: persisting the cap at cron add/update is the narrow owner-boundary fix because delayed runs only have the stored payload available. The remaining question is maintainer acceptance of the stricter compatibility behavior, not an obvious better code location.
