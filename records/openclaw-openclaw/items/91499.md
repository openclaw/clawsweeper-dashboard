## Summary

Keep open: the direction is useful, but the patch is not merge-ready because the new persisted cron `toolsAllow` cap is only enforced by the embedded cron runner path; CLI-backed scheduled turns still do not receive that cap. The PR also still has mock-only proof for a runtime scheduling boundary and carries a compatibility-sensitive change in scheduled-turn tool availability.

## What This Changes

The PR persists the creator's effective tool surface onto cron `agentTurn` add/update payloads and threads that cap through agent, gateway, skill-dispatch, and standalone OpenClaw tools MCP construction tests.

## Best Possible Solution

Make cron execution enforce the persisted tool cap on every runner path, failing closed for CLI backends or routing restricted cron jobs through an enforceable embedded runtime, then add redacted delayed-run proof.

## Reproduction Assessment

Yes, from source inspection: the PR writes `payload.toolsAllow`, the embedded cron path forwards it, and the adjacent CLI cron path does not. The CLI runner contract also shows it only fails closed when `toolsAllow` is actually supplied.

## Solution Assessment

No. Persisting the cap at cron add/update is only a partial fix unless the later cron execution path enforces it for both embedded and CLI-backed runs or fails closed where enforcement is impossible.
