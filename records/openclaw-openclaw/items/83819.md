## Summary

This PR should stay open for normal review, but it is not merge-ready: the new hook is not wired into any OpenClaw production caller, failure semantics are undefined, and the external no-op wake loop has no real behavior proof.

## What This Changes

Adds an optional `wakeGuard` callback to `startHeartbeatRunner` and invokes it before targeted heartbeat wakes call `runOnce`.

## Best Possible Solution

Put the wake eligibility check at the owner boundary where actionable tickets and assignee context are available, then expose a public runtime seam only if needed with tested skip and failure semantics.

## Reproduction Assessment

No for the claimed external no-op LLM loop: the Paperclip/Ronen queue and `heartbeat_runs.contextSnapshot` state are not present here, and the PR includes no live proof. The merge blocker is source-reproducible because current main has no production caller or plugin runtime seam that would pass `wakeGuard`.

## Solution Assessment

No; an uncalled internal hook is not the best way to solve the reported behavior. The safer path is a wired eligibility check at the adapter/runtime boundary with defined failure handling, regression coverage, and real deployment proof.
