## Summary

Keep open: current main still exposes ACPX child harnesses to ambient provider auth env and lacks the requested generic auth-contract/preflight surface. The open linked implementation PR with closing syntax should be reviewed before this issue is closed, and the later Claude ACP evidence strengthens that this is a generic ACP boundary problem rather than a Gemini-only report.

## What This Changes

The issue requests ACPX auth-boundary hardening so Gemini, Claude, and similar ACP harnesses can declare auth expectations, strip incompatible ambient provider env, preflight safely, and document mixed-credential hosts.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/81200 for the focused child-env boundary, then track broader auth-contract, doctor, and opt-in UX only if maintainers accept that larger surface.

## Reproduction Assessment

Yes for the narrow child-env exposure: current main source shows ACPX wrappers copying `process.env` into child harnesses, and upstream Gemini/Claude code consumes those env vars. The durable Gemini auth drift and real Claude billing behavior were not live-reproduced here.

## Solution Assessment

Yes for keeping this issue open: the linked open PR is the concrete implementation candidate and should be reviewed before closure. The focused wrapper strip is a maintainable first slice, while generic auth-contract, preflight, doctor, and opt-in behavior are product/security decisions.
