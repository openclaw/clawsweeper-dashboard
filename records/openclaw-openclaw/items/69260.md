## Summary

Keep open: current main still lacks the requested ACP/acpx auth-contract surface, per-agent provider-env scrubbing policy, and fail-closed preflight for Gemini-style mixed-credential hosts; the existing mitigation is only the narrower ACP bridge-client env scrub.

## What This Changes

Request to harden Gemini ACP/acpx launches with explicit auth metadata, ambient provider-env scrubbing, fail-closed preflight hooks, and mixed-credential documentation.

## Best Possible Solution

Design a plugin-facing ACP/acpx auth contract with expected auth mode, fallback policy, provider-env scrub policy, preflight failure behavior, focused tests, and mixed-credential docs.

## Reproduction Assessment

No, not for the full Gemini auth-drift symptom. Source review confirms the requested hardening seams are absent, but this review did not run a disposable Gemini CLI fixture with OAuth plus ambient API-key env against current main.

## Solution Assessment

Yes as a direction, but it needs maintainer/API review. A generic ACP/acpx auth contract is safer than hardcoding Gemini in core, especially because third-party ACP agents and the external acpx runtime are part of the boundary.
