## Summary

Keep open: current main and the latest stable release still expose ACPX child harnesses to inherited provider auth env and do not provide the requested auth-contract/preflight surface. The linked implementation candidate at https://github.com/openclaw/openclaw/pull/81200 is still open with closing syntax, so this issue should stay open until that PR is reviewed, landed, or explicitly rejected.

## What This Changes

The issue requests ACP/ACPX auth-boundary hardening so Gemini/Claude-style child harnesses cannot silently switch from durable OAuth auth to ambient provider API-key env fallback.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/81200 for the focused built-in child-env strip, then track broader auth-contract metadata, doctor warnings, docs, and opt-in API-key behavior only if maintainers accept that larger surface.

## Reproduction Assessment

Yes for the narrow child-env exposure: current main source shows ACPX wrappers copying process.env into child harnesses, and upstream Gemini source uses those env vars for auth selection. The durable Gemini settings drift and real Claude billing behavior were not live-reproduced here.

## Solution Assessment

Yes for keeping this issue open: the linked open PR is the concrete implementation candidate and should be reviewed before closure. The focused wrapper strip is a maintainable first slice, while generic auth-contract, preflight, doctor, and opt-in behavior need maintainer security/product judgment.
