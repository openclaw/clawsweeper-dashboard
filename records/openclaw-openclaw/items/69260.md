## Summary

Keep open: current main still has only a narrow bridge-client provider-env scrub, while the ACP/acpx harness path lacks generic auth-contract metadata, per-agent env scrubbing, and fail-closed preflight for Gemini-style mixed-credential hosts.

## What This Changes

Add ACP/acpx auth-contract metadata, provider env scrubbing hooks, fail-closed preflight, and mixed-credential docs so Gemini-style ACP launches cannot fall back to ambient API keys.

## Best Possible Solution

Keep this open for a maintainer-owned ACP/acpx auth contract: typed expected auth mode, fallback allowance, provider env scrub policy, fail-closed preflight, and mixed-credential docs, validated in a disposable Gemini stage environment.

## Reproduction Assessment

No. The issue gives a credible staged A/B and source review confirms the requested hardening seams are absent, but this review did not run a disposable Gemini CLI fixture against current main.

## Solution Assessment

Yes as a direction. A generic ACP/acpx auth contract is the maintainable path because it avoids hardcoding Gemini in core while giving Gemini and future ACP agents a first-class fail-closed boundary.
