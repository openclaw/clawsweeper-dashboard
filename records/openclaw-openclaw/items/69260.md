## Summary

Keep open: current main still lacks the requested ACP/acpx auth-contract surface, per-harness provider-env stripping policy, and fail-closed readiness/doctor checks, and the latest discussion adds a second Claude ACP case beyond the original Gemini report.

## What This Changes

Add ACP/acpx auth-contract metadata, per-harness env scrubbing, fail-closed preflight/doctor warnings, and mixed-credential docs so Gemini and Claude ACP harnesses do not silently fall back to ambient API keys.

## Best Possible Solution

Design a generic ACP/acpx auth-boundary contract for built-in and third-party harnesses, with per-harness metadata, child-process-only env stripping, explicit opt-in for API-key billing, doctor/readiness warnings, focused tests, and mixed-credential host docs.

## Reproduction Assessment

No, not for the full live auth-drift symptom. Source inspection confirms the requested auth-contract and per-harness scrub seams are absent, while the issue comments provide staged Gemini and Claude incident evidence that still needs disposable live validation.

## Solution Assessment

Yes as a product direction, but not as a quick one-off hardcode. A generic acpx/harness policy is more maintainable than Gemini-only or Claude-only scrubbing, and the final default/opt-out semantics need maintainer review.
