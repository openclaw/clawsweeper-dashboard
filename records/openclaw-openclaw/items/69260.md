## Summary

Keep open: current main still lets ACPX child harnesses inherit ambient provider auth environment, and the same author has an open linked PR with closing syntax that should be reviewed before this issue is closed. The later Claude comment materially strengthens the report by showing the same boundary problem outside Gemini.

## What This Changes

The issue requests ACP/ACPX auth-boundary hardening so provider-owned child harnesses can declare auth expectations, scrub incompatible ambient API-key env, and fail closed when needed.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/81200 for the focused child-env boundary, then track broader auth-contract and doctor/preflight UX only if maintainers want that larger surface.

## Reproduction Assessment

Yes for the narrow child-env exposure: current main shows ACPX wrappers copying `process.env` into child harnesses, and upstream Gemini/Claude dependencies consume those auth env vars. The durable Gemini auth drift and real billing behavior were not live-reproduced here.

## Solution Assessment

Yes for keeping this issue open: the linked open PR is the concrete implementation candidate and should be reviewed before closure. The focused wrapper strip is a maintainable first slice, while the generic auth-contract/preflight/doctor pieces are product decisions.
