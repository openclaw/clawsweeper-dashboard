## Summary

Keep open. Current main still only has a narrow provider-env scrub for the default `openclaw acp` bridge path; the ACP/acpx harness path lacks generic auth-contract metadata, per-agent provider env scrubbing, and fail-closed preflight hooks for Gemini-style mixed-credential hosts. The report is security-sensitive auth-boundary work and should stay with maintainer design rather than cleanup closure.

## What This Changes

The issue requests ACP/acpx auth-contract metadata, provider auth environment scrubbing, fail-closed preflight hooks, and mixed-credential host documentation for Gemini-style ACP agents.

## Best Possible Solution

Keep this open for a maintainer-owned ACP/acpx auth contract: typed per-agent expected auth mode, fallback allowance, provider env scrub policy, and fail-closed preflight in the runtime/backend contract, plus Gemini-specific policy layered through acpx and docs for mixed OAuth/API-key hosts. Validate with a disposable Gemini stage environment, not a live user profile.

## Reproduction Assessment

No. The issue gives a credible staged A/B, and static review confirms the requested OpenClaw hardening seams are absent, but this review did not have a runnable fixture that reproduces Gemini CLI durable-auth drift against current main.

## Solution Assessment

Yes as a direction. A generic ACP/acpx auth contract is the maintainable solution because it avoids hardcoding Gemini policy in core while giving Gemini and future ACP agents a first-class fail-closed boundary; the existing bridge-only scrub is too narrow.
