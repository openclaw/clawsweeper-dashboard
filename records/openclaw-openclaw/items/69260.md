## Summary

Keep open: current main still lacks a generic ACP/acpx auth-contract surface, per-harness provider-env scrub policy, and fail-closed readiness/doctor behavior; the newer Claude case strengthens the issue beyond the original Gemini-only report.

## What This Changes

Request to add generic ACP/acpx auth-contract metadata, child-process env scrubbing, fail-closed preflight, doctor warnings, and mixed-credential docs for ACP harnesses.

## Best Possible Solution

Design and land a plugin-facing ACP/acpx auth policy with per-harness expected auth mode, provider-env scrub metadata, explicit API-key opt-in, fail-closed doctor/readiness behavior, focused tests, and mixed-credential docs.

## Reproduction Assessment

No high-confidence live reproduction was established for the full Gemini or Claude auth drift on current main. Source inspection does confirm the requested auth-contract, per-harness strip policy, and fail-closed preflight seams are still absent.

## Solution Assessment

Yes as a direction, but not as an immediate narrow bug fix. A generic ACP/acpx auth contract is the maintainable solution; hardcoding Gemini or Claude behavior in core without an acpx/API design would be riskier.
