## Summary

Keep open: current main still lacks the requested generic ACP/acpx auth-contract surface, per-agent provider-env scrub policy, and fail-closed preflight; the newer command/args override and CLI bridge env stripping do not cover acpx harness launches for Gemini or Claude-style mixed-credential hosts.

## What This Changes

The issue requests ACP/acpx auth-contract metadata, provider auth environment scrubbing, fail-closed preflight hooks, and mixed-credential host documentation for Gemini and similar ACP harnesses.

## Best Possible Solution

Design a plugin-facing ACP/acpx auth contract that covers expected auth mode, fallback policy, provider-env scrub lists, fail-closed readiness checks, targeted tests, and mixed-credential documentation while keeping vendor-specific defaults in the owning harness/plugin layer.

## Reproduction Assessment

No, not for the full Gemini/Claude auth-drift symptom in a disposable live setup. Source review does provide a high-confidence path showing ambient provider env can still reach acpx child harnesses and that the requested auth-contract hooks are absent.

## Solution Assessment

Yes as a product direction, but it needs maintainer/API review. A generic ACP/acpx contract is safer than hardcoding Gemini or Claude policy in core, especially because current acpx explicitly treats ambient provider env as pass-through child-agent state.
