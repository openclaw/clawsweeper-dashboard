## Summary

Keep open: current main still has the ACPX child-process environment exposure the issue describes, and the active linked PR explicitly closes this issue but has not merged yet. The recent Claude case in the comments usefully broadens this from a Gemini-only concern to a generic ACPX auth-boundary hardening item.

## What This Changes

Request ACPX auth-boundary hardening so built-in ACP harnesses can strip incompatible provider auth env, declare auth expectations, fail closed, and document mixed-credential hosts.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/81200 for the narrow child-env boundary, then decide separately whether broader auth-contract, preflight, doctor, and opt-in config should follow.

## Reproduction Assessment

Yes for the narrow child-env exposure: current main shows ACPX-generated wrappers inherit `process.env`, and the issue/comments provide Gemini and Claude A/B evidence. I did not reproduce durable OAuth drift or billing behavior live in this read-only pass.

## Solution Assessment

Yes for keeping this issue open: the linked open PR is the concrete implementation path and should be reviewed before closing. The narrow wrapper strip is a maintainable first slice; the broader auth-contract and doctor/preflight UX should be a follow-up policy decision.
