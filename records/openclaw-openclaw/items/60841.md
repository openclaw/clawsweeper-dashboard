## Summary

Keep open. Current main still applies runtime `toolsAllow` only after `createOpenClawCodingTools()` and the policy pipeline can already filter out `read`, `write`, and `exec` under `group:openclaw`; the live closing PR remains open and unmerged.

## What This Changes

Embedded cron/isolated runs should let per-run `toolsAllow` re-expose explicitly requested core tools such as `exec` even when global `tools.allow` uses `group:openclaw`.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/60842 with a narrow patch that feeds runtime `toolsAllow` into core tool construction as a constrained known-core force-allow while preserving stronger policy gates.

## Reproduction Assessment

Yes. The source-level path is deterministic: `group:openclaw` excludes `exec`, `read`, and `write`, and current main applies runtime `toolsAllow` only after the core policy pipeline can already remove them.

## Solution Assessment

Yes at the direction level. The maintainable fix is to preserve explicitly requested known core tools before restrictive allowlists drop them, while keeping final trimming and stronger deny/runtime gates authoritative.
