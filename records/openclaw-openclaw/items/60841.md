## Summary

Keep open. Current main still has a source-reproducible path where `toolsAllow` reaches tool construction as a runtime allowlist but does not force re-add core tools through restrictive config allowlists, and the linked closing PR remains open rather than merged.

## What This Changes

The issue asks for embedded and cron `toolsAllow` to re-expose explicitly requested core tools like `exec` when config allowlists such as `group:openclaw` would otherwise remove them.

## Best Possible Solution

Land or replace the linked fix with a narrow core force-allow path that preserves explicitly requested known core tools before config allowlists drop them while keeping stronger deny and runtime gates authoritative.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from current-main source: `toolsAllow` becomes `runtimeToolAllowlist`, `group:openclaw` excludes `exec`/`read`/`write`, and the policy pipeline only filters downward before the final allow filter runs.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer policy review; the narrow technical fix is to preserve named runtime-requested core tools before restrictive config allowlists remove them, without bypassing explicit deny or runtime gates.
