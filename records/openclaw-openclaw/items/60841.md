## Summary

Keep open: current main still has a source-reproducible embedded `toolsAllow` path where requested core tools can be constructed but then removed by restrictive policy before the final allow filter, and the focused closing PR remains open and conflicting.

## What This Changes

The issue asks for embedded cron/isolated runs to let per-run `toolsAllow` re-expose explicitly requested core tools such as `exec` after restrictive global tool allowlists filter them out.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/60842 with a narrow deny-preserving force-allow path for explicitly requested known core tools, while coordinating separately with the broader Codex cron tool-surface work.

## Reproduction Assessment

Yes, at source level. Current main carries `toolsAllow` into construction as `runtimeToolAllowlist`, but `group:openclaw` excludes `exec`/`read`/`write` and the shared policy pipeline filters downward before the final embedded allow filter.

## Solution Assessment

Unclear until maintainers approve the policy boundary. The maintainable direction is a narrow deny-preserving core force-allow for explicitly requested known tools, not a broad bypass of sandbox, provider, subagent, or runtime restrictions.
