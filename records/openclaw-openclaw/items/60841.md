- `P2`: This is a deterministic embedded/cron tool exposure bug with real automation impact but a limited affected configuration set.
- `impact:security`: The issue is about authorization precedence for command and file tools such as `exec`, `read`, and `write` under restrictive allowlists.

## Summary

Keep open: current main still has a source-reproducible path where embedded cron `toolsAllow` is forwarded but cannot restore core tools after restrictive tool policy filters them out, and the linked implementation PRs remain open rather than landed.

## What This Changes

The issue reports that embedded cron/isolated runs cannot use per-run `toolsAllow` to restore requested core tools after a restrictive global tool allowlist has already filtered them out.

## Best Possible Solution

Keep this issue open as the canonical tracker until a focused current-main fix lands that lets explicitly requested known core tools survive restrictive allowlists while explicit deny, sandbox, subagent, provider/runtime, and final embedded filters still win.

## Reproduction Assessment

Yes. Source inspection shows cron passes `toolsAllow` into embedded runs, current main converts it to `runtimeToolAllowlist`, and the normal policy pipeline can remove `exec` before the final embedded allow filter sees it.

## Solution Assessment

Unclear until maintainers approve the security boundary. The narrow maintainable direction is to re-expose explicitly requested known core tools for the run without bypassing explicit deny or runtime gates, then prove that behavior with focused regression coverage and redacted runtime proof.
