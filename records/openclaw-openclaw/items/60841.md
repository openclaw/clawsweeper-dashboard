## Summary

Keep open: current main still forwards `toolsAllow` only as a runtime allowlist and does not re-expose core tools such as `exec`, `read`, or `write` after a restrictive `tools.allow: ["group:openclaw"]` policy filters them out. The issue is already paired with active fix paths, especially https://github.com/openclaw/openclaw/pull/60842, so it should stay open until that security-sensitive policy decision lands or is explicitly rejected.

## What This Changes

The issue reports that embedded cron runs with `toolsAllow` cannot regain core tools such as `exec` after a restrictive upstream tool allowlist filters them out.

## Best Possible Solution

Land or reject a narrow, security-reviewed policy fix that lets explicit per-run `toolsAllow` materialize requested core tools while preserving explicit deny and runtime/provider restrictions.

## Reproduction Assessment

Yes at source level: current main forwards `toolsAllow` into `runtimeToolAllowlist`, but the core policy pipeline still filters `exec`, `read`, and `write` under `tools.allow: ["group:openclaw"]`. I did not run tests because this review is read-only.

## Solution Assessment

Unclear until maintainers decide the security boundary. The narrowest maintainable direction is the linked fix shape: re-expose only explicitly requested known core tools, keep explicit deny stronger, and cover the forwarding and deny-precedence cases.
