## Summary

Keep open: current main still does not provide the requested finite `toolsAllow` behavior for core shell/file tools, and an open linked PR is already the candidate fix path. The later live comments usefully narrow the remaining problem to security-sensitive Codex isolated cron runs that can finish `ok` even when required tools are absent.

## What This Changes

The issue requests that embedded isolated cron runs either honor finite `toolsAllow` for requested core shell/file tools such as `exec`, `read`, and `write`, or fail before dispatch when those tools cannot be materialized.

## Best Possible Solution

Land or replace the linked fix with a maintainer-approved core/Codex cron path that either exposes only exactly requested enforceable tools under existing deny/runtime gates or fails preflight when exact materialization is impossible.

## Reproduction Assessment

Yes. Source inspection on current main shows requested core tools can still be built and then removed by existing policy/Codex dynamic filters, and multiple sanitized live comments reproduce the same finite-allowlist failure in released builds.

## Solution Assessment

Unclear. Re-exposing requested dynamic tools is plausible, but the best fix needs maintainer security direction between exact dynamic exposure and fail-closed preflight rather than enabling a broad native Codex surface.
