## Summary

Keep open. Current main still has a source-reproducible path where an embedded `toolsAllow` request can build the core tool family but then lose `exec`, `read`, or `write` to an upstream restrictive `tools.allow`, and the linked closing PR remains open and conflicting.

## What This Changes

The issue asks for embedded cron/isolated `toolsAllow` to re-expose explicitly requested core tools such as `exec` even when a broader config allowlist like `group:openclaw` would otherwise omit them.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/60842 with a narrow core force-allow path that preserves explicitly requested known core tools before config allowlists drop them, while stronger deny and runtime gates remain authoritative.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from current-main source: `toolsAllow` reaches construction, `group:openclaw` excludes `exec`/`read`/`write`, and the shared policy pipeline only filters downward before the final runtime allow filter.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer policy review because the right fix changes command/file tool exposure semantics; the narrow maintainable direction is to force-preserve only explicitly requested known core tools without bypassing deny or runtime gates.
