## Summary

Keep open. Current main still has the deterministic policy ordering gap, and #60842 is the open closing PR rather than a merged fix.

## What This Changes

Embedded/cron runs should let per-run `toolsAllow` re-expose explicitly requested core tools such as `exec`, unless a stronger deny or runtime/provider restriction blocks them.

## Best Possible Solution

Review and land #60842, or replace it with a narrow patch that feeds runtime `toolsAllow` into core tool construction as a constrained force-allow for known core tools with regression coverage for restrictive `group:openclaw` config.

## Reproduction Assessment

Yes. The source-level reproduction is deterministic: `group:openclaw` excludes `exec`, `read`, and `write`, and current main applies runtime `toolsAllow` only after the core policy pipeline has already had a chance to drop those tools.

## Solution Assessment

Yes at the direction level. The maintainable fix is to preserve explicitly requested known core tools before policy filtering removes them, while keeping final `toolsAllow` trimming and all stronger deny/runtime gates authoritative.
