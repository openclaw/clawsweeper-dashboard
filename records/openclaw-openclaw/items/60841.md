## Summary

Keep open. Current main still lets restrictive config allowlists remove core tools before the embedded run's final `toolsAllow` filter can select them, and the open linked implementation PR is still unmerged.

## What This Changes

The issue reports that embedded/isolated cron runs cannot use per-run `toolsAllow` to restore core tools like `exec` after restrictive config such as `tools.allow: ["group:openclaw"]` filters them out.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/60842 or a narrow replacement lands with regression coverage for restrictive `group:openclaw` config plus explicit-deny precedence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `group:openclaw` excludes `exec`/`read`/`write`, the core policy pipeline filters those tools before the embedded runner's final `toolsAllow` trim, and that final trim cannot restore dropped tools.

## Solution Assessment

Yes at the direction level. The narrow maintainable fix is to feed per-run `toolsAllow` into core tool construction/policy as a constrained force-allow for known core tools while preserving stronger deny and runtime gates.
