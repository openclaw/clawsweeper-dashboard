## Summary

Keep open: current main still has a source-reproducible path where embedded `toolsAllow` reaches tool construction but cannot re-add core tools already removed by restrictive policy, and the focused closing PR is still open/conflicting.

## What This Changes

The issue reports that embedded or isolated cron runs with `toolsAllow` cannot regain `exec`, `read`, or `write` after a restrictive `tools.allow` such as `group:openclaw` filters those core tools out.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/60842 with a narrow core force-allow path for explicitly requested known core tools, while stronger deny and runtime gates remain authoritative.

## Reproduction Assessment

Yes, by source inspection. `toolsAllow` reaches construction as `runtimeToolAllowlist`, `group:openclaw` excludes `exec`/`read`/`write`, and the shared policy pipeline filters downward before the final embedded allow filter.

## Solution Assessment

Unclear until maintainers approve the policy boundary. The narrow maintainable direction is to preserve explicitly requested known core tools before restrictive allowlists drop them, without bypassing explicit deny or runtime gates.
