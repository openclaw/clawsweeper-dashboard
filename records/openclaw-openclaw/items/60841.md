## Summary

Keep open. Current main still has a source-reproducible path where config policy can remove core tools before the embedded-run `toolsAllow` filter runs, and the linked open PR is the implementation candidate rather than a merged fix.

## What This Changes

Embedded cron or isolated runs should allow `toolsAllow` to re-expose explicitly requested core tools such as `exec` when a config allowlist like `group:openclaw` would otherwise filter them out.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/60842 with a narrow policy fix that preserves explicitly requested known core tools before restrictive allowlists drop them while keeping stronger deny and runtime gates authoritative.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source inspection: with `tools.allow: ["group:openclaw"]`, `exec`, `read`, and `write` are excluded before the final embedded `toolsAllow` filter can select them.

## Solution Assessment

Is this the best way to solve the issue? Unclear pending maintainer policy approval; the narrow technical shape is to force-allow named runtime-requested core tools before downward filtering while preserving explicit denies and sandbox/provider gates.
