## Summary

Keep open. Current main still applies per-run `toolsAllow` only as plugin materialization input plus a final post-filter, so restrictive config policy such as `tools.allow: ["group:openclaw"]` can remove `exec`, `read`, and `write` before the embedded run can select them; #60842 remains the open closing PR.

## What This Changes

Embedded cron/isolated runs should let per-run `toolsAllow` re-expose explicitly named core tools such as `exec` after restrictive config allowlists, while preserving stronger denies and runtime restrictions.

## Best Possible Solution

Review and land #60842, or replace it with a narrow patch that feeds runtime `toolsAllow` into core tool construction as a constrained force-allow for known core tools only, with regression coverage for restrictive `group:openclaw` config and embedded-runner forwarding.

## Reproduction Assessment

Yes. The source-level reproduction is deterministic: `group:openclaw` excludes `exec`, `read`, and `write`, and current main only applies runtime `toolsAllow` after the core policy pipeline has already filtered those tools out.

## Solution Assessment

Yes at the direction level. The maintainable fix is to preserve named runtime core tools before the policy pipeline drops them, while keeping the final `toolsAllow` trim and all stronger deny/runtime gates authoritative.
