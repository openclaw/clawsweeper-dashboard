## Summary

Keep open. Current main still filters embedded-run tools after core tool construction, so `toolsAllow` cannot restore `exec` once `tools.allow: ["group:openclaw"]` has removed it; the linked closing PR is still open and unmerged.

## What This Changes

The issue asks embedded cron/isolated runs to let `toolsAllow` re-expose explicitly requested core tools unless a stronger deny or runtime restriction blocks them.

## Best Possible Solution

Treat runtime `toolsAllow` as a constrained force-allow for known core tools before restrictive config allowlists run, while preserving deny/sandbox/provider gates and adding regression coverage.

## Reproduction Assessment

Yes, from source inspection. With `tools.allow: ["group:openclaw"]`, `exec`, `read`, and `write` are excluded before `applyEmbeddedAttemptToolsAllow` runs, so the reported embedded-run inventory gap is deterministic on current main.

## Solution Assessment

Yes at the implementation direction level, but it needs maintainer policy review because it deliberately creates a per-run exception for sensitive core tools. The narrow maintainable approach is a known-core-tool force-allow that still lets explicit denies and runtime gates win.
