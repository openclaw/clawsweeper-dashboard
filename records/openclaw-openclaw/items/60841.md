## Summary

Keep open. Current main and latest v2026.6.10 still disable the Codex app-server native tool surface for finite runtime allowlists and filter Codex-owned dynamic tool names before applying `toolsAllow`; the open linked PRs are candidate fixes, not merged replacements, and the remaining choice is security-sensitive.

## What This Changes

Fix isolated/embedded cron runs so finite `toolsAllow` can require core shell/file tools such as `exec`, `read`, and `write`, or fail before reporting `ok` when those tools cannot be exposed.

## Best Possible Solution

Keep this issue as the canonical tracker until a security-reviewed PR lands that either materializes requested tools under existing deny/sandbox gates or blocks the cron before dispatch with a non-ok result.

## Reproduction Assessment

Yes, source-reproducible with current main: finite Codex app-server `toolsAllow` disables native tools, then dynamic filtering removes native-owned `exec`/file tools before the allowlist can match. I did not run a live cron in this read-only sweep, but the source path and multiple release-era live repro comments point to the same failure class.

## Solution Assessment

Unclear until maintainers choose the security shape. The best fix is not another standalone repair lane; it is to review the linked PRs and land either exact safe dynamic/native tool materialization or deterministic preflight/block classification with cron status coverage.
