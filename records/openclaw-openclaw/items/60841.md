## Summary

Keep open. Current main still has the policy ordering gap: restrictive config allowlists can remove core tools before the embedded-run `toolsAllow` filter sees them, and the linked closing PR remains open rather than merged.

## What This Changes

Embedded cron/isolated runs should be able to use `toolsAllow` to make explicitly requested core tools such as `exec` callable unless deny, sandbox, provider, or runtime gates block them.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/60842, or replace it with a narrow patch that preserves explicitly requested known core tools before config allowlists drop them while keeping stronger gates authoritative.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure `tools.profile: "full"` with `tools.allow: ["group:openclaw"]`, run an embedded attempt with `toolsAllow: ["exec"]`, and current main can filter `exec` out before the final runtime allowlist.

## Solution Assessment

Yes at the technical direction level, but it needs maintainer policy review because it re-exposes sensitive core tools for a run. The narrow maintainable shape is a constrained known-core force-allow before downward filtering, with final `toolsAllow` trimming and all deny/runtime gates preserved.
