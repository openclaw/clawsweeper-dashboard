## What This Changes

Embedded cron and isolated runs with finite `toolsAllow` should either expose explicitly permitted core shell/file tools or fail before producing a false-green run.

## Best Possible Solution

Land a reviewed security-sensitive fix that either materializes only exactly enforceable requested tools under existing policy gates or emits a structured pre-dispatch failure when the finite allowlist cannot be honored.

## Reproduction Assessment

Yes, at source level. I did not run a live cron flow, but current source and tests show finite Codex `toolsAllow` disables the native surface and filters native-owned tool names out of dynamic tools, matching the provided live repros.

## Solution Assessment

Unclear until maintainers choose the security boundary. Exact finite materialization is operator-friendly, but current Codex code intentionally keeps native shell/file access fail-closed for restricted allowlists, so a structured preflight failure may be the safer fix.
