## Summary

Keep open. Current main still materializes tools, applies the config policy pipeline, and only then applies the embedded-run `toolsAllow` filter, so the `tools.allow: ["group:openclaw"]` plus `toolsAllow: ["exec"]` path remains source-reproducible; https://github.com/openclaw/openclaw/pull/60842 is the open closing PR, not a merged fix.

## What This Changes

Embedded cron and isolated runs should honor runtime `toolsAllow` for requested core tools even when an upstream allowlist such as `group:openclaw` omits them.

## Best Possible Solution

Review and land, rebase, or replace https://github.com/openclaw/openclaw/pull/60842 with a narrow policy fix that preserves explicitly requested known core tools before restrictive allowlists drop them, with regression coverage for `group:openclaw` and deny precedence.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source inspection: `group:openclaw` excludes `exec`, `read`, and `write`, and current main still applies the runtime `toolsAllow` filter only after the shared policy pipeline can drop those tools.

## Solution Assessment

Is this the best way to solve the issue? Yes at the technical direction level: the narrow fix is a constrained force-allow for known runtime-requested core tools before downward policy filtering, while final `toolsAllow` trimming and stronger deny/runtime gates remain authoritative.
