## Summary

Keep open. Current main still applies `toolsAllow` only after core tools have already been built and filtered by config policy, and the linked closing PR remains open, so this report is still the canonical issue for the bug.

## What This Changes

Embedded cron/isolated runs should let runtime `toolsAllow` re-expose explicitly requested core tools such as `exec`, `read`, and `write` when an upstream allowlist like `group:openclaw` would otherwise hide them.

## Best Possible Solution

Review and land #60842, or replace it with a narrow patch that feeds runtime `toolsAllow` into core tool construction as a constrained force-allow for known core tools only, while preserving deny precedence and adding regression coverage for the restrictive `group:openclaw` config path plus embedded-runner forwarding.

## Reproduction Assessment

Yes. The source-level path is deterministic: `group:openclaw` removes `exec`, `read`, and `write` during `createOpenClawCodingTools()`, and current `toolsAllow` only filters the already-reduced array afterward.

## Solution Assessment

Yes at the direction level. The maintainable fix is to merge a narrow runtime force-allow before the policy pipeline drops core tools, then keep the final `toolsAllow` filter as the authoritative per-run trim.
