## Summary

Keep open: current main still has a source-reproducible replay gap. Historical user-message sanitization runs before provider replay, but it only removes blank user content and does not call the existing inbound-metadata stripper; the fix should be coordinated with the related sender-attribution and Telegram context-bloat reports.

## What This Changes

The issue requests stripping OpenClaw-injected inbound metadata from historical user turns during prompt replay while preserving the current turn’s metadata envelope.

## Best Possible Solution

Add central replay-history cleanup that removes redundant OpenClaw inbound metadata from historical user turns while keeping current-turn context and preserving or replacing durable sender attribution deliberately.

## Reproduction Assessment

Yes, at source level: historical user replay passes through `sanitizeSessionHistory`, and the user-message sanitizer currently leaves nonblank string content unchanged instead of stripping inbound metadata. I did not run a live Telegram session, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear as an immediate automated fix: replay-time stripping is the right narrow area, but maintainers should decide how it interacts with durable sender attribution and the open Telegram context-duplication report before removing all historical metadata.
