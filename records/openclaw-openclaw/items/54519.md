## Summary

Keep open: current main and the latest release still do not provide a durable channel/account read-only mode, and the related WhatsApp-only PRs are either still open and blocked or closed unmerged while the shared channel/plugin contract remains a product decision.

## What This Changes

Request to add a per-account receive-only channel mode that preserves inbound message processing while blocking all outbound delivery at a hard gateway or transport boundary.

## Best Possible Solution

Define a maintainer-approved channel/plugin account contract for hard outbound suppression, document its security and upgrade semantics, then land channel-specific enforcement with focused tests and live transport proof.

## Reproduction Assessment

Yes, at source level: current main lacks a readOnly channel/account config field and no resolved WhatsApp account or socket send path carries a hard no-outbound policy. I did not run a live WhatsApp repro because the missing config and enforcement points are directly visible in source.

## Solution Assessment

Unclear: a hard transport or gateway gate is the right safety target, but maintainers still need to choose whether the durable API is shared across channel plugins, WhatsApp-specific, or externalized through wacli.
