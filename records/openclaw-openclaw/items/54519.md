## Summary

Keep open: current main and the latest release still do not provide the requested per-account/channel hard no-outbound mode, and the related WhatsApp implementation attempts are unmerged or closed rather than a safe canonical fix.

## What This Changes

The issue requests a per-account channel read-only mode that receives inbound messages while blocking all outbound channel delivery at a hard gateway or transport boundary.

## Best Possible Solution

Define an additive channel/account outbound policy with shared delivery enforcement plus channel-owned side-effect coverage, then decide whether a WhatsApp-only transport implementation should land first or wait for the shared contract.

## Reproduction Assessment

Yes for source reproduction: current main lacks a common or WhatsApp account-level read-only key, and shared outbound delivery does not consult a channel/account receive-only policy. I did not run a live WhatsApp or Signal repro because the missing config and enforcement path is clear from source.

## Solution Assessment

Unclear: hard enforcement is the right safety direction, but maintainers still need to choose whether the durable contract is a shared channel/account policy, a plugin-owned transport policy, or a staged WhatsApp-first implementation.
