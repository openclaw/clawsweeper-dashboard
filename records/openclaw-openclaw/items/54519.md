## Summary

Keep open. Current main has useful session-level send denial, but it still does not provide the requested per-account receive-only channel mode: there is no shared or WhatsApp `readOnly` account config, explicit gateway send/poll paths still deliver outbound traffic, and WhatsApp still has transport-level side effects that are not behind an account-level outbound-deny gate. The open WhatsApp PR is relevant implementation context but is not merged and does not cover the broader cross-channel request.

## What This Changes

Request a per-account channel read-only mode that continues ingesting inbound messages while hard-blocking all outbound delivery at the gateway or transport level.

## Best Possible Solution

Define a maintainer-approved channel/account outbound policy in shared config, schema/help, docs, and UI metadata, then enforce it before shared gateway delivery and every plugin-owned outbound side effect. Coordinate #71382 as the WhatsApp subset, but close this only after the generic policy lands or maintainers explicitly split the remaining cross-channel scope into narrower tracked work.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main has no accepted per-account `readOnly` config, `session.sendPolicy` blocks the inbound agent run rather than only delivery, and gateway/WhatsApp outbound paths remain callable without an account-level deny gate.

## Solution Assessment

No, current main is not the best solution yet. `session.sendPolicy` is useful but too session-scoped and early in the request path; the maintainable fix is a channel/account outbound-deny policy plus transport-side gates for side effects such as typing, read receipts, reactions, presence, messages, and polls.
