## Summary

Keep open: current main and v2026.5.28 still drop undeclared plugin tools through the manifest contract gate, `registerTool` still returns `void`, and the `tools.allow` warning still points generically at plugin enablement rather than the rejected manifest contract.

## What This Changes

Request clearer plugin-tool contract rejection observability and an API signal when `api.registerTool(...)` is rejected for missing `contracts.tools`.

## Best Possible Solution

Keep the manifest-first ownership gate, add operator-visible rejected-tool diagnostics, and make a maintainer decision on an additive `registerTool` acceptance signal before wiring `tools.allow` warnings to the rejected-tool record.

## Reproduction Assessment

Yes, source-level: a plugin that calls `api.registerTool(...)` while `contracts.tools` is empty hits the early diagnostic return, and `tools.allow` can still emit the generic unknown-entry warning. I did not run a live plugin because this review was read-only.

## Solution Assessment

Unclear as written: the observability problem is real, but the proposed return-or-throw behavior changes a plugin SDK contract. The safer direction is additive rejected-tool tracking and clearer warnings unless maintainers explicitly choose a breaking throw path.
