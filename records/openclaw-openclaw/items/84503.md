## What This Changes

The branch adds an iMessage conversation-repair helper and tests, then calls it from the monitor before inbound routing so anchorless watch payloads are recovered from RPC history or dropped fail-closed.

## Best Possible Solution

Fix the lint error, add redacted real iMessage/OpenClaw replay proof for recovery and fail-closed behavior, then land the focused pre-routing guard if maintainers accept the drop-on-unrecoverable policy.

## Reproduction Assessment

Yes, from source inspection and the linked live report: current main accepts the malformed payload, treats it as non-group when `is_group` is false, and routes replies by sender. I did not run a live iMessage reproduction in this read-only review.

## Solution Assessment

Mostly yes: the proposed recover-or-drop guard is the right boundary for this bug, because it acts before routing rather than asking sessions to infer missing group metadata. It still needs the lint fix and real behavior proof before merge.
