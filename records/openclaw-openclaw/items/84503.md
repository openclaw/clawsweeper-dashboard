## Summary

Keep open. The fix targets a real source-reproducible wrong-recipient iMessage routing bug, but the PR broadens “anchorless” to sender-only direct-message payloads that current main accepts and routes as DMs, and the proof is unit-test only for a privacy-sensitive live channel path.

## What This Changes

The PR adds an iMessage monitor conversation-repair helper, tests, and monitor-provider integration to recover `chat_id=0`/empty watch payloads through `chats.list` plus `messages.history` before routing, otherwise dropping them.

## Best Possible Solution

Keep the pre-routing recover-or-drop approach, but narrow the anchorless predicate to the observed malformed watch signature, preserve sender-only DMs, add compatibility coverage, and require redacted real iMessage proof before merge.

## Reproduction Assessment

Yes, source inspection plus the linked issue show current main accepts `chat_id: 0` with empty chat identifiers and `is_group: false`, then routes it as `imessage:<sender>`. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted. Pre-routing repair or drop is the right shape, but the implementation must avoid catching valid sender-only DMs and needs real iMessage proof for the routing change.
