## Summary

This remains necessary on current main and in v2026.6.6: the Qwen provider can emit `enable_thinking`, and current config accepts Qwen thinking formats, but the reported Coding Plan models still advertise no reasoning capability, so OpenClaw rejects `/thinking` before request construction. Alibaba’s current Coding Plan guide explicitly configures `enable_thinking: true` for `qwen3.5-plus`, `glm-5`, and `kimi-k2.5`; no merged or viable open PR currently owns the fix.

## What This Changes

Add official Ali Bailian Coding Plan reasoning support for supported subscription models.

## Best Possible Solution

In the Qwen provider, mark only Alibaba-confirmed Coding Plan models such as `qwen3.5-plus`, `glm-5`, and `kimi-k2.5` reasoning-capable, reuse the existing top-level `enable_thinking` adapter, and add catalog, stream, docs, and live endpoint proof for both on and off modes.

## Reproduction Assessment

Yes, at source level with high confidence: select a bundled Bailian model such as `qwen3.5-plus`, set `/thinking low`, and capability validation rejects it because the catalog declares `reasoning: false`; the June 12, 2026 comment reports the same result on v2026.6.5. A live authenticated Alibaba request was not available in this review.

## Solution Assessment

Yes in architecture: the narrow maintainable fix is provider-owned capability metadata feeding the existing Qwen payload adapter, not a new core provider or parallel config route. Maintainers still need to approve the exact supported model list and require live Coding Plan endpoint proof.
