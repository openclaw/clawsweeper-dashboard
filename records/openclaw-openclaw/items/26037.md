## Summary

Keep open. Current main and the latest stable release still reject non-off thinking for the reported Bailian/Qwen Coding Plan models because the bundled Qwen catalog marks `qwen3.5-plus`, `glm-5`, and `kimi-k2.5` as non-reasoning; the Qwen request adapter and custom-provider `thinkingFormat` support are already present, so the remaining decision is the exact vendor-approved Coding Plan model capability list and live proof.

## What This Changes

Request official Ali Bailian/Qwen Coding Plan support for reasoning/thinking mode in OpenClaw.

## Best Possible Solution

Update the Qwen provider catalog/static metadata for only Alibaba-confirmed Coding Plan reasoning models, reuse the existing `enable_thinking` adapter, and require focused catalog, stream, docs, and live Coding Plan proof for on/off thinking.

## Reproduction Assessment

Yes, at source level with high confidence: choose a bundled Coding Plan model such as `qwen3.5-plus`, request `/thinking low`, and the catalog-driven thinking profile rejects it because the model row says `reasoning: false`. A user also reported the same off-only rejection on `v2026.6.5`, but this review did not run an authenticated Alibaba request.

## Solution Assessment

Yes in architecture, but not as a broad toggle. The maintainable fix is provider-owned model capability metadata feeding the existing Qwen request adapter, with maintainer approval of the exact model list and live endpoint proof.
