## Summary

Keep this issue open. Current main has partial Qwen/Bailian Coding Plan support, but the named Coding Plan models still are not marked reasoning-capable and the custom-provider compat path still cannot express a Qwen/DashScope thinking format.

## What This Changes

Request official Ali Bailian/Qwen Coding Plan support for enabling thinking/reasoning on Coding Plan models such as qwen3.5-plus, glm-5, and kimi-k2.5.

## Best Possible Solution

Implement a focused provider-owned Qwen/Bailian fix: verify which Coding Plan models support top-level enable_thinking, mark only those bundled entries or compat paths as thinking-capable, route OpenClaw thinking levels to the DashScope payload for those paths, and update Qwen docs plus targeted regression tests.

## Reproduction Assessment

Yes for the source-level gap: selecting bundled qwen/glm-5, qwen/kimi-k2.5, or qwen/qwen3.5-plus leaves model.reasoning false, so the current Qwen wrapper will not emit enable_thinking. A live vendor reproduction remains missing because the issue has no API key-backed config or gateway log.

## Solution Assessment

Yes, a narrow Qwen provider/config compatibility change is the best direction. Broadly enabling Qwen-style payload flags in core for every OpenAI-compatible provider would risk unsupported-parameter failures and would violate the provider-owned boundary.
