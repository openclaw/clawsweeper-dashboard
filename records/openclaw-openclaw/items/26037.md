## Summary

Keep open. Current main still has only partial Qwen/Ali Bailian Coding Plan thinking support: the Qwen wrapper can emit `enable_thinking`, but the named Coding Plan models remain non-reasoning and the public custom-provider compat schema still cannot express a Qwen/DashScope thinking format.

## What This Changes

Request official Ali Bailian/Qwen Coding Plan support for enabling thinking/reasoning on Coding Plan models such as `qwen3.5-plus`, `glm-5`, and `kimi-k2.5`.

## Best Possible Solution

Implement a focused provider-owned Qwen/Bailian fix: verify which Coding Plan models support top-level `enable_thinking`, mark only those bundled entries or documented compat paths as thinking-capable, and update Qwen docs plus targeted regression tests.

## Reproduction Assessment

Yes for the source-level gap: selecting bundled `qwen/glm-5`, `qwen/kimi-k2.5`, or `qwen/qwen3.5-plus` leaves `model.reasoning` false, so the current Qwen wrapper will not emit `enable_thinking`. The live gateway restart failure is not reproduced because the issue does not include the exact config or API-backed logs.

## Solution Assessment

Yes, a narrow Qwen provider/config compatibility change is the best direction. Broadly enabling Qwen-style payload flags for every OpenAI-compatible provider would risk unsupported-parameter failures and would violate the provider-owned boundary.
