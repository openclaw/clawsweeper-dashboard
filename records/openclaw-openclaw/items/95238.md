## What This Changes

The PR changes Slack dispatch so thread participation for genuine Slack thread replies is recorded under the canonical root `thread_ts` instead of a delivered child reply timestamp.

## Best Possible Solution

Land a narrow Slack dispatch fix after real Slack behavior proof is added, with regression coverage tying dispatch recording to prepare-side `bot_thread_participant` lookup.

## Reproduction Assessment

Yes for source-level reproduction: current main can record one timestamp while prepare later checks `message.thread_ts`, and the cache lookup is exact-keyed. I did not run live Slack or tests because this review was read-only.

## Solution Assessment

Yes, this is the narrowest maintainable layer because it makes dispatch record the same canonical root thread key that prepare already uses. Broadening lookup to alternate child keys would create a second source of truth for participation state.
