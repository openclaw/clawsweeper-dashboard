## Summary

Keep open: source and release inspection show the reported delivered-true/no-message path still exists for identity-less cron announce text results, and the linked fix PR is still unmerged.

## What This Changes

Cron `--announce` fallback delivery should not report success unless the configured channel produces a usable delivery identity.

## Best Possible Solution

Land a shared outbound fix that filters identity-less delivery results before durable send and cron delivery state consume them, with regression coverage for cron announce text, formatted text, and media paths.

## Reproduction Assessment

Yes, at source level: current main and v2026.5.7 append identity-less text adapter results, then cron maps any non-empty result array to delivered. I did not run a live WeChat or Feishu repro in this read-only review.

## Solution Assessment

Yes: fixing shared outbound result accounting is the narrow maintainable direction; channel-specific WeChat or Feishu handling would leave the same false-positive path in other adapters. The linked PR follows that boundary but needs refresh/proof before landing.
