## Summary

Keep open: current main still has a source-level path where cron direct delivery can treat identity-less outbound text results as delivered, and https://github.com/openclaw/openclaw/pull/79811 is an open closing fix candidate rather than a merged fix.

## What This Changes

Cron `--announce` fallback delivery reports `delivered: true` for WeChat/Feishu runs even when the channel message never arrives.

## Best Possible Solution

Land a narrow shared outbound/durable cron delivery fix that filters identity-less adapter results before success accounting, with regression tests and redacted real cron/channel proof.

## Reproduction Assessment

Yes, at source level: current main can return a non-empty outbound result with no delivery identity, `sendDurableMessageBatch` treats that as sent, and cron derives `delivered` from the result count. The issue body adds concrete 2026.5.7 WeChat/Feishu live steps, but I did not execute the live channel flow in this read-only review.

## Solution Assessment

Yes, the maintainable direction is to fix the shared outbound result/receipt seam rather than adding channel-specific cron workarounds. The open PR is the right candidate path, but it should not close this issue until it is reviewed, proven, and merged.
