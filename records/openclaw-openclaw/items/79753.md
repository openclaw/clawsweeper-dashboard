## Summary

Keep open: current main still has a source-level path where non-empty but identity-less outbound results can become `delivered=true`, and an open closing PR owns the fix path.

## What This Changes

The issue requests that cron `--announce` only report delivery success when the configured WeChat/Feishu channel actually delivers, or otherwise report a clear delivery failure.

## Best Possible Solution

Land a reviewed fix that filters identity-less outbound adapter results before durable/cron delivery status is marked successful, with regression coverage and redacted real cron announce proof; keep this issue open until that PR merges or is replaced.

## Reproduction Assessment

Yes, at source level: cron direct delivery delegates to `sendDurableMessageBatch`, current outbound text/media paths can return a non-empty result without a platform identity, and cron then records `delivered=true` from that non-empty array. I did not run the live WeChat/Feishu flow.

## Solution Assessment

Yes, the narrow maintainable direction is to fix shared outbound result handling so cron/message delivery only count real delivery identities. The open linked PR is the right ownership path, but it still needs its review blockers and real behavior proof resolved before this issue closes.
