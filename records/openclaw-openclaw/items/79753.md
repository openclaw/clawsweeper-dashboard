## Summary

Keep open: current main still has a source-level path where identity-less outbound text/media results can be counted as delivered, and an open linked fix PR references this issue with closing metadata. Recent WeChat comments also suggest a related cold-session plugin/context-token failure that may need follow-up after the core false-success accounting is fixed.

## What This Changes

Cron `--announce` delivery should not report `delivered: true` when WeChat/Feishu fallback delivery produced no real channel message.

## Best Possible Solution

Land an updated version of the linked PR or an equivalent main change that filters identity-less outbound text/media results before durable/cron success accounting, then track any remaining cold WeChat context-token expiry as a separate plugin-specific follow-up if it still reproduces.

## Reproduction Assessment

Yes, for the core false-success path: current main can still carry an identity-less outbound text/media adapter result into `sendDurableMessageBatch`, and cron then derives `delivered: true` from a non-empty result array. I did not run a live WeChat/Feishu cron delivery in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is shared outbound delivery identity filtering with cron regression coverage, not WeChat/Feishu-specific cron status handling. The open linked PR is the current implementation candidate, but it needs rebase/review before this issue can close.
