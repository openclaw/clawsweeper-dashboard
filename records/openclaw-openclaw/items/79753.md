## Summary

## What This Changes

Cron `--announce` should deliver explicit WeChat/Feishu targets or report non-delivery instead of `delivered: true` when no channel adapter send happens.

## Best Possible Solution

Keep the current durable cron delivery path and have reporters retest after the next release; any remaining channel-specific failure should be filed with current-main logs for that plugin/runtime.

## Reproduction Assessment

Yes for the reported v2026.5.7 release: the issue includes concrete cron commands and run JSON showing `delivered: true` with no received message. For current main, the false-positive cron path is not reproduced by source inspection because empty/no-adapter results now remain not delivered.

## Solution Assessment

Yes. The current-main fix is the narrow maintainable path: route cron announce through the same durable message send lifecycle as direct sends and base delivered state on explicit outbound results, rather than adding channel-specific WeChat or Feishu workarounds.
