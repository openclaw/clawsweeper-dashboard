## Summary

Keep open: current main still has a source-level path where identity-less text delivery results can be counted as successful cron announce delivery, and https://github.com/openclaw/openclaw/pull/79811 explicitly targets this issue but remains unmerged.

## What This Changes

Cron `--announce` delivery should only report success when the configured channel actually returns a usable delivery receipt, or else report a clear non-delivery.

## Best Possible Solution

Finish the shared outbound receipt-identity fix in the open PR, covering text/media/payload paths with regression tests and redacted real cron channel proof, then close this issue through the merged fix.

## Reproduction Assessment

Yes, at source level: current main can return a non-empty outbound result with no usable platform identity, and cron maps any non-empty durable result to `delivered: true`. I did not execute the live WeChat/Feishu flow, but the reporter supplied concrete v2026.5.7 steps and a direct-send control.

## Solution Assessment

Yes in direction, not yet in repository state: filtering identity-less outbound results before cron counts them as delivered is the narrow maintainable fix. The current candidate is https://github.com/openclaw/openclaw/pull/79811, but it needs its review finding and real behavior proof resolved before merge.
