## Summary

Keep open. Current `main` still routes non-delivered hook completion/error system events to the default main session instead of the target agent session, so the PR's bug fix is not implemented on `main`. The existing hook tests also do not cover the claimed isolation behavior.

## Evidence

- **Current hook routing still hardcodes default main session:** `dispatchAgentHook()` resolves `mainSessionKey` from config once and uses it for both non-delivered hook summaries and error events, regardless of `value.agentId` or the request session key.
  - file: [src/gateway/server/hooks.ts:45](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/gateway/server/hooks.ts#L45)

- **Leak path is visible in the enqueue calls:** The non-delivered result path enqueues to `mainSessionKey` at lines 98-100, and the error path enqueues to the same `mainSessionKey` at lines 108-110.
  - file: [src/gateway/server/hooks.ts:98](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/gateway/server/hooks.ts#L98)

## Risks / Open Questions

- In multi-agent setups, non-delivered hook summaries and hook error messages can still be exposed in the default agent's main session instead of the intended target agent session.
- Because current tests do not assert session isolation for hook system events, this regression can persist unnoticed.
