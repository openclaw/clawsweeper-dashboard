## Summary

Keep open: current main still advertises configured Voice Call realtime tools to the provider without a user-facing handler binding path, and the concrete `end_call` case remains unimplemented on main.

## What This Changes

The issue requests a way for configured Voice Call realtime tools such as `end_call` to be backed by runtime handlers instead of being advertised to the realtime model without a callable binding.

## Best Possible Solution

Add a narrow, documented Voice Call realtime action contract: at minimum, an explicitly configured `end_call` built-in should bind to `manager.endCall(callId)` with regression coverage, while generic `realtime.tools[]` action mapping remains a deliberate follow-up API decision.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, and current main advertises it to the provider while any provider tool call for that name reaches `executeToolCall` with no registered handler and returns `Tool "end_call" not available`.

## Solution Assessment

Unclear as written. The suggested default auto-registration would solve the voicemail hangup case, but a safer maintainable path is to bind explicitly configured Voice Call built-ins first and handle generic action mapping as a separate API decision.
