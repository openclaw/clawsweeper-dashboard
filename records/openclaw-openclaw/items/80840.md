## Summary

Keep open: current main and the latest release still advertise configured Voice Call realtime tools to providers while only binding `openclaw_agent_consult`, so the realtime `end_call` or action-binding gap remains and needs maintainer API/safety direction.

## What This Changes

The issue requests a supported path for Voice Call realtime tool schemas, especially `end_call`, to be backed by runtime handlers instead of only being advertised to the realtime model.

## Best Possible Solution

Define a narrow Voice Call realtime call-control contract, then either bind a guarded built-in `end_call` handler or stop advertising unhandled configured tools until an explicit action-binding API is approved.

## Reproduction Assessment

Yes, from source inspection rather than a live call. Current main accepts and advertises configured `realtime.tools`, but a provider call to any unregistered name reaches the missing-handler branch and returns `Tool "<name>" not available`.

## Solution Assessment

Unclear. The suggested direct `end_call` registration is plausible and narrow, but maintainers should first choose whether realtime call control is a built-in tool only or a broader config/action-binding contract.
