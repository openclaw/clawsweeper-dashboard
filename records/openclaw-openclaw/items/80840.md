## Summary

Keep open: current main still passes configured Voice Call realtime tool definitions to the provider without a user-facing handler binding path, and only the shared `openclaw_agent_consult` handler is registered. The concrete `end_call` request is valid, but the fix needs owner judgment on whether to add an explicit built-in realtime hangup tool, bind only configured built-ins, or design a generic action-mapping surface.

## What This Changes

The issue requests a runtime binding path for configured Voice Call realtime tools, especially an `end_call` function the realtime model can invoke to terminate outbound calls cleanly.

## Best Possible Solution

Add a narrow, documented Voice Call realtime action contract: at minimum an explicitly configured `end_call` built-in should bind to `manager.endCall(callId)` with regression coverage, while any generic `realtime.tools[]` to plugin-action mapping should be a deliberate follow-up API decision.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, the runtime advertises it to the provider, and any provider tool call for that name reaches `executeToolCall` with no registered handler and returns `Tool "end_call" not available`. I did not run a live phone call in this read-only review.

## Solution Assessment

Unclear as written: the five-line default auto-registration would solve the reported voicemail hangup case, but it may expose a new model-callable hangup capability by default. The safer maintainable direction is to first bind explicitly configured built-in actions such as `end_call`, then separately design any generic action-mapping config.
