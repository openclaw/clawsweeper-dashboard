## Summary

Keep open. Current main still lets voice-call advertise configured realtime function declarations while registering only the shared consult handler, so an `end_call` realtime tool can be offered to the model and then fail as unavailable at dispatch time; the earlier closed agent-consult work fixed only `openclaw_agent_consult`, not this voice-call-owned hangup path.

## What This Changes

Bind voice-call realtime tool calls so a model-advertised `end_call` function can terminate the active call instead of falling through as unavailable.

## Best Possible Solution

Add a narrow, documented voice-call-owned realtime `end_call` tool binding that calls `manager.endCall(callId)` for the active call, with regression coverage for advertised tool declarations and unregistered-tool behavior.

## Reproduction Assessment

Yes at source level: current main accepts and advertises `realtime.tools[]`, but only registers `openclaw_agent_consult`, so any configured `end_call` declaration reaches dispatch without a handler. I did not run a live telephony repro in this read-only sweep.

## Solution Assessment

Yes, a narrow plugin-local realtime `end_call` binding is the best first fix; a generic configurable action-binding system would be broader than the reported failure.
