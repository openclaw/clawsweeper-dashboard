## Summary

Keep open: current main and the latest release still pass custom Voice Call `realtime.tools` definitions to the realtime provider while only binding the shared `openclaw_agent_consult` handler. The outer `voice_call.end_call` action exists, but it is not reachable from the realtime model tool path, so the reported `end_call` gap remains a valid maintainer decision item.

## What This Changes

The issue asks for Voice Call realtime tools, especially `end_call`, to have a runtime handler binding path instead of being advertised to the model without dispatch.

## Best Possible Solution

Add an explicit owner-approved Voice Call realtime tool binding contract: either an opt-in built-in `end_call` realtime tool or a narrow config action mapping, with handler registration, authorization docs, and regression tests.

## Reproduction Assessment

Yes from source, not from a live call: custom `realtime.tools` are included in the provider tool list, only the consult handler is registered by runtime setup, and missing handlers submit `Tool "<name>" not available`. No live Twilio/OpenAI realtime call was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the realtime call-control contract. The suggested built-in `end_call` path is plausible, but it is write-capable and should be explicitly opt-in or otherwise owner-approved with docs and tests.
