## Summary

Keep open: current main still advertises configured Voice Call realtime tools to the provider while only binding the shared `openclaw_agent_consult` handler, so a configured `end_call` tool still reaches the unbound-tool error path. The request is valid, but exposing realtime call-control tools or a generic binding surface needs owner/product judgment before implementation.

## What This Changes

Request a Voice Call realtime tool binding path so configured `realtime.tools[]` entries, especially `end_call`, can be handled instead of only advertised to the model.

## Best Possible Solution

Add a documented Voice Call realtime action contract that at least binds an explicitly configured `end_call` tool to `manager.endCall(callId)` with regression coverage, while treating generic custom tool-to-action mapping as a separate API decision.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, and current main will advertise it to the provider but dispatch any provider call for that name through an empty handler lookup that returns `Tool "end_call" not available`.

## Solution Assessment

Unclear as proposed. The suggested unconditional built-in registration would solve the hangup case but expands model-callable call control; the narrower maintainable path is explicit built-in binding first, with generic action mapping only after maintainer approval.
