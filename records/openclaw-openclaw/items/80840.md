## Summary

Keep open: current main and the latest release still pass configured Voice Call realtime tool definitions through to the provider while only auto-binding the shared `openclaw_agent_consult` handler. The requested direct `end_call` or generic action-binding path remains a real product and safety decision.

## What This Changes

The issue requests a supported Voice Call realtime tool binding path so configured `realtime.tools[]` entries such as `end_call` can be invoked by the realtime model with a runtime handler.

## Best Possible Solution

Define an explicit realtime Voice Call tool contract, then implement the chosen bounded handler path with source tests and docs that state which realtime tools can execute and under what policy.

## Reproduction Assessment

Yes, from source inspection. Current main passes configured realtime tool schemas into the provider, registers only the shared consult handler, and returns the missing-handler error for any unregistered tool name.

## Solution Assessment

Unclear. The suggested direct `end_call` registration is plausible, but the best maintainable fix depends on whether maintainers want a narrow built-in call-control tool or a broader action-binding contract with safety rules.
