## Summary

Keep open: current main still advertises configured Voice Call realtime tools to the provider while only binding the shared consult handler, so the reported unhandled end_call path remains valid and needs owner/product judgment before adding model-callable call-control behavior.

## What This Changes

The issue asks for Voice Call realtime tool definitions such as end_call to have a runtime handler binding path instead of being advertised without callable behavior.

## Best Possible Solution

Add an owner-reviewed, documented Voice Call realtime action contract: bind explicitly configured built-in call-control tools such as end_call to manager.endCall(callId), add regression coverage for provider tool calls, and leave any generic tools-to-actions mapping as a separate API decision.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure a custom realtime end_call tool, and any provider tool call for that name reaches executeToolCall without a registered handler and returns Tool "end_call" not available; I did not run a live phone call.

## Solution Assessment

Unclear as suggested. Auto-registering end_call whenever realtime.toolPolicy is not none would fix the reported voicemail hangup case, but the safer maintainable path is to bind only explicitly configured built-in call-control tools before considering a generic action-mapping config.
