## Summary

Keep open: current main and v2026.6.8 still pass configured Voice Call realtime tool schemas to providers while only binding `openclaw_agent_consult`, so the requested realtime `end_call` or explicit action-binding path remains unresolved and needs a maintainer API/safety decision.

## What This Changes

The issue asks for Voice Call realtime custom tool entries to have a callable handler path, especially an `end_call` realtime tool for clean voicemail termination.

## Best Possible Solution

Define a narrow Voice Call realtime call-control contract, then either bind a built-in `end_call` handler with explicit guardrails or stop advertising unhandled configured tools until an explicit action-binding API is approved.

## Reproduction Assessment

Yes, from source inspection rather than a live call. Current main accepts and advertises `realtime.tools`, but a provider call to an unregistered name reaches the missing-handler branch and returns `Tool "<name>" not available`.

## Solution Assessment

Unclear. The suggested direct `end_call` registration is plausible and narrow, but maintainers should first choose whether realtime call-control is a built-in tool only or a broader config/action-binding contract.
