## Summary

Keep open: current main still has the reported Voice Call realtime tool gap. Configured `realtime.tools[]` entries are advertised to the provider, but runtime startup only binds the shared consult handler, so an `end_call` tool call still reaches dispatch with no handler.

## What This Changes

The issue asks Voice Call realtime tools such as `end_call` to have a handler binding path instead of being merely advertised to the realtime provider.

## Best Possible Solution

Add a narrow, documented Voice Call realtime built-in action contract that binds explicitly configured `end_call` to `manager.endCall(callId)` with regression coverage; leave generic custom-tool-to-action mapping for a separate design decision.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, current runtime advertises it to the provider, and dispatch returns `Tool "end_call" not available` because no handler is registered.

## Solution Assessment

Unclear as suggested. A default auto-registration would solve the report but changes the realtime model's call-control surface; binding explicitly configured built-in tools is the safer first step, while a generic action-mapping surface needs separate product review.
