## Summary

Keep open: current main still accepts and advertises Voice Call `realtime.tools[]` entries but only registers a handler for `openclaw_agent_consult`, so a configured `end_call` realtime tool still reaches the reported unavailable-tool path. The issue is valid, but the fix needs owner judgment because it would expose model-callable call-control behavior or define a new generic binding contract.

## What This Changes

The issue requests a Voice Call realtime tool binding path so provider-advertised tools such as `end_call` can execute runtime handlers instead of returning an unavailable-tool error.

## Best Possible Solution

Add a documented Voice Call realtime action contract: at minimum bind an explicitly configured `end_call` built-in to `manager.endCall(callId)` with regression coverage, and treat generic `realtime.tools[]` action mapping as a separate deliberate API decision.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure a custom realtime `end_call` tool and current main will advertise it to the provider, then dispatch the provider tool call with no registered handler and return `Tool "end_call" not available`. I did not run a live carrier/provider call in this read-only review.

## Solution Assessment

Unclear as proposed: the requested five-line default registration would solve the voicemail hangup case, but it also exposes a write-capable hangup tool to realtime models by default. A safer maintainable direction is explicit built-in binding first, with a generic action-mapping contract only after owner review.
