## Summary

Keep open: current main still advertises configured Voice Call realtime tools without a public handler/action binding, so `end_call` remains unreachable from the realtime model. The gap is valid, but the fix needs owner judgment because exposing write-capable call-control tools changes the realtime model capability surface.

## What This Changes

The issue requests a Voice Call realtime tool binding path so configured `realtime.tools[]` entries such as `end_call` can be executed instead of advertised without handlers.

## Best Possible Solution

Add a narrow, documented Voice Call realtime action contract for explicitly configured built-ins such as `end_call`, with regression coverage; any generic `realtime.tools[]` to plugin-action mapping should be a separate API decision.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, current main advertises it to the provider, and a provider tool call reaches `executeToolCall` with no registered handler and returns `Tool "end_call" not available`. I did not run a live phone call.

## Solution Assessment

Unclear as proposed: auto-registering `end_call` by default would solve the voicemail hangup case, but it changes the model-callable call-control surface. The safer maintainable path is to bind explicitly configured built-in actions first and decide any generic action-mapping config separately.
