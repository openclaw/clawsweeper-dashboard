## Summary

Keep open: current main still advertises configured Voice Call realtime tool schemas to realtime providers but only binds `openclaw_agent_consult`, so the requested realtime `end_call` or explicit action-binding path remains unresolved and needs maintainer API/safety direction.

## What This Changes

The issue asks for Voice Call realtime tools to have a runtime handler path, especially an `end_call` tool or explicit action binding, so advertised model-callable schemas do not fail at dispatch.

## Best Possible Solution

Define a narrow Voice Call realtime call-control contract, then either bind a guarded built-in `end_call` handler or stop advertising unhandled custom tools until an explicit action-binding API exists.

## Reproduction Assessment

Yes, from source inspection rather than a live call: current main accepts and advertises `realtime.tools`, but provider calls to unregistered names reach the missing-handler branch and return `Tool "<name>" not available`. I did not run a live telephony call in this read-only review.

## Solution Assessment

Unclear. The suggested direct `end_call` registration is plausible and narrow, but maintainers should first choose whether realtime call control is a guarded built-in tool, a broader config/action-binding contract, or a docs/runtime change that stops advertising unhandled tools.
