## Summary

Keep open: current main still passes configured Voice Call realtime tool definitions to realtime providers, but runtime startup only binds `openclaw_agent_consult`. The reported `end_call` case remains valid, yet the fix needs owner judgment on the realtime call-control/tool-binding contract.

## What This Changes

The issue requests that Voice Call realtime tool definitions either bind to callable runtime handlers, especially `end_call`, or stop advertising unreachable tools.

## Best Possible Solution

Define a narrow, documented Voice Call realtime action contract: bind an explicitly configured `end_call` built-in to `manager.endCall(callId)` with regression coverage, and handle generic `realtime.tools[]` action mapping as a separate API decision.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, observe it is advertised through `resolveRealtimeVoiceAgentConsultTools`, then any provider tool call for that name reaches `executeToolCall` with no registered handler and returns `Tool "end_call" not available`.

## Solution Assessment

Unclear as written. A small built-in `end_call` binding would solve the reported voicemail hangup case, but default or generic realtime tool binding needs a deliberate call-control policy and API boundary.
