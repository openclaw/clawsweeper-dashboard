## Summary

Keep open: current main still advertises configured Voice Call realtime tools to providers without any user-facing handler binding path, and only the shared consult tool is registered. The reported `end_call` flow is source-reproducible, but the fix needs maintainer judgment because it would add model-callable call-control behavior.

## What This Changes

The issue requests a Voice Call realtime path for provider-advertised tools such as `end_call` to bind to runtime handlers.

## Best Possible Solution

Add a narrow, documented Voice Call realtime action contract: explicitly configured built-ins such as `end_call` should bind to `manager.endCall(callId)` with regression coverage, while generic custom tool-to-action mapping should be a deliberate follow-up API decision.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: configure a custom realtime `end_call` tool, the runtime advertises it, and any provider tool call for that name reaches `executeToolCall` without a registered handler and returns `Tool "end_call" not available`.

## Solution Assessment

Unclear as written. The suggested default auto-registration would solve the voicemail hangup case, but a safer maintainable solution should first decide whether `end_call` is an explicit configured built-in, a default realtime tool, or part of a generic action-mapping API.
