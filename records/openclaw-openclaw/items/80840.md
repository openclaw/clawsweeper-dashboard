## Summary

Keep open: current main still advertises configured Voice Call realtime tools to the provider without binding handlers for those names, while only the shared `openclaw_agent_consult` handler is registered. The reported `end_call` use case is valid, but the fix needs owner judgment because exposing model-callable hangup behavior or a generic action-mapping config changes the realtime call-control surface.

## What This Changes

The issue requests a Voice Call realtime tool-binding path so configured provider tools such as `end_call` can execute plugin actions instead of returning an unavailable-tool error.

## Best Possible Solution

Add an owner-approved Voice Call realtime action contract: at minimum bind an explicitly configured `end_call` built-in to `manager.endCall(callId)` with tests and docs, and treat generic tool-to-action mapping as a deliberate follow-up API design.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure a custom realtime `end_call` tool, current main advertises it to the provider, and a provider tool call for that name reaches `executeToolCall` with no registered handler and returns `Tool "end_call" not available`. I did not run a live phone call.

## Solution Assessment

Unclear as suggested: a small default auto-registration would solve the reported hangup case, but it may broaden realtime call-control behavior by default. The safer maintainable path is an explicit, documented built-in binding first, with any generic config action mapping handled as a separate product/API decision.
