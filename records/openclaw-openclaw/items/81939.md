## Summary

Keep open; the branch makes the narrow reset-command control-flow change for the linked regression, but it is not merge-ready because broader e2e/helper coverage and public docs still assert acknowledgement-only behavior.

## What This Changes

The PR changes non-ACP bare `/new` and `/reset` to fall through into the startup/model path unless reset hooks routed a reply, and updates reset-hook unit expectations.

## Best Possible Solution

Restore bare reset/new fallthrough through the existing startup prompt path, keep hook-routed replies terminal, preserve the empty-provider-call guard, and align e2e/helper tests plus docs with that contract.

## Reproduction Assessment

Yes, source-level. Current main returns a terminal reset/new command result before `get-reply-run.ts` can build the bare reset startup prompt; the PR body also includes live Gateway output for the after-fix `/new` path.

## Solution Assessment

No, not merge-ready yet. Returning `null` for non-routed bare resets is the narrow code direction, but the PR must update stale e2e/helper/docs expectations and preserve coverage against the old empty-provider-call regression.
