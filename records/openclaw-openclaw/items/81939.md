## Summary

Keep open with needs changes. The core control-flow direction is plausible and the supplied terminal proof is sufficient, but the PR still leaves broader e2e/helper expectations and public docs asserting the old acknowledgement-only behavior.

## What This Changes

The PR changes non-ACP bare `/new` and `/reset` command handling to fall through to the startup/model path unless reset hooks already routed a reply, and updates reset-hook unit expectations.

## Best Possible Solution

Restore bare reset/new fallthrough through the existing startup prompt path, keep hook-routed replies, ACP resets, and reset tails unchanged, and align e2e/helper tests plus public docs around that contract.

## Reproduction Assessment

Yes, source-level. On current main, bare `/new` and `/reset` return `shouldContinue: false` in `commands-reset.ts`, and the inline command handler returns that acknowledgement before `get-reply-run.ts` can resolve the bare reset startup prompt.

## Solution Assessment

No, not merge-ready yet. Returning `null` for non-routed bare resets is the narrow fix direction, but the PR must update stale e2e/helper/docs expectations and preserve the empty-provider-call regression guard.
