## Summary

Keep open: the run-id surrender direction is useful, but the current patch can drop the only visible assistant final when history reload adopts an empty in-flight run without creating a ChatLog streaming component.

## What This Changes

The PR adds TUI event-handler run-id surrender tracking and a ChatLog streaming-run query to avoid duplicate assistant output after sessions.changed history reloads.

## Best Possible Solution

Keep the run-id surrender approach only if it first proves history restored visible output or an in-flight component; otherwise let displayable finals render and dedupe with a narrower visible-row check.

## Reproduction Assessment

Yes at source level: current main can reload history on sessions.changed, restore an empty in-flight run without a ChatLog component, and later receive a displayable final for that run.

## Solution Assessment

No for the current patch shape: run-id-only final suppression is plausible for the duplicate case, but it is not the best fix until it proves the answer is already visible before returning.
