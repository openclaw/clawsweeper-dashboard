## Summary

Keep open: this PR targets a real TUI duplicate-message bug and has useful proof, but the current run-id surrender branch can suppress a displayable late final when history restored an empty in-flight run without a ChatLog streaming component.

## What This Changes

Adds TUI run-id surrender tracking across sessions.changed reloads, ChatLog.hasStreamingRun(), and regression tests to avoid duplicate assistant messages after external-channel history replay.

## Best Possible Solution

Keep run-id surrender only when history or ChatLog state proves the assistant answer is already visible; otherwise displayable finals should render, with tests for both visible-history and empty-in-flight orderings.

## Reproduction Assessment

Yes at source level: current main can adopt an empty in-flight run without a ChatLog streaming component, and PR head suppresses a surrendered displayable final when hasStreamingRun is false. I did not run a live Telegram/TUI reproduction in this read-only review.

## Solution Assessment

No for the current patch shape: run-id surrender is a good direction for the duplicate case, but final suppression must prove the answer is already visible before returning.
