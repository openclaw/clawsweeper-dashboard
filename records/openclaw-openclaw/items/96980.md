## Summary

## What This Changes

The PR adds persistent chat-finalized run maps in TUI event handling plus regression tests to suppress late chat events after `sessions.changed` clears finalized run state.

## Best Possible Solution

Land one canonical TUI fix that dedupes replayed-history/live-final duplicates only when the replayed row is actually visible, preserves stale-history recovery, and includes focused tests plus redacted real TUI/Gateway or Telegram proof.

## Reproduction Assessment

Yes at source level: current main clears the TUI on matching `sessions.changed` reloads, history replay may lack the just-finalized row, and this PR then drops a later displayable final for the same run. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The event-handler map is a plausible direction, but the guard uses prior display as proof of current visibility after a destructive reload; a safer fix must tie suppression to visible replay state or a narrowly scoped ChatLog-level duplicate check.
