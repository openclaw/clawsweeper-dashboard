## Summary

Keep open: current head addresses the earlier reset-path and proof blockers, but it still drops the finalized-run surrender marker on a late delta, so a later final for the same run can render a duplicate assistant row. The remaining blocker is a narrow state-machine repair plus regression coverage.

## What This Changes

This PR adds TUI runId-based surrender tracking around `sessions.changed` history reloads, exposes `ChatLog.hasStreamingRun()`, and expands ChatLog/event-handler tests for duplicate assistant replies.

## Best Possible Solution

Keep the event-handler runId surrender approach, but preserve finalized-run display suppression across late deltas before merge and add a regression test for the delta-then-final ordering.

## Reproduction Assessment

Yes, at source level: PR head allows finalized-run surrender to be deleted by a late delta, then a later final can pass the finalized guard because the display marker was cleared during reload. I did not run a live TUI reproduction for that exact delta-then-final ordering in this read-only review.

## Solution Assessment

No for the current head: event-handler runId surrender is the right owner boundary, but the implementation must keep finalized-run suppression intact across non-final late events before it is the best fix.

Fix PR https://github.com/openclaw/openclaw/pull/96980 on its branch. Preserve finalized-run suppression across late deltas, either by retaining the finalized surrender marker until terminal handling or by restoring finalized/chat finalized display markers before returning. Validate with `node scripts/run-vitest.mjs src/tui/tui-event-handlers.test.ts src/tui/components/chat-log.test.ts`; run the TUI PTY lane if the handler control flow changes beyond the narrow fix.
