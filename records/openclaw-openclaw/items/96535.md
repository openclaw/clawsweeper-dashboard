## Label Justifications

- `P2`: This is a normal-priority Slack outbound delivery-mirror session bug with limited blast radius to Slack thread routing and persisted mirror state.
- `impact:session-state`: The reported behavior creates sibling assistant-only sessions instead of updating the intended Slack thread session.
- `impact:message-loss`: Outbound mirror records are associated away from the conversation thread, leaving persisted channel-message history incomplete or misrouted for that thread.

## Summary

Keep open: current main and v2026.6.10 still have the source-level Slack outbound mirror route where `replyToId` wins before the recovered current thread session, and the two closing PRs are open candidate fixes rather than merged implementation.

## What This Changes

Fix Slack outbound delivery-mirror routing so inherited threaded replies append to the existing Slack thread session instead of creating per-reply mirror-only sessions.

## Best Possible Solution

Land one narrow Slack-owned fix for the canonical route gap, with redacted real Slack/session-store proof, then let the merged PR close this issue.

## Reproduction Assessment

Yes at source level: current main passes Slack `replyToId`, `threadId`, and `currentSessionKey` into a helper whose default precedence lets the inherited child reply timestamp win before the recovered current thread session. I did not run a live Slack workspace replay.

## Solution Assessment

Yes, the narrow maintainable direction is Slack-owned routing that distinguishes inherited same-thread replies from explicit reply targets; a broad generic precedence change would be riskier for other channels.
