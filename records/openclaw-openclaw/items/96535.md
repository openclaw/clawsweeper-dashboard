## Summary

Keep open: current main and the latest release still have a source-reproducible Slack outbound mirror path where an inherited child reply timestamp can win over the conversation's existing thread session and create a fresh mirror-only session entry.

## What This Changes

The issue reports Slack outbound delivery mirrors creating new single-message thread sessions instead of reusing the existing Slack conversation thread session.

## Best Possible Solution

Fix Slack outbound mirror routing so inherited same-thread replies mirror into the canonical/current Slack thread session while preserving explicit reply/thread targets and ordinary top-level channel behavior.

## Reproduction Assessment

Yes, at source level: an inherited Slack child reply timestamp can enter outbound mirror routing and win over the recovered current thread session before `ensureOutboundSessionEntry` creates the missing mirror session. I did not run a live Slack workspace replay.

## Solution Assessment

Yes. The narrow maintainable direction is to canonicalize Slack inherited same-thread reply routing to the thread root or existing thread session for delivery mirrors, rather than adding a config flag or changing generic precedence for every channel.
