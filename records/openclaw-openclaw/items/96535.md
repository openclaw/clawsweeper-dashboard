## Summary

Keep open: current main still has a source-reproducible Slack outbound mirror path where an inherited child reply timestamp can win over the existing thread-root session and create a fresh mirror-only session entry.

## What This Changes

The issue asks OpenClaw to mirror Slack outbound thread replies into the conversation's existing thread session instead of creating a new single-message session for each reply timestamp.

## Best Possible Solution

Fix Slack outbound mirror routing so inherited same-thread replies mirror into the canonical/current thread-root session while preserving explicit reply targets, `topLevel`, `threadId: null`, and ordinary flat top-level channel sessions.

## Reproduction Assessment

Yes, at source level: a Slack same-channel message action can inherit the child `CurrentMessageId` as `replyTo`, suppress root auto-thread resolution, and let route precedence build a mirror session from that child timestamp. I did not run a live Slack workspace replay.

## Solution Assessment

Yes, the narrow maintainable fix is to canonicalize Slack inherited reply routing to the root/current thread session for delivery mirrors. A new config flag or generic precedence change would be broader than the bug requires.
