## Label Justifications

- `P2`: This is a normal-priority Slack session-state bug with clear source evidence and limited blast radius to outbound delivery mirrors.
- `impact:session-state`: The reported behavior creates near-empty sibling session records instead of updating the intended Slack thread session.
- `impact:message-loss`: Outbound delivery-mirror records are mis-associated away from the conversation's thread session, making the persisted channel-message history incomplete for that thread.

## Summary

Keep open: current main and the latest release still have a source-reproducible Slack outbound mirror path where a child reply timestamp can win over the existing thread session and create a fresh mirror-only session entry.

## What This Changes

The issue asks Slack outbound delivery mirrors to reuse the conversation's existing thread session instead of creating a new single-message session per bot reply.

## Best Possible Solution

Fix Slack outbound mirror routing so inherited same-thread replies mirror into the canonical/current thread-root session, while preserving explicit reply/thread targets and ordinary flat top-level channel sessions.

## Reproduction Assessment

Yes, at source level: a Slack message action can inherit a child `replyToId`, skip root auto-thread resolution, and let `buildThreadAwareOutboundSessionRoute` choose that child timestamp before the recovered current thread session. I did not run a live Slack workspace replay.

## Solution Assessment

Yes, the narrow maintainable fix is to canonicalize Slack inherited reply routing to the thread root or existing thread session for delivery mirrors, rather than adding a config flag or changing generic precedence for every channel.
