## Summary

Keep open: current main and the latest release still have a source-reproducible Slack outbound mirror path where an inherited child reply timestamp can win over the existing thread session and create a fresh mirror-only session entry.

## What This Changes

The issue reports that Slack outbound replies create separate delivery-mirror sessions keyed by each reply timestamp instead of reusing the existing Slack thread session.

## Best Possible Solution

Fix Slack outbound mirror routing so inherited same-thread replies mirror into the canonical/current Slack thread session while preserving explicit reply/thread targets and ordinary top-level channel sessions.

## Reproduction Assessment

Yes, at source level: an inherited Slack child reply timestamp can enter outbound mirror routing and win over the recovered current thread session before ensureOutboundSessionEntry creates the missing mirror session. I did not run a live Slack workspace replay.

## Solution Assessment

Yes, the narrow maintainable direction is to canonicalize Slack inherited same-thread reply routing to the thread root or existing thread session for delivery mirrors, rather than adding a config flag or changing generic precedence for every channel.
