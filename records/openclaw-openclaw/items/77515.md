## Summary

Keep this PR open: it targets a real active-memory noise bug, but the new catch is broader than the stated missing-memory-plugin case and can silently hide a different existing configuration error.

## What This Changes

The PR catches the active-memory recall sub-agent's empty explicit memory-tool allowlist error and turns it into a `NONE` result, plus adds a changelog fix entry.

## Best Possible Solution

Keep the missing-memory-plugin no-op behavior, but only swallow the allowlist guard error when the reason is actually `no registered tools matched`; preserve unavailable diagnostics for unsupported-tool models or other tool-policy failures.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: active-memory always supplies the explicit memory-tool allowlist, and the embedded runner throws the guard error when that allowlist resolves to no callable tools. I did not run a live gateway/provider reproduction in this read-only review.

## Solution Assessment

No. The intended fix is good, but catching by prefix alone is not the narrowest maintainable solution because the same prefix also covers non-tool-capable model failures that should remain visible.
