## Summary

Keep open: the PR addresses real active-memory allowlist noise, but the catch is too broad and can hide a distinct unsupported-tools diagnostic when memory tools are present but the selected recall model cannot use tools.

## What This Changes

The PR makes active-memory return an empty `NONE` recall result for empty explicit memory-tool allowlist errors and adds a changelog fix entry.

## Best Possible Solution

Keep the graceful no-op for missing memory tools, but only swallow the guard error when its reason is `no registered tools matched` and add regression coverage for both missing-tools and unsupported-model cases.

## Reproduction Assessment

Yes. Source inspection shows active-memory always passes the explicit memory-tool allowlist, and the embedded-runner guard throws the reported error when that allowlist resolves to no callable tools; the PR regression is also source-reproducible because the same prefix covers unsupported-tool models.

## Solution Assessment

No. Returning `NONE` is appropriate for missing registered memory tools, but the catch must be narrowed to the missing-tools reason so unsupported-model failures remain visible.
