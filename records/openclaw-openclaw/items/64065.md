Race A and Race B remain lower-severity observations with existing generation/null-safe handling rather than a clear automation target.

## What This Changes

The issue reports three async interleaving races in command queues, global hook-runner access, and plugin-binding approval persistence.

## Best Possible Solution

Land a narrow Race C fix in `src/plugins/conversation-binding.ts` that serializes the persistent approval read/merge/write path and adds an out-of-order write regression test, while leaving the mitigated queue reset and hook-runner behaviors alone.

## Reproduction Assessment

Yes, at source level. Current main still lets overlapping `allow-always` approvals compute replacement approval files and await disk writes without a shared critical section, which can persist stale data after an interleaving.

## Solution Assessment

Yes for the actionable Race C path. Serializing the approval persistence critical section with the existing async-lock pattern is the narrowest maintainable fix; broader queue or hook-runner semantic changes are not justified by the current evidence.
