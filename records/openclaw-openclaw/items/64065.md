## Summary

Keep this open. Current main still has the un-serialized plugin binding approval persistence path, so the actionable Race C data-loss bug remains; the reported Race A and Race B behaviors still look like lower-severity, mitigated design edges rather than the repair target.

## What This Changes

The issue requests fixes or documentation for three async interleaving races, with the main unmitigated bug being stale on-disk plugin binding approvals after concurrent allow-always saves.

## Best Possible Solution

Land a narrow fix in src/plugins/conversation-binding.ts that serializes persistent approval updates so overlapping allow-always saves merge against the latest shared approvals and leave plugin-binding-approvals.json consistent with memory. Add a focused regression test in src/plugins/conversation-binding.test.ts and a user-facing changelog entry; leave command-queue reset and hook-runner initialization behavior unchanged unless a separate maintainer decision says otherwise.

## Reproduction Assessment

Yes. The actionable Race C has a high-confidence source-level reproduction path because current main still updates approvalsCache before awaiting writeJsonAtomic, with no lock around addPersistentApproval or saveApprovals; the reporter's RAPID scripts are external, but the checked source still matches the interleaving.

## Solution Assessment

Yes. Serializing the plugin binding approval read/merge/write path, preferably using the existing createAsyncLock pattern already used for file-backed state, is the narrowest maintainable fix for the unmitigated data-loss behavior.
