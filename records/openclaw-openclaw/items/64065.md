## Summary

Keep this open. Current main still has the un-serialized plugin binding approval persistence path, so the actionable Race C data-loss bug remains; the command-queue reset and hook-runner init observations still look like lower-severity mitigated edge cases.

## What This Changes

The issue reports three async interleaving races in command queue reset, global hook-runner initialization, and plugin binding approval persistence, with Race C called out as the unmitigated data-loss path.

## Best Possible Solution

Land a narrow fix in src/plugins/conversation-binding.ts that serializes the persistent approval read/merge/write path so overlapping allow-always approvals leave disk and memory with the same latest merged state.

## Reproduction Assessment

Yes. Current main still has the exact source-level interleaving: addPersistentApproval computes a new approvals file from shared cache state, while saveApprovals updates memory before awaiting writeJsonAtomic, allowing overlapping disk writes to finish out of order.

## Solution Assessment

Yes. Serializing the plugin binding approval critical section with the existing async lock helper, plus a focused out-of-order write regression test, is the narrowest maintainable fix; the lower-severity command-queue and hook-runner behaviors should stay out of this repair unless a maintainer asks for a separate product change.
