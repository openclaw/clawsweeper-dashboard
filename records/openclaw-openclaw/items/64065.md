Keep open: current main and v2026.5.12 still have the unsynchronized persistent approval read/merge/write path, so the Race C data-loss report remains actionable; Race A and Race B are lower-severity observations already mitigated by existing generation and null-safe patterns.

## What This Changes

The issue reports async interleaving races in command queue reset handling, global hook-runner initialization, and plugin conversation-binding approval persistence.

## Best Possible Solution

Land a narrow Race C fix in `src/plugins/conversation-binding.ts` that serializes the persistent approval read/merge/write path and adds an out-of-order write regression test, while leaving the mitigated queue reset and hook-runner behaviors alone.

## Reproduction Assessment

Yes, at source level. Current main still lets overlapping `allow-always` approvals compute replacement approval files and await disk writes without a shared critical section, which can persist stale data after an interleaving.

## Solution Assessment

Yes for the actionable Race C path. Serializing the approval persistence critical section with the existing async-lock pattern is the narrowest maintainable fix; broader command-queue or hook-runner semantic changes are not justified by the current evidence.

## Repair Work Prompt

Fix the Race C persistent plugin-binding approval race from https://github.com/openclaw/openclaw/issues/64065. In `src/plugins/conversation-binding.ts`, serialize the read/merge/write critical section for persistent `allow-always` approvals so concurrent approvals cannot leave `plugin-binding-approvals.json` with stale data while memory has newer approvals. Prefer the existing `createAsyncLock` pattern from `src/infra/json-files.ts`; keep the fix scoped to plugin binding approvals and do not change command queue reset semantics, global hook-runner null-safe behavior, or start a broad SQLite/runtime-state migration. Add a focused regression test in `src/plugins/conversation-binding.test.ts` that proves out-of-order overlapping saves cannot drop the later approval on disk, and add a `CHANGELOG.md` fix entry because this prevents user-visible approval loss on restart.
