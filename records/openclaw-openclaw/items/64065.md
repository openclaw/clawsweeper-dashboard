## Summary

Keep open: current main and the latest release still leave plugin-binding approval persistence unsynchronized, so the actionable Race C data-loss path remains. The command-queue reset and hook-runner observations still look lower severity and mitigated by existing generation/null-safe behavior.

## What This Changes

The issue asks to address three async interleaving races, with the actionable request to serialize plugin-binding approval persistence so disk and memory cannot diverge.

## Best Possible Solution

Land a narrow fix in `src/plugins/conversation-binding.ts` that serializes the persistent approval read/merge/write path and adds a focused out-of-order write regression test; leave the mitigated Race A and Race B behavior alone unless maintainers request separate policy changes.

## Reproduction Assessment

Yes, at source level. Current main still lets overlapping `allow-always` approvals compute and persist replacement approval files without a shared critical section, which is enough to reproduce the stale disk write interleaving described in the report.

## Solution Assessment

Yes. Serializing the plugin binding approval critical section with an existing async lock or equivalent single-flight queue plus a targeted regression test is the narrowest maintainable fix.
