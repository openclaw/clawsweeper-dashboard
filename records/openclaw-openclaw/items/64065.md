## Summary

Keep this open: current main still has the un-serialized plugin binding approval persistence path, so the actionable Race C data-loss bug remains. The command-queue reset and hook-runner initialization observations still look lower-severity and mitigated by existing generation/null-safe patterns.

## What This Changes

The issue reports async interleaving races in command queue resets, hook runner initialization, and plugin binding approval persistence, with Race C claiming stale disk writes can lose approvals after restart.

## Best Possible Solution

Land a narrow fix in src/plugins/conversation-binding.ts that serializes the persistent approval read/merge/write path so overlapping allow-always approvals leave disk and memory with the same latest merged state.

## Reproduction Assessment

Yes, at source level. Current main still lets addPersistentApproval compute a new approvals file from shared cache state while saveApprovals updates memory before awaiting writeJson, which is enough to reproduce the out-of-order write hazard without a product decision.

## Solution Assessment

Yes. Serializing the plugin binding approval critical section with the existing async lock helper plus a focused out-of-order write regression test is the narrowest maintainable fix; the lower-severity command-queue and hook-runner observations should stay out of this repair unless maintainers request separate changes.

Fix https://github.com/openclaw/openclaw/issues/64065 by addressing only Race C in src/plugins/conversation-binding.ts. Current main computes persistent allow-always approvals from shared in-memory cache and saveApprovals updates that cache before awaiting writeJson, so overlapping allow-always approval writes can persist stale data to plugin-binding-approvals.json.
