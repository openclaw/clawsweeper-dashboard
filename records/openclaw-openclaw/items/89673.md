## Summary

Keep open: this is an active MEMBER-authored bug-fix PR with a focused patch and no line-level blocking finding, but it changes the compatibility-sensitive meaning of `session.writeLock.maxHoldMs` from holder-watchdog-only to cross-process contender reclaim, so maintainers need to explicitly accept that session-state policy before merge.

## What This Changes

The PR removes `hold-exceeded` from the report-only stale lock reasons, adds a live-owner maxHoldMs reclaim regression test, and updates the existing maxHoldMs help/docs wording.

## Best Possible Solution

Land this only after maintainers explicitly accept `maxHoldMs` as the hard cross-process reclaim deadline for unchanged live-holder session locks, keeping the code, regression test, help text, and public docs aligned.

## Reproduction Assessment

Yes, source-reproducible: current main detects `hold-exceeded` when a lock exceeds the holder-recorded maxHoldMs, then refuses to remove locks whose stale reasons are all report-only. I did not run a live overnight gateway reproduction in this read-only review.

## Solution Assessment

Yes, if maintainers accept the policy: this is the narrow owner-boundary fix using the existing fs-safe unchanged-snapshot removal contract. If maxHoldMs must stay watchdog-only for existing users, the safer solution is an explicit opt-in reclaim policy instead of changing the default meaning.
