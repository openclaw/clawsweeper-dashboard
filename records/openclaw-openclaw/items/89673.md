## Summary

Keep open for normal review rather than maintainer cleanup close. The branch is a narrow, source-correct fix for the linked session-lock bug, but it changes when another process may reclaim a live holder's transcript lock and needs docs/help alignment plus explicit maintainer acceptance of that session-state policy.

## What This Changes

The PR removes `hold-exceeded` from the report-only stale-lock reasons so a contender can reclaim an unchanged lock after the holder's recorded `maxHoldMs`, and adds a regression test for that live-owner case.

## Best Possible Solution

Land the narrow reclaim fix after maintainers accept holder-recorded `maxHoldMs` as an authoritative cross-process deadline and update docs/config help to describe the contender reclaim behavior.

## Reproduction Assessment

Yes at source level: current main can classify an overdue live holder as `hold-exceeded`, but removal is blocked because that reason is report-only. I did not run a live overnight gateway reproduction in this read-only review.

## Solution Assessment

Mostly yes: removing `hold-exceeded` from the report-only set is the narrow fix for the observed acquisition failure, while keeping `too-old` and doctor cleanup conservative. The safer merge shape includes docs/config-help updates and maintainer acceptance of the session-lock policy.
