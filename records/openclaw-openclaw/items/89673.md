## Summary

Keep open for maintainer review: the PR is a narrow candidate fix for the linked live-holder `maxHoldMs` lock wedge, but it intentionally changes session-state concurrency semantics for an existing setting and is MEMBER-authored, so it should not be auto-closed or auto-repaired.

## What This Changes

The PR drops `hold-exceeded` from report-only stale lock reasons, adds a regression test for reclaiming a live OpenClaw-owned lock past its recorded `maxHoldMs`, and updates existing maxHoldMs help/docs text.

## Best Possible Solution

Land this only after maintainers explicitly accept `maxHoldMs` as the hard cross-process reclaim deadline for unchanged live-holder session locks, keeping code, regression coverage, help text, and docs aligned.

## Reproduction Assessment

Yes, source-reproducible: current main detects `hold-exceeded` for locks older than the holder-recorded `maxHoldMs`, then refuses to remove locks whose stale reasons are all report-only. I did not run a live overnight gateway reproduction in this read-only review.

## Solution Assessment

Yes, if maintainers accept the policy: this is the narrow owner-boundary fix using the existing fs-safe unchanged-snapshot removal contract. If `maxHoldMs` must stay watchdog-only for existing users, the safer solution is an explicit opt-in reclaim policy instead of changing the default meaning.
