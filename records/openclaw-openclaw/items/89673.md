## Summary

Keep this PR open for maintainer review: the runtime change is a narrow source-backed fix for live-but-overdue session locks, but it changes session-state concurrency semantics and still leaves user-facing maxHoldMs docs/help describing watchdog-only behavior.

## What This Changes

The PR makes contending session writers reclaim an unchanged live OpenClaw-owned lock after the holder's recorded maxHoldMs and adds a regression test for that case.

## Best Possible Solution

Merge only after maintainers accept maxHoldMs as the cross-process reclaim deadline and the public docs/config help describe that contract.

## Reproduction Assessment

Yes. Current main marks a live lock as hold-exceeded when respectMaxHold is enabled, then refuses removal because hold-exceeded is report-only, which the fs-safe lock manager converts into file_lock_stale instead of reclaiming.

## Solution Assessment

Not yet. The runtime patch is a plausible narrow fix, but the best merge state needs maintainer acceptance of the session-state policy and docs/help that define maxHoldMs as both watchdog and contender-reclaim behavior.
