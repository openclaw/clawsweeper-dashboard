## Summary

Keep open: this is a focused candidate fix for an unresolved session write-lock failure, but merging it changes the meaning of the existing `maxHoldMs` lock contract and needs maintainer acceptance of that session-state policy.

## What This Changes

The branch makes a live holder's expired `maxHoldMs` reclaimable by contenders, adds regression coverage, and updates the existing session write-lock help/docs.

## Best Possible Solution

Land this only after maintainers accept `maxHoldMs` as the hard reclaim deadline for unchanged live-holder session locks, with the code, regression test, and docs/help kept aligned.

## Reproduction Assessment

Yes, at source level: current main detects `hold-exceeded` but keeps it report-only, and the PR's added live-owner test exercises the reclaim path. The overnight gateway lockup itself was not live-reproduced in this review.

## Solution Assessment

Yes, if maintainers accept the policy: the patch is a narrow owner-boundary fix using the existing `fs-safe` remove-if-unchanged contract. A separate opt-in policy would be safer only if `maxHoldMs` must remain watchdog-only for existing users.
