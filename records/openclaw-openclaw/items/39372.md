## Summary

Keep this PR open because it targets a real requested capability, but the patch is not merge-ready: it changes a documented thread-bound contract and leaves a concrete ACP inline-delivery leakage path acknowledged in review discussion.

## What This Changes

The PR removes native subagent and ACP `mode:"session"` plus `thread:true` guards, adds headless-session notes and failed-start cleanup, and updates spawn-hook coverage for unthreaded persistent sessions.

## Best Possible Solution

If maintainers accept headless persistent sessions, land a rebased implementation that decouples persistence from thread binding in both native and ACP paths, gates ACP inline delivery on a real binding, and updates tests, docs, and changelog; otherwise keep the remaining decision on #53548.

## Reproduction Assessment

Yes. Source inspection of current main shows both native and ACP `sessions_spawn` reject `mode:"session"` without `thread:true`, and the native test suite asserts that rejection path.

## Solution Assessment

No, not as currently proposed. The decoupling direction may be maintainable, but the patch must first prevent ACP inline delivery without a real binding and align the documented product contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require thread binding before ACP inline delivery:** `src/agents/acp-spawn.ts:291-294`
  - body: Removing the ACP `mode:"session" && !thread` guard makes headless persistent ACP spawns reachable, but the delivery planner still enables `deliver: true` for any session-mode spawn with a requester target. That can send worker output into the requester conversation even though no thread or conversation binding was prepared; gate inline delivery on an actual binding or keep headless sessions internal.
  - confidence: 0.87
