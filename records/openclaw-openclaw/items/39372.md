## Summary

Keep open: the capability is still missing on current main and is tracked by https://github.com/openclaw/openclaw/issues/53548, but this patch is not mergeable because it changes a documented contract while leaving ACP inline delivery, tool schema, and docs inconsistent, and it lacks real behavior proof.

## What This Changes

The branch removes the native subagent and ACP `mode="session"`/`thread=true` guards, adds headless-session accepted notes plus subagent cleanup, and changes spawn-hook coverage to expect unthreaded persistent sessions to start.

## Best Possible Solution

Land a maintainer-approved implementation under https://github.com/openclaw/openclaw/issues/53548 that decouples persistence from thread binding across API, schema, docs, and tests, while allowing ACP inline delivery only when a binding exists.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: native and ACP `sessions_spawn` still reject `mode:"session"` without `thread:true`, and tests/docs encode that current behavior.

## Solution Assessment

No, not as currently proposed. Decoupling may be the right direction, but this branch must first fix ACP routing, expose the mode in the tool schema, update docs, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a bound ACP target before inline delivery:** `src/agents/acp-spawn.ts:291-294`
  - body: Removing the ACP `mode="session" && !thread` guard makes headless persistent ACP spawns reachable, but the delivery planner still enables inline delivery for any session-mode spawn with a requester target. Gate inline delivery on an actual binding or keep headless sessions internal so child output cannot be sent to the requester conversation without a prepared binding.
  - confidence: 0.9
- **[P2] Expose session mode when threads are unavailable:** `src/agents/tools/sessions-spawn-tool.ts:153-183`
  - body: The runtime guard removal is not enough for the advertised non-thread-channel flow because `createSessionsSpawnToolSchema` still restricts `mode` to `"run"` when no thread availability is detected. Agents on channels without thread binding will not be offered `mode:"session"`, so the schema and description need to decouple session persistence from thread availability too.
  - confidence: 0.86
- **[P2] Update the documented session contract:** `docs/tools/subagents.md:211-213`
  - body: This PR changes the public `sessions_spawn` contract, but the subagent and ACP docs still state that `mode:"session"` requires `thread:true`. Update the docs with the new headless-session semantics and the difference between persistence and thread binding before shipping this behavior.
  - confidence: 0.82
