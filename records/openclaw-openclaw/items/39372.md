## Summary

Keep open: the underlying decoupling request is still valid on current main, but this PR is not merge-ready because it is conflicting, lacks real behavior proof, and leaves headless ACP session output able to inline-deliver into requester chat.

## What This Changes

The PR removes native subagent and ACP thread-required guards for `mode="session"`, adds headless-session accepted notes/cleanup, and changes one spawn-hook test to expect the unthreaded path to proceed.

## Best Possible Solution

Land a rebased, maintainer-approved decoupling that makes headless native and ACP session delivery explicit, updates docs/tests/tool text, and proves both non-threaded and thread-bound flows.

## Reproduction Assessment

Yes. Source inspection on current main shows native and ACP guards rejecting `mode="session"` without `thread=true`, and the spawn-hooks test asserts that rejection path; I did not run a live channel repro in this read-only review.

## Solution Assessment

No, not as proposed. Decoupling session persistence from thread binding is plausible, but the PR must first gate or redefine ACP headless delivery, update the documented/tool contract, rebase, and provide real behavior proof.
