## Summary

Keep open: the PR addresses a real requested behavior, but it changes the documented session/thread contract and still has a blocking ACP routing defect plus missing real behavior proof.

## What This Changes

The branch removes native subagent and ACP `mode="session"`/`thread=true` guards, adds headless-session accepted notes and cleanup, and updates spawn-hook coverage to expect unthreaded persistent sessions to proceed.

## Best Possible Solution

After maintainer acceptance of https://github.com/openclaw/openclaw/issues/53548, land a hardened implementation that decouples persistence from thread binding, gates ACP inline delivery on an actual binding, preserves `thread:true` behavior, and updates tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection on current main shows native and ACP `sessions_spawn` reject `mode:"session"` without `thread:true`, and the current test suite explicitly asserts that rejection path.

## Solution Assessment

No, not as currently proposed. Decoupling may be the right direction if maintainers accept the product contract, but this patch must first prevent headless ACP inline delivery and include real behavior proof plus docs coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require a bound thread before ACP inline delivery:** `src/agents/acp-spawn.ts:294-297`
  - body: Removing the ACP session/thread guard makes headless ACP session-mode reachable, but the existing delivery planner still enables inline delivery for any session-mode spawn with a requester target. In chat-backed contexts that can send child ACP output into the requester conversation even though no thread or conversation binding was prepared, so gate inline delivery on an actual binding or keep headless sessions internal.
  - confidence: 0.88
