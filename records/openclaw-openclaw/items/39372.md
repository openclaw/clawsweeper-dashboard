## Summary

Keep open: current main and the latest release still enforce the thread-required contract, so this PR is not obsolete, but the proposed patch remains blocked by ACP inline-delivery semantics and missing real behavior proof.

## What This Changes

The PR removes native subagent and ACP `mode="session"` thread-required guards, adds headless-session accepted notes and cleanup, and updates one spawn-hook test expectation.

## Best Possible Solution

Land a maintainer-approved decoupling that updates native and ACP contracts, gates ACP delivery on a real binding or explicit parent-relay path, updates docs/tool text, and proves both headless and thread-bound flows.

## Reproduction Assessment

Yes. Source inspection on current main shows native and ACP guards rejecting `mode="session"` without `thread=true`, and the existing docs/tests still encode that contract; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not as proposed. Decoupling session persistence from thread binding is plausible, but the PR must first define safe ACP headless-session delivery semantics, update the documented/tool contract, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate ACP inline delivery for headless sessions:** `src/agents/acp-spawn.ts:294-297`
  - body: Removing this guard makes headless ACP session spawns reachable, but the PR head still enables inline delivery for any session-mode spawn with a requester target and no parent stream. In chat-backed contexts that can send child output to the requester conversation even though no thread was bound, so gate inline delivery on a real binding or explicit parent-relay contract before relaxing the guard.
  - confidence: 0.9
