## Summary

Keep this PR open: current main still rejects unthreaded session-mode spawns, but this branch is not mergeable as proposed because it makes headless ACP sessions reach inline requester-chat delivery and it still lacks required real behavior proof.

## What This Changes

The PR removes native subagent and ACP session-mode thread-required guards, adds headless-session accepted notes and cleanup, and changes a spawn-hook test expectation.

## Best Possible Solution

Land a maintainer-approved decoupling that updates native and ACP contracts, gates ACP inline delivery on a real binding or explicit parent-relay path, updates docs/tool text, and proves both headless and thread-bound flows.

## Reproduction Assessment

Yes. Source inspection on current main shows native and ACP guards reject `mode="session"` without `thread=true`, and current tests/docs still encode that contract; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No, not as proposed. Decoupling persistence from thread binding is plausible, but the PR must also define safe ACP delivery semantics, update the documented/tool contract, and provide real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require bound ACP delivery before inline delivery:** `src/agents/acp-spawn.ts:294-297`
  - body: Removing this guard makes headless ACP session spawns reachable, but `useInlineDelivery` still turns on for any session-mode spawn with a requester target and no parent stream. That can send child output into the requester conversation even though no thread was bound, so gate inline delivery on an actual binding or explicit parent-relay contract before relaxing the guard.
