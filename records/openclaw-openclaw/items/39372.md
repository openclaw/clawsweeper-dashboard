## Summary

Keep open. The PR targets a real requested capability that is still not on current main, but the patch changes the documented session/thread contract, still exposes a P1 ACP delivery-routing problem, and lacks required real behavior proof.

## What This Changes

The PR removes native subagent and ACP `mode="session"`/`thread=true` guards, adds headless-session result notes and cleanup, and updates spawn-hook coverage to expect unthreaded persistent sessions to proceed.

## Best Possible Solution

Review or replace this branch under https://github.com/openclaw/openclaw/issues/53548 with a rebased implementation that decouples persistence from thread binding, gates ACP delivery on an actual binding or safe parent relay, preserves `thread:true` behavior, and updates tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection on current main shows native and ACP `sessions_spawn` reject `mode:"session"` without `thread:true`, while the PR diff makes ACP headless session mode reachable without changing the session-mode inline-delivery planner.

## Solution Assessment

No, not as currently proposed. Decoupling may be the right direction if maintainers accept the product contract, but this PR must first prevent headless ACP inline delivery and provide real Telegram/headless plus Discord/thread-bound behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require a bound thread before ACP inline delivery:** `src/agents/acp-spawn.ts:289-294`
  - body: Removing the ACP thread-required guard makes headless session spawns reach the existing delivery plan, but `useInlineDelivery` is still true for any session-mode spawn with a requester target. That can deliver ACP child output into the requester conversation without a prepared binding, so require an actual binding/thread or route headless sessions through the parent relay/internal path.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the patch introduces a concrete ACP routing/privacy risk by making unbound session-mode delivery reachable.

Concerns:

  - body: By allowing ACP session mode without thread binding while leaving inline delivery keyed only to `spawnMode === "session"`, the patch can route child ACP output to a requester conversation instead of keeping the headless session internal or parent-relayed.
