## Summary

Keep open: this PR targets a real cron delivery leak, but the proposed fatal branch can still use the model self-debug text as the fatal delivery payload for the actual non-terminal warning shape, so it may still publish or persist the text it is meant to suppress.

## What This Changes

Adds a cron outcome classifier for unavailable-tool self-debug text and regression tests around `resolveCronPayloadOutcome`.

## Best Possible Solution

Carry a structured or error-payload diagnostic for exhausted unavailable-tool failures, skip normal announce delivery consistently for that fatal class, add metadata-shaped regression coverage, and prove a real cron/channel run no longer sends the self-debug text.

## Reproduction Assessment

Yes. Source inspection plus the linked real gateway trace show the path: repeated unavailable-tool recovery creates final assistant self-debug text, and cron outcome handling can treat non-terminal tool-warning payloads as recovered output.

## Solution Assessment

No. The PR is a plausible narrow mitigation, but as written it uses the self-debug text as fatal delivery text for the path that matters; the safer fix is a structured failure signal or last-error diagnostic that never enters normal announce delivery.
