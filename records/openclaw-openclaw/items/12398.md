## Summary

Keep open. Current main and v2026.6.5 still have no scheduler-owned post-run verifier or equivalent fail-capable plugin contract; this remains a broad cron API, runtime, UI, persistence, and command-execution security decision.

## What This Changes

Add scheduler-owned deterministic post-run checks for isolated agent cron jobs, with retries, escalation, and persisted verification state.

## Best Possible Solution

Add an additive scheduler-owned verification contract using bounded argv-based execution or a synchronous fail-capable plugin hook, with explicit security policy, durable run-log/state projection, and reuse of existing retry and failure-delivery machinery.

## Reproduction Assessment

No, the reporter's exact 18-of-50 batch incident cannot be reproduced from the available artifacts. However, current source provides a high-confidence deterministic path showing that an `ok` isolated-agent result is persisted without any outcome verifier.

## Solution Assessment

Unclear. Framework-controlled verification is the right ownership principle, but the proposed raw shell-string field is not yet demonstrably the safest or narrowest design compared with argv-based execution or a synchronous fail-capable plugin hook.
