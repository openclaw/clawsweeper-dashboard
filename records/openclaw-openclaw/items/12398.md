## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned verifier, and the existing issue discussion already routes this to design follow-up rather than cleanup closure.

## What This Changes

Request deterministic postcondition checks after cron agent jobs, with verifier command execution, retry context injection, escalation, and persisted state.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or a fail-capable cron post-run plugin hook, with typed config/state/protocol, bounded verifier execution, durable logs/UI/CLI/docs, retry context injection, and failure-alert escalation.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Source inspection gives high-confidence absence of the requested postconditions surface across cron types, protocol, runtime execution, CLI/UI, docs, and plugin hooks.

## Solution Assessment

Unclear. Deterministic post-run verification matches the reported outcome gap, but the proposed shell-command shape is not ready without core-vs-plugin ownership and host command-execution boundaries.
