## Summary

Keep this open: current main and v2026.6.5 still lack a cron postcondition contract or equivalent scheduler-owned post-run verifier for isolated agent jobs, and the remaining work is a broad product/security design rather than conservative cleanup.

## What This Changes

Requested an additive cron postcondition contract that runs deterministic post-run checks for isolated agent cron jobs, records verifier state, retries with failure context, and escalates when checks fail.

## Best Possible Solution

Choose a security-reviewed cron post-run verification contract, preferably at the narrowest core/plugin boundary that can fail a run, persist verifier state, inject bounded retry context, and reuse existing failure-alert/escalation delivery without duplicating command-cron or agent-finalize behavior.

## Reproduction Assessment

Yes at source level: current cron types, protocol schemas, and timer runtime do not expose or run a post-agent verifier, and isolated cron returns the agent result directly. I did not run a live cron because this is a missing feature rather than a broken existing contract.

## Solution Assessment

No final solution is selected yet. The proposed postconditions field is a plausible design, but maintainers still need to decide whether the durable fail/retry/escalation contract belongs in core cron or a new plugin lifecycle seam.
