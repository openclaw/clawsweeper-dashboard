## Label Justifications

- `P2`: This is a normal-priority reliability feature for autonomous cron workflows with broad but bounded cron/runtime impact.
- `impact:session-state`: The reported gap lets an isolated agent session terminate cleanly while durable cron state records success for an outcome that may not have happened.
- `impact:security`: The proposed solution would run configured host commands and expose verifier output, touching command-execution and sensitive-data boundaries.

## Summary

Keep open. Current main and the latest release still lack a cron postcondition verifier or scheduler-owned equivalent, and the requested shell-check design requires maintainer product and security review before implementation.

## What This Changes

Add optional deterministic postcondition checks to cron jobs so isolated agent runs can be verified, retried, or escalated after completion.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first define a fail-capable plugin hook that preserves scheduler-owned retry and escalation semantics with explicit command-execution boundaries.

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the exact core field versus fail-capable plugin hook design and shell-command security boundary need maintainer decisions first.
