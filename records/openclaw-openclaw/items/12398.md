## Label Justifications

- `P2`: This is a normal-priority reliability feature for autonomous cron workflows with broad but bounded cron/runtime impact.
- `impact:session-state`: The reported gap lets an isolated agent session terminate cleanly while durable cron state records success for an outcome that may not have happened.
- `impact:security`: The proposed solution would run configured host commands and expose verifier output, so it touches command-execution and sensitive-data boundaries.

## Summary

Keep open. Current main still lacks a cron postcondition verifier or scheduler-owned equivalent for turning a clean-but-incomplete isolated agent run into retry/escalation, and the requested shell-check design needs product and security review before implementation.

## What This Changes

The issue requests an opt-in cron postcondition contract that deterministically verifies isolated agent cron outcomes and can retry or escalate on verifier failure.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that preserves scheduler-owned retry and escalation semantics with explicit command-execution boundaries.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. Source inspection gives high-confidence proof that current main lacks the requested cron postcondition surface.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the exact core field versus fail-capable plugin hook design and shell-command security boundary need maintainer decisions first.
