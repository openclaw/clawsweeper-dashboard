## Summary

Keep open. Current main still lacks the requested cron postcondition contract, public protocol/state surface, and scheduler-owned deterministic verifier/retry path, while the existing maintainer-side comments already route this as cron runtime/protocol/security design work rather than cleanup closure.

## What This Changes

The issue requests an optional cron postconditions field that runs deterministic shell checks after isolated agent-turn jobs and uses failures to retry or escalate.

## Best Possible Solution

Add a reviewed additive post-run verification design spanning typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable cron post-run hook.

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection on current main gives high confidence that the requested field, protocol acceptance, verifier execution point, and retry/escalation state are absent.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but maintainers need to decide the core cron contract versus plugin seam and define host command-execution boundaries before implementation.
