## Summary

Current main still lacks a scheduler-owned verifier that can turn a cleanly terminated but incomplete isolated agent run into failure, retry, or escalation. Command cron jobs, failure alerts, Task Flow, and the observational `cron_changed` hook are useful adjacent capabilities, but none solves this issue’s central request; the feature should remain open for an explicit product and security design decision.

## What This Changes

Add deterministic post-run verification for isolated agent cron jobs, with failure state, bounded retries, context injection, and escalation.

## Best Possible Solution

Define an additive scheduler-owned assertion result with bounded execution authority, durable run-history projection, and explicit idempotent retry semantics; if core should not own verifier implementations, first add a fail-capable plugin post-run hook that can request canonical scheduler retry or escalation.

## Reproduction Assessment

Yes at the source-contract level: current main accepts an isolated agent runner’s clean terminal status without any deterministic outcome assertion. The reporter’s exact 18-of-50 production incident was not independently rerun, but the missing mechanism is directly reproducible from the current execution path.

## Solution Assessment

No, not exactly as proposed. Deterministic post-run assertions are a strong direction, but a raw shell string and automatic prompt injection are not yet the narrowest safe solution without an execution-security policy, ownership decision, and idempotent retry/state contract.

## Risks / Open Questions

- A persisted raw shell command would execute with scheduler process access and could expose filesystem data, environment variables, credentials, or other runtime authority unless it uses an explicit constrained execution policy.
- Retrying an agent with verifier output changes prompt and session state and can repeat non-idempotent work; continuation, deduplication, transcript ownership, and retry accounting need an explicit contract.
- Adding fields across stored cron jobs, gateway protocol, CLI, and UI is compatibility-sensitive and requires fresh-install plus upgrade behavior to be designed together.
