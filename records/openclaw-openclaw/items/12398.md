## Summary

Keep open. Current main still has no cron postcondition verifier or scheduler-owned equivalent, and the remaining work is a broad cron/API/security design item rather than conservative cleanup.

## What This Changes

The issue requests an opt-in cron `postconditions` contract that runs deterministic checks after agent-driven cron jobs and can retry or escalate when the outcome check fails.

## Best Possible Solution

Define a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that can request scheduler-owned retry and escalation while preserving explicit command-execution boundaries.

## Reproduction Assessment

No, not as a normal bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof that current main lacks a cron postcondition surface or equivalent scheduler-owned verifier.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
