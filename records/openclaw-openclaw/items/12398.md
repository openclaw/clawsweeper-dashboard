## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned verifier, and prior review discussion already routed this to maintainer design follow-up rather than cleanup closure.

## What This Changes

Add optional cron postcondition checks that deterministically verify isolated agent-turn job outcomes, then retry or escalate when verification fails.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable cron post-run plugin hook with scheduler-owned retry and escalation semantics.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Source inspection gives high-confidence absence of the requested postconditions surface across cron types, protocol, runtime execution, UI/CLI, docs, and plugin hooks.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the proposed shell-command shape needs maintainer decisions on core-vs-plugin ownership and host command execution boundaries.
