## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned deterministic verifier, and the existing discussion already routes this as broad cron runtime/protocol design work rather than cleanup closure.

## What This Changes

The issue requests an optional cron `postconditions` field that runs deterministic checks after isolated agent cron jobs and uses failures for retry or escalation.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection gives high confidence that current main lacks the requested field, protocol acceptance, verifier execution point, and retry/escalation state.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but the safest maintainable path needs maintainer design around core cron contract versus plugin seam and host command-execution boundaries.
