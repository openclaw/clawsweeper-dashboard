## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned verifier, and the live labels plus prior review comments route this to product and security review rather than conservative cleanup.

## What This Changes

The issue requests an optional cron `postconditions` contract that runs deterministic checks after isolated agent jobs and can trigger retry or escalation when an otherwise clean agent run did not achieve its goal.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that preserves scheduler-owned retry and escalation semantics with explicit command-execution security boundaries.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Source inspection gives high-confidence proof that the requested cron postcondition surface is absent from current main.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the proposed shell-command shape needs maintainer decisions on core-vs-plugin ownership and host command execution boundaries.
