## Summary

Keep open. Current main and the latest shipped release still do not provide a cron postcondition verifier or scheduler-owned equivalent, and the remaining work is a broad cron contract/security/product decision rather than a stale or duplicate cleanup item.

## What This Changes

Requests an optional `postconditions` cron-job contract that runs deterministic post-run checks, retries failed agent turns with verifier context, and escalates exhausted failures.

## Best Possible Solution

Define a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that can request scheduler-owned retry and escalation under explicit command-execution boundaries.

## Reproduction Assessment

No, not as a normal bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof that current main and v2026.5.22 lack a cron postcondition surface or equivalent scheduler-owned verifier.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
