## Summary

Keep this open: current main and v2026.6.5 still lack a cron postcondition contract or equivalent scheduler-owned verifier, and the remaining work needs maintainer product and security design rather than conservative cleanup.

## What This Changes

The issue requests an optional cron postconditions contract that runs deterministic post-run checks for isolated agent cron jobs and can retry or escalate on verifier failure.

## Best Possible Solution

Choose a security-reviewed cron post-run verification contract that can fail a run, persist verifier state, inject bounded retry context, and reuse failure-alert/escalation delivery at the narrowest maintainable core/plugin boundary.

## Reproduction Assessment

Yes, at source level: current cron types, protocol schemas, docs, hooks, and timer runtime do not expose or run a post-agent verifier. I did not run a live cron because this is a missing feature rather than a broken existing documented contract.

## Solution Assessment

Unclear as a final design: the proposed postconditions field is plausible, but maintainers still need to decide whether durable fail/retry/escalation belongs in core cron, a new plugin lifecycle seam, or a combined contract.
