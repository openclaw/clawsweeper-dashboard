## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned verifier, and the live discussion already routes this to design review rather than cleanup closure.

## What This Changes

The issue requests an optional cron `postconditions` field that runs deterministic post-run checks for isolated agent jobs and uses failures to retry or escalate.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or a fail-capable cron post-run plugin hook, with typed config/state/protocol, bounded verifier execution, durable logs/UI/CLI/docs, retry context injection, and failure-alert escalation.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Current-main source inspection provides high-confidence absence of the requested `postconditions` surface across types, protocol, CLI/UI, docs, runtime execution, and plugin hooks.

## Solution Assessment

Unclear. Deterministic post-run verification matches the reported outcome gap, but the proposed shell-command design is not ready without a core-vs-plugin decision and host command-execution boundaries.
