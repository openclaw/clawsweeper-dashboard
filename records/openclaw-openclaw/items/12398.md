## Label Justifications

- `P2`: The issue asks for a meaningful reliability improvement for scheduled agent work, but it is an additive feature and design decision rather than an urgent regression.
- `impact:session-state`: The issue concerns persisted cron run state treating a clean isolated session termination as success without verifying the intended task outcome.
- `impact:security`: The proposed verifier would execute configured commands and expose run/session context to them.

## Summary

Keep open: current main and the latest release still have no cron postcondition contract or equivalent scheduler-owned verifier. The request is valuable and vision-aligned, but it crosses cron runtime, protocol, CLI/UI/docs, retry state, and command-execution security boundaries, so it needs maintainer product/security design before implementation.

## What This Changes

Request adding optional deterministic post-run checks for isolated cron agent jobs, with retry/escalation and persisted verifier state.

## Best Possible Solution

Add a reviewed additive cron verifier contract with typed persisted results, bounded execution, retry/escalation semantics, and either a safe scheduler-owned runner or a fail-capable plugin adapter for optional verifiers.

## Reproduction Assessment

Yes at the source-contract level: current main can establish that isolated cron completion has no deterministic post-run verifier path. The specific 18-of-50 production incident was not independently replayed.

## Solution Assessment

No, not exactly as proposed: deterministic post-run verification is the right capability, but an unrestricted shell field is not yet the safest or narrowest maintainable contract. A typed, bounded verifier result plus a scheduler/plugin ownership decision is the safer path.
