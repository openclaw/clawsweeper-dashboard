## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier, and the existing discussion already frames this as broad cron/runtime design work rather than conservative cleanup closure.

## What This Changes

The issue requests optional cron postcondition checks that deterministically verify agent-run outcomes after scheduled jobs complete.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a missing feature. Source inspection gives high confidence that current main lacks the requested field, protocol acceptance, verifier execution point, and retry/escalation state.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but the safest maintainable path needs maintainer design around core contract versus plugin seam and host command-execution boundaries.
