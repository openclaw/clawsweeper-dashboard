## Label Justifications

- `P2`: This is a meaningful reliability improvement for scheduled agent work, but it is an additive feature/design decision rather than an urgent shipped-runtime regression.
- `impact:session-state`: The issue concerns cron run state treating clean isolated session termination as success without verifying the intended task outcome.
- `impact:security`: The proposed verifier would execute configured commands and expose run/session context to those commands.

## Summary

Keep open: current main and the latest release still lack a cron postcondition contract or equivalent scheduler-owned verifier for successful-but-incomplete isolated agent-turn runs, and the remaining work needs product/security review before implementation.

## What This Changes

The issue requests scheduler-owned cron postcondition checks that deterministically verify isolated agent-turn outcomes and drive retry or escalation when a cleanly terminated agent run under-completes its task.

## Best Possible Solution

Choose a maintainer-approved cron verifier boundary, then add a typed additive contract with bounded execution, persisted results, retry/escalation semantics, and either a safe scheduler-owned runner or a fail-capable plugin adapter.

## Reproduction Assessment

Yes at the source-contract level: current main shows isolated cron completion has no deterministic post-run verifier path and public schemas have no postcondition fields. The specific 18-of-50 production incident was not independently replayed.

## Solution Assessment

Unclear: deterministic post-run verification is the right capability, but an unrestricted shell field is not yet the safest or narrowest maintainable contract. Maintainers need to choose scheduler-owned core behavior versus a fail-capable plugin hook/adapter seam.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because the requested design introduces scheduler-owned command execution and run/session context exposure.

Concerns:
