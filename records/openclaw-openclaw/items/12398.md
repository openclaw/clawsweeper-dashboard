## Summary

Keep open. Current main still lacks the requested cron postcondition contract and scheduler-owned deterministic verifier, and the existing maintainer-side discussion already routes this as broad cron runtime/protocol/security design work rather than cleanup closure.

## What This Changes

The issue requests an optional cron job postconditions field that runs deterministic checks after isolated agent-turn cron jobs and drives retry or escalation from the check result.

## Best Possible Solution

Add a reviewed additive post-run verification design spanning typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable cron post-run hook.

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection on current main gives high confidence that the requested field, protocol acceptance, verifier execution point, and retry/escalation state are absent.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but the maintainable path needs maintainer design around core cron contract versus plugin seam and host command-execution boundaries.
