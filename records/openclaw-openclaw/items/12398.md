## Summary

Keep open. Current main and v2026.5.12 still lack a cron postcondition verifier or equivalent scheduler-owned retry/escalation path, and the requested shell-check design needs product and security review before implementation.

## What This Changes

The issue requests an opt-in cron postcondition contract that runs deterministic checks after isolated agent cron jobs and can retry or escalate when the check fails.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that preserves scheduler-owned retry and escalation semantics with explicit command-execution boundaries.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Source and release inspection give high-confidence proof that the requested cron postcondition surface is absent from current main and v2026.5.12.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the shell-command shape needs maintainer decisions on core-vs-plugin ownership and host command execution boundaries.
