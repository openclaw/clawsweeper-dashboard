## Summary

Keep open. Current main still lacks a cron postcondition contract or equivalent scheduler-owned verifier, and the prior substantive review comment already routed this to maintainer design follow-up rather than cleanup closure.

## What This Changes

Request adding optional cron postconditions that run deterministic post-run checks for isolated agent-turn cron jobs and use failures to retry or escalate.

## Best Possible Solution

Add a reviewed additive post-run verification design across cron config/state, gateway protocol, runtime execution, run logs, UI/CLI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable cron post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Source inspection on current main shows the proposed field, protocol acceptance, verifier execution point, retry context injection, and escalation state are absent.

## Solution Assessment

Unclear. Deterministic post-run verification matches the reported gap, but maintainers need to decide the core cron contract versus plugin seam and define host command-execution boundaries first.
