## Summary

Keep open. Current main still lacks a cron postcondition contract or equivalent scheduler-owned verifier, and the existing maintainer-side discussion already routes this as broad, security-sensitive cron design work rather than conservative cleanup closure.

## What This Changes

The issue requests an optional cron-job postcondition contract that runs deterministic post-run checks after isolated agent-turn cron jobs and can retry or escalate when verification fails.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery. If maintainers prefer plugin ownership, first expose a fail-capable post-run cron hook that can request scheduler-owned retry or escalation.

## Reproduction Assessment

Not applicable as a normal runtime bug reproduction; this is a missing feature. The absence is high-confidence from current-main searches plus inspection of cron types, closed gateway schemas, detached execution, state application, docs, UI/CLI surfaces, and plugin hooks.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but the safest maintainable path needs maintainer design around core contract versus plugin seam and host command-execution boundaries before implementation.
