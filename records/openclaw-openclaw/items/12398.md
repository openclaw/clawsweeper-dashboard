## Summary

Keep open: current main and v2026.6.8 still lack a scheduler-owned cron postcondition contract for successful-but-incomplete isolated agent-turn runs; command cron and `before_agent_finalize` are adjacent surfaces, but neither verifies a completed cron agent outcome and then drives scheduler retry/escalation.

## What This Changes

Request an optional cron post-run verifier that can deterministically check isolated agent-turn outcomes, persist postcondition state, and trigger retry or escalation.

## Best Possible Solution

Add a maintainer-approved additive cron verifier contract with typed persisted results, bounded execution, retry/escalation semantics, and either a safe scheduler-owned runner or a fail-capable plugin adapter for optional verifiers.

## Reproduction Assessment

Yes at the source-contract level: current main has no cron postcondition fields, public schema support, or verifier branch between isolated agent completion and state application. The specific 18-of-50 production incident was not replayed.

## Solution Assessment

No, not exactly as a raw `postconditions.check` shell string. Deterministic post-run verification is the right capability, but the maintainable path needs a typed, bounded verifier or plugin seam with explicit retry, idempotency, and security semantics.
