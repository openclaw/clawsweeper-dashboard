## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned deterministic verifier, and existing discussion already routes this as broad cron runtime/protocol design work with command-execution security implications.

## What This Changes

The issue requests an optional `postconditions` field on cron jobs to run deterministic post-run checks and drive retry or escalation for isolated agent cron jobs.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection on current main gives high confidence that the requested field, protocol acceptance, verifier execution point, and retry/escalation state are absent.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but the maintainable path needs maintainer design around core cron contract versus plugin seam and host command-execution boundaries.
