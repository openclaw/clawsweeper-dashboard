## Summary

Current main and v2026.6.5 still treat a clean agent termination as the cron execution outcome without a scheduler-owned goal verifier, and no canonical replacement issue or PR exists. The request is valuable but needs maintainer product and security decisions because the proposed raw shell contract spans public cron configuration, persisted session/run state, retry semantics, delivery, and privileged command execution.

## What This Changes

Add optional deterministic post-run checks for agent cron jobs that can fail an otherwise successful run, inject retry context, and escalate exhausted failures.

## Best Possible Solution

Define one additive scheduler-owned verification result and lifecycle that can convert an ok agent result into verified failure, reuse canonical retry and failure-delivery machinery, persist bounded audit data, and expose verifier implementations through a narrowly reviewed security boundary rather than making raw shell text the core semantic contract.

## Reproduction Assessment

Yes at the source-contract level: current main has a clear path where a clean agent termination becomes an ok cron result without deterministic goal verification. The specific 18-of-50 production incident was not independently replayed, but replay is not needed to establish the missing capability.

## Solution Assessment

No, not exactly as proposed. Deterministic verification is the right product direction, but the maintainable solution should establish a typed scheduler verification contract and security model first, then decide whether shell, plugin, or other verifier implementations sit behind it.

## Risks / Open Questions

- A raw shell-command field would execute operator-defined code with scheduler context, requiring explicit rules for permissions, environment exposure, working directory, timeout, output bounds, redaction, and escalation delivery.
- The feature changes public cron config and persisted run/session state across protocol, CLI, UI, retry, and delivery surfaces; implementing only the timer step would create inconsistent clients and competing outcome semantics.
- It is unresolved whether verifier implementations belong in core or plugins; adding both a built-in shell field and a hook without one canonical result contract would create long-term API drift.
