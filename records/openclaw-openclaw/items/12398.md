## Label Justifications

- `impact:session-state`: The gap lets durable cron state record success for an isolated agent run even when the intended external outcome may not have happened.

## Summary

Keep open. Current main and v2026.6.5 still lack a cron postcondition verifier or scheduler-owned equivalent, and the remaining work is a broad cron runtime/protocol/CLI/UI/security product decision rather than conservative cleanup.

## What This Changes

Request an optional cron job postconditions contract that runs deterministic checks after isolated agent turns and uses failures for retry or escalation.

## Best Possible Solution

Define a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that can request scheduler-owned retry and escalation under explicit command-execution boundaries.

## Reproduction Assessment

No, not as a normal bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof that current main and v2026.6.5 lack a cron postcondition surface or equivalent scheduler-owned verifier.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
