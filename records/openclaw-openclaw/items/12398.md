## Summary

Keep open. Current main and the latest release still lack the requested cron postcondition verifier surface, and the live labels plus prior review route this broad, security-sensitive feature to maintainer product/security design.

## What This Changes

Request to add optional cron job postconditions that run deterministic checks after isolated agent jobs and drive retry or escalation from the result.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that preserves scheduler-owned retry and escalation semantics with explicit command-execution security boundaries.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the shell-command shape needs maintainer decisions on core-vs-plugin ownership and host command execution boundaries.
