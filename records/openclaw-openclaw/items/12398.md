## Summary

Keep open. Current main still lacks the requested cron postcondition contract, public protocol/state surface, and scheduler-owned deterministic verifier/retry path; the existing comments already route this to maintainer design instead of cleanup closure.

## What This Changes

The issue requests an opt-in CronJob postconditions field that runs deterministic post-run checks after isolated agent cron jobs, with retry and escalation behavior.

## Best Possible Solution

Add a reviewed additive post-run verification design across cron config/state, gateway protocol, runtime execution, run logs, UI/CLI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable cron post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection on current main gives high confidence that the field, protocol acceptance, verifier execution point, and retry/escalation state are absent.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but maintainers need to decide the core cron contract versus plugin seam and define host command-execution boundaries first.
