- `P2`: This is a normal-priority reliability feature for autonomous cron workflows with broad but bounded cron/runtime impact.
- `impact:session-state`: The gap lets an isolated agent session terminate cleanly while durable cron state can record success for an outcome that may not have happened.

## Summary

Keep open. Current main and the latest release still lack a cron postcondition verifier or scheduler-owned equivalent, and the remaining work is a broad cron contract plus command-execution security design item rather than conservative cleanup.

## What This Changes

Issue requests deterministic postcondition checks for agent-driven cron jobs so clean session termination can be verified against a separate outcome contract.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first define a fail-capable plugin hook that can request scheduler-owned retry and escalation while preserving explicit command-execution boundaries.

## Reproduction Assessment

No, not as a normal bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof that current main lacks a cron postcondition surface or equivalent scheduler-owned verifier.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
