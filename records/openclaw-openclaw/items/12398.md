## Summary

Current main still treats an isolated cron run as successful when the agent runner returns `ok`; diagnostics, failure alerts, and the `cron_changed` hook improve observation but cannot deterministically reject a semantically incomplete result or request scheduler-owned retry. The request remains useful, but the exact shell-command design needs maintainer product and security decisions because it adds public config/state/protocol surfaces and executes operator-supplied commands with run context.

## What This Changes

Add optional deterministic post-run checks for isolated agent cron jobs, including persisted results, retry context, and failure escalation.

## Best Possible Solution

Define an additive scheduler-owned verification result contract first, with a narrow fail-capable plugin hook or trusted command runner that can convert an `ok` run into a verified failure before state application; then integrate bounded retries, idempotent attempt metadata, existing failure destinations, run logs, protocol, CLI/UI, and public docs.

## Reproduction Assessment

Yes at the source level: current main deterministically records the runner’s clean `ok` result without any semantic outcome verifier. The reporter’s exact early-stop incident was not recreated, but the missing framework capability is directly observable in the current contracts and state path.

## Solution Assessment

No, not yet as specified: scheduler-owned deterministic verification is the right direction, but a raw per-job shell string plus automatic full-turn retries is not sufficiently bounded for security, idempotency, and plugin/core ownership. A reviewed fail-capable verifier contract should precede the concrete command and retry syntax.

## Risks / Open Questions

- A direct `check: string` contract would execute operator-controlled shell commands after unattended jobs and expose run metadata through environment variables, requiring explicit approval, timeout, output-bound, working-directory, secret-redaction, and sandbox semantics.
- Adding postcondition config, persisted state, protocol fields, CLI/UI projections, and retry behavior creates a compatibility-sensitive public cron contract whose ownership and migration rules need agreement.
- Retrying the full agent turn with injected verifier output can duplicate external side effects unless the contract defines idempotency, attempt identity, and how delivery/session state is associated across attempts.
