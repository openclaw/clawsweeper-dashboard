## Label Justifications

- `P2`: This is a normal-priority reliability feature for autonomous cron workflows with a broad but bounded cron/runtime impact.
- `impact:session-state`: The reported gap lets an isolated agent session terminate cleanly while the durable cron state records success for an outcome that may not have happened.
- `impact:security`: The proposed solution would run configured host commands and expose verifier output, so it touches command-execution and sensitive-data boundaries.

## Summary

Keep open. Current main still has no cron postcondition verifier or scheduler-owned equivalent for turning a clean-but-incomplete isolated agent run into retry/escalation, and the proposed shell-check shape needs product and security review.

## What This Changes

The issue requests an optional `postconditions` cron job contract that runs deterministic checks after isolated agent turns and uses failures for retry or escalation.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that preserves scheduler-owned retry and escalation semantics with explicit command-execution boundaries.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/API request. Source and release inspection give high-confidence proof that current main and v2026.5.12 lack the requested cron postcondition surface.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
