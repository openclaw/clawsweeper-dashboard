## Summary

Current main still treats clean isolated-agent termination as a successful cron execution without a job-specific deterministic outcome check. The capability is valuable and vision-aligned, but its ownership, security boundary, retry semantics, and persistent contract require maintainer product review before implementation.

## What This Changes

Add scheduler-enforced deterministic post-run verification for isolated agent cron jobs, with verifier failures driving retry context and escalation.

## Best Possible Solution

Add an additive typed scheduler verification result with durable attempt state and bounded execution, using a fail-capable plugin adapter when optional verifiers can own checks without exposing unrestricted shell execution through cron configuration.

## Reproduction Assessment

Yes at the source-contract level: current main can establish that an isolated agent turn ended cleanly, but it cannot deterministically verify an arbitrary job-specific outcome. The reported 18-of-50 production incident itself was not independently replayed.

## Solution Assessment

No, not exactly as proposed: deterministic post-run verification is an appropriate capability, but an unrestricted shell field is not yet the safest or narrowest maintainable contract.
