## Summary

Keep open. Current main still lacks the requested cron postcondition contract or equivalent scheduler-owned verifier, and prior issue discussion explicitly kept this for maintainer design follow-up rather than cleanup closure.

## What This Changes

Request adding an opt-in CronJob postconditions field that runs deterministic post-run checks and can drive retry or escalation for isolated agent cron jobs.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that keeps scheduler-owned retry and escalation semantics in core.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Source inspection gives high-confidence absence of the requested postconditions surface across cron types, protocol, runtime execution, docs, UI-adjacent forms, and plugin hooks.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported outcome gap, but the proposed shell-command shape needs maintainer decisions on core-vs-plugin ownership and host command execution boundaries.
