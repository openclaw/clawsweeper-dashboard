## Summary

Keep open. Current main and the latest shipped release still lack a cron postcondition verifier or scheduler-owned equivalent, and the remaining work needs cron contract, command-execution security, and core-versus-plugin product decisions.

## What This Changes

The issue requests an optional cron `postconditions` contract that runs deterministic checks after isolated agent cron jobs and drives retry or escalation from the result.

## Best Possible Solution

Define a reviewed additive cron post-run verification contract, or first add a fail-capable plugin hook that can request scheduler-owned retry and escalation under explicit command-execution boundaries.

## Reproduction Assessment

No, not as a normal bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof that current main and v2026.5.20 lack a cron postcondition surface or equivalent scheduler-owned verifier.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
