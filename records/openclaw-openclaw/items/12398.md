## Summary

Keep open. Current main and v2026.5.20 still lack a cron postcondition verifier or scheduler-owned equivalent, and the request remains a broad cron contract plus host-command security design item rather than conservative cleanup.

## What This Changes

The issue requests an optional cron job postcondition contract that runs deterministic verification after isolated agent-turn cron executions and can retry or escalate on failure.

## Best Possible Solution

Add a reviewed additive cron post-run verification contract, or first define a fail-capable plugin hook that can request scheduler-owned retry and escalation while preserving explicit command-execution boundaries.

## Reproduction Assessment

No, not as a normal bug reproduction; this is a feature/API request. Source inspection gives high-confidence proof that current main lacks a cron postcondition surface or equivalent scheduler-owned verifier.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the outcome gap, but the exact core field versus plugin hook design and shell-command security boundary need maintainer decisions first.
