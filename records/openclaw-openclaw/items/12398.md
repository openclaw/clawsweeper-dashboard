- `impact:session-state`: The issue concerns cron run state treating clean isolated session termination as success without verifying the intended task outcome.

## Summary

Keep open: current main and the latest release still lack a cron postcondition contract or equivalent scheduler-owned verifier for successful-but-incomplete isolated agent-turn runs. Command cron and cron_changed hooks are useful adjacent surfaces, but they do not verify an agent outcome after the run or drive semantic retry/escalation.

## What This Changes

The issue requests an optional cron postcondition contract that deterministically verifies isolated agent-turn cron outcomes and drives retry or escalation when the verifier fails.

## Best Possible Solution

Add a reviewed additive cron verifier contract with typed persisted results, bounded execution, retry/escalation semantics, and either a safe scheduler-owned runner or a fail-capable plugin adapter for optional verifiers.

## Reproduction Assessment

Yes at the source-contract level: current main shows isolated cron completion has no deterministic post-run verifier path and public schemas have no postcondition fields. The specific 18-of-50 production incident was not independently replayed.

## Solution Assessment

No, not exactly as proposed: deterministic post-run verification is the right capability, but an unrestricted shell field is not yet the safest or narrowest maintainable contract. A typed, bounded verifier result plus a scheduler/plugin ownership decision is the safer path.
