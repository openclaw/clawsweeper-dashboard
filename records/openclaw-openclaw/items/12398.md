## Summary

Keep open: current main and v2026.6.8 still lack a scheduler-owned cron postcondition contract for successful-but-incomplete isolated agent-turn runs; existing command cron and plugin hooks are adjacent but do not verify a completed cron agent outcome and drive scheduler retry/escalation.

## What This Changes

The issue requests an optional cron-job postcondition contract that deterministically verifies isolated agent-turn outcomes after completion and can drive retry or escalation.

## Best Possible Solution

Add a maintainer-approved additive cron verifier contract with typed persisted results, bounded execution, retry/escalation semantics, and either a safe scheduler-owned runner or a fail-capable plugin adapter for optional verifiers.

## Reproduction Assessment

Yes at the source-contract level: current main has no cron postcondition fields, public schema support, or verifier branch between isolated agent completion and state application. The specific 18-of-50 production incident was not replayed.

## Solution Assessment

No, not exactly as a raw shell-string field. Deterministic post-run verification is the right capability, but the maintainable path needs a typed, bounded verifier or plugin seam with explicit retry, idempotency, and security semantics.

Status: needs_attention

Summary: The requested design is security-sensitive because it would add scheduler-owned command execution with run/session context and semantic retry behavior.

  - body: Retrying after semantic failure can repeat external side effects unless attempt identity and idempotency expectations are part of the cron verifier contract.
