## Summary

Keep open. Current main still lacks a cron postcondition contract, protocol/state fields, scheduler-owned verifier step, retry context injection, and escalation behavior; recent comments already route this to maintainer design follow-up rather than cleanup closure.

## What This Changes

The issue requests an optional cron `postconditions` field that runs a deterministic verifier after isolated agent cron jobs and drives retry or escalation from its result.

## Best Possible Solution

Add a reviewed additive post-run verification design across cron config/state, gateway protocol, runtime execution, run logs, UI/CLI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable cron post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection on current main gives high confidence that the field, protocol acceptance, verifier execution point, retry context injection, and escalation state are absent.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but maintainers need to choose the core cron contract versus plugin seam and define host command-execution boundaries first.
