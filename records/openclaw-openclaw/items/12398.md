## Summary

Keep this open: current main and v2026.6.5 still lack a cron postcondition contract or equivalent scheduler-owned verifier, and the remaining work is a broad product/security design rather than conservative cleanup.

## What This Changes

The issue requests an optional cron `postconditions` field that runs deterministic checks after isolated agent cron jobs and can retry or escalate when the check fails.

## Best Possible Solution

Choose a security-reviewed cron post-run verification contract that can fail a run, persist verifier state, inject bounded retry context, and reuse existing failure-alert/escalation delivery at the narrowest maintainable core/plugin boundary.

## Reproduction Assessment

Yes, at source level: current cron types, protocol schemas, UI form state, docs, and timer runtime do not expose or run a post-agent verifier, and isolated cron returns the agent result directly. I did not run a live cron because this is a missing feature rather than a broken existing contract.

## Solution Assessment

No final solution is selected yet. The proposed postconditions field is plausible, but maintainers still need to decide whether the durable fail/retry/escalation contract belongs in core cron or a new plugin lifecycle seam.
