## Label Justifications

- `P2`: Silent under-completion of scheduled agent work is a meaningful reliability problem, but resolving it is a normal-priority cross-surface feature rather than an emergency regression.
- `impact:session-state`: The requested verifier would reinterpret terminal isolated-session outcomes and inject failure context into subsequent agent runs.
- `impact:security`: The proposed design executes stored local commands and passes session- and agent-derived data across command, prompt, and notification trust boundaries.

## Summary

Keep open. Current main and v2026.6.5 still treat a cleanly completed agent run as the cron execution outcome without a scheduler-owned assertion mechanism; deterministic verification is a valid reliability gap, but the proposed shell execution, retry safety, public contract, and core-versus-plugin boundary require maintainer product and security decisions.

## What This Changes

Add deterministic post-run verification for isolated agent cron jobs so unsuccessful outcomes can trigger scheduler-owned retries and escalation despite clean agent termination.

## Best Possible Solution

Define one additive, closed scheduler assertion result that a trusted bounded verifier can return after agent execution, integrate assertion failure into the existing canonical run-log, retry, and failure-delivery path, and require explicit retry-safety plus output-security semantics; use a generic plugin hook if it can preserve scheduler ownership without duplicating the cron state machine.

## Reproduction Assessment

Yes for the reported product gap: current source and v2026.6.5 have no scheduler-owned assertion mechanism, so a cleanly terminated but incomplete agent run can remain `ok`. The reporter's exact 18-of-50 production workload was not independently replayed.

## Solution Assessment

No, not exactly as proposed. Deterministic post-run verification is an appropriate capability, but an unconstrained shell field with a parallel retry model is not yet the narrowest maintainable solution; it should use a generic typed assertion contract integrated with existing scheduler outcomes and security controls.

## Risks / Open Questions

- Automatically rerunning an agent after a verifier failure can duplicate emails, writes, purchases, deployments, or other irreversible side effects unless retry safety and continuation semantics are explicit.
- Persisted shell commands create a privileged execution surface involving command injection, executable/path resolution, environment leakage, timeouts, output limits, and secret exposure.
- Verifier output may contain untrusted or sensitive data; injecting it into later prompts or escalation messages requires bounded encoding, redaction, and external-content treatment.
- Adding cron configuration, persisted state, protocol fields, CLI/UI controls, and defaults is a compatibility-sensitive public surface that needs fresh-install and upgrade behavior defined together.
- A core-specific checker implementation could conflict with the repository's lean-core and plugin-first direction if a generic fail-capable scheduler hook can provide the capability.
