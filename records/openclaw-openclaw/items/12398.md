## Summary

Keep open. Current main still lacks a cron postcondition contract or scheduler-owned deterministic verifier, and the prior discussion already keeps this as cron runtime/protocol design work rather than cleanup closure.

## What This Changes

Add optional cron job postconditions that run deterministic checks after isolated agent runs and drive retry or escalation from the check result.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery; if plugin-owned, first expose a fail-capable post-run hook.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a missing feature request. Source inspection gives high confidence that current main lacks the requested field, protocol acceptance, verifier execution point, and retry/escalation state.

## Solution Assessment

Unclear. Deterministic post-run verification addresses the reported gap, but the narrow maintainable path needs maintainer design around core cron contract versus plugin seam and host command-execution boundaries.
