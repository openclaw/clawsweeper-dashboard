## Summary

Keep open. Current main has strong no-clobber mitigations for the destructive write paths, but the remaining root unknown-key and bundled-command allowlist behavior is still an intentional strict config/plugin policy that needs maintainer judgment before code changes.

## What This Changes

The issue requests preserving unknown top-level config fields across config round-trips and changing how bundled plugin command surfaces interact with restrictive `plugins.allow`.

## Best Possible Solution

Keep the shipped no-clobber protections, then make one explicit maintainer decision on opaque root-key preservation and bundled-command allowlist semantics with matching schema, docs, doctor behavior, and regression tests.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main still rejects unknown root keys through the strict root schema and still blocks bundled command aliases when the owning plugin is excluded by `plugins.allow`. I did not reproduce the original Windows global-npm byte-drop sequence.

## Solution Assessment

Unclear. A one-line root `.passthrough()` change is not the best standalone fix because current docs and tests intentionally fail closed; any change should settle the config/security policy and update doctor, docs, schema consumers, and regression coverage together.
