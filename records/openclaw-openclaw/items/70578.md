## Summary

Keep open. Current main has strong no-clobber mitigations for the destructive write paths, but preserving opaque root config keys and changing bundled command allowlist behavior remain config/security policy decisions rather than a narrow cleanup close.

## What This Changes

The issue asks OpenClaw to stop config round trips from losing schema-unknown top-level data and to clarify bundled command behavior under restrictive `plugins.allow`.

## Best Possible Solution

Keep the shipped no-clobber protections, then make one explicit maintainer decision on opaque root-key preservation and bundled-command allowlist semantics with schema, docs, doctor behavior, and regression tests aligned.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main still rejects unknown root keys through the strict root schema and still blocks bundled command aliases when the owning plugin is excluded by `plugins.allow`. I did not reproduce the original Windows global-npm byte-drop sequence.

## Solution Assessment

Unclear. A one-line root `.passthrough()` change is not the best standalone fix because current docs and tests intentionally fail closed; any change should settle the config/security policy and update doctor, docs, schema consumers, and regression coverage together.
