## Summary

Keep open. Current main has mitigations for the destructive write paths, but it still intentionally rejects unknown root config keys and still applies `plugins.allow` to bundled command surfaces, so the remaining work needs a maintainer config-contract and plugin-policy decision.

## What This Changes

The issue asks OpenClaw to preserve unknown top-level config data during round trips and to avoid bundled command lockouts caused by restrictive `plugins.allow` after config loss.

## Best Possible Solution

Keep the no-clobber protections, then make an explicit maintainer decision on opaque root-key preservation and bundled command allowlist semantics with matching schema, doctor, docs, and regression coverage.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main still rejects unknown root keys through the strict root schema and still blocks bundled command aliases when the owning plugin is excluded by `plugins.allow`. I did not reproduce the original Windows global-npm byte-drop sequence.

## Solution Assessment

Unclear. A one-line root `.passthrough()` change is not the best standalone fix because current docs and tests intentionally fail closed; any change should settle the config/security policy and update doctor, docs, schema consumers, and regression coverage together.
