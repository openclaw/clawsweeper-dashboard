## Summary

Keep open. Current main has mitigated the silent clobber mechanisms with source-preserving write projection, destructive-write rejection, and plugin-install allowlist updates, but the remaining root-schema passthrough request and bundled plugin CLI allowlist behavior are still unresolved maintainer policy decisions. Related issue #75091 was already closed as duplicate/superseded in favor of this canonical tracker.

## What This Changes

The issue asks OpenClaw to stop losing or blocking unknown top-level config data during config/plugin round-trips and to resolve bundled plugin CLI behavior under restrictive `plugins.allow` settings.

## Best Possible Solution

Maintainers should explicitly choose the config contract: either keep strict root validation with clear fail-closed diagnostics and regression tests for config writes, plugin installs, and gateway startup, or preserve unknown top-level keys as opaque data with matching docs, tests, and doctor/migration behavior. The bundled command allowlist question should be resolved separately as a narrow plugin-security policy change.

## Reproduction Assessment

Yes. For the remaining behavior, current source and tests give a focused reproduction: a config with `customUnknownField` is expected to fail strict validation, while the earlier projection and destructive-write paths now preserve or reject before commit. The exact historical Windows byte-drop sequence was not rerun in this read-only review.

## Solution Assessment

Unclear. The suggested root `.passthrough()` change would solve opaque preservation, but it is not clearly the best fix until maintainers choose between typo-detection strictness and legacy-data passthrough; the bundled command exemption is also a separate security-policy decision.
