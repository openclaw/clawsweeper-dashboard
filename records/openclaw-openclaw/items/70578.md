## Summary

Keep open. Current main has mitigated the silent-clobber paths with source-preserving config writes, validation-before-write, destructive-write rejection, and plugin-install allowlist updates, but the remaining root-schema passthrough and bundled command allowlist requests are explicit config-contract and plugin-security policy decisions.

## What This Changes

The issue asks OpenClaw to stop losing top-level config data during config writes by preserving unknown root fields, preserving source-owned write state, and reconsidering bundled command allowlist behavior.

## Best Possible Solution

Keep fail-closed no-clobber behavior unless maintainers explicitly choose an opaque unknown-root-key preservation contract with matching docs, doctor/schema/test updates, and a separate bundled-command allowlist policy.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main rejects unknown root keys through OpenClawSchema.safeParse, and command aliases such as wiki remain blocked unless the parent plugin is in plugins.allow. I did not rerun the original Windows global-npm byte-drop sequence.

## Solution Assessment

Unclear. The one-line .passthrough() direction is not clearly the best fix because current docs and tests encode strict fail-closed validation, and the bundled command behavior is a separate plugin allowlist policy decision.
