## Summary

Current main mitigates the destructive config-write paths, but the issue still contains unresolved config-contract and plugin-allowlist policy questions. Root unknown-key passthrough and bundled command alias exemptions are not implemented, and both would change intentional behavior currently encoded in tests.

## What This Changes

The issue asks OpenClaw to prevent config round-trip data loss for unknown top-level fields and reconsider bundled command aliases being blocked by restrictive plugins.allow.

## Best Possible Solution

Keep the no-clobber write protections already on main, then make an explicit maintainer decision on opaque root-key preservation and bundled-command allowlist policy with matching docs, schema, doctor behavior, and regression tests.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main still rejects unknown root keys through OpenClawSchema.safeParse, and bundled command aliases remain blocked unless the owning plugin is included in plugins.allow. I did not reproduce the original Windows global-npm byte-drop sequence.

## Solution Assessment

Unclear. The requested one-line .passthrough() change is not clearly the best solution because current tests encode strict root validation, and the bundled command exemption is a separate plugin-security policy choice.
