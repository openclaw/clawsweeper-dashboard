## Summary

Keep open. Current main mitigates the concrete silent-clobber paths with source-preserving writes, validation-before-write, destructive-write rejection, and plugin-install allowlist updates, but the remaining root-schema passthrough and bundled command allowlist behavior are still config-contract and plugin-security policy decisions.

## What This Changes

The issue requests preventing OpenClaw config writes from silently dropping unknown top-level fields and reconsidering restrictive `plugins.allow` handling for bundled CLI command surfaces.

## Best Possible Solution

Maintain an explicit config contract: either keep strict root validation with fail-closed no-clobber diagnostics, or design opaque unknown-root-key preservation with matching docs, doctor behavior, schema output, tests, and a separate bundled-command allowlist policy.

## Reproduction Assessment

Yes for the remaining source-level behavior: current main rejects unknown root keys via `OpenClawSchema.safeParse`, and command aliases such as `wiki` are blocked unless the parent plugin is in `plugins.allow`. I did not rerun the original Windows global-npm byte-drop sequence.

## Solution Assessment

Unclear. The closed one-line `.passthrough()` PR is not clearly the best solution because it changes documented fail-closed validation; maintainers need to choose the config contract and handle bundled command allowlisting as a separate policy decision.
