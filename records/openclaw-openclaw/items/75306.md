## Summary

Keep open: the bug is valid, but this PR is incomplete against current main because manager-level ACP runtime-control validation can reject `timeout` before the new ACPX runtime filter is reached.

## What This Changes

The PR hoists ACPX `timeout` and `timeout_seconds` filtering out of the Codex-only branch, updates the Claude ACP unit assertion, and adds a changelog entry.

## Best Possible Solution

Suppress manager-generated `timeout` and `timeout_seconds` session-config replay for ACP backends that do not advertise those keys, preserve the manager watchdog, keep non-timeout controls flowing, and retain ACPX runtime guard coverage as defense in depth.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main synthesizes a `timeout` config pair, manager validation can reject it before the ACPX runtime sees it, and the pinned Claude ACP dependency rejects unknown config ids while not advertising `timeout`.

## Solution Assessment

No. Hoisting the skip into `AcpxRuntime.setConfigOption` is directionally useful, but current main has an earlier manager/control-plane validation path, so the narrow maintainable fix must also stop or tolerate the manager-level timeout config pair for unsupported backends.
