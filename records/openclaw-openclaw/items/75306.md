## Summary

Keep open: the reported bug is real, but this PR still fixes the wrong layer for current main because the manager can reject the unsupported timeout control before the acpx wrapper guard is reached.

## What This Changes

The PR moves the acpx `timeout`/`timeout_seconds` skip out of the Codex-specific branch and updates the runtime test to assert Claude timeout controls are not forwarded.

## Best Possible Solution

Handle auto-injected timeout controls in the manager runtime-options/application path by skipping or fail-softing unsupported optional timeout keys while keeping explicit user config-option writes strict, then add manager coverage and live current-main Claude ACP proof.

## Reproduction Assessment

Yes. Current source still emits `timeout` from runtime options and can reject it before the acpx wrapper when Claude advertises config keys without `timeout`; multiple live reports reproduce the same Claude ACP `sessions_spawn` failure.

## Solution Assessment

No. The PR handles direct acpx wrapper forwarding, but current main can fail earlier in `applyManagerRuntimeControls`, so the narrow maintainable fix belongs at the manager runtime-control boundary.
