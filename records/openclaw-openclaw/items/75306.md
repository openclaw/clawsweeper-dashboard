## Summary

Keep open. Current main still forwards `timeout` to Claude through ACPX and the pinned/upstream Claude ACP adapter rejects that config key; the proposed patch is focused and I found no blocking code-review finding, but the external-PR proof gate is still not fully satisfied and the PR check rollup has unresolved failures.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` filtering so all backends ignore those session config keys, updates Claude ACP unit coverage, and adds a changelog entry.

## Best Possible Solution

Land the targeted ACPX timeout filter with regression coverage and changelog after redacted live after-fix proof and current checks are green; leave broader ACP config-key negotiation to the separate capability-aware follow-up work.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main maps `timeoutSeconds` to a `timeout` config option and forwards it for non-Codex ACPX, while Claude ACP 0.32.0 and 0.33.1 reject unknown config IDs. I did not run a live Claude session locally.

## Solution Assessment

Yes for the timeout-specific regression: filtering unsupported timeout keys in the ACPX wrapper is a narrow owner-boundary fix and preserves the manager-owned turn watchdog. Broader capability-aware filtering for other unsupported keys should remain separate from this small bug fix.
