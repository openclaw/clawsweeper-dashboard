## Summary

Keep this PR open: the Claude ACP timeout bug is real, but the current patch is downstream of current main's manager-side validation and also suppresses timeout controls for ACPX backends that explicitly advertise them.

## What This Changes

This PR hoists the ACPX `timeout`/`timeout_seconds` no-op out of Codex-specific handling and updates the Claude ACP runtime test to expect those config controls are not forwarded.

## Best Possible Solution

Filter unsupported timeout controls in the manager/runtime-option application path using advertised config keys, while continuing to forward timeout aliases for ACP backends that explicitly advertise them.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main applies runtime config controls before the turn, builds `timeout` from `timeoutSeconds`, and rejects it when Claude's advertised config options omit that key. The PR discussion adds live logs showing the same unsupported-timeout failure in shipped builds.

## Solution Assessment

No. Skipping inside `AcpxRuntime.setConfigOption` is too late for current main's manager validation, and making it unconditional is too broad for ACP backends that advertise timeout support.
