## Summary

Keep open: the reported Claude ACP timeout bug is valid, but this patch skips the timeout key too late in the current path. On current main, manager runtime-control application can reject the auto-injected unsupported timeout before `AcpxRuntime.setConfigOption` is reached.

## What This Changes

This PR changes the acpx runtime wrapper to no-op `timeout` and `timeout_seconds` config-option writes for ACP backends and updates the Claude runtime unit test to expect no forwarding.

## Best Possible Solution

Handle auto-injected timeout controls at the manager runtime-options/application boundary by not emitting or fail-softing unsupported timeout keys for backends that do not advertise them, while keeping explicit user config-option writes strict.

## Reproduction Assessment

Yes. Current main still emits `timeout` from runtime options and rejects it when Claude's advertised config keys omit that option, and multiple commenters reproduced the same failure on 2026.5.7 and 2026.5.8 with claude-agent-acp 0.32.x and 0.33.1.

## Solution Assessment

No. The PR handles direct acpx `setConfigOption` calls, but the reported `sessions_spawn` path can fail earlier in `applyManagerRuntimeControls`; the fix should prevent the manager from emitting or rejecting unsupported auto-injected timeout controls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move the timeout skip before manager control rejection:** `extensions/acpx/src/runtime.ts:899-901`
  - body: On current main, Claude sessions expose advertised config keys from `session/new`, and `applyManagerRuntimeControls` rejects an auto-injected `timeout` that is not advertised before it calls `runtime.setConfigOption`. This guard in `AcpxRuntime.setConfigOption` is therefore not reached for the reported `sessions_spawn` failure; move the skip/fail-soft behavior to the manager runtime option construction/application path.
