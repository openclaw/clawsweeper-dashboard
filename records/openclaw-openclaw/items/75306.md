## Summary

Keep open: the PR addresses the right Claude ACP timeout symptom for its older base, and later comments add useful live confirmation, but current main can now reject the unadvertised `timeout` config key in the manager before the ACPX runtime filter runs. The branch needs a current-main core-path fix or consolidation with the broader ACP timeout-control PRs before it is mergeable.

## What This Changes

Hoists ACPX `timeout`/`timeout_seconds` filtering out of the Codex-only branch, updates Claude ACP unit coverage, and adds a changelog entry.

## Best Possible Solution

Fix the core pre-turn runtime-control path so manager-owned `timeoutSeconds` is not sent to backends that do not advertise `timeout`, then keep any ACPX wrapper filter only as defense-in-depth with current-main Claude ACP proof.

## Reproduction Assessment

Yes. Source inspection shows current main still builds a `timeout` runtime config pair and can reject it against Claude ACP's advertised `mode`/`model`/`effort` keys before the prompt, and the PR discussion includes live reports of the same timeout rejection on recent OpenClaw/acpx/claude-agent-acp versions.

## Solution Assessment

No, not by itself. The ACPX runtime skip was a narrow fix for the older delegate-forwarding path, but current main's manager-level advertised-key guard can fail before `AcpxRuntime.setConfigOption` runs; the safer fix belongs in `runtime-options`/`manager.runtime-controls`, with ACPX filtering as optional defense-in-depth.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Fix the pre-turn timeout injection path:** `extensions/acpx/src/runtime.ts:531-538`
  - body: This skip sits inside `AcpxRuntime.setConfigOption`, but current main now rejects unadvertised config keys in `applyManagerRuntimeControls` before it calls the runtime delegate.   - confidence: 0.88
