## Summary

Keep open: the bug is real, but this adapter-only change is now too late in the current control path because main rejects unsupported advertised config keys before `AcpxRuntime.setConfigOption` runs. The PR also still needs stronger after-fix real behavior proof for the current head.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` filtering out of the Codex-only branch so the acpx wrapper ignores those keys for every backend and flips the Claude runtime unit test.

## Best Possible Solution

Move the optional timeout no-op or fail-soft handling into manager runtime-control pair building/application, keep the manager-owned turn watchdog, and cover it with a manager-level regression plus live Claude ACP proof.

## Reproduction Assessment

Yes, with high confidence from source and related live reports. Current main still synthesizes a `timeout` config pair and rejects it when Claude advertises only `mode`, `model`, and `effort`, before prompt execution.

## Solution Assessment

No. The adapter-level skip is too late for current main's manager-side advertised-key validation; the fix should happen where OpenClaw auto-injects or applies runtime controls.
