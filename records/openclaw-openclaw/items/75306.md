## Summary

Keep open: the reported Claude ACP timeout failure is real, but this PR’s ACPX-layer skip is no longer sufficient against current main. Current main can reject the auto-built `timeout` runtime control in the ACP manager before `AcpxRuntime.setConfigOption` is reached, so the fix needs to move into the ACP runtime-control path or be superseded by a canonical core fix.

## What This Changes

The PR hoists ACPX `timeout` and `timeout_seconds` filtering out of the Codex-only branch so those config keys are ignored for all ACPX backends, updates Claude ACP unit coverage, and adds a changelog entry.

## Best Possible Solution

Handle unsupported auto-injected timeout controls in the ACP control-plane/runtime-options layer, keep the manager-owned turn watchdog, and land one canonical core fix with regression coverage plus live Claude ACP proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main builds a `timeout` config pair and can validate it against Claude ACP’s advertised keys before ACPX runtime filtering runs. I did not perform a live Claude ACP spawn in this read-only review.

## Solution Assessment

No. The ACPX-layer skip was a plausible earlier patch, but current main can fail in `applyManagerRuntimeControls` before `AcpxRuntime.setConfigOption` is called, so the narrow maintainable fix belongs in runtime-option application or unsupported-control handling.
