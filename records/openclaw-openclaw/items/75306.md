## Summary

Keep open: this is a real Claude ACP timeout bug, but the current PR does not fix the current-main failure path because manager runtime-control validation can reject the injected timeout before the ACPX runtime wrapper guard is reached.

## What This Changes

The PR skips `timeout` and `timeout_seconds` forwarding in `extensions/acpx` for all ACP backends and changes the Claude ACP runtime test to expect no forwarding.

## Best Possible Solution

Land a narrow manager-level fix that skips or fail-softs unsupported auto-injected timeout controls based on advertised config keys while keeping explicit user config-option writes strict, with manager tests and live Claude ACP proof.

## Reproduction Assessment

Yes. The report and related comments give live Claude ACP failures, and current main still emits `timeout` then rejects it when Claude advertises config keys without that option; I did not run a live Claude ACP session locally in this read-only review.

## Solution Assessment

No. The PR's ACPX wrapper guard is too late for the current `applyManagerRuntimeControls` path, which can throw before `runtime.setConfigOption` is called; the fix belongs in manager runtime-option construction/application or classified fail-soft handling.

## Review Findings

Overall correctness: patch is incorrect
