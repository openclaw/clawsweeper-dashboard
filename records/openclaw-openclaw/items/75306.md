## Summary

Keep open: this PR is aimed at a real Claude ACP timeout failure, but the patch currently suppresses timeout controls for every ACPX backend and the after-fix proof still does not show a completed Claude ACP first response on the latest head.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` filtering ahead of backend-specific handling and updates the Claude ACP runtime test to expect those controls to be ignored.

## Best Possible Solution

Use a Claude-specific or capability-aware timeout skip that preserves advertised config option forwarding for other ACPX agents, with targeted runtime coverage and live proof of a completed Claude ACP spawn.

## Reproduction Assessment

Yes, from source and related live reports rather than a local live run: current main builds and forwards `timeout` for Claude ACP, while `claude-agent-acp` rejects unknown config ids and does not advertise a timeout option.

## Solution Assessment

No. Filtering unsupported timeout controls is the right direction, but this patch applies it before checking the ACP backend and can silently drop timeout controls for agents that advertise them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope timeout filtering to unsupported agents:** `extensions/acpx/src/runtime.ts:898-900`
  - body: This early return drops `timeout` and `timeout_seconds` for every ACPX backend. The manager can map runtime timeouts to an agent-advertised timeout alias, and `acpx` exposes those advertised `configOptions`, so any backend that actually supports this control will now silently stop receiving it; keep the skip scoped to Claude/Codex or make it capability-aware.
