## Summary

Keep open: this is a real Claude ACP bug, but the PR still fixes the wrong layer. Current main can reject the auto-injected timeout control in the manager before the ACPX wrapper guard is reached.

## What This Changes

The PR hoists the ACPX `timeout`/`timeout_seconds` no-op from Codex-only handling to all ACPX backends and updates the ACPX runtime unit test for Claude.

## Best Possible Solution

Handle unsupported auto-injected timeout controls in the manager runtime-options/application path, preserve strict behavior for explicit user config writes, and prove it with manager coverage plus live Claude ACP proof.

## Reproduction Assessment

Yes. Current main emits `timeout` from runtime options and can reject it when Claude's advertised config keys omit that option; PR and issue comments reproduce the same failure on 2026.5.7 and 2026.5.8 with claude-agent-acp 0.32.x and 0.33.1.

## Solution Assessment

No. The PR handles direct ACPX wrapper calls, but the reported `sessions_spawn` path can fail earlier in manager runtime-control application; the fix should skip or fail-soft unsupported auto-injected timeout controls before that strict rejection.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Move the timeout skip before manager control rejection:** `extensions/acpx/src/runtime.ts:899-901`

Move the timeout skip or fail-soft behavior into the manager runtime option construction/application path while keeping explicit user config writes strict.
