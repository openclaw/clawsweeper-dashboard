## Summary

Keep open: the reported Claude ACP timeout failure is valid and the PR has useful evidence, but the branch is now aimed too late in the call path for current main. Current main can reject the manager-injected `timeout` as `ACP_BACKEND_UNSUPPORTED_CONTROL` before `AcpxRuntime.setConfigOption` reaches this new guard.

## What This Changes

The PR hoists ACPX `timeout`/`timeout_seconds` filtering out of the Codex-only branch, updates the Claude ACP runtime unit test, and adds a changelog entry.

## Best Possible Solution

Handle manager-injected timeout as an optional client-side control in `src/acp/control-plane` before backend advertised-key rejection, while keeping ACPX runtime filtering as defense-in-depth and covering Claude ACP with regression tests.

## Reproduction Assessment

Yes, as source/report-reproducible rather than locally live-reproduced. Current main still synthesizes a `timeout` config pair, ACPX defaults provide `timeoutSeconds`, and the current Claude ACP package rejects unknown config ids; live comments corroborate the failure on 2026.5.7 and 2026.5.8.

## Solution Assessment

No for current main. The PR fixes the ACPX delegate-forwarding layer, but current main can now reject the unsupported key in manager runtime controls before that delegate guard executes; the narrow maintainable fix belongs in the manager/runtime-options apply path too.
