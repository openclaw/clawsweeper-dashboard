## Summary

Keep open: the Claude ACP timeout bug is valid, but this PR is now incomplete against current main because the manager can reject a non-advertised `timeout` control before the new ACPX wrapper skip is reached.

## What This Changes

This PR hoists ACPX `timeout`/`timeout_seconds` filtering from the Codex-only branch to all ACPX backends, updates the Claude ACP unit assertion, and adds a changelog entry.

## Best Possible Solution

Consolidate on a manager-level ACP runtime-control fix that skips or fail-softs auto-injected timeout controls, keeps explicit unsupported config writes strict, updates regression coverage, and includes live Claude ACP proof.

## Reproduction Assessment

Yes, there is a high-confidence source/report reproduction path: current main emits a `timeout` pair and rejects it against advertised Claude ACP keys, and the linked issue comments show live `ACP_BACKEND_UNSUPPORTED_CONTROL` for that path. I did not run a live Claude ACP session in this read-only review.

## Solution Assessment

No, this PR is not currently the best complete fix because filtering inside `AcpxRuntime.setConfigOption` is too late on current main. The narrower maintainable fix belongs in the manager/runtime-control layer for auto-injected timeout controls, with wrapper filtering only as defense-in-depth if still needed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
