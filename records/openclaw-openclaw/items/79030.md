## Summary

Keep open: current main still has a source-provable path where private model-provider fetches can select trusted env-proxy mode, while OpenClaw's NO_PROXY matcher intentionally mirrors Undici exact-host/suffix semantics and does not match CIDR or IPv4 wildcard entries.

## What This Changes

The issue reports that private model-provider requests can still use environment proxy routing when `NO_PROXY` only contains CIDR or IP wildcard entries.

## Best Possible Solution

Extend the target-aware env-proxy bypass decision for provider fetches with CIDR and narrowly scoped IPv4 wildcard handling, reuse existing CIDR helpers where possible, and leave managed proxy policy unchanged.

## Reproduction Assessment

Yes for source-level reproduction: current main selects trusted env-proxy mode for provider requests when `shouldUseEnvHttpProxyForUrl` returns true, and the matcher cannot match `100.64.*` or `100.64.0.0/10` to `100.64.0.3`. I did not run a live proxy capture in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to extend OpenClaw's own target-aware NO_PROXY decision before choosing trusted env-proxy mode. Defaulting every `allowPrivateNetwork` provider to direct transport would be a broader policy change and should not be the first repair path.

Current main lets model-provider fetches without an explicit dispatcher policy enter trusted env-proxy mode when `shouldUseEnvHttpProxyForUrl(url)` is true, but `matchesNoProxy` does not match CIDR entries such as `100.64.0.0/10` or narrowly scoped IPv4 wildcard entries such as `100.64.*`. Add regression tests for `100.64.0.0/10`, `100.64.*`, exact host and host:port, nonmatching ranges, public host nonmatches, and provider guarded-fetch no-dispatcher behavior; add a single-line user-facing changelog fix entry.
