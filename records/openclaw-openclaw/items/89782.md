## Summary

Keep open: current main still has the unguarded callback-dispatch bug and the PR is pointed at the right owner boundary, but the new guard should not merge as written because malformed non-string `pluginRoot` metadata is treated as unscoped instead of skipped.

## What This Changes

The PR adds a guarded conversation-binding resolved-callback registry enumerator and a regression test for an unreadable callback registration before a healthy Codex callback.

## Best Possible Solution

Keep the registry-read guard, but dispatch only sanitized callback registrations whose `pluginId`, optional `pluginRoot`, and handler satisfy the registry contract; malformed root metadata should be skipped before merge.

## Reproduction Assessment

Yes. Current main reads callback registration metadata before the per-handler try/catch, and the PR body reports a red focused Vitest where a throwing registration prevents the healthy callback from firing; I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The fix is in the right layer, but the helper should skip present non-string `pluginRoot` values instead of treating them as unscoped registrations.
