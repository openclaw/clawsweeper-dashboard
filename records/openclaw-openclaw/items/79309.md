## Summary

The report appears valid and actionable: current source can preserve an `openrouter/...` runtime model id and the OpenAI-compatible transport copies `model.id` directly into the chat-completions payload, which matches the reported OpenRouter 400 class.

## What This Changes

The issue asks OpenClaw to avoid sending the internal `openrouter/` provider prefix as the OpenRouter API request-body model for routed OpenRouter models.

## Best Possible Solution

Normalize the OpenRouter wire `model` for verified `openrouter.ai` chat-completions requests so routed refs send the upstream slug, while preserving valid OpenRouter-native ids and avoiding changes to unrelated OpenAI-compatible providers.

## Reproduction Assessment

Yes, at source level: when the OpenRouter runtime model id is `openrouter/<slug>`, the OpenAI-completions transport copies that id into `payload.model`. I did not run the live OpenRouter 400 path during this read-only review.

## Solution Assessment

The requested direction is correct, but the safest implementation is provider-specific wire normalization rather than a generic first-slash strip for every OpenAI-compatible provider.
