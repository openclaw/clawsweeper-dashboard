## Best Possible Solution

Keep the catalog scalar guard, then normalize direct pairing-list rows at the shared parser or renderer boundary and cover the pending-row CLI path with a focused regression test.

Yes. Source inspection shows persisted pairing records are blind-cast, `parsePairingList` passes pending rows through by cast, and pending-row renderers still call string-only operations on those scalars; I did not run tests because this review is read-only.

## Solution Assessment

No. The catalog chokepoint is the right fix for effective `node.list` and `node.describe` rows, but it is too narrow until the direct `node.pair.list` pending-row path is normalized too.
