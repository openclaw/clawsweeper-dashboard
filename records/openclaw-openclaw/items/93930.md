## Summary

Keep this PR open: it fixes the main malformed-scalar crash paths at the right read boundaries, but the catalog path still accepts pending rows with a malformed `requestId`, leaving an unapprovable pending state in `node.list`/`node.describe` while the direct pair-list parser drops the same row.

## What This Changes

The PR normalizes node catalog and `node.pair.list` scalar fields from permissive persisted pairing records and adds regression tests for malformed nodes CLI rendering.

## Best Possible Solution

Complete the boundary normalization by dropping catalog pending entries unless both `nodeId` and `requestId` normalize to strings, with regression coverage for the `node.list`/`nodes status` pending-approval output.

## Reproduction Assessment

Yes, source-reproducible. Current main blind-casts persisted pairing records and passes raw scalar fields into CLI renderers that call `.trim()` or `sanitizeTerminalText`, and the PR discussion includes terminal before/after proof through the real functions.

## Solution Assessment

No, not as-is. The boundary-level fix is the right layer and covers the main crash paths, but the catalog needs the same required-`requestId` drop semantics that the direct pair-list parser now has.
