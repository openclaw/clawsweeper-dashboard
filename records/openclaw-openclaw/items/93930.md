## Summary

Keep open: the bug is valid and the PR is mostly at the right boundary, but the catalog helper currently stops at the first non-nullish raw scalar and can drop a valid lower-priority fallback when a higher-priority persisted field is malformed.

## What This Changes

This PR normalizes malformed node catalog and pairing-list string fields plus invalid catalog ids so nodes CLI rendering does not crash on blind-cast persisted pairing records.

## Best Possible Solution

Keep the boundary-level hardening, but make the catalog scalar helper coalesce the first value that survives normalizeOptionalString and add a mixed malformed/valid fallback regression test.

## Reproduction Assessment

Yes. Source inspection shows persisted pairing records are blind-cast and current CLI renderers assume string scalars; the contributor also supplied terminal before/after proof through the real functions.

## Solution Assessment

No as submitted. Normalizing at the catalog and parser boundaries is the right approach, but the catalog helper should continue past invalid high-priority scalar values instead of erasing valid lower-priority metadata.
