## Summary

Keep open: the branch fixes the crash path, but `firstNormalizedString` currently stops on a malformed higher-priority scalar and can suppress a valid lower-priority catalog value; that should be repaired before merge.

## What This Changes

The PR normalizes malformed node catalog and `node.pair.list` string fields from persisted pairing state and adds regression coverage for nodes CLI rendering paths.

## Best Possible Solution

Keep the boundary-level normalization, but make catalog scalar selection return the first successfully normalized string across the existing precedence order before landing.

## Reproduction Assessment

Yes. Current main blind-casts persisted pairing records, then catalog/parser consumers assume string scalars and call `.trim()` or terminal sanitization on them; I verified the failing path from source but did not run a local repro in this read-only review.

## Solution Assessment

No as-is. The catalog and parser boundaries are the right layer, but the catalog helper must treat invalid high-priority scalars as absent and continue to lower-priority valid values.
