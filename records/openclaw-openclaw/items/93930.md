## Best Possible Solution

Keep the boundary-level hardening, but skip invalid scalars before falling back and filter direct pair-list rows with invalid required IDs before merge.

## Reproduction Assessment

## Solution Assessment

No. Boundary-level normalization is the right layer, but the implementation should treat invalid scalar candidates as absent and drop unusable direct pair-list rows instead of returning empty required IDs.

Full review comments:

- **[P2] Skip invalid scalars before falling back:** `src/gateway/node-catalog.ts:87`
  - body: `firstNormalizedString` normalizes the first non-null candidate, so a malformed higher-priority value such as a numeric `nodePairing.displayName` returns `undefined` and prevents a valid lower-priority `devicePairing.displayName` from being used. Treat invalid scalars as absent by returning the first candidate that `normalizeOptionalString` accepts, and add a regression that preserves the existing fallback behavior.
  - confidence: 0.9
- **[P2] Drop rows with invalid required IDs:** `src/shared/node-list-parse.ts:11`
  - body: Returning `""` for malformed `requestId` or `nodeId` leaves typed pair-list rows with unusable IDs, which can render blank CLI rows or let the agent fallback choose an empty node target. Match the catalog-side behavior by filtering pending/paired rows whose required IDs are not valid non-empty strings instead of using an empty-string sentinel.
  - confidence: 0.87
