## Summary

Current main and shipped releases already implement the central targetless `/steer <message>` behavior through the merged current-session steer command, so this PR’s subagent-handler fallback is superseded and should not be merged.

## What This Changes

The PR adds a fallback that auto-selects the sole active subagent for targetless `/steer` input and adds unit coverage for that fallback.

## Best Possible Solution

Keep the shipped current-session `/steer <message>` implementation and leave child-run steering explicit through `/subagents steer <id|#> <message>`.

## Reproduction Assessment

No against current main. Source, docs, and tests now route top-level `/steer <message>` through `handleSteerCommand` before subagent parsing, so the old first-word-as-subagent-id path no longer reproduces.

## Solution Assessment

No for this branch. The merged current-session handler is the narrower maintainable fix, and this branch would also broaden `/subagents steer` beyond the documented explicit-target contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep auto-select out of `/subagents steer`:** `src/auto-reply/reply/commands-subagents/action-send.ts:63-66`
  - body: This fallback runs for any `steerRequested` call, including `/subagents steer hurry` where `handledPrefix` is `/subagents` and `message` is empty. With one active child it silently treats `hurry` as the message instead of returning the documented `Usage: /subagents steer <id|#> <message>` response, so missing or mistyped child targets can be delivered to the wrong run.
