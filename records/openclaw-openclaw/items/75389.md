## Summary

Keep open. The PR is useful source work for the linked diagnostic-log growth issue, but it is not mergeable as-is because it wires cache-trace and Anthropic payload logs into a cap that drops future writes instead of rotating, and it leaves the new user-facing config/env surface incompletely documented.

## What This Changes

The PR adds max-byte defaults/env parsing for Anthropic payload and cache-trace JSONL writers, wires `diagnostics.cacheTrace.maxFileBytes` through config schema/types/generated schema, and adds focused max-byte handling tests.

## Best Possible Solution

Land a bounded rotation implementation for the queued diagnostic JSONL files, keep a small number of numbered archives, continue writing to a fresh active file after the cap, and publish the config/env/docs/changelog surface with regression coverage.

## Reproduction Assessment

Yes. Fill `cache-trace.jsonl` or `anthropic-payload.jsonl` to the configured cap; the PR routes both through `safeAppendFile(..., { maxFileBytes })`, whose cap branch returns before appending future diagnostic lines.

## Solution Assessment

No. Suppressing writes bounds disk usage but loses diagnostics for long-lived gateways; the maintainable fix is rotation/retention matching the primary logger's behavior, plus complete public config coverage.
