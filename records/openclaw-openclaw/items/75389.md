## Summary

Keep open. The PR is a useful start, but it wires the diagnostic JSONL logs to the existing capped queued writer, whose cap behavior drops all later writes instead of rotating to a fresh active file; the linked report asks for bounded rotation/archives. The proposed user-facing config/env surface is also incomplete across generated metadata, help/docs, env coverage, and changelog.

## What This Changes

The PR adds max-byte defaults/env parsing for Anthropic payload and cache-trace JSONL logging, wires `diagnostics.cacheTrace.maxFileBytes` through config types/schema/generated schema, and adds focused tests for max-byte handling.

## Best Possible Solution

Land a focused bounded-rotation implementation for queued diagnostic JSONL files that keeps a small retained archive set, continues writing to a fresh active file after the cap, preserves existing redaction and filesystem protections, and publishes the full config/env/docs/changelog surface with regression tests.

## Reproduction Assessment

Yes. Configure or force a small `maxFileBytes`, write enough cache-trace or Anthropic payload lines to exceed it, and the shared `safeAppendFile` cap branch returns without appending future lines; the existing queued-writer test already proves that drop behavior.

## Solution Assessment

No. The PR’s cap suppresses writes, which prevents unbounded growth but does not satisfy the requested rotation/retention behavior; rotating the queued writer and retaining bounded archives is the narrower maintainable fix.
