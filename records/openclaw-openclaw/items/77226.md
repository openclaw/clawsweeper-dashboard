## Summary

The report’s unresolved SecretRef evidence matches a systemd service environment setup problem, while current main already has Discord named-account SecretRef runtime targets and account-aware runtime snapshot resolution.

## What This Changes

The issue reports that named Discord account tokens configured as SecretRefs still fail to resolve at gateway startup in v2026.5.3.

## Best Possible Solution

Keep the current account-aware resolver and route this setup to the documented service environment path, with a new issue only if the service environment contains the variable and the SecretRef still remains unresolved.

## Reproduction Assessment

Yes for the support cause: if a systemd service does not receive the env var, an env SecretRef remains unresolved even when an interactive shell can use the token. I did not reproduce a current-main Discord resolver bug; the source and tests show named account refs are handled through the runtime snapshot.

## Solution Assessment

Yes: the narrowest maintainable answer is to put the token in the service environment (`~/.openclaw/.env` or install/reinstall from a shell that has it) rather than changing Discord token resolution.
