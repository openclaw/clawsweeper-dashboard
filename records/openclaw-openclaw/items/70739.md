## Summary

Keep open: the underlying gateway SSE idle-timeout bug is source-reproducible, but this PR still has a heartbeat interval validation blocker, test timer cleanup issue, missing posted real behavior proof, and an unresolved stream-lifetime/security policy question.

## What This Changes

Adds a `startSseHeartbeat` helper that writes SSE `: ping` comments to the streaming `/v1/chat/completions` and `/v1/responses` gateway paths, with unit coverage for timer, backpressure, and error cleanup behavior.

## Best Possible Solution

Land a hardened SSE-comment heartbeat after finite interval normalization, test timer cleanup, redacted proxy/curl proof, and an explicit stream-limit policy decision.

## Reproduction Assessment

Yes. This is source-reproducible: current `main` sets SSE headers for both streaming OpenAI-compatible endpoints but emits no periodic bytes during quiet upstream model windows, so an idle-timeout proxy can close the stream before `[DONE]` or `response.completed`.

## Solution Assessment

Unclear as-is. SSE comments are the right transport shape and match existing session-history precedent, but this patch needs interval hardening, test cleanup, real behavior proof, and a stream-limit policy decision before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate the heartbeat interval before scheduling:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` leaves `NaN`, `Infinity`, and `-Infinity` unsafe. Node coerces those delays to 1ms, so a malformed override can turn each open SSE stream into a high-frequency write loop; normalize with `Number.isFinite` before scheduling.
  - confidence: 0.9
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:302-304`
  - body: This suite enables fake timers in `beforeEach` but never restores real timers. In shared-worker runs, later tests in the file can inherit fake timers and stop exercising real timer behavior, so add cleanup that clears pending timers and calls `vi.useRealTimers()`.
  - confidence: 0.68
