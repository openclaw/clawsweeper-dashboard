## Summary

Keep open: current main still lacks downstream SSE heartbeats and the PR addresses a real gateway transport gap, but the branch needs a finite-interval fix, test cleanup, real behavior proof, and maintainer security/product judgment on long-lived stream limits before merge.

## What This Changes

The branch adds a `startSseHeartbeat` helper, wires it into streaming `/v1/chat/completions` and `/v1/responses`, and adds unit coverage for interval, backpressure, and error cleanup behavior.

## Best Possible Solution

Land the SSE-comment heartbeat after finite interval normalization, timer cleanup, redacted proxy/curl proof, and an explicit stream-limit policy decision.

## Reproduction Assessment

Yes. Source inspection shows current main has streaming `/v1/chat/completions` and `/v1/responses` paths with no periodic downstream bytes during quiet model windows, so an idle-timeout proxy can close the stream before `[DONE]` or `response.completed`; I did not run a live Envoy repro in this read-only review.

## Solution Assessment

No, not as-is. SSE comments match the right transport shape and existing session-history precedent, but this branch still needs interval normalization, test timer cleanup, real behavior proof, and maintainer agreement on stream lifetime/concurrency policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate the heartbeat interval before scheduling:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` still lets `NaN` and `Infinity` reach `setInterval`; Node 22 coerces those to a 1ms interval, so a malformed future override can create a high-frequency heartbeat loop per open stream. Normalize with `Number.isFinite` before clamping and falling back to the default.
  - confidence: 0.91
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:302-304`
  - body: This suite enables fake timers in `beforeEach` and never restores real timers. In shared-worker or `--isolate=false` runs, later tests can inherit fake timers and stop exercising real timer behavior, so clear pending timers and call `vi.useRealTimers()` in cleanup.
  - confidence: 0.72

## Security Review
