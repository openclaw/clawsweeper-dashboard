## Summary

Keep open for normal PR review. Current main still lacks downstream SSE keepalives on the two public OpenAI-compatible streaming endpoints, so the PR is directionally useful, but the proposed patch has a finite-interval bug, test timer cleanup gap, and a stream-lifetime security posture question that should be resolved before merge.

## What This Changes

The PR adds a shared SSE heartbeat helper with unit tests and wires it into streaming `/v1/chat/completions` and `/v1/responses` responses.

## Best Possible Solution

Land a hardened SSE-comment heartbeat for both streaming endpoints after normalizing the interval to a finite bounded value, restoring real timers in the new tests, preserving backpressure/error handling, and making an explicit gateway threat-model decision on stream duration or concurrency limits.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible because the streaming branches only write model events after `setSseHeaders()`, and the PR interval bug is reproducible with Node 22 coercing non-finite interval values to 1ms timers.

## Solution Assessment

Unclear as-is. SSE comments are the right transport-level direction and match existing session-history precedent, but this patch is not the best merge shape until the interval validation, test cleanup, and stream-lifetime security posture are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate the heartbeat interval before scheduling:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` leaves `NaN`, `Infinity`, and `-Infinity` unchanged. Node coerces those durations to a 1ms interval, so any future malformed numeric override can turn each open SSE stream into a high-frequency write loop despite the intended clamp. Normalize with `Number.isFinite` and fall back to the default before calling `setInterval`.
  - confidence: 0.9
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:301-303`
  - body: The new heartbeat suite enables fake timers in `beforeEach` but never restores real timers. That can leave later tests in this file running under fake timers in shared-worker runs, causing hangs or hiding real timer behavior. Add cleanup that clears pending timers and calls `vi.useRealTimers()`.
  - confidence: 0.68

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, or secret-handling change, but it changes stream lifetime behavior in a way that should get explicit gateway security/product review before merge.

Concerns:

- **[medium] Heartbeat bypasses proxy idle caps without stream limits:** `src/gateway/openai-http.ts:639`
  - body: The PR starts heartbeats unconditionally on authenticated streaming endpoints. Backpressure handling prevents unbounded heartbeat buffering, but current gateway code does not show a max stream lifetime or per-subject/IP concurrent-stream cap, so internet-exposed or `auth.mode: none` deployments lose proxy idle timeout as a fallback resource limiter for quiet streams.
