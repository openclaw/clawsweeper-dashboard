## Summary

Keep open. Current main still lacks downstream SSE heartbeats on the two OpenAI-compatible streaming endpoints, so the PR is useful source work, but the patch is not merge-ready because the helper accepts non-finite timer values, the new fake-timer tests leak timer mode, and the stream-lifetime security posture needs maintainer review.

## What This Changes

The PR adds a reusable `startSseHeartbeat()` helper with unit tests and calls it from the streaming `/v1/chat/completions` and `/v1/responses` gateway paths.

## Best Possible Solution

Land a hardened SSE-comment heartbeat for the two streaming endpoints after normalizing the interval to a finite bounded value, adding fake-timer cleanup in the tests, preserving the backpressure/error handling, and making an explicit gateway threat-model decision on stream duration or concurrency limits.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible because both streaming branches only write real model events after `setSseHeaders()`, and the PR's interval bug is locally reproducible with Node showing non-finite interval values become 1ms timers.

## Solution Assessment

Unclear as-is. SSE comments are the right transport-level direction and match existing session-history precedent, but this patch needs interval validation, test cleanup, and maintainer security review before it is the narrowest safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate the heartbeat interval before scheduling:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` leaves `NaN`, `Infinity`, and `-Infinity` unchanged. Node coerces those durations to a 1ms interval, so any future caller that passes a malformed numeric override can turn each open SSE stream into a high-frequency write loop despite the intended clamp. Normalize with `Number.isFinite` and fall back to the default before calling `setInterval`.
  - confidence: 0.9
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:301-303`
  - body: The new heartbeat suite enables fake timers in `beforeEach` but never restores real timers. That leaves the rest of this test file running under fake timers in shared-worker runs, which can make later async or timer-based assertions hang or stop exercising real timer behavior. Add cleanup that clears pending timers and calls `vi.useRealTimers()`.
  - confidence: 0.68

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it introduces a stream-lifetime security posture change that should be reviewed before merge.

Concerns:

- **[medium] Heartbeat bypasses proxy idle caps without stream limits:** `src/gateway/openai-http.ts:639`
  - body: The PR starts heartbeats unconditionally on authenticated streaming endpoints. Backpressure handling prevents unbounded heartbeat buffering, but current gateway code does not show a maximum stream lifetime or per-subject/IP concurrent-stream cap, so internet-exposed or `auth.mode: none` deployments lose proxy idle timeout as a fallback resource limiter for quiet streams.
  - confidence: 0.74
