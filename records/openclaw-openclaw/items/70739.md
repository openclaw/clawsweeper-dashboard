## Summary

This PR should stay open: it addresses a real current-main SSE keepalive gap, but the head patch still has a blocking interval-validation bug, test timer cleanup leak, missing observable real-behavior proof, and a stream-lifetime security posture question for maintainers.

## What This Changes

The PR adds a `startSseHeartbeat()` helper with tests and calls it from the streaming `/v1/chat/completions` and `/v1/responses` gateway paths.

## Best Possible Solution

Land a hardened SSE-comment heartbeat for both streaming endpoints after normalizing heartbeat intervals, restoring test timers, preserving backpressure/error handling, adding observable runtime proof, and making an explicit gateway decision on stream duration or concurrency limits.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible because both streaming endpoints only write model events after `setSseHeaders()`, and an idle-timeout proxy can close that quiet stream before `response.completed` or `[DONE]`.

## Solution Assessment

Unclear as-is. SSE comments are the right transport-level shape and match existing session-history precedent, but this patch needs interval normalization, timer cleanup, real behavior proof, and maintainer security judgment first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate the heartbeat interval before scheduling:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` leaves `NaN`, `Infinity`, and `-Infinity` unchanged. Node coerces those delays to a 1ms interval, so a malformed override can turn each open SSE stream into a high-frequency write loop; normalize with `Number.isFinite` before calling `setInterval`.
  - confidence: 0.9
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:301-303`
  - body: The new heartbeat suite enables fake timers in `beforeEach` but never restores real timers. In shared-worker runs, later tests can inherit fake timers and stop exercising real timer behavior, so add cleanup that clears pending timers and calls `vi.useRealTimers()`.
  - confidence: 0.68
