## Summary

Keep open. Current main and the latest release still only expose the generic Agents > Files editor for allowlisted bootstrap files plus root MEMORY.md; there is no dedicated Memory panel, no /memory route, and no editable daily memory note surface. The closest memory-view PR was closed unmerged, and the open workspace-file proposal does not settle this memory-specific product and security boundary.

## What This Changes

The issue requests a dedicated Control UI memory panel for reading and editing USER.md, MEMORY.md, and daily memory notes.

## Best Possible Solution

Choose the product/API boundary first, then add a scoped Gateway memory-files API and Control UI Memory surface for USER.md, root MEMORY.md, and validated daily notes with security tests, docs, and UI proof.

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence absence check: no Memory tab or /memory route, and agents.files still rejects daily memory note paths.

## Solution Assessment

Unclear. A dedicated memory panel is plausible, but maintainers should first choose between a memory-specific UI, broader workspace file management, and pinned-context UX before adding browser write access to personal context files.
