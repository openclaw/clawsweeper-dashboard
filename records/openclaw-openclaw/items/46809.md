## What This Changes

Request a dedicated Control UI Memory panel to read and edit USER.md, MEMORY.md, and today's memory/YYYY-MM-DD.md daily note.

## Best Possible Solution

Choose the product/API boundary first, then add a scoped Gateway memory-files API and Control UI Memory surface for USER.md, root MEMORY.md, and validated daily notes with security tests, docs, and UI proof.

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives a high-confidence absence check: no Memory tab or /memory route, and agents.files still rejects memory/YYYY-MM-DD.md.

## Solution Assessment

Unclear. A dedicated memory panel is plausible, but maintainers should first choose between memory-specific UI, broader workspace file management, pinned context, and plugin contribution surfaces before adding browser write access to personal context files.
