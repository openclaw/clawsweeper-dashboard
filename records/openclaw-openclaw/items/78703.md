## Summary

Keep open: current docs promise cron persistence and source shows a still-open hot-reload cron race, but the reported `cron: job removed` log maps to the explicit `cron.remove` RPC path, so the exact trigger still needs correlation with full logs.

## What This Changes

The issue asks OpenClaw to prevent successfully created cron jobs from being silently removed or emptied during Gateway config reloads, with possible recovery from `jobs.json.bak`.

## Best Possible Solution

Land or supersede the graceful cron hot-reload drain work, then add targeted logging/tests that distinguish explicit `cron.remove` calls from reload-time persistence overwrites.

## Reproduction Assessment

Unclear for the exact incident timeline: source inspection shows a current-main hot-reload cron race, but the observed `cron: job removed` log currently points to an explicit remove RPC rather than reload code. Full Gateway logs or RPC trace around 10:30:41 would make this high-confidence.

## Solution Assessment
