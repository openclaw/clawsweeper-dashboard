## Summary

Keep open: the report describes plausible cron data loss against the documented persistence contract, and current main still does not provide an implemented, duplicate, or cannot-reproduce close path. Source inspection narrows the observed `cron: job removed` log to a successful explicit `cron.remove` request, while config reload itself still does not directly remove jobs.

## What This Changes

Investigate whether a cron job can be silently removed around config reload after a failed then successful cron add.

## Best Possible Solution

Add jobId-scoped removal-origin audit first, then fix the confirmed caller or add guarded backup recovery only for a proven unintended-removal or corruption signal.

## Reproduction Assessment

No. The incident timeline is detailed and current source maps the observed `cron: job removed` log to `cron.remove`, but there is not a high-confidence current-main reproduction that config reload itself removes a job.

## Solution Assessment

Unclear. Backup restore may be useful for corruption or clobber cases, but the safer first step is removal caller attribution so intentional deletes are not undone.
