## Label Justifications

- `P0`: The reported behavior was silent loss of scheduled cron jobs, which is user data loss even though it is now fixed.
- `impact:data-loss`: The issue is about persisted cron job definitions disappearing from storage.
- `impact:message-loss`: Lost cron jobs can silently stop scheduled automation and expected channel deliveries.

## Summary

Current main and the latest release use the SQLite-backed cron store instead of the old active jobs.json path, so the reported config-reload data-loss path is implemented away.

## What This Changes

The issue reports cron jobs being silently removed from jobs.json around a runtime config reload in v2026.5.4 after a failed then successful cron.add sequence.

## Best Possible Solution

Keep the SQLite-backed cron store and route any remaining post-SQLite upgrade/default fallout through the newer canonical cron issues.

## Reproduction Assessment

No current-main repro remains: current source and v2026.6.6 store cron jobs in SQLite, and config reload rebuilds the service without calling the remove path. The original v2026.5.4 jobs.json sequence was not replayed.

## Solution Assessment

Yes. Replacing the active jobs.json store with transactional SQLite persistence plus doctor migration is a better fix than adding another runtime jobs.json.bak fallback reader.
