## Label Justifications

- `P0`: The reported failure silently deleted scheduled cron state and stopped automation without alerting the user.
- `impact:data-loss`: The issue is specifically about persisted cron job state disappearing from the active store.
- `impact:message-loss`: The lost cron job caused scheduled delivery/automation to miss its expected run.

## Summary

Current main and the shipped v2026.6.6 release implement the central fix: cron jobs are stored in SQLite rather than the old active jobs.json file, reload rebuilds the cron service without calling the remove path, and the ambiguous delivery failure is rejected before cron.add mutates state.

## What This Changes

The issue reports a v2026.5.4 cron job being silently removed around runtime config reload, leaving jobs.json empty while jobs.json.bak still had the job.

## Best Possible Solution

Keep the SQLite-backed cron store and route any remaining post-migration cron reliability reports through narrower current-version issues, rather than adding a jobs.json.bak runtime fallback.

## Reproduction Assessment

No high-confidence current-main reproduction remains: current source and v2026.6.6 use SQLite cron persistence, and the reported multiple-channel validation failure returns before cron.add mutates state. I did not replay the original v2026.5.4 live sequence.

## Solution Assessment

Yes. Replacing the active jobs.json persistence path with transactional SQLite storage plus doctor migration is a better current solution than adding another jobs.json.bak runtime fallback reader.
