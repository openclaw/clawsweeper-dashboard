## Label Justifications

- `P0`: The original report described silent loss of scheduled automation state, which can stop unattended workflows without alerting the user.
- `impact:data-loss`: The issue is about persisted cron job definitions disappearing from durable scheduler state.
- `impact:message-loss`: The lost cron job caused scheduled automation output to be missed.

## Summary

Current main and release v2026.6.6 already address the central failure: cron job definitions and runtime state are SQLite-backed, legacy jobs.json is migrated by doctor/preflight, hot reload restarts cron by stop/start without clearing persisted jobs, and the reported `cron: job removed` log now maps only to explicit cron.remove.

## What This Changes

Cron jobs created before a config reload should remain durable and not be silently removed from persistent scheduler state.

## Best Possible Solution

Keep the shipped SQLite-backed cron persistence and doctor/preflight legacy import path, with separate narrow issues for any remaining migration edge cases.

## Reproduction Assessment

No current-main failing reproduction is present. Source inspection shows the old jobs.json-backed active store has been replaced by SQLite, and the reload path now stops/restarts cron without clearing persisted jobs.

## Solution Assessment

Yes. Moving cron definitions and runtime state into SQLite, with doctor-owned legacy import, is the maintainable fix because reload/restart no longer depends on rewriting jobs.json as the active source of truth.
