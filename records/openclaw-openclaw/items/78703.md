## Summary

Keep open: this is a recent cron data-loss report against an existing persistence contract, and current main does not provide a high-confidence implemented, duplicate, or cannot-reproduce close path. Source inspection narrows the likely trigger because the reported gateway log is emitted after an explicit `cron.remove` RPC, while the failed `cron.add` validation returns before the cron store is mutated.

## What This Changes

The issue reports a successfully created cron job being silently removed around a runtime config reload after an initial failed `cron.add` and a successful retry.

## Best Possible Solution

Identify the actual remover from source-attributed cron audit logs, then fix the confirmed reload/timer/plugin path or add a guarded recovery mechanism that cannot resurrect intentional deletions.

## Reproduction Assessment

No. The incident report is detailed, but current-main inspection only narrows the path: failed add should not mutate cron state, and the observed gateway remove log maps to an explicit `cron.remove` request rather than a proven config-reload deletion path.

## Solution Assessment

Unclear. A backup restore alone is not the best fix until maintainers know whether the remover was reload logic, a plugin cleanup path, an agent/tool call, or an operator-initiated `cron.remove`.
