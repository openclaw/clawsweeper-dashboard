## Summary

Keep open: the report describes cron data loss against the documented persistence contract, and current main still does not provide an implemented, duplicate, or cannot-reproduce close path. Source inspection narrows the symptom because the cited `cron: job removed` gateway log is still emitted only after an explicit `cron.remove` request succeeds, while config reload itself does not directly remove jobs.

## What This Changes

The issue reports cron job definitions being silently removed around a runtime config reload after a successful `cron.add`.

## Best Possible Solution

Add source-attributed audit around cron removals, identify the actual remover, then fix that confirmed path with guarded recovery logic that cannot undo intentional deletes.

## Reproduction Assessment

No. The incident report is detailed and source inspection maps the observed log to `cron.remove`, but there is not yet a high-confidence current-main path that makes config reload remove a job.

## Solution Assessment

Unclear. Backup restore alone is not the best fix until maintainers know whether the remover was reload logic, an agent/tool call, plugin cleanup, or an operator action.
