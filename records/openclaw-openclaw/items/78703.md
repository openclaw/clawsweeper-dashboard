## Summary

Keep open: this is a recent cron data-loss report with a documented persistence contract, and current main does not provide a high-confidence implemented or duplicate close path. Source inspection narrows the likely trigger because the quoted `cron: job removed` log is emitted by the explicit `cron.remove` handler, while the failed `cron.add` delivery validation returns before `context.cron.add` is called.

## What This Changes

The issue requests investigation and a fix for successfully created cron jobs being removed around runtime config reloads, with optional recovery from `jobs.json.bak`.

## Best Possible Solution

Identify the actual remover from jobId-scoped gateway/tool-call logs, then either fix the confirmed reload/timer path or add source-attributed cron removal audit and a guarded recovery path that cannot undo intentional deletes.

## Reproduction Assessment

No. The incident report is detailed, but current-main source inspection only narrows the path: the failed add should not mutate cron state, and the observed removal log maps to an explicit `cron.remove` call rather than a proven config-reload deletion path.

## Solution Assessment

Unclear. A backup restore alone is not the best fix until maintainers know whether this was a reload persistence bug, an agent/plugin self-removal, an operator `cron.remove`, or another explicit remover.
