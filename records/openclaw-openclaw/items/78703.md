## Label Justifications

- `P0`: The reported behavior was silent loss of scheduled cron automation, which is severe data and delivery loss even though the fix is now shipped.
- `impact:data-loss`: The issue is about persisted cron job definitions disappearing from the old jobs.json store.
- `impact:message-loss`: The lost cron job caused scheduled automation and its expected delivery to be missed without alerting.

## Summary

Current main and the shipped v2026.6.6 release use the SQLite-backed cron store with doctor migration for legacy jobs.json state, so the reported jobs.json config-reload data-loss path is already implemented away.

## What This Changes

The issue reports cron jobs being silently removed around runtime config reload after a failed add followed by a successful add.

## Best Possible Solution

Keep the SQLite-backed cron store and route any remaining post-SQLite upgrade fallout through narrow new reports with current-version logs.

## Reproduction Assessment

No current-main repro remains: current source and v2026.6.6 store cron jobs in SQLite, and config reload rebuilds the service without calling the remove RPC path. The original v2026.5.4 jobs.json sequence was not replayed.

## Solution Assessment

Yes. Replacing active jobs.json persistence with transactional SQLite storage plus doctor migration is a better fix than adding another runtime jobs.json.bak fallback reader.
