## Summary

Keep open: the PR targets a real provider discovery bug, but falling back from `discoveryApiKey` to raw `apiKey` can turn non-secret markers or env placeholders into Authorization headers, and the supplied proof is a mocked resolver exercise rather than a real self-hosted `/models` run.

## What This Changes

The branch changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into the provider builder and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted live `/models` proof.

## Reproduction Assessment

Yes, source-reproducible. The resolver can expose a truthy public marker with `discoveryApiKey` undefined, and the PR head then passes that marker to a builder whose local model discovery turns any non-empty key into Bearer Authorization.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fix should preserve the concrete `discoveryApiKey` separation instead of treating non-secret markers as bearer tokens.
