## Summary

Keep open: this is the active fix candidate for the provider discovery auth bug, but it is not merge-ready because marker-only inputs still reach dynamic discovery with no concrete credential and the supplied proof is local script/unit output rather than a redacted live endpoint run.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? toDiscoveryApiKey(apiKey)` into provider builders and adds tests for concrete keys, non-secret markers, and missing-key cases.

## Best Possible Solution

Land a credential-aware discovery fix that forwards concrete credentials, avoids or explicitly scopes marker-only unauthenticated probes, and includes redacted live endpoint proof.

## Reproduction Assessment

Yes, at source level: marker-only `apiKey` plus missing `discoveryApiKey` produces no discovery credential, and vLLM/SGLang builders then call `/models` without Authorization. I did not establish a live endpoint reproduction in this read-only review.

## Solution Assessment

No, not yet. `toDiscoveryApiKey` is the right primitive for avoiding marker text in bearer headers, but the PR still needs marker-only policy resolution and live endpoint proof before it is the best fix.
