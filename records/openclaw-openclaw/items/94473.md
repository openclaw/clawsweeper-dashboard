## Summary

Keep open: this PR targets the right self-hosted provider discovery surface, but the proposed fallback can send non-secret marker or placeholder values as bearer credentials and the proof is still mock-only rather than a real self-hosted `/models` run.

## What This Changes

The branch changes self-hosted OpenAI-compatible provider discovery to fall back from `discoveryApiKey` to `apiKey` when building the provider and adds regression tests for discovery-key precedence, marker fallback, and missing keys.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless coverage and redacted real `/models` proof.

## Reproduction Assessment

Yes, at source level. Current resolvers can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` contract or filter marker values instead of treating non-secret markers as bearer tokens.
