## Summary

Keep open: the PR edits the right self-hosted provider discovery boundary, but the fallback treats public marker values as concrete bearer credentials, so it can regress marker-backed/authless discovery and still not authenticate servers that need a real key.

## What This Changes

This PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted live `/models` proof.

## Reproduction Assessment

Yes. Source inspection shows resolver paths can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that converts any non-empty key into a bearer header.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fix should preserve the concrete `discoveryApiKey` separation or filter the fallback instead of treating non-secret markers as bearer tokens.
