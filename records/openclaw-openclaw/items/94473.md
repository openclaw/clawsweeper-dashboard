## Summary

Keep this PR open but not merge-ready: the force-pushed patch fixed the earlier marker-as-bearer problem for known markers, but the new fallback can still turn unresolved env-ref names into bearer tokens, and marker-only discovery still appears to run the unauthenticated `/models` probe described by the linked issue. The PR body also confirms there is no live self-hosted endpoint proof after the patch.

## What This Changes

The PR changes self-hosted provider discovery to fall back through `toDiscoveryApiKey(apiKey)` when `discoveryApiKey` is missing and adds regression tests for discovery-key precedence, concrete keys, markers, and missing keys.

## Best Possible Solution

Preserve the concrete discovery-credential contract: only a resolved `discoveryApiKey` or another proven concrete secret should reach the provider builder, while marker/env-ref-only availability should skip discovery or stay authless only when that provider behavior is intentional and tested.

## Reproduction Assessment

Yes, source-level. Current tests and source show env-ref profile auth can expose an env var name with no discovery key, and marker-only resolver output still reaches local-model discovery without a concrete key.

## Solution Assessment

No. The patch filters known marker strings, but it can still forward unresolved env-ref names as bearer tokens and does not stop marker-only unauthenticated discovery for the linked symptom.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep env-ref names out of discovery Authorization:** `src/plugins/provider-self-hosted-setup.ts:436`
  - body: When an auth profile is backed by an env keyRef but the env value is unavailable, the resolver can return `apiKey: "OPENAI_PROFILE_KEY"` with `discoveryApiKey: undefined`. `toDiscoveryApiKey` does not filter arbitrary env-ref names, so this fallback can make self-hosted discovery send `Authorization: Bearer OPENAI_PROFILE_KEY`. Preserve the concrete `discoveryApiKey` contract or otherwise prove the fallback is a real secret before passing it to `buildProvider`.
  - confidence: 0.91
- **[P2] Skip marker-only discovery instead of forwarding no key:** `src/plugins/provider-self-hosted-setup.ts:436`
  - body: For marker-only auth such as `custom-local`, `ollama-local`, OAuth, AWS SDK, or SecretRef markers, this expression still produces `undefined`, so `buildProvider` can continue into `discoverOpenAICompatibleLocalModels` with no key and emit the same unauthenticated `GET /models` described by the linked issue. Filter marker-only auth before invoking the builder, or narrow the PR so it no longer claims to close that marker-only bug.
  - confidence: 0.85
