## Summary

Keep open: this is an active fix candidate for the linked provider discovery bug, but the current patch should not merge because it can turn non-secret markers into bearer tokens and the supplied proof is mock-only.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into the provider builder and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery Authorization based only on concrete resolved discovery credentials, skipping marker-only probes when necessary, with focused coverage and redacted real endpoint proof.

## Reproduction Assessment

Yes, source-reproducible. The resolver can expose a truthy public marker with `discoveryApiKey` undefined, and the PR head then passes that marker to a builder whose local model discovery turns any non-empty key into Bearer Authorization.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fix should preserve the concrete `discoveryApiKey` separation instead of treating non-secret markers as bearer tokens.
