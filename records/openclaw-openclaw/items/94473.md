## Summary

Keep this PR open, but it is not merge-ready: it edits the right self-hosted discovery boundary, yet the fallback can turn non-secret marker values into discovery bearer tokens and the supplied proof is a fake-provider capture rather than a real `/models` endpoint run.

## What This Changes

The PR changes self-hosted OpenAI-compatible provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for discovery-key precedence, marker fallback, and missing keys.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or apply marker-aware normalization before any fallback, then prove both auth-required and authless local behavior.

## Reproduction Assessment

Yes, at source level. Current resolvers can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` contract or filter marker values instead of treating non-secret markers as bearer tokens.
