## Summary

Keep open and do not merge as-is: the PR edits the implicated self-hosted discovery path, but falling back from marker-filtered `discoveryApiKey` to raw `apiKey` can send non-secret markers as bearer tokens, and the supplied proof is not a real endpoint run.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted live `/models` proof.

## Reproduction Assessment

Yes, at source level. Resolver paths can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that converts any non-empty key into a bearer header.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fix should preserve the concrete `discoveryApiKey` separation or filter the fallback instead of treating non-secret markers as bearer tokens.
