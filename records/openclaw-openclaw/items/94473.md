## Summary

Keep open: this PR targets a real linked provider-catalog auth bug, but the patch is not merge-ready because it can send non-secret marker strings as bearer credentials and the supplied proof is only a mocked expression capture.

## What This Changes

This PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted live `/models` proof.

## Reproduction Assessment

Yes, at source level. Resolver paths can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that converts any non-empty key into an Authorization header.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback must preserve concrete `discoveryApiKey` separation or filter `apiKey` through marker-aware discovery normalization instead of treating markers as bearer tokens.
