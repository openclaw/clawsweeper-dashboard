## Summary

Keep open: the linked provider-catalog bug is still present on current main and the latest release, but this PR is not merge-ready because the fallback can turn non-secret provider markers into bearer tokens and the supplied proof is mock-only.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds tests for that fallback.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted live `/models` proof.

## Reproduction Assessment

Yes, at source level. Current main can carry a truthy non-secret marker as `apiKey`, drop it from `discoveryApiKey`, and then issue `/models` discovery without an Authorization header; I did not run a live LiteLLM server in this read-only review.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` separation or filter/skip marker-only values instead of treating non-secret markers as bearer tokens.
