## Summary

Keep open: the PR is focused on the right provider-discovery surface, but it still has blocking auth correctness issues and no live after-fix endpoint proof. The new fallback can forward unresolved env-ref names as bearer text, and marker-only discovery can still run the unauthenticated `/models` path described by the linked issue.

## What This Changes

The branch changes self-hosted provider discovery to pass `discoveryApiKey ?? toDiscoveryApiKey(apiKey)` into the provider builder and adds regression tests for concrete keys, marker values, discovery-key precedence, and missing keys.

## Best Possible Solution

Keep discovery auth concrete-only: use resolved discovery keys for authenticated probes, skip marker-only dynamic discovery before the HTTP request, preserve persisted marker semantics, and require redacted after-fix provider logs.

## Reproduction Assessment

Yes, source-reproducible. Current main can pass marker-backed or env-ref-backed auth through the provider discovery path, and `discoverOpenAICompatibleLocalModels` omits Authorization when no concrete key reaches it; the issue also includes before-fix LiteLLM logs.

## Solution Assessment

No. The PR is in the right area, but it is not the best fix until unresolved env refs stay out of Authorization and marker-only dynamic discovery is skipped or explicitly scoped.
