## Summary

Keep open: the branch is a useful credential-filtering primitive, but it is not merge-ready because marker-only inputs still proceed into self-hosted discovery with no Authorization header and the submitted proof is source/mock output rather than a redacted live `/models` run.

## What This Changes

The PR imports `toDiscoveryApiKey`, passes `discoveryApiKey ?? toDiscoveryApiKey(apiKey)` into self-hosted provider discovery, and adds tests for discovery credential precedence and marker filtering.

## Best Possible Solution

Land a provider discovery fix that forwards only concrete discovery credentials and otherwise either skips marker-only probes or documents and proves an explicit authless-local discovery contract.

## Reproduction Assessment

Yes, at source level: marker-only `apiKey` values resolve through `toDiscoveryApiKey` to undefined, then the PR still calls the provider builder and the local model discovery path sends no Authorization header. I did not run a live LiteLLM, vLLM, or SGLang endpoint in this read-only review.

## Solution Assessment

No. Filtering fallback credentials is the right primitive, but the patch still needs a settled marker-only behavior: skip unauthenticated marker-backed probes, or explicitly preserve authless local discovery with live proof.
