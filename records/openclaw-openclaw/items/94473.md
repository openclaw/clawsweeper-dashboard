## Summary

Keep open: this is a focused linked bug-fix PR, but the current patch is not merge-ready because it can turn non-secret provider markers or env placeholders into discovery Authorization values, and the supplied proof only exercises a fake provider-builder capture rather than a real self-hosted `/models` endpoint.

## What This Changes

The branch changes self-hosted OpenAI-compatible provider discovery to pass `discoveryApiKey ?? apiKey` into the provider builder and adds regression tests for discovery-key precedence, marker fallback, and missing keys.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless coverage and redacted real `/models` proof.

## Reproduction Assessment

Yes, at source level. Current resolver code can expose a truthy non-secret marker as `apiKey` with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` contract or filter marker values instead of treating non-secret markers as bearer tokens.
