## Summary

Keep open: this PR is an active candidate for the linked provider discovery auth bug, but it is not merge-ready because the fallback conflates concrete discovery credentials with marker/apiKey placeholders, and the proof is unit-test-only.

## What This Changes

The PR changes self-hosted provider discovery to pass `apiKey` into `buildProvider` when `discoveryApiKey` is absent, and adds focused unit tests for that fallback.

## Best Possible Solution

Use only concrete resolved discovery credentials for discovery Authorization, preserve marker-backed catalog/auth semantics, and require focused regression tests plus redacted after-fix provider proof before merge.

## Reproduction Assessment

Yes, at source level. Current main can return a truthy marker or env-var-name `apiKey` with `discoveryApiKey` undefined, and this PR would then send that marker or placeholder into discovery Authorization.

## Solution Assessment

No. The fix is in the right self-hosted discovery area, but `apiKey` is a public marker/placeholder field as well as a credential field; the safer fix must use only concrete discovery credentials or explicitly narrow marker handling.

- **[P1] Keep marker values out of discovery Authorization:** `src/plugins/provider-self-hosted-setup.ts:436`
  - body: `apiKey` is not guaranteed to be a concrete credential here: current resolver paths use it for env-var names, non-env SecretRef markers, OAuth/AWS/local markers, and `custom-local` style placeholders while reserving `discoveryApiKey` for the actual discovery secret. Falling back to `apiKey` lets vLLM/SGLang discovery send headers such as `Bearer custom-local`, `Bearer secretref-managed`, or a plain env-var name, which can fail auth-required servers and alter no-auth local setups. The fix should use only a concrete discovery credential or narrow marker handling with explicit tests.
