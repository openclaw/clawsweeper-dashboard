## Summary

Keep open: the patch targets a real self-hosted provider discovery bug, but the proposed fallback treats public auth markers and env-var placeholders as concrete bearer tokens, which conflicts with the current resolver contract and can regress marker-backed local discovery.

## What This Changes

The PR changes self-hosted provider catalog discovery to pass the configured apiKey when discoveryApiKey is absent, and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery requests credential-aware by using only concrete resolved discovery credentials for Authorization, and handle marker-backed no-auth or auth-required providers through the resolver, config, or doctor path rather than treating markers as bearer tokens.

## Reproduction Assessment

Yes, at source level. Current main can return a truthy marker or env-var-name apiKey with discoveryApiKey undefined, and this PR would then send that marker/env-var placeholder as discovery Authorization.

## Solution Assessment

No. The fix is in the right area but uses the public apiKey marker as a concrete discovery secret, which conflicts with the resolver and marker contracts already present on main.
