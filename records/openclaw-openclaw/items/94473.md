## Summary

Keep open: this PR is a focused candidate for the linked provider discovery auth bug, but the current fallback can pass non-secret marker or env-placeholder values into discovery Authorization and the supplied proof is only focused Vitest output.

## What This Changes

The branch changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into the provider builder and adds regression tests asserting that fallback.

## Best Possible Solution

Keep discovery Authorization built only from concrete resolved discovery credentials, and repair marker-backed auth-required discovery through the resolver or a clearly tested provider-auth contract rather than treating markers as bearer tokens.

## Reproduction Assessment

Yes, at source level. Current main can expose a truthy marker/public `apiKey` with `discoveryApiKey` undefined, and this PR would pass that marker/public value into the provider builder used for `/models` discovery.

## Solution Assessment

No. The fix touches the right self-hosted discovery boundary, but falling back to the public `apiKey` value violates the existing separation between marker values and concrete discovery credentials.
