## Summary

Keep open: the PR changes the right self-hosted discovery boundary, but the fallback can turn non-secret marker values into bearer tokens and the supplied after-fix proof is still a mocked resolution script rather than a real self-hosted `/models` request.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for env-backed, marker-backed, and missing-key cases.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete marker-filtered discovery credentials, or skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted real endpoint proof.

## Reproduction Assessment

Yes, at source level. Current main can carry a truthy non-secret marker as `apiKey`, drop it from `discoveryApiKey`, and then issue `/models` discovery without an Authorization header; this PR would instead send the marker as a bearer value.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fix should preserve the concrete `discoveryApiKey` separation or filter/skip marker-only discovery instead of treating non-secret markers as bearer tokens.
