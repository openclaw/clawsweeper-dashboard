## Summary

Keep open: the PR targets a real provider-discovery auth bug, but the current patch is not merge-ready because it can send non-secret marker or env-placeholder strings as bearer tokens and the supplied proof is a mocked expression capture rather than a real self-hosted `/models` endpoint run.

## What This Changes

This PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for discovery-key precedence, marker fallback, and missing-key behavior.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless coverage and redacted live `/models` proof.

## Reproduction Assessment

Yes, at source level. Current resolvers can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` contract or filter marker values instead of treating non-secret markers as bearer tokens.
