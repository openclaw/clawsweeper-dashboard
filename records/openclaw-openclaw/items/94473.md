## Summary

Keep open: this PR is the active candidate for the canonical provider-catalog auth issue, but it is not merge-ready because the fallback can treat marker or env-placeholder values as bearer credentials and the supplied proof is synthetic rather than a real endpoint run.

## What This Changes

The branch changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for credential precedence, marker fallback, and missing keys.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or skip marker-only discovery, then prove auth-required and authless local behavior with focused tests and real endpoint output.

## Reproduction Assessment

Yes, at source level. Current resolvers can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` contract or filter marker values instead of treating non-secret markers as bearer tokens.
