## Summary

Keep open: the PR targets a real source-reproducible provider discovery bug, but the proposed fallback crosses the current credential boundary by sending public marker values as discovery bearer tokens.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? apiKey` into the provider builder and adds regression tests for that fallback.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or explicitly skip marker-only dynamic discovery, with focused marker/env/SecretRef/authless tests and redacted live `/models` proof.

## Reproduction Assessment

Yes, source-reproducible. Current resolver paths can expose a truthy public marker with `discoveryApiKey` undefined, and the PR head would pass that marker to discovery code that turns any non-empty key into Bearer Authorization.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fix should preserve the concrete `discoveryApiKey` separation instead of treating non-secret markers as bearer tokens.
