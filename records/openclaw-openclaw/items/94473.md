## Summary

Keep open: this PR edits the right self-hosted discovery boundary, but the fallback can send non-secret marker or placeholder values as bearer tokens and the supplied proof is only a fake `buildProvider` capture, not a real `/models` endpoint run.

## What This Changes

The PR changes self-hosted OpenAI-compatible provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for key precedence, marker fallback, and missing keys.

## Best Possible Solution

Use a marker-aware discovery credential selector, or skip marker-only discovery, while preserving concrete env/profile discovery credentials and authless local-provider behavior.

## Reproduction Assessment

Yes, at source level. Current resolvers can expose a truthy public marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right boundary, but the fallback should preserve the concrete discovery credential contract or filter marker values instead of treating non-secret markers as bearer tokens.
