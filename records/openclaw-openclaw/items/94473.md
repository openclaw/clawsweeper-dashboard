## Summary

Keep open: the patch avoids forwarding marker text as a bearer token, but marker-only self-hosted discovery still calls provider builders with no concrete discovery credential, so vLLM/SGLang-style builders can still issue unauthenticated `/models` requests. The PR also still has only mock/unit-style proof for an auth-provider runtime path.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? toDiscoveryApiKey(apiKey)` into provider builders and adds unit coverage for concrete keys and non-secret markers.

## Best Possible Solution

Land a provider discovery fix that forwards concrete discovery credentials and skips marker-only dynamic probes unless an explicitly supported authless-local path is documented, tested, and live-proven.

## Reproduction Assessment

Yes, source-reproducible: on PR head, marker-only `apiKey` with missing `discoveryApiKey` produces `discoveryKey: undefined`, then vLLM/SGLang builders can call `/models` without Authorization. I did not run a live self-hosted endpoint in this read-only review.

## Solution Assessment

No. `toDiscoveryApiKey` is the right primitive for avoiding marker text in bearer headers, but this is not yet the best fix because marker-only discovery should stop before the provider builder or be an explicit authless-local contract with live proof.
