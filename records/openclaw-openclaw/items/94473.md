## Summary

Keep open: the PR avoids forwarding marker text as bearer credentials, but marker-only inputs still invoke self-hosted provider builders with no discovery credential, preserving unauthenticated `/models` probes for vLLM/SGLang-style discovery. The contributor proof is still unit/script-only, so after-fix real endpoint proof is missing.

## What This Changes

The PR changes self-hosted provider discovery to pass `discoveryApiKey ?? toDiscoveryApiKey(apiKey)` into the provider builder, adds marker-matrix tests, and adds a root-level proof script.

## Best Possible Solution

Land a credential-aware self-hosted discovery fix that forwards concrete credentials, skips or explicitly scopes marker-only unauthenticated discovery, and includes focused tests plus redacted live endpoint proof.

## Reproduction Assessment

Yes, at source level: current main can pass a truthy marker as `apiKey`, provide no `discoveryApiKey`, and then local discovery omits Authorization when the builder receives no key. I did not run a live LiteLLM, vLLM, or SGLang endpoint in this read-only review.

## Solution Assessment

No, not yet. `toDiscoveryApiKey` is the right primitive for avoiding marker text in bearer headers, but the PR still needs marker-only discovery policy resolution and real endpoint proof.
