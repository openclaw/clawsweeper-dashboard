- `P2`: This is a bounded provider discovery/auth bug fix with limited blast radius around self-hosted model catalog discovery.

## Summary

Keep open: the PR targets a valid source-level provider-discovery auth bug, but the current patch is not merge-ready because the fallback can turn non-secret marker or env-placeholder values into bearer credentials, and the supplied proof does not exercise a real self-hosted `/models` endpoint.

## What This Changes

The PR changes self-hosted OpenAI-compatible provider discovery to pass `discoveryApiKey ?? apiKey` into `buildProvider` and adds regression tests for discovery key precedence, marker fallback, and missing keys.

## Best Possible Solution

Keep discovery Authorization sourced only from concrete resolved discovery credentials, or skip marker-only discovery, with focused tests and redacted live endpoint proof for auth-required and authless self-hosted providers.

## Reproduction Assessment

Yes, at source level. Current resolver paths can expose a truthy non-secret marker with `discoveryApiKey` undefined, and this PR would pass that marker into discovery code that builds bearer headers from any non-empty key.

## Solution Assessment

No. The PR edits the right self-hosted discovery boundary, but the fallback should preserve the concrete `discoveryApiKey` contract or filter marker values instead of treating non-secret markers as bearer tokens.
