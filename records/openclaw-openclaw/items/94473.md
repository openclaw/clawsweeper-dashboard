## Summary

Needs changes before merge: the branch adds a marker-aware fallback for concrete discovery credentials, but marker-only self-hosted discovery still calls the provider builder without a credential and can still issue the unauthenticated `/models` probe described by the linked bug; the PR also lacks redacted after-fix live endpoint proof.

## What This Changes

The PR routes self-hosted provider discovery through `toDiscoveryApiKey` for `apiKey` fallback and adds tests for discovery key precedence and marker filtering.

## Best Possible Solution

Land a provider discovery fix that forwards concrete discovery credentials and either skips marker-only dynamic probes or preserves a clearly documented, live-proven authless local discovery path.

## Reproduction Assessment

Yes, at source level: marker-only `apiKey` plus missing `discoveryApiKey` flows through the PR to `buildProvider({ apiKey: undefined })`, and the vLLM/SGLang builders then call `/models` without Authorization. I did not run a live LiteLLM/vLLM/SGLang endpoint in this read-only review.

## Solution Assessment

No. Filtering fallback credentials is the right primitive, but this is not yet the best fix because marker-only discovery still needs to stop before the provider builder or be an explicit authless-local contract with live proof.
