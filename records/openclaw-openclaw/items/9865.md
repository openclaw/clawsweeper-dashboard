## Summary

Keep open. Current main and v2026.6.10 still lack first-class provider Batch API routing for delayed/background agent turns; the only Batch API support found is memory-embedding specific, and the closed related reports are duplicate context rather than shipped fixes.

## What This Changes

The issue requests opt-in provider Batch API routing for non-real-time agent work such as cron jobs, subagents, heartbeats, and other background tasks.

## Best Possible Solution

Define an additive delayed/batch agent-turn lifecycle shared by provider runtime and the public Plugin SDK, then expose explicit background opt-ins for cron, sessions_send, sessions_spawn/subagents, and heartbeat/background runs with provider implementations, docs, accounting, cancellation, fallback, and delayed delivery semantics.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken documented behavior. Source inspection shows current main has no delayed/batch agent-turn opt-in or provider lifecycle to reproduce as existing functionality.

## Solution Assessment

No current implementation path is selected. The best fix needs a maintainer-approved generic delayed-turn contract rather than a provider-local or cron-only flag.
