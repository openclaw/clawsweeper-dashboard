## Summary

Current main still lacks delayed Batch API execution for agent turns. Existing background-task tracking helps with durable state, but provider hooks, model APIs, cron payloads, and agent runners remain designed for immediate streaming execution, so this canonical feature needs maintainer contract design rather than closure or an automated fix.

## What This Changes

Add opt-in provider Batch API execution for latency-tolerant cron, heartbeat, and sub-agent background turns.

## Best Possible Solution

Define an additive public delayed-turn lifecycle integrated with the background-task ledger, including durable request identity, polling or wake-up, cancellation, accounting, result delivery, provider and credential eligibility, and explicit fallback semantics; then expose opt-ins for supported background callers and implement OpenAI and Anthropic adapters.

## Reproduction Assessment

Not applicable: this issue requests a new delayed provider execution mode rather than reporting broken existing behavior. Source inspection does provide a high-confidence verification that current main has no such opt-in or lifecycle.

## Solution Assessment

No, a simple cron batch flag is not the best complete solution. The narrowest maintainable design is a generic delayed-turn provider and plugin contract shared by cron, heartbeat, sessions, and subagents, with provider-specific implementations behind that boundary.
