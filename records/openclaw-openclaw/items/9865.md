## Summary

Keep this canonical feature request open: current main and v2026.6.5 still lack delayed Batch API execution for agent turns, while the three narrower related reports were closed in favor of this tracker. The capability is valuable but requires a cross-cutting product and public-API decision before implementation.

## What This Changes

Add opt-in provider Batch API execution for latency-tolerant cron jobs, heartbeats, subagents, and other background agent turns.

## Best Possible Solution

Introduce one additive delayed agent-turn lifecycle through the public provider and Plugin SDK contracts, persist its canonical state in SQLite, and let selected background entry points opt into it with documented provider eligibility, cancellation, accounting, delivery, timeout, and fallback behavior.

## Reproduction Assessment

Not applicable; this requests a new delayed execution mode rather than reporting broken behavior under an existing contract. Source and release inspection clearly establish that the capability is absent.

## Solution Assessment

No, implementing a provider-specific cron flag directly would not be the best solution because heartbeat, sessions, subagents, persistence, and provider lifecycle concerns share the same invariant. A generic delayed-turn contract should precede entry-point and provider implementations.

## Risks / Open Questions

- The durable ownership model for queued provider jobs, session transcripts, cron run state, restart recovery, and eventual delivery is not yet defined.
- Provider eligibility and credential rules differ across OpenAI, Anthropic, OAuth-backed accounts, and API-key configurations.
- Cancellation, tool-loop continuation, usage accounting, timeout behavior, and whether failure should fall back to full-price real-time execution need explicit product semantics.
- Adding independent batch branches to cron, heartbeat, and subagent callers would create competing execution paths unless the provider/runtime contract is designed first.
