## Summary

Keep open. Current main and the latest release still lack first-class Batch API routing for delayed/background agent turns, while prior discussion and related closed reports establish this as the canonical product/API tracker rather than a duplicate or already-shipped fix.

## What This Changes

Request opt-in provider Batch API routing for non-real-time cron, heartbeat, subagent, and background agent work.

## Best Possible Solution

Define an additive delayed-turn lifecycle shared by provider runtime and the Plugin SDK, then expose explicit background opt-ins with provider implementations, docs, accounting, cancellation, fallback, and delivery semantics.

## Reproduction Assessment

Not applicable: this is a feature request for a new delayed provider execution mode, not a broken existing behavior. Source inspection gives high-confidence evidence that current main and v2026.6.8 have no such opt-in or lifecycle.

## Solution Assessment

No, a simple cron or provider-local batch flag is not the best complete solution. The maintainable path is a generic delayed-turn provider and Plugin SDK contract shared by cron, heartbeat, sessions, and subagents.
