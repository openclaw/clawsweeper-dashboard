## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the live discussion plus related closed reports treat this as the canonical enhancement rather than an implemented or duplicate cleanup item.

## What This Changes

Request provider Batch API support for non-real-time background agent turns, with opt-ins for cron jobs, subagents, heartbeat checks, and scheduled maintenance.

## Best Possible Solution

Keep this as the canonical design tracker for an additive delayed/batch agent-turn contract, then implement documented opt-ins across cron, sessions_send, sessions_spawn/subagents, heartbeat/background runs, and provider plugins.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high confidence that cron, sessions_send, sessions_spawn, and heartbeat/background agent turns have no Batch API opt-in or delayed-turn lifecycle today.

## Solution Assessment

No, a one-shot provider-local or cron-only patch is not the best path yet. The maintainable solution needs a reviewed additive provider/runtime and public Plugin SDK contract before OpenAI or Anthropic Batch API adapters are wired in.
