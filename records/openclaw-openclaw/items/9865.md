## Summary

Current main still lacks first-class provider Batch API routing for agent turns: provider execution is stream-oriented, and cron, heartbeat, session, and subagent entry points have no delayed-turn opt-in. The three related reports are closed satellites of this canonical product/API tracker, not implementations, so the feature remains necessary and requires maintainer contract design before coding.

## What This Changes

Add opt-in provider Batch API execution for latency-tolerant cron jobs, heartbeats, session sends, and subagent runs.

## Best Possible Solution

Define one additive, provider-owned delayed-turn lifecycle in the public Plugin SDK with durable job state, then expose explicit opt-ins on the approved background entry points while keeping immediate execution as the default and documenting cancellation, fallback, accounting, and delivery semantics.

## Reproduction Assessment

Not applicable: current main is behaving according to its immediate-turn contracts; this issue requests a new asynchronous provider execution mode rather than reporting a broken documented behavior.

## Solution Assessment

Yes at the architectural boundary, but the detailed proposal is not implementation-ready: a generic provider/Plugin SDK delayed-turn lifecycle is preferable to provider-specific cron logic, and maintainers must choose its public API and product semantics first.
