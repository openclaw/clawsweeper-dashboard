## Summary

Keep open. Current main and the latest release still lack a first-class delayed Batch API lifecycle for background agent turns; the existing batch code is scoped to memory embeddings/media-style paths, and the related cron-specific reports were closed as duplicate context for this canonical tracker.

## What This Changes

The issue requests opt-in provider Batch API routing for non-real-time background agent work such as cron jobs, heartbeats, and sub-agents.

## Best Possible Solution

Define an additive public delayed-turn lifecycle shared by provider runtime and the Plugin SDK, then expose explicit opt-ins for supported background callers with provider implementations, docs, accounting, cancellation, fallback, and delivery semantics.

## Reproduction Assessment

Not applicable: this is a feature request for a new delayed provider execution mode, not a broken existing behavior. Source inspection does give high-confidence evidence that current main has no such opt-in or lifecycle.

## Solution Assessment

No, a simple cron or provider-local batch flag is not the best complete solution. The maintainable path is a generic delayed-turn provider and Plugin SDK contract shared by cron, heartbeat, sessions, and subagents.
