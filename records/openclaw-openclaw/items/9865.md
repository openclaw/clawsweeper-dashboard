## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the issue discussion plus canonical search show the related batch-agent reports were closed as narrower duplicates of this open tracker rather than shipped fixes.

## What This Changes

Add opt-in provider Batch API routing for non-real-time background agent work such as cron jobs, subagents, heartbeat checks, and scheduled maintenance.

## Best Possible Solution

Keep this as the canonical design tracker for an additive delayed/batch agent-turn contract, then implement documented opt-ins across cron, sessions_send, sessions_spawn/subagents, heartbeat/background runs, and provider plugins.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high confidence that cron, sessions_send, sessions_spawn, and heartbeat/background agent turns have no Batch API opt-in or delayed-turn lifecycle today.

## Solution Assessment

No, a one-shot provider-local or cron-only patch is not the best solution yet. The maintainable path needs a reviewed additive provider/runtime and public Plugin SDK contract before OpenAI or Anthropic Batch API adapters are wired in.
