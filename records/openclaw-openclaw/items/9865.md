## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the recent issue discussion already treats this as the canonical tracker while the related closed reports are narrower context rather than shipped fixes.

## What This Changes

Add provider Batch API support for background agent work such as cron jobs, spawned subagents, heartbeat checks, and scheduled maintenance runs.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract, with provider runtime and public Plugin SDK lifecycle support plus protocol/config opt-ins for cron, sessions_send, sessions_spawn/subagents, and heartbeat/background runs.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence current-main verification path: inspect the strict cron/session/subagent schemas and provider runtime hooks, which show no batch opt-in or delayed-turn lifecycle for agent turns.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is a shared provider/runtime and Plugin SDK seam first, not a provider-specific cron flag that bypasses shared cancellation, fallback, accounting, and delivery behavior.
