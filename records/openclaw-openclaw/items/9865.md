## Summary

Keep open. Current main still lacks first-class delayed/batch agent-turn routing; existing batch code is scoped to memory embeddings, and the live discussion plus related closed reports treat this as the canonical enhancement requiring API design.

## What This Changes

Add opt-in Batch API routing for non-real-time background agent work such as cron jobs, spawned subagents, scheduled maintenance, and heartbeat checks.

## Best Possible Solution

Keep this as the canonical design tracker for an additive delayed/batch agent-turn contract, then implement documented opt-ins across cron, sessions_send, sessions_spawn/subagents, heartbeat/background runs, and provider plugins.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main shows the relevant cron and sessions schemas reject unknown batch fields and provider hooks expose immediate stream execution only.

## Solution Assessment

No, this is not best solved as a one-off cron/provider patch. The maintainable path is a reviewed additive provider/runtime and public Plugin SDK contract before OpenAI or Anthropic Batch API adapters are wired in.
