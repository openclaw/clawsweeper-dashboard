## Summary

Keep open. Current main still lacks first-class provider Batch API routing for delayed/background agent turns, and the remaining work is a concrete enhancement that needs maintainer API/product design rather than cleanup closure.

## What This Changes

Request Batch API support for non-real-time background agent work such as cron jobs, subagents, scheduled maintenance, and heartbeat checks.

## Best Possible Solution

Keep this as the canonical design tracker for an additive delayed/batch agent-turn contract, then implement documented opt-ins across cron, sessions_send, sessions_spawn/subagents, heartbeat/background runs, and provider plugins.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high confidence that cron, sessions_send, sessions_spawn, and heartbeat/background agent turns have no Batch API opt-in path today.

## Solution Assessment

No, a one-shot provider-local patch is not the best path yet. The maintainable solution needs a reviewed additive runtime/API contract before OpenAI or Anthropic Batch API adapters are wired in.
