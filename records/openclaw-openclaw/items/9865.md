## Summary

Keep open. Current main and the latest release still lack first-class Batch API or delayed execution for agent turns; existing batch code is scoped to memory embeddings, while this request crosses provider runtime, Plugin SDK, protocol, cron, sessions/subagents, heartbeat, delivery, cancellation, accounting, and auth decisions.

## What This Changes

The issue requests provider Batch API support for delayed/background OpenClaw agent work such as cron jobs, subagents, scheduled maintenance, and heartbeats.

## Best Possible Solution

Keep this as the canonical design tracker for an additive delayed-turn provider/plugin contract shared by cron, heartbeat, sessions, and subagents before provider-specific Batch API adapters are implemented.

## Reproduction Assessment

Not applicable as a bug reproduction: this requests a new delayed provider execution mode. Source inspection gives high-confidence verification that current main has no batch opt-in or delayed-turn lifecycle for agent turns.

## Solution Assessment

No, a cron-only flag would be too narrow. The maintainable solution is a generic delayed-turn lifecycle in provider/runtime and Plugin SDK contracts, then opt-ins for supported background callers.
