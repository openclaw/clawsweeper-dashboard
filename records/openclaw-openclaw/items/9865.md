## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the existing discussion already treats this as the canonical product/API tracker rather than a duplicate or shipped fix.

## What This Changes

Request support for provider Batch APIs as an opt-in route for non-real-time cron, subagent, heartbeat, and scheduled background agent turns.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract across provider runtime, public Plugin SDK, gateway protocol/config opt-ins, provider implementations, and documented delivery semantics.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence current-main verification path: inspect the strict cron/session/subagent schemas and provider runtime hooks, which show no batch opt-in or delayed-turn lifecycle for agent turns.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is a shared provider/runtime and Plugin SDK seam first, not a provider-specific cron flag that bypasses cancellation, fallback, accounting, and delayed delivery behavior.
