## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the existing discussion already treats this as the canonical product/API tracker rather than a duplicate or shipped fix.

## What This Changes

Request support for provider Batch APIs for non-real-time background work such as cron jobs, subagents, and heartbeat checks to reduce cost at acceptable latency.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract across provider runtime, public Plugin SDK, gateway protocol/config opt-ins, provider implementations, and documented delivery semantics.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence current-main verification path: inspect the strict cron/session schemas and provider runtime hooks, which show no batch opt-in or delayed-turn lifecycle for agent turns.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is a shared provider/runtime and Plugin SDK seam first, not a provider-specific cron flag that bypasses shared cancellation, fallback, accounting, and delivery behavior.
