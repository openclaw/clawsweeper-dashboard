## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the existing discussion treats this as the canonical product/API tracker rather than a resolved duplicate.

## What This Changes

Request batch/delayed provider API routing for async-tolerant background agent work such as cron jobs, subagents, and heartbeat checks.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract, then implement provider/plugin runtime support and opt-in cron/session/subagent/heartbeat routing after maintainers choose the lifecycle semantics.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence current-main verification path: inspect the strict cron/session/subagent schemas and provider runtime hooks, which show no batch opt-in or delayed-turn lifecycle for agent turns.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is a shared provider/runtime and Plugin SDK seam first, not a provider-specific cron flag that bypasses shared cancellation, fallback, accounting, and delivery behavior.
