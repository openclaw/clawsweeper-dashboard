## Summary

Current main and v2026.6.5 still lack a delayed provider-turn lifecycle, and no viable open PR owns the work. This remains the canonical feature tracker, but its cross-cutting API and state semantics require maintainer design before implementation.

## What This Changes

Add opt-in provider Batch API execution for latency-tolerant cron, heartbeat, session, and subagent work.

## Best Possible Solution

Define one additive asynchronous provider-turn lifecycle with durable run identity, capability negotiation, cancellation, fallback, accounting, and completion delivery, then add narrowly scoped opt-ins to approved background callers and provider plugins.

## Reproduction Assessment

Not applicable: current main consistently uses the immediate agent-turn path, and the issue requests a new asynchronous provider capability rather than reporting broken behavior.

## Solution Assessment

No, not as isolated per-caller batch flags; the narrowest maintainable solution starts with a shared provider/plugin delayed-turn contract and common run-state semantics.
