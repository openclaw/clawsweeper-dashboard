## Summary

Current main and v2026.6.5 still lack first-class delayed Batch API execution for agent turns. The request remains useful and canonical, but implementation requires a maintainer product/API decision because it crosses public provider contracts, persisted session state, background scheduling, delivery, cancellation, accounting, and fallback behavior.

## What This Changes

Add opt-in provider Batch API routing for latency-tolerant cron jobs, heartbeats, scheduled maintenance, and sub-agent work.

## Best Possible Solution

Define one additive delayed agent-turn lifecycle in the public provider/plugin contract, persist its canonical job and session state in SQLite, then expose narrowly scoped opt-ins for eligible background entry points with explicit cancellation, completion delivery, accounting, timeout, and fallback semantics.

## Reproduction Assessment

Not applicable: this issue requests a new delayed execution capability rather than reporting broken existing behavior. Source and release inspection reliably confirm that the capability does not exist today.

## Solution Assessment

No, the suggested per-cron Boolean alone is not the best complete solution. The maintainable direction is a generic delayed-turn provider lifecycle first, followed by explicit opt-ins at selected background entry points rather than provider-specific routing embedded in cron.
