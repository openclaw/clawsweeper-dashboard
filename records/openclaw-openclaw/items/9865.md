## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the discussion already treats this as the canonical tracker while the related closed reports are narrower context rather than shipped fixes.

## What This Changes

The issue requests opt-in Batch API routing for non-real-time background work such as cron jobs, heartbeats, scheduled maintenance, and background subagents.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract, then implement provider plugins and cron/session/subagent/heartbeat opt-ins on top of that shared lifecycle.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence current-main verification path: inspect the strict cron/session/subagent schemas, heartbeat runtime, and provider hooks, which show no batch opt-in or delayed-turn lifecycle for agent turns.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is an additive provider/runtime and public Plugin SDK seam first, not a one-off OpenAI or Anthropic cron flag that bypasses shared cancellation, delivery, accounting, and fallback behavior.
