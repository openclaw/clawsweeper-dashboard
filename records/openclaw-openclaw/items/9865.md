## Summary

Keep open: current main still lacks first-class delayed/batch routing for background agent turns, and the live discussion already treats this as the canonical product/API tracker rather than a duplicate of the narrower closed reports.

## What This Changes

Request support for routing async/background agent work such as cron jobs, subagents, and heartbeat checks through provider Batch APIs with opt-in latency and cost tradeoffs.

## Best Possible Solution

Keep this as the canonical design tracker for an additive delayed/batch agent-turn contract, then implement documented protocol/config opt-ins and provider adapters after the runtime and SDK shape is agreed.

## Reproduction Assessment

Not applicable; this is an enhancement request rather than a broken existing contract. Source inspection is high-confidence that current main has no accepted batch opt-in for cron, sessions_send, sessions_spawn/subagents, or heartbeat agent turns.

## Solution Assessment

No, not as a one-off cron flag or provider-only adapter. The maintainable solution is a reviewed additive provider-runtime, public Plugin SDK, and gateway protocol contract before OpenAI or Anthropic Batch API execution is wired in.
