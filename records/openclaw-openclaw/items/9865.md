## Summary

Keep open. Current main and v2026.6.8 still lack first-class provider Batch API routing for delayed/background agent turns, and the related closed reports are duplicate context rather than shipped fixes.

## What This Changes

Request Batch API support for non-real-time background agent work such as cron jobs, subagents, scheduled maintenance, and heartbeats.

## Best Possible Solution

Define an additive delayed-turn lifecycle shared by provider runtime and the Plugin SDK, then expose explicit background opt-ins with provider implementations, docs, accounting, cancellation, fallback, and delivery semantics.

## Reproduction Assessment

Not applicable: this is a feature request for a new delayed provider execution mode, not a broken existing behavior. Source inspection gives high-confidence evidence that current main and v2026.6.8 have no such opt-in or lifecycle.

## Solution Assessment

No: a provider-local or cron-only batch flag would be too narrow. The maintainable solution is a generic delayed-turn provider and Plugin SDK contract shared by cron, heartbeat, sessions, and subagents.
