## Summary

Keep open. Current main and v2026.6.8 still lack first-class provider Batch API routing for delayed/background agent turns; existing batch support is memory-embedding scoped, and the related closed reports are duplicate context for this canonical product/API tracker.

## What This Changes

Request to add provider Batch API support for delayed/background work such as cron jobs, heartbeats, scheduled maintenance, and sub-agents.

## Best Possible Solution

Define an additive public delayed-turn lifecycle shared by provider runtime and the Plugin SDK, then expose explicit opt-ins for supported background callers with provider implementations, docs, accounting, cancellation, fallback, and delivery semantics.

## Reproduction Assessment

Not applicable: this is a feature request for a new delayed provider execution mode, not a broken existing behavior. Source inspection gives high-confidence evidence that current main and v2026.6.8 have no such opt-in or lifecycle.

## Solution Assessment

No, a simple cron or provider-local batch flag is not the best complete solution. The maintainable path is a generic delayed-turn provider and Plugin SDK contract shared by cron, heartbeat, sessions, and subagents.
