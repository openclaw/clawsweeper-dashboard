## Summary

Keep open. Current main and the latest release still lack first-class provider Batch API routing for delayed/background agent turns; existing Batch API code is scoped to memory embeddings or media/tool paths, and the related closed reports remain duplicate context for this canonical product/API tracker.

## What This Changes

The issue requests opt-in Batch API routing for non-real-time background agent work such as cron jobs, subagents, scheduled maintenance, and heartbeat checks.

## Best Possible Solution

Keep this as the canonical product/API tracker until maintainers define an additive delayed-turn provider/Plugin SDK contract and wire it through cron, heartbeat, sessions, subagents, provider implementations, docs, and accounting.

## Reproduction Assessment

Not applicable: this is a feature request for a new delayed provider execution mode, not a broken existing behavior. Source inspection gives high-confidence evidence that current main has no such opt-in or lifecycle.

## Solution Assessment

No, a simple cron or provider-local batch flag is not the best complete solution. The maintainable path is a generic delayed-turn provider and Plugin SDK contract shared by cron, heartbeat, sessions_send, sessions_spawn/subagents, delivery, cancellation, and accounting.
