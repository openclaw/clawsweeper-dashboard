## Summary

Keep open. Current main still passes Discord thread starter context into the reply payload for thread messages, the shared reply path can still prepend that starter on follow-up turns when no thread history is present, and an open targeted PR is already paired with this issue but has not merged.

## What This Changes

The issue requests that Discord thread starter context be included only on the first effective thread session turn to avoid repeated prompt contamination and outbound echo risk.

## Best Possible Solution

Update and land #50522, or a narrow replacement, against current main: compute the previous timestamp for the effective Discord thread session before finalizing context, omit `ThreadStarterBody` for existing thread sessions, and add a Discord regression test for an existing thread session. Keep the broader shared reply fallback unchanged unless review proves it creates cross-channel duplication after the adapter-level fix.

## Reproduction Assessment

Yes. The static path is high-confidence: a Discord thread message with an existing effective session still receives `ThreadStarterBody`, and the reply runner can prepend it on a non-new session when thread history is absent; the reporter and linked comments also describe a local reproduction.

## Solution Assessment

Yes, the adapter-level guard matching Slack is the narrowest maintainable fix. A broad shared-session removal would risk changing other channel/media-only fallback behavior, so the Discord PR should be updated around the current context-builder file and verified with a focused regression test.
