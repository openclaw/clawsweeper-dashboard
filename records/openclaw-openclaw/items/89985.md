## Summary

Keep this PR open: the recovery-bundle idea is useful and the implementation has focused tests plus real-environment proof, but the default reapply behavior can carry corrupt or tampered packaged code across an update and needs maintainer direction before merge.

## What This Changes

The PR adds package `dist` content inventory metadata and an update-time local override capture/reapply path for package-manager `openclaw update`.

## Best Possible Solution

Preserve local changes in a recovery bundle and only reapply them when explicitly trusted or opted in, while adding package-acceptance validation for the new content inventory.

## Reproduction Assessment

Yes for the blocking behavior: source inspection shows a mismatched installed `dist` file is saved and copied back after the staged package swap when upstream did not change it. The PR's clean reapply test also proves that default path.

## Solution Assessment

No. The recovery bundle is a good direction, but default reapply treats corruption or tampering the same as an intentional local override; the safer solution needs an explicit trust/opt-in boundary and release-artifact validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not reapply untrusted package mutations by default:** `src/infra/package-local-overrides.ts:424`
  - body: Any hash mismatch in the installed `dist` tree is treated as a local override. If that mismatch is corruption or tampering, `openclaw update` now copies the modified code back into the verified package whenever upstream did not change the file, so the update path no longer reliably repairs corrupt managed installs. Preserve the recovery bundle, but require an explicit trust signal before reapplying packaged code.
  - confidence: 0.87
- **[P2] Validate the new content inventory in package acceptance:** `src/infra/package-dist-inventory.ts:498-500`
  - body: This writes a new package metadata file that the override-preservation path depends on, but the tarball integrity checker still only requires the old path inventory. A release tarball could pass package acceptance without `dist/postinstall-content-inventory.json` or with stale content metadata, silently disabling the new behavior. Please extend `scripts/check-openclaw-package-tarball.mjs` and its tests to require and validate the new inventory for current packages.
  - confidence: 0.78
