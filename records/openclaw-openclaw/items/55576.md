## Summary

Keep open. The Vertex Express config gap is real, but this patch is not merge-ready because the provider-level `api: "google-vertex"` path still does not propagate through model transport resolution, so the documented setup can fall back to the wrong adapter after validation starts accepting it.

## What This Changes

The PR adds `google-vertex` to the model API allowlist and both generated base config schema enum arrays.

## Best Possible Solution

Make `google-vertex` a first-class model API across config validation, generated schema, model transport normalization, provider-level and model-level resolution, focused regression tests, and the changelog before merging.

## Reproduction Assessment

Yes. A static reproduction path is the provider-level config from the PR discussion: after this patch validates `api: "google-vertex"`, current model resolution still drops that value in `normalizeResolvedTransportApi` and can resolve the model with the OpenAI Responses adapter.

## Solution Assessment

No. Adding the enum entries is necessary, but the narrow maintainable fix also needs the runtime transport and resolution path to preserve `google-vertex`, plus tests that cover the provider-level config shape users are being told to use.
