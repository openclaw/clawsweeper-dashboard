## Summary

Keep open: current main still has a cron watchdog path that can return the timeout-shaped cron result while the native Codex missing-tool-result or incomplete-turn error is produced deeper in the embedded/Codex run path, so the cron-visible primary cause is not reliably preserved. The related open PR only preserves provider/model/session attribution on timeout rows and does not fix this cause-preservation bug.

## What This Changes

The issue requests that cron agent failures preserve the underlying native Codex `missing_tool_result` / incomplete-turn cause instead of surfacing only a timeout-shaped cron error.

## Best Possible Solution

Add a focused cron/agent boundary fix so timeout cleanup can preserve or prefer the underlying incomplete-turn / `missing_tool_result` cause when the embedded run has already recorded it, with the cron timeout message kept as fallback.

## Reproduction Assessment

No live current-main repro was run in this read-only review. Source inspection gives a high-confidence path: `executeJobCoreWithTimeout` can return the watchdog timeout result before the embedded Codex result carrying `missing_tool_result` or `incomplete_turn` becomes the cron result.

## Solution Assessment

No, closing this or routing it to the attribution-only PR is not the best solution. The narrow fix belongs at the cron/agent handoff, where a finalized embedded/Codex failure cause should replace or annotate the watchdog timeout when available.
