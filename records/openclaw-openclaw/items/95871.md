## Summary

Current main still has the cron timeout/core-result race described in this issue: `executeJobCoreWithTimeout` can return the timeout-shaped cron error when the timeout marker wins, even though the Codex app-server layer can record a native missing tool-result failure underneath. Newer merged Codex fixes reduce related native-tool false positives and model-fallback confusion, but they do not make the cron-visible diagnostic preserve the underlying `missing_tool_result` cause.

## What This Changes

The issue requests that isolated cron agent failures preserve the native Codex `web_search` missing-tool-result/incomplete-turn cause instead of surfacing only a cron timeout wrapper.

## Best Possible Solution

Add a focused cron/agent boundary fix so timeout cleanup can preserve or prefer an already-recorded embedded incomplete-turn / `missing_tool_result` cause in cron-visible diagnostics, without weakening the existing cron timeout guard.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this read-only review. Source inspection still shows the relevant race: the cron timeout wrapper can be returned before the deeper Codex missing-tool-result result becomes the cron result.

## Solution Assessment

No, closing is not the best way to solve the issue. The narrow maintainable fix is to repair the cron/agent handoff so the real embedded failure cause survives when it is available, while keeping the timeout guard for true stalls.
