## Label Justifications

- `P2`: This is a normal-priority cron scheduler bug with a bounded surface and limited blast radius.
- `impact:other`: The issue can drop a scheduled missed cron catch-up run for a full period after restart.

## Summary

Keep open: current main still has the startup-only exemption, while read RPC and timer maintenance recomputes can still run future-slot repair without a pending deferral marker. The linked open fix PR is the right current owner for the remaining work.

## What This Changes

Cron startup overflow catch-up deferrals should survive read and timer maintenance recomputes until the deferred catch-up tick fires.

## Best Possible Solution

Land or replace the linked fix PR with a canonical in-memory pending catch-up deferral set consulted by every maintenance recompute caller, clearing ids once their deferred slots are due.

## Reproduction Assessment

Yes. Source inspection shows the read and timer maintenance paths still call `recomputeNextRunsForMaintenance` without the startup-only skip set, matching the reporter's list/status clobber scenario; I did not execute the repro because this review is read-only.

## Solution Assessment

Yes. Moving the exemption into service state is the narrow maintainable fix because the invariant belongs to all maintenance recompute callers, not only `start()`'s immediate pass.
