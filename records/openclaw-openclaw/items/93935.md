## Label Justifications

- `P2`: The bug can silently skip a deferred missed cron run after restart, but it is limited to the startup overflow catch-up window.
- `impact:other`: The concrete impact is cron scheduler reliability, which is meaningful but outside the owned message, auth, session, crash, security, and data-loss labels.

## Summary

Keep open: current main and the latest release still only protect startup overflow catch-up deferrals with the startup-local skip set, while read/status and timer maintenance recomputes can still run without that exemption. The issue has a live closing PR, so it should stay open until https://github.com/openclaw/openclaw/pull/94022 lands or is replaced.

## What This Changes

The issue reports that cron startup overflow catch-up deferrals can be clobbered by read/status and maintenance recomputes before the deferred run fires.

## Best Possible Solution

Land one canonical fix that stores pending startup catch-up deferral IDs in `CronServiceState`, exempts them across every maintenance recompute until the deferred slot fires, then close or supersede duplicate fix attempts.

## Reproduction Assessment

Yes. Current main still has read/status and timer maintenance recompute paths that call `recomputeNextRunsForMaintenance` without the startup deferral skip set; I did not execute the repro because this review is read-only.

## Solution Assessment

Yes. Moving the pending deferral exemption into shared cron service state is the narrow maintainable fix because all later maintenance recompute callers share the same future-slot repair invariant.
