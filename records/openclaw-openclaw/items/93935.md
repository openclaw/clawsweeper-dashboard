- `impact:other`: The issue can drop a scheduled missed cron catch-up run for a full period after restart.

## Summary

Keep open: current main still protects startup overflow catch-up deferrals only in `start()`'s immediate maintenance pass, while read RPCs and timer maintenance paths can run future-slot repair without that exemption.

## What This Changes

The issue reports that cron startup overflow catch-up deferrals can be overwritten by later maintenance recomputes such as read RPCs, timer finalization, or empty-due ticks.

## Best Possible Solution

Add a canonical in-memory pending catch-up deferral state used by every maintenance recompute caller, clearing ids when the deferred slot is due, while preserving ordinary stale-future cron repair for unrelated jobs.

## Reproduction Assessment

Yes, source-reproducible: after startup defers an overflow job, `cron list`/`status` style reads and timer finalization can call maintenance recompute without the startup-only skip set. I did not execute the provided repro because the review is read-only.

## Solution Assessment

Yes: moving the exemption into service state is the maintainable boundary because all maintenance recompute callers share the same future-slot repair invariant.
