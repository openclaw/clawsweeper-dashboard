## Summary

Keep open: current main still has the startup-local deferral skip set, but read/status and timer maintenance recomputes can still run without that exemption. The issue also has an open clean closing PR, so it should remain the canonical tracker until that PR lands or is replaced.

## What This Changes

The issue reports that startup overflow cron catch-up deferrals can be advanced to the next natural schedule by read/status or timer maintenance recomputes before the staggered catch-up fires.

## Best Possible Solution

Keep this issue open as the canonical tracker and review or land https://github.com/openclaw/openclaw/pull/94022, or replace it with an equivalent state-owned deferral marker that covers every maintenance recompute caller.

## Reproduction Assessment

Yes, as source-reproducible: current main still has read/status and timer maintenance recompute paths that call `recomputeNextRunsForMaintenance` without the startup deferral skip set. I did not execute the repro because this sweep is read-only.

## Solution Assessment

Yes in direction: moving the pending deferral exemption into shared cron service state is the narrow maintainable fix because all later maintenance recompute callers share the same future-slot repair invariant. The open PR still needs normal PR review before merge.
