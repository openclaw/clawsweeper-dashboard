## Summary

Keep open: current main still only protects startup overflow catch-up deferrals through the startup-local skip set, while read/status and timer maintenance recomputes can still run without that exemption. The issue also has an open closing PR, so it should remain open until https://github.com/openclaw/openclaw/pull/94022 lands or is replaced.

## What This Changes

The issue reports that cron startup overflow catch-up deferrals can be advanced to the next natural schedule by read/status or timer maintenance recomputation before the deferred catch-up fires.

## Best Possible Solution

Land one canonical fix that stores pending startup catch-up deferral IDs in `CronServiceState`, exempts them across every maintenance recompute until the deferred slot fires, and then close or supersede duplicate fix attempts.

## Reproduction Assessment

Yes, as source-reproducible: current main still has read/status and timer maintenance recompute paths that call `recomputeNextRunsForMaintenance` without the startup deferral skip set. I did not execute the repro because this review is read-only.

## Solution Assessment

Yes: moving the pending deferral exemption into shared cron service state is the narrow maintainable fix because all later maintenance recompute callers share the same future-slot repair invariant. Continuing to thread one-off skip sets only through selected callers is too narrow.
