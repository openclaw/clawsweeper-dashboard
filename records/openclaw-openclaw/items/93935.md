## Summary

Keep open: current main and the latest release still have the startup-local deferral skip only, while read and timer maintenance paths can recompute without that exemption. The issue is already paired with an open closing PR, so it should stay open until that candidate lands or is explicitly replaced.

## What This Changes

The issue requests preserving startup overflow cron catch-up deferrals across read RPC and timer maintenance recomputes so missed runs are not advanced to the next natural schedule slot.

## Best Possible Solution

Land or explicitly replace the open cron service fix so pending startup catch-up deferral IDs are protected by one service-state mechanism across every maintenance recompute caller.

## Reproduction Assessment

Yes for source reproduction: current main still has read and timer maintenance callers that invoke recomputeNextRunsForMaintenance without the startup-local skip set. I did not execute the repro test because this review is read-only.

## Solution Assessment

Yes, the best fix is the state-level deferral exemption described by the issue and implemented by the open candidate PR. The existing one-call-site skip set is already too narrow for the shared maintenance recompute invariant.
