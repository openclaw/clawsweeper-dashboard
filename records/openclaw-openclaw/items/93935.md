## Summary

Keep open: current main still lacks a state-owned pending catch-up deferral set, so read/status and timer maintenance recomputes can still run future-slot repair without the startup-only exemption. Several open linked PRs now target the bug, including https://github.com/openclaw/openclaw/pull/94346, so the issue should stay open until one lands.

## What This Changes

The issue requests preserving startup overflow cron catch-up deferrals across read RPCs and timer maintenance recomputes until their staggered catch-up slot fires.

## Best Possible Solution

Land one narrow fix PR that moves the deferral exemption into `CronServiceState`, applies it to every maintenance recompute caller, clears IDs when their deferred slot is due, and covers read RPC plus timer recompute regressions.

## Reproduction Assessment

Yes, source-reproducible: after startup defers an overflow job, current main lets read/status and timer maintenance callers run future-slot repair without the startup-only skip set. The reporter also supplied an in-process `CronService` repro, but I did not run it in this read-only review.

## Solution Assessment

Yes, the suggested state-owned pending deferral set is the narrow maintainable fix because all maintenance recompute callers share the same future-slot repair invariant. Reviewing and landing one linked PR is better than starting another repair lane.
