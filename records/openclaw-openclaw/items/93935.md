## Label Justifications

- `P2`: This is a bounded cron scheduler correctness bug with limited blast radius.
- `impact:other`: The issue can silently drop a scheduled cron catch-up run for a full schedule period after restart.

## Summary

Keep open: current main still preserves startup overflow catch-up deferrals only through `start()`'s immediate post-catchup maintenance pass, while read/status and timer maintenance recomputes can still run future-slot repair without that exemption. Several linked open PRs are active candidate fixes, so the issue should close only after one lands.

## What This Changes

The issue requests preserving startup overflow cron catch-up deferrals across every maintenance recompute path until the staggered catch-up slot fires.

## Best Possible Solution

Review and land the strongest existing candidate that centralizes pending-deferral protection, expires markers after the catch-up slot, and proves list/status plus timer recompute coverage.

## Reproduction Assessment

Yes. Current main still has unguarded read/status and timer maintenance recompute callers, and the helper only preserves startup deferrals when an explicit skip set is passed; I did not run tests because this review is read-only.

## Solution Assessment

Yes in direction, but not as another new repair branch. The best solution is to review and land one existing linked PR that makes the deferral skip a shared cron-service invariant with expiry and regression coverage.
