## What This Changes

The issue reports that startup overflow cron catch-up deferrals can be clobbered by later maintenance recomputes such as cron list/status before the deferred run fires.

## Best Possible Solution

Review and land one correct linked fix PR that preserves deferred catch-up IDs across all maintenance recompute callers while keeping ordinary stale-future repair intact.

## Reproduction Assessment

Yes. Source inspection shows current main still calls `recomputeNextRunsForMaintenance(state)` from read and timer maintenance paths without the startup deferral skip set; I did not run the reproducer because this is a read-only review.

## Solution Assessment

Yes in shape. Persisting the pending deferral exemption in `CronServiceState` and using it from all recompute callers is the narrow owner-boundary fix, provided the selected PR clears the state safely and preserves ordinary stale-future repair.
