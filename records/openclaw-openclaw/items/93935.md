## Label Justifications

- `P2`: A deferred missed cron run can be silently skipped after restart, but the blast radius is limited to startup overflow catch-up windows.
- `impact:other`: The concrete impact is cron scheduler reliability, which is meaningful but outside the owned message, auth, session, crash, security, and data-loss labels.

## Summary

Keep open: current main and the latest release still lack a durable pending catch-up deferral marker, while an open closing PR already owns the canonical fix path. Closing this issue before that PR lands would orphan the remaining cron scheduler bug.

## What This Changes

The issue requests that startup overflow cron catch-up deferrals survive read/status and timer maintenance recomputes until the staggered catch-up slot fires.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/94022 or an equivalent canonical fix that stores pending startup catch-up deferral IDs in `CronServiceState`, exempts them across all maintenance recomputes, and clears them after the deferred slot fires.

## Reproduction Assessment

Yes. Current main is source-reproducible because read/status and timer maintenance recompute paths can call `recomputeNextRunsForMaintenance` without the startup-local deferral skip set before the deferred slot fires.

## Solution Assessment

Yes. Moving pending deferral IDs into shared cron service state is the narrow maintainable fix because all affected maintenance recompute callers share the same future-slot repair invariant.
