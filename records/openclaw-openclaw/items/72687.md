## Summary

Keep open. The reported OpenAI `gpt-image-2` timeout problem is partly covered on current main by provider-owned OpenAI defaults, but this PR’s runtime-wide fallback is not safe to merge because it would collapse an omitted timeout into an explicit `180_000` before providers can apply endpoint-specific defaults such as Azure OpenAI’s longer 600s budget.

## What This Changes

The PR adds a 180000 ms image-generation runtime fallback, always forwards an effective timeout to providers, and adds runtime tests for default/config/per-request timeout precedence.

## Best Possible Solution

Keep timeout ownership at the provider or shared provider-helper boundary: preserve `undefined` from the runtime when users did not set a timeout, add any broader 180s fallback only for providers that lack a more specific default, cover Azure/runtime interaction in tests, and add the required changelog entry if new user-facing behavior remains.

## Reproduction Assessment

Yes. The regression path is reproducible by inspection against current main: the runtime currently preserves an omitted timeout, while the PR would pass `180_000` before OpenAI/Azure provider code can choose its tested `600_000` Azure default.

## Solution Assessment

No. The PR’s runtime-wide fallback is not the best fix because image providers already own endpoint-specific timeout policy; a safer solution keeps omitted caller/config timeouts distinguishable from explicit overrides and applies any generic fallback only where no provider default exists.
