## Summary

Close as implemented on main. The reported OpenAI `gpt-image-2` timeout problem is already handled by shipped provider-owned defaults and existing configurable `imageGenerationModel.timeoutMs`; the PR’s runtime-wide 180s fallback is unnecessary and would override provider-specific defaults such as Azure’s 600s image timeout.

## What This Changes

The PR adds a runtime-wide 180000 ms image-generation fallback timeout and tests runtime timeout precedence when per-call and config values are absent or overridden.

## Best Possible Solution

Keep timeout ownership at the provider/helper boundary: preserve omitted runtime timeouts, retain shipped OpenAI/Codex and Azure defaults, use per-call or config `timeoutMs` for slow backends, and handle any broader non-OpenAI default change as a separate provider-policy discussion with targeted tests.

## Reproduction Assessment

Yes. The reported OpenAI `gpt-image-2` path is reproducible by source and test inspection: current main sends an omitted timeout through to the OpenAI provider, which applies the shipped 180s default, while the PR would turn that omission into a runtime-selected 180s value for every provider.

## Solution Assessment

No. The PR is not the best fix because it moves timeout policy into the core runtime and erases provider-specific omitted-timeout behavior; the current provider-owned implementation is narrower and already shipped for the reported OpenAI path.
