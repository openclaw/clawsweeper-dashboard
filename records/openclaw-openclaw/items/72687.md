## Summary

Keep this PR open for maintainer judgment, but do not merge it as-is. Current main already handles the reported slow OpenAI/Codex image path with provider-owned defaults, while the PR’s runtime-wide fallback would override provider-specific omitted-timeout behavior and would shorten the shipped Azure OpenAI default from 600s to 180s.

## What This Changes

The PR adds a core image-generation runtime default timeout of 180000 ms, passes that value to providers when no per-call or config timeout is present, and adds runtime tests for fallback and precedence behavior.

## Best Possible Solution

Keep omitted `timeoutMs` distinguishable from explicit caller/config overrides at the runtime boundary. If maintainers want a broader image-generation fallback, add it through provider-owned defaults or an explicit provider metadata/helper contract, with Azure regression coverage and a changelog entry for any user-facing timeout change.

## Reproduction Assessment

Yes. By inspection, an Azure-configured OpenAI image request with no `timeoutMs` reaches provider code as omitted on current main and gets the tested 600000 ms provider default; the PR would pass 180000 from the runtime before the provider can apply that default.

## Solution Assessment

No. A runtime-wide 180s fallback is not the narrowest maintainable fix because image providers already own endpoint-specific timeout policy, including the shipped Azure 600s default.
