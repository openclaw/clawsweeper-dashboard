## Summary

Keep open. The PR is not landable as written because it converts omitted image-generation timeouts into a core 180s value before providers can apply their own defaults, which would regress the shipped Azure OpenAI 600s default; it also lacks the required changelog entry and GitHub reports the branch dirty. https://github.com/openclaw/openclaw/pull/72687

## What This Changes

The PR adds a 180000 ms fallback in the core image-generation runtime and tests omitted, config, and per-request timeout precedence.

## Best Possible Solution

Keep omitted timeoutMs distinguishable at the runtime boundary and apply longer defaults in provider-owned code or a typed provider-default seam, preserving Azure's 600s policy and documenting any user-facing timeout change.

## Reproduction Assessment

Yes. By inspection, current main leaves timeoutMs undefined when caller/config omit it, while the PR would pass 180000 to every provider; an Azure OpenAI image request through the runtime would therefore bypass the provider's tested 600000 ms default.

## Solution Assessment

No. A runtime-wide fallback is not the narrowest maintainable fix because image providers already own endpoint-specific timeout defaults, and the current diff would shorten an existing shipped Azure default.
