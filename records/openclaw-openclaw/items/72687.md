## Summary

Keep open. Current main already has shipped provider-owned defaults for the stated OpenAI gpt-image-2 path, but this PR is not merge-ready because its runtime-wide 180s fallback would override provider-specific timeout policy, including Azure OpenAI's tested 600s default. The remaining decision is whether non-OpenAI providers that still inherit shorter HTTP defaults should receive a generic fallback, and where that policy belongs.

## What This Changes

The PR adds a core image-generation runtime default timeout of 180000 ms and tests that omitted timeouts use that fallback while per-call timeoutMs still takes precedence.

## Best Possible Solution

Keep omitted timeoutMs distinguishable from explicit caller/config overrides at the runtime boundary. Apply longer defaults in provider-owned code or a shared provider-helper contract that can preserve endpoint-specific policies, add Azure/runtime interaction coverage, and include the required changelog entry if user-facing timeout behavior changes.

## Reproduction Assessment

Yes. By inspection, current main omits timeoutMs when the caller/config omit it, while the PR would pass 180000 to every provider; an Azure OpenAI image request through the runtime would therefore bypass the provider's tested 600000 ms default.

## Solution Assessment

No. A runtime-wide fallback is not the narrowest maintainable fix because image providers already own endpoint-specific timeout defaults; a provider-owned or metadata-backed fallback would avoid shortening Azure and other provider-local defaults.
