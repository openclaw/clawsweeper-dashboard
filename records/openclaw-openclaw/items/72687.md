## Summary

Keep open. The PR addresses a real timeout class, but the current diff is not mergeable because it turns an omitted runtime timeout into an explicit 180000 ms provider timeout for every provider, which bypasses provider-owned defaults such as Azure OpenAI's current 600000 ms image timeout. Current main already preserves omitted runtime timeouts, documents/configures per-call and config timeouts, and has OpenAI provider-specific defaults, so the remaining work is to revise the PR rather than close it as implemented.

## What This Changes

The PR adds a 180000 ms fallback timeout in the shared image-generation runtime and adds runtime tests for default, config, and per-request timeout precedence.

## Best Possible Solution

Keep the provider boundary intact: preserve undefined runtime timeouts so providers can apply endpoint-specific defaults, adjust only provider/helper defaults that genuinely still fall through to the guarded 60000 ms HTTP default, add targeted tests for both the intended slower-backend path and Azure no-timeout behavior, and add the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The introduced regression can be reproduced by applying the PR and exercising the existing Azure OpenAI image-generation no-timeout path: current main expects 600000 ms when timeoutMs is omitted, while the PR would pass 180000 before the provider can select the Azure default.

## Solution Assessment

No. A shared runtime-wide fallback is not the best fix because image providers already own endpoint-specific timeout policy; the safer fix is provider-local or helper-local defaulting that does not erase the distinction between omitted timeout and explicit override.
