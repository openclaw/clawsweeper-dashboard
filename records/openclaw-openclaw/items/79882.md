- **[P2] Recompute shared runtime TTLs after detaching a session:** `src/agents/pi-bundle-mcp-runtime.ts:687-690`
  - body: When a shared runtime keeps other attached sessions, this path returns without updating `idleTtlMsByRuntimeKey`. ## Repair Work Prompt

On openclaw/openclaw PR https://github.com/openclaw/openclaw/pull/79882, fix the shared bundled MCP runtime TTL bookkeeping without changing the default `session` behavior or the MCP catalog/connect/listTools logic. The bug: in `mcp.runtimeScope: "shared"`, `idleTtlMsByRuntimeKey` stores the max/zero TTL ever attached, but `disposeSession`/key-move detach does not recompute after a session leaves. Track each attached session's TTL, recompute the effective runtime TTL from currently attached sessions after detach/key changes, preserve the intended rule that `0` means unbounded while that session remains attached, and add regression coverage where a zero-TTL or long-TTL session detaches while a finite/default-TTL session remains and the runtime later sweeps according to the remaining session config. Do not add another config option, do not change docs beyond any necessary wording correction, and do not touch unrelated gateway or MCP transport code.
