## Summary

Keep open: this is a plausible opt-in performance feature with supplied Docker benchmark proof, but the patch regresses existing TTL update semantics and still needs public docs plus maintainer review for cross-session isolation.

## What This Changes

The PR adds an opt-in `mcp.runtimeScope` setting, refactors bundled MCP runtime caching to support shared workspace/config-keyed runtimes, and updates schema/help plus runtime tests.

## Best Possible Solution

Preserve existing session-scope TTL behavior, document `mcp.runtimeScope` in the public MCP/config docs with isolation guidance, then have maintainers decide whether to accept the opt-in shared runtime mode.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows current main applies the caller's TTL before sweeping, while the PR sweeps first. For the requested performance feature itself, reproduction is not a bug repro; the PR body supplies real benchmark output for the intended behavior.

## Solution Assessment

No as currently written: the shared-runtime direction is plausible, but the implementation should keep current TTL semantics before merge. A safer path is to apply the current TTL before sweeping an attached runtime, add a regression test for TTL changes, and update public docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Apply the current TTL before sweeping:** `src/agents/pi-bundle-mcp-runtime.ts:533`
  - body: Current main updates the existing session runtime's idle TTL before `sweepIdleRuntimes()`. This patch sweeps first, so a caller that raises `mcp.sessionIdleTtlMs` or sets it to `0` can still have its existing runtime evicted under the stale shorter TTL before the new config is applied, breaking the claimed session-scope compatibility and forcing an avoidable MCP cold start.
  - confidence: 0.88
- **[P3] Document the new runtime scope publicly:** `src/config/types.mcp.ts:44`
  - body: This adds a user-facing `mcp.runtimeScope` config key, but the public MCP/config docs still describe only session-scoped bundled MCP runtimes and `mcp.sessionIdleTtlMs`. Please update the MCP and configuration-reference docs so users see how to enable shared scope and when the single-tenant isolation caveat applies.
  - confidence: 0.82
