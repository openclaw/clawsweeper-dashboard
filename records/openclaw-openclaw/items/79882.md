## Summary

Keep open: this PR has useful real behavior proof and the earlier JSDoc finding was addressed, but current head still has two runtime-manager edge-case bugs around in-flight key switches and shared TTL merging. Maintainers also need to accept the opt-in cross-session MCP process-sharing model before merge.

## What This Changes

The PR adds an opt-in `mcp.runtimeScope` config setting to share bundled MCP runtimes by workspace and MCP server fingerprint, with schema, docs, and regression tests.

## Best Possible Solution

Keep the default `session` isolation, repair the shared-runtime manager edge cases, then land the opt-in shared scope only with maintainer approval for the isolation tradeoff.

## Reproduction Assessment

Not applicable as a bug reproduction: this is an opt-in performance feature. The PR body supplies Docker benchmark output for the intended shared-runtime behavior, and current main source inspection confirms the existing runtime cache is session-keyed.

## Solution Assessment

No, not as currently written: the runtime-key/ref-count approach is a narrow maintainable direction, but the current implementation should first fix in-flight stale-runtime disposal and `sessionIdleTtlMs: 0` merging semantics. After those repairs, the remaining decision is maintainer acceptance of the shared-isolation model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Dispose the previous in-flight runtime before switching keys:** `src/agents/pi-bundle-mcp-runtime.ts:573-579`
  - body: When a session already attached to an in-flight runtime asks for a different workspace, fingerprint, or scope, this branch deletes the old `createInFlight` entry but only disposes a runtime already present in `runtimesByKey`. The pending promise can still resolve and store the stale runtime under `previousRuntimeKey` with no attached sessions, leaving an orphan MCP child process until a later sweep, and possibly longer when no sweep timer is armed. Please await and dispose the previous in-flight runtime before moving the session to the new key.
  - confidence: 0.88
- **[P2] Preserve disabled TTL when sharing runtimes:** `src/agents/pi-bundle-mcp-runtime.ts:453-458`
  - body: `sessionIdleTtlMs: 0` means idle eviction is disabled, but this merge treats TTLs as plain numbers. If one attached session disables eviction and another matching workspace/server config later attaches with a nonzero TTL, the nonzero value wins and the shared runtime can be swept while the zero-TTL session is still attached. Handle `0` as unbounded when merging shared runtime TTLs.
  - confidence: 0.82
