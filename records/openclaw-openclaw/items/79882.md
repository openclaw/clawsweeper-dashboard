## Summary

Keep open: the prior blocking review finding still applies after checking current main and the PR head, and the new public config key still lacks public docs even though the performance proof is sufficient.

## What This Changes

The PR adds an opt-in `mcp.runtimeScope` setting, refactors bundled MCP runtime caching to support shared workspace/config-keyed runtimes, and updates schema/help plus runtime tests.

## Best Possible Solution

Preserve existing session-scope TTL behavior, document `mcp.runtimeScope` in the public MCP/config docs with isolation guidance, then have maintainers decide whether to accept the opt-in shared runtime mode.

## Reproduction Assessment

Yes for the blocking patch regression: source inspection shows current main applies the caller's TTL before sweeping, while the PR sweeps first. For the requested performance feature itself, reproduction is not a bug repro; the PR body supplies real benchmark output for the intended behavior.

## Solution Assessment

No as currently written: the shared-runtime direction is plausible, but the implementation should keep current TTL semantics before merge. A safer path is to apply the current TTL before sweeping an attached runtime, add regression coverage, and update public docs.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.82
