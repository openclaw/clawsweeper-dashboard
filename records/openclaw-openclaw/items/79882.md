## Summary

Keep open: the shared-runtime direction is useful and has real benchmark proof, but this branch is not merge-ready because the shared TTL bookkeeping still has a detach bug, the primary config docs understate the isolation tradeoff, the branch conflicts with current main, and maintainers need to choose the canonical path versus the newer same-direction PR.

## What This Changes

The PR adds an opt-in `mcp.runtimeScope` config that shares bundled MCP runtimes across sessions with the same workspace/config fingerprint, updates config schema/help/docs, and adds runtime manager tests.

## Best Possible Solution

Land one maintainer-approved shared-runtime implementation with `session` as the default, correct per-session TTL recomputation after detach, explicit trusted single-tenant isolation docs, conflict-free exact-head validation, and a clear branch choice versus https://github.com/openclaw/openclaw/pull/95591.

No, not as-is. Workspace/config-keyed sharing in the runtime manager is the right implementation layer for the fan-out, but this branch still needs TTL recomputation after detach, stronger isolation docs, conflict resolution, and maintainer acceptance of the new config/security boundary.

Full review comments:

- **[P2] Recompute shared runtime TTL after detach:** `src/agents/agent-bundle-mcp-runtime.ts:1184-1187`
  - body: When a shared runtime has one attached session with `sessionIdleTtlMs: 0` and another with a finite TTL, detaching the unbounded session returns while `idleTtlMsByRuntimeKey` remains `0`. The remaining finite-TTL session then keeps an unbounded runtime until final detach; track per-session TTLs or recompute the runtime TTL from remaining attachments before returning.
  - confidence: 0.86
- **[P2] Warn config readers about shared MCP isolation:** `docs/gateway/configuration-reference.md:179-184`
  - body: The primary configuration reference introduces `"shared"` without saying it is for trusted single-tenant deployments or that MCP clients, child processes, catalog state, server-side state, and failure/backoff state can cross sessions. Add that warning here so operators see the isolation boundary before enabling the new config.
  - confidence: 0.85
