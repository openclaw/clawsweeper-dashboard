## Summary

Keep open: the feature is plausible and well-proved, but the current PR head introduces two cleanup regressions in the shared runtime manager that should block merge until repaired.

## What This Changes

The PR adds opt-in `mcp.runtimeScope: "shared"` support for bundled MCP runtimes, including runtime-key ref counting, config schema/help/docs updates, and focused tests.

## Best Possible Solution

Repair the shared-runtime bookkeeping defects, keep `session` as the default, and land only after maintainer approval of the opt-in isolation model.

## Reproduction Assessment

Yes for the review findings: source inspection gives concrete paths for attaching shared sessions with different TTLs, detaching the long/zero-TTL owner, then sweeping, and for recreating a stale runtime after binding a session key. As an original user bug reproduction, this PR is an opt-in performance feature rather than a regression report.

## Solution Assessment

No, not as currently written: the cache/ref-count shape is reasonable, but it needs per-attached-session TTL recomputation and session-key preservation before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Recompute shared TTLs after detach:** `src/agents/pi-bundle-mcp-runtime.ts:687-690`
  - body: When a shared runtime has sessions with different TTLs, detaching the session that contributed the max TTL does not lower `idleTtlMsByRuntimeKey` for the remaining sessions. In particular, a `sessionIdleTtlMs: 0` session can detach and leave the runtime unbounded forever even though only finite-TTL sessions remain, so idle MCP children can be kept alive indefinitely. Track TTLs per attached session or recompute the runtime TTL whenever a session detaches.
  - confidence: 0.91
- **[P2] Preserve session-key cleanup after recreation:** `src/agents/pi-bundle-mcp-runtime.ts:609`
  - body: `getOrCreate` binds `params.sessionKey` before checking for a stale existing runtime, but this new eviction helper forgets all keys for the session and the replacement runtime is attached without rebinding the key. After a workspace/config drift recreation, `retireSessionMcpRuntimeForSessionKey` can no longer resolve the runtime, so subagent or one-shot cleanup may miss it until idle eviction. Rebind the session key after stale eviction or avoid deleting the current key during same-session replacement.
  - confidence: 0.86
