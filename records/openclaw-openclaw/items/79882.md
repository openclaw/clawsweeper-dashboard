## Summary

Keep open: the shared MCP runtime idea is still unique and current main does not implement it, but the branch is dirty against current main and the new opt-in sharing mode needs clearer isolation documentation before merge.

## What This Changes

The PR adds an opt-in `mcp.runtimeScope: "session" | "shared"` config mode for reusing bundled MCP runtimes across sessions with matching workspace and MCP server config, plus docs, schema metadata, and runtime-manager tests.

## Best Possible Solution

If maintainers want this core feature, land it as an explicitly opt-in single-tenant runtime mode after rebasing onto current main and documenting the isolation tradeoff everywhere the option is introduced.

## Reproduction Assessment

Yes, from source: current main keys the bundled MCP runtime cache by `sessionId`, and the nested-agent cleanup path retires by session key after each step. I did not rerun the contributor's Docker benchmark, but the current code supports the repeated cold-start behavior the PR targets.

## Solution Assessment

Unclear: shared scope is a plausible single-tenant optimization, but it is not merge-ready until the branch is rebased and the security/isolation tradeoff is documented at the primary config surface. A safer final shape is default session isolation with an explicit, well-documented opt-in shared mode.
