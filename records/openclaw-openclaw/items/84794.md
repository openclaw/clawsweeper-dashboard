## Summary

Keep open for maintainer handling: the patch is a focused bug fix with sufficient real Gateway proof and no blocking code findings, but it intentionally expands `deleteAfterRun` cleanup to isolated cron terminal paths that currently retain session/transcript state, so the existing human-review gate should be cleared only by a maintainer.

## What This Changes

The PR extracts isolated cron delete-after-run cleanup into a shared helper, calls it from the isolated runner `finally` block, preserves delivery-dispatch cleanup, and adds delivery-none regression coverage.

## Best Possible Solution

Land the focused finalizer cleanup after a maintainer accepts the `deleteAfterRun` session-state semantics, while keeping broader hook/session retention policy in separate issues.

## Reproduction Assessment

Yes, at source level. Current main creates run-scoped isolated cron sessions, but delete-after-run cleanup lives in delivery-dispatch paths, so delivery-none and runner-error paths can bypass immediate cleanup.

## Solution Assessment

Yes. Reusing the existing `sessions.delete` cleanup contract from a shared helper and invoking it from the isolated runner finalizer is the narrowest maintainable fix; the remaining question is maintainer acceptance of the session-state semantics, not a code repair.
