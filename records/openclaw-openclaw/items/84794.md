Keep open: this is a focused, still-useful bug-fix PR with sufficient real Gateway proof and no blocking code findings, but it carries a maintainer-visible session-state semantics change and already has the `clawsweeper:human-review` gate.

## What This Changes

This PR extracts isolated cron delete-after-run cleanup into a shared helper, calls it from the isolated runner finalizer, keeps delivery-dispatch cleanup through that helper, and adds delivery-none regression coverage.

## Best Possible Solution

Land this focused finalizer cleanup once maintainers accept `deleteAfterRun` as explicit permission to delete isolated cron run session and transcript state; keep hook cleanup and failed-job deletion policy separate.

## Reproduction Assessment

Yes, at source level. Current main creates run-scoped isolated cron sessions, `delivery.mode: "none"` bypasses delivery cleanup, and the runner finalizer currently disposes context without delete-after-run session cleanup.

## Solution Assessment

Yes, with maintainer acceptance of the retention semantics. Reusing the existing Gateway-backed cron cleanup from the runner finalizer is the narrow maintainable fix because delivery dispatch cannot cover delivery-none or runner-error terminal paths.
