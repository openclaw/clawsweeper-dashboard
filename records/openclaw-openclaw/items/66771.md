## Summary

Keep open. Current main still has a plausible MSTeams mixed thread-session key path: the handler mutates the resolved route, the thread helper appends to whatever base key it receives, and the route cache can retain the mutated miss result for later thread messages in the same channel.

## What This Changes

MSTeams should avoid producing mixed double-thread session keys when an old thread-qualified session is reselected before a new channel thread is qualified.

## Best Possible Solution

Fix the MSTeams-owned thread-session path so it derives a thread-qualified key from a clean immutable base route, strips any existing generic `:thread:` suffix before appending the current Teams thread id, avoids mutating the object returned by `resolveAgentRoute`, and adds a regression test for two routed channel-thread messages in the same normalized channel.

## Reproduction Assessment

Yes for a source-level reproduction: `resolveMSTeamsRouteSessionKey` with a base like `agent:main:msteams:channel:19:x@thread.tacv2:thread:old` and `conversationMessageId: "new"` deterministically produces a double suffix. A full live Teams reproduction is still missing.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it mutates a shared route object and blindly appends to an already-qualified key. The safer fix is local to MSTeams routing, with the existing generic thread-key parser used to recover the clean parent key before suffixing.
