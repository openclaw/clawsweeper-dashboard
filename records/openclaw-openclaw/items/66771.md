## Summary

Keep open. Current main and the latest tagged release still have a concrete MSTeams path that can append a new thread id onto an already thread-qualified session key, and the related #59294 work tracks #58615 rather than this double-suffix failure mode.

## What This Changes

The issue reports that MSTeams channel thread routing can reuse an old thread-qualified session key and append a new thread id, producing a malformed `...:thread:old:thread:new` key.

## Best Possible Solution

MSTeams should derive thread session keys from a clean immutable channel base, avoid mutating cached route objects, strip any prior `:thread:` suffix before applying the current thread id, and cover the old-thread-plus-new-thread cache path in regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: pass `baseSessionKey="agent:main:msteams:channel:19:x@thread.tacv2:thread:old"` and `conversationMessageId="new"` to `resolveMSTeamsRouteSessionKey`; the current code appends a second suffix. A live Teams minimal-room recipe is still unclear.

## Solution Assessment

Yes. The narrow maintainable solution is MSTeams-owned session-key derivation plus route immutability at the caller boundary, with a generic route-cache adjustment only if needed after checking callers.
