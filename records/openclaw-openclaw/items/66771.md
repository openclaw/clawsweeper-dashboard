## Summary

Keep open. Current main still has a concrete MSTeams source path for mixed thread keys: channel messages route by normalized channel id, the returned route is mutated after lookup, the thread helper appends to any supplied base, and the route cache can retain the mutated miss result.

## What This Changes

The issue reports that MSTeams channel-thread routing can append a new thread id onto an already thread-qualified old session key, producing malformed mixed keys with two `:thread:` suffixes.

## Best Possible Solution

Keep this as a valid bundled-plugin bug until MSTeams derives thread-qualified session keys from a clean immutable base route, prevents route-cache poisoning, and covers the old-thread-plus-new-thread shape with regression tests.

## Reproduction Assessment

Yes for a source-level reproduction: `resolveMSTeamsRouteSessionKey` given `baseSessionKey="agent:main:msteams:channel:19:x@thread.tacv2:thread:old"` and `conversationMessageId="new"` deterministically produces a double suffix. A full live Teams fresh-room reproduction is still missing.

## Solution Assessment

Yes, the best path is a narrow MSTeams-owned fix rather than a broad core migration: clean the base thread key before suffixing and avoid mutating the cached route object returned by `resolveAgentRoute`.
