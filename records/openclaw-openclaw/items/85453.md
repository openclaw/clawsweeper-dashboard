Keep open for PR review: the protected maintainer label prevents cleanup closure, and the patch still needs a small source-level correction because it strips ANSI after context parsing and creates a narrower duplicate stripper instead of the existing shared-style log display behavior.

The PR adds ANSI stripping to Control UI log parser display fields, updates focused parser tests for colored JSON/plain log lines, and adds a changelog entry.

## Best Possible Solution

Keep the UI-side display fix, but make ANSI normalization shared across Control UI log surfaces, apply it before context JSON parsing, preserve raw export lines, and retain the existing message-selection regression coverage.

## Reproduction Assessment

Yes. Current main lets ANSI-bearing log-tail text flow from `parseLogLine()` into `renderLogs()` without display normalization, and the linked issue provides a concrete Windows Control UI example.

## Solution Assessment

No. Parser-boundary display normalization is the right boundary, but this implementation should reuse one ANSI stripping strategy, sanitize the context candidate before JSON parsing, and keep the pre-existing slot-1 message regression test.
