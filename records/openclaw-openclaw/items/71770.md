## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown/probing, provider usage snapshots, and local usage/cost display, but it still does not implement configurable token, quota, or cost threshold routing, reverse failback, or user-facing threshold-switch notifications. The adjacent open items are relevant design context, but none cleanly supersedes this cross-provider threshold failover request.

## What This Changes

The issue requests configurable token or cost thresholds that automatically switch from a primary LLM provider/model to ordered fallbacks, optionally revert after reset, and notify the user when switching occurs.

## Best Possible Solution

Define a core usage-aware model-routing policy that consumes provider quota snapshots and local token/cost state before candidate selection, preserves the existing reactive cooldown/error fallback path, documents reset and reverse-failback behavior, and emits explicit status or notification events for threshold-triggered switches. Coordinate the design with #6966, #38248, #38775, and #45771 so rate-limit, spend, auth-profile balancing, and pace controls share one coherent contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source, schema, docs, tests, and targeted search: current main has usage visibility and reactive fallback, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product and config contract. The requested capability likely belongs in core model selection/fallback policy rather than a standalone plugin because it changes normal provider routing, but threshold units, failback semantics, and notification surfaces need design first.
