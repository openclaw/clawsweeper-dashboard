## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown/probing, and provider usage snapshots for status surfaces, but it still lacks configurable token, quota, or cost thresholds that proactively choose fallback providers, reverse failback, or threshold-triggered user notifications. The related pace-aware rate-limit issue overlaps but does not fully supersede this broader cross-provider threshold-routing request.

## What This Changes

The issue requests automatic switching among configured LLM providers based on token or cost thresholds, with optional reverse failback and notifications.

## Best Possible Solution

Define one usage-aware model-routing policy in core before candidate execution, backed by provider usage snapshots and local token/cost accounting where available. It should preserve the existing reactive cooldown/error fallback path, expose a documented config schema for thresholds and failback, and emit status or notification events through existing Gateway/session surfaces when a threshold switch occurs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from source, docs, and tests: current main exposes configured fallbacks and status usage snapshots, but no threshold policy participates in model selection.

## Solution Assessment

Unclear until maintainers define the product/config contract. The capability likely belongs in core model-selection/fallback policy with provider-plugin usage hints, not as an isolated plugin, because core owns retry, fallback, session override, status, and notification behavior.
