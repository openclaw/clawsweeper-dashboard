## Summary

Keep open. Current main has ordered reactive model fallback plus provider usage reporting, but it still lacks proactive token, quota, or cost threshold routing, reverse failback, and user-facing notifications for threshold-triggered switches. Related #45771 is adjacent pace-aware throttling work, not a canonical replacement for cross-provider threshold failover.

## What This Changes

Request automatic switching between configured LLM providers based on token or cost consumption thresholds, with ordered fallbacks, optional reverse failback, and notifications when switching occurs.

## Best Possible Solution

Define a core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves the existing reactive failure/cooldown path, documents reset and reverse-failback semantics, and emits explicit status or notification events for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has static configured fallback chains and usage reporting, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product policy. The capability likely belongs in core model selection because it changes normal provider routing, while provider plugins can remain responsible for usage snapshots.
