## Summary

Keep open. Current main and the latest release still provide ordered reactive fallback, profile cooldowns, provider usage display, and local cost reporting, but they do not implement token, quota, or cost thresholds that proactively select fallback models or notify/revert after failover.

## What This Changes

The issue requests configurable automatic model/provider failover and optional failback based on real-time token or cost consumption thresholds.

## Best Possible Solution

Define a core usage-aware model-routing contract that consumes explicit provider/model usage and local token/cost accounting before candidate selection, preserves current error-driven fallback by default, and documents threshold, hysteresis, failback, and notification semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks a threshold policy influencing model selection while supporting adjacent fallback, cooldown, usage display, and cost-reporting surfaces.

## Solution Assessment

Unclear until maintainers define the product and config contract. The maintainable path is a first-class routing policy with explicit accounting, freshness, hysteresis/failback, and notification rules rather than ad hoc checks inside the existing fallback loop.
