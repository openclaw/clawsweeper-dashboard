## Summary

Keep open. Current main supports ordered reactive model fallback, auth-profile cooldown/probe recovery, and provider usage/status snapshots, but it still has no configurable token, quota, or cost threshold policy that proactively changes model selection; related #45771 remains open and overlaps on usage pace, but it does not supersede this broader cross-provider failover request.

## What This Changes

Request configurable automatic switching between primary and fallback LLM providers based on token, quota, or cost thresholds, with optional failback and notifications.

## Best Possible Solution

Define a core usage-aware model-routing policy that runs before candidate execution, backed by provider usage hints and local token/cost accounting where available, while preserving the existing reactive cooldown/error fallback path and documenting config, failback, and notification behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from source, docs, and tests: current main supports ordered reactive fallback and usage displays, but no threshold policy participates in model selection.

## Solution Assessment

Unclear until maintainers define the product and config contract. The likely maintainable direction is core routing with provider-plugin usage hints because core owns candidate selection, fallback persistence, cooldown recovery, status, and notification surfaces.
