## Summary

Keep open. Current main has ordered/error-driven model fallback, auth-profile cooldowns, provider usage snapshots, and status/cost reporting, but it still lacks a token, quota, or cost threshold policy that proactively drives model selection; the related open issues are adjacent rather than a full replacement for this request.

## What This Changes

Request automatic switching between configured LLM providers/models based on token or cost thresholds, ordered fallbacks, optional reverse failover, and user notifications.

## Best Possible Solution

Define a core usage-aware model-routing policy that consumes explicit provider/model usage and local cost/token accounting before candidate selection, while preserving current error-driven fallback defaults and documenting config, hysteresis/failback, and notification semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks threshold policy inputs for model selection while supporting adjacent ordered fallback, cooldown, status usage, and cost reporting surfaces.

## Solution Assessment

Unclear until maintainers define the product/config contract. The maintainable direction is a core routing policy with explicit provider usage freshness, cost/token accounting, hysteresis, failback, and notification semantics rather than ad hoc checks in the fallback loop.
