## Summary

Keep open. Current main still has ordered reactive model fallback, auth-profile cooldown/probing, and provider usage snapshots for status/reporting, but it does not implement token-, quota-, or cost-threshold driven routing, reverse failback, or threshold-switch notifications. The related open issues remain useful design context rather than a merged canonical replacement.

## What This Changes

Request configurable automatic switching among LLM providers/models based on token or cost thresholds, ordered fallbacks, optional reverse failback, and failover notifications.

## Best Possible Solution

Define a core usage-aware model routing policy that consumes provider usage snapshots plus local token/cost state before candidate selection, preserves existing reactive error/cooldown fallback, documents reset and reverse-failback behavior, and emits explicit status or notification events for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has configured fallback chains and usage snapshots, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product and config contract. The requested capability likely belongs in core model-selection/fallback policy because core owns retry/fallback behavior, with provider plugins supplying usage or routing hints where needed.
