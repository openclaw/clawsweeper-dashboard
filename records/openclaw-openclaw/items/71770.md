## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown/probing, and provider usage snapshots, but it still does not implement token-, quota-, or cost-threshold driven model routing, reverse failback, or user-facing threshold-switch notifications. The related open issues are useful design context, but none cleanly supersedes this broader cross-provider threshold failover request.

## What This Changes

Request configurable automatic switching among LLM providers/models based on token or cost thresholds, ordered fallbacks, optional reverse failback, and failover notifications.

## Best Possible Solution

Define a core usage-aware model routing policy that consumes provider usage snapshots plus local token/cost state before candidate selection, preserves the existing reactive error/cooldown fallback path, documents reset and reverse-failback behavior, and emits explicit status or notification events for threshold-triggered switches. Coordinate it with #6966, #38248, #38775, and #45771 so rate-limit downgrades, spend guards, auth-profile balancing, and pace throttling share one coherent contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has configured fallback chains and usage snapshots, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product and config contract. The requested capability likely belongs in core model-selection/fallback policy rather than a standalone plugin because core owns retry/fallback behavior, but provider plugins may need to provide usage/routing hints.
