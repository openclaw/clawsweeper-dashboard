## Summary

Keep open. Current main has configured model fallback chains, reactive auth cooldown/probing, and provider usage/cost visibility, but token-, quota-, or cost-threshold policy still does not feed model routing, and the related open issues do not cleanly supersede this broader request.

## What This Changes

Request automatic model/provider switching based on configurable token or cost thresholds, ordered fallbacks, optional reverse failback, and user notifications.

## Best Possible Solution

Define a core usage-aware model routing policy that runs before candidate selection, consumes provider usage snapshots plus local token/cost state, preserves the existing reactive error/cooldown fallback path, documents reset and reverse-failback behavior, and emits explicit status or notification events for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main exposes fallback chains and usage snapshots, but no threshold policy participates in model routing.

## Solution Assessment

Unclear until maintainers define the product contract. The capability likely belongs in core model selection/fallback policy, with provider plugins supplying usage hints where needed, rather than as a standalone plugin that cannot govern Gateway-owned model calls.
