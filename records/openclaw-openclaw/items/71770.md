## Summary

Keep open. Current main has ordered, error-driven model fallback, rate-limit cooldown handling, provider usage readouts, and fallback/recovery notices, but it still lacks a proactive token, quota, or cost threshold policy that selects fallback models before failures occur.

## What This Changes

Requests configurable token or cost thresholds that automatically move configured agents from a primary model/provider to ordered fallbacks and optionally return when usage resets.

## Best Possible Solution

Define a usage-aware model-routing contract that consumes provider usage and local cost/token accounting before candidate selection, preserves current error-driven fallback defaults, and documents threshold, failback, and notification semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks threshold policy inputs while supporting adjacent ordered fallback, cooldown, usage, and cost visibility surfaces.

## Solution Assessment

Unclear until maintainers define the product/config contract. The maintainable direction is an explicit usage-aware routing policy rather than ad hoc checks inside the existing reactive fallback loop.
