## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldowns, and provider usage/status visibility, but it still lacks a token/cost/quota threshold policy that proactively changes model selection; the related open issues are adjacent design inputs, not a replacement for this request.

## What This Changes

The issue requests configurable token or cost thresholds that automatically switch from a primary model/provider to ordered fallbacks, optionally revert after reset, and notify users.

## Best Possible Solution

Define a usage-aware model-routing contract that consumes explicit provider/model usage and local cost/token accounting before candidate selection, preserves current error-driven fallback by default, and documents config, hysteresis, failback, and notification semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks threshold-driven model routing while supporting adjacent ordered fallback and usage visibility.

## Solution Assessment

Unclear until maintainers define the product and config contract. The maintainable direction is a core routing policy with explicit usage freshness, cost/token accounting, hysteresis, failback, and notification semantics rather than ad hoc checks in the fallback loop.
