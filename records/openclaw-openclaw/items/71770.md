## Summary

Keep open. Current main has ordered reactive fallback, cooldown/probe recovery, and provider usage snapshots for status, but it still lacks a configurable token, quota, or cost threshold policy that participates in model selection; the related rate-limit and pace trackers remain open and do not fully cover this token/cost failover scope.

## What This Changes

The issue requests automatic model failover based on configured token or cost thresholds, ordered fallback, optional reverse failover, and user notification.

## Best Possible Solution

Define one core usage-aware model-routing contract before candidate execution, backed by provider usage hints and local token/cost accounting where available, while preserving the current reactive cooldown/error fallback path by default.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source and test inspection gives high confidence that current main lacks a token, quota, or cost threshold policy influencing model selection while supporting adjacent ordered fallback and status usage snapshots.

## Solution Assessment

Unclear until maintainers define the product and config contract. The maintainable direction is likely a core routing policy with explicit provider usage freshness, token/cost accounting, hysteresis, failback, and notification semantics rather than ad hoc checks inside the fallback loop.
