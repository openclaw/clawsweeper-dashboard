## Summary

Keep open. Current main has ordered reactive fallback, model-scoped cooldown/probe behavior, and provider usage snapshots for status, but it still lacks token, quota, or cost thresholds that proactively participate in model selection.

## What This Changes

The issue requests configurable token or cost threshold-based automatic failover across ordered primary and fallback LLM providers, with optional reverse failover and notifications.

## Best Possible Solution

Define one core usage-aware model-routing contract that layers explicit token/cost/quota thresholds onto existing ordered fallback without changing current reactive failover by default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence that current main lacks a threshold policy that influences model selection.

## Solution Assessment

Unclear until maintainers choose the product and config contract. The maintainable direction is a core routing policy with provider usage freshness, accounting, hysteresis, failback, and notification semantics rather than ad hoc checks inside the current fallback loop.
