## Summary

Keep open. Current main and the latest shipped release have ordered reactive model fallback, primary recovery probes, fallback notices, and provider usage visibility, but they still do not provide proactive token, quota, or cost threshold routing; the adjacent pace and budget issues are related design inputs rather than terminal replacements.

## What This Changes

The issue requests configurable token or cost thresholds that automatically route between primary and fallback LLM providers, optionally fail back, and notify users on transitions.

## Best Possible Solution

Preserve the current reactive fallback defaults, then add a maintainer-approved usage-aware routing contract with explicit thresholds, freshness rules, hysteresis, failback behavior, and notices coordinated with the related pace and budget work.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, release-tag inspection, and related issue context give high confidence that proactive token/cost threshold routing is not implemented.

## Solution Assessment

Unclear until maintainers define the product and config contract. The maintainable path is a core routing policy with explicit usage freshness, accounting, hysteresis, failback, and notification semantics rather than ad hoc checks inside the existing fallback loop.
