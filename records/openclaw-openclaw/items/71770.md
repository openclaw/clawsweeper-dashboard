## Summary

Keep open. Current main has ordered reactive model fallback, model/profile cooldown behavior, provider usage snapshots, and cost reporting, but it still lacks any token, quota, or cost threshold policy that proactively drives model selection; the related open issues are adjacent rather than a full replacement for this request.

## What This Changes

Request configurable token or cost thresholds that automatically switch configured LLM providers/models, optionally fail back, and notify users.

## Best Possible Solution

Define a core usage-aware model-routing policy that consumes explicit provider/model usage and local cost/token accounting before candidate selection, preserves current error-driven fallback by default, and documents config, hysteresis/failback, and notification semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks a threshold policy influencing model selection while supporting adjacent ordered fallback, cooldown, status usage, and cost reporting surfaces.

## Solution Assessment

Unclear until maintainers define the product/config contract. The maintainable direction is a core routing policy with explicit provider usage freshness, cost/token accounting, hysteresis, failback, and notification semantics rather than ad hoc checks in the fallback loop.
