## Summary

Keep open. Current main has ordered, error-driven model fallback, cooldown probing/recovery, user-visible fallback notices, and provider usage readouts, but it still lacks a proactive token, quota, or cost-threshold policy that selects a fallback model before a provider fails.

## What This Changes

The issue requests automatic provider/model failover driven by token or cost thresholds, with ordered fallback, optional failback, and user notifications.

## Best Possible Solution

Define an explicit usage-aware model-routing policy that consumes provider quota snapshots and local token/cost accounting before candidate selection, while preserving today’s error-driven fallback as the default and documenting opt-in config, hysteresis, failback, and notices.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection gives high confidence that current main lacks threshold policy inputs while supporting adjacent ordered fallback, cooldowns, usage readouts, and fallback notices.

## Solution Assessment

Unclear until maintainers define the product and config contract. The maintainable path is a core opt-in routing policy with explicit usage freshness, accounting, hysteresis, failback, and notification semantics rather than ad hoc checks inside the current fallback loop.
