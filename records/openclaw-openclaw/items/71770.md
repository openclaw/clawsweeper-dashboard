## Summary

Keep open. Current main has ordered reactive model fallback, cooldown/probe recovery, and provider usage snapshots for status, but it still lacks a configurable token, quota, or cost threshold policy that participates in model selection.

## What This Changes

Request configurable token or cost thresholds that automatically fail over across ordered primary and fallback LLM models, optionally revert later, and notify users.

## Best Possible Solution

Define one core usage-aware model-routing policy before candidate execution, backed by provider usage hints and local token/cost accounting where available, while preserving the current reactive cooldown/error fallback path by default.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that current main lacks a threshold policy influencing model selection while supporting adjacent ordered fallback and status usage snapshots.

## Solution Assessment

Unclear until maintainers define the product/config contract. The maintainable direction is likely core routing with explicit provider usage freshness, cost/token accounting, hysteresis, failback, and notification semantics rather than ad hoc checks in the fallback loop.
