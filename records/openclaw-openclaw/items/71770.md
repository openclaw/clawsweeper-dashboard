## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown/probing, and provider usage reporting, but it still does not implement token-, quota-, or cost-threshold driven routing, reverse failback, or user-facing threshold-switch notifications. The closest related open items are useful design context, especially #6966, #38248, #38775, and #45771, but no single one cleanly supersedes this cross-provider threshold failover request.

## What This Changes

The issue requests automatic switching between configured LLM providers based on token or cost thresholds, with ordered fallback, optional reverse failback, and notifications.

## Best Possible Solution

Define a core usage-aware model routing policy that consumes provider usage snapshots and local cost/token state before candidate selection, preserves the existing reactive error/cooldown fallback path, documents reset and reverse-failback behavior, and emits explicit status/notification events for threshold-triggered switches. The design should be coordinated with #6966, #38248, #38775, and #45771 so model routing, spend guards, auth-profile balancing, and pace throttling share one coherent contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has configured fallback chains and usage snapshots, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product/API contract. The requested capability likely belongs in core model-selection/fallback policy rather than a standalone plugin because it changes normal provider routing, but threshold units, failback semantics, and notification surfaces need design first.
