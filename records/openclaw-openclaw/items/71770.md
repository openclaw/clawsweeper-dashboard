## Summary

Keep open. Current main has ordered reactive model fallback, primary recovery probes, fallback notices, and provider usage visibility, but it still lacks a proactive token, quota, or cost threshold policy that selects fallback models before failure and later fails back based on usage thresholds.

## What This Changes

The issue requests configurable token or cost threshold based automatic failover and optional failback across configured LLM providers.

## Best Possible Solution

Define a usage-aware model-routing contract that preserves current reactive fallback defaults while adding explicit threshold inputs, freshness rules, hysteresis, failback behavior, and visible transition notices.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source and docs inspection give high confidence that current main lacks proactive token/cost threshold model routing while supporting adjacent reactive fallback and usage visibility.

## Solution Assessment

Unclear until maintainers define the product and config contract. The maintainable path is a core routing policy with explicit usage freshness, accounting, hysteresis, failback, and notification semantics rather than ad hoc checks in the existing fallback loop.
