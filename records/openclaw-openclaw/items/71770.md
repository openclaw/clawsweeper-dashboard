## Summary

Keep open. Current main has ordered reactive model fallback, cooldown/probe recovery, and provider usage snapshots for status surfaces, but no configurable token, quota, or cost threshold policy participates in model selection; the open pace-aware rate-limit item overlaps but does not supersede this broader cross-provider failover request.

## What This Changes

Add automatic model/provider switching based on configurable token, quota, or cost thresholds, with ordered fallback, optional failback, and notifications.

## Best Possible Solution

Define one core usage-aware model-routing policy before candidate execution, backed by provider usage hints and local token/cost accounting where available, while preserving the existing reactive cooldown/error fallback path and documenting the config plus notification/failback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from source, docs, and tests: current main supports ordered reactive fallback and status usage snapshots, but no threshold policy influences model selection.

## Solution Assessment

Unclear until maintainers define the product/config contract. The capability likely belongs in core routing with provider-plugin usage hints because core owns model candidate selection, fallback persistence, cooldown recovery, status, and notification surfaces.
