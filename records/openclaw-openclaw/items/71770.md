## Summary

Keep open. Current main has ordered reactive model fallback, cooldown probing, and provider usage/status snapshots, but it still does not have configurable token, quota, or cost thresholds that proactively route across providers, optional failback, or threshold-triggered notifications.

## What This Changes

Request configurable token/cost threshold-based routing that switches among ordered model providers, optionally fails back, and notifies users.

## Best Possible Solution

Define a core usage-aware model-routing contract that preserves the existing reactive cooldown/error fallback path, consumes provider usage snapshots and local cost data where reliable, documents the config schema, and emits status or notification events when threshold routing changes the active model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from source, docs, and tests: current main exposes configured fallbacks and usage snapshots, but no threshold policy participates in model selection.

## Solution Assessment

Unclear until maintainers define the product/config contract. The capability likely belongs in core model-selection policy with provider-plugin usage hints because core owns fallback, session override, status, and notification behavior.
