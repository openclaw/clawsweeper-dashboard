## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown/probing, and provider usage snapshots for status surfaces, but it still lacks configurable token, quota, or cost thresholds that proactively route model selection, reverse failback, or threshold-triggered user notifications. The related open issues overlap, but none fully supersedes this broader cross-provider threshold failover request.

## What This Changes

Request automatic switching between configured primary and fallback LLM providers based on real-time token consumption or cost thresholds, with optional reverse failback and user notifications.

## Best Possible Solution

Define one core usage-aware model-routing policy that runs before fallback candidate execution, consumes provider usage snapshots plus local token/cost accounting, preserves the existing reactive cooldown/error fallback path, and emits explicit status or notification events when a threshold switch occurs. Coordinate the config contract with the related rate-limit, spend guard, auth balancing, and pace throttling issues so users do not get several incompatible routing policies.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from source, docs, and tests: current main exposes configured fallbacks and usage/status snapshots, but no threshold policy participates in model selection.

## Solution Assessment

Unclear until maintainers define the product/config contract. The capability likely belongs in core model-selection/fallback policy with provider-plugin usage hints, not as an isolated plugin, because core owns retry, fallback, session override, status, and notification behavior.
