## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown/probing, and provider usage snapshots, but it still lacks token-, quota-, or cost-threshold based model routing, reverse failback, and user-facing threshold-switch notifications. The related open issues overlap with parts of the design, but none cleanly supersedes this broader cross-provider threshold failover request.

## What This Changes

Request automatic switching between configured primary and fallback LLM providers/models when token consumption or cost thresholds are reached, with optional reverse failback and notifications.

## Best Possible Solution

Define a core usage-aware model-routing policy that consumes provider usage snapshots plus local token/cost state before candidate selection, preserves the existing reactive cooldown/error fallback path, documents reset and reverse-failback behavior, and emits explicit status or notification events for threshold-triggered switches. Coordinate the contract with #6966, #38248, #38775, and #45771 so rate-limit downgrades, spend guards, auth-profile balancing, and pace throttling share one coherent policy.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from source and docs inspection: current main exposes configured fallbacks and usage/status snapshots, but no threshold policy participates in model selection.

## Solution Assessment

Unclear until maintainers define the product/config contract. The capability likely belongs in core model-selection/fallback policy with provider-plugin usage hints, not as an isolated plugin, because core owns retry, fallback, session override, and status behavior.
