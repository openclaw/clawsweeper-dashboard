## Summary

Keep open. Current main still supports ordered reactive model fallback, cooldown/probe recovery, provider usage snapshots, and local cost reporting, but no token, quota, or cost threshold policy participates in model candidate selection; the related pace-aware rate-limit item is narrower and remains open rather than superseding this request.

## What This Changes

Request automatic switching between configured LLM providers based on token or cost consumption thresholds, with ordered fallback, optional reverse failback, and user notifications.

## Best Possible Solution

Define one core usage-aware model-routing policy before candidate execution, backed by provider usage hints and local token/cost accounting where available, while preserving the existing reactive cooldown/error fallback path and documenting config, notification, and failback behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The implementation gap is high-confidence from current source, docs, and tests: usage is reported in status/cost surfaces, while model selection remains ordered and reactive.

## Solution Assessment

Unclear until maintainers define the product/config contract. The capability likely belongs in core routing with provider-plugin usage hints because core owns model candidate selection, fallback persistence, cooldown recovery, status, and notification surfaces.
