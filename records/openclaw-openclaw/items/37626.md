## Summary

Current main still has the Feishu wiki pagination bug: the `nodes` action calls `spaceNode.list` once, cannot pass pagination inputs, and returns no pagination metadata, while the Feishu docs and SDK mark this endpoint as paginated. The earlier fix PR was closed unmerged, and the later duplicate was closed in favor of keeping this issue as the canonical tracker.

## What This Changes

The issue asks for `feishu_wiki` `nodes` to return or expose complete paginated Feishu wiki child-node results instead of only the first page.

## Best Possible Solution

Implement the pagination fix inside the Feishu plugin-owned wiki tool with a bounded loop or explicit pagination metadata, focused regression coverage, and a changelog entry for the user-visible data completeness fix.

## Reproduction Assessment

Yes, source-reproducible. A Feishu wiki parent with more than one `spaceNode.list` page will hit the current single-call `listNodes` path, which neither follows nor exposes the upstream pagination token.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/feishu/src/wiki.ts` and `extensions/feishu/src/wiki-schema.ts`, using the existing Feishu SDK pagination fields and preserving plugin ownership rather than changing core.

## Repair Work Prompt

Fix the canonical Feishu wiki pagination bug tracked by https://github.com/openclaw/openclaw/issues/37626. Current `feishu_wiki` `nodes` in `extensions/feishu/src/wiki.ts` calls `client.wiki.spaceNode.list` once and returns only `nodes`, while Feishu `space-node/list` and `@larksuiteoapi/node-sdk@1.62.1` expose `page_size`, `page_token`, response `has_more`, and response `page_token`. Keep the fix plugin-owned; do not move behavior into core. Implement a bounded, maintainable pagination path for `nodes` that prevents unbounded API/cost amplification, handles `has_more=true` with empty items and next `page_token`, and reports truncation or continuation metadata clearly. Update `extensions/feishu/src/wiki-schema.ts` only if exposing caller-controlled pagination fields is part of the chosen fix; avoid adding a global config option. Add focused regression tests, including multi-page aggregation or token passthrough, empty-page-with-has_more, missing next token/truncation, API error preservation, and account routing if the execute path changes. Add a single-line `CHANGELOG.md` fix entry crediting the reporter if practical. Use PR https://github.com/openclaw/openclaw/pull/37641 as context only; reimplement against current main and do not copy unsafe logging/resource-amplification behavior blindly.
