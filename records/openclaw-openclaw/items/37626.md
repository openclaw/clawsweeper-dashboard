## Summary

Keep open: current main still has the Feishu wiki pagination gap, because `feishu_wiki` `nodes` makes one `spaceNode.list` call, drops upstream pagination fields, and the related fix PRs were closed unmerged while a newer duplicate was closed back to this canonical issue.

## What This Changes

The issue asks for the Feishu plugin wiki `nodes` action to return all child nodes from paginated `spaceNode.list` results instead of only the first page.

## Best Possible Solution

Implement the fix in the Feishu plugin wiki tool with bounded pagination over `has_more` and `page_token`, preserving the existing `nodes` call shape while adding focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible. A Feishu wiki parent with more than one `spaceNode.list` page will hit the current single-call `listNodes` path, which neither follows `has_more` nor exposes a next `page_token`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/feishu/src/wiki.ts` and should use the upstream pagination fields with conservative caps, plus tests covering multi-page, empty-page-with-has-more, and error paths.

Fix https://github.com/openclaw/openclaw/issues/37626 in the Feishu plugin only. The observable bug is that `feishu_wiki` with `action: "nodes"` returns only the first `wiki.spaceNode.list` page for parents with more than one page of child nodes. Update `extensions/feishu/src/wiki.ts` so `listNodes` follows Feishu `has_more` and `page_token` using `page_size` capped at 50 and an explicit conservative page/node cap; preserve account routing and avoid unbounded API loops.
