## Summary

Keep this open. Current main has reduced the cron catch-up and session-store pressure described in the discussion, but it still does not provide durable Gateway heap headroom or a supported `gateway.nodeMaxOldSpaceMb`/service `NODE_OPTIONS` path for the reported V8 heap ceiling.

## What This Changes

The issue asks OpenClaw to prevent Gateway OOM crashes during isolated cron workloads, either by lowering memory use, adding service heap headroom, or further throttling cron catch-up.

## Best Possible Solution

Land a reviewed Gateway-managed heap-headroom path, likely a revised PR #42853 or equivalent, while keeping the existing cron throttling and session cleanup mitigations.

## Reproduction Assessment

Unclear for the full OOM workload: I did not run the WSL2/systemd cron scenario. Yes for the remaining source-level gap: current main has no Gateway heap config field, no service `NODE_OPTIONS` insertion, and no Node `--max-old-space-size` service argument.

## Solution Assessment

Yes, keeping this open is the best current handling. The narrow maintainable fix is a reviewed Gateway heap config/service-environment change, not a duplicate scheduler rewrite or unsupported arbitrary `NODE_OPTIONS` persistence.
