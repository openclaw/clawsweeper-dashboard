## Summary

Keep this issue open. Current main has cron catch-up, cron concurrency, isolated-session cleanup, and session-store maintenance mitigations, but it still does not implement the reported Gateway heap-headroom behavior: generated Gateway service environments and Node program arguments do not set a heap limit, and the Gateway config/schema has no `gateway.nodeMaxOldSpaceMb`-style contract. The linked open #42853 remains the closest implementation candidate rather than a reason to close this issue before it lands.

## What This Changes

The issue requests mitigation for Gateway Node.js heap exhaustion during cron-heavy workloads, either by reducing per-run memory, adding Gateway heap configuration/service defaults, or throttling cron catch-up.

## Best Possible Solution

Keep this as the cron-triggered Gateway heap/OOM tracker until a narrow service heap-headroom/config solution lands, likely a revised #42853 or equivalent, while leaving broader session-store payload and pruning work under the existing session-store OOM trackers.

## Reproduction Assessment

Unclear for the full OOM: the issue has detailed steps and a verified operator workaround, but this read-only review did not run the WSL2 cron workload. We do have a high-confidence source-level reproduction of the missing heap-headroom behavior because current service env, program args, and config/schema paths lack the requested setting.

## Solution Assessment

Yes, keeping this open is the best current handling. The narrow maintainable fix is a reviewed Gateway service heap option that preserves the existing cron/session mitigations and avoids arbitrary `NODE_OPTIONS` persistence.
