## Summary

Keep open. Current main and v2026.5.19 still lack a supported Gateway heap-limit config or generated heap-only service flag; cron catch-up, run reaping, and session-store bloat have been partially mitigated, but the durable daemon heap path remains unresolved and https://github.com/openclaw/openclaw/pull/42853 is still open/conflicting with proof and security-boundary blockers.

## What This Changes

The issue asks OpenClaw to prevent Gateway OOM restart loops under cron memory pressure by lowering memory use, throttling catch-up, or generating a durable V8 heap limit for managed services.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/42853 with a typed heap-only Gateway and node-service config path that preserves the `NODE_OPTIONS` blocklist, covers regenerated launchd/systemd/Windows service metadata, and keeps the existing cron/session pressure mitigations intact.

## Reproduction Assessment

No high-confidence live current-main reproduction of the WSL2/systemd OOM loop was established. The source-level gap is reproducible: current main and `v2026.5.19` lack a supported `gateway.nodeMaxOldSpaceMb` or generated heap-only service flag, while the issue and linked reports provide real crash evidence.

## Solution Assessment

Yes, keeping this open is the best current handling. The maintainable fix is a typed heap-only service configuration path with real regenerated-service proof, not another scheduler-only patch or arbitrary ambient `NODE_OPTIONS` passthrough.
