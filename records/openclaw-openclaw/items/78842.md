## Summary

Keep open: current main still leaves the Feishu guarded-fetch HTTP paths in strict mode, while the Feishu WebSocket path already follows ambient proxy env, so the requested HTTP proxy parity is not implemented.

## What This Changes

Request Feishu streaming-card and app-registration HTTP calls to honor proxy environment variables instead of using the strict guarded-fetch default.

## Best Possible Solution

Add a Feishu-owned guarded-fetch helper for fixed Feishu/Lark API hosts that opts into trusted env proxy mode while preserving hostname allowlists, audit contexts, SSRF checks, and the core strict default.

## Reproduction Assessment

Yes, at source level: set HTTPS_PROXY or HTTP_PROXY without managed proxy enabled, then Feishu streaming-card/app-registration calls still use strict guarded fetch while WSClient gets an ambient proxy agent. I did not run a live Feishu proxy-only account scenario.

## Solution Assessment

Yes, the narrow maintainable direction is plugin-local proxy parity for trusted Feishu/Lark guarded-fetch calls. Changing the core strict default or folding in the broader Lark SDK/WS proxy policy work would be too broad for this item.
