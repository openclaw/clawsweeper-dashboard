## Summary

Keep open: current main still leaves the Feishu streaming-card and app-registration guarded-fetch calls in default strict mode, so raw HTTP(S)_PROXY is not honored there. Related Feishu proxy threads ask for conflicting direct versus ambient behavior, so this needs maintainer proxy-policy review before a repair PR.

## What This Changes

The issue requests Feishu plugin HTTP API calls to honor raw proxy environment variables like the Feishu WebSocket path does.

## Best Possible Solution

Agree on Feishu proxy policy, then implement a plugin-local HTTP helper or policy path that preserves hostname allowlists and the core strict default while covering streaming-card and app-registration calls.

## Reproduction Assessment

Yes, at source level: set HTTPS_PROXY or HTTP_PROXY without OPENCLAW_PROXY_ACTIVE=1, then exercise Feishu streaming-card or app-registration fetches and those calls remain in strict guarded-fetch mode. I did not run a live Feishu account/proxy scenario.

## Solution Assessment

Unclear as a standalone raw-env patch until maintainers choose Feishu direct versus ambient proxy policy. The narrow code direction is plugin-local use of existing trusted env proxy behavior for known Feishu/Lark endpoints, not changing the core guarded-fetch default.
