## Label Justifications

- `P1`: The report describes Feishu HTTP API calls failing in proxy-only environments, which can block a real channel workflow for affected users.
- `impact:security`: The requested behavior changes whether credentialed Feishu/Lark HTTP traffic follows ambient or managed proxy egress.
- `impact:message-loss`: Streaming-card, token, and app-registration HTTP failures can prevent Feishu replies or setup from completing.
- `impact:auth-provider`: The central failure path includes Feishu tenant access token and channel credential validation HTTP calls.

## Summary

Keep open: current main and v2026.6.6 still leave Feishu streaming-card and app-registration guarded-fetch HTTP calls on the strict default, and the broader open Feishu proxy PR only covers the Lark SDK client path, not these call sites.

## What This Changes

Request Feishu HTTP API calls that use `fetchWithSsrFGuard` to honor proxy environment variables instead of inheriting the strict default mode.

## Best Possible Solution

Settle the Feishu plugin egress policy, then update these fixed Feishu/Lark guarded-fetch call sites, focused tests, and docs to consistently use the approved direct, ambient, or managed-proxy behavior without changing the core strict default.

## Reproduction Assessment

Yes, source-reproducible: with `HTTP_PROXY` or `HTTPS_PROXY` set and no active OpenClaw managed-proxy lifecycle, these Feishu guarded-fetch calls omit `mode`, so `resolveGuardedFetchMode` returns strict and ignores ordinary ambient proxy env. I did not run a live Feishu account/proxy scenario.

## Solution Assessment

Unclear as a product/security decision. The narrow technical shape is Feishu-local trusted proxy handling for fixed vendor hosts, but related issues request direct-by-default or mixed proxy behavior, so maintainers need to choose one policy before another fix PR is queued.
