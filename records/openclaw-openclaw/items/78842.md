## Summary

Keep open: current main still leaves the named Feishu guarded-fetch HTTP call sites in strict mode, while the WebSocket path has ambient proxy handling. I would not close or auto-repair it blindly because the same Feishu proxy cluster has conflicting direct-vs-ambient policy needs and the exact fix PR was closed unmerged for lack of real behavior proof.

## What This Changes

Request that Feishu plugin HTTP API calls for streaming cards, token refresh, and app registration honor HTTP(S)_PROXY instead of defaulting to strict guarded fetch.

## Best Possible Solution

Define a Feishu-owned proxy policy, then apply it consistently to guarded fetch, Lark SDK REST, and WebSocket paths while preserving hostname allowlists, NO_PROXY semantics, and the core strict default.

## Reproduction Assessment

Yes, at source level: set HTTPS_PROXY without OpenClaw managed proxy activation, then Feishu streaming-card and app-registration guarded-fetch calls remain in strict mode while Feishu WebSocket setup can get an ambient proxy agent. I did not run a live Feishu credential/proxy scenario.

## Solution Assessment

Unclear as an isolated change. Adding trusted_env_proxy to the named call sites is technically narrow, but the maintainable fix should be reviewed against the broader Feishu direct-versus-ambient proxy policy before another patch lands.
