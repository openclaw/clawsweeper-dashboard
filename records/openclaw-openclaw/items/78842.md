## Summary

Keep open: current main and v2026.6.10 still leave the exact Feishu guarded-fetch HTTP calls on the strict default. The open related PRs cover SDK HTTP, WebSocket defaults, or docs/policy slices, but not this guarded-fetch runtime gap.

## What This Changes

Request Feishu plugin HTTP calls for streaming cards, token retrieval, and app registration to support proxy environments instead of always using strict guarded-fetch routing.

## Best Possible Solution

Settle one Feishu-owned egress policy that covers guarded-fetch HTTP, Lark SDK HTTP, WebSocket traffic, docs, and tests without changing the core strict guarded-fetch default.

## Reproduction Assessment

Yes, source-reproducible: current Feishu guarded-fetch call sites omit `mode`, and fetch-guard defaults those calls to strict mode instead of ordinary env proxy routing. I did not run a live credentialed Feishu proxy-only scenario.

## Solution Assessment

Unclear until maintainers choose the Feishu egress policy. The implementation should stay Feishu-local and keep core guarded-fetch strict by default, but the direct/ambient/managed-proxy behavior needs explicit acceptance.
