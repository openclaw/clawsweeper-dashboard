## Summary

Keep open: current main still leaves the Feishu HTTP API guarded-fetch calls in strict mode, and the only linked implementation PR was closed unmerged after the contributor could not provide live Feishu proxy proof.

## What This Changes

Feishu plugin HTTP API calls should honor `HTTP_PROXY`/`HTTPS_PROXY` in proxy-only environments, matching the existing Feishu WebSocket proxy behavior.

## Best Possible Solution

Add a Feishu-owned HTTP helper or explicit `mode: "trusted_env_proxy"` at trusted Feishu/Lark API call sites while preserving hostname allowlists, audit contexts, and SSRF safeguards.

## Reproduction Assessment

Yes by source inspection: set `HTTPS_PROXY` without the managed proxy lifecycle, then Feishu HTTP calls remain in strict guarded-fetch mode while the Feishu WebSocket client gets a proxy agent. I did not run a live Feishu/proxy account scenario.

## Solution Assessment

Yes, the best direction is a narrow Feishu plugin change using the existing trusted env proxy mode for fixed vendor API hosts. Changing the core guarded-fetch default would be broader and riskier.
