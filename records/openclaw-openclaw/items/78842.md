## Summary

Keep open: current main and v2026.6.8 still leave the Feishu guarded-fetch HTTP call sites on the strict default, while the open related PRs cover SDK HTTP, WebSocket default, or docs-only paths rather than this exact runtime gap.

## What This Changes

The issue requests Feishu streaming-card and app-registration HTTP calls to honor proxy environment variables instead of inheriting `fetchWithSsrFGuard` strict mode.

## Best Possible Solution

Settle one Feishu-owned egress policy covering guarded-fetch HTTP, Lark SDK HTTP, WebSocket traffic, docs, and tests, then update the Feishu plugin without changing the core strict guarded-fetch default.

## Reproduction Assessment

Yes, source-reproducible: with ordinary `HTTP_PROXY` or `HTTPS_PROXY` set and no active managed-proxy lifecycle, these Feishu guarded-fetch calls omit `mode`, so `resolveGuardedFetchMode` returns strict and ordinary env proxy routing is not selected. I did not run a live Feishu account/proxy scenario.

## Solution Assessment

Unclear as a product/security decision. The narrow code shape is Feishu-local trusted proxy handling for fixed Feishu/Lark hosts, but related issues request direct-by-default or mixed proxy behavior, so maintainers need to choose the egress policy first.
