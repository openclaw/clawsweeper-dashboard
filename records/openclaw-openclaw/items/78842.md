## Summary

Keep open: current main and the latest stable tag still leave the Feishu guarded-fetch HTTP calls in strict mode, and the related Feishu proxy threads show that direct-vs-ambient egress policy needs maintainer judgment before a safe fix.

## What This Changes

The issue asks for Feishu streaming-card, token refresh, and app-registration HTTP calls to honor proxy environment variables instead of defaulting to strict guarded-fetch mode.

## Best Possible Solution

Define a Feishu-owned HTTP egress policy that covers guarded-fetch call sites and SDK HTTP calls, then opt trusted Feishu/Lark API hosts into the chosen proxy behavior with focused tests and docs while leaving the core guarded-fetch strict default intact.

## Reproduction Assessment

Yes, by source inspection: set `HTTPS_PROXY` without the managed proxy lifecycle, then Feishu guarded-fetch HTTP calls remain in strict mode while the Feishu WebSocket path can receive an ambient proxy agent. I did not run a live Feishu/proxy account scenario.

## Solution Assessment

Unclear as a product decision: adding `trusted_env_proxy` at fixed Feishu/Lark guarded-fetch hosts is narrow, but adjacent Feishu reports ask for direct or per-channel bypass behavior. The safer path is a maintainer-approved Feishu egress policy rather than changing guarded-fetch defaults or only one call family.
