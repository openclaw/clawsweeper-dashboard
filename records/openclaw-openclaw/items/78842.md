## Summary

Keep open: current main still leaves the Feishu guarded-fetch HTTP call sites on the strict default, the linked implementation PR closed unmerged, and related Feishu proxy-policy work is still unresolved.

## What This Changes

The issue requests Feishu app-registration and streaming-card guarded HTTP calls to honor HTTP(S)_PROXY instead of staying in strict guarded-fetch mode.

## Best Possible Solution

Add a Feishu-local HTTP helper or explicit trusted_env_proxy mode only at trusted Feishu/Lark API call sites, preserve hostname allowlists and NO_PROXY behavior, and leave the core strict default unchanged.

## Reproduction Assessment

Yes, source-reproducible: set HTTP_PROXY or HTTPS_PROXY without the managed proxy lifecycle, and these Feishu guarded-fetch calls still resolve to strict mode while the WebSocket path gets an ambient proxy agent. I did not run a live Feishu account/proxy scenario.

## Solution Assessment

Unclear until maintainers settle Feishu egress policy; the narrow maintainable direction is plugin-local trusted_env_proxy for fixed vendor hosts, not changing fetchWithSsrFGuard's strict default.
