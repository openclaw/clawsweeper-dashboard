## Summary

Keep open: current main and v2026.6.10 still classify the reported ZAI reset-text quota error as `rate_limit` and route generic provider failures through local stepped cooldown/profile-rotation logic, while the exact reset-timestamp contract and error-class-aware rotation policy remain unresolved product work.

## What This Changes

The issue asks OpenClaw to honor provider reset timestamps for rate-limit cooldowns and make profile rotation limits distinguish long quota exhaustion from transient overload.

## Best Possible Solution

Define a typed provider reset-hint path that can safely carry validated header or message reset metadata into auth-profile cooldown state, reconcile it with quota-exhaustion classification work, and preserve existing rotation defaults until maintainers choose a compatibility policy.

## Reproduction Assessment

Yes, at source level: the reported reset-text message is classified as `rate_limit`, and the generic auth-profile failure path has no reset timestamp input and falls back to local stepped cooldowns. I did not run a live exhausted ZAI account in this read-only review.

## Solution Assessment

No, the suggested regex alone is not the best solution. The maintainable path is a typed reset-hint contract across provider error classification, auth-profile cooldown mutation, docs/tests, and the overlapping quota-exhaustion work.
