## Summary

Keep this open: current main and the latest release still classify the reported periodic reset text as ordinary rate-limit handling, do not carry provider reset headers/body timestamps into auth-profile cooldown state, and leave the rotation-depth change as a compatibility-sensitive product decision. The open quota circuit-breaker PR overlaps part of the taxonomy question, but it is not merged and does not cover the full reset-timestamp and rotation-cap request.

## What This Changes

The issue requests provider reset timestamp parsing for auth-profile cooldowns plus error-class-aware rate-limit/profile rotation behavior for multi-key providers.

## Best Possible Solution

Define a typed, validated provider reset-hint path from transport/error normalization into auth-profile cooldown state, coordinate it with quota-exhaustion work, and keep rotation defaults stable unless maintainers explicitly approve a compatibility change.

## Reproduction Assessment

Yes at source level, but not with a live ZAI run in this review. Current code classifies the reported reset-text condition into existing failover buckets and the generic auth-profile failure path has no reset timestamp or header metadata input to persist as cooldownUntil.

## Solution Assessment

Unclear as proposed: a reset parser is directionally useful, but a local regex alone is not the best maintainable solution. The safer path is a typed reset-hint contract across provider error normalization, failover classification, auth-profile state, docs, and tests, with maintainer-approved rotation semantics.
