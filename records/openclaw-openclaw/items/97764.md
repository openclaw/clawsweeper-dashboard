## Summary

The issue should stay open: current main recognizes the Zhipu/ZAI weekly-monthly quota text as a failover-triggering rate limit, but the generic provider path still does not propagate reset timestamps into auth-profile cooldowns, and the rotation-depth part remains an overlapping product/compatibility decision rather than an implemented fix.

## What This Changes

Request provider reset timestamp parsing, quota-vs-transient rate-limit classification, and less aggressive same-provider auth-profile rotation behavior for multi-key providers.

## Best Possible Solution

Define one generic provider reset-hint path that safely passes validated reset metadata into auth-profile cooldown state, coordinates with quota-exhaustion classification work, and preserves explicit rotation defaults unless maintainers approve a compatibility-changing policy update.

## Reproduction Assessment

Yes at source level, but not with a live ZAI run in this review. Current code classifies the reported reset-text message as `rate_limit` and the generic auth-profile failure path has no reset timestamp input to persist a provider reset cooldown.

## Solution Assessment

No, the current implementation is not the best end state for this issue. The suggested regex is directionally useful, but the maintainable fix needs a typed reset-hint path across provider error classification, auth-profile cooldown mutation, docs/tests, and any quota-exhaustion work rather than a local string match alone.
