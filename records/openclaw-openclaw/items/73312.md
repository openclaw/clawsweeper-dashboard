## Summary

Keep this PR open. The intended Matrix presentation metadata direction is useful and builds on the already-merged Matrix approval metadata work, but the current head still misses the registered Matrix outbound facade path and forwards caller-supplied Matrix event-content fields too broadly.

## What This Changes

The PR adds Matrix MessagePresentation rendering/sendPayload support, targeted Matrix outbound tests, and a new Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the registered Matrix outbound facade exposes presentationCapabilities, renderPresentation, and sendPayload, while sendPayload only accepts validated com.openclaw.presentation metadata and the docs nav plus changelog are updated.

## Reproduction Assessment

Yes. Trace normal delivery through loadChannelOutboundAdapter into the registered Matrix plugin outbound facade: current main forwards only sendText, sendMedia, and sendPoll, while the PR adds the new hooks only to matrixOutbound.

## Solution Assessment

No. The metadata shape and plain-text fallback are a reasonable direction, but the current implementation is not mergeable until the registered facade path is wired and the Matrix event-content contract is constrained.
