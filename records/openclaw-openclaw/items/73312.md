## Summary

Keep open. The PR addresses a useful Matrix progressive-rendering use case and the author’s follow-up explains a concrete downstream iOS flow, but the latest head still is not mergeable: the new hooks are added only to the Matrix runtime adapter, the registered Matrix channel facade still does not expose them to core delivery, and the new sendPayload path forwards Matrix event content too broadly.

## What This Changes

The PR adds Matrix outbound MessagePresentation rendering under com.openclaw.presentation, a Matrix sendPayload implementation with media handling, direct adapter tests, and a new Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the registered Matrix channel facade forwards presentationCapabilities, renderPresentation, and sendPayload or the shared runtime delegate helper supports those hooks; constrain the Matrix sendPayload path to validated com.openclaw.presentation metadata only; then add a facade/delivery regression test plus the docs nav and changelog surfaces required for a user-facing Matrix feature.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace normal Matrix delivery through src/infra/outbound/deliver.ts into the registered outbound adapter from extensions/matrix/src/channel.ts; the PR adds hooks to matrixOutbound, but that registered facade still exposes only sendText, sendMedia, and sendPoll.

## Solution Assessment

No. The metadata shape and plain-text fallback direction are reasonable, but the PR is not the best mergeable solution until the facade/core delivery path is wired and the new Matrix metadata forwarding contract is restricted to validated presentation content.
