## Summary

Keep this PR open. The PR head adds useful Matrix presentation rendering code and tests, and the author clarified the downstream native Matrix client use case, but the patch still does not wire the new hooks through the registered Matrix channel facade, and it forwards arbitrary Matrix event-content fields through the new payload path.

## What This Changes

The PR adds Matrix outbound MessagePresentation rendering under `com.openclaw.presentation`, a Matrix presentation docs page, and outbound tests for rendering, extra content, media URLs, and schema metadata.

## Best Possible Solution

Revise this PR so the Matrix channel facade forwards `presentationCapabilities`, `renderPresentation`, and `sendPayload` or extends the shared runtime delegate helper to do so; add an integration-style delivery regression test that proves Matrix events receive `com.openclaw.presentation`; restrict the Matrix extra-content path to validated presentation metadata; then wire the docs page into navigation and add the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace normal Matrix delivery through `src/infra/outbound/deliver.ts` into the registered outbound adapter from `extensions/matrix/src/channel.ts`: the PR adds hooks on `matrixOutbound`, but that registered adapter still exposes only text, media, and poll forwarding.

## Solution Assessment

No. The presentation renderer and fallback shape are a reasonable direction, but this is not the best mergeable solution until the facade path is wired and the Matrix event-content contract is narrowed to the single validated metadata key.
