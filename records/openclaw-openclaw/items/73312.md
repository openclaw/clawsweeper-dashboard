## Summary

Keep this PR open: the Matrix presentation direction is useful and follows the already-merged `com.openclaw.approval` pattern, but the current diff is still not mergeable because normal Matrix delivery will not reach the new hooks and the event-content/package/docs surfaces need tightening.

## What This Changes

The PR adds Matrix outbound `MessagePresentation` metadata under `com.openclaw.presentation`, fallback text rendering, `sendPayload` media tests, a Matrix presentation docs page, and a root Matrix SDK mirror check.

## Best Possible Solution

Expose Matrix presentation through the registered outbound facade, validate and whitelist only OpenClaw presentation metadata, preserve existing reply fanout semantics, and align docs, changelog, package, and lockfile state before merge.

## Reproduction Assessment

Yes. Source inspection is enough: core delivery only sees hooks on the registered channel outbound adapter, while current Matrix registration forwards only text, media, and poll methods and the PR changes only the runtime adapter.

## Solution Assessment

No. The metadata key and fallback approach are consistent with the existing Matrix approval pattern, but the current patch is not the narrow maintainable solution until facade forwarding, validation, reply fanout, docs, changelog, and package metadata are fixed.
