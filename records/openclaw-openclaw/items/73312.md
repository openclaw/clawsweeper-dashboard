## What This Changes

The PR adds Matrix outbound MessagePresentation rendering, `com.openclaw.presentation` Matrix event metadata, docs, outbound tests, and a root Matrix SDK mirror check.

## Best Possible Solution

Keep the feature direction, but merge only after the Matrix facade forwards payload rendering and sending, Matrix event content is constrained to validated presentation metadata, docs/dependencies are aligned, and redacted real Matrix proof is added.

## Reproduction Assessment

Yes. Source inspection reproduces the PR defect: current main reads `renderPresentation` and `sendPayload` from the registered Matrix channel facade, while the PR adds those hooks only to the runtime outbound adapter.

## Solution Assessment

No. The metadata format and fallback idea match the existing Matrix approval metadata pattern, but the implementation is not the narrowest maintainable fix until facade forwarding, event-content validation, dependency/docs alignment, and real behavior proof are handled.
