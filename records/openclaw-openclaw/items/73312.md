## Summary

Keep open. The PR targets a real Matrix presentation gap, but the current head still leaves normal Matrix delivery unable to reach the new hooks and keeps the Matrix event-content forwarding contract too broad.

## What This Changes

The PR adds Matrix MessagePresentation metadata rendering, Matrix outbound sendPayload handling, targeted outbound tests, and a new Matrix presentation docs page.

## Best Possible Solution

Revise this PR so the Matrix registered outbound facade exposes the presentation/sendPayload hooks, the Matrix metadata path accepts only validated com.openclaw.presentation content, and the public docs navigation plus changelog are updated before merge.

Yes. A high-confidence source reproduction is to trace normal delivery through loadChannelOutboundAdapter into extensions/matrix/src/channel.ts: the PR adds hooks to matrixOutbound, but the registered facade still exposes only sendText, sendMedia, and sendPoll.

## Solution Assessment

No. The renderer and fallback format are a reasonable direction, but the current patch is not the narrowest mergeable solution until the facade path is wired and the Matrix event-content contract is constrained.
