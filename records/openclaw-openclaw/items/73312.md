## Summary

Keep this PR open. The feature direction is valid and the author’s comments explain a useful downstream Matrix/iOS progressive-rendering flow, but the latest head still adds presentation hooks only to the Matrix runtime adapter, not to the registered Matrix channel facade that normal delivery uses. The PR also forwards arbitrary Matrix event-content fields through `extraContent`, so it needs revision before merge.

## What This Changes

The PR adds Matrix MessagePresentation rendering on `matrixOutbound`, forwards Matrix `extraContent` through `sendPayload`, adds outbound tests for presentation and media handling, and adds a Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the Matrix channel facade forwards `presentationCapabilities`, `renderPresentation`, and `sendPayload`, or extend `createRuntimeOutboundDelegates` to support those hooks; add a delivery or facade regression test proving MessagePresentation reaches Matrix events; whitelist and validate only `com.openclaw.presentation` metadata for this path; then wire the docs nav and required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace normal Matrix delivery through `src/infra/outbound/deliver.ts` into the registered outbound handler from `extensions/matrix/src/channel.ts`: the PR adds hooks on `matrixOutbound`, but the registered facade still exposes only `sendText`, `sendMedia`, and `sendPoll`.

## Solution Assessment

No. The runtime renderer and fallback format are reasonable, but the PR is not the best mergeable solution until the facade path is wired and the Matrix metadata forwarding contract is constrained to validated presentation content.
