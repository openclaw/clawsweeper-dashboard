## Summary

Keep open. The downstream Matrix client use case is valid and follows the merged approval-metadata precedent, but the current PR still has merge-blocking source-level issues in the registered Matrix outbound facade, event-content validation, media reply fanout, dependency alignment, docs navigation, and changelog coverage.

## What This Changes

The PR adds Matrix MessagePresentation rendering and sendPayload support under com.openclaw.presentation, with outbound tests, a Matrix docs page, and a root Matrix SDK mirror dependency update.

## Best Possible Solution

Revise or replace the PR so Matrix presentation metadata is exposed through the registered channel facade, constrained to validated com.openclaw.presentation content, and shipped with preserved media/reply semantics plus aligned docs, changelog, package, and lockfile updates.

## Reproduction Assessment

Yes. Source inspection shows core delivery can only call renderPresentation/sendPayload from the registered Matrix outbound facade, while current main and the PR file list leave that facade limited to sendText, sendMedia, and sendPoll.

## Solution Assessment

No. The metadata format and fallback direction match the existing Matrix approval metadata pattern, but the patch is not the narrowest mergeable solution until facade forwarding, event-content validation, reply fanout, dependency sync, docs nav, and changelog are fixed.
