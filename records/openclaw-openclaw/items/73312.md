## Summary

Keep open: the feature direction fits the existing MessagePresentation and Matrix approval metadata patterns, but the current PR still misses the registered Matrix facade path and leaves Matrix event-content forwarding and media reply fanout unsafe.

## What This Changes

The PR adds Matrix MessagePresentation rendering, `sendPayload` handling with `com.openclaw.presentation` event content, outbound tests, Matrix docs, and a root Matrix SDK mirror dependency check.

## Best Possible Solution

Wire the generic runtime-forwarder/facade path for `renderPresentation` and `sendPayload`, constrain Matrix extra content to validated presentation metadata, preserve reply fanout semantics for media sequences, and align docs, changelog, package, and lockfile state before merge.

## Reproduction Assessment

Yes. Source inspection shows core calls `renderPresentation` and `sendPayload` on the registered outbound adapter, while Matrix registration currently forwards only `sendText`, `sendMedia`, and `sendPoll`; the PR diff only changes the runtime adapter.

## Solution Assessment

No. The metadata format and fallback approach are consistent with the existing `com.openclaw.approval` pattern, but the patch is not mergeable until facade wiring, event-content validation, reply fanout, docs, changelog, and dependency alignment are fixed.
