## Summary

This PR is useful and aligned with the existing Matrix approval-metadata direction, including the author's downstream iOS client context, but the current diff is not mergeable because the new runtime hooks do not reach the registered Matrix channel facade and the send path still has reply, content-forwarding, docs, changelog, and package-lock blockers.

## What This Changes

The PR adds Matrix outbound MessagePresentation rendering, sendPayload-based Matrix extraContent forwarding, outbound tests, a Matrix presentation docs page, and a root Matrix SDK mirror dependency/check.

## Best Possible Solution

Revise the PR so the registered Matrix facade exposes presentation/sendPayload hooks, Matrix custom content is constrained to validated OpenClaw metadata, reply fanout follows the shared policy, and docs, changelog, lockfile, and dependency versions align with current main.

## Reproduction Assessment

Yes. A high-confidence source trace shows core delivery calls renderPresentation/sendPayload from the registered outbound adapter, while current Matrix registration forwards only sendText, sendMedia, and sendPoll, and the PR diff does not change that path.

## Solution Assessment

No. The metadata format and fallback direction fit the existing Matrix approval metadata precedent, but the current implementation is not the narrowest maintainable solution until facade wiring, content validation, reply fanout, docs/changelog, and package metadata are fixed.
