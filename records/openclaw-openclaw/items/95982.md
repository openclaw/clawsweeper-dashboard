## Summary

Keep open: the reported parser bug is real and this PR is focused, but the new guard is overbroad for Windows paths and still undercovers marker-free Python block openers, so it should not merge as-is.

## What This Changes

The PR adds a code-context-marker guard to `looksLikeWindowsPathPrefix` and regression tests for Python/shell block-opener newlines in repaired JSON tool-call strings.

## Best Possible Solution

Refine the helper to classify the actual trailing candidate rather than the whole accumulated tail, preserve Windows path repair after ordinary punctuation, add focused regressions for both edge classes, and then provide real behavior proof.

## Reproduction Assessment

Yes. Source inspection plus the read-only regex probe gives a high-confidence path for the PR defect: marker-free `if x:` still matches the Windows-path regex, while unrelated markers earlier in the tail can suppress real path repair.

## Solution Assessment

No. The PR is a plausible mitigation for the observed heredoc shape, but the best fix needs to classify only the actual trailing code/path candidate and preserve the merged Windows-path behavior from the earlier repair.
