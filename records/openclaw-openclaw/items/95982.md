## What This Changes

The PR adds a code-context marker guard to `looksLikeWindowsPathPrefix` and regression tests for Python/shell block-opener newline repair.

## Best Possible Solution

Revise the parser heuristic so code-context rejection is scoped to the actual candidate suffix, preserves the pinned Windows-path repair cases, covers bare single-letter block openers, and is backed by real after-fix OpenClaw tool-run proof.

Yes, source-level reproduction is high confidence: the linked issue has byte-level v2026.6.9 failures, and a direct predicate probe shows the current and proposed heuristic behavior. I did not run a full real agent cron reproduction in this read-only review.

## Solution Assessment

No. The parser helper is the right layer, but this whole-tail guard is both too broad for existing Windows paths and too narrow for bare code openers; a candidate-suffix-aware rule is safer.

- **[P1] Scope the marker check to the candidate suffix:** `src/llm/utils/json-parse.ts:123-125`
  - body: The new guard checks the entire 160-character tail before the Windows-path regex. That disables repair for real unescaped Windows paths whenever prose or code punctuation appears earlier in the tail, for example `see, C:\new` or `Bob's C:\new`, so JSON parsing can turn `\n` into a newline instead of preserving the path. Limit the code-context rejection to the actual candidate segment or add an equivalent regression-preserving path check.
  - confidence: 0.91
- **[P2] Reject bare single-letter code prefixes:** `src/llm/utils/json-parse.ts:123-125`
  - body: The guard only helps when prior marker punctuation exists in the tail. If a tool argument starts with `if x:\n` or `if r:\n`, the proposed predicate still matches `x:` or `r:` as a Windows drive prefix and emits a literal backslash-n, so add isolated regressions and update the predicate for this headline case.
  - confidence: 0.86
