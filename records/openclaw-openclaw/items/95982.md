## Summary

Keep open: this is a focused candidate fix for a real shared-parser newline corruption bug, but the patch still has compatibility defects and lacks after-fix real OpenClaw agent/tool proof.

## What This Changes

The PR adds a guard to `looksLikeWindowsPathPrefix` and regression tests so code-block tails ending in `:` do not cause repaired JSON strings to preserve literal `\n` where real newlines are expected.

## Best Possible Solution

Keep the parser repair focused on the actual candidate suffix or use a stronger per-string signal, preserve pinned Windows-path repair behavior, cover the remaining bare code-opener cases, and add redacted real behavior proof before merge.

## Reproduction Assessment

Yes from source inspection and linked issue evidence: current `main` treats a prefix like `if x:` as a Windows path candidate, and the linked issue shows byte-level shipped failures after Python block openers. I did not run a live OpenClaw cron/tool repro in this read-only review.

## Solution Assessment

No. The parser helper is the right layer, but this whole-tail marker guard is too broad for existing Windows paths and too narrow for bare code openers; the safer fix is a more precise candidate-suffix distinction with focused regressions.
