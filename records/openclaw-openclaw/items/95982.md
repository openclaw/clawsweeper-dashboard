## Summary

Keep open: the fix targets the right parser, but the new guard regresses valid Windows-path repair when punctuation appears earlier in the tail and still misses bare `if x:` / `if r:` openers; external after-fix real behavior proof is also missing.

## What This Changes

Adds a code-context marker guard to `looksLikeWindowsPathPrefix` and regression tests for Python/shell block-opener newlines in repaired tool-call JSON.

## Best Possible Solution

Revise the heuristic so code-context rejection is scoped to the actual candidate suffix, preserves existing Windows-path repair, covers bare single-letter code block openers, and includes after-fix real behavior proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the linked issue has byte-level v2026.6.9 failures, and the parser predicate can be probed directly without writing files. I did not run a full current-main agent cron reproduction in this read-only review.

## Solution Assessment

No. The parser helper is the right layer, but this whole-tail marker guard is too broad for existing Windows paths and too narrow for bare code openers; the safer fix is a more precise candidate-suffix distinction with focused regressions.

- **[P1] Scope the marker check to the candidate suffix:** `src/llm/utils/json-parse.ts:123-125`
  - body: The new guard checks the whole 160-character tail before the Windows-path regex. That disables repair for real unescaped Windows paths whenever prose or code punctuation appears earlier in the tail, e.g. `see, C:\new` or `Bob's C:\new`: current main's regex treats those as path prefixes, but this PR returns false before the path regex and JSON parsing will turn `\n` into a newline. Limit the code-context rejection to the actual candidate segment, or add an equivalent regression-preserving path check.
  - confidence: 0.91
- **[P2] Cover bare single-letter block openers:** `src/llm/utils/json-parse.ts:123-125`
  - body: The PR says `if x:` and `if r:` should stop matching the Windows-path heuristic, but this marker guard does not fire when a tool argument starts with that code body. In that case the existing regex still returns true and the following `\n` is still emitted literally, so add coverage and extend the predicate for bare single-letter Python conditions.
  - confidence: 0.84
