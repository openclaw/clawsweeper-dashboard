## Summary

Keep open. This PR proposes a real new model-profile routing feature that current main still does not implement, but the patch is not ready because routed refs are not coordinated with async provider preparation and the public params-inheritance semantics remain unresolved alongside https://github.com/openclaw/openclaw/issues/59168 and https://github.com/openclaw/openclaw/issues/62629.

## What This Changes

Adds `routeTo` and per-entry `contextTokens` to `agents.defaults.models`, routes model resolution through the configured upstream ref, updates schema, and adds focused agent tests.

## Best Possible Solution

Define a backward-compatible model profile routing contract, then land a rebased patch that applies the effective routed ref consistently across schema/help/docs, picker/listing, async preparation, runtime resolution, params, context budgeting, sessions, and tests.

## Reproduction Assessment

Yes for the PR blockers. Source inspection shows `resolveModelAsync` prepares the original provider/model before calling `resolveModelWithRegistry`, while the PR only computes the routed ref inside `resolveModelWithRegistry`.

## Solution Assessment

No, not as-is. A `routeTo` field is a plausible direction, but the patch must route before async provider preparation and settle params inheritance/documentation before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prepare routed providers before dynamic resolution:** `src/agents/pi-embedded-runner/model.ts:625-628`
  - body: `resolveModelWithRegistry` rewrites to the routed provider/model here, but `resolveModelAsync` still calls `prepareProviderDynamicModel` with the original ref before this resolver runs. A cross-provider route to a dynamic-only model prepares the wrong provider and can still return an unknown model. Compute the effective routed ref before the async prepare step and use it for both prepare and resolution.
  - confidence: 0.91
- **[P2] Define params inheritance for routed refs:** `src/agents/context.ts:407-410`
  - body: This path reads `params` from the original derived ref while checking Anthropic 1M support against the routed upstream ref. If the upstream entry has `params.context1m` and the derived entry omits `params` and `contextTokens`, the budget falls through. Either inherit/fallback to routed params or explicitly document no inheritance and cover that case.
  - confidence: 0.86
