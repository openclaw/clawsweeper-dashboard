## Summary

Keep open: current main still lacks the proposed routed model-profile capability, but this PR is not mergeable as-is because routed dynamic-provider preparation, params/context semantics, and real behavior proof remain unresolved.

## What This Changes

The PR adds `routeTo` and per-entry `contextTokens` to `agents.defaults.models`, routes model resolution through the configured upstream ref, updates generated schema, and adds focused tests.

## Best Possible Solution

Define a backward-compatible model-profile routing contract, then rebase a complete patch that applies the effective routed ref consistently across schema/help/docs, picker/listing, async preparation, runtime resolution, params, context budgeting, sessions, and tests.

## Reproduction Assessment

Yes for the PR blockers by source inspection: `resolveModelAsync` prepares the original provider/model before `resolveModelWithRegistry`, while the PR only routes inside `resolveModelWithRegistry`. The context-budget asymmetry is also source-reproducible from the PR's changed `resolveContextTokensForModel` path.

## Solution Assessment

No, not as-is. `routeTo` is a plausible direction, but the patch must route before async provider preparation and explicitly define routed params inheritance or non-inheritance before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Prepare routed providers before dynamic resolution:** `src/agents/pi-embedded-runner/model.ts:616-633`
  - body: `resolveModelWithRegistry` rewrites to the routed provider/model here, but `resolveModelAsync` still calls `prepareProviderDynamicModel` with the original ref before this resolver runs. A cross-provider route to a dynamic-only target provider can still fail as an unknown model because the routed provider was never prepared.
  - confidence: 0.91
- **[P2] Define params inheritance for routed refs:** `src/agents/context.ts:407-410`
  - body: This path reads `params` from the original derived ref while checking Anthropic 1M support against the routed upstream ref. If the upstream entry has `params.context1m` and the derived entry omits both `params` and `contextTokens`, the budget falls through; either inherit/fallback to routed params or document and test non-inheritance.
  - confidence: 0.86
