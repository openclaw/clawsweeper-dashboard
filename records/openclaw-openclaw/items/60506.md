## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Route before preparing dynamic providers:** `src/agents/pi-embedded-runner/model.ts:619-628`
  - body: `resolveModelAsync` still calls `prepareProviderDynamicModel` with the original derived provider/model before this new routing runs. For dynamic providers that prime caches from `ctx.modelId` (OpenRouter/Ollama-style hooks), a derived ref can prepare the wrong model and then fail to resolve the routed upstream, surfacing `Unknown model` even though `routeTo` is valid. Resolve the effective routed ref before providerConfig/auth preparation and pass that same ref through prepare/run/error paths.
  - confidence: 0.89
- **[P2] Keep context params aligned with the routed model:** `src/agents/context.ts:408-409`
  - body: This checks the routed upstream model for Anthropic 1M support but still reads `context1m` from the original derived ref. Runtime resolution uses the routed upstream entry, so a derived ref that omits duplicate params but routes to an upstream entry with `params.context1m` will execute as 1M-capable while local context budgeting falls through to a smaller/default window. Read params from the same effective model or document that routeTo never inherits upstream params.
  - confidence: 0.84
- **[P2] Document and surface the new config fields:** `src/config/zod-schema.agent-defaults.ts:33-36`
  - body: `routeTo` and per-entry `contextTokens` are new user-facing config fields, but this patch only updates types/zod/generated schema. The config help/labels and current model config docs still describe `agents.defaults.models` entries as alias/params/runtime policy, so users and config tooling do not get a coherent contract. Add the config docs/help/labels or keep the surface out of public config until the model-profile design is decided.
  - confidence: 0.78
