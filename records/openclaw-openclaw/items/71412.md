## Summary

Keep open. Current main still has the reported channel lifecycle wedge: the stop-timeout path returns before clearing the stale task/abort entries, keeps the runtime marked running, and the health monitor then calls start through a guard that no-ops when the stale task remains. The latest release tag checked here still contains the same behavior, and provided related-item context shows open PR #72912 as the active implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Fix the channel manager so a stopChannel abort timeout retires stale task/abort bookkeeping and lets the health monitor start a fresh channel lifecycle.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged channel tasks during health-monitor recovery, clears stale task and abort bookkeeping, marks the old runtime stopped or disconnected, gates late callbacks from retired tasks, updates the regression tests to prove a fresh task starts after timeout, and adds any required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in src/gateway/server-channels.test.ts: a never-settling account task times out on stop, a second start is attempted, and the test expects the replacement start not to happen while the runtime remains running with the timeout error.

## Solution Assessment

Yes for the overall direction, but a maintainable fix should do more than only delete two map entries. The safer path is the open forced-retirement approach that also prevents late callbacks from the retired task from overwriting the replacement lifecycle.
