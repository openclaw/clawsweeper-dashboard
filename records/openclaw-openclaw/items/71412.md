## Summary

Keep open: current main still has the reported stop-timeout path that leaves stale channel task bookkeeping in place, and open PR #72912 remains the active implementation candidate for the forced-retirement fix. The discussion also adds Slack production confirmations, so this is broader than the original Telegram/macOS report.

## What This Changes

Health-monitor channel restarts should recover when `stopChannel` exceeds `CHANNEL_STOP_ABORT_TIMEOUT_MS` instead of leaving a stale task that blocks the next start.

## Best Possible Solution

Review and land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor channel tasks, clears stale bookkeeping, marks runtime state accurately, gates late old-lifecycle callbacks, and updates focused regression coverage plus changelog.

## Reproduction Assessment

Yes at source level. Current main contains the timeout branch, the stale-task start guard, and a unit-level test that still expects the second start to be blocked after a stop timeout.

## Solution Assessment

Yes for the forced-retirement direction in PR #72912. The original suggested map cleanup alone is likely too narrow because late callbacks from the timed-out lifecycle also need to be prevented from overwriting the replacement state.
