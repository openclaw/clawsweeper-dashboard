## Summary

Keep open: current main has the merged recovery-stop improvement from https://github.com/openclaw/openclaw/pull/77686, but the timeout path still leaves the old task registered and a health-monitor restart still no-ops until that old lifecycle settles. Later issue discussion reports post-merge Telegram/macOS failures, and the remaining forced-retirement or task-fencing contract is still unresolved.

## What This Changes

The issue asks OpenClaw to recover channel health-monitor restarts when `stopChannel` times out and leaves stale task bookkeeping that blocks a replacement polling/socket lifecycle.

## Best Possible Solution

Implement a guarded gateway lifecycle recovery contract that can recover from a never-settling timed-out channel task while preventing old task callbacks or transport side effects from corrupting the replacement lifecycle.

## Reproduction Assessment

Yes, at source level. Current main has a deterministic path where a timed-out stop leaves `store.tasks`, the health monitor calls `startChannel`, and the start guard no-ops until the old task settles; existing tests also pin that boundary.

## Solution Assessment

Unclear for the proposed direct cleanup. Clearing the maps fixes the no-op but is not sufficient without task identity/generation fencing or an explicitly accepted force-retire policy for old lifecycle side effects.
