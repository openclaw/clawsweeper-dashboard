## Summary

Keep open. Current main still has the reported gateway lifecycle bug: the stop-timeout branch returns before clearing stale task/abort bookkeeping, marks the runtime running, and the next health-monitor restart can no-op on the stale task. The latest tagged release checked still has the same behavior, and open PR #72912 is the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests that channel stop timeouts release stale lifecycle bookkeeping so health-monitor restarts can launch a replacement channel task.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged channel tasks during health-monitor recovery, clears stale task and abort bookkeeping, marks the old runtime stopped or disconnected, gates late callbacks from retired tasks, updates regression tests to prove a fresh task starts after timeout, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in `src/gateway/server-channels.test.ts`: a never-settling account task times out on stop, a second start is attempted, and the test expects the replacement start not to happen while the runtime remains running with the timeout error.

## Solution Assessment

Yes for the direction, but the best maintainable solution is broader than only deleting two map entries. The fix should retire the wedged lifecycle and gate stale task callbacks, matching the direction described by open PR #72912.
