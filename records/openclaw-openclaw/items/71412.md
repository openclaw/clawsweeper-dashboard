## Summary

Keep open: current main still has the stale-task restart failure at source level. The merged recovery-stop work in https://github.com/openclaw/openclaw/pull/77686 improved manual-versus-recovery state, but a never-settling timed-out lifecycle still keeps `store.tasks` registered so the health-monitor restart can no-op until that old lifecycle settles.

## What This Changes

The issue reports that a channel stop timeout leaves stale channel-manager task state, causing Telegram/Slack health-monitor restarts to no-op and leave the channel dead.

## Best Possible Solution

Implement a guarded gateway lifecycle recovery contract for never-settling stop-timeout tasks, with task identity or generation fencing, focused regression coverage, and redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main can leave a never-settling task in `store.tasks` after a stop timeout, then the health monitor immediately calls `startChannel` and the stale-task guard no-ops; I did not rerun a live Mac sleep or Slack deployment scenario.

## Solution Assessment

Unclear for the original direct cleanup alone. Clearing stale maps addresses the no-op, but the maintainable fix needs task identity, generation fencing, or an explicitly accepted force-retire policy so old lifecycle callbacks cannot clobber a replacement.
