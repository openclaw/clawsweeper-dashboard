## Summary

Keep open. Current main still has the reported lifecycle failure: stop timeout returns before clearing the task/abort bookkeeping, leaves the runtime marked running, and the health monitor then calls a start path that no-ops on the stale task. The issue is already paired with open PR #72912, so cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests a gateway channel lifecycle fix so stop-timeout recovery clears stale task bookkeeping and allows health-monitor restarts to create a fresh channel account lifecycle.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that lets health-monitor recovery force-retire wedged tasks, clears stale task and abort bookkeeping, marks retired runtime state accurately, guards late callbacks from retired tasks, and preserves conservative manual-stop semantics with focused tests and changelog coverage.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in src/gateway/server-channels.test.ts where a never-settling account task times out on stop, a second start is attempted, and the test expects no replacement start while runtime remains running with the timeout error.

## Solution Assessment

Yes for the forced-retirement direction represented by PR #72912; the minimal two-map-delete proposal is not the whole maintainable fix because late callbacks and task-scoped cleanup also need lifecycle gating.
