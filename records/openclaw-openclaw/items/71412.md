## Summary

Keep open. Current main still has the reported channel lifecycle failure: a stop timeout leaves the task registered, marks the runtime running, and the health monitor then calls a start path that no-ops on the stale task. The issue discussion adds Slack and Telegram production confirmations, and open PR #72912 remains the existing implementation candidate to review rather than opening another repair lane.

## What This Changes

Fix the gateway channel stop-timeout path so a wedged channel task does not block health-monitor restarts for Telegram, Slack, and other channel accounts.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires health-monitor-wedged channel tasks, clears stale task and abort bookkeeping, marks the retired runtime stopped or disconnected, prevents late callbacks from retired tasks from overwriting a replacement lifecycle, and updates focused regression tests plus changelog.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in `src/gateway/server-channels.test.ts`: a never-settling account task times out on stop, a second start is attempted, and the test expects the replacement start not to happen while the runtime remains running with the timeout error.

## Solution Assessment

Yes for the forced-retirement direction, but the minimal two-map-delete diff is not the whole maintainable fix. The safer solution should also gate stale task callbacks and preserve manual-stop semantics, matching PR #72912 or an equivalent core gateway patch.
