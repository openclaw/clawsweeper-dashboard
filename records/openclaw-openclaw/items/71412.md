## Summary

Keep open: current main still has the stop-timeout path that leaves stale channel task bookkeeping in place, and open PR #72912 is the active unmerged implementation candidate.

## What This Changes

The issue requests a gateway lifecycle fix so a channel stop timeout does not leave a zombie task that blocks health-monitor restarts.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor channel tasks, clears stale bookkeeping, marks runtime state accurately, gates late callbacks, and updates focused regression coverage plus changelog.

## Reproduction Assessment

Yes at source level. Current main contains a unit-level reproduction where a never-settling task times out on stop, a second start is attempted, and the test still expects no replacement task while runtime remains running with the timeout error.

## Solution Assessment

Yes for the forced-retirement direction in PR #72912. The original two-map-delete patch is too narrow by itself because late callbacks from the old timed-out lifecycle also need to be gated from overwriting the replacement state.
