## Summary

Keep open. Current main still has the reported stop-timeout lifecycle bug, the latest tagged release still contains the same behavior, and the discussion adds Slack production confirmations beyond the original Telegram/macOS report. Open PR #72912 is the implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Fix channel stop timeouts so stale channel tasks are retired and health-monitor restarts can register fresh polling or socket lifecycles.

## Best Possible Solution

Land #72912 or an equivalent gateway lifecycle fix that retires wedged health-monitor restarts, clears stale task and abort bookkeeping, marks the old runtime stopped or disconnected, gates late callbacks from retired tasks, and updates the regression tests to prove a fresh task starts after a stop timeout.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: a never-settling task times out on stop, the next start is attempted, and the test still expects the replacement start not to run.

## Solution Assessment

Yes for the direction, pending PR review. The narrow maintainable solution is a gateway lifecycle retirement path with stale bookkeeping cleanup and late-callback gating, not only a doctor/status diagnostic change.
