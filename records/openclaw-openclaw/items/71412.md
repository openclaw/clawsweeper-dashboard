## Summary

Keep open: current main still leaves a timed-out channel task registered, so the health-monitor stop-then-start path can still no-op at the task guard until the old lifecycle settles. The issue discussion also has post-partial-fix Telegram and Slack production confirmations, so this is not cleanup-closeable.

## What This Changes

The issue asks OpenClaw to recover channel restarts after a stopChannel abort timeout leaves stale task state that blocks health-monitor restarts.

## Best Possible Solution

Implement a guarded gateway lifecycle recovery contract that can recover from a never-settling timed-out channel task while preventing stale task callbacks or plugin transport side effects from corrupting the replacement lifecycle.

## Reproduction Assessment

Yes, at source level: current main has a deterministic path where a timed-out stop leaves store.tasks registered, the health monitor calls startChannel, and startChannelInternal no-ops. I did not rerun a live Telegram sleep/wake or Slack Socket Mode deployment.

## Solution Assessment

Unclear for the proposed direct cleanup alone: deleting stale map entries fixes the no-op, but the maintainable fix also needs task identity, generation fencing, or an explicitly accepted force-retire policy for old lifecycle side effects.
