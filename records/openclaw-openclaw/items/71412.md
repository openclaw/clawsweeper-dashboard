## Summary

Keep open: current main still leaves timed-out channel task bookkeeping in place, so the health monitor's immediate restart can no-op; the prior implementation PRs are closed unmerged and the remaining recovery contract needs maintainer judgment.

## What This Changes

The issue requests that channel stop timeouts stop leaving stale task and abort entries that block health-monitor restarts for Telegram and Slack channel lifecycles.

## Best Possible Solution

Define and implement a gateway lifecycle recovery contract that lets health-monitor recovery start a fresh account after a timed-out stop while fencing or hard-stopping old channel lifecycles before they can process side effects.

## Reproduction Assessment

Yes at source level. Current main has a deterministic path where a stop timeout leaves `store.tasks`, the health monitor calls `startChannel`, and the start guard no-ops; the existing unit test also pins that immediate replacement start remains blocked.

## Solution Assessment

Unclear. Direct map cleanup fixes the manager no-op, but prior PRs showed an old/new lifecycle overlap tradeoff, so maintainers should choose a fenced or hard-stop recovery contract before implementation.
