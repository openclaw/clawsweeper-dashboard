## Summary

Keep open: current main still has the reported stop-timeout path that leaves stale channel task state in place, and #72912 is the active open implementation candidate for the broader forced-retirement fix. The later Slack confirmations make this a cross-channel gateway lifecycle bug rather than only a Telegram/macOS polling report.

## What This Changes

The issue asks `stopChannel` and health-monitor restarts to recover after a channel stop abort timeout instead of leaving a stale task that blocks a fresh channel lifecycle.

## Best Possible Solution

Review and land #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor channel tasks, clears stale bookkeeping, marks runtime state accurately, gates late old-lifecycle callbacks, and updates focused regression coverage plus changelog.

## Reproduction Assessment

Yes at source level. Current main still contains the stale-task start guard, the stop-timeout early return before cleanup, and a unit test that expects a second start to be blocked after the timeout.

## Solution Assessment

No for the exact cleanup-only diff in the issue body; it is likely too narrow because late callbacks from the timed-out lifecycle can still overwrite replacement state. The forced-retirement and lifecycle-gating direction in #72912 is the more maintainable fix.
