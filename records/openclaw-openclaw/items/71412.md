## Summary

Keep open: current main still contains the stop-timeout path that leaves stale channel task bookkeeping in place, and the active implementation path is still an open PR rather than shipped code.

## What This Changes

The issue asks for health-monitor channel restarts to recover after a stopChannel abort timeout by releasing stale task bookkeeping and starting a fresh lifecycle.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged tasks, clears stale bookkeeping, gates late callbacks from retired lifecycles, and updates focused regression coverage plus changelog.

## Reproduction Assessment

Yes at source level. Current main has a focused unit scenario where a never-settling channel task times out on stop, a second start is attempted, and the test still expects no replacement task while runtime remains running with the timeout error.

## Solution Assessment

Yes for the forced-retirement direction tracked by PR #72912. The original two-map-delete patch is too narrow by itself unless late callbacks from the retired lifecycle are also prevented from overwriting the replacement state.
