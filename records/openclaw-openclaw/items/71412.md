## Summary

Keep open: current main still has the stop-timeout restart blocker. The timeout path returns without clearing the stale task/abort entries, the health monitor still does a stop-then-start recovery, and current tests still assert that an immediate replacement start is blocked; the prior implementation PRs are closed unmerged, with the broader force-retire approach explicitly abandoned.

## What This Changes

The issue reports that a channel stop timeout can leave stale lifecycle bookkeeping so health-monitor restarts no-op and Telegram or Slack polling remains silently dead.

## Best Possible Solution

Define and implement a maintainer-approved gateway lifecycle recovery contract: a timed-out health-monitor stop must allow a fresh account lifecycle to start without stale callbacks corrupting state, while either fencing or hard-stopping plugin-owned side effects from the retired lifecycle.

## Reproduction Assessment

Yes at source level. Current main follows the reported chain: stop timeout leaves `store.tasks`, health-monitor calls start, and the start guard no-ops; the existing unit tests also pin that immediate restart remains blocked.

## Solution Assessment

Unclear. The reporter's direct map cleanup fixes the central no-op, but prior reviews found a real old/new lifecycle overlap tradeoff, so the best fix needs explicit maintainer agreement on fencing or hard-stop semantics.
