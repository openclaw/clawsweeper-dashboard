## Summary

Keep open: current main and the latest release still ignore configured Discord `doneHoldMs`/`errorHoldMs` during final status-reaction cleanup, even though the config contract exposes those timing keys and the reporter narrowed the live repro to that behavior.

## What This Changes

The issue requests Discord parity for `messages.statusReactions`, narrowed in comments to honoring configured final done/error hold timing before cleanup.

## Best Possible Solution

Mirror Telegram's merged timing handling in Discord final cleanup, then add focused Discord regression coverage for configured done/error hold delays.

## Reproduction Assessment

Yes. The high-confidence source path is `removeAckAfterReply: true` with custom `messages.statusReactions.timing.doneHoldMs`: Discord reaches `setDone()` and then sleeps the default hold instead of the configured value, matching the reporter's live Discord observation.

## Solution Assessment

Yes. The narrow maintainable fix is to compute the merged Discord status-reaction timing once and use it for final done/error cleanup delays; changing permanent persistence semantics when `removeAckAfterReply` is false should stay out of this bug fix.
