## Summary

Keep this open: current main and v2026.5.5 already contain the Discord lifecycle wiring for explicit `messages.statusReactions.enabled`, but Discord still ignores configured `doneHoldMs`/`errorHoldMs` during final cleanup, so the documented timing behavior is not fully implemented.

## What This Changes

The issue asks for Discord to match the documented `messages.statusReactions` lifecycle, including done/error reaction persistence timing, instead of behaving like legacy ack-only reactions.

## Best Possible Solution

Keep the shipped lifecycle wiring, then make Discord finalization use merged status-reaction timing overrides like Slack and Telegram and clarify the `removeAckAfterReply` interaction if persistent done semantics are desired.

## Reproduction Assessment

Yes, from source inspection: Discord final cleanup currently uses `DEFAULT_TIMING.doneHoldMs/errorHoldMs` instead of the configured `messages.statusReactions.timing` values. I did not live-run Discord in this read-only review.

## Solution Assessment

No, the current implementation is not complete because it wires the lifecycle but does not honor the documented timing override on Discord. The narrow maintainable fix is to introduce a Discord `statusReactionTiming` merge, matching Slack and Telegram, and add focused regression tests.
