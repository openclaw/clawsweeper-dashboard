## Summary

Keep open: current main has Discord lifecycle status reaction wiring, but the narrowed timing bug remains because the final cleanup still ignores configured `doneHoldMs` and `errorHoldMs`. The reporter's live follow-up narrows the issue to that gap, and no canonical open PR or duplicate specifically covers Discord hold timing.

## What This Changes

The issue requests Discord status-reaction finalization to honor configured done/error hold timing before clearing reactions.

## Best Possible Solution

Mirror Telegram's merged timing handling in the Discord finalizer, use it for done/error cleanup delays, and add focused Discord regression coverage without changing persistence semantics.

## Reproduction Assessment

Yes, source-reproducible. With `removeAckAfterReply: true` and custom `messages.statusReactions.timing.doneHoldMs`, current Discord code reaches `setDone()` and then sleeps `DEFAULT_TIMING.doneHoldMs` before clearing, while the reporter observed the same done-flash behavior in a real Discord setup.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse a merged timing object in Discord final cleanup and cover configured `doneHoldMs`/`errorHoldMs`; broader lifecycle or persistence changes are not needed for this bug.
