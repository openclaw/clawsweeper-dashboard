## Summary

Keep open: current main still ignores configured Discord status-reaction final hold timing, so the narrowed doc-backed bug remains after the earlier lifecycle wiring.

## What This Changes

The issue asks Discord status reactions to honor the documented `messages.statusReactions.timing.doneHoldMs` and related lifecycle timing instead of behaving like the legacy ack cleanup path.

## Best Possible Solution

Add a Discord finalizer-level merged timing object, use it for done/error cleanup delays, and cover the custom hold timing with focused Discord tests.

## Reproduction Assessment

Yes by source inspection, with reporter live confirmation: with `removeAckAfterReply: true` and a custom `messages.statusReactions.timing.doneHoldMs`, current Discord reaches `setDone()` and then sleeps the default hold duration before clearing. I did not run a live Discord bot in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to mirror Telegram and Slack by merging `DEFAULT_TIMING` with configured status-reaction timing in the Discord finalizer; changing `removeAckAfterReply: false` persistence semantics should stay out of this bug fix.
