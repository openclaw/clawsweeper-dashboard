## Summary

Keep open: the PR adds a useful per-spawn fast-mode surface, but the advertised ACP behavior is incomplete because ACP initial turns do not consume the patched session fastMode value. The PR also still needs real after-fix behavior proof beyond mocked/unit validation.

## What This Changes

The PR adds a `sessions_spawn.fastMode` boolean, persists it on spawned native subagent and ACP child sessions, adds focused tests, and documents the override in subagent, ACP, thinking, and changelog docs.

## Best Possible Solution

Keep the native subagent support, but route ACP fast-mode overrides through the ACP runtime option/config-control path before the first turn, with regression coverage for a backend advertising `fast_mode`.

## Reproduction Assessment

Yes for the review finding from source inspection: the ACP spawn patch writes `SessionEntry.fastMode`, while ACP runtime initialization and turn control consume `AcpSessionRuntimeOptions` without fastMode. I did not run the PR branch because this is a read-only review.

## Solution Assessment

No: the native subagent part follows the existing session-entry fast-mode resolver, but the ACP path needs to apply the override through ACP runtime config controls or avoid advertising ACP support.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply fastMode to ACP runtime options:** `src/agents/acp-spawn.ts:1324`
  - body: The ACP path now accepts and documents `sessions_spawn.fastMode`, but this line only stores it in OpenClaw's local session entry. ACP child turns are initialized and dispatched through ACP runtime options/control application, which currently include model, thinking, timeout, and backend extras but not `fastMode`, so the first ACP child turn will still use the backend default. Please pass the override through the ACP runtime config-option path before dispatch, or do not expose it for `runtime="acp"`.
  - confidence: 0.89
