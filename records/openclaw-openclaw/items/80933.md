## Summary

Keep open: current main still ignores canonical Anthropic `contextTokens` caps for `claude-cli` runtime budgeting, so CLI compaction can run against the 1M fallback instead of the configured smaller budget. The closed companion PR is useful prior art, but it was not merged and no other direct fix appears to own this issue.

## What This Changes

Fix Claude CLI runtime sessions so explicit canonical Anthropic `contextTokens` budgets are persisted and used by CLI transcript compaction.

## Best Possible Solution

Use the existing runtime-provider alias table to resolve canonical provider `contextTokens` caps before Anthropic fallback windows, then cover resolver, session-store persistence, and CLI compaction budgeting with focused regressions.

## Reproduction Assessment

Yes, by source inspection. A canonical `anthropic/claude-opus-4-7` model routed with `provider: claude-cli` reaches the resolver path that returns the 1M Anthropic fallback before consulting the configured canonical `anthropic` `contextTokens` cap, and CLI compaction then uses the persisted `SessionEntry.contextTokens` budget.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing runtime alias contract for configured cap lookup before fallback defaults, not to add another config option or change compaction policy.
