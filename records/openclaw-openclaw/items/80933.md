## Summary

Keep open: current main still resolves explicit `claude-cli` Anthropic 1M-capable models to the 1M window before scanning canonical `anthropic` `contextTokens`, so session persistence can store an over-large budget and CLI compaction will not trigger at the configured cap. The targeted companion fix at https://github.com/openclaw/openclaw/pull/80937 is closed unmerged, and I did not find a merged or open canonical replacement for this exact bug.

## What This Changes

The issue asks OpenClaw to make `claude-cli` runtime sessions persist and compact against the configured model `contextTokens` budget instead of ignoring it.

## Best Possible Solution

Honor canonical provider `contextTokens` for CLI runtime aliases before Anthropic 1M/default fallbacks, persist that cap into session state, and cover the resolver/session-store/CLI-compaction path with focused regressions.

## Reproduction Assessment

Yes, by source inspection rather than a live OAuth run. `updateSessionStoreAfterAgentRun` passes `providerUsed` and `modelUsed` into `resolveContextTokensForModel`, and with `provider: claude-cli` plus an Anthropic 1M-capable model the resolver returns the 1M value before canonical `anthropic` config lookup.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing runtime-provider alias seam for configured-cap lookup before fallback defaults, instead of adding a new config knob or changing compaction policy.
