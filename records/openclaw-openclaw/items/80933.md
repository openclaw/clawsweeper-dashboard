## Summary

Keep this open. Current main still does not show an alias-aware lookup from the `claude-cli` runtime provider back to canonical `models.providers.anthropic` `contextTokens`, and the directly linked fix PR is closed unmerged rather than a safe canonical landing path.

## What This Changes

The issue reports that Claude CLI runtime sessions ignore configured `contextTokens`, causing CLI transcript compaction to run against a missing or inflated budget.

## Best Possible Solution

Resolve `claude-cli` runtime-provider aliases to the canonical Anthropic provider before configured-cap lookup, persist that configured cap on the session, and cover resolver, session-store, and CLI compaction behavior with focused regressions.

## Reproduction Assessment

Yes, by source inspection: Claude CLI run metadata is persisted with the runtime provider, while current budget resolution checks configured caps under that provider instead of canonical `anthropic` before falling back to the fixed 1M cap. I did not run a live Claude CLI OAuth compaction session.

## Solution Assessment

Yes. The narrow maintainable solution is alias-aware configured-cap lookup at the existing context resolver/session-store boundary, not a new setting or a compaction policy change.
