## Summary

Keep open: current main and the latest shipped tag still have the resolver ordering that can ignore a configured Claude CLI context-token cap, while the directly related fix PR was closed unmerged.

## What This Changes

The issue asks for Claude CLI runtime sessions to persist and compact against configured `models.providers.anthropic.models[].contextTokens` budgets.

## Best Possible Solution

Make explicit configured `contextTokens` caps authoritative for canonical Anthropic models routed through `claude-cli`, using the existing runtime alias mapping before Anthropic 1M fallback defaults.

## Reproduction Assessment

Yes, by source inspection. A Claude CLI run for a canonical Anthropic 1M model reaches session persistence with `provider: claude-cli`, and current resolver ordering can return 1M before checking the configured canonical Anthropic `contextTokens` cap.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve runtime-provider aliases for explicit configured `contextTokens` lookup before fallback context-window defaults, without changing auth, runtime selection, or compaction policy.
