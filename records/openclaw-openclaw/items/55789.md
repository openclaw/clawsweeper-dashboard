## What This Changes

The PR adds a gateway request-context guard that rejects reserved `x-openclaw-session-key` prefixes for Chat Completions and OpenResponses HTTP requests, maps that specific error to HTTP 400, and adds focused tests.

## Best Possible Solution

Land the shared guard after adding the reserved-namespace docs exception and redacted real HTTP proof, while tracking any broader `/tools/invoke`, hook, or history policy separately.

## Reproduction Assessment

Yes, source-reproducible. Current main returns explicit `x-openclaw-session-key` headers unchanged and both affected HTTP handlers use that shared context before dispatch; I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Yes for the runtime shape: a shared resolver guard with a dedicated error type is the narrow maintainable fix. The remaining question is not code shape, but docs, real behavior proof, and maintainer acceptance of the compatibility break.
