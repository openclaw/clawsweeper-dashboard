## Summary

Keep open: the runtime patch appears correct and current main still has the reported gap, but this external PR still needs contributor-supplied real HTTP proof before merge and a small public docs update for the changed header contract.

## What This Changes

Adds a gateway request-context guard that rejects internal `x-openclaw-session-key` overrides on the OpenAI-compatible and OpenResponses HTTP endpoints, returns `400 invalid_request_error`, and adds focused request-context and endpoint coverage.

## Best Possible Solution

Land the shared request-context guard after adding the docs exception and contributor-supplied real HTTP proof, while keeping the companion session-history hardening tracked separately.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit `x-openclaw-session-key` values are returned unchanged and both OpenAI-compatible HTTP handlers consume the shared context before dispatch; I did not run a live HTTP repro in this read-only review.

## Solution Assessment

Yes for the runtime approach. A shared resolver guard plus a dedicated `GatewaySessionKeyOverrideError` is the narrow maintainable fix; the remaining needs are docs and real behavior proof, not a different code design.
