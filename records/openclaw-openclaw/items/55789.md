## Summary

Keep open: current main still forwards reserved session-key headers unchanged, and the PR’s narrowed runtime approach looks correct, but contributor-supplied real HTTP proof is still missing and the public gateway docs would be inaccurate without a reserved-namespace exception.

## What This Changes

The PR adds a shared gateway request-context guard that rejects reserved internal `x-openclaw-session-key` overrides on the OpenAI-compatible and OpenResponses HTTP handlers with a 400 response, plus resolver and endpoint tests.

## Best Possible Solution

Land the shared request-context guard after adding the docs exception and contributor-supplied real HTTP proof, while keeping the separate session-history hardening in https://github.com/openclaw/openclaw/pull/55928 distinct.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit `x-openclaw-session-key` values are returned unchanged and both OpenAI-compatible HTTP handlers consume the shared context before dispatch; I did not run a live HTTP repro in this read-only review.

## Solution Assessment

Yes for the runtime approach. A shared resolver guard plus a dedicated `GatewaySessionKeyOverrideError` is the narrow maintainable fix; the remaining needs are docs and contributor real behavior proof, not a different code design.
