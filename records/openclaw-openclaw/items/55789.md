## What This Changes

The PR adds a reserved-prefix guard for `x-openclaw-session-key` in OpenAI-compatible and OpenResponses HTTP request context, maps that specific error to 400, and adds focused tests.

## Best Possible Solution

Land the narrow gateway guard only after maintainers accept the reserved-namespace compatibility break, document the exception, and see redacted real HTTP proof for both rejected reserved keys and accepted ordinary custom keys.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit `x-openclaw-session-key` values are returned directly by `resolveSessionKey`, while the PR adds a reserved-prefix rejection path; I did not run a live gateway in this read-only review.

## Solution Assessment

Mostly yes. The dedicated error class keeps the 400 mapping narrow and the guard is placed in the shared request-context path for the two targeted endpoints, but the compatibility/docs decision and related HTTP ingress scope remain maintainer calls.
