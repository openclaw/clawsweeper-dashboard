## Summary

Keep open: current main still forwards explicit reserved session keys, and this PR is a focused security hardening, but it still needs contributor-supplied real HTTP proof before merge plus a small docs update.

## What This Changes

The PR rejects reserved `x-openclaw-session-key` namespaces on OpenAI-compatible and OpenResponses HTTP requests, returns `400 invalid_request_error`, and adds focused gateway regression tests.

## Best Possible Solution

Land the shared request-context guard after adding a short docs note for the reserved namespace 400 behavior and redacted real HTTP proof from the contributor.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: send `x-openclaw-session-key: agent:main:subagent:worker` to either OpenAI-compatible HTTP endpoint on current main and the shared resolver returns it unchanged before dispatch. I did not run a live request in this read-only review.

## Solution Assessment

Yes for the runtime approach. A shared resolver guard plus a dedicated `GatewaySessionKeyOverrideError` is the narrow maintainable fix; the remaining gaps are docs and real behavior proof.
