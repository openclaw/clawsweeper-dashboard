## Summary

Keep open. Current main still accepts explicit `x-openclaw-session-key` values without an internal-namespace guard, and this PR is a focused, plausible hardening patch with no blocking code finding; the remaining blockers are contributor real-behavior proof and maintainer acceptance of the compatibility break.

## What This Changes

The PR adds a reserved session-key namespace guard for OpenAI-compatible and OpenResponses HTTP headers, maps only that guard to `400 invalid_request_error`, and adds request-context plus endpoint tests.

## Best Possible Solution

Land the narrow guard only after maintainers accept the reserved-namespace compatibility break, add docs or release-note coverage, and get redacted real HTTP proof for both rejected reserved keys and accepted ordinary keys.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit `x-openclaw-session-key` headers are returned directly, while the PR rejects reserved internal prefixes before dispatch.

## Solution Assessment

Mostly yes. The dedicated error class keeps the 400 mapping narrow and the guard is scoped to the two stated endpoints; the remaining question is the maintainer policy for the compatibility break and broader HTTP ingress coverage.
