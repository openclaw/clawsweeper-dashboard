## Summary

Keep open: current main still forwards explicit internal-looking `x-openclaw-session-key` values, while the PR head adds a narrow guard and the earlier broad-catch review issue has been addressed; the remaining blockers are contributor real behavior proof and a small docs contract update.

## What This Changes

The branch adds a shared gateway guard that rejects reserved internal `x-openclaw-session-key` overrides in the OpenAI-compatible and OpenResponses HTTP handlers and adds resolver plus endpoint coverage.

## Best Possible Solution

Land the shared request-context guard after adding the docs exception and contributor-supplied real HTTP proof, while keeping the session-history hardening in https://github.com/openclaw/openclaw/pull/55928 separate.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit `x-openclaw-session-key` values are returned unchanged and both OpenAI-compatible HTTP handlers consume the shared context before dispatch; I did not run a live HTTP repro in this read-only review.

## Solution Assessment

Yes for the runtime approach. A shared resolver guard plus a dedicated `GatewaySessionKeyOverrideError` is the narrow maintainable fix; the remaining needs are docs and contributor real behavior proof, not a different code design.
