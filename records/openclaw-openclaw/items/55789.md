## Summary

Keep open: current main still accepts explicit OpenAI-compatible HTTP session-key overrides, and this PR is a focused hardening patch, but it still needs real HTTP proof and an explicit compatibility/docs decision before merge.

## What This Changes

The PR rejects reserved `x-openclaw-session-key` values for OpenAI-compatible and OpenResponses HTTP requests, maps that guard to a 400 response, and adds request-context plus endpoint tests.

## Best Possible Solution

Land the narrow guard only if maintainers accept the reserved-namespace compatibility break, with docs or release-note context and redacted real HTTP proof for both rejected reserved keys and accepted ordinary keys.

## Reproduction Assessment

Yes, from source inspection. Current main returns any explicit `x-openclaw-session-key`, while the PR rejects reserved internal prefixes before dispatch; I did not run a live gateway in this read-only review.

## Solution Assessment

Mostly yes. The dedicated error class keeps the 400 mapping narrow, but maintainers still need to decide whether this compatibility break should land narrowly or as part of a broader HTTP session-key policy.
