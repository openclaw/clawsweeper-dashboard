## Summary

## What This Changes

Adds a reserved internal session-key namespace guard to the OpenAI-compatible Chat Completions and OpenResponses HTTP request-context path, returning 400 errors and adding focused tests.

## Best Possible Solution

Rebase and land the shared OpenAI/OpenResponses request-context guard with docs and real HTTP proof, while tracking any broader all-HTTP ingress policy separately if maintainers want it.

## Reproduction Assessment

Yes from source inspection: current main returns explicit `x-openclaw-session-key` headers unchanged, so a reserved key can reach the OpenAI/OpenResponses agent command path. I did not run a live HTTP repro in this read-only review.

## Solution Assessment

Mostly yes: `resolveGatewayRequestContext` is the narrow shared boundary for these two HTTP surfaces, and the dedicated error type avoids reclassifying unrelated context failures. It still needs docs, real behavior proof, and a rebase over newer gateway context handling.
