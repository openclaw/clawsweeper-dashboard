## Summary

Keep open. The runtime guard is a focused gateway security hardening and appears correct, but the PR still needs a docs update for the narrowed public HTTP header contract and external real behavior proof before merge.

## What This Changes

The PR adds a gateway HTTP guard that rejects `x-openclaw-session-key` overrides targeting `subagent:`, `acp:`, or `cron:` namespaces in OpenAI-compatible and OpenResponses endpoints.

## Best Possible Solution

Land the shared request-context guard after documenting the reserved namespace rejection and adding redacted real HTTP behavior proof for the 400 response.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send `x-openclaw-session-key: agent:main:subagent:worker` to either OpenAI-compatible HTTP endpoint on current main and the shared resolver returns it unchanged. I did not run endpoint tests because this was a read-only review.

## Solution Assessment

Yes for the runtime approach. A shared request-context guard plus narrow `GatewaySessionKeyOverrideError` handling is the maintainable fix; the remaining gaps are docs and real behavior proof, not a different code design.
