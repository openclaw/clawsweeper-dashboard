## Summary

Keep open. Current main still accepts explicit internal session-key overrides on the OpenAI-compatible HTTP paths, while PR head e58b3893aee1fcece10a02f8ff24bde298b45ec5 adds a narrow guard and has green checks; I found no runtime or security blocker, but the public docs and changelog still need the corresponding update before merge.

## What This Changes

The branch rejects explicit OpenAI-compatible and OpenResponses `x-openclaw-session-key` values that target `subagent:`, `acp:`, or `cron:` internal namespaces, maps that dedicated error to 400 responses, and adds request-context plus endpoint regression tests.

## Best Possible Solution

Land the narrow guard after adding a short docs note for both HTTP compatibility pages and a single-line Unreleased changelog entry, while keeping #55928 as the separate session-history follow-up.

## Reproduction Assessment

Yes. On current main, a request with `x-openclaw-session-key: agent:main:subagent:worker` follows a direct source path: `resolveSessionKey` returns it unchanged and both HTTP endpoint handlers use that resolved session key for the agent turn.

## Solution Assessment

Yes, with docs and changelog follow-through. Blocking in shared request-context resolution and translating only `GatewaySessionKeyOverrideError` to a 400 is the narrowest maintainable fix for these two endpoints without broadening unrelated HTTP surfaces.
