## Summary

Keep open for maintainer review. Current main still accepts explicit internal session-key overrides on the OpenAI-compatible HTTP surfaces, while the refreshed PR head applies a narrow guard with public checks passing; it should not be auto-closed or replaced. The only actionable gap I found is docs alignment for the changed header contract.

## What This Changes

The PR adds reserved-prefix validation for explicit `x-openclaw-session-key` values in gateway request-context resolution, maps the dedicated error to 400 responses in `/v1/chat/completions` and `/v1/responses`, and adds request-context plus endpoint regression tests.

## Best Possible Solution

Land the narrow header-override guard with a short public docs note for both HTTP compatibility pages, keep #55928 as the separate session-history follow-up, and only broaden the policy to other HTTP ingress paths if maintainers explicitly want all external session-key inputs to share the same reserved-namespace rule.

## Reproduction Assessment

Yes. On current main, source inspection gives a high-confidence reproduction path: send either compatibility endpoint a request with `x-openclaw-session-key: agent:main:subagent:worker`, and the shared resolver returns that value unchanged before the endpoint dispatches into the agent path.

## Solution Assessment

Yes, with docs follow-through. Blocking in shared request-context resolution and translating only the dedicated `GatewaySessionKeyOverrideError` to a 400 is the narrowest maintainable fix for these two endpoints while preserving normal custom session keys and avoiding the earlier internal-error exposure problem.
