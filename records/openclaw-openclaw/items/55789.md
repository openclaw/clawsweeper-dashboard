## Summary

Keep open: current main still forwards explicit reserved session-key headers unchanged, and this PR is a focused hardening path, but the external PR still has only test/CI proof and needs docs plus maintainer acceptance of the intentional compatibility break before merge.

## What This Changes

The PR adds a gateway guard that rejects `subagent:`, `acp:`, and `cron:` `x-openclaw-session-key` overrides on OpenAI-compatible HTTP endpoints and maps that dedicated error to `400 invalid_request_error`.

## Best Possible Solution

Land the shared request-context guard after documenting the reserved namespace exception and adding redacted real HTTP proof, while keeping the session-history and tools-invoke policy work separate unless maintainers choose a broader HTTP-ingress hardening pass.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send a current-main OpenAI-compatible HTTP request with `x-openclaw-session-key: agent:main:subagent:worker`, and the shared context resolver returns that explicit key unchanged before dispatch. I did not run a live HTTP repro in this read-only review.

## Solution Assessment

Yes for the runtime shape: a shared request-context guard plus a dedicated error type is the narrow maintainable fix. The remaining merge blockers are docs, real behavior proof, and maintainer acceptance of the scoped fail-closed policy.
