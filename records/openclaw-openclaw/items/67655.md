## Summary

Keep open: current main still only fails closed for Windows shell wrappers when `cmdInvocation` is true, while PowerShell/pwsh wrappers are recognized separately and can reach the policy with `shellWrapperInvocation=true`. The patch direction is narrow and useful, but this external security-sensitive PR still needs redacted real Windows runtime proof before merge.

## What This Changes

The PR changes node-host exec policy, regression tests, node docs/troubleshooting, and changelog so Windows PowerShell/pwsh shell-wrapper runs in allowlist mode require approval like cmd.exe wrappers.

## Best Possible Solution

Land this focused policy/docs/test change after redacted Windows runtime proof shows PowerShell or pwsh wrapper runs are denied in allowlist mode until approval, with exact-head checks green.

## Reproduction Assessment

Yes, source-reproducible. Current main recognizes PowerShell/pwsh shell wrappers and passes `shellWrapperInvocation` separately from `cmdInvocation`, while the policy guard only blocks Windows wrappers when `cmdInvocation` is true.

## Solution Assessment

Yes for the code direction. Changing the fail-closed predicate to use `shellWrapperInvocation` is the narrow maintainable fix; merge should wait for real Windows runtime proof rather than a different implementation.
