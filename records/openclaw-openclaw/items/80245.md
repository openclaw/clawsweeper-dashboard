## Summary

Keep open: the patch is not mergeable because it strips the only core-recognized send content before send validation, so the reported `message` plus `card` call would fail before plugin card handling. The external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds normalization and tests that move `message`/`text` into `_cardNotificationHint` and delete them when a send action includes a `card` object.

## Best Possible Solution

Treat raw card sends as first-class content before stripping visible text, or keep the fix plugin-owned; either path should preserve core validation and be proven with a real Feishu/Lark card send.

## Reproduction Assessment

No high-confidence live reproduction is available. Source review does show the PR's reported path would become `message required` because raw `card` is not currently counted as send content after `message` is removed.

## Solution Assessment

No. The proposed normalization runs too early for the current send pipeline; a safer fix would either use the established `presentation` payload path or update the send contract so `card` is recognized before deleting text.
