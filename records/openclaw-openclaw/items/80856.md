## Summary

Keep open: the branch fixes direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` leaf edits, but it still accepts parent `meta` object writes that can contain those stamped leaves and then be overwritten while reporting success. The contributor supplied real terminal proof for the direct-path behavior, so the remaining blocker is the incomplete parent-value handling.

## What This Changes

The PR adds config CLI guards for auto-managed `meta.lastTouchedVersion` and `meta.lastTouchedAt` set/patch/batch/unset operations plus regression tests.

## Best Possible Solution

Keep `stampConfigVersion` as the system-owned writer, and reject both direct leaf edits and parent `meta` object operations that contain `lastTouchedVersion` or `lastTouchedAt`, with focused regression coverage.

## Reproduction Assessment

Yes. The linked issue and comments provide live direct-command proof, and source inspection shows user values are overwritten after mutation during `stampConfigVersion`; the remaining parent-path gap is also source-reproducible through `config set meta ... --merge` or `config patch --replace-path meta`.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the patch is incomplete because parent `meta` object operations can still carry stamped leaves that are accepted and then overwritten.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject parent meta writes with stamped leaves:** `src/cli/config-cli.ts:1389-1391`
  - body: Because this guard only checks `operation.requestedPath`, parent writes such as `config set meta '{"lastTouchedAt":"1999-01-01T00:00:00.000Z"}' --strict-json --merge` or `config patch --replace-path meta` can still reach `replaceConfigFile`. `stampConfigVersion` then overwrites the supplied leaf while the CLI reports success, so detect auto-managed leaves inside `operation.value` or reject parent `meta` values containing them.
