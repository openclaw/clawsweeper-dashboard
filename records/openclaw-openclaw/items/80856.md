## Summary

Keep open: the direct-path guard is useful and now has real terminal proof, but the patch still lets parent `meta` object writes carrying stamped leaves reach the writer and be overwritten while reporting success.

## What This Changes

The branch adds config CLI validation to reject direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set, patch, batch, and unset requests, plus focused regression tests.

## Best Possible Solution

Keep `stampConfigVersion` as the system-owned writer, but reject direct stamped leaves and parent object, replace, or batch operations whose value tree contains those leaves, with regression coverage for each entry path.

## Reproduction Assessment

Yes. The linked issue and current source give a high-confidence path for the original false-success bug, and the remaining PR gap is source-reproducible through parent `meta` operations such as `config set meta ... --strict-json --merge` or `config patch --replace-path meta`.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the current patch is incomplete because it does not inspect parent `meta` value trees that contain the auto-managed leaves.
