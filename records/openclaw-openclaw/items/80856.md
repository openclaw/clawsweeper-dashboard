## Summary

Keep open: this is a valid narrow config CLI bug fix and the contributor has now supplied after-fix terminal proof, but the proposed guard still only checks the requested path and leaves parent `meta` object writes able to carry stamped leaves that will be overwritten while the CLI reports success.

## What This Changes

The PR adds config CLI validation and regression tests to reject direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set, patch, and unset operations as auto-managed fields.

## Best Possible Solution

Keep `stampConfigVersion` as the system-owned writer, but reject both direct managed leaf paths and parent `meta` operations whose value tree contains `lastTouchedVersion` or `lastTouchedAt`, with focused regression coverage.

## Reproduction Assessment

Yes. The linked issue and source path give a high-confidence reproduction: direct `config set meta.lastTouchedVersion` or `meta.lastTouchedAt` writes reach serialization, then `stampConfigVersion` overwrites the value while the CLI reports success.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the current PR is incomplete because parent `meta` object operations can still include stamped leaves that are accepted and then overwritten.

## Review Findings
