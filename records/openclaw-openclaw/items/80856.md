## Summary

Keep open: the PR fixes the direct leaf commands and now has adequate after-fix terminal proof, but the guard still misses parent `meta` operations that can carry or remove the stamped leaves and then report success while `stampConfigVersion` overwrites the result.

## What This Changes

The PR adds config CLI rejection for direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set/patch/unset operations, with focused config CLI regression tests.

## Best Possible Solution

Keep `stampConfigVersion` as the system-owned writer, but reject direct leaves, descendants, and parent `meta` set/patch/unset operations that would set or remove `lastTouchedVersion` or `lastTouchedAt`, with regression coverage for the parent cases.

## Reproduction Assessment

Yes. Source inspection shows current main applies user config mutations before final write-time stamping, and the linked issue/PR discussion includes terminal output for the direct leaf false-success path.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the proposed implementation is incomplete because parent `meta` object writes and parent `meta` unsets can still be accepted and then overwritten by the final stamp.
