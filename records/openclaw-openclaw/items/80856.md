## Summary

Keep open: the underlying CLI/config bug is valid, but this patch should not merge yet because parent `meta` object writes can still carry `lastTouched*` values that get stamped away after a success message, and the external PR does not include after-fix real behavior proof.

## What This Changes

The branch adds config CLI guards for direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set/patch/unset operations plus three regression tests.

## Best Possible Solution

Keep `stampConfigVersion` as the system-owned writer, but reject both direct leaf operations and parent `meta` merge/replace values that contain `lastTouchedVersion` or `lastTouchedAt`, with regression coverage for the parent-path case.

## Reproduction Assessment

Yes. The source path and linked issue proof give a high-confidence reproduction: direct `config set meta.lastTouchedVersion` or `meta.lastTouchedAt` writes are applied before serialization, then `stampConfigVersion` overwrites them while the CLI reports success.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the current implementation is incomplete because parent `meta` object operations can still carry auto-managed leaves that are accepted and then overwritten.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject parent meta writes containing stamped leaves:** `src/cli/config-cli.ts:1389`
  - body: Because this guard only inspects `operation.requestedPath`, parent-path writes such as `openclaw config set meta '{"lastTouchedAt":"1999-01-01T00:00:00.000Z"}' --strict-json --merge` or `config patch --replace-path meta` can still reach `replaceConfigFile`; `stampConfigVersion` then overwrites the supplied leaf while the CLI reports success. Detect auto-managed leaves inside parent `meta` values, or reject parent `meta` writes when they contain `lastTouchedVersion` or `lastTouchedAt`, so this PR actually closes the false-success class.
  - confidence: 0.88
