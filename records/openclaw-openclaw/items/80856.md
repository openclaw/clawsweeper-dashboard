The PR adds config CLI validation and regression tests to reject direct edits to `meta.lastTouchedVersion` and `meta.lastTouchedAt` through config set/patch/unset paths.

## Best Possible Solution

Reject both direct leaf operations and parent `meta` object set/merge/replace values that contain stamped leaves, with focused regression coverage for the parent-path case.

Yes. The source path and linked issue proof give a high-confidence reproduction: CLI config mutation accepts the stamped metadata values, then `stampConfigVersion` overwrites them during serialization while the CLI reports success.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the current PR is not the best complete fix because parent `meta` values containing the same stamped leaves can still be accepted and overwritten.
