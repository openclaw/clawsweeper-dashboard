## Summary

Keep open: the PR fixes direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set/unset attempts and now has real terminal proof, but the branch still misses parent `meta` object writes that can include those stamped leaves and be overwritten after a success message.

## What This Changes

The branch adds config CLI guards and regression tests to reject direct edits to auto-managed `meta.lastTouchedVersion` and `meta.lastTouchedAt` paths.

## Best Possible Solution

Reject both direct leaf operations and parent `meta` merge/replace values that contain `lastTouchedVersion` or `lastTouchedAt`, while keeping `stampConfigVersion` as the system-owned writer.

## Reproduction Assessment

Yes. The linked issue and source path show direct stamped-field writes are overwritten during serialization, and source inspection shows the PR leaves a parent `meta` object variant of the same false-success behavior.

## Solution Assessment

No. Rejecting direct leaf paths is the right direction, but the implementation should also inspect parent `meta` values before reporting success for values that `stampConfigVersion` will overwrite.
