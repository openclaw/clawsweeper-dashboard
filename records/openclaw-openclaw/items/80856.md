## Summary

Keep open: the direct-path guard is the right direction and the author has now supplied after-fix terminal proof, but the patch still permits parent `meta` object writes that contain auto-stamped leaves and would continue the same false-success behavior.

## What This Changes

The PR adds config CLI validation for direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set/patch/unset operations, plus three regression tests.

## Best Possible Solution

Reject auto-managed metadata both for direct leaf paths and for parent `meta` merge/replace values containing those leaves, while keeping `stampConfigVersion` as the system-owned writer.

## Reproduction Assessment

Yes. The issue has live CLI logs for direct writes, and current source shows the write pipeline stamps the managed fields after mutation handling; the remaining parent-path gap is also source-reproducible from the requestedPath-only guard.

## Solution Assessment

No. Blocking direct leaf paths is the right fix shape, but the current PR is incomplete because parent `meta` object operations can still carry values that are later overwritten while the CLI reports success.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject parent meta values with stamped leaves:** `src/cli/config-cli.ts:1389`
  - body: Because this helper only checks `operation.requestedPath`, parent writes such as `config set meta '{"lastTouchedAt":"1999-01-01T00:00:00.000Z"}' --strict-json --merge` or `config patch --replace-path meta` can still carry auto-managed leaves into the write path. `stampConfigVersion` then overwrites those leaves before serialization while the CLI reports success, so please inspect parent `meta` values or reject parent operations when they include `lastTouchedVersion` or `lastTouchedAt`.
  - confidence: 0.88
