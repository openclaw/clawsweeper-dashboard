## Summary

Keep open: the direct-path guard and after-fix terminal proof are useful, but this PR still lets parent `meta` mutations and `config unset meta` report success while `stampConfigVersion` overwrites or recreates the managed fields. A member reviewer confirmed the same parent-path gap, so this should be revised before merge.

## What This Changes

The PR adds config CLI validation that rejects direct `meta.lastTouchedVersion` and `meta.lastTouchedAt` set, patch, batch, and unset operations, plus regression tests for those direct paths.

## Best Possible Solution

Keep `stampConfigVersion` as the system-owned writer, but reject or inspect parent `meta` mutations that contain or remove `lastTouchedVersion` and `lastTouchedAt`, with regression coverage for parent set, merge, patch replace-path, batch, and unset cases.

## Reproduction Assessment

Yes. The linked issue and PR comments include live CLI proof for the direct-path bug, and source inspection shows parent `meta` operations can still reach the write path before `stampConfigVersion` overwrites or recreates the managed fields.

## Solution Assessment

No. Rejecting direct managed leaves at the CLI boundary is the right direction, but the current PR is incomplete because it does not handle parent `meta` values or deletion of `meta` itself.
