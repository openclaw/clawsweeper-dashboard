## Summary

Keep open: current main and the latest release still render the run-scoped isolated-cron session key in the Runtime prompt line, and the linked open fix PR is the active implementation candidate but has not merged.

## What This Changes

The issue reports that isolated cron runs since v2026.6.9 render a per-run `:run:<UUID>` session key in the system-prompt Runtime line, breaking prompt-prefix cache reuse.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/96686 or an equivalent fix that stabilizes only prompt-facing cron identity while preserving per-run routing, transcript, cleanup, and delivery state.

## Reproduction Assessment

Yes, at source level: current main constructs a `:run:` isolated-cron session key, passes it to embedded execution, and renders it in the Runtime prompt line. I did not independently reproduce the live metered-provider token jump.

## Solution Assessment

Yes: prompt-facing normalization is the narrow maintainable fix because reusing one cron session would weaken isolation, and the existing stable `promptCacheKey` path does not remove volatile Runtime prompt bytes.
