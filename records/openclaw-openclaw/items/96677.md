## Summary

Keep open: current main and v2026.6.10 still render the isolated-cron per-run session key into the Runtime prompt line, and the existing stable `promptCacheKey` does not remove that prompt-visible volatile id. The broader OpenAI prefix-cache issue is related, but this cron UUID mechanism is specific enough to preserve as its own repair target.

## What This Changes

The issue asks OpenClaw to keep isolated-cron per-run UUIDs out of cache-sensitive system-prompt Runtime content because they break prompt prefix caching and increase metered input cost.

## Best Possible Solution

Keep the per-run cron identity internal for routing and cleanup, but render or shape a stable cron prompt identity so repeated isolated runs keep a byte-identical stable system/tool prefix across providers.

## Reproduction Assessment

Yes at source level for the volatile prompt field: current main constructs an isolated cron `:run:` session key, passes it into embedded execution, and renders it in the Runtime line. I did not independently reproduce the reporter's live provider token jump.

## Solution Assessment

No, current behavior is not the best solution: a separate stable `promptCacheKey` is useful but does not stop the raw per-run session key from appearing in prompt text. The safer fix is to separate internal run identity from prompt-facing cache identity or provider payload ordering.
