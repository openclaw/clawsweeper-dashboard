## Summary

Keep open: current main and v2026.6.10 still render the isolated-cron `:run:<id>` key in the Runtime prompt line, and open candidate PRs now own the fix path but have not merged.

## What This Changes

The issue asks OpenClaw to keep volatile per-run isolated-cron IDs out of the cache-sensitive Runtime prompt prefix.

## Best Possible Solution

Land a narrow prompt-rendering fix that strips or normalizes only the terminal isolated-cron run scope from model-visible Runtime identity while preserving run-scoped execution state.

## Reproduction Assessment

Yes at source level: current main builds an isolated-cron `:run:` session key, passes it into embedded execution, and renders it in the Runtime prompt line. I did not independently reproduce the live provider token jump.

## Solution Assessment

Yes: prompt-facing identity normalization is the narrowest maintainable fix because reusing one session would weaken isolation and a new config option is unnecessary for this regression.
