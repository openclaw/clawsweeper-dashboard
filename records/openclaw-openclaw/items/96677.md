## Summary

Keep open: current main and v2026.6.10 still pass a per-run isolated cron session key into embedded execution and render that key in the prompt Runtime line, so the reported cache-busting mechanism remains source-reproducible.

## What This Changes

The issue asks OpenClaw to stop exposing volatile isolated-cron run UUIDs in cache-sensitive prompt/runtime text while preserving per-run cron isolation.

## Best Possible Solution

Keep the per-run cron identity internal for routing and cleanup, but render or route a stable cron prompt identity so repeated isolated runs keep a byte-identical cacheable prompt prefix.

## Reproduction Assessment

Yes at source level: current main constructs a `:run:` isolated-cron session key, passes it into embedded execution, and renders it in the Runtime prompt line. I did not independently reproduce the live provider token jump.

## Solution Assessment

Yes, the requested repair direction is the right narrow fix: keep run identity internal and make prompt/cache-facing identity stable. Reusing one session across runs would weaken isolation, and adding a new config knob is not required for the core bug.
