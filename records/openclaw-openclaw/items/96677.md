## Summary

Keep open: current main and the latest v2026.6.10 release still pass a volatile isolated-cron `:run:<id>` session key into embedded execution and render `runtimeInfo.sessionKey` into the prompt Runtime line. The existing stable `promptCacheKey` preserves observability/cache affinity metadata, but it does not remove the volatile value from prompt text.

## What This Changes

The issue reports isolated cron runs adding a per-run `:run:<UUID>` to the system-prompt Runtime session field, breaking prompt-prefix caching and increasing metered input cost.

## Best Possible Solution

Keep per-run cron identity internal for routing/transcripts/cleanup, but render a stable prompt/cache-facing cron identity so repeated isolated runs for the same job keep a byte-stable provider prefix.

## Reproduction Assessment

Yes at source level: current main constructs an isolated cron `:run:` session key, passes it into embedded execution, and renders `runtimeInfo.sessionKey` in the Runtime prompt line. I did not independently reproduce the reporter's live provider token jump.

## Solution Assessment

No, current behavior is not the best solution: the stable `promptCacheKey` is useful, but it does not stop the raw per-run session key from appearing in prompt text. The narrow fix is to separate internal run identity from prompt-facing cache identity without adding config or weakening isolation.
