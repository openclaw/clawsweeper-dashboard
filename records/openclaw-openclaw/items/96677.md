- `P2`: This is a source-backed isolated-cron prompt-cache cost regression with a bounded runtime surface and no outage, data loss, or security bypass.
- `impact:session-state`: The issue is caused by per-run isolated cron session identity leaking into model-facing Runtime prompt state.

## Summary

Keep open: current main and v2026.6.10 still render the run-scoped isolated-cron session key in the Runtime prompt line, and the linked candidate fix remains open rather than merged.

## What This Changes

Avoid rendering the per-run isolated-cron `:run:<UUID>` identity into the cache-sensitive Runtime prompt line so repeated runs preserve prompt prefix caching.

## Best Possible Solution

Land or close the linked candidate PR after maintainers accept a prompt-facing stable cron identity while keeping actual isolated-cron routing, transcript, cleanup, and delivery state per-run.

## Reproduction Assessment

Yes at source level: current main constructs a `:run:` isolated-cron session key, passes it into embedded execution, and renders it in the Runtime prompt line. I did not independently reproduce the live provider token jump.

## Solution Assessment

Yes, the narrow fix is to keep the per-run identity internal while rendering a stable prompt/cache-facing cron identity. Reusing one session across runs would weaken isolation, and a new config option is not required for the core bug.
