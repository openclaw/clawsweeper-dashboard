## Summary

Keep open: the PR is not merge-ready because it filters after validation can already convert the reported nullable string argument to an empty string, still missing the likely failure path, and it also drops schema-meaningful explicit nulls. The contributor has not provided after-fix real MCP behavior proof.

## What This Changes

The PR changes the session MCP runtime to remove top-level null-valued entries from MCP tool-call arguments before calling the MCP SDK client.

## Best Possible Solution

Preserve explicit JSON nulls by default and repair the validation/coercion path, or omit null only for schema-proven optional fields with regression coverage and redacted real MCP proof.

## Reproduction Assessment

Yes, source inspection provides a medium-confidence reproduction path for the suspected bug: the agent loop validates tool args before MCP execution, and plain JSON-schema `anyOf` coercion can turn `null` into `""`. This review did not run the awslabs EKS MCP server or capture a live wire payload.

## Solution Assessment

No. Unconditional top-level null stripping is too broad and likely too late for the reported `anyOf [string, null]` case because validation can coerce null to an empty string before this boundary.
