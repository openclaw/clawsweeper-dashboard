## Summary

Keep open. Current main still lacks the requested hook, and the PR is directionally useful, but the patch has a cache correctness blocker that can cause hook-enabled template loads to reuse earlier default cached content.

## What This Changes

The PR adds a typed `substitute_template` plugin hook, wires it into workspace template loading, updates docs/changelog/API baseline metadata, and adds workspace seeding tests.

## Best Possible Solution

Keep the load-time typed hook, but make template caching respect substitution state so hook-enabled loads cannot reuse default content cached by an earlier call.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Current main lacks the hook, and the PR blocker can be reproduced by loading a template once without hooks, then ensuring another workspace with a `substitute_template` runner in the same process and observing that the hook is skipped.

## Solution Assessment

Is this the best way to solve the issue? No, not as patched. A load-time typed hook is the right narrow seam, but the cache must distinguish hook-enabled/default loads or bypass the cache when substitutions are active.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Key template cache by hook state:** `src/agents/workspace.ts:110-112`
  - body: `workspaceTemplateCache` is still keyed only by template name, so any earlier non-hook load of `USER.md`/`AGENTS.md` causes later `ensureAgentWorkspace` calls with a `substitute_template` runner to return the cached default and skip `runSubstituteTemplate`. That makes the new hook silently fail in a long-lived process that loaded templates before plugins initialized; include hook state/runner identity in the cache key or bypass the cache when hooks are active.
  - confidence: 0.88
