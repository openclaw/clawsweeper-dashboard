## Summary

Keep open. The PR adds a useful narrow hook, but the latest head still has a blocking prompt-injection policy gap and the reverted cache fix means hook behavior can depend on whether templates were loaded earlier without a hook runner.

## What This Changes

The PR adds a typed `substitute_template` plugin hook, invokes it while loading workspace templates, updates plugin docs/changelog/API hash, and adds workspace seeding tests.

## Best Possible Solution

Keep the hook API, but merge only after `substitute_template` is covered by the prompt-injection policy, template caching is separated for hook-enabled and plain loads, and both paths have loader/workspace regressions plus docs alignment.

## Reproduction Assessment

Yes. Configure a plugin with `plugins.entries.<id>.hooks.allowPromptInjection=false`, register `substitute_template`, then load plugins and seed a workspace; source inspection shows the hook is not classified as prompt-injection, and a prior no-hook template load will also make later hook-enabled seeding return cached plain content.

## Solution Assessment

No. The public hook shape is narrow, but the current implementation is not the best merge path until it preserves the existing safety contract and makes template substitution deterministic across cache ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate template substitution with prompt-injection policy:** `src/plugins/hook-types.ts:103`
  - body: `substitute_template` can rewrite workspace files that are later loaded into agent prompt context, but it is not included in the prompt-injection hook list. A plugin with `hooks.allowPromptInjection=false` can still register this hook and persist prompt text into newly seeded workspaces, bypassing the existing policy gate.
  - confidence: 0.91
- **[P2] Separate template cache entries by hook runner:** `src/agents/workspace.ts:106-110`
  - body: `loadTemplate` now accepts a hook runner, but the cache lookup still uses only the template name before checking hooks. If a template is loaded once without hooks, a later hook-enabled workspace ensure in the same process returns the cached plain template and never calls `substitute_template`, so plugin customization becomes order-dependent.
  - confidence: 0.87
