## Summary

Keep open. The hook is a useful, narrow Plugin SDK addition, but this PR is not mergeable yet because it lets disabled prompt-injection plugins mutate prompt-loaded workspace templates and its final cache behavior can skip or leak substitutions across hook/no-hook loads.

## What This Changes

The PR adds a typed `substitute_template` plugin hook, wires it into workspace template loading and seeding, updates Plugin SDK docs/API baseline/changelog, and adds workspace tests.

## Best Possible Solution

Keep the hook, but classify it under the prompt-injection policy and make template caching hook-runner/plugin-reload aware, with loader and workspace regressions plus aligned docs/API baseline.

## Reproduction Assessment

Yes. Register `substitute_template` from a plugin configured with `hooks.allowPromptInjection=false` and the loader still accepts it because the hook is not in `PROMPT_INJECTION_HOOK_NAMES`; separately, call `ensureAgentWorkspace` once without hooks and then with a hook runner in the same process to hit the name-only template cache.

## Solution Assessment

No. The API shape is narrow, but the current implementation needs the existing prompt-injection safety contract and a hook-aware cache boundary before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate template substitution on prompt-injection policy:** `src/plugins/hook-types.ts:103`
  - body: `substitute_template` can rewrite workspace files that later become prompt context, but adding it only to the general hook list means `plugins.entries.<id>.hooks.allowPromptInjection=false` will not block it. Please classify this hook under the prompt-injection policy or add an equivalent explicit gate with a loader regression.
  - confidence: 0.9
- **[P2] Key template substitutions by hook runner:** `src/agents/workspace.ts:105`
  - body: `loadTemplate` still uses the module-level cache by template name before considering the `hookRunner`. If templates are first loaded without hooks, later hook-enabled workspace ensures reuse the no-hook promise and never call `runSubstituteTemplate`; if a hook result wins first, it can leak into later no-hook or different-runner loads. Please restore a hook-aware cache or clear it on runner/plugin changes.
  - confidence: 0.88
