## Summary

Keep open. The PR adds the requested hook, cache split, docs, changelog, and workspace tests, but the latest head still leaves a prompt-content mutation path outside the existing `allowPromptInjection=false` policy.

## What This Changes

The PR adds a typed `substitute_template` plugin hook, wires it into workspace template loading/caching, documents it, refreshes the Plugin SDK API baseline hash, and adds workspace seeding tests.

## Best Possible Solution

Keep the hook only after classifying `substitute_template` under the prompt-injection policy, or adding an equivalent explicit gate, with loader-policy coverage and docs aligned to that contract.

## Reproduction Assessment

Yes. Configure a plugin with `hooks.allowPromptInjection=false`, register `substitute_template`, and load plugins or seed a workspace; the PR adds the hook name but the policy gate still only checks `PROMPT_INJECTION_HOOK_NAMES`.

## Solution Assessment

No. The hook API and per-runner cache shape are reasonable, but the merge path is not complete until the new prompt-mutating capability respects the existing plugin safety policy.
