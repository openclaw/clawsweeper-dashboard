## Summary

Keep open. The earlier ClawSweeper findings were addressed, but the latest patch adds a prompt-shaping template hook without wiring it into the existing `allowPromptInjection=false` policy, so the PR should not merge as-is.

## What This Changes

The PR adds a typed `substitute_template` plugin hook, invokes it during workspace template loading before seeding, updates hook docs/changelog/API hash, and adds workspace seeding coverage.

## Best Possible Solution

Land the additive hook only after `substitute_template` is routed through prompt-injection policy, docs mention that gating, and focused registry/workspace tests prove disabled prompt injection blocks template substitution.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature PR. Static review gives a high-confidence policy reproduction: set a plugin's `allowPromptInjection=false`, register `substitute_template`, and the registry will not block it because the hook is absent from `PROMPT_INJECTION_HOOK_NAMES`.

## Solution Assessment

No. The template-load seam is the right narrow architecture, but the hook must be classified and tested as prompt-mutating before it is safe to expose.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate template substitution behind prompt-injection policy:** `src/plugins/hook-types.ts:103`
  - body: `substitute_template` can rewrite seeded `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `USER.md`, and related workspace files before they become prompt context, but this patch only adds the hook to the general hook-name list. Because `registerTypedHook` only blocks hooks listed in `PROMPT_INJECTION_HOOK_NAMES` when `allowPromptInjection=false`, an operator who disabled prompt injection for a plugin would still let it persist prompt-shaping content into new workspaces. Add this hook to the prompt-injection policy list and cover the blocked registration path before exposing it.
  - confidence: 0.89

## Security Review
