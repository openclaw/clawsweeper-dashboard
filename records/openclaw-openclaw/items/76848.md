## Summary

Keep open: current main still creates unlabeled Control UI dashboard sessions and the CLI session table/JSON path drops stored labels, so the reported usability gap remains.

## What This Changes

Request that WebUI `/new`/New Chat dashboard sessions receive friendly labels automatically and that session lists surface those labels.

## Best Possible Solution

Use the existing session `label` contract to assign a conservative default label for generated dashboard sessions and include stored labels in CLI JSON/text output while preserving raw keys.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `/new` reaches `createChatSession`, sends no label to `sessions.create`, Gateway stores only explicit labels, and CLI display rows omit `entry.label`.

## Solution Assessment

Yes. Reusing the existing `sessions.create`/`sessions.patch` label plumbing is the narrow maintainable fix; first-message derived titles can stay as a later enhancement.

## Repair Work Prompt

Fix openclaw/openclaw#76848. Current main creates Control UI `/new`/New Chat dashboard sessions without labels: `ui/src/ui/app-render.helpers.ts` calls `createSessionAndRefresh` without `label`, `src/gateway/server-methods/sessions.ts` only stores `p.label`, and `src/commands/sessions-table.ts`/`src/commands/sessions.ts` drop labels from `openclaw sessions` output. Use the existing session `label` contract; do not change session keys, add a config option, or implement AI/first-message title derivation. Add a conservative default label for generated dashboard sessions or the Control UI creation path, include stored labels in `openclaw sessions --json`, and render a readable label/name in the text table while keeping the raw key visible. Update docs and CHANGELOG because this is user-facing. Add/update focused tests for the Control UI create params, Gateway create behavior if the fallback lives there, CLI JSON/text output, and any helper formatting touched.
