## Summary

Keep open: the issue is recent, source-backed, and current main still appears able to replay persisted OpenAI/Codex encrypted reasoning from mirrored history into a later Responses request, producing the reported silent channel failure.

## What This Changes

The issue asks OpenClaw to sanitize stale Codex/OpenAI encrypted thinking blocks during replay, recover once from invalid encrypted content, and surface a channel-visible error when no reply can be delivered.

## Best Possible Solution

Add provider-aware replay sanitization in the shared Codex/OpenAI session-history path, retry once on invalid_encrypted_content with encrypted reasoning removed, and emit a non-sensitive channel error if the turn still cannot deliver.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: a persisted same-model OpenAI/Codex thinking block carrying thinkingSignature.encrypted_content can survive current replay sanitization and be emitted as a Responses reasoning item, although the live Telegram/OpenAI path was not rerun here.

## Solution Assessment

Yes. The fix should live in shared OpenAI/Codex replay sanitization and recovery, not in a Telegram-only workaround or a one-off transcript mirror loader.
