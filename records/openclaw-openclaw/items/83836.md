## Summary

Keep open: current main still has a source-level path that replays persisted Codex `thinkingSignature.encrypted_content` into native OpenAI Codex Responses requests, matching the reported Telegram non-delivery failure.

## What This Changes

The issue reports that Codex replay can resend persisted encrypted reasoning blocks into later OpenAI Responses requests, causing Telegram turns to fail without a deliverable reply.

## Best Possible Solution

Add provider-aware native Codex Responses replay sanitization that removes replay-unsafe encrypted reasoning payloads at request assembly while preserving needed reasoning/tool-call structure and clear non-delivery behavior.

## Reproduction Assessment

Yes, at source level: current main still parses stored `thinkingSignature` JSON into a Responses reasoning input item, and the native Codex replay test expects `encrypted_content` to remain. I did not run the live Telegram/OpenAI path because this review is read-only.

## Solution Assessment

Yes, the best fix is in shared OpenAI/Codex Responses replay sanitization rather than a Telegram-only workaround. It should stay narrower than the broader silent-reasoning transcript-persistence feature tracked separately.
