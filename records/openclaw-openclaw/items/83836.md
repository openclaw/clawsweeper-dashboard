## Summary

Keep open: current main still has a source-level path that preserves same-model OpenAI/Codex Responses `thinkingSignature` JSON and replays it as a reasoning item, matching the reported `invalid_encrypted_content` Telegram non-response failure.

## What This Changes

Fix Codex/OpenAI Responses replay so stale encrypted reasoning blocks from Telegram-backed session history do not poison later turns or leave the user without a reply.

## Best Possible Solution

Add provider-aware replay sanitization for OpenAI/Codex Responses history, retry `invalid_encrypted_content` once with stale reasoning removed, and emit a non-sensitive channel-visible failure if no reply can be delivered.

## Reproduction Assessment

Yes, at source level: a same-model OpenAI/Codex assistant `thinking` block with JSON `thinkingSignature.encrypted_content` survives current replay sanitization and is emitted as a Responses reasoning item. I did not run the live Telegram/OpenAI path in this read-only review.

## Solution Assessment

Yes, the right fix belongs in shared OpenAI/Codex replay sanitization and recovery rather than a Telegram-only workaround. It should preserve valid provider-owned reasoning replay while removing stale mirrored reasoning payloads and making final delivery failure visible.
