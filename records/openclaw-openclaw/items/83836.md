## Summary

Keep open: current main still has a source-level path that replays Codex Responses `thinkingSignature` JSON containing `encrypted_content` into later requests, and a current test explicitly asserts that native Codex replay preserves that field. Reporter and follow-up comments provide real Telegram evidence, and no closing PR or current-main implementation resolves the remaining bug.

## What This Changes

The issue requests sanitizing Codex/OpenAI Responses replay so stale encrypted reasoning blocks cannot make Telegram-bound turns fail without a deliverable reply.

## Best Possible Solution

Add provider-aware Codex Responses replay sanitization below the Telegram layer, stripping or retrying stale `encrypted_content` for native Codex while preserving valid reasoning/tool-call pairing and surfacing a non-sensitive channel failure if no reply can be delivered.

## Reproduction Assessment

Yes, at source level: a same-model Codex Responses assistant `thinking` block with JSON `thinkingSignature.encrypted_content` survives replay sanitization and is emitted back as a Responses reasoning item. I did not execute the live Telegram/OpenAI path in this read-only pass.

## Solution Assessment

Yes, the maintainable fix belongs in shared Codex/OpenAI Responses replay assembly and recovery, not in a Telegram-only cleanup. It should strip stale native Codex encrypted payloads while preserving valid provider-required reasoning replay structure.
