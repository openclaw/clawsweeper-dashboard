## Summary

Keep open: current main still has a source-level path that can replay OpenAI/Codex Responses `thinkingSignature.encrypted_content` into later Codex requests, matching the reported Telegram non-delivery failures and the newer live Telegram evidence.

## What This Changes

The issue reports Codex/OpenAI Responses replay preserving stale encrypted reasoning blocks from mirrored history, causing later Telegram-bound turns to fail before a deliverable reply.

## Best Possible Solution

Add Codex/OpenAI Responses replay sanitization at request assembly or shared session-history construction to remove stale encrypted reasoning payloads, retry once on `invalid_encrypted_content`, and preserve valid reasoning/tool pairing plus a non-sensitive channel-visible failure when no reply can be delivered.

## Solution Assessment

Yes, the best fix belongs below the Telegram layer in Codex/OpenAI replay sanitization and recovery. A Telegram-only workaround or one-time transcript cleanup would miss newly persisted encrypted reasoning after later successful turns.
