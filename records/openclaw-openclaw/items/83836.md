## Summary

Keep open: current main still has a source-level path that preserves same-model OpenAI/Codex Responses `thinkingSignature` JSON and replays it as a reasoning item, while the reported current-release Telegram logs show stale `encrypted_content` can terminate a turn without delivery.

## What This Changes

The issue asks OpenClaw to sanitize stale Codex/OpenAI Responses reasoning blocks from replay history and avoid silent Telegram non-responses when replay fails with `invalid_encrypted_content`.

## Best Possible Solution

Add provider-aware replay sanitization for OpenAI/Codex Responses history: strip or downgrade stale mirrored reasoning blocks before replay, retry `invalid_encrypted_content` once with sanitized history, and emit a non-sensitive channel-visible failure when delivery still cannot produce a reply.

## Reproduction Assessment

Yes, source-level reproduction is clear: a same-model OpenAI/Codex assistant `thinking` block with JSON `thinkingSignature.encrypted_content` survives current replay sanitization and is emitted as a Responses reasoning item. I did not run the live Telegram/OpenAI path in this read-only review.

## Solution Assessment

Yes, the best fix is in the shared OpenAI/Codex replay sanitization and recovery path, not a Telegram-only workaround. It must preserve valid provider-owned reasoning replay while removing stale mirrored reasoning payloads and making final delivery failure visible.
