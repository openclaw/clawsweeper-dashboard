## Summary

Keep open: current main still has a source-level path that can replay stale OpenAI/Codex Responses encrypted reasoning into a later request, and the likely repair is still an open draft PR rather than merged code.

## What This Changes

The issue asks OpenClaw to sanitize stale Codex Responses `thinkingSignature` / `encrypted_content` during replay, retry once on invalid encrypted reasoning, and avoid silent Telegram non-delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84267 or an equivalent shared OpenAI/Codex Responses recovery that drops invalid replayable reasoning, retries before any visible output, preserves tool pairing, and surfaces a non-sensitive channel failure if recovery still cannot deliver.

## Reproduction Assessment

Yes, at source level: a same-model OpenAI/Codex assistant `thinking` block with JSON `encrypted_content` can survive current replay sanitization and be converted back into a Responses reasoning item. I did not run a fresh live Telegram/OpenAI reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix belongs in shared OpenAI/Codex Responses replay recovery rather than a Telegram-only transcript cleanup. The linked draft PR appears to use the right boundary, but it still needs review and merge before this issue can close.
