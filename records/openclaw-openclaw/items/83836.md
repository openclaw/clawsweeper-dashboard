## Summary

Keep open: current main still has a source-level path that can replay same-model OpenAI/Codex Responses `thinkingSignature.encrypted_content` into later requests, matching the reported `invalid_encrypted_content` Telegram failures and lost normal replies.

## What This Changes

The issue requests Codex/OpenAI replay sanitization so stale encrypted reasoning blocks do not poison Telegram or channel-bound Codex turns.

## Best Possible Solution

Add provider-aware OpenAI/Codex replay sanitization below the channel layer, retry once on `invalid_encrypted_content` with stale encrypted reasoning removed, and keep a non-sensitive channel-visible failure if recovery still cannot deliver a reply.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a same-model OpenAI/Codex assistant `thinking` block with JSON `thinkingSignature.encrypted_content` followed by text survives current replay cleanup and is emitted as a Responses reasoning item. I did not run the live Telegram/OpenAI path in this read-only review.

## Solution Assessment

Yes, the best fix belongs in shared OpenAI/Codex replay sanitization and recovery rather than a Telegram-only workaround. It should preserve valid provider-owned reasoning replay while removing stale encrypted payloads and making final non-delivery visible.
