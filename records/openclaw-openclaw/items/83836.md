## Summary

Keep open: current main still has a source-level path that preserves and replays Codex/OpenAI `thinkingSignature.encrypted_content` into later Responses requests, matching the reported Telegram non-delivery failures and the two live field reports in the discussion.

## What This Changes

The issue asks OpenClaw to stop replaying stale Codex/OpenAI encrypted thinking payloads and to recover or surface a channel-visible error when that replay would otherwise fail silently.

## Best Possible Solution

Add provider-aware Codex/OpenAI replay sanitization at request assembly or shared replay construction, retry `invalid_encrypted_content` once with stale encrypted reasoning removed, and emit a non-sensitive channel-visible failure when no reply can be delivered.

## Reproduction Assessment

Yes, at source level: a same-model OpenAI Codex assistant `thinkingSignature` JSON object containing `encrypted_content` can pass current session-history sanitization and be emitted as a Responses `reasoning` item. I did not run a live Telegram/OpenAI turn in this read-only review.

## Solution Assessment

Yes, the best fix belongs in shared Codex/OpenAI replay sanitization and retry handling, not in a Telegram-only workaround. The implementation should keep replay structure and tool-call pairing intact while removing stale encrypted payloads that are unsafe to resend.
