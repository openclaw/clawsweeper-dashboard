## Summary

Keep open: current main still has a source-level path that replays JSON `thinkingSignature` reasoning items containing `encrypted_content` for native Codex Responses, and the field reports show Telegram non-delivery recurred until those encrypted fields were removed. No merged PR or current-main diff appears to have addressed the failing replay path.

## What This Changes

The issue requests a Codex/OpenAI replay fix so stale encrypted reasoning blocks from mirrored Telegram session history do not cause `invalid_encrypted_content` failures and silent non-delivery.

## Best Possible Solution

Add provider-aware sanitization in the shared Codex/OpenAI Responses replay path: strip only stale or unverifiable `encrypted_content` from replayed Codex reasoning, retry once on `invalid_encrypted_content`, preserve reasoning/tool-call pairing, and surface a non-sensitive channel-visible failure if no reply can be delivered.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can carry same-model Codex `thinkingSignature` JSON with `encrypted_content` into Responses replay input. I did not run the live Telegram/OpenAI path in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix belongs in shared Codex/OpenAI replay sanitization and recovery, not in a Telegram-only workaround or a broad silent-reasoning feature. The fix should preserve valid provider-owned reasoning replay while handling stale encrypted payloads safely.
