## Summary

Keep open: current main still has a source-level path that can replay persisted OpenAI/Codex Responses `thinkingSignature.encrypted_content` into a later request, matching the reported `invalid_encrypted_content` and channel non-delivery symptoms.

## What This Changes

The issue reports Codex/OpenAI Responses replay of persisted Telegram-bound `thinkingSignature.encrypted_content` blocks causing `invalid_encrypted_content` failures and missing replies.

## Best Possible Solution

Sanitize OpenAI/Codex Responses replay at request assembly or shared replay construction so stale `encrypted_content` is not resent, retry once on `invalid_encrypted_content` with sanitized history, and keep any final failure channel-visible without expanding into a broad silent-reasoning mode.

## Reproduction Assessment

Yes, at source level: a same-model OpenAI/Codex assistant `thinking` block with JSON `thinkingSignature.encrypted_content` can survive current sanitization and be emitted as a Responses reasoning item. I did not execute the live Telegram/OpenAI path in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix belongs below the Telegram layer in shared OpenAI/Codex replay sanitization and recovery. It should preserve valid provider-owned reasoning/tool replay while removing stale encrypted payloads that cannot be verified in the current request context.
