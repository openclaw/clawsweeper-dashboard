- `P1`: The PR fixes a user-visible WebChat transcript/session-state loss regression, but the current patch has a high-impact multi-agent transcript-mixing blocker.
- `merge-risk: 🚨 message-delivery`: Merging as-is can route a final event from one agent session into another visible session when the rest key matches.

## Summary

Keep open: the PR targets a real WebChat session-state loss path, but the new alias helper can collapse distinct non-main agent sessions and let one agent's transcript appear in another agent's Control UI session. That needs a focused fix before merge.

## What This Changes

Adds a Control UI per-session WebChat message cache, alias-aware cache keys, switched-away final-message caching, reset cleanup, regression tests, and a changelog entry.

## Best Possible Solution

Keep the per-session cache approach, but constrain aliasing to the configured main session only and preserve agent identity for all non-main keys before rebasing and landing.

## Reproduction Assessment

Yes, at source level: current main clears visible chat messages during session switching and drops final events for non-selected sessions. I did not run tests because this review is read-only, but the PR body reports focused RED/GREEN coverage for the same paths.

## Solution Assessment

No, not in its current form. The cache direction is maintainable, but the alias helper must keep non-main agent session keys distinct before this is the best merge shape.
