## Label Justifications

- `P1`: The report is a stable-release regression that can make reply-aware channel turns lose the context needed for correct agent replies.
- `impact:message-loss`: The replied-to message body is dropped from the model-visible inbound context.
- `impact:session-state`: Current-turn channel context diverges between the inbound metadata state and the prompt seen by the model.

## Summary

Keep open. Current main still drops `currentInboundContext` exactly on the `runtimeOnly` prompt path, while source shows reply-target metadata is built and handed to the embedded runner before being discarded at prompt assembly.

## What This Changes

Fix a regression where reply-target inbound context is lost from runtime-only model prompt assembly for Telegram and other reply-aware channels.

## Best Possible Solution

Preserve channel-provided `currentInboundContext` for runtime-only submissions, add a regression for `runtimeOnly` plus reply target context, and keep transcript persistence/internal runtime-event handling unchanged.

## Reproduction Assessment

Yes for source reproduction: combine `transcriptPrompt: ""` with non-empty runtime context and a `currentInboundContext` reply-target block, and current main passes `undefined` as the prompt context. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to carry the untrusted inbound context through the runtime-only prompt path, with a focused regression test proving the reply target reaches the model without changing transcript persistence.
