## Label Justifications

- `P1`: This is a reported latest-release regression in reply-aware channel behavior where the model loses replied-to message context.
- `impact:message-loss`: The dropped data is current-message reply/inbound context that should be visible to the model for the channel turn.

## Summary

Keep open: current main still drops `currentInboundContext` from the model-visible prompt when `runtimeOnly` is true, and the linked closing PR is still open rather than merged.

## What This Changes

The issue reports a latest-release regression where runtime-only agent turns lose reply-target or inbound channel context before prompt assembly.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83786 or an equivalent narrow fix that preserves channel inbound context in runtime-only prompt assembly while keeping hidden runtime context out of transcript history.

## Reproduction Assessment

Yes by source inspection: a runtime-only attempt with `transcriptPrompt: ""` and non-empty `currentInboundContext` passes `undefined` into `buildCurrentInboundPrompt`, so the reply-target block is absent from the model prompt. I did not run live Telegram proof or tests in this read-only review.

## Solution Assessment

Yes: preserving `params.currentInboundContext` at prompt assembly is the narrow owner-boundary fix, because channel inbound context is already model-visible prompt context while runtime-generated context remains handled by the system-prompt branch. The linked PR takes that shape, but the issue should remain open until it or an equivalent fix merges.
