## Label Justifications

- `P1`: The issue describes a latest-release regression in reply-aware channel behavior where the model loses the replied-to message body.
- `impact:message-loss`: The affected data is message context from the current reply target being dropped before the model can use it.

## Summary

Keep open: current main still drops `currentInboundContext` in the runtime-only prompt path, and the linked closing PR remains open and unmerged, so this issue should continue tracking the regression until a fix lands.

## What This Changes

The issue reports a regression where runtime-only prompt submission omits reply-target inbound context from the model-visible prompt.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/83786 or an equivalent narrow fix that always preserves channel inbound context in the model-visible current prompt while keeping hidden runtime context out of transcript history.

## Reproduction Assessment

Yes by source inspection: a runtime-only attempt with `transcriptPrompt: ""` and non-empty `currentInboundContext` passes `undefined` into `buildCurrentInboundPrompt`, so the reply-target block is absent from the model prompt. I did not run live Telegram proof in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to pass current inbound channel context into prompt assembly regardless of `runtimeOnly`, with regression coverage for a runtime-only turn carrying reply context. The linked PR takes that shape, but the issue should remain open until that PR or an equivalent fix is merged.
