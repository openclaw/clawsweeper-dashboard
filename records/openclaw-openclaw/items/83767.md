## Summary

Keep open: current main still has a runtime-only prompt branch that discards `currentInboundContext`, while nearby source shows reply-target blocks are built and carried separately before prompt assembly. The issue is narrow, source-reproducible, and I did not find a local merged fix or superseding item in the checked-out history.

## What This Changes

The issue reports that runtime-only prompt assembly drops channel inbound reply context, removing Telegram reply-target content from what the model sees.

## Best Possible Solution

Preserve channel inbound reply context in the model-visible current prompt even for runtime-only submissions, while keeping hidden runtime-generated context out of transcripts and historical user turns.

## Reproduction Assessment

Yes by source inspection: construct a runtime-only attempt with `transcriptPrompt: ""` and a non-empty `currentInboundContext`; current main passes `undefined` to `buildCurrentInboundPrompt`, so the reply-target block is absent. I did not run live Telegram proof in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to pass current inbound channel context into prompt assembly regardless of `runtimeOnly`, with regression coverage for a runtime-only turn that has reply context. The fix should not move hidden runtime-generated context into the transcript.
