## Summary

Keep open: this focused PR is not merge-ready because the new opt-in contract still relies on `meta.toolSummary.failures`, which can miss recovered tool failures, and the external PR still lacks after-fix real behavior proof.

## What This Changes

Adds an opt-in `payload.failOnToolFailure` field for isolated cron agent turns, persists it in cron SQLite state, and marks isolated cron runs as errors when the agent result reports tool failures.

## Best Possible Solution

Base the opt-in contract on a durable per-run failed-tool count, keep default behavior unchanged, validate payload/state round-trip and recovered-failure behavior, then require redacted real isolated-cron proof before merge.

## Reproduction Assessment

Yes, source-reproducible from current source: the PR reads `meta.toolSummary.failures`, while current main derives that from `Boolean(attempt.lastToolError)` and clears `lastToolError` after recovery. I did not run tests because this review is read-only.

## Solution Assessment

No: the payload/state plumbing is plausible, but `meta.toolSummary.failures` is not a durable any-failed-tool source. The safer fix is durable per-run failed-tool accounting or an explicitly narrower terminal-failure contract.
