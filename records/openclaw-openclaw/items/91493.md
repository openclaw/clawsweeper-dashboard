## Summary

Keep open: current main does not implement the new cron field, but this PR is not merge-ready because the opt-in contract reads terminal `toolSummary.failures`, which can miss failed-then-recovered tools, and contributor real behavior proof is still absent.

## What This Changes

Adds an opt-in `payload.failOnToolFailure` cron agent-turn field, persists it in SQLite cron projections, and marks isolated cron runs as `error` when the final agent result reports tool failures.

## Best Possible Solution

Base the opt-in contract on durable per-run failed-tool accounting, or explicitly narrow and name it as an unresolved-terminal-failure contract, then add focused regression coverage, rebase, and redacted real isolated-cron proof.

## Reproduction Assessment

Yes, source-level: the PR reads `meta.toolSummary.failures`, while current main derives that from `Boolean(attempt.lastToolError)` and recovery can clear `lastToolError`. I did not run tests or a live cron because this review is read-only.

## Solution Assessment

No: the payload/state plumbing is plausible, but terminal `meta.toolSummary.failures` is not the best source for an any-failed-tool contract. The safer fix is durable per-run failed-tool accounting, or an explicitly narrower terminal-failure contract.
