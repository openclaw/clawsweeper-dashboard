## Summary

Keep open: the additive cron contract is useful, but the patch is not merge-ready because it relies on a terminal tool summary that current main can clear after recovered tool failures, and the external PR only provides unit/type validation rather than real behavior proof.

## What This Changes

The PR adds an opt-in `payload.failOnToolFailure` cron agent-turn flag, persists it through cron SQLite columns/schema, validates it in the gateway protocol, and makes isolated cron runs report `status: "error"` when the flag sees tool failures.

## Best Possible Solution

Use a durable per-run failed-tool count or event source that survives recovery, wire that through the new cron payload/state/protocol contract, and require redacted real isolated-cron proof before merge.

## Reproduction Assessment

Yes for the review finding from source inspection: current main derives `toolSummary.failures` from the terminal `lastToolError`, and later successful tools can clear that marker before finalization. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No: adding the additive schema/storage flag is plausible, but using `meta.toolSummary.failures` is not the best enforcement point for an "any failed tool call" contract. The safer fix is to count failed tool events separately from recoverable terminal error state.

- **[P2] Count failures before recovery clears them:** `src/cron/isolated-agent/run.ts:443-444`
  - body: The new `failOnToolFailure` check reads `meta.toolSummary.failures`, but current main builds that from `Boolean(lastToolError)` and clears `lastToolError` after a later successful tool call.
