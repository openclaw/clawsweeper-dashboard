## Summary

Keep open: this PR contains meaningful cron contract work, but it is not merge-ready because the new opt-in contract relies on terminal tool-summary failure accounting, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

Adds an opt-in cron agent-turn `failOnToolFailure` payload field, persists it through cron SQLite state, and marks isolated cron runs as errors when the final agent result reports tool failures.

## Best Possible Solution

Base the opt-in contract on durable per-run failed-tool accounting, or explicitly narrow and name it as a terminal-unresolved-failure contract, then rebase and add redacted real isolated-cron proof.

## Reproduction Assessment

Yes at source level: the PR reads `meta.toolSummary.failures`, while current main derives that count from `Boolean(attempt.lastToolError)` and recovery can clear `lastToolError`. I did not run tests or a live cron because this review is read-only.

## Solution Assessment

No: the payload and state plumbing is plausible, but terminal `meta.toolSummary.failures` is not the best source for an any-failed-tool contract. A durable per-run failure counter, or an explicitly narrower terminal-failure API, is the safer shape.
