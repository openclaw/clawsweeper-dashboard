## What This Changes

The PR adds a `payload.failOnToolFailure` option for isolated cron `agentTurn` jobs, persists it in the cron SQLite projection, validates it through the Gateway protocol schema, and marks runs as `error` when tool-summary failures are present under that flag.

## Best Possible Solution

Define one opt-in strict tool-failure contract that keeps execution status, normal delivery, failure notification, retries, and persisted delivery state consistent, then prove it with a real scheduled isolated cron run plus focused regression tests.

## Reproduction Assessment

Yes at source level for the review finding: the PR sets a fatal run status before delivery-state persistence, while current `applyJobResult` records error runs with `delivered:true` as primary `not-delivered`. I did not run a live scheduled cron proof in this read-only review.

## Solution Assessment

No, not yet. The opt-in idea is plausible, but the proposed path needs a maintainer-owned execution-versus-delivery contract before adding a public payload knob and stored state column.
