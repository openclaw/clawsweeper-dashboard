## Summary

Keep open: the PR is a focused useful cron contract, but it is not merge-ready because it checks a recoverable terminal error marker instead of durable failed-tool events, and the external PR still lacks real isolated-cron behavior proof.

## What This Changes

Adds an opt-in `payload.failOnToolFailure` cron agent-turn field, persists it through gateway protocol and SQLite cron state, and makes isolated cron runs return `error` when reported tool failures are present.

## Best Possible Solution

Base the opt-in contract on a durable per-run failed-tool event count, keep the default behavior unchanged, prove payload/state upgrade round-trip, and add redacted real isolated-cron output before merge.

## Reproduction Assessment

Yes, source-reproducible: PR head checks `toolSummary.failures`, while current main derives that from `lastToolError` and clears it after recovery. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No: the additive payload/state plumbing is plausible, but `meta.toolSummary.failures` is the wrong enforcement source for an any-failed-tool contract. The maintainable fix is a durable failed-tool event count or ledger that survives recovery.
