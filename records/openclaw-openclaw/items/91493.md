## Summary

Keep open: the PR is a focused and useful cron contract, but it is not merge-ready because the enforcement source can miss recovered tool failures and the external PR still lacks real after-fix isolated-cron proof.

## What This Changes

Adds an opt-in cron agent-turn `payload.failOnToolFailure` protocol/state field, persists it in cron SQLite projections, and marks isolated cron runs as `error` when reported tool failures are present.

## Best Possible Solution

Base the opt-in contract on a durable per-run failed-tool count, keep the default behavior unchanged, test recovered failures and payload/state round-trip, then add redacted real isolated-cron output before merge.

## Reproduction Assessment

Yes, source-reproducible: the PR checks `meta.toolSummary.failures`, while current main derives that from `Boolean(attempt.lastToolError)` and clears `lastToolError` after recovery. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No: the additive payload/state plumbing is plausible, but `meta.toolSummary.failures` is not the best enforcement source for an any-failed-tool contract. The safer fix is a durable per-run failed-tool event count, or an explicitly narrower terminal-error contract.
