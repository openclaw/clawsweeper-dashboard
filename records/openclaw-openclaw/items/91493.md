## Summary

Keep open: the PR adds a public persisted cron contract, but the implementation relies on terminal `toolSummary.failures`, which current source derives from `lastToolError` and can miss failed-then-recovered tools; it also still lacks real isolated-cron proof and is conflicting with main.

## What This Changes

Adds an opt-in `payload.failOnToolFailure` cron agent-turn field, persists it in cron SQLite projections, and makes isolated cron runs return `error` when the final agent result reports tool failures.

## Best Possible Solution

Base the opt-in contract on durable per-run failed-tool accounting, or explicitly narrow and name it as an unresolved-terminal-failure contract, then rebase and add redacted real isolated-cron proof.

## Reproduction Assessment

Yes at source level: the PR reads `meta.toolSummary.failures`, while current main derives that from `Boolean(attempt.lastToolError)` and later successful tools can clear `lastToolError`. I did not run a live cron because this review is read-only.

## Solution Assessment

No: the payload/state plumbing is plausible, but terminal `toolSummary.failures` is not the best source for an any-failed-tool contract. The safer fix is durable per-run failed-tool accounting, or an explicitly narrower terminal-failure contract.
