## Summary

Keep open: this PR is not implemented on current main and still has a concrete correctness blocker because the new opt-in contract reads terminal `toolSummary.failures`, which can miss failed-then-recovered tool calls. It also still lacks contributor-supplied real behavior proof for an actual isolated cron run.

## What This Changes

Adds an opt-in `payload.failOnToolFailure` cron agent-turn field, persists it in cron SQLite state, and makes isolated cron runs return `error` when `meta.toolSummary.failures` is positive.

## Best Possible Solution

Base the opt-in contract on durable per-run failed-tool accounting, or explicitly narrow and name it as an unresolved-terminal-failure contract, then add focused regression coverage and redacted real isolated-cron proof.

## Reproduction Assessment

Yes, source-reproducible from current main: the PR would read `meta.toolSummary.failures`, while current main derives that from `Boolean(attempt.lastToolError)` and clears `lastToolError` after recovery. I did not run tests because this review is read-only.

## Solution Assessment

No: the payload/state plumbing is plausible, but `meta.toolSummary.failures` is not a durable any-failed-tool source. The safer fix is durable per-run failed-tool accounting or an explicitly narrower terminal-failure contract.
