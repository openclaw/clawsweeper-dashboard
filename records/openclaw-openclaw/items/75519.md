## Summary

Keep open. Current main still has a plausible generic channel lifecycle bug: a health-monitor restart calls `stopChannel`, but the stop-timeout path marks the account as manually stopped and leaves the old task registered, so the following `startChannel` can no-op instead of launching a fresh Telegram provider.

## What This Changes

The issue asks OpenClaw to recover Telegram polling accounts after a stop-timeout during polling-stall recovery instead of leaving the channel stopped or disconnected until a full gateway restart.

## Best Possible Solution

Implement a generic channel lifecycle recovery path that can force or schedule a replacement after a supervised stop timeout, while preserving the duplicate-provider guard and manual-stop semantics. Add regression coverage for a task that ignores abort, then verify Telegram polling recovery through the existing polling/session tests rather than relying on webhook mode as the only durable workaround.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is available by starting a managed channel task that ignores abort, triggering the health-monitor restart path, advancing the 5s stop timeout, and observing that the attempted restart does not create a new task because the old task and manual-stop marker remain.

## Solution Assessment

Yes, the requested direction is the right boundary if implemented generically: the channel manager/health monitor should distinguish recovery stops from manual stops and ensure a fresh start is possible after timeout, instead of adding a Telegram-only workaround.

## Repair Work Prompt

Fix the Telegram polling recovery issue reported at https://github.com/openclaw/openclaw/issues/75519. Current main can strand a channel after a health-monitor or recovery stop timeout: `stopChannel` marks the account as manually stopped and leaves the old task registered when `waitForChannelStopGracefully` times out, then the health monitor's following `startChannel` can no-op because `store.tasks` still has the account and/or `manuallyStopped` is still set. Implement the narrow generic channel lifecycle change needed so supervised restart/recovery stops can proceed to a fresh start after timeout without turning user-initiated manual stops into auto-restarts and without allowing uncontrolled duplicate providers for the same account/token. Keep the fix in the gateway channel manager/health-monitor boundary unless a tiny Telegram-specific hook is strictly necessary. Do not change Telegram auth, DNS defaults, provider defaults, or webhook behavior as part of this repair. Add regression coverage for a channel task that ignores abort during a health-monitor restart and proves a new start can occur or is scheduled once the forced-stop timeout path is reached; update any existing test that currently codifies the no-second-start timeout behavior. Include a user-facing CHANGELOG.md fix entry with `Thanks @ronakdinesh`.
