## Summary

Keep open. Current main still has the timeout/no-replacement lifecycle path described by the report, and the existing regression test enforces that a second account task is not started after the 5s stop timeout. Telegram polling has useful stall and transport mitigations, but they do not prove recovery from the channel-manager stop-timeout wedge.

## What This Changes

The issue reports that Telegram polling-stall recovery can leave the channel wedged after `stopChannel` times out because the health monitor restart path does not successfully start a replacement provider.

## Best Possible Solution

Add a restart-safe recovery path at the channel-manager and Telegram boundary so a health-monitor restart can replace an aborted timed-out provider without leaving the account wedged and without allowing duplicate active long pollers.

## Reproduction Assessment

Yes for the core lifecycle path: current tests simulate a never-settling channel task after abort and assert that a subsequent start does not create a replacement. The exact live Telegram/IPv6 network wedge is observational rather than locally reproduced.

## Solution Assessment

No for current main. Blindly bypassing the task guard would be risky; the maintainable solution is a scoped force-recovery or replacement mode that clears or supersedes only an aborted timed-out task and works with the Telegram polling lease.
