## Summary

Keep open. Current main has useful Telegram polling mitigations, but the channel supervisor still returns from the stop-timeout path with the old task registered, so the health monitor's follow-up start can no-op instead of replacing the wedged provider.

## What This Changes

The issue asks for Telegram polling-stall and health-monitor recovery to restart the channel after stop() times out instead of leaving it disconnected until a full gateway restart.

## Best Possible Solution

Add a scoped restart-safe recovery path in the channel manager or health-monitor restart flow that can supersede a timed-out task and start a fresh provider without allowing duplicate active pollers; keep Telegram transport rebuild and fetch-timeout mitigations in place and cover the lifecycle with regression tests.

## Reproduction Assessment

Yes for the core lifecycle path: current `src/gateway/server-channels.test.ts` simulates a task that never settles after abort, times out `stopChannel`, calls `startChannel`, and asserts no replacement start occurs. The live Telegram network wedge itself is documented by reporter logs but was not recreated here.

## Solution Assessment

No for current main: the existing polling restart, fetch abort, and transport rebuild mitigations reduce triggers but do not repair the channel-manager stop-timeout branch. The maintainable fix is a scoped force-recovery/supersede path, not a blind duplicate start.
