## Summary

Keep open. Current main still has the lifecycle gap behind the report: after a channel stop timeout, the old task remains registered, so the health monitor's follow-up start can return without creating a fresh provider. Telegram polling has useful stall and transport-rebuild mitigations, and webhook mode is a good workaround, but those do not repair the generic channel-manager stop-timeout branch.

## What This Changes

The issue requests reliable Telegram polling-stall recovery when channel stop times out, so the supervisor starts a fresh channel instance instead of leaving the bot unresponsive until a full gateway restart.

## Best Possible Solution

Add a scoped restart-safe recovery path in the channel manager or health-monitor restart flow that can supersede a timed-out task and start a fresh provider, while keeping duplicate active pollers prevented. Cover the manager timeout branch and health-monitor restart path with regression tests, and keep Telegram polling transport rebuild and webhook guidance as complementary mitigations.

## Reproduction Assessment

Yes for the core lifecycle path: current `src/gateway/server-channels.test.ts` already simulates a task that never settles after abort, times out `stopChannel`, calls `startChannel`, and proves no replacement start occurs. The live Telegram IPv6/NAT long-poll wedge is supported by reporter logs but was not recreated in this read-only review.

## Solution Assessment

Yes, a scoped manager-level supersede/restart path is the maintainable fix boundary. A Telegram-only polling tweak or webhook-only workaround would not solve the generic health-monitor stop-timeout state that blocks a fresh provider.
