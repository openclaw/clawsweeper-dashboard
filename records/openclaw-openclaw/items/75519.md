## Summary

Keep open. Current main has useful Telegram polling mitigations, but the channel-manager timeout branch still preserves the wedged task and an immediate health-monitor start remains a no-op, so the requested recovery path is not proven implemented.

## What This Changes

The issue reports that Telegram polling-stall recovery can leave the account unrecovered after a channel-level stop timeout, requiring a full gateway restart or migration to webhook mode.

## Best Possible Solution

Add a restart-safe recovery path for channel stop timeouts: either force-dispose the wedged Telegram polling transport before replacement, or add a scoped channel-manager restart mode that can recover a timed-out health-monitor stop without allowing duplicate long pollers.

## Reproduction Assessment

Yes for the core lifecycle path: the existing server-channels unit test simulates a never-settling task after abort and asserts no replacement starts after the stop timeout. The exact live residential IPv6 Telegram wedge is not reproduced here.

## Solution Assessment

No. Current main's fetch aborts, getUpdates timeout, and transport rebuilds are good mitigations, but the channel-level timeout branch still blocks the fresh start path the report asks for; a safer fix needs a scoped force-recovery seam rather than blindly starting a duplicate poller.
