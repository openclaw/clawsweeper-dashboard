## Summary

Keep open. Current main still shows the same lifecycle gap described in the report: a timed-out channel stop leaves the old task registered, and the follow-up start path returns without replacing it.

## What This Changes

The issue reports that Telegram polling stall recovery can leave the channel supervisor in a stopped or wedged state after `stop()` times out, preventing recovery without a full gateway restart.

## Best Possible Solution

Add a scoped restart-safe recovery path in the channel manager or health-monitor restart flow that can supersede an aborted task after the stop timeout and start a fresh provider, while preserving the duplicate-provider guard for normal overlapping starts.

## Reproduction Assessment

Yes for the core lifecycle path: `src/gateway/server-channels.test.ts` already models a task that never settles after abort, times out `stopChannel`, calls `startChannel`, and proves no replacement start occurs. The live Telegram network wedge itself remains environment-specific.

## Solution Assessment

Yes, the maintainable fix belongs in the generic channel lifecycle recovery path rather than in Telegram-only polling code. The fix should not blindly bypass `store.tasks`; it needs a narrow force-recovery or supersede state for timed-out, already-aborted tasks.
