## What This Changes

The issue requests restoring Feishu p2p DM reply delivery after a 2026.5.12 regression that receives messages but fails to deliver generated replies.

## Best Possible Solution

Land a narrow Feishu plugin fix that uses `user:<open_id>` for non-thread p2p top-level delivery while preserving inbound chat/message metadata for typing, thread replies, and group `chat:<chat_id>` delivery.

## Reproduction Assessment

Yes, source-reproducible. Current main computes `user:<open_id>` for Feishu DMs, then still builds the reply dispatcher with the raw p2p `ctx.chatId` while suppressing reply metadata before visible sends.

## Solution Assessment

Yes. The best fix is the narrow target split from the closed candidate PRs: keep `chatId` for inbound lifecycle state but send non-thread p2p visible replies through `user:<open_id>`.
