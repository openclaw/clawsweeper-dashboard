## Summary

Keep open: this is a fresh Feishu DM delivery regression with concrete logs, and current main plus the latest shipped tag still route p2p reply delivery through the inbound `oc_...` chat id even though the direct-user identity is the sender `open_id`. The related Feishu visible-reply and P2P-thread reports are useful context, but they do not cover this exact `queuedFinal=true, replies=1` plus `SUBSCRIPTION_NOT_FOUND` failure.

## What This Changes

The issue requests restoring Feishu WebSocket p2p DM replies after upgrading to 2026.5.12, where inbound messages dispatch but outbound replies fail with `SUBSCRIPTION_NOT_FOUND`.

## Best Possible Solution

Fix the Feishu plugin so normal p2p DM final replies send to the direct user identity or a safe reply API path, while preserving group, OpenChat, and p2p-thread behavior with focused regression coverage and redacted live Feishu proof before landing.

## Reproduction Assessment

Yes at source level, but not with a live tenant in this review. A p2p event with `chat_id=oc_...` and `sender.open_id=ou_...` reaches the dispatcher as direct, strips reply metadata, and sends via the `oc_...` chat id that the send layer maps to `chat_id`, matching the reported Feishu delivery failure.

## Solution Assessment

Yes, the likely best fix is a narrow Feishu plugin routing correction, not a global visible-reply or provider policy change. The patch should keep p2p thread and group behavior intact and add tests around the direct DM send target.
