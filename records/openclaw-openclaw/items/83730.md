## Summary

Keep open. This looks like a valid Feishu delivery regression: current main still sends p2p final replies as top-level message creates to the `oc_*` p2p chat id after DM reply-to is skipped, even though the handler already computes a safer `user:<open_id>` target.

## What This Changes

The issue reports that Feishu WebSocket p2p DMs are received and dispatched, but generated replies fail with `SUBSCRIPTION_NOT_FOUND` after upgrading to 2026.5.12.

## Best Possible Solution

Route Feishu p2p reply delivery through the sender open_id target (`user:<ou_...>`) while preserving message-id typing behavior and all group/topic chat-id routes.

## Reproduction Assessment

No live repro was run, but the issue is source-reproducible with high confidence: p2p events carry an `oc_*` chat id, DM reply-to is suppressed, and final delivery creates a message to that chat id. The supplied logs match this path because dispatch completed with one reply but delivery failed afterward.

## Solution Assessment

Yes: the narrow maintainable fix is to use the already-computed DM `user:<senderOpenId>` target for p2p reply delivery, not to add a new config option or retry fallback. Tests should cover p2p text/card/media or dispatcher target selection plus unchanged group/topic behavior.
