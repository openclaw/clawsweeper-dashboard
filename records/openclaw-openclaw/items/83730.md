## Summary

Keep open. Current main and the latest release still route top-level Feishu p2p final replies through the raw p2p `chatId` after reply metadata is suppressed, and the two explicit fix PRs for this issue are closed unmerged.

## What This Changes

The issue requests restoring Feishu p2p DM reply delivery after the 2026.5.12 regression where replies are generated but fail with `SUBSCRIPTION_NOT_FOUND`.

## Best Possible Solution

Land a narrow Feishu plugin fix that keeps inbound chat/message metadata for typing and thread lifecycle but sends non-thread p2p visible replies through `user:<open_id>`, with focused regression coverage and redacted live Feishu proof if feasible.

## Reproduction Assessment

Yes, source-level. A p2p event computes `user:<open_id>` for route metadata, but current main constructs the reply dispatcher with raw `ctx.chatId`; after `skipReplyToInMessages` clears reply metadata, visible sends use that raw `oc_*` chat target.

## Solution Assessment

Yes, the narrow maintainable fix is to separate p2p final delivery target from inbound chat id while preserving existing typing, reply, thread, group, and topic behavior. A broad fallback or session-routing change would be less targeted.
