## What This Changes

The issue reports a Feishu p2p DM regression where OpenClaw receives messages and generates replies, but users see SUBSCRIPTION_NOT_FOUND instead of receiving the reply after upgrading to 2026.5.12.

## Best Possible Solution

Land a narrow Feishu plugin fix that separates non-thread p2p visible delivery (`user:<open_id>`) from inbound chat/message lifecycle metadata, while preserving group/topic `chat_id` behavior and adding focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible with high confidence. A p2p event computes `user:<open_id>` route metadata, but current main constructs the reply dispatcher with the raw `ctx.chatId`; the dispatcher then suppresses DM reply metadata and sends visible replies to that raw target.

## Solution Assessment

Yes, the best fix shape is the linked PR direction: keep typing and reply lifecycle tied to the inbound message while using `user:<open_id>` for non-thread p2p visible delivery. The issue should remain open because that fix has not landed.
