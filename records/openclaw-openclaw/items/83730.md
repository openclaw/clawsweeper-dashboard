## What This Changes

The issue reports a Feishu WebSocket p2p DM regression after 2026.5.12 where agent replies are generated but fail with `SUBSCRIPTION_NOT_FOUND` instead of reaching the user.

## Best Possible Solution

Land a narrow Feishu plugin fix that separates p2p visible delivery from the inbound `oc_*` chat id while preserving typing and thread reply lifecycle, with regression tests and redacted live delivery proof when feasible.

## Reproduction Assessment

Yes, source-level. Current main computes `user:<open_id>` for Feishu DMs but still gives the reply dispatcher `ctx.chatId`, and that dispatcher suppresses reply metadata before direct visible sends.

## Solution Assessment

Yes, the best fix shape is known: keep inbound chat/message metadata for typing and threads, but send non-thread p2p visible replies through the sender `user:<open_id>` target instead of the p2p `oc_*` chat id.
