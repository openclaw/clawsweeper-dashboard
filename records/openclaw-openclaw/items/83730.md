## Summary

Keep open: current main and v2026.6.8 still build Feishu p2p reply dispatch from the raw p2p `oc_*` chat id after reply metadata is suppressed, and the open linked fix PRs are unmerged candidates rather than an implemented fix.

## What This Changes

Feishu WebSocket p2p DM replies after 2026.5.12 are received and generated but not delivered because visible reply sends target the p2p `oc_*` chat instead of the sender `open_id`.

## Best Possible Solution

Land a narrow Feishu plugin fix that preserves raw chat/message metadata for lifecycle state while using a normalized `user:<open_id>` delivery target for non-thread p2p visible sends, then close this issue after merge and proof.

## Reproduction Assessment

Yes, source-reproducible: current main computes `user:<open_id>` for Feishu DMs but still passes the raw p2p `ctx.chatId` to the reply dispatcher while suppressing reply metadata. I did not run a live Feishu tenant flow in this read-only review.

## Solution Assessment

Yes, the target split is the best shape: keep raw inbound chat/message ids for typing, thread, streaming, and context state, but use the computed user target for non-thread p2p visible delivery. The open PRs still need review/repair and proof before this issue can close.
