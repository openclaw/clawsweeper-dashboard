## Summary

Keep open. The underlying empty-output auto-reply gap still exists on current main, but the PR is not merge-ready because its fallback can send a visible error reply even when the session has explicitly suppressed delivery.

## What This Changes

The PR adds a direct/private empty-run fallback error final in `dispatchReplyFromConfig` plus regression tests for empty model output and channel no-op behavior.

## Best Possible Solution

Land a narrow empty-output fallback after final queued counts are known, gated by `!suppressDelivery`, direct-style conversation policy, no routed-origin delivery, no queued final/tool/block output, no block-stream/TTS output, and no intentional handled/no-text branch. Align the chat-type check with `normalizeChatType` unless maintainers intentionally add `private` as a supported alias, and add the required changelog entry.

## Reproduction Assessment

Yes. The current-main gap is reproducible by calling `dispatchReplyFromConfig` in a direct/private context with `replyResolver: async () => undefined` and no block/tool output; the PR regression is reproducible by adding `sendPolicy: "deny"` or message-tool-only delivery to that same path.

## Solution Assessment

No. The dispatch-level fallback is the right general shape, but this patch must honor the existing delivery-suppression boundary and should align chat-type semantics before merge.
