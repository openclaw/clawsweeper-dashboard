## Summary

Keep open: the current-main no-sendable-output gap is still source-visible, but this external PR needs real behavior proof and fixes for delivery-suppression, reasoning-only, and chat-type contract issues before merge.

## What This Changes

The PR adds a direct/private auto-reply fallback when `dispatchReplyFromConfig` completes with no final, tool, or block output, plus regression tests.

## Best Possible Solution

Land a narrow dispatch-level fallback after final queued counts are known, guarded by delivery policy, canonical direct-chat semantics, routed-origin state, queued output counts, block/TTS output, and intentional handled/no-text branches.

## Reproduction Assessment

Yes, at source level. Current main can complete a direct dispatch with an undefined resolver result, no block/tool callbacks, and zero queued output without sending a visible fallback; I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. A dispatch-level fallback is the right shape, but this patch must honor `suppressDelivery`, count sendable replies after filtering reasoning payloads, and use the canonical chat-type contract or deliberately extend it with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor delivery suppression before sending fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1048-1063`
  - body: The fallback guard does not check `suppressDelivery`, so a direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` can now call `sendFinalReply` even though current policy requires user delivery to remain suppressed.
  - confidence: 0.92
- **[P2] Count sendable replies after filtering reasoning:** `src/auto-reply/reply/dispatch-from-config.ts:1053`
  - body: The guard checks raw `replies.length === 0`, but dispatch skips `isReasoning` replies before final delivery. A resolver that returns only reasoning payloads still produces no sendable final/tool/block output, yet this fallback will not run.
  - confidence: 0.78
- **[P3] Use the canonical chat-type contract:** `src/auto-reply/reply/dispatch-from-config.ts:104-107`
  - body: The local `direct|dm|private` helper makes the fallback fire for `private`, but `normalizeChatType` does not currently support that value. Either use `normalizeChatType` here or intentionally extend the shared contract and tests.
  - confidence: 0.76
