## Summary

Keep open: current main still has the direct empty-output gap, but this PR as written can bypass delivery suppression and still lacks contributor real behavior proof.

## What This Changes

The branch adds a static error fallback in `dispatchReplyFromConfig` for direct/private empty-output runs and adds regression tests for direct fallback plus channel no-op behavior.

## Best Possible Solution

Land a narrow dispatch-level empty-output fallback guarded by delivery policy, canonical direct chat-type handling, zero queued final/tool/block outputs, no routed-origin delivery, and the existing intentional silent branches.

## Reproduction Assessment

Yes. Source inspection gives a focused current-main path: direct context, `replyResolver: async () => undefined`, no block/tool callbacks, and dispatch returns `queuedFinal: false` with zero final/tool/block counts.

## Solution Assessment

No. A dispatch-level fallback is the right shape, but this implementation must honor `suppressDelivery` and should use `normalizeChatType` unless maintainers deliberately extend the chat-type contract with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Honor delivery suppression before sending fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1048-1055`
  - body: The fallback condition is evaluated after normal delivery suppression, but it does not include `!suppressDelivery`. A direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver would now queue this static error final even though current delivery policy requires final/tool/block delivery to remain suppressed.
  - confidence: 0.91
