## Summary

Keep open: current main still has the direct empty-output dispatch gap, but this PR should not merge as-is because its fallback can bypass existing delivery-suppression policy and the contributor proof is only tests/CI.

## What This Changes

The PR adds a static error final reply in `dispatchReplyFromConfig` when a direct/private embedded dispatch produces no final, tool, or block output, plus regression tests for the fallback and channel no-op path.

## Best Possible Solution

Land a narrow empty-output fallback after queued counts are known, gated by delivery policy and canonical direct conversation handling, with regression coverage for suppressed delivery and channel no-op cases.

## Reproduction Assessment

Yes, by source inspection. A direct/private dispatch with `replyResolver: async () => undefined`, no block/tool callbacks, and no queued counts returns `queuedFinal: false` on current main; the PR's risky path is the same setup with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"`.

## Solution Assessment

No. A dispatch-level fallback is the right shape, but this implementation must honor `suppressDelivery` and should align with the canonical chat-type contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor delivery suppression before sending fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1048-1055`
  - body: The fallback condition runs outside the existing delivery-suppression gates and never checks `suppressDelivery`. A direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver would now queue the static error final even though current behavior and tests require final/tool/block delivery to stay suppressed.
  - confidence: 0.91
