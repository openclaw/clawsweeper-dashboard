## Summary

Keep open: current main still has the silent empty-output dispatch gap, but this PR is not mergeable as-is because the fallback can bypass delivery suppression, misses reasoning-only no-sendable output, and lacks required real behavior proof.

## What This Changes

The branch adds a static error final-reply fallback in `dispatchReplyFromConfig` for direct/private empty-output dispatches and adds regression tests for direct fallback and channel no-op behavior.

## Best Possible Solution

Land a narrow dispatch-level fallback after queued counts are known, gated by delivery suppression, canonical direct-chat policy, routed-origin state, queued output counts, block/TTS output, and intentional silent branches, with regression coverage for suppression, channel silence, and reasoning-only output.

## Reproduction Assessment

Yes, by source inspection. A direct dispatch with `replyResolver: async () => undefined`, no block/tool callbacks, and no queued counts still returns without any fallback on current main, while the docs say direct conversations disallow silence by default.

## Solution Assessment

No. A dispatch-level fallback is the right shape, but this implementation must honor `suppressDelivery`, count sendable output after filtering, and use the canonical `normalizeChatType` contract unless `private` is intentionally added with tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor delivery suppression before sending fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1048-1063`
  - body: The fallback guard never checks `suppressDelivery`, so a direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver would now call `sendFinalReply` even though current policy and tests require final delivery to remain suppressed.
  - confidence: 0.92
- **[P2] Count sendable output after filtering replies:** `src/auto-reply/reply/dispatch-from-config.ts:1053`
  - body: The guard requires `replies.length === 0`, but dispatch skips `isReasoning` replies before final delivery. If the resolver returns only reasoning payloads, final/tool/block counts remain zero and the new fallback will not send, leaving the same silent completion this PR is meant to cover.
  - confidence: 0.78
