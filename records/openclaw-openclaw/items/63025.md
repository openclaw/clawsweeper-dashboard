## Summary

Keep open: current main still has the zero-output silent completion path, but this PR needs changes before merge because the fallback can bypass explicit delivery suppression and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds a static error fallback in `dispatchReplyFromConfig` for direct/private zero-output dispatches and adds regression tests for the direct fallback and channel no-op path.

## Best Possible Solution

Land a narrow empty-output fallback after queued counts are known, guarded by delivery policy, canonical direct chat-type handling, no routed-origin delivery, and no queued final/tool/block or block-stream/TTS output.

## Reproduction Assessment

Yes by source inspection. A direct-style current-main dispatch with `replyResolver: async () => undefined`, no block/tool callbacks, and no routed-origin output reaches completion with `queuedFinal: false` and zero counts.

## Solution Assessment

No. A dispatch-level fallback is the right shape, but this patch should honor `suppressDelivery` and use or deliberately extend the canonical `normalizeChatType` contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor delivery suppression before sending the fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1048-1055`
  - body: The fallback condition runs after normal delivery-suppression checks but does not include `!suppressDelivery`. A direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver would now queue this error final even though current tests require final/tool/block delivery to stay suppressed.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff can weaken an outbound delivery-control boundary by sending the static fallback when delivery is explicitly suppressed.

Concerns:

- **[medium] Fallback can bypass no-delivery policy:** `src/auto-reply/reply/dispatch-from-config.ts:1048`
  - body: Because `shouldSendEmptyRunFallback` does not check `suppressDelivery`, sessions configured with `sendPolicy: deny` or `message_tool_only` can still receive the fallback final when dispatch produces no output.
  - confidence: 0.9
