## Summary

Keep open: current main still has the direct empty-output gap, but this branch sends the fallback outside delivery suppression and lacks after-fix real behavior proof.

## What This Changes

The PR adds a direct/private empty-output fallback final reply in `dispatchReplyFromConfig` and tests the direct fallback plus channel no-op case.

## Best Possible Solution

Land a narrow dispatch-level empty-output fallback guarded by delivery policy and canonical direct chat-type handling, with regression coverage for direct fallback, suppressed delivery, channel no-op, and handled/no-text branches.

## Reproduction Assessment

Yes. Source inspection gives a focused current-main path: direct context, `replyResolver: async () => undefined`, no block/tool callbacks, then dispatch returns `queuedFinal: false` with zero counts. The PR regression path is the same setup with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"`.

## Solution Assessment

No. The dispatch-level fallback is the right shape, but this implementation must honor `suppressDelivery` and should use the canonical chat-type contract unless maintainers intentionally add a `private` alias.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor delivery suppression before sending the fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1048-1055`
  - body: The fallback condition runs after the normal delivery-suppression checks but does not include `!suppressDelivery`. A direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver would now queue this error final even though current tests require final/tool/block delivery to remain suppressed.
  - confidence: 0.91
