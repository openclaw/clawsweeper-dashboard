## Summary

Keep open: the main branch still has a source-level silent no-reply path, but this PR should not merge as written because the fallback can bypass delivery suppression, misses filtered non-sendable reply payloads, and lacks real behavior proof from a real setup.

## What This Changes

The PR adds a static error final-reply fallback in `dispatchReplyFromConfig` for direct/private embedded dispatches with zero final/tool/block output and adds focused regression tests.

## Best Possible Solution

Land a narrow dispatch-level fallback after queued counts are known, guarded by delivery policy, canonical direct-chat handling, routed-origin state, queued output counts, block/TTS output, and intentional silent branches, with regression coverage for suppressed delivery, channel no-op, and filtered non-sendable payloads.

## Reproduction Assessment

Yes, by source inspection. A direct/private dispatch with `replyResolver: async () => undefined`, no block/tool callbacks, and no queued counts still returns `queuedFinal: false` on current main; the PR's risky path is the same setup with delivery suppression active.

## Solution Assessment

No. A dispatch-level fallback is the right shape, but this implementation must honor `suppressDelivery` and base the fallback on actual sendable output after filtering rather than raw `replies.length`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor delivery suppression before sending fallback:** `src/auto-reply/reply/dispatch-from-config.ts:1050-1063`
  - body: The fallback guard never checks `suppressDelivery`, so a direct turn with `sendPolicy: "deny"` or `sourceReplyDeliveryMode: "message_tool_only"` and an empty resolver would now call `sendFinalReply` even though current policy and tests require final delivery to remain suppressed.
  - confidence: 0.92
- **[P2] Count sendable output after filtering replies:** `src/auto-reply/reply/dispatch-from-config.ts:1053-1057`
  - body: The guard requires `replies.length === 0`, but dispatch later skips `isReasoning` replies before final delivery. If the resolver returns only reasoning payloads, final/tool/block counts remain zero and the new fallback will not send, leaving the same silent completion this PR is meant to cover.
  - confidence: 0.78
