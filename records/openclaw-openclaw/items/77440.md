## Summary

Keep open: current main still has a source-reproducible Control UI bug where the chat model selector builds the visible `Default (...)` label from global session defaults instead of the active agent's effective default model.

## What This Changes

The issue asks WebChat/dashboard to label the `default` model option using the current session agent's effective `model.primary` instead of the global `agents.defaults.model.primary`.

## Best Possible Solution

Resolve the `Default (...)` label from the active session agent's effective default model, fall back to global defaults only when agent-scoped data is unavailable, and cover the split global/agent default case in selector-state tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: when the active session is on an agent with its own default and the UI has `default` selected or cached as `null`, the label still comes from `sessionsResult.defaults` rather than that agent's effective model.

## Solution Assessment

Yes. A narrow Control UI selector-state fix is the maintainable path because backend execution and session resolution already use agent-aware model defaults; no new config or product decision is needed.
