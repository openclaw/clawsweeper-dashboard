## Summary

Keep open: current main still has a source-reproducible Control UI bug where the chat model selector's `Default (...)` label is derived from global session defaults instead of the active agent/session model.

## What This Changes

The issue requests that WebChat's `default` model option display the current session agent's effective default model rather than `agents.defaults.model.primary`.

## Best Possible Solution

Derive the `Default (...)` label from the active session row's effective model when available, fall back to global defaults only when needed, and add a focused selector-state regression test.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: when the UI is on a session whose cached model override is `null` or empty, the `Default (...)` label comes from `sessionsResult.defaults` even though the active row exposes the agent-scoped model. I did not run a live UI session because this review is read-only.

## Solution Assessment

Yes, a narrow Control UI selector-state fix is the best path because backend model execution and session row resolution already account for the current agent. No new config or product decision is needed.
