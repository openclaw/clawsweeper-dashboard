## Summary

Keep open: current main still has a source-visible mismatch where the WebChat default model label is built from global session defaults while backend session execution and rows resolve through the selected session's agent.

## What This Changes

The issue asks WebChat to label the `Default` model option using the active session agent's effective `model.primary` instead of the global `agents.defaults.model.primary`.

## Best Possible Solution

Resolve the picker's default label from the active session agent's configured/effective primary model, with fallback to global defaults only when the agent has no override.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with a default-selected session, `resolveChatModelSelectState()` formats `Default (...)` from `sessionsResult.defaults` even when the session key points at an agent whose effective model is different.

## Solution Assessment

Yes. A narrow UI or gateway contract fix is the right direction, preferably deriving the label from existing agent model metadata or an explicit per-session default field while leaving backend execution semantics unchanged.
