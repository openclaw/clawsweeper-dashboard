## Summary

Keep open. Current main still has credible Grok/xAI `web_search` abort paths: the bundled Grok provider inherits the shared 30 second timeout, passes it into the xAI Responses fetch path, the guarded fetch converts that timeout into an abort signal, and the agent tool adapter still rethrows internal `AbortError`s instead of returning a structured tool error. The linked #58733 report is a narrower confirmed timeout reproduction, and #43609 is useful source work, but neither is a merged/current-main fix for this broader issue.

## What This Changes

The issue reports that `web_search` configured with the Grok/xAI provider returns `This operation was aborted` immediately after the 2026.3.28 xAI Responses API migration.

## Best Possible Solution

Ship a narrow xAI plugin fix: keep user-configured timeouts respected, use a longer Grok default only when unset, convert provider-internal xAI timeout `AbortError`s into ordinary structured tool failures, and harden the xAI Responses parser against null or malformed output/content/annotation entries. Keep the behavior provider-owned unless tests prove a separate generic web-search abort contract gap.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the exact immediate-abort symptom because this read-only review did not run a live xAI request, but yes for the confirmed related path: #58733 reports a current-main Grok `requestXaiWebSearch` abort at the 30 second boundary that succeeds when only the timeout is raised to 60 seconds.

## Solution Assessment

Is this the best way to solve the issue? Yes: a narrow xAI plugin repair is the most maintainable path for the confirmed Grok/xAI behavior, with only minimal shared-tool changes if needed to distinguish user cancellation from provider-internal timeouts.
