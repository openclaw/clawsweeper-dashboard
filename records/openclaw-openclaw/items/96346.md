## Label Justifications

- `P1`: Awaited cron command results can lose auth/setup prompt lines that users need immediately to complete recovery or setup flows.
- `impact:message-loss`: The reported failure is a dropped subset of command output in the delivered run result when truncation occurs.
- `impact:auth-provider`: The concrete reproduced prompt is a device-code auth/setup flow whose code line can be lost from the result.
- `impact:security`: The requested fix must handle sensitive one-time auth/setup prompt output without leaking it through non-interactive delivery.

## Summary

Keep open: current main still bounds command output by retaining the newest bytes only, so an early device-code prompt can be absent from the stored and delivered cron result; the related open redaction PR covers non-interactive announce/webhook delivery rather than this interactive preservation path.

## What This Changes

Preserve action-critical auth/setup prompt lines in interactive async/cron command results when output is truncated.

## Best Possible Solution

Add a coordinated bounded-output preservation path for interactive cron command results that keeps flagged prompt lines and a recovery pointer, while keeping broadcast redaction separate and tested.

## Reproduction Assessment

Yes by source inspection: `outputMaxBytes` keeps only the newest captured bytes, and cron command summaries/run logs use that truncated stdout/stderr as the user-facing result. I did not run the live cron command in this read-only review.

## Solution Assessment

Mostly yes: preserving classified action-critical lines at the command-result truncation boundary is the right layer. The exact classifier and preserve-versus-redact split should be coordinated with the open cron delivery redaction PR before implementation.
