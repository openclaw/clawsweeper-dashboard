## Summary

Keep open. Current main has added useful timeout bounding, circuit-breaker behavior, and embedded-run stage timing diagnostics, but it still routes Active Memory through a blocking lightweight embedded Pi run and does not prove that the bootstrap path is fast enough for a blocking pre-reply plugin. The fresh v2026.4.24 discussion and related open performance issues make this an active latency investigation, not a stale or implemented cleanup item.

## What This Changes

The issue reports Active Memory’s blocking recall sub-agent spending seconds in embedded-run bootstrap despite `bootstrapContextMode: "lightweight"` and asks for profiling, optimization, and realistic latency guidance.

## Best Possible Solution

Keep this focused Active Memory latency issue open. Use the current embedded-run startup/prep stage timing to capture a cold Docker/VPS Active Memory run, then cache or bypass safe per-run invariants for lightweight blocking recall runs and update the Active Memory docs with measured timeout and latency guidance.

## Reproduction Assessment

Yes. The issue gives a concrete reproduction path: enable the bundled Active Memory plugin with logging/transcripts, send a direct message, and compare plugin start, transcript creation, embedded-run stage logs, prompt start, and final elapsed time; this review did not run the live Docker/provider benchmark.

## Solution Assessment

No, current main is not yet the complete fix. The hard timeout, circuit breaker, and stage timing are useful safeguards and diagnostics, but the requested lightweight bootstrap performance target still needs profiling-backed optimization of the pre-model embedded-run path.
