## What This Changes

The issue requests proxy-aware SSRF handling so fake-IP DNS from local proxy tools does not block Telegram media, WeChat, web_fetch, and related fetch surfaces.

## Best Possible Solution

Keep the shipped scoped proxy-aware SSRF contract, and route any remaining arbitrary private-network access or WeChat plugin runtime behavior to the narrower canonical issue or plugin-owned follow-up.

## Reproduction Assessment

Yes for the core behavior: the source and tests model the original strict DNS-pinned blocking path, the fake-IP opt-ins, and the fixed trusted-env-proxy dispatch path. Live WeChat reproduction is not in this checkout because that runtime is external.

## Solution Assessment

Yes. The current solution is narrower and safer than trusting every loopback proxy globally: operators explicitly opt web_fetch into trusted env proxy DNS handoff, while hostname checks, default DNS pinning, and dangerous-host blocking remain covered by tests.
