## Summary

Keep open. Current main has narrowed the Telegram and fake-IP cases, but it still does not implement the broad proxy-aware SSRF behavior requested for web_fetch and other surfaces; the remaining fix changes a security boundary and needs maintainer review.

## What This Changes

The issue requests a proxy-aware SSRF policy so fake-IP/local-proxy DNS does not block external Telegram media, WeChat, web_fetch, and related skill fetches.

## Best Possible Solution

Define and land a narrow, documented proxy-aware SSRF contract for web_fetch and any shared fetch helper, then coordinate channel-specific behavior with Telegram and the external WeChat plugin as needed.

## Reproduction Assessment

Yes for the core web_fetch/proxy-guard portion: a mocked DNS lookup returning 127.0.0.1, 198.18.x.x, or fc00::/7 for a public HTTPS hostname exercises the current blocking and opt-in paths. Live WeChat reproduction is unclear because that runtime is external to this checkout.

## Solution Assessment

No, the broad request to trust any loopback proxy for all requests is not the safest fix by itself. A scoped explicit opt-in with hostname policy checks, audit/docs, cache isolation, and proxy validation is the more maintainable direction.
