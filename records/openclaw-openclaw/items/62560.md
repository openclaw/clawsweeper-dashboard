## Summary

Keep open. Current main has narrowed parts of the fake-IP problem for provider/web-search and Telegram explicit-proxy media paths, but arbitrary `web_fetch` and trusted-env-proxy routing still keep strict local DNS/IP checks; the remaining behavior is a security-boundary decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to handle Clash/Surge-style fake-IP local proxies so public external URL fetches do not get blocked as private/internal SSRF targets across `web_fetch`, Telegram media, WeChat, and plugin/skill fetches.

## Best Possible Solution

Define a narrow, documented proxy-aware SSRF contract for shared fetch/web_fetch with explicit operator opt-in, tests, and docs, then coordinate any owner-specific Telegram, WeChat, or plugin follow-through without weakening default SSRF blocking.

## Reproduction Assessment

Yes for the source-level web_fetch/proxy-guard path: a mocked DNS lookup returning 127.0.0.1, 198.18.x.x, or fc00::/7 for a public HTTPS hostname reaches the current blocking and opt-in branches. Live WeChat behavior is unclear because that runtime is outside this checkout.

## Solution Assessment

No, the broad suggestion to trust all proxied requests is not the safest solution. The maintainable path is a scoped trusted-proxy DNS mode or config contract with preserved default blocking, proxy validation guidance, and targeted regression coverage.
