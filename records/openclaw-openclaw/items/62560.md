## Summary

Keep open. Current main has narrowed several related failures, but the broad proxy/fake-IP report is not fully implemented: managed proxy mode still resolves and checks the target hostname before routing through the proxy, web_fetch only has scoped fake-IP range opt-ins, Telegram media has a surface-specific mitigation, and WeChat runtime behavior is owned by an external plugin.

## What This Changes

The issue requests proxy-aware SSRF behavior so fake-IP DNS from local proxies such as Clash does not block Telegram media, WeChat polling, web_fetch, and related skill fetches.

## Best Possible Solution

Define a security-reviewed, scoped proxy-aware SSRF contract: keep hostname allowlists for known destinations, allow proxy-side DNS handoff only for operator-configured HTTPS paths with an explicit trust mode, document the web_fetch fake-IP knobs, and coordinate WeChat-specific transport behavior with the external plugin owner if a core seam is missing.

## Reproduction Assessment

Yes for the core web_fetch/proxy-guard portion: a mocked DNS lookup returning 127.0.0.1, 198.18.x.x, or fc00::/7 for a public HTTPS hostname exercises the current guarded-fetch rejection or opt-in paths. Live WeChat reproduction is unclear because that runtime is external to this checkout.

## Solution Assessment

No, the issue's broad suggestion to trust a loopback proxy for all requests is not the best safe fix by itself. A narrower explicit-proxy or per-surface policy with security review is safer and more maintainable.
