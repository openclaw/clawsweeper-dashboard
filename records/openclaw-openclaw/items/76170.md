## Summary

Current main covers the reported post-login blank Control UI failure: the browser bundle now uses the browser-safe slash-command registry and has a regression test for the prior `process is not defined` blank-screen path, while the login/overview `sessions.list` path is bounded and responsive on large stores.

## What This Changes

The issue reports that the Control UI goes blank immediately after password login on `2026.4.30-beta.1`, after a successful `sessions.list` RPC and WebSocket disconnect.

## Best Possible Solution

Keep the current browser-safe Control UI command metadata path and bounded session-list startup behavior, then let the next tagged release carry the fix to beta users.

## Reproduction Assessment

Yes for the original beta symptom: load `/openclaw/`, authenticate, observe a blank UI and WebSocket close shortly after `sessions.list`. I did not run the beta live in this read-only review; current-main verification is source/test/changelog based.

## Solution Assessment

Yes. Keeping browser code on the shared browser-safe registry and bounding the startup session query is narrower and more maintainable than adding reconnect retries around a client-side crash.
