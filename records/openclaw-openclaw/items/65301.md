## Label Justifications

- `P1`: The underlying false poll detection can block real user or agent message sends when optional poll metadata is present on a normal send action.
- `merge-risk: 🚨 message-delivery`: Merging the PR changes whether poll-looking send payloads are rejected or delivered, so the chosen intent boundary affects message delivery behavior.

## Summary

Current main still has the broad send-side poll guard, so the branch is not obsolete. The remaining blockers are PR-readiness items: the head is conflicting, the proof is unit-test-only, and overlapping open PRs propose different send/poll semantics.

## What This Changes

The PR adds a stricter poll-intent helper, changes `runMessageAction` send validation to reject only question/options poll payloads, and updates focused poll/send regression tests.

## Best Possible Solution

Land one rebased, maintainer-approved send/poll contract that lets metadata-only/default poll params accompany normal sends, preserves explicit poll creation validation, and includes focused tests plus redacted real channel proof.

## Reproduction Assessment

Yes from source inspection: current main still runs `hasPollCreationParams` before send routing, and focused tests still assert rejection for metadata-only poll fields. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes as a code direction: narrowing send rejection to question/options is a maintainable fix. It is not merge-ready until the branch is rebased, real behavior proof is added, and maintainers consolidate the competing poll/send semantics.
