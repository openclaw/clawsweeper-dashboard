## Summary

Keep open. Current main still uses broad poll-param detection before `message` send dispatch, while this PR targets the right outbound path and only needs mechanical cleanup before it is ready to land.

## What This Changes

The PR adds a stricter real-poll-intent helper, switches the `send` guard to it, and adds regression tests for metadata-only poll params versus real poll payloads.

## Best Possible Solution

Land the stricter action-scoped send guard with a required changelog entry and current-main validation, while leaving actual `poll` action validation strict.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: call `runMessageAction` with `action: "send"` plus metadata-only poll values such as a non-zero duration or true boolean flag and it reaches the broad `hasPollCreationParams` rejection path.

## Solution Assessment

Yes, with cleanup. A separate `hasRealPollIntent` helper for the send guard is narrower than weakening `hasPollCreationParams`, because poll creation can keep using the broader validation path.
