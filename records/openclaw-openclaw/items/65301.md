## Summary

Current main still has the broad send-side poll guard, and this PR’s code direction remains useful, but it should stay open because the external contributor has not provided real behavior proof and the remaining semantics overlap another open send/poll PR.

## What This Changes

The PR adds a `hasRealPollIntent` helper, switches `message-action-runner` send validation to use it, and updates poll/send regression tests so metadata-only poll fields no longer reject plain sends while real poll question/options still do.

## Best Possible Solution

Land one rebased consolidated fix that treats metadata-only/default poll params as absent for plain sends, preserves a clear `poll` action contract for real poll payloads, retains focused regression coverage, and includes real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main still checks `hasPollCreationParams` before send routing, and that detector treats non-question metadata fields as poll params; I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes as a code direction. Narrowing the guard to actual question/options is small and maintainable for metadata-only false positives, but maintainers still need to choose it over the overlapping strip-all-poll-params approach.
