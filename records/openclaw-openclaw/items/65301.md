## Summary

Keep open. Current main still uses broad poll-creation detection for `action="send"`, so this PR addresses a real regression; the code/test approach looks correct, but the user-facing fix still needs the required changelog entry before merge.

## What This Changes

The PR adds `hasRealPollIntent`, switches the send-side poll guard to it, and updates regression tests so metadata-only poll params no longer reject `action="send"`.

## Best Possible Solution

Land the narrower send-side poll-intent guard with focused regression tests and a single Unreleased Fixes changelog entry, while keeping actual `poll` action validation strict.

## Reproduction Assessment

Yes. On current main, a source-level reproduction is clear: pass `action: "send"` with metadata-only poll fields such as `pollDurationSeconds: "60"` or `pollPublic: "true"`, and the broad `hasPollCreationParams` guard rejects before send dispatch.

## Solution Assessment

Yes, with cleanup. A separate `hasRealPollIntent` helper is narrower than weakening `hasPollCreationParams`, because real poll creation and validation can keep using the broader path.
