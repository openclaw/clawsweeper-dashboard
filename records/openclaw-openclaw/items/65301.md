## Summary

Keep open: current main still rejects plain `send` calls when nonzero or true poll metadata is present, and this PR is a plausible narrow fix, but it still needs real behavior proof, a rebase from the current conflict, and maintainer consolidation with overlapping poll-send PRs.

## What This Changes

The PR adds a stricter poll-intent helper, switches `message-action-runner` send validation to use it, and updates tests so metadata-only poll fields are allowed on sends while poll question/options still reject.

## Best Possible Solution

Land one rebased contract that treats metadata-only/default poll params as absent for plain sends, preserves real poll payload handling on the `poll` action path, keeps focused regression tests, and includes redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main still runs `hasPollCreationParams` before send routing, and focused tests still preserve metadata-only poll rejection cases.

## Solution Assessment

Mostly yes as a code direction. Narrowing send rejection to actual poll question/options is cleaner than stripping all poll fields, but the landing branch needs a rebase, real proof, and consolidation with related PRs.
