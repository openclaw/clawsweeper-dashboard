## Summary

Keep open: the PR is aimed at the right gateway/session lifecycle bug and includes real helper-level proof, but it is not mergeable because the shutdown drain does not await `session_end` handlers and the new tracker does not cover the normal channel/compaction session hook paths that the linked issue depends on.

## What This Changes

The PR adds an in-memory active-session tracker, drains it from gateway shutdown/restart, extends `PluginHookSessionEndReason` with `shutdown` and `restart`, and updates docs, changelog, and tests.

## Best Possible Solution

Keep the shutdown/restart `session_end` behavior, but make the drain await bounded plugin hook completion and centralize active-session tracking across every session lifecycle emitter before merge.

## Reproduction Assessment

Yes, at source level: current main's gateway close handler has no active-session enumeration, while session lifecycle hooks exist in reset/create, channel reply, and compaction paths. I did not run a live gateway SIGTERM reproduction in this read-only review.

## Solution Assessment

No; the proposed direction is right, but this implementation is not the best fix yet because it reuses a fire-and-forget emitter for shutdown and tracks only one subset of session lifecycle emitters.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Await session_end handlers during the shutdown drain:** `src/gateway/session-reset-service.ts:232-240`
  - body: `drainActiveSessionsForShutdown` calls `emitGatewaySessionEndPluginHook`, but that helper fires `hookRunner.runSessionEnd(...)` with `void` and returns before plugin handlers settle. The close handler can then continue tearing down plugin services while a database-writing `session_end` plugin is still running, and the new 2s timeout will never report a slow handler because the raced work is only the synchronous loop. Add an awaited shutdown-only emit path, or make the drain build the payload and await `runSessionEnd` under the bounded timeout.
  - confidence: 0.95
- **[P1] Track the channel session lifecycle paths too:** `src/gateway/session-reset-service.ts:175-183`
  - body: The tracker is populated only from `emitGatewaySessionStartPluginHook`, but normal inbound channel sessions still emit `session_start`/`session_end` directly in `src/auto-reply/reply/session.ts`, and compaction does the same in `src/auto-reply/reply/session-updates.ts`. Those sessions will never be drained on shutdown, and compacted sessions can miss the forget path, so the linked ghost-session problem still survives for the main channel flow. Centralize the lifecycle emitters or update every emitter to note/forget the tracker.
  - confidence: 0.92
