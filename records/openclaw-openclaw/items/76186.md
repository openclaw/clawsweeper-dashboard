## Summary

Close as implemented. The report was filed against v2026.4.29, and the same embedded-run prep-latency cluster was fixed on main by 0f16edf329a979767b805568375d0fd07c14fb56 and shipped in v2026.5.2; current main contains targeted stable prompt-prefix caching, prompt-report/tool schema stat reuse, and plugin-tool descriptor caching for the repeated prep bottleneck.

## What This Changes

Cache or otherwise amortize repeated embedded-run prep work across consecutive turns so system prompt, tool prep, and stream setup do not add tens of seconds before every response.

## Best Possible Solution

Keep the shipped targeted cache design: reuse only stable prompt/tool-prep artifacts with explicit invalidation while leaving channel, session, auth, sandbox, and provider runtime pieces dynamic.

## Reproduction Assessment

Yes for the affected v2026.4.29 path: this report and canonical #75999/#76061 provide concrete repeated-turn prep-stage timings, and the same stage labels are in the runner. I did not establish a failing current-main reproduction because current main and v2026.5.2 contain the targeted cache fix.

## Solution Assessment

Yes. The shipped fix is safer than a blanket session cache because it caches stable prompt and descriptor work while keeping volatile channel/session/provider policy below the boundary or resolved per turn.
