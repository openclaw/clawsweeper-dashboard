## Summary

Keep open. Current main and v2026.5.2 include important hot-path cache reductions, and later issue comments report the severe local-provider delay is improved, but the requested session-level reuse is not fully implemented: the embedded attempt path still rebuilds/measures core tools, prompt assembly, and stream setup on each run.

## What This Changes

The issue requests session-lifetime caching for repeated embedded-run prep stages, especially `system-prompt`, `core-plugin-tools`, and `stream-setup`.

## Best Possible Solution

Define a measured stable/dynamic prep boundary, reuse only artifacts with safe invalidation keys, and benchmark repeated Telegram/local-provider turns before closing.

## Reproduction Assessment

Yes for the structural behavior: source inspection shows the same attempt path still marks and executes the `core-plugin-tools`, `system-prompt`, and `stream-setup` stages per run. The exact 48-50s latency is not reproduced without the reporter's provider/workspace benchmark.

## Solution Assessment

Unclear. Targeted stable-prefix and plugin-tool caching already landed, but a blanket session cache is not proven safe because prompt/tool/stream setup includes dynamic channel, sandbox, auth, policy, and hook state.
