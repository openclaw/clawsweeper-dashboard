## Summary

Keep open. Current main has provider-side prompt-cache stability work and one commenter reports the severe local-provider delay improved in v2026.5.2, but the local embedded prep stages named in this issue are still assembled inside each run path.

## What This Changes

Request session-scoped caching or reuse for embedded-run prep artifacts such as system prompt assembly, core tool materialization, and stream setup.

## Best Possible Solution

Introduce measured key-based reuse only for prep artifacts proven stable across turns, keep policy/auth/channel/sandbox/hook-sensitive pieces dynamic, and benchmark before/after on the embedded runner and affected Telegram/local-provider paths.

## Reproduction Assessment

Yes for the structural behavior: current main source shows the named prep stages are executed inside each embedded attempt. The exact 48-50s latency still needs the reporter's provider/workspace or a live benchmark on current release.

## Solution Assessment

Unclear. Session-level reuse is a plausible performance direction, but the proposed blanket cache is not yet the narrowest maintainable fix because the stages mix stable prompt/tool data with dynamic runtime policy.
