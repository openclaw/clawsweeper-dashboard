## Summary

Keep open. Current main has important targeted mitigations for prompt-prefix reuse and plugin/tool prep, but the embedded attempt path still rebuilds the named prep stages on each run, and the fresh Telegram/local-provider reports need a current-main benchmark before this can be closed as implemented.

## What This Changes

The issue requests session-scoped reuse of repeated embedded-run prep work such as system-prompt assembly, core plugin tools, and stream setup.

## Best Possible Solution

Define measured, key-based reuse only for stable artifacts, keep auth/channel/sandbox/tool-policy/provider setup dynamic, and split any local OpenAI-compatible or Telegram-specific slowdown into a narrower follow-up if current-main traces still show it.

## Reproduction Assessment

Yes for the structural behavior: enable embedded-run prep-stage tracing and run two turns in the same session, then compare repeated `core-plugin-tools`, `system-prompt`, and `stream-setup` stage entries. The exact 48-50s user timing needs the reporter's provider/workspace or a current-main live benchmark.

## Solution Assessment

Unclear. Session-level caching is directionally right, but a blanket cache of the named stages is not proven safe because those stages mix stable prompt text with dynamic permissions, auth, channel routing, sandbox state, and provider transport setup.
