## Summary

Keep open. Current main has provider prompt-cache stability and some narrower session reuse, but the embedded attempt path still rebuilds the reported local prep stages per run; the remaining fix needs a deliberate stable/dynamic cache design rather than a blanket session cache.

## What This Changes

Request session-lifetime reuse for repeated embedded-run prep stages, especially `system-prompt`, `core-plugin-tools`, and `stream-setup`, to reduce per-turn startup latency.

## Best Possible Solution

Add measured key-based reuse only for artifacts proven stable across turns, keep permission/channel/sandbox/auth/hook-sensitive pieces dynamic, and prove the change with before/after embedded-run prep benchmarks plus targeted prompt/tool/cache tests.

## Reproduction Assessment

Yes for the structural behavior: enable embedded-run trace logging, run two turns in the same session, and compare repeated `[trace:embedded-run] prep stages` entries. The exact 48-50s latency still needs the reporter's provider/workspace or a live benchmark.

## Solution Assessment

Unclear. Session-level reuse is directionally right, but caching the whole `system-prompt`, `core-plugin-tools`, and `stream-setup` stages is not yet proven safe because those stages mix stable prompt text with dynamic runtime policy and transport state.
