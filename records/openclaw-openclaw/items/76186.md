## Summary

Keep open. Current main still rebuilds the measured embedded-run prep stages per attempt; existing provider prompt-cache, skills, bootstrap, and session-file caches do not skip the local system-prompt, tool-materialization, or stream setup work reported here.

## What This Changes

The issue requests session-lifetime caching for embedded-run prep so system prompt assembly, core plugin tool loading, and stream setup are not recomputed on every turn.

## Best Possible Solution

Introduce keyed reuse only for artifacts proven stable across turns, with explicit invalidation for workspace files, skills/plugins, config, model/provider, auth, sandbox, channel/tool policy, and hooks, then prove the reduction with before/after embedded-run prep traces.

## Reproduction Assessment

Yes for the structural behavior: enable embedded-run trace logging and run two turns in the same session, then compare repeated `core-plugin-tools`, `system-prompt`, and `stream-setup` stages. The exact reported timing still needs the reporter's provider/workspace or a live benchmark.

## Solution Assessment

Unclear. Session-level prep reuse is the right performance direction, but the proposed blanket cache is not yet the safest implementation because the named stages mix stable prompt/tool data with run-specific policy, auth, sandbox, hook, and transport state.
