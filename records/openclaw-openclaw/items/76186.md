## Summary

Keep open. Current main includes targeted cache reductions and one commenter confirmed v2026.5.2 fixed their provider path, but source and docs still show the embedded runner assembles the local prep stages per run; a safe session cache needs scoped invalidation rather than a blanket reuse layer.

## What This Changes

Request session-lifetime caching for embedded-run prep stages so repeated turns avoid rebuilding the system prompt, core/plugin tools, and stream setup.

## Best Possible Solution

Add measured key-based reuse for only the stable prep artifacts, keep dynamic session/channel/auth/sandbox/policy work live, and verify repeated-turn traces across Telegram and provider paths.

## Reproduction Assessment

Yes for the structural current-main behavior: source inspection shows the same prep stages are constructed inside each `runEmbeddedAttempt` invocation. No for the exact 48-50s latency, because the original DeepSeek/macOS environment was not live-tested here and `pnpm` was unavailable for local benchmarks.

## Solution Assessment

Unclear. Session-level caching is a plausible performance direction, but the proposed blanket cache crosses stable prompt/tool artifacts and dynamic runtime policy; the safer path is artifact-specific keys plus trace benchmarks.
