## Summary

Keep open. Current `main` still rebuilds the reported embedded-run prep stages per attempt, and the new Telegram DM comment points to a live cache-prefix symptom rather than evidence that the issue is fixed.

## What This Changes

The issue requests session-lifetime caching so system prompt assembly, core plugin tool materialization, and stream setup are not recomputed on every turn.

## Best Possible Solution

Introduce measured, key-based reuse only for artifacts proven stable across turns, keep channel/auth/sandbox/hook-sensitive pieces dynamic, and prove the result with embedded-run trace and cache-token diagnostics.

## Reproduction Assessment

Yes for the structural behavior: enable embedded-run trace logging and compare two turns in the same session for repeated `core-plugin-tools`, `system-prompt`, and `stream-setup` stages. The exact reported latency still needs the reporter's provider/workspace or a live benchmark.

## Solution Assessment

Unclear. Session-level caching is the right performance direction, but the proposed blanket cache is not yet the safest fix because the named stages currently combine stable prompt/tool data with run-specific policy and transport state.
