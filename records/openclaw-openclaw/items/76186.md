## Summary

Keep open. Current main has provider prompt-cache support plus skills/bootstrap reuse, but `runEmbeddedAttempt` still rebuilds the measured local prep surfaces on each run; solving this safely needs an explicit stable/dynamic cache design rather than a blanket session cache.

## What This Changes

Request session-lifetime caching for embedded-run prep so repeated turns do not rebuild `system-prompt`, `core-plugin-tools`, and `stream-setup` when their inputs are unchanged.

## Best Possible Solution

Introduce measured, key-based prep reuse only for artifacts proven stable across turns, keep permission/channel/sandbox/auth/hook-sensitive pieces dynamic, and benchmark before/after with the embedded-runner hot-path tests and live cache probes.

## Reproduction Assessment

Yes for the structural behavior: enable embedded-run trace logging and run two turns in the same session, then compare the repeated `[trace:embedded-run] prep stages` entries. The exact 48-50s duration still needs the reporter's provider/workspace or a live benchmark.

## Solution Assessment

Unclear. Session-level caching is the right performance direction, but the proposed blanket cache of `system-prompt`, `core-plugin-tools`, and `stream-setup` is not yet proven to be the narrowest safe implementation because those stages mix stable prompt text with dynamic runtime policy.
