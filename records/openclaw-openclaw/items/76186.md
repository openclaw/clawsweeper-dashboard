## Summary

Keep open. Current main has partial prompt-cache stability work, but the embedded run path still rebuilds the reported local prep surfaces on each attempt; the remaining fix needs scoped stable/dynamic cache design rather than a blanket session cache.

## What This Changes

Request session-lifetime caching or reuse for repeated embedded-run prep stages such as system prompt assembly, core plugin tool materialization, and stream setup.

## Best Possible Solution

Add measured, key-based reuse only for artifacts proven stable across turns, keep channel/policy/auth/sandbox-sensitive pieces dynamic, and prove before/after behavior with embedded-run trace benchmarks plus prompt-cache regression coverage.

## Reproduction Assessment

Yes for the structural behavior: run two same-session embedded turns with prep trace logging and compare repeated `core-plugin-tools`, `system-prompt`, and `stream-setup` stage entries. The exact reported duration is not reproduced without the reporter's live Telegram/provider setup.

## Solution Assessment

Unclear. Session-level caching is directionally right, but blanket caching of the named stages is not the best proven fix because those stages mix cacheable prompt text with dynamic runtime policy and transport state.
