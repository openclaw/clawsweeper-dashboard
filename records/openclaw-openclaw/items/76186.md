## Summary

Keep open. Current main has provider prompt-cache stability work and a new stable-prefix cache inside system-prompt rendering, but the embedded attempt path still rebuilds the measured prep stages per run and the latest discussion reports a remaining Telegram-specific cache-prefix problem.

## What This Changes

Request session-lifetime reuse for embedded-run prep artifacts so `system-prompt`, `core-plugin-tools`, and `stream-setup` do not add about 48 seconds to every turn.

## Best Possible Solution

Design measured, key-based reuse for artifacts proven stable across turns, and separately fix any Telegram-specific prompt-cache churn without caching permission, auth, sandbox, or channel state too broadly.

## Reproduction Assessment

Yes for the structural behavior: enable embedded-run trace logging and run two turns in the same session, then compare repeated `core-plugin-tools`, `system-prompt`, and `stream-setup` entries. The exact reported duration still needs the reporter's provider, workspace, and Telegram channel setup or a live benchmark.

## Solution Assessment

Unclear. Session-level reuse is the right performance direction, but the proposed blanket cache is broader than the current safety boundaries; the maintainable path is a small set of explicit cache keys plus benchmark proof.
