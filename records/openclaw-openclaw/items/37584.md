## Label Justifications

- `P2`: This is a concrete multi-agent configuration improvement with a clear source-level gap, but it is not a current core outage.
- `impact:security`: The request concerns isolating API keys so credentials intended for one agent are not selected for another.
- `impact:auth-provider`: The feature would change web-search provider selection, API-key routing, env fallback, auth-profile, and SecretRef activation semantics.

## Summary

Keep open. Current main still supports web-search settings at root/plugin scope and active-agent auth-profile lookup, but it has no schema, type, or runtime contract for per-agent `tools.web.search` overrides; the related credential-scoping issues are useful design inputs, not merged replacements.

## What This Changes

The issue requests per-agent overrides for tool-specific web-search settings such as provider, timeout, and API key.

## Best Possible Solution

Define a documented agent-scoped web/tool override contract, then align schema, exported config types, SecretRef target discovery, runtime provider selection, docs, and credential-isolation tests around that contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a broken-behavior bug, but yes as a source-reproducible feature gap: `agents.list[].tools` is strict without `web`, the exported agent tools type lacks `web`, and runtime web-search config is resolved from root/plugin scope.

## Solution Assessment

Is this the best way to solve the issue? Unclear. `agents.list[].tools.web.search` is plausible, but maintainers should choose the public namespace and precedence model before implementation because this creates a credential-sensitive contract across agents, plugins, env fallbacks, auth profiles, and SecretRefs.
