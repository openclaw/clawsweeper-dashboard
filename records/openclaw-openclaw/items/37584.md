## Summary

Keep open. Current main still lacks an agent-scoped web/tool settings contract; web-search credentials and provider settings remain rooted in global or plugin-scoped config, and the related per-agent env issue is adjacent rather than a replacement. This needs maintainer product and security review before implementation because it defines credential precedence across agents.

## What This Changes

The issue requests per-agent tool settings such as agent-specific web search API keys under `agents.list[].tools.web.search`.

## Best Possible Solution

Define a documented agent-scoped web/tool override contract, then align schema, exported config types, SecretRef target discovery, runtime provider selection, docs, and credential-isolation tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but source inspection gives high confidence that current main does not accept `agents.list[].tools.web` and resolves web-search settings from root/plugin-scoped configuration.

## Solution Assessment

Is this the best way to solve the issue? Unclear. `agents.list[].tools.web.search` is plausible, but maintainers should choose the public namespace and precedence model first because this creates a credential contract across agents, plugins, env fallbacks, and SecretRefs.
