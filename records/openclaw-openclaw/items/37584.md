## Label Justifications

- `P2`: This is a normal-priority multi-agent configuration improvement with a clear current-main gap but no core outage.
- `impact:security`: The request concerns isolating API keys so credentials intended for one agent are not selected for another.
- `impact:auth-provider`: The feature would change web-search provider selection, API-key routing, and SecretRef activation semantics.

## Summary

Keep open. Current main still accepts web-search settings only through the root/plugin web-search configuration, while the strict per-agent tools schema and exported type have no `web` override, and the runtime resolves web search without an agent-scoped selector. The related per-agent env, MCP env, and credential-provider issues are useful design input but do not supersede this tool-specific request.

## What This Changes

Request to allow per-agent tool settings such as `agents.list[].tools.web.search.apiKey` to override global `tools.web.search` configuration.

## Best Possible Solution

Define a documented agent-scoped web/tool override contract, then align schema, exported config types, SecretRef target discovery, runtime provider selection, docs, and credential-isolation tests around that contract.

## Reproduction Assessment

Not applicable as a broken-behavior reproduction; this is a feature request. The current gap is high-confidence by source inspection because `agents.list[].tools` is strict without `web`, while web-search runtime/provider credential resolution has no agent-scoped selector.

## Solution Assessment

Unclear. `agents.list[].tools.web.search` is plausible, but maintainers should choose the public namespace and precedence model before implementation because this creates a credential-sensitive contract across agents, plugins, env fallbacks, and SecretRefs.
