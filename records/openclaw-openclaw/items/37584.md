## Summary

Keep open. Current main still lacks an agent-scoped web/tool settings contract; web-search credentials and provider settings are rooted in global/plugin-scoped config, and the related per-agent env request is adjacent rather than a replacement. This is useful work, but it needs maintainer product and security review before implementation because it defines credential precedence across agents.

## What This Changes

The issue requests per-agent nested tool settings, especially `agents.list[].tools.web.search.apiKey`, so different agents can use different web-search credentials and provider settings.

## Best Possible Solution

Define a documented agent-scoped web/tool override contract, then align schema, exported config types, SecretRef target discovery, runtime provider selection, docs, and multi-agent credential-isolation tests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main does not accept `agents.list[].tools.web` and resolves web-search settings from root/plugin-scoped configuration.

## Solution Assessment

Unclear. `agents.list[].tools.web.search` is plausible, but maintainers should choose the public namespace and precedence model first because this creates a credential contract across agents, plugins, env fallbacks, and SecretRefs.
