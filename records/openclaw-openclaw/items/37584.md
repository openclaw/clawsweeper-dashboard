## Label Justifications

- `P2`: This is a concrete multi-agent configuration improvement with a clear current-main gap but no core outage.
- `impact:security`: The request concerns isolating API keys so credentials intended for one agent are not selected for another.
- `impact:auth-provider`: The feature would change web-search provider selection, API-key routing, env fallback, and SecretRef activation semantics.

## Summary

Keep open. Current main still has global/root web-search configuration and no per-agent `tools.web` override in schema, exported config types, or runtime provider selection; the related per-agent env, MCP env, and plugin-config issues are useful design inputs but do not supersede this tool-specific credential-scoping request.

## What This Changes

Allow per-agent tool/web-search settings, especially web-search provider credentials, to override global `tools.web.search` values.

## Best Possible Solution

Define a documented agent-scoped web/tool override contract, then align schema, exported config types, SecretRef target discovery, runtime provider selection, docs, and credential-isolation tests around that contract.

## Reproduction Assessment

Not applicable as a broken-behavior reproduction; this is a feature request. The current gap is source-reproducible because `agents.list[].tools` is strict without `web`, the exported agent tools type lacks `web`, and web-search runtime has no agent-scoped selector.

## Solution Assessment

Unclear. `agents.list[].tools.web.search` is plausible, but maintainers should choose the public namespace and precedence model before implementation because this creates a credential-sensitive contract across agents, plugins, env fallbacks, and SecretRefs.
