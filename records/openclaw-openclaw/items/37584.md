## Label Justifications

- `P2`: This is a valid multi-agent credential-routing feature with limited blast radius, but it needs product and security design rather than urgent regression handling.
- `impact:security`: The request is about avoiding gateway-global web-search credentials being shared across agents that should have separate secret boundaries.

## Summary

Keep open: current main and v2026.6.10 still do not support `agents.list[].tools.web` or per-agent web-search provider credentials, and the remaining work is a product/security design question rather than conservative cleanup.

## What This Changes

Request to allow per-agent overrides for nested tool settings such as `tools.web.search.apiKey`, instead of using one global tool credential/config surface.

## Best Possible Solution

Define a narrow maintainer-approved agent-aware web-search/tool configuration contract with explicit precedence, SecretRef handling, redaction, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's strict agent tools schema has no `web` field, while web-search provider and credential resolution still read global/plugin config and Gateway/runtime sources.

## Solution Assessment

No, not as a broad recursive `agents.list[].tools.web` merge. The maintainable path is a narrower agent-aware web-search credential/config contract with explicit security, precedence, and plugin/core ownership rules.
