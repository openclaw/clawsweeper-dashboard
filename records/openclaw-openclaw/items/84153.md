## Label Justifications

- `P1`: The PR addresses an agent-turn hang path where a stuck compaction hook can block later Codex notifications and user-visible completion.
- `merge-risk: 🚨 compatibility`: Existing plugins that relied on unbounded compaction hook completion may now be timed out after 30 seconds unless they configure an override.
- `merge-risk: 🚨 session-state`: Fail-open timeout can let OpenClaw proceed before a slow memory or compaction hook finishes updating related state.

## Summary

Current main has a real source-level hang path: `before_compaction` and `after_compaction` reach `runVoidHook` without a default timeout and are awaited from the Codex compaction notification flow. The PR is narrow and appears correct, but merging it intentionally changes the plugin hook contract from unbounded wait to 30-second fail-open, so maintainers should own that compatibility tradeoff.

## What This Changes

The PR adds 30-second default timeouts for `before_compaction` and `after_compaction` plugin hooks plus focused Vitest coverage for hung and fast handlers.

## Best Possible Solution

Land a bounded default if maintainers accept the compatibility tradeoff, with the existing per-hook timeout override documented as the path for legitimately slow compaction plugins.

## Reproduction Assessment

Yes from source inspection: current main awaits compaction void hooks through `runVoidHook`, and without a default or registration timeout a never-settling handler keeps the caller open. I did not run a live hanging Codex turn because this read-only review cannot execute artifact-writing tests or scripts in the checkout.

## Solution Assessment

Yes with a maintainer-owned policy decision: adding these hooks to the existing default timeout table is the narrowest code fix and matches the `agent_end` precedent. The open question is whether 30 seconds is the intended compatibility default for slow compaction plugins.
