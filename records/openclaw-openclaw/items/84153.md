## Summary

Keep open: the patch targets a real source-proven availability gap and looks directionally correct, but it still needs contributor real behavior proof and maintainer acceptance of the new 30s fail-open default for compaction hooks.

## What This Changes

The PR adds 30-second default timeouts for `before_compaction` and `after_compaction` void hooks and adds Vitest coverage for hung and fast handlers.

## Best Possible Solution

Land the bounded default only after real Codex compaction proof and clear maintainer acceptance of the compatibility tradeoff for slow compaction hook plugins.

Yes. Source inspection shows current main awaits `before_compaction` and `after_compaction` through `runVoidHook` with no default timeout, including on the Codex app-server `contextCompaction` notification path; I did not run a live hanging plugin repro.

## Solution Assessment

Mostly yes. Reusing the existing hook timeout mechanism is the narrow fix, but the 30s fail-open default needs real behavior proof and maintainer acceptance of the compatibility tradeoff.
