Keep open: the patch is a focused, source-backed fix and I found no blocking code-review defects, but this external PR still needs after-fix real behavior proof before merge and maintainers should explicitly accept the default-timeout compatibility change.

## What This Changes

The PR adds 30s default fail-open timeouts for `before_compaction` and `after_compaction` plugin hooks and adds hook-runner tests for never-settling handlers.

## Best Possible Solution

Land the default timeout with focused tests and after-fix runtime proof, while preserving and documenting the `hook.timeoutMs` override for plugins that legitimately need longer compaction work.

## Reproduction Assessment

Yes. Source inspection on current main shows `before_compaction` and `after_compaction` are void hooks, have no default timeout entry, and are awaited by the Codex app-server compaction projector.

## Solution Assessment

Yes. Adding defaults in the central hook runner follows the existing `agent_end` pattern and preserves per-hook `timeoutMs` overrides; the missing piece is real after-fix proof and explicit acceptance of the compatibility tradeoff.

## Risks / Open Questions

- Existing plugins that relied on unbounded compaction hooks may now have those handlers marked failed after 30s unless they set a per-registration `timeoutMs`.
- The timeout releases OpenClaw's awaiter but does not cancel the plugin's underlying work, so a timed-out hook can still continue side effects in the background.
- The PR currently provides unit/type/lint proof only; no real Codex run, terminal output, or log shows the serialized notification queue continuing after a hung compaction hook.
