## Summary

## What This Changes

The issue asks startup ACP identity reconciliation to skip terminal one-shot ACP records so stale pending identities do not emit repeated false gateway warnings.

## Best Possible Solution

Skip `mode === "oneshot"` before startup pending-identity reconciliation increments `checked`, keep persistent pending identities eligible, and add a focused manager regression test without adding an unsupported `state === "closed"` branch.

## Reproduction Assessment

Yes, source-level. A persisted ACP record with `mode: "oneshot"`, `identity.state: "pending"`, and `acpxSessionId` passes the current reconcile predicate and can drive the gateway warning when runtime reconciliation fails.

## Solution Assessment

Yes. The narrow maintainable fix is to skip one-shot ACP records before startup pending-identity reconciliation, while leaving broader resolved-session warm-restore and pressure-eviction work to their own reviewed paths.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72013. In current main, `AcpSessionManager.reconcilePendingSessionIdentities` counts persisted ACP one-shot records with `identity.state: "pending"` and a stable `acpxSessionId` during gateway startup, which can emit false `acp startup identity reconcile` warnings. Add a narrow guard that skips `session.acp.mode === "oneshot"` before resolving/checking pending identity and before `checked += 1`; keep persistent pending identities with stable ids eligible. Add a focused regression test in `src/acp/control-plane/manager.test.ts` that constructs a one-shot pending identity with `acpxSessionId`, asserts `{ checked: 0, resolved: 0, failed: 0 }`, and verifies runtime ensure/status is not invoked. Do not add a `state === "closed"` guard unless the type/schema and legacy data evidence support it, and do not broaden ACP pressure eviction or warm-restore behavior. Add a changelog entry if the landing workflow treats the warning fix as user-visible.
