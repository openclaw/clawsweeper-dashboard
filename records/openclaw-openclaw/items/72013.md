## Summary

Keep open. Current main still lets persisted ACP records with `identity.state: "pending"` and a stable ACPX or agent session id enter startup identity reconciliation without first excluding terminal one-shot records, so the reported warning can recur for historical metadata. The focused source PR is closed unmerged, and the newer post-turn oneshot identity reconciliation does not repair already-persisted pending terminal records.

## What This Changes

Startup identity reconciliation should skip terminal one-shot or closed ACP session metadata so stale pending identities do not produce repeated gateway warnings.

## Best Possible Solution

Add a startup-only terminal-record guard before `checked` increments and before runtime work, preserving persistent-session identity reconciliation while adding focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. A focused unit fixture can mock `listAcpSessionEntries` with `mode: "oneshot"`, `identity.state: "pending"`, and `acpxSessionId` or `agentSessionId`; current main's predicates make that record eligible for startup reconciliation.

## Solution Assessment

Yes. The narrow startup skip is the right fix for historical pending terminal metadata; resolving future one-shot identities after successful runs does not prevent already-persisted pending one-shot records from being retried on restart.
