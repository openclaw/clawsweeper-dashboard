## Summary

Keep open. Current main still reconciles any pending ACP identity with a stable ACPX or agent session id, so persisted one-shot records like the report describes can still increment `checked` and produce the startup warning; the focused related PR was closed without merge.

## What This Changes

The issue asks OpenClaw to skip or de-noise ACP startup identity reconciliation for terminal one-shot session metadata that remains `pending`.

## Best Possible Solution

Add a narrow startup-only guard before `checked` increments and runtime work so persisted terminal one-shot ACP records are skipped, while preserving persistent-session identity reconciliation and documenting the user-facing warning fix in the changelog.

## Reproduction Assessment

Yes. A focused unit fixture can mock `listAcpSessionEntries` with `mode: "oneshot"`, `identity.state: "pending"`, and `acpxSessionId` or `agentSessionId`; the current code path still treats that as eligible for startup reconciliation.

## Solution Assessment

Yes. The startup skip is the narrowest maintainable fix for historical pending one-shot metadata; current post-turn one-shot identity resolution helps future successful runs but does not cleanly handle already-persisted pending one-shot records.

## Repair Work Prompt

Fix the ACP startup identity reconcile warning described in https://github.com/openclaw/openclaw/issues/72013. Current main still sends persisted `mode: "oneshot"` ACP records with `identity.state: "pending"` and a stable `acpxSessionId` or `agentSessionId` through `AcpSessionManager.reconcilePendingSessionIdentities`, which can produce misleading gateway startup warnings for terminal one-shot metadata. Add a narrow guard in `src/acp/control-plane/manager.core.ts` before `checked` increments and runtime work so terminal one-shot startup records are skipped, while preserving persistent pending-identity reconciliation. Add focused regression tests in `src/acp/control-plane/manager.test.ts` proving pending one-shot records with stable ids do not call runtime ensure/status and return `{ checked: 0, resolved: 0, failed: 0 }`; keep existing persistent/resolved/no-stable-id tests intact. Use related PR https://github.com/openclaw/openclaw/pull/72023 as source context, but reimplement against current main. Be careful with the issue body's `state === "closed"` suggestion: current `SessionAcpMeta.state` is typed as `idle | running | error`, so only add legacy closed-state handling if you can do it type-safely with evidence from current raw persisted data contracts. Add a user-facing `CHANGELOG.md` fix entry with appropriate non-bot credit, likely Thanks @scipher888 and Thanks @hclsys. Do not change acpx dependency versions, runtime backend readiness, broad startup ordering, or plugin APIs.
