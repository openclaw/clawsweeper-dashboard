## Summary

Keep open. Current main and v2026.6.10 still have adjacent read-only, sandbox, `update_plan`, `/btw`, approval, and plugin host-hook primitives, but not first-class Ask, Plan, and Modify modes with enforced read-only questioning, approvable previews, and an explicit transition to mutation.

## What This Changes

The issue requests first-class Ask, Plan, and Modify workflow modes that separate read-only questioning, approvable planning, and mutation.

## Best Possible Solution

Define an enforceable workflow-mode contract that reuses existing host-hook, sandbox/tool-policy, `update_plan`, `/btw`, and dry-run primitives where they fit, then implement through the chosen core/plugin/UI/API surface.

## Reproduction Assessment

Yes. By source inspection, current main has adjacent primitives but no first-class Ask/Plan/Modify mode contract that gates tool execution and transitions from preview to mutation.

## Solution Assessment

Unclear until maintainers define the product and security contract. The requested direction is reasonable, but the safest implementation is a designed authority contract over existing primitives, not a prompt-only label or parallel dry-run path.
