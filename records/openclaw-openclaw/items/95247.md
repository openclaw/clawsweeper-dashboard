## Summary

Keep this PR open, but not merge-ready: it fixes the reported context-engine happy path in the right area, yet the patch drops the existing explicit-empty/invalid allowlist fail-closed semantics when moving plugin policy into authority policy.

## What This Changes

The PR changes context-engine LLM capability construction to read the owning plugin's `llm.allowModelOverride` and `allowedModels` config into the runtime authority, with one runtime LLM test expectation updated.

## Best Possible Solution

Preserve the existing allowlist contract while hoisting plugin LLM policy into the trusted authority path, including the explicit-empty/invalid allowlist fail-closed case.

## Reproduction Assessment

Yes for the introduced PR bug: source inspection shows an explicit `allowedModels` array is tracked as configured in normalized plugin config, but the PR authority helper drops that marker and `assertAllowedModelOverride` allows overrides when the authority policy has no models. The original linked compaction bug has strong issue logs and a source-backed hardcoded-deny path, but I did not run the full live gateway compaction reproduction.

## Solution Assessment

No; the direction is plausible, but this is not yet the best fix because it changes model override policy semantics while bypassing the plugin-policy fail-closed allowlist state. The best fix is to adjust the authority API or reuse the existing policy builder so authority and plugin policy share the same allowlist contract.
