## Summary

Keep open. Current main still lacks the requested replay-context pruning for adjacent assistant silent replies, and the linked implementation PR is closed/unmerged, so this remains a narrow valid enhancement rather than cleanup-close material.

## What This Changes

Add built-in transcript hygiene that collapses consecutive assistant `NO_REPLY` or silent-reply turns in model replay context while preserving isolated silent replies.

## Best Possible Solution

Add documented in-memory replay-history hygiene in the embedded runner that collapses only adjacent assistant silent replies, keeps one by default, reuses the shared silent-reply payload detector, handles legacy string/text-block/strict JSON silent envelopes and known empty silent-stop shapes, and leaves stored transcripts unchanged.

## Reproduction Assessment

Yes. A replay history containing adjacent assistant turns with exact `NO_REPLY` or strict `{"action":"NO_REPLY"}` silent envelopes passes through the current sanitize/validate/heartbeat-filter/limit flow without a general silent-reply collapse; only separate heartbeat prompt/ack pairs are filtered.

## Solution Assessment

Yes. The narrow maintainable path is a replay-context pruning helper wired into the existing embedded-runner history hygiene flow with focused tests and docs; the closed prior PR is useful prior art, but a replacement should avoid broad config unless maintainers want it.

## Repair Work Prompt

Implement a focused fix for this issue in current main. Observable bug: the embedded-runner replay context can keep runs of adjacent assistant silent replies (`NO_REPLY`, strict `{"action":"NO_REPLY"}` envelopes, normalized text-block equivalents, and known empty silent-stop shapes), which can reinforce silence in later model turns. Add an in-memory replay-history pruning step that collapses only adjacent assistant silent replies and keeps one by default. Reuse the shared silent-reply detector from `src/auto-reply/tokens.ts`; do not invent a second `NO_REPLY` parser. Wire the helper into the existing embedded-runner history hygiene flow around `sanitizeSessionHistory` / `validateReplayTurns` / `filterHeartbeatPairs` / `limitHistoryTurns` before assigning replay messages back to the active session. Do not rewrite stored session JSONL or add storage-level cleanup. Avoid adding a new `agents.defaults` config surface unless maintainers explicitly request it. Update `docs/reference/transcript-hygiene.md` and add focused tests for exact text, text-block, strict JSON envelope, isolated silent replies, adjacent silent replies, and non-silent assistant content. Credit the report in a user-facing changelog entry if the touched behavior is treated as a shipped fix. Related refs: this issue, PR #13388 as prior art, #5191 for the cascade scenario.
