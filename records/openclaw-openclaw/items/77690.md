## Summary

Keep open: current main still has the source-reproducible WebChat selector-label bug, and this draft PR is a plausible narrow fix, but it still needs accepted after-fix real behavior proof before merge.

## What This Changes

The PR updates Control UI/WebChat model selector state to derive the `Default (...)` label from the active agent model, adds focused regression coverage, and adds a changelog entry.

## Best Possible Solution

Finish the draft with accepted after-fix WebChat or terminal proof, account for the plain-session default-agent lookup edge if maintainers agree, then land the selector-state fix with focused coverage.

## Reproduction Assessment

Yes. Source inspection shows current main formats `defaultLabel` from `sessionsResult.defaults` while gateway session rows expose the effective agent model, so a default-selected agent session with a different agent primary will render the global label.

## Solution Assessment

Mostly yes. The PR uses the narrow selector-state path and preserves the active-row fallback, but the new agent lookup should account for `agentsList.defaultId` for plain session keys to avoid leaving that edge label on the global default.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Use the configured default agent for plain session keys:** `ui/src/ui/chat-model-select-state.ts:36`
  - body: `resolveAgentIdFromSessionKey(...)` returns `main` for non-agent session keys, while other chat controls and gateway session rows treat plain keys as `agentsList.defaultId`. With `agentsList` loaded and an explicit current override, this can still label the default option from the wrong agent/global default when the configured default agent is not `main`; resolve the active agent with `parseAgentSessionKey(...) ?? state.agentsList?.defaultId` before looking up `agentsList`.
  - confidence: 0.75
