## Summary

Keep open: current main still has the WebChat default-label bug and this draft PR is the linked fix path, but it still lacks accepted real behavior proof and has a small plain-session default-agent edge to address before merge.

## What This Changes

The PR adds agent-aware default-model label resolution to the WebChat model selector, a regression test for cleared/default agent sessions, and a changelog entry.

## Best Possible Solution

Resolve the active agent from the parsed agent session key or configured default agent, keep the active-row fallback for unloaded agent lists, retain focused regression coverage, and add redacted after-fix proof before landing.

## Reproduction Assessment

Yes, as a source-level reproduction: current main builds the WebChat `Default (...)` label from global session defaults while WebChat renders that label directly and gateway rows expose agent-scoped model identity. I did not run a live WebChat session in this read-only review.

## Solution Assessment

Mostly yes. The selector-state helper is the narrow maintainable boundary, but the PR should resolve plain session keys through `agentsList.defaultId` and still needs accepted after-fix real behavior proof.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Use the configured default agent for plain session keys:** `ui/src/ui/chat-model-select-state.ts:39-40`
  - body: `resolveAgentIdFromSessionKey(state.sessionKey)` maps a plain key such as `main` to `main`, while nearby WebChat paths treat plain keys as `agentsList.defaultId`. With a non-`main` default agent and a cleared override, the new lookup can still label the Default option from `main` or global data; resolve the active agent as `parseAgentSessionKey(state.sessionKey)?.agentId ?? state.agentsList?.defaultId` before matching `agentsList.agents`.
  - confidence: 0.75
