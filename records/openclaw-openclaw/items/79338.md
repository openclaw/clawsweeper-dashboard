## Summary

Keep open: current main still fetches the Slack thread root but explicitly drops current-bot starter/history content, matching the reported missing context when a user replies to the bot's own Slack message.

## What This Changes

The issue asks Slack thread sessions to include the parent/root message content when a user replies in-thread to the bot's own Slack message.

## Best Possible Solution

Keep Slack thread sessions isolated, but include the current bot's root message once as sanitized thread starter/history context for the first turn of a new Slack thread session.

## Reproduction Assessment

Yes from source inspection, though not from a live Slack run: the plugin fetches the root via conversations.replies, then drops current-bot starter and history before building ThreadStarterBody/ThreadHistoryBody. Existing tests assert the same omission for bot-authored thread starters.

## Solution Assessment

Yes: the narrow maintainable fix is to change Slack thread context filtering so current-bot root messages are included for new thread sessions while keeping parent transcript inheritance off, preserving allowlist filtering for other senders, and avoiding repeated injection on existing sessions.
