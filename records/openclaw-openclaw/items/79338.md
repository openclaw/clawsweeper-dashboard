## Summary

Keep open: current main still omits the current bot's Slack root/thread-history content, and an open fix PR already specifically references this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Slack DM thread replies to the bot's own root message should include the parent/root message content in the new thread session.

## Best Possible Solution

Land or replace the focused Slack fix in https://github.com/openclaw/openclaw/pull/80409 so new thread sessions include the current bot's root message as assistant-labeled context while preserving existing-session de-duplication and sender filtering, then close this issue after merge.

## Reproduction Assessment

Yes, from source inspection: current main fetches Slack thread starter/history for new sessions and then removes current-bot starter/history before constructing the prompt context. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is to retain the current bot's root message for new Slack thread sessions while keeping existing-session filtering and third-party sender rules intact. The active PR is the right implementation candidate, so this issue should stay open until that PR lands or is superseded.
