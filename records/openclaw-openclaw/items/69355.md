## Summary

Keep open. The PR is a useful doctor feature, and current main still has no equivalent rebuild-vs-npm-release diagnostic, but the submitted patch has narrow correctness blockers already reflected in prior review comments: unbounded registry fetch, stale-cache fallback, missing package/build-info version parity, and conflated malformed/missing metadata.

## What This Changes

The PR adds a warn-only `openclaw doctor` install freshness check that compares local `dist/build-info.json` metadata with npm registry publish times, plus docs, tests, and changelog text.

## Best Possible Solution

Land a rebased or replacement doctor-owned diagnostic that stays warn-only, uses bounded registry fetching, treats expired registry data as inconclusive, validates local metadata parity, reports corrupt metadata accurately, and fits the current lazy doctor flow.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested doctor diagnostic is absent: the doctor flow has install notes but no build-info/npm publish comparison, and targeted symbol/docs searches only found unrelated release text.

## Solution Assessment

No, not as submitted. The direction is appropriate for `openclaw doctor`, but the patch should be tightened around bounded network behavior, TTL-respecting cache use, package/build-info parity, and clearer corrupt metadata diagnostics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Add a timeout around the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:154`
  - body: This doctor check calls the npm registry with a bare `fetchFn(NPM_REGISTRY_URL)`. On networks that blackhole outbound HTTPS, a non-essential install note can stall `openclaw doctor`; use the existing timeout fetch helper or an abort signal so failures become inconclusive quickly.
  - confidence: 0.92
- **[P2] Do not revive expired registry cache entries:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After `loadFreshCache` rejects stale cache data, the fallback still reads the same cache file without checking age. That can emit false `version-unreleased` or rebuild warnings from expired registry data during offline or outage runs, instead of staying silent/inconclusive as the PR contract describes.
  - confidence: 0.9
- **[P2] Validate build-info version against package.json:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence while the registry lookup uses `dist/build-info.json.version`. A partial install with those files out of sync can evaluate the wrong version and suppress the corruption signal this guard is meant to provide.
  - confidence: 0.89
- **[P3] Distinguish malformed JSON from missing metadata:** `src/commands/doctor-rebuild-guard.ts:67-73`
  - body: `readJsonFile` collapses missing files and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That points operators at the wrong remediation for a corrupt metadata file.
  - confidence: 0.86
