## Summary

Keep this PR open. Current main still lacks the requested warn-only doctor check for locally rebuilt installs versus pristine npm releases, but the submitted patch has concrete correctness and responsiveness issues that should be fixed before maintainer merge review.

## What This Changes

The PR adds a doctor rebuild guard that reads local package build metadata, compares it with npm registry publish times using a cache, warns on rebuilt/unreleased/corrupt installs, and adds doctor wiring, tests, docs, and a changelog entry.

## Best Possible Solution

Keep the feature as a doctor-owned diagnostic, but update it to fit the current lazy doctor flow, use bounded registry fetching, avoid warnings from expired registry data, treat mismatched local metadata as corruption, distinguish malformed from missing metadata, and bring docs/changelog/tests in line with repo policy.

## Reproduction Assessment

Yes. Current-main inspection reproduces the missing feature by showing no guard symbols, no doctor call, no contribution, and no docs section; the PR's synthetic tests describe the intended rebuilt/unreleased/corrupt/pristine cases, though they were not executed in this read-only review.

## Solution Assessment

No, not as submitted. The requested direction is reasonable, but the patch should be tightened around network timeout behavior, cache freshness, package/build-info version parity, metadata diagnostics, and current doctor integration before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Add a timeout around the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:154`
  - body: This new doctor path calls the npm registry with a bare `fetchFn(NPM_REGISTRY_URL)`. On networks that blackhole or severely delay outbound HTTPS, a non-essential install note can stall `openclaw doctor` instead of returning an inconclusive silent result; use the existing timeout fetch helper or an abort signal.
  - confidence: 0.92
- **[P2] Do not revive expired registry cache entries:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After rejecting stale cache entries with `loadFreshCache`, the fallback still reads the same cache file without checking age. That can emit false `version-unreleased` or rebuild warnings from old registry data during offline/outage runs, contrary to the PR's silent-on-inconclusive contract.
  - confidence: 0.9
- **[P2] Validate build-info version against package.json:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence, while the registry lookup uses `dist/build-info.json.version`. A partial or interrupted install with those files out of sync can evaluate the wrong version and suppress the corruption signal this guard is meant to provide.
  - confidence: 0.89
- **[P3] Distinguish malformed JSON from missing metadata:** `src/commands/doctor-rebuild-guard.ts:67-73`
  - body: `readJsonFile` collapses read failures and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That sends operators toward the wrong remediation for a corrupt metadata file.
  - confidence: 0.86
- **[P2] Fix the changelog section and attribution:** `CHANGELOG.md:7-9`
  - body: The added user-facing feature entry creates a `### Features` section with `#TBD` and no contributor credit. Current repo policy uses the active `### Changes`/`### Fixes` sections and requires real PR/reference text plus `Thanks @...` attribution for user-facing entries.
  - confidence: 0.84
