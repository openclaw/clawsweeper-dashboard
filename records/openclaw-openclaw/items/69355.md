## Summary

Keep open. Current main still lacks this doctor rebuild/freshness diagnostic, and the PR is a plausible implementation path, but the submitted head has merge-blocking issues around stale doctor wiring, unbounded network behavior, stale-cache fallback, metadata validation, and changelog policy.

## What This Changes

The PR adds an `openclaw doctor` rebuild guard that compares local package build metadata with npm registry publish times, caches registry data, emits install notes, and updates tests, docs, and changelog.

## Best Possible Solution

Land a doctor-owned, warn-only install freshness diagnostic that preserves the current lazy doctor flow, uses bounded npm registry fetching, treats expired or unavailable registry data as inconclusive, validates local metadata consistency, distinguishes missing from malformed metadata, and updates docs/changelog/tests in repo style.

## Reproduction Assessment

Yes. Source and diff inspection reproduce both sides: current main has no rebuild guard, and the PR patch visibly contains the stale doctor wiring and narrow guard defects. The PR's test matrix covers the intended rebuilt/unreleased/corrupt/pristine cases, but tests were not run during this read-only review.

## Solution Assessment

No, not as submitted. The feature belongs in `openclaw doctor`, but the implementation should be tightened and rebased before merge so the diagnostic cannot stall doctor, warn from expired cache data, or miss corrupted local metadata.

Full review comments:

- **[P2] Rebase onto the lazy doctor flow:** `src/flows/doctor-health.ts:6`
  - body: The PR head still has eager top-level doctor imports, including the new rebuild guard import, while current main lazy-loads doctor paths so headless doctor avoids plugin-heavy startup work. Wire the guard through the current dynamic import block near `noteSourceInstallIssues` instead of landing this stale file shape.
  - confidence: 0.9
- **[P2] Add a timeout around the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:154`
  - body: This doctor path calls the npm registry with a bare `fetchFn(NPM_REGISTRY_URL)`. On networks that blackhole or delay outbound HTTPS, a non-essential install note can stall `openclaw doctor`; use the existing timeout fetch helper or an abort signal and return inconclusive on timeout.
  - confidence: 0.92
- **[P2] Honor cache TTL when registry lookup falls back:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After rejecting stale cache entries with `loadFreshCache`, the fallback still reads the same cache file without checking age. That can emit false rebuild or `version-unreleased` warnings from old registry data during offline/outage runs, contrary to the silent-on-inconclusive contract.
  - confidence: 0.9
- **[P2] Validate build-info version against package.json:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence, while the registry lookup uses `dist/build-info.json.version`. A partial or interrupted install with those files out of sync can evaluate the wrong version and suppress the corruption signal this guard is meant to provide.
  - confidence: 0.89
- **[P2] Fix the changelog section and attribution:** `CHANGELOG.md:7-9`
  - body: The added user-facing feature entry creates a `### Features` section with `#TBD` and no contributor credit. Current repo policy uses the active `### Changes`/`### Fixes` sections and requires concrete reference text plus `Thanks @...` attribution for user-facing entries.
  - confidence: 0.86
- **[P3] Distinguish malformed JSON from missing metadata:** `src/commands/doctor-rebuild-guard.ts:67-73`
  - body: `readJsonFile` collapses read failures and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That points operators at the wrong remediation for a corrupt metadata file.
  - confidence: 0.84
