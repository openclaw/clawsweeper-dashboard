## Summary

Keep open. The diagnostic is useful and not already on current main, but the PR branch still has merge-blocking correctness issues around unbounded registry I/O, stale cache fallback, and misleading local metadata evaluation.

## What This Changes

The PR adds a warn-only `openclaw doctor` install freshness check comparing local `dist/build-info.json` metadata with npm publish times, plus docs, changelog, and tests.

## Best Possible Solution

Land a doctor-owned diagnostic that remains warn-only, uses bounded npm registry I/O, treats expired registry data as inconclusive, validates local metadata parity, and keeps docs/tests aligned.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the unbounded fetch, TTL-bypassing fallback, and missing package/build-info version parity, while current main inspection confirms the requested diagnostic is not already present.

## Solution Assessment

No, not as submitted. The feature belongs in `openclaw doctor`, but the implementation should be repaired so an optional warning cannot hang doctor or emit stale/misleading install diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Add a timeout to the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This doctor check calls the npm registry with a bare `fetchFn(NPM_REGISTRY_URL)`. On networks that blackhole outbound HTTPS, a non-essential install note can stall `openclaw doctor`; use the existing bounded fetch helper or an abort signal so failures become inconclusive quickly.
  - confidence: 0.92
- **[P2] Honor the cache TTL when registry lookup falls back:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After rejecting stale cache data through `loadFreshCache`, the fallback still reads the same cache file without checking age. Offline or outage runs can then emit false `version-unreleased` or rebuild warnings from expired registry data instead of staying silent when current data is unavailable.
  - confidence: 0.9
- **[P2] Validate build-info version against package.json:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence while the registry lookup uses `dist/build-info.json.version`. A partial install with those files out of sync can evaluate the wrong version and suppress the corruption signal this guard is meant to provide.
  - confidence: 0.89
- **[P3] Distinguish malformed metadata from missing files:** `src/commands/doctor-rebuild-guard.ts:67-72`
  - body: `readJsonFile` collapses missing files and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That sends operators toward the wrong remediation for a corrupt metadata file.
  - confidence: 0.86
