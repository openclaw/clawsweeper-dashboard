## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This runs in the normal `openclaw doctor` path, but the registry request has no abort signal or timeout. On networks that blackhole HTTPS, a nonessential diagnostic can stall doctor for the transport timeout instead of staying quick and inconclusive.
  - confidence: 0.93
- **[P2] Keep stale cache from producing warnings:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After rejecting an expired cache entry, the fallback still loads the same cache file unconditionally when fetch is unavailable or offline. That can turn old registry data into fresh `version-unreleased` or rebuild warnings, violating the PR's silent-on-inconclusive behavior.
  - confidence: 0.92
- **[P2] Reject mismatched package metadata:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is read but not compared with `dist/build-info.json.version`, so a partial update can evaluate the wrong version and report `pristine` or `unreleased` instead of corrupt local metadata. Treat mismatched versions as corrupt before looking up npm publish time.
  - confidence: 0.9
- **[P2] Scope the guard to package installs:** `src/flows/doctor-health.ts:47`
  - body: `doctorCommand` passes the resolved root for both source checkouts and package installs. In a normal source checkout with no `dist/build-info.json`, this emits a corrupt install warning even though there is no npm package rebuild to compare.
  - confidence: 0.86
- **[P3] Distinguish malformed metadata from missing files:** `src/commands/doctor-rebuild-guard.ts:67-72`
  - body: `readJsonFile` returns `null` for both missing files and malformed JSON, so malformed `build-info.json` is reported as `build-info-missing`. Keep those reasons separate so the warning sends operators to the right repair path.
  - confidence: 0.78
- **[P3] Remove the release-owned changelog edit:** `CHANGELOG.md:6-8`
  - body: This is a normal feature PR, but OpenClaw release generation owns `CHANGELOG.md`. Keep release-note context in the PR body or commit message and let the release process update the changelog.
  - confidence: 0.82
