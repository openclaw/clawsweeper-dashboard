## Summary

Keep open: the PR addresses a real missing doctor diagnostic on current main, but the submitted branch still has correctness blockers around unbounded registry I/O, stale cache fallback, local metadata parity, and corrupt-metadata reporting.

## What This Changes

Adds a warn-only `openclaw doctor` rebuild guard that compares `dist/build-info.json` with npm registry publish times, plus docs, changelog text, and focused tests.

## Best Possible Solution

Land a rebased doctor-owned diagnostic that stays warn-only, uses bounded registry I/O, treats expired registry data as inconclusive, validates local metadata parity, and keeps docs aligned.

## Reproduction Assessment

Yes. Current-main source inspection shows the doctor install-note path but no build-info/npm publish comparison, and the PR diff gives source-level reproductions for the stale-cache, version-mismatch, malformed-metadata, and unbounded-fetch defects.

## Solution Assessment

No, not as submitted. The direction fits `openclaw doctor`, but the implementation needs bounded network behavior, TTL-respecting cache fallback, package/build-info version parity, clearer corrupt-metadata handling, and a rebase onto the current doctor flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Add a timeout to the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This doctor check calls the npm registry with a bare `fetchFn(NPM_REGISTRY_URL)`. On networks that blackhole outbound HTTPS, a non-essential install note can stall `openclaw doctor`; use the existing timeout fetch helper or an abort signal so failures become inconclusive quickly.
  - confidence: 0.92
- **[P2] Honor cache TTL when registry lookup falls back:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After `loadFreshCache` rejects stale cache data, the fallback still reads the same cache file without checking age. That can emit false `version-unreleased` or rebuild warnings from expired registry data during offline or outage runs instead of staying silent when current registry data is unavailable.
  - confidence: 0.9
- **[P2] Validate build-info version against package.json:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence while the registry lookup uses `dist/build-info.json.version`. A partial install with those files out of sync can evaluate the wrong version and suppress the corruption signal this guard is meant to provide.
  - confidence: 0.89
- **[P3] Distinguish malformed metadata from missing files:** `src/commands/doctor-rebuild-guard.ts:67-73`
  - body: `readJsonFile` collapses missing files and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That points operators at the wrong remediation for a corrupt metadata file.
  - confidence: 0.86
