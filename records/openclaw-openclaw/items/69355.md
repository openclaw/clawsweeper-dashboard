Full review comments:

- **[P2] Skip source checkouts before warning on metadata:** `src/commands/doctor-rebuild-guard.ts:78-80`
  - body: `doctorCommand` can pass a source checkout root into install-posture checks, and source checkouts normally lack packaged `dist/build-info.json`. This branch would warn those users that install metadata is corrupt even though there is no npm package rebuild to compare, so prove the root is a packaged install before warning.
  - confidence: 0.9
- **[P2] Bound the npm registry request:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This check runs in normal `openclaw doctor`, but the registry request has no timeout or abort signal. On blackholed or very slow HTTPS paths, a nonessential warning check can stall doctor instead of becoming inconclusive.
  - confidence: 0.93
- **[P2] Keep stale registry cache from emitting warnings:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After the fresh-cache check rejects an expired entry, the fallback still loads the same cache file unconditionally when fetch is unavailable or offline. That can turn stale registry data into `version-unreleased` or rebuild warnings, which conflicts with the stated silent-on-inconclusive behavior.
  - confidence: 0.92
- **[P2] Validate build-info version against package.json:** `src/commands/doctor-rebuild-guard.ts:90-91`
  - body: `package.json.version` is read but not compared with `dist/build-info.json.version`, so a partial update can evaluate the wrong version and report `pristine` or `unreleased` instead of corrupt local metadata. Treat mismatched versions as corrupt before looking up npm publish time.
  - confidence: 0.9
- **[P2] Avoid adding a JSON cache under the state directory:** `src/commands/doctor-rebuild-guard.ts:50-51`
  - body: Normal doctor would create `~/.openclaw/rebuild-guard-cache.json`, but OpenClaw-owned runtime caches should use the canonical state storage rather than adding another sidecar JSON cache. Either avoid persistent caching here or route it through the current state-store owner.
  - confidence: 0.84
- **[P3] Drop the release-owned changelog edit:** `CHANGELOG.md:8`
  - body: `CHANGELOG.md` is release-owned in this repo, so a normal feature PR should keep release-note context in the PR body or squash message instead of editing the release changelog directly.
  - confidence: 0.88
