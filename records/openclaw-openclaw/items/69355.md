## Summary

Keep open. Current main still lacks this doctor diagnostic, and the PR is useful, but the submitted head has merge-blocking correctness issues around unbounded registry I/O, stale cache fallback, and mismatched local metadata handling.

## What This Changes

The PR adds a warn-only `openclaw doctor` rebuild guard, docs, tests, and a changelog entry comparing local `dist/build-info.json` metadata with npm registry publish timestamps.

## Best Possible Solution

Land a rebased warn-only doctor install diagnostic that uses bounded registry I/O, treats expired or unavailable registry data as inconclusive, validates package/build metadata parity, distinguishes malformed metadata, and keeps docs/tests aligned.

## Reproduction Assessment

Yes, at source level. Current main has no rebuild guard in `doctor`, and the PR-head blockers are directly visible in the new guard source without needing a live install mutation.

## Solution Assessment

No, not as submitted. The doctor-owned diagnostic is the right surface, but the patch should be repaired so an optional warning cannot stall doctor or warn from stale or mismatched metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Bound the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This warn-only doctor check calls `fetchFn(NPM_REGISTRY_URL)` without a timeout or abort signal. On networks that blackhole outbound HTTPS, `openclaw doctor` can stall on transport timeout; use bounded fetch and return inconclusive on timeout.
  - confidence: 0.92
- **[P2] Honor cache TTL on fallback:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After `loadFreshCache` rejects stale cache data, the fallback reloads the same cache file without checking age. Offline or outage runs can emit false rebuild or `version-unreleased` warnings from expired publish data instead of staying silent when current registry data is unavailable.
  - confidence: 0.9
- **[P2] Validate package and build-info versions:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence while the registry lookup uses `dist/build-info.json.version`. If an update leaves those files out of sync, the guard can evaluate the wrong version and suppress the corruption signal it is meant to provide.
  - confidence: 0.89
- **[P3] Report malformed metadata separately:** `src/commands/doctor-rebuild-guard.ts:67-72`
  - body: `readJsonFile` collapses missing files and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That sends operators toward the wrong remediation for a corrupt metadata file.
  - confidence: 0.86
