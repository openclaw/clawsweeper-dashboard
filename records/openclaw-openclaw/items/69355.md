## Summary

Keep open: the diagnostic is plausible, but the current branch is not ready because it would warn on normal source checkouts, uses unbounded registry I/O, and can reuse expired registry cache data after declaring the cache stale.

## What This Changes

The branch adds a warn-only `openclaw doctor` rebuild guard, docs, tests, and changelog entry comparing local build metadata to npm publish times.

## Best Possible Solution

Land a rebased doctor diagnostic that skips source workspaces, uses bounded registry fetches, treats expired cache as inconclusive, validates metadata parity, and preserves the current lazy-loaded doctor flow.

## Reproduction Assessment

Yes for the review findings: current-main source shows source checkouts resolve as OpenClaw package roots with source markers and no build-info file, while PR-head code treats missing build-info as corrupt. The PR diff also shows the direct unbounded fetch, stale cache fallback, version mismatch gap, and malformed-file diagnostic path.

## Solution Assessment

No, not as submitted. The feature belongs in `openclaw doctor`, but the implementation needs the focused safeguards above before it is a maintainable warning-only diagnostic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Skip source checkouts before evaluating metadata:** `src/commands/doctor-rebuild-guard.ts:329`
  - body: `resolveOpenClawPackageRoot` can return the source repo root, and source checkouts have `pnpm-workspace.yaml` plus `src/entry.ts` but no `dist/build-info.json`. This added call would warn that a normal source-run `openclaw doctor` has corrupt install metadata unless the guard skips source workspaces.
  - confidence: 0.9
- **[P2] Bound the npm registry fetch:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This warn-only doctor check calls the registry fetch directly without a timeout or abort signal. On networks that blackhole outbound HTTPS, `openclaw doctor` can stall on transport timeout instead of treating the optional registry check as inconclusive.
  - confidence: 0.92
- **[P2] Honor cache TTL on registry fallback:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After rejecting stale cache data via `loadFreshCache`, the fallback reloads the same cache file without checking age. Offline or outage runs can emit false rebuild or `version-unreleased` warnings from expired publish data instead of staying silent when current registry data is unavailable.
  - confidence: 0.9
- **[P2] Validate package and build-info versions:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is only checked for existence while the registry lookup uses `dist/build-info.json.version`. If an update leaves those files out of sync, the guard can evaluate the wrong version and miss the corruption signal it is meant to surface.
  - confidence: 0.89
- **[P3] Distinguish malformed metadata from missing files:** `src/commands/doctor-rebuild-guard.ts:67-72`
  - body: `readJsonFile` collapses missing files and JSON parse failures into `null`, so malformed `build-info.json` is reported as `build-info-missing`. That sends operators toward the wrong remediation for a corrupt metadata file.
  - confidence: 0.86
