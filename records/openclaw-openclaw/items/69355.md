## Review Findings

Full review comments:

- **[P2] Skip source checkouts before warning about corrupt metadata:** `src/commands/doctor-rebuild-guard.ts:79-80`
  - body: `doctorCommand` can pass a source checkout root, which usually has `package.json` but no packaged `dist/build-info.json`. This would warn source users that their install metadata is corrupt even though there is no npm package rebuild to compare, so the guard should prove it is inspecting a packaged install or stay silent.
  - confidence: 0.9
- **[P2] Bound the npm registry lookup:** `src/commands/doctor-rebuild-guard.ts:162`
  - body: This check runs in normal `openclaw doctor`, but the registry request has no timeout or abort signal. On blackholed or very slow HTTPS paths, a nonessential warning check can stall doctor instead of becoming inconclusive; use the existing bounded fetch pattern.
  - confidence: 0.93
- **[P2] Do not warn from expired registry cache:** `src/commands/doctor-rebuild-guard.ts:212`
  - body: After the fresh-cache check rejects an expired entry, the fallback still loads the same cache file unconditionally when fetch is unavailable or offline. That can turn stale registry data into `version-unreleased` or rebuild warnings, which conflicts with the stated silent-on-inconclusive behavior.
  - confidence: 0.92
- **[P2] Reject mismatched package metadata:** `src/commands/doctor-rebuild-guard.ts:87-91`
  - body: `package.json.version` is read but not compared with `dist/build-info.json.version`, so a partial update can evaluate the wrong version and report `pristine` or `unreleased` instead of corrupt local metadata. Treat mismatched versions as corrupt before looking up npm publish time.
  - confidence: 0.9
- **[P3] Remove the release-owned changelog entry:** `CHANGELOG.md:8-10`
  - body: `CHANGELOG.md` is release-generated in this repo, so normal PRs should keep release-note context in the PR body or squash message instead of adding an Unreleased entry.
  - confidence: 0.86
