## Summary

Keep this PR open. Current main still has the allowlist-mode heredoc approval regression, but this PR is not merge-ready because the new detector misses strip-tab heredocs and the added tests do not match the current gateway allowlist helper contract.

## What This Changes

This PR changes the gateway exec allowlist heredoc gate to detect only unquoted heredocs from raw segment text, updates the warning copy, and adds tests for quoted, unquoted, and backslash-quoted heredocs.

## Best Possible Solution

Keep the PR open and rework it narrowly around the existing shell parser semantics or a shared heredoc-classification helper. The final behavior should auto-approve quoted and backslash-quoted heredocs when the binary and enforced plan are allowlisted, still require approval for unquoted `<<EOF` and `<<-EOF` forms per the PR’s policy, and add direct gateway allowlist tests that capture warnings and assert the actual `processGatewayAllowlist` return contract.

## Reproduction Assessment

Yes. The linked bug has concrete allowlist-mode steps, and current main source still shows every `<<` argv token forcing approval after allowlist analysis succeeds; the PR defects are reproducible by inspecting the raw regex against `<<-EOF` and the undefined `warnings` assertions in the added tests.

## Solution Assessment

No. The direction is reasonable, but an ad hoc raw-text regex is too easy to drift from the analyzer, and the current tests would fail or assert the wrong execution path; a parser-backed helper plus corrected focused tests is the safer fix.
