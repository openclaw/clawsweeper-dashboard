## Summary

Keep this PR open, but do not merge it as submitted. Current main still has the reported quoted-heredoc approval regression, while the PR’s proposed replacement misses unquoted strip-tab heredocs and its added tests do not match the current `processGatewayAllowlist` contract; because this is exec approval policy, the remaining work needs maintainer/security review.

## What This Changes

This PR changes gateway exec allowlist heredoc approval logic to detect unquoted heredocs from raw segment text, updates the approval warning copy, and adds gateway allowlist tests for quoted, unquoted, and backslash-quoted heredocs.

## Best Possible Solution

Rework the PR around the existing shell analyzer semantics, ideally by exposing or sharing a heredoc classification helper instead of adding a second raw-text regex. The final behavior should allow quoted and backslash-quoted literal heredocs when the binary and enforced plan are allowlisted, require approval for unquoted `<<EOF` and `<<-EOF` if that policy is retained, add focused gateway tests that assert `processGatewayAllowlist` return values and warnings, and include the required changelog entry.

## Reproduction Assessment

Yes. The linked bug has concrete allowlist-mode steps, and current main source shows any `<<` argv token still forcing approval after allowlist analysis succeeds; the PR defects are reproducible by inspecting the proposed raw regex against `<<-EOF` and the added tests’ undefined `warnings` assertions.

## Solution Assessment

No. The intended direction is reasonable, but this implementation duplicates parser behavior with an incomplete regex and adds tests that do not compile or assert the wrong execution path; a parser-backed helper plus corrected gateway contract tests is the safer fix.
