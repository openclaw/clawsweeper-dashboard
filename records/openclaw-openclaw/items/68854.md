## Summary

Keep this PR open, but it is not merge-ready: it targets a real current-main gateway approval regression, yet the patch changes a host exec approval boundary with an incomplete heredoc detector and no real behavior proof.

## What This Changes

The PR changes gateway exec allowlist heredoc handling so quoted and backslash-quoted heredocs bypass the extra approval gate while unquoted heredocs still require approval.

## Best Possible Solution

Align the gateway decision with the existing heredoc analyzer semantics, add focused gateway tests for quoted, backslash-quoted, unquoted strip-tab, and unsafe expansion cases, then require redacted real CLI or gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main still treats any allowlist-satisfied heredoc argv token starting with `<<` as requiring approval in the gateway path. I did not run a live gateway flow because this review is read-only.

## Solution Assessment

No. The intended regression fix is valid, but this implementation duplicates heredoc parsing with an incomplete raw regex and adds tests that fail before proving the changed execution boundary.
