## Summary

Keep open. The underlying heredoc allowlist regression is still present on current main, but this PR is not mergeable as-is because the new detector misses unquoted strip-tab heredocs, the added tests are malformed, and external real behavior proof is still absent.

## What This Changes

The PR changes gateway exec allowlist heredoc detection so quoted and backslash-quoted heredocs bypass the extra heredoc approval gate while unquoted heredocs still require approval.

## Best Possible Solution

Rework the fix around the existing analyzer or a shared heredoc-classification helper, add gateway tests for quoted, backslash-quoted, unquoted strip-tab, and unsafe expansion cases, then attach redacted real CLI or gateway proof.

## Reproduction Assessment

Yes, source-reproducible. Current main still forces approval for any allowlist-satisfied heredoc token through `requiresHeredocApproval`; I did not run a live gateway flow in this read-only review.

## Solution Assessment

No. The intended regression fix is useful, but this implementation duplicates parser behavior with an incomplete raw regex and adds tests that do not compile or assert the wrong execution boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Recognize every unquoted heredoc delimiter:** `src/agents/bash-tools.exec-host-gateway.ts:165`
  - body: The detector only matches an identifier immediately after `<<`, so unquoted forms such as `<<-EOF` do not set `requiresHeredocApproval` even though the PR says they still need explicit approval. Reuse analyzer delimiter parsing or a shared helper before relaxing the gate.

  - confidence: 0.95
