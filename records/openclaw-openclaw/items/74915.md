## Summary

Keep this open. Current main still builds web_fetch fallback candidates from bundled-origin plugins before explicit or runtime provider matching, and the provided GitHub context shows #74962 is an open focused PR with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

External plugin-declared web fetch providers should be usable by web_fetch when selected explicitly or through runtime provider metadata.

## Best Possible Solution

Land #74962 or an equivalent narrow mainline change that lets non-sandboxed web_fetch resolve explicit and runtime-selected providers across active non-bundled plugin origins, while keeping sandboxed web_fetch on bundled providers and preserving guarded-fetch protections.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: an external provider is filtered out by the bundled-origin resolver, and the readability-disabled path throws the reported no-provider error when no bundled fallback resolves.

## Solution Assessment

Yes. Adjusting web_fetch provider resolution plus adding regression coverage is the narrow maintainable fix; documenting the public webFetchProviders contract as bundled-only would conflict with the current plugin SDK/docs direction.
