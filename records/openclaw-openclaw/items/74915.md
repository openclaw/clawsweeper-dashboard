## Summary

Keep this open. Current main still builds web_fetch provider candidates from bundled-origin plugin resolution before explicit or runtime provider matching, and the linked open PR with closing syntax is the active fix vehicle.

## What This Changes

The issue asks for web_fetch to resolve explicitly configured external webFetchProviders from active non-bundled plugins instead of treating provider lookup as bundled-only.

## Best Possible Solution

Review and land #74962 or an equivalent narrow fix that lets non-sandboxed, explicitly configured or runtime-selected web_fetch providers resolve across active non-bundled plugin origins while keeping sandboxed web_fetch restricted to bundled providers and preserving guarded-fetch, redirect, SSRF, and provider-payload normalization behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: external candidates are filtered out by the bundled-origin resolver, then the Readability-disabled path throws the reported no-provider error when provider fallback resolution returns null.

## Solution Assessment

Yes. The narrow maintainable solution is to adjust web_fetch provider resolution and add regression coverage for explicit/runtime non-bundled providers, rather than documenting the public plugin contract as bundled-only.
