## Summary

Keep this issue open. Current main still builds the `web_fetch` provider candidate list from bundled-origin plugins before matching explicit or runtime-selected providers, and the linked fix PR #74962 uses closing syntax but remains open/unmerged in the provided GitHub context.

## What This Changes

The issue requests that `web_fetch` resolve explicitly configured external `contracts.webFetchProviders` providers registered by non-bundled plugins instead of treating the fallback provider set as bundled-only.

## Best Possible Solution

The desired end state is for non-sandboxed `web_fetch` to resolve explicit and runtime-selected providers from the active plugin registry, while sandboxed `web_fetch` remains limited to bundled providers and existing SSRF, redirect, guarded fetch, payload normalization, and cache behavior stays intact.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: register an external web-fetch provider, configure it explicitly, set `tools.web.fetch.readability` to false, and current main still searches bundled providers before reaching the reported no-provider error.

## Solution Assessment

Yes. The narrow maintainable path is to adjust `src/web-fetch/runtime.ts` to use runtime/all-origin providers when appropriate and add regression coverage; documenting the contract as bundled-only would contradict the current plugin SDK and manifest docs.
