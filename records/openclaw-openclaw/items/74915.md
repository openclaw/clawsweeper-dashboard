## Summary

Keep this issue open. Current main still builds the `web_fetch` provider candidate set through the bundled-origin resolver before explicit or runtime-selected provider matching, and open PR #74962 is already linked with closing syntax for the focused fix.

## What This Changes

The issue asks for `web_fetch` to resolve explicitly configured external plugin web-fetch providers declared through `contracts.webFetchProviders` and `api.registerWebFetchProvider(...)`.

## Best Possible Solution

Review and land #74962 or an equivalent narrow change that lets non-sandboxed `web_fetch` resolve explicitly configured and runtime-selected providers across active external plugin origins, with regression coverage for explicit external providers and runtime-selected providers.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current main filters candidates to `origin: "bundled"`, then the Readability-disabled path throws the reported no-provider error when fallback resolution returns null.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust web-fetch provider resolution and tests; documenting `webFetchProviders` as bundled-only would conflict with the current public plugin contract.
