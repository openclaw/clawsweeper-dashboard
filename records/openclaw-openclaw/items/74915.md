## Summary

Keep this open. Current main still filters web_fetch provider candidates to bundled-origin plugins before explicit provider matching, and the linked open PR #74962 has closing syntax but has not merged yet.

## What This Changes

External web_fetch providers declared and registered by non-bundled plugins should be resolvable when web_fetch is explicitly configured or runtime-selected to use them.

## Best Possible Solution

The end state should let non-sandboxed web_fetch resolve explicitly configured or runtime-selected providers across the active plugin registry, while keeping sandboxed web_fetch restricted to bundled providers and preserving guarded fetch, redirect, SSRF, and provider payload normalization behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure an external webFetchProviders plugin, set tools.web.fetch.readability to false, and current main will still search only bundled providers before throwing the reported no-provider error.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust web_fetch provider resolution and regression tests for explicit/runtime non-bundled providers, rather than weakening the documented plugin contract or making webFetchProviders bundled-only.
