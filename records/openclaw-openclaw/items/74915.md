## Summary

Keep this open. Current main still resolves `web_fetch` fallback candidates through bundled-origin provider lookup before explicit or runtime provider matching, and the linked open PR with closing syntax is the active fix vehicle.

## What This Changes

External or non-bundled plugins that declare and register `webFetchProviders` should be usable by `web_fetch` when explicitly configured or runtime-selected.

## Best Possible Solution

Review and land #74962, or an equivalent narrow fix, so non-sandboxed `web_fetch` resolves explicitly configured and runtime-selected providers across active non-bundled plugin origins while sandboxed web fetch remains restricted to bundled providers.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: external providers are filtered out by the bundled-origin resolver, then the Readability-disabled path throws the reported no-provider error when fallback resolution returns null.

## Solution Assessment

Yes. The narrow maintainable direction is to make `web_fetch` honor runtime provider resolution like the adjacent `web_search` runtime while preserving the existing sandbox boundary and SSRF/fetch payload safeguards.
