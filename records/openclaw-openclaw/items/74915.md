## Summary

Keep this issue open. Current main still builds the `web_fetch` provider list from bundled-origin plugins before matching explicit or runtime-selected providers, while the plugin contract docs expose `webFetchProviders` and `api.registerWebFetchProvider(...)` for plugin authors. The linked fix PR #74962 uses closing syntax but is open and unmerged in the provided GitHub context, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

Request `web_fetch` to resolve explicitly configured external `contracts.webFetchProviders` registrations from non-bundled plugins instead of failing when Readability is disabled.

## Best Possible Solution

Review and land #74962 or an equivalent narrow change that resolves non-sandboxed `web_fetch` providers from the active/runtime plugin registry for explicit and runtime-selected provider ids, while keeping sandboxed fetch bundled-only and adding regression coverage for external providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register an external web-fetch provider, configure it explicitly, set `tools.web.fetch.readability` to `false`, and observe that current main still searches bundled providers before reaching the reported no-provider error.

## Solution Assessment

Yes. The narrow maintainable path is to change `src/web-fetch/runtime.ts` to honor runtime/all-origin providers only where appropriate and cover the behavior in `src/web-fetch/runtime.test.ts`; documenting the contract as bundled-only would conflict with the current plugin manifest and SDK docs.
