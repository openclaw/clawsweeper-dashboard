## Summary

Keep this issue open. Current main still builds web_fetch fallback candidates from bundled-origin plugins before explicit or runtime provider selection, and the focused PR that uses closing syntax for this issue is still open and unmerged.

## What This Changes

The issue asks OpenClaw to let web_fetch resolve explicitly configured external plugin webFetchProviders instead of limiting runtime lookup to bundled plugins.

## Best Possible Solution

Land a narrow resolver and test fix that uses runtime/all-origin provider resolution for non-sandboxed web_fetch explicit and runtime-selected providers, while keeping sandboxed web_fetch on bundled providers.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure an external webFetchProviders plugin, set tools.web.fetch.readability to false, and current main still searches only bundled providers before throwing the reported no-provider error.

## Solution Assessment

Yes for the requested direction. The narrow maintainable path is to adjust web_fetch provider resolution and regression tests, not to make the documented webFetchProviders plugin contract bundled-only.
