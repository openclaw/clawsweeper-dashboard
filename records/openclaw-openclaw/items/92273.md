## Label Justifications

- `P1`: A core automatic compaction workflow can silently lose durable memory for active long-running sessions.
- `impact:data-loss`: Transcript content can be evicted without being successfully written to the durable memory store.
- `impact:session-state`: Compaction can leave later turns or fresh sessions without context that existed before the maintenance turn.

## Summary

Current main and release v2026.6.5 still expose the reported failure class, while the open closing PR only improves unknown-tool recovery and has not verified the actual pre-compaction persistence path. The issue remains necessary as the canonical data-loss report until a fix lands with forced-flush proof.

## What This Changes

Prevent Tool Search unknown-id failures from causing pre-compaction memory flushes to lose durable session memories.

## Best Possible Solution

Make failed pre-compaction persistence non-destructive or explicitly unsuccessful before compaction, retain recoverable unknown-tool guidance as defense in depth, and add an end-to-end forced-flush regression that verifies the memory file and fresh-session recall.

## Reproduction Assessment

Yes at source level: current main contains the guessed-name input path, bare unknown-id response, silent memory-flush turn, and durability-sensitive compaction workflow, while the reporter supplies concrete gateway logs and a 3/3 behavioral sequence. The model choosing `file_write` is probabilistic, so this review does not claim an independently executed live reproduction.

## Solution Assessment

No, closest-match guidance alone is not the best complete solution because durable memory should not depend solely on a model deciding to retry during a silent maintenance turn. It is useful defense in depth, but the compaction boundary also needs failure-aware persistence behavior and direct regression proof.
