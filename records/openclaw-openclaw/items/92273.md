## Summary

Current main still permits a failed inner tool call to be treated as a completed memory flush, and the linked open PR only improves recovery guidance without proving the durable-memory path. Keep this issue open until the linked repair lands with end-to-end flush recovery coverage or the memory runner independently prevents false-success bookkeeping.

## What This Changes

Prevent compact tool-search failures from being recorded as successful pre-compaction memory flushes when no durable write occurred.

## Best Possible Solution

Propagate failed memory-flush tool calls as flush failures, preserve retry bookkeeping, and mark the cycle complete only when the maintenance run has no failed required write; retain closest-match Tool Search guidance as defense in depth.

## Reproduction Assessment

Yes, at source level with high confidence. A failed inner Tool Search call can occur during a completed silent maintenance run, after which current main records the memory-flush cycle as successful; the reporter also supplies repeated live logs and a threshold control.

## Solution Assessment

No, the linked closest-match change is useful but not sufficient by itself. The most reliable fix belongs at the memory-flush owner boundary, where failed required tool operations must prevent success bookkeeping and allow retry.
