## Summary

Current main still exposes the reported dead end: Tool Search accepts guessed names, returns a bare unknown-id error, and can compact the durability-critical flush tools behind that surface. The issue should remain open because https://github.com/openclaw/openclaw/pull/92278 explicitly claims it and is still open; its proposed recovery hint is relevant, but its own proof says the real model and memory-flush path were not tested.

## What This Changes

Prevent pre-compaction durable-memory loss when Tool Search receives a guessed, unavailable tool name during the silent memory-flush turn.

## Best Possible Solution

Land a narrow recovery fix only after proving the actual pre-compaction turn retries from an unknown guessed name to the restricted `write` tool and persists memory; if it does not, add explicit flush-success enforcement at the compaction boundary rather than relying on error wording alone.

## Reproduction Assessment

Yes, at source level with high confidence: tools mode permits by-name calls, unknown names receive terminal bare feedback, and the memory flush depends on selecting the restricted write tool. The reporter also supplied a concrete gateway sequence and 3/3 observations, though this review did not run the model-driven scenario.

## Solution Assessment

No, not yet proven as the best complete fix: nearest-match guidance is the narrowest class-level improvement, but the durability-critical path needs real recovery proof or explicit flush-success enforcement before compaction.
