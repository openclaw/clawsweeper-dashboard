## Label Justifications

- `P1`: A long-running agent can silently cross compaction after a falsely successful memory flush, causing user-visible recall loss.
- `impact:data-loss`: Context can be evicted after the only intended durable copy was never written.
- `impact:session-state`: The incorrect flush marker changes the memory available to later turns and fresh sessions.

## Summary

Current main supports the reported failure class: Tool Search accepts guessed names, unknown names receive dead-end feedback, and a resolved memory-flush run is marked successful without an explicit persistence outcome. This issue must also remain open because https://github.com/openclaw/openclaw/pull/92278 explicitly claims it and is still open.

## What This Changes

Prevent compact Tool Search failures during pre-compaction maintenance from marking durable memory as flushed when the intended memory write did not occur.

## Best Possible Solution

Give memory maintenance a closed persisted/no-op/failed outcome and record the compaction cycle as flushed only after an explicit successful write or legitimate no-op; retain closest-match guidance as complementary general Tool Search recovery.

## Reproduction Assessment

Yes at source level: an unknown tool call may remain inside a resolved agent run, after which the caller records a successful flush without an explicit persistence result. The reporter also supplied three repeated production observations of the probabilistic wrong-name trigger.

## Solution Assessment

No, closest-match suggestions alone are not clearly the best complete fix because the durability boundary still trusts a fallible silent agent turn. The narrower robust design is explicit maintenance success semantics, with improved unknown-tool feedback retained for general recovery.
