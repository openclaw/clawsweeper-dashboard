## What This Changes

The issue reports that Tool Search `mode: "tools"` can make memory flushes lose durable session memory when the model guesses an unavailable hidden tool name and receives an unrecoverable `Unknown tool id` error.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92278 or an equivalent narrow recoverable unknown-tool error fix, then open a separate follow-up only if maintainers want memory flush persistence to stop depending on model-selected file writes.

Yes for the source-level bug path: current main exposes `tool_call` with by-name ids and still returns a bare unknown-id error when a hidden or unavailable name is guessed. I did not run the full gateway/model memory-flush harness, so the behavioral model choice remains reporter-provided rather than locally reproduced.

## Solution Assessment

Mostly yes: the open linked PR addresses the smallest current code defect by making unknown tool ids recoverable without adding config or changing the memory storage model. Deterministic runtime-side memory persistence could be stronger, but that is a broader follow-up rather than a reason to close this issue before the narrow fix lands.
