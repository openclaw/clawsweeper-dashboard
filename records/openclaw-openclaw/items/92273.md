## Label Justifications

- `P1`: The report describes repeatable silent loss of intended durable memory during a core compaction workflow affecting long-running sessions.
- `impact:data-loss`: Memory content selected for durable persistence can be evicted without ever being written.

## Summary

Current main still contains the reported failure path, and the open closing PR only improves unknown-tool feedback without proving the pre-compaction durability boundary. Keep the issue open until that PR or a stronger replacement demonstrates that a failed flush cannot be followed by silent memory loss.

## What This Changes

Prevent compact Tool Search from causing pre-compaction durable-memory loss when the model calls an unavailable guessed tool name.

## Best Possible Solution

Retain recoverable unknown-tool guidance, but also prove or enforce that compaction cannot discard unpersisted durable memory after a failed flush tool call.

## Reproduction Assessment

Yes, at high confidence from current source plus the reporter's repeated gateway logs, missing-file checks, and fresh-session recall control. The model choosing the wrong name is probabilistic, so this review classifies the issue as source-reproducible rather than independently reproduced.

## Solution Assessment

No, not conclusively: closest-match feedback is a good generic recovery improvement, but it does not by itself guarantee the stronger durability requirement that a failed flush cannot be compacted away.
