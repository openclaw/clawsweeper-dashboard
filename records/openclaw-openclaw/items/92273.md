## Summary

Keep open: current main and the latest release still return a bare Tool Search unknown-id error, and this issue is already paired with the open closing PR https://github.com/openclaw/openclaw/pull/92278. The central memory-loss/session-state bug is not implemented on main yet.

## What This Changes

The issue asks OpenClaw to make Tool Search `mode: "tools"` recover from guessed or unavailable tool ids so pre-compaction memory flushes do not silently lose durable memory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92278 or an equivalent narrow recoverable unknown-tool error fix, then track deterministic memory-flush persistence separately only if maintainers want that broader design.

## Reproduction Assessment

Yes for the source-level bug path: current main exposes by-name `tool_call` ids and still returns a bare unknown-id error when a hidden or unavailable name is guessed. I did not rerun the full gateway/model memory-flush harness, so the exact model choice remains based on reporter logs.

## Solution Assessment

Yes for the immediate bug: a recoverable unknown-tool error is the narrowest maintainable fix without adding config or changing the memory storage model. Runtime-side memory persistence could be stronger, but that is a broader follow-up rather than a reason to close this issue before the linked fix lands.
