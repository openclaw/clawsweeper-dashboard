## Summary

Keep open: current main still has the recoverability bug in Tool Search, and the issue is paired with an open closing PR that has not merged yet. Closing now would hide an active data-loss/session-state bug from maintainer tracking.

## What This Changes

This issue reports that Tool Search tools mode lets a memory-flush turn call an unavailable guessed tool name, causing a bare unknown-id error and lost durable memory after compaction.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92278 or an equivalent narrow recoverable unknown-tool error fix, then track deterministic memory persistence as a separate hardening follow-up if maintainers want it.

## Reproduction Assessment

Yes for the source-level bug path: current main exposes only Tool Search controls in tools mode, permits `tool_call` by name, and throws a bare `Unknown tool id` for misses. I did not rerun the full gateway/model memory-flush harness, so the 3/3 live model behavior remains reporter-provided.

## Solution Assessment

Mostly yes: the open linked PR addresses the smallest current code defect by making unknown tool ids recoverable without adding config or changing the memory storage model. Runtime-side deterministic memory persistence would be stronger hardening, but it is a broader follow-up rather than a reason to close this issue before the narrow fix lands.
