## Summary

Keep open: the feature is not on current main, and the PR has blocking issues before merge. The mutating `/codex plugins` verbs are reachable by ordinary authorized `/codex` senders, and `/codex plugins add` can report success while leaving the parent `codexPlugins` policy disabled on a fresh config.

## What This Changes

The PR adds a Codex-owned `/codex plugins` subcommand for listing and mutating Codex sub-plugin config, widens first-party OpenAI marketplace handling, and adds focused Codex tests.

## Best Possible Solution

Keep the Codex-owned command direction, but merge only after mutating verbs are owner/admin-gated, fresh-config add behavior activates or clearly rejects the parent-disabled state, and the related marketplace support boundary is resolved.

## Reproduction Assessment

Not applicable for the requested feature itself; current main simply lacks `/codex plugins`. For the blocking review findings, yes: source inspection shows the missing owner gate and the parent `codexPlugins.enabled` mismatch without needing a live run.

## Solution Assessment

No. A plugin-owned `/codex plugins` command is a reasonable direction, but the current patch is not the best mergeable form until mutating chat actions are gated and add behavior matches the resolver contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate mutating plugin commands to owners:** `extensions/codex/src/command-handlers.ts:243`
  - body: This dispatch lets every authorized `/codex` sender reach `add`, `remove`, `enable`, `disable`, and `toggle`, and those verbs mutate `openclaw.json` globally. In shared channels, an authorized non-owner could enable first-party Codex plugins such as browser/computer-use or remove entries for everyone. Please owner/admin-gate the mutating verbs while keeping read-only `list` and `help` accessible as intended.
  - confidence: 0.88
- **[P2] Enable the parent codexPlugins block when adding:** `extensions/codex/src/command-plugins-management.ts:127-132`
  - body: `/codex plugins add` only creates an enabled child entry. On a config without `codexPlugins.enabled: true`, `resolveCodexPluginsPolicy` still sets the parent policy disabled and every plugin remains inactive, so the command reports success but the plugin never loads after restart. Set the parent flag during add or reject with a clear instruction.
  - confidence: 0.93
