## Summary

Keep this PR open, but it is not merge-ready: the useful Codex chat-picker direction still carries an ungated persistent config-write path, includes an `add` flow that a domain reviewer says needs an OAuth-aware design, and leaves the widened marketplace contract out of user docs.

## What This Changes

Adds `/codex plugins` management and a Telegram in-place `/codex` picker, widens first-party Codex plugin marketplace handling, and adds focused Codex/Telegram tests.

## Best Possible Solution

Revise toward a smaller list/enable/disable/help surface with an explicit write-authority gate, align docs with the final marketplace boundary, and leave OAuth-aware add to separate Codex plugin work.

## Reproduction Assessment

Yes for the review findings: source inspection against the PR head shows the ungated config mutation path and config-only `add` behavior. The original request is a feature addition, not a current-main bug reproduction.

## Solution Assessment

No. A plugin-owned chat surface is a reasonable direction, but this patch is too broad until mutating verbs are gated, the add/OAuth scope is reduced or redesigned, and docs match the final config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate plugin config mutations before dispatch:** `extensions/codex/src/command-handlers.ts:490`
  - body: This branch sends `/codex plugins` calls to handlers that can enable, disable, toggle, add, or remove persisted `openclaw.json` entries after only the normal `/codex` authorization. Mutating verbs need an owner or explicit operator write/admin check before reaching the IO layer.
  - confidence: 0.88
- **[P2] Keep add out until OAuth-aware install exists:** `extensions/codex/src/command-plugins-management.ts:123-151`
  - body: `add` only writes a config entry and returns success, but the review discussion notes plugins with apps need the Codex/ChatGPT OAuth flow to be properly added. This can leave users with a successful-looking entry that is inactive or unusable, so the command should be dropped from this PR or wired to the real flow.
  - confidence: 0.86
- **[P2] Update the documented marketplace boundary:** `extensions/codex/openclaw.plugin.json:117`
  - body: The manifest now accepts `openai-bundled` and `openai-primary-runtime`, but the public Codex plugin docs still say V1 only supports `openai-curated`. Users would see a schema that accepts values the docs call unsupported.
  - confidence: 0.84
