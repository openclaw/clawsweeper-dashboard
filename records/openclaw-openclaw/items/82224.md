Full review comments:

- **[P2] Handle the advertised plugins menu command:** `extensions/codex/src/command-handlers.ts:248`
  - body: The new picker advertises `/codex plugins menu` as a button value and text fallback, but the `plugins` branch still delegates `menu` to `handleCodexPluginsSubcommand`, which only supports list/enable/disable and returns an unknown-subcommand help message. Add a real menu branch before dispatch or stop advertising this command outside the Telegram-only `cdx_plugins_menu` path.
  - confidence: 0.91
- **[P2] Use the effective plugin state in the picker:** `extensions/telegram/src/codex-picker-buttons.ts:205`
  - body: The dynamic enable/disable picker treats a plugin as enabled whenever the entry is not explicitly `enabled: false`, but `/codex plugins list` marks every plugin OFF when global `codexPlugins.enabled` is false. In a common fresh config with entries but the global switch off, the picker hides those inactive plugins from Enable and offers them under Disable; pass the global flag through and use the same effective state as the list command.
  - confidence: 0.88
- **[P3] Keep plugin callbacks under Telegram's byte limit:** `extensions/telegram/src/codex-picker-buttons.ts:222`
  - body: The plugin-name buttons build `callback_data` from an unbounded config key and send the inline keyboard directly to `editMessageText`; Telegram rejects callback data over 64 bytes, so a long configured plugin key can make the whole dynamic picker edit fail. Reuse the existing callback-data length guard or skip/shorten entries that cannot fit.
  - confidence: 0.76
