## Summary

Keep open: this is a plausible Codex feature with supplied real-use proof, but the current branch has blocking issues around ungated persistent config writes, ineffective first-time plugin adds when the global Codex plugin switch is off, and a stacked marketplace-docs mismatch.

## What This Changes

The PR adds `/codex plugins` list/enable/disable/toggle/add/remove/help handling, picker replies, wider first-party Codex plugin marketplace support, and focused Codex tests.

## Best Possible Solution

Land this only after mutating `/codex plugins` verbs are owner/operator-write gated, first-time adds align with the global `codexPlugins.enabled` contract, the marketplace docs are updated or the stack is rebased, and the related data-layer PR is resolved.

## Reproduction Assessment

Yes for the review findings: source inspection shows `/codex plugins add` can report success while leaving `codexPlugins.enabled` false, and the new dispatch has no write-authority check. The original request is a feature addition, not a broken existing contract.

## Solution Assessment

No. The plugin-owned `/codex` surface is a reasonable direction, but persistent config mutation from chat needs explicit owner/operator-write gating and the add path must honor the global Codex plugin enablement contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate plugin config mutations before dispatch:** `extensions/codex/src/command-handlers.ts:243`
  - body: This dispatch sends every `/codex plugins` call into handlers that can persistently mutate `openclaw.json`, but the parent `/codex` command only requires a generally authorized sender. That lets an authorized non-owner channel user add or enable Codex plugins with the default destructive-action policy, so the mutating verbs should require owner status or an operator write/admin scope before they call the IO layer.
  - confidence: 0.86
- **[P2] Set the global Codex plugin switch when adding entries:** `extensions/codex/src/command-plugins-management.ts:128-132`
  - body: The add path writes only `codexPlugins.plugins.<name>.enabled = true`. Current policy resolution still requires `codexPlugins.enabled === true`, so adding the first plugin to a config without that global switch returns a success message and restart hint but the plugin remains disabled after restart.
  - confidence: 0.91
- **[P2] Update the documented marketplace boundary:** `extensions/codex/openclaw.plugin.json:117`
  - body: This widens the manifest enum to accept `openai-bundled` and `openai-primary-runtime`, but the public Codex plugin docs still describe the V1 boundary as `openai-curated` only. After merge, users would see a schema that accepts values the docs still call unsupported, so the docs need to be updated or the enum kept at the documented boundary.
  - confidence: 0.88
