## Summary

Keep open. The current proof is useful, but the patch still has merge-blocking Telegram callback authorization, picker state, and invalid fallback-command issues that should be fixed before maintainer merge consideration.

## What This Changes

The branch adds a `/codex` picker/menu flow, Telegram `cdx_*` in-place callback handling, dynamic Codex plugin enable/disable pickers, and unit tests for the Telegram picker builders.

## Best Possible Solution

Land the picker only after `cdx_*` callbacks share the existing command authorization policy, fallback commands are valid, dynamic picker state matches `/codex plugins list`, and maintainers explicitly accept the `/codex` default change.

## Reproduction Assessment

Yes for the review findings: source inspection shows `cdx_*` callbacks render picker content without command-auth gating, the dynamic picker ignores `codexPlugins.enabled`, and `/codex plugins menu` falls through to the current unknown-subcommand path. I did not run tests because this was a read-only review.

## Solution Assessment

No. The picker is a useful direction, but the maintainable solution is to reuse the existing Telegram command authorization contract, make every advertised fallback/button command real, and derive picker state from the same effective config as the canonical plugin command handler.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate Codex picker callbacks with command auth:** `extensions/telegram/src/bot-handlers.runtime.ts:2620-2621`
  - body: The new `cdx_*` branch builds and edits Codex picker payloads directly, including the dynamic plugin-name picker from runtime config, but it never checks the command authorization helper used by model callbacks. In a group where inline buttons are broader than command permissions, a non-command-authorized user can still tap the picker and inspect configured Codex plugin names, so gate this branch before rendering or editing picker content.
  - confidence: 0.86
- **[P2] Honor the global Codex plugin enabled flag:** `extensions/telegram/src/codex-picker-buttons.ts:205-209`
  - body: The dynamic picker treats `entry.enabled !== false` as ON, but `/codex plugins list` treats every entry as OFF when `codexPlugins.enabled` is not true, and typed enable also flips the global flag back on. With global plugins disabled, the picker will offer those entries under `disable` and show no `enable` option to restore them, so pass the global flag through and filter on the effective state.
  - confidence: 0.9
- **[P2] Do not advertise unsupported plugins menu commands:** `extensions/codex/src/command-handlers.ts:248`
  - body: The no-arg picker tells fallback/non-Telegram users to type `/codex plugins menu`, but the plugins handler still only implements list/enable/disable and returns an unknown-subcommand response for `menu` and `help`. Either add real `menu`/`help` handling for the plugins subcommand or change the advertised/button routes to commands that already work.
  - confidence: 0.88
