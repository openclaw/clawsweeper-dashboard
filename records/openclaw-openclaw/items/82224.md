## Summary

Keep open: the PR is useful but not merge-ready because it adds persistent chat-driven Codex plugin config writes without a concrete write-authority gate, has a first-time add path that can report success while plugins remain globally disabled, and widens the marketplace contract without updating the public docs.

## What This Changes

The branch adds `/codex plugins` management and Telegram in-place Codex pickers, widens first-party Codex plugin marketplace handling, and adds focused Codex/Telegram tests.

## Best Possible Solution

Land this only after mutating plugin verbs are explicitly gated, first-time adds activate the global Codex plugin policy correctly, docs match the widened marketplace contract, and the related data-layer PR is resolved.

## Reproduction Assessment

Yes for the review findings: source inspection shows the config mutation path has no write-authority gate and the add path does not set the global enablement flag. The original request is a feature addition, not a current-main bug reproduction.

## Solution Assessment

No. A plugin-owned chat surface is a reasonable direction, but this patch is not merge-ready until the mutation authority boundary, first-time global enablement behavior, docs contract, and stacked data-layer dependency are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate plugin config mutations before dispatch:** `extensions/codex/src/command-handlers.ts:490`
  - body: This new branch sends every `/codex plugins` call to handlers that can add, enable, disable, toggle, or remove persisted `openclaw.json` entries, but `/codex` only requires basic sender authorization. Mutating verbs need an owner or explicit operator write/admin check before they reach the IO layer.
  - confidence: 0.86
- **[P2] Enable the global Codex plugins switch on add:** `extensions/codex/src/command-plugins-management.ts:140-145`
  - body: `add` writes only the per-plugin entry with `enabled: true`. The resolver still requires `codexPlugins.enabled === true`, so adding the first plugin to a config without that global switch reports success while the plugin remains disabled after refresh.
  - confidence: 0.91
- **[P2] Update docs for the widened marketplace enum:** `extensions/codex/openclaw.plugin.json:117`
  - body: The manifest now accepts `openai-bundled` and `openai-primary-runtime`, but the public Codex plugin docs and gateway configuration reference still say V1 only supports `openai-curated`. Users would see a schema that accepts values the docs call unsupported.
  - confidence: 0.88
