## Summary

Keep open. The feature is useful, but the current branch still disagrees with main’s update channel, downgrade, and git freshness contracts, and the external PR still lacks real CLI behavior proof.

## What This Changes

The PR adds a read-only `openclaw update review` CLI subcommand with JSON output, tests, registration under `openclaw update`, and update CLI docs.

## Best Possible Solution

Add the command by sharing main’s update channel, target, downgrade, and git freshness helpers, then align risk output with a maintainer-owned release-readiness source and require redacted real CLI proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head and current-main update helpers shows the beta-installed-channel, git-downgrade, and zero-behind cases. The feature itself still lacks real behavior proof from actual `openclaw update review` and `openclaw update review --json` runs.

## Solution Assessment

No, not as currently patched. The useful direction is maintainable, but this implementation reimplements channel and availability decisions instead of reusing the existing update contracts and leaves release-readiness policy unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the shared channel resolver for beta installs:** `src/cli/update-cli/review.ts:216-221`
  - body: A package install running a beta `VERSION` with no config falls back to `stable` here. Main resolves that case to beta via the shared channel helpers, so beta users can review the wrong target version and release notes.
  - confidence: 0.84
- **[P2] Scope downgrade reporting to package installs:** `src/cli/update-cli/review.ts:260-261`
  - body: `isDowngrade` is computed from the npm target for git installs too. A git checkout whose local `VERSION` is ahead of registry tags can be shown as a downgrade even though `openclaw update` only applies downgrade handling in the package path.
  - confidence: 0.88
- **[P3] Preserve zero-behind git freshness:** `src/cli/update-cli/review.ts:249-254`
  - body: This collapses `behind: 0` to `null`, losing the distinction between a fetched current checkout and unknown git status. If the registry lookup fails after git proved the checkout is current, the review reports an unknown state instead of preserving the git result.
  - confidence: 0.78
