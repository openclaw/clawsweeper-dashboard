## Summary

Keep this PR open. Current main still lacks `openclaw update review`, and the PR is a plausible feature path, but the branch still has correctness issues in update-channel/downgrade/git-freshness semantics and lacks required real behavior proof before merge.

## What This Changes

The PR adds a read-only `openclaw update review` CLI subcommand, tests, CLI registration, and update docs for npm/GitHub release-note risk assessment.

## Best Possible Solution

Implement the command by reusing or closely mirroring current update target/channel/downgrade/git-freshness helpers, add focused regression coverage for beta package installs and git freshness, then require real CLI output proof before merge.

## Reproduction Assessment

Yes for the PR blockers: source inspection and mocked paths show the downgrade-on-git and lost zero-behind cases without needing a live update host. The feature itself still needs real CLI proof before merge.

## Solution Assessment

No, not as currently patched. The command direction is useful, but the maintainable fix is to share the existing update channel, target, downgrade, and git availability model instead of reimplementing those decisions locally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor beta installed-version channel selection:** `src/cli/update-cli/review.ts:219-221`
  - body: A package install running a beta `VERSION` with no config is reviewed against the stable default because the command only uses config/defaults for `effectiveChannel`. Current update status resolves that case to beta, so beta users can see the wrong target and release notes; use the shared channel resolution path instead of defaulting package installs to stable.
  - confidence: 0.82
- **[P2] Scope downgrade reporting to package updates:** `src/cli/update-cli/review.ts:260-261`
  - body: `isDowngrade` is computed from the npm target for every install kind, so git checkouts whose local `VERSION` is ahead of registry tags are reported as downgrades even though `openclaw update` only applies downgrade handling in the package update path. Gate this on the package-mode target logic so git users see git freshness guidance instead.
  - confidence: 0.88
- **[P3] Preserve zero-behind git status:** `src/cli/update-cli/review.ts:249-254`
  - body: `gitBehind` collapses a fetched `behind: 0` result to `null`, and `checkUnavailable` later treats that as no git signal. If the npm lookup fails but git fetch proved the checkout is current, the command reports an unknown review-required state instead of the current git status.
  - confidence: 0.78
