## Summary

Keep open. Current main still lacks `openclaw update review`, and the PR is a useful implementation vehicle, but the current head still has correctness blockers in git/downgrade parity plus the required changelog entry is missing.

## What This Changes

The PR adds a read-only `openclaw update review` CLI subcommand with JSON output, npm/GitHub release-note risk heuristics, tests, CLI registration, and update documentation.

## Best Possible Solution

Keep this PR as the implementation vehicle, but make `update review` share or closely mirror the existing update target, downgrade, and git-freshness semantics, then add focused tests, the changelog entry, and rebase onto current main.

## Reproduction Assessment

Yes. A unit mock with `installKind: git`, local `VERSION` ahead of the npm target, and git status available reproduces the false downgrade path; a separate mock with `behind: 0`, successful git fetch, and registry unavailable reproduces the lost current-git signal.

## Solution Assessment

No, not as currently patched. The command direction is maintainable, but the safer implementation is to reuse or mirror the existing `openclaw update` target, downgrade, and git status model instead of independently classifying those states.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope downgrade detection to package installs:** `src/cli/update-cli/review.ts:261`
  - body: `isDowngrade` is computed for every install kind and then wins before git-specific branches. For git checkouts whose local `VERSION` is ahead of the npm tag, `update review` reports a downgrade even though `openclaw update` scopes downgrade handling to non-git package paths; gate this on package-mode target semantics so git users get git freshness guidance.
  - confidence: 0.88
- **[P3] Preserve zero-behind git status:** `src/cli/update-cli/review.ts:249-254`
  - body: `gitBehind` stores only positive counts, so a fetched `behind: 0` git checkout is indistinguishable from unknown status. If npm tag lookup fails, `checkUnavailable` reports a review-required unknown state even though git fetch proved the checkout is current; retain the numeric behind value separately from `hasGitUpdate`.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/cli/update-cli.ts:160-161`
  - body: This introduces a user-facing CLI command and docs, but the branch does not update `CHANGELOG.md`. Add a single-line entry under the active `### Changes` section before merge.
  - confidence: 0.96
