## Summary

Keep open. Current main still does not have `openclaw update review`, but the PR needs real CLI proof and still diverges from main’s update channel, downgrade, and git freshness contracts; the release-risk policy also overlaps open maintainer-owned update safety discussions.

## What This Changes

## Best Possible Solution

If maintainers want this command, implement it by reusing main’s update channel, target, downgrade, and git freshness helpers, then tie risk guidance to a maintainer-owned release-readiness source and require redacted real CLI proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head against current-main update helpers shows the beta-installed-version, git-downgrade, and zero-behind cases. The feature itself still lacks real behavior proof from actual `openclaw update review` and `openclaw update review --json` runs.

## Solution Assessment

No, not as currently patched. The direction is plausible, but the implementation duplicates update targeting logic and release-risk policy instead of sharing main’s contracts and a maintainer-owned readiness source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the shared channel resolver for beta installs:** `src/cli/update-cli/review.ts:216-221`
  - body: This derives the review channel from config/defaults only. A package install already running a beta `VERSION` with no or stale stable config should resolve to beta per the shared channel helpers, otherwise the command reviews the wrong target version and release notes.
  - confidence: 0.86
- **[P2] Scope downgrade reporting to package installs:** `src/cli/update-cli/review.ts:260-261`
  - body: `isDowngrade` is computed for git installs from the npm target too. A git checkout whose local `VERSION` is ahead of registry tags can be reported as a downgrade even though `openclaw update` only applies downgrade handling in the package path.
  - confidence: 0.88
- **[P3] Preserve zero-behind git freshness:** `src/cli/update-cli/review.ts:249-254`
  - body: This collapses `behind: 0` to `null`, so a fetched current git checkout is indistinguishable from unknown freshness. If the registry lookup fails after git proved the checkout is current, the review reports an unknown state instead of preserving the git result.
  - confidence: 0.8
