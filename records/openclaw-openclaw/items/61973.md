## Summary

Keep open: current main does not have an `openclaw update review` subcommand, and this PR is the right implementation vehicle, but the current head still has update-parity bugs and a required changelog gap before it should merge.

## What This Changes

The PR adds a read-only `openclaw update review` CLI subcommand with npm/GitHub release-note risk heuristics, JSON output, tests, CLI registration, and update documentation.

## Best Possible Solution

Keep the new command, but make its target, downgrade, git freshness, and fallback semantics share or closely mirror the existing update command model before merging.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence mock reproduction: a git install with local `VERSION` ahead of the npm target triggers the false downgrade path, and a git install with `behind: 0` plus unavailable registry data loses the current-git signal.

## Solution Assessment

No, not as currently patched. The feature direction is maintainable, but the safest solution is to reuse or mirror the existing update channel/downgrade state model and add focused regression coverage plus the changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope downgrade reporting to package updates:** `src/cli/update-cli/review.ts:261`
  - body: `isDowngrade` is computed from the npm target for every install kind, so git checkouts whose local `VERSION` is ahead of registry tags are reported as downgrades even though `openclaw update` only applies downgrade handling in the package update path and suppresses beta fallback downgrades. Gate this on the same package-mode target logic so git users see git freshness guidance instead.
  - confidence: 0.88
- **[P3] Preserve zero-behind git status:** `src/cli/update-cli/review.ts:249-254`
  - body: `gitBehind` collapses a fetched `behind: 0` result to `null`, and `checkUnavailable` later treats that as no git signal. If the registry lookup fails but git fetch proved the checkout is current, the command reports an unknown review-required state instead of the current git status.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/cli/update-cli.ts:160-161`
  - body: This adds a user-facing CLI command and documentation, but the branch does not update `CHANGELOG.md`. Add a single-line entry under the active `### Changes` section before merge.
  - confidence: 0.96
