## Summary

Keep open: current main lacks the proposed command, but the PR still has correctness blockers in update channel resolution, git/package downgrade semantics, and git freshness handling, plus missing real behavior proof and an unresolved release-risk policy question.

## What This Changes

The PR adds a read-only `openclaw update review` CLI subcommand with JSON output, release-note risk heuristics, tests, and update documentation.

## Best Possible Solution

If maintainers want this command, reuse the shared update channel, target, downgrade, and git freshness contracts, then source risk guidance from maintainer-owned release readiness metadata.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR head against current-main update helpers shows the beta-installed-version, git downgrade, and zero-behind cases. The feature itself still lacks real `openclaw update review` and `openclaw update review --json` proof.

## Solution Assessment

No, not as currently patched. The direction is plausible, but the implementation duplicates update targeting and release-risk policy instead of sharing main’s update contracts and a maintainer-owned readiness source.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the shared channel resolver for beta installs:** `src/cli/update-cli/review.ts:216-221`
  - body: This derives the review channel from config/defaults only. A package install already running a beta `VERSION` with no or stale stable config should resolve to beta per the shared channel helpers, otherwise the command reviews the wrong target version and release notes.
  - confidence: 0.86
- **[P2] Scope downgrade checks to package installs:** `src/cli/update-cli/review.ts:260-261`
  - body: `isDowngrade` is computed for git installs from the npm target too. A git checkout whose local `VERSION` is ahead of registry tags can be reported as a downgrade even though the updater’s downgrade handling is package-scoped.
  - confidence: 0.88
- **[P3] Preserve zero-behind git freshness:** `src/cli/update-cli/review.ts:249-254`
  - body: This collapses `behind: 0` to `null`, so a fetched current git checkout is indistinguishable from unknown freshness. If registry lookup fails after git proved the checkout is current, the review reports an unknown state instead of preserving the git result.
  - confidence: 0.8
