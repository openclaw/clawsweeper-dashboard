## Summary

Keep open. Current main still lacks `openclaw update review`, and the PR is a useful implementation vehicle, but the current patch still has a package/git downgrade parity bug, loses a successful zero-behind git freshness signal, and lacks the required changelog entry.

## What This Changes

The PR adds a read-only `openclaw update review` CLI subcommand with JSON output, npm/GitHub release-note risk heuristics, tests, CLI registration, and update documentation.

## Best Possible Solution

Keep this PR or a narrow replacement as the implementation path, but make the review command share or closely mirror the existing update target, downgrade, and git freshness semantics before merging.

## Reproduction Assessment

Yes. A unit mock with `installKind: "git"`, local `VERSION` ahead of the npm target, and git status available reproduces the false downgrade path; a separate mock with `behind: 0`, `fetchOk: true`, and registry unavailable reproduces the lost current-git signal.

## Solution Assessment

No, not as currently patched. The command direction is maintainable, but the safest fix is to reuse or mirror the existing `openclaw update` channel/downgrade state model instead of independently classifying git installs from npm semver alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope downgrade reporting to package updates:** `src/cli/update-cli/review.ts:261`
  - body: `isDowngrade` is computed from the npm target for every install kind, so git checkouts whose local `VERSION` is ahead of registry tags are reported as downgrades even though `openclaw update` only applies downgrade handling in the package update path. Gate this on the package-mode target logic so git users see git freshness guidance instead.
  - confidence: 0.88
- **[P3] Preserve zero-behind git status:** `src/cli/update-cli/review.ts:249-254`
  - body: `gitBehind` collapses a fetched `behind: 0` result to `null`, and `checkUnavailable` later treats that as no git signal. If the npm lookup fails but git fetch proved the checkout is current, the command reports an unknown review-required state instead of up to date.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/cli/update-cli.ts:160-161`
  - body: This adds a user-facing CLI command and documentation, but the branch does not update `CHANGELOG.md`. Add a single-line entry under the active `### Changes` section before merge.
  - confidence: 0.96
