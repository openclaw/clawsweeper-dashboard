## Summary

Keep open: current main still has the source-level mismatch where sandboxed non-rw runs can switch to the sandbox workspace while reusing a host-built skills snapshot prompt unchanged. The focused path is still to review or supersede open #38789 rather than close this issue.

## What This Changes

The issue reports unreadable host-style skill locations being exposed to Docker non-main sandbox sessions after skills have been mirrored into the sandbox workspace.

## Best Possible Solution

Land #38789 or an equivalent root fix that rebuilds or remaps available-skills prompts from sandbox-readable skill entries for non-rw sandbox attempt and compaction runs while preserving skill filters, exposure metadata, and normal snapshot reuse outside isolated sandboxes.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a host-built skillsSnapshot with resolvedSkills and a non-rw sandbox run: current main switches to the sandbox workspace but skips live sandbox skill loading and returns the stored snapshot prompt unchanged.

## Solution Assessment

Yes. The narrow maintainable solution is to refresh or remap the snapshot for isolated sandbox workspaces in the runtime prompt path; prompt-only hardening such as #74161 cannot repair a stale host path already stored in the snapshot.
