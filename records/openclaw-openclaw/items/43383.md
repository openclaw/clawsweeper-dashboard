## Summary

Keep open. Current main still has the reported mismatch: non-rw sandbox setup mirrors eligible skills into the sandbox workspace, but attempt and compaction runs can still reuse a host-built skills snapshot prompt unchanged, so unreadable host-style skill locations may still be exposed. The focused path is to review or supersede the open #38789 fix candidate, not to close this issue now.

## What This Changes

The issue requests that sandboxed non-main sessions expose SKILL.md locations that are readable from the sandbox workspace instead of host-only managed skill paths.

## Best Possible Solution

Land #38789 or an equivalent root fix that rebuilds or remaps available-skills prompts from sandbox-readable skill entries whenever a non-rw sandbox attempt or compaction run receives a host-built snapshot, while preserving skill filters, exposure metadata, environment overrides, and normal snapshot reuse outside isolated sandboxes.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a host-built skillsSnapshot with resolvedSkills and a non-rw sandbox run: current main switches to the sandbox workspace but skips live sandbox skill loading and returns the stored snapshot prompt unchanged.

## Solution Assessment

Yes. The narrow maintainable fix is to refresh or remap the snapshot for isolated sandbox workspaces in the runtime/prompt path; #74161-style prompt wording is useful defense-in-depth but cannot repair a stale host path already stored in the snapshot.
