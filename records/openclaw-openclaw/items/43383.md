## Summary

Keep open: current main still has the source-level mismatch where a host-built skills snapshot can be reused after a non-rw sandbox run switches to the sandbox workspace, so the prompt can keep host-only skill paths even though the sandbox copy exists. The provided GitHub context shows open PR #38789 as the focused root-fix candidate, while #74161 is prompt hardening that does not remap an already stale snapshot path.

## What This Changes

The issue asks for sandboxed non-main sessions to expose skill paths that are readable inside the sandbox instead of host-only `~/.openclaw/skills/...` paths.

## Best Possible Solution

Review and land #38789 or an equivalent root fix that rebuilds or remaps available-skills prompts from sandbox-readable skill entries for non-rw sandbox attempt and compaction runs, while preserving filters, exposure metadata, and normal snapshot reuse outside isolated sandboxes.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a host-built `skillsSnapshot` with `resolvedSkills` plus a non-rw sandbox run: current main switches to the sandbox workspace, skips live sandbox skill loading, and returns the stored snapshot prompt unchanged.

## Solution Assessment

Yes. The narrow maintainable fix is to refresh or remap the snapshot for isolated sandbox workspaces in the runtime prompt path; prompt-only hardening such as #74161 cannot repair a stale host path already stored in the snapshot.
