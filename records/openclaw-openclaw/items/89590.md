## Summary

The central bug is valid and still needs a fix, but this branch does not consistently supply its new canonical transport field from the primary embedded and Codex side-question tool factories. Split provider/transport runs can therefore still use a policy provider as the approval channel or lose account/thread context, so the PR is not merge-ready.

## What This Changes

The PR forwards originating turn routing fields through embedded plugin approval contexts and adds gateway-level routing and fail-closed regression coverage.

## Best Possible Solution

Build one canonical turn-origin binding from the run's transport fields, pass it unchanged through every embedded and Codex tool-construction path, and prove split-provider routing plus a real transport approval round trip.

## Reproduction Assessment

Yes. Current main omits origin bindings from embedded plugin approval requests, and the PR's split-provider example provides a concrete source path where the proposed fix still supplies the wrong channel.

## Solution Assessment

No, not yet. Adding an explicit transport field is the right direction, but the maintainable fix must create and use one canonical binding at every tool-construction entry point rather than letting callers reconstruct it inconsistently.
