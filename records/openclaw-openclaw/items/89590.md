## Summary

Keep open: the underlying approval-routing bug is still present on current main, and the PR's turn-source propagation is the right direction, but the added no-route auto-allow weakens an existing approval/security boundary before it is safe to merge.

## What This Changes

The PR forwards embedded-runner turn-source routing fields into plugin before_tool_call approval requests and adds an unavailable-route allow behavior for skill workshop lifecycle approvals.

## Best Possible Solution

Merge the turn-source propagation with no-route approvals still fail-closed unless an explicit, documented maintainer-approved policy says otherwise, then require redacted live transport proof for the chat approval path.

Yes from source: current main omits turn-source fields from embedded plugin approval requests while the gateway requires those fields to keep a no-client approval pending for the originating chat route. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No as submitted. The turn-source propagation is the narrow maintainable fix, but the no-route auto-allow path is a separate policy/API change that should be removed or explicitly approved and documented.
