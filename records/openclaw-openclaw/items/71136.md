## Summary

Keep open. Current main still returns higher override/configured token values before lower model or discovery limits in central context paths, and core Anthropic maxTokens still accepts an oversized request before checking the model cap. This remains a narrow runtime correctness issue, not stale cleanup.

## What This Changes

The issue requests conservative contextTokens and maxTokens resolution by taking the minimum positive available value from agent config, provider model config, model discovery/runtime metadata, and defaults.

## Best Possible Solution

Add a shared positive-finite token-limit resolver with source metadata, then use it consistently in context-window guards, session persistence, embedded runner model normalization, config/discovery merge behavior, and provider request maxTokens while preserving provider-qualified lookup and documented long-context exceptions.

## Reproduction Assessment

Yes. Configure or request contextTokens/maxTokens above a lower configured, discovered, or runtime model cap; static inspection shows current resolver paths can still return the higher value first.

## Solution Assessment

Yes. The minimum-of-available-limits direction is the narrow maintainable fix, provided it is centralized in a shared resolver rather than scattered Math.min calls.
