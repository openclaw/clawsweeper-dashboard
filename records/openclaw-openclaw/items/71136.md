## Summary

Keep open. Current main still has precedence paths where an agent/configured/requested token value can win before lower model or discovery limits are considered, and the issue is not protected or paired with a merged fix in the provided context.

## What This Changes

Use the minimum positive available value from agent config, provider model config, discovered/runtime model metadata, and defaults when resolving effective contextTokens and maxTokens.

## Best Possible Solution

Add a shared positive-finite token-limit resolver with source metadata and use it consistently in context-window resolution, session persistence, embedded-runner model normalization, provider config/discovery merge behavior, and provider request maxTokens clamping while preserving special long-context opt-ins.

## Reproduction Assessment

Yes. Configure agents.defaults.contextTokens or a per-agent/requested maxTokens above a lower configured or discovered model cap; static inspection of current main shows the higher value can still be returned or forwarded first.

## Solution Assessment

Yes. The requested minimum-of-available-limits behavior is the narrow maintainable fix, provided it is centralized and covered by regression tests rather than added as isolated clamps.
