## Summary

Keep open. Current main still resolves several contextTokens and maxTokens paths by precedence rather than by the minimum available model/config/discovery cap, and both tests and docs still encode higher configured values winning over lower discovered/catalog limits.

## What This Changes

The issue requests resolving effective contextTokens and maxTokens as the minimum of agent config, provider model config, model discovery, and defaults so oversized overrides cannot exceed model limits.

## Best Possible Solution

Add one shared positive-finite token-limit resolver with source metadata and use it consistently in context-window guards, session persistence, embedded-runner model normalization, model config/discovery merge behavior, and provider request maxTokens handling, with tests and docs updated for the new minimum-of-known-limits contract.

## Reproduction Assessment

Yes. A static reproduction is enough: configure or request contextTokens/maxTokens above a lower configured, discovered, or runtime model cap, and current main still returns the higher override/configured value in central resolver paths before considering the lower model limit.

## Solution Assessment

Yes. The issue's minimum-of-available-limits direction is the narrow maintainable fix, provided it is centralized and preserves provider-qualified lookup semantics and documented long-context opt-ins.
