## Summary

Keep open. Current main still has precedence-based token resolution in several runtime paths, so a higher agent/config/requested value can bypass a lower model or discovery cap; the issue remains a narrow correctness fix rather than cleanup-close material.

## What This Changes

Use the minimum positive available value from agent config, provider model config, model discovery/runtime metadata, and defaults when resolving `contextTokens` and `maxTokens`.

## Best Possible Solution

Introduce one shared positive-finite token-limit candidate resolver, use it for context window guards, session persistence/status, embedded-runner model normalization, model merge/runtime metadata, and provider `maxTokens` request shaping, then update docs/tests to define the conservative cap contract.

## Reproduction Assessment

Yes. Configure `agents.defaults.contextTokens` or a per-agent/requested `maxTokens` above a lower configured/discovered/runtime model limit; static inspection of current main shows those higher values can still return before the lower cap is considered.

## Solution Assessment

Yes. The requested minimum-of-available-limits behavior is the right direction, with the maintainable implementation being a centralized resolver plus provider-qualified lookup safeguards instead of scattered `Math.min` calls.
