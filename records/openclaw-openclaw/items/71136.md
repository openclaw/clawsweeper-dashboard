## Summary

Keep open. Current main still uses precedence in the central token-limit paths: explicit agent/context overrides and configured provider values can win before lower discovered or runtime limits, and Anthropic maxTokens still accepts a higher requested value before considering the model cap. The issue remains a valid, focused runtime correctness repair; related PR #39099 was closed unmerged and only addressed an adjacent compaction-cap path.

## What This Changes

The issue requests conservative contextTokens and maxTokens resolution by taking the minimum positive available value from agent config, provider model config, model discovery/runtime metadata, and defaults.

## Best Possible Solution

Add a shared positive-finite token-limit resolver that collects named candidates from agent defaults/per-agent settings, provider/model config, runtime/discovery metadata, and fallback defaults, then uses the conservative minimum consistently for context-window guards, session persistence, embedded-runner model metadata, model merge/runtime normalization, and provider request maxTokens while preserving provider-qualified lookup behavior and documented context1m exceptions.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure or request contextTokens/maxTokens above a lower configured/discovered/runtime model limit and the current resolver paths can still return the higher override before the lower cap.

## Solution Assessment

Yes. The requested minimum-of-available-limits direction is the narrow maintainable fix, but it should be implemented through a shared resolver with named candidates rather than scattered Math.min calls.
