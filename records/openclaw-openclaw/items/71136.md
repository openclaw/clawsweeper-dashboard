## Summary

Keep open: the contextTokens over-reporting example is fixed on current main and in v2026.6.8 by https://github.com/openclaw/openclaw/pull/90889, but this issue also asks for maxTokens to use the same conservative minimum policy. Current docs and code still treat explicit maxTokens/configured contextTokens as compatibility-sensitive overrides in some paths, so the remaining behavior needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests resolving effective contextTokens and maxTokens as the minimum of agent config, provider model config, model discovery, and defaults instead of strict priority.

## Best Possible Solution

Keep the shipped context-window cap, then have maintainers decide whether maxTokens should become a conservative ceiling across provider/model discovery or remain an explicit override with narrower transport-level safeguards.

## Reproduction Assessment

Yes for the original contextTokens failure path from source and merged regression tests; current main no longer reproduces that concrete example. The remaining maxTokens request is source-verifiable as absent, but current docs make it a policy decision rather than a fully established current-contract bug.

## Solution Assessment

No, not as a blanket min-of-all change yet. The shared resolver fix was the right solution for contextTokens, but maxTokens needs a maintainer decision because current docs and merge code preserve explicit config over discovery in some cases.
