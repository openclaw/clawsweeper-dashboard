## Summary

Keep open. Current main and the latest release still contain override-first `contextTokens` paths, while related work only covers part of the context-window behavior and does not settle `maxTokens` or the broader config/discovery precedence decision.

## What This Changes

The issue requests resolving effective `contextTokens` and `maxTokens` as the minimum of agent settings, provider/model config, discovery metadata, and defaults so oversized agent settings cannot exceed known model limits.

## Best Possible Solution

Unify context and output-token budget resolution behind one owner-boundary helper that treats agent settings as ceilings when model limits are known, with focused tests for session persistence, status/metadata, provider config, discovery cache, and `maxTokens` request shaping.

## Reproduction Assessment

Yes from source inspection: the current resolver and session persistence paths still accept an agent context-token override before lower model/config/discovery limits. I did not run a live provider overflow scenario in this read-only review.

## Solution Assessment

Partly: using the minimum is the right cap behavior for known context windows, but applying it across all `maxTokens` and config/discovery precedence paths needs a maintainer compatibility decision before an automated fix.
