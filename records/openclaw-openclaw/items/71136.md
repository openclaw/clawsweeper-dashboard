## Summary

Keep open. Current main and the latest stable tag still have source-reproducible override-first `contextTokens` and `maxTokens` paths, and the related open PR only covers part of the context-window problem rather than the broader output-token and catalog-precedence behavior.

## What This Changes

The issue requests resolving effective `contextTokens` and `maxTokens` as the smallest finite known value across agent config, provider model config, discovery, and defaults.

## Best Possible Solution

Centralize finite positive token-limit resolution with source metadata, apply it consistently to runtime context budgeting, session/status reporting, and transport output caps, then update docs/tests once maintainers choose the catalog precedence contract.

## Reproduction Assessment

Yes, source-level. Configure an agent/default `contextTokens` above a known selected-model window, or request `maxTokens` above the model cap; current main still has resolver, session, status, and transport paths that choose the larger explicit value before the known model limit.

## Solution Assessment

Partly. Taking the minimum of known positive runtime caps is the right safety direction, but the final fix should be centralized and needs maintainer agreement on catalog merge and output-token precedence before it lands.
