## Best Possible Solution

Land one byte-size invariant: reject unsafe rounded string results and unsafe numeric byte-size config values, with focused parser/config tests and no compatibility shim for precision-corrupted values.

## Solution Assessment

No. The `parseByteSize` helper change is the right layer for string inputs, but the best fix should also align the documented numeric byte-size config helper with the same safe-integer invariant.
