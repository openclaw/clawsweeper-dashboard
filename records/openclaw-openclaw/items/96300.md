## Summary

Keep open: this PR fixes the unsafe string `parseByteSize` path, but it is still too narrow because documented numeric byte-size config can still pass through the finite/floor helper path.

## What This Changes

Adds a `Number.isSafeInteger` guard to `parseByteSize` and parser tests for unsafe byte-size strings.

## Best Possible Solution

Land one byte-size invariant across string parsing and numeric config helpers: reject unsafe integer results for both forms and add focused parser/config tests.

## Reproduction Assessment

Yes. Current main's parser source rounds and only checks `Number.isFinite`, and a read-only Node math probe confirms the PR examples are finite but not safe integers.

## Solution Assessment

No. The string parser helper change is the right layer for string inputs, but the best fix should also align numeric byte-size config handling with the same safe-integer invariant.
