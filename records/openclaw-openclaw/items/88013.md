## What This Changes

This PR changes `runSecretsApply` to stage satellite credential-file writes before renaming them and committing config, with a regression test for mid-commit write faults.

## Best Possible Solution

Keep the stage-then-commit ordering, but make staging preserve the same private-store/secret-file directory hardening and then provide after-fix proof where the injected fault actually reaches the new stage path.

## Reproduction Assessment

Yes from source inspection, though I did not run a local repro: current main commits config before looping over satellite writes, and rollback is best-effort. A write fault after config and before a later auth-store can therefore leave divergent credential state.

## Solution Assessment

No, not as submitted. The stage-before-config direction is maintainable, but the implementation should keep the existing private credential write contract and the proof must hit the new staging path.

- **[P1] Preserve private-store writes when staging secrets:** `src/secrets/apply.ts:775`
  - body: The current satellite writes go through `writeTextFileAtomic`, which sends 0o600 credential files through `privateFileStoreSync`; fs-safe private writes re-create/chmod credential directories to 0o700 and guard the directory before renaming. Please keep the two-phase ordering but stage through the same private file-store/secret-file contract.
