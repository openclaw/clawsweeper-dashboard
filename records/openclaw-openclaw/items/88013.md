## Summary

Keep this PR open: it fixes the reported satellite-write fault window, but the new ordering still renames credential satellites before the final config write, so a config write fault can leave the opposite half-migration if rollback is defeated. This is still a security-sensitive credential write-path change that needs owner review.

## What This Changes

The PR stages `secrets apply` satellite credential writes through `writeTextFileAtomic`, renames them before committing config, and adds a regression test for satellite write faults.

## Best Possible Solution

Make the write-fault boundary cover both satellite and config writes before any live file advances, or add a secops-approved proof that final config failures cannot leave persisted divergence.

## Reproduction Assessment

Yes by source and supplied live output for the original bug: current main commits config before satellite credential files and uses best-effort rollback, so a later satellite write fault can leave divergent state. The remaining final-config-write concern is also source-reproducible from the PR ordering.

## Solution Assessment

No. Staging satellite writes through the existing private-store path is the right narrow direction, but the final config write must also be staged, preflighted, or otherwise proven all-or-nothing before this is the best fix.
