## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for a test-only hook, while the strict Telegram config schema, exported types, UI hints, and Telegram docs expose no production key; closed #1811 and broader open #39343 do not supersede this narrower setting.

## What This Changes

Expose a user-facing Telegram media-group flush timeout config key so slow-arriving album items can be buffered longer than the current 500 ms default.

## Best Possible Solution

Add optional `channels.telegram.mediaGroupFlushMs` at top level and per account with the existing 500 ms default, wire it into the media-group timers, and update config contract docs/UI hints/changelog/tests without changing global inbound debounce behavior.

## Reproduction Assessment

Yes, from source inspection. A same-`media_group_id` update arriving after the 500 ms timer is processed as a new entry because the timer deletes and flushes the buffer; I did not run a live Telegram latency test.

## Solution Assessment

Yes. A Telegram-owned config key is the narrowest maintainable solution because the existing inbound debouncer intentionally excludes default-lane media-only entries, while gateway-level batching would broaden the behavior beyond this report.
