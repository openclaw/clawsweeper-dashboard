## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms outside the test-only timing hook, and the strict Telegram config/types/docs do not expose a production key; the linked broader gateway batching issue does not supersede this narrower Telegram config request.

## What This Changes

The issue requests a user-facing Telegram config key to tune the media-group flush window instead of relying on the hardcoded 500 ms production timeout.

## Best Possible Solution

Add a maintainer-approved optional Telegram media-group flush setting with the existing 500 ms default, schema/types/UI hints/docs/changelog, and focused media-group regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: after the 500 ms timer fires, the buffer entry is deleted and processed, so a later update with the same media_group_id is handled as a separate group; I did not run a live Telegram latency test.

## Solution Assessment

Yes on the direction: a Telegram-owned config key is narrower than changing the global inbound debouncer, which intentionally skips default-lane media-only entries. The exact key name and location should be decided before implementation.
