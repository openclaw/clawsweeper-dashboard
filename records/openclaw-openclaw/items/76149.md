## Summary

Keep open. Current main still has a fixed Telegram media-group flush window with only a test-only override, while Telegram config types, schema, UI hints, and docs expose neighboring media/network timing keys but not this production setting.

## What This Changes

The issue requests a user-facing Telegram media-group flush timeout configuration so slower pure-media Telegram albums are not split into partial agent turns.

## Best Possible Solution

Add an optional Telegram-owned `mediaGroupFlushMs` integer config with the current 500 ms default, wire it through the existing account config path into media-group buffering, and update schema/types/UI hints/docs/changelog plus focused tests.

## Reproduction Assessment

Yes. A code-level reproduction is clear: deliver two Telegram updates with the same `media_group_id` more than 500 ms apart, and current main flushes and deletes the first buffer before the later item arrives; no production config can extend that window today.

## Solution Assessment

Yes. A Telegram-specific config key is the narrowest maintainable fix because the shared inbound debounce deliberately excludes default-lane pure-media entries and changing it would affect broader message behavior.
