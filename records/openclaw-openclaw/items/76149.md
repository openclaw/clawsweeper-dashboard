## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for a test-only hook, and the strict Telegram config schema/types/docs do not expose a production key; the linked #1811 fix history and the broader #39343 gateway batching request do not supersede this narrower Telegram config request.

## What This Changes

Expose an optional Telegram media-group flush timeout config key so slow album updates can be buffered longer than the hardcoded 500 ms default.

## Best Possible Solution

Add an optional per-account Telegram `mediaGroupFlushMs` setting with the existing 500 ms default, wire it into the media-group buffer, and update schema/types/UI hints/docs/changelog plus focused Telegram regression coverage.

## Reproduction Assessment

Yes. Source inspection shows that a same-`media_group_id` item arriving after the 500 ms timer is processed in a new buffer because the original buffer is deleted during flush; I did not run a live Telegram latency test.

## Solution Assessment

Yes. A Telegram-owned account config key is the narrowest maintainable solution because the existing inbound debouncer intentionally excludes default-lane media-only entries, while changing that global predicate would affect broader message handling.
