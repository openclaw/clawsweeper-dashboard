## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500ms except for a test-only hook, and the public Telegram config schema/types/docs do not expose a production setting for pure media albums.

## What This Changes

Expose a user-facing Telegram media-group flush timeout so slower Telegram album deliveries can be buffered longer than the current 500ms default.

## Best Possible Solution

Add an optional Telegram account/channel `mediaGroupFlushMs` integer config with the current 500ms default, wire it into media-group buffering, and update schema/types/UI hints/docs/changelog plus focused Telegram tests.

## Reproduction Assessment

Yes. A code-level reproduction is clear: after 500ms the media-group buffer is deleted and processed, so a same-id Telegram album item arriving later becomes a second processing turn; no production config can change that path today.

## Solution Assessment

Yes. A Telegram-owned config key is the narrowest maintainable fix because global inbound debounce intentionally excludes default-lane media-only entries and would change broader message behavior.
