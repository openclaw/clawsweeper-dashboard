## Summary

Keep open. Current main still uses a fixed 500 ms Telegram media-group flush window except for a test-only timing hook, and the public Telegram config schema, type surface, UI hints, and docs do not expose a production setting; the broader batching issue and earlier closed report do not supersede this narrower config request.

## What This Changes

Expose a user-facing Telegram media-group flush timeout config key so deployments can increase the current 500 ms album buffer window.

## Best Possible Solution

Add an optional Telegram-owned mediaGroupFlushMs-style integer config with the current 500 ms default, wire it into media-group buffering, and update schema/types/UI hints/docs/changelog plus focused Telegram regression coverage.

## Reproduction Assessment

Yes, from source inspection. A same-media_group_id album item that arrives after the current 500 ms timer will be handled as a new group because the first timer deletes and flushes the existing buffer.

## Solution Assessment

Yes, subject to maintainer choice on the key name/location. A Telegram-owned config key is narrower than changing the shared inbound debouncer, which currently and intentionally skips default-lane media-only entries.
