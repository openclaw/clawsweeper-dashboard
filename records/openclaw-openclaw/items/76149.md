## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for a test-only timing hook, and the public Telegram config schema, types, UI hints, and docs do not expose a production setting; the older media-group report and broader batching issue do not supersede this narrower config request.

## What This Changes

Expose Telegram media-group flush timing as an optional user-facing configuration key while preserving the current 500 ms default.

## Best Possible Solution

Add an optional Telegram account/channel mediaGroupFlushMs integer config with the current 500 ms default, wire it into media-group buffering, and update schema, types, UI hints, docs, changelog, and focused Telegram regression tests.

## Reproduction Assessment

Yes, from source inspection. A same-media_group_id item that arrives after the 500 ms timer fires is processed as a new entry because the first timer deletes and flushes the buffer; I did not run a live Telegram latency test.

## Solution Assessment

Yes, subject to maintainer naming/location choice. A Telegram-owned config key is narrower than changing the shared inbound debouncer, which intentionally excludes default-lane media-only messages.
