## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for a test-only hook, and the public Telegram config schema/types/docs do not expose a production setting; the linked prior report and broader batching issue do not supersede this narrower config request.

## What This Changes

Expose Telegram media-group flush timing as a user-facing configuration key instead of only a test timing override.

## Best Possible Solution

Add an optional Telegram account/channel `mediaGroupFlushMs` integer config with the current 500 ms default, wire it into media-group buffering, and update schema/types/UI hints/docs/changelog plus focused Telegram regression tests.

## Reproduction Assessment

Yes, from source inspection. A same-`media_group_id` Telegram album item that arrives after the current 500 ms timer is processed as a new entry because the first timer deletes and flushes the buffer; I did not run a live Telegram network test.

## Solution Assessment

Yes. A Telegram-owned config key is the narrowest maintainable fix because the existing inbound debouncer intentionally excludes default-lane media-only entries and changing that global behavior would affect broader message handling.
