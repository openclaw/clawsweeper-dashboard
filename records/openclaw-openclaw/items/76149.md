## Summary

Keep open. Current main still uses a 500 ms Telegram media-group flush window unless the test-only `testTimings.mediaGroupFlushMs` hook is passed, and the strict Telegram config schema/types/docs do not expose a production setting.

## What This Changes

Expose a user-facing Telegram media-group flush timeout config key so slower album updates can be buffered longer than the 500 ms default.

## Best Possible Solution

Add an optional per-account `channels.telegram.mediaGroupFlushMs` setting with the existing 500 ms default, wire it into the Telegram media-group buffer, and cover schema/docs/changelog/tests.

## Reproduction Assessment

Yes. Source inspection shows a same-`media_group_id` update arriving after the 500 ms flush starts a new buffer because the original entry is deleted; I did not run a live Telegram latency test.

## Solution Assessment

Yes. A Telegram-owned account config key is the narrowest maintainable fix; changing the global inbound debounce predicate would broaden behavior beyond Telegram albums.
