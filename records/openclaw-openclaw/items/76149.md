## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for the test-only timing hook, and the public Telegram config schema/types/docs still do not expose a production setting for slower pure-media albums; the linked #1811 context supports the same root cause, while #39343 is broader and does not supersede this narrow Telegram follow-up.

## What This Changes

The issue requests a user-facing Telegram media-group flush timeout configuration so slower pure-media Telegram albums are not split into partial agent turns.

## Best Possible Solution

Add an optional Telegram-owned `mediaGroupFlushMs` integer config with the current 500 ms default, wire it into media-group buffering, and update schema/types/UI hints/docs/changelog plus focused Telegram tests.

## Reproduction Assessment

Yes. A code-level reproduction is clear: deliver two Telegram updates with the same `media_group_id` more than 500 ms apart, and current main flushes and deletes the first buffer before the later item arrives with no production config path to extend the window.

## Solution Assessment

Yes. A Telegram-specific config key is the narrowest maintainable fix because the shared inbound debounce deliberately excludes default-lane pure-media entries and changing it would alter broader message behavior.
