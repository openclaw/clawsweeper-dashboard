## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for a test-only hook, while the strict Telegram config schema/types/docs expose no production key; #1811 is closed without covering this config path and #39343 is a broader gateway batching request, not a replacement for this narrower Telegram setting.

## What This Changes

Expose a Telegram media-group flush timeout config key so pure-media albums can use a longer buffer than the current 500 ms default.

## Best Possible Solution

Add an optional Telegram account-level `mediaGroupFlushMs` setting with the existing 500 ms default, wire it into the media-group buffer, and update schema/types/UI hints/docs/changelog plus focused Telegram regression coverage.

## Reproduction Assessment

Yes, from source inspection. A same-`media_group_id` album item arriving after the 500 ms timer is processed as a new entry because the first timer deletes and flushes the buffer; I did not run a live Telegram latency test.

## Solution Assessment

Yes. A Telegram-owned config key is the narrowest maintainable solution because the existing inbound debouncer intentionally excludes default-lane media-only entries, and changing that global predicate would affect broader message handling.
