## Summary

Keep open. Current main still hardcodes Telegram media-group flushing to 500 ms except for a test-only timing hook, and the strict Telegram config schema, exported types, UI hints, and docs do not expose a production config key; the closed #1811 and broader open #39343 do not supersede this narrower Telegram config request.

## What This Changes

Expose a user-facing Telegram media-group flush timeout setting so slower album deliveries can raise the current 500 ms buffer window.

## Best Possible Solution

Add an optional per-account Telegram mediaGroupFlushMs setting with the existing 500 ms default, update schema/types/UI/docs/changelog/tests, and leave broader cross-channel batching to #39343.

## Reproduction Assessment

Yes. Source inspection shows that after the media-group timer fires, the handler deletes and processes the buffer, so a later same-media_group_id update will be handled as a new group; I did not run a live Telegram latency test.

## Solution Assessment

Yes. A Telegram-owned account config key is the narrowest maintainable solution because the existing inbound debouncer intentionally excludes default-lane media-only entries, while changing that global predicate would alter broader message handling.
