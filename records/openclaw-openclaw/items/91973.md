## Summary

The updated head still leaves the core provider lifecycle defect: after the first continuing response, the call ID remains in a set that blocks every later `response.create`, so the new periodic progress updates are queued rather than spoken when scheduled. The PR also lacks required after-fix real-call proof and adds compatibility-sensitive public surfaces that need maintainer confirmation.

## What This Changes

The PR adds configurable working speech and periodic consult progress for realtime voice calls, provider-authoritative barge-in handling, and new voice-call config, tool, Plugin SDK, provider capability, and gateway metadata surfaces.

## Best Possible Solution

Serialize interim and final tool-result responses explicitly so each scheduled progress update can create one response after the prior response completes, keep the final result ordered last, and retain only public config/API additions that have a confirmed external contract need.

## Reproduction Assessment

Yes at source level: submit two or more `willContinue: true` results for one call through the real OpenAI bridge and observe that only the first immediately emits `response.create`; a real voice-call reproduction has not been supplied.

## Solution Assessment

No, not yet: periodic talkback is a reasonable feature, but the current boolean-pending lifecycle cannot represent multiple queued progress responses, and the broad public contract expansion is not proven necessary.
