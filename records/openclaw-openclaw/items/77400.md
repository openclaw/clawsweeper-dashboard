## Summary

Keep open: current main does not have a doctor retention guard for active auth profiles tied to provider API-key config or fallback usage, and the reported outage path is plausible from the current repair and auth-selection code.

## What This Changes

Request that `openclaw doctor --fix` preserve `auth.profiles` entries that still have configured provider credentials or are used by model fallback chains, and warn before any removal.

## Best Possible Solution

Teach doctor repair to classify auth-profile metadata as active when its provider has usable configured credentials or appears in default/per-agent fallback chains, preserve those entries, and emit a visible warning for any skipped or removed profile.

## Reproduction Assessment

Yes, from source inspection rather than a live run: the current repair flow can remove schema-unknown auth-profile data without checking provider API keys or fallback references. A high-confidence live reproduction still needs a temporary config/auth-store fixture test.

## Solution Assessment

No, the current schema-only doctor cleanup is not the best way to solve this class of auth issue. The narrower maintainable fix is to add an active-auth-profile retention check and regression coverage around provider API keys plus fallback chains.
