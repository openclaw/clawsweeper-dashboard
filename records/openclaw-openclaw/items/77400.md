## Summary

Keep open: current main still applies schema-only unknown-key cleanup in doctor repair without checking whether an auth profile is active through provider credentials or fallback usage, and the linked closing PR is open but unmerged.

## What This Changes

The issue asks `doctor --fix` to preserve `auth.profiles` entries for providers with configured API keys or fallback-chain references, and to warn before removing active auth profile data.

## Best Possible Solution

Review and land #77482, or an equivalent guarded doctor repair that preserves active auth-profile credential data and emits a visible warning when preservation occurs.

## Reproduction Assessment

Yes, from source inspection: current main runs schema-only unknown-key stripping during `doctor --fix`, and strict `auth.profiles` metadata makes legacy credential fields removable. A live fixture is still needed to confirm the exact whole-entry deletion wording from the incident report.

## Solution Assessment

Yes, the narrow maintainable direction is to preserve unknown auth-profile data only when the profile is active through configured provider credentials, auth order, or model fallback references, and warn during doctor repair. Closing before #77482 merges would be premature.
