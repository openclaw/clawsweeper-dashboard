## Summary

Keep open: current main still applies schema-only doctor cleanup without an active-auth-profile guard for provider API-key config or fallback usage, so the reported data-loss/outage path remains plausible and actionable.

## What This Changes

The issue asks `openclaw doctor --fix` to preserve `auth.profiles` entries that are backed by configured provider credentials or referenced by model fallback chains.

## Best Possible Solution

Add a doctor repair guard and regression coverage that preserves active auth-profile metadata when the provider has usable configured credentials or the profile participates in default/per-agent fallback selection, while warning clearly about any removed inactive profile data.

## Reproduction Assessment

Yes, from source inspection: the current repair path can strip schema-unknown auth-profile data without checking provider API-key config or fallback-chain references. A high-confidence live fixture still needs to be added in a repair PR.

## Solution Assessment

Yes, the narrow maintainable solution is to add active-auth-profile retention before unknown-key cleanup, plus a regression test for provider API keys and fallback chains. Relaxing the full schema or preserving all unknown auth data would be broader and riskier.
