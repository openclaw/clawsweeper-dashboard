## Summary

Keep open: current main still runs doctor unknown-key cleanup without checking whether an auth profile is backed by configured credentials or referenced by fallback routing, and the incident follow-up says the problem remained present after upgrading to 2026.5.3-1.

## What This Changes

Request that `doctor --fix` preserve `auth.profiles` entries tied to configured provider API keys or active fallback chains, and warn before removing any active auth profile metadata.

## Best Possible Solution

Add an active-auth-profile retention check to the doctor repair path, preserve profiles backed by provider credentials or fallback references, emit visible warnings for skipped/removal decisions, and cover the incident shape with a regression test plus changelog entry.

## Reproduction Assessment

Yes from source inspection, but not from a live run: the current doctor repair path can clean auth-profile-shaped config using only schema unknown-key data and has no guard for provider API keys or fallback references. A high-confidence live reproduction still needs a temporary config/auth-store fixture for the reported `openai:default` deletion shape.

## Solution Assessment

No, the current schema-only cleanup is not the best way to solve this issue. The narrower maintainable fix is to classify active auth profiles before cleanup and warn instead of removing metadata needed by fallback auth.
