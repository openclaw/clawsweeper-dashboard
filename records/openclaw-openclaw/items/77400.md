## Summary

Keep open: current main still runs schema-only unknown-key cleanup during doctor repair without checking whether an auth profile is active through provider API-key config or model fallback usage, and an unmerged closing PR already targets this issue.

## What This Changes

The issue asks `doctor --fix` to preserve `auth.profiles` entries when the provider has configured credentials or is referenced by model fallback configuration.

## Best Possible Solution

Review and land a narrow doctor repair change that preserves active auth profile metadata based on provider credentials and fallback/auth-order references, with focused regression coverage for the reported OpenAI fallback case.

## Reproduction Assessment

Yes. Source inspection on current main shows the `doctor --fix` repair path can strip unknown `auth.profiles.*` data without checking provider API keys or fallback references; I did not run a live fixture because this sweep is read-only.

## Solution Assessment

Yes. The narrow maintainable fix is to add an active-auth-profile retention guard in the doctor cleanup path plus regression tests, rather than adding a config option or changing runtime fallback policy.
