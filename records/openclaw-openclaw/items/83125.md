## Summary

Keep open: the SIGTERM fallback is the right shape, but the SIGUSR1 catch can leave the in-flight restart token uncleared, and the supplied proof is a standalone Node simulation rather than patched gateway run-loop behavior.

## What This Changes

This PR wraps the gateway SIGTERM and SIGUSR1 async signal handlers in try/catch, logging lifecycle-loader failures and using a SIGTERM stop fallback while skipping failed SIGUSR1 restarts.

## Best Possible Solution

Keep the SIGTERM fallback, clear or roll back SIGUSR1 restart state on import failure without widening authorization, and add focused run-loop coverage plus redacted real gateway proof.

## Reproduction Assessment

Yes, source-reproducible: current main awaits the lifecycle loader inside fire-and-forget SIGTERM/SIGUSR1 handlers before request(...) runs, so a rejected loader can escape before shutdown or restart is submitted. I did not run an artifact-producing live repro because this review is read-only.

## Solution Assessment

No, not yet: the SIGTERM catch is maintainable, but the SIGUSR1 failure path should also consume or roll back the in-flight restart cycle. The safer fix is a narrow signal-handler fallback that preserves existing authorization semantics and proves the actual run-loop path.

## Review Findings

Overall correctness: patch is incorrect
