## Summary

Keep open: this is a focused gateway bug fix, but the SIGUSR1 failure path still appears to leave the restart cycle unconsumed, and the supplied proof is a standalone Node simulation rather than the patched OpenClaw gateway behavior.

## What This Changes

The PR wraps the gateway SIGTERM and SIGUSR1 async signal handlers in try/catch, logging lifecycle loader failures, falling back to a plain SIGTERM stop, and skipping failed SIGUSR1 restarts.

## Best Possible Solution

Keep the SIGTERM fallback, but update the SIGUSR1 failure path to consume or roll back the restart cycle before skipping, then prove the actual run-loop behavior with focused coverage or redacted real gateway output.

## Reproduction Assessment

Yes, source-reproducible: current main awaits the lifecycle loader inside fire-and-forget SIGTERM/SIGUSR1 handlers with no catch, so a rejection can escape before request runs. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

No, not yet: the SIGTERM try/catch is the right shape, but SIGUSR1 should also clear the restart-cycle state instead of only logging. The safer fix is a narrow signal-handler failure path plus focused regression proof against the actual run-loop contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Clear the SIGUSR1 restart token on failure:** `src/cli/gateway-cli/run-loop.ts:751`

Consume or roll back the SIGUSR1 cycle before skipping this failed restart.
