## Summary

Keep open: current main still has a source-level path where message-tool-only source replies can be delivered to the channel without an assistant text entry being appended to the durable transcript, so the reported restart amnesia is not proved fixed.

## What This Changes

The issue requests that Feishu/tool-call replies visible in the channel also be persisted to the session `.jsonl` transcript so restarts preserve the assistant response.

## Best Possible Solution

Add an owner-approved durable representation for successful message-tool replies, likely using existing `messagingToolSentTexts`/target evidence and ACP delivery state, so session history and memory can recover visible replies without corrupting replay order or duplicating channel sends.

## Reproduction Assessment

Yes, from source inspection: message-tool-only source delivery can produce no block payload text, `dispatch-acp` persists only the block accumulator, and the transcript writer skips the assistant append when that text is empty. I did not run a live Feishu gateway reproduction in this read-only review.

## Solution Assessment

No, current main is not yet the best solution. Removing the steer early return alone would not cover the maintainer-diagnosed message-tool-only path; the safer fix is to persist or index successful message-tool output through an explicit transcript/history contract.
