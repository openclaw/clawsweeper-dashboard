## Summary

This still looks like a real current-main transcript persistence bug, not a close candidate: the source path can deliver visible channel output while persisting only block-accumulated text, so message-tool-only or routed turns can leave the assistant reply absent from the session JSONL.

## What This Changes

The issue reports Feishu-visible assistant replies after tool-call turns not being persisted to the session `.jsonl`, causing context loss after gateway restart.

## Best Possible Solution

Persist the actual visible assistant output for message-tool-only/routed turns exactly once, with regression coverage for Feishu/group-channel delivery and a guard against duplicate delivery mirrors.

## Reproduction Assessment

Yes, at source level: current main can route visible ACP output while `persistAcpDispatchTranscript` receives only `getAccumulatedBlockText()`, which ignores non-block delivery. I did not run a live Feishu session in this read-only review.

## Solution Assessment

No, the proposed fix of removing the steer early return is not clearly the best repair; the safer target is the transcript persistence/mirroring boundary so delivered text is durable without changing active-run queue semantics.
