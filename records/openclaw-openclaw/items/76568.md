## Summary

Keep open: current main still has a source-level path where message-tool-visible replies can be delivered to the channel while transcript persistence sees no assistant text to append. The report is actionable, but the proposed steer early-return removal is not the safest proven fix.

## What This Changes

The issue reports Feishu-visible assistant replies after tool-call turns not being persisted to session JSONL transcripts in OpenClaw 2026.5.2.

## Best Possible Solution

Persist committed message-tool visible output into the session transcript for message_tool_only turns while preserving the tool-only delivery policy and adding regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a message_tool_only channel turn that sends visible text through the message tool but emits no block text leaves finalText empty for transcript persistence, so no assistant message is appended.

## Solution Assessment

Is this the best way to solve the issue? No for the proposed early-return removal; the safer fix is to persist committed message-tool outputs, or an equivalent transcript-visible assistant turn, without changing active-run steering semantics.
