## Summary

Do not close this issue: current `main` still has a source-level path where a message-tool-only ACP turn can deliver visible channel text without passing that text into the assistant transcript persistence path.

## What This Changes

The issue reports Feishu-visible assistant replies after tool-call turns not being appended to session `.jsonl` transcripts in OpenClaw 2026.5.2.

## Best Possible Solution

Persist the actual user-visible message-tool send text for ACP/message-tool-only turns into the session transcript without changing group/channel tool-only visible-reply policy.

## Reproduction Assessment

Yes from source inspection, but not from a live Feishu run: an ACP/message-tool-only turn with no block text reaches transcript persistence with empty final text, so this path can omit the assistant reply from the transcript.

## Solution Assessment

Yes for the narrow direction: fix the ACP delivery/transcript handoff to include visible message-tool sends. Removing the steer early return would be less targeted and does not address the block-only transcript accumulator shown in current source.

Current `tryDispatchAcpReply` persists only `delivery.getAccumulatedBlockText()`, while visible message-tool sends can be delivered as tool-path output and do not populate the block accumulator, so the session `.jsonl` can miss the assistant-visible reply after restart.
