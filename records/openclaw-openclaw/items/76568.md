## Summary

Keep open: current main still has the transcript gap described in the issue. The maintainer comment's mechanism matches source inspection: tool-only source replies can be delivered through `message(action=send)`, but ACP transcript persistence still records only accumulated block text, so a successful message-tool-only reply can leave no assistant text in the session JSONL.

## What This Changes

The issue asks OpenClaw to persist channel-delivered assistant replies to the session `.jsonl` transcript even when the visible reply is sent through the message tool after tool-call turns.

## Best Possible Solution

Record successful source-visible message-tool sends as assistant transcript content for the same turn, exactly once, while preserving the tool-only delivery policy and existing silent/no-reply behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: a group/channel ACP turn using the default `message_tool_only` policy can deliver via the message tool while `persistAcpDispatchTranscript` receives an empty block accumulator, causing no assistant transcript append. I did not run a live Feishu gateway in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow fix is to persist successful message-tool delivery text into the assistant turn transcript, not to revert the group visible-reply default or add another config option.
