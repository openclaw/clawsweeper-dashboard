## Summary

Keep open: the renderer-layer fix is the right direction, but this patch keys only top-level `id`/`messageId` while current history and `session.message` paths put transcript ids in `__openclaw.id`, so the reported shipped shape can still double-render; proof is also unit-test-only.

## What This Changes

The PR adds Control UI renderer logic and tests to collapse adjacent user/assistant chat items sharing a top-level `id` or `messageId`, preferring the unlabeled native copy over a relay-labeled copy.

## Best Possible Solution

Keep the renderer-layer dedupe, but key it from the canonical transcript id sources including `__openclaw.id`, add regression coverage for that shape, and provide real Control UI dmScope=main proof.

## Reproduction Assessment

Yes, source-reproducible: https://github.com/openclaw/openclaw/issues/93044 describes a dmScope=main duplicate, and current code shows chat final/session-message/history paths can feed the renderer while current main collapses only display signatures. I did not run a live Telegram plus Control UI reproduction in this read-only review.

## Solution Assessment

No. The renderer boundary is the right place, but this implementation should include canonical `__openclaw.id` transcript metadata rather than only top-level `id`/`messageId`.
