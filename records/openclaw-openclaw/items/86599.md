## Summary

Keep open. Current main has meaningful responsiveness improvements and a Windows main probe looked much better, but the report is recent, externally reproduced on Windows, and there is not enough current-main source or live proof to close the full local-provider model-call starvation symptom.

## What This Changes

The issue reports Windows local model calls blocking the Gateway event loop, making trivial prompts and unrelated Gateway RPCs take tens of seconds to minutes.

## Best Possible Solution

Keep this open until a maintainer runs a current-main Windows local-provider comparison that captures direct backend timing, OpenClaw infer/gateway timing, concurrent health/sessions/models RPC timings, and liveness logs, then either fix the remaining hot path or close with that proof.

Yes for the reported Windows beta/stable packages: the reporter and a second Windows repro provide concrete commands, logs, timings, and screenshots. No for current main: the only main check shows major improvement but explicitly does not prove the full UI/chat.history path.

## Solution Assessment

Unclear. Current main has good partial mitigations for model catalog/session-list responsiveness, but the best fix cannot be selected until the remaining current-main Windows local-provider hot path is isolated.
