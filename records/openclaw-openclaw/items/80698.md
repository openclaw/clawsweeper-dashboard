## Summary

Keep open. Current main still has only global config env support and host exec starts from the Gateway process environment, while the closest open implementation PR is unmerged and has security/behavior proof blockers. This needs maintainer/security design for a new credential-sensitive config contract, not cleanup closure.

## What This Changes

Request `agents.list[].env.vars` so each configured agent can receive only global env defaults plus its own env overrides during Bash/tool execution.

## Best Possible Solution

Keep this open and either rework https://github.com/openclaw/openclaw/pull/62011 or replace it with a documented, schema-backed per-agent env resolver that has explicit security semantics, redaction, and regression proof across exec and any supported credential consumers.

## Reproduction Assessment

Yes for the current limitation by source inspection: current main has no strict `agents.list[].env` schema/type field, and non-sandbox exec builds from the Gateway process env. This is not a broken existing contract reproduction; it is proof that the requested feature is still absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact config shape and inheritance rules need maintainer/security agreement; the closest open PR is not yet the best solution because it has scoped-env leakage, default-scope behavior, and proof blockers.
