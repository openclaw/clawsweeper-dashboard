## Summary

Keep open. Current main still has only root/global env configuration and host exec builds from the gateway process environment; the related open implementation PR is not merged and is currently human/security-gated, so this is not cleanup-close material.

## What This Changes

The issue requests `agents.list[].env.vars` so each agent's tool and process environment can inherit global env defaults without receiving other agents' credentials.

## Best Possible Solution

Define one schema-backed per-agent env contract with explicit merge, filter, redaction, SecretRef, service metadata, and host-exec baseline semantics, then land it through the active implementation lane or a replacement with focused isolation tests.

## Reproduction Assessment

Yes for the missing capability by source inspection: current main rejects an `agents.list[].env` block through the strict agent schema and host exec starts from gateway `process.env`. This is a feature/config gap rather than a broken existing documented behavior.

## Solution Assessment

Unclear. `agents.list[].env.vars` is plausible, but the best fix needs maintainer decisions on precedence, allowed host baseline values, SecretRef/redaction, service metadata, and how this composes with MCP/tool-specific credential surfaces.
