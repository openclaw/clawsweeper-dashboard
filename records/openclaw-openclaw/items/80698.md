## Summary

Keep open. Current main still lacks a per-agent env config surface, and host Bash execution still builds from the gateway process environment; the closest implementation is the open related PR, but it is conflicting, changes-requested, and missing real behavior proof.

## What This Changes

Requested support for per-agent `agents.list[].env.vars` so each agent’s tool/process environment receives only global env plus its own scoped variables.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filter, redaction, SecretRef, and service-metadata semantics, then cover it with regression tests proving cross-agent credential isolation.

## Reproduction Assessment

Yes. Source inspection on current main shows `agents.list[]` does not accept `env`, root `env.vars` is global, and host exec starts from gateway `process.env`; I did not run a live two-agent credential setup.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible direction, but the safest implementation needs maintainer agreement on precedence, filtering, redaction, SecretRef support, and composition with MCP/tool credential surfaces.
