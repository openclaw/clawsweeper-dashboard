## Summary

Keep open. Current main still has only a global `env.vars` config surface, and gateway-host Bash execution builds from the gateway process environment, so the reported multi-agent credential-isolation gap remains real but needs maintainer design because it adds a public, credential-sensitive config contract.

## What This Changes

The issue requests per-agent environment variable configuration at `agents.list[].env.vars` so each agent receives only global env vars plus its own scoped overrides.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge/filter/redaction semantics, then wire it through runtime env construction and service metadata with regression tests proving two agents cannot see each other's scoped credentials.

## Reproduction Assessment

Yes. Source inspection shows `agents.list[]` does not accept an `env` block, root `env.vars` is global, and host exec currently starts from the gateway `process.env`.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible shape, but the safest solution needs maintainer agreement on precedence, filtering, SecretRef/redaction, and how it composes with MCP/tool-specific credential surfaces.
