## Summary

Keep open. Current main still has only a root-level `env.vars` surface, and host Bash execution builds from the gateway process environment, so the reported credential-isolation gap remains real but needs maintainer design because it adds a public, security-sensitive config contract.

## What This Changes

Request per-agent `agents.list[].env.vars` config so each agent receives only global env defaults plus its own scoped env vars during tool and process execution.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filter, redaction, SecretRef, reload, and service metadata semantics, then wire it through runtime env construction with two-agent regression coverage.

## Reproduction Assessment

Yes for the reported current behavior. Source inspection on current main shows `agents.list[]` does not accept `env`, root `env.vars` is collected globally, and gateway-host Bash starts from `process.env`.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible direction, but the safest solution needs maintainer agreement on precedence, filtering, SecretRef/redaction, service metadata, and composition with MCP/tool-specific credential surfaces.
