## Summary

Keep open. Current main still lacks an `agents.list[].env.vars` config surface, root `env.vars` is applied service-wide, and host exec builds from the gateway process environment, so the requested credential-isolation feature remains valid and needs maintainer security/product design rather than cleanup closure.

## What This Changes

Add per-agent `agents.list[].env.vars` configuration so each agent's tool processes receive only global env plus that agent's scoped env.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filter, redaction, and service metadata semantics, then wire it through runtime env construction and docs/tests that prove two agents cannot see each other's scoped credentials.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: `agents.list[]` rejects `env`, root `env.vars` is global, and host exec starts from `process.env`; this is a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible shape, but maintainers should first decide precedence, allowed/blocked keys, SecretRef/redaction behavior, and interaction with MCP/tool-specific credential surfaces.
