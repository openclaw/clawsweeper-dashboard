## Label Justifications

- `P2`: This is a valid multi-agent credential-isolation improvement with limited but security-sensitive blast radius and no current core outage.
- `impact:security`: The issue is about preventing one agent's credentials from leaking into another agent's tool process environment.
- `impact:auth-provider`: The reported symptom involves GitHub CLI/token identity and credential selection across agents.

## Summary

Keep open. Current main still lacks an agent-scoped env config surface, and host Bash execution inherits the gateway process environment, so the reported credential-isolation gap remains valid but needs maintainer design for a public, security-sensitive config contract.

## What This Changes

Request to add `agents.list[].env.vars` so each agent receives only global env vars plus its own overrides, preventing other agents' credentials from appearing in tool processes.

## Best Possible Solution

Define one schema-backed per-agent env contract with explicit merge, filtering, SecretRef/redaction, and service metadata semantics, then wire it through Bash/tool env construction with regression tests proving agents cannot see each other's scoped credentials.

## Reproduction Assessment

Yes for the current-source feature gap. Source inspection shows the agent schema has no `env` field, root `env.vars` is global, and host exec starts from the gateway `process.env`, which is enough to verify the missing isolation path on current main.

## Solution Assessment

Unclear. `agents.list[].env.vars` is plausible, but maintainers should first settle precedence, filtering, SecretRef/redaction, and how the general env layer composes with MCP and tool-specific credential settings.
