## Summary

Keep open. Current `main` still lacks a per-agent `env.vars` config surface, and host Bash execution still derives non-sandbox env from the gateway process, so the credential-isolation gap remains real and needs maintainer security/product design rather than conservative cleanup.

## What This Changes

Requested per-agent `agents.list[].env.vars` configuration so each agent’s tool processes receive only global env vars plus that agent’s overrides, without other agents’ credentials.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filtering, redaction, SecretRef, and diagnostics semantics, then wire it through runtime env construction and service metadata with multi-agent isolation tests.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `agents.list[]` does not accept an `env` block, root `env.vars` is global, and non-sandbox host exec starts from the gateway `process.env`.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible shape, but maintainers should first decide precedence, filtering, SecretRef/redaction, and composition with MCP/tool-specific credential surfaces.
