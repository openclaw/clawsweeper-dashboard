- `impact:security`: The issue concerns preventing credentials intended for one agent from leaking into another agent's tool process.

## Summary

Keep open. Current main still has only a root-scoped `env.vars` surface, agent entries remain strict with no `env` field, and host exec still starts from the gateway process environment; the reported multi-agent credential-isolation gap is valid but needs maintainer design because it adds a credential-sensitive public config contract.

## What This Changes

Request support for `agents.list[].env.vars` so each agent's Bash/tool process sees only global env plus that agent's scoped variables.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filtering, SecretRef, redaction, and service metadata semantics, then wire it through runtime env construction with regression tests for two-agent isolation.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction of the gap: `agents.list[]` does not accept `env`, root `env.vars` is global, and host exec starts from `process.env`; I did not run the reporter's two-agent `gh` setup live.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible direction, but maintainers should first choose the public precedence, filtering, SecretRef, redaction, and service-env metadata contract.
