## Summary

Keep open. Current main still has only a global `env.vars` surface, strict per-agent config entries do not accept `env`, and host Bash execution still starts from the gateway process environment, so the reported credential-isolation gap remains valid and needs maintainer security/product design rather than cleanup closure.

## What This Changes

Request to add `agents.list[].env.vars` so each agent's Bash/tool processes receive global env vars merged only with that agent's own scoped vars.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filtering, redaction, and reload semantics, then wire it through runtime env construction and service metadata with two-agent isolation tests.

## Reproduction Assessment

Yes, by source inspection: current main rejects `agents.list[].env` through the strict agent schema, collects only root `env.vars`, and builds host exec env from gateway `process.env`. I did not run a live two-agent credential setup because the source path already shows the missing contract.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible public shape, but maintainers still need to choose precedence, filtering, SecretRef/redaction, and how this composes with MCP and tool-specific credential scopes.
