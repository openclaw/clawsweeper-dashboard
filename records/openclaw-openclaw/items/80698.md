## Summary

Keep open. Current `main` still has only process-wide `env.vars` plus sanitized ambient `process.env` for host/CLI execution, so the reported per-agent credential isolation request remains valid and is not covered by the narrower open related issues.

## What This Changes

Request per-agent `agents.list[].env.vars` config so each agent gets global env defaults plus its own env overrides without inheriting other agents' credentials.

## Best Possible Solution

Design and implement a schema-backed per-agent env contract that feeds host exec, sandbox exec, CLI backend runs, service-managed env metadata, redaction, docs, and tests without weakening existing host-env sanitization.

## Reproduction Assessment

Yes for the current source-level limitation: current `main` has no accepted `agents.list[].env` schema/type field, and host/CLI execution starts from sanitized ambient `process.env`. I did not run a live two-agent service reproduction.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible shape, but maintainers should choose the public precedence, reload, service-env, sandbox, and shared-HOME semantics before implementation.
