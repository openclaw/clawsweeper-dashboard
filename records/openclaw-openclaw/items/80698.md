## Summary

Keep open. Current main still has root-scoped env configuration and host exec inherits the gateway process environment, so the requested per-agent credential isolation remains a real feature gap that needs maintainer/security design.

## What This Changes

Add an `agents.list[].env.vars` config surface so each agent's Bash/tool process sees only root `env.vars` plus that agent's scoped variables.

## Best Possible Solution

Define a schema-backed per-agent env contract with explicit merge, filter, redaction, reload, and service-metadata semantics, then wire it through host exec/runtime env construction with two-agent regression coverage.

## Reproduction Assessment

Yes for the current-main feature gap: source inspection shows `agents.list[]` has no `env` field, root `env.vars` is global, and non-sandbox host exec starts from gateway `process.env`. I did not run the reporter's live two-agent credential setup.

## Solution Assessment

Unclear. `agents.list[].env.vars` is a plausible shape, but maintainers need to settle precedence, filtering, SecretRef/redaction, service metadata, and how this composes with MCP and tool-specific credential surfaces before implementation.
