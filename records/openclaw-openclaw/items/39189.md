## Summary

Keep open: this PR targets plausible bugs, but the branch still has two blocking correctness problems and lacks after-fix real behavior proof. Current main already has a safer route-scoped dedupe implementation, while the Anthropic billing classification gap still needs a narrower fix.

## What This Changes

The PR adds Anthropic usage-limit billing matchers, reorders billing checks ahead of rate-limit handling, and globally enables final-reply text dedupe after message-tool sends.

## Best Possible Solution

Land a narrowed replacement: add a precise Anthropic specified-API-usage-limits billing matcher with regression coverage, and extend current route- or alias-aware dedupe only with redacted real messaging proof.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main's matcher and classifier order, while the multi-identifier duplicate-reply scenario still lacks live iMessage or alias proof.

## Solution Assessment

No. The PR touches the right surfaces, but the billing matcher is too broad and the dedupe change replaces route-scoped suppression with global same-text suppression; a narrower replacement is safer.

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56-57`
  - body: The new `/reached your.*usage limits?/i` pattern can also match transient usage-window text such as "You have reached your monthly usage limit".   - confidence: 0.9

  - body: This enables messaging-tool dedupe whenever any sent text exists, even when the tool target differs from the originating route. Because `filterMessagingToolDuplicates` compares text only, a same-text send to another channel or user can incorrectly drop this conversation's final reply.
  - confidence: 0.94
