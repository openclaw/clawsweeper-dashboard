## Summary

Keep open: the PR addresses a real billing-classification gap, but the branch is not safe to merge as-is because it broadens usage-limit billing matching and replaces route-scoped final-reply dedupe with global same-text suppression, with no real after-fix messaging proof.

## What This Changes

The branch adds Anthropic usage-limit billing matchers, reorders billing checks before rate-limit handling, and enables text dedupe after any messaging-tool send.

## Best Possible Solution

Land a narrowed replacement: add a precise Anthropic specified-API-usage-limits billing matcher with regression coverage, and extend dedupe only through route- or alias-aware matching backed by redacted real messaging proof.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main's matcher/classifier order, while the multi-identifier duplicate-reply scenario still lacks live iMessage or alias proof.

## Solution Assessment

No. The PR touches the right surfaces, but the billing matcher is too broad and the dedupe change should preserve route-scoped or alias-aware suppression instead of global same-text suppression.
