## Summary

Keep open: the PR targets a real failover-classification gap, but it is not safe to merge as-is because the billing matcher is too broad, the reply-dedupe change can suppress cross-target same-text replies, the reply half is already superseded on current main, and the external real-behavior proof gate is still missing.

## What This Changes

The branch adds Anthropic usage-limit billing matchers/tests, moves billing classification ahead of transient rate-limit handling, and broadens message-tool final-reply dedupe to any sent text.

## Best Possible Solution

Split the work: add a narrow Anthropic specified-usage-limits billing matcher with regression coverage, and keep the route-scoped reply-dedupe design already on main instead of global same-text suppression.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main because the generic `usage limit` rate-limit matcher runs before the late billing fallback, but the multi-handle duplicate-reply case still lacks real iMessage/channel-alias proof.

## Solution Assessment

No. The narrow maintainable fix is a precise billing matcher plus the route-scoped dedupe behavior already present on main, not the PR's broad billing precedence and global same-text dedupe.

  - confidence: 0.9
