## Summary

Keep open: the billing bug is still source-reproducible on current main, but this PR is not ready as-is because it broadens billing classification and weakens route-scoped final-reply dedupe; it also still lacks after-fix real behavior proof.

## What This Changes

The PR adds Anthropic usage-limit billing matchers, reorders failover billing checks ahead of rate-limit handling, and enables message-tool final-reply text dedupe whenever sent text exists.

## Best Possible Solution

Land a narrowed replacement: a precise Anthropic specified-API-usage-limits billing matcher with regression coverage, plus route- or alias-aware final-reply dedupe backed by redacted real messaging proof.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible on current main from the matcher and ordering path, while the multi-identifier duplicate-reply scenario still lacks live iMessage or alias proof.

## Solution Assessment

No. The PR touches the right surfaces, but the billing matcher is too broad and the dedupe change replaces route-scoped suppression with global same-text suppression; a narrower replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56-57`
  - body: The new `/reached your.*usage limits?/i` pattern can also match transient usage-window text such as "You have reached your monthly usage limit". Since billing now runs before rate-limit handling, those messages can be classified as `billing` instead of staying in the cooldown/retry path covered by current tests.
  - confidence: 0.9
- **[P1] Keep final-reply dedupe route-scoped:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: This enables messaging-tool dedupe whenever any sent text exists, even when the tool target differs from the originating route. Because `filterMessagingToolDuplicates` compares text only, a same-text send to another channel or user can incorrectly drop this conversation's final reply.
  - confidence: 0.94
