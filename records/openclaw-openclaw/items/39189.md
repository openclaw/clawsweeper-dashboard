## Summary

Keep this PR open, but do not merge it as-is: the Anthropic billing bug still needs a narrow fix, while the reply-dedupe half is partly superseded by current main and this branch introduces two blocking regressions plus lacks real behavior proof.

## What This Changes

The PR adds Anthropic usage-limit billing matchers/tests, moves billing checks ahead of rate-limit handling, and enables messaging-tool text dedupe whenever sent text exists.

## Best Possible Solution

Split the work: keep the route-scoped reply dedupe design on main, and land a narrow Anthropic specified-API-usage-limits billing matcher with regression coverage that preserves periodic usage-window rate limits.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main through the generic `usage limit` rate-limit matcher; the multi-handle iMessage duplicate-reply scenario is not high-confidence reproduced because no live/channel-alias proof is present.

## Solution Assessment

No. The PR's current approach is not the best fix because it broadens billing classification before rate-limit handling and globally dedupes same text across targets; the maintainable path is a narrow billing matcher plus route-aware reply dedupe.
