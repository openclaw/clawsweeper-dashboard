## Summary

Keep open. The PR addresses a real Anthropic billing-classification gap, but the proposed patch is not mergeable as-is: one billing matcher is broad enough to shadow transient usage limits, and the reply-dedupe change can suppress legitimate same-text replies sent to a different target. Current main has a safer route-based dedupe seam, but it does not fully implement the PR's exact billing wording or the multi-handle identity behavior.

## What This Changes

The PR adds Anthropic usage-limit billing matchers, moves billing checks ahead of rate-limit handling, and enables messaging-tool text dedupe whenever any sent text exists even when the sent target differs from the originating target.

## Best Possible Solution

Split or rework the PR: add a narrow Anthropic billing matcher tied to the exact `specified API usage limits` and `regain access` wording while preserving transient usage-limit tests, and solve duplicate replies through identity-aware same-route matching in the channel/plugin suppression seam rather than content-only cross-target dedupe.

## Reproduction Assessment

Partly yes. The billing half has a high-confidence source-level reproduction through `classifyFailoverReason` on the exact Anthropic wording, while the multi-handle duplicate reply still needs a channel-specific or live alias reproduction to confirm the intended same-person route match.

## Solution Assessment

No. The billing direction is valid but too broad as written, and the reply-dedupe change bypasses the route guard; the safer solution is a narrow billing matcher plus channel-aware target alias matching that keeps cross-target same-text replies intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:55`
  - body: The new `/reached your.*usage limits?/i` pattern also matches transient quota messages such as `You have reached your monthly usage limit`. Because billing is checked before the periodic/rate-limit paths in this PR, those temporary limits can be misclassified as billing and trigger billing disable/backoff instead of normal cooldown handling.
  - confidence: 0.9
- **[P1] Keep text dedupe behind the target match:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: Adding `messagingToolSentTexts.length > 0` enables text dedupe even when the message-tool target did not match the originating target. In that cross-target case, `filterMessagingToolDuplicates` removes by text alone, so sending the same text elsewhere can incorrectly drop the current conversation's final reply.
  - confidence: 0.94
