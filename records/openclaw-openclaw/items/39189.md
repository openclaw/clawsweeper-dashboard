## Summary

Keep open: this PR targets plausible bugs, but the current branch should not merge because it can misclassify periodic usage limits as billing, can suppress cross-target same-text replies, and still lacks real behavior proof.

## What This Changes

The PR adds Anthropic usage-limit billing matchers/order changes and changes messaging-tool final-reply dedupe to run whenever sent text exists, with related tests.

## Best Possible Solution

Split the work: add a precise Anthropic specified-API-usage-limits billing matcher that preserves periodic rate-limit behavior, and solve multi-handle dedupe with route- or alias-aware target equivalence plus redacted runtime proof.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main, but the multi-handle duplicate-reply case still lacks live iMessage or channel-alias proof.

## Solution Assessment

No. The branch touches the right surfaces, but the billing matcher is too broad and the reply change replaces route-scoped dedupe with global same-text suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56`
  - body: The new `/reached your.*usage limits?/i` pattern can also match periodic usage-window messages such as "You have reached your monthly usage limit". Because billing now runs before rate-limit handling, those transient limits can be classified as billing instead of normal cooldown/failover.
  - confidence: 0.9
- **[P1] Keep text dedupe route-scoped:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: This enables text dedupe whenever any messaging-tool text exists, even when the tool target differs from the originating route. Since duplicate filtering compares text only, a same-text send to another channel or user can incorrectly drop this conversation's final reply.
  - confidence: 0.94
