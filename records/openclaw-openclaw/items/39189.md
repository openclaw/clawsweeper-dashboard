## Summary

Keep open: the PR targets real bug surfaces, but the branch is not mergeable as-is because both implementation changes are over-broad and the external PR still lacks required real behavior proof.

## What This Changes

The branch adds Anthropic usage-limit billing matchers, moves billing checks before transient rate-limit handling, and enables final-reply text dedupe after any messaging-tool send.

## Best Possible Solution

Land a narrowed replacement: add a precise Anthropic specified-API-usage-limits billing matcher with regression coverage, and extend dedupe only through route- or alias-aware matching backed by redacted real messaging proof.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main's matcher/classifier order, while the multi-identifier duplicate-reply scenario still lacks live iMessage or alias proof.

## Solution Assessment

No. The PR touches the right surfaces, but the billing matcher is too broad and the dedupe change should preserve route-scoped or alias-aware suppression instead of global same-text suppression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56-57`
  - body: The new `/reached your.*usage limits?/i` pattern can also match transient usage-window messages such as monthly usage-limit text. Because billing is moved earlier, those messages can be classified as `billing` instead of staying in the retry/cooldown path covered by current tests.
  - confidence: 0.9
- **[P1] Keep final-reply dedupe route-scoped:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: This enables messaging-tool dedupe whenever any sent text exists, even when the tool target differs from the originating route. Since duplicate filtering compares text only, a same-text send to another channel or user can incorrectly suppress this conversation's final reply.
  - confidence: 0.94
