## Summary

The PR should stay open because it targets a real failover-classification gap, but the current patch is not mergeable as-is: the billing matcher is too broad and the reply-dedupe half is superseded by route-scoped logic already on `main`.

## What This Changes

The PR adds Anthropic usage-limit billing classification changes, adjusts assistant error formatting/failover precedence, and changes messaging-tool final-reply dedupe logic with unit-test updates.

## Best Possible Solution

Land a narrow Anthropic billing matcher that preserves periodic usage-window rate-limit behavior, and keep reply dedupe route-scoped using the current `main` design.

## Reproduction Assessment

Partly yes. The Anthropic billing misclassification is source-reproducible from the current matcher/order; the multi-handle duplicate-reply claim still lacks live iMessage/channel-alias proof.

## Solution Assessment

No. The PR's approach is broader than needed for billing and replaces route-scoped reply dedupe with global same-text suppression.

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56`
  - body: The new `/reached your.*usage limits?/i` pattern can also catch retryable periodic usage-window messages. Because the PR moves billing ahead of rate-limit handling, those messages can be misclassified as billing instead of normal cooldown/failover.
  - confidence: 0.9

  - confidence: 0.94
