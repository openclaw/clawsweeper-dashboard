## Summary

Keep open: the billing half addresses a real current-main gap, but the PR is not mergeable because it broadens billing matching enough to shadow temporary limits and bypasses the route guard for messaging-tool dedupe.

## What This Changes

The PR adds Anthropic usage-limit billing matchers, moves billing checks earlier in failover/error formatting, and enables final-reply text dedupe whenever the messaging tool sent text.

## Best Possible Solution

Rework or split the PR: land a narrow Anthropic billing classifier for the exact billing-limit wording, and handle multi-handle reply suppression through route or channel-owned alias matching while preserving cross-target same-text delivery.

## Reproduction Assessment

Partly yes. The billing bug is source-reproducible from current-main matcher order, and the PR's dedupe regression is covered by existing cross-target tests; the original multi-handle iMessage-style duplicate still needs channel-specific reproduction proof.

## Solution Assessment

No. The billing direction is valid, but the matcher is too broad and the reply fix bypasses the existing route-suppression seam instead of teaching the channel route matcher about aliases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:55-56`
  - body: The new `/reached your.*usage limits?/i` pattern can match temporary quota text such as `You have reached your monthly usage limit`. Because the PR moves billing ahead of rate-limit handling, those cooldown-style limits can be misclassified as billing and trigger billing disable behavior instead of normal retry/cooldown handling.
  - confidence: 0.9
- **[P1] Keep text dedupe behind the route match:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: Adding `messagingToolSentTexts.length > 0` makes text-only dedupe run even when the message-tool target differs from the originating target. In that case, sending the same text to another channel/user can suppress the current conversation's final reply, which the current route-dedupe tests explicitly preserve.
  - confidence: 0.94
