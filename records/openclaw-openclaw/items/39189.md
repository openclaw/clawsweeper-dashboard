## Summary

Keep open: the billing bug appears real, but the PR is not safe to merge because its matcher is too broad and its reply-dedupe change can suppress valid cross-target replies. Current main has a same-route dedupe seam, so the multi-handle part needs channel-aware identity matching rather than content-only dedupe.

## What This Changes

The PR adds Anthropic usage-limit billing matchers, moves billing classification ahead of rate-limit handling, and enables final-reply text dedupe whenever the messaging tool sent text.

## Best Possible Solution

Split or rework the PR: add a narrow Anthropic billing matcher for the exact `specified API usage limits` plus `regain access` wording, and solve duplicate replies through channel-owned route or alias matching that preserves cross-target same-text delivery.

## Reproduction Assessment

Partly yes. The billing half is source-reproducible through `classifyFailoverReason` because generic `usage limit` text wins before billing; the multi-handle duplicate reply still needs channel-specific alias or live reproduction proof.

## Solution Assessment

No. The billing direction is valid but too broad as written, and the reply-dedupe change bypasses the target guard instead of using the existing channel-owned suppression seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:55-56`
  - body: The new `/reached your.*usage limits?/i` pattern can also match transient quota messages such as `You have reached your monthly usage limit`. Since this PR moves billing ahead of the periodic/rate-limit checks, those temporary limits can be misclassified as billing and trigger billing disable/backoff instead of normal cooldown handling.
  - confidence: 0.9
- **[P1] Keep text dedupe behind the target match:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: Adding `messagingToolSentTexts.length > 0` enables text dedupe even when the message-tool target differs from the originating target. In that cross-target case, `filterMessagingToolDuplicates` removes by text alone, so sending the same text elsewhere can incorrectly drop the current conversation's final reply.
  - confidence: 0.94
