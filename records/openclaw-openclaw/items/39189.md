## Summary

Keep open: the billing bug is still source-reproducible on current main, but this branch is not mergeable as-is because it broadens billing classification and replaces route-scoped reply dedupe with global same-text suppression.

## What This Changes

The PR adds Anthropic usage-limit billing matchers, moves billing checks ahead of rate-limit handling, and enables messaging-tool text dedupe whenever sent text exists.

## Best Possible Solution

Split the work: land a narrow Anthropic specified-usage-limits billing matcher that preserves periodic rate-limit semantics, and keep duplicate-reply handling route/alias-aware rather than global same-text suppression.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible from current main; the multi-handle duplicate-reply scenario still lacks live iMessage/channel-alias proof.

## Solution Assessment

No. The safe direction is a narrower billing matcher plus the route-scoped dedupe design already on main, not the PR's broad matcher and global text dedupe.

Overall correctness: patch is incorrect

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56`
  - body: The new `/reached your.*usage limits?/i` pattern can also match retryable periodic usage-window messages. Because the PR moves billing ahead of rate-limit handling, those messages can be misclassified as billing and push profiles into billing-disable behavior instead of normal cooldown/failover.
  - confidence: 0.9

  - body: This condition enables text dedupe whenever any messaging-tool text exists, even when the tool target differs from the originating route. `filterMessagingToolDuplicates` compares text only, so sending the same text elsewhere can incorrectly drop the current conversation's final reply.
  - confidence: 0.94
