## Summary

Keep open: the PR addresses plausible bug surfaces, but it should not merge as-is because the billing matcher is too broad, the reply-dedupe change can suppress cross-target replies, and current main already has a safer route-scoped reply-dedupe implementation while the Anthropic billing gap still needs a narrower fix.

## What This Changes

The PR adds Anthropic usage-limit billing matchers and billing-first error handling, enables messaging-tool final-reply text dedupe whenever sent text exists, and updates related tests.

## Best Possible Solution

Land a narrow Anthropic specified-usage-limits billing matcher that preserves periodic rate-limit semantics, and keep or extend the current route-scoped dedupe design with live channel-alias proof before changing reply suppression.

## Reproduction Assessment

Partly yes. The billing misclassification is source-reproducible for the PR's sample string on current main, but the multi-handle duplicate-reply scenario still lacks a high-confidence live iMessage or channel-alias reproduction.

## Solution Assessment

No. The PR's broad billing matcher and global same-text dedupe are not the narrowest maintainable fix; the safer direction is precise billing wording plus the route-scoped dedupe approach already present on current main.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Narrow the usage-limit billing matcher:** `src/agents/pi-embedded-helpers/failover-matches.ts:56`
  - body: The new `/reached your.*usage limits?/i` pattern can also match retryable periodic usage-window messages. Because the PR moves billing ahead of rate-limit handling, those messages can be misclassified as billing and put profiles on billing-disable backoff instead of normal cooldown/failover.
  - confidence: 0.9
- **[P1] Keep text dedupe route-scoped:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: This condition enables text dedupe whenever any messaging-tool text exists, even when the tool target differs from the originating route. `filterMessagingToolDuplicates` compares text only, so sending the same text elsewhere can incorrectly drop the current conversation's final reply.
  - confidence: 0.94
