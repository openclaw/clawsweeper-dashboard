## What This Changes

## Best Possible Solution

A mergeable path should get docs-owner scope approval, fix the SecretRef examples to match current structured refs and exec protocol, trim `docs/docs.json` to the intended nav entry, and cross-link the existing security, secrets, sandbox, and multi-agent pages.

## Reproduction Assessment

Yes. From source inspection, copying the PR's `${{ secrets.NAME }}` examples will not parse as SecretRefs, and the `jsonOnly` exec provider output lacks the `protocolVersion: 1` value that current resolver code and tests require.

## Solution Assessment

No. The requested docs direction may be useful, but this PR is not the best merge path until the copyable examples match current behavior and maintainers decide whether this broad guide belongs in the main docs.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Use supported SecretRef syntax in the guide:** `docs/guides/multi-agent-security-ops.md:62`
  - body: The guide tells readers to use `${{ secrets.NAME }}` in config files, but current SecretRef parsing only recognizes structured `{ source, provider, id }` refs or env templates matching `${NAME}`. Copying this syntax leaves a literal string instead of resolving through the configured provider, so the credential lifecycle example does not work.
  - confidence: 0.87
- **[P2] Return protocolVersion from the exec provider example:** `docs/guides/multi-agent-security-ops.md:95`
  - body: With `jsonOnly: true`, exec providers must output an object with `protocolVersion: 1` before the resolver reads `values`. This script emits only `{"values": ...}` in the empty and normal paths, so copied SOPS providers fail with `protocolVersion must be 1`.
  - confidence: 0.93
