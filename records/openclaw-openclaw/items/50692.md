## Summary

Keep open. Current main still does not implement Anthropic native web search, and this PR is the active implementation attempt for the linked feature request, but the branch is stale and not merge-ready because it bypasses the current manifest/public-artifact web-search contract, installs native search without a user-policy gate, drops configured Anthropic search options, and includes unrelated runtime logs.

## What This Changes

The PR adds an Anthropic native web-search provider, a stream wrapper and unit tests, registration in the Anthropic plugin and legacy bundled web-search registry, plus two root runtime log files.

## Best Possible Solution

Implement Anthropic native search in the Anthropic plugin on current main with an explicit `contracts.webSearchProviders` manifest/public-artifact surface, activation gated by `tools.web.search.enabled` and `tools.web.search.provider`, config propagation for tool version/domain filters/max uses/location, managed-tool suppression only while native search is active, response handling tests, docs, and no unrelated artifacts.

## Reproduction Assessment

Yes. The PR defects are reproducible by code inspection: the diff installs the wrapper for every Anthropic stream and passes no search config, while current main's registry and docs show Anthropic native search is not otherwise implemented.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the narrowest maintainable solution because it bypasses current manifest-first discovery, ignores user policy/config, loses configured credentials, and commits unrelated logs; a rebased provider-owned implementation with explicit activation is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate native search before wrapping Anthropic streams:** `extensions/anthropic/index.ts:390-394`
  - body: `wrapStreamFn` is installed for every Anthropic request, and the wrapper injects a server `web_search_*` tool whenever the model looks like direct Anthropic. That ignores `tools.web.search.enabled` and `tools.web.search.provider`, so users who disabled web search or selected another backend can still trigger Anthropic server-side searches and costs.
  - confidence: 0.93
- **[P1] Pass resolved Anthropic search config into the wrapper:** `extensions/anthropic/index.ts:392`
  - body: The wrapper is constructed with only `ctx.streamFn`, so `buildAnthropicWebSearchServerTool()` always sees `searchConfig` as undefined. Runtime settings such as `toolVersion`, domain filters, `maxUses`, and `userLocation` are therefore ignored even though the provider accepts and tests them.
  - confidence: 0.91
- **[P2] Register through the manifest web-search contract:** `src/bundled-web-search-registry.ts:1`
  - body: Adding an eager import to the legacy bundled registry does not match current main, where bundled web-search providers are discovered from `contracts.webSearchProviders` plus public artifacts/runtime entries. Without updating the Anthropic manifest and public artifact path, this provider will not be exposed correctly after rebase.
  - confidence: 0.86
- **[P2] Persist configured Anthropic credentials:** `extensions/anthropic/src/anthropic-web-search-provider.ts:145-148`
  - body: This provider declares a credential path but has no configured credential setter and `setCredentialValue` is a no-op. Configure/onboarding flows that call `setCredentialValue(...)` will accept a pasted key but not save it, leaving the provider unconfigured unless `ANTHROPIC_API_KEY` was already set.
  - confidence: 0.84
- **[P2] Remove committed runtime log artifacts:** `openclaw-2026-03-18.log:1`
  - body: The two root `openclaw-*.log` files are unrelated to the provider implementation and include local runtime metadata such as run IDs, installed paths, timestamps, and debug/error messages. They should not be part of the PR.
  - confidence: 0.89
