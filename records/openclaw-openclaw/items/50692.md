## Summary

Keep open. Native Anthropic web search is still a valid feature gap, but this PR is not mergeable as-is: it is stale/conflicting against current main, bypasses current manifest-first web-search discovery, can inject provider-hosted search outside configured policy, and lacks real behavior proof.

## What This Changes

The PR adds an Anthropic native web-search provider, a payload-injection stream wrapper, unit tests, legacy bundled registry wiring, and root runtime log artifacts.

## Best Possible Solution

Rebuild the feature on current main inside the Anthropic plugin using a manifest-declared webSearchProviders public artifact, explicit activation tied to tools.web.search policy, propagated settings/credentials, server-result handling, docs/tests, and no runtime artifacts.

## Reproduction Assessment

Yes for the blocking PR issues: they are source-reproducible from the diff and current-main code path. The feature itself is not a bug reproduction; current main still lacks Anthropic native web search.

## Solution Assessment

No. The direction is plausible, but this PR is not the best implementation because it bypasses manifest-first discovery, ignores configured web-search policy, drops Anthropic-specific settings, and carries unrelated log artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate native search before wrapping Anthropic streams:** `extensions/anthropic/index.ts:390-394`
  - body: `wrapStreamFn` is installed for every Anthropic request, and the wrapper injects a server `web_search_*` tool whenever the model looks Anthropic. That ignores `tools.web.search.enabled` and `tools.web.search.provider`, so disabled web search or another selected provider can still trigger Anthropic-hosted searches and costs.
  - confidence: 0.94
- **[P1] Pass resolved search config into the wrapper:** `extensions/anthropic/index.ts:392-393`
  - body: The wrapper is constructed with only `ctx.streamFn`, so `buildAnthropicWebSearchServerTool()` always sees `searchConfig` as undefined. Runtime settings such as `toolVersion`, domain filters, `maxUses`, and `userLocation` are silently ignored.
  - confidence: 0.91
- **[P2] Register through the manifest web-search contract:** `src/bundled-web-search-registry.ts:1`
  - body: Adding Anthropic to `src/bundled-web-search-registry.ts` does not match current main, where bundled web-search providers are discovered from `contracts.webSearchProviders` and public artifacts. After rebase, this needs manifest/runtime artifact wiring or the provider will not be exposed correctly.
  - confidence: 0.89
- **[P2] Persist or reuse configured Anthropic credentials:** `extensions/anthropic/src/anthropic-web-search-provider.ts:143-151`
  - body: This provider declares a web-search credential path but `setCredentialValue` is a no-op and `getCredentialValue` only reads `process.env.ANTHROPIC_API_KEY`. Configure/onboarding flows can accept a key without saving or later resolving it unless the env var was already set.
  - confidence: 0.81
- **[P2] Remove committed runtime log artifacts:** `openclaw-2026-03-18.log:1`
  - body: The root `openclaw-*.log` files are unrelated to the provider implementation and include local runtime metadata such as run IDs, installed paths, timestamps, and debug/error messages. They should not be part of the PR.
  - confidence: 0.9
