## Summary

Keep open. This is a plausible feature PR for an open Anthropic native web-search request, but it is not mergeable as written: the branch bypasses current manifest-first discovery, injects search outside configured web-search policy, drops config/domain controls, imports stale packages, commits runtime logs, and lacks real behavior proof.

## What This Changes

The PR adds an Anthropic native web-search provider, a payload-injection stream wrapper, bundled provider wiring, unit tests, and root runtime log files.

## Best Possible Solution

Rebuild the feature on current main inside the Anthropic plugin with manifest-declared webSearchProviders, explicit activation tied to tools.web.search policy, propagated config/credentials, server-result handling, docs/tests, and no runtime artifacts.

## Reproduction Assessment

Yes for the blocking PR issues: source inspection shows the wrapper is installed for all Anthropic streams and constructed without searchConfig, plus current-main package, manifest, and discovery mismatches. The underlying request is a feature gap, not a bug reproduction.

## Solution Assessment

No. The direction is plausible, but this branch needs a current-main rebuild around manifest-first plugin discovery and web-search policy instead of an always-on provider stream wrapper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate native search before wrapping Anthropic streams:** `extensions/anthropic/index.ts:393-395`
  - body: `wrapStreamFn` is installed for every Anthropic request, and the wrapper injects a server `web_search_*` tool whenever the model looks Anthropic. That ignores `tools.web.search.enabled` and `tools.web.search.provider`, so disabled web search or another selected backend can still trigger Anthropic-hosted searches and costs.
  - confidence: 0.94
- **[P1] Pass resolved search config into the wrapper:** `extensions/anthropic/index.ts:393-395`
  - body: The wrapper is constructed with only `ctx.streamFn`, so `buildAnthropicWebSearchServerTool()` always sees `searchConfig` as undefined. Runtime settings such as `toolVersion`, domain filters, `maxUses`, and `userLocation` are silently ignored.
  - confidence: 0.91
- **[P2] Register through the manifest web-search contract:** `src/bundled-web-search-registry.ts:1`
  - body: Current main discovers bundled web-search providers through manifest `webSearchProviders` and public artifacts, while this PR adds Anthropic to a legacy registry file that is absent on current main. After rebase, that does not expose the provider through the current plugin discovery boundary.
  - confidence: 0.89
- **[P2] Update the renamed pi package imports:** `extensions/anthropic/src/anthropic-native-search-wrapper.ts:1-2`
  - body: Current main uses `@earendil-works/pi-agent-core` and `@earendil-works/pi-ai`, but this new file imports the old `@mariozechner` package names. After rebase, module resolution will fail unless the imports are updated.
  - confidence: 0.9
- **[P2] Persist or reuse configured Anthropic credentials:** `extensions/anthropic/src/anthropic-web-search-provider.ts:142-148`
  - body: This provider declares a web-search credential path, but `setCredentialValue` is a no-op and `getCredentialValue` only reads `process.env.ANTHROPIC_API_KEY`. Configure flows can accept a key without saving or later resolving it unless the env var was already set.
  - confidence: 0.81
- **[P2] Remove committed runtime log artifacts:** `openclaw-2026-03-18.log:1`
  - body: The root `openclaw-*.log` files are unrelated to the provider implementation and include local runtime metadata such as run IDs, installed paths, timestamps, and debug/error messages. They should not be part of the PR.
  - confidence: 0.9
