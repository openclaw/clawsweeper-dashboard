## Summary

Keep this PR open, but do not merge the branch as-is. Current main still has the explicit `--agent` plus `--to` resolver bug this PR targets, while the PR head is stale relative to the merged default-agent routing compatibility work and needs to preserve that behavior when rebased or ported.

## What This Changes

The PR changes the agent session resolver so `--agent` plus `--to` derives an agent-scoped target session key before falling back to the agent main session, and adds regression tests for precedence, blank session keys, and qualified `--to` rebinding.

## Best Possible Solution

Port the focused resolver change onto current main: derive selected-agent `--to` keys before the implicit selected-agent main fallback, keep explicit `sessionKey` and `sessionId` precedence, preserve #72414 default-agent store selection for non-explicit-agent requests, and retain the new regression tests.

Yes. The static current-main path is high-confidence: `--agent mybot --to cw_111` has no `sessionId`, so the resolver gets `agent:mybot:main` from the explicit-agent fallback before the `--to` context can derive `agent:mybot:cw_111`.

## Solution Assessment

No, not as currently staged. The PR is the right narrow direction for the explicit-agent CLI bug, but the branch must be rebased or ported so it does not drop current main's default-agent session-store behavior from #72414.
