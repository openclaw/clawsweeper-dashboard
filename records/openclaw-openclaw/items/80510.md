## Summary

Keep this protected MEMBER draft PR open; the Control UI and diagnostic fixes look directionally correct, but the newly added signed-commit enforcement has a concrete verification gap that should block merge until repaired.

## What This Changes

The PR catches Control UI WebSocket constructor failures, adjusts diagnostic liveness stability severity, adds related tests and changelog entries, and adds a new mandatory signed-commit preflight/verification helper for commit hooks.

## Best Possible Solution

Keep the WebSocket and diagnostic fixes, but repair or split the signed-commit enforcement so the hook actually rejects SSH signing configurations that cannot be locally verified before merge.

## Reproduction Assessment

Yes, source-level: current main constructs the browser WebSocket without guarding synchronous constructor failures, and the PR diff shows a source-level bypass in the new signing verifier for private-key SSH signing configs. I did not run the focused tests under the read-only review policy.

## Solution Assessment

No for the full PR as-is: the Control UI and diagnostic changes are narrow, but the signed-commit enforcement needs a pre-commit verification repair before it is a maintainable merge candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reject unverifiable private-key signing configs:** `scripts/check-commit-signing.mjs:151-152`
  - body: `resolveSshSigningPublicKey` returns `null` for the common SSH `user.signingkey=/path/to/private_key` form, and the preflight only fails when `publicKey` is truthy. That lets normal `git commit` pass the new pre-commit hook with an `allowedSignersFile` that cannot verify the resulting commit; only `scripts/committer` catches it after the bad commit already exists. Derive the public key before commit, or fail the preflight when local verification cannot be proven.
  - confidence: 0.88

Ensure normal `git-hooks/pre-commit` cannot pass when the resulting SSH-signed commit would fail `git verify-commit HEAD`, and avoid leaving unverifiable commits behind in `scripts/committer`. No changelog entry is needed for this internal repair unless maintainers request one.
