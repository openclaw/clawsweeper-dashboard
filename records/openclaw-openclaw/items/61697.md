## Summary

Current main and the shipped Gmail webhook docs now implement the supported Gmail Pub/Sub path: OpenClaw owns the gog watcher lifecycle, separates the public Tailscale path from the local gog serve path, forwards to `/hooks/gmail` with the hook token, and maps that webhook into agent delivery. The report's manual external watcher setup is superseded by that shipped flow; the related open PR #61704 is only a diagnostics/log-message follow-up, not the delivery fix.

## What This Changes

The issue reports that a gog Gmail Pub/Sub watch remains active but no webhook requests reach OpenClaw's Gmail hook endpoint on OpenClaw 2026.4.5.

## Best Possible Solution

Keep the shipped integrated Gmail webhook flow as the supported path: use `openclaw webhooks gmail setup` or the documented `hooks.gmail` config with topic, push token, hook token, Tailscale path, and Gateway-owned watcher lifecycle. Treat #61704 as a separate diagnostics cleanup if maintainers still want clearer wording for external webhook setups.

## Reproduction Assessment

No live Gmail/Pub/Sub reproduction was run because it requires the reporter's Google account, gog auth state, and Tailscale Funnel. Static current-main verification is high confidence: the reported manual setup is superseded by shipped config/docs, and the old missing-topic/port-mismatch symptoms are covered by the current supported flow.

## Solution Assessment

Yes. The current solution is the narrower maintainable path because OpenClaw owns the watcher lifecycle and config generation instead of relying on a separate manual `gog gmail watch serve` process with hand-wired ports and paths.
