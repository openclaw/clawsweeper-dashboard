## Summary

Close as implemented on main: current main and the shipped v2026.4.23+ Gmail webhook flow cover the reported no-delivery setup by owning the gog watcher lifecycle, separating the public Tailscale path from the local serve path, and routing mapped `/hooks/gmail` requests into agent delivery.

## What This Changes

The issue reports that a gog Gmail Pub/Sub watch remains active but no webhook requests reach OpenClaw's Gmail hook endpoint on OpenClaw 2026.4.5.

## Best Possible Solution

Keep the shipped integrated Gmail webhook flow as the supported path: `openclaw webhooks gmail setup` or equivalent `hooks.gmail` config with topic, push token, hook token, Tailscale path, and Gateway-owned watcher lifecycle.

## Reproduction Assessment

No live Gmail/Pub/Sub reproduction was run because it requires the reporter's Google account, gog auth state, and Tailscale Funnel. Static current-main verification is high confidence because the reported manual setup is superseded by shipped config, docs, and hook-handler coverage.

## Solution Assessment

Yes. The current implementation is the narrower maintainable solution because OpenClaw owns setup, Tailscale path normalization, watcher lifecycle, and hook dispatch instead of relying on a separately hand-wired `gog gmail watch serve` process.
