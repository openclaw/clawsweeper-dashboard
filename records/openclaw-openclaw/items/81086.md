## Summary

Keep open. Current main still has a source-level path where a Telegram group session key can fall back to `group:<chatId>`, while Telegram send normalization rejects that non-numeric recipient; the related prior reports found by search are closed and do not leave an open canonical fix path.

## What This Changes

The issue requests that `sessions_send` announce delivery for Telegram group sessions resolve to a numeric Telegram chat ID instead of `group:<id>`.

## Best Possible Solution

Add Telegram-owned session-target normalization so session-derived group and topic keys produce a numeric chat ID plus preserved thread id for announce delivery, with regression tests and live Telegram proof before landing.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: `agent:<agent>:telegram:group:<negativeChatId>` can resolve to `group:<id>`, and the Telegram send path then rejects it as non-numeric. I did not execute a live Telegram send in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is a Telegram plugin-owned `resolveSessionTarget` or similarly scoped session-derived target normalization, not a core Telegram special case or a retry that could send to the wrong surface.
