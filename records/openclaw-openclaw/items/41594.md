## Summary

Keep this open: current main still has only global, per-agent, provider-model compatibility, and per-run tool controls, not the requested alias/model-scoped context profile for workspace files, skills, and tool visibility.

## What This Changes

The issue requests an opt-in per-model or per-alias context profile to control injected workspace files, skills, and tool visibility.

## Best Possible Solution

Define one backward-compatible alias/model-scoped profile surface, coordinated with #59168, then thread it through config schema/help, bootstrap file filtering, skills prompt assembly, tool/client-tool exposure, Control UI, docs, and focused default plus `/model` switching tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. The missing capability is source-verifiable on current main because the strict model-alias schema and prompt assembly paths have no model/alias context-profile selector.

## Solution Assessment

Unclear until maintainers settle the model-profile schema and precedence rules. Existing knobs are useful mitigations, but they are global, per-agent, provider-model compatibility, or per-run controls rather than the requested alias-scoped profile.
