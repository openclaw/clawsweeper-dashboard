## Summary

Keep this issue open. Current main still lacks an alias/model-scoped `contextProfile`; the available controls are global, per-agent, per-provider-model compatibility flags, or per-run allowlists, and they do not provide the requested workspace-file, skills, and tool visibility profile for `/model` switching.

## What This Changes

The issue requests an optional per-model or per-alias context profile that can independently filter injected workspace files, suppress skills prompt metadata, and hide tool definitions for multi-model workflows.

## Best Possible Solution

Define a backward-compatible selected-model or alias-scoped profile surface, preferably coordinated with the broader model-profile work in #59168, then thread it through bootstrap file filtering, skills prompt assembly, tool/client-tool schema exposure, system prompt reporting, generated config schemas/help, docs, Control UI, and focused tests for defaults and `/model` switching.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The missing capability is source-verifiable on current main because the proposed config key is absent from the strict schema and no runtime prompt path receives a model/alias context-profile selector.

## Solution Assessment

Unclear until maintainers decide the model-profile schema. Existing knobs are useful mitigations, but they are not the best complete solution because they are global, per-agent, provider-model, or per-run rather than opt-in alias-scoped profiles.
