## Summary

Keep open: the PR targets a real current-main and shipped TUI local-mode bug, but it is not merge-ready because the fix only intercepts two advertised shared commands while the same fallthrough remains for other advertised local commands, and the supplied proof is unit/static output rather than real local TUI behavior.

## What This Changes

Adds `/status` and `/compact` cases to the TUI command handler so local mode shows an unsupported-command system message instead of sending those command strings to chat, with focused unit tests.

## Best Possible Solution

Update the TUI local command boundary so every locally advertised shared command either has a local handler, a supported shared route, or an explicit unsupported response, then include focused unit coverage plus local TUI or PTY proof that blocked commands do not hit the model path.

## Reproduction Assessment

Yes from source: current main and `v2026.6.8` advertise shared slash commands and route unhandled local command names to `sendMessage(raw)`. I did not run the interactive TUI in this read-only review.

## Solution Assessment

No, not yet: adding two cases is a useful partial repair, but the linked issue's command-boundary invariant is broader than `/status` and `/compact`. The maintainable fix should cover all advertised unsupported shared commands or stop advertising them in local mode.
