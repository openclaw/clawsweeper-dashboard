## Summary

Keep open: the PR targets a real local TUI command-delivery bug, but it is not merge-ready because it only intercepts exact `/status` and `/compact` inputs while local discovery/shared-command parsing still leave colon-form and other advertised shared commands on the embedded model path, and the contributor proof is unit/static only.

## What This Changes

The PR adds local-mode `/status` and `/compact` cases to the TUI command handler and unit tests asserting they show an unsupported system message instead of calling `sendChat`.

## Best Possible Solution

Make local TUI discovery/help and dispatch agree so every advertised local slash command is handled, safely routed, or explicitly rejected before the embedded model path, then prove it with local PTY or live TUI output.

## Reproduction Assessment

Yes. Source shows slash input enters `handleCommand`, unhandled names reach `sendMessage(raw)`, local discovery/help expose shared commands, and the embedded backend passes raw text to `agentCommandFromIngress`; I did not run the PTY repro in this read-only review.

## Solution Assessment

No. The two-command switch case is a useful mitigation, but the best fix is the shared local command boundary, including colon-form normalization and unsupported-command filtering, with local PTY proof.
