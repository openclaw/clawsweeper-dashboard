## Summary

Keep open: current main still leaves `llm_input` and `llm_output` observe-only, so the central feature is not already implemented, but this PR should not merge as-is because source inspection shows provider-cache ordering, prompt-injection policy, payload-isolation, and documentation blockers. The contributor supplied useful live gateway and Docker policy-engine logs, so the remaining blocker is branch/design repair rather than missing proof.

## What This Changes

The PR changes plugin `llm_input` and `llm_output` hooks and embedded runner handling so plugins can block or rewrite model input and replace assistant output.

## Best Possible Solution

Land a maintainer-approved hook contract that runs before provider-visible prompt/cache construction, preserves current prompt-injection and payload-isolation contracts, and documents the public plugin API semantics.

## Reproduction Assessment

Yes. Source inspection against current main and the PR head gives high-confidence reproduction paths for the review findings: the hook runs after Google cache setup, the guarded-hook list is stale, and history cloning is shallow compared with current main.

## Solution Assessment

No. The feature direction is plausible, but this branch is not the best merge shape until it is rebased onto current prompt assembly and the security/policy/docs gaps are fixed.

A hook can mutate nested content or see runtime/tool details outside the explicit result contract; keep `cloneHookMessages(normalizeMessagesForLlmBoundary(...))` for this payload.
