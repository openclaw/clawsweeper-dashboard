## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Make result-bearing LLM hooks runner-consistent:** `src/plugins/hooks.ts:967`
  - body: This changes the shared `runLlmInput`/`runLlmOutput` contract to return modifying results, but current main's CLI and harness paths still call those methods as best-effort observation hooks and ignore returned blocks, rewrites, and redactions. A plugin would therefore work in the embedded runner but not in sibling runtimes, so the public API should either stay observation-only there or be implemented consistently across those paths before merge.
  - confidence: 0.9
- **[P1] Rebuild provider inputs after prompt rewrites:** `src/agents/embedded-agent-runner/run/attempt.ts:4144-4148`
  - body: The PR applies `llm_input` prompt/system-prompt overrides after prompt-local images and provider/cache state have already been prepared. If a security plugin removes or adds image references, the model call can still use stale `imageResult` state from the original prompt; move the hook before those preparations or rebuild the affected provider inputs after any rewrite.
  - confidence: 0.89
- **[P1] Replace downstream assistant state on output overrides:** `src/agents/embedded-agent-runner/run/attempt.ts:4948-4949`
  - body: `llm_output` redaction is saved in `llmOutputAssistantTextsOverride`, but the original `assistantTexts` and `lastAssistant` are still used for context maintenance, `agent_end`, empty-reply handling, trajectory records, and terminal classification before only the returned result field is swapped. That means original content can remain visible to downstream state/diagnostic/plugin consumers after a redaction hook claims to replace it.
  - confidence: 0.88
- **[P2] Add default budgets for awaited LLM hooks:** `src/plugins/hooks.ts:967-1003`
  - body: Switching `llm_input` and `llm_output` from fire-and-forget to awaited sequential modifying hooks puts plugin latency directly on the model-call path, but the default modifying-hook timeout map does not include either hook. A hung plugin can now stall prompt submission or final response handling indefinitely unless these hooks get bounded defaults or an explicit timeout policy.
  - confidence: 0.84
