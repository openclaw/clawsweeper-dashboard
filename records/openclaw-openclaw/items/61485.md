## Summary

Keep open for maintainer review: current main still treats `llm_input` and `llm_output` as observe-only hooks, while this PR is a security-sensitive public plugin API change with blocking correctness and policy-regression findings before merge.

## What This Changes

The PR changes `llm_input` and `llm_output` hook types, hook-runner behavior, embedded-runner handling, and tests so plugins can block or rewrite prompts/system prompts and replace assistant response text.

## Best Possible Solution

Land a rebased, documented hook contract that preserves current prompt-injection protections, uses immutable hook event snapshots, and proves the actual provider call and delivered output honor the modified values.

## Reproduction Assessment

Yes. A focused test can register an `llm_input` hook under `allowPromptInjection=false` that mutates nested `historyMessages` content and assert live session state is unchanged; code inspection shows the PR only shallow-copies those messages.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best merge candidate until the security regressions, stale main integration, docs, changelog, and overlapping hook API direction are resolved.
