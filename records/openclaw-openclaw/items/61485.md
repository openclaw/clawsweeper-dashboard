## What This Changes

The PR changes `llm_input` and `llm_output` plugin hook types and embedded-runner logic so hooks can return results that block LLM calls, rewrite prompts/system prompts, or replace assistant text.

## Best Possible Solution

Land only after maintainers choose and document the permanent plugin API contract, make `llm_input`/`llm_output` semantics consistent across embedded, CLI, and Codex/harness paths or intentionally scope the API, and refresh current-head real-behavior proof.

Yes for the review blockers: source inspection shows the public hook result types change while CLI and Codex harness helpers still discard results, and the embedded attempt path applies output overrides only at return time. This is a feature PR rather than a current-main bug reproduction.

## Solution Assessment

No; this is only a plausible implementation. The better fix is to settle the global plugin API semantics first, then apply one effective prompt/output state consistently across model submission, terminal bookkeeping, diagnostics, and all supported runners.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep llm hooks consistent across runners:** `src/plugins/hook-types.ts:1045-1052`
  - body: This changes the public handler map so plugins can return `block`, prompt overrides, or `assistantTexts`, but the CLI and Codex harness paths still invoke these hooks through fire-and-forget helpers that ignore returned values. A compliance plugin would therefore work in embedded runs and silently fail to block/redact the same hook contract in CLI/Codex runs; either make the contract runner-wide or do not advertise these returns globally.
  - confidence: 0.9
- **[P1] Submit systemPrompt overrides to the model:** `src/agents/embedded-agent-runner/run/attempt.ts:4147-4149`
  - body: Assigning `llmInputResult.systemPrompt` to `systemPromptForHook` only updates later prechecks/trajectory metadata; it does not call `setActiveSessionSystemPrompt` before `activeSession.prompt(...)`. That means a plugin can appear to rewrite the system prompt in telemetry while the model still receives the old session prompt, which breaks the advertised policy/compliance use case.
  - confidence: 0.88
- **[P1] Apply output overrides before terminal bookkeeping:** `src/agents/embedded-agent-runner/run/attempt.ts:5141`
  - body: The override is kept aside until the returned attempt object, but silent-reply decisions, terminal status, and trajectory/artifact records above this return still use the original `assistantTexts`. A redaction or suppression plugin can therefore leave the raw assistant text in diagnostics/artifacts and terminal decisions even though the final attempt result is replaced.
  - confidence: 0.87
- **[P2] Give awaited LLM hooks a default timeout:** `src/plugins/hooks.ts:967`
  - body: `llm_input` and `llm_output` are now awaited on the request path through `runModifyingHook`, but neither hook has a default modifying timeout. Unless every plugin author configures one, a hung policy/redaction plugin can stall model submission or response delivery indefinitely; add bounded defaults like the existing awaited prompt hooks.
  - confidence: 0.82
