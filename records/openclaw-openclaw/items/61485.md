## Summary

This PR should stay open because current main still has observe-only `llm_input`/`llm_output` hooks, but the branch is not merge-ready: the patch changes a public, security-sensitive plugin contract and still has blocking hook-timing and policy-regression issues.

## What This Changes

The PR changes `llm_input` and `llm_output` hook types and embedded-runner handling so plugins can block or rewrite prompts/system prompts and replace assistant response text, with hook-runner tests for the new merge behavior.

## Best Possible Solution

Land a rebased, documented hook contract that applies `llm_input` modifications before provider submission, image detection, and prompt-cache wrapping; preserves all existing prompt-injection policy hooks; and proves the actual embedded-runner model call and delivered output use the modified values.

## Reproduction Assessment

Yes. Register an `llm_input` hook returning a rewritten prompt and assert the embedded runner submits that rewritten prompt; code inspection shows the PR updates `effectivePrompt` after `promptSubmission` is already resolved, so that integration check should expose the bug.

## Solution Assessment

No. The feature direction is plausible, but this exact patch is not the best implementation until hook timing, Google prompt-cache state, prompt-injection guard preservation, docs/changelog follow-through, and overlapping hook API direction are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move llm_input rewrites before prompt submission:** `src/agents/pi-embedded-runner/run/attempt.ts:2504`
  - body: `promptSubmission` is resolved before this new hook result is applied, but the model call later submits `promptSubmission.prompt`. Updating only `effectivePrompt` here means a plugin's prompt rewrite or redaction can fail to reach the provider while precheck and diagnostics may evaluate a different prompt.
  - confidence: 0.94
- **[P1] Preserve existing prompt-injection guarded hooks:** `src/plugins/hook-types.ts:130-136`
  - body: Current main already includes `agent_turn_prepare` and `heartbeat_prompt_contribution` in `PROMPT_INJECTION_HOOK_NAMES`. This hunk must preserve those entries while adding `llm_input`; otherwise `allowPromptInjection=false` can stop constraining existing prompt-mutating hooks.
  - confidence: 0.9
- **[P2] Rebuild prompt-cache wrapping after system rewrites:** `src/agents/pi-embedded-runner/run/attempt.ts:2509`
  - body: `prepareGooglePromptCacheStreamFn` is called before the new `systemPrompt` override is applied. On Google prompt-cache eligible runs, a plugin system-prompt rewrite can be ignored or cached against the old prompt unless the hook moves earlier or the wrapper is rebuilt.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff grants installed plugins new authority over model prompts and responses, and the current patch has concrete policy-bypass and prompt-redaction correctness concerns.

Concerns:

- **[high] Prompt rewrites may not reach the provider call:** `src/agents/pi-embedded-runner/run/attempt.ts:2504`
  - body: Compliance or redaction plugins could believe `llm_input` changed the prompt while the embedded runner submits prompt state prepared before the rewrite, leaving original sensitive content eligible for model submission.
  - confidence: 0.94
- **[high] Prompt-injection policy can regress on existing hooks:** `src/plugins/hook-types.ts:130`
  - body: Adding `llm_input` to the prompt-injection guard must preserve the current guarded hook list; dropping existing entries would weaken `allowPromptInjection=false` for established prompt-mutating hooks.
  - confidence: 0.9
