## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Bound awaited LLM hooks with default timeouts:** `src/plugins/hooks.ts:957-959`
  - body: This changes `llm_input`/`llm_output` from background observation to awaited `runModifyingHook`, but the modifying-hook timeout defaults still do not include either hook. An existing plugin handler that never settles would now stall every model call instead of running fire-and-forget, so add explicit bounded defaults or another non-hanging execution contract before merge.
  - confidence: 0.91
- **[P1] Apply system-prompt overrides to the model session:** `src/agents/embedded-agent-runner/run/attempt.ts:4146-4148`
  - body: Assigning `systemPromptForHook` only updates later local bookkeeping; the active session system prompt and Google prompt-cache wrapper are already prepared from the old `systemPromptText`. A plugin returning `systemPrompt` can therefore be reported in diagnostics while the provider still receives the original system prompt/cache state.
  - confidence: 0.87
- **[P2] Reload prompt images after prompt overrides:** `src/agents/embedded-agent-runner/run/attempt.ts:4142-4144`
  - body: When `llm_input` returns a new `prompt`, `imageResult` has already been built from the original prompt. That misses image refs introduced by the plugin and can keep stale image attachments from the old prompt, so the prompt override needs image re-detection or rollback before the model call.
  - confidence: 0.86
- **[P1] Make output overrides the canonical answer source:** `src/agents/embedded-agent-runner/run/attempt.ts:5138`
  - body: The PR only replaces `assistantTexts`, but `buildEmbeddedRunPayloads` normally prefers canonical `lastAssistant` final-answer text over `assistantTexts`. A redaction plugin can still deliver the original assistant content unless the override also updates or masks the canonical assistant state used for payload construction and downstream metadata.
  - confidence: 0.88
