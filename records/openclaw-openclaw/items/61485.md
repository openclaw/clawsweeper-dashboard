  - body: The handler map now advertises `llm_input`/`llm_output` return values as plugin API, but CLI and harness paths still call these hooks as best-effort void side effects and continue with the original prompt/output. A plugin that blocks or redacts through the new contract would work in the embedded runner but be silently ignored in those sibling LLM paths, so the public contract needs to be implemented consistently or explicitly narrowed before merge.
  - confidence: 0.92

That means a security plugin can approve or rewrite one input while the model call still carries stale images or cached system-prompt state from the previous input.
  - confidence: 0.9
- **[P1] Replace downstream assistant state on output overrides:** `src/agents/embedded-agent-runner/run/attempt.ts:4948-4949`
  - body: When `llm_output` returns `assistantTexts`, the override is only saved for the final returned `assistantTexts`; context-engine finalization, `agent_end`, diagnostics, `messagesSnapshot`, and `lastAssistant` have already seen the original assistant content. A redaction plugin would therefore not actually redact the downstream state surfaces that other plugins and diagnostics consume.
  - confidence: 0.88
- **[P2] Add default budgets for awaited LLM hooks:** `src/plugins/hooks.ts:967`
  - body: This switches `llm_input` and `llm_output` onto `runModifyingHook`, which awaits handlers, but the default modifying-hook timeout map still has no entries for either hook. A hung plugin can now stall the model path until a broader run timeout instead of being bounded like the other awaited prompt hooks.
