## Label Justifications

- `P2`: This is a normal-priority plugin API feature with meaningful but bounded agent runtime impact.
- `merge-risk: 🚨 compatibility`: Changing existing hook return semantics can create different plugin behavior across runner surfaces and surprise plugin authors.
- `merge-risk: 🚨 security-boundary`: The PR lets enabled plugins block or rewrite model-bound prompts and assistant outputs, which is a prompt-integrity and sensitive-data boundary.
- `merge-risk: 🚨 availability`: Awaited LLM hooks without a default budget can stall model turns when a plugin is slow or hung.

## Summary

Keep open: current main still has observe-only `llm_input`/`llm_output` hooks, so this is a meaningful implementation candidate, but the PR is not merge-ready because the public hook contract is incomplete across runners and has blocking security/API correctness gaps.

## What This Changes

The PR changes plugin hook types, hook runner behavior, registry prompt-injection policy, and embedded-runner handling so `llm_input` can block or rewrite prompts/system prompts and `llm_output` can replace assistant text.

## Best Possible Solution

Maintainers should either approve a global modifying `llm_input`/`llm_output` contract and implement it consistently across embedded, CLI, and harness paths with tests/docs/proof, or explicitly narrow the contract before merge so plugin authors are not promised behavior unsupported runner paths discard.

- **[P1] Apply systemPrompt overrides to the session:** `src/agents/embedded-agent-runner/run/attempt.ts:4148`
  - body: This branch records a returned `systemPrompt` only in `systemPromptForHook`, which feeds diagnostics/prechecks, but the actual model path uses the active session system prompt set earlier via `setActiveSessionSystemPrompt`. A plugin returning `{ systemPrompt }` can therefore believe it changed the provider-bound system prompt while the old prompt is still used, including for Google prompt-cache setup.
  - confidence: 0.9
- **[P1] Keep modifying hooks consistent across runners:** `src/plugins/hook-types.ts:1042-1047`
  - body: The new result-bearing handler types promise plugins can block or rewrite LLM traffic, but CLI and harness dispatch still call `runLlmInput`/`runLlmOutput` as best-effort observations and ignore returned results. That makes the public plugin contract work in the embedded runner while silently failing in sibling runner paths.
  - confidence: 0.87
- **[P2] Give awaited LLM hooks a default budget:** `src/plugins/hooks.ts:957-961`
  - body: `llm_input` and `llm_output` become awaited modifying hooks on the model path, but the default modifying-hook timeout table does not include either hook name. Without a finite default, any plugin that hangs in these hooks can stall the LLM turn indefinitely unless every installation configures its own timeout.
  - confidence: 0.84
