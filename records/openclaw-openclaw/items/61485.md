## Summary

Keep open: current main still treats `llm_input` and `llm_output` as observe-only hooks, while this PR proposes a real new plugin capability. The branch is not ready to merge because it is stale against the current prompt/cache pipeline and still has policy, payload-isolation, and documentation blockers.

## What This Changes

This PR changes plugin `llm_input` and `llm_output` hooks to await sequential modifying results that can block or rewrite model input and replace assistant output, with sentinel handling and tests.

## Best Possible Solution

Land a maintainer-approved, rebased hook contract that runs before provider-visible prompt/cache construction, preserves existing prompt-injection policy, isolates hook payloads with real deep clones or immutable views, and updates public docs plus focused contract tests.

## Reproduction Assessment

Yes. Source inspection against current main and the PR head gives a high-confidence path for the blocking findings: prompt/cache construction moved ahead of the old hook site, the guard list is stale, and the history payload clone is shallow.

## Solution Assessment

No. The feature direction is plausible and the provided live logs are useful, but this branch is not the best merge shape until it is rebased onto current prompt assembly and closes the policy, payload-isolation, and docs gaps.
