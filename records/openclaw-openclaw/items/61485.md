  - body: The diff passes `evolveEvent` into `runModifyingHook`, but `ModifyingHookPolicy` has no such field and the runner never applies evolved events.   - body: When `llm_output` returns replacement `assistantTexts`, the attempt result only swaps that array and leaves `currentAttemptAssistant` unchanged. `buildEmbeddedRunPayloads` prefers `currentAssistant` over assistant text arrays, so a redaction or suppression plugin can still leak the original assistant content in the normal reply payload.
  - confidence: 0.9
- **[P1] Run input rewrites before cache and image setup:** `src/agents/embedded-agent-runner/run/attempt.ts:4147-4152`
  - body: The new prompt/systemPrompt overrides are applied at the old `llm_input` observation point, after Google prompt-cache setup and prompt image detection already used the original values. A compliance plugin can therefore rewrite a system prompt or prompt text while cached content and image metadata remain based on the pre-hook input.
  - confidence: 0.86
- **[P1] Budget the awaited LLM hooks:** `src/plugins/hooks.ts:956`
  - body: This converts `llm_input` and `llm_output` from fire-and-forget observation into awaited modifying hooks, but the default modifying-hook timeout table still omits both names. A stuck plugin handler can now block model submission or response delivery indefinitely unless an operator happened to configure a per-hook timeout.
  - confidence: 0.88
- **[P2] Keep modifying hook semantics consistent across runtimes:** `src/plugins/hook-types.ts:1042-1048`
  - body: The public handler types now advertise modifying results for `llm_input` and `llm_output`, but CLI, shared harness helpers, and the Codex adapter still call these hooks fire-and-forget and ignore returned values. A plugin author could believe a block/redaction policy is enforced while it silently does nothing outside the embedded runner.
  - confidence: 0.82
