## Summary

Keep open: current main still has observe-only `llm_input`/`llm_output`, older related PRs are closed unmerged, and this is the active focused implementation candidate. The PR is not merge-ready because the public hook contract is inconsistent across runtimes, embedded-runner ordering can leave stale provider-visible state, awaited hooks lack default budgets, the branch is currently conflicting, and current-head real behavior proof is insufficient.

## What This Changes

The PR changes `llm_input` and `llm_output` plugin hooks from fire-and-forget observers into awaited modifying hooks that can block LLM calls, rewrite prompts/system prompts, and replace assistant text.

## Best Possible Solution

Keep this PR open, refresh it against current main, and either implement one consistent modifying-hook contract across embedded, CLI, and harness paths with docs/tests/proof, or narrow the public API to a maintainer-approved hook boundary.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR changes public hook return types while current CLI/harness dispatchers ignore results, and awaited `llm_input`/`llm_output` have no default timeout. This is not a current-main user bug reproduction because the PR proposes a new capability.

## Solution Assessment

No. The embedded-runner path is only a partial solution if the public PluginHookHandlerMap contract applies to all hook runners; the safer fix is to wire the modifying result through every runtime surface or narrow the exposed contract to the supported runner.

  - body: Changing `llm_input`/`llm_output` to return modifying results advertises this behavior to every plugin, but the CLI/harness path still dispatches both hooks best-effort and ignores returned block/rewrite/redaction results (`runAgentHarnessLlmInputHook`/`runAgentHarnessLlmOutputHook`).   - body: The PR applies `llm_input` prompt/systemPrompt results after the embedded runner has already prepared prompt-cache state and prompt-local images on the original prompt/system prompt. That means a plugin can appear to rewrite or remove model-boundary content while stale images or cached system-prompt state remain provider-visible; the hook result needs to feed the authoritative prompt/system/image preparation path.
  - confidence: 0.86
- **[P1] Make output replacements authoritative:** `src/agents/embedded-agent-runner/run/attempt.ts:5138`
  - body: When `llm_output` returns `assistantTexts`, the patch only swaps the returned `assistantTexts` field while leaving `lastAssistant` and earlier terminal/trajectory decisions tied to the original assistant message. A redaction plugin can therefore replace the visible text but still leave original content or error metadata in downstream state; update the authoritative assistant message/state or keep the hook observe-only.
  - confidence: 0.82
- **[P2] Add default budgets for awaited LLM hooks:** `src/plugins/hooks.ts:957-991`
  - body: This changes `llm_input` and `llm_output` from fire-and-forget observers into awaited modifying hooks, but neither hook is added to `DEFAULT_MODIFYING_HOOK_TIMEOUT_MS_BY_HOOK`. A hung plugin can now stall the agent turn at the model boundary indefinitely, so the new awaited hooks need explicit fail-open timeouts or a different non-blocking contract.
