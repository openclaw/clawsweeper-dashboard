## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Apply system prompt overrides to the actual session:** `src/agents/embedded-agent-runner/run/attempt.ts:4141-4143`
  - body: The new `systemPrompt` result is assigned only to the hook/precheck metadata variable, but the provider path is already configured from `systemPromptText` and the active session system prompt. That means a plugin can return a system prompt rewrite that shows up in traces/accounting while the model and Google prompt-cache wrapper still use the original system prompt. Apply the override through the same active-session system prompt path as the existing prompt hooks, and rebuild any provider/cache wrapper that depends on it.
  - confidence: 0.9
- **[P1] Apply the shared hook contract consistently:** `src/plugins/hook-types.ts:277-287`
  - body: The shared `llm_input`/`llm_output` handler map now advertises returned block/rewrite/redaction values, but the patch only consumes those values in the embedded runner. Existing callers such as the CLI and agent harness helpers still invoke `runLlmInput`/`runLlmOutput` as best-effort hooks and ignore the result, so the same plugin API will silently mean modifying in one runner and observe-only in another. Either keep this API local to the supported path or wire/guard every public caller before exposing the shared return contract.
  - confidence: 0.88
- **[P2] Add default budgets before awaiting LLM hooks:** `src/plugins/hooks.ts:957-1017`
  - body: Switching `llm_input` and `llm_output` to `runModifyingHook` makes every handler awaited on the model path, but the default modifying-hook timeout table does not include either hook. A hung plugin that was previously fire-and-forget can now stall an agent turn indefinitely unless every plugin/operator configures a timeout manually. Add bounded defaults for these hook names before making them awaited.
  - confidence: 0.86
