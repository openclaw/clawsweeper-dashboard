## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep modifying hooks consistent across runners:** `src/plugins/hook-types.ts:1045-1052`
  - body: This changes the public `llm_input`/`llm_output` handler signatures to return mutations, but only the embedded runner consumes those mutations. CLI and Codex/harness callers still dispatch these hooks best-effort and ignore return values, so plugin authors get one API with runner-dependent block/rewrite/redact behavior. Either implement the contract across supported runners or narrow/document the API before exposing result types globally.
  - confidence: 0.92
- **[P1] Rebuild prompt-cache inputs after hook rewrites:** `src/agents/embedded-agent-runner/run/attempt.ts:4144-4149`
  - body: `llm_input` applies prompt/systemPrompt overrides after Google prompt-cache wrapping and prompt image loading have already used the old values. For cached-content runs the wrapper removes the runtime system prompt and uses the pre-hook cached one, so a compliance plugin's systemPrompt rewrite can be silently bypassed; prompt-local images also remain based on the pre-hook prompt. Run the modifying hook before these setup paths, or recompute the affected state after rewrites.
  - confidence: 0.9
- **[P1] Propagate output rewrites before terminal state:** `src/agents/embedded-agent-runner/run/attempt.ts:5141-5144`
  - body: The `llm_output` override is only applied to the returned `assistantTexts`. Before that return, empty-reply classification, terminal trajectory events, trace artifacts, and the returned `lastAssistant` still use the original assistant output, so redacted text can remain in diagnostics/state and output semantics can be based on content the plugin meant to replace. Apply the final assistant text/state before those consumers run.
  - confidence: 0.89
- **[P1] Give awaited LLM hooks a default budget:** `src/agents/embedded-agent-runner/run/attempt.ts:4114-4115`
  - body: This PR turns `llm_input` and `llm_output` from fire-and-forget observation hooks into awaited modifying hooks on every embedded LLM call, but the default modifying-hook timeout map still omits both names. An existing or new plugin handler that never settles can now stall the model turn indefinitely unless every registration supplies its own timeout. Add bounded defaults and tests for timeout behavior before awaiting these hooks in the hot path.
  - confidence: 0.86
