## Summary

Keep open: current main still keeps `llm_input` and `llm_output` observe-only, and this PR carries useful work plus sufficient live log proof, but the branch is dirty against current main and still has blocking API/security correctness issues before merge.

## What This Changes

The PR changes plugin `llm_input` and `llm_output` hook contracts and runner handling so plugins can block or rewrite model input and replace assistant output, with sentinel handling and tests.

## Best Possible Solution

Land a maintainer-approved, rebased hook contract that runs before provider-visible prompt/cache construction, preserves prompt-injection policy, isolates hook payloads, and documents the public plugin API.

## Reproduction Assessment

Yes. Source inspection against current main and the PR head gives a high-confidence path for the blocking findings: the hook runs after provider-visible prompt/cache preparation, the guard list is stale, and history cloning is shallow.

## Solution Assessment

No. The feature direction is plausible and the proof is useful, but this branch is not the best merge shape until it is rebased onto current prompt assembly and the policy, payload-isolation, and docs gaps are fixed.

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move llm_input mutations before provider prompt setup:** `src/agents/pi-embedded-runner/run/attempt.ts:2507-2514`
  - body: This applies prompt/systemPrompt overrides after `prepareGooglePromptCacheStreamFn` has already wrapped the stream with the old system prompt. On Google prompt-cache eligible runs, a plugin can believe it changed model input while the provider-visible cache wrapper still uses pre-hook state.
  - confidence: 0.88
- **[P1] Preserve existing prompt-injection guards:** `src/plugins/hook-types.ts:130-134`
  - body: This guarded-hook list is based on an older main and omits current prompt-mutating hooks such as `agent_turn_prepare` and `heartbeat_prompt_contribution`. Rebase this additively so `allowPromptInjection=false` does not lose existing protections while adding `llm_input`.
  - confidence: 0.92
- **[P1] Deep-clone llm_input history messages:** `src/agents/pi-embedded-runner/run/attempt.ts:2472-2475`
  - body: `activeSession.messages.map((m) => ({ ...m }))` only copies each top-level message object. Nested content remains shared, so a hook can mutate history in place and bypass both the explicit result contract and `allowPromptInjection=false` prompt-field stripping.
  - confidence: 0.86
- **[P2] Document the modifying llm hook contract:** `src/plugins/hook-types.ts:191-199`
  - body: The public docs still describe `llm_input` and `llm_output` as observation hooks only. Since these result types change plugin API and security semantics, the return shapes, ordering, and `allowPromptInjection=false` behavior need docs before merge.
  - confidence: 0.8
