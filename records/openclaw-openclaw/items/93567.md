## Summary

This PR is still necessary and appears mergeable after normal maintainer review: current main still has the cron run-log write/read jobId asymmetry, and the proposed write-side normalization matches the existing read-side validation without adding a new surface.

## What This Changes

The PR normalizes cron run-log `jobId` values inside `appendCronRunLog` before write-keying, SQLite insertion, and pruning, then adds regression tests for padded and unsafe IDs.

## Best Possible Solution

Land the narrow write-side normalization with its regression tests after normal maintainer review and required checks.

## Reproduction Assessment

Yes. Current main writes raw `entry.jobId` but trims and validates jobId before reads, and the PR body includes real SQLite proof plus a negative control; I did not run local tests because this review is read-only.

## Solution Assessment

Yes. Normalizing once in `appendCronRunLog` before the write key, insert, and prune is the narrow owner-boundary fix; caller-level normalization would miss direct store users and SQLite-only normalization would leave queue keys inconsistent.
