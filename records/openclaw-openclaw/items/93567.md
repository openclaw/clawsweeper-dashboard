Keep open: current main and the latest release still have the cron run-log write/read jobId asymmetry, and this PR is a narrow, well-proven fix with no blocking findings.

## What This Changes

The branch normalizes cron run-log jobIds on append using the read-side validation helper and adds regression tests for padded and unsafe IDs.

## Best Possible Solution

Land the narrow write-side normalization with the regression tests after normal maintainer review and required merge gates.

## Reproduction Assessment

Yes. Current main writes raw entry.jobId but trims and validates jobId before exact-key SQLite reads, and the PR body supplies after-fix real SQLite proof plus a negative control.

## Solution Assessment

Yes. Normalizing once in appendCronRunLog before the queue key, insert, and prune is the narrowest owner-boundary fix; caller-only normalization would miss direct store users, and SQLite-only normalization would leave queue/prune keys inconsistent.
