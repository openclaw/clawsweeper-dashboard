## Summary

Keep open for normal maintainer review: current main still has the write/read run-log jobId asymmetry, while this PR fixes the implicated write path with focused tests and sufficient real SQLite proof; I found no actionable review findings.

## What This Changes

The PR normalizes cron run-log job IDs on write before computing the write key, inserting the row, and pruning, and adds regression tests for padded and unsafe IDs.

## Best Possible Solution

Land this narrow write-side normalization with its regression tests after normal maintainer review and required checks.

## Reproduction Assessment

Yes. Current main writes raw entry.jobId but trims and validates jobId before reads, and the PR body includes real SQLite proof plus a negative control; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Normalizing once in appendCronRunLog before the write key, insert, and prune is the narrowest maintainable fix; caller-level normalization would miss direct store users and SQLite-only normalization would leave queue keys inconsistent.
