## Summary

Keep open: current main and the latest release still write cron run-log rows with raw `entry.jobId` while read paths trim and validate before exact SQLite lookup. This PR is a focused fix with sufficient real SQLite terminal proof, matching regression tests, and no blocking review findings.

## What This Changes

The PR normalizes cron run-log `jobId` in `appendCronRunLog` before write keying, insertion, and pruning, and adds regression tests for padded and unsafe job IDs.

## Best Possible Solution

Land the append-boundary normalization and regression tests after routine maintainer review and required checks complete.

## Reproduction Assessment

Yes. Source inspection on current main shows raw write IDs flowing into exact SQLite `job_id` storage while reads trim and validate before querying; the PR body also supplies real SQLite terminal proof and a negative control.

## Solution Assessment

Yes. Normalizing once in `appendCronRunLog` before the queue key, insert, and prune is the narrow owner-boundary fix; caller-only normalization would miss direct store users, and SQLite-only normalization would leave pending-write keys inconsistent.
