## Summary

This PR is still necessary because current main keeps an asymmetric cron run-log contract: writes store the raw job ID while reads trim and validate before querying. The proposed patch is a focused storage-boundary fix with matching regression tests and sufficient real SQLite proof, and I found no blocking correctness or security finding.

## What This Changes

The PR normalizes `appendCronRunLog` job IDs through the existing safe run-log job-id validator before computing the write key, inserting the row, and pruning, with regression tests for padded and unsafe IDs.

## Best Possible Solution

Land this narrow write-side normalization with its regression tests after normal maintainer review and required checks, keeping the existing read-side validator as the single run-log job-id contract.

## Reproduction Assessment

Yes. Source inspection shows current main writes raw `entry.jobId` but trims and validates `jobId` before reads; the PR also supplies real SQLite after-fix proof plus a negative-control description.

## Solution Assessment

Yes. Normalizing once in `appendCronRunLog` before the write key, insert, and prune is the narrowest maintainable fix; caller-level normalization would miss direct store users, and SQLite-only normalization would leave pending-write keys inconsistent.
