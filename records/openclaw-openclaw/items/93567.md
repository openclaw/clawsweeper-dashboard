## Summary

Keep open. Current main still has an asymmetric cron run-log contract: writes store `entry.jobId` verbatim while the read paths trim and validate before querying; this PR fixes that at the write boundary with focused tests and supplied real SQLite proof, and I found no blocking patch defect.

## What This Changes

The branch normalizes cron run-log `jobId` values in `appendCronRunLog` before write queue keying, SQLite insert, and pruning, and adds regression tests for padded and unsafe job IDs.

## Best Possible Solution

Land the focused write-side normalization and regression tests after required checks finish, keeping run-log reads and writes on one canonical job-id contract.

## Reproduction Assessment

Yes. Source inspection of current main shows `appendCronRunLog` writes `params.entry.jobId` verbatim while `readCronRunLogEntriesSync` and `readCronRunLogEntriesPage` normalize the query jobId before filtering; I did not run tests because this review is read-only.

## Solution Assessment

Yes. Normalizing once at the writer is the narrow maintainable fix because the write queue key, indexed SQLite row, serialized entry JSON, and prune query all need to share the same canonical jobId; loosening read validation or only trimming queries would preserve unsafe writes.
