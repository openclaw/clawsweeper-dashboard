- `P2`: This is a focused cron run-log persistence bug affecting readback consistency for unusual jobIds, with limited blast radius.

## Summary

This PR is still necessary: current main still writes cron run-log rows under the raw `entry.jobId`, while both read paths trim and validate before querying. The proposed fix is a narrow store-boundary correction with sufficient real SQLite proof and no blocking findings.

## What This Changes

The PR normalizes and validates cron run-log `jobId` values in `appendCronRunLog` before write-key selection, SQLite insertion, pruning, and adds regression tests for padded and unsafe IDs.

## Best Possible Solution

Keep the invariant in `appendCronRunLog` so the run-log store owns its canonical jobId key, then land the PR with the added regression coverage after required checks complete.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: append a padded jobId on current main, then read with the trimmed id; writes use the raw key while reads trim and query the trimmed key.

## Solution Assessment

Yes. Normalizing before write-key selection, insertion, and pruning is the narrowest maintainable fix because it keeps the indexed SQLite row and stored JSON payload under the same read-side key.
