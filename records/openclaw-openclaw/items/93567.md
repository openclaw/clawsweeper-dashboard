## Summary

Keep open: current main still has the write/read asymmetry, and this PR is a narrow, proven fix with no actionable Codex findings. No cleanup close reason applies; it should stay on the normal maintainer review and merge path.

## What This Changes

The PR normalizes cron run-log `jobId` values with read-side validation before computing the write key, inserting the SQLite row, and pruning, plus regression coverage for padded and unsafe IDs.

## Best Possible Solution

Land this narrow write-side normalization with the regression tests after normal maintainer review and required checks.

## Reproduction Assessment

Yes, source-reproducible. Current main writes raw `entry.jobId` but trims and validates jobId before exact-key SQLite reads, and the PR body supplies real SQLite proof plus a negative control.

## Solution Assessment

Yes. Normalizing once in `appendCronRunLog` before the queue key, insert, and prune is the narrowest owner-boundary fix; caller-only normalization would miss direct store users, and SQLite-only normalization would leave the write queue/prune keys inconsistent.
