## Summary

Keep open: current main still has asymmetric cron run-log jobId handling, and this PR is a narrow bug fix that aligns the write path with existing read-side validation. I found no blocking correctness or security findings.

## What This Changes

The PR normalizes cron run-log jobId values with read-side validation before writing SQLite rows and adds regression tests for padded and unsafe IDs.

## Best Possible Solution

Land this narrow write-side normalization with its regression tests after normal maintainer review and required checks.

## Reproduction Assessment

Yes, source-reproducible. Current main writes raw entry.jobId but trims and validates jobId before reads, and the PR body includes real SQLite proof plus a negative control; I did not run local tests because this review is read-only.

## Solution Assessment

Yes. Normalizing once in appendCronRunLog before the write key, insert, and prune is the narrowest owner-boundary fix; caller-level normalization would miss direct store users and SQLite-store-only normalization would leave queue keys inconsistent.
