## Summary

Keep open for normal maintainer review: current main and the latest release still have the write/read asymmetry, while this PR applies the existing read-side jobId normalization at the write boundary with focused tests and sufficient real SQLite proof.

## What This Changes

The PR normalizes cron run-log job IDs inside `appendCronRunLog` before write-keying, insertion, and pruning, then adds regression tests for padded and unsafe IDs.

## Best Possible Solution

Land the narrow write-side normalization with its regression coverage after normal maintainer review and required checks.

## Reproduction Assessment

Yes. Current main writes raw `entry.jobId` in `appendCronRunLog` while sync and paged reads trim and validate before exact job-id lookup; the PR body also includes real SQLite proof and a negative control.

## Solution Assessment

Yes. Normalizing once inside `appendCronRunLog` before queue keying, insertion, and pruning is the narrow owner-boundary fix; caller-level normalization would miss direct store users and SQLite-only normalization would leave write queue keys inconsistent.
