## Summary

Keep open: current main still has restart cooldowns, supervisor respawn settings, channel-level health recovery, and doctor-owned config repair, but not a gateway-level graduated crash-recovery ladder. The requested persistent counters, escalation tiers, destructive repair choices, maintenance mode, and alerting policy need maintainer product/ops design; the related open pull request only covers a first-rung startup throttle.

## What This Changes

Add a configurable graduated gateway crash-recovery ladder that tracks repeated failures and escalates through restart, config rollback, state cleanup, and operator alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new reliability capability request. Source inspection gives high confidence that current main lacks a `gateway.crashRecovery` ladder or equivalent process-level recovery controller.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native recovery likely belongs around service/gateway/doctor primitives, but the exact tiers, defaults, persistence, rollback scope, and alert policy need maintainer design review.
