## Summary

Keep open: current main has supervisor restart limits, SIGUSR1 cooldown/deferral, channel-level restart caps, and doctor-owned config repair, but it still lacks a gateway-level graduated crash-recovery ladder with persistent failure state, configurable tiers, state cleanup, rollback, maintenance mode, and alert policy.

## What This Changes

The issue requests a configurable graduated gateway crash-recovery ladder that escalates after repeated gateway failures from restart/backoff into config recovery, state repair, version/package recovery, and user alerting.

## Best Possible Solution

Design an opt-in, supervisor-aware recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new gateway recovery capability request, not a report that current documented behavior is broken. Source inspection gives high-confidence evidence that current main lacks the requested `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. The related startup-throttle PR at https://github.com/openclaw/openclaw/pull/79181 is a useful first rung, but the full ladder needs policy decisions around thresholds, rollback scope, state deletion, alerts, and opt-in behavior.
