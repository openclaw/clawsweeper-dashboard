## Summary

Keep open: current main still has supervisor respawn, SIGUSR1 restart cooldown/deferral, channel-level health restarts, and doctor-owned repair, but no graduated gateway crash-recovery ladder. The fresh production confirmation and related open first-rung throttle PR reinforce that this is a reliability/product design request, not an auto-close candidate.

## What This Changes

The issue asks for a configurable gateway crash-recovery ladder that tracks repeated process failures and escalates beyond flat supervisor respawn into repair, rollback, state cleanup, and operator alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request, not a broken existing contract. Source inspection on current main gives high confidence that no `gateway.crashRecovery` ladder exists.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native recovery likely belongs around supervisor, gateway, and doctor primitives, but the exact tiers, defaults, rollback scope, and alert policy need design review; https://github.com/openclaw/openclaw/pull/79181 is only a possible first rung.
