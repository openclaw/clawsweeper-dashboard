## Summary

Keep open: current main still has basic supervisor restart guards, in-process SIGUSR1 cooldown/coalescing, channel-level health restarts, and doctor-owned repair, but no gateway-level graduated crash recovery ladder. The reporter's latest comment confirms production impact while explicitly deferring the counter, escalation, maintenance-mode, and alert semantics to maintainer design.

## What This Changes

Request a configurable gateway crash-recovery ladder that tracks repeated gateway failures and escalates through restart, config or state repair, and operator alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive steps behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection and the reporter's production update give high confidence that current main lacks a gateway.crashRecovery ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. A native recovery controller likely belongs around gateway service and doctor primitives, but thresholds, persistence, rollback scope, halt behavior, and alerts need reliability design review.
