## Summary

Keep open: current main has service restarts, channel-level health restarts, doctor-owned last-known-good config repair, and update-specific restart checks, but no gateway-level crash-loop recovery ladder. This is a new core gateway/service recovery feature with safety and product-policy decisions, not a strict bug fix or already-implemented cleanup.

## What This Changes

The issue requests a configurable gateway crash-recovery ladder that escalates from restarts to config restore, state cleanup, and user alerting.

## Best Possible Solution

Design an opt-in core gateway/daemon recovery policy that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, preserves consent around destructive state/package rollback, and documents the config and alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source and docs inspection does give high-confidence evidence that current main lacks the requested `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native gateway crash-loop escalation belongs in the core gateway/service area, but the exact tiers and defaults need design review because they cross daemon supervision, doctor repair, update rollback, state deletion, and channel alerting.
