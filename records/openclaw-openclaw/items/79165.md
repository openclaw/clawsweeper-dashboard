## Summary

Keep open: current main still lacks a graduated gateway crash-recovery ladder. It has supervisor respawn guards, SIGUSR1 cooldown/coalescing, channel-level health restarts, and doctor/config recovery pieces, but not the requested opt-in crash-loop policy with persistent process-failure state, escalating recovery tiers, and operator alerting.

## What This Changes

Request an opt-in gateway crash-recovery ladder that tracks repeated gateway process failures and escalates from restart/backoff to config or state repair and operator notification.

## Best Possible Solution

Design a recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new reliability capability request, not a broken documented behavior. Source inspection gives high confidence that current main lacks a `gateway.crashRecovery` ladder while retaining narrower related safeguards.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native recovery is plausible, but the exact counter window, persistence model, destructive tiers, defaults, alert path, and interaction with the open first-rung throttle PR need design review.
