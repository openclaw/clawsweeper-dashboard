## Summary

Keep open: current main still has supervisor respawn, SIGUSR1 cooldown/coalescing, channel-level restart limits, and doctor-owned repair, but no gateway-level graduated crash-recovery ladder. The remaining request is a reliability/product design change, not a narrow bug fix.

## What This Changes

Request adds an opt-in gateway crash-recovery ladder with crash counters, maintenance mode, escalating repair tiers, and operator alerts.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection gives high-confidence evidence that current main lacks `gateway.crashRecovery` configuration, persistent process crash counters, and graduated escalation actions.

## Solution Assessment

Unclear until maintainers choose the reliability boundary. The proposed ladder is plausible, but the exact tiers, defaults, destructive-action consent, persistence window, and alert delivery need maintainer design review before implementation.
