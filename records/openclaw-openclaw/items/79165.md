## Summary

Keep open: current main still has supervisor respawn, SIGUSR1 restart cooldown/deferral, channel-level health restarts, restart sentinels, and doctor-owned repair, but no gateway-level graduated crash-recovery ladder. The requested escalation tiers add recovery policy, persistent state, config/state rollback behavior, and alerting semantics that need maintainer product and ops design.

## What This Changes

The issue requests a configurable gateway crash-recovery ladder that escalates from restart through config/state repair and user alerting after repeated process failures.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection on current main gives high-confidence evidence that OpenClaw lacks a gateway.crashRecovery ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native recovery likely belongs around service/gateway/doctor primitives, but the exact tiers, defaults, rollback scope, and alert policy need maintainer design review.
