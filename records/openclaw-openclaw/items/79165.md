## Summary

Keep open: current main still has supervisor restarts, SIGUSR1 cooldown/deferral, channel-level health restarts, and doctor/config repair primitives, but no gateway-level graduated crash-recovery ladder. The related open pull request only covers a first-rung startup throttle, while this request adds persistent recovery policy, config/state repair or rollback, and alert behavior that need maintainer product and ops design.

## What This Changes

The issue requests a configurable graduated gateway crash-recovery ladder that escalates beyond supervisor restarts into config/state repair, rollback, and alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive tiers behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection gives high-confidence evidence that current main lacks a `gateway.crashRecovery` ladder or equivalent graduated process-level controller.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native gateway recovery likely belongs around supervisor, gateway, and doctor primitives, but the exact tiers, defaults, rollback scope, and alert policy need maintainer design review.
