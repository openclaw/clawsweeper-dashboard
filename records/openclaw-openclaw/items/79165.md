## Summary

Keep open: current main has supervisor respawn/backoff, SIGUSR1 restart cooldown/deferral, channel-level health restarts, and targeted config/doctor recovery, but no persistent gateway-level crash recovery ladder. The requested escalation tiers add new config, state, destructive repair/rollback choices, and alert behavior that need maintainer product and ops design.

## What This Changes

Request a native configurable gateway crash-recovery ladder that tracks consecutive failures and escalates from restart to config or state repair and user alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses audited doctor/repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection gives high-confidence evidence that current main lacks a `gateway.crashRecovery` config surface or persistent crash-ladder controller.

## Solution Assessment

Unclear until maintainers choose the product boundary. A native controller likely belongs around gateway service and doctor primitives, but the exact tiers, defaults, rollback scope, maintenance mode, and alert policy need design review.
