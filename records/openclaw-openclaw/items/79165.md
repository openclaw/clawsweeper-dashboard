## Summary

Keep open: current main still has basic supervisor restarts, SIGUSR1 restart cooldown, channel-health recovery, and doctor-owned config repair, but no persistent gateway crash-loop recovery ladder with configurable thresholds, rollback/state cleanup tiers, maintenance mode, or alerting. The request is a new recovery capability and needs maintainer product/ops design rather than conservative cleanup closure.

## What This Changes

Add a configurable gateway crash recovery ladder that tracks repeated gateway failures and escalates from restart to config rollback, state cleanup, and user alerting.

## Best Possible Solution

Design an opt-in recovery controller outside the crashing process, persist crash-loop state in the gateway state area, reuse doctor-safe repair primitives, gate destructive tiers behind explicit policy, and document maintenance-mode and alert semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request, not broken behavior against an existing contract. Source inspection gives high-confidence evidence that current main lacks a gateway.crashRecovery ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native gateway recovery likely belongs around service/gateway/doctor primitives, but the exact tiers, defaults, rollback scope, maintenance behavior, and alert policy need maintainer design review.
