## Summary

Keep open: current main still has supervisor respawn, SIGUSR1 restart cooldown/coalescing, channel-level health restarts, and doctor-owned repair, but no gateway-level graduated crash-recovery ladder. The latest reporter comment explicitly defers implementation to maintainer reliability design, so this is not a cleanup-close candidate.

## What This Changes

Request a native gateway crash-loop recovery ladder with thresholds, maintenance mode, escalating repair actions, and operator alerts.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new reliability capability request, not broken documented behavior. Source inspection gives high-confidence evidence that current main lacks the requested gateway.crashRecovery ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. A maintainable solution should be designed around service, gateway, and doctor primitives with explicit policy for rollback, state deletion, alerts, and maintenance mode.
