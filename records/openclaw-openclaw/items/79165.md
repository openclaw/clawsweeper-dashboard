## Summary

Keep open: current main still has flat supervisor respawn, SIGUSR1 cooldown/coalescing, per-channel health restarts, and doctor-owned repair, but no native graduated gateway crash recovery ladder. The requested behavior adds new config, persistent recovery state, rollback or cleanup actions, maintenance-mode semantics, and alert policy that need maintainer product and ops design.

## What This Changes

The issue requests a graduated gateway crash recovery ladder with counters, escalation tiers, maintenance mode, and operator alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new gateway recovery capability request. Source inspection gives high-confidence evidence that current main lacks a `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. A native recovery controller likely belongs around gateway service and doctor primitives, but the tiers, defaults, rollback scope, persistence rules, and alert policy need maintainer design review.
