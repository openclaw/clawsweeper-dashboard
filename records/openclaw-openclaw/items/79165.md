## Summary

Keep open: current main has supervisor restarts, SIGUSR1 restart cooldown/deferral, channel-level health recovery, and doctor-owned config repair, but it still lacks the requested gateway-level graduated crash recovery ladder. The related open pull request https://github.com/openclaw/openclaw/pull/79181 only covers a first-rung rapid-start throttle and does not implement config rollback, state cleanup, alerting, maintenance mode, or the proposed config surface.

## What This Changes

The issue requests an opt-in gateway crash recovery ladder that tracks repeated failures and escalates through restart, config restore, state cleanup, and user alerting.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection gives high-confidence evidence that current main lacks a `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native gateway recovery likely belongs around service, gateway, and doctor primitives, but the exact tiers, defaults, rollback scope, and alert policy need maintainer design review.
