## Summary

Keep open: current main still has supervisor respawn, SIGUSR1 cooldown/coalescing, channel-level restart limits, and config last-known-good repair primitives, but no gateway-level graduated crash recovery ladder. The requested behavior adds new configurable recovery policy, persistent crash-loop state, destructive repair choices, and alert semantics that need maintainer product/ops design rather than conservative cleanup closure.

## What This Changes

Request a configurable gateway crash-recovery ladder that tracks repeated process failures and escalates through restart, config restore, state cleanup, and operator notification.

## Best Possible Solution

Design an opt-in gateway recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection gives high-confidence evidence that current main lacks a `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native gateway recovery likely belongs around service/gateway/doctor primitives, but the exact tiers, defaults, rollback scope, and alert policy need maintainer design review.
