## Summary

Keep open: current main still has supervisor respawn/rate-limit guards, SIGUSR1 restart cooldown/coalescing, channel-level health restarts, and doctor-owned repair, but it does not provide the requested gateway-level graduated crash recovery ladder. The remaining work is a feature and reliability policy design, not a narrow bug fix.

## What This Changes

The issue requests an opt-in gateway crash recovery controller with persistent failure counts, escalation thresholds, config/state repair actions, maintenance mode, and operator alerting.

## Best Possible Solution

Design a recovery controller outside the crashing process that persists crash-loop state, reuses doctor-safe repair primitives, gates destructive config/state rollback behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new reliability capability request. Source inspection gives high-confidence evidence that current main lacks the requested `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product and ops boundary. A native solution likely belongs around gateway service supervision and doctor repair primitives, but the tiers, defaults, persistence, rollback scope, and alert policy need design review.
