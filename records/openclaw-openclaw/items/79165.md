## Summary

Keep open: current main has supervisor restarts, SIGUSR1 restart cooldown/deferral, channel-level health restarts, and doctor-owned repair, but no gateway-level graduated crash recovery ladder. The requested tiers add configurable recovery policy, persistent state, destructive repair/rollback choices, and alert behavior that need maintainer product and ops design.

## What This Changes

Request adds an opt-in gateway crash-recovery ladder that escalates from restart to config restore, state cleanup, user alerting, and possible reinstall or rollback behavior.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance-mode plus alert behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new recovery capability request. Source inspection gives high-confidence evidence that current main lacks a `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native gateway recovery likely belongs around service/gateway/doctor primitives, but the exact tiers, defaults, rollback scope, and alert policy need maintainer design review.
