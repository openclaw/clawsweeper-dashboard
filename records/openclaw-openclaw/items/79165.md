## Summary

Keep open: current main still has basic supervisor respawn, SIGUSR1 restart cooldown/coalescing, and per-channel health restarts, but not a native graduated gateway crash-recovery ladder. The reporter also confirmed production impact while explicitly deferring the policy shape to maintainers, so this is a product/ops design item rather than a stale cleanup close.

## What This Changes

The issue requests a configurable gateway crash-loop recovery ladder with counters, maintenance mode, escalating config/state repair actions, and operator alerts.

## Best Possible Solution

Design an opt-in recovery controller that persists crash-loop state outside the crashing process, reuses doctor-safe repair primitives, gates destructive actions behind explicit policy, and documents maintenance mode plus alert semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new recovery capability. Source inspection gives high confidence that current main lacks the requested `gateway.crashRecovery` ladder.

## Solution Assessment

Unclear until maintainers choose the product boundary. Native recovery likely belongs around service/gateway/doctor primitives, but tiers, defaults, persistence, destructive actions, and alert policy need owner design review.
