## Summary

Keep open: current main has structured Gateway protocol-mismatch diagnostics and Sparkle wiring, but I did not find a desktop path that turns an old macOS app plus newer Gateway into a clear in-app update prompt or immediate Sparkle check. The related closed report at https://github.com/openclaw/openclaw/issues/80133 covered a stale launchd entrypoint, while this report is a stale desktop bundle after a Gateway/npm update.

## What This Changes

Request clear desktop guidance or coordinated update behavior when a newer Gateway rejects an older macOS app with a protocol mismatch.

## Best Possible Solution

Add a desktop-owned protocol-mismatch update-required flow that detects an older app from handshake details, triggers Sparkle when available, and surfaces a clear update action without weakening protocol negotiation.

## Reproduction Assessment

Yes, with source-level confidence: a client whose max protocol is below the current Gateway protocol floor is rejected, and the attached logs describe that exact old-app/new-Gateway skew. I did not run a packaged macOS before/after reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the ownership boundary. A desktop-side update-required prompt with a Sparkle check looks safer than making the Gateway accept incompatible clients, because protocol bumps protect the wire contract.
