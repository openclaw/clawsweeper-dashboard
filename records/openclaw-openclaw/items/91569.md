## Summary

Keep open: current main still rejects old desktop/control clients with structured protocol-mismatch details, but the macOS app does not turn that mismatch into an immediate Sparkle/appcast update prompt or coordinated desktop update recovery path.

## What This Changes

The issue requests actionable desktop update guidance when an npm-updated Gateway becomes incompatible with an older macOS desktop app.

## Best Possible Solution

Add a desktop-owned update-required recovery flow that detects protocol mismatch or app/Gateway version skew, triggers an immediate Sparkle check or download action, and keeps Gateway protocol negotiation strict.

## Reproduction Assessment

Yes, at source/log level: a v2026.4.26-style macOS client advertises protocol 3, while current Gateway requires protocol 4 and rejects that range with PROTOCOL_MISMATCH. The reporter's logs show old desktop/new Gateway skew; I did not replay a packaged macOS scenario.

## Solution Assessment

Unclear until maintainer choice: the issue lists Gateway-side, desktop-side, and coordinated-update shapes. Source evidence suggests a desktop-side update-required prompt is safer than relaxing Gateway protocol compatibility.
