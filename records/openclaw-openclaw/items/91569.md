## Summary

Keep open: current main and v2026.6.9 still strictly reject stale desktop protocol clients, and the macOS app still lacks a typed PROTOCOL_MISMATCH recovery path that triggers Sparkle or a clear update prompt. The related cloud Control UI issue and PR cover browser service-worker recovery, not the desktop app/Sparkle skew reported here.

## What This Changes

The issue asks for desktop/gateway version-skew recovery so an old macOS desktop app gets actionable update guidance instead of a generic protocol mismatch after the gateway updates.

## Best Possible Solution

Add a desktop-owned update-required recovery flow for PROTOCOL_MISMATCH that can launch Sparkle or a clear download action, while keeping Gateway protocol negotiation strict.

## Reproduction Assessment

Yes, source-level: stale protocol clients are rejected by the Gateway, and the current macOS/shared Swift path has no typed PROTOCOL_MISMATCH branch that triggers an update prompt. I did not run a packaged old-desktop/new-Gateway macOS scenario.

## Solution Assessment

Unclear until maintainers choose the owner boundary. Source review suggests a desktop update-check/prompt is safer than relaxing Gateway protocol compatibility, while Gateway-side structured UX and coordinated update signaling remain plausible alternatives.
