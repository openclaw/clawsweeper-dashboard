## Summary

Keep open: current main still rejects old desktop/control clients with protocol-mismatch details, but there is no desktop recovery path that forces a Sparkle/appcast check or presents an update-required prompt after a Gateway npm update.

## What This Changes

The issue asks OpenClaw to prevent or clearly recover from desktop/Gateway version skew after Gateway self-update.

## Best Possible Solution

Add a desktop-owned update-required recovery flow that detects protocol mismatch/version skew, triggers an immediate Sparkle check or download action, and keeps Gateway protocol negotiation strict.

## Reproduction Assessment

Yes, at source/log level: a non-probe desktop client whose max protocol is below the Gateway protocol floor reaches the protocol-mismatch rejection path, and the issue logs show old desktop/new Gateway skew. I did not run a packaged macOS before/after scenario.

## Solution Assessment

Unclear until maintainer choice: the issue lists Gateway-side, desktop-side, and coordinated-update shapes. Source evidence suggests a desktop-side update-required prompt is safer than relaxing Gateway protocol compatibility.
