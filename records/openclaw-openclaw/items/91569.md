## Summary

Keep open: current main and v2026.6.8 still strictly reject stale desktop/WebChat protocol clients, and the macOS app only maps the failure to a plain text “update the app/gateway” message. I found no current path that forces an immediate Sparkle/appcast check or presents an actionable update prompt after a Gateway npm update.

## What This Changes

The issue asks OpenClaw to recover clearly when an npm-updated Gateway and an older desktop app fall out of sync instead of leaving users with a generic protocol mismatch.

## Best Possible Solution

Add a desktop-owned update-required recovery flow that handles protocol mismatch/version skew by launching Sparkle or a clear download action, while keeping Gateway protocol negotiation strict.

## Reproduction Assessment

Yes at source/log level: a non-probe desktop client whose max protocol is below the Gateway protocol floor reaches the protocol-mismatch rejection path, and the reporter supplied old-desktop/new-Gateway logs. I did not run a packaged macOS before/after scenario.

## Solution Assessment

Unclear until a maintainer chooses the product boundary. Source review suggests a desktop update-check/prompt is safer than relaxing Gateway protocol compatibility, but the issue lists Gateway-side, desktop-side, and coordinated-update options.
