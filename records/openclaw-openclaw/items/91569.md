## Summary

Keep open: current main still hard-rejects incompatible desktop/control clients with protocol-mismatch details, and the macOS Sparkle updater is not tied to that failure path or to Gateway npm updates, so the reported desktop update prompt gap remains.

## What This Changes

The issue asks for a clear desktop update-required recovery path when a Gateway or CLI update leaves the macOS app on an older incompatible protocol version.

## Best Possible Solution

Add a desktop-owned update-required recovery flow that detects protocol mismatch/version skew, triggers an immediate Sparkle/appcast check or download action, and keeps Gateway protocol negotiation strict.

## Reproduction Assessment

Yes, at source/log level: a non-probe desktop/control client whose max protocol is below the Gateway protocol floor reaches the PROTOCOL_MISMATCH rejection path, and the issue logs show old desktop/new Gateway skew. I did not run a packaged macOS before/after scenario.

## Solution Assessment

No, the current behavior is not the best end state because it leaves users with only a generic mismatch. The safer direction is a desktop-side update-required prompt or structured update guidance, not weakening the Gateway protocol floor.
