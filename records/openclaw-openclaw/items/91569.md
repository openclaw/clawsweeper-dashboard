## Summary

Keep open: current main still has strict protocol rejection plus Sparkle/manual update plumbing, but no path converts an old macOS app plus newer Gateway protocol mismatch into an immediate in-app update prompt. The closed appcast issue and the open node-skew issue are related, but neither covers this desktop recovery flow.

## What This Changes

The issue asks for stale macOS desktop app recovery when a Gateway/npm update leaves the app on an older protocol and desktop chat only reports a generic protocol mismatch.

## Best Possible Solution

Add a desktop-owned stale-client recovery flow that detects protocol mismatch, triggers an immediate Sparkle/appcast check or download action when the app is older, and keeps Gateway protocol negotiation strict.

## Reproduction Assessment

Yes. A source-level repro is clear: an older macOS app advertising a protocol range below the current Gateway protocol reaches the protocol-mismatch rejection path, and the issue logs show that old-app/new-Gateway skew.

## Solution Assessment

No. Closing or auto-fixing this is not the best path yet; a desktop-owned update-required prompt looks safer than weakening Gateway protocol compatibility, but maintainers need to choose the ownership boundary.
