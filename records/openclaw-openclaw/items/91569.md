## Summary

Keep open: current main still rejects incompatible desktop/control clients with a protocol-mismatch error and has Sparkle update plumbing, but I did not find a desktop recovery path that forces an appcast check or presents an update-required prompt after a Gateway npm update.

## What This Changes

The issue requests clearer recovery when an updated Gateway rejects an older macOS desktop app with a protocol mismatch instead of prompting the app update.

## Best Possible Solution

Add a desktop-owned update-required recovery flow that detects protocol mismatch/version skew, triggers an immediate Sparkle check or download action, and keeps Gateway protocol negotiation strict.

## Reproduction Assessment

Yes, at source/log level: a non-probe client whose max protocol is below the Gateway protocol floor reaches the protocol-mismatch rejection path, and the issue logs show old desktop/new Gateway skew. I did not run a packaged macOS before/after scenario.

## Solution Assessment

No, closing or auto-fixing this is not the best path yet. A desktop-side update-required prompt with an immediate Sparkle/appcast check looks safer than weakening Gateway protocol compatibility, but maintainers need to choose that boundary.
