## Summary

Keep open: current main and the latest release still give the macOS app only generic protocol-mismatch guidance, with no path that immediately checks Sparkle or presents a targeted update action when the desktop bundle is the stale side. The detailed before/after logs make the reported upgrade failure credible, and the related appcast incident does not supersede it because that incident involved a stale published feed while this report says a current update was already available.

## What This Changes

Provide actionable update guidance when an outdated macOS desktop app connects to a newer Gateway after a CLI/npm update.

## Best Possible Solution

Propagate typed protocol/version incompatibility details into the macOS connection layer, determine whether the installed app is the stale side, and then offer an immediate Sparkle check with explicit recovery text while preserving generic guidance for other mismatch directions.

## Reproduction Assessment

Yes at source level and from the reporter’s detailed before/after logs: an old desktop bundle against a newer Gateway reaches generic mismatch handling and works after manual app replacement. A live current-main macOS run remains necessary to verify the exact UI and Sparkle lifecycle behavior.

## Solution Assessment

Partly: an immediate Sparkle check is the narrowest likely recovery when the app is stale, but it should be driven by typed mismatch direction rather than generic string parsing or unconditional coupling between npm and desktop updates.
