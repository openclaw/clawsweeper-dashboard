## Summary

Keep open: current main still rejects stale desktop/control clients with a protocol-mismatch response, and the macOS app has Sparkle plumbing but no recovery path that turns this skew into an immediate update prompt or appcast check.

## What This Changes

The issue requests clear desktop recovery when an npm-updated Gateway leaves an older macOS app speaking an incompatible Gateway protocol.

## Best Possible Solution

Add a desktop-owned update-required recovery flow that maps protocol mismatch/version skew to an immediate Sparkle appcast check or download action while keeping Gateway protocol negotiation strict.

## Reproduction Assessment

Yes, at source/log level: an older non-probe desktop/control client whose protocol range does not include the Gateway protocol reaches the protocol-mismatch rejection path, and the issue logs show the old-app/new-Gateway skew. I did not run a packaged macOS before/after scenario.

## Solution Assessment

No, closing or auto-fixing is not the best path yet. A desktop-side update-required prompt with an immediate Sparkle/appcast check looks safer than weakening Gateway protocol compatibility, but maintainers need to choose that boundary.
