## Summary

Keep open: the PR is a useful targeted fix for misleading config restart notices, but the current head regressed the completed-restart path that earlier branch state and the PR body still claim to handle. A manual config restart can now wake the session with `Gateway restart required (...)` after the restart has already completed, and the supplied terminal proof is stale for the current head.

## What This Changes

The PR adds `stats.requiresRestart` to config apply/patch restart sentinel payloads, formats marked config-write sentinels as restart-required, and adds focused gateway/restart-sentinel regression tests.

## Best Possible Solution

Repair current head so pending config writes and completed startup-consumed restart sentinels use distinct wording, then refresh the PR body/proof against the exact head and rerun the focused gateway/restart-sentinel checks.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main formats successful `config-patch` sentinels as `Gateway restart config-patch ok (...)`, and current PR head formats restart-required sentinels as `required` even when startup consumes them after restart. I did not run a live gateway restart in this read-only review.

## Solution Assessment

No. Adding restart-required metadata at the config write/formatter boundary is the right layer, but current head is too narrow because it no longer distinguishes pending restart-required notices from completed startup wake notices.
