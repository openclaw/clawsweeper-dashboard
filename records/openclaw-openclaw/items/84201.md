## Summary

Close as a duplicate of the open macOS LaunchAgent PATH policy issue at https://github.com/openclaw/openclaw/issues/59665. Current main still shows the reported service-env behavior, but the remaining decision belongs in that canonical thread rather than a separate duplicate issue.

## What This Changes

The issue reports that generated gateway service environment files overwrite PATH without including the active openclaw binary directory, breaking gateway-spawned CLI calls for npm-global and other non-Homebrew installs.

## Best Possible Solution

Use https://github.com/openclaw/openclaw/issues/59665 as the canonical place to choose the macOS service PATH contract, with this npm-global evidence informing a narrow, sanitized active-CLI-bin option if maintainers accept it.

## Reproduction Assessment

Yes, source-level. Current main generates macOS service PATH from a canonical list plus the selected Node runtime bin, and the reporter supplied concrete v2026.5.18 service-env output showing the npm-global openclaw bin directory missing.

## Solution Assessment

No, not as a standalone issue. The suggested active-binary-dir fix is a plausible narrower alternative to full login-shell PATH capture, but it should be decided in the canonical LaunchAgent PATH policy issue because it changes the service executable lookup boundary.
