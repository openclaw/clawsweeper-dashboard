## Summary

Keep open: the report matches current main. The gateway install path still builds a durable macOS service PATH from explicit runtime directories plus system paths, while the install plan only supplies the selected Node runtime bin directory, so an npm-global `openclaw` shim under a user prefix can be excluded from the LaunchAgent environment.

## What This Changes

The issue reports that `openclaw gateway install` writes a LaunchAgent service env `PATH` that omits the active `openclaw` binary directory for npm-global and similar installs.

## Best Possible Solution

Teach the gateway install plan to add the active `openclaw` command's bin directory as an explicit service PATH entry, while preserving the current minimal macOS PATH policy and avoiding a full ambient PATH merge.

## Reproduction Assessment

Yes, source-reproducible: current main omits macOS user bin defaults and only prepends the selected Node runtime directory, while the launchd wrapper sources the generated `PATH`. A live npm-global macOS service install was not run during this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to carry the installer OpenClaw CLI bin directory into `extraPathDirs` or an equivalent explicit service PATH input. Broadly preserving `process.env.PATH` would be riskier because the current audit path intentionally rejects version-manager/package-manager PATH drift.
