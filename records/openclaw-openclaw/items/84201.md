## Summary

Keep open: current `main` still matches the reported service PATH behavior, and two open linked PRs already target the fix, including a mergeable PR with closing syntax.

## What This Changes

The issue reports that `openclaw gateway install` writes a macOS service `PATH` that omits the active `openclaw` CLI bin directory for npm-global and similar prefix installs.

## Best Possible Solution

Review and land one linked fix PR that explicitly adds the active OpenClaw CLI bin directory to the managed service PATH without broad ambient PATH merging, then close this issue as implemented.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current `main` builds the managed macOS service PATH from the selected Node bin plus minimal system dirs, and the launchd wrapper sources that PATH before launching the gateway. I did not run a live macOS npm-global LaunchAgent install in this read-only review.

## Solution Assessment

Yes, the best fix is the linked-PR direction: carry the active `openclaw` command bin directory into the explicit service PATH inputs. Broadly merging the installer's full ambient PATH would weaken the existing minimal service PATH policy and is not needed for this bug.
