## Summary

Keep open. Current main and latest release v2026.6.9 still deliberately split the behavior: invalid external edits are skipped during reload, but the same invalid on-disk config fails startup before channels come back; fixing that requires a maintainer decision on last-good restore policy because related reports show stale restores can discard intentional config.

## What This Changes

This issue requests managed Gateway startup and restart recovery so an invalid on-disk `openclaw.json` does not strand channels when a valid last-known-good config exists.

## Best Possible Solution

Define one bounded managed-startup and restart-recovery contract that preserves channel availability without silently discarding intentional config changes, then cover it across startup, reload, doctor, status/notification, and docs.

## Reproduction Assessment

Yes at source level: invalid reload snapshots are skipped while invalid startup snapshots throw before services boot, and current tests pin both behaviors. I did not run the WSL2 npm-global Telegram managed-service scenario.

## Solution Assessment

Unclear. The requested recovery behavior is plausible for availability, but automatic last-good restore is not clearly the best default until maintainers decide how to avoid stale rollback and config-loss regressions.
