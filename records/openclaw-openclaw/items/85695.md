## Summary

Keep open. Current main still preserves the reported gap: invalid external config edits are skipped by hot reload, while startup still fails closed on that same invalid file; recent SecretRef/config-write hardening does not cover direct bad on-disk startup recovery.

## What This Changes

Request managed gateway startup/restart recovery so invalid on-disk `openclaw.json` does not take channels down when a valid last-known-good copy exists.

## Best Possible Solution

Maintainers should choose the fail-closed versus managed-restore policy, then implement bounded startup/restart preflight recovery with audit artifacts, owner notification, and docs/tests.

## Reproduction Assessment

Yes. Source inspection shows invalid hot-reload snapshots are skipped while invalid startup snapshots throw before services boot; I did not run a live WSL2 Telegram managed-service reproduction.

## Solution Assessment

Unclear until maintainers choose the policy. The likely implementation should reuse existing last-known-good primitives, but the default versus opt-in behavior needs an explicit decision.
