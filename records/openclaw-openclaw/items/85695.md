## Summary

Keep open. Current main and v2026.6.8 only partially address the delayed-outage path: CLI service start/restart now preflights invalid config, but Gateway startup still fails closed and hot reload still skips invalid direct edits without restoring last-known-good; choosing automatic restore remains a maintainer policy decision because it conflicts with stale rollback/data-loss risk tracked in https://github.com/openclaw/openclaw/issues/90551.

## What This Changes

The issue asks OpenClaw to prevent an invalid on-disk `openclaw.json` from taking down a managed Gateway on restart by preflighting restarts, recovering from last-known-good when safe, and steering agents toward validated config mutation paths.

## Best Possible Solution

Maintainers should define a single managed-startup/restart recovery contract that preserves channel availability without silently discarding intentional config changes, then implement it with audit artifacts, status/owner notification, and regression coverage across startup, hot reload, CLI lifecycle, and doctor recovery.

## Reproduction Assessment

Yes, at source level: invalid reload snapshots are skipped while invalid startup snapshots throw before services boot, and tests pin both behaviors. I did not run the WSL2/npm-global Telegram managed-service scenario in this read-only review.

## Solution Assessment

Unclear. The requested recovery behavior is plausible for availability, and CLI restart preflight is a useful partial mitigation, but automatic last-good restore is not clearly the best default until maintainers decide how to avoid stale rollback and config-loss regressions.
