## Summary

Keep open. Current main and the latest release still keep the split behavior reported here: invalid direct edits are skipped by hot reload, but the same invalid on-disk config fails closed at startup, and recovery remains doctor-owned. The open related PR https://github.com/openclaw/openclaw/pull/83698 covers rejected RPC writes, not managed startup recovery, while https://github.com/openclaw/openclaw/issues/90551 shows the opposite stale last-good rollback risk that makes this a maintainer policy decision.

## What This Changes

The issue asks for managed gateway startup/restart recovery so an invalid on-disk `openclaw.json` does not take down Telegram when a valid last-good config exists.

## Best Possible Solution

Keep this open until maintainers choose a policy-backed managed startup/restart contract, then implement either documented fail-closed behavior or bounded last-good recovery with audit artifacts, status/owner notification, and regression tests.

## Reproduction Assessment

Yes, at source level: invalid reload snapshots are skipped while invalid startup snapshots throw before services boot. I did not run the WSL2/npm-global Telegram managed-service scenario in this read-only review.

## Solution Assessment

Unclear. The requested recovery behavior is plausible for availability, but current docs/tests intentionally fail closed and the safer default depends on maintainer policy around automatic restore versus doctor-owned repair.
