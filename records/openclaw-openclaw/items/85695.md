## Summary

Keep open. Current main and the latest release still keep the reported split behavior: hot reload skips invalid external edits while startup fails closed on the same invalid file, and the linked open PR only covers rejected RPC writes rather than managed startup recovery. The remaining question is a maintainer policy decision because automatic last-good restore would improve channel availability but overlaps with the opposite stale-rollback risk tracked in https://github.com/openclaw/openclaw/issues/90551.

## What This Changes

The issue requests managed-gateway protection so invalid on-disk `openclaw.json` cannot turn a previously healthy last-known-good runtime into a startup/channel outage.

## Best Possible Solution

Keep this issue open until maintainers choose a policy-backed managed startup/restart contract, then implement either documented fail-closed behavior or bounded last-good recovery with audit artifacts, status/owner notification, and regression tests.

## Reproduction Assessment

Yes, at source level: invalid reload snapshots are skipped while invalid startup snapshots throw before services boot. I did not run the WSL2/npm-global Telegram managed-service scenario in this read-only review.

## Solution Assessment

Unclear. The requested recovery behavior is plausible for availability, but current docs/tests intentionally fail closed and the safest path depends on a maintainer decision about automatic restore versus explicit doctor-owned repair.
