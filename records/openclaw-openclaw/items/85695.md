## Summary

Keep open: current main still preserves the exact gap reported here. Invalid hot-reload snapshots are skipped while the runtime keeps its last accepted config, but startup still fails closed on the same invalid on-disk file and the docs explicitly say last-known-good is not restored automatically.

## What This Changes

The issue requests managed startup and restart preflight recovery so an invalid on-disk `openclaw.json` does not take down channels when a valid last-known-good config exists.

## Best Possible Solution

Maintainers should define the managed-startup recovery policy, then either keep doctor-owned fail-closed behavior as intentional or add bounded last-known-good startup/restart recovery with audit artifacts, owner notification, tests, and docs.

## Reproduction Assessment

Yes, from source and tests: invalid hot-reload snapshots are skipped, while invalid startup snapshots throw before services boot. I did not run a live WSL2/Telegram managed-service reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy. The maintainable path should reuse existing last-known-good recovery primitives, but the default versus opt-in behavior and rollback safety rules need explicit approval.
