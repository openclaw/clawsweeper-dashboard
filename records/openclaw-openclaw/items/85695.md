- `impact:message-loss`: The reported user impact is loss of the Telegram channel served by the failed gateway.

## Summary

Keep open. Current main still has the reported availability gap: hot reload skips an invalid config while keeping the runtime alive, but startup fails closed on the same invalid file, and the current docs/tests explicitly say startup and hot reload do not auto-restore last-known-good.

## What This Changes

The issue requests managed startup/restart protection so an invalid on-disk `openclaw.json` cannot take down the gateway when a valid last-known-good config is available.

## Best Possible Solution

Maintainers should decide the managed-startup policy, then either keep fail-closed doctor-owned repair as documented or add bounded last-known-good startup/restart recovery with tests, audit artifacts, owner notification, and docs.

## Reproduction Assessment

Yes from source inspection, but not from a live run: invalid hot-reload snapshots are skipped while startup invalid snapshots throw before services boot. The existing test suite also pins startup without automatic recovery.

## Solution Assessment

Unclear until maintainers choose the policy. The safest implementation path would reuse existing last-known-good recovery primitives but make the default versus opt-in behavior explicit and covered by startup/restart-preflight tests.
