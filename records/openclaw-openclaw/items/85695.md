## Label Justifications

- `P1`: A bad on-disk config can still block Gateway startup and remove the Telegram recovery channel for affected managed installs.
- `impact:crash-loop`: The central failure mode is Gateway startup failure or repeated managed restart failure on invalid config.
- `impact:message-loss`: The report specifically loses the Telegram channel served by the failed Gateway, cutting off remote message-based repair.

## Summary

Keep open. Current main and the latest release still keep the central split: invalid external edits are skipped while the running gateway stays on the last accepted config, but a later startup still fails closed; CLI restart preflight and doctor recovery only partially mitigate this, and automatic last-good restore needs a maintainer policy decision because of stale rollback risk.

## What This Changes

Request managed gateway startup/restart recovery so an invalid on-disk `openclaw.json` cannot take Telegram or other channel availability down when a valid last-known-good config exists.

## Best Possible Solution

Define one bounded managed-startup/restart recovery contract that preserves channel availability without silently discarding intentional config changes, then implement it with audit artifacts, status or owner notification, and regression coverage across startup, hot reload, CLI lifecycle, and doctor recovery.

## Reproduction Assessment

Yes, at source level: invalid reload snapshots are skipped while invalid startup snapshots throw before services boot, and current tests pin both behaviors. I did not run the WSL2 npm-global Telegram managed-service scenario in this read-only review.

## Solution Assessment

Unclear. The requested recovery behavior is plausible for availability, but automatic last-good restore is not clearly the best default until maintainers decide how to avoid stale rollback and config-loss regressions.
