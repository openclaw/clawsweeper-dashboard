## Summary

Keep open. Current main and v2026.6.9 still intentionally skip invalid hot reload snapshots and fail Gateway startup on invalid config instead of automatically restoring last-known-good; doctor can repair, but that does not preserve a managed Telegram recovery path and the automatic-restore policy conflicts with stale rollback risk.

## What This Changes

The issue asks OpenClaw to prevent an invalid on-disk `openclaw.json` from bricking a managed Gateway on restart by preflighting restart/startup and safely recovering from last-known-good when appropriate.

## Best Possible Solution

Define and implement a bounded managed-startup/restart recovery contract that preserves channel availability without silently discarding newer intentional config, with audit artifacts, status/owner notification, and regression coverage across startup, hot reload, CLI restart, and doctor repair.

## Reproduction Assessment

Yes, at source level: current startup rejects invalid config before services boot, while hot reload skips invalid external edits and leaves the bad file on disk. I did not run the full WSL2/npm-global Telegram managed-service scenario in this read-only review.

## Solution Assessment

Unclear. Automatic last-good restore is a plausible availability fix, but it is not clearly the best default until maintainers define how to avoid stale rollback and config-loss regressions.
