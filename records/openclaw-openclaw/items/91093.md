## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P1] Guard parentSessionKey like spawned lineage:** `src/gateway/sessions-patch.ts:204-213`
  - body: The patch exposes `parentSessionKey` through `sessions.patch`, but accepts it for any store key and lets it be cleared. `spawnedBy` is deliberately limited to subagent/acp sessions and immutable, and spawned-session visibility treats `parentSessionKey` as proof that a requester owns the child. A gateway client could mark an ordinary session as someone else's child or remove a real child link, changing what sandboxed session tools can resolve. Keep this field internal to spawn flows, require the same spawned-lineage key guard, and make existing links immutable like `spawnedBy`.
  - confidence: 0.87
- **[P2] Use real ACP activity for delegated expiry rows:** `src/agents/tools/sessions-list-tool.ts:398-402`
  - body: The delegated `sessions_list` row computes expiry from `entry.acp.lastActivityAt`, but `sessions.list` returns `GatewaySessionRow`s and the projection does not carry ACP metadata; this PR only adds the hub-delegated marker. That means idle expiry shown to agents is based on creation time, so a worker that received follow-ups can appear to expire too early. Include ACP activity in the gateway row or compute the expiry server-side before returning delegated rows.
  - confidence: 0.75
