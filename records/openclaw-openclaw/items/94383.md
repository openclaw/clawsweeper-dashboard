## Summary

Keep open: the generic durable approval backend is useful, but this protected maintainer PR is not merge-ready because the Skill Workshop opt-in lacks the durable owner required for reuse, exposes a raw workspace path in the approval key before hashing, and leaves the plugin permission docs internally inconsistent.

## What This Changes

The PR adds an `allowAlwaysKey` plugin approval contract, persists allow-always decisions in SQLite-backed core plugin state, wires reuse through before_tool_call and node.invoke approval paths, opts Skill Workshop lifecycle approvals into the mechanism, and updates plugin docs/tests.

## Best Possible Solution

Keep the generic SQLite-backed durable approval backend, but fix the built-in Skill Workshop opt-in with a stable non-sensitive owner/scope, add second-call reuse coverage, and align the docs before landing.

## Reproduction Assessment

Yes. Source inspection of the PR head shows durable reuse requires `pluginId`, `toolName`, and `allowAlwaysKey`, while the Skill Workshop path offers `allow-always` with only `toolName` and `allowAlwaysKey`; I did not run tests in this read-only review.

## Solution Assessment

No. The generic backend is the right layer, but this PR needs a stable non-sensitive Skill Workshop scope, second-call reuse coverage, and docs alignment before it is the best merge shape.
