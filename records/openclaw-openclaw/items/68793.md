## Summary

Keep open: current main still logs inbound QQ Bot message content at info level; the raw WebSocket dispatch JSON is debug-only, but the privacy-relevant chat-text exposure remains.

## What This Changes

The issue requests quieter QQ Bot default logs and removal of full inbound chat payloads from info-level logging.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and dispatch payloads behind debug or explicit diagnostics, and add focused regression coverage for inbound logging.

## Reproduction Assessment

Yes, via source inspection on current main: QQ dispatch stores raw ev.content in QueuedMessage.content and handleMessage logs that field at info level. I did not run a live QQ Bot account, so this is source-reproducible rather than a runtime log capture.

## Solution Assessment

Yes for the central issue: a QQ Bot plugin-local logging repair is the narrowest maintainable fix. A global redaction change or new channel-specific verbosity knob is unnecessary unless maintainers choose that policy explicitly.
