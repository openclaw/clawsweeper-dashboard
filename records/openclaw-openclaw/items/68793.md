## Summary

Keep this open. Current main still logs full inbound QQ Bot message text at info level in the normal handler; raw WebSocket dispatch JSON is debug-only now, but the central default-log privacy/noise request remains unresolved.

## What This Changes

The issue requests quieter QQ Bot default logs and no full inbound chat payloads or message bodies at info level.

## Best Possible Solution

Fix this inside the QQ Bot plugin by making info logs metadata-only while keeping raw message bodies and dispatch payloads behind debug or an explicit diagnostic path, with focused regression tests for default info logging.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: QQ dispatch events copy raw `content` into `QueuedMessage.content`, and `handleMessage` logs that value at info.

## Solution Assessment

Yes. A narrow QQ Bot plugin-local logging repair is the best path; changing global logging redaction or adding a broad verbosity knob is unnecessary for the reported leak.
