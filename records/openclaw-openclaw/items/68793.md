## Summary

Keep open: current main still routes QQ Bot inbound chat text into an info-level log message, and the latest shipped release contains the same message-body log. The full raw WebSocket dispatch payload is already debug-only, so the remaining issue is the default info-level exposure of user-visible message content.

## What This Changes

The issue requests quieter QQ Bot defaults and metadata-only info logs instead of full inbound chat payloads.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and payload dumps behind debug or explicit diagnostics, and add focused regression coverage for inbound and urgent-command logging.

## Reproduction Assessment

Yes. Source inspection on current main shows QQ Bot dispatch copies ev.content into QueuedMessage.content for C2C and group messages, and handleMessage logs that value at info level.

## Solution Assessment

Yes, with owner/security review. A QQ Bot plugin-local logging repair is the narrowest maintainable fix; a global redaction layer or new verbosity option is unnecessary unless owners choose a broader policy.
