## Summary

Keep this open: current main and v2026.6.10 still log inbound QQBot message content at info level, while the related merged fixes only covered debug-log sanitization and QQBOT_DEBUG parsing. Because this is a sensitive-content default-logging policy issue, it needs maintainer security/product review before a repair lane.

## What This Changes

The issue asks QQBot to reduce routine log verbosity and stop writing full inbound chat payloads or message bodies at info by default.

## Best Possible Solution

Keep this open and update QQBot inbound processing so info logs contain safe metadata only, with full message bodies available only through explicit debug/verbose behavior and focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction. dispatchEvent copies QQBot payload content into QueuedMessage.content, and startGateway logs that value at info level; no live QQBot gateway was run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy. The narrow fix boundary is clear, but the best permanent shape is metadata-only info logging or debug-only full content, not a new verbosity knob unless maintainers explicitly want more config surface.
