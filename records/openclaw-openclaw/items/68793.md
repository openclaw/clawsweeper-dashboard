## Summary

Keep this open. Current main still logs the full inbound QQ Bot message body at info level; the raw WebSocket dispatch payload is debug-only now, but the default info-log privacy/noise request remains unresolved.

## What This Changes

The issue asks for QQ Bot default logging to demote routine gateway noise and avoid emitting full inbound chat payloads at info level.

## Best Possible Solution

Fix this in the QQ Bot plugin by making default info logs metadata-only, moving full chat bodies and raw payloads to debug or explicit diagnostics, and adding focused tests that fail if info logging receives inbound message text.

## Reproduction Assessment

Yes. Source inspection on current main is enough: QQ dispatch events copy raw `ev.content` into `QueuedMessage.content`, and `handleMessage` logs that value at info level.

## Solution Assessment

Yes. A narrow QQ Bot plugin-local logging repair is the best path; a global redaction change or new channel verbosity knob is unnecessary for the reported default info-log leak.

## Security Review

Status: needs_attention

Summary: The issue is privacy-sensitive because current main can persist inbound QQ Bot chat text in default info-level logs.

Concerns:

- **[medium] Inbound QQ Bot chat content is logged at info:** `extensions/qqbot/src/engine/gateway/gateway.ts:126`
  - body: The normal QQ Bot message handler includes the inbound message body in an info-level log line. With file logs commonly configured at info, user chat text can be persisted unless a custom redaction pattern happens to catch it.
  - confidence: 0.95
- **[low] Urgent command previews may expose raw text:** `extensions/qqbot/src/engine/commands/slash-command-handler.ts:58`
  - body: The urgent slash-command path logs the first 20 characters of raw command content at info, which should be tightened under the same default content-logging policy.
  - confidence: 0.82
