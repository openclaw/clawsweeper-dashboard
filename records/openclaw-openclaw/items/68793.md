## Summary

Keeping this issue open: current main and the latest release still route QQ Bot inbound message bodies into the normal info logger, while the related merged QQBot security PR only sanitized debug-log output. The remaining change is security-sensitive default logging behavior, so it still needs maintainer/security/product direction rather than cleanup closure.

## What This Changes

The issue asks QQ Bot to reduce default info-level verbosity and avoid writing full inbound chat payloads or message bodies to normal logs.

## Best Possible Solution

Keep QQ Bot info logs metadata-only by default, move full content and raw payload detail behind debug or an explicit diagnostic path, and add focused regression coverage for that logging contract.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.6 shows the QQ Bot gateway still interpolates inbound event.content into an info-level log call; I did not run a live QQ Bot gateway session with credentials.

## Solution Assessment

Unclear as a product policy decision. The narrow maintainable direction is clear, but maintainers should choose whether safe metadata remains at info or the whole inbound processing line moves to debug.
