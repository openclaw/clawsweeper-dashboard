## Summary

Keep open: current main still sends full QQ Bot inbound message text through an info-level log when the gateway has the framework logger, while only the raw WebSocket dispatch dump is already debug-only. The latest release snapshot still has the same inbound content log, so the central sensitive-content logging problem remains.

## What This Changes

The issue requests quieter QQ Bot logging and avoiding full chat payloads in info-level gateway logs.

## Best Possible Solution

Change QQ Bot gateway logging to keep info-level messages to metadata such as account, message id, sender id, type, and content length, while keeping full content and raw dispatch payloads behind debug or explicit opt-in logging with regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: any QQ Bot inbound message that reaches `handleMessage` is formatted into `log.info` with `event.content`. I did not live-run QQ Bot in this read-only review.

## Solution Assessment

Not applicable to a patch yet; the requested direction is right, but the maintainable fix should be a narrow owner-reviewed logging-policy change rather than adding a new verbosity knob by default.
