## Summary

Keep this open. Current main still logs the full inbound QQ Bot message body at info level in the normal message handler; the raw WebSocket dispatch payload is now debug-only, but the central default-log privacy/noise request remains unresolved.

## What This Changes

The issue asks for QQ Bot default logs to be less verbose and to avoid writing full inbound chat payloads or message bodies at info level.

## Best Possible Solution

Fix this inside the QQ Bot plugin by making default info logs metadata-only, such as account id, message id, sender or peer id, chat type, and content length, while keeping full message bodies and raw dispatch payloads behind debug or an explicit diagnostic path. Add focused QQ Bot tests that fail if normal info logging includes inbound message text.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: QQ dispatch events copy raw QQ content into `QueuedMessage.content`, and `handleMessage` logs that value at info before any inbound processing or redaction-specific logic can narrow it.

## Solution Assessment

Yes. A narrow QQ Bot plugin-local logging repair is the best fit; changing global logging redaction or adding a broad verbosity knob is unnecessary for the reported default info-log leak.
