## Summary

Keep open: current main still logs QQ Bot inbound message text at info level, and the latest shipped release contains the same default message-body log. The raw WebSocket dispatch payload is already debug-only, so the remaining confirmed problem is default info-level exposure of user-visible chat content.

## What This Changes

The issue asks QQ Bot to reduce default info-level logging and avoid logging full inbound chat payloads or message bodies by default.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and payload dumps behind debug or explicit diagnostics, and add focused regression coverage for inbound and urgent-command logging.

## Reproduction Assessment

Yes. Source inspection on current main shows QQ Bot dispatch copies inbound event content into `QueuedMessage.content`, and `handleMessage` logs that value at info level.

## Solution Assessment

Yes, with owner/security review. A QQ Bot plugin-local logging repair is the narrowest maintainable direction; a global redaction layer or new channel-specific verbosity knob is unnecessary unless owners choose a broader policy.
