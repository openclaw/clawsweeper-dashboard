## Label Justifications

- `P2`: The issue reports a concrete QQ Bot logging privacy bug with limited channel-local blast radius.
- `impact:security`: The affected behavior can persist user chat content in normal gateway logs.

## Summary

Keep open: current main still has a direct QQ Bot path that copies inbound chat text into `QueuedMessage.content` and logs that value at info level. The raw WebSocket dispatch dump is already debug-only, so the remaining confirmed work is the normal inbound message log and related command-preview policy.

## What This Changes

The issue asks QQ Bot to demote routine operational logs and stop writing full inbound chat payloads to info-level logs by default.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and payload JSON behind debug or explicit diagnostics, and add focused regression coverage for inbound and urgent-command logging.

## Reproduction Assessment

Yes. Source inspection on current main shows QQ Bot dispatch copies inbound event content into `QueuedMessage.content`, and `handleMessage` logs that value at info level.

## Solution Assessment

Yes. A QQ Bot plugin-local logging repair is the narrowest maintainable path; a global redaction layer or new channel-specific verbosity knob is not needed for the confirmed raw-body info log.
