## Summary

Keep open: current main still logs inbound QQ Bot message text at info level. The raw WebSocket dispatch payload is now debug-only, but the privacy-relevant chat-content exposure remains and no linked fix PR or duplicate owner item was found.

## What This Changes

The issue asks QQ Bot to reduce routine info-level logging and stop logging full inbound chat payloads by default.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and large dispatch payloads behind debug or explicit diagnostics, and add focused regression coverage for inbound logging.

## Reproduction Assessment

Yes. Source inspection on current main shows QQ dispatch stores raw `ev.content` in `QueuedMessage.content`, and `handleMessage` logs that field at info level.

## Solution Assessment

Yes for the central issue. A QQ Bot plugin-local logging repair is the narrowest maintainable solution; a global redaction change or new channel-specific verbosity knob is unnecessary unless maintainers choose that policy explicitly.
