## Label Justifications

- `P2`: The issue reports a concrete QQ Bot logging privacy bug with limited channel-local blast radius.
- `impact:security`: The affected behavior can persist user chat content in normal logs.

## Summary

Keep open: current main still has a direct QQ Bot path that maps inbound chat text into `QueuedMessage.content` and logs that value at info level. The raw WebSocket dispatch dump is already debug-only, so the remaining confirmed issue is the normal inbound message log.

## What This Changes

Reduce QQ Bot default log verbosity and keep full inbound chat text or dispatch payloads out of normal info-level logs.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and payload dumps behind debug or explicit diagnostics, and add focused regression coverage for inbound and urgent-command logging.

## Reproduction Assessment

Yes. Source inspection on current main shows QQ Bot dispatch copies inbound event content into `QueuedMessage.content`, and `handleMessage` logs that value at info level.

## Solution Assessment

Yes. A QQ Bot plugin-local logging repair is the narrowest maintainable path; a global redaction layer or new channel-specific verbosity knob is not needed for the confirmed raw-body info log.
