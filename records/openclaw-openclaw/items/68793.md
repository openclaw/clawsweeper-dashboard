## Label Justifications

- `P2`: This is a scoped QQBot logging issue with limited blast radius but real sensitive-content exposure risk in operator logs.
- `impact:security`: The issue concerns user chat message bodies being written to default log output, which is a sensitive-data exposure class.

## Summary

Keep this open: current `main` and the latest release still log the inbound QQBot message body through the info logger, while the related merged QQBot logging PR only addressed debug-log sanitization and debug opt-in behavior.

## What This Changes

The issue asks QQBot to reduce default log verbosity and stop writing full inbound chat payloads at info level by default.

## Best Possible Solution

Update the QQBot inbound processing log so default info output contains only safe metadata, or demote the full-content line to debug, with focused regression coverage and no new config knob unless maintainers explicitly choose one.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `startGateway` logs `event.content` through `log.info` for inbound QQBot messages; I did not run a live QQBot gateway in this read-only review.

## Solution Assessment

Unclear. The narrow code boundary is clear, but the best permanent behavior depends on maintainer choice between metadata-only info logs and full debug demotion; adding a channel verbosity knob should be avoided unless maintainers explicitly want new config surface.
