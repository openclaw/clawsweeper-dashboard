## Label Justifications

- `P2`: The remaining leak is scoped to QQ Bot logging but can expose sensitive chat content in normal operator logs.
- `impact:security`: The issue is about sensitive user message bodies being written to default log output.

## Summary

Current main and the latest release still have the info-level QQ Bot inbound log that embeds the full message body, so this issue is still necessary. A prior merged PR improved QQ Bot debug-log sanitization, and the raw WebSocket dispatch payload is already debug-level, but the default info-level chat-body log remains.

## What This Changes

The issue asks for QQ Bot default logging to avoid info-level full chat payloads and demote routine noisy operational logs.

## Best Possible Solution

Keep this issue open and change QQ Bot inbound logging so default info logs never include message bodies, while full bodies and raw dispatch payloads require an explicit debug/verbose path.

## Reproduction Assessment

Yes for a source-level reproduction: current main still formats `event.content` into an info-level QQ Bot log in the inbound message handler. I did not run a live QQ Bot gateway because this read-only review cannot use credentials or create runtime artifacts.

## Solution Assessment

Unclear as a product choice: the narrow fix is clear, but maintainers should choose whether safe metadata remains at info or the whole inbound line moves to debug.
