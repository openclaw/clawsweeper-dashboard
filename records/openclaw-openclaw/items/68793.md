## Summary

Keep open: current main still logs QQ Bot inbound message content at info level; the raw WebSocket dispatch payload is already debug-only, so the remaining confirmed issue is default info-level chat content exposure.

## What This Changes

Request to reduce QQ Bot default info logging and stop emitting full inbound chat payloads at info level.

## Best Possible Solution

Make QQ Bot info logs metadata-only by default, keep raw message text and payload dumps behind debug or explicit diagnostics, and add focused regression coverage for inbound and urgent-command logging.

## Reproduction Assessment

Yes. Source inspection on current main shows QQ Bot dispatch copies inbound event content into `QueuedMessage.content`, and `handleMessage` logs that value at info level.

## Solution Assessment

Yes, pending owner/security policy review. A QQ Bot plugin-local logging repair is the narrowest maintainable path; a global redaction layer or new channel-specific verbosity knob is not necessary unless maintainers choose a broader policy.
