## Summary

Keep this open. Current main still logs the full QQ Bot inbound message body at info level in the normal message handler, so the central privacy/noise request remains unresolved; the raw WebSocket dispatch payload is already debug-only.

## What This Changes

The issue requests reducing QQ Bot plugin default info-level logging so full inbound chat payloads and high-volume WebSocket dispatch data are not written to normal logs.

## Best Possible Solution

Fix this inside the QQ Bot plugin by making default info logs metadata-only, such as account id, message id, sender or peer id, message type, and content length. Keep full message bodies, command text, and raw dispatch payloads behind debug or an explicit diagnostic path, and add focused tests that fail if normal info logging includes inbound message text.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: QQ Bot dispatch maps inbound payload content into `QueuedMessage.content`, and `handleMessage` logs that content with `log.info` under the documented normal file log level.

## Solution Assessment

Yes. A narrow QQ Bot plugin-local logging change is the best way to solve this; changing core logging or adding a new global verbosity knob is unnecessary for the observed leak.
