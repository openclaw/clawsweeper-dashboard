## Summary

Keep open. The non-decoding validator direction is narrow and appropriate, but the current PR still fails lint on the new padding check and lacks inspectable real gateway/WebChat proof.

## What This Changes

The PR replaces `isValidBase64`'s whole-string base64 regex in the gateway attachment parser with a non-decoding character-code loop.

## Best Possible Solution

Land one non-decoding linear validator after fixing lint, adding or confirming focused large-payload regression coverage, and attaching redacted real gateway/WebChat proof for the related attachment-crash bug.

## Reproduction Assessment

No. The related issue has exact shipped gateway logs and the current source path is clear, but a direct 10 MiB regex probe on current main did not reproduce the `RangeError`.

## Solution Assessment

Mostly yes. A non-decoding character scan is the narrow maintainable fix because it removes the full-payload regex without violating the existing estimate-before-decode size guard, but this PR still needs the lint fix and real behavior proof.
