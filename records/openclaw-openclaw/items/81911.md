## Summary

Keep this PR open, but it needs changes before merge: the proposed Buffer round-trip moves a full base64 decode ahead of the gateway size guard, which conflicts with current attachment safety coverage. The claimed stack-overflow repro is also not backed by inspectable real behavior proof.

## What This Changes

This PR replaces `isValidBase64`'s regex with a `Buffer.from(..., "base64")` round-trip in gateway attachment validation.

## Best Possible Solution

Keep validation non-decoding before size enforcement, using a linear character and padding scanner or a carefully shared canonicalizer, then add focused large-payload regression coverage and real gateway proof.

## Reproduction Assessment

No. A direct current-main regex probe with 10 MiB valid and invalid strings did not reproduce the claimed stack overflow, and the PR does not include inspectable commands or runtime output.

## Solution Assessment

No. The Buffer round-trip is not the best fix because it decodes untrusted attachment data before the existing size budget; a non-decoding linear validator would address regex concerns without reopening pre-budget allocation.
