## Summary

Keep this PR open, but it should not merge as-is: the intended gateway base64 hardening is plausible, but the patch decodes every attachment during validation before the existing size guard, reintroducing the memory-pressure path that current code and tests explicitly avoid. The external PR also lacks after-fix real behavior proof.

## What This Changes

The PR replaces gateway attachment base64 regex validation with a Buffer decode/re-encode round trip in `src/gateway/chat-attachments.ts`.

## Best Possible Solution

Use a non-decoding linear validator, or the existing `canonicalizeBase64` helper, so large valid base64 can be validated without regex recursion risk while preserving oversized-before-decode enforcement and regression coverage.

## Reproduction Assessment

For the claimed stack overflow, no: the PR gives no concrete payload, log, or artifact. For the introduced regression, yes from source: current tests assert oversized base64 is rejected without `Buffer.from(..., 'base64')`, and the diff calls `Buffer.from` during validation.

## Solution Assessment

No. A Buffer round trip is not the best fix because it decodes before `maxBytes`; the safer path is a non-decoding linear scan that preserves the existing security contract.
