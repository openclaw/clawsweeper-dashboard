## Summary

Keep open. Current main still preserves mixed-case Matrix thread IDs in the Matrix route/send helpers, but the shared session-recording and store-normalization path lowercases the full session key before persistence, so the reported split between mixed-case runtime routing and lowercased stored session keys remains valid and narrowly actionable.

## What This Changes

The issue reports that Matrix thread event IDs are lowercased when recorded as session keys, producing duplicate sessions and failed thread replies for case-sensitive Matrix event IDs.

## Best Possible Solution

Preserve Matrix event-id casing across the shared session-recording and session-store lookup boundary while keeping intentional lowercase normalization for agent/channel/base key components, then add focused regression coverage for mixed-case Matrix thread roots through inbound recording, store lookup, outbound route recovery, and reply relation delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a mixed-case Matrix m.thread root with Matrix threadReplies enabled: Matrix routing builds a mixed-case :thread: suffix, the turn kernel records it, and the shared session/store code lowercases it before persistence; the reporter's production Synapse logs match that path.

## Solution Assessment

Yes, the requested direction is the right fix, but it should be narrow. The maintainable solution is a shared canonicalization helper or channel-provided preservation seam for case-sensitive thread suffixes, not a blanket removal of all session-key normalization.
