## Summary

Keep open. Current main has Matrix-owned routing/send paths that preserve mixed-case thread IDs, but the shared inbound session recorder still lowercases the full session key before persistence and route updates, which can still split Matrix thread handling between the original route/run key and a lowercased stored key. The reported delivery failure path needs a focused regression test because immediate Matrix reply construction now passes the original thread ID through to `m.relates_to`.

## What This Changes

The issue asks OpenClaw to preserve Matrix event ID casing across thread session keys and reply relations so `threadReplies` does not create duplicate stuck sessions or unknown-event Matrix relation failures.

## Best Possible Solution

Preserve provider-sensitive thread suffix casing at the shared session-recording and session-key parsing boundary while keeping canonical lowercasing for the provider/channel/base portions that are intentionally storage-normalized, then add Matrix regression coverage for mixed-case thread roots through inbound recording and reply delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed a mixed-case Matrix thread event through the Matrix handler with the real `recordInboundSession` path and observe that `_route.sessionKey` preserves the suffix while `src/channels/session.ts` lowercases the persisted key; the report's live Synapse logs provide the corresponding production symptom.

## Solution Assessment

Unclear for the full fix. The current Matrix-local identity normalizers are necessary, but not sufficient; the safer fix is a narrow shared canonicalization helper or channel-owned seam that preserves case-sensitive thread suffixes instead of changing every session key consumer blindly.
