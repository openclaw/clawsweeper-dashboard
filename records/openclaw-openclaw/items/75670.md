## Summary

Keep open. Current main still has the reported split: Matrix routing and send paths preserve mixed-case thread event IDs, but the shared inbound session recorder lowercases the full opaque session key before persistence and last-route updates. The related stuck-session issue #71127 is broader recovery work; this issue has a distinct Matrix event-ID casing root cause.

## What This Changes

Preserve Matrix event ID casing when forming and recording Matrix thread session keys so Matrix threads use one session and replies relate to the original event ID.

## Best Possible Solution

Make session-key canonicalization segment-aware so Matrix thread suffixes remain byte-for-byte case-preserved while stable session-key prefixes continue to normalize, then add regression coverage for mixed-case Matrix thread roots through inbound recording and outbound/recovery delivery.

## Reproduction Assessment

Yes. A source-level reproduction is to route a mixed-case Matrix thread root through resolveMatrixInboundRoute and then recordInboundSession: the Matrix route preserves the suffix, but the shared recorder persists the lowercased full key.

## Solution Assessment

Yes. The best fix is a narrow case-preserving suffix path or channel-aware session-key normalization seam; removing all shared normalization would be broader and riskier than the reported Matrix event-ID bug requires.
