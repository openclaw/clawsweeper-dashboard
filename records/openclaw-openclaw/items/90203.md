## Summary

Keep open: current main and v2026.6.6 still report `MEMORY.md` presence from the configured agent workspace only, while the useful follow-up comments narrowed this into a product/security contract decision about sanctioned external memory metadata rather than a ready bug-fix patch.

## What This Changes

Allow workspace-file status to treat a sanctioned external `MEMORY.md` store as present, or configurable, instead of always requiring `MEMORY.md` at the workspace root.

## Best Possible Solution

Define a narrow sanctioned external-memory presence contract, then update `agents.files.list`, the Control UI status, protocol/docs, and tests while keeping `agents.files.get` and `agents.files.set` workspace-only.

## Reproduction Assessment

Yes, at source level. Current `agents.files.list` checks only the configured workspace path for `MEMORY.md`, so an external curated memory file cannot satisfy the current presence check.

## Solution Assessment

Unclear until maintainers choose the contract. The comment-proposed shape is safer than making external files editable, but config-vs-allowlist-vs-auto-detection needs a product/security decision first.
