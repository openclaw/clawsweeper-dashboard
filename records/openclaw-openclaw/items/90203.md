## Summary

Keep open: current main and the latest release still report `MEMORY.md` from the configured agent workspace only, while the useful discussion has narrowed the request to a product/security decision about sanctioned external memory metadata rather than a closeable implemented bug.

## What This Changes

The issue asks OpenClaw to stop reporting root `MEMORY.md` as missing when a sanctioned external memory store is intentionally used instead.

## Best Possible Solution

Define a narrow read-only metadata projection for sanctioned external memory, with workspace `MEMORY.md` taking precedence and `agents.files.get/set` remaining workspace-only.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.6.9` shows `agents.files.list` resolves only the configured workspace and stats `DEFAULT_MEMORY_FILENAME` there; I did not run a live UI scenario because the source path is sufficient.

## Solution Assessment

Unclear until maintainers choose the contract. The commenter-proposed list-only projection is safer than making external memory editable, but config registration, allowlisted roots, path display, and security checks need product approval first.
