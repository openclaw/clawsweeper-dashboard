## Summary

Keep open: current main and v2026.6.8 still report `MEMORY.md` from the configured agent workspace only, while the comments have narrowed this into a product/security decision about sanctioned external memory metadata rather than a ready bug-fix patch.

## What This Changes

The issue requests a way for the workspace-files UI to stop marking `MEMORY.md` missing when a supported curated memory store intentionally lives outside the configured agent workspace.

## Best Possible Solution

Define a narrow read-only metadata projection for sanctioned external memory that preserves workspace `MEMORY.md` precedence and keeps content reads/writes workspace-only.

## Reproduction Assessment

Yes, at source level: `agents.files.list` only checks `MEMORY.md` under the configured agent workspace, so an external curated memory file cannot satisfy the current presence check.

## Solution Assessment

Unclear until maintainers choose the contract; the comment-proposed list-only projection is safer than editable external files, but config-vs-allowlist-vs-auto-detection needs product/security direction first.
