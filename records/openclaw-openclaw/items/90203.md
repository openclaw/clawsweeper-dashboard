## Summary

Keep open: current main and the latest release still report `MEMORY.md` presence only from the configured agent workspace, but the requested external canonical-memory handling is a product/config contract decision rather than a strict bug under current docs.

## What This Changes

The issue asks OpenClaw to stop flagging `MEMORY.md` as missing when the user's canonical memory store intentionally lives outside the agent workspace root.

## Best Possible Solution

Define a narrow memory presence contract first, then update the gateway file-list API, UI copy/status, tests, and docs to distinguish optional absent root memory from a supported external canonical store if that is the chosen direction.

## Reproduction Assessment

Yes. Source inspection shows `agents.files.list` reports `MEMORY.md` by checking only the configured agent workspace path, so an external `MEMORY.md` store will not satisfy the current presence check.

## Solution Assessment

Unclear. Config override, auto-detecting a Claude memory path, and per-file opt-out are all plausible, but current docs define `MEMORY.md` under the OpenClaw agent workspace and maintainers need to choose the supported contract first.
