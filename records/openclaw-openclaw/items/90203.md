## Summary

Keep open: current main and v2026.6.8 still report `MEMORY.md` only from the configured agent workspace, and the discussion has narrowed this into a product/security decision about sanctioned external memory metadata rather than a closeable implemented bug.

## What This Changes

Request support for suppressing the workspace-root `MEMORY.md` missing state when a sanctioned external curated memory store exists.

## Best Possible Solution

Define a narrow read-only metadata projection for sanctioned external memory that keeps workspace `MEMORY.md` precedence and preserves workspace-only content reads and writes.

## Reproduction Assessment

Yes. Source inspection shows `agents.files.list` only checks `MEMORY.md` under the configured agent workspace, so an external curated memory file cannot satisfy the presence check.

## Solution Assessment

Unclear until maintainers choose the contract. The commenter-proposed list-only projection is safer than making external memory editable, but config registration, allowlisted roots, and auto-detection need product/security approval first.
