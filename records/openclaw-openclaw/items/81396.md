## Summary

Keep this PR open: it targets a real current-main provider auth regression, but the latest head should not merge as-is because known env markers now fall through as literal discovery credentials when the env var is unset.

## What This Changes

The PR changes configured provider auth resolution so explicit `${ENV}` apiKeys resolve through the environment while bare all-caps provider apiKeys can remain literal discovery keys, with focused auth provenance tests.

## Best Possible Solution

Fix the resolver so explicit SecretRefs and known provider env markers stay env-only when unset, while only non-marker all-caps strings fall back to literal configured keys.

## Reproduction Assessment

Yes, by source inspection. Current main returns undefined for a missing all-caps configured provider key, and PR head fixes that path but introduces a separate source-reproducible missing-env regression for known markers such as `VLLM_API_KEY`.

## Solution Assessment

No, not as currently written. The direction is right, but the implementation needs an early `undefined` return for missing known env markers plus regression coverage for that case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Keep known env markers env-only when unset:** `src/agents/models-config.providers.secrets.ts:243-253`
  - body: When `configuredApiKey` is a known provider marker like `VLLM_API_KEY` and the env var is missing, this branch now falls through to the generic marker/literal return below. Return `undefined` after a missing known-marker env lookup and add a regression test for that case.
