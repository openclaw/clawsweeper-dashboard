## Summary

Keep open. Current main still lacks a durable checkpoint/resume path for builtin full memory reindexing, and the existing maintainer/Codex discussion explicitly treats adjacent embedding-failure reports as separate from process-crash resume semantics.

## What This Changes

Request durable checkpoint/resume support for long-running builtin memory indexing so interrupted full reindexes can preserve progress.

## Best Possible Solution

Add a memory-core design for resumable builtin full reindexing that persists generation/checkpoint state or safely reuses incomplete temp databases while keeping status and search aware of incomplete generations.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use the default builtin backend, start a forced full memory index, interrupt it mid-build, and the next run creates a new random temp database with no checkpoint or temp reuse path.

## Solution Assessment

Unclear for the issue's proposed options. The request is valid, but the maintainable fix needs maintainer design review because it changes durable SQLite generation semantics, partial-index visibility, and crash recovery behavior.
