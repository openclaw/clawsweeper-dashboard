## Summary

Keep open. Current main still sends normal forced/full builtin memory reindexing through a fresh random temp database and an all-or-nothing atomic swap, with no durable checkpoint cursor, abandoned-temp discovery, incomplete-generation status, or resume-on-restart contract.

## What This Changes

Request durable checkpoint/resume support for long-running builtin memory indexing so interrupted forced/full reindexes can continue without losing all temp-db progress.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that can resume committed file or chunk progress while keeping incomplete generations separate from the last clean index, then align status, config/docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the exact design changes generation semantics, status/search visibility, and likely public config, so it needs maintainer product/design judgment first.
