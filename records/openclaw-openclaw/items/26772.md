## Label Justifications

- `P2`: The issue affects a real memory indexing reliability path for large stores, but it is design-heavy normal-priority work rather than an emergency regression.
- `impact:data-loss`: Interrupted long-running indexing can discard uncommitted indexing progress and force users to repeat hours of embedding work.

## Summary

Keep open. Current main still lacks durable checkpoint/resume behavior for builtin memory full reindex, and the live thread already treats this as unresolved memory-core design work rather than completed cleanup.

## What This Changes

The issue requests durable checkpoint/resume support for long-running builtin memory indexing so interrupted full reindexes do not lose all progress.

## Best Possible Solution

Define a durable memory-core generation/checkpoint contract that resumes committed file or chunk progress after restart while keeping incomplete generations separate from the last clean searchable index, then align status, config/docs, and interrupted-reindex regression coverage around it.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an all-or-nothing atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB and config shape changes search/status visibility and atomicity, so maintainers should choose the contract first.
