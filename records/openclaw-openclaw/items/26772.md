## Summary

Keep open. Current main still uses the builtin SQLite memory backend by default and forced/full reindex still builds a fresh random temp database, with no durable checkpoint cursor, temp DB discovery, incomplete-generation status, or resume-on-restart contract.

## What This Changes

Request durable checkpoint/resume support for long-running memory indexing so interrupted full reindexes can continue instead of restarting from zero.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that resumes committed file or chunk progress while keeping incomplete generations separate from the last clean index, then align status, config/docs, and regression tests around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin path creates a fresh random temp DB for forced/full reindex and uses an atomic helper that swaps only after the full build succeeds, with no persisted resume cursor or temp DB reuse.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the exact solution needs maintainer design because it changes generation semantics, status/search visibility, and possibly public config.
