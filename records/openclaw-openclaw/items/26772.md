## Summary

Keep open. Current main still routes forced/full builtin memory reindexing through a fresh random temp SQLite database and final atomic swap, with no durable checkpoint cursor or resume-on-restart path; the existing discussion also explicitly treats adjacent embedding-failure reports as separate work.

## What This Changes

Request durable checkpoint/resume support for long-running memory indexing so interrupted full reindexes do not lose all progress.

## Best Possible Solution

Add a memory-core design for resumable builtin full reindexing that persists generation/checkpoint state or safely reuses incomplete temp databases while keeping search and status aware of incomplete generations.

## Reproduction Assessment

Yes, by source inspection. Use the default builtin backend, start a forced/full memory index, interrupt it before the final swap, and the next run creates a new random temp DB with no checkpoint or temp reuse path.

## Solution Assessment

Unclear. The request is valid, but the proposed options need maintainer design review around durable SQLite generation semantics, partial-index visibility, and whether new memorySearch config should exist.
