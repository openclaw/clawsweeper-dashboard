## Label Justifications

- `P2`: The issue affects a real memory indexing reliability path for large stores, but it is a normal-priority design improvement rather than an emergency regression.
- `impact:data-loss`: Interrupted long-running indexing can discard uncommitted indexing progress and force users to repeat hours of embedding work.

## Summary

Keep open. Current main still uses the default builtin full-reindex path with a fresh random temp SQLite database and all-or-nothing atomic swap, so durable checkpoint/resume after interruption remains unresolved.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so interrupted forced/full reindex work can resume instead of restarting from zero.

## Best Possible Solution

Define a durable memory-core generation/checkpoint contract that resumes committed file or chunk progress after restart while keeping incomplete generations separate from the last clean searchable index, then align status, config/docs, and interrupted-reindex regression coverage around it.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an all-or-nothing atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB/config approach changes generation semantics, search/status visibility, and public configuration, so maintainers should choose the contract first.
