## Summary

Keep open. Current main still lacks durable checkpoint/resume for builtin full memory reindexing: forced/full reindex creates a fresh random temp SQLite database, swaps only after the full build succeeds, and has no persisted cursor or abandoned-temp reuse path.

## What This Changes

Request durable checkpoint/resume support for long-running builtin memory indexing, including progress visibility and optional `memorySearch` indexing controls.

## Best Possible Solution

Define and implement a memory-core resumable generation model that preserves atomic search correctness while allowing full reindexes to resume from durable progress, with status/config/docs/tests aligned.

## Reproduction Assessment

Yes, by source inspection. The default builtin backend sends forced/full reindex through `runSafeReindex`, which creates a new random temp DB each run, and the atomic helper has no durable checkpoint or reuse path for interrupted work.

## Solution Assessment

Unclear. The requested direction is valid, but direct-to-main incremental commits versus a resumable temp DB requires maintainer judgment around incomplete generation semantics, search/status visibility, and config surface.
