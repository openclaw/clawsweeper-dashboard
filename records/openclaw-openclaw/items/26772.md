## Summary

Current main still has the core reliability gap: default builtin memory full reindex builds a fresh random temp SQLite database and only atomically swaps it after the whole build succeeds, with no durable cursor, incomplete-generation state, or temp-DB reuse on restart. Keep this open for maintainer design because the adjacent rollback/cache PR at https://github.com/openclaw/openclaw/pull/59137 is still open, currently conflicting, and does not by itself implement process-crash checkpoint/resume.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so forced/full reindex progress survives process crashes or interruptions.

## Best Possible Solution

Define a durable memory-core generation/checkpoint contract that resumes committed file or chunk progress after restart while keeping incomplete generations separate from the last clean searchable index, then align status, config/docs if needed, and interrupted-reindex regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an all-or-nothing atomic helper, and has no persisted cursor or reusable temp DB discovery on restart.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB/config approach changes generation semantics, search/status visibility, and public configuration, so maintainers should choose the contract first.
