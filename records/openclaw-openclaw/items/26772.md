## Summary

Keep open. Current main still uses a fresh random temp SQLite database for forced/full builtin memory reindex and only swaps it after the full build succeeds, with no durable checkpoint cursor, abandoned-temp reuse path, or resume-on-restart behavior.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so interrupted full reindexes do not lose all progress.

## Best Possible Solution

Define a memory-core generation/checkpoint model that persists resumable progress while keeping incomplete generations distinguishable from the last clean index, then align status, config, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin backend sends forced/full reindex through `runSafeReindex`, which creates a fresh random temp DB and uses an atomic helper with no durable checkpoint or reuse path.

## Solution Assessment

Unclear. The requested direction is valid, but choosing direct-to-main incremental commits versus resumable temp DBs, and deciding status/search/config semantics, needs maintainer design before implementation.
