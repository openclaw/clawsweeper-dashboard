## Summary

Keep open. Current main still runs default builtin forced/full memory reindex through a fresh random temp database and all-or-nothing atomic swap, with no persisted checkpoint cursor, reusable temp DB discovery, or public generation/status contract for resume.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so partial progress survives process crashes or restarts.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that resumes committed file or chunk progress while keeping incomplete generations separate from the last clean index, then align status, config/docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB/config approach changes generation semantics, search/status visibility, and public config, so maintainers should choose the contract first.
