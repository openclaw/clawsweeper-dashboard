## Summary

Keep open. Current main still uses a fresh random temp SQLite database for full builtin memory reindexing and swaps only after the whole build succeeds, so there is still no durable checkpoint/resume path for interrupted long-running indexes.

## What This Changes

Add crash-safe checkpoint/resume support for long-running builtin memory indexing, with aligned progress, config, status, and search semantics if those controls are adopted.

## Best Possible Solution

Define a memory-core generation model that can persist resumable progress while keeping incomplete generations distinguishable from the last clean index, then align status, config, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin backend sends forced/full reindex through `runSafeReindex`, which creates a fresh random temp DB and uses an atomic helper with no durable checkpoint or reuse path.

## Solution Assessment

Unclear. The requested direction is valid, but choosing direct-to-main incremental commits versus resumable temp DBs, and deciding the status/search/config semantics, needs maintainer design before implementation.
