## Label Justifications

- `impact:data-loss`: The issue is about losing persisted memory-index progress after an interrupted long-running reindex.

## Summary

Keep open: current main and the latest release still rebuild builtin memory indexes into a fresh random shadow database, publish only after the full rebuild succeeds, and have cleanup for abandoned shadows rather than durable checkpoint/resume state. The central request remains valid, but the fix needs maintainer direction because it changes memory-core generation, invalidation, locking, and status/search semantics.

## What This Changes

Add durable checkpoint/resume support for long-running builtin memory indexing so interrupted full reindexes can continue instead of restarting from scratch.

## Best Possible Solution

Add memory-core-owned durable generation/checkpoint metadata for validated shadow full reindexes, resume only compatible incomplete generations, and keep public config minimal unless maintainers explicitly choose knobs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a forced full memory reindex creates a fresh random shadow DB, publishes only after completion, and has no durable cursor to reuse after a process kill. I did not run a destructive kill repro in this read-only review.

## Solution Assessment

No, the direct-write plus public config proposal is not clearly the best default shape. A validated resumable shadow generation better preserves the current database-first publish invariant with less new config surface.
