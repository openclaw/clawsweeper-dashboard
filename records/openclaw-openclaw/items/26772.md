## Summary

Keep open. Current main and latest release still rebuild the builtin memory index in a fresh temp database with completion-only promotion, so the requested durable checkpoint/resume capability remains missing.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so interrupted full reindexes do not discard completed work.

## Best Possible Solution

Preserve atomic promotion, but add plugin-owned durable generation identity and checkpoint state so only a matching abandoned temp generation can resume; expose status/docs as needed and avoid public tuning knobs unless operational evidence proves they are necessary.

## Reproduction Assessment

Yes, at source level: start a forced full memory index, terminate before completion, and restart; current main creates a new random temp DB and has no durable cursor or old-temp reuse path. I did not run a destructive multi-hour live reindex during this read-only review.

## Solution Assessment

No, the issue's direct-write and new-public-config proposal is not clearly the best default shape. A resumable temporary generation would preserve atomic-swap safety with less public configuration surface.
