## Summary

Keep open: current main improved the memory reindex path from whole-file swaps to a shadow database plus table publish and stale-shadow cleanup, but it still creates a fresh random shadow database for each full reindex and has no durable checkpoint/resume cursor. The central user problem remains for interrupted long-running indexing.

## What This Changes

Request durable checkpoint/resume support for long-running builtin memory indexing so interrupted full reindexes can continue without discarding prior indexing work.

## Best Possible Solution

Add memory-core-owned durable generation/checkpoint metadata for validated shadow full reindexes, resume only compatible incomplete generations, and keep public config minimal unless operators clearly need knobs.

## Reproduction Assessment

Yes. Source inspection shows a forced full memory reindex creates a new random shadow DB, publishes only after completion, and has no durable cursor to reuse after a process kill.

## Solution Assessment

No, the direct-write plus public config proposal is not clearly the best default shape. A validated resumable shadow generation better preserves the current database-first publish invariant with less new config surface.
