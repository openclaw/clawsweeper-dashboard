## Summary

Keep open. Current main still matches the reported reliability gap, and the prior detailed issue comments already kept this separate from adjacent retry and dirty-state reports because durable checkpoint/resume semantics are still missing.

## What This Changes

Request to add durable checkpoint/resume capability to builtin memory indexing so long-running full reindexes do not lose all progress after interruption.

## Best Possible Solution

Define a durable memory-core generation/checkpoint contract that can resume committed file or chunk progress while keeping incomplete generations out of clean search/status, then align config, docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. Current main's normal forced/full builtin reindex creates a fresh random temp DB and swaps it only after the build completes, with no persisted cursor or abandoned-temp discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the exact design changes generation semantics, status/search visibility, and likely public config, so maintainers should choose the contract first.
