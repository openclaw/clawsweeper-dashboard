## Summary

Keep open. Current main still lacks durable checkpoint/resume for the default builtin memory safe-reindex path, and the existing discussion/labels already route this as maintainer design work rather than cleanup.

## What This Changes

Request durable checkpoint/resume for long-running builtin memory indexing so forced/full reindex can recover after crashes instead of restarting from zero.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that can resume committed file or chunk progress while keeping incomplete generations separate from the last clean index, then align status, config/docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB/config approach changes generation semantics, search/status visibility, and public config, so maintainers should choose the contract first.
