## Label Justifications

- `P2`: Interrupted large reindexes can waste hours and substantial provider cost, while the previously committed searchable index remains protected.
- `impact:data-loss`: A process interruption discards completed indexing progress and successful embedding work from the in-progress generation.

## Summary

Current main still loses the in-progress generation after an interrupted full reindex: it creates a fresh random temporary SQLite database, removes temporary artifacts on build failure, and has no durable cursor or resume discovery in source, tests, config, status, or docs. The narrower rollback/cache work in https://github.com/openclaw/openclaw/pull/59137 does not implement process-crash resume, so this issue remains necessary and needs an owner-approved generation-state design.

## What This Changes

Add durable checkpoint and resume behavior so interrupted builtin memory full reindexes continue from committed progress instead of restarting from zero.

## Best Possible Solution

Keep the committed index immutable during rebuild, persist one lifecycle-owned incomplete generation plus source/config fingerprint and cursor in SQLite, resume only compatible work, expose incomplete progress through status, and atomically promote the generation only after full completion.

## Reproduction Assessment

Yes, source-reproducible with high confidence: interrupting the safe full-reindex build before atomic promotion leaves no durable cursor or stable generation discovery, and the next attempt creates a new random temporary database. This read-only review did not run a destructive multi-hour live interruption.

## Solution Assessment

No, writing partial generations directly into the committed database is not the safest solution because it weakens the known-good generation boundary. A resumable incomplete SQLite generation with atomic promotion is the narrower maintainable design, and new user configuration should be avoided unless fixed internal checkpoint policy proves inadequate.

## Risks / Open Questions

- A resumable generation must never replace or be reported as the known-good searchable index before completion.
- Concurrent index attempts and stale or incompatible incomplete generations need one deterministic lifecycle owner, locking rule, and invalidation policy.
- Public checkpoint interval or resume settings would enlarge an already broad configuration surface without evidence that fixed internal policy is insufficient.
- The open rollback/cache PR may reduce repeated embedding cost after handled failures, but it could obscure that ungraceful process death still has no deterministic resume path.
