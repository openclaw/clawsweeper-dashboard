## Label Justifications

- `P2`: The issue affects a real memory indexing reliability path for large stores, but it is a normal-priority design improvement rather than an emergency regression.
- `impact:data-loss`: Interrupted long-running indexing can discard uncommitted indexing progress and force users to repeat hours of work.

## Summary

Keep open. Current main and the latest release still use the builtin full-reindex path with a fresh random temp SQLite database and all-or-nothing atomic swap, so the requested checkpoint/resume capability remains unresolved.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory full reindex operations so interrupted indexing does not discard all partial progress.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that resumes committed file or chunk progress after restart while keeping incomplete generations separate from the last clean index, then align status, config/docs, and interrupted-reindex regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an all-or-nothing atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB/config approach changes generation semantics, search/status visibility, and public config, so maintainers should choose the contract first.
