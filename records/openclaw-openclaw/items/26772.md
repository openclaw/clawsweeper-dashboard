- `impact:data-loss`: The issue is about losing generated memory-index progress after an interrupted temporary generation.

## Summary

Keep open: current main and v2026.6.8 still rebuild the builtin memory index in a fresh temporary SQLite database with completion-only promotion, so durable checkpoint/resume remains missing. Related retry, WAL, cleanup, and search-race work reduces adjacent failure modes but does not resume interrupted indexing progress.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory index rebuilds so interrupted work can continue instead of restarting from zero.

## Best Possible Solution

Add plugin-owned durable generation/checkpoint state for the builtin full-reindex path, preferably by resuming only validated temporary generations while preserving atomic promotion and avoiding public tuning knobs unless operational evidence requires them.

## Reproduction Assessment

Yes, at source level: start a forced full memory index, terminate before completion, and restart; current main creates a new random temp DB and has no durable cursor or old-temp reuse path. I did not run a destructive multi-hour live reindex during this read-only review.

## Solution Assessment

No, the direct-write and new-public-config proposal is not clearly the best default shape. A resumable temporary generation with durable identity validation would preserve atomic-swap safety with less public configuration surface.
