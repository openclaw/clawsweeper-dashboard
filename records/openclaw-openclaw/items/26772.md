- `impact:data-loss`: The issue is about losing generated memory-index progress after an interrupted temporary generation.

## Summary

Keep open: current main still creates a fresh random temporary SQLite database for each builtin full reindex and atomically publishes only after the full build completes, so durable checkpoint/resume remains missing. The newer stale-temp cleanup reduces disk waste after hard restarts, but it deletes or preserves temp families; it does not validate or resume interrupted indexing progress.

## What This Changes

Request durable checkpoint/resume support for long-running builtin memory indexing so interrupted forced/full reindexes do not restart from zero.

## Best Possible Solution

Add memory-core-owned durable generation/checkpoint state for builtin full reindexing, preferably resuming only validated temporary generations while preserving atomic promotion and avoiding public tuning knobs unless operators clearly need them.

## Reproduction Assessment

Yes. At source level, start a forced full memory index, terminate before completion, and restart; current main creates a new random temp DB and has no durable cursor or old-temp resume path.

## Solution Assessment

No, the direct-write plus public config proposal is not clearly the best default shape. A validated resumable temporary generation looks safer because it preserves the existing atomic-swap invariant with less public configuration surface.
