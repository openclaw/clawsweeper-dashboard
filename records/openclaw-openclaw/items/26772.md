## Label Justifications

- `P2`: Interrupted large reindexes can waste hours of indexing and provider work, while the committed searchable index remains protected.
- `impact:data-loss`: The issue is about losing completed indexing progress and successful embedding work from an in-progress generation after interruption.

## Summary

Keep open: current main and the latest release still use the fresh random temp-database full-reindex path, with no durable checkpoint cursor, reusable incomplete generation, resume-on-restart status, or documented resume flow. The open rollback/cache PR is adjacent but does not implement process-crash checkpoint/resume.

## What This Changes

Request durable checkpoint/resume support for builtin memory indexing so interrupted long full reindexes can resume without discarding in-progress work.

## Best Possible Solution

Keep this open until memory-core has a lifecycle-owned resumable incomplete SQLite generation with a durable cursor, compatibility fingerprint, status visibility, and atomic promotion after completion.

## Reproduction Assessment

Yes. Source inspection shows interrupting the safe full-reindex build before atomic promotion leaves no durable cursor or reusable generation identity, and the next attempt creates a new random temp database; this review did not run a destructive multi-hour live interruption.

## Solution Assessment

No. The direct-write/config-heavy proposal is not clearly the best solution because it weakens the known-good generation boundary; a resumable incomplete SQLite generation with atomic promotion is the safer maintainable direction.

## Risks / Open Questions

- A resumable design must preserve the known-good committed index boundary so partial generations are never searched or reported as clean before completion.
- Concurrent index attempts, stale temp databases, source/config fingerprint changes, and checkpoint invalidation need one owner-approved lifecycle rule.
- Public checkpoint/resume knobs would enlarge an already broad config surface unless maintainers decide fixed internal policy is insufficient.
- The open rollback/cache PR may reduce repeated provider cost after handled failures, but it can be mistaken for a full process-crash resume fix.
