## Summary

Keep open: current main still rebuilds the builtin memory index through a fresh random temporary SQLite database with completion-only atomic promotion, so durable checkpoint/resume remains missing. The merged stale-temp cleanup reduces orphaned disk waste but does not validate or resume interrupted indexing progress.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so interrupted full reindexes do not lose completed embedding work.

## Best Possible Solution

Add plugin-owned durable generation/checkpoint state for builtin full reindexing, preferably resuming only validated temporary generations while preserving atomic promotion and avoiding public tuning knobs unless operators clearly need them.

## Reproduction Assessment

Yes, at source level: start a forced full memory index, terminate before completion, and restart; current main creates a new random temp DB and has no durable cursor or old-temp reuse path. I did not run a destructive multi-hour live reindex during this read-only review.

## Solution Assessment

No, the direct-write plus public config proposal is not clearly the best default shape. A resumable temporary generation with durable identity validation would preserve atomic-swap safety with less public configuration surface.
