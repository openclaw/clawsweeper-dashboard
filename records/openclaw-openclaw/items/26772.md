## Summary

Keep open. Current main and v2026.6.6 still implement builtin memory full reindex as completion-only temp database promotion with no durable cursor or resume path; recent temp-file cleanup and metadata-race work is adjacent, but it does not solve checkpoint/resume after interruption.

## What This Changes

The issue requests checkpoint/resume support for long-running builtin memory reindex operations so crashes do not discard hours of completed indexing work.

## Best Possible Solution

Preserve atomic promotion, but add plugin-owned durable generation identity and a checkpoint cursor so only a matching abandoned temp generation can resume; expose status/docs as needed and avoid public tuning knobs unless operational evidence proves they are necessary.

## Reproduction Assessment

Yes, at source level: start a forced full memory index, terminate before completion, and restart; current main creates a new random temp DB and has no durable cursor or reuse path. I did not run a destructive multi-hour live reindex during this read-only review.

## Solution Assessment

No, the issue's direct-write and new-config proposal is not the best default shape. A resumable temporary generation preserves atomic-swap safety with less public configuration surface.
