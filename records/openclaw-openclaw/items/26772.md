## Summary

Keep open. Current main still lacks durable checkpoint/resume in the default builtin safe-reindex path, and the existing discussion and labels point to maintainer product/design follow-up rather than conservative cleanup.

## What This Changes

The issue requests durable checkpoint/resume support for long-running builtin memory indexing so interrupted full reindexes do not lose all temp-index progress.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that can resume committed file or chunk progress while keeping incomplete generations separate from the last clean index, then align status, config/docs, and regression coverage around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin forced/full reindex path creates a fresh random temp DB, rebuilds through an atomic helper, and has no persisted cursor or reusable temp DB discovery.

## Solution Assessment

Unclear. Checkpoint/resume is the right capability direction, but the proposed direct-main-DB/config design changes generation semantics, search/status visibility, and public config, so maintainers should choose the contract first.
