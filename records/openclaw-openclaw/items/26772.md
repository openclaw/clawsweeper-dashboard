## Summary

Keep open. Current main still rebuilds the builtin memory index through a fresh random temp SQLite database and final atomic swap, with no durable checkpoint or resume cursor, and the existing issue discussion explicitly keeps this separate from adjacent embedding-failure reports.

## What This Changes

Request durable checkpoint/resume support for long-running builtin memory indexing so interrupted forced/full reindexes can restart without losing all partial progress.

## Best Possible Solution

Add a memory-core design for resumable builtin full reindexing that persists generation/checkpoint state or safely reuses incomplete temp databases while keeping search and status aware of incomplete generations.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: use the default builtin backend, start a forced/full memory index, interrupt it before the final swap, and the next run creates a new random temp DB with no checkpoint or temp reuse path.

## Solution Assessment

Unclear. The request is valid, but the proposed options need maintainer design review around durable SQLite generation semantics, partial-index visibility, and whether new memorySearch config should exist.
