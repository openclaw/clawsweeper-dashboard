## Summary

Keep open. Current main still builds forced/full builtin memory reindexes in a fresh random temp SQLite database and only swaps it after the whole build succeeds, so the requested durable checkpoint/resume capability remains unimplemented; prior review discussion also explicitly kept this issue open as distinct from adjacent retry/dirty-state reports.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so interrupted full reindexes do not lose all progress.

## Best Possible Solution

Define a memory-core generation/checkpoint model that can persist resumable progress while keeping incomplete generations distinguishable from the last clean index, then align status, config, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin backend sends forced/full reindex through runSafeReindex, which creates a fresh random temp DB and uses an atomic helper with no durable checkpoint or reuse path.

## Solution Assessment

Unclear. The requested direction is valid, but choosing direct-to-main incremental commits versus resumable temp DBs, and deciding status/search/config semantics, needs maintainer design before implementation.
