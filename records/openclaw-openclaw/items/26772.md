## Summary

Keep open. Current main still lacks durable checkpoint/resume for builtin full memory reindexing: forced/full reindex creates a fresh random temp SQLite database and swaps it only after the build completes, while failed builds remove the temp index files.

## What This Changes

Request durable checkpoint/resume support for long-running memory indexing, with optional progress/status visibility and proposed `memorySearch.indexing` controls.

## Best Possible Solution

Add a memory-core design for resumable builtin full reindexing that persists generation/checkpoint state or safely reuses incomplete temp databases while keeping search/status aware of incomplete generations and updating config/docs/tests if new controls ship.

## Reproduction Assessment

Yes, by source inspection. Use the default builtin backend, start a forced/full memory index, interrupt it before the final swap, and the next run creates a new random temp DB with no checkpoint or abandoned-temp reuse path.

## Solution Assessment

Unclear. The requested direction is valid, but the proposed options need maintainer design around durable SQLite generation semantics, partial-index visibility, and whether new `memorySearch` config should exist.
