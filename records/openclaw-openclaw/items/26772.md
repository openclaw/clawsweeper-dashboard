## Summary

Keep open. Current main and the latest release still rebuild the builtin memory index through a fresh temporary SQLite database with completion-only promotion, so durable checkpoint/resume after a process crash remains missing; recent work cleans stale temp files and adjacent metadata failures but does not make abandoned progress resumable.

## What This Changes

The issue requests durable checkpoint/resume support for long-running builtin memory indexing so interrupted full reindexes do not restart from zero.

## Best Possible Solution

Keep this open until memory-core has plugin-owned durable generation/checkpoint state for full reindex, safe temp reuse or discard, clear status semantics, and regression coverage for interrupted rebuilds.

## Reproduction Assessment

Yes, at source level: start a forced full memory index, terminate before completion, and restart; current main creates a new random temp DB and has no durable cursor or old-temp reuse path. I did not run a destructive multi-hour live reindex during this read-only review.

## Solution Assessment

No, the issue's direct-write plus public tuning config proposal is not proven to be the best shape. A resumable temporary generation is a safer likely direction because it can preserve atomic swap behavior with less public configuration surface.
