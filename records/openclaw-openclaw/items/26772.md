## Summary

Keep open. Current main still rebuilds the builtin memory index in a fresh random temp SQLite database and only swaps it after the full build succeeds, so process death still discards partial progress and there is no durable checkpoint/resume contract.

## What This Changes

Add checkpoint/resume support for long-running builtin memory indexing, including resumable progress and aligned status/config semantics.

## Best Possible Solution

Define a durable memory-core generation/checkpoint model that can resume committed file/chunk progress while keeping incomplete generations distinguishable from the last clean index, then align status, config, docs, and regression tests around that contract.

## Reproduction Assessment

Yes, by source inspection. The default builtin backend sends forced/full reindex through `runSafeReindex`, which creates a fresh random temp DB and uses an atomic helper that deletes temp files on failure rather than persisting a resumable cursor.

## Solution Assessment

Unclear. The requested checkpoint/resume direction is valid, but choosing direct-to-main incremental commits versus resumable temp DBs, plus status/search/config semantics, needs maintainer design before implementation.
