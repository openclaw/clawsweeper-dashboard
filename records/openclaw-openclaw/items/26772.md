## Label Justifications

- `impact:data-loss`: The current lifecycle discards all completed indexing progress in an interrupted temporary generation.

## Summary

Keep this issue open. Current main and v2026.6.6 still discard an interrupted full-reindex generation rather than resuming it, and the newer OOM report provides supporting evidence rather than a canonical replacement; maintainers still need to define the durable generation lifecycle before implementation.

## What This Changes

Add durable checkpoint and resume support so interrupted full memory indexing can continue without discarding completed work.

## Best Possible Solution

Preserve atomic promotion, but store generation identity, ownership, and a durable work cursor in plugin-owned SQLite; resume only a matching abandoned generation, keep the completed active database serving search until promotion, and avoid new tuning configuration unless operational evidence proves it necessary.

## Reproduction Assessment

Yes at source level: start a forced full index, terminate it before completion, and restart; current main creates a different random temporary database and has no durable cursor or reuse path. The destructive long-running scenario was not executed during this read-only review.

## Solution Assessment

No, direct writes to the active database plus three new configuration knobs are not the best default. A durable resumable temporary generation preserves the existing atomic-swap safety with a smaller public surface.
