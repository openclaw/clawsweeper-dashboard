## Summary

Keep open: current main still has the source-proven cron bug, but this PR should not merge as-is because it bundles unrelated changes and has a blocking session transcript repair regression.

## What This Changes

The PR enables completed one-shot cron skipping in timer collection and also changes WhatsApp append handling, session transcript repair, gateway secrets startup retries, and provider key filtering for exec environments.

## Best Possible Solution

Land a small cron-only change on current main that enables the completed-one-shot guard for normal timer collection and adds regression coverage for completed skips and retry eligibility, leaving unrelated WhatsApp, agent, gateway, and env changes to separate PRs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed an enabled `at` job with terminal `lastStatus`, no `runningAtMs`, and a past `nextRunAtMs`, then let `onTimer` collect runnable jobs without `skipAtIfAlreadyRan`.

## Solution Assessment

No. Passing `skipAtIfAlreadyRan` into normal timer collection is the right cron mechanism, but this PR is not the best vehicle because it is stale, bundled with unrelated changes, and introduces a session transcript ID mismatch.
