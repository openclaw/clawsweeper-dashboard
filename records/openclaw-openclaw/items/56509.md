## Summary

Keep open. Current main still has the cron scheduler gap this PR targets, but this branch is not safe to merge as-is because it is unmergeable against current main, bundles unrelated behavior changes, and introduces a blocking transcript-repair regression.

## What This Changes

The PR passes `skipAtIfAlreadyRan` into cron runnable collection and also changes WhatsApp append handling, session transcript repair, gateway secrets startup retries, and provider-key environment filtering.

## Best Possible Solution

Land a narrow current-main cron fix that enables the completed-one-shot guard in normal timer collection, preserves retry eligibility for failed one-shot jobs, adds focused regression coverage, and leaves the WhatsApp, transcript, secrets, and exec-env changes for separate review.

## Reproduction Assessment

Yes. A source-level reproduction is clear: persist an enabled completed `schedule.kind === "at"` job with `lastStatus: "ok"`, a past `nextRunAtMs`, and no `runningAtMs`, then let `onTimer` collect due jobs; current main does not pass the skip flag on that path.

## Solution Assessment

No. The cron call-site idea is the right narrow mechanism, but this PR is not the best vehicle because it bundles unrelated changes, is stale against current main, and the transcript repair hunk can emit invalid id pairs.
