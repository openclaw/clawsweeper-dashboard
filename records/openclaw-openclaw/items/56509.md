## Summary

Keep open. Current main still lacks the completed one-shot guard on the normal cron timer path, and this PR has the right cron idea but is not merge-ready because it bundles unrelated behavior changes and introduces a blocking transcript-repair mismatch.

## What This Changes

The PR adds `skipAtIfAlreadyRan` to cron runnable collection and also changes WhatsApp append handling, transcript pairing repair, gateway secrets startup retries, and host exec environment provider-key filtering.

## Best Possible Solution

Land a narrow current-main cron fix that enables the completed-one-shot guard on the normal timer path, adds focused regression coverage for completed one-shot skips and transient retry eligibility, and leaves the unrelated WhatsApp, transcript, secrets, and exec-env changes to separate reviews.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to seed an enabled `schedule.kind === "at"` job with terminal `lastStatus`, no `runningAtMs`, and a past `nextRunAtMs`, then invoke the normal `onTimer` path; current main calls `collectRunnableJobs` without the completed-one-shot guard.

## Solution Assessment

No. The cron call-site mechanism is the right narrow fix, but this PR is not the best vehicle because it mixes unrelated behavior changes and the transcript repair hunk can emit mismatched assistant/tool-result IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the new fallback maps a sanitized `rawId` back to the original tool-call ID, it stores the unchanged `toolResult` under that original ID. The emitted transcript can then contain an assistant call with the original ID next to a tool result whose `toolCallId`/`toolUseId` is still sanitized, which strict replay paths can reject; rewrite the result ID before adding it to `spanResultsById`.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as four or label the field as retry count.
  - confidence: 0.78
