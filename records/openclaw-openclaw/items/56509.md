## Summary

Keep open. Current main still has the completed one-shot cron re-trigger path, but this PR should not merge as-is because it is bundled with unrelated fixes, is reported unmergeable against current main, and introduces a blocking transcript-repair mismatch.

## What This Changes

The PR adds the completed-one-shot guard to cron runnable collection and also changes WhatsApp append handling, session transcript tool-result pairing, gateway secrets startup retries, and provider-key filtering for child process environments.

## Best Possible Solution

Land a narrow current-main cron fix that enables the completed-one-shot guard on the normal timer path, adds focused regression coverage for completed skips and retry eligibility, and leaves the unrelated changes to separate reviews.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: seed an enabled `schedule.kind === "at"` job with terminal `lastStatus`, no `runningAtMs`, and a past `nextRunAtMs`, then invoke the normal `onTimer` path. Current main reaches `collectRunnableJobs` without the completed-one-shot guard.

## Solution Assessment

No. The cron call-site guard is the right narrow mechanism, but this PR is not the best vehicle because it bundles unrelated changes and its transcript repair hunk can emit mismatched assistant/tool-result IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the fallback maps a sanitized `rawId` back to the original tool-call ID, it stores the unchanged `toolResult` under that original ID. The emitted transcript can then contain an assistant call with the original ID next to a tool result whose `toolCallId`/`toolUseId` is still sanitized, which strict replay paths can reject; rewrite the result ID before adding it to `spanResultsById`.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as four or label the field as retry count.
  - confidence: 0.78
