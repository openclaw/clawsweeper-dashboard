## Summary

Keep open. Current main still has the cron bug, and the cron part of this PR is directionally correct, but the branch is not safe to merge as-is because it bundles unrelated changes and introduces a blocking transcript-repair mismatch.

## What This Changes

The PR enables the completed one-shot guard in cron timer/due-job collection and also changes WhatsApp append handling, session transcript pairing repair, gateway secrets startup retries, and exec environment provider-key filtering.

## Best Possible Solution

Land a narrow cron-only fix that passes `skipAtIfAlreadyRan: true` in the normal timer and due-job paths, adds regression coverage for completed one-shot `at` jobs and retry eligibility, and leaves the WhatsApp, transcript, secrets, and exec-env changes for separate reviews.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction: seed an enabled completed `at` job with terminal `lastStatus`, past `nextRunAtMs`, and `runningAtMs` unset, then call `onTimer` or `runDueJobs`; both paths collect runnable jobs without the completed-one-shot guard.

## Solution Assessment

No. The cron call-site change is the right narrow mechanism, but this PR is not the best vehicle because it mixes unrelated behavior changes and the transcript repair change can produce invalid assistant/tool-result ID pairs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the fallback maps a sanitized `rawId` to the original tool-call ID, the code still stores the unchanged `toolResult`. The repaired transcript can then pair an assistant call with the original ID next to a tool result that still carries the sanitized ID, which strict replay validation can reject. Rewrite the result ID to the matched original before adding it to `spanResultsById`.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as 4 or label the field as retry count.
  - confidence: 0.78
