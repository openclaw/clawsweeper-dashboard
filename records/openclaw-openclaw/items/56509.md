## Summary

Keep open: current main and the latest release still leave the central cron bug reachable, but this PR is not merge-ready because it is bundled with unrelated changes, lacks real behavior proof, and introduces a blocking transcript repair regression.

## What This Changes

The branch enables `skipAtIfAlreadyRan` for cron due collection and also changes WhatsApp append handling, transcript repair, startup secret retry behavior, and exec environment API-key filtering.

## Best Possible Solution

Land or replace this with a focused cron fix that applies the terminal one-shot guard to every non-forced due scan while preserving transient-error retry eligibility and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with terminal `lastStatus`, no `runningAtMs`, and a past `nextRunAtMs`; current normal timer and non-forced due checks still treat it as due.

## Solution Assessment

No. Passing the existing guard through due scans is the right mechanism, but this PR is not the best vehicle because it is stale, bundled with unrelated fixes, missing proof, and has a blocking transcript repair regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When `rawId` only matches through the sanitized-ID fallback, the code stores the unchanged `toolResult` under the original id. That can emit an assistant tool call with the original id next to a tool result whose `toolCallId`/`toolUseId` is still sanitized, which strict replay paths can reject; rewrite the result id before storing it.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as four or label the field as retry count.
  - confidence: 0.78
