Full review comments:

- **[P1] Split the exec env credential blocklist out:** `src/infra/host-env-security.ts:186-192`
  - body: This hunk silently strips provider API keys such as `OPENAI_API_KEY` from all exec child processes. That may be the right security direction, but it is a security/compatibility policy change that can break existing agent-invoked CLIs, so it should not land hidden inside a cron fix without separate review and proof.
  - confidence: 0.87
- **[P1] Rewrite matched sanitized result IDs before storing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the sanitized-ID fallback maps `rawId` back to an original tool-call id, the code still stores the unchanged `toolResult`. That can emit an assistant tool call such as `call_1` paired with a result still carrying `call1`, which strict replay paths may reject; rewrite `toolCallId`/`toolUseId` to the matched id before storing.
  - confidence: 0.91
- **[P2] Move startup retry into the active activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main routes startup SecretRef activation through `createRuntimeSecretsActivator` in `server-startup-config.ts`. Adding retry around this stale inline `server.impl.ts` block will not cover the active startup failure path after rebase.
  - confidence: 0.9
- **[P2] Avoid retrying the whole secrets batch:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve` wraps the full runtime snapshot. Because current secret resolution groups refs by provider and stops on the first provider error, one persistent provider failure can rerun successful providers on every retry and multiply the external calls this fix is meant to reduce.
  - confidence: 0.84
- **[P2] Preserve append read receipts before skipping:** `extensions/whatsapp/src/inbound/monitor.ts:484-486`
  - body: Moving the stale append filter before `normalizeInboundMessage` skips `maybeMarkInboundAsRead`. Current behavior marks append catch-up messages read before skipping auto-reply work, so this can regress read receipts for old replayed WhatsApp messages.
  - confidence: 0.88
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total-attempts or retry-count wording so incident logs are unambiguous.
  - confidence: 0.9
