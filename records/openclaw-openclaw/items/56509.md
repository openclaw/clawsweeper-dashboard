The PR adds `skipAtIfAlreadyRan` to cron runnable collection and also changes WhatsApp append handling, session transcript repair, gateway secrets startup retries, and host exec environment filtering.

## Best Possible Solution

Land a small cron-only fix on current main that enables the completed-one-shot guard for normal timer collection and adds focused regression coverage for completed skips and retry eligibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed an enabled `at` job with terminal `lastStatus`, no `runningAtMs`, and a past `nextRunAtMs`, then let `onTimer` collect runnable jobs without `skipAtIfAlreadyRan`.

## Solution Assessment

No. Passing `skipAtIfAlreadyRan` on normal timer collection is the right mechanism, but this PR is not the best vehicle because it is stale against current code, bundles unrelated behavior changes, and has a blocking transcript repair regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When `rawId` only matches through the sanitized-ID fallback, the code stores the unchanged `toolResult` under the original ID. That can emit an assistant tool call with the original ID next to a tool result whose `toolCallId`/`toolUseId` is still sanitized, which strict replay paths can reject; rewrite the result ID before storing it.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as four or label the field as retry count.
  - confidence: 0.78
