## Summary

Keep open. The cron defect is still present on current main, but this PR is not a good merge vehicle because it bundles several unrelated fixes, one transcript-repair change introduces a strict replay mismatch, and the `runDueJobs` hunk is obsolete because current main removed that helper.

## What This Changes

The PR enables the completed one-shot cron guard in timer/due-job collection and also changes WhatsApp append handling, session transcript pairing repair, gateway secrets startup retries, and exec environment provider-key filtering.

## Best Possible Solution

Land a narrow cron-only fix on current main that enables `skipAtIfAlreadyRan` for the normal timer collection path, adds regression coverage for completed one-shot `at` jobs on normal ticks plus retry eligibility, and leaves the unrelated behavior changes for separate PRs.

## Reproduction Assessment

Yes. Seed an enabled one-shot `at` job with `lastStatus: "ok"`, a past `nextRunAtMs`, a populated `lastRunAtMs`, and no `runningAtMs`, then let `onTimer` collect due jobs; current main reaches the normal `nowMs >= nextRunAtMs` path because the guard is not enabled.

## Solution Assessment

No. Passing the guard at the cron timer call site is the right narrow mechanism, but this PR is not the best solution because it mixes unrelated changes, contains a transcript replay regression, and includes a stale hunk for a helper no longer present on current main.
