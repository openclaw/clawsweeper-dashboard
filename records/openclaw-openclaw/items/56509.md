## Summary

Keep open. The cron bug is still source-reproducible on current main, but this external PR is not merge-ready because it lacks after-fix real behavior proof, is dirty against main, bundles unrelated changes, and introduces a blocking transcript-repair regression.

## What This Changes

The PR applies the existing one-shot cron guard to normal due scans and also changes WhatsApp append handling, session transcript repair, gateway secret startup retry, and exec environment filtering.

## Best Possible Solution

Replace or update this with a focused cron fix that applies the terminal one-shot guard to every non-forced due scan, preserves transient-error retry eligibility, and adds regression coverage for a successful completed `at` job with a stale past `nextRunAtMs`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, no `runningAtMs`, and a past `nextRunAtMs`; current normal timer and non-forced due checks still treat it as due.

## Solution Assessment

No. Passing the existing guard through due scans is the right mechanism, but this PR is not the best vehicle because it is bundled with unrelated changes, lacks real behavior proof, and has a blocking transcript repair regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Rewrite sanitized result IDs before storing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the new fallback maps a sanitized `rawId` back to an original tool-call id, `spanResultsById` stores the unchanged `toolResult`. The repaired transcript can then pair an assistant id such as `call_1` with a result still carrying `call1`, which strict replay paths may reject; rewrite `toolCallId`/`toolUseId` to the matched id before storing.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as four or label the field as retry count.
  - confidence: 0.78
