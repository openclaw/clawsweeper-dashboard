## Summary

Keep open: the cron bug is still present on current main, but this external PR is not ready to merge because it lacks real behavior proof, bundles several unrelated fixes, and introduces a blocking transcript-repair regression.

## What This Changes

This PR applies the one-shot cron terminal guard to normal due scans and also changes WhatsApp append handling, session transcript repair, gateway secret startup retry, and child-process exec environment filtering.

## Best Possible Solution

Replace or update this with a focused cron fix that applies the terminal one-shot guard to every non-forced due scan on current main, preserves transient-error retry eligibility, and adds regression coverage.

## Reproduction Assessment

Yes. A high-confidence source path is to persist an enabled successful `kind: "at"` job with `deleteAfterRun: true`, `lastStatus: "ok"`, no `runningAtMs`, and a past `nextRunAtMs`; current normal timer and manual due checks still treat it as due.

## Solution Assessment

No. Passing the existing guard through due scans is the right mechanism, but this PR is not the best vehicle because it is bundled with unrelated changes, lacks real behavior proof, and has a blocking transcript repair regression.
