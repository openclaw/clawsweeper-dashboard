## Summary

Keep open. Current main still lacks the completed one-shot guard on the normal cron timer path, so the central cron defect remains valid, but this PR is not merge-ready because it bundles unrelated WhatsApp, transcript, secrets, and exec-environment changes and introduces a blocking transcript pairing regression.

## What This Changes

The PR enables the completed one-shot cron guard in timer/due-job collection and also changes WhatsApp append handling, session transcript pairing repair, gateway secrets startup retries, and exec environment provider-key filtering.

## Best Possible Solution

Land a narrow current-main cron fix that enables `skipAtIfAlreadyRan` on the normal runnable collection path, adds regression coverage for completed one-shot `at` jobs and transient retry eligibility, and leaves the WhatsApp, transcript, secrets, and exec-env changes to separate focused reviews.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to seed an enabled `schedule.kind === "at"` job with terminal `lastStatus`, a past `nextRunAtMs`, and no `runningAtMs`, then run the normal `onTimer` path; current main calls `collectRunnableJobs` without the completed-one-shot guard.

## Solution Assessment

No. The cron call-site idea is the right narrow mechanism, but this PR is not the best vehicle because it mixes unrelated behavior changes and the transcript repair hunk can emit mismatched assistant/tool-result IDs.

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the fallback maps a sanitized `rawId` to the original tool-call ID, the code still stores the unchanged `toolResult`. The repaired transcript can then pair an assistant call with the original ID next to a tool result that still carries the sanitized ID, which strict replay validation can reject. Rewrite the result ID to the matched original before adding it to `spanResultsById`.
  - confidence: 0.91

  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as four or label the field as retry count.
  - confidence: 0.78
