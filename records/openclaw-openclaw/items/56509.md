Full review comments:

- **[P1] Split credential env stripping out of this cron fix:** `src/infra/host-env-security.ts:186-192`
  - body: This hunk silently strips provider API keys such as `OPENAI_API_KEY` from all exec child processes. That may be the right security direction, but it is a security and compatibility policy change that can break agent-invoked CLIs, so it should not land hidden inside a cron fix.
  - confidence: 0.87
- **[P1] Rewrite remapped tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the sanitized-ID fallback maps `rawId` back to an original tool-call ID, the code still stores the unchanged `toolResult`. That can emit an assistant tool call such as `call_1` paired with a result still carrying `call1`, which strict replay paths may reject.
  - confidence: 0.91
- **[P2] Move the retry into the active secrets activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main routes startup SecretRef activation through `createRuntimeSecretsActivator` in `server-startup-config.ts`. Adding retry around this stale inline `server.impl.ts` block will not cover the active startup failure path after rebase.
  - confidence: 0.9
- **[P2] Retry the failing secret provider, not the whole snapshot:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve` wraps the full runtime snapshot. Because current secret resolution groups refs by provider and stops on the first provider error, one persistent provider failure can rerun successful providers on every retry and multiply the external calls this fix is meant to reduce.
  - confidence: 0.84
- **[P2] Keep stale append read receipts before skipping:** `extensions/whatsapp/src/inbound/monitor.ts:484-486`
  - body: Moving the stale append filter before `normalizeInboundMessage` skips `maybeMarkInboundAsRead`. Current behavior marks append catch-up messages read before skipping auto-reply work, so this can regress read receipts for old replayed WhatsApp messages.
  - confidence: 0.88
