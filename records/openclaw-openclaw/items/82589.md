## Summary

## What This Changes

The branch adds an `isCommandResponse` payload marker, tags replies normalized by `handleCommands`, skips Auto-TTS for that marker, mirrors the runtime-plan type, and adds focused command/type coverage.

## Best Possible Solution

Cover every command-generated reply path that can reach Auto-TTS, including direct skill command dispatch, then require redacted real TTS-enabled runtime proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main normalizes command replies as ordinary `ReplyPayload`s and the final dispatch gate invokes TTS whenever `ttsAuto` resolves enabled. I did not run a live TTS provider/channel reproduction in this read-only review.

## Solution Assessment

No, not as submitted: the central marker plus TTS guard is the right maintainable shape, but it should also mark direct skill command replies that bypass `handleCommands`. Real behavior proof is still needed before merge.
