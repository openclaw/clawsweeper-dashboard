## Summary

Close: the useful forcing-tool-use problem is superseded by the shipped OpenAI-compatible behavior in https://github.com/openclaw/openclaw/pull/71472 and the documented `params.extra_body.tool_choice` override, while this branch still carries stale current-main integration issues and no real behavior proof.

## What This Changes

The PR forwards `tool_choice`/`toolChoice` model params into embedded-runner stream options and adds a regression test that captures the forwarded stream options.

## Best Possible Solution

Keep the shipped `tool_choice: "auto"` default plus the documented `params.extra_body.tool_choice` override, and only reopen direct `params.tool_choice` support if a concrete runtime case shows the supported override path is insufficient.

## Reproduction Assessment

Yes, at source level: direct `params.tool_choice` is still not bridged by `createStreamFnWithExtraParams()`, while the OpenAI transport only serializes `tool_choice` from `options.toolChoice` or the proxy default path. The user-visible forced-tool-use flow is nevertheless covered by the shipped `extra_body.tool_choice` override.

## Solution Assessment

No, not as submitted. The branch uses a stale test target, does not preserve ordered alias precedence, and would need a typed OpenAI-completions stream-option extension; the current shipped override path is the narrower maintained solution.
