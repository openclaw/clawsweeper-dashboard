## Summary

Keep open: current `main` and the reported release still allow a session-derived terminal patch to publish `Done` before the deferred history reload that may render the final assistant text has completed. The related stale-typing fix from https://github.com/openclaw/openclaw/pull/84565 improved the inverse symptom but did not prove this ordering boundary.

## What This Changes

The issue reports that WebChat marks a run complete before the final assistant reply text has rendered.

## Best Possible Solution

Defer user-visible terminal `chatRunStatus` publication for session-derived terminal reconciliation until the final chat event appends content or the deferred `loadChatHistory` path completes.

## Reproduction Assessment

Yes, at source level: a terminal session patch for the active run can clear `chatRunId`, clear `chatStream`, and set `chatRunStatus.phase` to `done` before the async `loadChatHistory` path finishes rendering the final message. I did not run a live browser/Windows reproduction in this read-only pass.

## Solution Assessment

No, the current implementation is not the best end state because it couples stale-run cleanup with immediate user-visible `Done` status. The safer fix is to separate local cleanup from terminal status publication until rendered content or history reload completion is known.
