- `impact:session-state`: The issue is about WebChat run/session state racing ahead of the rendered transcript state.

## Summary

Keep open: current main still publishes the WebChat done status during session-derived terminal reconciliation before the async history reload or final chat event has necessarily rendered the assistant reply.

## What This Changes

The issue asks WebChat to show the run-complete indicator only after the final assistant reply text is visible in the conversation.

## Best Possible Solution

Separate stale-run cleanup from user-visible terminal status so session-derived reconciliation can clear local state while publishing `done` only after the final message is appended or the deferred history reload finishes.

## Reproduction Assessment

Yes, at source level: a terminal session patch for the active run can clear `chatRunId`, clear `chatStream`, and set `chatRunStatus.phase` to `done` before the deferred `loadChatHistory` path or a later final chat event renders the assistant message. I did not run a live browser/Windows reproduction in this read-only pass.

## Solution Assessment

No, the current implementation is not the best end state because it couples stale local run cleanup with immediate publication of the done indicator. The maintainable fix is to delay only the user-visible terminal status while keeping the stale typing cleanup intact.
