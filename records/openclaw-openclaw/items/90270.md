- `impact:auth-provider`: The reported stale key affects provider credential resolution through daemon-managed environment variables.

## Summary

Keep open: current main and the latest release cover more install-time `.env` staging, including the sibling MiniMax service-env export case, but a plain gateway restart still restarts the existing supervisor environment instead of regenerating service-env from an edited global `.env`. The remaining request is a real UX/config-contract decision, not conservative cleanup.

## What This Changes

The issue asks OpenClaw to make `~/.openclaw/.env` the single user-facing source for environment variables, or warn/document when generated service-env values override it.

## Best Possible Solution

Maintainers should choose one daemon-env contract: regenerate managed service-env from trusted `.env` on restart, warn when generated service-env conflicts with global `.env`, or explicitly document that `openclaw gateway install --force` is required after changing service-managed env keys.

## Reproduction Assessment

Yes for source-level reproduction: current restart callers do not re-run service-env planning from the edited global `.env`. No live macOS restart reproduction was run in this read-only review.

## Solution Assessment

Unclear: making `.env` the single source of truth is a plausible UX improvement, but it is not automatically the best fix because it changes the daemon process-env precedence contract; a warning or explicit reinstall contract may be safer.
