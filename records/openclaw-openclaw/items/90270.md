## Summary

Keep open: current main still keeps the generated service environment/process environment ahead of the global `.env`, and `gateway restart` reuses the existing LaunchAgent service environment instead of rebuilding managed values from the edited global `.env`. This is a real provider-auth ergonomics problem, but the fix changes the daemon env precedence contract and needs maintainer product direction.

## What This Changes

The issue asks OpenClaw to make the global `.env` the single env-management entry point, or otherwise warn/document when managed service-env values override it.

## Best Possible Solution

Maintainers should choose one daemon-env contract: regenerate managed service-env from trusted `.env` on restart, warn when generated service env conflicts with global `.env`, or clearly document the required reinstall/doctor flow after changing service-managed env keys.

## Reproduction Assessment

Yes at source level: the restart path reuses the existing service environment instead of rebuilding managed service-env from the edited global `.env`. A fresh live macOS LaunchAgent reproduction was not run in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the env contract. Making `.env` the single source of truth is plausible UX, but overriding generated service-env on restart changes current precedence and may need a warning, doctor flow, or explicit documentation instead.
