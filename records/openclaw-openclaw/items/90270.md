## Summary

Keep open: current main and v2026.6.8 still keep generated service-env/process environment ahead of the global `.env`, and `gateway restart` does not rebuild managed service-env from an edited global `.env`. The remaining work is a daemon env-contract/product decision, not conservative cleanup.

## What This Changes

The issue requests a unified daemon environment-variable contract where global `.env` is the user-facing source of truth or conflicts with generated service env are documented or warned.

## Best Possible Solution

Maintainers should choose one daemon-env contract: regenerate managed service-env from trusted `.env` on restart, warn when generated service env conflicts with global `.env`, or clearly document the required reinstall/doctor flow after changing service-managed env keys.

## Reproduction Assessment

Yes at source level: the current restart path reuses the existing service environment instead of rebuilding managed service-env from the edited global `.env`. A fresh live macOS LaunchAgent reproduction was not run in this read-only pass.

## Solution Assessment

Unclear: making `.env` the single source of truth is plausible UX, but it changes process-env precedence; a conflict warning or explicit reinstall/doctor contract may be safer.
