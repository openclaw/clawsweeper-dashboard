## Label Justifications

- `P2`: The issue can break provider authentication after key rotation, but it is bounded to daemon environment management rather than a whole-runtime outage.
- `impact:auth-provider`: The reported failure is stale API-key resolution from service-managed environment variables overriding the user's updated provider credential.

## Summary

Keep open: current main stages global `.env` values into managed service environments during install/regeneration, but the loaded `gateway restart` path does not rebuild that service-env plan from the updated `.env`, so the reported confusion remains actionable and needs a product decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make the global `.env` the single user-facing source for environment variables, or at least document or warn when managed service-env values override it.

## Best Possible Solution

Define one explicit contract for managed service env refresh: either re-stage `.env` into service-env during restart with safe conflict handling, or emit clear doctor/status/restart warnings and docs telling users when to regenerate the service environment.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: install/regeneration reads state-dir `.env`, while loaded `gateway restart` restarts the existing service definition and LaunchAgent wrapper environment without rebuilding it from `.env`. No live macOS daemon was started in this read-only review.

## Solution Assessment

Unclear: making `.env` the unconditional winner is not automatically the best fix because service/process env currently has intentional precedence. A safer maintainer decision is needed between restart-time re-staging, conflict warnings, and clearer docs/doctor repair.
