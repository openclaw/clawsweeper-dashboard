## Summary

Keep open: current main and v2026.6.9 still do not make the state-dir `.env` the single source of truth for managed gateway service env values. The macOS restart path restages the existing LaunchAgent environment rather than rebuilding managed values from `.env`, while changing that contract needs maintainer product/security direction.

## What This Changes

The issue asks OpenClaw to make the state-dir `.env` the single entry point for provider/API-key environment values, or to sync, warn, or document conflicts with generated service-env files.

## Best Possible Solution

Maintainers should choose one daemon environment contract, then align restart, install/doctor repair, docs, and focused service-env tests around that contract.

## Reproduction Assessment

Source-level yes, live no: the restart path reuses the existing service environment, while install planning is the path that reads state-dir `.env`. I did not run a live macOS LaunchAgent reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the env contract. Making `.env` the single source of truth is plausible setup UX, but it changes current process-env/service-env precedence and may be better as restart sync, conflict warning, doctor repair, or explicit documentation.
