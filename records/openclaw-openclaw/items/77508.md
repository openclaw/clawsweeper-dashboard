## Summary

Keep open: current main still has a source-reproducible path where `channels login` can save auth, then the gateway restart call reports `invalid channels.start channel` when the channel is not enabled in the active channel registry. The report is valid, but the final behavior needs a maintainer choice between clearer diagnostics, preflight failure, or config creation.

## What This Changes

Request clearer handling when `openclaw channels login --channel whatsapp` is run without a `channels.whatsapp` config block.

## Best Possible Solution

Add a clear unconfigured/not-enabled channel diagnostic, and decide whether `channels login` should preserve auth-only behavior with guidance or preflight-block missing channel config before credentials are saved.

## Reproduction Assessment

Yes. By source inspection, with `channels.whatsapp` absent, login does not mutate channel config, then `channels.start` normalizes against active registered channels and can return the reported `invalid channels.start channel` message.

## Solution Assessment

Unclear. A clearer gateway/CLI diagnostic is the narrow fix, but auto-writing a minimal stanza versus failing before saving auth is a product-policy choice because the current login path is explicitly auth-only.
