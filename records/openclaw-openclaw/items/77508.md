## What This Changes

The issue asks for actionable handling when `openclaw channels login --channel whatsapp` runs without a `channels.whatsapp` config block, either failing before auth save or adding the block.

## Best Possible Solution

Land or replace the canonical preflight fix that rejects missing or disabled channel blocks before auth save and returns the same guidance from `channels.start`.

## Reproduction Assessment

Yes, at source level: current main still saves auth before gateway reconciliation, and `channels.start` still returns `invalid channels.start channel` when channel normalization fails. I did not run a live WhatsApp login because it can mutate auth state.

## Solution Assessment

Yes for the preflight/fail-before-save direction: it preserves the current auth-only login policy while giving actionable config guidance. Auto-writing `channels.<name>` would be a broader behavior change and is not needed for the narrow bug fix.
