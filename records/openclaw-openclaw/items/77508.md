## Summary

Keep open: current main and the latest release still perform channel auth before the post-login `channels.start` reconciliation and can still return the generic invalid-channel error; https://github.com/openclaw/openclaw/pull/83081 explicitly owns the repair but remains open, draft, and conflicting.

## What This Changes

The issue asks for channel login to fail with actionable missing-config guidance before saving auth, or otherwise avoid the opaque post-auth `channels.start` failure when `channels.<id>` is absent.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/83081 so explicit channel login and gateway start reject missing or disabled authored channel config before auth/start side effects, with matching actionable guidance.

## Reproduction Assessment

Yes, at source level: current main still runs plugin login before gateway reconciliation, and gateway channel normalization can still emit the reported `invalid channels.start channel` response when the running registry cannot resolve the channel. I did not run the live auth flow because this review is read-only and that flow can mutate auth/config state.

## Solution Assessment

Yes: the best fix is the fail-before-save/start preflight direction in the linked maintainer PR, not auto-writing a channel block during auth. Auto-writing config would be broader setup behavior and could surprise existing operators.
