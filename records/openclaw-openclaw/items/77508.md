## Summary

Keep open: current main still lets the auth-only login path proceed before gateway reconciliation and does not preflight a missing channel config block, so the reported opaque post-save error path remains source-reproducible.

## What This Changes

The issue asks `channels login` to handle a missing `channels.whatsapp` config block with a clear pre-save error or minimal config creation instead of saving auth and later reporting an opaque gateway start error.

## Best Possible Solution

Add a narrow pre-save login guard for absent or disabled channel config that emits actionable `channels.<name>` setup guidance while preserving the existing auth-only login policy.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection shows an explicit auth-capable channel can proceed through login without a `channels.whatsapp` block and only afterward hit the generic `channels.start` reconciliation path.

## Solution Assessment

Yes. The narrowest maintainable fix is to fail before credentials are saved with explicit config guidance; auto-writing channel config would be a broader product-policy change.

## Repair Work Prompt

Fix openclaw/openclaw#77508. On current main, explicit `openclaw channels login --channel whatsapp` can run the plugin login with `cfg` lacking `channels.whatsapp`, save auth, and only afterward call `channels.start`, yielding a generic gateway error. Add a narrow pre-save guard for auth-capable channel login, likely in `src/cli/channel-auth.ts`, that detects an absent or disabled `channels.<channel>` block and fails before credentials are saved with actionable guidance such as `openclaw config set channels.whatsapp.enabled true` plus restart/setup wording. Preserve the existing auth-only policy: do not auto-write channel config or change provider defaults unless a maintainer explicitly expands scope. Add regression coverage in `src/cli/channel-auth.test.ts` and, if the message also belongs at the gateway boundary, `src/gateway/server-methods/channels.start.test.ts`; add a single-line `CHANGELOG.md` fix entry for the user-facing behavior change.
