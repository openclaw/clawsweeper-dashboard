## Summary

Keep open: current main still has the opaque missing channel-block login path, and the open closing PR remains the canonical fix candidate.

## What This Changes

The issue requests clearer handling when `openclaw channels login --channel <name>` runs without a matching `channels.<name>` config block.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77630 so explicit login fails before provider auth/save and `channels.start` returns actionable config-block guidance.

## Reproduction Assessment

Yes, at source level: current main still reaches plugin login before Gateway reconciliation and `channels.start` can return `invalid channels.start channel` when the active registry lacks the channel. I did not run a live WhatsApp login because it can mutate local auth state.

## Solution Assessment

Yes: the narrow maintainable fix is a pre-auth config-block failure plus matching `channels.start` guidance. Auto-writing `channels.<name>` would be a broader behavior change and is not needed to fix the reported bug.
