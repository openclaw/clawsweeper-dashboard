## Summary

Keep open: current main still has the missing channel-block login path, and a live closing PR already owns the narrow preflight fix with real behavior proof.

## What This Changes

The issue asks for clearer handling when `openclaw channels login --channel whatsapp` is run without a `channels.whatsapp` config block.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/77630 so explicit channel login rejects missing/disabled config blocks before auth save and `channels.start` gives the same actionable guidance.

## Reproduction Assessment

Yes, at source level: current main still runs provider login before gateway reconciliation, and `channels.start` can still return `invalid channels.start channel` before checking the channel config block.

## Solution Assessment

Yes. The preflight/fail-before-save direction is the narrow maintainable fix; auto-writing `channels.<name>` would be a broader behavior change and is not needed to fix the misleading error path.
