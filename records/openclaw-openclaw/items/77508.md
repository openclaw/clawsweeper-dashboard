## Summary

Keep open: current main still lacks a pre-save/pre-reconcile check for a missing channel config block, so the CLI can save auth and then surface a gateway restart failure instead of giving an actionable config message.

## What This Changes

Request clearer `channels login` handling when `channels.<name>` is absent, either by creating a minimal channel stanza or failing before credentials are saved.

## Best Possible Solution

Add a narrow login preflight that detects an absent or disabled channel config before credentials are saved and prints actionable `channels.<name>` setup guidance, with regression coverage and a changelog entry.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection shows the relevant path: `channels login` saves auth before gateway reconciliation and lacks a missing-channel config preflight, so the observable post-save error path is plausible.

## Solution Assessment

Yes. The narrowest maintainable fix is to fail before auth is saved with explicit config guidance, rather than changing the auth-only login flow to write channel config implicitly.

Fix https://github.com/openclaw/openclaw/issues/77508 on current `main`. The observable bug is that `openclaw channels login --channel whatsapp` can save local auth and only then report a confusing gateway `channels.start` failure when `channels.whatsapp` is missing from `openclaw.json`. Preserve the existing auth-only login policy in `src/cli/channel-auth.ts`; do not auto-write channel config unless existing helpers already do so for plugin install/auto-enable. Add a pre-save/pre-login check for explicit runtime-backed channel login when the resolved config lacks an enabled `channels.<channel>` block, and fail with actionable guidance such as adding/enabling the channel via `openclaw channels add --channel whatsapp` or `openclaw config set channels.whatsapp.enabled true` before retrying. Add focused regression coverage proving login is not called and credentials are not saved when the config block is absent or disabled. If you need to touch gateway `channels.start`, keep the change generic and plugin-safe. Add a single-line user-facing changelog entry.
