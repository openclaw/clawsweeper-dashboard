## Summary

Keep open: current main, the latest OpenClaw release, and the latest published acpx package still route the built-in OpenCode ACP command through `npx -y opencode-ai acp`, so the reported runtime reliability problem is not implemented away.

## What This Changes

The issue asks OpenClaw to stop launching the ACP OpenCode agent through an `npx` wrapper and use the direct `opencode acp` command after the acpx registry is fixed or deliberately overridden.

## Best Possible Solution

Fix the OpenCode launcher contract in `openclaw/acpx`, publish a new acpx release, then bump OpenClaw’s ACPX extension dependency and validate with a live OpenCode ACP spawn/bind smoke.

## Reproduction Assessment

Yes, partially: source and dependency inspection confirm current main still launches the built-in OpenCode ACP adapter through the reported `npx` command, and the reporter provided repeated spawn-failure observations. I did not run a fresh live OpenCode ACP spawn loop, so this is source-reproducible rather than fully reproduced here.

## Solution Assessment

Unclear until maintainers confirm ownership. The direct `opencode acp` command matches OpenCode’s ACP subcommand, but the best maintainable fix is an upstream acpx registry change plus an OpenClaw dependency bump, not an unreviewed local dist or private runtime patch.
