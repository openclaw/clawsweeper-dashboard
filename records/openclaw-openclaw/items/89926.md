## Summary

Keep open: current main and the latest OpenClaw release still consume `acpx@0.10.0`, while both that consumed package and current `acpx@0.11.0` still map OpenCode to `npx -y opencode-ai acp`. The reported default path remains unresolved, but changing it needs a maintainer decision because acpx documents first-run `npx` adapter download semantics and OpenClaw already has an explicit agent-command override path.

## What This Changes

The issue asks OpenClaw to stop launching OpenCode ACP through `npx -y opencode-ai acp` and use the direct `opencode acp` binary to avoid intermittent spawn/handshake failures.

## Best Possible Solution

Decide the OpenCode launch contract in `openclaw/acpx`, publish a fixed acpx release if the default changes, bump OpenClaw's ACPX dependency, align docs, and validate with the live OpenCode ACP bind lane.

## Reproduction Assessment

No live high-confidence reproduction was established in this review. Source inspection does reproduce the unresolved path: OpenClaw delegates built-in ACP agent commands to acpx, and the consumed plus latest acpx registries still launch OpenCode through `npx`.

## Solution Assessment

No for an OpenClaw-only dist patch or unconditional private override. The direct command may be the right default, but the maintainable fix is an upstream acpx command-contract decision followed by an OpenClaw dependency bump and live OpenCode ACP proof.
