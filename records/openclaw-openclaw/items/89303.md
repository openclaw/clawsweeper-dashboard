## Summary

Keep this PR open, but the current patch is not merge-ready: the reply-scoped Telegram stop logic is added after the existing fast-abort path, and `/cancel` is promoted into shared abort detection instead of staying scoped to the intended Telegram reply flow.

## What This Changes

The PR records Telegram outbound message IDs on session entries, resolves replied-to message targets for `/stop` and `/cancel`, adds `/cancel` command metadata/docs, and exports a session-store runtime helper for plugins.

## Best Possible Solution

Move Telegram reply-target resolution into the earliest stop/cancel consumption path, keep bare `/stop` behavior explicit, scope `/cancel` to the approved Telegram reply behavior or document it as global, and either approve or avoid the new SDK export before merge.

## Reproduction Assessment

Yes for the PR-head defects: source inspection shows fast abort still runs before the new reply-target handler, and the prepared Telegram Desktop proof reports a candidate failure. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is a plausible direction, but the best fix has to resolve reply-scoped stops before the fast-abort consumer and avoid making `/cancel` a shared abort primitive unless maintainers explicitly want that global behavior.
