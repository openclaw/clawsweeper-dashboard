## Summary

Keep open: the feature direction is useful, but source review shows reply-scoped Telegram stops are placed after the shared fast-abort path, so `/stop` and the new `/cancel` can still abort the current chat session before `ReplyToId` targeting runs; real Telegram proof is also missing.

## What This Changes

The PR adds Telegram reply-scoped `/stop` and `/cancel` handling by recording message-to-session targets on session entries, exposing a session-store runtime helper, and adding focused tests.

## Best Possible Solution

Move reply-scoped Telegram stop/cancel resolution into the pre-command fast-abort path or make that path defer Telegram replies to `messageWorkTargets`, preserve bare `/stop`, and prove it with real Telegram DM/topic evidence.

## Reproduction Assessment

Yes, from source inspection: `dispatchReplyFromConfig` calls `tryFastAbortFromMessage` before command handlers, and `tryFastAbortFromMessage` uses shared abort detection plus `CommandTargetSessionKey` without the new message target lookup. I did not run tests because this review is read-only.

## Solution Assessment

No. The best fix is to integrate Telegram reply target resolution into the early fast-abort path, or explicitly bypass that path for reply-scoped Telegram stop/cancel while preserving bare `/stop`.
