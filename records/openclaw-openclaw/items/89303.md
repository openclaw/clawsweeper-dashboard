## Summary

Keep open; the reply-scoped Telegram stop idea is useful, but this PR is not merge-ready because it breaks existing bare Telegram `/stop`, adds JSON sidecar runtime state plus a plugin-to-core deep import, and has no real Telegram behavior proof.

## What This Changes

The branch adds `/cancel`, makes Telegram `/stop` and `/cancel` resolve from replied-to messages, and persists outbound message-to-session targets for abort routing.

## Best Possible Solution

Land a narrower implementation where Telegram replies can target replied-to work, bare `/stop` still aborts the active session unless maintainers explicitly approve a breaking change, and the target mapping lives in database-backed core/channel state behind an SDK-safe seam.

## Reproduction Assessment

Yes for the blocking review findings: source and docs show current main's `/stop` aborts the active session, while PR head returns a reply-required message without aborting. No live Telegram run was performed.

## Solution Assessment

No. Reply-scoped targeting is a reasonable capability, but the maintainable fix should preserve the current `/stop` fallback, avoid JSON sidecar runtime state, and expose any needed channel/core seam without a Telegram plugin deep import.
