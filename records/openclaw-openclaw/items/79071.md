## Summary

Keep open. Current main has display-layer stripping for Control UI/WebChat history, but the backend subagent announce handoff still sends the raw `triggerMessage` into the requester agent path, and the embedded Pi persistence guard only annotates inter-session provenance rather than stripping or suppressing that user message.

## What This Changes

The issue requests that subagent completion announce trigger messages be filtered before Control UI/webchat display and before session JSONL persistence.

## Best Possible Solution

Keep the issue open and fix the backend announce handoff so internal completion payloads reach the requester as internal event/custom context or sanitized transcript data, with UI/history stripping remaining only as defense in depth.

## Reproduction Assessment

Yes, from source inspection: direct and queued subagent announce delivery pass raw `triggerMessage` into the Gateway agent call, and embedded Pi persistence does not strip that message before append. I did not run the live Control UI scenario, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes, the best fix is backend-side in the subagent announce or agent persistence boundary, not another UI-only filter or a new config option. The existing display filters should stay as defense in depth but do not solve JSONL pollution.
