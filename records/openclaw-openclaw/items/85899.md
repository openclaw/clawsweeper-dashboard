## Summary

The PR is useful and mostly well-scoped, but it still overstates startup-context behavior. Current main recursively indexes `memory/**/*.md` for memory search, while the reset/new startup prelude only selects direct dated or date-prefixed files from the immediate `memory` directory.

## What This Changes

The PR changes memory tool descriptors, prompt/config help, public memory docs, and generated config-baseline hashes from `memory/*.md` to recursive `memory/**/*.md` wording.

## Best Possible Solution

Merge after keeping recursive wording for indexed memory search while leaving startup-context docs precise about direct recent daily files, unless maintainers intentionally add recursive startup-prelude support with focused tests.

## Reproduction Assessment

Yes. Source review reproduces the mismatch: recursive indexed-memory collection walks directories, but `buildSessionStartupContextPrelude` builds startup paths from a non-recursive `readdir` of the immediate `memory` directory.

## Solution Assessment

No, not as submitted. The descriptor and indexed-search docs are the right narrow fix, but startup-context docs should distinguish the top-level reset/new prelude or be paired with a deliberate recursive startup-prelude implementation and tests.
