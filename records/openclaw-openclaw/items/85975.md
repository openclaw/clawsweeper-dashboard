## Summary

This issue should stay open: current main and v2026.5.22 still clear oversized Codex app-server startup bindings before the context-engine thread-bootstrap compatibility path can inspect them. The linked open fix PR, https://github.com/openclaw/openclaw/pull/85978, owns the narrow repair, while the later Discord slowness comment suggests any remaining non-bootstrap cold-start behavior may need separate follow-up.

## What This Changes

Defer the startup native token/byte guard for Codex context-engine `thread_bootstrap` bindings so compatibility checks can reuse the warmed native thread.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/85978 for the narrow thread-bootstrap binding bug, then split any remaining oversized Discord/Codex cold-start behavior into a separate focused report if it is not covered by that path.

## Reproduction Assessment

Yes, as a source-level path: with `truncateAfterCompaction` enabled, an oversized saved `thread_bootstrap` binding reaches the startup token/byte guard before the projection compatibility check can preserve it. I did not run a live or Vitest repro because this review is read-only.

## Solution Assessment

Yes for the central bug: deferring the startup size guard only for context-engine `thread_bootstrap` bindings is the narrow maintainable fix, while `startOrResumeThread` should continue rotating incompatible bindings and non-bootstrap sessions should keep the existing guard.
