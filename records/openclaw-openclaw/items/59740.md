## Summary

Keep open. Current main still has many standalone architecture and boundary guard scripts, package scripts still expose individual guard commands, and CI still groups/delegates those checks rather than providing the requested unified checker framework, config, ESLint plugin, or migration guide.

## What This Changes

The issue asks to consolidate custom architecture and boundary check scripts into unified tooling, migrate an initial set of checks, update CI, and document rule authoring.

## Best Possible Solution

Keep this as a maintainer-directed architecture cleanup: audit the guard taxonomy, extract shared file-walking and import-parsing primitives where behavior matches, migrate a small compatible subset while preserving current CI coverage, and document the rule-authoring path before retiring individual scripts.

## Reproduction Assessment

Not applicable as a bug reproduction. Current-main source inspection gives a high-confidence verification path: inspect the `scripts/check-*` inventory, package scripts, CI shard wiring, boundary runner, docs, tests, and absence of a unified `architecture-checks` or ESLint plugin implementation.

## Solution Assessment

Unclear. The config-driven runner is plausible for repeated file-walking and import-rule checks, but the mixed guard surface needs maintainer approval and a phased taxonomy before a broad rewrite is the best maintainable solution.
