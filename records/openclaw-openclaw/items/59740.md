## Summary

Keep open. Current main still has many individual architecture and boundary guard scripts, package scripts still expose them as separate commands, and the CI boundary shard runner is a hardcoded concurrent delegator rather than the requested unified checker framework, config, ESLint plugin, or migration guide.

## What This Changes

The issue requests consolidating the custom architecture and boundary check scripts into unified tooling, migrating an initial set of checks, updating CI, and documenting how to add rules.

## Best Possible Solution

Keep this as a maintainer-directed architecture cleanup: audit the existing guard taxonomy, extract shared file-walking/import-parsing primitives where behavior matches, migrate a small compatible subset while preserving current CI coverage, and document the rule-authoring path before retiring individual scripts.

## Reproduction Assessment

Not applicable as a bug reproduction. Current-main source inspection gives a high-confidence verification path: check the `scripts/check-*` inventory, package scripts, CI shard wiring, boundary runner, docs, and absence of a unified `architecture-checks` or ESLint plugin implementation.

## Solution Assessment

Unclear. The proposed config-driven runner is plausible for repeated file-walking and import-rule checks, but the mixed guard surface needs maintainer approval and a phased taxonomy before a broad automated rewrite is the best solution.
