## Summary

Keep open: current main has a config-only MCP registry and static-header HTTP transports, but no generic MCP OAuth login/logout or SDK auth-provider wiring. Notion’s hosted MCP flow requires user OAuth, so this is a valid feature request that needs maintainer design rather than cleanup closure. ([developers.notion.com](https://developers.notion.com/docs/get-started-with-mcp))

## What This Changes

Request support for `openclaw mcp login <server>` and related auth-clearing flows for OAuth-based remote MCP servers such as Notion MCP.

## Best Possible Solution

Add a generic remote-MCP OAuth client flow in core that persists per-server OAuth session state, wires the SDK auth provider into HTTP transports, and documents login/logout/clear-auth semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source and docs do give a high-confidence current-main check that the MCP CLI only manages config and lacks an OAuth login flow.

## Solution Assessment

Unclear as an implementation plan: a generic `openclaw mcp login` direction fits the missing capability, but maintainers need to choose the credential/session model and security posture before coding it.
