## Summary

Keep open, but do not merge as-is: current main still has the foreground-fence problem and the linked issue remains open, while this branch trades truncation for stale or out-of-order visible delivery and the Mantis Telegram proof did not show a positive candidate delta.

## What This Changes

The PR changes the shared auto-reply foreground fence so active older generations are allowed after a newer visible generation, updates freshness/fence tests to expect that behavior, and removes one unused agent import.

## Best Possible Solution

Use a delivery-time fence or equivalent freshness state that prevents truncation without allowing older deliveries or fresh-settled fallback after a newer visible reply; coordinate with https://github.com/openclaw/openclaw/pull/91963 or https://github.com/openclaw/openclaw/pull/93977 rather than merging this active-generation guard as-is.

## Reproduction Assessment

Yes, source-reproducible: current main creates the fence before deeper reply admission and the existing freshness tests encode newer-visible suppression. I did not run tests because this is a read-only checkout review.

## Solution Assessment

No as submitted. Guarding on activeGenerations is not the best fix because it tracks dispatch lifetime rather than visible-delivery freshness; the safer direction is a delivery-phase fence or equivalent state that preserves newer-visible stale suppression.
