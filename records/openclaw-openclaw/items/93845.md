## Summary

Keep open: current main and v2026.6.8 still have the foreground-fence bug, but this PR is not safe as-is because the active-generation guard can let stale older same-session output, including fresh-settled fallback, deliver after a newer visible reply.

## What This Changes

The PR changes the shared auto-reply foreground fence to allow active older generations after newer visible delivery, updates freshness/fence tests to expect that behavior, and removes one unused agent import.

## Best Possible Solution

Use a delivery-time fence or equivalent freshness state that prevents truncation without allowing older deliveries or fresh-settled fallback after a newer visible reply; coordinate with the canonical foreground-fence issue and open lazy-fence PRs before merge.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 create the foreground fence before same-session reply admission, and the existing freshness tests encode newer-visible suppression. I did not run tests in this read-only checkout.

## Solution Assessment

No as submitted. Guarding on activeGenerations is a plausible mitigation for truncation, but it tracks dispatch lifetime rather than visible-delivery freshness; the safer fix must prevent truncation while preserving newer-visible stale suppression.
