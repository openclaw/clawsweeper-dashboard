## Summary

Keep this PR open, but do not merge it as-is: the active-generation guard prevents truncation by allowing older dispatches to continue after a newer visible reply, and the PR’s own expectations show older visible output and settled fallback can now appear after newer output.

## What This Changes

The PR changes the shared auto-reply foreground fence so older active generations are not cancelled after a newer visible delivery, adds fence/freshness regression tests, and removes one unused import.

## Best Possible Solution

Preserve newer-visible stale suppression while fixing truncation, likely with delivery-time fence creation or equivalent ordering state that prevents both data loss and interleaving.

## Reproduction Assessment

Yes at source level. Current main still has eager fence creation and unconditional newer-visible cancellation, while this PR’s changed tests show the risky opposite ordering: newer output followed by older output or fallback.

## Solution Assessment

No. Guarding on activeGenerations is not the best fix because it tracks dispatch lifetime rather than visible-delivery freshness; the safer solution must prevent truncation without allowing stale older output after a newer visible reply.
