## Summary

Keep open: this PR targets a real foreground-reply truncation bug, but the active-generation guard is too broad and allows older stale deliveries and fresh-settled fallbacks after a newer visible reply.

## What This Changes

This PR changes the shared auto-reply foreground fence to allow active older generations after a newer visible delivery, adds concurrency/freshness tests, and removes one unused model-picker import.

## Best Possible Solution

Use a delivery-time fence or true active-visible-delivery state that prevents truncation without allowing older deliveries or fresh-settled fallback after a newer visible reply; coordinate with https://github.com/openclaw/openclaw/pull/93977 rather than merging this active-dispatch guard as-is.

## Reproduction Assessment

Yes at source level. Current main still cancels older replies after a newer visible delivery, and this PR's own rewritten tests show the risky opposite ordering, including newer final before older final and an older settled fallback after newer output.

## Solution Assessment

No. Guarding on activeGenerations is not the best fix because it tracks dispatch lifetime rather than visible-delivery freshness; the safer direction is a delivery-phase fence or equivalent state that preserves newer-visible stale suppression.

- **[P1] Keep stale replies suppressed after newer visible delivery:** `src/auto-reply/dispatch.ts:173`
  - body: This branch treats any still-active older dispatch as fresh, but `activeGenerations` is only cleared at the end of the whole dispatch.
