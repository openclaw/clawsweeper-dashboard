## Summary

Keep open: the provider-local TTL feature is plausible, but this PR head still wires it only through inference preloads while the shared native load helper is also used by LM Studio embedding warmup.

## What This Changes

The PR adds `models.providers.lmstudio.params.ttlSeconds`, sends it as `ttl` on LM Studio native model loads for inference preloads, includes it in the inference preload dedupe key, updates local-model docs/changelog/defaults, and adds LM Studio tests.

## Best Possible Solution

Route the validated provider TTL through every LM Studio native load caller, especially embeddings, with focused coverage while keeping OpenAI-compatible inference payloads unchanged.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The review finding is source-reproducible on the PR head: `embedding-provider.ts` still calls the new TTL-capable helper without passing `ttlSeconds`.

## Solution Assessment

No, the current patch is close but incomplete. The LM Studio plugin and native load helper are the right boundary, but provider-level TTL should apply consistently to inference and embedding preload paths or the public docs/config scope should be narrowed.
