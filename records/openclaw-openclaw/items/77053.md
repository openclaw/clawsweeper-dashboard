## Summary

Keep open: current main does not implement `ttlSeconds`, and the PR is close but still incomplete because the shared LM Studio native load helper now accepts TTL while the embedding warmup path still calls it without TTL.

## What This Changes

The PR adds opt-in `models.providers.lmstudio.params.ttlSeconds` handling for LM Studio native model-load preloads, with docs, changelog/default updates, and LM Studio tests.

## Best Possible Solution

Complete the provider-local TTL feature by applying the same validated TTL to LM Studio embedding warmups while preserving the existing guarantee that OpenAI-compatible inference payloads are not mutated.

## Reproduction Assessment

Not applicable as a current-main bug reproduction. The PR-head blocker is source-reproducible: configure `models.providers.lmstudio.params.ttlSeconds` and create the LM Studio embedding provider; its warmup call omits `ttlSeconds`.

## Solution Assessment

No, the PR direction is sound but incomplete. The narrow maintainable fix is to share the validated provider TTL with every LM Studio native load path, including embedding warmup, without adding a default or touching OpenAI-compatible payloads.
