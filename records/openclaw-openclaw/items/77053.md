## Summary

Keep open: the PR is close, but the provider-level TTL option is incomplete because LM Studio embedding warmup still uses the same native load helper without threading the configured TTL.

## What This Changes

The PR adds `models.providers.lmstudio.params.ttlSeconds` support for LM Studio native load requests, updates docs/changelog, and adds LM Studio tests for TTL validation, inference payload isolation, and preload dedupe.

## Best Possible Solution

Thread one validated LM Studio provider TTL through every native load caller covered by the provider config, especially embedding warmup, while keeping OpenAI-compatible inference payloads unchanged.

## Reproduction Assessment

Not applicable for the feature request itself; yes for the blocking PR defect by source inspection. `ensureLmstudioModelLoaded` is shared with embeddings, and `createLmstudioEmbeddingProvider` calls it without provider TTL.

## Solution Assessment

No. The LM Studio plugin boundary and native load endpoint are the right direction, but the provider-level option should also cover embedding warmups or the docs/changelog should explicitly narrow the feature to inference preload only.
