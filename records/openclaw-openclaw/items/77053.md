## Summary

Keep open: the PR is on the right LM Studio plugin boundary, but it still leaves provider-level `ttlSeconds` out of the embedding warmup path that uses the same native load helper.

## What This Changes

The PR adds an opt-in `models.providers.lmstudio.params.ttlSeconds` provider setting, threads it into LM Studio inference preloads, updates docs/changelog, and adds LM Studio preload tests.

## Best Possible Solution

Keep the provider-local TTL feature, but route the validated setting through every LM Studio native load caller, especially embedding warmup, while leaving OpenAI-compatible inference payloads unchanged.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature PR. The remaining review finding is source-reproducible: the PR head adds `ttlSeconds` to the shared native load helper, but the embedding provider still calls that helper without the new option.

## Solution Assessment

No, the current patch is close but incomplete. The LM Studio plugin is the right place for the feature, but provider-level TTL should either cover embedding warmups too or the public docs/config scope should be narrowed.
