## Summary

Keep open. The updated PR fixes the earlier missing-changelog issue, but it still leaves one user-facing TTL path incomplete: LM Studio memory embedding warmups call the same native load helper without forwarding the new provider-level TTL setting.

## What This Changes

The PR adds `models.providers.lmstudio.params.ttlSeconds` handling for LM Studio native model-load requests, updates inference preload dedupe/tests, documents the opt-in config, and adds a changelog entry.

## Best Possible Solution

Thread the validated LM Studio provider TTL through every native load caller covered by the provider-level config, especially memory embedding warmup, while keeping OpenAI-compatible inference payloads unchanged.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The remaining issue is source-reproducible from the shared load helper being used by both inference and embeddings while the PR only threads `ttlSeconds` through the inference stream wrapper.

## Solution Assessment

No, the current patch is close but incomplete. The LM Studio plugin boundary is the right place for this feature, but the provider-level TTL option should also reach the existing embedding warmup load path or the documented scope should be narrowed.
