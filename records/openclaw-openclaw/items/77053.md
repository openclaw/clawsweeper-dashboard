## Summary

Keep open: the provider-local TTL feature is useful and mostly well scoped, but the latest PR head still leaves LM Studio embedding warmups on the old native load lifecycle while the shared helper/docs describe native load TTL support.

## What This Changes

The PR adds `models.providers.lmstudio.params.ttlSeconds`, passes validated TTL through LM Studio inference preloads and their dedupe key, documents the setting, updates tests/changelog, and includes a small stream lint cleanup.

## Best Possible Solution

Keep the provider-local opt-in TTL design, but apply the same validated `ttlSeconds` to every LM Studio native load path, including embeddings, while leaving OpenAI-compatible inference payloads unchanged.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The remaining blocker is source-reproducible from the latest patch: the embedding warmup caller still uses the shared native load helper without passing `ttlSeconds`.

## Solution Assessment

No. The provider-local approach is maintainable, but the patch should thread the validated TTL through LM Studio embedding warmup or explicitly narrow docs/changelog to inference-only behavior before merge.
