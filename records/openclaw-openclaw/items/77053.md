## Summary

Keep open; the feature direction is provider-local and plausible, but the latest PR head still leaves LM Studio embedding warmups outside the new `ttlSeconds` path, so the patch is not ready to merge.

## What This Changes

The PR adds opt-in `models.providers.lmstudio.params.ttlSeconds` handling for LM Studio native model-load preloads, plus docs, changelog, and regression coverage for inference preloads.

## Best Possible Solution

Keep the provider-local TTL feature, but thread the validated setting through every LM Studio native load call, including memory embedding warmup, or explicitly narrow the feature and docs to inference-only preload behavior.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature PR. The remaining blocker is source-reproducible at head `93b73e1b40dbea77345f4e0d24717596b99e1945`: embedding warmup still omits `ttlSeconds`.

## Solution Assessment

No; the provider-local native-load approach is reasonable, but the current patch is incomplete. The narrow maintainable fix is to share the same validated provider TTL with the LM Studio embedding warmup native load call while keeping OpenAI-compatible payloads unchanged.
