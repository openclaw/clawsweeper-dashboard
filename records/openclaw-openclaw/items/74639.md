## Summary

Keep open. Current main still allows phase report and `memory.dream.completed` writes to succeed independently of detached Dream Diary narrative appends, and the reported date-only backfill heading plus rollback normalization behavior still match the memory-core implementation and tests.

## What This Changes

The issue reports that memory-core dreaming can keep writing phase artifacts while `DREAMS.md` stops advancing, and that `memory rem-backfill`/rollback can leave Dream Diary formatting or managed-block state inconsistent.

## Best Possible Solution

Memory-core should make Dream Diary recovery idempotent for legacy or partially marked `DREAMS.md` files, align backfill diary headings with the intended public diary format, and make cron narrative append failures visible or retryable enough that successful phase reports cannot be mistaken for a healthy diary append.

## Reproduction Assessment

Partly yes. The backfill date-only heading and rollback/header-normalization behavior are statically reproducible from current code and tests; the exact multi-day cron stale-diary case still needs live cron/subagent logs or a focused integration reproduction.

## Solution Assessment

No current solution is attached, and current main is not sufficient. The maintainable path is a scoped memory-core fix coordinated with #70332 and #72546 rather than closing this as duplicate or implemented.
