## Summary

Keep open. Current main still allows dreaming phase report/event writes to succeed separately from detached best-effort Dream Diary appends, and the backfill/rollback implementation still matches the reported date-only heading and legacy marker-normalization symptoms while #70332 and #72546 remain open.

## What This Changes

The issue reports memory-core dreaming phase reports continuing while `DREAMS.md` stops receiving diary entries, plus `rem-backfill` and rollback producing inconsistent Dream Diary formatting.

## Best Possible Solution

Harden memory-core so cron phase success makes diary append failures observable or retryable, backfill headings are structurally consistent with normal diary entries, and rollback is idempotent for legacy or partially marked `DREAMS.md` files.

## Reproduction Assessment

Partly yes. The backfill date-only heading and rollback marker-normalization behavior are statically reproducible from current source and test expectations; the multi-day cron stale-diary path still needs live cron/subagent evidence or a focused integration test.

## Solution Assessment

No current main implementation solves the whole issue. The narrow maintainable path is to coordinate diary recovery with #70332 and cron/artifact reliability with #72546, or split a focused replacement if those do not cover this report.
