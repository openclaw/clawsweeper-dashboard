## Summary

Keep open. Current main still allows phase report/event writes to complete separately from best-effort detached Dream Diary appends, and the backfill/rollback code still matches the reported date-only heading and managed-section duplication risk.

## What This Changes

The issue requests a fix for memory-core dreaming so `DREAMS.md` advances with successful phase reports and recovery/backfill keeps the Dream Diary structurally consistent.

## Best Possible Solution

Harden memory-core so cron sweeps make diary append failures observable or retryable, align backfill headings with the normal diary format or documented recovery format, and make rollback idempotent for legacy or partially marked `DREAMS.md` files.

## Reproduction Assessment

Partly yes. The backfill date-only heading and marker-prepend rollback behavior are statically reproducible from current source and tests; the exact multi-day cron stale-diary symptom still needs live cron/subagent evidence or a focused integration test.

## Solution Assessment

No current patch solves the whole issue. The maintainable path is to coordinate the diary recovery work with #70332 and the cron/artifact reliability work with #72546 rather than closing this as implemented or duplicate.
