## Summary

Keep open. Current main still separates successful dreaming phase report/event writes from best-effort detached Dream Diary appends, and the backfill/rollback code still matches the reported date-only heading and marker-normalization symptoms. Related #70332 and #72546 are open, but they do not make this recovery-format symptom safe to close yet.

## What This Changes

The issue reports that memory-core dreaming phase reports can continue advancing while `DREAMS.md` stops receiving diary entries, and that `memory rem-backfill`/rollback can leave inconsistent diary formatting or managed blocks.

## Best Possible Solution

Keep this open until memory-core makes Dream Diary recovery idempotent for legacy or partially marked `DREAMS.md` files, aligns backfill diary headings with the intended public format, and makes detached cron narrative failures visible or retryable enough that phase report success cannot silently imply a healthy diary append.

## Reproduction Assessment

Partly yes. The backfill date-only heading and rollback/header normalization paths are statically reproducible from current code and tests; the exact multi-day cron stale-diary symptom still needs live cron/subagent logs or a targeted integration test.

## Solution Assessment

No patch is attached, and current main is not sufficient. The narrow maintainable direction is a scoped memory-core fix coordinated with open #70332 and #72546 rather than closing this as duplicate or implemented.
