## Summary

Keep open. Current main still lets successful dreaming phase report/event writes complete independently from detached best-effort Dream Diary narratives, and the backfill/rollback code still matches the reported date-only heading and legacy marker-normalization symptoms; #70332 and #72546 remain open, so this is not implemented or terminally duplicated.

## What This Changes

Reports that memory-core dreaming can keep writing phase artifacts while `DREAMS.md` stops advancing, and that `rem-backfill`/rollback can leave Dream Diary entries or managed sections inconsistent.

## Best Possible Solution

Harden memory-core so cron sweeps make diary append failures observable or retryable, make backfill headings structurally consistent with normal diary entries, and make rollback idempotent for legacy or partially marked `DREAMS.md` files.

## Reproduction Assessment

Partly yes. The backfill date-only heading and rollback marker-normalization behavior are statically reproducible from current source and test expectations; the exact multi-day cron stale-diary symptom still needs live cron/subagent evidence or a focused integration test.

## Solution Assessment

No current patch solves the whole issue. The narrow maintainable path is to coordinate the diary recovery work with #70332 and the cron/artifact reliability work with #72546 rather than closing this as duplicate or implemented.
