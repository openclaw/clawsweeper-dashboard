## Summary

Keep open. Current main still has a split path where phase reports and completion events can be written independently while Dream Diary prose is detached best-effort work, and the reported recovery symptoms still map to current backfill date formatting and marker-normalization behavior. The related diary-hardening PR and broader cron-artifact issue are both still open, so there is not enough terminal evidence for a close decision.

## What This Changes

The issue reports that memory-core dreaming kept writing April 27-29 phase reports while `DREAMS.md` stopped receiving Dream Diary entries, and that `memory rem-backfill` or rollback could leave date-only headings or malformed managed diary blocks.

## Best Possible Solution

Keep this issue open and land a coordinated memory-core fix that makes Dream Diary recovery idempotent for legacy or partially marked `DREAMS.md` files, aligns or documents backfilled diary heading format, and makes cron narrative failures visible or retryable enough that phase-report success does not imply a healthy diary append.

## Reproduction Assessment

Partly yes. The date-only backfill heading and rollback/header normalization symptoms have high-confidence static reproduction paths from current source and tests; the multi-day stale diary symptom needs live cron/subagent logs to prove the exact failure mode.

## Solution Assessment

No current patch is attached, and current main is not sufficient. The safest path is a scoped maintainer plan across this issue, #72546, and PR #70332 before queuing or landing another diary-hardening repair.
