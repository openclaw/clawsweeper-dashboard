## Summary

Keep open. Current main still allows memory-core phase report/event writes to succeed independently from best-effort detached Dream Diary appends, and the recovery code still matches the reported date-only backfill heading plus rollback marker-normalization behavior. Related #70332 and #72546 overlap, but #70332 is open/unmerged and #72546 tracks a broader no-artifacts cron symptom, so neither makes this safe to close.

## What This Changes

The issue reports that memory-core dreaming phase reports keep advancing while `DREAMS.md` diary entries stall, and that `memory rem-backfill` or rollback can leave date-only or malformed Dream Diary recovery state.

## Best Possible Solution

Keep this open until memory-core makes Dream Diary recovery idempotent for legacy or partially marked `DREAMS.md` files, aligns backfill diary headings with the intended public diary format or documents the date-only contract, and makes detached cron narrative failures visible or retryable enough that phase report success cannot silently imply a healthy diary append.

## Reproduction Assessment

Partly yes. The backfill date-only heading and rollback marker-normalization paths are directly reproducible from current code and tests; the exact multi-day cron stale-diary symptom still needs live cron/subagent logs or a targeted integration test that leaves phase reports written while detached narrative append fails.

## Solution Assessment

No patch is attached, and current main is not sufficient. The narrow maintainable direction is a scoped memory-core fix coordinated with open #70332 and #72546 rather than closing this as duplicate or implemented.
