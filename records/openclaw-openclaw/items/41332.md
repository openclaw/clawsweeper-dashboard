## Summary

Keep open: current main has not adopted the gateway shutdown change, the PR’s cooldown behavior conflicts with the current tested same-provider transient probe contract, and the cron config piece overlaps an open public API design item rather than being a safe stale cleanup close.

## What This Changes

This PR adds a gateway-specific `process.reallyExit` runtime, changes model fallback cooldown probing, adds `cron.startupStaggerMs`, updates related tests, and allowlists fake sanitizer tokens.

## Best Possible Solution

Split or refresh the work into focused changes: prove or redesign gateway shutdown, preserve the established transient cooldown probe contract unless maintainers change policy, and route cron catch-up configuration through the canonical public API work.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the Node exit contract gap, the current cooldown contract conflict, and the missing public/generated cron config surface. The original GGML Metal shutdown crash itself is not reproduced here because the PR does not include live Metal-host proof.

## Solution Assessment

No. The branch combines three independent behavior changes, and the maintainable path is to split them while proving shutdown behavior, keeping established cooldown semantics unless maintainers change policy, and resolving public cron config in https://github.com/openclaw/openclaw/issues/42039.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Prove the gateway exit path bypasses native shutdown:** `src/runtime.ts:61`
  - body: The new gateway runtime documents `process.reallyExit` as skipping C++ atexit handlers, but Node v22's `ReallyExit` implementation calls `RunAtExit(env)` before exiting. As written, the patch does not prove it avoids the GGML cleanup path it is meant to fix.
  - confidence: 0.86
- **[P2] Keep transient same-provider cooldown probes:** `src/agents/model-fallback.ts:445-451`
  - body: The PR changes active cooldown handling so same-provider fallback models are skipped. Current main intentionally permits one transient same-provider probe for transient provider failures and has regression coverage for that contract, so this regresses model-scoped recovery unless maintainers explicitly change the policy.
  - confidence: 0.9
- **[P2] Route cron catch-up config through the public surface:** `src/config/zod-schema.ts:512`
  - body: Adding `cron.startupStaggerMs` only to Zod/types leaves labels, help, docs, and the canonical missed-job policy tracked by https://github.com/openclaw/openclaw/issues/42039 unresolved. The cron configuration should go through that public config surface instead of introducing a partial key.
  - confidence: 0.84
