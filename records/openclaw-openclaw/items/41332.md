## Summary

This PR should stay open because its central changes are not merged on current main, and the branch still has merge-blocking correctness and proof gaps around gateway shutdown, cooldown policy, and cron public configuration.

## What This Changes

The PR adds a gateway-specific runtime exit path, changes model fallback cooldown probing behavior, introduces `cron.startupStaggerMs`, updates related tests, and adds test-only secret-scan allowlist comments.

## Best Possible Solution

Split or refresh the work into focused changes: prove or redesign gateway shutdown, preserve same-provider transient cooldown probes unless maintainers change that contract, and handle cron catch-up controls through the canonical public config request.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the exit-contract gap, the tested cooldown behavior conflict, and the partial cron config surface. No for the original Metal crash itself, because this review did not have live Metal-host shutdown proof.

## Solution Assessment

No. The branch combines three independent behavior changes, and the maintainable path is to split them while proving the shutdown fix, preserving current cooldown semantics unless policy changes, and routing cron catch-up settings through the canonical public config surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Prove the gateway exit path bypasses native shutdown:** `src/runtime.ts:61`
  - body: `process.reallyExit` is documented here as bypassing C++ atexit handlers, but Node v22's `ReallyExit` implementation calls `RunAtExit(env)` before exiting. The patch therefore does not prove it avoids the GGML cleanup path it is meant to fix, and the PR still lacks live Metal-host proof.
  - confidence: 0.86
- **[P2] Keep transient same-provider cooldown probes:** `src/agents/model-fallback.ts:452`
  - body: This changes active cooldown handling so same-provider fallback models are skipped. Current main intentionally permits one transient same-provider probe for rate-limit and overloaded cooldowns and has regression coverage for that contract, so this regresses model-scoped recovery unless maintainers explicitly change policy.
  - confidence: 0.9
- **[P2] Route cron catch-up config through the public surface:** `src/config/zod-schema.ts:512`
  - body: Adding `cron.startupStaggerMs` only to Zod/types leaves labels, help, docs, and the canonical missed-job policy tracked by https://github.com/openclaw/openclaw/issues/42039 unresolved. The cron configuration should go through that public config surface instead of introducing a partial key.
  - confidence: 0.84
