## Summary

Keep open: the branch still contains potentially useful shutdown and cron work, but it is not merge-ready because the gateway exit contract is unproven, the cooldown change conflicts with current tested recovery behavior, and the cron config piece overlaps an open public-API decision.

## What This Changes

The PR adds a gateway-specific runtime exit path, changes model fallback cooldown probing, introduces `cron.startupStaggerMs`, updates related tests, and adds test-only secret-scan allowlist comments.

## Best Possible Solution

Split or refresh the work into focused changes: prove or redesign gateway shutdown, preserve current transient cooldown recovery unless maintainers choose a new policy, and handle cron missed-run controls through the canonical public config design.

Yes for the PR review findings: source inspection shows the exit-contract gap, the tested cooldown behavior conflict, and the partial cron config surface. No for the original Metal crash itself, because this review did not run a live Metal-host shutdown reproduction.

## Solution Assessment

No. The branch combines three independent changes, and the maintainable path is to split the work while proving the shutdown fix, preserving current cooldown semantics unless policy changes, and routing cron catch-up settings through https://github.com/openclaw/openclaw/issues/42039.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not rely on reallyExit to bypass native shutdown:** `src/runtime.ts:61`
  - body: `process.reallyExit` is treated as bypassing C++ atexit handlers, but Node v22's implementation calls `RunAtExit(env)` before exiting. That leaves the GGML atexit crash path unproven while moving gateway shutdown onto an undocumented API, so this needs a different exit path or live Metal proof.
  - confidence: 0.86
- **[P2] Preserve transient same-provider cooldown probes:** `src/agents/model-fallback.ts:452`
  - body: This makes only the primary eligible during cooldown, so fallback models on the same provider are skipped. Current main intentionally allows one transient same-provider probe for rate-limit, overloaded, and timeout recovery and tests that behavior, so this would regress recovery unless maintainers change the policy.
  - confidence: 0.91
- **[P2] Route missed-job controls through the public cron policy:** `src/config/zod-schema.ts:512`
  - body: Adding `cron.startupStaggerMs` alone creates a partial public key that does not cover the canonical `maxMissedJobsPerRestart`/`missedJobStaggerMs` request or generated config help/docs. The cron catch-up API should be refreshed through the open public-config design instead of landing a one-off key.
  - confidence: 0.84
