## Summary

Keep open: current main still lacks the REST-health-check failure termination that this PR targets, but the branch needs contributor real-environment proof, a refresh over the shipped ping/pong keepalive work, focused tests, and a small options-contract fix before merge.

## What This Changes

The PR adds a Mattermost WebSocket health-check failure threshold that terminates the socket after repeated `getBotUpdateAt` failures so the reconnect loop can recover.

## Best Possible Solution

Refresh the PR onto current main, preserve ping/pong keepalive, keep `createMattermostConnectOnce` options required, add focused threshold/reset/opt-out tests, and require redacted Mattermost proof before merge.

## Reproduction Assessment

Yes for the source path: current main logs and reschedules `getBotUpdateAt` failures without terminating, and the focused test asserts no termination when that call throws. The live Mattermost zombie incident from the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear as-is: a consecutive REST-failure threshold is a plausible narrow recovery mechanism, but the branch weakens the required options contract and needs maintainer agreement on the reconnect policy. The safer path is to keep the threshold internal unless maintainers explicitly want a tuning surface.
