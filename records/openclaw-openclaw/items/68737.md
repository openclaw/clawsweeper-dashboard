## Summary

Keep open: current main still leaves Mattermost REST health-check failures as log-only, while this PR's branch has a small type-safety regression and needs Mattermost owner judgment because ping/pong keepalive already ships for the overlapping silent-WebSocket failure mode.

## What This Changes

The PR adds a Mattermost health-check failure threshold that terminates the WebSocket after repeated REST poll failures so the reconnect loop can create a fresh connection.

## Best Possible Solution

If REST health-check failures should be a recovery signal, land it as a Mattermost-owned policy with the required options signature preserved, focused monitor tests, and a changelog entry; otherwise close this PR in favor of the shipped ping/pong keepalive behavior.

## Reproduction Assessment

Yes for the code path under review: the existing `monitor-websocket.test.ts` can make `getBotUpdateAt` throw, and current main asserts that this logs without terminating. The exact live zombie incident in the PR body was not reproduced in this read-only sweep.

## Solution Assessment

Unclear. A REST-failure threshold is a narrow possible recovery mechanism, but current main already ships WebSocket ping/pong liveness; deciding whether REST failures should also terminate the socket is a Mattermost policy choice, and this patch needs small correctness/process fixes before merge.
