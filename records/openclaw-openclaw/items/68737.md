## Summary

Keep open: the REST health-check reconnect behavior is not on current main, but this PR is not merge-ready because it weakens the required options contract, lacks after-fix Mattermost proof, and needs a maintainer decision on using REST failures as a reconnect signal alongside the shipped ping/pong keepalive.

## What This Changes

The PR adds a Mattermost health-check failure threshold that terminates the WebSocket after repeated `getBotUpdateAt` REST failures, with an internal opt-out value.

## Best Possible Solution

If maintainers want REST failures to trigger reconnect, refresh the Mattermost plugin change onto current main, preserve ping/pong keepalive, remove the defaulted options object, add focused threshold/reset/opt-out tests, and require redacted live proof.

## Reproduction Assessment

Yes for the source path: current source and tests show `getBotUpdateAt` failures are logged and rescheduled without terminating the socket. The live zombie Mattermost incident from the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear. A REST-failure threshold is a plausible narrow recovery mechanism, but the branch still has a required-options contract bug and needs maintainer agreement on treating REST failures as WebSocket liveness failures.
