## What This Changes

The PR adds a Mattermost `healthCheckFailureThreshold` path that terminates the WebSocket after repeated `getBotUpdateAt` health-check failures.

## Best Possible Solution

Refresh the change onto current main, preserve ping/pong keepalive, remove the defaulted options object, add focused threshold/reset/opt-out tests, and require redacted Mattermost proof before merge.

## Reproduction Assessment

Yes for the source path: current source and tests show `getBotUpdateAt` failures are logged and rescheduled without terminating the socket. The live Mattermost zombie connection described in the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear: a REST-failure threshold is a plausible narrow recovery mechanism, but this branch still weakens the required options contract and needs maintainer agreement on the policy. The safer path is to refresh it onto current main with tests and real behavior proof.
