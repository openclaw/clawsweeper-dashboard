## What This Changes

This PR adds a Mattermost `healthCheckFailureThreshold` option that terminates the WebSocket after repeated REST health-check failures so the reconnect loop can establish a fresh connection.

## Best Possible Solution

Refresh onto current main, preserve shipped ping/pong keepalive, remove the empty-options default, add focused threshold/reset/opt-out coverage, and merge only after maintainers accept the REST-failure reconnect policy with redacted real Mattermost proof.

## Reproduction Assessment

Yes for the source path: current main logs and reschedules `getBotUpdateAt` failures without terminating, and the focused test asserts `terminateCalls` remains 0. The live Mattermost zombie incident was not reproduced in this read-only review.

## Solution Assessment

Unclear as-is: a consecutive REST-failure threshold is a plausible narrow recovery mechanism, but the branch weakens the required options contract and needs maintainer agreement on the default reconnect policy. The safer path is to keep the threshold internal unless maintainers explicitly want a tuning surface.
