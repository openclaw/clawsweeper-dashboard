## Summary

Keep open: this is still a plausible Mattermost recovery fix, but current main does not have the REST-failure threshold and the PR still needs a small contract fix, refreshed integration with the shipped ping/pong keepalive, regression coverage, maintainer policy approval, and real Mattermost proof before merge.

## What This Changes

The PR adds a `healthCheckFailureThreshold` counter to Mattermost WebSocket health checks and terminates the socket after repeated `getBotUpdateAt` failures so `runWithReconnect` can reconnect.

## Best Possible Solution

Refresh the PR onto current main, preserve ping/pong keepalive, remove the defaulted options object, add focused threshold/reset/opt-out tests, and require redacted Mattermost proof before merge.

## Reproduction Assessment

Yes for the source path: current main logs and reschedules `getBotUpdateAt` failures without terminating, and the focused test asserts no termination when that call throws. The live Mattermost zombie incident from the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear as-is: a REST-failure threshold is a plausible narrow recovery mechanism, but this PR weakens the required options contract and must be refreshed onto current main with tests and real behavior proof. The safer path is to keep the threshold internal unless maintainers want a documented tuning surface.
