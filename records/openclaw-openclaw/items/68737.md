## Summary

Keep open. Current main already has Mattermost WebSocket ping/pong recovery, but this PR proposes an additional REST-health-failure reconnect policy that is not implemented on main and still needs Mattermost owner judgment, contributor proof, and a small type-safety fix.

## What This Changes

The PR adds a Mattermost health-check failure threshold that terminates the WebSocket after repeated `getBotUpdateAt` REST health-check failures.

## Best Possible Solution

If maintainers want REST health-check failures to force a Mattermost reconnect, land that policy in the Mattermost plugin with the required-options contract preserved, focused monitor tests, and clear ownership of the threshold knob.

## Reproduction Assessment

Yes for the code path under review: current source and tests show `getBotUpdateAt` failures are logged and do not terminate the socket. The exact live zombie incident described in the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear. A REST-failure threshold is a plausible narrow recovery mechanism, but current main already handles transport-level stale sockets via ping/pong, so accepting REST failures as a reconnect signal is a Mattermost policy decision and the current patch still has a type-contract bug.
