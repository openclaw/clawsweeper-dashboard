## Summary

Keep open for maintainer review. The PR addresses a plausible Mattermost recovery gap, but current main has since shipped a related ping/pong WebSocket keepalive in v2026.4.29 while still intentionally treating REST health-check failures as log-only, so the remaining REST-failure reconnect policy needs an owner decision and a rebase; the diff also still has a small type-safety regression.

## What This Changes

The PR adds a `healthCheckFailureThreshold` option to Mattermost WebSocket monitoring so consecutive REST health-check failures terminate the socket and let the reconnect loop create a fresh connection.

## Best Possible Solution

If maintainers still want REST health-check failures to force a Mattermost reconnect after the shipped ping/pong keepalive, rebase the change onto current main, keep the required options signature, and add focused tests for threshold termination, success counter reset, threshold opt-out, and interaction with the existing ping/pong reconnect path.

## Reproduction Assessment

Yes for the code path under review: the existing `monitor-websocket.test.ts` seam can force `getBotUpdateAt` to throw and current main asserts that this logs without terminating. The exact live `@dandan` zombie incident was not reproduced here.

## Solution Assessment

Unclear. The threshold is a narrow possible fix for REST-failure recovery, but current main's shipped transport-liveness solution is ping/pong based, and changing REST API failures into forced reconnects is now a policy choice that should be made against the current monitor code.
