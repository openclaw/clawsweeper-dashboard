## Summary

Keep open. Current main does not contain this PR's REST health-check failure threshold, and the patch is not merge-ready because it weakens the required options contract while overlapping the shipped Mattermost ping/pong keepalive recovery path.

## What This Changes

The PR adds a Mattermost health-check failure threshold that counts consecutive REST poll failures and terminates the WebSocket so the reconnect loop can create a fresh connection.

## Best Possible Solution

If maintainers want REST health-check failures to force a Mattermost reconnect in addition to the shipped ping/pong transport liveness check, rebase the change onto current main, keep the required options signature, and add focused monitor tests for the new threshold paths. If ping/pong is considered the complete recovery policy, close this PR in favor of the shipped implementation.

## Reproduction Assessment

Yes for the code path under review: `monitor-websocket.test.ts` can force `getBotUpdateAt` to throw, and current main asserts that this logs without terminating. The exact live zombie incident described in the PR body was not reproduced in this read-only sweep.

## Solution Assessment

Unclear. A REST-failure threshold is a narrow possible recovery mechanism, but current main's shipped solution is WebSocket ping/pong liveness; deciding whether REST failures should also terminate the socket is a Mattermost policy choice, and this patch needs the required-options fix before merge.
