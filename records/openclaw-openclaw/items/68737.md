## Summary

Keep open: the Mattermost REST-health reconnect problem is plausible and the central behavior is not on current main, but this PR is not merge-ready because it loosens a required options contract, lacks real after-fix Mattermost proof, and needs maintainer agreement on using repeated REST failures as a WebSocket reconnect signal.

## What This Changes

The PR adds an internal Mattermost health-check failure threshold that terminates the WebSocket after consecutive `getBotUpdateAt` failures, with `0` as an opt-out.

## Best Possible Solution

If maintainers want REST failures to trigger reconnect, refresh the Mattermost plugin change onto current main, preserve ping/pong keepalive, keep connect options required, add focused threshold tests, and require redacted live proof.

## Reproduction Assessment

Yes for the source path: current main logs and reschedules `getBotUpdateAt` failures without terminating the socket, while the PR changes that behavior. The live Mattermost zombie incident from the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear. A consecutive REST-failure threshold is a plausible narrow recovery mechanism, but maintainers need to accept that policy and the branch should remove the defaulted required options object, add tests, and provide real behavior proof.
