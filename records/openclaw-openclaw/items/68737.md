## Summary

## What This Changes

The branch adds a Mattermost `healthCheckFailureThreshold` option that terminates the WebSocket after repeated `getBotUpdateAt` health-check failures.

## Best Possible Solution

Refresh the branch on current main, preserve ping/pong keepalive, remove the defaulted options parameter, add focused threshold/reset/opt-out coverage, and require redacted Mattermost proof before merge.

## Reproduction Assessment

Yes for the source path: current main catches `getBotUpdateAt` failures, logs them, and reschedules without terminating, and the focused test asserts no termination when that call throws. The live Mattermost zombie incident from the PR body was not reproduced in this read-only review.

## Solution Assessment

Unclear as-is: a consecutive REST-failure threshold is a plausible narrow recovery mechanism, but the branch weakens the required options contract and needs policy agreement plus real proof. The safer path is to keep the internal API required and validate both compatibility and reconnect behavior before merge.
