## Summary

Keep open: current main still has a clear source path where Auto-TTS assigns the summarizer's returned text directly to audio input, and the inspected tests only cover ordinary summary text, not reasoning-tagged, prompt-echo, or original-text echo output from a reasoning summary model.

## What This Changes

The issue reports that long Auto-TTS replies can speak the summary model's prompt/reasoning/original text instead of only the generated summary.

## Best Possible Solution

Add a narrow TTS summarization cleanup path that strips hidden reasoning/control scaffolding from summarizer output and refuses or truncates unsafe prompt-echo/original-text echoes before synthesis, with focused regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence mocked reproduction path: return a TextContent summary containing `<think>...</think>` or prompt-echo text from completeSimple and summarizeText will return it for Auto-TTS audio input. I did not run a live Microsoft/MiniMax audio repro.

## Solution Assessment

Yes, the best fix is a narrow cleanup/regression test in the TTS summarization path, reusing existing reasoning-tag/user-facing text sanitizers where possible. Switching defaults to a non-reasoning model is a useful configuration workaround but does not protect other summary models that leak reasoning or echo prompt scaffolding.
