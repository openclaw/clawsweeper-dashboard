## Summary

Keep open: the bug is valid and the real-behavior proof is sufficient, but the patch still sends a fixed 4096-token request to every selected title model, which can exceed lower configured model output budgets and silently skip the Discord rename path.

## What This Changes

The PR raises Discord generated thread-title maxTokens from 512 to 4096, increases the background rename timeout from 10s to 60s, updates the focused assertion, and adds a changelog entry.

## Best Possible Solution

Keep the larger timeout and reasoning-model headroom, but compute the effective title maxTokens as the positive minimum of the Discord cap and the selected model's declared maxTokens, with focused regression coverage.

## Reproduction Assessment

Yes, at source and log level. Current main still uses the 512-token cap and 10s timeout, and the PR body includes before/after MiniMax M2.7 logs showing the reasoning-only failure and text-producing success with higher headroom.

## Solution Assessment

No, not as written. The timeout increase and larger reasoning-model budget are the right narrow direction, but the fixed 4096-token request should be clamped to the selected model's positive maxTokens before provider transports see it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clamp title tokens to the model limit:** `extensions/discord/src/monitor/thread-title.ts:28`
  - body: The new 4096 cap is sent as options.maxTokens for every selected title model. Since OpenClaw preserves caller maxTokens and configured models can declare lower output budgets, lower-cap models can receive an over-limit request; generateThreadTitle then catches the provider failure and silently skips the background rename. Clamp to the positive minimum of the Discord cap and params.model.maxTokens before calling the completion helper.
  - confidence: 0.88
