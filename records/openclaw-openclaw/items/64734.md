## Summary

Keep this PR open: it targets a real Discord generated auto-thread title failure for reasoning models, and the timeout increase is consistent with the fire-and-forget rename path, but the patch still forwards a fixed 4096-token request without respecting the selected model's lower output cap.

## What This Changes

The PR raises Discord generated auto-thread title maxTokens from 512 to 4096, raises the title generation timeout from 10 seconds to 60 seconds, updates the focused unit assertion, and adds a changelog entry.

## Best Possible Solution

Keep the larger timeout and reasoning-model headroom, but clamp the requested title token cap to the selected model's positive maxTokens and cover that with focused regression tests.

## Reproduction Assessment

Yes, at source and log level. Current main still uses a fixed 512-token cap and 10-second timeout, and the PR body includes MiniMax M2.7 runtime logs showing the after-fix path producing text; I did not rerun live Discord.

## Solution Assessment

No as written. The timeout and reasoning-model headroom are the right narrow direction, but the fixed 4096-token request should be clamped to the selected model's positive output cap before it reaches provider transports.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Clamp title tokens to the model limit:** `extensions/discord/src/monitor/thread-title.ts:28`
  - body: The raised 4096 cap is forwarded unchanged as `options.maxTokens` for every selected title model. The simple-completion helper preserves that value, and Anthropic/OpenAI transports prefer requested tokens over `model.maxTokens`, so a lower-cap model can receive an over-limit request and silently skip the rename. Clamp to the positive minimum of the Discord cap and `params.model.maxTokens` before calling the completion helper.
  - confidence: 0.88
