## Summary

Keep this PR open: it is a focused Discord bug fix with useful real-world evidence, but the current patch still forwards a fixed 4096-token request without respecting the selected model's declared output cap.

## What This Changes

The PR raises Discord generated auto-thread-title maxTokens from 512 to 4096, raises the background rename timeout from 10s to 60s, updates the unit assertion, and adds a changelog entry.

## Best Possible Solution

Keep the larger timeout and reasoning-model headroom, but clamp the title maxTokens to the prepared model's positive maxTokens when that limit is lower than 4096.

## Reproduction Assessment

Yes, at source and PR-log level. Current main forwards a fixed 512-token title request with a 10s abort, and the PR body includes MiniMax M2.7 logs showing thinking-only empty output before the larger cap/timeout and non-empty text afterward.

## Solution Assessment

No, not as written. Raising the timeout and title budget is the right narrow direction, but the fixed 4096 request should be clamped before provider transports see it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clamp title tokens to the selected model limit:** `extensions/discord/src/monitor/thread-title.ts:28`
  - body: The new 4096 title cap is forwarded unchanged as `options.maxTokens` for every selected title model. The simple-completion runtime preserves caller options, and Anthropic/OpenAI transports prefer runtime `maxTokens` over `model.maxTokens`, so a selected or custom model whose declared output cap is below 4096 can receive an over-limit request and silently lose the background rename. Clamp to the positive minimum of this cap and `params.model.maxTokens` before calling the completion helper.
  - confidence: 0.89
