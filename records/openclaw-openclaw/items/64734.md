## Summary

Keep open: this targets a real Discord generated auto-thread-title failure and includes credible live logs, but the patch can over-request output tokens for selected models whose declared cap is below 4096 because the raised value is forwarded directly.

## What This Changes

The branch raises Discord generated thread-title timeout and max-token constants, updates the title-generation unit expectation, and adds a changelog entry.

## Best Possible Solution

Keep the timeout increase and higher Discord-local title budget, but clamp requested maxTokens to the prepared model's declared output cap with a low-cap regression test before merging.

## Reproduction Assessment

Yes, by source: current main forwards a 512-token cap and 10-second abort signal into Discord generated-title completion, and thinking-only assistant content is ignored by text extraction. I did not run the live MiniMax/Discord path in this read-only review.

## Solution Assessment

No as written: raising the Discord-owned caps is the right boundary, but the token request should be bounded by the selected model's output cap so low-cap custom models do not receive over-limit requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clamp the title cap to the selected model limit:** `extensions/discord/src/monitor/thread-title.ts:31`
  - body: The new 4096 value is passed straight through as `maxTokens`, while model/provider metadata defines `maxTokens` as the known output cap. If a Discord title model is configured with a cap below 4096, this can turn generated-title calls into over-limit provider requests and silently skip the rename. Keep the higher ceiling for reasoning models, but pass a positive `min(4096, model.maxTokens)` and cover a low-cap model in the unit test.
  - confidence: 0.83
