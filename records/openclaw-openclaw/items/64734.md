## Summary

Keep open: the Discord timeout and larger reasoning-model budget are useful, but the PR still forwards a fixed 4096-token request without bounding it to the selected model's output cap, which can turn generated title renames into silent failures for lower-cap models.

## What This Changes

The branch raises Discord generated thread-title timeout from 10 seconds to 60 seconds and maxTokens from 512 to 4096, updates the unit expectation, and adds a changelog entry.

## Best Possible Solution

Keep the timeout increase and higher Discord title budget, but clamp the requested maxTokens to the selected model's positive output cap and cover that low-cap case in tests.

## Reproduction Assessment

Yes. Source inspection shows current main forwards a 512-token cap and 10-second abort signal into generated-title completion, and the PR body provides before/after MiniMax M2.7 logs for the thinking-only failure path; I did not rerun the live Discord path in this read-only review.

## Solution Assessment

No as written. Raising the Discord-owned timeout and budget is the right narrow direction, but the fixed 4096 request should be bounded by the selected model's declared output cap before it is forwarded.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Clamp the title cap to the model limit:** `extensions/discord/src/monitor/thread-title.ts:28`
