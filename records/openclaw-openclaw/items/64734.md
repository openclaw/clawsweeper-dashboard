## Summary

Keep open: this is a focused bug-fix PR with sufficient runtime-log proof, but the new fixed 4096-token request can exceed the selected model's declared output cap because the completion path forwards caller maxTokens unchanged.

## What This Changes

Raises Discord generated thread-title maxTokens from 512 to 4096 and timeout from 10 seconds to 60 seconds, updates the unit expectation, and adds an Unreleased changelog entry.

## Best Possible Solution

Keep the timeout increase and higher Discord title budget, but clamp the requested title maxTokens to the selected model's positive output cap and cover that low-cap case in tests.

## Reproduction Assessment

Yes, by source and PR logs: current main forwards a 512-token cap and 10-second abort signal into generated-title completion, and thinking-only responses produce no extracted title. I did not rerun the live Discord path in this read-only review.

## Solution Assessment

No as written: raising the Discord-owned timeout and budget is the right narrow direction, but maxTokens should be bounded by the selected model's declared output cap before forwarding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clamp the title cap to the selected model limit:** `extensions/discord/src/monitor/thread-title.ts:28`
  - body: The new 4096 cap is forwarded as maxTokens, while model metadata defines maxTokens as the known output cap and the Anthropic transport preserves a caller-provided request as-is. A configured thread-title model below 4096 can reject every generated-title request and silently keep raw thread names; clamp to positive min(DISCORD_THREAD_TITLE_MAX_TOKENS, model.maxTokens) and add a low-cap regression test.
