## Summary

Keep this PR open: the Discord-local bug fix is well motivated and has usable live-log proof, but the patch introduces an over-limit request risk for selected models with `maxTokens` below 4096 because the new cap is forwarded unbounded through simple completion.

## What This Changes

This PR raises Discord generated auto-thread-title completion max tokens and timeout, updates the unit expectation, and adds a changelog entry.

## Best Possible Solution

Keep the Discord-local timeout increase, raise the title ceiling for reasoning models, and bound the requested tokens by the selected model's declared output cap with focused regression coverage.

## Reproduction Assessment

Yes: current main forwards a 512-token cap and 10-second abort signal into Discord generated-title completion, and thinking-only assistant content is ignored by text extraction. I did not run a live MiniMax/Discord reproduction in this read-only review.

## Solution Assessment

No as written: raising the Discord-owned caps is the right boundary, but the token request should be clamped to the selected model's output cap so small/custom providers do not start receiving over-limit requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clamp the title cap to the selected model limit:** `extensions/discord/src/monitor/thread-title.ts:31`
  - body: The new 4096 cap is forwarded as the requested `maxTokens`, and the simple-completion transports use requested values directly while `maxTokens` is documented as the model output cap. If a user selects or configures a Discord title model with `maxTokens < 4096`, generated-title calls can now send an over-limit request and silently skip the rename. Keep the 4096 ceiling for high-cap reasoning models, but pass a positive `min(4096, model.maxTokens)` and add a low-cap regression expectation.
  - confidence: 0.83
