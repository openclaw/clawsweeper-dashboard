## Summary

Keep open: the fix direction is valid and backed by live logs, but the PR still forwards an unbounded 4096-token request, which can regress lower-cap title models into provider rejection and silent rename skips.

## What This Changes

The PR raises Discord generated thread-title maxTokens from 512 to 4096, timeout from 10s to 60s, updates the unit expectation, and adds a changelog entry.

## Best Possible Solution

Keep the timeout increase and larger reasoning-model title budget, but clamp the requested maxTokens to the selected model's positive output cap and cover that low-cap case in the Discord title-generation tests.

## Reproduction Assessment

Yes. Source inspection shows current main sends a fixed 512-token cap and 10s abort signal into generated-title completion, and the PR body includes MiniMax M2.7 before/after logs for the thinking-only failure path; I did not rerun live Discord.

## Solution Assessment

No as written. Raising the Discord-owned timeout and budget is the right narrow direction, but the fixed 4096 request should be bounded by the selected model's declared output cap before it is forwarded.
