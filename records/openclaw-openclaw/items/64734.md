## Summary

Keep this PR open: the reported Discord auto-thread-title failure is credible and the timeout/headroom direction is useful, but the patch still forwards a fixed 4096-token request without respecting the selected model's declared output limit.

## What This Changes

The PR raises Discord generated thread-title max tokens and timeout, updates the focused unit assertion, and adds a changelog entry.

## Best Possible Solution

Keep the larger timeout and reasoning-model headroom, but compute the effective title maxTokens as the positive minimum of the Discord cap and the selected model's declared maxTokens, with focused regression coverage.

## Reproduction Assessment

Yes, at source and log level. Current main still sends a 512-token cap with a 10s timeout, the PR body includes after-fix MiniMax M2.7 logs for the reported path, and the remaining defect follows from the PR's fixed 4096 cap overriding lower configured model limits.

## Solution Assessment

No, not as written. The timeout increase and larger reasoning-model budget are the right narrow direction, but the fixed token request should be clamped to the selected model's positive maxTokens before provider transports see it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Clamp title tokens to the selected model limit:** `extensions/discord/src/monitor/thread-title.ts:28`
  - body: The new 4096 cap is forwarded as `options.maxTokens` for every selected title model. OpenClaw transports prefer the caller value over the model's declared limit, so configured lower-cap models can receive an over-limit request, fail in the provider call, and silently skip the background rename. Clamp to the positive minimum of the Discord cap and `params.model.maxTokens` before calling the completion helper.
  - confidence: 0.89
