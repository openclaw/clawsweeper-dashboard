## Summary

Keep open. Current main has partial mitigations, including private-by-default group/channel final replies and newer Slack bot-room authorization, but it still lacks a dedicated guard for OpenClaw gateway-generated error replies in Slack rooms where sibling bot messages are intentionally allowed. The source path still accepts owner-present or explicitly allowlisted sibling bot messages with `allowBots: true`, and OAuth refresh failures still produce non-generic gateway auth guidance rather than a suppressible gateway-error marker.

## What This Changes

Prevent gateway-posted Slack error replies in shared `allowBots` channels from triggering sibling agents into repeated auth-failure loops.

## Best Possible Solution

Add a gateway-error-origin guard for Slack inbound handling so OpenClaw-generated auth/failure replies from sibling OpenClaw bot accounts are ignored without disabling legitimate bot-to-bot messages. Then add a small repeated-error suppression or auth-failure cooldown as defense in depth, with regression coverage for two Slack bot identities in a shared channel.

## Reproduction Assessment

Yes for a source-level reproduction: current code still accepts allowed sibling Slack bot room messages and still builds non-generic OAuth gateway auth guidance. A live Slack flood was not reproduced, so the test should model the two-bot `allowBots: true` room path deterministically.

## Solution Assessment

Yes, Option B from the issue is the best base fix: classify or tag gateway-generated error replies and suppress them on inbound before they start a new agent session. Error dedupe and auth cooldowns are useful follow-ups, but they should not replace origin-aware suppression.
