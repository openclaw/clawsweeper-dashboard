## Summary

Keep open: current main still lacks a Slack gateway-error-origin guard, and open PR #77045 is the targeted implementation path; this issue should stay open until that PR lands or is replaced.

## What This Changes

Suppress Slack shared-channel feedback loops where OpenClaw gateway error replies from one bot trigger sibling agents when `allowBots: true`.

## Best Possible Solution

Land or revise #77045 so Slack outbound gateway failure replies are tagged and Slack inbound handling drops tagged OpenClaw failure echoes while preserving legitimate bot-to-bot messages, with regression tests and changelog coverage.

## Reproduction Assessment

Yes for a source-level reproduction: current main can still accept allowed sibling Slack bot room messages, and OAuth refresh failures are still rendered as visible gateway auth guidance. A live Slack flood was not run in this read-only review.

## Solution Assessment

Yes: the active tagged-metadata approach in #77045 is the narrowest maintainable form of Option B from the discussion and avoids disabling legitimate `allowBots` workflows. Close this issue only after that PR lands or an equivalent fix reaches main.
