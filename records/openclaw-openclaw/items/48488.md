## Summary

Current main and the shipped v2026.4.29 release implement the central recovery path: queued tasks can now carry a task timeout, active work is raced against that timeout, embedded agent runs pass a timeout-derived lane watchdog, and regression coverage proves a never-settling task releases the lane and drains queued work.

## What This Changes

The issue requests task-level timeout or recovery for command-lane work so never-settling promises cannot permanently block serialized session and cron lanes.

## Best Possible Solution

Keep the shipped queue timeout implementation and close the stale candidate PR as superseded by the released maintainer fix, while tracking any narrower leftover timeout-policy questions separately.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains. Source inspection and regression coverage show a never-settling lane task with taskTimeoutMs now rejects, clears the active slot, and lets queued work drain.

## Solution Assessment

Yes. The shipped approach is the narrow maintainable fix: the queue exposes a generic per-task timeout and the embedded runner derives its watchdog from the existing run timeout instead of adding a separate user-facing policy knob.
