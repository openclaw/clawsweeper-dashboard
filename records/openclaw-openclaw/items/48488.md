## Summary

Current main now implements the requested lane task timeout/recovery path well enough to close: command queue entries support caller-owned task timeouts, timed-out active tasks reject and pump the lane, embedded agent turns wire their run timeout into the global command lane, and focused tests cover a never-settling task draining queued work. The fix is present on main but not in the latest release tag I could verify.

## What This Changes

Add task-level timeout/recovery for command lane work so never-settling promises do not permanently block serialized session and cron lanes.

## Best Possible Solution

Keep the current main implementation and close this issue as implemented, then resolve the now-superseded open PR separately so the contributor thread has a clear outcome.

## Reproduction Assessment

Yes. The focused reproduction is a maxConcurrent=1 lane with a first task that never settles and a second queued task; current main has a regression test proving taskTimeoutMs releases the active slot and drains the queued task.

## Solution Assessment

Yes. The current implementation is a maintainable variant of the suggested Promise.race fix: queue timeouts are caller-owned, embedded agent turns wire the existing run timeout plus grace, and cron retains its job timeout path rather than imposing one fixed timeout on every lane user.
