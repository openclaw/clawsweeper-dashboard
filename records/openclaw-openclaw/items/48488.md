## Summary

Current main and the latest stable release already implement the central queue-level recovery: active lane tasks can be bounded by caller-owned timeouts, timed-out tasks release the lane slot, and regression coverage proves queued work drains afterward.

## What This Changes

The issue requests automatic recovery for hung command-lane tasks so a never-settling promise cannot permanently block session lanes.

## Best Possible Solution

Keep the shipped caller-owned, progress-aware lane timeout implementation and resolve the stale linked PR separately rather than reviving its hard default timeout branch.
