## Summary

Current main and the shipped v2026.5.3 tree now include a command-lane task timeout path with regression coverage, so the reported permanent lane jam is implemented well enough to close.

## What This Changes

The issue requests a task-level timeout or recovery path for command lane tasks whose promises never settle.

## Best Possible Solution

Keep the shipped v2026.5.3 timeout implementation; file any future follow-up as a narrower caller-specific report if an unbounded internal lane remains observable.

## Reproduction Assessment

Yes: the original failure mode has a high-confidence source-level regression path in `src/process/command-queue.test.ts` using a never-settling first task and queued second task. Current main no longer reproduces the permanent jam for timed queue entries because the first promise times out, rejects, clears the active slot, and lets the next task run.

## Solution Assessment

Yes. The current solution adds a per-task queue timeout and propagates the embedded agent-run timeout into queued gateway work, which is narrower than an unconditional global reset loop and preserves caller-specific budgets for legitimate long-running work.
