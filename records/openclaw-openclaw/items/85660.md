## Label Justifications

- `P1`: The reported behavior is a provider-routing regression that could send a Copilot credential to the wrong provider endpoint.
- `impact:auth-provider`: The issue concerns model provider routing, Copilot auth profiles, and `agents.defaults.imageModel` selection.

## Summary

Current main and the latest shipped release already keep this image-model path on the GitHub Copilot provider: unknown Copilot model IDs are synthesized by the Copilot resolver, Gemini IDs use Copilot-compatible Chat Completions metadata, and the image runtime exchanges GitHub credentials before streaming through the Copilot provider.

## What This Changes

The issue reports that an unknown provider-prefixed GitHub Copilot `agents.defaults.imageModel` is routed to OpenAI instead of Copilot forward compatibility or a clear Copilot-owned error.

## Best Possible Solution

Keep the shipped provider-owned Copilot image-model route in core; handle broader live Copilot catalog refresh in the related open catalog issue.

## Reproduction Assessment

No current-main failing reproduction is needed for closure: source inspection shows current main and `v2026.5.20` route unknown Copilot Gemini image models through the Copilot resolver and token-exchange path. I did not run a live Copilot Enterprise replay because it requires the reporter's entitlement and credentials.

## Solution Assessment

Yes. The current implementation keeps model resolution, Gemini transport metadata, and token exchange in the GitHub Copilot provider boundary rather than adding a core special case or broad fallback.
