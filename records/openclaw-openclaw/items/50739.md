## Summary

Keep open. Current main and v2026.5.22 still route `openclaw system event --mode now` through queued system-event plus heartbeat delivery, and the heartbeat path intentionally skips while the main/session lane or target reply run is busy. The requested bypass mode is useful but needs maintainer product/safety direction because it can reorder prompt-visible writes against active session work.

## What This Changes

Request a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent in-session alerts do not wait behind saturated LLM/session lanes.

## Best Possible Solution

Define a maintainer-approved priority event contract that surfaces urgent operator alerts immediately while preserving active-turn and session-ordering guarantees, then update the CLI/RPC/schema/docs and focused regression tests.

## Reproduction Assessment

Yes, source-level: `system event --mode now` enqueues a system event and requests heartbeat, while heartbeat skips on busy main/session lanes or active reply runs. I did not run the original live provider-outage incident.

## Solution Assessment

Unclear: a priority or bypass flag is plausible, but the best fix depends on maintainer policy for whether urgent alerts may bypass active session ordering or need a safer side-channel.
