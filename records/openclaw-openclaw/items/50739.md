## Summary

Keep open. Current main and v2026.6.10 still route `openclaw system event --mode now` through the queued system-event plus immediate-heartbeat path; the heartbeat runner still skips busy main/session lanes and retries later, so the reported urgent-alert latency is not solved. The remaining work needs maintainer direction because true bypass would change prompt-visible session ordering and trust semantics.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent in-session alerts can avoid congested session lanes.

## Best Possible Solution

Define a maintainer-approved urgent-alert contract that is immediately operator-visible and separately decides when, if ever, the alert becomes prompt-visible session context, with tests for busy main lanes, busy session lanes, docs, and release-safe CLI/protocol behavior.

## Reproduction Assessment

Yes, from source inspection: `system event --mode now` enqueues a system event and requests a heartbeat, while the heartbeat runner skips busy main/session lanes and the wake layer retries later. I did not reproduce the original provider-rate-limit incident live.

## Solution Assessment

Unclear as a standalone flag: a bypass mode addresses the alert delay, but the maintainable fix needs an explicit operator-visible versus prompt-visible ordering contract. A narrow implementation should preserve active turn integrity while making urgent alerts visible immediately.
