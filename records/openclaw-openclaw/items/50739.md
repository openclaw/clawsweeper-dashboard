## Summary

Keep open. Current main and v2026.6.6 still route `openclaw system event --mode now` through queued system-event plus heartbeat delivery, and the related open wake/notification PRs do not settle the prompt-visible versus operator-visible alert contract requested here.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so incident alerts can appear immediately during a congested session lane.

## Best Possible Solution

Define a core alert contract that makes incident notifications visible immediately without racing active prompt/transcript state, with a separately reviewed path if prompt-visible mutation is required.

## Reproduction Assessment

Yes, at source level: current main has no priority/bypass CLI or wake contract, `mode now` enqueues then wakes heartbeat, and heartbeat skips busy session lanes. I did not run a live provider-outage saturated-lane reproduction in this read-only pass.

## Solution Assessment

No, not as a narrow flag alone. The maintainable fix needs a product/API decision about immediate operator-visible notification versus prompt-visible transcript injection before implementation starts.
