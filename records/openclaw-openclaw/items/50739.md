## Summary

Keep open. Current main fixed the adjacent session-targeting gap for `openclaw system event`, but it still has no priority/bypass mode and the heartbeat runner intentionally defers system-event delivery while the main or target session lane is busy.

## What This Changes

Request a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent in-session alerts surface immediately during saturated session lanes.

## Best Possible Solution

Define an explicit urgent system-alert contract that is immediately operator-visible, with separate ordering and trust rules for any transcript or prompt mutation.

## Reproduction Assessment

Yes, from source and focused tests: a system event is queued, `mode: "now"` requests heartbeat delivery, and a busy session lane makes heartbeat return `requests-in-flight` instead of delivering immediately. I did not run a live gateway/TUI outage reproduction in this read-only pass.

## Solution Assessment

Unclear. A raw bypass into the prompt-visible session stream is not obviously the safest fix; the maintainable path is likely an immediate operator-visible alert surface plus separately ordered transcript insertion.
