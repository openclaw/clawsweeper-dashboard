- `P2`: The issue affects timely operator alerts during degraded provider/session conditions but requires a bounded product decision rather than emergency remediation.
- `impact:message-loss`: The reported failure mode delays an operational alert until it no longer functions as a timely in-session message.
- `impact:session-state`: The requested bypass would change how prompt-visible system events are ordered into session state while other turns are active.

## Summary

Keep open. Current main and v2026.5.22 still route system events through queued heartbeat delivery, and the heartbeat runner deliberately skips/retries when the main lane, active reply, or target session lane is busy; the requested immediate alert path remains a product/safety decision.

## What This Changes

Request a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent in-session alerts surface during provider or session-lane congestion.

## Best Possible Solution

Define an explicit opt-in high-priority alert contract that can notify the operator immediately while keeping prompt/transcript mutation ordered and tested against busy lanes and active replies.

## Reproduction Assessment

Yes, at source level: current main enqueues the event and requests heartbeat delivery, and `runHeartbeatOnce` skips with `requests-in-flight` when the main queue, active reply, or target session lane is busy. I did not run a live provider-rate-limit incident replay.

## Solution Assessment

Unclear as proposed: a raw bypass flag would solve latency but can reorder prompt-visible writes against active work. The safer direction is an explicit opt-in alert contract that separates immediate operator visibility from transcript or prompt injection.
