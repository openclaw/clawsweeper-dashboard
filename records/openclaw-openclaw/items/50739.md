## Summary

Keep open. Current main still enqueues `openclaw system event --mode now` through the wake/heartbeat path, and heartbeat still skips busy main or resolved session lanes with `requests-in-flight`, so the requested reliable urgent in-session alert behavior is not implemented.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent alerts can appear immediately even when the session lane is congested.

## Best Possible Solution

Define a first-class urgent system alert contract that gives operators immediate TUI/UI visibility during degraded provider incidents, while handling transcript or prompt mutation through a separately ordered and trust-reviewed path across the CLI, Gateway wake protocol, heartbeat delivery, docs, and regression tests.

## Reproduction Assessment

Yes. The source-level path is high-confidence: `system event --mode now` sends `wake`, cron enqueues the event and requests heartbeat, and heartbeat returns `requests-in-flight` when the main or resolved session lane is busy; the busy-lane regression test covers that behavior.

## Solution Assessment

No/unclear for the raw bypass flag as proposed. Immediate alert delivery is valid, but directly bypassing into trusted prompt-visible system events is not the safest maintainable shape without a separate operator-alert surface and explicit transcript-ordering semantics.

## Risks / Open Questions

- A prompt-visible bypass could race an active streaming turn or reorder transcript state; current heartbeat code explicitly avoids busy session lanes.

- Because queued system events become trusted `System:` prompt lines by default, urgent delivery needs spoofing, trust, and prompt-injection review before it mutates an active transcript.
