## Summary

Keep open: current main still queues `openclaw system event --mode now` through the wake and heartbeat path, and the requested urgent alert behavior is a product and safety decision rather than a conservative cleanup close.

## What This Changes

Request adds a high-priority or bypass-queue mode for `openclaw system event --mode now` so in-session alerts can appear immediately during congested session lanes.

## Best Possible Solution

Define an urgent operator-alert contract that gives immediate TUI/UI visibility during degraded provider incidents, while transcript or prompt mutation remains separately ordered and trust-reviewed.

## Reproduction Assessment

Yes, at source level. The CLI sends only `{ mode, text }`, wake enqueues a system event, and heartbeat skips busy main or session lanes with tests covering that deferral path.

## Solution Assessment

No/unclear for the raw bypass flag as proposed. Immediate alert delivery is valid, but directly bypassing into trusted prompt-visible system events is not the narrowest safe shape without explicit ordering and trust semantics.

## Risks / Open Questions

- A prompt-visible bypass could race an active streaming turn or violate transcript ordering; current heartbeat code explicitly skips busy session lanes.

- System events default to trusted `System:` prompt-visible lines, so urgent delivery needs spoofing, trust, and prompt-injection semantics before bypassing the queue.
- The KeepMyClaw snapshot and quota-trigger comments are useful operational context, but they do not replace the requested immediate in-session alert contract.
