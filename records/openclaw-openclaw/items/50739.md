## Summary

Keep open. Current main and the latest release still expose `system event` only as queued system-event plus heartbeat wake delivery, and heartbeat deliberately skips active reply runs or busy session lanes, so the requested reliable in-session alert path is not implemented. Related KeepMyClaw and quota-threshold comments are useful operational context, but they do not replace an immediate operator-visible alert contract.

## What This Changes

The issue requests a high-priority or queue-bypass mode for `openclaw system event --mode now` so urgent in-session alerts are not delayed behind congested session lanes.

## Best Possible Solution

Define an explicit urgent system-alert contract that delivers immediate operator-visible notifications, with separate ordering and trust rules for any transcript or prompt mutation.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the CLI and wake RPC only send normal wake fields, wake enqueues a system event, and heartbeat returns `requests-in-flight` while the target reply run or session lane is busy. I did not run a live provider-outage/TUI incident reproduction in this read-only pass.

## Solution Assessment

Unclear. The proposed bypass flag addresses the alert-delay symptom, but the safer maintainable path is likely an immediate operator-visible alert surface plus a separately ordered transcript insertion path.

## Risks / Open Questions

- A prompt-visible bypass can race an active streaming turn and reorder or corrupt transcript state; current heartbeat code explicitly skips active reply runs and busy session lanes to avoid that class of issue.
- Because wake params allow opaque metadata, callers can already send a `priority`-like field that validates but is silently ignored by the current handler.
- The product shape is unresolved: immediate operator/TUI notification may need to be separate from prompt-visible `System:` transcript injection.
- Open adjacent issues such as https://github.com/openclaw/openclaw/issues/54488 and https://github.com/openclaw/openclaw/issues/64810 show that scheduler fairness and non-preemption safety need coordinated design.
