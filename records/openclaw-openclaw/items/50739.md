## Label Justifications

- `P2`: The issue describes delayed operational alerts during provider outages, but the requested fix is a new priority contract with bounded product scope rather than an emergency runtime outage.
- `impact:message-loss`: The alert is not necessarily deleted, but arriving many minutes late makes the operator message effectively lost as an incident alert.
- `impact:session-state`: The requested bypass would write into or alongside the active session stream and affects ordering, transcript, and active-run state semantics.

## Summary

Keep open. Current main and v2026.5.22 still implement `openclaw system event --mode now` as a queued system event plus heartbeat wake; when the target session is active or its lane is busy, the heartbeat intentionally skips and retries, so the requested high-priority in-session alert contract is still missing and needs maintainer product direction.

## What This Changes

Add an explicit high-priority or bypass-queue mode for `openclaw system event` so operator alerts can surface immediately during congested session or provider-failure states.

## Best Possible Solution

Define a maintainer-approved urgent-alert contract that preserves existing queued behavior by default, then add a narrowly scoped operator-controlled path for prompt-visible system alerts or a separate TUI alert surface with ordering and retry tests.

## Reproduction Assessment

Yes, at source level. Current `system event --mode now` enqueues a session event and requests heartbeat, while heartbeat skips with `requests-in-flight` when the target session run or session lane is active; I did not run a live outage replay.

## Solution Assessment

Unclear on the exact implementation. The need is valid, but a raw `--priority high` queue bypass is not obviously the safest solution until maintainers decide how urgent system alerts should interact with active streaming, transcript ordering, and foreground user/control messages.
