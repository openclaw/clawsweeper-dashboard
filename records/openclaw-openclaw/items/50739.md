## Summary

Keep open. Current main and v2026.5.20 still route `openclaw system event --mode now` through the normal system-event queue plus heartbeat wake path; the requested immediate operator-visible high-priority alert contract is still not implemented and should not be closed as duplicate because nearby scheduler issues cover different starvation/preemption directions.

## What This Changes

The issue requests a priority or bypass-queue mode for `openclaw system event --mode now` so degraded-system alerts can appear in-session without waiting behind congested LLM request lanes.

## Best Possible Solution

Define an explicit urgent system-alert contract that delivers immediate operator-visible notifications while keeping transcript or prompt mutation on a separately ordered, trust-reviewed path.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: the CLI and wake RPC have no priority contract, `wake()` enqueues then requests heartbeat, and heartbeat returns `requests-in-flight` while the target reply run or session lane is busy; no live provider-outage incident was run in this read-only pass.

## Solution Assessment

Unclear. The proposed bypass flag addresses the alert-delay symptom, but the maintainable solution likely needs an immediate operator-visible alert surface plus separately ordered transcript insertion to avoid racing active turns.
