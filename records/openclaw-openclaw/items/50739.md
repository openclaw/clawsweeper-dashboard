## Summary

Keep open. Current main and v2026.6.10 still route `openclaw system event --mode now` through queued system-event plus heartbeat delivery, with no typed priority or queue-bypass contract; the snapshot and pre-failure-trigger comments are useful adjacent mitigation context, but they do not satisfy the immediate in-session alert path.

## What This Changes

Request an in-session system event priority or bypass-queue mode so urgent alerts are not delayed behind congested model requests.

## Best Possible Solution

Define one explicit urgent-alert contract: immediate operator-visible delivery for incident notifications, with prompt-visible transcript injection added only through a separately ordered and trust-reviewed session path.

## Reproduction Assessment

Yes, at source level: current main has no priority/bypass CLI or wake contract, `mode: now` enqueues then wakes heartbeat, and heartbeat skips busy session lanes. I did not run a live provider-outage reproduction in this read-only review.

## Solution Assessment

No, not as a standalone flag yet. The maintainable path is to decide whether OpenClaw needs immediate operator-visible notification, prompt-visible transcript injection, or both before adding a typed CLI/protocol surface.
