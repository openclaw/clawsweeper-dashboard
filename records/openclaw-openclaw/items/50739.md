## Summary

Keep open. Current main and v2026.5.22 still enqueue `openclaw system event --mode now` as a system event plus heartbeat wake, and the heartbeat runner intentionally skips/retries when the main lane, target session lane, or reply run is busy. The alert-latency problem is valid from source, but a bypass mode needs maintainer product/safety direction because it can reorder prompt-visible events against active replies.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so degraded-session alerts can appear immediately instead of waiting behind queued agent work.

## Best Possible Solution

Define an explicit opt-in priority system-event contract that can surface operator alerts without running behind saturated LLM work while preserving the current safe default and preventing active-reply preemption.

## Reproduction Assessment

Yes, at source level: current main enqueues the event and requests a heartbeat, and `runHeartbeatOnce` skips with `requests-in-flight` when the main/session lane or reply run is busy. I did not run a live provider-outage replay.

## Solution Assessment

Unclear as proposed: a raw bypass flag would solve latency but can reorder prompt-visible writes against active work. The safer solution is an explicit, opt-in priority contract with tests for busy lanes, active replies, and default compatibility.
