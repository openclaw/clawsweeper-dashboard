## Label Justifications

- `P2`: This is a normal-priority reliability improvement with real incident evidence but it needs product direction before implementation.
- `impact:session-state`: The request would change how system events interact with serialized session lanes, active replies, and transcript ordering.
- `impact:message-loss`: Delayed alerts can arrive too late to be useful during provider or lane congestion, making them effectively lost as notifications.

## Summary

Keep open. Current main and v2026.5.22 still enqueue system events and wake heartbeat, but heartbeat intentionally skips while the target reply/session lane is active; the requested reliable priority/bypass alert path remains a product and session-ordering decision.

## What This Changes

Request adds a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent in-session alerts can surface during lane congestion.

## Best Possible Solution

Define a maintainer-approved high-priority system alert contract, likely separating immediate operator-visible notification from any ordered prompt/transcript mutation, then add focused busy-lane coverage for that contract.

## Reproduction Assessment

Yes, at source level: `system event --mode now` routes through `wake`, which enqueues a system event and requests heartbeat, while `runHeartbeatOnce` skips on active reply/session lanes. I did not run the original provider-outage incident live in this read-only review.

## Solution Assessment

Unclear. A raw bypass flag would address alert latency but may violate current ordering guards, so maintainers should first choose whether this belongs in a separate alert stream or an ordered high-priority session injection path.
