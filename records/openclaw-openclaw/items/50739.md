## Summary

Keep open. Current main still has no priority or bypass mode for `openclaw system event --mode now`; it routes through queued system events plus heartbeat wake delivery, and busy main or resolved session lanes still defer the heartbeat. The KeepMyClaw and quota-trigger comments are useful operational context, but they do not satisfy the requested immediate in-session alert path.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so urgent in-session alerts can appear during congested provider/request lanes.

## Best Possible Solution

Define a first-class urgent system alert contract that gives operators immediate TUI/UI visibility during degraded provider incidents, while handling transcript or prompt mutation through a separately ordered and trust-reviewed path across the CLI, Gateway wake protocol, heartbeat delivery, UI/TUI surfaces, docs, and regression tests.

## Reproduction Assessment

Yes. The source-level path is high-confidence: CLI sends only `{ mode, text }`, wake enqueues a system event and requests heartbeat, and heartbeat returns `requests-in-flight` when main or resolved session lanes are busy; current tests cover the busy-lane skip behavior.

## Solution Assessment

No/unclear for the raw bypass flag as proposed. Immediate alert delivery is valid, but directly bypassing into trusted prompt-visible system events is not the safest maintainable shape without a separate operator-alert surface and explicit transcript-ordering semantics.
