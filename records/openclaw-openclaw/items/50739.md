## Label Justifications

- `P2`: This is a concrete reliability improvement for system alerts during degraded sessions, but it needs product/safety design rather than emergency repair.
- `impact:message-loss`: The reported failure mode makes urgent system-event alerts arrive too late to be useful and appear lost during outages.
- `impact:session-state`: The requested bypass would intentionally interact with session lane ordering and active reply state.

## Summary

Keep open. Current main and v2026.5.22 still make `openclaw system event --mode now` enqueue a system event and request an immediate heartbeat, but the heartbeat runner intentionally defers while the target reply/session lane is active, so the requested reliable in-session alert bypass remains a product decision rather than implemented behavior.

## What This Changes

Request a high-priority or bypass-queue mode for `openclaw system event` so urgent system alerts can appear immediately during congested agent sessions.

## Best Possible Solution

Define an explicit alert contract that gives operators reliable degraded-system visibility without corrupting session ordering, either via a guarded in-session priority mode or an admin-scoped UI notification path with focused delay and preemption tests.

## Reproduction Assessment

Yes, at source level: enqueue a system event, request `--mode now`, then make the target reply run active or the session lane busy; current main skips the heartbeat with `requests-in-flight`. I did not run the original live provider-rate-limit incident.

## Solution Assessment

Unclear. A bypass flag would address alert latency, but the maintainable fix needs a product decision about whether alerts may interrupt active replies or should use a separate admin/UI notification channel.
