## Summary

Keep open. Current `main` and `v2026.6.6` have the session-targeted wake improvements from the related async-completion work, but they still do not provide a high-priority or bypass-queue system-event mode; `mode now` still enqueues a prompt-visible event and relies on heartbeat, which intentionally skips busy session lanes.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so incident alerts can appear immediately in an active TUI/session even when the normal LLM session lane is congested.

## Best Possible Solution

Define one explicit high-priority alert contract: immediate operator-visible notification delivery, with prompt/transcript mutation either kept separate or ordered through a reviewed session-state path.

## Reproduction Assessment

Yes at source level: current `main` exposes no priority/bypass surface, `mode now` enqueues then wakes heartbeat, and heartbeat skips busy session lanes. I did not run a live provider-outage saturated-lane reproduction in this read-only review.

## Solution Assessment

No, not as a narrow flag alone. The safer solution needs a maintainer decision on operator-visible notification delivery versus prompt-visible transcript injection, including ordering and trust semantics.
