- `impact:session-state`: The requested bypass would intentionally change how system events interact with the serialized session lane and transcript/order safety guards.

## Summary

Keep open. Current main and v2026.5.22 still enqueue `openclaw system event --mode now` through the heartbeat wake path, and the heartbeat runner deliberately skips while the target reply/session lane is active, so the requested reliable in-session bypass remains a core scheduling/product decision.

## What This Changes

Request adding a high-priority or bypass-queue mode for `openclaw system event` so urgent system alerts surface immediately during congested sessions.

## Best Possible Solution

Define a maintainer-approved alert-priority contract, likely either a dedicated system-alert event stream or an explicit high-priority system-event injection path, with ordering guarantees and busy-lane regression coverage.

## Reproduction Assessment

Yes, at source level: `openclaw system event --mode now` routes through `wake`, which enqueues the event and requests heartbeat, while `runHeartbeatOnce` skips on active reply/session lanes. I did not run the original provider-outage scenario live in this read-only review.

## Solution Assessment

Unclear. A raw bypass flag may solve the alert latency but could violate current session-ordering guards, so maintainers should choose the alert stream or injection semantics before a fix PR.
