## Summary

Keep open. Current main and the latest release still route `openclaw system event --mode now` through queued system-event plus heartbeat delivery, and the existing open notification trackers overlap but do not fully settle the TUI/prompt-visible alert contract requested here.

## What This Changes

The issue requests a high-priority or bypass-queue mode for `openclaw system event --mode now` so in-session alerts surface immediately during congested provider-failure lanes.

## Best Possible Solution

Define one deliberate high-priority alert contract: either a UI/TUI notification event separate from prompt mutation, a reviewed prompt-visible system-event bypass with ordering guarantees, or a two-path design that keeps those semantics distinct.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.6` expose no priority/bypass surface, `mode now` enqueues then wakes heartbeat, and heartbeat skips busy session lanes. I did not run a live provider-outage saturated-lane repro.

## Solution Assessment

No, not as a narrow flag alone. The safer maintainable solution needs a product/API decision about operator-visible notification delivery versus prompt-visible transcript injection before code work starts.
