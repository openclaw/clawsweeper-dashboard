## Summary

Keep open. Current main still has the active-session dead end this PR targets, and the PR is the right narrow area, but the latest added fallback-throw regression test is still inconsistent with the existing no-visible-payload fallback path and appears to fail by static control-flow inspection.

## What This Changes

The PR removes early `delivered: false` returns from active requester subagent completion fallback failures, adds regression tests, and adds a changelog entry for the Telegram/push-channel delivery fix.

## Best Possible Solution

Land the focused delivery fix after making the fallback-throw regression test match the actual successful gateway condition, or explicitly updating the source flow so a successful gateway delivery is not negated by repeating a fallback that already threw.

## Reproduction Assessment

Yes. A focused unit path sets `expectsCompletionMessage: true`, an active requester session, `queueEmbeddedPiMessage` returning `false`, a throwing `sendMessage`, non-empty completion events, and a gateway mock without visible payload; the latest PR then retries the same throwing fallback after the gateway call and returns `delivered: false`.

## Solution Assessment

No, not quite as submitted. Removing the early returns is the narrow maintainable fix for the empty-fallback case, but the updated throwing-fallback test needs a visible gateway response or the production path needs a small guard for already-failed pre-gateway fallback delivery.
