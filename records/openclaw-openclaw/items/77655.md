## Summary

Keep open: current main has the generic fallback loop and WebChat retry-grace handling, but I do not see a current-main reproduction or a merged fix that proves the reported Windows/NVIDIA WebChat path now tries the full configured chain before terminal state.

## What This Changes

The issue reports that model timeout fallback chooses a next candidate but the request terminates before all configured fallback models are attempted.

## Best Possible Solution

Add a focused regression that separates provider timeout fallback from run-budget or lane-watchdog timeout, then fix the shared agent/fallback boundary only if the WebChat embedded path still finalizes before the configured provider-timeout chain is exhausted.

## Reproduction Assessment

No. The issue has concrete historical logs and current source exposes the relevant fallback, lifecycle, and lane-timeout paths, but I did not establish a high-confidence current-main Windows/NVIDIA/WebChat reproduction showing later fallback candidates are never invoked.

## Solution Assessment

Unclear. Trying every configured model is correct for provider/model timeout fallback, but not necessarily for a run-budget or command-lane watchdog timeout; the narrow solution needs to preserve that distinction.
