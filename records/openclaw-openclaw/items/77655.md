## Summary

Keep open. Current main has the generic fallback loop and WebChat lifecycle-error grace, but the reported early termination still sits on an unresolved boundary between retryable provider/model timeouts and terminal run-budget or lane-watchdog timeouts.

## What This Changes

The issue asks OpenClaw to continue trying all configured model fallbacks in order when a model timeout occurs instead of ending the WebChat task after the first fallback decision.

## Best Possible Solution

Add a focused regression that distinguishes provider/model timeout fallback from run-budget and command-lane watchdog termination, then adjust the shared agent/fallback boundary only for the confirmed retryable path.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows the relevant current-main path: provider-timeout fallback is intended, but `CommandLaneTaskTimeoutError` is terminal and can interrupt the outer fallback loop after an embedded run timeout.

## Solution Assessment

Unclear as an automatic fix. Trying every configured model is right for retryable provider/model timeouts, but not for terminal run-budget or lane-watchdog timeouts; maintainers need to lock that distinction before implementation.
