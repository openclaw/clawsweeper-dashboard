## Summary

Keep open: the direct-Mistral fix points at the right compat gate, but the patch adds the flag to a shared patch that is also contributed to OpenRouter/custom Mistral-family routes and can override an explicit opt-out. The supplied terminal proof exercises source predicates, not a real OpenClaw/provider request, so merge still needs a scoped compatibility fix or stronger provider proof.

## What This Changes

Adds `supportsPromptCacheKey: true` to the Mistral model transport compat patch and updates Mistral compat tests to expect the new field.

## Best Possible Solution

Enable `prompt_cache_key` for proven direct Mistral endpoints while preserving proxy/custom-route opt-outs, with regression coverage for direct and contributed compat behavior.

## Reproduction Assessment

Yes for the source-level bug: current `main` lacks `supportsPromptCacheKey` in the Mistral patch while the transport only injects `prompt_cache_key` when that flag is true. I did not establish a live Mistral cache-hit reproduction.

## Solution Assessment

No; adding the flag to the shared patch is too broad because provider compat reuses it for proxy/OpenRouter Mistral-family models. The safer solution is a direct-Mistral-scoped opt-in or explicit false preservation with tests and provider proof.

## Review Findings

- **[P1] Preserve proxy opt-out before enabling cache keys:** `extensions/mistral/api.ts:22`
  - body: This constant is not only used for direct Mistral models: `contributeMistralResolvedModelCompat` returns the same object for OpenRouter/custom Mistral-family routes, and the plugin runtime applies patches as `{ ...compat, ...patch }`, so an operator's explicit `compat.supportsPromptCacheKey: false` would be overwritten. Please split the direct Mistral patch from contributed proxy compat, or preserve explicit false and prove those endpoints accept the field.
