## Summary

Keep open: the bug and proof look credible, but the patch still has a correctness gap in the new LRU lookup. It computes cache keys from one last-written registry-state hint, so alternating workspace-scoped and unscoped derived snapshots can still miss cached entries when their registry fingerprints differ.

## What This Changes

The PR changes plugin metadata snapshot memoization from one process-local slot to a capped LRU map, memoizes source-stale derived snapshots, and adds focused memo tests.

## Best Possible Solution

Keep the bounded process-local cache, but make cache probing use the registry state that belongs to each cached entry or another keying strategy that handles distinct derived registry fingerprints without weakening freshness checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level bug class: current main has one snapshot memo slot and the PR body gives beta2 timeout proof, though I did not run the live timeout scenario in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not yet; the LRU direction is right, but the lookup must handle cached entries whose derived registry states differ before this is a reliable fix.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.87

Full review comments:

- **[P2] Probe cached snapshots with their own registry state:** `src/plugins/plugin-metadata-snapshot.ts:716-719`
  - body: `memoKey` includes `registryState.fingerprint`, but this lookup now derives that fingerprint from the single last-written `pluginMetadataSnapshotRegistryStateMemo`. If a workspace-scoped call and an unscoped call cache derived snapshots with different plugin indexes, the later call overwrites the hint; the next call for the earlier scope computes its params with the other scope's registry fingerprint, misses the Map, and rebuilds. That leaves the alternating-bootstrap case unresolved whenever the scoped snapshot differs from the unscoped one; add a regression with different indexes and make lookup use the matching cached registry state.
  - confidence: 0.87
