## Summary

Keep open with one blocking finding. The PR is a focused bug fix with useful real behavior proof, but the new multi-entry cache still computes lookups from a single last-written registry-state hint, so alternating workspace-scoped and unscoped callers can keep missing when their derived registry fingerprints differ.

## What This Changes

The PR changes plugin metadata snapshot memoization from one process-local slot to an 8-entry key-indexed LRU cache and adds focused memo regression tests.

## Best Possible Solution

Keep the bounded per-key metadata snapshot cache, but make lookup use the registry-state fingerprint for the matching cached entry and add a regression test where workspace-scoped and unscoped derived indexes differ.

## Reproduction Assessment

Yes at source level: current main has a single-slot process memo and the PR demonstrates alternating bootstrap keys that need separate entries. I did not run the full CLI timeout path locally because this review is read-only.

## Solution Assessment

No, not yet. The LRU map is the right direction, but a single registry-state hint can still make valid multi-key derived snapshots miss alternately when workspace-scoped discovery changes the registry fingerprint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Match lookups against each cached registry state:** `src/plugins/plugin-metadata-snapshot.ts:716`
  - body: The map can now hold separate snapshots, but every lookup still builds the key from `pluginMetadataSnapshotRegistryStateMemo`, which is only the last cached registry state. If a workspace-scoped caller and an unscoped caller derive different plugin indexes, the last caller's registry fingerprint is used for the other lookup, so the existing entry is missed and the bootstrap rebuild loop continues. Store/use registry-state hints per cached entry, or add equivalent matching coverage with different workspace/unscoped indexes.
  - confidence: 0.82
