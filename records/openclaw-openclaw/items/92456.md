## Summary

Keep open: the feature is useful, but the PR is not merge-ready because the new exec-approvals config field is not part of the gateway protocol contract, and node-host denylist hits can deny instead of creating the promised approval prompt.

## What This Changes

The PR adds an optional per-agent exec approvals denylist that screens analyzed command segments and forces explicit approval for matching commands, with docs and focused tests.

## Best Possible Solution

Add the denylist entry schema to the gateway protocol read/write validators and wire node-host preflight so denylist hits create the same human approval flow as the gateway exec host.

## Reproduction Assessment

Yes for the main review blockers by source inspection: `ExecApprovalsAgentSchema` has `additionalProperties: false` and no `denylist`, while the branch documents and runtime-loads `agents.*.denylist`; node-host direct invoke also bypasses approval preparation for `security=full`/`ask=off`.

## Solution Assessment

No. The runtime matcher is a plausible core, but the best fix must make the new config field part of the protocol contract and either implement node-host approval prompting or explicitly narrow the shipped scope.
