## Summary

The gateway path is coherent, but the node-host implementation does not deliver the feature’s promised human-confirmation workflow. The full/off fast path bypasses approval registration, so the new node policy produces a terminal denial rather than an actionable prompt.

## What This Changes

The PR adds a persisted per-agent exec denylist that forces fresh human approval for matching or unanalyzable commands across gateway and node-host execution paths.

## Best Possible Solution

Evaluate the denylist during node-host preflight, route hits through the existing human approval request flow, and prove a real allow-once round-trip before merging.

## Reproduction Assessment

Yes. Configure a matching denylist and run host=node with security=full and ask=off: the caller takes the direct-invoke path, then the node returns denylist-hit without any registered approval request.

## Solution Assessment

No. Server-side fail-closed enforcement is useful defense in depth, but it is not the best implementation of an “always ask” feature until the node-host caller creates the approval interaction.
