## Summary

Keep open: current main still documents the Discord elevated fallback, but runtime authorization only consults the generic channel-plugin fallback hook, Discord does not register that hook, and `sandbox explain` still reads only explicit elevated allowlists. The open fixing PR at https://github.com/openclaw/openclaw/pull/53259 has closing syntax, so this issue should remain open until that PR lands, is updated, or is superseded.

## What This Changes

Restore the documented Discord elevated authorization fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted, and align `openclaw sandbox explain` diagnostics with runtime authorization.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned `elevated.allowFromFallback` adapter that shares runtime and `sandbox explain` resolution while preserving explicit empty overrides and per-agent restrictions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure the Discord sender only in `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and current main has no Discord fallback hook while `sandbox explain` reads only explicit elevated allowlists.

## Solution Assessment

Yes for the requested direction, but not by closing this issue now. The maintainable fix is to review, land, or supersede the linked PR with a Discord plugin-owned fallback shared by runtime authorization and diagnostics.
