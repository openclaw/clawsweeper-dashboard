## What This Changes

The issue asks OpenClaw to honor `channels.discord.allowFrom` as the documented fallback for Discord elevated authorization and `sandbox explain` diagnostics when `tools.elevated.allowFrom.discord` is omitted.

## Best Possible Solution

Land a narrow Discord fallback fix shared by runtime elevated auth and `sandbox explain`, with tests for omitted fallback and explicit empty override semantics.

## Reproduction Assessment

Yes, at source level: omit `tools.elevated.allowFrom.discord`, set only `channels.discord.allowFrom`, and current main has no Discord `allowFromFallback` implementation for runtime auth or diagnostics. I did not run a live Discord gateway repro because this review is read-only.

## Solution Assessment

Yes, the best fix shape is narrow: implement the documented Discord fallback through the channel/plugin-owned allowlist path and reuse it in `sandbox explain`. The existing open PR is a plausible starting point but needs maintainer/security handling because it is conflicting and unmerged.
