## Summary

Keep open. Current main still documents the Discord elevated allowFrom fallback, but runtime elevated authorization only consults a channel plugin fallback hook that Discord does not implement, and sandbox explain still reads only explicit elevated allowlists. The open #53259 PR is already the implementation candidate with closing syntax, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Discord elevated exec does not honor the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted, and that `sandbox explain` reports the same path as not allowed.

## Best Possible Solution

Land or supersede #53259 with a narrow Discord-owned fallback resolver reused by both runtime elevated authorization and sandbox explain, with regression coverage for omitted fallback, explicit empty override, and diagnostic consistency.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: configure only `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and runtime fallback remains undefined because Discord has no `elevated.allowFromFallback`; sandbox explain likewise reports false because it reads only explicit elevated allowlists.

## Solution Assessment

Yes. The narrow linked PR direction is the best maintainable fix: resolve the documented Discord fallback in one shared path for runtime and diagnostics, without changing provider-wide policy or weakening explicit elevated allowlist overrides.
