## Summary

Keep open: current main still documents the Discord elevated allowFrom fallback, but runtime authorization depends on a channel-plugin fallback adapter that Discord does not register, and sandbox diagnostics still read only explicit elevated allowlists. The open fixing PR at https://github.com/openclaw/openclaw/pull/53259 has closing syntax and remains the right path to review, update, or supersede.

## What This Changes

Discord elevated auth should fall back to `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted, and `sandbox explain` should report that path correctly.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned fallback adapter shared by runtime authorization and `sandbox explain`, preserving explicit empty overrides and per-agent restrictions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure only `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and current main leaves runtime fallback undefined because Discord has no `elevated.allowFromFallback`; diagnostics also read only explicit elevated allowlists.

## Solution Assessment

Yes. The narrow maintainable fix is to implement the documented Discord fallback through the channel plugin adapter and reuse that resolution for diagnostics; the open closing PR should be reviewed, updated, or superseded before creating any duplicate repair lane.
