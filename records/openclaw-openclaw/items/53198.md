## Summary

Keep open: current main still lacks the documented Discord elevated fallback from `channels.discord.allowFrom` into elevated authorization, and `sandbox explain` still appears to resolve only explicit elevated allowlists for this path. The open fork PR is an implementation candidate but has not merged, so this issue should remain open for maintainer security/boundary review rather than being closed as fixed.

## What This Changes

The issue reports that Discord elevated exec on 2026.3.22 blocks users allowlisted only in `channels.discord.allowFrom`, despite docs promising that value as the fallback when `tools.elevated.allowFrom.discord` is omitted, and also reports matching `sandbox explain` diagnostics drift.

## Best Possible Solution

Land or replace the linked narrow Discord-only fix with tests for runtime elevated auth and `sandbox explain`, preserving explicit elevated allowlist overrides and avoiding provider-wide fallback expansion without maintainer approval.

## Reproduction Assessment

Yes from source inspection: current main documents the fallback but the runtime and diagnostic code read only explicit `tools.elevated.allowFrom.discord`, so the reported config shape still denies authorization. I did not run a live Discord gateway repro in this read-only cleanup pass.

## Solution Assessment

Unclear until maintainer review: a narrow Discord-only fallback appears to match the documented contract, but because this authorizes elevated host commands it needs explicit security/boundary review and tests for explicit empty-list override behavior.
