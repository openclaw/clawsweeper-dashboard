## Summary

Keep open: current main and the latest shipped tag still document the Discord elevated fallback, but runtime authorization only uses the generic channel `elevated.allowFromFallback` hook and the Discord plugin does not register one; `sandbox explain` also still reads only explicit elevated allowlists. The open fixing PR at https://github.com/openclaw/openclaw/pull/53259 has closing syntax, so this issue should remain open until that PR lands, is updated, or is superseded.

## What This Changes

The issue asks Discord elevated authorization and `sandbox explain` to honor the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned `elevated.allowFromFallback` adapter that uses the Discord account allowlist resolver and is shared by runtime auth and `sandbox explain`, while preserving explicit empty overrides and per-agent fail-closed restrictions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure the Discord sender only in `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and current runtime has no Discord fallback hook while `sandbox explain` reads only explicit elevated allowlists.

## Solution Assessment

Yes, but not by closing this issue now. The maintainable fix is to review, land, or supersede the linked PR with a Discord plugin-owned fallback shared by runtime authorization and diagnostics.
