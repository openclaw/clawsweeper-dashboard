## Summary

Keep open: current main still documents the Discord elevated allowFrom fallback, but runtime authorization only consults the generic channel-plugin fallback hook, Discord does not register that hook, and sandbox diagnostics still read only explicit elevated allowlists. The open fixing PR at https://github.com/openclaw/openclaw/pull/53259 has closing syntax and should be reviewed, updated, or superseded before closing this issue.

## What This Changes

The issue reports that Discord elevated authorization does not honor the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned `elevated.allowFromFallback` adapter shared by runtime authorization and `sandbox explain`, preserving explicit empty overrides and per-agent restrictions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure only `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and current main leaves runtime fallback undefined while diagnostics read only explicit elevated allowlists.

## Solution Assessment

Yes. The narrow maintainable fix is to implement the documented Discord fallback through the existing channel plugin adapter and reuse that resolution for diagnostics; the open closing PR should be reviewed, updated, or superseded first.
