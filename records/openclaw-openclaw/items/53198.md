## Summary

Keep open: current main still documents the Discord elevated fallback, but runtime authorization only consults a channel-plugin fallback hook that Discord does not register, and `sandbox explain` still reads only explicit elevated allowlists. The open closing PR at https://github.com/openclaw/openclaw/pull/53259 remains the fix path to review, rebase, land, or supersede.

## What This Changes

The issue reports that Discord elevated exec no longer honors the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted, and that `sandbox explain` misreports the same path.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned `elevated.allowFromFallback` adapter backed by `resolveDiscordAccountAllowFrom`, shared by runtime auth and `sandbox explain`, with regression coverage for fallback, explicit-empty override, per-agent restrictions, and account-scoped Discord config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure the Discord sender only in `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and current runtime has no Discord fallback adapter while `sandbox explain` reads only explicit elevated allowlists.

## Solution Assessment

Yes, but not by closing this issue now. The maintainable path is to review, land, or supersede the linked PR with a Discord plugin-owned fallback shared by runtime authorization and diagnostics.
