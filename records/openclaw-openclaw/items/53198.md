## Summary

Keep open. Current main still documents the Discord elevated allowFrom fallback, but runtime authorization only uses a channel plugin fallback hook that Discord does not provide, and sandbox explain still reads only explicit `tools.elevated.allowFrom` values. The open #53259 PR remains the linked implementation candidate, so this issue should not be closed before that PR or an equivalent fix lands.

## What This Changes

Restore the documented Discord elevated authorization fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted, and make sandbox explain report the same effective allowlist.

## Best Possible Solution

Implement a narrow Discord-owned elevated fallback resolver or adapter that feeds both runtime elevated authorization and sandbox explain, with regression coverage for omitted fallback, explicit empty override, and diagnostics consistency.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: configure only `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and runtime fallback remains undefined because Discord has no `elevated.allowFromFallback`; sandbox explain likewise reports false because it reads only explicit elevated allowlists.

## Solution Assessment

Yes. The narrow maintainable path is a Discord-specific fallback resolver shared by runtime and diagnostics; a broad cross-provider policy change would be riskier because related Slack and Telegram reports describe different authorization paths.
