Keep open: current main and the latest release still do not satisfy the documented Discord elevated allowFrom fallback, and the diagnostic path still reads only explicit elevated allowlists. The same-repo fix PR was closed unmerged, while the remaining candidate path needs maintainer security and plugin-boundary review before adoption.

## What This Changes

The issue asks OpenClaw to honor `channels.discord.allowFrom` as the documented fallback for Discord elevated auth when `tools.elevated.allowFrom.discord` is omitted, and to make `sandbox explain` report that path correctly.

## Best Possible Solution

Land or adopt a focused Discord-plugin-owned `elevated.allowFromFallback` implementation shared by runtime auth and `sandbox explain`, with regression coverage for fallback, explicit empty override, per-agent restriction, and account-scoped Discord config.

## Reproduction Assessment

Yes. Source inspection shows the docs promise a Discord fallback, but current runtime auth only uses a plugin fallback hook that Discord does not implement, and `sandbox explain` reads only explicit elevated allowlists.

## Solution Assessment

No current fix has landed. The best repair should keep Discord-specific fallback ownership in the Discord plugin seam and reuse the same resolution path for runtime auth and diagnostics.
