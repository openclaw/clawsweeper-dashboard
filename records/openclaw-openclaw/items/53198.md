## Summary

Keep open: current main still documents the Discord elevated allowFrom fallback, but runtime elevated authorization only uses a channel-plugin fallback hook and the Discord plugin still does not expose that hook; `sandbox explain` also still reads only explicit elevated allowlists. The open fixing PR at https://github.com/openclaw/openclaw/pull/53259 has closing syntax and remains the right implementation path to review or supersede.

## What This Changes

The issue asks for Discord elevated authorization and `sandbox explain` diagnostics to honor the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned `elevated.allowFromFallback` adapter reused by runtime auth and `sandbox explain`, plus regression coverage for fallback, empty override, and account-scoped diagnostics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear on current main: configure only `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and runtime fallback remains undefined because Discord has no `elevated.allowFromFallback`; `sandbox explain` likewise reads only explicit elevated allowlists.

## Solution Assessment

Yes. The narrow maintainable direction is to implement the documented Discord fallback through the Discord channel plugin adapter and share that generic resolution path with diagnostics, rather than adding a duplicate repair lane while the linked PR is open.
