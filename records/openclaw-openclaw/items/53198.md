## Summary

Keep open: current main still documents the Discord elevated allowFrom fallback, but runtime authorization only asks the generic channel `elevated.allowFromFallback` hook and the Discord plugin does not register that hook. The linked fixing PR at https://github.com/openclaw/openclaw/pull/53259 remains open with closing syntax, so this issue should stay open until that PR is reviewed, updated, superseded, or closed.

## What This Changes

The issue reports that Discord elevated exec does not honor the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/53259 with a Discord-owned `elevated.allowFromFallback` adapter shared by runtime authorization and `sandbox explain`, preserving explicit empty overrides and per-agent restrictions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure only `channels.discord.allowFrom`, omit `tools.elevated.allowFrom.discord`, and current main leaves runtime fallback undefined because Discord has no `elevated.allowFromFallback`; diagnostics also read only explicit elevated allowlists.

## Solution Assessment

Yes for the reported issue, but not by closing this issue now. The narrow maintainable fix is to implement the documented Discord fallback through the channel plugin adapter and reuse that resolution for diagnostics, with the open fixing PR reviewed or superseded first.
