## Summary

Keep open: current main and v2026.6.10 still miss the documented Discord elevated allowFrom fallback, and the only open linked implementation path is a conflicting fork PR that changes an elevated authorization boundary.

## What This Changes

This issue reports that Discord elevated exec does not honor the documented fallback from `channels.discord.allowFrom` when `tools.elevated.allowFrom.discord` is omitted, and that `sandbox explain` misreports that path.

## Best Possible Solution

Land a narrow Discord plugin-owned fallback that runtime elevated auth and `sandbox explain` share, with tests for omitted fallback, explicit empty override, per-agent restriction, and account-scoped diagnostics.

## Reproduction Assessment

Yes. Source inspection reproduces the mismatch: docs promise the Discord fallback, runtime only calls a plugin fallback hook, Discord does not register that hook, and `sandbox explain` reads only explicit elevated allowFrom config.

## Solution Assessment

Yes, the narrow best fix is to implement the documented fallback through the Discord-owned plugin adapter and reuse the same resolution in diagnostics instead of duplicating Discord policy in core.
