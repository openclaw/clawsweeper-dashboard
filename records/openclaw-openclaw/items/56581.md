## Summary

Keep open: the PR still contains potentially useful security and approval fixes that are not implemented on current main, but this branch is conflicting, lacks contributor real behavior proof, and has concrete test-env security regressions.

## What This Changes

The PR scrubs known provider auth env vars from exec child environments, changes two-phase exec approvals to stay pending without connected approvers, adjusts webchat/internal approval follow-up routing and copy, and adds live-test HOME/profile staging docs and tests.

## Best Possible Solution

Rebase or split the work: preserve current main's implemented webchat path, move the no-route two-phase approval fix into `approval-shared.ts`, and add provider-env scrubbing only after maintainer/security review confirms the intended env boundary.

## Reproduction Assessment

Yes, source-level. Current main still expires no-client/no-route requests in `handlePendingApprovalRequest` before the two-phase accepted response, and the sanitizer still copies provider auth env vars from inherited `baseEnv` when they are present.

## Solution Assessment

No. The useful fixes are real, but this combined branch targets some obsolete paths, conflicts with current main, and adds unsafe test profile sourcing; a focused replacement or rebase is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move the no-route guard into the shared approval helper:** `src/gateway/server-methods/exec-approval.ts:205`
  - body: Current main routes exec approvals through `handlePendingApprovalRequest`, where no-client/no-route requests still expire before the two-phase accepted response. Apply the guard and regression coverage to the shared helper instead of the stale inline branch.
  - confidence: 0.9
- **[P2] Gate profile loading to live tests:** `test/test-env.ts:338`
  - body: `installTestEnv()` calls `loadProfileEnv(realHome)` unconditionally, so ordinary unit and integration tests can source a developer profile and inherit local provider keys before HOME isolation. Keep profile loading behind live-test mode or an explicit opt-in.
  - confidence: 0.93
- **[P2] Capture live env after profile import:** `test/test-env.ts:336`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, so profile-defined `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR` are absent when live state is staged. Capture the snapshot after any live profile import.
  - confidence: 0.9
- **[P2] Honor profile-loaded real-home opt-in:** `test/test-env.ts:334`
  - body: `allowRealHome` is computed before sourcing `.profile`, so `OPENCLAW_LIVE_USE_REAL_HOME=1` set there is ignored. Recompute the flag after profile loading so the documented opt-in works.
  - confidence: 0.9
