## Summary

Keep this PR open: it fixes a real WhatsApp DM session-isolation bug on current main, but the latest patch still has one narrow correctness blocker in scoped last-route recording for pinned main-DM setups.

## What This Changes

The PR adds a per-call `dmScopeOverride` route option, uses it for inbound WhatsApp DMs, records scoped last-route metadata, and updates WhatsApp docs, changelog, and tests.

## Best Possible Solution

Keep the route override, but adjust WhatsApp last-route handling so main-DM owner pinning only gates writes to the shared main session, while scoped per-contact sessions always record their own delivery context.

## Reproduction Assessment

Yes. Current main source shows inbound WhatsApp DMs pass a direct peer into `resolveAgentRoute` without an override, and the resolver defaults `dmScope` to `main`, producing shared `agent:<agentId>:main` sessions.

## Solution Assessment

No as submitted. The route override is the narrow maintainable direction, but the scoped last-route branch should not reuse the main-session pin guard when writing to a per-contact session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Record scoped WhatsApp last routes despite main-session pins:** `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:263`
  - body: The new scoped-session branch is still gated by `shouldUpdateMainLastRoute`, but that predicate exists to protect writes to `route.mainSessionKey`. In a pinned-main setup, an allowed non-owner DM now gets its own per-contact `sessionKey` but skips `updateLastRoute`, leaving that session without deliveryContext for later delivery. Apply the pin guard only when writing the shared main session.

## Repair Work Prompt

For openclaw/openclaw PR #76314 and issue #76263, keep the existing `dmScopeOverride` approach but fix `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts` so per-contact scoped WhatsApp DM routes update their own last-route deliveryContext even when `pinnedMainDmRecipient` is set. The main-DM owner pin should only suppress writes to `route.mainSessionKey`; when the resolved inbound last-route target is `route.sessionKey` and `route.sessionKey !== route.mainSessionKey`, call `updateLastRoute` for that scoped session regardless of `shouldUpdateMainLastRoute`. Add or update a regression test for a pinned non-owner direct DM with `sessionKey !== mainSessionKey`. Preserve the docs/changelog coverage, do not add a config option, and do not change group routing.
