## Summary

Keep open. The PR fixes the normal WhatsApp inbound DM route and has focused tests, but the same direct-message scoping still does not reach WhatsApp broadcast fan-out, so a configured broadcast contact can still collapse distinct senders into each broadcast agent's main session under default config.

## What This Changes

The PR adds a per-call route dmScope override, applies it to WhatsApp inbound direct messages, updates last-route handling, and adds docs, changelog, and regression coverage for per-contact WhatsApp DM session keys.

## Best Possible Solution

Extend the PR so every WhatsApp direct inbound path, including broadcast fan-out, uses the same per-channel-peer session key shape with regression coverage, then land the docs and changelog update.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp direct messages provide a peer while resolveAgentRoute defaults dmScope to "main", producing agent:<agentId>:main under default config; the remaining PR gap is also source-reproducible because broadcast.ts rebuilds direct fan-out keys from cfg.session?.dmScope.

## Solution Assessment

No. The normal inbound-route fix is narrow and reuses existing per-channel-peer support, but it is incomplete until the same scoped DM behavior is propagated through WhatsApp broadcast route-key construction.

## Review Findings

- **[P2] Propagate the scoped DM route into broadcast fan-out:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:97`

Please feed the WhatsApp per-channel-peer scope through broadcast route-key construction and add direct-broadcast coverage.
