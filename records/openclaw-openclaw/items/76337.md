## Summary

Keep open. Current main still preserves active embedded runs in diagnostics instead of escalating them, and cron timeout cleanup is a separate timeout path rather than the requested stuck-session recovery behavior.

## What This Changes

The issue requests that Gateway diagnostics abort or force-clear cron-triggered active embedded runs after a bounded no-progress timeout.

## Best Possible Solution

Define a maintainer-approved active-run escalation policy that coordinates diagnostic stalled-session recovery with cron timeoutSeconds, agent runtime timeout, and provider idle timeout semantics, then document and test that boundary.

## Reproduction Assessment

No high-confidence live reproduction of the full six-worker CPU incident was established. The recovery-skip behavior is source-reproducible because current main observes active embedded runs by default and the heartbeat never enables active abort.

## Solution Assessment

Unclear. A blanket five-minute force terminate is not obviously the safest fix because current docs and tests intentionally distinguish active work from stale session bookkeeping; tying escalation to explicit timeout policy is safer.
