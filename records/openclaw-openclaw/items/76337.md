## Summary

Keep open. Current main still does not provide the requested diagnostic escalation for active embedded cron runs: active work is classified as non-recoverable, the recovery helper observes active embedded runs unless explicitly opted into aborting, and cron cleanup is a separate outer timeout path.

## What This Changes

The issue asks for diagnostic stuck-session recovery to escalate active embedded cron runs from observe-only to forceful abort after a bounded no-progress timeout.

## Best Possible Solution

Define a maintainer-approved active-work escalation policy for diagnostics and cron, then implement the timeout/config path with focused recovery, cron cleanup, and documentation coverage.

## Reproduction Assessment

No for the full incident: I did not reproduce the six-worker CPU spin on current main. The observe-only recovery path is source-reproducible because active work is non-recoverable and heartbeat never passes `allowActiveAbort`.

## Solution Assessment

Unclear. A direct five-minute force terminate conflicts with current docs/tests that preserve active work, so the safer path is an explicit maintainer policy, likely with configuration and cron/provider timeout alignment.
