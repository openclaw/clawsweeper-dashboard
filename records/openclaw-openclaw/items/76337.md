## Summary

Keep open. Current main still lacks the requested max-age escalation: active work is classified as non-recoverable, and the recovery helper observes active embedded runs unless `allowActiveAbort` is passed; cron timeout cleanup is a separate outer timeout path, not diagnostic stuck-session recovery.

## What This Changes

The issue requests diagnostic stuck-session recovery to escalate active embedded cron runs from observe-only to forced termination after a bounded no-progress and empty-queue timeout.

## Best Possible Solution

Add a bounded, documented active-work escalation path that coordinates diagnostic classification, recovery, cron timeouts, and provider idle timeouts, with regression coverage for active embedded-run cleanup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full six-worker CPU incident, but yes for the core source behavior: current main keeps active embedded runs observe-only/non-recoverable by default and the heartbeat does not enable active abort.

## Solution Assessment

Is this the best way to solve the issue? Unclear; the proposed blanket five-minute force terminate conflicts with current docs and tests that protect active work, so a maintainer-approved escalation policy is safer than a narrow unconditional abort.
