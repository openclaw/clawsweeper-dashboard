## Summary

Current main no longer leaves stalled active embedded runs in indefinite observe-only recovery; diagnostics now escalate after an extended no-progress window and abort-drain the embedded run.

## What This Changes

The issue requests diagnostic recovery to escalate stuck `processing_without_queue` sessions with active embedded runs instead of observing indefinitely.

## Best Possible Solution

Keep the current conservative policy: observe active work briefly, then abort-drain stalled embedded runs after the extended no-progress window; a shorter or configurable timeout should be a separate policy request if needed.

## Reproduction Assessment

No, the indefinite current-main failure path no longer reproduces from source: stalled embedded runs now become recovery-eligible and pass `allowActiveAbort: true` after the extended no-progress threshold. I did not replay the reporter's live macOS CPU incident.

## Solution Assessment

Yes. The current implementation fixes the core failure, avoids aborting merely slow active runs, and has focused diagnostic recovery coverage; the reporter's suggested five-minute cutoff would be a more aggressive policy than current docs/tests choose.
