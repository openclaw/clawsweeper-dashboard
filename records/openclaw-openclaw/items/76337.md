## Summary

Current main and the v2026.5.3 release now escalate stalled embedded diagnostic runs instead of leaving active embedded work in observe-only recovery forever.

## What This Changes

The issue asks for stuck-session recovery to stop indefinitely skipping active embedded cron runs and to abort or escalate after a bounded no-progress timeout.

## Best Possible Solution

Keep the shipped diagnostic escalation policy and ask for a fresh report only if a post-v2026.5.3 run still spins past the extended stall threshold.

## Reproduction Assessment

No high-confidence live reproduction of the six-worker CPU incident was run. The fixed behavior is high-confidence from current source, docs, changelog, and regression coverage showing the formerly indefinite observe-only path now escalates after an extended no-progress stall.

## Solution Assessment

Yes. The shipped fix is safer than a blanket five-minute kill because it keeps active work observe-only initially, then abort-drains only after a documented extended no-progress window.
