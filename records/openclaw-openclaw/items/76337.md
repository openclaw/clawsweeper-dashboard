## Summary

Keep open. Current main still treats active embedded work as non-recoverable/observe-only in diagnostics, while cron timeout cleanup is a separate outer timeout path rather than the requested diagnostic escalation for active embedded runs.

## What This Changes

Request active embedded-run stuck-session recovery to escalate from observe-only to forced cleanup when a cron-spawned processing session has no queue progress past a timeout.

## Best Possible Solution

Define and implement a maintainer-approved active-work timeout/escalation policy across diagnostics, cron, and provider/model timeouts, then cover it with focused recovery and cron cleanup tests plus docs.

## Reproduction Assessment

No high-confidence live reproduction of the six-worker CPU incident is available here. The observe-only recovery path is source-reproducible because active embedded runs are non-recoverable by default and heartbeat never passes `allowActiveAbort`.

## Solution Assessment

Unclear. A blanket five-minute force terminate conflicts with current docs/tests that keep active work non-recoverable; a coordinated timeout policy or config knob tied to cron/agent/provider timeouts is safer.
