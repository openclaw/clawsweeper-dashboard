## Summary

Close as implemented. Current main and the shipped v2026.5.3 line now abort-drain stalled active embedded runs after an extended no-progress window, resolving the `active_embedded_run` observe-only recovery gap reported here.

## What This Changes

Request gateway diagnostics to escalate stuck cron sessions with active embedded runs from observe-only recovery into bounded abort/drain recovery instead of leaving CPU-heavy worker runs alive indefinitely.

## Best Possible Solution

Keep the shipped extended no-progress abort/drain policy; route any remaining OS-level SIGTERM or process-chain leaks to the separate cron/supervisor tracking issue #71710.

## Reproduction Assessment

No. I did not live-reproduce the six-worker CPU incident, and current main no longer reproduces the tracked active-embedded-run observe-only-only path; source and regression tests show it escalates after the extended no-progress window.

## Solution Assessment

Yes. The current implementation is a safer maintainable fix than a blanket five-minute kill because it waits for a documented extended no-progress stall, then uses the existing abort/drain and force-clear recovery path.
