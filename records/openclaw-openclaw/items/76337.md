## Summary

Close as implemented on current main: the permanent `active_embedded_run` observe-only gap is now covered by an extended stalled-embedded-run abort path, with tests and docs. The proof commit is on `main` after `v2026.5.2`, so it is fixed on main but not yet in the latest release.

## What This Changes

The issue requests diagnostic recovery for cron-spawned sessions stuck in `processing_without_queue` with `active_embedded_run` to escalate from observe-only to terminating or abort-draining the active run after a timeout.

## Best Possible Solution

Keep the current main abort-drain policy and ship it in the next release; requests for a shorter or configurable kill window should be tracked separately as product policy.

## Reproduction Assessment

No. The old permanent observe-only behavior does not reproduce on current main because stalled embedded runs now become recovery-eligible after the extended no-progress window and call recovery with `allowActiveAbort: true`.

## Solution Assessment

Yes. The current implementation is a conservative fix: it avoids cutting off slow active work immediately, then abort-drains and force-clears genuinely stalled embedded runs after a documented threshold.
