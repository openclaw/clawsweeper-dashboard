## Summary

Close as implemented. Current main now escalates stalled embedded runs after an extended no-progress window, passes `allowActiveAbort` into stuck-session recovery, abort-drains the active embedded run, and documents/tests that policy; the fix shipped in `v2026.5.3-beta.2`.

## What This Changes

The issue requests that gateway diagnostics stop observing active embedded cron runs forever and terminate or escalate no-progress `processing_without_queue` sessions before they leave worker processes spinning indefinitely.

## Best Possible Solution

Keep the shipped conservative stalled embedded-run abort-drain policy, while leaving separate event-loop saturation and SIGTERM process-lifecycle reports open for their distinct root causes.

## Reproduction Assessment

No live reproduction of the six-worker CPU incident was performed, but current-main source and tests show the reported observe-only-forever path no longer matches current behavior. Stalled embedded runs now become abort-drain eligible after the documented extended no-progress window.

## Solution Assessment

Yes. The current implementation is a narrower maintainable fix than a blanket five-minute kill because it escalates only stalled embedded runs after an extended no-progress window, avoiding interruption of merely slow active work.
