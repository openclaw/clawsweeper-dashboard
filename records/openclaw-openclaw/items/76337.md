## Summary

Close as implemented on current main: stalled embedded runs now escalate from observe-only to abort-drain after an extended no-progress window, and recovery can force-clear the active run/lane when normal abort cleanup does not drain. I found this on main only; no containing release tag was present in this checkout.

## What This Changes

The issue asks gateway diagnostics to escalate active embedded runs stuck in processing with no queue progress from observe-only to bounded abort/force cleanup.

## Best Possible Solution

Keep the current conservative main implementation that observes active work first, then abort-drains stalled embedded runs after the documented extended no-progress window; track separate event-loop saturation or SIGTERM handling reports independently.

## Reproduction Assessment

No live reproduction of the original six-worker incident was run. Current main no longer matches the reported indefinite observe-only behavior because heartbeat escalates stalled embedded runs with allowActiveAbort after the extended no-progress window.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable fix: it preserves observe-only behavior for merely slow active work, then abort-drains and force-clears only after a documented stalled embedded-run threshold.
