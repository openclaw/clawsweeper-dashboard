## Summary

Keep open. Current main still treats active embedded work as non-recoverable diagnostic activity and only force-cleans cron runs through the separate cron timeout path, so the requested active-run escalation is not implemented.

## What This Changes

The issue requests that Gateway stuck-session diagnostics escalate active embedded cron runs from observe-only to abort or force-clear after a bounded no-progress timeout.

## Best Possible Solution

Define a maintainer-approved active-run escalation policy that coordinates diagnostic stalled-session recovery with cron `timeoutSeconds`, agent runtime timeout, and provider idle timeout semantics, then document and test that boundary.

## Reproduction Assessment

No high-confidence live reproduction of the six-worker CPU incident was established. The reported recovery-skip path is source-reproducible because current main observes active embedded runs by default and the heartbeat never enables active abort.

## Solution Assessment

Unclear. A blanket five-minute force terminate is not obviously the safest fix because current docs and tests intentionally distinguish active work from stale session bookkeeping; tying escalation to explicit timeout policy is safer.
