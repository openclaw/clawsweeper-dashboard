## Summary

Keep open. Current main still does not implement the requested active embedded-run escalation: diagnostics either classify active embedded work as non-recoverable stalled/long-running work or, in the recovery helper, explicitly observe active embedded runs unless an unused opt-in flag is supplied. Cron now has timeout cleanup, but that is a separate 60-minute/default or per-job timeout path, not the reported stuck-session recovery behavior.

## What This Changes

The issue requests that Gateway stuck-session recovery escalate active embedded cron runs from observe-only to abort/force-clear after a bounded no-progress timeout.

## Best Possible Solution

Define a maintainer-approved active-run timeout/escalation policy that coordinates cron `timeoutSeconds`, agent/provider idle timeouts, and diagnostic stalled-session recovery, then cover the chosen behavior with focused diagnostics and cron cleanup tests.

## Reproduction Assessment

No full high-confidence live reproduction of the six-worker CPU incident was established. The reported recovery-skip path is source-reproducible: current main observes active embedded runs by default and the heartbeat never enables the force-abort branch.

## Solution Assessment

Unclear. A blanket five-minute force terminate is not obviously the safest fix because current docs/tests intentionally distinguish active work from stale session bookkeeping; a policy tied to explicit cron/agent/provider timeouts or a new config knob is safer.
