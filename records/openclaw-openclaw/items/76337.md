## Summary

Current main now implements the requested escalation: stalled embedded runs are abort-drained after an extended no-progress window instead of remaining indefinitely observe-only, with tests and docs covering the new policy.

## What This Changes

The issue asks for diagnostic stuck-session recovery to escalate active embedded runs instead of observing indefinitely when a cron session is stuck with no queue depth.

## Best Possible Solution

Keep the current conservative staged recovery policy: observe active work first, then abort-drain embedded runs only after the documented extended no-progress window.

## Reproduction Assessment

Yes for the source path: the previously observe-only active embedded run path is now routed by heartbeat into active abort after the extended no-progress threshold. I did not live-reproduce the original six-worker CPU incident.

## Solution Assessment

Yes. The current fix is narrower than an immediate forced kill because it preserves slow active work while preventing indefinite stuck embedded runs from blocking the lane.
