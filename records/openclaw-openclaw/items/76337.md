## Summary

Keep open. Current main still preserves active embedded work as non-recoverable/observe-only in diagnostics, while cron timeout cleanup is a separate outer timeout path rather than the requested active-run escalation.

## What This Changes

Request diagnostic stuck-session recovery to terminate or escalate active embedded cron runs after a bounded no-progress timeout.

## Best Possible Solution

Define a maintainer-approved timeout/escalation policy for stalled active work across diagnostics, cron, and provider/model timeouts, then cover it with focused recovery and cron cleanup tests.

## Reproduction Assessment

No high-confidence live reproduction of the CPU worker leak is available here. The non-escalation path is source-reproducible because active embedded runs are non-recoverable by default and the heartbeat never passes `allowActiveAbort`.

## Solution Assessment

Unclear. A blanket five-minute force terminate conflicts with current docs and tests that protect active work, so the safer path is a maintainer-approved policy or config knob before changing recovery behavior.
