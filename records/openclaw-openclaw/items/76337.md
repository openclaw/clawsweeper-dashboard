## Summary

Close as implemented. Current main now escalates stalled active embedded runs after a documented extended no-progress window instead of observing indefinitely, and the fix is present in v2026.5.3-beta.2.

## What This Changes

The issue requests gateway diagnostic recovery to abort or force-clear active embedded runs that stay processing with no queue progress instead of logging observe-only forever.

## Best Possible Solution

Keep the shipped extended no-progress abort-drain policy for diagnostic recovery; separate event-loop saturation or SIGTERM process-lifecycle problems should remain in their narrower reports.

## Reproduction Assessment

No high-confidence live reproduction of the six-worker CPU incident was attempted in this read-only review. Current main no longer has the reported indefinite observe-only path for stalled embedded runs, based on source, docs, and regression coverage.

## Solution Assessment

Yes. The current implementation is narrower than a fixed five-minute kill switch: it preserves observe-only for active progress, then abort-drains only after a documented extended no-progress window.
