## Summary

Keep this PR open. The direction matches the linked reconnect/Stop bug, but the patch still has a Chat-tab reload gap: the new Stop visibility depends on `sessionsResult`, while reconnect on the Chat tab subscribes to session events without loading `sessions.list`, so a fresh reload can still have no recovered active-run state.

## What This Changes

This PR adds Gateway WebSocket protocol pings, exposes active-run state on session rows, and uses that state in Control UI to keep Stop and session-scoped aborts available after reconnect.

## Best Possible Solution

Keep the PR direction, but make Chat-tab reconnect/reload eagerly recover `sessions.list` active-run state before the Stop control depends on it.

## Reproduction Assessment

Yes. Start a long Control UI run, reload or reconnect on the Chat tab, and inspect current code: `chatRunId` is cleared while the new active-run state is only available after `sessions.list`, which Chat reconnect does not request.

## Solution Assessment

No. The backend/UI model is the right narrow approach, but the patch should also load or bootstrap session active-run state on Chat reconnect/reload so the recovered Stop control is actually available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Load sessions before relying on recovered abort state:** `ui/src/ui/app-render.ts:2367`
  - body: This line now makes Stop depend on `hasAbortableSessionRun(state)`, but that helper only sees server-side active runs after `sessionsResult` is populated. On a fresh reload or reconnect while already on the Chat tab, `connectGateway` only subscribes to session events and `refreshActiveTab` has no Chat case that calls `loadSessions`, so `sessionsResult` can remain null and Stop stays hidden until some unrelated session refresh happens. Please trigger a sessions load on Chat reconnect/reload, or otherwise seed the active-run state before using it for `canAbort`.
