## Summary

Keep open. Current main and the latest release still expose `session_status` as a rendered `/status`-style card with minimal structured metadata, while raw session/last-turn token and cost data remains on gateway usage surfaces or transcript data rather than an agent-facing `session_status` usage contract.

## What This Changes

Expose raw per-session, last-turn, cost, and budget usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or a dedicated usage/budget tool, backed by visibility-aware session totals, last-turn usage, provider-window data, cost, and availability/source semantics.

## Reproduction Assessment

Yes as a source-level feature-gap check, not as a bug reproduction: current main and `v2026.5.20` accept only `sessionKey`/`model` and return rendered `statusText` without a structured `usage` object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate usage or budget tool may be cleaner if maintainers want display status and machine-readable accounting to stay separate.
