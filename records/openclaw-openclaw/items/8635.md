## Summary

Current main and v2026.6.5 still expose `session_status` primarily as a rendered status card; structured token and cost aggregates remain on gateway/operator surfaces, and the merged context-budget estimate is adjacent but does not provide the requested agent-facing session, last-turn, cost, and quota contract.

## What This Changes

Requests a structured agent-facing token, cost, and quota usage API through `session_status`.

## Best Possible Solution

Define a documented typed usage object on `session_status.details` or an equivalent dedicated tool, reuse canonical transcript and provider usage helpers, and preserve the existing visibility guard with explicit source, freshness, and unavailable states.

## Reproduction Assessment

Yes at the source-contract level: current main’s `session_status` schema and returned details have no structured usage object, although its rendered text can include transcript and provider usage lines.

## Solution Assessment

The direction is useful, but the example shape is not yet the best settled contract; maintainers should choose one canonical tool boundary and define visibility, freshness, unavailable, quota, and cost semantics before implementation.

- The proposed object combines transcript accounting, estimated context state, monetary cost, and provider quota windows that have different freshness and availability semantics.

- Provider quota and cost data may be unavailable or auth-dependent, so a stable contract needs explicit source, freshness, and unavailable metadata rather than omitted fields or inferred zeroes.
