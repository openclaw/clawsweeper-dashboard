## Summary

Keep open. Current main still exposes `session_status` as a rendered status-card result, while structured token/cost usage is only available through separate gateway usage surfaces; related #39400 is narrower and does not supersede this broader agent-facing API request.

## What This Changes

Request to expose per-session, last-turn, cost, and quota-window token usage through the agent-facing `session_status` tool.

## Best Possible Solution

Add a documented structured usage contract, likely `session_status.details.usage` or an equivalent supported agent tool, reusing transcript and `sessions.usage` parsers while preserving current visibility checks.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence verification path: `session_status` renders usage into `statusText` and does not expose structured `details.usage`.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. A stable sanitized `details.usage` contract or equivalent supported tool is more maintainable than requiring agents to parse session JSONL directly.
