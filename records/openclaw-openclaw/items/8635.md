## Summary

Keep open. Current main and the latest release still expose `session_status` as a rendered status card with minimal metadata, while reusable structured token/cost data remains on gateway usage RPCs rather than an agent-facing `session_status` usage contract.

## What This Changes

The issue requests a structured token, cost, last-turn, and quota/budget usage API for the agent-facing `session_status` tool.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or an equivalent supported tool, backed by existing session usage data and covered by visibility-aware tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction, but source and docs inspection provide a high-confidence current-behavior check: `session_status` returns rendered status text plus minimal details and does not expose the requested structured usage object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is a plausible narrow path, but an equivalent supported usage tool could also be cleaner if maintainers want to separate display status from machine-readable accounting.

## Risks / Open Questions

- The public schema still needs a product decision: session totals, last-turn usage, context headroom, provider quota windows, and user-defined budgets are related but not the same contract.
- Exposing transcript totals, cost, or provider quota metadata to agents must preserve existing session visibility, sandbox, cross-agent access, and privacy boundaries.
- Provider quota windows can be unavailable, source-dependent, or auth-dependent today, so a stable API should include explicit availability/source metadata rather than silent omission.
