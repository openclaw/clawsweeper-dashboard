## Summary

Keep open. Current main and the latest release still expose `session_status` as rendered status text plus minimal metadata, while the requested structured token, cost, last-turn, and budget API remains absent and is not superseded by the narrower open budget issue.

## What This Changes

Expose machine-readable per-session token usage, last-turn usage, costs, and budget/window data through `session_status` or an equivalent agent-facing API.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or a new supported usage tool, backed by existing session usage data with visibility-aware tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence current-behavior check: `session_status` accepts only `sessionKey`/`model` and returns rendered `statusText` without the requested structured usage object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate supported usage tool may be cleaner if maintainers want to separate display status from machine-readable accounting.

## Risks / Open Questions

- The public schema still needs a maintainer decision because session totals, last-turn usage, provider quota windows, context headroom, and user-defined budgets are related but distinct contracts.
- Exposing transcript totals, costs, or provider quota metadata to agents must preserve existing session visibility, sandbox, cross-agent access, and privacy boundaries.
- Provider usage windows can be unavailable, auth-dependent, or source-dependent, so a stable API should include explicit availability/source metadata instead of silently omitting fields.
