## Summary

Keep open. Current main and the latest release still expose `session_status` as a `/status`-style rendered card with minimal structured metadata, while the requested raw session, last-turn, token, cost, and budget usage API remains a product/API decision.

## What This Changes

Expose structured per-session and last-turn token usage, cost, and budget or quota data through `session_status` instead of requiring transcript JSONL parsing.

## Best Possible Solution

Define an additive documented agent-facing usage contract on `session_status.details` or a dedicated usage/budget tool that reuses existing transcript and gateway usage sources while preserving source, availability, cost, and privacy semantics.

## Reproduction Assessment

Yes, as a source-level feature-gap check rather than a runtime bug reproduction. Current main and `v2026.6.5` accept only `sessionKey` and `model` and return rendered `statusText` without a structured `usage` object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate usage or budget tool may be cleaner if display status and machine-readable accounting should stay separate.
