## Summary

Keep open. Current main still exposes `session_status` as a rendered `/status`-style card with only `statusText` in tool details, while structured token/cost usage remains on separate gateway usage surfaces; adjacent #39400 is narrower and does not supersede this broader agent-facing API request.

## What This Changes

Expose structured per-session and last-turn token/cost usage through the `session_status` tool so agents can budget usage without parsing transcript files.

## Best Possible Solution

Add a documented structured agent-facing usage surface, likely on `session_status.details`, reusing transcript and gateway usage parsers while preserving current visibility checks and unavailable-state metadata.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows the requested structured `session_status` usage API is still absent.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. The maintainable direction is a sanitized structured contract on `session_status` or an equivalent supported tool, not agent-side transcript parsing.
