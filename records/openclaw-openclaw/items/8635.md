## Summary

Keep open. Current main and the latest release still expose `session_status` as a rendered status-card response with minimal metadata, while structured per-session/last-turn token and cost usage remains available only through gateway usage surfaces; the open context-budget PR and adjacent budget issue do not cover this broader agent-facing usage API request.

## What This Changes

The issue requests raw per-session and last-turn token, cost, and budget usage fields from the agent-facing `session_status` tool.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or a dedicated usage/budget tool, backed by visibility-aware session totals, last-turn usage, provider-window data, costs, and availability/source semantics.

## Reproduction Assessment

Yes as a source-level feature-gap check, not as a bug reproduction: current main and `v2026.5.20` accept only `sessionKey`/`model` and return rendered `statusText` without a structured `usage` object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate usage or budget tool may be cleaner if maintainers want to separate display status from machine-readable accounting.
