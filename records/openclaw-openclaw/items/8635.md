## Summary

Keep open. Current main and the latest release still expose `session_status` as rendered status text plus minimal metadata, while the requested structured token, cost, last-turn, and budget API remains absent; https://github.com/openclaw/openclaw/issues/39400 points back here as the canonical structured-usage thread, and adjacent open issues are narrower rather than superseding this broader API request.

## What This Changes

Expose machine-readable per-session, last-turn, token, cost, and budget usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or a dedicated supported usage tool, backed by existing visibility-aware session/gateway usage data with explicit source, availability, cost, and budget semantics.

## Reproduction Assessment

Yes as a source-level behavior check, not as a bug reproduction: current `session_status` accepts only `sessionKey`/`model` and returns rendered `statusText` without a structured usage object. The missing capability is visible from source and docs inspection on current main and `v2026.5.19`.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate supported usage tool may be cleaner if maintainers want to separate display status from machine-readable accounting.

## Risks / Open Questions

- The public schema still needs a maintainer decision because session totals, last-turn usage, provider quota windows, context headroom, and user-defined budgets are related but distinct contracts.
- Exposing transcript totals, costs, or provider quota metadata to agents must preserve existing session visibility, sandbox, cross-agent access, and privacy boundaries.
- Provider usage windows can be unavailable, auth-dependent, or source-dependent, so a stable API should include explicit availability/source metadata instead of silently omitting fields.
- Current structured surfaces such as `sessions_list`, CLI status summaries, and gateway `sessions.usage` are partial or differently scoped; adding another surface without a clear contract could create conflicting meanings.
