## Summary

Keep open: current main still exposes `session_status` as rendered status text plus minimal metadata, while structured session and last-turn token/cost usage remains available only through separate gateway usage surfaces. This needs an explicit maintainer API/product decision before implementation.

## What This Changes

Request to expose raw per-session, last-turn, cost, and budget usage data to agents through `session_status` instead of requiring transcript JSONL parsing.

## Best Possible Solution

Define an additive documented agent-facing usage contract, likely `session_status.details.usage` or a dedicated usage tool, backed by `sessions.usage` and transcript parsers with explicit source/unavailable metadata and existing visibility checks preserved.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence current-behavior check: `session_status` has no structured usage fields, while raw usage data is available through separate gateway RPCs.

## Solution Assessment

Unclear until maintainers choose the public API contract. Reusing `sessions.usage` with visibility and availability metadata is safer than asking agents to parse rendered status text or transcript files directly.
