## Summary

Keep this open. Current main still exposes `session_status` as a rendered `/status`-style status card, while raw structured session/last-turn token and cost data remains available only through separate usage infrastructure such as gateway `sessions.usage`. The adjacent open #39400 is related but narrower, covering unstable displayed usage-window fields rather than this broader agent-facing usage API request.

## What This Changes

Expose per-session, last-turn, cost, and quota-window token usage data through the `session_status` tool so agents can budget usage without parsing transcript JSONL files.

## Best Possible Solution

Add a documented structured agent-facing usage surface, likely by extending `session_status.details` with stable `usage.session`, `usage.lastTurn`, cost, quota-window, source, and unavailable-state fields that reuse existing transcript and gateway usage parsers while preserving current session visibility checks.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows the requested structured `session_status` usage API is still absent.

## Solution Assessment

Unclear until maintainers choose the public schema, but the requested direction is maintainable if it is implemented as a narrow documented contract on `session_status` or an equivalent supported tool. The safer path is sanitized structured totals plus explicit availability metadata, not exposing raw transcript file parsing to agents.
