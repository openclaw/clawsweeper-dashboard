## Summary

Keep open. Current main and the latest release still expose `session_status` as rendered status text plus route metadata, while raw token/cost aggregates remain on separate gateway usage RPCs; the merged context-budget work is useful but only partial, and the related open issues are narrower or adjacent rather than superseding this request.

## What This Changes

Request structured per-session, last-turn, cost, and budget usage data through the agent-facing `session_status` tool so agents can reason about token capacity without parsing transcript files.

## Best Possible Solution

Define one documented structured usage result or dedicated usage tool behind existing session-visibility checks, reusing current transcript and gateway usage parsers while explicitly representing source, freshness, availability, estimated versus measured values, and privacy boundaries.

## Reproduction Assessment

Yes, at source level. Current `session_status` can be inspected directly to confirm structured usage fields are absent while related usage data exists only in rendered status text or separate gateway methods.

## Solution Assessment

Unclear as a final API shape. The requested capability is appropriate, but maintainers should choose the stable response contract and metadata semantics before implementation.
