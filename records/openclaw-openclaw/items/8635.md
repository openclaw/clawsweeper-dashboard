## Summary

Keep open. Current main still exposes `session_status` as a rendered `/status`-style tool result, while structured token/cost usage remains available through separate gateway usage RPCs rather than a documented agent-facing `session_status` usage contract.

## What This Changes

Request exposing per-session, last-turn, budget/window, token, and cost data through the `session_status` tool.

## Best Possible Solution

Add a documented structured agent-facing usage contract, likely on `session_status.details.usage` or an equivalent supported tool, reusing transcript and `sessions.usage` parsers while preserving current visibility checks.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows no structured usage payload in `session_status`.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. The maintainable direction is a sanitized structured contract on `session_status` or an equivalent supported tool, not agent-side transcript parsing.
