## Summary

Keep open. Current main still exposes `session_status` as a rendered `/status`-style tool result, while structured token/cost usage remains on separate gateway usage surfaces and prior maintainer context treats #39400 as adjacent rather than a replacement.

## What This Changes

The issue requests structured per-session token, cost, last-turn, and budget/window usage fields from the agent-facing `session_status` tool.

## Best Possible Solution

Add a documented structured agent-facing usage contract, likely on `session_status.details.usage` or an equivalent supported tool, reusing transcript and `sessions.usage` parsers while preserving current visibility checks.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows no structured usage payload in `session_status`.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. A stable sanitized `details.usage` contract or equivalent supported tool is more maintainable than asking agents to parse session JSONL directly.
