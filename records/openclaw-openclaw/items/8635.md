## Summary

Keep open: current main and latest release still expose `session_status` as a rendered `/status`-style tool result, while structured session, last-turn, cost, and quota/budget usage data remains in separate Gateway/status/transcript usage surfaces.

## What This Changes

Request to expose structured per-session, last-turn, cost, and quota/budget token usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Define a documented structured agent-facing usage surface under existing session visibility checks, with explicit measured, estimated, stale, unavailable, source, and privacy semantics for session totals, last-turn usage, costs, and quota windows.

## Reproduction Assessment

Yes, at source level: current `session_status` schema and return details show no structured `usage` object, while related raw usage data exists only as rendered status text or separate Gateway methods.

## Solution Assessment

Unclear. The requested capability is reasonable, but the sample response is not yet a proven best design because maintainers still need to choose freshness, privacy, quota, last-turn, and cost-source semantics for a stable public contract.
