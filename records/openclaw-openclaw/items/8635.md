## Summary

Keep open. Current main still exposes `session_status` as a rendered status card with minimal structured metadata, while raw session token/cost aggregates exist on separate gateway usage RPCs and are not yet an agent-facing `session_status` usage contract.

## What This Changes

Expose per-session, last-turn, and budget token/cost usage data through `session_status` so agents can budget capacity without parsing transcript JSONL files.

## Best Possible Solution

Define a documented additive agent-facing structured usage surface, likely `session_status.details.usage` or an equivalent dedicated tool, reusing gateway `sessions.usage` and provider usage summaries while preserving the existing `statusText` path and visibility checks.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-behavior check: `session_status` returns a status card and minimal details, while structured usage remains on gateway RPC surfaces.

## Solution Assessment

Unclear until maintainers choose the public API shape. An additive structured details payload is the narrowest likely direction, but source, unavailable-state, cost, and privacy semantics need product/API judgment first.
