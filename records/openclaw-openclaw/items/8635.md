## Summary

Keep open. Current main still exposes `session_status` as a rendered `/status`-style card with `statusText` only, while structured token/cost usage is available through separate gateway usage RPCs; this remains a valid feature/API request that needs maintainer design.

## What This Changes

Expose raw token, cost, last-turn, and budget/window usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Add a documented structured agent-facing usage contract, likely on `session_status.details.usage` or an equivalent supported tool, reusing existing transcript and gateway usage parsers while preserving current visibility checks.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows no structured usage payload in `session_status`.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. The maintainable direction is a sanitized structured contract on `session_status` or an equivalent supported tool, not agent-side transcript parsing.
