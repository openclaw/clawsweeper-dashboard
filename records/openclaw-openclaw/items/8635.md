## Summary

Keep open: current main and the latest release still return `session_status` as a rendered status card plus minimal details, while raw session, last-turn, cost, and quota/budget usage remain in separate Gateway/status/transcript surfaces rather than a stable agent-facing usage API.

## What This Changes

Request to expose raw per-session, last-turn, cost, and budget/quota usage data through the agent-facing `session_status` tool instead of requiring transcript parsing.

## Best Possible Solution

Define one documented structured agent-facing usage surface under existing session visibility checks, with explicit measured, estimated, stale, unavailable, source, cost, and quota semantics.

## Reproduction Assessment

Yes, at source level: current `session_status` schema and result details show no structured `usage` object, while related raw usage data exists only as rendered status text or separate Gateway methods.

## Solution Assessment

Unclear. The requested capability is reasonable, but the sample response is not yet a proven best design because maintainers still need to choose freshness, privacy, quota, last-turn, and cost-source semantics for a stable public contract.
