## Summary

Keep open. Current main still exposes `session_status` as a rendered status-card result with minimal structured details, while raw session totals, last-turn data, cost, and provider-window metadata remain text-only or available through separate gateway usage methods.

## What This Changes

The issue requests structured token, cost, and budget usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Define a documented additive agent-facing usage contract, either on `session_status.details.usage` or an equivalent tool, reusing transcript parsers, `sessions.usage`, and provider usage data while preserving visibility checks and availability metadata.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives high-confidence current-behavior proof: `session_status` accepts only `sessionKey`/`model` and returns rendered `statusText` without structured `details.usage`.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. A sanitized additive payload or dedicated tool that reuses existing usage sources is more maintainable than asking agents to parse JSONL files or rendered status text.
