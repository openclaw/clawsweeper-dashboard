## Summary

Keep open. Current main still exposes `session_status` as a rendered status-card tool result, while structured token/cost usage is available only through separate gateway usage surfaces; related #39400 is narrower and does not cover this broader agent-facing API contract.

## What This Changes

Expose per-session and last-turn token/cost/budget usage through the agent-facing `session_status` tool instead of requiring direct JSONL parsing.

## Best Possible Solution

Add a documented structured usage contract, likely `session_status.details.usage` or an equivalent supported agent tool, reusing transcript and `sessions.usage` parsers while keeping current visibility checks intact.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main gives a high-confidence verification path: `session_status` renders usage into `statusText` and does not expose a structured `details.usage` payload.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. A stable sanitized `details.usage` contract or equivalent supported tool is more maintainable than requiring agents to parse session JSONL directly.
