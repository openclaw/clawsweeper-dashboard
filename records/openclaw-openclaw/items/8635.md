## Summary

Keep open. Current main still exposes session usage through rendered status text or separate gateway usage RPCs, not through a documented structured agent-facing session_status usage payload.

## What This Changes

Expose per-session token, cost, last-turn, and budget/window usage data through session_status or an equivalent agent-facing API.

## Best Possible Solution

Define a documented additive session_status.details.usage contract, or an equivalent supported tool, that reuses transcript, sessions.usage, and provider-window data while preserving existing visibility checks and unavailable/source metadata.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection give a high-confidence current-behavior check: session_status returns rendered statusText and does not expose structured details.usage.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. A stable sanitized usage payload is more maintainable than requiring agents to parse session JSONL or rendered status text directly.
