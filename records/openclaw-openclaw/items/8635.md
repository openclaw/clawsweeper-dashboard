## Summary

Keep open. Current `main` and `v2026.5.20` still expose `session_status` as rendered status text plus minimal metadata, while raw session, last-turn, cost, and budget usage remain outside a documented agent-facing structured contract.

## What This Changes

Expose per-session and last-turn token, cost, and budget data through `session_status` or an equivalent agent-facing structured usage API.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or a dedicated usage/budget tool, backed by visibility-aware session, transcript, and provider usage data with explicit source, availability, cost, and budget semantics.

## Reproduction Assessment

Yes as a source-level feature-gap check, not as a bug reproduction: current `main` and `v2026.5.20` accept only `sessionKey`/`model` and return rendered `statusText` without a structured usage object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate usage or budget tool may be cleaner if maintainers want to separate display status from machine-readable accounting.
