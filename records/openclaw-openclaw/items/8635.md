## Summary

Keep this issue open. Current main and v2026.6.6 still expose `session_status` as a `/status`-style text result with route/model details, while structured session/last-turn token and cost data remains on separate gateway usage RPCs; the merged context-budget work is useful but only partially overlaps this broader agent-facing API request.

## What This Changes

Expose raw per-session, last-turn, cost, and budget usage data through the `session_status` tool so agents can programmatically track token capacity.

## Best Possible Solution

Define a documented structured usage surface behind existing session visibility checks, with explicit measured, estimated, stale, unavailable, and source metadata for session totals, last-turn usage, costs, and quota windows.

## Reproduction Assessment

Yes, at source level. Current `session_status` schema and return object can be inspected to show structured usage fields are absent while related raw usage data exists only as rendered status text or separate gateway methods.

## Solution Assessment

Unclear. The requested capability is reasonable, but the sample response is not yet a proven best design because maintainers still need to choose freshness, privacy, quota, last-turn, and cost-source semantics for a stable public contract.
