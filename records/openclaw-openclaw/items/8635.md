## Summary

Keep open. Current main and latest v2026.5.12 still expose `session_status` as rendered status text plus minimal metadata, while the requested structured token, cost, last-turn, and budget API is absent; the related budget issue is adjacent, not a replacement.

## What This Changes

The issue requests raw per-session token usage, last-turn usage, cost, and budget data from the agent-facing `session_status` tool.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or an equivalent supported tool, backed by existing session usage data with visibility-aware tests and docs.

## Reproduction Assessment

Not applicable as a bug reproduction, but source and shipped-release inspection give a high-confidence current-behavior check: `session_status` returns rendered `statusText` and does not expose the requested structured usage object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate supported usage tool may be cleaner if maintainers want to separate display status from machine-readable accounting.

## Risks / Open Questions

- The public schema still needs a maintainer product decision: session totals, last-turn usage, provider quota windows, context headroom, and user-defined budgets are related but not the same contract.
- Exposing transcript totals, costs, or provider quota metadata to agents must preserve existing session visibility, sandbox, cross-agent access, and privacy boundaries.
- Provider usage windows can be unavailable, auth-dependent, or source-dependent, so a stable API should expose availability/source metadata rather than silently omitting fields.
