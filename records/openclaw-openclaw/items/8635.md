## Summary

Keep open: current main still exposes `session_status` as rendered status-card text plus minimal metadata, while structured session/last-turn token and cost data is only available through separate gateway usage RPCs. The adjacent budget request at https://github.com/openclaw/openclaw/issues/57053 does not replace this broader usage API request, and https://github.com/openclaw/openclaw/issues/39400 was already closed in favor of this issue.

## What This Changes

The issue requests a structured agent-facing token and cost usage API on `session_status`, including session totals, last-turn usage, and budget/quota windows.

## Best Possible Solution

Define an additive documented agent-facing usage contract, likely `session_status.details.usage` or a dedicated usage tool, backed by `sessions.usage` and transcript parsers with explicit source/unavailable metadata and existing visibility checks preserved.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-behavior check: `session_status` has no structured usage fields, while raw usage data is available through separate gateway RPCs.

## Solution Assessment

Unclear until maintainers choose the public API contract. Reusing `sessions.usage` with visibility and availability metadata is safer than asking agents to parse rendered status text or transcript files directly.
