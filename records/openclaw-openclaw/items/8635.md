## Summary

Keep open: current main still exposes `session_status` as a rendered status-card result, while raw token/cost aggregates are available only through separate gateway usage RPCs and the agent-facing API shape needs maintainer product/privacy decisions.

## What This Changes

Request to add structured per-session, last-turn, cost, and budget usage data to the agent-facing `session_status` response.

## Best Possible Solution

Define a documented structured agent-facing usage contract, likely on `session_status.details.usage` or an equivalent supported tool, reusing `sessions.usage` while preserving session visibility checks and explicit unavailable/source metadata.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the requested structured usage object is absent from current `session_status`, while raw session usage is available through separate gateway RPCs.

## Solution Assessment

Unclear until maintainers decide the API contract. Reusing the existing `sessions.usage` data path with visibility and availability metadata is safer than having agents parse rendered status text or transcript files directly.
