## Summary

Keep open. Current main and v2026.6.5 still expose `session_status` primarily as rendered status text with model and route metadata; structured token and cost aggregates remain on operator-scoped gateway RPCs. The merged context-budget work is useful adjacent progress but does not provide the requested agent-facing session, last-turn, cost, and quota-window contract.

## What This Changes

Add a documented structured agent-facing API for per-session and last-turn token usage, cost, and provider budget windows through `session_status` or an equivalent supported tool.

## Best Possible Solution

Define one typed read-only usage result that reuses canonical transcript aggregation and provider-window loaders, includes source, freshness, timestamp, and unavailable metadata, and applies the existing session visibility guard before exposing data for a visible session.

## Solution Assessment

Unclear pending maintainer choice. Extending `session_status.details` is plausible, but a separate read-only usage tool may better preserve the lightweight rendered-status path and make expensive aggregation explicit.

## Risks / Open Questions

- An agent-facing usage API could expose transcript-derived costs or provider quota metadata across sessions unless it applies the existing visibility guard and explicitly limits fields to the requested visible session.
- Session totals, last-turn usage, context estimates, and provider quota windows use different sources and freshness semantics; combining them without source, timestamp, and unavailable metadata would create an unstable automation contract.
- Transcript aggregation may be materially more expensive than the current lightweight status path, so the API shape should avoid making full-history scanning an implicit cost of every `session_status` call.
