## Summary

Keep open. Current main still exposes `session_status` as rendered `/status` text plus minimal metadata, while the requested structured session, last-turn, token, cost, and budget/window usage contract is not present; adjacent structured usage exists only on operator gateway RPCs and does not supersede the agent-facing API request.

## What This Changes

The issue requests structured per-session token, cost, last-turn, and budget/window usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Define an additive documented agent-facing usage contract, likely `session_status.details.usage` or an equivalent supported tool, backed by transcript usage, `sessions.usage`, and provider-window data while preserving existing visibility and privacy checks.

## Reproduction Assessment

Not applicable: this is a feature request, not a bug report. Source and docs inspection provide a high-confidence current-behavior check that `session_status` returns rendered `statusText` and no structured `details.usage`.

## Solution Assessment

Unclear until maintainers choose the public schema and privacy boundary. A stable sanitized usage payload is the maintainable direction; parsing transcript JSONL or rendered status text should remain a workaround, not the contract.

## Risks / Open Questions

- A public agent-facing usage payload must preserve existing session visibility, sandbox, cross-agent access, and privacy boundaries before exposing transcript totals, costs, or provider quota metadata.
- Provider usage windows can be unavailable, stale, or source-dependent today; a stable API should model availability and source explicitly instead of silently omitting fields.
- Maintainers still need to choose whether this belongs as `session_status.details.usage` or as a separate supported agent tool/RPC surface.
