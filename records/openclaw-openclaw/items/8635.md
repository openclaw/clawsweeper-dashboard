## Summary

Keep open: current main and the latest release still expose `session_status` as a rendered status-card tool result, while structured session, last-turn, cost, and quota/budget usage data remains in separate Gateway/status/transcript usage surfaces. The merged context-budget PR is adjacent but does not implement this broader agent-facing usage API.

## What This Changes

Request to expose structured per-session and last-turn token/cost/budget usage data through the agent-facing `session_status` tool.

## Best Possible Solution

Define a documented structured agent-facing usage surface under existing session visibility checks, with explicit measured, estimated, stale, unavailable, source, and privacy semantics for session totals, last-turn usage, costs, and quota windows.

## Reproduction Assessment

Yes. At source level, current `session_status` schema and returned details have no structured usage object, while existing structured usage data is implemented and documented under separate Gateway usage RPCs instead.

## Solution Assessment

Unclear. The requested capability is reasonable, but the sample response is not yet a proven best design because maintainers still need to choose freshness, privacy, quota, last-turn, and cost-source semantics for a stable public contract.
