## Summary

Keep open. Current `main` and `v2026.5.19` still expose `session_status` as rendered status text plus minimal metadata, while raw session, last-turn, cost, and budget usage remain unavailable through an agent-facing structured contract; related items are adjacent rather than superseding this canonical API request.

## What This Changes

The issue requests structured per-session, last-turn, cost, and budget usage data from the `session_status` tool so agents can programmatically track capacity.

## Best Possible Solution

Define an additive documented agent-facing usage contract, either on `session_status.details` or a dedicated supported usage tool, backed by visibility-aware session/gateway usage data with explicit source, availability, cost, and budget semantics.

## Reproduction Assessment

Yes as a source-level behavior check, not as a bug reproduction: current `main` and `v2026.5.19` accept only `sessionKey`/`model` and return rendered `statusText` without a structured usage object.

## Solution Assessment

Unclear until maintainers choose the public API shape and privacy semantics. Extending `session_status.details` is plausible, but a separate usage tool may be cleaner if maintainers want to separate display status from machine-readable accounting.

## Risks / Open Questions

- The public API shape still needs a maintainer decision because session totals, last-turn usage, provider quota windows, context headroom, and user-defined budgets are related but distinct contracts.
- Exposing transcript totals, costs, or provider quota metadata to agents must preserve existing session visibility, sandbox, cross-agent access, and privacy boundaries.
- Provider usage windows can be unavailable, auth-dependent, or source-dependent, so a stable API should include explicit availability/source metadata instead of silently omitting fields.
- Adjacent open issues should be coordinated so `session_status`, budget preflight, OpenRouter exact cost, and gateway `sessions.usage` do not grow conflicting meanings.
