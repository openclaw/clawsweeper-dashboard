## Summary

Keep open: current main still returns `session_status` as a rendered status-card result, while raw session, last-turn, cost, and quota/budget usage remain separate display or Gateway surfaces rather than a stable agent-facing usage API.

## What This Changes

Request to expose per-session, last-turn, cost, and budget token usage fields through the `session_status` agent tool.

## Best Possible Solution

Define a documented structured agent-facing usage surface, likely `session_status.details.usage` or an equivalent supported tool, reusing gateway/transcript usage while preserving visibility checks and explicit source, freshness, and unavailable metadata.

## Reproduction Assessment

Yes, at source level: current `session_status` schema and return details can be inspected to show structured usage fields are absent while related raw usage data exists only as rendered status text or separate Gateway methods.

## Solution Assessment

Unclear. The requested capability is reasonable, but the sample response is not yet a proven best design until maintainers choose freshness, privacy, quota-window, last-turn, and cost-source semantics for a stable public contract.
