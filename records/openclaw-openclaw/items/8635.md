## Summary

Keep open: current main and v2026.6.10 still expose `session_status` as a rendered status-card result plus limited details, while structured session, last-turn, cost, and quota/window usage data remains in separate Gateway/status/transcript surfaces. The useful direction is clear, but the agent-facing response contract and exposure policy still need maintainer product/API review.

## What This Changes

Request adding structured raw token, cost, last-turn, and budget/window usage data to the agent-facing `session_status` tool.

## Best Possible Solution

Define a documented agent-facing usage contract, either on `session_status.details.usage` or an equivalent supported tool, reusing `sessions.usage` and transcript usage parsers while preserving current visibility checks.

## Reproduction Assessment

Yes, at source level: current `session_status` returns a rendered status card and limited details, while the structured usage data remains on separate Gateway usage RPCs and transcript parsers. I did not run runtime tests because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the API shape. Extending `session_status.details` may be the narrowest path, but an equivalent documented usage tool could be better if maintainers want to keep the status-card tool display-oriented.
