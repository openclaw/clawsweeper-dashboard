## Summary

Keep open. Current main does not implement durable Matrix approval reaction targets, and this PR is a draft with the protected maintainer label; I did not find a concrete correctness or security blocker in the proposed patch.

## What This Changes

This PR adds best-effort SDK state persistence for Matrix approval reaction target bindings, updates Matrix reaction handling to read persisted targets on cache miss, and documents/changelogs the behavior.

## Best Possible Solution

Have maintainers choose whether to land this Matrix-only split or the broader #75053 approach, then merge the selected in-memory-first SDK persistence path after the targeted Matrix tests and changed gate have substantive green proof.

## Reproduction Assessment

Not applicable as a feature PR. The current-main verification path is source inspection: Matrix reaction targets are currently process-local only, while the PR adds SDK-store register/lookup/delete behind that cache.

## Solution Assessment

Yes, the proposed direction appears narrow and maintainable: it uses the existing bundled-plugin `api.runtime.state.openKeyedStore` seam, keeps the memory cache as the hot path, and degrades to existing behavior on store failure. The open question is maintainer selection between this split and the broader related PR.
