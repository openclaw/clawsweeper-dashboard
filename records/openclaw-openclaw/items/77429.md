## Summary

Keep open: current main and the latest release still compute Discord startup delay from the sorted configured account list without promoting the configured default account, and the open proof-backed linked PR remains the canonical fix path.

## What This Changes

This issue asks Discord multi-account startup to place the configured primary/default account in the zero-delay startup slot before staggering secondary accounts.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/89744, then close this issue only after the candidate fix merges or maintainers choose not to make default-account-first startup behavior product policy.

## Reproduction Assessment

Yes, source-reproducible: with multiple configured Discord accounts where channels.discord.defaultAccount sorts after secondary accounts, current main uses the sorted account list directly and gives the default account a nonzero startup delay. I did not run a live Discord restart in this read-only review.

## Solution Assessment

Yes, the linked PR is the best current implementation path pending maintainer acceptance of the compatibility behavior. It keeps the change inside the Discord plugin startup helper, reuses existing default-account and runtime/configured predicates, and preserves the existing stagger for secondary accounts.
