## Summary

Keep open: current main still computes Discord account startup delay from the sorted enabled account list, so a configured `defaultAccount` such as `main` can be delayed behind alphabetically earlier specialist accounts. The bug is narrow and source-reproducible in the Discord plugin.

## What This Changes

Request that Discord multi-account startup place the configured default/main account first while preserving the 10s stagger for remaining enabled accounts.

## Best Possible Solution

Use the existing Discord default-account resolver to order the startup-delay candidate list with the resolved default account first, keep the current enabled/configured filtering and deterministic ordering for the rest, and add regression coverage.

## Reproduction Assessment

Yes, by source inspection: with accounts `billy`, `farber`, `lumbergh`, `main`, `maverick`, and `scout`, the shared helper sorts IDs alphabetically and `resolveDiscordStartupDelayMs` assigns `main` index 3, producing a 30s delay. I did not run live Discord startup in this read-only review.

## Solution Assessment

Yes: fixing the ordering inside Discord's startup-delay calculation is the narrowest maintainable solution. The repair should avoid hard-coded account aliases and use the existing `defaultAccount` contract.
