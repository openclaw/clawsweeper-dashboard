## Summary

Keep open: current main and the latest release still compute Discord account startup delay from the filtered account list without promoting the configured default account, and the open proof-backed PR at https://github.com/openclaw/openclaw/pull/89744 is the canonical linked fix path.

## What This Changes

The issue asks Discord multi-account startup to put the configured default or primary account in the zero-delay slot before secondary accounts.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/89744; the desired end state is default-account-first Discord startup while preserving the existing stagger for enabled secondary accounts.

## Reproduction Assessment

Yes. Source inspection shows a config with enabled billy, farber, and main plus channels.discord.defaultAccount set to main can assign main a nonzero startup index because ids are sorted before delay calculation; I did not run a live Discord restart in this read-only review.

## Solution Assessment

Yes. The linked PR's Discord-local fix uses the existing default-account and runtime/configured predicates, which is narrower than hard-coded account aliases or adding another config knob.
