## Summary

Keep open: current main and the latest release still compute Discord startup delay from the sorted enabled/configured account list without promoting the configured default account, and the open proof-backed PR at https://github.com/openclaw/openclaw/pull/89744 remains the canonical fix path.

## What This Changes

The issue asks OpenClaw to start the configured Discord default or primary account before secondary accounts while preserving the existing startup stagger.

## Best Possible Solution

Land or explicitly reject the linked Discord PR after maintainer review, with the configured runtime-enabled default account receiving the zero-delay startup slot and secondary accounts retaining the existing stagger.

## Reproduction Assessment

Yes. Source inspection on current main shows Discord startup delay still uses the sorted startup-account list index, so a configured default account that sorts later can receive a staggered delay.

## Solution Assessment

Yes at the issue level. The best fix belongs in the Discord plugin startup-delay path and should derive priority from the configured, runtime-enabled default account rather than hard-coded account names.
