## Summary

Keep open: current main and the latest release still assign Discord startup delay from the sorted enabled/configured account list without promoting the configured default account, and the open proof-backed PR at https://github.com/openclaw/openclaw/pull/89744 remains the canonical fix path.

## What This Changes

Issue requests that Discord multi-account startup promote the configured primary/default account ahead of secondary accounts while preserving the existing stagger.

## Best Possible Solution

Land or explicitly reject the linked Discord PR after maintainer review, with a regression test that the runtime-enabled configured default account receives the zero-delay startup slot.

## Reproduction Assessment

Yes. Source inspection on current `main` shows Discord startup delay still uses the sorted account-list index while the default-account resolver is separate; a configured default account that sorts later can still get a staggered delay.

## Solution Assessment

Yes, at the issue level. The best fix belongs in the Discord plugin startup-delay path and should derive priority from the configured/runtime-enabled default account rather than hard-coded account names.
