## Summary

Keep open: current main still computes Discord startup delay from the sorted enabled account list, so a configured default account can be delayed behind specialist accounts. The linked fix PR is open and references this issue with closing syntax, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue asks Discord multi-account startup to start the configured default or primary account before applying the existing stagger to secondary accounts.

## Best Possible Solution

Land or reject https://github.com/openclaw/openclaw/pull/89744 after maintainer review; the desired end state is default-account-first Discord startup while preserving the existing stagger for enabled secondary accounts.

## Reproduction Assessment

Yes. Source inspection shows a config with enabled `billy`, `farber`, and `main` plus `channels.discord.defaultAccount: "main"` can assign `main` a nonzero startup index because account ids are sorted before delay calculation; I did not run a live Discord restart in this read-only review.

## Solution Assessment

Yes. Reusing the existing Discord default-account resolver to order startup candidates first is the narrowest maintainable fix; hard-coded account aliases or a new config knob are not needed.
