## Label Justifications

- `P2`: This is a normal-priority Discord multi-account startup bug with a clear source path and limited blast radius.
- `impact:message-loss`: The default/front-desk Discord account can be delayed during restart windows, leaving users talking to specialist accounts while the primary message path is not yet online.

## Summary

Keep open: current main and the latest release still compute Discord startup delay from the sorted enabled account list, so a configured default account such as `main` can start after alphabetically earlier specialist accounts.

## What This Changes

The issue requests Discord multi-account startup ordering that starts the configured primary/default account first while preserving the existing 10s stagger for the remaining accounts.

## Best Possible Solution

Order Discord startup-delay candidates with the resolved configured default account first, then preserve the existing enabled/configured filtering and 10s stagger for all remaining accounts without hard-coded account aliases.

## Reproduction Assessment

Yes, as a source-level reproduction: configure enabled Discord accounts such as `billy`, `farber`, and `main` with `channels.discord.defaultAccount: "main"`; the sorted account list gives `main` a nonzero startup index. I did not run a live Discord restart in this read-only review.

## Solution Assessment

Yes: using the existing `resolveDefaultDiscordAccountId` contract in the Discord startup-delay ordering is the narrowest maintainable fix. The repair should not add a new config option, change the 10s stagger, or hard-code operator account names.
