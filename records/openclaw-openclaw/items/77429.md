## Summary

Keep open: current main and the latest v2026.5.12 source still compute Discord startup delay from the sorted enabled account list, so a configured default account such as `main` can still start after alphabetically earlier specialist accounts. The related closed issue points back here as the canonical tracker, and the reporter added fresh live-release confirmation that the local startup-priority patch is still needed.

## What This Changes

The issue asks Discord multi-account startup staggering to start the configured default or primary account before secondary accounts.

## Best Possible Solution

Order Discord startup-delay candidates with the resolved configured default account first, then preserve the existing enabled-account filtering and 10s stagger for the remaining accounts.

## Reproduction Assessment

Yes. Configure enabled Discord accounts such as `billy`, `farber`, and `main` with `channels.discord.defaultAccount: "main"`; current main gives `main` a nonzero startup index because the sorted account list is indexed directly before the startup sleep.

## Solution Assessment

Yes. The narrow maintainable fix is to use Discord's existing resolved default-account contract when ordering startup-delay candidates, without changing the stagger duration or adding hard-coded deployment-specific aliases.
