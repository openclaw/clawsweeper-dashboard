## Summary

This PR is still a live implementation candidate: current main does not mount host Claude CLI credentials into the Podman setup container, and the branch directly addresses the linked onboarding failure. I found no new line-level correctness defect, but the credential mount remains auth/security sensitive and should stay in maintainer review before merge.

## What This Changes

The branch adds `OPENCLAW_HOST_CLAUDE_DIR` to the Podman launcher allowlist, conditionally mounts a guarded host Claude credential directory read-only during setup, and documents the setup-only override.

## Best Possible Solution

Land this shape only if maintainers accept a setup-only, read-only, private-guarded Claude credential mount as the supported Podman onboarding path; otherwise keep the linked bug open for a different auth handoff design.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the linked bug: current main launches setup without mounting host `~/.claude`, while the Claude CLI credential reader looks under the container home and Anthropic setup throws when that read is null.

## Solution Assessment

Yes, this is a narrow and maintainable fix for the specific Podman setup failure if maintainers accept the credential boundary. The safer alternative is a different auth handoff that avoids mounting host Claude credentials, which is a product/security decision rather than a mechanical PR repair.
