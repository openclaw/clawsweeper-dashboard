## What This Changes

The PR changes `scripts/run-openclaw-podman.sh` to add a setup-only Podman bind mount from a host Claude credentials directory to `/home/node/.claude` when `.credentials.json` exists.

## Best Possible Solution

Land a validated setup-only Claude credential mount that uses the script’s resolved home fallback, validates the host source like other bind mounts, and includes real rootless Podman/SELinux proof.

## Reproduction Assessment

Yes, by source inspection: current setup runs with `HOME=/home/node` and no host `.claude` mount, while the Anthropic setup path reads home-relative Claude CLI credentials and throws the reported error when absent. I did not run the full Fedora/rootless Podman flow.

## Solution Assessment

Not yet. A setup-only mount is the right direction, but this patch should use `EFFECTIVE_HOME`, validate `OPENCLAW_HOST_CLAUDE_DIR`, and include real Podman proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use the resolved home for Claude defaults:** `scripts/run-openclaw-podman.sh:532`
  - body: The script already handles an unset `HOME` by resolving `EFFECTIVE_HOME`, but this new default reads `$HOME` under `set -u`. Environments that rely on the existing fallback now exit with `HOME: unbound variable` before Podman launches, so default from `EFFECTIVE_HOME/.claude` or otherwise guard `HOME`.
  - confidence: 0.92
- **[P2] Validate the Claude bind-mount source:** `scripts/run-openclaw-podman.sh:539`
  - body: `OPENCLAW_HOST_CLAUDE_DIR` is interpolated into a Podman `-v` source without the same mount-source validation used for config and workspace. A value containing `:` or invalid path segments can make Podman reinterpret the volume spec or mount the wrong host path, so validate the source before adding the bind mount.
  - confidence: 0.84
