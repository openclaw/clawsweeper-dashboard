## Summary

Keep open: the new dependency-ordered approach fixes the earlier provider-upsert hazard, but Phase 0 also commits provider deletes before target/auth-store migration, which can leave existing SecretRefs pointing at a deleted provider alias after a later write fault.

## What This Changes

The PR changes `runSecretsApply` to commit provider mutations, satellite/env writes, per-agent SQLite auth-store writes, and final config in dependency order, with two regression tests.

## Best Possible Solution

Split provider additions from deletions: commit only additive provider definitions before store writes, defer provider deletes and full target migrations to the final config commit, add the missing deletion-fault regression, then get secops approval.

## Reproduction Assessment

Yes. Source inspection shows current main commits config before later credential-store writes with only best-effort rollback, and the PR body supplies terminal before/after proof for that fault path; I did not rerun the fault harness in this read-only review.

## Solution Assessment

No. The dependency ordering is closer than the previous revision, but it treats provider deletes like additive definitions; deletes should be deferred until the final config commit or otherwise guarded with rollback-proof coverage.
