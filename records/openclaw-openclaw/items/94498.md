## Summary

Keep open: this is a focused candidate fix for the open secrets-apply credential divergence bug, and the force-pushed head addresses the earlier provider-alias ordering concern with a provider-definition phase plus regression coverage. It still needs human/secops review because the PR changes credential source-of-truth failure states across config and per-agent SQLite auth stores.

## What This Changes

The PR changes `runSecretsApply` to commit provider definitions first, satellite/auth-store writes next, and the full config last, with regression tests for auth-store commit faults and provider-alias resolvability after a config-write fault.

## Best Possible Solution

Land a secops-approved dependency-ordered commit sequence that keeps persisted SecretRefs resolvable and prevents config from claiming credential migrations before the referenced stores commit.

Yes. Current main shows config commits before later credential stores, and the PR body includes terminal before/after proof for an auth-store commit fault with rollback failure; I did not rerun that harness in this read-only review.

## Solution Assessment

Yes, as an acceptable narrow mitigation rather than full cross-store atomicity. The new head preserves provider alias resolvability before store refs and keeps the final migrated config claim last, but secops still needs to approve the residual partial-state contract.
