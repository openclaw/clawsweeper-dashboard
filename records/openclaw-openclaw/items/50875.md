## Summary

Keep open: the feature is useful, but the branch is stale against current hook APIs, has concrete routing correctness gaps, and still lacks inspectable real behavior proof.

## What This Changes

The PR adds a `before_identity_resolve` plugin hook, SDK helper, resolved-peer routing parameter, Slack/MS Teams/WhatsApp integrations, tests, and docs for dynamic direct-message identity resolution.

## Best Possible Solution

Rebase the feature onto `hook-types`, define first-valid identity and session-isolation semantics, propagate resolved identities through all primary and secondary DM routing paths, and add docs/tests plus real setup proof.

## Reproduction Assessment

Yes for review purposes: source inspection on current main confirms the requested hook/helper/resolved-peer route surface is absent, and PR-head inspection exposes concrete stale API and secondary-routing defects. No live channel run was needed to identify those blockers.

## Solution Assessment

No. The additive hook direction is plausible, but this branch is not the best merge shape until it targets current hook APIs, ignores blank resolver results before first-winner selection, covers secondary Slack/WhatsApp routing paths, and documents session-isolation trust semantics.
