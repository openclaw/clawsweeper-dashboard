## Summary

Keep open: the dynamic identity hook is not on current main, but this PR still has rebasing, correctness, security-contract, and real behavior proof blockers before it is mergeable.

## What This Changes

The branch adds a `before_identity_resolve` plugin hook, SDK helper, resolved-peer routing parameter, Slack/MS Teams/WhatsApp integrations, tests, and docs for dynamic DM identity resolution.

## Best Possible Solution

Rebase the hook into the current `hook-types` contract, define first-valid identity semantics and session-isolation trust boundaries, and propagate resolved identities through all primary and secondary DM routing paths before maintainer review.

## Reproduction Assessment

Yes for review purposes: source inspection on current main confirms the requested hook/helper/resolved-peer route surface is absent, and the PR diff exposes concrete stale API and secondary-routing issues. No live channel run was needed to identify those blockers.

## Solution Assessment

No. The additive hook direction is plausible, but this branch is not the best merge shape until it targets current hook APIs, ignores blank resolver results before first-winner selection, covers secondary Slack/WhatsApp routing paths, and documents session-isolation trust semantics.

  - body: Current main keeps hook names, validation, and handler maps in `src/plugins/hook-types.ts`, while this PR adds `before_identity_resolve` only to the older `types.ts` surface. After rebase, `api.on()` validation and the handler map will not know about the new hook.

  - confidence: 0.86

  - confidence: 0.85
