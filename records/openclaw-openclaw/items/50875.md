## Summary

Keep open. Current main does not implement this hook, and the PR still has blocking compatibility and session-isolation gaps: the hook contract is added to an obsolete type surface, the modifying-hook merge policy is passed in the wrong shape for current main, and secondary Slack/WhatsApp routing paths can still split direct-message history.

## What This Changes

The PR adds a `before_identity_resolve` plugin hook, an SDK helper, route/session-key support, Slack/MS Teams/WhatsApp integrations, docs, and focused tests.

## Best Possible Solution

Rebase the feature onto current main, define the new hook in `src/plugins/hook-types.ts`, expose a narrow SDK helper, pass a proper modifying-hook policy with blank-result filtering and first-result stop semantics, thread resolved peer IDs through every direct-message session construction path including Slack system events and WhatsApp broadcast, and document/test the session-isolation trust contract.

## Reproduction Assessment

Yes. The missing feature is reproducible on current main with the symbol search showing no hook/helper/resolved-peer surface, and the PR bugs are reproducible by code inspection with two identity hooks, a blank high-priority result, a Slack DM interaction event, or a WhatsApp direct chat with broadcast enabled.

## Solution Assessment

No. The additive hook direction is plausible, but this PR is not the best current solution until it is rebased onto the active hook contract surface and closes the priority, blank-result, secondary-routing, and session-isolation gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move the hook contract into hook-types:** `src/plugins/types.ts:1368-1396`
  - body: On current main, plugin hook names and handler maps live in `src/plugins/hook-types.ts`, and `src/plugins/types.ts` imports them. Adding `before_identity_resolve` only in `types.ts` leaves `isPluginHookName` and registration validation unaware of the hook, so `api.on("before_identity_resolve", ...)` will not be a reliable public contract after rebase.
  - confidence: 0.87
- **[P2] Pass the identity merge policy as an object:** `src/plugins/hooks.ts:455-457`
  - body: `runModifyingHook` now reads `policy.mergeResults`, but this call passes a bare function. That means the merge callback is not wired as intended on current main, so multiple identity hooks can lose the documented priority behavior or fail type-checking during rebase.
  - confidence: 0.88
- **[P2] Ignore blank canonical IDs before choosing a winner:** `src/plugins/hooks.ts:456`
  - body: The merge expression treats `""` or whitespace as the selected `canonicalPeerId`. `resolveInboundPeerIdentity` later trims that final value to `null`, so a high-priority not-found hook can block a lower-priority resolver from supplying the real canonical identity.
  - confidence: 0.84
- **[P2] Keep Slack system events on the resolved DM session:** `extensions/slack/src/monitor/message-handler/prepare.ts:344-352`
  - body: The new Slack inbound path resolves canonical peer IDs for normal DMs, but block-action and modal system events still rebuild their route through the synchronous system-event resolver with the raw sender ID. Interactive DM history can split between the canonical session and the raw Slack user session.
  - confidence: 0.86
- **[P2] Pass resolved peer IDs into WhatsApp broadcasts:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:78`
  - body: The primary WhatsApp route receives `resolvedPeerId`, but broadcast fan-out still rebuilds per-agent sessions from raw `peerId` in `buildBroadcastRouteKeys`. Direct chats with broadcast enabled will put the main reply path in the canonical session while broadcast agents use the raw peer session.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: The diff needs security attention because plugin-provided canonical IDs become direct-message session keys and can merge user transcripts without a fully documented trust/isolation contract.

Concerns:

- **[medium] Identity plugins can collapse DM session isolation:** `src/routing/session-key.ts:146`
  - body: A plugin-provided `canonicalPeerId` takes precedence over `identityLinks` for direct-message session keys. Since session keys isolate transcript and last-route state, a buggy or compromised resolver can map distinct channel users to the same canonical ID; the hook contract needs explicit isolation warnings, blank/failure semantics, and regression coverage before merge.
  - confidence: 0.82
