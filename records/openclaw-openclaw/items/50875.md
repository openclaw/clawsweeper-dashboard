## Summary

Keep this PR open. Current main still lacks the dynamic DM identity-resolution hook, but this branch is not ready to close or merge as-is because the hook contract has drifted on main and the diff still leaves concrete session-splitting and multi-hook precedence bugs in the proposed implementation.

## What This Changes

The PR adds a `before_identity_resolve` plugin hook, an SDK helper for inbound peer identity resolution, routing/session-key support for a hook-resolved peer id, Slack/MS Teams/WhatsApp DM integrations, docs, and focused tests.

## Best Possible Solution

Add the dynamic identity seam as a current-main-compatible plugin contract: define it in the active hook type files, expose a narrow SDK helper, route an optional resolved peer id through all DM session-key construction paths that can create or append to the same conversation, and cover priority, blank-result fallback, secondary Slack events, WhatsApp broadcast, and session-isolation behavior with focused tests and docs.

## Reproduction Assessment

Yes. The missing feature is reproducible by searching current main for the proposed hook symbols, and the PR bugs are reproducible by code inspection with two identity hooks, a Slack DM interaction/modal event, or a WhatsApp direct chat with broadcast enabled.

## Solution Assessment

No. The feature direction is plausible and additive, but this PR is not the best current solution until it is rebased onto the current hook contract and fixes the priority, blank-result, secondary-routing, and session-isolation issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass the identity merge policy as an object:** `src/plugins/hooks.ts:455-457`
  - body: On current main, `runModifyingHook` reads `policy.mergeResults`; this call passes a bare function, so the merge callback is never used and each non-null hook result replaces the previous one. With two identity hooks, the lower-priority resolver can override the documented first result and route DMs to the wrong canonical session.
  - confidence: 0.88
- **[P2] Ignore blank canonical IDs before choosing a winner:** `src/plugins/hooks.ts:456`
  - body: The proposed merge expression treats `""` or whitespace as an already chosen `canonicalPeerId`. `resolveInboundPeerIdentity` later trims the final value back to `null`, so a high-priority not-found hook can block a lower-priority plugin from resolving the same peer.
  - confidence: 0.84
- **[P2] Route Slack system events through the resolved identity:** `extensions/slack/src/monitor/message-handler/prepare.ts:344-352`
  - body: The new Slack DM path resolves a canonical peer id for normal inbound routing, but block action and modal system events still rebuild their session via `resolveSlackSystemEventSessionKey` using the raw sender id. Interactive DM history can split between `...:direct:<canonical>` and `...:direct:<raw user>` sessions.
  - confidence: 0.86
- **[P2] Pass resolved peer IDs into WhatsApp broadcasts:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:78`
  - body: The primary WhatsApp route receives `resolvedPeerId`, but broadcast fan-out still rebuilds each agent route from raw `peerId` in `buildBroadcastRouteKeys`. Direct chats with broadcast enabled will send the main reply path to the canonical session while fan-out agents use the raw peer session.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: The diff introduces a session-identity control point for plugins, and the current contract does not yet sufficiently constrain or document the cross-user session-collapse risk.

Concerns:

- **[medium] Identity plugins can collapse DM session isolation:** `src/routing/session-key.ts:146`
  - body: A plugin-provided `canonicalPeerId` takes precedence over `identityLinks` for direct-message session keys. Because session keys control transcript and last-route isolation, a buggy or compromised resolver can map two distinct channel users to the same canonical id and expose shared conversation state; this needs explicit contract wording, tests, and ideally clear failure/blank semantics before merge.
  - confidence: 0.82
