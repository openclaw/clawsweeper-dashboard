## Summary

Current main already fixes the reported WebChat stale run-state problem by separating active-run state from session liveness/status and by covering stale `status: "running"` refreshes after local terminal reconciliation. The fix is present on `origin/main` but is not contained in the latest `v2026.5.26` release tag.

## What This Changes

Fix WebChat so completed assistant replies clear the Send/Stop button and In progress indicator even after `sessions.changed` or `sessions.list` refreshes report the session status as running.

## Best Possible Solution

Keep the current-main `hasActiveRun`-based reconciliation and reopen only with a post-`7121f674` build or release that still shows stuck WebChat run state.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current tests model both a stale `sessions.list` row and a later `sessions.changed` event that report `status: "running"` after local terminal completion. I did not run a live WebChat repro during this read-only review.

## Solution Assessment

Yes. Current main uses the narrow maintainable fix: backend payloads expose `hasActiveRun`, and the frontend honors that field instead of deriving active work solely from session `status`.
