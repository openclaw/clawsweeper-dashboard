## Summary

Keep open: this PR targets a real memory-core fallback recovery bug, but it cannot merge as-is because the recovery probe can accept another fallback provider as a successful primary recovery, the new test file currently fails type checking, and real behavior proof is still only test/CI claims.

## What This Changes

The PR adds a memory-core search-time cooldown path that tries to restore the configured primary embedding provider after fallback activation, plus new provider recovery tests.

## Best Possible Solution

Repair the recovery probe so only the configured primary provider counts as recovery, fix the test type errors, then require redacted real outage/recovery proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main's provider creation can return a fallback provider with fallbackFrom, and the PR recovery path currently only checks for any provider. The linked issue gives a concrete Ollama memory_search outage sequence, but no live run was provided in the PR.

## Solution Assessment

No, not as written: search-time recovery is the right layer, but the recovery probe must not use a fallback-enabled request as proof that the primary recovered. The narrow fix is to disable fallback for the probe or treat fallbackFrom as recovery failure while preserving index-identity guards.
