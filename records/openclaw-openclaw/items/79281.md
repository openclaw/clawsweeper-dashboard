## Summary

Keep open: current main has generic current-conversation binding support that reduces some adapter work, but it still lacks the requested sender-scoped default ACP thread-binding SDK preset, explicit group-sharing policy, and manifest diagnostic. This is a public SDK/API and ACP isolation design feature, not a stale cleanup item.

## What This Changes

The issue requests a public SDK preset or opt-in config for default ACP thread binding so channel plugins can avoid custom per-channel binding implementations.

## Best Possible Solution

Design a narrow documented plugin-SDK helper with explicit opt-in, sender-scoped group defaults if accepted, and a manifest diagnostic only when a channel advertises binding placement without either a manager or generic current-conversation support.

## Reproduction Assessment

Yes. Source inspection on current main shows the requested SDK preset, sender-scoped binding key, group-sharing option, and manifest diagnostic are absent; I did not live-run the external WeChat setup.

## Solution Assessment

Unclear pending maintainer design review. A documented explicit SDK helper looks like the narrowest maintainable direction, but automatic defaults and group sharing need product/security sign-off before code work.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because default group binding policy controls sharing of persistent host-side ACP runtime state.

Concerns:

- **[medium] Preserve per-sender ACP isolation in groups:**
  - body: A default binding preset that keys only by conversation could let multiple group members share one persistent ACP process with cwd, files, and tool cache state; the design should make sender scoping the default or require explicit shared-group opt-in.
