## Summary

Keep open: current main has low-level session-binding APIs and a generic current-conversation fallback, but it still lacks the requested sender-scoped default ACP thread-binding preset, opt-in SDK/config design, and manifest-vs-runtime diagnostic. This remains a feature/API design item rather than a cleanup-close candidate.

## What This Changes

The issue requests a plugin-SDK default ACP thread-binding helper or opt-in config path so channel plugins can avoid reimplementing large binding adapters.

## Best Possible Solution

Design a backward-compatible, documented plugin-SDK helper for explicit channel-plugin opt-in, include sender-scoped group isolation semantics if accepted, and add a focused manifest/runtime diagnostic for advertised binding placement without a usable runtime path.

## Reproduction Assessment

Yes for the source-level limitation: current main lacks the requested SDK preset and sender-scoped default, while ACP thread spawning still depends on binding capabilities. I did not live-run openclaw-weixin, so that external impact remains issue-context evidence.

## Solution Assessment

Unclear pending maintainer API/product review. A documented explicit SDK helper is a plausible maintainable direction, but the per-sender default, shared-group opt-in, and possible config flag should be decided before coding.
