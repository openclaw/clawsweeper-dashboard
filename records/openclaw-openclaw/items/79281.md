## Label Justifications

- `impact:session-state`: The requested binding preset controls how persistent ACP runtime state is associated with channel conversations and senders.

## Summary

Keep open. Current main and the latest release still lack the requested public per-sender ACP thread-binding preset or shared-group opt-in policy; the existing generic current-conversation binding is a partial implementation but keys by conversation rather than sender, and the related open items cover adjacent channel/policy work rather than superseding this SDK/API decision.

## What This Changes

The issue requests a public default ACP thread-binding preset or opt-in config flag so channel plugins can register per-sender ACP session bindings without reimplementing large channel-specific adapters.

## Best Possible Solution

Expose one narrow documented SDK helper only after maintainers choose the per-sender default, shared-group opt-in, and plugin-owned config shape, then cover it with SDK contract tests and one reference channel/plugin adoption.

## Reproduction Assessment

Yes at source level: current main lacks the requested SDK preset, the current binding keys do not include sender identity, and ACP thread spawn still fails when no binding capability is available. I did not run a live WeChat, Slack, or LINE repro.

## Solution Assessment

Unclear as a final implementation: a generic SDK helper fits the plugin boundary, but the security default and config/API contract should be chosen by maintainers before implementation.
