## Summary

Keep open. Current main and v2026.6.8 have a generic current-conversation binding path that partially reduces adapter boilerplate, but they still do not provide the requested public per-sender ACP thread-binding preset or the group-sharing opt-in policy; the related open issues cover adjacent session decoupling, auto-injection, and Telegram-specific work rather than superseding this SDK/API decision.

## What This Changes

The issue requests a default ACP thread-binding SDK preset or opt-in config path so channel plugins can register safe per-sender session binding without bespoke adapters.

## Best Possible Solution

Add a narrow documented SDK helper only after maintainers choose the per-sender default, shared-group opt-in, and plugin-owned config shape, then cover it with SDK contract tests and one reference channel/plugin adoption.

## Reproduction Assessment

Yes, at source level: current main lacks the requested SDK preset and the existing generic/account-scoped binding keys do not include sender identity. I did not run a live WeChat, Slack, or LINE channel repro.

## Solution Assessment

Unclear as a final implementation. A generic SDK helper fits the plugin boundary, but current main's generic current-conversation binding is only a partial answer and maintainers still need to choose the per-sender and shared-group policy.
