## Summary

Keep open: current main still has only low-level thread-binding SDK pieces and generic current-conversation binding, not the requested per-sender default ACP binding preset or manifest sanity diagnostic. The exact default keying, group sharing behavior, and optional config flag are public SDK/product decisions, so this is not a conservative cleanup close.

## What This Changes

The issue requests a public plugin-SDK default ACP thread-binding helper or opt-in flag, with per-sender group-safe keys and a warning for manifest binding declarations that lack runtime support.

## Best Possible Solution

Design an additive, documented plugin-SDK helper with explicit plugin opt-in, sender-safe group defaults if accepted, and a manifest-registry warning for binding placement declarations that have neither a manager nor a supported generic path.

## Reproduction Assessment

Yes for the current limitation from source: current main lacks the requested preset, uses account/conversation rather than sender-scoped keys in the shared manager, and ACP thread spawn still requires binding capability. I did not live-run the external WeChat plugin, so that impact remains issue-context evidence rather than direct local reproduction.

## Solution Assessment

Unclear pending maintainer API/product review. A plugin-SDK preset is a plausible maintainable direction, but the group-sharing default and any OpenClaw config flag need explicit design before implementation.
