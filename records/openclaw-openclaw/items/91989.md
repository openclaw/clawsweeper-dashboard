## Label Justifications

- `P1`: The bundled fal video workflow fails after successful billable jobs for all models returning fal’s documented raw result shape.
- `impact:other`: OpenClaw discards successfully generated paid video output and reports failure, which is significant provider functionality loss outside the specific impact taxonomy.

## Summary

The report is valid and still necessary: current `main` and v2026.6.5 discard fal’s documented model-specific top-level video result after a queue job completes, causing successful paid generations to end as an OpenClaw error. No open or merged related PR currently owns this repair.

## What This Changes

Fix fal video generation so completed queue result bodies preserve top-level `video` or `videos` output fields instead of failing with a missing output URL.

## Best Possible Solution

Fetch the completed result once, parse the documented raw model output as `FalVideoResponse`, retain nested-envelope handling only as an explicitly tested compatibility shape, and add regression tests for top-level `video`, top-level `videos`, and malformed results.

## Reproduction Assessment

Yes from source inspection: a completed raw `{ video: ... }` result deterministically loses that field in `readFalQueueResponse` and reaches the missing-output-URL error. A credentialed live fal run was not performed during this read-only review.

## Solution Assessment

Yes, with a small refinement: the best fix is provider-local parsing of the completed endpoint’s model payload, with narrowly tested envelope compatibility if it is intentionally supported; the separate resolution-normalization observation should not broaden this repair.
