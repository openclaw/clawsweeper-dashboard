## Label Justifications

- `impact:other`: The defect can make a bundled paid video-generation workflow unusable, but it does not fit a narrower owned impact category.

## Summary

Keep this issue open: current `main` parses the completed fal result endpoint with a queue-envelope reader that removes raw top-level video fields, and no canonical open PR or merged fix was found. The earlier https://github.com/openclaw/openclaw/issues/61736 addressed oversized-media delivery and passed a live MiniMax check, but it did not cover this distinct result-shape defect.

## What This Changes

Fix fal video queue completion parsing so raw model outputs retain their top-level `video` or `videos` URLs.

## Best Possible Solution

Normalize the fetched result once at the fal plugin boundary according to the current result contract, preserving raw `video` and `videos` outputs while retaining only intentionally supported envelope handling, then add focused regression tests.

## Reproduction Assessment

Yes, from current-main source with high confidence: when fal returns a raw model result, the completed branch passes it through a parser that cannot retain its video fields. A credentialed paid invocation would provide additional live proof but is not required to establish this parser defect.

## Solution Assessment

Yes in scope and ownership: the narrow maintainable fix belongs at the completed-result parsing boundary in the fal plugin. It should use an explicit normalized result shape rather than add a broad speculative fallback stack.
