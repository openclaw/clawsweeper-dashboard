## Summary

Keep open. Current main still has a request-visible auth-profile persistence cascade path, and this issue is paired with the open closing PR at https://github.com/openclaw/openclaw/pull/67077, which should be reviewed, rebased, landed, or replaced before the issue is closed.

## What This Changes

The issue reports that Windows EPERM failures while saving `auth-profiles.json` can turn post-request auth-profile bookkeeping into a fatal gateway-wide model fallback cascade.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/67077 or an equivalent focused patch that makes only post-completion auth-profile success/failure/cooldown persistence best-effort while keeping real credential and OAuth writes fatal.

## Reproduction Assessment

Yes for the source-level cascade path: force `saveAuthProfileStore` to throw after `updateAuthProfileStoreWithLock` returns null, and current main still lets the awaited success/failure bookkeeping save escape. The underlying Windows ReadOnly timing race was not live-reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is to catch and warn-log persistence failures inside post-completion auth-profile bookkeeping paths, not to weaken `saveAuthProfileStore` globally or silently ignore OAuth/credential persistence failures.
