## Summary

Keep open. Current main still has the request-visible cascade path: successful embedded runs await auth-profile bookkeeping, lock-protected persistence failures return `null`, and the direct fallback saves can still throw. The linked #67077 remains open and explicitly closes this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Windows `EPERM` failures while writing `auth-profiles.json` can turn post-success auth-profile bookkeeping into request-level gateway failures.

## Best Possible Solution

Review and land #67077 or an equivalent narrow patch that warn-logs post-success auth-profile bookkeeping persistence failures while preserving fatal behavior for credential and OAuth writes.

## Reproduction Assessment

Yes for the cascade path: current main awaits the affected mark functions, converts lock-save failures to `null`, and then performs uncaught direct fallback saves. The underlying Windows ReadOnly timing race was not live-reproduced here.

## Solution Assessment

Yes for the linked direction: the narrow maintainable fix is to catch and warn-log persistence failures inside the post-success bookkeeping `mark*` paths, not to weaken `saveAuthProfileStore` globally.
