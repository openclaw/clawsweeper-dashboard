## Summary

Keep open. Current main still has the request-visible cascade path: successful embedded runs await auth-profile bookkeeping, lock-protected persistence failures return null, and the direct fallback saves in the mark* bookkeeping functions can still throw. The issue is also paired with open, unmerged PR #67077 using closing syntax, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that auth-profile bookkeeping write failures, especially Windows EPERM on auth-profiles.json after hot reload, stop aborting completed LLM requests and cascading through model fallbacks.

## Best Possible Solution

Keep the issue open until #67077 or an equivalent focused patch makes last-good, usage, failure, and cooldown bookkeeping persistence failures warn-only, with regression coverage for both lock-guarded and direct fallback save paths, while preserving hard failures for credential and OAuth token saves.

## Reproduction Assessment

Yes for the request-level cascade: current source shows successful runs awaiting markAuthProfileGood/Used and the mark* fallback save paths can still throw after a lock-path persistence error. The exact Windows ReadOnly race was not live-reproduced in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to catch and warn-log persistence failures inside the post-result bookkeeping mark* functions. Making saveAuthProfileStore globally non-fatal would be riskier because OAuth token and credential writes still need hard failure semantics.
