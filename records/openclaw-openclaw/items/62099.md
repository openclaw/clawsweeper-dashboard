## Summary

Keep open. Current main still has a request-visible auth-profile persistence cascade path, and https://github.com/openclaw/openclaw/pull/67077 remains the open closing implementation candidate rather than a merged fix.

## What This Changes

The issue reports that Windows EPERM failures while saving `auth-profiles.json` should not turn post-completion auth-profile bookkeeping into full gateway request failure.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/67077 or an equivalent focused patch that makes only post-completion auth-profile success/failure/cooldown bookkeeping persistence best-effort while keeping OAuth and credential writes fatal.

## Reproduction Assessment

Yes for the source-level cascade path: force the locked auth-store update to return null and then have the direct auth-profile save throw EPERM; current main still awaits that mark path. The original Windows ReadOnly race itself was not live-reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is to catch and warn-log persistence failures inside post-completion auth-profile bookkeeping paths, not to weaken `saveAuthProfileStore` globally or make OAuth token saves silently succeed.
