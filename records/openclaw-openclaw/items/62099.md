## Summary

Keep open. Current main still has the request-visible cascade path: successful embedded runs await auth-profile bookkeeping, lock-protected save failures fall through to direct fallback saves, and those direct saves can still throw EPERM. The open, unmerged #67077 is explicitly paired with this issue, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Windows EPERM on `auth-profiles.json` can make post-result auth-profile bookkeeping abort LLM requests and cascade through model fallback.

## Best Possible Solution

Land #67077 or an equivalent focused patch that catches and warn-logs last-good, usage, failure, and cooldown bookkeeping persistence failures while preserving hard failures for credential and OAuth token writes, with regression coverage for lock-guarded and direct fallback save paths.

## Reproduction Assessment

Yes for the cascade path: current source shows successful runs awaiting bookkeeping, lock-path persistence errors returning null, and direct fallback saves that can still throw. The underlying Windows ReadOnly attribute race was not live-reproduced in this read-only review.

## Solution Assessment

Yes for the linked direction: the narrow maintainable fix is to catch and warn-log persistence failures inside the bookkeeping `mark*` functions rather than weakening `saveAuthProfileStore` globally. OAuth/token save failures should continue to propagate.
