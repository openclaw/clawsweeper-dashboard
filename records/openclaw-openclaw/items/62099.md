## Summary

Keep open. Current main still has the request-visible cascade path, and the provided context shows an open, unmerged closing PR (#67077) for this exact issue, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Windows `EPERM` failures while saving `auth-profiles.json` bookkeeping not abort otherwise successful LLM requests or cascade through model fallback.

## Best Possible Solution

Review and land #67077, or an equivalent focused patch, that catches and warn-logs post-success auth-profile bookkeeping persistence failures while preserving fatal credential and OAuth saves.

## Reproduction Assessment

Yes for the cascade path from source inspection: current main awaits the affected mark functions, lock-save failures return `null`, and direct fallback saves can still throw. I did not live-reproduce the underlying Windows ReadOnly timing race.

## Solution Assessment

Yes for the linked direction: the narrow maintainable fix is to catch and warn-log persistence failures inside the post-success bookkeeping `mark*` paths, not to weaken `saveAuthProfileStore` globally.
