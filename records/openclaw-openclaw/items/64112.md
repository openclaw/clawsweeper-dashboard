## Summary

Keep open. Current main still has the raw plugin allowlist fallback for arbitrary missing primaries, so this PR is not implemented-on-main; the PR direction is useful, but it needs a current-main port to the extracted CLI policy helper and the required changelog entry before it is mergeable.

## What This Changes

The PR changes CLI missing-plugin diagnostics to verify bundled plugin CLI metadata before emitting `plugins.allow` or `plugins.entries.*.enabled` guidance, with regression coverage for unknown roots, aliases, and denylist-neutral probing.

## Best Possible Solution

Implement the policy on current main so plugin allowlist and disabled-entry diagnostics are emitted only for real bundled plugin CLI roots or manifest-owned command aliases, while arbitrary unknown roots fall through to the normal unknown-command path.

## Reproduction Assessment

Yes. A source-level reproduction is `resolveMissingPluginCommandMessage("run", { plugins: { allow: ["quietchat"] } })`, which reaches the current raw allowlist branch; `runCli` still calls that helper for unregistered primaries after lazy registration.

## Solution Assessment

Unclear as-is. The PR’s direction is the right maintainable shape, but the best fix must be ported to `src/cli/run-main-policy.ts`, preserve existing alias diagnostics, and include a changelog entry.

## Repair Work Prompt

Treat https://github.com/openclaw/openclaw/pull/64112 and the overlapping https://github.com/openclaw/openclaw/pull/63336 as source context. On current main, update `src/cli/run-main-policy.ts` so `resolveMissingPluginCommandMessage` emits `plugins.allow` or `plugins.entries.<id>.enabled=false` diagnostics only when the missing primary is a real bundled plugin CLI root or a manifest-owned command alias. Unknown roots such as `run` and `lossless` should return null and fall through to the normal unknown-command path. Preserve existing runtime-slash and parent-plugin alias guidance for `dreaming`/`wiki`, handle real bundled roots whose command differs from plugin id such as `voicecall` and `ltm`, make detection independent of runtime allow/deny/disabled state, and fail open to the normal unknown-command path if metadata probing fails. Add focused tests in `src/cli/run-main.test.ts` and a single-line user-facing `CHANGELOG.md` entry. Do not hardcode plugin ids in core beyond registry/manifest metadata and do not change public plugin SDK APIs unless the existing registry surface is insufficient.
