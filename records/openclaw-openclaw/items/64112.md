## Summary

Keep this PR open. Current main still emits the false `plugins.allow` fallback for arbitrary missing primary commands, and this PR contains the right kind of fix, but the branch is now dirty against the current `run-main-policy.ts` split and overlaps another open maintainer-labeled PR in the same diagnostic area.

## What This Changes

This PR changes CLI missing-plugin diagnostics to probe bundled plugin CLI root metadata before emitting `plugins.allow` or `plugins.entries.*.enabled` guidance, with regression coverage for unknown roots, aliased plugin commands, denylist-neutral probing, and probe failures.

## Best Possible Solution

Implement the current-main policy so `plugins.allow` and `plugins.entries.<id>.enabled=false` diagnostics are emitted only for real bundled plugin CLI roots or manifest-owned command aliases, while unknown roots such as `run` fall through to the normal unknown-command path and aliases such as `wiki` or runtime slash commands keep their parent-plugin guidance.

## Reproduction Assessment

Yes. The source-level reproduction is `resolveMissingPluginCommandMessage("run", { plugins: { allow: ["quietchat"] } })`: on current main, the raw allowlist branch still returns a plugin allowlist message for `run`, and `runCli` still calls that helper for unregistered primaries.
