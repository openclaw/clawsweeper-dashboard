## Summary

Keep open. The report is current, concrete, and corroborated by another 2026.4.29 user; current main still does the embedded prep work inside the gateway process, with synchronous system-prompt assembly and synchronous skill/frontmatter loading. The post-release `models.list` timeout fix reduces one catalog-discovery symptom but does not address event-loop starvation while agent prep itself is running.

## What This Changes

The issue asks OpenClaw to prevent agent preparation, especially system-prompt and plugin-tool setup, from blocking the gateway event loop and hanging RPC requests.

## Best Possible Solution

Treat this and #75329 as one gateway performance incident: profile packaged 2026.4.29 and current main while polling `models.list`, `sessions.list`, and `node.list` during an agent run, then move, cache, yield, or offload the blocking prep work so gateway RPC handling stays responsive under normal agent startup.

## Reproduction Assessment

Unclear. The issue plus the Ubuntu corroboration provide strong field evidence, but a high-confidence reproduction still needs the reporter workspace/config/skills or an equivalent packaged setup with event-loop diagnostics enabled during an agent run.

## Solution Assessment

Not applicable because there is no patch here. The adjacent `models.list` timeout is a useful partial mitigation, but the maintainable fix should reduce or isolate blocking prep work rather than only surfacing event-loop health or timing out one RPC dependency.
