## Summary

Keep open. Current main still performs the PI embedded attempt prep and system-prompt assembly inside the gateway process, the skill/frontmatter loader still uses synchronous filesystem work, and the event-loop health/readiness code is diagnostic rather than a mitigation for RPC starvation. The linked #75329 and #75214 overlap, but this issue has distinct current evidence for agent-prep-related API hangs and failover fallout, so it should not be closed as a duplicate yet.

## What This Changes

Report that agent preparation blocks the Gateway event loop, causing `models.list`, `sessions.list`, `node.list`, and failover behavior to stall during OpenClaw 2026.4.29 runs.

## Best Possible Solution

Treat this as a gateway performance incident shared with #75329 and informed by #75214: profile packaged 2026.4.29 and current main during an agent turn, then remove, yield, cache, or isolate the blocking prep work so gateway RPCs and in-flight streams remain responsive while prompt, skill, plugin-tool, and session-resource setup run.

## Reproduction Assessment

Unclear. The report and corroborating comment provide strong timings across two Linux setups, but a high-confidence reproduction still needs a comparable packaged/current-main run with event-loop diagnostics enabled while polling `models.list`, `sessions.list`, and `node.list` during agent preparation.

## Solution Assessment

Not applicable because there is no patch in this issue. The best fix is not to suppress stuck-session or liveness warnings; it should make the prep pipeline non-starving and add regression proof for bounded gateway RPC latency during agent preparation.
