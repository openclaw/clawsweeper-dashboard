## Summary

Keep open. Current main still performs the relevant agent-prep work inside the gateway process, with synchronous prompt/skill assembly paths and diagnostics that expose event-loop saturation rather than preventing RPC starvation. The open #75329 and #75214 reports overlap, but this report has distinct API-hang and downstream failover evidence, so closing it as implemented or duplicate would be premature.

## What This Changes

The issue reports gateway API hangs and severe Node event-loop starvation during agent preparation in OpenClaw 2026.4.29, especially around system-prompt and plugin/tool prep.

## Best Possible Solution

Treat this as a gateway performance incident: profile packaged 2026.4.29 and current main during an agent turn, then make prep non-starving by caching, yielding, moving blocking work off the gateway event loop, or narrowing cold plugin/skill/prompt work, with regression proof that lightweight RPCs stay responsive during prep.

Unclear. The issue has detailed multi-platform measurements and corroboration, but a high-confidence reproduction still needs a packaged/current-main run with event-loop diagnostics enabled while polling models.list, sessions.list, and node.list during agent preparation.

## Solution Assessment

Not applicable because there is no patch in this issue. The best fix is not to suppress diagnostics; it should remove or isolate the blocking prep work and add a bounded-latency proof for gateway RPCs during agent preparation.
