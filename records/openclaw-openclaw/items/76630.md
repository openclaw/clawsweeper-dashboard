## Summary

Close as implemented on current main: doctor repair now handles the Brave web_search externalization path by detecting provider-only Brave config, resolving the official @openclaw/brave-plugin install, and installing missing payloads before final validation. This is present on main after v2026.5.2, but no release tag contains the proof commit yet.

## What This Changes

The issue reports that upgrading to 2026.5.2 leaves configs with tools.web.search.provider set to brave unable to start, and doctor --fix fails instead of installing or enabling the Brave plugin.

## Best Possible Solution

Keep the current main doctor repair path for Brave external-plugin recovery and let the next release ship it; broader update rollback remains separate from this plugin-repair issue.

## Reproduction Assessment

Yes. The reported release failure is source-reproducible from a config with tools.web.search.provider set to brave and no active Brave plugin; current main now routes that same config through doctor missing-plugin repair instead of leaving doctor stuck on validation.

## Solution Assessment

Yes. The current implementation fixes the narrow broken doctor repair path in plugin repair and official catalog resolution, rather than adding a startup migration or a new config fallback.
