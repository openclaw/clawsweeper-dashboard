## Summary

Keep open: the invoke parser and stream-normalizer expansion is useful and the contributor supplied credible terminal proof, but the current PR head still weakens two user-visible sanitizer paths by returning original text before the shared plain-text tool-call stripper runs.

## What This Changes

The PR extends shared plain-text tool-call repair, provider/embedded stream normalization, visible-text sanitizers, and tests to recognize invoke-style XML tool-call text with named parameters and `toolUse` terminal events.

## Best Possible Solution

Keep the invoke parser and stream-normalizer expansion, but narrow the sanitizer exception so explicit prose examples survive without letting true XML-ish tool-call blocks through delivery or history sanitizers.

## Reproduction Assessment

Yes. Source inspection of the PR head shows both new sanitizer helpers return original text before stripping when ordinary prose precedes `<invoke` or `<function=`, which reproduces the leak path without needing to run tests.

## Solution Assessment

No. The shared repair package is the right layer for invoke parsing, but bypassing the whole user-visible stripper is too broad; the safer fix is a narrow example-preservation guard plus regression coverage.
