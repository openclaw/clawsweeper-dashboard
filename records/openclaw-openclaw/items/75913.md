## Summary

Keep open: the PR fixes the noisy recovered IPv4 fallback signal, but it changes the shared fallback-promotion log call and also demotes the more serious pinned Telegram API IP escalation.

## What This Changes

The PR demotes Telegram transport fallback promotion logging from warn to debug, adds a sticky IPv4 fallback logger assertion, and records an Unreleased changelog fix.

## Best Possible Solution

Demote only the recovered sticky IPv4 fallback promotion to debug, keep pinned alternative-IP escalation at warn, and test both log levels.

## Reproduction Assessment

Yes. The existing mocked Telegram fetch tests can trigger both the recovered ETIMEDOUT IPv4 fallback and the ETIMEDOUT plus EHOSTUNREACH pinned-IP escalation without a long live dual-stack run.

## Solution Assessment

No. The current patch changes the shared fallback log site; the narrower maintainable fix is to choose the log level per fallback attempt or encode severity on the attempt metadata.
