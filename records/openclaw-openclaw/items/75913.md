## Summary

Keep open. The PR is directionally useful for the noisy recovered IPv4 fallback report, but it changes the shared promotion log call and also demotes the pinned alternative-IP escalation, which is a broader observability regression than the linked issue needs.

## What This Changes

The branch demotes Telegram transport fallback promotion logging from warn to debug, adds a sticky fallback logger assertion, and records an Unreleased changelog fix.

## Best Possible Solution

Demote only the recovered sticky IPv4 fallback promotion to debug, keep pinned alternative-IP escalation visible at warn, and add a focused test for both log levels.

## Reproduction Assessment

Yes. The existing mocked fetch tests can trigger the recovered sticky IPv4 fallback with `ETIMEDOUT`, and the adjacent pinned-IP escalation test exercises the same shared log path without needing a 24-hour dual-stack live run.

## Solution Assessment

No. The current PR changes the shared log site; a narrower per-attempt log-level choice would solve the noisy warning while preserving the stronger signal for escalation beyond DNS-resolved IPv4.
