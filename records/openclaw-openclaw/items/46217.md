## Summary

Keep open: current main and v2026.6.10 still leave visible Settings text in English, including toolbar chrome, quick-settings chrome, section metadata, and schema-derived field labels. The linked open PRs are useful but partial and currently not merged, so this issue remains the canonical Settings localization tracker.

## What This Changes

The issue requests fuller Chinese localization of the Control UI Settings section.

## Best Possible Solution

Adopt one Settings i18n contract that routes toolbar, quick-settings, section metadata, and schema/uiHint field labels through generated UI locale keys with English fallback, then land focused slices against that contract.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: switch Control UI to zh-CN and open Settings; toolbar, quick-settings, section metadata, and schema-derived form labels still come from English literals or schema metadata.

## Solution Assessment

Unclear for implementation shape. Existing `t()` keys are the right boundary for toolbar and chrome, but schema/uiHint field-label localization needs a maintainer-approved key shape and missing-key fallback contract before a full fix should be automated.
