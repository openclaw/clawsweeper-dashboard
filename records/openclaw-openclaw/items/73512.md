## Summary

Keep open. The linked QMD scheduling bug is real, but this branch is not merge-ready because it bypasses the lexical-mode guard using a resolved default interval and conflicts with current docs and tests.

## What This Changes

The PR changes QMD embed execution and timer scheduling so a positive resolved embed interval bypasses the lexical search-mode guard, and adds a regression test for lexical search with an explicit embed interval.

## Best Possible Solution

Decide the lexical-mode QMD contract first; if explicit `embedInterval` is an opt-in, carry an explicit raw-config flag through resolved QMD update config and align tests, docs, and changelog, otherwise keep the current BM25-only behavior and direct vector users to `vsearch`, `query`, or manual `qmd embed`.

## Reproduction Assessment

Yes. Current main has a focused unit scenario with `searchMode: "search"` and `embedInterval: "5m"` that advances timers and expects no QMD commands, while this PR adds the opposite expectation for the same effective setup.

## Solution Assessment

No. The proposed implementation is not the narrowest maintainable fix because it treats a defaulted positive `embedIntervalMs` as explicit user intent; the safer path needs explicit-config tracking plus contract-aligned docs and tests.
