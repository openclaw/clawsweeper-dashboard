## Summary

Keep open. The PR targets a real report in #73432, but the patch changes the documented QMD lexical-mode contract without preserving whether `embedInterval` was explicitly configured, so default `searchMode: "search"` setups can start running embeds again.

## What This Changes

The PR changes QMD embed execution and timer scheduling so a positive resolved `embedIntervalMs` bypasses the lexical search-mode vector guard, and adds a regression test expecting `update` plus `embed` for `searchMode: "search"` with `embedInterval: "5m"`.

## Best Possible Solution

Decide the QMD lexical-mode contract first. If explicit `embedInterval` should opt lexical setups into embedding, carry an explicit raw-config flag through `ResolvedQmdUpdateConfig`, use that flag in both embed guards, update the conflicting tests, docs/schema help, and changelog. If lexical mode must stay BM25-only, keep the current contract and point users to `query`/`vsearch` or manual `qmd embed` when vectors are desired.

## Reproduction Assessment

Yes. Current main has a focused unit-level reproduction: configure QMD with `searchMode: "search"` and `embedInterval: "5m"`, advance timers, and the existing test expects no `update` or `embed` calls; the PR adds the opposite expectation for the same effective setup.

## Solution Assessment

No. The proposed direction may be valid after maintainer approval, but this implementation uses resolved `embedIntervalMs > 0` as user intent even when the value came from the default; the narrower maintainable fix needs explicit-config tracking plus aligned tests and docs.
