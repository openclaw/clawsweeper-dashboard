## Summary

Keep open. This PR targets real QMD embed scheduling pain from #73432, but it is not merge-ready because it changes the lexical-mode contract using a resolved default value and leaves tests/docs pointing in the opposite direction.

## What This Changes

The PR changes QMD embed execution and timer scheduling so positive `embedIntervalMs` bypasses the lexical search-mode vector guard, and adds a regression test for `searchMode: "search"` with `embedInterval: "5m"`.

## Best Possible Solution

Decide the lexical-mode QMD contract first. If explicit `embedInterval` should opt BM25 setups into embedding, carry an explicit raw-config flag through the resolved QMD update config, use that flag in both embed guards, and align tests, docs, and changelog; otherwise preserve the current BM25-only behavior and direct vector users to `vsearch`, `query`, or manual `qmd embed`.

## Reproduction Assessment

Yes. Current main has a focused unit reproduction with `searchMode: "search"` and `embedInterval: "5m"` that advances timers and expects no QMD commands; this PR adds the opposite expectation for the same effective setup.

## Solution Assessment

No. The proposed implementation is not the narrowest maintainable fix because it treats a defaulted positive `embedIntervalMs` as explicit user intent; the safer implementation needs explicit-config tracking plus contract-aligned docs/tests.
