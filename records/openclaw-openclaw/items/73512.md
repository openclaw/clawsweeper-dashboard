## Summary

Keep open. The linked QMD scheduling problem is plausible, but the latest branch is still not merge-ready because it infers explicit user intent from a resolved interval that defaults to 60m, which can make default lexical QMD configurations run embeddings despite the current BM25-only contract.

## What This Changes

The PR changes QMD embed execution and timer scheduling to bypass the lexical search-mode guard when the resolved embed interval is positive, and updates QMD manager tests for that expectation.

## Best Possible Solution

Preserve explicit interval intent in resolved QMD config if maintainers accept it as an opt-in, then align tests, docs, and changelog with that settled contract.

## Reproduction Assessment

Yes, source-level. Current main has a focused lexical-mode test with explicit `embedInterval: "5m"` expecting no QMD commands, and the PR adds the opposite expectation for the same effective behavior.

## Solution Assessment

No. The proposed guard is not the best fix because it treats the defaulted positive `embedIntervalMs` as explicit user intent; the safer path needs explicit-config tracking plus docs/tests that match the chosen contract.
