## Summary

Keep open: the PR targets a real linked QMD scheduler complaint, but current docs/tests define lexical `searchMode: "search"` as no-embed, and this diff infers explicit opt-in from a resolved default `embedIntervalMs`. It needs maintainer product review and a narrower implementation before merge.

## What This Changes

The PR changes QMD embed run/timer guards to allow a positive resolved embed interval in lexical search mode and updates QMD manager tests around that behavior.

## Best Possible Solution

If maintainers want explicit `embedInterval` to opt into embeddings in lexical mode, carry explicitness through resolved config and align docs/tests; otherwise keep the BM25-only contract and direct users to `query` or `vsearch` for embedding maintenance.

## Reproduction Assessment

Yes, source-level. Current main resolves omitted/default QMD search mode to lexical `search`, then the manager blocks both embed execution and scheduling; the focused current test asserts no QMD commands for `searchMode: "search"` plus `embedInterval: "5m"`.

## Solution Assessment

No. The proposed guard uses resolved `embedIntervalMs`, which defaults to 60 minutes and loses explicitness; the safer path is an explicit-config signal plus a maintainer-confirmed docs/test contract.
