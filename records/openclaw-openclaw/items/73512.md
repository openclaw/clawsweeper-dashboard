## Summary

Keep open. The linked QMD scheduler problem is plausible, but this PR is not merge-ready because it infers explicit user intent from a defaulted resolved interval and conflicts with the current BM25-only docs and tests.

## What This Changes

The PR changes QMD embed execution and timer scheduling so a positive resolved embed interval bypasses the lexical search-mode guard, and updates QMD manager tests for that behavior.

## Best Possible Solution

Settle the lexical-mode QMD contract first; if explicit `embedInterval` is accepted as an opt-in, preserve raw-config explicitness in resolved QMD config and align tests, docs, and changelog.

## Reproduction Assessment

Yes, source-level. Current main has a focused `searchMode: "search"` plus `embedInterval: "5m"` test expecting no QMD commands, while this PR adds the opposite expectation for the same effective setup.

## Solution Assessment

No. The proposed guard is not the narrowest maintainable fix because it treats the default 60m `embedIntervalMs` as explicit user intent; a safer path needs explicit-config tracking and contract-aligned docs/tests.

## Review Findings

Overall correctness: patch is incorrect

Default lexical QMD configs can therefore run `qmd embed` through boot or first update paths, contrary to the current BM25-only contract.
