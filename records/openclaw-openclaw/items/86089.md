## Label Justifications

- `merge-risk: 🚨 message-delivery`: The diff changes which channel target receives recovered restart replies and can still use the wrong persisted route under overlap.

## Summary

Keep open: the PR addresses a valid restart-recovery message-loss bug and now includes useful QA Gateway proof, but the direct-agent path still writes persisted restart delivery context without the ownership guard needed to avoid stale or overlapping runs overwriting the route recovery will later trust.

## What This Changes

The branch persists restart recovery delivery context for direct agent and auto-reply runs, uses it for resumed or failure delivery, normalizes new session-store fields, reserves plugin slot keys, and adds focused tests.

## Best Possible Solution

Keep the restart-recovery route persistence design, but make every durable route write atomic against the current session/run owner before allowing recovery to use it for visible delivery.

## Reproduction Assessment

Yes from source, but not from an executed live repro in this read-only review. Current main resumes restart-aborted sessions with delivery disabled, while the linked report describes the resulting silent Discord recovery path.

## Solution Assessment

No, not yet. The proposed storage and recovery boundary is the right direction, but the direct-agent write needs the same atomic ownership guard already used in the auto-reply path.
