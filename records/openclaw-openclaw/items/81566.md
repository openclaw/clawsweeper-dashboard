## Summary

Keep open: current main and the latest release still reject signed non-message Nextcloud Talk webhook variants with HTTP 400 before the existing ignore path can acknowledge them. The related fix PR had a focused implementation and real webhook proof, but it closed unmerged, so the issue remains actionable.

## What This Changes

The issue asks the Nextcloud Talk channel webhook handler to acknowledge signed non-message Talk events instead of returning 400 Invalid payload format.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so signed JSON is parsed before strict message validation, signed non-message Activity Streams events are acknowledged without invoking the message handler, and malformed JSON or malformed message-shaped payloads still return 400.

## Reproduction Assessment

Yes. A signed JSON Talk Activity Streams payload outside the strict Create/Person/Note/Collection schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format; I verified the path from current source and shipped tag, not from a live Nextcloud instance.

## Solution Assessment

Yes. The narrow maintainable fix is to distinguish malformed JSON and malformed supported message payloads from signed non-message Talk events after signature verification; the closed related PR followed that shape but did not merge.
