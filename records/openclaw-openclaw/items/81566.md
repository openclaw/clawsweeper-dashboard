## Summary

Keep open: current main and the latest release still reject signed Nextcloud Talk webhook payloads that do not match the strict chat-message schema, while the only linked fix PR was closed unmerged.

## What This Changes

The issue asks the Nextcloud Talk channel to acknowledge signed non-message webhook events, such as file shares, instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so malformed JSON and malformed message-shaped payloads still return 400, while signed non-message Talk events are acknowledged without invoking the message handler.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a signed JSON Talk Activity Streams payload outside the strict Create/Person/Note/Collection schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format. I did not run a live Nextcloud deployment in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to parse signed JSON before strict message validation, return 400 only for malformed JSON or message-shaped invalid payloads, and treat other signed Talk event variants as ignored. The closed related PR followed that shape but did not merge.
