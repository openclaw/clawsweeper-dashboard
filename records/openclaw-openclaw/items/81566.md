## Label Justifications

- `P2`: The issue is a bounded but real channel reliability bug that can make Nextcloud Talk treat the bot as unhealthy in active rooms.
- `impact:message-loss`: The reported 400 responses can increment Talk bot error counts and may stop later message delivery if the bot is disabled by Nextcloud.

## Summary

Keep open: current `main` and the latest release still validate every signed Nextcloud Talk webhook against the strict chat-message `Create`/`Note` schema before deciding whether to ignore unsupported events, so signed non-message Activity Streams payloads can still return 400. The closed exact-fix PR was not merged, and the open rich-object attachment PR is adjacent rather than a proven canonical fix for this acknowledgment contract.

## What This Changes

The issue reports that the Nextcloud Talk channel returns HTTP 400 for signed non-message Talk webhooks such as file-share/setup events instead of acknowledging them without invoking message handling.

## Best Possible Solution

Keep one narrow Nextcloud Talk webhook fix: authenticate and JSON-parse first, acknowledge signed valid JSON events outside the supported chat-message shape without calling the message handler, and preserve existing 400/401 behavior for malformed JSON, malformed supported message payloads, missing headers, bad backend, and bad signatures.

## Reproduction Assessment

Yes at source level, though not with a live Nextcloud instance in this read-only review. A signed valid JSON Activity Streams payload whose shape is not the strict chat-message `Create`/`Note` schema will go through signature verification, fail `parseWebhookPayload()`, and return 400.

## Solution Assessment

Yes: separating authenticated JSON parsing from supported chat-message decoding is the narrow maintainable fix. It addresses the reported Nextcloud error-count pollution while preserving useful failures for malformed requests and authentication problems.
