## Summary

Keep open: current main still routes signed non-message Nextcloud Talk webhook payloads through the strict chat-message schema and returns HTTP 400 on schema miss. The linked fix PR had a plausible focused implementation and real webhook proof, but it was closed unmerged, so the issue remains actionable.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed non-message Talk events instead of returning 400 Invalid payload format.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so signed JSON is parsed before strict message validation, supported malformed message payloads still return 400, and signed non-message Talk events are acknowledged without invoking the message handler.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a signed JSON Talk Activity Streams payload outside the strict Create/Person/Note/Collection schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format; I did not run live Nextcloud in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to distinguish malformed JSON and malformed supported message payloads from signed non-message Talk events after signature verification; the closed related PR followed that shape but did not merge.
