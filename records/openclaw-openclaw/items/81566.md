## Summary

Keep open: current main and the latest release still reject signed Nextcloud Talk webhook payloads that do not match the strict chat-message Note schema, and the open draft fix at https://github.com/openclaw/openclaw/pull/90637 specifically owns this issue with closing syntax. The issue should stay open until that PR merges or maintainers choose a replacement path.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed non-message Talk Activity Streams events instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Land or replace the narrow linked fix so signed non-message Activity Streams events are acknowledged without dispatch while malformed JSON, malformed supported Notes, missing headers, bad backend, and bad signatures still fail explicitly.

## Reproduction Assessment

Yes at source level. A signed valid JSON Activity Streams payload outside the strict Create/Note schema reaches signature verification, fails parseWebhookPayload, and returns HTTP 400 on current main; I did not run a live Nextcloud server in this read-only review.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: split JSON parsing from supported chat-message decoding, ignore signed non-message events, and preserve explicit failures for malformed or unauthenticated requests. The remaining blocker is proof and draft readiness, not product direction.
