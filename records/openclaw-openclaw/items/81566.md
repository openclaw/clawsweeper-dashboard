## Summary

Keep open: current main and the latest release still parse signed Nextcloud Talk webhooks through the strict chat-message schema before classifying ignored events, so signed non-message hooks still return HTTP 400. The focused fix PR for this issue was closed unmerged, and the open rich-object file-share PR is related but does not close this non-message webhook contract.

## What This Changes

The issue requests that signed non-message Nextcloud Talk webhook events be acknowledged instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so signed JSON is parsed before strict message validation, signed non-message Talk events are acknowledged without invoking the message handler, and malformed JSON or malformed message-shaped payloads still return 400.

## Reproduction Assessment

Yes, at source level. A signed JSON Talk hook outside the strict chat-message schema still reaches decodeWebhookCreateMessage, fails parseWebhookPayload, and writes HTTP 400 Invalid payload format before the ignore branch can run.

## Solution Assessment

Yes. The narrow maintainable fix is to parse signed JSON first, classify supported chat-message shapes separately from non-message Activity Streams hooks, and preserve 400 handling for malformed JSON or malformed message-shaped payloads.

## Repair Work Prompt

Fix the Nextcloud Talk webhook handling for signed non-message Talk Activity Streams events. In `extensions/nextcloud-talk/src/monitor.ts`, parse signed JSON before applying the strict chat-message schema, return 200 with an empty body for valid signed non-message events such as reaction, bot lifecycle, and file-share-like payloads, keep 400 Invalid payload format for malformed JSON and malformed message-shaped payloads, and do not invoke `onMessage` for ignored events. Add focused regression coverage in `extensions/nextcloud-talk/src/monitor.replay.test.ts` for non-message variants returning 200 and malformed supported message payloads still returning 400. Do not bundle the broader rich-object media parsing from https://github.com/openclaw/openclaw/pull/49869 into this fix unless maintainers explicitly ask to combine them. A changelog entry may be needed at landing because this changes user-visible channel error behavior.
