## Summary

Keep open: current main and the latest release still return HTTP 400 for signed Nextcloud Talk webhook payloads that miss the strict chat-message Note schema, and the open closing pull request has not merged yet.

## What This Changes

The issue asks Nextcloud Talk webhooks to acknowledge signed non-message events, such as file-share Activity Streams payloads, instead of returning 400 Invalid payload format.

## Best Possible Solution

Land or replace the focused Nextcloud Talk fix from https://github.com/openclaw/openclaw/pull/90637: parse signed JSON separately from strict chat-message schema validation, acknowledge valid non-message Activity Streams events without dispatch, and preserve 400/401 behavior for malformed or unauthenticated requests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a signed JSON webhook whose Activity Streams object is not the strict Note payload fails parseWebhookPayload and returns 400 after signature verification. I did not run a live Nextcloud server in this read-only review.

## Solution Assessment

Yes, the linked two-stage parser approach is the best narrow fix: authenticate and parse JSON first, then distinguish malformed supported Note payloads from valid non-message Activity Streams events. It avoids adding config or a broader attachment parser to solve this acknowledgement bug.
