## Summary

Keep open: current main and v2026.5.18 still route signed non-message Nextcloud Talk webhook payloads through the strict chat-message schema and return HTTP 400 on schema miss. The exact fix PR was closed unmerged, and the open file-share rich-object PR is related but does not own this non-message acknowledgement contract.

## What This Changes

The issue asks for signed Nextcloud Talk webhook payloads that are not chat messages, such as file-share, reaction, or bot lifecycle events, to be acknowledged instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Land a focused Nextcloud Talk webhook classifier fix that parses signed JSON before strict message validation, keeps malformed JSON and malformed supported message payloads on the 400 path, and acknowledges signed non-message Talk events without invoking the message handler.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a signed JSON Activity Streams payload outside the strict Create/Person/Note/Collection schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format; I did not run live Nextcloud in this read-only pass.

## Solution Assessment

Yes. The narrow maintainable fix is to classify signed JSON before strict message validation, preserving 400 for malformed supported message payloads while returning 200/empty for non-message events; optional debug logging can be a follow-up logger-surface decision.
