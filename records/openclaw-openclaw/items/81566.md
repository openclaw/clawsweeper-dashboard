## Summary

Keep open: current main still has the reported failure mode, because the Nextcloud Talk webhook handler validates every signed payload against a narrow chat-message schema before it can classify documented non-message Talk webhook variants as ignorable.

## What This Changes

The issue asks the Nextcloud Talk channel to acknowledge signed non-message webhook events instead of returning HTTP 400 `Invalid payload format`.

## Best Possible Solution

Add a plugin-local post-signature classifier that separates malformed JSON/auth errors from valid signed Talk webhook variants, processes supported chat messages, and acknowledges benign non-message events without incrementing Nextcloud bot error counts.

## Reproduction Assessment

Yes from source inspection: a signed JSON payload whose Activity Streams shape is outside the strict `Create`/`Note` schema reaches `decodeWebhookCreateMessage` and returns HTTP 400 `Invalid payload format`. I did not establish a live Nextcloud reproduction in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is in the Nextcloud Talk webhook classifier: acknowledge recognized benign Talk variants after signature verification while keeping strict failures for malformed JSON, missing headers, invalid signatures, invalid backends, and malformed supported chat messages.
