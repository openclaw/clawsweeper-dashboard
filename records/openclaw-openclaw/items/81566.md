## Summary

The current source path matches the reported failure mode: the webhook server only accepts a narrow chat-message schema before it can classify non-message Talk hooks, so signed but non-chat Activity Streams variants are treated as bad requests.

## What This Changes

The issue asks for Nextcloud Talk webhook handling to acknowledge non-message Talk webhook events instead of returning HTTP 400 invalid payload errors.

## Best Possible Solution

Loosen the post-signature payload classifier so valid JSON signed by Nextcloud can be classified as a supported chat message or a benign ignored Talk event, while malformed JSON, missing auth headers, invalid signatures, and truly invalid supported message payloads still return errors.

## Reproduction Assessment

Yes from source inspection: a signed JSON payload whose Activity Streams shape is not the strict `Create`/`Note` chat-message schema reaches `decodeWebhookCreateMessage` and returns HTTP 400 `Invalid payload format`. I did not establish a live Nextcloud reproduction in this read-only review.

## Solution Assessment

Yes, the reporter's suggested direction is the right boundary: treat recognized non-message Talk webhook variants as ignored after signature verification, but keep strict errors for malformed/auth-invalid requests and invalid chat-message payloads.
