## Label Justifications

- `P2`: This is a bounded channel reliability bug that can make Nextcloud Talk mark the bot unhealthy, while ordinary chat replies can still work between error bursts.
- `impact:message-loss`: Repeated 400 responses can increase Nextcloud Talk bot error counts and may stop future message delivery if the bot is disabled.

## Summary

Keep open: current main and v2026.6.6 still return 400 for signed Nextcloud Talk payloads that miss the strict chat-message Note schema, and the remaining work is already owned by the open draft fix PR at https://github.com/openclaw/openclaw/pull/90637.

## What This Changes

Nextcloud Talk should acknowledge signed non-message Talk webhook events instead of returning HTTP 400 while preserving bad-request handling for malformed or unauthenticated requests.

## Best Possible Solution

Land the focused PR at https://github.com/openclaw/openclaw/pull/90637, or a narrow replacement, with proof that signed non-message events return 200 and malformed message-shaped payloads still return 400.

## Reproduction Assessment

Yes, at source level. A signed valid JSON Activity Streams payload outside the strict Create/Note schema reaches decodeWebhookCreateMessage and returns HTTP 400 before the existing ignore path can run.

## Solution Assessment

Yes, the best fix shape is to split authenticated JSON parsing from supported chat-message schema validation, acknowledge valid signed non-message events, and keep 400/401 behavior for malformed or unauthenticated requests.
