## Label Justifications

- `impact:message-loss`: Repeated 400 responses can increment Talk bot error counts and may eventually disable delivery to the bot.

## Summary

Keep open: current main and v2026.6.6 still send signed payloads that miss the strict Nextcloud Talk chat-message schema to HTTP 400, and the active closing fix at https://github.com/openclaw/openclaw/pull/90637 is open/draft rather than merged.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed non-message Talk events instead of returning 400 `Invalid payload format`.

## Best Possible Solution

Review and resolve https://github.com/openclaw/openclaw/pull/90637, with the final behavior acknowledging signed valid non-message Activity Streams events while preserving 400/401 responses for malformed or unauthenticated requests.

## Reproduction Assessment

Yes. A signed valid JSON Activity Streams payload whose object is not the strict `Note` chat-message shape reaches signature verification, then `safeParseJsonWithSchema()` returns null and the handler writes HTTP 400.

## Solution Assessment

Yes. The best fix is the linked two-stage decode: distinguish malformed JSON or malformed supported Notes from valid unsupported Activity Streams events, then acknowledge unsupported signed events without dispatching them.
