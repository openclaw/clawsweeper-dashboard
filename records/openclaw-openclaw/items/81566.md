## Summary

Keep open: current main and v2026.5.18 still parse signed Nextcloud Talk webhooks through the strict chat-message schema before classifying event type, so signed non-message Activity Streams variants still return HTTP 400. The focused fix PR had useful signed-webhook proof but was closed unmerged, and the adjacent open rich-object/mention PRs do not close this exact acknowledgement path.

## What This Changes

The issue requests that Nextcloud Talk signed non-message webhook events be acknowledged instead of rejected as invalid payloads.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so signed JSON is parsed after signature verification, supported malformed message payloads still return 400, and signed non-message Talk events are acknowledged without invoking the message handler.

## Reproduction Assessment

Yes. A signed JSON Activity Streams payload outside the strict chat-message schema reaches decodeWebhookCreateMessage after signature verification and returns HTTP 400; I did not run a live Nextcloud server in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to separate malformed JSON and malformed supported message payloads from signed non-message Talk events, preserving 400 for true bad requests and returning 200/empty for ignored events.
