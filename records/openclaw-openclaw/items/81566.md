## Summary

Keep open: current main and the latest release still route signed non-message Nextcloud Talk webhook payloads through the strict chat-message schema and return HTTP 400 on schema miss. The exact focused fix PR closed unmerged, while the remaining open rich-object PR is adjacent rather than a fix for this 400-on-non-message contract.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed Talk events outside the chat-message schema instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier after signature verification so malformed JSON and malformed supported message payloads still return 400, while signed non-message Talk Activity Streams events are acknowledged and skipped without invoking the message handler.

## Reproduction Assessment

Yes, by source inspection. A signed JSON Activity Streams payload outside the strict Create/Person/Note/Collection chat-message schema reaches decodeWebhookCreateMessage, fails parseWebhookPayload, and is returned as HTTP 400 Invalid payload format; I did not run a live Nextcloud instance.

## Solution Assessment

Yes. The narrow maintainable fix is to classify the signed JSON envelope before strict message validation, preserving 400 for malformed supported message payloads while acknowledging non-message Talk events.
