## Summary

Keep open: current main and the latest release still route signed non-message Nextcloud Talk webhook payloads through the strict chat-message schema and return HTTP 400 on schema misses. The related fix PR at https://github.com/openclaw/openclaw/pull/81569 had a focused implementation and real signed-webhook proof, but it closed unmerged, so the issue remains actionable.

## What This Changes

The issue requests that signed Nextcloud Talk webhook events outside the chat-message schema be acknowledged and ignored instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Update the Nextcloud Talk webhook classifier so signed JSON is parsed before strict message validation, message-shaped malformed payloads still return 400, and signed non-message Talk events are acknowledged without invoking the message handler.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a signed JSON Activity Streams payload outside the strict Create/Person/Note/Collection schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format before the ignore branch.

## Solution Assessment

Yes. The narrow maintainable fix is to distinguish malformed JSON and malformed supported message payloads from signed non-message Talk events after signature verification; the closed related PR followed that shape but did not merge.
