## Summary

Keep open: current main and the latest release still validate signed Nextcloud Talk webhooks against the strict chat-message Note schema before distinguishing valid non-message Activity Streams events, so file-share-like Document events still return 400. Two focused fix PRs were closed unmerged, while the currently open Nextcloud Talk PRs are adjacent structured-payload work rather than an active closing fix for this exact 400/acknowledgement bug.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed non-message Talk events such as file shares without dispatching them to the bot, while preserving 400 responses for malformed JSON or malformed supported message payloads.

## Best Possible Solution

Land a narrow monitor-level fix that parses signed JSON before strict Note validation, returns 200/no body for valid non-message Talk Activity Streams events, preserves 400 for malformed JSON and malformed message-shaped payloads, and coordinates with the adjacent structured-payload PRs.

## Reproduction Assessment

Yes, source-level: a signed JSON payload with a non-Note object such as Document fails the strict NextcloudTalkWebhookPayloadSchema, and decodeWebhookCreateMessage writes HTTP 400 before the ignore branch. I did not run a live Nextcloud server in this read-only review.

## Solution Assessment

Yes for the narrow repair shape: the best fix is to classify signed JSON before strict message validation, acknowledge non-message Activity Streams events, and keep malformed JSON or partial message-shaped payloads on the existing 400 path. Reusing the closed candidate shape is better than solving this in inbound mention or attachment parsing.
