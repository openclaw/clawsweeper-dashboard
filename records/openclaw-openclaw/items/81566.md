## Summary

Keep open: current main and the latest release still route signed non-message Nextcloud Talk Activity Streams payloads through the strict chat-message schema and return HTTP 400. The exact fix PR was closed unmerged, so the issue remains actionable.

## What This Changes

The issue asks the Nextcloud Talk webhook handler to acknowledge signed non-message Talk events instead of returning HTTP 400 Invalid payload format.

## Best Possible Solution

Classify signed JSON after signature verification into supported chat messages, benign ignored Talk events, and malformed supported-message payloads; acknowledge ignored events while preserving errors for malformed JSON, missing auth headers, invalid signatures, and invalid message-shaped payloads.

## Reproduction Assessment

Yes from source inspection: a signed JSON payload whose Activity Streams shape is not the strict Create/Note chat-message schema reaches decodeWebhookCreateMessage and returns HTTP 400 Invalid payload format. I did not run a live Nextcloud Talk server in this read-only review.

## Solution Assessment

Yes, the reporter's direction and the closed reference PR are the narrowest maintainable fix: distinguish malformed/auth-invalid requests from signed non-message Talk events and acknowledge the latter without invoking onMessage.
