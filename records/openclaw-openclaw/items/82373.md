## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the channel task promise tracked:** `src/gateway/server-channels.ts:548-559`
  - body: This starts the channel with `void startAccount(...)`, so the promise stored in `store.tasks` resolves as soon as handoff finishes instead of when the channel actually exits. A long-running Discord/Telegram/etc. transport will then hit the `channel exited without an error` cleanup/restart path while the real transport work is still active, which can break delivery and restart accounting.
  - confidence: 0.92
- **[P2] Do not report expected account states as failures:** `src/gateway/server-channels.ts:820-826`
  - body: `startChannelInternal` records disabled and unconfigured accounts by setting `lastError` to reasons like `disabled` or `not configured`, then returns normally. This new detector treats any non-running account with `lastError` as a startup failure, so normal installs with an installed but disabled/unconfigured channel can make `channelsFailed` and `readiness.fullyReady` look degraded.
  - confidence: 0.88
- **[P2] Remove or implement the missing canon scripts:** `package.json:1363-1365`
  - body: These package scripts invoke `scripts/canon.mjs`, but that file is not present in the PR head. Landing this would publish broken `pnpm canon:*` commands unrelated to the startup readiness feature.
  - confidence: 0.95
- **[P3] Classify the local TCP probe:** `src/cli/gateway-cli/register.ts:355-357`
  - body: The new startup-doctor opens a raw `net.createConnection` callsite. Even though it is loopback-only, the repo's security review already flagged this class of raw socket egress for classification, so the code should document or encode that this is a local diagnostic probe.
  - confidence: 0.78
