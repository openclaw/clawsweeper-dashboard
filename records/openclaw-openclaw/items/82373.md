## Summary

Keep this PR open. The startup diagnostics direction is plausible, but the current branch introduces blocking channel lifecycle regressions and the supplied proof is focused tests only, not after-fix real gateway behavior.

## What This Changes

The PR adds gateway startup runtime-state persistence, a `gateway startup-doctor` command, safe-mode startup flags, channel startup summaries/timeouts, docs, and focused gateway tests.

## Best Possible Solution

Preserve the existing channel restart/backoff lifecycle while adding startup diagnostics, report disabled or unconfigured channels as skipped rather than failed, wire the docs into the Gateway docs section, and require a real gateway startup-doctor proof before merge.

## Reproduction Assessment

Yes for the review findings by source inspection: current main sends immediate channel task failures through the tracked restart/backoff chain, while the PR throws before installing that chain. I did not run tests because this was a read-only review.

## Solution Assessment

No. The overall direction is useful, but this implementation should preserve current restart semantics and unmanaged-account readiness semantics before it becomes the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep immediate failures on the restart path:** `src/gateway/server-channels.ts:567-576`
  - body: This throws when `startAccount` rejects during the first few microtasks, before `trackedPromise` is installed. On current main the same task is always wrapped by the catch/finally/backoff chain, so an immediate transient startup failure is recorded and auto-restarted; with this branch it is reported once as failed and never enters the existing restart loop. Please derive the startup summary without bypassing the tracked task/restart machinery.
  - confidence: 0.9
- **[P2] Skip unmanaged accounts when summarizing startup failures:** `src/gateway/server-channels.ts:805-811`
  - body: `getImmediateChannelFailure` treats any stopped account with `lastError` as a startup failure. Disabled and unconfigured accounts intentionally set `lastError` (`disabled` / `not configured`) and readiness treats them as unmanaged, so ordinary disabled or incomplete channel config will now inflate `channelsFailed` and keep `readiness.fullyReady` false. Please ignore `enabled === false` / `configured === false` here or report them as `skipped`.
  - confidence: 0.87
- **[P3] Wire the startup resilience docs into navigation:** `docs/gateway-startup-resilience.md:1`
  - body: The new page is added at the docs root and `docs/docs.json` is not updated, so the Mintlify Gateway docs will not expose it from the Health and diagnostics section. Please move/link it under the Gateway docs path and add it to the navigation, or fold the content into an existing Gateway diagnostics page.
  - confidence: 0.75
