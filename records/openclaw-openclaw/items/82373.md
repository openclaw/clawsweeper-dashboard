Keep this PR open: the startup diagnostics direction is useful and the contributor has now supplied real runtime output, but the current head still has build-blocking source defects, stale test/type contracts, build-script regressions, and an unclassified raw socket diagnostic callsite.

The PR adds gateway startup runtime-state reporting, a `startup-doctor` command, safe-mode startup flags, per-channel startup summaries/timeouts, focused tests, docs, and related build-script compatibility changes.

## Best Possible Solution

Keep the startup diagnostics idea, but first restore the existing channel lifecycle/restart contract, align tests and build profiles with the new summary type, classify the local diagnostic socket probe, wire docs into the Gateway nav, and get maintainer approval for the new CLI/runtime-state contract.

The claimed user behavior also has contributor-supplied real startup-doctor output from a Windows source checkout.

No. The direction is useful, but the current implementation is not the narrowest maintainable solution until it preserves the existing channel restart lifecycle, respects unmanaged channel semantics, keeps build/test contracts green, and gets the new diagnostic API reviewed.

- **[P1] Restore the startAccount handoff block:** `src/gateway/server-channels.ts:548-561`
  - body: The current head leaves the `measureStartup` callback malformed around `startAccount` and drops the `await startAccountTask` that current main uses to keep the channel lifecycle attached to `trackedPromise`.   - confidence: 0.98

  - body: This throws immediate startup failures before `trackedPromise` is installed. Current main routes those failures through the catch/finally/backoff chain, so transient startup errors can be cleaned up and restarted; this branch reports them once and skips that lifecycle.
  - confidence: 0.9

  - body: `getImmediateChannelFailure` treats any stopped account with `lastError` as a startup failure.   - confidence: 0.88
