## Summary

Keep this PR open. The startup-diagnostics direction is useful and relates to open gateway startup-boundary reports, but the branch is not mergeable: it regresses channel restart semantics, misclassifies unmanaged channels, leaves CI/type/lint/security guard failures, and does not include real behavior proof.

## What This Changes

The PR adds a gateway `startup-doctor` command, safe-mode startup flags, runtime-state readiness fields, channel startup summaries/timeouts, docs, and focused gateway tests.

## Best Possible Solution

Preserve the existing channel restart/backoff lifecycle while adding startup diagnostics, report disabled or unconfigured channels as skipped/unmanaged, wire the docs into the Gateway docs section, classify the local socket probe, and validate with real `gateway startup-doctor` output plus green checks.

## Reproduction Assessment

Yes for the review findings: source inspection and CI logs show the PR's early throw bypasses current restart/backoff flow, its `lastError` scan conflicts with the unmanaged-account health contract, and several checks fail on the branch. I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is useful, but the implementation should derive startup summaries from the existing tracked lifecycle, preserve unmanaged-channel semantics, satisfy the diagnostic/security guardrails, and add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Keep immediate failures on the restart path:** `src/gateway/server-channels.ts:567-576`
  - body: This throws an immediate `startAccount` rejection before `trackedPromise` is installed. On current main, that same task is wrapped by the catch/finally/backoff chain, so transient startup failures are recorded and restarted; with this branch they are reported once and skip the restart loop. Derive the startup summary from the tracked lifecycle instead of short-circuiting it.
  - confidence: 0.9
- **[P2] Skip unmanaged accounts when summarizing failures:** `src/gateway/server-channels.ts:805-811`
  - body: `getImmediateChannelFailure` treats any stopped account with `lastError` as a startup failure. Disabled and unconfigured accounts intentionally get `lastError` values such as `disabled` or `not configured`, while readiness treats them as unmanaged, so ordinary disabled or incomplete channel config will inflate `channelsFailed` and keep full readiness false.
  - confidence: 0.88
- **[P2] Update callers for the new startup summary contract:** `src/gateway/server-channels.ts:213-215`
  - body: Changing `startChannels` from `Promise<void>` to `Promise<ChannelStartupSummary>` leaves existing test doubles and startup post-attach mocks returning `undefined` or `{ pluginServices: null }`; `check-test-types` fails across `channel-health-monitor.test.ts` and `server-startup-post-attach.test.ts`. Either preserve the old callable contract or update all typed call sites and test doubles.
  - confidence: 0.92
- **[P2] Keep server runtime env mocks compatible:** `src/gateway/server-runtime-services.ts:275`
  - body: `activateGatewayScheduledServices` now imports `isTruthyEnvValue`, but existing tests mock `../infra/env.js` without that export, so the startup-runtime shard fails before exercising the intended behavior. Update the mock seam or avoid adding a new mocked export dependency here.
  - confidence: 0.9
- **[P3] Classify the raw socket probe before landing:** `src/cli/gateway-cli/register.ts:357`
  - body: The new `net.createConnection` call is a local startup diagnostic probe, but the repository's raw-socket CodeQL guard requires every new raw client callsite to be classified as local-only, managed/proxied, diagnostic guarded, or unsupported. Add the appropriate classification or reuse an already classified helper so the network-runtime-boundary check passes.
  - confidence: 0.93
- **[P3] Use the lint-approved NaN form:** `src/gateway/server-channels.ts:787`
  - body: The startup timeout parser introduces the global `NaN`, and `check-lint` rejects it with `Use Number.NaN instead of the global NaN`. Swap this to the repo-approved form before landing.
  - confidence: 0.86
- **[P3] Wire the startup resilience docs into navigation:** `docs/gateway-startup-resilience.md:1`
  - body: The new docs page is added at the docs root and `docs/docs.json` is not updated, so the Mintlify Gateway docs will not expose it from the Health and diagnostics section. Move or link it under the Gateway docs path and add it to navigation, or fold the content into an existing Gateway diagnostics page.
  - confidence: 0.78
