## Summary

Keep open: this is a draft maintainer PR with the protected `maintainer` label, and source review found a blocking diagnostic false-positive in the new outbound channel path.

## What This Changes

The PR adds catalog-backed repair hints in status and outbound send selection when configured official external channel plugins such as Feishu or WhatsApp are unavailable after an upgrade.

## Best Possible Solution

Keep the repair hints, but gate missing-plugin language on the plugin actually being absent and add a regression test for installed-but-unconfigured official external channels.

## Reproduction Assessment

Yes from source inspection: configure `channels.whatsapp.enabled = true`, have the WhatsApp plugin installed but unlinked so `isConfigured` returns false, and call channel selection without an explicit channel. The PR's new branch would report a missing plugin even though the installed plugin needs login/config repair.

## Solution Assessment

No; the overall direction is right, but the no-explicit-channel path should mirror the explicit-channel path and only show install/doctor missing-plugin repair text when the channel plugin cannot be resolved or loaded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.76

Full review comments:

- **[P2] Check plugin absence before emitting missing-plugin hints:** `src/infra/outbound/channel-selection.ts:252-256`
  - body: The new no-explicit-channel path builds repair hints from raw `channels.<id>` config whenever no configured message channel is found. If the official external plugin is installed but not linked/configured, `listConfiguredMessageChannels` can still return empty, so this reports “missing its plugin” even though the next action is login/config repair. Gate this hint on the channel plugin actually being absent or unresolved, like the explicit-channel branch does.
