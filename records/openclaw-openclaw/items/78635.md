## Summary

Keep open: the linked Telegram media-delivery bug is plausible, but this PR is not ready because it lacks real behavior proof and the new branch can bypass the current per-media delivery path that preserves multi-media results and first-message pinning.

## What This Changes

The PR changes `src/infra/outbound/deliver.ts` so payloads with `mediaUrl` or `mediaUrls` take the `handler.sendPayload` path during `agent --deliver`.

## Best Possible Solution

Keep the fix in outbound delivery, but preserve current media fan-out/result/pin semantics, add a focused regression test, and require real Telegram `agent --deliver` proof before merge.

## Reproduction Assessment

No high-confidence current-main reproduction was established. The linked issue has concrete Windows/Telegram steps, but this review only source-inspected the path and current main still has a generic media delivery branch after the sendPayload condition.

## Solution Assessment

No. The proposed condition is too broad because it sends all media payloads through a single-result payload branch; a safer fix should target the missing Telegram path or make payload delivery preserve the existing media fan-out contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve multi-media delivery results:** `src/infra/outbound/deliver.ts:1449`
  - body: Routing any payload with `mediaUrl`/`mediaUrls` into `sendPayload` skips the existing media branch that fans out `mediaUrls`, records each delivery, and pins the first media message. Telegram's payload helper returns one result from the sequence, so a multi-media `agent --deliver` payload can report, notify, and pin only that single result instead of preserving all delivered media. Keep the media fan-out semantics or make payload delivery return equivalent results before taking this branch.
  - confidence: 0.87
