## Summary

Keep open: current main and the latest release still have a source-reproducible path where a custom provider model without `maxTokens` can inherit a broad fallback cap and then forward it as `max_completion_tokens`. The merged Xiaomi catalog fix covers bundled Xiaomi Token Plan rows, but it does not settle the reporter's custom omitted-metadata case.

## What This Changes

The issue asks OpenClaw to avoid sending an invalid oversized `max_completion_tokens` for custom Xiaomi MiMo model entries when `maxTokens` is omitted.

## Best Possible Solution

Stop steady-state runtime from sending a synthesized oversized output cap for configured Xiaomi/OpenAI-compatible models without `maxTokens`; prefer known catalog metadata for matching rows or fail early with a clear configuration error before the HTTP request.

## Reproduction Assessment

Yes from source inspection, but not from a fresh live Xiaomi call: current fallback resolution can produce a broad `model.maxTokens`, and the OpenAI-completions transport forwards it as `max_completion_tokens`. The reporter supplies the live provider failure and working config workaround.

## Solution Assessment

Yes: the narrow maintainable fix is in model resolution or request-cap guarding, not a new config option. Reusing known Xiaomi catalog metadata when available or rejecting/omitting an unsafe synthesized cap would address the broken existing behavior.
