## Summary

Keep open. Current main still has a source-plausible path where a custom Xiaomi model without a resolved maxTokens cap can fall back to the generic 200k token default before the OpenAI-compatible transport sends max_completion_tokens, and no exact duplicate or merged fix was found.

## What This Changes

The issue asks OpenClaw to avoid sending an over-limit max_completion_tokens value for custom Xiaomi MiMo models when the configured model row omits maxTokens.

## Best Possible Solution

Keep this issue open and fix model metadata resolution so custom Xiaomi rows either inherit a known catalog/provider maxTokens cap, preserve explicit configured maxTokens, or fail with a clear configuration error instead of sending an over-limit request.

## Reproduction Assessment

Yes, at source level: a custom Xiaomi model row without maxTokens can resolve to the generic DEFAULT_CONTEXT_TOKENS maxTokens fallback and the OpenAI-compatible transport then clamps against that wrong value. I did not perform a live Xiaomi call, so this is not marked as fully reproduced.

## Solution Assessment

Yes, the maintainable fix is a narrow provider/model metadata repair or explicit configuration error, not a new setting. The existing workaround of adding maxTokens is useful evidence but should not be required for a known Xiaomi model cap.
