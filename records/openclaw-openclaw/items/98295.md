## Summary

Keep open: current main has correct bundled Xiaomi Token Plan catalog limits, but the reported custom-provider path can still resolve a configured Xiaomi model without `maxTokens` to an oversized default and forward it as `max_completion_tokens`. The issue is fresh, externally reported, and not superseded by a merged fix for the omitted-custom-metadata case.

## What This Changes

The issue reports Xiaomi MiMo custom model requests exceeding the provider completion-token limit when `maxTokens` is omitted from `models.providers.xiaomi.models[]`.

## Best Possible Solution

Stop steady-state runtime from sending a synthesized oversized output cap for configured Xiaomi/OpenAI-compatible models without `maxTokens`; prefer existing catalog metadata when available or fail early with a clear configuration error instead of sending an invalid request.

## Reproduction Assessment

Yes from source inspection, but not from a fresh live Xiaomi call: configured model fallback can supply a broad default maxTokens when custom metadata omits it, and the OpenAI-completions transport forwards the resolved value as `max_completion_tokens`. Reporter evidence supplies the live provider failure and workaround.

## Solution Assessment

Yes, the best fix is a narrow provider/model-resolution guard rather than a new config option: reuse known Xiaomi catalog metadata for matching MiMo rows when available, or reject/omit the generated output cap before the HTTP request. Merging the related catalog PR would help pay-as-you-go model discovery but would not by itself cover all custom omitted-maxTokens entries.
