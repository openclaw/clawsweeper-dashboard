## Summary

Keep open: the latest head improves the default progress-mode suppression path, but the new deferred fallback is still cleared when a normal final reply lands, so the linked fast-tool dead-air case can remain for turns that do produce a final answer. The PR also still has only harness/unit proof, not real Discord behavior proof.

## What This Changes

The branch changes shared progress-draft suppression and Discord draft/block delivery so tool/block replies are deferred until a progress draft becomes visible and flushed as fallback when no draft/final reply is delivered, with regression tests.

## Best Possible Solution

Preserve the fallback progress/block update until it can be emitted before final delivery when no draft became visible, keep duplicate suppression once a draft is visible, and add redacted real Discord proof.

## Reproduction Assessment

Yes from source inspection: current main suppresses progress-mode standalone tool progress before any draft is visible, while the first work event can remain behind the delayed draft gate. I did not run a live Discord/Codex reproduction in this read-only review.

## Solution Assessment

No: the PR is a plausible direction, but clearing deferred progress blocks on final delivery leaves the reported block-plus-final fast-tool path undercovered. The best fix should preserve a visible fallback before final delivery when no progress draft became visible.
