## Summary

Keep open. Current main still contains the Feishu startup path that awaits a 30-second bot-identity probe before account transport starts, the focused fix PR was closed unmerged, and the related Feishu SDK timeout report adds diagnostics rather than fully superseding this Windows startup case.

## What This Changes

Prevent Gateway startup on Windows from waiting 30 seconds on Feishu API/Lark SDK timeouts while keeping the Feishu channel usable after startup.

## Best Possible Solution

Fix this inside the Feishu plugin by making startup use a startup-safe, cancellable identity path that does not block account transport, then recover bot identity in the background while preserving group mention gating.

## Reproduction Assessment

Yes, source-reproducible: configure Feishu in websocket mode and make the startup bot identity probe hang until its timeout; current `monitorFeishuProvider` awaits that 30,000 ms path before starting `monitorSingleAccount`. I did not run a live Windows/Feishu credential repro.

## Solution Assessment

Yes, the narrow maintainable fix is Feishu-owned startup decoupling, not a core Gateway timeout change. The closed PR in #74187 is useful prior art, but it is not merged and should be replaced or revised with focused startup tests.

## Repair Work Prompt

Fix the Feishu startup responsiveness bug tracked by https://github.com/openclaw/openclaw/issues/63257, considering the related diagnostics in https://github.com/openclaw/openclaw/issues/72721 and the closed unmerged prior attempt in https://github.com/openclaw/openclaw/pull/74187. Current main awaits `fetchBotIdentityForMonitor` during `monitorFeishuProvider` startup with a 30,000 ms default, and `probeFeishu` passes that timeout through to the Lark SDK request. Make the startup path avoid blocking account transport for a full external HTTP timeout: use a shorter startup-safe identity probe and/or start `monitorSingleAccount` promptly with unknown identity while background recovery continues. Preserve strict `requireMention` and group policy behavior while bot identity is unknown; do not weaken Feishu group gating, alter global Gateway networking, or add a new config option unless a maintainer explicitly asks. Add focused Feishu tests that simulate a hanging or timeout probe and assert transport startup is not delayed, background identity recovery still runs, and timeout wiring remains bounded. Add a user-facing `CHANGELOG.md` fix entry; update docs only if the user-visible config or env semantics change.
