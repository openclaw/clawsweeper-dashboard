## Summary

Keep open. Current main still has Feishu startup/status paths that can wait on SDK-backed Feishu API calls for the reported timeout shape, and the overlapping Feishu timeout/startup PRs are open with review blockers or closed unmerged rather than shipped.

## What This Changes

The issue asks OpenClaw to avoid a Windows gateway startup delay caused by Feishu SDK/API requests timing out before Feishu continues normally.

## Best Possible Solution

Land or replace a Feishu-owned fix that bounds or decouples startup/status and WebSocket token waits, preserves gateway monitor lifetime and mention gating, and includes focused regression tests plus redacted real Feishu startup proof.

## Reproduction Assessment

Yes at source level: configure Feishu in WebSocket mode and make the startup bot-identity or SDK token request hang; current main can await the 30,000 ms startup probe path before transport setup, and the WSClient token path still uses the SDK HTTP path. I did not establish a live Windows credential reproduction.

## Solution Assessment

No shipped solution exists yet. The best fix is Feishu-local and bounded, but the open branches propose conflicting latency/serialization/lifecycle tradeoffs that need maintainer reconciliation before implementation is safe.
