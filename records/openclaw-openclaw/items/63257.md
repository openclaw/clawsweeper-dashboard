## Summary

Keep open. Current main still has the reported Feishu startup wait shape: a 30-second startup bot-identity probe is awaited before Feishu account transport starts, while the focused fix PR remains open and the prior responsiveness PR was closed unmerged.

## What This Changes

The issue asks for Feishu Gateway startup on Windows to avoid a 30-second Feishu API/Axios timeout before startup continues.

## Best Possible Solution

Land or replace #67466 with a lifecycle-safe Feishu fix that bounds startup/status probes, starts transport promptly when identity lookup is slow, recovers identity in the background, and keeps mention gating correct.

## Reproduction Assessment

Yes. The report gives concrete Windows/Node/OpenClaw logs, and current main statically reproduces the same 30-second wait path because monitorFeishuProvider awaits fetchBotIdentityForMonitor, which defaults to 30,000ms and passes that budget to probeFeishu before transport starts.

## Solution Assessment

No current-main solution exists yet. The focused PR is aimed at the right area, but the final fix needs maintainer review around monitor lifecycle and requireMention safety.
