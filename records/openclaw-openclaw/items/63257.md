## Summary

Keep open. Current main still lets the Feishu startup bot-identity probe use a 30-second default and await that probe before Feishu account transport starts; related open PRs at https://github.com/openclaw/openclaw/pull/67466 and https://github.com/openclaw/openclaw/pull/63481 are unmerged and conflicting, so the reported behavior is not implemented yet.

## What This Changes

Ensure Windows Feishu gateway startup does not block on a 30-second SDK/Axios API timeout before Feishu transport starts.

## Best Possible Solution

Land or revise a Feishu-local fix that bounds or decouples startup bot-identity probing while keeping the gateway monitor promise tied to account lifecycle and preserving mention-gating behavior.

## Reproduction Assessment

Yes, source-level: configure Feishu in WebSocket mode and make the startup bot identity probe hang until timeout; current main awaits the 30,000 ms probe before account transport starts. I did not run a live Windows/Feishu credential reproduction.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable solution is Feishu-owned startup probe bounding or decoupling, coordinated with the open Feishu PRs, not a new core gateway timeout policy.
