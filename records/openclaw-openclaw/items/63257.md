## Summary

Keep open. Current main and v2026.5.12 still allow Feishu startup to wait on the 30-second bot-identity probe before transport starts, and the overlapping Feishu startup/probe PRs remain open or unmerged.

## What This Changes

The issue reports that Windows gateway startup hits a 30-second Feishu SDK/Axios API timeout even though normal system network checks succeed.

## Best Possible Solution

Land a Feishu-local fix that bounds or decouples startup/status identity probing while keeping the gateway-owned monitor promise long-lived and covered by focused regression tests plus redacted live startup proof.

## Reproduction Assessment

Yes, source-level: configure Feishu in WebSocket mode and make the startup bot-identity probe hang; current main awaits the 30,000 ms fetchBotIdentityForMonitor path before transport startup. I did not run a live Windows/Feishu credential reproduction.

## Solution Assessment

Yes for the direction, but no shipped fix exists yet. The narrow maintainable solution belongs in Feishu startup/probe code and should reconcile the open timeout and serialization PRs without detaching gateway lifecycle ownership.
