## Summary

Keep open. Current main and the latest checked release still allow Feishu startup to wait on a 30,000 ms bot-identity probe before transport setup, and the related implementation PRs at https://github.com/openclaw/openclaw/pull/63481 and https://github.com/openclaw/openclaw/pull/67466 remain open, conflicting, and unmerged.

## What This Changes

The issue reports Windows 11 Feishu gateway startup waiting 30 seconds on a Lark SDK/Axios Feishu API call before continuing.

## Best Possible Solution

Land or revise one Feishu-local fix that bounds or decouples startup identity/status/WS-token probing while preserving gateway lifecycle tracking and mention-gating behavior.

## Reproduction Assessment

Yes, source-level: configure Feishu in WebSocket mode and make the startup bot identity probe hang or time out; current main awaits the 30,000 ms probe before account transport setup. I did not run a live Windows/Feishu credential reproduction.

## Solution Assessment

Not applicable for a patch on this issue. The maintainable path is a Feishu-owned startup/probe fix coordinated with the existing open PRs, not closing this report or adding a separate core gateway timeout policy.
