## Summary

Keep this PR open. Current main still lacks the two behaviors the PR targets: Slack block-action system events are queued without waking the routed session, and Slack interaction session resolution still cannot receive `threadTs`, so threaded clicks resolve to the base session. The branch is useful source work, but it needs rebase/reconciliation with the broader Slack interaction tracker and overlapping wake PRs before maintainers pick a canonical path.

## What This Changes

The PR adds an optional wake hint to system-event enqueueing, threads Slack `threadTs` into block-action session resolution, derives thread-aware Slack session keys, and adds focused system-event and Slack interaction tests.

## Best Possible Solution

Land one reconciled Slack interaction fix that wakes the routed session only after a non-duplicate event is queued, derives the same thread-aware session key as normal Slack routing, preserves delivery/routing context, covers block actions and reply controls in tests, and includes the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Slack block-action payload with `container.thread_ts`: current main keeps `threadTs` in the event payload but omits it from session resolution and does not wake after enqueue.

## Solution Assessment

Unclear as the final merge path. The PR's wake-after-enqueue and `resolveThreadSessionKeys` direction is narrow, but it needs rebase, changelog coverage, and reconciliation with the overlapping Slack interaction wake work before it is clearly the best solution.
