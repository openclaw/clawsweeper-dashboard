## Summary

Keep open. Current main supports the reported silent-drop path: WhatsApp group messages are rejected when `channels.whatsapp.groups` is configured without the group or `"*"`, and the runtime only writes a verbose log before returning without dispatch.

## What This Changes

The issue asks for WhatsApp group inbound messages missing from `channels.whatsapp.groups` to route or emit a visible warning instead of disappearing silently.

## Best Possible Solution

Keep the existing group allowlist semantics, but add a default-visible, rate-limited warning or diagnostic when WhatsApp drops an inbound group because `channels.whatsapp.groups` lacks that group or `"*"`.

## Reproduction Assessment

Yes, from source inspection and existing focused tests: configure a WhatsApp `groups` map without the inbound group or `"*"`, then a mentioned group message reaches `applyGroupGating` and returns `shouldProcess: false`. I did not run a live WhatsApp bridge in this read-only pass.

## Solution Assessment

The best fix is not to reinterpret `groupAllowFrom` as a group allowlist; it should preserve the documented access boundary and surface a visible warning or doctor/status hint for this exact drop reason.
