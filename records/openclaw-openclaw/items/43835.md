## Summary

Keep this PR open, but do not merge it as-is. The linked leak remains valid on current main, yet the PR's cleanup path can delete the wrong or still-live workspace and is now stale against the backend-aware sandbox prune flow.

## What This Changes

This PR adds recursive sandbox workspace directory cleanup to `src/agents/sandbox/prune.ts` after sandbox-container prune, skipping shared scope and adding an agent-scope remaining-entry guard.

## Best Possible Solution

Rework the fix on current main so backend-aware sandbox prune records or resolves the exact per-entry workspace identity, deletes only after confirmed runtime removal, skips cleanup while any remaining registry/browser/runtime entry uses the same scope, preserves shared/agent/remote backend semantics, and adds focused prune coverage plus a changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. The original leak has a concrete reproduction in #43797, and current main source confirms prune removes runtimes and registry entries without removing local workspace directories. The PR regressions are statically reproducible from the diff and current registry/scope-key contracts.

## Solution Assessment

No. Cleaning orphaned workspaces is the right direction, but this patch is not the best solution because it re-normalizes stored scope keys, deletes after best-effort runtime removal, only guards duplicate workspace use for agent scope, and ignores per-entry workspace root history.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key. For an agent-scope entry such as `agent:ops`, calling `resolveSandboxScopeKey("agent", entry.sessionKey)` treats the two-part value as a malformed agent session and can resolve to `agent:main`, deleting the wrong workspace while leaving the intended one behind.
  - confidence: 0.91
- **[P1] Delete workspaces only after confirmed runtime removal:** `src/agents/sandbox/prune.ts:83`
  - body: The PR runs `docker rm -f` with `allowFailure: true` and then still reaches `fs.rm`. If Docker is unavailable or removal fails, prune can delete the host workspace while the container still exists or is still running with that mount.
  - confidence: 0.89
- **[P2] Check remaining entries for every scope key:** `src/agents/sandbox/prune.ts:70-76`
  - body: The remaining-entry guard only runs for `agent` scope. Session scope can also have multiple registry entries for the same scope key, for example after a container-prefix or backend naming change, so pruning the stale entry can delete the workspace used by the newer entry.
  - confidence: 0.78
- **[P2] Persist the workspace root used by each entry:** `src/agents/sandbox/prune.ts:82-83`
  - body: This cleanup uses the current `params.cfg.workspaceRoot` for every registry entry, but sandbox config can be overridden per agent and can change over time. A stale entry created under a different root would leave its real workspace behind or delete an unrelated workspace under the current root.
  - confidence: 0.76

## Security Review
