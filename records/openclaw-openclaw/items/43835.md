## Summary

Keep this PR open, but do not merge the current patch. The underlying bug from #43797 still exists on current main, but this branch is based on an older prune implementation and its workspace cleanup can delete the wrong directory or live workspace data.

## What This Changes

This PR changes `src/agents/sandbox/prune.ts` to delete sandbox workspace directories during sandbox-container prune, skip shared scope, and add an agent-scope remaining-entry guard.

## Best Possible Solution

Rework the fix against current main so workspace cleanup is owned by the current backend-aware prune flow, uses the stored scope key/path information, runs only after confirmed runtime removal, and skips deletion while any registry/browser/runtime entry still uses the same scope. Add targeted tests for session, agent, shared, browser, and removal-failure cases.

## Reproduction Assessment

Yes. The original leak has a direct reproduction in #43797, and the PR regression is statically reproducible from current code: registry entries store `sessionKey: scopeKey`, while the PR re-normalizes that value before deleting the workspace.

## Solution Assessment

No. Cleaning orphaned workspace directories is the right direction, but this patch is not the best implementation because it targets the wrong path for agent-scope entries and deletes after best-effort runtime removal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key, not the raw session key. For an agent-scope entry such as `agent:ops`, passing it back through `resolveSandboxScopeKey("agent", ...)` treats it as a malformed agent session key and resolves to `agent:main`, so prune can delete the wrong workspace and leave the intended one behind.
  - confidence: 0.91
- **[P1] Delete the workspace only after runtime removal succeeds:** `src/agents/sandbox/prune.ts:83`
  - body: `docker rm -f` is called with `allowFailure: true`, so a daemon outage or permission error still reaches the new `fs.rm` call. That can remove the host workspace while the container still exists or is running with that mount.
  - confidence: 0.88
- **[P2] Check all remaining entries for the same scope key:** `src/agents/sandbox/prune.ts:70`
  - body: The duplicate-workspace guard only runs for `agent` scope. Session scope can also have two registry entries for the same scope key, for example after a container prefix change, and pruning the stale entry would delete the workspace used by the newer container.
  - confidence: 0.78

## Security Review

Status: needs_attention
