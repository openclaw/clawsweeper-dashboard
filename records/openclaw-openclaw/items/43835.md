## Summary

Keep open. Current main still leaves sandbox workspaces behind during prune, but this PR is dirty against current main and introduces unsafe recursive workspace deletion without proving the exact workspace is unused or that runtime removal succeeded.

## What This Changes

The PR adds sandbox workspace directory cleanup to the prune path after removing expired sandbox containers.

## Best Possible Solution

Rework this on current main as a backend- and scope-aware prune cleanup that deletes only the exact unused workspace after confirmed runtime removal, preserves shared/agent/remote semantics, and adds focused prune coverage plus a changelog entry.

## Reproduction Assessment

Yes. Current main source shows expired sandbox prune removes runtimes and registry entries but never deletes the non-shared workspace directory.

## Solution Assessment

No. Cleaning orphaned workspaces is the right direction, but this patch is not the best solution because it adds destructive host deletion without a reliable per-entry workspace identity, successful-runtime-removal guarantee, or current-main backend compatibility.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Delete workspaces only after confirmed runtime removal:** `src/agents/sandbox/prune.ts:82-83`
  - body: `removeRuntime` can swallow Docker removal failures, so this cleanup can delete the host workspace even when the container still exists or is still running with that mount. Make runtime removal report success before allowing recursive workspace deletion.
  - confidence: 0.89
- **[P1] Use the stored scope key for cleanup:** `src/agents/sandbox/prune.ts:66-68`
  - body: Registry entries already store the normalized scope key as `sessionKey`; recomputing it with the current config can derive a different workspace for agent-scoped entries. Use the stored key or a persisted workspace identity for deletion.
  - confidence: 0.88
- **[P2] Check remaining entries for every cleanup scope key:** `src/agents/sandbox/prune.ts:70-76`
  - body: The remaining-entry guard only applies to agent scope, but stale duplicate entries can share a session-scope workspace after container naming or backend changes. Pruning one stale entry can delete a workspace still referenced by another entry.
  - confidence: 0.78
- **[P2] Persist the workspace root used by each entry:** `src/agents/sandbox/prune.ts:82-83`
  - body: Cleanup uses the current `cfg.workspaceRoot` for stale entries. If an agent or global sandbox root changed since the entry was created, prune can miss the real orphaned workspace or delete an unrelated directory under the current root.
  - confidence: 0.76
- **[P3] Add the required changelog entry:** `src/agents/sandbox/prune.ts:61`
  - body: This is a user-facing `fix(sandbox)` change, but the PR only changes code. Add a single-line `CHANGELOG.md` entry for the active version before merge.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff introduces recursive host workspace deletion in prune, which is security-sensitive until tied to a confirmed-unused runtime and stable workspace identity.

Concerns:

- **[medium] Recursive deletion can target live workspace data:** `src/agents/sandbox/prune.ts:83`
  - body: The PR adds `fs.rm(..., { recursive: true, force: true })` after best-effort runtime removal and fragile scope/path derivation, so prune can remove host data still used by another sandbox or by a runtime that failed to stop.
