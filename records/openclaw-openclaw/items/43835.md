## Summary

Keep open. The underlying sandbox workspace leak is still present on current main, but this PR is not safe to merge because its recursive cleanup can target the wrong or still-live workspace and the branch predates the current backend-aware prune contract.

## What This Changes

This PR adds recursive sandbox workspace directory cleanup after sandbox-container prune, skipping shared scope and adding an agent-scope remaining-entry guard.

## Best Possible Solution

Rework this on current main as a backend- and scope-aware prune cleanup that deletes only the exact unused workspace after confirmed runtime removal, preserves shared/agent/remote semantics, and adds focused prune coverage plus a changelog entry.

## Reproduction Assessment

Yes. Current main source shows expired sandbox prune removes runtimes and registry entries without deleting the non-shared workspace directory; the PR regressions are also statically reproducible from the PR head code.

## Solution Assessment

No. Cleaning orphaned workspaces is the right direction, but this patch is not the best solution because it adds destructive host deletion without a reliable per-entry workspace identity or successful-runtime-removal guarantee and is stale against backend-aware prune.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key. For an agent-scope entry like `agent:ops`, calling `resolveSandboxScopeKey("agent", entry.sessionKey)` can treat that stored key as a malformed raw session and resolve a different workspace, deleting the wrong directory while leaving the intended one behind.
  - confidence: 0.91
- **[P1] Delete workspaces only after confirmed runtime removal:** `src/agents/sandbox/prune.ts:83`
  - body: This adds `fs.rm` after `docker rm -f` is run with `allowFailure: true` and failures are swallowed. If Docker is unavailable or removal fails, prune can delete the host workspace while the container still exists or is still running with that mount.
  - confidence: 0.89
- **[P2] Check remaining entries for every scope key:** `src/agents/sandbox/prune.ts:70-76`
  - body: The remaining-entry guard only runs for agent scope. Session scope can also have multiple registry entries for the same scope key after naming or backend changes, so pruning a stale entry can delete the workspace used by a newer live entry.
  - confidence: 0.78
- **[P2] Persist the workspace root used by each entry:** `src/agents/sandbox/prune.ts:82-83`
  - body: Cleanup uses the current `params.cfg.workspaceRoot` for every stale entry. If the agent or global sandbox root changed since the entry was created, prune can miss the real orphaned workspace or delete an unrelated directory under the current root.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff adds recursive host workspace deletion without enough proof that the targeted runtime and scope are no longer live.

Concerns:

- **[medium] Recursive deletion can target live workspace data:** `src/agents/sandbox/prune.ts:83`
  - body: `fs.rm(..., { recursive: true, force: true })` runs after best-effort Docker removal and with fragile scope/path derivation, so prune can delete host workspace data still used by another sandbox or by a container that removal failed to stop.
  - confidence: 0.87
