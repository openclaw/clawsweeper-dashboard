## Summary

Keep open. Current main has partial Telegram transport fallback through sticky IPv4 and one hard-coded pinned Telegram API IP, but the requested configurable or discoverable fallback-IP transport is still absent; the added restricted-network comment is useful evidence, while DoH discovery, proxy precedence, and network-bypass policy need maintainer review before automation.

## What This Changes

Request configurable Telegram Bot API fallback-IP transport for restricted networks, including optional IP discovery, sticky successful-IP selection, proxy fallback, and `fallbackIps`/`disableFallback` config knobs.

## Best Possible Solution

Keep the work in the Telegram plugin, but define the public network policy first: decide whether fallback IPs are manual-only or auto-discovered, preserve TLS/SNI and existing proxy/env behavior, keep media SSRF protections intact, document the config, and add focused tests for timeout-triggered fallback paths including `sendChatAction`.

## Reproduction Assessment

Unclear. The report and follow-up comment provide concrete restricted-network symptoms, and current tests reproduce the lower-level fallback machinery, but there is no high-confidence local or CI reproduction for the exact China-mainland `sendChatAction` timeout restart loop.

## Solution Assessment

Unclear. The current pinned-IP fallback is a useful partial solution, but the proposed automatic discovery and bypass behavior need a maintainer policy/security decision; a safer first step may be a documented manual `fallbackIps` config plus targeted timeout fallback tests.
