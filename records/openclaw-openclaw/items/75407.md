## Summary

Keep open. Current main only has a partial Telegram transport fallback with one hard-coded pinned IP and sticky retry behavior; it still lacks the requested configurable fallback list, disable switch, discovery policy, and broader timeout-forced coverage noted in the discussion.

## What This Changes

Add Telegram channel fallback-IP transport for restricted networks, with configurable or discovered IPs, sticky selection, and graceful fallback through proxy or error handling.

## Best Possible Solution

Keep the implementation boundary in the Telegram plugin, but first define a maintainer-approved fallback policy. A complete fix should add validated public config, preserve TLS/SNI and proxy precedence, document restricted-network behavior, and cover configured fallback IPs plus timeout-triggered fallback behavior with focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs and observations, and static review verifies the requested config/discovery path is missing, but this review did not reproduce the live `sendChatAction` restart loop on a restricted Telegram network.

## Solution Assessment

Unclear. A Telegram-plugin fallback is the right boundary, but configurable or DoH-discovered IP bypass is not clearly the safest maintainable solution until maintainers decide the network policy and security constraints.
