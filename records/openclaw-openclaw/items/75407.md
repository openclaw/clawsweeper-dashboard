## Summary

Keep open. Current main has sticky IPv4 retry plus one hard-coded pinned Telegram API IP and timeout-forced fallback for `sendChatAction`, but it still lacks the requested configurable or discoverable fallback-IP transport; the remaining work changes Telegram egress/DNS/proxy policy and needs maintainer product/security direction.

## What This Changes

Add a Telegram Bot API fallback-IP transport for restricted networks, with configurable or discoverable fallback IPs, sticky selection, and defined fallback/proxy behavior.

## Best Possible Solution

Keep this in the Telegram plugin after approved egress policy: add validated operator-controlled fallback-IP configuration or an approved discovery alternative, preserve TLS/SNI and proxy precedence, document the behavior, and cover direct, proxy, and timeout-forced fallback paths with focused tests.

## Reproduction Assessment

No high-confidence live restricted-network reproduction was established. Source inspection does confirm that current main lacks the requested configurable or discoverable fallback-IP path, while retaining only one built-in pinned IP.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs cannot be called the best maintainable solution until maintainers decide egress, DNS, and proxy-precedence policy.
