## Summary

Keep open. Current main has a useful partial Telegram transport fallback with IPv4 and one pinned Telegram API IP, but it does not implement the requested configurable fallback IP list, disable switch, DoH/discovery path, or timeout-forced fallback coverage for `sendChatAction`; the issue comments add concrete restricted-network evidence rather than making the request obsolete.

## What This Changes

The issue requests Telegram Bot API fallback-IP transport for restricted networks, including fallback IP discovery or manual configuration, sticky selection, and graceful proxy/error behavior.

## Best Possible Solution

Keep the work in the Telegram plugin, but first define a maintainer-approved fallback policy. A complete implementation should add validated opt-in config, preserve TLS/SNI and proxy precedence, document restricted-network behavior, and cover configured fallback IPs plus timeout-triggered fallback behavior with focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs and observations, and current tests cover lower-level fallback behavior, but there is no high-confidence live reproduction of the exact current-main `sendChatAction` restart loop.

## Solution Assessment

Unclear. The existing hard-coded sticky fallback is a partial mitigation; the requested configurable or discoverable fallback path may be useful, but the safer direction likely needs maintainer approval before choosing manual IPs, shipped defaults, or DoH discovery.
