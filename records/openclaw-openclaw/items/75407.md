## Summary

Keep open. Current main partially mitigates Telegram network failures with sticky IPv4 and one pinned Telegram API IP, plus timeout fallback for sendChatAction, but it still lacks the requested configurable or discoverable fallback IP list and disable policy.

## What This Changes

Request a Telegram channel fallback-IP transport for restricted networks, including discovered or configured IPs, sticky selection, proxy precedence, and an opt-out.

## Best Possible Solution

Keep the fix in the Telegram plugin, but only after an approved transport policy defines validated fallback IPs, proxy precedence, TLS/SNI preservation, SSRF boundaries, opt-out behavior, docs, and focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up comment provide concrete restricted-network logs, and static review confirms the requested config/discovery path is missing, but I did not live-reproduce the network failure from a restricted environment.

## Solution Assessment

Unclear. The Telegram plugin is the right boundary, but DoH or configurable IP fallback is not clearly the best maintainable solution until maintainers settle egress and proxy-precedence policy; current main is only a partial mitigation.
