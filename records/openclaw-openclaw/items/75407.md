## Summary

Keep open. Current main partially mitigates Telegram network failures with sticky IPv4, timeout-forced control retries, proxy support, and one built-in pinned API IP, but it still lacks the requested configurable or discoverable fallback-IP transport and needs maintainer network/security policy before implementation.

## What This Changes

Add configurable or discoverable Telegram Bot API fallback-IP transport for restricted networks, with sticky selection and clear proxy/error fallback behavior.

## Best Possible Solution

Keep the change in the Telegram plugin after maintainer-approved egress policy, then add validated fallback-IP configuration or an approved alternative that preserves TLS/SNI, defines proxy precedence, updates docs, and covers configured-IP plus timeout-forced fallback paths with focused tests.

## Reproduction Assessment

No high-confidence live restricted-network reproduction was established. Source inspection confirms the requested configurable/discoverable fallback path is absent, and the issue comments provide concrete mainland China logs and follow-up testing context.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs need maintainer decisions on egress policy, DNS behavior, validation, and proxy precedence before this can be called the best fix.
