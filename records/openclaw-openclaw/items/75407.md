## Summary

Keep open. Current main partially mitigates Telegram restricted-network failures with sticky IPv4, one built-in pinned Telegram API IP, and sendChatAction timeout fallback, but the requested configurable or discoverable fallback-IP transport remains absent and needs maintainer egress-policy review.

## What This Changes

The issue requests a Telegram fallback-IP transport for restricted networks, including configured or discovered fallback IPs, sticky selection, and proxy/error precedence.

## Best Possible Solution

Keep the work in the Telegram plugin after maintainer-approved egress policy; if approved, add validated fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, update docs/config metadata, and cover configured-IP plus timeout-forced fallback paths.

## Reproduction Assessment

No high-confidence live reproduction path was established in this review. The issue body and follow-up testing give concrete restricted-network evidence, and source inspection verifies that configurable or discoverable fallback IPs are missing on current main.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs cannot be called the best maintainable solution until maintainers decide egress routing, DNS, and proxy-precedence policy.
