## Summary

Keep open. Current main and v2026.5.2 only provide a partial Telegram restricted-network mitigation with sticky IPv4 and one hard-coded pinned IP; the requested configurable/discoverable fallback-IP list and opt-out policy remain absent and need maintainer egress-policy direction.

## What This Changes

Request to add Telegram Bot API fallback-IP transport with auto/manual fallback IPs, sticky selection, and opt-out config for restricted networks.

## Best Possible Solution

Keep the work in the Telegram plugin after maintainer-approved network policy; add validated fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, document it, and cover configured-IP plus timeout-forced fallback paths with focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs, and source review verifies the requested config/discovery path is missing, but I did not reproduce the live Telegram network failure from a restricted environment.

## Solution Assessment

Unclear. Telegram plugin transport is the right implementation boundary, but configurable or DoH fallback cannot be called the best maintainable solution until maintainers set egress, discovery, and proxy-precedence policy.
