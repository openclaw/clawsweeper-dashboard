## Summary

Keep open. Current main has only a partial Telegram transport fallback with one hard-coded pinned IP and sticky retry; the requested configurable fallback list, disable/discovery policy, and broader timeout-triggered coverage are still missing.

## What This Changes

The issue requests Telegram Bot API fallback-IP transport for restricted networks, including configurable or discovered IPs, sticky selection, opt-out behavior, and proxy-aware degradation.

## Best Possible Solution

Keep the fix in the Telegram plugin, but define a maintainer-approved fallback policy first; then add validated config/env support, preserve TLS/SNI and proxy precedence, document the behavior, and cover configured IPs plus timeout-forced fallback with focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs, and static review verifies the requested config/discovery path is missing, but I did not reproduce the live sendChatAction restart loop on a restricted Telegram network.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IP bypass is not clearly the safest maintainable solution until maintainers decide the network policy and security constraints.
