## Summary

Keep open. Current main has a partial Telegram transport fallback with a hard-coded pinned IP and sticky retry path, but it does not implement the requested configurable fallback IP list, disable switch, DoH discovery, or timeout-forced fallback coverage for `sendChatAction`; the follow-up comment adds useful real-world restricted-network evidence rather than making the request obsolete.

## What This Changes

The issue requests a Telegram channel fallback-IP transport for restricted networks, including auto-discovered or configured fallback IPs, sticky selection, and public config/env controls.

## Best Possible Solution

Keep the work in the Telegram plugin, but first define a maintainer-approved fallback policy. A complete fix should add validated opt-in/public config, preserve TLS/SNI and proxy precedence, document restricted-network behavior, and cover configured fallback IPs plus timeout-triggered fallback behavior with focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete logs and restricted-network observations, and current tests reproduce lower-level fallback behavior, but there is no high-confidence live reproduction of the exact current-main `sendChatAction` restart loop.

## Solution Assessment

Unclear. The current hard-coded sticky fallback is a useful partial mitigation, but the requested configurable/discoverable fallback path is not yet proven to be the narrowest maintainable or safest solution without maintainer security/product review.
