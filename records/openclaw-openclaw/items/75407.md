## Summary

Keep open. Current main has a partial Telegram transport fallback with one hard-coded pinned Telegram API IP and sticky retry behavior, but the requested configurable fallback list, disable switch, discovery policy, and broader timeout-forced coverage remain absent. Because the remaining request changes network-bypass and fallback-IP policy, it should get maintainer security/product direction before an automated fix PR is queued.

## What This Changes

Requests configurable Telegram Bot API fallback-IP transport with discovery, sticky selection, and proxy degradation for restricted networks.

## Best Possible Solution

Keep the implementation boundary in the Telegram plugin, but first define a maintainer-approved fallback policy. If approved, add validated public config/env for fallback IPs, an explicit disable switch, clear proxy precedence, TLS/SNI-preserving transport behavior, focused tests, and docs for restricted-network use; avoid automatic DoH discovery unless maintainers explicitly accept that egress model.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs and observations, and static inspection verifies the requested knobs are missing, but this review did not reproduce the live Telegram restart loop on a restricted network.

## Solution Assessment

Unclear. A Telegram-plugin fallback is the right boundary, but configurable or DoH-discovered fallback IP transport is not clearly the safest maintainable solution until maintainers decide the network-bypass policy and security constraints.
