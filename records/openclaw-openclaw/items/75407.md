## Summary

Keep open. Current main partially mitigates Telegram restricted-network failures with sticky IPv4 and a single pinned-IP retry plus sendChatAction timeout fallback, but the requested configurable/discoverable fallback IP list and disable policy are still absent, and the egress-bypass behavior needs maintainer direction.

## What This Changes

Request to add Telegram Bot API fallback-IP transport with auto/manual fallback IPs, sticky selection, and opt-out config for restricted networks.

## Best Possible Solution

Keep the work in the Telegram plugin after maintainer-approved network policy; add validated fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, document it, and cover configured-IP plus timeout-forced fallback paths with focused tests.

Unclear. The issue and follow-up provide concrete restricted-network logs and static review verifies the requested config/discovery path is missing, but I did not reproduce the live Telegram network failure from a restricted environment.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH fallback cannot be called the best maintainable solution until maintainers set egress and proxy-precedence policy; current main is only a partial mitigation.
