## Summary

Keep open. Current main partially mitigates Telegram restricted-network failures with sticky IPv4, one built-in pinned IP, and timeout-forced control retries including `sendChatAction`, but it still lacks configurable or discoverable fallback IP transport and needs maintainer egress-policy review.

## What This Changes

Request configurable or auto-discovered Telegram Bot API fallback-IP transport for restricted networks where DNS or direct TCP to `api.telegram.org` fails.

## Best Possible Solution

Keep the work in the Telegram plugin after maintainer-approved egress policy; add documented operator-controlled fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, and cover timeout-forced plus configured-IP paths with focused tests.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection confirms the requested configurable/discoverable fallback path is absent, and the issue plus follow-up provide plausible restricted-network logs and testing context.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs cannot be called the best maintainable solution until maintainers decide the egress, DNS, and proxy-precedence policy.
