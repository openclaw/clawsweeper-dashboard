## Summary

Keep open. Current main still only partially mitigates Telegram restricted-network failures with sticky IPv4 and one built-in pinned-IP retry; the requested configurable or discoverable fallback-IP transport remains absent and needs maintainer network-policy review.

## What This Changes

Request configurable or auto-discovered Telegram Bot API fallback IP transport for networks where DNS or direct TCP access to api.telegram.org is blocked.

## Best Possible Solution

Keep the work in the Telegram plugin after maintainer-approved egress policy; add validated fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, document it, and cover configured-IP plus timeout-forced fallback paths with focused tests.

No high-confidence live reproduction path was established here. The issue and follow-up provide concrete restricted-network logs, and source inspection confirms the requested configurable/discoverable fallback path is missing, but I did not verify the network failure from a restricted environment.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs cannot be called the best maintainable solution until maintainers decide egress, DNS, and proxy-precedence policy.
