## Summary

Keep open. Current main has a partial Telegram transport fallback, but it is still a single hard-coded pinned IP with no configured list, DoH discovery, or disable policy; the requested restricted-network bypass behavior also needs maintainer egress-policy review.

## What This Changes

Request configurable or auto-discovered Telegram Bot API fallback-IP transport for restricted networks where `api.telegram.org` DNS or direct TCP access fails.

## Best Possible Solution

After maintainer approval, keep the work inside the Telegram plugin: add validated fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, document the behavior, and cover configured-IP plus timeout-forced fallback paths with focused tests.

## Reproduction Assessment

Unclear. The issue provides concrete restricted-network logs and a follow-up restricted-network proxy test, and static review confirms the requested config/discovery path is missing, but I did not reproduce the live Telegram failure from a restricted network.

## Solution Assessment

Unclear. The Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs should not be treated as the best maintainable solution until maintainers settle the egress-bypass and proxy-precedence policy.
