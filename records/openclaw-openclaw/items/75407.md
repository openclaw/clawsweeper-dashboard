## Summary

Keep open. Current main has a partial Telegram transport fallback, but it does not implement the requested public `fallbackIps`/`disableFallback` configuration, DoH discovery, or broader timeout-forced fallback coverage for `sendChatAction`; the follow-up comment adds useful real-world restricted-network evidence rather than making the item obsolete.

## What This Changes

The issue requests configurable and discoverable Telegram fallback-IP transport so the Telegram plugin can keep reaching `api.telegram.org` in restricted networks when normal DNS or direct TCP fails.

## Best Possible Solution

Keep the work in the Telegram plugin, define a maintainer-approved fallback policy, then add a safe public config surface with validation, documented proxy precedence, TLS/SNI-preserving pinned lookup behavior, and focused tests for configured fallback IPs plus timeout-triggered fallback on affected control/delivery methods.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete logs and environment details, and current tests reproduce low-level `ETIMEDOUT`/`EHOSTUNREACH` fallback behavior, but there is no high-confidence live reproduction of the exact restricted-network `sendChatAction` restart loop on current main.

## Solution Assessment

Unclear. The existing hard-coded pinned-IP fallback is a useful partial mitigation, but the proposed configurable/discoverable fallback list and broader forced-fallback behavior need maintainer policy and security review before this is the best fix.
