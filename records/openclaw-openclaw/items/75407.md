## Summary

Keep open. Current main has a useful partial Telegram transport fallback, but it does not implement the requested configurable fallback IP list, opt-out, DoH discovery, or broader timeout-forced fallback coverage for `sendChatAction`; the follow-up comment also adds concrete restricted-network evidence rather than making the report obsolete.

## What This Changes

Request adding a Telegram Bot API fallback-IP transport for restricted networks, including configurable or discoverable fallback IPs, sticky selection, and proxy-aware graceful degradation.

## Best Possible Solution

Keep this in the Telegram plugin, define a maintainer-approved fallback policy, then add a validated public config surface with documented proxy precedence, TLS/SNI-preserving pinned lookup behavior, safe optional discovery, and focused tests for configured fallback IPs plus timeout-triggered fallback on affected Bot API methods.

## Reproduction Assessment

Unclear. The report and follow-up provide concrete logs and environment details, and current tests reproduce low-level fallback behavior, but there is no live reproduction of the exact restricted-network `sendChatAction` restart loop on current main.

## Solution Assessment

Unclear. The existing single pinned-IP fallback is a partial mitigation, but the proposed configurable/discoverable fallback list and broader timeout fallback need maintainer policy and security review before they are clearly the best fix.
