## Summary

Keep open. Current main has a partial Telegram transport fallback from normal DNS to sticky IPv4 and then to one hard-coded pinned Telegram API IP, and proxy/DNS tuning is documented, but the requested configurable or auto-discovered fallback IP surface is still missing. The follow-up restricted-network testing also keeps the report useful, especially because timeout-triggered `sendChatAction` still does not use the forced fallback allowlist.

## What This Changes

Request to add Telegram fallback-IP transport for restricted networks, including discovery or configurable IPs, sticky selection, and graceful fallback to proxy or error.

## Best Possible Solution

Keep the work in the Telegram plugin, define a maintainer-approved policy for fallback IP configuration or discovery, preserve TLS/SNI and pinned-host safeguards, document proxy precedence, and add focused tests for configurable fallback behavior plus timeout-triggered `sendChatAction` fallback before shipping it.

## Reproduction Assessment

Unclear. The report includes concrete logs and a China-mainland environment, and the follow-up comment adds real restricted-network proxy testing, but there is no high-confidence live reproduction for the exact GFW `sendChatAction` restart loop on current main.

## Solution Assessment

Unclear. The current hard-coded pinned-IP fallback is only a partial solution; the requested configurable/DoH fallback path may be reasonable, but the safest maintainable design needs explicit proxy, privacy, and SSRF-boundary decisions first.
