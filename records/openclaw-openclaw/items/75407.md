## Summary

Keep open. Current main has a partial Telegram transport fallback chain with sticky IPv4 and one pinned Bot API IP, plus timeout-forced retry for sendChatAction, but it does not provide the requested configurable or discoverable fallback IP transport and the egress-bypass behavior needs maintainer policy review.

## What This Changes

Request configurable or auto-discovered Telegram Bot API fallback IP transport for restricted networks where DNS or direct TCP access to api.telegram.org is blocked.

## Best Possible Solution

Keep this in the Telegram plugin after maintainer-approved network policy; add validated fallback-IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, document the behavior, and cover configured-IP plus timeout-forced fallback paths with focused tests.

## Reproduction Assessment

No high-confidence live reproduction path was established here. The issue and follow-up provide concrete restricted-network logs, and source review confirms the requested configurable/discoverable fallback path is missing, but I did not verify the network failure from a restricted environment.

## Solution Assessment

Unclear. Telegram plugin transport is the right boundary, but configurable or DoH-discovered fallback IPs cannot be called the best maintainable solution until maintainers decide egress, DNS, and proxy-precedence policy.
