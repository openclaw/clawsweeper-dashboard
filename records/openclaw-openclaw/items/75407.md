## Summary

Keep open. Current main has only a partial Telegram transport fallback: direct transport, sticky IPv4-only retry, and one hard-coded pinned Telegram API IP; the requested configurable or discoverable fallback IP list, disable policy, and broader timeout-triggered coverage are still missing.

## What This Changes

Request configurable Telegram Bot API fallback-IP transport for restricted networks, including manual/discovered IPs, sticky selection, and fallback/proxy ordering.

## Best Possible Solution

Keep the change in the Telegram plugin after maintainer-approved network policy: add validated fallback IP configuration or an approved alternative, preserve TLS/SNI and proxy precedence, document it, and cover configured IP plus timeout-forced fallback paths with focused tests.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs, and static review verifies the requested config/discovery path is missing, but I did not reproduce the live Telegram network failure from a restricted environment.

## Solution Assessment

Unclear. The Telegram plugin is the right implementation boundary, but IP/DoH fallback as a bypass mechanism needs explicit maintainer security and product direction before it can be called the best maintainable solution.
