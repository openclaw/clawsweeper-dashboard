## Summary

Keep open. Current main and the reported release only provide a partial Telegram transport fallback: one hard-coded pinned Telegram API IP with sticky retry behavior. The requested configurable fallback IP list, disable switch, discovery policy, and broader timeout-forced coverage remain missing, and the network-bypass policy needs maintainer security/product direction before implementation.

## What This Changes

The issue requests configurable Telegram Bot API fallback-IP transport for restricted networks, including optional IP lists, sticky selection, discovery, and opt-out behavior.

## Best Possible Solution

Keep the boundary in the Telegram plugin, but first define a maintainer-approved fallback policy. A complete solution should add validated public config, an opt-out or opt-in model, clear proxy precedence, TLS/SNI preservation, docs for restricted-network operation, and focused tests for configured fallback IPs plus timeout-triggered fallback behavior.

## Reproduction Assessment

Unclear. The issue and follow-up provide concrete restricted-network logs and observations, and static review verifies the requested config/discovery path is missing, but this review did not reproduce the live `sendChatAction` restart loop on a restricted Telegram network.

## Solution Assessment

Unclear. Telegram-plugin fallback is the right implementation boundary, but automatic DoH or fallback-IP bypass is not clearly the safest maintainable solution until maintainers decide the network and security policy; an operator-configured, validated list with explicit proxy precedence looks safer than silent discovery.
