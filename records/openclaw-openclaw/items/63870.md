## Summary

Keep this PR open, but it is not mergeable as-is: the branch adds a useful operator-facing config idea that is not on current main, yet the patch has a TypeScript scoping error, targets the old eager gateway MCP startup path, and lacks after-fix real behavior proof.

## What This Changes

The branch adds `gateway.mcpLoopback.port` config type/schema/help/generated metadata, uses it when starting the gateway MCP loopback server, improves bind-failure logging, and adds a fixed-port loopback test.

## Best Possible Solution

If maintainers want this feature, route the validated port through the current lazy CLI MCP startup path, document the operator behavior and collision tradeoff, preserve loopback/bearer protections, and require real gateway-plus-CLI proof before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the PR head references a block-scoped variable outside its `try`, and current main starts bundle MCP from `prepareCliRunContext()` via `ensureMcpLoopbackServer()` without a configured port. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but this branch should rebase onto the lazy CLI preparation path and add public docs plus real behavior proof instead of editing stale eager gateway startup code.
