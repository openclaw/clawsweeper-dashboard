## Summary

Keep this PR open, but do not merge the branch as-is. Current main still lacks the requested `gateway.mcpLoopback.port` setting, while the PR is stale against the current lazy bundled-MCP startup path and contains a compile-blocking scope error in the proposed gateway startup change.

## What This Changes

The PR adds a `gateway.mcpLoopback.port` config field, generated schema/help/label metadata, a fixed-port MCP loopback test, and gateway startup wiring intended to let operators pin the local MCP loopback HTTP port.

## Best Possible Solution

If OpenClaw still wants operator-pinned MCP URLs, implement the setting as a documented public gateway config surface wired into the current lazy CLI backend preparation path, with tests for config validation, fixed-port startup, bind-failure diagnostics, and unchanged loopback/token security. If maintainers decide stable resume hashing is the preferred solution, record that decision and close or supersede this PR explicitly.

No. The requested direction is plausible, but this branch is not the best current implementation because it wires the port into the old eager gateway startup path instead of `src/agents/cli-runner/prepare.ts`, misses public docs/changelog follow-through, and contains a compile blocker.
