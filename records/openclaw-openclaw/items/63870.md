## Summary

Keep open: current main still lacks the requested configurable MCP loopback port, but this PR is not merge-ready because it does not compile, targets the old eager gateway startup path, and lacks after-fix real behavior proof.

## What This Changes

The PR adds `gateway.mcpLoopback.port` config/schema/help/generated-schema metadata, a fixed-port helper test, and attempts to pass that port into gateway MCP startup.

## Best Possible Solution

If the feature is still desired, add the config to the current lazy CLI backend preparation path with docs, config validation, fixed-port coverage, and preserved 127.0.0.1/bearer/origin protections.

## Reproduction Assessment

Yes. Static inspection of current main shows the strict gateway schema has no `mcpLoopback` field and `prepareCliRunContext()` still calls `ensureMcpLoopbackServer()` without a configured port; the PR diff also has a clear block-scoping compile error.

## Solution Assessment

No. The feature direction is plausible, but this branch updates the obsolete eager gateway startup path; the maintainable fix is to pass the validated port into lazy CLI preparation and document the new operator setting.
