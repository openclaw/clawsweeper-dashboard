## Summary

## What This Changes

The PR adds a `gateway.mcpLoopback.port` config/schema/help surface, uses it when starting the MCP loopback server, and adds a fixed-port loopback test.

## Best Possible Solution

If maintainers accept the config, implement it at the lazy CLI MCP startup seam, pass a validated port into `ensureMcpLoopbackServer()` from run preparation, document collision behavior, and prove a real restart flow.

## Reproduction Assessment

Yes for the PR defects: the scoped-variable error is visible in the diff, and current main shows the active MCP startup path is lazy `prepareCliRunContext()` with no configured port. The underlying operator need still lacks real gateway-plus-CLI proof.

## Solution Assessment

No. The feature direction may be acceptable, but this branch is not the narrowest maintainable solution until maintainers accept the new config contract and the implementation is rebased onto the current lazy startup path.
