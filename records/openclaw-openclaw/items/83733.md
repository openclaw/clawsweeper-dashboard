## Summary

Keep open for normal PR review and merge: current main still has the pinned setup-registry path that can hide a send-capable runtime registry, and this PR is a narrow fix with focused regression coverage and no blocking review findings.

## What This Changes

The branch updates outbound channel registry resolution to fall back from a pinned setup-only channel entry to the active runtime registry and adds regression tests for bootstrap and delivery.

## Best Possible Solution

Land the focused pinned-first registry fallback after normal CI, preserving existing pinned-adapter behavior while allowing active-runtime lookup only when the pinned channel entry cannot send.

## Reproduction Assessment

Yes, at source level. Current main prefers a non-empty pinned channel registry, the loader only reads that selected registry, and bootstrap skips runtime loading when any matching channel id exists, matching the setup-only pinned-entry failure path.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix I found: pinned registry entries still win when they have the requested sender, and active runtime fallback is limited to absent send-capable surfaces.
