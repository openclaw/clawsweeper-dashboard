## Summary

Keep open: the reported workspace-bootstrap symlink problem is real, but this PR is not safe to merge as-is because it relaxes a shared boundary primitive rather than only the fixed bootstrap-file reader. The safer path is the open scoped replacement work plus maintainer/security review of the prompt-bootstrap trust boundary.

## What This Changes

The PR adds `~/.openclaw` containment fallbacks to three shared boundary-path checks so workspace symlinks can resolve to sibling OpenClaw state directories.

## Best Possible Solution

Keep the shared fs-safe boundary root-confined by default and land or explicitly supersede a scoped fixed-bootstrap reader implementation such as https://github.com/openclaw/openclaw/pull/73401 with negative security coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a fixed bootstrap symlink resolving outside the workspace root reaches `openRootFile`, fails the workspace boundary, and is recorded as a missing bootstrap entry. I did not create a filesystem fixture because this review is read-only.

## Solution Assessment

No. Relaxing `src/infra/boundary-path.ts` globally is not the narrowest maintainable solution; a workspace-bootstrap-specific reader exception or dedicated boundary mode is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict the fallback to symlink traversal:** `src/infra/boundary-path.ts:642-646`
  - body: `assertLexicalBoundaryOrCanonicalAlias` runs for any absolute path outside the root, not only for followed symlinks. With this return, direct paths under sensitive OpenClaw state directories can pass the workspace boundary without symlink traversal, so the exception must not live in this generic outside-lexical guard.
  - confidence: 0.9
- **[P1] Scope the exception to bootstrap readers:** `src/infra/boundary-path.ts:316-317`
  - body: `applyResolvedSymlinkHop` is a shared boundary primitive used beyond workspace bootstrap. Any sandbox, hook, install, media, or generic safe-file root that contains a symlink to `~/.openclaw` would inherit a broader target boundary than the caller requested, so this needs a bootstrap-specific mode or reader-layer fallback.
  - confidence: 0.88
- **[P2] Use the configured canonical state root:** `src/infra/boundary-path.ts:316-317`
  - body: The fallback computes `path.join(os.homedir(), '.openclaw')`, which ignores configured OpenClaw state locations and compares a lexical path with realpath-derived values. That makes the trust decision wrong for overridden or symlinked state roots, so the effective state root must be resolved canonically before use.
  - confidence: 0.82

## Security Review
