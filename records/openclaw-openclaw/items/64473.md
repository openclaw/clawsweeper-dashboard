## Summary

Keep open: the linked user problem is still real on current main, but this PR is not safe to merge because it widens shared boundary checks beyond workspace bootstrap symlinks and now targets code that has moved behind the fs-safe wrapper. The paired issue is still open, and https://github.com/openclaw/openclaw/pull/73401 is the narrower active replacement path.

## What This Changes

The PR adds hard-coded `~/.openclaw` containment fallbacks to three `src/infra/boundary-path.ts` checks so symlinks resolving outside the workspace root but inside the OpenClaw state tree pass.

## Best Possible Solution

Keep shared fs-safe boundary helpers root-confined by default, then land or explicitly supersede a scoped fixed-bootstrap reader implementation with negative security coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live fixture. A fixed bootstrap symlink resolving outside the workspace reaches `openRootFile` with the workspace root, the failed read is marked missing, and prompt rendering emits `[MISSING] Expected at: ...`.

## Solution Assessment

No. A global `~/.openclaw` fallback in shared boundary checks is broader than the reported bootstrap-file symptom; a scoped reader exception such as the active replacement PR is the safer maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict the fallback to symlink traversal:** `src/infra/boundary-path.ts:642-646`
  - body: `assertLexicalBoundaryOrCanonicalAlias` runs for any absolute path outside the root, not only followed symlinks. Returning here for every canonical path under `~/.openclaw` can let direct state paths pass the workspace boundary, so this exception must not live in the generic guard.
  - confidence: 0.9
- **[P1] Scope the exception to bootstrap reads:** `src/infra/boundary-path.ts:313-317`
  - body: `applyResolvedSymlinkHop` is a shared boundary primitive. Any sandbox, hook, install, media, or generic safe-file root containing a symlink to `~/.openclaw` would inherit a broader target boundary than the caller requested, so this needs a bootstrap-specific mode or reader-layer fallback.
  - confidence: 0.88
- **[P2] Use the configured canonical state root:** `src/infra/boundary-path.ts:316-317`
  - body: The fallback computes `path.join(os.homedir(), '.openclaw')`, which ignores `OPENCLAW_STATE_DIR` and the configured home directory and compares a lexical path with realpath-derived values. That makes the trust decision wrong for overridden or symlinked state roots.
  - confidence: 0.82
