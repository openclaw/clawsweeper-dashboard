## Summary

Keep open. Current main still rejects the reported workspace-bootstrap symlink pattern, but this PR is not safe to merge as written because it relaxes a shared boundary-path primitive beyond workspace bootstrap reads. The same user-visible behavior is also being handled in open maintainer-labeled PR #73401, while linked issue #64472 remains open, so the safe next step is maintainer/security review rather than cleanup closure.

## What This Changes

The PR changes `src/infra/boundary-path.ts` so boundary checks allow canonical paths under `os.homedir()/.openclaw` even when they are outside the configured root.

## Best Possible Solution

Use one canonical, security-reviewed workspace-bootstrap reader fix: allow only explicit fixed bootstrap-file symlinks under an agreed trust policy, preserve direct-path and non-workspace rejection, keep extra/glob bootstrap paths boundary-bound, and add positive plus negative regression coverage for credentials, sessions, dangling links, hardlinks, non-regular files, oversized files, and race-sensitive reads.

## Reproduction Assessment

Yes. The reported bug has a high-confidence source reproduction path: a workspace `AGENTS.md` symlink to a sibling `~/.openclaw/kb` target is read through `openBoundaryFile` with the workspace as root, then rejected by the current boundary containment and recorded as missing. The PR regressions are also source-reproducible by passing a direct path under `~/.openclaw/credentials` or by using a non-workspace boundary root with a symlink into `~/.openclaw`.

## Solution Assessment

No. This PR fixes the missing-bootstrap symptom by widening generic containment in `boundary-path.ts`, but the narrow maintainable solution belongs in the workspace bootstrap reader or an explicit boundary mode with a canonical effective state root and negative security tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict the fallback to symlink traversal:** `src/infra/boundary-path.ts:645-649`
  - body: `assertLexicalBoundaryOrCanonicalAlias` runs for any absolute path that is lexically outside the root, not only followed symlinks. With this fallback, a direct path under `~/.openclaw/credentials` can pass the workspace boundary without symlink traversal, so the exception needs to stay out of this generic guard or be tied to an actual symlink-hop state.
  - confidence: 0.9
- **[P1] Gate the exception to workspace roots:** `src/infra/boundary-path.ts:316-317`
  - body: `applyResolvedSymlinkHop` is shared by sandbox, tmp, managed-media, SSH upload, and generic safe-file callers. As written, any such boundary root can contain a symlink to `~/.openclaw/...` and inherit a broader trust boundary than the caller requested, so the exception must be scoped to the intended workspace-bootstrap root or a dedicated mode.
  - confidence: 0.88
- **[P2] Use the configured canonical state root:** `src/infra/boundary-path.ts:316-317`
  - body: The fallback computes `path.join(os.homedir(), ".openclaw")`, which ignores configured state locations such as `OPENCLAW_STATE_DIR` and compares a lexical path with canonical realpaths. That makes the trust decision wrong for overridden or symlinked state roots, so resolve the effective state root canonically before using it.
  - confidence: 0.83
