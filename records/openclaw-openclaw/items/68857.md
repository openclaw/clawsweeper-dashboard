## Summary

Keep this PR open. Current main still lacks a configurable default for gateway `node.invoke`, but the proposed patch is not ready to merge because the advertised config key is not added to the strict runtime schema and the configured default is not sent to node hosts when per-call `timeoutMs` is omitted.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry.defaultInvokeTimeoutMs` fallback, and startup wiring from gateway config into the node registry.

## Best Possible Solution

Finish this as a complete gateway config contract: accept and document `gateway.nodes.invokeTimeoutMs`, validate it as a finite positive integer with an agreed bound, use one resolved effective timeout for both the gateway wait timer and `node.invoke.request.timeoutMs`, preserve per-call override precedence, and cover the default/configured/override paths with focused tests.

## Reproduction Assessment

Yes. A static reproduction is clear: current main’s strict `gateway.nodes` schema rejects the new key, and the PR diff still sends `params.timeoutMs` to the node before applying its default fallback, so omitted per-call timeouts leave node hosts on their existing 30000 ms default.

## Solution Assessment

No, not as written. The approach is narrow and plausible, but the best fix needs schema/help/docs/changelog/test coverage plus resolved-timeout propagation to the node payload and positive-integer validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:398-399`
  - body: The PR only adds `invokeTimeoutMs` to the TypeScript config type. The runtime `gateway.nodes` zod object is strict and does not include this key, so a user who follows the new example will get an unknown-property config error before the gateway can use the value.
  - confidence: 0.94
- **[P2] Send the resolved default timeout to nodes:** `src/gateway/node-registry.ts:142-143`
  - body: The new fallback is computed after the `node.invoke.request` payload is sent, while the payload still contains `timeoutMs: params.timeoutMs`. When callers omit per-call `timeoutMs`, Swift node hosts receive `undefined` and keep their own 30000 ms timeout, so the configured default only changes how long the gateway waits.
  - confidence: 0.91
- **[P2] Reject non-positive configured timeouts:** `src/gateway/server.impl.ts:488-490`
  - body: This startup guard accepts any number, so `0` or a negative `invokeTimeoutMs` can be copied into the registry and make invocations time out immediately or near-immediately. The new config should be validated as a finite positive integer, preferably with the same bound used in the zod schema.
  - confidence: 0.88
- **[P3] Add release notes for the new config key:** `src/config/types.gateway.ts:398-399`
  - body: This adds a user-facing gateway config option, but there is no `CHANGELOG.md` entry in the PR. OpenClaw policy expects user-facing feature/config changes to be listed under Unreleased with contributor credit.
  - confidence: 0.83
