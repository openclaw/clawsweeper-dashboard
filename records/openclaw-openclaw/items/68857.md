## Summary

Keep open. The PR targets a real gateway-node timeout gap, but the branch is incomplete for a new public config contract: the advertised key is not accepted by the strict runtime schema, the resolved default is not sent to node hosts, invalid numeric values are accepted, and there is no real behavior proof.

## What This Changes

The branch adds a `gateway.nodes.invokeTimeoutMs` TypeScript field, a `NodeRegistry` default timeout fallback, and startup wiring from gateway config to the registry.

## Best Possible Solution

Land this only as a complete gateway config contract: schema/help/labels/docs, finite positive bounded validation, one resolved timeout used by the gateway timer and node payload, caller-timeout coverage or clear scope, focused tests, and real behavior proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the strict schema rejects the advertised key, the node payload still sends only the per-call timeout, and the caller request budget remains separate. This is a feature PR, not a current-main bug reproduction.

## Solution Assessment

No, not as written. The direction is reasonable, but the patch needs the full config contract, validation, resolved-timeout propagation, caller-budget handling or explicit scope, tests, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:377`
  - body: The PR adds the public TypeScript field, but the runtime `gateway.nodes` zod object is strict and still lacks `invokeTimeoutMs`. A user following the new example would hit an unknown-property config error before startup can wire the value.
  - confidence: 0.94
- **[P2] Send the resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The fallback is computed only after `node.invoke.request` is sent, while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, node hosts receive `undefined` and keep their own 30000 ms default, so the configured default only changes how long the gateway waits.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. That can make invocations time out immediately or behave inconsistently, so the schema and startup path should require a finite positive bounded integer.
  - confidence: 0.88
