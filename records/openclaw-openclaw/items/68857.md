## Summary

Keep open. The PR is a useful direction and remains the canonical implementation path for https://github.com/openclaw/openclaw/issues/68090, but the current patch is incomplete: the advertised strict-config key is not accepted by the runtime schema, the resolved default is not sent to node hosts, configured values are not validated, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a mutable `NodeRegistry` default timeout fallback, and startup wiring from gateway config.

## Best Possible Solution

Expose the option only as a complete gateway config contract: runtime schema, help/labels/docs, finite positive bounded validation, one resolved timeout used for both the gateway timer and node payload, and focused regression coverage.

## Reproduction Assessment

Yes. Static inspection of the PR and current main is enough for the PR findings: the advertised strict-config key is absent from the runtime schema, and node request payloads still send only `params.timeoutMs` when callers omit a per-call timeout.

## Solution Assessment

No, not as written. The direction is reasonable, but the patch needs runtime schema/metadata/docs, finite positive bounded validation, and resolved-timeout propagation before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:377-378`
  - body: The PR adds the public TypeScript field, but `gateway.nodes` is validated by a strict zod object that still lacks `invokeTimeoutMs`. A user following the new example would hit an unknown-property config error before startup can wire the value.
  - confidence: 0.94
- **[P2] Send the resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The fallback is computed only after `node.invoke.request` is sent, while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, native node hosts receive no timeout and keep their own 30000 ms default, so the configured default only changes how long the gateway waits.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. That can make invocations time out immediately or behave inconsistently, so the schema and startup path should require a finite positive bounded integer.
  - confidence: 0.88
