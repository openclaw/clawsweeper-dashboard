## Summary

Keep open. The feature direction is plausible, but current main has not implemented it and this PR is still incomplete: the strict runtime schema would reject the advertised key, node hosts would not receive the resolved default timeout, validation is missing, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry` default timeout fallback, and startup wiring from gateway config into node invocation.

## Best Possible Solution

Land this only as a complete gateway config contract: strict schema, help/labels/docs, bounded finite validation, one resolved timeout used by both the gateway timer and node payload, and focused config/node-registry coverage.

## Reproduction Assessment

Yes for the PR findings. Source inspection on current main and the PR diff shows the advertised config key is absent from the strict zod schema, and the node request payload still uses only the per-call `timeoutMs`.

## Solution Assessment

No, not as written. The direction is maintainable, but the patch needs runtime schema support, config metadata/docs, bounded validation, resolved-timeout propagation to node hosts, and tests before it is the right fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:377-378`
  - body: The new TypeScript field is advertised as `openclaw.json` config, but the `gateway.nodes` zod object is strict and still omits `invokeTimeoutMs`. A user setting the example key would hit an unrecognized-property config error before startup can wire it.
  - confidence: 0.94
- **[P2] Send the resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The branch computes the fallback only for the gateway timer; the `node.invoke.request` payload still sends `params.timeoutMs`. With no per-call timeout, Swift node hosts decode `nil` and keep their own 30000 ms default, so the configured default does not govern node-side execution.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. The config/schema path should require a finite positive bounded integer, or explicitly design a no-timeout mode, before exposing this key.
  - confidence: 0.88
