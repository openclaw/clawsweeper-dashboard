## Best Possible Solution

Land this only as a complete gateway config contract: schema/help/docs, finite positive bounded validation, one resolved timeout used by both the gateway timer and node payload, and focused config plus node-registry coverage.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the strict schema rejects the advertised key and the invoke payload still sends only the per-call timeout. This is a feature PR rather than a broken-current-main bug reproduction.

## Solution Assessment

No, not as written. The direction is reasonable, but the patch needs runtime schema support, validation, resolved-timeout propagation, tests/docs, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add runtime schema support for invokeTimeoutMs:** `src/config/types.gateway.ts:377`
  - body: The PR adds the public TypeScript field, but the runtime `gateway.nodes` zod object is strict and still lacks `invokeTimeoutMs`. A user following the new example would hit an unknown-property config error before startup can wire the value.
  - confidence: 0.94
- **[P2] Send the resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The fallback is computed only after `node.invoke.request` is sent, while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, node hosts receive no value and keep their own 30000 ms default, so the configured default only changes how long the gateway waits.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. That can make invocations time out immediately or behave inconsistently, so the schema and startup path should require a finite positive bounded integer.
  - confidence: 0.88
