## Summary

Keep open. Current main does not implement a configurable node invoke default, and the live PR still has merge-blocking config-contract gaps plus missing real behavior proof.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry` default timeout fallback, and startup wiring from gateway config into the node registry.

## Best Possible Solution

Land this only as a complete gateway config contract: strict schema, help/labels/docs, bounded positive validation, one resolved timeout used by both the gateway timer and node payload, and focused config/node-registry tests.

## Reproduction Assessment

Yes. Static inspection is enough for the PR findings: the branch advertises a strict-config key that current schema does not accept, and the PR still sends `undefined` to node hosts when callers omit per-call `timeoutMs`.

## Solution Assessment

No, not as written. The direction is maintainable, but the patch needs runtime schema support, bounded validation, resolved-timeout propagation, docs/metadata, and tests before it is the right fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:377`
  - body: The PR adds the public TypeScript field, but the runtime `gateway.nodes` zod object is strict and still lacks `invokeTimeoutMs`. A user following the new example would hit an unknown-property config error before startup can wire the value.
  - confidence: 0.94
- **[P2] Send the resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The fallback is computed only after `node.invoke.request` is sent, while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, node hosts receive `undefined` and keep their own 30000 ms default, so the configured default only changes how long the gateway waits.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. That can make invocations time out immediately or behave inconsistently, so the schema and startup path should require a finite positive bounded integer.
  - confidence: 0.88
