## Summary

Keep open. Current main still has the fixed 30s node.invoke default, and the PR is a reasonable feature direction but remains incorrect because the advertised config key is not accepted by the strict runtime schema, the resolved timeout is not sent to node hosts, invalid numeric values are not guarded, and no real gateway/node-host proof has been posted.

## What This Changes

This PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry` default timeout fallback, and gateway startup wiring from config into the registry.

## Best Possible Solution

Land this only as a complete gateway config contract: strict schema/help/docs support, finite positive bounded validation, one resolved timeout used for both the gateway timer and node payload, focused tests, and redacted real behavior proof.

## Reproduction Assessment

Yes for the PR review findings: source inspection of current main and the PR diff shows the strict schema omission, payload timeout mismatch, and missing value validation. This is a feature PR rather than a broken-current-main bug reproduction.

## Solution Assessment

No, not as written. The requested direction is plausible, but the patch needs schema/docs/test coverage, finite positive bounded validation, resolved-timeout propagation to node hosts, and real behavior proof before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add runtime schema support for invokeTimeoutMs:** `src/config/types.gateway.ts:377`
  - body: The PR adds the public TypeScript field, but `gateway.nodes` is parsed by a strict zod object that still omits this key. A user following the new example config would hit an unknown-property config error before startup can copy the value.
  - confidence: 0.94
- **[P2] Send resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The fallback is computed only after `node.invoke.request` is sent, while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, node hosts receive no value and keep their own 30000 ms default even if the gateway waits longer.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. The schema and startup path should require a finite positive bounded integer so configured defaults cannot make invokes time out immediately or behave inconsistently.
  - confidence: 0.88
