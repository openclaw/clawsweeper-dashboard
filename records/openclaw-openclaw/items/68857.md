## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry` default timeout fallback, and startup wiring from gateway config into the node registry.

## Best Possible Solution

Land this only as a complete gateway config contract: schema, help/labels/tags, docs, bounded validation, one resolved timeout used by both gateway and node payload, override precedence tests, and focused node-registry/config coverage.

## Reproduction Assessment

Yes. Static inspection is enough for the PR findings: the branch advertises a config key that current strict schema does not accept, and the branch still sends `undefined` to node hosts when callers omit per-call `timeoutMs`.

## Solution Assessment

No, not as written. The direction is maintainable, but the patch needs runtime schema support, validation, resolved-timeout propagation, docs/metadata, and tests before it is the right fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:375-377`
  - body: The PR only adds `invokeTimeoutMs` to the TypeScript config type. The runtime `gateway.nodes` zod object is strict and still lacks this key, so a user who follows the new example will get an unknown-property config error before startup can use it.
  - confidence: 0.94
- **[P2] Send the resolved timeout to nodes:** `src/gateway/node-registry.ts:140-143`
  - body: The new fallback is computed after `node.invoke.request` is sent, while the payload still uses `timeoutMs: params.timeoutMs`. When callers omit a per-call timeout, node hosts receive `undefined` and keep their own 30000 ms default, so the configured default only changes how long the gateway waits.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: The startup guard accepts any number, so `0`, negative, `NaN`, or infinite values can be copied into the registry and make invocations time out immediately or behave inconsistently. The config should be finite, positive, integral, and bounded in the schema and startup wiring.
  - confidence: 0.88
