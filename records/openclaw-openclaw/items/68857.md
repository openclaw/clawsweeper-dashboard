## Summary

Keep open: the feature is still not implemented on current main, and this PR remains the canonical implementation candidate for the closed config request, but it is not merge-ready because the public config contract is incomplete, the resolved timeout is not sent to node hosts, invalid timeout values are accepted, and no real behavior proof is attached.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry` default timeout fallback, and startup wiring from gateway config into the node registry.

## Best Possible Solution

Land this only as a complete gateway config contract: strict schema, help/labels/docs, finite positive bounded validation, one resolved timeout used by both the gateway timer and node payload, and focused config/node-registry tests.

## Reproduction Assessment

Yes. Static inspection of the PR diff plus current main reproduces the merge blockers: the strict schema rejects the new key, and node hosts still receive no default timeout when callers omit per-call `timeoutMs`.

## Solution Assessment

No, not as written. The direction is maintainable, but the patch needs the full config contract, validation, resolved-timeout propagation, docs/metadata, and tests before it is the right fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:377`
  - body: The PR adds a public TypeScript config field, but `gateway.nodes` is validated by a strict zod object that still lacks `invokeTimeoutMs`. A user following the new example would hit an unknown-property config error before startup can wire the value.
  - confidence: 0.94
- **[P2] Send the resolved timeout to node hosts:** `src/gateway/node-registry.ts:140-143`
  - body: The fallback is computed only for the gateway-side timer while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, native node hosts receive no timeout and keep their own 30000 ms default, so the configured default would not extend node-side execution.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: Startup copies any number into the registry, including `0`, negative, `NaN`, or infinite values. That can make invocations time out immediately or behave inconsistently, so the schema and startup path should require a finite positive bounded integer.
  - confidence: 0.88
