## Summary

Keep open: the feature direction is valid, but the PR is still incomplete because the new config key is not accepted by the strict runtime schema, the resolved default timeout is not sent to node hosts, and invalid configured values are not guarded.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a mutable `NodeRegistry` default timeout fallback, and startup wiring from gateway config into node invoke timeouts.

## Best Possible Solution

Finish this as a complete gateway config contract while leaving the broader streaming/progress primitive to #74845: schema, metadata, docs, changelog, validation, tests, and one effective timeout shared by gateway and node host.

## Reproduction Assessment

Yes. Static source review is enough: current main’s strict `gateway.nodes` schema lacks the key, and the PR diff still sends `undefined` to node hosts when only the configured default should apply.

## Solution Assessment

No. The requested config option is a reasonable narrow fix for #68090, but this patch needs runtime schema support, validation, resolved-timeout propagation, docs/changelog, and tests before it is mergeable.

Full review comments:

- **[P1] Add runtime schema for the timeout key:** `src/config/types.gateway.ts:398-399`
  - body: The PR adds `invokeTimeoutMs` only to the TypeScript config type. The runtime `gateway.nodes` zod object is strict and still lacks this key, so a user who follows the new example will get an unknown-property config error before startup can use it.
  - confidence: 0.94
- **[P2] Send the resolved timeout to nodes:** `src/gateway/node-registry.ts:142-143`
  - body: The fallback is computed only after `node.invoke.request` is sent, while the payload still uses `params.timeoutMs`. When callers omit a per-call timeout, node hosts receive `undefined` and keep their own 30000 ms default, so the configured default does not extend node-side execution.
  - confidence: 0.91
- **[P2] Validate configured timeout values:** `src/gateway/server.impl.ts:488-490`
  - body: The startup guard copies any number into the registry, including `0`, negative, `NaN`, or infinite values. That can make invocations time out immediately or behave inconsistently, so the config should be finite, positive, integral, and bounded.
  - confidence: 0.88
- **[P3] Add release notes for the config key:** `src/config/types.gateway.ts:398-399`
  - body: This is a user-facing gateway config option, but the PR does not update `CHANGELOG.md`. OpenClaw policy expects user-facing feature/config changes to be listed under the active release notes.
  - confidence: 0.83
