Full review comments:

- **[P1] Add schema support for the new config key:** `src/config/types.gateway.ts:376-377`
  - body: `invokeTimeoutMs` is added only to the TypeScript config type. The runtime schema for `gateway.nodes` is strict and still allows only `browser`, `pairing`, `allowCommands`, and `denyCommands`, so a user who copies the PR's `openclaw.json` example will get an unrecognized-key validation failure instead of the new timeout behavior.
  - confidence: 0.94
- **[P2] Send the resolved default timeout to node clients:** `src/gateway/node-registry.ts:141`
  - body: The PR changes the gateway wait timer fallback, but the `node.invoke.request` payload is built earlier with `timeoutMs: params.timeoutMs`. When no per-call timeout is provided, Swift and Android nodes still receive no timeout and can enforce their own 30s default, so the configured default does not reliably allow long-running node commands.
  - confidence: 0.9
- **[P2] Reject non-positive configured timeouts:** `src/gateway/server.impl.ts:488-490`
  - body: The startup wiring accepts any number from config, including `0` or negative values. Those values flow into timer resolution and can make every node invoke time out immediately or behave unlike the advertised positive millisecond budget; the config schema should require a positive finite integer and startup should not bypass that contract.
  - confidence: 0.84
