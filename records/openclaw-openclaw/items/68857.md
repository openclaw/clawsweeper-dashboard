## Summary

Keep open. The PR is a plausible fix direction for the timeout feature tracked by https://github.com/openclaw/openclaw/issues/68090, but the current patch is incomplete: the advertised config key is not accepted by the strict runtime schema, the resolved default timeout is not sent to node clients, and no real behavior proof was provided.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` config field, a mutable `NodeRegistry` default timeout, and startup wiring from gateway config for `node.invoke` calls.

## Best Possible Solution

Finish the config contract end-to-end: add bounded schema/help/docs/test coverage, resolve one positive timeout value, send it to the node payload and gateway timer, then provide redacted real node.invoke proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config with `gateway.nodes.invokeTimeoutMs` would hit the strict `gateway.nodes` zod object without that key, and an omitted per-call timeout would still leave node clients on their own 30s fallback.

## Solution Assessment

No. The feature direction is reasonable, but this patch is not the best complete implementation until schema validation, positive bounds, node-payload propagation, and focused proof are added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Add schema support for the new config key:** `src/config/types.gateway.ts:378-379`
  - body: Adding the TypeScript field is not enough because the runtime config validator keeps `gateway.nodes` strict and does not include `invokeTimeoutMs`. A user following the PR usage example will have the key rejected before startup, so the advertised option cannot be used; add the zod schema entry with bounds and regression coverage.
  - confidence: 0.96
- **[P2] Send the resolved timeout to node clients:** `src/gateway/node-registry.ts:140-142`
  - body: The PR changes the gateway-side wait budget but leaves the request payload as `params.timeoutMs`. When callers omit a per-call timeout, Swift and TypeScript node clients receive no timeout and keep their own 30s fallback, so a configured 60s default can still abort on the node side before the gateway timer expires.
  - confidence: 0.9
- **[P2] Reject non-positive configured timeouts:** `src/gateway/server.impl.ts:488-490`
  - body: The startup wiring accepts any number from config. Once schema support is added, `0` or negative values would be assigned to the registry default and make missing-timeout invocations time out immediately via `setTimeout`, so the config path should enforce a positive bounded integer.
  - confidence: 0.86
