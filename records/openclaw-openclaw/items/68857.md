## Summary

Keep open. The PR addresses a valid gateway/node timeout feature request, but current main still lacks the config option and the proposed patch is incomplete: the strict runtime config schema would reject the new key, the resolved default is not sent to node hosts, invalid numeric config values are accepted, and the user-facing config contract is missing docs/changelog/test coverage.

## What This Changes

The PR adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a `NodeRegistry.defaultInvokeTimeoutMs` fallback, and startup wiring from gateway config into the node registry.

## Best Possible Solution

Finish this as a complete gateway config contract: add `gateway.nodes.invokeTimeoutMs` to the runtime schema, help/labels/tags, docs, and changelog; validate it as a finite positive integer with an agreed upper bound; use one effective timeout for both the gateway timer and `node.invoke.request.timeoutMs`; preserve per-call override precedence; and cover default/configured/override behavior with focused tests.

## Reproduction Assessment

Yes. A static reproduction is enough: on current main, `gateway.nodes.invokeTimeoutMs` is absent from the strict schema, and in the PR patch omitted per-call `timeoutMs` still sends `undefined` to node hosts before applying the new gateway fallback.

## Solution Assessment

No, not as written. The direction is narrow and maintainable, but the patch needs runtime schema support, validation, resolved-timeout propagation, docs/changelog, and tests before it is the best fix.

  - body: The PR only adds `invokeTimeoutMs` to the TypeScript config type. The runtime `gateway.nodes` zod object is strict and still lacks this key, so a user who follows the new example will get an unknown-property config error before startup can use it.
  - confidence: 0.94
- **[P2] Send the resolved timeout to nodes:** `src/gateway/node-registry.ts:142-143`
  - body: The new fallback is computed after `node.invoke.request` is sent, while the payload still uses `timeoutMs: params.timeoutMs`.   - confidence: 0.91

  - body: The startup guard accepts any number, so `0`, negative, `NaN`, or infinite values can be copied into the registry and make invocations time out immediately or behave inconsistently. The config should be finite, positive, integral, and bounded in the schema and startup wiring.
  - confidence: 0.88

  - body: This adds a user-facing gateway config option, but the PR does not update `CHANGELOG.md`. OpenClaw policy expects user-facing feature/config changes to be listed under the active release notes.
