## Summary

Keep this PR open because current main still lacks the requested run-scoped `chat.history` capability, but the branch is not merge-ready: the matcher still has valid-ID prefix collisions, the run filter is still downstream of the recent transcript tail read, generated protocol artifacts are missing, and the patch no longer applies cleanly to current main.

## What This Changes

The PR adds optional `runId` support to gateway `chat.history`, filters transcript messages through a new helper before history projection/budgeting, and adds focused Vitest coverage.

## Best Possible Solution

Land the additive read-only `chat.history({ runId })` API after implementing exact run-marker matching for the full `NonEmptyString` contract, making runId lookup happen before recent-response tail limiting, regenerating protocol artifacts, adding the required changelog entry, and rebasing onto current main.

## Solution Assessment

No. The chosen Gateway surface is appropriate and earlier review fixes improved the direction, but the current patch is not the narrowest maintainable solution until it uses exact matching, scans at the right transcript layer, updates generated clients, and adapts to current main.

- **[P2] Match only complete run marker values:** `src/gateway/server-methods/chat.history-run-id-filter.ts:25-38`
  - body: `runId` is accepted as any `NonEmptyString`, but this boundary treats punctuation and symbols as separators. A valid query like `run` matches `run_id=run:2`, and `🚀` matches `run_id=🚀🎯`, so run-scoped history can include another run. Match explicit marker forms or structured fields exactly instead of relying on this partial token boundary.
  - confidence: 0.9
- **[P2] Search before the recent-message tail cap:** `src/gateway/server-methods/chat.ts:1641`
  - body: This filters `rawMessages`, but those messages are still loaded from `readRecentSessionMessagesAsync` with `maxMessages: max` where `max` is the response limit. In a busy session, unrelated newer messages can keep the target run out of `rawMessages`, so `chat.history({ runId })` returns empty even though the transcript contains that run. The runId path needs a bounded run-aware scan before the response limit is applied.
  - confidence: 0.86

  - body: Adding `runId` changes the Gateway protocol schema, but this PR only changes the TypeBox source. `protocol:check` requires the generated JSON schema and Swift GatewayModels to be committed too; otherwise typed clients keep a stale `ChatHistoryParams` shape without `runId`.
