## Summary

Keep open. Current main still lacks a `runId` parameter for `chat.history`, and this PR is a useful implementation direction, but the head is not merge-ready: the matcher can still return prefix matches for valid run IDs containing punctuation or symbols, the public protocol generated artifacts are missing, and GitHub reports the branch as dirty against main. Earlier Greptile and Codex findings were mostly addressed by extracting the matcher and moving filtering before budget projection, so the remaining work is narrower.

## What This Changes

The PR adds an optional `runId` parameter to gateway `chat.history`, filters raw transcript messages through a new run-ID matcher before history projection/budget handling, and adds matcher-focused Vitest coverage.

## Best Possible Solution

Land the additive `chat.history({ runId })` capability after making run identity matching exact across the full `NonEmptyString` contract, preserving pre-budget filtering, regenerating the protocol artifacts, and rebasing onto current main. A good final implementation should prefer explicit marker forms or structured fields over a broad substring boundary rule.

## Reproduction Assessment

Yes. The remaining matcher bug is reproducible by evaluating the PR helper logic against `run_id=run:2` with query `run` and `run_id=🚀🎯` with query `🚀`; both currently match even though they are different run IDs. Current main cannot reproduce the feature behavior because it has no `runId` filter yet.

## Solution Assessment

No. The PR chooses the right gateway surface and the earlier fixes improved the sequencing and test structure, but it is not the best final patch until matcher semantics are exact, generated protocol clients are updated, and the dirty branch is reconciled with current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat every runId character as part of the token:** `src/gateway/server-methods/chat.history-run-id-filter.ts:25`
  - body: `runId` is accepted as any non-empty string, but this boundary only treats letters, numbers, marks, `_`, and `-` as token characters. A valid ID like `run:2` or `🚀🎯` can therefore be returned when the caller asks for the prefix `run` or `🚀`, breaking the run-scoping guarantee. Match exact marker forms or use a boundary rule that is valid for the whole schema contract.
  - confidence: 0.9
- **[P2] Regenerate the protocol artifacts with the schema change:** `src/gateway/protocol/schema/logs-chat.ts:36`
  - body: Adding `runId` to `ChatHistoryParamsSchema` changes the public Gateway protocol, but this PR does not update the generated JSON schema or Swift protocol models that `protocol:check` diffs. Typed macOS/iOS clients will keep a stale `ChatHistoryParams` shape until those generated artifacts are committed.
  - confidence: 0.84
