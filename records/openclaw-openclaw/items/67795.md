## Summary

Keep this PR open, but do not merge it as-is. The reported `OSTYPE=linux` failure is real on current `main`, yet the patch only moves openSUSE/SLES users past OS detection into later Linux installer branches that still have no `zypper` path for build tools, Node.js, or Git.

## What This Changes

This PR broadens `scripts/install.sh` Linux OS detection from `linux-gnu*` to `linux*` so Bash `OSTYPE=linux` is treated as Linux.

## Best Possible Solution

Land a complete installer fix that accepts Bash `OSTYPE=linux` for Linux while adding focused zypper/openSUSE coverage for build tools, Node.js, and Git, with shell regression tests, installer docs, and a user-facing changelog entry. If zypper support is not desired, keep OS detection scoped to Linux variants the installer can actually bootstrap.

## Reproduction Assessment

Yes. Current `main` can be reproduced by sourcing `scripts/install.sh` with `OPENCLAW_INSTALL_SH_NO_RUN=1` and `OSTYPE=linux`, which exits as unsupported; a zypper-only probe shows the PR would then fail later at package-manager detection.

## Solution Assessment

No. The pattern change is a useful first step, but it is not the best complete fix for the reported openSUSE install problem unless the downstream zypper package-manager branches are handled too.
