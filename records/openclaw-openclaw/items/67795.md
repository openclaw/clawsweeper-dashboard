## Summary

Keep this PR open, but do not merge it as-is. The one-line `OSTYPE` broadening fixes the first openSUSE/SLES failure, and the reporter plus prior #36975 context make the gap credible; however, current installer code still lacks zypper handling for Node, Git, and build-tool installation, so a clean openSUSE/SLES host can pass OS detection and then fail later.

## What This Changes

This PR changes `scripts/install.sh` so `detect_os_or_die` treats `$OSTYPE` values beginning with `linux` as Linux instead of only matching `linux-gnu`, while preserving the WSL check.

## Best Possible Solution

Land a complete installer fix that treats Bash `OSTYPE=linux` as Linux and adds focused zypper/openSUSE coverage for build tools, Node.js, and Git, with shell unit tests and docs/changelog updates for the expanded Linux support matrix.

## Reproduction Assessment

Yes. Current main exits through `Unsupported operating system` when `detect_os_or_die` is called with `OSTYPE=linux`, while the PR head detects Linux; the remaining zypper failure path is source-backed by the package-manager branches and by prior #36975 context.

## Solution Assessment

No. The pattern change is the right first step, but it is not the best complete fix for the reported openSUSE install problem unless the downstream zypper package-manager paths are handled too.
