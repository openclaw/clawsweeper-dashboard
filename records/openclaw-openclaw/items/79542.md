## Summary

Keep this PR open: the diagnostic command is useful, but the current patch reports a Copilot state path for every ACP session even though ACP supports many non-Copilot harnesses, and the contributor still needs after-fix real behavior proof from a patched setup.

## What This Changes

The PR adds an `openclaw sessions link` CLI command with tests, ACPx docs, and a changelog entry to print ACP session keys, acpx session IDs, Copilot state paths, and status output.

## Best Possible Solution

Clarify the command scope, then either filter to Copilot ACP sessions or make state-path/status resolution backend-aware for all ACP harnesses, with regression coverage for a non-Copilot ACP row and redacted after-fix runtime proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows the PR scans every `entry.acp` session while hardcoding `.copilot/session-state`, and current docs/tests show ACP sessions for non-Copilot harnesses. The original feature need has before-only real-environment evidence, not after-fix proof.

## Solution Assessment

No. The current approach is not the best maintainable shape until it either becomes explicitly Copilot-scoped or resolves state diagnostics through ACP backend/harness-aware contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope state-dir checks to Copilot sessions:** `src/commands/acp-sessions-link.ts:64`
