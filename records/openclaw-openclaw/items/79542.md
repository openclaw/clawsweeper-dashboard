## Summary

Keep open: the diagnostic command is plausible, but the patch currently treats every ACP session as if it has a Copilot state directory, which will misreport non-Copilot ACP sessions. The contributor also provided useful before-fix live context, but not convincing after-fix proof from the PR branch.

## What This Changes

Adds an `openclaw sessions link` CLI command, docs/changelog entry, and tests to print ACP session keys, `acpxSessionId` values, Copilot state paths, and status flags.

## Best Possible Solution

Make the command semantics explicit: either scope it to Copilot ACP sessions or make the output agent/backend-aware, then add non-Copilot coverage and real after-fix terminal proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create any non-Copilot ACP session with an `acpxSessionId` and the new loop will still print a `~/.copilot/session-state/<id>` path for it.

## Solution Assessment

No. The narrow maintainable fix is to either filter to `entry.acp.agent === "copilot"` for a Copilot helper or introduce backend/agent-specific path resolution instead of applying a Copilot path to every ACP session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope link rows to Copilot sessions:** `src/commands/acp-sessions-link.ts:56-64`
  - body: `openclaw sessions link` defaults to all ACP session stores, but this loop accepts every ACP entry and then constructs a `~/.copilot/session-state/<id>` path. Current ACPX docs/config support many agents, so Codex/Claude/Gemini/etc. sessions will be reported as Copilot paths and usually `MISSING_STATE_DIR`, giving operators false diagnostics. Filter this helper to Copilot sessions or make the state-path resolver backend/agent-aware.
