## Summary

Keep open. Current main still has the reported cron isolated-agent gap: the cron fallback caller does not pass the embedded-runner result classifier, and the delivery/state path can still persist an error run with delivered content state when the warning payload is successfully sent.

## What This Changes

The issue requests that isolated cron fallback failures retry or alert instead of delivering the incomplete-turn warning as normal output, and that error runs not be persisted as delivered successful content.

## Best Possible Solution

Cron should reuse the embedded-runner fallback classifier when invoking `runWithModelFallback`, continue to the next eligible fallback or fail with failure-notification semantics when all candidates return incomplete terminal output, and persist failed output separately from successful cron content delivery.

## Reproduction Assessment

Yes. A deterministic test can mock an isolated embedded run that returns only the incomplete-turn warning as an error payload, let direct announce delivery succeed, and assert that current main can return or persist `status: "error"` with delivered state.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing classifier into the cron fallback call and normalize failure-output delivery persistence; a regex-only `lastDelivered` patch would leave fallback acceptance and monitoring semantics split across layers.
