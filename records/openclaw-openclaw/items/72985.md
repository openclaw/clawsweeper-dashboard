## Summary

Keep open. Current main still lets cron call `runWithModelFallback` without the embedded-runner result classifier, and cron state persists `delivered=true` independently of `status=error`, so the reported warning-delivery and monitoring contradiction remains reachable.

## What This Changes

The issue asks cron isolated-agent runs to treat incomplete-turn fallback text as a failed fallback result and to avoid persisting error runs as successfully delivered user content.

## Best Possible Solution

Cron isolated runs should reuse the embedded-runner fallback classifier and persist fatal-output notification separately from successful cron content delivery.

## Reproduction Assessment

Yes. A deterministic test can mock a cron isolated embedded result containing only the incomplete-turn warning as an error payload, let announce delivery succeed, and assert current main can still produce `status: "error"` with delivered state.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing classifier into the cron fallback call and normalize fatal-output delivery semantics; a last-mile warning-text regex alone would leave fallback behavior split across layers.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72985. Cron isolated-agent execution currently calls `runWithModelFallback` without the embedded Pi result classifier, so an incomplete-turn warning can be accepted from a fallback provider and then delivered/persisted as `status: error` plus `lastDelivered: true`. Wire `classifyEmbeddedPiRunResultForModelFallback` into the cron fallback path through the existing lazy/runtime seam, and adjust cron fatal-output delivery persistence so sending an error/failure notification is not recorded as successful cron content delivery. Keep the fix generic: no provider-specific special cases, no broad retry-policy rewrite, and no regex-only patch at the service persistence layer when the existing classifier can decide the fallback outcome earlier. Add focused regression tests for the cron fallback call and the persisted `status=error`/delivery-state behavior. Add a user-facing `CHANGELOG.md` fix entry. Validate with targeted cron/agent fallback tests and changed-gate in Testbox before handoff.
