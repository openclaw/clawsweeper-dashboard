## Summary

Keep open. Current main still lets cron isolated-agent model fallback accept an embedded incomplete-turn error payload as the fallback result, then can persist a terminal cron error with successful delivery state because delivery success is tracked independently from run success.

## What This Changes

The issue asks cron isolated-agent fallback failures to avoid silently delivering incomplete-turn warning text and to prevent error runs from persisting `lastDelivered=true`.

## Best Possible Solution

Reuse the embedded-runner fallback classifier in cron isolated model fallback, then persist fatal-output notification separately from successful cron content delivery.

## Reproduction Assessment

Yes. A focused test can mock an embedded cron fallback result containing only the incomplete-turn warning as an error payload, let announce delivery succeed, and assert current main returns/persists `status: "error"` with delivered state.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing classifier at the cron fallback boundary and normalize fatal delivery semantics; a warning-text regex only in the delivery layer would leave fallback behavior split across layers.

Current main's cron isolated executor calls `runWithModelFallback` without `classifyEmbeddedPiRunResultForModelFallback`, so an embedded fallback provider can return only the incomplete-turn warning as an error payload and still be accepted as the terminal fallback result.
