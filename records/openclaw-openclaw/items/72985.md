## Summary

Keep open. Current main still has the reported isolated-cron gap: cron does not pass the embedded-runner fallback classifier into the model fallback runner, and the error-output delivery path can still produce `status: "error"` while forwarding and persisting `delivered: true`.

## What This Changes

The issue reports that isolated cron fallback can deliver the embedded-runner incomplete-response warning as normal cron output and persist failed cron rows with delivered state.

## Best Possible Solution

Cron should reuse the embedded-runner fallback classifier when invoking `runWithModelFallback`, continue to the next eligible fallback or fail with failure-notification semantics when all candidates return incomplete terminal output, and persist failed output separately from successful cron content delivery.

## Reproduction Assessment

Yes. A deterministic test can mock an isolated embedded run that returns only the incomplete-turn warning as an error payload, let announce delivery succeed, and assert that current main can return or persist `status: "error"` with delivered state.

## Solution Assessment

Yes. The narrow maintainable direction is to wire the existing classifier into the cron fallback call and normalize failure-output delivery persistence; a regex-only `lastDelivered` patch would leave fallback acceptance and monitoring semantics split across layers.
