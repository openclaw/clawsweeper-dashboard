## Summary

Keep open. Current main still accepts cron isolated embedded-runner incomplete-turn error payloads as fallback results, can direct-deliver that fatal text, and then persists the channel send as `lastDelivered=true` even while the run status is `error`.

## What This Changes

The issue asks cron isolated-agent fallback failures to stop treating incomplete-turn warning text as normal delivered cron output and to persist unambiguous delivery state.

## Best Possible Solution

Cron isolated runs should reuse the embedded-runner result classifier before accepting fallback candidates, and persisted cron state should distinguish failure-notification delivery from successful cron content delivery.

## Reproduction Assessment

Yes. A focused test can mock `runWithModelFallback` returning an embedded result whose only payload is the incomplete-turn warning, let direct delivery succeed, and assert current main can return/persist `status: "error"` with `delivered: true`.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing classifier into the cron fallback call and normalize fatal-output delivery semantics; a late regex only in persistence would leave fallback behavior split across layers.
