## Summary

Keep open. Current main still matches the reported cron isolated-agent gap: the cron fallback call does not pass the embedded-runner result classifier into `runWithModelFallback`, and the delivery/state path can still carry `delivered: true` into an `error` run when outbound delivery of an error payload succeeds.

## What This Changes

The issue asks for cron isolated runs to stop treating embedded-runner incomplete-turn fallback text as successful cron output delivery and to make fallback failure/delivery state unambiguous.

## Best Possible Solution

Cron isolated runs should reuse the existing embedded-runner fallback classifier when invoking model fallback, continue to another eligible candidate when a candidate returns only incomplete terminal output, and persist fatal-output delivery separately from successful cron content delivery.

## Reproduction Assessment

Yes. A deterministic test can mock an isolated embedded run that returns only the incomplete-turn warning as an error payload, let announce delivery succeed, and assert that current main can return and persist `status: "error"` with delivered state.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing classifier into the cron fallback call and normalize fatal-output delivery persistence; a late regex-only `lastDelivered` patch would leave fallback acceptance and monitoring semantics split across layers.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72985. Current main lets cron isolated-agent fallback accept embedded-runner incomplete-turn output because `createCronPromptExecutor` calls `runWithModelFallback` without `classifyResult`, and fatal error payload delivery can still persist `lastDelivered: true`. Wire `classifyEmbeddedPiRunResultForModelFallback` into the cron `runWithModelFallback` call for embedded results so incomplete terminal output rejects that candidate and allows remaining fallback candidates to run. Normalize cron fatal-output delivery persistence so failed cron content is not recorded as successful user content delivery; if an error notification is delivered, keep that semantically separate from `lastDelivered` for useful cron output. Add focused tests around cron fallback classification and `status:error` persistence with successful announce delivery. Update `CHANGELOG.md` with a user-facing Fixes entry crediting `Thanks @Allenbluff`. Do not add provider-specific defaults, do not change unrelated fallback policy, and do not implement only a warning-text regex at the state layer while leaving fallback acceptance unchanged.
