## Summary

Keep open. Current main still lets cron isolated-agent model fallback accept the incomplete-turn warning as a successful fallback result, then can persist a fatal cron output as delivered when channel delivery succeeds.

## What This Changes

The issue asks cron isolated-agent fallback failures to stop being silently delivered as successful cron output and to avoid persisting error runs with delivered=true.

## Best Possible Solution

Cron isolated runs should share the existing embedded-runner fallback classifier and persist fatal-output notification separately from successful cron content delivery.

## Reproduction Assessment

Yes. A deterministic test can mock a cron isolated embedded run that returns only the incomplete-turn error payload, let outbound announce delivery succeed, and assert current main can produce status=error with delivered=true.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing classifier into the cron fallback call and normalize fatal-output delivery semantics; a last-minute warning-text regex in state persistence would leave fallback behavior split across layers.

Fix https://github.com/openclaw/openclaw/issues/72985. Wire cron isolated-agent execution through the existing embedded-runner fallback classifier so incomplete-turn error payloads are rejected by runWithModelFallback before cron accepts a fallback provider result. Then normalize cron delivery/state so fatal error output is not counted as successful cron content delivery; if an error notification is sent, track it separately or leave delivered=false while preserving useful diagnostics. Do not solve this with only a late jobs-state warning-text regex, and do not change unrelated provider fallback policy. Add focused coverage for the cron fallback classifier call and for an error run whose outbound delivery succeeds but must not persist lastDelivered=true.
