## Summary

Keep open. Current main still accepts isolated cron embedded-runner incomplete-turn outputs without passing the existing result classifier into the model fallback runner, and the fatal-output delivery path can still return/persist `status: "error"` together with `delivered: true` when the warning text is successfully sent.

## What This Changes

The issue requests that isolated-agent cron fallback failures stop treating the embedded-runner incomplete-turn warning as delivered user content, and expose or alert failed fallback semantics clearly enough for `jobs-state.json` monitoring.

## Best Possible Solution

Cron should reuse the embedded-runner fallback classifier when invoking `runWithModelFallback`, and failed cron output should be persisted separately from successful user-content delivery. If all fallback candidates produce incomplete/error payloads, the run should remain an error, avoid marking the warning as delivered content, and use the existing failure-notification path where configured.

## Reproduction Assessment

Yes. A deterministic test can mock an isolated embedded run returning only the incomplete-turn warning as an error payload, allow announce delivery to succeed, and observe the current path returning an error run whose delivered flag is persisted as delivered.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the existing embedded-result classifier into the cron fallback call and adjust fatal-output delivery persistence; a regex-only `lastDelivered` patch in job-state persistence would leave fallback acceptance and monitoring semantics split across layers.
