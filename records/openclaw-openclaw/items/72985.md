## Summary

Keep open. Current main still lets cron isolated-agent fallback accept embedded incomplete-turn error payloads without the existing result classifier, then preserves successful channel delivery separately from the terminal cron error, so the reported `status=error` plus `delivered=true` state remains possible.

## What This Changes

The issue requests that cron isolated-agent provider fallback failures stop silently delivering incomplete-turn warning text as normal output and stop persisting failed runs as delivered cron content.

## Best Possible Solution

Wire the embedded-runner fallback classifier into cron isolated model fallback, then normalize fatal cron content delivery so error fallback text is recorded as failed cron output delivery rather than successful delivered content.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: mock `runWithModelFallback` to return an embedded `isError` payload containing the incomplete-turn warning, let direct cron delivery succeed, and observe `status: error` with delivered state preserved. I did not run the test because this review is read-only and `pnpm` is unavailable in this shell.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing embedded-runner result classifier at the cron fallback boundary and make fatal cron output delivery semantics explicit; a warning-text-only delivery regex would miss empty and reasoning-only fallback failures.
