## Summary

Keep open. Current main still lets cron isolated-agent fallback accept embedded incomplete-turn/error payloads without the embedded-runner result classifier, and fatal error payloads can still be delivered and persisted independently from the terminal cron error status.

## What This Changes

Cron isolated-agent fallback should treat incomplete-turn fallback output as a failed model attempt and should not persist error runs as successful content delivery.

## Best Possible Solution

Reuse the embedded-runner result classifier at the cron model-fallback boundary and normalize fatal payload delivery semantics so error runs persist as not delivered, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence focused reproduction: mock runWithModelFallback to return an embedded isError incomplete-turn payload, make direct delivery succeed, and observe status error with delivered=true on current main.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing embedded-runner classifier into cron fallback and make fatal cron output count as failed content delivery; a warning-text regex only in delivery would miss empty and reasoning-only fallback failures.

In cron isolated-agent runs, pass classifyEmbeddedPiRunResultForModelFallback into runWithModelFallback so embedded empty/reasoning-only/incomplete-turn results are treated as failed fallback candidates instead of accepted terminal output.
