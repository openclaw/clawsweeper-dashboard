## What This Changes

The branch adds memory-core startup cleanup for aged reindex temp SQLite triplets, writes a PID lock around safe reindexes, and expands manager-db probe tests for stale, young, live-owned, and stale-lock temp files.

## Best Possible Solution

Land one focused memory-core startup cleanup with an active-reindex race guard; this PR is the stronger current candidate because it adds PID-lock protection and broader regression coverage.

## Reproduction Assessment

Yes. Current main creates atomic reindex temp DBs and only removes them through in-process success/error cleanup, so a hard kill can leave orphaned triplets; the PR body also shows an isolated after-fix CLI proof for a seeded stale triplet.

## Solution Assessment

Yes. The cleanup belongs in memory-core before the live DB opens, and the PID-lock guard is a better fit than an age-only startup sweep because it protects long-running active reindexes without adding config.
