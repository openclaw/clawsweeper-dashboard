## Summary

Keep open. Current main still protects Discord slash-command execution after a user selects a command, but it does not implement per-channel command-picker visibility for same-guild multi-bot deployments; the new user comment confirms the remaining problem is Discord UI visibility, not OpenClaw routing or execution authorization.

## What This Changes

Request per-channel Discord slash-command visibility so multi-bot same-guild deployments show only the commands for the bot configured for the current channel.

## Best Possible Solution

Keep this as a Discord plugin feature request until maintainers choose the supported end state: implement safe account/guild/channel application-command visibility reconciliation with explicit operator setup, or document the supported workaround path such as per-account native command disablement, text commands, or server isolation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure multiple Discord accounts in one guild with separate allowed channels and native commands enabled; current main deploys each account's full command catalog and rejects only after interaction submission. The May 1 user comment also confirms the same behavior in a live setup.

## Solution Assessment

Unclear. The requested UX is valid, but the best implementation needs maintainer judgment around Discord command visibility semantics, permission reconciliation, OAuth/operator setup, and stale command cleanup rather than a narrow local filter.
