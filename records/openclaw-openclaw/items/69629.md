## Summary

Keep open. Current main still registers each Discord account's native command set without deriving picker visibility from per-channel bindings, while authorization is enforced only after Discord submits the interaction; the remaining work is a product/design choice for Discord command permission or registration policy.

## What This Changes

Request per-channel Discord slash-command visibility so multiple bot accounts in one guild only appear in their assigned channels.

## Best Possible Solution

Keep this as an open Discord plugin feature until maintainers choose and document a supported per-account, per-channel native command visibility model with targeted tests and live Discord proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure multiple Discord accounts in one guild with separate channel bindings and native commands enabled; current main registers the account command set and only rejects disallowed channels after the slash interaction is submitted. I did not live-run Discord, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear. The requested UX is valid, but the best solution needs maintainer choice between Discord permission reconciliation, guild/channel-scoped registration, per-account command inclusion, text-only controls, or documented platform limits.
