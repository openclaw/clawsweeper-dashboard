## Summary

Keep open: current `main` still uses generic visible-answer retry text for reasoning-only and empty-response continuations, and the production confirmation specifically says that retry path can answer an older user intent despite Telegram ordering fixes.

## What This Changes

Ensure tool/reasoning-only retry continuations answer the latest inbound user message rather than a stale nearby intent.

## Best Possible Solution

Add a generic embedded-runner latest-inbound-turn anchor to reasoning-only and empty-response retry construction, with regression coverage for stale-intent retry continuation.

## Reproduction Assessment

No high-confidence deterministic live replay was established here. The current-main source path and production confirmation make the retry-boundary failure source-reproducible, but not yet a reproduced local or live Telegram test.

## Solution Assessment

Yes for the narrow bug: the maintainable fix is in the embedded-runner retry prompt/state boundary, not a Telegram-only dispatch change or the broader finalize-time stale-reply feature requested in https://github.com/openclaw/openclaw/issues/79614.
