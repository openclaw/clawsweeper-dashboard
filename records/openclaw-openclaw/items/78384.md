## Summary

Keep open: current main still builds reasoning-only and empty-response retries with generic visible-answer continuation text and no latest-inbound-turn anchor. The later Telegram supersession and foreground freshness fixes help stale delivery ordering, but they do not cover the semantic retry path confirmed in production comments for this issue.

## What This Changes

The issue requests a runtime guard so tool continuations, reasoning-only retries, and empty-response retries answer the latest inbound user message instead of an older nearby intent.

## Best Possible Solution

Add a generic embedded-runner latest-inbound-turn anchor to reasoning-only and empty-response retry construction, with regression coverage that prevents a retry continuation from answering an older nearby user turn.

## Reproduction Assessment

No high-confidence deterministic live replay is available from this review. The source path is clear on current main and production confirmation narrows the symptom to the embedded-runner retry continuation, but the exact Telegram stale-intent final answer was not reproduced here.

## Solution Assessment

Yes, the narrow maintainable direction is a core embedded-runner latest-turn anchor with focused regression coverage. A Telegram-only dispatch change is not sufficient because the reported mismatch happens inside the retry continuation semantics.
