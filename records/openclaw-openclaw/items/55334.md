## Summary

Keep this open. Current main has real session maintenance mitigations, but the central bloat path remains: retained session rows can still carry `skillsSnapshot.prompt` and `systemPromptReport` inline, and saves still stringify the whole session store; related PRs #48425 and #47635 are useful but not merged into current main.

## What This Changes

The issue asks OpenClaw to prevent `sessions.json` from growing into an OOM/lock-latency source by pruning ephemeral sessions, avoiding duplicated skill/system-prompt payloads, and cleaning orphan artifacts.

## Best Possible Solution

Keep this as the canonical session-store bloat tracker and land a backward-compatible persistence design that keeps `sessions.json` lightweight, compacts legacy rows, and coordinates the narrower skills-snapshot and archive-cleanup PRs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: retain many session entries with `skillsSnapshot.prompt` and `systemPromptReport`, then trigger session updates or list/startup paths that read and serialize the store.

## Solution Assessment

No. Current main's pruning, disk-budget, and `resolvedSkills` stripping are useful mitigations, but they do not remove the static prompt/report duplication from retained session rows; a schema-compatible externalization or recomputation path is the narrower durable fix.
