## Summary

Keep this open. Current main and v2026.5.19 have useful pruning, capping, and artifact-cleanup mitigations, but they still persist `skillsSnapshot.prompt` and `systemPromptReport` inline and still parse/clone the full session store; focused fixes are open but not merged.

## What This Changes

The issue reports gateway OOM/session-store lock latency from `sessions.json` growth due to duplicated `skillsSnapshot`/`systemPromptReport`, retained cron/subagent rows, orphan artifacts, and eager whole-store reads.

## Best Possible Solution

Keep this as the broad session-store bloat/OOM tracker, review or land https://github.com/openclaw/openclaw/pull/78259 and https://github.com/openclaw/openclaw/pull/84693 as appropriate, then close only after shipped proof covers prompt/report storage, clone/read memory, orphan artifact cleanup, and upgrade-safe session loading.

## Reproduction Assessment

Yes at source level: current main still persists `skillsSnapshot.prompt` and `systemPromptReport` and still parses/clones the full session store. I did not run a long-uptime RSS/OOM scenario, so the exact heap-growth path remains live-proof pending.

## Solution Assessment

No complete fix is on main yet. The narrowest maintainable path is to sequence the focused prompt-stripping and session-store memory PRs, then decide the backward-compatible schema/read-path cleanup needed for the remaining OOM modes.
