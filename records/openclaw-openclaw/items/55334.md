## Summary

Keep this issue open. Current main and v2026.5.18 include real mitigations for session-store growth, but they still persist `skillsSnapshot.prompt` and `systemPromptReport` inline per retained session row; the active prompt-stripping PR is still open, and the broader schema/artifact cleanup decisions are not finished.

## What This Changes

The issue reports gateway OOM and session-store lock latency from duplicated per-session `skillsSnapshot`/`systemPromptReport` payloads plus incomplete pruning of ephemeral sessions and orphan transcripts.

## Best Possible Solution

Keep this as the high-impact session-store bloat tracker, land or replace the focused prompt-stripping work, then decide the compatible storage/compaction path for prompt reports, snapshot references, and orphan transcript artifacts with upgrade tests.

## Reproduction Assessment

Yes, for the storage-bloat path at source level: current main still persists `skillsSnapshot.prompt` and `systemPromptReport` per retained row and serializes/parses the whole store. I did not run a live gateway heap/OOM workload in this read-only review.

## Solution Assessment

Unclear as a final implementation: stripping or referencing large static prompt payloads is the right direction, but the active PR and any `systemPromptReport`/artifact cleanup need maintainer review for resume semantics and upgrade compatibility.
