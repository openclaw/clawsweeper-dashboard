## Label Justifications

- `P1`: Multiple users report gateway OOMs, lock stalls, timeouts, and broken cron/channel workflows during normal session churn.
- `impact:session-state`: The affected surface is persisted session metadata, skill snapshots, prompt reports, cron/subagent rows, and transcript/artifact maintenance.
- `impact:crash-loop`: The reported failure mode includes gateway OOM kills, unresponsiveness, restart-dependent recovery, and repeated service crashes.

## Summary

Keep this open. Current main has useful pruning and disk-budget mitigations, but it still persists `skillsSnapshot.prompt` and `systemPromptReport` inline in session rows, and the focused prompt-stripping work at https://github.com/openclaw/openclaw/pull/78259 is still open rather than merged.

## What This Changes

The issue asks OpenClaw to bound `sessions.json` growth by pruning ephemeral cron/subagent sessions, deduplicating or externalizing skill snapshots, cleaning orphan transcripts, and reducing whole-store memory costs.

## Best Possible Solution

Keep this as the canonical session-store bloat tracker, land a correct prompt-stripping fix, then design backward-compatible static snapshot/report externalization and safe orphan-artifact cleanup with heap proof for the remaining RSS growth.

## Reproduction Assessment

Yes at source level for the retained-row bloat: current main still persists `skillsSnapshot.prompt` and `systemPromptReport` while only stripping `resolvedSkills`. I did not run a long-uptime heap reproduction, so the exact 50-100 MB/min RSS curve remains unproven here.

## Solution Assessment

No complete fix is on main yet. The current retention, capping, disk-budget, and `resolvedSkills` stripping work are useful mitigations, but the remaining solution needs maintainer-approved persisted schema, resume semantics, cleanup defaults, and heap-proof sequencing.
