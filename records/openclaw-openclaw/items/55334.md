## Summary

Keep open: current main and v2026.6.10 have important mitigations for duplicated `skillsSnapshot.prompt` data, but the broader gateway OOM/session-store pressure is not fully solved and remains split across prompt-report persistence, whole-store hydration/cache retention, transcript/artifact cleanup, and maintainer retention-policy choices.

## What This Changes

The issue requests bounded session-store growth by pruning ephemeral sessions, deduplicating large skills/system-prompt metadata, cleaning orphan transcripts, and avoiding full-store memory pressure.

## Best Possible Solution

Keep this broad tracker open until maintainers land or reject the focused cache, prompt-report, transcript, and artifact-cleanup paths, then close only with current-main or release proof that representative long-running session stores and gateway RSS stay bounded.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: current source still shows whole-store hydration/cache retention and durable `systemPromptReport` writes, and the discussion has repeated production reports, but I did not run a live current-main OOM soak.

## Solution Assessment

Is this the best way to solve the issue? Not yet as a single fix: prompt-blob externalization is a good partial mitigation, but the final solution needs maintainer-approved cache caps, prompt-report compaction, artifact cleanup, and retention semantics.
