## Label Justifications

- `P1`: The issue has repeated production reports of gateway OOM, lock stalls, and unresponsive agent workflows under normal cron/subagent/session churn.
- `impact:session-state`: The affected surface is the persisted session registry, per-session metadata, and retained transcript/session state.
- `impact:crash-loop`: The reported user impact includes gateway OOM kills, unresponsiveness, and restart-dependent recovery.

## Summary

Keep this open. Current main and the latest release have useful mitigations for session-store growth, but retained session rows still persist large static `skillsSnapshot.prompt` and `systemPromptReport` payloads inline, so the central bloat/OOM report is not fully resolved.

## What This Changes

The issue asks OpenClaw to bound `sessions.json` growth by pruning ephemeral sessions and avoiding duplicated per-session skill snapshot and system prompt payloads.

## Best Possible Solution

Keep this as the canonical tracker and design a backward-compatible session metadata format that stores static skill snapshot and system prompt report payloads by hash/reference or recomputation, compacts legacy rows, and preserves existing session replay behavior while retaining current pruning/capping mitigations.

## Reproduction Assessment

Yes at source level for the retained-row bloat: current main still persists `skillsSnapshot.prompt` and `systemPromptReport` per session entry, and the tests intentionally preserve the snapshot prompt on disk. The exact reported 50-100 MB/min heap curve was not live-reproduced in this read-only review.

## Solution Assessment

No complete fix is present yet. The existing pruning/capping and `resolvedSkills` stripping are good mitigations, but the best remaining solution is a maintainer-approved schema/storage change rather than another narrow cleanup script.
