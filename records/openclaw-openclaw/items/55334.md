## Summary

Keep open: current main and v2026.6.10 mitigate the largest duplicated `skillsSnapshot.prompt` payload with content-addressed prompt blobs and enforce session-entry maintenance, but the broader reported gateway OOM/session-store pressure is not implemented away because full-store hydration/object-snapshot cache retention, durable `systemPromptReport`, null-by-default disk budgets, and transcript/artifact cleanup policy still leave meaningful work open.

## What This Changes

The issue reports unbounded `sessions.json` growth from repeated `skillsSnapshot` and `systemPromptReport` metadata, missing ephemeral-session/transcript cleanup, and gateway OOM or lock-latency under cron/subagent workloads.

## Best Possible Solution

Keep this broad tracker open until maintainers land or revise the focused cache-retention, prompt/report metadata, and artifact-cleanup paths, then close it only with current-main or release proof that session-store size and gateway RSS stay bounded for representative long-running workloads.

## Reproduction Assessment

Yes at source level, but not as a full live OOM reproduction in this read-only pass. Current main still has whole-store hydration/cache paths and durable `systemPromptReport`, while the issue discussion provides multiple field reports of OOM, lock latency, and artifact buildup.

## Solution Assessment

Unclear as a final fix: prompt-blob externalization is the right mitigation for the largest duplicated prompt payload, but cache-retention limits, cleanup defaults, and durable prompt-report policy still need maintainer-guided convergence and soak proof.
