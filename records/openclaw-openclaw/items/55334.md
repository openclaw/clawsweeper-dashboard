## Summary

Keep this open. Current main has important mitigations for session-store memory retention and cron runtime cleanup, but the persisted session schema still keeps `skillsSnapshot.prompt` and `systemPromptReport` inline and the remaining prompt/report storage fix is still open.

## What This Changes

The issue asks OpenClaw to prevent session-store OOM by pruning ephemeral sessions, avoiding per-row skill/system prompt duplication, cleaning orphan artifacts, and reducing full-store load memory.

## Best Possible Solution

Keep this as the broad tracker, land or replace the prompt-stripping/schema work, then close only after current-main proof covers retained-row size, full-store read memory, orphan artifact cleanup, and upgrade behavior.

## Reproduction Assessment

Yes for the retained-row bloat at source level: current main still persists inline `skillsSnapshot.prompt` and `systemPromptReport` and still has full-store parse/clone paths. I did not run a long-uptime heap/RSS scenario after the recent mitigations, so the exact current-main OOM slope still needs live proof.

## Solution Assessment

No complete fix is on main yet. The best path is to sequence the focused prompt/report storage change with the already-merged cache/runtime mitigations and require upgrade-safe heap or lock-latency proof before closing.
