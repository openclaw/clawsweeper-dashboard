## Summary

Keep open. Current main has meaningful mitigations for session-store accumulation, including enforced entry caps, load-time pruning, cron run-session pruning, and cache/rotation reductions, but the central retained-row bloat remains: session entries still persist full skills snapshots and system prompt reports inline. Fresh discussion also reports the same retained-row cost on v2026.4.26, so this is not implemented away yet.

## What This Changes

The issue reports gateway memory, lock-latency, and OOM problems from growing `sessions.json`, duplicated inline `skillsSnapshot` and `systemPromptReport` payloads, and lingering session artifacts.

## Best Possible Solution

Keep this as the broad tracker for session-store static-payload bloat. The desired end state is a backward-compatible persisted-session design that keeps pruning and disk-budget safeguards, stores large skills and prompt-report data out of line by hash/reference or recomputes it safely, compacts legacy rows, and proves many retained chat, cron, and subagent sessions no longer duplicate large static payloads while old sessions still load.
