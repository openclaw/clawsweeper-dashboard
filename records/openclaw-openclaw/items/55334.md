## Summary

Keep this issue open. Current main has real guardrails for count, age, disk budget, and cron run-session retention, but it still persists large static payloads inline and still uses whole-store read/stringify/write paths, so the central storage and lock-latency problem remains unresolved.

## What This Changes

The issue requests a fix for `sessions.json` growth by pruning ephemeral sessions, avoiding duplicated `skillsSnapshot` and `systemPromptReport` payloads, cleaning orphan artifacts, and reducing whole-store memory/write costs.

## Best Possible Solution

Keep this as the broad canonical session-store bloat tracker. The end state should be a backward-compatible storage design that keeps current pruning safeguards, removes or references large static session payloads, preserves warning and skill-refresh semantics, compacts legacy rows, and adds focused coverage for chat, cron, subagent, reset, cleanup, and large-store write paths.

## Reproduction Assessment

Yes. The issue has concrete production reproduction data, and current main gives a high-confidence source-level reproduction: create many recent session entries with large inline `skillsSnapshot` and `systemPromptReport` payloads, then time a session update or startup load.

## Solution Assessment

No, current main is not yet the best solution. The pruning and cleanup mitigations help, and PR #48425 is a useful partial direction, but the maintainable fix still needs a session-entry-aware persisted shape plus compatibility and write-path design.
