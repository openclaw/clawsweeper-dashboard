## Summary

Keep open: current main and v2026.6.8 have important mitigations for the original sessions.json bloat, but the broad OOM/session-store problem is not fully closed because oversized store cache retention, partial/opt-in artifact cleanup, and remaining per-row metadata still need maintainer follow-up.

## What This Changes

The issue asks OpenClaw to prevent unbounded sessions.json growth from duplicated skills snapshots, retained ephemeral cron/subagent sessions, and orphaned session artifacts that can drive gateway OOMs.

## Best Possible Solution

Keep this as the broad session-store bloat tracker until maintainers land or explicitly split the cache-retention, artifact-cleanup, and persisted metadata follow-ups, then close only after representative regression and soak proof show bounded session storage and gateway memory.

## Reproduction Assessment

Yes at source level, with multiple field reports. Current main still retains whole-store runtime caches without a byte cap and has only partial/opt-in artifact cleanup, although I did not run a fresh live OOM soak.

## Solution Assessment

No: closing or treating this as a pruning-only fix would be too narrow. The maintainable path is to keep the shipped prompt-blob mitigation, review the focused cache-cap PR, and settle artifact/session metadata cleanup policy with targeted tests and soak proof.
