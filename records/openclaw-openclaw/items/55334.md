## Summary

Keep this issue open. Current main has useful mitigations, including session maintenance caps and stripping `resolvedSkills`, but it still persists `skillsSnapshot.prompt` and `systemPromptReport` inline in `sessions.json`; the latest production data in the issue reports the same retained-row duplication after cleanup on 2026.5.7.

## What This Changes

The issue requests a bounded, metadata-light session store that prunes ephemeral sessions, avoids duplicated skill/system-prompt payloads, and cleans orphan session artifacts.

## Best Possible Solution

Keep this as the canonical session-store bloat tracker and land a backward-compatible metadata-light persistence design that strips, deduplicates, externalizes, or lazily rebuilds static skill/system-prompt payloads while preserving the existing pruning safeguards.

## Reproduction Assessment

Yes. Source inspection can reproduce the storage path by retaining many session entries with `skillsSnapshot.prompt` and `systemPromptReport`, then using the load/save paths that parse and stringify the full store; live heap proof was not run in this read-only review.

## Solution Assessment

Yes directionally: retention and cleanup are useful mitigations, but the durable fix is to stop storing repeated rendered prompt/report payloads per session row. The exact schema, migration, and resume semantics need maintainer design before implementation.
