## Summary

Keep this open. Current main has meaningful mitigations for session-store growth, but the central static-payload duplication remains: retained rows can still carry rendered `skillsSnapshot.prompt` and `systemPromptReport` inline, and saves still serialize the whole store.

## What This Changes

The issue asks OpenClaw to stop `sessions.json` growth and gateway OOM/lock latency by pruning ephemeral sessions, de-duplicating large per-session skill/system-prompt payloads, cleaning orphan transcripts, and reducing whole-store load/write pressure.

## Best Possible Solution

Keep this as the canonical session-store bloat tracker and land a backward-compatible persistence design that keeps `sessions.json` lightweight, externalizes or reconstructs large static skill/system-prompt payloads, compacts legacy rows, and preserves the current pruning, writer-queue, disk-budget, and cron-reaper safeguards.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: retain many session entries with `skillsSnapshot.prompt` and `systemPromptReport`, then trigger session updates or startup/list paths that read and stringify the store. I did not run a live heap profile because this review is read-only.

## Solution Assessment

No. Current main is a useful partial mitigation because it strips `resolvedSkills` and improves writer/cleanup behavior, but it still persists large rendered/static fields inline; PR #48425 is a promising partial direction that still needs maintainer review against the broader schema and resume semantics.
