## Summary

Keep this PR open: the reported data-loss path is source-reproducible on current main and the proposed catch split is the right narrow code direction, but the external PR still needs after-fix real behavior proof before merge.

## What This Changes

The PR changes memory-core's phase-signal store read helper to return an empty store only for missing or corrupt JSON and to rethrow other read errors.

## Best Possible Solution

Land this narrow memory-core fix or an equivalent replacement after the contributor provides real read-failure proof and the PR adds the required changelog and targeted regression coverage.

## Reproduction Assessment

Yes from source inspection: current main catches all `readPhaseSignalStore` errors and `recordDreamingPhaseSignals` writes the returned store, so a non-ENOENT read failure can become an empty phase-signal write. I did not run a live chmod/EACCES reproduction because this review was read-only.

## Solution Assessment

Yes, the catch split is the narrowest maintainable code fix because it preserves ENOENT and SyntaxError recovery while letting the existing workspace-level dreaming catch handle unexpected I/O errors. Merge readiness still needs real behavior proof, a changelog entry, and ideally a focused regression test.
