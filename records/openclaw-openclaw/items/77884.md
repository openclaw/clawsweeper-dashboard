## Summary

Keep open: the bug remains source-reproducible on current main and the one-file fix is directionally correct, but this external PR still lacks after-fix real behavior proof for an actual non-ENOENT phase-signal read failure.

## What This Changes

This PR changes memory-core's phase-signal reader to return an empty store for missing or corrupt JSON but rethrow other read errors so the existing dreaming workspace catch handles them.

## Best Possible Solution

Land this narrow catch split, updated to current `readJsonIfExists` code, with contributor real read-failure proof and a focused regression test showing unexpected read errors do not overwrite stored phase signals.

## Reproduction Assessment

Yes, from source inspection: a non-ENOENT read failure in `readPhaseSignalStore` currently becomes an empty store, and `recordDreamingPhaseSignals` can write that store back. I did not run a live chmod/EACCES scenario in this read-only review.

## Solution Assessment

Yes. Rethrowing non-ENOENT, non-SyntaxError read errors is the narrow maintainable fix because it preserves first-run/corrupt-file recovery while using the existing per-workspace dreaming failure handler.
