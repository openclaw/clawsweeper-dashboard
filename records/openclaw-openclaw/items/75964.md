## Summary

Keep open. Current main still exits recurring `wakeMode: "now"` main-session cron jobs after the first retryable busy heartbeat skip, records `ok`, and has regression coverage asserting that ghost-shaped result; #73189 only covers payload delivery once the heartbeat reaches the run path.

## What This Changes

The issue asks OpenClaw to stop recording fast successful recurring main-session cron runs when a retryable busy heartbeat skip prevents the queued cron event from being consumed.

## Best Possible Solution

Retry recurring wake-now main cron jobs on temporary `requests-in-flight` and `lanes-busy` skips within the existing bounded wait, preserve `cron-in-progress` deferral, and update regression coverage plus changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a recurring main-session `wakeMode: "now"` systemEvent job with mocked `requests-in-flight` records `ok` after one heartbeat attempt and a deferred wake.

## Solution Assessment

Yes, with a per-reason boundary. The narrow maintainable fix is to remove the recurring fast exit for temporary busy reasons while keeping the `cron-in-progress` fallback because direct wake-now cannot succeed while the cron job's own active marker is set.
