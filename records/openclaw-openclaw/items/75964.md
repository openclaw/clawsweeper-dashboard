## Summary

Keep open. Current main still has the reported recurring wake-now main-session cron path: a retryable busy heartbeat skip immediately schedules a follow-up wake and records `ok`, and existing coverage asserts the fast successful result.

## What This Changes

The issue reports that recurring main-session wake-now cron jobs can record fast successful runs after a retryable busy heartbeat skip without guaranteeing the queued cron event is processed.

## Best Possible Solution

Recurring wake-now main-session cron jobs should use the bounded retry loop for temporary `requests-in-flight` and `lanes-busy` skips, while preserving the `cron-in-progress` deferred wake and updating the pinned regression coverage plus changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists in current tests: a recurring main-session wake-now job with a mocked `requests-in-flight` heartbeat skip records fast `ok` after only one heartbeat attempt. I did not run tests because this review was explicitly read-only.

## Solution Assessment

Yes, with a per-reason refinement. The maintainable fix is not a broad cron rewrite; it is to remove the recurring fast exit for temporary busy reasons while keeping the bounded fallback and `cron-in-progress` special case.
