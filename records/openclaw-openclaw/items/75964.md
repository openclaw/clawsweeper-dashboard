## Summary

Keep open. Current main still immediately returns `ok` for recurring wake-now main-session cron jobs after the first retryable busy heartbeat skip, and existing regression coverage pins the short successful run instead of requiring the queued cron event to be processed.

## What This Changes

The issue asks OpenClaw to stop recording recurring main-session wake-now cron runs as successful when their first heartbeat attempt is skipped for temporary busy-lane reasons.

## Best Possible Solution

Retry recurring wake-now main cron jobs on temporary `requests-in-flight` and `lanes-busy` skips within the existing bounded wait, preserve the `cron-in-progress` deferral, and update the regression test plus changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a recurring `sessionTarget: "main"`, `wakeMode: "now"` cron with mocked `requests-in-flight` records `ok` in under 100 ms after only one heartbeat attempt.

## Solution Assessment

Yes for the narrow direction, with one important boundary: temporary busy skips should use the bounded retry loop, while `cron-in-progress` should continue to defer via `requestHeartbeatNow`.
