## Summary

Keep open. Current main still fast-exits recurring `wakeMode: "now"` main-session cron jobs on retryable busy heartbeat skips and records `ok`, with regression coverage asserting the fast successful result; #73189 only covers payload delivery after the heartbeat actually runs.

## What This Changes

Recurring main-session wake-now cron should not record a successful run before a retryable busy heartbeat skip has either been retried within the bounded wait or genuinely deferred.

## Best Possible Solution

Retry recurring wake-now main cron jobs on temporary `requests-in-flight` and `lanes-busy` skips within the existing bounded wait, preserve the `cron-in-progress` deferral, and update the pinned regression coverage plus changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a recurring main-session `wakeMode: "now"` cron with mocked `requests-in-flight` records `ok` in under 100 ms after only one heartbeat attempt.

## Solution Assessment

Unclear for the exact proposed one-line diff. The best fix is per-reason: remove the recurring fast exit for temporary busy reasons while preserving the `cron-in-progress` fallback and the bounded fire-and-forget cap.

## Repair Work Prompt

Fix the recurring main-session wake-now cron ghost-run reported in https://github.com/openclaw/openclaw/issues/75964. In `src/cron/service/timer.ts`, recurring `sessionTarget: "main"`, `wakeMode: "now"`, `payload.kind: "systemEvent"` jobs should not immediately return `ok` after a first retryable busy skip for temporary `requests-in-flight` or `lanes-busy`; they should share the existing bounded retry behavior used by one-shot wake-now jobs and only fall back to `requestHeartbeatNow` after the max wait. Preserve the fast deferred wake behavior for `HEARTBEAT_SKIP_CRON_IN_PROGRESS`, because that can be caused by the cron job's own active marker. Update the current regression test that asserts `lastDurationMs < 100`, add/adjust coverage for recurring temporary busy retry success and bounded fallback, preserve heartbeat `{ target: "last" }`, and add a single-line `CHANGELOG.md` fix entry under the active version. Do not implement the broader #63106 ghost-run warning in this PR and do not change next-heartbeat semantics except where required by shared helpers.
