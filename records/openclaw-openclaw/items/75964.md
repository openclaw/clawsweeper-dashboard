## Summary

Keep open. Current main still fast-exits recurring `wakeMode: "now"` main-session cron jobs on retryable busy heartbeat skips and records `ok`, with existing regression coverage asserting the short successful result; the linked #73189 fix covers cron payload delivery only after a heartbeat actually reaches the run path.

## What This Changes

Fix recurring `wakeMode: "now"` main-session cron jobs so retryable busy heartbeat skips do not become short `ok` ghost runs before the queued cron event is processed.

## Best Possible Solution

Retry recurring wake-now main cron jobs on temporary `requests-in-flight` and `lanes-busy` skips within the existing bounded wait, preserve the `cron-in-progress` deferral, and update regression coverage plus changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a recurring main-session `wakeMode: "now"` cron with mocked `requests-in-flight` records `ok` after one heartbeat attempt and a deferred wake.

## Solution Assessment

Yes, if scoped per skip reason. The narrow maintainable fix is to retry temporary busy skips within the cap while still using the queued wake fallback for `cron-in-progress` and timeout cases.

Fix openclaw/openclaw#75964. Current main records a short `ok` run for recurring `sessionTarget: "main"`, `payload.kind: "systemEvent"`, `wakeMode: "now"` cron jobs when the first `runHeartbeatOnce` returns retryable temporary busy (`requests-in-flight` or `lanes-busy`). Update the regression test that currently pins the short-ok behavior so it proves recurring jobs retry and only report ok after a heartbeat run, while keeping one-shot and `cron-in-progress` behavior covered. Add a single-line changelog fix entry. Do not attempt the broader #63106 ghost-run warning/detector work in this PR.
