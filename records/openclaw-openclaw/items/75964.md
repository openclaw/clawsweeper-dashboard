## Summary

Keep open. Current main still has the recurring `wakeMode: "now"` fast-exit path that records `ok` after the first retryable busy heartbeat skip, and the current regression test explicitly pins that short successful run rather than proving the cron event was processed.

## What This Changes

The report asks OpenClaw to stop recording recurring main-session `wakeMode: "now"` cron jobs as successful when the first heartbeat attempt only hit a retryable busy skip.

## Best Possible Solution

Handle temporary `requests-in-flight` and `lanes-busy` skips with a bounded retry or explicit deferred status, preserve the `cron-in-progress` queued-wake escape hatch, and update regression coverage plus the changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: a recurring main-session `wakeMode: "now"` cron with mocked `requests-in-flight` records `ok` quickly after one heartbeat attempt and a deferred wake request.

## Solution Assessment

Yes with guardrails. The narrow fix should remove the recurring-job fast exit for temporary busy reasons while preserving the `cron-in-progress` fallback and bounded wait so the #58872 drift fix is not lost.
