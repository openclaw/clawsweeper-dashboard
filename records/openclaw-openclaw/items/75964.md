## Summary

Keep open. Current main still has the reported recurring `wakeMode: "now"` main-session cron path: a retryable busy heartbeat skip immediately schedules a follow-up wake and records `ok`, and existing coverage asserts the fast `ok` result. The closed #73189 payload fix only covers the case where the heartbeat actually runs.

## What This Changes

Recurring main-session wake-now cron jobs should retry retryable busy heartbeat skips instead of recording a fast successful run after only scheduling a follow-up wake.

## Best Possible Solution

Main-session recurring wake-now cron should share the one-shot bounded retry behavior for temporary `requests-in-flight` and `lanes-busy` skips, while keeping the self-induced `cron-in-progress` case as a deferred wake.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists in current tests: a recurring main-session cron job with `wakeMode: "now"` and a mocked `requests-in-flight` heartbeat skip records `ok` in under 100 ms without a real heartbeat run.

## Solution Assessment

Unclear for the literal one-line diff. The safer fix is per-reason: remove the recurring fast exit for temporary busy reasons, but preserve the `cron-in-progress` fast exit and update the existing #58833 regression contract.
