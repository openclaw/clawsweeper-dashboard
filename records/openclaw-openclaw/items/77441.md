## Summary

Keep open. Current main and the latest release still prefer the normalized built-in channel alias over an installed plugin claim, so WeCom and Yuanbao can be auto-added to `plugins.allow` as non-plugin IDs and then reported as missing plugins.

## What This Changes

Configured WeCom and Yuanbao channels should auto-add their owning plugin IDs to `plugins.allow` instead of the channel aliases.

## Best Possible Solution

Update channel auto-enable resolution so installed plugin claims win over built-in channel aliases for external channel plugins, while preserving bundled-channel fallback and `preferOver` behavior.

## Reproduction Assessment

Yes, source-level: configured channel IDs flow into `collectPluginIdsForConfiguredChannel`, which currently returns the built-in alias when `wecom` or `yuanbao` also has a plugin manifest claim. I did not run the gateway live, but the current-main and release source path is direct.

## Solution Assessment

Yes. The narrow maintainable fix is in `collectPluginIdsForConfiguredChannel`, with regression tests for built-in channel IDs whose owning plugin ID differs from the channel ID; doctor cleanup alone would not stop restart-time regeneration.
