## Summary

Keep open: current main still lets a normalized built-in channel alias win over an installed plugin claim, and two open closing PRs already target the same narrow bug.

## What This Changes

Configured WeCom/Yuanbao channel aliases should auto-enable their owning plugin IDs in `plugins.allow` instead of the channel IDs.

## Best Possible Solution

Land one reviewed fix that makes manifest plugin claims win over built-in channel aliases, add focused regression coverage and a changelog entry, then close the duplicate PR.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: `channels.wecom` or `channels.yuanbao` flows through the collector, matching plugin claims exist, and the current fallthrough returns `builtInId` before the plugin manifest ID.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is in `collectPluginIdsForConfiguredChannel`, with regression tests for built-in channel IDs whose owning plugin ID differs from the channel ID.
