## Summary

Keep this PR open, but do not merge the global request-shape change as written: current main does put the prompt in `input`, yet OpenAI’s Responses contract permits system/developer input messages, and this shared adapter also serves custom compatible endpoints whose `instructions` support is not established. The linked issue and PR provide no live API rejection or incorrect-response proof, so the safest path is a provider-scoped, request-tested fix after reproducing the actual failure.

## What This Changes

The PR excludes the system prompt from the regular OpenAI Responses input array and sends it through the top-level `instructions` parameter instead.

## Best Possible Solution

Reproduce the concrete endpoint failure, then introduce a provider-owned compatibility decision that uses top-level `instructions` only for contracts that require it, with final serialized-request tests for OpenAI and representative custom-provider behavior.

## Reproduction Assessment

No. Current main’s JSON shape is easy to establish from source, but neither the issue nor PR demonstrates an API rejection, ignored prompt, or incorrect model output, and the official OpenAI contract permits the current message role.

## Solution Assessment

No. A global switch for all `openai-responses` endpoints is not the best fix without provider-specific failure proof; a compatibility-scoped request builder with serialized-request coverage is safer.
