## Summary

Keep open. Current main still allows undeclared stored auth-profile credentials to remain eligible for provider auth and then flow into Pi runtime auth, while the related closed issues cover configured-key precedence and Ollama marker fallback rather than removed `auth.profiles` declarations.

## What This Changes

The issue reports that Pi runtime can silently select credentials left in per-agent `auth-profiles.json` after their declaration is removed from `openclaw.json`.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, warn or block orphaned stored credentials in the Pi path, and add focused resolver/Pi auth coverage that preserves intentional configless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: omit the provider profile from `cfg.auth.profiles`, leave a compatible credential in `auth-profiles.json`, then `resolveAuthProfileOrder` can return that stored profile and Pi can inject it into runtime auth.

## Solution Assessment

No for destructive startup pruning as the default. The safer direction is a declaration-aware runtime guard or high-visibility warning, with an explicit compatibility rule for configless stored profiles and profile-id drift repair.
