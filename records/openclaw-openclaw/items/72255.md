## Summary

Keep open. Current main still permits provider credentials stored in per-agent `auth-profiles.json` to be selected when no matching `auth.profiles` declaration remains, and the Pi embedded path can inject that credential into runtime auth. The related stale-key and Ollama fallback fixes are useful context but do not supersede this removed-declaration billing route.

## What This Changes

The issue asks OpenClaw to stop silently using per-agent stored credentials after their `openclaw.json` auth profile declarations are removed, or at least warn and diagnose that drift.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime auth selection should ignore or loudly flag stored provider profiles whose ids are no longer declared for that provider, while `openclaw doctor` or `openclaw models status` reports stored-but-undeclared profiles with a non-destructive remediation path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure no declared `auth.profiles` entry for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile for the Pi auth controller to resolve and inject.

## Solution Assessment

No. The current silent reuse behavior is not the best solution; automatic pruning is also risky without policy, so a resolver allowlist or selection-time warning plus diagnostic drift reporting is the safer maintainable direction.
