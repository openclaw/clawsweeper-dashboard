## Summary

Keep open. Current main still permits a stored provider credential to be selected when there are no matching declared `auth.profiles` entries, and the Pi runtime can inject that resolved credential without a declaration-drift guard or high-visibility warning; the current Claude CLI auth-alias path narrows one part of the reported setup but does not satisfy the requested strict-or-loud behavior.

## What This Changes

The issue asks OpenClaw to stop silently using stored per-agent auth-profile credentials after their `openclaw.json` declarations are removed, especially through the embedded Pi fallback path.

## Best Possible Solution

Make runtime auth resolution declaration-aware and add diagnostics for undeclared stored credentials, with any pruning/quarantine handled through an explicit recovery-safe doctor or secrets flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the matching `auth.profiles` declaration while leaving a valid stored provider credential, then trace `resolveAuthProfileOrder` to Pi `profileCandidates`, `getApiKeyForModel`, and `setRuntimeApiKey`.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for removed declarations; the safer direction is an allowlist or warning/quarantine policy before considering destructive cleanup.
