## Summary

Keep open. Current main and v2026.6.8 still allow provider-compatible stored auth profiles to become runtime auth candidates when config declarations are absent or drifted, so the reported silent credential reuse remains a real auth-provider/security issue.

## What This Changes

The issue reports that removing an auth profile declaration from openclaw.json does not prevent a matching stored per-agent credential from being selected by the embedded/Pi auth path.

## Best Possible Solution

Define the authoritative contract for removed auth profile declarations, then update resolver eligibility plus doctor/status diagnostics with focused upgrade and security tests.

## Reproduction Assessment

Yes. Source inspection shows current main can still select provider-compatible stored profiles when no explicit auth order or compatible declared profile constrains the provider, although I did not run a live paid-provider reproduction.

## Solution Assessment

Unclear until maintainers choose the auth authority contract. A resolver eligibility rule plus doctor/status warning or migration is safer than only logging after a credential has already been selected.
