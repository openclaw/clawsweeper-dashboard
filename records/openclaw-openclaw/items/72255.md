## Summary

Keep open. Current main still lets a provider credential stored only in per-agent `auth-profiles.json` remain eligible after the matching `openclaw.json` declaration is gone, and the Pi runner can inject that resolved credential without an undeclared-profile guard or high-visibility warning.

## What This Changes

The issue asks OpenClaw to stop or warn when Pi uses credentials left in per-agent `auth-profiles.json` after their `openclaw.json` `auth.profiles` declaration is removed.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured profiles, add a non-destructive warning or diagnostic for undeclared stored credentials, and preserve legacy drift recovery only under an explicit compatibility rule.

## Reproduction Assessment

Yes, source-level. With the provider declaration removed from `auth.profiles` but a valid stored credential left in `auth-profiles.json`, current main can follow `resolveAuthProfileOrder` into `resolveApiKeyForProvider` and Pi `setRuntimeApiKey`.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for removed declarations; the safer narrow path is a declaration-aware guard or prominent warning before considering destructive cleanup.
