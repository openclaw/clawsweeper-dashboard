## Summary

Keep open. Current main and v2026.6.8 still let provider-compatible stored auth profiles become runtime candidates when matching config declarations are absent, and the remaining fix needs maintainer/security direction because it changes the authority contract between config metadata, stored credentials, runtime fallback, and doctor/audit repair.

## What This Changes

The issue asks OpenClaw to stop or loudly warn when removed auth-profile declarations leave per-agent stored credentials usable by the Pi/runtime auth path.

## Best Possible Solution

Choose a maintainer-approved credential authority model: either add a clear doctor/audit/list/remove path for undeclared stored credentials, or add a tested runtime allowlist/migration that prevents undeclared stored credentials from silently authenticating.

## Reproduction Assessment

Yes, at source level: current main uses stored provider-compatible profiles when config has no matching explicit profile/order, and model auth returns those profiles before later fallback sources. I did not run a live credential/billing repro in this read-only review.

## Solution Assessment

Unclear: a strict runtime allowlist would stop the leak, but current docs make config auth.profiles metadata/routing rather than the credential source of truth. The best fix needs maintainer/security choice around migration, warnings, and stored-profile-only upgrade behavior.
