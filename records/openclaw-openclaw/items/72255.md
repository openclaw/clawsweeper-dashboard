## Summary

Keep open. Current main still allows a stored per-agent auth-profile credential to be selected when no matching `auth.profiles` declaration remains, and the Pi embedded path can pass that selected credential into runtime auth without an undeclared-profile guard or warning. The related stale-key and Ollama fallback reports overlap the same auth-store drift theme, but they do not cover this removed-declaration billing path.

## What This Changes

The issue asks for Pi/runtime auth to stop or clearly warn when using per-agent `auth-profiles.json` credentials whose `auth.profiles` declarations were removed from `openclaw.json`.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime selection should ignore or prominently flag stored provider credentials whose profile ids are no longer declared for that provider, while `doctor` or `models status` exposes stored-but-undeclared drift and an explicit cleanup or quarantine command preserves recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no `auth.profiles` entry for a provider, keep a valid stored credential for that provider, then observe `resolveAuthProfileOrder` returning the stored profile and the Pi auth controller injecting the resolved key into runtime auth.

## Solution Assessment

No. The current silent reuse of stored credentials is not the best solution for a removed-declaration billing case; the safer maintainable direction is a declaration-aware guard or prominent drift warning plus diagnostics, not hidden runtime selection or automatic destructive pruning by default.
