## Summary

Keep open. Current main still allows a stored provider credential to remain eligible when the matching `auth.profiles` declaration is absent, and the Pi path can inject that resolved credential into runtime auth without a declaration-drift guard or warning.

## What This Changes

The issue asks OpenClaw to stop or loudly diagnose Pi/runtime use of stored auth-profile credentials after their `openclaw.json` declarations are removed.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured profiles, surface undeclared stored credentials in a high-visibility warning or doctor/secrets diagnostic, and preserve intentional legacy store-only auth only under an explicit compatibility rule.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remove the provider declaration from `auth.profiles`, leave a matching stored credential, then follow `resolveAuthProfileOrder` into `profileCandidates` and `setRuntimeApiKey`.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for a removed declaration; the narrow maintainable fix is a declaration-aware guard or prominent drift warning before any destructive cleanup.
