## Summary

Keep open. Current main still lets stored per-agent auth-profile credentials remain eligible when the matching openclaw.json declaration is gone, and the Pi embedded path can inject that selected credential without an undeclared-profile guard or warning.

## What This Changes

The issue requests that Pi/runtime auth stop silently selecting credentials left in per-agent auth-profiles.json after their openclaw.json declarations are removed.

## Best Possible Solution

Adopt declaration-aware credential authority with a non-destructive migration or diagnostic path so removed profile declarations cannot be silently used while intentional read-through and legacy drift recovery remain explicit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the provider’s auth.profiles entry, leave a usable stored credential for that provider, then follow resolveAuthProfileOrder into Pi profileCandidates and setRuntimeApiKey.

## Solution Assessment

Unclear. The strict allowlist direction is likely right for removed declarations, but the safest implementation needs secops review of fail-closed, warning, quarantine, and legacy compatibility semantics.
