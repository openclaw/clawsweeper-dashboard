## Summary

Keep open. Current main still lets a provider-compatible credential that exists only in per-agent `auth-profiles.json` enter the Pi auth path, and the two related closed issues fixed different configured-key and Ollama-marker paths rather than removed `auth.profiles` declarations.

## What This Changes

The issue asks OpenClaw to stop or loudly warn when Pi runtime uses credentials left in per-agent `auth-profiles.json` after their `openclaw.json` auth-profile declaration is removed.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, add a loud warning or diagnostic for undeclared stored credentials, and preserve configless/drift-repair compatibility under an explicit rule.

## Reproduction Assessment

Yes. Source inspection against current main gives a high-confidence path: when config no longer declares a compatible profile, auth order can fall back to stored provider profiles, and Pi then resolves and injects that selected credential.

## Solution Assessment

Unclear for the suggested destructive prune as the default fix. The safer maintainer direction is a declaration-aware guard or high-visibility warning plus an explicit compatibility rule for configless stored profiles and profile-id drift repair.
