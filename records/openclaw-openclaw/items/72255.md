## Summary

Keep open. Current main still lets a stored per-agent auth-profile credential remain eligible when its `auth.profiles` declaration is absent, and the Pi embedded runtime can inject that selected credential into runtime auth without a declaration-drift guard or warning.

## What This Changes

The issue requests that Pi runtime auth stop silently selecting credentials left in per-agent `auth-profiles.json` after their `auth.profiles` declarations are removed from `openclaw.json`.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, add a loud diagnostic or warning for undeclared stored credentials, and preserve configless/drift-repair compatibility only under an explicit documented rule.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: remove the compatible `auth.profiles` declaration while leaving a provider-matching stored credential, then Pi profile ordering can select that stored profile and inject its key. I did not run a live provider call because this was a read-only review and the path uses real credentials/billing.

## Solution Assessment

No for destructive startup pruning as the default fix; it risks breaking documented stored-profile and drift-repair behavior. The better solution is a declaration-aware guard or high-visibility warning with an explicit compatibility rule for configless profiles.
