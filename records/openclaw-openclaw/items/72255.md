## Summary

Keep open. Current main still allows stored per-agent auth profiles to be considered when no matching `openclaw.json` declaration remains, and the Pi path can inject the selected credential without a declaration-drift guard or warning.

## What This Changes

The issue requests that removed auth profile declarations stop leaving per-agent stored credentials silently eligible for Pi fallback runtime calls.

## Best Possible Solution

Adopt declaration-aware credential authority for runtime selection: ignore or prominently warn on stored credentials whose profile ids are no longer declared, with a non-destructive diagnostic or quarantine path for recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the provider’s `auth.profiles` declaration, keep a usable stored credential for that provider, follow `resolveAuthProfileOrder` into Pi `profileCandidates`, and observe `setRuntimeApiKey` receiving the resolved credential.

## Solution Assessment

No. The current stored-profile fallback is intentional for legacy compatibility, but it is not the best behavior for a removed declaration because it can silently route live calls to a credential the user believes retired.
