## Summary

Keep open. Current main still allows a credential that remains only in `auth-profiles.json` to be selected and injected into the Pi runtime when the matching `openclaw.json` declaration is gone, and the related closed reports do not supersede this removed-declaration path.

## What This Changes

The issue asks OpenClaw to stop or loudly warn when Pi/runtime auth uses stored per-agent credentials whose `auth.profiles` declaration was removed from `openclaw.json`.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, add a non-destructive warning or diagnostic for undeclared stored credentials, and preserve legacy/configless compatibility only under an explicit rule.

## Reproduction Assessment

Yes. Source inspection against current main shows that removing the `auth.profiles` declaration leaves `explicitProfiles` empty, so `resolveAuthProfileOrder` falls back to stored provider profiles; that result flows through `resolveApiKeyForProvider` into Pi `setRuntimeApiKey`.

## Solution Assessment

No. A destructive startup prune is risky because current tests preserve configless and drift-repair behavior; the safer solution is a declaration-aware guard or loud warning with explicit compatibility semantics.
