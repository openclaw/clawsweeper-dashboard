## Summary

Keep open. Current main still lets provider-compatible stored auth profiles remain selectable after their `openclaw.json` declaration is absent, and the Pi path can inject that selected credential without an orphan/declaration guard; because docs also describe stored profiles as part of runtime selection, the fix needs secops/product judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop, warn about, or diagnose Pi runtime use of credentials left in per-agent `auth-profiles.json` after their `openclaw.json` auth profile declaration is removed.

## Best Possible Solution

Define a declaration-aware runtime rule for configured providers, then block or loudly warn on undeclared stored-profile selection in the Pi path and add focused resolver/Pi plus diagnostic coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remove a compatible provider profile from `cfg.auth.profiles`, leave its credential in `auth-profiles.json`, and the resolver can still return it for Pi runtime auth injection.

## Solution Assessment

No for destructive startup pruning as the default. A safer fix is a declaration-aware selection guard or high-visibility warning plus diagnostics, after secops/product define how this interacts with documented stored-profile fallback and drift repair.
