## Summary

Keep open. Current main still permits a valid credential stored only in `auth-profiles.json` to be selected for a provider after the matching `openclaw.json` declaration is gone, and the Pi runtime can inject that credential without an undeclared-profile guard or prominent warning.

## What This Changes

The issue asks OpenClaw to stop or clearly warn when Pi fallback uses per-agent auth-profile credentials whose `openclaw.json` declarations were removed.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured profiles, add a non-destructive warning or doctor/models-status diagnostic for undeclared stored credentials, and keep legacy drift recovery only under an explicit compatibility rule.

## Reproduction Assessment

Yes. A high-confidence source-level path is to remove the provider's `auth.profiles` declaration while leaving a valid stored credential, then follow `resolveAuthProfileOrder` into `resolveApiKeyForProvider` and the Pi auth controller's `setRuntimeApiKey` call.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for removed declarations; the narrow maintainable fix is a declaration-aware guard or prominent warning before any destructive cleanup is considered.
