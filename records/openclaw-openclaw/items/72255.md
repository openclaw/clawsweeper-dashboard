## Summary

Keep open. Current main still treats stored auth-profile credentials as selectable when no matching `auth.profiles` declaration remains, and the Pi embedded path can inject the selected credential into runtime auth; the adjacent stale-key and Ollama fallback issues do not fully cover this removed-declaration billing path.

## What This Changes

Request that Pi/runtime auth stop silently using per-agent stored credentials after the corresponding `openclaw.json` `auth.profiles` declaration is removed, or at least warn and diagnose that drift.

## Best Possible Solution

Adopt a declaration-aware credential authority model for runtime selection: ignore or prominently flag stored provider profiles whose ids are no longer declared for that provider, and add non-destructive `doctor`/`models status` diagnostics plus an explicit cleanup command for stored-but-undeclared profiles.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no `auth.profiles` entry for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile before the Pi auth controller resolves and injects it into runtime auth.

## Solution Assessment

No. The current silent stored-credential reuse is not the best solution for the reported billing/security case; the safer direction is an explicit allowlist or loud drift warning with diagnostic remediation, not hidden runtime use or automatic destructive pruning.
