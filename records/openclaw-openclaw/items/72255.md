## Summary

Keep open. Current main still has the source-level path described in the report: undeclared credentials can remain eligible through the stored auth-profile order, and the Pi runtime can inject the resolved key without a declaration-drift guard or warning.

## What This Changes

The issue asks OpenClaw to stop or visibly warn when Pi/runtime auth uses credentials left in per-agent `auth-profiles.json` after their `openclaw.json` profile declarations are removed.

## Best Possible Solution

Runtime auth selection should become declaration-aware for user-managed profiles, with a clear warning/audit or quarantine path for undeclared stored credentials and focused tests covering the Pi candidate path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the provider's `auth.profiles` declaration while leaving a valid stored credential, then follow `resolveAuthProfileOrder` into Pi `profileCandidates` and `setRuntimeApiKey`.

## Solution Assessment

No. Current main's silent stored-profile fallback is not the best solution for removed declarations; a safer fix should add a declaration-aware guard or visible diagnostic before considering destructive cleanup.
