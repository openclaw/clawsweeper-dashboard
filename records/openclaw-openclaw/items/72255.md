## Summary

Keep open. Current main still permits a per-agent stored auth profile to be selected when no matching `auth.profiles` declaration remains, and the embedded Pi auth path can inject that resolved credential without an undeclared-profile guard or warning. The related closed reports overlap auth-store drift and fallback behavior, but they do not fully cover this removed-declaration billing path.

## What This Changes

The issue requests that runtime auth stop silently using per-agent stored credentials after their `openclaw.json` `auth.profiles` declarations are removed, or at least warn/diagnose that drift.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime selection should ignore or prominently flag stored provider credentials whose profile ids are no longer declared, while `doctor` or `models auth` exposes stored-but-undeclared drift with an explicit cleanup or quarantine flow and updated auth semantics docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no `auth.profiles` entry for a provider, keep a valid stored credential for that provider in the auth store, then observe `resolveAuthProfileOrder` returning the stored profile and the Pi auth controller injecting its API key into runtime auth.

## Solution Assessment

No. The current silent reuse of stored credentials is not the best way to solve a removed-declaration billing case; the safer maintainable path is a maintainer-approved declaration guard or high-visibility warning plus diagnostics, rather than hidden runtime selection or default destructive pruning.
