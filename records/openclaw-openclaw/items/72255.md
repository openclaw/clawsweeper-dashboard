## Summary

Keep open. Current main and the latest release still allow a provider-compatible stored auth profile to become the runtime candidate when no compatible declared profile constrains that provider, so the reported silent credential/provider reuse remains a real auth-provider and billing/security concern.

## What This Changes

The issue reports that removing an Anthropic auth profile declaration from `openclaw.json` can leave a stored profile selectable by the OpenClaw/Pi runtime, causing silent credential/provider reuse.

## Best Possible Solution

Define the authoritative contract for removed auth profile declarations, then implement resolver behavior and doctor/probe diagnostics consistently with security review and focused upgrade tests.

## Reproduction Assessment

Yes, source-reproducible: current main still falls back to compatible stored profiles when no compatible declared profile or explicit order constrains the provider, and the embedded runner consumes that order. I did not perform a live paid-provider run.

## Solution Assessment

Unclear until maintainers choose the auth authority contract. A resolver rule plus doctor/probe warning or migration is safer than only logging after a credential has already been selected.
