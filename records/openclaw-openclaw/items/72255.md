## Summary

Keep open. Current main still lets per-agent auth-profile credentials remain eligible when the matching `openclaw.json` declaration is absent, and the Pi embedded path can forward and inject that selected credential without an undeclared-profile guard or warning.

## What This Changes

The issue requests preventing or loudly surfacing Pi/runtime use of stored auth-profile credentials whose `openclaw.json` declarations were removed.

## Best Possible Solution

Adopt declaration-aware credential authority for runtime selection, paired with non-destructive doctor/status drift reporting so users can recover legacy profiles without silent billing changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the provider's `auth.profiles` declaration, leave a usable provider credential in the per-agent store, and follow `resolveAuthProfileOrder` through Pi `profileCandidates` into `setRuntimeApiKey`.

## Solution Assessment

Unclear. Strict pruning at startup is riskier than a declaration-aware resolver plus warning/quarantine path because current behavior intentionally preserves some legacy stored-profile drift repair.
