## Label Justifications

- `P1`: Silent reuse of a removed credential can affect live provider routing and billing for real agent runs.
- `impact:security`: The issue concerns stored credentials being reused after their user-visible auth-profile declaration was removed.
- `impact:auth-provider`: The affected behavior is auth-profile resolution and provider routing for Pi/runtime model calls.

## Summary

Keep open. Current main and the latest release still allow provider-compatible credentials from per-agent `auth-profiles.json` to remain eligible after their `openclaw.json` declaration is removed, and the Pi path can inject the selected credential without a declaration-drift guard.

## What This Changes

The issue requests that Pi/runtime auth stop silently using per-agent stored credentials after their `openclaw.json` auth-profile declaration is removed, or at least warn and diagnose that drift.

## Best Possible Solution

Define the credential authority rule between `openclaw.json` declarations and per-agent stored credentials, then add a resolver/Pi guard or high-visibility warning plus `doctor` diagnostics and regression coverage.

## Reproduction Assessment

Yes, at source level. Remove an `openclaw.json` auth-profile declaration while leaving a provider-compatible entry in per-agent `auth-profiles.json`; current `resolveAuthProfileOrder` can still select the stored profile and Pi can inject its key.

## Solution Assessment

Unclear until maintainers decide the credential authority policy. Destructive startup pruning is risky by default; a declaration-aware selection guard or loud warning with `doctor` diagnostics is safer than silently deleting stored credentials.
