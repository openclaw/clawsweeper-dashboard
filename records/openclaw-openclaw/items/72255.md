## Summary

Keep open. Current main still allows a credential stored only in per-agent `auth-profiles.json` to remain eligible for provider auth when the matching `openclaw.json` declaration is gone, and the Pi path can inject that resolved credential without an undeclared-profile guard or visible warning.

## What This Changes

The issue asks for Pi/runtime auth selection to stop or clearly warn when a credential remains only in per-agent `auth-profiles.json` after its `openclaw.json` `auth.profiles` declaration is removed.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured profiles, add a non-destructive warning or doctor/models-status diagnostic for undeclared stored credentials, and preserve legacy drift recovery only under an explicit compatibility rule.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: remove the provider's `auth.profiles` declaration while leaving a valid stored credential, then follow `resolveAuthProfileOrder` into Pi `profileCandidates` and `setRuntimeApiKey`.

## Solution Assessment

Is this the best way to solve the issue? No for the current behavior: silent stored-profile fallback is not the right end state for removed declarations, but the exact fail-closed versus warn-first compatibility rule needs maintainer/security judgment.
