## Summary

Keep open. Current main still lets stored per-agent auth-profile credentials remain eligible when the matching `openclaw.json` declaration is gone, and the Pi embedded path can inject that selected credential without an undeclared-profile guard or warning.

## What This Changes

The issue asks for removed `auth.profiles` declarations to stop, warn about, or diagnostically surface matching credentials that remain in per-agent `auth-profiles.json`.

## Best Possible Solution

Adopt declaration-aware credential authority: runtime selection should ignore or prominently flag stored provider credentials whose profile ids are no longer declared, with a non-destructive diagnostic or quarantine path for recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide no `auth.profiles` entry for a provider, keep a usable stored credential for that provider, and follow `resolveAuthProfileOrder` through Pi `profileCandidates` into `setRuntimeApiKey`.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for removed declarations; a safer fix should add a declaration-aware guard or warning before considering destructive pruning.
