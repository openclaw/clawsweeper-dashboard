## Label Justifications

- `P1`: Silent selection of a credential the user believed removed can affect live provider routing and billing in real agent runs.
- `impact:security`: The issue concerns stored credentials remaining usable after their user-visible declaration is removed.
- `impact:auth-provider`: The affected behavior is auth-profile resolution and provider routing for Pi/runtime model calls.

## Summary

Keep open. Current main and the latest release still allow provider-compatible credentials in `auth-profiles.json` to be selected when their `openclaw.json` declaration is absent, and the Pi embedded path can pass that selected credential into runtime auth; the related closed issues cover adjacent stale-key and local-fallback cases, not this removed-declaration billing path.

## What This Changes

The issue asks OpenClaw to stop or loudly surface Pi runtime use of stored auth-profile credentials after their user-visible `openclaw.json` declarations are removed.

## Best Possible Solution

Define a declaration-aware credential authority rule, then implement either a resolver/Pi guard or high-visibility warning plus doctor/status diagnostics with focused tests for removed declarations and existing drift-repair compatibility.

## Reproduction Assessment

Yes, at source level. Remove the config declaration while leaving a provider-compatible credential in `auth-profiles.json`; `resolveAuthProfileOrder` can return the stored profile and the Pi runner can pass it into runtime auth storage.

## Solution Assessment

Unclear until maintainers decide the compatibility rule. Destructive startup pruning is not obviously the best default because current tests preserve stored-profile fallback, but a declaration-aware guard or loud warning plus diagnostics is a maintainable path.
