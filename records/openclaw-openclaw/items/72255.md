## Summary

Keep open. Current main still allows stored per-agent auth-profile credentials to be selected when no matching `auth.profiles` declaration remains, and the Pi embedded path can inject that selected credential into runtime auth without an undeclared-profile guard or warning. The related stale-key and Ollama fallback reports overlap the auth-store drift theme, but they do not cover this removed-declaration billing path.

## What This Changes

The issue reports that removing an auth profile declaration from `openclaw.json` does not stop the Pi embedded runtime from selecting the remaining per-agent stored credential, causing unexpected provider billing or quota use.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime selection should ignore or loudly flag stored provider credentials whose profile ids are no longer declared for that provider, while `doctor`/`models status` expose stored-but-undeclared drift and an explicit cleanup or quarantine command preserves recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no `auth.profiles` entry for a provider, keep a valid stored credential for that provider, then observe `resolveAuthProfileOrder` returning the stored profile and the Pi auth controller injecting the resolved key into runtime auth.

## Solution Assessment

No. The current silent reuse of stored credentials is not the best solution for a removed-declaration billing case; the safer maintainable direction is a declaration-aware guard or prominent drift warning plus diagnostics, not hidden runtime selection or automatic destructive pruning by default.
