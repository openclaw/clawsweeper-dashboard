## Label Justifications

- `P1`: Silent selection of a credential the user believed removed can affect live provider routing and billing for real agent runs.
- `impact:security`: The issue concerns stored credentials being reused after their user-visible declaration is removed.
- `impact:auth-provider`: The affected behavior is auth-profile resolution and provider routing for Pi/runtime model calls.

## Summary

Keep open. Current main and the latest release still allow a provider-compatible stored auth profile to remain selectable when its `openclaw.json` declaration is absent, and the Pi path can inject that credential; the related closed reports cover adjacent stale-key and local-fallback cases but not this removed-declaration billing path.

## What This Changes

The issue asks OpenClaw to stop or warn when Pi/runtime auth selects a stored auth profile whose `openclaw.json` declaration was removed.

## Best Possible Solution

Define a declaration-aware credential authority rule, then add a resolver/Pi guard or high-visibility warning plus doctor/status diagnostics and focused tests without breaking intentional stored-profile fallback.

## Reproduction Assessment

Yes at source level. Remove the config declaration while leaving a provider-compatible credential in `auth-profiles.json`; `resolveAuthProfileOrder` can return the stored profile and the Pi runner can pass it into runtime auth storage.

## Solution Assessment

No for destructive startup pruning as the default without a maintainer decision. A safer path is a declaration-aware selection guard or loud warning plus diagnostics after secops/product decide how to preserve intentional stored-profile fallback and drift repair.
