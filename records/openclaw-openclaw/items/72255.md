## Summary

Keep open. Current main still permits compatible stored auth-profile credentials to be selected when their `auth.profiles` declaration is absent, and that selected credential can flow into Pi runtime auth; the related closed issues fixed configured-key precedence and Ollama marker fallback, not this removed-declaration billing path.

## What This Changes

Prevent Pi runtime auth from silently using stored credentials whose `auth.profiles` declaration was removed from `openclaw.json`, or at least warn and diagnose that drift.

## Best Possible Solution

Define the auth-profile authority rule for configured providers, add a declaration-aware block or high-visibility warning in the runtime/Pi path, and cover it with focused resolver and Pi auth tests while preserving intentional configless/drift-repair behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: omit the provider profile from `cfg.auth.profiles`, leave a compatible credential in `auth-profiles.json`, and `resolveAuthProfileOrder` can return that stored profile for Pi runtime auth.

## Solution Assessment

No for destructive startup pruning as the default. A declaration-aware runtime guard or high-visibility warning is safer, but maintainers need to choose how it coexists with configless stored profiles and profile-id drift repair.
