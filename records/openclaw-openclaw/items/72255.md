## Summary

Keep open. Current main still allows provider-compatible stored auth profiles to be selected when their `openclaw.json` declaration is absent, and the Pi path can carry the selected credential into runtime auth without an orphan/declaration guard.

## What This Changes

The issue reports that Pi runtime can silently use a stored per-agent auth profile after its declaration is removed from `openclaw.json`, causing unexpected provider billing or quota use.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, warn or block orphaned stored credentials in the Pi path, and add focused resolver/Pi coverage that preserves intentional configless and drift-repair behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remove the provider profile from `cfg.auth.profiles`, leave a compatible credential in `auth-profiles.json`, and `resolveAuthProfileOrder` can still return it for Pi to inject into runtime auth.

## Solution Assessment

No for destructive startup pruning as the default. The safer direction is a declaration-aware runtime guard or high-visibility warning, with an explicit compatibility rule for configless stored profiles and profile-id drift repair.
