## Summary

Keep open. Current main and the latest release still allow provider-compatible stored auth profiles to become runtime candidates when declarations are absent or drifted, so the reported silent credential reuse remains a real auth-provider/security issue.

## What This Changes

The issue reports that Pi fallback can silently reuse per-agent stored credentials after the matching `openclaw.json` auth profile declaration is removed.

## Best Possible Solution

Define the authoritative contract for removed auth profile declarations, then update resolver eligibility plus doctor/status diagnostics with focused upgrade tests.

## Reproduction Assessment

Yes, source-reproducible: current main still lets compatible stored profiles become candidates when no explicit auth order or compatible declared profile constrains the provider. I did not run a live paid-provider reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the auth authority contract. A resolver eligibility rule plus doctor/status warning or migration is safer than only logging after a credential has already been selected.
