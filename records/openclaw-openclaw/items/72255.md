## Summary

Keep open. Current main still lets provider-compatible stored auth profiles remain selectable after their `openclaw.json` declaration is absent, and the Pi path can inject that selected credential without an orphan/declaration guard.

## What This Changes

The issue asks for Pi embedded auth resolution to stop silently using credentials left in per-agent `auth-profiles.json` after their declarations are removed from `openclaw.json`.

## Best Possible Solution

Define a declaration-aware runtime rule for configured providers, block or loudly warn on undeclared stored-profile selection in the Pi path, and preserve intentional configless and profile-id drift repair behavior with focused resolver/Pi tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: remove a compatible provider profile from `cfg.auth.profiles`, leave its credential in `auth-profiles.json`, and the resolver can still return it for Pi runtime auth injection.

## Solution Assessment

No for destructive startup pruning as the default. A safer fix is a declaration-aware selection guard or high-visibility warning, plus diagnostics, after secops/product define how this interacts with configless stored profiles and drift repair.
