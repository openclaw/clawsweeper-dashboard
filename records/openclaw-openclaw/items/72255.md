## Summary

Keep open. Current `main` still permits a provider-compatible credential that exists only in `auth-profiles.json` to enter the resolved auth order, flow through model auth, and be injected into the Pi runtime without a declaration-drift guard or warning.

## What This Changes

The issue reports that removing an auth profile declaration from `openclaw.json` can leave Pi eligible to use the matching stored credential from per-agent `auth-profiles.json`.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, add a loud diagnostic for undeclared stored credentials, and preserve configless/drift-repair compatibility under an explicit rule.

## Reproduction Assessment

Yes. Source inspection on current main shows that a removed declaration leaves no matching `explicitProfiles`, so `resolveAuthProfileOrder` falls back to stored provider profiles and Pi later injects the resolved key.

## Solution Assessment

No for the destructive startup-prune suggestion as-is. A declaration-aware guard or high-visibility warning is safer because current tests and docs preserve stored-profile and drift-repair compatibility.
