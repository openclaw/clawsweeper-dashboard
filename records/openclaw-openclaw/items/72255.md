## Summary

Keep open. Current main and the latest release still allow provider-compatible stored auth profiles to become runtime credential candidates when the matching `auth.profiles` declaration is absent, so the reported silent credential reuse remains unresolved and needs maintainer/security direction.

## What This Changes

The issue requests preventing or warning on Pi/OpenClaw runtime use of stored auth credentials whose `auth.profiles` declaration was removed from `openclaw.json`.

## Best Possible Solution

Define the authoritative contract for removed auth profile declarations, then update resolver eligibility plus doctor/status diagnostics with focused upgrade and security tests for the SQLite auth store.

## Reproduction Assessment

Yes, source-level. Current main still selects provider-compatible stored profiles when no compatible configured declaration constrains the provider; I did not run a live paid-provider reproduction.

## Solution Assessment

Unclear until maintainers choose the auth authority contract. A resolver eligibility rule plus doctor/status warning or migration is safer than only logging after a credential has already been selected.
