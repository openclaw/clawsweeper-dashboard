## Summary

Keep open. Current main still lets stored provider credentials be selected when no matching `auth.profiles` declaration remains for that provider, and the Pi embedded path can inject that resolved credential into runtime auth without an undeclared-profile guard. The linked stale-key and silent-fallback issues are useful adjacent context, but they do not supersede this removed-declaration billing path.

## What This Changes

The issue requests that OpenClaw stop silently selecting per-agent stored credentials after their `openclaw.json` `auth.profiles` declarations are removed, especially in the Pi embedded fallback path.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime selection should ignore or prominently warn on stored provider profiles whose ids are no longer declared for that provider, while `openclaw doctor` or `models status` reports stored-but-undeclared profiles with a non-destructive cleanup path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no `auth.profiles` entry for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile for the Pi auth controller to inject into runtime auth.

## Solution Assessment

No. The current silent reuse behavior is not the best way to solve the issue; the narrow maintainable direction is a resolver allowlist or selection-time warning plus diagnostic drift reporting, not hidden billing through removed declarations or automatic secret deletion.
