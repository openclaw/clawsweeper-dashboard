## Summary

Keep open. Current main still lets stored auth-profile credentials remain selectable when the matching `openclaw.json` declaration is gone, and Pi can inject that resolved credential without an undeclared-profile guard or prominent warning.

## What This Changes

The issue requests that removing an auth profile declaration from `openclaw.json` stop Pi/runtime use of the leftover per-agent credential, or at least emit a prominent diagnostic.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured profiles, add a non-destructive warning or doctor/models-status diagnostic for undeclared stored credentials, and preserve legacy drift recovery only under an explicit compatibility rule.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the provider's `auth.profiles` declaration while leaving a valid stored credential, then follow `resolveAuthProfileOrder` into `profileCandidates` and `setRuntimeApiKey`.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for removed declarations; the narrow maintainable fix is a declaration-aware guard or prominent warning before any destructive cleanup is considered.
