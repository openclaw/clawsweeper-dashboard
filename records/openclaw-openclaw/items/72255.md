## Summary

Keep open. Current main still allows stored provider credentials to be selected when no matching `auth.profiles` declaration remains, and the Pi embedded path can inject that resolved credential into runtime auth without an undeclared-profile guard. The related stale-key and silent-fallback reports are useful context, but they do not fully supersede this removed-declaration billing route.

## What This Changes

The issue asks OpenClaw to stop silently selecting provider credentials that remain in per-agent `auth-profiles.json` after their `openclaw.json` `auth.profiles` declaration has been removed, especially in the Pi embedded fallback path.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime selection should ignore or loudly flag stored provider profiles whose ids are no longer declared for that provider, while `openclaw doctor` or `models status` reports stored-but-undeclared profiles with non-destructive remediation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no `auth.profiles` entry for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile, which the Pi auth controller then resolves and injects into `authStorage`.

## Solution Assessment

No. The current silent reuse behavior is not the best way to solve the issue; the narrow maintainable direction is a resolver allowlist or selection-time warning plus diagnostic drift reporting, not hidden billing through removed declarations or automatic secret deletion.
