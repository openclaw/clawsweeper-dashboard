## Summary

Keep open. Current main still lets a stored auth-profile credential remain eligible when its `auth.profiles` declaration is absent, and the Pi runtime path can inject that selected credential without a declaration-drift guard or warning.

## What This Changes

The issue requests preventing or loudly warning about Pi runtime use of per-agent auth-profile credentials after their declarations are removed from `openclaw.json`.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, add a high-visibility warning or diagnostic for undeclared stored credentials, and preserve configless or drift-repair compatibility only under an explicit documented rule.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: with the declaration removed, `resolveAuthProfileOrder` can fall back to stored provider profiles, and Pi passes the selected profile through to runtime auth storage. I did not run a live provider billing request in this read-only sweep.

## Solution Assessment

No, destructive startup pruning is not clearly the best default. A declaration-aware guard or loud warning plus an explicit compatibility rule for configless stored profiles and profile-id drift repair is the safer maintainable direction.
