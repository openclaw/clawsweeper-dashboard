## Summary

Keep open. Current main still allows undeclared stored auth-profile credentials to remain eligible for provider auth and Pi runtime injection, while the related closed issues cover configured-key precedence and Ollama marker fallback rather than removed `auth.profiles` declarations.

## What This Changes

The issue asks OpenClaw to stop or warn on Pi runtime use of credentials left in per-agent `auth-profiles.json` after their `openclaw.json` auth-profile declarations are removed.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, warn or block orphaned stored credentials in the Pi path, and add focused resolver/Pi auth coverage that preserves intentional configless behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `resolveAuthProfileOrder` can fall back to compatible stored profiles when config lacks an explicit profile, and Pi then passes those candidates into runtime auth; no live Anthropic billing call was run.

## Solution Assessment

No for destructive startup pruning as the default. The safer direction is a declaration-aware runtime guard or high-visibility warning with an explicit compatibility rule for configless stored profiles and profile-id drift repair.
