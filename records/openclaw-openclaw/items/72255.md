## Summary

Keep open. Current main and the latest release still allow compatible stored auth profiles to become runtime candidates when no declared `auth.profiles` entry constrains that provider, and the issue is security/billing-relevant enough to need maintainer product and secops review before choosing strict pruning versus loud diagnostics.

## What This Changes

The issue requests preventing or warning when Pi/OpenClaw runtime uses stored per-agent credentials whose `openclaw.json` auth profile declaration was removed.

## Best Possible Solution

Define one authoritative auth-profile contract for removed declarations, then implement it consistently in resolver selection and `openclaw doctor --fix` with focused upgrade/security tests.

## Reproduction Assessment

Yes, source-reproducible: the current resolver falls back from absent configured profiles to compatible stored profiles, and the embedded runner consumes that order. I did not perform a live paid-provider run.

## Solution Assessment

Unclear until maintainers choose the auth authority contract. A resolver guard plus doctor migration/warning is more maintainable than only adding a late log after the wrong credential is already selected.
