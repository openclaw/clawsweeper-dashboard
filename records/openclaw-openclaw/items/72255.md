## Summary

Keep open. Current main still lets stored auth-profile credentials remain selectable when the matching `openclaw.json` declaration is absent, and the Pi embedded path can inject the resolved credential without an undeclared-profile guard or loud runtime warning.

## What This Changes

The issue asks OpenClaw to stop or clearly warn when the Pi runtime uses per-agent auth-profile credentials whose `openclaw.json` declarations have been removed.

## Best Possible Solution

Runtime auth resolution should become declaration-aware for live calls, with non-destructive diagnostics or quarantine for undeclared stored credentials and a reviewed cleanup command for explicit user remediation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to remove the provider's `auth.profiles` declaration while leaving a valid stored credential, then follow `resolveAuthProfileOrder` into `resolveApiKeyForProvider`, Pi `profileCandidates`, and `setRuntimeApiKey`.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for removed declarations; a safer fix should add a declaration-aware runtime guard or warning before considering destructive cleanup.
