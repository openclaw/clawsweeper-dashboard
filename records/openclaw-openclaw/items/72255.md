## Summary

Keep open. Current main still allows an undeclared stored auth-profile credential to remain eligible for Pi runtime auth, and the related closed issues only cover configured-key precedence and Ollama marker fallback, not removed `auth.profiles` declarations.

## What This Changes

Prevent Pi/runtime auth from silently using per-agent `auth-profiles.json` credentials after their `openclaw.json` `auth.profiles` declarations are removed, or surface that drift clearly.

## Best Possible Solution

Make runtime auth selection declaration-aware for configured providers, add a loud non-destructive warning or diagnostic for undeclared stored credentials, and preserve configless/drift-repair compatibility only under an explicit rule.

## Reproduction Assessment

Yes. Source inspection against current main shows that a missing `auth.profiles` declaration leaves no compatible explicit profile, `resolveAuthProfileOrder` falls back to stored provider profiles, and the selected credential flows into Pi runtime auth storage.

## Solution Assessment

No for a destructive startup prune as the default fix. The safer direction is a declaration-aware guard or high-visibility warning plus an explicit compatibility rule for configless stored profiles and profile-id drift repair.
