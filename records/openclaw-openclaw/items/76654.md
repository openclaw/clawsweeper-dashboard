## Summary

Keep open: current main still has a source-visible WebChat path where tool-final or silent-final events reload `chat.history`, and the merge helper only protects locally optimistic tail messages. The corrected reporter note and second user confirmation make this a credible remaining bug rather than an already-shipped cleanup candidate.

## What This Changes

The issue requests that WebChat keep previously rendered assistant responses visible after heartbeat-triggered tool calls, especially in the MiMo V2 Pro workflow.

## Best Possible Solution

Land a generic WebChat history reconciliation fix that preserves already-visible assistant responses through heartbeat/tool-final reloads, with regression coverage around the reload and projection boundary rather than a MiMo-specific workaround.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main MiMo/heartbeat run was established here; source inspection does show a credible reload path where a stale `chat.history` response can replace already visible assistant UI state.

## Solution Assessment

Is this the best way to solve the issue? Unclear whether a separate new PR is the best route because #70391, #74733, and #75776 overlap this area; the best technical direction is a narrow generic history merge/projection regression fix, not a model-specific policy change.
