## Summary

Keep open. Current main still has a credible WebChat history-reload path where heartbeat/tool-final events can replace visible chat state, and the existing guard only preserves local optimistic messages rather than already-rendered persisted assistant messages.

## What This Changes

Prevent WebChat heartbeat/tool-call history reloads from removing already-rendered assistant responses, especially in the reported MiMo V2 Pro heartbeat workflow.

## Best Possible Solution

Land or fold a focused Control UI history-reconciliation fix that preserves already-visible persisted assistant messages through heartbeat/tool-final reloads, with regression coverage for the heartbeat-after-response case.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was established, but source inspection shows a current-main path from heartbeat/tool-final events into `loadChatHistory` replacement that can drop already-visible persisted assistant messages.

## Solution Assessment

Is this the best way to solve the issue? The best direction is a narrow WebChat history merge/reload guard, not a MiMo-specific workaround; the exact implementation should be coordinated with the overlapping open WebChat reliability PRs.
