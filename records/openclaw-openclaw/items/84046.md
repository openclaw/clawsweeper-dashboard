## Summary

Keep this PR open: the reported false-positive takeover is well supported and the fix direction is right, but the current patch still leaves the new fence snapshot able to pair a stale stat fingerprint with bytes read from a later file state.

## What This Changes

The PR changes the embedded runner session-file fence to tolerate append-only canonical runner-owned assistant/tool transcript entries during the released prompt window while rejecting other mutations as takeovers.

## Best Possible Solution

Land the runner-owned append tolerance after making snapshot creation describe one consistent file state, while leaving broader co-tenant write semantics to the canonical follow-up issue.

## Reproduction Assessment

Yes. Current-main source shows every runner-owned append changes the stat fingerprint after releaseForPrompt, and the PR body plus production validation comment provide real long-turn logs for the false takeover path.

## Solution Assessment

No, not yet. The append-only runner-owned slow path is the right narrow direction, but snapshotSessionFileFence should first guarantee that its fingerprint and prefix hash came from the same file state.

- **[P2] Re-stat the file before storing the snapshot:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:193-196`
  - body: snapshotSessionFileFence records the fingerprint, then hashes whatever readFile returns without proving the file still has that same stat.
