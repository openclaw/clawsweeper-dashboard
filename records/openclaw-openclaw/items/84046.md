## Summary

Keep open: the latest revision fixes the earlier role and non-message tail problems, but the slow path still carries the old prefix hash forward after accepting a runner-owned tail, so a later valid append can false-trip the same takeover fence.

## What This Changes

The PR changes the embedded attempt session-lock fence to tolerate verified append-only runner-owned transcript growth during the prompt-lock release window and adds regression tests.

## Best Possible Solution

Keep the append-only dev/ino/birthtime/prefix-hash design, but advance the accepted snapshot hash to the full current file bytes, validate accepted tail message schemas, and keep the broader linked issue open or narrow the closing language before merge.

## Reproduction Assessment

Yes. Current main throws on any stat mismatch after prompt-lock release, and the PR body supplies redacted Telegram gateway logs showing that false takeover shape after a successful long DM turn.

## Solution Assessment

No, not yet. The approach is maintainable, but the accepted slow-path snapshot must hash the full accepted file and validate accepted message shapes so repeated runner-owned appends do not reintroduce the false takeover.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Advance the snapshot hash after accepting a tail:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:264`
  - body: After the slow path accepts one append-only runner-owned tail, `nextSnapshot` pairs the new larger `fingerprint.size` with `prefixHash`, which was computed only over the old `fenceOffset`. On the next valid runner-owned append, verification hashes bytes `[0,newSize)` and compares them to that old-prefix hash, so the prefix check fails and throws another `EmbeddedAttemptSessionTakeoverError`. Recompute the accepted snapshot hash over the full current file before storing it, and add a two-step append regression.
  - confidence: 0.93
- **[P2] Validate accepted tail message schemas:** `src/agents/pi-embedded-runner/run/attempt.session-lock.ts:257-259`
  - body: The tail parser now rejects non-message entries, but any JSON message with role `assistant`, `toolResult`, or `bashExecution` is accepted even if it is not a valid persisted message shape. Since this check is the replacement for the ownership fence's fail-closed path, it should share or mirror the canonical transcript-file-state validation for those roles instead of blessing malformed session entries by role string alone.
  - confidence: 0.78
