## Summary

Keep open. Current main still leaves persisted cron jobs without an `id` on the load path and still applies outcomes with an unguarded `find()` by `result.jobId`; #72853 is an open unmerged closing PR for the same bug, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue reports that persisted cron jobs missing `id` can collapse runtime state and error attribution into the first id-less job.

## Best Possible Solution

Land #72853 or an equivalent focused cron runtime fix that assigns stable IDs before state merge and rejects undefined job IDs before outcome matching.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load multiple persisted jobs with neither `id` nor `jobId`, let a run produce `jobId: undefined`, and `applyOutcomeToStoredJob()` will match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive undefined-id guards is the narrow maintainable fix; a doctor-only repair would not protect a gateway that loads malformed rows before repair runs.
