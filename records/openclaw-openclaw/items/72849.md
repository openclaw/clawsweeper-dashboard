## Summary

Keep open: current main and the latest shipped source still leave id-less persisted cron jobs without a stable ID, and an open linked fix PR already references this issue with closing syntax.

## What This Changes

The issue requests canonicalizing id-less persisted cron jobs and guarding runtime state updates so outcomes cannot match the first undefined-id job.

## Best Possible Solution

Land a focused cron fix that assigns stable persisted IDs before state merge, refuses invalid outcome IDs, and covers existing split-state sidecars.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted jobs missing both `id` and `jobId`, then let a run outcome or sidecar state merge use the missing job ID path; current main still reaches `stateFile.jobs[job.id]` and `entry.id === result.jobId`.

## Solution Assessment

Yes. Stable ID canonicalization before state merge plus defensive invalid-ID guards is the narrow maintainable fix, and https://github.com/openclaw/openclaw/pull/72853 is already the open implementation candidate.
