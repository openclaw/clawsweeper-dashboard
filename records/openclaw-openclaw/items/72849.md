## Summary

Keep open. Current main still has the reported cron missing-id collision path, and open PR #72853 is explicitly paired as the implementation candidate, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

Persisted cron jobs loaded without stable IDs should be backfilled and runtime state application should reject missing job IDs instead of matching undefined identities.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that assigns and persists stable IDs for missing or blank persisted job identities during runtime load, rejects missing job IDs in timer/state application paths, preserves doctor repair behavior, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let a run outcome carry `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first job whose `id` is also undefined.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around timer/state application is the narrow maintainable fix; doctor-only repair is not sufficient because the Gateway can still load and execute malformed persisted rows before doctor is run.
