## Summary

Keep open: the cron state-collision bug is still present on current main, and an open linked PR is already the implementation path, so this issue should wait for that PR to merge or be replaced.

## What This Changes

The issue requests stable IDs for persisted cron jobs missing `id` so runtime state cannot collapse onto the first id-less job.

## Best Possible Solution

Canonicalize missing cron job IDs before state merge, guard invalid outcome IDs, cover the `jobs-state.json` `undefined` case, and land https://github.com/openclaw/openclaw/pull/72853 or a corrected replacement.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: load multiple persisted cron jobs missing both `id` and `jobId`, then let sidecar state merge or timer outcome lookup use the missing ID path. I did not run a live Gateway repro because this review is read-only.

## Solution Assessment

Yes, the right fix is in the cron store/timer identity boundary, not a product change. The implementation should backfill before sidecar state merge and add defensive outcome guards with regression coverage.
