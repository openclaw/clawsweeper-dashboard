## Summary

Keep open: current main still lets persisted cron rows without a usable `id` reach the scheduler, and an open closing PR already targets the fix. This should close only after that PR or an equivalent runtime fix is merged.

## What This Changes

The issue asks cron to assign stable ids to hand-authored or migrated jobs missing `id` and to prevent undefined job ids from corrupting runtime state.

## Best Possible Solution

Land #72853 or an equivalent cron-owned fix that backfills stable ids before scheduling and refuses undefined job ids in state/outcome paths.

## Reproduction Assessment

Yes. This is source-reproducible on current main: missing ids survive load, due jobs can carry an undefined `jobId`, and the outcome lookup uses `Array#find` on `entry.id === result.jobId`.

## Solution Assessment

Yes. Load-time stable id backfill plus defensive undefined-id guards is the narrow maintainable fix; doctor-only repair is useful but does not protect normal gateway startup from malformed persisted rows.
