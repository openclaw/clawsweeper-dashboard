## Summary

Keep open: current main still permits the source-reproducible cron identity collision, and an open fix PR with closing syntax already owns the implementation path, so the issue should not be closed before that fix or an equivalent one lands.

## What This Changes

The issue reports that hand-authored cron jobs missing both `id` and legacy `jobId` can share an undefined identity and have runtime state applied to the first id-less job.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72853, or an equivalent focused fix, so cron load/hydration persists stable IDs for id-less jobs and timer/state paths reject invalid job IDs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs missing both `id` and `jobId`, then apply a run outcome; current main can carry an undefined `result.jobId` into `entry.id === result.jobId` and select the first id-less row.

## Solution Assessment

Yes. Stable ID backfill during cron store load/hydration plus defensive guards before outcome/state lookup is the narrow maintainable fix, and the open linked PR is already the implementation candidate.
