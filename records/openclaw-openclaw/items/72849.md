## Summary

Keep open. Current main still leaves persisted cron rows without `id` or legacy `jobId` id-less, and the timer/state paths can still compare `undefined` job IDs; #72853 is open and explicitly paired as the implementation candidate, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

Persisted cron jobs missing both `id` and legacy `jobId` should receive stable identities so runtime state cannot collapse onto the first id-less job.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that assigns and persists stable ids for missing or blank persisted job identities during runtime load/hydration, rejects missing job ids in timer/state application paths, preserves doctor repair behavior, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let a run outcome carry `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first job whose `id` is also undefined.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around timer/state application is the narrow maintainable fix; doctor-only repair is not sufficient because the Gateway can still load and execute malformed persisted rows before doctor is run.
