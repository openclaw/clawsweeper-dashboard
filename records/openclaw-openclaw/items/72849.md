## Summary

Keep open. Current main still leaves the cron missing-id collision path live, and #72853 is an open unmerged closing PR for the same bug, so this issue should wait for that PR to land or be replaced.

## What This Changes

The issue reports that cron jobs loaded from `jobs.json` without `id` or legacy `jobId` can share an undefined identity, causing runtime state and outcomes to update the first id-less job.

## Best Possible Solution

Review and land #72853, or an equivalent focused cron runtime fix, so malformed persisted jobs get stable IDs before state merge and timer outcome application, with regression coverage for multiple id-less jobs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, then let an outcome with `jobId: undefined` reach `applyOutcomeToStoredJob()` and match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state and outcome application is the narrow maintainable fix; doctor-only repair is useful but does not protect a gateway that loads malformed rows first.
