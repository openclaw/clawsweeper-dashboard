## Summary

Keep open. Current main still has the missing-id cron state collision path, and open PR #72853 already uses closing syntax for the same bug, so the issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests stable ID backfill and defensive guards so persisted cron jobs missing `id` cannot share runtime state through undefined-ID matching.

## Best Possible Solution

Review and land #72853, or an equivalent focused cron fix, so load-time identity canonicalization runs before state merge and timer outcomes reject invalid job IDs with regression coverage for multiple id-less persisted jobs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, then let an outcome with `jobId: undefined` reach `applyOutcomeToStoredJob()` and match the first id-less row.

## Solution Assessment

Yes. Load-time canonicalization plus defensive outcome/state guards is the narrow maintainable fix; doctor-only repair leaves the Gateway exposed until a user runs repair manually.
