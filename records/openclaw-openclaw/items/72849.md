## Summary

Keep open. Current main still leaves the missing-ID cron state-collision path live, and the issue is already paired with open, unmerged closing PR #72853, so it should wait for that PR to land or be replaced.

## What This Changes

The issue asks for cron store loading to assign stable IDs to jobs missing `id`/`jobId` and to guard outcome/state paths so runtime state cannot collapse into the first id-less job.

## Best Possible Solution

Review and land #72853, or an equivalent focused cron fix, so persisted jobs missing both `id` and legacy `jobId` get stable IDs before state merge and timer outcome application, with regression coverage for multiple id-less persisted jobs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, then let an outcome with `jobId: undefined` reach `applyOutcomeToStoredJob()` and match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state/outcome application is the narrow maintainable fix; doctor-only repair would not protect a Gateway that loads malformed rows before repair runs.
