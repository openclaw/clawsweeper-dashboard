## Summary

Keep open. Current main still has the cron missing-ID state-collision path, and the issue is already paired with open, unmerged closing PR #72853, so it should wait for that PR to land or be replaced.

## What This Changes

The issue requests stable ID backfill and defensive guards for persisted cron jobs missing `id` so runtime state cannot collapse onto the wrong job.

## Best Possible Solution

Review and land #72853, or an equivalent focused cron fix, so persisted jobs missing `id` or legacy `jobId` receive stable IDs before state merge and timer outcome application, with regression coverage for multiple id-less jobs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, then let a run outcome with `jobId: undefined` reach `applyOutcomeToStoredJob()` and match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state/outcome application is the narrow maintainable fix; relying only on `openclaw doctor --fix` would not protect a Gateway that loads malformed rows before repair runs.
