## Summary

Keep open: current main still has the source-reproducible cron identity collision, and an open linked fix PR already references this issue with closing syntax.

## What This Changes

The issue reports that persisted cron jobs missing `id` can share undefined identity and misattribute runtime state to the first id-less job.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72853, or an equivalent focused fix, so cron load/hydration persists stable IDs for id-less jobs and timer/state paths reject invalid job IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows multiple persisted jobs missing both `id` and `jobId` can flow through load with undefined IDs, then an outcome lookup can match the first id-less row through `entry.id === result.jobId`; I did not execute the runtime flow in this read-only pass.

## Solution Assessment

Yes. Stable ID backfill during cron store load/hydration plus defensive rejection of missing outcome IDs is the narrow maintainable fix, and the open linked PR is already the implementation candidate.
