## Summary

Keep open. Current main and the latest shipped release still allow id-less cron jobs through the load, sidecar-state merge, and outcome lookup paths, and the directly linked fix PR is still open and unmerged.

## What This Changes

The issue reports that hand-authored or legacy cron jobs missing `id` can share an undefined job identity, causing runtime state and outcome updates to land on the first id-less job.

## Best Possible Solution

Land a fix that canonicalizes id-less cron jobs before sidecar state merge, guards invalid outcome job IDs, and adds regression coverage for both `jobs.json` and `jobs-state.json` collision cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: load multiple persisted `jobs[]` rows with no `id` or `jobId`, then let `jobs-state.json` merge or `applyOutcomeToStoredJob()` compare against an undefined job ID.

## Solution Assessment

Yes, with one important refinement. The requested backfill-plus-guard is the right direction, but ID canonicalization needs to happen before `jobs-state.json` merge or invalid sidecar keys need to be ignored.
