## Summary

Keep open. Current main still has the reported cron missing-id collision path, and PR #72853 is open with the intended closing fix, so this issue should wait for that PR to be reviewed, landed, or closed rather than being swept independently.

## What This Changes

Request to backfill stable cron job IDs for persisted hand-authored or migrated jobs missing `id`, and to guard runtime state application so undefined job IDs cannot match the first id-less job.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that backfills and persists stable IDs for missing or blank persisted job identities during runtime load/hydration, rejects missing job IDs before timer/state application, keeps doctor repair behavior aligned, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let one due run produce an outcome whose `jobId` comes from the missing `job.id`, and observe `applyOutcomeToStoredJob()` matching the first id-less row via `undefined === undefined`.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around timer/state application is the narrow maintainable fix; doctor-only repair is not sufficient because Gateway runtime can still load and execute malformed persisted rows before doctor runs.
