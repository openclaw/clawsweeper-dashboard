## Summary

Keep open. Current main still leaves persisted cron jobs without load-time UUID backfill and still matches timer outcomes with `Array#find` on `id`; #72853 is an open unmerged closing PR, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that persisted cron jobs without `id` can share runtime state because scheduler and outcome matching compare undefined IDs.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that canonicalizes missing IDs before runtime state merge/outcome application, persists the canonical shape, and adds regression coverage for multiple id-less persisted jobs.

## Reproduction Assessment

Yes. Load two persisted jobs with neither `id` nor legacy `jobId`, then let a timed run outcome with missing `jobId` reach `applyOutcomeToStoredJob`; current main can match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state and outcome application is the narrow maintainable fix; doctor-only repair is useful but does not protect a gateway that loads malformed rows first.
