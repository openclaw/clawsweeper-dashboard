## Summary

Keep open. Current main still leaves persisted cron jobs with neither `id` nor legacy `jobId` on the runtime path, and #72853 is an open unmerged closing PR for the same bug.

## What This Changes

Ensure persisted cron jobs without `id` receive stable identities and prevent timer/state paths from matching undefined job IDs.

## Best Possible Solution

Land #72853 or an equivalent focused cron runtime fix that canonicalizes IDs before state merge and guards timer outcome matching.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: persisted jobs with neither `id` nor `jobId` enter `ensureLoaded()` without an ID, and an undefined `result.jobId` can match the first undefined-id job in `applyOutcomeToStoredJob()`.

## Solution Assessment

Yes. Load-time identity canonicalization before sidecar merge plus defensive timer guards is the narrow maintainable fix; doctor-only repair does not protect gateway startup/runtime paths.
