## Summary

Keep open. Current main still leaves persisted cron jobs with neither `id` nor legacy `jobId` on the runtime path, and #72853 is an open unmerged closing PR for the same bug.

## What This Changes

Cron jobs loaded from persisted config without an `id` should not share or misattribute runtime state through undefined job identity matches.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that assigns stable IDs before state merge and rejects undefined IDs before outcome matching.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load multiple persisted jobs with neither `id` nor `jobId`, let one run with `jobId: undefined`, and `applyOutcomeToStoredJob()` can match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization before sidecar state merge plus defensive timer guards is the narrow maintainable fix; doctor-only repair is useful but does not protect startup/runtime paths.
