## Summary

Keep open: current main still has the source-reproducible cron identity collision, and an open fix PR already references this issue with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that persisted cron jobs missing both `id` and legacy `jobId` can share an undefined identity, causing runtime state and outcome updates to attach to the first id-less job.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72853, or an equivalent focused fix, so cron load/hydration assigns stable persisted IDs before state merge and timer/state paths reject invalid job IDs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs missing both `id` and `jobId`, then apply a run outcome; current main can still carry an undefined `result.jobId` into `entry.id === result.jobId` and select the first id-less row.

## Solution Assessment

Yes. Stable ID canonicalization before state merge plus defensive guards before outcome/state lookup is the narrow maintainable fix, and the open linked PR is already the implementation candidate.
