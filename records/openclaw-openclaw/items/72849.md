## Summary

Keep open. Current main still has the missing-id cron state collision path, and this issue is paired with open, unmerged closing PR #72853, so it should wait for that PR or an equivalent focused fix to land.

## What This Changes

Persisted cron jobs without a canonical `id` can have runtime state and timer outcomes misattributed to the first id-less job.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that assigns stable IDs during runtime load before state merging/outcome application and rejects missing job IDs defensively, with regression coverage for multiple id-less persisted jobs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, let a due job produce an outcome with `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state and outcome application is the narrow maintainable fix; doctor-only repair is useful but does not protect a Gateway that loads malformed rows before repair runs.
