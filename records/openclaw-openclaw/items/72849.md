## Summary

Keep open. Current main still has the missing-id cron state collision path, and the issue is already paired with open, unmerged closing PR #72853, so the issue should wait for that PR to land or be replaced.

## What This Changes

Cron should assign stable identities to persisted jobs missing `id` so runtime state cannot be applied to the wrong job.

## Best Possible Solution

Review and land #72853, or an equivalent focused cron fix, so persisted jobs missing `id` or legacy `jobId` get stable IDs before state merge and timer outcome application, with regression coverage for multiple id-less persisted jobs.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, then let an outcome with `jobId: undefined` reach `applyOutcomeToStoredJob()` and match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state/outcome application is the narrow maintainable fix; doctor-only repair would not protect a Gateway that loads malformed rows before repair runs.
