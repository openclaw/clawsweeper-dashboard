## Summary

Keep open. Current main still leaves persisted cron jobs without `id` or legacy `jobId` id-less, and timer/state paths still compare possibly undefined job IDs; #72853 is already open and explicitly paired as the implementation candidate.

## What This Changes

The issue requests that cron runtime load/hydration backfill stable IDs for hand-authored or migrated jobs missing `id`, and that timer/state application paths refuse missing job IDs instead of colliding on `undefined`.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that assigns and persists stable IDs for missing or blank persisted job identities during runtime load/hydration, rejects missing job IDs in timer/state application paths, preserves doctor repair behavior, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let a run outcome carry `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first job whose `id` is also undefined.

## Solution Assessment

Yes. Runtime load-time identity canonicalization plus defensive guards around timer/state application is the narrow maintainable fix; doctor-only repair is not sufficient because the Gateway can still load and execute malformed persisted rows before doctor is run.
