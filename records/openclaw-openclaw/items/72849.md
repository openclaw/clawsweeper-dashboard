## Summary

Keep open: current main still leaves the missing-id cron runtime collision in place, and #72853 is an open unmerged closing PR for the same bug.

## What This Changes

Persisted cron jobs with neither `id` nor legacy `jobId` should receive stable IDs, and timer outcome matching should reject missing job IDs so runtime state cannot collapse onto the first id-less job.

## Best Possible Solution

Land #72853 or an equivalent focused cron runtime fix that canonicalizes IDs before state merge and guards outcome matching against missing IDs.

## Reproduction Assessment

Yes. Source inspection on current main shows multiple persisted jobs without `id`/`jobId` can load with `undefined` IDs, a due run can return `jobId: undefined`, and `applyOutcomeToStoredJob()` can match the first id-less job.

## Solution Assessment

Yes. Load-time identity canonicalization before sidecar state merge plus a defensive timer guard is the narrow maintainable repair; doctor-only repair does not protect gateway startup/runtime paths.
