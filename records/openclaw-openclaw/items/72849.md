## Summary

Keep open. Current main still leaves persisted cron jobs with neither `id` nor legacy `jobId` on the runtime path, and #72853 is an open unmerged closing PR for the same bug.

## What This Changes

Backfill or reject missing cron job IDs so runtime state cannot collapse onto the first id-less persisted job.

## Best Possible Solution

Land #72853 or an equivalent focused cron fix that assigns stable IDs before state merge and rejects undefined IDs before outcome matching.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load multiple persisted jobs with neither `id` nor `jobId`, run one so its outcome carries `jobId: undefined`, and `applyOutcomeToStoredJob()` can match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization before sidecar state merge plus defensive timer guards is the narrow maintainable fix; doctor-only repair is useful but does not protect gateway startup/runtime paths.
