## Summary

Keep open: current main still leaves id-less cron jobs without stable identity, and an open closing PR already tracks the focused fix, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that persisted cron jobs without an `id` or legacy `jobId` can share undefined identity and have runtime state applied to the first id-less job.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/72853, or an equivalent focused fix, so cron load/hydration persists stable IDs and timer/state paths reject malformed job IDs.

## Reproduction Assessment

Yes. A high-confidence source path exists: load multiple persisted cron jobs missing both `id` and `jobId`, then apply a run outcome whose `jobId` is undefined; current main can match the first id-less row through `entry.id === result.jobId`.

## Solution Assessment

Yes. The narrow maintainable solution is stable ID backfill during cron store load/hydration plus defensive guards before state merge/outcome lookup; the linked open PR is already the implementation candidate.
