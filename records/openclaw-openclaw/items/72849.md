## Summary

Keep open. Current main still leaves persisted cron jobs without IDs on the runtime path, and #72853 is an open unmerged closing PR for the same bug, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

Cron runtime should assign stable IDs to persisted jobs that lack both `id` and legacy `jobId`, then reject missing job IDs before state/outcome matching.

## Best Possible Solution

Land #72853 or an equivalent focused runtime fix that backfills stable IDs before state merge and rejects undefined job IDs before outcome matching.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: load multiple persisted jobs with neither `id` nor `jobId`, allow a run result with `jobId: undefined`, and `applyOutcomeToStoredJob()` can match the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive timer/state guards is the narrow maintainable fix; doctor-only repair is useful but does not protect the gateway runtime before repair runs.
