## Summary

Keep open. Current main still leaves id-less persisted cron jobs on the runtime path, and #72853 is an open unmerged closing PR for the same bug, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

The issue asks for persisted cron jobs missing `id` to receive stable IDs before runtime state merge/outcome application, with guards against undefined job IDs.

## Best Possible Solution

Land #72853 or an equivalent focused cron runtime fix that assigns stable IDs before state merge and rejects undefined job IDs before outcome matching.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`, let one run produce `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first id-less row.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive guards around state and outcome application is the narrow maintainable fix; doctor-only repair is useful but does not protect a gateway that loads malformed rows first.
