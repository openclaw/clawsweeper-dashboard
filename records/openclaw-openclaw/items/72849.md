## Summary

Keep open. Current main still has the missing-ID cron state collision path, and #72853 is an open unmerged closing PR for the same bug, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

Persisted cron jobs missing both `id` and legacy `jobId` should receive stable IDs and avoid merging runtime state through undefined job IDs.

## Best Possible Solution

Land #72853 or an equivalent focused cron runtime fix that assigns stable IDs before state merge and rejects undefined job IDs before timer outcome matching.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: load multiple persisted cron jobs with neither `id` nor `jobId`, let one outcome carry `jobId: undefined`, and current `applyOutcomeToStoredJob()` can match the first undefined-id row.

## Solution Assessment

Is this the best way to solve the issue? Yes: load-time stable-ID canonicalization plus defensive undefined-jobId guards is the narrow maintainable fix; doctor-only repair does not protect gateway runtime before repair runs.
