## Summary

Keep open. Current main c81c017 still leaves persisted cron jobs with neither `id` nor legacy `jobId` without a generated runtime ID, and #72853 is an open unmerged closing PR for the same bug.

## What This Changes

The issue asks for persisted cron jobs missing `id` to get stable IDs and for timer/state paths to reject undefined job IDs so runtime state cannot collapse onto the first job.

## Best Possible Solution

The cron runtime should canonicalize persisted job IDs before state merge/run dispatch and refuse outcome/state application when the ID is missing.

## Reproduction Assessment

Yes. From source, persist two jobs with neither `id` nor `jobId`, load them, let one produce an outcome with `jobId: undefined`, and `applyOutcomeToStoredJob()` will match the first undefined-id row.

## Solution Assessment

Yes. Load-time identity backfill before state merge plus defensive guards around outcome matching is the narrow maintainable fix; doctor-only repair is useful but insufficient for gateway runtime safety.
