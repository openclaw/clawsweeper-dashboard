## Summary

Keep open: the underlying inline-directive bug is real, but this PR reuses directive flags that also drive session persistence and it lacks after-fix real behavior proof from the contributor.

## What This Changes

The PR captures parsed think/verbose levels before mixed inline directives are cleared and re-adds them for authorized senders in `resolveReplyDirectives`.

## Best Possible Solution

Carry authorized mixed-content think/verbose values into only the current run using a non-persistent path, then add focused regression coverage for raw inline directives and gateway/session send behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a normal chat body with real text plus `/think <level>` or `/verbose <level>` reaches `clearInlineDirectives`, loses the parsed level, and then falls back during run resolution; I did not execute tests in this read-only sweep.

## Solution Assessment

No. Preserving the levels is the right direction, but setting `hasThinkDirective` and `hasVerboseDirective` back to true is not the best fix because those flags are also used to persist session defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep preserved inline levels out of session defaults:** `src/auto-reply/reply/get-reply-directives.ts:298-302`
  - body: This restores `hasThinkDirective`/`hasVerboseDirective` after clearing a mixed-content message. `applyInlineDirectiveOverrides` then calls `persistInlineDirectives`, which writes session defaults from those flags, but normal-chat inline directives are documented as per-message only. Preserve the run-level values without reusing the persistence flags, or make persistence skip this preserved mixed-inline case.
  - confidence: 0.9
