## Summary

Keep this PR open, but do not merge it as-is. Current main still drops mixed-content `/think` and `/verbose` levels, so the bug remains real; however, the PR restores the levels by setting `hasThinkDirective` and `hasVerboseDirective` back to true, and downstream directive handling treats those flags as session-persistence signals, contrary to the documented per-message behavior.

## What This Changes

The PR modifies `resolveReplyDirectives` to capture parsed think and verbose levels before `clearInlineDirectives` and reapply them for authorized mixed-content messages.

## Best Possible Solution

Implement an equivalent fix that carries authorized mixed-content think and verbose levels into the current run without reusing persistence flags for session-default updates. The clean end state should have focused directive-resolution tests plus Gateway regression coverage showing `chat.send`/`sessions.send` thinking affects the run but does not mutate `sessionEntry.thinkingLevel` unless the message is directive-only.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `chat.send` with a `thinking` value and non-command message synthesizes `/think <level> <message>`, current `resolveReplyDirectives` detects remaining content, calls `clearInlineDirectives`, and then resolves the run from cleared `thinkLevel`/`verboseLevel` fields.

## Solution Assessment

No. The PR addresses the right bug, but setting `hasThinkDirective` and `hasVerboseDirective` back to true is not the best fix because those flags drive downstream session persistence. A safer fix needs a per-message preservation path or persistence skip for mixed inline directives.
