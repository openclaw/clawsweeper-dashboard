## Summary

Keep open: the underlying mixed-inline directive bug is real on current main, but this PR reuses `hasThinkDirective` and `hasVerboseDirective` in a way that can persist per-message `/think` or `/verbose` values as session defaults, and the external PR still lacks real behavior proof.

## What This Changes

The PR changes `resolveReplyDirectives` to preserve parsed think and verbose levels when mixed-content inline directives are cleared for authorized senders.

## Best Possible Solution

Carry mixed-content think/verbose values into current-run resolution without setting session-persistence flags, and add regression coverage for both run application and non-persistence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an authorized message containing real text plus `/think <level>` or `/verbose <level>` reaches `clearInlineDirectives`, which drops the parsed levels before current-run resolution.

## Solution Assessment

No. Preserving the levels is the right direction, but setting `hasThinkDirective` and `hasVerboseDirective` reuses persistence semantics; the safer fix is to preserve run-only values or explicitly bypass persistence for this mixed-inline case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep preserved inline levels out of session defaults:** `src/auto-reply/reply/get-reply-directives.ts:298-305`
  - body: This re-adds `hasThinkDirective` and `hasVerboseDirective` after clearing a mixed-content message. Those flags later make the directive path eligible for fast-lane handling and persistence, while the docs say inline `/think` and `/verbose` affect only the current message. Preserve the run-level values without marking them as session-default directives, or explicitly skip persistence for this preserved mixed-inline case.
