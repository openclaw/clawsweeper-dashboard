## Summary

Keep open: the reported directive-loss bug is source-reproducible on current main, but this PR still restores the session-persistence flags and lacks required after-fix real behavior proof.

## What This Changes

The PR captures parsed think/verbose levels before mixed inline directive clearing and re-injects them for authorized senders in `resolveReplyDirectives`.

## Best Possible Solution

Carry mixed-content think/verbose values into the current run only, keep persistence flags false or explicitly bypass persistence, and add regression coverage for run application plus non-persistence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: send a normal authorized message containing real text plus `/think <level>` or `/verbose <level>` and current main clears the parsed level before run resolution.

## Solution Assessment

No. Preserving the levels is the right direction, but restoring `hasThinkDirective` and `hasVerboseDirective` is not the best fix because those flags also drive session persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep preserved inline levels out of session defaults:** `src/auto-reply/reply/get-reply-directives.ts:298-305`
  - body: This re-adds `hasThinkDirective` and `hasVerboseDirective` after clearing a mixed-content message. Those flags later make the directive path eligible for fast-lane handling and persistence, while the docs say inline `/think` and `/verbose` affect only the current message. Preserve the run-level values without marking them as session-default directives, or explicitly skip persistence for this preserved mixed-inline case.
  - confidence: 0.9
