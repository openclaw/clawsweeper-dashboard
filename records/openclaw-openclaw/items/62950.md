## Summary

Keep open: the underlying directive-loss bug is still source-reproducible on current main, but this PR is not mergeable as-is because it reuses session-persistence flags for per-message think/verbose values and lacks after-fix real behavior proof.

## What This Changes

The PR modifies directive resolution so mixed-content messages retain parsed think/verbose levels after inline directive syntax is cleared.

## Best Possible Solution

Carry authorized mixed-content think/verbose values into only the current run, avoid marking them as session-default directives, and add regression coverage for run-level application plus non-persistence.

## Reproduction Assessment

Yes. Source inspection shows a normal message with real text plus `/think <level>` or `/verbose <level>` reaches clearInlineDirectives on current main, loses the parsed level, and then falls back during run resolution.

## Solution Assessment

No. Preserving the values is the right direction, but restoring hasThinkDirective/hasVerboseDirective is not the best fix because those flags also drive session persistence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep preserved inline levels out of session defaults:** `src/auto-reply/reply/get-reply-directives.ts:298-305`
  - body: This re-adds `hasThinkDirective` and `hasVerboseDirective` after clearing a mixed-content message. Those flags are not only run-level markers: the directive path later calls the fast-lane handler and persistence helper, both of which can write session defaults from these flags, while docs say inline `/think` and `/verbose` affect only that message. Preserve the run-level values without setting persistence flags, or make the preserved mixed-inline case explicitly non-persistent.
  - confidence: 0.9
