## Summary

Keep open. Current main partially covers the area with per-assistant-message token/cost metadata, a high-context warning, a Usage page, and provider-usage APIs, but it does not implement the requested persistent chat header quota/cost widget or the named OpenRouter/LLMAPI/Ollama quota states. The related open PR is useful but explicitly treats this issue as broader follow-up scope rather than closing it.

## What This Changes

Request to add a prominent Control UI chat token-usage widget with session totals, remaining quota/cost, per-message usage, and provider-specific quota handling for OpenRouter, LLMAPI, and Ollama.

## Best Possible Solution

Keep this issue open as the broader chat usage/quota UX request. The best end state is a maintainer-scoped Control UI chat status surface that composes fresh session context totals, per-response metadata, permitted cost display, and provider quota availability with clear unavailable/unlimited/local states, while reusing existing session/provider usage APIs and avoiding duplicate status bars across the Usage page, Overview card, and #73744.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current source/docs/tests provide a high-confidence gap check: per-message metadata and warning/report surfaces exist, but the persistent chat header quota/cost widget and requested provider quota support do not.

## Solution Assessment

Unclear for a single implementation path. The requested direction is valid, but the best fix should be scoped by maintainers because it combines chat UI placement, provider quota APIs, cost permissions, and overlapping open status/usage requests.
