## Summary

Keep open: the bug is valid and the mirror-credit direction is right, but the current predicate can still credit the wrong requester mirror when identical text appears from another current-window run.

## What This Changes

The branch adds a subagent lifecycle check for matching requester-session delivery mirrors and regression coverage so completion cleanup can treat a prior visible reply as delivered when the parent announce returns NO_REPLY.

## Best Possible Solution

Bind mirror credit to durable current-run evidence, preferably the delivery mirror idempotency/run metadata, with timestamp and text checks as supporting evidence plus same-window duplicate coverage.

## Reproduction Assessment

Yes, source-reproducible: current main finalizes cleanup from the parent announce didAnnounce result, and the linked Slack report shows retry-limit after a requester delivery mirror already existed. I did not run a live current-main Slack repro in this read-only pass.

## Solution Assessment

No, not yet: the mirror-credit direction is the right small fix, but text plus a run-window timestamp is not enough identity for repeated completions. The safer solution is to require mirror metadata tied to this child run.
