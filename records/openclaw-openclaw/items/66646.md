## Summary

Keep open. Current main contains the typed lock-timeout classifier fix from #68700, but the central fallback loop still records and advances unclassified errors while more candidates remain, and each agent fallback attempt acquires the same session write lock inside that loop. The latest issue comments also show current agent/Discord paths still hit `SessionWriteLockTimeoutError` under contention, so the cascade stop remains a narrow actionable core bug.

## What This Changes

The issue asks OpenClaw to treat session write-lock contention as infrastructure/session-busy instead of a model failure that advances through fallback candidates.

## Best Possible Solution

Add a narrow session-busy/infrastructure escape before model fallback records a failed candidate or advances to the next model when the error is a session write-lock timeout. Preserve generic unknown-error fallback, live-session model-switch behavior, and existing stale-lock cleanup policy; queue/backoff can be handled separately if maintainers want that product behavior.

## Reproduction Assessment

Yes. A focused unit reproduction can call `runWithModelFallback` with a two-candidate config and make the first callback throw `SessionWriteLockTimeoutError`; current source shows it will call the next candidate because the error is unclassified and candidates remain.

## Solution Assessment

No. The current classifier-only solution is incomplete because returning `null` still flows into the generic unclassified-error fallback path; the safer repair is an explicit session-lock branch at the fallback wrapper or agent-attempt boundary.
