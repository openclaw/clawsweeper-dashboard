## Summary

Keep open. Current main has the partial classifier fix from #68700, but direct session lock timeouts still become unclassified errors and `runWithModelFallback` continues to later candidates, so the reported cascade remains valid.

## What This Changes

The issue asks for session file lock timeouts to fail fast as session-busy infrastructure errors instead of consuming model fallback candidates.

## Best Possible Solution

Add a generic session-lock infrastructure stop before model fallback records a failed candidate or advances, while leaving stale-lock reclaim and queue/backoff policy to separate focused work.

## Reproduction Assessment

Yes. Configure a multi-candidate fallback chain and make the first candidate callback throw `SessionWriteLockTimeoutError`; source and existing tests show current main will treat it as unknown and try the next candidate.

## Solution Assessment

Yes on the requested direction, but the current classifier-only implementation is incomplete because unclassified direct lock errors still flow through the generic fallback path.
