## Summary

Keep open. Current main no longer classifies typed session write-lock timeouts as provider timeouts, but the model fallback runner still advances unclassified errors to later candidates, so direct lock contention can still cascade through the fallback chain described in the report.

## What This Changes

The issue asks OpenClaw to treat session write-lock contention as local infrastructure/session-busy pressure instead of a model failure that advances through configured fallback models.

## Best Possible Solution

Keep this as the canonical core agents bug and add a narrow infrastructure/session-busy escape before model fallback records a failed candidate or advances. Generic unknown-error fallback should remain intact, stale-lock reclaim policy should not change, and queue/backoff can be handled separately if maintainers want that behavior.

## Reproduction Assessment

Yes. Configure a multi-candidate fallback chain and make the first run callback throw SessionWriteLockTimeoutError; current main will suppress timeout classification but still call the next candidate because the error is unclassified and candidates remain.

## Solution Assessment

No. The current classifier-only solution is incomplete because returning null still flows through the generic unknown-error fallback path; the safer solution is an explicit session-lock/infrastructure branch at the fallback wrapper or agent-attempt boundary.
