## Summary

Keep this PR open, but the current import-only patch is not sufficient: production status summary still passes `cfg` and `allowAsyncLoad: false`, while the shared resolver skips runtime cache loading whenever `cfg` is supplied, so the claimed discovery-cache fix does not run on the cold CLI status path.

## What This Changes

The PR changes `src/commands/status.summary.runtime.ts` to export `resolveContextTokensForModel` from `src/agents/context.ts` instead of aliasing the cache-only resolver from `src/agents/context-resolution.ts`.

## Best Possible Solution

Land a focused status/context fix that preserves the read-only status contract while resolving discovered or prepared model context windows, then verify it with a real redacted CLI status run before closing the linked issue.

## Reproduction Assessment

Yes, from source inspection rather than an executed CLI run. The production status path passes `cfg` and `allowAsyncLoad: false`, while the imported resolver skips runtime cache loading when `cfg` is present.

## Solution Assessment

No. Importing the shared resolver is the right direction, but this PR alone does not change the production cold-cache path; the fix needs to address the status caller/cache-warming contract and include real CLI proof.
