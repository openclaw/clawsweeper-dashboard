## Summary

Keep open: current main still awaits Twitch `disconnectAll()` before deleting the registry entry, so a thrown cleanup path can leave a stale manager registered. I found no duplicate or linked fix for this removal path; the adjacent open PR only covers test registry clearing for a related issue.

## What This Changes

The issue requests guaranteed cleanup of the Twitch client-manager registry when `removeClientManager` sees `disconnectAll()` throw or reject.

## Best Possible Solution

Move the registry deletion for `removeClientManager` into guaranteed cleanup, preserve the original disconnect error signal, and add focused regression coverage for failed disconnect cleanup.

## Reproduction Assessment

Yes, source-level: create a Twitch manager for an account, make that manager's `disconnectAll()` throw or reject, call `removeClientManager`, catch the error, and `getClientManager` or `getOrCreateClientManager` can still observe the stale entry on current main. I did not execute this because the review is read-only.

## Solution Assessment

Yes. A `try/finally` around the disconnect with `registry.delete(accountId)` in the finalizer is the narrowest maintainable fix, provided the original disconnect error still propagates.
