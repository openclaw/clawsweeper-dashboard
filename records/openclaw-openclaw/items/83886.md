## What This Changes

The issue asks to guarantee Twitch client-manager registry eviction when `removeClientManager` observes a throwing or rejecting `disconnectAll`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84375 if its proof and checks satisfy maintainers; after merge, close this issue as implemented by that PR.

## Reproduction Assessment

Yes, source-level: create a Twitch manager through `getOrCreateClientManager`, make that manager's `disconnectAll` reject, call `removeClientManager`, and current main skips `registry.delete(accountId)` before a later `getClientManager`/`getOrCreateClientManager` lookup. I did not run the repro because this review is read-only.

## Solution Assessment

Yes; putting `registry.delete(accountId)` in a `finally` block is the narrow maintainable fix as long as the original disconnect error still propagates. The linked PR follows that shape and is the right current implementation path to review.
