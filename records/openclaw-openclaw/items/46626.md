## Summary

Keep open: this is a real silent-reply leak class and current main still has source-reproducible exact-only gaps, but this PR is not mergeable as-is because it has correctness blockers, missing real behavior proof, and overlapping open proposals for the canonical `NO_REPLY` semantics.

## What This Changes

The branch adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron delivery, subagent follow-up, gateway live chat/history projection, and related tests.

## Best Possible Solution

Choose one canonical, cached, case-insensitive silent-reply classifier, apply it consistently through normalization, directive parsing, cron/subagent delivery, live/history projection, and runner handling as appropriate, while preserving media/channelData and legitimate inline mentions.

## Reproduction Assessment

Yes, source-reproducible. Current main still has exact/shared-helper checks in directive parsing, subagent follow-up, and live chat projection, while the normalizer test still proves `summary\n\nNO_REPLY` is stripped and delivered rather than suppressed.

## Solution Assessment

No. The branch covers useful paths, but it should not merge as-is because the parser test fails, own-line matching is case-sensitive, channelData can be dropped, and maintainers still need to settle the canonical silent-reply semantics across overlapping open PRs.
