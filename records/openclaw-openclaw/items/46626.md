## Summary

Keep open: current main has only partially addressed this class of silent-token leaks, while this branch still has correctness blockers, lacks inspectable real behavior proof, and overlaps competing open silent-reply proposals that need maintainer policy selection.

## What This Changes

The PR adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron delivery, sub-agent follow-up, gateway streaming/history filtering, and tests.

## Best Possible Solution

Adopt one documented, case-insensitive silent-reply classifier across normalization, outbound planning, cron, gateway display/history, and runner handling while preserving media, channelData, and legitimate inline mentions.

## Reproduction Assessment

Yes for the current behavior: source inspection shows current main strips trailing own-line `NO_REPLY` and returns the preamble, and a current regression test asserts that behavior. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction may be useful, but this branch needs parser/test fixes, case-insensitive matching, channelData preservation, real behavior proof, and a maintainer policy choice across related alternatives.

  - confidence: 0.8

  - body: `isSilentTokenOnOwnLine` compiles a new `RegExp` on every call, and this PR calls it from streaming and cron paths. Mirror the existing cached token regex helpers to avoid per-chunk allocation pressure.
  - confidence: 0.74
