## Summary

Keep open: current main does not implement the PR’s intended own-line silent suppression, and this branch still needs reconciliation with overlapping silent-reply work plus several blocking correctness fixes.

## What This Changes

The PR adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron/subagent delivery, gateway live/history projection, and tests.

## Best Possible Solution

Define one shared, cached, case-insensitive silent-reply classifier, then apply it consistently through normalization, outbound planning, cron/subagent follow-up, gateway live/history projection, and runner handling after the silent-token semantics are settled.

## Reproduction Assessment

Yes, source-reproducible. Current main still strips trailing `NO_REPLY` while delivering the preamble in `normalizeReplyPayload`, keeps exact/JSON-only directive parsing, and has exact-only subagent follow-up checks.

## Solution Assessment

No. The branch targets real leak paths, but it should not merge as-is because the inline parser test fails, own-line matching is case-sensitive, channelData can be dropped, and overlapping PRs need a single maintainer-approved contract.

  - confidence: 0.97

  - body: Existing silent-token helpers accept mixed-case `no_reply`, but this new regex has no `i` flag. A payload like `summary\nNo_Reply` can still leak through every newly wired own-line path.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`

  - confidence: 0.84

  - body: The new helper compiles a fresh `RegExp` on every call even though the PR adds it to streaming and polling paths. Cache it like the neighboring silent-token regex helpers to avoid per-chunk allocation.
