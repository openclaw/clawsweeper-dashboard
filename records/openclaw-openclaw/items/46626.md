## Summary

Keep open. Current main does not fully implement this PR's end-to-end own-line `NO_REPLY` suppression, but the PR is not merge-ready because it adds a failing parser assertion, misses existing case-insensitive token semantics, can drop channelData-only payloads, and overlaps with newer silent-reply work that needs maintainer consolidation.

## What This Changes

The PR adds own-line `NO_REPLY` suppression through reply normalization, directive parsing, cron/subagent delivery, gateway live/history filtering, and related tests.

## Best Possible Solution

Define one shared, cached, case-insensitive silent-reply classifier and apply it consistently through parser, normalization, outbound planning, cron/subagent follow-up, and gateway live/history projection while preserving inline text, media, interactive payloads, and channelData.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main still treats several own-line `NO_REPLY` payloads as non-silent outside newer cron handling, and the PR branch's inline parser test fails against its own parser logic.

## Solution Assessment

No. The PR is useful source work, but it is incomplete and stale; the safer path is to consolidate the silent-reply contract with the related open issue and PRs before merging or replacing this branch.

  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but this patch only clears text for exact or own-line silent replies. Inline text returns unchanged, so the test fails unless the parser strips inline tokens or the expectation is corrected.
  - confidence: 0.97

  - body: Existing silent-token helpers match case-insensitively, and current cron coverage includes mixed-case `No_Reply`. The new own-line regex has no `i` flag, so mixed-case own-line tokens still leak through the new suppression paths.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning drop channelData-only payloads in allow-policy paths because it checks only media before treating the payload as silent. That can discard channel-specific cards or custom payloads instead of sending them with empty text.
  - confidence: 0.84
- **[P3] Cache the own-line token regex:** `src/auto-reply/tokens.ts:64`
  - body: The new helper is called from streaming and polling paths but allocates a fresh `RegExp` on every call. Cache it like the neighboring silent-token regex helpers to avoid per-chunk allocation in hot paths.
  - confidence: 0.79
