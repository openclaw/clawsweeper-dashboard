## Summary

Keep open: current main and the latest release still preserve text before a trailing own-line `NO_REPLY` in the shared normalizer/gateway display path, but this branch is not merge-ready because it changes unsettled silent-reply policy, lacks inspectable real behavior proof, and introduces concrete parser, case-sensitivity, and channelData regressions.

## What This Changes

The PR adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron delivery, sub-agent follow-up, gateway streaming/history filtering, and tests.

## Best Possible Solution

Land or replace with one documented, case-insensitive silent-reply classifier shared across normalization, outbound planning, cron, gateway display/history, and runner handling while preserving media, channelData, and legitimate inline token mentions.

## Reproduction Assessment

Yes at source level: current main strips trailing own-line `NO_REPLY` and returns the preamble from `normalizeReplyPayload`, and current tests assert that behavior. I did not run runtime tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction may be useful, but this branch needs parser/test fixes, case-insensitive matching, channelData preservation, real behavior proof, and a maintainer decision across related alternatives.

  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the changed parser only clears exact or own-line silent payloads. Inline text is returned unchanged, so this assertion fails unless the parser strips inline tokens or the test is corrected.
  - confidence: 0.96
- **[P2] Match own-line silent tokens case-insensitively:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentReplyText` and the docs accept `no_reply`, but the new own-line helper compiles a case-sensitive regex and the normalizer guard checks `text.includes(silentToken)`. A payload like `summary\nno_reply` would still pass through instead of being treated like the silent token.
  - confidence: 0.9
- **[P2] Preserve channelData when marking own-line replies silent:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line tokens as silent here makes outbound planning treat no-media payloads as silent before `channelData` affects visibility. A channelData-only reply can be dropped or rewritten instead of sending its custom payload content.
  - confidence: 0.82
- **[P3] Cache the own-line silent-token regex:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentTokenOnOwnLine` compiles a new `RegExp` on every call, and this PR calls it from streaming and cron paths. Mirror the existing cached token regex helpers to avoid per-chunk allocation pressure.
  - confidence: 0.74
