## Summary

Keep open: the PR targets a source-reproducible silent-token leak class, but the branch still has correctness blockers, missing real behavior proof, and an unresolved silent-reply policy choice across related alternatives.

## What This Changes

The branch adds an own-line `NO_REPLY` classifier and wires it through reply normalization, directive parsing, cron delivery/follow-up, gateway chat streaming/history, and related tests.

## Best Possible Solution

Define one shared silent-reply classifier and policy, document it, then land a rebased implementation across normalization, outbound planning, cron/subagent handling, gateway projection/history, and runner paths while preserving media, channelData, and legitimate inline token mentions.

## Reproduction Assessment

Yes at source level: current main strips a trailing own-line `NO_REPLY` and returns the preamble from `normalizeReplyPayload`, and current tests assert that behavior. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction may be useful, but the branch needs parser/test fixes, case-insensitive matching, channelData preservation, a rebase over current cron/gateway changes, real behavior proof, and a maintainer policy decision across related alternatives.
