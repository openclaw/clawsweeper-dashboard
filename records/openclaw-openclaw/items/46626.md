## Summary

Keep open: the underlying silent-token leak is still only partially handled on current main, but this PR is not merge-ready because it has a failing test expectation, inconsistent case matching, no real behavior proof artifact, and conflicts with newer silent-reply work.

## What This Changes

The PR adds an own-line `NO_REPLY` detector and wires it into reply normalization, directive parsing, cron delivery/subagent follow-up, gateway streaming, chat-history filtering, and related tests.

## Best Possible Solution

Adopt one shared, case-insensitive silent-reply classifier across normalizer, outbound, gateway, and cron paths, while preserving legitimate inline mentions, media, channel data, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main still delivers the preamble in `normalizeReplyPayload` and does not suppress own-line `NO_REPLY` in live gateway projection, although I did not run tests in this read-only review.

## Solution Assessment

No, not as submitted. The direction is plausible, but the PR needs the failing test fixed, case-insensitive matching, conflict resolution against newer main behavior, and maintainer agreement on the canonical silent-token policy.
