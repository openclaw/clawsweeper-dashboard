## Summary

Keep open: current main still has the central silent-token leak in non-cron reply and gateway paths, but this PR is not merge-ready because it has correctness issues, lacks inspectable real behavior proof, and overlaps several active policy alternatives.

## What This Changes

The PR adds an own-line `NO_REPLY` helper and wires it through reply normalization, directive parsing, cron delivery, sub-agent follow-up, gateway streaming/history filtering, and tests.

## Best Possible Solution

Adopt one documented, case-insensitive silent-token classifier and apply it consistently across normalization, outbound planning, cron, gateway display/history, and runner handling while preserving media, channelData, and legitimate inline mentions.

## Reproduction Assessment

Yes. Source inspection shows current main strips trailing own-line `NO_REPLY` and returns the preamble, and the current regression test asserts that behavior; I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction may be useful, but the PR needs parser/test fixes, channelData preservation, case-insensitive cached matching, real behavior proof, and a maintainer policy choice across related alternatives.

- **[P2] Fix the inline NO_REPLY parser expectation:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the changed parser only clears exact or own-line silent payloads. Inline text is returned unchanged, so this assertion fails unless the parser strips inline tokens or the test is corrected.
  - confidence: 0.96
- **[P2] Match own-line silent tokens case-insensitively:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentReplyText` and the docs accept `no_reply`, but the new own-line helper compiles a case-sensitive regex. A payload like `summary\nno_reply` would still pass through the new delivery paths instead of being treated like the silent token.
  - confidence: 0.9
- **[P2] Preserve channelData for own-line silent replies:** `src/auto-reply/reply/reply-directives.ts:34`
  - body: Marking own-line tokens as silent here makes outbound planning treat no-media payloads as silent before `channelData` affects visibility. A channelData-only reply can be dropped or rewritten instead of sending its custom payload content.
  - confidence: 0.8
- **[P3] Cache the own-line silent-token regex:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentTokenOnOwnLine` compiles a new `RegExp` on every call, and this PR calls it from streaming and cron paths. Mirror the existing cached token regex helpers to avoid per-chunk allocation pressure.
  - confidence: 0.74
