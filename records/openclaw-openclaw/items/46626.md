## Summary

Keep open. The PR targets a real silent-reply leak family that current main has not fully resolved across parser, live chat/history, and subagent follow-up paths, but this branch is stale against newer silent-reply work and has merge-blocking correctness issues.

## What This Changes

The PR adds own-line `NO_REPLY` suppression and wires it through reply normalization, directive parsing, cron/subagent delivery, gateway live chat/history filtering, and regression tests.

## Best Possible Solution

Consolidate the silent-reply contract into a shared, cached, case-insensitive detector on current main, then apply it consistently to normalization, outbound planning, subagent follow-up, live projection, and history display while preserving inline-token delivery, media, interactive payloads, JSON envelopes, direct rewrite policy, and channelData semantics.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current tests still expect `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` to return deliverable preamble text, and parser/live/history paths are exact-control-token based rather than own-line aware.

## Solution Assessment

No. The PR points in a useful direction, but this branch is not the best mergeable solution until it is rebased and reconciled with newer cron fixes, open silent-reply PRs, case-insensitive matching, inline parser expectations, and channelData preservation.

## Review Findings

Full review comments:

- **[P2] Align the inline-token test with parser behavior:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the patch only clears text for exact or own-line silent replies. Inline text returns unchanged, so this test fails unless the parser strips inline tokens or the expectation is corrected.
  - confidence: 0.97
- **[P2] Make own-line token matching case-insensitive:** `src/auto-reply/tokens.ts:64`
  - body: Existing silent-token helpers match mixed-case variants, and current cron coverage includes mixed-case `NO_REPLY`. The new own-line regex lacks the `i` flag, so `summary\nNo_Reply` can still leak on paths that rely on this helper.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning treat channelData-only payloads as silent because it only excludes media from silence. Group/internal allow-policy paths can then drop channel-specific data instead of sending it with empty text.
  - confidence: 0.84
- **[P3] Cache the own-line token regex:** `src/auto-reply/tokens.ts:64`
  - body: This helper compiles a new `RegExp` on every call while the patch adds it to streaming and polling paths. Cache it like the neighboring silent-token regex helpers to avoid avoidable per-chunk allocation.
  - confidence: 0.79
