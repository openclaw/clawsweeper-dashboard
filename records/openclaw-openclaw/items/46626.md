## Summary

Keep open: current main only partially covers the silent-token leak class, and this PR still targets real remaining paths, but it is not mergeable as-is because of correctness findings, missing real behavior proof, and an unresolved maintainer policy choice across overlapping silent-reply proposals.

## What This Changes

The branch adds own-line `NO_REPLY` suppression and applies it across reply normalization, directive parsing, cron delivery/follow-up, gateway streaming/history filtering, and related tests.

## Best Possible Solution

Adopt one documented, case-insensitive silent-reply classifier across the normalizer, outbound plan, cron, gateway display/history, and runner paths while preserving media, channelData, and legitimate inline `NO_REPLY` mentions.

## Reproduction Assessment

Yes. Source inspection of current main shows the normalizer strips trailing own-line `NO_REPLY` and returns the preamble, and the current regression test asserts that behavior; I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction may be useful, but the PR needs parser/test fixes, case-insensitive cached matching, channelData preservation, real behavior proof, and maintainer selection of the canonical silent-token policy.

  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the changed parser only clears exact or own-line silent payloads. Inline text is returned unchanged, so this assertion fails unless the parser strips inline tokens or the test is corrected.
  - confidence: 0.96
- **[P2] Match own-line silent tokens case-insensitively:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentReplyText` and the docs accept `no_reply`, but the new own-line helper compiles a case-sensitive regex. A payload like `summary\nno_reply` would still pass through these new delivery paths instead of being treated like the silent token.
  - confidence: 0.9
- **[P2] Preserve channelData for own-line silent replies:** `src/auto-reply/reply/reply-directives.ts:34`
  - body: Marking own-line tokens as silent here makes outbound planning treat no-media payloads as silent before `channelData` affects visibility. A channelData-only reply can be dropped or rewritten instead of sending its custom payload content.
  - confidence: 0.8
- **[P3] Cache the own-line silent-token regex:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentTokenOnOwnLine` compiles a new `RegExp` on every call, and this PR calls it from streaming and cron paths. Mirror the existing cached token regex helpers to avoid per-chunk allocation pressure.
  - confidence: 0.74
