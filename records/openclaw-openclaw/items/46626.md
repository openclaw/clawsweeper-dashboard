## Summary

Keep open. The user-facing leak is still real on current main, but this PR branch is not mergeable as-is: it has a failing added test, incomplete mixed-case handling, and a channelData regression risk, while newer main already supersedes part of the cron work.

## What This Changes

The PR adds own-line NO_REPLY suppression and wires it through reply normalization, directive parsing, cron/subagent delivery, gateway live/history filtering, and regression tests.

## Best Possible Solution

Define one shared, cached, case-insensitive silent-token contract, then apply it through normalization, directives, outbound planning, subagent follow-up, live/history projection, and tests while preserving inline text, media, interactive payloads, and channelData.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main still has tests and code paths that deliver preamble text before a trailing own-line `NO_REPLY`, and the PR branch's new inline parser assertion fails by direct code inspection.

## Solution Assessment

No. The PR is useful source work, but it should be consolidated with the newer silent-reply fixes and current-main refactors before merge.

- **[P2] Align the inline-token parser test with the parser behavior:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the patch only clears text for exact or own-line silent replies. Inline text returns unchanged, so this assertion fails unless the parser also strips inline tokens or the expectation is corrected.
  - confidence: 0.97
- **[P2] Make own-line silent matching case-insensitive:** `src/auto-reply/tokens.ts:64`
  - body: Existing silent-token helpers match case-insensitively, and current cron coverage includes mixed-case `No_Reply`. The new own-line regex lacks the `i` flag, so `summary\nNo_Reply` still misses the new suppression paths and can leak.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning treat channelData-only payloads as silent because it only excludes media from silence. In allow-policy paths this can drop channel-specific data instead of sending it with empty text.
  - confidence: 0.84
- **[P3] Cache the own-line token regex:** `src/auto-reply/tokens.ts:64`
  - body: This helper is added to streaming and polling paths but compiles a new `RegExp` on every call. Cache it like the neighboring silent-token regex helpers to avoid avoidable per-chunk allocation.
  - confidence: 0.79
