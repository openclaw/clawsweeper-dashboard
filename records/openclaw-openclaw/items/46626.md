## Summary

Keep open: current main still does not implement the full own-line/trailing `NO_REPLY` policy across the main normalizer and gateway display path, but this PR is not mergeable as-is because it has correctness issues, lacks inspectable real behavior proof, and overlaps several competing open policy proposals.

## What This Changes

The PR extends own-line `NO_REPLY` suppression across reply normalization, directive parsing, cron delivery/followup, gateway streaming, chat history filtering, and adds focused tests.

## Best Possible Solution

Choose one canonical silent-token policy and implement it through a shared classifier used by normalization, outbound planning, cron, gateway display/history, and runner handling while preserving media, channelData, and legitimate inline mentions.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main strips `summary\n\nNO_REPLY` in the main normalizer while preserving the preamble, and gateway display projection still suppresses exact control-token text only. I did not execute tests because this review is read-only.

## Solution Assessment

No, not as-is. The suppression direction is plausible, but this branch needs the inline parser expectation fixed, own-line matching made case-insensitive and cached, channelData semantics preserved, contributor proof added, and policy reconciled with the related open alternatives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Fix the inline NO_REPLY expectation:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the changed parser only clears exact or own-line silent payloads. Inline text is returned unchanged, so this assertion will fail unless the parser strips inline tokens or the test is corrected.
  - confidence: 0.97
- **[P2] Make own-line matching case-insensitive:** `src/auto-reply/tokens.ts:64-65`
  - body: The existing silent-token helpers match case-insensitively, but `isSilentTokenOnOwnLine` builds a regex without the `i` flag. Mixed-case outputs such as `text\nNo_Reply` would still leak through every new own-line guard added by this PR.
  - confidence: 0.88
- **[P2] Preserve channelData on own-line silent directives:** `src/auto-reply/reply/reply-directives.ts:34`
  - body: Marking own-line tokens as `isSilent` here makes outbound planning treat no-media payloads as silent before `channelData` affects visibility. A channelData-only reply can be dropped or rewritten instead of sending its custom payload content.
  - confidence: 0.8
- **[P3] Cache the own-line silent-token regex:** `src/auto-reply/tokens.ts:64-65`
  - body: `isSilentTokenOnOwnLine` compiles a new `RegExp` on every call, and this PR calls it from streaming and cron paths. Please mirror the existing cached token regex helpers to avoid per-chunk allocation pressure.
  - confidence: 0.74
