## Summary

Keep open for maintainer review: current main and the latest release still do not implement the PR's broad own-line `NO_REPLY` classifier across all paths, but this branch is not merge-ready because it has blocking review findings, no inspectable real behavior proof, and overlaps unresolved silent-reply policy work.

## What This Changes

The PR adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron delivery/subagent follow-up, gateway streaming/history filtering, and tests.

## Best Possible Solution

Choose one documented silent-reply boundary and land or replace this with a shared case-insensitive implementation across normalization, outbound planning, cron, gateway, and runner paths while preserving media, channelData, and legitimate inline token mentions.

## Reproduction Assessment

Yes, at source level. Current main turns `summary\n\nNO_REPLY` into deliverable preamble text in `normalizeReplyPayload`, current tests assert that behavior, and the gateway live projector lacks an own-line classifier.

## Solution Assessment

No, not as-is. The direction may be useful, but this branch needs parser/test alignment, case-insensitive matching, channelData preservation, real behavior proof, and a maintainer decision on silent-token policy before merge.
