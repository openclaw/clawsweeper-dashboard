## Summary

Keep open: the PR targets a real silent-reply leak class that is not fully solved on current main, but it still has blocking correctness issues, missing inspectable real behavior proof, and an unresolved policy choice across related open alternatives.

## What This Changes

The PR adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron delivery/followup, gateway streaming/history filtering, and focused tests.

## Best Possible Solution

Choose one shared silent-token classifier and policy, apply it consistently across normalization, outbound planning, cron, gateway display/history, and runner handling, while preserving media/channelData and legitimate inline mentions.

## Reproduction Assessment

Yes. Source inspection and current tests show the main normalizer still strips a trailing own-line `NO_REPLY` while preserving the preamble, and the gateway display path still suppresses exact control tokens only; I did not execute tests because this review is read-only.

## Solution Assessment

No, not as-is. The suppression direction is plausible, but this branch needs the inline expectation fixed, channelData semantics preserved, regex allocation addressed, contributor proof added, and policy reconciled with related open alternatives.
