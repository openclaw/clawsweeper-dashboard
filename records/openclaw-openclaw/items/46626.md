## Summary

Keep open: current main still does not provide the PR's end-to-end own-line silent-reply suppression, but this branch is not merge-ready because it has concrete correctness findings and overlaps newer silent-reply proposals with different contracts.

## What This Changes

The PR adds own-line `NO_REPLY` suppression across reply normalization, directive parsing, cron/subagent delivery, gateway live/history filtering, and related tests.

## Best Possible Solution

Define one shared, cached, case-insensitive silent-reply classifier and apply it consistently through tokens, reply directives, normalization, outbound planning, cron/subagent follow-up, gateway live/history projection, and runner handling with current-main regression coverage.

## Reproduction Assessment

Yes. Source inspection is high confidence: current main still uses exact/JSON silent-reply classification in shared helpers and gateway projection, while existing normalization coverage expects trailing-token preamble text to be delivered after stripping.

## Solution Assessment

No. The branch is useful source work, but the maintainable fix should first settle the shared silent-reply contract and then rebase the implementation across current-main reply, outbound, cron, gateway, and runner seams.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Align the inline-token parser test with parser behavior:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the patch only clears text for exact or own-line silent replies. Inline text returns unchanged, so this assertion fails unless the parser strips inline tokens or the expectation is corrected.
  - confidence: 0.97
- **[P2] Make own-line silent matching case-insensitive:** `src/auto-reply/tokens.ts:64`
  - body: Existing silent-token helpers match case-insensitively, and current cron coverage includes mixed-case `No_Reply`. The new own-line regex has no `i` flag, so `summary\nNo_Reply` can still miss the new suppression paths.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning drop silent entries under allow-policy paths without checking `channelData`. That can discard channel-specific payloads that should be sent with empty text.
  - confidence: 0.84
- **[P3] Cache the own-line token regex:** `src/auto-reply/tokens.ts:64`
  - body: The new helper compiles a fresh `RegExp` on every call even though it is added to streaming and polling paths. Cache it like the neighboring silent-token regex helpers to avoid per-chunk allocation.
  - confidence: 0.79
