## Summary

Keep open: current main has not implemented the PR’s own-line silent-reply contract across the requested paths, but the proposed branch is still not merge-ready because it conflicts with existing silent-reply semantics and has concrete review blockers.

## What This Changes

The PR adds an own-line `NO_REPLY` classifier and wires it into reply normalization, directive parsing, cron/subagent delivery, gateway live/history projection, and related tests.

## Best Possible Solution

Settle one shared silent-reply classifier contract, implement it centrally in the auto-reply token/normalization layer, then apply it consistently through outbound delivery, cron/subagent follow-up, gateway live/history projection, and runner handling with regression tests and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible. Current main still has exact/JSON-only shared classification plus exact-only subagent follow-up checks, so `text\n\nNO_REPLY` can remain visible in paths this PR targets.

## Solution Assessment

No. The branch identifies real gaps, but it should not merge as-is because the inline parser test fails, own-line matching is case-sensitive, channelData can be dropped, and maintainers still need to choose the canonical silent-reply contract.
