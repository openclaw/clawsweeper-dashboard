## Summary

Keep open: the central NO_REPLY suppression behavior is not implemented consistently on current main, but this PR is not mergeable as-is because its patch has blocking parser/classifier regressions and lacks inspectable real behavior proof.

## What This Changes

The PR adds an own-line NO_REPLY helper and wires it through reply normalization, directive parsing, cron/subagent delivery, gateway chat projection/history, and tests.

## Best Possible Solution

Choose one shared, documented, case-insensitive silent-reply classifier and apply it consistently across normalization, outbound planning, cron/subagent handling, gateway display/history, and runner behavior while preserving media, channelData, and legitimate inline token mentions.

## Reproduction Assessment

Yes, at source level. Current main still turns `summary\n\nNO_REPLY` into deliverable preamble text in the shared normalizer, and current tests assert that strip-and-deliver behavior; I did not run runtime tests because this review is read-only.

## Solution Assessment

No, not as-is. The direction is plausible, but this branch needs parser/test fixes, case-insensitive matching, channelData preservation, contributor real behavior proof, and maintainer alignment with the overlapping canonical work.

- **[P2] Fix the inline NO_REPLY parser expectation:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but the changed parser only clears exact or own-line silent payloads. Inline text is returned unchanged, so this assertion fails unless the parser strips inline tokens or the test is corrected.
  - confidence: 0.96
- **[P2] Keep own-line silent tokens case-insensitive:** `src/auto-reply/tokens.ts:64-65`
  - body: Current helpers and docs accept both `NO_REPLY` and `no_reply`, but the new own-line helper uses a case-sensitive regex and the normalizer guard checks `text.includes(silentToken)`. A payload like `summary\nno_reply` can still leak instead of being treated as silent.
  - confidence: 0.9
- **[P2] Preserve channelData when marking own-line replies silent:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line tokens as `isSilent` here makes outbound planning drop no-media payloads before `channelData` contributes to renderability. A channelData-only reply can be discarded instead of sending its custom payload content with empty text.
  - confidence: 0.82
