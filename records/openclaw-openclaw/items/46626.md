Keep open: this is a plausible silent-reply leak fix, but it should not merge as-is because current main only partially covers the cron path, the branch has blocking parser/helper regressions, real behavior proof is not attached, and several open PRs still require a maintainer decision on the canonical silent-token contract.

## What This Changes

The branch adds an own-line `NO_REPLY` helper and wires it through reply normalization, directive parsing, cron/subagent delivery, gateway live/history filtering, and tests.

## Best Possible Solution

Choose one shared, cached, case-insensitive silent-reply classifier and apply it consistently through normalization, final delivery planning, cron/subagent follow-up, gateway live/history projection, and runner handling while preserving legitimate inline mentions and channelData payloads.

## Reproduction Assessment

Yes. Source inspection shows current main still strips and delivers `summary\n\nNO_REPLY` in the main normalizer, and subagent follow-up plus gateway live projection do not classify own-line trailing sentinels; I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The branch covers useful paths, but it introduces a failing inline parser test, a case-sensitive hot-path helper, possible channelData loss, and overlaps https://github.com/openclaw/openclaw/pull/42873, https://github.com/openclaw/openclaw/pull/69394, and https://github.com/openclaw/openclaw/pull/73153, so maintainers should first choose the canonical policy.
