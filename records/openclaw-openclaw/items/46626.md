## Summary

Keep this PR open, but do not merge it as-is. Current main still has source-reproducible silent-reply leak paths that this PR is trying to address, while the branch has correctness blockers, missing real behavior proof, and overlaps several open silent-reply policy fixes.

## What This Changes

The PR extends own-line NO_REPLY suppression across reply normalization, directive parsing, cron/subagent delivery, gateway live/history filtering, and tests.

## Best Possible Solution

Choose one shared, cached, case-insensitive silent-reply classifier and apply it consistently through normalization, outbound planning, cron/subagent follow-up, gateway live/history projection, and runner handling while preserving legitimate inline mentions and channelData payloads.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main still strips and delivers a preamble before trailing NO_REPLY in normalizeReplyPayload, and subagent follow-up plus gateway live projection do not classify own-line trailing sentinels. I did not run tests because this review is read-only.

## Solution Assessment

No, not as-is. The PR covers useful paths, but it introduces a failing inline parser test, case-sensitive own-line matching, possible channelData loss, and overlaps several open PRs where maintainers need to choose the canonical policy.
