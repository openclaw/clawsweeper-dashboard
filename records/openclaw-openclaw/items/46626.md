## Summary

Keep open: this branch targets a real unresolved silent-reply leak class, but it is not merge-ready because the patch has blocking correctness issues and no inspectable after-fix real-behavior proof.

## What This Changes

The branch adds an own-line `NO_REPLY` helper and wires it into reply normalization, directive parsing, cron delivery/follow-up, gateway streaming/history sanitization, plus focused token/directive tests.

## Best Possible Solution

Choose one canonical shared silent-reply classifier for own-line/trailing `NO_REPLY`, preserve media and channelData semantics, then land it with dispatcher, gateway, cron, and outbound regression coverage plus real channel proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the shared parser/normalizer still treats `text\nNO_REPLY` as non-silent or strip-and-deliver in key paths; no runtime test was run because this review is read-only.

## Solution Assessment

No, not as written. Propagating a shared own-line classifier is a reasonable direction, but this branch adds an inconsistent inline-token test, can drop channelData payloads through outbound planning, and still needs maintainer agreement on the suppression boundary.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P2] Align the inline NO_REPLY test with the parser behavior:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The new test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but this patch only clears text when the exact or own-line checks pass. Inline text remains unchanged, so the added test will fail until the parser strips inline tokens or the assertion is corrected.
  - confidence: 0.95
- **[P2] Preserve channelData when marking own-line replies silent:** `src/auto-reply/reply/reply-directives.ts:34`
  - body: Marking own-line text as `isSilent` flows into outbound planning, where `parsed.isSilent && mergedMedia.length === 0` causes the whole entry to be skipped before channelData, presentation, or interactive content are considered. Payloads carrying channelData plus `text\nNO_REPLY` would now be discarded instead of sent as channelData-only replies.
  - confidence: 0.78
- **[P3] Cache the own-line silent-token regex:** `src/auto-reply/tokens.ts:64`
  - body: `isSilentTokenOnOwnLine` allocates a new `RegExp` on every call, and this patch calls it from streaming/gateway paths. Reuse the same token-keyed cache pattern as the existing silent-token regex helpers to avoid per-chunk allocation.
  - confidence: 0.79
