## Label Justifications

- `P2`: Repeated model-backed reminders can cause material avoidable cost and throttling, but the impact is limited to configured cron workflows and has existing alternatives.
- `impact:session-state`: Main-session cron events are injected into heartbeat processing and consume agent context and model turns.

## Summary

Current main still routes main-session cron `systemEvent` reminders through the heartbeat agent reply path, so the cost concern remains valid; however, bypassing the model requires an explicit product decision because cron events can require reasoning, tools, visibility filtering, and summarization.

## What This Changes

Add a low-cost deterministic delivery path for simple recurring reminders so they do not require a model turn every time.

## Best Possible Solution

Define an explicit cron payload or delivery mode for literal reminders that sends caller-supplied text through the existing outbound delivery owner, while retaining `systemEvent` plus heartbeat for events that require agent interpretation or tools.

## Reproduction Assessment

Yes, from source with high confidence: create a main-session cron job with a non-empty `systemEvent`; current main queues it as a cron event, builds the scheduled-reminder prompt, and invokes the agent reply path. A live billable OpenAI run was not performed.

## Solution Assessment

No, implementing generic local string formatting is not the best solution because some cron events intentionally need semantic processing. The safer design is an explicit literal-delivery contract selected by the job author, not heuristic prompt classification.
