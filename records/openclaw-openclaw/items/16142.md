- `impact:other`: The issue concerns avoidable model API usage, cost, and rate-limit pressure, which do not fit the more specific owned impact categories.

## Summary

Current main still converts main-session cron reminders into model-backed heartbeat turns, and existing tests and docs preserve that behavior. The cost concern remains valid, but bypassing the model requires a product decision because some cron events need interpretation, filtering, tools, or summarization.

## What This Changes

Request a lower-cost deterministic path for simple recurring reminders instead of sending every cron notification through a model reply.

## Best Possible Solution

Add an explicit literal-notification cron payload or delivery mode that sends trusted text through the normal channel-delivery boundary without an agent turn, while retaining system-event payloads for reasoning, tools, filtering, and summarization.

## Reproduction Assessment

Yes at source level: a main-session cron job with a non-empty systemEvent payload necessarily enters the heartbeat agent-reply path on current main. This review did not repeat the reporter's live API logging or quantify token volume.

## Solution Assessment

No—the proposed blanket string formatting is not the safest general solution because system events can require agent semantics. A separate explicit literal-delivery payload or mode is more maintainable than heuristically bypassing existing systemEvent behavior.
