## Label Justifications

- `impact:session-state`: Each reported reminder creates cron-owned session context and enters heartbeat reply processing.

## Summary

Current main still sends main-session cron reminders through a model-backed heartbeat turn. Deterministic command jobs now cover operator-authored scripts, but they do not replace watcher-generated system events with equivalent session-aware delivery, so the central request remains a valid product decision rather than an implemented fix.

## What This Changes

Add a deterministic delivery path for simple recurring cron or watcher reminders so model calls are reserved for events that need reasoning, tools, or summarization.

## Best Possible Solution

Reuse command jobs for deterministic scripts where their routing is sufficient; otherwise add one typed direct-text cron payload with explicit delivery and visibility semantics, while keeping system events model-backed.

## Reproduction Assessment

Yes, at source level with high confidence: a main-session system event is enqueued into a cron run session and processed by the model-backed heartbeat cron-event path. A live paid-provider run was not necessary to establish that call path.

## Solution Assessment

No, generic string formatting or heuristic payload detection is not the best solution. The narrow maintainable design is an explicit deterministic payload/delivery contract, reusing command jobs when they already preserve the required behavior.
