## Summary

Keep open. Current main still routes main-session cron `systemEvent` reminder text through the heartbeat cron-event prompt and model-backed reply path; related static/direct cron issues remain open rather than shipped replacements, so conservative cleanup should not close this.

## What This Changes

The issue requests a low-cost deterministic cron reminder delivery/formatting path so simple scheduled Slack/email watcher alerts do not repeatedly call ChatGPT to relay already-formed text.

## Best Possible Solution

Define a maintainer-approved low-cost path for simple main-session cron reminders, likely an explicit direct/static payload option or conservative simple-payload boundary, while preserving the model-backed cron-event path for complex, internal, tool-using, and reasoning-dependent events.

## Reproduction Assessment

Yes, at source level. Main-session cron enqueues `systemEvent` text with a `cron:` context key, heartbeat selects `buildCronEventPrompt`, and `getReplyFromConfig` sends that prompt through the model-backed reply path.

## Solution Assessment

Unclear. Deterministic formatting or direct delivery is plausible for simple reminders, but maintainers need to define which cron events may bypass the model and how that relates to the open static/direct cron contracts.
