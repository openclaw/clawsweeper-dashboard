## Summary

Close as implemented: current main and v2026.6.5 provide command cron jobs that run deterministic watcher scripts and announce captured output without an agent/model call, covering the reported high-frequency formatter-cost problem while preserving the model-backed systemEvent path for cron events that still need reasoning.

## What This Changes

Request a no-token deterministic cron delivery path so OpenAI/ChatGPT is not used to reformat high-frequency Slack/email watcher reminders.

## Best Possible Solution

Keep the shipped command cron path as the core no-token solution for deterministic watcher output, and leave systemEvent cron model-mediated for complex reminders that need reasoning or user-facing rewriting.

## Reproduction Assessment

Yes, source-level: the legacy main-session systemEvent cron path still queues cron text into heartbeat and then calls the model-backed reply path. This is not a failing strict bug repro because current main now provides command cron as the shipped no-LLM path for deterministic watcher output.

## Solution Assessment

Yes for the central cost problem: command cron is a narrower maintainable solution than making every systemEvent reminder bypass the model. It preserves existing cron-event semantics while giving deterministic Slack/email watcher scripts a direct delivery path.
