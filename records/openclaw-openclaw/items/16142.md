## Summary

Keep open. Current main still routes main-session cron reminder text through the heartbeat cron-event prompt and model-backed reply path, so the reported cost/throttle concern remains valid. The fix needs maintainer product/API judgment because not every cron/system event can safely bypass model reasoning, and related PR #48883 is open rather than merged and does not by itself settle a deterministic low-cost delivery mode.

## What This Changes

Request a lower-cost deterministic path for simple recurring cron/heartbeat reminders so OpenClaw does not repeatedly call OpenAI/ChatGPT just to reformat watcher alerts.

## Best Possible Solution

Define a narrow low-cost delivery mode for simple main-session cron reminders, preferably through an explicit cron option or conservative simple-payload detector, while preserving the existing model-backed cron-event path for complex, internal, tool-using, or reasoning-dependent events. Reconcile the relevant parts of #48883, then update docs/tool guidance and regression tests for both deterministic and model-backed paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: enqueue a tagged main-session cron system event, run heartbeat, and observe the cron-event prompt plus getReplyFromConfig call; current tests assert this model-backed behavior.

## Solution Assessment

Unclear. Deterministic formatting is directionally right for simple reminders, but the requested blanket replacement is too broad unless maintainers first define which cron events are safe to deliver without model interpretation and how that interacts with #48883's tokenized main-session approach.
