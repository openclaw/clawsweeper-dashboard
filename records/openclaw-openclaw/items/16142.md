## Summary

Keep open. Current main still routes main-session cron `systemEvent` reminder text through heartbeat cron-event prompting and the model-backed reply path, while the related static-message/direct cron work and open hardening PRs are not shipped replacements for the reporter's dynamic watcher-generated reminder flow.

## What This Changes

The issue requests a lower-cost cron reminder delivery path so simple scheduled watcher messages are not repeatedly sent through ChatGPT for formatting.

## Best Possible Solution

Define a maintainer-approved low-cost cron delivery path for simple main-session `systemEvent` reminders, likely through an explicit direct/static delivery option or conservative simple-payload boundary, while preserving the model-backed path for complex or reasoning-dependent cron events.

## Reproduction Assessment

Yes, at source level. Main-session cron enqueues `systemEvent` text with a `cron:` context key, heartbeat selects `buildCronEventPrompt`, and `getReplyFromConfig` sends that prompt through the model-backed reply path.

## Solution Assessment

Unclear. Deterministic formatting or direct delivery is plausible for simple reminders, but maintainers need to define which cron events may bypass the model and how that relates to open static-message, direct-exec, and direct-invoke cron contracts.
