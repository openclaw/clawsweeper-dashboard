## Label Justifications

- `P2`: Frequent model-backed reminder turns can cause meaningful API cost and throttling, but only for users who configure affected automation workflows.
- `impact:session-state`: The current implementation creates repeated cron run sessions and model-backed transcript turns for reminder text that may already be fully rendered.

## Summary

Current main still routes every non-empty main-session cron `systemEvent` through a generated cron-event prompt and a model-backed reply turn, so the reported cost/throttling problem remains. The issue should stay open because deterministic command cron is a useful alternative for operator-authored scripts but does not replace watcher-generated main-session reminders, and maintainers still need to define which notification semantics may safely bypass agent reasoning.

## What This Changes

Add a lower-cost deterministic path for simple scheduled notifications instead of using a model turn solely to rephrase and relay their text.

## Best Possible Solution

Introduce a typed deterministic notification payload or explicit delivery mode for already-rendered text, route it through the existing channel delivery contract without a model turn, and retain `systemEvent` for model-backed agent semantics with focused fresh-install and upgrade tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: create a main-session cron job with a non-empty `systemEvent`; cron enqueues it with a cron context, heartbeat builds the scheduled-reminder relay prompt, and `getReplyFromConfig` starts the model-backed reply. The read-only review did not execute the reporter's live five-minute Slack/email watcher setup.

## Solution Assessment

No, replacing model use with generic string formatting is not the safest solution because existing `systemEvent` jobs can require reasoning or internal handling. The maintainable fix is an explicit deterministic notification contract separate from model-backed cron events.
