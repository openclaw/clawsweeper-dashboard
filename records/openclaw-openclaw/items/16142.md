## Summary

Keep open. Current main and v2026.6.8 still route main-session cron reminder text through the heartbeat cron-event prompt and getReplyFromConfig model path, so the cost/throttling concern remains valid; the missing piece is a maintainer decision on which reminders may use literal delivery without losing cron-event agent semantics.

## What This Changes

The issue requests a lower-cost deterministic delivery path so simple cron reminders do not use ChatGPT/OpenAI as a string formatter every few minutes.

## Best Possible Solution

Add an explicit literal-notification cron payload or delivery mode for trusted simple reminders, while keeping systemEvent on the existing model-backed heartbeat path for reasoning-dependent events.

## Reproduction Assessment

Yes, at source level: a main-session cron job with non-empty systemEvent text enters heartbeat, becomes a cron-event prompt, and calls getReplyFromConfig. I did not repeat the reporter's live OpenAI logging or measure token volume.

## Solution Assessment

No for a blanket string-formatting bypass: systemEvent can need agent reasoning, tool use, or internal handling. The safer direction is an explicit literal-delivery payload or mode with focused tests and docs defining the boundary.
