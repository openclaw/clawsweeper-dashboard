## Summary

Keep open. Current main and v2026.5.20 still send main-session cron `systemEvent` reminders through heartbeat cron-event prompting and `getReplyFromConfig`; the open static-message/direct-exec cron tracks are useful context, but they are not shipped replacements for dynamic watcher-generated reminder text.

## What This Changes

The issue requests a low-cost deterministic path for simple cron reminder delivery so OpenClaw does not spend OpenAI tokens formatting repeated watcher alerts.

## Best Possible Solution

Define a maintainer-approved low-cost cron delivery path for simple main-session `systemEvent` reminders, likely through an explicit direct/static delivery option or conservative simple-payload boundary, while preserving the model-backed path for complex, internal, tool-using, and reasoning-dependent cron events.

## Reproduction Assessment

Yes, at source level. Main-session cron enqueues `systemEvent` text with a `cron:` context key, heartbeat selects `buildCronEventPrompt`, and `getReplyFromConfig` sends that prompt through the model-backed reply path.

## Solution Assessment

Unclear. Deterministic formatting or direct delivery is plausible for simple reminders, but maintainers need to define which cron events may bypass the model and how that relates to the open static-message, direct-exec, and direct-invoke cron contracts.
