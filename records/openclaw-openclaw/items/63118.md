## Summary

Keep open. Current main still has useful lower-level abort/delete/reset plumbing, but the public `openclaw sessions` surface and Sessions UI do not provide the requested stop, kill, pause, or resume lifecycle controls; the related open PR #35735 only covers deletion cleanup, so it does not supersede this broader lifecycle request.

## What This Changes

The issue requests public session lifecycle management commands, Sessions UI controls, and resource/cost-management limits for running OpenClaw sessions.

## Best Possible Solution

Keep this as the broad session lifecycle tracker. The maintainable first slice is a public `openclaw sessions stop <session-key>` backed by `sessions.abort` with docs and tests, followed by Sessions-page Stop controls only when the UI can show real active-run state and Gateway abort results; pause/resume/resource quotas should wait for enforceable runtime and accounting design.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the sessions CLI registration, docs, Gateway method list, and Sessions UI can be inspected on current main and none expose the requested public stop/kill/pause/resume controls.

## Solution Assessment

Yes for the triage outcome: keeping the issue open is the best current answer because current main only partially covers the underlying need with lower-level abort plumbing. The safer implementation path is to add a narrow stop command/UI action first rather than closing this as implemented or folding it into deletion cleanup.
