## Summary

Keep open. Current main still only exposes lower-level or narrower controls: `sessions.abort`/`chat.abort`, a chat Stop path, task cancellation for background task records, and an HTTP subagent kill route, but not the requested public `openclaw sessions stop/kill/pause/resume` commands or Sessions-page lifecycle controls. The related open PR #35735 remains deletion cleanup only, so it does not supersede this broader lifecycle tracker.

## What This Changes

The issue requests public session lifecycle management commands, Sessions UI controls, and resource/cost-management limits for running OpenClaw sessions.

## Best Possible Solution

Keep this as the broad session lifecycle tracker. The maintainable first product slice is a public `openclaw sessions stop <session-key>` backed by `sessions.abort`, with docs and focused CLI/Gateway tests; Sessions-page Stop controls should follow only when the UI can show active-run state and abort results clearly. Pause/resume and quota enforcement should be split into explicit design-backed follow-ups.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main can be inspected to show the sessions CLI/docs and Sessions UI still lack `stop`, `kill`, `pause`, and `resume` lifecycle controls, while lower-level abort/task/subagent paths only cover narrower cases.

## Solution Assessment

Yes for the triage outcome: keeping this issue open is the best current answer because current main only partially covers the need with lower-level or narrower primitives. The safer implementation path is a scoped stop command/UI action first, not closing this as implemented or folding it into deletion cleanup.
