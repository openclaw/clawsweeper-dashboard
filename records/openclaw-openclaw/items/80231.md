## Summary

Keep open: the report matches a current iOS/shared chat event plumbing gap. Current main has a gateway `session.message` contract and a macOS transport that subscribes/decodes it, but the iOS transport does neither and the shared view model still drops assistant session-message payloads.

## What This Changes

The issue requests realtime or near-realtime updates for new agent messages in iOS group chats without requiring the user to leave and reopen the conversation.

## Best Possible Solution

Wire the iOS chat transport into the existing gateway session-message subscription/event contract and update shared chat state so active-session assistant transcript messages appear live with focused Swift coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: an active iOS chat session receiving a gateway `session.message` assistant payload will not display it live because iOS does not subscribe/decode the event and shared chat state filters assistant session messages.

## Solution Assessment

Yes. The narrowest maintainable solution is to use the existing `sessions.messages.subscribe` and `session.message` contract in iOS/shared chat code, not to add a polling mode, config flag, or new gateway protocol surface.
