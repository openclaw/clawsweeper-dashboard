## Summary

Keep open: current main still has a source-visible iOS/shared chat event gap for live group-chat updates. The gateway already exposes subscribed `session.message` transcript updates, but the iOS transport does not subscribe to or decode that event and the shared view model currently ignores assistant `session.message` payloads.

## What This Changes

The issue requests real-time iOS group-chat updates for new agent messages without leaving and reopening the chat view.

## Best Possible Solution

Wire the iOS chat transport into the existing `sessions.messages.subscribe` and `session.message` contract, then update shared chat handling so active-session assistant transcript events appear without a manual history reload.

## Reproduction Assessment

Yes by source inspection: an active-session `session.message` assistant payload will not reach or update the iOS/shared chat UI because iOS does not decode that event and shared handling filters non-user roles. I did not perform a live iOS gateway run.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing gateway subscription/event contract in `IOSGatewayChatTransport` and `OpenClawChatViewModel`, not add a new polling mode or protocol surface.
