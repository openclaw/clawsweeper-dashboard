## Summary

Keep open: the report matches a current iOS chat event plumbing gap rather than stale or external behavior. The gateway has session transcript events for subscribed clients, but the iOS transport does not subscribe/decode `session.message`, and the shared chat view model currently ignores assistant `session.message` payloads.

## What This Changes

The issue reports that iOS group chats do not display agent replies in real time until the user leaves and re-enters the chat.

## Best Possible Solution

Wire iOS chat into the existing gateway session/message event contract and update the shared view model so active-session assistant transcript events appear without a manual reload, with focused Swift regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: emit or receive an active-session `session.message` assistant payload and the current iOS/shared path will not show it. I did not perform a live iOS gateway run.

## Solution Assessment

Yes, the best fix is narrow event plumbing in `IOSGatewayChatTransport` and `OpenClawChatViewModel`, using the existing gateway protocol instead of adding a new polling mode or core protocol surface.
