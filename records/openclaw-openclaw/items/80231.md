## Summary

Keep open: current main still has an iOS/shared chat event plumbing gap that matches the reported group-chat symptom, and no merged or open iOS-specific fix appears to supersede it.

## What This Changes

The issue requests real-time iOS group-chat updates so new multi-agent replies appear without leaving and reopening the chat.

## Best Possible Solution

Wire iOS chat into the existing Gateway `sessions.messages.subscribe` and `session.message` contract, then reconcile active-session assistant transcript events in the shared view model without adding polling or a new protocol surface.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an active iOS chat receiving an assistant `session.message` payload will not render it because iOS neither subscribes/decodes that event nor does the shared view model accept assistant session-message payloads.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing Gateway session-message event contract in the iOS transport and shared view model, rather than adding a new polling mode or changing core protocol behavior.
