## Summary

Keep open: current main still has a clear iOS/shared chat event plumbing gap for group-chat transcript updates; the gateway exposes and emits subscribed `session.message` events, but iOS does not subscribe/decode them and the shared view model drops assistant payloads.

## What This Changes

The issue requests real-time iOS group-chat agent message updates without leaving and re-entering the chat.

## Best Possible Solution

Wire the iOS chat transport into the existing `sessions.messages.subscribe` / `session.message` contract and update `OpenClawChatViewModel` to show active-session assistant transcript events with focused Swift regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: deliver an active-session `session.message` assistant payload on current main and the iOS transport will not yield it, while the shared view model would reject it by role if decoded. I did not run a live iOS gateway flow.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrowest maintainable fix is client-side plumbing against the existing Gateway contract, not a new polling mode, config option, or core protocol change.

## Repair Work Prompt

Fix the iOS group-chat real-time update bug reported at https://github.com/openclaw/openclaw/issues/80231. On current main, `IOSGatewayChatTransport.setActiveSessionKey` is a no-op, its event mapper drops `session.message`, and `OpenClawChatViewModel.handleSessionMessageEvent` only appends `user` messages. Implement the existing Gateway contract: subscribe the active iOS chat session with `sessions.messages.subscribe`, map `session.message` to `.sessionMessage`, and update the shared view model so active-session assistant transcript payloads become visible without leaving and re-entering the chat. Preserve existing one-on-one streaming behavior, avoid new polling/config/protocol fields, and handle dedupe if `chat` final refresh and `session.message` deliver the same visible assistant reply.
