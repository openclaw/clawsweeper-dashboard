## Summary

Keep open: current main still has the same source-proven iOS group-chat update gap, while the existing Gateway and macOS paths already expose the needed session-message contract.

## What This Changes

The issue asks for iOS group chats to show new agent messages in real time without leaving and reopening the chat.

## Best Possible Solution

Use the existing Gateway session-message subscription path in the iOS transport and let the shared view model merge active-session assistant transcript messages with focused Swift regression coverage.

## Reproduction Assessment

Yes, from source inspection against current main: iOS never subscribes to or decodes `session.message`, and the shared handler currently ignores assistant session messages. I did not run a live iOS device or simulator flow in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse the existing `sessions.messages.subscribe` and `session.message` Gateway contract already used by macOS, rather than adding polling, a new protocol event, or group-chat-specific policy.
