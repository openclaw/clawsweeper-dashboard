## Summary

Keep open: current `main` still has the iOS/shared chat gap described in the report, with no matching open fix PR or iOS-specific duplicate found. The Gateway already exposes `sessions.messages.subscribe` and emits `session.message`, but the iOS transport does not subscribe or decode that event and the shared view model currently ignores assistant session-message payloads.

## What This Changes

The issue requests real-time iOS group-chat updates for new agent messages without leaving and re-entering the conversation.

## Best Possible Solution

Wire the iOS active-session path to the existing Gateway session-message subscription, decode `session.message`, and let the shared view model merge active-session assistant transcript events with focused Swift regression coverage.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here: current `main` does not subscribe the iOS active chat session to transcript updates, does not decode `session.message`, and filters assistant session-message payloads in the shared view model.

## Solution Assessment

Yes: the narrow maintainable fix is to consume the existing `sessions.messages.subscribe` and `session.message` Gateway contract instead of adding polling, a new protocol event, or a group-chat-specific workaround.
