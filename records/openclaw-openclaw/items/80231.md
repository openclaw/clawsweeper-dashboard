## Summary

Keep open: current main still has the iOS/shared event-plumbing gap described in the report and in the prior ClawSweeper comment. The Gateway and macOS chat transport already expose the intended `sessions.messages.subscribe` / `session.message` path, but the iOS transport does not subscribe or decode those frames and the shared view model still ignores assistant session-message payloads.

## What This Changes

The issue reports that iOS group chat assistant messages do not appear live until the user leaves and re-enters the chat.

## Best Possible Solution

Wire iOS chat to the existing Gateway session-message subscription and decoding path, then merge active-session assistant transcript events into the shared view model with focused Swift regression coverage.

## Reproduction Assessment

Yes: current-main source inspection gives a high-confidence reproduction path because iOS never subscribes to or decodes `session.message`, and the shared handler filters out assistant messages. I did not run a live Aight/iOS group-chat flow in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to use the existing `sessions.messages.subscribe` and `session.message` contract already used by macOS, not to add polling, a new protocol event, or a group-chat-specific side path.
