## Summary

Keep open: current main still matches the reported iOS group-chat real-time update gap. The Gateway already has a session-message subscription/event contract, but the iOS transport does not subscribe to or decode those events, and the shared chat view model currently ignores assistant `session.message` payloads.

## What This Changes

The issue requests real-time iOS group-chat updates for agent replies without leaving and reopening the chat.

## Best Possible Solution

Wire iOS chat to the existing `sessions.messages.subscribe` and `session.message` contract, decode assistant transcript events, and merge them into the active shared chat model with focused Swift regression coverage.

## Reproduction Assessment

Yes from source inspection, though not from a live iOS run: current main does not subscribe to `session.message`, does not decode that event in the iOS transport, and filters assistant session messages in the shared view model.

## Solution Assessment

Yes: the narrowest maintainable fix is to consume the existing Gateway `sessions.messages.subscribe`/`session.message` path rather than adding polling, a new protocol event, or a group-chat-specific workaround.
