## Summary

Keep open: current main still has a source-backed iOS/shared chat event gap that matches the report. The gateway exposes and emits `session.message`, but the iOS transport does not subscribe/decode it and the shared chat view model drops assistant session-message payloads.

## What This Changes

The issue asks for iOS group chat agent replies to update in real time without leaving and reopening the chat.

## Best Possible Solution

Wire iOS chat into the existing `session.message` subscription/event contract and let the shared chat view model render active-session assistant transcript events without duplicate bubbles.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: emit an active-session assistant `session.message` payload on current main and the iOS/shared path will not show it. I did not perform a live iOS gateway run.

## Solution Assessment

Yes, the narrowest maintainable fix is iOS/shared event plumbing against the existing gateway protocol, not a new polling mode or core protocol change.
