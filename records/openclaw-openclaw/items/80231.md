## Summary

Keep open: this still matches a current iOS/shared chat event plumbing bug, not stale support noise. Current main has a gateway real-time `session.message` path and macOS consumes it, but the iOS transport does not subscribe/decode it and the shared view model currently ignores assistant transcript events.

## What This Changes

Reporter asks for iOS group chat agent replies to appear in real time without leaving and re-entering the chat.

## Best Possible Solution

Wire iOS chat into the existing `sessions.messages.subscribe` and `session.message` contract, then update the shared view model to accept active-session assistant transcript events with focused dedupe and regression coverage.

## Reproduction Assessment

Yes from source inspection: receiving an active-session assistant `session.message` payload on current main will not update iOS chat because iOS neither subscribes/decodes that event nor displays assistant session-message payloads. I did not perform a live iOS gateway run.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing gateway session-message event contract already documented and consumed by macOS, not to add a new polling mode, config flag, or protocol surface.
