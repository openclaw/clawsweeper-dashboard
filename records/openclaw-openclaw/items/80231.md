## Summary

Keep open: this matches a current iOS/shared chat event plumbing bug. Current main exposes gateway transcript events, but the iOS transport does not subscribe to or decode `session.message`, and the shared chat view model ignores assistant `session.message` payloads even if they arrive.

## What This Changes

The issue requests real-time display of agent replies in iOS/Aight group chats without leaving and reopening the chat.

## Best Possible Solution

Wire iOS chat into the existing gateway session/message event contract and update the shared view model so active-session assistant transcript events appear without adding a new polling mode or protocol surface.

## Reproduction Assessment

Yes for source-level reproduction: emit or receive an active-session assistant `session.message` payload and current main will not show it through the iOS/shared path because iOS neither subscribes nor decodes that event and the view model filters out assistant roles. I did not run a live iOS gateway session.

## Solution Assessment

Yes, the narrowest maintainable fix is to use the existing `sessions.messages.subscribe`/`session.message` gateway contract in `IOSGatewayChatTransport` and adjust `OpenClawChatViewModel` handling, rather than adding a separate polling path.
