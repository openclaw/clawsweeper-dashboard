## Summary

Keep open: this matches a current iOS/shared chat event plumbing bug, not stale or external behavior. The gateway already has a subscribed `session.message` transcript path, but the iOS transport does not subscribe/decode it and the shared view model still drops assistant `session.message` payloads.

## What This Changes

The issue asks for iOS group chat agent replies to appear in real time without leaving and reopening the chat.

## Best Possible Solution

Wire iOS chat into the existing gateway `sessions.messages.subscribe` and `session.message` contract, then let the shared view model append active-session assistant transcript updates with focused Swift regression coverage.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: emit an active-session assistant `session.message` payload and the current iOS transport will not surface it, while the shared view model would drop the assistant role even if decoded. I did not run a live Aight device flow.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing gateway subscription/event contract and update iOS/shared handling, not add polling, a new protocol method, or a new config knob.
