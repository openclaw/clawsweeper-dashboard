## Summary

Keep open. Current main still has no native systemd readiness-notify contract: the generated gateway unit lacks `Type=notify`, the ready boundary only updates OpenClaw state/logging, and update/restart flows still fall back to OpenClaw polling.

## What This Changes

Request `sd_notify`/`Type=notify` support so systemd start/restart waits until the OpenClaw gateway is actually ready.

## Best Possible Solution

Add a Linux gateway readiness contract: send `READY=1` at the existing post-attach gateway-ready boundary when systemd provides `NOTIFY_SOCKET`, render generated units with notify readiness, document it, and leave watchdog recovery to the separate watchdog track.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no notify terms, generated systemd units lack notify directives, and the gateway-ready boundary only changes internal readiness/logging state.

## Solution Assessment

Yes. Native systemd notification at the existing gateway ready boundary plus matching generated unit directives is the narrowest maintainable solution; polling should remain fallback diagnostics.
