## Summary

Keep open. Current `main` still lacks a systemd readiness-notify contract: generated Linux units do not use `Type=notify` or `NotifyAccess=main`, the gateway ready boundary only updates OpenClaw readiness/logging, and `systemctl restart` is still followed by OpenClaw’s own polling rather than being made readiness-aware for direct systemd users. The related watchdog request is adjacent but does not supersede this READY=1 feature.

## What This Changes

The issue requests systemd `sd_notify` readiness support so generated gateway units can use `Type=notify` and `systemctl restart` waits until the gateway is actually ready.

## Best Possible Solution

Keep this open for a focused Linux gateway/daemon PR: add a small systemd notify helper, send `READY=1` only after the existing gateway ready boundary, update generated and documented systemd units to use `Type=notify` with `NotifyAccess=main`, and leave watchdog heartbeat support to #11973 unless maintainers choose to combine the work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current `main`: searching for notify-related terms finds no implementation, `buildSystemdUnit` lacks notify directives, and the ready boundary only logs/updates internal readiness state.

## Solution Assessment

Yes, the requested `sd_notify READY=1` plus generated `Type=notify` unit is the narrowest maintainable solution for direct systemd readiness semantics. Existing CLI health polling should remain as a fallback/diagnostic path, but it is not a substitute for systemd’s native readiness contract.
