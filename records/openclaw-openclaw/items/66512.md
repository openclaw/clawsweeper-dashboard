## Summary

Keep open. Current main still lacks the requested systemd readiness-notify contract: generated Linux gateway units do not opt into notify mode, and the existing gateway-ready boundary does not send `READY=1`; the adjacent WatchdogSec work is related but does not supersede this readiness request.

## What This Changes

Request Type=notify and systemd `sd_notify` readiness support so Linux service restarts and dependent health checks wait until the gateway is actually ready.

## Best Possible Solution

Add a best-effort `READY=1` notification at the existing gateway-ready boundary and generate matching `Type=notify`/`NotifyAccess=main` for Linux systemd gateway units, while keeping WatchdogSec/WATCHDOG pings separate unless maintainers explicitly combine the scope.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: generated systemd units lack notify directives, and the gateway-ready boundary has no systemd notification call; no live systemd timing repro was run in this read-only review.

## Solution Assessment

Yes. A best-effort `READY=1` call at the existing post-attach ready boundary plus matching unit directives is the narrow maintainable path; watchdog pings should remain separate or be coordinated with the adjacent watchdog issue.
