## Summary

Keep open. Current main still lacks the requested systemd readiness-notify contract: generated gateway units do not opt into notify mode, and the existing gateway-ready boundary does not send a `READY=1` notification.

## What This Changes

Request support for systemd `sd_notify` readiness by emitting `READY=1` when the gateway is ready and generating `Type=notify` systemd units.

## Best Possible Solution

Add a best-effort Linux `READY=1` notifier at the existing post-attach gateway-ready boundary, render matching notify directives for generated gateway units, and keep watchdog pings separate unless maintainers intentionally combine them.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: generated systemd units still lack notify directives, and the gateway-ready boundary has no systemd notification call.

## Solution Assessment

Yes. A best-effort `READY=1` at the existing ready boundary plus matching unit directives is the narrow maintainable path; watchdog support should stay separate or be coordinated with the adjacent watchdog PR.

Scope this to `READY=1` readiness: add a best-effort Linux notifier that is a no-op when `NOTIFY_SOCKET` is unset, handles systemd socket forms correctly, avoids shell execution, treats failures as nonfatal, and sends `READY=1` after the existing post-attach `gateway ready` boundary once HTTP listening is already established. Render matching generated gateway unit directives, likely `Type=notify` plus appropriate `NotifyAccess`, in `buildSystemdUnit`. Do not add watchdog pings unless maintainers explicitly expand scope; coordinate with the open watchdog PR if it lands first. Add focused regression tests for the unit renderer, notifier helper, and post-attach ready boundary; update Linux/gateway docs and changelog for the user-visible service behavior.
