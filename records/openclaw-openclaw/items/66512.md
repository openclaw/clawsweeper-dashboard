## Summary

Keep open. Current main still lacks a systemd readiness-notify contract: the generated Linux unit has no `Type=notify`/`NotifyAccess`, the gateway ready boundary only flips OpenClaw readiness/logging state, and restart/update paths still rely on OpenClaw polling instead of native systemd readiness.

## What This Changes

Request support for systemd `sd_notify` readiness by sending `READY=1` after the gateway is ready and generating Linux units that use `Type=notify`, with watchdog pings left as an optional extension.

## Best Possible Solution

Add a Linux gateway readiness contract: send a non-fatal systemd `READY=1` notification from the existing post-attach ready boundary when `NOTIFY_SOCKET` is present, render supported systemd units with `Type=notify` and `NotifyAccess=main`, update the gateway docs, and keep watchdog recovery as a separate follow-up unless maintainers choose otherwise.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the relevant search terms are absent, the unit renderer lacks notify directives, and the ready boundary only updates internal readiness/logging state.

## Solution Assessment

Yes. Native `sd_notify` at the existing gateway ready boundary plus `Type=notify` in generated Linux units is the narrowest maintainable way to make `systemctl restart` reflect RPC readiness; OpenClaw health polling should remain as fallback diagnostics, not the systemd readiness contract.
