## Summary

Keep open. Current main still lacks the native systemd readiness-notify contract requested here: the generated gateway unit has no `Type=notify` or `NotifyAccess=main`, the gateway ready boundary only updates OpenClaw state/logging, and restart/update paths still use OpenClaw polling rather than systemd readiness notification.

## What This Changes

Add systemd `sd_notify` readiness support for the gateway and generate notify-capable systemd units so service restarts wait until the WebSocket gateway is actually ready.

## Best Possible Solution

Implement a Linux gateway readiness contract at the existing post-attach ready boundary, render generated systemd units with notify directives, add focused regression coverage, and leave watchdog pings as a separate follow-up unless maintainers explicitly combine them.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: generated systemd units lack notify directives, the gateway ready boundary has no systemd notification call, and notify-related terms are absent from the relevant code/docs surfaces.

## Solution Assessment

Yes. Best-effort `READY=1` at the existing gateway-ready boundary plus matching generated unit directives is the narrow maintainable direction; the current polling code should remain fallback diagnostics rather than the primary systemd contract.
