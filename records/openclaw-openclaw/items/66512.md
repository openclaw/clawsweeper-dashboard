## Summary

Keep open. Current main still lacks a native systemd readiness-notify contract: generated Linux gateway units do not render `Type=notify` or `NotifyAccess`, the gateway ready boundary only updates internal state/logging, and update/restart flows still rely on OpenClaw polling instead of systemd start readiness. The optional watchdog portion should stay scoped to the separate open watchdog issue.

## What This Changes

Request systemd `sd_notify` readiness for the gateway, including `READY=1` at the gateway-ready boundary and `Type=notify` generated units.

## Best Possible Solution

Add `READY=1` notification when `NOTIFY_SOCKET` is present at the existing post-attach gateway-ready boundary, render supported Linux units with `Type=notify` and `NotifyAccess=main`, update docs/changelog/tests, and leave watchdog recovery to #11973.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no notify terms, generated systemd units omit notify directives, and the ready boundary only changes internal readiness/logging state.

## Solution Assessment

Yes. Native systemd notification at the existing post-attach ready boundary plus matching generated unit directives is the narrowest maintainable solution; existing OpenClaw health polling should remain fallback diagnostics.
