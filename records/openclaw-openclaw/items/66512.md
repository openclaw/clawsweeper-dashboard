## Summary

Keep open. Current `main` still does not implement the requested systemd readiness-notify contract: generated gateway units remain non-notify units, and the existing gateway ready boundary does not send `READY=1` through `NOTIFY_SOCKET`.

## What This Changes

Request systemd `sd_notify` readiness support so `openclaw gateway install` can generate `Type=notify` gateway units that become active only after RPC readiness.

## Best Possible Solution

Add best-effort Linux `READY=1` notification at the existing gateway-ready boundary, render matching `Type=notify` and `NotifyAccess=main` unit directives for gateway installs, and keep watchdog pings as separate scope unless maintainers intentionally combine them.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: generated systemd units still lack notify directives, and the gateway ready boundary has no systemd notification call.

## Solution Assessment

Yes. A best-effort `READY=1` at the existing gateway-ready boundary plus matching generated unit directives is the narrow maintainable direction; OpenClaw polling should remain fallback diagnostics rather than the primary systemd contract.
