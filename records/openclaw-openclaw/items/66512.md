## Summary

Keep open. Current main still lacks a native systemd readiness-notify contract: generated Linux gateway units do not use `Type=notify`, the gateway ready boundary only updates OpenClaw state/logging, and restart/update flows still rely on OpenClaw polling rather than systemd start readiness.

## What This Changes

The issue requests systemd `sd_notify` readiness support so generated gateway units can use `Type=notify` and `systemctl restart` waits for gateway RPC readiness.

## Best Possible Solution

Add a Linux gateway readiness contract: send `READY=1` at the existing post-attach gateway-ready boundary when systemd provides `NOTIFY_SOCKET`, render supported systemd units with notify readiness, update docs/changelog, and leave WatchdogSec recovery to the separate watchdog issue.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no notify terms, generated systemd units lack notify directives, and the ready boundary only changes internal OpenClaw readiness/logging state.

## Solution Assessment

Yes. Native systemd notification at the existing gateway ready boundary plus matching generated unit directives is the narrowest maintainable fix; OpenClaw restart health polling should remain fallback diagnostics rather than the systemd readiness contract.
