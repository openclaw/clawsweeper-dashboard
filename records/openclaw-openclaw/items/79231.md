## Summary

Keep open: this is a draft, maintainer-labeled Telegram feature PR with useful reply-chain work, but source review found a multi-account cache ownership bug and the PR body only provides test verification, not after-fix real behavior proof.

## What This Changes

The PR adds a persisted Telegram observed-message cache, threads nearest-first ReplyChain data through Telegram inbound context into embedded agent prompt context, and updates tests, docs, and CHANGELOG.

## Best Possible Solution

Finish this as a maintainer-reviewed Telegram feature: keep reply-chain context extension-owned, share or isolate the persisted cache per account/path, add multi-instance cache coverage, and attach redacted real Telegram proof before merge.

## Reproduction Assessment

No live reproduction was run. The multi-account cache clobber path is source-reproducible from two Telegram bot handler/cache instances sharing one persisted path and recording alternating account messages.

## Solution Assessment

No. The reply-chain direction is reasonable, but the patch should first match the existing shared-bucket cache pattern or use per-account persisted files, then show real Telegram behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Share the message cache across handler instances:** `extensions/telegram/src/message-cache.ts:251-253`
  - body: Each call to `createTelegramMessageCache` gets its own in-memory `Map` after reading the persisted file once, and `record()` rewrites the whole file from that private map. In a configured multi-account gateway, each account starts its own Telegram bot/handler against the same session-store-derived cache path, so interleaved messages from account B can overwrite account A's cached parents and make reply-chain hydration disappear for that account. Please make the cache shared per persisted path or persist per account, and cover two live cache instances writing the same path.
  - confidence: 0.84
