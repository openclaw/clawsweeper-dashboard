## Summary

Keep open, but this patch should not merge as-is. The status-sink plumbing is useful, and a repository member recently repaired the branch, but the current diff marks ordinary Feishu app/webhook traffic as transport liveness, which can opt quiet Feishu accounts back into false stale-socket restarts; external real behavior proof is also still missing.

## What This Changes

The PR threads Feishu channel status updates into gateway health snapshots and adds tests for connection, event, and validated transport activity reporting.

## Best Possible Solution

Keep the status-sink plumbing, but publish `lastTransportActivityAt` only from real Feishu transport heartbeat/socket activity; app events and webhook requests should update app-level timestamps while quiet WebSocket/webhook accounts remain healthy.

## Reproduction Assessment

Yes for the blocking review finding by source inspection: the PR writes `lastTransportActivityAt` from ordinary Feishu events and webhook requests, and current health policy restarts connected channels when that timestamp becomes stale. I did not run a live Feishu setup.

## Solution Assessment

No. The proposed status path is directionally right, but treating app/webhook traffic as transport liveness conflicts with the existing health-policy contract; the safer fix is to separate app activity from true transport activity and add quiet-channel regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Stop counting app events as transport liveness:** `extensions/feishu/src/monitor.account.ts:479`
  - body: With this flag enabled for WebSocket accounts, `markEventReceived()` writes `lastTransportActivityAt` for ordinary Feishu messages, reactions, bot membership, and card events. The gateway health policy treats an aged `lastTransportActivityAt` on a connected account as a stale socket, so after any prior event a quiet but healthy Feishu WebSocket can be restarted once the threshold passes. Keep app events to `lastEventAt` unless the Feishu SDK exposes actual heartbeat/transport activity.
  - confidence: 0.91
- **[P2] Do not treat webhook requests as socket liveness:** `extensions/feishu/src/monitor.transport.ts:316`
  - body: Webhook mode has no persistent upstream socket, but this status patch records `lastTransportActivityAt` for a signed challenge or event and leaves the account connected. After the first valid webhook, a quiet listening server will look like a stale socket once no more requests arrive for the health threshold, causing unnecessary restarts. Publish app/request activity separately and reserve `lastTransportActivityAt` for real transport liveness.
  - confidence: 0.88
