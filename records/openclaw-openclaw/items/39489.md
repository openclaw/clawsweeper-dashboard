## Summary

Keep open, but do not merge as written. The status-sink plumbing is useful and current main still lacks it, but the PR publishes `lastTransportActivityAt` from Feishu application/webhook events, which conflicts with the current health-policy contract and can recreate false stale-socket restarts after normal quiet periods.

## What This Changes

The PR threads Feishu channel status reporting through startup, monitor, account, transport, and tests so gateway health snapshots receive connection, event, and activity updates.

## Best Possible Solution

Keep the Feishu-owned status-sink path, but publish `lastTransportActivityAt` only from genuine transport/heartbeat proof; use `connected`, `mode`, `lastConnectedAt`, and `lastEventAt` for lifecycle and event observability otherwise.

## Reproduction Assessment

Yes by source inspection: once this PR writes `lastTransportActivityAt` from an event or webhook request, the current health policy will flag the channel stale after the threshold even if the channel is merely quiet. I did not run a live Feishu account reproduction.

## Solution Assessment

No as written. Passing `ctx.setStatus` through the Feishu monitor stack is the right narrow direction, but using application/request timestamps as generic transport liveness is not the best fit for the current health-policy contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Stop treating Feishu app events as transport liveness:** `extensions/feishu/src/monitor.account.ts:256`
  - body: This sets `lastTransportActivityAt` whenever a WebSocket-mode Feishu application event is handled. The current health policy intentionally uses that field for transport liveness and ignores app events because quiet connected channels can still be healthy; after one event, a quiet Feishu WebSocket can be restarted as stale. Only publish this field from real transport or heartbeat evidence, or leave it unset for WebSocket.
  - confidence: 0.88
- **[P2] Do not arm stale-socket checks from webhook requests:** `extensions/feishu/src/monitor.transport.ts:316`
  - body: A signed webhook request proves one request was handled, but the absence of later webhook traffic is normal and does not mean a long-lived socket went stale. Since the generic policy treats any `lastTransportActivityAt` as stale-socket input, a quiet Feishu webhook can be restarted after the threshold following a valid challenge or event. Leave transport activity unset here unless there is a periodic transport health signal.
  - confidence: 0.82
