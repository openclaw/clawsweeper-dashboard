## Summary

Keep open: current main still lacks the Feishu health status path, but this PR is not merge-ready because it reports WebSocket connectivity before SDK readiness and must be rebased without dropping current main's terminal-error recovery loop.

## What This Changes

The PR threads a Feishu status sink through channel startup, monitor/account/transport code, records connection/event/transport health fields, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a rebased Feishu status-sink fix that reports WebSocket state from SDK lifecycle callbacks, preserves current terminal-error recovery, and keeps webhook activity reporting after validation.

## Reproduction Assessment

Yes. Source inspection on current main shows the missing Feishu status-sink path, and PR-head plus pinned Lark SDK source inspection shows `WSClient.start()` does not prove handshake readiness.

## Solution Assessment

No, not yet. The status-sink approach is the right narrow fix, but WebSocket health should be emitted from `onReady`/`onReconnected` and disconnect/error callbacks while retaining current main's monitor-owned retry loop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Wait for WebSocket readiness before marking connected:** `extensions/feishu/src/monitor.transport.ts:195-200`
  - body: `WSClient.start()` only stores the dispatcher and kicks off `reConnect(true)`; it does not wait for the first WebSocket handshake. Reporting `connected: true` here can make Feishu look healthy before the socket is ready, so move the connected/lastConnectedAt status update into the SDK `onReady` and `onReconnected` callbacks.
  - confidence: 0.9

This branch calls `createFeishuWSClient(account)` and then only waits for abort, so a rebase must keep the terminal-error path instead of dropping recovery.
  - confidence: 0.86
