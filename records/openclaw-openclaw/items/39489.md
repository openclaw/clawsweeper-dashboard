## Summary

Keep open: current main still lacks the Feishu health-signal plumbing, and this PR is the right narrow direction, but the PR head still publishes WebSocket `connected: true` immediately after `WSClient.start()` instead of waiting for SDK readiness/reconnect callbacks.

## What This Changes

The PR threads a Feishu status sink through channel startup, monitor orchestration, account/event handlers, and transport handlers, then adds Feishu health-status tests and a changelog entry.

## Best Possible Solution

Land one Feishu-owned implementation that keeps the status-sink plumbing, records webhook transport activity only after validation, reports WebSocket lifecycle state from Lark SDK callbacks, and preserves current main's recovery behavior.

## Reproduction Assessment

Yes. Source inspection shows current main has no Feishu status-sink path, and the PR-specific blocker is visible from the PR head plus the pinned Lark SDK source showing `start()` does not wait for the first WebSocket handshake.

## Solution Assessment

No, not yet. The status-sink direction is the narrow maintainable fix, but WebSocket health state should be emitted from `onReady`/`onReconnected`/`onReconnecting`/`onError` callbacks rather than immediately after `start()`.
