## Summary

Keep open. Current main still lacks Feishu health-status publication, and the PR remains useful, but the current PR head still publishes WebSocket connected state from `start()` instead of the Lark SDK readiness lifecycle and is behind current main's WebSocket recovery loop. That is a merge-blocking correctness issue.

## What This Changes

The PR threads a Feishu status sink through channel startup, monitor orchestration, account event handlers, and transport handlers, then adds Feishu health-status tests and a changelog entry.

## Best Possible Solution

Land one Feishu-owned implementation that keeps the status-sink plumbing, records webhook transport activity only after request/signature/body validation, reports WebSocket connected/disconnected/error state from the Lark SDK readiness and reconnect callbacks, and preserves current main's retry-exhaustion recovery behavior.

## Reproduction Assessment

Yes. Source inspection reproduces the missing current-main status-sink path, and the PR-specific bug is reproducible from the PR head plus the pinned Lark SDK source showing `start()` does not wait for the first WebSocket handshake.

## Solution Assessment

No, not yet. The status-sink direction is the right narrow fix, but WebSocket health state should be emitted from SDK `onReady`/`onReconnected`/`onReconnecting`/`onError` callbacks rather than immediately after `start()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for WebSocket readiness before marking connected:** `extensions/feishu/src/monitor.transport.ts:195-200`
