Full review comments:

- **[P1] Do not mark WebSocket events as transport liveness:** `extensions/feishu/src/monitor.account.ts:479-480`
  - body: `recordTransportActivityWithEvents: connectionMode === "websocket"` makes ordinary Feishu app events update `lastTransportActivityAt`. Current health policy treats that field as socket liveness, not app activity, so after one message a quiet but healthy Feishu WebSocket can be restarted as stale 30 minutes later. Keep WebSocket event handling to `lastEventAt`/connection state unless a true SDK transport/heartbeat signal is available.
  - confidence: 0.88
- **[P1] Preserve terminal-error reconnects when rebasing:** `extensions/feishu/src/monitor.transport.ts:185-202`
  - body: The PR head still creates the WS client without the current `onError` callback and waits only for abort after `start()`. Current main uses `waitForFeishuWsCycleEnd` so terminal SDK reconnect exhaustion closes and recreates the client; dropping that loop can bring back Feishu message gaps after a terminal WebSocket failure.
  - confidence: 0.87
- **[P2] Keep the current webhook rate-limit key helper:** `extensions/feishu/src/monitor.transport.ts:255`
  - body: The PR head builds webhook rate-limit keys from `req.socket.remoteAddress`. Current main uses `resolveRequestClientIp` plus `buildFeishuWebhookRateLimitKey`, so the refresh must keep that helper while adding activity reporting instead of falling back to the older raw-address key.
  - confidence: 0.84
