## Summary

Keep open: current main still lacks Feishu status-sink reporting, but this PR is not merge-ready because the WebSocket health signal is emitted before SDK readiness, the branch can drop current terminal-error recovery during rebase, and the external PR lacks real behavior proof.

## What This Changes

The branch threads a Feishu status sink from channel startup through monitor/account/transport code, reports connection and activity fields, adds regression tests, and updates the changelog.

## Best Possible Solution

Land one rebased Feishu-owned health-monitor fix that wires the gateway status sink end to end, reports WebSocket readiness from SDK lifecycle callbacks, preserves current terminal-error recovery, keeps webhook activity after validation, and includes focused tests plus redacted real runtime proof.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu execution. Current main does not pass a status sink from Feishu channel startup into the monitor pipeline, while the current health policy depends on connected and lastTransportActivityAt to evaluate stale/disconnected channels.

## Solution Assessment

No, not yet. The status-sink direction is the narrow maintainable fix, but this PR should report WebSocket connected state from SDK readiness callbacks and preserve current main's terminal-error recovery before it can be merged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Report WebSocket connected from readiness callbacks:** `extensions/feishu/src/monitor.transport.ts:192-200`
  - body: `WSClient.start()` stores the dispatcher and kicks off `reConnect(true)` without awaiting the first handshake; the SDK exposes `onReady` and `onReconnected` for that state. Reporting `connected: true` immediately after `start()` can mark Feishu healthy before the socket is ready, so move the connected/lastConnectedAt patch into those callbacks.
  - confidence: 0.92
- **[P2] Preserve terminal-error recovery when rebasing:** `extensions/feishu/src/monitor.transport.ts:185-202`
  - body: Current main passes `onError` into `createFeishuWSClient` and waits for terminal SDK errors so the monitor recreates the client after retry exhaustion. This branch creates the client without that callback path and then waits only for abort, so resolving the conflict this way would drop existing recovery.
  - confidence: 0.88
