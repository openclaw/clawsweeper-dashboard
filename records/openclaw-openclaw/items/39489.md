## Summary

Keep this PR open: it addresses a real Feishu health-monitor gap, but the current merge result still reports WebSocket connectivity too early and would drop the current-main terminal-error recovery loop.

## What This Changes

The PR threads a Feishu status sink through channel startup, monitor orchestration, account handlers, and transports, then adds Feishu health-status tests and a changelog entry.

## Best Possible Solution

Land the Feishu-owned status-sink implementation after rebasing onto current main, emitting WebSocket health from SDK lifecycle callbacks and keeping the existing terminal-error reconnect cycle.

## Reproduction Assessment

Yes. Source inspection shows current main has no Feishu status-sink path, and the PR merge result still marks WebSocket connected immediately after WSClient.start even though the pinned SDK only starts the asynchronous reconnect flow there.

## Solution Assessment

No, not yet. The status-sink direction is the narrow maintainable fix, but WebSocket connected/lastConnectedAt should come from Lark SDK readiness or reconnect callbacks while preserving current main's terminal-error recovery loop.
