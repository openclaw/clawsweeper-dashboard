## Summary

Keep open: the Feishu status-sink direction is useful, but the PR branch still reports WebSocket connectivity immediately after `WSClient.start()` and needs to be rebased over current main's WebSocket recovery loop before merge.

## What This Changes

The PR threads a Feishu status sink from channel startup through monitor/account/transport paths, records Feishu connection, event, and transport activity, adds Feishu regression tests, and updates the changelog.

## Best Possible Solution

Repair and rebase the Feishu status-sink implementation onto current main, emit WebSocket health state from Lark SDK lifecycle callbacks, keep webhook activity reporting after validation, and preserve the existing recovery tests.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the Feishu status-sink path, and the PR-specific blocker is visible in the PR head plus the pinned Lark SDK source showing readiness is reported by lifecycle callbacks rather than by `start()` completion.

## Solution Assessment

No, not yet. The status-sink approach is the narrow maintainable fix, but WebSocket connected status should come from `onReady`/`onReconnected` and disconnect/error callbacks while retaining current main's terminal-error retry loop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
