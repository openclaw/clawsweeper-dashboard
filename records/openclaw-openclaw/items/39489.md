## Summary

Keep open. Current main still lacks Feishu status-sink plumbing, so the PR addresses a real health-monitor gap, but the branch is not ready to merge because it reports WebSocket `connected: true` before the Lark SDK has actually completed the initial handshake and GitHub reports the PR as dirty against main.

## What This Changes

This PR threads a Feishu status sink from channel startup through the monitor stack, publishes connection/event/transport health patches, adds Feishu monitor and webhook coverage, and adds a changelog entry.

## Best Possible Solution

Land one Feishu-owned implementation that threads a typed status sink from channel startup through monitor.ts, monitor.account.ts, and monitor.transport.ts, preserves webhook activity updates only after request/signature/body validation, and reports WebSocket connected/disconnected/error state from the Lark SDK readiness/error lifecycle rather than from start() returning.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection: Feishu startup never passes ctx.setStatus beyond the initial port patch, and the monitor/transport types have no status-sink path. The PR-specific readiness bug is reproducible from the pinned Lark SDK source, where WSClient.start() schedules reConnect(true) and returns before onReady.

## Solution Assessment

No, not yet. The status-sink direction and webhook validation ordering are the right shape, but the WebSocket status update should be moved to a readiness-aware wrapper or SDK onReady/onError callbacks before the PR can be considered the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Wait for WebSocket readiness before reporting connected:** `extensions/feishu/src/monitor.transport.ts:195-199`
  - body: `WSClient.start()` in the pinned Lark SDK only stores the dispatcher and kicks off `reConnect(true)`; the initial handshake result is delivered through `onReady`/`onError`. Reporting `connected: true` immediately after this await can mark Feishu healthy even when the long connection never establishes, so wire the status patch to the SDK readiness/error lifecycle instead.
