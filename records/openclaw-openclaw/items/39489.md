## Summary

Keep open. Current main still lacks Feishu runtime health-signal wiring, and this PR remains the canonical useful implementation path after prior webhook/type review feedback was repaired. It is not merge-ready yet because the PR head reports WebSocket `connected: true` immediately after `WSClient.start()`, while the pinned Lark SDK reports actual first-handshake success/failure through `onReady` and `onError`.

## What This Changes

The PR threads a Feishu status sink through channel startup, monitor orchestration, account handlers, and transport handlers, then adds Feishu health-status tests and a changelog entry.

## Best Possible Solution

Land one Feishu-owned implementation that keeps the status sink plumbing, preserves webhook activity updates only after request/signature/body validation, and reports WebSocket connected/disconnected/error state from the Lark SDK readiness and reconnect callbacks while preserving current main's retry-exhaustion recovery logic.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the missing status-sink path, and the PR-specific bug is reproducible from the PR head plus the Lark SDK 1.62.1 source showing start() does not wait for the first WebSocket handshake.

## Solution Assessment

No, not yet. The status-sink direction and repaired webhook ordering are the right shape, but WebSocket connected state should be emitted from onReady/onReconnected and cleared or errored from onReconnecting/onError rather than immediately after start().
