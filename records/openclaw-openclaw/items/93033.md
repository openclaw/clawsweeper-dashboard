## Summary

Keep open. Current main and v2026.6.8 document that BWS exec SecretRefs require `BWS_ACCESS_TOKEN` in the Gateway service environment and can carry configured `passEnv` values into service installs, but the reported upgrade case is not fully resolved: the bundled resolver still hard-fails when that env var is absent, and I did not find an automatic repair or explicit BWS cache-to-service-env migration for old LaunchAgents that relied only on the BWS cache.

## What This Changes

The issue reports a v2026.6.6 upgrade regression where macOS LaunchAgent gateways using Bitwarden Secrets fail after cache expiry when `BWS_ACCESS_TOKEN` is absent from the service environment.

## Best Possible Solution

A maintainer should choose and document the BWS upgrade policy, then either make doctor/update fail fast with explicit `BWS_ACCESS_TOKEN` repair guidance for managed gateways or deliberately restore a bounded cache-compatible path.

## Reproduction Assessment

Source-reproducible, but not live-reproduced: the BWS resolver throws when `BWS_ACCESS_TOKEN` is absent, and service install only copies configured `passEnv` values that are already visible. The reported four-hour cache-expiry trigger still needs live macOS/BWS proof if maintainers want to validate the timing exactly.

## Solution Assessment

No, not yet: current docs and passEnv handling are useful but do not fully solve old LaunchAgents that relied on BWS cache without a service-visible token. The best path needs maintainer confirmation of strict fail-fast policy versus cache-compatible upgrade behavior.
