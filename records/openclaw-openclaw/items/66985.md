## What This Changes

The PR canonicalizes configured and per-call node selectors before comparing bound and requested exec nodes, then adds regression coverage for matching, mismatched, and missing selectors.

## Best Possible Solution

Land this canonical-selector fix, or deliberately choose the equivalent open fix path, so node-host exec compares canonical node identity while preserving fail-closed behavior for stale or mismatched bindings.

## Reproduction Assessment

Yes. Source inspection shows current main rejects a configured full-id `boundNode` plus a display-name or prefix `requestedNode` before the shared node resolver can prove both selectors name the same node; I did not run tests because this review is read-only.

## Solution Assessment

Yes. `resolveNodeExecutionTarget` is the narrow owner boundary for node selection, binding enforcement, connected/capability checks, and downstream invoke preparation, so canonicalizing both selectors there is the maintainable fix rather than changing callers or duplicating resolver policy.
