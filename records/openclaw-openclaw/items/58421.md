## Summary

Keep open: current main still lacks the plugin-facing reset helper, but this PR head exposes the mutating helper through discovery-capable API construction and still lacks real after-fix behavior proof.

## What This Changes

The PR adds optional plugin SDK `api.resetSession(key, reason?)`, updates gateway reset locking/config snapshot behavior and lifecycle broadcasts, and adds docs, changelog, API baseline, and focused contract tests.

## Best Possible Solution

Expose the helper only during full native plugin registration when `sessions.reset` exists, add discovery and tool-discovery absence coverage, keep the single reset notification path, and require redacted real plugin/gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection is enough: the PR head adds `resetSession` inside `capabilityHandlers`, while current main enables that group for `discovery` and `tool-discovery` even though those modes are documented as read-only/capability discovery.

## Solution Assessment

No for the current patch. The feature direction is plausible, but the mutating helper should be gated to full registration mode and covered by discovery/tool-discovery absence tests before the SDK contract can be judged cleanly.
