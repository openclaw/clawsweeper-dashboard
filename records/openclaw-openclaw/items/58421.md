## What This Changes

The PR adds optional plugin SDK `api.resetSession(key, reason?)`, wires it to gateway session reset, updates reset-service config/locking/broadcast behavior, docs, changelog, API baselines, and focused tests.

## Best Possible Solution

Rebase onto current main, keep the helper on a full native plugin runtime surface, make one reset notification path authoritative, and provide redacted live gateway/plugin reset proof before merge.

## Reproduction Assessment

Yes. Source inspection reproduces the review issues: PR head exposes `resetSession` through `capabilityHandlers`, while current main enables that surface for discovery modes and still has direct reset notification callers.

## Solution Assessment

No for the current head. The maintainable shape should be full-runtime-only, fit the current grouped session workflow SDK organization, and avoid split or duplicate reset broadcast ownership.

  - body: `resetSession` is added inside the `capabilityHandlers` spread, but current main enables that block for `full`, `discovery`, and `tool-discovery`; those modes are documented as discovery rather than live activation.
