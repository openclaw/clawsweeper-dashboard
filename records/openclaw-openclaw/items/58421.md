## Summary

Keep this PR open, but do not merge it yet. The API direction is still useful because current main lacks a plugin-facing reset helper, but the latest diff still exposes `api.resetSession()` in discovery and tool-discovery registration modes via the broad capability-handler block.

## What This Changes

The PR adds optional plugin-facing `api.resetSession(key, reason?)`, reuses gateway reset internals, updates SDK exports/docs/changelog/baselines, and adds focused reset-session contract tests.

## Best Possible Solution

Keep the plugin reset API direction, but expose the helper only during full registration and lock that contract with discovery and tool-discovery absence tests.

## Reproduction Assessment

Yes. Inspecting the latest PR diff against current main is enough: the helper is added under `capabilityHandlers`, and current main enables that block for `discovery` and `tool-discovery`.

## Solution Assessment

No for the current patch. The SDK helper is a narrow maintainable seam, but the implementation needs an explicit full-mode gate rather than inheriting the broader discovery-capability handler block.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Gate resetSession to full registration mode:** `src/plugins/registry.ts:2494-2496`
  - body: `resetSession` is added inside the broad `registrationCapabilities.capabilityHandlers` block, but current main enables that block for `discovery` and `tool-discovery` as well as `full`. Those discovery modes are documented as read-only capability collection paths, so a plugin that feature-detects and calls this helper during discovery can reset sessions outside live activation. Please gate this helper on `registrationMode === "full"` and add absence tests for both discovery modes.
  - confidence: 0.95
