## Summary

Keep open: the SDK direction is useful, but the latest head still exposes a session-mutating helper during discovery/tool-discovery, and the shared reset notification move needs to account for current main reset callers. The external PR also still needs real behavior proof before merge.

## What This Changes

The branch adds optional plugin SDK `api.resetSession(key, reason?)` support with public types/docs/tests and gateway reset-service changes for config snapshot reuse, in-flight guarding, and reset notifications.

## Best Possible Solution

Keep the SDK helper direction, but expose it only during full native plugin registration, add discovery/tool-discovery absence tests, and make reset notification ownership consistent across all current reset callers.

## Reproduction Assessment

Yes. Source inspection is enough: PR head adds `resetSession` under `capabilityHandlers`, current main enables that block for discovery/tool-discovery, and current main also has a reset caller that already emits `sessions.changed` after `performGatewaySessionReset`.

## Solution Assessment

No for the latest patch. The maintainable shape is a full-mode-only SDK helper plus one authoritative reset notification path, with tests covering discovery/tool-discovery absence and current reset event subscribers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Gate resetSession to full registration mode:** `src/plugins/registry.ts:2494-2496`
  - body: `resetSession` is added inside the broad `capabilityHandlers` block, but current main enables that block for `discovery` and `tool-discovery`, which are documented as read-only capability discovery modes. A discovery-loaded plugin can feature-detect and call this session-mutating helper outside live activation; gate it on `registrationMode === "full"` and add discovery/tool-discovery absence tests.
  - confidence: 0.96
- **[P2] Reconcile reset broadcasts for current reset callers:** `src/gateway/session-reset-service.ts:722-728`
  - body: Emitting lifecycle events from `performGatewaySessionReset` makes every caller broadcast `sessions.changed`. Current main already has a sessions.create reset-in-place branch that calls `performGatewaySessionReset` and then emits `sessions.changed` directly, so after rebase that path will broadcast twice unless all direct reset emits are removed or notification ownership stays in the handlers.
  - confidence: 0.88
