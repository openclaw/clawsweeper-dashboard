## Best Possible Solution

Define `resetSession` as an explicitly authorized full-runtime plugin workflow helper, guard it against discovery/inactive registry paths, make it late-callable if plugins are expected to use it after registration, and land it with focused tests plus redacted real plugin-gateway proof.

- **[P1] Keep resetSession out of discovery snapshots:** `src/plugins/registry.ts:3175-3177`
  - body: `resetSession` is added on the plugin capability-handler path, but current main enables that path for `full`, `discovery`, and `tool-discovery`. Attach it only for full active registration and guard side effects like the other workflow helpers.
  - confidence: 0.88
- **[P1] Make resetSession late-callable or keep it registration-only:** `src/plugins/api-builder.ts:288`
  - body: The new API is added to `buildPluginApi`, but `createGuardedPluginRegistrationApi` returns `undefined` for methods called after `register()` closes unless they are listed in `src/plugins/api-lifecycle.ts`. A real plugin that captures `api` and calls `api.resetSession(...)` from a hook, command, compaction flow, or other later runtime path would not get the documented Promise. Add the method to the runtime late-callable policy with a regression test, or make the contract explicitly registration-only.
  - confidence: 0.86
