Full review comments:

- **[P1] Keep one reset broadcast owner:** `src/gateway/session-reset-service.ts:1050-1053`
  - body: `performGatewaySessionReset()` now emits a lifecycle event for every reset, but existing agent and webchat reset callers still emit `sessions.changed` directly after calling it. In normal gateway mode the lifecycle subscription also broadcasts `sessions.changed`, so those reset flows can deliver duplicate reset updates to subscribers.
  - confidence: 0.94
- **[P1] Carry the selected agent into global resets:** `src/gateway/session-reset-service.ts:883`
  - body: For selected global resets, `primaryKey` is `global`, so this fallback chooses the default agent instead of the selected target agent. That means `agent:work:main` global-scope resets can build the replacement row with the wrong agent's model/session-file context and hook metadata.
  - confidence: 0.9
- **[P1] Gate resetSession behind an authority boundary:** `src/plugins/registry.ts:3176-3178`
  - body: This exposes `api.resetSession` to every full-mode plugin whenever the gateway has `sessions.reset`, with no manifest capability, operator opt-in, or plugin-owned-session limit. Because the helper can reset arbitrary session keys, the public SDK contract needs an explicit approved authority boundary before merge.
  - confidence: 0.86
