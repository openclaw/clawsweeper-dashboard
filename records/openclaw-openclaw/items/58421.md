## Summary

Keep open: the feature is not implemented on current main, and the latest PR head still has source-level blockers around discovery-mode exposure, reset notification ownership after current-main changes, and missing real behavior proof.

## What This Changes

Adds an optional plugin SDK `api.resetSession(key, reason?)` backed by gateway session reset, plus reset-service locking/config reuse, docs, changelog, API baseline, and focused tests.

## Best Possible Solution

Expose the reset helper only from full native plugin registration, align it with the current session workflow API shape, keep reset notifications owned by one path across all current callers, and require redacted live plugin/gateway proof before merge.

## Reproduction Assessment

Yes, source-level. A plugin API created with `registrationMode: "discovery"` and a `sessions.reset` gateway method still receives `resetSession` because the helper is inside `capabilityHandlers`, which is true for discovery and tool-discovery modes.

## Solution Assessment

No for the latest patch. The feature direction is plausible, but the helper is not actually full-mode-only and the shared reset broadcast change conflicts with current-main direct reset emitters unless ownership is reconciled on rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Gate resetSession to full registration mode:** `src/plugins/registry.ts:2494-2496`
  - body: `resetSession` is added inside the broad `capabilityHandlers` block, but that block is enabled for `discovery` and `tool-discovery`. Those modes are documented as capability discovery rather than live activation, and the new docs/type comment promise full-mode availability only; gate this helper on `registrationMode === "full"` and add discovery/tool-discovery absence tests.
  - confidence: 0.96
- **[P2] Keep reset notifications single-owned:** `src/gateway/session-reset-service.ts:722-728`
  - body: Emitting `emitSessionLifecycleEvent` from the shared reset service makes every reset caller broadcast through lifecycle events. Current main also has a webchat reset-in-place path that calls this service and then emits `sessions.changed` directly, so rebasing this PR can produce duplicate reset updates unless the direct emit is removed there or notification ownership stays outside the service.
  - confidence: 0.88
