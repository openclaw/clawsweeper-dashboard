## Summary

Keep open: the feature direction is plausible, but the latest head still exposes a session-mutating helper during discovery-mode plugin API construction and lacks after-fix real behavior proof.

## What This Changes

The PR adds an optional plugin SDK `api.resetSession(key, reason?)` helper, gateway reset-service in-flight/config-snapshot changes, reset lifecycle broadcast plumbing, docs/changelog/API-baseline updates, and contract tests.

## Best Possible Solution

Expose the helper only in full native plugin registration, add discovery and tool-discovery absence coverage, preserve one reset notification path, and require redacted live proof plus maintainer approval of the SDK boundary.

## Reproduction Assessment

Yes. Source inspection is enough: current main makes `capabilityHandlers` true for discovery/tool-discovery, and the PR head adds `resetSession` under that spread.

## Solution Assessment

No for the latest patch. The maintainable version should gate `resetSession` on full registration mode, cover discovery/tool-discovery absence, and treat this as a deliberate public plugin SDK/session mutation API.
