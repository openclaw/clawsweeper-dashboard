## Summary

Keep this PR open. Current main still lacks the proposed plugin-facing reset session API, and the replacement for the closed stale branch is useful, but the current patch is not merge-ready because it removes the existing `sessions.reset` change broadcast without adding an equivalent reset-service broadcast.

## What This Changes

The PR adds an optional plugin-facing `api.resetSession(key, reason?)` SDK method, result type/export, registry wiring through gateway reset support, reset-service in-flight/config-snapshot changes, generated SDK API baseline updates, and focused tests.

## Best Possible Solution

Land a narrow, documented, backwards-compatible plugin SDK reset helper that delegates reset resolution to the gateway, preserves exactly one `sessions.changed` broadcast for every successful reset path, keeps the helper absent outside full registration mode, and updates the SDK baseline, docs, changelog, and focused contract tests.

## Reproduction Assessment

Yes. A focused subscriber test on the PR branch that invokes the existing `sessions.reset` RPC should show no `sessions.changed` event after the patch removes the handler broadcast and does not add a reset-service replacement.

## Solution Assessment

No for the current patch. The overall API direction is reasonable, but the patch should first keep reset mutation broadcasting consistent, preferably by restoring the handler broadcast for `sessions.reset` while leaving plugin-only lifecycle emission in `api.resetSession()` or by centralizing all reset broadcasts exactly once.
