## Summary

Keep this PR open. Current main still lacks the proposed plugin-facing reset API, and the direction fits the documented plugin SDK contract model, but the current patch is not merge-ready because it removes the existing `sessions.reset` broadcast path while only replacing lifecycle emission for plugin-initiated resets; it also needs SDK docs, a changelog entry, and a rebase onto the current split gateway test layout.

## What This Changes

The PR adds an optional plugin-facing `api.resetSession(key, reason?)` SDK method, result type/export, registry wiring through gateway reset support, reset-service config/in-flight changes, generated SDK API baseline updates, and focused tests.

## Best Possible Solution

Land a narrow, backwards-compatible plugin SDK reset helper that delegates resolution to the gateway, is present only in full registration mode when reset is available, preserves exactly one `sessions.changed` notification for every successful reset path, and ships with SDK docs, API baseline updates, focused current-file tests, and a changelog entry.

## Reproduction Assessment

Yes. A focused subscriber test that calls the existing `sessions.reset` RPC on the PR branch should reproduce the regression by observing no `sessions.changed` event after the handler broadcast is removed and no reset-service replacement is emitted.

## Solution Assessment

No for the current patch. The API direction is reasonable, but the maintainable solution is to keep reset mutation notification semantics intact, preferably by restoring the RPC handler broadcast and emitting lifecycle only for plugin-initiated resets, or by centralizing all successful reset broadcasts exactly once.
