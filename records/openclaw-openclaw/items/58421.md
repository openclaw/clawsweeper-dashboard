## Summary

Keep open: the PR adds a useful plugin reset-session surface, but the latest diff still appears to double-emit `sessions.changed` for reset RPCs and lacks after-fix proof from a real plugin setup.

## What This Changes

Adds optional plugin SDK `api.resetSession(key, reason?)`, wires it through plugin registry to gateway session reset, updates docs/changelog/API baselines, and adds reset-session contract tests.

## Best Possible Solution

Keep the API optional and capability-gated if maintainers accept the boundary, but make reset notification ownership single-source and require redacted real plugin-to-gateway reset proof before merge.

## Reproduction Assessment

Yes for the PR defect: source inspection shows `performGatewaySessionReset()` emits a lifecycle event and the `sessions.reset` handler still emits `sessions.changed`, while current gateway runtime subscriptions forward lifecycle events to the same event name. I did not run a live gateway scenario because this review is read-only.

## Solution Assessment

No: service-level lifecycle emission is a reasonable way to support plugin-initiated resets, but the current diff leaves two broadcast owners for existing reset RPCs. The safer fix is a single reset notification path plus a real plugin proof artifact.
