## What This Changes

The PR adds an optional plugin SDK `api.resetSession(key, reason?)` helper, gateway reset-service coordination changes, SDK exports, docs, API baseline updates, changelog text, and focused tests.

## Best Possible Solution

Expose `api.resetSession` only during full native plugin activation behind `sessions.reset`, preserve the current `loadSessionEntry` options contract while adding config-snapshot reuse, cover discovery/tool-discovery and scoped lookup tests, and merge only after API-owner review plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection is enough: the PR head adds `resetSession` inside `capabilityHandlers`, current main enables those handlers for discovery modes, and the PR head changes `loadSessionEntry`'s second parameter away from the current-main options object.

## Solution Assessment

No. The direction is plausible, but the current patch is not the best form because it needs full-mode-only gating and a current-main-compatible `loadSessionEntry` options shape before the API can be reviewed safely.

- **[P1] Gate resetSession to full registration mode:** `src/plugins/registry.ts:2494-2496`
  - body: `resetSession` is added whenever `capabilityHandlers` is true, but current main sets that for `discovery` and `tool-discovery`.   - confidence: 0.9
