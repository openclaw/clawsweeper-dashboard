## Summary

Keep open: the feature is still useful and not implemented on current main, but the PR head still exposes a session-mutating helper through discovery-capable plugin API construction and lacks real behavior proof.

## What This Changes

The PR adds an optional plugin SDK `api.resetSession(key, reason?)` helper backed by gateway session resets, plus reset-service locking/config changes, SDK exports, docs, changelog, API baseline updates, and focused tests.

## Best Possible Solution

Expose `api.resetSession` only during full native plugin registration when `sessions.reset` exists, add discovery/tool-discovery absence coverage, keep the single lifecycle broadcast path, and require redacted real plugin/gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough: `resetSession` is added under `capabilityHandlers`, while current main enables that group for `discovery` and `tool-discovery` and documents those modes as discovery/read-only.

## Solution Assessment

No for the current patch. The feature direction is plausible, but the mutating helper should be full-mode gated, covered by discovery and tool-discovery absence tests, and backed by real plugin/gateway proof.
