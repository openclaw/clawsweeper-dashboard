## Summary

Keep open: the SDK direction is plausible, but the latest PR head still exposes a session-mutating helper through discovery capability handlers and needs reset-event ownership reconciled against current main before merge. The external PR also still lacks after-fix real behavior proof beyond tests and CI/local check lists.

## What This Changes

The PR adds optional plugin SDK `api.resetSession(key, reason?)`, wires it through gateway reset support, updates reset-service behavior, docs, changelog, API baselines, and focused contract tests.

## Best Possible Solution

Keep the plugin reset helper direction, but rebase onto current main, expose it only for full native registration, align it with the current session workflow facade, and make one reset notification path authoritative across all reset callers.

## Reproduction Assessment

Yes. Source inspection is enough for the review findings: PR head adds `resetSession` under `capabilityHandlers`, current main enables that surface for discovery modes, and current main has a reset-in-place caller that already emits `sessions.changed` after calling the shared reset service.

## Solution Assessment

No for the latest patch. The maintainable version should be full-mode-only, should fit current `api.session.workflow` SDK organization, and should keep reset broadcasts owned in exactly one place.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Gate resetSession to full registration mode:** `src/plugins/registry.ts:2494-2496`
  - body: `resetSession` is added inside the broad `capabilityHandlers` block, but current main enables that block for `discovery` and `tool-discovery`, which are documented as capability discovery modes rather than live activation. A discovery-loaded plugin can feature-detect and call this session-mutating helper; gate it on `registrationMode === "full"` and add discovery/tool-discovery absence tests.
  - confidence: 0.96

Current main also has a webchat reset-in-place path that calls this service and then emits `sessions.changed` directly, so after rebase that path will broadcast twice unless direct reset emits are removed or notification ownership stays in the handlers.
  - confidence: 0.88
