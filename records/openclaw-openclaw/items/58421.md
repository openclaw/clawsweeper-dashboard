## Summary

This PR should stay open, but it is not ready to merge: current main still lacks the requested plugin SDK reset helper, while the PR head still exposes a session-mutating helper during discovery/tool-discovery loads and has no real behavior proof.

## What This Changes

The PR adds optional plugin SDK `api.resetSession(key, reason?)`, adjusts gateway reset locking/config snapshot behavior, updates docs/changelog/API baseline hashes, and adds focused contract tests.

## Best Possible Solution

Expose the helper only from full native plugin registration when `sessions.reset` exists, add discovery/tool-discovery absence coverage, keep a single reset notification path, and require redacted real plugin/gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection is enough: the PR head puts `resetSession` inside `capabilityHandlers`, and current main makes that capability true for `discovery` and `tool-discovery`.

## Solution Assessment

No for the current patch. The direction is plausible, but the mutating helper should be full-mode gated and covered by discovery/tool-discovery absence tests before maintainer review can focus on the new SDK contract.
