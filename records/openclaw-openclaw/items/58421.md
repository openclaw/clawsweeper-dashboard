## Summary

Keep open: current main does not expose `api.resetSession`, and this PR is the active implementation, but the latest head still has a blocking reset-notification/type defect and no real behavior proof for the new session-mutating plugin SDK surface.

## What This Changes

This PR adds a full-mode, capability-gated plugin SDK `api.resetSession(key, reason?)` API, reset-service in-flight/config-snapshot changes, docs/changelog updates, SDK API baseline changes, and focused contract tests.

## Best Possible Solution

Keep the helper full-mode and capability-gated only if maintainers accept the SDK/session boundary, then make gateway and plugin resets share exactly one `sessions.changed` path with preserved reason semantics, tests, and real plugin proof.

## Reproduction Assessment

Yes for the PR defect: source inspection of the latest head shows the changed `sessions.reset` handler passing a payload that no longer matches `emitSessionsChanged` and coexisting with the new reset-service lifecycle emission. Real end-to-end plugin behavior is not reproduced because no real plugin proof is present.

## Solution Assessment

No: exposing the API may be a reasonable direction, but the current patch needs one reset-notification source of truth, valid changed-event payload semantics, real behavior proof, and explicit maintainer acceptance of the plugin session-reset boundary before merge.
