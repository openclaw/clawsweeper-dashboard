## Summary

Keep open. The latest PR head still exposes the session-mutating `api.resetSession` helper through the broad capability-handler surface that current main enables for discovery and tool-discovery loads, and the external PR still lacks real behavior proof.

## What This Changes

Adds optional plugin SDK `api.resetSession(key, reason?)` support with structured results, registry/gateway reset wiring, reset-service guard changes, docs, changelog, generated API baseline updates, and focused contract tests.

## Best Possible Solution

Expose the helper only in full native plugin registration, add discovery/tool-discovery absence coverage, preserve one reset notification path, and require redacted live plugin/gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection is enough: current main makes `capabilityHandlers` true for discovery and tool-discovery, and the PR head adds `resetSession` inside that spread.

## Solution Assessment

No. The API direction is plausible, but the current patch should gate `resetSession` on full registration mode and test discovery/tool-discovery absence before merge.
