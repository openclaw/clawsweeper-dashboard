## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Gate resetSession to full registration mode:** `src/plugins/registry.ts:3048-3050`
  - body: `resetSession` is added inside the `registrationCapabilities.capabilityHandlers` spread, but current main enables that bucket for `full`, `discovery`, and `tool-discovery`. That lets discovery-mode plugin code call a mutating session reset even though the new docs say the helper is full-mode-only; gate this on `registrationMode === "full"` and add discovery/tool-discovery absence tests.
  - confidence: 0.95
- **[P1] Keep reset broadcasts on one path:** `src/gateway/session-reset-service.ts:951-957`
  - body: `performGatewaySessionReset()` now emits a lifecycle event that is forwarded as `sessions.changed`, while the `sessions.reset` RPC handler still calls `emitSessionsChanged()` after success. A normal reset can therefore notify subscribers twice for the same mutation; choose one reset broadcast source and cover exactly-one-event behavior.
  - confidence: 0.91
- **[P1] Pass a valid sessions.changed payload:** `src/gateway/server-methods/sessions.ts:2063-2066`
  - body: `emitSessionsChanged` still requires a `reason` field, but this changed call passes `action: "updated"` and omits `reason`, which is enough to break the TypeScript check and matches the current red CI status. Keep the existing reason contract or update the helper and all callers consistently.
  - confidence: 0.97
