## Summary

Keep open: the SDK reset helper is useful, but the latest head is not merge-ready because it still exposes a session-mutating API through discovery-capable plugin API construction, the `sessions.reset` broadcast edit fails type checks, and no real plugin/gateway proof was provided.

## What This Changes

The PR adds optional plugin SDK `api.resetSession(key, reason?)`, gateway reset coordination changes, SDK docs/exports, generated API baseline updates, and focused reset-session tests.

## Best Possible Solution

Expose `api.resetSession` only in full native plugin registration, delegate to the gateway reset service, keep one type-correct reset notification path that preserves `reason`, cover discovery/tool-discovery absence plus reset event behavior, and require redacted real plugin/gateway proof before merge.

## Solution Assessment

No, not yet. The SDK seam may be the right direction, but the implementation should gate the helper to full registration and keep one canonical reset event path before maintainers evaluate the product boundary.
