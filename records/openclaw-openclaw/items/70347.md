## Summary

Keep open. Current main still cleans up timed-out isolated cron runs and records the cron error, but the timeout-winning path does not emit a terminal lifecycle event, so the existing gateway lifecycle persistence path is not invoked synchronously.

## What This Changes

The issue requests that a started isolated cron run whose outer timeout fires emit a terminal lifecycle error event so `sessions.json` finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session identity and timeout metadata through the existing gateway lifecycle persistence path rather than writing `sessions.json` directly from cron.

## Reproduction Assessment

Yes, at source level. A focused fake-timer harness can start an isolated cron run, capture `sessionId` and `sessionKey` via `onExecutionStarted`, keep `runIsolatedAgentJob` unresolved, advance the outer timeout, and observe cleanup plus cron error without lifecycle persistence.

## Solution Assessment

Yes. Emitting a terminal lifecycle error through the existing gateway lifecycle path is the narrow maintainable fix; direct session-store writes from cron would duplicate session lifecycle semantics.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/70347. Current main's `executeJobCoreWithTimeout` timeout-winning branch aborts the run, calls `cleanupTimedOutCronAgentRun`, and returns a cron error, but it does not emit a terminal lifecycle event that would let the gateway persist session finalization. Add a narrow cron/gateway lifecycle fix for started isolated cron runs only: carry the captured `CronAgentExecutionStarted` identity (`sessionId`, `sessionKey`, and any needed run identifier) into a `lifecycle` terminal error event with timeout metadata such as `phase: "error"`, timeout error text, `errorKind`/`stopReason` for timeout, and `endedAt`. Reuse the existing agent event/gateway lifecycle persistence path; do not add a separate direct `sessions.json` write from cron. Add or update focused regression coverage proving that a timed-out started isolated cron run emits or routes the terminal lifecycle event and finalizes the matching session row promptly, while preserving the existing cleanup behavior. Be careful to avoid duplicate/damaging finalization if the embedded runner later emits its own terminal lifecycle event. Do not change cron timeout policy, lane concurrency, provider behavior, or unrelated session status semantics.
