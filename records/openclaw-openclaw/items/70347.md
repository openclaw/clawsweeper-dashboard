## Summary

Keep open. Current main still aborts and cleans up timed-out isolated cron runs, but the timeout-winning branch returns a cron error without emitting the terminal lifecycle event that would drive synchronous session-store finalization; the related open PR overlaps direct cron session status persistence but does not fix this lifecycle-event path.

## What This Changes

The issue asks for started isolated cron runs that hit the outer cron timeout to emit a terminal lifecycle error so session persistence finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session identity and timeout metadata through the existing gateway lifecycle persistence path rather than writing `sessions.json` directly from cron.

## Reproduction Assessment

Yes, at source level. A focused harness can start an isolated cron run, have `onExecutionStarted` capture `sessionId` and `sessionKey`, keep `runIsolatedAgentJob` unresolved, advance the timeout, and observe cleanup plus cron error without lifecycle persistence.

## Solution Assessment

Yes, the requested lifecycle-event path is the narrow maintainable fix. Direct session-store writes from cron would duplicate gateway lifecycle semantics and risk diverging from normal agent event handling.
