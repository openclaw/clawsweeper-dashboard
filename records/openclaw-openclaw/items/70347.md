## Summary

Keep open. Current main still aborts and bounded-cleans timed-out isolated cron agent runs, but the timeout-winning path returns a cron error without emitting a terminal lifecycle event, so the existing gateway session finalization path is not invoked synchronously.

## What This Changes

The issue asks cron outer-timeout handling to emit a terminal lifecycle error carrying the active session identity so sessions.json finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session identity and timeout metadata through the existing gateway lifecycle persistence path with regression coverage.

## Reproduction Assessment

Yes, at source level. A focused unit harness can start an isolated cron run, call onExecutionStarted with sessionId/sessionKey, keep runIsolatedAgentJob unresolved, advance the timeout, and observe cleanup plus cron error without lifecycle persistence.

## Solution Assessment

Yes. Reusing the existing lifecycle event/session persistence path is the narrow maintainable fix; direct sessions.json writes from cron would duplicate gateway lifecycle semantics.
