## Summary

Keep open. Current main still aborts and cleans up timed-out isolated cron runs, but the timeout-winning branch does not publish a terminal lifecycle event, so the existing gateway session lifecycle persistence path is not invoked synchronously.

## What This Changes

Cron outer timeout should emit a terminal lifecycle error event for started isolated cron sessions so `sessions.json` finalizes immediately.

## Best Possible Solution

Add an injected cron timeout lifecycle-error emission for started isolated runs and route it through the existing gateway session lifecycle persistence path, with regression coverage for same-flow finalization.

## Reproduction Assessment

Yes, source-level. A fake-timer cron service harness can keep `runIsolatedAgentJob` unresolved after `onExecutionStarted` supplies `sessionId`/`sessionKey`, advance the outer timeout, and observe cleanup plus cron error without lifecycle persistence.

## Solution Assessment

Yes. Emitting a terminal lifecycle error through the existing agent-event/gateway lifecycle path is the narrow maintainable fix; direct `sessions.json` writes from cron would duplicate lifecycle semantics and risk drift.

Fix https://github.com/openclaw/openclaw/issues/70347. Current main captures `activeExecution` in `src/cron/service/timer.ts`, then on outer timeout calls `cleanupTimedOutCronAgentRun` and returns a cron error without emitting a terminal lifecycle event. Add a cron-owned terminal lifecycle error emission for started isolated runs carrying the captured `sessionId`/`sessionKey`, timeout error text, `errorKind: "timeout"`, and `stopReason: "timeout"`, routed through the existing agent-event/gateway lifecycle persistence path. Preserve architecture boundaries: prefer an injected dependency/callback from gateway or existing event-bus seam; do not write `sessions.json` directly from cron. Avoid double-finalization if the aborted embedded runner later emits its own terminal lifecycle. Do not merge or copy the direct status-persistence shape from the conflicting adjacent PR. Add a changelog fix entry if the repair is treated as user-visible.
