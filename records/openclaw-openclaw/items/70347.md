## Summary

Keep open. Current main still captures the started cron session identity and cleans up the timed-out embedded run, but the timeout-winning path returns a cron error without publishing the terminal lifecycle event that drives synchronous session persistence.

## What This Changes

The issue requests that cron outer wall-timeout paths emit a terminal lifecycle error for already-started isolated sessions so session persistence finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session id/key and timeout metadata through the existing gateway lifecycle persistence path.

## Reproduction Assessment

Yes, at source level. A fake-timer cron service harness can keep `runIsolatedAgentJob` unresolved after `onExecutionStarted` supplies `sessionId` and `sessionKey`, advance the outer timeout, and observe abort plus cleanup and cron error state without any lifecycle persistence call.

## Solution Assessment

Yes. Emitting a terminal lifecycle error through the existing gateway lifecycle path is the narrow maintainable fix; direct session-store writes from cron would duplicate lifecycle semantics and risk diverging from normal end/error handling.
