## Summary

Keep open. Current main still records and cleans up the cron timeout, but the timeout winner returns a plain error without emitting a terminal lifecycle event or preserving the active session identity into the cron finished event, so the reported synchronous session-finalization gap remains.

## What This Changes

The issue asks cron execution timeouts that fire after an isolated session has started to emit a terminal lifecycle error event so the gateway session store finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the active run session key/session id and timeout metadata through the existing agent event/session lifecycle persistence path, with regression coverage proving the session row finalizes without watchdog repair.

## Reproduction Assessment

Yes. A focused unit harness can call onExecutionStarted with a sessionId/sessionKey, keep runIsolatedAgentJob unresolved, advance the cron timeout, and observe that current main records the cron timeout and cleanup but has no lifecycle emission or sessions.json finalization assertion.

## Solution Assessment

Yes. Reusing the existing lifecycle persistence path is the narrow maintainable fix; adding a second direct sessions.json write path from cron would duplicate gateway lifecycle semantics and increase drift risk.
