## Summary

Keep open. Current main still aborts and cleans up the timed-out cron run, but the timeout-winning path does not emit a terminal lifecycle event or return the active session identity, so the reported synchronous session-store finalization gap remains.

## What This Changes

The issue requests that isolated cron execution timeouts emit a terminal lifecycle error event so `sessions.json` finalizes immediately after a started run times out.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session identity and timeout metadata through the existing agent-event/session-lifecycle persistence path with regression coverage.

## Reproduction Assessment

Yes. A focused unit harness can call `onExecutionStarted` with a `sessionId`/`sessionKey`, keep `runIsolatedAgentJob` unresolved, advance the cron timeout, and observe that current main records cleanup without lifecycle emission or session-store finalization.

## Solution Assessment

Yes. Reusing the existing lifecycle persistence path is the narrow maintainable fix; writing `sessions.json` directly from cron would duplicate gateway lifecycle semantics.
