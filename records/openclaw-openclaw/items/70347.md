## Summary

Keep open. Current main and the latest release still abort and clean up a timed-out started isolated cron run, but the timeout-winning path returns a cron error without publishing the terminal lifecycle event that drives gateway session-store finalization.

## What This Changes

The issue asks for cron outer-timeout handling to emit a terminal lifecycle error event so started isolated cron sessions finalize in `sessions.json` immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session identity and timeout metadata through the existing gateway lifecycle persistence path.

## Reproduction Assessment

Yes, at source level. A fake-timer cron service harness can keep `runIsolatedAgentJob` unresolved after `onExecutionStarted` supplies `sessionId` and `sessionKey`, advance the outer timeout, and observe abort, cleanup, and cron error state without lifecycle persistence.

## Solution Assessment

Yes. Emitting a terminal lifecycle error through the existing gateway lifecycle path is the narrow maintainable fix; direct session-store writes from cron would duplicate lifecycle semantics.

Fix https://github.com/openclaw/openclaw/issues/70347 by adding terminal lifecycle error emission for isolated cron runs whose outer timeout fires after execution has started. Keep the fix inside the cron/gateway lifecycle boundary: capture the active `sessionId`/`sessionKey` from `CronAgentExecutionStarted`, publish a lifecycle `error` event with timeout metadata through the existing agent event/gateway session lifecycle path, and avoid direct bespoke session-store writes from the cron runner. Do not solve queued pre-execution abort semantics from https://github.com/openclaw/openclaw/issues/43141, and do not reuse the direct status-write approach from https://github.com/openclaw/openclaw/pull/56624 as-is. Add focused regression coverage proving the timeout branch emits or persists terminal lifecycle state for the captured session key while preserving the existing abort/cleanup behavior.
