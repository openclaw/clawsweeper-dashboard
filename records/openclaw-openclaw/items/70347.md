## Summary

Keep open. Current main still cleans up timed-out isolated cron runs and records the cron error, but the timeout-winning path does not publish a terminal lifecycle event, so gateway session lifecycle persistence is not invoked synchronously.

## What This Changes

The issue asks for isolated cron outer timeouts to publish a terminal lifecycle error so gateway session persistence finalizes the started session immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle error emission for started timed-out isolated runs, carrying the captured session identity and timeout metadata through the existing gateway lifecycle persistence path rather than writing sessions.json directly from cron.

## Reproduction Assessment

Yes, at source level. A fake-timer cron service harness can keep runIsolatedAgentJob unresolved after onExecutionStarted supplies sessionId/sessionKey, advance the outer timeout, and observe cleanup plus cron error without a lifecycle persistence call.

## Solution Assessment

Yes. Emitting a terminal lifecycle error through the existing gateway lifecycle path is the narrow maintainable fix; direct session-store writes from cron would duplicate lifecycle semantics and risk drift.

Fix https://github.com/openclaw/openclaw/issues/70347 on current main. In src/cron/service/timer.ts, executeJobCoreWithTimeout currently wins the outer timeout race, runs cleanupTimedOutCronAgentRun, and returns a cron error without publishing a terminal lifecycle event. Add a narrow cron/gateway-owned path that emits or routes a lifecycle error for started isolated agent runs using the captured CronAgentExecutionStarted sessionId/sessionKey and timeout metadata, so existing gateway lifecycle persistence finalizes the session synchronously. Prefer reusing the server-chat/session-lifecycle event path; do not add a bespoke sessions.json write from cron. Avoid double-finalizing if the embedded runner later emits its own terminal event. Add focused regression coverage for a started isolated run that ignores abort: after advancing the timeout, assert cleanup still happens and the terminal lifecycle/persistence path receives the timeout event with sessionKey, stopReason/errorKind timeout semantics, and the cron error remains recorded. Coordinate with adjacent open PRs only as context; do not implement unrelated runTimeoutSeconds, reaper recovery, or session_end hook feature work.
