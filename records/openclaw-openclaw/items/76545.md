## Summary

Keep this PR open: current main still skips stale base cron sessions, and the predicate swap is the right core surface, but the PR needs the required changelog entry and focused regression coverage before it should land.

## What This Changes

The PR changes the cron session reaper to use `isCronSessionKey`, so retention sweeps both base `agent:...:cron:<jobId>` entries and `...:run:<runId>` entries.

## Best Possible Solution

Land the predicate change with a regression test for expired base cron entries, align the reaper wording with the new scope, and add an Unreleased changelog fix entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: create an expired `agent:main:cron:<jobId>` store entry and run `sweepCronRunSessions`; current main skips it because `isCronRunSessionKey` rejects base cron keys.

## Solution Assessment

Mostly yes. Switching to `isCronSessionKey` is the narrowest maintainable code fix for the reported filter bug, but the PR should also add regression coverage, a changelog entry, and updated run-only wording.
