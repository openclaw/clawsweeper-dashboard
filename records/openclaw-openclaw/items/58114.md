## Summary

Keep open. Current main still has the reported gap: Voice Call’s Twilio stream path forwards raw text into telephony synthesis, while the telephony runtime/provider contract has no directive-derived override path. The existing message-channel TTS path and docs already support `[[tts:...]]`, so the remaining work is targeted wiring for telephony, not cleanup closure.

## What This Changes

The issue requests parsing and applying `[[tts:voiceId=...]]` directives in the bundled Voice Call telephony TTS path so per-utterance voice/model overrides work during calls.

## Best Possible Solution

Add directive handling to the generic telephony TTS flow so Voice Call and any other telephony caller get the same single-utterance override semantics as regular TTS: strip directive text, honor the existing modelOverrides policy, route provider selection when allowed, and pass providerOverrides through telephony preparation and provider synthesis.

## Repair Work Prompt

Fix the Voice Call telephony TTS directive gap from https://github.com/openclaw/openclaw/issues/58114. Current main forwards raw text containing `[[tts:...]]` into telephony synthesis. Implement a narrow additive fix that reuses the existing TTS directive parser/modelOverrides policy and does not introduce a second parser. Prefer handling this in the generic telephony TTS runtime flow where `resolveTtsConfig`, provider configs, and `parseTtsDirectives` already exist: strip directive text before telephony synthesis, use `ttsText` when present, honor `provider=...` only under the existing policy, and pass the selected provider override bag into `prepareSpeechSynthesis` and `synthesizeTelephony`.
