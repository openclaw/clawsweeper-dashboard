## Summary

Keep open: current main still only partially addresses the report. Generic `openai-responses` payloads already use array `input` and `stream: true`, but there is still no stable opt-in for fallback top-level `instructions` or for omitting `max_output_tokens` on stricter proxy endpoints, and the linked duplicate was closed in favor of this tracker.

## What This Changes

The issue requests stable `openai-responses` request-normalization options for stricter Responses-compatible proxy endpoints, including fallback `instructions`, array input/stream handling, and omitting unsupported fields such as `max_output_tokens`.

## Best Possible Solution

Add opt-in model-compat and payload-policy controls for fallback `instructions` and `max_output_tokens` omission, with schema, catalog normalization, tests, and docs while preserving native defaults.

## Reproduction Assessment

No, not with a high-confidence live reproduction against a strict third-party relay. Source inspection does confirm that generic `openai-responses` still lacks fallback top-level `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the requested opt-in direction: model compat plus payload policy is the narrowest maintainable place for proxy-specific normalization. The exact config names and defaults still need maintainer/API approval before implementation.
