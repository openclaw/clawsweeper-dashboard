## Summary

Keep open. Current main partially covers the report with array-shaped `input`, `stream: true`, prompt-cache stripping, and native Codex request sanitization, but generic `openai-responses` proxy models still lack stable opt-in controls for fallback top-level `instructions` and omitting `max_output_tokens`.

## What This Changes

The issue requests configurable OpenAI Responses request normalization for third-party compatible endpoints, including fallback instructions and optional omission of unsupported fields.

## Best Possible Solution

Add per-model Responses proxy compatibility fields for fallback top-level `instructions` and omitting `max_output_tokens`, wire them through config/schema/catalog normalization/payload policy/transport/docs/tests, and preserve existing native defaults.

## Reproduction Assessment

No, not for the exact relay-specific HTTP 400 because it depends on a stricter third-party Responses-compatible endpoint. Static inspection does give high confidence that current main cannot configure the remaining fallback `instructions` or omit-`max_output_tokens` behavior.

## Solution Assessment

Yes, an opt-in per-model compatibility layer is the narrowest maintainable direction. Global fallback instructions or globally omitting token caps would risk regressions for native OpenAI, Codex, Azure Responses, and compatible proxies.
