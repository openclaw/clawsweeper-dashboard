## Summary

Keep open: current main still only partially covers the reported Responses proxy compatibility problem. The transport already sends array `input` and `stream: true`, but there is still no stable config/payload-policy control for generic `openai-responses` fallback `instructions` or for omitting `max_output_tokens`; https://github.com/openclaw/openclaw/issues/10140 is already closed as the duplicate that points here.

## What This Changes

Request stable OpenAI Responses proxy compatibility configuration for normalizing request payloads, including fallback instructions and optional omission of unsupported fields.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, docs, and changelog while preserving native provider defaults.
