## Summary

Keep open: current main still only partially covers the Responses proxy compatibility request. Generic `openai-responses` payloads already use array `input` and `stream: true`, but there is still no stable opt-in for top-level fallback `instructions` or omitting `max_output_tokens` for stricter proxy endpoints.

## What This Changes

Request stable request-normalization compatibility controls for strict third-party `openai-responses` proxy or relay endpoints.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, public docs, and changelog while preserving native provider defaults.

## Reproduction Assessment

No, not with a high-confidence live reproduction against a strict third-party relay. Source inspection does confirm that generic `openai-responses` payloads still lack fallback `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the requested opt-in direction: a model-compat and payload-policy change is narrower than changing default native OpenAI Responses payloads. The exact stable config names and defaults still need maintainer approval before implementation.
