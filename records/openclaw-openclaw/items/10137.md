## Summary

Keep open: current main still only partially covers the Responses proxy compatibility request. The transport already sends array `input` and `stream: true`, but generic `openai-responses` models still lack stable opt-in top-level fallback `instructions` and a way to omit `max_output_tokens`; the duplicate thread points back here rather than superseding it.

## What This Changes

The issue requests stable request-normalization and compatibility options for third-party `openai-responses` proxy or relay endpoints that reject OpenClaw's current Responses payload shape.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, public docs, and changelog while preserving native provider defaults.

## Reproduction Assessment

No high-confidence live reproduction is available because I did not establish a third-party relay returning the reported 400/no-body failure. Source inspection does confirm that generic `openai-responses` payloads still lack fallback `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the direction: an opt-in model-compat and payload-policy change is narrower than changing default native OpenAI Responses payloads. The exact stable config names and defaults still need maintainer confirmation before implementation.
