## Summary

Keep open. Current main covers some Responses compatibility pieces, including array `input`, `stream: true`, prompt-cache stripping controls, and native Codex payload sanitization, but generic `openai-responses` proxy models still cannot opt into fallback top-level `instructions` or omit `max_output_tokens`; the prior comments correctly treat this as the canonical tracker after #10140.

## What This Changes

The issue requests stable per-model compatibility normalization for strict OpenAI Responses-compatible proxy endpoints that return HTTP 400/no body.

## Best Possible Solution

Add per-model Responses proxy compatibility controls for fallback top-level `instructions` and `max_output_tokens` omission, wired through config/schema/catalog normalization/payload policy/transport/docs/tests while preserving native defaults.

## Reproduction Assessment

No high-confidence live reproduction exists here for the exact relay-specific HTTP 400 because it depends on a strict third-party Responses-compatible endpoint. Source inspection does give a high-confidence path for the remaining config gap: generic `openai-responses` has no fallback-instructions or omit-`max_output_tokens` control on current main.

## Solution Assessment

Yes, the best direction is a narrow opt-in per-model compatibility layer. Global top-level instructions or globally omitting token caps would risk regressions for native OpenAI, Codex, Azure Responses, and proxies that accept the current payload.
