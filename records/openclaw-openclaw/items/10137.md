## Summary

Keep open. Current main covers several adjacent Responses fixes, including array `input`, `stream: true`, prompt-cache stripping controls, and native Codex `instructions`/payload sanitization, but generic `openai-responses` proxy models still cannot opt into fallback top-level `instructions` or omit `max_output_tokens`.

## What This Changes

The issue requests stable `openai-responses` request normalization and compatibility options for stricter Responses-compatible proxy endpoints.

## Best Possible Solution

Add per-model Responses proxy compatibility fields for fallback top-level `instructions` and omitting `max_output_tokens`, wire them through config/schema/catalog normalization/payload policy/transport/docs/tests, and preserve existing native defaults.

## Reproduction Assessment

No, not for the exact relay-specific HTTP 400 because it depends on a stricter third-party Responses-compatible endpoint. Static inspection does give high confidence that current main cannot configure the remaining fallback `instructions` or omit-`max_output_tokens` behavior.

## Solution Assessment

Yes, an opt-in per-model compatibility layer is the narrowest maintainable direction. Global fallback instructions or globally omitting token caps would risk regressions for native OpenAI, Codex, Azure Responses, and proxies that accept the current payload.
