## Summary

Keep open: current main still only partially covers the Responses proxy compatibility request. It already sends array `input` and `stream: true`, but generic `openai-responses` proxy payloads still lack stable opt-in fallback `instructions` and omit-`max_output_tokens` controls, while the closed duplicate at https://github.com/openclaw/openclaw/issues/10140 points back here as the canonical tracker.

## What This Changes

Request stable `openai-responses` request-normalization and compatibility options for third-party Responses-compatible proxy endpoints that require top-level `instructions` or reject fields such as `max_output_tokens`.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, public docs, and changelog while preserving native OpenAI, Codex, and Azure defaults.

## Reproduction Assessment

No high-confidence live reproduction is available because I did not establish a third-party relay returning the reported 400/no-body failure. Source inspection on current main does confirm the generic `openai-responses` payload still lacks fallback `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the direction: opt-in model-compat and payload-policy controls are narrower than changing default native Responses payloads. The exact stable config names and defaults still need maintainer confirmation before implementation.
