## Summary

Keep open. Current main still only partially covers the reported Responses-compatible proxy problem: payloads are array-shaped and streamed, and prompt-cache stripping is configurable, but there is still no stable opt-in control for generic `openai-responses` fallback `instructions` or for omitting `max_output_tokens` on stricter relays. The existing discussion already treated this issue as the canonical tracker after #10140 closed, and the latest read-only search did not find an open PR with closing syntax for the remaining work.

## What This Changes

Requests stable per-model compatibility options for `openai-responses` proxy/relay endpoints to normalize request shape and optionally omit unsupported fields such as `max_output_tokens`.

## Best Possible Solution

Add explicit per-model Responses proxy compatibility fields for fallback top-level `instructions` and omitting `max_output_tokens`, wire them through config types/schema, catalog normalization, payload policy, transport construction, focused tests, generated config docs, provider/local-model docs, and changelog, while preserving all native defaults.

## Reproduction Assessment

No, not for the relay-specific HTTP 400 because it depends on a stricter third-party Responses-compatible endpoint. Static inspection does give high confidence that current main cannot configure the remaining fallback `instructions` or omit-`max_output_tokens` behavior.

## Solution Assessment

Yes, an opt-in per-model compatibility layer is the narrowest maintainable solution. Applying fallback instructions or omitting token caps globally would risk regressions for native OpenAI, Codex, Azure Responses, and compatible proxies that accept the current payload.
