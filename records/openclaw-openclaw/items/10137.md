Keep open: current main still only partially covers the requested Responses compatibility behavior. The transport already sends array `input` and `stream: true`, but generic `openai-responses` proxy models still have no stable opt-in controls for fallback `instructions` or omitting `max_output_tokens`.

## What This Changes

Request stable configuration for stricter `openai-responses` proxy normalization, including fallback `instructions` and optional omission of unsupported fields such as `max_output_tokens`.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, docs, and changelog while keeping native provider defaults unchanged.
