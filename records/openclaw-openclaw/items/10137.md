## Summary

Keep open: current main only partially addresses the reported Responses proxy compatibility problem. The transport already sends array `input` and `stream: true`, but generic `openai-responses` proxy payloads still have no stable opt-in fallback `instructions` control and no way to omit `max_output_tokens`; the only related item found is closed as a duplicate pointing here.

## What This Changes

The issue requests stable `api: "openai-responses"` compatibility normalization for stricter proxy endpoints that require `instructions`, array `input`, `stream: true`, or omission of unsupported fields such as `max_output_tokens`.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, public docs, and changelog while preserving native provider defaults.

## Reproduction Assessment

No high-confidence live reproduction is available because I did not establish a third-party relay returning the reported 400/no-body failure. Source inspection does confirm the current generic `openai-responses` payload still lacks fallback `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the direction: an opt-in model-compat and payload-policy change is narrower than changing default native OpenAI Responses payloads. The exact stable config names and defaults still need maintainer confirmation before implementation.
