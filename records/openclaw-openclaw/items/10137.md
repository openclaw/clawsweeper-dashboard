## Summary

Keep open: current `main` still only partially addresses the reported Responses proxy compatibility problem. The transport already sends array `input` and `stream: true`, but generic `openai-responses` proxy payloads still have no stable opt-in fallback `instructions` control and no way to omit `max_output_tokens`; the closed duplicate points back to this issue as the canonical tracker.

## What This Changes

Request stable compatibility controls for generic `openai-responses` proxy payloads that may require fallback `instructions`, array input, streaming, or omission of unsupported fields.

## Best Possible Solution

Add narrow opt-in Responses proxy compatibility fields through model config/schema, catalog normalization, payload policy, transport construction, tests, public docs, and changelog while preserving native provider defaults.

## Reproduction Assessment

No high-confidence live reproduction is available because I did not establish a third-party relay returning the reported 400/no-body failure. Source inspection against current `main` does confirm the generic `openai-responses` payload still lacks fallback `instructions` and cannot omit `max_output_tokens`.

## Solution Assessment

Yes for the direction: an opt-in model-compat and payload-policy change is narrower than changing default native Responses payloads. The exact stable config names and defaults still need maintainer confirmation before implementation.
