## Summary

Close as implemented on main. The requested memory-core-independent embedding provider SDK surface now exists as the generic `openclaw/plugin-sdk/embedding-providers` contract, current memory-core can consume generic providers, and the implementation is present in shipped releases.

## What This Changes

Request a stable public plugin SDK path for embedding provider APIs so third-party memory-slot plugins do not depend on memory-core-owned internals.

## Best Possible Solution

Keep the shipped generic `plugin-sdk/embedding-providers` and `contracts.embeddingProviders` seam; handle narrower provider-specific bugs as separate follow-up issues or PRs.

## Reproduction Assessment

Not applicable: this is a feature/API-surface request, not a broken-behavior bug report. Source and docs inspection verify current main now exposes the requested stable provider path and memory bridge.

## Solution Assessment

Yes. The shipped solution uses a narrow generic `plugin-sdk/embedding-providers` subpath plus manifest-owned registration instead of freezing a memory-core-owned alias, which matches the plugin SDK boundary better than the original sketch.
