## Summary

Keep open. Current main still exposes the embedding helper surface only through the memory-core-named SDK subpath, while package exports, SDK entrypoint metadata, docs, and bundled imports do not contain a neutral `openclaw/plugin-sdk/embeddings` path. The closed related PR attempted this change but was not merged, and this remains a core Plugin SDK boundary/API-shape question rather than ClawHub-only plugin work.

## What This Changes

Request to expose the memory embedding provider API through a stable documented Plugin SDK subpath that is independent of the `memory-core` plugin lifecycle.

## Best Possible Solution

Add an owner-approved, additive `openclaw/plugin-sdk/embeddings` facade backed by the generic memory-host embedding contracts/helpers, keep `plugin-sdk/memory-core-host-engine-embeddings` backward compatible, document the preferred path and migration note, update package exports/entrypoint metadata/API baseline, and cover the contract with focused SDK/provider tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the generic SDK file/export/docs row are absent, while the existing embedding facade and bundled imports still use `plugin-sdk/memory-core-host-engine-embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears maintainable and consistent with the SDK boundary rules, but the exact public contract and old-path deprecation scope should be approved before implementation.
