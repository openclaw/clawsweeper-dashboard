## Summary

Keep open. Current main still lacks a generic stable embeddings SDK subpath; the only public embedding helper surface remains the memory-core-named facade, and related PR #67242 is closed unmerged. This is an OpenClaw SDK boundary/API-shape question rather than ClawHub-only plugin work, so it needs maintainer/API-owner direction before implementation.

## What This Changes

Request to expose the embedding provider API through a stable `openclaw/plugin-sdk/embeddings` subpath that works independently of the memory-core plugin lifecycle while preserving the existing memory-core-named path for compatibility.

## Best Possible Solution

Add an owner-approved, additive `openclaw/plugin-sdk/embeddings` facade for the agreed embedding contracts/helpers, keep `plugin-sdk/memory-core-host-engine-embeddings` backward compatible, document the preferred path, update package exports/SDK entrypoint metadata/API baseline, and lock it with focused SDK/provider contract tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static source inspection is enough: an import of `openclaw/plugin-sdk/embeddings` has no source file, package export, entrypoint metadata, or docs row on current main, while the memory-core-named facade remains present.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears consistent with the SDK boundary rules, but the exported symbol set and old-path deprecation policy should be approved before a fix PR is queued.
