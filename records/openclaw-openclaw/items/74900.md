## Summary

Keep open. Current main still lacks a generic stable embeddings SDK subpath, and the only published/documented facade for these helpers remains the memory-core-named path. The request is a plausible Plugin SDK contract change, but because it creates a new public SDK surface and the related implementation PR is closed unmerged, it needs maintainer/API-owner agreement before cleanup automation should implement it.

## What This Changes

Expose a stable, documented embeddings SDK subpath independent of memory-core so third-party memory-slot plugins can use shared embedding factories, registry APIs, types, and batch helpers.

## Best Possible Solution

Add an owner-approved, additive generic embeddings SDK facade; preserve the existing memory-core-named path for compatibility; document the preferred import and any deprecation wording; align package exports, SDK entrypoint metadata, API/export checks, focused contract tests, and a user-facing changelog entry.

## Reproduction Assessment

Yes. Static inspection against current main reproduces the gap: there is no openclaw/plugin-sdk/embeddings source file, package export, SDK entrypoint metadata entry, or docs row, while the memory-core-named facade remains present and documented.

## Solution Assessment

Unclear pending maintainer/API-owner input. A narrow additive SDK subpath appears consistent with the plugin boundary, but the public name, exact export set, and compatibility wording should be settled before implementation.
