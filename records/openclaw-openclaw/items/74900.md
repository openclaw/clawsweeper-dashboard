## Summary

Keep open. Current main still lacks the requested plugin-neutral embeddings SDK subpath, the linked implementation PR was closed unmerged, and the remaining decision is a public SDK contract/API-shape question for SDK and memory owners.

## What This Changes

Expose the embedding provider API through a stable plugin-neutral Plugin SDK subpath that works independently of memory-core being loaded, while preserving the old memory-core-named path for compatibility.

## Best Possible Solution

Ship an owner-approved additive `openclaw/plugin-sdk/embeddings` facade, or explicitly document the existing path as the stable contract, with compatibility and SDK metadata/docs/checks kept in sync.

## Reproduction Assessment

Yes for the SDK gap: static inspection on current main shows no `src/plugin-sdk/embeddings.ts`, no package export/entrypoint metadata for it, and docs still listing only the memory-core-named facade. This is a feature/API request rather than a failing runtime bug.

## Solution Assessment

Unclear until SDK and memory owners choose the public contract shape. A narrow additive plugin-neutral facade looks maintainable, but the name and export set should be approved before implementation.
