## Summary

Keep open. Current main still exposes the embedding helper surface only through the memory-core-named SDK subpath, and the requested generic `openclaw/plugin-sdk/embeddings` contract is absent from source, package exports, SDK entrypoint metadata, and docs. This is an SDK boundary decision, not a ClawHub-only plugin request, and the linked implementation PR was closed unmerged after being judged too broad.

## What This Changes

The issue requests a stable public plugin SDK subpath for embedding provider APIs that third-party memory-slot plugins can use without depending on memory-core-owned imports.

## Best Possible Solution

Add an owner-approved, additive `openclaw/plugin-sdk/embeddings` facade for the agreed embedding contracts/helpers, keep `plugin-sdk/memory-core-host-engine-embeddings` backward compatible, document the preferred path, update package exports and SDK entrypoint metadata/API baseline, cover the contract with focused SDK/provider tests, and include a user-facing changelog entry if the surface changes.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the reported SDK gap: the generic subpath is absent while the existing helper surface is published and documented only under `memory-core-host-engine-embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears consistent with the plugin SDK boundary rules, but the exact stable contract and compatibility/deprecation scope should be approved before a repair PR is queued.
