## Summary

Keep open. Current main still publishes only the memory-core-named embedding SDK facade; the linked implementation PR was closed unmerged, and the remaining work needs SDK/memory owner agreement on a new public contract shape.

## What This Changes

Expose embedding provider helpers and registry APIs through a stable, plugin-neutral `openclaw/plugin-sdk/embeddings` subpath that works independently of the memory-core plugin lifecycle.

## Best Possible Solution

Ship an owner-approved additive `openclaw/plugin-sdk/embeddings` facade, keep the current memory-core-named path compatible or explicitly documented, and update exports, SDK metadata, docs, API checks, focused contract coverage, and changelog in one narrow change.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the requested SDK gap: `plugin-sdk/embeddings` is absent from source, package exports, entrypoint metadata, and docs, while the old memory-core-named facade remains published.

## Solution Assessment

Unclear pending SDK/memory owner input. A narrow additive plugin-neutral facade matches the SDK boundary guidance, but the public name and exact export set should be approved before implementation.
