## Summary

Keep open. Current main still lacks the requested plugin-neutral embeddings SDK subpath, the earlier implementation PR was closed unmerged with a request for a narrower owner-backed change, and the remaining decision is a public SDK contract shape for SDK and memory owners.

## What This Changes

Request a stable `openclaw/plugin-sdk/embeddings` subpath so third-party memory-slot plugins can use embedding helpers without depending on the memory-core-named facade.

## Best Possible Solution

Ship an owner-approved additive `openclaw/plugin-sdk/embeddings` facade, or explicitly bless the existing memory-core-named facade as the stable contract, with package exports, SDK metadata, docs, and compatibility checks kept aligned.

## Reproduction Assessment

Yes for the SDK gap: current main has no `src/plugin-sdk/embeddings.ts`, no `./plugin-sdk/embeddings` package export, and docs/metadata still list only `plugin-sdk/memory-core-host-engine-embeddings`. This is a feature/API-surface request rather than a runtime bug.

## Solution Assessment

Unclear until SDK and memory owners choose the public contract shape. A narrow additive facade looks maintainable, but the name and export set should be approved before implementation because this expands supported SDK surface.
