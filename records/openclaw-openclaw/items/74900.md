## Summary

Keep open. Current main still lacks a generic `openclaw/plugin-sdk/embeddings` subpath; the only published/documented embeddings facade remains the memory-core-named SDK path. The runtime now has manifest-backed cold lookup for memory embedding providers, so the remaining question is an SDK contract/API naming decision, and the related implementation PR #67242 is closed unmerged.

## What This Changes

Request to expose memory embedding provider factories, registry helpers, types, and batch helpers through a stable plugin-neutral SDK subpath independent of the memory-core plugin lifecycle.

## Best Possible Solution

Add an owner-approved additive generic embeddings SDK facade, keep `memory-core-host-engine-embeddings` backward compatible, document the preferred import and compatibility status, update package exports and SDK entrypoint metadata, refresh API/export checks, add focused contract coverage, and include a user-facing changelog entry.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the gap: the generic `openclaw/plugin-sdk/embeddings` specifier is absent from source, package exports, entrypoint metadata, docs, and tests while the memory-core-named facade remains present.

## Solution Assessment

Unclear until a maintainer/API owner settles the public contract. A narrow additive SDK subpath appears maintainable and consistent with the SDK boundary rules, but the exact name, export set, and old-path deprecation language should be approved first.
