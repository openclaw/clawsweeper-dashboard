## Summary

Keep open. Current main still lacks a generic stable embeddings SDK subpath; the only published/documented embedding helper surface remains the memory-core-named facade, and the related implementation PR is closed unmerged. Because this would create a new public SDK contract, the next step is maintainer/API-owner agreement on the subpath, export set, and compatibility wording before implementation.

## What This Changes

The issue requests an additive, documented plugin SDK embeddings subpath that exposes embedding factories, registry helpers, types, and batch helpers independently of whether memory-core is the active memory plugin.

## Best Possible Solution

Add an owner-approved, additive generic embeddings SDK facade, keep the existing memory-core-named path backward compatible, document the preferred import and any deprecation status, update export metadata/API baseline, and cover the contract with focused SDK/provider tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static source inspection is enough to reproduce the gap: `openclaw/plugin-sdk/embeddings` has no source file, package export, entrypoint metadata, or docs row on current main, while the memory-core-named facade remains present.

## Solution Assessment

Unclear pending maintainer/API-owner input. A narrow additive generic subpath appears consistent with the SDK boundary rules, but the exact export set and compatibility wording should be approved before a fix PR is queued.
