## Summary

Keep open. Current main still lacks the requested generic `openclaw/plugin-sdk/embeddings` SDK subpath; the published source, package exports, entrypoint metadata, and docs continue to expose only the memory-core-named embedding facade. This is an OpenClaw SDK boundary/API-shape request, not a ClawHub-only plugin idea, and the related implementation PR was closed unmerged pending a narrower owner-backed shape.

## What This Changes

Expose a stable, documented `openclaw/plugin-sdk/embeddings` subpath so third-party memory-slot plugins can use embedding provider contracts/helpers without importing a memory-core-named facade.

## Best Possible Solution

Add an owner-approved, additive embeddings SDK facade for the agreed stable contracts/helpers, keep `plugin-sdk/memory-core-host-engine-embeddings` backward compatible, update package exports/entrypoint metadata/API baseline/docs, and lock the new public path with focused SDK/provider contract coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static source inspection is enough: `openclaw/plugin-sdk/embeddings` has no source file, package export, entrypoint metadata, or docs row on current main, while the memory-core-named facade remains present.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears maintainable and consistent with the SDK boundary rules, but the exported symbol set and compatibility language should be approved before a fix PR is queued.
