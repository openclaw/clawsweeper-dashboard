## Summary

Keep open. Current main still lacks a plugin-neutral `openclaw/plugin-sdk/embeddings` subpath; the only published and documented embedding SDK facade remains the memory-core-named path, and the related implementation PR #67242 was closed unmerged pending a narrower owner-backed shape.

## What This Changes

Expose embedding provider helpers and registry APIs through a stable plugin-sdk subpath independent of the bundled memory-core plugin lifecycle.

## Best Possible Solution

Add an owner-approved additive `openclaw/plugin-sdk/embeddings` facade, keep the current memory-core-named path compatible, and align exports, entrypoint metadata, docs, API checks, tests, and changelog together.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the gap: `plugin-sdk/embeddings` is absent from source, package exports, entrypoint metadata, and docs, while `plugin-sdk/memory-core-host-engine-embeddings` remains the only published embedding facade.

## Solution Assessment

Unclear pending SDK and memory owner input. A narrow additive generic subpath fits the SDK boundary direction, but the exact public name, export set, and deprecation wording should be approved before implementation.
