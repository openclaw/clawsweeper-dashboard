## Summary

Keep open. Current main still lacks a generic stable embeddings SDK subpath; the available public import remains the memory-core-named facade, and the related implementation PR #67242 is closed unmerged. This is a plausible Plugin SDK contract change, but it needs maintainer/API-owner agreement before automation should implement it.

## What This Changes

Request to expose memory embedding provider helper and registry APIs through a stable generic plugin SDK subpath independent of the memory-core plugin lifecycle.

## Best Possible Solution

Add an owner-approved, additive generic embeddings SDK facade; keep the existing memory-core-named path backward compatible; document the preferred import and any deprecation status; update package exports, entrypoint metadata, API/export checks, focused contract tests, and a user-facing changelog entry.

## Reproduction Assessment

Yes. Static source inspection reproduces the gap: `openclaw/plugin-sdk/embeddings` is absent from source, package exports, SDK metadata, and docs on current main, while only the memory-core-named facade exists.

## Solution Assessment

Unclear pending maintainer/API-owner input. A narrow additive SDK subpath appears maintainable and consistent with the plugin SDK boundary rules, but the public name, export set, and compatibility wording should be chosen before implementation.
