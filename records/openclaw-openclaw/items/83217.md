## Summary

Keep open: the request is coherent, current main and the latest release still lack first-class WhatsApp list prompt extraction and a row-id list reply action, and the related merged pairing/content-gate work only covers interactive responses after a user selection.

## What This Changes

Request support for exposing inbound WhatsApp interactive list rows/row ids to agents and sending outbound list replies by selected row id.

## Best Possible Solution

Implement this inside the WhatsApp plugin boundary: parse listMessage prompts into readable body plus untrusted structured context, keep the inbound gate wrapper-aware, add a validated row-id list-reply action through the existing message action/config contracts, and cover it with focused WhatsApp extension tests and docs.

## Solution Assessment

Yes in direction, but the API shape needs maintainer judgment. The maintainable path is a WhatsApp plugin implementation plus small shared message-action/config updates, not raw Baileys payload exposure or plain-text guessing.
