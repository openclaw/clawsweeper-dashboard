## Label Justifications

- `P1`: The reported behavior can delay real inbound, outbound, and stop/control messages for minutes on active channel workflows.
- `impact:message-loss`: Messages are not permanently deleted, but they can be silently queued long enough to look lost or miss the useful dispatch window.
- `impact:session-state`: The affected code path is the per-session followup queue and serialized session lane that protect transcript and run ordering.

## Summary

Keep open. Current main still queues active non-heartbeat followup work into the same FIFO drain, and the drain still runs until empty before the next priority decision; recent session-lane priority work helps direct enqueue ordering but does not give fresh user/control messages a followup-queue preemption point.

## What This Changes

The issue asks for session-lane fairness so followup drain cannot monopolize inbound and outbound dispatch for long periods.

## Best Possible Solution

Define a scheduler-level fairness contract that lets fresh user/control work win over accumulated background/followup backlog while preserving per-session serialization, route metadata, collect batching, dedupe, and queue cap/drop semantics.

## Reproduction Assessment

Yes, at source level: active non-heartbeat work can still enqueue into the same FIFO followup queue, and the drain runs that queue until empty. I did not run the full 20-30 minute live multi-channel reproduction in this read-only pass.

## Solution Assessment

Unclear on the exact implementation. The best direction is scheduler-level fairness; isolated session-lane concurrency or command-lane priority alone does not address the followup queue's selection and preemption problem.
