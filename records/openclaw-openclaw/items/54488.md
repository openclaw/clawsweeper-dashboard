## Summary

Keep open. Current main still has the core fairness problem: busy-session inbound work is appended to the same FIFO followup queue, the drain runs until that queue is empty, and every drained followup still re-enters the same per-session command lane.

## What This Changes

The issue reports that followup queue draining can monopolize the per-session lane and delay fresh inbound or outbound messages for 20-30 minutes.

## Best Possible Solution

Add scheduler-level fairness to the auto-reply followup path so fresh user/control work can run ahead of accumulated system/followup backlog while preserving FIFO ordering within each class and existing safety/routing checks.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: keep a session active, enqueue followup backlog, then send fresh inbound work and it is appended behind the same FIFO drain that re-enters the per-session lane; I did not run the full 20-30 minute live gateway scenario.

## Solution Assessment

Yes. A narrow core scheduler-fairness fix is the maintainable solution; lane timeouts, reset-command bypasses, context-engine background maintenance, and heuristic fast-lane modes reduce adjacent pain but do not fix backlog monopolization for normal fresh messages.
