## Summary

Keep this open: current `main` still admits active non-heartbeat user work into the same follow-up queue and drains that queue through the same session lane, so the central starvation problem is not implemented away.

## What This Changes

This issue reports that background/follow-up drain work can monopolize a session lane for 20-30 minutes, delaying inbound Discord, WhatsApp, outbound send, and stop/control messages.

## Best Possible Solution

Define and implement a scheduler-level fairness policy so fresh foreground user/control turns can overtake, interrupt, or defer accumulated background/follow-up work on the same session key while preserving deterministic transcript ordering within each priority class.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: with an active non-heartbeat run and follow-up enabled, later user work is appended to the same follow-up queue and drained through the same session-lane path. I did not run the long live multi-channel timing scenario in this read-only pass.

## Solution Assessment

Unclear: the proposed caps, priority split, and maintenance movement target the right surfaces, but maintainers still need to choose the permanent foreground/background session scheduler policy before implementation.
