## Summary

Keep open. Current main and the latest release still have the central scheduler fairness gap: active non-heartbeat work can be appended to the same FIFO followup queue, the drain runs until the queue is empty, and session-lane priority only affects later command-queue ordering rather than giving fresh user/control work a preemption point.

## What This Changes

The issue reports session-lane starvation where followup/background drains monopolize a per-session lane and delay inbound messages, outbound sends, and stop/interrupt handling for minutes.

## Best Possible Solution

Define a scheduler-level fairness contract that lets fresh user/control work preempt or defer accumulated background/followup backlog while preserving per-session serialization within each priority class.

## Reproduction Assessment

Yes, at source level: active non-heartbeat work still enters the same FIFO followup queue and the drain loops until empty. I did not run the full 20-30 minute multi-channel reproduction in this read-only review.

## Solution Assessment

Unclear on the exact implementation. Session-lane concurrency or a short-message fast lane alone does not establish the needed fairness contract; the maintainable fix should make user/control work structurally win over background followups.
