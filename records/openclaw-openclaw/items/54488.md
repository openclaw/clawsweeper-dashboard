## Summary

Keep open. Current main still has the central fairness gap: active non-heartbeat inbound/control work can be appended to the same FIFO followup queue, the drain runs until exhausted, and each followup turn still enters the same serialized session lane. Recent comments add current confirmations and stop-safety impact rather than making the issue stale or superseded.

## What This Changes

The issue requests scheduler fairness so followup/background work cannot monopolize a single session lane and delay fresh inbound user/control messages for minutes.

## Best Possible Solution

Define and implement a scheduler-level contract where fresh user/control work can preempt or run after a bounded slice of background/followup work, with explicit ordering and cancellation rules for collect, heartbeat, system-event, and stop paths.

## Reproduction Assessment

Yes by source inspection: active non-heartbeat work still resolves to `enqueue-followup`, appends to the same FIFO followup queue, and drains through the same serialized session lane. I did not run the full multi-channel 20-30 minute live setup in this read-only pass.

## Solution Assessment

Unclear on the exact implementation. A priority-aware scheduler fix is the maintainable direction, while simply widening session-lane concurrency is not sufficient because it can race session state and still does not make user/stop work win over background work.
