## Summary

Keep open. Current main still has the scheduler fairness bug: fresh busy-session inbound work is appended to the same FIFO followup queue, the drain awaits followup turns until the queue is empty, and those turns re-enter the same per-session command lane; a May 1 user comment also reports the same latency class on outbound sends.

## What This Changes

The issue asks OpenClaw to prevent followup drain work from monopolizing a session lane and delaying fresh inbound or outbound dispatch for 20-30 minutes.

## Best Possible Solution

Add scheduler-level fairness in the auto-reply followup path so fresh user/control work can yield or run ahead of accumulated followup/system backlog while preserving FIFO ordering within each priority class.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: keep a session active, accumulate followup queue work, then send fresh non-heartbeat inbound work under queue/steer/followup mode; it appends to the same FIFO queue and waits for the drain to await earlier turns.

## Solution Assessment

Yes. A narrow scheduler-fairness fix in the followup queue is the maintainable path; config mitigations, lane timeouts, and post-turn cleanup reductions do not by themselves prevent backlog monopolization.
