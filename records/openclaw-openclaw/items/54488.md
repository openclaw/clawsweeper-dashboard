- `impact:message-loss`: Inbound and outbound channel messages can be silently queued for long periods before delivery or dispatch.

## Summary

Keep open. Current main has added useful mitigations, including command-lane priorities, lane task timeouts, and deferred context-engine maintenance, but the central fairness gap remains: accepted followups are still appended to one queue, the drain still runs until empty, and queued followup turns still use the same serialized session lane without a clear user/control preemption contract.

## What This Changes

User messages and control commands can wait behind an unbounded followup drain on the same serialized session lane, causing long inbound and outbound delays.

## Best Possible Solution

Define a scheduler-level fairness contract where fresh user/control work can preempt or run after a bounded slice of background/followup work while preserving per-session serialization, collect authorization grouping, route metadata, dedupe, and per-priority FIFO guarantees.

## Reproduction Assessment

Yes, at source level: active non-heartbeat work can still enqueue into the same followup queue, that queue still drains until empty, and queued followup turns still run through the serialized session lane. I did not perform the full 20-30 minute multi-channel live reproduction in this read-only pass.

## Solution Assessment

Unclear on the exact implementation. The maintainable direction is a priority-aware scheduler contract; the current partial mitigations reduce some contributors but do not make fresh user/control work win over accumulated followup backlog.
