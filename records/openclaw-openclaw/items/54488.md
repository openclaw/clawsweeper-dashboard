## Summary

Keep open: current main and the latest release still route active non-heartbeat same-session work into the FIFO follow-up queue and drain it until empty, so the reported foreground starvation remains real despite some adjacent maintenance and recovery fixes.

## What This Changes

The issue requests scheduler fairness so fresh inbound user/control messages are not delayed for many minutes behind accumulated follow-up or background work on the same session lane.

## Best Possible Solution

Define a core scheduler policy where fresh foreground user/control work can overtake, interrupt, or defer accumulated background/follow-up work while preserving ordering within each priority class, then land it with regression coverage across follow-up drain, active reply admission, and stuck-session recovery.

## Reproduction Assessment

Yes. Source inspection shows active non-heartbeat turns enqueue into the same FIFO follow-up queue and the drain loop continues until empty; recent comments add live field evidence, though I did not run a long multi-channel reproduction.

## Solution Assessment

No single narrow patch is proven best yet. A queue admission bypass or lane concurrency increase would be only partial; maintainers need to choose the foreground/background scheduler semantics before an implementation is safe.
