- `impact:session-state`: The remaining bug is about same-session run ordering and queued follow-up state monopolizing the session lane.

## Summary

Keep open: current main still routes active non-heartbeat follow-ups through the same FIFO follow-up queue and drain loop, so accumulated background or follow-up work can monopolize the session lane before fresh user/control work reaches the existing foreground priority layer.

## What This Changes

Issue #54488 requests a scheduler fairness fix so follow-up drains cannot starve fresh same-session user messages, control commands, or other foreground work.

## Best Possible Solution

Define and implement scheduler-level fairness so fresh foreground user/control turns can overtake, interrupt, or defer accumulated background/follow-up work while preserving ordering within priority classes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: during an active non-heartbeat run, later user/control work can be enqueued into the same FIFO follow-up queue and drained until empty; I did not run the long live multi-channel stall scenario.

## Solution Assessment

Unclear. The proposed caps and priority split target the right surfaces, but maintainers still need to choose the scheduler semantics for overtaking, interruption, collect batching, and transcript ordering.
