## Summary

Keep open. Current main still has the central scheduler fairness problem: busy inbound work is appended to the same followup FIFO queue, the drain runs until that queue is empty, and every drained followup turn still uses the same FIFO session lane. The May 1 field report against v2026.4.29 reinforces that this is still observable in the latest shipped release and now affects outbound CLI sends too.

## What This Changes

The issue requests scheduler fairness for session followup drains so queued followup and system work cannot monopolize a session lane and delay inbound or outbound dispatch for tens of minutes.

## Best Possible Solution

Add scheduler-level fairness to the auto-reply followup path: bound consecutive followup turns or drain elapsed time, allow fresh user/control inbound work to run ahead of accumulated system/followup backlog on the same session key, and preserve FIFO ordering within each class with focused regression coverage.

## Reproduction Assessment

Yes. The issue includes concrete setup and lane-wait logs, the May 1 comment extends the repro to outbound CLI send on v2026.4.29, and current main still shows the same unbounded FIFO source path; no live reproduction was run in this read-only review.

## Solution Assessment

Yes, a narrow scheduler fairness fix is the best direction. Moving more maintenance out of the lane may reduce latency, but it does not by itself prevent an accumulated followup drain from monopolizing the same session lane.
