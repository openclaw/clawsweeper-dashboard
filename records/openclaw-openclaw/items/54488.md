## Summary

Keep open. Current main still has the central scheduler fairness problem: busy inbound work is appended to the same followup FIFO queue, the drain loops until that queue is empty, and each followup turn still re-enters the per-session lane; a newer user comment also reports the same starvation class on the outbound send path.

## What This Changes

Fix session-lane starvation so followup draining cannot monopolize a session lane and delay fresh inbound or outbound dispatch for many minutes.

## Best Possible Solution

Add scheduler-level fairness in the auto-reply followup path so fresh user/control work can run ahead of accumulated followup/system backlog while preserving FIFO ordering within each class.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: accumulate followup queue work, keep the session run active, then send fresh inbound work and it is appended behind the same FIFO drain; I did not run a live 20-30 minute gateway reproduction.

## Solution Assessment

Yes. A narrow scheduler-fairness fix in the followup queue is the maintainable path; lane timeouts and cleanup-before-drain are useful mitigations but do not solve backlog monopolization.

## Repair Work Prompt

Fix the session-lane starvation reported in https://github.com/openclaw/openclaw/issues/54488. On current main, enqueueFollowupRun appends work to a FIFO queue, scheduleFollowupDrain drains until empty, active inbound followup/collect/steer work is enqueued behind older followups, and each followup turn re-enters the same per-session lane. Implement a narrow fairness repair in the auto-reply followup queue: bound consecutive followup turns or drain time, or otherwise yield/preempt so fresh user/control inbound work for the same session can run before accumulated system/followup backlog. Preserve FIFO ordering within each priority class, collect-mode authorization grouping, cross-channel routing, queue cap/drop summaries, and existing reset/stop behavior. Do not add a config option and do not implement the broader subagent lifecycle notification architecture from adjacent issues. Add focused regression coverage for fresh inbound/control work not being starved by an active followup drain; include a one-line CHANGELOG.md fix entry if the behavior change is user-visible.
