## Summary

Keep open. Current main still has the core scheduler fairness problem: followup work is appended FIFO, the drain loops until the queue is empty, and drained followup turns continue through the same per-session lane; later steering, reset-bypass, lane-timeout, and deferred-maintenance work mitigate adjacent stalls but do not bound or prioritize the followup drain itself.

## What This Changes

The issue asks OpenClaw to prevent followup-drain backlog from monopolizing a session lane and delaying inbound or outbound dispatch for tens of minutes.

## Best Possible Solution

Add scheduler-level fairness to the auto-reply followup path so fresh user/control work can run ahead of or after a bounded slice of accumulated system/followup backlog while preserving routing, authorization, and per-class FIFO guarantees.

## Reproduction Assessment

Yes, by source inspection: an active non-heartbeat run can enqueue followup work, append it FIFO, and drain it through the same per-session lane until the queue is empty. I did not run the full live 20-30 minute multi-channel gateway scenario in this read-only pass.

## Solution Assessment

Yes on direction, but the exact policy needs maintainer judgment: a bounded/priority-aware followup scheduler is the narrow maintainable fix, while existing reset bypasses, timeouts, default steering, and deferred maintenance only reduce adjacent contributors.
