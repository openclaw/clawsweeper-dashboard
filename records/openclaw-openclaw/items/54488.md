## Summary

Keep open. Current main still has the central scheduler fairness problem: user/control work can be converted into FIFO followup work while background work is active, and the followup drain continues through the same single-concurrency session lane without a bounded yield or user-priority check.

## What This Changes

The issue asks OpenClaw to prevent followup/background drain work from starving inbound user dispatch on the same session lane for long periods.

## Best Possible Solution

Add scheduler-level fairness so user/control messages can preempt or run after a bounded slice of background/followup work, while preserving per-class FIFO ordering and adding regression coverage for stop/user-message priority during active heartbeat or followup drains.

## Reproduction Assessment

Yes by source inspection: a fresh user run arriving during active background/followup work can be enqueued as followup and drained FIFO through the same per-session lane. I did not establish the full live 20-30 minute multi-channel reproduction in this read-only pass.

## Solution Assessment

Yes on direction, unclear on the exact policy: a bounded or priority-aware scheduler fix is the maintainable path, while simply widening session-lane concurrency can race transcript/session state and does not make user/stop work win over background work.
