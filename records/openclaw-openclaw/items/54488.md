## Summary

Keep open: current main still routes active non-heartbeat same-session work into the follow-up queue and drains that queue until empty, so the reported foreground starvation is not implemented away. Merged #95039 fixed a narrower dormant-drain race, and open/related items cover admission or recovery slices, but they do not replace this canonical scheduler-fairness issue.

## What This Changes

Issue requests preventing follow-up drain/session lane starvation from delaying fresh inbound user/control messages for long periods.

## Best Possible Solution

Define a core scheduler policy where fresh foreground user/control work can overtake, interrupt, or defer accumulated background/follow-up work while preserving ordering within each priority class and one active writer per session.

## Reproduction Assessment

Yes at source level: active non-heartbeat follow-up turns still enqueue, the queue appends FIFO, and the drain loop awaits queued follow-ups until the queue/dropped work is empty. I did not run the full 20-30 minute Discord/WhatsApp live scenario in this read-only review.

## Solution Assessment

Unclear as an automated fix: the owner boundary is core reply/session scheduling, but maintainers need to choose the policy for user priority, background abort/yield, and session-state safety before a new repair branch is safe.
