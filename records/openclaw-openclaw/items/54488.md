## Label Justifications

- `P1`: The issue affects real user/channel responsiveness and can delay stop/control messages behind background work.
- `impact:message-loss`: Inbound and outbound channel messages can sit queued for long periods before dispatch or delivery.
- `impact:session-state`: The affected behavior is tied to serialized session lanes, followup queues, transcript ordering, and safe session-state preservation.

## Summary

Keep open. Current `main` still has the source-level fairness gap: non-heartbeat work during an active run can be accepted as followup work, appended to one per-session followup queue, drained until empty, and then executed through the serialized session lane without a clear fresh user/control preemption contract.

## What This Changes

The issue reports followup-drain starvation where background/followup work monopolizes a serialized session lane and delays inbound dispatch, outbound sends, and control messages.

## Best Possible Solution

Define a scheduler-level fairness contract where fresh user/control work can preempt or run after a bounded slice of background/followup work while preserving per-session serialization, collect authorization grouping, route metadata, dedupe, and FIFO guarantees within each priority class.

## Reproduction Assessment

Yes, at source level: active non-heartbeat work can still enqueue into the same followup queue, that queue drains until empty, and queued turns still run through the serialized session lane. I did not run the full long-duration multi-channel live reproduction in this read-only pass.

## Solution Assessment

Unclear on the exact implementation. The maintainable direction is a priority-aware scheduler contract; current mitigations reduce some contributors but do not make fresh user/control work consistently win over accumulated followup backlog.
