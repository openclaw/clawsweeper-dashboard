## Summary

Keep open. Current main and the latest release still show the central fairness gap: busy non-heartbeat work is appended to a FIFO followup queue, the drain runs until empty, and followup turns enter the same serialized session lane; related open items are scoped tracks, not merged replacements.

## What This Changes

The issue asks for scheduler fairness so followup/background work cannot monopolize a serialized session lane and delay inbound or control messages.

## Best Possible Solution

Define a scheduler-level fairness contract where fresh user/control work can preempt or run after a bounded slice of background/followup work, while preserving collect-mode authorization grouping, routing, dedupe, and per-priority ordering with focused regression tests.

## Reproduction Assessment

Yes, at source level: current main routes active non-heartbeat followup/collect work into the same FIFO followup queue and serialized session lane. I did not run the full multi-channel live setup in this read-only pass.

## Solution Assessment

Unclear on the exact implementation. A priority-aware scheduler contract is the maintainable direction; simply widening session concurrency, as in https://github.com/openclaw/openclaw/pull/63864, does not make user/stop work win over background work and can race session state.
