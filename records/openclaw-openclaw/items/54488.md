## Summary

Keep open. Current `main` still has the central scheduler fairness problem: fresh user/control work can be converted into FIFO followup work while a session is active, and the drain still runs through the same single-concurrency session lane without a bounded yield or priority check.

## What This Changes

The issue asks OpenClaw to prevent followup queue draining from monopolizing a session lane and delaying inbound user/control dispatch for long periods.

## Best Possible Solution

Add scheduler-level fairness so user/control work can preempt or run after a bounded slice of background/followup work, while preserving ordering within each priority class and covering stop/user-message priority during active heartbeat or followup drains.

## Reproduction Assessment

Yes by source inspection: a fresh user/control run arriving during active background or followup work can be appended to the same FIFO followup queue and later drained through the same session lane. I did not run the full live 20-30 minute Discord/WhatsApp reproduction in this read-only pass.

## Solution Assessment

Yes on direction, unclear on exact policy: a bounded or priority-aware scheduler fix is the maintainable path. Widening session-lane concurrency alone is not the best fix because it can race session state and still does not make user/stop work win over background work.
