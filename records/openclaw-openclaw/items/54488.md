## Label Justifications

- `P1`: The issue describes a live regression where inbound and stop/control messages can wait behind background work long enough to break user workflows and side-effect safety.
- `impact:message-loss`: Fresh inbound, outbound send, and stop/control messages are reported and source-supported as being delayed or effectively suppressed behind followup backlog.
- `impact:session-state`: The affected path serializes per-session runs and followup work, so the fix must protect transcript/session ordering and active-run state.

## Summary

Keep open. Current main and the latest release still show the central scheduler fairness gap: active non-heartbeat work can be appended to the same FIFO followup queue, the drain runs until empty, and followup turns still enter the serialized session lane without a user/control preemption contract.

## What This Changes

The issue asks OpenClaw to prevent followup/background drains from monopolizing a session lane and delaying fresh inbound or stop/control messages for minutes.

## Best Possible Solution

Define an explicit scheduler fairness contract where user/control inbound work can preempt or run after a bounded slice of background/followup work, while preserving collect-mode authorization grouping, per-priority ordering, routing, dedupe, and focused regression coverage.

## Reproduction Assessment

Yes at source level: current main still routes active non-heartbeat followup/collect/steer work into a FIFO followup queue that drains through the same serialized session lane. I did not run the full Discord/WhatsApp/Telegram setup needed to reproduce the exact 20-30 minute waits.

## Solution Assessment

Unclear on the exact implementation. A priority-aware fairness contract is the maintainable direction; simply increasing session-lane concurrency, as proposed in https://github.com/openclaw/openclaw/pull/63864, does not guarantee that user or stop/control work wins over background work and may race session state.
