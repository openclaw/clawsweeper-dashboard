## Summary

Keep open. Current main still has the core fairness problem: accepted followup work is appended FIFO, the drain loops until the queue is empty, and each drained followup still runs through the same per-session lane; recent steering, timeout, reset, and deferred-maintenance work mitigates adjacent causes but does not supersede this report.

## What This Changes

The issue reports session-lane starvation where followup drain monopolizes a session lane and delays inbound or outbound dispatch for 20-30 minutes.

## Best Possible Solution

Add core scheduler fairness to the auto-reply followup path so fresh user/control work can run ahead of accumulated system/followup backlog while preserving existing routing, authorization, and per-class ordering guarantees.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: enqueue followup backlog for an active session, then send fresh inbound work and it is appended behind the same FIFO drain that re-enters the per-session lane; I did not run the full live 20-30 minute gateway scenario.

## Solution Assessment

Yes. A narrow scheduler-fairness repair is the best direction; lane task timeouts, reset bypasses, default steering, and background maintenance reduce adjacent stalls but do not bound or prioritize the followup drain itself.
