## Summary

Keep open. Current main still drains followup work FIFO until the queue is empty through the same single-concurrency session lane, so accumulated followup turns can still starve fresh inbound work; the recent lane-timeout and deferred-maintenance changes reduce adjacent stalls but do not add fairness to this drain.

## What This Changes

The issue asks OpenClaw to prevent followup queue drain from monopolizing a session lane and delaying inbound or CLI message dispatch for tens of minutes.

## Best Possible Solution

Add scheduler-level fairness to the auto-reply followup path so fresh user/control work can run ahead of accumulated system/followup backlog while preserving FIFO order within each priority class.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: accumulate followup queue work, keep a session run active, then enqueue fresh inbound work and it appends behind the same FIFO drain that runs through the single-concurrency session lane; I did not run the live 20-30 minute gateway scenario.

## Solution Assessment

Yes. A narrow scheduler-fairness fix is the maintainable direction; current main's cleanup-before-drain, lane timeout, and background maintenance mitigations do not bound or prioritize the followup drain itself.
