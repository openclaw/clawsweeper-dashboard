## Summary

Keep open. Current main still has the followup-drain starvation shape: active inbound work can be appended to the same FIFO followup queue, the drain loops until the queue is empty, and each followup turn still enters the same per-session lane; recent comments and the active priority-queue PR are useful context but are not a current-main fix for this issue.

## What This Changes

The issue reports that followup/background drain work can monopolize a session lane and delay fresh inbound user or control messages for minutes.

## Best Possible Solution

Add scheduler-level fairness so fresh user/control work can run ahead of or after a bounded slice of accumulated followup/background backlog, coordinated with https://github.com/openclaw/openclaw/pull/80199 and covered by focused queue and session-lane regression tests.

## Reproduction Assessment

Yes by source inspection: active non-heartbeat work can enqueue followup items, accepted items append FIFO, and scheduleFollowupDrain awaits each item until the queue is empty through the same per-session lane. I did not run the full live multi-channel 20-30 minute gateway scenario.

## Solution Assessment

Yes on direction, but not as a simple concurrency bump. The maintainable fix is a bounded or priority-aware scheduler policy, ideally integrated with the active priority-queue work, rather than allowing parallel turns on the same session transcript.
