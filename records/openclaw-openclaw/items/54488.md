## Label Justifications

- `P1`: The issue reports a broken real-user agent/channel workflow where fresh messages and stop-like controls can wait behind background follow-up work for minutes.
- `impact:message-loss`: Inbound channel and CLI messages are not lost from storage, but they are delayed long enough to be effectively suppressed from the user workflow.
- `impact:session-state`: The affected code path serializes per-session agent runs, follow-up queues, transcript maintenance, and interrupt semantics on the same session state.

## Summary

Keep open. Current main and v2026.6.6 still let active non-heartbeat turns enter the same follow-up queue and drain that queue until empty; related open PRs address narrower admission or idle-drain races, but not the broader foreground-vs-background scheduler fairness reported here.

## What This Changes

The issue reports that follow-up and background turns can monopolize a per-session lane and delay fresh inbound Discord, WhatsApp, and CLI messages for tens of minutes.

## Best Possible Solution

Define a scheduler-level fairness contract where fresh user/control turns can run ahead of or interrupt background follow-up work without racing transcript/session state, then land only the partial PRs that fit that contract.

## Reproduction Assessment

Yes, at source level: active non-heartbeat turns still resolve to enqueue-followup, enqueue FIFO into a per-session follow-up queue, and drain until the queue is empty. I did not run the full live 20-30 minute channel reproduction in this read-only review.

## Solution Assessment

Unclear. The reporter’s cap/yield/priority suggestions are plausible, but the best durable fix needs a maintainer decision on foreground-vs-background session scheduling, preemption, ordering, and interrupt semantics.
