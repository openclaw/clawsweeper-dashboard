## Summary

Keep open. Current main still has the misleading degraded memory warning path, and open PR #75637 already tracks a narrow fix with closing syntax but is not merged.

## What This Changes

The issue asks memory indexing diagnostics to distinguish actual sqlite-vec load failures from no-embedding-provider or unresolved-dimensions vector degradation.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec wording for actual extension load failures and uses embedding-provider or unresolved-dimensions wording when vector rows are skipped without a sqlite-vec load error.

## Reproduction Assessment

Yes. The source-level path is clear: provider-less indexing writes FTS-only chunks with `vectorReady: false`, and the helper currently emits sqlite-vec-specific wording even when `loadError` is absent.

## Solution Assessment

Yes. Splitting the warning message on existing vector load state is the narrowest maintainable fix; indexing and sqlite-vec loading semantics do not need to change.
