## Summary

Keep open. Current main still has the misleading memory-core warning path: FTS-only indexing without an embedding provider writes chunks with `vectorReady: false`, then the degraded-write helper emits sqlite-vec-specific wording even when no sqlite-vec load error is present. Open PR #75637 already uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for memory indexing warnings to distinguish actual sqlite-vec load failures from missing embedding-provider or unresolved-dimensions cases.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec wording for actual extension load failures and emits embedding-provider or unresolved-dimensions wording when chunks are indexed without vector rows and no sqlite-vec load error is present.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with no embedding provider, `indexFile` enters FTS-only mode, calls `writeChunks` with `vectorReady: false`, and the helper emits the sqlite-vec warning even when `loadError` is absent.

## Solution Assessment

Yes. The narrow maintainable fix is to split warning selection on existing vector load state rather than changing sqlite-vec loading or FTS-only indexing semantics.
