## Summary

Keep open. Current `main` still emits the sqlite-vec-specific degraded write warning from the no-embedding-provider path, and open PR #75637 is already the active closing implementation candidate.

## What This Changes

The issue asks memory-core to report missing embeddings or unresolved vector dimensions instead of blaming sqlite-vec when vector writes are skipped after sqlite-vec can load.

## Best Possible Solution

Review and land or close PR #75637; the intended end state is a narrow diagnostic split that preserves sqlite-vec wording for actual load failures and reports missing provider or dimensions for FTS-only/vector-skipped writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with no embedding provider, `indexFile` writes FTS-only chunks with `vectorReady: false`, and the current helper emits sqlite-vec-specific wording even when no `loadError` is present.

## Solution Assessment

Yes. Branching the warning text on existing vector load state and embedding/provider readiness is the narrow maintainable fix; sqlite-vec loading and FTS-only indexing do not need behavioral changes for this report.
