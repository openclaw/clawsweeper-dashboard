## Summary

Keep open. Current main still emits a sqlite-vec-specific degraded-write warning for the no-embedding-provider path even when no sqlite-vec load error is present, and open PR #75637 already uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests clearer memory-indexing diagnostics that distinguish actual sqlite-vec load failures from missing embedding-provider or unresolved-dimensions states.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec wording for actual `vector.loadError` failures and emits embedding-provider or unresolved-dimensions wording when vector writes are skipped without a sqlite-vec load error.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `logMemoryVectorDegradedWrite` with vectors enabled, `vectorReady: false`, chunks present, warning not yet shown, and no `loadError`; current FTS-only indexing reaches that state through `writeChunks(..., [], false)`.

## Solution Assessment

Yes. The best fix is a narrow warning-selection change in memory-core with focused tests; no sqlite-vec loader, storage schema, or plugin API change is needed for the reported behavior.
