## Summary

Keep open. Current main still emits the sqlite-vec-specific degraded-write warning whenever vector writes are skipped, even when the no-provider FTS-only path is the reason and no sqlite-vec load error is present; an open closing PR (#75637) already targets this issue, so this issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests that memory indexing diagnostics distinguish real sqlite-vec load failures from missing embedding-provider or unresolved-dimension cases.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec wording for actual extension load failures and emits an embedding-provider or unresolved-dimensions message when vector writes are skipped without `vector.loadError`; use #75637 as the active implementation candidate.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: call `logMemoryVectorDegradedWrite` with vectors enabled, `vectorReady: false`, chunks present, `warningShown: false`, and no `loadError`; the no-provider `indexFile` path reaches that state via `writeChunks(..., [], false)`.

## Solution Assessment

Yes. The narrow maintainable solution is to split warning selection in the memory-core diagnostic helper, with matching focused tests; no sqlite-vec loader or plugin API change is needed for the reported behavior.
