## Summary

Keep open. Current main still has the misleading sqlite-vec-specific warning on the no-provider FTS-only indexing path, and open PR #75637 already uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks memory-core indexing diagnostics to distinguish actual sqlite-vec load failures from vector-write skips caused by missing embedding providers or unresolved dimensions.

## Best Possible Solution

Review and land PR #75637, or an equivalent narrow memory-core diagnostic change, so sqlite-vec wording is reserved for actual extension load failures and missing-provider or unresolved-dimension cases get their own warning.

## Reproduction Assessment

Yes. The source-level path is clear: with no embedding provider, `indexFile` enters FTS-only mode, calls `writeChunks` with `vectorReady: false`, and the helper emits the sqlite-vec warning even when `loadError` is absent.

## Solution Assessment

Yes. Splitting the warning message based on existing vector load state is the narrow maintainable fix; changing sqlite-vec loading or disabling FTS-only indexing would solve the wrong layer.
