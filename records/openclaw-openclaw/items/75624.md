## Summary

Keep open. Current main still emits the sqlite-vec-specific degraded write warning from the generic memory vector warning helper, and an open unmerged PR already uses closing syntax for this exact diagnostic split, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks memory indexing to distinguish actual sqlite-vec load failures from missing embedding provider or unresolved vector dimensions when warning that `chunks_vec` was not updated.

## Best Possible Solution

Review and land, revise, or close PR #75637 so the memory-core warning remains sqlite-vec-specific only for real extension load failures and uses embedding-provider or unresolved-dimensions wording for FTS-only indexing.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with no embedding provider, `indexFile` enters the FTS-only branch, calls `writeChunks` with `vectorReady: false`, and the helper emits the sqlite-vec warning even when no load error is present.

## Solution Assessment

Yes. The narrow maintainable fix is to split warning message selection on the existing load-error/vector state rather than changing sqlite-vec loading or FTS-only indexing behavior.
