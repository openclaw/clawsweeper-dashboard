## Summary

Keep open. Current main still has the reported diagnostic bug: the memory-core degraded-write helper names sqlite-vec whenever vector writes are skipped, even when the caller is in FTS-only/no-embedding-provider mode and there is no sqlite-vec load error.

## What This Changes

The issue requests clearer memory indexing diagnostics when vector writes are skipped because embeddings or dimensions are unavailable rather than because sqlite-vec failed to load.

## Best Possible Solution

Split the memory-core degraded-write diagnostic by cause: keep the sqlite-vec message for real extension load failures, and use an embedding/provider/dimensions message when chunks are written in FTS-only or otherwise non-vector-ready mode without a sqlite-vec load error.

## Reproduction Assessment

Yes. Current main can be reproduced at source level by calling `logMemoryVectorDegradedWrite` with vectors enabled, `vectorReady: false`, chunks, no prior warning, and no `loadError`; the no-provider `indexFile` path reaches that state via `writeChunks(..., [], false)`.

## Solution Assessment

Yes. The narrow maintainable fix is in memory-core diagnostic selection and tests; no core/plugin API change or sqlite-vec loader change is needed for this report.
