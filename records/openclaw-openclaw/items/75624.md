## Summary

Keep open. Current main still emits the sqlite-vec-specific degraded-write warning whenever vector writes are skipped with vectors enabled and chunks present, even when `loadError` is absent; the provided timeline also shows open, unmerged PR #75637 is already the active closing implementation candidate.

## What This Changes

The issue requests clearer memory indexing diagnostics so skipped vector writes blame sqlite-vec only when the extension load failed and otherwise report missing embeddings or unresolved vector dimensions.

## Best Possible Solution

Keep the issue open until the paired fix lands, then ship a narrow memory-core diagnostic change that selects sqlite-vec wording only for actual extension load failures and uses embedding-provider or unresolved-dimensions wording for no-provider vector skips, with focused tests for both branches.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `logMemoryVectorDegradedWrite` with vectors enabled, `vectorReady: false`, chunks present, `warningShown: false`, and no `loadError`; the current no-provider `indexFile` path reaches that state through `writeChunks(..., [], false)`.

## Solution Assessment

Yes. The best fix is the narrow diagnostic split already targeted by #75637, plus focused tests; this does not need sqlite-vec loader changes or a plugin API change.
