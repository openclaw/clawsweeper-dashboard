## Summary

Keep open. Current main and the latest release still over-attribute the no-embedding-provider FTS-only indexing path to sqlite-vec, and the directly related fix PR at https://github.com/openclaw/openclaw/pull/75637 is closed without merge.

## What This Changes

The issue asks memory-core degraded vector warnings to distinguish actual sqlite-vec load failures from missing embedding-provider or unresolved-dimensions cases.

## Best Possible Solution

Update memory-core’s degraded-write and CLI index warnings to use sqlite-vec wording only when a vector load error exists, and otherwise report missing embedding readiness without changing FTS-only indexing or sqlite-vec loading.

## Reproduction Assessment

Yes, by source inspection. With no embedding provider, current main writes FTS-only chunks using `writeChunks(..., [], false)` and the warning helper still emits sqlite-vec-specific wording when `loadError` is absent.

## Solution Assessment

Yes. A narrow diagnostic split on the existing `loadError` and embedding-readiness state is the maintainable fix; sqlite-vec loading, provider selection, and FTS-only indexing should stay unchanged.
