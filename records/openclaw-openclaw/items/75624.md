## Summary

Keep open. Current main still has the misleading memory-core warning path, and open PR #75637 already uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests clearer memory indexing diagnostics so sqlite-vec is blamed only when sqlite-vec loading actually failed, not when vector writes are skipped because embeddings or dimensions were unavailable.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec-specific wording for actual extension load failures and emits embedding-provider or unresolved-dimensions wording when chunks are indexed without vector rows and no sqlite-vec load error is present.

## Reproduction Assessment

Yes. The source-level path is high confidence: without an embedding provider, `indexFile` enters FTS-only mode, calls `writeChunks(..., [], false)`, and the warning helper emits sqlite-vec wording even when no load error is present.

## Solution Assessment

Yes. The narrow maintainable solution is to split warning wording on existing vector load state and add focused tests, without changing sqlite-vec loading or FTS-only indexing semantics.
