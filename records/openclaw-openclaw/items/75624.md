## Summary

Keep open. Current main still over-attributes the no-embedding-provider FTS-only indexing path to sqlite-vec in the manager warning, and the prior closing PR is now closed unmerged.

## What This Changes

The issue requests clearer memory indexing diagnostics that distinguish sqlite-vec load failures from missing embedding providers or unresolved vector dimensions.

## Best Possible Solution

Update memory-core’s degraded-write warning to use sqlite-vec wording only when `loadError` is present, add a no-loadError regression test, and keep FTS-only indexing behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with no embedding provider, FTS-only indexing calls `writeChunks(..., [], false)` and the warning helper emits sqlite-vec-specific wording even when `loadError` is absent.

## Solution Assessment

Yes. A narrow diagnostic split on the existing `loadError` state is the maintainable fix; sqlite-vec loading, provider selection, and FTS-only indexing do not need behavior changes for this report.
