## Summary

Keep open. Current main still has the reported diagnostic bug, and open PR #75637 already references this issue as its fix path, so the conservative maintainer cleanup outcome is to leave this issue open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks memory-core to distinguish real sqlite-vec load failures from missing embedding provider or unresolved dimensions when logging degraded vector-write warnings.

## Best Possible Solution

Update the memory-core degraded-write diagnostic so actual `loadError` cases keep the sqlite-vec-specific wording, while no-load-error vector skips report missing embeddings or unresolved dimensions, with focused regression coverage for the helper and any CLI/status path that surfaces the same cause.

## Reproduction Assessment

Yes. Current main can be reproduced at source level by calling `logMemoryVectorDegradedWrite` with vectors enabled, `vectorReady: false`, chunks, no previous warning, and no `loadError`; the FTS-only `indexFile` path reaches that same state through `writeChunks(..., [], false)`.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostic selection inside memory-core plus focused tests; no sqlite-vec loader, dependency, or plugin API change is needed for this report. PR #75637 appears to target that path, but the issue should stay open until the PR lands or is replaced.
