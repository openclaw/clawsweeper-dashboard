## Summary

Keep open. Current main still has the misleading sqlite-vec-specific degraded-write warning on the no-provider vector-not-ready path, and open PR #75637 is already the active closing implementation candidate.

## What This Changes

The issue requests that memory indexing warnings distinguish actual sqlite-vec load failures from missing embedding provider or unresolved vector dimensions.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec wording for actual extension load failures and uses embedding-provider or unresolved-dimensions wording when vector writes are skipped without `vector.loadError`, with focused tests for both helper and CLI behavior.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: call `logMemoryVectorDegradedWrite` with vectors enabled, `vectorReady: false`, chunks present, `warningShown: false`, and no `loadError`; the no-provider FTS-only `indexFile` path reaches that state through `writeChunks(..., [], false)`.

## Solution Assessment

Yes. Splitting the warning by load-error/provider-readiness state is the narrowest maintainable fix; no sqlite-vec loader or plugin API change is required for the reported behavior.
