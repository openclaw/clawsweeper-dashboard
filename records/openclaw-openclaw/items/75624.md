## Summary

Keep open. Current main still has the reported diagnostic bug, and an open PR already references this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue asks OpenClaw to stop blaming sqlite-vec in memory indexing warnings when sqlite-vec loaded successfully but vector writes are unavailable because embeddings or vector dimensions were not resolved.

## Best Possible Solution

Ship a narrow memory-core fix that preserves sqlite-vec-specific wording for actual extension load failures, uses an embedding/provider/dimensions message when vector writes are skipped without a sqlite-vec load error, and covers both the manager helper and CLI warning path with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by calling the degraded-write helper with vectors enabled, chunks present, vectorReady false, warningShown false, and no loadError; the no-provider FTS-only index path reaches that state via writeChunks(..., [], false).

## Solution Assessment

Yes. The requested diagnostic split is the narrowest maintainable fix; no sqlite-vec loader change, dependency change, or plugin API change is needed for this report.
