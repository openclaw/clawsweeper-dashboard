## Summary

Keep open. Current main still emits the sqlite-vec-specific degraded write warning from the no-embedding-provider path, and open PR #75637 is already the active closing implementation candidate.

## What This Changes

The issue asks memory indexing diagnostics to distinguish sqlite-vec load failure from missing embedding provider or unresolved vector dimensions.

## Best Possible Solution

Review and land or close PR #75637; the end state should be a narrow diagnostic split that keeps sqlite-vec wording for actual load failures and reports missing provider or dimensions for FTS-only/vector-skipped writes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `runSync` can load sqlite-vec successfully, then the no-provider `indexFile` path writes FTS-only chunks with `vectorReady: false`, causing the helper to emit sqlite-vec wording even when `loadError` is absent.

## Solution Assessment

Yes. Branching the warning text on existing vector load state and embedding/provider readiness is the narrow maintainable fix; sqlite-vec loading and FTS-only indexing behavior do not need to change for this report.
