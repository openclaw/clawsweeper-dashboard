## Summary

Keep open. Current main still has the misleading diagnostic path: FTS-only indexing writes chunks with `vectorReady = false`, while the warning helper reports every non-ready vector write as `sqlite-vec unavailable` even when the sqlite-vec loader has succeeded.

## What This Changes

The issue requests separate memory index warnings so sqlite-vec is blamed only for extension load failures, while missing embedding providers or unresolved dimensions are reported as the actual degradation cause.

## Best Possible Solution

Update memory-core’s degraded-write warning contract to carry the actual degradation reason, add focused tests for both sqlite-vec load failure and FTS-only/no-provider indexing, and include a user-facing changelog fix entry credited to the reporter.

## Reproduction Assessment

Yes. The provided steps map to a high-confidence source path on current main: no embedding provider leads to FTS-only `writeChunks(..., [], false)`, which triggers the sqlite-vec warning helper even though sqlite-vec load state is tracked separately.

## Solution Assessment

Yes. The narrow maintainable fix is to branch the warning on actual `vector.loadError` versus provider/dimensions unavailability, rather than changing the sqlite-vec loader or dependency pins.
