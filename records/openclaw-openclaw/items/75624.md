## Summary

Keep open. Current main still emits the sqlite-vec-specific degraded-write warning whenever vector writes are skipped with chunks present and no prior warning, and the no-provider FTS-only indexing path reaches that helper with `vectorReady: false` even when no sqlite-vec load error is recorded. Open PR #75637 already uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks memory indexing diagnostics to distinguish actual sqlite-vec load failures from missing embedding provider or unresolved vector dimensions when vector writes are skipped.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that keeps sqlite-vec wording for actual extension load failures and emits embedding-provider or unresolved-dimensions wording when chunks are written without vector rows and no sqlite-vec load error is present.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with no embedding provider, `indexFile` enters FTS-only mode, calls `writeChunks` with `vectorReady: false`, and the helper emits the sqlite-vec warning even when `loadError` is absent.

## Solution Assessment

Yes. The maintainable fix is to split warning selection on the existing vector load state instead of changing sqlite-vec loading or the FTS-only indexing behavior.
