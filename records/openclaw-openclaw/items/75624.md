## Summary

Keep open. Current main still has the misleading sqlite-vec-specific warning for non-ready vector writes, and the no-embedding-provider FTS-only path can reach that helper with `vectorReady: false` even when sqlite-vec did not fail. Open PR #75637 is already the active closing implementation candidate, so this issue should stay open until that PR is merged or closed.

## What This Changes

The issue asks memory indexing to distinguish sqlite-vec load failures from missing embedding provider or unresolved vector dimensions when reporting degraded vector writes.

## Best Possible Solution

Review and land a narrow memory-core and CLI diagnostic fix that keeps sqlite-vec wording for actual extension load failures and uses embedding-provider or unresolved-dimensions wording when vector rows are skipped without a sqlite-vec load error.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with no embedding provider, `indexFile` enters FTS-only mode, calls `writeChunks` with `vectorReady: false`, and the helper emits sqlite-vec wording even when `loadError` is absent.

## Solution Assessment

Yes. The narrow maintainable solution is to split warning selection on the existing vector load state and provider/dimension state rather than changing sqlite-vec loading or disabling FTS-only indexing.
