## Summary

Keep open. Current main still has the misleading memory-core warning path, and open PR #75637 is already the active closing implementation candidate for this issue.

## What This Changes

Request to distinguish sqlite-vec load failures from missing embedding-provider or unresolved-dimension degraded vector writes in memory indexing warnings.

## Best Possible Solution

Land a narrow memory-core diagnostic fix that preserves sqlite-vec wording for real extension load failures and uses embedding-provider or unresolved-dimensions wording when vector rows are skipped without a sqlite-vec load error.

## Reproduction Assessment

Yes. The source-level path is high confidence: with no embedding provider, FTS-only indexing calls `writeChunks` with `vectorReady: false`, and the helper emits sqlite-vec-specific wording even when `loadError` is undefined.

## Solution Assessment

Yes. Splitting the warning on the existing vector load state is the narrowest maintainable fix; it avoids changing sqlite-vec loading or FTS-only indexing semantics.
