## Summary

Keep open: the JSON bounding work is useful and well targeted, but the PR still turns oversized non-JSON 2xx Mattermost responses into a successful 64 KiB prefix instead of a failure, so it needs a narrow fix or explicit maintainer acceptance before merge.

## What This Changes

The PR replaces Mattermost REST success JSON reads in client, upload, and probe paths with `readProviderJsonResponse`, caps non-JSON success text, and adds oversized-stream cancellation tests.

## Best Possible Solution

Use the shared bounded JSON reader plus a throwing bounded text-success reader, with regressions for JSON and text overflow and maintainer acceptance of the response-size caps.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main uses unbounded `res.json()`/`res.text()` in the Mattermost success paths, and the PR body includes terminal proof from a no-Content-Length streaming server; I did not run tests in this read-only review.

## Solution Assessment

No, not as written. The JSON, upload, and probe changes use the right shared helper, but the text fallback should fail on overflow rather than returning silent partial data.
