## Summary

Keep open: the PR fixes a real Mattermost unbounded success-response read path, but latest head adds a local bounded text reader that duplicates the exported `readProviderTextResponse` helper. That should be replaced before merge; maintainers still need to accept the intentional hard caps for unusual oversized Mattermost 2xx responses.

## What This Changes

The branch replaces Mattermost REST success JSON reads in request, upload, and probe paths with bounded readers, adds a throwing 64 KiB text-success cap, and adds oversized-stream cancellation tests.

## Best Possible Solution

Use the shared bounded JSON and text success readers in the Mattermost owner boundary, keep the overflow cancellation regressions, and have maintainers explicitly accept or tune the response-size budgets before merge.

## Reproduction Assessment

Yes, as a source-level reproduction: current main still calls unbounded `res.json()`/`res.text()` in the Mattermost success paths, and the PR body includes a no-Content-Length streaming-server proof showing bounded cancellation after the fix.

## Solution Assessment

No, not as written. The response cap placement is right, but the text fallback should use the existing `readProviderTextResponse` helper instead of creating a duplicate lower-level bounded text reader inside the plugin.
