## Summary

Keep open. The PR targets real WebChat ordering and reset problems, but current main does not already implement the full change and the proposed diff still has blocking review findings around bare reset semantics, reset cleanup, and the required changelog entry.

## What This Changes

The PR updates WebChat and Gateway chat history projection, reset/session-change reconciliation, stream/tool rendering, terminal error display, related UI/Gateway tests, a CLI test mock, and QA scenario timeouts to stabilize chat ordering.

## Best Possible Solution

Update this PR in place so bare WebChat `/new` and `/reset` keep the current fresh-session greeting contract or run an equivalent post-reset agent turn, centralize reset cleanup through `resetToolStream` before any history reload can race or fail, and add the active-version changelog entry before rerunning the focused WebChat/Gateway checks.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: compare current main’s `/new` WebChat path and gateway agent tests with the PR diff that calls `sessions.reset` directly, then reset after a tool/stream card while `chat.history` is delayed or fails to see stale tool stream state remain.

## Solution Assessment

No. The PR is aimed at the right surfaces, but direct `sessions.reset` is not the narrowest maintainable fix unless it preserves the existing fresh-session agent greeting, and the reset cleanup should reuse the existing tool-stream reset helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the fresh-session greeting on resets:** `ui/src/ui/app-chat.ts:556-557`
  - body: Current main sends bare WebChat `/new` and `/reset` through `chat.send`, and the gateway agent path resets the session then runs the fresh-session startup prompt. This branch sends bare resets straight to `sessions.reset`, whose handler only resets and emits `sessions.changed`, so WebChat stops producing the expected first assistant greeting and matches the open #75314 failure mode.
  - confidence: 0.88
- **[P2] Clear tool stream state during synchronous resets:** `ui/src/ui/app-chat.ts:649-655`
  - body: The new bare `/new` and `/reset` path clears visible chat state after `sessions.reset`, but it leaves `chatStreamSegments`, `chatToolMessages`, `toolStreamById`, `toolStreamOrder`, and the sync timer alive until history succeeds or a later gateway event arrives. Those artifacts render independently of `chatMessages`, so old tool/stream cards can remain visible after a reset when reload or event delivery is delayed or fails. Call `resetToolStream` in this path before reloading history.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `ui/src/ui/app-chat.ts:556-557`
  - body: This PR changes user-facing WebChat reset, history ordering, and error-display behavior, but the branch does not update `CHANGELOG.md`. OpenClaw policy requires user-facing `fix` changes to add an active changelog entry with contributor credit before merge.
  - confidence: 0.9
