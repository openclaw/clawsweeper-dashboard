## Summary

Keep open. Current main does not already contain this PR’s intended WebChat ordering and reset work, and the PR branch is an active draft with a concrete correctness blocker in the new synchronous reset path: it clears visible chat state but leaves tool-stream artifacts that render independently. The branch also lacks the required user-facing changelog entry.

## What This Changes

The PR updates WebChat and Gateway history projection, reset handling, stream/tool rendering, deferred final reconciliation, terminal error display, and related tests/timeouts to stabilize message ordering.

## Best Possible Solution

Keep this PR as the implementation candidate, update the synchronous reset path to reuse resetToolStream before reloading history/sessions, add focused coverage for reset-after-tool-output when history is delayed or fails, and add an Unreleased changelog entry for the user-facing WebChat fixes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a bare WebChat /new or /reset after tool/stream output while chat.history or sessions.changed delivery is delayed or fails: clearChatHistory clears chatMessages, but buildChatItems still renders chatStreamSegments and chatToolMessages.

## Solution Assessment

No, not as-is. The PR is scoped to the right WebChat/Gateway surfaces, but the synchronous reset path should use the existing resetToolStream cleanup already used by the PR’s Gateway reset-event path, and the branch needs the required changelog entry before landing.
