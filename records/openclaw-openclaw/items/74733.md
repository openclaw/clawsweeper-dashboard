## Summary

Keep open. This is a useful but still-draft WebChat/Gateway stabilization PR, and the PR head still has a concrete reset-state regression before merge: the new synchronous reset path clears visible messages but leaves tool-stream artifacts until a later history reload succeeds. It also lacks the required user-facing changelog entry.

## What This Changes

The PR updates WebChat and Gateway chat history projection, reset handling, stream/tool rendering, terminal error display, related tests, and QA timeouts to stabilize message ordering.

## Best Possible Solution

Update this PR in place: call resetToolStream in the synchronous clearChatHistory reset path before reloading history/sessions, add the required changelog entry, rebase against current main if needed, and rerun the focused WebChat/Gateway checks listed in the PR body.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a bare WebChat /new or /reset after tool/stream output while chat.history or sessions.changed delivery is delayed or fails; clearChatHistory clears chatMessages, but buildChatItems still renders chatStreamSegments and chatToolMessages.

## Solution Assessment

No, not as-is. The PR is aimed at the right WebChat/Gateway surfaces, but the synchronous reset path should reuse the existing resetToolStream cleanup before any async reload can fail or race.
