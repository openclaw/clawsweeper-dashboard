## Label Justifications

- `P1`: The reported behavior can replace a returning user's active WebChat context after reopen, which is a broken agent/session workflow affecting real use.
- `impact:data-loss`: The old conversation context is silently removed from the active session by reset/archive behavior even if the underlying transcript may still exist elsewhere.
- `impact:session-state`: The issue is centered on session lifecycle state drifting from a resumable WebChat session to a new/archived session without user intent.

## Summary

Keep open: current main still has a WebChat `sessions.create` path that can emit `command:new` and reset/archive the parent session, and I did not find an implemented-on-main fix or canonical replacement issue for the unsolicited reconnect case.

## What This Changes

The issue reports that WebChat reconnects can send an unsolicited `action:"new"` from `senderId:"unknown"`, causing completed sessions to be archived/reset and active chat context to disappear.

## Best Possible Solution

Prevent WebChat startup/reconnect/bootstrap from invoking the command-hook `sessions.create` reset path without explicit user action, while preserving the existing `/new` and New session button behavior.

## Reproduction Assessment

No live reproduction was run, but the reporter's command log plus current source give a medium-confidence source reproduction for how WebChat `action:"new"` resets/archives the active session. The missing piece is the exact reconnect caller that triggers the create path without a user click or slash command.

## Solution Assessment

Yes for the narrowed direction: the maintainable fix is to stop unsolicited startup/reconnect session creation while keeping explicit New Chat semantics intact. Adding a new config option should not be the first solution because the root policy is no hidden session reset without user action.
