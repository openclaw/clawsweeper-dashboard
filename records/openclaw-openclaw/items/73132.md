## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path can report success or degraded success after launchd state checks without verifying that the OpenClaw gateway listener exited or that the effective Gateway port is free.

## What This Changes

The issue asks for `openclaw gateway stop` on macOS to terminate or fail when a LaunchAgent stop leaves the underlying Gateway process and port alive.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective Gateway port, waits for verified OpenClaw listener PIDs to exit, escalates only for verified Gateway processes, and returns non-zero if the port remains busy or unattributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection reproduces the control-flow gap: the loaded LaunchAgent path can report stopped or degraded without checking the actual listener process or port.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extract existing verified Gateway-process and port-release helpers around the macOS managed-stop path, rather than killing by a broad `pgrep` pattern or redesigning launchd ownership.
