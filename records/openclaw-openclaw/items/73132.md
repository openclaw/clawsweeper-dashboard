## Summary

Keep open. Current `main` still has the reported macOS managed-stop gap: the LaunchAgent stop path verifies launchd state, but the loaded-service path does not verify that the OpenClaw gateway listener exited or that the configured port is free before reporting success.

## What This Changes

The issue requests that `openclaw gateway stop` on macOS terminate any wedged gateway process, verify the gateway port is free, and fail clearly if stop did not actually release the listener.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port, waits for safely attributed OpenClaw listener PIDs to exit after launchd stop or bootout, escalates only for verified gateway processes, and returns a non-zero failure if the port remains busy or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection reproduces the control-flow gap: loaded LaunchAgent stop can return success without listener or port verification.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing verified gateway-process and port-inspection helpers after managed LaunchAgent stop, rather than redesigning launchd handling or killing arbitrary `node` processes.
