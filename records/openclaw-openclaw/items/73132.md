## Summary

Keep open. Current main still has the macOS managed-stop gap: a loaded LaunchAgent stop can return success after launchd state checks without verifying that the OpenClaw gateway listener exited or that the configured port is free, and no source/changelog search shows this issue was fixed after the earlier ClawSweeper review.

## What This Changes

Ensure `openclaw gateway stop` on macOS actually terminates the managed gateway process/listener and fails if the gateway port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the configured service port, waits for the verified OpenClaw listener to exit after `launchctl stop` or bootout, escalates only against verified lingering gateway PIDs, and returns a non-zero failure if the port remains busy or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps, and current main can be source-verified to skip gateway listener and port checks on the loaded LaunchAgent stop path; a live macOS LaunchAgent smoke should be the final runtime proof.

## Solution Assessment

Yes. A narrow post-stop lifecycle check is the best maintainable fix; it should reuse existing verified gateway-process and port-inspection helpers rather than redesigning launchd handling or killing arbitrary `node` processes.

## Repair Work Prompt

Fix the macOS `openclaw gateway stop` managed LaunchAgent gap from https://github.com/openclaw/openclaw/issues/73132. Current main lets a loaded LaunchAgent stop return after launchd state checks without verifying the OpenClaw listener or freeing the configured port. Add a narrow macOS managed-stop postcondition: resolve the service gateway port from installed command metadata/config, after `launchctl stop` or bootout wait for verified OpenClaw listener PIDs on that port to exit, send SIGTERM and then SIGKILL/force cleanup only for verified OpenClaw gateway PIDs if needed, and fail non-zero with a clear message if the port remains busy or cannot be safely attributed. Reuse existing gateway-process/port-inspection helpers and the Windows Scheduled Task cleanup pattern where appropriate. Do not kill arbitrary `node` processes, do not change Windows/Linux lifecycle semantics, and do not redesign launchd install/restart behavior. Add focused tests for loaded LaunchAgent stop cleanup, degraded bootout cleanup, port-still-busy failure, and preserving the not-loaded unmanaged fallback. Add a user-facing CHANGELOG.md fix entry crediting `Thanks @Adam-Researchh`.
