## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent path checks launchd state, but the loaded-service stop flow does not verify that the OpenClaw listener exited or that the configured gateway port was released before reporting a stopped result.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate or fail on lingering OpenClaw gateway processes/listeners after LaunchAgent stop or bootout instead of returning success while the port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port from the installed service command/config, waits for safely attributed OpenClaw listener PIDs to exit after launchd stop or bootout, escalates only for verified gateway processes, and returns a non-zero failure if the port remains busy or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps and process evidence, and current source inspection shows the loaded LaunchAgent stop path can return without listener or port verification.

## Solution Assessment

Yes, with a safety constraint. Reusing the existing verified gateway-process and port-inspection helpers after LaunchAgent stop is narrower and safer than redesigning launchd handling or killing arbitrary `node` processes.
