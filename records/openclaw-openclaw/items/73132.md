## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the loaded LaunchAgent stop path verifies launchd state but does not verify that the OpenClaw gateway listener exited or that the configured port is free before reporting success.

## What This Changes

`openclaw gateway stop` should terminate wedged macOS LaunchAgent-backed gateway processes and fail if the gateway port remains bound after stop.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port, waits for verified OpenClaw listener PIDs to exit after launchd stop/bootout, escalates only for safely attributed gateway processes, and returns a failure if the port remains busy or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current main’s loaded LaunchAgent stop path can be verified from source to skip listener and port checks; a live macOS LaunchAgent smoke is the final runtime confirmation.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing verified gateway-process and port-inspection helpers after managed LaunchAgent stop rather than redesigning launchd handling or killing arbitrary `node` processes.
