## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path can return after launchd state checks without verifying that the Gateway listener exited or that the configured port is free, while verified-listener cleanup is only wired into the not-loaded/unmanaged stop path.

## What This Changes

The issue asks for `openclaw gateway stop` on macOS to terminate wedged OpenClaw Gateway processes left behind by launchd and fail clearly if the Gateway port remains occupied.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the configured Gateway port after `launchctl stop` or bootout, waits for the verified OpenClaw listener to exit, escalates only for verified lingering Gateway PIDs, and returns a non-zero failure if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps, and current main's loaded LaunchAgent stop path can be verified from source to skip Gateway listener and port checks; a live macOS LaunchAgent smoke should be the final proof.

## Solution Assessment

Yes. A narrow macOS lifecycle postcondition is the best fix; it should reuse existing verified Gateway-process and port-inspection helpers rather than redesigning launchd handling or killing arbitrary `node` processes.
