## Summary

Keep open. Current main still has the reported macOS managed-stop gap: a loaded LaunchAgent stop can report success after launchd state checks or bootout fallback without verifying that the OpenClaw gateway listener exited or that the configured port is free.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate lingering LaunchAgent-backed OpenClaw gateway processes and fail if the gateway port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port, waits for verified OpenClaw listener PIDs to exit, escalates only for verified gateway processes, and returns non-zero if the port remains busy or unattributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection reproduces the control-flow gap: the loaded LaunchAgent stop path can report success without checking the listener process or configured port.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing verified gateway-process and port-inspection helpers after managed LaunchAgent stop, rather than redesigning launchd handling or killing by a broad `pgrep` pattern.

## Repair Work Prompt

Fix the macOS managed `openclaw gateway stop` gap reported at https://github.com/openclaw/openclaw/issues/73132. Current main only verifies launchd state for loaded LaunchAgent stops and skips the verified gateway listener/port cleanup used for unmanaged stops and Windows Scheduled Tasks. Add a narrow post-stop path for managed Gateway stops on macOS: resolve the effective gateway port from service args/env/config, wait for verified OpenClaw listener PIDs to exit after `stopLaunchAgent`/bootout, escalate only for verified gateway processes, and return non-zero with a clear message if the port remains busy or cannot be safely attributed.
