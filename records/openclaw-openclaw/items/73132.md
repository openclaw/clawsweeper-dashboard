## Summary

Keep open. Current main still has the macOS managed-stop gap: `openclaw gateway stop` can finish from launchd state checks without verifying that the effective Gateway listener exited or that the port is free.

## What This Changes

Make macOS `openclaw gateway stop` terminate lingering OpenClaw Gateway processes, verify the port is released, and fail clearly if it is not.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective service port, waits for verified OpenClaw listener PIDs to exit, escalates only against verified Gateway processes, and returns non-zero with diagnostics if the port remains busy.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection reproduces the control-flow gap: loaded LaunchAgent stop can report success or degraded success without checking the actual listener process or port.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse or extract the existing verified Gateway-process and port-release helpers around the macOS managed-stop path, not a broad `pgrep`/`kill -9` fallback or a launchd redesign.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/73132. Current macOS `openclaw gateway stop` can return stopped/degraded after launchd checks while the underlying OpenClaw Gateway listener keeps the port bound. Add a macOS managed-stop postcondition that resolves the effective Gateway port from LaunchAgent command metadata/config, verifies and terminates only OpenClaw Gateway listener PIDs, waits for the port to become free, and fails with clear diagnostics if the port remains busy or unattributed. Prefer reusing/extracting existing verified process and port-release helpers from `src/infra/gateway-processes.ts`, `src/infra/restart-stale-pids.ts`, `src/infra/ports.ts`, and the Windows `schtasks` stop path.
