## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path can return success after launchd state checks or bootout fallback without verifying that the OpenClaw gateway listener exited or that the configured port is free.

## What This Changes

The issue asks for `openclaw gateway stop` on macOS LaunchAgent installs to terminate any lingering OpenClaw gateway process and fail if the gateway port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port from service args/env/config, waits for verified OpenClaw listener PIDs to exit after launchd stop or bootout, escalates only for verified gateway processes, and returns non-zero if the port remains busy or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection reproduces the control-flow gap: the loaded LaunchAgent stop path can report success without checking the listener process or configured port.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing verified gateway-process and port-inspection helpers after managed LaunchAgent stop, rather than redesigning launchd handling or killing by a broad `pgrep` pattern.
