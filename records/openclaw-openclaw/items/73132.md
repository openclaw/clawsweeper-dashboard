## Summary

Keep open. Current main still lets the loaded macOS LaunchAgent stop path finish without verifying that the actual OpenClaw Gateway listener exited or that the gateway port was freed.

## What This Changes

Request that `openclaw gateway stop` on macOS terminate surviving OpenClaw Gateway processes and fail if the gateway port remains occupied.

## Best Possible Solution

Add a macOS LaunchAgent managed-stop postcondition that resolves the effective service port, waits for verified OpenClaw listener PIDs to exit, escalates only those verified gateway processes, and fails if the port remains busy or unattributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current-main source inspection shows a loaded LaunchAgent stop can complete without checking verified gateway listener PIDs or port release.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extract the existing verified listener and port-release helpers around the macOS managed-stop path, not to add a broad `pgrep`/`pkill` workaround or redesign launchd ownership.

## Repair Work Prompt

Fix the macOS `openclaw gateway stop` path for https://github.com/openclaw/openclaw/issues/73132. Current main stops/disables the LaunchAgent but does not verify that the real OpenClaw Gateway listener exited or that the configured gateway port is free. Add a managed-stop postcondition for loaded macOS LaunchAgents: resolve the effective service port, wait briefly for verified OpenClaw Gateway listener PIDs to exit, send SIGTERM then SIGKILL only to verified OpenClaw Gateway listener processes if needed, and fail with a clear non-zero error if the port remains busy or cannot be attributed safely. Reuse or extract existing verified listener/port-release helpers where possible. Do not add broad `pgrep`/`pkill` process-name killing, do not change system-level LaunchDaemon support, and do not add a config option.
