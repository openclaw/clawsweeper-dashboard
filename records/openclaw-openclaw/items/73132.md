## Summary

Keep open. Current main still reports managed macOS LaunchAgent stop success after launchd commands without verifying that the actual OpenClaw gateway listener exited or that the configured port became free; the related macOS bootout/disable work did not cover this listener-cleanup postcondition.

## What This Changes

The issue requests that `openclaw gateway stop` fully terminate the managed macOS gateway process and fail if the configured port remains bound.

## Best Possible Solution

Add a macOS LaunchAgent stop postcondition that resolves the effective gateway port, waits for verified OpenClaw listeners to exit, escalates only those listeners when needed, and fails if the port remains busy or unattributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps and current source still returns from the managed LaunchAgent stop path without checking the real listener process or port; I did not run a live macOS launchd reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse or extract the existing verified gateway-process and port-release helpers around macOS managed stop, not to add a broad `pgrep` kill or redesign launchd ownership.
