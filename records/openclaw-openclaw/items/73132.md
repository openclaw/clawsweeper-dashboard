## Summary

Keep open. Current main still matches the reported macOS managed-stop gap: the LaunchAgent stop path can declare success or degraded success after launchd state checks without verifying the configured gateway port is free or terminating a verified lingering OpenClaw listener. The existing listener cleanup is only wired for the not-loaded/unmanaged path, while Windows already has the safer post-stop port release pattern this macOS path lacks.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate lingering OpenClaw gateway processes, verify the gateway port is free, and fail clearly when the port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition after `launchctl stop` and bootout fallback that resolves the installed gateway port, waits for any verified OpenClaw listener to exit, escalates only verified lingering gateway PIDs when needed, and returns a non-zero failure if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps, and current main has a source-level reproduction path: a loaded LaunchAgent stop reaches `service.stop(...)` without the listener/port cleanup that `onNotLoaded` uses; the existing lifecycle test also asserts that port resolution is skipped in that path.

## Solution Assessment

Yes. A narrow macOS lifecycle postcondition reusing existing verified gateway-process and port-inspection helpers is the best maintainable fix; redesigning launchd handling or killing unverified processes would be riskier.
