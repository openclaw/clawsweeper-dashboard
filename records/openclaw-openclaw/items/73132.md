## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path can report success after launchd state checks without verifying that the OpenClaw gateway listener exited or that the effective gateway port is free.

## What This Changes

Ensure `openclaw gateway stop` on macOS actually terminates the managed gateway listener and fails if the configured port remains busy.

## Best Possible Solution

Add a macOS LaunchAgent managed-stop postcondition that resolves the effective gateway port, waits for verified OpenClaw listener PIDs to exit, escalates only for verified gateway processes, and returns non-zero if the port remains busy or unattributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current-main source inspection reproduces the control-flow gap: loaded LaunchAgent stop can complete without checking the real listener process or port.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extract the existing verified gateway-process and port-release helpers around the macOS managed-stop path, not to kill by a broad process-name pattern or redesign launchd ownership.
