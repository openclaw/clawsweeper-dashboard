## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path can report stopped or degraded after launchd checks without verifying that the OpenClaw gateway listener exited or that the effective gateway port is free.

## What This Changes

The issue reports that macOS `openclaw gateway stop` can return success while the underlying OpenClaw gateway Node process continues listening.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port, waits for verified OpenClaw listener PIDs to exit, escalates only for verified gateway processes, and exits non-zero if the port remains busy or unattributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection reproduces the control-flow gap: the loaded LaunchAgent stop path can report success without checking the listener process or configured port.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse or extract the existing verified gateway-process and port-inspection patterns after managed LaunchAgent stop, rather than redesigning launchd handling or killing by a broad `pgrep` pattern.
