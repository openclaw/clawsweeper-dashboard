## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path confirms launchd state but does not verify that the OpenClaw gateway listener exited or that the configured port is free before reporting success.

## What This Changes

The issue asks for `openclaw gateway stop` on macOS to terminate any wedged underlying gateway process and fail if the gateway port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port, waits for safely attributed OpenClaw listener PIDs to exit after launchd stop/bootout, escalates only for verified gateway processes, and returns a failure if the port remains busy or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current main can be source-verified to skip listener and port checks after loaded LaunchAgent stop; a live macOS LaunchAgent smoke is the final runtime confirmation.

## Solution Assessment

Yes, with a safety boundary. The narrow maintainable fix is to reuse the existing verified gateway-process and port-inspection helpers after managed LaunchAgent stop, rather than killing arbitrary `node` processes or redesigning launchd handling.
