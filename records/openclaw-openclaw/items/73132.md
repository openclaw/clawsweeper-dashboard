## Summary

Keep open. Current main still lacks a macOS LaunchAgent stop postcondition that verifies the actual OpenClaw gateway listener exited or that the configured port is free before reporting stop success; the adjacent bootout-by-default work in https://github.com/openclaw/openclaw/pull/79235 changed stop semantics but did not close this listener-cleanup gap.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate any still-running managed gateway process and verify the listener port is free before returning success.

## Best Possible Solution

Add a macOS LaunchAgent stop postcondition that resolves the effective gateway port, waits for verified OpenClaw listener PIDs to exit, escalates only those verified listeners, and fails if the port remains busy or unattributed.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main still returns from the LaunchAgent stop path without checking the real listener process or port, and the issue gives concrete macOS steps to observe the wedged process. I did not run a live macOS launchd reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse or extract the existing verified gateway-process and port-release helpers around macOS managed stop, not to add a broad `pgrep` kill or redesign launchd ownership.
