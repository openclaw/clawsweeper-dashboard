## Summary

Keep open. Current main still has the macOS managed-stop gap: the LaunchAgent stop path can report success after launchd state checks or bootout fallback without verifying that the OpenClaw gateway listener exited or that the configured port is free.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate lingering LaunchAgent gateway node processes and fail if the gateway port remains bound.

## Best Possible Solution

Add a macOS managed-stop postcondition that resolves the effective gateway port from service args/env/config, waits for verified OpenClaw listener PIDs to exit after `launchctl stop` or `bootout`, escalates only when the listener is safely attributed, and returns non-zero if the port remains busy or attribution is unsafe.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source inspection shows the loaded LaunchAgent stop path can report success without checking the listener process or port release.

## Solution Assessment

Yes. The narrow maintainable direction is to reuse existing verified gateway-process and port-inspection helpers after managed LaunchAgent stop, rather than broad `pgrep` cleanup or a redesign of launchd handling.

## Repair Work Prompt

Fix the macOS `openclaw gateway stop` managed LaunchAgent gap from https://github.com/openclaw/openclaw/issues/73132. Current `stopLaunchAgent`/`runServiceStop` can report success after launchd state checks while an OpenClaw gateway listener still owns the configured port. Add a focused macOS managed-stop postcondition: resolve the effective gateway port from service command args/env/config, after `launchctl stop` or bootout verify any OpenClaw gateway listener PIDs on that port, wait for graceful exit/port release, escalate only for verified OpenClaw gateway processes, and fail non-zero with clear diagnostics if the port remains busy or cannot be safely attributed. Reuse existing gateway-process and port-inspection helpers where possible; do not add broad `pgrep` cleanup, kill unrelated listeners, redesign launchd restart semantics, or change Windows/systemd behavior except for shared helper extraction if needed. Add focused regression coverage in launchd/lifecycle tests and a user-facing `CHANGELOG.md` fix entry crediting `@Adam-Researchh`. Related refs: #73132 and the already-implemented Windows analogue #52049.
