## Summary

Keep open. Current main still has the reported macOS managed-stop gap: a loaded LaunchAgent stop can return after launchd state checks without verifying that the effective OpenClaw Gateway listener exited or that the gateway port is free.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate or escalate lingering Gateway listener processes and fail if the port remains bound.

## Best Possible Solution

Add a macOS LaunchAgent managed-stop postcondition that resolves the effective service port, waits for verified OpenClaw listener PIDs to exit, escalates only those verified gateway processes, and fails if the port remains busy or unattributed.

## Reproduction Assessment

Yes, source-level. The issue gives concrete macOS steps, and current-main source inspection reproduces the control-flow gap: loaded LaunchAgent stop completes without checking the real listener process or port.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse or extract the existing verified gateway-process and port-release helpers around the loaded macOS managed-stop path, without adding broad `pgrep`-style killing or changing stop/start semantics into restart.

## Repair Work Prompt

Fix the macOS `openclaw gateway stop` managed-stop gap tracked by https://github.com/openclaw/openclaw/issues/73132. Current main stops loaded LaunchAgents through `stopLaunchAgent` and reports success after launchd state checks without verifying that the OpenClaw Gateway listener exited or that the effective service port is free. Add a narrow macOS stop postcondition that resolves the effective gateway port from the installed service command/config, waits for verified OpenClaw listener PIDs to exit, escalates only verified OpenClaw gateway processes if needed, and returns a non-zero failure if the port remains busy or cannot be safely attributed. Reuse or extract existing helpers such as `findVerifiedGatewayListenerPidsOnPortSync`, `signalVerifiedGatewayPidSync`, `terminateStaleGatewayPids`, restart-health/port inspection, or the Windows Scheduled Task stop pattern. Do not add broad `pgrep`/process-name kills, do not change stop/start semantics into restart, do not broaden this into system LaunchDaemon support, and preserve the intentional macOS behavior where `gateway stop` disables the LaunchAgent. Add focused tests for loaded LaunchAgent stop verifying port resolution, listener cleanup, and failure when the port remains busy; update lifecycle tests that currently assert the skip. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter if appropriate.
