## Summary

Keep open. Current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking` support: the config type/schema reject those top-level fields, docs omit them, and skill invocation has no override carrier; the related implementation PRs are closed unmerged.

## What This Changes

Requests per-skill `model` and `thinking` defaults under `skills.entries.<skill>`.

## Best Possible Solution

Add validated optional `model` and `thinking` fields to skills config, thread resolved per-skill defaults into invocation/model selection with explicit precedence and allowlist semantics, then update schema/runtime tests and public docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection gives high confidence that current main rejects top-level `model` and `thinking` under `skills.entries.<skill>` and has no invocation plumbing for them.

## Solution Assessment

Unclear; first-class config fields are the right area, but the previous directive-injection approaches raised precedence and parser-scope concerns. Maintainers should settle the model-selection contract before implementation.
