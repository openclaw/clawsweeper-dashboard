## Summary

Keep open. Current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`; the direct config PRs are closed unmerged, and the open frontmatter PR is a different design that does not settle this config-and-thinking request.

## What This Changes

The issue requests per-skill `model` and `thinking` config fields under `skills.entries.<skill>` and applying them as per-turn defaults when that skill runs.

## Best Possible Solution

Define the public per-skill defaults contract first, then add typed schema fields, invocation plumbing, model/thinking validation, docs, and regression tests that reuse existing model-selection and thinking policy.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main shows the requested config fields are absent from type/schema/docs and are not carried through skill invocation.

## Solution Assessment

Unclear until maintainers choose the config-versus-frontmatter contract and precedence/security rules. The requested config shape is plausible, but this is a product/API decision rather than a narrow bug fix.
