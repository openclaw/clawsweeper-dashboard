## Summary

Keep open. Current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`: the type and strict schema omit both fields, docs only describe `enabled`/`env`/`apiKey` plus custom `config`, and skill command invocation only rewrites the prompt body without carrying model or thinking defaults into model selection.

## What This Changes

Request adding validated per-skill `thinking` and `model` config fields under `skills.entries.<skill>` and applying them as per-turn defaults when that skill is invoked.

## Best Possible Solution

Implement this in core skills config and invocation plumbing: validate optional fields, resolve them by skill key, apply them as non-persistent per-turn defaults, and document precedence.

## Reproduction Assessment

Yes, from source inspection. A top-level `model` or `thinking` under `skills.entries.<skill>` is rejected by the strict schema today, and the skill invocation path has no field that can carry those defaults into model/thinking selection.

## Solution Assessment

Yes. First-class core config and invocation plumbing is the narrowest maintainable solution because a custom `config` bag or inline SKILL.md directive cannot provide validated per-user defaults before model selection.
