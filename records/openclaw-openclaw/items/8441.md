The issue requests optional per-skill `model` and `thinking` overrides under `skills.entries.<skill>` so skills can choose cheaper, stronger, or different reasoning defaults than the session default.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract, then implement schema/types, invocation and provider-aware model selection, thinking validation, precedence, allowlist or consent behavior, fallback semantics, focused tests, real provider proof, and public docs together.

## Security Review

Status: needs_attention

Summary: Per-skill model/thinking routing is security-sensitive because it can steer prompts and credentials across provider/model boundaries.

Concerns:

- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:120`
  - body: The models docs make `agents.defaults.models` the allowlist for `/model` and session overrides, so skill-level model routing should honor that policy instead of creating a separate bypass.
  - confidence: 0.86
- **[medium] Define consent for skill-influenced routing:** `docs/tools/skills.md:181`
  - body: The skills docs treat third-party skills as untrusted code and note that per-skill env/API key config can inject secrets into the host process, so routing needs explicit operator-controlled semantics.
  - confidence: 0.84
- **[medium] Align with plugin override gates:** `docs/gateway/configuration-reference.md:230`
  - body: Current plugin LLM and subagent model overrides use explicit `allowModelOverride` and optional `allowedModels` gates; per-skill routing should not be less controlled.
  - confidence: 0.82
- **[high] Apply routing before provider resolution:** `src/agents/pi-embedded-runner/run/setup.ts:56`
  - body: Current hook routing runs before model resolution so provider/model overrides are auth-aware; skill routing should use an equivalent early boundary rather than swapping only a late model identifier.
  - confidence: 0.86
