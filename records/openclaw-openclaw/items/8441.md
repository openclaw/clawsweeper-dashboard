## Summary

Keep open: current main and the latest release still do not provide first-class `skills.entries.<skill>.model` or `skills.entries.<skill>.thinking`, and the open frontmatter-only model-routing work does not supersede this config-plus-thinking request. The feature is clear, but it changes provider/model routing and reasoning policy, so maintainers should choose the public contract before implementation.

## What This Changes

Add per-skill `thinking` and `model` overrides under `skills.entries.<skill>` so skill invocations can use configured reasoning and model defaults.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract, then implement schema/types, invocation/model-selection plumbing, precedence, allowlist and consent behavior, thinking validation, focused tests, and public docs together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config-contract request. Source, tests, docs, latest release code, and related GitHub state show the requested first-class fields and runtime consumer remain absent.

## Solution Assessment

Unclear until maintainers choose the contract. Adding raw fields alone is not the best solution without explicit precedence, allowed-model, provider/auth, fallback, consent, and thinking-level semantics.

## Security Review

Status: needs_attention

Summary: Per-skill model/thinking routing is security-sensitive because it can steer prompts and credentials across provider/model boundaries.

Concerns:

- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:120`
  - body: The docs make `agents.defaults.models` the allowlist for `/model` and session overrides, so skill-level model routing should honor the same policy rather than creating a separate path.
  - confidence: 0.86
- **[medium] Define consent for skill-influenced routing:** `docs/tools/skills.md:182`
  - body: The skills docs treat third-party skills as untrusted code and note that per-skill env/API key config can inject secrets into the host process, so model/provider routing needs explicit operator-controlled semantics.
  - confidence: 0.84
- **[medium] Align with existing plugin override gates:** `docs/gateway/configuration-reference.md:230`
  - body: Current plugin LLM and subagent model overrides use explicit `allowModelOverride` and optional `allowedModels` gates; per-skill routing should not create a less-controlled path.
  - confidence: 0.82
