## Summary

Keep open. Current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`: the type and strict schema omit both fields, docs only describe `enabled`/`env`/`apiKey` plus custom `config`, skill command specs cannot carry overrides, and the prior implementation PR is closed unmerged while the narrower related issue points back here.

## What This Changes

Add optional per-skill `model` and `thinking` config fields under `skills.entries.<skill>` so skill invocations can use default model/thinking overrides like cron agent-turn payloads.

## Best Possible Solution

Implement this as a core config and invocation feature: validate optional `model` and `thinking` on skill entries, resolve them by skill key when a skill command is invoked, apply them as per-turn defaults before model/thinking selection, and document precedence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a top-level `model` or `thinking` under `skills.entries.<skill>` is rejected by the strict schema today, and skill invocation lacks any field or timing path to influence model/thinking selection.

## Solution Assessment

Yes. First-class config and invocation plumbing is the narrowest maintainable solution because a custom `config` bag or inline SKILL.md directive cannot provide validated per-user defaults before model selection while preserving explicit override precedence.
