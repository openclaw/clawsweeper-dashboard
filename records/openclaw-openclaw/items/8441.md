## Summary

Keep open. Current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`, the direct config-based implementation attempts are closed unmerged, and the open frontmatter-only model-routing work does not cover this config-and-thinking request.

## What This Changes

The issue requests schema-backed per-skill `thinking` and `model` defaults under `skills.entries.<skill>` that apply when a skill is invoked.

## Best Possible Solution

Define the per-skill defaults contract first, then implement typed schema fields and invocation plumbing that reuse existing model and thinking selection policy, with clear precedence, observability, docs, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection on current main shows the requested fields are absent from the config type/schema/docs and are not carried through skill invocation.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the config-versus-frontmatter contract and precedence/security rules; the requested config shape is plausible, but this is a product/API decision rather than a narrow bug fix.
