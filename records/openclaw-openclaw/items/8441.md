## Summary

Keep open. Current main still does not implement or accept first-class per-skill `model` and `thinking` config, related config-field PRs are closed unmerged, and the remaining open related PR uses a different SKILL.md frontmatter routing design that needs maintainer product/security direction.

## What This Changes

Add first-class `thinking` and `model` keys under `skills.entries.<skill>` so skill invocations can override per-turn reasoning level and model selection.

## Best Possible Solution

Define the product contract, then implement typed and schema-backed per-skill defaults that flow through skill invocation into existing model and thinking selection with docs and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection on current main shows the requested config fields are absent, rejected by strict schema, and not carried through skill command invocation.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the config-vs-frontmatter contract and precedence/security semantics; the narrow implementation area is core skills config, invocation plumbing, model selection, thinking validation, tests, and docs.
