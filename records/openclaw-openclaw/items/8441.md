## Security Review

Status: needs_attention

Summary: Per-skill model/thinking routing is security-sensitive because it can steer prompts and credentials across provider/model boundaries.

Concerns:

- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:120`
  - body: The docs make `agents.defaults.models` the allowlist for `/model` and session overrides, so skill-level model routing should honor the same policy rather than creating a separate route around it.
  - confidence: 0.86
- **[medium] Define consent for skill-influenced routing:** `docs/tools/skills.md:181`
  - body: Skills are documented as untrusted code and per-skill env/API key config can inject secrets into the host process, so model/provider routing needs explicit operator-controlled semantics.
  - confidence: 0.84
- **[medium] Align with existing plugin override gates:** `docs/gateway/configuration-reference.md:230`
  - body: Current plugin LLM and subagent model overrides use explicit `allowModelOverride` and optional `allowedModels` gates; per-skill routing should not create a less-controlled path.
  - confidence: 0.82
