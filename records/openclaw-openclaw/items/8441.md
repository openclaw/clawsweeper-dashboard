## Summary

Keep open: current main and v2026.5.22 still do not support first-class `skills.entries.<skill>.model` or `skills.entries.<skill>.thinking`, and the active related frontmatter PR does not settle this config contract.

## What This Changes

The issue requests validated per-skill `model` and `thinking` fields under `skills.entries.<skill>` that apply as per-turn defaults when a skill runs.

## Best Possible Solution

Define one operator-controlled per-skill routing contract, then implement schema/types, invocation plumbing, provider-aware model selection, thinking validation, precedence, tests, docs, and security semantics together.

## Reproduction Assessment

Yes for the missing capability from source inspection: the current strict schema and config type do not accept top-level `model` or `thinking` under a skill entry. This is a feature/config-contract gap rather than a broken existing behavior reproduction.

## Solution Assessment

Unclear until maintainers choose the routing and security contract. Adding raw fields alone is not enough without explicit precedence, allowed-model, provider/auth, fallback, consent, turn-boundary, and thinking-level semantics.
