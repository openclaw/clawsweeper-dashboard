## Summary

Keep open. Current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking` support: the config type and strict schema omit both fields, tests still preserve unknown-key rejection, docs only describe `enabled`/`env`/`apiKey` plus custom `config`, and skill command handling rewrites the prompt after model selection rather than carrying per-skill defaults into that selection path.

## What This Changes

Add per-skill `model` and `thinking` config fields under `skills.entries.<skill>` so skill invocations can use configured model and reasoning defaults.

## Best Possible Solution

Implement this as a core config and invocation feature: validate optional `model` and `thinking` on skill entries, resolve those values by skill key when a skill command is invoked, feed them into model/thinking selection before the agent run starts, and document the precedence and validation behavior.

## Reproduction Assessment

Yes. The config-contract part is reproducible by parsing a `skills.entries.<skill>` object with top-level `model` or `thinking`, which current strict schema rejects; the runtime gap is also visible because skill command handling currently rewrites only the prompt body after model selection has already run.

## Solution Assessment

Yes. First-class core plumbing is the right direction because a custom `config` bag or inline SKILL.md directive cannot reliably apply validated, user-specific model and thinking defaults before model selection while preserving explicit user/session precedence.
