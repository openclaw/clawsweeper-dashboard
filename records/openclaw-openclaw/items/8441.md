## Summary

Keep open: current main and the latest shipped release still lack first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`, and the open related frontmatter issue only covers part of this request. The remaining work needs a maintainer product/security decision before another implementation is queued.

## What This Changes

Request to add first-class `model` and `thinking` per-skill config under `skills.entries.<skill>` and apply those defaults during skill invocation.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract across config and SKILL.md, then implement schema, invocation plumbing, precedence, provider/auth allowlist policy, focused tests, and public docs together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config contract request. Source and docs inspection on current main and `v2026.5.12` show the requested first-class skill-entry fields and runtime consumer are absent.

## Solution Assessment

Unclear until maintainers choose the public contract. Adding raw fields alone is not enough; the implementation needs explicit precedence, allowed-model, provider/auth, fallback, and thinking-level semantics.

## Security Review

Status: needs_attention

Summary: Per-skill model routing is security-sensitive because skill configuration could influence provider choice, prompt destination, credential path, and cost.

Concerns:

- **[medium] Keep model allowlists authoritative:** `docs/concepts/models.md:119`

  - confidence: 0.86

  - body: The skills docs warn that third-party skills are untrusted code, so config or metadata that changes model/provider selection needs explicit operator-controlled policy and observability.
  - confidence: 0.82
