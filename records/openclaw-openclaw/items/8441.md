## Summary

Keep open: current main and the latest release still lack first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`, and the open SKILL.md/frontmatter PR does not fully supersede this config-plus-thinking request. The request has clear user value, but it is a public config/provider-routing contract that needs maintainer product and security decisions before implementation.

## What This Changes

Add per-skill `model` and `thinking` configuration fields under `skills.entries.<skill>` so skill invocations can use skill-specific model and thinking defaults.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract, then implement schema/types, invocation and model-selection plumbing, precedence, allowlist/consent policy, thinking-level validation, focused tests, and public docs together.

Not applicable as a bug reproduction; this is a feature/config-contract request. Source, docs, tests, latest release code, and related PR state show the requested first-class fields and runtime consumer remain absent.

## Solution Assessment

Unclear until maintainers choose the contract. Adding raw fields alone is not the best solution without explicit precedence, allowed-model, provider/auth, fallback, consent, and thinking-level semantics.
