## Summary

Keep open: current main still lacks first-class `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking`, and the open SKILL.md/frontmatter work does not fully cover this config-plus-thinking request. The feature has clear user value, but it changes public config, provider routing, and trust boundaries, so it needs maintainer product/security review before implementation.

## What This Changes

Add first-class `model` and `thinking` fields under `skills.entries.<skill>` and apply them when a skill is invoked.

## Best Possible Solution

Define one maintainer-approved per-skill routing contract, then implement schema/types, invocation and provider-aware model selection, precedence, allowlist/consent policy, thinking validation, tests, and public docs together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config-contract request. Source, docs, tests, latest-release code, and related PR state show the requested first-class fields and runtime consumer remain absent.

## Solution Assessment

Unclear until maintainers choose the contract. Adding raw fields alone is not the best solution without explicit precedence, allowed-model, provider/auth, fallback, consent, and thinking-level semantics.
