## Summary

Keep open; the tool-policy cache is scoped to the config object, but the provider hook cache can persist a transient in-flight empty result and make later hook discovery skip provider plugins.

## What This Changes

The PR adds config-scoped caches for effective tool policy resolution and provider hook plugin discovery, plus small spacing/comment changes in `src/agents/openclaw-tools.ts`.

## Best Possible Solution

Keep the performance optimization only if provider hook caching excludes in-flight guard results or caches only stable post-load results, with focused regression coverage for config-scoped reentry.

## Reproduction Assessment

Yes. By source inspection, a config-scoped `resolveProviderPluginsForHooks` call made during provider loading would cache the temporary `[]`, then a later same-key call after loading completes would still get no hook plugins.

## Solution Assessment

No. The provider hook cache wraps the reentry guard result; the guard should stay outside the cached load or the cache should skip storing guard-only empty results.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not cache in-flight empty provider hooks:** `src/plugins/provider-hook-runtime.ts:132-137`
  - body: `resolveProviderPluginsForHooksUncached` returns `[]` while provider loading is already in flight as a temporary reentry guard. This wrapper stores that empty array in the config-scoped cache, so later same-key calls can keep seeing no provider hook plugins after loading completes, skipping normalization or fallback hooks.
