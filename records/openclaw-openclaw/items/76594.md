## Summary

Keep open: the performance idea is narrow, but the provider hook cache in this PR can memoize a temporary in-flight empty result and then skip real provider hooks after plugin loading completes.

## What This Changes

The branch adds config-scoped caches for effective tool policy resolution and provider hook discovery, plus small comment/spacing changes in `src/agents/openclaw-tools.ts`.

## Best Possible Solution

Keep the performance optimization only if provider hook caching excludes in-flight guard results or caches only stable post-load results, with regression coverage for config-scoped reentry.

## Reproduction Assessment

Yes. By source inspection, a config-scoped `resolveProviderPluginsForHooks` call during provider loading can cache the temporary `[]` reentry guard and later return no hooks for the same cache key after loading finishes.

## Solution Assessment

No. The provider hook cache wraps the reentry guard result; the safer fix is to keep that guard outside the cached load or skip storing guard-only empty results.

## Review Findings

Overall correctness: patch is incorrect

This wrapper stores that empty array in the config-scoped cache, so later same-key calls can keep seeing no provider hook plugins after loading completes, skipping normalization or fallback hooks.
