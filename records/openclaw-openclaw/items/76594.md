## Summary

Keep open. The performance idea is narrow, but this PR is not safe to merge as-is because the new provider hook cache can persist the temporary in-flight `[]` reentry result and hide real provider hook plugins after loading completes.

## What This Changes

The PR adds config-scoped caches for effective tool policy resolution and provider hook plugin discovery, plus small spacing/comment changes in `src/agents/openclaw-tools.ts`.

## Best Possible Solution

Keep the optimization only if provider hook discovery caches stable post-load results or skips caching while the in-flight guard is active, with focused nested-load regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a config-scoped `resolveProviderPluginsForHooks` call during provider loading would cache the guard-only `[]`, and later same-key calls would read that cached empty result instead of resolving real hooks.

## Solution Assessment

No. The proposed provider hook wrapper puts the reentry guard inside the cached load path; the narrower maintainable fix is to keep that guard outside the cache or skip storing guard-only empty results.
