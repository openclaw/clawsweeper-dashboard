## Summary

Keep open: the provider-hook cache is not merge-ready because it can persist a temporary in-flight empty result, and this external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds config-scoped caches for effective tool policy resolution and provider hook discovery, plus small spacing/comment changes in the OpenClaw tools file.

## Best Possible Solution

Keep provider-resolution performance work in an operation-scoped cache design, and keep any tool-policy optimization only if it has benchmark proof plus focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a config-scoped resolveProviderPluginsForHooks call made during provider loading would cache the temporary [] guard result, and later same-key calls could keep seeing no hook plugins after loading completes.

## Solution Assessment

No. Caching around the provider-hook reentry guard is not the narrow safe solution; operation-scoped provider resolution caching or skipping in-flight guard results is safer, as in https://github.com/openclaw/openclaw/pull/77848.
