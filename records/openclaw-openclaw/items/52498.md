## Summary

Keep open: current main still carries machine provenance for inter-session turns, but it does not provide the human-readable sender, recipient, or filterable per-message attribution requested for sessions_send. The request overlaps related open identity work, yet it has a sessions_send-specific sender-side recipient and troubleshooting angle that is not terminally superseded.

## What This Changes

Request to show who sent a sessions_send message, and in follow-up also show the recipient or id in the sender's session for troubleshooting.

## Best Possible Solution

Define an additive sender/recipient metadata contract for inter-session turns, persist and expose it through history/live events, render and filter it in Control UI, and coordinate the narrower model-facing requester-name work separately.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main carries source session/channel/tool and optional user senderLabel, but not sessions_send sender and recipient identity suitable for display, filtering, or troubleshooting.

## Solution Assessment

Unclear as a small patch. The maintainable direction is a durable provenance/protocol/history/UI contract rather than appending a name to message text, with https://github.com/openclaw/openclaw/issues/38570 covering the narrower prompt-context requester-name bug.
