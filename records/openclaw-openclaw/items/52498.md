## Summary

Keep open: current main and the latest release still carry only machine-oriented inter-session provenance for `sessions_send`, while Control UI can show a human-readable sender only when a top-level `senderLabel` is already present. The related open issues cover adjacent model-facing identity and broader multi-role UI work, but this focused session-message attribution gap still has useful standalone scope.

## What This Changes

The issue requests human-readable sender and recipient attribution for `sessions_send` messages in session history/UI.

## Best Possible Solution

Add an additive `sessions_send` attribution contract that preserves existing provenance, records or resolves display-safe source and target identity at the session/history boundary, and feeds the existing `senderLabel` UI path without bundling broader multi-role filtering.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send via `sessions_send` and inspect receiving or sender session history/UI; current main persists only machine provenance and renders a human label only when `senderLabel` is already present.

## Solution Assessment

Yes in direction, with a narrower implementation boundary than raw message-prefix injection. The maintainable fix is structured provenance/history/UI attribution, while broader filtering and avatar roles should stay with the larger multi-role tracker.
