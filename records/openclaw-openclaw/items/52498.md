## Summary

Keep open: current main still records only machine-oriented `sessions_send` provenance, and Control UI only renders a human-readable sender when a top-level `senderLabel` is already present. The related open issues cover adjacent model-facing identity and broader multi-role UI work, but this focused message-level session attribution request still has standalone scope.

## What This Changes

Request to show a human-readable sender or recipient identity for messages sent through `sessions_send` in agent sessions.

## Best Possible Solution

Add an additive `sessions_send` attribution contract that preserves existing provenance, records or resolves display-safe source and target identity at the session/history boundary, and feeds the existing `senderLabel` UI path without bundling broader multi-role filtering.

## Reproduction Assessment

Yes, as a source-level capability gap rather than a bug reproduction. Send through `sessions_send` and inspect persisted history/UI: current main carries source session/channel/tool provenance but no display-safe sender or recipient label for rendering.

## Solution Assessment

Yes in direction, with a narrower implementation boundary than raw message-prefix injection. The maintainable fix is structured provenance/history/UI attribution, while broader avatar roles and filtering should stay with the larger multi-role tracker.
