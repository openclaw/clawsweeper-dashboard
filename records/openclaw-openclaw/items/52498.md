## Summary

Keep this open. Current main still preserves only machine-oriented inter-session provenance for sessions_send, while the UI can display a sender label only when a top-level senderLabel is already present; it does not persist or resolve a human-readable sender/recipient identity for the session history view requested here.

## What This Changes

Show sender identity, and ideally sender-side recipient or session id context, for session_send messages in agent sessions.

## Best Possible Solution

Implement an additive core attribution path for sessions_send: either store display-safe source and target identity in inter-session provenance or resolve it when chat.history/sessions.get prepares history, then surface it through senderLabel or equivalent UI metadata without changing existing provenance fields. Cover receiving-session sender display and sender-session target/id visibility with focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is a sessions_send between sessions: the receiving transcript gets inter_session provenance with sourceSessionKey/sourceTool only, while Control UI history rendering only uses an existing top-level senderLabel and has no resolver from that provenance to a human-readable sender.

## Solution Assessment

No proposed patch is under review. The best fix should be additive and contract-safe: preserve existing provenance fields, add or resolve display identity at the session/history boundary, then let the UI consume that label rather than hard-coding sessions_send-specific UI guesses.
