## Summary

Keep open: current main still records `sessions_send` provenance as machine-oriented source session/channel/tool data, while the Control UI only shows a human-readable sender when a top-level `senderLabel` is already present. Related #22774, #38570, and merged #69817 do not fully cover this focused session-message attribution gap.

## What This Changes

Request to show a human-readable sender, and ideally recipient context, on session messages created by `sessions_send`.

## Best Possible Solution

Add an additive `sessions_send` attribution contract that preserves existing provenance fields, records or resolves display-safe source and target identity at the session/history boundary, and feeds the existing `senderLabel` UI path without broad multi-role UI expansion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send via `sessions_send` and inspect receiving session history/UI; current main persists only machine provenance and UI renders a human label only if `senderLabel` is already present.

## Solution Assessment
