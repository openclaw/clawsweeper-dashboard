## Summary

Keep this open: current main still preserves only machine-oriented `sessions_send` provenance and the Control UI only displays sender attribution when a `senderLabel` is already present. The related open #22774 is broader multi-role UI work, #38570 tracks model prompt identity rather than session-history display metadata, and merged #69817 fixed an A2A loop without adding visible sender/recipient attribution.

## What This Changes

Request to show and possibly filter human-readable sender and recipient identity for `sessions_send` messages in session history.

## Best Possible Solution

Add an additive sessions_send attribution contract that preserves existing provenance fields, records or resolves display-safe source and target identity at the session/history boundary, and feeds the existing `senderLabel` UI path without hard-coding sessions_send-specific UI guesses.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: send between sessions and inspect history. The receiving transcript gets `inter_session` provenance with source session/channel/tool only, while the Control UI can display a sender label only if one is already present.

## Solution Assessment

No proposed patch is under review. The best path is an additive contract-safe implementation at the provenance/history boundary, not a UI-only inference from raw session-key strings.
