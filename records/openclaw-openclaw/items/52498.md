## Summary

Keep open: current main still records inter-session provenance for sessions_send, but it does not provide the human-readable sender, recipient, or filterable per-message attribution requested here. The issue overlaps with https://github.com/openclaw/openclaw/issues/22774 and https://github.com/openclaw/openclaw/issues/38570, but neither makes this a terminal close because this issue also asks for sender-side recipient visibility and troubleshooting metadata.

## What This Changes

Request to add visible sender and recipient attribution to sessions_send messages in session history, with possible filtering and logging support.

## Best Possible Solution

Define an additive sender/recipient metadata contract for inter-session turns, persist and expose it through history/live events, and render it in Control UI while coordinating the narrower model-facing requester-name work with the broader durable sender-identity tracker.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence capability gap: current provenance and UI paths carry source session/channel/tool or optional user senderLabel, not sessions_send sender and recipient identity suitable for display or filtering.

## Solution Assessment

Unclear as a small patch. The maintainable direction is a product/API design across provenance, transcript persistence, gateway history/live payloads, Control UI rendering, and sender-side recipient visibility rather than only appending a name to prompt text.
