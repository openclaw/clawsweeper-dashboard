## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Export the broker runtime setter:** `extensions/channel-broker/index.ts:17-18`
  - body: `defineBundledChannelEntry` will load `./runtime-api.js` and require `setChannelBrokerRuntime`, but this PR's `runtime-api.ts` does not export that symbol. When the bundled entry registers and applies the runtime, the loader will throw `missing export "setChannelBrokerRuntime"`, so the channel cannot be packaged/registered in this state.
  - confidence: 0.93
- **[P2] Align capability config with the SDK shape:** `extensions/channel-broker/src/config-schema.ts:39`
  - body: The config schema accepts flat flags like `threads`, `topics`, and `draftPreview`, but the SDK and `brokerPlatformSupports` read nested `delivery`, `live`, `receive`, and `native` capability groups. Configured provider capabilities will therefore validate into a shape the runtime does not honor, making the documented capability contract drift immediately.
  - confidence: 0.86
- **[P2] Validate broker receipts before trusting JSON:** `extensions/channel-broker/src/runtime.ts:46-50`
  - body: The HTTP transport casts any object response to `BrokerReceiptV1` and the send path then trusts fields like `status` and `messageIds`. A malformed provider response can crash or produce confusing delivery errors instead of a controlled broker receipt error, so this external JSON boundary needs schema or explicit field validation.
  - confidence: 0.84
