## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Package the broker runtime before documenting it:** `package.json:46`
  - body: This PR defines `channel-broker` as a bundled channel entry, adds docs/navigation, and adds tests for the plugin, but the root package now excludes `dist/extensions/channel-broker/**`. The existing pack-artifact helper filters those excluded extension dirs, so an installed OpenClaw package would not contain the runtime files needed to load the channel. Either remove this exclusion and add the package-artifact smoke coverage, or make the broker an explicit external/catalog plugin instead of documenting it as bundled.
  - confidence: 0.91
- **[P2] Align capability config with the SDK shape:** `extensions/channel-broker/src/config-schema.ts:39`
  - body: `ChannelBrokerProviderConfig.capabilities` is typed as broker SDK capability metadata and `accounts.ts` passes those objects through to resolved accounts, but the Zod schema only permits flat booleans like `inbound`, `topics`, and `finalEdit`. A user configuring the documented SDK-shaped `delivery`/`live`/`receive` capability metadata would fail config validation. Please make the schema accept the same nested broker capability shape, or remove the config field until that contract lands.
  - confidence: 0.86
- **[P2] Validate provider receipts before trusting JSON:** `extensions/channel-broker/src/runtime.ts:50`
  - body: `parseBrokerReceipt` accepts any object as `BrokerReceiptV1`, but the send path immediately indexes `receipt.messageIds[0]` and maps `receipt.messageIds`. A broker provider returning a successful HTTP response with missing or malformed `messageIds` will produce internal TypeErrors or invalid commits instead of a controlled provider error. This external HTTP boundary should use a Zod schema or narrow guard for version, status, request/provider/platform ids, and message id arrays before returning the receipt.
  - confidence: 0.88
