## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Export the advertised broker runtime setter:** `extensions/channel-broker/index.ts:16-18`
  - body: `index.ts` declares `runtime: { specifier: "./runtime-api.js", exportName: "setChannelBrokerRuntime" }`, and the bundled entry loader throws when that named export is missing. Since `runtime-api.ts` does not export the setter, loading or runtime-setting `channel-broker` can fail before the new channel is usable; export `setChannelBrokerRuntime` from the sidecar and lock it in the guard test.
  - confidence: 0.95
- **[P2] Accept the SDK capability shape in config:** `extensions/channel-broker/src/config-schema.ts:5-22`
  - body: The provider config type and conformance path use the SDK capability groups `delivery`, `live`, `receive`, and `native`, but the Zod schema only accepts old flat booleans such as `outbound`, `topics`, and `draftPreview`. A valid broker provider config using the documented SDK shape will be rejected by the strict schema, so the schema and tests need to match the public contract.
  - confidence: 0.88
- **[P2] Validate broker receipts before committing sends:** `extensions/channel-broker/src/runtime.ts:46-50`
  - body: `parseBrokerReceipt` only checks that provider JSON is an object before casting it to `BrokerReceiptV1`. A malformed successful receipt, for example with a string or missing `messageIds`, can flow into message-id selection and durable receipt creation; validate version, status, provider/request ids, and `messageIds` before treating the send as committed.
  - confidence: 0.87
