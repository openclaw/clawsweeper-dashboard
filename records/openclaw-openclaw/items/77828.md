## Summary

Keep open: current main does not implement the Hermes arbiter metadata contract, but this branch still mixes an unvalidated public metadata field, core-hardcoded Hermes routing, and a media delivery policy regression that needs human design/security review.

## What This Changes

The PR adds Hermes arbiter metadata propagation through Gateway send/outbound hooks, task and async follow-up paths, direct-mode gateway routing, generated Swift models, and tests.

## Best Possible Solution

Define a generic, validated hook metadata contract with a maintainer-approved caller policy, keep async media completions requester-session mediated, and move Hermes routing values into plugin/config-owned code with live metadata-delivery proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main keeps async media completions requester-session mediated and has no public send metadata field, while the PR adds an early direct-send return and arbitrary Gateway metadata. I did not run a live Hermes setup.

## Solution Assessment

No. The proposed path mixes a service-specific Hermes contract, arbitrary client-controlled metadata, and a media delivery policy change into core; a generic validated contract plus plugin/config-owned routing is the safer maintainable direction.

  - confidence: 0.82
