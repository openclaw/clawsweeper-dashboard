## Summary

Keep this PR open for human review: the proposed feature is not implemented on current main, and the PR still has missing real behavior proof, a source-visible media completion regression, and an unresolved Gateway metadata trust-boundary concern.

## What This Changes

The PR adds Hermes arbiter metadata plumbing through Gateway send, outbound message hooks, task/media/exec follow-up delivery, Swift protocol models, and related tests.

## Best Possible Solution

Keep async media completions agent-mediated, design any outbound hook metadata as a generic validated contract, update protocol docs/generated clients together, and require real Gateway/task/media proof before merge.

## Reproduction Assessment

Yes, for the introduced regression by source inspection: current main docs and tests keep async media `directSend` agent-mediated, while the PR changes media completion to call direct delivery and rewrites tests to expect that path. I did not run the PR branch in a live setup.

## Solution Assessment

No. The proposed broad Hermes-specific core/protocol plumbing and unvalidated client metadata path are not the narrowest maintainable solution; a safer direction is a generic, documented, validated hook metadata contract that preserves current media completion behavior.
