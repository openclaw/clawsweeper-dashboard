## Summary

Keep open: current main does not implement this feature, and the PR still needs maintainer design/security review because it adds a public arbitrary metadata surface, hard-coded Hermes routing in core, and changes documented async media completion behavior.

## What This Changes

The PR adds Hermes arbiter metadata propagation through Gateway send, outbound delivery hooks, task and async follow-up paths, media completion sends, Swift protocol models, and focused tests.

## Best Possible Solution

Define a generic, validated hook metadata contract with maintainer-approved caller policy, keep async media completions requester-session mediated, and move Hermes routing values into plugin/config-owned code with live metadata-delivery proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: current docs/types/tests keep async media completions requester-session mediated, while the PR adds an early direct-send return and exposes arbitrary Gateway metadata. I did not run a live Hermes setup.

## Solution Assessment

No. The proposed path mixes a service-specific Hermes contract, arbitrary client-controlled metadata, and a media delivery policy change into core; a generic validated contract plus plugin/config-owned Hermes routing is the safer maintainable direction.

  - body: This adds public Gateway `send.metadata` as arbitrary JSON and forwards it into `message_sending` hook metadata. Hook-control metadata needs a schema, allowlist, and caller policy before clients can set it through the wire protocol.
  - confidence: 0.86

  - body: This hard-codes one Hermes topic and bot name in core task delivery. Core delivery should stay plugin-agnostic and resolve service-specific routing from plugin/config-owned policy, otherwise every deployment reaching this path emits the same Hermes route.
  - confidence: 0.82
