## Summary

Keep open: current main does not implement the Hermes arbiter metadata contract, and this PR still has blocking design/security issues around arbitrary Gateway metadata, hard-coded Hermes routing in core, and changing deprecated async media completion behavior. The contributor updated the body with Gateway status and tests, but it still does not prove live Hermes metadata delivery through the changed path.

## What This Changes

The branch adds Hermes arbiter metadata builder/plumbing through Gateway send, outbound message_sending hooks, task/media/exec follow-up sends, Swift protocol models, and focused tests.

## Best Possible Solution

Define a generic, validated hook metadata contract with maintainer-approved caller policy, keep async media completions requester-session mediated, and move Hermes routing values into plugin/config-owned code with live metadata-delivery proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: current main has a closed send schema with no metadata field and keeps async media completions requester-session mediated, while the PR adds arbitrary Gateway metadata and an early direct-send branch. I did not run a live Hermes setup.

## Solution Assessment

No. The proposed path mixes a service-specific Hermes contract, arbitrary client-controlled metadata, and a media delivery policy change into core; a generic validated contract plus plugin/config-owned Hermes routing is the safer maintainable direction.
