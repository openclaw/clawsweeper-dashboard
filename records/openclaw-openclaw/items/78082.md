## Label Justifications

- `P2`: This is a Discord-specific multi-account reliability problem with real user impact but limited blast radius.
- `impact:message-loss`: A stale specialist-owned slash command can consume a `/side` invocation before the intended main Discord account sees it.

## Summary

Keep open: current main and v2026.6.6 intentionally skip Discord slash-command cleanup when `commands.native=false`, so the stale specialist-command problem is not fixed and needs a maintainer product decision before implementation.

## What This Changes

The issue requests a Discord multi-account behavior where disabling native commands on specialist accounts does not leave stale `/side` or other generic slash commands registered to those specialist apps.

## Best Possible Solution

Define Discord disabled-native semantics so operators can explicitly clean or reconcile stale commands while preserving a zero-REST startup mode, then cover multi-account specialist setups in tests and docs.

## Reproduction Assessment

Yes for the source-level behavior: current main and v2026.6.6 explicitly skip Discord cleanup when native commands are disabled, and tests lock in no deploy and no clear call. I did not run a live Discord registry/client-cache reproduction for the full hijack flow.

## Solution Assessment

Unclear: automatic cleanup would address stale commands, but it conflicts with the current documented zero-REST disabled mode. The safer direction is a maintainer-approved explicit cleanup or deploy-policy path rather than changing `native=false` semantics silently.
