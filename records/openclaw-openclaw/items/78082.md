## Summary

Keep open. Current main still treats Discord `commands.native=false` as a no-REST slash-command lifecycle state, with tests and docs explicitly preserving that behavior; the reporter’s v2026.5.12 follow-up says local cleanup remains necessary for this multi-account setup. The remaining fix needs a maintainer policy decision because cleanup-on-disable conflicts with the shipped no-REST startup contract and overlaps the broader open Discord command-visibility work.

## What This Changes

Request that Discord per-account native-command disable either removes stale registered slash commands or provides supported command ownership controls so specialist apps cannot retain generic commands like `/side`.

## Best Possible Solution

Define an explicit Discord native-command ownership and cleanup policy that preserves the no-REST disabled-startup use case while adding a supported cleanup/status path or per-account command ownership controls with docs and tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register native commands for a specialist account, later set that account's `commands.native=false`, and current main skips command lifecycle calls, leaving provider-side registrations untouched unless an operator cleans them up separately.

## Solution Assessment

Unclear. Automatically deleting commands on `native=false` would address stale registrations, but it conflicts with the shipped no-REST startup behavior; an explicit cleanup/status or command ownership policy is safer.
