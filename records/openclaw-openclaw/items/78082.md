## Label Justifications

- `P2`: The issue affects Discord multi-account slash-command routing but is bounded to specialized deployments rather than the core runtime for all users.
- `impact:message-loss`: A stale specialist-owned slash command can prevent the intended main Discord account from receiving the user request.

## Summary

Keep open: current main and the latest release still intentionally skip Discord slash-command cleanup when `commands.native=false`, so stale specialist-owned commands can remain registered and the reported multi-account `/side` hijack is not fixed. The remaining work needs maintainer product direction because automatic cleanup would reverse the current zero-REST disabled-native semantics.

## What This Changes

Request that Discord per-account native-command disable remove stale application commands or provide an operator-controlled way to keep generic slash commands main-only in multi-account setups.

## Best Possible Solution

Define an explicit Discord command-ownership policy: preserve zero-REST disabled mode by default, then add an operator-controlled cleanup/status or deploy mode that can clear disabled specialist app commands with docs, tests, and live Discord proof.

## Reproduction Assessment

Yes for the source-level core problem: current main and `v2026.6.6` skip Discord command cleanup when `commands.native=false`, while `/side` is a native alias and disallowed specialist interactions return `This channel is not allowed.` I did not live-reproduce Discord’s global/client cache behavior in this read-only review.

## Solution Assessment

Unclear: deleting commands on every disabled account would address the stale registry, but it conflicts with the current zero-REST disabled-native semantics. The safer solution is an explicit cleanup/status or deploy-policy choice with upgrade docs and live proof.
