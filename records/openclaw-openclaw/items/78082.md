## Summary

Keep open. Current main still skips Discord slash-command deploy and cleanup when native commands are disabled, and the reporter's latest v2026.5.12 follow-up says their multi-account install still depends on manual registry cleanup, so this remains a Discord product/design issue rather than a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to prevent stale Discord native slash commands from remaining on specialist accounts after per-account native commands are disabled.

## Best Possible Solution

Adopt an explicit Discord native-command lifecycle model that preserves the no-REST disabled-startup use case while giving multi-account operators a deliberate cleanup or command-ownership path with docs and live Discord proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: register commands for a specialist account, later set that account's `commands.native=false`, and current main skips both deploy and cleanup, leaving provider-side Discord registrations untouched unless the operator removes them separately.

## Solution Assessment

Unclear. Cleanup-on-disable would address stale registrations, but it conflicts with the shipped low-power no-REST startup contract; an explicit cleanup, deploy-policy, or per-account command ownership design is safer.
