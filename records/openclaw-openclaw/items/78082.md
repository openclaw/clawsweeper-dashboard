## Summary

Keep open: current main and v2026.6.10 intentionally skip Discord slash-command registration and cleanup when native commands are disabled, so stale specialist application commands can still remain provider-side. The issue is still necessary, but it needs maintainer product judgment because the requested cleanup conflicts with the current documented zero-REST disabled-native contract.

## What This Changes

The issue requests a supported Discord lifecycle that prevents disabled specialist accounts from retaining stale native commands such as `/side`.

## Best Possible Solution

Define one Discord command lifecycle contract: keep zero-REST disable plus add an explicit operator cleanup/status path, or intentionally restore cleanup-on-disable with docs, tests, and live multi-account proof.

## Reproduction Assessment

Yes, at source level: current main skips the Discord slash-command lifecycle entirely when native commands are disabled, and docs say stale commands may remain. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Unclear. Deleting stale commands on disable would solve the report, but the safest maintainable solution depends on a maintainer decision about whether disabled native commands should remain zero-REST or gain an explicit cleanup lifecycle.
