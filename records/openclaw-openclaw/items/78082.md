## Label Justifications

- `P2`: This is a Discord-specific multi-account reliability problem with real user impact but limited blast radius.
- `impact:message-loss`: A stale specialist-owned slash command can consume a `/side` invocation before the intended main Discord account sees it.

## Summary

Keep open: current main and the latest release intentionally skip Discord slash-command cleanup when `commands.native=false`, so the stale specialist-command problem is not fixed. Any repair needs a maintainer product decision because restoring cleanup conflicts with the zero-REST disabled-native behavior added for related Discord startup failures.

## What This Changes

Requests Discord per-account native-command disable to reconcile or otherwise prevent stale specialist slash commands from capturing generic commands such as `/side`.

## Best Possible Solution

Define Discord disabled-native semantics so operators can either keep zero-REST startup or explicitly clean/reconcile stale commands, with focused tests and docs for multi-account specialist setups.

## Reproduction Assessment

Yes for the source-level behavior: current main and v2026.6.6 explicitly skip Discord cleanup when native commands are disabled. I did not run a live Discord registry/client-cache reproduction for the full hijack flow.

## Solution Assessment

Unclear. Reverting to unconditional cleanup would address stale commands, but it conflicts with the current documented zero-REST disabled mode; maintainers should choose an explicit cleanup, deploy-policy, or per-account ownership model first.
