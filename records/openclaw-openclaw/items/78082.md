## Summary

Keep open. Current main still intentionally skips Discord slash-command registration and cleanup when native commands are disabled, and the shipped docs warn that previously registered commands may remain; the reported multi-account `/side` hijack is credible but needs a maintainer policy choice rather than cleanup closure.

## What This Changes

Request that Discord specialist accounts with native commands disabled either delete stale registered slash commands or provide a supported way to keep generic commands such as `/side` main-only.

## Best Possible Solution

Define the Discord native-command ownership policy first, preserving the no-REST disabled-startup use case while adding an explicit cleanup, include/exclude, or scoped-registration path with docs and tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: register commands for a specialist account, later set that account's `commands.native=false`, and current main skips all command lifecycle calls, leaving provider-side registrations untouched. I did not run live Discord in this read-only review.

## Solution Assessment

Unclear. Automatically deleting commands on `native=false` would address stale registrations, but it conflicts with the shipped low-power no-REST startup behavior; an explicit cleanup or per-account command ownership policy is safer.
