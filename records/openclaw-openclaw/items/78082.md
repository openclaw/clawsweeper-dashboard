## Summary

Current `main` intentionally skips Discord slash-command registration and cleanup when native commands are disabled, so the stale specialist-command behavior is real but not already solved; changing it needs maintainer policy judgment because it conflicts with the documented #76202 low-power startup behavior.

## What This Changes

The issue requests a Discord multi-account path that prevents or removes stale generic native commands, such as `/side`, from specialist bot applications after native commands are disabled for those accounts.

## Best Possible Solution

Choose a Discord native-command ownership policy that preserves the low-power disabled-startup use case while giving multi-account operators an explicit cleanup or per-account generic-command registration path, then update tests and docs together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear path: after commands were previously registered, setting a Discord account's `commands.native` to false makes current `main` skip all command lifecycle calls, leaving provider-side registrations untouched. I did not live-run Discord here, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the policy: automatic cleanup on `native=false` would address stale commands but would reverse the documented startup behavior added for #76202, while include/exclude or explicit cleanup adds a new user-facing control.
