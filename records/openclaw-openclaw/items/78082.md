## Summary

Keeping this open: current main still skips Discord native-command deployment and cleanup when an account resolves `commands.native=false`, so stale provider-side commands can remain; resolving it needs maintainer policy because that behavior is documented and tied to the recent #76202 low-power startup fix.

## What This Changes

The issue asks Discord multi-account setups to remove or avoid stale native slash commands on accounts with native commands disabled, so specialist bots cannot keep owning generic commands like `/side`.

## Best Possible Solution

Choose a Discord command lifecycle policy that preserves disabled-startup no-REST behavior while adding an explicit cleanup or per-account command ownership path, then update Discord tests and docs together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: after commands were registered, setting the Discord account to `commands.native=false` makes current main skip command spec listing, deploy, and cleanup, leaving existing provider-side registrations untouched.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the policy: automatic deletion would solve stale commands but conflicts with the documented #76202 no-REST startup behavior, while include/exclude or explicit cleanup would add a new user-facing control.
