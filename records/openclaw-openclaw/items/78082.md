## Summary

Keep open: current main and v2026.6.10 still intentionally skip Discord command registration and cleanup when native commands are disabled, so the stale specialist-command problem remains a real product decision rather than an implemented cleanup.

## What This Changes

The issue requests a Discord multi-account command lifecycle that prevents stale specialist slash commands such as `/side` from remaining active after that account disables native commands.

## Best Possible Solution

Define a maintainer-approved Discord command lifecycle that preserves a zero-REST disabled mode while adding an explicit cleanup, deploy-policy, or status path for stale specialist commands, with multi-account tests and docs.

## Reproduction Assessment

Yes at the source level: current main and v2026.6.10 explicitly skip Discord command cleanup when native commands are disabled, and tests lock in no deploy or clear call. I did not run a live Discord client-cache reproduction for the full hijack flow.

## Solution Assessment

Unclear: automatic cleanup would address stale commands, but it conflicts with the current documented zero-REST disabled mode. The best path is an explicit maintainer-approved cleanup or deploy-policy contract rather than silently changing `commands.native=false` semantics again.
