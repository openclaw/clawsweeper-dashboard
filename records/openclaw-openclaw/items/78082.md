## Label Justifications

- `P2`: The issue can misroute Discord slash-command invocations in multi-account installs, but the blast radius is limited to a specific Discord specialist-account configuration.
- `impact:message-loss`: A stale specialist `/side` command can capture the interaction before the main account sees it, suppressing the intended command delivery path.

## Summary

Keep open: current main and v2026.6.6 still intentionally skip Discord slash-command registration and cleanup when native commands resolve false, so stale specialist commands are not reconciled. The remaining fix needs maintainer direction because automatic cleanup would conflict with the zero-REST disabled-native startup behavior added for a related startup-timeout bug.

## What This Changes

The issue asks OpenClaw to prevent disabled Discord specialist accounts from leaving stale native slash commands such as `/side` registered where they can capture main-orchestrator commands.

## Best Possible Solution

Define disabled-native semantics for Discord, then implement either a safe explicit cleanup/status path or a documented reconcile mode that removes stale global and guild commands without regressing the zero-REST startup default.

## Reproduction Assessment

Yes from source inspection: with `commands.native=false`, current startup creates no native command specs and skips deploy/cleanup, so stale Discord-side commands are not reconciled. I did not run a live Discord registry cleanup in this read-only review.

## Solution Assessment

Unclear: restoring automatic cleanup is a plausible fix for the stale-command hijack, but it conflicts with the current zero-REST disabled-native startup behavior. Maintainers should choose the intended operator behavior before a fix PR is queued.
