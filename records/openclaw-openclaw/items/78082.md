## Label Justifications

- `P2`: The issue affects a real multi-account Discord command workflow but is limited to Discord native-command ownership and needs design review rather than emergency handling.
- `impact:message-loss`: A stale specialist application command can receive `/side` instead of the main account, preventing the intended agent from seeing the request.

## Summary

Keep open. Current main and the latest release still intentionally skip Discord slash-command registration and cleanup when `commands.native=false`, so the reported multi-account stale-command hijack remains a real product/design gap rather than a cleanup-close candidate.

## What This Changes

Request that disabling native Discord commands per account reconcile/delete stale application commands, or otherwise prevent specialist bot apps from owning generic commands like `/side`.

## Best Possible Solution

Decide Discord disabled-command semantics explicitly, then implement either a deliberate cleanup/reconcile path with multi-account tests and operator messaging, or keep no-REST disabled mode and add a first-class cleanup/status workflow for stale commands.

## Reproduction Assessment

Yes, by source inspection: configure a Discord account with native commands enabled, then set that account to `commands.native=false`; current main skips command deploy and cleanup, and tests assert no cleanup REST path runs. I did not live-run Discord, but the reporter's registry dump matches the current code path.

## Solution Assessment

Unclear until maintainers choose the product contract. Cleanup-on-disable would address stale registrations, but it changes the shipped no-REST disabled behavior; a cleanup/status command or explicit reconcile mode may be the safer permanent shape.
