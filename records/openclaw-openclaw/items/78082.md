## Summary

Keep open: current main intentionally skips Discord slash-command registration and cleanup when native commands are disabled, and the docs/tests describe that behavior, so this is a real product-change request rather than a cleanup-close candidate.

## What This Changes

The issue requests that disabling Discord native commands for a specialist account also removes or avoids stale provider-side commands such as `/side`.

## Best Possible Solution

Define a Discord-owned command reconciliation policy for disabled per-account native commands, then update the startup path, regression tests, and public docs to match that policy.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path for stale commands: enable Discord native commands once, then set that account to `commands.native=false`; startup skips deploy and cleanup. I did not live-run Discord in this read-only review.

## Solution Assessment

Unclear. The requested deletion behavior is plausible for specialist accounts, but current docs/tests intentionally preserve provider-side commands on disable, so the narrow implementation should follow an explicit maintainer decision.
