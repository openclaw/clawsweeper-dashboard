- `impact:message-loss`: A stale specialist-owned slash command can consume a `/side` invocation before the intended main Discord account receives it.

## Summary

Keep open: current main and v2026.6.8 still intentionally skip Discord slash-command registration and cleanup when native commands are disabled, so stale specialist commands can remain provider-side unless an operator cleans them up outside OpenClaw.

## What This Changes

The issue requests a supported Discord multi-account command lifecycle so disabling native commands on specialist accounts does not leave stale `/side`-style commands registered to the wrong bot application.

## Best Possible Solution

Define a maintainer-approved Discord command lifecycle contract that preserves a zero-REST disabled mode while offering an explicit cleanup, deploy-policy, or status path for clearing stale specialist commands, with multi-account tests and docs.

## Reproduction Assessment

Yes for the source-level behavior: current main and v2026.6.8 explicitly skip Discord command cleanup when native commands are disabled, and tests lock in no deploy or clear call. I did not rerun a live Discord client-cache reproduction for the full hijack flow.

## Solution Assessment

Unclear: automatic cleanup would address stale commands, but it conflicts with the current documented zero-REST disabled mode. The best path is an explicit maintainer-approved cleanup or deploy-policy contract rather than silently changing `commands.native=false` semantics again.
