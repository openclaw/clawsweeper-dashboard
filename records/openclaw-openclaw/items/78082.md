## Summary

Keep open: current main and v2026.6.6 still intentionally skip Discord native-command cleanup when `commands.native=false`, so the stale specialist command problem is not fixed and needs maintainer product direction rather than stale cleanup.

## What This Changes

The issue requests Discord per-account native-command disable semantics that remove stale specialist slash commands or provide a first-class main-only/per-account command ownership path.

## Best Possible Solution

Define a documented Discord command-lifecycle policy that preserves zero-REST disabled startup while giving operators an explicit per-account cleanup/status or main-only ownership path for stale specialist slash commands.

## Reproduction Assessment

Yes for the source-level behavior: current main and v2026.6.6 explicitly skip Discord cleanup when native commands are disabled. I did not run a live Discord registry/client-cache reproduction for the full specialist hijack flow.

## Solution Assessment

Unclear. Unconditional cleanup would address stale specialist commands, but it conflicts with the shipped zero-REST disabled-native behavior, so maintainers should choose the explicit cleanup, deploy-policy, or command-ownership model first.
