## Summary

Keep open. The PR targets a real plugin-tool hot path, but the proposed global `pluginId` cache conflicts with the current request-scoped factory contract and can reuse trusted browser, sandbox, session, and delivery context across requests; it also has a cache-hit timing type bug.

## What This Changes

The PR adds a global plugin tool factory result cache keyed by `pluginId`, a reset helper/test, and a changelog entry for the Windows plugin-tool slowdown.

## Best Possible Solution

Redesign caching around a context-safe opt-in contract, complete context-aware key, or cheaper manifest/lazy-tool metadata path, with regression tests for sandbox, browser, session, and delivery-context-sensitive factories.

## Reproduction Assessment

Yes for the PR regression: call `resolvePluginTools()` twice for the same plugin id with different `sandboxed`, browser, session, or delivery context and the PR cache reuses the first tool. Existing Browser, Lobster, Zalo Personal, and TaskFlow tests show those context fields intentionally change factory output or execution behavior.

## Solution Assessment

No. Caching may be the right performance direction, but this implementation is not the narrowest safe fix because it globally memoizes trusted context-bound tool objects and also appears to fail type checking on the cache-hit timing branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Unpack the factory timing result:** `src/plugins/tools.ts:465`
  - body: `describePluginToolFactoryResult()` returns `{ result, resultCount }`, but this cache-hit branch assigns that whole object to `PluginToolFactoryTiming.result`, which is typed as the string union consumed by `formatPluginToolFactoryTiming()`. This should fail type checking and would format timing output incorrectly even if compiled loosely.
  - confidence: 0.97
- **[P1] Preserve request-scoped factory context:** `src/plugins/tools.ts:457`
  - body: Several current factories intentionally close over `OpenClawPluginToolContext`: browser captures sandbox/host policy and session key, Lobster gates on `ctx.sandboxed` and binds TaskFlow to `ctx.sessionKey`, and Zalo Personal captures ambient `deliveryContext`. A cache keyed only by `pluginId` reuses the first tool object for later requests, so later sessions or channels can inherit stale policy or routing.
  - confidence: 0.93
