## Summary

Keep open. Current main and the latest release still omit reasoning on the HTTP surfaces: OpenResponses and Chat Completions only consume assistant text events, while channel-side reasoning exists on a separate thinking path and the docs still say Responses `reasoning` is ignored.

## What This Changes

Request to expose Claude CLI thinking blocks as structured reasoning on Gateway `/v1/responses` and `/v1/chat/completions`, gated by existing reasoning visibility settings.

## Best Possible Solution

Keep this issue as the canonical HTTP reasoning API decision, then land a current-main implementation that is default-silent, explicitly gated, covered for streaming and non-streaming Responses, and deliberate about any Chat Completions compatibility shape.

## Reproduction Assessment

Yes, at source level. Current main documents Responses `reasoning` as ignored and both HTTP handlers project assistant text only, while the separate channel-side thinking event path is not consumed by the HTTP endpoints.

## Solution Assessment

Unclear as a final implementation. The requested direction is the right surface to discuss, but maintainers still need to choose the opt-in policy, event/item shape, and Chat Completions compatibility contract before code should land.
