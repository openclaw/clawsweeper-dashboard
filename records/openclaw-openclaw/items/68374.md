## Summary

Keep open: current main and the latest release still omit Claude CLI thinking from both Gateway HTTP response surfaces, while the direct implementation candidate remains open and blocked on API/security/product review rather than merged.

## What This Changes

Expose Claude CLI reasoning/thinking blocks on `/v1/responses` and `/v1/chat/completions`, preferably behind the existing reasoning visibility controls or an explicit HTTP opt-in.

## Best Possible Solution

Keep this issue open as the canonical product/API decision, then land or replace the open implementation with a gated, documented HTTP reasoning contract that covers streaming and non-streaming Responses plus the chosen Chat Completions compatibility shape.

## Reproduction Assessment

Yes. Source inspection on current main shows Claude CLI `thinking_delta` is ignored by the parser and both HTTP bridges serialize assistant text only; I did not run a live Claude CLI gateway turn in this read-only review.

## Solution Assessment

Unclear as a final design. Exposing reasoning on HTTP is the right surface to discuss, but the best implementation still depends on a maintainer decision for opt-in gating, event shape, and `/v1/chat/completions` compatibility.
