## What This Changes

The issue reports that a cron-triggered `cron.update` tool call without a `patch` makes the agent unresponsive while retrying in the background and consuming tokens.

## Best Possible Solution

Keep this open and fix the cron agent-turn failure path so repeated malformed cron.update tool errors terminate or back off cleanly without starving the active session or rapidly consuming tokens.

## Reproduction Assessment

Yes for the trigger, not for the entire loop: current source can produce the exact `cron failed: patch required` tool error for a cron.update without patch, but this review did not live-reproduce the sustained token-consuming background loop.

## Solution Assessment

No current-main solution is proven; the best fix is likely in cron/agent run failure handling or default loop guards, not in loosening cron.update validation to accept malformed input.
