## Summary

Keep open: current main still leaves the Control UI chat input and mobile controls dependent on `env(safe-area-inset-bottom)` without a defensive floor in a documented standalone PWA surface, and the earlier iOS PWA viewport fix does not cover this composer/control clipping path.

## What This Changes

The issue asks for defensive Control UI bottom clearance for the webchat composer and mobile controls in iOS standalone PWA mode, with cache invalidation considered if stale service-worker caching blocks delivery.

## Best Possible Solution

Land a scoped Control UI layout fix that gives the actual chat input and mobile controls a defensive bottom safe-area floor, adds focused responsive/PWA regression coverage, and only changes service-worker cache behavior if stale shell delivery is proven.

## Reproduction Assessment

No high-confidence live reproduction was established here. The report gives concrete iOS standalone steps, and source inspection confirms the current mobile composer/control clearance still collapses to a small value when the safe-area env value is zero.

## Solution Assessment

Yes for the central layout problem: a scoped CSS safe-area floor on the rendered chat input and mobile controls is the narrowest maintainable fix. Service-worker cache changes should remain secondary unless a patched build still fails to update in standalone mode.
