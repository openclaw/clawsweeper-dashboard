## Summary

Keep open: current `main` still has the bounded transcript readline path, and the same author has an open paired PR for this exact cleanup. A focused Node probe shows the descriptor can stay open after an early parse error while the stream finishes reading, but it did not prove a permanent leak, so the issue should track the paired PR rather than be closed independently.

## What This Changes

The issue asks for explicit cleanup of the readline interface and read stream in `TranscriptsStore.readUtterancesFromDir` when bounded JSONL parsing exits on error.

## Best Possible Solution

Resolve https://github.com/openclaw/openclaw/pull/98140 first: either land a narrow defensive cleanup with proof or close the PR and issue together if maintainers agree the reported leak is not real.

## Reproduction Assessment

Yes, with nuance: a focused Node probe matching the current source shows an FD remains open briefly after an early `JSON.parse` throw on a large stream. It did not reproduce a permanent leak because the descriptor closed after the stream finished reading.

## Solution Assessment

Yes, conditionally: explicit `rl.close()` plus `stream.destroy()` is the narrow defensive cleanup and matches the sibling helper. The paired PR still needs maintainer review because the issue’s permanent-leak wording is stronger than the observed delayed-release behavior.
