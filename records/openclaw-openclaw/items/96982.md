## Summary

Keep open: current main and the latest release still preserve Telegram Bot API `/bot<TOKEN>` path segments in fetch-timeout diagnostic URLs, so the reported credential-redaction gap remains. The fix is narrow, but it should go through security-aware maintainer handling rather than an automated repair lane.

## What This Changes

The issue requests redacting Telegram Bot API `/bot<TOKEN>` path segments from fetch-timeout diagnostic URLs while preserving existing userinfo/query/hash sanitization.

## Best Possible Solution

Add a small Telegram Bot API path-token sanitizer in `src/utils/fetch-timeout.ts` for both parsed and fallback URL paths, with focused security regression tests.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: call `buildTimeoutAbortSignal` with a Telegram Bot API URL containing `/bot<TOKEN>/sendMessage`, and the current sanitizer keeps that pathname. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the narrow fix belongs in the timeout URL sanitizer rather than relying only on broader log redaction, because this helper already owns timeout diagnostic URL sanitization and general log redaction can be configured off.
