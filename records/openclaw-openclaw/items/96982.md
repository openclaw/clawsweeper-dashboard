## Summary

Keep open: current main still lets the timeout URL sanitizer preserve Telegram Bot API tokens in URL path segments, and the latest release has the same behavior. This is a narrow security-hardening issue, but it should go through security-aware maintainer handling rather than an automated repair lane.

## What This Changes

The issue asks for fetch-timeout URL diagnostics to replace Telegram Bot API `/bot<TOKEN>` path segments with `/bot<redacted>` in both parsed and fallback sanitization paths.

## Best Possible Solution

Add a small timeout URL sanitizer for Telegram Bot API `/bot<TOKEN>` path segments in `src/utils/fetch-timeout.ts`, with focused tests covering parseable and fallback URL paths while keeping non-Telegram paths unchanged.

## Reproduction Assessment

Yes, there is a high-confidence source reproduction path: call the timeout helper with a Telegram Bot API URL containing `/bot<TOKEN>/sendMessage`, and the current sanitizer keeps that pathname. I did not execute tests because this review is read-only.

## Solution Assessment

Yes, the narrow fix belongs in the timeout URL sanitizer rather than relying on broader log redaction, because timeout tests already treat this helper as a separate safety boundary and logging redaction can be configured off.
