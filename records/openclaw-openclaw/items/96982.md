## Summary

Keep open: current main still has a narrow credential-redaction gap in the fetch-timeout URL sanitizer. Telegram probe and audit paths pass Bot API URLs containing `/bot${token}` into `fetchWithTimeout`, and the sanitizer strips userinfo/query/hash but leaves path tokens intact before putting the URL into warning metadata and console text.

## What This Changes

The issue asks fetch-timeout URL logging to redact Telegram Bot API `/bot<TOKEN>` path segments while preserving endpoint context and existing userinfo/query/hash sanitization.

## Best Possible Solution

Patch `src/utils/fetch-timeout.ts` so both parsed URLs and fallback cleaned strings replace Telegram Bot API `/bot...` path segments before truncation, then add focused regression cases in `src/utils/fetch-timeout.test.ts`.

## Reproduction Assessment

Yes from source inspection, though no runtime test was run in this read-only review. Telegram probe/audit callers build `/bot${token}` URLs and the timeout sanitizer serializes or cleans the path without replacing that token segment.

## Solution Assessment

Yes: a narrow sanitizer change in `src/utils/fetch-timeout.ts` with regression tests is the best fit. Relying on later logger redaction is weaker because the helper still emits pre-redacted structured metadata and may miss noncanonical or test-token shapes.
