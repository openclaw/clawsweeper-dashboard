## Summary

Keep this issue open. Current main implements `openclaw config validate`, write-time validation, SecretRef builder mode, and env-reference restoration, but bare `openclaw config` is still intentionally documented and implemented as the guided configure wizard entrypoint.

## What This Changes

The issue requests safer `openclaw config` behavior: show help instead of launching the wizard on configured systems, add explicit config validation, and avoid persisting env-resolved secrets during config writes.

## Best Possible Solution

Keep the implemented validation and SecretRef/env-preservation pieces, then make a maintainer decision on whether bare `openclaw config` should show help on fully configured systems and update CLI behavior, tests, docs, and changelog together.

## Reproduction Assessment

Yes, from source inspection: the root `config` action calls `configureCommandFromSectionsArg`, and the no-section path runs `runConfigureWizard`. The docs and tests align with that current-main behavior.

## Solution Assessment

No, closing is not the best solution yet. The implemented validation and secret-preservation work should stay, but the remaining root-command and value-mode UX needs a maintainer compatibility/product decision.
