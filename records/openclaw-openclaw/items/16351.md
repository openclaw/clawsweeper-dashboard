## Summary

Keep this open. Current main already has `openclaw config validate` and existing `${ENV}` write-back preservation, but bare `openclaw config` is still a documented wizard entrypoint and value-mode `config set` can still persist newly supplied literal secret strings.

## What This Changes

Request `openclaw config` to avoid launching the wizard by default, add explicit validation, and prevent config writes from turning secrets or invalid config into unsafe files.

## Best Possible Solution

Keep the implemented validation and env-preservation work, then make a maintainer-approved CLI semantics and secret-writing policy change with matching docs and focused regression tests.

## Reproduction Assessment

Yes, by source inspection: bare `openclaw config` still dispatches to the configure wizard, and value mode still accepts direct string assignment while warning that dry-run has limited checks. Current code and docs give a high-confidence path for the remaining behavior without mutating the checkout.

## Solution Assessment

Unclear until maintainer direction is given. Showing help for the root command and tightening literal-secret writes are plausible, but both change documented CLI behavior or credential-writing policy, so the maintainable path is an explicit product/security decision before implementation.
