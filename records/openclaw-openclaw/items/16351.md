## Summary

Keep this issue open. Current main implements the validation and existing env-placeholder preservation pieces, but the remaining root `openclaw config` wizard behavior and plaintext value-mode secret policy are still documented current behavior and need maintainer direction.

## What This Changes

The issue requests safer `openclaw config` behavior: show help instead of the root wizard on configured systems, add explicit config validation, and avoid persisting resolved environment-placeholder secrets.

## Best Possible Solution

Keep the implemented validation and env-preservation work, then make a maintainer-approved CLI semantics and secret-writing policy change with matching docs and focused regression tests.

## Reproduction Assessment

Yes for the remaining observable behavior. Source inspection shows bare `openclaw config` still dispatches to the configure wizard, and docs describe that as the supported current behavior; the value-mode secret concern is also visible from the dry-run code path and docs.

## Solution Assessment

No, closing or queueing an automated fix is not the best way to solve this yet. The narrow maintainable path is a maintainer decision on root command semantics and value-mode secret UX, followed by CLI, docs, and test updates.
