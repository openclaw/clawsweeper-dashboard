## Summary

Current main resolves two of the three requests but intentionally retains the remaining behavior: bare `openclaw config` opens the guided configure wizard, and literal-value mode remains distinct from explicit SecretRef assignment. The issue should stay open for a focused maintainer product/security decision; the linked open maintainer PR only hardens malformed input paths and does not own those remaining decisions.

## What This Changes

Improve `openclaw config` so bare invocation is less surprising, configuration can be validated explicitly, and resolved credentials are not persisted as plaintext.

## Best Possible Solution

Retain `config validate`, schema-checked writes, explicit SecretRef assignment, and existing-placeholder restoration; separately define a documented root-command dispatch rule and an explicit secret-entry workflow that avoids plaintext without silently inventing credential storage policy.

## Reproduction Assessment

Yes for the current behavior: source and docs deterministically show that bare `openclaw config` opens the wizard, and source distinguishes literal values from explicit SecretRef assignments. The issue does not provide a redacted runtime example for the exact newly-added secret path, so this is source reproduction rather than a live reproduced incident.

## Solution Assessment

No as one blanket implementation: validation and existing-placeholder preservation are already solved, while root-command behavior and creation of new secret-backed values need explicit, separately tested compatibility and security decisions.
