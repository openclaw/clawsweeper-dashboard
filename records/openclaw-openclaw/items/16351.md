## Summary

Keep open: current main and v2026.5.12 already cover explicit validation and existing env-placeholder restoration, but bare `openclaw config` is still a documented wizard entrypoint and value-mode literal-secret policy remains unresolved.

## What This Changes

Request to make `openclaw config` safer by showing help on configured systems, adding explicit config validation, and preventing `config set` from persisting resolved secret placeholders.

## Best Possible Solution

Keep the shipped validation and env-preservation work, then make a maintainer-approved CLI semantics and secret-writing policy change with matching docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.12 shows bare `openclaw config` still dispatches to the configure wizard, and value-mode dry-run explicitly skips schema and SecretRef checks.

## Solution Assessment

Unclear until maintainers decide the product and security policy. Showing help for the root command and tightening literal-secret writes are plausible, but both change documented CLI behavior or credential-writing semantics.
