## Summary

Keep open: current main implements `openclaw config validate` and preserves existing environment-backed placeholders on write-back, but bare `openclaw config` is still documented and implemented as a guided wizard entrypoint, and secret-safe literal `config set` behavior still needs maintainer product/security direction.

## What This Changes

The issue asks for `openclaw config` to be safer by showing help instead of entering the wizard, adding explicit config validation, and avoiding plaintext secret writes from resolved environment placeholders.

## Best Possible Solution

Keep this issue open for a maintainer-approved CLI/security design that preserves the shipped validation command and existing placeholder restoration while deciding the root command and secret-safe literal value behavior.

## Reproduction Assessment

Yes for the remaining root-command behavior: current source routes bare `openclaw config` to `configureCommandFromSectionsArg`, which opens the wizard on an interactive terminal when no sections are supplied. The literal-secret concern is source-supported through value-mode parsing, but the exact desired guardrail is a product/security decision rather than a strict existing-behavior bug.

## Solution Assessment

No, current main is not the complete solution: validation and existing placeholder restoration are implemented, but the root command and secret-safe literal write policy remain unsettled. The safer path is a maintainer-sponsored design that keeps `openclaw configure` and `--section` compatibility explicit while defining secret-looking literal write behavior.
