## Summary

Keep open. Current main has implemented `openclaw config validate` and existing `${ENV}` placeholder restoration, but the remaining central behavior is still unresolved: bare `openclaw config` is documented and implemented as the configure wizard entrypoint, and value-mode secret-safe UX still needs maintainer direction.

## What This Changes

The issue requests safer `openclaw config` behavior: avoid root-command wizard surprises, add explicit validation, and prevent resolved environment-backed secrets from being persisted into config files.

## Best Possible Solution

Keep this issue as the canonical maintainer decision point: keep the shipped validation and existing-placeholder restoration, then decide whether bare `openclaw config` should become help-only on configured systems and whether value-mode credential writes need an explicit safer UX.

## Reproduction Assessment

Yes for the root command behavior: current source and docs show bare `openclaw config` enters the configure wizard. The new-secret plaintext write case is source-plausible because new keys are kept as-is, but I did not run a live CLI repro in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The best fix needs a maintainer-approved CLI/security UX decision that preserves `config validate`, SecretRef builder mode, and existing env-placeholder restoration while resolving the root-command and new-secret value-mode behavior.
