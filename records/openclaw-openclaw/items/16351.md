## Summary

Keep open. Current main has implemented `openclaw config validate`, write-time validation, SecretRef builder UX, and env-placeholder restoration, but the central root-command request is still unresolved: bare `openclaw config` remains implemented and documented as the guided configure wizard entrypoint.

## What This Changes

The issue requests safer `openclaw config` behavior: show help instead of starting onboarding on configured systems, add explicit config validation, and avoid persisting resolved environment-backed secrets.

## Best Possible Solution

Keep the implemented validation and env-restoration behavior, then make a maintainer product decision on root `config` semantics. If the root command should stop launching the wizard on configured systems, update `src/cli/config-cli.ts`, the configure fallback path, focused CLI tests, and the public config/configure docs together, while preserving `openclaw configure` and `openclaw config --section ...` as explicit wizard paths.

## Reproduction Assessment

Yes for the remaining behavior: the source trace shows bare `openclaw config` dispatches `configureCommandFromSectionsArg`, and empty sections call the configure wizard. Live CLI execution was not needed for this read-only review because code and docs agree on the behavior.

## Solution Assessment

Unclear. The implemented validate and env-placeholder pieces are appropriate, but changing the root command is a compatibility/product decision because current code and public docs intentionally expose the wizard there.
