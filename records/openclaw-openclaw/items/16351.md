## Summary

Keep this issue open. Current main and v2026.6.6 already cover `openclaw config validate` and existing `${ENV}` write-back restoration, but bare `openclaw config` is still documented and implemented as the guided configure wizard, and the remaining secret-safe value-mode behavior needs maintainer product/security judgment.

## What This Changes

Request to make `openclaw config` safer by showing help instead of entering the wizard, adding explicit config validation, and avoiding plaintext secret write-back from resolved environment placeholders.

## Best Possible Solution

Decide the root-command UX and secret-safe value-mode policy in one focused maintainer-sponsored change, preserving the shipped `config validate`, SecretRef builder, and existing env-placeholder restoration behavior.

## Reproduction Assessment

Yes for the bare command behavior: current source and docs show `openclaw config` enters the configure wizard. The intermittent plaintext-secret write-back case is only partially source-proven because existing env refs are restored but new value-mode keys remain literal.

## Solution Assessment

No final fix direction is established yet. The best path is a maintainer decision on whether to change the documented root command contract and how strongly `config set` should steer or reject secret-looking value-mode writes.
