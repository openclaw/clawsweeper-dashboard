## Summary

Keep open: current main implements the validation subcommand and preserves existing authored `${ENV}` placeholders on write-back, but bare `openclaw config` is still documented and wired as a guided setup entry point, and literal value-mode writes to secret-like paths still require a maintainer security/product policy decision.

## What This Changes

The issue requests safer `openclaw config` behavior: show help instead of starting guided setup in some cases, add explicit config validation, and avoid persisting secrets when config values come from env placeholders.

## Best Possible Solution

Keep the shipped validation and existing-placeholder preservation, then make a maintainer-approved CLI semantics and credential-write policy change with focused docs and regression coverage.

## Reproduction Assessment

Yes, from source inspection. Bare `openclaw config` still routes to the configure wizard, and value mode still carries raw assigned strings while env restoration only preserves existing authored placeholders.

## Solution Assessment

Unclear until maintainers decide the product and security policy. Showing help for the root command and tightening literal secret writes are plausible, but both change documented CLI or credential-writing semantics.
