## Summary

Keep open: current main has implemented validation and existing env-placeholder restoration, but bare `openclaw config` still intentionally opens the wizard and the remaining literal-secret write policy needs maintainer product/security judgment.

## What This Changes

The issue asks to make `openclaw config` safer by showing help instead of the wizard on configured systems, adding explicit config validation, and avoiding plaintext secret writes when env placeholders are involved.

## Best Possible Solution

Keep the shipped validation, SecretRef builder, doctor warning, and env-placeholder preservation, then make a maintainer-approved CLI semantics and literal-secret policy change with focused docs and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still routes bare `openclaw config` into the configure wizard, and value mode can still persist raw strings while env restoration only preserves existing `${ENV}` placeholders.

## Solution Assessment

Unclear until maintainers decide the product and security policy. Showing help for the root command and tightening literal secret writes are plausible, but both change documented CLI or credential-writing semantics.
