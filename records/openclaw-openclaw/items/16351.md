## Summary

Keep open: current main has implemented `openclaw config validate`, write-time validation, dry-run SecretRef checks, and preservation of existing environment-backed secrets, but bare `openclaw config` is still documented and implemented as the guided wizard entrypoint and literal `config set` secret handling still needs product/security direction.

## What This Changes

The issue asks for safer `openclaw config` behavior: help instead of the wizard for bare config, explicit config validation, and avoiding resolved secrets being written back.

## Best Possible Solution

Keep this issue open for a maintainer-approved CLI/security design that preserves the shipped validation command and existing placeholder restoration while deciding the root command and secret-safe literal write policy.

## Reproduction Assessment

Yes for the root-command part by source inspection: `registerConfigCli` calls `configureCommandFromSectionsArg`, and that path runs the configure wizard when no section is supplied on an interactive terminal. The literal-secret concern is also source-supported because value/json mode parses and writes direct values, but the desired guardrail is a product/security policy choice.

## Solution Assessment

No, current main is only a partial solution: validation and existing placeholder restoration are implemented, but the root command and raw secret-like value behavior remain unsettled. The safer path is a maintainer-sponsored design that keeps `openclaw configure` and `--section` compatibility explicit while defining secret-looking literal write behavior.
