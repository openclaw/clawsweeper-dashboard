## Summary

Keep open. Current main has implemented the validation and existing env-placeholder preservation pieces, but the remaining root-command behavior is still intentionally documented and implemented as the guided configure wizard, so this needs maintainer UX/product follow-up rather than cleanup closure.

## What This Changes

The issue requests safer `openclaw config` behavior: help instead of launching the wizard, explicit config validation, and avoiding plaintext secret writes when env placeholders are involved.

## Best Possible Solution

Keep the issue open, decide the root `openclaw config` help-versus-wizard behavior, then update the CLI, focused tests, and docs while preserving the existing `validate`, SecretRef builder, and env-placeholder restoration behavior.

## Reproduction Assessment

Yes, the remaining behavior is source-reproducible: the root `config` action dispatches to the configure wizard and the docs say that is expected. The implemented validation/env-preservation pieces are also source-backed with tests.

## Solution Assessment

No, closing is not the best outcome because a central requested behavior remains intentionally present. The narrow maintainable path is a maintainer decision on root-command UX followed by a small CLI/docs/test change.
