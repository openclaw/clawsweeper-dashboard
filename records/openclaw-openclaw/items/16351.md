## Summary

Keep this issue open. Current main already implements `openclaw config validate`, write-time validation, env-reference restoration, and safer SecretRef/provider builder flows, but bare `openclaw config` is still intentionally implemented and documented as a configure-wizard entrypoint, and plain value-mode secret policy still needs maintainer direction.

## What This Changes

The issue requests safer `openclaw config` behavior: show help instead of launching onboarding on configured systems, add explicit config validation, and avoid persisting resolved environment-backed secrets.

## Best Possible Solution

Keep the implemented validation, SecretRef builder, dry-run, and env-preservation pieces, then align root `config` semantics and secret-writing UX across CLI behavior, docs, tests, and changelog after a maintainer decision.

## Reproduction Assessment

Yes for the remaining observable behavior, by source inspection rather than a live CLI run. The root `config` action still dispatches to the configure wizard path, and docs still describe that as the current behavior.

## Solution Assessment

No, closing is not the best way to solve this issue yet. The narrow maintainable path is a maintainer decision on the documented root command and value-mode secret UX, followed by focused CLI, docs, tests, and changelog changes.
