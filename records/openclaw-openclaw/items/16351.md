## Summary

Keep this issue open. Current main implements the validation, write-time validation, env-reference restoration, and safer SecretRef/builder pieces, but the remaining root-command behavior and plain value-mode secret policy are still documented current behavior and need maintainer direction.

## What This Changes

Request to make `openclaw config` safer by showing help instead of the wizard on configured systems, adding explicit config validation, and avoiding plaintext secret persistence from env placeholders.

## Best Possible Solution

Keep the implemented validation, SecretRef builder, dry-run, and env-preservation pieces, then align root `config` semantics and secret-writing UX across CLI behavior, docs, tests, and changelog after a maintainer decision.

## Reproduction Assessment

Yes for the remaining observable behavior. Source inspection shows the root `config` action still calls `configureCommandFromSectionsArg`, the no-section path runs the configure wizard, and docs describe the same current-main behavior; I did not run the interactive command in this read-only pass.

## Solution Assessment

No, closing or queueing an automated fix is not the best way to solve this yet. The narrow maintainable path is a maintainer decision on the documented root command and value-mode secret UX, followed by focused CLI, docs, tests, and changelog changes.
