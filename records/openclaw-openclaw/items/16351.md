## Summary

Keep this issue open. Current main already covers `openclaw config validate`, write-time validation, env-reference restoration, and safer SecretRef/builder flows, but the remaining root-command behavior is still intentionally implemented and documented as the configure wizard, and plain value-mode secret policy still needs maintainer direction.

## What This Changes

The issue requests safer `openclaw config` behavior: show help instead of entering the wizard on configured systems, add config validation, and avoid persisting resolved environment secrets.

## Best Possible Solution

Keep the implemented validation, SecretRef builder, dry-run, and env-preservation pieces, then align root `config` semantics and secret-writing UX across CLI behavior, docs, tests, and changelog after a maintainer decision.

## Reproduction Assessment

Yes for the remaining observable behavior. Source inspection shows the root `config` action still calls `configureCommandFromSectionsArg`, the no-section path runs the configure wizard, and docs describe the same current-main behavior.

## Solution Assessment

No, closing is not the best way to solve this issue yet. The narrow maintainable path is a maintainer decision on the documented root command and value-mode secret UX, followed by focused CLI, docs, tests, and changelog changes.
