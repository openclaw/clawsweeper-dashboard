## Summary

Keep open: current main has already added `config validate` and existing `${ENV}` placeholder restoration, but bare `openclaw config` still intentionally enters the configure wizard and value-mode secret writes still need a maintainer security/product decision.

## What This Changes

The issue requests safer `openclaw config` behavior: help instead of the root wizard on configured systems, explicit config validation, and avoiding resolved secret values in config writes.

## Best Possible Solution

Keep the shipped validation, SecretRef builder, and env-placeholder preservation, then make a maintainer-approved CLI semantics and literal-secret policy change with focused docs and regression coverage.

Unclear until maintainers decide the product and security policy. Showing help for the root command and tightening literal secret writes are plausible, but both change documented CLI or credential-writing semantics.
