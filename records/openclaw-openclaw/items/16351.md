## Summary

Keep this open. Current main has implemented `openclaw config validate` and existing `${ENV}` write-back preservation, but bare `openclaw config` is still documented and implemented as the guided configure wizard, and value-mode `config set` still permits literal string writes that can include newly supplied secrets.

## What This Changes

The issue asks to make `openclaw config` safer by showing help instead of launching the wizard, validating config explicitly, and avoiding plaintext secret writes or resolved env-placeholder persistence.

## Best Possible Solution

Keep the implemented validation and env-preservation work, then make a maintainer-approved CLI semantics and secret-writing policy change with matching docs and focused regression tests.

## Reproduction Assessment

Yes, by source inspection: bare `openclaw config` still dispatches to the configure wizard, and value mode still accepts direct string assignment while warning that dry-run has limited checks. I did not run the CLI because this was a read-only review, but current code and docs give a high-confidence reproduction path for the remaining behavior.

## Solution Assessment

Unclear until maintainer direction is given. Showing help for the root command and tightening literal-secret writes are plausible, but both change documented CLI behavior or credential-writing policy, so the maintainable path is an explicit product/security decision before implementation.
