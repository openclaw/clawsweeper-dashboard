## Summary

Keep open: current main has implemented `config validate` and preserves existing env placeholders on write-back, but the remaining behavior is still unresolved because bare `openclaw config` is documented and implemented as a configure-wizard alias, and value mode can still persist newly supplied literal secret strings. The related draft hardening PR is useful adjacent work, but it does not close the root-command or literal-secret policy questions.

## What This Changes

The issue requests safer `openclaw config` behavior: help instead of an unexpected wizard, explicit config validation, and avoiding plaintext secret writes when env placeholders are involved.

## Best Possible Solution

Keep the shipped validation and env-placeholder preservation, then make a maintainer-approved CLI semantics and credential-write policy change with focused docs and regression coverage.

## Reproduction Assessment

Yes, from source inspection. Bare `openclaw config` still routes to the configure wizard, and value mode can still carry raw strings while env restoration only preserves existing placeholders on unchanged paths.

## Solution Assessment

Unclear until maintainers decide the product and security policy. Showing help for the root command and tightening literal secret writes are plausible, but both change documented CLI or credential-writing semantics.
