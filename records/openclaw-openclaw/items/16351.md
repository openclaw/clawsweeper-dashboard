## Summary

Keep this open. Current main has implemented the validation command and existing env-placeholder write-back preservation, but the remaining behavior still needs maintainer direction: bare `openclaw config` is documented and implemented as a guided wizard entrypoint, and value-mode `config set` can still persist newly supplied literal secret strings.

## What This Changes

The issue requests safer `openclaw config` behavior: help instead of accidental wizard entry, explicit config validation, and avoiding plaintext secret write-back from resolved environment placeholders.

## Best Possible Solution

Keep the implemented validation and env-preservation work, then make a maintainer-approved CLI semantics and secret-writing policy change with matching docs and focused regression tests.

## Reproduction Assessment

Yes, by source inspection: bare `openclaw config` still dispatches to the configure wizard, and value mode still accepts direct string assignment while warning that dry-run has limited checks. I did not run the CLI because this was a read-only review, but current code and docs give a high-confidence reproduction path for the remaining behavior.

## Solution Assessment

No, closing or queueing an automated fix is not the best path yet. The remaining work changes documented CLI behavior and credential-writing policy, so the narrow maintainable path is maintainer direction followed by CLI, docs, and test updates.
