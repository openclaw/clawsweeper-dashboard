## Summary

Keep this issue open. Current main has implemented the validation and much of the secret-safe write-back work, but bare `openclaw config` is still documented and implemented as the guided configure wizard entrypoint, so the remaining request needs maintainer product judgment rather than cleanup closure.

## What This Changes

The issue asks for `openclaw config` to show safer non-interactive behavior, add explicit config validation, and avoid writing resolved secret placeholders back to config files.

## Best Possible Solution

Keep the implemented `config validate`, SecretRef builder, write validation, and env-ref restoration, then make a maintainer decision on whether bare `openclaw config` should switch to help on configured systems and update tests/docs accordingly.

## Reproduction Assessment

Yes from source inspection: the root `config` command calls the configure path, and the no-section configure helper runs the wizard. I did not run the CLI because this review is read-only and `pnpm` is unavailable, but current code, docs, and tests are consistent.

## Solution Assessment

No closure is not the best solution yet. The implemented validation and secret-preservation pieces should stay, but the remaining root-command behavior is a compatibility/product decision rather than a completed fix.
