Keep open. Current main still has the reported provider-error diagnostics gap, and the PR's shared-helper fix with focused tests and a changelog entry appears narrow and correct; it should continue through normal maintainer review and exact-head CI rather than be closed.

## What This Changes

The PR updates shared provider HTTP error extraction to catch response-reader failures, parse SSE `data:` error envelopes, use HTTP status text as an empty-body fallback, and add regression tests plus a changelog entry.

## Best Possible Solution

Land the shared-helper fix after exact-head CI finishes green and maintainer review, keeping the behavior generic in `src/agents/provider-http-errors.ts` with the targeted regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main returns no detail for empty provider error bodies and does not catch body-reader failures before formatting provider HTTP errors. The Google streaming transport routes non-OK responses through that same helper.

## Solution Assessment

Yes. Centralizing the fallback and SSE-envelope parsing in the shared provider HTTP helper is the narrowest maintainable fix because the same SDK surface is used by Google streaming and other provider/capability plugins.
