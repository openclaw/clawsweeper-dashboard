## Summary

Keep open. Current main still has the reported provider-error detail gap, and this PR is a narrow shared-helper fix with focused tests and a changelog entry; I found no blocking diff findings, but it is still draft and exact-head status is pending.

## What This Changes

The PR updates shared provider HTTP error extraction to catch response-body read failures, parse SSE `data:` error envelopes, fall back to `response.statusText`, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land the shared-helper fix after the draft state is resolved and exact-head CI is green, keeping the behavior generic in `src/agents/provider-http-errors.ts` with the targeted regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main returns no detail for empty provider error bodies and can let body-reader failures escape before formatting provider HTTP errors. The Google streaming transport routes non-OK responses through that same helper.

## Solution Assessment

Yes. Centralizing the fallback and SSE-envelope parsing in the shared provider HTTP helper is the narrowest maintainable fix because the same SDK surface is used by Google streaming and other provider/capability plugins.
