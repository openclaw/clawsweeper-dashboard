## Summary

Keep open: the PR targets a valid MiniMax Portal OAuth refresh bug through the right provider-owned seam, but the latest diff still misroutes released CN profiles that lack `resourceUrl` metadata and pulls OAuth implementation into the eager provider-registration path.

## What This Changes

The PR adds a MiniMax Portal refresh-token grant, stores `resourceUrl` on new OAuth credentials, wires `refreshOAuth`, adds Global/CN tests, and updates the changelog plus raw-fetch allowlist offsets.

## Best Possible Solution

Land MiniMax Portal refresh after preserving or safely rejecting unknown legacy regions, keeping OAuth code behind the runtime boundary, and covering Global, CN, and missing-metadata profiles.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main has no `minimax-portal` `refreshOAuth` hook, and core refresh falls through to pi-ai built-ins that do not include MiniMax; I did not run a live wait-for-expiry flow.

## Solution Assessment

No. The provider-owned refresh hook is the right direction, but this PR is not yet the safest solution because it guesses Global for missing legacy metadata and moves a runtime helper into the eager registration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing MiniMax OAuth profiles from current releases do not store `resourceUrl`; their CN/Global choice lives in provider config. This line turns missing metadata into Global, so upgraded CN profiles send refresh tokens to `api.minimax.io` and still fail. Derive the legacy region from a reliable source, migrate/fill metadata, or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.9
- **[P3] Keep the resolver behind the lazy OAuth boundary:** `extensions/minimax/provider-registration.ts:22`
  - body: Provider registration currently reaches OAuth code through `oauth.runtime.ts`. Adding a value import from `./oauth.js` pulls the OAuth implementation into the eager registration path while refresh still dynamically imports the runtime module. Move the resolver behind the runtime export or into a lightweight helper that does not import the full OAuth implementation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependencies, but it can submit legacy CN refresh tokens to the Global MiniMax endpoint when region metadata is absent.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
  - body: When `resourceUrl` is absent, `refreshOAuth` defaults to Global. Legacy CN profiles from current releases can therefore submit their stored refresh token to `api.minimax.io` instead of `api.minimaxi.com`, which is both a functional failure and unnecessary token exposure.
  - confidence: 0.87
