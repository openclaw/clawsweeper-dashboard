## Summary

Keep this PR open: it fixes a real MiniMax Portal OAuth refresh gap in the right provider-owned area, but the current diff still has a blocking legacy CN-region regression/security concern and lacks real after-fix MiniMax refresh proof.

## What This Changes

The PR adds a MiniMax Portal OAuth refresh-token flow, stores login-time `resourceUrl` on new OAuth credentials, wires `refreshOAuth`, adds provider tests, updates the changelog, and adjusts raw-fetch allowlist line offsets.

## Best Possible Solution

Land a provider-owned MiniMax Portal refresh hook after it handles legacy profiles without cross-region token submission, preserves the lazy OAuth boundary, and includes real after-fix refresh evidence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: expired `minimax-portal` OAuth credentials call the plugin refresh seam, find no MiniMax hook, then fall through to pi-ai built-ins that omit MiniMax.

## Solution Assessment

No. The provider-owned `refreshOAuth` hook is the right direction, but this diff is not the safest implementation until legacy region metadata is handled without defaulting to Global and the lazy runtime boundary is preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing MiniMax OAuth profiles from current releases do not store `resourceUrl`; their CN/Global choice lives in provider config. This line turns missing metadata into Global, so upgraded CN profiles send refresh tokens to `api.minimax.io` and still fail. Derive the legacy region from a reliable source, backfill metadata, or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.91
- **[P3] Keep the resolver behind the lazy OAuth boundary:** `extensions/minimax/provider-registration.ts:22`
  - body: Provider registration currently reaches OAuth code through `oauth.runtime.ts`. Adding a value import from `./oauth.js` pulls the OAuth implementation into the eager registration path while refresh still dynamically imports the runtime module. Move the resolver behind the runtime export or into a lightweight helper that does not import the full OAuth implementation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Missing legacy region metadata can submit CN refresh tokens to the Global MiniMax endpoint.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
  - body: When `resourceUrl` is absent, `refreshOAuth` defaults to Global. Legacy CN profiles from current releases can therefore submit their stored refresh token to `api.minimax.io` instead of `api.minimaxi.com`, which is both a functional failure and unnecessary token exposure.
  - confidence: 0.88
