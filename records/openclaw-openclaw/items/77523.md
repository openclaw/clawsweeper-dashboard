## Summary

Keep open: the PR is the right provider-owned direction and later commits addressed the raw-fetch/no-floating issues, but latest head still has a legacy CN token-routing bug plus an eager OAuth import that should be repaired before merge.

## What This Changes

The PR branch adds a MiniMax Portal OAuth refresh-token grant, stores `resourceUrl` on new OAuth login credentials, exports the refresh helper, adds Global/CN refresh tests, and updates the changelog/raw-fetch allowlist offsets.

## Best Possible Solution

Land MiniMax Portal refresh with legacy-safe region handling, no wrong-region token submission, preserved lazy runtime loading, and regression coverage for Global, CN, and missing-metadata profiles.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no `minimax-portal` `refreshOAuth` hook, and core refresh returns null when neither the plugin nor pi-ai can handle the provider. I did not run a live wait-for-expiry MiniMax flow.

## Solution Assessment

No, not yet. The PR uses the right provider-owned seam, but it defaults unknown legacy region metadata to Global and imports the OAuth implementation eagerly; the safer fix keeps routing legacy profiles correctly or fails safely before refresh.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing current-release MiniMax OAuth profiles do not store `resourceUrl`; the only CN/Global choice is the provider config written during login. This branch resolves missing metadata as Global, so upgraded CN profiles send refresh tokens to `api.minimax.io` and still fail. Use stored config via refresh context, migrate/fill metadata, or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.9
- **[P3] Keep the resolver behind the lazy OAuth boundary:** `extensions/minimax/provider-registration.ts:22`
  - body: Provider registration currently reaches OAuth through `oauth.runtime.ts`; the added value import from `./oauth.js` pulls the implementation into the eager registration path while refresh still dynamically imports it. Move the resolver behind the runtime export or into a light helper that does not import the OAuth implementation.
  - confidence: 0.78

## Security Review
