## Summary

This PR is a valid fix direction for a source-reproducible MiniMax Portal OAuth refresh bug, but it should not merge yet because the current diff can route legacy CN refresh tokens to the Global endpoint and also weakens the existing lazy OAuth boundary.

## What This Changes

The PR adds MiniMax Portal OAuth token refresh support, stores login-time resource URL metadata, adds Global/CN refresh tests, and updates the changelog plus raw-fetch allowlist offsets.

## Best Possible Solution

Land provider-owned MiniMax Portal refresh after preserving or safely rejecting unknown legacy regions, keeping OAuth implementation behind the runtime boundary, and covering Global, CN, and missing-metadata profiles.

## Reproduction Assessment

Yes. Source inspection shows current main has no `minimax-portal` `refreshOAuth` hook, core falls through to pi-ai, and pi-ai 0.73.0 has no MiniMax OAuth provider; I did not run a live expiry wait.

## Solution Assessment

No. The provider-owned refresh hook is the right narrow solution, but this PR should first avoid guessing Global for legacy credentials and restore the lazy OAuth import boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve legacy MiniMax OAuth regions:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing MiniMax OAuth credentials from current releases do not store `resourceUrl`; their CN/Global choice is only reflected in provider config. This line turns missing metadata into Global, so upgraded CN profiles send refresh tokens to `api.minimax.io` and still fail. Derive the legacy region from a reliable source, migrate/fill metadata, or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.9
- **[P3] Keep region resolution out of eager registration:** `extensions/minimax/provider-registration.ts:22`
  - body: Provider registration currently reaches OAuth code through `oauth.runtime.ts`. Adding a value import from `./oauth.js` pulls the OAuth implementation into the eager registration path while refresh still dynamically imports the runtime module. Move the resolver behind the runtime export or into a lightweight helper that does not import the full OAuth implementation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or permission change, but missing legacy region metadata can submit CN refresh tokens to the Global MiniMax endpoint.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
  - body: When `resourceUrl` is absent, `refreshOAuth` defaults to Global. Legacy CN profiles can therefore submit their stored refresh token to `api.minimax.io` instead of `api.minimaxi.com`, which is both a functional failure and unnecessary token exposure.
  - confidence: 0.87
