## Summary

Keep open: the PR targets a valid MiniMax Portal OAuth refresh bug, but the latest diff still defaults legacy credentials without `resourceUrl` to Global and statically imports `oauth.ts` from provider registration, so it needs a focused repair before merge.

## What This Changes

The PR adds MiniMax Portal OAuth refresh helpers, stores `resourceUrl` on new OAuth logins, registers `refreshOAuth`, extends MiniMax tests, adjusts raw-fetch lint allowlist line numbers, and adds a changelog entry.

## Best Possible Solution

Implement MiniMax Portal refresh while preserving the original CN/Global endpoint for both new and existing profiles, keep the OAuth implementation behind the lazy runtime boundary, and cover Global, CN, and legacy-metadata cases with regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main registers MiniMax Portal OAuth login but no `refreshOAuth` hook, and core refresh returns null when neither plugin nor shared pi-ai refreshers can handle the provider; I did not run the two-hour expiry flow in this read-only review.

## Solution Assessment

No. The PR is the right general direction, but it is not yet the best complete fix because missing legacy region metadata is treated as Global and the resolver is imported through the eager provider-registration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing credentials from current releases do not include `resourceUrl`; the CN/Global choice only exists in the saved provider config. This default sends legacy CN refresh tokens to `api.minimax.io`, so upgraded CN users still fail and unnecessarily expose tokens cross-region. Use the saved provider base URL, migrate metadata, or explicitly require reauth before refreshing.
  - confidence: 0.9
- **[P3] Keep the OAuth resolver behind the lazy boundary:** `extensions/minimax/provider-registration.ts:22`
  - body: Provider registration currently reaches MiniMax OAuth through `oauth.runtime.ts`; adding a static `./oauth.js` import loads the implementation eagerly and risks the ineffective-dynamic-import build guard. Move the resolver behind the runtime export or into a small pure helper that does not import the OAuth implementation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff still has a concrete cross-region token-submission risk for legacy CN profiles that lack `resourceUrl`.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
  - body: When `resourceUrl` is absent, `refreshOAuth` defaults to Global. Legacy CN profiles from current releases can therefore send their stored refresh token to `api.minimax.io` instead of the configured CN host, which is both a functional failure and unnecessary token exposure.
  - confidence: 0.86
