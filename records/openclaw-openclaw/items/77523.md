## Summary

Keep this PR open: current main has the reported MiniMax Portal refresh gap and `refreshOAuth` is the right provider-owned surface, but the patch still misses legacy CN credentials created before `resourceUrl` is saved and can refresh them against the Global endpoint.

## What This Changes

The PR adds MiniMax Portal OAuth refresh-token helpers, stores `resourceUrl` on new logins, registers `refreshOAuth` for `minimax-portal`, extends MiniMax tests, and adds a changelog entry.

## Best Possible Solution

Implement MiniMax Portal refresh so it preserves the originally selected CN/Global endpoint for both new and existing profiles, keeps the OAuth runtime boundary lazy, and covers Global, CN, and legacy-metadata cases with regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main has MiniMax Portal OAuth login but no `refreshOAuth`, and core refresh returns null when neither the plugin nor shared OAuth providers can refresh the expired credential. I did not run an end-to-end two-hour expiry test in this read-only review.

## Solution Assessment

No, this PR is not yet the best complete fix because missing `resourceUrl` is treated as Global, leaving upgraded CN users on the wrong refresh endpoint. The safer path is to use persisted provider config or a migration/explicit reauth path before making the refresh request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing credentials from current releases do not include `resourceUrl`; the CN/Global choice is only in the config patch written at login. This line turns missing metadata into `global`, so upgraded CN users refresh against `api.minimax.io` and still lose auth. Pass/use the saved provider base URL, or explicitly migrate/reauth legacy profiles before refreshing.
  - confidence: 0.9
- **[P3] Keep the OAuth runtime helper lazy:** `extensions/minimax/provider-registration.ts:22`
  - body: Adding a static import from `./oauth.js` makes provider registration load the same OAuth implementation that the login path imports through `oauth.runtime.js`, defeating the lazy boundary and risking the repo's ineffective-dynamic-import build check. Move the resolver behind `oauth.runtime.js` or a small pure helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch adds refresh-token POSTs and currently can choose the wrong regional endpoint for legacy profiles without `resourceUrl`.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
  - body: When `resourceUrl` is absent, `refreshOAuth` defaults to the Global MiniMax endpoint. Legacy CN profiles created by current releases can therefore send their stored refresh token to `api.minimax.io` instead of the configured CN host, which is both a functional failure and unnecessary cross-region token exposure.
