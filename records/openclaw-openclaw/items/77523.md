## Summary

Keep open: this PR targets a valid MiniMax Portal OAuth refresh bug, but the current diff still has a blocking legacy-region defect and lacks required real behavior proof; the prior review flagged the same core blockers, and later comments only address raw-fetch/lint cleanup.

## What This Changes

The PR adds a MiniMax Portal refresh-token grant, stores `resourceUrl` on new OAuth login credentials, wires a `minimax-portal` `refreshOAuth` hook, adds provider tests, updates the changelog, and adjusts raw-fetch allowlist line offsets.

## Best Possible Solution

Land a provider-owned MiniMax Portal refresh hook only after missing legacy region metadata is preserved, derived, or rejected safely, and after real after-fix refresh proof is attached.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main has no `minimax-portal` `refreshOAuth` hook, and core then falls through to pi-ai OAuth providers that omit MiniMax; I did not run a live wait-for-expiry flow.

## Solution Assessment

No. The provider-owned refresh hook is the right direction, but this diff is not yet the safest fix because it defaults missing legacy metadata to Global and pulls OAuth implementation into the eager registration path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing MiniMax OAuth profiles from current releases do not store `resourceUrl`; their CN/Global choice lives in provider config. This line turns missing metadata into Global, so upgraded CN profiles send refresh tokens to `api.minimax.io` and still fail. Derive the legacy region from a reliable source, migrate/fill metadata, or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.91
- **[P3] Keep the resolver behind the lazy OAuth boundary:** `extensions/minimax/provider-registration.ts:20`
  - body: Provider registration currently reaches OAuth code through `oauth.runtime.ts`. Adding a value import from `./oauth.js` pulls the OAuth implementation into the eager registration path while refresh still dynamically imports the runtime module. Move the resolver behind the runtime export or into a lightweight helper that does not import the full OAuth implementation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependencies, but missing region metadata can submit legacy CN refresh tokens to the Global MiniMax endpoint.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
