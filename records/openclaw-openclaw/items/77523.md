## Summary

Keep open: current main still needs MiniMax Portal OAuth refresh, and this PR uses the right provider-owned hook, but latest head still has a legacy CN-region safety bug and a lazy-boundary regression.

## What This Changes

The PR adds MiniMax Portal OAuth refresh-token support, stores a resource URL on new OAuth credentials, exports refresh helpers, adds provider tests, and updates the changelog/raw-fetch allowlist offsets.

## Best Possible Solution

Land MiniMax Portal refresh after it preserves region for existing profiles or fails safely, keeps OAuth code behind the lazy runtime boundary, and covers Global, CN, and missing-metadata refresh.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main has no `minimax-portal` `refreshOAuth` hook, and core refresh returns null when neither the plugin nor pi-ai can handle the provider. I did not run a live wait-for-expiry MiniMax flow.

## Solution Assessment

No, not yet. The PR uses the right provider-owned seam, but it defaults unknown legacy region metadata to Global and imports the OAuth implementation eagerly; the safer fix keeps routing legacy profiles correctly or fails safely before refresh.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve the legacy MiniMax OAuth region:** `extensions/minimax/provider-registration.ts:289`
  - body: Existing MiniMax OAuth profiles from current releases do not store `resourceUrl`; the only CN/Global signal is the provider config written at login. This branch resolves missing metadata as Global, so upgraded CN profiles send refresh tokens to `api.minimax.io` and still fail. Use stored config via refresh context, migrate/fill metadata, or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.9
- **[P3] Keep the resolver behind the OAuth runtime boundary:** `extensions/minimax/provider-registration.ts:22`
  - body: Provider registration currently reaches OAuth through `oauth.runtime.ts`; the added value import from `./oauth.js` pulls the OAuth implementation into the eager registration path while refresh still dynamically imports it. Move the resolver behind the runtime export or into a light helper that does not import the OAuth implementation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Needs attention because legacy credentials without `resourceUrl` can send CN refresh tokens to the Global MiniMax endpoint; the guarded fetch and dependency surface otherwise look narrow.

Concerns:

- **[medium] Avoid cross-region refresh-token submission:** `extensions/minimax/provider-registration.ts:289`
  - body: When `resourceUrl` is absent, `refreshOAuth` defaults to Global. Legacy CN profiles from current releases can therefore submit their stored refresh token to `api.minimax.io` instead of `api.minimaxi.com`, which is both a functional failure and unnecessary token exposure.
  - confidence: 0.87
