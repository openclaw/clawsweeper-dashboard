## Summary

Keep open: current main and the latest release still have the dry-run formatter bug, and this PR is a correct focused fix with sufficient terminal proof. The remaining maintainer decision is choosing this proof-positive branch versus the older open branch that includes focused formatter tests.

## What This Changes

The PR changes `formatMessageCliText` to render dry-run output when `result.dryRun` is true instead of relying on `result.handledBy === "dry-run"`.

## Best Possible Solution

Land one narrow formatter fix that checks `dryRun`, pair it with focused send/poll formatter regression coverage and real CLI proof, then close or supersede the other fix candidate.

## Reproduction Assessment

Yes. I did not execute the CLI in this read-only review, but current-main source plus the canonical issue's captured CLI output show send and poll dry-runs carry `dryRun: true` while the formatter still checks `handledBy === "dry-run"`.

## Solution Assessment

Yes. Checking `result.dryRun` in the formatter is the narrowest maintainable fix because delivery already short-circuits dry-run and the runner contract already carries `dryRun` separately; changing outbound `handledBy` semantics would broaden the contract unnecessarily.
