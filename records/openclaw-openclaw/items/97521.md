## Summary

Keep open: current main and the latest stable release still have the unbounded bodyless MCP fetch fallback, while the related bounded-read PR was closed after review showed it would not safely solve this bodyless compatibility path.

## What This Changes

The issue asks maintainers to choose an explicit policy for bodyless MCP SDK fetch responses that expose `text()` without a readable body.

## Best Possible Solution

Maintainers should choose the bodyless-response policy first, then land one focused MCP fix with compatibility coverage and real loopback MCP/HTTP proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a bodyless non-204/205/304 MCP response with `text()` reaches the unbounded `response.text()` call. I did not execute a runtime repro because this review was read-only.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing `readProviderTextResponse` is not the best fix for this bodyless path because it depends on `arrayBuffer()` and checks the cap after allocation.
