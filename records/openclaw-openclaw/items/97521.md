## Summary

Keep open: current main and v2026.6.10 still have the unbounded bodyless MCP fetch `text()` fallback, and the previous bounded-reader PR was closed after review showed that path needs an explicit compatibility/security policy decision.

## What This Changes

The issue requests a maintainer decision for how MCP HTTP fetch should handle bodyless foreign responses: preserve text-only compatibility with a cap, fail closed, or defer.

## Best Possible Solution

Keep this as the canonical policy tracker, then land one focused MCP transport fix with explicit compatibility/security behavior, regression coverage, and real loopback MCP/HTTP proof.

## Reproduction Assessment

Yes. Source inspection shows a bodyless non-204/205/304 MCP response with `text()` reaches the unbounded `response.text()` call, and the existing compatibility test exercises that text-only foreign-response shape.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing `readProviderTextResponse` is not the best fix for this bodyless path because it depends on `arrayBuffer()` after `body` is null and changes the text-only contract.
