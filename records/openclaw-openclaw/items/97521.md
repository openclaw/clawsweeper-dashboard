## Summary

Keep open: current main and the latest release still have the unbounded bodyless MCP fetch fallback, and the related bounded-read PR was closed after review showed that a shared-reader swap would not safely solve this compatibility path.

## What This Changes

Requests a maintainer policy decision for bodyless foreign MCP HTTP responses: preserve text-only compatibility with a cap, fail closed, or defer broader transport hardening.

## Best Possible Solution

Maintainers should choose the bodyless-response policy first, then land one focused MCP transport fix with compatibility coverage and real loopback MCP/HTTP proof.

## Reproduction Assessment

Yes. Source inspection shows a bodyless non-204/205/304 MCP response with text() reaches the unbounded response.text() call, and the existing compatibility test exercises the bodyless text-only shape.

## Solution Assessment

Unclear until maintainers choose the policy. Reusing readProviderTextResponse is not the best fix for this bodyless path because it depends on arrayBuffer() after body is null and changes the text-only contract.
