## Summary

Keep open. The report is plausible on current `main`: Telegram final delivery has durable queue coverage for a single outbound send, but restart-continuation recovery can still replay the whole queued agent turn after a successful send if the session-delivery entry is not acked, and there is no persisted final-answer idempotency tying that replay to the accepted Telegram message.

## What This Changes

Telegram can receive duplicate assistant replies after gateway restart and auto-compaction retry even when the session history contains one final assistant response.

## Best Possible Solution

Carry a deterministic turn/final-delivery id through restart-continuation and Telegram final delivery, persist the accepted receipt or terminal send state, and skip replayed finals while preserving normal chunk and media behavior.

## Reproduction Assessment

No: we do not have a high-confidence live reproduction on current `main`. Source inspection does show a credible replay path around queued restart continuations and final Telegram delivery, but the exact runtime trigger still needs a focused regression test or live proof.

## Solution Assessment

No current patch is proposed. The requested session/response/final-answer idempotency direction is appropriate, but it should be implemented through the existing durable delivery and session-delivery queues rather than a Telegram content-only dedupe cache.
