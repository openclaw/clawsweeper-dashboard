## Summary

Keeping this open: current main still has the source-level fan-out path, and two open PRs already reference this issue with closing syntax, so the issue should stay open until a fix merges.

## What This Changes

The issue requests that Telegram direct replies send only the final user-facing assistant reply once when multiple assistant text blocks accumulate around context-engine maintenance.

## Best Possible Solution

Review and land a focused fix from https://github.com/openclaw/openclaw/pull/79665 or https://github.com/openclaw/openclaw/pull/79761 that collapses non-block final text without breaking explicit media, chunking, or reply metadata, then close this through the merge.

## Reproduction Assessment

Yes, at source level: passing multiple non-empty `assistantTexts` through current main produces multiple reply payloads, and the dispatcher sends each final payload separately. I did not establish a live Telegram reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because it treats accumulated assistant text blocks as independent final replies. The maintainable fix belongs at the final payload selection/delivery boundary while preserving explicit streaming, media, and reply-tag behavior.
