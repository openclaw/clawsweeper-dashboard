## Summary

Keeping this open: current main still has a source-level path where multiple accumulated `assistantTexts` become multiple final reply payloads, and the Telegram/default preview path does not prove those are collapsed before final delivery.

## What This Changes

The issue reports duplicate Telegram direct-chat replies when a single assistant turn accumulates multiple visible assistant text blocks after context-engine maintenance.

## Best Possible Solution

Constrain non-block final delivery to the final user-facing assistant answer while keeping explicit streaming/chunking/media paths multi-part, and add a regression that combines accumulated assistant texts with context-engine maintenance status.

## Reproduction Assessment

Yes for a source-level reproduction, but not from a live Telegram run in this read-only review. Passing multiple non-empty `assistantTexts` into the current final-delivery path can produce multiple reply payloads, and the dispatcher sends each payload separately.

## Solution Assessment

No, the current source path is not the best solution because it treats every accumulated assistant text as an independent final reply. The narrow maintainable fix is to select or derive the final user-facing answer for non-block final delivery and leave explicit block streaming/chunking behavior alone.
