## Summary

Keep open: the fix is aimed at the right heartbeat/Telegram path, but the new marker parser misses `notify=false` when the model output has a normal final newline, and the PR still only provides synthetic test-harness proof rather than live Telegram behavior proof.

## What This Changes

The PR strips a final standalone heartbeat fallback `notify=false` marker, forwards `silent` through durable delivery and the Telegram test shim, adds regression coverage, and updates the changelog.

## Best Possible Solution

Keep the narrow heartbeat fallback normalizer, but make it tolerant of final line terminators, add the missing newline regression case, and attach live or redacted Telegram proof that the marker is hidden and the send is silent.

## Reproduction Assessment

Yes for the patch defect: the new helper regex can be exercised directly and leaves `notify=false` intact when the marker is followed by a final newline. The original Telegram symptom still needs live Telegram confirmation after the code fix.

## Solution Assessment

No, not yet: the approach is the right narrow path, but the current regex is too strict about the marker being the final bytes. Allow final LF/CRLF or trim terminal whitespace before matching, then prove the Telegram delivery path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle final newlines after the notify marker:** `src/infra/heartbeat-runner.ts:748`
  - body: The new normalizer only matches `notify=false` when it is the last byte or followed by spaces/tabs. If the model output has a normal trailing newline, such as `No interruption needed.\nnotify=false\n`, the replacement does not run, so the marker is still delivered and marker-only `notify=false\n` is not suppressed. Please allow final line terminators or trim terminal whitespace before matching, and add a regression case for this form.
  - confidence: 0.92
