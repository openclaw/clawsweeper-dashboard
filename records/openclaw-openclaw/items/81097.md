## Summary

Keep open: the PR targets a real heartbeat/Telegram bug and uses the existing silent-delivery path, but the patch currently treats ordinary trailing whitespace as a silent marker and only provides synthetic harness proof.

## What This Changes

The PR strips a final standalone `notify=false` from heartbeat fallback replies, propagates `silent` into durable Telegram delivery, updates heartbeat test-shim coverage, and adds a changelog entry.

## Best Possible Solution

Keep the fix in heartbeat fallback normalization, but only set `silent` when the marker regex actually removes `notify=false`, then require redacted real Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main passes heartbeat fallback text through token stripping and durable delivery without parsing a trailing `notify=false` marker or setting `silent`. I did not run tests because this review is read-only.

## Solution Assessment

No as written: the owner boundary is right, but the helper should compare marker removal before trimming so ordinary trailing whitespace does not request silent delivery. A real Telegram proof artifact should cover the final behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Set silent only when the marker was removed:** `src/infra/heartbeat-runner.ts:747-748`
  - body: The helper trims the string before checking whether the regex matched. A normal reply ending in whitespace, for example `"Status done\n"`, becomes `"Status done"`, so `stripped !== text` and the code returns `silent: true` even though there was no `notify=false` marker. Compare the replaced value before `trimEnd()` or move trailing whitespace into the marker-matched branch.
