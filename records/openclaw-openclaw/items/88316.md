## Summary

Keep open: the PR is useful, but the patch derives `activeChannel` from stored `deliveryContext`, which current main intentionally preserves as the external route when webchat views an existing channel session, so it can still report the origin route for the reported path.

## What This Changes

The branch adds origin/active route metadata to the `session_status` tool result and adds one regression test for a Discord-origin, webchat-active fixture.

## Best Possible Solution

Expose origin from the session key or stored origin and active route from the live run context, with regression coverage where the stored delivery route remains Discord while the current run is webchat.

## Reproduction Assessment

Yes, at source level: current main preserves the external stored `deliveryContext` when webchat views an existing channel session, while the PR reads `activeChannel` from that stored context. I did not run a live Discord-to-webchat E2E in this read-only review.

## Solution Assessment

No. The additive field direction is right, but this implementation must derive the active route from live run context rather than persisted delivery context to solve the reported issue.
