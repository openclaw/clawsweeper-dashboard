## Summary

## What This Changes

The PR adds `originChannel`, `activeChannel`, and nested origin/active delivery details to `session_status`, plus a regression test for a Discord-origin, webchat-active fixture.

## Best Possible Solution

Expose origin from stored origin/session-key data and active route from live tool-run context, keep persisted delivery metadata separate, and cover the preserved-external-route plus webchat-active case before merge.

## Reproduction Assessment

Yes. Source and existing tests show current main can preserve an external stored delivery route while webchat is the active inbound surface, and this PR reads `activeChannel` from that persisted route.

## Solution Assessment

No. The additive field direction is right, but active route details need to come from live run/tool context rather than persisted delivery context.
