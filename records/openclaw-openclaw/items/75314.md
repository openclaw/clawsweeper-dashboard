## Summary

Keep open. Current main fixes part of the toolbar New Chat path by creating a new session directly, but the typed WebChat `/new` and `/reset` path still goes through `chat.send`, the reset handler still returns only a no-model acknowledgement, and the UI still lacks reset-divider projection for `gateway-injected` assistant-shaped transcript messages. Draft PR #74733 targets this WebChat reset/display contract but is still open, so the issue is not implemented on main yet.

## What This Changes

The issue asks WebChat fresh-session reset acknowledgements to either trigger a real startup greeting or render as clearly non-assistant session-control events instead of ordinary `gateway-injected` assistant bubbles.

## Best Possible Solution

Land PR #74733 or an equivalent narrow patch that keeps bare reset as a no-model control operation unless maintainers explicitly change that policy, but renders reset acknowledgements as session lifecycle events/dividers rather than ordinary assistant turns and aligns the reset prompt/UI copy with that behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence for typed WebChat `/new` or `/reset`: current UI sends the command through `chat.send`, the reset handler returns a synthetic acknowledgement without starting an agent run, and the no-agent final path can persist the reply as a `gateway-injected` assistant message.

## Solution Assessment

Yes, the best current fix direction is to separate reset acknowledgements from assistant output instead of forcing an automatic model greeting by default. That matches existing no-model reset tests while addressing the misleading WebChat presentation; a product decision can still choose a real greeting later by updating the reset policy and tests together.
