## Summary

Keep open. Current main still routes typed WebChat `/new` and `/reset` through `chat.send`, bare resets still stop after the synthetic acknowledgement, and the UI does not yet convert `gateway-injected` reset acknowledgements into lifecycle events; draft PR #74733 targets the same area but remains open and unmerged.

## What This Changes

The issue asks WebChat fresh-session reset acknowledgements to stop looking like the assistant's first real conversational turn, either by running the startup greeting or by rendering the acknowledgement as a distinct lifecycle event.

## Best Possible Solution

Ship a narrow WebChat fix that keeps documented no-model bare resets but renders reset acknowledgements as lifecycle dividers or system events, with regression coverage for typed `/new` and `/reset`.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: typed WebChat `/new` uses `chat.send`, the reset handler returns the synthetic acknowledgement without model execution, and current tests assert no model call for bare `/new`.

## Solution Assessment

Yes for the planned direction, not for current main. Separating lifecycle acknowledgements from assistant output is the narrowest maintainable fix because current docs intentionally define bare chat resets as no-model operations.
