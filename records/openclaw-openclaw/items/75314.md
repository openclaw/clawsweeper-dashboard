## Summary

Keep open. Current main still leaves the typed WebChat `/new` and `/reset` path using `chat.send`, where bare reset returns only the synthetic acknowledgement and no model run; the UI does not yet project `gateway-injected` reset acknowledgements as lifecycle events. Draft PR #74733 targets the same WebChat reset/display contract, but it is open and unmerged.

## What This Changes

The issue requests that WebChat fresh-session reset acknowledgements either trigger a real startup greeting or render clearly as session-control events instead of assistant replies.

## Best Possible Solution

Land PR #74733 or an equivalent narrow WebChat patch that keeps bare reset as a no-model control operation while rendering reset acknowledgements as session lifecycle events instead of ordinary assistant turns.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: typed WebChat `/new` dispatches through `chat.send`, the reset handler returns the synthetic acknowledgement without starting a model run, and current tests assert that no model is invoked for bare `/new`.

## Solution Assessment

Yes for the planned direction, not for current main. Separating lifecycle acknowledgements from assistant output is narrower and more consistent with current docs/tests than forcing every bare reset to run the model.
