## Summary

Keep open. Current main still routes WebChat fresh-session actions through `chat.send`, the bare reset command handler still returns only the synthetic acknowledgement without starting an agent run, and the resulting reply can still be persisted/broadcast as an assistant-shaped `gateway-injected` message. Open PR #74733 appears to address much of the same WebChat reset-acknowledgement contract, but it is unmerged, broad, and not linked with closing syntax, so this issue should remain open until that work lands or a narrower fix is merged.

## What This Changes

The issue requests that WebChat fresh `/new` or reset sessions stop showing only a gateway-injected assistant-looking acknowledgement, either by producing a real startup greeting or by rendering the acknowledgement as a distinct session-control event.

## Best Possible Solution

Preserve the current no-model behavior for bare `/new` and `/reset` unless maintainers intentionally change that policy, but make WebChat represent exact reset acknowledgements as session-control dividers/events rather than normal assistant turns. The most efficient path is to review and land the relevant parts of #74733, or split out a narrow PR that routes bare WebChat reset actions through the reset/session event path and adds focused history/rendering tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger Control UI New Session or send bare `/new`, observe the request goes through `chat.send`, the reset handler returns only `✅ New session started.` with `shouldContinue: false`, and the WebChat final/history path can surface that as a `gateway-injected` assistant entry without any model run.

## Solution Assessment

Yes for a narrow repair. Classifying/rendering the reset acknowledgement separately in WebChat and history is the safest maintainable fix; forcing an automatic greeting should be handled separately because current tests intentionally assert no model invocation for bare reset commands.
