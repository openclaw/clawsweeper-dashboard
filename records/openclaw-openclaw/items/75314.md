## Summary

Keep open. Current main still has the reported WebChat slash-reset gap: typed `/new` and `/reset` are sent through `chat.send`, bare reset returns only the synthetic acknowledgement without a model run, and the chat item builder does not yet project `gateway-injected` reset acknowledgements as lifecycle dividers. Related PR #74733 is open and draft, so this is not implemented on main yet.

## What This Changes

Request to make fresh WebChat reset/new-session UX produce a real startup greeting or render the synthetic reset acknowledgement as a non-assistant lifecycle event.

## Best Possible Solution

Land PR #74733 or an equivalent narrow WebChat patch that keeps bare reset semantics consistent with docs while rendering reset acknowledgements as session lifecycle events instead of ordinary assistant turns.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: typed WebChat `/new` is preserved as a `chat.send` command, the bare reset handler returns the synthetic acknowledgement with `shouldContinue: false`, and tests assert no model invocation for bare `/new`.

## Solution Assessment

Yes for the linked planned direction, not for current main. Separating lifecycle acknowledgements from assistant output is narrower and more consistent with current docs/tests than forcing every bare reset to run the model.
