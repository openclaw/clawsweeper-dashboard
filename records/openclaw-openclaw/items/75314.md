## Summary

Keep open. Current main still has a source-level path where bare WebChat `/reset` stops after a gateway-injected acknowledgement that WebChat renders as an ordinary assistant message, while typed `/new` now creates a session without starting a greeting run and a new comment reports the same no-greeting symptom on Signal.

## What This Changes

WebChat fresh-session reset/new flow should not leave users with only a gateway-injected acknowledgement that looks like an assistant turn instead of a real startup greeting or clear system event.

## Best Possible Solution

Settle the reset/new contract once, then either keep bare resets no-model and render gateway acknowledgements as lifecycle/system events across affected surfaces, or intentionally run a startup turn where docs and tests say one should run.

## Reproduction Assessment

Yes, source-level. Current main shows typed WebChat `/reset` still dispatches through `chat.send`, the shared reset handler returns before a model run for no-tail resets, and WebChat renders non-compaction transcript entries as ordinary messages.

## Solution Assessment

Unclear for the requested automatic greeting because current docs and tests intentionally preserve no-model bare chat resets. The narrower maintainable path is to render gateway acknowledgements as lifecycle/system events unless maintainers choose to restore automatic startup turns.
