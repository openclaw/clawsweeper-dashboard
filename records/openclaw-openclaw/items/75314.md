## Summary

Keep open. Current main still sends Control UI/WebChat fresh-session actions through `chat.send`, the bare reset handler still returns only the synthetic acknowledgement without starting an agent run, and that reply can still be persisted/broadcast as an assistant-shaped `gateway-injected` message. Related #75235 overlaps on reset readiness, but this issue adds a distinct WebChat display/transcript contract problem.

## What This Changes

The issue asks WebChat to avoid showing a synthetic `✅ New session started.` acknowledgement as the only assistant-looking startup response when no real agent greeting has run.

## Best Possible Solution

Preserve the documented no-model behavior for bare `/new` and `/reset` unless maintainers explicitly change product policy, but make WebChat represent exact reset acknowledgements as session-control/system events rather than normal assistant turns. If the prompt wording remains confusing, align the reset prompt/docs so paths that do not invoke the model no longer imply an immediate greeting.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger Control UI New Session or send `/new`, observe it goes through `chat.send`, the reset handler returns `✅ New session started.` with no agent run, and the WebChat final/history path can surface that as an assistant-shaped `gateway-injected` entry.

## Solution Assessment

Yes for a narrow repair. The safest implementation is to classify/render the reset acknowledgement separately in WebChat and history while preserving the parent-chain append behavior and the current no-model reset contract; forcing a greeting should be a separate maintainer product decision.
