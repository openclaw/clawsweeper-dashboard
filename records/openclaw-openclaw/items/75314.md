## Summary

Keep open. Current main still has the reported WebChat/session-control mismatch: Control UI sends fresh-session actions through `chat.send`, bare `/new` returns only the synthetic acknowledgement without starting a model run, and the Gateway can persist/broadcast that reply as an assistant-shaped `gateway-injected` transcript entry. Related #75235 overlaps on post-reset readiness, but this issue has a distinct WebChat display/transcript classification bug.

## What This Changes

The issue reports that WebChat fresh-session/reset actions show a synthetic `✅ New session started.` assistant-shaped bubble without triggering or clearly distinguishing a real agent startup greeting.

## Best Possible Solution

Preserve the documented no-model behavior for bare `/new` and `/reset`, but classify exact reset acknowledgements in WebChat/history as session-control or system events rather than normal assistant turns. If maintainers want an automatic startup greeting, that should be a separate product decision with docs and tests updated together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger Control UI New Session or send bare `/new`, observe it enters `chat.send`, returns `✅ New session started.` with no agent run, and the no-agent finalization path can append/broadcast that reply as an assistant-shaped `gateway-injected` message.

## Solution Assessment

Yes for a narrow repair. The maintainable fix is to separate reset acknowledgements from assistant transcript/rendering semantics while keeping parent-chain transcript integrity; forcing model invocation is broader and conflicts with current tested behavior.
