## Summary

Keep open. Current main still has the reported source-level behavior, and the older open item carries a WebChat-specific UI/transcript concern that is not fully superseded by the broader bare reset greeting issue.

## What This Changes

The issue requests that fresh WebChat reset/new sessions not present a synthetic assistant-looking acknowledgement as the only visible startup response.

## Best Possible Solution

Maintainers should decide the reset/new contract once, then either restore/configure a startup model turn or keep the no-model acknowledgement and render `gateway-injected` reset/new acknowledgements as non-assistant lifecycle events in WebChat and affected transports.

## Reproduction Assessment

Yes, source-level. Current main still has a bare reset/new path that returns a synthetic acknowledgement before model execution, stores gateway-injected acknowledgements as assistant messages, and renders non-compaction history entries as ordinary WebChat messages.

## Solution Assessment

Unclear. Removing the short-circuit may restore greetings for some users but conflicts with current docs/tests that encode no-model acknowledgement; lifecycle-event rendering is narrower if maintainers intend bare resets to stay no-model.
