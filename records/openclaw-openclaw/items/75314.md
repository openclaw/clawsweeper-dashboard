## Summary

Keep open. Current main still matches the reported behavior: WebChat reset actions travel through the normal chat send path, bare reset handling returns only the synthetic acknowledgement without a model run, and the UI/history filters do not classify that acknowledgement as a separate system event. Related #75235 overlaps on reset readiness, but this issue has a distinct WebChat display/transcript contract problem.

## What This Changes

The issue requests that fresh WebChat sessions stop showing only a synthetic `✅ New session started.` assistant-looking bubble without either running the startup greeting or rendering the acknowledgement as a clear system/session-control event.

## Best Possible Solution

Keep the existing no-model behavior for bare `/new` and `/reset` unless maintainers explicitly change product policy, but make WebChat represent exact reset acknowledgements as session-control/system events rather than normal assistant turns. If the startup prompt wording remains misleading for paths that do not invoke the model, align that prompt or docs in the same focused repair.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger the Control UI New Session button or send `/new`; the UI sends `chat.send`, the reset handler returns the acknowledgement with `shouldContinue: false`, and the non-agent final reply can be appended and rendered as an assistant-shaped `gateway-injected` message.

## Solution Assessment

Yes for a narrow repair. The maintainable fix is to classify or render reset acknowledgements separately in WebChat/history while preserving the tested no-model reset contract; forcing a model greeting should be a separate product decision.
