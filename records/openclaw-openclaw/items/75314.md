## Summary

Keep open. Current main still lets typed WebChat `/new` and `/reset` go through `chat.send`, where bare reset returns only the synthetic acknowledgement and stops before a model run; the UI still lacks reset-specific projection for `gateway-injected` acknowledgement messages. Draft PR #74733 targets the same WebChat reset/display contract but is open and unmerged.

## What This Changes

Fresh WebChat `/new` or `/reset` should not leave users with only a gateway-injected assistant-looking acknowledgement without either a real startup greeting or a clear lifecycle/system distinction.

## Best Possible Solution

Land PR #74733 or an equivalent narrow WebChat patch that keeps bare reset semantics aligned with docs while rendering reset acknowledgements as lifecycle/session events rather than ordinary assistant turns.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: typed WebChat `/new` dispatches through `chat.send`, the reset command handler returns the synthetic acknowledgement with `shouldContinue: false`, and current tests assert no model invocation for bare `/new`.

## Solution Assessment

Yes for the planned direction, but it is not implemented on main. Since current docs define bare reset as no-model, separating lifecycle acknowledgements from assistant output is narrower than forcing every bare reset to run the model.
