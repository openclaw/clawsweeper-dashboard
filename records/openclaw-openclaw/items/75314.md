## Summary

Keep open. Current `main` still has the reported behavior: WebChat New Session uses `/new` through `chat.send`, bare reset handling stops after the synthetic acknowledgement, and that acknowledgement can still be persisted/broadcast as an assistant-shaped `gateway-injected` message. Open PR #74733 appears to target this exact WebChat display/reset contract, but it is not merged yet.

## What This Changes

The issue asks WebChat fresh-session/reset acknowledgements to stop appearing as the only assistant-looking reply, either by producing a real startup greeting or by rendering the control acknowledgement separately from assistant output.

## Best Possible Solution

Keep bare `/new` and `/reset` as no-model control operations unless maintainers explicitly change that policy, but WebChat should route or render those acknowledgements as session lifecycle events/dividers rather than ordinary assistant turns. If #74733 lands with that behavior and adequate tests, this issue can close as implemented.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: trigger the Control UI New Session action, which sends `/new` via `chat.send`; the reset handler returns only the synthetic acknowledgement without an agent run; the WebChat final path can append and broadcast it as a `gateway-injected` assistant message.

## Solution Assessment

Yes for the narrow UI/contract repair, not for forcing a model greeting by default. The maintainable fix is to keep reset acknowledgement separate from assistant output and preserve the tested no-model reset policy unless product owners decide otherwise.
