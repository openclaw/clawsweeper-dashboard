## Summary

Keep open. Current main still has a source-level WebChat `/reset` path where the gateway acknowledgement is persisted as a `gateway-injected` assistant message and the UI has no reset/new lifecycle divider, while draft PR #74733 is open and targets this same display/dispatch contract.

## What This Changes

WebChat fresh-session/reset flow should not leave users with only a synthetic gateway-injected acknowledgement that looks like the assistant's first conversational turn.

## Best Possible Solution

Review and land PR #74733 or a narrower equivalent that keeps bare chat resets as no-model acknowledgements while rendering reset/new acknowledgements as lifecycle events and refreshing the active WebChat history.

## Reproduction Assessment

Yes, source-level. Typed WebChat `/reset` dispatches through `chat.send`, the reset handler returns a synthetic no-model acknowledgement, and WebChat currently renders non-compaction transcript entries as messages.

## Solution Assessment

Yes for the planned direction, not current main. Given the documented no-model bare reset contract, separating gateway acknowledgements from assistant output is narrower than forcing every bare reset to run a model greeting.
