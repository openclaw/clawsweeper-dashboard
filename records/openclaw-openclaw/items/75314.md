## Summary

Keep open. Current main still has a high-confidence source-level reproduction for typed WebChat `/reset`: the UI sends it through `chat.send`, the command path returns only a synthetic no-model acknowledgement, and history projection still renders the `gateway-injected` assistant-shaped entry as a normal message; draft PR #74733 targets this but is unmerged.

## What This Changes

Make WebChat fresh-session/reset acknowledgements either produce a real startup greeting or appear as non-assistant lifecycle events rather than assistant-shaped `gateway-injected` bubbles.

## Best Possible Solution

Land PR #74733 or a narrower equivalent that preserves the documented no-model bare reset contract while rendering gateway-injected reset/new acknowledgements as lifecycle/system dividers.

## Reproduction Assessment

Yes. Typed WebChat `/reset` dispatches through `chat.send`, the reset handler returns `shouldContinue: false` with a synthetic acknowledgement, and `buildChatItems` pushes that assistant-shaped transcript entry as a message because only compaction markers become dividers.

## Solution Assessment

Yes for the planned direction, not current main. Since docs now explicitly state bare chat `/new` and `/reset` do not invoke the model, separating the acknowledgement from assistant output is narrower than forcing a startup greeting on every reset.
