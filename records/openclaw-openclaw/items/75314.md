## Summary

Keep open. Current main partially moved the toolbar New Session flow to `sessions.create`, but the WebChat typed `/new` and `/reset` path still goes through `chat.send`; the bare reset handler still returns only the synthetic acknowledgement without starting a model run, and the current UI renderer has no special reset-divider handling for `gateway-injected` assistant-shaped transcript messages. Open PR #74733 appears to target this same WebChat reset/display contract, but it is still open and draft, so this issue is not implemented on main yet.

## What This Changes

The issue requests that fresh WebChat reset/new-session UX either trigger a real startup greeting or render the synthetic reset acknowledgement as a clearly non-assistant system/session event.

## Best Possible Solution

Review and land PR #74733, or an equivalent narrow patch, that keeps bare reset as a no-model control operation unless product direction changes, but projects reset acknowledgements as session lifecycle events/dividers rather than ordinary assistant turns in WebChat.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence for typed WebChat `/new`: the UI sends it through `chat.send`, the reset handler returns a synthetic acknowledgement without invoking the model, and the no-agent final path can persist/broadcast it as a `gateway-injected` assistant message.

## Solution Assessment

Yes, the best direction is to separate reset acknowledgements from assistant output rather than forcing an automatic model greeting by default. That preserves the current tested no-model reset policy while fixing the misleading WebChat presentation.
