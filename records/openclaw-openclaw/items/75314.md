## Summary

Keep open. Current main has moved WebChat `/new` away from the old `chat.send` acknowledgement path, but typed WebChat `/reset` still dispatches through `chat.send`, produces a no-model gateway acknowledgement, and WebChat still renders that injected assistant-shaped transcript row as an ordinary assistant group; draft PR #74733 remains the active related implementation candidate.

## What This Changes

Requests that WebChat fresh/new/reset sessions not show a `gateway-injected` assistant bubble as the only first visible response without either a real startup greeting or a system/lifecycle distinction.

## Best Possible Solution

Keep the no-model bare reset contract, but project gateway-injected reset/new acknowledgements in WebChat as lifecycle/system dividers and refresh the active history after reset/session rotation.

## Reproduction Assessment

Yes, source-level: type `/reset` in WebChat and the current code routes it through `chat.send`, returns a no-model reset acknowledgement, persists it as `gateway-injected`, and renders it as a normal assistant group. I did not need a live browser run because the dispatch, handler, transcript, projection, and tests line up directly.

## Solution Assessment

Yes for the related PR direction, not for current main. Rendering injected reset acknowledgements as lifecycle/system UI is narrower than forcing every bare reset to run a model greeting, and it preserves the existing no-model reset tests.
