## Summary

Keep open. Current main still has a source-level WebChat `/reset` path that stops after a gateway-injected acknowledgement instead of producing a real first conversational turn, and the related draft PR is still open and unmerged.

## What This Changes

The issue asks WebChat fresh/reset sessions to avoid showing a gateway-injected assistant-looking acknowledgement as the only first turn, either by running the startup greeting or rendering the acknowledgement as a system/lifecycle event.

## Best Possible Solution

Land PR #74733 or a narrower equivalent that keeps the reset contract explicit while ensuring WebChat does not render gateway-injected reset acknowledgements as ordinary assistant replies.

## Reproduction Assessment

Yes. A high-confidence source reproduction remains for typed WebChat `/reset`: it dispatches through `chat.send`, the reset command returns `shouldContinue: false`, and the resulting acknowledgement is persisted as a gateway-injected assistant-shaped transcript entry.

## Solution Assessment

Yes for the planned direction, not for current main. Given current docs describe bare reset as no-model, the narrowest fix is to render/reset-audit the acknowledgement as a lifecycle/system event or otherwise route WebChat reset through a dedicated UI/session path instead of an assistant bubble.
