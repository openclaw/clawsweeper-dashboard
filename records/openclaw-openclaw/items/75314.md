## Summary

Keep open. Current main still has the reported source-level no-startup-run behavior, and this older issue remains the WebChat/Control UI-specific canonical thread rather than being fully covered by the broader bare-reset greeting issue.

## What This Changes

Fresh WebChat/Control UI sessions should not leave users with only a synthetic `gateway-injected` reset acknowledgement that looks like the assistant's first conversational turn.

## Best Possible Solution

Keep this as the WebChat-specific canonical thread, decide the reset/new contract once, then align code, docs, tests, and UI rendering so synthetic reset acknowledgements cannot be mistaken for real assistant greetings.

## Reproduction Assessment

Yes, source-level. Current main short-circuits bare reset/new before model execution, stores non-agent acknowledgements as assistant-shaped `gateway-injected` transcript messages, and Control UI session creation only starts a run when a `task` or `message` is provided.

## Solution Assessment

Unclear. Restoring bare reset fallthrough is the direct regression fix, but it conflicts with current docs and tests; if no-model acknowledgement is intentional, rendering these entries as lifecycle/system events is the narrower UI solution.
