## Summary

Keep open. Current main still routes Control UI/WebChat fresh-session actions through `chat.send`, the bare reset handler still returns only a no-model acknowledgement, and the gateway path can still persist/broadcast that acknowledgement as an assistant-shaped `gateway-injected` entry. Related #75235 overlaps on reset readiness, but this issue adds a distinct WebChat transcript/display contract problem.

## What This Changes

WebChat fresh-session resets should not leave the user with only an assistant-looking `✅ New session started.` gateway acknowledgement and no real startup signal or system-event distinction.

## Best Possible Solution

Preserve the current no-model behavior for bare `/new` and `/reset` unless maintainers explicitly choose a greeting policy change, but make WebChat classify exact reset acknowledgements as session-control/system events rather than normal assistant turns. If the reset prompt remains reachable only on agent RPC or reset-tail paths, align prompt/docs copy so users are not promised an immediate greeting where no run starts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger Control UI New Session or send `/new`, observe it goes through `chat.send`, the reset handler returns `✅ New session started.` with `shouldContinue: false`, and the WebChat final/history path can surface that reply as a `gateway-injected` assistant entry.

## Solution Assessment

Yes for a narrow repair. The safest fix is to classify/render the reset acknowledgement separately in WebChat and history while preserving the current no-model reset contract; forcing a greeting should be handled only if maintainers decide to change product policy.
