## Summary

Keep open. Current `main` fixes the typed `/new` WebChat path by routing it to the New Chat action, but typed `/reset` still follows the generic chat command path, stops after a synthetic no-model acknowledgement, and WebChat still renders `gateway-injected` reset acknowledgements as assistant-shaped messages; draft PR #74733 targets the same contract but is open and unmerged.

## What This Changes

The issue asks WebChat fresh-session/reset flows to avoid showing only a synthetic assistant-looking `gateway-injected` acknowledgement when no real model turn has run.

## Best Possible Solution

Land PR #74733 or a narrower equivalent that keeps the documented no-model bare reset contract while rendering reset/new acknowledgements as lifecycle/system events instead of assistant replies.

## Reproduction Assessment

Yes. A source-level reproduction remains for typed WebChat `/reset`: the UI sends `/reset` through `chat.send`, the reset handler returns a synthetic acknowledgement without a model run, and current WebChat projection does not convert that assistant-shaped transcript entry into a lifecycle divider.

## Solution Assessment

Yes for the planned direction, not current main. Given the current documented contract, separating reset acknowledgements from assistant output is narrower and more maintainable than forcing every bare reset to run a greeting model turn.
