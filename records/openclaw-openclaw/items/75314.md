## Summary

Keep open. Current main no longer routes typed WebChat `/new` through the generic `chat.send` acknowledgement path, but typed `/reset` still does; the reset handler still stops after a synthetic acknowledgement, and the WebChat history renderer still treats `gateway-injected` reset acknowledgements as ordinary assistant messages rather than lifecycle dividers. Draft PR #74733 targets the same display/reset contract, but it is open and unmerged.

## What This Changes

The issue requests that WebChat fresh-session/reset acknowledgements not appear as the only assistant-like startup response without either a real greeting or clear lifecycle-event rendering.

## Best Possible Solution

Land PR #74733 or a narrower equivalent that keeps bare reset no-model if desired, while rendering `gateway-injected` reset acknowledgements as lifecycle/system events and keeping WebChat reset history reloads coherent.

## Reproduction Assessment

Yes. A source-level reproduction remains for typed WebChat `/reset`: it dispatches through `chat.send`, the reset handler returns a synthetic acknowledgement without a model run, and current WebChat projection does not convert that `gateway-injected` assistant-shaped transcript entry into a reset lifecycle divider.

## Solution Assessment

Yes for the planned direction, not current main. Because docs and tests now intentionally define bare reset as no-model, separating reset acknowledgements from assistant output is the narrowest maintainable fix; forcing every bare reset to run the model would fight the current contract.
