## Summary

Keep open. Current main only partially avoids the reported UX for the toolbar New Session button; typed WebChat `/new` and `/reset` still go through `chat.send`, the bare reset handler still returns a no-model synthetic acknowledgement, and the UI still has no reset-divider projection for `gateway-injected` assistant-shaped transcript messages. Draft PR #74733 appears to target the same WebChat reset acknowledgement/display contract, but it is open and unmerged, so the issue is not implemented on main.

## What This Changes

The issue reports that fresh WebChat sessions can show only a synthetic gateway-injected reset acknowledgement as an assistant bubble instead of running or clearly separating the expected startup greeting.

## Best Possible Solution

Land PR #74733 or an equivalent narrow patch that keeps the intentional no-model bare reset policy unless maintainers choose otherwise, but renders WebChat reset acknowledgements as lifecycle events/dividers rather than ordinary assistant turns, with focused UI and gateway regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: typed WebChat `/new` dispatches through `chat.send`, the bare reset handler returns a synthetic acknowledgement without a model run, and the injected final reply can be persisted as a `gateway-injected` assistant message that the UI currently renders as an assistant group.

## Solution Assessment

Yes, separating reset acknowledgements from assistant output is the narrowest maintainable direction on current main. Forcing an automatic greeting would contradict existing tests that intentionally keep bare `/new` and `/reset` no-model operations unless maintainers change that product policy.
