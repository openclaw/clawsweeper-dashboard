## Summary

Keep open. Current main still preserves the reported typed WebChat reset path: `/new` and `/reset` are sent through `chat.send`, the bare reset handler returns only the synthetic acknowledgement without starting a model run, and the UI still treats the resulting `gateway-injected` assistant-shaped transcript entry as a normal message rather than a lifecycle divider. The toolbar New Session flow has moved to `sessions.create`, but that only covers one entry point; open draft PR #74733 appears to target the remaining WebChat reset/display contract and has not landed.

## What This Changes

The issue asks WebChat fresh-session reset/new acknowledgements to either trigger a real startup greeting or be rendered clearly as session-control events rather than assistant replies.

## Best Possible Solution

Review and land PR #74733, or an equivalent focused patch, that keeps bare reset as a no-model session-control operation unless product direction changes, while making WebChat display reset/new acknowledgements as explicit session lifecycle events rather than assistant bubbles.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: typed WebChat `/new` still goes through `chat.send`, the bare reset handler returns `shouldContinue: false` with only the acknowledgement, and current tests assert the model is not invoked for bare `/new`.

## Solution Assessment

Yes. Separating session-control acknowledgements from assistant output is the narrowest maintainable fix for the observed ambiguity because it preserves the current tested no-model reset behavior while removing the misleading assistant-shaped UI presentation.
