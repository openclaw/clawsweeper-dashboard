## Summary

Keep open. Current main still has a source-level WebChat `/reset` path where a no-model gateway acknowledgement is stored and rendered as assistant output; the no-model behavior is now documented, so the remaining issue is the UI/session lifecycle distinction already targeted by draft PR #74733.

## What This Changes

WebChat reset/new startup acknowledgements should not appear as real assistant replies or leave users waiting for a second message to know the session is usable.

## Best Possible Solution

Review and land PR #74733 or a narrower equivalent that keeps documented no-model bare resets while rendering reset/new acknowledgements as lifecycle/system events instead of assistant messages.

## Reproduction Assessment

Yes, source-level. Typed WebChat `/reset` dispatches through `chat.send`, the reset handler returns a no-model acknowledgement, the gateway persists it as `gateway-injected` assistant output, and WebChat currently projects it as a normal assistant message.

## Solution Assessment

Yes for the planned direction, not current main. Since current docs and tests preserve no-model bare resets, the narrow fix is to separate gateway acknowledgements from assistant output rather than forcing every reset to run a model greeting.
