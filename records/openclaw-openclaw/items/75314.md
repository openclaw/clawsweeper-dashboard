## Summary

Keep open. Current main partially narrows the report by routing typed `/new` through the dedicated New Chat flow and documenting no-model bare reset semantics, but typed WebChat `/reset` still goes through `chat.send`, stops after a synthetic acknowledgement, and renders the gateway-injected assistant-shaped transcript entry as a normal message; draft PR #74733 is open and unmerged.

## What This Changes

Request that WebChat fresh session/reset handling not leave users with only an assistant-looking gateway-injected acknowledgement and no clear real startup or ready signal.

## Best Possible Solution

Land PR #74733 or a narrower equivalent that keeps the documented no-model bare reset contract while rendering gateway-injected reset/new acknowledgements as lifecycle/system events instead of assistant replies.

## Reproduction Assessment

Yes. A source-level reproduction remains for typed WebChat `/reset`: the UI sends `/reset` through `chat.send`, the reset command returns without continuing to a model run, and the injected acknowledgement is projected as a normal assistant message.

## Solution Assessment

Yes for the planned direction, not current main. Given the current docs say bare `/new` and `/reset` do not invoke the model, separating reset acknowledgements from assistant output is narrower and more maintainable than forcing every bare reset to run a greeting turn.
