Keep open. Current main still has the source-level WebChat/no-startup-run behavior, and related cleanup comments now point WebChat-specific duplicates back to this issue while the broader bare reset contract remains open separately.

## What This Changes

The issue requests that fresh WebChat sessions not show only a synthetic `âś… New session started.` assistant-looking bubble without either a real startup greeting or clear lifecycle/event rendering.

## Best Possible Solution

Define a single reset/new UX contract, then align gateway reset handling, `sessions.create`, gateway-injected transcript rendering, WebChat UI, docs, and tests around either a real startup turn or a clearly non-assistant lifecycle acknowledgement.

## Reproduction Assessment

Yes, source-level. Current main has the bare reset/new short-circuit, a Control UI create-session path without an initial `task` or `message`, and ordinary WebChat rendering for gateway-injected assistant messages.

## Solution Assessment

Unclear. Removing the short-circuit would restore a greeting for some users but conflicts with current no-model docs/tests, so maintainers should first choose the reset/new UX contract and then update code, docs, and tests together.
