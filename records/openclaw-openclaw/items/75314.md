## Summary

Keep open. Current main still has the reported source-level behavior: bare reset/new acknowledgements stop before a model run, the acknowledgement is stored as a `gateway-injected` assistant message, and WebChat does not convert that message into a lifecycle/system event. The broader greeting regression is also active in https://github.com/openclaw/openclaw/issues/77733, but this issue still carries the WebChat/UI distinction that is not fully superseded.

## What This Changes

The issue asks for WebChat fresh/new/reset sessions to avoid showing only a synthetic `gateway-injected` assistant-looking acknowledgement instead of a real startup greeting or clearly separated session-control event.

## Best Possible Solution

Maintainers should decide the reset contract once, then either restore/configure a startup model turn or keep the no-model acknowledgement and render `gateway-injected` reset/new acknowledgements as non-assistant lifecycle events across affected chat surfaces.

## Reproduction Assessment

Yes, source-level. Current main still has a bare reset/new path that returns a synthetic acknowledgement before model execution, persists gateway-injected acknowledgements as assistant messages, and renders non-compaction history entries as ordinary WebChat messages.

## Solution Assessment

Unclear. Removing the short-circuit would restore the greeting for some users but risks undoing the empty-provider-call fix from the reset acknowledgement change; a lifecycle-event rendering fix is narrower if maintainers intend bare resets to stay no-model.
