## Summary

Keep open. Current main still matches the reported WebChat/reset behavior: the New Session control sends `/new` through the generic chat path, bare reset handling returns only a synthetic acknowledgement without starting an agent run, and the no-agent final path can persist/broadcast that acknowledgement as a `gateway-injected` assistant-shaped transcript message. Related PR #74733 appears to target this reset/display contract, but it is still open and unmerged in the provided context.

## What This Changes

The issue requests that fresh WebChat sessions not leave users with only a synthetic `gateway-injected` “New session started” assistant bubble, either by producing a real startup greeting or by rendering the session-control acknowledgement separately from assistant output.

## Best Possible Solution

Keep bare `/new` and `/reset` as no-model control operations unless maintainers explicitly change that policy, but route or render WebChat reset acknowledgements as session lifecycle events/dividers instead of ordinary assistant turns. If #74733 or an equivalent focused patch lands with tests for the reset display path, this issue can close as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger the Control UI New Session action, which sends `/new` through `chat.send`; the reset handler returns only a synthetic acknowledgement without an agent run; the no-agent final path can then append/broadcast a `gateway-injected` assistant-shaped message.

## Solution Assessment

Yes for separating reset acknowledgement from assistant output; that is narrower than forcing a model greeting and preserves the currently tested no-model reset policy. The active PR direction of using `sessions.reset`/session dividers for bare WebChat resets is the maintainable path to review.
