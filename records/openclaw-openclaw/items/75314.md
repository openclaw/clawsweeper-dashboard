## Summary

Keep open. Current main documents bare `/new` and `/reset` as no-model acknowledgements and routes typed WebChat `/new` away from the old chat-send path, but typed WebChat `/reset` still goes through `chat.send`, stops after a synthetic acknowledgement, and the UI still projects `gateway-injected` acknowledgements as normal assistant messages rather than lifecycle dividers.

## What This Changes

Make WebChat fresh-session/reset acknowledgements either trigger a real startup greeting or render as non-assistant lifecycle events instead of assistant-shaped `gateway-injected` bubbles.

## Best Possible Solution

Land PR #74733 or a narrower equivalent that keeps the documented no-model bare reset contract while rendering reset/new acknowledgements as lifecycle/system events instead of assistant replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction remains for typed WebChat `/reset`: it dispatches through `chat.send`, the reset handler returns a synthetic no-model acknowledgement, and WebChat does not convert that `gateway-injected` transcript entry into a lifecycle divider.

## Solution Assessment

Yes for the planned direction, not current main. Given the current documented no-model contract, separating reset acknowledgements from assistant output is narrower than forcing every bare reset to run a greeting turn.
