## Summary

Keep open. Current main still preserves the no-model bare reset policy, but WebChat can still surface the reset acknowledgement through assistant-shaped `gateway-injected` transcript/display paths, so the reported UX/contract mismatch remains actionable rather than implemented or stale.

## What This Changes

WebChat fresh sessions should not leave users with only a synthetic `✅ New session started.` assistant-looking bubble when no real agent startup greeting or model run has occurred.

## Best Possible Solution

Preserve the documented no-model bare reset behavior, but make WebChat treat exact reset acknowledgements as session-control/system events in display and history instead of normal assistant turns. Align any prompt or docs wording that implies an automatic greeting on paths where OpenClaw intentionally does not invoke the model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger the Control UI New Session action or send bare `/new`, observe that `chat.send` routes to the reset command handler, the handler returns only `✅ New session started.` with no agent run, and WebChat can append that reply as a `gateway-injected` assistant-shaped transcript row.

## Solution Assessment

Yes for a narrow repair. The best fix is to classify/render reset acknowledgements as session-control UI while preserving existing transcript parent-chain and replay-filter behavior, rather than reintroducing an automatic startup model run without maintainer product approval.
