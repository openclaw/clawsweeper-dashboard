## Summary

Keep open. Current main still has a source-level WebChat reset path where the Gateway acknowledgement can be persisted as a `gateway-injected` assistant-shaped message and rendered as a normal chat message, while open draft PR #74733 targets the same display and reset-dispatch behavior.

## What This Changes

The issue asks WebChat fresh-session/reset acknowledgements to either trigger a real startup greeting or render as non-assistant lifecycle events instead of a synthetic assistant bubble.

## Best Possible Solution

Review and land PR #74733 or a focused equivalent that preserves documented no-model bare reset behavior while rendering reset/new acknowledgements as lifecycle events and refreshing active WebChat history.

## Reproduction Assessment

Yes, source-level. Typed WebChat `/reset` dispatches through `chat.send`, the reset handler returns a no-model acknowledgement, and WebChat currently renders non-compaction transcript entries as messages.

## Solution Assessment

Yes for the proposed direction, not current main. Since current docs say bare chat `/new` and `/reset` do not invoke the model, separating gateway acknowledgements from assistant output is narrower than forcing every reset to run a greeting.
