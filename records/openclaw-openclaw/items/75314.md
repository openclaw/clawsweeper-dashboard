## Summary

Keep open. Current main partially addresses the report by routing typed WebChat `/new` through the New Chat session-creation path and documenting bare reset as no-model, but typed `/reset` still goes through `chat.send`, can stop after a synthetic acknowledgement, and WebChat still renders gateway-injected reset acknowledgements as normal assistant-shaped history entries; draft PR #74733 is the active candidate for that remaining UI/reset contract.

## What This Changes

Ensure a fresh WebChat new/reset session does not leave users with only a synthetic `gateway-injected` assistant-looking acknowledgement instead of a real greeting or clearly separated system event.

## Best Possible Solution

Review and land PR #74733, or a narrower equivalent, that keeps the documented bare-reset no-model contract while rendering reset acknowledgements as lifecycle/system events and keeping WebChat reset history reloads coherent.

## Reproduction Assessment

Yes. A source-level reproduction remains for typed WebChat `/reset`: it dispatches through `chat.send`, the reset command returns a synthetic no-model acknowledgement, and the current WebChat projection has no reset divider for the resulting `gateway-injected` assistant-shaped row.

## Solution Assessment

Yes for the planned direction, not current main. Because docs now define bare reset as no-model, the narrow maintainable fix is UI/projection separation of reset acknowledgements from assistant output rather than forcing every bare reset to run a model greeting.
