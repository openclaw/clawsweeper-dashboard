## Summary

Keep open: current main still has a credible routed Discord diagnostic-state bug path, and neither the related closed Discord retry issue nor the open session-reset PR fully covers this report.

## What This Changes

Routed non-main Discord sessions should clear diagnostic processing/queue state promptly after run completion, and `/stop` should clear the target session’s queued/processing state.

## Best Possible Solution

Add a narrow regression for routed non-main Discord diagnostic state, then align queue, idle, and stop cleanup on the final target session identity or intentionally alias the diagnostic refs.

## Reproduction Assessment

No live high-confidence reproduction was run, but source inspection gives a credible current-main path: diagnostic queue depth is incremented and only decremented by an idle transition on the same session ref, while routed Discord can queue and dispatch under different session keys. A focused test should reproduce the stale `processing,q=1,last=run:completed` state before fixing it.

## Solution Assessment

Yes: the narrowest maintainable fix is to preserve routing behavior while making diagnostic queue/idle and `/stop` cleanup use the same resolved target session identity. Avoid adding config or changing Discord routing policy unless the regression test proves the identity contract itself is wrong.
