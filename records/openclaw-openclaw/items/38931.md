## Summary

Keep open: current main still implements browser SSRF as static deny/allow plus hostname exceptions, while per-request confirmation, hard blocklist semantics, audit logging, and temporary access remain missing and security/product-sensitive. Related open issues cover narrower URL denylist and web_fetch private-network access, but they do not supersede this browser approval design item.

## What This Changes

Requests a browser SSRF policy mode that can ask for user confirmation, support allow/block lists, hard-block cloud metadata endpoints, and temporarily allow private-network access.

## Best Possible Solution

Define a typed browser SSRF policy contract with confirmation semantics, hard-deny precedence, approval routing, logging, docs, config schema/help updates, and regression coverage in the shared SSRF and browser navigation layers.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing security/config feature. Source, docs, and tests show current main still has static browser SSRF allow/deny and hostname-list behavior, not a confirmable request path.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. The maintainable path is a maintainer-approved browser SSRF policy contract before implementation, not a narrow config-only patch.
