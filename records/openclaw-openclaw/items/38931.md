## Summary

Keep open: current `main` and the latest release still provide only static browser SSRF allow/deny behavior, while the requested confirmation mode, IP/CIDR allow/block lists, audit logging, and temporary CLI workflow remain absent. The prior discussion and related searches point to this as a security/product design item rather than a stale duplicate.

## What This Changes

The issue requests a browser private-network SSRF policy that can ask for user confirmation, support allow/block lists, log attempts, and temporarily enable access.

## Best Possible Solution

Keep this open as the canonical browser private-network approval design item until maintainers define a typed SSRF policy covering confirmation, precedence, hard-deny rules, approval delivery, audit behavior, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing security/config feature. Source and docs inspection show current main and `v2026.5.12` still lack confirm, blocklist, audit, and temporary CLI fields and only support immediate static allow/deny behavior.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. The maintainable path is a maintainer-approved browser SSRF policy contract that defines confirmation semantics, hard-deny precedence, routing, logging, docs, and regression tests before implementation.
