## Summary

Keep open: current main and the latest release still expose browser SSRF as strict/default blocking plus boolean private-network opt-in and hostname exceptions, while the requested confirmation, blocklist, and temporary access modes are new security-sensitive product/config work.

## What This Changes

The issue requests a browser SSRF policy mode that can ask for user confirmation, allowlist or blocklist private targets, and temporarily enable private-network access.

## Best Possible Solution

Define a reviewed typed browser SSRF policy contract with hard-deny precedence for metadata/link-local targets, explicit approval routing, timeout/rate-limit behavior, audit semantics, config schema/help/docs updates, and regression coverage across the shared SSRF and browser navigation layers.
