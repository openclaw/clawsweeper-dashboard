## Summary

Keep open: current source, tests, docs, and the latest release still model browser SSRF as static allow/deny plus host exceptions. The requested approval, audit, blocklist, and temporary-access behavior remains a security/product design item rather than a stale duplicate.

## What This Changes

The issue requests a browser SSRF policy that can prompt before private-network access, support allow/block lists, hard-block metadata endpoints, log attempts, and optionally expose temporary CLI controls.

## Best Possible Solution

Keep this open as the canonical browser private-network approval design item until maintainers define a typed SSRF policy covering confirmation, precedence, hard-deny metadata rules, approval delivery, audit behavior, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing security/config feature. Current-main source, docs, and tests show the browser policy still has only static allow/deny and hostname-list behavior.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. The maintainable path is a maintainer-approved browser SSRF policy contract that defines confirmation semantics, hard-deny precedence, approval routing, logging, docs, and regression tests before implementation.
