## Summary

Keep open: current main and v2026.5.12 still expose browser SSRF as static private-network allow/deny plus hostname exceptions, with no confirmation state, timeout, request approval routing, user-facing allow/block list contract, audit option, or temporary access command. The issue is security-sensitive and needs maintainer product/security design before implementation.

## What This Changes

Requests a browser SSRF policy mode that can confirm private-network access, support allow/block lists, hard-block metadata endpoints, audit attempts, and temporarily enable access from the CLI.

## Best Possible Solution

Define a typed browser SSRF policy contract with reviewed confirmation semantics, hard-deny precedence, approval routing, logging, docs, config schema/help updates, and regression coverage in the shared SSRF and browser navigation layers.
