## Summary

Keep open: current main and v2026.6.5 still provide strict blocking, broad private-network opt-in, and static hostname exceptions, but not per-request SSRF confirmation or temporary grants; this needs explicit product and security design before implementation.

## What This Changes

Requested an SSRF private-network policy mode that asks for user confirmation, supports allow/block lists, or grants temporary access instead of only broad allow/deny behavior.

## Best Possible Solution

Define one browser-owned, fail-closed approval contract that revalidates every redirect and connection, binds grants to the narrowest approved destination and session scope, and keeps maintainer-selected sensitive destinations non-overridable.

## Reproduction Assessment

Not applicable: current main behaves according to its documented strict and static-exception SSRF policy, while this issue requests a new intermediate approval capability.

## Solution Assessment

No, not as three parallel configuration and CLI policy paths. The maintainable direction is one canonical browser approval model integrated with the existing SSRF guard and approval infrastructure.
