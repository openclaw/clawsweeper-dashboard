## Summary

Keep open. Current main still has a static browser SSRF policy and immediate allow/block navigation guards, while the request asks for a security-sensitive middle policy with confirmation, allow/block precedence, metadata hard-blocking, timeout/rate-limit/audit behavior, and possible CLI controls.

## What This Changes

The issue requests a browser SSRF private-network access policy that can ask for user confirmation and support allowlists, blocklists, metadata blocking, logging, rate limits, and temporary enablement.

## Best Possible Solution

Keep this as the canonical browser private-network confirmation design request. The end state should be a maintainer-approved browser SSRF contract covering deny/allow/confirm precedence, CIDR/host/port allow and block rules, non-overridable cloud metadata blocks, timeout/rate-limit/audit behavior, redirect and DNS-rebinding handling, UI approval delivery, docs, and focused security tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/security design gap. The absence is high-confidence from current source and tests: proposed confirm/blocklist fields are outside the strict schema, and navigation currently returns or throws instead of waiting for approval.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. Keeping the issue open for maintainer security/product design is the right path before implementation, because a safe solution needs typed policy semantics and approval UX rather than a one-file toggle.
