## Summary

Keep open: current main and the latest release still implement browser SSRF as strict static allow/deny plus hostname exceptions, while user confirmation, hard block/allow lists, audit logging, and temporary access are not implemented and need security/product review.

## What This Changes

Request a browser SSRF policy mode that can ask users to approve private-network navigation, support allow/block lists and temporary access, and hard-block metadata endpoints.

## Best Possible Solution

Define a typed browser SSRF policy contract with confirmation semantics, hard-deny precedence, approval routing, logging, docs, config schema/help updates, and regression coverage in the shared SSRF and browser navigation layers.
