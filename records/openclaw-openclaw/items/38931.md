## Summary

Keep open: current main and the latest release still expose browser SSRF as static private-network allow/deny plus hostname exceptions, with no confirmation state, timeout, approval routing, blocklist contract, audit option, or temporary access command. The requested behavior changes a security boundary and needs product/security review before implementation.

## What This Changes

The issue requests a browser SSRF policy mode that can prompt for user confirmation, support allow/block lists, and temporarily allow private-network access.

## Best Possible Solution

Define a typed browser SSRF policy contract with reviewed confirmation semantics, hard-deny precedence, approval routing, logging, docs, config schema/help updates, and regression coverage in the shared SSRF and browser navigation layers.
