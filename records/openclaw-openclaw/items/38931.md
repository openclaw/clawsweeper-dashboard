## Summary

Keep open: current main still has a static boolean browser SSRF policy with immediate allow/deny behavior. The related open metadata-hardening PR is useful partial work, but it does not implement the requested confirmation mode, allow/block-list policy, audit behavior, or temporary CLI workflow.

## What This Changes

The issue requests an SSRF private-network policy mode that can prompt for user approval, support allow/block lists, hard-block cloud metadata, and optionally expose temporary CLI controls.

## Best Possible Solution

Keep this as the canonical browser private-network approval design item until maintainers settle a typed SSRF policy contract and matching browser guard, UI/approval delivery, hard-deny precedence, audit/logging behavior, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a security/product feature request. Source inspection confirms current main lacks confirm, blocklist, audit, and CLI fields and only has immediate SSRF allow/deny paths.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. The maintainable path is a maintainer-approved core/browser SSRF design covering policy semantics, approval delivery, hard-deny precedence, audit behavior, docs, and tests.
