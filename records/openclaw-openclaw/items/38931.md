## Summary

Keep open: current main still exposes browser SSRF as static deny/allow plus hostname exceptions, while the requested per-request confirmation, hard blocklist, audit logging, and temporary access mode remain missing and security/product-sensitive.

## What This Changes

The issue requests a browser SSRF policy mode that asks for user confirmation or temporary approval before private-network navigation instead of only static allow/deny behavior.

## Best Possible Solution

Keep this as the canonical browser private-network approval design item until maintainers define a typed SSRF policy with confirmation semantics, hard-deny precedence, approval routing, logging, docs, and regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a missing security/config feature. Source, docs, and tests show the current browser policy still has static allow/deny and hostname-list behavior.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. The maintainable path is a maintainer-approved SSRF policy contract before implementation, not a narrow bug patch.
