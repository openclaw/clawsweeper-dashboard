## Summary

Keep open: current main and the live issue state still show browser SSRF as static allow/deny plus host exceptions. The requested confirmation, blocklist, audit, and temporary-access behavior remains unimplemented and needs maintainer security/product design.

## What This Changes

The issue requests an intermediate browser SSRF policy that can confirm, allowlist/blocklist, log, or temporarily permit private-network access instead of only allowing or denying it.

## Best Possible Solution

Keep this open as the canonical browser private-network approval design item until maintainers define a typed SSRF policy with confirmation semantics, hard-deny precedence, approval routing, logging, docs, and regression tests.

Not applicable as a bug reproduction; this is a missing security/config feature. Source, docs, and tests show the current browser policy still has static allow/deny and hostname-list behavior.
