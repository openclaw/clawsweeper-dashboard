## Summary

Keep open: current main and the latest release still expose a static browser SSRF policy with immediate allow/deny behavior, while the requested confirmation mode, IP/CIDR allow/block lists, audit behavior, and temporary CLI workflow remain absent. The related metadata-hardening PR is useful partial work but does not cover the central approval workflow.

## What This Changes

The issue requests a browser SSRF policy that can confirm private-network access per request, support allow/block lists, hard-block metadata endpoints, log attempts, and optionally expose temporary CLI controls.

## Best Possible Solution

Keep this as the canonical browser private-network approval design item until maintainers settle a typed SSRF policy contract spanning config, browser guard behavior, approval delivery, hard-deny precedence, audit/logging, docs, and tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a security/product feature request. Source inspection confirms current main lacks confirm, blocklist, audit, and CLI fields and only has immediate SSRF allow/deny paths.

## Solution Assessment

Unclear whether the exact proposed JSON and CLI shape is the best final API. The maintainable path is a maintainer-approved core/browser SSRF design covering policy semantics, approval delivery, hard-deny precedence, audit behavior, docs, and tests.
