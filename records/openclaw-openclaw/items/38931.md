## Summary

Keep open: both current main and v2026.6.5 provide strict blocking, broad private-network opt-in, and static hostname exceptions, but no per-request confirmation or temporary grant. The goal fits OpenClaw’s security direction, while the approval identity, lifetime, immutable denies, and UI/protocol contract require explicit maintainer product and security decisions.

## What This Changes

Requests an interactive or scoped approval mechanism for browser access to private-network destinations instead of the current global allow-or-deny policy.

## Best Possible Solution

Define one browser-owned, fail-closed approval contract that revalidates every redirect and connection, binds grants to the narrowest approved destination and session scope, and keeps maintainer-selected sensitive destinations non-overridable.

## Reproduction Assessment

Not applicable: current main behaves according to its documented strict and static-exception policy, while this issue requests a new intermediate approval capability.

## Solution Assessment

No, not as three parallel configuration and CLI policy paths; the maintainable solution is one canonical browser approval model integrated with the existing SSRF guard and generic approval infrastructure.

## Risks / Open Questions

- An approval must remain narrowly bound across DNS rebinding, redirects, ports, protocols, and concurrent sessions; a hostname-only grant could authorize a later network pivot.
- A separate browser approval policy could drift from static hostname exceptions and the generic plugin approval queue, creating competing authorization sources.
- The issue proposes permanently blocked metadata destinations, but the exact special-use ranges and whether any are operator-overridable require an explicit security decision.
- Allow-always or timed grants introduce persisted or session state whose ownership, expiry, auditing, and upgrade behavior are not yet defined.
