## Summary

Keep open: the Nostr relay validation bug is still present on current main, but this PR is not merge-ready because it only hardens schema parsing while setup paths can still write remote plaintext `ws://` relays, and the branch is stale against current test/baseline layout.

## What This Changes

The PR adds loopback-aware Nostr relay URL validation to `NostrConfigSchema`, adds relay schema tests in the old `types.test.ts`, and updates obsolete secret-baseline line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that exposes one shared relay URL policy used by config schema and both setup parsers, with coverage in current `channel.test.ts`/setup tests and no obsolete baseline churn.

## Reproduction Assessment

Yes. Source inspection on current main shows schema parsing accepts arbitrary relay strings and both setup parsers accept any `ws:` or `wss:` host; I did not run tests because this was a read-only review and `pnpm` is unavailable here.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should centralize the relay URL policy and reuse it in setup as well as schema validation, then move coverage to the current test file layout.
