## Summary

Keep this PR open: it is useful Nostr relay hardening, but the patch still leaves setup paths accepting remote plaintext `ws://` relays and lacks after-fix real behavior proof.

## What This Changes

The PR adds a Nostr relay URL schema validator for `wss://` plus loopback `ws://`, schema tests, and `.secrets.baseline` line-number updates.

## Best Possible Solution

Land a rebased Nostr-local hardening that exports or reuses one relay URL policy across config schema, guided setup, and delegated setup, with focused coverage in the current Nostr tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts arbitrary relay strings in `NostrConfigSchema`, and both setup parsers accept any `ws:` or `wss:` URL; no runtime command was run because this was a read-only review.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should centralize the relay policy and reuse it in setup so user-facing setup cannot accept relays the schema rejects.
