## Summary

Keep open: the relay schema validation is useful, but the PR still leaves Nostr setup paths on a looser protocol-only policy, so setup can persist remote plaintext ws:// relays that the new schema rejects. The external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds Nostr relay URL validation to the config schema, restricts ws:// to local hosts, adds relay validation tests to the old Nostr types test file, and updates the old secrets baseline.

## Best Possible Solution

Share one Nostr-local relay URL validator across config-schema.ts, setup-surface.ts, and channel.setup.ts, with current-main tests covering direct config and setup persistence.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: direct config accepts arbitrary relay strings, and both setup parsers accept any ws: or wss: URL before persisting relays.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should reuse the same relay policy in direct config and both setup paths so setup cannot save values the schema rejects.
