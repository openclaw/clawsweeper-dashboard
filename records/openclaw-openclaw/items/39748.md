## Summary

Keep this PR open: the Nostr relay validation problem is still present on current main, but this branch is stale and incomplete because it validates only the config schema while current setup paths can still persist remote plaintext `ws://` relays.

## What This Changes

The PR adds a Nostr relay URL validator for `channels.nostr.relays`, adds relay protocol tests in the old `types.test.ts`, and updates obsolete secret-scan baseline line numbers.

## Best Possible Solution

Maintain one shared Nostr relay URL contract across schema, setup adapter, setup wizard, tests, and docs: remote relays require `wss://`, and plaintext `ws://` is loopback-only.

## Reproduction Assessment

Yes. Source inspection is enough: direct schema validation accepts any relay string, while the two setup parsers reject non-WebSocket protocols but still accept remote `ws://` hosts.

## Solution Assessment

No. The loopback-only plaintext policy is the right direction, but the PR needs to share the validator with current setup paths, move tests into the current layout, and drop stale baseline churn.
