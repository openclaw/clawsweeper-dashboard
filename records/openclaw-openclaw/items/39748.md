## Summary

Keep open: this is a real Nostr relay validation bug, but the PR is not merge-ready because it only validates direct config while both setup parsers can still persist remote plaintext `ws://` relays; it also lacks external after-fix real behavior proof.

## What This Changes

The PR adds a Nostr relay URL validator for `wss://` and loopback `ws://`, applies it to `NostrConfigSchema.relays`, adds schema tests in `extensions/nostr/src/types.test.ts`, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Refactor a Nostr-local relay URL validator shared by direct config, guided setup, and dependency-light setup, move regression coverage into the current Nostr test layout, refresh the branch, and add redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection is enough: current main accepts arbitrary direct relay strings and both setup parsers accept and persist any `ws:` or `wss:` relay URL.

## Solution Assessment

No. The direct config validator is a useful building block, but the maintainable fix must apply the same relay policy to `config-schema.ts`, `setup-surface.ts`, and `channel.setup.ts` with tests in the current layout.
