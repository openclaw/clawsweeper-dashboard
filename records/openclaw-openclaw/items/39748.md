## Summary

Keep open: the underlying Nostr relay URL validation problem still exists on current main, but this patch only covers direct config, leaves setup paths able to persist remote `ws://` relays, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a Nostr relay URL schema for `wss://` and loopback `ws://`, applies it to direct Nostr config relays, adds schema tests in `extensions/nostr/src/types.test.ts`, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Share one Nostr-local relay URL validator across direct config, guided setup, and dependency-light setup, enforce remote `wss://` plus loopback-only `ws://`, and add coverage in the current Nostr test files.

## Reproduction Assessment

Yes. Source inspection of current main shows direct config accepts `z.array(z.string())` and both setup parsers accept and persist any `ws:` or `wss:` relay URL.

## Solution Assessment

No. The direct schema validator is useful, but the maintainable fix must apply the same relay policy to `config-schema.ts`, `setup-surface.ts`, and `channel.setup.ts` with tests in the current layout.
