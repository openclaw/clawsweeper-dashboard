## Summary

Keep open: the Nostr relay validation bug is still present on current main and this PR is useful, but the branch is incomplete because setup paths can still persist remote plaintext `ws://` relays, the branch is conflicting with current main, and external real behavior proof is still missing.

## What This Changes

The PR adds a Nostr relay URL schema for `wss://` and loopback-only `ws://`, applies it to config `relays`, adds relay validation tests in the old `types.test.ts`, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Share one Nostr-local relay URL policy across `config-schema.ts`, `setup-surface.ts`, and `channel.setup.ts`, then move focused coverage into current Nostr tests and include redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows direct config accepts arbitrary relay strings, while both setup parsers accept any `ws:` or `wss:` URL before writing config.

## Solution Assessment

No. The schema validator is the right direction, but the best fix should reuse the same relay policy in both setup paths so setup cannot persist values the schema rejects.
