## Summary

Keep open: the Nostr relay validation gap is still present on current main, but this PR is not merge-ready because it hardens only the config schema while both setup paths can still persist remote plaintext `ws://` relays.

## What This Changes

This PR adds Nostr relay URL validation to the config schema, adds schema coverage for `wss://` and loopback-only `ws://`, and updates secrets-baseline line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that shares one relay URL policy across the config schema, guided setup, and dependency-light setup, with focused schema/setup coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main accepts arbitrary relay strings in `NostrConfigSchema`, and both setup parsers accept any `ws:` or `wss:` URL before writing config.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should reuse the same relay policy in `setup-surface.ts` and `channel.setup.ts` so setup cannot persist values the schema forbids.
