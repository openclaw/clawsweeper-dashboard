## Summary

Keep open: the underlying Nostr relay validation problem is real on current main, but this PR is not merge-ready because it only tightens the config schema, leaves setup paths accepting remote plaintext `ws://` relays, targets files current main removed, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a Nostr relay URL schema that permits remote `wss://` and loopback `ws://`, applies it to `relays`, adds schema tests, and updates generated secrets-baseline line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that shares one relay URL validator across config schema and setup parsers, adds coverage in the current Nostr test file, and leaves obsolete generated baselines untouched.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main accepts arbitrary `relays` strings in `NostrConfigSchema`, and setup accepts any `ws:` or `wss:` host; I did not execute tests because this was a read-only review.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should centralize the relay policy and reuse it in setup as well as config parsing, with tests moved to the current `channel.test.ts` structure.
