## Summary

This PR should stay open: the relay URL hardening is a useful bug fix, but the current branch is incomplete against current main, still targets obsolete files, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a Nostr relay URL validator for `wss://` and loopback `ws://`, applies it to `relays`, adds tests under the old `types.test.ts`, and changes generated secrets-baseline line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that shares one relay URL policy across config validation, generated metadata, setup paths, and `channel.test.ts`, while dropping obsolete baseline churn.

## Reproduction Assessment

Yes. Source inspection on current main shows `NostrConfigSchema` still accepts arbitrary relay strings, and both Nostr setup parsers accept any `ws:` or `wss:` URL.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix needs to enforce the same relay policy in generated config validation and both setup paths, with tests moved to the current Nostr test file.
