## Summary

Keep open. The PR is a useful Nostr-local security hardening, but current main still has the relay validation gap, the branch is stale against current main, setup flows would remain less strict than the new schema, and the external PR still lacks inspectable real behavior proof.

## What This Changes

The PR adds Nostr relay URL validation so configured relays must use `wss://` or loopback-only `ws://`, with schema tests for accepted and rejected protocols.

## Best Possible Solution

Land a rebased Nostr-local fix that centralizes one relay URL validator across `NostrConfigSchema`, `setup-surface.ts`, and `channel.setup.ts`, adds current tests in `channel.test.ts`, and omits obsolete baseline churn.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: current main accepts arbitrary relay strings in `NostrConfigSchema`, and both setup parsers accept any `ws:` or `wss:` URL. I did not run tests because this was a read-only review.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should share one relay policy with both setup parsers and place regression coverage in the current Nostr test file.
