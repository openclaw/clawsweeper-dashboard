## Summary

Keep open: the PR is a useful Nostr relay validation hardening change, but the current branch is conflicting, lacks after-fix real behavior proof, and still leaves setup flows able to persist remote plaintext `ws://` relays.

## What This Changes

The PR adds Nostr relay URL validation to `NostrConfigSchema`, adds relay validation tests in the old Nostr types test file, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Share one Nostr-local relay URL policy across the config schema and both setup paths, move focused coverage to current Nostr tests, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows direct config accepts arbitrary relay strings, while both setup parsers accept any `ws:` or `wss:` URL before writing config.

## Solution Assessment

No. The schema validator is the right direction, but the best fix should reuse the same relay policy in `config-schema.ts`, `setup-surface.ts`, and `channel.setup.ts` so setup cannot persist values the schema rejects.
