## Summary

Keep this PR open: the validation goal is valid and still missing on current main, but the patch only tightens the direct schema while setup paths can still persist remote plaintext `ws://` relays; it also lacks after-fix real behavior proof.

## What This Changes

The PR adds Nostr relay URL validation to the config schema, adds relay validation tests in the old Nostr types test file, and updates a secrets baseline that no longer exists on current main.

## Best Possible Solution

Share one Nostr-local relay URL policy across `config-schema.ts`, `setup-surface.ts`, and `channel.setup.ts`, then add current-main tests that cover direct config and setup persistence paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: direct config accepts arbitrary relay strings, and both setup parsers accept any `ws:` or `wss:` URL before persisting config.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should reuse the same relay policy in the schema and both setup paths so setup cannot save values the schema rejects.
