## Summary

Keep open: the underlying Nostr relay-validation bug still exists on main, and this PR is useful but incomplete because setup validators still accept remote plaintext `ws://` relays and contributor real behavior proof is missing.

## What This Changes

The PR adds a Nostr relay URL validator to the config schema, restricts `ws://` to loopback hosts after review, adds schema tests, and updates secrets-baseline line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that exposes or reuses one relay URL policy for config schema, non-interactive setup, and guided setup, with focused schema/setup tests and no unrelated baseline churn.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main accepts arbitrary relay strings in `NostrConfigSchema`, and both setup parsers accept any `ws:` or `wss:` URL; no tests were run because this is a read-only review.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should centralize the relay policy and reuse it in setup so user-facing setup cannot accept a relay the schema rejects.
