## Summary

Keep open. The PR addresses a real Nostr relay-validation gap, and its later commit fixed the original unrestricted `ws://` review concern, but the patch is still partial because setup validators can persist relay values the new schema would reject.

## What This Changes

The PR adds Nostr config-schema relay URL validation, schema tests for `wss://` and loopback-only `ws://`, and `.secrets.baseline` line-number updates.

## Best Possible Solution

Land a rebased Nostr-local hardening that shares one relay URL policy across the config schema, guided setup, and dependency-light setup, with focused coverage and redacted real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main accepts arbitrary relay strings in `NostrConfigSchema`, and both setup parsers accept any `ws:` or `wss:` URL before writing config.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should reuse the same relay policy in `setup-surface.ts` and `channel.setup.ts` so setup cannot persist values the schema forbids.
