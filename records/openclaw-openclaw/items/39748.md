## Summary

Keep open: the Nostr relay validation hardening is still useful on current main, but the PR remains incomplete because setup paths can still accept remote plaintext `ws://` relays and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds a Nostr relay URL schema validator restricting relays to `wss://` plus loopback `ws://`, adds schema tests, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that shares one relay URL policy across config schema, guided setup, and delegated setup, with focused tests in the current Nostr test file.

## Reproduction Assessment

Yes. Source inspection shows current main accepts arbitrary relay strings in `NostrConfigSchema`, while both setup parsers accept any `ws:` or `wss:` URL; no runtime tests were run because this was a read-only review.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should centralize the relay policy and reuse it in setup so user-facing setup cannot accept a relay the schema rejects.
