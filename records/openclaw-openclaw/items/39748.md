## Summary

Keep this PR open: it addresses a real Nostr relay validation bug, but the branch is not merge-ready because it only validates direct config schema input while current setup paths can still persist remote plaintext `ws://` relays, and the tests/baseline edits target paths that no longer exist on current main.

## What This Changes

The PR adds a Nostr relay URL validator allowing remote `wss://` and loopback `ws://`, applies it to `relays`, adds protocol tests, and updates secret-scan baseline line numbers.

## Best Possible Solution

Maintain one shared Nostr relay URL contract: remote relays require `wss://`, plaintext `ws://` is loopback-only, and schema, setup flows, metadata, tests, and docs agree.

## Reproduction Assessment

Yes. Source inspection of current main shows direct schema validation accepts any relay string, while the two setup parsers reject non-WebSocket protocols but still accept remote `ws://` hosts.

## Solution Assessment

No. The loopback-only plaintext policy is the right direction, but the schema-only helper and stale test/baseline paths are not the narrow maintainable fix for current main.
