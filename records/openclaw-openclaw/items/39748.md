## Summary

Keep open. The PR is useful Nostr security hardening, but it is not merge-ready because the same relay URL policy is not reused by the guided and dependency-light setup paths, and the external contributor has not provided after-fix real behavior proof.

## What This Changes

The PR adds Nostr relay URL schema validation for `wss://` plus loopback-only `ws://`, adds schema tests, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Land a rebased Nostr-local hardening that shares one relay URL policy across the config schema, guided setup, and dependency-light setup, with focused coverage in the current Nostr tests.

## Reproduction Assessment

Yes, by source inspection. Current main accepts arbitrary relay strings in `NostrConfigSchema`, and both setup parsers accept any `ws:` or `wss:` URL; no runtime command was run because this was a read-only review.

## Solution Assessment

No. The schema validator is the right direction, but the maintainable fix should centralize or export the relay policy and reuse it in all config-writing setup paths before merge.
