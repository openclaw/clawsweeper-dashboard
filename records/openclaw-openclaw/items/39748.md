## Summary

Keep open: the Nostr relay validation gap is still present on current main, but this branch is not merge-ready because real behavior proof is absent and the direct-config-only schema would create upgrade and setup-path compatibility risk.

## What This Changes

The PR adds direct Nostr relay URL schema validation for remote `wss://` and loopback `ws://`, adds tests under the old `types.test.ts` path, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Use one Nostr-local relay URL normalizer/validator across direct config and setup paths, with an explicit maintainer decision on alias compatibility plus upgrade coverage and redacted real setup proof.

## Reproduction Assessment

Yes, from source inspection. Current main accepts arbitrary direct relay strings, setup parsers use a separate `ws:`/`wss:` parser, and the pinned `nostr-tools` source normalizes HTTP(S) aliases before relay connection; I did not run a live Nostr relay.

## Solution Assessment

No. The direct-config-only schema is not the best merge shape because it can break normalized aliases and leaves setup validation on a separate contract; the safer path is a shared validator/normalizer with upgrade proof.
