## Summary

Keep open: the PR improves direct Nostr config validation, but the same relay policy is still bypassable through setup paths and the external real behavior proof gate is still missing.

## What This Changes

The PR adds a Nostr relay URL schema for `wss://` and loopback `ws://`, applies it to direct `relays` config, adds schema tests, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Share one Nostr-local relay URL validator across direct config, guided setup, and dependency-light setup, enforcing remote `wss://` plus loopback-only `ws://` with focused tests and real behavior proof.

## Reproduction Assessment

Yes. Source inspection of current main shows direct config accepts `z.array(z.string())`, while both setup parsers accept any `ws:` or `wss:` relay and persist through a helper that does not rerun schema validation.

## Solution Assessment

No. Validating only `NostrConfigSchema` is useful but incomplete; the same relay policy should be shared by direct config, guided setup, and dependency-light setup.
