## Summary

Keep this PR open: it is a focused Nostr hardening change, but it is not merge-ready because the current patch can break existing relay URL aliases, leaves setup validation on a different contract, conflicts with current main, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds Nostr direct-config relay URL validation for remote `wss://` and loopback `ws://`, adds relay schema tests under the old `types.test.ts` path, and updates `.secrets.baseline` line numbers.

## Best Possible Solution

Use one Nostr-local relay normalizer/validator across direct config and setup, preserving or explicitly migrating alias behavior with focused tests and redacted real setup proof.

## Reproduction Assessment

Yes, from source inspection. Current main accepts arbitrary direct relay strings, setup parsers use a separate `ws:`/`wss:` parser, and the pinned `nostr-tools` code normalizes HTTP(S) aliases before relay connection; I did not run a live Nostr relay.

## Solution Assessment

No. The direct-config-only schema is not the best merge shape because it can break normalized aliases and leaves setup validation on a separate contract; the safer path is a shared validator/normalizer with upgrade proof.
