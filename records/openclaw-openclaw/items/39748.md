## Summary

Keep open: current main still leaves Nostr relay config permissive, but this PR is not merge-ready because it hardens only direct config, can reject relay aliases that currently work through dependency normalization, conflicts with current main, and lacks after-fix real setup proof.

## What This Changes

The PR adds Nostr relay URL schema validation and tests so direct config accepts remote `wss://` relays plus loopback `ws://` development relays while rejecting other schemes.

## Best Possible Solution

Use one Nostr-local relay URL normalizer/validator across direct config, dependency-light setup, and guided setup, preserving or explicitly migrating currently normalized aliases with focused tests and real setup proof.

## Reproduction Assessment

Yes, from source inspection. Current main accepts arbitrary direct relay strings, both setup parsers accept any `ws:` or `wss:` URL, and the pinned `nostr-tools` source normalizes `https:` to `wss:` before connecting; I did not run a live Nostr relay.

## Solution Assessment

No. The loopback-only follow-up addressed the first review concern, but the maintainable fix needs shared validation plus explicit upgrade handling across every Nostr relay entry point.
