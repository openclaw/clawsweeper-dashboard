## Summary

Keep open. Current main does not implement the central gateway restart transaction/RPC path, and the PR is not mergeable as-is because source/API inspection confirms blocking lifecycle and restart-recovery regressions in the proposed patch.

## What This Changes

The PR adds a `gateway.restart` RPC with durable restart transaction and post-boot handoff behavior, routes config/update/tool/CLI restarts through that path, updates gateway docs/protocol/tests, and changes bundled plugin resolution under Vitest to prefer the current worktree.

## Best Possible Solution

Split this into a narrow bundled-dir Vitest/worktree resolver fix and a separate gateway restart transaction/RPC handoff PR. The restart PR should keep protocol schema, server handlers, CLI behavior, docs, changelog, and tests together, preserve existing JSON/Windows lifecycle behavior, and prove the missing-sentinel transaction recovery path from startup.

## Reproduction Assessment

Yes. Source inspection of the PR head reproduces the review findings: the RPC restart branch returns before daemon JSON output handling, the transaction fallback is unreachable when startup skips wake scheduling without a sentinel file, and the active run is aborted before restart scheduling is confirmed.

## Solution Assessment

No. The bundled-dir resolver change may be a narrow fix, but combining it with a broad security-sensitive gateway restart RPC and transaction handoff makes this PR harder to review and currently introduces blocking lifecycle regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve daemon JSON output after RPC restarts:** `src/cli/daemon-cli/lifecycle.ts:386-399`
  - body: When the RPC-first restart succeeds, this branch returns before `runServiceRestart` creates the daemon action/output context, and it passes a fresh `warnings: []` to the post-check. `openclaw gateway restart --json` can therefore emit no structured success response and drop post-check warnings, unlike the existing service-manager path.
  - confidence: 0.93
- **[P2] Run transaction recovery without a sentinel file:** `src/gateway/server-restart-sentinel.ts:160-162`
  - body: The new fallback synthesizes a payload from a pending restart transaction inside `scheduleRestartSentinelWake`, but startup only calls that function after `hasRestartSentinel()` is true. If the sentinel write is missing, this recovery path is never reached and the interrupted-turn follow-up can be skipped.
  - confidence: 0.9
- **[P2] Schedule the restart before aborting the active run:** `src/gateway/restart-transaction.ts:102-119`
  - body: The terminal-handoff path aborts the active run before `scheduleGatewaySigusr1Restart` and the following transaction write succeed. If either step throws, the user-facing turn has already been aborted with no restart queued.
  - confidence: 0.87
- **[P2] Keep the Windows restart wait budget:** `src/cli/daemon-cli/lifecycle.ts:373-374`
  - body: The refactor removes the current Windows-specific 180s restart wait and computes the wait from the default attempt budget for all platforms. That can make managed Windows restarts fail early compared with current main's intentional longer timeout.
  - confidence: 0.82
