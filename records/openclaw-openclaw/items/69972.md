## Summary

Keep this PR open. Current main still lacks the PR’s central gateway restart RPC, restart transaction, RPC-first CLI path, and Vitest worktree resolver behavior; the PR is also an XL admin lifecycle change with concrete blocking review findings, so it is not safe cleanup-closure material.

## What This Changes

The PR adds a `gateway.restart` RPC and durable restart transaction/handoff flow, wires config/update/tool/CLI restart paths through it, updates Gateway docs/protocol/tests, and changes bundled plugin resolution under Vitest to prefer the current worktree.

## Best Possible Solution

Split this into a narrow bundled-dir Vitest/worktree resolver fix and a separate gateway restart transaction/RPC handoff PR. The restart PR should keep schema/docs/tests together, preserve existing CLI JSON behavior and Windows restart timing, and prove transaction recovery reaches the missing-sentinel case.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows the RPC restart path returns before daemon JSON output handling, the transaction fallback is unreachable when the sentinel file is absent, and active runs are aborted before restart scheduling is confirmed. Current-main inspection also reproduces the cleanup decision: the PR’s intended behavior is not implemented on main.

## Solution Assessment

No, not as currently packaged. The bundled-dir resolver change may be a narrow fix, but the restart transaction/RPC handoff is a broad admin lifecycle feature and should be reviewed separately after the blocking behavior findings are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve daemon JSON output after RPC restarts:** `src/cli/daemon-cli/lifecycle.ts:392-399`
  - body: When the RPC-first restart succeeds, this branch returns before `runServiceRestart` creates the daemon action/output context, and it passes a fresh `warnings: []` to the post-check. `openclaw gateway restart --json` can therefore emit no structured success response and drop post-check warnings, unlike the existing service-manager path.
  - confidence: 0.93
- **[P2] Run transaction recovery without a sentinel file:** `src/gateway/server-restart-sentinel.ts:150-155`
  - body: The new fallback synthesizes a payload from a pending restart transaction inside `scheduleRestartSentinelWake`, but startup only calls that function after `hasRestartSentinel()` is true. If the sentinel write is missing, this recovery path is never reached and the interrupted-turn follow-up can be skipped.
  - confidence: 0.9
- **[P2] Schedule the restart before aborting the active run:** `src/gateway/restart-transaction.ts:102-119`
  - body: The terminal-handoff path aborts the active run before `scheduleGatewaySigusr1Restart` and the following transaction write succeed. If either step throws, the user-facing turn has already been aborted with no restart queued.
  - confidence: 0.87
- **[P2] Keep the Windows restart wait budget:** `src/cli/daemon-cli/lifecycle.ts:373-374`
  - body: The refactor removes the current Windows-specific 180s restart wait and computes the wait from the default 60s attempt budget for all platforms. That can make managed Windows restarts fail early compared with current main’s intentional longer timeout.
  - confidence: 0.82
