## Summary

Keep open. Current main has several mitigations for this family, but the Codex app-server path still treats missing `turn/completed` after tool/native progress as an incomplete-turn timeout, and related open reports continue to cover the remaining terminal-state/recovery problem.

## What This Changes

The issue reports a regression where Codex app-server turns after tool-heavy work stop before a terminal `turn/completed`, leaving agent, channel, cron, and hook workflows without reliable final confirmation.

## Best Possible Solution

Maintainers should settle the Codex app-server terminal-state contract and side-effect recovery policy, then prove with live or close-to-live Codex workloads that side-effectful turns either receive `turn/completed` or fail/recover with durable, non-duplicating state.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The source path and current tests do reproduce the core missing-`turn/completed` shape as a timeout/failure, and the issue has multiple field reports across released builds.

## Solution Assessment

No, closing as implemented is not the best resolution. Current main mitigates and classifies the failure, but the best fix still needs maintainer judgment on Codex protocol recovery, side-effect replay safety, and live proof.
