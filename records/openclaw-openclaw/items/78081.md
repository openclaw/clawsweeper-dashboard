## Summary

Keep open: current main still has the silent Teams failure behavior, and this PR is the right owner-local direction, but the branch still has a narrow correctness gap in the new JWKS network-error logging path.

## What This Changes

The PR updates the Microsoft Teams plugin to surface blocked Bot Framework JWKS and Bot Connector network failures via error logging/hints, removes optional `runtime.error` calls, adds targeted tests, and adds a changelog entry.

## Best Possible Solution

Tighten the Teams middleware so every validator-rethrown JWKS network error is logged at `runtime.error`, keep strict 401 behavior, and land after exact-head CI is green.

## Reproduction Assessment

Yes. Source inspection of current main shows JWKS fetch errors are caught in `sdk.ts`, collapsed to `false`, and then logged only at debug level before 401 in `monitor.ts`; the PR tests cover the `ECONNREFUSED` slice but I did not run a live blocked-egress Teams setup.

## Solution Assessment

No, not quite. The Teams-plugin-local approach is the narrow maintainable fix, but the middleware should reuse the validator's network predicate or treat validator rejections as network so wrapped JWKS fetch failures are not downgraded back to debug-only logging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Log all rethrown JWKS network failures visibly:** `extensions/msteams/src/monitor.ts:312-314`
