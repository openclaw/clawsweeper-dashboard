## Summary

Keep open. Current main still has the final-buffer leak path this PR targets, but the current PR head still needs maintainer review before merge because the new suppression provenance is reduced to a process-wide `clientRunId` flag and the branch is far behind the current gateway chat state refactor.

## What This Changes

The PR adds Gateway chat-run suppression provenance state to hide buffered `NO_REPLY` underscore lead fragments from final Control UI chat payloads, adds regression coverage for `NO_` and uppercase `NO`, and adds a changelog entry.

## Best Possible Solution

Land a rebased, narrow Gateway fix that records lead-fragment provenance precisely with the current chat buffer lifecycle, preserves intentional uppercase `NO`, and keeps the broader prefix/suffix hardening discussion in #40165 separate.

## Reproduction Assessment

Yes. Register a chat run, emit one assistant event with `data.text: "NO_"`, then emit lifecycle end; current main suppresses the delta but resolves the final buffer with lead-fragment suppression disabled, making `NO_` eligible as the final chat message.

## Solution Assessment

No. The PR direction is narrow and useful, but this head is not the best landing state until it is rebased and the final suppression decision is scoped to the current buffered fragment or server-owned run lifecycle rather than a bare `clientRunId` flag.
