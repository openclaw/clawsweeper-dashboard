## What This Changes

The PR records streamed Gateway `NO_REPLY` lead-fragment provenance, suppresses matching final Control UI chat payloads, and adds focused regression tests plus a changelog entry.

## Best Possible Solution

Land a current-main Gateway fix that carries streamed lead-fragment provenance into finalization, preserves intentional uppercase `NO`, and includes redacted real Gateway/WebChat proof.

## Reproduction Assessment

Yes, at source level: current main suppresses `NO_` while streaming, stores that buffer, then finalizes with lead-fragment suppression disabled, so lifecycle end at `NO_` can emit a final message. I did not run a live Gateway/WebChat reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug: provenance-based suppression is more maintainable than dropping every uppercase `NO` final reply. The broader mixed prefix/suffix `NO_REPLY` policy belongs in https://github.com/openclaw/openclaw/issues/40165.
