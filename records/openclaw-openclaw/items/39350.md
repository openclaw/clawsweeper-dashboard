## Summary

Keep open: the user-visible Gateway bug is still present on current main, and this PR targets the right surface, but the current head is merge-conflicted and the final-suppression guard is scoped by run-level provenance instead of the current buffered fragment.

## What This Changes

The PR adds Gateway chat state, regression tests, and a changelog entry to suppress streamed `NO_`/`NO_REPLY` lead fragments from final Control UI chat payloads while preserving uppercase `NO` replies.

## Best Possible Solution

Land a rebased Gateway fix that derives final silent-fragment suppression from the current raw/buffered text or a server-owned run lifecycle, keeps coverage for `NO_` suppression and uppercase `NO` preservation, and leaves broader prefix/suffix policy to #40165.

## Reproduction Assessment

Yes, at source level. On current main, an assistant event with `data.text: "NO_"` is buffered before delta suppression returns, then lifecycle finalization resolves that buffer with lead-fragment suppression disabled.

## Solution Assessment

No. The PR's tests and changelog target the right bug, but the landing fix should suppress only when the current buffered/raw text is a silent-token lead fragment, not from a process-wide `clientRunId` flag.
