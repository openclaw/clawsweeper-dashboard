## Summary

Keep open: current main still has a source-inspectable final-event edge for underscore-bearing `NO_REPLY` fragments, but this branch is currently dirty against main and needs maintainer conflict/scope handling before it can land.

## What This Changes

The PR adds per-run provenance state so buffered `NO_REPLY` lead fragments such as `NO_` are suppressed from final gateway chat payloads while uppercase `NO` remains visible, plus regression tests and a changelog entry.

## Best Possible Solution

Rebase the branch onto current main, keep the narrow underscore-provenance final suppression with uppercase `NO` preserved, and leave broader prefix/suffix token policy to #40165.

## Reproduction Assessment

Yes, source-reproducible. On current main, a single assistant event containing `NO_` is suppressed during delta projection but is re-projected for the final event with lead-fragment suppression disabled, so it can become a visible final message.

## Solution Assessment

Yes for the narrow edge. The provenance-set approach avoids the earlier regression where a legitimate uppercase `NO` final reply would be hidden, but the branch must be rebased before it is mergeable.
