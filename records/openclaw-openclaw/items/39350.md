## Summary

Keep open: the current-main Gateway path still has a source-reproducible final-message leak for buffered `NO_`, and the PR’s repaired approach is narrowly targeted, but it still needs rebase/conflict resolution and contributor-provided real behavior proof before merge.

## What This Changes

The PR records streamed `NO_REPLY` lead-fragment suppression in Gateway chat state, uses that provenance during final chat emission, adds regression tests for `NO_` and uppercase `NO`, and updates the changelog.

## Best Possible Solution

Land a rebased Gateway fix that carries streamed lead-fragment provenance into finalization, preserves standalone uppercase `NO`, and leaves broad `NO_REPLY` prefix/suffix policy to https://github.com/openclaw/openclaw/issues/40165.

## Reproduction Assessment

Yes, at source level: current main suppresses `NO_` only when lead-fragment suppression is enabled for streaming/flush, but finalization explicitly calls the projector with `suppressLeadFragments: false`, so a lifecycle ending at `NO_` can emit it as a final message. I did not run a live Gateway/WebChat flow in this read-only review.

## Solution Assessment

Yes for the narrow bug after rebase and proof: carrying streaming lead-fragment provenance into finalization avoids dropping legitimate uppercase `NO`, unlike an unconditional final prefix check. The broader silent-token policy should stay in the linked follow-up issue.
