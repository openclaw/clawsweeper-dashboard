## What This Changes

The PR records suppressed `NO_REPLY` lead-fragment provenance in Gateway chat state, uses it to omit final Control UI chat payloads for buffered `NO_` fragments, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a current-main Gateway fix that suppresses final payloads only when buffered silent-token lead-fragment provenance exists, preserves intentional uppercase `NO`, keeps focused regression tests, and leaves broader mixed-position `NO_REPLY` policy to https://github.com/openclaw/openclaw/issues/40165.

## Reproduction Assessment

Yes. Source inspection shows a current-main path where an assistant `NO_` event is buffered as suppressed during delta handling, then lifecycle finalization reprojects it with lead-fragment suppression disabled and can emit it as final chat text.

## Solution Assessment

Yes for the narrow bug. The latest PR uses streamed underscore-fragment provenance rather than a broad prefix check, which avoids dropping legitimate uppercase `NO` replies and keeps the broader delivery policy out of this patch.
