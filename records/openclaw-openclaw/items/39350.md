## Summary

Keep open. Current main still has the observable final-fragment gap when a chat-linked assistant stream ends at a buffered `NO_` fragment, and this PR targets the right Gateway surface, but the latest head is dirty against main and the current Set-based provenance needs state-scoping or exact-fragment matching before merge.

## What This Changes

The PR adds Gateway chat suppression provenance state, final-message suppression logic, regression coverage, and a changelog entry for buffered `NO_REPLY` lead fragments while preserving intentional uppercase `NO` replies.

## Best Possible Solution

Land a rebased, narrow Gateway fix that records lead-fragment provenance precisely, clears it with the chat buffer lifecycle, and keeps regression coverage for lifecycle-ending `NO_`/`NO_RE`, exact `NO_REPLY`, and intentional uppercase `NO`. Keep the broader suffix/prefix hardening discussion in #40165 separate.

## Reproduction Assessment

Yes. Register a chat run, emit one assistant event with `data.text: "NO_"`, then emit lifecycle end; current main suppresses the delta but finalization reprojects with lead-fragment suppression disabled, so the fragment can become the final chat message.

## Solution Assessment

No. The PR direction is narrow and maintainable, but this head is not the best landing state until it is rebased and the suppression provenance is scoped or compared precisely enough to avoid unrelated final suppression.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve provenance when suppressing final fragments:** `src/gateway/server-chat.ts:715`
  - body: The new `Set` turns lead-fragment provenance into a boolean keyed only by `clientRunId`, and finalization suppresses on `.has(clientRunId)` rather than proving the current buffered text is the suppressed fragment. Keep the stored fragment-text equality check from the first commit or namespace the state by a server-owned run/session identity so unrelated finals cannot disappear through key reuse or stale provenance.
