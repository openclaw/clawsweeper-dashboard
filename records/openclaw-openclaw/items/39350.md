## Summary

Keep open: current main still has a source-level path where a lone buffered `NO_` lead fragment can be emitted as the final chat message, and the latest PR diff is a narrow fix that also preserves legitimate uppercase `NO`. The remaining merge blocker is missing after-fix real behavior proof from a real Gateway/WebChat setup.

## What This Changes

The PR adds Gateway chat-run suppression state for buffered `NO_REPLY` lead fragments, regression tests for `NO_` and uppercase `NO`, and a changelog entry.

## Best Possible Solution

Land a current-main Gateway fix that suppresses final payloads only for proven buffered silent-token lead fragments, preserves intentional uppercase `NO`, and keeps the broader mixed-token policy in the separate follow-up issue.

## Reproduction Assessment

Yes, at source level: feed an assistant event with `data.text: "NO_"`, let delta suppression buffer it, then emit lifecycle end; current main finalizes that buffer with lead-fragment suppression disabled. I did not execute the Vitest harness in this read-only review because `pnpm` is not on PATH here.

## Solution Assessment

Yes for the narrow bug: the latest PR head uses streamed underscore-fragment provenance to suppress `NO_` finals while preserving uppercase `NO`. It should stay scoped away from the broader mixed-token policy tracked at https://github.com/openclaw/openclaw/issues/40165.
