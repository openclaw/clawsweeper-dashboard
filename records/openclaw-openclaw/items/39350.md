## What This Changes

The branch adds Gateway chat state to remember suppressed streamed `NO_REPLY` lead-fragment provenance, uses it to suppress final Control UI chat payloads, adds regression tests for `NO_` and uppercase `NO`, and adds a changelog entry.

## Best Possible Solution

Land a rebased Gateway fix that carries streamed lead-fragment provenance into finalization, preserves intentional uppercase `NO`, and leaves broader silent-token semantics to the dedicated follow-up issue.

## Reproduction Assessment

Yes, at source level: current main suppresses `NO_` while streaming, but finalization reprojects the buffered text with lead-fragment suppression disabled and no provenance state, so it can emit `NO_` as the final chat message. I did not run a live Gateway/WebChat reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug after proof and rebase: carrying streaming lead-fragment provenance into finalization is more maintainable than an unconditional final prefix check because it preserves legitimate standalone uppercase `NO` replies.
