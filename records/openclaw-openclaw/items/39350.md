## Summary

Keep open: current main still has a source-level finalization gap for buffered NO_REPLY lead fragments, and the PR head is a plausible narrow fix, but this external PR is missing real behavior proof and is currently merge-conflicting.

## What This Changes

The branch adds per-run NO_REPLY lead-fragment provenance to Gateway chat state, suppresses matching final chat payloads, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a rebased Gateway fix that carries lead-fragment provenance from streaming into finalization, preserves intentional uppercase NO replies, and leaves broader mixed/suffix policy to https://github.com/openclaw/openclaw/issues/40165.

## Reproduction Assessment

Yes, at source level: current main suppresses NO_ while streaming, but finalization reprojects the buffered text with lead-fragment suppression disabled and has no provenance state to suppress that final payload. I did not run a live Gateway/WebChat reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug after proof and rebase: carrying streaming lead-fragment provenance into finalization is the maintainable fix and preserves legitimate uppercase NO replies. Broader NO_REPLY prefix/suffix semantics should stay in the linked policy issue rather than being folded into this small PR.
