## Summary

Keep open: current main still has the NO_REPLY final-message leak and this PR is aimed at the right Gateway path, but the latest branch suppresses finals from run-level provenance instead of the current buffered fragment, so it needs a narrow branch repair before merge.

## What This Changes

The PR adds Gateway chat-state provenance, regression tests, and a changelog entry to suppress buffered NO_REPLY lead fragments in final Control UI chat payloads while preserving uppercase NO replies.

## Best Possible Solution

Land a small Gateway fix that ties silent lead-fragment provenance to the exact current buffered/raw text or a server-owned run lifecycle, with this PR’s focused regression tests kept separate from the broader #40165 cleanup.

## Reproduction Assessment

Yes, at source level. Register a chat run, emit assistant text `NO_`, then emit lifecycle end; current main buffers the suppressed fragment and finalizes with lead-fragment suppression disabled, so the final chat payload can expose the fragment.

## Solution Assessment

No. The PR direction is narrow and useful, but the latest implementation should suppress based on the current buffered fragment rather than a run-level `clientRunId` flag; broader prefix/suffix token cleanup belongs in #40165.
