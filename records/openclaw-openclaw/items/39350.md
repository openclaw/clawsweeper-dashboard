## Summary

Keep open: current main still has the final-buffer edge where a suppressed `NO_` lead fragment can be emitted as a final chat message, and the PR's latest head is a plausible narrow fix, but it still needs real behavior proof before merge.

## What This Changes

The branch adds Gateway chat-run state for streamed `NO_REPLY` lead-fragment provenance, suppresses matching final chat payloads, preserves uppercase `NO` replies, and adds regression tests plus a changelog entry.

## Best Possible Solution

Land a current-main Gateway fix that suppresses final payloads only for buffered silent-token lead fragments with clear provenance, preserves intentional uppercase `NO`, and keeps the broader mixed-token policy in the separate follow-up thread.

## Reproduction Assessment

Yes, at source level: feed an assistant event with `data.text: "NO_"`, let delta suppression buffer it, then emit lifecycle end; current main resolves the buffer with lead-fragment suppression disabled. I did not execute the Vitest harness because this was a read-only review.

## Solution Assessment

Yes for the narrow bug: the latest branch uses streamed underscore-fragment provenance to suppress `NO_` finals while preserving uppercase `NO`. It should stay scoped to that edge and not absorb the broader prefix/suffix policy work tracked separately.
