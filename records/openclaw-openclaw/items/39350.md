## What This Changes

The PR records suppressed Gateway `NO_REPLY` lead-fragment provenance, uses it to suppress matching final chat payloads, adds regression tests for `NO_` and uppercase `NO`, and adds a changelog entry.

## Best Possible Solution

Land a current-main Gateway fix that carries streamed lead-fragment provenance into finalization, preserves legitimate uppercase `NO` replies, and includes redacted real Gateway/WebChat proof.

## Reproduction Assessment

Yes, at source level. Current main suppresses `NO_` during streaming, stores that buffer, then finalizes with lead-fragment suppression disabled, so lifecycle end at `NO_` can emit a visible final chat message.

## Solution Assessment

Yes for the narrow bug. Carrying provenance from the streaming suppression path into finalization is more precise than broad prefix suppression and preserves legitimate uppercase `NO`, but the branch still needs real proof and reconciliation with current main.
