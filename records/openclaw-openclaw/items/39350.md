## Summary

Keep open: the underlying WebChat `NO_REPLY` fragment leak is still source-reproducible on current main, but this PR head should not merge as-is because the remaining suppression provenance is stored as run-level state instead of being tied to the current buffered fragment.

## What This Changes

The PR adds Gateway chat-run suppression provenance, regression tests, and a changelog entry so buffered `NO_REPLY` lead fragments are hidden from final Control UI chat payloads while uppercase `NO` remains visible.

## Best Possible Solution

Land a rebased Gateway fix that scopes silent lead-fragment provenance to the current raw/buffered text or a server-owned run lifecycle, keeps regression coverage for `NO_` and uppercase `NO`, and leaves broader prefix/suffix hardening to #40165.

## Reproduction Assessment

Yes, at source level. Register a chat run, emit one assistant event with `data.text: "NO_"`, then emit lifecycle end; current main buffers the suppressed fragment but finalization resolves it with lead-fragment suppression disabled.

## Solution Assessment

No. The PR direction is narrow and useful, but the landing fix should derive final suppression from the current buffered fragment or a server-owned run lifecycle rather than a process-wide `clientRunId` flag.
