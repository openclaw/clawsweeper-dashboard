## Summary

Keep open. Current main and v2026.5.19 still swallow QueuedFileWriter mkdir/append failures without any warning path, and https://github.com/openclaw/openclaw/pull/57089 remains an open closing PR for the same work.

## What This Changes

QueuedFileWriter should keep best-effort diagnostic writes but emit a rate-limited signal after repeated write failures instead of swallowing every failure silently.

## Best Possible Solution

Keep best-effort non-crashing diagnostic writes, but add rate-limited failure reporting in the current safeAppendFile/flush writer path with focused coverage for threshold, repeat cadence, and reset-on-success behavior.

## Reproduction Assessment

Yes. Source inspection shows that a mkdir or appendRegularFile failure flows into `.catch(() => undefined)` and `flush()` resolves without any warning path.

## Solution Assessment

Yes in direction, with a boundary caveat. Rate-limited reporting is the maintainable fix, but it should be applied to the current safe writer implementation and preserve filesystem-safety behavior.
