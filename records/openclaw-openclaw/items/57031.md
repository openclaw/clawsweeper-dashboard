Keep open. Current main and the latest release still swallow queued writer mkdir/append failures; the remaining work is a low-priority diagnostic visibility decision with an open fork PR already proposing the warning path.

## What This Changes

Request to make QueuedFileWriter surface repeated diagnostic write failures instead of silently absorbing mkdir/append errors.

## Best Possible Solution

Land or adopt a narrow fix that preserves non-fatal diagnostic writes while adding bounded, non-secret warning visibility for repeated failures, or explicitly decide that silent lossy diagnostic queues remain intentional.

## Solution Assessment

Unclear. A repeated-failure warning is a plausible small fix, but maintainers still need to decide the logging threshold, destination, and acceptable warning noise for optional diagnostics.
