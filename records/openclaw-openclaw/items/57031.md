## Summary

Keep open. Current main and the latest release still suppress QueuedFileWriter setup/append failures without warning, and the open closing PR remains the implementation candidate; the reporter narrowed this to a low-severity diagnostic signal improvement rather than user data loss.

## What This Changes

Request adding operator-visible, rate-limited reporting when QueuedFileWriter diagnostic writes repeatedly fail.

## Best Possible Solution

Keep best-effort, non-crashing diagnostic writes, but add rate-limited failure reporting in the current safeAppendFile/flush writer path with focused coverage for threshold, periodic repeat, and reset-on-success behavior.

## Reproduction Assessment

Yes, from source inspection: point the current writer at a path that makes mkdir or safeAppendFile fail, enqueue writes, and await flush; current main catches the failure without any warning path. I did not run a live disk-full scenario because this review is read-only.

## Solution Assessment

Partly: rate-limited reporting is the right boundary, but it should be ported onto the current safe writer implementation and reviewed for log-noise behavior across all diagnostic sinks.
