## Summary

Keep open. Current main and the latest release still absorb queued diagnostic writer mkdir/append failures without a warning, and the remaining change is a low-priority diagnostic logging policy decision rather than an already-implemented fix.

## What This Changes

The issue requests a bounded warning or similar operator-visible signal when `QueuedFileWriter` experiences consecutive write failures for diagnostic JSONL sinks.

## Best Possible Solution

Keep diagnostic writes non-fatal, but add a maintainer-approved bounded warning or debug signal with threshold/reset tests and no sensitive path or payload leakage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: force `safeAppendFile` or the initial directory creation to fail and the queued writer resolves through `.catch(() => undefined)` without warning. I did not execute a filesystem repro because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. A warning after consecutive failures is a plausible narrow fix, but the best shape needs maintainer policy on warning level, threshold, rate limiting, and whether file paths should be omitted or redacted.
