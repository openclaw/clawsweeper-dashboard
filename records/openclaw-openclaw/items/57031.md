## Summary

Keep open. Current main still suppresses QueuedFileWriter setup and append failures without a warning or failure counter, and https://github.com/openclaw/openclaw/pull/57089 is an open closing-syntax implementation candidate; the reporter narrowed the impact to a low-severity diagnostic signal improvement rather than user data loss.

## What This Changes

Add diagnostic failure reporting when QueuedFileWriter setup or append writes fail instead of silently suppressing every error.

## Best Possible Solution

Keep best-effort, non-crashing diagnostic writes, but add rate-limited failure reporting in the current safe writer path with focused coverage for threshold, repeat warning, and reset-on-success behavior.

## Reproduction Assessment

Yes from source inspection: point the current writer at a path that fails setup or append, enqueue writes, and await `flush()`; current main catches the failure without any warning path. I did not run a live disk-full reproduction because this review is read-only.

## Solution Assessment

Unclear as stated: thresholded or rate-limited reporting is the right boundary, but it should be ported onto the current safeAppendFile/flush implementation and reviewed for log-noise behavior across diagnostic sinks.
