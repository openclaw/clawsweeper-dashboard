## Summary

Keep open. Current main and the latest release still suppress QueuedFileWriter setup and append failures without a warning path, and the open closing PR remains the implementation candidate; the reporter narrowed this to a low-severity diagnostic signal improvement rather than user data loss.

## What This Changes

Request is to make QueuedFileWriter report repeated diagnostic write failures instead of silently swallowing them.

## Best Possible Solution

Keep best-effort non-crashing diagnostic writes, but add rate-limited failure reporting in the current safeAppendFile/flush writer path with focused coverage for warning threshold, periodic repeat, and reset-on-success behavior.

## Reproduction Assessment

Yes. From source inspection, point the current writer at a path that makes mkdir or safeAppendFile fail, enqueue writes, and await flush; current main catches the failure without any warning path.

## Solution Assessment

Yes, with a boundary caveat. Rate-limited reporting is the maintainable fix, but it should be applied to the current safe writer implementation rather than restoring older direct append behavior.
