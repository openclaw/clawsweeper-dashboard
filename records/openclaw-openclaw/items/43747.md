## Summary

Keep open. Current main and the latest release still expose separate builtin SQLite, QMD, pre-compaction flush, and single memory-plugin-slot surfaces; the linked multi-slot PR is open and related, but it is not merged and does not fully settle the builtin/QMD/flush/per-workspace contract requested here.

## What This Changes

The issue requests a coherent memory-management contract so builtin SQLite memory, QMD Markdown memory, mixed mode, and per-workspace behavior are predictable and configurable.

## Best Possible Solution

Define one maintainer-approved memory contract that reconciles backend selection, flush behavior, plugin-role composition, storage defaults, docs, and doctor migration before implementation.

## Reproduction Assessment

Partly yes: source and docs show the fragmented current surfaces, including builtin/QMD backend selection, separate flush config, per-agent SQLite storage, and a single memory plugin slot. No high-confidence live reproduction proves the exact three-user divergence end to end.

## Solution Assessment

No: a raw `memory.type` switch is only one plausible API. The best solution needs maintainer product/API design across config, plugin roles, migration, docs, and current open memory-role work.
