## Summary

Keep open. Current main and v2026.6.6 still expose separate builtin SQLite, QMD, memorySearch, compaction memory-flush, and single memory-plugin slot surfaces; the open memory role-slot PR overlaps part of the problem but does not resolve the full unified memory-mode/product contract.

## What This Changes

The issue requests a stricter unified memory-management contract so builtin SQLite, QMD daily Markdown, disabled memory, workspace scoping, and mixed memory behavior are predictable across users.

## Best Possible Solution

Define a maintainer-approved memory mode or role contract that reconciles backend selection, memorySearch, QMD, compaction flush, plugin ownership, workspace scoping, docs, and doctor migration before changing runtime behavior.

## Reproduction Assessment

Yes at source level: current main still exposes separate `memory.backend`, `memory.qmd.*`, `agents.defaults.memorySearch.*`, `plugins.slots.memory`, and compaction memory-flush surfaces. I did not run a live three-user reproduction.

## Solution Assessment

Unclear. The proposed `memory.type` switch is plausible, but the best fix needs maintainer API direction and should be reconciled with the open memory role-slot work before implementation.
