## Summary

Keep open for maintainer product direction: current main and the latest release still expose separate builtin SQLite, QMD, memory search, and compaction memory-flush controls rather than the requested unified memory contract and storage semantics.

## What This Changes

The issue requests a unified memory-management contract so builtin SQLite, QMD markdown memory, mixed behavior, and workspace or agent storage behave predictably across users.

## Best Possible Solution

Maintainers should define the public memory contract first, then align schema/help/docs, backend resolution, storage placement, migration or doctor guidance, diagnostics, and regression tests around that contract.

## Reproduction Assessment

Source-level, yes: current main lacks `memory.type` and keeps backend, QMD, memorySearch, and memoryFlush controls split. Live, no: the exact three-user mismatch lacks each user's config and logs.

## Solution Assessment

No automated one-off patch is the best way to solve this yet. The proposed modes may be useful, but maintainers need to decide the public config contract, migration story, storage placement, and mixed-backend semantics before implementation.
