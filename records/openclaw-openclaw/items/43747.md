## Summary

Keep open. Current main still splits memory behavior across backend, search, QMD, plugin-slot, and compaction-flush controls, so the stricter public memory-mode contract requested in the discussion is not implemented or superseded.

## What This Changes

The issue requests a unified memory-management contract, including a `memory.type`-style mode for builtin-only, QMD-only, and mixed behavior plus clearer workspace/agent storage semantics.

## Best Possible Solution

Define one maintainer-owned public memory-mode contract that aligns config schema, generated metadata, docs, runtime backend selection, plugin-slot disablement, QMD fallback/indexing, compaction daily Markdown flush behavior, and workspace/agent storage scope while preserving compatibility for existing `memory.backend` users.

## Reproduction Assessment

No for the original three-user divergence because the report does not include each user's config, logs, plugin-slot state, QMD state, or embedding-provider state. Yes for the contract gap: static inspection of current main shows no `memory.type` enum and the relevant controls remain split.

## Solution Assessment

Unclear. A unified memory mode may be the right product direction, but the proposed `built-in-only`/`qmd-only`/`mixed` shape needs maintainer design because it must settle migration, fallback, disabled-memory, plugin-slot, compaction-flush, and workspace SQLite semantics together.
