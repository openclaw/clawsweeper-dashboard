## Summary

Keep open. Current main and v2026.6.6 still expose separate global backend, QMD, Markdown flush, memory-search, and single memory-plugin-slot surfaces, while the requested coherent memory contract remains a product/config design decision and the related multi-slot PR is open but not merged.

## What This Changes

The issue asks OpenClaw to define one consistent memory-management contract across builtin SQLite memory, QMD/Markdown memory, disabled memory, and workspace or agent scoping.

## Best Possible Solution

Keep this open until maintainers either fold it explicitly into the canonical memory architecture work or define a separate coherent contract for backend selection, role composition, workspace/agent storage, Markdown flush behavior, docs, and doctor migration.

## Reproduction Assessment

No high-confidence live reproduction of the three reported machines was run. Source inspection does give a high-confidence way to see the current split: global `memory.backend`, single `plugins.slots.memory`, per-agent `memorySearch`, builtin SQLite store paths, and Markdown memory flush all remain separate surfaces on current main.

## Solution Assessment

Unclear whether the proposed `memory.type` shape is the best final API. A maintainable solution should come from the memory architecture decision, likely coordinating role slots, durable storage, workspace/agent scoping, migration, and docs rather than adding one ad hoc config key.
