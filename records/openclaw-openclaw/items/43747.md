## Summary

Keep open. Current main still has separate `memory.backend`, `agents.*.memorySearch`, and `agents.defaults.compaction.memoryFlush` controls, but no unified `memory.type`/mixed-mode contract or single strict memory-management definition; this is a product/config design request rather than a reproducible regression.

## What This Changes

Request to replace scattered memory backend, search, and flush behavior with a unified memory type/config contract and consistent storage semantics across users and workspaces.

## Best Possible Solution

Define a backward-compatible memory configuration design that documents current behavior and, if approved, maps a unified mode to builtin, QMD, mixed, disablement, storage paths, and migration/doctor behavior.

## Reproduction Assessment

No high-confidence bug reproduction is available or really applicable. The reporter supplied three observed setups and explicitly asked for a rewrite/strict definition; source inspection confirms separate current config surfaces but not a violation of an existing contract.

## Solution Assessment

Unclear. A unified memory mode may be a good product direction, but the maintainable fix needs maintainer scoping across backend selection, daily Markdown writes, SQLite store paths, active memory, docs, and compatibility rather than a narrow bug patch.
