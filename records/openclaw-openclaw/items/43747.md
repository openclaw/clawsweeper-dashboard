## Summary

Keep open. Current main and latest v2026.6.6 still expose separate builtin SQLite, QMD, memorySearch, single memory-plugin slot, and compaction memory-flush surfaces; the related memory-role PR is open but unmerged, so this remains a maintainer product/API decision.

## What This Changes

The issue requests one clear memory-management contract so builtin SQLite memory, QMD Markdown memory, and disabled or misconfigured memory do not diverge across users.

## Best Possible Solution

Define a maintainer-approved memory mode or role contract that reconciles backend selection, memorySearch, QMD, compaction flush, plugin ownership, workspace scoping, docs, and doctor migration before changing runtime behavior.

## Reproduction Assessment

Yes at source level: current main still exposes separate `memory.backend`, `memory.qmd.*`, `agents.defaults.memorySearch.*`, `plugins.slots.memory`, and compaction memory-flush surfaces. I did not run a live three-user reproduction.

## Solution Assessment

No final solution is selected yet. The proposed `memory.type` switch is plausible, but the best fix needs maintainer API direction and should be reconciled with the open memory-role PR before implementation.
