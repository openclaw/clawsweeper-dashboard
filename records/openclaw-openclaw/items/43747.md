## Summary

Keep this open as a maintainer product-design issue, not as a reproduced regression: current behavior intentionally separates canonical Markdown memory from derived search indexes, but VISION.md still calls for convergence on one recommended memory path and no viable canonical issue or PR owns that decision.

## What This Changes

Define a coherent, predictable memory experience across Markdown capture, built-in SQLite indexing, QMD retrieval, and agent or workspace scoping.

## Best Possible Solution

Retitle and narrow this into the canonical memory-convergence discussion: define one recommended default experience, clearly distinguish capture from indexing, expose effective per-agent configuration and diagnostics, and split any approved config or migration work into focused follow-ups.

## Reproduction Assessment

No, there is no high-confidence regression reproduction because the three installations were not shown to use matching versions and effective configuration. Source inspection does confirm that Markdown capture and SQLite or QMD indexing can intentionally coexist.

## Solution Assessment

No, the proposed `memory.type` enum is not yet the best solution because it treats canonical files and derived indexes as mutually exclusive backends, while current releases already provide backend selection and per-agent storage.
