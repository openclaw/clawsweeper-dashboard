## Summary

Keep open. Current main and the latest release still expose separate SQLite builtin memory, QMD backend, memorySearch, compaction memory-flush, and memory-plugin slot controls; the open memory role-slot work overlaps one part of the problem but is not merged and does not settle the unified memory-mode, workspace-scoping, and upgrade contract requested here.

## What This Changes

The issue asks for a unified memory-management contract because installations can currently use SQLite memory, QMD Markdown files, or no memory depending on configuration and setup.

## Best Possible Solution

Define a maintainer-approved memory mode or role contract that reconciles backend selection, memorySearch, QMD, compaction flush, plugin ownership, workspace scoping, docs, and doctor migration before changing runtime behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 still expose separate `memory.backend`, `memory.qmd.*`, `agents.defaults.memorySearch.*`, `plugins.slots.memory`, and compaction memory-flush surfaces. I did not run a live three-user setup.

## Solution Assessment

Unclear. The requested `memory.type` switch is plausible, but the best fix needs a maintainer-approved contract across backend selection, memorySearch, QMD, compaction flush, workspace storage, docs, and doctor migration.
