## Summary

Keep open. Current main and v2026.6.6 still expose separate builtin SQLite, QMD, pre-compaction memory-flush, memorySearch, and single memory-slot surfaces; the open memory-role PR is related but does not fully settle the builtin/QMD/flush/per-workspace contract requested here.

## What This Changes

Request to replace fragmented builtin/QMD/flush memory behavior with a strict unified memory contract and mode-style configuration.

## Best Possible Solution

Define one maintainer-approved memory contract that reconciles backend selection, flush behavior, plugin-role composition, storage defaults, docs, and doctor migration before implementation.

## Reproduction Assessment

No high-confidence live reproduction proves the exact three-user divergence. Source and docs do show the fragmented current surfaces: builtin/QMD backend selection, separate memorySearch settings, pre-compaction Markdown flush, and a singular memory plugin slot.

## Solution Assessment

No, the proposed `memory.type` switch is only one plausible API. The best fix needs a maintainer-approved memory contract across config, plugin roles, storage scope, doctor migration, and docs.
