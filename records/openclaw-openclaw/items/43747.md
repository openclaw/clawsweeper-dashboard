## Summary

Keep open. Current main has clearer docs and substantial memory work, but it still intentionally exposes separate builtin SQLite indexing, QMD backend selection, `memorySearch`, and pre-compaction memory flush controls; the request for one strict memory contract and a `memory.type`-style mode remains a product/config design item, not an already-fixed bug.

## What This Changes

The issue asks for a stricter, unified memory management contract so different users do not see SQLite memory, daily Markdown memory, or no memory depending on configuration.

## Best Possible Solution

Keep this as maintainer-scoped memory UX work: decide the public memory contract and whether to unify backend, write, and flush behavior behind a clearer config model while coordinating with the narrower memory CRUD work.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection confirms the confusion comes from multiple intentional memory surfaces and configuration paths, while the original report does not include the configs needed to reproduce the three users’ exact states.

## Solution Assessment

No, not as a one-shot bug rewrite. A unified `memory.type` may be a useful design direction, but the maintainable path needs maintainer scoping across backend selection, Markdown write policy, SQLite location, QMD fallback, and memory tool permissions.
