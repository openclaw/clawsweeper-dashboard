## Summary

Keep open. Current main still exposes separate memory backend, memory-search, QMD, plugin-slot, and compaction-flush controls, but it does not implement the unified public memory mode contract requested in the discussion.

## What This Changes

The request is to define one strict memory-management contract so builtin SQLite, QMD daily Markdown, mixed behavior, disabled memory, and workspace/agent storage behave predictably across users.

## Best Possible Solution

Define a maintainer-owned public memory-mode contract that aligns config schema, generated metadata, docs, runtime backend selection, plugin-slot behavior, QMD fallback/indexing, compaction flush, and storage scope while preserving compatibility for existing memory configs.

## Reproduction Assessment

No for the original three-user divergence because the report lacks each user's config, logs, plugin-slot state, QMD state, and embedding-provider state. Yes for the contract gap: current source and docs have no `memory.type` enum and still split the relevant controls.

## Solution Assessment

Unclear. A unified memory mode may be the right direction, but the proposed `built-in-only`/`qmd-only`/`mixed` shape needs product design before implementation because it spans migration, fallback, disabled-memory, plugin-slot, compaction-flush, and workspace SQLite semantics.
