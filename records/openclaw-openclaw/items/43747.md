## Summary

Keep open. Current main documents and implements separate memory surfaces, but it does not implement the requested unified `memory.type` contract or a single strict mode that coordinates SQLite indexing, QMD daily Markdown, disabled memory, and workspace scoping; the related memory PRs/proposal in the context are closed, not canonical open follow-ups.

## What This Changes

The issue requests a stricter unified memory-management definition so builtin SQLite memory, QMD daily Markdown memory, and disabled memory behave consistently across users.

## Best Possible Solution

Maintainers should decide the intended memory contract first: either add a unified mode/config that coordinates backend, flush, search, and store scoping, or explicitly keep the current split and improve diagnostics/docs around those independent knobs.

## Reproduction Assessment

No. There is no high-confidence current-main failing command or log for the three installations; source inspection confirms parallel configurable memory surfaces, but that is expected behavior rather than a reproduced regression.

## Solution Assessment

Unclear. The proposed `memory.type` could be a useful UX direction, but the best implementation needs maintainer design because it crosses memory plugin selection, search backend behavior, compaction flush writes, and storage scoping.
