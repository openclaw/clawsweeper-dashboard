## Summary

Keep open for maintainer product direction: current main still exposes separate builtin SQLite search, QMD backend configuration, memorySearch scoping, and compaction Markdown flush controls instead of the requested unified memory-mode and storage contract.

## What This Changes

The issue requests a strict, unified memory management contract so builtin SQLite, QMD Markdown, mixed storage, and disabled/misconfigured memory do not behave inconsistently across users.

## Best Possible Solution

Maintainers should define the public memory contract first, then align schema/help/docs, doctor/status diagnostics, backend resolution, storage placement, migration behavior, and regression tests around that contract.

## Reproduction Assessment

No high-confidence live repro exists for the three-user mismatch because configs and logs are missing. Source inspection does reproduce the design gap: current main has separate `memory.backend`, `memorySearch`, QMD, and `memoryFlush` controls rather than a unified `memory.type` contract.

## Solution Assessment

No automated one-off patch is the best path yet. The proposed `memory.type` modes may be useful, but public config, storage placement, migrations, fallback behavior, and mixed-backend semantics need maintainer product direction first.
