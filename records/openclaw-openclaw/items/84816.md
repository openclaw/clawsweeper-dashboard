## Label Justifications

- `P1`: The PR targets a broken Discord `/models` workflow and introduces auth-provider behavior that could affect real channel users immediately.
- `merge-risk: 🚨 auth-provider`: The proposed global auth map can override caller-specific discovery flags and change which providers appear authenticated.
- `merge-risk: 🚨 availability`: The change moves a long auth-discovery sweep into gateway startup/background runtime, which could affect early interaction responsiveness if not proven.

## Summary

Keep open: the performance direction is useful and the PR includes real Discord timing proof, but the patch currently lets a global warmed auth map override caller-specific auth-discovery settings and does not actually rewarm after config reload.

## What This Changes

The PR adds a gateway-startup provider-auth prewarm path and uses the prepared map to speed up `/models` and other model-listing calls.

## Best Possible Solution

Keep the prewarm design, but scope prepared auth answers to the same auth context the caller requested and add a reload/auth-change lifecycle that rebuilds the map with tests for read-only gateway lists and config reloads.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: warm the provider map with broad discovery, then call a gateway-style model list with `runtimeAuthDiscovery: false`; the prepared answer returns before the false discovery flags can take effect.

## Solution Assessment

No. Prewarming the canonical provider auth facts is the right direction, but the current implementation is not the best fix until it preserves caller auth scope and has a real reload rewarm lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve auth-discovery scope when using warmed answers:** `src/agents/model-provider-auth.ts:38-40`
  - body: `models.list` intentionally calls the visibility helper with `runtimeAuthDiscovery: false`, which flows into `allowPluginSyntheticAuth` and `discoverExternalCliAuth`. This new early return ignores those caller options and returns the startup-warmed answer, which was built with external CLI and plugin synthetic auth enabled, so read-only gateway lists can expose providers they previously filtered out and agent/env-specific calls can get the wrong auth result. Please gate or key the prepared map by the requested auth context before consulting it.
  - confidence: 0.91
- **[P2] Rewarm the provider auth map after reload invalidation:** `src/gateway/server-reload-handlers.ts:318`
  - body: The reload handler now clears the prepared auth state, but the branch only calls `warmCurrentProviderAuthState` from startup. After a relevant config reload, the map stays null and later `/models` calls fall back to the old per-call discovery path until restart, so the advertised reload lifecycle does not hold. Schedule a rewarm with the new config, or make the next read explicitly rebuild the prepared state.
  - confidence: 0.87
