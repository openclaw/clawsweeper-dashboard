## Summary

This PR should stay open: the performance direction is useful and the live Discord timing proof is strong, but the latest head still lets model listing fall back to the old slow provider-auth sweep after a 10s TTL and during post-ready warm windows.

## What This Changes

This PR adds a prepared provider-auth snapshot warmed at gateway startup, hot reload, and auth logout so model-listing auth checks can short-circuit per-provider discovery.

## Best Possible Solution

Keep provider-auth availability as a lifecycle-owned runtime fact that never sends user-facing model-listing calls back to synchronous discovery; refresh it asynchronously or invalidate it from known auth/config/plugin mutations with regression coverage for startup, reload, logout, and expiry/freshness behavior.

## Reproduction Assessment

Yes for the PR blockers by source inspection: the branch test itself proves fallback after 10,001ms, and startup code schedules the warm after `gateway ready`. I did not run tests because this review is read-only.

## Solution Assessment

No. Pre-warming provider auth is the right direction, but the current implementation still exposes synchronous discovery on user-facing model-listing paths; a stale-while-refresh or explicit-invalidation design is safer.

  - body: After this 10s TTL expires, `matchesWarmedScope` becomes false and each later `/models` request can fall back to `hasRuntimeAvailableProviderAuth` plus `ensureAuthProfileStore` instead of the prepared map.   - confidence: 0.92

Either publish/complete a usable snapshot before channels accept interactions, or make the picker ack/defer and use a nonblocking pending refresh path.
