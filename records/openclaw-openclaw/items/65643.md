## Summary

Keep open. Current main still preserves inherited `OPENCLAW_LAUNCHD_LABEL` after explicit profile selection, and upstream PR #66000 is the active unmerged implementation candidate that directly closes this issue.

## What This Changes

Ensure explicit `--profile` gateway status invocations inspect the selected profile plist instead of a stale inherited launchd label.

## Best Possible Solution

Review and land PR #66000 or an equivalent narrow fix that makes explicit CLI profile selection take precedence over stale inherited launchd labels for gateway status, while leaving the broader restart behavior tracked separately in #47555.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: set `OPENCLAW_LAUNCHD_LABEL` to one profile label, run `openclaw --profile <other> gateway status`, and the status path still resolves the plist through the inherited label first.

## Solution Assessment

Yes. Fixing the CLI profile-env boundary is the narrowest maintainable direction because it preserves intentional launchd label overrides elsewhere while preventing stale gateway-shell labels from defeating explicit `--profile`.
