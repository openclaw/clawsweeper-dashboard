## Summary

Keep open. Current `main` still preserves a stale inherited `OPENCLAW_LAUNCHD_LABEL` after explicit profile selection, and launchd plist resolution still checks that label before `OPENCLAW_PROFILE`; PR #66000 is open and unmerged as the active implementation candidate for this issue.

## What This Changes

This issue asks for `openclaw --profile <name> gateway status` to resolve the profile-specific launchd plist instead of an inherited gateway process label.

## Best Possible Solution

Review and land PR #66000 or an equivalent narrow fix that makes explicit CLI profile selection override stale inherited launchd labels for gateway status, while leaving broader restart behavior to #47555 unless deliberately included.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: set `OPENCLAW_LAUNCHD_LABEL` to one profile label, run `openclaw --profile <other> gateway status`, and current `main` still resolves the plist through the inherited label first.

## Solution Assessment

Yes. Fixing the CLI profile-env boundary is the narrow maintainable direction because it prevents stale gateway-shell labels from defeating explicit `--profile` without removing intentional launchd label support elsewhere.
