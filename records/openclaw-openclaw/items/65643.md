## Summary

Keep open. Current main still leaves an inherited `OPENCLAW_LAUNCHD_LABEL` in place after an explicit CLI profile is applied, and launchd status resolution still prioritizes that label over `OPENCLAW_PROFILE`; PR #66000 is the open linked implementation candidate and should be reviewed or landed before this issue closes.

## What This Changes

The issue requests that explicit `--profile` CLI selection override or clear a stale inherited `OPENCLAW_LAUNCHD_LABEL` so `gateway status` resolves the profile-specific macOS LaunchAgent plist.

## Best Possible Solution

Land PR #66000 or an equivalent narrow fix that makes explicit CLI profile selection take precedence over stale OpenClaw-managed launchd labels for gateway status, with focused tests for conflicting labels, matching labels, custom labels, and plist resolution. Keep broader restart-command behavior tracked separately by #47555 unless the same patch intentionally covers it.

## Reproduction Assessment

Yes. The source path is enough to reproduce logically: set `OPENCLAW_LAUNCHD_LABEL` to one profile label, run `openclaw --profile <other> gateway status`, and current main applies the profile while leaving the label that launchd plist resolution reads first.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because it changes the CLI profile-env boundary and adds targeted coverage, instead of weakening launchd's generic support for intentional label overrides everywhere.
