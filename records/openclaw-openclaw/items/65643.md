## Summary

Keep open. Current main still leaves an inherited `OPENCLAW_LAUNCHD_LABEL` in place after explicit `--profile`, and the status path resolves the LaunchAgent plist through that label first; https://github.com/openclaw/openclaw/pull/66000 is an open closing implementation candidate, so the issue should stay open until that PR or an equivalent fix merges.

## What This Changes

Make explicit `openclaw --profile <name> gateway status` resolve the selected profile's LaunchAgent plist instead of a stale inherited `OPENCLAW_LAUNCHD_LABEL`.

## Best Possible Solution

Land or replace the focused PR so explicit CLI profile selection wins over stale OpenClaw-managed service labels while preserving intentional custom label overrides outside root-profile commands.

## Reproduction Assessment

Yes, at source level. Set `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.batch`, run a root `--profile interactive` gateway status path, and current main leaves the label intact before `resolveLaunchAgentLabel()` chooses it over `OPENCLAW_PROFILE`.

## Solution Assessment

Yes. Fixing the CLI profile-env boundary is the narrowest maintainable solution because `--profile` is parsed before command routing and can prevent stale managed launchd labels from reaching status/plist resolution.
