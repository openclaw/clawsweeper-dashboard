## What This Changes

The issue asks for `openclaw --profile <name> gateway status` to ignore or clear a stale inherited launchd label so it targets the selected profile's plist.

## Best Possible Solution

Land or replace the focused open PR so explicit CLI profile selection wins over stale OpenClaw-managed service labels while preserving intentional custom launchd overrides.

## Reproduction Assessment

Yes, at source level. With `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.batch` and an explicit root `--profile interactive gateway status` path, current main leaves the stale label in `process.env` before launchd resolution chooses it over `OPENCLAW_PROFILE`.

## Solution Assessment

Yes. Fixing the CLI profile-env boundary is the narrowest maintainable direction because `--profile` is parsed before gateway status resolves the LaunchAgent plist, while global launchd-label precedence can still preserve intentional custom overrides.
