## Summary

Keep open. Current main still preserves an inherited `OPENCLAW_LAUNCHD_LABEL` after explicit profile selection, and the launchd status path still reads that label before deriving the profile label; linked PR #66000 remains the active unmerged implementation candidate.

## What This Changes

The issue asks for `openclaw --profile <name> gateway status` to inspect the selected profile’s launchd plist instead of a stale launchd label inherited from the caller environment.

## Best Possible Solution

Review and land PR #66000 or an equivalent narrow fix that makes explicit CLI profile selection take precedence over stale inherited launchd labels for gateway status, with the broader restart behavior kept in #47555 unless deliberately covered.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: set `OPENCLAW_LAUNCHD_LABEL` to one profile label, run `openclaw --profile <other> gateway status`, and the status path still resolves the plist through the inherited label first.

## Solution Assessment

Yes. Fixing the CLI profile-env boundary is the narrow maintainable direction because it preserves intentional launchd label overrides elsewhere while preventing stale gateway-shell labels from defeating explicit `--profile`.
