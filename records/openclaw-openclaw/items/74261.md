## Summary

Keep open. Current main still has no WhatsApp create-group agent tool or live runtime method, even though the pinned Baileys API exposes `groupCreate`; the request would create real groups and invite phone-number participants through the operator's linked WhatsApp account, and the latest discussion adds credible account-abuse risk. This needs maintainer product/security approval rather than automatic cleanup or a ClawHub redirect, because a standalone tool plugin does not currently have a safe public surface for the existing linked WhatsApp session's group-creation operation.

## What This Changes

Add a WhatsApp-owned `whatsapp_create_group` agent tool that accepts a group name and participant phone numbers, then creates a WhatsApp group through the linked Baileys session.

## Best Possible Solution

If maintainers approve the tradeoff, implement this inside the WhatsApp plugin as an explicitly gated, WhatsApp-owned capability: normalize participants to WhatsApp JIDs, call `groupCreate` through an additive live-runtime method, require an owner or trusted-requester policy, redact identifiers in logs, record outbound activity, document account-abuse limits, and add focused unit/runtime tests. If maintainers reject agent-created group invites, document that limitation in the WhatsApp channel docs.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence because source and docs inspection shows login, send, poll, reaction, typing, media, and group-reply handling only, with no group-create path.

## Solution Assessment

Unclear until maintainer product/security review. If approved, a WhatsApp-owned gated tool plus additive live-runtime seam is the narrow maintainable solution; a core-generic helper or standalone ClawHub tool would not safely access the existing linked WhatsApp session's group-creation operation today.
