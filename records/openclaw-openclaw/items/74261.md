## Summary

Keep open. Current main still has no WhatsApp group-creation agent tool or live runtime method, and the discussion adds credible account-abuse risk around repeated Baileys group creation, so this needs explicit maintainer product/security review rather than cleanup closure or an automated fix.

## What This Changes

The issue requests a `whatsapp_create_group` agent tool with `groupName` and participant phone-number parameters, backed by Baileys `groupCreate`.

## Best Possible Solution

If maintainers approve the tradeoff, implement this inside the WhatsApp plugin as an explicitly gated WhatsApp-owned capability: normalize participant numbers to WhatsApp JIDs, call group creation through an additive live-runtime method, restrict who can invoke it, redact identifiers in logs, record outbound activity, document account-abuse limits, and add focused tests. If they reject agent-created group invites, document that limitation in the WhatsApp docs.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high-confidence because source and docs inspection shows login, send, poll, reaction, typing, media, and group-message handling only, with no group-create path.

## Solution Assessment

Unclear until maintainer product/security review. A direct always-available Baileys-backed tool is not clearly the safest solution given the abuse-risk comment; a WhatsApp-owned gated runtime/tool surface, or documenting the limitation, would be the maintainable path.
