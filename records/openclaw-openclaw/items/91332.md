## Summary

Keep open, but this PR is not merge-ready: the fallback cleanup is not already on current main, yet the PID-only path deletion can remove a same-process lock acquired after the pre-read no-holder check, and the external real-behavior proof is still CI/code-review only.

## What This Changes

The PR wraps `acquireSessionWriteLock` release to best-effort remove a lingering `.lock` file after release when no in-process holder remains and the lock payload PID matches the current process.

## Best Possible Solution

Make the cleanup ownership-safe with a snapshot/inode-aware removal path or an immediate ownership recheck before deletion, add focused regression coverage, and require redacted runtime or synthetic after-fix proof before merge.

## Reproduction Assessment

Yes, source-level: fs-safe can leave a lock file behind on snapshot mismatch, and the PR's active-lock deletion race is visible from the awaited payload read between the holder check and path deletion. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix must preserve mutual exclusion with atomic ownership proof or a snapshot/inode-aware deletion path, plus regression coverage and real behavior proof.

- **[P1] Recheck ownership before removing the leftover lock:** `src/agents/session-write-lock.ts:998-1000`
  - body: The no-holder check happens before `await readLockPayload`. After `innerRelease()` has removed the held entry, another same-process acquire can reclaim/create this session lock while the fallback is awaiting I/O; because that active lock also has `pid === process.pid`, the later `fs.rm` can unlink it and break mutual exclusion.
