## Summary

Keep this PR open, but it is not merge-ready: the proposed fallback is focused on a real session-lock failure class, yet deleting the lock path by current pid after the lock manager has dropped its held state can remove a fresh same-process lock and break transcript mutual exclusion.

## What This Changes

The PR wraps `acquireSessionWriteLock` release with a best-effort post-release cleanup that force-removes a remaining `.lock` file when its payload pid matches the current process and no tracked reentrant holder remains.

## Best Possible Solution

Make lock cleanup identity-aware in the lock manager or expose a dependency API that proves the same observed file is being removed, then cover snapshot-mismatch and same-process reacquire races with focused tests and redacted runtime/log proof.

## Reproduction Assessment

No high-confidence live reproduction is present for the reported production race. Source inspection confirms the dependency can leave a mismatched snapshot lock behind, but the PR does not demonstrate the fallback path in a real or fully simulated setup.

## Solution Assessment

No. A session-layer `pid` check is only a plausible mitigation; the safer fix keeps deletion tied to the observed file identity in the lock manager or an explicit dependency contract.
