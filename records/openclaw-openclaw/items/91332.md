## Summary

Keep this PR open, but not merge-ready: the fallback cleanup targets a real session-lock failure mode, yet the patch can still remove an active same-process lock and the contributor has not provided after-fix real behavior proof for the new cleanup path.

## What This Changes

The PR wraps `acquireSessionWriteLock` release so it best-effort removes a remaining `.lock` file after release when no in-process holder remains and the lock payload PID matches the current process.

## Best Possible Solution

Make the cleanup ownership-safe with a focused regression for snapshot mismatch plus same-process reacquire, then require redacted runtime or synthetic after-fix proof unless a maintainer explicitly accepts a proof override.

## Reproduction Assessment

Yes, source-level. The fs-safe release contract can leave a lock file behind on snapshot mismatch, and the PR's active-lock deletion race is visible from the awaited payload read between the holder check and path removal; I did not run a live gateway reproduction.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix must preserve mutual exclusion by making fallback removal atomic with ownership proof or by using the dependency's snapshot/inode contract, with regression coverage and real behavior proof.
