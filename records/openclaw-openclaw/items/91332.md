## Summary

Keep open: the PR is focused on a real session-lock concern, but the fallback still bypasses `fs-safe` snapshot ownership with PID-only path removal after an async gap, and the external-contributor real behavior proof is missing.

## What This Changes

The PR wraps session write-lock release with best-effort `.lock` file removal after `fs-safe` release when no in-process holder remains and the lock payload PID matches the current process.

## Best Possible Solution

Make any release-side fallback cleanup snapshot- or ownership-token-aware, add focused regression coverage for same-process reacquire during cleanup, and require redacted after-fix runtime proof unless a maintainer explicitly accepts a proof override.

## Reproduction Assessment

Yes, at source level. The `fs-safe` release path removes the held entry before awaiting snapshot-based cleanup, and the PR then performs a PID-only path removal after an awaited payload read, leaving a same-process reacquire window.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix must preserve mutual exclusion with snapshot-aware or ownership-token-aware cleanup plus focused regression coverage and real behavior proof.
