## Summary

Keep open: this is a focused fix for a credible session write-lock leak, but the current patch is not merge-ready because its fallback can unlink a newly acquired same-process lock and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR wraps `acquireSessionWriteLock` release with a best-effort post-release cleanup that deletes a leftover `.lock` file when no reentrant holder remains and the lock payload PID is the current process.

## Best Possible Solution

Make the cleanup ownership-safe with a focused regression for snapshot mismatch plus same-process reacquire, then merge only with redacted runtime proof or an explicit maintainer proof override while leaving broader delivery contention on https://github.com/openclaw/openclaw/issues/86538.

## Reproduction Assessment

Yes, source-level. The fs-safe release contract can leave a lock file behind on snapshot mismatch, and the proposed fallback race is visible from the awaited read between holder check and path removal; I did not run a live gateway reproduction.

## Solution Assessment

No. The proposed direction is a plausible mitigation, but the best fix must preserve mutual exclusion by making the fallback removal atomic with ownership proof or by using the fs-safe snapshot/inode contract, with regression coverage and real behavior proof.
