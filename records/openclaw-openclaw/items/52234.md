## What This Changes

The PR changes auto-reply followup queues, reply delivery, and gateway session reset cleanup to preserve and rebind queued replies while suppressing stale output after resets.

## Best Possible Solution

Rebase or replace this with a narrower patch that preserves queued followups across reset while keeping current main's session cleanup, threading, and preserved session metadata behavior intact.

  - body: `cleanupSessionBeforeMutation` requires `target` and `entry`, and immediately dereferences `params.target`, but this call omits those fields and even sets `reason` twice. As written, `sessions.reset`/`sessions.new` cannot typecheck or complete the cleanup path reliably.
  - confidence: 0.97
- **[P1] Remove the duplicate activeSessionEntry declaration:** `src/auto-reply/reply/followup-runner.ts:294-295`
  - body: This function already declared `activeSessionEntry` from `rebindQueuedRunToCurrentSession()` above, then declares another `let activeSessionEntry` in the same block. That redeclaration prevents the followup runner from compiling.
  - confidence: 0.95
- **[P1] Keep one browser tab cleanup import:** `src/gateway/session-reset-service.ts:17`
  - body: The module imports `closeTrackedBrowserTabsForSessions` twice under the same local name from different modules. That duplicate module binding is a compile blocker before the reset code can run.
  - confidence: 0.94
- **[P2] Preserve existing reset entry fields:** `src/gateway/session-reset-service.ts:345-370`
  - body: This reset path rebuilds `nextEntry` from a narrow field list, dropping current-main fields such as TTS, exec, queue, delivery, CLI binding, and ACP state. After a reset, users would silently lose persisted per-session behavior instead of only clearing conversation state.
  - confidence: 0.88
