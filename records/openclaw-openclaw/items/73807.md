## Summary

Keep open: the PR targets a real OpenAI Codex OAuth refresh problem, but the proposed cap only changes stored `expires`, so the next capped recheck can repeat the same JWT short-circuit instead of reaching an authoritative refresh or controlled CLI re-import. It also still lacks after-fix real behavior proof from a real Codex OAuth setup.

## What This Changes

This PR adds an OpenAI Codex OAuth access-token JWT `exp` short-circuit and provider tests for stale stored `expires` values.

## Best Possible Solution

Land a rebased provider-local fix that skips unnecessary Codex refreshes for valid cached JWTs while making capped rechecks reach authoritative refresh or controlled CLI re-import and deriving identity metadata on the short-circuit path.

## Reproduction Assessment

Yes, at source level. Seed an OpenAI Codex OAuth credential with stale stored `expires` and a long-lived JWT access token: current main proceeds to provider refresh, while this PR short-circuits and repeats the same short-circuit after the synthetic cap.

## Solution Assessment

No. The provider-local JWT check is the right boundary, but storing the cap only in `expires` means the post-cap path repeats the early return instead of enforcing the claimed refresh or controlled external re-import.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Force capped rechecks through authoritative refresh:** `extensions/openai/openai-codex-provider.ts:327-330`
  - body: The 30-minute cap only changes the stored `expires` value. After that cap passes, the OAuth manager calls this hook again with the same long-lived JWT, and this condition returns before reaching `refreshOpenAICodexToken` or a controlled CLI re-import, so revoked or externally rotated credentials can stay trusted until JWT expiry despite the PR's stated bound.
  - confidence: 0.9

  - confidence: 0.78
