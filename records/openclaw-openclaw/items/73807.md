## Summary

Keep this PR open, but do not merge it as-is. The provider-local JWT short-circuit addresses a real current-main gap, but the proposed 30-minute recheck cap only re-enters the same provider hook and then repeats the short-circuit for the same long-lived JWT, so it does not enforce the stated revocation or external-rotation bound.

## What This Changes

The PR adds an OpenAI Codex OAuth refresh short-circuit based on the cached access-token JWT `exp` claim, caps the corrected stored `expires` value at 30 minutes, and adds provider unit tests for JWT, opaque-token, and malformed-token cases.

## Best Possible Solution

Land a rebased provider-local fix that stops unnecessary OpenAI Codex refreshes when stored `expires` drifts behind a valid access-token JWT, while making any capped recheck actually reach an authoritative refresh or external credential re-import, preserving #75111 account metadata behavior, and recording the user-facing reliability fix in the changelog.

## Reproduction Assessment

Yes. A deterministic unit path can seed an OpenAI Codex OAuth credential with stale stored `expires` and a long-lived JWT access token; current main refreshes immediately, while this PR short-circuits, and advancing past the synthetic 30-minute cap would show the PR short-circuits again instead of reaching HTTP refresh.

## Solution Assessment

No, not as currently implemented. The provider-local JWT check is the right boundary, but the cap is not an effective revocation or external-rotation bound unless the post-cap path can distinguish a prior synthetic `expires` value and perform a real refresh or re-import.
