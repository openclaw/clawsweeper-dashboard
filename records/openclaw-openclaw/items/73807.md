## Summary

Keep this PR open, but do not merge it as-is. The provider-local JWT short-circuit addresses a real current-main gap, but the proposed 30-minute recheck cap does not actually force a later HTTP refresh or external credential re-import, so long-lived JWTs can keep bypassing refresh until their JWT exp despite the PR's stated revocation/rotation bound.

## What This Changes

The PR adds an OpenAI Codex OAuth refresh short-circuit based on the cached access-token JWT exp claim, caps the corrected stored expires value at 30 minutes, and adds provider unit tests for JWT, opaque-token, and malformed-token cases.

## Best Possible Solution

Land a provider-local fix that stops unnecessary OpenAI Codex refreshes when stored expires drifts behind a valid access-token JWT, while making any capped recheck actually reach an authoritative refresh or external credential re-import, preserving the #75111 account metadata behavior, and documenting the user-facing reliability fix in the changelog.

## Reproduction Assessment

Yes. A deterministic unit path can seed an OpenAI Codex OAuth credential with stale stored expires and a long-lived JWT access token; current main refreshes immediately, while this PR short-circuits, and advancing past the synthetic 30-minute cap would show the PR short-circuits again instead of reaching HTTP refresh.

## Solution Assessment

No, not as currently implemented. The provider-local JWT check is the right boundary, but the cap is not an effective revocation or external-rotation bound unless the post-cap path can distinguish a prior synthetic expires value and perform a real refresh or re-import.

## Review Findings

Full review comments:

- **[P2] Make the recheck cap force a real refresh:** `extensions/openai/openai-codex-provider.ts:327-330`
  - body: The 30-minute cap only makes the OAuth manager call this provider hook again. Add logic and a time-advanced test so the post-cap call reaches an authoritative refresh or re-import path.

## Security Review
