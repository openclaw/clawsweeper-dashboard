## Summary

Keep open for revision: current main still unconditionally refreshes stale OpenAI Codex OAuth credentials, but this PR’s synthetic 30-minute cap re-enters the same JWT short-circuit instead of forcing authoritative refresh or controlled external re-import.

## What This Changes

The PR adds an OpenAI Codex OAuth access-token JWT `exp` short-circuit with a 30-minute synthetic `expires` cap and provider tests for JWT, opaque-token, malformed-token, and preservation cases.

## Best Possible Solution

Land a rebased provider-local fix that skips unnecessary OpenAI Codex refreshes for valid cached JWT access tokens while making any capped recheck reach a real refresh or controlled external re-import path, preserving account metadata, and adding the required changelog entry.

## Reproduction Assessment

Yes. A deterministic unit path can seed an OpenAI Codex OAuth credential with stale stored `expires` and a long-lived JWT access token; current main refreshes immediately, while this PR short-circuits and then repeats the same short-circuit after the synthetic 30-minute cap.

## Solution Assessment

No. The provider-local JWT check is the right boundary, but the current cap is not an effective revocation or external-rotation bound unless the post-cap path can distinguish synthetic expiry and force authoritative refresh or re-import.
