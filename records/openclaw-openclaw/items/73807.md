## Summary

Keep open: the PR targets a real OpenAI Codex OAuth refresh bug, but the proposed short-circuit still has a blocking correctness/security issue around the 30-minute cap and lacks after-fix real behavior proof from a real Codex OAuth setup.

## What This Changes

This PR adds an OpenAI Codex OAuth access-token JWT `exp` short-circuit plus provider tests for stale stored `expires` values.

## Best Possible Solution

Land a rebased provider-local fix that skips unnecessary Codex refreshes for valid cached JWTs while making capped rechecks reach authoritative refresh or controlled external re-import and preserving identity metadata.

## Reproduction Assessment

Yes, at source level. Seed a stored OpenAI Codex OAuth credential with stale `expires` and a long-lived JWT access token: current main proceeds to provider refresh, while this PR short-circuits and then repeats the same short-circuit after the synthetic cap.

## Solution Assessment

No. The provider-local JWT check is the right boundary, but this implementation stores the cap only in `expires`, so post-cap re-entry repeats the early return instead of enforcing the claimed revocation or external-rotation check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Force the cap to reach authoritative refresh:** `extensions/openai/openai-codex-provider.ts:327-329`
  - body: The 30-minute cap only updates `expires`. Once that cap passes, the OAuth manager calls this hook again with the same long-lived JWT, and this condition returns before `refreshOpenAICodexToken` again; Codex external CLI sync is bootstrap-only once local inline tokens exist, so `codex login` rotations are not reliably picked up here either. Store state or change control flow so the post-cap path reaches real refresh or a controlled re-import, and add a time-advanced regression test.
  - confidence: 0.9
- **[P3] Preserve Codex identity metadata when skipping refresh:** `extensions/openai/openai-codex-provider.ts:329`
  - body: Current main's refresh path derives and persists Codex `accountId` and `chatgptPlanType` from the access token. Returning only `{ ...cred, expires }` skips that enrichment for older profiles that predate the metadata, leaving account-scoped usage/auth handoff without the fields that a normal refresh would add.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes OAuth token authority, and the stated 30-minute revocation/rotation recheck bound is not enforced by the proposed control flow.

Concerns:

- **[medium] Ineffective revocation recheck cap:** `extensions/openai/openai-codex-provider.ts:327`

Because the cap is stored only as `expires`, the next post-cap refresh repeats the same short-circuit until JWT expiry, leaving revoked or externally rotated Codex credentials unrefreshed longer than the PR claims.
