## Summary

Keep open: current main still does not implement the JWT-exp short-circuit, but this PR is not mergeable as-is because the short-circuit can bypass the promised recheck path and lacks real Codex OAuth proof.

## What This Changes

The PR adds an OpenAI Codex OAuth access-token JWT `exp` short-circuit and provider tests for stale stored `expires` values.

## Best Possible Solution

Land a rebased provider-local fix that avoids unnecessary Codex refreshes for valid cached JWTs only in non-forced contexts, makes capped rechecks authoritative, and preserves Codex identity metadata.

## Reproduction Assessment

Yes, at source level. Seed an OpenAI Codex OAuth credential with stale stored `expires` and a long-lived JWT access token: current main proceeds to provider refresh, while this PR short-circuits and would repeat the same short-circuit after the synthetic cap.

## Solution Assessment

No. The provider-local JWT check is a plausible boundary, but the current PR stores only a capped `expires` value and does not account for current main's forced-refresh or identity-enrichment contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor forced and capped refresh before trusting the JWT:** `extensions/openai/openai-codex-provider.ts:327-330`
  - body: The early return runs whenever the cached JWT `exp` is past the margin. Because the 30-minute cap is stored only as `expires`, the next post-cap entry sees the same long-lived JWT and returns early again; after rebase this also bypasses current main's forced-refresh contract, so revocation, external rotation, or explicit refresh can remain blocked until JWT expiry.
  - confidence: 0.9
- **[P3] Preserve Codex identity on the short-circuit path:** `extensions/openai/openai-codex-provider.ts:330`
  - body: Returning `{ ...cred, expires }` skips current main's identity enrichment from the access token. Older profiles without `accountId` or `chatgptPlanType` would keep missing those fields even though the trusted JWT contains the data used by same-account adoption and fallback checks.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes OAuth token authority, and the proposed shortcut does not currently force the promised authoritative recheck.

Concerns:

  - body: The short-circuit trusts cached access-token JWT `exp` on every provider refresh entry. Since the cap is stored only in `expires`, the next post-cap refresh repeats the same early return and can extend trust in revoked or externally rotated credentials beyond the stated 30-minute bound.
  - confidence: 0.88
