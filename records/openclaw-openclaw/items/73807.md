## Summary

Keep open. The PR targets a real OpenAI Codex OAuth reliability bug, but the proposed 30-minute JWT recheck cap does not actually force an authoritative refresh or external credential re-import, so the stated revocation/rotation bound is ineffective.

## What This Changes

The PR adds an OpenAI Codex OAuth access-token JWT `exp` short-circuit with a 30-minute synthetic `expires` cap and provider tests for JWT, opaque-token, and malformed-token cases.

## Best Possible Solution

Land a rebased provider-local fix that stops unnecessary OpenAI Codex refreshes when stored `expires` drifts behind a valid access-token JWT, while making any capped recheck actually reach an authoritative refresh or controlled re-import path, preserving current account metadata enrichment, and adding the required changelog entry.

## Reproduction Assessment

Yes. A deterministic unit path can seed an OpenAI Codex OAuth credential with stale stored `expires` and a long-lived JWT access token; current main refreshes immediately, while this PR short-circuits, and advancing past the synthetic 30-minute cap would show the PR short-circuits again instead of reaching HTTP refresh.

## Solution Assessment

No. The provider-local JWT check is the right boundary, but the cap is not an effective revocation or external-rotation bound unless the post-cap path can distinguish a prior synthetic `expires` value and perform a real refresh or controlled re-import.

- **[P2] Make the recheck cap force a real refresh:** `extensions/openai/openai-codex-provider.ts:327-330`
  - body: The 30-minute cap only makes the OAuth manager call this provider hook again. On that post-cap call, the same long-lived JWT still satisfies this condition, so the function returns before `refreshOpenAICodexToken` again; a 10-day token keeps skipping HTTP refresh every 30 minutes instead of bounding revocation or external-login rotation exposure. Add state or control flow so the post-cap call reaches an authoritative refresh or re-import path, with a time-advanced test.

This early return only updates `expires`, so older profiles that lack the new metadata will not be enriched on the path that replaces refresh. Reuse the same identity helper before returning the short-circuited credential.

## Security Review

Status: needs_attention

Summary: The diff changes OAuth credential refresh behavior, and the stated 30-minute revocation/rotation bound is not enforced by the proposed control flow.

Concerns:

- **[medium] Ineffective revocation recheck cap:** `extensions/openai/openai-codex-provider.ts:327`
  - body: The short-circuit trusts the cached access-token JWT `exp` on every provider refresh entry.
