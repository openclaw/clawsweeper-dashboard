## Summary

Keep this PR open, but do not merge it as-is. The provider-local JWT expiry check addresses a real Codex OAuth stale-`expires` problem, but the proposed 30-minute cap does not actually force an authoritative refresh or external credential re-import after the cap passes, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR adds an OpenAI Codex OAuth refresh short-circuit based on the cached access-token JWT `exp` claim, caps the corrected `expires` value at 30 minutes, and adds provider unit tests for JWT, opaque-token, and malformed-token cases.

## Best Possible Solution

Land a rebased provider-local fix that avoids unnecessary OpenAI Codex refreshes when stored `expires` drifts behind a valid access-token JWT, while ensuring any capped recheck actually reaches an authoritative refresh or safe external credential re-import path, preserves current account metadata behavior, and records the reliability fix in the changelog.

## Reproduction Assessment

Yes. A deterministic unit path can seed an OpenAI Codex OAuth credential with stale stored `expires` and a long-lived JWT access token; current `main` refreshes immediately, while this PR short-circuits, and advancing past the synthetic 30-minute cap would show the PR short-circuits again instead of reaching HTTP refresh or re-import.

## Solution Assessment

No, not as currently implemented. The provider-local JWT check is the right boundary, but the cap is not an effective revocation or external-rotation bound unless the post-cap path can distinguish a synthetic `expires` value and perform an authoritative refresh or safe re-import.
