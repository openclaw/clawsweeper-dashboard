## Label Justifications

- `P2`: The optional Dashboard voice workflow fails for a valid local-speech-only setup, while normal text chat remains usable.
- `impact:auth-provider`: The failure results from realtime provider selection occurring despite the absence of an intentionally configured realtime provider.

## Summary

Keep this issue open: current main and the latest release expose the native Talk control whenever the Dashboard is connected, then independently resolve a realtime provider, despite documentation treating realtime Talk and local speech synthesis as separate capabilities. No open PR currently owns the repair.

## What This Changes

Prevent the native Dashboard Talk control from attempting an OpenAI Realtime session when only a local speech synthesis provider is configured.

## Best Possible Solution

Expose canonical realtime Talk availability to the Control UI and disable or reject the control with an actionable localized message before microphone activation when no usable realtime provider exists, while keeping local speech synthesis independent.

## Reproduction Assessment

Yes, from source inspection: the connected composer always exposes Talk, and clicking it requests a realtime session even though the documented capability requires separate realtime provider configuration. A live browser run was not performed.

## Solution Assessment

Yes in direction: shared realtime capability gating is the narrowest maintainable repair. Local TTS should not be reinterpreted as a duplex realtime transport, and no new configuration option is needed.
