## Label Justifications

- `P2`: The issue breaks an optional but user-visible Dashboard voice workflow with a bounded provider-routing blast radius.
- `impact:auth-provider`: The failure depends on OpenAI realtime provider selection and whether general API-key authentication counts as configured realtime capability.

## Summary

Current main and v2026.6.6 still expose browser Talk without a UI capability gate, while the provider resolver intentionally auto-selects any provider considered configured; the issue should remain open because the generic 500 is real, but maintainers must decide whether browser Talk requires explicit realtime configuration or may continue using general OpenAI API-key availability.

## What This Changes

Prevent native Dashboard Talk from failing through an unintended OpenAI Realtime WebRTC attempt when the user has local TTS but no explicitly selected realtime provider.

## Best Possible Solution

Define one canonical Gateway result for whether browser realtime Talk is usable, preserve or deliberately revise auto-selection semantics, and have the Control UI disable or explain Talk before microphone/session startup while returning an actionable provider error instead of a generic 500.

## Reproduction Assessment

Yes at source level: current main exposes the control without a provider gate and auto-selects providers from their configured-state contract. The external OpenAI 500 was not live-reproduced during this read-only review.

## Solution Assessment

Unclear until maintainers decide whether general OpenAI API-key availability should continue enabling browser realtime Talk; after that decision, the best implementation is a shared Gateway capability/preflight result rather than UI-only heuristics.
