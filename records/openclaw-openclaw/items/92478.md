## Summary

Keep open: current main and v2026.6.6 still let the Dashboard Talk control start a client-owned OpenAI Realtime path through provider auto-selection, while the related merged OAuth-only fix does not settle whether general OpenAI Platform API-key availability should enable browser Talk without explicit realtime configuration.

## What This Changes

The issue asks for Dashboard Talk to avoid starting OpenAI Realtime WebRTC when only local/non-Realtime voice configuration is intended, and to show a clear disabled or actionable error state instead.

## Best Possible Solution

Define a Gateway-owned browser Talk availability/preflight result, then have the Control UI disable or explain Talk before microphone startup while preserving or deliberately revising provider auto-selection semantics.

## Reproduction Assessment

Yes at source level: current main still exposes the Talk control, starts talk.client.create, and can auto-select an API-key-configured OpenAI Realtime provider. The reported external OpenAI HTTP 500 was not live-reproduced in this read-only review.

## Solution Assessment

Unclear until maintainers choose the Realtime availability policy. After that, the best fix is a Gateway-owned capability/preflight result plus UI disabled/actionable error handling, not independent UI heuristics.
