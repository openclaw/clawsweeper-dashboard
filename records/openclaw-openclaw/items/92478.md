## Summary

Keep this open: current main still exposes the Control UI Talk button without a realtime capability gate and can still enter the OpenAI WebRTC path before the user gets an actionable provider/configuration decision. The report is source-reproducible enough to remain valid, but it needs maintainer product judgment before an automated repair because changing provider auto-selection could affect existing Talk users.

## What This Changes

The issue requests that the Dashboard native Talk button stop attempting OpenAI Realtime WebRTC when realtime Talk is not explicitly usable or configured.

## Best Possible Solution

Define one canonical Gateway/catalog result for whether browser realtime Talk is usable, then have Control UI disable or explain Talk before microphone/WebRTC startup while preserving any intentional provider auto-selection policy.

## Reproduction Assessment

Yes at source level: current main exposes the Talk control without a realtime capability gate and can enter the OpenAI WebRTC `/v1/realtime/calls` path. The exact external OpenAI 500 response was not live-reproduced in this read-only review.

## Solution Assessment

Unclear until maintainers choose the provider policy: explicit realtime configuration versus current auto-selection from general OpenAI auth. The maintainable fix should use a shared Gateway/catalog capability result rather than hardcoded UI heuristics.
