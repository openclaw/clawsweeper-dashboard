## Summary

Keep open: current main still has the Dashboard Talk control wired to start a browser Realtime/WebRTC session without a clear availability preflight, so the reported confusing 500 path is not fully resolved by source, docs, or related issue history.

## What This Changes

The issue asks for the native Dashboard Talk button to avoid starting OpenAI Realtime/WebRTC when Realtime voice is not configured, and instead disable, explain the missing setup, or respect the local Talk/TTS configuration.

## Best Possible Solution

Expose a canonical Gateway Realtime Talk availability/preflight result and have the Dashboard disable or explain the Talk control before microphone/WebRTC startup while preserving the intended provider-selection contract.

## Reproduction Assessment

Yes for the source path, but not for the external 500 in a live environment. Current main shows the connected Talk button starting `talk.client.create` and then the WebRTC `/v1/realtime/calls` path without a catalog/preflight gate.

## Solution Assessment

Unclear until maintainers decide the product contract for implicit Realtime enablement. The best repair is not a local UI special case; it should use Gateway-owned capability/preflight data so provider availability, auth, and UI state stay aligned.
