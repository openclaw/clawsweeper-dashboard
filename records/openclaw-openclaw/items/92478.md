## Summary

Keep open: current main still renders browser Talk without a realtime capability gate and can auto-select a configured OpenAI realtime provider, while the reporter’s follow-up confirms the remaining product question is how browser Talk should behave when only local/non-realtime Talk config is intended.

## What This Changes

The issue asks for Dashboard Talk to avoid starting OpenAI Realtime WebRTC when Realtime is not configured or usable, and instead disable the control, show an actionable message, or respect local non-realtime voice configuration.

## Best Possible Solution

Define a canonical Gateway browser-realtime readiness result, settle explicit-provider versus auto-select semantics, and have the Control UI disable or explain Talk before microphone/WebRTC startup.

## Reproduction Assessment

No for a high-confidence live reproduction of the exact OpenAI 500; yes at source level for the OpenClaw path that exposes Talk, creates a realtime session, and posts to the OpenAI Realtime calls endpoint without a UI readiness gate.

## Solution Assessment

Unclear until maintainers choose the provider-selection contract. The maintainable fix should be Gateway-owned readiness/capability plus UI state, not separate UI heuristics.
