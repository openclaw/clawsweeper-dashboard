## Label Justifications

- `P2`: The issue breaks an optional but user-visible Dashboard voice workflow with limited blast radius while text chat remains usable.
- `impact:auth-provider`: The behavior depends on realtime provider selection and whether general OpenAI Platform credentials implicitly make realtime voice available.

## Summary

Keep this issue open: current main and v2026.6.6 still expose native realtime Talk whenever the Dashboard is connected and auto-select any credentialed realtime provider, even when the user configured only local speech synthesis. The reported OpenAI 500 is not reproducible from source alone, but the confusing capability and provider-selection behavior remains current and has no open implementation PR.

## What This Changes

Make native Dashboard Talk avoid unexpectedly starting OpenAI Realtime WebRTC for users who intended to configure only local speech synthesis.

## Best Possible Solution

Define one upgrade-safe realtime Talk availability contract, expose it through the existing Talk catalog/config surface, and use it to disable or reject the Control UI action before microphone capture while preserving intentional configured and documented auto-selection behavior.

## Reproduction Assessment

Partly: source inspection reproduces the unconditional Talk launch and credential-based realtime provider auto-selection, but not the provider's reported HTTP 500 response. A current-main browser run with the reporter's auth shape or redacted response body is needed to establish the exact failure.

## Solution Assessment

Unclear: capability gating is the correct boundary, but maintainers must decide whether existing credential-based provider auto-selection remains intentional or whether realtime Talk should require explicit configuration.
