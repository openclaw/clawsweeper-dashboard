## Summary

Keep open: current main and the latest release still allow the Control UI Talk button to start the browser realtime path and surface the generic WebRTC failure after the provider call, while the remaining fix depends on a maintainer decision about explicit realtime configuration versus API-key-based auto-selection.

## What This Changes

The issue asks OpenClaw to stop the Dashboard native Talk control from attempting OpenAI Realtime WebRTC when Realtime voice is not explicitly configured or available.

## Best Possible Solution

Have maintainers decide the realtime Talk enablement contract, then wire the Control UI to a Gateway-backed availability/preflight result so the button is disabled or shows a clear provider-specific message before microphone/WebRTC startup.

## Reproduction Assessment

Yes at source level: current main still exposes the Talk control, starts talk.client.create, can auto-select OpenAI realtime from API-key availability, and throws the generic WebRTC status after the OpenAI call. I did not live-reproduce the external 500 response on current main.

## Solution Assessment

Unclear until maintainers choose the product contract for implicit realtime enablement. The maintainable fix should centralize availability/preflight in the Gateway path rather than adding a standalone UI guess.
