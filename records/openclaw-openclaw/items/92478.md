## Label Justifications

- `P2`: The issue breaks an optional but user-visible Dashboard voice workflow while normal text chat still works.
- `impact:auth-provider`: The failure depends on Realtime provider selection and whether OpenAI API-key authentication counts as configured browser Talk capability.

## Summary

Keep open: current main and v2026.6.6 still expose browser Talk without a UI provider-availability gate, and the Gateway still auto-selects any Realtime provider whose configured-state contract passes, so the confusing OpenAI Realtime WebRTC path is not implemented away.

## What This Changes

The issue asks OpenClaw to stop the Dashboard Talk button from starting an OpenAI Realtime WebRTC call when Realtime voice is not explicitly configured or available.

## Best Possible Solution

Add a canonical Gateway capability/preflight result for browser Realtime Talk, then have the Control UI disable or clearly explain Talk before microphone/session startup when no supported Realtime voice path is usable.

## Reproduction Assessment

Yes at source level: current main exposes the Talk control without a provider gate, starts `talk.client.create`, and the WebRTC transport posts to OpenAI `/v1/realtime/calls`. No live current-main browser/API repro was run, so the external 500 itself remains unverified here.

## Solution Assessment

Unclear until maintainers decide whether general OpenAI API-key availability should continue enabling browser Realtime Talk. After that decision, the maintainable fix is a shared Gateway capability/preflight result rather than UI-only heuristics.
