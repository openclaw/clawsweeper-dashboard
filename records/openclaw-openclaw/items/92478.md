## Summary

Keep open: current main still lets the Dashboard Talk button start browser realtime Talk before a clear capability/preflight decision, and the remaining behavior depends on a maintainer product choice about whether a general OpenAI Platform API key should auto-enable realtime Talk without explicit `talk.realtime` selection.

## What This Changes

The issue reports that the Dashboard native Talk button starts an OpenAI Realtime WebRTC call and fails with a generic 500 when Realtime voice is not configured.

## Best Possible Solution

Have Gateway expose a canonical realtime Talk availability/preflight result and make the Dashboard disable or explain Talk before microphone/WebRTC startup, after maintainers decide whether explicit `talk.realtime.provider` selection is required.

## Reproduction Assessment

Yes at source level: current main still exposes the Talk button, starts `talk.client.create`, auto-selects configured realtime providers, and surfaces a generic WebRTC status error after the OpenAI call. I did not live-reproduce the external 500 on current main.

## Solution Assessment

Unclear until maintainers settle the product contract for implicit API-key auto-selection. Once that is decided, the best fix is a shared Gateway capability/preflight result consumed by the Dashboard, not a separate UI-only provider guess.
