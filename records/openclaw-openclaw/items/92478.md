## Summary

Keep open: current main and v2026.6.9 still expose the Control UI Talk button and can proceed into the OpenAI browser WebRTC path before surfacing a generic provider failure, while the remaining fix depends on the maintainer decision about explicit Realtime enablement versus API-key-based auto-selection.

## What This Changes

The issue asks for the Control UI Talk button to avoid starting OpenAI Realtime WebRTC when Realtime voice is unavailable or not intentionally configured, and to disable itself or show a clear provider message instead.

## Best Possible Solution

Have maintainers decide the realtime Talk enablement contract, then wire the Control UI to a Gateway-backed availability or preflight result so the button disables or shows an actionable provider message before microphone and WebRTC startup.

## Reproduction Assessment

Yes at source level: current main exposes the Talk control, starts talk.client.create, can resolve a configured OpenAI realtime provider, and throws the generic WebRTC status after the OpenAI call. I did not live-reproduce the external 500 response on current main.

## Solution Assessment

Unclear until maintainers choose the product contract for implicit realtime enablement. The maintainable fix should centralize availability or preflight in the Gateway path rather than adding a standalone UI guess.
