Keep open: current main still lets a connected Dashboard user start client-owned Realtime Talk, then the WebRTC transport posts to OpenAI `/v1/realtime/calls` and surfaces the generic setup status error. ## What This Changes

The issue reports that the Dashboard native Talk microphone control starts an OpenAI Realtime WebRTC call and shows a generic 500 error even when the user expects a local/non-Realtime Talk setup.

## Best Possible Solution

Define a single Gateway-owned Realtime Talk usability result, then make the Control UI disable or explain the Talk action before microphone/session startup while preserving any maintainer-approved auto-selection behavior.

## Reproduction Assessment

Yes at source level: current main can start the Dashboard RealtimeTalkSession while connected, request microphone access, post SDP to OpenAI Realtime, and surface `Realtime WebRTC setup failed (<status>)`. The external 500 was not live-reproduced during this read-only review.

## Solution Assessment

No for a UI-only heuristic; the maintainable fix is a shared Gateway capability/preflight result plus a policy-approved UI disabled state or actionable provider error. The exact policy still needs maintainer decision because current provider resolution intentionally auto-selects configured Realtime providers.
