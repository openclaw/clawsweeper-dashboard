## Summary

Keep open: current main still exposes the Dashboard Start Talk control based only on connection state, and the browser WebRTC path can request microphone access before posting to the OpenAI Realtime calls endpoint. The merged OAuth-only fix addresses a related failure mode, but this issue still needs the product decision noted in the discussion: whether browser Talk requires explicit realtime configuration or may be enabled by general OpenAI Platform API-key availability.

## What This Changes

The issue reports that the Dashboard native Talk button starts an OpenAI Realtime WebRTC call and fails with a generic 500 when Realtime voice is not configured.

## Best Possible Solution

Define one canonical browser Talk readiness contract, then make the Dashboard consume it so the Talk control is disabled or fails before microphone/WebRTC setup when Realtime is not explicitly usable.

## Reproduction Assessment

Yes for the source-level UI/Gateway path: current main renders Start Talk without realtime capability state and can start WebRTC setup before the failing OpenAI call. No live run was performed to reproduce the exact upstream 500.

## Solution Assessment

Unclear until maintainers choose the policy. A UI/Gateway capability preflight is the likely right shape, but the product decision is whether general OpenAI Platform API-key availability should enable browser Talk or explicit `talk.realtime` configuration should be required.
