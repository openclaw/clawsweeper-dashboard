## Summary

## What This Changes

The issue asks for the Dashboard Talk control to avoid starting OpenAI Realtime WebRTC when Realtime voice is not intentionally usable or configured.

## Best Possible Solution

Define one canonical Gateway realtime availability/preflight result, then have the Control UI disable or explain Talk before microphone capture while preserving or deliberately changing auto-selection semantics after maintainer approval.

## Reproduction Assessment

Yes at source level for the OpenClaw behavior: current main exposes and starts browser Talk without a provider readiness gate, then can auto-select a configured Realtime provider. The external OpenAI HTTP 500 was not live-reproduced in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product contract for browser Talk auto-selection. The best implementation should use a shared Gateway capability/preflight result rather than local UI heuristics or another config knob.
