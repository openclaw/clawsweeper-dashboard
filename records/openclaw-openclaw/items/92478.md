## Summary

Keep open: current main and the latest release still expose the Dashboard Start Talk control without a realtime capability/preflight gate, while Gateway/provider resolution can still enter OpenAI Realtime from configured API-key availability even when the user's configured Talk speech path is local TTS.

## What This Changes

Request that Dashboard native Talk avoid starting OpenAI Realtime WebRTC when realtime voice is not configured or usable, or show a clear disabled/error state before microphone startup.

## Best Possible Solution

Define canonical Gateway browser-Talk usability/preflight semantics, preserve or revise OpenAI Realtime auto-selection deliberately, then have Control UI disable Talk or show an actionable provider message before microphone/session startup.

## Reproduction Assessment

Yes for the OpenClaw source path: current main shows the button can start WebRTC Talk, request microphone access, and post to the OpenAI Realtime calls endpoint before surfacing the generic 500 status. No live provider run was performed for the exact reporter environment.

## Solution Assessment

Unclear until maintainers decide whether browser Talk requires explicit realtime selection or whether general OpenAI API-key availability should continue to auto-enable it. The best implementation should be Gateway-backed capability/preflight behavior rather than UI-only heuristics.
