## Summary

Keep open: current main still wires the Dashboard Talk control to start browser realtime Talk when the Gateway is connected, and the OpenAI provider can be considered configured from general OpenAI API-key auth even when the user's Talk playback provider is local TTS. The remaining work needs a maintainer decision on whether browser Talk should require explicit realtime availability/preflight or keep the existing auto-selection semantics.

## What This Changes

The issue asks for the Dashboard Talk button to avoid starting OpenAI Realtime WebRTC when realtime voice is not explicitly usable, and to show a clear disabled/actionable state or respect local Talk/TTS configuration.

## Best Possible Solution

Use the existing Gateway Talk catalog or one canonical preflight result to expose configured realtime availability, then gate the Control UI before microphone/WebRTC startup and return an actionable provider error while preserving the chosen provider-selection policy.

## Reproduction Assessment

Partially: the source-level path is clear for the visible Talk control and OpenAI WebRTC call, and the reporter confirmed it locally. The exact 500 from OpenAI Realtime still needs live reproduction with a real current setup.

## Solution Assessment

Unclear until maintainers choose the provider policy. The maintainable fix should use Gateway-owned capability/preflight state rather than hard-coded UI heuristics or a second provider-availability policy.
