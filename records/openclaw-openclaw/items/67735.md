## Summary

Keep open: current main still defaults WebChat/Control UI and TUI empty session selection to the selected agent main session, while `session.identityLinks` only canonicalizes after routing has a concrete direct peer. The focused implementation candidate at https://github.com/openclaw/openclaw/pull/61145 is still open and has unresolved product/API and coverage questions, so this issue should remain the canonical tracking thread.

## What This Changes

The issue requests a supported WebChat/Control UI mode or documentation that reconciles WebChat session selection with `session.dmScope`, `session.identityLinks`, and Telegram direct-session continuity.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/61145 as the focused candidate, then either ship an explicit opt-in WebChat/Control UI peer/default mode that resolves through existing `dmScope` and `identityLinks`, or document WebChat as an intentional operator/main-session exception.

## Reproduction Assessment

Yes. Source inspection shows WebChat/default Control UI and TUI selection resolve to main, while `identityLinks` only takes effect when session-key construction receives a concrete direct peer id.

## Solution Assessment

Unclear pending product/API review. An opt-in peer/default mode is safer than changing WebChat defaults globally, but it needs browser Control UI, TUI, config metadata, docs, and named-account semantics aligned before this can close.
