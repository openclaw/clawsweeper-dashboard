## Summary

Keep this open because current main still has the reported split: WebChat/Control UI defaults to the selected session or main session, while identityLinks canonicalization only applies when routing has a concrete direct peer. The related #61145 PR is still open and unmerged, so the remaining work is maintainer review of that opt-in session-mode approach plus the newer named Telegram-account variant.

## What This Changes

The issue requests that WebChat/Control UI support attaching to a canonical direct-human session via session.identityLinks, or clearly document that WebChat remains operator/main-session scoped.

## Best Possible Solution

Review #61145 as the current focused implementation candidate, then choose an explicit opt-in WebChat/Control UI identity or default-peer mode that resolves through the existing dmScope and identityLinks rules. If maintainers decide the operator/main-session behavior is intentional, update the Session, Channel Routing, and WebChat docs to state that boundary clearly.

## Reproduction Assessment

Yes. The deterministic source/test path reproduces the split: a webchat route with peer null resolves to agent:main:main, while identityLinks only applies when buildAgentPeerSessionKey receives a concrete direct peer id.

## Solution Assessment

Unclear pending maintainer review. The #61145 opt-in default-peer approach looks narrower and safer than changing WebChat defaults globally, but it still needs product/API review and a decision on whether it covers the named Telegram-account variant.
