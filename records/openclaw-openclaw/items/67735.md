## Summary

Keep open: current main and the latest release still route WebChat/Control UI empty/default session selection to the agent main session, while `session.identityLinks` only canonicalizes after routing has a concrete direct peer. The focused implementation candidate is still open and needs product/API review plus proof before this issue can close.

## What This Changes

The issue asks WebChat/Control UI to resolve a linked direct human identity into the same canonical direct session used by Telegram DMs, or to document an intentional main-session/operator mode.

## Best Possible Solution

Settle the product contract through the open implementation candidate: either ship an explicit opt-in WebChat/Control UI/TUI direct-peer mode wired through browser UI, TUI, config metadata, docs/protocol, and named-account semantics, or clearly document WebChat as operator/main-session scoped unless the user switches sessions.

## Reproduction Assessment

Yes. Source inspection on current main shows WebChat/default Control UI and empty TUI selection resolve to the main session, while `identityLinks` only takes effect when session-key construction receives a concrete direct peer id.

## Solution Assessment

Unclear pending product/API review. An opt-in peer/default mode is safer than changing WebChat defaults globally, but it must be wired through all public surfaces and account-routing variants before this issue can close.
