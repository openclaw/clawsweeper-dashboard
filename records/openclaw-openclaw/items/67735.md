## Summary

Keep open: current main still defaults WebChat/Control UI and empty TUI selection to the selected agent's main session, while `session.identityLinks` only canonicalizes after routing has a concrete direct peer. The focused implementation candidate at https://github.com/openclaw/openclaw/pull/61145 remains open and unmerged, so the issue still needs maintainer product/API review rather than cleanup closure.

## What This Changes

The issue asks WebChat and the Control UI to support canonical direct-session routing through `session.identityLinks` instead of always defaulting to the selected agent main session.

## Best Possible Solution

Decide the WebChat product contract: either ship an explicit opt-in peer/default mode wired through Control UI, TUI, config metadata, protocol/docs, and named-account semantics, or document WebChat as intentionally operator/main-session scoped unless manually switched.

## Reproduction Assessment

Yes by source inspection, though I did not run a live WebChat/Telegram setup in this read-only review. Current main routes WebChat/default UI and empty TUI selection to main, while `identityLinks` only applies after a concrete direct peer id is available.

## Solution Assessment

Unclear pending maintainer product/API review. An opt-in peer/default mode is safer than changing WebChat defaults globally, but it must be consistently wired and proved before this issue can close.
