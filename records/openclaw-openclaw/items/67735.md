## Summary

Keep open. Current main still has the reported split: WebChat/Control UI defaults to the selected agent main session, while `identityLinks` canonicalization only runs when routing has a concrete direct peer; the focused #61145 implementation candidate is still separate from main and needs maintainer product/API review.

## What This Changes

Request WebChat/Control UI support for attaching to a canonical direct session from `session.identityLinks` instead of always defaulting to the agent main session.

## Best Possible Solution

Add or explicitly reject an opt-in WebChat/Control UI identity/session mode that resolves through existing `dmScope` and `identityLinks`; if maintainers keep WebChat operator-scoped, document that boundary in the session, routing, and WebChat docs.

## Reproduction Assessment

Yes. The current source and route test provide a high-confidence reproduction path: WebChat without a peer resolves to `agent:main:main`, while identity linking only takes effect during direct peer session-key construction.

## Solution Assessment

Unclear pending maintainer product/API review. An opt-in default peer or canonical-direct mode is safer than changing WebChat defaults globally, but it needs to cover the named-account and `dmScope` edge cases before this issue should close.
