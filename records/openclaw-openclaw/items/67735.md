## Summary

Keep open: current main still defaults WebChat/Control UI and empty TUI selection to the selected agent main session, while `session.identityLinks` only canonicalizes direct-message routing after a concrete peer id exists. The focused implementation candidate at https://github.com/openclaw/openclaw/pull/61145 is still open and has proof/product-contract gaps, so this issue remains the right canonical thread.

## What This Changes

Requests a WebChat / Control UI mode that can attach to the same canonical direct-human session as linked Telegram DMs instead of always defaulting to the agent main session.

## Best Possible Solution

Review the open implementation candidate at https://github.com/openclaw/openclaw/pull/61145 and decide the product contract: either ship an explicit opt-in WebChat/Control UI peer/default mode wired through browser UI, TUI, config metadata, protocol/docs, and named-account semantics, or document WebChat as an intentional operator/main-session exception.

## Reproduction Assessment

Yes. Source inspection on current main shows WebChat/default Control UI and empty TUI selection resolve to main, while `identityLinks` only takes effect when session-key construction receives a concrete direct peer id; I did not run tests because this review is read-only.

## Solution Assessment

Unclear pending maintainer product/API review. Reusing the peer-session builder behind an explicit opt-in mode looks safer than changing WebChat defaults globally, but the solution is not complete until browser WebChat, TUI, config metadata, docs, proof, and named-account behavior line up.
