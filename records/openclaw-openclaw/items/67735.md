## Summary

Keep open: current main still defaults WebChat/Control UI and empty TUI session selection to the agent main session, while `session.identityLinks` only canonicalizes once routing has a concrete direct peer. The related implementation candidate remains open and has unresolved browser WebChat, config metadata, named-account, and product/API questions.

## What This Changes

The issue requests a supported WebChat/Control UI mode that can attach to a canonical direct-human session via `session.dmScope` and `session.identityLinks` instead of always defaulting to `agent:main:main`.

## Best Possible Solution

Review the open implementation candidate and decide the product contract: either ship an explicit opt-in WebChat/Control UI peer/default mode wired through browser UI, TUI, config metadata, protocol/docs, and named-account semantics, or document WebChat as an intentional operator/main-session exception.

## Reproduction Assessment

Yes. Source inspection on current main shows WebChat/default Control UI and TUI empty session selection resolve to main, while `identityLinks` only takes effect when session-key construction receives a concrete direct peer id.

## Solution Assessment

Unclear pending product/API review. An opt-in peer/default mode is safer than changing WebChat defaults globally, but it must be wired through all public surfaces and account-routing variants before this issue can close.
