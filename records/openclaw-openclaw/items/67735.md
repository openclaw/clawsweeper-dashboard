## Summary

Keep open. Current main still has the reported split: WebChat/Control UI defaults to a selected or main session, while identityLinks canonicalization only happens when routing has a concrete direct peer. The focused #61145 implementation candidate is open and unmerged, and the later #73226 report adds a related WebChat dmScope variant rather than proving this issue is obsolete.

## What This Changes

The issue requests that WebChat/Control UI either honor canonical direct-session identityLinks like Telegram DMs or clearly document an explicit main-session versus canonical-direct mode.

## Best Possible Solution

Review #61145 as the focused implementation candidate and decide on an explicit opt-in WebChat/Control UI identity/session mode that resolves through existing dmScope and identityLinks rules. If maintainers decide the operator/main-session boundary is intentional, update the Session, Channel Routing, WebChat, and macOS WebChat docs to state that boundary clearly and link #73226 as the related remaining variant if needed.

## Reproduction Assessment

Yes. The deterministic source/test path reproduces the issue: WebChat without a concrete peer resolves to agent:main:main, while identityLinks only applies during direct peer session-key construction.

## Solution Assessment

Unclear pending maintainer product/API review. A narrow opt-in default-peer or canonical-direct mode looks safer than changing WebChat defaults globally, but it still needs review against named-account and dmScope variants.
