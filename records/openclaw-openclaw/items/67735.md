## Summary

Keep open. Current main still defaults WebChat/Control UI to the selected agent main session while identity-link canonicalization only applies after routing has a concrete direct peer; the focused related implementation PR is still open and needs maintainer product/API review.

## What This Changes

The issue asks for WebChat/Control UI to honor configured direct-session identity links, or to clearly document that it intentionally remains main-session scoped.

## Best Possible Solution

Define an explicit WebChat/Control UI identity/session mode that resolves through existing dmScope and identityLinks, or deliberately document WebChat as an operator/main-session exception across the session, routing, WebChat, and security docs.

## Reproduction Assessment

Yes. Source inspection shows WebChat/default Control UI selection resolves to the main session, while identityLinks only take effect when direct-peer session-key construction receives a non-empty peer id.

## Solution Assessment

Unclear pending maintainer product/API review. An opt-in WebChat default-peer or canonical-direct mode is safer than changing WebChat defaults globally, but it needs to cover browser Control UI and named-account edge cases before this can close.
