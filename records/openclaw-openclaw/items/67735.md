## Summary

Keep open: current main still routes WebChat/Control UI defaults to the selected agent main session while identity-link canonicalization only runs after a concrete direct peer is supplied; the focused implementation candidate at https://github.com/openclaw/openclaw/pull/61145 is still open and needs product/API review.

## What This Changes

Request WebChat / Control UI support for canonical direct-human session binding through `session.dmScope` and `session.identityLinks`, instead of always defaulting to the agent main session.

## Best Possible Solution

Review https://github.com/openclaw/openclaw/pull/61145 as the focused candidate, then either ship an explicit opt-in WebChat/Control UI peer-default mode that resolves through existing `dmScope` and `identityLinks`, or document WebChat as an operator/main-session exception.

## Reproduction Assessment

Yes. Source inspection shows WebChat/default Control UI selection resolves to the main session, while `identityLinks` only takes effect when direct-peer session-key construction receives a non-empty peer id.

## Solution Assessment

Unclear pending product/API review. An opt-in WebChat default-peer or canonical-direct mode is safer than changing WebChat defaults globally, but it must cover browser Control UI, TUI, docs, config metadata, and named-account edge cases before this can close.
