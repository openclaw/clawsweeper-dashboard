## Summary

Keep open: current main still has the intended Matrix sibling token-hash storage selection and docs mention reuse, but there is no warning, doctor signal, or cleanup guidance for accumulated populated sibling stores. This is a Matrix observability/docs feature request rather than a broken existing contract.

## What This Changes

The issue asks for Matrix crypto-store observability and documentation when multiple populated token-hash sibling directories accumulate after token rotations.

## Best Possible Solution

Add a Matrix-owned diagnostic for multiple populated token-hash roots that redacts sensitive details, leaves storage selection unchanged, and documents safe archival cleanup.

Not applicable: this is an observability/docs request for an intended Matrix storage-selection design, not a regression report. Source inspection verifies the sibling enumeration path and the absence of a warning or cleanup documentation.

## Solution Assessment

Unclear: a structured warning plus docs is a plausible minimal solution, but maintainers should choose between startup logging and doctor diagnostics and define redaction/log-noise policy before implementation.
