## Summary

Keep open. Current main has targeted lifecycle helpers, but it still does not provide the requested shared async resource lifecycle protocol, and the remaining scope overlaps open retry/tool and plugin-channel cleanup follow-ups that need maintainer design judgment.

## What This Changes

The issue requests a shared async resource lifecycle manager or shutdown protocol so retry, cleanup, and plugin/channel shutdown paths stop, drain or cancel, and dispose resources consistently.

## Best Possible Solution

Keep this as the broad lifecycle-design tracker while maintainers land or replace the narrower retry/tool and Feishu fixes, then decide whether a small versioned core/plugin SDK lifecycle contract should be introduced.

## Reproduction Assessment

No high-confidence single reproduction covers the full umbrella request. Source inspection does confirm motivating current-main paths: Feishu still deletes monitor state immediately after server.close(), and retry cleanup still uses a narrow waitForIdle-then-flush helper.

## Solution Assessment

Unclear. A shared ResourceManager may be the right end state, but current evidence supports maintainer-owned design plus targeted fixes before adding a new core or plugin SDK lifecycle contract.
