## Summary

Keep open: current main has several targeted lifecycle and cleanup primitives, but not the requested shared ResourceManager-style shutdown, cancellation, or drain protocol; this remains a cross-cutting maintainer API decision.

## What This Changes

Request asks OpenClaw to define a shared ResourceManager-style lifecycle protocol for async work across retry, cleanup, channel, and plugin paths.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers choose a minimal lifecycle primitive, then migrate only proven high-risk retry, shutdown, channel/plugin, and delivery paths with focused contract tests.

## Reproduction Assessment

No, not as a single current-main bug repro. This is a cross-cutting lifecycle/API proposal built from related bugs; the motivating examples have source evidence and targeted fixes, but the shared ResourceManager request itself needs maintainer design review.

## Solution Assessment

Unclear. A shared lifecycle primitive is plausible, but the best solution needs maintainer agreement on owner boundaries and should reuse existing drain, abort, host-cleanup, channel queue, and generation primitives where they already fit.
