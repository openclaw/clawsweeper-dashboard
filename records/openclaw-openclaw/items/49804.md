## Summary

Keep open: current main has several targeted lifecycle fixes and opt-in helper primitives, but it does not implement the requested standard ResourceManager-style shutdown/cancellation/drain protocol, and that remaining work crosses agent retry cleanup, gateway shutdown, channel/plugin lifecycle, and outbound delivery boundaries that need maintainer design judgment.

## What This Changes

Request to standardize async resource lifecycle management across retry and cleanup paths so pending work is cancelled or drained before resources are disposed.

## Best Possible Solution

Keep this as a maintainer design tracker until the team chooses a minimal lifecycle primitive, then migrate only proven high-risk paths with focused contract tests instead of adding a broad manager by default.

## Reproduction Assessment

No, not as a single current-main bug repro. This issue is an architecture/API proposal built from related bugs; the motivating examples have source evidence and targeted fixes, but the ResourceManager request itself needs design review rather than one failing path.

## Solution Assessment

Unclear. A shared lifecycle primitive is plausible, but the best solution needs maintainer agreement on the owner boundary and should reuse existing drain, abort, and lifecycle-generation primitives where they already fit.
