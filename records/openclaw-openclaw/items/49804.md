## Summary

Keep open: current main has targeted lifecycle drains and cleanup fixes, but not the requested shared ResourceManager-style shutdown/cancellation/drain protocol; the channel-only follow-up was closed into this issue, so this remains the canonical maintainer API decision.

## What This Changes

The issue asks OpenClaw to standardize async resource lifecycle handling with a shared drain/cancel/shutdown protocol instead of ad hoc retry and cleanup guards.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers choose a minimal lifecycle primitive, then migrate only proven high-risk retry, shutdown, channel/plugin, and delivery paths with focused contract tests.

## Reproduction Assessment

No, not as a single current-main bug repro. This is a cross-cutting lifecycle/API proposal built from related bugs; the motivating examples have source evidence and targeted fixes, but the shared ResourceManager request itself needs maintainer design review.

## Solution Assessment

Unclear. A shared lifecycle primitive is plausible, but the best solution needs maintainer agreement on the owner boundary and should reuse existing drain, abort, host-cleanup, and lifecycle-generation primitives where they already fit.
