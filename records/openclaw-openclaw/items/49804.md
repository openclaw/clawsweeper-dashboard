## Summary

Keep open: current main has targeted lifecycle fixes and opt-in cleanup helpers, but it does not implement a standard ResourceManager-style drain/cancel/shutdown protocol across agent retry, channel/gateway, outbound delivery, and plugin SDK boundaries.

## What This Changes

The issue asks OpenClaw to standardize async resource lifecycle management with a ResourceManager-like protocol for retry and cleanup paths where work can outlive disposal.

## Best Possible Solution

Keep this as a maintainer design tracker until a minimal lifecycle primitive is chosen, then migrate only proven high-risk paths with contract tests and leave narrow symptom fixes on their own tracks.

## Reproduction Assessment

Not applicable as a single bug reproduction: this is an architecture/API proposal. The motivating examples have source/history evidence, but the ResourceManager request itself needs design review rather than one current-main failing path.

## Solution Assessment

Unclear. A ResourceManager-like direction is plausible, but the best solution is a maintainer-approved minimal lifecycle contract that reuses existing drain/abort primitives instead of adding a broad duplicate layer.
