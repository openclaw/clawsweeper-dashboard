## Summary

Keep open. Current main has useful targeted lifecycle pieces, including a one-shot wait-for-idle tool-result flush guard, plugin cleanup callbacks, and channel lifecycle helpers, but it does not provide the requested shared async resource lifecycle protocol across retry/tool cleanup and plugin/channel shutdown paths. The motivating Feishu cleanup path still closes HTTP servers and clears state without waiting for close, while the related retry/tool PR and the narrower PluginResourceManager issue remain open.

## What This Changes

Request a standardized async resource lifecycle manager or shutdown protocol so retry, tool, channel, and plugin cleanup paths stop accepting work and drain or cancel pending operations before disposing resources.

## Best Possible Solution

Keep this as the broad lifecycle-design tracker. Maintainers should review or replace the open retry/tool PR, fix the Feishu HTTP cleanup path directly, and then decide whether core or the plugin SDK needs a small shared lifecycle contract with explicit stop-accepting-work, drain/cancel, timeout, and forced-dispose semantics before migrating more call sites.

## Reproduction Assessment

No high-confidence single reproduction covers the full architectural request; it is a design-level cross-subsystem cleanup tracker. There is high-confidence source-level reproduction for motivating paths: Feishu still deletes monitor state immediately after server.close(), and retry cleanup still uses a narrow waitForIdle-then-flush helper.

## Solution Assessment

Unclear. A broad ResourceManager might be the eventual right shape, but current evidence supports starting with targeted fixes and a maintainer-owned design decision before introducing a new shared core/plugin SDK contract.
