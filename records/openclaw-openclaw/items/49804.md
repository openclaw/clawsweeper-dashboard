## Summary

Keep open: current main has targeted lifecycle repairs and restart-drain helpers, but it still does not implement the requested standard ResourceManager-style cancellation/drain/shutdown contract, and the remaining work is a cross-boundary product/API design decision rather than a stale duplicate.

## What This Changes

The issue requests a standard async resource lifecycle manager or shutdown protocol for retry, cleanup, and delivery paths where async work can outlive disposal.

## Best Possible Solution

Keep this as the canonical maintainer design tracker until the team chooses a minimal lifecycle primitive, then migrate only proven high-risk paths with focused contract tests and public SDK/docs follow-through.

## Reproduction Assessment

No, not as a single current-main bug repro. This issue is an architecture/API proposal built from related bugs; source inspection confirms targeted fixes and missing central manager, but the ResourceManager request itself needs design review rather than one failing path.

## Solution Assessment

Unclear. A shared lifecycle primitive is plausible, but the best solution needs maintainer agreement on the owner boundary and should reuse existing drain, abort, generation, and restart helpers where they already fit.
