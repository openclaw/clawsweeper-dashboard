## Summary

The PR still needs maintainer review: it improves date/timer hardening, but the current auth ownership predicate can discard a recoverable child OAuth credential when the main credential is merely date-valid, and the `/session` expiry display can now diverge from the lifecycle enforcement path. The PR also still has only test/lint/autoreview evidence, not real child-agent auth-store proof.

## What This Changes

The PR changes OAuth/profile timestamp handling and broadly clamps timer/date values across agents, gateway, plugin SDK, bundled plugins, media, secrets, and tests.

## Best Possible Solution

Keep the timestamp helpers, but make the auth-owner predicate preserve recoverable local child OAuth credentials unless the main credential is usable/newer or token-identical, and normalize session lifecycle display and enforcement through the same helper before merge.

## Reproduction Assessment

Yes. Source inspection shows the current PR can classify an out-of-range local OAuth expiry as `undefined` and still let a date-valid main credential own the child profile, and it can report `/session` expiry from `boundAt` while lifecycle expiry uses raw `lastActivityAt`.

## Solution Assessment

No. The broad clamp direction is useful, but the safest solution needs the auth-owner predicate and session lifecycle/display paths to share the same validity rules and focused regressions before merge.

Full review comments:

- **[P1] Require a usable main credential before dropping local OAuth:** `src/agents/auth-profiles/store.ts:147`
  - body: When `local.expires` is outside the date range, this returns true for any date-valid main credential with the same identity. A child store with different local access/refresh tokens can then drop its recoverable local profile and route refreshes to an older or expired main profile. Preserve the local record unless the tokens are identical or the main credential is actually usable/newer than the malformed local record.
  - confidence: 0.88
- **[P2] Keep `/session` expiry reporting aligned with lifecycle expiry:** `src/auto-reply/reply/commands-session.ts:113`
  - body: Falling back an invalid `lastActivityAt` to `boundAt` here makes `/session idle` report a concrete expiry, but `resolveThreadBindingLifecycle` still expires from the raw `lastActivityAt`, so the runtime may not enforce the date the command reports. Normalize the shared lifecycle input the same way or avoid reporting an expiry that enforcement will not honor.
  - confidence: 0.82
