## Summary

Keep open: current main still has the transcript rewrite/truncation path that can branch at the first replacement and re-append the remaining suffix, so the raw JSONL duplication/write-amplification part of the report is not implemented away. `sessions_history` itself is a bounded read-only tool, and current history readers follow the active branch for display, but the underlying rewrite behavior still needs maintainer follow-up through #69486, #72511, or an equivalent fix.

## What This Changes

The issue requests that `sessions_history` and related large tool-result truncation stop causing duplicate transcript rows with fresh IDs but identical prior message content in session JSONL files.

## Best Possible Solution

Land #69486, #72511, or an equivalent ancestry-preserving rewrite/truncation fix that prevents duplicate raw JSONL rows from suffix replay, then add a regression around a large `sessions_history` or tool-result truncation case that inspects the raw transcript file and the active-branch history view.

## Reproduction Assessment

Yes. The issue includes concrete JSONL and trajectory evidence, a follow-up comment reproduced the mechanism on main, and current main still contains the large tool-result truncation path into branch-and-suffix replay; this review stayed read-only and did not create live artifacts.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the right boundary because it only reads; the durable fix belongs in transcript rewrite/truncation, with maintainer review choosing an ancestry-safe cleanup or replay-dedupe invariant.
