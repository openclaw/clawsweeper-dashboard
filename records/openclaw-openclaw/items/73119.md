## Summary

Keep open: current main still sends large tool-result truncation through the branch-and-suffix transcript rewrite path, and the persistence path appends replayed suffix entries to JSONL. `sessions_history` itself is read-only and newer history readers select the active branch, but that mitigates visibility rather than eliminating the raw transcript-write duplication reported here.

## What This Changes

The issue requests preventing `sessions_history` and large tool-result truncation from causing duplicate transcript JSONL entries while preserving single provider-call behavior.

## Best Possible Solution

Adopt an ancestry-preserving transcript rewrite cleanup or equivalent invariant so abandoned suffix rows do not inflate raw transcripts/history, with regression coverage for large `sessions_history` tool results and sibling branches.

## Reproduction Assessment

Yes. The issue includes concrete JSONL and trajectory evidence, a follow-up comment reproduced the mechanism on main, and current source still contains the truncation-to-branch-and-suffix-rewrite path.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the right boundary because it only reads; the maintainable fix belongs in transcript rewrite/truncation or an equivalent active-branch/raw-file invariant.
