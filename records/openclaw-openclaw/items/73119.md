## Summary

Keep open: current main still routes large tool-result truncation through a branch-and-suffix transcript rewrite that appends unchanged suffix entries, while `sessions_history` itself remains a bounded read-only tool. The issue discussion already localized the mechanism, and the related fixes #69486 and #72511 are still open rather than merged.

## What This Changes

Prevent `sessions_history`-triggered tool-result truncation or rewrite maintenance from duplicating prior transcript messages in raw JSONL history.

## Best Possible Solution

Preserve active-branch ancestry while ensuring rewrite-abandoned suffix entries do not remain visible to flat transcript/history readers, with regression coverage for large history/tool-result truncation.

## Reproduction Assessment

Yes. The issue includes concrete JSONL and trajectory evidence, a follow-up comment reproduced the mechanism on main, and current source still has the truncation-to-branch-and-suffix-rewrite path; this read-only review did not create live artifacts.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the right boundary because it only reads; the maintainable fix belongs in transcript rewrite/truncation, with maintainer review choosing the ancestry-aware, identity-dedupe, or equivalent invariant-preserving approach.
