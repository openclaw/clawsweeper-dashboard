## Summary

Keep open: current main still has the transcript rewrite path that can re-append unchanged suffix entries after a large tool-result truncation, while `sessions_history` itself remains a bounded read-only tool. The issue discussion correctly localized the mechanism, and the related fixes #69486 and #72511 are still open proposals rather than merged current-main behavior.

## What This Changes

The issue requests that `sessions_history` and related active-session transcript reads stop causing duplicate JSONL transcript entries, especially after large history/tool-result truncation paths.

## Best Possible Solution

Land #69486, #72511, or an equivalent maintained fix that preserves active-branch ancestry while preventing duplicate raw JSONL rows from abandoned suffix replay, then add a regression around a large `sessions_history`/tool-result truncation case that inspects the raw transcript file.

## Reproduction Assessment

Yes. The issue includes concrete JSONL and trajectory evidence, the follow-up comment reproduced the mechanism on main, and current main still has the large tool-result truncation to branch-and-suffix-rewrite path; this review stayed read-only and did not create live artifacts.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the right boundary because it only reads; the durable fix belongs in transcript rewrite/truncation, with maintainer review choosing between ancestry-aware cleanup, suffix dedupe, or an equivalent invariant-preserving approach.
