## Summary

Keep open. Current main mitigates the visible `chat.history`/`sessions_history` duplicate-read symptom by projecting tree transcripts through the active branch, but the transcript rewrite path still branches at the first replacement and re-appends the suffix, so the raw JSONL duplication mechanism remains and the competing cleanup PRs still need maintainer resolution.

## What This Changes

The issue requests preventing `sessions_history` and large tool-result truncation from creating duplicate transcript JSONL suffix entries that inflate downstream history consumers without extra provider calls.

## Best Possible Solution

Keep this open until #69486, #72511, or an equivalent maintained fix is accepted or explicitly rejected. The end state should keep `sessions_history` bounded and read-only while either preventing low-value raw suffix replay or documenting and testing why append-only abandoned entries are acceptable for all downstream consumers.

## Reproduction Assessment

Yes. The report includes concrete raw JSONL evidence, the follow-up comment reproduced and localized the mechanism, and current main still contains the truncation-to-rewrite path that replays suffix entries; this read-only review did not create live transcript artifacts.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the right boundary because it is a bounded read path; the maintainable fix belongs in transcript rewrite/truncation or in a deliberate active-branch projection policy for every downstream reader.
