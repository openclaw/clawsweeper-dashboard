## Summary

Keep open: current main still contains the transcript rewrite path that can re-append unchanged active-branch suffix entries after tool-result truncation. The issue is better handled by maintainer review of the already-open transcript-rewrite PRs than by closing or queueing a separate repair.

## What This Changes

The issue reports that large `sessions_history` results can trigger tool-result truncation to duplicate prior messages in raw JSONL transcript records without extra provider calls.

## Best Possible Solution

Land #69486, #72511, or an equivalent maintained fix that prevents low-value raw JSONL suffix replay while preserving active-branch ancestry and sibling-branch integrity, then verify it against the large `sessions_history` raw transcript case.

## Reproduction Assessment

Yes. The issue provides concrete JSONL and trajectory evidence, the follow-up comment reproduced the mechanism on main, and current source inspection still shows the truncation-to-rewrite path with suffix replay.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the best boundary because it only reads; the maintainable solution belongs in transcript rewrite/truncation, with maintainer review choosing between cleanup, dedupe, or a combined invariant.
