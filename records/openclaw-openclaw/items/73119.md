## Summary

Keep open: current main keeps `sessions_history` itself read-only and now reads active transcript branches for chat/history surfaces, but the write-side transcript rewrite path still branches and re-appends the suffix into append-only JSONL. The existing open PRs #69486 and #72511 cover the same rewrite-duplication cluster, so this should stay open for maintainer review rather than be closed as implemented.

## What This Changes

The issue reports duplicate JSONL transcript records after large `sessions_history` output triggers tool-result truncation and transcript rewrite behavior.

## Best Possible Solution

Preserve active-branch ancestry while preventing rewrite-abandoned suffix entries from remaining as duplicate raw transcript records, with regression coverage for large tool-result/history truncation.

## Reproduction Assessment

Yes. The issue includes concrete JSONL and trajectory evidence, a follow-up comment reproduced the rewrite mechanism on main, and current source still has the truncation-to-branch-and-suffix-append path; this read-only review did not create live artifacts.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the right boundary because it only reads; the maintainable fix belongs in transcript rewrite/truncation persistence, with maintainer review choosing the ancestry-aware cleanup or equivalent invariant-preserving approach.
