## Summary

Keep open: current main still has the transcript rewrite path that can replay unchanged suffix entries into JSONL after large tool-result truncation, and the issue is not fixed by making `sessions_history` read-only because that tool already only reads history.

## What This Changes

The issue reports duplicate JSONL transcript rows after `sessions_history` returns large history content, with no extra provider calls or double billing.

## Best Possible Solution

Land one coherent transcript rewrite invariant that prevents abandoned replayed rows from inflating raw JSONL, with regression coverage for large `sessions_history` tool results and repeated rewrite/truncation passes.

## Reproduction Assessment

Yes. The issue gives concrete session-file and trajectory evidence, the follow-up comment reproduced the source mechanism on main, and current source still contains the branch-and-suffix rewrite path that explains the duplicate rows.

## Solution Assessment

No for a direct `sessions_history` fix: the tool already only reads via `chat.history`. The maintainable fix belongs in transcript rewrite/truncation cleanup, likely by reviewing or superseding the overlapping open PRs.
