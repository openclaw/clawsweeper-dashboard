## Summary

Keep open. Current main still has the transcript maintenance path that branches at the first replacement and re-appends the active-branch suffix, so the raw JSONL duplication mechanism described here is still present even though the `sessions_history` tool itself is a bounded read path.

## What This Changes

Prevent transcript maintenance triggered after large `sessions_history` reads or similar large tool results from duplicating prior assistant and tool-result messages in raw JSONL session transcripts.

## Best Possible Solution

Keep this open until #69486, #72511, or an equivalent maintained fix lands and is verified against the raw JSONL `sessions_history` case. The end state should preserve bounded read-only `sessions_history` behavior while preventing low-value duplicate suffix replay or safely cleaning abandoned rewrite entries without breaking active branch ancestry.

Yes. The issue includes concrete raw JSONL evidence, a follow-up comment reproduced and localized the mechanism, and current-main source still contains the large-tool-result truncation to suffix-rewrite path; this review did not create live artifacts because it was read-only.

Unclear. Fixing `sessions_history` itself is not the best boundary because the tool only reads; the maintainable fix belongs in transcript rewrite/truncation, with maintainer review choosing between the open cleanup strategies or a combined equivalent.
