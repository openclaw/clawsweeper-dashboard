## Summary

Keep open: current main still has the large tool-result truncation path that branches and replays the transcript suffix, so the raw JSONL duplication mechanism remains plausible even though `sessions_history` itself is read-only and chat history follows the active branch.

## What This Changes

Prevent `sessions_history`-triggered large tool-result truncation from duplicating prior transcript rows in session JSONL while preserving the existing billing and model-context behavior.

## Best Possible Solution

Land an ancestry-preserving transcript rewrite cleanup, or an equivalent invariant, so abandoned replayed suffix rows do not inflate raw JSONL transcripts while active-branch history and sibling branches remain intact.

## Reproduction Assessment

Yes. The issue provides concrete transcript and trajectory evidence, the follow-up comment reproduced the mechanism on main, and current source still contains the truncation-to-branch-and-suffix-rewrite path.

## Solution Assessment

No for fixing `sessions_history` directly: that tool is already a bounded read surface. The maintainable fix belongs in transcript rewrite/truncation cleanup or raw-transcript active-branch reconciliation.
