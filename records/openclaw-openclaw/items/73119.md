## Summary

Keep open. Current main still has the transcript maintenance path that truncates oversized tool results by branching before the first replacement and replaying the active-branch suffix, so the raw JSONL duplication mechanism remains. The `sessions_history` tool itself is already a bounded read path, and related fix work is open in #69486 and #72511, so this should stay open as useful reproduction evidence rather than spawning another repair lane.

## What This Changes

The issue reports transcript-only duplicate JSONL writes after large `sessions_history` results enter the tool-result truncation and transcript rewrite path.

## Best Possible Solution

Keep this open until #69486, #72511, or an equivalent maintained fix lands and is verified against the raw JSONL `sessions_history` trigger. The end state should keep `sessions_history` as a bounded read surface while preventing low-value suffix replay or safely reconciling abandoned rewrite entries without breaking branch ancestry.

## Reproduction Assessment

Yes. The issue includes concrete raw JSONL evidence, a follow-up comment reproduced and localized the mechanism, and current main still contains the large-tool-result truncation to suffix-rewrite path; this review did not create live artifacts because it was read-only.

## Solution Assessment

Unclear. Changing `sessions_history` itself is not the best boundary because the tool only reads; the maintainable fix belongs in transcript rewrite/truncation, with maintainers choosing or combining the open cleanup strategies.
