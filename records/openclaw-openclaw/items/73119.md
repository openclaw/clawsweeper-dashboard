## Summary

Keep open. Current main still has the write-side transcript rewrite path that branches at the first replacement and re-appends the remaining suffix after large tool-result truncation; the newer active-branch history reader mitigates some `chat.history` exposure but does not stop raw JSONL duplicate/sibling entries from being written. The issue discussion already gives a concrete reproduction and two open PRs target the same transcript-rewrite area, so this should stay open for maintainer review rather than be swept closed.

## What This Changes

The issue reports that invoking `sessions_history` can push large tool-result truncation through transcript rewrite, causing prior assistant/toolResult messages to be re-appended into the JSONL transcript with fresh IDs as a parallel branch.

## Best Possible Solution

Land one maintained write-side fix, or an equivalent combined fix, that prevents low-value suffix replay or safely reconciles rewrite-abandoned entries while preserving active-branch ancestry. Add a regression that exercises a large `sessions_history` tool result and asserts the raw JSONL does not accumulate duplicate assistant/toolResult suffix entries.

## Reproduction Assessment

Yes. The issue includes concrete raw JSONL evidence and provider trajectory counts, the follow-up comment reproduced the mechanism, and current main still has the traceable truncation-to-rewrite-to-suffix-replay path.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the best boundary because the tool only reads; the maintainable fix belongs in transcript rewrite/truncation, with maintainer review choosing between the existing cleanup and dedupe approaches or a combined equivalent.
