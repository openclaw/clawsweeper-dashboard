## Summary

The original 404 was explained by the enforced 2 MB local-avatar limit, and current main now exposes `too_large` through avatar metadata, but the Control UI still appears to discard that reason and show only a broken or missing avatar. An open closing PR also remains paired with this issue, so conservative cleanup should keep the issue open while maintainers resolve the narrow UI-diagnostic remainder or close the non-viable PR.

## What This Changes

Make Webchat display workspace-relative avatars from `IDENTITY.md` reliably and report a clear error instead of a generic 404 when a local avatar is rejected.

## Best Possible Solution

Keep the secure 2 MB and workspace-containment policy, surface the existing safe `avatarReason` as a user-friendly Control UI message, and consolidate any reproducible under-limit avatar failure with https://github.com/openclaw/openclaw/issues/41201 before resolving the paired PR.

## Reproduction Assessment

Yes for the oversized-file path: configure a workspace-local avatar larger than the enforced limit and the resolver deterministically returns `too_large`, while the image request cannot be served. The separate under-limit flicker or null-metadata report is not yet a high-confidence current-main live reproduction.

## Solution Assessment

No, the linked PR is not the best current solution because it is conflicting, broadly contaminated with unrelated changes, and addresses path parsing rather than the reporter-confirmed size failure. The narrow maintainable fix is to render the gateway’s safe rejection reason in the Control UI and separately verify any under-limit Windows failure.
