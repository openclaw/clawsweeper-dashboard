## Summary

Keep this open. Current main still has a source-reproducible Windows path-containment failure path for valid workspace-relative avatars, and open PR #38923 still references this issue with closing syntax but is unmerged and dirty.

## What This Changes

Valid workspace-relative `Avatar:` paths in `IDENTITY.md` should make `/avatar/{agentId}` and `/avatar/{agentId}?meta=1` return the configured avatar instead of 404 or null metadata.

## Best Possible Solution

Keep this issue open until #38923 is reviewed and landed or replaced with a narrower avatar-only fix that normalizes Windows namespace/drive-letter paths, preserves symlink and size protections, and adds route-level regression coverage.

## Reproduction Assessment

Yes, at source level. The issue has concrete Windows steps, and current main's containment helper can be shown to reject a same-workspace Windows namespaced root plus drive-letter avatar path without running a live Windows UI.

## Solution Assessment

Yes for keeping the issue open. The maintainable fix belongs in the avatar resolver/policy and route tests; if #38923 remains broad or dirty, maintainers should extract only the avatar normalization and parsing coverage.
