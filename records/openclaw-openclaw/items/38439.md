## What This Changes

The issue requests that `/avatar/{agentId}` serve a valid workspace-relative avatar declared in an agent workspace `IDENTITY.md` instead of returning 404 in WebChat/Control UI.

## Best Possible Solution

Keep this issue open until #38923 is reviewed and either landed or replaced with an equivalent narrow fix. The desired end state is a small avatar-resolution change that normalizes Windows namespace and drive-letter path forms before workspace containment checks, preserves the existing symlink and 2 MB protections, handles backtick-wrapped `IDENTITY.md` avatar values if maintainers want to cover the related report shape, and adds regression coverage for a non-default Windows agent through both `/avatar/<agentId>?meta=1` and `/avatar/<agentId>`.

Yes, at source level. The issue includes concrete Windows steps, a contributor validated the namespace-path root cause, and a `path.win32.relative()` simulation against current main reproduces the containment false negative; a live Windows UI run was not performed in this read-only pass.
