Keep open. Current main has targeted required-parameter guards for read/write/edit and general loop-detection hooks, but it still lacks the requested schema-derived, configurable validation surface plus an explicit auto-prompt/retry policy before generic tool dispatch.

## What This Changes

The issue requests configurable pre-dispatch validation for missing tool parameters and optional automatic prompting or retry behavior.

## Best Possible Solution

Add a typed, schema-derived validation seam that composes with existing per-tool validators and before-tool-call hooks, then design automatic prompting or retrying as a separate explicit opt-in policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection confirms the requested config keys and generic schema-derived validation policy are absent on current main, while existing behavior is targeted validation, handler-level errors, hooks, and threshold-based loop detection.

## Solution Assessment

Unclear as proposed. Pre-dispatch validation is plausible, but the config names and auto-prompt/retry semantics need maintainer design alongside schema normalization, before-tool hooks, per-tool validators, and loop detection.
