## Summary

Keep open. Current main still has targeted runtime checks for read/write/edit and handler-level checks for exec, but it does not provide the requested generic configurable pre-dispatch required-parameter validation or automatic retry/prompt policy across tool schemas. The related #56562 is closed, but it is not an open canonical implementation path and does not prove the requested behavior shipped.

## What This Changes

The issue requests configurable pre-call validation for missing required tool parameters, with optional automatic prompting or retry before invoking tool handlers.

## Best Possible Solution

Add a core dispatch-level validator that can reject missing required fields from normalized tool schemas before invoking handlers, while preserving richer per-tool validators. Treat automatic retry or prompting as a separate opt-in policy after maintainer review of loop behavior and plugin/client-tool contracts.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A focused current-main source check reproduces the remaining gap: exec missing command is rejected inside createExecTool.execute, read/write/edit use targeted wrappers, and no config or dispatch layer validates all tool schemas before handler invocation.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Schema-derived pre-dispatch validation is a plausible narrow core seam, but the requested automatic prompting/retry behavior is broader than validation and should be designed separately.
