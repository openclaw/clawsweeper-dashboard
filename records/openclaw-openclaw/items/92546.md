## Label Justifications

- `P1`: The report describes a user-facing WhatsApp channel regression that deletes linked session credentials and breaks the channel until QR re-pair.
- `impact:data-loss`: The affected files are persistent WhatsApp Baileys credential/session artifacts, and current cleanup can remove them without a recovery copy.
- `impact:session-state`: The issue is specifically about losing the linked WhatsApp Web session state across plugin update/reconnect cleanup.
- `impact:crash-loop`: The reporter observed the gateway entering an auto-restart crash loop after the credentials were cleared.

## Summary

Keep this open: current main and the latest v2026.6.6 release still delete the managed WhatsApp Baileys auth directory on terminal logged-out/conflict cleanup, and the only active fix path is the unmerged https://github.com/openclaw/openclaw/pull/92948, which still needs patch/proof follow-up.

## What This Changes

The issue asks OpenClaw to preserve or safely recover WhatsApp Baileys session credentials when a plugin update/reconnect path triggers credential cleanup.

## Best Possible Solution

Land a WhatsApp-owned mitigation that preserves or safely restores the Baileys auth folder before terminal reconnect cleanup can delete it, with regression coverage and redacted real WhatsApp/plugin-update proof, while leaving the broader SQL auth-state migration to https://github.com/openclaw/openclaw/issues/9544.

## Reproduction Assessment

Yes, at source level: current main routes terminal WhatsApp logged-out/conflict cleanup through `logoutWeb()`, which deletes the managed auth directory. I did not perform a live WhatsApp plugin-update repro in this read-only review.

## Solution Assessment

Yes for a narrow WhatsApp auth-store/reconnect fix; closing now is not the best path because the active fix PR is still unmerged and reviewed as needing changes/proof. The broader Baileys SQL migration is useful but should remain separate from this immediate session-loss regression.
