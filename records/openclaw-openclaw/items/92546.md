## Label Justifications

- `P1`: A channel-scoped upgrade regression can delete WhatsApp session credentials and break the user's linked WhatsApp workflow until QR relink.
- `impact:data-loss`: The issue reports deletion of persisted WhatsApp Baileys credential files during plugin update.
- `impact:session-state`: The affected files determine whether the WhatsApp Web session remains linked across restarts and plugin updates.
- `impact:crash-loop`: The reported post-update state enters an auto-restart loop after credentials are cleared.

## Summary

Keep open. Current main and the latest release still delete the managed WhatsApp auth directory from `logoutWeb`, and https://github.com/openclaw/openclaw/pull/92948 is paired but unmerged and still needs review/proof.

## What This Changes

The issue requests preventing `openclaw plugins update whatsapp` from deleting WhatsApp Baileys session credentials and forcing QR relink after plugin updates.

## Best Possible Solution

Fix the bundled WhatsApp plugin so update/reconnect terminal cleanup preserves or recoverably backs up active Baileys credentials while explicit user logout still deletes them, then validate plugin-update, startup recovery, and logout paths.

## Reproduction Assessment

Yes at source level, but not as a live replay. `clearTerminalWebAuthState` reaches `logoutWeb`, and `logoutWeb` still deletes the owned auth directory and emits the reported cleanup log line on current main.

## Solution Assessment

No, not if the fix backs up every logout. The safer solution is scoped preservation or recovery for unintended update/reconnect cleanup while keeping documented explicit logout deletion semantics intact.
