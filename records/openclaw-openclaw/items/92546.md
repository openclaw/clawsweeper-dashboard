## Label Justifications

- `P1`: The reported regression could silently destroy a linked WhatsApp session and break a real channel workflow until QR re-pair.
- `impact:data-loss`: The issue is about deletion of persisted WhatsApp Baileys credential files during an update/restart path.
- `impact:session-state`: The affected data is the WhatsApp linked-session state that determines whether the channel remains paired.
- `impact:crash-loop`: The report says the gateway entered an auto-restart crash loop after the credentials were cleared.

## Summary

Current main removes the destructive WhatsApp monitor terminal-close cleanup and adds regression coverage proving auth files survive logged-out/conflict terminal statuses; the fix is main-only and not in v2026.6.8.

## What This Changes

The issue reports that updating the WhatsApp plugin can trigger gateway cleanup that deletes Baileys session credentials and forces QR re-pair.

## Best Possible Solution

Keep the current main behavior: gateway terminal logged-out/conflict states should stop and ask for relink without deleting auth, while the broader Baileys auth-store redesign remains tracked at https://github.com/openclaw/openclaw/issues/9544.

## Reproduction Assessment

Yes for the historical release/source path: v2026.6.8 called clearTerminalWebAuthState, which called logoutWeb and deleted managed auth. Current main no longer reproduces that path because the monitor no longer clears auth on terminal logged-out/conflict states and has regression coverage for preserved creds.

## Solution Assessment

Yes. Removing implicit gateway cleanup is the narrower maintainable fix than broad backup-and-restore behavior: explicit logout remains destructive, while terminal monitor shutdown no longer deletes session files.
