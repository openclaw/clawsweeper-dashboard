## Label Justifications

- `P1`: A reported WhatsApp upgrade path can silently remove channel session credentials and leave a real messaging workflow unusable until QR re-pairing.
- `impact:data-loss`: The issue is specifically about recursive deletion of Baileys credential files under the managed WhatsApp auth directory.
- `impact:session-state`: The deleted files are the persisted linked-session state that determines whether WhatsApp remains paired.
- `impact:crash-loop`: The report says the gateway enters an auto-restart crash loop after the credentials are cleared.

## Summary

Keep this open. Current main does not directly show `plugins update` deleting WhatsApp credentials, but it does contain a shipped source-backed path where a post-update terminal/logged-out WhatsApp close calls `logoutWeb()` and recursively removes the managed Baileys auth directory, matching the reported log and session loss.

## What This Changes

Investigate a reported WhatsApp plugin update regression where updating WhatsApp appears to trigger Baileys credential deletion and QR re-pairing.

## Best Possible Solution

Make WhatsApp terminal-close/auth cleanup preserve, back up, or explicitly quarantine existing linked-session credentials across plugin updates unless the user intentionally logs out, then prove the full linked-session update/restart path.

## Reproduction Assessment

Yes for the destructive cleanup branch by source inspection: terminal/logged-out WhatsApp close paths call `logoutWeb()`, which removes managed `credentials/whatsapp/<accountId>` and emits the reported log; no live WhatsApp plugin-update repro was run.

## Solution Assessment

Unclear. The likely fix belongs in WhatsApp auth cleanup policy, but maintainers need to decide whether terminal close should preserve, back up, quarantine, or delete session credentials.
