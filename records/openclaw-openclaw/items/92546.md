## Label Justifications

- `P1`: The issue describes a repeatable plugin update path that can destroy WhatsApp credentials and break a real channel workflow until QR re-pair.
- `impact:data-loss`: The reported failure deletes persisted Baileys credential files from the user's OpenClaw credential store.
- `impact:session-state`: The deleted files invalidate the linked WhatsApp Web session state across restart or update.
- `impact:security`: The affected files contain sensitive WhatsApp authentication and signal-key material, so backup and retention behavior needs security-aware handling.

## Summary

This issue should stay open: current `main` and the latest release still delete the managed WhatsApp Baileys auth directory through `logoutWeb`, and the only direct repair path is an open PR that has not landed and currently has review blockers.

## What This Changes

The issue asks OpenClaw to preserve or safely handle WhatsApp Baileys credentials during WhatsApp plugin updates so users do not need to QR re-pair after each update.

## Best Possible Solution

Land a narrow WhatsApp-owned fix that preserves or migrates credentials for update/reconnect cleanup without turning explicit logout into silent credential retention, with tests and redacted real update proof.

## Reproduction Assessment

Yes at source level: the reported log line and auth deletion path are present on current `main`, and Baileys stores the session across files in that directory. I did not run a live WhatsApp plugin update in this read-only review.

## Solution Assessment

Unclear: preserving credentials is the right end state, but maintainers need to choose the exact split between update/reconnect recovery, explicit logout deletion, backup retention, and the broader SQLite auth-state migration tracked separately.
