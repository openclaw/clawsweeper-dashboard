## Label Justifications

- `P1`: The report describes a current WhatsApp channel upgrade workflow that can delete session credentials and leave users unable to reconnect without QR re-pair.
- `impact:data-loss`: The central symptom is deletion of Baileys credential files under the WhatsApp auth directory.
- `impact:session-state`: The deleted credentials are the persisted WhatsApp Web session state needed to remain linked across restarts and plugin updates.
- `impact:crash-loop`: The report says the gateway enters an auto-restart crash loop after the credentials are cleared.

## Summary

Keep open: current main still deletes managed WhatsApp Baileys auth state through `logoutWeb()`, the latest release has the same behavior, and the only active fix path is an open PR that has not merged yet.

## What This Changes

The issue asks OpenClaw to preserve WhatsApp Baileys credentials during WhatsApp plugin updates instead of deleting the session and requiring QR re-pair.

## Best Possible Solution

Land a WhatsApp-plugin-owned fix that preserves or backs up Baileys auth before terminal cleanup, reports failed backups accurately, covers owned and legacy auth dirs, and is proven through a real plugin-update/reconnect path.

## Reproduction Assessment

Partly. Source inspection gives a high-confidence deletion path once WhatsApp terminal-close cleanup calls `logoutWeb()`, but I did not live-reproduce the plugin-update step that triggers that close state.

## Solution Assessment

Unclear for the active PR shape. Backup-before-clear is a narrow acceptable mitigation, but the current open PR still needs reliable backup outcome handling, focused tests, and real plugin-update proof before it is the best fix.
