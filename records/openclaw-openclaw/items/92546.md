## Label Justifications

- `P1`: The report describes a repeatable plugin-update regression that silently removes WhatsApp session credentials and breaks a real channel workflow until QR re-pair.
- `impact:data-loss`: The affected state is the Baileys credential file set used to preserve a linked WhatsApp session.
- `impact:session-state`: After the cleanup path runs, the channel status becomes not linked and the prior WhatsApp session cannot be reused.
- `impact:crash-loop`: The reporter observed the gateway entering an auto-restart loop immediately after the credential clear.

## Summary

Keep open: current main and v2026.6.6 still remove managed WhatsApp auth state on terminal logout/conflict and log the reported message, while the active fix path is an open PR that still needs code/proof follow-up.

## What This Changes

Prevent `openclaw plugins update whatsapp` and the resulting restart/reconnect path from silently clearing Baileys credentials, preserving or backing up the linked WhatsApp session instead.

## Best Possible Solution

Review and land a corrected version of https://github.com/openclaw/openclaw/pull/92948 or an equivalent WhatsApp-owned mitigation that preserves plugin-update sessions while keeping explicit logout semantics safe.

## Reproduction Assessment

Yes, at source level. The reporter gave concrete steps and the exact log line, and current source reaches `logoutWeb()` from terminal-close reconnect paths where it deletes the managed auth directory; I did not run a live WhatsApp plugin update.

## Solution Assessment

No final solution exists on main. The narrow maintainable path is a corrected backup/restore or migration in the WhatsApp auth cleanup path, with explicit logout still deleting credentials and with real plugin-update proof before merge.
