## Label Justifications

- `P1`: The report describes a real WhatsApp channel workflow breaking after plugin update by losing linked-session credentials until QR re-pair.
- `impact:data-loss`: The issue is about silently deleting WhatsApp Baileys credential files from the user's OpenClaw credential store.
- `impact:session-state`: The affected data is the persisted WhatsApp linked-session/auth state used to keep the channel connected.
- `impact:crash-loop`: The report says the gateway enters an auto-restart crash loop after the credentials are cleared.

## Summary

Keep this open: current `main` and latest `v2026.6.6` still clear WhatsApp Baileys auth state without any pre-logout/update backup path, and the active linked fix PR has not merged yet.

## What This Changes

The issue asks OpenClaw to preserve, warn, back up, or migrate WhatsApp Baileys session credentials when `openclaw plugins update whatsapp` triggers an update/restart.

## Best Possible Solution

Ship a WhatsApp-owned fix that preserves or recoverably backs up Baileys session state during update-triggered terminal reconnects, keeps explicit logout semantics intentional, and makes credential-backup failures observable before deletion.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not with a live plugin update in this read-only review, but source inspection shows the terminal reconnect path reaches `logoutWeb()` and deletes the managed WhatsApp auth directory with no pre-clear backup.

## Solution Assessment

Is this the best way to solve the issue? Keeping this issue open is the right cleanup outcome until https://github.com/openclaw/openclaw/pull/92948 or a better owner-local fix lands; the broader SQL-backed auth-state work in https://github.com/openclaw/openclaw/issues/9544 is related but does not replace the narrow update-regression fix.
