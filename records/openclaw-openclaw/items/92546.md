- `impact:data-loss`: The affected behavior deletes the on-disk Baileys credential/session files for a linked WhatsApp account.

## Summary

Keep this open: current main and the latest release still have a source-visible path where WhatsApp terminal reconnect cleanup calls the destructive logout flow, and the two linked fix PRs were closed unmerged rather than landing a bounded credential-preservation fix.

## What This Changes

The issue reports that `openclaw plugins update whatsapp` can lead to deletion of the Baileys auth directory, forcing QR re-pair after plugin updates.

## Best Possible Solution

Add a maintainer-approved WhatsApp update/reconnect preservation path that keeps explicit logout destructive, backs up or restores only at a bounded proven update/migration boundary, prunes sensitive backups, and includes real `plugins update whatsapp` proof.

## Reproduction Assessment

Yes at source level: current main still routes terminal WhatsApp reconnect cleanup into `logoutWeb`, and `logoutWeb` deletes the owned auth directory. I did not run a live Docker/WhatsApp plugin update, so the exact update-triggered sequence remains unproven end to end.

## Solution Assessment

No current linked solution is the best fix yet. The closed `backupOnClear` PRs are useful prior art, but the maintainable fix needs a narrow update/reconnect boundary that preserves explicit logout semantics and avoids unbounded sensitive credential backups.
