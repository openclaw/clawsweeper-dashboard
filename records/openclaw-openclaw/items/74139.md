## Summary

Keep open. The PR targets a valid disabled-tools false positive, but the current diff is too broad because it suppresses the existing fail-closed runtime `toolsAllow` conflict that current main still tests.

## What This Changes

This PR changes the empty explicit tool allowlist guard to return `null` for `disableTools: true` runs and updates the colocated guard test accordingly.

## Best Possible Solution

Land a narrowed guard change that ignores inherited ambient allowlist sources for intentional no-tool runs while preserving runtime `toolsAllow` conflicts, with regression tests and a changelog entry.

## Reproduction Assessment

Yes, at source level. A direct guard call with `runtime toolsAllow`, no callable tools, `toolsEnabled: true`, and `disableTools: true` is covered by current main and demonstrates the regression introduced by this PR; the original inherited-allowlist false positive follows the same embedded-run guard path.

## Solution Assessment

No. Returning `null` for every `disableTools` run is broader than the bug boundary; the safer solution is to skip only inherited config/profile/agent/group/sandbox allowlists for intentional no-tool calls while keeping explicit runtime `toolsAllow` fail-closed.
