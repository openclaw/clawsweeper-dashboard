## Summary

Keep open. Current main still has the reported no-tool `llm-task` false-positive path, but this PR should not merge as-is because its unconditional `disableTools` bypass also removes the existing fail-closed behavior for runtime `toolsAllow` conflicts.

## What This Changes

This PR adds an unconditional `disableTools === true` early return to `buildEmptyExplicitToolAllowlistError` and changes the colocated test to expect `runtime toolsAllow` plus disabled tools to return `null`.

## Best Possible Solution

Land a narrowed fix that treats inherited config/profile allowlists as harmless for intentional no-tool `llm-task` runs while preserving an error for trusted runtime `toolsAllow` combined with disabled tools; keep the broader `modelRun` path coordinated with #73031/#74810.

## Reproduction Assessment

Yes. A direct guard call with config-derived allowlist sources, no callable tools, `toolsEnabled: true`, and `disableTools: true` reproduces the current false positive; a direct call with `runtime toolsAllow` and `disableTools: true` reproduces the fail-closed contract this PR would remove.

## Solution Assessment

No. The fix direction is right, but an unconditional early return is broader than the bug because it makes runtime `toolsAllow` conflicts silently run with no tools instead of erroring.
