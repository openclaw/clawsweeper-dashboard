## Summary

Keep open. Current main still has the reported false-positive path, but this PR is broader than the bug because it also suppresses the existing runtime `toolsAllow` plus `disableTools` conflict contract.

## What This Changes

The PR makes `buildEmptyExplicitToolAllowlistError` return `null` for all `disableTools` runs and changes the runtime `toolsAllow` disabled-tools test to expect no error.

## Best Possible Solution

Keep the disabled-tool bypass only for inherited ambient allowlist sources while preserving the runtime `toolsAllow` conflict and adding a user-facing changelog entry.

## Reproduction Assessment

Yes, source-level. A direct guard call with inherited allowlist sources, zero callable tools, and `disableTools: true` follows the current-main error path; the PR’s changed runtime `toolsAllow` test demonstrates the broader regression.

## Solution Assessment

No. The fix direction is right, but the unconditional `disableTools` return is not the narrowest maintainable solution because runtime `toolsAllow` plus disabled tools should still fail closed.
