## Summary

Keep open. The reported llm-task allowlist false positive is real, but this PR bypasses the guard for runtime `toolsAllow` plus disabled tools, which current main explicitly keeps fail-closed.

## What This Changes

This PR changes the empty explicit tool allowlist guard to return `null` whenever `disableTools` is true and updates the colocated guard test to expect no error for that case.

## Best Possible Solution

Land a narrowed guard fix that allows inherited config/profile allowlists for no-tool llm-task and model-only runs while preserving runtime allowlist conflicts and adding the required changelog entry.

## Reproduction Assessment

Yes. A direct guard call with `runtime toolsAllow`, no callable tools, `toolsEnabled: true`, and `disableTools: true` is already covered on current main and demonstrates the regression introduced by this PR.

## Solution Assessment

No. The fix direction is valid, but returning `null` for every disabled-tools run is broader than needed; the safer solution should distinguish inherited policy from explicit runtime `toolsAllow`.
