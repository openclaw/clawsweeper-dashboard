## Summary

Keep open. The PR targets a real current-main false positive for `llm-task` disabled-tool runs, but the proposed guard change is broader than the reported failure and removes the existing fail-closed behavior for runtime `toolsAllow` conflicts.

## What This Changes

The PR changes `buildEmptyExplicitToolAllowlistError` to return `null` whenever `disableTools` is true and updates the colocated guard test to expect no error for that case.

## Best Possible Solution

Land a narrowed fix that treats the inherited config/profile allowlist on intentional no-tool `llm-task` runs as valid, while preserving an error for trusted callers that pass runtime `toolsAllow` together with disabled tools. Coordinate any generalized `modelRun` behavior with #73031/#74810 rather than silently broadening this PR.

## Reproduction Assessment

Yes. A direct guard call with non-empty config-derived allowlist sources, no callable tools, `toolsEnabled: true`, and `disableTools: true` reproduces the current false-positive error; the embedded `llm-task` path reaches that shape.

## Solution Assessment

No. The fix direction is right, but returning `null` for every `disableTools` run is not the narrowest maintainable solution because it also bypasses runtime `toolsAllow` conflicts that current tests intentionally fail closed.
