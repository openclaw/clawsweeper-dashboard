## Summary

Keep open: the BYOK discovery bug is still plausible on current main, but this PR is not merge-ready because it reads only ambient env credentials, is based on an older raw-fetch path, and lacks attached real BYOK proof.

## What This Changes

The branch builds Kilocode model-discovery headers from trimmed `KILOCODE_API_KEY` and `KILOCODE_ORGANIZATION_ID`, adds header tests, and moves the raw-fetch allowlist line.

## Best Possible Solution

Rebase onto current main, pass the catalog-resolved Kilocode `discoveryApiKey` into discovery, add auth headers inside `fetchWithSsrFGuard`, keep focused tests, and attach redacted live BYOK proof.

## Reproduction Assessment

Yes. Current main's source shows Kilocode discovery sends only `Accept`, and https://github.com/Kilo-Org/cloud/issues/2384 documents authenticated versus unauthenticated BYOK catalog results; I did not run a live BYOK account flow.

## Solution Assessment

No. Adding discovery auth is the right direction, but the PR should use the catalog-resolved discovery credential and preserve current main's SSRF-guarded request instead of reading only ambient env on the older raw-fetch base.
