## Summary

Keep open. Current main still does not authenticate Kilocode model discovery, so this PR is not obsolete, but the branch is not ready because it reads only process.env credentials and is stale against the guarded fetch path now on main.

## What This Changes

The PR adds Kilocode model-discovery auth headers from `KILOCODE_API_KEY` and `KILOCODE_ORGANIZATION_ID`, extends header tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Update the PR to pass the resolved Kilocode discovery credential from `ProviderCatalogContext` into the current SSRF-guarded discovery request, add focused tests for env and profile/env-ref credentials, and include the required changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main shows `discoverKilocodeModels()` sends only `Accept`; profile-backed auth remains reproducible because the function has no path to receive `discoveryApiKey`.

## Solution Assessment

No. Sending a bearer token is the right direction, but the maintainable fix should use the catalog credential resolver and preserve the current `fetchWithSsrFGuard` network boundary.
