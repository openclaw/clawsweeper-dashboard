## Summary

Keep this PR open: the BYOK discovery-auth problem is not implemented on current main, but the branch still has blocking review issues around credential resolution, stale raw-fetch code, and missing real behavior proof.

## What This Changes

The PR adds Kilocode model-discovery Authorization and organization headers from environment variables, tests that header behavior, and updates a raw-fetch allowlist line.

## Best Possible Solution

Pass the resolved Kilocode discovery credential from the plugin catalog context into `discoverKilocodeModels()` and apply it inside the existing `fetchWithSsrFGuard` request, with focused tests for env/config/profile credential paths and redacted live BYOK proof.

## Reproduction Assessment

Yes. By source inspection, current main sends only `Accept` during Kilocode discovery while the existing catalog context can resolve a `discoveryApiKey`; the related Kilo cloud issue also documents authenticated versus unauthenticated BYOK catalog behavior, though I did not run a live BYOK account flow.

## Solution Assessment

No. Adding a bearer header is directionally right, but the PR should use the resolved discovery credential and preserve the guarded fetch path instead of reading only `process.env` on a stale raw-fetch base.
