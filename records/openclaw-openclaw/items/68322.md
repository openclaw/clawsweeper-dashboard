## Summary

Keep open: current main still sends Kilocode model discovery without auth, so the PR addresses a real BYOK discovery gap, but the branch is conflicting and not merge-ready because it applies env-only auth to the older raw-fetch path and lacks inspectable after-fix BYOK proof.

## What This Changes

The PR adds Kilocode model-discovery headers from `KILOCODE_API_KEY` and `KILOCODE_ORGANIZATION_ID`, adds header-focused tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Rebase onto current main, thread the resolved Kilocode `discoveryApiKey` from the provider catalog context into discovery, add auth headers inside `fetchWithSsrFGuard`, keep focused tests, and attach redacted live BYOK proof.

## Reproduction Assessment

Yes, from source and upstream issue evidence: current main's discovery init sends only `Accept`, while Kilo-Org/cloud issue 2384 documents authenticated versus unauthenticated BYOK catalog results. I did not run a live BYOK account flow in this review.

## Solution Assessment

No. Adding discovery auth is the right direction, but this PR should use the catalog-resolved discovery credential and preserve the current SSRF-guarded request instead of reading only ambient env on the older raw-fetch base.
