## Summary

Keep open: the PR targets a real Kilocode discovery bug that is still visible in current main, but the branch is stale against the guarded fetch path, reads only ambient env credentials, and still needs real after-fix behavior proof before merge.

## What This Changes

The PR adds env-derived Kilocode model-discovery auth headers, extends fetch-header tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Rebase the PR and pass the resolved Kilocode `discoveryApiKey` from `ProviderCatalogContext` into the existing `fetchWithSsrFGuard` discovery request, with focused env/profile credential tests.

## Reproduction Assessment

Yes, by source inspection: current main's Kilocode discovery request sends only `Accept`, and the shorthand catalog path has no way to pass `discoveryApiKey` from profile/config/env resolution into `discoverKilocodeModels()`. I did not run a live BYOK account flow.

## Solution Assessment

No. Adding a bearer header is the right direction, but this branch should use the catalog credential resolver and preserve the current SSRF-guarded fetch boundary instead of reading only `process.env` on an outdated raw-fetch base.
