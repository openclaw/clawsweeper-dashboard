## Summary

Keep open: this is a plausible Kilocode BYOK discovery bug fix, but the branch is not merge-ready because it is stale against current main's guarded discovery path, reads only ambient environment variables instead of OpenClaw's resolved discovery credential, and still lacks posted real behavior proof.

## What This Changes

The PR adds Kilocode model-discovery Authorization and organization headers from environment variables, tests those headers, and updates the raw-fetch allowlist line.

## Best Possible Solution

Thread the resolved Kilocode `discoveryApiKey` from `ProviderCatalogContext` into `discoverKilocodeModels()` and apply the auth headers inside the existing `fetchWithSsrFGuard` request, with focused tests and redacted live BYOK proof.

## Reproduction Assessment

Yes. Current main's Kilocode discovery request sends only `Accept`, and the related Kilo-Org/cloud issue documents authenticated versus unauthenticated BYOK catalog results; I did not run a live BYOK account flow in this review.

## Solution Assessment

No. Adding an Authorization header is directionally right, but the current PR should use the catalog-resolved discovery credential and preserve the guarded fetch path instead of reading only ambient env on a stale raw-fetch base.
