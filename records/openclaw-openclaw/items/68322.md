## Summary

Keep open. The PR targets a real Kilocode model-discovery gap, but the current patch only reads `process.env` during discovery, so it can still miss the already-resolved catalog credential used for auth-profile and env-ref flows; it also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds trimmed `KILOCODE_API_KEY` and `KILOCODE_ORGANIZATION_ID` headers to Kilocode plugin model-discovery fetches, adds header tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Keep the narrow Kilocode plugin-path fix, but route the resolved catalog `discoveryApiKey` into model discovery before building the request headers, then document or drop the organization-id env surface according to maintainer intent and include the required changelog entry.

## Reproduction Assessment

Yes. On current main, the fetch-path test and source inspection show model discovery sends only `Accept`; with an auth profile or env-ref key resolved through `ProviderCatalogContext`, the proposed patch can still reproduce an unauthenticated discovery fetch because it reads only `process.env.KILOCODE_API_KEY`.

## Solution Assessment

No. Sending a bearer token is the right direction, but the best implementation is to use the provider catalog credential resolver (`discoveryApiKey`) so env, profile, and env-ref credentials behave consistently with other dynamic provider catalogs.
