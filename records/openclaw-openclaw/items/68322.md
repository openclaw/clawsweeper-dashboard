## Summary

Current main still lacks authenticated Kilocode BYOK discovery, so the PR should remain open, but the branch is not merge-ready because it reads only ambient env credentials, predates the current SSRF-guarded discovery path, and still lacks attached after-fix BYOK proof.

## What This Changes

The PR adds `Authorization` and organization headers from `KILOCODE_API_KEY`/`KILOCODE_ORGANIZATION_ID` to Kilocode model discovery, adds header tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Rebase onto current main, pass the resolved Kilocode `discoveryApiKey` from the provider catalog context into discovery headers inside `fetchWithSsrFGuard`, keep the env fallback if needed for Kiloclaw, and attach redacted live BYOK proof.

## Reproduction Assessment

Yes from source and related reports, with medium confidence. Current main shows unauthenticated Kilocode discovery, and Kilo-Org/cloud#2384 documents BYOK models appearing only when the gateway models request is authenticated; I did not run a live BYOK account flow.

## Solution Assessment

No. Adding discovery auth is the right direction, but this patch should use the catalog-resolved discovery credential and current `fetchWithSsrFGuard` path rather than only reading ambient env on a stale raw-fetch branch.
