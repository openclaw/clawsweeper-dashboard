## Summary

Keep open: the bug is still present on current main, and this PR is not merge-ready because it reads only ambient env vars, is based on an outdated raw-fetch discovery path, and lacks posted real behavior proof.

## What This Changes

The PR adds Kilocode model-discovery Authorization and organization headers from environment variables, with fetch-header tests and a raw-fetch allowlist line update.

## Best Possible Solution

Pass the resolved Kilocode discovery credential from `ProviderCatalogContext` into `discoverKilocodeModels()` and apply it inside the current `fetchWithSsrFGuard` request, with focused env/profile/config coverage and redacted live BYOK proof.

## Reproduction Assessment

Yes, by source inspection: current main's Kilocode discovery request sends only `Accept`, while the existing credential resolver can provide a `discoveryApiKey` that never reaches `discoverKilocodeModels()`. I did not run a live BYOK account flow.

## Solution Assessment

No. Adding a bearer header is directionally right, but this branch should use the resolved `discoveryApiKey` from `ProviderCatalogContext` and preserve `fetchWithSsrFGuard` instead of reading `process.env` in a stale raw-fetch path.

## Review Findings

  - confidence: 0.86

Rebase and add the auth headers inside the guarded request so a credential-bearing model discovery call does not bypass the SSRF guard.
  - confidence: 0.87
