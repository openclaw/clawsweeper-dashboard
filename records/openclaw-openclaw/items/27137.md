## Summary

Keep open. Current main still lacks the requested unified `authorizeIngress()`-style contract: shared DM/group helpers exist, and the message-turn kernel centralizes message turns, but representative non-message ingress paths still authorize through channel-local code. This remains a security-sensitive architecture RFC needing maintainer/secops and SDK design approval rather than cleanup closure.

## What This Changes

The issue requests a unified ingress authorization gate that centralizes policy decisions across channel and event ingress surfaces while leaving channel-specific identity/context adapters in plugins.

## Best Possible Solution

Approve a backwards-compatible security/SDK ingress contract, centralize policy evaluation behind channel-owned adapters, and migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract tests.

## Reproduction Assessment

Not applicable as a single runtime reproduction because this is an architecture RFC. The structural verification is high-confidence: the requested contract is absent, and current docs/tests/source still keep representative non-message events plugin-local or channel-local.

## Solution Assessment

Unclear until maintainer/secops review approves the contract shape. A shared ingress gate is a plausible direction, but the maintainable solution must be additive, versioned where public SDK surfaces change, and preserve channel-specific identity normalization.
