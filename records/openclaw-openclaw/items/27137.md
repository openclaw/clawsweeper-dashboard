## Summary

Keep open. Current main still lacks the requested unified ingress authorization gate, and representative message-adjacent and non-message ingress paths continue to authorize through lower-level shared helpers or channel-local code. The existing maintainer discussion is consistent with this being a security/design RFC rather than a cleanup-close candidate.

## What This Changes

Request to add a shared `authorizeIngress()` security/SDK contract that centralizes DM/group/allowlist authorization for channel and event ingress while leaving channel identity and context normalization adapter-owned.

## Best Possible Solution

Keep the RFC open until maintainers and secops approve an additive shared ingress authorization contract. The end state should centralize policy decisions while preserving channel-specific identity/context adapters, then migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract tests.

## Reproduction Assessment

Not applicable as a single runtime reproduction because this is a security architecture RFC. There is high-confidence structural verification: the requested gate is absent, and representative ingress paths still authorize locally or through narrower helpers.

## Solution Assessment

Unclear until maintainer/secops design review approves the contract shape. A shared additive gate is plausible, but the maintainable solution must preserve plugin SDK compatibility and channel-specific normalization rather than forcing one unsafe generic adapter.
