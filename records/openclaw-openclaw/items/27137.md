## Summary

Keep open. Current main still lacks the requested unified `authorizeIngress()` contract, and representative non-message ingress paths continue to authorize through lower-level shared helpers or channel-local code; this remains a security-sensitive architecture RFC needing maintainer/secops design approval.

## What This Changes

Request an additive shared `authorizeIngress()` gate that centralizes DM/group allowlist and policy decisions across channel/event ingress while leaving identity and context normalization to channel adapters.

## Best Possible Solution

Approve a backwards-compatible security/SDK ingress contract, centralize policy evaluation behind channel-owned adapters, and migrate representative message, reaction, pin, interaction, callback, slash, voice, and media ingress paths with contract tests.

## Reproduction Assessment

Not applicable as a single runtime reproduction because this is an architecture RFC. The structural verification is high-confidence: the requested symbols are absent, and docs/tests still keep non-message events plugin-local.

## Solution Assessment

Unclear until maintainer/secops review approves the contract shape. A shared gate is plausible, but the maintainable solution must be additive and preserve plugin SDK compatibility plus channel-specific identity normalization.
