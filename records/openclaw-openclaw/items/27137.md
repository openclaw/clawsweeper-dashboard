## Summary

Keep open: current main still lacks the requested unified `authorizeIngress()` contract. The codebase has useful shared policy primitives and per-surface hardening, including the merged Discord voice allowlist work, but ingress authorization is still assembled per channel and event type, and a maintainer review in the issue already kept this RFC open for security design follow-up.

## What This Changes

The issue requests a unified ingress authorization gate that centralizes cross-channel policy decisions while leaving channel-specific identity and context extraction to adapters.

## Best Possible Solution

Keep the RFC open until maintainers approve an additive security/SDK contract, then migrate representative bundled plugin ingress paths to the shared policy decision point with channel-specific identity adapters and regression tests.

## Reproduction Assessment

Not applicable. This is a security architecture RFC for a new cross-channel authorization contract; source inspection confirms current main still has lower-level helpers and per-channel authorization paths rather than a single failing reproduction path.

## Solution Assessment

Unclear until maintainer/security design approval. The requested direction is plausible, but the maintainable fix should be an additive contract with channel adapters and staged migration tests, not a narrow automated rewrite.
