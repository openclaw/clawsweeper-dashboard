## What This Changes

The branch filters Docker and browser sandbox read-only workspace skill mounts when a configured bind uses the same container destination, and updates the related tests to expect user-bind precedence.

## Best Possible Solution

Fix duplicate Docker targets with one shared bind-collision policy that preserves protected read-only skill mounts, filters or rejects colliding custom binds across shell and browser paths, reuses the existing bind parser, and includes redacted live Docker proof.

## Reproduction Assessment

Yes from source, but not from live Docker in this review. Current main appends configured binds before read-only skill mounts in both sandbox create paths, so an overlapping destination can emit duplicate `-v` targets when the reserved-target override is enabled.

## Solution Assessment

No. Filtering duplicate targets is the right area, but dropping the protected skill mount is not the best fix because current source and tests require protected skills to remain read-only; the colliding custom bind should be filtered, rejected, or handled by a shared policy instead.
