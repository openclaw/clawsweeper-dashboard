## Summary

Keep open: this targets a valid duplicate Docker mount bug, but the branch is not merge-ready because it lets custom binds shadow protected read-only skill mounts, uses ad hoc bind parsing despite an existing Windows-aware parser, and lacks live Docker proof.

## What This Changes

The PR filters read-only workspace skill mounts out of shell and browser Docker create args when configured binds target the same container path, and updates two sandbox tests to expect that precedence.

## Best Possible Solution

Preserve protected read-only skill mounts and filter or reject only colliding custom binds through a shared parser used by shell and browser sandbox paths, then add live Docker proof.

## Reproduction Assessment

Yes, from source at medium confidence: current main appends custom binds and then protected read-only skill mounts in both sandbox create paths, so an overlapping target can reach Docker twice when reserved-target validation is bypassed. I did not establish a live Docker daemon reproduction in this read-only review.

## Solution Assessment

No. Filtering duplicate targets is the right area, but dropping the protected skill mount is not the best fix because current tests and filesystem bridge policy require protected skills to stay read-only.

- **[P1] Preserve protected skill mounts over conflicting binds:** `src/agents/sandbox/docker.ts:605-607`
  - body: This filter removes the managed read-only skill mount whenever a user bind targets the same container path. Current main explicitly keeps protected skill overlays authoritative so checked-in skills cannot become writable; filter or reject the colliding custom bind instead, and apply the same policy to the browser path.
  - confidence: 0.92
- **[P2] Reuse the existing bind spec parser:** `src/agents/sandbox/docker.ts:600-601`
