## Summary

Keep this PR open. The bug and fix direction are real, but the patch changes a published auth-dir string export into a runtime object, which breaks callers that pass the value to Node path/fs APIs, and the external PR proof is only unit-test/lint output rather than a real profile/gateway behavior check.

## What This Changes

The PR lazily resolves WhatsApp default and account auth directories against the active process environment and adds regression tests for profile state directory isolation.

## Best Possible Solution

Land a WhatsApp-owned fix that defers default auth-dir resolution without changing the primitive string/path contract, with focused regression coverage and redacted real profile-gateway or CLI proof.

## Reproduction Assessment

Yes. Current main is source-reproducible because `WA_WEB_AUTH_DIR` is still computed at import time, and the linked issue provides concrete profile-gateway commands and logs showing credentials under the default state dir.

## Solution Assessment

No. The direction is right, but exporting a class instance as a string is not the safest fix because the published contract and Node path APIs expect a primitive string.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the auth-dir string contract:** `extensions/whatsapp/src/auth-store.ts:62`
  - body: `WA_WEB_AUTH_DIR` is re-exported through the WhatsApp runtime APIs and typed as a string by the web-channel loader. This replaces it with a class instance, and Node path/fs APIs such as `path.join(WA_WEB_AUTH_DIR, ...)` reject string-like objects before calling `toString`, so callers can fail with a TypeError even though the new test only checks `String(...)`. Keep this surface primitive or move the laziness behind a resolver/normalization boundary.
  - confidence: 0.9
