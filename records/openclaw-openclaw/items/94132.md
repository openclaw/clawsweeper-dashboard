Keep open: this PR is a focused candidate for the canonical custom-scheme origin bug, and current main plus v2026.6.8 still compare exact allowlist entries against `URL.origin`. ## What This Changes

The PR updates Gateway origin parsing so hosted custom schemes like `app://localhost`, `electron://localhost`, and `tauri://localhost` can match exact `gateway.controlUi.allowedOrigins` entries, with focused unit coverage.

## Best Possible Solution

Land this or an equivalent narrow current-main parser fix after redacted Gateway proof and a clean rerun, preserving exact configured allowlist semantics while keeping literal `Origin: null` and unlisted custom origins rejected.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 compare configured entries against `URL.origin`, and a Node probe shows custom schemes serialize that origin as `null` while retaining host data.

## Solution Assessment

Yes for the parser layer, with merge-readiness caveats. The PR normalizes only incoming hosted opaque origins and preserves exact configured allowlist strings, which is narrower than reparsing config entries or changing wildcard/host fallback policy.
