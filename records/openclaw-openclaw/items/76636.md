## Summary

Keep open. Current main still has the source-level startup gap for selected context-engine plugins without `activation.onStartup`, and this PR's code direction matches the narrow fix, but GitHub currently reports the branch as dirty/diverged so it needs an update before merge.

## What This Changes

The PR adds selected `plugins.slots.contextEngine` plugins to the gateway startup plan, restores cold plugin-tool registry loading, and adds regression tests plus changelog entries for both fixes.

## Best Possible Solution

Update the branch onto current main, preserve the generic startup-plan fix for selected non-legacy context-engine plugins, and validate both plugin-startup and plugin-tool regression paths before merge.

## Reproduction Assessment

Yes. The failure is source-reproducible on current main with a selected external `kind: "context-engine"` plugin that lacks `activation.onStartup`: the startup planner can omit it, and the later resolver then follows the reported legacy fallback path.

## Solution Assessment

Yes, after rebasing. The maintainable fix is to include the selected non-legacy context-engine slot plugin in the generic startup plan, while the tool-registry fallback also follows the existing standalone runtime loader pattern.
