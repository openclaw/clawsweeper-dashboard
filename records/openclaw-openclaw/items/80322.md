## What This Changes

Reports a Codex beta regression where the app-server strips OpenClaw `exec`/`process` dynamic tools, breaking SSH sandbox-backed shell routing for Codex agents.

## Best Possible Solution

Restore a sandbox-backend-aware Codex execution path: either expose OpenClaw `exec`/`process` for non-native sandbox backends or make Codex-native shell execution honor OpenClaw sandbox backends, with focused SSH/OpenShell regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main strips `exec` and `process`, rejects/removes the old compatibility key, and still documents SSH sandboxing as the backend for OpenClaw `exec`; I did not run a live SSH sandbox in this read-only pass.

## Solution Assessment

No. The current always-strip behavior is too broad for configured non-native sandbox backends; the maintainable fix should preserve Codex native ownership for local tools while keeping remote sandbox routing intact.
