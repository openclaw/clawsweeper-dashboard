## Summary

Keep open: this is protected by the `beta-blocker` label, and current main still matches the reported Codex app-server filtering behavior while OpenClaw's documented SSH sandbox exec path remains separate from Codex-native shell execution.

## What This Changes

The issue reports a Codex app-server regression where OpenClaw dynamic `exec` and `process` are stripped, preventing SSH-backed sandbox agents from using the execution path that honors `tools.exec.host: "sandbox"`.

## Best Possible Solution

Restore a sandbox-backend-aware Codex execution path: either expose OpenClaw `exec`/`process` for non-native sandbox backends or make Codex-native shell execution honor OpenClaw sandbox backends, with focused SSH sandbox regression coverage while preserving native-first local workspace ownership.

## Reproduction Assessment

Yes. Source inspection on current main shows Codex app-server filters out `exec` and `process`, rejects/removes the old compatibility key, and still documents/implements SSH sandbox execution through OpenClaw's exec backend path; I did not run a live SSH sandbox.

## Solution Assessment

No. The current always-strip behavior is too broad for configured non-default sandbox backends; the narrower maintainable fix should preserve Codex-native ownership for local tools while keeping remote sandbox routing intact.
