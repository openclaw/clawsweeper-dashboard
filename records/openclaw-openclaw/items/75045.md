## Summary

Keep this open. Current main still reuses the plugin registry when the existing loader cache key matches, and that key does not include packaged OpenClaw version or package-root identity; the linked open PR is an implementation candidate, not a reason to close the issue before merge.

## What This Changes

The issue reports a bundled browser plugin load failure after upgrading from 2026.4.26 to 2026.4.27 where the Gateway keeps referencing an old versioned plugin-runtime-deps path even after doctor repair and restarts.

## Best Possible Solution

Land PR #75071 or an equivalent shared plugin-loader fix that adds packaged OpenClaw version/package-root identity to the loader cache key with focused regression coverage, while keeping the repair generic to bundled plugin loading rather than browser-specific cleanup.

## Reproduction Assessment

Yes. The report gives a concrete upgrade path, and current main still has the matching source-level gap: versioned runtime-deps roots can change while the in-process loader cache key does not.

## Solution Assessment
