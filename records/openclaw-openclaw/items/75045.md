## Summary

Keep this open. Current main still has a plugin registry cache key that can match across an in-place package upgrade, and open PR #75071 is already linked as the closing implementation candidate, so closing the issue before a fix lands would be premature.

## What This Changes

The issue reports the browser plugin failing after a 2026.4.26 to 2026.4.27 upgrade because the gateway keeps importing from the old plugin-runtime-deps root even after doctor repair and restarts.

## Best Possible Solution

Land PR #75071 or an equivalent generic plugin-loader fix that adds packaged OpenClaw version/package-root identity to the loader cache key, with a regression test for same-path package upgrades. Keep the fix at the shared loader/runtime-deps seam rather than adding browser-only or doctor-only recovery behavior.

## Reproduction Assessment

Yes. The issue gives a concrete upgrade path and failure command, and current main source shows the cache can still be reused without package-version/package-root identity.

## Solution Assessment

Likely yes. A shared loader cache-key invalidation is the narrow maintainable fix because the stale import can occur before browser-specific code or doctor repair logic gets another chance to restage runtime deps.
