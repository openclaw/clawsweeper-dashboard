## Summary

Keep open. Current main still omits session duration/start visibility from the shared status footer even though session start timestamps are already persisted and recoverable; the related timing items do not implement or supersede this narrower footer request.

## What This Changes

Request to add elapsed session duration, or the requester-approved absolute start timestamp, to the shared status footer.

## Best Possible Solution

Add a shared footer segment sourced from `sessionStartedAt`/`resolveSessionLifecycleTimestamps()` so `/status` and `session_status` show the same `started <timestamp>` text unless maintainers explicitly choose elapsed duration.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a status request with `sessionEntry.sessionStartedAt` still reaches `buildStatusMessage()`, whose footer construction only renders the session key and update age/no-activity text.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared status renderer with focused tests, rather than duplicating formatting in the slash command or agent tool path.
