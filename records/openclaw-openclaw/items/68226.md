## Summary

Keep open. Current `origin/main` and the latest release still omit session lifetime visibility from the shared status footer, while the session start timestamp is already persisted or recoverable; adjacent timing work covers different surfaces and does not supersede this request.

## What This Changes

Add session lifetime visibility, either elapsed duration or a started timestamp, to the shared status footer.

## Best Possible Solution

Implement one shared footer segment from `sessionStartedAt` or `resolveSessionLifecycleTimestamps()` so `/status` and `session_status` show the same session lifetime information before the existing update age.

## Reproduction Assessment

Yes. Source inspection is enough: a status request with `sessionEntry.sessionStartedAt` still reaches `buildStatusMessage()`, whose footer construction only renders the session key and update age/no-activity text.

## Solution Assessment

Yes. The narrowest maintainable path is a shared status renderer change with focused footer tests, not duplicated formatting in the slash-command or agent-tool callers.
