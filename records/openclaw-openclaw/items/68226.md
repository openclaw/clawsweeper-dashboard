## Summary

Keep open. Current main and the latest tagged release still render the shared status footer with only the session key plus last-updated/no-activity text, while the session start timestamp already exists in session lifecycle state and docs. This remains a narrow status-rendering feature request suitable for a focused fix PR.

## What This Changes

Requested adding session elapsed duration, or the requester-approved absolute session start timestamp, to the shared status footer.

## Best Possible Solution

Add session-start visibility in the shared status renderer so `/status` and `session_status` produce identical footer output, using `sessionEntry.sessionStartedAt` and the lifecycle fallback when needed. Given the requester follow-up, `started <timestamp>` is the lowest-maintenance display unless maintainers explicitly prefer elapsed duration.

## Reproduction Assessment

Yes. A source-level reproduction is enough: inspect `buildStatusMessage()` on current main and it ignores the available session start timestamp while rendering only `Session: <key> • updated ...` or `no activity`.

## Solution Assessment

Yes. The best fix is a narrow shared-renderer change with focused tests, not duplicated formatting in the slash command or agent tool paths; the safer display choice is the requester-approved absolute start timestamp.
