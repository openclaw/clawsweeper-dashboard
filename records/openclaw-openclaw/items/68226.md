## Summary

Keep open. Current main still omits session lifetime visibility from the shared status footer, while the session start timestamp is already persisted and recoverable; the related timing PRs target different surfaces and do not supersede this request.

## What This Changes

Add elapsed session duration or the requester-approved session start timestamp to the shared status footer.

## Best Possible Solution

Add one shared footer segment sourced from `sessionStartedAt` or `resolveSessionLifecycleTimestamps()` so `/status` and `session_status` show the same `started <timestamp>` text before the existing update age.

## Reproduction Assessment

Yes. Source inspection is enough: a status request with `sessionEntry.sessionStartedAt` still reaches `buildStatusMessage()`, whose footer construction only renders the session key and update age/no-activity text.

## Solution Assessment

Yes. The narrowest maintainable path is a shared status renderer change with focused footer tests, not duplicated formatting in the slash-command or agent-tool callers.
