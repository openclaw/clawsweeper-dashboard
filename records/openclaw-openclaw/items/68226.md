## Summary

Keep this open. Current main and the latest tagged release still render the shared status footer with the session key and last-updated/no-activity text only, while the session start timestamp is already stored and recoverable; this remains a narrow status-rendering feature rather than an implemented or stale cleanup item.

## What This Changes

The issue requests adding current-session elapsed duration or an absolute session start timestamp to the `🧵 Session:` footer shown by `/status` and `session_status`.

## Best Possible Solution

Add session-start visibility in the shared status renderer so `/status` and `session_status` produce identical footer behavior, using `sessionStartedAt` first and the lifecycle fallback when needed, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is enough: `buildStatusMessage()` currently ignores `sessionStartedAt` and only renders the session key plus `updated ...` or `no activity` even when a start timestamp exists.

## Solution Assessment

Yes. The narrowest maintainable solution is to update the shared status rendering path rather than duplicating formatting in `/status` or `session_status`; the requester-approved absolute `started <timestamp>` format avoids live duration recalculation if maintainers prefer it.
