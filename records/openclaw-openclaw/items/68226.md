## Summary

Keep open. Current main and v2026.5.18 still render the shared status footer with only the session key plus update age/no-activity text, while the session lifecycle timestamp already exists and the related timing items cover different surfaces.

## What This Changes

The issue requests a session elapsed-duration or absolute session-start segment in the shared status footer.

## Best Possible Solution

Render a single `started <timestamp>` segment from the existing session lifecycle timestamp in the shared status renderer before the existing updated/no-activity segment, with fallback to the transcript session header for legacy rows.

## Reproduction Assessment

Yes. Source inspection on current main shows a status request with a session entry reaches buildStatusMessage(), whose footer only renders the session key plus updated/no-activity text despite the available sessionStartedAt source.

## Solution Assessment

Yes. The maintainable path is to update the shared status renderer from resolveSessionLifecycleTimestamps()/sessionStartedAt so `/status` and `session_status` stay aligned, rather than appending text in individual callers.
