## Summary

Keep open. Current main and the latest release still render the shared status footer with only the session key plus update age/no-activity text, while the session start timestamp already exists and the related timing items cover different surfaces.

## What This Changes

Request to add session elapsed duration or the requester-approved absolute start timestamp to the shared status footer.

## Best Possible Solution

Add one shared footer segment from the existing lifecycle timestamp source so `/status` and `session_status` show consistent session start visibility without duplicating formatting in command/tool callers.

## Reproduction Assessment

Yes. Source inspection on current main is enough: a status request with `sessionEntry.sessionStartedAt` still reaches `buildStatusMessage()`, whose footer only renders the session key plus update age or no-activity text.

## Solution Assessment

Yes. The narrow maintainable path is to update the shared status renderer using `sessionStartedAt`/`resolveSessionLifecycleTimestamps()`, not the separate current-time or per-reply response-duration surfaces.
