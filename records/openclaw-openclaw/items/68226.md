## Summary

Keep open. Current main still renders the shared status footer with only the session key and update age, while session start timestamps already exist in session state and the related timing items do not cover this operator-facing footer request.

## What This Changes

Add session elapsed duration or the requester-preferred absolute start time to the shared status footer.

## Best Possible Solution

Add a shared footer segment sourced from `sessionStartedAt`/`resolveSessionLifecycleTimestamps()` so `/status` and `session_status` consistently show `started <timestamp>` before the existing update-age text.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a status request with `sessionEntry.sessionStartedAt` reaches `buildStatusMessage()`, but the footer construction only renders the session key and update age/no-activity text.

## Solution Assessment

Yes. The narrowest maintainable path is to update the shared status renderer once, with focused coverage for both `/status` and `session_status`, instead of adding separate formatting at each caller.

## Repair Work Prompt

Implement this issue as a narrow shared status footer update. Current main builds `sessionLine` in `src/status/status-message.ts` from only the session key and `updated ...`/`no activity`, while `SessionEntry.sessionStartedAt` and `resolveSessionLifecycleTimestamps()` already provide a start timestamp. Add a footer segment before the update-age text using the requester-preferred absolute form, for example `started 2026-04-17 09:22 MDT`, falling back cleanly when no start timestamp can be resolved. Keep the change in the shared renderer path so both chat `/status` and the `session_status` tool get the same output. Do not implement response latency from #57442, machine-readable current time from #51683, new config options, or unrelated status-card changes. Add focused regression coverage and a user-facing changelog entry.
