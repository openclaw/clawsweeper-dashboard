## Summary

Keep open. Current main still renders the shared session footer with only the session key plus update age/no-activity text, while the requested session start/duration source already exists and the related timing items cover different surfaces.

## What This Changes

Request adding session lifetime visibility, either elapsed duration or the requester-preferred absolute start timestamp, to the shared status footer.

## Best Possible Solution

Add one shared session-lifetime segment in `buildStatusMessage()` using `sessionEntry.sessionStartedAt` with the existing lifecycle fallback, so `/status` and `session_status` stay consistent.

## Reproduction Assessment

Yes. Source inspection on current main is enough: a status request with session data reaches `buildStatusMessage()`, whose footer currently renders only the session key plus update age or no-activity text.

## Solution Assessment

Yes. The narrow maintainable solution is a shared status-renderer change using the existing session lifecycle timestamp source, not duplicated formatting in the slash-command or agent-tool callers.

## Repair Work Prompt

Implement https://github.com/openclaw/openclaw/issues/68226 as a narrow shared status-footer feature. Add a session lifetime segment to `buildStatusMessage()` in `src/status/status-message.ts`, using `sessionEntry.sessionStartedAt` and, for legacy rows, `resolveSessionLifecycleTimestamps()` with the existing `agentId`/`sessionStorePath` context. Prefer the requester-approved `started <timestamp>` display unless the surrounding status format clearly favors compact duration; in either case place the segment before the existing `updated ...` text. Do not add a config option, do not duplicate formatting in `session_status` or slash-command callers, and do not mix in per-reply response-duration work from https://github.com/openclaw/openclaw/pull/49762 or machine-readable current-time work from https://github.com/openclaw/openclaw/pull/51683. Add focused regression coverage for the shared renderer and at least one caller path, and add a single active-version changelog entry because the footer is user-visible.
