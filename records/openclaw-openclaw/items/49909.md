## Summary

Keep open: current main and the latest release still omit `docker-compose.override.yml` from `_clawdock_compose()`, so `clawdock-start` can recreate the gateway from only the base and extra Compose files. Docker’s Compose docs confirm that explicit `-f` files define the merge set/order, while default override auto-loading is the no-`-f` path. ([docs.docker.com](https://docs.docker.com/compose/how-tos/multiple-compose-files/merge/))

## What This Changes

ClawDock should preserve user Docker Compose override-file config when `clawdock-start` uses explicit `-f` compose arguments.

## Best Possible Solution

Update `_clawdock_compose()` to include the user override file after the base and existing extra overlay, with focused coverage for Compose file ordering and docs only where supported overlay files are enumerated.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `_clawdock_compose()` always passes explicit `-f` files but never appends the override file, and `clawdock-start()` goes through that helper. Docker’s docs make the explicit file list the merge set when `-f` is used. ([docs.docker.com](https://docs.docker.com/compose/how-tos/multiple-compose-files/merge/))

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing optional Compose-file pattern for `docker-compose.override.yml`; the open sandbox-overlay PR is related but does not solve this standard override-file case.
