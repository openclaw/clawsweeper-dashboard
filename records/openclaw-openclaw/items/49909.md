## Label Justifications

- `P2`: The bug affects a real Docker helper recreate workflow but is scoped to ClawDock users rather than the core runtime.
- `impact:data-loss`: User-supplied deployment configuration can be silently dropped from recreated containers when the override file is not passed to Compose.

## Summary

Keep this open: current main and the latest release still build ClawDock commands from only `docker-compose.yml` plus optional `docker-compose.extra.yml`, so a user-provided `docker-compose.override.yml` remains omitted from recreate flows.

## What This Changes

ClawDock should include a standard `docker-compose.override.yml` when `clawdock-start` and related helpers pass explicit Compose files.

## Best Possible Solution

Add `docker-compose.override.yml` to the ClawDock compose-file list in a documented order, preserving existing `docker-compose.extra.yml` behavior and adding focused regression proof for the generated command.

## Reproduction Assessment

Yes for source-level reproduction: current main's `_clawdock_compose()` never appends `docker-compose.override.yml`, so `clawdock-start` cannot pass that file to Docker Compose. I did not run Docker in this read-only review, but the source and Docker Compose contract make the failing path clear.

## Solution Assessment

Yes. The narrow maintainable fix is to append an existing `docker-compose.override.yml` through the same helper seam as `docker-compose.extra.yml`, document the supported order, and cover the command construction with focused proof.
