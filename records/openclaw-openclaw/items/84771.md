## Summary

Keep open: the report is recent, externally confirmed on v2026.5.20, and current main still has the implicated startup sidecars and synchronous session-store load path without proof that the Discord READY window is protected under load.

## What This Changes

The issue asks OpenClaw to prevent startup event-loop stalls from model prewarm, session-lock cleanup/recovery, and immediate agent work so Discord readiness and message delivery are not starved.

## Best Possible Solution

Adopt a maintainer-approved startup readiness policy that keeps heavyweight prewarm, provider-auth warmup, recovery, and session-store work outside the Discord handshake window, then prove loaded startup with liveness metrics and real channel readiness.

## Reproduction Assessment

No, not for the full current-main 28-64s symptom; I did not run a live loaded Discord startup. Source inspection and two field reports show the named sidecars and synchronous session-store path still exist and plausibly match the failure.

## Solution Assessment

Unclear: deferring prewarm is likely necessary, but the durable fix needs a maintainer decision on startup readiness, session recovery timing, and how much work may run before providers are READY.
