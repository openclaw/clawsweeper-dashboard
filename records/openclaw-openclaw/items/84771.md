## Summary

Keep open: this is a fresh external report with concrete production timing data, and current `main` still schedules model prewarm alongside channel startup and post-ready session-lock cleanup that can touch synchronous session-store parsing. The exact 28-64s loaded Discord symptom still needs live proof, but there is not enough evidence to close it as fixed, duplicate, stale, or out of scope.

## What This Changes

The issue requests fixing startup event-loop stalls from model prewarm and session-lock cleanup that can delay Discord READY, typing indicators, session recovery, memory pressure, and restart behavior.

## Best Possible Solution

Define and implement a startup readiness policy that moves heavyweight prewarm and recovery work outside Discord’s handshake window, bounds synchronous session-store work, and proves loaded Discord startup with liveness metrics.

## Reproduction Assessment

No high-confidence live reproduction for the full 28-64 second symptom was established here. Source inspection does show the named sidecars and synchronous session-store parse path still exist on current `main`, so the report remains plausible and actionable.

## Solution Assessment

Unclear. Deferring or lazy-running model prewarm is likely part of the solution, but startup readiness, restart recovery, and queued agent work need a maintainer decision before a narrow patch is safe.
