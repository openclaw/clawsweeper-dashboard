## Summary

Keep open: current main and the latest release contain several partial mitigations, but they do not prove the full startup/session-lock/Discord readiness problem is fixed, and active related PRs only cover parts of the cluster.

## What This Changes

The issue reports startup event-loop saturation from model prewarm, session-lock cleanup, and queued agent work causing Discord readiness failures, message delays, memory pressure, and restart cascades.

## Best Possible Solution

Adopt a maintainer-approved startup readiness policy that keeps heavyweight prewarm and session recovery work out of channel handshake windows, then prove it with a loaded real gateway startup and liveness metrics.

## Reproduction Assessment

Partly: current source still contains the implicated startup sidecars and synchronous store-load pieces, and the issue has multi-platform logs. I did not reproduce the full loaded Discord startup stall on current main.

## Solution Assessment
