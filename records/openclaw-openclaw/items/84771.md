## Summary

Keep open: the report is recent, has a second live confirmation on v2026.5.20, and current main still has the implicated startup and session-store paths without proof that loaded Discord startup is protected from event-loop stalls.

## What This Changes

The issue asks OpenClaw to prevent startup event-loop saturation from model prewarm, session-lock cleanup, and recovery work so Discord readiness and message delivery are not delayed or dropped.

## Best Possible Solution

Define and implement a maintainer-approved startup readiness policy that keeps heavyweight model/auth warmups and session recovery out of the Discord handshake window, bounds synchronous session-store work, and proves loaded startup with liveness metrics.

## Reproduction Assessment

No high-confidence live current-main reproduction was run here. Source inspection shows the named prewarm, session-lock, synchronous session-store, and Discord readiness paths still exist, and a second reporter confirmed the same signature on v2026.5.20.

## Solution Assessment

Unclear: deferring or lazily running model/auth prewarm and bounding session recovery are likely parts of the fix, but the safest permanent behavior needs a maintainer startup-readiness decision and live Discord proof.
