## Summary

Keep open: the report is recent, externally filed, and the implicated startup surfaces still exist on current `main`; current code has partial mitigations but does not prove the Discord handshake window is protected from model prewarm, session-lock cleanup, or restart recovery work.

## What This Changes

The issue reports startup event-loop saturation from model prewarm, session-lock cleanup/recovery, and immediate channel/agent work that can delay Discord readiness and restart recovery.

## Best Possible Solution

Adopt a maintainer-approved startup readiness policy that moves heavyweight prewarm and recovery work outside the Discord handshake window, bounds synchronous session-store work, and proves loaded Discord startup with liveness metrics.

## Reproduction Assessment

No, not for the full 28-64 second symptom: I did not run a live loaded Discord startup. Source inspection does show the named startup sidecars and synchronous session-store parse path still exist on current `main`, so the report is plausible and actionable.

## Solution Assessment

Unclear: deferring or lazy-running model prewarm is likely part of the fix, but session recovery and queued work need a maintainer decision on startup readiness semantics before a narrow patch is safe.
