## Summary

Keep open. The PR is a plausible focused fix for the live-backup EOF race, but the current head is not mergeable because its volatile filter still misses OpenClaw's current session and cron run-log paths, JSON mode can be prefixed with human retry/skip notices, and the user-visible backup content change lacks docs/changelog coverage.

## What This Changes

The branch adds a backup volatile-file filter, tar EOF retry/backoff with cleanup and enriched errors, `skippedVolatileCount` reporting, command logger plumbing, and focused backup retry/filter tests.

## Best Possible Solution

Revise this PR so the volatile filter covers current `stateDir/agents/<agentId>/sessions/*.jsonl` and `cron/runs/*.jsonl` paths, preserve machine-readable `--json` output by suppressing or rerouting non-fatal notices, document and changelog the changed backup semantics, then merge it as the narrow EOF-race fix while leaving ENOENT and broader exclusion design to their linked items.

## Reproduction Assessment

Yes. The PR-introduced path bug is reproducible by calling `isVolatileBackupPath` for `{stateDir}/agents/main/sessions/live.jsonl` or `{stateDir}/cron/runs/job.jsonl`; the JSON regression is reproducible when `openclaw backup create --json` emits a retry or volatile-skip notice before the JSON payload.

## Solution Assessment

No, not as currently written. Retry-plus-filter is a reasonable narrow fix direction for the tar EOF race, but the filter must match current storage contracts and JSON mode must stay clean before this is the best mergeable solution.
