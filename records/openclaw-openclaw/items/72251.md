## Summary

Keep open: current main still lacks tar EOF retry/volatile-file handling, and this PR is a useful focused fix, but the latest diff still misses the current live session and cron run-log paths that are most likely to reproduce the reported failure.

## What This Changes

This PR adds backup volatile-file filtering, tar EOF retry/backoff handling, enriched backup failures, skippedVolatileCount reporting, command logger plumbing, and focused backup tests.

## Best Possible Solution

Revise the branch so the volatile filter matches current storage contracts, JSON/direct command callers stay clean, and docs/changelog explain the skipped-content/count behavior while leaving the ENOENT deletion race and configurable excludes to their existing follow-ups.

## Reproduction Assessment

Yes. Source inspection shows current main stores live session transcripts under `agents/<agentId>/sessions/*.jsonl` and cron run logs under `cron/runs/*.jsonl`, while the PR predicate does not match those paths.

## Solution Assessment

No, not as currently written. Retry-plus-filter is a narrow maintainable direction for the tar EOF race, but the filter and user-facing docs need to match current backup/session/cron contracts before merge.
