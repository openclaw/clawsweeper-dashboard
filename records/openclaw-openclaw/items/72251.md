## Summary

Keep open. The PR is a useful narrow fix for the tar EOF race, but the current branch still misses the live session and cron paths used by current main, can route retry notices through the same sink as JSON output for direct callers, and lacks the required docs/changelog coverage for a user-facing backup content/API change.

## What This Changes

The PR adds backup volatile-file filtering, tar EOF retry/backoff handling, enriched backup failures, skippedVolatileCount reporting, command logger plumbing, and focused backup tests.

## Best Possible Solution

Revise the branch so the built-in volatile filter matches current storage contracts, JSON callers receive clean JSON, and docs/changelog explain the backup content/API change while leaving ENOENT deletion races and configurable excludes to their existing follow-ups.

## Reproduction Assessment

Yes. Check the branch predicate against `{stateDir}/agents/main/sessions/live.jsonl` and `{stateDir}/cron/runs/job.jsonl`; current main shows those are live append paths, while the PR rules return false.

## Solution Assessment

No, not as currently written. Retry-plus-filter is a reasonable narrow approach for the node-tar EOF race, but it must match current path contracts and preserve JSON/docs/changelog expectations before merge.
