## Summary

Keep open: the branch still misses the main live append paths that trigger the reported backup failure, can emit retry/skip notices on the same stream as JSON output, and needs docs/changelog follow-through before merge.

## What This Changes

This PR adds backup volatile-file filtering, tar EOF retry/backoff handling, enriched backup failures, skippedVolatileCount reporting, command logger plumbing, and focused backup tests.

## Best Possible Solution

Revise the branch so the built-in volatile filter matches current storage contracts, retry/skip notices cannot corrupt JSON callers, and the backup content/API change is documented and changeloged while leaving ENOENT deletion races and configurable excludes to their existing follow-ups.

## Reproduction Assessment

Yes. Source inspection reproduces the remaining PR blocker by comparing the new predicate against current main's `agents/<agentId>/sessions/*.jsonl` and `cron/runs/<jobId>.jsonl` paths, which the proposed rules do not match.

## Solution Assessment

No, not as currently written. Retry-plus-filter is a narrow maintainable fix for the node-tar EOF race, but it must cover the actual live-write paths and preserve JSON/docs/changelog expectations before merge.

- **[P2] Match current live log paths:** `src/infra/backup-volatile-filter.ts:68-75`
  - body: Current session transcripts live under `stateDir/agents/<agentId>/sessions/*.jsonl`, and cron run logs are `cron/runs/<jobId>.jsonl`. This filter only covers `stateDir/sessions` and cron `*.log`, so the files most likely to trigger the reported live-write EOF race remain archived.
