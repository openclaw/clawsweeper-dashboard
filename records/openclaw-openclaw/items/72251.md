## Summary

Keep open: the PR is a plausible fix for the live-backup tar EOF race, but the current head still misses the current session and cron log paths, can emit retry notices through the JSON output sink for plain RuntimeEnv callers, lacks user-facing docs/changelog coverage, and does not include inspectable real behavior proof.

## What This Changes

The PR adds backup volatile-file filtering, tar EOF retry/backoff handling, enriched backup failures, skippedVolatileCount reporting, command logger plumbing, and focused backup tests.

## Best Possible Solution

Revise the branch so the built-in volatile filter matches current storage contracts, retry/skip notices cannot corrupt JSON callers, and the backup content/API change is documented and changeloged while leaving ENOENT deletion races and configurable excludes to their existing follow-ups.

## Reproduction Assessment

Yes for the remaining PR blocker by source inspection: the PR predicate can be compared directly with current main's stateDir/agents/<agentId>/sessions/*.jsonl and cron/runs/<jobId>.jsonl path contracts, which it does not match.

## Solution Assessment

No, not as currently written. Retry-plus-filter is a narrow maintainable approach for the node-tar EOF race, but it must cover the actual live-write paths, preserve JSON output, and document the changed backup contents before merge.
