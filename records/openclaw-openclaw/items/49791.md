## Summary

Keep open. Current main still computes Feishu group command authorization from group/top-level sender allowlists and the shared directive-only path still returns no reply when that authorization is false; the reporter confirmed the allowFrom workaround, but the requested visible rejection is not implemented and the related PR was closed unmerged.

## What This Changes

The issue requests visible handling for Feishu group `/think` commands that are normalized but silently dropped when sender command authorization fails.

## Best Possible Solution

Keep sender authorization strict, but add a narrow non-sensitive rejection for explicitly addressed Feishu group directive-only commands blocked by command auth, with docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a Feishu group `/think medium` without matching sender command authorization reaches dispatch with `CommandAuthorized: false`, then the directive-only branch returns an undefined reply, matching `queuedFinal=false, replies=0`.

## Solution Assessment

Yes. The narrow maintainable solution is visible rejection for addressed blocked group directives, not opening group command execution or changing allowlist semantics.
