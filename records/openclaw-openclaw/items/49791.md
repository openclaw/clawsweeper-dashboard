## Summary

Keep open. Current main still routes Feishu group `/think medium` through group sender command authorization, and when that authorization is false the shared directive-only path returns no reply; the reporter-confirmed allowFrom workaround resolves execution, but the silent blocked-command UX remains unfixed.

## What This Changes

Feishu group chat should return a normal acknowledgement or actionable authorization rejection for `@Bot /think medium`, instead of completing with no reply while the same command works in direct chat.

## Best Possible Solution

Keep group command authorization strict, but return a narrow generic rejection for explicitly addressed blocked directive-only commands and document the required Feishu group sender allowlist.

## Reproduction Assessment

Yes. Source inspection shows a Feishu group `/think medium` without matching sender command authorization reaches dispatch with `CommandAuthorized: false`, then the directive-only branch returns an undefined reply matching `queuedFinal=false, replies=0`.

## Solution Assessment

Yes. The narrow maintainable fix is visible rejection for addressed blocked group directives, not changing `groupPolicy`, `allowFrom`, or command authorization semantics.
