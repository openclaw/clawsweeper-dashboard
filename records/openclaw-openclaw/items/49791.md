## Summary

Keep open. Current main still makes Feishu group `/think` and other directive/command handling depend on explicit sender command authorization, and the unauthorized directive-only path can still complete with no user-visible reply. The reporter’s workaround confirms the root cause is group command authorization, but the remaining UX request for a visible rejection is not implemented, and the linked fixing PR is closed unmerged.

## What This Changes

The issue asks Feishu group messages like `@Bot /think medium` to either work like direct chat or return a visible authorization rejection instead of silently producing no reply.

## Best Possible Solution

Keep the command authorization boundary intact, but add a narrow non-sensitive rejection for explicitly mentioned Feishu group command/directive-only messages blocked by sender command authorization, with docs and regression tests for both unauthorized and authorized group `/think` behavior.

## Reproduction Assessment

Yes. A Feishu group command with no matching per-group or top-level sender command allowlist finalizes with `CommandAuthorized: false`; the shared directive-only path then returns an undefined reply, matching the logged `queuedFinal=false, replies=0` symptom.

## Solution Assessment

Yes, the requested visible rejection is the right direction if implemented narrowly. The safer alternative to making group commands implicitly open is to keep authorization strict and return a generic configuration hint only when the bot was explicitly addressed with a command/directive-only message.

Implement a focused fix for Feishu group command UX from https://github.com/openclaw/openclaw/issues/49791. Observable bug: in Feishu group chat, an explicitly mentioned command/directive-only message such as `@Bot /think medium` is normalized and dispatched, but if the sender is not command-authorized it returns no visible reply (`queuedFinal=false, replies=0`). Expected behavior: keep command authorization strict, but return a generic non-sensitive rejection/configuration hint for explicitly mentioned group command/directive-only messages that are blocked by command authorization. Prefer a generic core seam if it cleanly covers mentioned group command/directive-only text without plugin IDs; otherwise keep the Feishu-specific reply path inside the Feishu plugin using current plugin/runtime APIs. Update docs to replace the blanket silent behavior where this new scoped rejection applies. Add regression coverage for unauthorized Feishu group `/think medium`, authorized Feishu group `/think medium`, and the existing no-reply behavior for unmentioned or normal unauthorized inline text where applicable. Related source PR #49821 is closed unmerged and should be treated only as context, not as implemented.
