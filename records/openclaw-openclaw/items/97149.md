## Summary

Keep open: the spend-accounting idea is useful, and the new PR body adds real terminal/log proof, but the proof also confirms the submitted branch does not record spend without an extra diagnostics allowlist change that is not in the PR. The remaining blockers are maintainer product/API/security decisions around the trusted diagnostics seam, the canonical usage-ledger source of truth, SQLite-backed state, and dependency-graph authorization.

## What This Changes

The PR adds a new `@openclaw/tokenomics` plugin with a startup `model.usage` diagnostics subscriber, JSONL spend ledger, optional `pricing.json`, authenticated Gateway report route, docs, tests, package metadata, shrinkwrap, and root lockfile importer entry.

## Best Possible Solution

Define or sponsor one canonical usage-accounting seam first, then land tokenomics only after it consumes that seam, stores durable state through approved SQLite/plugin-state contracts, resolves dependency authorization, adds routing metadata, and proves the submitted branch end to end without local out-of-branch patches.

Yes. Source inspection shows the added service depends on `ctx.internalDiagnostics?.onEvent`, and current main only grants that capability to `diagnostics-otel` and `diagnostics-prometheus`; the PR body's live proof confirms unmodified main logs that tokenomics cannot record spend.

## Solution Assessment

No. The implementation is a plausible prototype, but the best fix needs maintainer selection of the diagnostics/usage-ledger boundary, approved durable storage, and dependency/security handling before this can be considered the right OpenClaw solution.
