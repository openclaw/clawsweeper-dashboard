## What This Changes

The PR changes backup output resolution so a non-existent `--output` path with no extension is treated as a destination directory, while archive/custom suffix paths remain explicit filenames.

## Best Possible Solution

Keep the shipped atomic backup publication/cleanup path; if maintainers want different `--output` disambiguation for missing paths, open a fresh focused issue with docs and regression tests.

## Reproduction Assessment

No. I did not run a live backup command in this read-only review, but current main’s source and atomic tests show the linked 10-byte temp-file failure is handled; the no-extension directory heuristic is a separate policy change.

## Solution Assessment

No. The best fix for the linked bug is the current atomic publish/cleanup implementation; changing missing no-extension paths to directories should not ride along as an obsolete bug-fix branch without a fresh product decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Do not rewrite explicit no-extension output paths:** `src/commands/backup.ts:120-123`
  - body: `--output` accepts an archive path or a destination directory. This branch rewrites every missing path with no extension into a directory, so a script that intentionally passes `--output /tmp/backup` as an exact archive path would silently write to `/tmp/backup/<timestamp>.tar.gz` instead. The linked temp-archive bug is already fixed on main, so this heuristic needs a separate product decision plus docs/tests before merging.
  - confidence: 0.78
