## Summary

This PR addresses a real backup output-path bug, but it is not mergeable as-is because the branch patches the old command-level resolver while current main resolves backup output paths in the infra implementation.

## What This Changes

The PR branch changes `src/commands/backup.ts` output resolution so missing extensionless output paths become destination directories while existing paths and paths with suffixes remain explicit archive filenames.

## Best Possible Solution

Move the output-path heuristic into `src/infra/backup-create.ts`, rethrow non-`ENOENT` stat failures, add focused regression coverage for missing extensionless output directories and explicit suffix filenames, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call `backupCreateCommand` or `createBackupArchive` with `output` set to a missing extensionless path; current main catches the `ENOENT`, returns that path, and publishes the archive there without appending the generated `.tar.gz` basename.

## Solution Assessment

No. The refined policy in the PR is reasonable, but the branch applies it to the wrong module for current main; the maintainable fix belongs in the active infra resolver and its tests.
