## Summary

Keep this PR open. The reported backup output-path bug still exists on current main, and the PR's intended behavior is useful, but the branch edits the old command-level resolver while the active implementation now resolves output paths in `src/infra/backup-create.ts`.

## What This Changes

The PR changes backup output resolution so missing extensionless output paths are treated as destination directories while existing files and paths with suffixes are treated as explicit archive filenames.

## Best Possible Solution

Implement the heuristic in `src/infra/backup-create.ts`: classify missing extensionless output paths as destination directories, keep existing directories and trailing-slash paths as directories, preserve existing files and non-empty suffixes as explicit filenames, rethrow non-`ENOENT` stat errors, and add focused resolver regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `backupCreateCommand` or `createBackupArchive` with a mocked backup plan and `output` set to a missing extensionless path; current main will return that path as `archivePath` instead of appending the generated `.tar.gz` basename inside it.

## Solution Assessment

No. The PR's policy direction is reasonable, but this branch is not the best current fix because it edits a stale resolver surface; the maintainable fix belongs in the active infra resolver with tests around missing extensionless paths, explicit suffixes, existing directories, and stat errors.
