## Summary

Keep open: the PR addresses a real backup output-path bug, but it is stale against current main. The active backup command now delegates to the infra resolver, and that resolver still treats a missing extensionless output path as the final archive filename instead of a destination directory.

## What This Changes

The branch changes backup output resolution in `src/commands/backup.ts` so missing extensionless output paths become destination directories while existing paths and paths with suffixes remain explicit archive filenames.

## Best Possible Solution

Move the output-path heuristic into `src/infra/backup-create.ts`, rethrow non-`ENOENT` stat failures, add focused regression coverage for missing extensionless output directories and explicit suffix filenames, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `backupCreateCommand` or `createBackupArchive` with `output` set to a missing extensionless path; current main catches the `ENOENT`, returns that path, and publishes the archive there without appending the generated `.tar.gz` basename.

## Solution Assessment

No. The refined PR policy is reasonable, but the branch applies it to the wrong module for current main; the maintainable fix belongs in the active infra resolver and its tests.
