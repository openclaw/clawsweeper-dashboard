## Summary

Keep open. Current `main` still runs the generic `sessions_send` A2A reply-back loop and announce step for normal peer/persistent sends, while the merged skip only covers requester-owned child sessions and the related open PRs address announce routing slices rather than the broader persistent-session lifecycle policy.

## What This Changes

The issue asks OpenClaw to prevent `sessions_send` persistent-to-persistent A2A ping-pong and announce handling from producing duplicate persisted messages.

## Best Possible Solution

Define the persistent-peer `sessions_send` A2A contract, then implement either explicit lifecycle control or a context-aware skip with regression coverage that preserves intentional peer conversation flows and the existing requester-owned child skips.

## Reproduction Assessment

Yes, at source/test level: a normal peer send that does not match the requester-owned child skips still enters `runSessionsSendA2AFlow`, and the current peer test expects two reply steps plus an announce. I did not run a live persistent-session setup in this read-only review.

## Solution Assessment

Unclear. A per-call lifecycle control or context-aware persistent-peer skip could solve the duplicate behavior, but broadly removing A2A would conflict with current documented and tested peer-session semantics without maintainer product/API direction.
