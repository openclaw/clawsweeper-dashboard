## Summary

Keep open. Current main still runs the default `sessions_send` peer A2A reply-back loop and announce step for normal persistent/peer sends; only narrower parent-owned ACP/native child cases are skipped, and current docs/tests still describe peer A2A as intentional behavior.

## What This Changes

The issue asks OpenClaw to prevent or expose control over `sessions_send` A2A ping-pong/announce behavior for persistent-to-persistent sessions so only the direct send and inline reply appear.

## Best Possible Solution

Define and document the persistent-peer A2A policy, then implement the chosen behavior in the `sessions_send` A2A boundary with regression coverage that preserves intentional peer announce workflows.

## Reproduction Assessment

Yes, at source/test level. A normal requester/target `sessions_send` that does not match the parent-owned child skip still enters `runSessionsSendA2AFlow`, and current tests assert peer reply steps plus announce delivery.

## Solution Assessment

Unclear. The narrow code path is well localized, but the current requested directions change public `sessions_send` semantics or schema, so maintainers should choose the product/API policy before implementation.
