## Summary

Keep open. Current main and v2026.5.12 still route normal peer or persistent `sessions_send` calls through the A2A reply-back loop and target announce path; the merged related fixes only cover parent-owned child/native subagent slices, while the requested persistent-peer behavior conflicts with current docs/tests and needs maintainer API/product direction.

## What This Changes

The issue asks for persistent-to-persistent `sessions_send` to return only the direct send and inline reply, without extra A2A ping-pong or announce-generated duplicate messages.

## Best Possible Solution

Keep this as the canonical persistent-peer A2A behavior tracker until maintainers choose a backward-compatible lifecycle rule or explicit per-call control that preserves intentional peer workflows.

## Reproduction Assessment

Yes, at source/test level: a normal peer or persistent send that is not a requester-owned child still enters `runSessionsSendA2AFlow`, and tests assert the reply-step plus announce behavior. I did not run a live persistent-session setup in this read-only review.

## Solution Assessment

Unclear. Broadly disabling A2A is not the best fix because current docs/tests preserve peer conversation semantics; the maintainable path needs an explicit persistent-peer policy or tool control chosen by maintainers.
