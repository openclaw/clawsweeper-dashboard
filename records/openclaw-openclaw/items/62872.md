## Summary

Keep open. Current main still preserves the generic peer and persistent `sessions_send` A2A reply loop plus target announce behavior; the merged ACP-child fix only covers requester-owned ACP background targets, so the broader persistent-session duplicate report is not implemented.

## What This Changes

The issue requests that `sessions_send` between persistent sessions return only the direct target reply, without framework-generated ping-pong turns or announce-step messages causing duplicate delivery.

## Best Possible Solution

Keep this as the canonical tracker for persistent-session duplicate behavior. The desired end state is an explicit, documented `sessions_send` A2A policy or per-call control that prevents unwanted persistent peer follow-up turns while preserving intentional peer announce workflows and the requester-owned ACP child skip.

## Reproduction Assessment

Yes. At source/test level, a normal non-skipped peer or persistent `sessions_send` with the default ping-pong setting starts the target run, then enters the bounded reply loop and announce step; current tests assert that peer behavior rather than suppressing it.

## Solution Assessment

Unclear. A per-call `maxPingPongTurns` or context-aware skip could solve the report, but either changes documented peer A2A semantics or the public tool API, so maintainers should choose the product boundary before a narrow repair PR starts.
