## Summary

Keep open. Current main still runs the documented peer/persistent sessions_send A2A reply-back and announce flow, while the merged ACP-child guard only skips parent-owned child sessions and the open announce-routing PR is narrower than the persistent-session ping-pong policy requested here.

## What This Changes

The issue requests that persistent-to-persistent sessions_send avoid duplicate A2A ping-pong/announce messages or expose a way to disable that flow.

## Best Possible Solution

Keep this as the canonical tracker until maintainers define the persistent-peer sessions_send A2A policy, then implement either context-aware skipping or explicit per-call lifecycle control with regression coverage that preserves intentional peer workflows.

## Reproduction Assessment

Yes, at source/test level. A normal peer send that does not match the parent-owned child skips still enters runSessionsSendA2AFlow, and current tests assert the reply-step plus announce behavior.

## Solution Assessment

Unclear. A per-call max-turn/disable option or context-aware persistent-peer skip could solve the reported duplicate flow, but maintainers need to choose the public A2A semantics before changing behavior that current docs and tests intentionally describe.
