## Summary

Keep open. Current main still runs the default `sessions_send` A2A ping-pong plus announce flow for normal peer/persistent session sends; recent fixes only cover narrower parent-owned ACP/native subagent slices and do not add per-call ping-pong or announce control.

## What This Changes

The issue requests preventing `sessions_send` persistent-session A2A ping-pong/announce turns from creating duplicate messages beyond the direct send and inline reply.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose and document a persistent-peer A2A policy or per-call control that prevents unwanted duplicate follow-up turns without regressing intentional peer announce workflows.

## Reproduction Assessment

Yes, at source/test level. A distinct requester/target `sessions_send` that does not match the parent-owned ACP/native subagent skip still enters `runSessionsSendA2AFlow`, and current tests assert peer reply steps plus announce delivery.

## Solution Assessment

Unclear. A per-call disable/max-turn parameter or context-aware persistent-peer skip could solve the report, but either changes the public `sessions_send` policy and needs maintainer API/product judgment first.
