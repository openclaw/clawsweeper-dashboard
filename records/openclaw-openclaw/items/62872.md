## Summary

Keep open. Current main and the latest release still run the documented peer/persistent `sessions_send` A2A reply-back plus announce flow for normal peer sends; existing fixes only skip requester-owned child sessions or cover narrower announce-routing work.

## What This Changes

Persistent-to-persistent `sessions_send` should avoid duplicate ping-pong/announce messages or expose a way to disable that flow.

## Best Possible Solution

Keep this as the canonical tracker until maintainers define persistent-peer `sessions_send` A2A semantics, then implement a context-aware lifecycle change or explicit per-call control with regression coverage for intentional peer workflows.

## Reproduction Assessment

Yes, at source/test level. A normal peer send that does not match the owned-child skip still enters `runSessionsSendA2AFlow`, and current tests assert reply-step plus announce behavior; I did not run a live persistent-session setup in this read-only review.

## Solution Assessment

Unclear. A context-aware persistent-peer skip or explicit per-call ping-pong control could solve the reported duplicates, but broad removal would conflict with current docs and tests until maintainers choose the public A2A policy.
