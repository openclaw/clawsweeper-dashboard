## Summary

Keep open. Current main still preserves the generic peer `sessions_send` A2A ping-pong plus announce flow; the merged ACP/native child skips do not cover normal persistent peer sessions, so the requested behavior remains a maintainer policy/API decision.

## What This Changes

The issue requests preventing `sessions_send` between persistent sessions from generating duplicate A2A ping-pong and announce messages, potentially via per-call ping-pong control.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose and document the persistent-peer A2A lifecycle, with regression coverage that preserves intentional peer announce workflows while preventing unwanted duplicate follow-up turns.

## Reproduction Assessment

Yes, at source/test level. A normal visible peer `sessions_send` still resolves the global ping-pong count, enters `runSessionsSendA2AFlow`, runs bounded reply steps, and then runs the announce step; current tests and docs preserve that behavior.

## Solution Assessment

Unclear. A per-call disable/max-turn parameter or a context-aware persistent-peer skip could solve the report, but either path changes the public `sessions_send` policy and needs maintainer API/product judgment first.
