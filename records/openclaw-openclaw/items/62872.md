## Summary

Keep open. Current main still runs the documented A2A ping-pong and announce path for normal persistent or peer `sessions_send` targets; newer skips only cover requester-owned ACP or native child sessions, so the remaining work needs a maintainer semantics/API decision.

## What This Changes

Request to prevent persistent-session `sessions_send` calls from generating duplicate A2A ping-pong and announce messages, either through context-aware skipping or per-call ping-pong control.

## Best Possible Solution

Define the persistent-peer A2A policy, then implement either context-aware skipping or explicit per-call control with regression coverage that preserves intentional peer announce workflows.

## Reproduction Assessment

Yes, at source/test level. A normal requester/target `sessions_send` that is not a requester-owned child session still enters `runSessionsSendA2AFlow`, and current tests assert peer reply steps plus announce delivery.

## Solution Assessment

Unclear. The narrow code path is localized, but the requested direction changes public `sessions_send` A2A semantics or schema, so maintainers should choose the policy before implementation.
