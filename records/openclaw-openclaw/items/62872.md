## Summary

Keep open. Current main and the latest release still run the documented peer/persistent `sessions_send` A2A reply-back plus announce flow for normal peer sends; recent fixes only skip requester-owned child sessions or address narrower announce routing, so the remaining behavior needs maintainer product/API direction.

## What This Changes

The issue requests disabling or controlling `sessions_send` A2A ping-pong and announce behavior for persistent-to-persistent sends to avoid duplicate messages.

## Best Possible Solution

Define the persistent peer A2A policy, then implement either a context-aware skip or explicit per-call lifecycle control with regression coverage that preserves intentional peer workflows.

## Reproduction Assessment

Yes, at source/test level, but not by running a live setup in this read-only review. A normal peer send that does not match the owned-child skip still enters `runSessionsSendA2AFlow`, and current tests assert reply-step plus announce behavior.

## Solution Assessment

Unclear. Per-call control or a context-aware persistent-peer skip could solve the reported duplicate flow, but broadly removing A2A would conflict with current docs and tests until maintainers choose the public semantics.
