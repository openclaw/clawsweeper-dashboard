## Summary

Keep open. Current main still runs the generic `sessions_send` ping-pong plus announce flow for normal peer or persistent sends; recent owned-child skips reduce adjacent loops but do not satisfy the broader persistent peer behavior requested here.

## What This Changes

Requests preventing `sessions_send` between persistent sessions from generating duplicate A2A ping-pong/announce messages, either through context-aware behavior or caller control over ping-pong turns.

## Best Possible Solution

Define the intended persistent-peer A2A policy, then implement the chosen behavior in `sessions_send` with docs and regression coverage that preserve owned-child skips and intentional peer announce workflows.

## Reproduction Assessment

Yes, at source/test level. A normal peer or persistent `sessions_send` that is not an owned child target still enters `runSessionsSendA2AFlow`, and current tests assert the ping-pong plus announce behavior.

## Solution Assessment

Unclear. The code path is localized, but current docs and tests intentionally support peer A2A, so the best fix needs a maintainer decision between changing persistent-peer defaults and adding explicit API control.
