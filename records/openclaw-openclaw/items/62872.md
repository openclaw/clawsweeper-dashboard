## Summary

Keep open. Current main still runs the default peer `sessions_send` A2A reply-back loop and announce step for normal persistent/peer sends; the newer parent-owned ACP/native subagent skips do not cover the persistent-peer behavior reported here, and changing that surface needs maintainer policy/API choice.

## What This Changes

Request that `sessions_send` persistent-to-persistent communication avoid extra A2A ping-pong/announce turns or expose control to disable them per send.

## Best Possible Solution

Keep this as the canonical tracker, define and document the persistent-peer A2A policy, then implement the chosen behavior with regression coverage that preserves intentional peer announce flows and the parent-owned child skips.

## Reproduction Assessment

Yes, at source/test level. A normal peer send that does not match the parent-owned ACP/native subagent skips still enters `runSessionsSendA2AFlow`, and current tests assert the peer reply steps plus announce delivery.

## Solution Assessment

Unclear. A per-call max-turn option is plausible, but maintainers should first choose the public persistent-peer A2A semantics because current docs and tests intentionally preserve peer follow-up behavior.
