## Summary

Keep open. Current main still runs the `sessions_send` peer A2A reply-back loop and announce step for normal persistent/peer sends, while the existing skip only covers requester-owned child sessions and the related open PR addresses announce routing rather than the broader persistent-session ping-pong policy.

## What This Changes

The issue requests that `sessions_send` avoid duplicate persistent-session messages from A2A ping-pong/announce flow, or expose control to disable that flow.

## Best Possible Solution

Define and document the persistent-peer `sessions_send` A2A policy, then implement either context-aware skipping or explicit per-call lifecycle control with regression coverage that preserves intentional peer conversation flows and existing parent-owned child skips.

## Reproduction Assessment

Yes, at source/test level. A normal peer send that does not match the parent-owned child skips still enters `runSessionsSendA2AFlow`, and current tests assert the reply-step plus announce behavior.

## Solution Assessment

Unclear. A per-call max-turn/disable option or context-aware persistent-peer skip could solve it, but maintainers should choose the public A2A semantics before changing behavior that current docs and tests intentionally describe.
