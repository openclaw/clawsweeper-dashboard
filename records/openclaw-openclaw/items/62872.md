## Summary

Keep open. Current main still documents and implements the generic `sessions_send` peer A2A reply-back loop plus target announce path for normal or persistent peer sends; merged adjacent work only skips requester-owned child sessions, so the remaining persistent-peer behavior needs maintainer API/product direction rather than cleanup closure.

## What This Changes

The issue requests preventing persistent-session `sessions_send` from generating A2A ping-pong and announce duplicate messages, potentially by adding per-call ping-pong control or context-aware persistent-session behavior.

## Best Possible Solution

Keep this as the persistent-peer A2A lifecycle tracker until maintainers choose an explicit dispatch/no-reply mode or context-aware persistent-session rule, with regression coverage preserving requester-owned child skips and intentional peer workflows.

## Reproduction Assessment

## Solution Assessment

Unclear. Broadly disabling A2A for all peer sessions is not the best maintained fix because current docs/tests preserve peer semantics; the safer path is a maintainer-approved persistent-peer policy or explicit per-call control.
