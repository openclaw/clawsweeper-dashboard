## Summary

Keep open. Current main still runs the default `sessions_send` A2A ping-pong and announce path for normal peer or persistent sessions; the merged ACP work only skips requester-owned ACP child sessions, and the tool schema still has no per-call ping-pong control.

## What This Changes

The issue requests preventing `sessions_send` between persistent sessions from producing duplicate A2A ping-pong/announce messages, or exposing per-call ping-pong control.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose and document an explicit A2A policy or per-call control that prevents unwanted persistent peer follow-up turns while preserving intentional peer announce workflows and the ACP parent-child skip.

## Reproduction Assessment

Yes, at source/test level. A distinct requester/target `sessions_send` still enters `runSessionsSendA2AFlow`, and current tests assert reply-step calls plus target announce delivery for that peer-session shape.

## Solution Assessment

Unclear. A per-call `maxPingPongTurns`/disable option or context-aware persistent-peer skip could solve the report, but changing defaults affects documented peer A2A semantics and needs maintainer product/API judgment.
