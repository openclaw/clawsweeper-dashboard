## Summary

Keep open: current main and the latest release still clone plugin hook context, so the linked bug remains. The patch is a plausible narrow fix, but this PR is draft and still needs after-fix real behavior proof plus maintainer acceptance of the mutable `registerHook` context compatibility change.

## What This Changes

The PR changes `registerHook` wrapping to pass the original hook context to plugin handlers, temporarily inject `pluginConfig`, restore or delete it afterward, and add loader regression coverage.

## Best Possible Solution

Land a narrow plugin-registry fix that preserves documented mutable hook context, restores `pluginConfig` between handlers, and proves the behavior through a real plugin run plus focused tests.

## Reproduction Assessment

Yes, source-reproducible without running tests: current main clones `evt.context` in the plugin wrapper, while `applyBootstrapHookOverrides` reads the original event context after `triggerInternalHook`. A plugin that assigns `event.context.bootstrapFiles` through `api.registerHook` therefore loses the reassignment.

## Solution Assessment

Yes, with merge-gate caveats: preserving the original context while restoring `pluginConfig` matches the documented mutable bootstrap contract and is narrower than redesigning hooks. The PR still needs real-run proof and maintainer acceptance of the compatibility semantics before merge.
