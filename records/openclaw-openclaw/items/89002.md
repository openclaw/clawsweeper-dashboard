## Summary

This PR remains necessary: current main still clones the public hook context and therefore drops top-level mutations before bootstrap consumes the original event. The proposed generic registry fix matches the runner’s explicit sequential execution model, restores the existing mutable contract, and has convincing focused and real-loader proof; no actionable correctness or security defect was found.

## What This Changes

The PR preserves the original mutable context across registered plugin hook handlers, temporarily injects and restores each handler’s `pluginConfig`, and adds regressions for mutation propagation and config cleanup.

## Best Possible Solution

Merge the generic registry-boundary repair after ordinary maintainer approval, keeping the shared mutable context contract and per-handler configuration cleanup rather than adding a bootstrap-specific workaround or field-by-field merge.

## Reproduction Assessment

Yes. Current main deterministically gives the plugin a cloned context and then reads the original context after dispatch, so replacing `bootstrapFiles` cannot propagate through `registerHook`; the linked issue also provides a real-run manifestation.

## Solution Assessment

Yes. Repairing context identity once at the generic plugin registry boundary is narrower and more maintainable than special-casing bootstrap or copying selected fields back, and the sequential runner plus `finally` cleanup supports the implementation.
