## Summary

Keep this PR open: current main still clones the hook context and can drop plugin `agent:bootstrap` mutations, while the documented/core contract expects those mutations to survive. The patch looks correct, the newly supplied real plugin-loader proof satisfies the external proof gate, and the remaining issue is maintainer acceptance of the compatibility-sensitive hook semantics.

## What This Changes

The PR changes plugin `registerHook` wrapping to pass the shared hook context, inject and restore `pluginConfig` per handler, and add loader regression tests for context mutation and config cleanup.

## Best Possible Solution

Land the shared-context fix after maintainers accept the plugin API compatibility tradeoff and normal merge gates pass, preserving the per-handler `pluginConfig` cleanup behavior in this patch.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main clones `evt.context`, while core reads the original `event.context.bootstrapFiles` after hook dispatch. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

Yes, this is the narrowest maintainable fix: pass the original context object to preserve generic hook mutation semantics, while restoring/removing `pluginConfig` in `finally` to keep per-handler config isolated. Special-casing `agent:bootstrap` or adding a new return API would be broader than needed for the documented contract.
