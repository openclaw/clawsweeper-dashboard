## Summary

Keep this PR open: current main still drops `registerHook` context reassignments, and the patch is a narrow plausible fix, but merge still needs real plugin-loader proof and maintainer acceptance of the plugin API compatibility change.

## What This Changes

The PR changes plugin `registerHook` wrapping to pass the original hook context to handlers, temporarily inject `pluginConfig`, restore it afterward, and add regression coverage for `agent:bootstrap` context mutation.

## Best Possible Solution

Land the narrow registry fix after redacted real plugin-loader proof confirms context mutation persistence and per-handler `pluginConfig` cleanup.

## Reproduction Assessment

Yes. Current main clones `evt.context` in the plugin wrapper while `applyBootstrapHookOverrides` reads the original event context after `triggerInternalHook`, so an `api.registerHook` handler that assigns `bootstrapFiles` loses that reassignment.

## Solution Assessment

Yes, with merge-gate caveats. Preserving the original context while restoring `pluginConfig` is the narrowest fix for the documented mutable bootstrap contract; a larger immutable return-value API would be a separate design decision.
