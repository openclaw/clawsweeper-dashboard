## Summary

Keep open for maintainer review: the patch appears to be the narrow registry fix for the lost mutable hook-context bug, with sufficient real loader proof, but it intentionally changes plugin hook compatibility semantics before merge.

## What This Changes

The PR changes plugin `registerHook` wrapping to pass the shared hook context while temporarily injecting/restoring `pluginConfig`, and adds loader regression coverage for context mutation and config cleanup.

## Best Possible Solution

Land the narrow registry fix after maintainer acceptance of the mutable hook-context compatibility tradeoff and normal CI gating; if maintainers reject shared mutation, design an explicit return-value hook API instead of keeping the current mutable-looking clone.

## Reproduction Assessment

Yes from source inspection: current main clones `evt.context` in the plugin hook wrapper, while the bootstrap caller reads the original `event.context.bootstrapFiles` after `triggerInternalHook`. The linked issue also includes real-run evidence of the lost mutation.

## Solution Assessment

Yes, with maintainer acceptance of the compatibility tradeoff; changing the registry wrapper is the narrow owner-boundary fix, while merging fields back risks config leakage and a return-value API would be a larger plugin SDK change.
