## Summary

The PR demonstrates a substantial real latency improvement, but it is not merge-ready: the shared provider-data cache is an unbounded process-lifetime map without a production lifecycle owner, provider-wildcard browse semantics are changed from full discovery to read-only data, and the current head conflicts with main.

## What This Changes

The PR bounds model-catalog waits, reduces repeated provider-auth discovery, caches read-only catalog and shared `/models` provider data, and adds lighter provider-menu loading for Telegram and Discord model pickers.

## Best Possible Solution

Rebase the branch, make shared provider data a bounded active-generation cache cleared by the canonical model/auth hot-reload boundary, and retain or prove complete dynamic discovery for provider-wildcard views.

## Reproduction Assessment

Yes. The linked issue provides repeatable CLI timings, source inspection identifies the repeated catalog/auth path, and the PR supplies real before-and-after Telegram timings for the shared picker path.

## Solution Assessment

No, not in its current form. Sharing in-flight work and bounding catalog waits are appropriate, but the final design needs bounded lifecycle ownership and must preserve the documented provider-wildcard model contract.
