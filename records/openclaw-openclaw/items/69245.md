## Summary

Keep open. Current main still excludes generic `openai-completions` models from cache-TTL eligibility, so the PR addresses a real runtime gap, but broad eligibility for every OpenAI-compatible transport remains a maintainer product-policy decision and the user-facing feature still needs the required changelog entry before merge.

## What This Changes

The PR adds `modelApi === "openai-completions"` to cache-TTL provider eligibility and adds unit tests for OpenAI-compatible providers such as OpenAI, Astron Coding Plan, and DeepSeek.

## Best Possible Solution

Land a policy-explicit version only if maintainers want cache-TTL pruning to apply to all OpenAI-compatible Chat Completions transports; otherwise keep eligibility provider-owned through `isCacheTtlEligible` hooks for providers such as DeepSeek. Either path should keep the existing plugin override priority and include focused tests plus a single-line changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, `isCacheTtlEligibleProvider("deepseek", "deepseek-r1", "openai-completions")` falls through to false because there is no generic `openai-completions` branch and DeepSeek has no provider hook.

## Solution Assessment

Unclear. The PR is the narrowest generic code change and preserves provider-hook overrides, but maintainers should decide whether the product policy should be broad API-type eligibility or provider-by-provider opt-in for cache-backed transports.
