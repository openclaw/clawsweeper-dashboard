## What This Changes

The branch adds a `modelApi === "openai-completions"` fallback to cache-TTL provider eligibility and unit assertions for OpenAI-compatible providers.

## Best Possible Solution

Maintainers should decide whether cache-TTL eligibility is API-family-wide; if yes, land it with real provider proof, otherwise keep support provider-owned through hooks or explicit compatibility signals.

Yes. Source inspection shows an `openai-completions` provider such as DeepSeek reaches `buildContextPruningFactory`, passes its model API into `isCacheTtlEligibleProvider`, and is rejected on current main unless a provider hook opts it in.

## Solution Assessment

Unclear. The diff is narrow and preserves provider hook precedence, but making every `openai-completions` provider eligible is a maintainer policy choice; provider hooks or explicit compatibility flags are safer if support should stay capability-specific.
