## Summary

The Discord symptom matches current documented group/channel visible-reply policy: guild replies are message-tool-only by default, so a plain assistant final in the transcript is intentionally private unless the model uses the message tool or config restores automatic replies.

## What This Changes

The issue asks OpenClaw to treat Discord assistant finals that appear in the transcript but are not posted to the guild channel as a delivery bug.

## Best Possible Solution

Keep the current message-tool-only group/channel default and direct affected operators to use a stronger tool-calling model or set messages.groupChat.visibleReplies to automatic when they want legacy auto-posted finals.

## Reproduction Assessment

Yes, the observed symptom is source-reproducible without a live Discord run: current main routes Discord guild turns to message_tool_only delivery by default and skips ordinary final replies before outbound delivery. A true transport bug would need evidence that message-tool delivery or automatic delivery was attempted and then failed silently.

## Solution Assessment

No, changing core to always deliver these Discord finals is not the best fix because it would override the documented group/channel visibility policy. The maintainable path is the existing config/model choice, with a new bug only if logs show didSendViaMessagingTool: true or another outbound delivery attempt that failed without an error.
