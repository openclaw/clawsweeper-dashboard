## What This Changes

The PR adds a new `tool_error` failover/runtime/auth reason, classifies messages containing `missing_tool_result` as that reason, and preserves transient cooldown probe slots for it.

## Best Possible Solution

Keep the missing_tool_result failure local to the failed turn: either abort at the local-error boundary or explicitly exclude `tool_error` from run failover and auth-profile health, then prove the path with focused tests and redacted real Codex output.

## Reproduction Assessment

Yes at source level: current main synthesizes `missing_tool_result`, classifies prompt errors into failover reasons, and routes concrete non-timeout reasons through fallback/profile policies. I did not run a live long-hanging Codex shell scenario in this read-only review.

## Solution Assessment

No: classifying the sentinel as `tool_error` is plausible, but this patch is not the best complete fix while current run and auth-profile policies still treat the new value as a provider-routing/profile-health signal. A safer shape handles it as a local non-provider error at the fallback boundary or explicitly carves it out everywhere it can be consumed.
