## Summary

Keep open: the reported `/model status` auth-label bug is plausible, but this patch changes shared harness routing for every bare `codex` provider and would bypass existing auto harness support and priority semantics; after-fix real behavior proof is also still missing.

## What This Changes

The PR changes `isOpenAICodexProvider` to treat bare `codex` as `openai-codex` and adds predicate plus harness-policy tests.

## Best Possible Solution

Fix the bare `codex` status auth label in the status-auth planning path or a local status alias while preserving auto harness selection, then prove the visible Telegram output.

## Reproduction Assessment

Yes, by source inspection: current main leaves the bare `codex` status row on `runtime: "auto"`, so `buildAgentRuntimeAuthPlan` does not expose the `openai-codex` harness auth provider for that row. I did not run tests because this review is read-only.

## Solution Assessment

No. Changing the shared OpenAI Codex provider predicate fixes the label through a broad harness-routing side effect; a narrower `/model status` auth mapping would avoid changing provider-based plugin selection.
