## Summary

Keep open: the reported `/model status` bug is valid, but this patch changes shared harness routing so `provider: "codex"` no longer uses the current auto support/priority selection path. The external PR also still needs real after-fix behavior proof before merge.

## What This Changes

The PR changes `isOpenAICodexProvider` to accept bare `codex` and adds predicate and harness-policy unit coverage for that routing.

## Best Possible Solution

Fix the effective auth label for bare `codex` status rows without redefining the shared OpenAI Codex provider predicate used by harness selection, then prove the visible `/model status` output in a real setup.

## Reproduction Assessment

Yes, by source inspection: current main leaves `provider: "codex"` on `runtime: "auto"`, so the status auth fallback never gets an `openai-codex` harness auth provider for that row. I did not run tests because this was a read-only review.

## Solution Assessment

No. Changing `isOpenAICodexProvider` fixes the reported chain too broadly because it also changes `selectAgentHarness` behavior for every bare `codex` provider run; a narrower status-auth alias or runtime-auth planning adjustment would be safer.
