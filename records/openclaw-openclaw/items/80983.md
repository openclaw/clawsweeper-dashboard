## Summary

Keep open: source inspection supports the reported auth-label gap, but the proposed fix changes shared harness routing in a way that conflicts with current auto-selection semantics and lacks real behavior proof.

## What This Changes

The PR changes the OpenAI Codex provider predicate to treat `codex` as an `openai-codex` alias and adds unit tests for predicate and harness-policy routing.

## Best Possible Solution

Fix the bare `codex` status-row auth label in the `/model status` auth-resolution path while preserving provider-based harness auto-selection and plugin priority semantics.

## Reproduction Assessment

Yes. Source inspection shows current main leaves `provider: "codex"` on `runtime: "auto"`, so the status auth fallback never receives `openai-codex` as the effective harness auth provider for that row.

## Solution Assessment

No. Changing the shared OpenAI Codex provider predicate fixes the label by redefining harness routing too broadly; a narrower status-auth fallback or runtime-auth planning adjustment is safer.
