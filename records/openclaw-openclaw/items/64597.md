## Summary

Keep open; the current branch is not safe to merge because the new truncation skip can rely on stale run-loop prompt usage and suppress an existing recovery path, and the PR still has only test-based proof plus failing CI.

## What This Changes

The PR gates overflow recovery truncation retries based on estimated tool-result reduction potential, adds overflow loop coverage, and narrows auto-reply session reset behavior for embedded compaction failures.

## Best Possible Solution

Revise the overflow skip gate to use current-attempt observed or estimated overflow pressure, or preserve fallback truncation whenever usage provenance is uncertain; consider landing the auto-reply reset narrowing separately if maintainers want that policy change.

## Reproduction Assessment

Yes. Source inspection of PR head shows the skip gate derives overflow size from run-loop-scoped lastRunPromptUsage and then marks truncation attempted, while current main still attempts fallback truncation when oversized tool output is present.

## Solution Assessment

No. Avoiding futile truncation is reasonable, but this implementation should use attempt-local overflow evidence or keep the fallback when usage provenance is uncertain; the auto-reply reset behavior should not ride along with a blocked recovery change unless maintainers explicitly want that coupling.
