## Summary

Keep this PR open, but it is not ready to merge: the main overflow-recovery idea is useful, yet the skip gate still relies on stale-prone run-loop usage and can bypass a truncation recovery that current main would attempt.

## What This Changes

The PR adds an estimated tool-result reduction gate to embedded overflow recovery, test coverage for bounded fallback truncation, and narrows auto-reply session resets to embedded compaction failures.

## Best Possible Solution

Keep the reduction-potential gate only when prompt pressure is attempt-local or otherwise fall back to the existing truncation recovery path, then prove the fixed overflow path with real redacted runtime output.

## Reproduction Assessment

Yes, from source inspection. A prior large lastRunPromptUsage can survive into a later overflow attempt with oversized tool output and no fresh usage, making this PR skip truncation that current main would still attempt.

## Solution Assessment

No. The direction is narrow, but the implementation should use current-attempt prompt pressure, such as the precheck estimate, or preserve truncation when usage provenance is uncertain.
