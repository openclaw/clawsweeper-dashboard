## Summary

Keep open: the PR targets a real overflow-recovery problem, but the latest head still skips truncation from loop-level prompt usage that can be stale, so it can bypass a recoverable tool-result truncation path; it also lacks the required changelog entry.

## What This Changes

The PR adds an overflow-recovery skip gate based on estimated tool-result reduction, one-shot truncation coverage, and embedded-error reset narrowing to compaction failures.

## Best Possible Solution

Keep the reduction-potential gate, but only skip truncation when prompt pressure is known to be from the current failed attempt; otherwise preserve the existing truncation fallback and add the Unreleased Fixes entry.

## Reproduction Assessment

Yes. A focused runner harness can reproduce the PR-introduced blocker by carrying a high prior `lastRunPromptUsage`, then returning a current overflow with oversized tool output and no fresh usage, which lets the PR skip truncation even though truncation could recover.

## Solution Assessment

No. The direction is narrow and maintainable, but the current implementation should use current-attempt prompt pressure or fall back to truncation when usage provenance is uncertain.
