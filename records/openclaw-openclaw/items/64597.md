## Summary

Keep open: the PR addresses a real overflow-recovery problem, but the latest head still skips truncation using loop-level `lastRunPromptUsage`, which can be stale for the current failed attempt. The external PR also has only focused test/CI proof, so it needs real after-fix behavior evidence before merge.

## What This Changes

This PR gates embedded-runner overflow recovery by estimated tool-result reduction, adds one-shot truncation coverage, and narrows auto-reply session resets to embedded compaction failures.

## Best Possible Solution

Keep the reduction-potential gate, but only skip truncation when prompt pressure is known to come from the current failed attempt; otherwise preserve the existing truncation fallback and have maintainers handle the release changelog during landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: carry a high prior `lastRunPromptUsage`, then hit a current overflow with oversized tool output and no fresh usage, causing the PR skip gate to bypass truncation that current main would attempt.

## Solution Assessment

No. The direction is narrow and maintainable, but the implementation should use current-attempt prompt pressure or fall back to truncation when usage provenance is uncertain.
