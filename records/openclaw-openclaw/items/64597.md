## Summary

Keep this PR open for maintainer review: current main has not implemented the proposed overflow retry gate, but the latest PR diff still has a blocking correctness issue already raised in review. The skip decision uses stale-prone `lastRunPromptUsage` instead of an attempt-local prompt estimate, so it can skip the only recovery path that would have fixed a current overflow.

## What This Changes

The PR changes embedded runner overflow recovery to estimate whether tool-result truncation can clear a context overflow, carries attempt-local token estimates, adjusts compaction-failure reset handling, and adds focused regression tests.

## Best Possible Solution

Keep the reduction-potential idea, but make the skip decision depend only on a prompt-size estimate known to belong to the current failed attempt, or fall back to attempting truncation when usage provenance is uncertain. The embedded-error reset narrowing can remain if its focused tests continue to prove plain context overflow surfaces a `/new` hint while true compaction failures reset.

## Reproduction Assessment

Yes. A focused runner harness can reproduce the blocking case by carrying a high previous `lastRunPromptUsage`, then returning a current context overflow with oversized tool output and no fresh usage; the PR would skip truncation even though truncation could recover the run.

## Solution Assessment

No. The direction is narrow and maintainable, but this implementation is not the best fix until it uses `attempt.estimatedContextTokens` or another attempt-local estimate for the skip gate, instead of stale loop-level usage.
