## Summary

Keep open. Current main still uses the existing oversized-tool-result truncation fallback and does not implement the PR’s new skip gate, while the latest PR head still bases that skip gate on stale-prone loop-level usage and is dirty against main. The direction is useful, but the branch needs revision before merge.

## What This Changes

The PR changes embedded runner overflow recovery to estimate tool-result reduction before fallback truncation, carry local estimated context tokens into compaction, narrow embedded-error resets to compaction failures, and add focused overflow and auto-reply tests.

## Best Possible Solution

Keep the reduction-potential idea, but make the skip decision depend only on a prompt-size estimate known to belong to the current failed attempt, such as parsed provider overflow tokens or `attempt.estimatedContextTokens`. When usage provenance is uncertain, keep the existing fallback and attempt tool-result truncation.

## Reproduction Assessment

Yes. A focused runner harness can reproduce the blocker by carrying a high previous `lastRunPromptUsage`, then returning a current context overflow with oversized tool output and no fresh usage; the PR can skip truncation even though truncation could recover the run.

## Solution Assessment

No. The proposed direction is narrow and maintainable, but this implementation is not the best fix until the insufficiency gate uses attempt-local prompt pressure or falls back to truncation when provenance is uncertain.
