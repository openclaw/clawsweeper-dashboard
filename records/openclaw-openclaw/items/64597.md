## Summary

Keep this PR open, but do not merge as-is: the overflow truncation skip still uses stale-prone run-loop usage, so it can skip the current fallback truncation path incorrectly, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a tool-result reduction gate to embedded overflow truncation, a bounded truncation regression test/harness update, and narrows auto-reply session resets to embedded compaction_failure payloads.

## Best Possible Solution

Use attempt-local prompt pressure, such as a precheck or current-attempt estimate, before skipping truncation; when usage provenance is uncertain, keep the existing truncation fallback and require redacted real overflow-recovery proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: lastRunPromptUsage is loop-scoped and can reflect a prior successful attempt, while the PR uses it to decide whether the current overflow should skip truncation.

## Solution Assessment

No. The direction is narrow, but the implementation should use current-attempt prompt pressure or preserve the existing truncation path when the prompt-usage snapshot is not proven current.
