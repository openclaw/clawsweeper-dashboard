## Summary

Keep open: current main still lacks structured inbound mention metadata, and this PR's implementation would drop valid non-`open_id` Feishu mentions while overlapping the broader open arbitration work in #38939.

## What This Changes

The PR adds a Feishu `Mentions` template-context field, populates it from inbound message mention metadata excluding the receiving bot, and serializes it as `mentions` in inbound conversation info.

## Best Possible Solution

Revise or fold the work into #38939 so Feishu has one tested mention-extraction contract that excludes the receiving bot, preserves supported ID shapes, emits bounded untrusted metadata if still needed, and includes the required changelog entry.

## Reproduction Assessment

Yes. A Feishu group with multiple bot accounts and `requireMention=false` can deliver a rich-text mention of another bot; current main passes only `WasMentioned` and gives the agent no structured non-self mention list to inspect.

## Solution Assessment

No, not as submitted. The metadata direction is reasonable, but the implementation should use a shared extractor with `open_id`/`user_id`/`union_id` fallback and coordinate with #38939 rather than relying on an inline `open_id`-only filter.
