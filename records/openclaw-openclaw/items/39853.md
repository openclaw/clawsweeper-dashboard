## Best Possible Solution

Adopt one Feishu multi-bot mention contract: either populate the existing generic mention metadata from Feishu facts or land a maintainer-approved arbitration path, with supported ID forms, broadcast filtering, focused coverage, and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes mention text, and forwards only the current bot mention result into inbound metadata. I did not establish a live Feishu multi-bot tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable path should reuse the existing generic mention metadata contract or an accepted Feishu arbitration policy, while this branch creates a parallel schema and misses ID and broadcast semantics.

  - body: Current main already renders generic mention metadata through `MentionedUserIds`, `MentionedSubteamIds`, `ImplicitMentionKinds`, and `MentionSource`.   - body: Filtering on `m.id.open_id` drops Feishu mention records that only carry `user_id` or `union_id`, which the current event type allows. In those deliveries the metadata stays empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list serializes every non-self `open_id`, but current Feishu handling treats `@all`, `@_all`, and ID `all` as broadcast mentions rather than targeted users. Including broadcasts in `mentions` can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.84
