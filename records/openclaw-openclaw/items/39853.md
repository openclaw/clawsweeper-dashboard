## Summary

Keep open: current main still lacks Feishu non-self mention metadata for agents, but this branch is not merge-ready because it adds a parallel metadata contract, drops valid Feishu mention IDs, serializes broadcast mentions, and lacks real Feishu proof. ## What This Changes

The PR adds Feishu non-self mention data to the auto-reply template context and renders it in inbound conversation metadata.

## Best Possible Solution

Adopt one Feishu multi-bot mention contract: populate the existing generic mention metadata or land a maintainer-approved arbitration path, while preserving supported Feishu ID forms, filtering broadcasts, adding focused coverage, and requiring redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu event.message.mentions, normalizes mention text, and forwards only WasMentioned into inbound metadata. I did not establish a live Feishu multi-bot tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The useful fix should reuse current generic mention metadata or an accepted Feishu arbitration policy; this branch creates a parallel Mentions schema and misses Feishu ID and broadcast semantics.

Full review comments:

- **[P1] Use the existing mention metadata contract:** `src/auto-reply/templating.ts:130-136`
  - body: Current main already exposes provider-native mention metadata through MentionedUserIds and MentionSource, and inbound-meta renders those fields. Adding a separate Feishu-only Mentions array creates a competing prompt contract for the same targeting signal; populate the generic fields from Feishu mentions unless maintainers explicitly choose a new schema.
  - confidence: 0.86
- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on m.id.open_id drops Feishu mention records that only carry user_id or union_id, which the current event type allows. In those deliveries the metadata is empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list serializes every non-self open_id, but current Feishu handling treats @all, @_all, and ID all as broadcast mentions, not targeted users. Including broadcasts in mentions can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.84
