## Summary

Keep open: the reported Feishu multi-bot problem is still visible on current main, but this PR should not merge as-is because it drops valid mention records that lack `open_id` and is missing the required changelog entry; related #38939 covers the same duplicate-reply product problem and needs maintainer coordination.

## What This Changes

The PR adds a Feishu `Mentions` template-context field and serializes non-self mention IDs and names into inbound conversation info.

## Best Possible Solution

Choose one Feishu mention-arbitration contract, ideally with a tested extractor that preserves supported Feishu ID shapes, excludes the receiving bot only when identity is known, emits bounded untrusted metadata if needed, and includes the required changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main has `event.message.mentions`, normalizes mentions into message text, and passes only `WasMentioned` into inbound metadata, so a multi-bot Feishu group lacks a structured non-self mention list for the agent to inspect.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but the implementation drops supported non-`open_id` mention entries and should be reconciled with #38939's broader channel-side arbitration work.
