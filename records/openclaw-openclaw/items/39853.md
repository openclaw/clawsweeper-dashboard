## Summary

Keep open: the Feishu multi-bot problem is still valid, but the submitted branch has correctness gaps in mention extraction and broadcast handling, lacks real Feishu proof, and overlaps unresolved arbitration direction in https://github.com/openclaw/openclaw/pull/38939.

## What This Changes

The branch adds a Feishu-derived `Mentions` template field and serializes it as `mentions` in inbound conversation metadata.

## Best Possible Solution

Choose one Feishu mention contract, then populate generic mention metadata or land a reviewed Feishu arbitration policy with focused tests and redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes mention text, and forwards only the current bot mention signal into inbound metadata. I did not establish a live Feishu multi-bot reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but the patch should align with the existing generic mention metadata or an accepted Feishu arbitration policy and must preserve Feishu ID and broadcast semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on `m.id.open_id` drops Feishu mention records that only carry `user_id` or `union_id`, which the current event type allows. In those events the new metadata is empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list serializes every non-self `open_id`, but current Feishu handling treats `@all`/`@_all` and ID `all` as broadcast mentions, not targeted users. Including broadcasts in `mentions` can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.84
