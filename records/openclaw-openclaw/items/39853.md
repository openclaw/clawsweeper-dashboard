## Summary

Keep open for maintainer handling: current main still does not expose Feishu non-self mentions to inbound metadata, but this patch drops valid mention records, lacks real Feishu proof, and needs direction against the current generic mention-metadata/arbitration contracts.

## What This Changes

The PR adds a Feishu-populated `Mentions` template-context field and serializes it as a `mentions` array in inbound conversation metadata.

## Best Possible Solution

Expose Feishu mentions through the current generic mention metadata contract or land a chosen channel-side arbitration policy, preserving all Feishu ID forms plus broadcast/self filtering with focused tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: current main receives `event.message.mentions`, strips/normalizes mention text, and only forwards `WasMentioned` from Feishu into inbound metadata. I did not establish a live Feishu multi-bot reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but the patch adds a parallel `Mentions` shape instead of the current generic mention metadata and loses valid non-`open_id` mention records.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on `m.id.open_id` makes the new metadata empty for Feishu mention records that only carry `user_id` or `union_id`, which the current event type and debounce path already support. In those deliveries the agent still cannot see that another bot or user was addressed, so derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list treats any non-self `open_id` as an addressed user or bot. Feishu broadcast mentions can arrive as `@all`/`@_all` or ID `all`, and current main already has an `isFeishuBroadcastMention` guard for this; serializing broadcast as a normal mention can make agents suppress replies when no sibling bot was targeted.
  - confidence: 0.84
