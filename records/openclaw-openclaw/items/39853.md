## Summary

Keep open for maintainer review: the problem is real at source level, but this branch is not merge-ready because it introduces an `open_id`-only parallel metadata shape on top of the newer generic mention contract and still lacks contributor-supplied live Feishu proof.

## What This Changes

The PR adds a Feishu-populated `Mentions` template-context field and serializes it as a `mentions` array in inbound conversation metadata.

## Best Possible Solution

Implement Feishu mention exposure through the current generic mention metadata or chosen arbitration contract, preserving `open_id`/`user_id`/`union_id`, broadcast/self filtering, focused Feishu tests, and redacted live multi-bot proof.

Yes, at source level: current main receives `event.message.mentions`, normalizes non-self mentions into the body, and only forwards `WasMentioned` into Feishu inbound metadata. I did not establish a live Feishu multi-bot reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but current main now has a generic mention metadata contract and this PR adds a parallel `Mentions` shape while losing `user_id`/`union_id` mention records.
