## Summary

Keep open: the Feishu duplicate-reply problem is source-reproducible and the PR addresses a real gap, but it should not merge as submitted because it drops valid non-`open_id` mentions, predates the current generic mention-metadata contract, lacks real Feishu after-fix proof, and overlaps an open channel-side arbitration PR.

## What This Changes

The branch adds a Feishu-populated `Mentions` template-context field and serializes it as a `mentions` array in inbound conversation metadata.

## Best Possible Solution

Port Feishu to the current generic mention metadata or a channel-side arbitration policy after maintainer direction, preserving `open_id`/`user_id`/`union_id`, filtering broadcast/self mentions correctly, and adding focused Feishu regression plus real multi-bot proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes non-self mentions into message text, and only forwards `WasMentioned` from Feishu into inbound metadata. I did not establish a live Feishu multi-bot repro in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but current main now has a generic mention metadata contract and this PR adds a parallel `Mentions` shape while losing `user_id`/`union_id` mention records.
