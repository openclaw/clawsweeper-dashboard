## Summary

Keep this PR open: the Feishu duplicate-reply problem is still real on current main, but the submitted patch needs real Feishu proof and fixes for mention ID and broadcast semantics before it is mergeable.

## What This Changes

The PR adds Feishu non-self mentions to the template context and serializes them into inbound conversation metadata.

## Best Possible Solution

Adopt one Feishu mention contract, then preserve supported Feishu IDs, filter broadcasts, add focused coverage, and require redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes mention text, and forwards only the current bot mention signal into inbound metadata. I did not establish a live Feishu multi-bot tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but the patch should align with the existing generic mention metadata or an accepted Feishu arbitration policy and must preserve Feishu ID and broadcast semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on `m.id.open_id` drops Feishu mention records that only carry `user_id` or `union_id`, which the current event type allows. In those deliveries the new metadata is empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list serializes every non-self `open_id`, but current Feishu handling treats `@all`/`@_all` and ID `all` as broadcast mentions, not targeted users. Including broadcasts in `mentions` can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.84
