Keep open: current main still lacks structured non-self Feishu mention metadata, but this PR should not merge as submitted because the extractor is open_id-only, after-fix real behavior proof is missing, and the same multi-bot duplicate-reply problem overlaps with open channel-side arbitration work.

## What This Changes

The PR adds a Feishu `Mentions` template-context field populated from non-self `event.message.mentions` and serializes it into inbound conversation metadata.

## Best Possible Solution

Adopt one Feishu-owned mention contract that preserves `open_id`, `user_id`, and `union_id`, excludes the receiving bot only when identity is comparable, and either emits bounded untrusted metadata or performs dispatch arbitration with focused tests and live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes mentions into body text, and only forwards `WasMentioned` into inbound metadata. I did not establish a live multi-bot Feishu reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction is plausible, but the extractor is `open_id`-only and the final contract should be reconciled with the open channel-side arbitration PR before merge.
