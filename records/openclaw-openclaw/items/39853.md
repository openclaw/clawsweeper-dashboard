## Summary

Keep open: the reported Feishu multi-bot mention gap is plausible and not solved on current main, but this PR is not merge-ready because it adds a parallel mention metadata contract, misses supported Feishu ID and broadcast semantics, lacks real Feishu proof, and currently conflicts with main.

## What This Changes

The PR adds a Feishu-sourced `Mentions` array to the auto-reply template context and renders it as `mentions` in inbound conversation metadata.

## Best Possible Solution

Use one Feishu multi-bot mention contract: populate the existing generic mention metadata or land maintainer-approved arbitration, while preserving supported Feishu ID forms, filtering broadcasts, adding focused coverage, and requiring redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes mention text, and forwards only `WasMentioned` into inbound metadata. I did not establish a live Feishu multi-bot tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The narrow maintainable fix should reuse current generic mention metadata or an accepted Feishu arbitration policy; this branch creates a parallel schema and misses Feishu ID and broadcast semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the existing mention metadata contract:** `src/auto-reply/templating.ts:130-136`
  - body: Current main already exposes provider-native mention metadata through `MentionedUserIds` and `MentionSource`, and inbound-meta renders those fields. Adding a separate `Mentions` array creates a competing prompt contract for the same targeting signal; populate the generic fields from Feishu mentions unless maintainers explicitly choose a new schema.
  - confidence: 0.86
- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on `m.id.open_id` drops Feishu mention records that only carry `user_id` or `union_id`, which the current event type allows. In those deliveries the metadata is empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list serializes every non-self `open_id`, but current Feishu handling treats `@all`, `@_all`, and ID `all` as broadcast mentions, not targeted users. Including broadcasts in `mentions` can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.84
