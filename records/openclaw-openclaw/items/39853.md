## Summary

Current main still lacks Feishu-populated structured mention metadata, so the contribution is not obsolete, but the submitted patch should not merge as-is because it drops valid Feishu mention ID forms, does not filter broadcast mentions, lacks real Feishu proof, and overlaps an unresolved arbitration approach.

## What This Changes

The PR adds a Feishu-populated `Mentions` template-context field and serializes it as `mentions` in inbound conversation metadata.

## Best Possible Solution

Choose one Feishu multi-bot mention strategy, then implement it on current main using the existing generic mention metadata contract or a reviewed arbitration policy, preserving all Feishu ID forms and broadcast/self filtering with focused tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: current main receives Feishu `event.message.mentions`, normalizes mention text, and only forwards `WasMentioned` into inbound metadata. I did not run a live Feishu multi-bot tenant in this read-only review.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but the patch adds a parallel `Mentions` shape instead of using the current generic mention metadata and drops valid non-`open_id` or broadcast cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on `m.id.open_id` makes the new metadata empty for Feishu mention records that only carry `user_id` or `union_id`, which the current event type and debounce path already support. In those deliveries the agent still cannot see that another bot or user was addressed, so derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list treats any non-self `open_id` as an addressed user or bot. Feishu broadcast mentions can arrive as `@all`/`@_all` or ID `all`, and current main already has an `isFeishuBroadcastMention` guard for this; serializing broadcast as a normal mention can make agents suppress replies when no sibling bot was targeted.
  - confidence: 0.84
