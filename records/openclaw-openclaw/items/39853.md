## Summary

Keep open for human review: the Feishu multi-bot problem is source-reproducible and the PR is focused, but the submitted branch lacks real Feishu proof, conflicts with current main, and introduces a parallel mention metadata schema with extraction gaps that can affect reply targeting.

## What This Changes

The PR adds a Feishu-derived `Mentions` template field and renders non-self `{ id, name }` mention entries into inbound conversation metadata.

## Best Possible Solution

Adopt one Feishu mention-handling contract: populate the existing generic mention metadata or land an approved arbitration policy, with ID fallback, broadcast filtering, focused tests, and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current Feishu dispatch passes `WasMentioned` but does not pass non-self mention IDs into inbound context, and current `mentionTargets` only covers bot-plus-other mention-forward requests. I did not establish a live Feishu multi-bot tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable path should reuse the existing generic mention metadata contract or an accepted Feishu arbitration policy, while this branch creates a parallel schema and misses ID and broadcast semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the existing mention metadata contract:** `src/auto-reply/templating.ts:130-136`
  - body: Current main already renders generic mention metadata through `MentionedUserIds`, `MentionedSubteamIds`, `ImplicitMentionKinds`, and `MentionSource`. Adding a separate Feishu-only `Mentions` array creates a competing prompt contract for the same targeting signal; populate the generic fields from Feishu mention facts unless maintainers explicitly approve a new schema.
  - confidence: 0.87
- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1335`
  - body: Filtering on `m.id.open_id` drops Feishu mention records that only carry `user_id` or `union_id`, which the current event type allows. In those deliveries the metadata stays empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1335-1336`
  - body: The new list serializes every non-self `open_id`, but current Feishu handling treats `@all`, `@_all`, and ID `all` as broadcast mentions rather than targeted users. Including broadcasts in `mentions` can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.84
