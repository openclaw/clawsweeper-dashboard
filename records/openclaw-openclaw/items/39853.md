## Summary

Keep open, but this PR is not merge-ready: the Feishu multi-bot duplicate-reply problem is plausible from source, but the patch adds a parallel mention metadata schema, misses Feishu ID and broadcast semantics, and lacks real after-fix Feishu proof.

## What This Changes

The PR adds a Feishu-derived `Mentions` template context field and renders it as a `mentions` array in inbound conversation metadata.

## Best Possible Solution

Adopt one Feishu multi-bot mention contract: populate the existing generic mention metadata from Feishu facts or land an approved dispatch-arbitration path, with ID fallbacks, broadcast filtering, focused tests, and redacted live Feishu proof.

## Reproduction Assessment

Yes at source level: current main receives `event.message.mentions`, normalizes/strips mention text, and forwards `WasMentioned`, but it does not expose structured non-self Feishu mentions to the generic metadata contract. I did not run a live Feishu multi-bot tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted. The maintainable path should reuse the existing generic mention metadata contract or an accepted Feishu arbitration policy instead of adding a Feishu-only `Mentions` schema with incomplete ID and broadcast handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the existing mention metadata contract:** `src/auto-reply/templating.ts:130-136`
  - body: Current main already renders generic mention metadata through `MentionedUserIds`, `MentionedSubteamIds`, `ImplicitMentionKinds`, and `MentionSource`. Adding a separate Feishu-only `Mentions` array creates a competing prompt contract for the same targeting signal; populate the generic fields from Feishu mention facts unless maintainers explicitly choose a new schema.
  - confidence: 0.88
- **[P2] Preserve mentions without open_id:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: Filtering on `m.id.open_id` drops Feishu mention records that only carry `user_id` or `union_id`, which the current event type allows. In those deliveries the metadata stays empty, so the receiving agent still cannot tell that another bot or user was addressed; derive a stable ID from all supported fields before mapping.
  - confidence: 0.91
- **[P2] Filter broadcast mentions before serializing:** `extensions/feishu/src/bot.ts:1332-1335`
  - body: The new list serializes every non-self `open_id`, but current Feishu handling treats `@all`, `@_all`, and ID `all` as broadcast mentions rather than targeted users. Including broadcasts in `mentions` can make agents suppress replies even when no sibling bot was addressed; reuse the existing broadcast guard before mapping.
  - confidence: 0.85
