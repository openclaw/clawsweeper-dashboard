## Summary

Keep open: the PR targets a real Feishu multi-bot duplicate-reply gap, but the submitted patch is not merge-ready because it drops valid mention records that lack `open_id` and it is missing the required user-facing changelog entry. It also overlaps #38939's broader Feishu mention-arbitration work, so the final direction needs maintainer coordination rather than cleanup closure.

## What This Changes

The PR adds a `Mentions` template-context field, populates it from Feishu inbound mentions excluding the receiving bot, and serializes it as `mentions` in inbound conversation info.

## Best Possible Solution

Land one coordinated Feishu mention contract that preserves supported ID shapes, excludes self and broadcast mentions consistently, adds focused regression tests plus a changelog entry, and reconciles this metadata surface with #38939's arbitration policy.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main passes only `WasMentioned` into inbound context while Feishu mention records are only normalized into message text or stripped. A multi-bot Feishu group with `requireMention=false` therefore has no structured non-self mention list for the receiving agent to inspect.

## Solution Assessment

No, not as submitted. The metadata direction may be useful, but the extractor should share a tested Feishu mention helper with `open_id`/`user_id`/`union_id` fallback and be coordinated with #38939 rather than adding an inline `open_id`-only path.
