## Summary

Keep open: current main and v2026.6.6 still ship the bundled xurl skill with write-capable examples but without explicit confirm-before-write or audit-log instructions. Because this is a public-posting skill and the requested audit/enforcement semantics are security-sensitive, maintainer/security review should choose the exact policy shape before implementation.

## What This Changes

The issue asks OpenClaw to harden the bundled xurl skill with explicit user confirmation for public/write actions, redacted audit logging guidance, and rate-limit/backoff instructions.

## Best Possible Solution

Add a maintainer-approved safety section to the bundled xurl skill that classifies read versus write commands, requires exact user confirmation for writes, defines redacted audit expectations, and documents rate-limit/backoff behavior, or explicitly split any runtime-enforcement requirement into a separate security design item.
