## Summary

Current main still has the reported xurl skill safety gap. The bundled skill documents public/account-mutating X commands and raw POST usage, while the safety text is limited to credential handling and a one-line 429 backoff note, so this remains a security-policy item for maintainer review rather than a cleanup close.

## What This Changes

The issue asks for the bundled xurl skill to classify read/write actions, require user confirmation before write actions, add redacted write-audit guidance, and document safer rate-limit/backoff behavior.

## Best Possible Solution

Update `skills/xurl/SKILL.md` with explicit read-only/write classifications, confirmation wording that shows account/action/target/content, bounded redacted write-audit guidance, and conservative Retry-After/backoff rules; track hard runtime enforcement separately if maintainers require it.

## Reproduction Assessment

Yes for a source-level gap: current main and v2026.5.20 show the xurl skill still has write-capable examples without xurl-specific confirmation, audit, or detailed backoff instructions. This is not a runtime bug reproduction.

## Solution Assessment

Partly yes: SKILL.md hardening is the narrowest first fix for agent-followed guidance. If maintainers want non-bypassable enforcement, that should be a separate runtime or plugin-hook design rather than implied by skill text.
