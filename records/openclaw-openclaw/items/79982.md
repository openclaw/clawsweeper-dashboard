## Summary

Keep open: the proposed `group:core` shorthand is useful and not present on current `main`, but the current PR still has a blocking plugin-policy/security bug because it expands core groups inside plugin tool resolution.

## What This Changes

The PR adds a documented `group:core` tool-policy shorthand, updates tool-group docs/tests, and expands plugin allowlist normalization through shared group expansion.

## Best Possible Solution

Keep the additive `group:core` API if maintainers accept it, but implement it only for core/global tool policy and keep plugin resolver matching limited to plugin groups, plugin IDs, `*`, and explicit plugin tool names.

## Reproduction Assessment

Yes. Source inspection shows current `main` lacks `group:core` and documents `group:openclaw` too broadly; the PR regression is also source-reproducible through plugin allowlist expansion plus core-name plugin collisions.

## Solution Assessment

No, not as-is. Deriving `group:core` from the core catalog is a narrow shape, but expanding core groups inside plugin allowlist normalization breaks the documented plugin exclusion boundary.
