## Summary

Keep open: the `group:core` idea is useful, but the latest PR still has a P1 plugin-boundary bug because core groups are expanded inside plugin allowlist normalization, which can expose plugin-owned tools despite the new docs saying core excludes plugins.

## What This Changes

This PR adds a documented `group:core` tool-policy shorthand, revises `group:openclaw` docs, and expands tool group handling in plugin allowlist normalization with focused tests.

## Best Possible Solution

Keep the additive `group:core` core-policy/docs/test work, but make plugin resolution ignore or preserve core groups as opaque entries so only `group:plugins`, plugin ids, explicit plugin tool names, wildcard/default behavior, and existing deny rules affect plugin tools.

## Reproduction Assessment

Yes. Current main lacks `group:core`, and the PR-introduced plugin leak is source-reproducible because `expandToolGroups` runs inside plugin allowlist normalization while the standalone plugin-tools MCP path supplies no core `existingToolNames` set.

## Solution Assessment

No, not as-is. Adding `group:core` to the core catalog/docs is the right narrow shape, but plugin resolution should keep core groups out of plugin allowlists instead of expanding them into plugin tool ids.
