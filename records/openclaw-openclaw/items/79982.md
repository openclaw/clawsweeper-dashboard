## Summary

Keep this PR open, but it still needs changes before merge: the `group:core` idea addresses a real tool-policy gap, while the current plugin resolver change can let core-group entries select plugin-owned tools in plugin-only paths.

## What This Changes

The PR adds a `group:core` tool-policy shorthand, updates tool-policy docs/tests, and expands group entries in plugin tool allowlist normalization.

## Best Possible Solution

Keep the additive `group:core` concept if maintainers accept the API, but make plugin tool resolution treat core groups as non-plugin policy and add collision-name regression coverage.

## Reproduction Assessment

Yes. Current main source lacks `group:core` and the docs overstate `group:openclaw`; the PR-introduced boundary bug is source-reproducible because plugin-only callers can match expanded core tool names without an `existingToolNames` core set.

## Solution Assessment

No, not as-is. Deriving `group:core` from the core catalog is a reasonable narrow shape, but expanding it inside plugin allowlist normalization breaks the documented plugin exclusion; filter or ignore core groups on the plugin side instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

  - body: Expanding `group:core` here turns it into concrete core tool ids before plugin matching. Keep core groups opaque or filtered in plugin normalization and add a collision-name regression.
