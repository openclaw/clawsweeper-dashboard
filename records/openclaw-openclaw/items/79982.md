## Summary

Keep open: the `group:core` idea and proof are useful, but the current patch expands core tool groups inside plugin tool allowlist normalization and can expose plugin-owned tools that collide with core ids.

## What This Changes

This PR adds a documented `group:core` tool-policy shorthand, updates tool-group docs/tests, and expands group entries in plugin tool allowlist normalization.

## Best Possible Solution

Keep the additive core-policy/docs/test work, but make plugin resolution ignore or preserve core groups as opaque entries so only `group:plugins`, plugin ids, explicit plugin tool names, wildcard/default behavior, and deny rules affect plugin tools.

## Reproduction Assessment

Yes. Current main lacks `group:core`, and the PR-introduced plugin leak is source-reproducible because `expandToolGroups` runs inside plugin allowlist normalization while the standalone plugin-tools MCP path supplies no core `existingToolNames` set.

## Solution Assessment

No, not as-is. Adding `group:core` to the core catalog/docs is a plausible narrow shape, but plugin resolution should keep core groups out of plugin allowlists instead of expanding them into plugin tool ids.
