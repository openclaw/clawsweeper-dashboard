## Summary

This PR should stay open: it addresses a real tool-policy/docs gap, but the current diff still expands core groups inside plugin allowlist normalization, which can expose plugin-owned tools through a policy documented as core-only.

## What This Changes

The PR adds a `group:core` tool-policy group, updates tool-policy docs/tests, and expands tool groups during plugin allowlist normalization.

## Best Possible Solution

Keep the additive `group:core` concept in core policy/docs, but make plugin resolution ignore or filter core groups and add regression coverage for plugin tools whose names collide with core tools.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `group:core` and documents `group:openclaw` too broadly; the PR regression is also source-reproducible through plugin-only `resolvePluginTools` callers that receive expanded core tool names without a core-name conflict set.

## Solution Assessment

No, not as-is. Adding a core catalog group is the narrow shape, but expanding `group:core` inside plugin allowlist normalization violates the documented plugin exclusion; plugin policy should keep core groups opaque or filter them out.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep core groups out of plugin allowlists:** `src/plugins/tools.ts:88-89`
  - body: Expanding tool groups in `normalizeAllowlist` turns `group:core` into names like `browser`, `canvas`, and `memory_search` before plugin matching. Plugin-only callers such as `src/mcp/plugin-tools-serve.ts` pass no `existingToolNames`, so enabled plugins whose manifests declare core-named tools can be selected by `tools.allow: ["group:core"]` even though the new docs say core excludes plugins. Keep core groups opaque or filtered for plugin resolution and add a colliding-name regression.
  - confidence: 0.88
