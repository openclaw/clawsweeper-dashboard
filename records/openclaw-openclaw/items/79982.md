## Summary

Keep this PR open: the core-group direction is plausible and has real after-fix proof, but the current patch introduces a tool-policy boundary bug by expanding core group names inside plugin tool allowlist matching.

## What This Changes

The PR adds a documented `group:core` shorthand for all built-in core tools, updates tool-policy docs/tests, and expands `group:*` entries in plugin tool allowlist normalization.

## Best Possible Solution

Keep the additive `group:core` concept, but make plugin tool resolution treat core groups as non-plugin policy and add regression coverage for core-named plugin tools before maintainer/API approval.

## Reproduction Assessment

Yes. Source inspection shows current main lacks `group:core` and documents `group:openclaw` too broadly; the blocking PR edge is also source-reproducible because plugin-only callers can match expanded core tool names without an `existingToolNames` core set.

## Solution Assessment

No, not as-is. Deriving `group:core` from the core catalog is the right narrow shape, but expanding it inside plugin allowlist normalization breaks the documented plugin exclusion; filter or ignore core groups on the plugin-policy side instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

  - body: Expanding `group:core` here turns it into concrete core tool ids before plugin matching. Keep core groups opaque or filtered in plugin normalization and add a collision-name regression.
