## Summary

Keep open: the `group:core` shorthand is useful and the supplied real-environment proof is sufficient, but the current diff introduces a P1 tool-boundary bug by expanding core groups inside plugin allowlist normalization.

## What This Changes

This PR adds a documented `group:core` tool-policy shorthand, updates `group:openclaw` docs/tests, and changes plugin tool allowlist normalization to expand tool groups.

## Best Possible Solution

Keep the additive `group:core` catalog/docs/tests, but make plugin resolution ignore or preserve core groups as opaque entries so only `group:plugins`, plugin ids, explicit plugin tool names, wildcard/default behavior, and deny rules affect plugin tools.

## Reproduction Assessment

Yes. Current main lacks `group:core`, and the PR-introduced plugin leak is source-reproducible because `expandToolGroups` would run inside plugin allowlist normalization while `plugin-tools-serve` supplies no core `existingToolNames` set.

## Solution Assessment

No, not as-is. Adding `group:core` to the core catalog/docs is the right narrow shape, but plugin resolution should keep core groups out of plugin allowlists instead of expanding them into plugin tool ids.
