## Summary

Keep open: the idea is useful and the contributor supplied real setup proof, but the current patch introduces a concrete plugin-policy boundary regression by expanding core groups inside plugin tool resolution.

## What This Changes

The PR adds a `group:core` tool-policy shorthand, updates tool-policy docs/tests, and expands `group:*` entries during plugin tool allowlist normalization.

## Best Possible Solution

Keep the additive `group:core` concept, but expand it only for core tool-policy paths and make plugin resolution ignore or filter core groups so plugin tools still require `group:plugins`, a plugin id, or a plugin tool name.

## Reproduction Assessment

Yes. Source inspection shows the PR would expand `group:core` to concrete core names before plugin matching, and the plugin-only MCP caller passes no core `existingToolNames` set while bundled plugins declare colliding tool names.

## Solution Assessment

No, not as-is. Adding the core group in the catalog is the right narrow shape, but expanding it in plugin allowlist normalization breaks the documented core-vs-plugin boundary; filter core groups out of plugin resolution instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep core groups out of plugin allowlists:** `src/plugins/tools.ts:87-91`
  - body: Expanding `group:core` here turns it into concrete core tool ids before plugin matching. Plugin-only callers such as `src/mcp/plugin-tools-serve.ts` pass no `existingToolNames`, and bundled plugins already declare core-named tools like `browser`, `canvas`, `memory_get`, `memory_search`, `code_execution`, and `x_search`, so `tools.allow: ["group:core"]` can expose plugin tools despite the new docs saying core excludes plugins. Keep core groups opaque or filtered in plugin normalization and add a collision-name regression.
