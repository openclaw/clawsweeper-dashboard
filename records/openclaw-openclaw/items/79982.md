## Summary

Keep open. The proposed `group:core` direction addresses a real policy/docs gap, but the current patch still has a blocking plugin-boundary bug already flagged in the prior ClawSweeper review: expanding core groups inside plugin allowlist normalization can allow plugin-owned tools despite the new docs saying core excludes plugins.

## What This Changes

The PR adds `group:core` to core tool groups, docs, and tests, and changes plugin tool allowlist normalization to expand `group:*` entries through the shared core tool group expander.

## Best Possible Solution

Keep the additive core-group concept in the core catalog and docs, but keep plugin resolution limited to `group:plugins`, explicit plugin ids, explicit plugin tool names, wildcards, and existing default behavior, with a regression for core-name plugin collisions.

## Reproduction Assessment

Yes. Current main source shows the original missing all-core group, and the PR-introduced plugin leak is source-reproducible through `src/plugins/tools.ts` plus `src/mcp/plugin-tools-serve.ts` without needing to run a writable test suite.

## Solution Assessment

No, not as-is. Adding `group:core` to the core policy surface is the narrow useful direction, but expanding core groups inside plugin allowlist normalization breaks the documented core-only boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep core groups out of plugin allowlists:** `src/plugins/tools.ts:87-92`
  - body: Expanding `group:core` here converts it to concrete core tool names before plugin matching. `src/mcp/plugin-tools-serve.ts` calls `resolvePluginTools` without `existingToolNames`, and bundled plugins publish core-named tools such as `browser`, `canvas`, `memory_get`, `memory_search`, `code_execution`, and `x_search`, so `tools.allow: ["group:core"]` can expose plugin-owned tools despite the docs saying core excludes plugins. Keep core groups opaque or ignored on the plugin-policy side and add a collision-name regression.
  - confidence: 0.88
