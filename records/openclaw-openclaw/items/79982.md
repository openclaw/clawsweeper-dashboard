## Summary

Keep open: this addresses a real tool-policy/docs gap, but the current patch expands `group:core` inside plugin tool policy and can expose plugin-owned tools with core names in plugin-only surfaces.

## What This Changes

The PR adds a `group:core` tool-policy group, updates tool-policy docs/tests, and expands tool groups inside plugin allowlist normalization.

## Best Possible Solution

Keep the additive `group:core` concept, but expand core groups only in core tool policy and keep plugin resolution limited to `group:plugins`, explicit plugin ids, explicit plugin tool names, and existing wildcard/default behavior.

## Reproduction Assessment

Yes. Current main lacks `group:core` and still documents `group:openclaw` too broadly; the PR-introduced plugin leak is source-reproducible through `src/mcp/plugin-tools-serve.ts` calling `resolvePluginTools` without a core `existingToolNames` set.

## Solution Assessment

No, not as-is. Adding `group:core` to the core catalog/docs is the right narrow shape, but the plugin resolver should filter or ignore core groups instead of expanding them into plugin tool ids.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep core groups out of plugin allowlists:** `src/plugins/tools.ts:87-89`
  - body: Expanding `group:core` here converts it to concrete core tool names before plugin matching. `src/mcp/plugin-tools-serve.ts` calls `resolvePluginTools` without `existingToolNames`, and bundled plugins publish tools named `browser`, `canvas`, `memory_get`, `memory_search`, `code_execution`, and `x_search`, so `tools.allow: ["group:core"]` can materialize plugin tools despite the new docs saying core excludes plugins. Keep core groups filtered or opaque on the plugin-policy side and add a collision-name regression.
  - confidence: 0.88
