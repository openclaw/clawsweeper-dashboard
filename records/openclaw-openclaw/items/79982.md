## Label Justifications

- `merge-risk: 🚨 security-boundary`: Merging this patch could weaken the documented boundary between core tool access and plugin-owned tool access.

## Summary

Keep open: the core-group idea is useful and real proof is supplied, but the current patch still expands core groups inside plugin resolution, so `group:core` can select plugin-owned tools with core-colliding names despite the docs saying plugins are excluded.

## What This Changes

The branch adds a `group:core` tool-policy shorthand, updates tool-policy docs/tests, and changes plugin allowlist normalization to expand tool groups.

## Best Possible Solution

Keep the additive `group:core` concept only if maintainers accept the public config surface, and keep plugin resolution limited to `group:plugins`, explicit plugin ids, explicit plugin tool names, wildcard/default behavior, plus collision-name coverage.

## Reproduction Assessment

Yes for the PR-introduced issue by source inspection: `normalizeAllowlist` expands `group:core`, and plugin manifest filtering later matches plugin tools by normalized name. I did not run tests in this read-only review.

## Solution Assessment

No as-is. Adding `group:core` to the core catalog/docs is a maintainable shape, but plugin policy should keep core groups opaque or filtered instead of using them to select plugin-owned tools.
