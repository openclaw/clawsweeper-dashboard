## Summary

Keep open. Current main still lacks the requested public config field and still uses the shared package-level install execution root by default; the related open PR #75511 appears to address only the opt-out knob and explicitly leaves isolated-root defaulting and channel-only scope out of scope.

## What This Changes

Request to expose a public bundled runtime-dependency install opt-out, make isolated execution the default for bundled plugin runtime-dependency repair, and document the resulting operator matrix.

## Best Possible Solution

Land a cohesive config/docs/test change that exposes the opt-out, threads it through loader and gateway prestage, makes package-level runtime-deps repair use an isolated execution root by default, and explicitly documents how plugin and channel staging behave when operators manage dependencies themselves.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: inspect current main and see that the public schema rejects the requested key, gateway prestage has no user opt-out check, and package-level repair omits `installExecutionRoot`. The production npm storm/disk-bloat behavior is supported by the issue comments, but this read-only review did not run live staging.

## Solution Assessment

Yes for the overall direction, with one maintainer scope decision. Exposing the existing loader seam plus changing the package-level execution root is the narrow maintainable fix; the only unclear part is whether bundled channel staging should share the same `plugins` flag or get its own config surface.
