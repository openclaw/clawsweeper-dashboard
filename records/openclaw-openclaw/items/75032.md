## Summary

Keep open. Current `main` still has only an internal loader opt-out, no public `plugins.installBundledRuntimeDeps` config surface, no docs/schema/help coverage for the requested matrix, and the package-level runtime-deps repair path still defaults installs to the shared root unless a caller passes an isolated execution root.

## What This Changes

Requests a public config opt-out for bundled runtime-dependency installs and a default isolated execution root for bundled plugin runtime-dependency staging.

## Best Possible Solution

Add a public boolean config field, thread it through gateway prestage and plugin loading so `false` skips bundled runtime-dependency installation and reports missing dependencies clearly, make package-level runtime-deps repair use an isolated execution root with atomic replacement by default, and update docs, generated config docs, tests, and changelog for the supported matrix.

## Reproduction Assessment

Yes for the code-level gap: current `main` has a strict public config schema without the requested key, an internal loader option only, gateway prestage that ignores any user opt-out, and package-level repair that defaults execution to the shared install root. The production field reports are credible, but this review did not run the live npm-staging workload.

## Solution Assessment

Yes, with one scope decision. Exposing the existing loader seam and defaulting runtime-deps repair to isolated execution is the narrowest maintainable direction; maintainers should explicitly choose how the flag applies to bundled channel startup staging.
