## Summary

Keep open. Current main still lacks the requested persisted config field, gateway startup still runs bundled runtime-dependency prestage without a user opt-out, and package-level repair still omits an isolated execution root by default; the open related PR appears to cover only part of the requested surface.

## What This Changes

The issue requests a public `plugins.installBundledRuntimeDeps: false` opt-out, isolated runtime-dependency staging by default, and docs for the resulting operator-managed dependency matrix.

## Best Possible Solution

Land a cohesive config, loader, gateway, docs, and test change that exposes the opt-out, keeps missing-dependency failures explicit when operators manage deps themselves, defaults package-level repair to isolated staging, and documents how plugin and channel dependency staging are selected.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: inspect current main and see the config schema rejects the requested key, gateway startup always calls package-level repair for startup plugins, and package-level repair does not pass an isolated execution root.

## Solution Assessment

Unclear until maintainer scope is settled. Exposing the existing loader seam and isolating package-level repair are narrow, maintainable changes, but the channel-scope behavior raised in the discussion needs explicit product/API direction before a second fix branch should be queued.
