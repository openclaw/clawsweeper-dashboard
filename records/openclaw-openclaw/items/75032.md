## Summary

Keep open. Current main still does not expose the requested persisted config key and still sends gateway startup runtime-deps repair through package-level staging without a user opt-out; the related open PR #75511 covers the opt-out seam only and explicitly leaves the isolated-root default and channel scope out of scope.

## What This Changes

The issue requests exposing `plugins.installBundledRuntimeDeps: false`, making isolated execution roots the default for bundled runtime-dependency installs, and documenting the resulting operator-managed versus auto-managed staging matrix.

## Best Possible Solution

Land or revise a cohesive config/docs/test change that exposes the public opt-out, threads it through gateway prestage and loader calls, makes package-level bundled runtime-deps repair use an isolated execution root by default or explicitly preserves a legacy mode, and documents plugin plus channel behavior for operator-managed dependencies.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: inspect current main and the strict public schema lacks the requested key, gateway startup still performs package-plan repair without a config opt-out, and package-level repair defaults the execution root to the install root. The live disk-bloat/stall reproduction is supported by reporter comments but was not rerun in this read-only review.

## Solution Assessment

Yes for the general direction, with a maintainer scope decision still needed. Exposing the existing loader seam and isolating package-manager execution are narrow maintainable fixes, but channel behavior should be decided before considering the issue complete.
