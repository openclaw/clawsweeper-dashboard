## Summary

Keep open. Current main still has the internal runtime-deps opt-out seam but not the requested public config field, and package-level startup repair still defaults to installing directly in the shared runtime-deps root rather than an isolated execution root. The production reports and related comments make this actionable rather than stale cleanup.

## What This Changes

Request to expose a public `plugins.installBundledRuntimeDeps: false` operator escape hatch, make isolated bundled runtime-deps execution the default, and document the resulting behavior matrix.

## Best Possible Solution

Add the public boolean config, thread it through gateway prestage, setup-channel loading, and normal plugin loading, make package-level repair use an owned isolated execution root by default, and update tests, generated config docs, user docs, and the changelog for the supported operator-managed dependency mode.

## Reproduction Assessment

Yes for the code-level gap: current main has a strict public schema without the requested key, an internal-only loader option, and package-level repair that still executes in the shared install root. I did not run a live npm staging reproduction because this review is read-only.

## Solution Assessment

Yes, with one scoped API decision. Exposing the existing loader seam and changing the runtime-deps install owner path is the narrowest maintainable fix; the safer alternative is not to add ad hoc environment-only workarounds beyond the existing `OPENCLAW_PLUGIN_STAGE_DIR`.

## Repair Work Prompt

Implement the runtime-deps control requested by openclaw/openclaw#75032. Add a public `plugins.installBundledRuntimeDeps?: boolean` config field with default `true` in config types, zod schema, labels/help, generated config docs, and tests. When false, skip bundled runtime-deps installation in normal bundled plugin loading, setup-channel plugin loading, and Gateway startup pre-staging, then surface missing dependency failures clearly instead of silently reinstalling. Make package-level bundled runtime-deps repair/install use an owned isolated execution root by default with atomic `node_modules` replacement instead of executing package-manager installs directly in the shared install root; preserve explicit legacy/shared-root behavior only where an existing internal API requires it. Preserve script-disabled install defaults, package-manager safety env, materialization checks, locking, and low-disk warnings. Update docs for `/tools/plugin`, `/cli/plugins`, and `/install/updating` with the supported matrix, and add a single-line `CHANGELOG.md` entry crediting the reporter (`Thanks @Charpup`) plus the Windows channel data point (`Thanks @jgriff0723`) if the channel path is covered. Do not change dependency versions, package-manager overrides, or unrelated plugin activation policy.
