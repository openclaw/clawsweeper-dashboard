## Summary

Keep this open. Current main still routes OpenRouter replay policy through the passthrough-Gemini hook, which preserves the Gemini cleanup behavior but does not select strict9 tool-call-id sanitization for Mistral-family model IDs; the older provider-capabilities fixes are obsolete after the plugin-owned replay refactor.

## What This Changes

The issue requests restoring strict9 tool-call-id sanitization for Mistral-family models when they are routed through proxy providers such as OpenRouter.

## Best Possible Solution

Add a provider-owned replay policy path, preferably via a shared helper composed by OpenRouter and any clearly affected bundled proxy provider, that recognizes wrapped Mistral-family model IDs and selects strict9 while preserving existing Gemini thought-signature cleanup and OpenRouter turn-validation behavior.

## Reproduction Assessment

Yes. A focused code-level reproduction is `resolveTranscriptPolicy({ provider: "openrouter", modelApi: "openai-completions", modelId: "mistralai/devstral-2512:free" })`: current main follows OpenRouter's passthrough-Gemini hook and never sets `toolCallIdMode: "strict9"`.

## Solution Assessment

Yes, the best direction is a narrow plugin-owned/shared replay helper in the new provider replay surface. The older provider-capabilities fallback is no longer the right fix because that module has been removed and provider hooks now intentionally take precedence.
