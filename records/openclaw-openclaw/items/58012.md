## Summary

Keep open. Current main still gives OpenRouter the passthrough-Gemini replay hook, and that hook does not select strict9 tool-call-id sanitization for wrapped Mistral model IDs; direct Mistral already has strict9 support, so this is a remaining proxy-provider policy gap rather than missing sanitizer infrastructure.

## What This Changes

The issue requests restoring strict9 tool-call-id sanitization for Mistral-family models routed through proxy providers such as OpenRouter after provider replay/capability refactors.

## Best Possible Solution

Add provider-owned replay policy composition for OpenRouter and similar proxy transports so Mistral-family wrapped model IDs select strict9 tool-call-id sanitization while preserving existing passthrough-Gemini thought-signature cleanup and OpenRouter turn-validation behavior.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve transcript policy for provider `openrouter`, model ID `mistralai/devstral-2512:free` or `mistral-large-latest`, and OpenAI-compatible API; the current OpenRouter hook returns no `sanitizeToolCallIds` or `toolCallIdMode`, while direct Mistral returns `strict9`.

## Solution Assessment

Yes, the best fix direction is a narrow provider/plugin replay helper or OpenRouter policy composition update. Restoring the old provider-capabilities fallback would work against the current plugin-owned replay architecture and is not the maintainable path.
