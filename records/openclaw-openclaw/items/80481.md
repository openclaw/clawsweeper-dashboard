## Summary

Keep open: the PR has a real Kimi thinking-stream bug and good live after-fix proof, but the patch introduces a payload invariant bug when Kimi thinking budgets combine with a lower configured maxTokens.

## What This Changes

The PR updates the Kimi plugin, Anthropic transport fetch path, and incomplete-turn retry guard to support Kimi Anthropic thinking streams.

## Best Possible Solution

Keep the Kimi-local fix, but make the wrapper preserve a valid Anthropic thinking payload by keeping `budget_tokens < max_tokens` after any configured maxTokens override.

## Reproduction Assessment

Yes. The PR body gives live pre-fix timeout/no-visible-text evidence, and source inspection gives a high-confidence reproduction for the review finding: Kimi `thinkingLevel: "high"` plus configured `maxTokens: 4096` produces `budget_tokens: 8192` with `max_tokens: 4096`.

## Solution Assessment

No. The Kimi transport direction is appropriate, but the budget/max token clamp needs one more guard so user-configured maxTokens cannot make the Anthropic thinking payload invalid.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep Kimi thinking budgets below max_tokens:** `extensions/kimi-coding/stream.ts:66`
  - body: When a user has a lower `maxTokens` extra param, this leaves that smaller payload value in place while still adding the session thinking budget. For example, `thinkingLevel: "high"` plus `maxTokens: 4096` sends `thinking.budget_tokens: 8192` with `max_tokens: 4096`, which violates the Anthropic extended-thinking contract that `budget_tokens` be less than `max_tokens`. Clamp the budget against the final max_tokens or raise max_tokens enough for the reserve before sending the request. ([docs.anthropic.com](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking))
  - confidence: 0.9

## Repair Work Prompt

Fix the Kimi Anthropic thinking payload invariant in PR 80481. Preserve the intended live Kimi behavior, but ensure the wrapper never sends `thinking.budget_tokens >= max_tokens` when session thinking levels or explicit Kimi thinking budgets combine with a lower configured `maxTokens`. Prefer a Kimi-local fix in `extensions/kimi-coding/stream.ts` that either lowers the budget to leave the visible-answer reserve within the final max_tokens or raises max_tokens to the documented budget-plus-reserve policy; add focused tests for low configured maxTokens plus high/session and explicit budgets. Do not broaden core provider policy or change unrelated providers.
