## Summary

Keep open. The hardening is useful, but it only covers direct `/hooks/agent` payload normalization; mapped hook agent dispatches can still render and dispatch oversized metadata, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds max-length checks and unit tests for hook agent payload metadata fields in `normalizeAgentPayload()`.

## Best Possible Solution

Move metadata length validation into a shared hook-agent action validator used by both direct and mapped dispatch paths, with mapped-path regression coverage and `message` still bounded by the HTTP body limit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped agent hook that templates `name`, `sessionKey`, `to`, `model`, or `thinking` from a large payload value, then post to that mapped hook; I did not execute it because this review is read-only.

## Solution Assessment

No. Checking only `normalizeAgentPayload()` is not the best fix because mapped hook actions bypass that function; the maintainable fix is a shared validator before replay-key construction and dispatch for both direct and mapped agent hooks.
