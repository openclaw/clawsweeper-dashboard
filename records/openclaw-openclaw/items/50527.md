## Summary

Keep open: the PR hardens only the direct `/hooks/agent` normalizer, while mapped hook agent actions can still carry the same metadata fields to replay-key construction and dispatch without the proposed limits. It also still lacks real gateway behavior proof from the external contributor.

## What This Changes

The branch adds maximum-length checks for hook agent metadata in `normalizeAgentPayload()` and unit tests for oversized and boundary values.

## Best Possible Solution

Use one shared hook-agent metadata validator before both direct and mapped agent dispatch, while leaving `message` bounded by the existing hook body-size limit.

## Reproduction Assessment

Yes for the review finding. Source inspection of current main shows mapped `/hooks/<name>` actions can render the same metadata fields and reach replay-key construction and `dispatchAgentHook()` without the new direct-path validator.

## Solution Assessment

No. Direct-only validation is not the best fix because mapped hooks are a documented ingress path for the same metadata shape; shared validation is narrower and more maintainable.
