## Label Justifications

- `P2`: This is a focused security hardening PR for gateway hooks with a concrete but limited bypass and no evidence of an emergency production outage.
- `merge-risk: 🚨 security-boundary`: Merging the patch as-is could leave maintainers believing hook metadata is bounded while mapped hook inputs remain unbounded.
- `merge-risk: 🚨 compatibility`: The new direct endpoint validation intentionally converts some currently accepted oversized metadata payloads into 400 responses.

## Summary

Keep open: the hardening is useful, but the current patch only validates direct hook payloads while mapped hook agent actions can still carry oversized request-derived metadata, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds direct `/hooks/agent` metadata length checks for `name`, `to`, `model`, `thinking`, `agentId`, and `sessionKey` in `normalizeAgentPayload()` plus helper tests.

## Best Possible Solution

Move metadata length validation into a shared hook-agent action validator used by both direct and mapped dispatch before replay-key construction, with mapped regression coverage and redacted live request proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped agent hook that templates `name`, `sessionKey`, `to`, `model`, or `thinking` from a large payload value, then post to that mapped hook; I did not execute it because this review is read-only.

## Solution Assessment

No. Checking only `normalizeAgentPayload()` is not the best fix because mapped hook actions bypass that function; the maintainable fix is a shared validator for both direct and mapped agent hooks.
