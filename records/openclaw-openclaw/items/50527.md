## Summary

Keep open; the hardening is useful but not merge-ready because it validates only direct hook payloads while mapped hook agent actions can still carry oversized metadata, and contributor proof is only unit-test/CI text.

## What This Changes

The PR adds per-field metadata length constants and rejection checks inside `normalizeAgentPayload()` for direct hook agent payloads, plus direct normalizer tests for oversized and boundary values.

## Best Possible Solution

Move metadata length validation into a shared hook-agent action validator used by direct and mapped hooks before replay-key construction and dispatch, with mapped regression tests and explicit compatibility proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped agent hook that templates `name`, `sessionKey`, `to`, `model`, or `thinking` from a large payload value, then post to that mapped hook.

## Solution Assessment

No. Checking only `normalizeAgentPayload()` is not the best fix because mapped hook actions bypass that function; the maintainable fix is shared validation for both direct and mapped agent hooks.
