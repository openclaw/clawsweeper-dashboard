## Summary

Keep open: the hardening is useful, but it is incomplete because mapped hook agent actions can still carry the same oversized metadata into replay-key construction and dispatch, and the external PR has no real after-fix runtime proof.

## What This Changes

The PR adds maximum-length validation for direct hook agent payload metadata in `normalizeAgentPayload()` and unit tests for oversized and boundary values.

## Best Possible Solution

Use one shared hook-agent metadata validator before both direct and mapped agent dispatch, while leaving `message` bounded by the existing hook body-size limit.

## Reproduction Assessment

Yes for the review finding. Source inspection shows mapped `/hooks/<name>` actions can render the same metadata fields and reach replay-key construction and dispatch without `normalizeAgentPayload()`; no live request was run because this review is read-only.

## Solution Assessment

No. Direct-only validation is not the best solution because mapped hooks are a documented ingress path for the same metadata shape; shared validation is narrower and more maintainable.
