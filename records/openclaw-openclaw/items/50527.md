## Summary

Keep open: the hardening is useful but incomplete because mapped hooks can still render and dispatch the same metadata fields without the new normalizer, and this external PR still lacks required real gateway proof.

## What This Changes

The PR adds max-length constants and validation in `normalizeAgentPayload()` for hook agent metadata fields, plus unit tests for oversized and boundary values.

## Best Possible Solution

Add one shared hook-agent metadata validator used by both `normalizeAgentPayload()` and mapped agent dispatch, with regression coverage for both ingress paths while leaving `message` bounded by the existing body-size limit.

## Reproduction Assessment

Yes for the review finding. Source inspection shows mapped `/hooks/<name>` agent actions can render the same metadata fields and reach replay-key construction and dispatch without `normalizeAgentPayload()`; I did not run a live gateway request in this read-only review.

## Solution Assessment

No. Direct-only validation is not the best fix because mapped hooks are a supported ingress path for the same metadata shape; shared validation before both dispatch paths is narrower and more maintainable.
