## Summary

Keep this PR open: the defensive validation is useful, but the patch only covers direct `/hooks/agent` normalization while mapped hook agent actions still carry the same metadata fields into replay-key construction and dispatch.

## What This Changes

The PR adds per-field maximum length checks for hook agent payload metadata inside `normalizeAgentPayload()` and adds helper-level tests for oversized and exact-boundary values.

## Best Possible Solution

Add a shared hook-agent metadata validator used by both direct payload normalization and mapped action dispatch, with regression coverage for both ingress paths while leaving `message` bounded by the existing body-size limit.

## Reproduction Assessment

Yes, by source inspection: mapped `/hooks/<name>` agent actions can render or carry the same metadata fields and are dispatched without `normalizeAgentPayload()`. I did not run a live gateway request in this read-only review.

## Solution Assessment

No. Direct-only validation is not the best fix because mapped hooks are a documented ingress path for the same metadata shape; a shared validator before both dispatch paths is narrower and more maintainable.
