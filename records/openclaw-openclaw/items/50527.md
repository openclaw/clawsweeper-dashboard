## Summary

Keep open: the PR is useful security hardening, but the current patch only validates direct `/hooks/agent` payloads and leaves the documented mapped hook path able to dispatch oversized metadata. It also lacks contributor real behavior proof and is currently conflicting against main.

## What This Changes

The branch adds per-field max-length validation for hook agent payload metadata inside `normalizeAgentPayload()` plus unit tests for oversized and boundary values.

## Best Possible Solution

Add one shared hook-agent metadata validator used by both direct payload normalization and mapped agent action dispatch, with regression coverage for both ingress paths while keeping `message` bounded by the HTTP body limit.

## Reproduction Assessment

Yes, by source inspection: direct `/hooks/agent` enters `normalizeAgentPayload()`, while mapped `/hooks/<name>` renders and dispatches the same metadata fields without that helper. I did not run a live gateway request in this read-only review.

## Solution Assessment

No. A direct-only normalizer is bypassable on the documented mapped hook surface; the maintainable fix is a shared validator applied before both direct and mapped agent dispatch.
