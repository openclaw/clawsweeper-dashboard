## Summary

Keep this PR open. The branch is useful defense-in-depth work, but it validates only direct hook payload normalization while the supported mapped hook path can still render and dispatch the same oversized metadata fields.

## What This Changes

The branch adds maximum-length constants and checks for direct hook agent payload metadata in `normalizeAgentPayload()` plus focused helper tests.

## Best Possible Solution

Add a shared hook-agent metadata validator for the common direct and mapped action shape, call it before replay-key construction and dispatch on both paths, and keep `message` bounded by the existing HTTP body limit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped `/hooks/<name>` agent action whose template or transform renders an over-limit `name`, `sessionKey`, `to`, `model`, or `thinking`, and current main passes it from `mapped.action` into replay-key construction and dispatch without the direct normalizer.

## Solution Assessment

No. The PR is directionally correct, but direct-only validation is not the narrowest maintainable fix because mapped hooks carry the same metadata shape through a separate supported path.
