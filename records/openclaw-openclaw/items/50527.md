## Summary

Keep open: the PR is useful security hardening for the direct hook agent path, but it leaves the documented mapped hook path able to dispatch the same oversized metadata fields and the contributor has not provided required real gateway behavior proof.

## What This Changes

The branch adds per-field max-length checks for direct hook agent payload metadata in `normalizeAgentPayload()` and helper tests for oversized and boundary values.

## Best Possible Solution

Add a shared hook-agent metadata validator used before both direct payload dispatch and mapped action dispatch, with regression coverage for both ingress paths while leaving `message` bounded by the HTTP body limit.

## Reproduction Assessment

Yes, by source inspection: direct `/hooks/agent` goes through `normalizeAgentPayload()`, while mapped `/hooks/<name>` renders and dispatches the same metadata fields without that helper. I did not run a live gateway request in this read-only review.

## Solution Assessment

No. A direct-only normalizer is bypassable on the documented mapped hook surface; the narrower maintainable fix is a shared validator for both direct payloads and mapped agent actions.
