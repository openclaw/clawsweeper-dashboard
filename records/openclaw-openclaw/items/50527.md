## Summary

Keep open. Current main still lacks per-field hook agent metadata limits, and the PR is useful, but the patch only validates direct `/hooks/agent` payloads while mapped `/hooks/<name>` agent actions can render and dispatch the same metadata fields without those limits.

## What This Changes

The branch adds per-field maximum length constants and over-limit errors for direct `normalizeAgentPayload()` metadata fields, plus helper tests for oversized and boundary values.

## Best Possible Solution

Use one shared hook-agent metadata validator for both direct payloads and mapped agent actions after template/transform rendering, preserving the existing 256 KB body limit for `message` while rejecting oversized metadata before replay-key construction, logging, storage, or dispatch.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a mapped `/hooks/<name>` agent mapping whose `name`, `sessionKey`, `to`, `model`, or `thinking` template renders an over-limit string; the current mapped route dispatches those fields without the direct normalizer.

## Solution Assessment

No. The PR is directionally right, but the narrowest maintainable fix is to validate the shared agent-action shape used by both direct and mapped hooks, not only `normalizeAgentPayload()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Validate mapped hook metadata before dispatch:** `src/gateway/hooks.ts:374-375`
  - body: The new length checks only run inside `normalizeAgentPayload()`, which is used by direct `/hooks/agent` requests.
