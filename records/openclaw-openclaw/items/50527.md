## Summary

Keep open: the PR is directionally useful, but the hardening is incomplete because it validates only direct hook agent payloads while mapped hook agent actions can still carry the same oversized metadata into replay-key construction and dispatch.

## What This Changes

The branch adds maximum-length constants and checks in `normalizeAgentPayload()` plus helper tests for oversized direct hook agent metadata.

## Best Possible Solution

Add one shared hook-agent metadata validation path that runs before both direct and mapped agent dispatch while leaving `message` governed by the existing HTTP body-size limit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a mapped `/hooks/<name>` agent mapping or transform can render over-limit metadata, and current main passes `mapped.action` into replay-key construction and dispatch without the direct normalizer.

## Solution Assessment

No. The PR is directionally correct, but direct-only validation is not the best fix because mapped hooks carry the same metadata shape through a separate documented path.
