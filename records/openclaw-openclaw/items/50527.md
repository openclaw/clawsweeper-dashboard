## Summary

Keep open: this is useful security hardening, but the patch only covers direct hook agent payloads; mapped hook agent dispatch can still carry the same metadata fields through replay-key construction and dispatch, and the PR has only test/CI proof so far.

## What This Changes

The PR adds per-field maximum length validation for direct hook agent payload metadata in `normalizeAgentPayload()` plus unit tests for oversized and boundary values.

## Best Possible Solution

Add a shared hook-agent metadata validator used by both direct payload normalization and mapped action dispatch, with regression coverage for both ingress paths while keeping `message` bounded by the existing body-size limit.

## Reproduction Assessment

Yes for the review finding by source inspection: mapped `/hooks/<name>` agent actions can render or carry the same metadata fields and are dispatched without `normalizeAgentPayload()`. I did not run a live gateway request in this read-only review.

## Solution Assessment

No. Direct-only validation is not the best fix because mapped hooks are a documented ingress path for the same metadata shape; a shared validator before both dispatch paths is narrower and more maintainable.
