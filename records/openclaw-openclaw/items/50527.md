## Summary

Keep open: the PR is useful security hardening, but it is not merge-ready because mapped hook agent dispatches bypass the new direct-payload validator and the external PR has no after-fix real behavior proof.

## What This Changes

The branch adds per-field length constants and checks for hook agent metadata inside `normalizeAgentPayload()` plus unit tests for oversized direct payload values and exact-length boundaries.

## Best Possible Solution

Use one shared hook-agent metadata validator before both direct and mapped agent dispatch, while leaving `message` bounded by the existing hook body-size limit.

## Reproduction Assessment

Yes for the review finding. Source inspection shows mapped `/hooks/<name>` actions can render the same metadata fields and reach replay-key construction and dispatch without the new normalizer; I did not run a live gateway request because this review is read-only.

## Solution Assessment

No. Direct-only validation is not the best fix because mapped hooks are a documented ingress path for the same metadata shape; shared validation is narrower and more maintainable.
