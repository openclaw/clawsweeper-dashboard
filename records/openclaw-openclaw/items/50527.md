## Summary

Keep open: the hardening is still useful, but the branch is not merge-ready because it only validates direct `/hooks/agent` metadata while mapped hook agent actions can still render and dispatch the same request-controlled fields without shared limits.

## What This Changes

The branch adds max-length checks and unit tests for `name`, `to`, `model`, `thinking`, `agentId`, and `sessionKey` in direct hook agent payload normalization.

## Best Possible Solution

Move metadata length validation into a shared hook-agent action validator used before replay hashing and dispatch for both direct and mapped hooks, with mapped regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes, at source level. Configure a mapped agent hook that templates `name`, `sessionKey`, `to`, `model`, or `thinking` from a large payload value, then post to that mapped hook.

## Solution Assessment

No. Checking only `normalizeAgentPayload()` is not the best fix because mapped hook actions bypass that function; the maintainable fix is shared validation for both direct and mapped agent hooks.
