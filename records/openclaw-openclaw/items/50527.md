## Summary

Keep open. The PR adds useful direct `/hooks/agent` metadata limits, but it leaves mapped `/hooks/<name>` agent metadata unvalidated before replay-key construction and dispatch.

## What This Changes

The branch adds per-field maximum length checks for direct hook agent payload metadata in `normalizeAgentPayload()` and helper tests for oversized and exact-boundary values.

## Best Possible Solution

Use one shared hook-agent metadata validator after direct payload normalization and after mapped action rendering, preserving the HTTP body limit for `message` while rejecting oversized metadata before replay-key construction, logging, storage, or dispatch.

## Reproduction Assessment

Yes. Configure a mapped `/hooks/<name>` agent action whose `name`, `sessionKey`, `to`, `model`, or `thinking` template renders an over-limit string; current main sends those mapped fields into replay-key construction and dispatch without the direct normalizer.

## Solution Assessment

No. The PR is directionally right, but the narrowest maintainable fix is to validate the shared agent-action shape used by both direct and mapped hooks, not only `normalizeAgentPayload()`.
