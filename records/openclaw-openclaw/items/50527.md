## Summary

Keep open: the PR is useful hardening, but it only validates direct hook agent payloads and leaves mapped hook agent metadata on the same webhook ingress path unbounded; the external PR also still lacks real behavior proof.

## What This Changes

The branch adds per-field length limits and rejection errors for direct hook agent payload metadata in `normalizeAgentPayload()`, plus direct normalizer unit tests.

## Best Possible Solution

Move metadata length validation into a shared hook-agent action validator used by both direct and mapped dispatch before replay-key construction, with mapped regression coverage and explicit compatibility proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped agent hook that templates `name`, `sessionKey`, `to`, `model`, or `thinking` from a large payload value, then post to that mapped hook.

## Solution Assessment

No. Checking only `normalizeAgentPayload()` is not the best fix because mapped hook actions bypass that function; the maintainable fix is shared validation for both direct and mapped agent hooks.
