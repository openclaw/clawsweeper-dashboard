## Summary

Keep open: the PR is focused and useful, but it only hardens direct `/hooks/agent` normalization while mapped hook agent actions can still render the same metadata fields from request-controlled data and dispatch them unbounded. It also still lacks contributor-provided real behavior proof.

## What This Changes

The branch adds per-field length checks for direct hook agent payload metadata in `normalizeAgentPayload()` and tests oversized plus exact-limit direct normalizer inputs.

## Best Possible Solution

Move metadata length validation into a shared hook-agent action validator used by both direct and mapped dispatch before replay-key construction, with mapped regression coverage and documented threshold intent.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a mapped agent hook that templates `name`, `sessionKey`, `to`, `model`, or `thinking` from a large payload value, then post to that mapped hook.

## Solution Assessment

No. Checking only `normalizeAgentPayload()` is not the best fix because mapped hook actions bypass that function; the maintainable fix is shared validation for both direct and mapped agent hooks.
