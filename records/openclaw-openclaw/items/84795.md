## Summary

Keep open: current main and the latest v2026.5.22 release still let the optional shell-env fallback reach the POSIX `/bin/sh` probe on native Windows, and two open closing PRs already own the narrow fix path.

## What This Changes

The issue asks OpenClaw to skip the optional login-shell environment fallback on native Windows so CLI/config startup does not spawn `/bin/sh` or warn/fail.

## Best Possible Solution

Land one focused PR that makes `loadShellEnvFallback`/`probeLoginShellEnv` a no-op on win32, preserves POSIX behavior, and adds regression coverage; close the duplicate fix PR after the chosen fix merges.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: enable shell-env import on win32 with a missing expected key and the current fallback reaches the `/bin/sh` probe. I did not run native Windows in this read-only review, but the reporter also confirmed the same failure on OpenClaw 2026.5.22.

## Solution Assessment

Yes, the narrow maintainable solution is to skip the POSIX login-shell probe on win32 before resolving or executing `/bin/sh`, with regression coverage. The best path is to review and land one of the linked focused PRs rather than start a third fix.
