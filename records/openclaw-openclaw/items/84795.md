## Summary

Keep open: current main still routes the optional shell-env fallback through the POSIX `/bin/sh` probe on Windows, the latest shipped v2026.5.22 tag shows the same path, and the two linked closing PRs are still open implementation candidates rather than merged fixes.

## What This Changes

Request that OpenClaw skip the optional POSIX login-shell env fallback on native Windows so startup, status, doctor, and completion commands no longer try to spawn `/bin/sh`.

## Best Possible Solution

Land one narrow fix PR that makes the shell-env fallback/probe a win32 no-op, preserves POSIX behavior, adds regression coverage, and includes a native Windows or Testbox smoke before closing this issue.

## Reproduction Assessment

Yes. Source inspection shows that enabling shell-env import on win32 with a missing expected key reaches the `/bin/sh` exec path, and the reporter supplied current v2026.5.22 Windows confirmation; I did not run native Windows locally.

## Solution Assessment

Yes. A platform-aware skip in the shell-env fallback/probe path is the narrow maintainable fix; adding a PowerShell environment importer would be a separate product decision.
