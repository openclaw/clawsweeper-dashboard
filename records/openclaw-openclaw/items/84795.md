## Summary

Keep open: current main and the latest shipped release still let the enabled shell-env fallback reach the POSIX login-shell probe on native Windows, and the issue is now paired with an open closing PR that should be reviewed or landed before closure.

## What This Changes

The issue requests skipping the POSIX login-shell shell-env fallback on native Windows so startup/status does not warn about missing `/bin/sh`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84838, or an equivalent win32 no-op guard with regression coverage, then close this issue after merge and Windows smoke/source proof.

## Reproduction Assessment

Yes, source-level: enable shell-env fallback on native Windows with a missing expected key, and config loading reaches `loadShellEnvFallback`, which calls `probeLoginShellEnv` and defaults to `/bin/sh` with no win32 guard. I did not run a native Windows host.

## Solution Assessment

Yes, the linked PR's platform guard is the narrow maintainable fix if it lands with regression coverage and native Windows proof. This should not expand into a PowerShell environment importer in this issue.
