## Summary

Keep open: current main and the latest release still route enabled shell-env fallback through the `/bin/sh` probe on Windows, and two open PRs already reference this issue with closing syntax. The issue should close only after one of those fixes lands and is verified.

## What This Changes

The issue requests skipping the optional login-shell env fallback on native Windows so startup/status no longer attempts `/bin/sh` and warns with `spawnSync /bin/sh ENOENT`.

## Best Possible Solution

Review and land one of the linked PRs with focused regression coverage and acceptable Windows/runtime proof, then close this issue as fixed.

## Reproduction Assessment

Yes for source reproduction: enabling `OPENCLAW_LOAD_SHELL_ENV=1` or `env.shellEnv.enabled` on native Windows with a missing expected key reaches `loadShellEnvFallback` and then the unguarded `/bin/sh` probe. I did not run a native Windows host in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the fallback/probe path no-op on win32, mirroring the existing `getShellPathFromLoginShell` guard, with regression coverage. A PowerShell-specific env importer would be a separate feature decision.
