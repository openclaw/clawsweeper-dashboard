## Summary

This is a valid, narrow Windows startup bug on current main: the shell-env fallback still reaches the `/bin/sh` probe path, and only the separate PATH helper has a Windows skip.

## What This Changes

The issue requests skipping the login-shell environment fallback on native Windows so startup does not warn about missing `/bin/sh`.

## Best Possible Solution

Add a testable win32 no-op for shell-env import in the fallback/probe path, preserve existing POSIX behavior, and add focused regression coverage plus a changelog entry if this ships as a user-visible fix.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: enable shell-env import on native Windows with a missing expected key, and current main reaches the `/bin/sh` exec path. I did not run native Windows here, so this is source-reproducible rather than live reproduced.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix if implemented as a platform-aware skip in the fallback/probe path with regression coverage. It should not grow into an implicit PowerShell environment importer without a separate product decision.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84795. On native Windows, enabled `env.shellEnv` or `OPENCLAW_LOAD_SHELL_ENV=1` with missing expected keys currently routes `loadShellEnvFallback` through `probeLoginShellEnv`, which falls back to `/bin/sh` and warns `spawnSync /bin/sh ENOENT`. Add a platform-aware skip for the shell-env fallback/probe path without changing POSIX login-shell import behavior and without adding implicit PowerShell env import. Prefer a small testable seam such as threading an optional platform through `loadShellEnvFallback`/`probeLoginShellEnv`. Add regression coverage in `src/infra/shell-env.test.ts` proving `loadShellEnvFallback` on `win32` returns ok with no applied keys, does not call exec, and does not warn when expected keys are missing. Add a changelog fix entry if this is user-visible for release. Do not change unrelated shell command behavior or broad Windows exec policy.
