## Summary

Keep open: current main and v2026.5.22 still lack a win32 guard in the shell-env fallback probe, while two open closing PRs now own the repair path.

## What This Changes

The issue asks OpenClaw to skip the optional login-shell environment fallback on native Windows so CLI startup and completion do not try to spawn `/bin/sh`.

## Best Possible Solution

Land one focused shell-env fix that makes the fallback/probe path a win32 no-op, keeps POSIX behavior unchanged, adds regression coverage, and includes a native Windows or Testbox smoke before closing.

## Reproduction Assessment

Yes. Source inspection shows `OPENCLAW_LOAD_SHELL_ENV=1` or config-enabled shell env import with a missing expected key reaches `/bin/sh` on win32, and the reporter supplied v2026.5.22 native Windows confirmation; I did not run a fresh native Windows command here.

## Solution Assessment

Yes. A platform-aware no-op in the existing fallback/probe path with focused regression coverage is the narrow maintainable fix, and it should not turn into a PowerShell environment importer without a separate product decision.
