## Summary

Keep open: the current PR head fixes the central docs/gateway timeout mismatch and has sufficient real gateway-backed proof, but it still needs maintainer acceptance because it intentionally changes omitted gateway CLI timeout behavior for existing users.

## What This Changes

The PR updates agent timeout docs and CLI help, changes gateway-backed `openclaw agent` dispatch to omit `params.timeout` unless `--timeout` is explicit, and adds regression tests for omitted and explicit timeout paths.

## Best Possible Solution

Land this shape after maintainers accept the compatibility tradeoff and normal merge gates pass, keeping explicit `--timeout` or server config as the supported shorter-cap path.

## Reproduction Assessment

Yes, at source level. Current main falls back to 600 seconds in `agent-via-gateway` while the shared runtime resolver defaults omitted agent timeouts to 172800 seconds; I did not run tests because this was a read-only review.

## Solution Assessment

Yes. Omitting `params.timeout` when the CLI flag is absent is the right owner boundary because the gateway server owns server config/defaults, while explicit `--timeout` remains a per-run override.
