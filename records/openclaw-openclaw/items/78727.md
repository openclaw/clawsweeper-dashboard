## Summary

Keep open. Current main still has a plausible device-pairing bug path: approval merges the paired device's approved scope baseline, but the operator token minted for a non-empty pending request is replaced with only the pending request scopes, which matches the reported dropped prior scopes and can leave reconnects stuck behind another upgrade prompt.

## What This Changes

The issue reports that `openclaw mcp serve` repeatedly creates scope-upgrade requests and that approving them rotates the device operator token without preserving prior scopes.

## Best Possible Solution

Fix the generic paired-device approval/token issuance path so scope-upgrade approval preserves the effective approved operator scopes needed for reconnects without widening beyond caller-approved scopes, with a focused regression around MCP-style read/write/approvals reconnects.

## Reproduction Assessment

Unclear for a live current-main run: the issue provides detailed commands and logs, and source inspection reproduces the token-scope replacement path, but I did not execute a real Gateway/MCP loop in this read-only review.

## Solution Assessment

No implemented solution or PR is present. The best direction is a generic device-pairing/token-scope fix plus regression coverage, not an MCP-only shared-token bypass.
