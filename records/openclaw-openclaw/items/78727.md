## Summary

Keep open. Current main still has a source-visible token-rotation gap: approval merges `approvedScopes`, but the rotated role token is minted from the pending request scopes first, so an upgrade can drop previously usable scopes and keep the client in another upgrade/request cycle.

## What This Changes

The issue reports that `openclaw mcp serve` scope-upgrade approvals rotate the CLI device operator token with only the latest approved scopes, causing repeated pending approval loops.

## Best Possible Solution

Fix device-pairing approval so rotated role tokens preserve the merged, role-scoped approved baseline while staying bounded by caller authorization, then add regression coverage for the MCP/device-list upgrade loop.

## Reproduction Assessment

Yes, source-reproducible: seed a paired operator device with prior approved/token scopes, approve a pending MCP-style operator scope upgrade, and current `resolveApprovedTokenScopes` will choose pending scopes before the merged baseline. I did not run the live Gateway/MCP process in this read-only environment.

## Solution Assessment

Yes, the narrow fix is in token-scope resolution during `approveDevicePairing`, not a new `mcp serve` master-token bypass. The fix should preserve existing approved role scopes in the newly minted token while keeping the existing caller-scope containment checks.
