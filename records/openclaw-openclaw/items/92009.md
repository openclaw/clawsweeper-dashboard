## Summary

This is a credible 2026.6.5 regression: the released CLI reports the configured Google default as resolved while refusing to inspect or run it, yet current source, tests, docs, onboarding, and retirement metadata all define `google/gemini-3.1-pro-preview` as supported. Current main predates the report and no merged or open canonical fix owns this exact failure, so the issue remains necessary.

## What This Changes

Repair the upgrade path so a retained `google/gemini-3.1-pro-preview` default is cataloged and runnable, or is migrated or diagnosed before agent execution.

## Best Possible Solution

Make the existing Google plugin catalog the authoritative model set consumed consistently by list, inspect, and run, then add an upgrade regression test showing a retained default remains executable; if the model is intentionally retired upstream, update the plugin default/docs and add a plugin-owned doctor migration in the same change.

## Reproduction Assessment

No current-main live reproduction was run, but the released reproduction is concrete and current source establishes a high-confidence contradictory contract: the model is cataloged, documented, and selected by onboarding while v2026.6.5 rejects it.

## Solution Assessment

Yes, the best direction is a narrow catalog-consistency and upgrade repair. A silent downgrade to Gemini 2.5 Pro would conflict with current defaults, docs, normalization, and retirement guidance unless upstream retirement is first proven and all those contracts are migrated together.
