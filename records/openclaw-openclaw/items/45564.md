## Summary

Keep open. Current main only partially addresses the accidental-wipe problem: Control UI `/new` now creates and switches to a fresh dashboard session, but typed `/reset` and shared channel/native `/new` and `/reset` still perform immediate reset behavior without a confirmation step.

## What This Changes

The issue requests a confirmation step, ideally configurable, before `/new` or `/reset` wipes a session.

## Best Possible Solution

Define a shared destructive-session-reset confirmation contract for bare `/new` and `/reset`, apply it consistently to in-place reset paths, and explicitly preserve non-destructive fresh-session creation plus `/reset soft` and reset-tail behavior.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: send typed Control UI `/reset` or a native/channel bare `/new` or `/reset`, and the existing code/tests show immediate reset handling with no confirmation state.

## Solution Assessment

No, the original 2-3 click/configurable shape is not clearly the best default. The issue discussion points to a narrower single repeat-command confirmation, but maintainers should approve the cross-surface contract before implementation.
