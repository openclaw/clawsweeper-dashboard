## Summary

Keep open. Current main only partially reduces accidental wipes: Control UI typed `/new` now creates a fresh dashboard session and toolbar New Session has a confirmation guard, but typed `/reset` and shared native/gateway bare `/new` and `/reset` still reset immediately without the repeat-command confirmation discussed in this issue.

## What This Changes

The issue asks for a confirmation step before destructive `/new` and `/reset` session resets, ideally with configurable confirmation depth.

## Best Possible Solution

Define a shared destructive-session-reset confirmation contract for bare in-place `/reset` and destructive `/new` paths, using a short repeat-command timeout while preserving non-destructive fresh-session creation, reset tails, `/reset soft`, and documented automation behavior.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: typed Control UI `/reset` dispatches without `confirm`, and shared native/gateway bare `/new` and `/reset` paths immediately reset or acknowledge in tests.

## Solution Assessment

Unclear. The original configurable 2-3 step shape is not clearly the best fit; the issue discussion points to a narrower single repeat-command confirmation, but maintainers should approve the cross-surface behavior change first.
