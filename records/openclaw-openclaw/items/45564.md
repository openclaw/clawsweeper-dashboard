## Summary

Keep open. Current main only partially mitigates accidental resets: button-triggered Control UI new-session resets have a confirmation guard, but typed `/new` and `/reset` plus shared channel and Gateway reset paths still proceed immediately, and the remaining cross-surface behavior is a product/config decision rather than a cleanup close.

## What This Changes

Request a confirmation flow, preferably configurable, before `/new` or `/reset` wipes the current session.

## Best Possible Solution

Define one shared destructive reset confirmation contract for bare in-place `/reset` and destructive `/new` paths, preserving non-destructive fresh-session creation, reset tails, `/reset soft`, and documented automation behavior.

## Reproduction Assessment

Yes for the current behavior, though this is a feature request rather than a broken current contract. Static inspection shows typed Control UI `/reset`, shared channel bare `/new` and `/reset`, and Gateway agent bare resets still proceed immediately.

## Solution Assessment

Unclear. The discussed single repeat-command confirmation is narrower than the original 2-3 click request, but maintainers should choose the exact surfaces, timeout, config shape, and automation compatibility before implementation.
