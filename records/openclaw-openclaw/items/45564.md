## Summary

Keep open. Current main and v2026.6.6 fixed the Control UI button-triggered New Chat path, but typed `/reset` and channel `/new`/`/reset` still execute hard session resets immediately; the remaining work is a product/UX policy decision, not a stale or already-implemented item.

## What This Changes

The issue requests a confirmation step, possibly configurable, before hard `/new` or `/reset` starts a fresh session.

## Best Possible Solution

Decide the destructive command policy first; if accepted, implement one shared default-on confirmation for hard `/new` and `/reset` across channel and typed Control UI paths, avoiding new config unless maintainers explicitly approve it.

## Reproduction Assessment

Yes, source-level. Current tests and source show typed `/new` and `/reset` bypass confirmation, while the channel reset path marks those commands as reset-triggering immediately.

## Solution Assessment

Unclear until maintainers choose the policy. A single shared hard-reset confirmation looks more maintainable than another UI-only guard, while the configurable `confirmSteps` part needs explicit product approval.
