## Label Justifications

- `P2`: This is a normal-priority cross-channel session-safety improvement requiring bounded product and compatibility design.
- `impact:session-state`: Accidental reset replaces the active session and conversation context, even though prior sessions may remain discoverable.

## Summary

Current main and v2026.6.5 intentionally execute authorized typed `/new` and `/reset` immediately; the merged Control UI safeguard only covers button-triggered resets, so the command-level safety request remains useful but needs a maintainer product decision before implementation.

## What This Changes

Add confirmation before typed `/new` and `/reset`, potentially showing the session size and allowing configurable confirmation depth.

## Best Possible Solution

If maintainers approve changing the command contract, add one fixed short-lived confirmation for authorized human-authored bare reset commands, preserve a clearly documented immediate reset escape path, and leave Control UI affordance work to https://github.com/openclaw/openclaw/issues/76138.

## Reproduction Assessment

Yes. Current source and tests establish that authorized typed `/new` and `/reset` immediately rotate session state without a confirmation phase.

## Solution Assessment

No as proposed. Configurable two-to-three-step confirmation is broader than necessary; one fixed confirmation plus an immediate escape path is the narrower maintainable design if maintainers choose to revise the command contract.
