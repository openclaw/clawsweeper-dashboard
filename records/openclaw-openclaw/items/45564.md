## Summary

Keep open. Current main and v2026.6.8 still preserve immediate typed `/new` and `/reset` behavior without confirmation, while related closed UI-button issues only fixed the button-triggered New Session path and explicitly left broader typed-command policy here.

## What This Changes

The issue requests a confirmation guard for destructive session-reset commands such as `/new` and `/reset`, with an optional configurable confirmation count.

## Best Possible Solution

Define one maintainer-approved destructive-command confirmation contract, likely a single timed repeat/confirm flow, then apply it consistently to typed Control UI and Gateway reset command surfaces without adding config unless maintainers explicitly want that knob.

## Reproduction Assessment

Yes, source-level. Current tests assert typed `/new` and `/reset` do not call `confirm`, and Gateway reset handling accepts bare reset commands directly.

## Solution Assessment

No, not exactly as written. The 2-3 click configurable version is likely heavier than necessary; the narrower path is a maintainer-approved single confirmation or timed repeat contract across the command surfaces.
