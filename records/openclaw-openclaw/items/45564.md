## Summary

Keep open. Current main and v2026.6.10 still leave typed `/new` and `/reset` unconfirmed across the Control UI and TUI, while the merged button-confirmation work only covers adjacent UI affordance cases; the remaining behavior needs maintainer UX/config direction.

## What This Changes

The issue requests a confirmation flow, optionally configurable, before explicit `/new` and `/reset` commands clear or rotate session state.

## Best Possible Solution

Define one maintainer-approved hard-reset confirmation contract for explicit commands across Control UI, TUI, and channel/Gateway paths, with no new config knob unless maintainers explicitly accept the compatibility cost.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 tests and handlers show typed `/new` and `/reset` bypass confirmation in Control UI, and the TUI command handler resets or rotates sessions immediately.

## Solution Assessment

Unclear as an implementation request. A single confirmation step may be the right UX, but confirmation depth, config shape, timeout behavior, and which command surfaces change need maintainer approval first.
