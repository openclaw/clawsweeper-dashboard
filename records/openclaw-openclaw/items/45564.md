## Summary

Keep open. Current main and v2026.5.20 still preserve immediate typed `/new` and `/reset` behavior outside the button-triggered Control UI guard, so this remains a real cross-surface destructive-session UX/config policy decision rather than cleanup.

## What This Changes

The issue requests a confirmation step for destructive `/new` and `/reset` commands to reduce accidental session loss.

## Best Possible Solution

Adopt one shared safe-default confirmation contract for user-authored destructive chat resets, with explicit API/automation bypass semantics plus aligned config, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows typed Control UI `/new` bypasses the only `confirmReset` guard, typed `/reset` is sent to `chat.send`, shared channel and TUI reset paths execute immediately, and tests assert no-confirm typed commands.

## Solution Assessment

Unclear. The repeat-command timeout pattern discussed in comments is plausible, but maintainers still need to choose covered surfaces, timeout/config shape, loss-summary behavior, and API/automation compatibility.
