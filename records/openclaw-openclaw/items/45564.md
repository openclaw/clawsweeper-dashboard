## Summary

Keep open. Current main only confirms the Control UI toolbar New Session action; typed WebChat `/new` and `/reset`, shared channel/native reset command handling, and gateway agent reset handling still perform destructive reset behavior immediately. The issue discussion narrowed the UX toward a single repeat-command confirmation, but that shared cross-surface contract is not implemented.

## What This Changes

The issue requests a confirmation step before `/new` or `/reset` wipes a session, with discussion converging on a single repeat-command confirmation and optional lost-session summary/configuration.

## Best Possible Solution

Define one shared destructive-session-reset confirmation contract for bare `/new` and `/reset`, with safe default behavior and clear timeout semantics; route typed Control UI/WebChat commands, channel/native command handling, and gateway agent reset paths through it while preserving `/reset soft` and reset-tail behavior.

## Reproduction Assessment

Yes. Current source and regression tests give a high-confidence static reproduction: typed WebChat `/new` or `/reset` bypasses confirmation, and shared reset-command tests assert immediate bare reset acknowledgements.

## Solution Assessment

Unclear. The existing toolbar-only confirmation is not the best complete solution for this issue; the maintainable solution needs a maintainer-approved cross-surface confirmation contract before implementation.
