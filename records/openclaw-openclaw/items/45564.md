Keep open. Current main still executes user-facing `/new` and `/reset` paths without a shared confirmation contract, and the related toolbar issue only covers one Control UI affordance; the remaining work needs maintainer UX/config direction before implementation.

## What This Changes

The issue requests a confirmation step, ideally configurable, before `/new` and `/reset` wipe or replace session state.

## Best Possible Solution

Adopt one shared safe-default confirmation contract for user-authored destructive session resets, with explicit bypass semantics for automation/API paths and aligned config, docs, and regression coverage.
