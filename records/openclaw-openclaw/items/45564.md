## Summary

Keep open. Current main only confirms button-triggered Control UI new-session resets; typed `/new` and `/reset`, shared channel reset triggers, and Gateway agent resets still proceed immediately, so the requested cross-surface confirmation behavior remains unresolved.

## What This Changes

Add a confirmation step before destructive `/new` and `/reset` session wipes, with an optional configurable confirmation policy.

## Best Possible Solution

Define one shared destructive-reset confirmation contract across typed Control UI, shared channel/native commands, and Gateway agent resets while preserving reset tails, `/reset soft`, non-destructive fresh-session creation, and documented automation flows.

## Reproduction Assessment

Yes, by source inspection rather than live execution. Typed Control UI commands, shared channel reset triggers, native reset acknowledgements, and Gateway agent resets all still have immediate current-main paths, although this is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear. The single repeat-command confirmation discussed in comments is narrower than the original multi-click request, but maintainers still need to choose the exact surfaces, timeout, config shape, and automation compatibility before implementation.
