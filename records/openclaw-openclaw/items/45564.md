## Summary

Keep open. Current main only partially addresses nearby problems: button-triggered New Session has a confirmation guard and `/reset soft` args are preserved, but typed `/new` and hard `/reset` still execute without confirmation, and the requested configurable/multi-step policy needs maintainer product direction.

## What This Changes

Request to add a confirmation flow, optionally configurable, before destructive `/new` and `/reset` session resets.

## Best Possible Solution

Adopt one maintainer-approved reset-confirmation policy for destructive typed resets, likely one default confirmation with an optional session-loss summary, and avoid adding a new config surface unless explicitly approved.

## Reproduction Assessment

Yes, by source inspection. Current main and the latest release both have tests and dispatch code showing typed `/new` and `/reset` bypass confirmation, while the gateway reset handler executes the reset path immediately.

## Solution Assessment

Unclear. A confirmation step is a plausible safety fix, but the best implementation depends on maintainer choices about command scope, confirmation count, summary text, and whether the requested config surface is worth adding.
