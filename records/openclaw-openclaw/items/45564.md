## Summary

Keep open. Current main and v2026.6.8 still preserve immediate typed `/new` and `/reset` behavior without a confirmation step; the merged Control UI work only addressed button-triggered New Session paths.

## What This Changes

The issue requests a confirmation step, with an optional config knob, before typed `/new` and `/reset` commands wipe or rotate session context.

## Best Possible Solution

Define one maintainer-approved destructive-command confirmation contract, likely a single timed repeat/confirm flow, then apply it consistently to typed Control UI and Gateway reset-command surfaces without adding config unless maintainers explicitly approve that knob.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.8 tests assert typed `/new` and `/reset` do not call `confirm`, and the Gateway reset handler immediately acknowledges bare reset commands.

## Solution Assessment

No, not exactly as written. The 2-3 click configurable version is likely heavier than necessary; a narrower maintainer-approved single confirmation or timed-repeat contract across command surfaces is the more maintainable path.
