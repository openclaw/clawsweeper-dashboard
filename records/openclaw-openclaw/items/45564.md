## Summary

Keep open. Current main still executes typed `/new` and `/reset`, shared channel reset commands, TUI resets, and Gateway agent reset calls without a shared confirmation policy; this is a useful UX/config request but needs maintainer product direction before implementation.

## What This Changes

The issue requests a confirmation step, ideally configurable, before `/new` or `/reset` resets or replaces a session.

## Best Possible Solution

Define one shared safe-default confirmation contract for user-authored destructive session resets, with explicit bypass semantics for automation/API paths and aligned config, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the remaining typed/shared/TUI/Gateway reset paths execute immediately, and tests explicitly preserve immediate typed behavior; this is source-reproducible as a feature gap rather than a broken existing contract.

## Solution Assessment

Unclear. The repeat-command timeout pattern discussed in comments is plausible, but maintainers still need to decide the covered surfaces, timeout/config shape, loss-summary behavior, and automation compatibility before implementation.
