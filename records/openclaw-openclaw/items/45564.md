## Summary

Keep open. Current main and the latest release still intentionally leave typed `/new` and `/reset` unconfirmed; related button-confirmation and daily-reset work only partially overlaps, so the remaining explicit-command policy needs maintainer direction.

## What This Changes

Add a confirmation or multi-step guard for explicit `/new` and `/reset` session resets, ideally with configurable confirmation depth.

## Best Possible Solution

Define a maintainer-approved hard-reset confirmation contract across Control UI typed commands, TUI, and channel commands, with default-safe behavior and no new config knob unless product owners explicitly accept the config surface.

## Reproduction Assessment

Yes. Current source and tests show typed `/new` and `/reset` bypass confirmation in the Control UI, and TUI command handlers reset immediately without a guard.

## Solution Assessment

Unclear as a direct implementation request. A single confirmation step may be enough, but confirmation depth, config shape, and which surfaces should change need maintainer UX/product approval first.
