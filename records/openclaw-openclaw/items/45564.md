## Best Possible Solution

Adopt one maintainer-approved destructive-session confirmation policy for typed reset/new commands, safe by default, and avoid adding a new config surface unless maintainers explicitly accept that tradeoff.

## Reproduction Assessment

Yes, by source inspection. Current main tests and handlers explicitly preserve unconfirmed typed `/new` and `/reset` behavior while only confirming the override/button path.

Unclear. A single repeat-within-timeout confirmation is plausible, but the best fix depends on maintainer choices about command scope, `/reset soft`, summary text, timeout, and config surface.
