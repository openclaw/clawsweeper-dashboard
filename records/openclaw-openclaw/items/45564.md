## Summary

Keep open: current main and the latest release have mitigated the old Control UI plus-button loss path, but the typed `/new` and `/reset` command paths still acknowledge and execute reset directly, so the central command-confirmation request remains necessary and needs maintainer product direction.

## What This Changes

The issue requests a confirmation step for destructive `/new` and `/reset` session actions, ideally with an optional configurable confirmation count.

## Best Possible Solution

Keep the shipped separate-session Control UI behavior, and decide a narrow default-on confirmation policy for typed destructive reset commands before adding code or config surface.

## Reproduction Assessment

Yes. Source and tests show current main immediately acknowledges bare `/new` and `/reset`, and the latest release still has the same typed command behavior.

## Solution Assessment

Unclear until maintainers choose the policy. A single short repeat-to-confirm guard is likely narrower than 2-3 confirmations or a new `confirmSteps` config surface, but that is a product decision.
