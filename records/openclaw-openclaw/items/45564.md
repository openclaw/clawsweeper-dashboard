## Summary

Keep open. Current main and the latest release still let typed `/new` and hard `/reset` execute without confirmation, while the related merged work only covered a narrower Control UI button path and the remaining command/config policy needs maintainer direction.

## What This Changes

The issue requests a confirmation guard for destructive `/new` and `/reset` session commands, ideally with a configurable confirmation count.

## Best Possible Solution

Adopt one maintainer-approved reset-confirmation policy for destructive typed resets, likely one default confirmation with an optional session-loss summary, while avoiding a new config surface unless explicitly approved.

## Reproduction Assessment

Yes, by source inspection. Current main and v2026.6.6 tests explicitly assert typed `/new` and typed `/reset` do not call `confirm`, while the gateway handler immediately handles bare `/new` and `/reset`.

## Solution Assessment

Unclear. A confirmation step is a plausible safety fix, but the best implementation depends on maintainer choices about command scope, confirmation count, summary text, and whether the requested config surface is worth adding.
