## Summary

Keep open. Current main only partially addresses the accidental-wipe problem: Control UI `/new` now creates and switches to a fresh dashboard session, but typed `/reset` and shared native/channel `/new` and `/reset` still execute immediate reset behavior without confirmation.

## What This Changes

The issue requests a configurable confirmation step before `/new` or `/reset` wipes a session.

## Best Possible Solution

Define a shared destructive-session-reset confirmation contract for bare `/reset` and destructive native/channel `/new`, while preserving non-destructive fresh-session creation plus `/reset soft` and reset-tail behavior.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction: send typed Control UI `/reset` or native/channel bare `/new` or `/reset`, and the current code/tests show immediate reset handling with no confirmation state.

## Solution Assessment

No, the original 2-3 click/configurable shape is not clearly the best default. The discussion points to a narrower repeat-command confirmation, but maintainers should approve the cross-surface contract before implementation.
