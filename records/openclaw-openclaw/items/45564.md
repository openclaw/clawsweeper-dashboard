## Label Justifications

- `P2`: Accidental typed resets can disrupt active conversations, but the change is a bounded compatibility-sensitive UX improvement rather than an urgent outage.
- `impact:session-state`: The issue concerns rotating the active session ID and replacing current conversation context.

## Summary

Current main still intentionally executes typed `/new` and `/reset` immediately; the shipped Control UI confirmation solved only button-triggered resets, leaving this command-level behavior as a genuine product and compatibility decision.

## What This Changes

Add confirmation before typed `/new` and `/reset`, ideally with configurable confirmation depth, to reduce accidental active-session replacement.

## Best Possible Solution

Keep the shipped Control UI confirmation, then explicitly decide whether human-authored chat resets should use a short repeat-to-confirm interaction while programmatic/API reset paths remain immediate, avoiding new configuration unless a fixed safe default proves inadequate.

## Reproduction Assessment

Yes. Current source and focused gateway tests establish that authorized typed `/new` and `/reset` execute immediately without confirmation.

## Solution Assessment

Unclear. Repeat-to-confirm may be suitable for human chat commands, but applying it globally or adding configurable confirmation counts is not yet proven to be the narrowest compatible design.
