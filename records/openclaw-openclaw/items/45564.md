## Summary

Keep open. Current main has partially reduced the Control UI New Chat risk, but typed `/reset`, shared channel reset commands, and gateway agent reset commands still proceed immediately, and the requested configurable confirmation policy remains unresolved.

## What This Changes

The issue requests a confirmation flow, ideally configurable, before `/new` or `/reset` wipes a session.

## Best Possible Solution

Define one shared destructive-reset confirmation contract for user-authored reset commands, with explicit exclusions for automation/API reset paths or documented reasons they remain immediate.

## Reproduction Assessment

Yes, by source inspection: typed Control UI `/reset`, shared channel reset triggers, and gateway agent reset paths still reset immediately on current main. This is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear. The repeat-command confirmation discussed in comments is plausible, but maintainers still need to decide the covered surfaces, timeout/config shape, and automation compatibility before implementation.
