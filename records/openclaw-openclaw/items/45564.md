## Summary

Keep open. Current main still lacks a shared confirmation policy for destructive `/new` and `/reset` flows: the Control UI toolbar, typed Control UI commands, shared channel reset handling, and Gateway agent reset handling all still have immediate reset paths, while the requested config surface is absent.

## What This Changes

The issue requests a confirmation step, ideally configurable, before `/new` or `/reset` wipes or rotates a session.

## Best Possible Solution

Define one shared destructive reset confirmation contract for user-authored `/new` and `/reset`, with explicit exclusions or bypasses for automation/API paths and aligned config, docs, and tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the relevant `/new` and `/reset` paths still reset or create sessions immediately, and tests lock in immediate typed Control UI behavior; this is a feature request rather than a broken existing contract.

## Solution Assessment

Unclear. The repeat-command confirmation discussed in comments is plausible, but maintainers still need to decide the covered surfaces, timeout/config shape, and automation compatibility before implementation.
