## Summary

Keep open. Current main and latest release still only confirm button-triggered Control UI reset overrides; typed `/new` and `/reset`, shared channel reset handling, TUI commands, and the Gateway reset API remain immediate, so the central request is still unresolved and needs a maintainer UX/config decision.

## What This Changes

The issue requests a confirmation flow, ideally configurable, before `/new` or `/reset` can wipe or rotate session state.

## Best Possible Solution

Adopt one safe-default confirmation contract for user-authored destructive reset/new-session commands, with explicit API/automation bypass semantics, aligned config/docs, and regression coverage across Control UI, shared channel, TUI, and Gateway paths.

## Reproduction Assessment

Yes. Source inspection on current main shows typed Control UI `/new` bypasses the only confirmation guard, typed `/reset` dispatches immediately, shared channel and TUI reset paths execute immediately, and tests assert no-confirm typed commands.

## Solution Assessment

Unclear. The repeat-command timeout pattern discussed in comments is plausible, but maintainers still need to choose covered surfaces, timeout/config shape, loss-summary behavior, and API/automation compatibility.
