## Summary

Keep open. Current main and the latest release still preserve unconfirmed typed `/new` and `/reset` behavior, and the related merged work intentionally covered only button-triggered Control UI new-session resets.

## What This Changes

The issue requests a confirmation step for destructive `/new` and `/reset` session wipes, with discussion favoring a single repeat-confirmation pattern and possible configuration.

## Best Possible Solution

Adopt one maintainer-approved destructive-session confirmation policy for typed reset/new commands, likely safe by default, and avoid adding a config surface unless maintainers explicitly accept that tradeoff.

## Reproduction Assessment

Yes, by source inspection. Current main tests and handlers explicitly preserve unconfirmed typed `/new` and `/reset` behavior while only confirming the button-triggered override path.

## Solution Assessment

Unclear. A single repeat-within-timeout confirmation is plausible, but the best fix depends on maintainer choices about command scope, `/reset soft`, summary text, timeout, and config surface.
