## Label Justifications

- `P1`: This is a confirmed regression that blocks the Mattermost slash-command workflow for real users across released builds.
- `impact:message-loss`: Mattermost slash commands are accepted by Mattermost but never reach the initialized command handler, returning the placeholder 503 instead.

## Summary

Keep open: current main and the latest tagged release still preserve the Mattermost slash-command split-state failure mode, and the focused fix path remains the open PR at https://github.com/openclaw/openclaw/pull/68089.

## What This Changes

Mattermost native slash-command callbacks should execute after successful activation instead of returning HTTP 503 "not yet initialized."

## Best Possible Solution

Keep this issue as the canonical bug until https://github.com/openclaw/openclaw/pull/68089 or an equivalent Mattermost plugin fix lands with regression coverage proving the initialized callback no longer returns the 503 placeholder.

## Reproduction Assessment

Yes. Current main still has the route-side 503 fallback reading a module-local accountStates map while activation writes through the monitor import path, and user comments confirm the same patch removes the failure in installed releases.

## Solution Assessment

Yes. The best path is to keep this issue open while maintainers land, revise, or close the focused Mattermost fix PR rather than opening a duplicate repair lane.
