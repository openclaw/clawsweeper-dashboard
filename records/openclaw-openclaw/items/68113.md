## Summary

Keep this open: current main still has the Mattermost slash route loaded through the sync bundled-entry loader while slash activation writes to a module-local state map, and the focused fix candidate #68089 is still open rather than merged. #69038 was a confirmed workaround patch but closed unmerged, so the shipped user problem is not resolved in this repository yet.

## What This Changes

Mattermost native slash command callbacks should stop returning the permanent uninitialized 503 after command activation in v2026.4.15 and later.

## Best Possible Solution

Land or revise one Mattermost plugin fix, likely #68089 or an equivalent, that makes route registration and monitor activation share the same slash-command state across supported loader paths and adds a regression test for the initialized callback path.

## Reproduction Assessment

Yes. The report has repeated user reproductions on v2026.4.15 through v2026.4.22, and current source still shows the route-side empty-state 503 path combined with split route/runtime loading.

## Solution Assessment

Yes. The active #68089 direction is the narrowest maintainable fix because it removes the duplicate module instance at the Mattermost plugin boundary without changing slash command semantics or core routing policy.
