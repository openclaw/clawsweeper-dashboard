## Summary

Current main and shipped releases no longer preserve the reported npm-package split-state failure: the bundled-entry loader now native-requires compiled dist sidecars before falling back to jiti, so the Mattermost route registrar and monitor activation path share Node's ESM module instance for `slash-state`. The original Mattermost-local symptoms and the related PRs were useful in identifying the module-identity bug, but the generic loader fix shipped in `v2026.4.25` and is present in the latest release line.

## What This Changes

The report asks OpenClaw to restore Mattermost native slash commands so `/oc_*` callbacks stop returning persistent HTTP 503 "not yet initialized" after command activation in v2026.4.15.

## Best Possible Solution

Keep the generic bundled-entry loader fix and close this report as shipped. The open Mattermost-specific PR can be reviewed separately as potentially redundant cleanup or additional source-dev guardrail work, but release users should upgrade to `v2026.4.25` or newer.

## Reproduction Assessment

Yes. The original v2026.4.15 source path is reproducible from code: non-Windows packaged sidecar loads used jiti while monitor activation used native ESM, leaving `accountStates` empty in the route-side module. Current packaged main no longer follows that path because built JavaScript sidecars use native require before jiti.

## Solution Assessment

Yes. The shipped generic loader fix is the narrowest maintainable release fix because it removes the duplicate-module cause for all compiled bundled sidecars without adding Mattermost-specific global state. #68089's static import approach is also plausible, but it is no longer required to solve the reported npm release failure.
