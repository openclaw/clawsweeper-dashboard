- `P1`: This is a corroborated regression that breaks a real channel command workflow for Mattermost users.
- `impact:message-loss`: The reported failure suppresses all Mattermost slash-command responses behind a 503 callback result.

## Summary

Keep this open: current main and the latest tagged release still preserve the Mattermost slash-command split-state path that can leave the callback route seeing an empty state map and returning the reported 503, while the focused fix path remains an open conflicting PR.

## What This Changes

Mattermost native slash-command callbacks should stop returning 503 "not yet initialized" after command activation in v2026.4.15 and later.

## Best Possible Solution

Keep this issue as the canonical bug until https://github.com/openclaw/openclaw/pull/68089, or an equivalent Mattermost plugin fix, lands with regression coverage proving route registration and monitor activation share one slash-command state instance.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main still registers the callback route through the bundled-entry loader while monitor activation imports the same state module through the runtime path, and the route-side empty state returns the exact reported 503 text.

## Solution Assessment

Yes, keeping this open and routing through the focused Mattermost fix PR is the right next step. The durable fix should remove the duplicate module instance hazard, or otherwise share state across those loader paths, with a regression test for the initialized callback path.
