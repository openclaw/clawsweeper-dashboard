## Summary

Keep this open: current main and the latest release still preserve the Mattermost slash-command split-state failure mode, and the focused fix path remains open. The issue is well corroborated by user reports, source inspection, and related PR proof.

## What This Changes

The issue reports Mattermost native slash commands returning a permanent HTTP 503 "not yet initialized" response after gateway activation.

## Best Possible Solution

Keep this issue as the canonical bug until https://github.com/openclaw/openclaw/pull/68089 or an equivalent Mattermost plugin fix guarantees one shared slash-command state instance across supported loader paths with regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible with high confidence: the route can load slash state through the bundled-entry loader while monitor activation writes through the native ESM import path, and the route-side empty state returns the reported 503 text.

## Solution Assessment

Yes. The narrow maintainable fix is to keep this issue open while a Mattermost plugin change makes route registration and monitor activation share slash-command state across loader paths; a separate repair lane would duplicate the existing open implementation candidate.
