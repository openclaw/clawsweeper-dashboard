## Summary

Keep this open: current main and the latest release still preserve the Mattermost slash-command split-state failure mode, and the focused fix PR remains open/unmerged. User comments, the duplicate report, and the two related PRs all support the same root cause and expected fix boundary.

## What This Changes

Mattermost slash commands return HTTP 503 after startup even though command registration and activation logs succeed.

## Best Possible Solution

Keep this issue as the canonical bug while maintainers land or revise https://github.com/openclaw/openclaw/pull/68089, or an equivalent Mattermost plugin fix, with regression coverage for the initialized callback no longer returning the 503 placeholder across loader paths.

## Reproduction Assessment

Yes. Current main is source-reproducible with high confidence: the route can load slash state through the bundled-entry loader while monitor activation writes through the native ESM import path, and the empty route-side map returns the reported 503 text.

## Solution Assessment

Yes. The best path is to keep this issue open while maintainers land or revise the focused Mattermost fix PR so route registration and monitor activation share one state instance with a dual-loader regression test.
