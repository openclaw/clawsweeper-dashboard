## Summary

Current main still contains the Mattermost slash-command split-state failure mode, and the active fix candidate is still open rather than merged. This issue should remain open until #68089 or an equivalent Mattermost plugin fix lands and, if maintainers require it, ships in a release.

## What This Changes

Mattermost native slash commands report a persistent 503 "not yet initialized" response after startup even though command registration logs show successful activation.

## Best Possible Solution

Keep this issue as the canonical Mattermost slash-command regression until one reviewed fix lands. The desired end state is a Mattermost plugin change that guarantees route registration and monitor activation share the same slash-command account state across supported loader paths, ideally with a regression test for the initialized callback no longer returning the 503 placeholder.

## Reproduction Assessment

Yes. The report has a high-confidence reproduction path from user logs and current source: activate commands through the Mattermost monitor path, then hit the callback route loaded through the sync bundled-entry path and observe the route-side empty state returning the 503 placeholder.

## Solution Assessment

Yes. #68089's static import approach is a narrow maintainable fix for the observed module-identity bug; an equivalent single-instance route/runtime seam plus a dual-loader regression test would also be acceptable.
