## Label Justifications

- `P1`: The regression blocks the native Mattermost slash-command workflow for affected users even after command registration succeeds.
- `impact:message-loss`: Mattermost slash-command callbacks fail before reaching the command handler and return the 503 placeholder instead.

## Summary

Keep this open: current main and the latest release still preserve the Mattermost split-state path that can leave slash callbacks on an empty route-side state map, and the active implementation candidate remains open and unmerged. The reporter comments, duplicate issue, and linked PRs all point to the same regression and confirm that sharing the route/activation state resolves the 503 symptom.

## What This Changes

The issue reports Mattermost native slash commands returning a persistent 503 not-initialized response after gateway startup.

## Best Possible Solution

Keep this as the canonical bug until https://github.com/openclaw/openclaw/pull/68089 or an equivalent Mattermost plugin fix makes route registration and monitor activation share one slash-command state instance with dual-loader regression coverage.

## Reproduction Assessment

Yes. Current main still has the split loader/import path, module-local slash state, and exact 503 fallback, so the issue is source-reproducible with high confidence even without a live Mattermost server.

## Solution Assessment

Yes. The best path is to review and land or revise the existing focused Mattermost fix PR, rather than opening a duplicate ClawSweeper repair lane.
