## Summary

Keep this open: current `main` and the latest release still preserve the Mattermost slash-command split-state failure mode, and the focused fix path remains the open PR at https://github.com/openclaw/openclaw/pull/68089. Multiple user comments also corroborate that the alternate local patch resolved the 503, but that patch was not merged or shipped.

## What This Changes

Mattermost native slash commands return HTTP 503 "not yet initialized" after successful activation in v2026.4.15 and later releases.

## Best Possible Solution

Land or revise https://github.com/openclaw/openclaw/pull/68089, or an equivalent Mattermost plugin fix, so route registration and monitor activation share one slash-command state instance across supported loader paths with regression coverage for the initialized callback no longer returning the 503 placeholder.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current `main` can register the route through the bundled-entry loader while monitor activation writes through the ESM import path, and the route-side empty state still returns the exact reported 503 response.

## Solution Assessment

Yes, keeping this open while maintainers review the focused Mattermost fix PR is the best current path. A separate automated repair would duplicate https://github.com/openclaw/openclaw/pull/68089 unless maintainers first close or abandon that PR.
