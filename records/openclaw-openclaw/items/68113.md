## Summary

Keep this open: current main and the latest tagged release still preserve the Mattermost slash-command split-state failure mode, and the active fix remains the open Mattermost PR at https://github.com/openclaw/openclaw/pull/68089 rather than merged code.

## What This Changes

The issue reports that Mattermost native slash commands return a permanent HTTP 503 "not yet initialized" response after OpenClaw logs successful slash-command activation.

## Best Possible Solution

Keep this as the canonical bug until a Mattermost plugin fix lands, ideally by reviewing and landing https://github.com/openclaw/openclaw/pull/68089 or an equivalent fix with regression coverage for the initialized callback path.

## Reproduction Assessment

Yes, source inspection plus repeated user reports give a high-confidence reproduction path: enable native Mattermost slash commands, wait for the activation log, then hit the callback and observe the route-side 503 fallback instead of the initialized handler's 405/command response. I did not run a live Mattermost instance in this read-only review.

## Solution Assessment

Yes, keeping this issue open while the focused Mattermost fix PR is reviewed is the narrowest maintainable path. The fix should make route registration and monitor activation share the same slash-command state across supported loader paths without changing command semantics.
