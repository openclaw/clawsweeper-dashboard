## Summary

This should stay open: current main still prompts inline-button-capable agents to use a non-existent top-level `buttons=[[...]]` message-tool argument, while the shipped shared contract and Telegram runtime render buttons from `presentation` or `interactive`.

## What This Changes

The issue reports Gemini emitting Telegram inline-button JSON as visible text instead of sending structured inline keyboard buttons.

## Best Possible Solution

Update the agent prompt and regression tests to steer inline-button-capable runs toward `presentation.blocks[].buttons` and keep raw Telegram JSON parsing out of the primary path.

## Reproduction Assessment

Yes. Source inspection on current main shows a deterministic contract mismatch: the prompt advertises `buttons=[[...]]`, while the message tool and Telegram renderer expect `presentation` or `interactive` payloads.

## Solution Assessment

No for parsing raw `[[ ... ]]` text as the primary fix. The narrow maintainable solution is to align prompt guidance and tests with the already-shipped shared `presentation` contract.
