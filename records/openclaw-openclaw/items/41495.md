## Summary

Keep open: current main and the latest release still advertise the stale raw `buttons=[[...]]` hint for inline-button-capable agents, while the current message-tool and Telegram paths use the structured `presentation` contract, so the reported Gemini/Telegram button breakage remains source-reproducible.

## What This Changes

The issue asks OpenClaw to make Gemini-backed Telegram replies render inline buttons through the message tool instead of showing raw button JSON in chat.

## Best Possible Solution

Align the shared agent prompt and regression tests with `message(action=send)` using `presentation.blocks[].buttons`, and avoid making final-text `[[...]]` parsing the primary button delivery contract.

## Reproduction Assessment

Yes, at source level: current main still prompts inline-button-capable agents to use raw `buttons=[[...]]`, while the Telegram message-tool schema exposes `presentation` and omits top-level `buttons`. I did not establish a live Gemini/Telegram failing run in this read-only review.

## Solution Assessment

Yes. A narrow prompt and regression-test update is the maintainable fix; parsing raw button JSON out of final assistant text would create a second delivery contract outside the shared presentation renderer.
