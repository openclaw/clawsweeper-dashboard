## Summary

Keep open: current main and the latest release still tell inline-button-capable agents to use raw `buttons=[[...]]` syntax, while the current Telegram message-tool path exposes `presentation.blocks[].buttons` and no top-level `buttons` property. The related group-prompt PR is useful adjacent work, but it does not fix the shared inline-button prompt guidance that can steer Gemini into emitting JSON text.

## What This Changes

The issue asks OpenClaw to stop steering Gemini-backed Telegram agents into raw inline-button JSON text and instead use the message tool's native button format.

## Best Possible Solution

Align the shared agent prompt and prompt tests with `message(action=send)` plus `presentation.blocks[].buttons`, and avoid adding raw final-text JSON parsing as the primary fix.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main and `v2026.5.20` still advertise raw `buttons=[[...]]` while the current Telegram message tool exposes `presentation.blocks[].buttons` and tests assert no top-level `buttons` property. I did not run a live Gemini plus Telegram scenario in this read-only review.

## Solution Assessment

Yes, a narrow prompt/schema alignment is the best fix. Parsing raw final-text `[[...]]` JSON would create a second source of truth outside the shared presentation contract.
