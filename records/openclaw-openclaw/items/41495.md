## Summary

Keep open: current main and the latest release still tell inline-button-capable agents to use raw `buttons=[[...]]` prompt syntax, while the supported shared message-tool and Telegram rendering contract is `presentation.blocks[].buttons`. The related open group-prompt PR is useful adjacent work, but it does not close this issue and does not fix the shared inline-button prompt guidance that can steer Gemini into emitting JSON text.

## What This Changes

The issue reports that Gemini-backed Telegram agents emit inline-button JSON as message text instead of using the supported message-tool presentation button contract.

## Best Possible Solution

Align the shared agent prompt and prompt regression tests with `message(action=send)` plus `presentation.blocks[].buttons`, and avoid adding raw final-text JSON parsing as the primary fix.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: current main still advertises raw `buttons=[[...]]` in the shared prompt while the tool schema, docs, and Telegram runtime use `presentation.blocks[].buttons`. I did not run a live Gemini plus Telegram scenario in this read-only review.

## Solution Assessment

Yes for a narrow prompt/schema alignment repair; no for parsing raw final-text `[[ ... ]]` JSON, because that would add a second source of truth outside the shared presentation contract.
