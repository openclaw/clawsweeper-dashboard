## Summary

Keep open: current main still advertises a raw Telegram `buttons=[[...]]` prompt while the message tool and Telegram runtime render inline buttons from structured `presentation`/`interactive` payloads, so the reported Gemini failure remains a valid source-backed bug.

## What This Changes

The issue reports that Gemini-generated Telegram inline buttons appear as raw JSON text instead of rendering as interactive inline keyboard buttons.

## Best Possible Solution

Align the system prompt and Telegram docs with the existing `presentation.blocks[].buttons` contract, keep button rendering structured, and add focused regression coverage for the prompt/tool/runtime contract.

## Reproduction Assessment

Yes, source-reproducible: current main still emits raw `buttons=[[...]]` guidance while the message tool schema and Telegram runtime only render structured `presentation` or `interactive` button payloads. A live Gemini/Telegram run was not performed, so the provider-specific frequency is not measured here.

## Solution Assessment

Yes: the narrow maintainable fix is to correct prompt/docs/tests to use the existing presentation payload. Parsing arbitrary raw button JSON from message text would add a new directive surface and should not be the default repair.
