## Summary

Keep this issue open: current main still contains the reported prompt/runtime contract mismatch, so this is not stale or implemented.

## What This Changes

Gemini-backed agents are reported to show Telegram inline button JSON as message text instead of sending structured button payloads that render as Telegram inline keyboards.

## Best Possible Solution

Align agent prompt guidance and Telegram docs around `message(action="send")` with `presentation.blocks[{type:"buttons", buttons:[{label,value,style?}]}]`, then add focused regressions proving Telegram presentation buttons render as inline keyboards.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: the prompt instructs legacy `buttons=[[...]]`, while the schema/runtime only attach inline keyboards from structured `presentation` or `interactive` payloads.

## Solution Assessment

Yes. The narrow maintainable fix is contract alignment in prompt/docs/tests plus Telegram presentation regression coverage; parsing arbitrary raw JSON-like assistant text should be optional compatibility work, not the default fix.

Update `src/agents/system-prompt.ts` so inline-button-capable non-Slack runtimes tell models to call `message(action="send")` using the shared `presentation` payload, e.g. `presentation={"blocks":[{"type":"buttons","buttons":[{"label":"...","value":"...","style":"success"}]}]}`. Update `src/agents/system-prompt.test.ts` to assert the new presentation guidance and no longer assert `buttons=[[{text,callback_data,style?}]]`. Align Telegram-facing docs so they point agents/users to the generic presentation contract for inline buttons; avoid introducing provider-native top-level `buttons` as the shared-tool recommendation. Add or adjust focused Telegram regression coverage showing presentation button blocks render to Telegram inline keyboards. Do not add a broad raw-text JSON parser unless maintainers explicitly decide to preserve legacy assistant text syntax. Changelog only if the final change is user-facing enough under repo policy.
