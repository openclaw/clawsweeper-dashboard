## Summary

Keep open: current main still contains a source-level mismatch where the shared agent prompt advertises raw Telegram-style `buttons=[[...]]` syntax, while the actual shared message tool and Telegram renderer expect semantic `presentation` button payloads. The related open group-prompt PR is adjacent and useful context, but it does not replace the shared prompt/schema repair for this issue.

## What This Changes

The issue asks OpenClaw to make Gemini-driven Telegram inline buttons render as real buttons instead of raw JSON text.

## Best Possible Solution

Align the shared agent prompt and regression tests with the existing `presentation.blocks[].buttons` contract, and avoid parsing raw `[[ ... ]]` button JSON as the primary fix.

## Reproduction Assessment

Yes. Source inspection on current main shows a deterministic mismatch: the prompt advertises raw `buttons=[[...]]`, while the message tool schema, tests, docs, and Telegram runtime expect `presentation` or legacy `interactive` payloads.

## Solution Assessment

Yes for a narrow prompt/schema alignment repair; no for raw final-text JSON parsing because that would bypass the shared presentation contract and add a second source of truth.
