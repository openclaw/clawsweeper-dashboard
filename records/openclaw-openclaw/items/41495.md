## Summary

Keep open: current main has structured Telegram button delivery, but the reported Gemini failure is still plausible because the prompt advertises a raw-looking `buttons=[[...]]` shorthand while the current message tool schema and Telegram send path use structured `presentation`/`interactive` payloads, and OpenClaw intentionally does not execute JSON-like tool calls emitted as assistant text.

## What This Changes

The issue reports Gemini producing Telegram inline button JSON as message text instead of invoking the message tool buttons contract.

## Best Possible Solution

Keep the issue open and align the structured message-tool guidance/schema around `presentation` or a supported typed buttons alias, instead of silently executing arbitrary assistant-text JSON as callbacks.

## Reproduction Assessment

No high-confidence live repro was established. Source inspection does show that if Gemini emits the reported `[[...callback_data...]]` block as assistant text, current main has no parser for it and the documented raw-tool-text policy leaves JSON-like tool calls as text.

## Solution Assessment

No, the regex/post-processing parser proposed in the issue is not clearly the best first fix. The safer direction is to make Gemini use the structured message tool payload that Telegram already renders, or intentionally add a typed compatibility alias with tests and docs.
