## Summary

Keep open: current main and v2026.5.19 still advertise raw `buttons=[[...]]` prompt syntax, while the supported shared message-tool and Telegram rendering path uses `presentation.blocks[].buttons`. The related open group-prompt PR is useful adjacent work, but it does not fix the shared inline-button prompt guidance that can steer Gemini into emitting JSON text.

## What This Changes

The issue asks OpenClaw to stop Gemini-backed Telegram agents from emitting inline-button JSON as visible text and instead use the supported interactive button tool payload.

## Best Possible Solution

Align the shared agent prompt and prompt regression tests with `message(action=send)` plus `presentation.blocks[].buttons`, and avoid adding raw final-text JSON parsing as the primary fix.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: current main still advertises raw `buttons=[[...]]` while the message tool, docs, and Telegram runtime use `presentation.blocks[].buttons`. I did not run a live Gemini plus Telegram scenario in this read-only review.

## Solution Assessment

Yes for a narrow prompt/schema alignment repair; no for parsing raw final-text `[[ ... ]]` JSON, because that would add a second source of truth outside the shared presentation contract.
