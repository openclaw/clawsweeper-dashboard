## Summary

Keep open: current main and the latest release still tell inline-button-capable agents to use raw `buttons=[[...]]` even though the message tool contract exposes `presentation`, so Gemini can still produce the reported raw JSON instead of interactive Telegram buttons.

## What This Changes

The issue asks OpenClaw to make Gemini-backed Telegram agents use the message-tool presentation button contract instead of writing inline button JSON in final text.

## Best Possible Solution

Align the shared agent prompt and prompt tests with `message(action=send)` using `presentation.blocks[].buttons`, and avoid adding final-text `[[...]]` JSON parsing as the primary delivery contract.

## Reproduction Assessment

Yes, at source level: current main still prompts inline-button-capable agents to use raw `buttons=[[...]]`, while the Telegram message-tool schema exposes `presentation` and omits top-level `buttons`. I did not establish a live Gemini/Telegram failing run in this read-only review.

## Solution Assessment

Yes. A narrow prompt and regression-test update is the maintainable path; parsing raw button JSON out of final assistant text would create a second delivery contract outside the shared presentation renderer.
