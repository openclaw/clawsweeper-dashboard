## Summary

Keep open: current main still has a Telegram inline-button contract mismatch that plausibly explains the Gemini raw-JSON output, and the remaining fix is narrow enough to attempt without closing this issue.

## What This Changes

The issue reports that Gemini-generated Telegram inline button intents appear as raw JSON text instead of interactive inline keyboards.

## Best Possible Solution

Align the model prompt and Telegram docs with the existing `presentation` button contract, then add regression coverage showing inline-button guidance matches the tool schema and Telegram renderer.

## Reproduction Assessment

No, not with a live current-main Gemini run in this review. Source inspection does show a reproducible contract mismatch: the prompt/docs advertise raw/top-level Telegram buttons while the tool schema and renderer expect structured `presentation` or `interactive` payloads.

## Solution Assessment

Yes for a narrow repair: update guidance and docs to use the existing `presentation` payload and test that contract. Parsing arbitrary raw JSON button text would be a broader directive-surface change and should not be the default fix.

Fix https://github.com/openclaw/openclaw/issues/41495. Current main tells agents to use Telegram `buttons=[[{text,callback_data,style?}]]` and the Telegram docs show a top-level `buttons` action example, but the message tool schema exposes `presentation` and Telegram renders buttons from `presentation`/`interactive`. Update the system prompt inline-button guidance to show structured `message(action=send, presentation={blocks:[{type:"buttons", buttons:[{label,value,style?}]}]})` usage, update tests/snapshots that assert the old `buttons=[[...]]` hint, and correct Telegram docs to prefer `presentation` for inline keyboards. Do not add a raw text regex parser unless maintainers explicitly choose a new Telegram directive surface. Add a CHANGELOG.md fix entry because this affects user-visible Telegram/Gemini behavior.
