## Summary

Keep open: current main still teaches Telegram agents to send inline buttons using raw `buttons=[[...]]` text while the message tool and Telegram runtime render buttons from structured `presentation` or `interactive` payloads, so the reported Gemini raw-JSON symptom remains source-backed.

## What This Changes

The issue requests that Gemini-generated Telegram inline buttons render as interactive buttons instead of raw JSON text.

## Best Possible Solution

Correct the Telegram inline-button prompt and docs to use the existing `presentation.blocks[].buttons` message-tool payload, then add focused regression coverage; raw-text JSON parsing should remain a separate product choice if maintainers want it.

## Reproduction Assessment

Yes, source-reproducible: current main emits raw `buttons=[[...]]` guidance while the message tool schema exposes `presentation` and the Telegram runtime reads `presentation` or `interactive` only. I did not run a live Gemini plus Telegram session.

## Solution Assessment

Yes: the narrow maintainable fix is to align prompt, docs, and tests with the existing structured presentation contract. Parsing arbitrary raw button JSON out of message text would add a new directive surface and should not be the default repair.
