## Summary

Keep open: current main still has a source-backed mismatch between Telegram inline-button guidance and the advertised message tool schema, so the Gemini raw-text behavior is plausibly still unresolved.

## What This Changes

The issue asks for Gemini Telegram replies with inline button choices to render as Telegram inline keyboards instead of leaking JSON-like button text.

## Best Possible Solution

Align the model-facing Telegram button guidance and docs with the canonical `presentation.blocks[{type:"buttons"}]` contract, while keeping Telegram-specific rendering inside the plugin.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced: current prompt text tells models to use a `buttons` argument, while the current message tool schema exposes `presentation` and tests assert `buttons` is not exposed. That mismatch is a high-confidence path for Gemini to emit the shown JSON-like text instead of a structured tool argument.

## Solution Assessment

No to raw text post-processing as the first fix; the narrowest maintainable repair is to make the prompt, docs, and regression tests point models at the existing presentation contract that Telegram already renders as inline keyboards.
