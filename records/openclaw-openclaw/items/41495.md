## Summary

Keep open: current main still has a deterministic prompt/schema mismatch. The shared system prompt tells inline-button-capable agents to use raw `buttons=[[...]]`, while the message tool, Telegram runtime, tests, and public presentation docs all route rich buttons through `presentation.blocks[].buttons`.

## What This Changes

The issue asks OpenClaw to prevent Gemini-driven Telegram replies from emitting inline buttons as raw JSON text and instead use the existing message-tool button format.

## Best Possible Solution

Align the shared agent prompt and regression tests with `presentation.blocks[].buttons`, keeping raw `[[...]]` Telegram JSON parsing out of the primary path.

## Reproduction Assessment

Yes, at source level. Current main deterministically advertises a raw `buttons=[[...]]` prompt shape while the tool schema, tests, docs, and Telegram runtime expect `presentation` or `interactive` payloads.

## Solution Assessment

Yes for a narrow prompt/schema alignment repair; parsing raw `[[...]]` text as the primary fix would bypass the shared presentation contract and add a second source of truth.
