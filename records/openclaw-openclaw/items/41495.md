## Summary

Current main still has a source-backed contract mismatch: the agent prompt tells inline-button-capable runs to use a top-level `buttons=[[...]]` message-tool argument, while the message tool and Telegram send path render buttons from `presentation` or `interactive`. This should stay open as a narrow bug rather than a stale cleanup close.

## What This Changes

The issue asks OpenClaw to stop Gemini-driven Telegram replies from showing inline-button JSON as visible text and instead render real inline keyboard buttons.

## Best Possible Solution

Update the agent prompt and regression tests to steer inline-button-capable sends toward `presentation.blocks[].buttons` and avoid adding raw `[[ ... ]]` text parsing as the primary runtime path.

## Reproduction Assessment

Yes. Source inspection on current main shows a deterministic mismatch: the prompt advertises `buttons=[[...]]`, while the message tool and Telegram renderer expect `presentation` or `interactive` payloads.

## Solution Assessment

Yes for prompt/test alignment, no for parsing raw `[[ ... ]]` text as the primary fix. The narrow maintainable repair is to align model guidance with the already-shipped shared presentation contract.
