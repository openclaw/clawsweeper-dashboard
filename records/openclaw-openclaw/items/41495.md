## Summary

This should stay open: current main and the latest shipped release still advertise a `buttons=[[...]]` message-tool shape for inline-button-capable runs, while the shared message tool and Telegram runtime use the `presentation`/`interactive` contract instead.

## What This Changes

The issue requests that Gemini-generated Telegram inline buttons render as interactive buttons instead of raw `[[...]]` JSON text.

## Best Possible Solution

Align the shared agent prompt and regression tests with the existing `presentation.blocks[].buttons` contract, and avoid adding raw Telegram JSON parsing as the primary path.

## Reproduction Assessment

Yes. Source inspection on current main shows a deterministic mismatch: the prompt advertises `buttons=[[...]]`, while the message tool schema, tests, docs, and Telegram runtime expect `presentation` or `interactive` payloads.

## Solution Assessment

No for parsing raw `[[ ... ]]` text as the primary fix. The narrow maintainable solution is to update prompt guidance and tests to use the already-shipped shared `presentation` contract.
