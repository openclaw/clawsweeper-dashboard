## Summary

This should stay open: current main still has a source-visible mismatch where the agent prompt tells inline-button-capable Telegram runs to use a `buttons=[[...]]` message-tool argument, while the supported shared tool contract is `presentation`, and Telegram renders buttons from `presentation`/`interactive` rather than top-level `buttons`.

## What This Changes

The issue asks OpenClaw to stop Gemini-driven Telegram inline button replies from leaking raw JSON text and render them as interactive inline buttons.

## Best Possible Solution

Align the agent prompt and tests with the existing `presentation` button contract so Gemini is steered to structured message-tool calls instead of raw Telegram JSON text.

## Reproduction Assessment

Yes, from source inspection rather than a live Telegram run: current main prompts inline-button-capable agents to use `buttons=[[...]]`, while the tool schema and Telegram runtime expect `presentation`/`interactive` payloads. That explains the reporter's raw JSON symptom without needing a new feature path.

## Solution Assessment

No, parsing raw `[[ ... ]]` model text would be the wrong primary fix because the supported contract is `presentation`. The safer fix is to update prompt guidance and regression tests so Gemini uses the existing message-tool `presentation.blocks[].buttons` format.
