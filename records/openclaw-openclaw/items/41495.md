## Summary

Keep open: current main and the latest release still advertise raw `buttons=[[...]]` inline-button syntax to agents, while the current Telegram message-tool schema exposes `presentation.blocks[].buttons` and explicitly omits a top-level `buttons` property. No canonical merged or closing PR fixes this; the open group-prompt PR is useful adjacent work but only changes group reply guidance.

## What This Changes

The issue reports that Gemini-backed Telegram agents emit inline-button JSON as message text instead of using the message tool’s button presentation contract.

## Best Possible Solution

Align the shared agent prompt and prompt tests with the current `message(action=send)` presentation contract, and avoid adding final-text `[[...]]` JSON parsing as the primary fix.

## Reproduction Assessment

Yes. Current-main source still tells inline-button-capable agents to use raw `buttons=[[...]]`, while the Telegram message-tool schema exposes `presentation` and the schema test asserts there is no top-level `buttons` property; I did not run a live Gemini/Telegram scenario in this read-only review.

## Solution Assessment

Yes. A narrow prompt/schema alignment is the maintainable fix; parsing raw `[[...]]` JSON out of final text would create a second delivery contract outside the shared presentation path.
