## Summary

Keep open: current main and the latest release still advertise the stale raw `buttons=[[...]]` prompt hint while the Telegram message-tool surface now exposes structured `presentation`, so the reported Gemini/Telegram button breakage remains source-reproducible.

## What This Changes

The issue reports that Gemini-backed Telegram agents emit inline-button JSON as plain text instead of sending buttons through OpenClaw's structured message tool.

## Best Possible Solution

Align the shared agent prompt, docs if needed, and regression tests with `message(action=send)` using `presentation.blocks[].buttons`, and avoid making final-text `[[...]]` parsing the primary button delivery contract.

## Reproduction Assessment

Yes, at source level: current main still prompts inline-button-capable agents to use raw `buttons=[[...]]`, while the Telegram message-tool schema exposes `presentation` and omits top-level `buttons`. I did not establish a live Gemini/Telegram failing run in this read-only review.

## Solution Assessment

Yes. A narrow prompt and regression-test update is the maintainable fix; parsing raw button JSON out of final assistant text would create a second delivery contract outside the shared presentation renderer.
