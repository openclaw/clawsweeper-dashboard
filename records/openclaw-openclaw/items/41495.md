## Summary

Keep open: current main still tells inline-button-capable agents to use raw `buttons=[[...]]`, while the current Telegram message-tool path exposes `presentation.blocks[].buttons` and does not expose a top-level `buttons` schema field. The adjacent open group-prompt PR is useful context, but it does not fix the shared generic inline-button prompt that matches this report.

## What This Changes

The issue asks OpenClaw to stop Gemini-backed Telegram agents from emitting inline-button JSON as visible text and instead send interactive buttons through the message tool.

## Best Possible Solution

Align the shared agent prompt and prompt tests with `message(action=send)` using `presentation.blocks` buttons, and avoid adding a raw final-text `[[...]]` JSON parser as the primary fix.

## Reproduction Assessment

Yes for a source-level reproduction: current main still tells inline-button-capable agents to use raw `buttons=[[...]]`, while the Telegram message-tool schema exposes `presentation` and asserts no top-level `buttons` property. I did not run a live Gemini plus Telegram scenario in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is prompt and regression-test alignment with the existing presentation contract; parsing raw `[[...]]` JSON from final text would create a second delivery contract.
