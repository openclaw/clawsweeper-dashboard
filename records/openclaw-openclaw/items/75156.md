## Summary

Current main already covers the reported Telegram mixed text plus MEDIA attachment-loss path: block streaming no longer consumes MEDIA directives, final payload assembly preserves unsent media after streamed text, and the same behavior is in the v2026.6.6 release.

## What This Changes

The issue reports Telegram direct-chat replies dropping the attachment on the first mixed text plus MEDIA assistant response while delivering the text.

## Best Possible Solution

Keep the shipped reply finalization behavior; open a fresh report only with v2026.6.6 or newer Telegram logs if a specific mixed reply still drops an attachment.

## Reproduction Assessment

No high-confidence current-main failure reproduction remains from this review. The old code path is clear from the report and source, but current source and tests show the mixed text/media loss path now preserves the unsent media.

## Solution Assessment

Yes. The current implementation solves this in the narrow reply-finalization path by leaving MEDIA directives for final payload handling and preserving only the unsent media side after streamed text.
