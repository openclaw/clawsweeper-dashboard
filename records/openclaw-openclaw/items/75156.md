## Summary

Keep this open. Current main still has the reported block-reply media accounting path, and open PR #75180 is the active closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that a Telegram direct-chat assistant reply containing visible text plus a MEDIA attachment can deliver only the text on first send because block-streamed media is deduped from the final payload.

## Best Possible Solution

Land PR #75180 or an equivalent narrow fix that makes block reply sent-media tracking delivery-aware while preserving duplicate suppression for media that actually shipped.

## Reproduction Assessment

Yes. The reporter provided live Telegram DM steps, and current main has a high-confidence source path: parse mixed text-plus-media into a block payload, record its media URL as sent, then dedupe that media from the final payload while keeping text.

## Solution Assessment

Yes. The narrow maintainable fix belongs in generic block-reply sent-media accounting rather than Telegram delivery, because Telegram delivery already supports captioned media payloads.
