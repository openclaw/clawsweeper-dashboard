## Summary

Keep this open. Current main still has the reported block-stream/final-dedupe path, and open PR #75180 is the active implementation candidate with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Fix Telegram direct-chat replies so a first assistant response containing visible text plus a MEDIA attachment delivers both the text/caption and the attachment.

## Best Possible Solution

Land PR #75180 or an equivalent narrow fix that makes block reply sent-media tracking delivery-aware, preserves media-only duplicate suppression, and adds regression coverage for mixed text-plus-media block replies and final payload dedupe.

## Reproduction Assessment

Yes. The reporter provided live Telegram DM steps and local validation, and current source gives a high-confidence unit path: enqueue a mixed text-plus-media block reply, observe it enter sentMediaUrls, then build final payloads with block streaming enabled and see dedupe strip the media while leaving text.

## Solution Assessment

Yes. The best fix belongs in the generic block reply sent-media accounting boundary rather than Telegram transport, because Telegram delivery already supports captioned media and the strip happens before final Telegram send.
