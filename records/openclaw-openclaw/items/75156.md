## Summary

Keep this open: current main still contains the mixed block-media accounting path that can strip the attachment from the final payload, and PR #75180 is the open closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that Telegram can deliver text but drop the attachment from the first assistant reply that combines visible text with a MEDIA attachment.

## Best Possible Solution

Adopt delivery-confirmed media accounting in the generic block reply pipeline so final dedupe suppresses only attachments known to have reached the channel.

## Reproduction Assessment

Yes. The reporter provided live Telegram DM steps, and the static path is reproducible in current main by enqueueing a mixed text+media block, observing it enter `sentMediaUrls`, and then seeing final payload media dedupe remove that attachment.

## Solution Assessment

Yes. A generic block-reply accounting fix is the narrow maintainable solution because Telegram delivery already supports captioned media; the transport layer should not paper over premature final-payload dedupe.
