## Summary

Keep this open. Current main still contains the reported block-stream/final-media-dedupe path, and open PR #75180 is the active closing implementation candidate, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests a fix for Telegram direct chats where a first assistant reply containing both visible text and a `MEDIA:` attachment can deliver only the text because block-stream media tracking causes final payload dedupe to strip the attachment.

## Best Possible Solution

Review and land PR #75180, or an equivalent narrow replacement, so block reply media is only treated as sent when delivery semantics justify final-payload dedupe, while preserving duplicate suppression for truly delivered media-only block replies.

## Reproduction Assessment

Yes. The reporter provided live Telegram DM steps, and current main gives a source-level reproduction path: enqueue a mixed text-plus-media block reply, record its media in sentMediaUrls, then let final payload media dedupe remove the attachment while preserving text.

## Solution Assessment

Yes for the direction, but the implementation should land through the open PR or an equivalent reviewed fix. The maintainable boundary is the generic block reply sent-media accounting path, not Telegram transport, because Telegram delivery can send captioned media downstream.
