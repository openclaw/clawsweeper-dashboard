## Summary

Keep this open. Current main still records every media URL from a successful block reply as sent and later uses that set to strip final payload media, so the reported text-without-attachment path remains plausible; open PR #75180 is the active implementation candidate and should resolve or replace this issue only after merge.

## What This Changes

Fix Telegram direct-chat replies so a first assistant response containing visible text plus a `MEDIA:` attachment delivers both the text and the attachment on the first send.

## Best Possible Solution

Land PR #75180 or an equivalent narrow fix that makes block reply sent-media accounting reflect media actually delivered by the block callback, while adding regression coverage for mixed text+media first delivery and media-only duplicate suppression.

## Reproduction Assessment

Yes. The reporter provided live Telegram direct-message steps and current source gives a high-confidence unit path: enqueue a mixed text+media block reply, let the block callback return, observe the media URL enter `getSentMediaUrls()`, then build final payloads and see matching media stripped while text remains.

## Solution Assessment

Yes. The fix belongs at the generic block reply sent-media accounting boundary; Telegram's downstream delivery already supports captioned media, so changing Telegram transport alone would treat a symptom and leave the premature dedupe state intact.
