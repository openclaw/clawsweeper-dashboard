## Summary

Keep this open. Current main still has the reported block-stream/final-dedupe path: block reply delivery records every media URL as sent after the callback returns, and final reply construction later strips those URLs from media-bearing payloads. The linked open PR #75180 is the active implementation candidate and should land or be replaced before this issue closes.

## What This Changes

Fix Telegram direct chats so the first assistant reply containing visible text plus a MEDIA attachment delivers both instead of stripping the attachment during block-stream/final dedupe.

## Best Possible Solution

Review and land PR #75180, or replace it with an equivalent narrow fix that makes block reply sent-media accounting delivery-aware and adds regression coverage for mixed text-plus-media replies plus media-only duplicate suppression.

## Reproduction Assessment

Yes. The reporter supplied live Telegram DM steps and local validation, and current source gives a high-confidence unit path: enqueue a mixed text/media block, let the pipeline mark its media URL sent, then build final payloads with block streaming enabled and observe media dedupe strip the attachment.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the generic block reply sent-media accounting boundary with regression tests; Telegram captioned media delivery already exists, so changing only Telegram transport would address a downstream symptom.
