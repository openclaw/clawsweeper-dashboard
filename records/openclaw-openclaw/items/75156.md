## Summary

Keep this open. Current main still records block-reply media URLs before the final payload dedupe pass, and the linked open PR is the active closing implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue asks for Telegram direct-chat replies containing visible text plus a `MEDIA:` attachment to deliver both text and attachment on the first send instead of dropping the attachment after block-streaming dedupe.

## Best Possible Solution

Land PR #75180 or an equivalent narrow fix that makes block-reply sent-media tracking delivery-aware, preserves media-only duplicate suppression, and adds regression coverage for mixed text-plus-media block replies before final dedupe.

## Reproduction Assessment

Yes. The reporter provided Telegram DM steps and live local validation, and current source gives a focused unit path: enqueue a mixed text-plus-media block reply, observe its media enter `sentMediaUrls`, then let final payload media dedupe strip that URL while preserving text.

## Solution Assessment

Yes. The fix belongs in the generic block-reply sent-media accounting boundary rather than Telegram transport, because Telegram delivery supports captioned media and the strip happens before the Telegram send path.
