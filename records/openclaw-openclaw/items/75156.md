## Summary

Keep this open. Current main still has the reported block-stream/final-dedupe path: mixed text-plus-media block replies are recorded as sent media, and final reply construction can then strip the attachment while leaving text; open PR #75180 is the active closing implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that Telegram direct-chat replies containing visible text plus a MEDIA attachment deliver both parts on the first assistant reply instead of dropping the attachment during block-stream/final-reply dedupe.

## Best Possible Solution

Land PR #75180 or an equivalent narrow fix that makes block-reply sent-media accounting delivery-aware, then add regression coverage for mixed text-plus-media replies and for media-only duplicate suppression.

## Reproduction Assessment

Yes. The reporter supplied live Telegram DM steps and local validation, and current source gives a high-confidence reproduction path: enqueue a mixed text-plus-media block reply, observe its media URL enter getSentMediaUrls(), then build final payloads with block streaming enabled and see that media eligible for stripping while text remains.

## Solution Assessment

Yes at the fix direction level. The narrow maintainable solution is in the generic block reply sent-media accounting boundary with tests; changing Telegram transport would address a downstream symptom even though Telegram delivery already supports captioned media.
