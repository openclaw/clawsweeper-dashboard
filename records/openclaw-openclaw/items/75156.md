## Summary

Keep this open. Current main still records media from mixed block replies as already sent before final-payload dedupe, and open PR #75180 is the active closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that a Telegram direct-chat assistant reply containing visible text plus a MEDIA attachment can deliver the text while dropping the attachment on the first send.

## Best Possible Solution

Review and land PR #75180, or an equivalent narrow fix that makes block-stream media dedupe depend on media actually delivered while preserving duplicate suppression for media-only streamed payloads.

## Reproduction Assessment

Yes. The reporter provided live Telegram DM steps, and current main has a source-level reproduction path: mixed text-plus-media block payloads can be recorded as sent media, then final dedupe can strip the attachment while leaving text.

## Solution Assessment

Yes. The narrow maintainable fix belongs in generic block-reply media accounting rather than Telegram transport, because Telegram delivery already supports captioned media payloads.
