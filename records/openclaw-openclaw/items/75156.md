## Summary

Keep this open. Current main still has the reported block-stream/final-dedupe path: mixed text-plus-media block replies are recorded as sent media, and final reply construction can then strip that media while leaving the text. PR #75180 is the active open implementation candidate with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Fix Telegram direct replies so the first assistant response containing visible text plus a `MEDIA:` attachment delivers both parts instead of dropping the attachment during block-stream finalization.

## Best Possible Solution

Review and land PR #75180, or replace it with an equivalent narrow fix that makes block-stream sent-media accounting delivery-aware and adds regression coverage for mixed text-plus-media replies plus media-only duplicate suppression.

## Reproduction Assessment

Yes. The reporter supplied live Telegram direct-message steps and local validation, and current source gives a high-confidence unit path: enqueue a mixed text-plus-media block reply, observe the media URL enter `getSentMediaUrls()`, then build final payloads with block streaming enabled and see the same media eligible for stripping while text remains.

## Solution Assessment

Yes. The maintainable fix belongs in the generic block reply sent-media accounting boundary with regression tests; Telegram delivery already supports captioned media, so changing Telegram transport would treat a downstream symptom rather than the source of the strip.
