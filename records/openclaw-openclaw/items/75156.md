## Summary

Keep this open. Current main still records media from successful block replies as already sent before the final payload dedupe pass, and open PR #75180 is the active closing implementation candidate, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Ensure Telegram direct-chat replies containing visible text plus a `MEDIA:` attachment deliver both the text/caption and attachment on the first send when block streaming and final dedupe are enabled.

## Best Possible Solution

Land PR #75180 or an equivalent narrow fix that makes block reply sent-media tracking delivery-aware, preserves media-only duplicate suppression, and adds regression coverage for mixed text-plus-media block replies before final payload dedupe.

## Reproduction Assessment

Yes. The reporter provided live Telegram DM steps and local validation, and current source gives a high-confidence unit path: enqueue a mixed text-plus-media block reply, observe it enter `sentMediaUrls`, then build final payloads with block streaming enabled and see dedupe strip the media while leaving text.

## Solution Assessment

Yes. The fix belongs in the generic block-reply sent-media accounting boundary rather than Telegram transport, because Telegram can send captioned media and the strip happens before final Telegram delivery.
