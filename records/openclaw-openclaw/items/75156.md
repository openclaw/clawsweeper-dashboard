## Summary

Keep this open. Current main still marks media from mixed text-plus-media block replies as sent before final reply construction, and the later dedupe pass can strip the attachment while preserving text. Open PR #75180 references this issue with closing syntax and should be reviewed or replaced before this issue closes.

## What This Changes

Telegram DM replies that combine visible text with a MEDIA attachment should deliver both the text and attachment on the first assistant send.

## Best Possible Solution

Review PR #75180 as the active implementation candidate and land its delivery-aware sent-media accounting with regression coverage, or close that PR in favor of an equivalent narrow fix that preserves media-only duplicate suppression while keeping mixed text-plus-media final attachments deliverable.

## Reproduction Assessment

Yes. The reporter supplied live Telegram DM steps and local validation, and the current source gives a high-confidence unit path: enqueue a mixed text-plus-media block payload, observe its media in getSentMediaUrls(), then build final payloads with block streaming enabled and see matching media stripped while text remains.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the block reply sent-media accounting boundary, not in Telegram delivery, because Telegram already has captioned-media support and the observed stripping happens in generic final payload dedupe.
