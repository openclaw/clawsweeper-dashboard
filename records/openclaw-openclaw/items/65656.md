## Summary

Keep this open. Current main still routes LINE text-plus-table replies through a text Reply API call followed by a separate Push API Flex message, while the only directly linked fix PR was closed unmerged after review showed it did not change the reachable behavior.

## What This Changes

The issue requests that LINE text plus generated table Flex messages be delivered in the same Reply API interaction instead of pushing the table separately.

## Best Possible Solution

Fix the LINE plugin delivery path so eligible generated Flex/media messages fill unused Reply API slots before the reply token is consumed, then push only overflow while preserving quick-reply behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call deliverLineAutoReply with text that processLineMessage splits into cleaned text plus a table Flex message; current main then replies with text and pushes the Flex table separately.

## Solution Assessment

Yes. The maintainable fix is to batch extras before sendLineReplyChunks consumes the token; the closed PR's allowReplyToken argument change was not sufficient because the token was already marked used.

Fix https://github.com/openclaw/openclaw/issues/65656 in the LINE plugin only. Implement batching so non-quick-reply generated Flex/media extras can fill unused LINE Reply API slots in the same reply call before the reply token is marked used, and push only overflow beyond the five-message Reply API limit. Preserve existing quick-reply ordering/visibility semantics. Do not move behavior into core or add a config option.
