## Summary

Keep open. Current main still splits LINE text-plus-table responses into a Reply API text call followed by a Push API Flex message, and the only linked fix PR was closed unmerged after review showed it was a no-op.

## What This Changes

The issue asks OpenClaw to deliver LINE responses containing explanatory text plus a Markdown table without losing the generated Flex table when Push API quota is exhausted.

## Best Possible Solution

Update the LINE plugin delivery path so eligible Flex/media messages fill unused Reply API slots before the token is consumed, then push only overflow while preserving quick-reply behavior and tests.

## Reproduction Assessment

Yes, by source inspection: call deliverLineAutoReply with text that processLineMessage splits into cleaned text plus a table Flex message, and current main replies with text before pushing the Flex table separately. I did not run a live exhausted-quota LINE account in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to batch eligible rich/media extras into sendLineReplyChunks before it consumes the reply token; the closed allowReplyToken-only PR was not sufficient because the token was already marked used.

Fix https://github.com/openclaw/openclaw/issues/65656 in the LINE plugin. Current main converts Markdown tables into Flex messages, sends text chunks through sendLineReplyChunks, then forces rich/media messages through Push API with allowReplyToken=false; this drops tables when Push quota is exhausted. Implement the fix in the LINE delivery path so eligible generated Flex/media messages are bundled into unused Reply API slots before the reply token is consumed, and only overflow is pushed. Preserve quick-reply behavior and text/rich ordering semantics. Add or update regression tests covering text plus table/Flex using one Reply API call when capacity allows, overflow push behavior beyond five messages, and quick-reply cases.
