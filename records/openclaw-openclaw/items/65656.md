## What This Changes

The issue requests that LINE text-plus-table replies batch the table Flex message into the same Reply API interaction instead of dropping it on exhausted Push quota.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/81299 or an equivalent focused batching fix, then close this issue after the fix is on main.

## Reproduction Assessment

Yes, at source level: current main extracts Markdown tables into Flex messages, replies with text first, then sends Flex content through the quota-bound Push path. I did not establish a live exhausted-quota LINE account reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to batch eligible rich/media messages into the same Reply API call before the reply token is consumed, then push only overflow; the open focused PR owns that direction.
