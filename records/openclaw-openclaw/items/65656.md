## Label Justifications

- `P1`: The report describes a broken LINE channel workflow where structured table content is dropped for affected real users once Push quota is exhausted.
- `impact:message-loss`: The affected table Flex message is not delivered to the user even though the text portion arrives.

## Summary

Keep open. Current main and the latest release still route the LINE text-plus-table path through Reply API for text and Push API for the table Flex message, while https://github.com/openclaw/openclaw/pull/81299 is an open closing PR for the same bug.

## What This Changes

The issue reports LINE auto-replies losing Markdown-derived table Flex messages when text and table are returned together after the Push API quota is exhausted.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/81299; the desired end state is one Reply API batch for eligible rich/media/text up to five messages, with only overflow sent through Push API.

## Reproduction Assessment

Yes, source-reproducible: current main extracts Markdown tables into Flex messages, sends text chunks through the Reply API first, then forces rich/media through Push API. I did not run a live exhausted-quota LINE account.

## Solution Assessment

Yes. The narrow maintainable solution is to batch eligible rich/media/text before the reply token is consumed and push only overflow, which is exactly the active direction in the open linked PR.
