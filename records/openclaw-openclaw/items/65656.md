## Summary

Keep open. Current main and latest release still route the extracted LINE table Flex message through Push after text consumes the Reply API token, while the linked closing PR is open and not shipped.

## What This Changes

The issue reports that LINE replies containing text plus a Markdown table can deliver only the text because the table Flex message is sent through Push API and fails with 429 after Push quota is exhausted.

## Best Possible Solution

Review and land or explicitly reject the focused batching PR at https://github.com/openclaw/openclaw/pull/81299, then close this issue only after the fix is on main.

## Reproduction Assessment

Yes, at source level. Current main extracts Markdown tables into Flex messages, replies with text first, then sends the Flex messages through the Push API path; a live exhausted-quota LINE account would be needed for full end-to-end reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to batch eligible rich/media messages into the same Reply API call before the reply token is consumed and push only overflow; the open linked PR appears to own that direction.
