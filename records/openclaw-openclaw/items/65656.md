## Summary

Keep open. Current main still turns Markdown tables into LINE Flex messages, sends the text through the Reply API first, then forces the table Flex message through the Push API; the linked fix PR was closed unmerged after review found it did not change that reachable path.

## What This Changes

The issue asks for LINE text plus Markdown-table Flex messages to be delivered within the same Reply API interaction instead of replying with text and then pushing the table.

## Best Possible Solution

Batch eligible rich/media messages into unused Reply API slots before consuming the reply token, then push only overflow while preserving quick-reply ordering semantics.

## Reproduction Assessment

Yes. By source inspection on current main, a reply payload whose text becomes one text chunk plus one table Flex message replies with text first and then pushes the Flex message separately; I did not run a live exhausted-quota LINE account.

## Solution Assessment

Yes. The maintainable fix is to extend the reply-chunk path to accept eligible extra message objects before it consumes the token; the closed allowReplyToken-only PR is not sufficient because the token is already marked used by then.
