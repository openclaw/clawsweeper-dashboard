## Label Justifications

- `P1`: The report describes a broken LINE channel reply workflow where structured table content is silently lost for real users on exhausted Push quota.
- `impact:message-loss`: The affected table Flex message is dropped while only the text portion reaches the user.

## Summary

Keep open. Current main and the latest release still split LINE text-plus-table replies into Reply API text followed by Push API Flex/table delivery, and the open closing PR at https://github.com/openclaw/openclaw/pull/81299 owns the same bug.

## What This Changes

The issue asks for LINE auto-replies that contain text plus Markdown-table Flex messages to use remaining Reply API slots before falling back to Push API overflow.

## Best Possible Solution

Land a narrow LINE delivery fix that fills Reply API slots for mixed rich/text replies before using Push API overflow, while preserving quick-reply ordering and covering the five-message limit.

## Reproduction Assessment

Yes. Source inspection shows Markdown tables become Flex messages, and current LINE auto-reply delivery replies with text before pushing rich/table messages, matching the reporter's 429 loss path; I did not run a live exhausted-quota LINE account.

## Solution Assessment

Yes. Batching eligible rich/media plus text into the Reply API up to LINE's five-message limit is the narrowest maintainable fix, with Push reserved for overflow and quick-reply ordering handled separately.
