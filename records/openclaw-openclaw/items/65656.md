- `P1`: This is a real LINE channel workflow where structured table content can be silently lost for users after Push quota exhaustion.
- `impact:message-loss`: The affected table Flex message is dropped while only the text portion reaches the LINE user.

## Summary

Keep open. Current main and the latest release still route LINE text-plus-table replies as Reply API text followed by Push API Flex/table delivery, and the open closing PR at https://github.com/openclaw/openclaw/pull/81299 is the active repair path.

## What This Changes

Ensure LINE replies that include text plus Markdown tables/Flex messages use available Reply API slots before falling back to Push API overflow.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/81299, or an equivalent narrow LINE delivery fix, so eligible rich/media/table messages are batched into Reply API slots before Push overflow while preserving quick-reply behavior.

## Reproduction Assessment

Yes, source-level. Current main converts Markdown tables to Flex messages, sends text through the Reply API first, and then pushes the Flex/table messages, matching the reported 429 loss path when Push quota is exhausted.

## Solution Assessment

Yes. The narrow maintainable direction is to batch eligible rich/media plus text into the same Reply API opportunity before Push overflow; the open linked PR implements that direction and should be reviewed instead of starting a second fix lane.
