## Summary

Keep this open. Current main still has the reported LINE delivery split: Markdown tables become Flex messages, text chunks consume the reply token first, and generated Flex/media messages are then sent through the Push API path; the directly related PR #65673 is closed and unmerged.

## What This Changes

The issue requests that LINE Markdown table Flex messages be bundled with available Reply API slots instead of being forced through Push API after the text reply.

## Best Possible Solution

Fix this inside the LINE plugin delivery path by letting eligible generated Flex/media messages fill unused Reply API slots in the same reply batch as text, then push only overflow under focused tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available by calling deliverLineAutoReply with text that processLineMessage splits into cleaned text plus a table Flex message; live quota exhaustion is only needed to observe the external 429 symptom.

## Solution Assessment

Yes, the requested direction is the right maintainable fix, but the closed PR's simple allowReplyToken argument change is insufficient because sendLineReplyChunks has already marked the reply token used. The safer fix is to batch extras before the token is consumed and return how many were consumed so only overflow is pushed.
