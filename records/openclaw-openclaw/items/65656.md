## Summary

Keep open: current main and v2026.5.18 still split LINE text-plus-table delivery into Reply API text followed by Push API rich-message delivery, and an open closing PR already targets the same source-reproducible bug.

## What This Changes

The issue requests that LINE text plus Markdown-table Flex replies be delivered together through available Reply API slots instead of dropping the table when Push quota is exhausted.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/81299; the desired end state is one Reply API batch for eligible rich/media/text up to five messages, with only overflow sent through Push API.

## Reproduction Assessment

Yes, by source inspection: current main extracts Markdown tables into Flex messages, sends text chunks through the Reply API first, then forces rich/media through Push API. I did not run a live exhausted-quota LINE account in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to batch eligible rich/media/text before the reply token is consumed and push only overflow, rather than editing built dist files or toggling allowReplyToken after the token is already used.
