## Label Justifications

- `P1`: The issue describes a broken LINE channel reply workflow where table Flex content is silently lost for users whose Push API quota is exhausted.
- `impact:message-loss`: The table Flex message can be dropped while only the text portion reaches the LINE user.

## Summary

Keep open. Current main and the latest release still route mixed LINE text-plus-table replies as Reply API text followed by Push API Flex/table delivery, while the open closing PR at https://github.com/openclaw/openclaw/pull/81299 owns the focused fix.

## What This Changes

The issue asks for LINE text plus Markdown-table Flex messages to be delivered through the same Reply API interaction instead of silently losing the Flex/table when Push API quota is exhausted.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81299 so LINE fills Reply API slots for eligible rich/media plus text replies, preserves quick-reply behavior, and pushes only overflow.

## Reproduction Assessment

Yes, at source level. Markdown tables become Flex messages, and current LINE auto-reply delivery replies with text before forcing the table Flex through Push API, matching the reported exhausted-quota 429 loss path.

## Solution Assessment

Yes. The narrow maintainable fix is to batch eligible rich/media and text into the Reply API up to LINE's five-message limit, then push only overflow; the linked PR implements that direction but remains unmerged.
