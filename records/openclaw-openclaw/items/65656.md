## Summary

Keep open: current main still routes text-plus-table LINE replies through a text Reply API call followed by a separate Push API call for the table Flex Message, and the linked fix PR was closed unmerged after reviewers identified it as a no-op.

## What This Changes

The issue requests bundling LINE text and Markdown-table Flex messages into the same Reply API interaction when reply capacity remains, instead of pushing the table separately.

## Best Possible Solution

Batch eligible rich/media extras with the initial LINE Reply API text batch up to LINE's five-message limit, push only overflow, preserve existing quick-reply ordering rules, and add regression coverage for Markdown table extraction plus delivery.

## Reproduction Assessment

Yes, at source level: a payload whose text yields one text chunk and one processed Flex table reaches sendLineReplyChunks for the text, then sendLineMessages(richMessages, false) for the table. I did not run a live LINE quota exhaustion test in this read-only review.

## Solution Assessment

No, the closed linked PR was not the right fix because flipping allowReplyToken after sendLineReplyChunks consumes the token leaves the rich/table send on Push API. The maintainable fix is to include extra rich/media messages in the first reply batch before marking the token used, then push only overflow.
