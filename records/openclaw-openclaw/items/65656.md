## Summary

Keep this open. Current main still has the reported LINE delivery split: Markdown tables become Flex messages, text chunks consume the reply token first, and the generated table/card messages are then sent through the Push API path; the directly related fix PR #65673 is closed and unmerged, and its proposed false-to-true change was reviewed as a no-op.

## What This Changes

The issue requests that LINE table Flex messages generated from Markdown tables be bundled into the Reply API response with the explanatory text when reply slots are available, instead of being forced through the Push API.

## Best Possible Solution

Fix this inside the LINE plugin delivery path by letting generated table/code Flex messages and media messages fill unused Reply API slots in the same reply batch as text chunks, then push only overflow while keeping the quick-reply ordering contract covered by tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available by calling deliverLineAutoReply with text that processLineMessage splits into cleaned text plus a table Flex message; current tests already assert the resulting replyMessageLine-for-text plus pushMessagesLine-for-Flex split. Live quota exhaustion is only needed to observe the external 429 symptom from LINE.

## Solution Assessment

Yes, the requested direction is the right maintainable fix, but the closed PR's simple false-to-true change is not sufficient because sendLineReplyChunks has already marked the reply token used. The safer implementation is to batch eligible extra Flex/media messages before the reply token is consumed and return how many extras were consumed so only overflow is pushed.
