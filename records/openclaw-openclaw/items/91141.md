## Summary

Keep open: current main does not prove this session-takeover/message-loss path is fixed, and the report adds a concrete default-config concurrent-inbound-user-message case beyond the broader open takeover reports. The source still has a fence that accepts owned writes or transcript-only OpenClaw assistant appends, while ordinary user appends are not part of that benign whitelist.

## What This Changes

Prevent legitimate concurrent inbound user-message appends during embedded prompt-lock release from being treated as session takeovers and causing silent reply or transcript loss.

## Best Possible Solution

Keep this open and fix the session ownership boundary so legitimate in-process user turns are preserved and retried or serialized without weakening detection of real external transcript writes.

## Reproduction Assessment

No live high-confidence reproduction was run, but the source path is credible: a released prompt fence accepts only owned writes or assistant-only benign appends, while the orphan repair path still removes trailing user leaves. The missing proof is a live or focused regression that creates the exact concurrent inbound Matrix timing on current main.

## Solution Assessment

Unclear: publishing legitimate in-process user appends as owned writes is plausible, but maintainers should choose the permanent boundary between owned-write publication, non-destructive recovery/retry, and stricter per-session serialization. The solution must preserve the external takeover guard and add regression coverage for the concurrent user-turn case.
