## Summary

Current main and v2026.6.5 still exclude macOS from a Linux-only `openclaw doctor` startup-optimization hint, but this path does not provide memory telemetry to the agent. The issue should remain open because https://github.com/BingqingLyu/openclaw/pull/769 is an open closing implementation and maintainers still need to choose whether macOS should receive these low-power tuning recommendations, especially since treating every Apple Silicon Mac as a target is broader than the reported low-memory case.

## What This Changes

Extend the doctor startup-optimization target detection so appropriate macOS hosts can receive compile-cache and respawn tuning guidance.

## Best Possible Solution

Keep this scoped to doctor guidance: define an explicit macOS targeting rule—preferably demonstrated by a real low-memory Mac or VM—then update the helper and Darwin-focused tests without claiming or adding agent memory telemetry.

## Reproduction Assessment

Yes for the actual doctor behavior: passing `platform: "darwin"` to the current helper deterministically skips the note. No high-confidence path supports the stronger claim that this function supplies memory data to the agent, because its only production caller is the doctor flow.

## Solution Assessment

No, the proposed unconditional Darwin-plus-ARM64 rule is not yet proven to be the best fix because it broadens low-power guidance to every Apple Silicon Mac. The narrow maintainable solution is to decide and test the intended macOS target criteria within the existing doctor helper.
