## Summary

Current main still excludes macOS, but the reported path is an `openclaw doctor` startup-hint path, not agent memory telemetry. Keep this issue open while the linked implementation at https://github.com/BingqingLyu/openclaw/pull/769 is open; its broad ARM64 rule needs maintainer judgment because it would classify every Apple Silicon Mac as a low-power host.

Likely related people: Vincent Koc (high confidence) introduced and recently maintained this doctor hint surface; rocky-d and shuicici (medium confidence) developed the earlier focused fix attempts.

## What This Changes

Extend startup optimization guidance to appropriate macOS hosts rather than limiting the doctor hint to Linux.

## Best Possible Solution

Define macOS eligibility explicitly in the existing doctor function—prefer genuinely low-memory darwin hosts, preserve Linux ARM behavior, and add tests proving low-memory macOS is included while high-memory Apple Silicon is excluded or receives platform-appropriate wording.

## Reproduction Assessment

Yes for the concrete doctor behavior: call `noteStartupOptimizationHints` with `platform: "darwin"` and low memory, and current main exits without emitting a note. No high-confidence reproduction supports the broader claim that this path withholds memory data from the agent, because its only caller is the doctor flow.

## Solution Assessment

No, the proposed unconditional darwin/ARM64 expansion is not the best solution because it produces misleading low-power guidance on unconstrained Macs. The narrow maintainable solution is an explicit macOS targeting rule plus focused low- and high-memory tests.
