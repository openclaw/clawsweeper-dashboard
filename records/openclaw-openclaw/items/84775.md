Keep open: the PR targets a valid diffs viewer bug, but it is not merge-ready because the served runtime asset remains unchanged, one failure path can leave a broken controller registered, CI lint is failing, and the supplied proof is only mocked jsdom output.

## What This Changes

The branch wraps diffs viewer card hydration in a per-card try/catch, exports test seams, adds a test auto-start guard, and adds jsdom coverage for continued hydration after one card throws.

## Best Possible Solution

Update the source and checked-in viewer-runtime asset together, register controllers only after full card success, fix lint, and verify with both focused jsdom coverage and a real generated-viewer/browser proof.

## Reproduction Assessment

Yes, source-level: current main has one unguarded hydration loop, so a first-card FileDiff construction, hydrate, or applyState throw rejects before later cards are processed. I did not run a browser repro in this read-only review.

## Solution Assessment

No. The per-card guard is the right direction, but the PR must update the shipped runtime asset and avoid keeping failed applyState controllers before it is the maintainable fix.
