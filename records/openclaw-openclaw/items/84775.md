## Summary

Keep open: the PR targets the right diffs viewer hydration loop, but as submitted it does not update the served viewer runtime asset, so real viewers would still run the old unguarded bundle. It also leaves a failed controller registered if state application throws, and the supplied proof is jsdom/mock-only rather than real viewer behavior.

## What This Changes

The PR wraps diffs viewer per-card hydration in a try/catch, exports the hydration loop/controller list for tests, disables auto-start under a test global, and adds jsdom coverage for continuing after one card fails.

## Best Possible Solution

Update the source and generated diffs viewer runtime together, ensure failed cards are not retained as controllers after any hydration/state failure, then land with focused regression coverage plus real browser proof.

## Reproduction Assessment

Yes. Current main has an unguarded per-card hydration loop in extensions/diffs/src/viewer-client.ts, and the linked report gives a concrete two-card failure path; I did not execute tests because this was a read-only review.

## Solution Assessment

No as submitted. A per-card catch is the right source-level direction, but the branch must also regenerate the shipped runtime asset and avoid retaining controllers when applyState() fails.
