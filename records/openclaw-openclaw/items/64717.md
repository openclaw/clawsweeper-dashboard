## Summary

Keep open. Current main still ignores configured Telegram text chunk limits inside `buildChunkedTextPlan`, so this PR targets a real existing config bug, but the branch is dirty against current main, lacks required real Telegram proof, and needs a rebase that preserves the newer rendered-HTML planner shape.

## What This Changes

The PR replaces hardcoded Telegram text chunk planning limits with `resolveTextChunkLimit` capped at 4096 and adds a send test for `channels.telegram.textChunkLimit = 4096`.

## Best Possible Solution

Rebase onto current main, resolve the Telegram text chunk limit once inside `buildChunkedTextPlan`, clamp it to 4096, use it consistently for HTML and plain fallback chunking while preserving rendered-HTML planning, and attach redacted real Telegram proof.

## Reproduction Assessment

Yes. By source inspection, configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text; current main still uses literal `4000` in `buildChunkedTextPlan`, so it splits below the configured limit.

## Solution Assessment

Yes in direction, but not as submitted against current main. The existing resolver is the right seam, but the branch must rebase, keep rendered HTML as planner input, pass explicit `cfg` in the test, and provide real Telegram proof.
