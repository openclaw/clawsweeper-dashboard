## Summary

Keep open. Current main still ignores the configured Telegram text chunk limit in the send planner, and this PR is a plausible narrow fix, but it is not merge-ready without real Telegram proof and a careful rebase onto the current chunking path.

## What This Changes

The PR changes Telegram send chunk planning to use the configured text chunk limit clamped at 4096 and adds send-path regression coverage for the 4096 case.

## Best Possible Solution

Keep the fix in the Telegram plugin: resolve `channels.telegram.textChunkLimit`, clamp to Telegram's 4096 cap, apply it consistently in the rendered-HTML/plain-text planner, preserve current default-markdown chunking, add focused regression coverage, and attach real Telegram proof.

## Reproduction Assessment

Yes, by source inspection. Configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text through `sendMessageTelegram`; current main's planner still uses literal `4000`, so it splits below the configured limit.

## Solution Assessment

Yes in direction, but not as submitted. The existing `resolveTextChunkLimit` seam is the narrow maintainable fix, but the branch needs a current-main rebase, explicit `cfg` in the new test, and real Telegram proof before merge.
