The PR changes Telegram send chunk planning to use the configured `textChunkLimit` capped at 4096 and adds a focused regression test for the 4096 case.

## Best Possible Solution

Keep the fix inside the Telegram plugin by resolving the effective text chunk limit, clamping it to 4096, applying it consistently to rendered HTML and plain fallback chunks, and preserving current markdown rendering plus final-chunk reply markup behavior.

## Reproduction Assessment

Yes. Source inspection is enough: configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text through `sendMessageTelegram`; current main still plans chunks with literal `4000`, so it will split below the configured limit.

## Solution Assessment

Yes in direction, but no as submitted. Reusing `resolveTextChunkLimit` with a 4096 cap is the narrow maintainable fix, but this branch needs a current-main rebase that preserves rendered HTML input, passes `cfg` in the test, and adds real Telegram proof.
