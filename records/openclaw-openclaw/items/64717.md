## Summary

Keep open: current main still ignores the documented Telegram `textChunkLimit` in the send chunk planner, but this PR needs a current-main rebase, focused test repair, and real Telegram proof before merge.

## What This Changes

The branch imports the shared text chunk-limit resolver, uses it to cap Telegram text chunks at 4096, and adds a send-path regression test for `textChunkLimit: 4096`.

## Best Possible Solution

Keep the fix in the Telegram send planner: resolve the effective configured limit through the SDK resolver, clamp it to Telegram’s 4096 maximum, apply it consistently to rendered HTML and plain fallback chunks, and cover both cap and lower-limit behavior.

## Reproduction Assessment

Yes. Source inspection is enough: configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text through `sendMessageTelegram`; current main still plans chunks with literal `4000`, so it splits below the configured limit.

## Solution Assessment

No as submitted, but yes in direction. Reusing `resolveTextChunkLimit` with a 4096 cap is the narrow maintainable fix, but this branch must be rebased to preserve rendered HTML input, pass `cfg` in the test, and add real Telegram proof.
