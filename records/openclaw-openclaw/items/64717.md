## Summary

Keep open. The PR addresses a real Telegram send-path bug that is still present on current main, and the production direction matches the existing config resolver seam, but the branch still has a concrete test type issue and is missing the required changelog entry for a user-facing fix.

## What This Changes

The PR changes Telegram send chunk planning to use the configured text chunk limit capped at 4096 and adds a regression for the 4096 configured-limit case.

## Best Possible Solution

Keep the fix in the Telegram plugin: resolve `channels.telegram.textChunkLimit` through the existing SDK chunk-limit helper, clamp the send effective limit to Telegram's 4096 maximum, use it consistently in `buildChunkedTextPlan`, add focused regression coverage with `cfg` passed explicitly, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction: configure `channels.telegram.textChunkLimit = 4096` and send an HTML-mode message whose rendered length is over 4000 but at or under 4096; the current send planner still chunks against 4000.

## Solution Assessment

No, not as submitted. The production approach is the narrow maintainable fix, but the PR should be amended to pass `cfg` in the new regression and add the required changelog entry before merge.
