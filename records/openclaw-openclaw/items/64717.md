## Summary

Keep open. Current main still hardcodes `4000` inside Telegram send chunk planning even though the config/docs/resolver already support `channels.telegram.textChunkLimit`; the PR targets a real bug, but it is conflicting, lacks real behavior proof, and needs a careful rebase to preserve current rendered-HTML markdown chunking.

## What This Changes

This PR changes Telegram send chunk planning from hardcoded `4000` limits to a resolved `textChunkLimit` capped at 4096 and adds a send-path regression test for the 4096 configured limit.

## Best Possible Solution

Keep the fix in the Telegram plugin: resolve `channels.telegram.textChunkLimit`, clamp it to Telegram's 4096 cap, apply it throughout `buildChunkedTextPlan`, preserve current rendered-HTML markdown chunking, and add focused regression coverage plus real Telegram proof.

## Solution Assessment

Yes in direction, but not as submitted. The existing `resolveTextChunkLimit` seam is the narrow maintainable fix, but the branch needs a current-main rebase, explicit `cfg` in the new test, and real Telegram proof before merge.
