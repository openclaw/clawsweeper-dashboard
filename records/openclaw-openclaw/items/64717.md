## Summary

Keep open: current main still has the configured-limit gap, but this external Telegram PR is not merge-ready because it conflicts with current main, would need to preserve the newer rendered-HTML planner shape on rebase, and lacks required real Telegram proof.

## What This Changes

This PR changes Telegram text send chunk planning to use the configured chunk limit capped at 4096 and adds focused send-path regression coverage for the 4096 case.

## Best Possible Solution

Keep the fix inside the Telegram plugin by resolving the effective text chunk limit in `buildChunkedTextPlan`, clamping it to Telegram's 4096 maximum, applying it consistently to HTML and plain fallback chunks, and preserving current markdown-to-HTML rendering behavior.

## Reproduction Assessment

Yes. Configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text through `sendMessageTelegram`; source inspection of current main shows the planner still uses literal `4000` and will split below the configured limit.

## Solution Assessment

Yes in direction, but not as submitted. Reusing `resolveTextChunkLimit` with a 4096 clamp is the narrow maintainable fix, but the branch needs a rebase that keeps rendered HTML as planner input, passes `cfg` in the test, and adds real Telegram proof.
