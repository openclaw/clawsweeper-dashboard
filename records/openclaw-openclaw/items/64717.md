## Summary

Keep open: current main still ignores the configured Telegram text chunk limit in the send chunk planner, and this PR follows the right production seam but needs small test and changelog repairs before merge.

## What This Changes

The PR replaces hardcoded 4000-character Telegram send chunk planning limits with the configured `textChunkLimit` capped at 4096 and adds a send-path regression test.

## Best Possible Solution

Keep the fix in the Telegram plugin: resolve the configured Telegram text chunk limit through the existing SDK helper, clamp it to Telegram's 4096 maximum, use it throughout the send chunk planner, preserve current markdown/html chunking behavior, add focused regression coverage, and record the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. On current main, configure `channels.telegram.textChunkLimit = 4096` and send an HTML or default-markdown Telegram text whose rendered length is over 4000 but at or under 4096; the planner still uses literal 4000 and splits despite the configured limit.

## Solution Assessment

No, not as submitted. The production approach is the narrow maintainable fix, but the added regression must pass the configured `cfg`, the PR needs a changelog entry, and the rebase must preserve current main's markdown-mode chunking fix.
