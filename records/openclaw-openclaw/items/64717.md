## Summary

Keep open. Current main still hardcodes Telegram send chunk planning at 4000 despite the documented configurable limit, so the bug is real, but this PR is stale against current main, lacks required real Telegram proof, and needs small correctness fixes before merge.

## What This Changes

The PR updates Telegram send chunk planning to use the configured text chunk limit capped at 4096 and adds a regression test for the configured 4096 case.

## Best Possible Solution

Keep the fix in the Telegram plugin by resolving `channels.telegram.textChunkLimit` inside current main's rendered-HTML planner, clamping to Telegram's 4096 maximum, using the value consistently across fallback chunking, and adding focused configured-limit coverage plus redacted real Telegram proof.

## Reproduction Assessment

Yes. By source inspection, configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text; current main still uses literal `4000` inside `buildChunkedTextPlan`, so it splits below the configured limit.

## Solution Assessment

Yes in direction, but not as submitted. Reusing the existing resolver is the narrow maintainable fix, but the branch must be rebased onto current main, keep `renderHtmlText(rawText)` as the planner input, pass explicit `cfg` in the regression, and provide real Telegram proof.
