## Summary

Keep open: current main still hardcodes `4000` inside Telegram send chunk planning even though the config/docs expose `channels.telegram.textChunkLimit`, so the PR targets a real bug. ## What This Changes

The PR changes Telegram text send chunk planning to use the resolved configured `textChunkLimit` capped at 4096, and adds a regression test for the 4096 configured-limit case.

## Best Possible Solution

Keep the fix in the Telegram plugin: resolve `channels.telegram.textChunkLimit`, clamp the effective send limit to Telegram's 4096 cap, apply it throughout `buildChunkedTextPlan` after current markdown-to-HTML rendering, and cover configured-limit behavior with focused tests plus real Telegram proof.
