## Summary

Keep open: current main still ignores the configured Telegram `textChunkLimit` inside `buildChunkedTextPlan`, so this PR targets a real narrow bug. The submitted branch is not merge-ready because it is stale against the current send API/rendered-HTML planner shape and lacks required real Telegram proof.

## What This Changes

The PR changes Telegram send chunk planning to use a config-resolved text chunk limit capped at 4096 and adds a send-path regression test.

## Best Possible Solution

Keep the fix in the Telegram plugin: resolve `channels.telegram.textChunkLimit`, clamp it to Telegram's 4096 maximum, apply it consistently in `buildChunkedTextPlan`, preserve rendered HTML plus plain fallback behavior, and land focused regression coverage.

## Reproduction Assessment

Yes. Configure `channels.telegram.textChunkLimit = 4096` and send a 4001-4096 character Telegram text through `sendMessageTelegram`; source inspection of current main shows the planner still uses literal `4000`, so it will split below the configured limit.

## Solution Assessment

Yes in direction, but not as submitted. Reusing `resolveTextChunkLimit` with a 4096 clamp is the narrow maintainable fix, but the branch needs a rebase that keeps rendered HTML as planner input, passes `cfg` in the test, and adds real Telegram proof.
