## Summary

Keep open: the report is a concrete scheduler bug with documented expected behavior, but the exact May 13/14/15 jump did not reproduce in a direct Croner 10.0.1 probe, and current source shows a bad future jobs-state slot could be preserved across restart.

## What This Changes

The issue reports that six-field Asia/Shanghai daily cron schedules such as `0 0 21 * * *` can persist `nextRunAtMs` days or weeks beyond the next valid run after Gateway restart or runtime recomputation.

## Best Possible Solution

Add a focused restart/state regression for the reported six-field Asia/Shanghai schedules, then adjust cron state reconciliation or schedule computation so a persisted slot cannot skip the next valid occurrence while preserving missed-run safeguards.

## Reproduction Assessment

Unclear: the report gives concrete schedules and observed dates, but a direct Croner 10.0.1 probe for those schedules returned the expected next runs. Source inspection does show that an already bad future `jobs-state.json` slot can survive restart.

## Solution Assessment

Unclear: recomputing through Croner alone appears correct, so the safest path is first to create a failing Gateway restart/state test before changing future-slot preservation logic.
