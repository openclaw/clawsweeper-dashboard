## Summary

Keep open: the patch is not ready because it omits the WhatsApp `onReasoningStream` wiring promised in the PR body and instead suppresses reasoning in shared runner/final-payload paths that other channels use for documented reasoning lanes.

## What This Changes

The PR changes shared auto-reply runner/payload handling to suppress `isReasoning` streams/final payloads, adds focused tests, and adds a WhatsApp changelog entry.

## Best Possible Solution

Use a WhatsApp-owned `onReasoningStream: () => {}` sink with regression coverage, preserve shared reasoning payloads for channels with reasoning lanes, and point the changelog at the PR that actually lands.

## Reproduction Assessment

Yes for the PR defect: the downloaded diff omits the WhatsApp dispatch file, while current main still lacks `replyOptions.onReasoningStream` and the runner only creates the reasoning callback when an option is supplied. A live WhatsApp reproduction was not attempted because the reported behavior can expose private reasoning to an external channel.

## Solution Assessment

No. The current patch broadens suppression in shared runner/payload helpers and leaves the WhatsApp callback unwired; the safer path is the narrower WhatsApp dispatch callback plus tests that preserve channel-owned reasoning lanes.
