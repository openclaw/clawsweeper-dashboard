## Summary

Keep open: the PR is still useful in the #74886 area, but the current diff no longer wires the WhatsApp `onReasoningStream` sink and instead drops reasoning globally in core paths that channel-specific reasoning lanes depend on.

## What This Changes

The PR adds core filtering for `isReasoning` runner/final payloads, tests that filtering, and a changelog entry for the WhatsApp reasoning leak.

## Best Possible Solution

Wire the WhatsApp-owned `replyOptions.onReasoningStream: () => {}` sink at the inbound dispatch boundary, keep core reasoning payloads available for channel-specific lanes, and leave #74886 open for the separate fallback/stability investigation.

## Reproduction Assessment

Yes for the PR regression: a source-level reproduction is to emit `isReasoning` block/final payloads under a reasoning-visible channel lane and observe that the patched core runner drops them before channel delivery. The full live WhatsApp leak was not rerun in this read-only review.

## Solution Assessment

No. The narrow maintainable fix is WhatsApp-boundary opt-in to the existing reasoning suppression path, not global core filtering that changes explicit reasoning visibility behavior for other lanes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve reasoning payloads for channel lanes:** `src/auto-reply/reply/agent-runner-payloads.ts:187-190`
  - body: This drops every `isReasoning` final payload before channel dispatch. That breaks the documented `/reasoning on` behavior where reasoning can be delivered as its own lane/message, and it also does not fix the WhatsApp-specific missing callback that can produce untagged contaminated chunks. Keep these payloads available to channel-specific dispatch and suppress them at the WhatsApp/generic delivery boundary instead.
  - confidence: 0.88
- **[P2] Do not skip reasoning callbacks in the runner:** `src/auto-reply/reply/agent-runner-execution.ts:1113-1115`
  - body: Returning `skip` for `payload.isReasoning` here prevents `onBlockReply`/`onToolResult` reasoning lanes from seeing marked reasoning payloads at all. Current main intentionally emits reasoning-visible blocks with `isReasoning: true`; WhatsApp should provide a no-op `onReasoningStream` sink rather than changing this shared callback contract.
  - confidence: 0.84
