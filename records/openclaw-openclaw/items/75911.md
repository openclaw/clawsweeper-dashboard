## Summary

Keep open: the PR still misses the stated WhatsApp `onReasoningStream` wiring and instead adds shared reasoning filtering that can regress channel-specific reasoning lanes.

## What This Changes

The PR filters `isReasoning` payloads in shared auto-reply runner/final-payload paths, adds focused runner/payload tests, and adds a WhatsApp changelog entry for #74886.

## Best Possible Solution

Use a WhatsApp-owned no-op reasoning sink with inbound-dispatch regression coverage, and avoid global final-payload stripping unless a shared contract test proves every channel expects it.

## Reproduction Assessment

Yes for the PR defect: current main's WhatsApp `replyOptions` lack `onReasoningStream`, shared dispatch only forwards that callback when supplied, and the PR diff does not modify the WhatsApp dispatch file. I did not live-test WhatsApp because the suspected failure mode can disclose private reasoning.

## Solution Assessment

No. The proposed shared filtering diverges from the stated WhatsApp callback fix and risks channel-owned reasoning lanes; the narrower maintainable repair is to wire the WhatsApp sink and test that boundary.
