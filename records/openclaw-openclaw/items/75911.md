## Summary

Keep open: the latest PR head still applies a shared reasoning-payload filter instead of the WhatsApp-owned no-op sink described by the title/body, and that shared filter risks channel-specific reasoning lanes that current docs and tests preserve.

## What This Changes

The PR adds shared auto-reply runner filtering for `isReasoning` payloads, related runner/payload tests, and a WhatsApp changelog entry for #74886.

## Best Possible Solution

Use a WhatsApp-owned no-op reasoning sink and delivery-boundary regression coverage, while leaving shared/channel-specific reasoning payload contracts intact unless reviewed separately.

## Reproduction Assessment

Yes, source-reproducible for the PR defect: the live diff for head 783dc4805532 has no WhatsApp dispatch change, while current WhatsApp replyOptions still omit `onReasoningStream` and the new filter would run in a shared final-payload path.

## Solution Assessment

No. The narrower maintainable fix is to wire the WhatsApp boundary sink and test that boundary, not to globally drop all `isReasoning` payloads before channel-specific delivery paths can decide what to do.
