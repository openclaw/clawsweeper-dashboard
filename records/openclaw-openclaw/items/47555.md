## Summary

Keep open. Current main still lets an explicit root `--profile` coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`, and `gateway restart` resolves the launchd target from that inherited label before the requested profile-derived label; the related PR #66000 remains the open implementation path for the same stale-label root cause.

## What This Changes

The issue reports that `openclaw --profile <name> gateway restart` can restart the current agent session's gateway instead of the requested profile when run from an inherited launchd environment.

## Best Possible Solution

Explicit root profile selection should override inherited gateway service identity for lifecycle and status resolution; land PR #66000 or an equivalent narrow fix with restart/status coverage, then close both stale-label reports as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets that label.

## Solution Assessment

Yes. The narrow maintainable solution is to clear or re-derive stale launchd service identity when an explicit root `--profile` is accepted, instead of changing generic lifecycle routing or asking users to call `launchctl` directly.
