## Summary

Keep open. Current main still has the stale launchd-label path: an explicit root `--profile` sets profile state but does not clear inherited `OPENCLAW_LAUNCHD_LABEL`, and macOS restart resolution still prefers that inherited label before deriving the profile-specific LaunchAgent target. Prior discussion usefully documented the `launchctl` workaround and state-loss impact, but the core lifecycle behavior remains unresolved.

## What This Changes

The issue asks for `openclaw --profile <name> gateway restart` to target the requested profile instead of the current agent session's inherited LaunchAgent label.

## Best Possible Solution

Explicit root `--profile` selection should clear or re-derive conflicting inherited launchd service identity for lifecycle/status resolution, then land PR #66000 or an equivalent narrow fix with restart/status coverage.

## Reproduction Assessment

Yes. A high-confidence source path exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets that label.

## Solution Assessment

Yes. The narrow maintainable solution is to sanitize stale launchd service identity when an explicit root profile is accepted, rather than changing generic lifecycle routing or relying on users to call `launchctl` directly.
