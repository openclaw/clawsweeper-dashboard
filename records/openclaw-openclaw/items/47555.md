## Summary

Keep open. Current main still allows an explicit root `--profile` to coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`; the macOS launchd path resolves that inherited label before the profile-derived label, so `gateway restart` can still target the wrong LaunchAgent. The related PR #66000 is already an open narrow implementation candidate for the same stale-label root cause, so this issue should wait for that PR or an equivalent fix to land or be rejected.

## What This Changes

The issue asks for `openclaw --profile <profile> gateway restart` to honor the explicitly requested profile when run from a session that inherited another gateway's launchd identity.

## Best Possible Solution

Land PR #66000 or an equivalent narrow patch that makes explicit root profile selection clear or re-derive conflicting OpenClaw-managed launchd labels before lifecycle/status resolution, with regression coverage for restart, status/plist lookup, and custom-label preservation. After that lands, close this issue and #65643 as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` builds the launchctl target from it.

## Solution Assessment

Yes. The best direction is a narrow profile-environment/service-identity fix, not a broader lifecycle rewrite; PR #66000 appears to target that boundary, but it should be reviewed for restart coverage and custom-label semantics before this issue closes.
