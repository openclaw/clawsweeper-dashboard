## Summary

Keep open. Current main still lets an explicit root `--profile` coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`; the launchd restart path resolves that inherited label before the profile-derived label, so it can still target the session LaunchAgent rather than the requested profile. The related PR #66000 is an open implementation candidate for the same stale-label root cause, so this should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue requests that `openclaw --profile <name> gateway restart` honor the explicit profile when run from an agent or gateway session instead of restarting the inherited session gateway.

## Best Possible Solution

Explicit root `--profile` selection should override inherited gateway service identity for launchd lifecycle and status resolution, while preserving custom launchd labels in managed contexts that did not request a different profile. Review and land PR #66000, or an equivalent narrow patch with restart/status coverage, then close this issue and the related status issue as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets it. I did not perform a live macOS launchd restart in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable solution is to clear or re-derive launchd service identity when an explicit root profile is accepted, rather than changing generic lifecycle routing; PR #66000 appears to target that boundary but should be reviewed for restart coverage and custom-label semantics.
