## Summary

Keep open. Current main still applies an explicit root `--profile` without clearing inherited `OPENCLAW_LAUNCHD_LABEL`, and macOS launchd restart resolution still uses that inherited label before the profile-derived label, so the wrong LaunchAgent target remains source-reproducible; PR #66000 is the open implementation path for the same stale-label root cause.

## What This Changes

The issue asks for `openclaw --profile <name> gateway restart` to target the requested profile instead of an inherited current-session LaunchAgent.

## Best Possible Solution

Profile-targeted gateway lifecycle and status commands should derive OpenClaw-managed service identity from the explicit profile, while preserving intentional custom launchd labels when no conflicting profile target is requested.

## Reproduction Assessment

Yes. A high-confidence source path exists: with `OPENCLAW_PROFILE=rescue` from root `--profile` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, launchd label resolution returns the inherited label and `gateway restart` targets it.

## Solution Assessment

Yes. The narrow maintainable repair is to clear or re-derive stale OpenClaw-managed launchd labels when an explicit root profile is accepted, then cover both `gateway status` and `gateway restart` resolution paths.
