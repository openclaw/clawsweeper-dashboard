## Summary

Keep open. Current main still lets an explicit root `--profile` coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`, and the macOS launchd restart path resolves the inherited label first, so the command can still target the current session's LaunchAgent instead of the requested profile; #66000 is the existing open implementation candidate for the same stale-label root cause.

## What This Changes

The issue asks for `openclaw --profile <name> gateway restart` to target the requested profile even when invoked from an agent session with inherited gateway launchd environment.

## Best Possible Solution

Explicit root profile selection should override inherited gateway service identity for lifecycle/status resolution; land #66000 or an equivalent narrow fix with restart/status/handoff coverage, then close the stale-label reports as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets that LaunchAgent.

## Solution Assessment

Yes. The narrow maintainable solution is to clear or re-derive stale launchd service identity when an explicit root `--profile` is accepted, rather than changing generic lifecycle routing or asking users to call `launchctl` directly.
