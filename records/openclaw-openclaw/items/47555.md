## Summary

Keep open. Current main still allows root `--profile` to coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`, and the launchd restart path resolves that inherited label before the profile-derived label, so the reported wrong-gateway restart remains source-reproducible.

## What This Changes

The issue requests that `openclaw --profile <name> gateway restart` from an agent session target the requested profile instead of the session-inherited gateway LaunchAgent.

## Best Possible Solution

Land PR #66000 or an equivalent narrow fix that makes explicit root profile selection override stale inherited launchd service identity for restart/status resolution, with focused coverage for conflicting and matching label cases.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: after root `--profile rescue` sets `OPENCLAW_PROFILE=rescue`, an inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway` still wins in `resolveLaunchAgentLabel`, so `restartLaunchAgent` targets the inherited service.

## Solution Assessment

Yes. Clearing or re-deriving stale launchd service identity when an explicit root profile is accepted is the narrow maintainable fix; asking users to call `launchctl` directly would leave the CLI contract broken.
