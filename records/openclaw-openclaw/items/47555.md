## Summary

Keep open. Current main still contains the stale launchd-label path: root `--profile` updates the CLI profile environment, but launchd resolution still prefers inherited `OPENCLAW_LAUNCHD_LABEL`, and `gateway restart` uses that resolved label as the launchctl target. PR #66000 is an open implementation candidate for the same root cause, so this issue should remain open until that PR, or an equivalent patch, lands or is rejected.

## What This Changes

The issue requests that `openclaw --profile <name> gateway restart` from an agent or gateway session restart the explicitly requested profile's LaunchAgent instead of the current session's inherited service label.

## Best Possible Solution

Explicit root `--profile` selection should override inherited gateway service identity for lifecycle and status resolution, while preserving custom launchd labels in managed contexts that did not request a different profile. Review and land PR #66000, or an equivalent narrow patch with restart/status coverage, then close this issue and the related status issue as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets it. This review did not perform a live macOS LaunchAgent restart.

## Solution Assessment

Yes. The narrowest maintainable direction is to sanitize or re-derive launchd service identity when an explicit root profile is accepted, rather than changing generic lifecycle routing. PR #66000 appears to target the right boundary but still needs maintainer review for restart coverage and custom-label semantics.
