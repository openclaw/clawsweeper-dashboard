## Summary

Keep open. Current main still lets an explicit root `--profile` coexist with an inherited `OPENCLAW_LAUNCHD_LABEL`, and `gateway restart` resolves the LaunchAgent target from that inherited label before using the requested profile. The related PR #66000 remains an open implementation candidate for the same stale-label root cause, so this should not be closed until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw --profile <name> gateway restart` from an agent session to restart the requested profile's LaunchAgent instead of the current session's inherited gateway service.

## Best Possible Solution

Explicit root profile selection should override inherited gateway service identity for lifecycle and status resolution, while preserving custom launchd labels in managed contexts that did not request a different profile. Review and land PR #66000 or an equivalent narrow patch with restart/status coverage, then close this issue and the related status tracker as implemented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `OPENCLAW_PROFILE=rescue` and inherited `OPENCLAW_LAUNCHD_LABEL=ai.openclaw.gateway`, current `resolveLaunchAgentLabel` returns the inherited label and `restartLaunchAgent` targets that label. Live macOS launchd execution was not run because this was a read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to clear or re-derive stale launchd service identity when an explicit root `--profile` is accepted, rather than changing generic lifecycle routing or requiring users to call `launchctl` directly.
