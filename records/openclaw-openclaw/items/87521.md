## Summary

Keep open: the reported proxy-fetch bug is plausible and the PR targets the right shared boundary, but the current patch drops non-body `Request` options while flattening and still lacks after-fix real behavior proof.

## What This Changes

The PR normalizes `Request`-like inputs in `proxy-fetch` before forwarding them through undici so method, headers, body, and signal are preserved.

## Best Possible Solution

Preserve the full Request-input fetch contract in proxy-fetch normalization, keep the fixed header replacement behavior, and add redacted after-fix proxy upload proof before merge.

## Reproduction Assessment

Yes from source inspection: current main only normalizes separate `init`, while Request-carried FormData lives on `input` and bypasses OpenClaw's existing undici FormData conversion. I did not run a live proxy upload repro in this read-only review.

## Solution Assessment

No in its current form. Fixing the shared proxy-fetch boundary is the right layer, but the implementation must preserve the full Request contract instead of replaying only method, headers, body, and signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Request options when flattening:** `src/infra/net/proxy-fetch.ts:157-158`
  - body: `makeProxyFetch` is exported as a fetch-like helper, but converting a `Request` to `input.url` only replays method/headers/body/signal. Undici's `new Request(request, init)` preserves redirect, credentials, cache, mode, referrer/referrerPolicy, integrity, keepalive, and the URL list, so `fetch(new Request(url, { redirect: "manual" }))` through this proxy would silently go back to default redirect behavior. Please carry those Request fields through the normalized init and add a regression test for at least `redirect` or another non-body option.
  - confidence: 0.91
