## Summary

Need to keep this open: current main still has the reported main-session cron wake behavior where cron can enqueue a system event without requesting delivery, and the related long-running task progress/watch UX is only partly covered by existing process/task primitives.

## What This Changes

The issue reports that long-running agent work can go silent because main-session cron self-wakes with systemEvent payloads do not create visible assistant turns, and asks for better task-watching, delivery-status surfacing, model-loop guidance, and sudo recovery UX.

## Best Possible Solution

Keep this open as the canonical UX bug; first decide and fix the main-session cron wake contract so a scheduled systemEvent either reliably produces a visible assistant turn or is clearly labeled and documented as non-delivery, then split watcher, model, and sudo improvements into focused follow-ups.

## Reproduction Assessment

Yes, from source inspection: a main-session cron with systemEvent payload can enqueue into a cron-run lane and record not-requested delivery without producing a visible assistant turn. The full droplet timing sequence was not live-reproduced.

## Solution Assessment

Unclear: the cron wake behavior needs a focused contract decision before implementation. A narrow first fix should address visible/non-visible delivery semantics before adding broader watcher, model-warning, or sudo-elevation features.
