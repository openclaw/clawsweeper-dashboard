## Summary

Keep open. The patch is a narrow source-plausible fix with focused coverage and no concrete code/security finding, but it changes the mutating-tool warning safety boundary and still needs after-fix real behavior proof before merge.

## What This Changes

The PR moves `messages.suppressToolErrors` ahead of mutating-tool warning generation in embedded run payloads and updates the regression table to expect no warning when the option is enabled.

## Best Possible Solution

Maintainers should choose the mutating-warning policy boundary, then either land this narrow reordering with real Telegram or equivalent runtime proof, or keep the broader delivery-mode design on https://github.com/openclaw/openclaw/issues/39406.
