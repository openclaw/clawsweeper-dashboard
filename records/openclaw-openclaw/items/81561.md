## Summary

Keep open. Current main still lets mutating tool warnings bypass `messages.suppressToolErrors`, this PR is the focused replacement for the older closed attempt, and I found no blocking correctness issue in the diff.

## What This Changes

The PR moves `messages.suppressToolErrors` ahead of mutating-tool warning generation in `buildEmbeddedRunPayloads` and updates the regression expectation for a mutating `write` error.

## Best Possible Solution

Merge the focused policy reorder and regression after normal maintainer gates; leave broader all/final/off warning-delivery policy in the open product discussion.

## Reproduction Assessment

Yes. Current-main source classifies `write` as mutating and returns from the mutating-warning branch before `messages.suppressToolErrors`, and the current test encodes that warning as expected behavior.

## Solution Assessment

Yes. Moving the existing suppression guard before mutating warning generation is the narrow maintainable fix, while preserving the special `sessions_send` suppression and existing assistant-error safeguards.
