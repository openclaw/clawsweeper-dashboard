## Summary

Keep this open: the report describes credible restart-time cron state drift, and current main shows deletion is authoritative inside core cron storage but does not identify the external or plugin reconciliation path that recreates this named job.

## What This Changes

Deleting a recurring cron job via `openclaw cron delete` should persist across gateway restarts instead of reappearing with a new `job_id`.

## Best Possible Solution

Identify the component that owns `open-gauge-watchdog` and make its startup reconciliation respect an explicit user delete, or add a narrowly scoped managed-cron deletion contract if owner-managed jobs are intended to self-heal.
