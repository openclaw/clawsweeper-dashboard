## Summary

Keep open: the PR is a focused, correct Firecrawl hardening change, and current main plus the latest release still use the unbounded success-path `response.json()` read. Contributor proof is now sufficient; the remaining decision is maintainer acceptance of the 16 MiB compatibility boundary.

## What This Changes

The PR changes the bundled Firecrawl client to parse successful JSON responses through the exported response-limit runtime helper with a 16 MiB cap and adds regression tests for normal and oversized streamed bodies.

## Best Possible Solution

Land the scoped success-path cap using the shared SDK helper once maintainers accept the 16 MiB boundary, and track the existing error-path JSON gap separately if full Firecrawl coverage is desired.

## Reproduction Assessment

Yes from source and PR proof: current main and `v2026.6.10` call `response.json()` in the shared Firecrawl success parser, while the PR body shows the patched parser cancelling an oversized no-Content-Length stream at the cap.

## Solution Assessment

Yes, this is the best local fix shape: it caps the shared Firecrawl success parser and reuses the existing plugin SDK response-limit helper instead of adding custom stream code. The only remaining choice is maintainer acceptance of the cap and whether the existing error-path gap should be separate work.
