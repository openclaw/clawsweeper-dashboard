## Summary

Keep open for normal maintainer review: current main and v2026.6.10 still use an unbounded Firecrawl success-body `response.json()` read, while this PR applies the existing SDK byte limiter with focused tests and sufficient terminal proof. I found no blocking code findings, but the 16 MiB cap is an intentional compatibility boundary maintainers should accept before merge.

## What This Changes

The PR changes Firecrawl success JSON parsing to use the plugin SDK response byte limiter with a 16 MiB cap and adds regression tests for small and oversized streamed bodies.

## Best Possible Solution

Land the scoped Firecrawl success-body cap with the shared SDK helper after maintainers accept the 16 MiB compatibility boundary; track the existing error JSON pre-read separately if full Firecrawl coverage is desired.

## Reproduction Assessment

Yes from source and PR proof: current main and v2026.6.10 parse Firecrawl success bodies with `response.json()`, while the PR body's terminal harness shows the patched helper cancels an oversized stream at the cap and the old shape drains it.

## Solution Assessment

Yes, this is the best local fix shape: it caps the shared Firecrawl success parser and reuses the existing plugin SDK limiter instead of adding custom stream code. The remaining decision is whether maintainers accept the 16 MiB policy, not a different implementation layer.
