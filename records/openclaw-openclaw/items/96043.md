## What This Changes

This PR routes Firecrawl success JSON parsing through the shared response byte-limit helper with a 16 MiB cap and adds regression tests for small and oversized streamed bodies.

## Best Possible Solution

Land the scoped Firecrawl success-body cap with the shared SDK helper after real behavior proof is added and the 16 MiB compatibility tradeoff is accepted; track the existing error-path JSON gap separately if maintainers want full Firecrawl coverage.

## Reproduction Assessment

Yes from source, not from a live run: current main and `v2026.6.9` parse Firecrawl success bodies with `response.json()`, and search/scrape both call that helper. The dependency source shows `json()` fully reads the body before parsing.

## Solution Assessment

Yes, this is the best local fix shape: it puts the cap at the shared Firecrawl success parser and reuses the existing plugin SDK response-limit helper instead of adding custom stream code. The remaining blockers are proof and maintainer acceptance of the cap, not a different implementation layer.
