## Label Justifications

- `impact:message-loss`: The reported path drops the useful cron answer and delivers generic failure text after accepting a fallback-safe failed primary result.

## Summary

Keep open: current main still lacks result-level fallback classification in the isolated cron embedded fallback call, and the latest release has the same gap. The best path is to review and land the open proof-positive fix PR rather than close this issue before a merge.

## What This Changes

The issue asks for isolated cron embedded `agentTurn` jobs to use the configured model fallback chain when the primary returns a fallback-safe result-level failure such as reasoning-only, empty, or `incomplete_turn`.

## Best Possible Solution

Review and land the open proof-positive cron fallback PR if it is still the best implementation; keep this issue open until a fix merges.

## Reproduction Assessment

Yes. Source inspection shows cron omits result classification while `runWithModelFallback` only advances on returned result-level failures when a classifier is supplied, matching the provided failing repro shape.

## Solution Assessment

Yes. Reusing the existing embedded classifier and exhausted-result merge helper in the cron embedded fallback path is the narrowest maintainable fix because sibling answer-bearing paths already use that contract.
