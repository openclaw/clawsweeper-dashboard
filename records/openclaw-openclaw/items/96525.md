## Label Justifications

- `P1`: A recurring scheduled agent workflow can skip configured fallback models and deliver a generic failure instead of the expected answer.
- `impact:message-loss`: The useful cron answer is lost when the primary returned result is accepted instead of promoting to a healthy fallback model.

## Summary

Keep open: current main still omits result-level classification from the isolated cron fallback call, so fallback-safe incomplete/reasoning-only embedded results can be accepted as the primary result instead of advancing to configured fallback models. Open pull requests already target the same bug, so the issue should stay open until one canonical fix merges.

## What This Changes

The issue reports that isolated cron embedded agentTurn runs do not engage configured model fallbacks for fallback-safe result-level failures such as reasoning-only, empty-visible, or incomplete_turn results.

## Best Possible Solution

Land one canonical fix, preferably the better-proven open candidate, that wires the existing embedded result classifier and exhausted-result merge helper into the cron embedded fallback path with focused regression coverage for embedded and CLI scoping.

## Reproduction Assessment

Yes. Source inspection shows the cron call omits the classifier and merge helper while runWithModelFallback requires classifier output to treat returned fallback-safe failures as failed candidates; I did not create the provided repro file because this review is read-only.

## Solution Assessment

Yes. Reusing the existing embedded classifier and merge helper in the cron embedded path is the narrowest maintainable fix because the interactive and auto-reply paths already use that contract for the same returned-result failure class.
