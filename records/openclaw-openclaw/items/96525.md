## Label Justifications

- `impact:message-loss`: The reported failure loses the answer the fallback model would have delivered and replaces it with a generic error payload.

## Summary

Keep open: current `main` still has the cron-specific gap. The shared fallback loop and embedded classifier already support result-level failures, and the interactive/auto-reply callers pass that classifier, but isolated cron does not, so a configured fallback can still be skipped for returned embedded error results.

## What This Changes

The issue reports that isolated cron `agentTurn` jobs do not invoke configured model fallbacks when an embedded primary returns reasoning-only, empty, or fallback-safe `incomplete_turn` results.

## Best Possible Solution

Add the existing embedded result classifier and exhausted-result merger to the isolated cron embedded fallback path, preserving CLI-provider behavior and adding a focused regression test for fallback-safe returned results.

## Reproduction Assessment

Yes at source level: current cron code omits the classifier that the shared fallback loop needs to advance after returned embedded terminal failures. I did not run the supplied repro because this review is read-only and tests can create artifacts.

## Solution Assessment

Yes: reusing the existing embedded classifier and merge helper is the narrow maintainable fix because the sibling interactive and auto-reply paths already use that contract. A cron-only classifier or new config surface would add duplicate behavior.
