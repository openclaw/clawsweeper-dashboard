## Summary

Keep open: current main still omits result-level fallback classification in the isolated cron embedded fallback call, so returned fallback-safe incomplete/reasoning-only results can be accepted as the primary result instead of advancing to configured fallback models. The issue is already paired with open fix candidates, especially https://github.com/openclaw/openclaw/pull/96529, so it should stay open until a fix merges.

## What This Changes

The issue requests that isolated cron agentTurn model fallback classify reasoning-only, empty-visible, and incomplete_turn result-level failures so configured fallback models are tried instead of dropping the answer.

## Best Possible Solution

Land one focused cron fix that passes the existing embedded result fallback classifier and exhausted-result merge helper to the isolated cron embedded fallback call, with regression coverage and a guard that leaves CLI-provider cron classification unchanged.

## Reproduction Assessment

Yes, this is source-reproducible: current main shows the cron runWithModelFallback call omits classifyResult/mergeExhaustedResult while runWithModelFallback only advances returned result-level failures when a classifier is supplied. I did not run the pasted repro because this review is read-only.

## Solution Assessment

Yes, the narrow best fix is to reuse the existing embedded result classifier and exhaustion merge helper in isolated cron, scoped away from CLI candidates. The open fix candidate at https://github.com/openclaw/openclaw/pull/96529 follows that shape and should be reviewed or replaced by an equivalent focused patch.
