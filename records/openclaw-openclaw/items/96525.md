## Summary

Keep open: current main still lacks result-level fallback classification in the isolated cron embedded fallback call, so returned fallback-safe incomplete/reasoning-only results can be accepted as the primary result instead of advancing to configured fallback models.

## What This Changes

The issue reports that isolated cron embedded agent runs do not pass result-level fallback classification into the model fallback executor, so fallback-safe empty/reasoning-only/incomplete results stop at the primary model instead of trying configured fallbacks.

## Best Possible Solution

Reuse the existing embedded result fallback classifier and exhausted-result merge helper in the cron embedded fallback path, with focused regression coverage proving returned fallback-safe incomplete/reasoning-only results advance to configured fallbacks while CLI-provider behavior stays unchanged.

## Reproduction Assessment

Yes. Source inspection shows cron omits result classification while `runWithModelFallback` only advances on returned result-level failures when a classifier is supplied, matching the provided failing repro shape.

## Solution Assessment

Yes. The best fix is to reuse the existing embedded classifier and exhausted-result merge helper in the cron embedded path, because interactive and auto-reply already use that contract for the same returned-result failure class.
