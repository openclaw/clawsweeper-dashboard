## Summary

Keep this PR open: it fixes a real ClawHub skill overwrite bug, but the shared helper currently relaxes the existing owner-qualified mismatch rule for legacy tracked skills with no stored owner and the user-facing `--force` docs still need the new refusal behavior.

## What This Changes

The PR adds a shared ClawHub owner-match guard on install/update paths plus a regression test for refusing cross-owner `skills install --force` overwrites.

## Best Possible Solution

Keep the shared lifecycle guard, but treat a missing tracked owner as a mismatch for owner-qualified requests, add regression coverage for legacy no-owner tracking, update skills docs, and then have maintainers accept the fail-closed compatibility change.

Yes. Current main is source-reproducible because owner-qualified installs normalize to the bare local slug, `--force` bypasses the existing-target refusal, and lock/origin writes use that same slug key; the PR also includes live before/after CLI output.

## Solution Assessment

No, not yet. The shared guard is the right boundary, but it must preserve the no-owner mismatch behavior already used by update/verify and the docs need to match the new `--force` contract.
