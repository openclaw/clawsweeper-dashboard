## Summary

Keep open for maintainer review: the patch appears correct and well proven, but it intentionally changes owner-qualified cross-owner `--force` installs from overwrite to hard error, so the remaining decision is compatibility acceptance.

## What This Changes

The PR adds a shared ClawHub owner-match guard for install/update flows and a regression test so `skills install @other/slug --force` refuses to overwrite a skill tracked under a different owner.

## Best Possible Solution

Land the lifecycle-level owner-match guard after maintainer acceptance, preserving fresh installs, same-owner reinstalls, and bare-slug force behavior while blocking cross-owner tracked-state clobbering.

## Reproduction Assessment

Yes. Source inspection shows current main can parse `@bob/weather` to the `weather` target, let `--force` bypass the existing-target check, then write Bob into origin/lock; the contributor also supplied live before/after CLI output.

## Solution Assessment

Yes. Reusing the existing update-path owner invariant inside the ClawHub lifecycle is the narrowest maintainable fix; CLI, gateway, or archive-install-only checks would leave another caller or owner-state path asymmetric.
