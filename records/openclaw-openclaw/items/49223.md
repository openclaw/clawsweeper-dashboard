## Summary

Keep open. Current main still sends explicit `sessions_send` delivery requests through the generic inter-session/A2A prompt path with no delivery-intent signal or carve-out that prevents `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY` from suppressing a normal outbound channel message.

## What This Changes

Request that WhatsApp group `sessions_send`/`inter_session` delivery requests prefer emitting the requested normal channel message over generic skip or silent-control-token behavior.

## Best Possible Solution

Add a backward-compatible delivery-intent guard in the sessions_send/A2A prompt/runtime path so explicit normal channel-delivery requests cannot be satisfied by control tokens, with focused regression coverage and docs updates if the tool contract changes.

## Reproduction Assessment

Yes, at source level: current main deterministically routes `sessions_send` input as inter-session data through A2A guidance where skip/silent control tokens remain valid and no delivery-intent override exists. A full live WhatsApp reproduction was not run.

## Solution Assessment

Unclear which exact implementation shape is best, but the issue targets the right seam. A structured internal delivery-intent signal is safer than phrase-only heuristics; a narrow prompt/runtime guard is a smaller backward-compatible first fix.
