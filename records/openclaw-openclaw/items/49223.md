## Summary

Keep open: current main still has the source-level path described by the report. `sessions_send` marks the target turn as `inter_session`, gives broad skip-token guidance, has no delivery-intent/mode field, and suppresses exact skip/silent tokens before visible delivery; the related Telegram/requester-routing work is adjacent but does not solve this WhatsApp delivery-intent precedence bug.

## What This Changes

The issue asks OpenClaw to make explicit WhatsApp/inter-session delivery requests prefer normal outbound group messages over `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY` suppression.

## Best Possible Solution

Add a narrow `sessions_send`/inter-session delivery-intent guard in prompt/runtime behavior so explicit channel-delivery requests produce a normal allowed message, while coordination-only chatter can still use skip tokens.

## Reproduction Assessment

Yes, at source level: current main annotates `sessions_send` as `inter_session`, encourages broad skip/silent exits, and suppresses those exact tokens before gateway delivery. I did not establish a live WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes for keeping this open and routing a narrow bugfix. The first fix should refine the existing prompt/control-token contract with focused regression coverage; a typed delivery mode can remain a follow-up only if the narrow fix is not reliable.
