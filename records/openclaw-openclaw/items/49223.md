## Summary

Keep open: current main and the latest release still mark sessions_send target turns as generic inter_session data, still offer broad skip-token exits, and have no delivery-intent path that would make an explicit allowed channel-delivery request outrank REPLY_SKIP, ANNOUNCE_SKIP, or NO_REPLY.

## What This Changes

The issue requests that WhatsApp/group inter_session turns with explicit outbound delivery intent prefer normal channel message emission over generic silent/skip-token handling.

## Best Possible Solution

Add a narrow sessions_send inter-session delivery-intent guard in the existing prompt/runtime contract so explicit allowed channel-delivery requests produce visible messages while coordination-only chatter can still use skip tokens.

## Reproduction Assessment

Yes at source level: current main annotates sessions_send target turns as inter_session, provides broad skip/silent exits, and suppresses exact control tokens before visible delivery. I did not establish a live WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes for keeping this open and routing a narrow bugfix. The first fix should refine the existing prompt/control-token contract with focused regression coverage; a typed delivery mode can remain a follow-up if maintainers want a larger public API contract.
