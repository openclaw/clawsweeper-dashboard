## Summary

Keep open: current main and v2026.5.22 still route sessions_send target turns as inter-session data, still provide broad skip/silent-token guidance, and still suppress exact control-token replies before any visible WhatsApp-style group delivery can happen.

## What This Changes

The issue asks OpenClaw to make explicit sessions_send inter-session channel-delivery requests outrank generic REPLY_SKIP, ANNOUNCE_SKIP, and NO_REPLY suppression when policy allows delivery.

## Best Possible Solution

Add a narrow sessions_send inter-session delivery-intent guard in prompt/runtime behavior so explicit allowed channel-delivery requests produce a normal visible message while coordination-only chatter can still use skip tokens.

## Reproduction Assessment

Yes, at source level: current main annotates sessions_send target turns as inter_session, provides broad skip/silent exits, and suppresses exact control tokens before visible delivery. I did not establish a live WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes for keeping this open and routing a narrow bugfix. The first fix should refine the existing prompt/control-token contract with focused regression coverage; a typed delivery mode can remain a follow-up if maintainers want a larger public API contract.
