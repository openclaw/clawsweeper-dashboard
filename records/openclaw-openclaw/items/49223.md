## Summary

Keep open. Current main still routes explicit sessions_send delivery requests through the generic inter_session/A2A path, with no typed delivery intent or prompt/runtime carve-out that would make normal channel delivery outrank REPLY_SKIP, ANNOUNCE_SKIP, or NO_REPLY behavior.

## What This Changes

The issue requests that explicit WhatsApp inter_session delivery requests emit a normal channel message instead of being suppressed by REPLY_SKIP, ANNOUNCE_SKIP, or NO_REPLY.

## Best Possible Solution

Add a backward-compatible delivery-intent signal or explicit prompt/runtime guard in the sessions_send/A2A path so clear requests to post a normal channel message cannot be satisfied by REPLY_SKIP, ANNOUNCE_SKIP, or NO_REPLY, while keeping those tokens valid for real coordination turns. Cover it with focused sessions_send tests and update the session-tool docs if the tool contract changes.

## Reproduction Assessment

Yes, at code-path level: current main deterministically sends sessions_send input as inter_session data through the A2A reply/announce flow where REPLY_SKIP and ANNOUNCE_SKIP are accepted suppression outcomes. A full live WhatsApp reproduction was not run in this read-only review.

## Solution Assessment

Unclear which exact implementation shape is best without maintainer choice, but the issue points at the right seam. A structured delivery-intent flag or mode is safer than phrase-only heuristics; a prompt/runtime guard is a smaller fallback if the public tool schema should not change yet.
