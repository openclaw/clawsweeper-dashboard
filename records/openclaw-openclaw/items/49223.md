## Summary

Keep open: current main and the latest release still expose the reported source-level path where `sessions_send` target turns are marked as inter-session data, broad skip/silent guidance remains available, and exact control tokens are filtered before visible delivery. ## What This Changes

The issue requests that explicit WhatsApp `inter_session` delivery requests be treated as normal outbound channel messages instead of being suppressed by skip/no-reply heuristics.

## Best Possible Solution

Add a narrow `sessions_send`/inter-session delivery-intent guard in prompt/runtime behavior so explicit channel-delivery requests produce a normal allowed message while coordination-only chatter can still use skip tokens.

Yes at source level: current main annotates `sessions_send` messages as `inter_session`, gives broad skip/silent exits, and suppresses exact control tokens before gateway delivery. I did not establish a live WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes for keeping this open and routing a narrow bugfix. The first fix should refine the existing prompt/control-token contract with focused regression coverage; a typed delivery mode can remain a follow-up if maintainers want a larger public API contract.
