## Summary

Keep open: current main still has the source-level path where `sessions_send`/A2A turns are marked as generic `inter_session` data and exact skip/no-reply tokens suppress visible delivery. Adjacent Telegram/thread routing and requester-routing work does not add an explicit-delivery exception for WhatsApp group handoffs.

## What This Changes

The issue asks OpenClaw to make explicit `sessions_send`/`inter_session` requests to post a normal WhatsApp group message prefer visible delivery over `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`.

## Best Possible Solution

Add a narrow prompt/runtime contract so explicit `sessions_send` delivery requests emit normal channel text when policy allows, while preserving skip tokens for coordination-only ping-pong suppression.

## Reproduction Assessment

Yes, at source level: current main marks `sessions_send` turns as `inter_session`, gives broad skip-token guidance, and suppresses exact control-token replies before gateway delivery. I did not establish a live WhatsApp group reproduction in this read-only review.

## Solution Assessment

Yes for keeping this open and routing a narrow bugfix. The safest first fix is prompt/control-token guidance plus regression coverage; a typed delivery mode should stay a follow-up unless the narrow contract cannot make explicit delivery reliable.
