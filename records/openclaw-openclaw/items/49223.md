## Summary

Keep open. Current main still routes `sessions_send` delivery requests through the generic inter-session/A2A prompt path without a delivery-intent guard, so the skip-token path can still outrank an explicit request to post a normal channel message.

## What This Changes

The issue requests that explicit WhatsApp group delivery requests sent through `sessions_send` inter-session turns produce the requested normal channel message instead of being suppressed as `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`.

## Best Possible Solution

Add a backward-compatible delivery-intent guard in the `sessions_send` inter-session/A2A path so explicit normal channel-delivery requests cannot be satisfied by control tokens, with regression coverage and docs/changelog updates if the tool contract changes.

## Reproduction Assessment

Yes, at source level: current main deterministically routes `sessions_send` input as inter-session data through A2A guidance where skip/silent control tokens remain valid and no delivery-intent override exists. A full live WhatsApp reproduction was not performed in this read-only review.

## Solution Assessment

Yes on the direction, but the safest implementation should be narrower than a public schema change at first: add an internal delivery-intent distinction or guarded prompt/runtime path, then expand the tool API only if maintainers want that product surface.
