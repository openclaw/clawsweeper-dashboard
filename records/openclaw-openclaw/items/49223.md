## Summary

Keep open: current main still gives `inter_session` recipients broad skip-token guidance and does not mark explicit channel-delivery intent separately. The related cleanup on https://github.com/openclaw/openclaw/issues/43848 treated this issue as the canonical remaining skip-token follow-up, while https://github.com/openclaw/openclaw/pull/57697 only covers requester-side announce routing.

## What This Changes

The issue requests that explicit `sessions_send`/`inter_session` requests to post a normal channel message not be suppressed as `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`.

## Best Possible Solution

Add a narrow sessions-send delivery-intent contract or prompt guard, with regression coverage proving explicit channel-delivery requests cannot be reduced to skip tokens while coordination chatter can still stop ping-pong.

## Reproduction Assessment

Yes, at source level: current main marks `sessions_send` turns as `inter_session`, teaches skip tokens, and drops exact control-token replies before gateway delivery. I did not establish a live WhatsApp group reproduction in this read-only pass.

## Solution Assessment

Unclear for automation: the reported bug is valid, but maintainers should choose between a prompt-only guard and a typed delivery-intent signal before implementation. A WhatsApp-specific transport fix would not be the right layer because the suppression happens in generic `sessions_send` A2A handling.
