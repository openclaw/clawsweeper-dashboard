## Summary

Keep open: current main still has the reported source-level precedence problem. `sessions_send` only accepts a free-text message, marks it as `inter_session`, and the A2A prompts still broadly offer `REPLY_SKIP`/`ANNOUNCE_SKIP` while the runtime suppresses those control replies before visible delivery; the open related PR targets requester-side announce routing, not the WhatsApp explicit-delivery intent described here.

## What This Changes

The issue requests that explicit WhatsApp group delivery intents received through `inter_session` prefer normal outbound message emission over `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`.

## Best Possible Solution

Add a narrow `sessions_send`/A2A prompt and runtime guard that preserves coordination silence tokens but tells recipients explicit channel-delivery requests should emit a normal message when policy allows, with focused regression coverage for the WhatsApp group shape.

## Reproduction Assessment

Source-level yes, live end-to-end no. Current main still has the free-text inter-session prompt path, broad skip-token guidance, and tests proving control-token replies suppress gateway sends; I did not run a real WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes for keeping this open and routing it to a narrow bugfix. The safest first fix is prompt/control-token guidance plus regression tests; a new typed delivery mode should be a follow-up only if the narrow fix cannot make explicit delivery intents reliable.
