## Summary

Keep open: current main and the latest release still have the reported source-level path. `sessions_send` marks target turns as inter-session data, has no delivery-intent/mode field, and the A2A runtime suppresses exact skip/silent tokens before visible delivery; the adjacent Telegram/requester-routing issues do not own this WhatsApp delivery-intent precedence bug.

## What This Changes

Request that explicit WhatsApp inter-session delivery requests emit normal group messages instead of being suppressed as `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`.

## Best Possible Solution

Add a narrow `sessions_send`/inter-session delivery-intent guard in prompt/runtime behavior so explicit channel-delivery requests produce a normal allowed message while coordination-only chatter can still use skip tokens.

## Reproduction Assessment

Yes at source level: current main annotates `sessions_send` messages as `inter_session`, gives broad skip/silent exits, and suppresses exact control tokens before gateway delivery. I did not establish a live WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes for keeping this open and routing a narrow bugfix. The first fix should refine the existing prompt/control-token contract with focused regression coverage; a typed delivery mode can remain a follow-up only if the narrow fix is not reliable.
