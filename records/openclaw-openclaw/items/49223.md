## Summary

Keep open. Current main still sends sessions_send payloads through the generic inter-session/A2A prompt path, where skip/silent control tokens remain valid and there is no structured delivery-intent override for explicit normal channel-delivery requests.

## What This Changes

The issue asks OpenClaw to prefer normal outbound channel message emission over REPLY_SKIP, ANNOUNCE_SKIP, or NO_REPLY when a sessions_send inter-session turn is clearly an explicit delivery request.

## Best Possible Solution

Add a backward-compatible delivery-intent guard in the existing sessions_send inter-session/A2A path so explicit normal channel-delivery requests cannot be satisfied by exact skip/silent control tokens, while preserving anti-ping-pong behavior for genuine coordination chatter.

## Reproduction Assessment

Yes at source level: current main deterministically routes sessions_send input as inter-session data through guidance where REPLY_SKIP, ANNOUNCE_SKIP, and NO_REPLY remain valid outcomes and no delivery-intent override exists. A full live WhatsApp reproduction was not performed.

## Solution Assessment

Yes on direction: the narrowest maintainable fix is an internal delivery-intent distinction or guarded prompt/runtime path in the existing sessions_send/A2A flow. A public tool-schema mode should be added only if maintainers want that API surface.
