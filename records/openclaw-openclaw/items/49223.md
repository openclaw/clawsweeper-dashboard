## Summary

Keep open: current main still has the reported source-level path. `sessions_send` wraps the target turn as `inter_session`, gives broad skip/silent-token guidance, has no explicit delivery intent signal, and filters exact control tokens before any visible send; the related requester-routing PR is adjacent but does not resolve this delivery-intent precedence bug.

## What This Changes

The issue asks for explicit WhatsApp `inter_session` channel-delivery requests to emit a normal group message instead of being suppressed as `REPLY_SKIP`, `ANNOUNCE_SKIP`, or `NO_REPLY`.

## Best Possible Solution

Add a narrow `sessions_send`/inter-session delivery-intent guard in prompt/runtime behavior so explicit allowed channel-delivery requests produce normal visible text, while coordination-only chatter can still use skip tokens.

## Reproduction Assessment

Yes, at source level: current main annotates `sessions_send` turns as `inter_session`, encourages exact skip/silent exits, and filters those tokens before gateway delivery. I did not establish a live WhatsApp group repro in this read-only pass.

## Solution Assessment

Yes, a narrow fix in the existing prompt/control-token contract with regression coverage is the best first step. A typed delivery mode can stay a follow-up only if the smaller guard proves unreliable.
