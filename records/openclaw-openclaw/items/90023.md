## Summary

Keep open. Current main and the latest release still run the synchronous zero-hit `memory_search` forced-sync path for QMD, and two open closing PRs now own the concrete fix path.

## What This Changes

Prevent QMD-backed zero-hit `memory_search` calls from running synchronous forced QMD sync/update work inside the interactive tool call.

## Best Possible Solution

Review and land the backend-aware guard from https://github.com/openclaw/openclaw/pull/90030, or replace it with an async/debounced QMD refresh that stays outside the interactive `memory_search` deadline.

## Reproduction Assessment

Yes, at source level: current main still awaits forced sync after zero `memory_search` hits, and QMD sync awaits update work with a longer timeout budget. The reporter also provided live `v2026.6.5` timing evidence, but I did not run a live QMD repro in this read-only pass.

## Solution Assessment

Yes, the best current fix shape is backend-aware zero-hit repair: preserve non-QMD sync-and-retry behavior while preventing QMD update maintenance from running synchronously inside `memory_search`. The linked PRs implement that narrow direction, so this issue should remain open until one is merged or replaced.
