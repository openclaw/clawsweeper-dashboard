## Summary

Keep open. Current main and v2026.5.28 still synchronously force-sync after a zero-hit `memory_search`; for QMD that awaits `qmd update` outside the search timeout, so the reported interactive turn stall remains a real memory-core bug.

## What This Changes

Prevent QMD-backed zero-hit `memory_search` calls from running synchronous forced QMD maintenance in the user-visible tool path.

## Best Possible Solution

Patch `memory_search` so QMD zero-hit results return promptly and any QMD refresh is skipped, debounced, or scheduled outside the tool call, with focused regression coverage for `memory.backend = "qmd"`.

## Reproduction Assessment

Yes at source level: current `memory_search` awaits a forced sync after zero hits, and a QMD manager sync awaits `qmd update`. A live QMD timing run was not performed in this read-only review.

## Solution Assessment

Yes, the best fix direction is backend-aware zero-hit repair: keep any needed non-QMD retry behavior, but do not run QMD forced maintenance synchronously inside the search tool.
