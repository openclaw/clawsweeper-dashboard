## Summary

Keep open: current main and the latest release still run the synchronous zero-hit forced-sync path in `memory_search`, and the linked fix PRs remain open rather than merged.

## What This Changes

QMD-backed `memory_search` zero-hit queries should return quickly instead of synchronously forcing QMD sync/update work inside the interactive tool call.

## Best Possible Solution

Land one focused memory-core fix that keeps QMD zero-hit refresh work out of the blocking `memory_search` hot path, with regression coverage for QMD zero-hit behavior and timeout/cooldown semantics.

Yes from source inspection: current `memory_search` still awaits a forced sync after zero hits, and QMD sync runs update work. I did not run the live QMD stall path in this read-only review.

## Solution Assessment

Yes, the requested direction is the right layer: fix the memory-core tool path so QMD zero-hit searches do not synchronously force maintenance during an interactive turn. The maintainer choice is which open PR shape to land, not whether the issue is still necessary.
