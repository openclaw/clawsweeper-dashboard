## Summary

Keep open: current main and v2026.6.8 still run the synchronous zero-hit forced-sync path for QMD-backed memory_search, while the linked fix PRs remain open and unmerged.

## What This Changes

The issue asks memory-core to stop QMD-backed zero-hit memory_search calls from forcing synchronous QMD sync/update work in the interactive tool path.

## Best Possible Solution

Land one focused memory-core fix that keeps QMD zero-hit refresh work out of the user-visible memory_search deadline, with regression coverage for the QMD zero-hit path and preserved or explicitly bounded non-QMD retry behavior.

## Reproduction Assessment

Yes, from source inspection and reporter live updates. Current main still awaits forced sync after zero-hit memory_search results, and QMD sync invokes update work with a much larger default timeout; I did not run a live large-corpus QMD timing repro in this read-only pass.

## Solution Assessment

Yes, the narrow memory-core direction is the best fit. The fix should remove, bound, or background QMD zero-hit freshness repair in the interactive tool path rather than adding a new user-facing mode.
