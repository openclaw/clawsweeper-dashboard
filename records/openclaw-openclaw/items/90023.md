## Summary

Keep open. Current `main` and the latest release still carry the synchronous zero-hit forced-sync path for QMD-backed `memory_search`, and the linked fix PRs remain open rather than merged.

## What This Changes

The issue reports that QMD-backed zero-hit `memory_search` calls synchronously force a memory sync inside the interactive tool path, causing long stalls and turn timeouts.

## Best Possible Solution

Land one focused memory-core fix that keeps QMD zero-hit refresh work out of the user-visible `memory_search` deadline while preserving intentional non-QMD retry behavior or replacing QMD refresh with an async/debounced path.

## Reproduction Assessment

Yes from source inspection and reporter live updates: current `memory_search` still synchronously forces `manager.sync({ reason: "search", force: true })` after zero hits, and QMD sync waits on update maintenance. I did not run a live large-corpus QMD timing repro in this read-only pass.

## Solution Assessment

Yes, the narrow memory-core fix direction is best: keep the existing retry path for non-QMD managers, but do not put QMD update work in the interactive zero-hit search path. If maintainers want QMD freshness repair preserved, it should be async or debounced outside the tool call.
