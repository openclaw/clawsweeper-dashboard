## Label Justifications

- `P2`: This is a concrete memory-core availability bug with a narrow QMD-specific fix surface and open fix PRs, but it is not a repo-wide outage or security emergency.
- `impact:crash-loop`: The reported synchronous QMD update can make interactive OpenClaw/Codex turns time out or appear hung during `memory_search`.

## Summary

Keep open. Current main and the latest release still have the synchronous zero-hit forced-sync branch in QMD-backed `memory_search`, and open linked PRs are candidate fixes rather than merged implementation.

## What This Changes

The issue asks for QMD-backed zero-hit `memory_search` calls to return promptly instead of synchronously forcing QMD maintenance inside the interactive tool path.

## Best Possible Solution

Land one focused memory-core fix that keeps QMD zero-hit refresh work out of the user-visible `memory_search` deadline while preserving non-QMD retry behavior or moving QMD refresh to an async/debounced path.

## Reproduction Assessment

Yes at source level: current main still awaits a forced manager sync after zero memory hits, and QMD manager sync runs update work with a much larger timeout budget. I did not run a live large-corpus QMD timing reproduction in this read-only review.

## Solution Assessment

Yes, keeping this issue open until a linked fix lands is the best path. The narrow fix belongs in `extensions/memory-core/src/tools.ts`, either by skipping the synchronous QMD zero-hit retry or replacing it with nonblocking refresh semantics.
