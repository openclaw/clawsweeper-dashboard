## Summary

Keep open: current main and v2026.6.9 still run the synchronous zero-hit forced-sync path for memory_search, while the direct closing PRs remain open and unmerged.

## What This Changes

The issue reports that QMD-backed zero-hit memory_search calls synchronously force a memory sync and can stall an interactive turn.

## Best Possible Solution

Land one focused memory-core fix that keeps QMD zero-hit refresh work out of the user-visible memory_search deadline while preserving intentional non-QMD retry behavior or replacing the retry with a bounded/background refresh path.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.9 still await a forced sync after a zero-hit memory_search result, and QMD sync awaits update maintenance. I did not run a live large-corpus QMD timing repro in this read-only review.

## Solution Assessment

Yes, the narrow memory-core direction is the best fit: remove, bound, or background QMD zero-hit sync work in the interactive search path without adding new config surface.
