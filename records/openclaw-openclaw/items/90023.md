## Summary

Keep open: current main and the latest release still run the synchronous zero-hit forced-sync path for QMD-backed memory_search, and the linked closing PRs remain open rather than merged.

## What This Changes

The issue reports that QMD-backed zero-hit memory_search calls synchronously force a memory sync inside the interactive tool path, causing stalls or timeouts.

## Best Possible Solution

Land one focused memory-core fix that keeps QMD zero-hit refresh work out of the user-visible memory_search deadline while preserving intentional non-QMD retry behavior or replacing it with a bounded asynchronous refresh path.

## Reproduction Assessment

Yes from source inspection and reporter live updates: current main still synchronously forces manager.sync({ reason: "search", force: true }) after zero-hit memory_search results, and QMD sync waits on update maintenance. I did not run a live large-corpus QMD timing repro in this read-only pass.

## Solution Assessment

Yes, the narrow memory-core direction is the best fit: avoid synchronous QMD update work in the interactive zero-hit search path, with any QMD freshness repair moved to a bounded, async, or debounced path.
