## Summary

Current main and v2026.6.5 still synchronously force a full QMD update after a zero-hit search, so the reported interactive stall remains necessary to fix. Keep the issue open while maintainers select and land the proof-positive linked fix; https://github.com/openclaw/openclaw/pull/90030 is clean and has supplied real QMD behavior proof, while https://github.com/openclaw/openclaw/pull/90117 lacks that proof.

## What This Changes

Prevent QMD-backed zero-hit `memory_search` calls from synchronously forcing index maintenance and retrying inside an interactive turn.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/90030 after maintainer acceptance of bounded QMD search latency, preserving non-QMD retry coverage and keeping QMD refresh in existing non-interactive update paths.

## Reproduction Assessment

Yes. Current main and v2026.6.5 deterministically await QMD update work after zero hits, and the reporter supplied live release-level before/after timing; this review did not independently run QMD.

## Solution Assessment

Yes. A QMD-specific guard is the narrowest maintainable fix because it removes maintenance from the interactive hot path, preserves generic backend retry behavior, and avoids a new configuration surface.
