The issue asks OpenClaw to stop QMD-backed `memory_search` zero-hit results from triggering synchronous forced QMD maintenance inside the interactive tool call.

## Best Possible Solution

Review and land one narrow memory-core fix that skips or schedules QMD zero-hit refresh outside the interactive `memory_search` path while preserving non-QMD retry behavior and adding focused regression coverage.

## Solution Assessment

Yes, the best direction is a backend-aware zero-hit repair: do not run QMD forced maintenance synchronously inside the search tool, while keeping any intended non-QMD retry behavior. The open linked PR with real QMD proof appears to be the strongest current review target.
