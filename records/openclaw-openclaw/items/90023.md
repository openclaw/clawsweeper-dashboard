## Summary

Keep open. Current main and the latest release still run the synchronous zero-hit `memory_search` forced-sync path, and the two open closing PRs are the right place to resolve the issue before closing it.

## What This Changes

The issue reports that QMD-backed `memory_search` zero-hit queries synchronously force a QMD sync/update and can stall interactive turns.

## Best Possible Solution

Review and land one of the linked fix PRs, preserving non-QMD zero-hit retry behavior while moving QMD refresh out of the user-visible `memory_search` deadline.

## Reproduction Assessment

Yes, at source level: current main still awaits a forced sync after zero `memory_search` hits, and QMD sync awaits update work with a much longer timeout budget. The reporter also provided live v2026.6.5 timing evidence, but I did not run a live QMD timing repro in this read-only pass.

## Solution Assessment

Yes. The best fix shape is backend-aware zero-hit repair: keep non-QMD sync-and-retry behavior while preventing QMD update maintenance from running synchronously inside `memory_search`.
