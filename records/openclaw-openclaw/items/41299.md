## Summary

Keep this PR open, but it is not merge-ready: the reported metadata/user-message ambiguity is plausible, but the branch inserts the marker through the generic current-inbound prompt combiner, so compact current-message and room-event contexts can persist or replay `---openclaw:user-msg---` instead of having it stripped. The submitted proof is helper-level terminal output, not a real channel/provider path.

## What This Changes

The PR inserts a `---openclaw:user-msg---` delimiter when combining current inbound context with the user prompt and teaches inbound-metadata sanitizers to remove that delimiter, with separator-focused tests.

## Best Possible Solution

Tie any delimiter to generated inbound metadata at a structured boundary that all sanitizer and replay paths can remove, then prove the real channel/provider flow does not leak the marker or lose user text.

## Reproduction Assessment

Yes from source inspection, not by executing the branch. Current tests and prompt-prelude code show non-metadata current-inbound contexts, while the PR inserts the marker for every prefix and only strips a leading marker after recognized metadata removal.

## Solution Assessment

No, not as implemented. The separator idea may be a reasonable mitigation, but applying it in the generic current-inbound prompt combiner is too broad; the safer fix needs a structured/generated-metadata boundary and matching sanitizer coverage.
