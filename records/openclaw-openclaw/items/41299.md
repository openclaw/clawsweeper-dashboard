## Label Justifications

- `P2`: This is a bounded agent prompt/session bugfix with meaningful model-comprehension impact, but it is not an emergency or broad runtime outage.
- `merge-risk: 🚨 compatibility`: The PR changes the generic current-inbound prompt join contract by inserting a new marker into contexts that existing callers and tests treat as plain prompt text.

## Summary

Keep open: the separator mitigation is not merge-ready because it changes shared current-inbound prompt assembly beyond metadata blocks and can leak the marker on room-event and compact current-message paths; overlapping hidden-runtime-context PRs are better candidates but not all merged or clean.

## What This Changes

The PR inserts `---openclaw:user-msg---` between current inbound context and the active prompt, strips that marker from inbound-metadata sanitizer paths, and adds separator tests.

## Best Possible Solution

Land one canonical hidden-runtime-context path that keeps inbound metadata available to the model without storing it as user-authored prompt text; if this branch continues, constrain and prove the delimiter only on metadata shapes.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the source-level prompt shape: current main and `v2026.6.8` inline `currentInboundContext` into session/model prompts, but the intermittent live model "message is empty" symptom was not reproduced here.

## Solution Assessment

Is this the best way to solve the issue? No. A delimiter is a plausible mitigation, but hidden runtime-context routing is the cleaner owner-boundary fix because it removes generated metadata from user-authored prompt/session text instead of adding another visible prompt token.
