## Summary

Keep open: current main still has the inline inbound-metadata prompt problem, but this PR is not merge-ready because it changes a shared current-inbound helper and would insert the marker into non-metadata `Current message` and room-event prompt paths.

## What This Changes

The PR adds an `---openclaw:user-msg---` delimiter between current inbound context and prompt text, strips that marker from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Use one prompt-contract direction for the cluster: preferably move inbound metadata into hidden runtime context, or if this delimiter branch continues, gate it to real metadata prefixes and prove Telegram reply, room-event, and sanitizer paths.

## Reproduction Assessment

Yes for the patch regression: source inspection shows `buildCurrentInboundPrompt` is shared by metadata, `Current message`, and room-event paths, and the PR adds the marker for all non-empty contexts. The original intermittent provider-level empty-message symptom was not live-reproduced in this read-only review.

## Solution Assessment

No. The delimiter is a plausible mitigation for fenced metadata, but adding it in the shared helper is not the best fix because it changes non-metadata prompt contracts; the safer direction is hidden runtime context or a metadata-only delimiter gate.
