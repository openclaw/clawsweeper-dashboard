## Summary

Keep open: the PR targets a real prompt-boundary problem, but the patch is not merge-ready because it adds the marker in a shared helper used by non-metadata current-message and room-event contexts. The stripper only removes a leading marker after metadata stripping, so marker leakage and session-prompt contamination remain possible.

## What This Changes

The PR inserts a `---openclaw:user-msg---` delimiter when current inbound context is joined with the user prompt and adds sanitizer tests/helpers to remove it from metadata-stripped text.

## Best Possible Solution

Preserve current non-metadata prompt joins and either move generated inbound metadata into hidden runtime context or narrowly delimit and strip only proven metadata prefixes with regression coverage across embedded, CLI, current-message, and room-event paths.

## Reproduction Assessment

Yes. Source inspection of the potential merge commit shows the marker is inserted for all nonempty current-inbound contexts, and unchanged current tests cover `Current message` and room-event joins that do not expect that marker; I did not run tests because this review is read-only.

## Solution Assessment

No. A delimiter may be a plausible mitigation for leading metadata blocks, but adding it in the shared helper is too broad; the safer direction is a metadata-only delimiter with full sibling coverage or the hidden-runtime-context path tracked by the related PR.
