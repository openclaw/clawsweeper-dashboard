## Summary

Keep open, but the current patch is not merge-ready: it changes a shared prompt helper so non-metadata current-turn contexts can receive an internal separator marker, while current tests and sibling runtime paths still expect plain joins. The related hidden-runtime-context direction in https://github.com/openclaw/openclaw/pull/89428 looks like a stronger path for the broader metadata-in-prompt problem, but this PR still needs maintainer review or contributor revision rather than cleanup closure.

## What This Changes

The PR adds `---openclaw:user-msg---` during current inbound prompt composition and strips that marker after inbound metadata removal, with focused strip tests.

## Best Possible Solution

Keep generic current-turn prompt composition unchanged for non-metadata contexts, and either gate any separator to generated metadata prefixes or land the hidden runtime-context direction from https://github.com/openclaw/openclaw/pull/89428.

## Reproduction Assessment

Yes for the PR regression: source inspection plus existing `runtime-context-prompt.test.ts` assertions show non-metadata `promptJoiner` and room-event contexts should join without an internal marker. I did not establish the original intermittent live LLM “message is empty” symptom in a provider run.

## Solution Assessment

No. The separator is a plausible mitigation for metadata ambiguity, but placing it in the shared prompt helper is too broad; the safer fix is to keep marker logic on the generated metadata boundary or route metadata through hidden runtime context.
