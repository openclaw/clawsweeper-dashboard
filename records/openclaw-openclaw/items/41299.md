## Summary

Keep open: the PR still has a merge-blocking prompt/session regression because the marker is inserted for every current-inbound context, not only generated metadata, and the better hidden-runtime-context direction is still only tracked by an open conflicting PR.

## What This Changes

The PR adds a generated `---openclaw:user-msg---` separator between inbound metadata and user content, strips that marker from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Resolve the prompt-boundary direction first: either move generated inbound metadata into hidden runtime context as in the canonical open work, or narrowly gate any delimiter to metadata-only writers and prove it cannot leak into CLI, room-event, replay, TUI, gateway, and session-cost surfaces.

## Reproduction Assessment

Yes for the PR regression: source inspection shows non-metadata currentInboundContext values flow through the changed helper and would receive the marker, while the sanitizers no-op for non-sentinel text. I did not run tests because this review was read-only.

## Solution Assessment

No. The inline marker is only a plausible mitigation, and this diff applies it too broadly; the safer end state is a metadata-only contract or the hidden runtime-context approach already tracked in the related open work.
