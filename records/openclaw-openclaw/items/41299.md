## Summary

Keep open: the PR is a plausible mitigation for inline inbound metadata, but the current patch changes the shared current-inbound prompt helper for non-metadata CLI, compact current-message, and room-event contexts too, so it is not merge-ready as written.

## What This Changes

The PR adds a `---openclaw:user-msg---` delimiter when joining current inbound context with user prompt text, strips that delimiter after inbound metadata removal, and adds separator stripping tests.

## Best Possible Solution

Use one owner-approved prompt contract: either move inbound metadata into hidden runtime context, or narrowly gate any delimiter to actual metadata-block contexts and prove the CLI, compact current-message, room-event, and Telegram-visible paths.

## Reproduction Assessment

Yes for the source-level prompt-shape behavior: current main joins currentInboundContext and prompt text directly, and the PR changes that join globally. The intermittent live LLM “message is empty” symptom was not reproduced here.

## Solution Assessment

No. The delimiter is a plausible mitigation for metadata blocks, but this helper is broader than metadata and the safer end state is either hidden runtime-context routing or a metadata-only delimiter with proof for sibling callers.
