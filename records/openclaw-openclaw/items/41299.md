## Summary

Keep open: current main has not solved the metadata/user-message ambiguity, but this PR is not safe to merge because it adds the marker in a generic current-inbound prompt helper used by non-metadata room-event and compact-current-message paths too.

## What This Changes

The PR inserts `---openclaw:user-msg---` between current inbound context and the active prompt, then strips that marker from inbound metadata sanitizers with separator-focused tests.

## Best Possible Solution

Move current inbound metadata into hidden runtime context, or narrowly gate this branch so the marker is emitted only for generated inbound metadata and prove the compact current-message, room-event, CLI, and sanitizer paths stay clean.

## Reproduction Assessment

Yes. Source inspection shows the patched helper is shared by metadata and non-metadata current inbound contexts, and current main tests pin marker-free joins for compact current-message and room-event prompts.

## Solution Assessment

No. The separator is a plausible mitigation for generated metadata blocks, but inserting it in the generic prompt helper is too broad; hidden runtime-context routing or metadata-only gating is the safer fix.
