## Summary

Keep open: this branch is not merge-ready because the separator is added by a shared prompt helper for non-metadata current-inbound contexts, while the strip path only handles metadata-derived leading markers. The same root problem is better handled by the open hidden-runtime-context direction in https://github.com/openclaw/openclaw/pull/89428.

## What This Changes

The PR inserts `---openclaw:user-msg---` between current inbound context and prompt text, adds separator stripping, and extends inbound metadata stripping tests.

## Best Possible Solution

Move current inbound metadata into hidden runtime context, or at minimum gate this delimiter to contexts proven to be metadata and cover every current-inbound caller path before merge.

## Reproduction Assessment

Yes for source-level behavior: current main joins `currentInboundContext` with the active prompt, and the PR head inserts a marker for all non-empty context prefixes. The intermittent model-level “message is empty” symptom was not reproduced in a live provider run during this read-only review.

## Solution Assessment

No. The separator is a plausible mitigation for metadata parsing, but the best current fix is to keep generated inbound metadata out of user-authored prompt/session text and carry it through hidden runtime context.
