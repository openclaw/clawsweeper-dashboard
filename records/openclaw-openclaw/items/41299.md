## Summary

Keep open: the direction is plausible, but the current PR head still introduces user-content stripping, lacks real behavior proof, and needs rebase around the current `currentTurnContext` prompt path.

## What This Changes

The PR adds a generated `---openclaw:user-msg---` separator between inbound metadata and user text, strips it from metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Rebase onto the current `currentTurnContext` path, add the boundary only where runtime context is attached to the model prompt, strip only the generated adjacent boundary, align prompt fixtures, and provide redacted real runtime proof.

## Reproduction Assessment

Yes for the PR regression: the helper regex can be exercised directly and deletes later user-authored marker text. No for the original live-provider false-empty symptom, which the PR body explicitly did not verify end to end.

## Solution Assessment

No. A clearer boundary may be the right fix, but this implementation strips too broadly and is stale against the current prompt-context architecture.
