## Summary

I found the same core blocker as the prior review: the PR is trying to address a real inbound-metadata comprehension problem, but the patch changes a shared current-inbound helper and can inject `---openclaw:user-msg---` into compact current-message, room-event, and CLI prompt/session text where the sanitizer cannot reliably remove it. The cleaner hidden-runtime-context direction remains tracked by the canonical issue and open candidate PRs, so this PR should stay open only for maintainer decision or a substantial repair, not close as superseded yet.

## What This Changes

The PR adds an `---openclaw:user-msg---` marker when joining current inbound context with the current prompt, strips that marker from inbound-metadata sanitizers, and adds separator-focused sanitizer tests.

## Best Possible Solution

Move current inbound metadata into hidden runtime context so user-authored prompt/session text stays clean, or explicitly narrow this delimiter branch to metadata-only contexts with regression proof across embedded, CLI, room-event, and sanitizer paths.

## Reproduction Assessment

Yes. ## Solution Assessment

No. The separator is a plausible mitigation for one metadata-block shape, but the shared helper also handles compact and room-event contexts; the more maintainable fix is the hidden-runtime-context route or a much narrower gated delimiter with full caller proof.
