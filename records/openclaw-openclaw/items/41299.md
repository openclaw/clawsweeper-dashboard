## Summary

Keep open. Current main still joins inbound metadata and user text with blank lines, so the reported problem is not implemented there, and the PR is useful but not merge-ready because the patch leaves the media-without-caption path on the old ambiguous join and globally strips later user-authored marker lines once any metadata was removed.

## What This Changes

The PR changes auto-reply prompt construction to insert an internal `---openclaw:user-msg---` separator between inbound metadata and user text, adds sanitizer removal for that marker, and adds separator-focused sanitizer tests.

## Best Possible Solution

Keep the PR open but rework it around one separator-aware prompt-body composition helper used by normal text, media-only fallback, and prompt fixtures. The sanitizer should remove only the synthetic first boundary adjacent to stripped OpenClaw metadata, preserve later user-authored marker text, and include focused regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. The current-main issue is reproducible by tracing a metadata-bearing turn through `buildInboundUserContextPrefix` into the blank-line join in `runPreparedReply`; the PR regression is reproducible from the supplied diff with metadata followed by user text that later contains a `---openclaw:user-msg---` line.

## Solution Assessment

No. The separator direction is reasonable, but this patch is not the best implementation because composition remains split across prompt paths and sanitizer removal is broader than the injected boundary; a shared composer plus first-boundary-only stripper is the safer path.
