## Summary

Keep open: source inspection shows this branch still has a blocking prompt/session bug because the marker is added in a shared helper used by non-metadata current-message and room-event contexts, not only fenced inbound metadata.

## What This Changes

The PR inserts an `---openclaw:user-msg---` separator between inbound metadata and the active user prompt, strips that marker from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Settle one prompt-boundary design before merge: preferably move current inbound metadata into hidden runtime context, or narrowly gate this separator to actual metadata prefixes with coverage for compact current-message, room-event, CLI, and display/history sanitizer paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: pass a non-metadata `currentInboundContext` such as `Current message:` with `promptJoiner: " "` through the patched `buildCurrentInboundPrompt`, and the marker is inserted even though no metadata stripper will remove it.

## Solution Assessment

No. A marker can be an acceptable mitigation only if scoped to actual inbound metadata, while the cleaner direction is the hidden-runtime-context path that removes generated metadata from user-authored prompt/session text entirely.
