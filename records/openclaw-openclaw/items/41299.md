## Summary

Keep open: current main has not implemented this delimiter, and the PR addresses a plausible prompt-boundary bug, but the latest patch still has a concrete sanitizer regression that can delete later user-authored marker text after metadata stripping.

## What This Changes

The PR injects `---openclaw:user-msg---` between inbound metadata and user text, strips that marker from sanitized user-facing text, and adds separator-focused metadata stripping tests.

## Best Possible Solution

Keep the delimiter approach only after the stripper removes exactly the synthetic metadata boundary, preserves user-authored marker text, adds regression coverage and a changelog entry, then reruns the focused auto-reply checks.

## Reproduction Assessment

Yes for the PR regression: the proposed global regex will remove later `---openclaw:user-msg---` lines from sanitized user text once any metadata block was stripped. The original LLM parsing symptom is source-plausible but not deterministically live-reproduced in the provided evidence.

## Solution Assessment

No, not yet. The separator direction is narrow and maintainable, but the current stripper is broader than the injected boundary; a safer repair should remove only the first separator associated with stripped metadata and preserve later body content.
