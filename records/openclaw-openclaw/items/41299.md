## Summary

Keep open: the underlying inline-metadata problem is real, but this PR is not merge-ready because it inserts the marker in the shared current-inbound prompt helper even for non-metadata contexts, and the canonical hidden-runtime-context fix path is still open.

## What This Changes

The branch adds an `---openclaw:user-msg---` delimiter when `buildCurrentInboundPrompt` combines current inbound context with a prompt, strips that delimiter after inbound metadata sanitization, and adds separator tests.

## Best Possible Solution

Land one canonical hidden-runtime-context route for current-turn metadata, or narrowly gate any delimiter to generated inbound metadata with runner/CLI/room-event coverage and real transport proof before merge.

## Reproduction Assessment

Yes, at source level. Current main still prepends `currentInboundContext` into session/model prompt strings, and the PR's unconditional helper change can be traced into non-metadata `Current message:` and room-event contexts without running a live provider.

## Solution Assessment

No. A delimiter is only a plausible mitigation; the current patch applies it at the wrong shared boundary, while the hidden-runtime-context direction keeps generated metadata out of user-authored prompt/session text entirely.
