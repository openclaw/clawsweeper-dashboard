## Summary

Keep open. Current main still has the reported ambiguous metadata/user-body join, and the PR is a useful implementation candidate, but the proposed patch is not merge-ready because it can delete user-authored marker text and does not cover the current media-without-caption prompt path.

## What This Changes

The PR changes auto-reply prompt construction to insert `---openclaw:user-msg---` between inbound metadata and user text, adds sanitizer stripping for that marker, and adds separator-focused sanitizer tests.

## Best Possible Solution

Keep the PR open and rework it around a single prompt-body composition helper used by normal text, media-without-caption fallback, raw-body tests, and prompt fixtures. The sanitizer should remove only the synthetic separator adjacent to stripped OpenClaw metadata while preserving later user-authored marker text, and the contract should cover all generated metadata labels that can be stripped from user-facing views.

## Reproduction Assessment

Yes. The current-main issue is reproducible by following `runPreparedReply` from `buildInboundUserContextPrefix` into the blank-line `baseBodyForPrompt` and media fallback joins; the PR regression is reproducible from the diff with a metadata block followed by user text containing a later `---openclaw:user-msg---` line.

## Solution Assessment

No. The separator direction is reasonable, but this patch is not the narrowest maintainable fix because composition remains split across paths and the global stripper removes more than the injected boundary. A shared composer plus a first-boundary-only stripper would be safer.
