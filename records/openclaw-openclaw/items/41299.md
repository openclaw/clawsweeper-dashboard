## Summary

Keep open: the PR still has a blocking prompt/session correctness issue because it inserts the marker in the generic current-inbound prompt helper, while current main uses that helper for non-metadata contexts such as Telegram compact reply context and room events. The related hidden-runtime-context PR is the better canonical direction, but it is still open and live GitHub reports it as conflicting, so this branch is not safely closeable as superseded yet.

## What This Changes

The PR inserts `---openclaw:user-msg---` between current inbound context and prompt text, adds separator stripping to inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Prefer the hidden-runtime-context direction from https://github.com/openclaw/openclaw/pull/89428; if this delimiter mitigation continues, gate it to actual metadata-only shapes and prove every buildCurrentInboundPrompt caller.

## Reproduction Assessment

Yes for the patch regression: source inspection shows `Current message:` and room-event currentInboundContext shapes flow through the same helper and would receive the marker. The original intermittent provider-level “message is empty” symptom was not reproduced in a live model run during this read-only review.

## Solution Assessment

No. The separator is a plausible mitigation, but placing it in the generic helper is too broad; hidden runtime context is the cleaner fix for generated metadata, and any delimiter fallback must be metadata-specific.
