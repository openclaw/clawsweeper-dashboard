## Summary

Keep open: the PR is a plausible mitigation for ambiguous inline metadata, but the current patch changes a shared prompt helper in a way that also inserts the internal marker into non-metadata `Current message`, room-event, and CLI prompt joins that current tests intentionally keep marker-free.

## What This Changes

The PR adds `---openclaw:user-msg---` between current inbound context and the user prompt, then strips that marker from metadata-sanitized display paths.

## Best Possible Solution

Prefer the hidden-runtime-context direction for generated channel metadata; if this delimiter approach continues, gate it to metadata-only contexts and add coverage for current-message, room-event, CLI, and sanitizer edge cases before merge.

## Reproduction Assessment

Yes. Source inspection shows the changed helper is shared by metadata, current-message, room-event, and CLI prompt paths, and current tests pin marker-free joins for those non-metadata contexts.

## Solution Assessment

No. A delimiter is only a partial mitigation; the cleaner solution is to keep generated metadata out of user-authored prompt/session text or narrowly gate this marker to metadata-only contexts.
