## Summary

Keep open: the change targets a plausible metadata/user-message ambiguity, but the current branch is conflicting with main, still has a source-proven sanitizer regression, and lacks required real provider proof.

## What This Changes

The PR adds an `---openclaw:user-msg---` delimiter between inbound metadata and user text, strips it from metadata sanitizers, and adds separator tests.

## Best Possible Solution

Rebase onto the current `currentTurnContext` / `buildCurrentTurnPrompt` flow, add the delimiter at the actual model prompt boundary, strip only the adjacent generated marker from visible/history views, align prompt-composition coverage, and add redacted live-provider proof.

## Reproduction Assessment

Yes for the PR-introduced sanitizer regression: source inspection shows the global regex can remove later user-authored marker lines after metadata is stripped. No for the original false empty-message symptom because the PR body says live end-to-end provider behavior was not verified.

## Solution Assessment

No. A delimiter may be the right direction, but this branch needs to be rebased onto current main and must remove only the generated adjacent marker before it is a maintainable fix.
