## Summary

Keep open: the branch still has a blocking runtime/session regression because the marker is inserted by a shared current-context helper, so Telegram inline current-message contexts can retain `---openclaw:user-msg---` after metadata stripping; the proof is also helper-level terminal output rather than a real metadata-bearing channel/provider turn.

## What This Changes

The PR adds a `---openclaw:user-msg---` delimiter to current inbound prompt assembly and strips that delimiter in inbound metadata sanitizers, with separator tests.

## Best Possible Solution

Move the delimiter ownership to the metadata-producing boundary, or make the prefix structurally distinguish metadata from preserved current-message context, then prove injection and stripping across metadata-only, inline reply, room-event/resumable, CLI, TUI, gateway, and session-replay paths.

## Reproduction Assessment

Yes, source-level: combine the PR's changed `buildCurrentInboundPrompt()` with a Telegram inline reply context where `buildInboundUserContextPrefix()` appends `Current message:` after metadata and `resolveInboundUserContextPromptJoiner()` returns a single space; metadata stripping then leaves the marker after the preserved inline context.

## Solution Assessment

No. The separator idea is plausible, but putting it in the shared current-context combiner is too broad; the safer fix is to place or track the boundary inside the metadata-owned path and cover the sibling context shapes.
