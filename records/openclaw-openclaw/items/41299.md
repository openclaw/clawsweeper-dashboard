## Summary

Keep this PR open, but the current patch is not merge-ready: it adds the marker in a generic current-context prompt helper used by CLI and room-event contexts, so the marker can enter prompts that are not metadata-prefix-plus-user-body prompts.

## What This Changes

The PR adds a generated `---openclaw:user-msg---` separator when current inbound context is joined with the user prompt, strips that marker from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Keep the PR open, but move marker insertion/stripping to an explicit generated-inbound-metadata boundary or add a context kind that preserves the generic helper's existing CLI and room-event semantics.

## Reproduction Assessment

Yes for the patch regression: current source and existing tests show `buildCurrentInboundPrompt()` is a shared helper for CLI and room-event contexts. The original intermittent provider-level “message is empty” symptom still lacks a high-confidence live reproduction.

## Solution Assessment

No. The current implementation is a plausible mitigation for metadata ambiguity, but the best fix needs a metadata-specific insertion point or explicit context kind instead of changing the generic current-context join helper.
