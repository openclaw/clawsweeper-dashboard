## Summary

Keep open: the PR addresses a real prompt-clarity problem, but the current branch changes a shared prompt join helper for every current inbound context, not only fenced metadata, so it can inject the internal marker into Telegram reply-context and room-event prompts where the sanitizer will not remove it.

## What This Changes

The PR inserts `---openclaw:user-msg---` between current inbound context and prompt text, strips that marker after inbound metadata stripping, and adds separator-focused sanitizer tests.

## Best Possible Solution

Either keep this PR narrow by gating the marker to actual fenced inbound metadata contexts and adding tests for non-metadata contexts, or replace this delimiter approach with the hidden runtime-context direction once that path is cleanly mergeable.

## Reproduction Assessment

Yes, source-reproducible: applying the PR’s helper change to the existing `Current message` test input makes the joined prompt include `---openclaw:user-msg---`, and current main tests document the expected marker-free join. I did not run tests because this review is read-only and the checkout must remain clean.

## Solution Assessment

No. The delimiter is a plausible mitigation for fenced metadata, but the implementation is too broad for the shared helper; the safer solution is metadata-only gating with sibling-context tests or the hidden runtime-context approach.
