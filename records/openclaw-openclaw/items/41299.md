## Summary

Keep open: the separator idea may address a real metadata/prompt ambiguity, but this patch is not merge-ready because it inserts the marker through a shared prompt helper that also serves room-event and CLI current-context prompts, and the supplied proof does not exercise a real metadata-bearing channel/provider turn.

## What This Changes

The PR injects `---openclaw:user-msg---` when `buildCurrentInboundPrompt()` combines a context prefix and prompt, strips that marker from inbound metadata sanitizers, and adds separator tests.

## Best Possible Solution

Gate the separator to generated inbound metadata contexts, preserve the existing generic prompt-join contract for CLI and room-event contexts, add sibling coverage, and require real channel/provider proof before merge.

## Reproduction Assessment

Yes for the patch regression: source and existing tests show `buildCurrentInboundPrompt()` is shared by CLI and room-event contexts that are not simple metadata prefixes. No high-confidence reproduction was provided for the original intermittent provider-level “message is empty” symptom.

## Solution Assessment

No. The current diff is a plausible metadata-prefix mitigation, but the best fix must scope marker insertion to generated inbound metadata or an explicit context kind instead of changing the generic current-context join helper.
