## Summary

Keep open, but this PR is not merge-ready: the delimiter is added in a shared current-inbound prompt helper, so it also changes non-metadata `Current message` and room-event prompt/session text while leaving related sibling runtime behavior split.

## What This Changes

The branch adds an `---openclaw:user-msg---` delimiter between current inbound context and prompt text, strips that delimiter from inbound metadata sanitizers, and adds separator tests.

## Best Possible Solution

Adopt one owner-approved current-inbound prompt contract across embedded and Codex-style runtimes, preferably the hidden-runtime-context direction for generated metadata; if a delimiter remains, gate it to actual fenced metadata and prove no visible or persisted marker leakage.

Yes, at source level: current main joins generated inbound context and user prompt with only the configured joiner, and the PR body describes local terminal proof for the delimiter path. I did not run a live provider or Telegram reproduction in this read-only review.

## Solution Assessment

No. The delimiter is a plausible mitigation for fenced metadata, but this implementation is too broad for the shared helper, one-sided against sibling runtimes, and should wait for a maintainer prompt-contract choice against the hidden-runtime-context path.
