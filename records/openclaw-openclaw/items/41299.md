## Summary

Keep open: current main still uses the ambiguous blank-line join, but this PR is not mergeable as-is because the proposed global separator stripping can remove later user-authored marker text and the prompt-composition fixtures remain out of sync with runtime prompts.

## What This Changes

The PR adds an `---openclaw:user-msg---` delimiter between inbound metadata and the user body, strips that delimiter from user-facing metadata sanitizers, and adds separator regression tests.

## Best Possible Solution

Keep the separator fix narrow: remove only the synthetic delimiter paired with stripped inbound metadata, preserve later marker text in the real body, align prompt-composition helpers with runtime prompt construction, and add the required changelog entry before merge.

## Reproduction Assessment

Yes for the PR regression: the added helper regex can be exercised directly and removes later `---openclaw:user-msg---` text from a normal body. The original LLM false-empty behavior is source-plausible from current prompt construction but was not live-reproduced.

## Solution Assessment

No. Adding a boundary marker is a reasonable direction, but the current global post-processing is too broad; the safer fix should consume only the marker associated with stripped metadata and keep tests aligned with runtime prompt assembly.
