## Best Possible Solution

Land one canonical inbound-metadata boundary that keeps generated channel context out of user-authored prompt/session text, or narrow this separator branch to metadata-only contexts with regression proof for the shared helper callers.

## Reproduction Assessment

Yes for the PR regression: source and tests show `buildCurrentInboundPrompt` is shared by metadata and non-metadata current inbound contexts. The original intermittent model “message is empty” symptom has source support but was not live-provider reproduced in this read-only review.

## Solution Assessment

No. The separator is a plausible mitigation for metadata-only prompts, but placing it in the shared helper is too broad; the safer path is metadata-only guarding or the hidden runtime-context fix already represented by the related PR.
