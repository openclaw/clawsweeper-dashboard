## Summary

Keep open: the delimiter idea targets a real inline-metadata problem, but the patch changes the shared current-inbound prompt helper and therefore injects the marker into CLI current-message and room-event contexts that are not metadata sanitizer paths.

## What This Changes

The PR adds an `---openclaw:user-msg---` delimiter after any current inbound context, strips that marker from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Keep this PR open for now, but resolve the prompt-contract direction first; the durable fix is either a metadata-only delimiter with full sibling-path proof or the hidden-runtime-context path already tracked by the overlapping PRs.

## Reproduction Assessment

Yes for the patch regression: source inspection shows the changed helper is shared by metadata, `Current message`, CLI, and room-event contexts. The original intermittent provider-level empty-message symptom was not live-reproduced in this read-only review.

## Solution Assessment

No. The delimiter is a plausible mitigation for inline metadata, but placing it in the shared helper is too broad; a metadata-specific formatter or hidden-runtime-context routing is safer.
