## Summary

Keep open: the branch is not merge-ready because the delimiter is added in a shared prompt helper that also handles non-metadata current-context paths, so it can leak the internal marker into model, session, or user-visible text.

## What This Changes

The PR adds an `---openclaw:user-msg---` delimiter when current inbound context is joined to the active prompt, strips that delimiter after inbound metadata sanitization, and adds separator-focused tests.

## Best Possible Solution

Use the hidden-runtime-context direction or a metadata-owned trusted decoration contract so current inbound metadata remains model-available without becoming user-authored prompt/session text.

## Reproduction Assessment

Yes at source level. The changed helper is used for all `CurrentInboundPromptContext`, while current source produces non-metadata `Current message:` and room-event contexts that are not recognized by `stripInboundMetadata`; tests were not run because this review is read-only.

## Solution Assessment

No. The delimiter is a plausible mitigation for fenced metadata blocks, but the best fix is to route generated inbound context through hidden runtime context or a trusted decoration contract instead of adding a visible marker in the generic prompt joiner.
