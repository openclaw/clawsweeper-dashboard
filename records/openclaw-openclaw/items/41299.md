## Summary

Keep open: the bug-fix intent is useful, but the branch is not merge-ready because the marker is injected through a shared prompt helper that also handles non-metadata Telegram reply and room-event contexts.

## What This Changes

The branch inserts an `---openclaw:user-msg---` marker between current inbound context and prompt text, strips it from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Choose one prompt contract: preferably move generated inbound metadata into hidden runtime context, or narrow this delimiter so it applies only to actual metadata prefixes with regression proof for metadata, `Current message`, and room-event paths.

## Reproduction Assessment

Yes for the patch regression: source inspection shows `buildCurrentInboundPrompt` is shared by metadata, Telegram `Current message`, and room-event paths, and the PR adds the marker for all non-empty contexts. The original intermittent provider-level empty-message symptom was not live-reproduced in this read-only review.

## Solution Assessment

No. A delimiter is a plausible mitigation for fenced metadata, but this implementation is too broad; the safer direction is hidden runtime context or a metadata-only marker with tests and proof for sibling prompt paths.
