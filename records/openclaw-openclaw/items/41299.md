## Summary

Keep open: the delimiter idea targets a real metadata/user-text ambiguity, but the current patch applies it inside a shared prompt helper and would inject `---openclaw:user-msg---` into non-metadata current-message, room-event, and CLI prompt paths.

## What This Changes

The PR adds an `---openclaw:user-msg---` delimiter to `buildCurrentInboundPrompt()` output when current inbound context and prompt are both present, plus sanitizer logic and tests to remove it after metadata stripping.

## Best Possible Solution

Either gate delimiter insertion to a typed/generated metadata context and update all sibling prompt tests, or consolidate this fix behind the hidden-runtime-context approach that removes metadata from user-authored prompt/session text.

## Reproduction Assessment

Yes for source-level reproduction: the PR merge ref shows the helper always injecting the marker for any context, while merge-ref tests still expect direct joins for non-metadata contexts. I did not run tests because this review is read-only.

## Solution Assessment

No. A delimiter may be a useful mitigation, but applying it in the shared helper without a metadata-only contract is too broad; the safer fix is a typed/generated metadata boundary or the hidden-runtime-context path.
