## Summary

Keep open: the useful metadata/user-message ambiguity is real, and prior review feedback improved this branch, but the current patch is not landable because it changes the shared current-context prompt helper for non-metadata contexts and leaves the canonical hidden-runtime-context direction unresolved.

## What This Changes

The branch inserts an `---openclaw:user-msg---` delimiter when `buildCurrentInboundPrompt` joins current inbound context with the user prompt, strips that delimiter after inbound metadata sanitization, and adds separator tests.

## Best Possible Solution

Land one canonical prompt contract that keeps generated inbound metadata out of user-authored prompt/session text, or narrowly gate this delimiter to actual metadata-only contexts with matching runner, CLI, sanitizer, and live transport proof.

## Reproduction Assessment

Yes at source level. The potential merge result inserts the marker in the shared helper while existing merge-result tests still assert marker-free current-message and room-event prompt shapes.

## Solution Assessment

No. The delimiter is a plausible mitigation, but the best current fix is an out-of-band hidden runtime-context or trusted bare-body contract; if this delimiter remains, it must be metadata-gated and proven through the real runner/channel path.
