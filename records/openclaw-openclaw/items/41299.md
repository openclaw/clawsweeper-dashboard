## Summary

Keep open: the PR targets a real inbound-metadata prompt ambiguity, but the current patch is not merge-ready because it adds the marker in a shared current-inbound helper used by non-metadata current-message, room-event, and CLI paths where the sanitizer will not remove it. The cleaner hidden-runtime-context direction is already represented by overlapping open PRs, so maintainers should choose that contract before continuing this delimiter branch.

## What This Changes

The PR changes `buildCurrentInboundPrompt` to insert `---openclaw:user-msg---` between current inbound context and prompt text, then strips that marker from inbound metadata sanitizers with focused tests.

## Best Possible Solution

Land one canonical hidden-runtime-context routing path that keeps current inbound metadata available to the model without storing delimiter-marked metadata in user-authored prompt/session text.

## Reproduction Assessment

Yes, from source for the patch defect: the changed helper is shared by metadata, current-message, room-event, and CLI paths, and the sanitizer only removes the marker after stripping metadata blocks. I did not run tests or live Telegram proof in this read-only review.

## Solution Assessment

No. A delimiter is only a plausible mitigation; the narrower maintainable direction is to move current inbound metadata into hidden runtime context, or at minimum gate and prove the delimiter across every current-inbound shape.
