## Summary

Keep open: this is a focused bug-fix PR, but the current patch adds the separator through a shared helper that also handles compact current-message, room-event, and CLI current-inbound prompt paths where the marker is not metadata and will not be stripped.

## What This Changes

The PR adds an `---openclaw:user-msg---` marker when joining current inbound context with the user prompt, then strips that marker from inbound-metadata sanitizers and adds separator tests.

## Best Possible Solution

Settle one prompt-boundary design before merge: preferably move current inbound metadata into hidden runtime context, or narrowly gate this separator to actual metadata prefixes with coverage for compact current-message, room-event, CLI, and display/history sanitizer paths.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pass a non-metadata `currentInboundContext` such as `Current message:` with `promptJoiner: " "` through the patched `buildCurrentInboundPrompt`, and the marker is inserted even though no metadata stripper will remove it. I did not run tests because this was a read-only checkout review.

## Solution Assessment

No, this is not the best fix as-is. A marker can be an acceptable mitigation only if it is scoped to actual inbound metadata, while the cleaner direction is the hidden-runtime-context path that removes generated metadata from user-authored prompt/session text entirely.
