## Summary

Keep open: the PR targets a real current-main prompt ambiguity, but the proposed sanitizer still removes later literal separator lines from user content and the prompt-composition fixtures do not match the new runtime prompt body.

## What This Changes

The PR inserts `---openclaw:user-msg---` between inbound metadata and the user body, strips that marker from metadata sanitizers, and adds separator-specific stripping tests.

## Best Possible Solution

Keep the separator approach only after the stripper removes exactly the synthetic boundary adjacent to stripped metadata, prompt-composition fixtures cover the new body shape, and the changelog records the user-facing fix.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection confirms the current-main prompt builder joins metadata and user text with only double newlines, but I did not establish a deterministic live LLM reproduction of the intermittent empty-message response.

## Solution Assessment

Is this the best way to solve the issue? No; adding a boundary is a reasonable direction, but this implementation must remove only the injected boundary and keep prompt-composition tests aligned before merge.
