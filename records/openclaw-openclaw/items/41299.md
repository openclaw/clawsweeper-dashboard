## Summary

Keep open: the separator direction is useful and earlier review feedback was mostly addressed, but the current diff still removes later user-authored occurrences of the marker after any metadata strip, so it is not ready to merge.

## What This Changes

The PR inserts `---openclaw:user-msg---` between inbound metadata and the user body, then updates metadata sanitizers and tests to hide that marker from user-facing output.

## Best Possible Solution

Keep the separator fix narrow, but remove exactly the synthetic marker adjacent to a stripped metadata block and preserve any marker text that appears later in the real user body.

## Reproduction Assessment

Yes for the PR regression: the helper regex from the diff can be exercised directly and removes later marker text from a normal body. The original LLM false-empty behavior is source-plausible but was not live-reproduced in this read-only pass.

## Solution Assessment

No. Adding a boundary marker is a reasonable fix direction, but the current global post-processing is too broad; the safer implementation should consume at most the marker paired with the stripped metadata block.
