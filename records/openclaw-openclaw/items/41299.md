## Summary

Keep open, but the current patch is not merge-ready: it inserts the marker in a shared prompt helper used by non-metadata `Current message` and room-event contexts, while the added stripper coverage only proves fenced metadata sanitization. The branch also overlaps the broader prompt-contract direction tracked by https://github.com/openclaw/openclaw/issues/72704 and https://github.com/openclaw/openclaw/pull/89428.

## What This Changes

The PR adds an internal `---openclaw:user-msg---` separator to current inbound prompt construction, strips that marker from inbound metadata sanitizers, and adds separator-focused tests.

## Best Possible Solution

Preserve the shared helper's existing non-metadata output and either scope any delimiter to the generated fenced inbound metadata boundary or let maintainers choose the hidden-runtime-context approach as the canonical fix.

## Reproduction Assessment

Yes by source inspection. Current tests define non-metadata `buildCurrentInboundPrompt` outputs for `Current message` and room-event contexts, and the PR's changed return line would add the marker to those outputs; tests were not run because this review is read-only.

## Solution Assessment

No. The delimiter may be a plausible metadata mitigation, but adding it in the shared join helper is too broad; a metadata-specific boundary or the hidden-runtime-context approach is the safer owner-boundary fix.
