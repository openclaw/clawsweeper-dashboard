## Summary

Keep open. Current main and the latest stable release still lack a typed Gemini safety-settings config surface and the Google GenerateContent transport still builds requests without `safetySettings`; the only matching implementation PR is closed unmerged, while the later duplicate issue already points back here as the canonical tracker.

## What This Changes

The issue requests configurable Gemini safety thresholds for Google/Gemini providers or models so OpenClaw can reduce blocked or empty responses.

## Best Possible Solution

Define a narrow Google-owned safety-settings contract, validate it in config/schema/help, apply it in the Google GenerateContent transport without overriding explicit payload values, document the safety implications, and prove the serialized request with focused tests plus redacted live Gemini proof.

## Reproduction Assessment

Yes, at source level: current main's strict provider schema lacks a `safetySettings` key and the Google request builder returns Gemini payloads without `safetySettings`. I did not run a live Gemini request in this read-only review.

## Solution Assessment

Unclear as proposed: the provider/config direction is plausible and aligned, but the exact public schema, category set, threshold policy, and safety posture need maintainer selection before implementation. The previous PR is not the best landing path because it closed unmerged after proof and request-shape concerns.
