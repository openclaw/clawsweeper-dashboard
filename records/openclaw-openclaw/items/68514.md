## Summary

Keep open: the PR targets a real Feishu URL-rendering bug that is still present on current main, but the submitted helper still mutates some existing Markdown links and the PR lacks real after-fix Feishu behavior proof.

## What This Changes

The PR adds a Feishu-local helper that wraps bare http(s) URLs in post markdown payloads as explicit Markdown links, then adds helper and send-path regression tests.

## Best Possible Solution

Land a Feishu plugin-local URL-boundary fix at post payload construction that preserves full clickable bare URLs without mutating existing Markdown/link/code regions, backed by focused tests and redacted real Feishu proof.

## Reproduction Assessment

Yes, source-level: the linked bug gives concrete Feishu send steps plus a screenshot, and current main still routes the text into a raw Feishu post `md` element. I did not establish a live Feishu-client reproduction in this read-only review.

## Solution Assessment

No, not as currently submitted. A Feishu-local wrapper in `buildFeishuPostMessagePayload` is the right boundary, but the mask needs to preserve nested or escaped existing Markdown links and the PR still needs real after-fix behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mask nested-bracket Markdown links before wrapping URLs:** `extensions/feishu/src/markdown-urls.ts:8`
  - body: The Markdown-link mask only accepts link text without `]`, so an existing link like `[foo [bar]](https://example.com/a_b)` is not protected before the bare URL pass. The helper then rewrites the URL inside an already-linked destination, breaking content it promises to leave untouched.
  - confidence: 0.91
