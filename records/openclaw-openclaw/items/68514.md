## Summary

Keep open: current main still has the Feishu URL-boundary bug and this PR is a plausible plugin-local fix, but the submitted helper can rewrite existing Markdown links with nested bracket text and the external PR only has test proof, not real Feishu after-fix evidence.

## What This Changes

The PR adds a Feishu-local helper that wraps bare http(s) URLs as explicit Markdown links before constructing post `md` payloads, plus helper and send-path tests.

## Best Possible Solution

Land one Feishu-local URL-boundary fix that preserves full clickable bare URLs without mutating existing Markdown/link/code regions, then coordinate closure of the duplicate Feishu URL attempts.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main passes `messageText` directly into Feishu post `md`, the linked bug at https://github.com/openclaw/openclaw/issues/41860 has concrete Feishu send steps and a screenshot, and the PR-specific defect follows from the `MARKDOWN_LINK_RE` failing on `[foo [bar]](...)`.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently submitted. A Feishu-local wrapper is the right boundary, but it must mask nested or escaped existing Markdown link text and needs real Feishu after-fix proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Mask nested-bracket Markdown links before wrapping URLs:** `extensions/feishu/src/markdown-urls.ts:8`
  - body: The Markdown-link mask only accepts link text without `]`, so an existing link like `[foo [bar]](https://example.com/a_b)` is not protected before the bare URL pass. The helper then rewrites the URL inside an already-linked destination, breaking content it promises to leave untouched.
  - confidence: 0.91
