## What This Changes

The PR records the most recent visible non-preview Telegram delivery, uses that timestamp to force a fresh final message after intermediate output, adds a dispatcher regression test, and documents the fix in the changelog.

## Best Possible Solution

Land the PR after exact-head CI and the targeted Telegram tests/typecheck finish green; the runtime approach and changelog coverage look sufficient.

## Reproduction Assessment

Yes at source level: current base can deliver a visible block/tool bubble through `sendPayload` after a short-lived preview exists, then finalize by editing that older preview because only the 60-second age gate forces a fresh final. I did not run a live Telegram client reproduction.

## Solution Assessment

Yes. Comparing the latest visible non-preview delivery timestamp with the preview `visibleSinceMs` is a narrow fix that preserves existing long-lived-preview behavior while correcting the reported ordering case.
