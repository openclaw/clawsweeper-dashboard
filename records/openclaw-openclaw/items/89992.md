## Summary

Keep this PR open, but it is not merge-ready: the editor-schema idea is useful and the supplied real-behavior proof is strong, but the patch makes schema sidecar/root writes part of existing config-write success paths, which can break upgrades or permission-managed configs.

## What This Changes

The PR writes `openclaw.schema.json` beside the active config during OpenClaw-owned config writes, stamps missing or blank root `$schema` values, reuses the schema helper for `openclaw config schema`, and updates docs/tests.

## Best Possible Solution

Land only after maintainers choose the schema sidecar contract and the implementation preserves existing config-write behavior by default, or documents and tests an intentional hard-fail upgrade path.

## Reproduction Assessment

Yes at source level: current main writes only the config target, while the PR adds mandatory sidecar/root writes before or around the config commit. I did not run a filesystem-permission repro because this was a read-only review.

## Solution Assessment

No: automatic schema output is useful, but the current implementation is not the safest default because it expands existing config-write failure modes. A better solution preserves config writes when editor-schema generation fails, or makes hard-fail behavior an explicit maintainer-approved contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Do not block config writes on schema sidecar failures:** `src/config/io.ts:2488-2492`
  - body: This new call writes `openclaw.schema.json` before `openclaw.json` is committed. That means any EACCES/EPERM/ENOSPC on the adjacent sidecar now aborts `openclaw config set/patch` even when the requested config write could have succeeded, and a later config commit failure can leave the sidecar behind. Please make sidecar generation non-blocking/post-commit by default, or add an explicit hard-fail contract with failure/rollback coverage.
  - confidence: 0.88
- **[P1] Keep include-backed writes from requiring root-file writes:** `src/config/mutate.ts:400-405`
  - body: The include fast path has already committed the included fragment before this block writes the sidecar and stamps the root config. Existing include-backed setups can keep the root `openclaw.json` read-only while the included fragment is writable; this change makes those writes fail or roll back solely because the root/sidecar cannot be edited. Preserve the include-only success path unless maintainers explicitly opt into root stamping.
  - confidence: 0.84
