## Summary

Keep this PR open with one blocking compatibility finding: the main editor-schema idea is useful and has real proof, but the include-backed mutation repair now rewrites the root config through `JSON.stringify`, which can silently erase JSON5 comments/formatting on a path current main keeps untouched.

## What This Changes

The PR writes an `openclaw.schema.json` sidecar beside OpenClaw-owned config writes, stamps a local `$schema` reference when missing or blank, reuses that schema for `openclaw config schema`, and documents/tests the behavior.

## Best Possible Solution

Keep sidecar schema generation for OpenClaw-owned full config writes, but repair include-backed writes so editor metadata does not erase existing root JSON5 authoring.

## Reproduction Assessment

Yes for the review finding from source inspection: the PR writes `params.snapshot.parsed` back through `JSON.stringify` on include-backed mutations, while docs allow JSON5 comments. I did not run a filesystem repro because this review was read-only.

## Solution Assessment

No as submitted. Config IO is a reasonable owner for full-root writes and CLI schema reuse, but the include-backed path needs a raw-preserving stamp or an explicit exception instead of normalizing the root file.
