## What This Changes

The PR writes `openclaw.schema.json` beside active config files, stamps missing or blank root `$schema` references during OpenClaw-owned config writes, reuses that schema for `openclaw config schema`, and documents the behavior.

## Best Possible Solution

Make root `$schema` stamping token/root-object aware, add regression coverage for leading JSON5 trivia, then merge only if maintainers accept config IO as the owner for the new sidecar artifact behavior.

## Reproduction Assessment

Yes from source inspection. Current config parsing supports JSON5, docs show commented JSON5 include roots, and the PR helper inserts at the first `{` byte rather than the actual root object start.

## Solution Assessment

No, not as currently patched. The local schema direction is plausible, but the raw JSON5 insertion must be root-aware or safely skip ambiguous files before this is the maintainable fix.
