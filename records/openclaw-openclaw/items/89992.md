## Best Possible Solution

Keep the central config IO approach, but make the editor-schema update cover every OpenClaw-owned config mutation path and preserve only non-empty user schema refs.

## Solution Assessment

No: the config IO location is plausible, but the current implementation is too narrow. The better fix is to share the schema sidecar/stamp behavior across config mutation commit paths and correct the blank `$schema` overwrite.

Full review comments:

- **[P2] Cover include-backed config writes:** `src/config/io.ts:2508-2512`
  - body: This only writes `openclaw.schema.json` from the full-root `writeConfigFileLocal` path. `replaceConfigFile` can commit supported single-file top-level `$include` updates through `tryWriteSingleTopLevelIncludeMutation` and return without calling this writer, so those OpenClaw-owned writes still do not create the sidecar or stamp the root `$schema`, contradicting the new docs/PR contract. Please move the editor-schema update to the shared mutation path or add an include-write hook with regression coverage.
  - confidence: 0.91
- **[P3] Stamp blank schema values:** `src/config/io.ts:396-398`
  - body: The guard treats blank or whitespace `$schema` as missing, but the returned object spreads `cfg` after the local ref, so `{ $schema: "" }` overwrites `./openclaw.schema.json` and persists the blank value. Build the object with the local ref last, or strip blank values before returning, and add a regression test for the promised non-empty preservation behavior.
  - confidence: 0.88
