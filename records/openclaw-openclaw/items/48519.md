## Summary

Keep open: current main still has message-level input provenance, external-content marker text, and transcript/message hooks, but no durable provenance chain through tool results, persistent writes, or later read-time policy. The PitStop comment and related open security/provenance issues make this an active product/security design gap, not stale or implemented work.

## What This Changes

The issue requests durable provenance tracking for agent-written persistent state across sessions, including tool-result provenance, persistence-write metadata, and read-time policy visibility.

## Best Possible Solution

Define a versioned framework-owned provenance/taint contract that carries through tool outputs and durable writes, exposes redacted metadata to plugin/read-time policy, and covers memory, file, cron, config, preference, transcript, and SDK paths consistently.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/security design proposal. Source inspection gives high confidence that current main lacks durable provenance propagation through tool results, persistence writes, and read-time policy.

## Solution Assessment

Unclear as a final implementation. The direction is sound, but maintainers need to choose provenance shape, redaction, coverage, mutation/blocking semantics, and coordination with related hook and memory-trust work before a narrow patch would be safe.
