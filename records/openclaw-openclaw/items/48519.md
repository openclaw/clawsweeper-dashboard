## Summary

Keep this open: current main and v2026.6.10 still provide adjacent input provenance, external-content markers, transcript hooks, memory citations, and memory event logs, but not the requested provenance chain through tool results, durable writes, and future read-time policy. The PitStop comment adds useful operational pressure for immutable mutation tags and auditability, so this remains a security-sensitive plugin/persistence design item rather than conservative cleanup.

## What This Changes

Request to carry provenance metadata from user and external inputs through tool results, durable state writes, and later reads so persistent state can be audited and policy-checked across sessions.

## Best Possible Solution

Define a host-authored persistent provenance contract spanning tool results, durable write/read hooks, memory/search schemas, and plugin policy APIs, then land it with maintainer security review and compatibility tests.

Yes, at source level: current main exposes narrow user-message InputProvenance and transcript hooks, but no structured provenance chain through tool results, durable writes, and read-time memory policy. I did not run a live exploit because the requested architecture is absent from source.
