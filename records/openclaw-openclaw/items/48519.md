## Summary

Keep this open: current main has user-message provenance, external-content marker strings, persistence hooks, and memory recall framing, but it does not carry structured provenance chains from untrusted tool results into durable state or expose read-time provenance for future policy. The remaining work is a security/product design decision across plugin hooks, persistence, and memory, not a stale cleanup close.

## What This Changes

Request durable provenance tracking from tool results into persistent state and later reads so future sessions can evaluate the trust and origin of stored data.

## Best Possible Solution

Define a framework-owned provenance-chain model and persistence/read-time contract, then implement it consistently across tool results, durable writes, memory/search/dreaming flows, and plugin SDK hooks.

## Reproduction Assessment

Yes at source level: current main shows provenance on user messages and marker text for external content, while hooks and memory schemas lack provenance-chain fields; I did not run a live poisoning scenario because this is a feature/design request.

## Solution Assessment

The direction is sound, but the proposal is not a ready narrow patch; the best fix needs maintainer choices on provenance authority, hook/API shape, storage/migration, and enforcement modes.
