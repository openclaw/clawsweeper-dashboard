## Summary

Keep open. Current main still has only user-message provenance, marker-wrapped external content, transcript-scoped persistence hooks, and local web_fetch metadata; it does not carry a structured provenance chain through tool results into durable state writes or expose read-time provenance for stored state.

## What This Changes

Request to extend provenance across tool results, durable state writes, and future stored-state reads so persistent data can be audited and policy-checked across sessions.

## Best Possible Solution

Keep this open as a core security/plugin-API design item, or explicitly canonicalize it under a broader taint umbrella while preserving the durable-state requirements.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security architecture feature request. The high-confidence verification path is current-main source inspection, which shows the requested durable provenance chain and stored-state read metadata are absent.

## Solution Assessment

Yes: keeping this open for maintainer/security design is the best current path. A plugin-only workaround or one-backend memory filter would not solve the requested cross-surface provenance contract.
