## Summary

Keep open. Current main still has ingress/message provenance, external-content text markers, transcript-scoped hook interception, plugin approvals, and memory-wiki sidecar provenance, but it does not carry structured provenance from tool results into durable state writes or expose read-time provenance for stored state. The PitStop comment adds practical validation for immutable state-mutation tags and an audit trace endpoint, and related tracks #48503, #48509, #54508, and #28913 remain open rather than merged/superseding this request.

## What This Changes

Add a core/plugin provenance chain that persists from input and tool results into durable state writes and future reads.

## Best Possible Solution

Keep this as a core security/plugin-API design item, or explicitly canonicalize it under #54508 while preserving the durable-state details. The desired end state is a generic provenance/taint contract for tool results and durable writes, coordinated with #48503 and #48509, with read-time metadata, policy hooks, docs, and regression coverage across memory, file, cron/config, and auto-capture surfaces.

## Reproduction Assessment

Not applicable as a feature/security architecture request. The high-confidence verification path is source and docs inspection of hook types, input provenance, external-content wrapping, and memory persistence surfaces, which shows the requested durable provenance chain is absent.

## Solution Assessment

Yes: keeping this open for maintainer/security design is the best current outcome. A plugin-only workaround or a single memory filter would not satisfy the requested cross-surface provenance contract because the core SDK/runtime still lacks durable-write and read-time provenance seams.
