## Summary

Keep open. Current main still has only adjacent provenance and boundary primitives, not the requested structured chain from tool results through durable state writes into future read-time policy; the linked durable-write, hook-enrichment, memory-trust, taint-model, and CaMeL work remains open rather than shipped.

## What This Changes

Request to carry structured provenance across tool results, persistent state writes, and future reads of stored memory, files, preferences, and scheduled tasks.

## Best Possible Solution

Define an additive, documented core/plugin provenance contract that assigns framework-owned source metadata, propagates it through tool results and durable writes, exposes read-time metadata for policy, and covers memory, files, tasks, config, preferences, compaction, and plugin SDK tests.

## Reproduction Assessment

Not applicable as a runtime bug. The high-confidence check is source inspection on current main, which shows the requested durable provenance chain and read-time stored-state metadata are absent.

## Solution Assessment

Unclear as an implementation path. The direction is valid, but maintainers need to decide the cross-cutting SDK/security contract, trust assignment model, and coverage boundaries before a safe patch exists.
