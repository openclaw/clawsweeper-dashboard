## Summary

Keep open: current main still has only message-level input provenance, prompt-text external-content wrappers, and transcript/message hooks; it does not carry durable provenance through tool results, persistence writes, or later read-time policy. The PitStop comment and related open issues support this as an active security/product design gap rather than implemented or superseded work.

## What This Changes

Request a framework-level provenance chain that survives tool results, durable state writes, and future reads of persistent state.

## Best Possible Solution

Define a versioned framework-assigned provenance/taint contract that carries through tool outputs and durable writes, exposes safe metadata to plugin/read-time policy, and covers representative memory, file, cron, config, transcript, and SDK paths.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/security design proposal. Source inspection gives high confidence that current main lacks the requested durable provenance chain in hook and memory contracts.

## Solution Assessment

Unclear as a final implementation. The direction is sound, but maintainers need to choose provenance shape, redaction, coverage, and enforcement semantics before a narrow code change would be safe.
