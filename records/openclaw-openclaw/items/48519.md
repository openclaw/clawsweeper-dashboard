## Summary

Keep open: current main has message-level input provenance, external-content text wrappers, and transcript/message hooks, but it still lacks durable provenance propagation through tool results, persistence writes, and future read-time policy. The PitStop comment and related open security/provenance threads make this an active cross-session design gap, not an implemented or clearly superseded request.

## What This Changes

Add framework-level provenance tracking that follows tool-derived data into persistent state and remains available when future sessions read that state.

## Best Possible Solution

Define a versioned framework-assigned provenance/taint contract that is carried through tool results, durable writes, retrieval, and plugin policy surfaces, with explicit redaction and trust semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/security architecture request. Source inspection gives a high-confidence state check that current main lacks tool-result provenance, persistence-write provenance payloads, and read-time persistent provenance policy.

## Solution Assessment

Unclear as a final implementation. The requested direction is sound, but maintainers need to choose the provenance shape, assignment authority, redaction rules, and coverage across memory, files, cron, config, transcript, and SDK surfaces before a narrow patch is safe.
