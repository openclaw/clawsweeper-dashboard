## Summary

Keep open: current main still lacks a durable provenance chain from tool-derived content into persistent state and back into later session reads. The related issues overlap but are still open, and Kelisi808's PitStop comment adds concrete mutation-tag and audit-trace requirements rather than making this obsolete.

## What This Changes

The issue requests provenance metadata that follows tool-derived content through persistent writes and remains available for future-session policy decisions.

## Best Possible Solution

Define a versioned framework provenance contract spanning tool outputs, durable writes, read-time policy, memory backends, plugin SDK hooks, approval/redaction semantics, and audit surfaces.
