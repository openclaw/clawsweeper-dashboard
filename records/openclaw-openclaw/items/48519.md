## Summary

Keep open. Current main has ingress/user-message provenance, external-content boundary markers, transcript-scoped hooks, and memory-wiki sidecar provenance, but it still does not carry structured provenance from tool results into durable state writes or expose read-time provenance for stored state; the PitStop comment adds concrete state-mutation and audit-trace asks, and related tracks #48503, #48509, #54508, and #28913 remain open.

## What This Changes

Request to preserve structured data provenance from inputs and tool results through durable writes and future reads of persistent memory, files, preferences, and tasks.

## Best Possible Solution

Keep this open as a core security/plugin-API design item, or explicitly canonicalize it under #54508 while preserving the durable-state requirements; the end state should be a generic provenance/taint contract for tool results, durable writes, read-time metadata, policy hooks, docs, and regression coverage across memory, file, cron/config, and auto-capture surfaces.

## Reproduction Assessment

Not applicable as a security architecture feature request. The high-confidence verification path is current-main source and docs inspection of hook types, input provenance, external-content wrapping, and memory persistence surfaces, which shows the requested durable provenance chain is absent.

## Solution Assessment

Yes: keeping this open for maintainer/security design is the best current outcome. A plugin-only workaround or single memory filter would not satisfy the requested cross-surface contract because the public SDK/runtime lacks durable write and read-time provenance seams.
