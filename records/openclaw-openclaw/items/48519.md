## Summary

Keep open: current main has message-level input provenance, external-content text wrappers, and transcript/message hooks, but not durable provenance propagation through tool results, persistence writes, or future read-time policy. The related PitStop comment and open provenance/security issues reinforce that this is a real cross-session security design gap, not an implemented or superseded request.

## What This Changes

The issue requests structured provenance to persist from inputs and tool results into durable state, then remain available when future sessions read that state.

## Best Possible Solution

Define a versioned provenance/taint contract that is assigned by the framework, carried through tool results and durable writes, exposed safely to plugins and read-time policy, and covered across representative memory, file, cron, config, transcript, and SDK surfaces.

## Reproduction Assessment

Not applicable as a runtime bug. Source inspection gives high confidence that current main lacks a durable provenance chain in hook and memory contracts.

## Solution Assessment

Unclear as a final design. The requested direction is sound, but maintainers need to choose provenance shape, redaction, coverage, and enforcement semantics before a narrow implementation would be safe.
