## Summary

Keep this open: current main and v2026.6.10 still have adjacent message provenance, external-content markers, transcript hooks, and memory source citations, but not the requested provenance chain through tool results, durable writes, and future read-time policy.

## What This Changes

Request to carry data provenance from input and tool-result boundaries into persistent state so future sessions can evaluate trust before using stored memory, files, preferences, or scheduled-task data.

## Best Possible Solution

Define a host-authored persistent provenance contract spanning tool results, durable write/read hooks, memory/search schemas, and plugin policy APIs, then land it with maintainer security review and compatibility tests.

## Reproduction Assessment

Yes, at source level: current main exposes narrow user-message InputProvenance and transcript hooks, but no structured provenance chain through tool results, durable writes, and read-time memory policy. I did not run a live exploit because the requested architecture is absent from source.

## Solution Assessment

Unclear as an implementation plan: the direction fits OpenClaw's security priorities, but maintainers still need to choose the public provenance, durable-write, audit, and read-time policy contracts before code should be queued.
