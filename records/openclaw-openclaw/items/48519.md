## Summary

Keep this issue open: current main and v2026.6.8 still provide adjacent protections, but not the requested host-authored provenance chain through tool results, durable writes, and future read-time policy. The PitStop comment adds useful real-world pressure for immutable mutation tags and auditability, and the remaining work is security-sensitive plugin/API and persistent-state design rather than conservative cleanup.

## What This Changes

The issue requests durable provenance tracking so agent-written persistent state can retain source and trust information across future sessions.

## Best Possible Solution

Define a maintainer-approved, host-authored provenance contract that flows through tool results, durable write hooks, storage metadata, and read-time policy, while keeping policy engines plugin-owned where current or new SDK seams can support them.

## Reproduction Assessment

Yes at source level: current main shows message provenance, external-content wrappers, and transcript hooks, while hook and memory surfaces lack durable provenance-chain fields or a before_persistence_write-style durable write hook. I did not run a live memory-poisoning scenario because this is a feature/design request and the checkout is read-only.

## Solution Assessment

Unclear as an implementation, but the direction is sound. The best fix is not a narrow patch; maintainers need to settle provenance authority, SDK/hook shape, storage metadata, migration impact, and read-time enforcement semantics first.
