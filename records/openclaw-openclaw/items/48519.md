## Summary

Keep open. Current main has user-message provenance, external-content boundary markers, transcript-scoped hooks, active-memory filtering for external marker noise, and Memory Wiki provenance, but it still does not carry a structured provenance chain through generic tool results into durable state writes or expose read-time provenance for stored memory/files/tasks. The related durable-write, hook-enrichment, memory-trust, and broader taint-tracking items remain open, and the same author also has an open CaMeL implementation PR that should not cause this issue to close before merge.

## What This Changes

The issue requests end-to-end provenance tracking from tool results into persistent state writes and future reads.

## Best Possible Solution

Define an additive, documented core/plugin provenance contract that carries source metadata through tool results, durable writes, and future reads across memory, files, tasks, config, and preferences, with policy hooks and regression coverage for the persistent-state surfaces.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security architecture feature request. The high-confidence verification path is source and docs inspection on current main, which shows the requested durable provenance chain and stored-state read metadata are absent.
