## Summary

Keep open. Current main has user-message provenance, external-content boundary markers, transcript-scoped hooks, plugin approvals, and Memory Wiki provenance, but it still does not carry a structured provenance chain through tool results into durable state writes or expose read-time provenance for stored state; the related provenance/security tracks remain open.

## What This Changes

Request structured provenance metadata that follows user, inter-session, and external tool content through tool results and durable state writes, then remains available when stored state is read in later sessions.

## Best Possible Solution

Keep this open as a core security/plugin-API design item, or explicitly canonicalize it under the broader taint umbrella while preserving the durable-state requirements; the end state should be a generic provenance contract for tool results, durable writes, read-time metadata, policy hooks, docs, and regression tests across the persistent state surfaces.

## Reproduction Assessment

Not applicable as a runtime bug; this is a security architecture feature request. The high-confidence verification path is current-main source and docs inspection, which shows the requested durable provenance chain and stored-state read metadata are absent.

## Solution Assessment

Yes: keeping this open for maintainer/security design is the best current path. A plugin-only workaround or a one-backend memory filter would not solve the requested cross-surface provenance contract.
