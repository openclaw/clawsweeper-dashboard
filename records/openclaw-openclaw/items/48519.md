## Summary

Keep open: current main still lacks durable provenance that follows tool-derived content into persistent state and back into future session reads. The related open issues overlap but do not supersede this one because this issue ties tool-result provenance, durable write provenance, and read-time policy into one security/product design.

## What This Changes

Request a framework provenance chain that survives tool results and durable writes so future sessions can evaluate stored state by origin.

## Best Possible Solution

Define a versioned framework provenance contract spanning tool outputs, durable writes, read-time policy, memory backends, plugin SDK hooks, approval/redaction semantics, and audit surfaces.

Not applicable as a runtime bug. Source inspection gives high confidence that current main lacks the requested durable provenance path, but this is a security/product feature proposal rather than a failing command path.

## Solution Assessment

Unclear as a final implementation. The direction is sound, but maintainers need to choose schema, trust assignment, redaction, SDK exposure, audit endpoints, and enforcement semantics before a safe patch exists.
