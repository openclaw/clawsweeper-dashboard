## Label Justifications

- `P2`: This is a normal-priority cron/channel delivery correctness bug with visible control-token leakage but limited blast radius.
- `impact:message-loss`: The issue is about channel delivery suppression semantics: a reply that should suppress posting is instead delivered to the configured channel.

## Summary

Keep open. Current main and the latest release still suppress only `NO_REPLY` in cron announce delivery, so `REPLY_SKIP` and `ANNOUNCE_SKIP` remain source-reproducible leaks until one of the open linked fix PRs lands.

## What This Changes

The issue asks for cron announce delivery to suppress control-reply tokens such as `REPLY_SKIP` and `ANNOUNCE_SKIP` instead of posting them to external channels.

## Best Possible Solution

Review and land the strongest existing fix path, preferably https://github.com/openclaw/openclaw/pull/85471 if maintainers agree with its exact-token scope, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows exact `REPLY_SKIP` and `ANNOUNCE_SKIP` remain non-empty after cron normalization and can proceed to outbound delivery, and the reporter supplied repeated Discord observations.

## Solution Assessment

Yes for the exact-token leak: reuse the shared control-reply suppression predicate in cron delivery and add focused direct/text regressions. The issue should stay open until one linked PR merges.
