## Label Justifications

- `P2`: This is a normal-priority cron/channel delivery correctness bug with visible control-token leakage but a limited announce-delivery surface.
- `impact:message-loss`: The issue concerns channel delivery suppression semantics: a control reply that should suppress posting is delivered as a visible channel message instead.

## Summary

Keep open. Current main and the latest release still normalize only `NO_REPLY` in the cron announce delivery path while chat-facing suppression already recognizes `ANNOUNCE_SKIP` and `REPLY_SKIP`; the two linked open PRs are implementation candidates, so the issue should close only after one lands.

## What This Changes

Cron announce delivery should suppress exact `REPLY_SKIP` and `ANNOUNCE_SKIP` control replies instead of posting them to external channel targets.

## Best Possible Solution

Land one narrow linked fix that reuses the shared control-reply predicate in cron delivery, then close this issue and leave markdown-wrapped token handling as a separately tested follow-up if desired.

## Reproduction Assessment

Yes. Source inspection on current main shows exact `REPLY_SKIP` remains non-empty after cron normalization and reaches the outbound send path; the reporter also supplied repeated real Discord observations.

## Solution Assessment

Yes for the exact-token bug: reusing the existing shared suppression predicate in cron delivery is the narrow maintainable fix. https://github.com/openclaw/openclaw/pull/85471 appears closer to that shape than the duplicate local-token implementation in https://github.com/openclaw/openclaw/pull/85492.
