## Label Justifications

- `P2`: This is a normal-priority delivery correctness bug with visible channel leakage but a limited cron announce surface.
- `impact:message-loss`: The issue concerns channel message delivery suppression semantics: a control token that should suppress posting is delivered to the channel instead.

## Summary

Keep open. Current main still has a split between gateway chat-display suppression, which recognizes `ANNOUNCE_SKIP` and `REPLY_SKIP`, and cron announce delivery, which only normalizes `NO_REPLY` before outbound send, so the reported leak remains source-reproducible.

## What This Changes

Request wiring existing control-reply suppression tokens into isolated cron announce final delivery so `REPLY_SKIP` and `ANNOUNCE_SKIP` are not posted to configured channels.

## Best Possible Solution

Use the canonical control-reply suppression predicate or a shared normalizer in cron final delivery, suppress exact control-token-only payloads before outbound send, and add focused regressions for text and structured cron paths.

## Reproduction Assessment

Yes, source-reproducible without a live Discord run: an announce cron result of `REPLY_SKIP` remains non-empty after cron normalization and reaches outbound delivery. The reporter also supplied concrete repeated Discord observations.

## Solution Assessment

Yes for the core exact-token bug: the narrow maintainable fix is to reuse the existing control-reply suppression semantics in cron delivery before the outbound adapter. Markdown-wrapped variants should only be added with explicit narrow tests.
