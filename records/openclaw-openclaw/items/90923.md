## Summary

Keep open: this is a useful, well-evidenced fix for the external-APFS-home LaunchAgent failure, but it is not merge-ready because the macOS app still reads the gateway plist from the raw home-directory LaunchAgents path while the PR moves daemon-managed plists to the boot-volume home.

## What This Changes

The PR adds a boot-volume-aware macOS LaunchAgent home resolver and routes daemon install, uninstall, doctor service scanning, restart handoff, supervisor restart fallback, update restart scripts, and tests through it for external-APFS-home setups.

## Best Possible Solution

Land one canonical boot-volume LaunchAgent plist path across both the daemon CLI and macOS app, while preserving OpenClaw state, logs, and the attach-only disable marker under the user's real HOME.

## Reproduction Assessment

Yes. Current main and v2026.6.6 still build LaunchAgent plist paths from HOME, and the linked issue plus PR terminal proof show launchd returning error 5 when that plist is on an external APFS volume.

## Solution Assessment

No, not yet. A single daemon resolver is the right fix shape, but the same canonical LaunchAgent path needs to reach the macOS app snapshot/log surface before this is the best complete solution.
