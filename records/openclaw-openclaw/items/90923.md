## Label Justifications

- `P1`: The PR targets a broken macOS gateway install and restart workflow that prevents external-APFS-home users from running the managed gateway.
- `merge-risk: 🚨 compatibility`: The PR intentionally changes the persisted LaunchAgent plist location for external-home macOS users and affects manual workaround paths.
- `merge-risk: 🚨 availability`: If any lifecycle or app path still targets the external HOME plist, gateway install, restart, or app connection can remain broken at runtime.

## Summary

Needs changes before merge: the daemon-side boot-volume resolver is the right direction and has strong live launchd proof, but the macOS app still reads the Gateway LaunchAgent plist from the home-derived path and can miss plist-backed token, password, and log metadata after this PR moves the managed plist to the boot volume.

## What This Changes

The PR adds a boot-volume-aware macOS LaunchAgent home resolver and routes daemon install, uninstall, inspection, restart handoff, supervisor restart, update restart helper, and related tests through it for external-APFS-home setups.

## Best Possible Solution

Land one canonical boot-volume LaunchAgent plist path across both the daemon CLI lifecycle and the macOS app snapshot/log reader, while keeping OpenClaw state, logs, and the attach-only disable marker under the user’s real HOME.

## Reproduction Assessment

Yes. Current main and v2026.6.6 still build LaunchAgent plist paths from HOME, and the linked issue plus PR terminal proof show launchd returning error 5 when bootstrapping a plist from an external APFS volume; I did not run a local macOS APFS repro in this Linux checkout.

## Solution Assessment

No, not yet. A single daemon resolver is the right fix shape, but the same canonical LaunchAgent path needs to reach the macOS app snapshot/log surface before this is the best complete solution.
