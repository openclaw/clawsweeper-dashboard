## Label Justifications

- `P1`: The PR addresses a broken macOS gateway install and restart workflow that prevents external-APFS-home users from running the managed gateway.
- `merge-risk: 🚨 compatibility`: The PR intentionally changes the persisted LaunchAgent plist location for external-home macOS users, which is upgrade-sensitive for existing manual workarounds and cleanup behavior.
- `merge-risk: 🚨 availability`: If the new resolver targets the wrong boot-volume account or misses a launchd lifecycle path, gateway install/restart can still fail at runtime.

## Summary

Keep open. Current main still derives macOS LaunchAgent plist paths from HOME, while the PR’s merge result centralizes the boot-volume fallback across install, repair, restart, uninstall, inspection, and update restart paths; I found no blocking patch defect, but the path-location change remains upgrade-sensitive.

## What This Changes

The PR changes macOS LaunchAgent path resolution so gateway launchd plists are managed under the boot-volume home when the configured home directory is on an external APFS volume.

## Best Possible Solution

Land the canonical boot-volume LaunchAgent path after maintainer acceptance of the upgrade-sensitive plist-location change, while keeping logs and OpenClaw state under the user’s real HOME/state directory.

## Reproduction Assessment

Yes. Current main resolves the LaunchAgent plist from HOME, and the related report plus PR proof show launchd failing with error 5 when that plist is on an external APFS volume.

## Solution Assessment

Yes. A single launchd-specific resolver is the best fix shape because install-only copying would leave repair, restart, uninstall, status/audit, and update restart fallbacks targeting the failing external path.
