## Summary

Keep open: the PR addresses a real macOS gateway install failure that is still present on current main and the latest release, but it should not merge until the macOS app uses the same canonical LaunchAgent plist path as the daemon. The daemon-side resolver is the right direction; the remaining gap is a sibling app lookup that can still read the external-home plist path after this PR writes the boot-volume plist.

## What This Changes

The PR adds a boot-volume-aware macOS LaunchAgent home resolver and routes daemon install, uninstall, inspection, restart handoff, supervisor restart, update restart helper, and tests through it for external-APFS-home setups.

## Best Possible Solution

Land one canonical boot-volume LaunchAgent plist path across both the daemon CLI and macOS app, while keeping OpenClaw state, logs, and the attach-only disable marker under the user's real HOME.

## Reproduction Assessment

Yes. Current main and v2026.6.6 still build LaunchAgent plist paths from HOME, and the linked issue plus PR live proof show launchd returning error 5 when that plist is on an external APFS volume.

## Solution Assessment

No, not yet. A single daemon resolver is the right fix shape, but the same canonical LaunchAgent path needs to reach the macOS app snapshot/log surface before this is the best complete solution.
