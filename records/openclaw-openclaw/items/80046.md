## Summary

Keep open: the PR targets a real CLI/API parity bug and has credible live proof, but the parser fallback can emit empty tool args before the documented complete assistant snapshot arrives, so it needs a narrow fix before merge.

## What This Changes

The PR adds claude-cli stream-json tool_use/tool_result parsing, sanitized CLI tool-event bus emission, a CLI-to-channel onToolStart bridge, tests, and a changelog entry.

## Best Possible Solution

Land the CLI parity fix after preserving streamed tool args across the documented Claude stream order and rebasing the branch onto current main.

## Reproduction Assessment

Yes. Current main is source-reproducible because the claude-cli JSONL parser only forwards text_delta records and the CLI run path only bridges assistant events, while the PR body supplies live stock v2026.5.7 logs showing no CLI tool bus events for a Telegram tool-heavy turn.

## Solution Assessment

No, not yet. The parser-to-sanitized-bus-to-existing-onToolStart bridge is the right shape, but the parser should accumulate input_json_delta or defer fallback so documented stream order does not drop tool args.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve streamed tool input before stop:** `src/agents/cli-output.ts:470`
  - body: The fallback emits a start with `{}` as soon as `content_block_stop` arrives. Claude/Claude Code streaming docs put the complete `AssistantMessage` after `content_block_stop`, with `input_json_delta` chunks before it, so this marks the id as started and prevents the later full `input` snapshot from delivering file/command args. Accumulate and parse the `input_json_delta` by index, or defer the empty fallback until after the complete assistant snapshot, otherwise Read/Bash previews lose the details this PR is meant to surface. ([code.claude.com](https://code.claude.com/docs/en/agent-sdk/streaming-output?utm_source=openai))
  - confidence: 0.78
