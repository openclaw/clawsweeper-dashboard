## Summary

Keep open. This PR targets a real CLI-preview parity bug and includes useful live proof, but the patch should not merge as-is because it emits raw CLI tool payloads without the existing redaction/capping path and assumes a Claude Code stream order that the official streaming docs contradict. ([code.claude.com](https://code.claude.com/docs/en/agent-sdk/streaming-output?utm_source=openai))

## What This Changes

The PR adds Claude CLI stream-json tool-use/result parsing, emits those events on the agent-event bus from both CLI execution paths, bridges start/update tool events into reply preview callbacks, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the bridge direction, but align CLI tool events with embedded runtime contracts by sanitizing/capping payloads, honoring the documented Claude Code message order, adding a regression for stop-before-assistant snapshots, and resolving the draft/conflict state before merge.

## Reproduction Assessment

Yes. Current-main source inspection shows the CLI parser filters out non-text stream-json records and the CLI runner only bridges assistant deltas, while the PR body supplies live stock-vs-patched logs for the Telegram claude-cli behavior.

## Solution Assessment

No. The intended fix belongs in this parser/runner/preview bridge, but the current patch must first preserve existing tool payload sanitization and handle the documented Claude Code ordering without emitting empty args too early.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Sanitize CLI tool payloads before emitting previews:** `src/agents/cli-runner/execute.ts:482`
  - body: The new CLI callbacks emit raw `args` and `result` onto `stream: "tool"`, unlike the embedded path which runs tool args/results through `sanitizeToolArgs`/`sanitizeToolResult` before bus emission. A CLI Bash tool call containing an API key in its command can now reach preview consumers such as Telegram, so sanitize and cap these payloads before emitting in both CLI branches.
  - confidence: 0.94
- **[P2] Wait for the complete assistant snapshot before starting:** `src/agents/cli-output.ts:470`
  - body: This fallback fires at `content_block_stop` with `{}` args and records the id as started, but Claude Code documents the complete `AssistantMessage` as arriving after `content_block_stop`/`message_stop`. In normal stream order the later snapshot cannot replace the empty args, so channel previews lose the path/command details this PR is meant to surface. ([code.claude.com](https://code.claude.com/docs/en/agent-sdk/streaming-output?utm_source=openai))
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete secret-exposure risk by bypassing existing tool payload redaction for CLI tool events.

Concerns:

- **[high] Raw CLI tool args can leak secrets to previews:** `src/agents/cli-runner/execute.ts:482`
  - body: `execute.ts` forwards CLI tool args/results directly to the global agent-event bus. Existing embedded tool events redact credential-like strings before emission, and preview consumers can display these payloads to channels, so a command containing an API key or bearer token could leak.
  - confidence: 0.94
