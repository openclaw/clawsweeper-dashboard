## Summary

Keep this PR open: bounding the DuckDuckGo success body is useful security hardening, but the current patch parses and caches a truncated response as if it were complete, and the supplied proof does not exercise the patched DuckDuckGo client path.

## What This Changes

The PR changes the DuckDuckGo client success path from unbounded `response.text()` to `readResponseText(response, { maxBytes: 2 * 1024 * 1024 })`.

## Best Possible Solution

Keep the bounded read, check `truncated` before bot-challenge detection or parsing, fail with a clear response-too-large error, and add focused DuckDuckGo client coverage plus provider-path proof.

## Reproduction Assessment

Yes, at source level: current main uses unbounded `response.text()` on the DuckDuckGo success path, and the bounded helper returns `truncated` instead of throwing. I did not run live DuckDuckGo or the PR proof in this read-only review.

## Solution Assessment

No as submitted: the bounded helper is the right primitive, but this provider must check truncation before parsing and caching. The safer sibling shape is the SearXNG pattern that throws when the capped body is truncated.
