## Summary

Keep this PR open, but it is not merge-ready: the cap uses the right shared reader, but the patch discards the `truncated` signal and can treat partial DuckDuckGo HTML as complete search results.

## What This Changes

The PR changes the DuckDuckGo web-search success path to read HTML via `readResponseText` with a 2 MiB byte cap.

## Best Possible Solution

Keep the bounded read, check `truncated` before bot-challenge detection or parsing, fail with a clear response-too-large error, and add focused DuckDuckGo client coverage plus provider-path proof.

## Reproduction Assessment

Yes, at source level: current main uses unbounded `response.text()` on the DuckDuckGo success path, and the bounded helper returns `truncated` instead of throwing. I did not run live DuckDuckGo or the PR proof in this read-only review.

## Solution Assessment

No as submitted: the bounded helper is the right primitive, but the provider must check truncation before parsing and caching. The safer sibling shape is the SearXNG pattern that throws when the capped body is truncated.
