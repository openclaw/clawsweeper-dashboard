## Summary

Keep this issue open. Current main still provides advisory prompt guidance, tool registration, text-shaped tool-call diagnostics, and missing-result repair only after structured tool calls exist; it does not enforce a fresh calendar/gog/exec invocation before accepting a plain final answer to an explicit factual or raw-output request.

## What This Changes

The issue reports that factual calendar/raw shell-output requests can receive fabricated final answers without the expected live gog/calendar tool invocation.

## Best Possible Solution

Define a narrow runtime contract for explicit freshness or raw-output turns, then enforce it in the embedded runner, tool policy, or a documented hook boundary so the assistant retries, blocks, or asks a clarifying question instead of finalizing when the required tool event is missing.

## Reproduction Assessment

Yes for a code-level reproduction: simulate an assistant final answer to an explicit calendar/raw-output request with no structured tool-call block, and current main has no guard that blocks it. The reporter's gog shim/log path is concrete, but live macOS gog validation was not run here.

## Solution Assessment

No, current main's advisory prompt and diagnostics are not sufficient to solve the reported behavior. The safer fix is a narrow enforceable contract for explicit freshness/raw-output requests, not broad unconditional tool forcing for every factual-looking question.
