## Summary

Keep this issue open. Current main has advisory prompt guidance, explicit-allowlist fail-closed behavior, tool-call diagnostics, and repair for missing results after structured tool calls, but I found no runtime contract that requires a fresh gog/calendar/exec invocation before delivering a plain final answer to an explicit factual or raw-output request.

## What This Changes

The issue reports that factual calendar/raw shell-output requests can receive fabricated final answers without the expected live gog/calendar tool invocation.

## Best Possible Solution

Define a narrow runtime contract for explicit freshness or raw-output turns, then enforce it in the embedded runner, tool policy, or a documented hook boundary so the assistant retries, blocks, or asks a clarifying question instead of finalizing when the required tool event is missing.

## Reproduction Assessment

Yes for a code-level reproduction: simulate an assistant final answer to an explicit calendar/raw-output request with no structured tool-call block and current main has no guard that blocks it. The reporter’s gog shim/log path is also concrete, but live macOS gog validation was not run here.

## Solution Assessment

No, current main’s advisory prompt and diagnostics are not sufficient to solve the reported behavior. The safer fix is a narrow enforceable contract for explicit freshness/raw-output requests, not broad unconditional tool forcing for every factual-looking question.
