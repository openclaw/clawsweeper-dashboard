## Summary

Keep open: current main and the latest release still have the Telegram CLI bridge gap. The channel-owned remap exists, but the exported Telegram plugin action wrapper does not expose `resolveCliActionRequest`, so `openclaw message thread create --channel telegram` can still fall back to the unsupported `thread-create` action.

## What This Changes

The issue reports that Telegram CLI thread creation dispatches `thread-create` instead of remapping to Telegram’s supported `topic-create` action.

## Best Possible Solution

Keep the generic `message thread create` CLI contract and make the exported Telegram plugin action surface own the alias mapping to `topic-create`.

## Reproduction Assessment

Yes from source: the CLI asks the exported channel plugin for `resolveCliActionRequest`, but the current exported Telegram action wrapper does not expose that hook. I did not run a live Telegram command in this read-only review.

## Solution Assessment

Yes, the narrow wrapper forwarder plus exported-surface regression test is the best repair. Hard-coding Telegram back into the core CLI would duplicate plugin policy that the current plugin hook is meant to own.
