## Summary

Keep open: current main still has the reported Telegram CLI bridge bug. The Telegram static action adapter contains the `thread-create` to `topic-create` remap, but the exported Telegram plugin action wrapper does not forward `resolveCliActionRequest`, so the CLI can still dispatch bare `thread-create` into Telegram handling.

## What This Changes

The issue reports that `openclaw message thread create --channel telegram` dispatches `thread-create` instead of remapping to `topic-create`, causing the gateway to reject the Telegram action.

## Best Possible Solution

Forward the existing Telegram CLI remap through the exported Telegram action wrapper, cover that plugin surface with a focused regression test, and keep the current topic result contract backward-compatible.

## Reproduction Assessment

Yes. Source inspection on current main shows the CLI asks the channel plugin for a `thread-create` remap, the Telegram static adapter implements that remap, and the exported Telegram wrapper omits the hook, leaving a clear failure path without needing a bot token.

## Solution Assessment

Yes. The narrowest maintainable fix is to forward `resolveCliActionRequest` through `telegramPlugin.actions` and test that real exported wrapper; adding a broad Telegram runtime alias for `thread-create` would bypass the channel-owned CLI normalization seam.
