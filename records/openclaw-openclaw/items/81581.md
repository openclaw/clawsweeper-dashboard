## Summary

Keep open: current main and the latest release still have a source-reproducible Telegram CLI bridge gap, and two open PRs already target this issue, so it should wait for one focused fix to land.

## What This Changes

The issue asks for Telegram `openclaw message thread create` to remap `thread-create` to `topic-create` before gateway dispatch.

## Best Possible Solution

Land one focused wrapper-forwarding PR with regression coverage for the real Telegram plugin surface and redacted real Telegram proof when feasible.

## Reproduction Assessment

Yes. Source inspection is high-confidence: the CLI calls the plugin remap hook, Telegram implements that hook, the exported Telegram wrapper does not expose it, and the fallback sends unsupported `thread-create` to Telegram handling.

## Solution Assessment

Yes. Forwarding the existing channel-owned `resolveCliActionRequest` through the Telegram wrapper is the narrowest maintainable fix; adding a broad runtime alias would bypass the existing CLI normalization seam.
