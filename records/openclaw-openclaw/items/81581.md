## Summary

Keep open: current `main` still has the Telegram remap only on the internal action adapter, while the exported Telegram plugin wrapper that the CLI calls does not forward it, so the reported unsupported `thread-create` path remains source-proven.

## What This Changes

The issue reports that `openclaw message thread create --channel telegram` dispatches generic `thread-create` instead of Telegram's supported `topic-create` action.

## Best Possible Solution

Forward the existing Telegram CLI resolver through `telegramPlugin.actions`, cover the exported plugin surface with a focused regression test, and preserve the current topic result contract while optionally adding a compatible `message_thread_id` alias.

## Reproduction Assessment

Yes at source level. Current `main` shows the CLI calls the exported plugin resolver, the internal Telegram remap exists, and the exported wrapper lacks that resolver, so bare `thread-create` can reach unsupported Telegram action handling.

## Solution Assessment

Yes. The narrow maintainable fix is to forward `resolveCliActionRequest` through the Telegram wrapper and test `telegramPlugin.actions`; adding a generic `thread-create` Telegram runtime alias would bypass the channel-owned CLI normalization seam.
