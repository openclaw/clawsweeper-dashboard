## Summary

Keep open: current main still has the Telegram CLI remap only on the internal action adapter, while the exported Telegram plugin action wrapper that the CLI calls still does not forward `resolveCliActionRequest`. That leaves the reported source path intact: `thread-create` falls back unchanged and Telegram action handling rejects it as unsupported.

## What This Changes

The issue asks for `openclaw message thread create --channel telegram` to remap generic `thread-create` requests into Telegram `topic-create` requests so forum topic creation succeeds.

## Best Possible Solution

Forward the existing Telegram CLI resolver through `telegramPlugin.actions`, add exported-plugin regression coverage, and keep the topic creation result contract backward-compatible.

## Reproduction Assessment

Yes, at source level. Current main's CLI calls the exported channel hook, the internal Telegram adapter contains the remap, the exported wrapper omits that hook, and Telegram runtime aliases reject bare `thread-create`.

## Solution Assessment

Yes. The narrow maintainable fix is to forward `resolveCliActionRequest` through the Telegram wrapper and test `telegramPlugin.actions`; adding a broad runtime `thread-create` alias would bypass the channel-owned CLI normalization seam.
