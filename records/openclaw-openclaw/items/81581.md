## Summary

Keep open. Current main and released v2026.5.7 still expose the Telegram thread-create remap only on the inner action implementation, while the exported Telegram plugin action wrapper read by the CLI omits `resolveCliActionRequest`, so generic `thread-create` can still reach the gateway.

## What This Changes

The issue requests restoring `openclaw message thread create --channel telegram` so CLI `thread-create` resolves to Telegram `topic-create` with `threadName` mapped to `name` before dispatch.

## Best Possible Solution

Forward the existing Telegram CLI action resolver through the exported Telegram plugin action wrapper and add coverage against the real Telegram plugin/CLI path.

## Reproduction Assessment

Yes, from source inspection: the CLI falls back when the selected plugin has no resolver, and the exported Telegram plugin wrapper currently lacks the resolver even though the inner implementation has it. A live Telegram credential run was not performed in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to forward the existing resolver through `extensions/telegram/src/channel.ts` and test the real Telegram wrapper path, rather than adding a broad runtime alias for generic `thread-create`.

## Repair Work Prompt

Fix the Telegram CLI thread-create remap regression. Current main has `src/cli/program/message/register.thread.ts` calling `getChannelPlugin(channel).actions.resolveCliActionRequest`, and `extensions/telegram/src/channel-actions.ts` already maps `thread-create` to `topic-create` with `threadName` -> `name`, but `extensions/telegram/src/channel.ts` wraps `telegramMessageActionsImpl` without forwarding `resolveCliActionRequest`. Forward the resolver through the exported Telegram plugin action wrapper, preserving runtime override behavior where appropriate, keep non-Telegram `thread-create` unchanged, and add regression coverage using the real Telegram plugin or bundled plugin path so `openclaw message thread create --channel telegram --thread-name Foo --dry-run` resolves to `topic-create` with `name` and without `threadName`. Do not make a broad runtime `thread-create` alias the primary fix. Add a changelog entry if the repair lane owns release metadata for this user-visible CLI bug.
