Keep open: current main still has the Telegram CLI bridge gap, and the direct repair PRs that referenced this issue are closed unmerged. The bug is narrow, source-proven, and still needs a focused fix.

## What This Changes

Make `openclaw message thread create --channel telegram` remap CLI `thread-create` requests to Telegram `topic-create` before gateway dispatch.

## Best Possible Solution

Forward `resolveCliActionRequest` through the exported Telegram actions wrapper and add regression coverage on `telegramPlugin.actions` so CLI normalization stays channel-owned.

## Reproduction Assessment

Yes, at source level. The CLI calls the exported channel hook, the exported Telegram wrapper lacks that hook, the fallback dispatches `thread-create`, and Telegram runtime rejects unsupported actions.

## Solution Assessment

Yes. The narrow fix is to forward the existing Telegram remap through the exported wrapper and test that public plugin surface; adding a runtime alias would bypass the intended CLI normalization seam.

## Repair Work Prompt

Fix the Telegram CLI thread-create remap for https://github.com/openclaw/openclaw/issues/81581. In current main, `src/cli/program/message/register.thread.ts` asks `getChannelPlugin('telegram').actions.resolveCliActionRequest` to remap `thread-create`, and `extensions/telegram/src/channel-actions.ts` already implements the `thread-create` to `topic-create` rewrite, but `extensions/telegram/src/channel.ts` exposes a wrapper that does not forward `resolveCliActionRequest`. Forward the hook through the exported Telegram actions wrapper, preserving runtime override/default behavior for other action methods. Add or update regression coverage that asserts `telegramPlugin.actions.resolveCliActionRequest({ action: 'thread-create', ... })` returns `topic-create` with `name` and without `threadName`, and keep the existing synthetic CLI remap coverage. Do not add a generic runtime alias for `thread-create`, do not change Telegram Bot API topic behavior, and do not touch unrelated Telegram reliability/topic routing work. Add a changelog entry only if maintainers require one during landing.
