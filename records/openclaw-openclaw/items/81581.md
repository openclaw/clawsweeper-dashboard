## Summary

Keep open: current main still has the reported Telegram CLI bridge bug. The CLI asks the selected channel plugin to remap `thread-create`, and the Telegram implementation has the remap, but the exported Telegram plugin wrapper still does not forward `resolveCliActionRequest`, so the bare action can reach Telegram handling and be rejected.

## What This Changes

The issue reports that Telegram `openclaw message thread create` dispatches `thread-create` instead of the Telegram-supported `topic-create` action.

## Best Possible Solution

Forward the existing Telegram CLI remap through `telegramPlugin.actions`, add a focused regression test against the real Telegram plugin wrapper, and keep the existing topic result contract backward-compatible.

## Reproduction Assessment

Yes, by source inspection on current main: the CLI asks the Telegram plugin for a `thread-create` remap, but the exported Telegram wrapper does not expose the hook that already performs that remap. A live bot token is useful for final proof, but it is not needed to establish this source-level failure path.

## Solution Assessment

Yes, the narrowest maintainable fix is to forward `resolveCliActionRequest` through the Telegram channel action wrapper and test `telegramPlugin.actions` directly. Adding a broad runtime alias for `thread-create` would bypass the channel-owned CLI normalization seam and is less targeted.
