The issue asks for `openclaw message thread create --channel telegram` to remap CLI `thread-create` requests to Telegram `topic-create` before gateway dispatch.

## Best Possible Solution

Forward the existing Telegram CLI remap through the exported Telegram plugin action wrapper and add regression coverage on `telegramPlugin.actions`, keeping any topic result field changes additive.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the CLI requests a channel-owned remap, the exported Telegram wrapper lacks that hook, the fallback dispatches `thread-create`, and the Telegram runtime rejects unsupported actions.

## Solution Assessment

Yes. The narrow maintainable fix is to forward `resolveCliActionRequest` through the exported Telegram action wrapper and test that public plugin surface, not to add a core Telegram special case or a broad runtime alias.
