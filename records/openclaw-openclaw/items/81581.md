## Summary

Keep open: current main still has the Telegram CLI/plugin action bridge gap, and the direct repair PRs that reference this issue are closed unmerged. The bug is narrow, source-proven, and suitable for a focused repair PR.

## What This Changes

The issue reports that Telegram `openclaw message thread create` dispatches `thread-create` instead of remapping to `topic-create`, causing the gateway to reject the action.

## Best Possible Solution

Forward `resolveCliActionRequest` through the exported Telegram plugin action wrapper and add a regression test on `telegramPlugin.actions`, preserving the existing `topic-create` runtime contract.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the CLI asks the exported Telegram plugin for a remap, the exported wrapper lacks the hook, the CLI falls back to `thread-create`, and the Telegram runtime rejects that action.

## Solution Assessment

Yes. The narrowest maintainable fix is to forward the existing channel-owned resolver through the exported Telegram action wrapper and test that public plugin surface; adding a broad runtime alias would bypass the established CLI normalization seam.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81581. Current main already has `extensions/telegram/src/channel-actions.ts` mapping CLI `thread-create` to Telegram `topic-create`, but the exported `telegramMessageActions` wrapper in `extensions/telegram/src/channel.ts` does not forward `resolveCliActionRequest`, so `openclaw message thread create --channel telegram` can still dispatch bare `thread-create` and fail. Add the missing forwarding through the exported Telegram plugin action wrapper, add or update regression coverage against `telegramPlugin.actions` in `extensions/telegram/src/channel-actions.contract.test.ts`, and keep the fix inside the Telegram plugin boundary. Do not solve this by adding a broad `thread-create` alias to the Telegram runtime unless a maintainer explicitly asks for that fallback. Include a changelog entry if the repair lane is expected to handle user-facing fixes.
