## Summary

Keep open: the report is source-reproducible on current main. The CLI already asks the selected channel plugin to remap `thread-create`, and the Telegram action implementation has the remap, but the exported Telegram channel action wrapper does not forward `resolveCliActionRequest`, so the generic `thread-create` can still reach Telegram action handling.

## What This Changes

The issue asks for `openclaw message thread create --channel telegram` to remap `thread-create` into Telegram `topic-create` and create a forum topic instead of failing as unsupported.

## Best Possible Solution

Forward the existing Telegram CLI remap through `telegramPlugin.actions`, cover the real Telegram wrapper with a focused regression test, and preserve the existing topic result contract while exposing any new ID alias additively.

## Reproduction Assessment

Yes, by source inspection: the CLI asks the channel plugin for a `thread-create` remap, but the real Telegram plugin wrapper does not expose the existing remap hook. That makes the reported `Unsupported Telegram action: thread-create` path credible on current main without needing a live bot token.

## Solution Assessment

Yes, the narrowest maintainable fix is to forward `resolveCliActionRequest` through the Telegram channel action wrapper and test the real Telegram plugin surface. Adding a broad runtime alias for `thread-create` would bypass the existing channel-owned CLI normalization seam and is less targeted.

Fix the Telegram CLI thread-create bridge reported at https://github.com/openclaw/openclaw/issues/81581. Current main already has `resolveThreadCreateRequest` calling `getChannelPlugin(channel)?.actions?.resolveCliActionRequest` and `extensions/telegram/src/channel-actions.ts` already maps `thread-create` to `topic-create`, but `extensions/telegram/src/channel.ts` wraps `telegramMessageActionsImpl` without forwarding `resolveCliActionRequest`, so the real Telegram plugin drops the remap. Forward that hook through the Telegram action wrapper, preferring runtime override behavior consistent with the other wrapper methods if applicable. Add focused regression coverage against the real Telegram plugin action surface, not only a synthetic remapping channel. Preserve existing `topicId` output; if adding `messageThreadId` or `message_thread_id` for the issue acceptance, make it additive. Add a user-facing changelog entry. Do not move Telegram topic behavior into core, do not add a broad generic runtime alias unless the wrapper seam cannot work, and do not change Telegram auth or group permission policy.
