## Summary

Keep open: current main and the latest release still expose the Telegram CLI remap only on the static action adapter, while the exported Telegram plugin wrapper that the CLI calls does not forward it. That leaves a source-proven path where bare `thread-create` reaches Telegram action handling and is rejected as unsupported.

## What This Changes

The issue asks for `openclaw message thread create --channel telegram` to remap `thread-create` to Telegram `topic-create` and create a forum topic.

## Best Possible Solution

Forward the existing Telegram CLI remap through `telegramPlugin.actions`, cover that exported plugin surface with a focused regression test, and preserve the current topic result contract.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.12` shows the CLI calls the exported plugin hook, the static Telegram remap exists, and the exported wrapper does not expose that hook, so the fallback dispatches bare `thread-create`.

## Solution Assessment

Yes. The narrowest maintainable fix is to forward `resolveCliActionRequest` through the Telegram action wrapper and test `telegramPlugin.actions` directly; adding a broad `thread-create` runtime alias would bypass the channel-owned CLI normalization seam.
