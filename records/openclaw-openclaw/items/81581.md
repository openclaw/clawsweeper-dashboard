## Summary

Keep open: current main and the latest release still have the Telegram exported-action wrapper gap, and the direct repair PRs linked from the discussion are closed unmerged. The reporter’s CLI failure remains source-proven.

## What This Changes

The issue reports that Telegram `openclaw message thread create` dispatches `thread-create` to the gateway instead of remapping it to `topic-create`.

## Best Possible Solution

Forward the existing Telegram CLI remap through the exported Telegram plugin action wrapper and add regression coverage against `telegramPlugin.actions`, without adding a broad runtime alias.

## Reproduction Assessment

Yes. The issue is source-reproducible on current main: the CLI calls the exported plugin hook, the exported Telegram wrapper does not provide it, and the Telegram runtime rejects bare `thread-create`.

## Solution Assessment

Yes. The narrowest maintainable fix is to forward `resolveCliActionRequest` through the exported wrapper and test that public plugin surface; runtime aliasing would duplicate the channel-owned CLI normalization seam.
