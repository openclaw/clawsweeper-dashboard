## Summary

Keep open: current main still drops the initiating agent identity before Google Meet and Voice Call create outbound call records, and the linked open PR is the active fix path rather than a reason to close this issue early.

## What This Changes

The issue asks for google_meet and voice_call calls in multi-agent deployments to use the initiating agent instead of the single configured voice agent.

## Best Possible Solution

Review, repair, and land the linked per-agent routing PR or a narrow replacement that threads agentId through tool dispatch, gateway RPC, call creation, CallRecord persistence, and downstream response/realtime resolution while preserving existing config fallback.

## Reproduction Assessment

Yes by source inspection: ToolContext exposes agentId, but current Google Meet and Voice Call paths do not serialize, store, or consume it for outbound call handling. I did not run the live Twilio/Meet multi-agent flow in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to resolve the originating agent at call creation, persist it on CallRecord, and make response/realtime consumers prefer that per-call value with the existing configured-agent fallback for legacy calls.
