- `impact:session-state`: The issue is about losing the initiating agent identity across tool, gateway, and voice-call session state.

## Summary

Keep this issue open: current main still lacks per-call initiating-agent identity through the Google Meet to Voice Call path, and the open closing PR remains the implementation candidate rather than a merged fix.

## What This Changes

The issue reports that Google Meet and Voice Call outbound calls in multi-agent deployments use the configured Voice Call agent instead of the initiating agent.

## Best Possible Solution

Land or replace the linked routing fix after review so the initiating agent is captured at tool/RPC dispatch, stored on each CallRecord at creation, and used by response-generation and realtime consult paths.

## Reproduction Assessment

Yes, source-level. Current main has no per-call agentId in CallRecord, voicecall.start, direct voice_call initiation, or downstream Voice Call response/realtime paths, so the reported multi-agent routing loss follows from the code path.

## Solution Assessment

Yes for the issue direction: the maintainable fix is to capture the initiating agent at tool/RPC dispatch, persist it once on the call record, and consume that call-level identity downstream. The linked PR still needs conflict resolution and review before merge.
