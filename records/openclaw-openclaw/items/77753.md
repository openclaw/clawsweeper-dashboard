## Summary

Keep open: current main and the latest release still drop the initiating agent identity before Google Meet and Voice Call create/consume outbound call records, and the open closing PR is the active fix path rather than a reason to close the issue early.

## What This Changes

The issue asks OpenClaw to route Google Meet and Voice Call outbound voice sessions to the agent that initiated the tool call in multi-agent deployments.

## Best Possible Solution

Review, repair, and land https://github.com/openclaw/openclaw/pull/77763 or a narrow replacement that persists the initiating agent on CallRecord and makes downstream response/realtime paths prefer that per-call value with existing config fallback.

## Reproduction Assessment

Yes, by source inspection: ToolContext carries agentId, but the Google Meet and Voice Call outbound paths do not serialize, store, or consume it before falling back to voiceConfig/effectiveConfig agentId or main. I did not run the live Twilio/Meet multi-agent flow in this read-only review.

## Solution Assessment

Yes, the maintainable fix direction is to resolve the initiating agent at call creation, persist it on CallRecord, and centralize downstream resolution so legacy calls still fall back to configured/default agent behavior.
