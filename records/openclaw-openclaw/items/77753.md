## Summary

Keep this issue open: current main still lacks per-call agent identity across the Google Meet to Voice Call path, and the open closing PR is still an implementation candidate rather than a merged fix.

## What This Changes

The issue reports that Google Meet and Voice Call outbound calls in multi-agent deployments route to the configured default agent instead of the initiating agent.

## Best Possible Solution

Land or replace the linked routing fix so the initiating agent is captured at tool/RPC dispatch, stored on each CallRecord at creation, and used by response-generation and realtime consult paths.

## Reproduction Assessment

Yes, source-level. Current main has no per-call agentId in CallRecord, voicecall.start, or the voice_call initiate path, and downstream response paths still resolve the agent from config defaults.

## Solution Assessment

Yes for the issue direction: the maintainable fix is to capture the initiating agent at dispatch, persist it once on the call record, and consume that call-level identity downstream. The exact linked PR still needs conflict resolution and review before merge.
