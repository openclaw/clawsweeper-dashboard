## Label Justifications

- `P1`: The bug can misroute live Google Meet or Voice Call interactions to the wrong agent in multi-agent deployments.
- `impact:session-state`: The issue is about losing the initiating agent identity across the tool, gateway, and voice-call session path.
- `impact:message-loss`: Voice responses can be delivered from the wrong agent identity, effectively misrouting the user-facing call conversation.

## Summary

Current main and the latest release still lose the initiating agent across the Google Meet to Voice Call path, and the linked closing PR is open and conflicting, so this issue remains necessary.

## What This Changes

The issue requests that Google Meet and Voice Call outbound calls in multi-agent deployments route to the agent that initiated the call instead of the single configured voice agent.

## Best Possible Solution

Land or replace the linked implementation PR with a focused fix that records the initiating agent on each call and resolves downstream voice responses from that call-level identity.

## Reproduction Assessment

Yes. Current-main source inspection shows the tool/RPC path never serializes a per-call `agentId`, `CallRecord` has no `agentId`, and downstream response paths resolve from config instead.

## Solution Assessment

Yes. The maintainable fix is to capture the initiating agent at tool/RPC dispatch, persist it on `CallRecord` at creation, and consume that call-level identity in response generation and realtime consult paths.
