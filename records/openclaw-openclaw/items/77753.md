## Label Justifications

- `P1`: The issue affects a real multi-agent voice-call workflow where calls from different users can be routed to the wrong agent identity.
- `impact:message-loss`: The reported failure misroutes voice-call and Google Meet talk-back behavior to the configured default agent instead of the initiating user's agent.
- `impact:session-state`: The failure mis-associates call transcripts, voice responses, and agent workspace/session context with `main` or another configured default.

## Summary

Current main and the latest release still do not preserve the initiating agent through the Google Meet to Voice Call path, and the open linked fix PR is the active implementation candidate, so this issue should stay open until that PR or a narrow replacement lands.

## What This Changes

Route `google_meet` and `voice_call` outbound voice calls to the agent that initiated the call in multi-agent deployments instead of the configured Voice Call default agent.

## Best Possible Solution

Land a rebased, narrow fix through https://github.com/openclaw/openclaw/pull/77763 or a replacement that threads `ctx.agentId` into `CallRecord.agentId` and centralizes downstream resolution without broadening the gateway trust boundary.

## Reproduction Assessment

Yes, source-reproducible: current main still never carries `ctx.agentId` into `voicecall.start` or `CallRecord`, and downstream response generation reads `voiceConfig.agentId ?? "main"`. I did not run a live Twilio or Google Meet reproduction in this read-only review.

## Solution Assessment

Yes, the issue's proposed invariant is the right direction: resolve the call agent once at call creation and consume that persisted value downstream. The linked PR still needs repair/review before merge because its current branch has broader gateway-boundary concerns.
