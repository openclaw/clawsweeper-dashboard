## Summary

Keep open: current main still drops the initiating tool context before the Google Meet to Voice Call path and downstream voice response code still resolves a configured/static agent instead of the call's originating agent.

## What This Changes

Request to make Google Meet and Voice Call outbound calls preserve the initiating multi-agent identity through RPC, call records, and downstream response generation.

## Best Possible Solution

Thread the initiating agentId through Google Meet and Voice Call call creation, persist it on CallRecord, and use one shared resolver downstream while preserving the existing config fallback for legacy and single-agent calls.

## Reproduction Assessment

Yes, by source inspection: multi-agent routing defines an agent-scoped workspace/session, but current tool and gateway paths drop toolContext.agentId before creating voice calls. I did not run a live Twilio/Meet scenario in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to resolve the originating agent once at call creation, store it on CallRecord, and read call.agentId in response/realtime code. The proposal should skip reworking google_meet registration beyond adding the missing factory opts/agent payload because that tool is already factory-registered on current main.
