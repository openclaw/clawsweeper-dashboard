## Label Justifications

- `P1`: The issue affects a real multi-agent voice-call workflow where calls from different users can be routed to the wrong agent identity.
- `impact:session-state`: The failure can mis-associate calls, transcripts, and workspace/session context with the configured default agent.
- `impact:message-loss`: Voice replies can be misrouted through the wrong agent identity instead of the initiating user's agent.

## Summary

Current main and the latest release still do not preserve the initiating agent through Google Meet and Voice Call outbound routing, and the linked closing PR is still open/conflicting, so this issue remains necessary.

## What This Changes

The issue requests routing Google Meet Twilio joins and direct Voice Call initiations to the agent that initiated the call in multi-agent deployments.

## Best Possible Solution

Land a rebased, narrow fix through the linked PR or a replacement that carries the initiating agent id into CallRecord and centralizes downstream resolution without broadening the gateway trust boundary.

## Reproduction Assessment

Yes. Source inspection on current main shows agentId is not serialized into googlemeet.join/testSpeech or voicecall.start, CallRecord has no agentId field, and downstream voice response still falls back to voiceConfig.agentId ?? "main".

## Solution Assessment

Yes in direction, but not complete on main. The maintainable solution is to resolve and persist the call agent once at call creation and consume that per-call value downstream; the existing linked PR still needs repair before merge.
