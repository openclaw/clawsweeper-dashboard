## Summary

Keep open: current main still has a source-confirmed multi-agent identity bug in the Google Meet to Voice Call path and in direct `voice_call` outbound calls, where initiating tool context is not persisted and downstream voice response code falls back to static config or `main`.

## What This Changes

Ensure Google Meet and Voice Call outbound calls in multi-agent deployments use the initiating agent instead of a single configured voice response agent.

## Best Possible Solution

Thread the initiating agentId through Google Meet and Voice Call call creation, persist the resolved value on CallRecord, and use one shared resolver downstream while preserving existing config fallback for legacy and single-agent calls.

## Reproduction Assessment

Yes, by source inspection: multi-agent routing and plugin tool context expose the initiating agent, but current Google Meet and Voice Call call-creation paths drop it before storing or using the voice call. I did not run a live Twilio/Meet scenario in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to resolve the originating agent at call creation, persist it on CallRecord, and read that value in response generation and realtime consult code. The proposal should account for current main already having `google_meet` factory registration, so the missing work is propagation and persistence rather than converting that tool from static form.
