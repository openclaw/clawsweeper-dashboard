## Summary

Keep open: current main and the latest release still keep the WhatsApp watchdog thresholds as internal `monitorWebChannel` tuning, while the gateway start path and public config schema do not expose those values. The request is valid, but it is a new public config surface and should get maintainer product/owner review before implementation.

## What This Changes

The issue requests supported WhatsApp watchdog timeout configuration for `messageTimeoutMs`, `transportTimeoutMs`, and `watchdogCheckMs` instead of the current hardcoded monitor defaults.

## Best Possible Solution

Choose a maintainer-approved WhatsApp-owned config shape, then wire bounded watchdog values into `monitorWebChannel` with schema/docs/tests while leaving the lower-level stuck-session race tracked by https://github.com/openclaw/openclaw/issues/89115.

## Reproduction Assessment

Yes for source-level reproduction: current main shows the hardcoded monitor defaults, the gateway caller omits timeout tuning, and the public schemas/docs expose only socket timing. A live 30-minute WhatsApp idle run was not performed in this read-only review.

## Solution Assessment

Unclear: exposing config is a plausible fix, but the repository has a high bar for new config surfaces and maintainers should first decide whether to expose watchdog tuning or change the watchdog policy itself.
