## Summary

## What This Changes

The issue asks OpenClaw to apply DeepSeek V4 thinking-mode `reasoning_content` request handling to DeepSeek models reached through generic OpenAI-compatible proxy providers.

## Best Possible Solution

Keep one canonical implementation path in https://github.com/openclaw/openclaw/issues/75105, likely by wiring the existing model-level `compat.thinkingFormat: "deepseek"` signal into the shared DeepSeek V4 OpenAI-compatible wrapper with docs, changelog, and focused payload tests.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: configure a custom `openai-completions` provider with a DeepSeek V4 model and thinking enabled, and current main does not install the DeepSeek V4 payload wrapper unless a named provider plugin owns the route. I did not run a live proxy request in this read-only review.

## Solution Assessment

No, this should not be solved as a separate issue. The safer maintainable path is the existing canonical model-level compat work in https://github.com/openclaw/openclaw/issues/75105 rather than adding another one-off provider or exact-id fallback here.
