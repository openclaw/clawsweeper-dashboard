## Summary

Keep open: current main still applies the DeepSeek V4 replay wrapper only through provider-owned routes, while generic `models.providers` OpenAI-compatible proxy routes can fall through unwrapped; the new Fireworks comment adds live evidence that this remains user-visible.

## What This Changes

Apply DeepSeek V4 `reasoning_content` replay and stream handling to generic OpenAI-compatible proxy provider routes configured through `models.providers`.

## Best Possible Solution

Add a guarded OpenAI-compatible DeepSeek V4 fallback at the provider stream-wrapper seam, reuse the existing shared wrapper, and leave broader arbitrary per-model opt-in policy to https://github.com/openclaw/openclaw/issues/75105.

## Reproduction Assessment

Yes at source level: a custom `models.providers` OpenAI-compatible DeepSeek V4 model has no owning runtime plugin, so current main leaves the outgoing replay payload unpatched. I did not run a live external proxy roundtrip in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse `createDeepSeekV4OpenAICompatibleThinkingWrapper` behind an `openai-completions` DeepSeek V4 guard, without importing DeepSeek plugin internals into core or adding a new config knob.
