## Summary

Keep open: current main still has the reported DeepSeek V4 proxy gap. Direct DeepSeek, OpenRouter, OpenCode Go, and Venice have provider-specific replay handling, but generic or differently owned OpenAI-compatible routes still do not get the wrapper that backfills DeepSeek `reasoning_content`, and the issue now includes live Fireworks evidence in the same area.

## What This Changes

The issue requests DeepSeek V4 `reasoning_content` replay handling for models reached through generic OpenAI-compatible `models.providers` proxy routes.

## Best Possible Solution

Add a guarded OpenAI-compatible DeepSeek V4 proxy wrapper path that reuses the shared helper, respects provider-specific wrappers, matches exact or path-segment DeepSeek V4 model ids safely, and includes regression coverage for unowned custom providers plus Fireworks-style model ids.

## Reproduction Assessment

Yes at source level: current main leaves generic or non-DeepSeek-owned `openai-completions` DeepSeek V4 routes without the wrapper that backfills `reasoning_content`. The thread also includes live Fireworks/OpenClaw evidence on v2026.5.7, but I did not run an external proxy roundtrip in this read-only pass.

## Solution Assessment

Yes in direction: the narrow maintainable fix is to compose the existing shared DeepSeek V4 OpenAI-compatible wrapper at the provider stream seam or from existing model compat metadata, without importing DeepSeek extension internals into core or adding more provider-id hacks.
