## Summary

Keep open: current main still has the DeepSeek V4 replay wrapper, but custom or non-matching OpenAI-compatible proxy routes can still miss it, and the newer Fireworks reproduction adds live evidence in the same provider/runtime area.

## What This Changes

The issue requests DeepSeek V4 `reasoning_content` handling for DeepSeek V4 models reached through OpenAI-compatible proxy providers configured with `models.providers`.

## Best Possible Solution

Add a narrow OpenAI-compatible DeepSeek V4 compat path that covers custom/unowned proxy routes and provider-owned proxy model ids without adding a new config option or hard-coding unrelated provider families.

## Reproduction Assessment

Yes at source level: current main leaves generic or non-matching OpenAI-compatible DeepSeek V4 proxy routes without the wrapper that backfills `reasoning_content`. The Fireworks comment also provides live user evidence, but I did not run an external proxy roundtrip in this read-only pass.

## Solution Assessment

Yes in direction: the existing shared DeepSeek V4 wrapper and provider stream hook seam are the right place to fix this. The implementation should be more careful than a broad provider-id patch, especially for Fireworks-style model ids and proxies that may reject native DeepSeek thinking controls.
