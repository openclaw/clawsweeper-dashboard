## Summary

Keep open: current main still lacks a DeepSeek V4 reasoning-content wrapper for generic or proxy OpenAI-compatible routes when no provider hook handles that exact model, and the newer Fireworks reproduction adds live evidence in the same routing area.

## What This Changes

The issue requests DeepSeek V4 `reasoning_content` handling for V4 models reached through custom or proxy OpenAI-compatible providers.

## Best Possible Solution

Add a guarded OpenAI-compatible DeepSeek V4 stream-wrapper path for proxy routes, plus any matching Fireworks response handling if verified, with regression coverage that leaves non-DeepSeek and non-OpenAI transports untouched.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a proxy `openai-completions` DeepSeek V4 model can reach `wrapProviderStreamFn` with no applicable wrapper, and `applyExtraParamsToAgent` then keeps the unpatched stream; I did not live-test the external proxy.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: compose the existing shared DeepSeek V4 OpenAI-compatible wrapper at the provider hook/fallback seam, gated to exact V4 ids and `openai-completions`, rather than importing DeepSeek plugin internals or adding a user config knob.
