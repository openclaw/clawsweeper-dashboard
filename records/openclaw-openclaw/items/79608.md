## Summary

Keep open: current main still has the DeepSeek V4 replay wrapper for direct or provider-owned routes, but a generic unowned OpenAI-compatible proxy can still fall through without the `reasoning_content` backfill; the linked fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to apply DeepSeek V4 `reasoning_content` request shaping when DeepSeek V4 is reached through custom `models.providers` OpenAI-compatible proxy providers.

## Best Possible Solution

Add a guarded OpenAI-compatible DeepSeek V4 replay wrapper for unowned proxy routes, reusing the existing shared wrapper and covering exact DeepSeek V4 ids plus the existing safe compat signal for reseller-style ids without hard-coding individual proxy providers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: a custom `models.providers` OpenAI-compatible DeepSeek V4 model has no owning runtime plugin, so `wrapProviderStreamFn` returns `undefined` and the runner preserves the unpatched stream.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: route only OpenAI-compatible DeepSeek V4 proxy traffic through the existing shared DeepSeek V4 wrapper, rather than importing DeepSeek extension internals into core or adding another proxy-provider id list.
