## Summary

Keep open: the linked schema gap is valid, but this draft PR still needs stronger real behavior proof and a small correctness fix because the new description names `webchat`, which current outbound routing treats as internal-only rather than a deliverable provider.

## What This Changes

The PR adds provider-id guidance and a registry-derived enum to the message tool `channel` schema, with focused regression coverage for the schema shape.

## Best Possible Solution

Land a narrow schema/test update that describes `channel` as configured deliverable provider ids, derives any enum from loaded channel plugins, excludes internal-only `webchat` from examples, and verifies the schema in a real OpenClaw run.

## Reproduction Assessment

Yes for the source-level problem: current main exposes `channel` as `Type.Optional(Type.String())`, and the linked issue describes resulting `Unknown channel` errors when agents pass Slack/Discord target ids. I did not run a live Slack or Discord workspace.

## Solution Assessment

No, not yet: a plugin-derived enum plus schema guidance is the right shape, but the description should not list internal-only `webchat` as a provider example and the after-fix proof should exercise the real tool schema path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Remove internal webchat from provider examples:** `src/agents/tools/message-tool.ts:119-120`
  - body: The new schema text lists `webchat` as a channel provider example, but outbound routing treats `webchat` as an internal source-reply channel, not a deliverable provider. If a model follows this text and sets `channel: "webchat"`, that explicit route bypasses the internal source-reply sink and can fail instead of replying to the current run. Please remove `webchat` from this provider-id example or handle it separately from deliverable channel ids.
  - confidence: 0.86
