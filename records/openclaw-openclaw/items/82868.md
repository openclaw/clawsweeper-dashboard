## Summary

Keep open: this is a focused fix for a real message-tool schema gap, but the draft PR still has a blocking wording defect and its supplied proof is only a source-level probe rather than a real OpenClaw tool/schema invocation.

## What This Changes

The PR changes the message tool `channel` schema from an unconstrained string to a described provider-id enum derived from loaded channel plugins, and adds a focused schema regression test.

## Best Possible Solution

Land or replace this with a narrow schema/test update that describes `channel` as a deliverable provider or channel-plugin id, derives any enum from the actual registered deliverable channel surface, excludes internal-only `webchat` from provider examples, and verifies the real message-tool schema path.

## Reproduction Assessment

Yes for the source-level issue: current main and the latest release expose `channel` as a bare optional string while routing expects provider/plugin ids. I did not run a live Slack or Discord workspace repro in this read-only review.

## Solution Assessment

No, not yet: schema guidance plus a plugin-derived enum is the right direction, but the provider examples should not include internal-only `webchat` and the PR still needs real after-fix behavior proof.

## Review Findings
