## Summary

Keep open. Current main still resolves one Telegram inbound route for the receiving account and has no opt-in cross-account fan-out config, while related #63380 only addresses account-level `agentId` validation for #62985.

## What This Changes

Request an opt-in Telegram multi-account routing mechanism so bot-to-bot group messages can reach other configured OpenClaw accounts while preserving separate bot identities.

## Best Possible Solution

Design an additive, opt-in Telegram or generalized broadcast routing model with destination allowlists, identity preservation, loop prevention, docs, and focused multi-account group coverage.

## Reproduction Assessment

Yes for the source-level OpenClaw routing gap: current code resolves one Telegram route for the receiving account and a named-account group test expects the fallback path to drop. I did not run a live Telegram Bot-to-Bot Mode group test in this read-only review.

## Solution Assessment

Unclear. `forwardToAccounts` is plausible, but the narrowest maintainable fix depends on maintainer choice between Telegram-specific allowlisted fan-out, a Telegram expansion of broadcast groups, or documenting the group pattern as unsupported.
