## Summary

Keep open. Current main still routes Telegram inbound updates to one route for the receiving account, drops named-account group fallback without an explicit binding, and has no Telegram `forwardToAccounts`/`broadcastToAccounts` fan-out config; related #63380 is narrower account-level `agentId` schema work for #62985 and does not resolve this bot-to-bot group routing request.

## What This Changes

Request opt-in Telegram multi-account routing so a group message received through one bot account can be forwarded or broadcast to other bot accounts or agents without sharing one bot token.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with explicit destination allowlists, preserved account identity, bot-authored message handling, loop prevention, docs, and focused multi-account group coverage; if unsupported, document the Telegram channel-post or single-account alternatives clearly.

## Reproduction Assessment

Yes for the OpenClaw routing gap: the existing named-account group test sets a non-default Telegram account, forces mention/activation, and currently expects `null`. Live Telegram bot-to-bot delivery still needs validation before final product design.

## Solution Assessment

Unclear. The requested forward/broadcast direction is plausible, but the best maintainable fix requires maintainer choice between Telegram-specific routing, generalized broadcast groups, or documenting the pattern as unsupported.
