## Summary

Keep open. Current main still has single-target Telegram inbound routing and intentionally drops named-account group fallback, while related #63380 only addresses account-level `agentId` validation for #62985 and does not add cross-account fan-out.

## What This Changes

Request opt-in Telegram multi-account routing so group messages received by one bot account can activate other configured bot accounts or agents while preserving per-bot identity.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or generalized broadcast routing model with explicit destination allowlists, identity preservation, loop prevention, docs, and focused multi-account group coverage; if maintainers reject the pattern, document the supported channel or single-account alternatives.

## Reproduction Assessment

Yes for the OpenClaw routing gap: source inspection shows the receiving Telegram account resolves one route and the named-account group test currently expects `null` without an explicit binding. Live Telegram bot-to-bot group delivery still needs validation before product design is finalized.

## Solution Assessment

Unclear. The requested `forwardToAccounts` or broadcast direction is plausible, but the narrowest maintainable solution depends on maintainer choice between Telegram-specific account fan-out, extending broadcast groups, or documenting the group pattern as unsupported.

  - body: Current Telegram handling skips self-authored messages for the same bot account; cross-account fan-out would also need rules for other bot-authored messages so multiple bot accounts cannot trigger duplicate replies or loops.
