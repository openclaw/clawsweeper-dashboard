## Summary

Keep open. Current main still lacks Telegram cross-account or broadcast routing, and the related open PR only addresses a narrower account-level `agentId` validation regression; the requested fan-out affects bot identity, authorization, and loop prevention, so it needs maintainer design before implementation.

## What This Changes

Request an opt-in Telegram routing feature that forwards or broadcasts inbound group messages from one bot account to other configured bot accounts or agents while preserving separate bot identities.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with explicit allowlists, identity preservation, loop prevention, docs, and focused multi-account group coverage; otherwise document the unsupported multi-bot group pattern clearly.

## Reproduction Assessment

Not applicable as a feature request; source inspection gives high confidence current main does not support it. The Telegram path builds one route for the receiving account and drops named-account group fallback, and tests assert that behavior.

## Solution Assessment

Unclear. `forwardToAccounts` may be viable, but maintainers should first choose between Telegram-specific account forwarding, generalized broadcast groups, or explicitly documenting this as unsupported because the choice affects identity and authorization policy.

## Risks / Open Questions

- A naive fan-out model could create duplicate replies or reply loops unless bot-authored messages, self-echoes, mention handling, and Telegram bot visibility rules are designed explicitly.
- Account identity and authorization semantics need an explicit allowlist so one bot account cannot implicitly authorize another agent/account to process or answer a group message.
- Open PR #63380 is adjacent but only handles account-level `agentId` validation for #62985, so it should not be treated as the canonical fix for this cross-account routing request.
