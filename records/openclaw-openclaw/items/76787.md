## Summary

Keep open. Current main still passes Telegram `t.me/c/...` links through as ordinary inbound text and exposes no Telegram fetch/link-resolution action; the request is useful but needs maintainer policy because it would inject historical chat content and may require Bot API operations with visible side effects.

## What This Changes

Resolve Telegram `t.me/c/<chat>/<thread>/<message>` links in inbound Telegram messages and inject the referenced message content into agent context.

## Best Possible Solution

Design a Telegram plugin-owned, opt-in resolver that parses `t.me/c` links, prefers already-ingested retained history when possible, and uses any Bot API fallback only under a documented access, retention, and side-effect policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Telegram message containing a `t.me/c/...` URL flows through formatting-only text expansion into `BodyForAgent` and `RawBody`; I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best maintainable solution because maintainers first need to choose privacy, authorization, retention, and Telegram-side-effect policy; an opt-in resolver that prefers local history is safer.
