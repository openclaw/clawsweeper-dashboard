## Label Justifications

- `P3`: This is a useful Telegram channel capability request, but it is a product/security design item rather than an urgent regression or broken runtime.
- `impact:session-state`: The requested behavior would inject fetched Telegram message content into the session context and transcript-visible prompt path.

## Summary

Keep open: current main and v2026.6.6 still pass Telegram `t.me/c/...` links through as ordinary message text, and the remaining work is a security-sensitive Telegram context-fetch design rather than an already-implemented fix.

## What This Changes

The issue requests Telegram `t.me/c/<chat>/<thread>/<message>` link resolution so referenced message content can be fetched and injected into the active session context.

## Best Possible Solution

Design this as a Telegram-plugin-owned context feature layered on the canonical Telegram read/history work, with scoped link parsing, observed-message limits, contextVisibility/allowlist enforcement, bounded retention, docs, tests, and redacted live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main renders inbound Telegram text into `rawBody`/`bodyText` and the Telegram message-action adapter does not advertise or map any read/fetch action that could resolve a `t.me/c/...` reference.

## Solution Assessment

Unclear as proposed. The issue identifies the right user problem, but the best implementation needs maintainer approval for observed-history semantics, link-scope authorization, and whether automatic context injection should exist on top of the broader Telegram read action.
