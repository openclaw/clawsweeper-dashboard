## Summary

Keep open. Current main still leaves Telegram `t.me/c/...` links as ordinary inbound text and has no Telegram read/fetch/link-resolution action, while the remaining work is a new Telegram feature that needs maintainer privacy and side-effect policy.

## What This Changes

Resolve Telegram `t.me/c` message links by retrieving the referenced Telegram message and injecting its content into session context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested history when available, applies existing supplemental-context visibility rules, and documents any Bot API fallback before injecting linked content.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: a Telegram message containing a `t.me/c/...` URL flows through text expansion into `BodyForAgent`, and no Telegram read/fetch resolver is exposed; I did not run a live bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best maintainable fix because maintainers first need to choose the privacy, authorization, retention, and Telegram-side-effect policy.

## Security Review

Status: needs_attention

Summary: Resolving Telegram message links is security-sensitive because it can inject historical chat content into an agent turn.

Concerns:

- **[medium] Define linked-message access policy first:**
  - body: Automatic `t.me/c` resolution can expose prior group or DM content to the agent, so maintainers should define who may trigger it, which chats/messages are eligible, retention behavior, and the safe default before implementation.
  - confidence: 0.88
- **[medium] Validate Bot API retrieval side effects:**
  - body: The proposed copy/forward/delete path may create visible messages or fail cleanup; grammY also shows `copyMessage` returns only a sent message id, so any implementation needs explicit handling for permissions, deletion failures, and observable side effects.
  - confidence: 0.8
