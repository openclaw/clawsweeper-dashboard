## Summary

Keep open. Current main still treats Telegram `t.me/c/...` links as ordinary inbound text and exposes no Telegram read/fetch/resolve action, while the requested resolver needs a Telegram-owned privacy and side-effect policy before implementation.

## What This Changes

Add Telegram plugin support for resolving `t.me/c/<chat_id>/<thread>/<msg_id>` links into referenced message content in session context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested cached messages when available and only uses Bot API copy/forward behavior behind explicit privacy policy, docs, and regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: inbound Telegram text flows through `expandTextLinks` into `rawBody`/`bodyText`, then `BodyForAgent`, and no Telegram action/docs surface exposes a message fetch resolver.

## Solution Assessment

Unclear. The suggested Bot API copy/forward-and-delete path is plausible but not clearly the best default until maintainers define privacy, permissions, caching, and side-effect policy.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because resolving Telegram message links can expose historical chat content and may require side-effecting Bot API operations.

Concerns:

- **[medium] Define access policy before resolving linked messages:**
  - body: Automatic `t.me/c` resolution could inject content the current user did not paste directly, including group or DM history, and copy/forward based retrieval may create visible Telegram side effects. Maintainers should define opt-in scope, authorization, retention, and failure behavior before implementation.
