## Summary

Keep open. Current main still passes Telegram `t.me/c/...` links through as message text, and resolving them would be a new Telegram capability with access, retention, opt-in, and Bot API side-effect policy to settle first; the prior closed history/read-action PR is useful context but does not make this issue implemented or superseded.

## What This Changes

The issue requests resolving Telegram `t.me/c/...` message links into referenced message content in the Telegram agent context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-observed message cache data and only adds Bot API fallback behavior after maintainers define access, retention, and side-effect policy.

## Reproduction Assessment

Yes for the current behavior by source inspection: a Telegram message containing a `t.me/c/...` URL flows through text expansion into `rawBody` and `bodyForAgent` with no resolver or read/fetch action on current main. I did not run a live Telegram bot because the missing path is clear in source.

## Solution Assessment

No. The proposed copy/forward/delete retrieval path is not yet the best maintainable solution because privacy, authorization, retention, and Telegram-visible side effects need maintainer policy before implementation.
