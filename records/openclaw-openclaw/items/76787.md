## Summary

Keep open. Current main still passes Telegram message links through inbound text and exposes no Telegram read/link-resolution action; this is a new Telegram plugin feature with privacy and side-effect policy decisions.

## What This Changes

The issue asks the Telegram plugin to resolve `t.me/c/...` message links and inject the referenced message content into agent context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that first respects existing history/context-visibility policy and only uses Bot API retrieval through an explicitly documented, side-effect-aware path.

## Reproduction Assessment

Yes. Source inspection shows a `t.me/c/...` link would flow through `expandTextLinks` into `BodyForAgent` as text, while the Telegram action/config/docs surfaces have no resolver; I did not run a live Telegram bot.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best fix until maintainers decide the privacy, permission, retention, and side-effect policy; a gated Telegram plugin resolver is the safer direction.

## Security Review

Status: needs_attention

Summary: Resolving linked Telegram messages would inject historical chat content that was not directly provided in the current turn.

Concerns:

- **[medium] Define linked-message access policy first:**
  - body: Automatic `t.me/c` resolution can expose prior group or DM content to the agent, so the plugin needs an explicit policy for who may trigger it, which chats/history may be read, and whether it is enabled by default.
  - confidence: 0.86
- **[medium] Handle Bot API retrieval side effects:**
  - body: The proposed copy/forward-style retrieval can create visible messages or depend on deletion success, so any implementation should model failures, protected content, and auditability instead of silently treating it as a read-only fetch.
  - confidence: 0.78
