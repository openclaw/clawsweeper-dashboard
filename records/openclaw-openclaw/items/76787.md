## Summary

Keep open. Current main still treats Telegram `t.me/c/...` links as ordinary inbound text and exposes no Telegram read/fetch/link-resolution action, while resolving historical messages needs maintainer privacy and Bot API side-effect policy before implementation.

## What This Changes

Add Telegram handling that parses `t.me/c/...` message links, retrieves the referenced Telegram message, and injects the resolved content into agent context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested history when possible, documents any Bot API fallback, and gates historical content injection by clear access and retention policy.

## Reproduction Assessment

Yes for the current behavior. Source inspection shows a Telegram message containing a `t.me/c/...` URL flows through text expansion into `BodyForAgent`, and no Telegram read/fetch resolver is exposed; I did not run a live bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best maintainable fix until maintainers choose the privacy, authorization, retention, and Telegram-side-effect policy.
