The issue requests resolving Telegram `t.me/c/...` message links and injecting the referenced message content into the session context.

## Best Possible Solution

Keep this open for a Telegram-plugin-owned context-fetch design with explicit authorization, bounded retention, topic/account scoping, docs, focused tests, and live Telegram proof.

Yes, at source level. Current main renders inbound Telegram text into `rawBody`/`bodyText` and has no Telegram read/fetch/link-resolution action, so a `t.me/c/...` link remains ordinary message text.

No approved best implementation exists yet. The issue identifies a real user need, but the safest solution needs maintainer approval for observed-history semantics, link-scope authorization, retention, and automatic context injection before coding.
