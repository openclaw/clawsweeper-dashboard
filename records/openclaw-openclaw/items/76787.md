## Summary

Keep open. Current main still passes Telegram `t.me/c/...` links through as ordinary inbound text, and resolving linked historical messages is a new Telegram plugin capability with privacy and permission policy decisions still open.

## What This Changes

The issue requests parsing Telegram `t.me/c/...` links, retrieving the referenced message, and injecting that content into Telegram session context.

## Best Possible Solution

Design a Telegram-owned, opt-in or policy-gated resolver that prefers already-ingested history when possible and documents any accepted Bot API fallback before injecting linked content.

## Reproduction Assessment

Yes. Source inspection shows incoming Telegram text is normalized through `expandTextLinks` into `rawBody`/`bodyText` and then `BodyForAgent`, while the action, config, and docs surfaces have no message-link fetch/read resolver.

## Solution Assessment

No. The requested capability is useful, but copy/forward/delete retrieval is not clearly the best path until maintainers define the privacy, permission, retention, and default-enable policy.

## Security Review

  - confidence: 0.86
- **[medium] Validate Bot API retrieval side effects:**
  - body: The proposed copy/forward-style retrieval path needs clear handling for visible messages, deletion failures, permissions, and media/text coverage before making resolution implicit.

## Risks / Open Questions

- Automatic link resolution could bypass existing Telegram `contextVisibility` and sender allowlist semantics if not designed as an explicit supplemental-context path.
- A Bot API copy/forward/delete retrieval strategy may have visible side effects, deletion failures, or incomplete media coverage that need policy and test coverage before enabling.
- The safe default is not established: always-on resolution, opt-in resolution, or history-only resolution would have materially different privacy behavior.
