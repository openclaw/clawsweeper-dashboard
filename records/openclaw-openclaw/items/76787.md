## Summary

Keep open. Current main still treats Telegram `t.me/c/...` message links as ordinary inbound text, and the requested linked-message retrieval is a new Telegram capability that needs explicit privacy and Bot API side-effect policy before implementation.

## What This Changes

The issue requests resolving Telegram `t.me/c/...` message links and injecting the referenced message content into Telegram session context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested history when available, documents any Bot API fallback, and gates historical content injection by clear chat, sender, retention, and side-effect policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: a Telegram message containing a `t.me/c/...` URL flows through text expansion into `BodyForAgent`, and no Telegram fetch or link resolver is exposed.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best maintainable fix because maintainers first need to choose the privacy, authorization, retention, and Telegram-side-effect policy.

## Security Review

  - body: The proposed copy/forward/delete retrieval path can create visible messages or leave cleanup/audit failures, so any implementation should handle permissions, deletion failures, and observable side effects explicitly.
