## Summary

Keep open. Current main still treats Telegram `t.me/c/...` links as ordinary inbound text and exposes no Telegram read/fetch/link-resolution action, while adding one is a new Telegram capability with historical-content privacy and Bot API side-effect policy decisions.

## What This Changes

Resolve Telegram `t.me/c/<chat>/<thread>/<message>` links in inbound Telegram messages and inject the referenced message content into agent context.

## Best Possible Solution

Add a Telegram-owned resolver only after maintainer policy is set, reusing the supplemental context visibility model where possible and documenting any opt-in Bot API retrieval behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the current behavior: a Telegram message containing a `t.me/c/...` URL is expanded as text into `rawBody`/`BodyForAgent`, and no Telegram read/fetch resolver is exposed; I did not run a live bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete approach is a plausible implementation detail, but the best fix should first settle privacy, authorization, retention, and visible-side-effect behavior, then implement it inside the Telegram plugin rather than core.

## Security Review

Status: needs_attention

Summary: Resolving Telegram message links is security-sensitive because it can inject historical chat content that was not directly submitted in the current turn.

  - confidence: 0.86

  - body: The proposed copy/forward/delete retrieval path can create visible messages or leave cleanup/audit failures, so any implementation should handle permissions, deletion failures, and observable side effects explicitly.
