## Summary

Keep open: current main and v2026.6.8 still do not parse Telegram `t.me/c/...` links, expose a Telegram read/fetch action, or inject linked message content into session context. The open Telegram read-action issue overlaps, but this issue remains a distinct product/security decision about automatic link expansion into model context.

## What This Changes

The issue requests resolving Telegram `t.me/c/<chat>/<thread>/<message>` links and adding the referenced message content to the Telegram session context.

## Best Possible Solution

Keep this as a Telegram-plugin-owned design item that aligns `t.me/c` resolution with the broader read/cache surface, explicit authorization, bounded retention, docs, focused tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a `t.me/c` parser, Telegram read/fetch action discovery, runtime dispatch, and linked-message supplemental context path, so the link remains ordinary inbound text.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The likely maintainable path is to align link resolution with Telegram read/cache semantics and explicit authorization rather than adding an independent copy/forward injection path first.
