## Summary

Keep open. Current main still passes Telegram `t.me/c/...` links through inbound text without dereferencing and exposes no Telegram read/fetch action; adding this would be a new security-sensitive Telegram capability that needs maintainer policy first.

## What This Changes

Resolve Telegram `t.me/c/...` message links and inject the referenced message content into Telegram session context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested visible history when possible and documents any Bot API fallback with explicit access, retention, and side-effect policy.

## Reproduction Assessment

Yes for the current behavior by source inspection: a Telegram message containing a `t.me/c/...` URL flows through text expansion into `BodyForAgent`/`RawBody`, and no resolver or read/fetch action exists. I did not live-run a Telegram bot in this read-only review.

## Solution Assessment

No for the exact proposed copy/forward/delete path as-is. Maintainers should first choose the privacy, authorization, retention, and Telegram-side-effect policy, then implement the narrow Telegram-owned resolver that matches that policy.
