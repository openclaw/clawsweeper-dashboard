## Summary

Keep open: current main and v2026.6.10 still do not resolve Telegram `t.me/c/...` links into session context. The open Telegram read-action issue overlaps, but automatic link expansion remains a distinct product and security decision.

## What This Changes

Request automatic Telegram `t.me/c/...` message-link parsing, message lookup, and referenced-content injection into agent context.

## Best Possible Solution

Keep this as a Telegram-plugin design item that aligns `t.me/c` expansion with a bounded message-read/cache contract, explicit authorization, docs, tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a `t.me/c` parser, Telegram read/fetch action wiring, and a linked-message supplemental context path, so the link remains ordinary inbound text.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The safer direction is to align link resolution with bounded Telegram read/cache semantics and explicit authorization instead of adding an independent hidden copy/forward injection path.
