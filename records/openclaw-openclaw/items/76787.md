## Summary

Keep open. Current main and the latest release still treat Telegram `t.me/c/...` links as ordinary inbound text; the adjacent observed-message cache helps replies and nearby context, but it does not resolve arbitrary Telegram message links or fetch referenced content.

## What This Changes

Resolve Telegram `t.me/c/...` message links and inject the referenced Telegram message content into the session context.

## Best Possible Solution

Add a Telegram-plugin-owned, opt-in resolver that first uses the observed-message cache when possible, then uses a bounded Bot API path only after maintainers define access, cleanup, media, retention, and context-visibility rules.

## Reproduction Assessment

Yes, from source rather than a live Telegram run: current inbound handling copies Telegram text into `rawBody`/`bodyText`, and neither current main nor v2026.5.27 has a `t.me/c` parser, fetch action, or copy/forward resolver.

## Solution Assessment

No, the requested copy/forward approach is plausible but not yet the best complete solution because the access, opt-in, cleanup, media, and side-effect policy needs to be settled first.
