## Summary

Keep open. Current main still has no Telegram `t.me/c/...` message-link resolver or fetch/read action, and automatically injecting referenced chat history is a new security-sensitive Telegram plugin capability that needs maintainer policy first.

## What This Changes

Request resolving Telegram `t.me/c` message links into referenced message content and injecting that content into Telegram session context.

## Best Possible Solution

Design a Telegram-owned, policy-gated resolver that prefers already-ingested history when possible and documents any accepted Bot API fallback before adding linked content to agent context.

## Reproduction Assessment

Yes for the current behavior: source inspection shows inbound Telegram text flows into BodyForAgent without a t.me/c resolver, and the action/docs surface has no fetch/read action. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path may be feasible, but it is not clearly the best solution until maintainers decide the privacy, permission, retention, and side-effect policy; a gated Telegram plugin resolver is safer.

## Security Review

Status: needs_attention

Summary: Resolving Telegram message links can inject historical chat content that was not directly supplied in the current turn, so policy and gating should be defined before implementation.

Concerns:

- **[medium] Define linked-message access policy first:**
  - body: Automatic t.me/c resolution could expose past group or DM content to the agent; maintainers should decide who may trigger it, what history may be read, how failures are surfaced, and whether the default is opt-in.
  - confidence: 0.86
- **[medium] Validate Bot API retrieval side effects:**
  - body: The proposed copy/forward/delete approach can create visible or partially deleted messages and has Telegram permission constraints, so any implementation needs explicit handling and tests for those side effects.
  - confidence: 0.78
