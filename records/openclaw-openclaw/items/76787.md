## Summary

Keep open: current main and v2026.6.10 still do not resolve Telegram `t.me/c/...` links into model context, and this is a feature/security design item rather than stale cleanup.

## What This Changes

The issue requests automatic Telegram `t.me/c/<chat>/<thread>/<msg>` link parsing, message retrieval, and injection of the referenced message content into session context.

## Best Possible Solution

Define a Telegram-plugin-owned `t.me/c` resolver on top of the approved read/cache semantics, with explicit opt-in authorization, contextVisibility behavior, docs, tests, and live Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main lacks a `t.me/c` parser, Telegram read/fetch action discovery, runtime dispatch, and linked-message supplemental context path, so the link remains ordinary inbound text.

## Solution Assessment

Unclear until maintainers choose the product/security contract. The maintainable direction is likely to align link resolution with Telegram read/cache semantics instead of adding an independent copy/forward injection path first.
