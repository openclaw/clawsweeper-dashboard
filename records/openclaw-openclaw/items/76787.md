## Summary

Keep open. Current main still treats Telegram `t.me/c/...` links as ordinary inbound text and exposes no Telegram read/fetch/link-resolution action; the requested historical-message injection is a new Telegram feature with privacy and Bot API side-effect policy questions.

## What This Changes

Add Telegram support for resolving `t.me/c/<chat>/<thread>/<message>` links and injecting the referenced message content into the agent context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested history when available, documents any Bot API fallback, and gates historical content injection by explicit access, retention, and side-effect policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: a Telegram message containing a `t.me/c/...` URL flows through text expansion into `BodyForAgent`, and no Telegram read/fetch/link resolver is exposed; I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best maintainable fix because maintainers first need to choose the privacy, authorization, retention, and Telegram-side-effect policy.

Summary: The issue itself is security-sensitive because resolving Telegram links can expose historical chat content to the agent and may require side-effecting Bot API retrieval.
