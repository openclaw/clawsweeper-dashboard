## Summary

Keep open. Current main still treats Telegram `t.me/c/...` references as ordinary inbound text and exposes no Telegram read/fetch/link-resolution action; the requested historical-message injection is a new Telegram capability that needs privacy, authorization, and Bot API side-effect policy before implementation.

## What This Changes

Resolve Telegram `t.me/c/<chat>/<thread>/<message>` links from inbound messages and inject the referenced message content into the agent context.

## Best Possible Solution

Design a Telegram-owned, opt-in resolver that prefers already-ingested history when possible, documents any Bot API fallback, and gates historical content injection by clear access and retention policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Telegram message containing a `t.me/c/...` URL flows through text expansion into `BodyForAgent`, and no Telegram read/fetch resolver is exposed; I did not run a live bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet the best way to solve the issue because maintainers first need to choose the privacy, authorization, retention, and Telegram-side-effect policy.
