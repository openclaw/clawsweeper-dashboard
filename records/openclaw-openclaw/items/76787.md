## Summary

Keep open. Current main still lacks Telegram `t.me/c/...` message-link resolution and no merged fix or canonical follow-up supersedes this request; the remaining work needs explicit Telegram privacy and Bot API side-effect policy before implementation.

## What This Changes

The issue requests Telegram-owned parsing of `t.me/c/<chat_id>/<thread>/<msg_id>` links, retrieval of the referenced message, and injection of that message content into agent context.

## Best Possible Solution

Design an opt-in Telegram resolver that prefers already-ingested history when possible, documents any Bot API fallback, and gates historical content injection by explicit access, privacy, retention, and side-effect policy.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: a Telegram message containing a `t.me/c/...` URL is treated as ordinary text through body construction and session finalization, and no Telegram fetch/read resolver is exposed; I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. The proposed copy/forward/delete retrieval path is not yet the best maintainable solution because maintainers first need to choose the privacy, authorization, retention, opt-in, and Telegram-side-effect policy.
