## Summary

Keep open. Current main still passes Telegram `t.me/c/...` links through as inbound text and exposes no Telegram-owned message-link fetch/read resolver; this is a new, security-sensitive Telegram plugin capability that needs maintainer policy first.

## What This Changes

Request Telegram inbound handling to resolve `t.me/c/<chat_id>/<thread>/<msg_id>` links and inject the referenced message content into the agent context.

## Best Possible Solution

Design a Telegram-owned, opt-in or policy-gated resolver that injects clearly labeled linked-message context only after access, retention, and side-effect rules are defined.

## Reproduction Assessment

Yes. Source inspection shows a `https://t.me/c/...` message would flow through Telegram inbound body construction as text and become `BodyForAgent`; I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No. The requested copy/forward/delete retrieval path is not yet proven to be the best fix because it can have privacy and visible-message side effects; a gated Telegram plugin resolver is the safer direction.

## Security Review

Status: needs_attention

Summary: Automatic Telegram message-link resolution would add historical chat content to model context, so access policy needs maintainer attention before implementation.

Concerns:

- **[medium] Define linked-message access policy first:**
  - body: Resolving `t.me/c` links can expose past group or DM content that was not directly pasted into the current turn, so maintainers should decide who may trigger it, what history may be used, and what default is safe.
  - confidence: 0.84
- **[medium] Validate Bot API retrieval side effects:**
  - body: The proposed copy/forward/delete approach may create visible messages or leave artifacts when deletion fails, so any implementation should make those effects explicit and handle permission failures safely.
