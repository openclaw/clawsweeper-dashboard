## Summary

Keep open: current main and the latest release still key the foreground reply fence by group target rather than sender or inbound message, so the reported WhatsApp automatic-delivery loss remains source-reproducible. The directly linked fix attempts are closed unmerged, and the still-open related lazy-fence PR covers a different queued-admission race rather than explicitly proving this two-participant group case.

## What This Changes

The issue asks OpenClaw to stop automatic foreground reply fencing from suppressing completed replies to earlier concurrent WhatsApp group messages from different users.

## Best Possible Solution

Define a participant- or reply-lineage-aware foreground fence for group conversations while preserving target-level freshness for direct and same-participant prompts, then prove it with focused freshness tests and a redacted live WhatsApp run.

## Reproduction Assessment

Yes, at source level: two dispatches sharing a WhatsApp group-target fence can complete out of order, and a newer visible delivery causes the older generation to cancel before transport delivery. I did not run a live WhatsApp group reproduction in this read-only review.

## Solution Assessment

No, not as a universal per-message key or a new config escape hatch first. The maintainable fix is to choose a stable group participant or reply-lineage boundary that preserves existing direct-chat and same-participant freshness behavior.
