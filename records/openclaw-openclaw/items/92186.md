## Summary

Keep this open: current main and v2026.6.8 still use the same-target foreground fence, and the linked fix PRs are closed unmerged with the permanent fence boundary still needing maintainer choice.

## What This Changes

The issue reports silent WhatsApp delivery loss when automatic foreground replies for concurrent group mentions complete in the dashboard but the older completed reply is cancelled before delivery.

## Best Possible Solution

Define a participant- or reply-lineage-aware foreground fence that isolates independent WhatsApp group replies while preserving stale-reply suppression for direct chats and same-participant rapid-fire prompts.

## Reproduction Assessment

Yes at source level: two WhatsApp group dispatches sharing a group target can complete out of order, and the newer visible delivery can cancel the older generation before transport delivery. I did not run a live WhatsApp group reproduction in this read-only review.

## Solution Assessment

No final code solution is selected yet. The fix belongs in the foreground fence identity, but maintainers should choose a narrower participant or reply-lineage boundary instead of a broad per-message key that disables valid freshness suppression.
