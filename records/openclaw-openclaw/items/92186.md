## Summary

Keep this open: current main and the latest release still use a target-scoped foreground reply fence, so a newer visible WhatsApp group reply can still cancel an older completed reply that shares the group target.

## What This Changes

The issue requests fixing silent WhatsApp group message loss when automatic visible replies for concurrent group mentions share one foreground reply fence.

## Best Possible Solution

Define a stable participant- or reply-lineage-aware foreground fence for group conversations while retaining target-level freshness for direct chats and genuinely superseding prompts, then prove both independent group delivery and stale-reply suppression in focused tests plus a redacted live WhatsApp run.

## Reproduction Assessment

Yes at source level: two buffered dispatches with the same group target share the current fence key, and a newer visible delivery makes the older `beforeDeliver` return null before transport delivery. I did not run a live WhatsApp group in this read-only review.

## Solution Assessment

No for the literal broad per-message/config escape-hatch direction; the best fix is a narrow reply-lineage or participant partition that preserves the existing freshness contract where prompts genuinely supersede each other.
