## Summary

Keep open: current main and v2026.6.6 still use a per-target foreground fence, so a newer visible WhatsApp delivery can cancel an older completed same-group reply before transport delivery. The open foreground-fence PR at https://github.com/openclaw/openclaw/pull/91963 is relevant, but it is not merged and is not yet explicit proof for this two-sender WhatsApp automatic-mode path.

## What This Changes

Prevent automatic foreground reply fencing from suppressing completed WhatsApp group replies to earlier concurrent messages, and add visibility for those cancellations.

## Best Possible Solution

Validate and land a narrow foreground-fence fix that preserves intended stale-reply suppression while allowing independent group participants' completed replies to deliver, with focused regression coverage and redacted WhatsApp proof.

## Reproduction Assessment

Yes, at source level: two buffered dispatches sharing one WhatsApp group target can let the newer visible delivery advance the shared foreground generation, causing the older snapshot to return null before transport delivery. I did not run a live WhatsApp group in this read-only review.

## Solution Assessment

Unclear for the final implementation path: the per-sender and per-message attempts were closed unmerged, while the open lazy-fence PR is a plausible better fix but needs explicit two-sender WhatsApp automatic-mode coverage before this can close.
