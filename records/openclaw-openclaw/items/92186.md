## Label Justifications

- `P1`: The source path can silently discard a completed reply during a concurrent multi-user group workflow affecting real channel users.
- `impact:message-loss`: A generated reply is present in agent state but suppressed before WhatsApp delivery.

## Summary

Current main still shares one foreground reply fence across concurrent messages in the same group target, allowing a newer visible delivery to suppress an earlier completed reply; keep the issue open because https://github.com/openclaw/openclaw/pull/92269 is an open closing PR that now owns the focused fix.

## What This Changes

Prevent completed replies to distinct concurrent group messages from cancelling one another at the foreground delivery fence.

## Best Possible Solution

Key the fence by the canonical inbound-message identity while preserving target/session scope as context, retain freshness suppression only among deliveries competing for that same inbound turn, add focused shared-dispatch regression coverage, and confirm both replies in a live WhatsApp group.

## Reproduction Assessment

Yes, at source level with high confidence: two current-main dispatches sharing the same group fence key reproduce the cancellation condition when the newer generation records visible delivery before the older delivery runs. The reported live WhatsApp observation was not independently rerun in this read-only review.

## Solution Assessment

Yes, per-inbound-message isolation is the narrowest maintainable direction because sender scoping still conflates separate messages from one sender and disabling the fence would discard its valid same-turn freshness behavior; the linked PR still needs regression and live proof.
