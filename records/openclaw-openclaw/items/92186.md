## Label Justifications

- `P1`: A normal concurrent group workflow can silently lose a completed user-facing reply with no delivery error shown to the operator.
- `impact:message-loss`: The generated response is recorded in agent state but suppressed before reaching the WhatsApp group.

## Summary

Current main still shares one foreground reply fence across distinct inbound messages for the same conversation target, allowing a newer visible reply to suppress an earlier completed reply. The report is actionable, and the open closing PR at https://github.com/openclaw/openclaw/pull/92269 already owns the repair, so this issue should remain open until that PR is merged or replaced.

## What This Changes

Prevent automatic reply fencing from suppressing completed replies to distinct concurrent inbound messages in the same WhatsApp group.

## Best Possible Solution

Key the fence by stable inbound-message or reply-lineage identity, retain target-level fallback only when that identity is unavailable, and prove both distinct-message delivery and same-lineage stale suppression before validating the fix in a live multi-user WhatsApp group.

## Reproduction Assessment

Yes, at source level with high confidence: distinct same-target dispatches share a fence, and a newer visible delivery deterministically cancels the older generation before delivery. This review did not independently run the live WhatsApp scenario.

## Solution Assessment

Yes in direction: inbound-message or reply-lineage scoping is narrower and more maintainable than sender-only scoping or a new configuration option. The final patch must preserve the original same-lineage freshness invariant and add focused regression coverage.
