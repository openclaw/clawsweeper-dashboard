## Summary

Keep open. Current main does not implement the requested gateway-owned ACP node-backed runtime, and the PR head still has blocking correctness issues plus a security-sensitive node command classification concern. The related open follow-up PR is unmerged, so it does not make this PR implemented or safe to close.

## What This Changes

The branch adds a durable gateway-owned ACP node-backed runtime with store/state, gateway-to-node worker commands and events, replayable outward projection, startup recovery, tests, docs, and some landing-only Feishu/Google/Google Chat test isolation changes.

## Best Possible Solution

Keep the PR open for maintainer rework or replacement. The maintainable end state is a smaller set of reviewable slices for durable ACP store/protocol, node worker transport, durable projection/replay, and startup recovery, with the correctness blockers fixed and the node command trust boundary explicitly classified, documented, and tested.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review decision: current-main source inspection reproduces the absence of the acp-node runtime, and PR-head source inspection reproduces the blocking defects at the cited lines. I did not run tests because this was a read-only cleanup review.

## Solution Assessment

Is this the best way to solve the issue? No for the current patch as-is: the product direction is plausible, but the patch should not land with duplicated projection loops, best-effort ACP terminal/event delivery, run-id aliasing, reconnect races, and unresolved command-surface classification. A split implementation with targeted tests and security review is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:425-427`
  - body: The ACP worker bridge sends events through the existing best-effort `sendNodeEvent` helper, which catches and ignores `client.request("node.event", ...)` failures. ACP terminals/events are durable state transitions, so a dropped terminal can look successful on the node while the gateway never records it; use a non-swallowing send path for ACP worker traffic.
  - confidence: 0.93
- **[P1] Return the active projection promise without restarting it:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-456`
  - body: When a projection target is already active, `ensureProjection()` calls `existing.start()`. `start()` launches `this.run()` every time, so a second ensure call creates another polling/delivery loop for the same target and can duplicate outbound delivery or race checkpoints. Return the existing settled promise instead.
  - confidence: 0.91
- **[P2] Stop the projection runner when run creation fails:** `src/auto-reply/reply/dispatch-acp.ts:394-396`
  - body: If `acpManager.runTurn()` fails before creating a durable run, this branch rethrows after reading `getRun(requestId)`, but the projection runner started with `waitForRunStart: true` is never stopped. It will keep polling for a run that will never appear, leaking a background loop for each early failure.
  - confidence: 0.89
- **[P1] Reject run ID collisions across sessions:** `src/acp/store/store.ts:532-534`
  - body: `startRun()` treats any existing `store.runs[runId]` as an idempotent hit without verifying `sessionKey` or `requestId`. If two sessions reuse the same request-derived run ID, the second session aliases the first run and later lease/session validation fails or corrupts delivery state; require the existing run to match the session and request before returning it.
  - confidence: 0.92
- **[P1] Await reconnect reconciliation before worker events can arrive:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: The reconnect path launches `handleNodeConnected()` fire-and-forget and then continues normal node setup. A reconnecting node can send `acp.worker.event` or `acp.worker.terminal` before suspect lease reconciliation finishes, causing the gateway store to reject valid progress as missing an active lease. Gate ACP worker traffic until reconciliation completes or queue it behind that promise.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch expands the gateway-to-node ACP command/event surface and classifies execution-capable ACP commands as default system node commands.

Concerns:

- **[medium] Classify ACP node commands outside the default system bucket:** `src/gateway/node-command-policy.ts:56`
  - body: The PR adds `NODE_ACP_COMMANDS` to `SYSTEM_COMMANDS`, which enables ACP session/turn control commands wherever normal desktop system commands are default-allowed. Because these commands start and control node-host runtime work over the gateway-node trust boundary, they need explicit policy classification, documentation, and tests instead of silently inheriting the existing system command bucket.
  - confidence: 0.74
