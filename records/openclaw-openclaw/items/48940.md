## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:426`
  - body: `handleAcpInvokeCommand` receives the generic `sendNodeEvent` callback, but that helper swallows `client.request("node.event")` failures. ACP worker events and terminals are durable protocol inputs, so a dropped send can let the node settle locally while the gateway never records the terminal and recovery/replay stays inconsistent.
  - confidence: 0.9
- **[P1] Return the existing projection promise:** `src/auto-reply/reply/dispatch-acp-replay.ts:456`
  - body: When a target is already active, `ensureProjection` calls `existing.start()`, and `start()` launches `this.run()` every time. Re-entering this path can create multiple polling/delivery loops for the same target, racing checkpoints and duplicating outbound sends.
  - confidence: 0.92
- **[P1] Stop the projection runner when turn creation fails:** `src/auto-reply/reply/dispatch-acp.ts:395-396`
  - body: The durable ACP path starts a projection with `waitForRunStart: true`, then throws `turnError` if no durable run record exists. That leaves the runner polling forever for a run that will never appear, leaking a background loop per early turn-start failure.
  - confidence: 0.88
- **[P1] Await ACP reconnect before accepting worker events:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: `handleNodeConnected` is launched fire-and-forget, leaving a window where a reconnecting node can send worker events while the lease is still `suspect`. Those events can be rejected as not bound to an active lease, and the node-host ACP path does not provide another guaranteed terminal replay after that drop.
  - confidence: 0.84
- **[P1] Validate idempotent run ownership:** `src/acp/store/store.ts:532-534`
  - body: `startRun` returns any existing `store.runs[runId]` without checking that it belongs to the same session and request. A run-id collision across sessions would alias the second turn onto the first run record, causing later worker events and delivery bookkeeping to use the wrong durable state.
  - confidence: 0.87
- **[P3] Reject array worker envelopes at the object guard:** `src/acp/store/gateway-events.ts:50-51`
  - body: This `isRecord` helper accepts arrays, unlike the sibling ACP-node parser. Arrays are later rejected by field validators, but accepting them here gives misleading field-level errors instead of the top-level envelope error and keeps the two worker parsers inconsistent.
  - confidence: 0.76
