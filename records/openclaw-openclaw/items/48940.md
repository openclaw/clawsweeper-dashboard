## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use shared SQLite for ACP runtime state:** `src/acp/store/store.ts:112`
  - body: This new store is the durable source of truth for ACP sessions, runs, events, leases, delivery targets, and checkpoints, but it lives at `state/acp/gateway-node-runtime-store.json`. OpenClaw runtime state is supposed to live in the shared SQLite state DB, so landing this sidecar would create a second migration/recovery path for core session state before the runtime is even shipped.
  - confidence: 0.93
- **[P1] Reject cross-session run id collisions:** `src/acp/store/store.ts:532-535`
  - body: `startRun` returns an existing `store.runs[runId]` before checking `sessionKey` or `requestId`. Since the ACP dispatch path derives `runId` from message/request ids, two sessions can collide and the second session will reuse the first run record, after which worker events and delivery targets fail validation against the wrong session.
  - confidence: 0.91
- **[P1] Do not restart an active projection runner:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-457`
  - body: When a projection for the same target is already active, `ensureProjection` calls `existing.start()`, but `start()` always launches `this.run()` again. A second ensure on the same target can therefore run another polling/delivery loop against the same checkpoint and duplicate or race outward sends instead of deduplicating the active projection.
  - confidence: 0.9
- **[P2] Stop the projection runner after early turn-start failure:** `src/auto-reply/reply/dispatch-acp.ts:394-397`
  - body: On the durable ACP path, `ensureProjection(... waitForRunStart: true)` is started in `Promise.all`; if `runTurn` fails before creating a durable run, the catch rethrows when `getRun(requestId)` is null but never stops the projection service. That leaves a background loop polling forever for a run that will never appear for every early dispatch failure.
  - confidence: 0.87
- **[P1] Propagate ACP worker event delivery failures:** `src/node-host/invoke.ts:425-427`
  - body: The ACP node command handler awaits `sendNodeEvent`, but the helper catches and ignores `client.request("node.event", ...)` failures. If a worker event or terminal is dropped at the gateway RPC boundary, the node can settle locally while the gateway never records the terminal, leaving the run stuck in recovery and breaking replay/cancel semantics.
  - confidence: 0.9
- **[P1] Await reconnect reconcile before accepting worker events:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: `handleNodeConnected` is launched fire-and-forget after node registration, so a reconnecting node can immediately deliver `acp.worker.event` or `acp.worker.terminal` before its suspect lease has been reactivated. Those events are rejected by the store as non-active/stale progress, and the node-host path does not retry them as heartbeat-backed recovery.
  - confidence: 0.86
