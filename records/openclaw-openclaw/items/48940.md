## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:425-427`
  - body: The ACP worker bridge uses a best-effort node-event send path that can swallow `node.event` request failures. If a terminal or event is dropped, the node can treat the turn as settled while the gateway never records the state transition, leaving recovery and replay inconsistent.
  - confidence: 0.93
- **[P1] Return the active projection promise without restarting it:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-456`
  - body: When a projection target is already active, `ensureProjection()` calls `existing.start()`, but `start()` launches another run loop. A second ensure for the same target can create duplicate polling and delivery loops that race checkpoint updates.
  - confidence: 0.91
- **[P2] Stop the projection runner when run creation fails:** `src/auto-reply/reply/dispatch-acp.ts:394-396`
  - body: The durable dispatch path starts a projection runner before `runTurn()`, then rethrows if no durable run record exists. That leaves the runner polling forever for a run that will never appear after early start failures.
  - confidence: 0.89
- **[P1] Reject run ID collisions across sessions:** `src/acp/store/store.ts:532-534`
  - body: `startRun()` treats any existing `runId` as idempotent without verifying the same `sessionKey` and `requestId`. Cross-session request ID reuse can alias one session onto another run and break lease/session validation.
  - confidence: 0.92
- **[P1] Await reconnect reconciliation before worker events arrive:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: The reconnect path launches ACP lease reconciliation fire-and-forget, then continues normal node setup. A reconnecting worker can send events before a suspect lease is reactivated, causing valid progress or terminals to be rejected.
  - confidence: 0.86
