## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move ACP runtime state into SQLite:** `src/acp/store/store.ts:111-112`
  - body: This creates a new core runtime state file under `state/acp/gateway-node-runtime-store.json` and writes sessions, runs, events, leases, checkpoints, delivery targets, and idempotency through JSON. Repository policy requires OpenClaw-owned runtime state to live in SQLite, so this would add a second recovery/migration/concurrency path for durable gateway truth.
  - confidence: 0.95
- **[P1] Reject cross-session run id reuse:** `src/acp/store/store.ts:532-534`
  - body: `startRun` returns any existing `store.runs[runId]` before verifying that it belongs to the same session and request. If two sessions reuse a provider/request-derived run id, the second session aliases the first run and later worker events fail lease/session validation or corrupt delivery bookkeeping.
  - confidence: 0.94
- **[P1] Do not restart active projection runners:** `src/auto-reply/reply/dispatch-acp-replay.ts:455-456`
  - body: When a projection already exists for the target, `ensureProjection` calls `existing.start()`, but `start()` launches `run()` again every time. A second ensure call can therefore create another polling/delivery loop for the same run and race or duplicate outbound effects instead of deduplicating work.
  - confidence: 0.92
- **[P1] Propagate ACP worker event delivery failures:** `src/node-host/invoke.ts:425-426`
  - body: The ACP worker bridge is given `sendNodeEvent`, but that helper swallows `node.event` request failures. A dropped worker event or terminal can look accepted locally while the gateway never persists it, leaving the run stuck in recovery with inconsistent replay and cancel semantics.
  - confidence: 0.9
- **[P1] Await reconnect reconciliation before worker events:** `src/gateway/server/ws-connection/message-handler.ts:1024-1034`
  - body: Node connect handling starts `handleNodeConnected` in the background, so a reconnecting worker can send ACP events before the suspect lease is reactivated. Those events are rejected as non-active lease progress, and the worker path does not guarantee a later heartbeat/terminal resend to recover the run.
  - confidence: 0.88
- **[P2] Stop projection when turn start fails early:** `src/auto-reply/reply/dispatch-acp.ts:395-396`
  - body: The durable path starts a wait-for-run projection and then rethrows when `runTurn` fails before any durable run exists. That branch never stops the runner, so it can keep polling forever for a run that will never be created.
  - confidence: 0.86
