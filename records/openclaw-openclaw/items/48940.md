## Summary

Keep open. Current main still lacks the node-backed ACP runtime, and this PR is not merge-ready because PR-head source review shows blocking durability, reconnect ordering, run identity, and command trust-boundary issues, plus no real node/gateway behavior proof beyond tests.

## What This Changes

The PR adds a gateway-owned ACP node-backed runtime with durable store/replay, node-host worker commands/events, startup recovery, docs, tests, and extension test isolation changes.

## Best Possible Solution

Keep the PR open for maintainer-led rework or replacement, ideally split into smaller reviewed slices for durable store/protocol, worker transport, replay recovery, startup reconciliation, and command policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR review findings: PR-head source inspection shows deterministic failure paths at the cited lines, though no live gateway/node repro was run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently patched; the gateway-owned ACP runtime direction is plausible, but this implementation should not land with unresolved durability, reconnect ordering, run identity, and command policy defects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:425-427`
  - body: The ACP worker bridge sends progress and terminal events through the node event helper that swallows `node.event` request failures. If an event or terminal is dropped, the node can treat the turn as settled while the gateway never records the durable transition, leaving replay and recovery inconsistent.
  - confidence: 0.93
- **[P1] Return the active projection promise without restarting it:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-456`
  - body: When a projection target is already active, `ensureProjection()` calls `existing.start()`, but `start()` launches another run loop. Re-ensuring the same target can therefore duplicate polling and outbound delivery while racing checkpoint writes.
  - confidence: 0.91
- **[P2] Stop the projection runner when run creation fails:** `src/auto-reply/reply/dispatch-acp.ts:394-396`
  - body: The durable dispatch path starts a wait-for-run projection before `runTurn()`, then rethrows early failures when no durable run exists. That branch leaves the runner polling forever for a run that will never be created.
  - confidence: 0.89
- **[P1] Reject run ID collisions across sessions:** `src/acp/store/store.ts:532-534`
  - body: `startRun()` treats any existing `runId` as idempotent without checking `sessionKey` or `requestId`. A reused request-derived id from another session can alias to the wrong durable run and break lease/session validation.
  - confidence: 0.92
- **[P1] Await ACP reconnect reconciliation before worker events arrive:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: The node reconnect path starts ACP lease reconciliation in the background and then continues normal setup. A reconnecting worker can send progress before the suspect lease is reactivated, causing valid events or terminals to be rejected.
  - confidence: 0.86
- **[P3] Reject array worker envelopes at the object boundary:** `src/acp/store/gateway-events.ts:50-52`
  - body: The ACP worker envelope `isRecord` helper accepts arrays, so malformed array payloads fall through to misleading field-level validation instead of being rejected as a non-object envelope. Keep this consistent with the sibling runtime helper.
  - confidence: 0.78
