## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Store ACP runtime state in SQLite:** `src/acp/store/store.ts:112`
  - body: This adds durable sessions, runs, events, leases, checkpoints, delivery targets, and idempotency under `state/acp/gateway-node-runtime-store.json`. OpenClaw-owned runtime state and replay/checkpoint data need to live in the shared SQLite/doctor migration model, otherwise upgrades, recovery, locking, and state inspection get a second source of truth outside the canonical store.
  - confidence: 0.93
- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:425-426`
  - body: The ACP command path passes worker events through the existing `sendNodeEvent` helper, but that helper catches and ignores `client.request("node.event", ...)` failures. A node can then finish locally while the gateway never records the worker event or terminal, leaving durable state stuck in recovery while the node believes delivery succeeded.
  - confidence: 0.9
- **[P1] Return the active projection instead of starting another loop:** `src/auto-reply/reply/dispatch-acp-replay.ts:456`
  - body: When a projection already exists for the target, `ensureProjection` calls `existing.start()`, and `start()` launches `this.run()` every time. A second ensure for the same target can therefore create duplicate polling/delivery loops that race checkpoints and duplicate outbound sends.
  - confidence: 0.9
- **[P1] Await reconnect reconciliation before worker frames:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: The connect handler starts `handleNodeConnected` fire-and-forget, so the socket can process `acp.worker.event` or `acp.worker.terminal` before the suspect lease is reconciled back to active. In that window the store rejects progress against the suspect lease and the completed worker output can be lost or stranded in recovery.
  - confidence: 0.88
- **[P1] Reject run ID reuse across sessions:** `src/acp/store/store.ts:532-535`
  - body: `startRun` returns any existing `store.runs[runId]` without checking `sessionKey` or `requestId`. If two sessions produce the same request/run id, the second session aliases the first run and later worker events fail session/lease validation or corrupt delivery bookkeeping.
  - confidence: 0.86
- **[P1] Gate ACP node commands explicitly:** `src/gateway/node-command-policy.ts:54-57`
  - body: The new ACP node commands are added to the default system command set for desktop nodes. Because they start, cancel, close, and status-check a runtime worker over the gateway/node trust boundary, they should require an ACP-specific policy/capability gate instead of being default-allowed with ordinary system commands.
  - confidence: 0.84
- **[P2] Stop the projection runner on early turn-start failure:** `src/auto-reply/reply/dispatch-acp.ts:394-396`
  - body: The durable path starts a projection runner with `waitForRunStart`, then rethrows when `runTurn` fails before a durable run exists. That branch never stops the runner, so it can keep polling forever for a run record that will never appear after the dispatch already failed.
  - confidence: 0.82
