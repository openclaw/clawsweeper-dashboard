## Summary

Keep open. Current main does not contain the proposed ACP node-backed runtime, and this recent PR still has source-confirmed runtime correctness blockers, a command-policy security concern, and missing real paired-node behavior proof before merge.

## What This Changes

The PR adds a gateway-owned ACP node-backed runtime with durable store/replay, ACP worker node commands/events, startup recovery, docs, tests, and extension test isolation updates.

## Best Possible Solution

Keep the PR open for maintainer-led rework or replacement, preferably split into smaller reviewed slices for durable store/protocol, node worker transport, replay recovery, startup reconciliation, and command policy, each with real behavior proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:426`
  - body: ACP worker events and terminals are sent through sendNodeEvent, but that helper catches and ignores node.event request failures. A dropped terminal can make the node settle locally while the gateway never records the durable transition, leaving replay and recovery inconsistent.
  - confidence: 0.93
- **[P1] Return the active projection promise without restarting it:** `src/auto-reply/reply/dispatch-acp-replay.ts:456`
  - body: When a projection target is already active, ensureProjection calls existing.start(), and start() launches run() again. Re-ensuring the same target can therefore duplicate polling and outbound delivery while racing checkpoint writes.
  - confidence: 0.91
- **[P2] Stop the projection runner when run creation fails:** `src/auto-reply/reply/dispatch-acp.ts:396`
  - body: The durable dispatch path starts a wait-for-run projection before runTurn, then rethrows early failures when no durable run exists. That branch leaves the runner polling forever for a run that will never be created.
  - confidence: 0.89
- **[P1] Reject run ID collisions across sessions:** `src/acp/store/store.ts:534`
  - body: startRun treats any existing runId as idempotent without checking sessionKey or requestId. A reused request-derived ID from another session can alias to the wrong durable run and break lease/session validation.
  - confidence: 0.92
- **[P1] Await ACP reconnect reconciliation before worker events:** `src/gateway/server/ws-connection/message-handler.ts:1024`
  - body: The node reconnect path starts ACP lease reconciliation in the background and then continues setup. A reconnecting worker can send progress before the suspect lease is reactivated, causing valid events or terminals to be rejected.
  - confidence: 0.86
- **[P3] Reject array worker envelopes at the object boundary:** `src/acp/store/gateway-events.ts:51`
  - body: The ACP worker envelope isRecord helper accepts arrays, so malformed array payloads fall through to misleading field-level validation instead of being rejected as a non-object envelope. Keep it consistent with the sibling runtime helper.
  - confidence: 0.78
