## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move ACP runtime state into SQLite:** `src/acp/store/store.ts:111-112`
  - body: This creates `state/acp/gateway-node-runtime-store.json` for sessions, runs, events, leases, checkpoints, delivery targets, and idempotency. OpenClaw runtime state is supposed to live in the shared SQLite store, so this adds a second migration, locking, backup, and recovery path for gateway truth.
  - confidence: 0.95
- **[P1] Reject cross-session run ID reuse:** `src/acp/store/store.ts:532-534`
  - body: `startRun` returns any existing `store.runs[runId]` before checking that it belongs to the same session and request. If two sessions reuse a provider/request-derived run id, the second session can alias the first run and corrupt or wedge later worker event validation.
  - confidence: 0.94
- **[P1] Do not restart active projection runners:** `src/auto-reply/reply/dispatch-acp-replay.ts:455-456`
  - body: When a projection already exists for the target, `ensureProjection` calls `existing.start()`, but `start()` launches `run()` again every time. A repeated ensure can therefore create another polling/delivery loop for the same run and race or duplicate outward effects.
  - confidence: 0.92
- **[P1] Propagate ACP worker event delivery failures:** `src/node-host/invoke.ts:425-426`
  - body: The ACP bridge is wired to `sendNodeEvent`, but that helper catches and ignores failed `node.event` requests. A dropped worker event or terminal can look accepted locally while the gateway never persists it, leaving replay and cancel semantics inconsistent.
  - confidence: 0.9
- **[P1] Await ACP reconnect reconciliation before worker events:** `src/gateway/server/ws-connection/message-handler.ts:1024-1034`
  - body: Node connect starts `handleNodeConnected` in the background, so a reconnecting worker can send ACP events before the suspect lease is reactivated. Those events are then rejected as non-active lease progress and may leave the run stuck in recovery.
  - confidence: 0.88
- **[P2] Stop projection when turn start fails early:** `src/auto-reply/reply/dispatch-acp.ts:395-396`
  - body: The durable path starts a wait-for-run projection and then rethrows when `runTurn` fails before any durable run exists. That branch never stops the runner, so it can keep polling forever for a run that will never be created.
  - confidence: 0.86
