## Summary

Keep open: this is a substantial new gateway/node ACP runtime feature, and the current PR still has blocking source-level durability bugs plus missing real behavior proof for the new execution path.

## What This Changes

The PR adds a gateway-owned, node-backed ACP runtime with durable ACP store state, node-host worker commands, replay/recovery projection, tests, and architecture docs.

## Best Possible Solution

Fix the blocking ACP durability issues, add real node-backed runtime proof from a redacted live setup, and then have the gateway/ACP owners review the architecture and trust-boundary changes before landing.

## Reproduction Assessment

Yes for the blocking review findings via source inspection: ACP event send failures are swallowed, duplicate projection loops can be started, and `startRun` can alias an existing run ID across sessions. I did not run a live gateway/node E2E because this review is read-only.

## Solution Assessment

No. The direction may be useful, but the current patch is not the best safe landing path until ACP event delivery is strict, projection start/stop is idempotent, run IDs are scoped, and real setup proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:425-426`
  - body: `handleAcpInvokeCommand` relies on `sendNodeEvent` failures to put the node-host session into an explicit terminal handoff failure state, but this passes the existing best-effort helper that catches every `client.request("node.event")` error. A dropped `acp.worker.event` or `acp.worker.terminal` is therefore treated as delivered locally while the gateway store never records it, leaving the durable run stuck in recovery. Use a strict sender for ACP events or make this helper optionally fail-fast.
  - confidence: 0.93
- **[P1] Return the active projection instead of restarting it:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-456`
  - body: When a target already has an active projection, `ensureProjection` calls `existing.start()`, but `start()` launches `this.run()` every time. A second ensure for the same target will start another polling/delivery loop with its own coordinator, which can duplicate outbound sends and race checkpoint updates. Store and return the original settled promise, or make `start()` idempotent.
  - confidence: 0.9
- **[P1] Validate run identity before idempotent start:** `src/acp/store/store.ts:532-534`
  - body: `startRun` returns any existing `store.runs[runId]` without checking that it belongs to the same `sessionKey` and `requestId`. Since the dispatcher uses provider message IDs as the ACP request/run ID when present, two sessions can collide on a run ID and the second session will reuse the first run, corrupting later lease/session validation. Reject mismatches or namespace run IDs by session.
  - confidence: 0.88
- **[P2] Stop the projection runner when no run was created:** `src/auto-reply/reply/dispatch-acp.ts:395-396`
  - body: The durable path starts a projection runner with `waitForRunStart: true` before `acpManager.runTurn` has necessarily created a run. If `runTurn` throws before the durable run exists, this branch rethrows without stopping the runner, so it keeps polling for a run that will never appear. Stop the local projection service or start projection only after the run exists on this failure path.
  - confidence: 0.86
