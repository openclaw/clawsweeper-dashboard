## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Move ACP runtime state into the shared SQLite store:** `src/acp/store/store.ts:111-130`
  - body: This adds `state/acp/gateway-node-runtime-store.json` as the durable store for ACP sessions, runs, events, leases, checkpoints, delivery targets, and idempotency. OpenClaw's current ACP metadata and replay state already live in `state/openclaw.sqlite`, and the repo policy requires runtime state to use SQLite rather than new JSON sidecars; keeping a parallel JSON store would create a second migration/recovery/permission path for core gateway truth.
  - confidence: 0.94
- **[P1] Reject run IDs that belong to another session:** `src/acp/store/store.ts:532-535`
  - body: `startRun()` returns any existing `store.runs[runId]` before checking `sessionKey` or `requestId`. Because the durable path uses the message-derived request id as the run id, two sessions that produce the same id can alias to one run record, after which worker events and delivery bookkeeping are validated against the wrong session.
  - confidence: 0.9
- **[P1] Return the active projection instead of starting it again:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-456`
  - body: When a target already has an active runner, `ensureProjection()` calls `existing.start()`, but `start()` launches `this.run()` every time. A second ensure for the same target can therefore start another polling/delivery loop over the same run and race or duplicate outbound effects instead of deduplicating projection ownership.
  - confidence: 0.88
- **[P2] Stop the projection runner when turn start fails early:** `src/auto-reply/reply/dispatch-acp.ts:374-381`
  - body: The durable dispatch path starts a projection runner with `waitForRunStart: true` in parallel with `runTurn()`. If `runTurn()` fails before a durable run exists, the code rethrows the turn error but never stops the runner, leaving it polling forever for a run that will never appear.
  - confidence: 0.86
- **[P1] Use a non-best-effort sender for ACP worker events:** `src/node-host/invoke-acp.ts:865-873`
  - body: The ACP worker treats awaited `sendNodeEvent()` calls as durable event/terminal handoff, but the concrete sender in `src/node-host/invoke.ts` swallows `node.event` RPC failures as best-effort. A node can settle locally while the gateway never records the event or terminal, leaving the run stuck in recovery with incorrect replay/cancel semantics.
  - confidence: 0.89
- **[P1] Serialize reconnect reconciliation before accepting ACP events:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: Node registration starts `handleNodeConnected()` as fire-and-forget, so ACP worker events can be handled while the lease is still `suspect`. In that window `validateActiveLeaseBinding()` rejects progress, and the node-host path has no guaranteed heartbeat/replay to recover the dropped terminal or event.
  - confidence: 0.87
