## Summary

Keep this PR open. Current main still lacks the proposed gateway-owned ACP node runtime, and the latest PR head still carries blocking correctness and security-review work around durable replay, node event delivery, reconnect ordering, and run identity isolation.

## What This Changes

This PR adds a gateway-owned ACP node-backed runtime with durable ACP store state, node worker commands/events, replayable outbound projection, startup recovery, focused tests, and architecture/verification docs.

## Best Possible Solution

Keep the PR open for maintainer review or replacement. The best path is to split or rework it into reviewable slices: durable gateway ACP store/protocol, node command and worker bridge, durable projection/replay, and startup recovery, with the blocking findings fixed and a focused security review of the node command surface before broad validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Propagate ACP worker event send failures:** `src/node-host/invoke.ts:425-427`
  - body: The ACP worker bridge sends events through the existing best-effort `sendNodeEvent` helper, which catches and ignores `client.request("node.event", ...)` failures. ACP terminals/events are now durable state transitions, so a dropped terminal can look successful on the node while the gateway never records it; use a non-swallowing send path for ACP worker traffic.
  - confidence: 0.93
- **[P1] Return the active projection promise without restarting it:** `src/auto-reply/reply/dispatch-acp-replay.ts:454-456`
  - body: When a projection target is already active, `ensureProjection()` calls `existing.start()`. `start()` launches `this.run()` every time, so a second ensure call creates another polling/delivery loop for the same target and can duplicate outbound delivery or race checkpoints. Return the existing settled promise instead of starting the runner again.
  - confidence: 0.91
- **[P2] Stop the projection runner when run creation fails:** `src/auto-reply/reply/dispatch-acp.ts:394-396`
  - body: If `acpManager.runTurn()` fails before creating a durable run, this branch rethrows after reading `getRun(requestId)`, but the projection runner started with `waitForRunStart: true` is never stopped. It will keep polling for a run that will never appear, leaking a background loop for each early failure.
  - confidence: 0.89
- **[P1] Reject run ID collisions across sessions:** `src/acp/store/store.ts:532-534`
  - body: `startRun()` treats any existing `store.runs[runId]` as an idempotent hit without verifying `sessionKey` or `requestId`. If two sessions reuse the same request-derived run ID, the second session aliases the first run and later lease/session validation fails or corrupts delivery state; require the existing run to match the session and request before returning it.
  - confidence: 0.92
- **[P1] Await reconnect reconciliation before worker events can arrive:** `src/gateway/server/ws-connection/message-handler.ts:1024-1038`
  - body: The reconnect path launches `handleNodeConnected()` fire-and-forget and then continues normal node setup. A reconnecting node can send `acp.worker.event` or `acp.worker.terminal` before suspect lease reconciliation finishes, causing the gateway store to reject valid progress as missing an active lease. Gate ACP worker traffic until reconciliation completes or queue it behind that promise.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR intentionally expands the node-host ACP command and event surface; one concrete command-policy classification concern needs maintainer security review before landing.

Concerns:

- **[medium] ACP node commands become default desktop system commands:** `src/gateway/node-command-policy.ts:56`
  - body: The PR adds `NODE_ACP_COMMANDS` to `SYSTEM_COMMANDS`, which makes ACP session/turn commands part of the default macOS/Linux/Windows node command set. Because those commands start and control node-host ACP runtime work, this should be explicitly classified, documented, and tested as an execution-capable node surface rather than silently inheriting the existing system command bucket.
  - confidence: 0.74
