## Summary

Keep open: current main still wires Kilo through a payload patch that normalizes proxy reasoning only, while Kilo is a supported OpenAI-compatible provider and the report gives a concrete 400 when a scalar `stop` is sent.

## What This Changes

Normalize scalar `payload.stop` to a one-element string array for Kilo chat-completion requests.

## Best Possible Solution

Add Kilo-owned payload normalization and regression coverage that preserves array stop values and wraps only scalar string `stop` before the Kilo request leaves the stream wrapper.

## Reproduction Assessment

No, not with a live Kilo credential in this read-only review. Source inspection shows the current Kilo wrapper would leave a scalar `stop` untouched, and the issue includes a concrete provider 400 for that shape.

## Solution Assessment

Yes, a provider-scoped Kilo wrapper normalization is the narrowest maintainable fix. It keeps global OpenAI-compatible behavior unchanged while handling Kilo's stricter validation at the provider-owned boundary.
