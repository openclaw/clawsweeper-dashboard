## Summary

Keep open. Current main still supports only binding-level `session.dmScope` for direct-message routing, while guild/channel peers remain isolated and the requested cross-surface session unification needs maintainer product and security policy.

## What This Changes

The issue requests a binding-level session target or alias so Discord guild/channel bindings can share an agent session with direct-message surfaces.

## Best Possible Solution

Add an explicit typed route-binding session target or alias contract, validate it to safe agent-owned targets, preserve default group/channel isolation, and update schema/help/docs plus regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows binding session config only accepts `dmScope`, the resolver only forwards that override, and the session-key helper always isolates non-direct peers; current tests and docs assert the same behavior.

## Solution Assessment

Unclear as proposed. A raw `sessionKey` override is probably too permissive; the safer maintainable solution is a typed, validated target or alias contract with preserved default isolation.
