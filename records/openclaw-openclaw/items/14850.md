## Summary

Keep open. Current main still lacks a binding-level session target or alias for routing a Discord guild/channel binding into the same agent main or DM session, and the current schema, resolver, tests, and Discord docs continue to preserve isolated session keys for non-direct peers. The related merged work in #71750 only added `bindings[].session.dmScope` for direct-message scoping, while the related open identity hook PR #50875 explicitly leaves group/channel peers out of scope.

## What This Changes

Request an opt-in binding-level session key, alias, or scope override so a channel/guild route binding can share an agent session across Telegram and Discord surfaces.

## Best Possible Solution

Add an explicit, opt-in, typed route-binding session target contract in config schema/types and route resolution, validate it to safe canonical agent-owned targets, preserve default group/channel isolation, and update generated schema/help/docs plus regression coverage for configured channel-to-main sharing and unchanged unconfigured channel behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static inspection and existing tests show a Discord channel peer resolves to `agent:<agentId>:discord:channel:<id>`, while the strict binding session schema rejects any binding session field other than `dmScope`.

## Solution Assessment

Is this the best way to solve the issue? Unclear as proposed. A raw `sessionKey` on bindings is likely too permissive; the safer maintainable direction is a typed, validated, explicit opt-in target or alias contract with channel isolation preserved by default.
