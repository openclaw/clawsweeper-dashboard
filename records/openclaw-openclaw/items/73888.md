## Summary

Keep open: the PR targets a valid current-main gateway/systemd bug and is not obsolete, but the current head introduces a medium security issue by forwarding merged service-unit environment values to `systemctl` children.

## What This Changes

The branch adds DBus/XDG env synthesis for Linux `systemctl --user` gateway probes, adjusts root/non-root scope handling and unavailable classification, adds regressions, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the DBus/XDG recovery direction, but synthesize the child env from inherited process env plus only `XDG_RUNTIME_DIR` and `DBUS_SESSION_BUS_ADDRESS`, with a regression proving service env secrets are not forwarded.

## Reproduction Assessment

Yes, source-level. #63561 gives concrete Ubuntu status/install steps, current main still calls direct `systemctl --user` without env recovery, and the PR-specific secret-forwarding regression follows from the new `{ ...process.env, ...env }` spread plus current service env merging.

## Solution Assessment

No, not as-is. The behavioral fix is narrow and useful, but the implementation should not pass parsed service `Environment`/`EnvironmentFile` values to `systemctl`; limiting the child env to the required DBus/XDG overrides is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forwarding service secrets to systemctl:** `src/daemon/systemd.ts:394-402`
  - body: `resolveSystemctlUserBusEnv` builds the child env from `{ ...process.env, ...env }`. Status paths pass an env merged with parsed service `Environment`/`EnvironmentFile` values, including gateway tokens and other credentials, so when bus vars are missing this forwards service-only secrets to every `systemctl` child. Build the child env from inherited process env plus only the DBus/XDG overrides needed for the probe.
  - confidence: 0.88

## Security Review
