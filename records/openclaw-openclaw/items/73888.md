## Summary

Keep this PR open, but do not merge the current head: the intended systemd user-bus recovery is still useful, while the patch currently forwards parsed service-unit environment values to `systemctl` when synthesizing the bus env.

## What This Changes

The branch adds Linux user-bus env synthesis for direct `systemctl --user` gateway probes, adjusts root/user scope handling and unavailable classification, adds regressions, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the direct bus-recovery direction, but build the synthesized child env from inherited process env plus only `XDG_RUNTIME_DIR` and `DBUS_SESSION_BUS_ADDRESS`, with a regression proving service env secrets are not forwarded.

## Reproduction Assessment

Yes. #63561 provides concrete Ubuntu status/install steps, current `main` shows direct `systemctl --user` calls have no env recovery, and the PR-specific regression is reproducible with a service `EnvironmentFile` token plus missing bus vars.

## Solution Assessment

No, not as-is. The behavioral fix is appropriately scoped, but the implementation should not pass parsed service command env to `systemctl`; limiting the child env to the required DBus/XDG values is the safer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forwarding service secrets to systemctl:** `src/daemon/systemd.ts:394-402`
  - body: `resolveSystemctlUserBusEnv` builds the child env from `{ ...process.env, ...env }`. Status paths pass an env merged with parsed service `Environment`/`EnvironmentFile` values, including gateway tokens and other credentials, so when bus vars are missing this forwards service-only secrets to every `systemctl` child. Build the child env from inherited process env plus only the DBus/XDG overrides needed for the probe.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete local secret-exposure risk by forwarding merged service-unit environment values to `systemctl`.

Concerns:

- **[medium] Service secrets can be forwarded to systemctl:** `src/daemon/systemd.ts:394`
  - body: `readGatewayServiceState` merges parsed service env into systemd probes, and the PR then spreads that env into the child process environment. Service unit `EnvironmentFile` values can include gateway tokens or provider credentials, so the fix should only add DBus/XDG variables instead of forwarding all service env.
  - confidence: 0.86
