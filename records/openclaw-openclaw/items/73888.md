## Summary

Keep open. The PR targets a real current-main gateway/systemd bug, but the current head still introduces a security-sensitive regression by forwarding merged service-unit environment values into spawned `systemctl` processes.

## What This Changes

The PR derives missing Linux user-bus environment for `systemctl --user`, adjusts root/non-root scope selection, narrows systemd-unavailable classification, adds regressions, and updates the changelog.

## Best Possible Solution

Keep the env-recovery approach, but synthesize the child env from inherited process env plus only `XDG_RUNTIME_DIR` and `DBUS_SESSION_BUS_ADDRESS`, with a regression proving service EnvironmentFile secrets are not forwarded.

## Reproduction Assessment

Yes. #63561 provides concrete Ubuntu status/install steps, current main lacks direct `systemctl --user` env recovery, and the PR-specific regression is reproducible by passing a service EnvironmentFile token with missing bus vars.

## Solution Assessment

No, not as-is. The behavioral direction is narrow and useful, but the implementation should not pass parsed service command environment to `systemctl`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forwarding service secrets to systemctl:** `src/daemon/systemd.ts:394-402`
  - body: When DBus/XDG vars are missing, this helper spreads the caller `env` into the `systemctl` child. Gateway status/service-state paths pass an env merged with parsed service `EnvironmentFile` values, so service-only tokens can be exposed to unrelated `systemctl` child processes; build from process env plus only the DBus/XDG overrides needed for the probe.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff can expose service EnvironmentFile secrets to spawned `systemctl` processes.

Concerns:

  - confidence: 0.86
