## Summary

Keep open: current main still lacks the user-bus env recovery needed for the linked gateway/systemd bug, but this PR should not merge as-is because it forwards merged service-unit environment values to `systemctl` children.

## What This Changes

The PR adds Linux `systemctl --user` DBus/XDG environment recovery for gateway status/install probes, adjusts root target-user handling, narrows unavailable classification, and adds daemon tests plus a changelog entry.

## Best Possible Solution

Keep the DBus/XDG recovery, but build the `systemctl` child env from inherited process env plus only the required DBus/XDG overrides, with a regression proving service secrets are not forwarded.

## Reproduction Assessment

Yes, source-level. The linked issue has concrete Ubuntu status/install logs, and current main still lacks a child env override while preserving the permission-denied user-bus false-unavailable path.

## Solution Assessment

No, not as-is. The env-recovery direction is narrow and maintainable, but this implementation should not pass parsed service `Environment=` or `EnvironmentFile=` values to `systemctl` children.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forwarding service secrets to systemctl:** `src/daemon/systemd.ts:394`
  - body: `resolveSystemctlUserBusEnv` builds the child env from `{ ...process.env, ...env }`. Status reads the installed unit and merges parsed `Environment=`/`EnvironmentFile=` values into `env` before probing systemd; those values can include gateway tokens or provider credentials, so missing bus env causes service-only secrets to be forwarded to the local `systemctl` child. Build the child env from inherited process env plus only the DBus/XDG overrides needed for the probe.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete local secret-exposure risk by forwarding merged service-unit environment values to `systemctl`.

Concerns:

- **[medium] Service secrets can be forwarded to systemctl:** `src/daemon/systemd.ts:394`
  - body: `readGatewayServiceState` merges parsed service env into systemd probes, and the PR then spreads that env into the child process environment. Service unit `EnvironmentFile` values can include gateway tokens or provider credentials, so the fix should only add DBus/XDG variables instead of forwarding all service env.
  - confidence: 0.86
