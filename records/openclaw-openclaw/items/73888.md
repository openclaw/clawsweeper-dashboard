## Summary

Keep open. The PR addresses a real current-main gateway/systemd gap and carries forward useful context from #63561 and #64020, but the current diff is not merge-ready because its synthesized `systemctl --user` environment forwards the merged service environment, including parsed unit `EnvironmentFile` values, to child `systemctl` processes.

## What This Changes

The branch adds Linux `systemctl --user` DBus environment synthesis for gateway status/install, broadens root-wrapper machine-scope selection, narrows systemctl-missing classification, adds regressions, and updates `CHANGELOG.md`.

## Best Possible Solution

Keep the direct user-bus recovery and classifier changes, but construct the synthesized child environment from the inherited process environment plus only the whitelisted `XDG_RUNTIME_DIR` and `DBUS_SESSION_BUS_ADDRESS` values, with a regression proving service-unit secrets are not forwarded.

## Reproduction Assessment

Yes. #63561 gives concrete Ubuntu gateway status/install steps, and the PR-specific regression is reproducible with a systemd service `EnvironmentFile` token plus missing DBus/XDG vars, then inspecting the `systemctl` child env.

## Solution Assessment

No, not as-is. The fix boundary is appropriately narrow, but the child env construction should whitelist only DBus/XDG recovery values instead of spreading the full merged gateway service env.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forwarding service secrets to systemctl:** `src/daemon/systemd.ts:394-402`
  - body: `resolveSystemctlUserBusEnv` builds the child env from `{ ...process.env, ...env }`, but status paths pass an env merged with parsed service `Environment`/`EnvironmentFile` values. When bus vars are missing this forwards service-only tokens and provider credentials to every direct `systemctl --user` child; build the env from inherited process env plus only the DBus/XDG overrides needed for the probe.
