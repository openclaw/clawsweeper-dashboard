## Summary

Keep this PR open: the direction is useful, but it is not merge-ready because the future-version guard still runs after a write-capable preflight path, interactive doctor can still keep repair prompts under detected skew, and real CLI proof is missing.

## What This Changes

The PR adds a future-version guard around doctor repair and gateway-token config writes, downgrades selected doctor repair prompts, suppresses the generic doctor --fix hint under skew, and adds focused tests.

## Best Possible Solution

Run the future-version decision before any write-capable doctor preflight or health repair, key the read-only prompter to detected skew, and add redacted real CLI proof for the skewed-config path.

## Reproduction Assessment

Yes, from source inspection: the PR calls a write-capable preflight before resolving the future-config block, and current main shows both prefix recovery and last-known-good recovery write the config file. I did not execute the CLI because this review is read-only.

## Solution Assessment

No: the proposed direction is right, but the guard needs to run before repair-capable preflight writes and the prompter should become read-only for any detected future skew, not only explicit repair mode.
