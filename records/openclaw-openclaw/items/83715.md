## Summary

Keep open: the change is directionally right, but the guard still runs after a write-capable config preflight and the prompter downgrade does not cover every detected future-version skew. The PR also has only test/check command claims, not after-fix real behavior proof from a real doctor run.

## What This Changes

The PR adds a future-version guard to doctor config repair, gateway-token generation, downstream health repair prompts, and the generic doctor --fix hint, with focused tests.

## Best Possible Solution

Run the future-version decision before any write-capable doctor preflight or health repair, make the read-only prompter apply to every detected version skew unless the explicit downgrade override is set, and prove the real CLI path with redacted terminal output.

## Reproduction Assessment

Yes from source inspection: the PR calls a write-capable preflight before resolving the future-config block, and current main shows both prefix recovery and last-known-good recovery write the config file. I did not execute the CLI because this review was required to keep the checkout read-only.

## Solution Assessment

No: the proposed direction is right, but the guard needs to run before any repair-capable preflight write and the prompter should become read-only for any detected future skew, not only explicit repair mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Guard before preflight can write config:** `src/commands/doctor-config-flow.ts:79`
  - body: This still runs `runDoctorConfigPreflight({ repairPrefixedConfig: true })` before resolving `futureConfigBlock`. That preflight can strip a prefixed config or restore last-known-good and write `openclaw.json`, so `doctor --fix` or `--generate-gateway-token` can mutate a newer-written config before the mismatch block runs. Move the future-version guard before write-capable recovery, or make recovery enforce the same guard.
  - confidence: 0.88
- **[P2] Downgrade the prompter on any future skew:** `src/flows/doctor-health.ts:59-68`
  - body: `repairBlockedByFutureConfig` is set from `requestedRepair`, so an interactive `openclaw doctor` that detects a newer-written config still keeps the original prompter. Normal doctor mode can still call `confirmAutoFix` or `confirmRuntimeRepair`, so later health contributions can perform accepted repairs instead of staying read-only. Key this off the detected skew, or return a separate block flag for the prompter.
  - confidence: 0.78
