## Summary

Keep open: the PR addresses a real doctor/config downgrade safety gap, but the implementation still lets a repair-capable preflight run before the new guard, keeps repair-capable prompts for normal future-skew doctor runs, and lacks external real behavior proof.

## What This Changes

The PR adds future-version checks to doctor config repair/token flows, suppresses doctor --fix hints during version skew, makes selected doctor prompters read-only, and adds focused regression tests.

## Best Possible Solution

Rebase and move future-version detection ahead of every write-capable doctor preflight or recovery path, make all future-skew doctor prompts read-only unless the explicit override is set, then add focused tests plus redacted CLI proof.

## Solution Assessment

No. The direction is useful, but the best fix needs the future-version guard before repair-capable preflight writes and needs downstream doctor prompts read-only for every detected future skew unless the explicit override is set.
