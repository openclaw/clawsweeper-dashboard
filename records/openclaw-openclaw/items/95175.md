## Summary

Keep open: this draft, maintainer-labeled PR addresses a real current-main reviewer-device gap in chat-triggered exec approvals, and the patch shape looks correct after source review. Because it broadens an approval authorization path, it should stay in human maintainer review until the security-boundary behavior is accepted and ideally proven on a real mobile/operator client.

## What This Changes

The branch carries the initiating Gateway device id through chat-triggered agent turns, lets trusted approval runtimes bind that device as an exec approval reviewer, and adds focused protocol/gateway/agent coverage.

## Best Possible Solution

Land a maintainer-approved version that preserves requester-bound approval behavior while adding reviewer-device visibility only for trusted approval runtimes, with focused coverage and one real mobile/operator approval proof.

## Reproduction Assessment

Yes, from source inspection: current main binds chat-triggered backend exec approvals to the runtime requester device and does not carry the initiating Gateway device id through the agent run. I did not run a live mobile repro in this read-only review.

## Solution Assessment

Yes, this appears to be the right layer: Gateway chat owns the initiating device identity, the agent run carries turn context, and approval-shared remains the central visibility/resolve gate. The safer alternative would be to reject reviewer-device binding entirely, which is a product/security decision rather than a smaller code fix.
