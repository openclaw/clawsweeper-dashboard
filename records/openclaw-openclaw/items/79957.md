## Summary

Keep open: current main still has a source-reproducible OpenRouter DeepSeek V4 payload path where generic OpenRouter reasoning injection can add nested `reasoning.effort` before the DeepSeek V4 wrapper adds top-level `reasoning_effort`. The existing coverage checks the top-level value but does not assert that the nested field is absent, so this fresh report is not already covered by the earlier OpenRouter DeepSeek fixes.

## What This Changes

The issue requests that OpenRouter DeepSeek V4 Flash requests stop sending both top-level `reasoning_effort` and nested `reasoning.effort` when thinking is enabled.

## Best Possible Solution

Make the final OpenRouter DeepSeek V4 payload choose exactly one reasoning-effort representation, preferably by clearing `reasoning` when the DeepSeek V4 wrapper emits top-level `reasoning_effort` or by having the OpenRouter path consistently use the nested OpenRouter shape, then add a composed-wrapper regression test that asserts the other field is absent.

## Reproduction Assessment

Yes, at source level: a non-off OpenRouter DeepSeek V4 request can pass through the generic OpenRouter wrapper that creates `reasoning.effort`, then through the DeepSeek V4 wrapper that adds `reasoning_effort`. I did not establish a live HTTP 400 against OpenRouter in this read-only pass.

## Solution Assessment

Yes, the best fix is narrow and belongs in the OpenRouter-owned stream wrapper or the shared DeepSeek V4 payload helper, with coverage for the composed wrapper order. This should not require a new config knob or broader product decision.
