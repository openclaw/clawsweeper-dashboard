## Summary

Keep open: current main still has a source-reproducible OpenRouter DeepSeek V4 payload path that can send both nested `reasoning.effort` and top-level `reasoning_effort`, and an open fix PR already explicitly closes this issue.

## What This Changes

The issue requests that OpenRouter DeepSeek V4 requests send exactly one reasoning-effort field instead of both top-level and nested forms.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79961 or an equivalent narrow fix with regression coverage that proves OpenRouter DeepSeek V4 emits only one reasoning-effort field.

## Reproduction Assessment

Yes, at source level: OpenAI-compatible OpenRouter payload construction can create nested `reasoning.effort`, the OpenRouter wrapper preserves it, and the DeepSeek V4 wrapper then adds top-level `reasoning_effort`. I did not run a live OpenRouter request.

## Solution Assessment

Yes, the narrow fix is to make the final DeepSeek V4 OpenRouter payload choose one field, and the linked PR's deletion of nested `reasoning` after setting top-level `reasoning_effort` matches the documented OpenClaw route behavior.
