## Summary

Keep open: current main still has a source-reproducible duplicate reasoning payload path for OpenRouter DeepSeek V4 when thinking is enabled, and an open closing PR already targets this exact bug.

## What This Changes

The issue reports that OpenRouter DeepSeek V4 requests with thinking enabled send both top-level `reasoning_effort` and nested `reasoning.effort`, causing OpenRouter to reject the payload with HTTP 400.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79961 or an equivalent narrow fix that makes the final OpenRouter DeepSeek V4 payload emit exactly one reasoning-effort shape, with regression coverage for the composed wrapper path.

## Reproduction Assessment

Yes, at source level: current main can add nested `reasoning.effort` in the OpenRouter path and then add top-level `reasoning_effort` in the DeepSeek V4 wrapper for the same payload. I did not run a live OpenRouter request in this read-only review.

## Solution Assessment

Yes, the linked PR's direction is the narrow maintainable fix because it removes the conflicting nested field at the DeepSeek V4 wrapper branch that emits flat `reasoning_effort`. The issue should remain open until that fix or an equivalent one merges.
