## Summary

Keep open. The PR addresses a real current-main gateway cleanup gap, but the new non-Linux argv fallback can still miss long LaunchAgent/Homebrew commands because it calls `ps` without wide output, and the required user-facing changelog entry is missing.

## What This Changes

The PR changes Unix stale gateway listener detection to inspect `node`/`nodejs`/`bun` process argv with `isGatewayArgv` and adds a regression test for a node-launched gateway.

## Best Possible Solution

Land a focused argv-based stale listener detector that preserves the existing command fast path, requests full argv on non-Linux, covers long LaunchAgent paths, and includes the required changelog entry.

## Reproduction Assessment

Yes. Current main can be checked with an `lsof -Fpc` payload whose command is `node` and whose argv is an OpenClaw gateway command; main ignores it because the command field lacks `openclaw`. The remaining PR-specific gap is reproducible with a long non-Linux `ps` command string that would be truncated without wide mode.

## Solution Assessment

No, not as currently patched. The code direction is narrow and maintainable, but the non-Linux `ps` call should request untruncated output and the user-facing fix needs a changelog entry before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Request untruncated argv from ps:** `src/infra/restart-stale-pids.ts:116`
  - body: The non-Linux path uses `ps -p <pid> -o command=` without wide mode. On BSD/macOS this output can be width-truncated, so long Homebrew/LaunchAgent commands can lose `dist/index.js` or `gateway` and this detection path will still miss the stale PID. Add wide output, such as `-ww`, and cover that long-command case.
