## Summary

Keep open. Current main and the latest release still lack a broad private-network opt-in for arbitrary `web_fetch` URLs, and live discussion says this remains a security-sensitive config/API decision with open implementation candidates rather than a cleanup close.

## What This Changes

Add a default-off `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach private/internal network URLs when explicitly enabled.

## Best Possible Solution

Land or replace the open work with one canonical default-off `tools.web.fetch.ssrfPolicy` private-network opt-in that preserves existing RFC2544/IPv6 ULA behavior, partitions cache entries by policy, updates docs/generated metadata/changelog, and has targeted SSRF tests after maintainer security approval.

## Reproduction Assessment

Yes, as source proof rather than a runtime bug repro. Current main's strict `ToolsWebFetchSchema` rejects the requested broad key, and `runWebFetch` only forwards the two narrow fake-IP SSRF flags.

## Solution Assessment

No, not as the exact top-level key proposed. The capability is valid, but the maintainable path is a maintainer-approved nested SSRF policy shape with explicit metadata, LAN, redirect, cache, docs, and test coverage.
