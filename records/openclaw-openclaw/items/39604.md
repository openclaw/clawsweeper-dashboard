## Summary

Keep open: current main and v2026.5.22 still lack a broad private-network opt-in for arbitrary `web_fetch`, and the linked implementation remains open/conflicting with an explicit maintainer note that the SSRF API and threat model still need review.

## What This Changes

Request to add a default-off `tools.web.fetch.allowPrivateNetwork` configuration option so `web_fetch` can explicitly reach localhost and private/internal network targets.

## Best Possible Solution

Keep this issue open until maintainers choose one canonical default-off `web_fetch` SSRF policy surface and land it with metadata/link-local, redirect, DNS-rebinding, cache, audit, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.22 have a strict `ToolsWebFetchSchema` without the requested broad key, and `runWebFetch` forwards only the two narrow fake-IP SSRF flags while tests still assert private targets are blocked by default.

## Solution Assessment

No, not as an unreviewed top-level key alone. The safer maintainable path is a maintainer-approved nested or per-agent SSRF policy contract that composes with existing fake-IP flags and documents the security boundary.
