## Summary

Keep open: current main and the latest release still lack a general private-network opt-in for arbitrary `web_fetch` URLs, and the linked implementation path is unmerged while maintainer discussion calls this a separate security-sensitive config decision.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach private or internal addresses when explicitly enabled.

## Best Possible Solution

Keep strict web_fetch defaults, and if maintainers approve the capability, land one canonical default-off SSRF policy shape with docs and focused coverage for default blocking, explicit opt-in, metadata/link-local handling, redirects, and cache isolation.

## Reproduction Assessment

Yes, at source level: current main accepts only narrow `tools.web.fetch.ssrfPolicy` fake-IP range flags and does not accept the requested private-network key. A live localhost fetch was not necessary to prove the missing schema/runtime surface.

## Solution Assessment

Unclear: exposing a default-off private-network opt-in is plausible, but the exact flat key versus nested SSRF policy shape and metadata/LAN/redirect behavior need maintainer security/product approval before it can be called the best fix.
