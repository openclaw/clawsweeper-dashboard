## Summary

Current main and v2026.6.8 still keep arbitrary `web_fetch` private-network targets blocked; this remains the canonical security-sensitive config decision, with draft/stale PRs as implementation candidates rather than a cleanup close.

## What This Changes

The issue requests an opt-in `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach localhost and private network addresses when explicitly enabled.

## Best Possible Solution

Keep this as the canonical issue until maintainers settle a default-off `web_fetch` SSRF policy shape that composes with current fake-IP flags, has explicit hard-deny/redirect/cache semantics, and lands with focused tests and real configured-run proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 accept only narrow `tools.web.fetch.ssrfPolicy` fake-IP flags while docs and tests keep private/internal targets blocked for arbitrary `web_fetch` URLs. I did not run a live localhost fetch because the config/schema/runtime gap is directly visible in source.

## Solution Assessment

No final solution is selected yet. A broad top-level `allowPrivateNetwork` key is likely not the best current shape because current main already uses nested `ssrfPolicy`; the safer path is a maintainer-approved nested or per-agent policy that preserves shipped fake-IP behavior and defines metadata/link-local handling.
