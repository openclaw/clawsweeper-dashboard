## Summary

Keep open: current main and v2026.6.6 still expose only narrow fake-IP `web_fetch` SSRF knobs, while maintainer discussion identifies broad private-network access as a separate security-sensitive config decision with no merged landable fix yet.

## What This Changes

Add a default-off `web_fetch` private-network SSRF configuration option so configured agents can fetch loopback, LAN, and private/internal addresses.

## Best Possible Solution

Define one maintainer-approved, default-off `tools.web.fetch.ssrfPolicy` contract that preserves existing fake-IP flags, documents metadata/link-local and redirect behavior, partitions cache by every policy bit, and lands with focused tests plus real configured-run proof.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.6 shows a strict `tools.web.fetch` schema without a broad private-network key and runtime policy forwarding only the narrow fake-IP SSRF flags.

## Solution Assessment

No for the literal top-level key as proposed. The maintainable path is a nested SSRF policy contract, or a maintainer-approved per-agent variant, with explicit security semantics and upgrade proof.
