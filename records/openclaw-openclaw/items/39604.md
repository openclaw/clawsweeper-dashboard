## Summary

Keep open: current main and the latest release still lack a broad private-network opt-in for arbitrary `web_fetch`, and the remaining work is already covered by open implementation discussions that need security/product approval rather than cleanup closure.

## What This Changes

Request an opt-in `tools.web.fetch.allowPrivateNetwork` configuration key so `web_fetch` can reach private and internal network addresses when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers approve one default-off `web_fetch` SSRF policy contract, then land a rebased implementation through https://github.com/openclaw/openclaw/pull/39630, https://github.com/openclaw/openclaw/pull/67421, or a narrow replacement with docs, audit, cache, redirect, metadata/link-local, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.22` have a strict `ToolsWebFetchSchema` without the requested broad private-network key, and `runWebFetch` forwards only the two narrow fake-IP SSRF flags.

## Solution Assessment

Unclear for the exact top-level key. The safer maintainable path is a maintainer-approved nested or per-agent SSRF policy contract that composes with existing fake-IP flags and documents the security boundary.
