## Summary

Keep open: current main and the latest release still do not expose broad private-network access for arbitrary `web_fetch`, and related open PRs show this remains an active security/product decision rather than cleanup closure.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` config option so `web_fetch` can reach localhost and private/internal network targets when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers approve and land a default-off, audited `web_fetch` SSRF policy contract with cache isolation and explicit metadata, redirect, DNS, and per-agent or allowlist behavior.

## Reproduction Assessment

Yes, at source level: the current strict schema rejects the requested broad private-network key and `runWebFetch` forwards only the two narrow fake-IP SSRF flags. I did not run a live localhost fetch because the config/runtime contract is enough to prove the missing capability.

## Solution Assessment

Unclear for the exact flat `allowPrivateNetwork` key proposed. A maintainer-approved nested or per-agent SSRF policy with audit visibility, cache isolation, redirect/DNS coverage, and hard-deny metadata handling looks safer than exposing a broad global flag directly.
