## Summary

Keep open: current main and the latest release still do not expose broad private-network access for arbitrary `web_fetch`, and the linked open PRs show the remaining work is an active security/product decision rather than cleanup closure.

## What This Changes

Add a default-off `tools.web.fetch` private-network SSRF opt-in so trusted deployments can fetch localhost or internal services.

## Best Possible Solution

Keep this issue open until maintainers approve and land a default-off, audited `web_fetch` SSRF policy contract, ideally scoped per trusted agent and isolated in cache keys.

## Reproduction Assessment

Yes, at source level: the current schema rejects the requested broad private-network key and `runWebFetch` forwards only the two narrow fake-IP SSRF flags. I did not run a live localhost fetch because the source/config contract is enough to prove the missing capability.

## Solution Assessment

Unclear for the exact flat `allowPrivateNetwork` key proposed. A maintainer-approved nested/per-agent SSRF policy with audit visibility, cache isolation, redirect/DNS coverage, and hard-deny metadata handling looks safer than exposing a broad global flag directly.
