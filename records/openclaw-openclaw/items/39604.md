## Summary

Keep open: current `main` still lacks a general `web_fetch` private-network opt-in, and maintainer discussion says the remaining work is a security-sensitive config/API decision rather than an already-implemented fix.

## What This Changes

The issue requests an opt-in `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach private/internal network addresses when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers either land a reviewed nested `tools.web.fetch.ssrfPolicy` private-network policy, likely through https://github.com/openclaw/openclaw/pull/67421 or a replacement, or explicitly reject the SSRF boundary change.

## Reproduction Assessment

Yes. Source inspection shows strict `ToolsWebFetchSchema` accepts only narrow fake-IP `ssrfPolicy` flags and `runWebFetch` only forwards those flags, so a general private/internal target remains blocked on current `main`.

## Solution Assessment

No, not as the originally requested top-level key without more maintainer direction. The safer current path is a single nested `tools.web.fetch.ssrfPolicy` contract with explicit metadata/link-local, redirect, DNS rebinding, audit, docs, and cache-key semantics.
