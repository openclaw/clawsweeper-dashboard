## Summary

Keep open: current main and v2026.6.8 still do not expose a broad private-network opt-in for arbitrary `web_fetch` URLs. The shipped fake-IP flags solve narrower proxy ranges, while this issue remains a security-sensitive public config decision for localhost/RFC1918/LAN targets.

## What This Changes

Request to add a default-off `tools.web.fetch.allowPrivateNetwork`-style config option so `web_fetch` can reach private/internal network targets when explicitly enabled.

## Best Possible Solution

Define one maintainer-approved, default-off nested web_fetch SSRF policy that composes with the existing fake-IP flags, documents metadata/link-local and redirect behavior, partitions caches by every policy bit, and ships with focused security tests.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.8 shows strict web_fetch config accepts only two narrow fake-IP policy flags, while the lower SSRF guard has private-network policy primitives that web_fetch does not expose.

## Solution Assessment

No, not as the original top-level-key proposal. The safer direction is a maintainer-approved nested SSRF policy with explicit metadata/link-local, redirect, cache, docs, and audit semantics.
