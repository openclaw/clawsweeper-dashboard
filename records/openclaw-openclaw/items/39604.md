## Summary

Keep open: current main and the latest release still lack a broad private-network opt-in for arbitrary `web_fetch`, and the linked implementation path remains security-sensitive, conflicting, and not yet maintainer-approved.

## What This Changes

Add a default-off `tools.web.fetch.allowPrivateNetwork` configuration option so `web_fetch` can reach localhost and private/internal network addresses when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers approve one default-off `web_fetch` SSRF policy contract with explicit cache, redirect, DNS rebinding, metadata/link-local, audit, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.22` have a strict `ToolsWebFetchSchema` without the requested broad private-network key, and `runWebFetch` forwards only the two narrow fake-IP SSRF flags.

## Solution Assessment

Unclear for the exact top-level key. The safer maintainable path is a maintainer-approved nested or per-agent SSRF policy contract that composes with existing fake-IP flags and documents the security boundary.
