## Summary

Keep open. Current main still lacks a general web_fetch private-network opt-in, and the issue is paired with open implementation candidates while maintainer discussion says the recent fake-IP/provider change does not close it.

## What This Changes

Request adding a default-off config option so web_fetch can reach private/internal network destinations when explicitly enabled.

## Best Possible Solution

Add a maintainer-approved, default-off private-network opt-in under the existing web_fetch SSRF policy surface, with schema/types/help/docs, cache isolation, and SSRF regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: the strict schema rejects the requested key, runtime forwards only the two fake-IP SSRF flags, and current tests still assert private web_fetch targets are blocked.

## Solution Assessment

No for the exact top-level key as proposed. The safer maintainable path is a maintainer-approved, dangerous-named opt-in under tools.web.fetch.ssrfPolicy with explicit redirect, DNS, metadata, docs, and cache coverage.
