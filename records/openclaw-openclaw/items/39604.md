## Summary

Keep this issue open: current main and v2026.6.6 still lack a broad private-network `web_fetch` opt-in, and maintainer discussion says this remains a security-sensitive product/API decision rather than implemented work.

## What This Changes

The issue requests an explicit config opt-in so `web_fetch` can reach private/internal network addresses while staying blocked by default.

## Best Possible Solution

Define one default-off, maintainer-approved nested web_fetch SSRF policy that preserves existing fake-IP flags, documents the private-network trust boundary, partitions cache by every policy bit, and ships with focused tests plus real configured-run proof.

## Reproduction Assessment

Yes at source level: current main's strict `web_fetch` schema rejects the requested broad private-network key, and runtime only forwards the two existing narrow SSRF policy flags. I did not run a live localhost fetch because this was a read-only review.

## Solution Assessment

No for the exact top-level key proposed in the issue: current code and maintainer discussion point toward a nested `tools.web.fetch.ssrfPolicy` contract. The maintainable solution needs a deliberate security policy for metadata/link-local and redirect/cache behavior.
