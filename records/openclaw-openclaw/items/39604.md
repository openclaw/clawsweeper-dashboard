## Summary

Keep open: current main and the latest release still lack a broad private-network `web_fetch` config surface, and maintainer discussion explicitly says the merged fake-IP work does not close this. The remaining work is a security-sensitive product/API decision, not stale cleanup.

## What This Changes

The issue requests an opt-in `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach private/internal network addresses when enabled.

## Best Possible Solution

Define one maintainer-approved, default-off `tools.web.fetch.ssrfPolicy` contract for broad private-network access that preserves existing fake-IP flags, blocks or documents metadata/link-local behavior, partitions cache entries by all policy bits, and ships with focused tests plus public docs.

## Reproduction Assessment

Yes for source-level reproduction: current strict config rejects the requested key and current `web_fetch` policy only forwards narrow fake-IP flags, so private/internal targets stay blocked by the SSRF guard. I did not run a live local-service fetch in this read-only review.

## Solution Assessment

No, the originally requested top-level key is not clearly the best final shape because current main has moved web_fetch SSRF options under `tools.web.fetch.ssrfPolicy`. The safer solution is a single nested, default-off policy with explicit security semantics and maintainer approval.
