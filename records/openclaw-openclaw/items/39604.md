## Summary

Keep open. Current main still lacks a broad private-network opt-in for arbitrary `web_fetch`, and the linked implementation PR remains open, conflicting, and security-sensitive with maintainer guidance saying this needs explicit review rather than cleanup closure.

## What This Changes

The issue requests a default-off config key allowing `web_fetch` to reach private or internal network addresses when an operator explicitly opts in.

## Best Possible Solution

Choose one canonical default-off `web_fetch` SSRF policy surface, then land or replace the linked implementation with maintainer-approved metadata policy, docs, generated config metadata, cache partitioning, and focused SSRF tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the strict `web_fetch` schema rejects the requested key, the runtime forwards only narrow fake-IP SSRF flags, and existing tests/docs still treat private hosts as blocked.

## Solution Assessment

Unclear as proposed. A default-off opt-in is a plausible direction, but maintainer discussion points toward a nested SSRF policy shape and requires security review before this is the maintainable solution.
