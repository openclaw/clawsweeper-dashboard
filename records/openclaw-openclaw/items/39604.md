## Summary

Keep open. Current main still lacks a web_fetch private-network opt-in, and the remaining decision is a security-sensitive SSRF policy expansion with open implementation candidates already linked.

## What This Changes

The issue requests a default-off config option allowing web_fetch to reach localhost and private-network targets when explicitly enabled.

## Best Possible Solution

Add a maintainer-approved default-off private-network policy under the existing web_fetch SSRF policy surface, with schema/types/help/docs, runtime forwarding, cache-key isolation, and SSRF regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows the strict schema rejects the requested key, web_fetch forwards only the existing narrow SSRF policy bits, and current tests still assert private targets are blocked.

## Solution Assessment

No for the exact top-level key as proposed. The safer maintainable direction is a maintainer-approved default-off field under the existing tools.web.fetch.ssrfPolicy surface, likely using dangerous private-network naming consistent with the security-sensitive behavior.
