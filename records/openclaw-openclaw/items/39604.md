## Summary

Keep open. Current main still lacks a broad private-network opt-in for arbitrary `web_fetch` URLs, the issue is paired with the still-open implementation PR at https://github.com/openclaw/openclaw/pull/39630, and maintainer discussion says the merged fake-IP work does not settle this security-sensitive config decision.

## What This Changes

Add a default-off `tools.web.fetch.allowPrivateNetwork` configuration path so `web_fetch` can reach private/internal network destinations when explicitly enabled.

## Best Possible Solution

Land or replace the paired work with one maintainer-approved, default-off `web_fetch` SSRF policy contract that preserves fake-IP flags, handles metadata/link-local policy, partitions cache keys by policy, covers redirects, and updates docs, config metadata, changelog, and targeted tests.

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is a feature/config request; source inspection gives high confidence that current main lacks the requested `web_fetch` config surface.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed top-level key; the safer maintainable path is a maintainer-approved nested SSRF policy shape with explicit metadata, redirect, cache, docs, and operator-trust boundaries.
