## What This Changes

The issue requests an opt-in `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach private/internal addresses when enabled.

## Best Possible Solution

Define one default-off, maintainer-approved web_fetch SSRF policy contract that composes with existing fake-IP flags, is audited and documented, and lands through a canonical PR with source and real behavior proof.

## Reproduction Assessment

Yes, at source level: current main accepts only narrow `web_fetch` fake-IP SSRF policy keys and still blocks localhost/private targets through the guarded fetch path. I did not run a live localhost fetch because the missing public config contract is visible in source and tests.

## Solution Assessment

No for the exact top-level key as proposed; the safer direction is a maintainer-approved nested or per-agent `web_fetch` SSRF policy that composes with existing fake-IP flags, partitions cache by effective policy, and documents the operator trust boundary.
