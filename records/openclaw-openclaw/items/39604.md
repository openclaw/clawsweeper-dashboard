## Summary

Keep open: current main and latest release still do not expose a broad private-network opt-in for arbitrary `web_fetch`, and the linked implementation at https://github.com/openclaw/openclaw/pull/39630 remains open, conflicting, and security-sensitive.

## What This Changes

Request an opt-in `tools.web.fetch.allowPrivateNetwork` config key so `web_fetch` can reach localhost, LAN, and private/internal network targets when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers approve one default-off `web_fetch` SSRF policy contract with cache isolation, redirect handling, DNS rebinding behavior, metadata/link-local rules, audit output, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.22` have a strict `ToolsWebFetchSchema` without the requested broad private-network key, and `runWebFetch` forwards only the two narrow fake-IP SSRF flags.

## Solution Assessment

Unclear for the exact top-level key. The safer maintainable path is a maintainer-approved nested or per-agent SSRF policy contract that composes with existing fake-IP flags and documents the security boundary.
