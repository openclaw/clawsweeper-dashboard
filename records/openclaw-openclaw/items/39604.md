## Summary

Keep open: current main and the latest release still lack a broad private-network opt-in for arbitrary `web_fetch` URLs, and an open implementation candidate still references this issue while maintainer discussion says the remaining blocker is security/product policy.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` config option so `web_fetch` can reach private/internal network targets when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers choose and land one canonical default-off `web_fetch` SSRF policy surface with metadata/link-local, redirect, DNS rebinding, allowlist, audit, cache, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main's strict `ToolsWebFetchSchema` rejects the requested broad key and `runWebFetch` forwards only narrow fake-IP SSRF flags, so the config gap is high-confidence without running a write-producing test.

## Solution Assessment

No for the exact top-level key as the only answer. The safer maintainable path is a maintainer-approved nested or per-agent SSRF policy contract that composes with existing fake-IP flags and documents the security boundary.
