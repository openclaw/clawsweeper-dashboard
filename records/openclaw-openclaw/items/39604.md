## Summary

Keep open: current main and latest release still do not expose a broad private-network `web_fetch` opt-in, and the remaining work is a security-sensitive config/product decision rather than stale cleanup.

## What This Changes

The issue requests an opt-in `tools.web.fetch.allowPrivateNetwork` setting so `web_fetch` can reach private or internal network addresses when explicitly enabled.

## Best Possible Solution

Keep this as the canonical issue until maintainers choose and land a default-off nested `tools.web.fetch.ssrfPolicy` contract, likely through https://github.com/openclaw/openclaw/pull/67421 or a replacement, with docs, audit coverage, cache partitioning, and focused SSRF tests.

## Reproduction Assessment

Yes at source level. Current main and `v2026.6.6` both show a strict `web_fetch` schema with only narrow fake-IP SSRF keys, while the lower SSRF guard already has a broader private-network primitive that is not exposed through `web_fetch`.

## Solution Assessment

No for the originally requested top-level key as-is. The better direction is a maintainer-approved nested `ssrfPolicy` contract that composes with existing fake-IP flags and explicitly documents metadata/link-local, redirect, DNS rebinding, cache, and audit behavior.
