## Summary

Keep open: current main and v2026.5.20 still lack a broad private-network opt-in for arbitrary `web_fetch`, and the remaining work is an SSRF API/security decision with open implementation candidates rather than cleanup-close material.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` configuration option so `web_fetch` can reach private or internal network targets when explicitly enabled.

## Best Possible Solution

Keep this issue open until maintainers choose one canonical default-off `web_fetch` SSRF policy surface and land it with metadata/link-local, redirect, DNS-rebinding, cache, audit, docs, and regression coverage handled in https://github.com/openclaw/openclaw/pull/39630, https://github.com/openclaw/openclaw/pull/67421, or a replacement.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.20 have a strict `ToolsWebFetchSchema` without the requested broad key, and `runWebFetch` forwards only the two narrow fake-IP SSRF flags.

## Solution Assessment

No for the exact top-level key as the only answer. The safer maintainable path is a maintainer-approved nested or per-agent SSRF policy contract that composes with existing fake-IP flags and documents the security boundary.
