## Summary

Keep open. Current main still lacks a broad private-network opt-in for arbitrary `web_fetch` URLs, the issue is paired with the still-open implementation PR at https://github.com/openclaw/openclaw/pull/39630, and a recent maintainer comment says this remains a separate security-sensitive config decision after https://github.com/openclaw/openclaw/pull/76887.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` configuration option so `web_fetch` can reach private and internal network addresses when explicitly enabled.

## Best Possible Solution

Land or replace the open implementation with one canonical default-off `web_fetch` private-network opt-in under the approved `ssrfPolicy` shape, preserving existing fake-IP flags, cache partitioning, metadata protections, docs, generated config metadata, changelog, and targeted SSRF tests.

## Reproduction Assessment

Yes, as source proof rather than a live runtime repro: current main's strict `web_fetch` schema rejects the requested broad private-network key, and the runtime forwards only the two narrow fake-IP SSRF flags. The existing SSRF tests and docs also preserve private/internal blocking by default.

## Solution Assessment

No, not as the exact top-level key proposed. The capability is valid, but the maintainable path needs maintainer-approved nested SSRF policy shape and security review for metadata, LAN, redirect, and cache behavior before shipping.
