## Summary

Keep this open: current main still does not expose the requested arbitrary private-network opt-in for `web_fetch`, and the active discussion explicitly says the remaining work is a security-sensitive config decision rather than implemented cleanup.

## What This Changes

Request to add an opt-in `tools.web.fetch.allowPrivateNetwork` boolean so `web_fetch` can reach localhost, LAN, and private IP targets when explicitly enabled.

## Best Possible Solution

Define the accepted private-network policy shape for `web_fetch`, then update and review one canonical implementation PR with schema, runtime, docs, changelog, SSRF tests, redirect handling, and metadata-host coverage.

Yes for the current boundary by source inspection: current main still blocks localhost/private targets in `web_fetch`, and the requested config key is absent from the strict schema. I did not run a live fetch because this is a feature/config-policy request rather than a broken existing contract.

Unclear. A private-network opt-in is plausible, but the maintainer discussion points to unresolved metadata, LAN, and redirect policy, so the safer path is a reviewed `ssrfPolicy`/dangerously named config design rather than closing or auto-applying the broad boolean as-is.
