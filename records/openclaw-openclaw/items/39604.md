## Summary

Keep open. Current main still does not expose a broad private-network opt-in for arbitrary `web_fetch` URLs, this issue is paired with the still-open implementation PR at https://github.com/openclaw/openclaw/pull/39630, and recent maintainer discussion explicitly keeps this as a separate security-sensitive config decision after https://github.com/openclaw/openclaw/pull/76887.

## What This Changes

Request an opt-in `tools.web.fetch` configuration setting that lets `web_fetch` reach loopback and private/internal network addresses while preserving strict SSRF blocking by default.

## Best Possible Solution

Land or replace the paired implementation with one maintainer-approved, default-off `web_fetch` SSRF policy shape that preserves existing fake-IP flags, metadata protections, redirect behavior, cache isolation, docs, generated config metadata, changelog, and targeted tests.

## Reproduction Assessment

Yes, as source proof rather than a live runtime repro: current main's strict `web_fetch` schema does not accept a broad private-network key, and runtime forwarding only includes the two narrow fake-IP SSRF flags.

## Solution Assessment

Unclear for the exact proposed top-level key. The capability is valid, but the maintainable solution needs maintainer-approved SSRF policy shape and threat-boundary review before changing arbitrary `web_fetch` URL access.
