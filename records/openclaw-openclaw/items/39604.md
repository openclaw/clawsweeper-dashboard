## Summary

Keep open: current main and the latest release still do not expose broad private-network access for arbitrary `web_fetch` URLs, and the remaining work is a security-sensitive public config decision with active but unmerged implementation candidates.

## What This Changes

The issue requests an opt-in `tools.web.fetch.allowPrivateNetwork` config key so arbitrary `web_fetch` calls can reach private/internal network addresses.

## Best Possible Solution

Define one maintainer-approved, default-off web_fetch SSRF policy that composes with existing fake-IP flags, partitions cache by every policy bit, documents the operator trust boundary, covers metadata/link-local/redirect behavior, and lands with focused tests plus security review.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` expose only narrow `web_fetch` fake-IP SSRF policy keys and still document private/internal hosts as blocked. I did not run a live localhost fetch because the remaining blocker is the missing public config/security contract, not a disputed runtime path.

## Solution Assessment

No for the exact top-level key as proposed; the safer likely solution is a nested, default-off `tools.web.fetch.ssrfPolicy` capability that preserves current fake-IP behavior and gets explicit maintainer security approval. The open per-agent PR may inform the shape, but it has not merged and is not yet canonical.
