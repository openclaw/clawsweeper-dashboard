## Summary

Keep open: current main and the latest release still do not expose a broad private-network opt-in for arbitrary `web_fetch` URLs, and the remaining work is a security-sensitive config/product decision rather than completed cleanup.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` config option so `web_fetch` can reach private or internal network targets when explicitly enabled.

## Best Possible Solution

Define one default-off nested web_fetch SSRF policy that composes with the existing fake-IP flags, keeps metadata/link-local handling explicit, partitions cache by policy, documents the operator trust boundary, and lands through maintainer security review.

## Reproduction Assessment

Yes, at source level: current main's strict web_fetch schema lacks the requested broad private-network key, and the runtime forwards only the two narrow fake-IP policy flags. I did not run a live localhost fetch because this is a feature/security config decision, not a broken existing contract.

## Solution Assessment

No for the exact top-level key as proposed; the safer likely solution is a nested, default-off `tools.web.fetch.ssrfPolicy` capability with explicit metadata/link-local, redirect, and cache semantics. Maintainers still need to approve that security boundary before implementation is final.
