## Summary

Keep open. Current main still lacks a web_fetch private-network opt-in, and the issue is already paired with open implementation candidates that need maintainer review because this is a security-sensitive SSRF policy expansion.

## What This Changes

Add a default-off web_fetch configuration option so operators can allow private/internal network destinations.

## Best Possible Solution

Keep this issue open while maintainers review, land, replace, or close the linked implementation work; the durable fix should add a default-off private-network web_fetch policy under the approved SSRF policy surface with schema, types, help/docs, runtime forwarding, cache-key isolation, and SSRF regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows the strict schema rejects the requested key, the web_fetch runtime forwards only fake-IP SSRF flags, and existing SSRF tests assert localhost/private targets remain blocked.

## Solution Assessment

No for the exact top-level key as proposed. The narrow maintainable direction is a maintainer-approved default-off field under the existing `tools.web.fetch.ssrfPolicy` surface, likely using dangerous private-network naming consistent with adjacent policy surfaces.
