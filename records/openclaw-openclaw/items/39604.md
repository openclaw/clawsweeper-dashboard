## Summary

Keep open. Current main and v2026.6.8 still keep arbitrary web_fetch private/internal targets blocked and expose only narrow fake-IP SSRF flags; the remaining request is a security-sensitive config/product decision with an open linked implementation candidate.

## What This Changes

The issue requests an opt-in tools.web.fetch.allowPrivateNetwork configuration key so web_fetch can access localhost and private/internal network addresses when explicitly enabled.

## Best Possible Solution

Keep this as the canonical web_fetch private-network access decision and resolve it with a maintainer-approved, default-off SSRF policy shape that composes with existing fake-IP flags, partitions cache by all policy bits, preserves metadata protections, and updates tests and public docs.

## Reproduction Assessment

Yes, at source level: current main's strict web_fetch schema rejects the requested private-network key, and runWebFetch forwards only the existing narrow fake-IP SSRF flags. This is a feature/config gap rather than a broken existing contract.

## Solution Assessment

No, not exactly as proposed yet: a private-network opt-in is plausible, but the durable fix should use the maintainer-approved nested SSRF policy shape and settle metadata, redirect, DNS rebinding, and cache behavior before landing.
