## Summary

Keep open: current main and the latest release still do not expose a broad private-network opt-in for arbitrary `web_fetch`, and the linked implementation work remains an active security/product decision rather than cleanup closure.

## What This Changes

The issue requests a default-off `tools.web.fetch.allowPrivateNetwork` configuration option so `web_fetch` can intentionally reach loopback and private/internal network targets.

## Best Possible Solution

Keep this issue open until maintainers approve one default-off `web_fetch` SSRF policy contract, then land https://github.com/openclaw/openclaw/pull/39630, https://github.com/openclaw/openclaw/pull/67421, or a narrow replacement with schema/types/help/docs, audit visibility, cache-key isolation, redirect/DNS-rebinding coverage, and real private-target proof.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.5.22` have a strict `ToolsWebFetchSchema` without the requested broad private-network key, and `runWebFetch` forwards only the two narrow fake-IP SSRF flags.

## Solution Assessment

Unclear for the exact top-level key proposed. The safest maintainable path is a maintainer-approved SSRF policy contract, likely nested and/or per-agent scoped, that composes with the existing fake-IP flags and documents the security boundary.
