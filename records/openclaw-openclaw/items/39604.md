## Summary

Keep open. Current main still lacks a web_fetch private-network opt-in; it only exposes narrow fake-IP proxy exceptions under `tools.web.fetch.ssrfPolicy`, while private/local addresses remain blocked. The issue is also paired with open implementation candidates, so it should stay open until maintainers land, revise, or close the linked work.

## What This Changes

The issue requests a default-off `web_fetch` config option that allows private/internal network targets when explicitly enabled.

## Best Possible Solution

Keep this issue open and review the linked implementation work. The desired end state is a default-off web_fetch private-network policy under the established SSRF policy surface, with strict schema/types/help/docs updates, runtime forwarding, cache-key isolation, and SSRF coverage for direct hosts, DNS resolution, redirects, and default-block behavior.

## Reproduction Assessment

Yes. The source and existing tests provide a high-confidence reproduction path: the strict web_fetch schema does not accept the requested key, and current SSRF tests still assert localhost/private targets are blocked while only the fake-IP exceptions can be enabled.

## Solution Assessment

No for the exact top-level key as proposed; the current maintainable direction is likely to align with `tools.web.fetch.ssrfPolicy` and the dangerous private-network naming used elsewhere. The current main implementation is not sufficient because it still cannot opt web_fetch into private/internal targets.
