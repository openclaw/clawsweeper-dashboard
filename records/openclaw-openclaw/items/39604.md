## Summary

Keep open. Current main still lacks a broad private-network opt-in for arbitrary `web_fetch`, and an open PR with closing syntax is still the implementation candidate while maintainers decide the SSRF policy shape and threat boundary.

## What This Changes

Add a default-off `tools.web.fetch.allowPrivateNetwork` config option so `web_fetch` can reach localhost and private/internal network addresses when explicitly enabled.

## Best Possible Solution

Choose one canonical default-off `web_fetch` SSRF policy surface, then land or replace the linked implementation with approved metadata policy, docs, generated config metadata, cache partitioning, and focused SSRF tests.

## Reproduction Assessment

Yes. Current-main source inspection shows the strict `web_fetch` schema rejects the requested key, the runtime forwards only narrow fake-IP SSRF flags, and tests/docs still treat private hosts as blocked.

## Solution Assessment

Unclear. A default-off opt-in is plausible, but the maintainable solution needs maintainer approval for the nested SSRF policy shape and metadata/LAN/redirect threat model before landing.
