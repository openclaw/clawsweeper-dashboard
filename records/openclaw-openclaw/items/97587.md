## Label Justifications

- `P2`: This is a concrete security/availability hardening fix for a limited Google OAuth path, but the current patch does not yet resolve the issue.

## Summary

Keep open: the intended hardening is at the wrong layer because Google OAuth `fetchWithTimeout` pre-buffers the upstream body before the new bounded JSON reader runs, so the OOM/DoS issue remains until the cap is applied earlier and proven through the actual OAuth path.

## What This Changes

The PR imports `readProviderJsonResponse` in the Google OAuth project and token modules and replaces six `response.json()` calls on userinfo, token, loadCodeAssist, onboard, and operation polling paths.

## Best Possible Solution

Apply the byte cap at the OAuth guarded fetch/read boundary, or return the streaming response with release handled around the bounded reader, then prove an oversized streamed response through the actual token/project OAuth functions is rejected before full buffering.

## Reproduction Assessment

Yes, from source inspection: the changed readers all receive `Response` objects returned by `fetchWithTimeout`, which already awaited `response.arrayBuffer()` on the upstream body. A focused repro would stream an oversized body through an actual OAuth token/project function and observe the pre-buffering before the new cap runs.

## Solution Assessment

No. Reusing `readProviderJsonResponse` is the right helper, but this PR applies it too late; the best fix moves the bound before or into `fetchWithTimeout` and covers that exact path with proof.
