## Summary

Keep this PR open: it is a useful Google OAuth hardening direction, but the patch still applies the JSON cap after `fetchWithTimeout` has already buffered the provider-controlled body, and it also raises plugin SDK surface budgets without a matching SDK surface change.

## What This Changes

The PR routes Google OAuth project and token JSON parses through `readProviderJsonResponse`, adds oversized-body OAuth regression tests, and increases plugin SDK surface budget constants.

## Best Possible Solution

Move the byte limit into the Google OAuth guarded fetch/read lifecycle, remove unrelated SDK budget bumps unless an actual SDK surface change requires them, and prove both oversized streamed and normal OAuth paths.

## Reproduction Assessment

Yes. Source inspection shows the changed OAuth readers are fed by `fetchWithTimeout`, and that helper drains the upstream response with `arrayBuffer()` before the new bounded JSON reader can run.

## Solution Assessment

No. `readProviderJsonResponse` is the right primitive, but the best fix applies the limit before or inside the OAuth fetch helper so the original provider stream is bounded and released safely.
