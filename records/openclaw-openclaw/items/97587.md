## Summary

Keep open: this addresses a real Google OAuth response-bound hardening gap, but the patch only caps the buffered JSON parse step while `fetchWithTimeout` still materializes the upstream response body before the cap can run.

## What This Changes

The PR replaces Google OAuth `response.json()` reads with `readProviderJsonResponse`, adds oversized-body regression tests, and refreshes plugin SDK surface budgets.

## Best Possible Solution

Move the byte cap into the Google OAuth guarded fetch/read lifecycle, preserve proxy and SSRF guard release behavior, and prove oversized streamed plus normal OAuth responses through the actual path.

## Reproduction Assessment

Yes. Source inspection shows `fetchWithTimeout` drains `response.arrayBuffer()` before caller parsing, so a parser-side cap cannot bound the original OAuth response stream; I did not run a live Google OAuth repro.

## Solution Assessment

No. `readProviderJsonResponse` is the right primitive, but the maintainable fix applies it before or inside the OAuth fetch helper so the original provider stream is bounded and released safely.
