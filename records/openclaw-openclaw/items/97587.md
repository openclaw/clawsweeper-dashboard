## Summary

Keep this PR open: the security hardening goal is valid, but the patch applies the JSON cap after the Google OAuth fetch helper has already buffered the whole upstream response, so the claimed OOM protection is not effective on the real path.

## What This Changes

The PR replaces six Google OAuth `response.json()` reads with `readProviderJsonResponse` in the project-discovery and token-exchange modules.

## Best Possible Solution

Move the byte cap to the Google OAuth guarded fetch/read boundary, preserve the existing proxy and release behavior, and prove an oversized streamed OAuth response is rejected before full buffering.

## Reproduction Assessment

Yes, from source inspection: every changed OAuth reader receives a `Response` returned by `fetchWithTimeout`, and that helper already awaits `response.arrayBuffer()` before returning. A focused repro would stream an oversized body through an actual token or project OAuth function and observe the pre-buffering before the new cap runs.

## Solution Assessment

No. Reusing `readProviderJsonResponse` is the right primitive, but the best fix applies the limit before or inside the OAuth fetch helper so the original stream is capped and cancelled.
