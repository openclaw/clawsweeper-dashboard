## Summary

Keep open: the reported Telegram hot-reload wedge is credible, but this PR only changes the worker's eventual exit code and still leaves the stop-timeout/stale-task path that blocks the restart.

## What This Changes

The PR changes the Telegram ingress worker to exit 0 after a requested stop and adds a changelog entry for the linked hot-reload wedge.

## Best Possible Solution

Add regression coverage for isolated Telegram ingress hot-reload shutdown, then fix the worker/supervisor lifecycle so requested stops cannot leave stale task state blocking the restart.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. The source path is credible: isolated ingress shutdown can exceed the 5s stop helper, the timeout path retains store.tasks, and hot reload immediately calls startChannel.

## Solution Assessment

No. Changing the stopped worker's exit code does not address the timeout path that returns before stale task cleanup; the fix should cover supervisor cleanup/restart behavior and add a regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fix the timeout path that still blocks hot reload:** `extensions/telegram/src/telegram-ingress-worker.runtime.ts:174`
  - body: The gateway stop helper already treats both resolved and rejected tasks as settled, so changing a stopped worker's exit code only helps after the worker has completed. The reported wedge happens when stopChannel times out first, leaves store.tasks/store.aborts in place with running: true, and hot reload immediately calls startChannel, which returns because the old task is still tracked. This patch needs to handle that timeout/stale-task path or add a regression proving it no longer wedges.
  - confidence: 0.88
