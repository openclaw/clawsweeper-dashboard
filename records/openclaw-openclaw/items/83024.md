## Summary

Keep open. The PR targets a real Telegram hot-reload wedge, but the patch only changes the worker exit code and does not cover the current stop-timeout path that can still leave stale task state blocking the immediate restart; the PR also lacks required real Telegram/gateway after-fix proof.

## What This Changes

The PR changes Telegram ingress worker top-level error handling to exit 0 after a requested stop and adds a changelog bullet for the linked hot-reload wedge.

## Best Possible Solution

Add focused regression coverage for Telegram isolated-ingress reload shutdown, then fix the worker/supervisor lifecycle so requested-stop worker failures and manual stop timeouts cannot leave stale task state blocking restart.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection plus the linked logs show a credible repro path: a `channels.telegram.*` hot reload where the isolated ingress task does not settle before the stop timeout leaves stale task state that makes restart return early.

## Solution Assessment

No. The one-line exit-code change can help when the worker settles before the supervisor timeout, but it does not fix or prove safe the 5s timeout path that preserves `store.tasks`/`store.aborts` and blocks `startChannel`.

## Review Findings

Overall correctness: patch is incorrect

Please fix or prove that timeout path no longer blocks restart.
