## Summary

Keep open. The patch is a plausible small part of the Telegram shutdown path, but source inspection shows the hot-reload wedge can still happen after the 5s stop timeout because changing the worker exit code does not make the already-timed-out supervisor restart the account.

## What This Changes

The PR changes the Telegram ingress worker to exit with code 0 after a requested stop and adds a changelog entry for the hot-reload channel wedge reported in https://github.com/openclaw/openclaw/issues/83008.

## Best Possible Solution

Add a failing regression for Telegram isolated-ingress hot reload shutdown, then fix the supervisor/worker lifecycle so a requested stop cannot leave stale task state blocking the restart.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The issue logs plus current source give a credible source-reproducible path around isolated ingress worker shutdown, stop timeout, retained store.tasks, and immediate hot-reload start.

## Solution Assessment

No. The one-line exit-code change does not address the timeout path that retains stale task state, and the stop helper already treats rejected tasks as settled when they finish before the timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Fix the timeout path that still blocks hot reload:** `extensions/telegram/src/telegram-ingress-worker.runtime.ts:168`
  - body: The gateway stop helper treats both fulfillment and rejection as a settled stop, so changing a stopped worker's exit code does not make stopChannel finish before its 5s timeout. In the reported hot-reload path, once the timeout returns with store.tasks retained, startChannel immediately no-ops; a later exit 0 still leaves no restart. The patch needs to cover the timeout/stale-task path or add a failing regression around it.
  - confidence: 0.87
