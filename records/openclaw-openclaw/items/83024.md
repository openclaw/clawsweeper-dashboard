## Summary

Keep open. The patch only changes how a completed stopped worker is classified, while current source shows the hot-reload wedge can occur when the stop timeout returns before stale task state is cleared; the PR also still lacks inspectable real Telegram/gateway proof.

## What This Changes

The PR changes the Telegram ingress worker to exit 0 when `main()` rejects after a requested stop and adds a changelog entry for the linked hot-reload wedge.

## Best Possible Solution

Add focused regression coverage for isolated Telegram ingress shutdown during channel hot reload, then fix lifecycle cleanup so requested stops cannot leave tracked tasks blocking restart.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection and the existing stop-timeout test show a credible repro path around isolated Telegram ingress shutdown plus immediate hot-reload restart.

## Solution Assessment

No. The one-line exit-code gate is not sufficient for the timeout path because `stopChannel` can return with `store.tasks` retained before the worker exits; the fix should cover supervisor cleanup/restart behavior and prove it with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fix the timeout path that still blocks hot reload:** `extensions/telegram/src/telegram-ingress-worker.runtime.ts:174`
  - body: The gateway stop helper already treats both resolved and rejected tasks as settled, so changing the exit code only helps after the worker has completed. When `stopChannel` times out first, it leaves `store.tasks`/`store.aborts` in place with `running: true`, and hot reload immediately calls `startChannel`, which returns because the old task is still tracked. Please handle that timeout/stale-task path or add a regression proving it no longer wedges.
  - confidence: 0.88
