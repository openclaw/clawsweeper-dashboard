## Summary

Keep open. The patch addresses one worker exit-code classification, but the reviewed source path still leaves the hot-reload timeout case able to retain stale channel task state and skip the immediate restart; the external PR also still lacks inspectable real Telegram/gateway proof.

## What This Changes

The PR changes the Telegram ingress worker to exit 0 when a requested stop has already been observed and adds a changelog entry for the linked hot-reload wedge.

## Best Possible Solution

Add regression coverage for isolated Telegram ingress hot-reload shutdown, then fix the worker/supervisor lifecycle so requested stops and manual stop timeouts cannot leave stale task state blocking the restart.

## Reproduction Assessment

No. The linked issue includes real logs and current-main source shows a credible timeout/no-op path, but this review did not run a live Telegram hot reload or inspect after-fix runtime output.

## Solution Assessment

No. The one-line exit-code change does not cover the 5s stop-timeout path that can return before stale task cleanup and cause the immediate `startChannel` call to no-op.

## Review Findings
