## Summary

Keep open: this is a maintainer-authored draft PR with the protected `maintainer` label, and current `main` still has the unguarded verbose logging read. The patch is a narrow bug fix and I found no blocking correctness or security findings.

## What This Changes

The PR guards hosted media resolver failure logging so an unreadable `pluginId` falls back to `unknown`, and adds focused regression coverage for continuing to a healthy fallback resolver.

## Best Possible Solution

Land this narrow logging guard with the sibling registry-metadata guard, or rebase the second PR, so hosted media resolution tolerates both malformed registry metadata and malformed per-entry logging metadata without expanding plugin API surface.

## Reproduction Assessment

Yes, by source inspection: on current `main`, a resolver can throw, then verbose logging reads `entry.pluginId` inside the catch block and can throw before the fallback resolver runs. I did not execute the repro in this read-only review, but the PR body includes a matching terminal repro.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the catch-path logging metadata read non-throwing while preserving the existing fallback resolver behavior; the broader unreadable-registry case remains covered by https://github.com/openclaw/openclaw/pull/89778.
