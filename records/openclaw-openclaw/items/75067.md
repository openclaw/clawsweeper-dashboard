## Summary

Keep this PR open, but it is not mergeable as-is. Current main still lacks the proposed pricing resilience work, while the PR head returns from fresh disk-cache warming before preserving configured pricing and schedules both warm-cache and open-circuit retries through the generic 24h timer.

## What This Changes

The PR lowers the Gateway pricing fetch timeout, adds disk-backed pricing cache warming and persistence, adds a pricing fetch circuit breaker, exposes a startup-pending readiness reason, and updates related Gateway tests.

## Best Possible Solution

Keep the pricing resilience direction, but revise the branch so disk cache warming overlays current configured and manifest pricing, retry timers use the relevant remaining TTL or breaker cooldown, the changelog documents the user-facing behavior, and the rebase preserves current readiness semantics.

## Reproduction Assessment

Yes. The findings are reproducible from PR-head code inspection, and focused fake-timer/unit tests can cover fresh disk cache with changed configured costs plus breaker cooldown retry behavior without live network access.

## Solution Assessment

No. The design is reasonable, but this implementation is not the best fix until it preserves pricing precedence and schedules refreshes from the actual cache age or circuit-breaker cooldown instead of the generic daily refresh helper.

- **[P2] Apply configured pricing before disk cache warm return:** `src/gateway/model-pricing-cache.ts:1182-1185`
  - body: This branch returns as soon as any fresh disk cache exists, before collectSeededPricing reads current models.providers.*.models[].cost. If a user changes local pricing while the cache is fresh, stale disk data can win for the cache window; merge or overlay current configured and manifest pricing before returning.
  - confidence: 0.91
- **[P2] Retry when the circuit cooldown expires:** `src/gateway/model-pricing-cache.ts:1232`
  - body: The open-circuit path logs the remaining cooldown but schedules through scheduleRefresh, which waits the 24h cache TTL. After three failures, a recovered pricing service may be ignored until the daily timer instead of being retried after the 10-minute cooldown.
  - confidence: 0.9
- **[P3] Schedule disk refresh for remaining TTL:** `src/gateway/model-pricing-cache.ts:1184`
  - body: A fresh disk cache always schedules a full 24h delay from process start. A 23-hour-old cache can remain in use for almost another day, and repeated restarts can keep postponing the live refresh; schedule from the loaded cache's remaining TTL instead.
  - confidence: 0.84
