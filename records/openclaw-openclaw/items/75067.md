## Summary

Keep open: the pricing resilience work is still useful and not already on current main, but this PR remains incorrect at its current head, is merge-conflicting with current main, and lacks observable real behavior proof.

## What This Changes

The PR reduces the Gateway pricing fetch timeout, adds disk-backed pricing cache warm/persist behavior and a pricing fetch circuit breaker, adds a readiness startupBlockedReason field, and updates focused Gateway tests.

## Best Possible Solution

Rebase onto current main, preserve the current readiness and pricing-health behavior, fix disk-cache precedence and refresh timing, then add redacted real gateway proof for the startup/degraded-network path.

## Reproduction Assessment

Yes for the review blockers: PR-head source shows the disk warm path returns before local/manifest pricing is collected, and the circuit-open path schedules through the 24h helper. I did not run live Gateway restart proof in this read-only review.

## Solution Assessment

No. The resilience direction is maintainable, but this implementation should treat persisted data as a bootstrap source below current configured/manifest pricing and schedule retries from actual remaining TTL or breaker cooldown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Overlay current pricing before disk warm return:** `src/gateway/model-pricing-cache.ts:1182-1185`
  - body: A fresh disk cache returns before `collectSeededPricing` reads current `models.providers.*.models[].cost` and manifest catalog costs. If a user changes local pricing while the persisted cache is fresh, stale disk data can win for the cache window; merge or overlay current configured and manifest pricing before returning.
  - confidence: 0.91
- **[P2] Retry after the circuit cooldown:** `src/gateway/model-pricing-cache.ts:1232`
  - body: When the circuit is open this calls `scheduleRefresh`, but that helper waits the 24h cache TTL. After the upstream service recovers, pricing refresh can remain skipped until the daily timer instead of retrying after the 10-minute cooldown.
  - confidence: 0.9
- **[P3] Schedule disk refresh for remaining TTL:** `src/gateway/model-pricing-cache.ts:1184`
  - body: A fresh disk cache always schedules the next refresh for a full TTL from process start. A 23-hour-old cache can remain in use for almost another day, and repeated restarts can keep extending that delay; schedule from the loaded cache's remaining TTL.
  - confidence: 0.84
