## Label Justifications

- `P1`: The PR targets production Slack gateway crash/restart loops that can take down the core runtime during network outages.

## Summary

Keep this open. The empty Slack Web API request-wrapper fix still addresses a real gap on current main, but the PR is conflicting, lacks after-fix real Slack proof, and the global nullish-rejection guard duplicates already-shipped Slack-specific reconnect handling while changing process-wide crash policy.

## What This Changes

The PR adds Slack Web API request-wrapper transient classification, treats null/undefined global unhandled rejection reasons as non-fatal, and adds focused infra regression tests.

## Best Possible Solution

Keep the Slack Web API empty-wrapper classifier and focused tests, rebase onto current main, and drop or Slack-scope the global nullish guard unless a new producer and explicit policy decision justify it.

## Reproduction Assessment

Yes, at source level for the empty Slack wrapper path: current main skips empty messages in `isTransientNetworkError`, while upstream Slack Web API builds `A request error occurred: ${original.message}`. I did not run a live Slack outage reproduction in this read-only review.

## Solution Assessment

No, not as a combined patch. The empty-wrapper classifier is the right narrow fix, but the global nullish unhandled-rejection branch should be removed or scoped because current main already handles the Slack undefined Socket Mode path inside the Slack reconnect code.
