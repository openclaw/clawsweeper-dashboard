## Summary

Keep this open. Current main still lacks a Dreams-tab agent selector, and the Dreams UI/controller plus doctor.memory gateway handlers remain scoped to the default agent instead of a selected agent, so the reported multi-agent inspection workflow is still missing.

## What This Changes

The issue requests a Control UI Dreams/Dreaming tab agent selector and matching backend agent scoping so users can inspect and act on each agent's own dreaming status and diary.

## Best Possible Solution

Add a Dreams-tab selected-agent state backed by agents.list, show the selected agent in the Dreams header, pass the selected agentId through all Dreams status/diary/action requests, and teach the doctor.memory.* handlers to accept an optional agentId with default-agent fallback and validation against configured agents.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: open the Dreams tab path and inspect the render/controller/gateway flow, which has no selector and always calls doctor.memory.* without an agentId before the gateway resolves the default agent.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse existing Control UI agent-list/selection patterns and add optional agentId scoping to the existing doctor.memory.* contract; broader per-agent dreaming config and corpus isolation should stay in their own related issues.
