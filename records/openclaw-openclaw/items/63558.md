## Summary

Keep this open. Current main still has no Dreams-tab agent selector, the Dreaming UI/controller does not carry selected-agent state, and the existing doctor.memory.* handlers still resolve the default agent rather than a user-selected agent workspace.

## What This Changes

The issue requests a Control UI Dreams-tab agent selector so users can inspect and act on Dreaming status, diary, and related controls for different configured agents.

## Best Possible Solution

Add a Dreams-tab selected-agent state backed by agents.list, show a compact current-agent selector in the Dreams header, pass the selected agentId through Dreaming status/diary/action requests, and update doctor.memory.* handlers to validate that id against configured agents before resolving the workspace, with default-agent fallback for compatibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: inspect the Dreams tab render/controller/gateway flow and it has no selector while all doctor.memory.* calls are unscoped and resolve the default agent.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse existing Control UI agent-list selection patterns and add validated optional agent scoping to the existing doctor.memory.* contract, without folding in broader per-agent Dreaming configuration or memory-corpus redesign.
