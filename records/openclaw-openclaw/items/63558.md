## Summary

Keep this open. Current main still lacks a Dreams-tab agent selector, and the Dreaming UI/controller/gateway path cannot request or mutate a selected agent workspace.

## What This Changes

Add a Control UI Dreaming tab agent selector so users can inspect and act on Dreaming artifacts for different agent contexts.

## Best Possible Solution

Add Dreams-tab selected-agent state backed by agents.list, pass a validated optional agentId through doctor.memory.* status/diary/action requests, and preserve default-agent fallback for compatibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the Dreams tab has no selector, the controller sends empty doctor.memory.* params, and gateway diary/action handlers resolve the default agent.

## Solution Assessment

Yes. The narrow maintainable path is to reuse the existing Control UI agent selector pattern and add validated optional agent scoping to the existing doctor.memory.* contract without broadening into #63829 memory-wiki vault isolation.
