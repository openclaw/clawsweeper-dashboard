## Summary

Keep open: current main still scopes doctor service handling to gateway-like `.service` units and suppresses inactive non-legacy Linux units, so the enabled-but-inactive `.path`/`.timer` user-unit problem remains. The request is source-reproducible and useful, but it needs maintainer lifecycle policy before implementation because `doctor --fix` would start non-gateway systemd user units.

## What This Changes

Add an `openclaw doctor` check and repair path that detects OpenClaw-owned enabled-but-inactive `systemd --user` units beyond the gateway and starts them under explicit repair consent.

## Best Possible Solution

Keep this open for a maintainer-sponsored design that reports owned enabled inactive user units read-only by default, emits structured lint findings, and starts only explicitly approved safe unit types under repair consent.

## Reproduction Assessment

Yes for source reproduction: current doctor service detection scans gateway-like `.service` files and filters inactive non-legacy units, while no inspected path enumerates enabled `.path` or `.timer` user units through `list-unit-files`/`FragmentPath`. I did not run a live multi-agent `systemd --user` host reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the lifecycle policy. Doctor is a plausible surface, but the best fix must define OpenClaw-owned unit discovery, safe unit classes, explicit repair consent, and preview/lint behavior before starting non-gateway units.
