## Summary

Keep open: the PR fixes the image-level Dockerfile grace period, but it leaves the bundled Compose healthcheck on a separate 20s start period, so the bundled Compose path can still report unhealthy during legitimate cold starts.

## What This Changes

The PR changes the root Dockerfile healthcheck start period from 15s to 600s and adds an Unreleased changelog entry for the Docker cold-start healthcheck fix.

## Best Possible Solution

Land one narrow Docker operator fix that applies the same cold-start grace period to both the image HEALTHCHECK and the bundled Compose healthcheck, preserves the existing probe command and other healthcheck knobs unless intentionally changed, and adds static coverage to prevent drift.

## Reproduction Assessment

Yes. The linked issue provides a concrete docker run plus docker ps reproduction against v2026.4.29, and the current-main static check shows the Dockerfile still at 15s while Compose remains at 20s.

## Solution Assessment

No. The Dockerfile change is directionally right, but Docker Compose can override image healthcheck values, so the complete maintainable fix should update the bundled Compose healthcheck and add a focused sync guard. ([docs.docker.com](https://docs.docker.com/reference/compose-file/services/))
