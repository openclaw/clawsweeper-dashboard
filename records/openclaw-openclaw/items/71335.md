## Summary

Keep this open. Current main still defaults an unset memory sync watcher to enabled, and normal gateway-side memory_search use can still create long-lived builtin or QMD chokidar watchers unless the user explicitly sets watch false.

## What This Changes

Request defaulting memorySearch sync.watch to false for headless gateway deployments while preserving explicit user overrides and the existing desktop/CLI default behavior.

## Best Possible Solution

Add a runtime-aware memory sync default policy so unset watch resolves false only for the intended headless long-lived gateway manager context, while explicit true/false config remains authoritative and CLI/status/desktop freshness paths stay covered by focused tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: unset sync.watch resolves to true, memory_search opens the full manager path, and both builtin and QMD full managers start chokidar watchers when watch is true.

## Solution Assessment

Yes, with a boundary adjustment. The best fix is not a raw gateway.mode check; it should pass an explicit headless/gateway default-policy context into memory sync resolution or manager creation so interactive behavior remains unchanged.
