## Label Justifications

- `P2`: This is a normal-priority gateway reliability/default-policy issue with a known workaround and partial mitigations, not an emergency for all installs.
- `impact:crash-loop`: The reported FD pressure can push long-running gateway processes toward file descriptor exhaustion and availability failures.
- `impact:session-state`: The affected watcher controls memory-index freshness and retained memory-search state across gateway sessions.

## Summary

Keep open. Current main and the latest release still default an unset memory watcher to enabled, while the merged warning mitigation and maintainer discussion explicitly leave the gateway default-off policy decision unresolved.

## What This Changes

Request to make an unset `memorySearch.sync.watch` resolve to `false` for headless gateway mode while preserving explicit overrides and CLI/desktop default-on behavior.

## Best Possible Solution

Maintainers should choose the gateway memory freshness policy, then either implement and document a gateway-only default-off path with fresh-install and upgrade coverage or explicitly keep default-on with warning/adaptive-watcher mitigations.

## Reproduction Assessment

Yes, at source level. Current `main` and `v2026.6.8` both resolve an unset `memorySearch.sync.watch` to `true`, and the runtime starts watchers whenever that resolved value is true.

## Solution Assessment

Unclear. Gateway default-off is a plausible reliability improvement, but the merged mitigation and maintainer comment show it is a product tradeoff against live memory freshness for smaller gateway deployments.
