## Summary

Keep this open. Current main still resolves an unset memory sync watcher to true, and the normal gateway memory tool path can still create long-lived builtin or QMD chokidar watchers; the related #64549 freshness bug is adjacent and already closed, not a replacement for this gateway default-policy change.

## What This Changes

The issue requests that unset `memorySearch.sync.watch` default to false for headless gateway mode while explicit config overrides and CLI/desktop defaults keep their current behavior.

## Best Possible Solution

Add a runtime-aware memory sync default policy so unset `sync.watch` resolves to false only for the intended headless gateway manager context, while explicit true/false config stays authoritative and existing CLI/desktop freshness behavior remains covered by tests and docs.

## Reproduction Assessment

Yes. With `sync.watch` unset, the resolver returns `watch: true`; a normal gateway memory_search call enters the full manager path, where builtin and QMD managers create chokidar watchers unless the resolved value is false.

## Solution Assessment

Unclear as originally phrased. The policy is reasonable, but the safest implementation should pass an explicit headless/default-policy context into memory sync resolution or manager creation instead of keying only on raw `gateway.mode`.
