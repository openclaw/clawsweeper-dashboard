## Label Justifications

- `P2`: This is a normal-priority gateway reliability/default-policy issue with a known workaround and partial mitigations, not an emergency for all installs.
- `impact:session-state`: The affected watcher controls memory-index freshness and retained memory-search state across gateway sessions.
- `impact:crash-loop`: The report concerns file-descriptor pressure in long-running gateways that can degrade availability or lead to runtime failure.

## Summary

Keep open. Current `main` and the latest release still resolve an unset memory watcher to enabled, while the related merged mitigation and maintainer discussion explicitly preserve default-on behavior and leave the gateway default-off policy decision unresolved.

## What This Changes

The issue requests making `memorySearch.sync.watch` resolve to `false` by default in non-CLI gateway mode while preserving explicit overrides and CLI/desktop freshness behavior.

## Best Possible Solution

Maintainers should choose the gateway memory freshness policy, then either implement a documented gateway-only default-off path with fresh-install and upgrade coverage, or close this as intentionally default-on in favor of warnings, lower-FD watcher backends, and explicit opt-out.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.6` both resolve an unset `memorySearch.sync.watch` to `true`, and the runtime creates watchers whenever that resolved value is true.

## Solution Assessment

Unclear. Gateway default-off is a plausible reliability improvement, but the merged mitigation and maintainer comment show it is a product tradeoff against live memory freshness for smaller gateway deployments.
