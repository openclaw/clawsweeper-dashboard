## Summary

Keep this open. Current main still resolves an unset memory sync watcher to true, and normal gateway memory_search calls still use long-lived builtin/QMD manager paths that can start chokidar watchers unless the resolved config is explicitly false.

## What This Changes

The issue requests that an unset `memorySearch.sync.watch` default resolve to `false` for headless gateway mode while preserving explicit overrides and existing CLI/desktop defaults.

## Best Possible Solution

Add an explicit runtime-aware memory sync default policy so an unset `watch` resolves to false only for intended headless gateway long-lived manager contexts, while explicit true/false remains authoritative and CLI/status/interactive freshness paths stay covered by tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: leave `memorySearch.sync.watch` unset, trigger gateway `memory_search`, and the resolver returns `watch: true` before the full builtin or QMD manager can create chokidar watchers.

## Solution Assessment

Yes, with one implementation constraint. The narrow maintainable fix is to pass an explicit gateway/headless default-policy context into memory sync resolution or manager creation, not to key blindly on raw `gateway.mode` in a way that could change desktop/local interactive behavior.
