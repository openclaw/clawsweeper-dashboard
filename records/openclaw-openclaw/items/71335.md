- `impact:crash-loop`: The report is about file-descriptor pressure in long-running gateways that can degrade process availability or lead to spawn failures.

## Summary

Keep open. Current main still resolves unset `memorySearch.sync.watch` to `true`, and the gateway memory-search path can still create long-lived built-in or QMD watchers; related merged work reduced watcher FD pressure but did not implement the requested gateway-mode default policy.

## What This Changes

Request defaulting unset `agents.defaults.memorySearch.sync.watch` to `false` in headless gateway mode while preserving explicit user overrides and CLI/desktop behavior.

## Best Possible Solution

Define a runtime-aware memory sync policy where unset watch is false only for approved long-lived headless gateway contexts, explicit true/false still wins, and tests/docs cover CLI, desktop gateway, built-in, and QMD behavior.

## Reproduction Assessment

Yes for the requested default-behavior path: source inspection shows unset `memorySearch.sync.watch` still resolves to true and full memory managers still arm watchers. I did not live-reproduce the original FD count after the related FD-pressure mitigation.

## Solution Assessment

Unclear as an automated fix. Defaulting watch off in gateway mode is plausible, but maintainers should first define the headless-versus-desktop gateway boundary and account for the merged watcher FD-pressure mitigation.
