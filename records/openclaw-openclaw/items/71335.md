## Summary

Keep open. Current main still resolves an unset memory watcher to enabled and gateway memory search still uses long-lived builtin/QMD manager paths that can arm chokidar; the merged ENOSPC work only changes crash handling, and the open large-extraPaths report is related but does not supersede this gateway-default policy request.

## What This Changes

Request to make unset `memorySearch.sync.watch` resolve to `false` in gateway/headless mode while preserving explicit overrides and desktop/CLI watch defaults.

## Best Possible Solution

Define a runtime-aware memory sync policy where unset watch is false only for approved long-lived headless gateway contexts, explicit true/false still wins, and CLI/status/desktop freshness remains covered by tests and docs.

## Reproduction Assessment

Yes for the default-behavior path. Source inspection on current main shows unset `memorySearch.sync.watch` resolves to true, gateway `memory_search` uses the default manager purpose, and full builtin/QMD managers can arm chokidar watchers; I did not live-reproduce the exact FD count.

## Solution Assessment

Unclear as an automated fix. Defaulting watch off in gateway mode is plausible, but maintainers should first define the exact gateway/headless contexts and coordinate with the open large-extraPaths watcher issue.
