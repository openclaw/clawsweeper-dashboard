## Summary

Keep open. Current main still resolves an unset `memorySearch.sync.watch` to enabled and the gateway memory-search path still opens long-lived builtin/QMD watchers; the merged FD-pressure fix removes chokidar write polling but does not implement the requested gateway-mode default policy.

## What This Changes

Request: make `memorySearch.sync.watch` default to `false` in gateway/headless mode while preserving explicit user overrides and desktop/CLI freshness behavior.

## Best Possible Solution

Define a runtime-aware memory sync policy where unset watch is false only for approved long-lived headless gateway contexts, explicit true/false still wins, and tests/docs cover CLI, desktop gateway, builtin, and QMD behavior.

## Reproduction Assessment

Yes for the default-behavior path: source inspection shows unset `memorySearch.sync.watch` still resolves to true and gateway `memory_search` still opens non-transient watcher-capable managers. I did not live-reproduce the original FD count after the merged write-polling mitigation.

## Solution Assessment

Unclear as an automated fix. Defaulting watch off in gateway mode is plausible, but maintainers should first define which gateway contexts are headless versus desktop-like and account for the already-merged FD-pressure mitigation in https://github.com/openclaw/openclaw/pull/81802.
