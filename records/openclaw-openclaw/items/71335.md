## Label Justifications

- `P2`: This is a normal-priority runtime default improvement with a real gateway resource-pressure report, an existing workaround, and a partial shipped mitigation.
- `impact:crash-loop`: The report concerns file watcher descriptor pressure on multi-agent gateways, which can degrade process availability when descriptors are exhausted.
- `impact:session-state`: The requested policy changes memory-search file watch behavior, which controls freshness of indexed memory state.

## Summary

Keep open. Current main and the latest shipped release still default an unset memory watcher to enabled, while the related FD-pressure fix only removed chokidar write polling and did not implement the requested gateway/headless default policy.

## What This Changes

The issue requests making `memorySearch.sync.watch` default to `false` for gateway/headless operation while preserving explicit user overrides and CLI/desktop watch behavior.

## Best Possible Solution

Define an explicit runtime-aware memory sync policy for long-lived headless gateway managers only: unset watch becomes false there, explicit true/false still wins, and builtin plus QMD tests/docs cover CLI, status, desktop-like local gateway, and headless gateway behavior.

## Reproduction Assessment

Yes for the source-level default behavior: current main resolves unset `memorySearch.sync.watch` to true and gateway `memory_search` opens default/full managers that create watchers. I did not establish a fresh live FD-count reproduction after the shipped https://github.com/openclaw/openclaw/pull/81802 mitigation.

## Solution Assessment

Unclear as an automated fix. Defaulting watch off for gateway mode is plausible, but maintainers need to define the headless versus desktop-like gateway boundary and the compatibility story before changing a runtime default.
