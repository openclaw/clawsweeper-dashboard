## Summary

Keep this open. Current main still resolves an unset memory sync watcher to enabled, and normal gateway `memory_search` calls still use full long-lived memory managers that can create chokidar watchers unless the user explicitly sets `sync.watch: false`.

## What This Changes

Request defaulting `agents.defaults.memorySearch.sync.watch`/agent `memorySearch.sync.watch` to false only for headless gateway mode while preserving explicit overrides and keeping CLI/desktop defaults unchanged.

## Best Possible Solution

Add a runtime-aware memory sync default policy so an unset `watch` resolves to false only for intended headless gateway long-lived manager contexts, while explicit true/false remains authoritative and CLI/status/interactive freshness paths stay covered by tests and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: leave `memorySearch.sync.watch` unset, trigger gateway `memory_search`, and the resolver returns `watch: true` before the full builtin or QMD manager can create chokidar watchers.

## Solution Assessment

Yes, with one implementation constraint. The narrow maintainable fix is to pass an explicit gateway/headless default-policy context into memory sync resolution or manager creation, not to key blindly on raw config in a way that changes desktop or local interactive behavior.

Add focused tests for unset gateway/headless default false, explicit true winning, explicit false winning, and CLI/status behavior staying unchanged.
