## Summary

Keep this open. Current main still resolves an unset memory sync watcher to enabled, and normal gateway `memory_search` tool calls still open long-lived memory managers that can create chokidar watchers unless the user explicitly configures `sync.watch: false`.

## What This Changes

Default `memorySearch.sync.watch` to false for headless gateway mode while preserving explicit overrides and existing CLI/desktop watch behavior.

## Best Possible Solution

Add a runtime-aware memory sync default policy so an unset watcher resolves to false only for intended headless gateway long-lived manager contexts, while explicit true/false remains authoritative and CLI/status/interactive freshness paths stay covered by tests, docs, and changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: leave `memorySearch.sync.watch` unset, trigger gateway `memory_search`, and the resolver returns `watch: true` before the full builtin or QMD manager can create chokidar watchers.

## Solution Assessment

Yes, with one implementation constraint. The maintainable fix is to pass an explicit gateway/headless default-policy context into sync resolution or manager creation, not to key blindly on raw config in a way that changes desktop or one-shot CLI behavior.
