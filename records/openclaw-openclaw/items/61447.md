## Summary

Keep open. Current main still exposes only compaction metadata to the public `after_compaction` plugin hook, while the generated summary is available in nearby compaction plumbing; the remaining work is a product/security decision about opt-in access to transcript-derived content.

## What This Changes

Request to add generated compaction summary content and related token metadata to the public `after_compaction` plugin hook payload.

## Best Possible Solution

Define a permissioned, opt-in `after_compaction` summary-content contract, populate it only when a runtime produced real summary text, and document unavailable, truncation, and token-count semantics with focused hook tests.

## Reproduction Assessment

Not applicable as a runtime bug, but yes as a source-level contract gap: current main generates or carries compaction summaries while the public `after_compaction` payload still exposes only metadata.

## Solution Assessment

Partially: adding an optional summary field is the right direction, but the best solution needs an explicit opt-in or permission contract plus documented absence and bounds before exposing transcript-derived content.

## Risks / Open Questions

- Compaction summaries can contain transcript-derived private content, tool output, credentials, or operational state, so exposing them to plugins needs explicit opt-in, permission, audit, size, and redaction semantics.
- Not every compaction path appears to have real summary text available, especially Codex/native or context-engine-owned paths, so the API must document `undefined` or unavailable-summary behavior.
- A `summaryTokens` field should not be promised unless the implementation can compute a trustworthy token count for the actual generated summary across supported runtimes.
