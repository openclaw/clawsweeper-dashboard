## Summary

Keep open. Current main still omits generated compaction summary text from the public `after_compaction` plugin hook payload while direct and queued compaction paths keep the generated summary nearby, and I did not find a canonical duplicate or merged fix for this specific hook payload gap.

## What This Changes

The issue requests exposing generated compaction summary text, and optionally a summary token metric, on the public `after_compaction` plugin hook payload.

## Best Possible Solution

Add a documented optional generated-summary field to the public `after_compaction` hook contract and populate it only from compaction paths that already have a real summary string, with focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows generated compaction summaries remain on `result.summary` or `result.result.summary`, while the typed public hook event and emitters pass only counts and session file metadata.

## Solution Assessment

Yes. A backward-compatible optional hook field is the narrow maintainable solution; parsing inline `<summary>` tags or changing transcript persistence would broaden the feature and keep plugin consumers on a fragile workaround.
