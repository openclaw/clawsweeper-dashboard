- `impact:session-state`: The requested payload exposes compaction output used for session memory, transcript continuity, and recovery workflows.

## Summary

Keep open. Current main and the latest stable release still expose only compaction metadata through the public `after_compaction` plugin hook, while the requested summary-content payload remains a real SDK feature with privacy and opt-in semantics that need maintainer approval.

## What This Changes

The issue requests adding generated compaction summary text, and possibly summary token metadata, to the public `after_compaction` plugin hook payload.

## Best Possible Solution

Define a permissioned `summary?: string` contract for `after_compaction`, populate it only when a real generated summary is available and the plugin has opted into summary-content access, and document absence, size, and token-count semantics.

## Reproduction Assessment

Not applicable as a runtime bug, but yes at source level: current main has generated summary text in compaction paths while the public `after_compaction` payload and tests remain metadata-only.

## Solution Assessment

Partially: an optional summary field is the right SDK direction, but the best fix needs an explicit opt-in or permission contract and documented unavailable-summary behavior before exposing transcript-derived content.

## Risks / Open Questions

- Compaction summaries can contain transcript-derived private or tool-derived content, so exposing them to plugins needs explicit opt-in, permission, and audit semantics.
- Some compaction paths, especially harness-owned or context-engine-owned paths, may not have generated summary text available, so the API must document absence semantics instead of synthesizing empty strings.
- `summaryTokens` should not be promised unless the implementation can compute a trustworthy token count for the generated summary across supported runtimes.
