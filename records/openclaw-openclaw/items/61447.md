## Summary

Keep open. Current main still omits generated compaction summary text from the typed `after_compaction` plugin hook payload, even though direct, queued, runner-owned, and subscribed compaction paths already have or receive summary-bearing result objects before emitting the hook.

## What This Changes

The issue requests a backward-compatible `after_compaction` plugin hook payload field for generated compaction summary text and optional summary token metadata.

## Best Possible Solution

Add an optional public `summaryText` field, and only add `summaryTokens` if a reliable existing token estimate is available, then populate it from current summary-bearing compaction results and document unsupported native paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: inspect the current hook event type and hook emitters, then compare them with nearby compaction result objects that already carry `summary`.

## Solution Assessment

Yes. Extending the optional typed plugin hook payload is the narrow maintainable fix; parsing inline `<summary>` tags or session JSONL would keep the fragile workaround as the contract.
