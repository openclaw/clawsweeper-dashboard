## Summary

Keep open: the branch targets a real WebChat wrong-recipient normalization path, but the current prefix heuristic is too broad and the implementation is stale relative to current main's route-helper normalization.

## What This Changes

The PR changes delivery context normalization to drop unstructured WebChat recipients while preserving colon-prefixed targets and adds regression coverage for a valid WebChat target plus a phone-number-shaped target.

## Best Possible Solution

Update the fix on current main so WebChat contexts preserve only WebChat-owned direct targets and clear stale unstructured or non-WebChat structured recipients without changing other channel route semantics.

## Reproduction Assessment

Yes for the PR defect: with the proposed diff, `normalizeDeliveryContext({ channel: "webchat", to: "telegram:123" })` keeps the recipient because the colon-prefix heuristic passes. Current-main `extractDeliveryInfo` shows that any preserved `channel + to` context remains routable downstream.

## Solution Assessment

No. The current approach distinguishes structured from unstructured targets, but the maintainable boundary is WebChat-owned versus cross-channel targets, ideally enforced in the current `normalizeChannelRouteTarget` path or immediately after it.
