## Summary

Keep open: the underlying recipient-routing bug is plausible and source-reproducible, but the current PR still preserves cross-channel structured recipients in WebChat contexts and lacks required real behavior proof.

## What This Changes

The PR changes delivery-context normalization so WebChat drops unstructured `to` values while preserving colon-prefixed targets, and adds normalization tests for WebChat recipients.

## Best Possible Solution

Normalize WebChat delivery so only WebChat-owned direct targets survive, clear unstructured and cross-channel recipients before downstream fallback treats them as routable, and cover both normalization and extraction paths with proof.

## Reproduction Assessment

Yes. From source inspection, this PR preserves `normalizeDeliveryContext({ channel: "webchat", to: "telegram:123" })` as routable, and current `extractDeliveryInfo` treats any `channel` plus `to` as a deliverable context; I did not run tests because this review is read-only.

## Solution Assessment

No. Preserving any colon-prefixed target is too broad; the maintainable fix should preserve only WebChat-owned targets such as `webchat:...` and clear non-WebChat structured targets.
