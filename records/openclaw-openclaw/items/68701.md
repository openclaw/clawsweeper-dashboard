## Summary

Keep open. The PR targets a real WebChat recipient-contamination bug, but the current diff still preserves any colon-prefixed non-WebChat recipient, so stale targets like `telegram:...` remain routable; the prior ClawSweeper discussion raised the same blocker and there is still no real behavior proof.

## What This Changes

The PR changes `normalizeDeliveryContext` to drop unstructured WebChat `to` values while preserving structured targets, plus tests for `webchat:user-123` and a phone-shaped recipient.

## Best Possible Solution

Normalize WebChat delivery so only WebChat-owned direct targets survive, clear unstructured and non-WebChat structured recipients, and add regression coverage against the current route-helper path.

## Reproduction Assessment

Yes, from source inspection: the PR predicate keeps `normalizeDeliveryContext({ channel: "webchat", to: "telegram:123" })` routable, and current `extractDeliveryInfo` treats any `channel` plus `to` as routable. I did not run tests because this review kept the checkout read-only.

## Solution Assessment

No. Structured versus unstructured is not the right boundary; the narrow maintainable fix is WebChat-owned versus cross-channel targets, implemented against the current route-normalization helper.
