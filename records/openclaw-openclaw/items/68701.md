## Summary

Keep open: the PR targets a real recipient-contamination path, but the branch preserves any colon-prefixed WebChat `to`, so stale `telegram:` or `channel:` recipients can still remain routable. It also lacks after-fix real behavior proof from a real setup.

## What This Changes

The branch adds WebChat-specific recipient stripping in `normalizeDeliveryContext` and tests preserving `webchat:user-123` while dropping a phone-shaped `to`.

## Best Possible Solution

Constrain WebChat delivery normalization to preserve only WebChat-owned targets and clear stale unstructured or non-WebChat structured recipients without changing other channel route semantics.

## Reproduction Assessment

Yes for the PR defect from source inspection: the proposed predicate keeps `normalizeDeliveryContext({ channel: "webchat", to: "telegram:123" })` routable, and current `extractDeliveryInfo` treats any `channel` plus `to` as routable. I did not run tests because this review kept the checkout read-only.

## Solution Assessment

No. The branch distinguishes structured from unstructured targets, but the maintainable boundary is WebChat-owned versus cross-channel targets, ideally enforced in the current route-helper path or immediately after it.
