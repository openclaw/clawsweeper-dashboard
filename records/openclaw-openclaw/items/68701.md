## Summary

Keep this PR open. The underlying WebChat delivery-context problem is still valid on current main, but the proposed guard is incomplete because it preserves any colon-prefixed target, including stale non-WebChat routes that downstream delivery code still treats as routable.

## What This Changes

The PR changes `normalizeDeliveryContext` to drop unstructured `to` values for WebChat contexts while preserving colon-prefixed targets, and adds regression coverage for a WebChat target plus a phone-number-shaped stale target.

## Best Possible Solution

Update the current shared delivery-context normalization path so WebChat contexts preserve only WebChat-owned direct targets and strip both unstructured stale values and structured non-WebChat targets. Keep other channels’ route normalization unchanged, add focused regression coverage for the structured cross-channel case, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes for the PR defect: under the proposed guard, `normalizeDeliveryContext({ channel: "webchat", to: "telegram:123" })` still keeps `to` because the value has a generic structured prefix. Current main’s `extractDeliveryInfo` then shows that any preserved `channel + to` context remains routable downstream.

## Solution Assessment

No. The proposed direction is close, but checking for any structured prefix is too broad; the safer fix should validate the target as WebChat-owned, or clear `to` for WebChat contexts when no supported WebChat direct-target contract applies.
