## Summary

Keep open: current main still excludes accepted Discord guild messages from the early typing cue, the queue test asserts that suppression, and the docs/queue contract still describe immediate typing while work is queued. An active implementation PR appears to cover the same Discord typing lifecycle, so this should stay open until that path lands or is replaced.

## What This Changes

The issue requests immediate Discord guild/channel typing feedback when `agents.defaults.typingMode` is `instant`, without hidden debounce delays.

## Best Possible Solution

Keep this issue open until Discord accepted guild/channel turns start a single best-effort typing feedback path promptly when policy allows it, likely by landing or narrowing https://github.com/openclaw/openclaw/pull/76091.

## Reproduction Assessment

Yes, from source inspection: accepted guild messages reach the early typing cue call, but the helper rejects guild contexts before checking `typingMode`, and the current queue test asserts no early typing is sent. I did not perform a live Discord guild timing run.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the Discord typing lifecycle at the accepted-message/queued-processing boundary rather than changing core debounce or adding config; the active PR is the current review path for that approach.
