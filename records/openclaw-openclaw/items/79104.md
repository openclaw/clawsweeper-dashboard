## Summary

Keep open: current main still suppresses the accepted Discord guild-channel early typing cue even when `typingMode: "instant"`, while the docs describe group typing as following that setting. The related Discord typing lifecycle PR is open and appears to be the active fix path, so this issue should stay as the canonical tracker until that work lands or is replaced.

## What This Changes

The issue asks OpenClaw to show Discord guild-channel typing promptly after an accepted message when `agents.defaults.typingMode` is `"instant"`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/76091 with a focused Discord plugin fix that starts accepted guild typing feedback for `typingMode: "instant"` while preserving rejected, aborted, empty, and opt-out cases.

## Reproduction Assessment

Yes from source inspection: an accepted guild preflight context reaches `queueAcceptedDiscordTypingCue`, but the helper rejects guild contexts before checking `typingMode`, and the current queue test asserts no early typing call. I did not run a live Discord guild reproduction in this review.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the Discord accepted-typing lifecycle, preferably through the existing related PR or a smaller replacement, rather than adding new configuration or changing core typing policy.
