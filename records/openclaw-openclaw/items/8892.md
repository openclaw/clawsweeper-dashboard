## Summary

Keep open: current main still lacks a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, cwd workspace inference, remembered sessions, and post-launch `/agent` switching are useful adjacent paths, but they do not satisfy selecting a non-default agent from an arbitrary directory at startup.

## What This Changes

Request a new `openclaw tui --agent <id>` flag that starts the TUI against a specific configured agent's model, workspace, and context instead of defaulting to `main`.

## Best Possible Solution

Add a typed `openclaw tui --agent <id>` startup selector with validation, documented precedence, launch/relaunch plumbing, focused TUI coverage, and CLI/TUI docs updates if maintainers accept the UX.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence the requested surface is absent: there is no TUI CLI flag, no TUI option field, and startup agent resolution only uses session key, cwd, or the default agent.

## Solution Assessment

Yes, a first-class startup selector is the narrow direct solution if maintainers accept the UX and precedence contract. Existing cwd inference, canonical session keys, remembered sessions, and post-launch `/agent` switching remain useful but do not satisfy arbitrary-directory startup selection with `--agent <id>`.
