## Summary

Keep this open: current main still lacks a native `openclaw tui --agent <id>` startup selector, and the existing workarounds only cover agent-scoped session keys, workspace inference, or post-launch `/agent` switching.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` startup option so the TUI opens directly against a configured non-default agent.

## Best Possible Solution

Add a typed TUI startup agent selector with validation and documented precedence, then cover it in focused TUI tests and the public TUI CLI/guide docs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the missing capability: current main registers no `--agent` flag and has no TUI option field to carry it. I did not run the CLI because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, a native typed `--agent <id>` startup selector is the cleanest fit if maintainers accept the behavior. It should not be modeled as the invalid shorthand `--session agent:<id>`; it needs explicit precedence and validation.
