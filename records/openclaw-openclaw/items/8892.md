## Summary

Keep open: current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, workspace inference, and post-launch `/agent <id>` are useful workarounds, but they do not satisfy the requested generic-directory startup flag and no open PR now owns this exact change.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` startup selector so the TUI can bind to a non-default configured agent.

## Best Possible Solution

Ship a focused additive TUI CLI design that validates `--agent <id>`, threads a typed startup agent id through TUI startup, preserves existing session/workspace behavior, and updates focused tests plus public TUI docs after precedence is decided.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` has no `--agent` option, the TUI option type cannot carry one, and startup resolution only considers a canonical agent-scoped session key, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, if maintainers confirm the public CLI direction. A native startup selector is the maintainable user-facing surface for arbitrary directories, but it should be additive, validated, and explicit about precedence rather than treating two-part `agent:<id>` shorthand as a hidden alias.
