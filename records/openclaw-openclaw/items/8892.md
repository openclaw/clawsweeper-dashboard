## Summary

Keep open: current main still lacks a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, workspace inference, remembered-session restore, and post-launch `/agent` switching are useful adjacent paths, but they do not satisfy selecting a non-default agent from an arbitrary directory at launch.

## What This Changes

Request a startup `--agent <id>` flag for `openclaw tui` so the TUI can start directly against a configured non-default agent and its workspace/model context.

## Best Possible Solution

Add a typed `openclaw tui --agent <id>` startup selector with validation, documented precedence, focused TUI/CLI coverage, and TUI docs updates.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence the requested surface is absent: no CLI flag, no TUI option field, and startup agent resolution only from session key, cwd workspace inference, or the default agent.

## Solution Assessment

Yes, if maintainers accept the UX, a first-class startup selector is the narrow direct solution. Existing cwd inference, canonical session keys, remembered sessions, and `/agent` are useful but do not satisfy arbitrary-directory startup selection with `--agent <id>`.
