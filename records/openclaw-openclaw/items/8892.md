## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, but it is not an emergency regression.
- `impact:session-state`: The selected startup agent determines the TUI session owner and workspace context.
- `impact:auth-provider`: The selected startup agent also determines model/provider routing and can otherwise route the user to the wrong configured model.

## Summary

Keep open. Current main and the latest shipped tag still lack a native `openclaw tui --agent <id>` startup selector; workspace inference, canonical `agent:<id>:...` session keys, and post-launch `/agent` switching are useful but do not satisfy selecting a configured agent from an arbitrary directory at launch.

## What This Changes

Add an `openclaw tui --agent <id>` startup flag so a configured non-default agent can own the TUI session, model/provider routing, and workspace context from launch.

## Best Possible Solution

Add a typed `--agent <id>` path through the TUI CLI, option type, and startup resolver, validate the id against configured agents, and document/test the intended precedence with existing session and workspace behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` has no `--agent` option, `TuiOptions` cannot carry one, and startup resolution only uses a canonical session key, CWD workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native startup selector is the narrowest maintainable way to select a non-default configured agent from arbitrary directories; current workspace inference and post-launch `/agent` switching do not cover that launch case.
