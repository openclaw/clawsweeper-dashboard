## Label Justifications

- `P2`: This is a normal-priority TUI ergonomics and multi-agent routing improvement with limited blast radius, not a crash or urgent regression.
- `impact:session-state`: The requested selector changes which agent session and workspace context a TUI startup attaches to.
- `impact:auth-provider`: Selecting a different startup agent can change the model/provider/auth-profile routing used for the session.

## Summary

Keep open: current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector, while the existing agent-scoped session key, workspace inference, and post-launch `/agent` paths do not cover startup from arbitrary directories.

## What This Changes

Add a native `openclaw tui --agent <id>` flag so a TUI session can start on a configured non-default agent with that agent's model and workspace context.

## Best Possible Solution

Ship a focused TUI CLI change that validates `--agent <id>`, threads a typed startup agent id through `TuiOptions`/`runTui`, defines precedence with session and workspace inference, and updates focused TUI tests plus the TUI docs.

## Reproduction Assessment

Yes. Source inspection against current main shows the TUI command has no `--agent` option, the TUI options type cannot carry one, and startup resolution only considers an agent-scoped session key, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, if maintainers confirm the public CLI direction. A native startup selector is the maintainable surface for arbitrary directories, but it should reuse existing agent/session helpers and avoid treating the two-part `agent:<id>` shorthand as a silent compatibility alias.
