## Label Justifications

- `P2`: This is a clear TUI ergonomics feature with bounded blast radius, not an urgent regression or runtime outage.
- `impact:session-state`: The requested startup agent controls the TUI session owner, workspace context, remembered session scope, and loaded persona/context.
- `impact:auth-provider`: Selecting a configured agent at startup can change model/provider routing and auth-backed provider configuration for the session.

## Summary

Keep open. Current main and v2026.5.22 still lack a native `openclaw tui --agent <id>` startup selector; agent-scoped session keys, cwd workspace inference, and post-launch `/agent` do not cover the requested arbitrary-directory startup flow.

## What This Changes

Request a TUI startup flag that selects a configured non-default agent for the session, model/provider routing, workspace context, and agent persona files.

## Best Possible Solution

Add a typed `--agent <id>` selector through the TUI CLI, option contract, startup resolver, validation, focused tests, and TUI docs, after maintainers choose the precedence rules.

## Reproduction Assessment

Yes, source-level. Current main has no TUI `--agent` option, no startup-agent field in `TuiOptions`, and startup resolution is limited to canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; current workarounds do not cover that launch case.
