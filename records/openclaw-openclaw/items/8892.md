## Label Justifications

- `P2`: This is a normal-priority TUI ergonomics feature with limited blast radius but real multi-agent usability impact.
- `impact:session-state`: The requested startup selector controls which agent workspace and session context the TUI starts in.
- `impact:auth-provider`: Selecting the wrong agent can route the TUI through the wrong configured model/provider/auth profile.

## Summary

Keep open: current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, cwd workspace inference, and post-launch `/agent` switching are useful adjacent paths, but they do not satisfy the requested generic-directory startup flag.

## What This Changes

The issue requests a native TUI startup flag that selects which configured agent handles the session before the terminal UI connects or sends the initial message.

## Best Possible Solution

Add a native `--agent <id>` selector for `openclaw tui`, validate it against configured agents, define precedence against canonical session keys and cwd inference, and update focused TUI tests plus the public TUI docs.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--agent` option in `registerTuiCli`, no startup agent field in `TuiOptions`, and an initial-agent resolver that only uses canonical session keys, cwd inference, then the default agent.

## Solution Assessment

Unclear until maintainers choose the startup precedence. A native flag is a clean, bounded solution if it validates configured agent IDs and preserves the existing session-key and workspace-inference contracts.
