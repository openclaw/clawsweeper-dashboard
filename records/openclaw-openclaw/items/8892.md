## Label Justifications

- `P2`: This is a normal-priority TUI ergonomics and multi-agent routing feature with limited blast radius, not an urgent regression or crash.
- `impact:session-state`: The requested selector changes which agent session and workspace context the TUI attaches to at startup.
- `impact:auth-provider`: Selecting a different startup agent can change the model, provider, and auth-profile routing used for the session.

## Summary

Keep open: current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, workspace-directory inference, and post-launch `/agent <id>` are useful partial paths, but they do not cover selecting a configured agent from an arbitrary directory at TUI startup.

## What This Changes

Request a new `openclaw tui --agent <id>` flag that starts the TUI against a selected configured agent, including that agent's model, workspace, and context.

## Best Possible Solution

Ship a focused TUI CLI change that validates `--agent <id>`, threads a typed startup agent id through TUI startup, defines precedence with session/workspace/remembered-session behavior, and updates focused tests plus TUI docs.

## Reproduction Assessment

Yes. Source inspection against current main and `v2026.6.8` shows the TUI command has no `--agent` option, the TUI options type cannot carry one, and startup resolution only considers a canonical agent-scoped session key, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, if maintainers confirm the public CLI direction. A native startup selector is the maintainable surface for arbitrary directories, but it should reuse existing agent/session helpers and avoid treating two-part `agent:<id>` session shorthand as a silent alias.
