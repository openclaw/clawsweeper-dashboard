## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, not an urgent regression or runtime outage.
- `impact:session-state`: The startup agent determines the TUI session owner, workspace context, remembered-session behavior, and bootstrap context.
- `impact:auth-provider`: The selected agent controls model/provider routing, so missing startup selection can route the TUI to the wrong configured model or auth profile.

## Summary

Keep open. Current main and the latest shipped tag still do not expose a native `openclaw tui --agent <id>` startup selector; existing canonical session keys, workspace inference, and post-launch `/agent` switching are useful but do not cover the requested arbitrary-directory launch path.

## What This Changes

Request to add an `openclaw tui --agent <id>` startup selector so the TUI can launch directly against a configured non-default agent.

## Best Possible Solution

Add a typed `--agent <id>` startup path through the TUI CLI, options, startup resolver, validation, focused tests, and TUI docs, with explicit precedence against session keys and cwd workspace inference.

## Reproduction Assessment

Yes. Source inspection on current main and `v2026.5.20` shows `openclaw tui` has no `--agent` option, `TuiOptions` cannot carry one, and startup resolution only uses canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; the existing workspace workaround and post-launch `/agent` command do not cover that launch case.
