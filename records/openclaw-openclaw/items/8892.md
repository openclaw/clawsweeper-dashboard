## Summary

Current main still lacks a native startup selector for choosing an agent from an arbitrary directory. Agent-scoped session keys, workspace inference, the in-TUI agent picker, and the workspace workaround discussed in https://github.com/openclaw/openclaw/pull/40176 are useful adjacent paths, but they do not provide the requested `openclaw tui --agent <id>` interface; this issue should remain the canonical request pending a maintainer decision on precedence and remembered-session semantics.

Likely related people:
- Peter Steinberger — high confidence as the primary historical contributor across the current TUI CLI, runtime, tests, and docs.
- Vincent Koc — medium confidence from the recent refactor/import of the current TUI startup resolver and documentation.

## What This Changes

Add a native `openclaw tui --agent <id>` startup option that routes the session through the selected configured agent.

## Best Possible Solution

Add a validated typed startup agent id at the TUI CLI boundary and feed it into the existing canonical agent/session resolver with documented precedence, focused startup tests, and matching CLI/TUI documentation.

## Reproduction Assessment

Not applicable as a bug reproduction. Current source and documentation provide a high-confidence verification that the requested public startup capability is absent.

## Solution Assessment

Yes in direction: an explicit selector is the narrowest user-facing solution for generic directories. It should extend the existing startup resolver rather than establish a parallel routing mechanism, and maintainers must first define precedence and remembered-session behavior.
