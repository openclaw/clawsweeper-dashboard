## Summary

Current main still lacks the requested explicit startup selector. Agent-scoped session keys, workspace inference, and post-launch switching are useful alternatives, but none provides `openclaw tui --agent <id>` from an arbitrary directory.

## What This Changes

Requests a native `openclaw tui --agent <id>` option that starts the TUI with the selected configured agent’s model, workspace, and session context.

## Best Possible Solution

Add one validated startup agent selector that feeds the existing canonical TUI agent/session state, with explicit precedence rules and matching CLI, resolver, and documentation coverage rather than a parallel routing path.

## Reproduction Assessment

Yes, from source: invoking `openclaw tui` from a generic directory offers no explicit agent argument; startup selection is limited to session-prefix, cwd, and default resolution.

## Solution Assessment

Yes in direction, but the flag should reuse the existing agent/session resolver and requires a documented precedence decision before implementation.
