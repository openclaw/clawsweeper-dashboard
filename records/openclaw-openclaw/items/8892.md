## Label Justifications

- `P2`: This is a normal-priority TUI ergonomics and multi-agent routing feature with limited blast radius, not a crash or urgent regression.
- `impact:session-state`: The requested startup selector changes which agent session and workspace context the TUI attaches to.
- `impact:auth-provider`: Selecting a different startup agent can change the model, provider, and auth-profile routing used for the session.

## Summary

Keep open: current main and the latest release still do not provide a native `openclaw tui --agent <id>` startup selector, and the existing agent-scoped session key, cwd workspace inference, and post-launch `/agent <id>` paths do not cover the requested arbitrary-directory startup workflow.

## What This Changes

Request a native `openclaw tui --agent <id>` flag that starts the TUI on a selected configured agent, model, session scope, and workspace context.

## Best Possible Solution

Ship a focused additive TUI CLI change that validates `--agent <id>`, threads a typed startup agent id through TUI startup, reuses existing agent/session helpers, and updates focused tests plus public TUI docs after the precedence decision is settled.

## Reproduction Assessment

Yes. Source inspection against current main shows the TUI command has no `--agent` option, the TUI options type cannot carry one, and startup resolution only considers an agent-scoped session key, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, if maintainers confirm the public CLI direction. A native startup selector is the maintainable surface for arbitrary directories, but it should be additive and should not turn the two-part `agent:<id>` shorthand into a silent compatibility alias.
