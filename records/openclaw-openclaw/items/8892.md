## Label Justifications

- `P2`: This is a normal TUI feature request with clear scope and limited blast radius, but it is not an emergency regression.
- `impact:session-state`: The request is about which agent owns the TUI session key and workspace context at startup.
- `impact:auth-provider`: The selected agent controls model/provider routing, so the missing selector can route users to the wrong configured model/provider.

## Summary

Keep open. Current main still lacks a native `openclaw tui --agent <id>` startup selector; the existing canonical session-key and cwd workspace-inference paths are useful workarounds but do not cover selecting a non-default configured agent from an arbitrary directory at launch.

## What This Changes

The issue requests a TUI startup `--agent <id>` flag so a session can route to a configured non-default agent with that agent's model and workspace context.

## Best Possible Solution

Add a typed `--agent <id>` path through `src/cli/tui-cli.ts`, `TuiOptions`, and `runTui`, validate it against configured agents, define precedence with existing session/workspace behavior, and document/test the startup semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` has no `--agent` option, `TuiOptions` cannot carry one, and startup resolution only uses canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native startup selector is the narrowest maintainable way to select a non-default configured agent from arbitrary directories; current workspace inference and post-launch `/agent` switching do not cover that launch case.
