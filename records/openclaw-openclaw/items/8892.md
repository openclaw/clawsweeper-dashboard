## Summary

Keep open: current main still lacks a native `openclaw tui --agent <id>` startup selector, and the existing session-key, cwd workspace inference, remembered-session, and post-launch `/agent` paths do not cover the requested generic-directory launch flow.

## What This Changes

Request a `--agent` flag for `openclaw tui` so startup can route the interactive session to a configured agent instead of relying on the default, session key, or cwd workspace inference.

## Best Possible Solution

Add a typed `openclaw tui --agent <id>` startup selector with validation, documented precedence, focused TUI coverage, and CLI/TUI docs updates.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence the requested surface is absent: there is no CLI flag, no TUI option field, and startup agent resolution only uses session key, cwd, remembered session behavior, or the default agent.

## Solution Assessment

Yes, a first-class startup selector is the narrow direct solution if maintainers accept the UX. Existing cwd inference, canonical session keys, remembered sessions, and post-launch `/agent` remain useful but do not satisfy arbitrary-directory startup selection with `--agent <id>`.
