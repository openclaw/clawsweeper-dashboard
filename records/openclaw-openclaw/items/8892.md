## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, not an urgent regression or runtime outage.
- `impact:session-state`: The selected startup agent determines the TUI session owner, workspace context, and bootstrap context for the conversation.
- `impact:auth-provider`: Selecting a different configured agent can change model/provider routing and auth-backed provider configuration used by the session.

## Summary

Keep open. Current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector, and the existing agent-scoped session key, cwd workspace inference, and post-launch `/agent` paths do not cover the requested arbitrary-directory launch flow.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` flag so the terminal UI starts under a selected configured agent instead of always using the default agent or a workaround.

## Best Possible Solution

Add a typed `--agent <id>` startup selector through the TUI CLI, option contract, startup resolver, validation, focused tests, and TUI docs, with maintainer-approved precedence against existing session and workspace selection paths.

## Reproduction Assessment

Yes for the reported current behavior. Source inspection on current main shows no TUI `--agent` option, no startup-agent field in `TuiOptions`, and startup resolution limited to canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; current workarounds do not cover that launch case.
