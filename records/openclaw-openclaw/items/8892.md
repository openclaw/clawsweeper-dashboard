## Summary

Keep open: current main still lacks a native `openclaw tui --agent <id>` startup selector, and the existing session-key, workspace inference, remembered-session, and post-launch `/agent` paths do not cover selecting a non-default agent from an arbitrary directory at launch.

## What This Changes

Request a user-facing `openclaw tui --agent <id>` flag that starts the TUI bound to a configured non-default agent, including that agent's model and workspace context.

## Best Possible Solution

Add a typed `openclaw tui --agent <id>` selector with validation, documented precedence, focused TUI launch tests, and CLI/TUI docs updates.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high confidence the requested surface is absent: there is no CLI flag, no TUI option field, and startup resolution only considers session key, cwd workspace inference, and the default agent.

## Solution Assessment

Yes, a first-class startup selector is the narrow direct solution if maintainers accept the UX. Existing cwd inference, canonical session keys, remembered sessions, and `/agent` are useful but do not solve non-default agent selection from arbitrary directories at launch.
