## Summary

Keep open: current `main` and the latest release still lack a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, cwd workspace inference, remembered sessions, and post-launch `/agent <id>` are adjacent paths, but they do not cover the reported arbitrary-directory startup use case.

## What This Changes

Request a native `openclaw tui --agent <id>` flag that starts the TUI on a selected configured agent instead of relying on session names, cwd inference, or post-launch switching.

## Best Possible Solution

Add a typed startup agent selector in the TUI CLI/runtime path, validate it against configured agents, document precedence, and cover it with focused TUI and CLI tests if maintainers accept the UX.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence the requested CLI surface is absent on current `main` and the latest release.

## Solution Assessment

Unclear until maintainers decide the CLI UX and precedence contract. If accepted, a first-class `--agent <id>` selector is the narrow direct solution; existing cwd inference, canonical session keys, remembered sessions, and `/agent` switching do not satisfy arbitrary-directory startup selection.
