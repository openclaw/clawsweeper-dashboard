## Summary

Keep open. Current main still lacks a native `openclaw tui --agent <id>` startup selector; CWD workspace inference, canonical `agent:<id>:...` session keys, and post-launch `/agent` switching are useful but do not cover the requested generic-directory startup flow.

## What This Changes

Request adds an `openclaw tui --agent <id>` flag to start the TUI directly on a configured non-default agent.

## Best Possible Solution

Add a typed TUI startup agent selector, normalize and validate it consistently with existing agent switching, preserve current session/CWD behavior, and document the precedence with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main has no `--agent` option in `registerTuiCli`, no TUI option field for a startup agent id, and no explicit startup-agent branch in `resolveInitialTuiAgentId`.

## Solution Assessment

Yes. A dedicated `--agent <id>` startup option is the narrowest maintainable solution because it leaves session keys, CWD workspace inference, and post-launch `/agent` switching intact while covering the generic-directory startup use case.

## Repair Work Prompt

Implement a native `openclaw tui --agent <id>` startup selector for this issue. Add the Commander option in `src/cli/tui-cli.ts`, extend `TuiOptions`, propagate the option through `runTui` and `tui-launch` relaunch args, and update startup agent resolution so a bare `--session <key>` scopes to the selected agent. Codify precedence with tests: recommended behavior is that a conflicting canonical `--session agent:<other>:...` and `--agent <id>` fails clearly, a matching canonical session is accepted, `--agent` wins over CWD workspace inference and remembered last session startup, and existing post-launch `/agent` behavior stays unchanged. Do not change `parseAgentSessionKey` to treat two-part `agent:<id>` as canonical. Update focused tests, docs (`docs/cli/tui.md`, `docs/web/tui.md`), and `CHANGELOG.md` because this is user-facing.
