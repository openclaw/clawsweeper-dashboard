## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, but it is not an emergency regression.
- `impact:session-state`: The request is about which configured agent owns the TUI startup session and workspace context.
- `impact:auth-provider`: The selected agent controls the model/provider routing, so missing startup selection can route users to the wrong configured model/provider.

## Summary

Keep open. Current main still lacks a native `openclaw tui --agent <id>` startup selector; canonical session keys, cwd workspace inference, and post-launch `/agent` switching are useful but do not cover selecting a configured agent from an arbitrary directory at launch.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` flag so TUI startup can bind directly to a configured non-default agent.

## Best Possible Solution

Add a typed `--agent <id>` path through `src/cli/tui-cli.ts`, `TuiOptions`, and `runTui`, validate it against configured agents, define precedence with existing session/workspace behavior, and document/test the startup semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` has no `--agent` option, `TuiOptions` cannot carry one, and startup resolution only uses canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native startup selector is the narrowest maintainable way to select a non-default configured agent from arbitrary directories; current workspace inference and post-launch `/agent` switching do not cover that launch case.
