## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, not an urgent regression or runtime outage.
- `impact:session-state`: The startup agent determines the TUI session owner, workspace context, and remembered-session scope.
- `impact:auth-provider`: The selected agent controls model/provider routing, so missing startup selection can route to the wrong configured model.

## Summary

Keep open. Current main still lacks a native `openclaw tui --agent <id>` startup selector; workspace inference, canonical `--session agent:<id>:...`, and post-launch `/agent` switching are useful adjacent paths but do not cover choosing a configured agent from an arbitrary directory.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` flag so TUI starts with a configured non-default agent and its model/workspace context.

## Best Possible Solution

Add a typed `--agent <id>` startup path through the TUI CLI, option type, and resolver, validate it against configured agents, and document/test its precedence with session and workspace behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` has no `--agent` option, `TuiOptions` cannot carry one, and startup resolution only uses canonical session keys, CWD workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; the existing workspace workaround does not cover that launch case.
