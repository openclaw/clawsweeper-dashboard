## Label Justifications

- `P2`: This is a clear TUI feature request with bounded blast radius, not an urgent regression or core runtime outage.
- `impact:session-state`: The startup agent determines the TUI session owner, workspace context, remembered session scope, and bootstrap context.
- `impact:auth-provider`: Selecting a different configured agent can change model/provider routing and auth-backed provider configuration for the session.

## Summary

Keep open. Current main and the latest release still do not provide a native `openclaw tui --agent <id>` startup selector; the existing alternatives are workspace-directory inference, canonical agent-scoped session keys, and post-launch `/agent`, which do not satisfy the arbitrary-directory startup flow requested here.

## What This Changes

Add a `--agent <id>` flag to `openclaw tui` so a TUI session can start directly against a configured non-default agent.

## Best Possible Solution

Add a typed, validated `openclaw tui --agent <id>` startup selector through the TUI CLI, option contract, startup resolver, tests, and docs, with maintainer-approved precedence against existing session and workspace selection paths.

## Reproduction Assessment

Yes. Source inspection shows current main has no TUI `--agent` option, no startup-agent field in `TuiOptions`, and startup resolution is limited to canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; current workarounds do not cover that launch case.
