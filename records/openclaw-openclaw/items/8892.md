## Summary

Keep open: current main still lacks a native `openclaw tui --agent <id>` startup selector, and the existing canonical session-key, cwd workspace inference, and post-launch `/agent` paths do not satisfy the generic-directory startup use case.

## What This Changes

Request adding a native `openclaw tui --agent <id>` flag so the TUI starts directly against a configured non-default agent.

## Best Possible Solution

Add a typed `--agent <id>` startup selector in the TUI CLI/runtime, validate it against configured agents, define precedence with existing session and cwd inference behavior, then update focused TUI tests and the public TUI docs.

## Reproduction Assessment

Yes. Source inspection on current main shows the TUI command has no `--agent` option and the startup resolver has no explicit agent-id input, while existing tests cover only session-prefix and cwd-inference paths.

## Solution Assessment

Yes in direction, with a product detail to settle. A first-class `--agent <id>` option is cleaner than overloading malformed `--session agent:<id>` shorthand, but maintainers should define precedence before implementation.
