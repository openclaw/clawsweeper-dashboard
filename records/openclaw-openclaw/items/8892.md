## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, not an urgent regression or core runtime outage.
- `impact:session-state`: The selected startup agent determines the TUI session owner, workspace context, and bootstrap context for the conversation.
- `impact:auth-provider`: Selecting a different configured agent can change model/provider routing and auth-backed provider configuration used by the session.

## Summary

Keep open. Current main and the latest release still do not provide a native `openclaw tui --agent <id>` startup selector; existing support covers canonical agent-scoped session keys, cwd workspace inference, and post-launch agent switching, but not the requested arbitrary-directory startup flow.

## What This Changes

The issue requests a TUI `--agent <id>` flag that starts an interactive session under a selected configured agent, including that agent's model, workspace, and context.

## Best Possible Solution

Add a typed `--agent <id>` startup selector through the TUI CLI, option contract, startup resolver, validation, focused tests, and TUI docs, with owner-approved precedence against existing session and workspace selection paths.

## Reproduction Assessment

Yes, source-level. Current main has no TUI `--agent` option, no startup-agent field in `TuiOptions`, and startup resolution is limited to canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; current workarounds do not cover that launch case.
