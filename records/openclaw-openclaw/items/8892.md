## Summary

Keep this open: current main and the latest release still do not provide a native `openclaw tui --agent <id>` startup selector. The existing canonical `agent:<id>:...` session key, cwd workspace inference, remembered-session handling, and post-launch `/agent <id>` command are useful, but they do not satisfy selecting a configured agent from an arbitrary directory at TUI startup.

## What This Changes

Request a new `openclaw tui --agent <id>` flag that starts the TUI against a specific configured agent, including that agent's model, workspace context, and agent-scoped session state.

## Best Possible Solution

Add a typed `--agent <id>` startup selector for `openclaw tui`, validate the configured agent id, define precedence explicitly, and update focused TUI tests plus the public TUI CLI/guide docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main registers no `--agent` flag, has no TUI option field for a startup agent id, and routes startup through session parsing or cwd inference only.

## Solution Assessment

Yes, if maintainers accept the new CLI mode. A native typed `--agent <id>` startup selector is the narrowest maintainable direction; modeling this as the invalid `--session agent:<id>` shorthand would not solve the reported workflow.
