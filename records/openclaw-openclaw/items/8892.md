## Label Justifications

- `impact:session-state`: The request changes which agent-scoped session and workspace context the TUI attaches to at startup.
- `impact:auth-provider`: Selecting a different agent at startup changes the configured model/provider and related auth profile path used by the session.

## Summary

Keep open. Current main still has no native TUI startup agent selector; canonical agent-scoped session keys, cwd workspace inference, and post-launch `/agent` are useful but do not satisfy selecting a configured agent from arbitrary directories at launch.

## What This Changes

Add an `openclaw tui --agent <id>` startup option that selects the configured agent, model, workspace, and agent context for the session.

## Best Possible Solution

Add a typed `--agent <id>` path through `src/cli/tui-cli.ts`, `TuiOptions`, `runTui`, and focused TUI tests, with docs that define validation and precedence against existing session and workspace inference behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `openclaw tui` has no `--agent` option, `TuiOptions` cannot carry one, and startup resolution only uses canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes. A native startup selector is the narrowest maintainable solution; existing workspace inference and post-launch `/agent` switching do not cover launching a non-default agent from an arbitrary directory.
