## Summary

Keep open: current main and the latest release still do not provide a native `openclaw tui --agent <id>` startup selector, and the existing session-key, workspace-inference, and in-TUI switching paths do not cover the requested generic-directory startup workflow.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` flag so TUI startup can bind directly to a configured non-default agent and its model/workspace context.

## Best Possible Solution

Ship a focused TUI CLI change that validates `--agent <id>`, threads a typed startup agent id through TUI startup, reuses existing agent/session helpers, and updates focused TUI tests plus public TUI docs after the precedence decision is settled.

## Reproduction Assessment

Yes. Source inspection against current main shows the TUI command has no `--agent` option, the TUI options type cannot carry one, and startup resolution only considers an agent-scoped session key, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, if maintainers confirm the public CLI direction. A native startup selector is the maintainable surface for arbitrary directories, but it should be additive and should not turn the two-part `agent:<id>` shorthand into a silent compatibility alias.
