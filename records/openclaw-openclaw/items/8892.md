## Label Justifications

- `P2`: This is a clear TUI feature request with limited blast radius, not an urgent regression or runtime outage.
- `impact:session-state`: The startup agent determines the TUI session owner, workspace context, and bootstrap context for the conversation.
- `impact:auth-provider`: Selecting a different configured agent can change the model/provider routing and auth-backed provider configuration used by the session.

## Summary

Keep open. Current main still lacks a native `openclaw tui --agent <id>` startup selector, and the existing canonical session-key, cwd workspace inference, and post-launch agent switching paths do not satisfy the requested arbitrary-directory launch flow.

## What This Changes

Request a TUI `--agent <id>` flag that starts an interactive session under a configured non-default agent instead of only naming the session.

## Best Possible Solution

Add a typed `--agent <id>` startup selector through the TUI CLI, options contract, resolver, validation, focused tests, and TUI docs, with maintainer-approved precedence against existing session and workspace selection paths.

## Reproduction Assessment

Yes for the reported current behavior. Source inspection shows no TUI `--agent` option, no startup-agent field in `TuiOptions`, and startup resolution limited to canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; current workarounds do not cover that launch case.
