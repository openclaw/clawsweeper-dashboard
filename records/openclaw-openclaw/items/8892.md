## Label Justifications

- `P2`: This is a clear TUI feature request with bounded blast radius, not an urgent regression or core runtime outage.
- `impact:session-state`: The startup agent determines the TUI session owner, workspace context, remembered session scope, and bootstrap context.
- `impact:auth-provider`: Selecting a different configured agent can change model/provider routing and auth-backed provider configuration for the session.

## Summary

Keep open. Current main and latest release still lack a native `openclaw tui --agent <id>` startup selector; the existing session-prefix, cwd workspace inference, and post-launch `/agent` paths do not cover the requested arbitrary-directory startup flow.

## What This Changes

Request adds a TUI `--agent <id>` startup flag so a terminal session can begin under a selected configured agent, model, workspace, and session context.

## Best Possible Solution

Add a typed `--agent <id>` startup selector through the TUI CLI, option contract, startup resolver, validation, focused tests, and TUI docs, with maintainer-approved precedence against existing session and workspace selection paths.

## Reproduction Assessment

Yes, source-level. Current main has no TUI `--agent` option, no startup-agent field in `TuiOptions`, and startup resolution is limited to canonical session keys, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, with maintainer-defined precedence. A native typed startup selector is the narrowest maintainable way to choose a non-default configured agent from arbitrary directories; current workarounds do not cover that launch case.
