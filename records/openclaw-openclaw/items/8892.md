## Summary

Keep open: current main still does not provide a native `openclaw tui --agent <id>` startup selector, and the existing session-key, workspace inference, remembered-session, and post-launch `/agent` paths do not cover the requested generic-directory launch flow.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` flag that starts the TUI under a configured non-default agent with that agent's model and workspace context.

## Best Possible Solution

Add a typed `openclaw tui --agent <id>` startup selector with validation, documented precedence, focused TUI coverage, and CLI/TUI docs updates.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence the requested surface is absent: there is no CLI flag, no TUI option field, and startup agent resolution only uses session key, cwd, or the default agent.

## Solution Assessment

Yes, a first-class startup selector is the narrow direct solution if maintainers accept the UX. Existing cwd inference, canonical session keys, remembered sessions, and post-launch `/agent` remain useful but do not satisfy arbitrary-directory startup selection with `--agent <id>`.
