## Summary

Keep open: current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, workspace inference, and post-launch `/agent <id>` are useful workarounds, but they do not cover the requested generic-directory startup workflow, and no open PR currently owns this exact change.

## What This Changes

Add a native `openclaw tui --agent <id>` flag so startup can bind the TUI session to a configured non-default agent, model, workspace, and context.

## Best Possible Solution

Ship a focused additive TUI CLI design that validates `--agent <id>`, threads a typed startup agent id through TUI startup, preserves existing session/workspace behavior, and updates focused tests plus public TUI docs after precedence is decided.

## Reproduction Assessment

Yes. Current-main source inspection shows `openclaw tui` has no `--agent` option, the TUI option type cannot carry one, and startup resolution only considers an agent-scoped session key, cwd workspace inference, then the default agent.

## Solution Assessment

Yes, if maintainers confirm the public CLI direction. A native startup selector is the maintainable surface for arbitrary directories, but it should be additive, validated, and explicit about precedence rather than treating `agent:<id>` shorthand as a hidden alias.
