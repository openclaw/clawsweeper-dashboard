## Summary

Keep this open: current main and the latest release still lack a native `openclaw tui --agent <id>` startup selector. Existing agent-scoped session keys, workspace inference, remembered sessions, and post-launch `/agent <id>` are useful adjacent paths, but they do not satisfy selecting a configured agent from an arbitrary directory at startup.

## What This Changes

The issue requests a native `openclaw tui --agent <id>` flag so the TUI starts directly on a configured non-default agent.

## Best Possible Solution

Add a typed `--agent <id>` startup selector for `openclaw tui`, validate the configured agent id, define precedence explicitly, and update focused TUI tests plus the public TUI CLI and guide docs.

## Reproduction Assessment

Yes. Source inspection gives high confidence: the TUI CLI does not define `--agent`, `TuiOptions` cannot carry it, and startup routing only uses a canonical session key, cwd workspace inference, or the default agent.

## Solution Assessment

Yes, if maintainers accept the new CLI mode. A native typed `--agent <id>` startup selector is narrower and clearer than overloading the invalid `--session agent:<id>` shorthand; the unresolved part is the precedence contract.
