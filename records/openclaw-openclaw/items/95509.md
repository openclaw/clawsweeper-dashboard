## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Deep-merge nested TTS provider overrides:** `ui/src/ui/views/agents-panels-overview.ts:65`
  - body: This resolver says it mirrors `resolveEffectiveTtsConfig`, but it only replaces top-level keys. If `messages.tts.providers.elevenlabs` holds the API key/model and an agent override only sets `providers.elevenlabs.speakerVoiceId`, the UI drops the inherited provider fields while runtime keeps them, so the card can show an empty credential/default model and save a misleading override.
  - confidence: 0.91
- **[P1] Reflect TTS auto mode in the enable control:** `ui/src/ui/views/agents-panels-overview.ts:75`
  - body: The enable switch reads only `mergedTts.enabled`, but current TTS uses session/prefs/config `auto` before falling back to the deprecated `enabled` boolean. Existing configs with `auto: "always"` render as off here, while `auto: "off"` remains off even if this switch writes `enabled: true`; model the actual auto mode or avoid a boolean control that runtime can ignore.
  - confidence: 0.88
- **[P2] Render effective TTS from runtime config:** `ui/src/ui/views/agents-panels-overview.ts:39`
  - body: resolveAgentTts chooses only the agent/default block, but runtime TTS starts from messages.tts and deep-merges per-agent overrides. Existing configs can keep auto mode and provider credentials globally while agents override only voice fields, so this card can show TTS as disabled or incomplete for a valid runtime setup.
  - confidence: 0.9
- **[P2] Write ElevenLabs modelId so the selector affects synthesis:** `ui/src/ui/app-render.ts:3561`
  - body: The model handler writes providers.elevenlabs.model, but the ElevenLabs speech provider reads modelId when building the synthesis request. With this path, changing the UI selector can leave runtime synthesis on the provider default model.
  - confidence: 0.86
