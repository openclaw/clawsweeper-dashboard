## Summary

Keep open: current main has no Exo or RequestEventLog implementation, but the generic node pending-work surface can hand a node a baseline status request on every drain, so this needs OpenClaw-side log evidence or a maintainer decision on the node polling contract before cleanup closure.

## What This Changes

The issue asks OpenClaw to stop excessive RequestEventLog polling against Exo during normal gateway operation.

## Best Possible Solution

Define the node pending-work cadence contract, then either throttle baseline status responses in the gateway or require the Exo-Service node to back off, with regression coverage for the chosen behavior.

## Reproduction Assessment

No, not end to end: source inspection shows repeated node.pending.drain calls can receive baseline status work every time, but current main has no Exo or RequestEventLog sender tying that directly to the reporter's Exo log.

## Solution Assessment

Unclear: the desired lower polling rate is reasonable, but the maintainable fix depends on whether OpenClaw owns the polling cadence or the external Exo-Service node must apply backoff.
