Keep this open: current main and v2026.6.6 still only cover the stale IPv4 self-loop portproxy case, and the linked repair PR was closed unmerged because it overgeneralized Chrome behavior rather than because the issue was solved.

## What This Changes

The issue asks OpenClaw to document and diagnose Windows/WSL2 Chrome CDP empty replies when Chrome is reachable on `[::1]:9222` but an IPv4 `v4tov4` portproxy points at `127.0.0.1:9222`.

## Best Possible Solution

Land a narrowly scoped docs and diagnostic update that tells users how to distinguish IPv4 self-loop from observed IPv6-only CDP reachability and when to use `v4tov6`, without adding config, automatic `netsh` repair, or a WSL-side `[::1]` probe.

## Reproduction Assessment

Yes for the OpenClaw gap: current source, tests, docs, and v2026.6.6 all still lack IPv6/v4tov6 guidance. The external Chrome behavior has detailed Windows/WSL2 reporter commands, but I did not run a live Windows Chrome repro here.

## Solution Assessment

Yes, with narrower wording than the closed PR used. The maintainable fix is to document and hint the observed netstat/curl split and v4tov6 remedy while avoiding a universal Chrome-version claim or invasive Windows mutation.
