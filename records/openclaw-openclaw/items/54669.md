## Summary

Keep open: current main and v2026.5.22 still lack the reported `[::1]` / `v4tov6` WSL2 recovery path in the guide and browser diagnostics. The merged self-loop hint helps a related empty-reply case, but it does not cover the IPv6-only Chrome listener mismatch preserved by https://github.com/openclaw/openclaw/issues/68666.

## What This Changes

The issue requests WSL2 Windows remote CDP docs and browser doctor diagnostics for Chrome binding CDP on IPv6 `[::1]` while IPv4 `v4tov4` portproxy rules fail silently.

## Best Possible Solution

Land a focused WSL2 browser guide and browser CDP diagnostic update that teaches users to test `[::1]`, inspect the Windows listener family, use `v4tov6 connectaddress=::1` when appropriate, and keep Windows network changes as explicit user-run recovery steps.

## Reproduction Assessment

Yes, source-backed and manually reproducible: this issue plus https://github.com/openclaw/openclaw/issues/68666 give concrete Windows/WSL2 curl and listener checks, and current main still lacks the `[::1]` / `v4tov6` recovery path. I did not live-run the Windows setup here.

## Solution Assessment

Yes. The narrow maintainable fix is docs plus a targeted browser CDP diagnostic hint/test; automatically mutating Windows `netsh portproxy` rules would be broader and riskier.
