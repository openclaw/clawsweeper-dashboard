## Repair Work Prompt

Fix the WSL2 + Windows remote Chrome CDP diagnostics gap tracked by https://github.com/openclaw/openclaw/issues/54669. Update the WSL2 browser troubleshooting guide to add a conditional IPv4-vs-IPv6 CDP listener diagnosis: test Windows `curl.exe http://127.0.0.1:9222/json/version`, `curl.exe http://[::1]:9222/json/version`, and `netstat -ano | findstr :9222`; explain that when Chrome responds on `[::1]` but IPv4 returns empty, the Windows portproxy bridge should use `v4tov6 connectaddress=::1` rather than `v4tov4 connectaddress=127.0.0.1`. Also mention the Chrome 136+ non-default `--user-data-dir` requirement from Chrome's official remote-debugging change. Add a targeted browser doctor/status hint for remote CDP profiles when CDP HTTP/WebSocket is unreachable that points WSL2 Windows users to the IPv6 listener/portproxy check; keep it read-only and do not run or auto-repair `netsh`, firewall, or Windows services. Coordinate wording with the separate portproxy self-loop report at https://github.com/openclaw/openclaw/issues/59209 but do not broaden this fix into automatic Windows network management. Before docs edits, run `pnpm docs:list` and read only relevant docs. Add focused regression coverage for the hint and update docs checks; add a changelog fix entry if CLI/doctor output changes.
