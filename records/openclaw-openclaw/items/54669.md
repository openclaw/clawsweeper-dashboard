## Summary

Keep open: current main and the latest release still do not document the reported Windows Chrome `[::1]`/`v4tov6` WSL2 CDP recovery path, and the browser diagnostics still collapse this failure into generic CDP reachability hints. The closed duplicate preserves useful extra Chrome 147 and `--user-data-dir` context, while the broader layered-diagnostics PR that once superseded related work was closed unmerged.

## What This Changes

The issue asks OpenClaw to document and diagnose Windows Chrome CDP IPv6 loopback binding so WSL2 remote browser profiles do not fail silently behind IPv4-only portproxy rules.

## Best Possible Solution

Update the WSL2 browser guide with conditional IPv4/IPv6 CDP diagnosis and `v4tov6` guidance, then add a targeted browser doctor/status hint without automatically mutating Windows networking rules.

## Reproduction Assessment

Yes, for a high-confidence source-backed/manual path: this issue and the closed duplicate provide concrete Windows/WSL2 curl and netstat steps, and current main still lacks the `[::1]`/`v4tov6` docs and diagnostic path. I did not live-run Windows/WSL2 here.

## Solution Assessment

Yes. The narrow maintainable fix is docs plus a targeted diagnostic hint; automatic management of Windows `netsh portproxy` rules would be broader and riskier.
