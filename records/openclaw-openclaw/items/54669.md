## Summary

Keep this open: current main and v2026.6.8 still only hint the stale IPv4 portproxy self-loop case, while the reported IPv4-vs-IPv6 CDP binding remedy remains undocumented and unimplemented.

## What This Changes

The issue asks OpenClaw to document and diagnose Windows/WSL2 Chrome CDP cases where IPv4 portproxy rules fail because the reachable CDP listener is on IPv6 loopback.

## Best Possible Solution

Land a narrow docs and diagnostic update that tells users how to distinguish stale self-loop portproxy from the observed IPv4-vs-IPv6 CDP split and when to use `v4tov6`, without adding config, automatic Windows mutation, or a WSL-side `[::1]` auto-probe.

## Solution Assessment
