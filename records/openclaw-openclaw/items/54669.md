## Summary

Keep open. Current main still has the WSL2 + Windows remote CDP documentation gap and still reports generic configured-endpoint CDP diagnostics, while the related #68666 report was closed as a duplicate of this issue rather than replacing it as the canonical tracker.

## What This Changes

Request documentation and browser doctor/status diagnostics for Chrome on Windows binding CDP to IPv6 loopback, which breaks WSL2 v4tov4 portproxy setups.

## Best Possible Solution

Keep this as the canonical tracker and land a narrow browser-plugin docs/diagnostics update: add Windows-side IPv4 vs IPv6 CDP checks and v4tov6 portproxy guidance to the WSL2 guide, then add a browser doctor/status hint that explains the likely Windows Chrome IPv6 loopback mismatch without changing generic CDP transport or SSRF policy.

## Reproduction Assessment

Yes. The issue and duplicate provide concrete Windows/WSL2 curl and netstat steps, and current-main source inspection confirms the missing docs and diagnostic hint paths; I did not run a live Windows Chrome instance in this Linux checkout.

## Solution Assessment

Yes. The requested docs update plus a scoped browser doctor/status hint is the narrowest maintainable fix; changing core CDP connection behavior would be broader and riskier than helping users diagnose and configure the Windows portproxy correctly.
