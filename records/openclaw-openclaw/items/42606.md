## Summary

Keep open. The security hardening remains valid, but current main still serves the live noVNC bootstrap without the proposed CSP, `nosniff`, or frame-deny headers, and this PR branch targets the stale `src/browser` path instead of the active browser plugin files.

## What This Changes

The PR adds a nonce-based CSP, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, and matching header/nonce assertions for the noVNC bootstrap response in the old `src/browser` bridge files.

## Best Possible Solution

Port the same narrow hardening to `extensions/browser/src/browser/bridge-server.ts` and `extensions/browser/src/browser/bridge-server.auth.test.ts`, preserving the one-time observer token flow and fragment-only password behavior while adding nonce-based CSP, `nosniff`, and frame-deny coverage on the live route.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to start the browser bridge with `resolveSandboxNoVncToken`, fetch `/sandbox/novnc?token=<valid-token>`, and inspect headers/body; current source and tests show the live route lacks the requested hardening headers.

## Solution Assessment

No, not as currently proposed. The nonce-based CSP approach is the right narrow fix, but the PR must be applied to the active `extensions/browser` implementation rather than the stale `src/browser` files.
