## Summary

Keep open. The hardening is still valid, but this PR patches the old `src/browser` bridge path while current `main` serves the live noVNC bootstrap from the browser plugin path, so merging as written would not harden the active route.

## What This Changes

The PR branch adds a nonce-based CSP, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, and matching regression assertions to the old browser bridge files.

## Best Possible Solution

Apply the nonce-based CSP, `nosniff`, and frame-deny hardening to `extensions/browser/src/browser/bridge-server.ts` with focused coverage in `extensions/browser/src/browser/bridge-server.auth.test.ts`, while preserving one-time token redemption and fragment-only password behavior.

## Reproduction Assessment

Yes. A high-confidence path is to start the browser bridge with `resolveSandboxNoVncToken`, request `/sandbox/novnc?token=<valid-token>` with bridge auth, and inspect the headers/body; current source and tests show the active route lacks the requested hardening.

## Solution Assessment

No, not as currently proposed. The nonce-based CSP approach is the right narrow fix, but the PR must be ported from the stale `src/browser` files to the active browser plugin implementation.
