## Summary

Keep open: current main and the latest release still apply native-Ollama object normalization to OpenAI-compatible Ollama payloads, matching the reporter's captured 400 on second-turn tool-call replay.

## What This Changes

The issue asks OpenClaw to keep Ollama Cloud/OpenAI-compatible `tool_calls[].function.arguments` serialized as JSON strings when replaying prior tool calls.

## Best Possible Solution

Split the Ollama native and OpenAI-compatible replay contracts: keep object coercion only for native `/api/chat`, and preserve string arguments for Ollama Cloud/OpenAI-compatible Chat Completions while retaining `num_ctx` injection.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the issue provides a captured failing payload and current code shows the Ollama OpenAI-compatible wrapper converting string tool-call arguments back to objects before send. I did not run a live Ollama Cloud credential test in this read-only pass.

## Solution Assessment

Yes, the best fix is narrow: stop applying native Ollama object coercion to OpenAI-compatible Chat Completions payloads, and add regression coverage proving both API contracts remain distinct.
