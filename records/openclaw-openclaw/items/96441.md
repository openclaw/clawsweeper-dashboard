## Summary

Keep open: current main and v2026.6.10 still apply native Ollama object normalization to OpenAI-compatible Ollama payloads, and the viable fix is still an open linked PR rather than merged code.

## What This Changes

The issue reports a regression where Ollama Cloud/OpenAI-compatible second-turn tool-call replay sends `tool_calls.function.arguments` as an object instead of the required JSON string, causing Ollama Cloud to reject the request.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96474 as the canonical proof-positive fix, then close this issue as implemented and close or supersede duplicate candidate PRs.

## Reproduction Assessment

Yes. Source inspection reproduces the reported payload-shape bug: current main stringifies Chat Completions tool-call arguments, then the Ollama compat wrapper parses those strings back into objects before send.

## Solution Assessment

Yes. The narrow best fix is to remove the OpenAI-compatible wrapper's object coercion while preserving native Ollama object conversion, and the open canonical PR already follows that shape.
