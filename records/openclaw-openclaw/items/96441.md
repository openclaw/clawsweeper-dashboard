## Summary

Keep open: current main and latest release still contain the Ollama OpenAI-compatible wrapper mutation that converts Chat Completions tool-call argument strings back to objects, and the open proof-positive fixing PR has not merged yet.

## What This Changes

The issue reports that Ollama Cloud/OpenAI-compatible second-turn tool-call replay sends `tool_calls.function.arguments` as a JSON object instead of the JSON string required by the Chat Completions contract.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96474 or an equivalent wrapper-level fix, then close this issue after merge with proof that OpenAI-compatible replay keeps string arguments and native Ollama replay keeps object arguments.

## Reproduction Assessment

Yes. Source inspection shows the OpenAI-compatible converter stringifies tool-call arguments, then the Ollama compat wrapper parses those strings back into objects before send; the issue also includes a captured Ollama Cloud 400 for that payload shape.

## Solution Assessment

Yes. The best fix is narrow and already represented by the linked candidate PR: remove native-object coercion from the OpenAI-compatible Ollama wrapper while leaving native `/api/chat` object replay conversion intact.
