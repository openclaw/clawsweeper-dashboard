## Summary

Keep open: current main and the latest release still apply native-Ollama object normalization to the Ollama OpenAI-compatible payload patch, and several open PRs already claim this issue with closing syntax. The issue should stay open until one canonical fix PR lands.

## What This Changes

The issue reports that Ollama Cloud OpenAI-compatible tool-call history is replayed with object arguments instead of JSON strings, causing second-turn 400 errors.

## Best Possible Solution

Land one narrow canonical fix, preferably the proof-sufficient https://github.com/openclaw/openclaw/pull/96474 if it still matches current main, preserving OpenAI-compatible string arguments while keeping native Ollama object arguments covered.

## Reproduction Assessment

Yes. The issue includes a captured failing provider payload, and current main shows the OpenAI-compatible payload starts with stringified arguments before the Ollama wrapper converts them back to objects.

## Solution Assessment

Yes. The maintainable fix is to split the native Ollama and OpenAI-compatible replay contracts, not add a new config flag or a broad runtime fallback.
