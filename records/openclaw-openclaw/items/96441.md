## Summary

Keep open: current main and the latest shipped tag still apply native-Ollama object normalization to the OpenAI-compatible Ollama payload path, so the reported second-turn 400 remains unresolved until one linked fix PR lands.

## What This Changes

The issue asks OpenClaw to preserve `tool_calls.function.arguments` as JSON strings when replaying tool-call history through Ollama Cloud's OpenAI-compatible Chat Completions path.

## Best Possible Solution

Land one narrow fix, preferably the proof-positive path in https://github.com/openclaw/openclaw/pull/96474, that keeps native Ollama object replay while preventing the OpenAI-compatible wrapper from converting string arguments back to objects.

## Reproduction Assessment

Yes, at source level: current main stringifies tool-call arguments in the shared OpenAI Chat Completions converter and then the Ollama OpenAI-compatible wrapper converts those fields back to objects before send. I did not establish a live Ollama Cloud round trip in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow split-contract repair in `extensions/ollama/src/stream.ts`, preserving native `/api/chat` object arguments while leaving OpenAI-compatible `/v1/chat/completions` arguments as JSON strings.
