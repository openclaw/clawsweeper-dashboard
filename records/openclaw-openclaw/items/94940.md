## Summary

Keep this PR open: the keyless Google Chat direction is useful, but the current patch adds an ADC auth path that can reach GCE metadata-token requests outside the existing OpenClaw guarded Google auth transport, and the shared proof still does not show the outbound ADC-authenticated reply path.

## What This Changes

The PR adds an opt-in `channels.googlechat.serviceAccountAdc` config key, threads a new `adc` credential source through Google Chat account resolution, updates generated config metadata/docs, and adds resolver/schema tests.

## Best Possible Solution

Keep the opt-in keyless Google Chat direction, but land it only after metadata-token ADC is either routed through an explicit OpenClaw-owned guard or accepted as a documented maintainer/security exception with focused tests and redacted live proof.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows the new `adc` source reaches GoogleAuth ADC, and the published dependency source shows Compute ADC uses direct `gcp-metadata` requests. I did not run a live GCE reproduction in this read-only review.

## Solution Assessment

No: using GoogleAuth's ADC fallback is a plausible direction, but enabling it without an OpenClaw-owned metadata-server policy or explicit maintainer security exception is not the best final shape for this channel auth path.
