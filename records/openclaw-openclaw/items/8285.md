## Label Justifications

- `P2`: This is a meaningful cross-channel messaging UX improvement, but it is neither an urgent regression nor a core availability failure.
- `impact:other`: Users can receive no clear textual progress feedback during long-running agent work across multiple messaging channels.

## Summary

Current main still lacks supported cross-channel acknowledgment text before agent processing, and the related Feishu and waiting-tips implementations both closed unmerged. The request remains useful, but its permanent core-config versus plugin-hook contract requires maintainer product judgment.

## What This Changes

Add configurable, scoped text acknowledgment that is delivered before normal agent processing across messaging channels.

## Best Possible Solution

Define one canonical interim-message lifecycle with scope, optional delay, cancellation, cleanup, and failure semantics, expose it through the narrowest approved core or plugin API contract, and keep channel adapters responsible only for transport-specific delivery.

## Reproduction Assessment

Yes at the source-contract level: current main provides typing and reaction feedback but no supported configurable text acknowledgment before processing. This is a missing capability rather than a failure of an existing documented contract.

## Solution Assessment

No, isolated per-channel timers are not the best general solution. A canonical cross-channel interim-message lifecycle or explicitly approved non-terminal plugin seam would avoid duplicated policy and inconsistent cleanup behavior.
