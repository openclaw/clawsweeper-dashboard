## Summary

Keep open. Current main and the latest release still use built-in bare reset/startup prompt strings and expose no root or per-agent `resetPrompt` config surface; related duplicate cleanup has already pointed the same prompt-config request back to this issue, while greeting-model and WebChat acknowledgement reports track adjacent but different behavior.

## What This Changes

Add a global and per-agent `session.resetPrompt` override for the bare `/new` and `/reset` startup prompt, falling back to the current built-in prompt when unset.

## Best Possible Solution

Add a typed reset-prompt resolver that applies the maintainer-approved root/per-agent precedence, preserves the built-in fallback when unset, keeps bootstrap-specific behavior explicit, and updates schema, types, docs, help metadata, and focused reset prompt/Gateway/config tests.

## Reproduction Assessment

Yes, source-level. Current main still routes reset model-input paths through `resolveBareSessionResetPromptState()` and strict root/per-agent schemas expose no requested reset-prompt override shape.

## Solution Assessment

Unclear until maintainers settle the public config contract. A full replacement prompt with the current behavior as fallback looks like the narrowest direction, but bootstrap-specific prompts and nearby reset-greeting policy should be handled deliberately.
