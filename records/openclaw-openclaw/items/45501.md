## Summary

Keep open. Current main and the latest release still hardcode the bare reset startup prompt and expose no root or per-agent config key to replace it; the comments include concrete Ollama/gemma4 impact tied to the exact wording, and related reset-greeting or multi-session items do not supersede this narrower request.

## What This Changes

Add a configurable global and per-agent `session.resetPrompt` override for the startup message used after bare `/new` or `/reset`, with the current prompt as the fallback.

## Best Possible Solution

Keep this as the canonical narrow prompt-override feature: implement full replacement text with fallback to the current default, support per-agent-over-root precedence, wire both gateway and reply-producing reset paths through one helper, and document that reset backups, greeting-model selection, suppression, and multi-session architecture are separate work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send a bare `/new` or `/reset` through a reply-producing or gateway agent path and current main reaches `buildBareSessionResetPrompt()`, while strict root and per-agent schemas reject the requested override shape.

## Solution Assessment

Yes, if maintainers accept this public key. A full replacement string with current behavior as fallback is the narrowest maintainable fix for the reported wording problem; bootstrap safety semantics should be explicit rather than folded into unrelated reset-backup or greeting-model work.

Implement the narrow configurable reset prompt feature for https://github.com/openclaw/openclaw/issues/45501. Add a documented root `session.resetPrompt?: string` and per-agent `agents.list[].session.resetPrompt?: string` override with precedence per-agent > root > existing default. Treat the configured value as a full replacement for the normal completed-bootstrap bare `/new` and `/reset` startup prompt, while preserving existing fallback behavior when unset and preserving BOOTSTRAP.md pending/limited safety behavior unless the implementation explicitly documents and tests a safe override policy.
