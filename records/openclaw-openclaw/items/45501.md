## Summary

Keep open. Current main and the latest release still build the bare `/new`/`/reset` startup prompt from fixed in-tree constants, and neither root `session` config nor `agents.list[]` exposes the requested `resetPrompt` override; duplicate reports now point back to this as the canonical tracker.

## What This Changes

The issue requests a global and per-agent config key for overriding the startup prompt injected after bare `/new` or `/reset`.

## Best Possible Solution

Define one typed reset-prompt resolver with maintainer-approved precedence, preserve built-in fallback and bootstrap-specific behavior, then update schema, exported types, help/docs, and focused reset prompt/Gateway/config tests together.

## Reproduction Assessment

Yes, source-level. Current main routes bare reset/new model-producing paths through `resolveBareSessionResetPromptState()` and strict root/per-agent schemas expose no requested reset-prompt override shape.

## Solution Assessment

Unclear until maintainers settle the public config contract. A full replacement prompt with current behavior as fallback looks narrow, but bootstrap-specific prompts and adjacent reset-greeting policy need deliberate handling.
