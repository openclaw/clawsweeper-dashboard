## Label Justifications

- `P2`: The missing control has concrete model and UI impact but remains a bounded feature needing product design rather than urgent remediation.
- `impact:session-state`: The hardcoded startup message becomes the first context of a reset session and can redirect its greeting and tool behavior.

## Summary

Current main and the latest release still hardcode the reset startup prompt, and the later duplicate was closed in favor of this canonical request. The feature remains necessary, but its exact shape needs maintainer judgment because unrestricted replacement could discard newer bootstrap and time-context safeguards.

## What This Changes

Add global and per-agent configuration to customize the startup prompt used after `/new` or `/reset`.

## Best Possible Solution

Keep mandatory bootstrap and time context core-owned, then define one canonical reset-start policy that exposes only the configurable greeting or supplemental instruction layer with clear global and per-agent precedence.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.5` still use fixed reset prompt variants, with focused tests asserting that behavior. The reported Ollama tool-call loop is credible environment evidence but was not independently run during this read-only review.

## Solution Assessment

No, not exactly as proposed: complete prompt replacement could remove current bootstrap safeguards. A bounded greeting or supplemental-text override within a unified reset-start policy is the safer maintainable design.
