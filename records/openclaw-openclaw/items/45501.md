## What This Changes

The issue requests a global and per-agent `session.resetPrompt` configuration option to override the hardcoded reset/startup prompt used around `/new` and `/reset`.

## Best Possible Solution

Maintain one reset-start policy where core keeps bootstrap and time safeguards, and if approved exposes the intended customizable greeting or supplemental-instruction layer with schema, docs, and focused reset-path tests.

## Reproduction Assessment

Yes at source level: `origin/main` and `v2026.6.6` still hardcode the reset startup prompt, and strict config schemas do not accept the requested key. I did not live-reproduce the Ollama-specific tool-call loop in this read-only review.

## Solution Assessment

No, not exactly as proposed: unrestricted prompt replacement risks dropping bootstrap and time-context safeguards now owned by the reset prompt builder. A bounded configurable layer under a maintainer-approved reset-start policy is the safer maintainable path.
