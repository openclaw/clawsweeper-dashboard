## Summary

Keep open. Current main still has a source-reproducible path where OpenAI-compatible transports emit the resolved `maxTokens` value directly as `max_tokens`/`max_completion_tokens`, while the prompt preflight only reserves the compaction reserve budget and does not reduce the requested output cap by the current prompt size.

## What This Changes

The issue asks OpenClaw to cap `max_tokens`/`max_completion_tokens` to the remaining context after already-used input tokens.

## Best Possible Solution

Add a shared prompt-aware output-budget clamp for OpenAI-compatible request construction so emitted token caps never exceed the remaining context, while preserving existing max-token precedence and field-name compatibility.

## Reproduction Assessment

Yes from source inspection: configure an OpenAI-compatible model with a 262144 context window and a requested output cap of 262144, accumulate a prompt around 43364 tokens, and current main will still emit 262144 as the completion cap. I did not run the live StepFun API call.

## Solution Assessment

Yes, the maintainable fix is a shared request-budget clamp in the OpenAI-compatible transport or immediately before it, not a StepFun-specific workaround or a new config option.
