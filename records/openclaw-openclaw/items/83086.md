## Summary

Keep open: current main still forwards the resolved OpenAI-compatible output-token cap directly to providers, and the reported release has the same behavior. The source path matches the StepFun error, so this is a valid narrow provider-runtime bug.

## What This Changes

The issue asks OpenClaw to subtract estimated prompt/input tokens before sending `max_tokens` or `max_completion_tokens` to OpenAI-compatible providers.

## Best Possible Solution

Clamp the outbound OpenAI-compatible generation-token field at request build time to the remaining estimated context budget while preserving explicit smaller caps and provider compatibility rules.

## Reproduction Assessment

Yes, at source level. A request built with a large prompt and `contextWindow`/`maxTokens` near 262144 still emits the full cap because the OpenAI-compatible request builder never subtracts estimated prompt tokens.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the request-construction path, not in model resolution or a StepFun-specific catalog workaround.

## Repair Work Prompt

Fix the OpenAI-compatible output-token budgeting bug reported at https://github.com/openclaw/openclaw/issues/83086. In the OpenAI completions request path, clamp the outbound `max_tokens` or `max_completion_tokens` value to the remaining estimated context budget: `min(resolved output cap, contextWindow - estimatedInputTokens)`, with a sensible floor/omit behavior when no positive output budget remains. Preserve existing provider compatibility field selection, explicit smaller caps, runtime `maxTokens` precedence, model params precedence, and zero/undefined semantics unless the existing behavior is demonstrably invalid. Do not make this StepFun-specific and do not add a config option. Add focused regression coverage for a 262144-token model with a large prompt so the payload no longer sends the full context window as completion budget, and update adjacent tests that currently assert direct forwarding.
