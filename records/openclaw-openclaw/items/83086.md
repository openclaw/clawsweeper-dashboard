## Summary

Keep open: current main still forwards the configured/model output token cap directly to OpenAI-compatible providers, and the StepFun failure described by the reporter maps to that source path. This is a valid, focused provider-runtime bug rather than stale cleanup.

## What This Changes

The issue reports that model requests set `max_tokens` or `max_completion_tokens` to the full configured output cap instead of the remaining context budget after input tokens are counted.

## Best Possible Solution

Clamp outbound OpenAI-compatible generation-token fields to the remaining estimated context budget at request construction time, while preserving existing provider field selection and configured smaller caps.

## Reproduction Assessment

Yes. A focused source-level reproduction can build an OpenAI-completions request with `contextWindow: 262144`, a large input context, and `maxTokens: 262144`; current main will still emit the full cap because the request builder never subtracts estimated input tokens.

## Solution Assessment

Yes. The maintainable fix is a narrow token-budgeting helper in the provider request path, not a provider-specific StepFun catalog change or a new config knob.
