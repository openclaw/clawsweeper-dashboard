## Label Justifications

- `P1`: The regression can make large-context provider sessions fail on every LLM call once prompt tokens accumulate.
- `impact:auth-provider`: The affected behavior is provider/model request construction for OpenAI-compatible routes.

## Summary

Keep open: current main still forwards the configured/model max-token cap directly in the OpenAI-compatible request path, and the linked fixing PR is open and unmerged. The report is source-reproducible and should close only after a corrected PR lands.

## What This Changes

The issue reports that OpenAI-compatible calls send max_tokens or max_completion_tokens without subtracting already-used input tokens from the model context window.

## Best Possible Solution

Clamp the outbound OpenAI-compatible max-token field to the remaining estimated context budget from the finalized request payload, preserve explicit smaller caps and no-cap semantics, and close this issue through the linked PR after merge.

## Reproduction Assessment

Yes, at source level. Build an OpenAI-completions request with a large prompt and a maxTokens value near the contextWindow; current main forwards the full cap without subtracting input tokens.

## Solution Assessment

Yes on direction, but not yet landed. The narrow maintainable fix belongs in OpenAI-compatible request budgeting, and the existing open PR should be reviewed or replaced before this issue closes.
