## Summary

Keep open: current main still forwards OpenAI-compatible output-token caps directly from runtime/model settings into the provider request, so the reported StepFun context overflow maps to a real provider-runtime bug rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to subtract estimated input tokens from a model's remaining context before sending `max_tokens` or `max_completion_tokens`.

## Best Possible Solution

Add a request-time token-budget clamp in the OpenAI-compatible transport path that preserves smaller explicit caps and provider-specific field selection while preventing `input + output cap` from exceeding the model context budget.

## Reproduction Assessment

Yes. A focused source-level reproduction can call `buildOpenAICompletionsParams` with a 262144-token context model, a large prompt context, and a full-context `maxTokens`; current main will emit the full cap because the request builder never subtracts estimated input tokens.

## Solution Assessment

Yes. The best fix is a narrow request-builder clamp based on estimated prompt/input tokens and context budget, not a StepFun-only catalog change or a new user-facing config knob.
