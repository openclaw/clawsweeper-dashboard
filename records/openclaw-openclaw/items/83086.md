## Label Justifications

- `impact:session-state`: The failure depends on accumulated conversation/context tokens and breaks long-running session progression.
- `impact:auth-provider`: The affected surface is provider/model request budgeting for OpenAI-compatible routes.

## Summary

Keep open: current main still forwards the OpenAI-compatible output-token cap without subtracting prompt/input tokens, and the linked fix PR is open but not merged. The issue should stay open until that PR lands or maintainers choose a better request-time budgeting fix.

## What This Changes

The issue asks OpenClaw to cap outbound max_tokens/max_completion_tokens to the remaining context budget after estimated input tokens are accounted for.

## Best Possible Solution

Clamp OpenAI-compatible generation-token fields at request-build time using the final serialized input estimate, preserve existing smaller caps, and land regression coverage plus redacted provider proof.

## Reproduction Assessment

Yes. A source-level reproduction can build an OpenAI-completions request with a large prompt and full context-window maxTokens; current main forwards the full cap without subtracting input tokens.

## Solution Assessment

Yes, the maintainable direction is request-time remaining-budget clamping in the provider request builder. The open PR is the active implementation candidate, but the issue should remain open until that fix is merged or replaced.
