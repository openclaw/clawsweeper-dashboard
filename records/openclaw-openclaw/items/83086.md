## Summary

Keep open: current main still forwards the resolved output-token cap to OpenAI-compatible request payloads without subtracting the estimated input tokens, so the reported provider rejection remains a valid focused runtime bug.

## What This Changes

The issue asks OpenClaw to cap outbound `max_tokens` / `max_completion_tokens` to the remaining context budget instead of sending the model's full output cap after prompt tokens are already used.

## Best Possible Solution

Clamp generation-token request fields to the remaining estimated context budget in the shared provider request path while preserving existing field selection and smaller configured caps.

## Reproduction Assessment

Yes, at source level: build an OpenAI-compatible request with a 262,144 context window, a large prompt, and a 262,144 output cap, and current main still emits the full cap without reserving prompt tokens. I did not run a live StepFun call in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is request-time token-budget clamping in the shared transport path, not a StepFun-specific catalog edit or a new user config knob.
