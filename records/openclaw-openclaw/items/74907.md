## Summary

Keep open. Current main still sends Anthropic OAuth/Claude CLI sessions through the Claude live stdio path with `--replay-user-messages` and per-turn stdin writes, while the direct Anthropic HTTP grouping and newer Claude history-import coalescing do not cover that live replay path.

## What This Changes

The issue reports a v2026.4.x regression where Claude/Anthropic multi-tool turns replay as assistant `tool_use` blocks without immediately adjacent matching `tool_result` blocks on follow-up messages.

## Best Possible Solution

Land a focused Claude CLI live-session replay fix that preserves strict tool-use/tool-result adjacency for retained multi-tool turns, or falls back to an OpenClaw-owned sanitized replay path when Claude's own replay cannot be trusted.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The issue provides a fresh-session reproduction, and current main still uses the implicated Claude live stdio route with only the current prompt written to stdin on follow-up turns.

## Solution Assessment

No patch is proposed yet. The best repair target is the Claude CLI live-session/replay boundary, not the direct Anthropic HTTP converter, because current main already groups direct HTTP `toolResult` records.
