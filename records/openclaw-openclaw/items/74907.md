## Summary

Keep open. The report remains current on `main`: direct Anthropic Messages conversion already coalesces consecutive split tool results, but the affected Claude CLI/OAuth path still uses the separate live `claude-stdio` session with `--replay-user-messages` and writes only the current user prompt to stdin on follow-up turns. Current tests cover warm-session reuse and the replay flag, but not a retained assistant turn with multiple tool calls plus split tool-result records.

## What This Changes

The issue reports a 2026.4.x regression where Claude CLI/OAuth sessions containing multi-tool assistant turns later replay into Anthropic orphan `tool_use` rejections.

## Best Possible Solution

Add focused regression coverage for Claude CLI live stdio replay after a parallel tool-call turn, then fix that boundary so every retained assistant tool call is replayed with all matching results adjacent/grouped or the assistant tool calls are pruned together with missing results. Keep the direct Anthropic HTTP converter behavior intact and check the GitHub Copilot Claude route for the same invariant.

## Reproduction Assessment

Yes. The reporter gives a high-confidence fresh-session path: start a Claude CLI OAuth session, trigger an assistant turn with multiple parallel tool calls, then send any follow-up user message and observe Anthropic rejecting orphan `tool_use` IDs.

## Solution Assessment

Unclear because there is no proposed patch. The narrowest maintainable fix appears to be in the Claude CLI live-session/stdin replay boundary, reusing shared replay helpers only where that preserves the already-correct direct Anthropic converter behavior.
