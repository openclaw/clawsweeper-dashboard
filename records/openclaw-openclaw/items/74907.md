## Summary

Keep open. The report is current and source-narrowed: current main still routes bundled Anthropic OAuth Claude CLI runs through the live `claude-stdio` process with `--replay-user-messages`, while writing only the current follow-up prompt to stdin; the direct Anthropic HTTP converter already groups split tool results, but the discussion and source both show that is not the failing route.

## What This Changes

The issue asks OpenClaw to preserve Anthropic tool-use/tool-result adjacency when replaying sessions that contain assistant turns with multiple parallel tool calls, especially through the Claude CLI live-session path after follow-up user messages.

## Best Possible Solution

Add a focused regression for the Claude CLI live stdio replay path after a parallel tool-call turn, then fix that boundary so a retained assistant tool-call message is replayed with all matching results adjacent/grouped or the assistant tool calls are pruned together with missing results. Keep the direct Anthropic HTTP converter's existing grouping behavior intact and check the Copilot Claude route only if the same invariant is observable there.

## Reproduction Assessment

Yes. The issue discussion gives a high-confidence fresh-session path: use `anthropic:claude-cli`, trigger one assistant turn with multiple parallel tool calls, then send any follow-up user message and observe Anthropic rejecting orphan `tool_use` ids; current source confirms that route uses the live Claude CLI stdin replay boundary.

## Solution Assessment

Unclear because there is no patch yet. The narrowest maintainable direction is not to change the already-correct direct Anthropic converter first, but to add coverage and repair the Claude CLI live-session/stdin replay or disable that live replay mode when it cannot preserve strict tool-result adjacency.

## Repair Work Prompt

Fix the Claude CLI live-session replay regression reported in https://github.com/openclaw/openclaw/issues/74907. The observable bug is: with `anthropic:claude-cli` / OAuth routed through `claude-stdio`, a fresh session that has an assistant turn with multiple parallel tool calls can fail on the next user message with Anthropic orphan `tool_use` ids. Do not start by changing `convertAnthropicMessages`; current `src/agents/anthropic-transport-stream.ts` already groups consecutive `toolResult` messages for the direct HTTP path. Add a focused regression around `src/agents/cli-runner/claude-live-session.ts` / `src/agents/cli-runner.spawn.test.ts` that models a retained assistant multi-tool turn plus split tool results before a follow-up user message. Fix the Claude CLI live stdio/replay boundary so every retained assistant tool call is replayed with all matching results adjacent/grouped, or close/restart/fallback from live replay when that invariant cannot be guaranteed. Keep third-party CLI backend behavior compatible and avoid broad changes to generic transcript repair unless the helper can be reused without altering direct Anthropic HTTP behavior. Check whether the GitHub Copilot Claude report shares the same invariant before widening the fix. Add a user-facing `CHANGELOG.md` Fixes entry crediting @wiziswiz, and credit @Marvae only if the fix explicitly covers their Copilot confirmation.
