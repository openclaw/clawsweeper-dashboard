## Summary

Keep open. The reporter's fresh-session repro and the current source both point at the Claude CLI live-session replay path, not the direct Anthropic HTTP converter; main still defaults `claude-cli` to stdio replay and writes only the current prompt on each reused live turn, without focused coverage for a retained multi-tool assistant turn followed by split tool results.

## What This Changes

The issue reports that 2026.4.x Claude CLI/OAuth replay can resend assistant turns with multiple `tool_use` blocks without adjacent matching `tool_result` blocks after a follow-up user message.

## Best Possible Solution

Add a focused Claude CLI live-session replay regression and repair the stdio replay boundary so retained assistant tool calls are always paired with all matching results or pruned symmetrically before Claude/Anthropic can reject the follow-up turn.

## Reproduction Assessment

Yes. The issue discussion gives a high-confidence reproduction using an `anthropic:claude-cli` OAuth session: create a fresh session, trigger one assistant turn with parallel tool calls, then send any follow-up user message and observe the orphan `tool_use` rejection.

## Solution Assessment

Unclear because there is no proposed patch. The narrowest maintainable direction is to fix the Claude CLI live-session/replay boundary while keeping the already-correct direct Anthropic converter and existing compaction pairing safeguards intact.
