## Summary

Keep open. The report is current, detailed, and still matches current main: the direct Anthropic Messages converter already coalesces split tool results, but the affected Claude CLI live-session path still uses stdio replay and sends each follow-up as one user stream-json message, with no targeted replay repair visible after the latest release.

## What This Changes

The issue asks OpenClaw to fix Claude/Anthropic replay after parallel tool-call turns so assistant tool_use blocks are replayed only with all matching tool_result blocks adjacent/grouped, especially on the Claude CLI OAuth path.

## Best Possible Solution

Add a focused regression for Claude CLI live stdio replay after a parallel tool-call turn, then fix that boundary so retained assistant tool calls are replayed only with all matching results adjacent/grouped or are pruned symmetrically. Keep the direct Anthropic HTTP converter’s existing grouping behavior intact and check whether the same invariant applies to GitHub Copilot Claude replay.

## Reproduction Assessment

Yes. The reporter provides a fresh-session reproduction on 2026.4.x, and current main still has the Claude CLI live-session route that reuses stdio with --replay-user-messages and writes only the next user prompt for follow-up turns.

## Solution Assessment

Unclear until the regression is written, but the narrowest maintainable fix is likely in the Claude CLI live-session/replay boundary or a shared replay sanitizer invoked before that boundary, not in convertAnthropicMessages, which already groups consecutive toolResult records.
