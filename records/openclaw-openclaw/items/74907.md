## Summary

Keep open. Current main still routes Claude CLI/OAuth follow-up turns through warm `claude-stdio` with `--replay-user-messages` and writes only the next user prompt, while the already-correct direct Anthropic HTTP converter and generic replay repair do not cover the live CLI replay path called out in the discussion.

## What This Changes

The issue asks OpenClaw to stop Anthropic/Claude multi-tool history replay from producing orphan `tool_use` blocks on follow-up turns after split `toolResult` records.

## Best Possible Solution

Add a focused Claude CLI live-session replay fix and regression coverage that preserves strict adjacency between retained multi-tool assistant turns and all matching tool results, or avoids Claude CLI replay when OpenClaw cannot guarantee that shape.

## Reproduction Assessment

Yes. The high-confidence path is `/new`, induce one `anthropic:claude-cli` assistant turn with multiple tool calls, then send any follow-up user message; current source still uses the implicated live stdio replay route.

## Solution Assessment

No patch is proposed yet. The narrowest maintainable fix belongs in the Claude CLI live-session replay/resume boundary, not in the direct Anthropic HTTP converter that already groups consecutive tool results.
