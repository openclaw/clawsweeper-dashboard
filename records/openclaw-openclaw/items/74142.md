## Summary

Keep open. The reported bug is valid and current main still lacks full empty-store channel-plugin handling, but this PR does not fully fix the linked repro because its fallback still depends on finding the requester key in the session store; it also overlaps the more complete open #72306 branch and lacks the required changelog entry.

## What This Changes

The PR adds a `session_status` fallback that rewrites literal `sessionKey="current"` to the effective requester key for channel-style clients and adds a stored-requester regression test.

## Best Possible Solution

Land one canonical implementation that preserves literal `current` key/sessionId precedence, handles channel-plugin requester sessions even when no store row exists, keeps visibility self-scoped, adds focused regression coverage, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. A channel-style requester with an empty session store and explicit `sessionKey="current"` can be traced through current main to the `Unknown sessionKey: current` throw, matching the linked #74141 log slice.

## Solution Assessment

No. This PR is narrow, but it only covers the stored-requester case; #72306 or an equivalent final fallback that can synthesize requester-scoped status after all literal `current` precedence paths miss is safer.
