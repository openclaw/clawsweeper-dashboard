## Summary

Keep open for maintainer review. Current main already handles the stored-requester case this branch tests, while the linked live log shows an unstored requester-key failure that this PR still appears to miss; the overlapping open PR #72306 is the more complete candidate for the remaining behavior.

## What This Changes

The PR adds a second `sessionKey="current"` fallback in `session-status-tool.ts` for channel-style requesters and adds a stored channel-plugin requester regression test.

## Best Possible Solution

Land one canonical fix that preserves literal `current` key/sessionId precedence and existing visibility checks, while explicitly handling the sparse channel-plugin requester case, then retire the overlapping branch.

## Reproduction Assessment

Yes. A channel-style requester with an empty session store and explicit `sessionKey="current"` can be traced through current main to the unknown-session path, and #74141 provides concrete Scope-channel logs for the same class of failure.

## Solution Assessment

No. This PR is not the best fix because its fallback still depends on finding the requester key in the store; the safer direction is #72306 or an equivalent requester-scoped fallback after all literal `current` precedence paths miss.
