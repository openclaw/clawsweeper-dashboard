## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Exclude Wear identity files from backup:** `apps/android/wear/src/main/res/xml/backup_rules.xml:3`
  - body: The Wear manifest enables backup and these rules include the whole `file` domain, while `GatewayDeviceIdentityStore` writes `openclaw/identity/device.json` under `context.filesDir` with `privateKeyPkcs8Base64`. That makes the watch's gateway identity/private key eligible for cloud backup and device transfer; exclude the identity/auth paths or move them to no-backup storage before merge.
  - confidence: 0.92
- **[P2] Build and test the Wear module in Android release lanes:** `apps/android/settings.gradle.kts:22`
  - body: Adding `:wear` to the Gradle project does not make the existing Android CI build or test it: the current workflow only runs `:app:testPlayDebugUnitTest`, `:app:testThirdPartyDebugUnitTest`, and `:app:assemblePlayDebug`, and `:app` does not depend on `:wear`. A Wear app MVP should add explicit Wear unit/build/release coverage so the new artifact is actually validated and shippable.
  - confidence: 0.86
- **[P2] Clear bootstrap auth when password credentials change:** `apps/android/wear/src/main/java/ai/openclaw/wear/ui/ConnectionSettingsScreen.kt:51`
  - body: The password editor keeps any existing `bootstrapToken` in `WearGatewayConfig`, but `buildWearConnectParams` tries `bootstrapToken` before `password` when `token` is blank. A watch that previously received bootstrap fallback config can ignore the newly entered password and keep failing with stale bootstrap auth; clear bootstrap auth when the user edits direct token/password credentials.
  - confidence: 0.84
