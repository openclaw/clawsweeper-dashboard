- **[P1] Exclude Wear secrets from backup:** `apps/android/wear/src/main/AndroidManifest.xml:10-12`
  - body: The Wear app enables backup and includes all app files, while the new shared identity store writes an Ed25519 private key under `filesDir/openclaw/identity/device.json` and the Wear config store persists gateway credentials. This should disable backup for the Wear app or narrowly exclude identity/config/secure-pref files before merge.
  - confidence: 0.93
- **[P1] Wire the Wear module into Android gates:** `apps/android/settings.gradle.kts:20`
  - body: Adding `:wear` to Gradle is not enough for review safety because the existing Android CI and CodeQL workflows still run only `:app` tasks. A broken or unanalyzed Wear app could merge green unless the PR adds Wear build/test/security coverage or a deliberate release-gate alternative.
  - confidence: 0.89
- **[P2] Clear bootstrap auth when password changes:** `apps/android/wear/src/main/java/ai/openclaw/wear/ui/ConnectionSettingsScreen.kt:50-52`
  - body: The password editor preserves any synced `bootstrapToken`, but `buildWearConnectParams` prefers `bootstrapToken` over `password`. A user switching direct mode to password auth can keep sending stale bootstrap auth with no field in this screen to clear it.
  - confidence: 0.86
- **[P2] Use shared Android versioning for Wear:** `apps/android/wear/build.gradle.kts:17-18`
  - body: The Wear APK shares the phone app's application ID for Data Layer pairing but hardcodes its own versionCode/versionName. That can drift from OpenClaw's Android release train and produce confusing update/package metadata, so it should use the same `Config/Version.properties` source or an explicitly maintained Wear release scheme.
  - confidence: 0.82
