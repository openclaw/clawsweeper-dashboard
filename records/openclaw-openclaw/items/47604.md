## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Exclude Wear identity files from backups:** `apps/android/wear/src/main/res/xml/backup_rules.xml:3`
  - body: The Wear manifest enables backup and points both backup rule files at the entire `file` domain, while `GatewayDeviceIdentityStore` writes `openclaw/identity/device.json` with `privateKeyPkcs8Base64` under `filesDir`. That makes the watch's gateway identity/private key eligible for cloud backup and device transfer; exclude the identity/auth paths or move this material to a non-backed-up/keystore-owned location before merge.
  - confidence: 0.9
- **[P2] Wire the Wear app into Android release paths:** `apps/android/wear/build.gradle.kts:17-18`
  - body: The PR adds a shipping-looking `:wear` application with its own fixed `versionCode`/`versionName`, but the existing Android scripts, docs, and CI only test/build/bundle `:app`. As merged, Wear would not be versioned, signed, release-bundled, or covered by the Android CI lane; add explicit Wear release/CI tasks or mark it as non-shipping.
  - confidence: 0.84
- **[P2] Clear bootstrap auth when editing direct password:** `apps/android/wear/src/main/java/ai/openclaw/wear/ui/ConnectionSettingsScreen.kt:50-51`
  - body: Editing the direct password preserves any synced `bootstrapToken`, and `buildWearConnectParams` chooses `bootstrapToken` before `password` when no token is set. A user switching from fallback bootstrap auth to password auth can keep sending the stale bootstrap credential with no UI control to clear it; clear bootstrap auth on token/password edits or make the auth choice explicit.
  - confidence: 0.86

## Security Review

Summary: The diff introduces watch-side auth and identity storage, and one concrete backup rule issue needs attention before merge.
