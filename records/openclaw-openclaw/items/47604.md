## Summary

Keep this PR open. Current main still has no Wear OS app or phone-side Wear proxy, and the contributor provided useful device-tested implementation context, but the proposed patch is not merge-ready because the Wear manifest and backup rules can export the device-auth private key.

## What This Changes

The PR branch adds Android `:shared` and `:wear` modules, a Wear OS companion app, phone-side Wear Data Layer proxying, direct watch gateway access, tile/complication/notification support, Android/Wear tests, and related docs.

## Best Possible Solution

Keep the PR open while maintainers decide whether Wear OS belongs in-tree. If it does, first disable Wear backup or exclude identity/auth files, then continue with staged review around shared gateway helpers, phone proxying, direct watch auth, Wear UI, notifications, tile/complication services, docs, tests, and fresh device proof.

## Reproduction Assessment

Yes for the blocking review finding: source inspection of the PR head shows `allowBackup=true` plus include-all file backup rules, and the shared identity store writes the private signing key under `files/openclaw/identity/device.json`. Full functional Wear behavior still needs paired phone/watch validation rather than source-only reproduction.

## Solution Assessment

No, merging this PR as-is is not the best way to solve the Wear OS feature. The safer path is to fix the backup/private-key exposure first and then land or replace the work in reviewable stages with Android security and device proof.
