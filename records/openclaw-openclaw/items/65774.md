## Summary

Keep open: current main partially mitigates stale persisted cron runs and non-main cron task cancellation, but the reported safety problem still spans missed startup replay, main-session/in-flight cancellation, and an operator-visible emergency stop path.

## What This Changes

The issue reports WhatsApp cron jobs sending outside their configured schedule after gateway reconnect and lacking a reliable stop/delete path for active outbound work.

## Best Possible Solution

Define a maintainer-approved cron/gateway safety policy that gates missed startup replay, fences or cancels queued and active cron delivery on delete/stop, and proves the chosen behavior with focused cron tests plus channel-delivery proof.

## Reproduction Assessment

Yes for a source-level current-main reproduction, but not as a live WhatsApp repro. The scheduler still has a missed-startup replay path without an explicit safe-send window policy, and current tests document that main-session cron rows have no active cancellation handle.

## Solution Assessment

Unclear. The issue is valid, but the best fix requires maintainer choices about suppressing, deferring, or replaying missed jobs and about how broad delete, task cancel, gateway stop, and emergency outbound stop guarantees should be.
