## Summary

Keep open: current main has useful mitigations for interrupted startup jobs and stale direct delivery, but the source still supports the remaining stop/delete and replay-policy gaps described in the issue.

## What This Changes

The issue asks OpenClaw to prevent out-of-hours WhatsApp cron replay and provide reliable stop/delete or emergency halt behavior for active outbound cron sends.

## Best Possible Solution

Define one operator-safe policy: late cron replays must not surprise external contacts, cron-owned active work should be cancellable where feasible, and an emergency outbound halt should explicitly block new sends.

## Reproduction Assessment

Yes, at source level: seed an overdue cron job or persisted running cron state, then inspect startup catch-up, remove, and gateway shutdown paths on current main. I did not live-run the original Windows WhatsApp incident.

## Solution Assessment

No patch is present, and a narrow startup catch-up tweak alone would not cover the stop/delete and emergency-halt expectations. The maintainable path needs an explicit scheduler replay and outbound cancellation policy with regression coverage.
