## Summary

Keep open: current main and v2026.6.8 still retain the unsafe startup catch-up replay shape, and the current cancellation work only partially covers active cron runs. The open schedule-activation PR is a credible fix path for the replay subset, but cron remove, main-session cron cancellation, gateway stop, and emergency outbound-stop semantics still need maintainer judgment.

## What This Changes

A user reports WhatsApp cron jobs firing at 1 AM outside their configured hours after gateway reconnect, with cron delete and gateway stop failing to halt active sends.

## Best Possible Solution

Land a maintainer-approved activation/window-boundary fix for startup replay, then define and document how cron delete, task cancellation, gateway stop, and any emergency stop halt active outbound sends.

## Reproduction Assessment

Yes, at source level on current main: startup catch-up can still select overdue cron jobs from persisted run state without a schedule activation/current-window guard, and cron.remove does not abort active execution. I did not run a live WhatsApp reproduction.

## Solution Assessment

Unclear for the whole issue. The open schedule-activation PR appears to be the best path for the replay subset, but the full request also needs maintainer decisions for delete cancellation, gateway stop guarantees, and an emergency outbound-stop contract.
