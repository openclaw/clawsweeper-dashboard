## Summary

Keep open: current main has useful mitigations for interrupted startup jobs, stale cron delivery, and reconnect duplicate sends, but the active-run stop/delete path is still source-backed as incomplete and the remaining replay policy needs maintainer judgment.

## What This Changes

The issue requests that WhatsApp-backed cron jobs respect configured schedule windows after reconnect and that cron delete/gateway stop immediately halt active sends.

## Best Possible Solution

Define one operator-safe policy: late replays must not surprise external contacts, cron-owned active work should be cancellable by delete/stop where feasible, and any emergency outbound halt should be explicit and documented.

## Reproduction Assessment

Yes, at source level: seed overdue or previously running cron state and inspect startup catch-up, remove, and gateway shutdown paths on current main. I did not live-run the original Windows WhatsApp incident, but the code and tests clearly show replay and non-aborting stop/delete behavior.

## Solution Assessment

No proposed patch is present, and a narrow catch-up tweak alone would not satisfy the stop/delete/emergency expectations. The best fix needs an explicit scheduler and outbound-halt policy with focused regression coverage.
