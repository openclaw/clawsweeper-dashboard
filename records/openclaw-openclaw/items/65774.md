## Summary

Keep open: current main has shipped partial protections for interrupted cron startup replay and stale direct delivery, but the stop/delete emergency behavior remains source-backed as incomplete and the remaining replay policy still needs maintainer judgment.

## What This Changes

The issue reports WhatsApp cron jobs replaying outside their configured hours after Gateway reconnect and continuing despite Gateway stop, process kills, and cron deletion attempts.

## Best Possible Solution

Define one operator-safe policy across cron replay, active-run cancellation, and outbound delivery: late replays must not surprise external contacts, deletion/stop should cancel cron-owned work where possible, and any emergency halt should be explicit and documented.

## Reproduction Assessment

Yes, at source level: seed overdue or previously running cron state and inspect startup catch-up/remove/stop paths on current main. The exact WhatsApp incident was not live-run, but the code and tests show the relevant replay and non-aborting stop/delete behavior.

## Solution Assessment

No proposed patch is present, and current main is only a partial solution. The maintainable fix needs an explicit scheduler replay and outbound-halt policy rather than another isolated startup catch-up tweak.
