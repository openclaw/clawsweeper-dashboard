## Summary

Keep open: current main partially handles stale persisted running markers, but it still does not fully cover the reported safety problem around after-hours startup catch-up and emergency cancellation of active outbound cron work.

## What This Changes

The issue asks for cron startup catch-up, cron deletion, and gateway stop/emergency controls to prevent scheduled WhatsApp messages from sending outside configured hours or continuing after cancellation.

## Best Possible Solution

Keep this open for a maintainer-owned cron/gateway design that makes startup catch-up respect an explicit safe-send policy and provides an operator-visible way to cancel active outbound cron delivery across session targets.

## Reproduction Assessment

Yes for source-level current-main paths, but not as a live WhatsApp repro. The scheduler still permits startup replay from missed cron history without checking that the current time is inside the cron window, and current cancellation tests show main-session cron runs have no active cancellation handle.

## Solution Assessment

Unclear as an implementation because the right behavior is product-sensitive. A good fix needs maintainers to choose whether missed jobs should be suppressed, deferred until the next allowed slot, or made configurable, and how broad the emergency stop should be.
