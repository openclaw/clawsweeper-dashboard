## Summary

Keep this open. Current main still has source-backed scheduler replay and cancellation gaps matching the report: overdue cron work can be replayed from persisted due or missed-slot state without a contact-window guard, cron removal/stop does not own an active-run abort registry, and Gateway shutdown stops schedulers/channels rather than exposing an emergency outbound halt.

## What This Changes

The issue requests that cron catch-up, cron deletion, Gateway stop, and emergency outbound controls prevent WhatsApp cron messages from sending outside configured contact hours.

## Best Possible Solution

Design a core operator-safe scheduler path: startup catch-up should not replay work outside the configured delivery/contact window, cron remove/disable/stop should abort matching active executions when possible, and Gateway operators should have a documented emergency outbound halt that blocks new sends and cancels or drains in-flight delivery safely.

## Reproduction Assessment

Yes for the core cron replay path: seed a recurring cron job with a past `nextRunAtMs` or missed slot and start the cron service; current tests already assert immediate replay. The full WhatsApp 1 AM incident was not live-reproduced, but the scheduler and cancellation gaps are source-backed.

## Solution Assessment

No proposed patch is present, and current main is not the best solution yet. The existing startup deferral and timeout-abort work are useful mitigations, but they do not define contact-window replay policy, deletion/stop abort semantics, or an emergency outbound halt.
