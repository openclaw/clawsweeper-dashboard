## Summary

Keep this open. Current main still replays overdue cron work from persisted due/missed-slot state without an operator-facing contact-window guard, cron removal/stop still does not own a service-level active-run abort registry, and gateway shutdown still stops services/channels rather than providing an emergency outbound halt.

## What This Changes

The issue requests that WhatsApp cron jobs respect configured daytime schedules after reconnect, that deleting/stopping cron jobs abort active sends, and that operators have an emergency stop for outbound message delivery.

## Best Possible Solution

Keep this open and design a core scheduler/operator fix: startup catch-up should have an explicit replay policy that cannot violate contact windows, cron remove/disable/stop should abort matching active executions where possible, and gateway operators should have a documented emergency outbound halt that blocks new sends and drains or cancels in-flight delivery safely.

## Reproduction Assessment

Yes for the cron core replay path: seed a recurring cron job with a past nextRunAtMs or a missed slot and start the cron service; current tests already assert immediate replay. The full WhatsApp business-impact path was not live-reproduced in this read-only review, but the scheduler and stop/cancel gaps are source-backed.

## Solution Assessment

No proposed patch is present, and current main is not the best solution yet. The adjacent startup sidecar deferral and timeout cleanup changes are useful but do not define contact-window replay policy, active-run abort semantics, or an emergency outbound halt.
