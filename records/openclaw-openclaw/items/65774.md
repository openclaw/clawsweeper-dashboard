## Summary

Keep this open: current main still has source-backed gaps matching the report, especially startup replay of overdue cron work without a configured-hours/contact-window guard and no cron-owned abort path for removal or gateway stop.

## What This Changes

Requests that cron catch-up respect the configured WhatsApp sending window, that deleting/stopping cron work cancel active sends, and that operators have an emergency outbound halt.

## Best Possible Solution

Define an operator-safe scheduler policy: missed cron replay must not violate configured delivery windows, cron remove/disable/stop should cancel owned active executions where possible, and Gateway should expose a documented emergency outbound halt.

## Reproduction Assessment

Yes for the core scheduler path: seed a recurring cron job with a past `nextRunAtMs` or missed slot and start cron; current source and tests show startup catch-up replays it. The exact WhatsApp production incident was not live-reproduced.

## Solution Assessment

No proposed patch is present, and current main is not the best solution yet. Delaying startup replay and timeout-aborting individual runs are useful mitigations, but they do not cover contact-window replay policy, deletion/stop abort semantics, or an emergency outbound halt.
