## Label Justifications

- `P1`: The report describes a broken real-user channel workflow where external WhatsApp messages were sent outside the configured schedule and could not be halted through operator commands.
- `impact:message-loss`: The core impact is mis-timed outbound channel delivery to external contacts.
- `impact:session-state`: The report and current code path involve stale persisted cron running/next-run state across Gateway restart or reconnect.

## Summary

Keep open: current main has partial mitigations for interrupted startup runs and stale direct delivery, but startup replay still intentionally runs overdue jobs and stop/delete still do not abort active cron execution.

## What This Changes

The issue requests that WhatsApp cron jobs respect configured hours after gateway reconnect, stop/delete abort active sends, and provide an emergency halt for outbound delivery.

## Best Possible Solution

Define one operator-safe policy: late replays must not surprise external contacts, cron-owned active work should be cancellable by delete/stop where feasible, and any emergency outbound halt should be explicit, documented, and covered by focused regression tests.

## Reproduction Assessment

Yes at source level: seed overdue or active cron state and inspect startup catch-up, remove, and Gateway shutdown paths on current main. I did not live-run the original Windows WhatsApp incident, but the code and tests clearly show replay behavior and non-aborting stop/delete behavior.

## Solution Assessment

No proposed patch is present, and a narrow catch-up tweak alone would not satisfy the reported stop/delete/emergency expectations. The best fix needs an explicit scheduler and outbound-halt policy with focused regression coverage.
