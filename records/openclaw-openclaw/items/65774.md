## Summary

Keep open. Current main fixes the stale running-lock replay path and adds task-level cron cancellation, but `cron remove/delete` still does not abort in-flight runs and missed-slot replay outside the current wall-clock schedule remains unresolved.

## What This Changes

The issue requests that cron jobs not send WhatsApp messages outside configured schedule windows after reconnect and that operators can immediately stop or delete active scheduled sends.

## Best Possible Solution

Keep the interrupted-run protection, then define one canonical operator stop path and startup catch-up rule: `cron remove/delete` should abort live cron sends, and missed cron windows should skip or defer according to an explicit documented policy.

## Reproduction Assessment

Yes at source level for the remaining gaps: `cron.remove` only removes stored jobs, and startup catch-up still treats due or missed cron slots as runnable without a wall-clock active-window guard.

## Solution Assessment

Unclear until maintainers choose catch-up semantics; wiring removal and stop commands into active cron abort handles is narrow, but skipping missed windows versus replaying them is a product decision.
