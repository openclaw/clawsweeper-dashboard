## Summary

Keep open: current main still has source-backed gaps matching the report, including startup replay of overdue cron slots without a configured-hours guard and no cron-owned abort path for remove or gateway stop.

## What This Changes

The issue asks cron-triggered WhatsApp sends to respect configured hours after gateway reconnects and to provide an immediate way to stop or delete active sends.

## Best Possible Solution

Define an operator-safe scheduler policy: missed replay must not violate configured delivery windows, cron remove/disable/stop should cancel owned active work where possible, and Gateway should expose a documented emergency outbound halt.

## Reproduction Assessment

Yes for the core scheduler path: seed a recurring cron job with a past nextRunAtMs or missed slot and start cron; current source and tests show startup catch-up replays it. The exact WhatsApp production incident was not live-reproduced.

## Solution Assessment

No proposed patch is present, and current main is not the best solution yet. Delaying startup replay and timeout-aborting individual runs are useful pieces, but the final fix needs a policy decision across cron replay, active-run cancellation, gateway stop semantics, and outbound delivery recovery.
