## Summary

Keep open: current main mitigates one stale-running startup replay path, but the reported safety problem still has remaining source-level gaps around off-window missed cron replay and operator cancellation of active outbound cron work.

## What This Changes

The issue reports WhatsApp cron jobs replaying after-hours after gateway reconnect and lacking an effective stop/delete path for active sends.

## Best Possible Solution

Define a maintainer-owned cron safety policy that makes startup catch-up respect safe-send windows and exposes an operator-visible cancellation path for active outbound cron delivery across supported session targets.

## Reproduction Assessment

Yes for a source-level current-main path, but not as a live WhatsApp reproduction. The scheduler still permits missed startup replay from cron history without a current-window gate, and removal/cancellation is not a complete emergency stop surface.

## Solution Assessment

Unclear as an implementation because the right behavior is product-sensitive. Maintainers should choose whether missed jobs are suppressed, deferred to the next allowed slot, or made configurable, and how broad active outbound cancellation must be.
