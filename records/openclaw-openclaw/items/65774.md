## Summary

Keep open: current main still has source-backed gaps matching the report, and the related PRs only address adjacent startup/stale-lock pieces rather than the out-of-hours replay plus stop/delete behavior.

## What This Changes

The issue asks OpenClaw to prevent missed WhatsApp cron jobs from replaying outside configured hours and to make cron delete or gateway stop halt active sends.

## Best Possible Solution

Define an operator-safe scheduler and outbound stop policy: missed replay must not violate configured delivery windows, delete/stop should cancel active cron-owned work where possible, and Gateway should expose a documented emergency outbound halt.

## Reproduction Assessment

Yes for the core scheduler path: seed a recurring cron job with a past nextRunAtMs or missed slot and start cron; current source and restart catch-up tests show startup replay. The exact WhatsApp production incident was not live-reproduced in this read-only review.

## Solution Assessment

No proposed patch is present, and current main is not the best solution yet. A safe fix needs a policy decision across cron replay semantics, active-run cancellation, gateway lifecycle, and channel outbound recovery.
