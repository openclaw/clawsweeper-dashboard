## Summary

Keep open: current main and the latest release still leave the central cron safety problem unresolved. Startup catch-up can still replay overdue cron work based on stale `nextRunAtMs`, `cron.remove` still only removes stored jobs, and main-session cron runs still lack the active cancellation handle that non-main cron tasks now have.

## What This Changes

The issue reports WhatsApp cron jobs replaying outside configured local hours after gateway reconnect and lacking a reliable stop, delete, or emergency kill path.

## Best Possible Solution

Define a maintainer-approved cron safety contract for missed replay, delete/task cancellation, gateway stop, and emergency outbound halt behavior, then implement it with focused cron, task, and channel-delivery proof.

## Reproduction Assessment

Yes, at source level: current main still admits overdue cron jobs through startup catch-up and has a documented main-session cancellation gap. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

Unclear as an implementation path. The issue is valid, but maintainers need to choose whether missed jobs should be suppressed, deferred, replayed with a window, or controlled through an explicit emergency-stop contract.
