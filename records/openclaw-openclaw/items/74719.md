## What This Changes

The PR adds runtime guards around reply payload text normalization, chat send message sanitization, chat abort stop-command parsing, and TUI stream text trimming.

## Best Possible Solution

Land a narrow rebased fix that narrows `payload.text` before silent-token handling, preserves attachment-only `chat.send`, and adds focused regression coverage.

## Reproduction Assessment

Yes for source reproduction: current main can pass a truthy non-string `payload.text` through `normalizeReplyPayload` when other payload content keeps it non-empty, then call a helper that trims it. I did not run a live gateway or EasyClaw reproduction in this read-only review.

## Solution Assessment

No. The normalizer guard is directionally right, but the branch must be rebased onto the current `hasContent()` and `normalizeOptionalString` code and must not reject valid attachment-only `chat.send` requests.
