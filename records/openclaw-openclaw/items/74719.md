## Summary

Keep open. The reported runtime crash still has a clear current-main source path, but this PR is dirty against main and introduces an attachment-only `chat.send` regression, so the useful normalizer idea needs a focused repair or replacement before merge.

## What This Changes

The PR adds defensive runtime guards around reply payload text normalization, `chat.send` sanitization, chat abort text parsing, and TUI stream text trimming.

## Best Possible Solution

Land a narrow rebased fix that normalizes reply payload text once before token helpers, preserves attachment-only `chat.send`, adds regression coverage, and includes the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `normalizeReplyPayload` with truthy non-string `text` plus media or channel data, which keeps the payload non-empty and then reaches token helpers that assume a string.

## Solution Assessment

No. The normalizer guard is directionally right, but this PR is stale against current main and the `chat.send` sanitizer change rejects valid attachment-only sends; the safer fix is a narrow rebased guard plus tests.
