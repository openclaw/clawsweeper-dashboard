## Summary

Keep open. The runtime text crash is still source-reproducible on current main, but this PR is stale against the current reply normalizer and would also reject supported attachment-only chat sends.

## What This Changes

The PR adds runtime string guards around reply payload normalization, `chat.send` sanitization/trim handling, chat abort stop-command parsing, and TUI stream text trimming.

## Best Possible Solution

Land a rebased fix that narrows reply payload text before string helpers while preserving existing attachment-only `chat.send` behavior and adding regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `normalizeReplyPayload` with truthy non-string `text` plus media or channel data, which keeps the payload non-empty and then reaches token helpers that call `text.trim()`.

## Solution Assessment

No. The normalizer guard is directionally right, but this branch is stale against current main and the `chat.send` sanitizer change rejects valid attachment-only sends; a narrow rebased guard plus tests is safer.
