## Summary

Keep open. The PR targets a real remaining reply-payload runtime gap, but it should not merge as-is because the chat.send path is already schema-guarded for missing/non-string messages, the sanitizer change would reject supported attachment-only sends, and the reply normalizer hunk needs to be reapplied to current main.

## What This Changes

The PR adds defensive runtime guards around message/text trimming in chat send, chat abort command parsing, reply payload normalization, and TUI stream assembly.

## Best Possible Solution

Port only the useful runtime text narrowing into current main: derive one typed optional string from payload.text before all silent-token, heartbeat, prefix, and sanitization helpers, preserve media-only and channelData-only replies, and leave chat.send's schema validation plus attachment-aware empty-message gate intact.

## Reproduction Assessment

Yes for the remaining reply gap: pass a runtime ReplyPayload with media or channelData plus a truthy non-string text through normalizeReplyPayload, and current main can let that raw value reach string-only helpers. The undefined chat.send message path is not reproducible through the RPC handler because validateChatSendParams rejects it before sanitization.

## Solution Assessment

No, this PR is not the best way to solve the issue as written. The safer fix is a current-main normalizer change with regression coverage, without moving blank-message rejection into sanitizeChatSendMessageInput or adding redundant optional chaining after existing typeof checks.
