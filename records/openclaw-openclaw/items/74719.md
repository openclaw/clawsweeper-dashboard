## Summary

Keep open, but do not merge as-is: the reported reply normalizer crash is still source-reproducible on current main, while this PR would break attachment-only chat sends, targets an older normalizer shape, and lacks real after-fix behavior proof.

## What This Changes

The PR adds runtime guards around reply payload text, chat.send message sanitization, chat abort stop-command parsing, and TUI stream text trimming.

## Best Possible Solution

Land a narrow rebased fix that narrows `payload.text` before silent-token handling, preserves attachment-only `chat.send`, and adds focused regression coverage.

## Reproduction Assessment

Yes for source reproduction: current main can pass a truthy non-string `payload.text` through `normalizeReplyPayload` when other payload content keeps it non-empty, then call a helper that trims it. I did not run a live EasyClaw, gateway, or IPC reproduction in this read-only review.

## Solution Assessment

No. The normalizer guard is directionally right, but this PR must be rebased onto the current `hasReplyPayloadContent`/`normalizeOptionalString` code and must not reject valid attachment-only `chat.send` requests.
