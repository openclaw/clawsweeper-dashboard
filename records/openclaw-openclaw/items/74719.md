## Summary

Keep open. The PR has a useful hardening goal, and current main still has a reply-normalization runtime gap for truthy non-string `payload.text`, but the branch is not merge-ready because it would reject currently supported attachment-only `chat.send` requests and its reply-normalizer hunk is based on an older implementation.

## What This Changes

The PR adds defensive message/text guards in reply normalization, `chat.send` sanitization, chat abort command parsing, and TUI stream text extraction.

## Best Possible Solution

Reapply the useful runtime narrowing on current main by normalizing `payload.text` to `string | undefined` once in `normalizeReplyPayload`, add a focused regression test for non-string runtime text with media or channel data, and leave `chat.send` schema validation plus attachment-only behavior intact.

## Reproduction Assessment

Yes. A focused unit test can call `normalizeReplyPayload` with a truthy non-string `text` plus media or channel data; current main can pass that raw value from `src/auto-reply/reply/normalize-reply.ts:56` into string-only silent-token helpers.

## Solution Assessment

No. The PR's reply-normalization idea is directionally right, but the branch applies it to an older code shape and changes `sanitizeChatSendMessageInput` in a way that breaks supported attachment-only sends; the safer fix is a narrow current-main reply normalizer guard.
