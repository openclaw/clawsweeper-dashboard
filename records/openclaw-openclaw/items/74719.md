## Summary

Keep open: the reported runtime hardening is still useful, but this PR should not merge as-is because it breaks supported attachment-only chat sends and its reply-normalizer hunk targets an older code shape.

## What This Changes

The PR adds runtime guards around reply text normalization, `chat.send` sanitization, chat abort parsing, and TUI stream text trimming to avoid undefined `trim()` calls.

## Best Possible Solution

Land a narrow current-main fix that narrows reply payload text once, adds regression coverage, preserves attachment-only `chat.send`, and includes the required changelog entry.

## Reproduction Assessment

Yes. A focused unit test can call `normalizeReplyPayload` with a truthy non-string `text` plus media or channel data and prove the current raw `payload.text` assignment can reach string-only helpers.

## Solution Assessment

No. The PR is directionally useful, but the sanitizer change rejects valid attachment-only sends and the normalizer change needs to be reapplied to current main's `hasContent`/`normalizeOptionalString` shape.
