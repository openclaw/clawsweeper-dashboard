## Label Justifications

- `P2`: This is a normal-priority heartbeat prompt metadata bug with noisy user impact and limited blast radius to synthetic scheduled turns.
- `impact:session-state`: Synthetic heartbeat route/session metadata is being presented to the agent as current per-turn conversation metadata.
- `impact:message-loss`: The reported false warning can be delivered to other active channels as an unintended user-visible message.

## Summary

Keep open: current main still has the source path that can put heartbeat delivery targets into the user-role `Conversation info` block, and the latest release still shows the same metadata emission. Two open PRs now target the same repair, so the issue should stay open until one canonical fix lands.

## What This Changes

The issue reports that cron heartbeat turns inject partial inbound conversation metadata with an unqualified chat id, causing false prompt-injection warnings and noisy cross-channel alerts.

## Best Possible Solution

Land one focused heartbeat metadata suppression fix that keeps route state available for delivery but prevents synthetic heartbeat, cron-event, and exec-event turns from presenting route targets as real inbound `Conversation info`.

## Reproduction Assessment

Yes for source reproduction: on current main, a heartbeat context with `OriginatingTo` reaches `buildInboundUserContextPrefix`, which emits `Conversation info.chat_id` before the heartbeat transcript marker. I did not run a live QQ/M3 heartbeat to reproduce the model warning itself.

## Solution Assessment

Yes: suppressing user-role inbound metadata for synthetic heartbeat-style runs is narrower and more maintainable than adding a QQ-only prefix, because the same partial-metadata hazard can occur for other heartbeat delivery routes.
