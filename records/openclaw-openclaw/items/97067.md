## Summary

Keep open: current main can still build a heartbeat agent turn with `OriginatingTo` copied from the resolved delivery route while `buildInboundUserContextPrefix()` turns that into a user-role `Conversation info` block, so the reported partial heartbeat metadata path is source-reproducible and not covered by a canonical issue or merged fix.

## What This Changes

The issue requests fixing cron/heartbeat prompt composition so synthetic heartbeat turns do not inject a partial `Conversation info` block with an unqualified channel target.

## Best Possible Solution

Fix heartbeat prompt composition so synthetic heartbeat turns do not prepend user-role `Conversation info` from the last delivery route, while preserving normal inbound metadata for real channel messages and adding regression coverage.

## Reproduction Assessment

Yes for source reproduction: a heartbeat context with `delivery.to` set to an unqualified `c2c:...` value will flow into `Conversation info.chat_id`. I did not run a live QQ heartbeat, so the downstream model warning is not manually reproduced here.

## Solution Assessment

Yes: the best fix is to keep synthetic heartbeat metadata out of the user-role inbound conversation block, or otherwise only emit canonical real-inbound metadata. A QQ-specific prefix patch would be narrower but would leave the same synthetic-metadata hazard for other heartbeat routes.
