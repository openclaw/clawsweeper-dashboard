## Summary

Keep open: current main still lets synthetic heartbeat runs pass `delivery.to` into the user-role `Conversation info` block, so the reported partial `chat_id` prompt shape remains source-reproducible and no merged or open canonical item owns this exact repair.

## What This Changes

The issue requests fixing cron/heartbeat prompt composition so synthetic heartbeat turns do not inject partial `Conversation info.chat_id` metadata that can be misread as prompt injection.

## Best Possible Solution

Suppress user-role inbound `Conversation info` for synthetic heartbeat turns, or only emit canonical real-inbound metadata, while preserving normal channel metadata for actual user messages.

## Reproduction Assessment

Yes for source reproduction: a heartbeat context whose `delivery.to` is an unqualified `c2c:...` target flows into `Conversation info.chat_id` before the heartbeat poll body. I did not execute a live QQ/M3 run, so the exact model warning is not manually reproduced here.

## Solution Assessment

Yes: the narrow maintainable fix is to stop synthetic heartbeat route metadata from being emitted as user-role inbound conversation metadata. Adding a QQ-only prefix would be too narrow because the same synthetic-metadata hazard exists for other heartbeat delivery routes.
