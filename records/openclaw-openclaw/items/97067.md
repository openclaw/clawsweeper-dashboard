## Summary

Keep open: current main and the latest release still have the source-level path that lets synthetic heartbeat route metadata become a `Conversation info` block, and the linked repair PRs are open rather than merged.

## What This Changes

The issue reports that cron heartbeat turns can inject partial inbound conversation metadata with an unqualified chat id, causing false prompt-injection warnings and noisy cross-channel alerts.

## Best Possible Solution

Review and land one linked narrow fix that suppresses inbound per-message metadata for synthetic heartbeat turns while preserving normal inbound channel metadata.

## Reproduction Assessment

Yes for source reproduction: current main builds heartbeat contexts with `OriginatingTo` and the normal prompt path emits `Conversation info.chat_id` from that field. I did not run a live QQ/M3 heartbeat turn in this read-only review.

## Solution Assessment

Yes: suppressing user-role inbound conversation metadata for synthetic heartbeat providers is narrower than adding a QQ-specific prefix and covers heartbeat, cron-event, and exec-event turns.
