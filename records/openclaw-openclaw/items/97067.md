## Summary

Keep open: current main still lets synthetic heartbeat delivery metadata reach the inbound `Conversation info` prompt path, and both linked fixes are still open rather than merged.

## What This Changes

The issue reports cron heartbeat turns adding an incomplete `Conversation info` metadata block with an unqualified `chat_id`, causing false prompt-injection warnings and unwanted cross-channel alerts.

## Best Possible Solution

Land one canonical caller-owned heartbeat metadata suppression, preferably https://github.com/openclaw/openclaw/pull/97072 or an equivalent patch, while preserving normal inbound channel metadata for real user messages.

## Reproduction Assessment

Yes for source reproduction: current main builds heartbeat contexts with `OriginatingTo`, passes them through `buildInboundUserContextPrefix`, and can emit `Conversation info.chat_id`. I did not live-run the downstream QQ/M3 warning path.

## Solution Assessment

Yes: suppressing inbound per-message metadata for synthetic heartbeat turns is the narrowest maintainable end state. Adding a QQ-specific prefix would leave the same synthetic-metadata hazard for other heartbeat delivery routes.
