## Summary

Keep open: current main still routes synthetic heartbeat delivery metadata into the inbound `Conversation info` prompt path, and the linked candidate fixes are open rather than merged.

## What This Changes

The issue asks OpenClaw to stop cron heartbeat prompts from exposing partial inbound conversation metadata with an unprefixed `chat_id`.

## Best Possible Solution

Review and land one narrow fix, preferably suppressing inbound per-message metadata for synthetic heartbeat, cron-event, and exec-event turns while preserving normal inbound channel metadata.

## Reproduction Assessment

Yes for source reproduction: current main can carry heartbeat `OriginatingTo` into `buildInboundUserContextPrefix`, which emits `Conversation info.chat_id` as model-visible inbound context. I did not run a live QQ/M3 heartbeat turn in this read-only review.

## Solution Assessment

Yes: suppressing inbound per-message metadata for synthetic heartbeat turns before it reaches model-visible context is the narrowest maintainable fix. Adding only a QQ-specific prefix would leave the same synthetic-metadata hazard for other delivery routes.
