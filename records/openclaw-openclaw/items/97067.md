## Summary

Keep open: current main still lets synthetic heartbeat route metadata reach the inbound `Conversation info` prompt path, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports that cron heartbeat turns can inject partial inbound conversation metadata with an unqualified `chat_id`, triggering false prompt-injection warnings and cross-channel alerts.

## Best Possible Solution

Review and land the linked narrow fix, or replace it with an equivalent change that suppresses inbound per-message metadata for synthetic heartbeat turns while preserving normal inbound channel metadata.

## Reproduction Assessment

Yes for source reproduction: a heartbeat context can carry `OriginatingTo` into `buildInboundUserContextPrefix`, which emits `Conversation info.chat_id` before heartbeat prompt assembly. I did not run a live QQ/M3 heartbeat turn in this read-only review.

## Solution Assessment

Yes: suppressing user-role inbound conversation metadata for synthetic heartbeat turns is the narrowest maintainable fix. Adding a QQ-specific prefix would leave the same synthetic-metadata hazard for other heartbeat delivery routes.
