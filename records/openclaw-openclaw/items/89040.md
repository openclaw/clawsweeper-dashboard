## Summary

Keep open. The PR targets a real P1 bootstrap-context/event-loop starvation report, but the proposed async fd reader has a blocking correctness regression: it treats one `fs.read` callback as a complete file read and can cache truncated bootstrap context; after that is fixed, the external PR still needs real embedded_run proof before merge.

## What This Changes

The PR changes agent bootstrap file loading, extra bootstrap glob walking, internal bootstrap hook dispatch, and embedded_run bootstrap-context timing instrumentation to reduce event-loop stalls.

## Best Possible Solution

Keep the performance direction, but first make the async fd reader preserve full-file semantics, add a short-read regression test, and require after-fix embedded_run liveness proof before merge.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection plus Node's `fs.read` contract shows a nonzero short read would be returned and cached as complete bootstrap file content. The original live 14-22s stall remains supported by issue logs but not re-proven by this PR.

## Solution Assessment

No. The scheduling and profiling direction is plausible, but replacing full-file sync reads with one callback read is not the safest implementation; use an async full-file read loop or equivalent helper and then prove the real embedded_run path.
