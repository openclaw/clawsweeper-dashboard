## Summary

Keep open. Current main still routes new direct messages according to `session.dmScope`, but it does not reconcile, retire, archive, or clearly mark stale peer-keyed direct-session rows after switching back to `main`; the existing cleanup path is age/count/disk/missing-transcript maintenance only.

## What This Changes

The issue asks for stale peer-keyed direct DM sessions to be safely reconciled, retired, marked inactive, or cleaned up after `session.dmScope` is changed back to `main`.

## Best Possible Solution

Add a channel-agnostic sessions cleanup or doctor repair path that detects direct-session rows inconsistent with current main-scoped DM routing, shows them in dry-run output, and preserves history while retiring, archiving, or marking them inactive only under an explicit repair/enforce action.

## Reproduction Assessment

Yes. Seed a store with `agent:main:main` plus `agent:main:telegram:direct:123` while config has `session.dmScope: "main"`; current route resolution returns the main key, and current cleanup/listing code has no dmScope-specific action for the peer-keyed row.

## Solution Assessment

Yes. The maintainable fix is a generic audited cleanup/doctor path with dry-run and history preservation; silent startup deletion or a Telegram-specific special case would be riskier.
