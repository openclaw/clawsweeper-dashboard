Keep open: current main and v2026.6.8 still contain the Synology Chat 120s deliver race, while this focused PR removes that competing timeout and adds targeted coverage, so it remains a viable merge candidate pending normal gates.

## What This Changes

The PR removes Synology Chat's local 120s `Promise.race` around webhook delivery and adds a regression test asserting no 120000ms timer is scheduled.

## Best Possible Solution

Land the narrow timeout-removal patch after normal CI and merge gates so Synology Chat relies on the shared agent runtime timeout instead of a channel-local deadline.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.8 still show `setTimeout(..., 120_000)` raced against `deliver` in the authorized Synology webhook path, and the linked reports provide user steps/logs. I did not run a live NAS repro.

## Solution Assessment

Yes. Removing the Synology-only local race is the narrowest maintainable fix because `dispatchSynologyChatInboundEvent` flows into shared channel inbound/reply dispatch, while adding or honoring a Synology-specific response timeout would keep a competing timeout policy.
