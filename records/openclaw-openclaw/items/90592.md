## What This Changes

The PR removes Synology Chat's hardcoded local `Promise.race` delivery timeout and adds a regression test asserting authorized webhook delivery no longer schedules a 120000ms timer.

## Best Possible Solution

Land this narrow timeout removal after the current PR-head merge gates are green or maintainer-accepted, keeping agent-turn timeout ownership in shared session/runtime policy.

## Reproduction Assessment

Yes. Current main and v2026.6.9 still race authorized Synology webhook delivery against `setTimeout(..., 120_000)`, and the duplicate report's logs match that 120s boundary; I did not run a live NAS reproduction.

## Solution Assessment

Yes. Removing the Synology-only local `Promise.race` is the best fix because the channel docs assign agent-turn timeouts to shared session/runtime policy, and adding or honoring a Synology-specific response timeout would keep a competing timeout source.
