- `P1`: The reported failure can poison provider credentials and break real agent/provider workflows until credentials are repaired.
- `impact:data-loss`: The issue describes persisted config secrets being overwritten by redacted placeholders.
- `impact:security`: The issue concerns the secret-redaction boundary between tools, transcripts, and the agent model.
- `impact:auth-provider`: Redacted placeholders written into provider API key fields can make provider auth fail with 401s.

## Summary

Keep open: current main and the latest release contain partial mitigations such as `sessions_history.contentRedacted` and polluted-placeholder config guards, but they do not provide the requested stable machine-readable marker across general agent-visible tool output; the related marker PR was closed unmerged and is not a canonical fix.

## What This Changes

Add a stable machine-readable signal to redacted agent-visible tool output so agents can tell masked placeholders are not real secrets.

## Best Possible Solution

Define one maintainer-approved agent-visible redaction contract at the shared tool-output/result boundary, cover `sessions_history` plus ordinary tool-result/file-read/exec redaction paths, and document how agents must treat marked output.

## Reproduction Assessment

Partially yes by source inspection: current code can redact agent-visible transcript/tool-result content without a general marker, while only `sessions_history` exposes `contentRedacted`; I did not run a live writeback reproduction.

## Solution Assessment

No: the current `contentRedacted` flag is useful for `sessions_history`, but it is not the general tool-output boundary marker requested; the best fix needs a single stable contract rather than isolated per-tool flags.
