## Label Justifications

- `P1`: The reported flow can poison provider API keys and leave real users unable to authenticate until manual repair.
- `impact:data-loss`: The issue is about real config values being overwritten with redacted placeholders during agent writeback.
- `impact:security`: The affected behavior is a secret-redaction boundary where masked sensitive data is indistinguishable from real data to agents.
- `impact:auth-provider`: The reported failure mode turns provider API keys into placeholders and causes provider authentication failures.

## Summary

Keep open. Current main and the latest release still return redacted agent-visible session-history content without a stable machine-readable marker, and the linked closing implementation is still open and conflicting rather than merged.

## What This Changes

Request a stable machine-readable marker or notice on redacted agent-visible tool output so agents can avoid writing masked placeholders back to config.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68483 or a narrow replacement that emits a stable marker only when final agent-visible output contains redaction, with docs and regression coverage.

## Reproduction Assessment

Yes, at source level. A `sessions_history` response containing an API-key-shaped token is masked and sets `contentRedacted`, while current main returns no stable marker, warning, or redacted-field metadata.

## Solution Assessment

Yes, directionally. An additive marker or notice at the final agent-visible redaction boundary is the maintainable direction, but the linked PR or a corrected replacement still needs to land before this issue can close.
