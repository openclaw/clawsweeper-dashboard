## Label Justifications

- `P1`: The report describes real credential poisoning that broke provider authentication, and current source still has an unmarked redaction path.
- `impact:data-loss`: The issue is about real config secrets being overwritten by redacted placeholders after an agent read redacted output.
- `impact:security`: The affected behavior is OpenClaw's secret redaction boundary for agent-visible and persisted transcript data.
- `impact:auth-provider`: The reported failure mode replaces provider API keys with placeholders, causing provider authentication failures.

## Summary

Keep open. Current main and v2026.5.28 have a partial `sessions_history` redaction flag, but the broader transcript/tool-result redaction path still rewrites model-visible tool-result content without a stable marker, and the linked marker PR was closed unmerged.

## What This Changes

The issue requests a stable machine-readable redaction marker in agent-visible tool output so agents do not treat redacted placeholders as real secrets.

## Best Possible Solution

Define one canonical redaction marker contract for model-visible and persisted tool results, keep the existing `sessions_history` flag compatible, and add focused coverage showing agents can distinguish redacted placeholders from real config values.

## Reproduction Assessment

Yes, from source inspection: tool-result and transcript redaction rewrite sensitive strings before persistence/replay, while the general redaction path returns only the rewritten message. I did not run a live repro because this review is read-only.

## Solution Assessment

No, current main is not the full solution. `sessions_history` has `contentRedacted`, but the broader redacted tool-result/transcript path still needs a canonical marker contract rather than another one-off field.
