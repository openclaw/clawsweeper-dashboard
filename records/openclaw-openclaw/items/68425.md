## Label Justifications

- `impact:data-loss`: The reported failure mode overwrites real credential values with redacted placeholders in persisted config.
- `impact:security`: The issue concerns sensitive-value redaction at an agent-visible output boundary.
- `impact:auth-provider`: Writing placeholders over provider API keys leads directly to provider authentication failures.

## Summary

Keep open. Current main and the latest release have partial coverage through `sessions_history.contentRedacted`, but ordinary shared redaction paths still return redacted text or objects without a common machine-readable marker for model-visible tool outputs.

## What This Changes

The issue requests a stable machine-readable signal when agent-visible tool output has had secrets redacted, to prevent agents from writing placeholder values back into config.

## Best Possible Solution

Define a shared redaction-result contract at the model-visible tool-output boundary, keep the existing `sessions_history.contentRedacted` partial signal, and validate the chosen contract against read/exec/config writeback flows before implementation.

## Reproduction Assessment

Yes at source level: the shared redaction helpers and tool-result sanitizer do not propagate redaction state, while `sessions_history` only adds a local `contentRedacted` flag. I did not run the live writeback incident path in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract shape. A per-tool `sessions_history` flag is not enough for the reported read/exec/config writeback path; the durable fix should live at a shared model-visible tool-output boundary.
