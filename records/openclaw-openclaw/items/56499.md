## Summary

Keep open: the linked startup secret crash-loop is still source-reproducible on current main, but this PR is mixed-scope, stale against the moved gateway startup path, and lacks after-fix real behavior proof.

## What This Changes

The branch adds startup-only retry/backoff around gateway SecretRef snapshot preparation in `src/gateway/server.impl.ts` and also changes WhatsApp append-upsert pairing suppression.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator` with bounded startup resilience, provider-aware regression coverage, docs/changelog updates, and keep the WhatsApp append behavior separate.

## Reproduction Assessment

Yes. A high-confidence source-level path is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls it once and throws the startup failure.

## Solution Assessment

No. The retry direction is useful, but this patch targets stale `server.impl.ts` code, retries the full provider batch, and mixes an unrelated WhatsApp behavior change into the secrets fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move startup retry into the active secrets activator:** `src/gateway/server.impl.ts:451-477`
  - body: On current main, `startGatewayServer` constructs `createRuntimeSecretsActivator`, and startup SecretRef activation runs in `src/gateway/server-startup-config.ts`. Adding retry around this stale inline block will not cover the current startup failure path after rebase.
  - confidence: 0.9
- **[P2] Avoid retrying the entire provider batch:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` wraps the whole `prepareSecretsRuntimeSnapshot` call. Because current resolution groups refs by provider and runs provider tasks for the snapshot, one persistent provider failure can rerun successful providers on each startup retry, multiplying external `op read` calls during the incident this is meant to reduce.
  - confidence: 0.82
- **[P2] Preserve stale append read marking:** `extensions/whatsapp/src/inbound/monitor.ts:484-486`
  - body: Moving the append age filter before `normalizeInboundMessage` skips `maybeMarkInboundAsRead` for stale append events. The existing code explicitly marks append catch-up messages read before suppressing auto-reply, so this leaves stale catch-up messages unread for allowed chats.
  - confidence: 0.86
- **[P3] Report total attempts in startup retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or retry-count wording so incident logs are unambiguous.
  - confidence: 0.92
