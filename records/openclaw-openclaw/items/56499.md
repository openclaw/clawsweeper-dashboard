## Summary

Keep this PR open, but do not merge it as-is. The underlying #56217 startup SecretRef crash-loop is still valid on current main, while the PR is stale against the active startup-secret activator and also bundles unrelated WhatsApp inbound behavior into the same branch.

## What This Changes

The PR adds startup-only exponential backoff around gateway SecretRef snapshot resolution and also changes WhatsApp append-upsert access control to suppress pairing replies for replayed messages.

## Best Possible Solution

Land a focused current-main replacement for #56217 in `createRuntimeSecretsActivator`: bounded startup resilience with deterministic tests for transient recovery and exhaustion, clear operator logs, preserved reload/restart-check last-known-good behavior, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A focused reproduction is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; the current test harness already proves current main rejects on the first startup preparation failure.

## Solution Assessment

No. The retry direction is valid, but this PR targets the obsolete inline startup path and retries the full snapshot; the narrower maintainable fix belongs in `src/gateway/server-startup-config.ts` with tests and retry behavior that does not amplify hard provider failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move startup retry to the active activator:** `src/gateway/server.impl.ts:454-477`
  - body: On current main, startup secret resolution runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. This retry block lands in the old inline path, so the reported startup crash-loop would remain after rebasing unless the retry moves to the active activator.
  - confidence: 0.88
- **[P2] Do not retry every provider batch on hard failures:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` retries the whole `prepareSecretsRuntimeSnapshot` call. Current resolution groups all SecretRefs by provider and runs provider tasks for each snapshot, so a persistent 1Password/auth/rate-limit failure can turn one supervisor restart into four complete provider-resolution runs, increasing the command volume this fix is meant to reduce.
  - confidence: 0.8
- **[P3] Report total startup attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the logs say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts, or otherwise make the retry count unambiguous for operators reading incident logs.
  - confidence: 0.9
