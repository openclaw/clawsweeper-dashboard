## Summary

Keep open: the linked startup SecretRef crash-loop bug still exists on current main, but this PR is not mergeable as-is because it patches stale startup wiring, retries the whole secrets snapshot, bundles unrelated WhatsApp behavior, and lacks external real behavior proof.

## What This Changes

The branch adds startup-only exponential retry around secret snapshot preparation and changes WhatsApp append-message handling to suppress pairing replies before inbound normalization.

## Best Possible Solution

Land a focused current-main replacement in createRuntimeSecretsActivator or a provider-aware resolver with bounded startup resilience and regression coverage, keep broader batching/memoization on https://github.com/openclaw/openclaw/issues/72278, and do not bundle WhatsApp changes.

## Reproduction Assessment

Yes, source-level: inject a rejecting prepareRuntimeSecretsSnapshot into createRuntimeSecretsActivator with reason `startup`; current main calls the resolver once and throws the fatal startup error. I did not run a live launchd plus 1Password service-account loop.

## Solution Assessment

No. Startup retry/backoff is useful, but this branch patches a stale server.impl.ts path, retries the entire snapshot, and bundles a WhatsApp change that regresses append read receipts; the safer fix belongs in the active activator or provider-aware resolver boundary.

Overall correctness: patch is incorrect

- **[P2] Move startup retry into the active activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main routes startup SecretRef activation through createRuntimeSecretsActivator in server-startup-config.ts. Adding retry around this stale inline server.impl.ts block will not cover the active startup failure path after rebase.

  - confidence: 0.84

  - body: Moving the stale append filter before normalizeInboundMessage skips maybeMarkInboundAsRead. Current coverage expects append catch-up messages to be marked read before auto-reply is skipped.
  - confidence: 0.92

  - body: With STARTUP_MAX_RETRIES = 3, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or retry-count wording so incident logs are unambiguous.
  - confidence: 0.92
