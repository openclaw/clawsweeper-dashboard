## Summary

Keep open: the linked startup SecretRef crash-loop bug is still valid on current main, but this PR is not mergeable as-is because it is stale against the active startup activator path, lacks required real behavior proof, retries the whole secrets snapshot, and bundles a WhatsApp change that regresses current append read-receipt behavior.

## What This Changes

The PR adds startup-only exponential backoff around secret runtime snapshot preparation and threads an `isAppend` flag through WhatsApp inbound access control while moving append filtering before normalization.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator` or a provider-aware resolver with bounded startup resilience and regression coverage, keep broader batching/memoization on https://github.com/openclaw/openclaw/issues/72278, and do not bundle the obsolete WhatsApp change.

## Reproduction Assessment

Yes, source-level: inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator` with reason `startup`; current main calls the resolver once and throws the fatal startup error. I did not run a live launchd plus 1Password service-account loop.

## Solution Assessment

No. Startup retry/backoff is useful, but this branch patches a stale `server.impl.ts` path, retries the entire snapshot, and bundles a WhatsApp change that regresses append read receipts; the safer fix belongs in the active activator or provider-aware resolver boundary.
