## Summary

Keep open: the underlying startup SecretRef crash-loop bug is still present on current main, but this PR is not mergeable as-is because it patches stale startup code, retries the whole secret snapshot, mixes in an already-superseded WhatsApp change, and still lacks contributor real-behavior proof.

## What This Changes

The PR adds startup-only exponential backoff around secret snapshot preparation and threads an append-history flag through WhatsApp inbound access control.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator` or the resolver with bounded startup resilience, provider-aware regression coverage, docs/changelog follow-through, and keep WhatsApp append behavior separate.

## Reproduction Assessment

Yes, source-level: inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls the resolver once and throws the fatal startup error. I did not run a live launchd plus 1Password service-account loop.

## Solution Assessment

No. The retry direction is useful, but this patch targets stale startup code, retries the entire provider batch, and bundles an unrelated WhatsApp change that regresses the read-receipt contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move startup retry into the active activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main routes startup SecretRef activation through `createRuntimeSecretsActivator` in `server-startup-config.ts`. Adding retry around this stale inline block in `server.impl.ts` will not cover the active startup failure path after rebase.
  - confidence: 0.9
- **[P2] Avoid retrying the whole secrets batch:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` wraps the full runtime snapshot. Since current resolution groups refs by provider and stops on the first provider error, one persistent provider failure can rerun successful providers on every retry, multiplying external calls this fix is meant to reduce.
  - confidence: 0.84
- **[P2] Preserve append read receipts before skipping:** `extensions/whatsapp/src/inbound/monitor.ts:484-486`
  - body: Moving the stale append filter before `normalizeInboundMessage` skips `maybeMarkInboundAsRead`. Existing coverage expects append catch-up messages to be marked read before auto-reply is skipped.
  - confidence: 0.92
- **[P3] Report total attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or retry-count wording so incident logs are unambiguous.
  - confidence: 0.92
