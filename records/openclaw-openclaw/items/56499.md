## Summary

Keep open: the PR targets a real startup SecretRef crash-loop problem, but the branch is not mergeable against current main because the retry is added to a stale `server.impl.ts` path, it retries the whole provider batch, it mixes in unrelated WhatsApp changes with a read-marking regression, and it lacks after-fix real behavior proof.

## What This Changes

The branch adds startup-only exponential retry/backoff around secret snapshot preparation in `src/gateway/server.impl.ts` and changes WhatsApp append-upsert handling to suppress pairing replies.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator` or the resolver with bounded startup resilience, provider-aware regression coverage, docs/changelog updates, and keep any WhatsApp append behavior in a separate validated PR.

## Reproduction Assessment

Yes. A high-confidence source path is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls it once and throws the fatal startup error.

## Solution Assessment

No. The retry direction is useful, but this patch targets stale startup code, retries the entire provider batch, and mixes an unrelated WhatsApp behavior change into the secrets fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move startup retry into the active secrets activator:** `src/gateway/server.impl.ts:451-477`
  - body: On current main, startup SecretRef activation runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`. Adding retry around this stale inline block in `server.impl.ts` will not cover the current startup failure path after rebase.
  - confidence: 0.9
- **[P2] Avoid retrying the entire provider batch:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` wraps the whole runtime snapshot preparation. Because current resolution groups refs by provider and stops on the first provider error, one persistent provider failure can rerun successful providers on every startup retry, multiplying the external calls this change is meant to reduce.
  - confidence: 0.82
- **[P2] Preserve stale append read marking:** `extensions/whatsapp/src/inbound/monitor.ts:484-486`
  - body: Moving the append age filter before `normalizeInboundMessage` skips `maybeMarkInboundAsRead` for stale append events. The existing path marks append catch-up messages read before suppressing auto-reply, so this leaves old catch-up messages unread for allowed chats.
  - confidence: 0.86
- **[P3] Report total attempts in startup retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or retry-count wording so incident logs are unambiguous.
  - confidence: 0.92
