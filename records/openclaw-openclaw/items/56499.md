## Summary

Keep open: the linked #56217 reliability bug still exists on current main, but this PR is not merge-ready because it targets the old startup secret-resolution location and retries the whole SecretRef snapshot in a way that can amplify provider calls.

## What This Changes

The branch adds startup-only retry/backoff around gateway SecretRef snapshot preparation in `src/gateway/server.impl.ts` and also changes WhatsApp append-upsert pairing suppression.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator` with bounded startup resilience, deterministic regression coverage, docs/changelog updates, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A source-level reproduction is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls it once and throws the startup failure.

## Solution Assessment

No. The retry direction is useful, but this patch targets stale `server.impl.ts` code and retries the full provider batch; the narrower fix belongs in the active secrets activator.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move startup retry into the active secrets activator:** `src/gateway/server.impl.ts:451-477`
  - body: On current main, startup SecretRef activation runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. Adding the retry block here will not cover the current startup failure path after a rebase.
  - confidence: 0.88
- **[P2] Avoid retrying the entire provider batch:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` retries the full `prepareSecretsRuntimeSnapshot` call. Current resolution groups refs by provider and runs provider tasks for the snapshot, so one persistent provider failure can multiply all provider calls during a single startup, which works against the rate-limit fix.
  - confidence: 0.8
- **[P3] Report total attempts in startup retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or clearer retry-count wording so incident logs are unambiguous.
  - confidence: 0.9
