## Summary

Keep this PR open, but do not merge it as-is. Current main still has the startup SecretRef fail-fast behavior and no startup retry marker, so the underlying #56217 problem remains valid; however this branch is stale against the active startup-secret path, retries the whole snapshot in a way that can increase provider command volume on hard failures, and also carries unrelated WhatsApp changes.

## What This Changes

The PR adds startup-only exponential backoff around gateway secret-provider snapshot resolution and includes separate WhatsApp append-upsert pairing suppression changes.

## Best Possible Solution

Land a focused current-main fix for #56217 in the active startup-secret activation path: bounded startup resilience with deterministic tests for transient recovery and exhaustion, no retry for obvious permanent configuration failures, preserved reload/restart-check last-known-good behavior, clear operator logs, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A focused reproduction is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first failed preparation, which is the startup crash-loop that a supervisor can amplify externally.

## Solution Assessment

No. The retry direction is valid, but this PR targets the obsolete inline activation path and retries the full snapshot; the safer solution is a narrow current-main implementation in `src/gateway/server-startup-config.ts`, with provider/error-aware retry behavior and tests.
