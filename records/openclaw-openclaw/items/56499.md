## Summary

Keep open, but do not merge as-is: the PR is useful source work for the startup SecretRef crash-loop, yet current main now runs startup secret activation through `createRuntimeSecretsActivator`, while this branch patches the obsolete inline path and also carries unrelated WhatsApp changes.

## What This Changes

The PR adds startup-only retry/backoff around gateway SecretRef snapshot resolution in `server.impl.ts` and also changes WhatsApp append-upsert pairing suppression.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator`: bounded startup resilience for #56217, deterministic transient/exhaustion tests, clear operator logs, docs/changelog updates, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A focused reproduction is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects on the first startup preparation failure.

## Solution Assessment

No. The retry direction is valid, but this patch is in the wrong current-main location and retries the whole snapshot in a way that can amplify hard provider failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move startup retry into the active secrets activator:** `src/gateway/server.impl.ts:454-477`
  - body: On current main, startup secret resolution runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. Adding the retry block here leaves the reported startup failure path unchanged after rebasing.
  - confidence: 0.88
- **[P2] Avoid retrying the entire provider batch on hard failures:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` retries the full `prepareSecretsRuntimeSnapshot` call. Current resolution groups all SecretRefs by provider and runs provider tasks for the snapshot, so a persistent 1Password/auth/rate-limit failure can turn one startup into multiple complete provider-resolution runs, increasing the command volume this fix is meant to reduce.
  - confidence: 0.8
- **[P3] Report total attempts in startup retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the logs say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or otherwise make the retry count unambiguous for incident logs.
  - confidence: 0.9
