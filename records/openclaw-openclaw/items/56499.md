## Summary

Keep this PR open, but do not merge it as-is. The underlying startup-secret crash-loop bug is still present on current main, while this branch adds retry logic to a stale inline `server.impl.ts` path, retries the whole secret snapshot in a way that can increase provider calls on persistent failures, and mixes unrelated WhatsApp changes into the same patch.

## What This Changes

The PR adds startup-only exponential backoff around secret provider snapshot resolution in the gateway and threads WhatsApp append-upsert state into inbound pairing/access-control handling.

## Best Possible Solution

Land a focused current-main fix for #56217 in `createRuntimeSecretsActivator`: bounded startup resilience with deterministic tests for transient recovery and exhaustion, clear operator logs, preserved reload/restart-check last-known-good behavior, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A focused reproduction is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first failed preparation, matching the startup crash-loop that a supervisor can amplify externally.

## Solution Assessment

No. The retry direction is valid, but this PR targets the obsolete inline activation path and retries the full snapshot; the safer solution is a narrow current-main replacement in `src/gateway/server-startup-config.ts` with tests and tuned call-volume behavior.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Move startup retry to the active activator:** `src/gateway/server.impl.ts:454-477`
  - body: On current main, startup secret resolution runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. This retry block lands in the old inline path, so the reported startup crash-loop would remain after rebasing unless the retry moves to the active activator.
  - confidence: 0.88
- **[P2] Do not retry every provider batch on hard failures:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` retries the whole `prepareSecretsRuntimeSnapshot` call. Current resolution groups all SecretRefs by provider and runs every provider task for each snapshot, so a persistent 1Password/auth/rate-limit failure can turn one supervisor restart into four complete provider-resolution runs, increasing the command volume this fix is meant to reduce.
  - confidence: 0.8
- **[P3] Report total startup attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the logs say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts, or otherwise make the retry count unambiguous for operators reading incident logs.
  - confidence: 0.9
