## Summary

Keep open: the linked startup SecretRef crash-loop problem is still present on current main, but this PR is not mergeable as-is because its secrets retry hunk targets stale gateway wiring and the branch also carries unrelated WhatsApp behavior changes.

## What This Changes

The PR adds startup-only exponential retry/backoff around secret snapshot resolution in `src/gateway/server.impl.ts` and also changes WhatsApp append-upsert pairing suppression in `extensions/whatsapp/src/inbound/*`.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator`: bounded startup resilience for #56217, provider-aware retry behavior, deterministic tests, clear operator logs, docs/changelog updates, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A source-backed reproduction is to inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first preparation failure and the existing test encodes that path.

## Solution Assessment

No. The retry direction is reasonable, but this patch targets stale `server.impl.ts` startup wiring, retries the whole provider snapshot, and mixes an unrelated WhatsApp fix into the same branch.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Move startup retry into the active secrets activator:** `src/gateway/server.impl.ts:454-477`
  - body: On current main, startup secret resolution runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. This hunk will be stale after rebase and leaves the reported startup failure path unchanged.
  - confidence: 0.88
- **[P2] Avoid retrying the full SecretRef snapshot:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` wraps the entire `prepareSecretsRuntimeSnapshot({ config })` call. Current SecretRef resolution groups all refs by provider for the snapshot, so one persistent provider failure can re-run every provider batch on each retry and multiply the command volume this fix is meant to reduce.
  - confidence: 0.8
- **[P3] Fix the startup retry attempt count:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Report total attempts or otherwise make retry count wording unambiguous for incident logs.
  - confidence: 0.9
