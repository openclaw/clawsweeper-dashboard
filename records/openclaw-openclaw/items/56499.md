## Summary

Keep this PR open, but do not merge it as-is. The underlying #56217 gateway startup-secret crash-loop is still valid on current main, but this branch is stale against the current startup-secret activator and mixes in WhatsApp work for a separate closed issue.

## What This Changes

The PR adds startup-only exponential backoff around gateway secret snapshot resolution in `src/gateway/server.impl.ts` and also changes WhatsApp append-message handling to suppress pairing replies for replayed history messages.

## Best Possible Solution

Land a focused current-main fix for #56217 in `createRuntimeSecretsActivator`: bounded startup resilience with deterministic transient-recovery and exhaustion tests, preserved reload/restart-check last-known-good behavior, clear operator logging, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. Inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first failed preparation, matching the startup crash-loop path that a supervisor can amplify externally.

## Solution Assessment

No. The retry direction is valid, but this PR applies it to a stale inline startup path and retries the full secret snapshot, so the safer solution is a narrow current-main repair in `src/gateway/server-startup-config.ts` with call-volume safeguards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move startup retry to the active activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main runs startup secret activation through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. This retry block lands in the old inline path, so after rebasing the reported startup crash-loop would remain.
  - confidence: 0.88
- **[P2] Avoid retrying the full secret batch on hard failures:** `src/gateway/server.impl.ts:459-477`
  - body: `attemptResolve()` retries the entire `prepareSecretsRuntimeSnapshot` call. Since current resolution groups and runs every provider task in the snapshot, a persistent exec/1Password failure can make each supervisor restart execute all configured providers four times, increasing the rate-limit burn this fix is meant to reduce.
  - confidence: 0.8
- **[P3] Report total startup attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the logs say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or otherwise make the retry count unambiguous for operators reading incident logs.
  - confidence: 0.9
