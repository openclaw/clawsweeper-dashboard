## Summary

Keep open; this PR is useful source work for the linked secret-provider crash-loop bug, but it should not merge as-is because current main moved startup SecretRef activation into `server-startup-config.ts`, while the branch patches stale `server.impl.ts` wiring and mixes in unrelated WhatsApp behavior.

## What This Changes

The PR adds startup-only exponential backoff around secret snapshot preparation and also changes WhatsApp append-upsert pairing suppression.

## Best Possible Solution

Land a focused current-main fix in `createRuntimeSecretsActivator` or the lower secret-provider resolver with regression coverage, docs/changelog updates, and no unrelated WhatsApp changes in the same branch.

## Reproduction Assessment

Yes, at source level. Current main calls `prepareRuntimeSecretsSnapshot` once during startup activation and throws the wrapped startup failure when that dependency rejects; the existing startup secrets test already exercises that shape.

## Solution Assessment

No. Startup retry/backoff is a reasonable direction, but this patch targets stale wiring and retries the full provider batch; the narrower maintainable fix belongs in the active activator or provider-resolution boundary.

- **[P2] Move startup retry into the active secrets activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main runs startup SecretRef activation through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. Adding retry around the old inline snapshot call here will miss the current startup failure path after rebase.
  - confidence: 0.88

  - body: `attemptResolve()` retries the entire snapshot preparation. Since current SecretRef resolution groups refs by provider and runs all provider tasks for the snapshot, one persistent provider failure can cause every previously successful provider group to run again, increasing the provider-call amplification this fix is trying to reduce.
  - confidence: 0.8

  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or explicit retry-count wording so incident logs are not contradictory.
  - confidence: 0.9

  - body: This user-facing gateway reliability fix, plus the WhatsApp behavior change, needs a `CHANGELOG.md` entry under the active release before merge per repo policy.
  - confidence: 0.78
