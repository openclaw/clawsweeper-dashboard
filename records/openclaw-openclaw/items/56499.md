## Summary

Keep open. The secrets backoff direction is useful, but this PR is not merge-ready: it targets stale startup code, retries the whole secrets batch, lacks contributor real-behavior proof, is currently conflicting, and its unrelated WhatsApp change fails the existing append read-marking contract.

## What This Changes

The PR adds startup-only exponential retry/backoff around secret snapshot preparation and changes WhatsApp append-upsert handling to suppress pairing replies.

## Best Possible Solution

Land a focused current-main replacement in `createRuntimeSecretsActivator` or the resolver with bounded startup resilience, provider-aware regression coverage, docs/changelog updates, and keep WhatsApp append behavior in a separate validated path.

## Reproduction Assessment

Yes, source-level. Inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator(..., { reason: "startup" })` or use an active failing exec SecretRef provider; current main calls the resolver once and throws the fatal startup error.

## Solution Assessment

No. The retry direction is useful, but this patch targets stale startup code, retries the entire provider batch, and mixes an unrelated WhatsApp behavior change that is already superseded and currently breaks read-marking coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move startup retry into the active activator:** `src/gateway/server.impl.ts:451-477`
  - body: Current main routes startup SecretRef activation through `createRuntimeSecretsActivator` in `server-startup-config.ts`. Adding retry around this stale inline block in `server.impl.ts` will not cover the active startup failure path after rebase.
  - confidence: 0.9
- **[P2] Avoid retrying the whole secrets batch:** `src/gateway/server.impl.ts:463-477`
  - body: `attemptResolve()` wraps the full runtime snapshot. Since current resolution groups refs by provider and stops on the first provider error, one persistent provider failure can rerun successful providers on every retry, multiplying calls to the same external services this fix is meant to protect.
  - confidence: 0.84
- **[P2] Preserve append read receipts before skipping:** `extensions/whatsapp/src/inbound/monitor.ts:484-486`
  - body: Moving the stale append filter before `normalizeInboundMessage` skips `maybeMarkInboundAsRead`. Existing coverage expects append catch-up messages to be marked read before auto-reply is skipped, and the live PR checks fail on that contract.
  - confidence: 0.92
- **[P3] Report total attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the log can say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts or retry-count wording so incident logs are unambiguous.
  - confidence: 0.92
