## Summary

Keep open. The PR addresses the direct gateway cron `runHeartbeatOnce` adapter, but current cron can normally fall back to a queued `requestHeartbeatNow({ heartbeat: { target: "last" } })` path that still re-merges configured heartbeat destinations and can preserve the same cross-context route leak reported in #73900.

## What This Changes

The PR adds a helper that removes inherited heartbeat `to` and `accountId` fields for cron-forced `target: "last"` direct heartbeat runs, plus a gateway cron regression test and changelog entry.

## Best Possible Solution

Move the destination-field omission to a shared cron wake boundary used by both direct `runHeartbeatOnce` and queued `requestHeartbeatNow` cron-forced `target: "last"` wakes. Add regression coverage for busy and `cron-in-progress` fallback wakes while preserving normal explicit heartbeat destinations outside this cron/systemEvent path.

## Reproduction Assessment

Yes. The source-backed reproduction path is a main-session `systemEvent` cron job with `wakeMode: "now"` and configured heartbeat `to`/`accountId`, where the direct wake skips as busy or `cron-in-progress` and the queued fallback re-merges those configured destination fields before outbound delivery.

## Solution Assessment

No. The PR is directionally correct for the direct adapter, but the best fix needs to cover the queued fallback path that cron already uses for the same user-visible wake.
