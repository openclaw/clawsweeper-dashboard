## Summary

Keep open. The PR fixes the direct cron `runHeartbeatOnce` adapter path, but the linked routing leak can still occur when main-session cron falls back to queued `requestHeartbeatNow` wakes; that path still merges `target: "last"` over configured heartbeat destinations.

## What This Changes

The PR strips inherited heartbeat `to` and `accountId` fields from cron `runHeartbeatOnce` overrides when forcing `target: "last"`, adds a gateway cron regression test, and adds a changelog entry.

## Best Possible Solution

Move the destination-field omission to a boundary shared by direct and queued cron-forced `target: "last"` wakes, or update the heartbeat wake runner so cron target-last overrides do not inherit explicit destinations. Add regression coverage that exercises the actual main-session cron fallback/queued path while preserving normal explicit heartbeat destination behavior outside this cron case.

## Reproduction Assessment

Yes. The source-backed path is: main-session cron enqueues a system event, `runHeartbeatOnce` can skip as `cron-in-progress`, the queued wake merges `{ target: "last" }` with configured heartbeat defaults, and outbound resolution still honors inherited `to`/`accountId`.

## Solution Assessment

No. The helper is directionally correct for the direct adapter call, but it is not the narrow complete fix because the normal queued fallback path still carries the conflicting destination fields.

## Review Findings

The linked route leak can still happen on queued wakes, so apply the omission at a shared wake boundary and cover that path.
