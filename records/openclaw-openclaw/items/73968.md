## Summary

Keep open. The PR is directionally useful for the direct cron heartbeat path, but it does not fully fix the linked privacy-sensitive routing bug because queued cron heartbeat wakes still carry `target: "last"` into the heartbeat wake runner where configured `to` and `accountId` can be re-inherited.

## What This Changes

The PR strips inherited heartbeat `to` and `accountId` fields from the gateway cron adapter's direct `runHeartbeatOnce` override when cron forces `target: "last"`, and adds a gateway cron regression test plus a changelog entry.

## Best Possible Solution

Put the destination-field omission at a shared boundary used by both direct and queued cron-forced `target: "last"` wakes, or make the heartbeat wake runner treat that cron override as non-inheriting for explicit destinations. Cover the actual main-session cron fallback path while preserving normal explicit heartbeat destinations outside cron-forced wakes.

## Reproduction Assessment

Yes. The source-backed reproduction path is a main-session `systemEvent` cron job whose direct heartbeat wake skips as busy or `cron-in-progress`, causing `requestHeartbeatNow({ heartbeat: { target: "last" } })`; the queued runner then merges that request with configured heartbeat defaults and outbound resolution honors inherited `to`/`accountId`.

## Solution Assessment

No. The PR sanitizes the direct `runHeartbeatOnce` adapter only, but the same user-visible route leak remains on the queued `requestHeartbeatNow` path that current cron code can reach normally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Sanitize queued cron heartbeat wakes too:** `src/gateway/server-cron.ts:294-296`

Main-session cron runs can fall back to `requestHeartbeatNow({ heartbeat: { target: "last" } })` when the wake is busy or `cron-in-progress`, and the queued heartbeat runner then merges that request with the agent heartbeat config, preserving configured destinations. That leaves the linked cross-context delivery leak possible on fallback wakes, so the sanitization needs to live at a shared wake boundary and be covered by the actual cron execution path.
