## Summary

Keep open; the patch is partial because it only sanitizes direct cron wakes while the queued fallback path can still inherit explicit heartbeat destinations and leak replies across delivery contexts.

## What This Changes

The branch updates `CHANGELOG.md`, `src/gateway/server-cron.ts`, and `src/gateway/server-cron.test.ts` to drop inherited heartbeat destinations for direct cron target-last `runHeartbeatOnce` wakes.

## Best Possible Solution

Move destination-field omission to a shared cron forced-target-last wake boundary used by both direct and queued wakes, with regression coverage for busy, cron-in-progress, and next-heartbeat queued cases.

## Reproduction Assessment

Yes. Configure a main-session `systemEvent` cron job bound to a group/topic session with global/default heartbeat `to` and `accountId`, then make `runHeartbeatOnce` skip as `cron-in-progress` or busy until the queued fallback path runs.

## Solution Assessment

No. The direct-path helper is directionally correct, but the narrow maintainable fix must also sanitize queued `requestHeartbeat` cron wakes before delivery resolution re-merges defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Sanitize queued cron heartbeat wakes too:** `src/gateway/server-cron.ts:294-296`
  - body: This strips inherited `to`/`accountId` only on the direct `runHeartbeatOnce` path. When wake-now skips as busy or `cron-in-progress`, cron queues `requestHeartbeat({ heartbeat: { target: "last" } })`; that queued runner still re-merges configured heartbeat defaults, so inherited destination fields can still override the session-bound route.
  - confidence: 0.9
