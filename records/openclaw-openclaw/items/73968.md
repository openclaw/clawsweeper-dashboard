## Summary

Keep open. The PR fixes the direct cron heartbeat adapter, but current source still has a queued fallback path for the same main-session wake that re-merges configured heartbeat destinations before outbound delivery.

## What This Changes

The PR adds a helper to drop inherited heartbeat `to` and `accountId` fields for cron direct `runHeartbeatOnce` target-last wakes, plus a gateway cron regression test and changelog entry.

## Best Possible Solution

Move the destination-field omission to a shared cron forced-target-last wake boundary used by both direct and queued wakes, with regression coverage for busy and cron-in-progress fallback cases.

## Reproduction Assessment

Yes. A main-session `systemEvent` cron job with `wakeMode: "now"` and global heartbeat `to`/`accountId` can reproduce this when `runHeartbeatOnce` skips as busy or `cron-in-progress`, because the queued fallback is later merged with configured heartbeat defaults.

## Solution Assessment

No. The PR is directionally correct for direct wakes, but the narrow maintainable fix must also cover queued `requestHeartbeat` cron wakes for the same observable reminder route.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P1] Sanitize queued cron heartbeat wakes too:** `src/gateway/server-cron.ts:294-296`
