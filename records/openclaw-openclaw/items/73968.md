## Summary

Keep open. The PR fixes the direct cron wake path, but source inspection confirms queued fallback wakes can still inherit configured `heartbeat.to`/`accountId` and route a session-bound reminder to the wrong delivery context.

## What This Changes

The PR strips `to` and `accountId` from cron-forced `target: "last"` direct heartbeat overrides, and adds a gateway cron regression test plus changelog entry.

## Best Possible Solution

Move destination-field omission to a shared cron forced-target-last wake boundary used by both direct `runHeartbeatOnce` and queued `requestHeartbeat`, with busy and `cron-in-progress` regression coverage.

## Reproduction Assessment

Yes. A main-session `systemEvent` cron job with `wakeMode: "now"`, a bound group/topic session key, and global `heartbeat.to`/`accountId` can reproduce this when the direct wake skips as busy or `cron-in-progress`, because the queued wake is later merged with those configured defaults.

## Solution Assessment

No. The direct-wake sanitization is directionally correct, but the narrow maintainable fix must also cover queued cron fallback wakes for the same user-visible route.
