## Summary

Keep open. The PR addresses a real privacy-sensitive cron routing bug and improves the direct wake path, but it only strips inherited destinations before the direct `runHeartbeatOnce` call; the queued `requestHeartbeatNow` fallback can still merge `target: "last"` with configured `heartbeat.to`/`accountId` and leak a session-bound reminder to an explicit destination.

## What This Changes

The PR strips inherited heartbeat `to` and `accountId` fields from the gateway cron adapter's direct `runHeartbeatOnce` override when cron forces `target: "last"`, and adds a gateway regression test plus a changelog entry.

## Best Possible Solution

Move the destination-field omission to a shared cron-forced `target: "last"` boundary used by both direct and queued wakes, or make the heartbeat wake runner treat that cron override as non-inheriting for explicit destinations. Add coverage for busy/`cron-in-progress` fallback and `next-heartbeat` main-session cron paths while preserving normal explicit heartbeat destinations outside this cron case.

## Reproduction Assessment

Yes. A high-confidence source path is a main-session `systemEvent` cron job bound to a topic/group session with global or agent `heartbeat.to`/`accountId`, where the direct wake skips as busy/`cron-in-progress` or the job uses `next-heartbeat`; the queued wake then merges `target: "last"` over the configured heartbeat and keeps explicit destinations.

## Solution Assessment

No. The proposed fix is too narrow because it sanitizes only the direct gateway adapter path, while current main can reach the same user-visible route leak through the queued wake path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Sanitize queued cron heartbeat wakes too:** `src/gateway/server-cron.ts:294-296`
  - body: This only strips `to`/`accountId` on the direct `runHeartbeatOnce` path. Main-session cron runs can fall back to `requestHeartbeatNow({ heartbeat: { target: "last" } })` when the wake is busy or `cron-in-progress`, and the queued heartbeat runner then merges that request with the agent heartbeat config, preserving configured destinations. That leaves the reported cross-context delivery leak possible on fallback wakes, so the sanitization needs to live at a shared wake boundary and be covered by the actual queued cron path.
  - confidence: 0.9
