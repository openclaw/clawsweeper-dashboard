## Summary

Keep this issue open for a maintainer product decision: current main does not provide temporary-artifact provenance or reject runtime-critical system-temp paths, but the request combines a focused daemon-safety invariant with a much broader artifact-lifecycle policy that should not be implemented as one feature.

## What This Changes

Add provenance classification and durable routing for agent-created temporary artifacts, cleanup-aware handling, and safeguards against runtime-critical paths resolving into system temp directories.

## Best Possible Solution

Split the work into a focused cross-platform daemon/doctor invariant that flags ephemeral executable, wrapper, symlink-target, and service paths, plus a separate product design for explicit tool-level artifact disposition backed by canonical SQLite storage if maintainers decide it belongs in core rather than a plugin or workflow.

## Risks / Open Questions

- The umbrella scope crosses daemon installation and doctor, agent tool contracts, persistent state, artifact routing, and cleanup policy, so a single implementation would create unclear ownership and a large validation surface.
- A metadata sidecar or temporary registry would create parallel file-based application state contrary to OpenClaw's database-first storage policy.
- Automatically inferring whether arbitrary agent output is durable could misclassify unique work or retain sensitive scratch data unless the product contract and security model are explicitly designed.
