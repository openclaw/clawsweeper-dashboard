## Summary

Keep open. Current main has adjacent temp-path hardening, temp cleanup helpers, trajectory exports, diffs TTL metadata, and App SDK artifact placeholders, but it still lacks the requested scratch/reproducible/durable temp provenance contract, registry or sidecar semantics, janitor behavior, and broad runtime-critical volatile-path validation.

## What This Changes

Request to add built-in provenance and guardrails for agent-created temp artifacts so disposable scratch, reproducible artifacts, and durable work are distinguishable and cleanup/runtime checks can avoid unsafe `/tmp` drift.

## Best Possible Solution

Define a narrow shared artifact-provenance contract in core, then expose it through the appropriate plugin/App SDK and runtime helper surfaces. The end state should let first-class tools and plugins register scratch, reproducible, or durable artifacts with minimal metadata, let doctor/cleanup understand those classes, and add targeted volatile-path checks for service wrappers, module targets, and active install/build/runtime paths while coordinating with the open artifact RPC design.

## Reproduction Assessment

Not applicable as a conventional bug reproduction. The gap is high-confidence from source inspection: current main has helper-only temp APIs and explicitly unsupported artifact RPCs, but no general temp artifact provenance or cleanup classification contract.

## Solution Assessment

Unclear until maintainers choose the product and API boundary. The requested direction is plausible, but the best path is a small shared contract coordinated with artifact RPC work rather than a broad one-shot janitor implementation.
