## Summary

Keep this open: current main has useful temp-root hardening, service-command temp-path checks, and temp helper APIs, but it does not implement the requested temp artifact classes, provenance registry, durable routing, or janitor semantics. The broader core safety concern also prevents a clean ClawHub close even though a narrower review-staging variant was already closed that way.

## What This Changes

The issue requests temp artifact provenance, durable-path routing, cleanup awareness, and runtime-critical temp-path guardrails for OpenClaw-created or agent-created files outside the workspace.

## Best Possible Solution

Keep this issue as a maintainer-scoped design item: core should own runtime-critical temp-path guardrails and any generic artifact provenance API, while workflow-specific janitor/SOP classification should stay in plugin or ClawHub surfaces unless maintainers identify a missing core seam.

## Reproduction Assessment

Partly, at source level: current main has no central temp artifact class/registry or janitor semantics, and generic harness temp-file creation has no provenance. No concrete current-main command reproduces the reported reboot/runtime temp-drift failure.

## Solution Assessment

Unclear as one implementation: adding a broad temp registry/default model needs maintainer product and storage-boundary decisions. The safer direction is to separate core runtime-critical path checks from optional workflow provenance and use existing plugin/ClawHub surfaces where they are enough.
