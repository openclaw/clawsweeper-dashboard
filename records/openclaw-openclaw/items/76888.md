## Summary

Keep open: current main still contains the source-backed orphan-user repair path that prepends older queued user text ahead of the latest triggering prompt, matching the reported stale-reply shape and conflicting with the documented latest-input steering contract.

## What This Changes

The issue asks OpenClaw to prevent queued/orphaned user-message repair from letting an older unresolved user turn steer the next visible reply over the latest triggering instruction.

## Best Possible Solution

Change the embedded runner's orphan-trailing-user repair so older orphan content cannot outrank the latest trigger, then add regression coverage proving latest-prompt-wins while preserving provider-safe role ordering.

## Reproduction Assessment

No, not for a high-confidence end-to-end live channel/model stale reply. The source path is high-confidence: current main prepends the older orphan leaf before the latest prompt, removes that leaf, and logs the same orphaned-user repair shape reported here.

Fix the queued/orphaned user-message stale-reply bug reported in https://github.com/openclaw/openclaw/issues/76888. In current main, mergeOrphanedTrailingUserPrompt prepends older orphaned user text before the latest prompt, and runEmbeddedAttempt removes that leaf before submitting the next model call. Adjust the embedded Pi runner orphan-trailing-user repair so older orphan content cannot outrank the latest triggering user instruction while still preventing provider-invalid consecutive user turns.
