## Summary

Keep open: current main still has the source-backed orphan-user repair path that prepends the older orphaned user text before the latest triggering prompt and submits that merged prompt to the model, while focused tests still lock in that older-before-newer ordering.

## What This Changes

The issue reports that a queued/orphaned user-message merge can make an embedded-runner response answer an earlier unresolved user turn instead of the latest triggering instruction.

## Best Possible Solution

Adjust the embedded runner's orphan-trailing-user repair so older orphan content cannot outrank the latest trigger while preserving provider-safe role ordering, then update regression coverage and the changelog.

## Reproduction Assessment

No, not as a high-confidence end-to-end channel/model run. Source inspection reproduces the risky path: current main prepends the older orphaned user leaf before the latest prompt, removes that leaf, and logs the same repair shape reported here.

## Solution Assessment

Unclear because there is no proposed patch yet. The narrow maintainable fix is to change prompt shaping or pruning in the embedded-runner orphan repair so the latest triggering prompt remains authoritative without adding a new queue mode or config knob.
