## Repair Work Prompt

Fix the stale-reply risk in https://github.com/openclaw/openclaw/issues/76888. In current main, `mergeOrphanedTrailingUserPrompt` prepends older orphaned user content ahead of the latest triggering prompt, and `runEmbeddedAttempt` then removes the user leaf; this can let the older turn steer the next model response. Keep the fix inside the embedded Pi runner/message-merge boundary. Make the latest triggering prompt clearly authoritative while preserving provider-safe role ordering and without silently dropping orphaned queued content. Do not add a config option or new queue mode. Update/add regression coverage proving latest-prompt-wins for orphan repair, structured orphan content remains represented safely, duplicate orphan text is not repeated, and the active runtime contract still avoids consecutive user turns. Add a single-line CHANGELOG.md fix entry because this is user-facing. Related context: #76875 was a broader stale-prioritization report, #76990 tracks a related but distinct assistant-transcript persistence stale-reply path.
