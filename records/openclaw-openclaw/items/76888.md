## Summary

Keep open: current main still has a source-backed orphan-user repair path that can place older queued user text ahead of the latest triggering prompt, matching the reported stale-reply shape and conflicting with the documented latest-input steering behavior.

## What This Changes

The issue reports that queued/orphaned user-message merging can make an embedded runner response answer an earlier unresolved user turn instead of the latest user instruction.

## Best Possible Solution

Change the embedded runner's orphan-trailing-user repair so older orphan content cannot outrank the latest trigger, while preserving provider-safe role ordering and adding regression coverage for latest-prompt-wins behavior.

## Reproduction Assessment

No, not for a full live channel/model stale reply in this read-only review. The source-level reproduction is high confidence: mergeOrphanedTrailingUserPrompt currently prepends the older orphan leaf before the latest prompt, and runEmbeddedAttempt invokes that path before the model call.

## Repair Work Prompt

Fix the embedded Pi runner stale-reply bug tracked by https://github.com/openclaw/openclaw/issues/76888. Current main prepends an older orphaned trailing user leaf before the latest triggering prompt in mergeOrphanedTrailingUserPrompt, then runEmbeddedAttempt removes that leaf and submits the merged prompt. Adjust the orphan-trailing-user repair so older orphan content cannot outrank or steer over the latest trigger while still preventing consecutive user turns for provider compatibility. Add or update focused regression coverage proving latest-prompt-wins for user-triggered repairs, structured orphan content remains handled safely, duplicate orphan text is not reintroduced, and runtime strategy contracts still preserve valid role ordering. Do not add a new queue mode, config knob, provider-specific policy, or broad transcript migration. Add a single-line CHANGELOG.md fix entry because this is user-facing channel behavior.
