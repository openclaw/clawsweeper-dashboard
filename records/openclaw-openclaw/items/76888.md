## Summary

Keep open: current main still has a source-backed orphan-user repair path that can put older orphaned user content ahead of the latest triggering prompt, matching the reported stale-reply shape and conflicting with the documented latest-input steering behavior.

## What This Changes

The issue reports that queued/orphaned user-message repair can let an earlier unresolved user turn steer a later run even when the submitted prompt is the latest user instruction.

## Best Possible Solution

Adjust the embedded runner's orphan-trailing-user repair so older orphan content cannot outrank the latest trigger, while preserving provider-safe role ordering and adding regression coverage for latest-prompt-wins.

## Reproduction Assessment

No, not as a high-confidence end-to-end live channel/model run. The source path is reproducible: current main prepends the older orphan leaf before the latest prompt, removes that leaf, and records the same repair log shape reported here.

## Solution Assessment

Unclear because there is no proposed patch yet. The narrow maintainable solution is to change prompt shaping or pruning in the embedded-runner orphan repair so the latest triggering prompt remains authoritative without adding a queue mode or config knob.
