## Summary

Keep open: current main still has a source-backed orphan-user repair path that prepends the older orphaned user turn ahead of the latest trigger, matching the reported stale-reply shape and conflicting with the documented latest-input steering contract.

## What This Changes

The issue reports that queued/orphaned user-message repair can let an older unresolved user turn steer a later run even though the submitted prompt is the latest instruction.

## Best Possible Solution

Adjust the embedded runner's orphan-trailing-user repair so older orphan content cannot outrank the latest trigger while preserving provider-safe role ordering, then add regression coverage for latest-prompt-wins behavior.

## Reproduction Assessment

No, not as a high-confidence end-to-end live channel/model run. The source path is reproducible on current main: the helper prepends the older orphaned user leaf before the latest prompt, removes that leaf, and records the same repair log shape reported here.

## Solution Assessment

Unclear because there is no proposed patch yet. The narrow maintainable solution is to change prompt shaping or pruning in the embedded-runner orphan repair so the latest triggering prompt remains authoritative without adding a queue mode or config knob.
