## Summary

Keep open: current main still has the orphan-user repair path that prepends older orphan text before the latest triggering prompt, removes the leaf, and submits that merged prompt to the model. The open transcript-persistence work is adjacent, but it does not by itself resolve this issue's latest-trigger authority problem.

## What This Changes

The issue reports that queued/orphaned user-message merging can let an earlier unresolved user turn steer a later run and produce a stale reply.

## Best Possible Solution

Adjust the embedded-runner orphan-user repair so provider-safe role ordering is preserved without letting older orphan text outrank the latest trigger, then update the focused helper/runtime tests and coordinate with the adjacent persistence work.

No, not as a high-confidence end-to-end channel/model run. Source inspection on current main shows the risky path: older orphan text is prepended before the latest prompt, the leaf is removed, and the same repair warning is emitted.
