## Summary

Keep this PR open. Current main still lacks the SPA WebView2 bridge, but the latest PR head introduced a blocking handshake regression: it sends `ready` before listener registration and then sends a second `ready`, contradicting the PR contract and its own tests.

## What This Changes

The PR adds a Control UI WebView2 bridge module and tests, then wires `OpenClawApp` connect/disconnect lifecycle to initialize and clean up native draft injection.

## Best Possible Solution

Restore a listener-first, single `ready` handshake, add the required changelog entry, keep the narrow `draft-text` surface, and land only after targeted UI tests plus a WebView2 smoke pass.

## Reproduction Assessment

Yes, for the introduced PR bug: inspect PR head `807165cb29ca0e8abe3d3f80e464f28090564c79` lines 64-66 or run `pnpm test ui/src/ui/app-native-bridge.test.ts`; the added tests expect listener registration before exactly one ready handshake.

## Solution Assessment

No, not as latest head stands. The intended minimal bridge is sound, but `initNativeBridge` should register the listener first and send exactly one lint-safe ready handshake.
