## Summary

Keep open: current main and the latest release still have the unguarded final recovery rename that can surface ENOENT into the Telegram isolated polling drain failure counter. The report is narrow, source-backed, and queueable rather than an obsolete or duplicate cleanup item.

## What This Changes

Handle ENOENT/EEXIST races while recovering stale Telegram `.json.processing` spool claims so harmless concurrent removals do not log drain failures or increment failure counters.

## Best Possible Solution

Add a narrow recovery-path guard that treats vanished or already-restored processing files as benign, mirrors the existing release-claim ENOENT/EEXIST behavior, and preserves the existing spool ordering semantics.

## Reproduction Assessment

Unclear for a deterministic live setup, but yes at source level: current main can throw ENOENT from the final recovery rename after `stat`, `readJsonFileWithFallback`, or `pathExists` observe an entry that another drain path removes before rename.

## Solution Assessment

Yes, the narrowest maintainable fix is to handle ENOENT/EEXIST at the final recovery handoff like `releaseTelegramSpooledUpdateClaim`, with a focused regression test rather than broader Telegram spool redesign.

## Repair Work Prompt

Fix the Telegram isolated polling spool recovery ENOENT race reported in https://github.com/openclaw/openclaw/issues/87847. In `extensions/telegram/src/telegram-ingress-spool.ts`, make `recoverStaleTelegramSpooledUpdateClaims` treat final recovery handoff races as benign: if the `.json.processing` source disappears before rename, continue without incrementing recovered; if the pending destination already exists, remove the processing file as `releaseTelegramSpooledUpdateClaim` does. Consider skipping when `shouldRecover` is enabled and `parseSpooledUpdate` returns null so a vanished or unreadable claim is not blindly renamed. Add focused regression coverage in `extensions/telegram/src/telegram-ingress-spool.test.ts` for ENOENT and EEXIST from the final recovery rename/path handoff. Do not broaden this into the older stale-backlog issues, do not add config, and do not edit `CHANGELOG.md`.
