## Summary

Keep open: this PR addresses a valid source-reproducible cron ID collision, but it generates IDs after sidecar state has already been merged, so existing state under an `undefined` key can still be copied into every id-less job before persistence. Contributor real behavior proof is also still absent.

## What This Changes

The PR adds load-time UUID generation and persistence for cron jobs without `id` or legacy `jobId`, guards timer outcome application from invalid IDs, and updates cron tests, docs, and changelog.

## Best Possible Solution

Canonicalize missing IDs before state sidecar merge, or ignore invalid sidecar keys during backfill, add regression coverage for an existing `jobs-state.json` keyed by `undefined`, then land with redacted Gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main can load multiple persisted cron jobs without `id` or `jobId`, merge sidecar state by `job.id`, and later match an undefined outcome ID to an id-less row.

## Solution Assessment

No. The direction is right, but generating IDs in `ensureLoaded()` after `loadCronStore()` has merged sidecar state leaves existing polluted state files incorrectly copied; move canonicalization earlier or reject invalid sidecar keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Backfill IDs before merging sidecar state:** `src/cron/service/store.ts:61-65`
  - body: The UUID is generated in `ensureLoaded()` only after `loadCronStore()` has already merged `jobs-state.json` by `job.id`. On an install that previously ran id-less jobs, the state sidecar can contain a string `undefined` key; this load will copy that same entry into each id-less job, and the later persist writes polluted state under each generated ID. Generate IDs before sidecar merge, or ignore invalid sidecar keys during backfill, and cover that existing-sidecar case.
  - confidence: 0.78
