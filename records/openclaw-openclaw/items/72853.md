## Summary

Keep this PR open. The cron ID backfill and invalid-outcome guard address a real bug, but the proposed load-time persistence path would overwrite `jobs.json.bak` during automatic normalization/backfill, regressing the existing backup-preservation contract from #35195/#35234.

## What This Changes

The PR generates and persists stable IDs for cron jobs missing both `id` and legacy `jobId`, guards cron outcome application against invalid job IDs, and adds cron tests, docs, and changelog coverage for #72849.

## Best Possible Solution

Keep the focused cron fix, but change the automatic missing-ID backfill persistence to preserve existing backups and add regression coverage for that path before merge.

## Reproduction Assessment

Yes. For the original bug, load multiple cron jobs missing both `id` and `jobId` and let an outcome with an undefined job ID apply; current main can match the first id-less row. For the PR-introduced blocker, the existing #35195 backup test pattern shows why normalization/backfill persistence must not overwrite `.bak`.

## Solution Assessment

No as submitted. The backfill-at-load plus timer guard is the right narrow design, but the automatic persist should call the existing skip-backup path and cover that behavior in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve cron backups during ID backfill:** `src/cron/service/store.ts:154`
  - body: This automatic load-time normalization calls `persist(state)` without `skipBackup`. Since `saveCronStore` copies `jobs.json` to `.bak` whenever config content changed, generating IDs during Gateway load can overwrite an existing pre-edit backup, regressing the #35195 contract that normalization persistence leaves `.bak` intact. Pass `skipBackup: true` here and add missing-ID coverage to the backup regression test.
  - confidence: 0.92
