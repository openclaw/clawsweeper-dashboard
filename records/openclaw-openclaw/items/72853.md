## Summary

Keep open: the PR targets a valid source-reproducible cron state-collision bug, but the diff still backfills IDs after sidecar state has already been merged and it lacks contributor-provided real Gateway proof.

## What This Changes

The PR generates and persists stable IDs for hand-authored cron jobs without `id`/`jobId`, adds an invalid-ID outcome guard, and updates cron tests, docs, and changelog.

## Best Possible Solution

Move missing-ID canonicalization before sidecar state merge or drop invalid sidecar keys during migration, add regression coverage for an existing `jobs-state.json` `undefined` key, then land with redacted Gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load multiple persisted cron jobs missing both `id` and `jobId`, then merge sidecar state or apply an outcome keyed by a missing job ID, and current lookups can collapse state by `undefined`.

## Solution Assessment

No. The PR is pointed at the right bug, but generating IDs in `ensureLoaded()` after `loadCronStore()` has merged sidecar state is not the narrowest safe fix; canonicalize earlier or reject invalid sidecar keys.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Backfill IDs before merging sidecar state:** `src/cron/service/store.ts:61-65`
  - body: The UUID backfill runs in `ensureLoaded()` only after `loadCronStore()` has already merged `jobs-state.json` by `job.id`. If an affected install already has a sidecar entry under `undefined`, each id-less job can inherit that same polluted state before the new IDs are persisted, so the migration writes the wrong runtime state under multiple generated IDs. Move ID canonicalization before sidecar merge or ignore invalid sidecar keys, and cover the existing-sidecar case.
  - confidence: 0.82
