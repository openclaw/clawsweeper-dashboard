## Summary

Keep this PR open, but do not merge it as-is. The bug and the proposed direction are valid, and prior Greptile/Codex comments correctly identified this as a focused fix, but the current patch generates missing IDs only after `loadCronStore()` has already merged sidecar runtime state by `job.id`, leaving the first-upgrade path able to preserve polluted `jobs-state.json` data under the new UUIDs.

## What This Changes

The PR adds load-time UUID backfill and persistence for cron jobs missing both `id` and legacy `jobId`, guards timer outcome application for invalid result job IDs, and updates store coverage, cron docs, and the changelog.

## Best Possible Solution

Land a corrected cron fix that canonicalizes missing job identities before both sidecar state merging and timer outcome application, preserves legacy `jobId` behavior, and includes regression coverage for id-less jobs with a preexisting `jobs-state.json` undefined-key entry.

## Reproduction Assessment

Yes. A high-confidence reproduction is a `jobs.json` with two cron rows missing both `id` and `jobId`, plus a `jobs-state.json` containing a `jobs.undefined` runtime state entry; the current PR still lets `loadCronStore()` merge that undefined-key state before assigning UUIDs.

## Solution Assessment

No. The overall direction is right, but the implementation is not the best fix until missing-ID canonicalization happens before the `loadCronStore()` state-file lookup or that lookup explicitly refuses missing IDs.

## Repair Work Prompt

Repair PR #72853 for the cron missing-job-id bug tracked by #72849. The current diff generates UUIDs in `ensureLoaded()` after `loadCronStore()`, but `loadCronStore()` has already merged `jobs-state.json` by `stateFile.jobs[job.id]`; id-less rows can all consume an existing `jobs.undefined` entry and then persist polluted runtime state under generated UUIDs. Move missing/blank ID generation into the cron store load path before state-file merge, or add an equivalent pre-merge canonicalization seam so state is never merged through undefined. Preserve legacy `jobId` normalization behavior and do not turn legacy `jobId` persistence into an unrelated startup migration unless needed for this fix. Add regression coverage for two jobs missing both `id` and `jobId` plus a preexisting `jobs-state.json` undefined-key entry, proving backfilled jobs do not both inherit that sidecar state. Keep the timer invalid-ID guard. Update docs/changelog only if the user-visible behavior wording changes.
