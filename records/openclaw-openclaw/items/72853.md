## Summary

Keep open. Current main still has the cron missing-ID collision, and this PR is the right focused implementation candidate, but the patch needs one correctness revision because the new UUID backfill runs after sidecar state has already been merged by the old missing ID.

## What This Changes

The PR backfills and persists stable IDs for cron jobs missing both `id` and legacy `jobId`, adds a timer guard for invalid run outcome IDs, and updates store tests, cron docs, and the changelog.

## Best Possible Solution

Revise the cron fix so persisted job identity is canonical before any sidecar state lookup, or make sidecar state merging ignore invalid IDs, while keeping the defensive timer guard and adding regression coverage for multiple id-less jobs with both inline and sidecar runtime state.

## Reproduction Assessment

Yes. A high-confidence reproduction is a `jobs.json` with two jobs missing both `id` and `jobId` plus a `jobs-state.json` entry under the `undefined` key; the current load path merges that sidecar state before the PR's generated IDs would exist.

## Solution Assessment

No. The direction is right, but generating IDs in `ensureLoaded()` is too late for the sidecar state merge; the safer solution canonicalizes IDs before `loadCronStore()` indexes state by job ID, or guards that merge from missing IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Backfill IDs before sidecar state merge:** `src/cron/service/store.ts:62-66`
  - body: This UUID generation happens after `loadCronStore()` has already merged `jobs-state.json` via `stateFile.jobs[job.id]`. If an existing store has id-less jobs and a sidecar entry under the `undefined` key, that state is copied into the id-less rows before the new IDs are generated, then persisted under the new IDs. Generate or normalize IDs before the state-file merge, or make the merge ignore missing IDs, and cover that sidecar-state case in the regression test.
  - confidence: 0.87
