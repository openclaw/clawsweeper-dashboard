## Summary

Keep this PR open. Current main still has the cron missing-id collision, and the PR is a focused implementation candidate with no blocking functional or security findings from static review and public exact-head check status.

## What This Changes

The PR backfills and persists stable IDs for cron jobs missing both `id` and legacy `jobId`, rejects invalid cron outcome IDs before state lookup, and adds cron tests, docs, and a changelog entry.

## Best Possible Solution

Land this focused cron fix after maintainer review, keeping the ID repair in the cron store hydration seam, the defensive outcome guard in the timer path, and the operator-facing docs/changelog note.

## Reproduction Assessment

Yes. The linked bug gives a high-confidence path: create multiple persisted cron jobs without `id` or `jobId`, restart the Gateway, and let a cron run produce an outcome; current main source confirms the id-less load and unsafe outcome lookup.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix I found: make hand-authored persisted rows satisfy the existing `CronJob.id` contract at load time and reject any remaining invalid outcome ID before it can update the wrong row.
