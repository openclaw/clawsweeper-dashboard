## Summary

Keep open: the linked cron bug is valid, but this PR still backfills IDs after split sidecar state has already been merged, so an existing `jobs-state.json` entry keyed as `undefined` can be copied under the new generated IDs; the external PR also still lacks real Gateway proof.

## What This Changes

The PR generates and persists UUIDs for cron jobs missing both `id` and `jobId`, adds an invalid-job-ID timer guard, and updates cron tests, docs, and changelog.

## Best Possible Solution

Canonicalize missing cron IDs before split sidecar state merge, or drop invalid sidecar keys during migration, then cover an existing `jobs-state.json` `undefined` key and land with redacted Gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: load multiple persisted cron jobs missing both `id` and `jobId`, then let split sidecar merge or an outcome lookup use the missing job ID path; current main still reaches `stateFile.jobs[job.id]` and `entry.id === result.jobId`.

## Solution Assessment

No. The PR targets the right bug, but generating IDs in `ensureLoaded()` after `loadCronStore()` has merged sidecar state is not the narrowest safe fix; canonicalization needs to happen before state merge or invalid sidecar keys need to be rejected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Backfill IDs before merging sidecar state:** `src/cron/service/store.ts:61-65`
  - body: The UUID backfill runs in `ensureLoaded()` only after `loadCronStore()` has already merged `jobs-state.json` by `job.id`. If an affected install already has a sidecar entry under `undefined`, each id-less job can inherit that same polluted state before the new IDs are persisted, so the migration writes the wrong runtime state under multiple generated IDs. Move ID canonicalization before sidecar merge or ignore invalid sidecar keys, and cover the existing-sidecar case.
  - confidence: 0.82
