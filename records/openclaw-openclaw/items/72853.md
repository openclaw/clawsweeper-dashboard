## Summary

Keep this PR open, but it needs a focused fix before merge. The branch addresses the main missing-id timer collision, but it generates and persists IDs only after `loadCronStore()` has already merged the split runtime state by the possibly undefined `job.id`, so users who already have a `jobs-state.json` entry under `undefined` can have that polluted state copied and re-persisted under every newly generated ID.

## What This Changes

The PR backfills and persists stable IDs for persisted cron jobs missing both `id` and legacy `jobId`, guards cron outcome application from invalid job IDs, and adds cron tests, docs, and a changelog entry for #72849.

## Best Possible Solution

Move missing/blank cron ID repair to the store-load layer before sidecar state lookup, or explicitly ignore sidecar state for id-less rows, then keep the timer outcome guard and document/test both fresh id-less jobs and existing `jobs-state.json` pollution.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs with neither `id` nor `jobId`; current main can apply an undefined run outcome to the first id-less row, and the split store can also merge `jobs-state.json` through an undefined `job.id`.

## Solution Assessment

No, not yet. The PR is the right narrow direction, but the backfill needs to happen before split-state merge or avoid applying sidecar state to id-less rows so the first fixed load does not persist already misattributed runtime state under new IDs.
