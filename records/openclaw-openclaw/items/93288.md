## Summary

Keep open: current main and the latest release still derive Microsoft Teams proactive send placement only from static `replyStyle` config, while existing shared `topLevel` message-action semantics are not carried into the Teams send context. The reporter's local workaround/proof is consistent with the current source gap, and no merged or open PR appears to own this per-call override yet.

## What This Changes

Request adds a per-call Microsoft Teams `topLevel` send override so proactive channel announcements can create new root posts while normal replies keep using configured thread behavior.

## Best Possible Solution

Implement Teams support for existing `topLevel` message-tool semantics as an explicit per-call override, preserving configured `replyStyle` behavior when omitted and avoiding a new config option.

## Reproduction Assessment

Yes, at source level: current Teams sends resolve proactive placement from static config only, while `topLevel` is recognized by shared message-action threading but not consumed by Teams. I did not run a live Teams tenant send in this read-only review.

## Solution Assessment

Mostly yes: a per-call `topLevel` override is the narrow maintainable direction because it reuses existing message-tool semantics instead of changing static config. The implementation should carry the suppression intent through core outbound and Teams send context rather than duplicating the whole send path.

## Repair Work Prompt

Implement support for per-call `topLevel: true` on Microsoft Teams `message` action `send` so proactive channel sends can force a new root post while omitted/false preserves existing `channels.msteams.replyStyle` resolution. Keep the change in the Teams/message-tool owner boundary and do not add a new config key. Audit the actual dispatch path before coding: shared message-tool schema/discovery, core outbound threading context, `msteamsOutbound` text/media/payload/adaptive-card delivery, `sendMessageMSTeams`, `sendAdaptiveCardMSTeams`, and `resolveMSTeamsSendContext`. The fix must carry explicit top-level suppression intent through whichever core/plugin path sends Teams text, media, and presentation payloads, then apply a Teams `replyStyle` override only for that per-call send; DMs and group chats should remain top-level as today. Add tests proving `topLevel: true` uses `replyStyle: "top-level"` even when config would choose `thread`, and proving omission preserves config-derived thread behavior. Update Teams docs and message-tool discovery/schema text if `topLevel` becomes user-visible for Teams. Do not change `channels.msteams.replyStyle` defaults or unrelated Teams attachment threading work.
