## Summary

Keep open: current main and v2026.6.6 still lack a per-call Teams `topLevel` override, and https://github.com/openclaw/openclaw/pull/93292 is now the open implementation candidate for the same request. Closing this issue before that PR lands or is replaced would be premature.

## What This Changes

Request a per-call `topLevel` Microsoft Teams message-tool send override so proactive announcements can create new root posts while static `replyStyle: "thread"` remains the default for thread replies.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/93292, or a replacement, that carries `topLevel` through the existing Teams outbound send path, preserves omitted-flag behavior, defines media/presentation scope, and includes redacted live Teams proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 compute Teams proactive `replyStyle` only from stored reference/config, while shared message-action plumbing recognizes `topLevel` and Teams messenger already honors top-level once selected. I did not run a live Teams tenant send.

## Solution Assessment

Yes on direction, but not complete through the linked PR yet: a per-call `topLevel` override is narrower than a new config option and reuses existing message-tool semantics. The implementation should route the override through the existing outbound path and explicitly cover or reject media/presentation combinations.
