## Summary

Keep open: current main and the latest release still lack the Microsoft Teams per-call `topLevel` send override, and overlapping open PRs already own the implementation path.

## What This Changes

Requests a per-call Microsoft Teams `topLevel` send option so proactive channel announcements can post as new root threads while normal mention replies keep using configured thread replies.

## Best Possible Solution

Review and land one narrow MS Teams plugin PR that exposes `topLevel` for `send` and carries that intent through the Teams outbound/send-context path while preserving omitted-parameter behavior.

## Reproduction Assessment

Yes, at source level: current main exposes no Teams `topLevel` send parameter and derives proactive placement only from static `replyStyle` config. I did not run a live Teams tenant send in this read-only review.

## Solution Assessment

Yes, the per-call override direction is the narrow maintainable solution because static `replyStyle` cannot express both in-thread replies and proactive root posts in the same channel. The implementation should reuse the Teams outbound/send-context path rather than add another config knob.
