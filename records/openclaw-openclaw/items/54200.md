## Summary

Keep open: current main still skips sibling `~/.openclaw/moltbot.json` during config resolution and doctor preflight, so the reported upgrade layout can still strand the real config behind an empty or skeletal `openclaw.json`. The related closed issue only addressed the missing `gateway.mode` error wording, not this migration recovery gap.

## What This Changes

Recover or clearly warn when an upgrade leaves the primary config at `~/.openclaw/moltbot.json` while canonical `openclaw.json` is empty or incomplete.

## Best Possible Solution

Add a narrow doctor/startup recovery path for `~/.openclaw/moltbot.json` when canonical `openclaw.json` is missing, empty, or skeletal, with docs and regression tests that preserve `openclaw.json` as the canonical runtime contract.

## Reproduction Assessment

Yes, at source level. A home with `~/.openclaw/openclaw.json` as `{}` or otherwise missing `gateway.mode` plus a valid `~/.openclaw/moltbot.json` resolves the canonical file, skips the sibling legacy file, and reaches only the generic clobbered-config guard.

## Solution Assessment

Yes. A targeted doctor/startup diagnostic and repair is the narrow maintainable fix; re-adding `moltbot.json` to normal config candidate resolution would revive the legacy fallback that current architecture intentionally removed.
