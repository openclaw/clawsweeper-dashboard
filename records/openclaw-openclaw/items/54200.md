## Summary

Keep open: current main still misses the reported migration layout, and the closed related wording fix only improves the `gateway.mode` symptom rather than recovering or naming the sibling legacy config.

## What This Changes

The issue asks OpenClaw to warn or recover when a legacy-named `$HOME/.openclaw/moltbot.json` remains beside a skeletal canonical `openclaw.json` after the v2026.3.22 migration.

## Best Possible Solution

Add a narrow recovery path that detects `$HOME/.openclaw/moltbot.json` when canonical config is missing, empty, or skeletal, surfaces a clear gateway/doctor hint, provides a safe doctor repair, and documents the migration without restoring broad normal runtime probing.

## Reproduction Assessment

Yes at source level. A home with `$HOME/.openclaw/openclaw.json` as `{}` or otherwise missing `gateway.mode` and a valid `$HOME/.openclaw/moltbot.json` will resolve the canonical file, skip the sibling legacy file, and produce only the generic suspicious-config gateway guidance.

## Solution Assessment

Yes, the best fix is a targeted doctor/startup diagnostic and repair path. Re-adding `moltbot.json` to normal config candidate resolution would risk reviving stale legacy-file behavior that earlier cleanup intentionally removed.
