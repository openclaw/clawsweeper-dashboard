## Label Justifications

- `P1`: This is an upgrade regression that can leave real users unable to start the gateway against their actual channel and agent configuration.
- `impact:data-loss`: The user's full config remains on disk but is silently ignored, making channels, agents, bindings, models, hooks, and plugins appear lost.
- `impact:session-state`: Loading a skeletal config can make existing agents, sessions, and instances disappear from UI and status surfaces.
- `impact:crash-loop`: The reported service repeatedly restarted after loading an incomplete config, even though newer generated systemd units mitigate restart storms for config failures.

## Summary

Keep open: current `main` and `v2026.5.20` still ignore a sibling `~/.openclaw/moltbot.json` when `openclaw.json` already exists, while the related closed issue only addressed the missing-`gateway.mode`/restart-storm symptom rather than this migration recovery gap.

## What This Changes

The issue requests recovery and diagnostics for upgrades where `~/.openclaw/moltbot.json` remains beside a skeletal `openclaw.json` and is no longer loaded.

## Best Possible Solution

Add a narrow `doctor --fix` recovery and gateway diagnostic path for sibling `~/.openclaw/moltbot.json` only when canonical `openclaw.json` is missing, empty, or skeletal, while keeping `openclaw.json` canonical.

Yes, at source level. A home state with skeletal `~/.openclaw/openclaw.json` plus a valid sibling `~/.openclaw/moltbot.json` still resolves and diagnoses only the canonical file while skipping the legacy source.

## Solution Assessment

Yes. The maintainable fix is targeted doctor recovery plus startup guidance for this upgrade state, not re-adding `moltbot.json` as a normal config fallback.
