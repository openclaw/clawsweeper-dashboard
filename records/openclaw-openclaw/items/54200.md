## Summary

Keep open: current main still resolves the canonical `openclaw.json` and remaining `clawdbot.json` aliases, but it does not detect or recover the reported sibling `$HOME/.openclaw/moltbot.json` layout. The related closed issue fixed the generic `gateway.mode` error wording, not the migration recovery gap.

## What This Changes

Request recovery and clearer diagnostics for upgrades where `$HOME/.openclaw/openclaw.json` is empty or skeletal while the real legacy config remains in `$HOME/.openclaw/moltbot.json`.

## Best Possible Solution

Add a narrow doctor/startup recovery path that detects `$HOME/.openclaw/moltbot.json` only when canonical `openclaw.json` is missing, empty, or skeletal, then guide or repair migration while preserving `openclaw.json` as the canonical runtime contract.

## Reproduction Assessment

Yes, at source level. A home with an empty or skeletal `$HOME/.openclaw/openclaw.json` plus a valid sibling `$HOME/.openclaw/moltbot.json` still resolves and reads the canonical file, skips the sibling legacy file, and reaches only generic clobbered-config handling.

## Solution Assessment

Yes, the maintainable fix is a targeted doctor/startup diagnostic and repair path. Re-adding `moltbot.json` to ordinary config candidate resolution would be broader and would undo the intended legacy fallback removal.
