## Label Justifications

- `P1`: The report describes an upgrade regression that can leave users in a blank UI and gateway crash-loop until config is recovered.
- `impact:data-loss`: The active runtime silently ignores the user's real config, effectively dropping channels, agents, bindings, models, hooks, and plugins from the running gateway.
- `impact:crash-loop`: The observed failure mode is a gateway restart loop caused by loading an empty or skeletal config with missing `gateway.mode`.
- `impact:session-state`: The blank Control UI and missing agents/sessions/instances are session and runtime-state visibility failures caused by the config split.

## Summary

Keep open: current main still ignores the reported sibling `~/.openclaw/moltbot.json` layout when `openclaw.json` exists but is empty or skeletal. The related closed issue improved the missing-`gateway.mode` diagnosis, but did not add detection or repair for the legacy sibling config that contains the real channels, agents, and session-facing configuration.

## What This Changes

The issue asks OpenClaw to detect or recover a legacy `moltbot.json` left inside `~/.openclaw/` after upgrade, and to clarify the migration/error guidance for that layout.

## Best Possible Solution

Add a narrow doctor/preflight recovery path that detects `$HOME/.openclaw/moltbot.json` only when canonical `openclaw.json` is missing, empty, or skeletal, then guides or repairs migration while keeping `openclaw.json` as the canonical runtime config.

## Reproduction Assessment

Yes, at source level. A home with an empty or skeletal `$HOME/.openclaw/openclaw.json` plus a valid sibling `$HOME/.openclaw/moltbot.json` still resolves the canonical file, skips the sibling legacy file, and only reaches generic clobbered-config handling.

## Solution Assessment

Yes, the best fix is a targeted doctor/preflight diagnostic and repair path. Re-adding `moltbot.json` to ordinary config resolution would be a broader compatibility rollback and is not the maintainable boundary.
