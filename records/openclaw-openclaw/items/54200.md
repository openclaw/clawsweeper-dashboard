## Summary

Keep this open. Current main still ignores a legacy-named `$HOME/.openclaw/moltbot.json` beside a missing or skeletal canonical config, and the later gateway/config recovery work only covers last-known-good or invalid-config recovery, not this sibling legacy filename migration gap.

## What This Changes

The issue asks OpenClaw to warn, diagnose, and safely recover when an upgraded install has its real config in `$HOME/.openclaw/moltbot.json` while the canonical `openclaw.json` is missing, empty, or skeletal.

## Best Possible Solution

Add a narrow recovery path: detect `$HOME/.openclaw/moltbot.json` when the canonical config is missing, empty, or skeletal; surface a gateway/doctor hint that names the nearby legacy file; provide a safe `openclaw doctor --fix` repair or backup-assisted copy/merge path; and update migration/troubleshooting docs and the changelog without adding `moltbot.json` back to normal runtime config candidate resolution.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a home directory with `$HOME/.openclaw/openclaw.json` as `{}` or another skeletal config and a valid `$HOME/.openclaw/moltbot.json`: current path resolution selects the canonical file, doctor preflight returns early, and gateway startup reports missing `gateway.mode` without naming the real sibling config.

## Solution Assessment

Yes, but the best fix is a targeted doctor/diagnostic repair rather than restoring broad legacy runtime probing. The current startup wording and last-known-good recovery are useful adjacent fixes, but they do not solve the reported migration layout.
