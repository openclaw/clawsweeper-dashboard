## Label Justifications

- `P1`: The report describes an upgrade regression that can leave existing users with a crash-looping gateway and blank agent/session UI.
- `impact:data-loss`: The user config remains on disk but is silently ignored at runtime, making channels, agents, bindings, models, hooks, and plugins appear lost.
- `impact:crash-loop`: The active empty or skeletal config can make gateway startup fail repeatedly because `gateway.mode` is missing.
- `impact:session-state`: Starting from the wrong skeletal config can leave the Control UI with no agents, sessions, or instances visible.

## Summary

Keep open: current main and the latest shipped release still ignore a sibling `~/.openclaw/moltbot.json` when `openclaw.json` exists but is empty or skeletal. The related closed report improved the generic `gateway.mode` error wording, but it did not add migration recovery for the ignored sibling config.

## What This Changes

The issue requests upgrade recovery and clearer diagnostics when a valid `~/.openclaw/moltbot.json` is left beside an empty or skeletal canonical `openclaw.json`.

## Best Possible Solution

Add a narrow doctor/startup diagnostic and repair path for `~/.openclaw/moltbot.json` only when canonical `openclaw.json` is missing, empty, or skeletal, while preserving `openclaw.json` as the canonical runtime contract.

## Reproduction Assessment

Yes, at source level. With `~/.openclaw/openclaw.json` present, the resolver checks only the canonical file and `clawdbot.json`, and doctor preflight returns before checking any sibling `moltbot.json`.

## Solution Assessment

Yes. A targeted doctor/startup recovery path is the narrowest maintainable fix; restoring `moltbot.json` to ordinary config resolution would revive broader legacy fallback behavior that was intentionally removed.
