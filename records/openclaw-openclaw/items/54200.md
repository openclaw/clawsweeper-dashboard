## Label Justifications

- `P1`: The report describes an upgrade regression that can leave existing users with a crash-looping gateway and blank agent/session UI.
- `impact:data-loss`: The user config is still on disk but is silently ignored at runtime, making channels, agents, bindings, models, hooks, and plugins appear lost.
- `impact:crash-loop`: The reported gateway repeatedly exits because the active empty config lacks `gateway.mode`.
- `impact:session-state`: The Control UI can show no agents, sessions, or instances when OpenClaw starts against the wrong skeletal config.

## Summary

Keep open: current main and the latest release still ignore a sibling `~/.openclaw/moltbot.json` when `openclaw.json` exists but is empty or skeletal. The related closed report fixed the generic `gateway.mode` wording, but not this migration recovery gap.

## What This Changes

The issue asks OpenClaw to detect or repair a legacy `moltbot.json` config left inside `~/.openclaw/` after the `v2026.3.22` migration change.

## Best Possible Solution

Add a narrow doctor/startup recovery path that detects `~/.openclaw/moltbot.json` only when canonical `openclaw.json` is missing, empty, or skeletal, then guides or repairs migration while preserving `openclaw.json` as the canonical runtime contract.

## Reproduction Assessment

Yes, at source level. A home with an empty or skeletal `~/.openclaw/openclaw.json` plus a valid sibling `~/.openclaw/moltbot.json` still resolves the canonical file, skips the sibling legacy file, and reaches only generic clobbered-config handling.

## Solution Assessment

Yes, the best fix is a targeted doctor/startup diagnostic and repair path. Re-adding `moltbot.json` to normal config candidate resolution would be broader than the reported upgrade recovery case.
