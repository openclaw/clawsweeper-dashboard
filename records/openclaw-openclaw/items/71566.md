## Summary

Keep open: current main still lacks Gateway directory RPCs, and the proposed PR is not merge-ready because it advertises a new public Gateway surface without the protocol contract follow-through and has unresolved directory list semantics.

## What This Changes

The PR adds Gateway handlers for `directory.self`, `directory.peers.list`, and `directory.groups.list`, registers them in method discovery, and maps them to `operator.read`.

## Best Possible Solution

Land this only as a complete Gateway protocol addition: typed params/result schemas and exports, handler/method-discovery/scope tests, protocol docs, directory list behavior aligned with the chosen cache/live semantics, and the required single-line changelog entry with contributor credit.

## Reproduction Assessment

Not applicable as a bug reproduction for the original feature request. The review issues are high-confidence from source inspection: current main lacks these Gateway RPCs, the PR advertises methods without protocol schema/docs coverage, and current resolver tests show the live-on-miss behavior the handlers should either preserve or explicitly supersede.

## Solution Assessment

No, not as-is. The direction is reasonable, but advertising public Gateway RPCs before the typed contract, docs, tests, changelog, and final directory list semantics land is not the narrowest maintainable solution.
