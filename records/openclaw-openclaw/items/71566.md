## Summary

Keep open. Current main does not implement these Gateway directory RPCs, and the PR is useful but incomplete: it advertises public methods without typed protocol schema/docs/validators, reads disk config instead of the active Gateway config, and drops live-on-miss directory lookup behavior for channels that have both config-backed and live directory adapters.

## What This Changes

The PR adds `directory.self`, `directory.peers.list`, and `directory.groups.list` Gateway handlers, registers them in method discovery, and maps them to `operator.read`.

## Best Possible Solution

Land this only as a complete Gateway protocol addition: typed params/result schemas and validators, focused handler/method-discovery/scope/protocol tests, docs for the new RPCs, cache-first plus live-on-miss directory semantics, active Gateway runtime config usage, and the required single-line changelog entry with contributor credit.

## Reproduction Assessment

Not applicable as an original bug reproduction; this is a feature PR. The review findings are reproducible by source inspection of PR head `ce3e3a2` against current main: the new methods are advertised without protocol schemas/docs, and the handlers diverge from current Gateway config and directory fallback patterns.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but a public Gateway RPC surface should land with protocol contracts, validation, docs, tests, and directory semantics aligned with the existing resolver before it is advertised.
