## Summary

Keep open. Current main does not already implement these Gateway directory RPCs, and the PR still has merge-blocking protocol and behavior gaps: it advertises public methods without typed protocol schemas and loses the existing live-on-miss directory lookup semantics. Earlier Greptile feedback about runtime error codes and one unsupported-capability message appears to have been addressed by follow-up commits, but the later protocol and fallback findings remain valid.

## What This Changes

The PR adds Gateway handlers for `directory.self`, `directory.peers.list`, and `directory.groups.list`, registers those methods in discovery, and maps them to `operator.read`.

## Best Possible Solution

Land this only as a complete Gateway protocol addition: add typed params/result schemas and validators, document the new RPCs, add focused handler/method-discovery/scope/protocol tests, preserve cache-first plus live-on-miss lookup behavior for peers and groups, and include the required single-line changelog entry with contributor credit.

## Reproduction Assessment

Not applicable as a bug reproduction for the original feature request. The review issues are high-confidence from source inspection: current main’s resolver and tests show live-on-miss semantics, while the PR file list advertises new Gateway methods without adding protocol schema/docs/test coverage.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but the maintainable fix needs protocol contract follow-through and directory lookup semantics aligned with the existing resolver before this public Gateway surface is advertised.
