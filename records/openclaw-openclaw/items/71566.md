## Summary

Keep open. Current main still lacks the Gateway directory RPC surface, and this PR is the active implementation candidate, but the branch should not merge yet because it advertises new public methods without the typed Gateway protocol schema/validator follow-through and does not preserve the existing cache-first, live-on-miss directory lookup behavior. Earlier Greptile feedback about runtime error codes and unsupported-method copy appears to have been addressed in later PR commits.

## What This Changes

The PR adds `directory.self`, `directory.peers.list`, and `directory.groups.list` Gateway handlers, registers them in method discovery, and maps them to `operator.read`.

## Best Possible Solution

Land this as a complete Gateway protocol addition: define typed directory params/result schemas and validators, document the new methods, add focused handler/method-discovery/scope/protocol tests, and preserve cache-first plus live-on-miss lookup behavior where live directory adapters exist.

## Reproduction Assessment

Not applicable as a feature PR. Source inspection gives a high-confidence gap and review path: current main lacks the Gateway directory handlers/method/scope entries, while the PR diff adds them but misses protocol contract coverage and live fallback behavior.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but the narrow maintainable solution needs protocol schema follow-through and directory lookup semantics aligned with the existing resolver before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Add protocol schemas before advertising directory methods:** `src/gateway/server-methods-list.ts:18-20`
  - body: These entries make the new RPCs visible in Gateway method discovery, but the PR does not add params/result schemas, validators, or `ProtocolSchemas` exports. Gateway methods are part of the public wire contract, so clients cannot validate or generate against this advertised surface until the typed protocol contract lands with it.
  - confidence: 0.86
- **[P2] Fall back to live directory lookups after empty config results:** `src/gateway/server-methods/directory.ts:126`
  - body: For channels that expose both config-backed and live directory adapters, this chooses `listPeers` once and never calls `listPeersLive` when the config-backed list returns no matches. The existing resolver does config/cache first and then live-on-miss; without that fallback, an operator query can return an empty list even though the channel live adapter has results. The same issue applies to the groups handler below.
  - confidence: 0.78
