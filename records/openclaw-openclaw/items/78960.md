## Summary

Keep open: this PR addresses a real Qwen explicit-config normalization bug, but the current patch is over-broad because it also advertises `qwen3.6-plus` in the built-in Coding Plan catalog contrary to the current conservative provider policy.

## What This Changes

The PR removes Qwen provider filtering so `qwen3.6-plus` remains in Coding Plan provider model lists and changes the Qwen catalog helper to treat all base URLs as supported.

## Best Possible Solution

Keep the explicit-config fix, but preserve Coding Plan built-in catalog suppression unless maintainers explicitly choose the broader provider-truth policy.

## Reproduction Assessment

Yes, from source inspection: the models.json planning path applies provider-specific normalization, and Qwen currently removes `qwen3.6-plus` from Coding Plan `providerConfig.models`. I did not run a live gateway because this review was read-only.

## Solution Assessment

No. Removing the explicit-model normalization filter is the narrow fix, but making `isQwen36PlusSupportedBaseUrl` always true also changes built-in catalog policy that current docs, tests, and the merged replacement PR intentionally keep conservative.
