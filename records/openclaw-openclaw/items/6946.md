## Summary

Keep open for maintainer product review. Current main provides early typing, acknowledgement reactions, status reactions, and draft streaming, but still lacks the requested standalone Telegram placeholder-message lifecycle. The related open presence PR uses typing plus phased reactions rather than a temporary message and is currently conflicting, so it is not yet a safe canonical replacement.

## What This Changes

Requests an opt-in Telegram placeholder message that appears during processing and is deleted after a successful reply or changed to an error on failure.

## Best Possible Solution

Choose one canonical Telegram-plugin-owned processing-feedback design, then implement it as opt-in typed configuration with explicit scope, concurrency, streaming, success cleanup, and failure behavior; either align the open presence PR to that decision or close it before landing a standalone-message path.

## Reproduction Assessment

Yes for the capability gap: current main clearly sends typing and manages adjacent reaction feedback without any configurable standalone placeholder that is later deleted or edited. This establishes missing requested behavior, not a regression against an existing contract.

## Solution Assessment

Unclear. The proposed lifecycle is technically feasible at the Telegram plugin boundary, but it is not yet demonstrably the best solution because maintainers must choose between standalone messages, phased reactions, or a broader acknowledgment model before creating another configuration path.
