## Summary

Keep open. Current main and the latest release still lack the requested standalone Telegram processing-placeholder message lifecycle, and the open related presence PR is an overlapping but blocked reaction/typing implementation rather than the same visible message-and-delete behavior.

## What This Changes

The issue requests an opt-in Telegram processing indicator that sends a visible placeholder while a turn is running, deletes it after a reply, and can show an error on failure.

## Best Possible Solution

Choose the intended Telegram feedback UX, then implement it as an opt-in Telegram-owned config/schema/docs change with dispatch wiring that tracks, deletes, and edits or replaces the placeholder with focused tests and live Telegram proof.

## Reproduction Assessment

Yes for the missing feature by source inspection: current main accepts typing, ack reactions, and status reactions, but has no config or dispatch lifecycle for a standalone Telegram placeholder message. This is not a bug reproduction; it proves the requested feature is absent.

## Solution Assessment

Unclear until maintainers choose the product shape. A Telegram-owned opt-in placeholder lifecycle is plausible, but it should be compared with the open presence-reaction PR and the broader cross-channel ack-text request before adding config surface.
