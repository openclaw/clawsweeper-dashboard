## Summary

Keep open. Current main has Telegram typing cues, ack/status reactions, and streaming/progress drafts, but it still does not expose or drive the requested standalone processing-placeholder lifecycle with delete-on-reply or edit-on-failure semantics. The related direct implementation PRs and broader waiting-tips PR are closed unmerged, and the live labels still mark this as a maintainer/product decision.

## What This Changes

The issue requests an opt-in Telegram processing indicator that posts a visible placeholder message while a turn is running, then deletes it on reply or edits it on failure.

## Best Possible Solution

Choose the intended bundled Telegram UX, then either document the existing progress-mode path as the supported answer or land an opt-in Telegram plugin change with typed config/schema/help/docs, placeholder tracking, delete-on-success, and edit-or-replace-on-failure coverage.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main is enough to show the requested standalone config and message lifecycle are absent, while adjacent progress streaming partially covers a different UX path.

## Solution Assessment

No, treating current main as fully implemented would be too broad. The maintainable path is a maintainer product choice: reuse/clarify Telegram progress streaming or add a separate opt-in processing-indicator lifecycle with focused tests.
