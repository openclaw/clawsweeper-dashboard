## Summary

Keep open. Current main still lacks the requested Telegram standalone processing-placeholder message lifecycle and config surface; the existing typing, ack/status reaction, and draft/streaming paths do not send a visible placeholder message that is later deleted or edited, so this remains a Telegram UX/config product decision.

## What This Changes

Requests an opt-in Telegram processing placeholder message that appears while a request is running and is deleted or edited when the reply finishes.

## Best Possible Solution

Choose the intended Telegram waiting UX, then implement an opt-in plugin-owned config/schema/docs path with authorized send/store/delete-or-edit lifecycle handling, focused tests, and real Telegram proof.

## Reproduction Assessment
