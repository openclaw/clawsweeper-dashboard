## Summary

Keep open. Current main still provides Telegram typing cues, ack/status reactions, draft streaming, and generic error replies, but it does not accept or drive a standalone processing-placeholder message that is sent while work runs and deleted or edited on completion/failure; the overlapping #40421 PR is still open and the direct implementations were closed unmerged, so this remains a maintainer UX/config decision.

## What This Changes

Request an opt-in Telegram processing indicator that sends a visible placeholder while a message is being processed and deletes or edits it when the turn completes.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land one opt-in Telegram plugin implementation with typed config/schema/help/docs, placeholder send/track/delete/edit wiring, and focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed keys are not accepted, and Telegram inbound/dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader #40421 waiting-tips design is the best product solution. The maintainable path is a maintainer UX/config decision first, followed by a Telegram plugin change without broad core work unless the chosen design needs a shared channel helper.
