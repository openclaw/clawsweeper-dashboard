## Summary

Keep open. Current main has adjacent Telegram typing, ack/status reactions, and preview/tool-progress messages, but it still lacks the requested opt-in standalone processing-placeholder lifecycle and config surface; the open #40421 waiting-tips PR overlaps the same UX choice while the direct implementation attempts #43021 and #43720 were closed unmerged.

## What This Changes

Add an opt-in Telegram processing indicator message that is sent while an inbound request is being processed, deleted on successful reply, and edited or replaced on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then implement one opt-in Telegram plugin path with typed config/schema/help/docs, placeholder send/track/delete/edit wiring, and focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed keys are not accepted, and Telegram inbound/dispatch inspection shows no immediate stored placeholder lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader #40421 waiting-tips design is the best product solution. The maintainable path is a maintainer UX/config decision first, followed by a Telegram plugin-scoped implementation rather than broad core work unless a shared helper is needed.
