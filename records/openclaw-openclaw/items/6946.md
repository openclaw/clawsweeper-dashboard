## Summary

Current main still lacks the requested standalone Telegram placeholder-message lifecycle, and no merged or safely viable canonical item replaces it. The open presence-indicator PR uses typing plus reactions, is currently conflicting, and does not implement delete-on-reply/edit-on-failure for a separate message; the broader acknowledgment-text issue also leaves the Telegram-specific product/config choice unresolved.

## What This Changes

Add an opt-in Telegram processing placeholder that is sent when work begins, deleted after a successful reply, and edited or replaced when processing fails.

## Best Possible Solution

Choose one Telegram processing-feedback contract, then implement an opt-in plugin-owned placeholder with a typed schema/help/docs surface, a tracked message handle, deletion only after successful final delivery, and a generic failure edit or replacement that does not expose internal errors.

## Reproduction Assessment

Not applicable. This is a well-specified new capability request, and source inspection gives high confidence that current main does not provide the requested standalone send/delete/edit lifecycle.

## Solution Assessment

Unclear. A Telegram-local opt-in placeholder is a maintainable solution, but maintainers must first choose it over the adjacent typing/reaction presence design and decide the safe failure text policy.
