## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-placeholder message lifecycle and config surface; the adjacent typing, ack-reaction, status-reaction, and draft-preview paths do not provide a visible message that is sent while processing and then deleted or edited on completion/failure. The overlapping #40421 PR remains the active product/config decision point, while the direct implementations #43021 and #43720 were closed unmerged.

## What This Changes

Request an opt-in Telegram processing indicator that sends a visible placeholder while a request runs, deletes it after a successful reply, and edits or replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then implement one opt-in Telegram plugin path with typed config/schema/help/docs, placeholder send/track/delete/edit wiring, and focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed keys are not accepted, and Telegram inbound/dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader #40421 waiting-tips design is the best product solution. The maintainable path is a maintainer UX/config decision first, followed by a Telegram plugin change without broad core work unless the chosen design needs a shared helper.
