## Summary

Keep open. Current main and the latest release still lack the requested standalone Telegram placeholder-message lifecycle and config surface; the related open presence PR at https://github.com/openclaw/openclaw/pull/80690 uses typing/reactions rather than a send/delete-or-edit message and is currently conflicting, while the direct implementation attempts were closed unmerged.

## What This Changes

Request to add an opt-in Telegram processing indicator that posts a visible placeholder while a message is being processed, then deletes or edits it when the turn completes or fails.

## Best Possible Solution

Choose the intended Telegram waiting UX, then land one opt-in Telegram plugin change with typed config/schema/help/docs, authorization-aware send/store/delete-or-edit lifecycle handling, focused regression tests, and redacted live Telegram proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source search, strict-schema inspection, docs, tests, and latest-release checks give high confidence that the requested placeholder lifecycle is absent.

## Solution Assessment

Unclear pending product decision. A minimal Telegram placeholder is technically plausible, but maintainers should decide whether it belongs as a separate Telegram option, a broader acknowledgement-text contract, or the existing reaction/typing presence direction before implementation.
