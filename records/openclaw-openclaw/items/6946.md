## Summary

Keep open. Current main has early Telegram typing, ack/status reactions, and progress-draft streaming, but it still does not expose or drive the requested standalone Telegram processing-placeholder message that is sent on receipt and deleted or edited on completion/failure.

## What This Changes

Request an opt-in Telegram processing indicator message, such as an hourglass, with send, delete-on-reply, and failure edit behavior.

## Best Possible Solution

Choose one first-class opt-in Telegram waiting UX, then implement it in the Telegram plugin with typed config/schema/help/docs, lifecycle tests, send/store/delete wiring, and failure edit-or-replace behavior.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source inspection gives high confidence that the exact standalone placeholder config and lifecycle are absent on current main.

## Solution Assessment

Unclear until maintainers choose the intended UX. A narrow processing indicator is technically feasible, but implementing it automatically would add new user-facing Telegram config and could conflict with existing progress-draft and reaction feedback paths.
