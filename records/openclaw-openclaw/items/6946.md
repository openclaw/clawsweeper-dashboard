## Summary

Keep open. Current main still has Telegram typing cues, ack reactions, status reactions, draft/preview reply handling, and generic fallback error messages, but it does not expose or drive a standalone Telegram processing-placeholder message that is sent while a turn runs and deleted or edited on completion. The issue also overlaps open PR #40421 and two closed direct attempts, so this needs maintainer UX/config direction rather than conservative cleanup closure.

## What This Changes

Request to add an opt-in Telegram processing indicator that sends a visible placeholder while an inbound message is processed, then deletes it on reply or edits it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed keys are not accepted, and Telegram inbound/dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader #40421 waiting-tips design is the best product solution. The maintainable path is a maintainer UX/config decision first, followed by a Telegram plugin implementation without broad core changes unless the chosen design needs a shared channel helper.
