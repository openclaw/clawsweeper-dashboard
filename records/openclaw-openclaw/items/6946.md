## Summary

Current main still provides typing and reaction-based progress feedback but not the requested temporary standalone Telegram message, and no merged or viable canonical replacement fully owns those semantics. The issue should remain open for a maintainer product/config decision because the open presence PR is conflicting and implements a different reaction-plus-typing UX, while the broader acknowledgment-text issue overlaps without replacing this Telegram-specific delete/edit lifecycle.

## What This Changes

Add an opt-in Telegram processing message that appears while a turn runs, is deleted after a successful reply, and is changed to an error on failure.

## Best Possible Solution

Choose one canonical Telegram processing-feedback model, then implement it inside the Telegram plugin with one typed opt-in configuration surface, explicit success/error/abort cleanup semantics, focused lifecycle tests, updated Telegram docs, and redacted native Telegram proof.

## Reproduction Assessment

Not applicable: current-main source clearly shows that the requested standalone processing-message lifecycle does not exist, but the item requests a new capability rather than reporting broken existing behavior.

## Solution Assessment

Unclear as proposed: the Telegram plugin is the correct ownership boundary, but maintainers must choose whether standalone placeholders, shared acknowledgment text, or reaction-and-typing presence is the permanent UX before accepting a new config contract.
