## What This Changes

The PR caches WhatsApp bot-authored outbound message metadata from the send-tracking path and adds quote-cache tests for group, direct-chat, and LID fallback cases.

## Best Possible Solution

Land this WhatsApp-plugin cache-at-send fix only after redacted live WhatsApp Desktop/group proof, or an explicit maintainer decision to accept the remaining real-client validation risk.

## Reproduction Assessment

Yes at source level: current `main` records bot-sent message IDs only for echo suppression, then self-echo filtering prevents inbound metadata caching, so a later quote lookup can miss and build `fromMe:false` with the replying group sender. A fresh live WhatsApp Desktop reproduction/proof has not been supplied.

## Solution Assessment

Yes, this appears to be the best code boundary: caching outbound metadata where the sent message id is first known complements the existing echo-suppression path and avoids guessing later in reply construction. The remaining gap is real Desktop rendering proof, not a different implementation layer.
