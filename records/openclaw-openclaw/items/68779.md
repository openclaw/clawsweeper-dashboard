## Summary

Keep open. Maintainer discussion explicitly kept the split browser existing-session stability work open, and current main still has fixed attach/readiness budgets, next-call-only recovery, and route limits; pageReady, doctor, and skill guidance are useful partial fixes but do not resolve multi-step stability.

## What This Changes

The issue requests stabilization of browser `profile="user"` existing-session automation by tuning timeouts, adding session recovery/diagnostics, and revisiting route/config limits.

## Best Possible Solution

Split this into measured attach/readiness budgets, observable reconnect/readiness states, in-flow session refresh semantics, and focused regression/live proof; handle public timeout config and route-limit expansion as separate product decisions.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection on current main shows the fixed attach-readiness window, next-call-only recovery, and short navigation guard that match the remaining concerns, but the multi-step `gateway closed (1000)` failure was not run against a loaded user profile.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this. The maintainable path is measured timeout tuning plus explicit reconnect/session-refresh behavior, diagnostics, and regression coverage for the affected existing-session path.
