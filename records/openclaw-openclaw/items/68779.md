## Summary

Keep open. Current main and v2026.5.18 include useful partial fixes for existing-session status, page readiness, click, and navigation timeout handling, but current source and maintainer discussion still leave the broader reconnect/readiness-budget work unresolved.

## What This Changes

The issue requests more stable browser existing-session automation by relaxing attach/readiness timeouts, adding Chrome MCP session recovery, improving diagnostics, and considering timeout configuration overrides.

## Best Possible Solution

Split the work into observable Chrome MCP attach/page-readiness states, measured timeout budgets, explicit in-flow session refresh semantics, and focused regression/live proof; treat public timeout config and route-limit expansion as separate product decisions.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the reported multi-step `gateway closed (1000)` failure on current main. Source inspection does confirm the remaining fixed budgets, next-call recovery semantics, and route limits that match the reported area.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this issue. The maintainable path is measured timeout tuning plus observable attach/page-readiness states, explicit reconnect or session-refresh behavior, and regression/live coverage for the existing-session path.
