## Summary

Keep open. Current main and v2026.5.18 have partial Chrome MCP existing-session improvements, but the remaining in-flow recovery, attach/readiness budget, and trace-backed acceptance criteria are still open in source and recent maintainer discussion; the linked FSB note also supports splitting attach, tab ownership, readiness, and action retry before tuning constants.

## What This Changes

Stabilize browser existing-session multi-step automation by revisiting Chrome MCP attach/readiness timeouts, session recovery, diagnostics, and current route limits.

## Best Possible Solution

Split the work into observable Chrome MCP attach/page-readiness states, measured timeout budgets, explicit in-flow session refresh semantics, and focused regression/live proof; handle public timeout config and route-limit expansion as separate product decisions.

## Reproduction Assessment

No, there is not a high-confidence live current-main reproduction of the reported multi-step gateway closed failure. Source inspection does show the fixed attach/readiness budgets, next-call-only recovery, route limits, and short navigation guard that match the remaining concerns.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this issue. The maintainable path is measured timeout tuning plus observable attach/page-readiness states, explicit reconnect or session-refresh behavior, and regression/live coverage for the existing-session path.
