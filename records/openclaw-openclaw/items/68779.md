## Summary

Keep open. Current main has useful diagnostics improvements, including pageReady status, but the reported existing-session instability is not fully solved: attach/readiness budgets remain fixed, Chrome MCP transport failures still reset the session for the next call rather than recovering the failed action, and the maintainer discussion explicitly kept reconnect, timeout-budget, and regression-test work open.

## What This Changes

The issue asks for more stable browser existing-session automation by improving Chrome MCP timeouts, reconnect/recovery, diagnostics, configuration, and multi-step action handling.

## Best Possible Solution

Keep this as browser plugin stability work and split it into measured attach/readiness budget changes, explicit in-flow reconnect/session refresh semantics with regression coverage, and separate decisions on public timeout configuration or doctor diagnostics.

## Reproduction Assessment

No. Source inspection confirms current-main paths that match the remaining concern, but I did not establish a loaded profile="user" multi-step failure that reproduces gateway closed (1000) on current main.

## Solution Assessment

No for a constants-only change. The maintainable solution is measured timeout tuning plus explicit readiness/reconnect state semantics and focused regression or live-trace proof; the merged pageReady status signal helps diagnostics but is not the full fix.
