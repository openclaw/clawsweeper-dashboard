## Summary

Keep open. Current main added useful page-readiness/status work, but it still leaves the core stability work unresolved: fixed attach/readiness budgets remain, transport failures reset the cached Chrome MCP session and rethrow the current call, existing-session route limits remain, and the maintainer discussion explicitly kept this split work open.

## What This Changes

The issue asks to make browser existing-session automation with `profile="user"` stable across multi-step flows by improving timeouts, recovery, diagnostics, and possible timeout configuration.

## Best Possible Solution

Keep this as browser plugin stability work and split it into measured attach/readiness budgets, explicit in-flow reconnect or session-refresh semantics, observable diagnostics, focused regression coverage, and a separate decision on any public timeout configuration.

## Reproduction Assessment

No live high-confidence reproduction was established. Source inspection shows the current timeout constants, next-call-only recovery, and short navigation guard that match the remaining concerns, but the multi-step `gateway closed (1000)` failure was not run against a loaded user profile.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this. The maintainable fix is measured timeout tuning plus explicit reconnect/session-refresh behavior, diagnostics, and regression tests for the affected existing-session path.
