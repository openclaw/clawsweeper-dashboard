## Summary

Keep open. Current main and v2026.5.18 have useful partial fixes for existing-session status, page readiness, click, and navigation timeout handling, but the issue’s remaining reconnect/readiness-budget work is still explicitly called out in maintainer discussion and still visible in source.

## What This Changes

The issue asks to make `browser --browser-profile user` existing-session automation stable by relaxing tight attach/probe budgets, adding session recovery, improving diagnostics, and considering timeout overrides.

## Best Possible Solution

Split the work into observable Chrome MCP attach/page-readiness states, measured timeout budgets, explicit in-flow session refresh semantics, and focused regression/live proof; treat public timeout config and route-limit expansion as separate product decisions.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the reported multi-step `gateway closed (1000)` failure on current main. Source inspection does show the fixed attach/readiness budgets, next-call-only recovery, route limits, and short navigation guard that match the remaining concerns.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this issue. The maintainable path is measured timeout tuning plus explicit reconnect/session-refresh behavior, clearer readiness diagnostics, and regression/live coverage for the existing-session path.
