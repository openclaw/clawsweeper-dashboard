## Summary

Keep open. The live maintainer discussion explicitly split the work and kept it open, and current main plus latest release still show fixed attach/readiness budgets, next-call-only transport recovery, existing-session route limitations, and a short post-interaction navigation guard; pageReady/status work is useful but only partial.

## What This Changes

The issue requests making browser existing-session `profile=user` automation more stable by relaxing timeout budgets, adding reconnect/recovery, improving diagnostics, and optionally exposing timeout overrides.

## Best Possible Solution

Split the work into observable Chrome MCP attach/page readiness states, measured timeout budgets, in-flow session refresh semantics, and focused regression/live proof; treat public timeout config and route-limit expansion as separate product decisions.

## Reproduction Assessment

No, there is not a high-confidence live reproduction of the reported multi-step `gateway closed (1000)` failure on current main. Source inspection does show the fixed attach-readiness budget, next-call-only recovery, route limits, and short navigation guard that match the remaining concerns.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this issue. The maintainable path is measured timeout tuning plus explicit reconnect/session-refresh behavior, clearer readiness diagnostics, and regression/live coverage for the existing-session path.
