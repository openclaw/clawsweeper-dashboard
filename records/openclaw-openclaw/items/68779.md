## Summary

Keep open. Current main and the latest release still leave the central existing-session concerns in place: fixed attach/readiness budgets, next-call-only Chrome MCP recovery, limited existing-session action support, and a short interaction navigation guard; the FSB link is useful prior art for splitting attach/page/action states, but it does not make the OpenClaw browser-plugin work obsolete.

## What This Changes

The issue requests more stable Chrome MCP existing-session browser automation by improving timeout budgets, reconnect behavior, diagnostics, route support, and possible timeout configuration.

## Best Possible Solution

Split the work into observable Chrome MCP attach/page readiness states, measured timeout budgets, in-flow reconnect or session-refresh semantics, and focused regression/live proof; keep public timeout config and broader route-limit expansion as separate maintainer decisions.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection does show the fixed attach/readiness budgets, next-call recovery, route limits, and short navigation guard that match the remaining concerns.

## Solution Assessment

No, a constants-only timeout increase is not the best way to solve this issue. The maintainable path needs measured timeout tuning, explicit reconnect/session-refresh behavior, clearer readiness diagnostics, and regression or live proof for existing-session multi-step flows.
