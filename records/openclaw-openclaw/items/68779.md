## Summary

Keep open. Current main has partial Chrome MCP timeout and recovery improvements, but the remaining existing-session stability work is still real: attach/readiness budgets are still fixed, transport errors recover on the next call rather than preserving the current workflow, and several limitations are still documented as current behavior.

## What This Changes

The issue asks for existing-session browser automation to be more stable by relaxing timeout budgets, adding recovery/reconnection, improving diagnostics, and allowing timeout configuration.

## Best Possible Solution

Keep this as a maintainer-scoped browser plugin issue and split it into measured attach/readiness budget changes, in-flow session refresh/reconnect behavior with regression coverage, and a separate decision on public timeout configuration.

## Reproduction Assessment

No. Source inspection shows current-main paths that match the remaining instability concerns, but I did not establish a live loaded `profile="user"` multi-step failure with `gateway closed (1000)`.

## Solution Assessment

No. A constants-only increase is not the best fix; the maintainable path is measured timeout budget changes plus explicit reconnect/session-refresh behavior and focused regression tests.
