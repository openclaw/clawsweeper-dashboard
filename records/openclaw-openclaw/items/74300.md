## Summary

Keep this open. Current main still documents machine migration as install-copy-doctor without requiring version alignment, and the doctor command has no source/target migration-version preflight; the existing newer-config guard is adjacent but only protects selected startup/service mutation paths.

## What This Changes

The issue requests a migration version-alignment requirement and doctor-side source/target compatibility preflight to prevent silent state incompatibility during blue/green moves.

## Best Possible Solution

Add a read-only migration version preflight to doctor and update the public migration guide so machine moves require aligning OpenClaw versions before copying state; keep provider-auth fixes, rsync-secret validation, and automatic schema rewrites out of the first repair.

## Reproduction Assessment

Yes. The actionable gap is reproducible by source inspection: current docs and CLI wiring show no required version-alignment step and no doctor migration-version input, even though the existing future-config guard demonstrates the adjacent version-comparison behavior.

## Solution Assessment

Yes. A conservative doctor warning/error plus docs update is the narrowest maintainable path, preferably reusing existing version parsing and config metadata rather than adding startup-time migrations or owner-specific provider recovery.
