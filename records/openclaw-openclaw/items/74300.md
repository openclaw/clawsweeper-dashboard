## Summary

Keep this open. Current `main` still has no migration source/target version preflight and the public migration guide still describes install-copy-doctor without requiring version alignment, while the existing newer-config guard only covers selected local service/gateway mutation paths.

## What This Changes

Request a migration safety check so OpenClaw warns or blocks when state copied from one environment was produced by a different source version than the target binary, plus docs that require version alignment before transfer.

## Best Possible Solution

Add a conservative doctor-side migration version preflight, backed by the existing OpenClaw version parser and config/version metadata, and update the migration guide to require same-version state transfer before any intentional upgrade. Keep provider-auth recovery, rsync secret validation, and broad schema rewrites as separate follow-ups.

Yes. The actionable gap is reproducible by source inspection: current docs and CLI wiring provide no required version-alignment step or migration-version input, while the existing future-config guard demonstrates adjacent comparison behavior only for selected local mutation paths.

## Solution Assessment
