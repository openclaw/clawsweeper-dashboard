## Summary

Keep this open: current main has adjacent newer-config and backup-version metadata, but it still lacks a migration/cutover preflight that compares source state version with the running target binary, and the migration docs do not require version alignment before copying state.

## What This Changes

Request doctor and migration documentation to detect or prevent cross-environment OpenClaw version mismatches before migrated state is accepted.

## Best Possible Solution

Define a generic migration preflight contract: record the source OpenClaw version during supported backup/export or accept it explicitly, have doctor compare it with the running binary before service restart, and update the migration guide to require alignment or an intentional upgrade path.

## Reproduction Assessment

Yes. The high-confidence reproduction is a static current-main check: doctor has no migration version input, the existing guard only blocks older binaries reading newer-written config, and the documented machine-to-machine path copies state without a version-alignment preflight.

## Solution Assessment

Unclear as an implementation because no patch is proposed and the exact source-version contract needs maintainer choice. The narrowest maintainable direction is to reuse the existing version comparison helpers and doctor warning style instead of adding per-state ad hoc migrations.
