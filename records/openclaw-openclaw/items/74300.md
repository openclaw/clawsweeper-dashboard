## Summary

Keep open: current `main` and `v2026.6.6` still do not implement a cross-environment source-version versus target-binary migration preflight. Existing guards cover newer-written local config and service mutation safety, but the machine-move guide still tells users to install, copy state, and run doctor without required version alignment.

## What This Changes

The issue requests migration documentation and doctor/preflight checks that catch OpenClaw source/target version mismatches before imported state is accepted.

## Best Possible Solution

Define one supported machine-move version contract, then add a narrow doctor/docs path that uses existing version parsing and existing metadata where possible, without adding ad hoc state sidecar files unless maintainers explicitly choose that storage shape.

## Reproduction Assessment

Yes for the diagnostic gap from source inspection, but not from a live BLUE/GREEN VPS replay. Current doctor preflight has no source/target version input and the existing future-config guard only catches older binaries reading newer-written config.

## Solution Assessment

Yes, keeping this open is the best current outcome. The narrow implementation path is plausible, but maintainers should first decide the source-version carrier and whether mismatches warn or fail before a repair PR is queued.
