Keep this open. Current main still lacks required version-alignment guidance and a doctor-side source/target migration preflight; the existing newer-config guard is adjacent but only blocks selected service/gateway mutation paths.

Request to require version alignment and add a doctor migration preflight for source/target OpenClaw version mismatches during machine moves.

Add a read-only migration version preflight to doctor that compares imported state/source version with the running binary using the existing version-comparison helpers, then update the migration and doctor docs to require aligned versions before copying state.

## Repair Work Prompt

Fix the machine-migration version mismatch preflight tracked by https://github.com/openclaw/openclaw/issues/74300. Add a conservative, read-only `openclaw doctor` preflight that compares imported/source state version with the running OpenClaw binary, preferably reusing `meta.lastTouchedVersion`, `src/config/version.ts`, and `src/config/future-version-guard.ts` semantics unless a small explicit source-version stamp is necessary. The preflight should warn or fail when copied state was last touched by a newer source version than the target binary, and its guidance should tell operators to align versions before copying state or upgrade intentionally after the move. Update `docs/install/migrating.md`, `docs/cli/doctor.md`, and `docs/gateway/doctor.md` so machine moves require version alignment before state transfer. Add focused tests for the helper/preflight and CLI option plumbing if a flag is introduced. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter/commenter. Do not add startup-time core migrations, provider-specific auth recovery, token/env repair, plugin-specific defaults, dependency changes, or broad migration rewrites in this PR.
