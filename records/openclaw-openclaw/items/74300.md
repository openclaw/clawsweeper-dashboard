## Summary

Keep this open. Current main has local newer-config downgrade protection and adjacent backup/version metadata, but it still does not provide a migration/cutover preflight that compares imported source state or backup runtime version against the target binary before operators accept transferred state.

## What This Changes

Request a migration/DR preflight and documentation that warn or fail when source and target OpenClaw versions differ before state transfer or import.

## Best Possible Solution

Add an explicit doctor migration preflight that reuses existing version parsing and state/backup metadata, documents source/target version alignment as required, and leaves normal startup/update policy unchanged.

## Reproduction Assessment

Yes for the repository gap: current-main source inspection shows doctor reads local config snapshots and state migrations but has no source/target migration version comparison. The full production cascade still needs the original incident state to reproduce end to end.

## Solution Assessment

Yes, the narrowest maintainable repair is a doctor/docs migration preflight using existing version helpers, config stamps, or backup manifest runtime metadata. A broad startup block or new general downgrade policy would be riskier than the reported cutover boundary needs.
