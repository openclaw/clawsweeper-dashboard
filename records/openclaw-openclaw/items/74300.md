## Summary

Keep this open. Current main still lacks the requested migration version-alignment guidance and has no doctor-side source/target version preflight; the existing newer-config guard is real but only protects selected service/process mutations after config metadata is already present, so it does not cover the reported cross-machine state import boundary.

## What This Changes

The issue asks OpenClaw to require or preflight source/target version alignment when migrating state between machines, especially for BLUE/GREEN cutovers.

## Best Possible Solution

Add a read-only migration-version preflight to doctor that compares copied state or an explicit source version with the running target binary, emits a clear mismatch warning/block according to the existing version semantics, and update the public migration guide to require matching OpenClaw versions before copying state; keep provider-specific auth and .env incident fixes in their own issues.

## Reproduction Assessment

Yes. The actionable gap is reproducible by inspecting current main: the migration guide has no required version-alignment step, doctor exposes no migration version flag, and the only version guard is wired to service/startup mutations rather than migration import preflight.

## Solution Assessment

Yes for a narrow repair. A doctor warning/preflight plus docs update is the maintainable first fix; automatic schema rewrites and provider-specific credential recovery would be broader follow-up work.
