## Summary

Keep this open. Current main still documents the machine-migration path as install-copy-doctor without requiring source/target version alignment, and the doctor CLI has no source-version, target-version, or migration preflight option wired. The existing newer-config guard is adjacent but narrower: it protects service/process mutations from an older binary after config metadata is already present, rather than preflighting a cross-environment state import.

## What This Changes

Request to make OpenClaw machine migrations version-aware by documenting same-version alignment and adding doctor/DR preflight checks for source/target version mismatch.

## Best Possible Solution

Add a read-only migration-version preflight to doctor and update the public migration guide so machine moves require aligning OpenClaw versions before copying state; keep automatic schema rewrites and provider-specific incident fixes out of this first repair.

## Reproduction Assessment

Yes for the actionable gap: current docs and CLI wiring can be inspected directly and show no migration version-alignment step or doctor option. The production auth/plugin failures are incident-specific, but the missing preflight is reproducible from source and docs.

## Solution Assessment

Yes, a conservative doctor warning/error plus docs update is the narrowest maintainable path. It should reuse existing version parsing and state metadata rather than adding broad startup-time migrations.
