## Summary

Keep this open. Current main has adjacent future-config downgrade protection and backup runtime metadata, but it still lacks a migration/cutover preflight that compares transferred source or backup version metadata against the running target binary before operators accept imported state.

## What This Changes

The issue requests migration version-alignment checks and documentation so source and target OpenClaw environments do not silently accept incompatible transferred state.

## Best Possible Solution

Add a scoped migration/backup/doctor preflight that surfaces transferred source or backup runtime version versus the running target binary, update the migration guide to require deliberate version alignment, and leave ordinary startup/update behavior unchanged unless maintainers explicitly choose a stricter policy.

## Reproduction Assessment

Yes for the repository gap: source inspection on current main shows doctor and backup verification have local version metadata but no source/target migration compatibility check. No for the complete production cascade, which still needs the original incident state to replay end to end.

## Solution Assessment

Unclear until maintainers choose the strictness policy. The narrowest maintainable direction appears to be doctor/docs migration preflight using existing version helpers and backup manifest metadata, rather than a broad startup blocker.
