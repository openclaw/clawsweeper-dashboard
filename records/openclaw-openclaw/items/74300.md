## Summary

Keep open: current main and the latest release still have local newer-config guards, but not the requested cross-environment source-version versus target-binary migration preflight before copied state is accepted.

## What This Changes

The issue asks for OpenClaw migration/doctor checks and docs that prevent silent state incompatibility when source and target environments run different OpenClaw versions.

## Best Possible Solution

Define one supported machine-move version contract, then add a narrow doctor/docs preflight that compares imported source metadata against the running binary and distinguishes blocked downgrades from intentional upgrades.

## Reproduction Assessment

Yes, by source inspection: current doctor preflight, migration docs, and release-tag source do not enforce a cross-environment source/target version check before copied state is accepted. I did not run a live BLUE/GREEN VPS migration replay.

## Solution Assessment

Unclear that the proposed doctor flag is the best permanent shape. The safer solution is a maintainer-chosen version metadata carrier plus warn/fail policy, then a narrow doctor/docs implementation.
