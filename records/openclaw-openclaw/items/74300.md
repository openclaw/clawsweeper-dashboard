## Summary

Keep this open. Current main still has the actionable gap described in the comments: machine migration docs tell users to install, copy state, then run doctor without requiring source/target version alignment, and doctor has no source-version, target-version, or migration preflight option wired. The existing future-config guard is useful but narrower because it blocks selected destructive actions when the running binary is older than `meta.lastTouchedVersion`; it does not preflight a cross-machine state import or warn on ordinary version mismatch before copied state is used.

## What This Changes

The issue requests migration documentation and doctor preflight behavior so copied OpenClaw state is checked against the source and target runtime versions before use.

## Best Possible Solution

Add a read-only migration-version preflight to doctor and update the public migration guide to require aligning OpenClaw versions before copying state; keep automatic schema rewrites and the related auth/.env incident fixes in their own issues.

## Reproduction Assessment

Yes for the actionable gap: current source and docs can be inspected directly and show no migration version-alignment step or doctor option. The full VPS incident is environment-specific, but the missing preflight is reproducible from current main.

## Solution Assessment

Yes. A conservative doctor warning/preflight plus documentation update is the narrowest maintainable solution; it should reuse existing version parsing and config metadata rather than adding startup-time migrations or provider-specific recovery behavior.

## Repair Work Prompt

Fix this issue by adding a conservative migration version preflight and updating the migration docs. Current main documents machine migration as install-copy-doctor and has no doctor source/target version input. Add a read-only doctor preflight that compares a supplied or stamped source OpenClaw version with the running target binary version using existing version parsing helpers. Prefer an explicit CLI option such as `openclaw doctor --migration-source-version <version>` because the target is the current runtime; warn on any source/target mismatch with clear guidance to align versions before state transfer, and preserve the stronger existing downgrade/blocking semantics for newer-source/older-target cases where appropriate. Wire the option through `registerMaintenanceCommands`, `DoctorOptions`, and the doctor flow with focused unit tests. Update `docs/install/migrating.md` to make same-version migration required before copying state, update `docs/cli/doctor.md`, and add a user-facing `CHANGELOG.md` fix entry thanking @lolaopenclaw. Do not change provider-specific Anthropic auth behavior, Telegram/Discord `.env` handling, automatic schema rewrites, or startup/load-time migration policy in this PR.
