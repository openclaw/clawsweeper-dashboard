## Summary

Keep open: current main and v2026.6.8 still have local newer-config and CLI/gateway skew guards, but not a cross-environment source-version versus target-binary migration preflight before imported state is accepted.

## What This Changes

The issue requests migration/DR version-alignment enforcement so state copied between different OpenClaw versions does not fail silently across cron, auth profiles, sessions, or plugin config.

## Best Possible Solution

Define one supported machine-move version contract, then add a narrow doctor/docs path that compares imported source metadata against the running binary and distinguishes blocked downgrades from intentional upgrades.

## Reproduction Assessment

Yes, by source inspection: current doctor preflight, migration docs, and backup verification still do not enforce a cross-environment source/target version check. I did not run a live BLUE/GREEN VPS migration replay.

## Solution Assessment

Unclear that the suggested doctor flag is the best permanent shape. The safer path is for maintainers to choose the source-version carrier and warn/fail policy, then implement the narrow preflight and docs update.
