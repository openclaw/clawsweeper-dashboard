## Summary

Keep this issue open. Current main still has adjacent future-config downgrade guards and backup runtime-version metadata, but no doctor-side source/target migration preflight and no required version-alignment step in the machine migration docs.

## What This Changes

Request a migration preflight and documentation that require or verify source and target OpenClaw version alignment before transferring state between environments.

## Best Possible Solution

Add a read-only migration compatibility check to doctor that compares imported/source state version metadata with the running binary using the existing version-comparison policy, then update migration and doctor docs to require source/target version alignment before copying state. Keep provider-specific auth repair and broad automatic schema rewrites as separate owner-specific follow-ups.

## Reproduction Assessment

Yes. The actionable gap is reproducible by source inspection on current main: migration docs lack the required alignment step, doctor has no source/target version option or typed input, and focused searches found no migration preflight call path.

## Solution Assessment

Yes. A conservative doctor warning or blocking preflight plus docs guidance is the narrowest maintainable solution; it builds on existing version-guard behavior without turning this into a broad schema migration project.
