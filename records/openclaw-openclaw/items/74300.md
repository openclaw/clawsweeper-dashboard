## Summary

Keep this open. Current main still documents machine migration as install-copy-doctor without requiring source/target version alignment, and `openclaw doctor` has no migration source/target version option or preflight path. The existing future-config guard is relevant but narrower: it blocks selected gateway/service actions when a copied config was last touched by a newer binary, rather than preflighting a cross-environment state import. The issue discussion usefully links the same cutover cluster (#74297, #74298) while keeping those provider/env failures separate from this migration-version gap.

## What This Changes

Require migration docs and doctor preflight coverage for source/target OpenClaw version mismatches before copying state between machines.

## Best Possible Solution

Add a read-only doctor migration-version preflight and update the public migration guide so machine moves require version alignment before copying state; keep provider auth repair, `.env` verification, and any automatic schema rewrites as separate follow-ups unless their own reports prove a shared implementation need.

## Reproduction Assessment

Yes for the actionable gap: current docs and CLI wiring directly show no required version-alignment step and no doctor source/target version preflight. The full production failure cascade is incident-specific, but it is not needed to reproduce the missing guard.

## Solution Assessment

Yes, the requested docs plus a conservative doctor warning/error using existing version parsing and future-config semantics is the narrowest maintainable fix. A broad startup-time migration policy would be riskier and should stay out of this first repair.
