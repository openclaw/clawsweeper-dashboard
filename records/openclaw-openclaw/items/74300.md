## Summary

Keep open: current main and v2026.6.8 still cover local newer-config and CLI/Gateway skew warnings, but not the requested cross-environment source-version versus target-binary preflight before copied state is accepted.

## What This Changes

Issue requests migration preflight and documentation so copied state from one OpenClaw version is checked against the target binary before a blue/green cutover silently accepts incompatible state.

## Best Possible Solution

Define one supported machine-move version contract, then add a narrow doctor/docs preflight that compares imported source metadata against the running binary and distinguishes blocked downgrades from intentional upgrades.

## Reproduction Assessment

Yes, by source inspection: current doctor preflight, migration docs, and v2026.6.8 release source do not enforce a cross-environment source/target version check before copied state is accepted. I did not run a live BLUE/GREEN VPS migration replay.

## Solution Assessment

Unclear that the proposed doctor flag is the best permanent shape. The safer path is a maintainer-chosen version metadata carrier plus warn/fail policy, then a narrow doctor/docs implementation.
