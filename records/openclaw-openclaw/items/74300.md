- `impact:session-state`: The issue is about copied agent/session state becoming subtly incompatible across source and target OpenClaw versions.

- `impact:auth-provider`: The report and linked companion issue involve auth-profile shape drift and provider credential behavior during migration.

## Summary

Keep open: current `main` still lacks a cross-environment migration preflight that compares imported state/source version with the target OpenClaw binary before accepting or migrating state. Adjacent fixes now cover CLI-vs-Gateway runtime skew and missing migrated channel env tokens, but they do not close the central silent source/target state-compatibility gap.

## What This Changes

The issue requests migration-time version alignment guidance and a doctor/preflight check so copied state from one OpenClaw version is not silently used by a different target version.

## Best Possible Solution

Define one canonical migration preflight that compares imported state/source version metadata with the running target binary before state migrations, documents same-version transfer as the default path, and only proceeds with intentional upgrades through doctor-owned migrations.

## Reproduction Assessment

No live BLUE/GREEN migration was reproduced in this read-only review. Source inspection gives a high-confidence reproduction of the missing guard: doctor preflight migrates imported state without any source-version versus target-binary comparison, and the public migration guide does not require version alignment.

## Solution Assessment

Unclear until maintainers choose the version source and fail/warn policy. A narrow doctor-owned preflight plus migration-guide update is the maintainable direction; adding an operator flag should be secondary unless existing metadata cannot cover real migrations.
