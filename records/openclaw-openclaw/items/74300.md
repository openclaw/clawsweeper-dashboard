## Summary

Keep open: current main and v2026.6.6 still lack a cross-environment migration preflight that compares imported source state version against the running target binary. Existing guards cover older binaries mutating newer-written local config, and recent state/cron migrations cover adjacent surfaces, but the documented machine-move path still permits install-copy-doctor without required version alignment.

## What This Changes

The issue requests a doctor/docs migration preflight that detects OpenClaw source/target version skew before imported state is accepted.

## Best Possible Solution

Define one supported machine-move version contract, then add a narrow doctor/docs path that uses existing version parsing and existing metadata where possible.

## Reproduction Assessment

Yes for the diagnostic gap by source inspection: current doctor preflight has no source/target version input, and the future-config guard only catches older binaries reading newer-written config. I did not run a live BLUE/GREEN VPS replay.

## Solution Assessment

Unclear that the suggested doctor flag is the best permanent shape. The safer direction is for maintainers to choose the source-version carrier and warn/fail policy, then implement the narrow preflight and docs update.
