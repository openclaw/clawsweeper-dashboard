## Summary

Keep this open. Current main has a newer-config guard and a backup manifest runtimeVersion, but there is still no explicit migration/cutover preflight that compares imported source state against the running target binary before accepting state.

## What This Changes

The issue requests a migration/DR preflight and docs that require OpenClaw version alignment before transferring state between environments.

## Best Possible Solution

Add an explicit doctor migration preflight for operator-driven state transfers, backed by docs, while leaving normal startup and supported upgrade migrations unchanged.

## Reproduction Assessment

Yes for the repository gap: current main can be inspected to show doctor lacks any source/target migration version comparison. The exact production cascade is only partially reproducible from the issue because the incident state files and environment are not included.

## Solution Assessment

Unclear until maintainers choose the contract. A narrow opt-in doctor preflight or flag that compares an imported source stamp/backup manifest runtimeVersion to the running binary is safer than a broad startup block.
