## Summary

Keep this open. Current main has a newer-config downgrade guard and backup manifests record a runtime version, but there is still no explicit migration/cutover preflight comparing imported source state or source runtime version against the target binary before the operator accepts migrated state.

## What This Changes

Request a migration safeguard and documentation for OpenClaw source/target version mismatches during BLUE/GREEN state transfer.

## Best Possible Solution

Add a doctor-side migration preflight that can compare a source state stamp or backup manifest runtimeVersion against the running target binary, document version alignment in the migration guide, and leave normal startup/update migrations unchanged.

## Reproduction Assessment

Yes for the repository gap: current main can be inspected to show no source/target migration version comparison in doctor, backup verification, or migration docs. The exact production cascade is only partially reproducible from the report because the incident state files and environment are not included.

## Solution Assessment

Unclear until maintainers choose the exact compatibility contract. The safest direction is a narrow operator-invoked doctor/migration preflight and docs, not a broad startup block for every version difference.
