## Summary

Keep open. Current main and the latest shipped release still allow direct `.bak` restore after only parsing and checking `gatewayMode`, while the linked closing PR remains open and conflicting rather than merged.

## What This Changes

The issue requests that config recovery refuse polluted `.bak` files before restore, with follow-up hardening around backup creation, rotation depth, and config validation.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/68462 with a current-main restore guard before `copyFile`/`copyFileSync`, then track backup-creation scanning, deeper rotation, and a config validation CLI separately if maintainers want that broader hardening.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a valid-looking `.bak` with `gateway.mode` and a redacted sensitive value, make the active config suspicious, and the async or sync recovery path reaches the restore copy without inspecting `backupParsed`.

## Solution Assessment

Yes for the immediate restore guard. Refusing polluted backups before direct restore is the narrow maintainable fix; backup creation scanning, rotation policy, and a validation CLI are separate follow-up scope.
