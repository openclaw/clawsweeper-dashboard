## Summary

Keep open. Current main still allows direct `.bak` restore after parsing and `gatewayMode` checks, while the linked closing PR remains open and conflicting rather than merged.

## What This Changes

The issue asks OpenClaw to refuse polluted config backups containing redacted secret placeholders before recovery and to consider related backup/validation hardening.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/68462 with a current-main guard before async and sync backup copies, then track broader backup hardening separately if desired.

## Reproduction Assessment

Yes by source inspection. Seed a valid-looking `${configPath}.bak` with `gateway.mode` and a redacted sensitive value, make the live config suspicious, and the async or sync recovery path reaches `copyFile`/`copyFileSync` without inspecting `backupParsed`.

## Solution Assessment

Yes for the immediate restore guard. Refusing polluted backups before copying is the narrow maintainable fix; backup creation scanning, rotation policy, and backup-scanning CLI behavior should be separate follow-up scope.
