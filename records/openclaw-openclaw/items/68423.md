## Summary

Keep open. Current main still restores from `.bak` after only a gateway-mode fingerprint check, and the open closing PR has not merged yet.

## What This Changes

Validate config backups for redacted secret placeholders before `.bak` restore and consider preventing polluted config writes from entering the backup ring.

## Best Possible Solution

Review and land #68462, or replace it with an equivalent main-ready patch that validates parsed backup contents before both async and sync `.bak` restore copies; handle backup creation and CLI scanning as explicit follow-ups if maintainers want that broader hardening.

## Reproduction Assessment

Yes. Seed a valid-looking `.bak` with `gateway.mode` and a secret field such as `gateway.auth.token: "***"`, make the active config suspicious, then call either recovery function; current main will copy the backup because it does not inspect the parsed backup for redacted placeholders.

## Solution Assessment

Yes for the immediate fix. Validating parsed `.bak` contents in `src/config/io.observe-recovery.ts` before copy is the narrowest maintainable repair; backup-ring and CLI validation changes are useful but separate scope.
