## Summary

Keep open. Current main still restores a parsed `.bak` over a suspicious live config without checking the parsed backup for redacted secret placeholders, and the open closing PR is still unmerged and conflicting.

## What This Changes

The issue requests validating `.bak` config backups for redacted secret placeholders before `maybeRecoverSuspiciousConfigRead` restores them, with broader backup and CLI hardening proposed as follow-up scope.

## Best Possible Solution

Land a current-main guard before both `copyFile` and `copyFileSync` that refuses parsed backups containing redacted secret placeholders, then handle backup creation scanning and validation CLI behavior as separate follow-up work.

## Reproduction Assessment

Yes by source inspection. Seed a suspicious live config and a valid-looking `${configPath}.bak` with `gateway.mode` plus a redacted sensitive value; current main reaches the async or sync backup copy path without scanning `backupParsed`.

## Solution Assessment

Yes for the immediate bug boundary. Refusing polluted backups before direct restore is the narrow maintainable fix, while backup rotation policy and `openclaw config validate` should be separate follow-up decisions.
