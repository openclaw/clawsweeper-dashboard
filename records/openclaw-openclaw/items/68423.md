## Summary

Keep open. Current main still restores `.bak` through both async and sync suspicious-config recovery after only parsing and gateway-mode checks, and the closing PR #68462 is open/unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Validate `.bak` backups before config auto-restore so redacted secret placeholders are not restored or fixated.

## Best Possible Solution

Review and land #68462, or replace it with an equivalent patch that rejects polluted parsed backups before both async and sync `.bak` restore copies; handle backup-creation scanning and CLI validation as explicit follow-ups if maintainers want the broader hardening.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a valid-looking `.bak` with `gateway.mode` and `gateway.auth.token: "***"`, make the active config an update-channel-only root, then the async or sync recovery path reaches `copyFile`/`copyFileSync` without inspecting the parsed backup for pollution.

## Solution Assessment

Yes for the immediate fix. The narrow maintainable solution is to reuse or factor the existing redacted-placeholder collector before copying `.bak`; backup rotation policy and CLI scanning should stay separate follow-ups.
