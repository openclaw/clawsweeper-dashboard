## Summary

Keep open. Codex review finds current main still restores direct `.bak` backups without checking for redacted secret placeholders, and the linked fix PR at https://github.com/openclaw/openclaw/pull/68462 is still open rather than merged.

## What This Changes

The issue requests validation that config backups are not polluted with redacted secret placeholders before restore, plus broader backup rotation and CLI scanning hardening.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/68462 or an equivalent current-main patch for direct `.bak` restore refusal, then split backup creation, rotation, and CLI scanning hardening into explicit follow-up decisions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a valid-looking `.bak` with `gateway.mode` and a redacted secret placeholder, make the active config suspicious, and the async or sync recovery path reaches the backup copy without a pollution check.

## Solution Assessment

Yes for the immediate fix. Validating parsed backup contents before both direct restore copies is the narrowest maintainable repair; the rotation and CLI pieces are broader hardening that should not block landing the focused fix.
