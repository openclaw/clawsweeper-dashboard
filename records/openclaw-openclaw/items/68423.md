## Summary

Keep this open. Current main still has the reported gap: the last-known-good recovery path rejects redacted secret placeholders, but the legacy `.bak` auto-restore path named in this issue still reads, parses, and copies `openclaw.json.bak` without checking for redacted placeholder pollution. An open, unmerged PR (#68462) explicitly targets this issue, so the issue should remain open while that PR is reviewed or replaced.

## What This Changes

The issue asks OpenClaw to reject redacted placeholder pollution in `.bak` config backups before restoring them and to avoid spreading polluted config data through backup rotation.

## Best Possible Solution

Keep this issue open and review PR #68462, or replace it with a maintainer-owned equivalent. The first fix should validate parsed `.bak` contents before both async and sync restore copies; backup-creation validation, backup retention changes, and backup-scanning CLI behavior can be handled as explicit follow-ups if maintainers want that broader scope.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a valid-looking `.bak` with `gateway.mode` plus a secret field such as `gateway.auth.token: "***"`, make the active config suspicious, then call either recovery function; current main will copy the `.bak` because it checks `gatewayMode` but not redacted placeholders.

## Solution Assessment

Yes for the immediate path: refusing polluted `.bak` restore in `src/config/io.observe-recovery.ts` is the narrowest maintainable first fix. The backup-rotation and CLI validation ideas are related hardening, but they do not need to block the restore guard if reviewed as follow-ups.
