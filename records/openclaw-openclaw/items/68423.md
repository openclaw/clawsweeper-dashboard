## Label Justifications

- `P1`: The issue describes a credential-pollution chain that can leave provider auth broken for real users until config and backups are manually repaired.
- `impact:data-loss`: Redacted placeholders can replace real credential values in live config and backups, effectively losing the usable secrets.
- `impact:security`: The affected behavior is credential-safety handling around redacted secret placeholders in config recovery.
- `impact:auth-provider`: The reported failure mode corrupts API keys or tokens and produces provider authentication failures.

## Summary

Keep open. Current main still copies a parsed `.bak` over the active config in both suspicious-config recovery paths without checking the parsed backup for redacted credential placeholders, and the linked closing PR at https://github.com/openclaw/openclaw/pull/68462 remains open and unmerged.

## What This Changes

The issue asks OpenClaw to validate config backup files for redacted secret placeholders before restore, with broader follow-ups for backup creation, retention, and validation tooling.

## Best Possible Solution

Land or replace the linked fix with a guard before both direct `.bak` restore copy paths, reusing the existing sensitive-placeholder semantics; handle backup creation scanning and CLI validation as separate follow-ups if maintainers want that broader scope.

## Reproduction Assessment

Yes, by source inspection. Seed a suspicious live config and a parseable `${configPath}.bak` with `gateway.mode` plus a redacted sensitive value such as `gateway.auth.token: "***"`; both direct restore paths can copy the backup without scanning `backupParsed`.

## Solution Assessment

Yes for the immediate bug boundary: refusing polluted backups before direct restore is the narrow maintainable fix. The backup-creation scan, retention changes, and validation CLI should be separate maintainer decisions if pursued.
