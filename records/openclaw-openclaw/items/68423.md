## Summary

Keep open. Current main and the latest release still restore direct `.bak` backups after parsing and checking `gatewayMode`, and the linked closing PR is open/conflicting rather than merged.

## What This Changes

Request to stop config recovery from restoring or propagating `.bak` files that contain redacted secret placeholders.

## Best Possible Solution

Land, rebase, or replace https://github.com/openclaw/openclaw/pull/68462 with a current-main guard before both async and sync direct `.bak` copy paths, then handle backup-creation scanning and CLI diagnostics as follow-up scope if maintainers want them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a valid-looking `.bak` with `gateway.mode` plus a redacted sensitive value, make the active config suspicious, and the async or sync recovery path reaches `copyFile`/`copyFileSync` without inspecting `backupParsed` for placeholders.

## Solution Assessment

Yes for the immediate restore guard. Refusing polluted `.bak` restore before the copy is the narrowest maintainable fix; backup creation scanning, rotation policy, and config validation diagnostics should be scoped separately rather than blocking the core bug fix.
