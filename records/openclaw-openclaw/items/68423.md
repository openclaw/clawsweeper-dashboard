## Summary

Keep open. Current main still restores direct `.bak` backups after parsing and a `gatewayMode` check without scanning the backup for redacted secret placeholders, and the linked closing PR remains open and unmerged.

## What This Changes

The issue asks OpenClaw to refuse restoring polluted config backups that contain redacted secret placeholders and to avoid preserving those polluted backups.

## Best Possible Solution

Land, rebase, or replace https://github.com/openclaw/openclaw/pull/68462 with a current-main guard that checks parsed `.bak` contents before async and sync restore copies; handle backup-creation scanning, rotation policy, and CLI scanning as follow-ups if maintainers want that broader hardening.

## Reproduction Assessment

Yes, at source level. Seed a valid-looking `.bak` with `gateway.mode` and a redacted secret value, make the active config suspicious, and current async or sync recovery reaches `copyFile`/`copyFileSync` without scanning `backupParsed`.

## Solution Assessment

Yes for the immediate restore guard. Refusing polluted `.bak` restore before the copy is the narrowest maintainable fix; backup creation scanning, rotation policy, and CLI validation should not block that core repair.
