## Label Justifications

- `P1`: The issue describes a real credential-pollution chain that can leave provider auth broken until a human repairs config and backups.
- `impact:data-loss`: Redacted placeholders can overwrite or preserve lost real credential values in live config and backups.
- `impact:security`: The affected behavior is credential-safety handling around redacted secret placeholders in config recovery.
- `impact:auth-provider`: The reported failure mode corrupts API keys or tokens and produces provider 401 failures.

## Summary

Keep open. Current main still copies a parsed `.bak` over the active config in both async and sync suspicious-config recovery without checking the parsed backup for redacted credential placeholders, and the linked closing PR remains open.

## What This Changes

The issue requests that suspicious config recovery reject polluted `.bak` files containing redacted secret placeholders before restoring them over `openclaw.json`.

## Best Possible Solution

Land or replace the linked fix with a current-main guard before both direct backup copy paths, reusing the existing sensitive-path placeholder semantics; track backup creation scanning and validation CLI work separately if maintainers want that broader scope.

## Reproduction Assessment

Yes, by source inspection. Seed a suspicious live config and a valid-looking `${configPath}.bak` with `gateway.mode` plus a redacted sensitive value; current main reaches the async or sync backup copy path without scanning `backupParsed`.

## Solution Assessment

Yes for the immediate bug boundary: refusing polluted backups before direct restore is the narrow maintainable fix. Backup creation scanning, retention changes, and `openclaw config validate` should be separate maintainer decisions if pursued.
