## Summary

Keep open. Current main still restores direct `.bak` backups after only parsing and checking `gatewayMode`, and the linked closing PR is still open and conflicting rather than merged.

## What This Changes

The issue asks OpenClaw to refuse restoring polluted config backups that contain redacted credential placeholders and to avoid spreading those placeholders through backup handling.

## Best Possible Solution

Land, rebase, or replace https://github.com/openclaw/openclaw/pull/68462 with a current-main guard that checks parsed `.bak` contents before async and sync restore copies, then track backup-creation and CLI scanning hardening separately if maintainers want them.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed a valid-looking `.bak` with `gateway.mode` and a redacted sensitive value, make the active config suspicious, and the async or sync recovery path reaches `copyFile`/`copyFileSync` without inspecting the parsed backup for placeholders.

## Solution Assessment

Yes for the immediate restore guard. Refusing polluted `.bak` restore before the copy is the narrowest maintainable fix; backup creation scanning, rotation policy, and a config validation CLI should be scoped as follow-ups rather than blocking the core bug fix.
