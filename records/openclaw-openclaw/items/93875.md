## Summary

## What This Changes

The issue reports that Discord `message_tool_only` source-channel turns can append a stale tool-failed warning after a successful `message(action=send)` reply.

## Best Possible Solution

Review and land or reject the open linked PR; if it lands, close this issue as implemented and keep the duplicate fix attempts retired.

## Reproduction Assessment

Yes from source inspection, but not from a live run in this read-only review. Current `main` can set `deliveredSourceReplyViaMessageTool` while leaving `hasUserFacingAssistantReply` false, so a non-recoverable non-mutating `lastToolError` can append the stale warning.

## Solution Assessment

Yes. Counting the existing runner-owned delivery fact in the payload warning gate is the narrowest maintainable fix; Discord-specific warning logic or globally weakening tool-error warnings would duplicate ownership or risk suppressing unrelated failures.
