## Summary

Keep open: current main still has the same accounting gap, the latest release also contains it, and several open PRs are already candidate fixes. This issue should stay open until one focused fix lands.

## What This Changes

The issue asks OpenClaw to treat a successful Discord source reply sent through `message(action=send)` in `message_tool_only` mode as a user-facing assistant reply so earlier tool errors do not append stale warnings.

## Best Possible Solution

Land one focused PR that adds `deliveredSourceReplyViaMessageTool` to `hasUserFacingAssistantReply` and includes a regression test for delivered `message_tool_only` plus `lastToolError`; then close this issue as fixed.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a `message_tool_only` turn with `didDeliverSourceReplyViaMessageTool: true`, no source mirror payload, and a non-recoverable non-mutating `lastToolError` reaches `resolveToolErrorWarningPolicy` with `hasUserFacingReply: false`.

## Solution Assessment

Yes. Counting `deliveredSourceReplyViaMessageTool` as user-facing reply evidence is the narrow owner-boundary fix because the delivery hook already excludes failed, dry-run, explicit-route, and non-send message-tool calls.
