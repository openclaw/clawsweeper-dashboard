## Summary

Keep open: current main and v2026.6.9 still have the reported message-tool delivery accounting gap, and the open linked closing PR is the live fix candidate rather than a reason to close this issue before merge.

## What This Changes

Discord `message_tool_only` turns should count a successful `message(action=send)` source delivery as a user-facing reply so stale tool-error warnings are not appended.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94072; if it lands, close this issue as implemented and keep the duplicate closed PR attempts retired.

## Reproduction Assessment

Yes, at source level. Current main still lets `message_tool_only` set `deliveredSourceReplyViaMessageTool` while leaving `hasUserFacingAssistantReply` false, so a non-recoverable non-mutating `lastToolError` can append the stale warning.

## Solution Assessment

Yes. The linked PR's one-line gate alignment is the narrowest maintainable fix because it reuses the existing runner-owned delivery fact instead of adding Discord-specific warning logic or weakening tool-error warnings globally.
