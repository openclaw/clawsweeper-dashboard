## What This Changes

The issue reports that Discord `message_tool_only` turns can send the final reply successfully via `message(action=send)` and then append a stale tool-failed warning.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/94072 or an equivalent narrow payload-gate fix, then close this issue and retire duplicate fix PRs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main can set `deliveredSourceReplyViaMessageTool` while leaving `hasUserFacingAssistantReply` false, then `resolveToolErrorWarningPolicy` can append a warning for a non-recoverable non-mutating tool error.

## Solution Assessment

Is this the best way to solve the issue? Yes: adding the existing `deliveredSourceReplyViaMessageTool` evidence to the payload-builder reply gate fixes the accounting boundary without Discord-specific branching or a new config surface.
