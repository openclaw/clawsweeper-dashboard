## Summary

Keep open. The PR targets a real duplicate-reply bug, but the proposed wording conflicts with current main’s Discord guild/channel delivery policy: default visible replies are message-tool-only, so telling the agent to rely on final assistant delivery can make replies private instead of visible.

## What This Changes

The PR adds a changelog entry and rewrites the Codex embedded developer instruction plus bundled Discord skill so normal current-conversation replies use the final assistant message while explicit Discord operations use the `message` tool.

## Best Possible Solution

Keep the fix prompt/docs/changelog scoped, but make the guidance reflect the existing delivery mode: when final replies are private or message-tool-only, visible Discord output must use `message(action=send)`; when automatic final delivery is active, avoid a duplicate same-conversation `message.send`.

## Reproduction Assessment

Yes. Related issue #73278 provides concrete OpenClaw 2026.4.26 Discord `autoThread` reproduction steps, and current-main source/tests provide a high-confidence source-level reproduction for the PR regression: default guild/channel turns are `message_tool_only`, so final-only guidance can suppress visible output.

## Solution Assessment

No. A prompt-only repair is plausible, but this implementation is not the best form because it ignores `sourceReplyDeliveryMode`; the safer solution is visibility-policy-aware wording in both the Codex developer instruction and Discord skill.
