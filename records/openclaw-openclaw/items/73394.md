## Summary

Keep open: the bug is real, but this PR’s wording conflicts with current Discord channel delivery policy, where default guild/channel replies are tool-only and automatic final source delivery is suppressed.

## What This Changes

The PR rewrites Codex embedded developer instructions and the bundled Discord skill, plus adds a changelog entry, so normal autoThread/current-conversation replies use final assistant delivery while explicit Discord operations use the `message` tool.

## Best Possible Solution

Make the Codex and Discord guidance delivery-mode aware: use `message(action=send)` for visible output in `message_tool_only` turns, and warn against duplicate same-conversation sends only when automatic final delivery is active.

## Reproduction Assessment

Yes. #73278 provides concrete OpenClaw 2026.4.26 autoThread duplicate-reply steps, and current-main source/tests provide a high-confidence PR regression path through default Discord `message_tool_only` channel delivery.

## Solution Assessment

No. Prompt-only may be the right boundary, but this version is not the best fix because it contradicts `sourceReplyDeliveryMode`; the safer fix is conditional wording that preserves tool-only visible replies.
