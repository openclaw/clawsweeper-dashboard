## Summary

Keep open: this PR targets a real Discord autoThread duplicate-reply report, but it should not merge as-is because the new final-only guidance conflicts with current main's default guild/channel visible-reply policy. Current source, tests, and docs show Discord guild/channel replies are message-tool-only unless `messages.groupChat.visibleReplies` is set to `automatic`, so this wording can make default visible Discord replies disappear instead of merely preventing duplicates.

## What This Changes

The PR edits `CHANGELOG.md`, the Codex embedded developer instructions, and the bundled Discord skill so agents use final assistant messages for normal current-conversation replies and reserve the `message` tool for explicit out-of-band Discord actions.

## Best Possible Solution

Make the Codex prompt and Discord skill visibility-policy aware: visible output in tool-only group/channel turns should use `message(action=send)`, while automatic final delivery or autoThread final routing should avoid a duplicate same-conversation `message.send`. Keep the change scoped to prompt/skill/changelog unless source review shows a missing delivery-mode seam.

## Reproduction Assessment

Yes. #73278 gives concrete OpenClaw 2026.4.26 Discord autoThread reproduction steps, and current-main source/tests give a high-confidence static reproduction for the PR regression: default Discord guild/channel turns are `message_tool_only` while final source delivery is suppressed.

## Solution Assessment

No. A prompt-only fix is plausible, but this implementation overcorrects by assuming final assistant delivery is always visible; the safer solution must preserve current tool-only visible-output policy and only suppress duplicate message-tool sends when automatic final delivery is active.
