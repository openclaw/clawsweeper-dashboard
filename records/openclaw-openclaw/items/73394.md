## Summary

Keep open. The PR targets a real Discord autoThread duplicate-reply bug, but the proposed prompt wording is too broad for current Discord channel delivery because default guild/channel turns can require `message(action="send")` for visible output.

## What This Changes

The PR changes Codex embedded developer instructions, the bundled Discord skill text, and the changelog to route current-conversation replies through final assistant output while reserving `message` for out-of-band Discord actions.

## Best Possible Solution

Make the Codex and Discord guidance delivery-mode aware: use final assistant output when automatic source delivery is active, and preserve `message(action="send")` for visible Discord output in `message_tool_only` turns.

## Reproduction Assessment

Yes for review purposes. The linked bug report gives concrete Discord `autoThread: true` duplicate-reply steps, and source/tests on current main show the delivery-mode path that this patch would regress; I did not run a live Discord session in this read-only review.

## Solution Assessment

No. Prompt-only may be the right boundary, but this version is not the best fix because it makes final delivery unconditional instead of respecting `sourceReplyDeliveryMode`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve tool-only guidance for Discord channel replies:** `skills/discord/SKILL.md:12`
  - body: Current Discord guild/channel turns can resolve to `sourceReplyDeliveryMode: "message_tool_only"`, which suppresses automatic final delivery for visible channel output. This added skill text tells agents not to call `message` for the current conversation and to rely on final delivery, so default guild replies can become private or invisible instead of being sent to Discord. Please make the guidance conditional on the turn's delivery mode.
