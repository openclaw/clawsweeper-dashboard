## Summary

Keep open. The duplicate autoThread report is real, but this PR is not mergeable as written because it unconditionally tells Discord/Codex turns to rely on the final assistant message; current main defaults Discord guild/channel turns to message-tool-only delivery and suppresses automatic final delivery in that mode, so default visible channel replies can disappear.

## What This Changes

The PR adds a changelog entry and rewrites the Codex embedded developer instruction plus bundled Discord skill so normal current-conversation replies use the final assistant message while the `message` tool is reserved for explicit out-of-band Discord operations.

## Best Possible Solution

Rework the prompt and skill text so it is visibility-policy aware: when the current turn says final replies are private or message-tool-only, visible Discord output must use `message(action=send)`; when automatic final delivery is active or autoThread final delivery is routing the conversational reply, avoid a duplicate same-conversation `message.send`. Keep the fix prompt/docs/changelog scoped unless runtime evidence shows a missing delivery-mode seam.

## Reproduction Assessment

Yes. #73278 gives concrete OpenClaw 2026.4.26 Discord autoThread reproduction steps, and current-main source/tests give a high-confidence path for the PR regression: default guild/channel turns are `message_tool_only`, so final-only guidance can suppress visible output.

## Solution Assessment

No. A prompt-only repair is plausible, but this implementation is not the best form because it ignores `sourceReplyDeliveryMode`; the safer fix must make the Codex prompt and Discord skill conditional on the current visible-reply policy.
