## Summary

Keep open. The PR targets a real duplicate-reply failure from #73278, but the proposed prompt text is too broad for current main: Discord guild/channel turns default to tool-only visible replies, and automatic final delivery is suppressed in that mode. Telling the agent not to use the message tool for the current conversation can make default visible Discord replies private or invisible instead of fixing only the autoThread duplicate path.

## What This Changes

The PR adds a changelog entry and rewrites the Codex embedded developer instruction plus bundled Discord skill so normal current-conversation replies use final assistant delivery while explicit out-of-band Discord operations use the message tool.

## Best Possible Solution

Make the prompt text delivery-mode aware: in message_tool_only turns, visible Discord output must use message(action=send) and final answers stay private; in automatic turns, rely on the final assistant reply and warn against duplicate same-conversation message sends. Keep the fix scoped to prompt/docs/changelog unless a targeted test exposes a missing runtime seam.

## Reproduction Assessment

Yes. #73278 gives concrete OpenClaw 2026.4.26 autoThread duplicate-reply steps, and current-main source/tests give a high-confidence regression path for this PR: default Discord guild/channel turns resolve to message_tool_only, where final assistant delivery is suppressed.

## Solution Assessment

No. A prompt-only fix may be the right boundary, but this implementation is not the best fix because it makes final-assistant delivery sound unconditional and contradicts current sourceReplyDeliveryMode policy.
