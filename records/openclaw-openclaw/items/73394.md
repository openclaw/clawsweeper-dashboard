## Summary

Keep open. The PR targets a real Discord autoThread duplicate-reply report, but the proposed wording conflicts with current main's Discord channel visibility policy: default guild/channel turns are `message_tool_only`, so visible Discord output still needs `message(action=send)` unless automatic final replies are explicitly enabled.

## What This Changes

The PR adds a changelog entry and rewrites the Codex embedded developer instruction plus bundled Discord skill so normal current-conversation replies use the final assistant message while explicit Discord operations continue to use the `message` tool.

## Best Possible Solution

Make the Codex and Discord prompt text delivery-mode aware: for `message_tool_only` turns, visible Discord output uses `message(action=send)` to the current source; for automatic final delivery, avoid duplicate same-conversation `message.send` and rely on the final assistant response. Keep the change prompt/docs/changelog scoped unless tests show a missing runtime seam.

## Reproduction Assessment

Yes. #73278 provides concrete OpenClaw 2026.4.26 autoThread duplicate-reply steps, and current-main source/tests provide a high-confidence PR regression path: default Discord guild/channel turns resolve to `message_tool_only`, where final assistant delivery is suppressed.

## Solution Assessment

No. A prompt-only repair may be the narrowest boundary, but this implementation is not the best fix because it makes final-assistant delivery sound unconditional and contradicts the current `sourceReplyDeliveryMode` policy.
