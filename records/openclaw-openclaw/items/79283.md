## Summary

Keep open: current main already protects normal backgrounded exec sessions from run aborts, but the report still matches the documented process-denied path where exec runs synchronously and can be killed with the parent run. The remaining change is already represented by an open PR and needs maintainer judgment because it broadens process lifecycle behavior when the management tool is hidden.

## What This Changes

The issue reports long-running exec commands being terminated after a session heartbeat timeout and asks for survival or an opt-in detached lifecycle.

## Best Possible Solution

Resolve the remaining process-denied exec case through a narrowed PR that preserves finite timeout and management guarantees, or explicitly document why process-denied exec must remain synchronous.

## Reproduction Assessment

Partially: source inspection gives a clear current-main path with tools.deny: ["process"], a long exec with yieldMs, and a run abort killing the foreground process. The issue does not include config proving that was the reporter's exact webchat setup.

## Solution Assessment

Unclear: the existing backgrounded-exec guard is the right behavior for normal sessions, but changing process-denied exec lifecycle is not a mechanical bug fix. The safer path is to revise or replace the related PR with a finite-timeout guard and synchronized docs after maintainer policy review.
