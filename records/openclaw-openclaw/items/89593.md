## Summary

Keep open. Current main and v2026.6.6 added privacy-safe diagnostics for iMessage echo/from-me drops, but the inbound delivery policy still drops from-me group rows and the merged related work explicitly left this same-group inbound-drop behavior as remaining work.

## What This Changes

The issue asks the iMessage plugin to stop silently dropping follow-up inbound group messages after an agent reply and to diagnose or narrow the echo/from-me filter that caused the loss.

## Best Possible Solution

Use the current diagnostics to capture the dropped row reason, then adjust only the confirmed iMessage echo/from-me branch with focused regression coverage and live iMessage proof that the follow-up reaches the existing group session.

## Reproduction Assessment

No high-confidence exact reproduction of the reporter's two-minute group-chat loss is available yet. Source inspection does show current main still drops is_from_me group rows as from-me and logs that reason, so the next reproduction path is to collect the new diagnostic for the reported chat.

## Solution Assessment

No, the current merged work is not the complete fix: it is a useful diagnostics step, not a delivery-policy change. The best fix should wait for the specific current-main drop reason and then narrowly change that branch without weakening echo-loop protection.
