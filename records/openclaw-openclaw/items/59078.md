## Summary

Keep open: current main and v2026.6.8 still only satisfy `requires.env` from process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot. The requested env-or-config-file readiness behavior remains a product/security contract decision, not a stale or already-implemented issue.

## What This Changes

The issue asks OpenClaw to stop marking skills as needing setup when credentials are available through a skill-supported config-file fallback instead of environment variables alone.

## Best Possible Solution

Define a declarative skill credential-readiness contract and wire it through status, runtime eligibility, CLI/UI, doctor, docs, and regression tests without arbitrary status-time skill code execution.

## Reproduction Assessment

Yes, by source inspection: a skill with `requires.env` remains missing unless the required key is present in process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot. I did not run a live UI reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the credential-readiness contract. A declarative env-or-file credential source looks safer than executing arbitrary skill code during status checks.
