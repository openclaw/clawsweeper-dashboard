- `impact:security`: The issue concerns credential availability checks and how OpenClaw can safely reason about sensitive credential-backed setup state.

## Summary

Keep open: current main and the latest shipped release still treat `requires.env` as satisfied only by process env, OpenClaw skill env config, or the `primaryEnv` API-key slot. The related setup-hook issue and ClawHub plugin are useful context, but they do not resolve this narrower credential-readiness contract.

## What This Changes

The issue asks OpenClaw skill status/readiness to recognize credential sources beyond required environment variables, such as skill-owned config files or skill-reported setup status.

## Best Possible Solution

Define a safe declarative credential-readiness contract, then wire it through skill status, runtime eligibility, CLI/UI display, docs, and focused regression tests without executing arbitrary skill code during routine status checks.

## Reproduction Assessment

Yes, by source inspection. A skill declaring only `metadata.openclaw.requires.env` remains missing unless the value is in process env, OpenClaw skill env config, or the `primaryEnv` API-key slot; skill-owned credential files are never checked.

## Solution Assessment

Unclear as proposed. Skill self-reporting is likely too risky as a first fix; a declarative env-or-file requirement contract is narrower, but maintainers still need to settle the public metadata and credential-check trust model.
