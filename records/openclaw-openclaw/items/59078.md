## Summary

Keep open: current main and v2026.6.10 still treat `requires.env` as satisfied only by process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot, with no skill-owned credential-file readiness or setup-status contract. The related setup-hook issue overlaps, but it explicitly tracks install/update setup execution and does not replace this credential-readiness decision.

## What This Changes

The issue requests skill setup readiness detection that can account for credentials supplied by config files or skill-defined setup status instead of environment variables only.

## Best Possible Solution

Define a declarative skill credential-readiness contract and wire it consistently through status, runtime eligibility, CLI/UI, doctor, docs, and tests without arbitrary status-time skill code execution.

## Reproduction Assessment

Yes, by source inspection: a skill with `requires.env` remains missing unless the required key is present in process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot. I did not run a live UI reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the credential-readiness contract. A declarative env-or-file credential source looks safer than executing arbitrary skill code during status checks.
