## Summary

Keep open: current main and v2026.6.8 still satisfy skill `requires.env` only from process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot. Skill-owned credential files or skill-authored readiness remain a public contract and security decision.

## What This Changes

Request support for skill readiness checks that treat skill-owned credential config files, or a skill-defined setup-status contract, as satisfying credentials instead of only `requires.env`.

## Best Possible Solution

Define a declarative credential-readiness contract for skills, then wire it through status, runtime eligibility, CLI/UI, docs, and regression tests without executing arbitrary skill code during routine checks.

## Reproduction Assessment

Yes, by source inspection. A skill with `requires.env` remains missing unless the required key is present in process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot; skill-owned config files are not checked.

## Solution Assessment

Unclear until maintainers choose the credential-readiness contract. A declarative env-or-file credential source looks safer than executing arbitrary skill code during status checks.
