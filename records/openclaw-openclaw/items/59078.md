## Summary

Keep this issue open. Current main still treats skill readiness as a declared-requirements check, with env requirements satisfied only by process env or OpenClaw skill config; the linked setup-hook issue is adjacent and explicitly treats this credential/readiness conflict as separate rather than superseding it.

## What This Changes

The issue requests skill setup readiness detection that can recognize credentials supplied through skill-owned config files, or a safe way for skills to report setup status.

## Best Possible Solution

Define a safe declarative skill readiness contract for credential alternatives such as env-or-file, then wire it through status, runtime eligibility, CLI/UI rendering, docs, and regression tests without executing untrusted skill code during status checks.

## Reproduction Assessment

Yes, by source inspection. A skill declaring metadata.openclaw.requires.env with credentials only in skill-owned config files remains ineligible because the current env predicate never probes those files.

## Solution Assessment

Unclear. A self-report hook may be too risky as the first fix; a declarative env-or-file requirement contract looks narrower, but the public metadata/API shape needs maintainer review.
