## Summary

This issue should stay open because current main still treats skill readiness as a declared-requirements check: env requirements are satisfied only by process env, OpenClaw skill config env, or `apiKey` mapped through `primaryEnv`, not by skill-owned credential files such as the reported `~/.config/ima/*` fallback. The adjacent setup-hook proposal is related but explicitly separate rather than a canonical replacement.

## What This Changes

The issue requests a skill setup/readiness contract that avoids reporting `need_setup` when credentials are available through config files instead of declared environment variables.

## Best Possible Solution

Define a safe declarative readiness contract for credential alternatives such as env-or-file, then wire it through status, runtime eligibility, CLI/UI rendering, docs, and regression tests without executing arbitrary skill code during routine status checks.

## Reproduction Assessment

Yes, by source inspection. A skill that declares credentials only through `metadata.openclaw.requires.env` remains ineligible when those credentials exist only in skill-owned config files because the current predicates never probe those files.

## Solution Assessment

Unclear. The requested self-report option may be too risky as the first solution; a declarative env-or-file requirement contract is likely narrower, but the public metadata/API shape needs maintainer design review.
