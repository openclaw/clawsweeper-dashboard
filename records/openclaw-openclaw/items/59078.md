## Summary

Keep this issue open. Current main still derives skill readiness from declared metadata requirements, with env requirements satisfied only by process env or OpenClaw skill config; it has no safe contract for skill-owned credential files or skill setup-status reporting.

## What This Changes

The issue asks for skill setup/status detection to recognize credentials stored in skill-specific config files, or to let skills report setup status, instead of relying only on required env vars.

## Best Possible Solution

Define a safe declarative skill setup contract that can express credential alternatives such as env-or-file without executing untrusted skill code, then wire it through eligibility, skills.status, UI/CLI rendering, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a skill with metadata.openclaw.requires.env and credentials only in a skill-owned config file remains ineligible because the current eligibility path never reads those files.

## Solution Assessment

Unclear. The requested self-report hook may be too risky as a first step; a declarative credential-alternative requirement looks narrower and more maintainable, but it needs maintainer API review.
