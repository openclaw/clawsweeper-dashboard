## Summary

Keep open: current main and the latest shipped release still treat skill readiness as `requires.env` satisfied only by process env or OpenClaw skill config, so skill-owned credential files cannot clear the Skills page status or runtime eligibility. The related setup-hook discussion and published ClawHub plugin are useful context, but they do not settle this credential-readiness contract.

## What This Changes

The issue asks OpenClaw to avoid reporting a skill as needing setup when credentials are available through skill-owned config files instead of declared environment variables.

## Best Possible Solution

Define a safe declarative readiness contract for credential alternatives such as env-or-file, then wire it through status, runtime inclusion, CLI/UI rendering, docs, and regression tests without executing arbitrary skill code during routine status checks.

## Reproduction Assessment

Yes, by source inspection. A skill that declares credentials only through `metadata.openclaw.requires.env` remains ineligible when those credentials exist only in skill-owned config files because current main never probes those files.

## Solution Assessment

Unclear. The requested self-report option may be too risky as the first solution; a declarative env-or-file requirement contract is likely narrower, but the public metadata/API shape needs maintainer design review.
