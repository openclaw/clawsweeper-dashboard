## Summary

Keep open: current main and the latest release still satisfy `requires.env` only from process env, OpenClaw skill env config, or the matching `primaryEnv` API-key slot. The related setup-hook issue is useful context, but it is broader and does not settle this narrower credential-readiness contract or the security model for probing skill-owned credential files.

## What This Changes

The issue asks OpenClaw to avoid marking skills as needing setup when credentials are available through skill-supported config files rather than declared environment variables only.

## Best Possible Solution

Define a declarative credential-readiness contract for skills, then wire it through status, runtime eligibility, CLI/UI, docs, and regression tests without executing arbitrary skill code during routine checks.

## Reproduction Assessment

Yes, by source inspection. A skill with `requires.env` remains missing unless the required key is present in process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot; skill-owned config files are not checked.

## Solution Assessment

Unclear. Checking actual credential availability or adding skill self-reporting needs a maintainer decision; a declarative env-or-file readiness contract looks safer than running skill code during status checks.
