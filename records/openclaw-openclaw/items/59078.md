## Summary

Keep open: current `main` and the latest release still evaluate `requires.env` from process env, OpenClaw skill config env, or the `primaryEnv` API-key slot only, so a skill-owned config-file credential fallback still reports missing setup. The related setup-hook issue remains useful context, but it explicitly treats this `requires.env` credential-readiness conflict as separate remaining design work.

## What This Changes

Allow skill setup/readiness detection to recognize credentials supplied through skill-owned config files or a skill-defined readiness contract instead of marking `requires.env` missing.

## Best Possible Solution

Define a safe declarative credential-readiness contract, such as env-or-file or another explicit metadata shape, then wire it through status, runtime eligibility, CLI/UI rendering, docs, and regression tests without arbitrary status-time code execution.

## Reproduction Assessment

Yes, by source inspection. A skill declaring only `metadata.openclaw.requires.env` remains missing unless the value is in process env, OpenClaw skill env config, or the `primaryEnv` API-key slot; skill-owned credential files are never checked.

## Solution Assessment

Unclear. Checking arbitrary credential files or letting skills self-report setup is not automatically the best fix; a declarative readiness contract designed around redaction and no status-time code execution is the safer direction.
