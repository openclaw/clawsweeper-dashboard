## Summary

Keep open: current main still treats skill readiness as a declared `requires.env` check, so skill-owned credential files such as the reported `~/.config/ima/*` fallback cannot satisfy the Skills page or runtime eligibility. The adjacent setup-hook issue and open PR are related, but they explicitly leave this dual-purpose `requires.env` readiness problem as separate work.

## What This Changes

Support skill readiness when credentials are available through skill-owned config files instead of only declared environment variables.

## Best Possible Solution

Define a safe declarative readiness contract for credential alternatives such as env-or-file, then wire it through status, runtime eligibility, CLI/UI rendering, docs, and regression tests without executing arbitrary skill code during routine status checks.

## Reproduction Assessment

Yes, by source inspection. A skill that declares credentials only through `metadata.openclaw.requires.env` remains ineligible when those credentials exist only in skill-owned config files because current main never probes those files.

## Solution Assessment

Unclear. The requested self-report option may be too risky as the first solution; a declarative env-or-file requirement contract is likely narrower, but the public metadata/API shape needs maintainer design review.
