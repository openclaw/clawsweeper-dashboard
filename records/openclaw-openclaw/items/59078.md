## Summary

Keep open: current main and v2026.6.10 still treat `requires.env` readiness as satisfied only by process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot, so skill-owned credential files can still look missing. The related setup-hook request overlaps, but it tracks install/update setup execution rather than this credential-readiness/status contract.

## What This Changes

Request skill setup/readiness detection to avoid reporting setup needed when a skill can satisfy credentials from its own config files instead of environment variables.

## Best Possible Solution

Define a security-reviewed declarative skill credential-readiness contract and wire it consistently through status, runtime eligibility, CLI/UI, doctor, docs, and tests without arbitrary status-time skill code execution.

## Reproduction Assessment

Yes. Source inspection shows a skill with `requires.env` remains missing unless the required key is present in process env, `skills.entries.<skill>.env`, or the matching `primaryEnv` API-key slot; I did not run a live UI reproduction in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the credential-readiness contract. A declarative env-or-file credential source looks safer than executing arbitrary skill code during routine status checks.
