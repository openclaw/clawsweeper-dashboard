## Summary

Keep open: current main and the latest shipped release still satisfy `requires.env` only from process env, OpenClaw skill config env, or the `primaryEnv` API-key slot. The related setup-hook issue and published ClawHub plugin are useful context, but they leave this narrower credential-readiness contract unresolved.

## What This Changes

The issue requests skill readiness detection that can recognize credential alternatives such as skill-owned config files instead of only env vars or OpenClaw skill config.

## Best Possible Solution

Define a safe declarative credential-readiness contract for alternatives such as env-or-file, then wire it through status, runtime eligibility, CLI/UI rendering, docs, and regression tests without executing arbitrary skill code during routine status checks.

## Reproduction Assessment

Yes, by source inspection. A skill declaring only `metadata.openclaw.requires.env` remains missing unless the value is in process env, OpenClaw skill env config, or the `primaryEnv` API-key slot; skill-owned credential files are never checked.

## Solution Assessment

Unclear. The requested self-report option is likely too risky as the first fix; a declarative env-or-file requirement contract is narrower, but the public metadata and credential-check trust model need maintainer design review.
