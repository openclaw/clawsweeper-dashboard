## Summary

Keep open: the underlying Control UI streaming auto-scroll bug is source-reproducible and the patch is in the right area, but the latest guard still skips scroll-position bookkeeping for suppressed programmatic events and the external real-behavior proof is not yet verifiably sufficient.

## What This Changes

The PR adds a Control UI chat programmatic-scroll guard, host state, focused app-scroll tests, and a changelog entry for streaming auto-scroll behavior.

## Best Possible Solution

Keep the app-scroll guard, update scroll-position bookkeeping before suppressing only near-bottom/new-message mutations, add a regression for the next real upward scroll after a suppressed programmatic event, and land only after viewable redacted Control UI streaming proof is in the PR body.

## Reproduction Assessment

Yes, source-level. Current main schedules scrolls during chatStream updates and then handleChatScroll can treat the resulting programmatic scroll event as user state after content growth changes distanceFromBottom.

## Solution Assessment

No. The guard belongs in app-scroll, but this implementation returns before updating chatLastScrollTop, which is not the narrowest safe fix for the documented upward-scroll control behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve scroll direction bookkeeping before returning:** `ui/src/ui/app-scroll.ts:153-157`
  - body: This guarded return still skips the chatLastScrollTop update below it. Update the scroll-position baseline before suppressing only the near-bottom/new-message state changes.
