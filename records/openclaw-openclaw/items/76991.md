## Summary

Keep open: the linked WebChat auto-scroll bug is real on current main, but the PR's blanket programmatic-scroll guard can drop genuine user scroll-up events while the guard is active, violating the existing pause-on-user-scroll behavior.

## What This Changes

The PR adds a `chatIsProgrammaticScroll` guard around Control UI chat auto-scroll writes, focused tests, and a changelog entry for streaming WebChat pinning.

## Best Possible Solution

Make the guard suppress only scroll events attributable to OpenClaw's own scroll write while preserving immediate user scroll-up detection, then keep the focused regression coverage and changelog entry.

## Reproduction Assessment

Yes, source-level: current main's `scheduleChatScroll` writes to the bottom and `handleChatScroll` can then mark `chatUserNearBottom` false from a large post-growth distance. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No; the proposed boolean early return fixes the reported source path but is not the narrowest maintainable solution because it also hides user-originated scroll events while active. A safer fix should consume or identify only the programmatic scroll event and add a regression test for user scroll during the guard window.
