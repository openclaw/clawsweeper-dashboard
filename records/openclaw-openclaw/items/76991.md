## Summary

Keep open. The source-level Control UI streaming-scroll bug is real and this PR is focused, but the patch still skips scroll-position bookkeeping on guarded programmatic events, and the attached recording could not be visually verified while the PR body still says no live browser streaming smoke was performed.

## What This Changes

The branch adds a Control UI chat programmatic-scroll guard, host state, focused app-scroll tests, and a changelog entry for streaming auto-scroll behavior.

## Best Possible Solution

Keep the app-scroll guard, update scroll-position bookkeeping before suppressing only near-bottom/new-message mutations, add a header-direction regression test, and require viewable redacted Control UI streaming proof in the PR body before merge.

## Reproduction Assessment

Yes, source-level. Current main calls scheduleChatScroll for streaming chat updates and then handleChatScroll treats the resulting scroll event as user state, which can flip chatUserNearBottom after streamed content grows.

## Solution Assessment

No. The guard belongs in app-scroll, but returning before updating chatLastScrollTop is not the narrowest safe fix because it can regress the documented scroll-up header-control behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve scroll bookkeeping for ignored events:** `ui/src/ui/app-scroll.ts:153-157`
  - body: When this guarded programmatic event returns before updating chatLastScrollTop, the next real upward scroll can be compared against a stale lower value and look like a downward scroll. That can hide the desktop chat controls even though Control UI docs say scrolling up restores them; update scroll-position bookkeeping before suppressing only the near-bottom/new-message state changes.
  - confidence: 0.84
