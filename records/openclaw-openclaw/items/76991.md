## Summary

Keep open: the reported Control UI webchat bug is source-reproducible and the PR is on the right surface, but the current guard still skips scroll-position bookkeeping and the attached recording was not inspectable here, so it should not merge yet.

## What This Changes

The PR adds a Control UI webchat programmatic-scroll guard, related app-scroll unit tests, state fields, and a changelog entry for streaming auto-scroll behavior.

## Best Possible Solution

Keep the app-scroll guard, but update scroll-position bookkeeping before suppressing only the near-bottom and new-message mutations, then provide viewable redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main schedules chat scrolls during streaming updates and handles the resulting scroll event as user state, so content growth between the programmatic scroll and event handling can flip `chatUserNearBottom` false.

## Solution Assessment

No. The guard belongs in `app-scroll`, but returning before `chatLastScrollTop` and direction bookkeeping is too broad; update that bookkeeping first, then suppress only the state mutations that should ignore programmatic scrolls.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve scroll bookkeeping for guarded scrolls:** `ui/src/ui/app-scroll.ts:153-157`
  - body: Returning here before computing `scrollTop`/`delta` and updating `chatLastScrollTop` leaves the last recorded position stale after an auto-scroll. The next real upward scroll from the bottom can look like a downward delta and keep or hide the desktop chat controls instead of restoring them, so update scroll-position bookkeeping before suppressing only near-bottom and new-message state changes.
