## Summary

Keep open: the PR targets a valid Control UI streaming-scroll bug, but the guard is too broad and can drop real user scroll events while it is set, so the patch needs a narrow revision before merge.

## What This Changes

The PR adds a `chatIsProgrammaticScroll` guard, tests, and a changelog entry so Control UI chat auto-scroll events do not flip the near-bottom state during streaming.

## Best Possible Solution

Revise the guard so it suppresses only scroll events attributable to OpenClaw’s own scroll write while preserving immediate user scroll-up detection during streaming, then keep the focused tests and changelog coverage.

## Reproduction Assessment

Yes, source-level: on current main, `handleChatScroll` with `scrollHeight` 3000, `scrollTop` 500, and `clientHeight` 400 sets `chatUserNearBottom` false with no programmatic-scroll distinction. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No; the proposed boolean guard fixes the reported source path but is not yet the best narrow fix because it ignores all scroll events while active. A safer version should preserve user-initiated scroll detection even if that event lands during the programmatic guard window.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

- **[P2] Preserve user scrolls while the guard is active:** `ui/src/ui/app-scroll.ts:143-145`
