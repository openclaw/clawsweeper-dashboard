## Summary

Keep open: the reported bug is source-reproducible and the PR targets the right scroll code, but current head still suppresses scroll events before updating scroll-position bookkeeping, and the posted recording could not be inspected enough to satisfy the real behavior proof gate.

## What This Changes

The PR adds a Control UI webchat programmatic-scroll guard, focused app-scroll tests, and a changelog entry for streaming auto-scroll behavior.

## Best Possible Solution

Keep the app-scroll guard idea, but update scroll-position bookkeeping before suppressing only the near-bottom and new-message state mutations, then add a regression test for the next user scroll after a suppressed programmatic event.

## Reproduction Assessment

Yes, source-level. Current main calls scheduleChatScroll on chat updates and handleChatScroll treats the resulting scroll event as user state, which can flip chatUserNearBottom after streamed content grows.

## Solution Assessment

No. The guard belongs in app-scroll, but returning before updating chatLastScrollTop is too broad because it can regress the existing header-control direction logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve scroll bookkeeping for guarded scrolls:** `ui/src/ui/app-scroll.ts:153-157`
  - body: Returning here before updating chatLastScrollTop leaves the last recorded scroll position stale after an auto-scroll. The next real upward scroll from the bottom can then have a positive delta and keep or hide the desktop chat controls instead of restoring them, so update scroll-position bookkeeping before suppressing only the near-bottom/new-message mutations.
  - confidence: 0.84
