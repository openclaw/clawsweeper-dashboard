## Summary

Keep open: the PR targets a valid Control UI streaming scroll bug and includes focused tests plus a changelog entry, but the current boolean guard still suppresses genuine user scroll events during the active programmatic-scroll window.

## What This Changes

The PR adds a Control UI chat programmatic-scroll flag, guards `handleChatScroll` during auto-scroll, resets/tests that state, and adds a changelog entry.

## Best Possible Solution

Keep the focused programmatic-scroll fix, but narrow the guard so OpenClaw suppresses only its own scroll write while immediate user scroll-up still pauses auto-scroll.

## Reproduction Assessment

Yes, source-level: current main can flip `chatUserNearBottom` false after an auto-scroll event when content growth makes `distanceFromBottom` exceed the threshold. The PR regression is also source-reproducible because a real scroll event during the rAF guard window returns before updating the near-bottom state.

## Solution Assessment

No; the proposed boolean guard is too broad because it drops all scroll events while active. The safer narrow fix should distinguish OpenClaw’s own scroll event from user-initiated scrolls and add a regression test for that timing window.
