## Summary

Keep open: this is a plausible narrow fix for the linked fallback-loss bug, but the current PR head still interpolates config-derived model strings into doctor output and debug logs without sanitizing ANSI/control characters.

## What This Changes

Adds a debug warning, a doctor warning, and resolver unit coverage for string-form `agents.defaults.model` configs that cannot carry fallbacks.

## Best Possible Solution

Keep the diagnostic approach, but sanitize model refs before both sinks and add unsafe-character regression coverage while leaving the documented string-vs-object semantics unchanged.

## Reproduction Assessment

Yes, at source level: current main returns `[]` for string model input and the runtime fallback path consumes that helper directly. I did not execute a live overload run because this review is read-only.

## Solution Assessment

No. Warning users is a reasonable narrow direction, but this implementation should sanitize config-derived strings before terminal or log output.
