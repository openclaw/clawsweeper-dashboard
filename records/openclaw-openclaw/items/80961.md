## Summary

Keep open: the diagnostic direction is useful and tied to a real source-level fallback loss, but the latest PR head still interpolates config-derived model strings into doctor output and debug logs without sanitizing ANSI/control characters.

## What This Changes

The PR adds a debug warning, a doctor diagnostic, and resolver coverage for string-form `agents.defaults.model` configs that cannot carry fallbacks.

## Best Possible Solution

Keep the diagnostic approach, route displayed model refs through the existing terminal/log sanitizers, and preserve the documented string-primary/object-fallback semantics.

## Reproduction Assessment

Yes, at source level: current main returns `[]` for string model input, and the gateway runtime consumes that helper for default fallbacks. I did not execute a live overload run in this read-only review.

## Solution Assessment

No. Warning users is a reasonable narrow direction, but this implementation should sanitize config-derived model strings before terminal or log output and add unsafe-character regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Sanitize the doctor model warning:** `src/commands/doctor-config-analysis.ts:216`
  - body: `agents.defaults.model` comes from local config, and this new note interpolates it directly into terminal output. A value with ANSI or control characters can manipulate the displayed doctor warning, so sanitize the model string before building the note and add unsafe-character coverage.
  - confidence: 0.87
- **[P2] Sanitize the debug model warning:** `src/config/model-input.ts:48`
  - body: The new debug message embeds the raw config model string before passing it to the subsystem logger. That logger redacts secrets but does not strip control characters from message text before console/file output, so sanitize this value before logging it.
  - confidence: 0.84
