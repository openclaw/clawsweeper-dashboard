## Summary

This should stay open: current source and the reporter's logs point to a real `/v1/responses` image-delivery bug where accepted native images can be discarded when the resolved Anthropic runtime model is treated as text-only.

## What This Changes

The issue reports that `/v1/responses` accepts `input_image` blocks but drops them before Anthropic Claude vision models receive the prompt.

## Best Possible Solution

Keep `/v1/responses` native image support aligned with the provider catalog so configured Anthropic Claude vision models resolve with `image` capability and preserve image blocks through the embedded runner.

## Reproduction Assessment

Yes, partially from source: the current path accepts `input_image`, forwards it as native images, then drops all images if the resolved model has `input: ["text"]`; the reporter supplied matching runtime logs. I did not establish a live Anthropic request against current main.

## Solution Assessment

Unclear until patched, but the likely best fix is to preserve provider-catalog image capability during Anthropic model resolution rather than bypassing sanitization or adding a user-facing option.
