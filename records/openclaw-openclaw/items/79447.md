## Summary

Keep open. The PR is a plausible focused fix with sufficient terminal proof, but it is not merge-ready because the new runtime profile-reference path is not mirrored in status/auth-label behavior and the credential-class guard can still let OAuth profiles bypass an explicit api-key provider entry.

## What This Changes

The branch adds per-entry provider `apiKey` profile-ID dereferencing in model auth, re-exports an auth-profile compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Centralize provider-entry profile-reference resolution so runtime auth and status labels use the same rule, with auth-mode-aware credential compatibility and focused coverage for split OpenRouter, visible status output, and incompatible OAuth references.

## Reproduction Assessment

Yes. Current main is source-reproducible: the resolver can try profile order before the late provider-entry `apiKey` fallback, and non-marker `apiKey` strings are treated as literal models.json credentials; the PR's remaining status-label mismatch is also source-reproducible from `resolveModelAuthLabel`.

## Solution Assessment

No. The runtime resolver direction is plausible, but the patch is not the best mergeable form until the new profile-reference semantics are shared with the auth label/status path and the credential guard respects api-key versus OAuth auth classes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Mirror profile references in auth labels:** `src/agents/model-auth.ts:745-764`
  - body: The new resolver path can return `profile:openrouter:key-b` for the split-provider config, but `/status` still goes through `resolveModelAuthLabel`, which only accepts stored profiles whose provider matches the selected provider and then falls back to `resolveUsableCustomProviderApiKey`. For `openrouter-minimax.apiKey: "openrouter:key-b"`, status will still report `api-key (models.json)` or a stale override instead of the profile the request uses. Add the same per-entry profile-ref handling to the label/status path and cover it.
  - confidence: 0.86
- **[P2] Reject OAuth refs for api-key provider entries:** `src/agents/model-auth.ts:296-300`
  - body: For non-api-key/token credentials, this helper delegates to provider compatibility, so a same-provider OAuth profile is accepted and returned before the explicit `auth: "api-key"` branch. That lets an `apiKey` profile reference override an api-key provider entry with an OAuth access token, contrary to the new credential-class guard. Restrict this path to api_key/token profiles unless the provider entry is explicitly using OAuth auth, and add a same-provider OAuth regression.
  - confidence: 0.78
