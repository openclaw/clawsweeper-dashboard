## Summary

Keep open. The PR targets a real split-provider auth failure and now has real terminal proof, but the patch dereferences any matching stored profile ID from `models.providers.<id>.apiKey` without the provider-compatibility and terminal-failure behavior used elsewhere in auth routing.

## What This Changes

The PR adds per-entry `models.providers.<id>.apiKey` profile-ID resolution in `resolveApiKeyForProvider`, regression tests, and a changelog entry.

## Best Possible Solution

Define a provider-compatible profile-reference binding for split provider entries, centralize the rule in runtime auth, and cover incompatible-provider, explicit-auth, and fallback regressions.

## Reproduction Assessment

Yes, by source inspection. Current main resolves auth-profile order before the provider-entry `apiKey` fallback and treats non-marker `apiKey` strings as literal credentials, so a split provider can miss or misuse the intended stored profile.

## Solution Assessment

No. The happy path is useful, but this patch is not the best mergeable solution because it bypasses existing provider-compatibility checks and does not make matched-but-unresolvable profile references terminal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Constrain profile references to compatible providers:** `src/agents/model-auth.ts:688`
  - body: This lookup treats any `apiKey` string that matches any stored profile ID as usable for the current provider. `resolveApiKeyForProfile` validates the named profile itself, but it does not validate the target provider or endpoint, so a provider entry can route another provider's stored credential to its own endpoint. Require same-provider, aliased-provider, or another explicit safe compatibility check before returning the secret.
  - confidence: 0.87
- **[P2] Treat matched profile references as terminal:** `src/agents/model-auth.ts:700-713`
  - body: Once `store.profiles[perEntryRawKey]` matches, the value is known to be a profile reference. If resolving that profile returns null or throws, this block continues into env/profile/custom-key fallback, which can silently use a different credential or eventually send the profile ID string as the bearer token again.
  - confidence: 0.84
- **[P2] Resolve profile references before api-key overrides:** `src/agents/model-auth.ts:687-688`
  - body: This check runs after the existing `auth: "api-key"` early return, so a provider entry that sets both `auth: "api-key"` and `apiKey: "openrouter:key-b"` still returns the profile ID as a literal `models.json` key before reaching the new logic. Move the reference handling earlier or teach the custom-key resolver to distinguish stored profile IDs.
  - confidence: 0.78
