## Summary

Keep open: the PR targets a real, source-reproducible auth routing problem, but the patch needs real behavior proof and safer credential-eligibility handling before merge.

## What This Changes

The PR updates `resolveApiKeyForProvider`, adds model-auth profile tests, and adds a changelog entry so per-entry `apiKey` values matching stored auth profile IDs resolve to stored credentials.

## Best Possible Solution

Keep the PR open and revise the resolver so profile-id references are terminal, provider-compatible credential references with regression coverage for mismatched, unresolved, and explicit `auth: "api-key"` cases.

## Reproduction Assessment

Yes, by source inspection: current main treats `models.providers.<id>.apiKey: "openrouter:key-b"` as a non-marker custom key and can return that profile ID string as the bearer token when no auth order is available for the split provider.

## Solution Assessment

No. The proposed direction is useful, but the direct store lookup bypasses provider eligibility and does not stop fallback after a matched profile reference fails to resolve.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Constrain profile references to compatible providers:** `src/agents/model-auth.ts:693`
  - body: This lookup treats any `apiKey` string that matches any stored profile ID as usable for the current provider. `resolveApiKeyForProfile` only validates the named profile itself, while the existing auth-order path rejects provider mismatches; with this patch, a custom provider entry can route another provider's stored credential to its own `baseUrl`. Please require same/aliased provider compatibility or an explicit safe mapping before returning the secret.
  - confidence: 0.87
- **[P2] Stop falling through after a matched profile reference fails:** `src/agents/model-auth.ts:702-720`
  - body: Once `store.profiles[perEntryRawKey]` matches, the value is known to be a profile reference, but if `resolveApiKeyForProfile` returns null or throws, this block continues into auth-order/env/custom-key fallback. That can silently use a different profile or eventually send the profile ID string as the bearer token again. Please surface the profile resolution failure or otherwise keep the reference terminal.
  - confidence: 0.84
- **[P2] Resolve references before the explicit api-key override:** `src/agents/model-auth.ts:689-693`
  - body: This block runs after the `auth: "api-key"` early return, so a provider entry that sets both `auth: "api-key"` and `apiKey: "openrouter:key-b"` still returns the profile ID as a literal models.json key before reaching the new logic. Move the profile-reference check earlier or teach the custom-key resolver to distinguish stored profile IDs.
  - confidence: 0.78
