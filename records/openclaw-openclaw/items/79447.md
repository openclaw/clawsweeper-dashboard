## Summary

Keep open. The branch fixes the earlier early-return and fallthrough problems and includes useful terminal proof, but it still makes `models.providers.*.apiKey` dereference stored profile IDs with only credential-class compatibility, which can route stored same-class credentials to an arbitrary provider entry and needs maintainer security/product judgment before merge.

## What This Changes

The PR adds per-entry `apiKey` profile-ID dereferencing in model auth, re-exports an auth-profile compatibility helper, adds resolver regression tests, and adds a changelog entry.

## Best Possible Solution

Define an explicit provider-entry auth-profile binding or a provider-compatible profile-reference rule, then apply it consistently in runtime auth with focused split-provider and incompatible-provider tests.

## Reproduction Assessment

Yes, by source inspection. Current main can treat `openrouter:key-b` in a provider entry's `apiKey` as a literal `models.json` bearer after profile-order resolution misses the split provider ID.

## Solution Assessment

No. The PR fixes the happy path and prior fallthrough issues, but class-only compatibility is not a sufficient final contract for dereferencing stored secrets from a provider config field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Require provider-compatible profile references:** `src/agents/model-auth.ts:289-290`
  - body: This helper accepts any stored `api_key` or `token` profile before checking whether that credential belongs to the target provider. That means a provider entry can set `apiKey` to a known profile ID for another provider and OpenClaw will send that stored bearer to the entry's endpoint; keep the split-provider fix behind an explicit safe mapping or provider-compatible alias instead of class-only acceptance.
  - confidence: 0.82
