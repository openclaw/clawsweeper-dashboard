## Summary

Keep open. The added PR-body terminal proof is sufficient for the real-behavior gate, but the patch still has blocking credential-routing issues: it dereferences any matching stored profile ID without provider compatibility checks, falls through after a matched profile reference fails, and misses the existing `auth: "api-key"` early-return path.

## What This Changes

The PR adds direct profile-ID dereferencing for per-entry `models.providers.<id>.apiKey` values in `resolveApiKeyForProvider`, plus regression tests and a changelog entry.

## Best Possible Solution

Define a provider-compatible profile-reference path for per-entry provider auth, reuse the existing auth-profile eligibility checks, treat a matched-but-unresolvable reference as terminal, and cover both split-OpenRouter and incompatible-provider regressions.

## Reproduction Assessment

Yes, by source inspection. Current main can miss a split custom provider's intended stored profile because the custom provider id has no OpenRouter auth alias, then later treats the `apiKey` profile ID string as a literal bearer token.

## Solution Assessment

No. The patch proves the intended happy path, but it bypasses existing provider-compatibility checks and leaves fallback/early-return paths that can still send the wrong credential or literal profile ID.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Constrain profile references to compatible providers:** `src/agents/model-auth.ts:688`
  - body: This lookup treats any `apiKey` string that matches any stored profile ID as usable for the current provider. `resolveApiKeyForProfile` validates the named profile itself, but it does not validate the target provider/base URL, so a provider entry can route another provider's stored credential to its own endpoint. Please require same/aliased provider compatibility or an explicit safe mapping before returning the secret.
  - confidence: 0.87
- **[P2] Treat matched profile references as terminal:** `src/agents/model-auth.ts:697-711`
  - body: Once `store.profiles[perEntryRawKey]` matches, the value is known to be a profile reference. If resolving that profile returns null or throws, this block currently continues into env/profile/custom-key fallback, which can silently use a different credential or eventually send the profile ID string as the bearer token again.
  - confidence: 0.84
- **[P2] Resolve profile references before api-key overrides:** `src/agents/model-auth.ts:684-688`
  - body: This check runs after the existing `auth: "api-key"` early return, so a provider entry that sets both `auth: "api-key"` and `apiKey: "openrouter:key-b"` still returns the profile ID as a literal models.json key before reaching the new logic. Move the reference handling earlier or teach the custom-key resolver to distinguish stored profile IDs.
  - confidence: 0.78
