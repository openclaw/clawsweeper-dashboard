## Summary

Keep open. The PR addresses a real split-provider auth bug and now includes sufficient terminal proof, but the current patch introduces credential-routing and fallback bugs that should block merge until the provider-entry profile-reference contract is repaired.

## What This Changes

The PR makes `resolveApiKeyForProvider` dereference a provider-entry `apiKey` value that matches an auth profile ID, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Define a provider-compatible per-entry auth-profile binding, reuse existing eligibility checks, make matched profile references terminal on failure, and cover split-OpenRouter plus incompatible-provider regressions.

## Reproduction Assessment

Yes. Current main is source-reproducible: split provider ids without an auth alias can miss the intended OpenRouter profile and then treat a non-marker provider `apiKey` string as the literal bearer token.

## Solution Assessment

No. The proposed path proves the happy case, but it bypasses provider compatibility checks and leaves fallback/early-return behavior that can still route the wrong credential.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Constrain profile references to compatible providers:** `src/agents/model-auth.ts:688`
  - body: This lookup treats any `apiKey` string that matches any stored profile ID as usable for the current provider. `resolveApiKeyForProfile` validates the named profile itself, but it does not compare the stored credential with the target provider/base URL, unlike `resolveAuthProfileOrder`; a custom provider entry can send another provider's stored secret to its endpoint. Require same/aliased provider compatibility before returning the secret.
  - confidence: 0.87
- **[P2] Treat matched profile references as terminal:** `src/agents/model-auth.ts:697-711`
  - body: Once `store.profiles[perEntryRawKey]` matches, the value is known to be a profile reference. If resolving that profile returns null or throws, this block falls through to env/profile/custom-key fallback, which can silently use a different credential or eventually send the profile ID string as the bearer token again.
  - confidence: 0.84
- **[P2] Resolve profile references before api-key overrides:** `src/agents/model-auth.ts:684-688`
  - body: This new lookup runs after the existing `auth: "api-key"` early return, so a provider entry that sets both `auth: "api-key"` and `apiKey: "openrouter:key-b"` still returns the profile ID as a literal models.json key before reaching the new logic. Move the reference handling earlier or teach the custom-key resolver to distinguish stored profile IDs.
  - confidence: 0.78
