## Summary

Keep open. The PR targets a real split-provider auth bug, but the current patch is not mergeable because it dereferences stored profile IDs from provider config without target-provider compatibility, can fall through after a matched profile reference fails, misses the explicit `auth: "api-key"` path, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a direct `resolveApiKeyForProvider` lookup for per-entry `apiKey` strings that match stored auth profile IDs, adds three profile-resolution tests, and adds a changelog entry.

## Best Possible Solution

Define an explicit provider-entry auth-profile binding or safe compatibility mapping, then apply it consistently across runtime resolution, session profile selection, status labels, docs/schema, and focused split-provider regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main treats a non-marker `models.providers.<id>.apiKey` string as a literal models.json credential, while the related issue gives a concrete split-OpenRouter setup where that string is intended to name a stored profile.

## Solution Assessment

No. The direction is useful, but the patch should not bypass provider eligibility or silently fall through; the safer solution is an explicit, documented provider-entry profile binding or compatibility rule shared by all auth surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Constrain profile references to compatible providers:** `src/agents/model-auth.ts:688`
  - body: This lookup treats any `apiKey` string that matches any stored profile ID as usable for the current provider. `resolveApiKeyForProfile` validates the named profile itself, but it does not validate the current provider, so a provider entry can route another provider's stored credential to its own `baseUrl`. Please require same/aliased provider compatibility or an explicit safe mapping before returning the secret.
  - confidence: 0.87
- **[P2] Treat matched profile references as terminal:** `src/agents/model-auth.ts:697-711`
  - body: Once `store.profiles[perEntryRawKey]` matches, the value is known to be a profile reference. If `resolveApiKeyForProfile` returns null or throws, this block continues into auth-order/env/custom-key fallback, which can silently use a different profile or eventually send the profile ID string as the bearer token again.
  - confidence: 0.84
- **[P2] Resolve profile references before api-key overrides:** `src/agents/model-auth.ts:684-688`
  - body: This new check runs after the `auth: "api-key"` early return on the existing path, so a provider entry that sets both `auth: "api-key"` and `apiKey: "openrouter:key-b"` still returns the profile ID as a literal models.json key before reaching the new logic. Move the reference check earlier or teach the custom-key resolver to distinguish stored profile IDs.
  - confidence: 0.78
