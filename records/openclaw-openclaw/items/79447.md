## Summary

Keep open. The PR addresses the reported literal profile-ID-as-bearer path and includes useful terminal proof, but the latest head still accepts any stored bearer-shaped profile ID across provider IDs/endpoints, which needs a provider-compatible auth contract before merge.

## What This Changes

The PR adds per-entry `models.providers.<id>.apiKey` profile-ID dereferencing in `resolveApiKeyForProvider`, re-exports an auth-profile compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Define a provider-compatible profile-reference rule or explicit provider-entry auth-profile binding, then apply it consistently across runtime auth, session/status surfaces, docs/schema, and regression tests.

## Reproduction Assessment

Yes. Current main is source-reproducible: profile order can run before the late provider-entry `apiKey` fallback, and non-marker `apiKey` strings are still treated as literal `models.json` credentials.

## Solution Assessment

No. The PR fixes the requested happy path and prior fallthrough cases, but class-only compatibility is not the narrowest safe contract for dereferencing stored secrets from a provider config field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require provider-compatible profile references:** `src/agents/model-auth.ts:290-291`
  - body: This accepts every stored `api_key` or `token` profile before checking whether the credential belongs to the target provider. A provider entry can therefore set `apiKey` to another provider's profile ID and OpenClaw will send that stored bearer to the entry's endpoint; keep the split-provider fix behind an explicit safe mapping or provider-compatible alias rule.
  - confidence: 0.86
