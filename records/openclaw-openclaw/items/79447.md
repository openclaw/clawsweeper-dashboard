## Summary

Keep open. The PR fixes the focused runtime resolver happy path and includes usable terminal proof, but it still needs maintainer judgment because the new profile-reference contract allows same-class cross-provider stored bearer routing and is not shared with the status/model-label surfaces.

## What This Changes

The PR adds per-entry `models.providers.<id>.apiKey` stored profile-ID dereferencing in `resolveApiKeyForProvider`, re-exports an auth-profile compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Define an explicit provider-entry auth-profile binding or alias rule, then apply it consistently across runtime auth, status/model labels, docs/schema, and focused tests without allowing arbitrary cross-provider secret routing.

## Reproduction Assessment

Yes. The current-main path is source-reproducible: profile-looking provider `apiKey` strings are literal credentials, and the PR body supplies after-fix terminal proof for the patched resolver.

## Solution Assessment

No. The runtime happy path is useful, but class-only compatibility is not the safest final contract and the new binding is not shared with status/model-label behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require provider-compatible profile references:** `src/agents/model-auth.ts:290-291`
  - body: This accepts every stored `api_key` or `token` profile before checking whether it belongs to the target provider. A custom provider entry can therefore reference another provider's stored bearer and OpenClaw will send that secret to the entry's `baseUrl`; keep split-provider support behind an explicit alias/binding or provider-compatible rule.
  - confidence: 0.84
- **[P2] Share profile-reference resolution with status labels:** `src/agents/model-auth.ts:760-765`
  - body: The new dereference path only runs inside `resolveApiKeyForProvider`. `/status` and model-list labels still use the unchanged `resolveModelAuthLabel` fallback, so `openrouter-minimax.apiKey: "openrouter:key-b"` resolves as `profile:openrouter:key-b` at runtime but is still labeled as `models.json` instead of the matched profile.
  - confidence: 0.78
