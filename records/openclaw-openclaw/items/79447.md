## Summary

Keep open: this targets a real split-provider auth bug and includes sufficient terminal proof, but the patch still changes stored-secret routing with class-only cross-provider bearer acceptance and does not share the new profile-reference resolution with status/model labels.

## What This Changes

The PR adds per-entry provider `apiKey` profile-ID dereferencing in model auth, re-exports a compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Define an explicit provider-entry auth-profile binding or alias rule, then apply it consistently across runtime auth, session auto-selection, status/model labels, docs/schema, and focused tests without arbitrary cross-provider bearer routing.

## Reproduction Assessment

Yes. Current main is source-reproducible: non-marker provider `apiKey` strings are treated as literal `models.json` credentials, and runtime/status paths still consult auth-profile order without a provider-entry binding rule.

## Solution Assessment

No. The runtime dereference solves the happy path, but class-only compatibility is not a safe final credential contract and the new rule is not shared with status/model-label behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require provider-compatible profile references:** `src/agents/model-auth.ts:290-291`
  - body: This returns true for every stored `api_key` or `token` profile before checking whether the credential belongs to the target provider. A custom provider entry can therefore reference another provider's stored bearer and OpenClaw will send that secret to the entry's `baseUrl`; keep split-provider support behind an explicit alias, binding, or provider-compatible rule.
  - confidence: 0.84
- **[P2] Share profile-reference resolution with labels:** `src/agents/model-auth.ts:760-765`
  - body: The new dereference path only runs inside `resolveApiKeyForProvider`. `/status` and model-list labels still use `resolveModelAuthLabel` and fall back to `models.json`, so `openrouter-minimax.apiKey: "openrouter:key-b"` can authenticate with the matched profile at runtime while still displaying the wrong auth source.
  - confidence: 0.78
