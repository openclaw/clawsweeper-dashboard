## Summary

Keep open. The PR fixes the runtime resolver happy path, but it still defines a broad cross-provider stored-secret reference contract and does not carry the new binding into status or session auth-profile selection, so the linked split-provider issue is not fully solved.

## What This Changes

The PR adds per-entry `models.providers.<id>.apiKey` profile-ID dereferencing in `resolveApiKeyForProvider`, re-exports an auth-profile compatibility helper, adds resolver regression tests, and updates the changelog.

## Best Possible Solution

Define or document a provider-compatible profile binding for split provider entries and reuse one shared resolver across runtime auth, session auto-selection, status labels, schema/help, and regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main returns non-marker provider `apiKey` values as literal `models.json` credentials and status/session selection still follows provider auth-profile order, matching the split-OpenRouter wrong-profile path.

## Solution Assessment

No. Runtime-only apiKey profile-ID dereferencing is a useful part of the fix, but the class-only cross-provider rule and unupdated status/session paths are not the narrowest maintainable final contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Require provider-compatible profile references:** `src/agents/model-auth.ts:290-291`
  - body: This accepts every stored `api_key` or `token` profile as compatible before checking the target provider, so a provider entry can point at an unrelated stored bearer and OpenClaw will send it to that entry's endpoint. Keep the split-provider case behind an explicit provider-compatible alias or binding instead of class-only matching.
  - confidence: 0.82
- **[P2] Share the profile binding with status and session auth:** `src/agents/model-auth.ts:760-767`
  - body: The new dereference only runs in `resolveApiKeyForProvider`; `/status` labels still call `resolveModelAuthLabel` and session auto-selection still calls `resolveAuthProfileOrder`, so the reported split-provider setup can continue to display or persist the wrong canonical profile even after dispatch uses the intended key. Reuse one provider-entry binding path across these surfaces.
  - confidence: 0.78
