## Summary

Keep open: the PR addresses the runtime happy path, but it still leaves a security-sensitive auth binding contract and the user-visible auth label path inconsistent.

## What This Changes

The branch adds per-entry provider `apiKey` profile-ID dereferencing in model auth, exports an auth-profile compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Define one explicit provider-entry auth-profile binding rule and apply it consistently across runtime auth, status/model labels, docs/schema, and focused split-provider tests.

## Reproduction Assessment

Yes. Current main is source-reproducible: profile-looking provider `apiKey` strings still fall through as literal `models.json` credentials, while auth-profile order and label paths remain provider/order oriented.

## Solution Assessment

No. The runtime dereference is useful, but class-only compatibility and a runtime-only implementation are not the safest final contract for split-provider credentials.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require provider-bound profile references:** `src/agents/model-auth.ts:290-291`
  - body: This accepts every stored `api_key` or `token` profile before checking whether the credential belongs to the target provider or an explicit alias. A config entry can therefore reference an unrelated stored bearer profile and OpenClaw will send that secret to the custom provider endpoint; require an explicit provider binding, alias, or provider-compatible rule for these references.
  - confidence: 0.84
- **[P2] Share profile-reference resolution with auth labels:** `src/agents/model-auth.ts:760-765`
  - body: The new profile-reference rule only runs in `resolveApiKeyForProvider`. `/status` and model-list labels still use `resolveModelAuthLabel` and then fall back to `api-key (models.json)`, so the same `openrouter-minimax.apiKey: "openrouter:key-b"` setup can authenticate with the matched profile while showing the wrong user-visible auth source.
  - confidence: 0.78

## Security Review

Status: needs_attention
