## Summary

Keep open. The PR addresses a real split-provider auth bug, and the supplied terminal proof is enough for the real-behavior gate, but the patch is not ready because the new profile-reference rule is runtime-only and can also intercept env SecretRef-backed apiKey values.

## What This Changes

The PR adds per-entry provider `apiKey` profile-ID dereferencing in `resolveApiKeyForProvider`, re-exports an auth-profile compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Centralize the provider-entry auth-profile binding so runtime auth, status labels, and tests all share one rule that distinguishes literal credentials, SecretRefs, and stored profile IDs, with docs/schema updates if maintainers accept this config contract.

## Reproduction Assessment

Yes. Source inspection on current main shows provider-entry `apiKey` strings are literal `models.json` credentials and runtime auth resolves profile order before the late provider apiKey fallback, matching the split-provider failure mode.

## Solution Assessment

No. The PR improves the runtime happy path, but it applies the new rule through a helper that also returns env SecretRef ids and does not share the rule with status/model auth labels.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve env SecretRef apiKey values:** `src/agents/model-auth.ts:310-315`
  - body: The new helper reads `apiKey` through `getCustomProviderApiKey`, which returns the `id` for env SecretRefs as well as literal strings. If a SecretRef-backed provider uses an env id that matches a stored profile id, this branch dereferences the profile and never resolves the configured env variable. Inspect the raw provider config and only treat literal string values as profile references.
  - confidence: 0.75
- **[P2] Reuse profile-ref resolution for auth labels:** `src/agents/model-auth.ts:760-765`
  - body: The profile-reference path only runs in `resolveApiKeyForProvider`; `resolveModelAuthLabel` still falls through to `resolveUsableCustomProviderApiKey`. The split-provider config fixed at runtime is therefore still shown as `api-key (models.json)` in status instead of the matched profile, leaving the reported status symptom misleading.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The diff changes stored-secret routing and needs an explicit maintainer-approved contract for profile IDs inside the provider apiKey field.

Concerns:

- **[medium] Clarify profile ID versus credential routing:** `src/agents/model-auth.ts:310`
  - body: The new branch lets a provider `apiKey` value resolve a stored profile and allows bearer-shaped profiles across provider IDs, while the same field is also a literal/SecretRef credential surface. Without a clearer binding rule, configuration can silently route a stored credential source that the operator or status label does not expect.
