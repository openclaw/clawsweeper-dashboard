## Summary

Keep open. Current main still has the split-provider auth gap and this PR fixes the happy path, but the latest head still allows same-class stored bearer profiles to be dereferenced across provider boundaries, which needs security/product review before merge.

## What This Changes

The PR adds per-entry `models.providers.<id>.apiKey` profile-ID dereferencing in `resolveApiKeyForProvider`, re-exports an auth-profile compatibility helper, adds regression tests, and updates the changelog.

## Best Possible Solution

Define a provider-compatible profile-reference rule or explicit provider-entry auth-profile binding, then apply it consistently across runtime auth, status/session surfaces, docs/schema, and regression tests.

## Reproduction Assessment

Yes, by source inspection. Current main can resolve profile order before the late provider-entry `apiKey` fallback, and non-marker `apiKey` strings are treated as literal `models.json` credentials.

## Solution Assessment

No. The PR fixes the requested happy path and previous fallthrough issues, but class-only compatibility is not the safest final contract for dereferencing stored secrets from a provider config field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Require provider-compatible profile references:** `src/agents/model-auth.ts:290-291`
  - body: This returns true for every stored `api_key` or `token` profile before checking whether the credential belongs to the target provider. A provider entry can therefore set `apiKey` to another provider's profile ID and OpenClaw will send that stored bearer to the entry's endpoint; keep the split-provider fix behind an explicit safe mapping or provider-compatible alias rule.
