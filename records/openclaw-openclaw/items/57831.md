## Summary

Keep open: the PR targets a real cron/session isolation bug, but the current patch still has blocking compile and classifier defects, and the external security-sensitive change lacks required after-fix real behavior proof.

## What This Changes

The PR adds cron session-target guarding during normalization, tool/gateway update paths, runtime fallback, and regression tests to keep direct chat sessions out of cron runs.

## Best Possible Solution

Use a shared, well-tested cron session-key classifier at persistence and runtime that blocks real direct-message sessions, preserves group/channel and documented custom named sessions, and pairs with the narrower deleteAfterRun ownership guard where needed.

## Reproduction Assessment

Yes. Current main source plus the linked live-DM report provide a high-confidence path: current can persist a live direct session key, runtime reuses session:* targets, and deleteAfterRun cleanup can delete that same key.

## Solution Assessment

No, not as written. The layered guard is the right direction, but this patch must compile, classify only real direct chat session keys, preserve custom sessions, and include real after-fix behavior proof before merge.

  - confidence: 0.93

  - body: When parseAgentSessionKey fails, the guard returns false before applying the chat-type classifier, so legacy or unscoped direct keys like telegram:dm:123 can still be persisted or reused by cron instead of downgraded.
  - confidence: 0.87
- **[P1] Prefer direct markers before broad group tokens:** `src/cron/session-target-guard.ts:17`
  - body: The guard delegates to a helper that returns group or channel before checking direct/dm, so account-qualified direct keys with a group or channel account token can be misclassified as reusable.
  - confidence: 0.82
- **[P2] Preserve arbitrary custom session IDs:** `src/cron/session-target-guard.ts:8-9`
  - body: Treating every parseable agent:<id>:... value as a channel session means custom targets such as session:agent:billing:direct:daily can be downgraded even though session:<custom-id> is documented as a persistent named session.
  - confidence: 0.84
