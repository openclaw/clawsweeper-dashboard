## Summary

Keep open: the underlying cron/session isolation concern is source-reproducible, but this branch is not merge-ready because it is stale against current main, has a broken import and unsafe classifier behavior, lacks real behavior proof, and changes documented session-target semantics.

## What This Changes

The PR adds cron session-target normalization/runtime guards, update-path session context forwarding, and focused cron/tool/gateway tests to isolate direct-message cron jobs instead of reusing live DM session keys.

## Best Possible Solution

Rebase and implement a shared canonical session-target eligibility helper that blocks real direct-message cron targets, preserves documented custom sessions, covers legacy keys, and coordinates with the existing cron-owned cleanup guard.

## Reproduction Assessment

Yes, at source level: current main resolves `current` to `session:<sessionKey>` and runtime `session:<id>` targets to that exact key. I did not run live Feishu/Lark/Discord proof in this read-only review.

## Solution Assessment

No, not as written. The layered guard is a plausible direction, but it must compile on current main, use a canonical classifier, preserve custom-session compatibility, and get maintainer/security policy approval plus real behavior proof.
