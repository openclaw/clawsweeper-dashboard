## Summary

Keep this PR open. Current main still has the direct-session cron isolation gap, and the PR targets the right core surface, but the branch is not merge-ready because it has a current-main import break and unresolved direct-key classification regressions.

## What This Changes

The PR adds a cron session-target guard that downgrades direct-chat `current` and `session:<id>` cron targets to isolated sessions, threads session context through update paths, and adds cron/tool/gateway tests.

## Best Possible Solution

Keep the PR or replace it with a focused core fix that uses one shared, well-tested session-key classifier for cron persistence and runtime fallback, covers canonical and legacy direct DM keys, preserves documented custom named sessions and safe group/channel reuse, and updates tests plus changelog for the user-visible behavior change.

## Reproduction Assessment

Yes. Static inspection of current main shows `current` cron targets still persist direct session keys and runtime still trusts explicit `session:<id>` targets; the PR review findings are also reproducible from the diff and current exported session helper modules.

## Solution Assessment

No, not as written. The direction is right, but the implementation should first compile against current main and classify only real direct chat session keys across canonical, legacy, and account-qualified shapes without breaking arbitrary custom session IDs.

Full review comments:

- **[P1] Import the chat-type helper from its exporting module:** `src/cron/session-target-guard.ts:1`
  - body: `session-target-guard.ts` imports `deriveSessionChatType` from `session-key-utils`, but current main does not export that symbol from that file. The helper is exported from `src/sessions/session-chat-type.ts`, so this new file will fail typecheck/build as soon as it is compiled.
  - confidence: 0.93
- **[P1] Cover legacy direct session keys in the guard:** `src/cron/session-target-guard.ts:15-16`
  - body: Returning `false` whenever `parseAgentSessionKey` fails lets existing unscoped direct keys such as `telegram:dm:123456` pass through as safe. Those keys are classified as direct elsewhere, so cron can still persist or reuse a live DM session instead of falling back to isolated.
  - confidence: 0.87
- **[P1] Classify direct keys before broad group/channel tokens:** `src/cron/session-target-guard.ts:18`
  - body: The guard relies on `deriveSessionChatType`, whose current implementation returns `group` or `channel` before checking `direct`/`dm` when those tokens appear anywhere. An account-qualified direct key with an account id like `group` can therefore be treated as reusable and bypass the fallback this PR is meant to enforce.
  - confidence: 0.82
- **[P2] Preserve arbitrary custom session IDs:** `src/cron/session-target-guard.ts:8-9`
  - body: Treating every parseable `agent:<id>:...` value as a channel session means custom targets like `session:agent:billing:direct:daily` are downgraded to `isolated` even though `session:<custom-id>` is documented as a persistent named session. Restrict the guard to canonical chat-session shapes before blocking direct chats.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/agents/tools/cron-tool.ts:460`
  - body: This is a user-visible bug fix and security hardening for cron session behavior, but the PR only changes source and tests. Add a `CHANGELOG.md` entry with appropriate contributor credit before merge.

## Security Review

Status: needs_attention

Summary: The diff is security-relevant and reduces one direct-session reuse path, but the proposed guard still leaves concrete DM-session bypasses and named-session regressions to resolve.

Concerns:

- **[high] Incomplete direct-session isolation guard:** `src/cron/session-target-guard.ts:15`
  - body: The guard only evaluates agent-scoped keys and delegates to a chat-type helper that can prefer broad group/channel tokens before direct markers. Legacy direct keys and some account-qualified direct keys can still be considered safe, leaving cron able to reuse a live DM session.
  - confidence: 0.84

  - body: The guard treats arbitrary `agent:<id>:direct:<name>` custom session IDs as real direct chats, which can silently change persistence for legitimate named cron sessions and conflicts with the documented `session:<custom-id>` behavior.
