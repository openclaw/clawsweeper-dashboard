## Summary

Keep open. The underlying cron/session bug is valid on current main and this PR is useful in direction, but the current patch is not merge-ready because it fails the import contract, leaves direct-session classifier bypasses, can over-isolate valid custom sessions, lacks after-fix real behavior proof, and is currently merge-conflicting.

## What This Changes

The PR adds cron session-target guards for direct-message sessions during create/update/runtime paths and adds targeted cron normalization, tool, and gateway tests.

## Best Possible Solution

Use a shared, well-tested cron session-key classifier and cleanup ownership guard that blocks real direct-message sessions, preserves documented custom session IDs, coordinates with https://github.com/openclaw/openclaw/pull/76177, and proves the behavior with regression tests plus redacted runtime evidence.

## Reproduction Assessment

Yes. Current main source still composes the path: `current` persists a live session key, runtime reuses `session:*`, one-shot jobs default cleanup on, and direct cleanup deletes the supplied key; https://github.com/openclaw/openclaw/issues/69467 provides sanitized live logs for the same path.

## Solution Assessment

No, not as written. The layered guard is the right direction, but the patch must compile, classify only real direct chat session keys, preserve documented custom sessions, resolve conflicts, and include after-fix real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Import the chat-type helper from its exporter:** `src/cron/session-target-guard.ts:1`
  - body: `deriveSessionChatType` is exported from `src/sessions/session-chat-type.ts`, not `session-key-utils.ts`. Importing it from this module will fail build/typecheck as soon as the new guard file is compiled.
  - confidence: 0.93
- **[P1] Classify legacy direct keys before returning safe:** `src/cron/session-target-guard.ts:14-16`
  - body: When `parseAgentSessionKey` fails, the guard returns `false` without consulting the chat-type classifier. Legacy or unscoped DM keys that the rest of the session code can classify as direct would still be persisted and reused by cron.
  - confidence: 0.86
- **[P1] Check direct markers before broad group tokens:** `src/cron/session-target-guard.ts:17`
  - body: The guard relies on a helper that returns `group` or `channel` before checking `direct`/`dm`, so keys containing both a broad token and a direct marker can be treated as reusable and bypass the isolation this PR intends to add.
  - confidence: 0.82
- **[P2] Preserve arbitrary custom session IDs:** `src/cron/session-target-guard.ts:8-9`
  - body: `parseAgentSessionKey` accepts any `agent:<id>:...` value, so custom targets such as `session:agent:billing:direct:daily` can be downgraded even though `session:<custom-id>` is documented and tested as a persistent named session.
  - confidence: 0.84
