## Summary

Keep open for maintainer/security review: current main has the destructive cleanup guard from the related issue, but it still permits direct-chat session targets and this PR’s unique isolation work is not on main. The branch is conflicting, has blocking classifier/build issues, and lacks after-fix real behavior proof.

## What This Changes

The PR adds cron normalization/runtime guards and tests intended to downgrade direct-chat `current`/`session:<id>` cron bindings to isolated sessions while preserving safe group/custom session reuse.

## Best Possible Solution

Rebase and replace the PR guard with a shared, well-tested session-key classifier that blocks real direct-message cron targets, preserves documented custom session IDs, and coordinates cleanly with the current cron-owned cleanup guard.

## Reproduction Assessment

Yes for the remaining direct-session reuse path from source: current main still resolves `current` to `session:<direct key>` and still passes explicit `session:*` targets into isolated cron runs. The previously reported destructive `deleteAfterRun` deletion path is no longer reproducible from source because cleanup now skips non-cron session keys.

## Solution Assessment

No, not as written. The layered guard is the right direction, but the branch must compile on current main, classify only real direct chat keys, preserve custom session IDs, and include redacted after-fix runtime proof before it is mergeable.
