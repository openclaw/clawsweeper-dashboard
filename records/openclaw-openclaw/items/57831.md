## Summary

Keep this PR open, but do not merge it as-is. Current main still has the cron direct-session isolation gap, and this PR targets the right cron/tool/gateway surfaces, but the latest diff still has a build blocker plus incomplete and overbroad session-key classification.

## What This Changes

The PR adds a cron session-target guard for direct-chat `current` and `session:<id>` targets, passes session context through cron update paths, and adds cron/tool/gateway regression tests.

## Best Possible Solution

Use one shared, well-tested session-key classifier at cron persistence and runtime that blocks canonical and legacy direct DM chat sessions, preserves safe group/channel and documented custom named sessions, and ships with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. Static inspection of current main shows `current` targets still persist the caller session and runtime still trusts explicit `session:<id>` targets; related issue #69467 provides concrete live-DM reproduction evidence.

## Solution Assessment

No, not as written. The direction is right, but the implementation should compile and classify only real direct chat session keys across canonical, account-qualified, and legacy shapes without breaking arbitrary custom session IDs.
