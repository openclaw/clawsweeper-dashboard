## Summary

Keep open. The reported cron/session isolation bug is still present on current main, and this external security-hardening PR is not merge-ready because the branch is conflicting, still has blocking classifier/build issues, overlaps with the cleanup guard in a related open PR, and lacks after-fix real behavior proof.

## What This Changes

The PR adds cron normalization, tool/gateway update, runtime fallback, and tests intended to keep direct-message session keys out of reusable cron agent-turn sessions.

## Best Possible Solution

Use a shared, well-tested cron session-key classifier that blocks real direct-message session keys, preserves documented custom session IDs, keeps destructive cleanup limited to cron-owned sessions, coordinates with the related cleanup PR, and includes redacted after-fix runtime proof.

## Reproduction Assessment

Yes. The related issue provides sanitized live logs, and current main still composes the path: current persists a live key, one-shot jobs default cleanup on, runtime reuses session:* keys, and direct cleanup deletes the supplied key.

## Solution Assessment

No, not as written. The layered guard is the right direction, but the patch must compile, classify only real direct chat session keys, preserve custom session IDs, coordinate with the cleanup guard, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the chat-type helper from its exporter:** `src/cron/session-target-guard.ts:1`
  - body: `deriveSessionChatType` is exported from `src/sessions/session-chat-type.ts`, not `session-key-utils.ts`. Importing it from this module will fail build/typecheck as soon as the new guard file is compiled.
  - confidence: 0.93
- **[P1] Classify legacy direct keys before returning safe:** `src/cron/session-target-guard.ts:14-16`
  - body: When `parseAgentSessionKey` fails, the guard returns `false` before consulting the chat-type classifier. Legacy direct keys such as `telegram:dm:123456` are still recognized as direct elsewhere, so they could still be persisted or reused by cron.
  - confidence: 0.86
- **[P1] Check direct markers before broad group tokens:** `src/cron/session-target-guard.ts:17`
  - body: The guard relies on a helper that returns `group` or `channel` before checking `direct`/`dm`. Direct keys with an account or scope segment named `group` or `channel` can be misclassified as reusable and bypass the intended isolation.
  - confidence: 0.82
- **[P2] Preserve arbitrary custom session IDs:** `src/cron/session-target-guard.ts:8-9`
  - body: `parseAgentSessionKey` accepts any `agent:<id>:...` value, so custom targets such as `session:agent:billing:direct:daily` can be downgraded even though `session:<custom-id>` is documented as a persistent named session.
  - confidence: 0.84
