## Summary

Keep open. This PR targets a real cron/session isolation bug, but the proposed patch still has blocking build/classifier issues, lacks contributor-supplied real behavior proof, and overlaps with the still-open cleanup guard work in https://github.com/openclaw/openclaw/pull/76177 and the canonical live-DM report at https://github.com/openclaw/openclaw/issues/69467.

## What This Changes

The PR adds cron session-target guards for direct-message session keys during normalization, tool/gateway update handling, runtime fallback, and related tests.

## Best Possible Solution

Use a shared, well-tested cron session-key classifier that blocks real direct-message session keys, preserves documented custom session IDs, guards destructive cleanup to cron-owned sessions, coordinates with https://github.com/openclaw/openclaw/pull/76177, and includes redacted after-fix runtime proof.

## Reproduction Assessment

Yes, from source inspection and the linked sanitized live logs. I did not rerun the destructive flow, but current main still persists `current` session keys, reuses `session:*` at runtime, defaults one-shot jobs to `deleteAfterRun`, and direct cleanup deletes `params.agentSessionKey`.

## Solution Assessment

No, not as written. The layered guard is the right direction, but the patch must compile, classify only real direct chat session keys, preserve custom session IDs, coordinate with the cleanup guard, and include real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the chat-type helper from its exporter:** `src/cron/session-target-guard.ts:1`
  - body: `deriveSessionChatType` is exported from `src/sessions/session-chat-type.ts`, not `session-key-utils.ts`. Importing it from this module will fail build/typecheck as soon as the new guard file is compiled.
  - confidence: 0.93
- **[P1] Classify legacy direct keys before returning safe:** `src/cron/session-target-guard.ts:14-16`
  - body: When `parseAgentSessionKey` fails, the guard returns `false` without consulting the chat-type classifier. Legacy or unscoped DM keys that session code can still classify as direct would be persisted and reused by cron.
  - confidence: 0.86
- **[P1] Check direct markers before broad group tokens:** `src/cron/session-target-guard.ts:17`
  - body: The guard relies on a helper that returns `group` or `channel` before checking `direct`/`dm`, so mixed-token direct keys can be treated as reusable and bypass the intended isolation.
  - confidence: 0.82
- **[P2] Preserve arbitrary custom session IDs:** `src/cron/session-target-guard.ts:8-9`
  - body: `parseAgentSessionKey` accepts any `agent:<id>:...` value, so custom targets such as `session:agent:billing:direct:daily` can be downgraded even though `session:<custom-id>` is documented as a persistent named session.
  - confidence: 0.84

## Security Review
