## What This Changes

The PR adds a cron session-target guard that downgrades direct `current`/`session:<direct-key>` targets to `isolated`, forwards session context through cron update paths, and falls back to `cron:<jobId>` for persisted direct targets.

## Best Possible Solution

Adopt a maintainer-approved cron session eligibility rule that blocks only real unsafe direct-channel reuse, preserves documented `current` and custom-session workflows or makes strict isolation explicit, and backs create/update/runtime behavior with tests plus redacted real setup proof.

Yes, at source level for the remaining behavior: current main still resolves direct `current`/`session:<id>` cron targets to persistent session keys and runs non-isolated jobs on that key. I did not run a live channel reproduction in this read-only review.

No, not as written. The layered guard is a plausible hardening direction, but it needs a maintainer policy decision, a rebased canonical classifier, custom-session compatibility protection, and real behavior proof before merge.
