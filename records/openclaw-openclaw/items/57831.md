## Summary

Keep open: this PR targets a real current-main cron/direct-message session isolation bug, but the branch still has blocking build/classifier defects and lacks required after-fix real behavior proof.

## What This Changes

The PR adds cron session-target guarding across normalization, tool/gateway update paths, runtime fallback for legacy dirty direct-session jobs, and regression tests.

## Best Possible Solution

Use a shared, well-tested cron session-key classifier at persistence and runtime that blocks real direct-message sessions while preserving safe group/channel and documented custom named sessions.

## Reproduction Assessment

Yes. Current main composes the failing path by persisting current as session:<caller sessionKey>, reusing explicit session:* at runtime, and allowing deleteAfterRun direct cleanup to delete that same key; the linked live-DM issue adds concrete sanitized logs.

## Solution Assessment

No, not as written. The layered guard is the right direction, but the patch must compile and classify only real direct chat session keys without legacy/account-qualified bypasses or custom-session regressions.

  - confidence: 0.84
