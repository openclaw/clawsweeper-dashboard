## Summary

Keep open. The remaining direct-session cron isolation policy is not implemented on current main, and this branch is not merge-ready because it has a stale import, incomplete/over-broad session classification, missing real behavior proof, and unresolved compatibility/security policy risk.

## What This Changes

The PR adds cron session-target guards, update-path session context forwarding, runtime fallback for direct session targets, and cron/tool/gateway tests to keep direct-message session keys out of cron runs.

## Best Possible Solution

Use a maintainer-approved canonical session-target eligibility helper that blocks real direct-message cron targets, preserves documented custom sessions, covers legacy keys, and coordinates with the existing cron-owned cleanup guard.

## Reproduction Assessment

Yes, at source level. The import/export mismatch and classifier behavior are visible in current main, and current main still resolves `current` and explicit direct `session:<id>` values into persistent session keys, though the destructive deletion half is now guarded separately.

## Solution Assessment

No, not as written. The layered guard is a plausible direction, but it needs a canonical classifier, custom-session compatibility protection, maintainer security/product approval, and real behavior proof before merge.

  - confidence: 0.86
- **[P1] Keep custom session IDs reusable unless they are real DMs:** `src/cron/session-target-guard.ts:9-17`
  - body: `parseAgentSessionKey()` accepts any `agent:<id>:...` shape, so a documented custom session like `session:agent:billing:direct:daily` can be downgraded to `isolated` even though it is not a channel DM. That silently breaks persistent named-session cron jobs.
  - confidence: 0.88
