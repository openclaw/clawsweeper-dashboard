## Summary

Keep open: current main and the latest release still pass selected subagent completion text into the parent task-completion event without a dedicated child-result injection cap. The newer metadata-only completion issue overlaps, but it does not fully supersede this narrower cap/default/config decision, and the linked implementation PR is closed unmerged.

## What This Changes

Request to add a configurable cap for subagent result text before it is injected into parent-session task-completion events.

## Best Possible Solution

Choose the subagent completion contract first, then bound child-result prompt injection while preserving the prompt-data wrapper, frozen-result recovery, and explicit child-history/artifact inspection paths.

## Reproduction Assessment

Yes, at source level: current main still passes selected child completion text into a parent task-completion internal event without a dedicated result cap. I did not run a live oversized-subagent scenario in this read-only review.

## Solution Assessment

Unclear as proposed: bounding the requester handoff is the right area, but the exact `session.maxSubagentResultChars` key and default are product/API choices. The broader metadata-only completion request should be considered before adding a narrower config knob.
