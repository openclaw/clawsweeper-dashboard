## Summary

Keep open. Current main still has the reported 10s session write-lock acquire default, several transcript/session writers either omit `timeoutMs` or hard-code `10_000`, and the strict session config surface exposes no public lock-timeout knob.

## What This Changes

The issue requests a configurable session write-lock acquire timeout and threading that setting through transcript/session write-lock call sites for long legitimate lock holds.

## Best Possible Solution

Add a documented config-backed session write-lock acquire timeout, route production transcript/session lock acquisitions through a shared resolver, preserve the current lock-timeout error contract, and coordinate with the partial #69159 work.

## Reproduction Assessment

Yes. Hold a session write lock for longer than 10 seconds, then invoke a defaulted or hard-coded path such as `appendSessionTranscriptMessage`, embedded run, transcript rewrite, or Codex transcript mirroring; current main waits about 10 seconds and throws the reported lock-timeout error.

## Solution Assessment

Yes. A config knob plus shared call-site threading is the narrow maintainable fix; changing only `maxHoldMs` or one embedded-run call site would not address the acquire timeout default or the hard-coded transcript writers.
