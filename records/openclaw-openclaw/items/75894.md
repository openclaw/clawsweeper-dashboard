## Summary

Keep open. Current main still has a 10s session write-lock acquire default, multiple transcript-writing paths either rely on that default or hard-code 10s, and the strict session config surface has no lock acquire timeout knob.

## What This Changes

The issue requests a configurable session write-lock acquire timeout, with a higher default, so legitimate long prep-stage lock holds do not fail transcript writes after 10s.

## Best Possible Solution

Add a documented config-backed session write-lock acquire timeout and route production transcript lock acquisitions through one shared resolver, with focused tests for the schema, default, hard-coded call sites, and long-lock contention path.

## Reproduction Assessment

Yes. Hold one session write lock longer than 10s, then invoke a defaulted or hard-coded transcript writer such as embedded run, transcript append, transcript rewrite, or Codex transcript mirror; current main will wait 10s and throw `SessionWriteLockTimeoutError`.

## Solution Assessment

Yes. A config knob plus call-site threading is the narrow maintainable fix because changing only the helper default would leave hard-coded 10s paths and would not give operators a way to tune slow environments.
