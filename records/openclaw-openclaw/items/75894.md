## Summary

Keep open. Current main still has the reported 10s session write-lock acquire default and 10s transcript-writer literals, while the strict session config schema and public config docs still expose no write-lock acquire timeout knob.

## What This Changes

The issue requests a configurable session write-lock acquire timeout and a higher default so legitimate long prep-stage lock holds do not fail interactive messages.

## Best Possible Solution

Add a documented config-backed session write-lock acquisition timeout, likely under top-level `session`, raise the default above legitimate prep-stage holds, and route production transcript lock acquisitions through one resolver while preserving stale-lock and max-hold semantics.

## Reproduction Assessment

Yes. Hold one session write lock for longer than 10s, then run a defaulted or 10s-literal `acquireSessionWriteLock` path such as embedded run, shared transcript append, or Codex transcript mirror; current main will wait 10s and throw the reported timeout.

## Solution Assessment

Yes. A config knob plus shared call-site threading is the narrow maintainable fix; only raising the helper default would miss hard-coded callers and would not give operators a per-install escape hatch.
