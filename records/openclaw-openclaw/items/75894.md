## Summary

Keep open. Current `origin/main` still has the reported 10s acquire behavior in the lock helper and production transcript writers, and the strict session config schema still has no session write-lock acquire timeout knob.

## What This Changes

The issue asks for a configurable session write-lock acquire timeout and for transcript-writing paths to stop relying on a hard-coded or default 10s acquire timeout.

## Best Possible Solution

Add a documented config-backed session write-lock acquire timeout, route all production transcript lock acquisitions through one resolver, and cover the schema, docs, changelog, and key call sites with focused tests.

## Reproduction Assessment

Yes. Hold a session write lock longer than 10s, then invoke a defaulted or literal-10s transcript writer; existing lock tests already prove the contended acquire path rejects once the caller timeout expires.

## Solution Assessment

Yes. A config knob plus call-site threading is the narrow maintainable fix; changing only the helper default would leave hard-coded call sites and no operator control.
