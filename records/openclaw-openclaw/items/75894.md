## Summary

Keep open. Current main still has a 10s session write-lock acquire default plus hard-coded/defaulted transcript-writing call sites, and the strict session config schema/types/docs expose no knob for users to raise that acquire timeout.

## What This Changes

Request to add a configurable session write-lock acquire timeout and route transcript/session lock acquisitions through it instead of the 10s default or literals.

## Best Possible Solution

Add a documented session-level acquire-timeout setting, choose a safer default for slow prep contention, and thread the resolved value through production session transcript/store lock acquisitions while leaving unrelated locks scoped to their own policies.

## Reproduction Assessment

Yes. Hold a session write lock for longer than 10 seconds, then invoke a defaulted or hard-coded transcript path such as embedded run, transcript rewrite, transcript append, or Codex transcript mirror; current main waits 10s and throws `SessionWriteLockTimeoutError`.

## Solution Assessment

Yes. A config-backed acquire timeout with shared resolution at session-writing call sites is the narrow maintainable fix; only changing the helper default would leave users without control and risks affecting unrelated generic lock users.
