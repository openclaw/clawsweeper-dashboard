## Summary

Keep open. Current main still has a 10s session write-lock acquire default and several transcript-writing call sites either rely on that default or pass `10_000`, while the strict config schemas expose no session write-lock timeout knob.

## What This Changes

The issue requests a configurable session write-lock acquire timeout, plus a higher default, so legitimate long transcript lock holds do not fail after 10 seconds.

## Best Possible Solution

Add a documented config-backed session write-lock acquire timeout, likely under top-level `session`, and route all production transcript lock acquisitions through a shared resolver instead of helper defaults or per-call literals.

## Reproduction Assessment

Yes. Hold one session write lock for longer than 10s, then invoke another defaulted `acquireSessionWriteLock({ sessionFile })` path such as embedded run, transcript append, or Codex transcript mirror; current main will wait 10s and throw the reported timeout.

## Solution Assessment

Yes, a config knob plus call-site threading is the maintainable direction. The exact public key and default should be chosen carefully, but leaving the helper default/literals in place does not solve the reported failure mode.
