## Summary

Keep open. Current main still has the reported 10s session write-lock acquire default and multiple production transcript-writing paths either hard-code 10000 ms or inherit the helper default, while the strict config schema and user docs expose no session write-lock timeout knob.

## What This Changes

Requests a configurable session write-lock acquire timeout and a higher default for legitimate long lock holds during slow prep stages.

## Best Possible Solution

Add a documented config-backed session write-lock acquire timeout, thread it through a shared resolver for production transcript lock acquisitions, and preserve the existing max-hold watchdog semantics separately.

## Reproduction Assessment

Yes. Hold one session write lock for longer than 10s, then invoke a second transcript writer path that omits `timeoutMs` or passes `10_000`; current main will wait 10s and throw `SessionWriteLockTimeoutError`.

## Solution Assessment

Yes. A config knob plus shared call-site threading is the narrow maintainable fix; only raising the helper default would leave hard-coded 10s callers and unclear public configuration behavior.
