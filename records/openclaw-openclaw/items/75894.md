## Summary

Keep open. Current main still has the reported 10s session write-lock acquire default and hard-coded 10s transcript writer paths, while the strict session config schema exposes no session write-lock acquire timeout knob.

## What This Changes

The issue requests a configurable session write-lock acquire timeout and removal of hard-coded 10s lock acquisition paths.

## Best Possible Solution

Add a documented config-backed session write-lock acquire timeout under the session config surface, route production transcript/session lock acquisitions through one resolver, and cover the schema plus representative call sites with focused tests.

## Reproduction Assessment

Yes. Hold one session write lock longer than 10s, then invoke a defaulted or hard-coded acquisition path such as embedded run, transcript append, transcript rewrite, or the Codex transcript mirror; the current helper waits 10s and throws SessionWriteLockTimeoutError.

## Solution Assessment

Yes. A config knob plus shared call-site threading is the narrow maintainable fix; only raising one literal would leave other lock users and the public config gap unresolved.
