## Summary

Keep open. Current main still has a 10s session write-lock acquire default plus hard-coded/defaulted transcript-writing call sites, and the strict session config schema exposes no user knob to raise that acquire timeout.

## What This Changes

Add a configurable session write-lock acquire timeout and route session transcript lock acquisitions through it instead of 10s hard-coded/default values.

## Best Possible Solution

Add a documented session-level acquire-timeout setting, choose a safer default for slow prep contention, and thread the resolved value through production session transcript/store lock acquisitions while leaving unrelated locks scoped to their own policies.

## Reproduction Assessment

Yes. Hold a session write lock for longer than 10 seconds, then invoke a defaulted or hard-coded transcript path such as embedded run, transcript rewrite, transcript append, Codex transcript mirror, or tool-result truncation; current main waits 10s and throws `SessionWriteLockTimeoutError`.

## Solution Assessment

Yes. A config-backed acquire timeout with shared resolution at session-writing call sites is the narrow maintainable fix; only changing the helper default would still leave users without control and could affect unrelated generic lock users.
