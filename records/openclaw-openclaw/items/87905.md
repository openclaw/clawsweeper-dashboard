## Label Justifications

- `P1`: The report describes a successful cron job whose announce delivery fails after target corruption, breaking a real scheduled channel workflow.
- `impact:message-loss`: The final announce message is not delivered even though the cron job body succeeds.

## Summary

Keep open: current main still post-processes a plugin-resolved cron announce target with selected-provider prefix stripping, which matches the reported target corruption path.

## What This Changes

Cron announce delivery should preserve plugin-canonical delivery targets after channel/plugin target resolution instead of stripping a matching provider prefix unconditionally.

## Best Possible Solution

Preserve plugin-resolved canonical cron delivery targets, and only strip selected-provider prefixes on the generic fallback path or an explicitly route-canonicalized send target.

## Reproduction Assessment

Yes for a source-level reproduction: current main resolves the channel target, then unconditionally strips the selected provider prefix from the resolved target. I did not run a live bncr transport repro in this read-only pass.

## Solution Assessment

Yes: fixing the cron delivery boundary to preserve plugin-resolved canonical targets is narrower than adding a new config option or requiring every plugin to accept corrupted bare targets.
