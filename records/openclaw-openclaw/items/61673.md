## Summary

Keep open. Current main still only supports static `sandbox.docker.binds`, and there is no accepted `resolveBinds` field or pre-container sandbox lifecycle hook; this is a core sandbox/API design gap rather than a ClawHub-only plugin request.

## What This Changes

Add a host-side pre-container hook or resolver so Docker sandbox bind mounts can be computed per session from channel/user/context metadata.

## Best Possible Solution

Keep this as an open sandbox design item for a narrow, security-reviewed pre-container bind resolver or trusted plugin capability that runs before Docker create, reuses bind validation, and defines deterministic cache/reuse behavior.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The gap is high-confidence by source inspection: the schema accepts only static Docker bind config, resolution only merges arrays, and the create path appends only `cfg.binds`.

## Solution Assessment

Unclear. The requested capability is real, but the proposed raw executable hook may not be the safest final API compared with a documented trusted lifecycle seam or plugin capability.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it would add host-side authority over which filesystem paths enter sandbox containers.

Concerns:

- **[medium] Design the bind resolver as a trusted boundary:**
  - body: A pre-container resolver can decide which host paths become visible inside a sandbox. Any implementation needs explicit trust rules, timeout/error handling, validation through existing bind checks, and container reuse semantics that prevent cross-user mount leakage.
