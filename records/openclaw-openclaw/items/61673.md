- `impact:security`: The requested behavior controls which host filesystem paths become visible inside sandbox containers.

## Summary

Keep open. Current main and v2026.5.18 still model Docker sandbox binds as static config, with no pre-create session-aware resolver or existing plugin hook that can safely alter the mount set before container creation.

## What This Changes

Request a host-side, pre-container-creation resolver for Docker sandbox bind mounts using session context such as channel, user, and agent metadata.

## Best Possible Solution

Design a security-reviewed pre-create sandbox lifecycle contract that resolves per-session mounts before Docker create, validates the final mount set through existing bind policy, and incorporates the resolved set into sandbox reuse identity.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that the requested capability is absent: schema/types expose static binds, bind resolution has no session metadata, and Docker create consumes cfg.binds before setupCommand.

## Solution Assessment

Unclear. The user problem is valid, but the proposed resolveBinds executable should be weighed against a trusted plugin or lifecycle API with explicit trust, validation, timeout, audit, and reuse semantics.
