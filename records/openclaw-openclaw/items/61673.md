## Summary

Keep open. Current main still treats Docker sandbox bind mounts as static config merged from global and per-agent settings, and there is no pre-container resolver hook or accepted `resolveBinds` field. This is not a ClawHub-only close because the missing part is a core sandbox lifecycle seam before `docker create`, not an external plugin that can be built with today's hooks.

## What This Changes

Request to add a host-side pre-container sandbox bind resolver so Docker bind mounts can vary by session, channel, user, or metadata while retaining static binds.

## Best Possible Solution

Keep this as an open sandbox design item. The maintainable end state is a narrow, security-reviewed pre-container bind resolver or trusted plugin capability that runs before Docker create, receives a documented minimal session context, reuses existing bind validation, has deterministic timeout/error behavior, and defines how dynamic output interacts with sandbox scope and container reuse.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The current gap is high-confidence by source inspection: the config schema rejects unknown Docker keys, the resolver only merges static `binds`, and the create path appends only that resolved array.

## Solution Assessment

Unclear. The requested capability is real, but a raw executable hook may not be the best final API without maintainer security design around trust, auditability, cache identity, and plugin boundaries.
