## Summary

Keep open. Current main still lacks a dynamic pre-container bind resolver; the sandbox schema, resolver, and Docker create path only handle static bind arrays, so this remains a core sandbox/API and security design item.

## What This Changes

Request a host-side pre-container hook or resolver that computes sandbox Docker bind mounts per session from channel, user, or metadata context.

## Best Possible Solution

Keep this as an open sandbox design item for a trusted pre-create bind resolver or plugin capability that runs before Docker create, validates dynamic output with the existing bind checks, and defines deterministic container reuse semantics.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The feature gap is high-confidence by source inspection: the schema lacks a resolver field, config resolution only merges static arrays, and container creation only consumes `cfg.binds`.

## Solution Assessment

Unclear. The requested capability is real, but a raw host executable may not be the narrowest maintainable interface compared with a documented trusted lifecycle seam or plugin capability.

## Risks / Open Questions

- The raw executable hook shape may not be the safest API compared with a trusted plugin lifecycle capability or declarative resolver contract.
- Dynamic per-session mounts must define how `scope: shared`, `scope: agent`, container reuse, and config hashes avoid cross-user mount leakage.
- The resolver would introduce host-side authority over filesystem exposure, so timeout, trust, validation, error reporting, and audit semantics need explicit design.
