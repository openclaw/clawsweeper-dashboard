## Label Justifications

- `impact:security`: The request controls which host filesystem paths become visible inside sandbox containers.

## Summary

Keep open. Current main and v2026.5.18 still expose static Docker bind arrays with no pre-create, session-aware bind resolver, and the related volume-strategy PR plus adjacent bind-policy issues do not supersede this security-sensitive lifecycle/API request.

## What This Changes

The issue requests a host-side pre-container hook or callback that resolves Docker sandbox bind mounts per session from channel, user, or metadata context.

## Best Possible Solution

Design a security-reviewed pre-create sandbox lifecycle contract that resolves per-session mounts before Docker create, validates the final mount set through existing bind policy, and hashes the resolved set into container reuse identity.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that the requested capability is absent: schema/types expose static binds, resolution has no session metadata, and Docker create consumes cfg.binds before setupCommand.

## Solution Assessment

Unclear. The user problem is valid, but the proposed resolveBinds executable should be weighed against a trusted plugin or lifecycle API with explicit trust, validation, timeout, audit, and reuse semantics.

## Risks / Open Questions

- A raw host executable resolver may be the wrong final API compared with a trusted plugin or versioned sandbox lifecycle contract.
- Resolved dynamic binds must be included in container identity and reuse semantics to prevent one session from reusing another session's mount set.
- The resolver output would expand host-path mount authority, so timeout, validation, audit logging, and user-facing failure-mode rules need security review.
