## Summary

Keep open: current main and v2026.5.28 still have the LM Studio keyless local marker path, and the config audit path still treats any non-empty `models.providers.*.apiKey` string as plaintext. The exact current marker is `lmstudio-local`, not `lm-studio`, but the reported false-positive warning remains source-reproducible.

## What This Changes

Stop keyless LM Studio setup from producing a plaintext security-audit warning for its non-secret local provider marker.

## Best Possible Solution

Make the secrets audit and LM Studio keyless setup agree on registered non-secret markers so keyless LM Studio reports no plaintext count while real provider credentials still warn.

## Reproduction Assessment

Yes from source, but not by executing the command in this read-only review. Current setup persists `lmstudio-local` for keyless LM Studio, and the audit path marks any non-empty `models.providers.*.apiKey` string as plaintext.

## Solution Assessment

Yes, a narrow marker-aware audit fix is the best first repair because it matches the existing plugin manifest contract and generated `models.json` behavior. Omitting the marker from setup is also possible only if runtime synthetic auth remains covered by focused tests.
