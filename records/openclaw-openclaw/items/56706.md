## Summary

Keep this PR open and do not merge it as-is. Current main and the latest release still document and test inline, same-line block, and bare TTS directive behavior; this PR changes that public contract and also bundles unrelated Slack/config-validation work that needs to be split or dropped.

## What This Changes

This PR replaces regex-based TTS directive parsing with standalone-line scanning and updates TTS contract tests, while also changing Slack Bolt `ignoreSelf` handling and plugin validation known-ID handling.

## Best Possible Solution

Rework this into focused changes: decide the intended TTS directive contract first, then update the parser, unit and contract tests, runtime prompt hints, public TTS docs, and changelog together; split any Slack or config-validation fixes into separate PRs with their own validation.

## Reproduction Assessment

Yes. Current source, docs, and contract tests provide a high-confidence reproduction path for the semantic mismatch: inline provider directives and same-line `[[tts:text]]...[[/tts:text]]` blocks are expected today, while this PR makes those forms non-triggering unless rewritten as standalone lines.

## Solution Assessment

No. Standalone-only parsing may be acceptable after a maintainer product/API decision, but this PR is not the best path because it changes a public contract without the matching docs/changelog migration and bundles unrelated Slack/config fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve documented inline TTS directives:** `src/tts/directives.ts:70-78`
  - body: The new scanner only recognizes `[[tts:text]]` and `[[tts:...]]` when they occupy a full trimmed line, but current docs and contract tests still support inline provider directives and same-line text blocks. In tagged mode, `[[tts:text]]Hello world[[/tts:text]]` would stop triggering synthesis, and inline provider overrides would be left as visible text. Keep the existing forms or update the public contract, docs, tests, and changelog in the same focused change.
  - confidence: 0.94
- **[P2] Guard self messages when disabling Bolt ignoreSelf:** `extensions/slack/src/monitor/provider.ts:281`
  - body: Setting `ignoreSelf: false` lets Bolt deliver bot-authored events, but this patch does not add an equivalent self-event guard in the same Slack code path. Without the current-main `bot_id` filter, `bot_message` events can pass when `allowBots` is enabled and cause self-processing loops. Split this change and carry the explicit self filter with it.
  - confidence: 0.82
