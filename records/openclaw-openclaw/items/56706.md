## Summary

Keep open, but do not merge as-is: current main and the latest release still document and test permissive inline TTS directives, while this PR changes that public contract without the matching docs/changelog migration and bundles unrelated Slack/config-validation work. The maintainer discussion already asks for this to be split and reworked before it can land.

## What This Changes

This PR replaces regex-based TTS directive parsing with a standalone-line scanner and updates TTS contract tests, while also changing Slack Bolt ignoreSelf handling and plugin-validation known-ID handling.

## Best Possible Solution

Decide the intended TTS directive contract first, then land a focused parser change with matching unit and contract tests, runtime prompt hints, public TTS docs, and changelog coverage; handle Slack ignoreSelf and bundled-provider validation in separate focused PRs.

## Reproduction Assessment

Yes. Source, docs, and contract-test inspection give a high-confidence reproduction path: current main accepts inline provider directives and same-line [[tts:text]] blocks, while this PR's scanner would only activate standalone directive lines.

## Solution Assessment

No. Standalone-only parsing may be a valid product/API decision, but this PR is not the best way to solve it because it changes a public contract without the full migration surface and bundles unrelated behavior changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve documented inline TTS directives:** `src/tts/directives.ts:70-78`
  - body: The new scanner only recognizes [[tts:text]] and [[tts:...]] when they occupy a full trimmed line, but current docs and contract tests still support inline provider directives and same-line text blocks. In tagged mode, [[tts:text]]Hello world[[/tts:text]] would stop triggering synthesis, and inline provider overrides would be left as visible text. Keep the existing forms or update the public contract, docs, tests, and changelog in the same focused change.
  - confidence: 0.94
- **[P2] Guard self messages when disabling Bolt ignoreSelf:** `extensions/slack/src/monitor/provider.ts:281`
  - body: Setting ignoreSelf: false lets Bolt deliver bot-authored events, but this patch does not add an equivalent self-event guard in the same Slack path. Without a guard matching OpenClaw's bot identity, bot_message events can pass when allowBots is enabled and cause self-processing loops. Split this change and carry the explicit self filter with it.
  - confidence: 0.82
