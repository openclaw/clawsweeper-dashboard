## Summary

Keep open, but do not merge as-is. Current main still intentionally supports permissive inline/same-line TTS directives outside markdown code, and the prior maintainer discussion rejected the broader ordinary-prose literal-tag expectation as not planned; this branch changes that public contract, lacks real behavior proof, and mixes in stale unrelated Slack/config hunks.

## What This Changes

The PR replaces TTS directive parsing with a standalone-line scanner and tests, while also adding Slack Bolt `ignoreSelf: false` and bundled provider plugin known-ID validation changes.

## Best Possible Solution

Decide the TTS directive contract first; if standalone-only is desired, land a focused parser/docs/tests/changelog migration, and keep Slack plus config-validation fixes in separate rebased PRs.

## Reproduction Assessment

Yes. Source inspection shows current main parses inline provider directives and same-line text blocks, while the PR scanner only activates full-line directives; I did not run tests because this was a read-only review.

## Solution Assessment

No. Standalone-only parsing may be valid after a maintainer product decision, but this branch is not the best vehicle because it changes the public contract without the migration surface and bundles unrelated stale changes.
