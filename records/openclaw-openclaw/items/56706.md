## Summary

Keep open, but do not merge as-is: current main still treats inline and compact TTS directives as supported behavior, and the existing review discussion says this needs an API/product decision plus branch splitting rather than cleanup closure.

## What This Changes

This PR replaces regex-based TTS directive parsing with a standalone-line scanner, updates TTS contract tests, and also includes Slack Bolt `ignoreSelf` plus config-validation known-ID changes.

## Best Possible Solution

Decide the directive contract first; if standalone-only is wanted, land a focused TTS parser, docs, tests, and changelog migration, with Slack and config validation handled by separate current-main PRs.

## Reproduction Assessment

Yes for the behavior delta: current main accepts inline provider directives and same-line text blocks, while the PR scanner only activates full-line markers. No high-confidence proof shows current main is broken under the current documented contract.

## Solution Assessment

No. Standalone-only parsing may be valid after a product/API decision, but this branch is not the right vehicle because it changes public semantics without migration docs and bundles unrelated stale changes.

  - body: The new line scanner only recognizes `[[tts:...]]` and `[[tts:text]]` markers when they occupy a full trimmed line. Current docs and contract tests support inline provider directives, same-line hidden text blocks, and bare tags, so tagged-mode replies using those forms would stop applying overrides or triggering TTS unless the contract, docs, tests, and changelog are intentionally migrated.
  - confidence: 0.94

  - body: This hunk disables Bolt’s built-in self filtering in the branch’s older direct App construction without adding equivalent OpenClaw self-event middleware. Bot-authored messages can pass when `allowBots` is enabled; current main solves this through `createSlackBoltApp`, so the Slack change should be split or rebased rather than merged as-is.
  - confidence: 0.83

  - body: The added import points at `../plugins/bundled-capability-metadata.js`, but current main keeps bundled capability metadata under contract/test inventory and has a boundary test preventing runtime imports from it. Rebase the known-ID fix onto the current plugin metadata boundary before merging.
