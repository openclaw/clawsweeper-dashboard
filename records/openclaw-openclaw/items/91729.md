## Summary

Keep open: the MS Teams long-reply duplicate fix is still needed on current main, but this PR is not merge-ready because the new trim state can stay latched and truncate later text segments after a stream failure, and branch-specific real behavior proof is still insufficient.

## What This Changes

The PR trims already-streamed MS Teams reply text before block-delivery fallback so long replies do not repost the streamed prefix.

## Best Possible Solution

Make suffix trimming a one-shot fallback for the failed streamed segment, keep later segments on normal block delivery, and add redacted live Teams proof before merge.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced here: current main emits a prefix, latches `streamFailed` on a non-cancel emit error, then returns the full final payload for block delivery. The linked issue also gives a concrete Teams DM reproduction path for OpenClaw 2026.6.1.

## Solution Assessment

No as written. Trimming at the MS Teams reply stream controller is the right boundary, but the PR should make the trim one-shot for the failed segment so later block-delivered text is not sliced by stale prefix length.
