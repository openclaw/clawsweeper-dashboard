## Summary

Keep this PR open, but it is not merge-ready: the draft addresses plausible MS Teams duplicate-delivery paths, yet the failed-stream trim can corrupt later reply segments and the author’s later production check no longer proves the claimed >4000-character fix.

## What This Changes

The PR changes the MS Teams reply stream controller and tests to trim already-streamed text on failed fallbacks and suppress the SDK final create for streamed replies at or above 4000 characters.

## Best Possible Solution

Land a Teams-streaming fix that scopes fallback trimming to the current failed segment, preserves or explicitly decides final-message metadata behavior, and includes live Teams proof for a >4000-character DM reply.

## Reproduction Assessment

Yes for the PR-introduced bug: source inspection shows streamFailed remains latched while reply-dispatcher sends later payloads through preparePayload, so later segments can be trimmed by stale emittedTextLength. No for the claimed live >4000-character fix: the latest discussion reports a negative production check and lacks branch-specific Teams proof.

## Solution Assessment

No. The current patch is a plausible hardening path, but the best fix needs a per-payload stream-failure boundary or state reset plus a deliberate answer for the skipped stream-final metadata path.
