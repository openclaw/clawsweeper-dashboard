## Summary

Keep this draft PR open, but do not merge it as-is. The patch is a plausible hardening of the MS Teams failed-stream fallback path, but the author’s later live result says the observed >4000-character duplicate does not go through that path, and the diff can truncate later reply segments after a failed stream because the stale emitted length remains applied to every later payload.

## What This Changes

The PR changes the MS Teams reply stream controller so failed-stream and finalize fallbacks trim already-streamed text, with regression tests for partial stream failures and close fallback behavior.

## Best Possible Solution

Rework the Teams fix around the actual stream.close/collapse failure path, and keep any failed-stream suffix trim scoped to one failed streamed segment with a regression test for later post-tool payloads.

## Reproduction Assessment

Yes for the PR-introduced truncation risk from source: after a non-cancel stream failure, the controller keeps streamFailed and emittedTextLength while the dispatcher continues to call preparePayload for later segments. No for the claimed live >4000-character fix; the branch lacks positive Teams proof and the author reported a negative equivalent-patch result.

## Solution Assessment

No. Trimming the failed-stream fallback may be a useful hardening step, but this implementation is not the best fix because it is not scoped to one failed segment and the observed duplicate now appears to be in the SDK close/collapse path rather than streamFailed fallback.
