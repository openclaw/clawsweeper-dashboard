## Summary

Keep open: the transient native stdio MCP process overlap is plausible from the current session-scoped runtime design, but pooling by agent would change the documented isolation and lifecycle contract rather than fix an already-defined bug.

## What This Changes

Request agent-scoped pooling for native-binary stdio MCP runtimes so repeated agent runs do not spawn overlapping idle server processes before idle reap.

## Best Possible Solution

Define the intended MCP runtime scope first; if agent-scoped native stdio reuse is desired, implement it centrally in the runtime manager with workspace/config/isolation tests and leave the Docker and Hermes-specific leaks to their existing reports.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show current main caches by session ID and reaps by idle TTL, which makes overlapping runtimes possible when separate session IDs are active faster than the TTL.

## Solution Assessment

Unclear: pooling by `(agentId, serverName)` may reduce spawn overhead, but it is not obviously the safest fix because the current contract is session-scoped and isolation-sensitive.
