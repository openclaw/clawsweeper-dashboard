## Summary

## What This Changes

The issue asks OpenClaw to avoid OpenAI Responses 400s when failover or replay sends an assistant `message` item without its required `reasoning` item.

## Best Possible Solution

Keep the current main replay behavior: capture OpenAI reasoning items at the output-item boundary, replay paired reasoning before messages, and strip foreign replay IDs across provider/model boundaries.

## Reproduction Assessment

No current-main failure path was reproduced. The historical failure is well explained by the reporter logs and follow-up source diagnosis, and current main now contains the source and regression coverage for that payload shape.

## Solution Assessment

Yes. Capturing the OpenAI reasoning item when the stream item is added/done and avoiding stale cross-model message IDs is narrower than changing failover policy or adding a user-facing configuration.
