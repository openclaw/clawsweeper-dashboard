## Summary

Keep open: the report is new, concrete, and not superseded, and current main has partial OpenAI Responses replay protections but no proof that the Gemini-to-OpenAI `message`/`reasoning` pairing failure is fixed.

## What This Changes

The issue reports that failover from Gemini reasoning history into OpenAI Responses reasoning models sends an assistant `message` item without its required `reasoning` item.

## Best Possible Solution

Add a focused OpenAI Responses replay sanitizer and regression test so a `message` item id is not replayed after its required `reasoning` item has been removed during provider/model route switching.

## Reproduction Assessment

No, not as a live current-main reproduction. Source inspection gives a plausible payload-construction path and a clear test gap, but the exact Gemini-to-OpenAI failover should still be reproduced or captured with a focused payload test.

## Solution Assessment

Yes, keeping this open is the safest current outcome. The narrowest maintainable fix is in OpenAI Responses replay sanitation, preferably stripping or downgrading message item ids when their required reasoning item cannot be replayed rather than changing failover policy or adding config.
