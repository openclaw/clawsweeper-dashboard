## Summary

Keep open: current main still appears to preserve a replay path where OpenAI Responses assistant `message` item ids can be sent without the required `reasoning` item, and the later same-provider 2026.5.7 report makes this a current shipped bug rather than stale cross-provider-only fallout.

## What This Changes

The issue requests that OpenAI Responses replay preserve or sanitize reasoning/message item pairs when Gemini-originated or OpenAI-originated conversation history is retried through an OpenAI reasoning model.

## Best Possible Solution

Fix the OpenAI Responses stream/replay sanitizer so any replayed assistant message id is accompanied by its required reasoning item, or the message id is stripped when the reasoning item cannot be safely replayed, with focused regression coverage for both same-provider and cross-provider histories.

## Reproduction Assessment

Yes for a source-level reproduction path: current review evidence and the 2026.5.7 comment identify how a persisted `msg_*` text item can be replayed without a captured `rs_*` reasoning item. I did not establish a live provider reproduction in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable solution is to repair OpenAI Responses replay/stream sanitation and add regression tests, not to change failover policy or add a configuration workaround.
