## Summary

Keep open: current main captures OpenAI reasoning items when the stream emits them, but the reported failover shape still has a plausible source-level gap when route-switch cleanup drops replayable `rs_*` reasoning while a same-model OpenAI assistant text item can still replay its `msg_*` id.

## What This Changes

The issue reports OpenAI Responses replay failures where an assistant `message` item is sent without its required `reasoning` item after Gemini-to-OpenAI failover or related OpenAI reasoning replay paths.

## Best Possible Solution

Add a narrow OpenAI Responses replay sanitizer that omits assistant `message.id`/`textSignature` when the matching replayable reasoning item has been removed or is unavailable, with regression coverage for model-snapshot failover and same-provider no-summary reasoning streams.

## Reproduction Assessment

No, not as a live end-to-end reproduction in this review. Source inspection gives a credible payload-shape reproduction path: model-snapshot failover can drop replayable OpenAI reasoning while preserving a same-model assistant `textSignature` that later becomes `message.id`.

## Solution Assessment

Yes, the best fix is a focused OpenAI Responses replay sanitation change plus fixture coverage; failover policy changes or new configuration would be broader than the observed item-pairing contract problem.
