## Summary

Keep open: current main appears to have fixed the later same-provider empty-summary stream case, but the original route-switch replay sanitizer can still remove an OpenAI `rs_*` reasoning item while leaving a server-bound `msg_*` assistant message id replayable. No merged related item or closing PR proves the Gemini-to-OpenAI direction is fully fixed.

## What This Changes

The issue requests that OpenAI Responses replay preserve or safely remove required reasoning/message item pairings when a Gemini-originated conversation fails over to an OpenAI reasoning model.

## Best Possible Solution

Add focused OpenAI Responses replay sanitation that keeps required reasoning/message pairs together or rewrites/omits server-bound message ids when their `rs_*` parent is removed during a route switch.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes: construct current-main history with an OpenAI `thinkingSignature`/`textSignature` pair and a changed model snapshot, then the sanitizer removes the reasoning block while the transport can still replay the `msg_*` id. A live Gemini-to-OpenAI ladder was not run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix belongs in OpenAI Responses replay sanitation/payload construction, not in failover policy or a new config option.
