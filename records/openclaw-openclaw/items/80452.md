## Summary

Keep open: current main still appears to carry a source-level OpenAI Responses replay gap for provider/model switches, where OpenAI reasoning items can be dropped while their paired assistant message ids remain replayable. The existing related fixes cover the inverse Gemini path, Copilot reasoning-id preservation, and native Codex replay isolation, but I did not find a merged fix for Gemini-to-OpenAI message/reasoning pairing.

## What This Changes

The issue asks OpenClaw to handle Gemini-to-OpenAI failover without sending OpenAI Responses assistant message items whose required reasoning items were removed or unavailable.

## Best Possible Solution

Add OpenAI Responses replay sanitation that removes or repairs assistant message item ids whenever the required paired reasoning item will not be sent, then cover the provider-switch fixture that preserves a textSignature beside dropped reasoning.
