## Summary

Keep open: current main appears to have fixed the same-provider empty-summary capture path, but the original replay-sanitization gap is still source-reproducible when route-switch cleanup drops `rs_*` reasoning while a target-model OpenAI `msg_*` text signature remains replayable. The related model-switch discussion at https://github.com/openclaw/openclaw/issues/78545 and PR review at https://github.com/openclaw/openclaw/pull/78547 show the same dependent-item-id concern, so this should stay open for a focused fix rather than close as implemented.

## What This Changes

The issue reports OpenAI Responses failover requests replaying assistant `message` item IDs without their required paired `reasoning` items after Gemini-to-OpenAI routing.

## Best Possible Solution

Add a coupled OpenAI Responses replay sanitizer so dropping or suppressing replayable `rs_*` reasoning also clears dependent `msg_*`/Responses item IDs while preserving visible assistant text, with focused regression coverage for route switches and same-provider empty-summary replay.

A focused fixture should capture the exact sanitized-history to payload shape before a live provider proof.

## Solution Assessment

Yes, the best direction is a narrow Responses replay sanitation fix rather than failover-policy changes or new configuration. The fix should couple reasoning removal with dependent Responses item-ID removal and preserve current same-model replay behavior.
