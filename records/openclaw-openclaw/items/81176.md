## Summary

Keep open. The PR is a plausible feature direction, but it is not merge-ready because the new share config is accepted before all compaction/preflight consumers resolve the same budget, and the public config docs/help surfaces are still absolute-token only.

## What This Changes

The PR adds optional `reserveTokensShare`, `keepRecentTokensShare`, and `reserveTokensFloorShare` config fields, resolves them in Pi settings against `contextTokenBudget`, and adds focused Pi settings tests.

## Best Possible Solution

Land relative compaction budgets only after one backward-compatible API is settled, all compaction and preflight budget consumers use the same resolver, and public docs/help/baselines describe the fields and precedence.

## Reproduction Assessment

Not applicable as a bug reproduction for the original feature request. For the patch defect, source inspection gives a high-confidence path: a share-only floor config is accepted by the new schema but memory preflight and memory flush still read only absolute `reserveTokensFloor`.

## Solution Assessment

No. Additive share fields are a reasonable candidate API, but the implementation needs consistent budget resolution across threshold consumers plus docs/help/baseline coverage before merge.
