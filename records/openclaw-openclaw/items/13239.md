## Summary

This issue remains necessary. Current main now has useful plugin compatibility/deprecation infrastructure, but it still does not publish provider/model/capability lifecycle states, define public provider support guarantees, generate the requested matrix documentation, or expose provider lifecycle status through doctor/models output. The linked Matrix report is useful motivation but was a closed plugin-specific regression, not a replacement for this broader policy proposal.

## What This Changes

Publish a machine-readable provider/model capability compatibility matrix, define deprecation guarantees, and expose it in documentation and CLI diagnostics.

## Best Possible Solution

Define the provider support contract and ownership first, then reuse the existing compatibility-registry pattern for one provider/model/capability lifecycle source that generates public documentation and concise CLI notices while keeping live probe health separate.

## Reproduction Assessment

Not applicable; this proposes a new provider support-policy and discovery surface rather than reporting broken existing behavior.

## Solution Assessment

Yes in direction, but not yet implementation-ready: a single generated registry is the maintainable shape, while lifecycle meanings, provider ownership, and deprecation guarantees require maintainer approval first.
