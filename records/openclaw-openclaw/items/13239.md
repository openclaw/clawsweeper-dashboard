## Summary

Keep open: current main still lacks the requested provider/model-family/capability lifecycle matrix and deprecation policy. Existing docs and CLI code cover provider setup, model metadata, live auth/probe health, and plugin compatibility, but not the broader provider support-lifecycle contract this proposal asks maintainers to define.

## What This Changes

The issue proposes a machine-readable provider compatibility matrix with stable/experimental/deprecated/broken states, deprecation windows, and docs plus doctor/models CLI surfacing.

## Best Possible Solution

Approve one provider lifecycle source of truth keyed by provider, model family, capability, status, deadline, replacement, and canonical issue link, then generate public docs and concise doctor/models output from it while keeping live probe health separate.

## Solution Assessment

Unclear until maintainers choose the support guarantees. The maintainable direction is a single lifecycle registry feeding docs and CLI output without mixing policy state into live auth/probe health.
