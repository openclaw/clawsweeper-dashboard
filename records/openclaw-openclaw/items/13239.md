## Summary

Keep this issue open for an explicit maintainer product decision. Current main publishes provider catalogs and reports authentication/runtime health, but it still does not define provider/model lifecycle guarantees, deprecation windows, replacements, or canonical broken-state tracking. The Matrix plugin regression at https://github.com/openclaw/openclaw/issues/52899 is useful motivation but is closed and does not supersede this broader policy proposal.

Likely related people:
- steipete — high confidence; prior policy reviewer and primary contributor around the relevant provider docs and models CLI surfaces.
- Vincent Koc — low confidence; recent adjacent contributor visible in the checkout’s boundary-collapsed history.

## What This Changes

Publish a machine-readable provider/model capability matrix, define stable, experimental, deprecated, and broken lifecycle states, and expose them in generated docs and CLI diagnostics.

## Best Possible Solution

Approve a small provider-owned lifecycle schema and maintenance policy, then generate the public compatibility matrix and concise CLI lifecycle warnings from that metadata while keeping live authentication and probe health separate.

## Reproduction Assessment

Not applicable; this issue proposes a new compatibility-policy and metadata surface rather than reporting broken behavior under an existing contract.

## Solution Assessment

The generated-matrix outcome is reasonable, but a hand-maintained central registry is not the best ownership shape; generic output should consume provider-owned lifecycle metadata after maintainers define enforceable guarantees.

## Risks / Open Questions

- Publishing lifecycle labels and minimum notice windows would create ongoing support guarantees across many provider plugins without a currently defined owner, verification standard, or enforcement process.
- A centralized hand-maintained matrix could drift from provider-owned catalogs and live runtime behavior unless it is generated from the same canonical metadata.
- Lifecycle support state must remain distinct from transient authentication, billing, rate-limit, and availability probe results to avoid misleading users.
