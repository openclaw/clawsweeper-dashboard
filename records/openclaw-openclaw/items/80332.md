## Summary

Keep this open: current main still has source-provable paths that can place a truthy Ollama `think` at the top level, but the requested blanket move to `options.think` conflicts with current OpenClaw docs and upstream Ollama source, so the remaining fix needs maintainer contract judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop sending Ollama `think` as a top-level native `/api/chat` field and route it through `options` instead.

## Best Possible Solution

Keep native Ollama aligned with upstream top-level `think`, then have a maintainer decide whether explicit non-reasoning model params should be rejected, clamped to `false`, or left as an operator-forced configuration error.

## Reproduction Assessment

Yes from source for one path: explicit `params.think` or `params.thinking` can still become top-level `requestParams.think` on a native Ollama request. The issue's `thinkingDefault` route is not high-confidence from source alone because current validation can remap unsupported non-explicit defaults to `off`.

## Solution Assessment

No as proposed: moving all Ollama thinking control into `options.think` conflicts with the current OpenClaw docs and upstream Ollama `ChatRequest.Think` contract. A capability-aware guard or config-validation policy would be safer if maintainers decide this should change.
