## Summary

Keep open: current main still has a source-provable Ollama path that can send a truthy top-level `think` value, but the requested blanket move to `options.think` conflicts with current OpenClaw docs/tests and upstream Ollama's typed API contract. The linked fix PR was useful context, but it was closed unmerged after review flagged that contract issue.

## What This Changes

The issue asks OpenClaw to stop sending Ollama `think` as a top-level `/api/chat` field for non-thinking models and route it through `options` instead.

## Best Possible Solution

Preserve top-level Ollama `think` for reasoning-capable native models, and add capability-aware validation or clamping so non-reasoning models cannot receive a truthy top-level `think` unless upstream live proof supports a different contract.

## Reproduction Assessment

Yes, at source level for explicit/configured params: current main can build a native Ollama request with truthy top-level `think`, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact default-thinking route still needs live/config confirmation because current validation can remap unsupported non-explicit defaults to `off`.

## Solution Assessment

No, not as written. Moving all native Ollama thinking control into `options.think` is not the narrowest maintainable fix because upstream Ollama reads thinking from top-level `ChatRequest.Think`; a capability-aware guard is safer.
