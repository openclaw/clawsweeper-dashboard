## Summary

Keep open: current main and the latest release still send native Ollama thinking control as top-level `think`, and the linked replacement PR was closed unmerged. The reported blanket move to `options.think` conflicts with current OpenClaw docs/tests and upstream Ollama’s chat request contract, but there is still a source-provable path where truthy explicit Ollama thinking params can reach a non-thinking model and trigger Ollama’s 400, so this needs provider-owner direction rather than cleanup closure.

## What This Changes

The issue reports that OpenClaw sends Ollama `think` at the `/api/chat` top level, causing 400 errors for non-thinking models, and asks to move `think` into `options`.

## Best Possible Solution

Preserve top-level Ollama `think` for reasoning-capable native models, then add provider-owned capability-aware guarding, diagnostics, or rejection for unsupported truthy thinking settings if maintainers decide explicit params should not be allowed to reach non-thinking models.

## Reproduction Assessment

No for the exact report as written, because I did not live-run Ollama and current source shows some unsupported `thinkingDefault` paths are remapped before execution. Yes for a narrowed source path: truthy explicit `params.think`/`params.thinking` still reaches top-level Ollama `think`, and upstream Ollama rejects truthy `req.Think` for non-thinking models.

## Solution Assessment

No, the requested blanket move to `options.think` is not the best fix because it conflicts with current OpenClaw docs/tests and upstream Ollama’s `ChatRequest.Think` contract. A capability-aware guard or diagnostic inside the Ollama plugin is the safer maintainable direction.
