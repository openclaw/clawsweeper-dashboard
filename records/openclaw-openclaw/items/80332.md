## Summary

Keep open: current main and the latest release still send Ollama native thinking as top-level `think`, and the linked fix PR was closed unmerged. The reported blanket move to `options.think` conflicts with current OpenClaw docs/tests and upstream Ollama’s chat request contract, so the remaining decision is how to guard or diagnose truthy thinking for non-thinking models.

## What This Changes

The issue asks to move Ollama `think` from top-level native `/api/chat` requests into `options` for all models, or otherwise avoid truthy top-level thinking for non-thinking models.

## Best Possible Solution

Preserve upstream-compatible top-level `think` for reasoning-capable Ollama models, then add an owner-approved guard, warning, or rejection path for truthy thinking on models known not to support thinking.

## Reproduction Assessment

Yes for a narrowed source path: current main can put truthy configured `params.think`/`params.thinking` on the top-level Ollama chat payload, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact `thinkingDefault: "low"` gateway route is not high-confidence because current supported-level validation can downgrade unsupported defaults to `off`.

## Solution Assessment

No, not as suggested: moving all native Ollama thinking control into `options.think` conflicts with current OpenClaw docs/tests and upstream Ollama source. A capability-aware guard or diagnostic is the safer maintainable direction.
