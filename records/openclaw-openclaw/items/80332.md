## Summary

Keep this open: current `main` still has Ollama paths that can put a truthy `think` on the top-level native `/api/chat` payload, but the suggested blanket move to `options.think` conflicts with Ollama's current documented/source contract and needs maintainer confirmation before implementation.

## What This Changes

The issue asks OpenClaw to stop sending Ollama `think` as a top-level `/api/chat` field for non-thinking models and instead place it in `options`.

## Best Possible Solution

Keep Ollama `think` aligned with the upstream top-level API for reasoning-capable models, and harden OpenClaw so non-reasoning Ollama models cannot receive a truthy top-level `think` from defaults, overrides, or explicit params unless live upstream proof supports another contract.

## Reproduction Assessment

Yes, partially from source: current `main` can build a native Ollama request with top-level truthy `think` from explicit model params, and upstream Ollama source rejects truthy `Think` for non-thinking models. The exact `thinkingDefault` route still needs live confirmation because OpenClaw also has validation that remaps unsupported non-explicit defaults to `off`.

## Solution Assessment

No, not as written: moving all Ollama thinking control into `options` is not the safest fix because Ollama's current API contract models `think` as top-level. A narrower fix should guard or clamp truthy `think` for non-reasoning models while preserving top-level `think` for models with the `thinking` capability.
