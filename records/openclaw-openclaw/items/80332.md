## Summary

Keep this open: current main still has native Ollama request paths that can emit a truthy top-level `think`, but the requested blanket move to `options.think` conflicts with current OpenClaw docs and upstream Ollama source, so the remaining fix needs maintainer direction.

## What This Changes

The issue asks OpenClaw to stop sending Ollama `think` as a top-level `/api/chat` field for non-thinking models and instead route it through `options` or otherwise guard it by model capability.

## Best Possible Solution

Keep native Ollama `think` aligned with the upstream top-level API for reasoning-capable models, and add a narrow capability-aware guard so non-reasoning models cannot receive a truthy top-level `think` from defaults, overrides, or explicit model params unless upstream live proof supports another contract.

## Reproduction Assessment

Yes, but only at source level: current main can emit top-level `think: "low"` from explicit Ollama model params, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact `thinkingDefault` path still needs live/config confirmation because OpenClaw also validates and remaps unsupported defaults.

## Solution Assessment

No, the requested blanket `options.think` move is not the best proven fix because current Ollama source and OpenClaw docs define `think` as a top-level native chat request field. A safer solution is to preserve top-level `think` for capable models and prevent truthy values from reaching non-thinking models.
