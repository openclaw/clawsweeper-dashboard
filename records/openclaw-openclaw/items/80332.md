## Summary

Keep this open: current `main` still has native Ollama request paths that can put a truthy `think` on the top-level `/api/chat` payload, while the proposed blanket move to `options.think` conflicts with the upstream Ollama source/docs contract. The related fix PR is closed unmerged, so there is no merged replacement that resolves the issue.

## What This Changes

The issue requests routing Ollama `think` through the request `options` object instead of the top-level `/api/chat` field for non-thinking models.

## Best Possible Solution

Preserve top-level Ollama `think` for reasoning-capable native models, and harden OpenClaw so non-reasoning models cannot receive a truthy top-level `think` from defaults, session overrides, or explicit params unless live upstream proof supports a different contract.

## Reproduction Assessment

Yes for the core failure path from source: current `main` can still emit truthy top-level native Ollama `think`, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact `thinkingDefault` route still needs live/config confirmation because current OpenClaw also has supported-level validation.

## Solution Assessment

No, not as written. Moving all Ollama thinking control into `options.think` is not the narrowest maintainable fix because upstream Ollama models `think` as a top-level chat request field; a capability-aware guard or clamp is safer.
