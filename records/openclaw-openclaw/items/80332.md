## Summary

Keep open: current main and the latest release still send native Ollama thinking control as top-level `think`, and the linked fix PR was closed unmerged after review. The report has a source-provable failure path, but the requested blanket move to `options.think` conflicts with upstream Ollama request handling and current OpenClaw docs/tests, so this needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests moving Ollama `think` request parameters from the native `/api/chat` top level into the `options` block to avoid 400s on non-thinking models.

## Best Possible Solution

Preserve top-level Ollama `think` for reasoning-capable native models, then add a capability-aware guard, warning, or rejection path for truthy thinking values on known non-reasoning models if maintainers confirm that policy.

## Reproduction Assessment

Yes, source-reproducible for the narrowed path: current main can put truthy `params.think`/`params.thinking` on the top-level Ollama chat payload, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact `thinkingDefault: "low"` route still needs live Ollama confirmation.

## Solution Assessment

No, the suggested blanket move to `options.think` is not the best fix because upstream Ollama reads thinking from top-level `ChatRequest.Think` and ignores unknown options. A capability-aware guard or diagnostic preserves supported native thinking while preventing the non-thinking failure path.
