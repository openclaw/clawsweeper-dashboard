## Summary

Keep open: current main and the latest release still intentionally send Ollama native thinking control as top-level `think`, and there is still a source-provable path where explicit truthy `params.think`/`params.thinking` reaches the top-level request. The requested blanket move to `options.think` conflicts with current OpenClaw docs/tests and Ollama's upstream request contract, so this needs maintainer direction rather than cleanup closure.

## What This Changes

Requests changing Ollama native `/api/chat` thinking handling so non-thinking models do not receive truthy top-level `think`, with a suggested move to `options.think`.

## Best Possible Solution

Preserve upstream-compatible top-level `think` for reasoning-capable native Ollama models, then decide whether OpenClaw should guard, warn, or reject truthy explicit thinking params for models known not to support thinking.

## Reproduction Assessment

Yes, for a narrowed source path: current main can put a truthy configured `params.think`/`params.thinking` on the top-level Ollama chat payload, and upstream Ollama rejects truthy `req.Think` for non-thinking models. The exact `thinkingDefault: "low"` gateway route still needs live confirmation because current OpenClaw also has supported-level validation.

## Solution Assessment

No, not as suggested: moving all native Ollama thinking control to `options.think` conflicts with current OpenClaw docs/tests and upstream Ollama source. A safer solution is capability-aware guarding or diagnostics while preserving top-level `think` for supported models.
