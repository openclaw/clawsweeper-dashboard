## Summary

Keep open: the linked Markdown/Telegram bug is source-reproducible and this PR is a plausible narrow fix, but the current patch can corrupt blockquote/code-block span boundaries and still lacks after-fix real behavior proof.

## What This Changes

This PR trims a trailing paragraph separator before closing blockquote style spans, adds blockquote span boundary tests, and adds a changelog fix entry.

## Best Possible Solution

Fix the shared Markdown IR blockquote boundary without mutating text covered by existing child spans, then add IR and Telegram HTML regressions for paragraph and fenced-code blockquote cases.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: `paragraph_close` appends `\n\n` while the blockquote style is still open, and Telegram renders that span as `<blockquote>` HTML.

## Solution Assessment

No. The PR targets the right boundary but trims based only on `target.text.endsWith("\n\n")`; the safer fix should move only a known paragraph separator or adjust the blockquote span end without disturbing existing child spans.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve code-block span boundaries when trimming separators:** `src/markdown/ir.ts:688`
  - body: The new `endsWith("\n\n")` branch cannot distinguish a paragraph separator from the newline pair produced by a fenced code block in a blockquote. `renderCodeBlock` records `code_block` through the code's trailing newline and then appends a second newline, so slicing two bytes before `closeStyle` leaves `code_block.end` past the shortened blockquote end and can render the code block outside or around the quote. Move only a separator known to come from `paragraph_close`, or adjust the blockquote span end without mutating shared text.
  - confidence: 0.88
