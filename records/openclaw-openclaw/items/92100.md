## Summary

The issue is valid on current main: Feishu collapses several failed or no-visible outcomes into one generic English message and logs the reason only server-side. The proposed patch is directionally useful, but it combines channel-local recovery copy with global sub-agent prompt policy and exposes raw internal reason strings, so maintainer product and ownership judgment is needed before implementation.

## What This Changes

Add bilingual, reason-aware Feishu recovery messaging and require visible status replies while delegated work is still pending.

## Best Possible Solution

Define stable localized recovery categories for Feishu, preserve detailed raw reasons in operator logs, and separately add or test a generic visible-status invariant only after tracing the exact terminal-outcome path that produced the tool-only turn.

## Reproduction Assessment

Yes for the opaque fallback: current source and focused tests show multiple failure reasons produce the same generic English message. The exact delegated or sub-agent trajectory has not been reproduced live, so confidence is medium rather than high.

## Solution Assessment

No, not as one bundled change. Improving Feishu recovery text is appropriate, but stable localized categories are safer than raw internal codes, and global prompt changes should be justified by a traced orchestration root cause.
