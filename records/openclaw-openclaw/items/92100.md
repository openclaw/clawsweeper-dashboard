## Summary

Current main still sends the opaque English-only Feishu fallback, and the supplied diagnostic reason remains operator-log-only; the issue is a valid, bounded delivery/recovery defect, although the proposed raw reason-code exposure and prompt-only reinforcement are not the best complete fix.

## What This Changes

Make Feishu’s no-visible-reply fallback actionable after delegated turns and prevent pending-work status requests from ending without visible assistant text.

## Best Possible Solution

Send concise bilingual Feishu recovery guidance with a stable user-facing diagnostic category, keep detailed internal reasons in structured operator logs, and add focused coverage that a pending delegated-work status request produces visible assistant text rather than a tool-only terminal turn.

## Reproduction Assessment

Yes at source level: current main has explicit, tested no-visible fallback paths for empty, failed-final, and zero-final outcomes. The exact delegated-turn trajectory has not been reproduced live, so confidence is medium rather than high.

## Solution Assessment

No, not exactly as proposed. Localization and recovery guidance are appropriate, but raw internal reason codes should remain operator-facing and the shared delegated-status delivery invariant needs direct regression coverage rather than relying only on prompt wording.
