## Summary

Keep open: the PR targets a valid source-reproducible model resolver bug, but the submitted comparator collapses distinct model IDs that share the same numeric segments and still needs resolver-path real behavior proof before merge.

## What This Changes

The PR adds a custom numeric-segment comparator in `src/agents/sessions/model-resolver.ts` and uses it for alias and dated-version candidate sorting.

## Best Possible Solution

Use numeric-aware full-string collation at the existing resolver sort points, add focused `parseModelPattern` regression coverage, and require resolver-path proof before merging this or an equivalent candidate.

## Reproduction Assessment

Yes, source-reproducible. `parseModelPattern("opus", [claude-opus-4-9, claude-opus-4-10])` reaches `tryMatchModel`, and current main's plain descending `localeCompare` orders `4-9` ahead of `4-10`.

## Solution Assessment

No, not as submitted. The sort point is the right layer, but the helper should preserve full-ID ordering through numeric-aware full-string collation or an explicit full-ID tie-breaker plus focused tests.
