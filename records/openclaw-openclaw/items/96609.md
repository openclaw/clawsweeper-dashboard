## What This Changes

The PR adds a custom numeric-segment comparator and uses it for alias and dated-version sorting in `src/agents/sessions/model-resolver.ts`.

## Best Possible Solution

Use numeric-aware full-ID collation at the existing `tryMatchModel` sort points, preserve deterministic tie ordering, and add focused `parseModelPattern` regression coverage.

## Reproduction Assessment

Yes. Source inspection shows `parseModelPattern` reaches `tryMatchModel`, and current main still sorts matching aliases with plain descending `localeCompare`, which orders `claude-opus-4-9` ahead of `claude-opus-4-10`.

## Solution Assessment

No, not as submitted. The existing resolver sort point is the right layer, but the implementation should use numeric-aware full-string collation or an explicit full-ID tie-breaker and should be proven through `parseModelPattern`.
