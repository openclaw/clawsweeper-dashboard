## Summary

Keep open. Current main still has a source-visible Discord missing-reference path where a normal reply can reach the agent without hydrated `ReplyToBody`, and the adjacent open PR only covers part of the hydration/reconnect problem rather than the restart/session-provenance fail-closed behavior reported here.

## What This Changes

Fix Discord referenced-message turns so replies after restart do not answer from unrelated prior task context when the referenced message is unavailable.

## Best Possible Solution

Fix Discord inbound turns so reply references are hydrated or explicitly marked unavailable before model execution, then add a restart/session freshness guard that prevents stale transcript context from answering direct referenced-message questions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for the full live Discord/gateway leak; yes at source level for the missing-reference subpath, where normal text with `message_reference` and no `referencedMessage` bypasses hydration and reaches the agent without `ReplyToBody`.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a complete fix: hydrating missing Discord reply references is narrow and already covered by an open PR, but the restart/session-provenance fail-closed behavior needs maintainer policy before a separate repair should run.
