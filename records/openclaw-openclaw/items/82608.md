## Summary

Keep open. The report is concrete and current main still has a source-visible missing-reference path: ordinary Discord replies with `message_reference` but no hydrated `referencedMessage` can reach the agent without `ReplyToBody`, while existing session/history context remains available. An open adjacent PR addresses Discord reconnect backfill and reply-reference hydration, but it is unmerged and explicitly does not cover process-restart recovery or the fail-closed policy requested here.

## What This Changes

The issue reports Discord reply/reference turns answering from stale unrelated task context when the referenced Discord message is unavailable after restart or gateway/session drift.

## Best Possible Solution

Fix the Discord inbound path so referenced-message turns either hydrate and inject the referenced content or carry an explicit unavailable-reference marker that prevents stale session context from satisfying the direct reply question; coordinate with the open reconnect-backfill PR instead of duplicating its recovery work.

## Reproduction Assessment

No for the full cross-session leak in a live Discord gateway. Yes for the core missing-reference path from source: current main can process a normal text reply with `message_reference` but no `referencedMessage` without hydrating or marking the quote as unavailable.

## Solution Assessment

Unclear as a complete fix. Hydrating missing references is narrow and already covered by an open adjacent PR, but the fail-closed behavior and process-restart/session-provenance guard need maintainer judgment before a second automated repair should be spawned.
