## Summary

Keep open. Current main still has a source-visible Discord reply-reference gap: a normal reply can carry `message_reference` without hydrated `referencedMessage`, then reach the agent without `ReplyToBody` or an explicit unavailable-reference marker while session/history context remains available. The adjacent Discord reconnect/backfill PR is still open and does not cover process-restart recovery or the fail-closed policy requested here.

## What This Changes

The issue reports Discord reply/reference turns after restart answering from unrelated prior task context when the referenced message cannot be loaded.

## Best Possible Solution

Hydrate Discord referenced messages before model execution or carry an explicit unavailable-reference marker that makes direct reply questions ask for missing context instead of relying on unrelated session history, coordinated with the open reconnect/backfill PR.

## Reproduction Assessment

No for the full live cross-session leak: I did not reproduce a restarted Discord gateway leaking the Telegram task. Yes for the core source path: current main can process a normal Discord reply with `message_reference` but no hydrated `referencedMessage` without injecting `ReplyToBody` or an unavailable-reference marker.

## Solution Assessment

Unclear as a complete fix. Hydrating missing references is a narrow code fix and is partly covered by the adjacent open PR, but the restart/session-provenance and fail-closed policy need maintainer direction before a separate automated repair is safe.
