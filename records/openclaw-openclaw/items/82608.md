## Summary

Keep open. Current main has a credible source-level gap for Discord replies whose `message_reference` is present but `referenced_message` is missing: the turn can still dispatch without `ReplyToBody`, leaving the model to answer from session history.

## What This Changes

A Discord reply/reference turn reportedly answered from unrelated prior Telegram task context after restart when the referenced Discord message could not be read.

## Best Possible Solution

Resolve Discord referenced messages before agent dispatch, and fail closed with a clear fallback when the referenced content cannot be loaded instead of answering a short reply from stale session context.

## Reproduction Assessment

Yes for the source-level failure path: a Discord message with `message_reference` but no materialized `referenced_message` can reach dispatch without `ReplyToBody`. I did not live-reproduce the exact Ada restart transcript leak.

## Solution Assessment

No, the current implementation is not the best fix because it silently drops missing reply context and continues dispatch. The safer path is a narrow Discord preflight/hydration guard that fetches the reference or stops the turn with an explicit missing-reference response.

## Repair Work Prompt

Fix Discord inbound replies that have `message_reference` but no materialized `referenced_message`. In the Discord monitor preflight/hydration/context path, resolve the referenced message by `message_reference.channel_id` and `message_reference.message_id` before `buildDiscordMessageProcessContext`, preserving existing abort/REST timeout/access/contextVisibility behavior. If the referenced message cannot be loaded and the inbound turn depends on that reference, fail closed with an explicit safe response or handled/drop path; do not let a short reply such as “Is this done?” dispatch using only stale session transcript context. Do not add config knobs or change generic session routing. Add regression tests for successful REST reference resolution, unavailable/deleted reference fallback, and existing materialized `referenced_message` behavior.
