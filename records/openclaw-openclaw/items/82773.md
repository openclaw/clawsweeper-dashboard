## Summary

Keep open: current main still has the two reported gateway request-path callers using `refreshMode: "sync-when-empty"`, and the cache helpers still await rebuild/direct scan work under that mode. The issue is already paired with open fix PR https://github.com/openclaw/openclaw/pull/82778, so this should stay open until that PR lands or is rejected.

## What This Changes

The issue reports that `usage.cost` and `sessions.usage` still synchronously refresh usage caches on the gateway request path in `v2026.5.12`, causing large-profile gateway stalls despite the durable usage-cache work.

## Best Possible Solution

Land or revise the focused background-refresh PR after acceptable gateway/runtime proof, then close this issue from that merged fix.

## Reproduction Assessment

Yes, at source level: current main still uses `sync-when-empty` at both reported request-path call sites, and the helper implementation synchronously rebuilds or directly scans in that mode. I did not run the reporter's multi-GB profile locally.

## Solution Assessment

Yes, the narrowest maintainable fix is to switch only these gateway request-path callers to the existing `background` helper mode and update the direct tests. The issue should remain open while https://github.com/openclaw/openclaw/pull/82778 is reviewed and proven.
