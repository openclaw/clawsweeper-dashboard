## Summary

Keep open: current main still matches the reporter's incomplete-fix claim, with both gateway usage request paths using `sync-when-empty` and infra awaiting synchronous cache refresh or direct summary loading in that mode.

## What This Changes

The issue reports that `usage.cost` and `sessions.usage` still synchronously refresh usage cache data in 2026.5.12, blocking large gateway profiles despite the durable-cache work from https://github.com/openclaw/openclaw/pull/76650.

## Best Possible Solution

Change only these gateway request-path refresh modes to background refresh, keep the durable cache worker in infra, and add focused regression coverage for cold or stale cache requests returning promptly with cache status.

## Reproduction Assessment

Yes for source-level reproduction: current main still uses `sync-when-empty` at both reported gateway callsites, and the infra loader synchronously refreshes or directly scans under that mode. I did not run a live multi-GB profile reproduction in this read-only pass.

## Solution Assessment

Yes: the reporter's direction is the narrow maintainable fix, because switching these request-path calls to background refresh addresses the blocking path without a broader cache architecture change. Tests that currently assert synchronous mode should be updated to lock in the intended behavior.
