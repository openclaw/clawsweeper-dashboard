## Summary

Keep open: the report points at a real availability problem, but this PR changes OpenClaw’s documented exact user model-pin behavior, has a concrete fallback-default edge case, and still lacks convincing after-fix runtime proof.

## What This Changes

The PR makes `resolveEffectiveModelFallbacks()` return agent fallbacks for `modelOverrideSource: "user"` and updates the user-override command test to expect a fallback retry.

## Best Possible Solution

Keep user-selected model pins strict by default and fix stalls into a visible/recoverable failure path, or add an explicit maintainer-approved fallback policy with aligned docs, tests, and redacted runtime proof.

## Reproduction Assessment

Partial/source: current main clearly disables configured fallbacks for `modelOverrideSource: "user"`, and the PR body includes incident logs showing `fallbackConfigured=false`. I did not reproduce the full live provider-outage stall end to end.

## Solution Assessment

No: the proposed patch can address the stall, but it changes a documented exact model-selection contract and has an `undefined` fallback override edge case. A safer path is strict default behavior plus a visible recovery/failure fix, or an explicit fallback policy if maintainers approve it.
