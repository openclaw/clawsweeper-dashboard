## Summary

Keep open: the PR targets a real availability symptom, but the current patch changes the documented exact model-selection contract and does not yet prove an after-fix selected-model failover in a real run.

## What This Changes

The PR makes `resolveEffectiveModelFallbacks()` return configured agent fallbacks for `modelOverrideSource: "user"` and updates an agent command test to expect a fallback retry.

## Best Possible Solution

Decide the user-selection policy first, then either preserve exact pins and fix the stall-to-visible-error path or add an explicit opt-in fallback policy with aligned docs, tests, and redacted live failover proof.

## Reproduction Assessment

Partial/source: current main clearly disables configured fallbacks for `modelOverrideSource: "user"`, and existing tests assert that strict behavior. I did not reproduce the full live provider-outage stall in this read-only review.

## Solution Assessment

No: returning configured fallbacks for every user model pin is not the narrowest safe fix because it changes documented routing semantics. A safer path is a maintainer-approved policy choice: visible failure for strict pins, or an explicit opt-in fallback mode.
