## Summary

Keep open. Current main still uses process-local Discord component/modal registries, and this draft PR is a plausible narrow fix that moves only the Discord registry path onto the existing SDK keyed state API with memory-first fallback. I did not find a blocking correctness or security finding, but the PR has the protected `maintainer` label, is still draft, and should go through maintainer review plus normal CI before merge.

## What This Changes

The PR adds best-effort Discord component/modal registry persistence through `api.runtime.state.openKeyedStore`, wires async persistent lookup/consume into Discord interaction handlers, and updates tests, Discord docs, and the changelog.

## Best Possible Solution

Land a Discord-owned persistence implementation that keeps the in-memory registry as the hot path, uses the bundled plugin SDK keyed store only as best-effort restart durability, preserves single-use consume semantics, documents the behavior, and ships with targeted registry and interaction-flow tests plus the changed gate.

## Reproduction Assessment

Yes. Current-main source shows the registry lives only in process-global maps, so a Gateway restart necessarily drops component/modal entries; the PR's mocked persistence tests model clearing memory and resolving from the keyed store. I did not run tests because this review is read-only.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable solution: keep the change inside the Discord plugin and reuse the existing SDK state API rather than adding core Discord-specific migration logic. It still needs maintainer review and CI because the PR is draft/protected and changes interaction runtime behavior.
