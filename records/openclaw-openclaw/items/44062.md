## Summary

Keep open. The reported bug is valid, and the PR has useful source work, but it is not mergeable as-is against current main because outbound media roots now flow through a central resolver that still has no channel/account config inputs, and the advertised Telegram/Feishu config keys are still not accepted by the strict schemas.

## What This Changes

The PR threads channel/account `mediaLocalRoots` into outbound media root construction, adds Telegram config typing, and adds outbound plus Feishu regression coverage.

## Best Possible Solution

Rework the fix on current main so configured roots are resolved once in the shared media-access layer, accepted by the real channel/account schemas and metadata, and looked up through the same default/canonical account resolution used by outbound sends.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure channel/account `mediaLocalRoots`, send local media through generic outbound delivery or plugin message actions, and current main has no path for those configured roots to enter `resolveAgentScopedOutboundMediaAccess`; strict Telegram/Feishu schemas also reject the advertised keys.

## Solution Assessment

No. The PR direction is right, but the current patch targets stale direct call sites and raw config lookup; the maintainable fix should be in the shared resolver with schema, metadata, default-account, and normalized-account coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Thread configured roots through the central resolver:** `src/infra/outbound/deliver.ts:548`
  - body: Current main no longer builds outbound roots at this direct delivery call site; it routes media through `resolveAgentScopedOutboundMediaAccess`. Rebase the fix into that shared resolver and pass channel/account context from delivery, send-service, and message-action paths, or generic outbound delivery will still miss configured roots.
  - confidence: 0.9
- **[P1] Accept mediaLocalRoots in parsed channel schemas:** `src/media/local-roots.ts:65-71`
  - body: The helper reads channel/account `mediaLocalRoots`, but the real Telegram and Feishu schemas are strict and this PR only adds the Telegram TypeScript field. Normal user configs for those advertised cases are rejected before these values reach the helper, so add schema and generated metadata coverage.
  - confidence: 0.9
- **[P2] Resolve the default account before reading account roots:** `src/media/local-roots.ts:63`
  - body: Account roots are only loaded when `options.accountId` is explicitly set. Default-account sends can resolve the account later in the channel path, so `channels.<channel>.accounts.<default>.mediaLocalRoots` remains omitted for common sends without an explicit account id.
  - confidence: 0.84
- **[P2] Normalize account ids before account-root lookup:** `src/media/local-roots.ts:63`
  - body: The direct `accountsObj[accountId]` lookup is case-sensitive, while channel account resolution canonicalizes ids. Passing a differently cased account id can still select the account but silently skip its trusted media roots here.
  - confidence: 0.8
