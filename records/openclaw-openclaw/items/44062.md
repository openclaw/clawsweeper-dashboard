## Summary

Keep open. The PR is a useful direction, but it is conflicting and incomplete against current main: outbound media now flows through the shared media-access resolver, strict Telegram/Feishu schemas still do not accept the advertised config keys, account-scoped roots are looked up non-canonically, and the PR still lacks real after-fix media-send proof.

## What This Changes

The PR merges configured channel/account `mediaLocalRoots` into outbound delivery/action send paths, adds a Telegram config type field, and adds focused outbound media-root tests.

## Best Possible Solution

Define and validate one canonical configured local-media-root contract, route it through `resolveAgentScopedOutboundMediaAccess` with canonical account resolution, and prove a real configured-root media send.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure channel/account `mediaLocalRoots`, send a local media payload through outbound delivery, and current main has no configured-root input in the shared resolver while strict Telegram/Feishu schemas omit the key.

## Solution Assessment

No. The direction is useful, but this PR needs to be rebased around the shared media-access resolver, schema/account-resolution contracts, and a maintainer-approved filesystem trust policy before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread configured roots through the shared resolver:** `src/infra/outbound/deliver.ts:548-552`
  - body: Current main computes outbound media roots through `resolveAgentScopedOutboundMediaAccess`, but this patch only adds channel/account context to older direct `getAgentScopedMediaLocalRoots` call sites. After rebase, generic delivery can still drop configured roots unless the shared resolver accepts and applies that context.
  - confidence: 0.9
- **[P1] Accept mediaLocalRoots in runtime config schemas:** `src/config/types.telegram.ts:152-153`
  - body: Adding the Telegram TypeScript field is not enough for production configs. The strict Telegram and Feishu Zod schemas still omit `mediaLocalRoots`, so the advertised channel/account keys can be rejected before the helper ever reads them.
  - confidence: 0.9
- **[P2] Resolve account roots through canonical account lookup:** `src/media/local-roots.ts:66`
  - body: The helper reads `accounts[accountId]` directly. Sends that use a configured default account, or an account id whose casing differs from the config key, can still resolve/send on that account while losing its account-scoped media roots.
  - confidence: 0.84
