## Summary

Keep open. The underlying configured `mediaLocalRoots` bug is still source-reproducible, but this PR is now conflicting and does not satisfy current main's shared resolver, schema validation, account-resolution, proof, and security-policy requirements.

## What This Changes

The PR adds channel/account `mediaLocalRoots` merging to outbound delivery/message-action/send-service paths, declares a Telegram config type field, and adds focused outbound regression tests.

## Best Possible Solution

Define one canonical configured local-media-root contract in the shared outbound media-access resolver, validate it in the relevant strict schemas, use canonical account resolution, and prove a real media send after the fix.

## Reproduction Assessment

Yes, by source inspection. Configure channel/account `mediaLocalRoots`, send a local media payload through outbound delivery, and current main still has no configured-root input in the shared resolver while strict Telegram/Feishu schemas omit the key.

## Solution Assessment

No. The direction is useful, but this PR needs to be reworked around the current shared resolver, real schema validation, canonical account lookup, and host-media policy before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Thread configured roots through the shared resolver:** `src/infra/outbound/deliver.ts:539`
  - body: Current main computes outbound media roots through `resolveAgentScopedOutboundMediaAccess`. Adding channel/account context only to the older direct `getAgentScopedMediaLocalRoots` call site leaves the canonical delivery path without configured roots after rebase.
  - confidence: 0.9
- **[P1] Accept mediaLocalRoots in runtime schemas:** `src/config/types.telegram.ts:153-154`
  - body: Adding the Telegram TypeScript field is not enough for production config. The strict Telegram and Feishu Zod schemas still omit `mediaLocalRoots`, so real configs can be rejected before the new root-merging helper sees them.
  - confidence: 0.9
- **[P2] Resolve account roots through canonical lookup:** `src/media/local-roots.ts:64-66`
  - body: The helper reads `accounts[accountId]` directly. Sends that use the configured default account, or account IDs with different casing from the config key, can still resolve an account elsewhere while losing its account-scoped media roots here.
  - confidence: 0.84
