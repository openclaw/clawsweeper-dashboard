## Summary

This PR should close in favor of the cleaner same-root-cause candidate because it suppresses every no-tool heartbeat reply whenever the heartbeat tool is forced, while the supported contract only makes missed-tool final text private after an explicit `visibleReplies: "message_tool"` opt-in.

## What This Changes

The branch adds a heartbeat-runner guard that suppresses raw reply delivery when `usesHeartbeatResponseTool` is true, no `heartbeat_respond` result is present, and the reply payload is not marked for source-suppression delivery.

## Best Possible Solution

Close this branch and review or land https://github.com/openclaw/openclaw/pull/94088 as the canonical fix for the linked heartbeat privacy bug.

## Reproduction Assessment

Yes. The linked report and current source show that `sourceReplyDeliveryMode: "message_tool_only"` can be set, no `heartbeat_respond` result can be present, and current main then falls through to durable channel delivery when `shouldSkipMain` is false.

## Solution Assessment

No. The runner-level guard is the right area, but this branch scopes it too broadly; the safer solution is the canonical candidate that requires explicit message-tool visibility and preserves Codex default fallback behavior.
