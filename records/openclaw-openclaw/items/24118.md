## Summary

This remains a valid but under-specified Signal feature. Current main and v2026.6.6 do not expose preview controls, while direct signal-cli inspection confirms two distinct mechanisms: primary-device-only `updateConfiguration(linkPreviews)` and explicit per-message preview fields. Because the requested default conflicts with its no-change requirement and metadata handling overlaps a live security concern, maintainers should decide the product boundary before implementation.

## What This Changes

Add a `channels.signal.linkPreview` setting and wire Signal link-preview configuration and metadata fields into signal-cli JSON-RPC sending.

## Best Possible Solution

Define separate Signal-plugin-owned controls for upstream-generated previews and explicit per-message metadata, preserve existing behavior when unset, document primary-device and native/container behavior, and keep externally fetched metadata out of agent context.

## Reproduction Assessment

Not applicable: this requests a new configuration and transport capability rather than reporting broken behavior under an existing contract. Current source and release inspection establish that the capability is absent.

## Solution Assessment

No. A single default-on boolean plus broad field pass-through conflates two upstream mechanisms; the maintainable solution should separate their semantics, preserve unset behavior, and define security and backend constraints first.

## Risks / Open Questions

- The requested default of `true` conflicts with the requirement that unset configuration cause no behavior change.
- The proposal combines account/device configuration for Signal-generated previews with explicit per-message preview metadata, although signal-cli exposes them as separate mechanisms.
- `updateConfiguration` is primary-device-only, so automatic invocation could fail for common linked-device deployments unless startup and error behavior are defined.
- Native signal-cli and the supported container backend need an explicit capability policy to avoid mode-dependent silent behavior.
- Fetching preview metadata inside OpenClaw could introduce untrusted URL content into the system and should remain out of agent context unless the security design in https://github.com/openclaw/openclaw/issues/22060 is resolved.
