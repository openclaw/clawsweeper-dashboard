## Summary

Keep open: current main and v2026.6.6 still lack Signal link-preview config, send payload preview fields, and updateConfiguration wiring, and the requested shape needs maintainer product/security decisions because upstream separates primary-device configuration from explicit per-message previews.

## What This Changes

The issue requests a first-class `channels.signal.linkPreview` config option plus Signal JSON-RPC preview pass-through for outbound messages.

## Best Possible Solution

Define Signal-plugin-owned preview controls that preserve unset behavior, separate upstream-generated previews from explicit per-message metadata, document native/container and primary-device limits, and keep fetched metadata out of agent context unless the security design approves it.

No. The proposed single default-on boolean plus broad pass-through is not the best shape because upstream exposes generated previews and explicit metadata as separate mechanisms with different compatibility and security constraints.

## Risks / Open Questions

- The requested default `true` conflicts with the issue's no-behavior-change requirement for users who do not set the key.
- A single boolean conflates upstream-generated previews controlled by `updateConfiguration(linkPreviews)` with explicit per-message preview metadata fields.
- `updateConfiguration` is primary-device-only upstream, so linked-device Signal deployments need explicit failure, retry, and startup semantics.
- Native signal-cli and container REST mode may not have the same preview capability, so mode-dependent behavior needs a documented policy.
- Fetching or injecting preview metadata can overlap the open prompt-injection concern in https://github.com/openclaw/openclaw/issues/22060 if untrusted URL metadata ever reaches agent context.
