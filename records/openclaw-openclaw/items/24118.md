## Risks / Open Questions

- The requested default `true` conflicts with the issue's no-behavior-change requirement for users who do not set the key.
- A single boolean would conflate upstream-generated previews controlled by `updateConfiguration(linkPreviews)` with explicit per-message preview metadata fields.
- `updateConfiguration` is primary-device-only upstream, so linked-device Signal deployments need explicit failure, retry, and startup semantics.
- Native signal-cli and bbernhard container REST mode may not have the same preview capability, so mode-dependent behavior needs a documented policy.
