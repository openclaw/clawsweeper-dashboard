## Summary

Keep open. The PR fixes the right source path by registering plugin SDK subagent runs, but as submitted it changes session cleanup semantics and still has only unit/source proof for an external PR.

## What This Changes

The PR adds lazy gateway-side subagent registry registration and focused tests so plugin SDK `api.runtime.subagent.run()` runs can emit `subagent_ended`.

## Best Possible Solution

Register plugin SDK subagent runs so `subagent_ended` fires once, but preserve session retention by default and prove the wait/read/delete workflow with focused tests plus real plugin-harness evidence.

## Reproduction Assessment

Yes, at source level. Current main dispatches `api.runtime.subagent.run()` through Gateway `agent` and returns a `runId` without creating the `SubagentRunRecord` that `emitSubagentEndedHookOnce()` requires; I did not run a live plugin harness.

## Solution Assessment

No, not as submitted. Registering the run in the gateway adapter is the right boundary, but the registration should preserve the SDK's existing session retention contract instead of hard-coding delete cleanup.

- **[P1] Preserve plugin subagent sessions after completion:** `src/gateway/server-plugins.ts:508`
  - body: `api.runtime.subagent` is documented as `run()` -> `waitForRun()` -> `getSessionMessages()` with explicit `deleteSession()` cleanup, and current main does not delete these sessions automatically.
