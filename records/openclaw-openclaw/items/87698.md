## Summary

Keep this PR open: the central fix is valid and current main still lacks the plugin-SDK subagent registry registration, but the proposed patch still has two blocking lifecycle edge cases around cached Gateway responses and canonical session keys, and the external real-behavior proof remains short of a real plugin process driving a running Gateway.

## What This Changes

The PR adds best-effort subagent registry registration after plugin SDK `api.runtime.subagent.run()` dispatches the Gateway `agent` method, with regression tests for hook delivery and transcript readback.

## Best Possible Solution

Register only fresh accepted Gateway agent responses, use the accepted canonical `payload.sessionKey` for the registry child session, add focused regression coverage for both cases, and then attach redacted real plugin/Gateway proof before merge.

## Reproduction Assessment

Yes. Source inspection on current main shows the plugin SDK `run()` adapter dispatches the Gateway `agent` method and returns the `runId` without registering a subagent record, while the documented SDK flow expects `run() -> waitForRun() -> getSessionMessages() -> deleteSession()`.

## Solution Assessment

No, not as currently written. Adding registration at the plugin SDK adapter is the right shape, but it must respect the Gateway response contract by registering only fresh accepted runs and using the accepted canonical session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Skip cached agent responses before registering:** `src/gateway/server-plugins.ts:597-603`
  - body: The Gateway `agent` handler returns cached `in_flight` responses for repeated idempotency keys. This new registration path only checks for a `runId`, so a plugin retry would call `registerSubagentRun` again for an existing run; the registry stores by `runId` and replaces the entry, resetting lifecycle state and risking duplicate or stale `subagent_ended` handling. Please register only a fresh accepted response, or otherwise skip cached/non-accepted payloads.
  - confidence: 0.9
- **[P1] Register the canonical accepted session key:** `src/gateway/server-plugins.ts:599`
  - body: The Gateway accepts an agent run with `payload.sessionKey` set to the resolved canonical session key, which can differ from `params.sessionKey` for alias/global routes. Registering the raw requested key ties the subagent registry and hook payload to a session the run may not actually use, so readback, cleanup, and ownership can drift. Please use the accepted response session key when present and cover the canonicalization case.
  - confidence: 0.86
