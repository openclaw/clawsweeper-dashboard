## Summary

Keep open: the bug and fix direction are valid, but the PR still makes plugin-created subagent runs enter two task ownership paths, which can make operator task listing/status/cancellation ambiguous.

## What This Changes

The PR registers plugin SDK subagent runs after `api.runtime.subagent.run()` so `subagent_ended` hooks can fire, and adds regression tests for hook registration plus transcript retention.

## Best Possible Solution

Make plugin-spawned subagents use one canonical task/subagent ownership path that emits `subagent_ended`, preserves transcript readback, and keeps task listing/cancellation semantics unambiguous.

## Reproduction Assessment

Yes, by source inspection: current main's plugin SDK subagent runtime dispatches the Gateway `agent` method and returns the runId without creating a subagent registry record. I did not run a live installed-plugin repro in this read-only review.

## Solution Assessment

No, not as currently patched: registering the subagent run is the right direction, but the branch must avoid creating a second task record for the same plugin-created run and still needs real Gateway proof.
