## Summary

Keep open. The PR targets a valid source-reproducible plugin SDK lifecycle bug, but the proposed registry metadata would newly delete plugin-created subagent sessions after completion, breaking the documented wait/read/delete workflow.

## What This Changes

The PR adds a gateway-side registration step and regression tests so plugin SDK `api.runtime.subagent.run()` runs enter the subagent registry and can emit `subagent_ended`.

## Best Possible Solution

Register plugin SDK subagent runs with metadata that fires `subagent_ended` once while preserving existing session retention, then prove the wait/read/delete workflow with focused tests and real plugin-harness evidence.

## Reproduction Assessment

Yes, at source level. Current main dispatches `api.runtime.subagent.run()` through Gateway `agent` and returns a `runId` without creating the `SubagentRunRecord` that `emitSubagentEndedHookOnce()` requires; I did not run a live plugin harness.

## Solution Assessment

No, not as submitted. Registering the run in the gateway adapter is the right boundary, but the registration should preserve the SDK's existing session retention contract instead of hard-coding delete cleanup.

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve plugin subagent sessions after completion:** `src/gateway/server-plugins.ts:508`
  - body: `api.runtime.subagent` is documented as `run()` -> `waitForRun()` -> `getSessionMessages()` with explicit `deleteSession()` cleanup, and current main does not delete these sessions automatically. Registering every plugin run with `cleanup: "delete"` plus `expectsCompletionMessage: false` enters the lifecycle path that calls `sessions.delete` on completion, so existing plugins can lose the transcript before they read it. Use the registry bridge with keep-style cleanup by default, or add an explicit cleanup option with docs/tests if deletion is intentional.
  - confidence: 0.91
