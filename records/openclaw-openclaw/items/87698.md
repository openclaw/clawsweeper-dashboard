## Summary

This PR targets a real plugin SDK lifecycle bug, but it should stay open because the current patch still registers cached/in-flight Gateway responses as fresh runs, records the raw requested session key instead of the Gateway-accepted canonical key, and lacks real plugin-process proof against a running Gateway.

## What This Changes

The branch adds best-effort subagent registry registration after plugin SDK `api.runtime.subagent.run()` plus Gateway regression tests for hook firing and transcript retention.

## Best Possible Solution

Register only fresh accepted Gateway responses with the Gateway-accepted child session key, keep transcript-preserving cleanup semantics, add idempotency and alias/global regression coverage, then provide redacted real plugin-process proof before merge.

## Reproduction Assessment

Yes. Current main dispatches `api.runtime.subagent.run()` through the Gateway `agent` method without registering a `SubagentRunRecord`, and the linked issue gives a concrete plugin hook reproduction path; I did not run a live plugin harness here.

## Solution Assessment

No. Registering at the plugin SDK adapter is the right seam, but this patch needs to key registration off fresh accepted Gateway responses and `payload.sessionKey` before it is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Skip cached agent responses before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: The Gateway `agent` method returns cached `{ status: "in_flight" }` payloads for repeated `idempotencyKey` calls, but this code registers every response that contains a `runId`. Since `registerSubagentRun` replaces the map entry for that run id, a retry can reset lifecycle fields for the already-running child; only register fresh accepted runs and cover cached replies.
  - confidence: 0.9
- **[P1] Use the Gateway-accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: This passes `params.sessionKey` into the registry even though the Gateway canonicalizes aliases and returns the accepted session key in the agent payload. If a plugin passes `main`, `global`, or another alias-like target, the run can execute under `payload.sessionKey` while hooks, cleanup, and readback point at the raw requested key; register the accepted key and cover the alias case.
  - confidence: 0.88
