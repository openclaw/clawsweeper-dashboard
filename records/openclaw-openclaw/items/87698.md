## What This Changes

This PR adds best-effort subagent registry registration after plugin SDK `api.runtime.subagent.run()` and Gateway regression tests for hook delivery plus transcript retention.

## Best Possible Solution

Register only newly accepted plugin SDK agent runs, use the Gateway-accepted session key for the registry entry, preserve existing registry state on retries, and then provide redacted real plugin/Gateway proof or an explicit maintainer proof override.

Yes. Source inspection on current main shows `api.runtime.subagent.run()` dispatches the Gateway `agent` method without registering a subagent run, while `subagent_ended` emission reads from the registry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Skip cached agent replies before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: When the same plugin retries `api.runtime.subagent.run()` with the same idempotency key, the Gateway can return a cached `status: "in_flight"` response for the existing run. This code still calls `registerSubagentRun`, and the registry unconditionally overwrites `runs.set(runId, entry)`, resetting lifecycle/hook state and starting another wait for a run that was not newly spawned. Please inspect the `agent` response status/meta and skip registration for cached or in-flight replies.
  - confidence: 0.9
- **[P1] Use the Gateway-accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: The Gateway `agent` method returns the resolved `sessionKey` in the accepted payload after alias/global canonicalization, but this registers `params.sessionKey` instead. If a caller passes an alias, global key, or any value the Gateway normalizes, `subagent_ended` and cleanup/readback state can point at a different session from the one that actually ran. Capture `payload.sessionKey` and register that canonical key, with a regression test for a normalized session key.
  - confidence: 0.86
