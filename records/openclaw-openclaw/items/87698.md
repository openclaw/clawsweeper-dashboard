## Summary

Keep open: the hook-registration direction addresses the linked plugin SDK bug, but the current patch still creates overlapping task-ledger ownership for one plugin-spawned run and lacks installed-plugin proof, so it is not merge-ready.

## What This Changes

The PR adds best-effort subagent registry registration after plugin SDK `api.runtime.subagent.run()` dispatch and adds Gateway regression tests for hook emission and transcript retention.

## Best Possible Solution

Make plugin SDK subagent runs produce one canonical registry/task record that emits `subagent_ended`, preserves transcript readback until explicit deletion, and is proven with task-list/cancel regression coverage plus a redacted installed-plugin Gateway run.

## Reproduction Assessment

Yes from source inspection: current `main` shows the plugin SDK path dispatching to `agent` without subagent registration, while the PR path plus existing task writers show the duplicate task-record failure. I did not run an installed plugin harness in this read-only review.

## Solution Assessment

No: registering after dispatch is the right hook direction, but the patch is not yet the best solution because it adds a second task owner for the same plugin-spawned run. The maintainable fix should reconcile the Gateway task and subagent registry task into one canonical record.
