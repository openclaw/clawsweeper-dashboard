## Summary

Keep open: the hook fix targets a real Plugin SDK bug, but the current patch still overlays subagent registry tracking on a Gateway agent path that already creates a task row, so task listing/lookup/cancel behavior can diverge before merge.

## What This Changes

The PR adds post-dispatch subagent registry registration for plugin SDK `api.runtime.subagent.run()` plus Gateway regression coverage for hook emission and transcript retention.

## Best Possible Solution

Keep the hook fix, but make plugin-created subagent runs produce one canonical task record and prove hook delivery, transcript retention, and task API behavior together before merge.

## Reproduction Assessment

Yes, at source level. Current main dispatches `api.runtime.subagent.run()` through Gateway `agent` and returns the run id without subagent registry registration, while the public docs present `subagent_ended` as the lifecycle hook for subagent completion.

## Solution Assessment

No. The proposed registration point is plausible for hook delivery, but the current patch overlays subagent task creation on an already tracked Gateway task; the maintainable fix needs one canonical task representation and task API coverage.
