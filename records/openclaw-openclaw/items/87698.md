## Label Justifications

- `P2`: This is a normal-priority plugin SDK/Gateway lifecycle bug with limited blast radius but real task-ledger merge risk.
- `merge-risk: 🚨 compatibility`: Existing task clients and operators can see an extra task row for plugin-created runs and get different cancellation behavior depending on the selected row.
- `merge-risk: 🚨 session-state`: The patch changes subagent registry ownership and child-session lifecycle tracking for plugin SDK runs, which can mis-associate task/session state unless the ownership model is unified.

## Summary

Keep open: the underlying hook bug is real, but this patch still adds subagent registry task creation on top of the Gateway `agent` task record, leaving duplicate task-ledger rows and different cancellation semantics for the same plugin-created run.

## What This Changes

The branch adds Gateway-side registration so `api.runtime.subagent.run()` creates a subagent registry record for `subagent_ended` hook delivery, plus regression tests for hook registration and transcript retention.

## Best Possible Solution

Keep one canonical task-ledger owner for plugin-created subagent runs while registering enough lifecycle state to emit `subagent_ended` and preserve transcript readback.

## Reproduction Assessment

Yes. Source inspection on current main shows `api.runtime.subagent.run()` dispatches the Gateway `agent` method and returns `runId` without creating a `SubagentRunRecord`, while hook emission depends on registry tracking.

## Solution Assessment

No. Registering lifecycle state is the right direction, but stacking `registerSubagentRun` on top of the existing Gateway agent task record creates two task-ledger sources for one run.
