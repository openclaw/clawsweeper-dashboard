## Summary

Keep this PR open: the hook-registration direction matches the linked bug, but the current patch can create two task-ledger rows for one plugin SDK subagent run and still lacks contributor-side installed-plugin Gateway proof.

## What This Changes

The PR registers `api.runtime.subagent.run()` results in the subagent registry and adds regression tests for `subagent_ended` hook delivery plus transcript retention after completion.

## Best Possible Solution

Make plugin SDK subagent runs use one canonical task owner, cover hook delivery/transcript readback/single-row status and cancellation, then attach redacted installed-plugin Gateway proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main's plugin SDK `run()` dispatches Gateway `agent` and returns the `runId` without registering a subagent run, while the subagent registry is the path that creates `SubagentRunRecord` state and emits `subagent_ended`. I did not run a live plugin harness in this read-only review.

## Solution Assessment

No, not as submitted: registering after Gateway dispatch fixes the missing hook path, but it also layers subagent task tracking on top of the existing Gateway `cli` task for the same run. The safer solution is one canonical task owner with tests for hook delivery, transcript readback, task listing, status, and cancellation.
