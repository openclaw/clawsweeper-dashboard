## Summary

## Best Possible Solution

Land a version that registers only new accepted Gateway runs using the accepted canonical session key, preserves transcript-readback cleanup, adds retry and alias regressions, and includes redacted real Gateway/plugin proof or maintainer override.

## Solution Assessment

No, the direction is right but this implementation is not the best mergeable shape yet. It should use the raw Gateway response to avoid cached retry registration and use the accepted canonical session key before registering the subagent run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Skip cached agent replies before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: The Gateway `agent` method returns cached accepted replies with `meta.cached: true` for idempotent retries, but this code unwraps the payload and registers every returned `runId`. Since `registerSubagentRun` replaces the map entry for that `runId`, retrying `api.runtime.subagent.run()` with the same idempotency key can reset lifecycle fields and disturb hook/session cleanup state; use the raw dispatch response and add a retry regression that skips registration for cached replies.
  - confidence: 0.9
- **[P1] Register the accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: This passes the raw requested `params.sessionKey` into the registry, but the Gateway resolves aliases and legacy keys before accepting the run and includes the canonical `sessionKey` in the accepted payload. When those differ, the subagent registry and `subagent_ended` hook point at a different child session than the run actually used; register the accepted session key and add alias/canonicalization coverage.
  - confidence: 0.87
