## Summary

Keep open: the branch fixes a real source-reproducible plugin SDK hook gap, but it is not merge-ready because it registers a gateway-spawned run into the subagent registry after the gateway agent path already creates a `cli` task for the same runId, leaving task status and cancellation ambiguous.

## What This Changes

The PR registers plugin SDK subagent runs after `api.runtime.subagent.run()` and adds gateway tests for hook registration plus transcript retention.

## Best Possible Solution

Plugin-spawned subagents should have one canonical registry/task ownership path that emits `subagent_ended`, preserves transcript readback, and keeps `tasks.list` plus cancellation unambiguous.

Yes by source inspection: current main’s plugin SDK runtime dispatches Gateway `agent` and returns the runId without creating a `SubagentRunRecord`; I did not run a live installed-plugin repro in this read-only review.

## Solution Assessment

No, not as patched: registering the subagent run is directionally right, but the implementation must avoid creating a second task record and needs real Gateway/plugin proof.

- **[P1] Avoid creating a second task record for plugin subagent runs:** `src/gateway/server-plugins.ts:594-600`
