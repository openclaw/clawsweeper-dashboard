## Summary

## What This Changes

The PR registers plugin SDK subagent runs after `api.runtime.subagent.run()` dispatch, preserves the child session for transcript readback, and adds gateway regression tests.

## Best Possible Solution

Make plugin SDK subagent runs use one canonical task record, cover hook delivery plus transcript readback plus task list/status/cancel behavior, then attach redacted installed-plugin Gateway proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main's plugin SDK `run()` dispatches Gateway `agent` and returns a runId without registering a subagent run, while the registry is the path that creates `SubagentRunRecord` state and emits `subagent_ended`. I did not run a live plugin harness in this read-only review.

## Solution Assessment

No, not as submitted: registering after Gateway dispatch fixes the missing hook path, but it also layers subagent task tracking on top of the existing Gateway `cli` task for the same run. The safer solution is one canonical task owner with focused task-list/status/cancel coverage.
