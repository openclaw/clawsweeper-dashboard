## Summary

Keep open: the PR is aimed at a real documented plugin lifecycle bug, but the current patch can overwrite subagent lifecycle state on idempotent retries and can associate hook state with the raw requested session key instead of the Gateway-accepted key. The external proof is still an in-process/stubbed proof rather than a plugin process against a running Gateway or maintainer override.

## What This Changes

The PR adds best-effort subagent registry registration in the plugin SDK Gateway runtime and a new Gateway regression test for `subagent_ended` delivery plus transcript retention.

## Best Possible Solution

Land the Gateway/plugin SDK fix only after registration is idempotent for cached replies, uses the Gateway-accepted child session key, preserves transcript readback, and has real plugin-process proof or a maintainer override.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main dispatches `api.runtime.subagent.run()` through Gateway `agent` without registering a `SubagentRunRecord`, so registry-driven `subagent_ended` delivery cannot happen for that path. I did not execute tests because this review is read-only.

## Solution Assessment

No. Registering plugin SDK runs in this seam is the right direction, but the implementation must skip cached retries and use the accepted session key before it is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Skip cached agent replies before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: `agent` can return an in-flight cached reply for the same idempotency key, and `registerSubagentRun` replaces any existing record for that runId. Registering on every successful response means a normal retry can overwrite lifecycle state for the active or already-completed subagent run; use the raw dispatch response and skip registration when `meta.cached === true`, or make the registry insert idempotent.
  - confidence: 0.89
- **[P1] Use the Gateway-accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: The Gateway accepted payload carries the canonical `sessionKey` after alias/global resolution, but this code registers the raw request key. When those differ, `subagent_ended` and registry ownership can be attached to the wrong session; read `payload.sessionKey` from the accepted response and register that value as the child session key.
  - confidence: 0.86
