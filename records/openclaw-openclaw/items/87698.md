## Summary

Keep open: the PR is aimed at a real plugin SDK subagent lifecycle bug, but the new registration path still registers cached `agent` retry responses and stores the raw requested session key instead of the Gateway-accepted child session key. Those two cases can reset subagent registry state or point `subagent_ended`/cleanup/readback at the wrong session, and the attached proof is still an in-process capture rather than a real plugin process against a running Gateway.

## What This Changes

The branch adds best-effort subagent registry registration after plugin SDK `api.runtime.subagent.run()` and a new Gateway regression test file for hook registration and transcript retention.

## Best Possible Solution

Keep the plugin SDK registration seam, but register only newly accepted `agent` responses, use the Gateway-accepted child session key, cover cached-idempotency and canonical-session cases, and attach redacted real plugin/Gateway proof before merge.

No. Adding registry registration at the plugin SDK seam is the right direction, but the current implementation must avoid cached responses, use the accepted session key, and be backed by real plugin/Gateway proof.

- **[P1] Skip cached agent replies before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: This registers a subagent run for every `agent` response with a `runId`, but current Gateway retries with the same idempotency key return a cached `{ status: "in_flight" }` response without starting a new run. Since `registerSubagentRun` overwrites the existing record for the same `runId`, a plugin retry can reset lifecycle fields, start another wait, and disturb cleanup/hook state for the already running child; please preserve the raw response status/meta and register only the newly accepted run.
  - confidence: 0.89
- **[P1] Use the Gateway-accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: The registry stores `params.sessionKey`, but the `agent` method responds with the canonical `sessionKey` it actually accepted after resolving aliases, global sessions, or selected-agent routing. If a plugin passes one of those forms, the child run executes under the accepted key while `subagent_ended`, cleanup, and transcript lookup are tied to the raw key; register with the accepted payload session key and add alias/global coverage.
  - confidence: 0.86
