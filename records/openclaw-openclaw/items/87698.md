## Summary

The PR fixes a real plugin SDK lifecycle gap and has strong after-fix proof, but it still has one merge-blocking idempotency bug: duplicate `api.runtime.subagent.run()` retries can re-register the same gateway run and overwrite registry state. Keep it open for a narrow repair rather than closing or landing as-is.

## What This Changes

The PR adds gateway-side registration for plugin SDK subagent runs so `subagent_ended` hooks fire, with regression coverage for hook delivery and transcript readback using `cleanup: "keep"`.

## Best Possible Solution

Land the gateway-side registration with `cleanup: "keep"` after making registration idempotent for duplicate gateway runIds and keeping the focused hook/readback coverage.

## Reproduction Assessment

Yes from source inspection: current main dispatches `api.runtime.subagent.run()` through the gateway `agent` method and returns the runId without creating a SubagentRunRecord, while docs and hook types define `subagent_ended` as the completion hook. I did not run tests in this read-only review.

## Solution Assessment

No as submitted: registering after gateway dispatch is the right narrow seam, and `cleanup: "keep"` preserves the documented readback flow, but the registration must be idempotent for duplicate gateway runIds before merge.
