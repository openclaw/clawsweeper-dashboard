## Summary

Keep this PR open: it targets a valid plugin SDK bug, but the current patch still creates competing task-ledger ownership for the same plugin-spawned run and the external real-behavior proof remains test-only rather than an installed plugin against a running Gateway.

## What This Changes

The PR adds best-effort registration of `api.runtime.subagent.run()` runs in the subagent registry and adds Gateway regression tests for hook emission and transcript retention.

## Best Possible Solution

Keep the hook fix direction, but make plugin SDK subagent runs have exactly one task-ledger/session owner while still firing `subagent_ended`, preserving transcript readback until explicit delete, and proving it with an installed plugin against a running Gateway.

Yes, source inspection gives a high-confidence path: current `api.runtime.subagent.run()` dispatches `agent` and returns a run id without registering a `SubagentRunRecord`, while the hook emission path depends on the registry record. ## Solution Assessment

No, not yet. Registering the run after Gateway dispatch fixes the missing hook direction, but the proposed path also creates a second task-ledger record unless the Gateway `cli` task producer is suppressed, adopted, or otherwise made the single canonical owner.
