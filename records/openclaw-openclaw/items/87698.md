## Label Justifications

- `P2`: This is a normal-priority plugin SDK bug fix with bounded but real lifecycle impact for plugins using subagent hooks.
- `merge-risk: 🚨 compatibility`: The PR changes public plugin SDK runtime behavior and can affect existing plugin retries that rely on idempotent `api.runtime.subagent.run()` calls.

## Summary

Keep open: this is a useful fix direction for a real plugin SDK lifecycle bug, but the current patch still registers cached/in-flight Gateway replies, uses the caller's raw session key instead of the Gateway-accepted key, and only has test/stub proof rather than a real plugin process against a running Gateway.

## What This Changes

The PR adds best-effort subagent registry registration to plugin SDK `api.runtime.subagent.run()` and a Gateway regression test for hook delivery plus transcript retention.

## Best Possible Solution

Register only newly accepted plugin SDK agent runs, use the Gateway-accepted session key for the registry entry, preserve existing registry state on retries, and then provide real plugin/Gateway proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes. Source inspection on current main shows `api.runtime.subagent.run()` dispatches the Gateway `agent` method and returns a run id without registering a `SubagentRunRecord`, while `subagent_ended` emission depends on registry state.

## Solution Assessment

No. Registering at the plugin SDK adapter is the right boundary, but the implementation must skip cached Gateway replies and use the Gateway-accepted session key before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Skip cached agent replies before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: When a plugin retries `api.runtime.subagent.run()` with the same idempotency key, the Gateway can return a cached `status: "in_flight"` response for an existing run. This code still calls `registerSubagentRun`, and the registry overwrites `runs.set(runId, entry)` and starts another completion wait for a run that was not newly spawned. Please register only fresh accepted runs and add retry coverage.
  - confidence: 0.9
- **[P1] Use the Gateway-accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: The Gateway `agent` method returns the resolved `sessionKey` in the accepted payload after alias/global canonicalization, but this registers the caller's raw `params.sessionKey`. If the Gateway normalizes the key, `subagent_ended`, cleanup, and readback state can attach to the wrong session. Please register `payload.sessionKey` with a focused canonicalization regression test.
  - confidence: 0.86
