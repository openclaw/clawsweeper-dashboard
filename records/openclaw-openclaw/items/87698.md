## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Skip cached agent replies before registering:** `src/gateway/server-plugins.ts:594-600`
  - body: `dispatchGatewayMethod` unwraps both fresh `accepted` replies and cached/in-flight dedupe replies, then this call registers every returned `runId`. The Gateway explicitly returns cached `in_flight` responses for idempotent retries, and `registerSubagentRun` overwrites the run record for that id, so a retry can reset lifecycle state and schedule another completion wait instead of leaving the existing child run alone. Use the raw response metadata/status and register only a newly accepted Gateway run.
  - confidence: 0.9
- **[P1] Use the Gateway-accepted child session key:** `src/gateway/server-plugins.ts:596`
  - body: The Gateway's accepted payload includes the resolved `sessionKey`, which can differ from the caller's requested key after routing or canonicalization. Registering `params.sessionKey` makes hook events, cleanup, and transcript retention point at the wrong child session for those cases; carry `payload.sessionKey` through when present and test a canonicalized-key response.
  - confidence: 0.86
