## Summary

Keep open: the hardening is valid security work, but this branch applies it to stale `src/web` files that current main no longer uses, while the active WhatsApp plugin credential reads still follow symlinks and the PR lacks inspectable real behavior proof.

## What This Changes

The branch adds `lstat`-based symlink rejection and auth-directory mode/chmod hardening to WhatsApp credential/session files under `src/web`.

## Best Possible Solution

Retarget the hardening to `extensions/whatsapp` credential reads and auth-dir creation, using existing filesystem safety helpers where appropriate and adding focused symlink and permission regression tests.

## Reproduction Assessment

Yes. Source inspection plus a Node filesystem probe show current active readers use `stat`/`readFile` on `creds.json`, and Node follows a symlink there while `lstat` detects it; no full WhatsApp live session was run.

## Solution Assessment

No. The helper behavior is reasonable, but applying it under removed `src/web` files is not the maintainable fix; the checks need to cover the active `extensions/whatsapp` paths.
