## Summary

Keep open: the security hardening intent is valid, but the branch edits old `src/web` WhatsApp files that current `main` no longer uses, so the active WhatsApp credential readers remain unprotected.

## What This Changes

The PR adds lstat-based symlink rejection and `0o700` auth-directory hardening for WhatsApp credential handling in old `src/web` files.

## Best Possible Solution

Port the hardening into `extensions/whatsapp`, centralize credential-file reads around an lstat-based helper, harden active auth-directory creation, and add focused symlink regression coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: place a valid JSON symlink at the active `creds.json` path and the current `stat`/`readFile` readers will follow it. I did not run that filesystem repro because this review is read-only.

## Solution Assessment

No. The PR solves the right hardening problem in the wrong location; the maintainable fix belongs in the active `extensions/whatsapp` credential and session paths with regression tests.
