## Summary

Keep open: the hardening goal is valid, but this PR edits stale `src/web` WhatsApp files after the channel moved to `extensions/whatsapp`; the active credential readers and session directory setup on current main remain unchanged, and the external PR still lacks real behavior proof.

## What This Changes

This PR adds lstat-based symlink rejection and auth-directory mode/chmod hardening to WhatsApp credential/session code under `src/web`.

## Best Possible Solution

Retarget the hardening to the active `extensions/whatsapp` credential helpers and session setup, with focused symlink and directory-mode regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live run: a symlinked `creds.json` pointing at a regular file passes the active `fs.stat`/`statSync` checks, and Node documents that `stat` follows symlinks. No live WhatsApp session was exercised.

## Solution Assessment

No. The PR's helper is reasonable in isolation, but applying it under removed `src/web` paths is not the maintainable fix; the checks should be shared in the active WhatsApp plugin.
