## Summary

Keep open: the hardening goal is reasonable, but the PR changes stale `src/web` WhatsApp files while current main runs the WhatsApp implementation from `extensions/whatsapp`, where the credential readers and auth-directory setup remain unchanged. It also still lacks contributor-supplied real behavior proof for the after-fix symlink behavior.

## What This Changes

The PR adds lstat-based symlink rejection and auth-directory `0o700`/`chmod` hardening to old `src/web` WhatsApp credential/session files.

## Best Possible Solution

Retarget the hardening to `extensions/whatsapp` credential reads and auth-dir creation, reuse existing filesystem safety helpers where practical, and add focused symlink and permission regression tests.

## Reproduction Assessment

Yes. Source inspection shows the active `extensions/whatsapp` readers call `stat`/`readFile`/`readFileSync` on `creds.json` without an `lstat` guard, so the current-main path is source-reproducible even though I did not run a live WhatsApp session.

## Solution Assessment

No. The desired hardening is appropriate, but applying it under stale `src/web` files is not the maintainable fix; the active plugin paths need the checks and tests.
