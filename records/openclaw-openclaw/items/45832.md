## Summary

Keep open: the security hardening direction is useful, but this PR edits stale `src/web` WhatsApp files while current `main` serves WhatsApp from `extensions/whatsapp`, where the credential-read paths remain unguarded.

## What This Changes

The PR adds symlink rejection and `0o700` auth-directory handling to the old `src/web` WhatsApp auth-store/session files.

## Best Possible Solution

Port the hardening into `extensions/whatsapp`, with a shared credential-file read helper that rejects symlinked primary and backup files, directory mode enforcement in the active session path, and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection of current `main` shows `stat`/`readFile` paths that would follow a symlinked `creds.json`; a focused test can symlink `creds.json` to valid JSON and assert linked state, identity, and age reads treat it as missing.

## Solution Assessment

No. The PR solves the right problem in the wrong location because the active WhatsApp plugin now lives under `extensions/whatsapp`, not `src/web`.
