## Summary

Keep open: the hardening is useful, but the branch still changes stale `src/web` WhatsApp files while current `main` runs WhatsApp credential/session code from `extensions/whatsapp`, so the shipped credential reads and auth-directory creation remain unhardened. The external PR also still lacks after-fix real behavior proof.

## What This Changes

The branch adds symlink checks and `0o700` mkdir/chmod hardening to legacy `src/web` WhatsApp credential and session files.

## Best Possible Solution

Retarget symlink rejection and auth-directory mode hardening to `extensions/whatsapp` credential readers and session creation, with focused regression tests and redacted after-fix proof.

## Reproduction Assessment

Yes, at source level: active WhatsApp helpers still stat and read `creds.json` without first rejecting a final-path symlink. I did not run a live WhatsApp login/session in this read-only review.

## Solution Assessment

No. The `lstat`/`chmod` direction is reasonable, but applying it to `src/web` is not the maintainable fix for current users because the active implementation now lives under `extensions/whatsapp`.
