## Summary

Keep open. The security hardening is still relevant on current `main`, but this PR is stale: it modifies the old `src/web` WhatsApp files while the active credential code now lives under `extensions/whatsapp`. The prior review discussion was useful, and the author addressed the directory chmod note on the branch, but the branch still needs a current plugin-side port before it can be landable.

## What This Changes

The PR adds lstat-based symlink rejection around WhatsApp credential reads and auth-directory `0o700` chmod hardening in the old `src/web` WhatsApp auth-store/session files.

## Best Possible Solution

Port the hardening into `extensions/whatsapp`: add a shared credential-file read/inspection helper that rejects symlinked primary and backup credential files, use it from presence/status/identity/age/restore paths, enforce `0o700` plus best-effort chmod for the auth directory, preserve the current atomic save path, and add focused regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to place a symlink at `creds.json` pointing to a regular JSON credential file; current `main` source inspection shows the active plugin's `stat`/`readFile` paths would follow it. I did not execute tests because this review was read-only.

## Solution Assessment

No. The requested security direction is sound, but this branch is not the best implementation because it hardens removed `src/web` files rather than the current `extensions/whatsapp` plugin paths.
