## Summary

Keep open: the hardening goal is valid, but the patch is applied to stale `src/web` files while current `main` reads and creates WhatsApp credentials under `extensions/whatsapp`, so it would not protect active sessions.

## What This Changes

The PR adds `lstat`-based symlink rejection and `0o700`/`chmod` auth-directory hardening to WhatsApp credential/session code under `src/web`.

## Best Possible Solution

Retarget the hardening to `extensions/whatsapp` credential reads and auth-dir creation, reuse existing filesystem safety helpers where practical, and add focused symlink and permission regression tests.

## Reproduction Assessment

Yes. Source inspection plus a focused Node filesystem probe show the active readers use `stat`/`readFile` on `creds.json`, and Node follows symlink targets there; no live WhatsApp session was run.

## Solution Assessment

No. The desired hardening is reasonable, but applying it under removed `src/web` files is not the maintainable fix; the checks need to cover the active `extensions/whatsapp` paths.
