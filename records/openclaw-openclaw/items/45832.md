## Summary

Keep open: the hardening direction is valid, but this branch now targets stale `src/web` WhatsApp files while current main's active WhatsApp plugin lives under `extensions/whatsapp` and still has primary credential read paths without symlink rejection. The PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds lstat-based symlink rejection and 0o700/chmod auth-directory hardening to legacy `src/web` WhatsApp credential paths.

## Best Possible Solution

Retarget the hardening to `extensions/whatsapp` credential readers and `createWaSocket` auth-directory creation, reuse the existing secret-file safety pattern or explicit `lstat` guards, and add focused regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main's active WhatsApp plugin reads and stats `creds.json` through paths that follow symlinks. I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No. The proposed checks are directionally right, but landing them under stale `src/web` files is not the maintainable fix for current users.
