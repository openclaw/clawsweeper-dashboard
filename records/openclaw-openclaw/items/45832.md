## Summary

Keep this PR open. The hardening is still relevant, but current main does not implement the requested WhatsApp credential-file symlink rejection or auth-directory mode hardening in the active plugin code, and this branch targets removed `src/web` paths.

## What This Changes

The PR adds lstat-based symlink rejection around WhatsApp credential reads in the former `src/web` auth store and changes session auth-directory creation to mkdir/chmod `0o700`.

## Best Possible Solution

Port the hardening into the active WhatsApp plugin: add a shared credential-file inspection/read helper in `extensions/whatsapp/src/creds-files.ts` or `auth-store.ts`, use it before all primary and backup credential reads, enforce `0o700` plus best-effort chmod for the auth directory in `session.ts`, preserve the current atomic save path, and add focused regression tests for primary symlink reads, backup restore, identity/status/age reads, and directory mode behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to put a symlink at `creds.json` pointing to a regular JSON credential file; current main's `stat`/`readFile` based paths follow it, as shown by the source inspection, though I did not execute tests because this review was read-only.

## Solution Assessment

No. The intended security direction is sound, but this PR is no longer the best implementation because it hardens removed `src/web` files instead of the current `extensions/whatsapp` plugin paths.
