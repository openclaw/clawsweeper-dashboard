## Summary

Keep this PR open, but do not merge it as-is: the hardening goal is valid, while the branch targets stale `src/web` WhatsApp files that current `main` no longer ships, leaving the active WhatsApp plugin paths unprotected.

## What This Changes

The PR adds lstat-based symlink rejection and `0o700` auth-directory hardening to the old `src/web` WhatsApp auth-store/session files.

## Best Possible Solution

Port the hardening into `extensions/whatsapp`: centralize credential-file reads behind a helper that rejects symlinked primary and backup files, enforce `0o700` plus best-effort chmod for the auth directory, preserve the current atomic save path, and add focused regression tests.

## Reproduction Assessment

Yes. Source inspection of current `main` shows primary WhatsApp credential reads use `stat`/`readFile`, which follow a symlinked `creds.json`; a focused regression can symlink `creds.json` to valid JSON and assert auth presence, identity, and age reads treat it as missing.

## Solution Assessment

No. The security direction is sound, but this branch is not the best implementation because it hardens removed `src/web` files instead of the active `extensions/whatsapp` plugin paths.
