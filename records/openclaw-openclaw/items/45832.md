## Summary

Keep open: the hardening goal is valid, but this branch still patches deleted legacy `src/web` files while current WhatsApp credential/session code lives under `extensions/whatsapp`, where the active read paths still follow symlinks. The external PR also lacks after-fix real behavior proof.

## What This Changes

The PR adds lstat-based symlink rejection around WhatsApp credential reads and sets the auth directory to `0o700`/chmod in legacy `src/web` WhatsApp files.

## Best Possible Solution

Retarget symlink rejection and auth-directory mode hardening to the active `extensions/whatsapp` credential readers and session setup, with focused regression tests and redacted real behavior proof.

## Reproduction Assessment

Yes, at source level: active `extensions/whatsapp` helpers still use `stat`/`readFile` on `creds.json` without rejecting a final-path symlink. I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No. The `lstat`/chmod direction is reasonable, but applying it to deleted legacy `src/web` files cannot fix current WhatsApp behavior; the fix belongs in `extensions/whatsapp`.
