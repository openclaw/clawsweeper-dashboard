## Summary

Keep open: the hardening goal is valid, but this PR changes stale `src/web` WhatsApp files while current `main` runs WhatsApp auth from `extensions/whatsapp`, where the credential read paths still lack a symlink guard. The PR also still needs contributor-supplied real behavior proof before merge.

## What This Changes

The PR adds `lstat` symlink checks and `0o700`/`chmod` auth-directory hardening to old WhatsApp credential/session files under `src/web`.

## Best Possible Solution

Retarget the hardening to `extensions/whatsapp` credential reads and auth-directory creation, reuse existing filesystem safety helpers where practical, and add focused symlink and permission regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows the active WhatsApp plugin still uses `stat`/`readFile`/`readFileSync` paths for `creds.json` without an `lstat` guard, although I did not run a live WhatsApp session.

## Solution Assessment

No. The desired hardening is appropriate, but applying it under stale `src/web` files is not the maintainable fix; the active plugin paths need the checks and tests.
