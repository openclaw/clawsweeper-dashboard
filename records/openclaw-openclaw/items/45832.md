## Summary

Keep open: the security hardening goal is valid, but this branch still targets legacy `src/web` WhatsApp files while current main serves WhatsApp from `extensions/whatsapp`, so the shipped credential reads and auth-directory creation remain unchanged; the external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR adds lstat-based symlink guards and auth-directory mkdir/chmod hardening to legacy `src/web` WhatsApp credential and session files.

## Best Possible Solution

Retarget symlink rejection and auth-directory mode hardening to the active `extensions/whatsapp` credential readers and `createWaSocket`, using the existing secret-file/fs-safe pattern where practical, with focused regression coverage and redacted after-fix proof.

## Reproduction Assessment

Yes, at source level: current main's active WhatsApp helpers read and stat `creds.json` without an `lstat` guard, so a symlinked credential path would be followed. I did not run a live WhatsApp login/session in this read-only review.

## Solution Assessment

No. The proposed checks are directionally right, but applying them to stale `src/web` files is not the maintainable fix for current users; the same hardening needs to move to `extensions/whatsapp`.
