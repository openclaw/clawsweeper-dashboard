## Summary

Keep this PR open, but it should not merge as-is. The hardening is useful, yet the branch still patches stale `src/web` files while current main runs WhatsApp credential/session code from `extensions/whatsapp`, so the proposed security change would not protect the shipped path.

## What This Changes

The PR adds symlink rejection around old WhatsApp credential-file reads and changes old auth-directory creation to `0o700` plus best-effort `chmod`.

## Best Possible Solution

Retarget the hardening to `extensions/whatsapp` credential reads and auth-directory creation, reuse the existing filesystem safety helpers where practical, and add focused symlink and directory-permission regression coverage.

## Reproduction Assessment

Yes for the hardening gap: source inspection shows the active WhatsApp plugin still uses `stat`/`readFile`/`readFileSync` on `creds.json` without an `lstat` guard. I did not run a live WhatsApp session in this read-only review.

## Solution Assessment

No. The desired hardening is reasonable, but applying it under stale `src/web` files is not the maintainable fix; it needs to land in the active `extensions/whatsapp` paths with tests.
