## Summary

Keep open. Current main still rejects self-hosted Firecrawl baseUrl values, so the PR addresses a valid remaining behavior gap, but the proposed endpoint-mode split still has a blocking SSRF trust-boundary issue and leaves the published Firecrawl docs contradicting the new behavior. Related Firecrawl self-hosting work is also still active, so a maintainer/security review should choose the canonical path rather than closing this PR as stale or implemented.

## What This Changes

The PR changes Firecrawl endpoint validation to allow custom HTTPS baseUrl values and private or loopback HTTP baseUrl values, selects strict versus trusted web-tools fetch mode, updates Firecrawl tests, and adds a changelog entry.

## Best Possible Solution

Land one canonical Firecrawl plugin change that keeps the default and public endpoints on strict SSRF handling, permits explicit self-hosted HTTPS endpoints, allows cleartext HTTP only for private or loopback targets using fetch-time pinned resolution or an explicit operator opt-in, and updates Firecrawl docs, tests, and changelog before retiring sibling PRs.

## Reproduction Assessment

Yes. Current main can be verified statically: local HTTP Firecrawl baseUrl values fail the protocol check, and custom HTTPS hosts fail the api.firecrawl.dev allowlist; #63877 gives the matching user reproduction.

## Solution Assessment

No, not as currently proposed. The feature direction is right, but trusted mode should be derived from the same pinned resolution used by the actual guarded fetch or from an explicit private self-hosting opt-in, and the Firecrawl docs need to change with the behavior.

  - confidence: 0.84

  - confidence: 0.78
