## Summary

Keep this open. Current main partially addresses the argument-level surface with `argPattern`, but the broader security request still has meaningful remaining work: cross-policy design, deny/generalized rules, approval-log learning, user-confirmed pattern proposals, and related UX/config semantics are not implemented or owned by a single merged change.

## What This Changes

Request a progressive exec permission model that moves from binary-only allow-always approvals to argument-scoped matches and later user-confirmed generalized rules inferred from approval history.

## Best Possible Solution

Keep this as an umbrella security/design tracker until maintainers either define one progressive permission contract or split the remaining exact-match, denylist, path-scope, audit-log, and pattern-proposal work into linked canonical issues.

## Reproduction Assessment

Yes for the current-state check. Source inspection on current main shows optional `argPattern` support exists, but generalized allow-always argument persistence, approval-log pattern mining, user-confirmed rule proposals, and native deny/generalized-rule policy surfaces are not implemented.

## Solution Assessment

Unclear for implementation, but keeping this open is the right cleanup outcome. The requested direction is plausible, yet the best solution needs maintainer security/product design before a narrow code change is safe.
