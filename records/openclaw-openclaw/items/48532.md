## Summary

Keep open. Current main has partial exec-approval hardening, but POSIX allow-always still persists and matches ordinary commands by executable path rather than binary plus argv, and the requested deny-over-allow rules, dry-run matcher, approval-log clustering, and progressive rule proposal workflow are not implemented.

## What This Changes

Requests generalizing exec allow-always from binary-only trust to a progressive binary-plus-argument permission model with explicit user-confirmed pattern generalization.

## Best Possible Solution

Keep this as a maintainer-owned exec-approval policy design: define a cross-platform binary+argv rule format, deny-over-allow full-command matching, a dry-run matcher, approval/audit history, migration behavior for existing path-only entries, and an opt-in rule-generalization workflow that proposes scoped patterns only after explicit user confirmation.

## Reproduction Assessment

Yes. The current-main static reproduction is high confidence: a POSIX allowlist entry with an argPattern is accepted on path match without argv evaluation, and allow-always derives no argPattern outside Windows. I did not run tests because this review is read-only.

## Solution Assessment

No current implementation is the best complete solution yet. Exact-match allow-always is a useful first phase, but the maintainable fix needs maintainer design across policy semantics, compatibility, dry-run validation, and UX for pattern generalization.
