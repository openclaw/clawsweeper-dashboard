## Summary

Current main partially addresses the concern with `argPattern` matching and interpreter allow-always hardening, but it does not implement the requested progressive permission model or rule-generalization workflow. This should stay open for maintainer security/product design.

## What This Changes

Request progressive exec permissions so allow-always can match binary plus arguments and later propose scoped generalized rules from repeated approvals.

## Best Possible Solution

Define a versioned exec-permission design that keeps existing `argPattern` support, decides how durable allow-always should bind arguments, and separates learned rule proposals from denylist/path-policy work before implementation.

## Reproduction Assessment

Yes, by source inspection rather than a live run. Current main can still represent path-only durable trust, while the exact Python interpreter example is partly mitigated by interpreter allow-always hardening, so this is a source-reproducible design gap rather than a fully reproduced runtime bug.

## Solution Assessment

Unclear as a single implementation. Exact argument matching is a plausible incremental step, but approval-log mining and generalized regex rules need maintainer security and UX decisions before a safe PR.
