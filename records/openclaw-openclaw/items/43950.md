## Summary

This should stay open because current main still treats `skills.search` as ClawHub-only while local skills are exposed through inventory/list/status paths, so the requested local search/register behavior remains a product/API feature decision.

## What This Changes

The issue requests a way to register locally authored skills so they become discoverable through the skill search surface.

## Best Possible Solution

Define a local skill discovery contract, likely by reusing `loadWorkspaceSkillEntries` or `buildWorkspaceSkillStatus` behind either a scoped `skills.search` mode or a separate local search endpoint while keeping ClawHub registry semantics clear.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs give a high-confidence current-main check: local skills load for prompt/status/list surfaces, but `skills.search` and `openclaw skills search` still query ClawHub only.

## Solution Assessment

Unclear. The requested direction may be right, but current contracts deliberately separate local inventory from registry discovery, so maintainers should choose the API/UX boundary before implementation.
