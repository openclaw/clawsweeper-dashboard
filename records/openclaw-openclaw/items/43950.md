## Summary

Keep open: current main still separates local skill loading from search. Local skills are loaded for prompt/status/list surfaces, but `skills.search` and `openclaw skills search` are ClawHub-only today, so the requested local search/index capability remains a feature/API decision rather than an implemented bug fix.

## What This Changes

Add a way to register or auto-index locally authored skills so they are discoverable through the skill search surface.

## Best Possible Solution

Define a source-aware local skill discovery path that reuses `loadWorkspaceSkillEntries`, keeps ClawHub and local results distinguishable, and updates gateway protocol, CLI/UI behavior, docs, and regression tests together.

## Reproduction Assessment

Not applicable as a bug reproduction because the current docs define search as ClawHub-backed. Source inspection still gives high confidence that local skills under `~/.openclaw/skills` will not appear in `skills.search` today.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrowest maintainable fix is probably source-aware local search over existing workspace skill entries, but the requested CLI register, auto-scan, and API endpoint options have different protocol and UX implications.
