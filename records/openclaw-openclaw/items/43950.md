## Summary

Keep open: current main still has local skill inventory and ClawHub-backed search as separate paths, so local skills under managed/workspace/agent roots are not registered into the `skills.search`/`openclaw skills search` result set. The related runtime-discovery issue overlaps but does not supersede this narrower local-indexing request, and the remaining shape needs maintainer product/API direction.

## What This Changes

Request a way to register or auto-index locally authored skills so they are discoverable through the skill search surface.

## Best Possible Solution

Define an additive local skill discovery/search contract that reuses existing local loaders, preserves agent visibility rules, and clearly separates local inventory from ClawHub install semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high confidence that local skills are loaded for inventory/status/prompt flows while search still delegates to ClawHub only.

## Solution Assessment

Unclear until maintainers choose the product/API shape. CLI registration, auto-scan, API registration, or source-filtered search have different protocol, metadata, trust, and agent-scope tradeoffs.
