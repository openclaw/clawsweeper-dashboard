## Summary

Keep open. Current main still has the reported gap: local skills are loaded for runtime/status/list surfaces, but the search/discovery surfaces remain registry-scoped and there is no local registration API or CLI command. The earlier ClawSweeper comment reached the same conclusion, and the current code still supports that finding.

## What This Changes

Request a way to register or index locally authored skill directories so they become discoverable through skill search.

## Best Possible Solution

Add a source-aware discovery contract that can surface local skill entries from the existing loaders alongside registry results, or add an explicit local registration flow with equivalent search visibility. The endpoint, CLI, Control UI, docs, and tests should keep local skill paths distinct from ClawHub install/update/detail behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: local skills are loaded by `loadWorkspaceSkillEntries` and shown by status/list paths, while `skills.search` and `openclaw skills search` are hardwired to ClawHub search.

## Solution Assessment

Unclear until maintainers pick the product/API shape. The narrowest maintainable direction appears to be extending `skills.search` with source-aware local results from the existing loader rather than inventing a separate mutable index without clear path and source semantics.

## Risks / Open Questions

- Combining local and ClawHub search results without a source discriminator could make local names or paths look like ClawHub slugs for detail, install, or update flows.
- A register or auto-scan design needs explicit path trust, realpath containment, precedence, allowlist, and refresh semantics before implementation.
- The issue body mentions `_meta.json`, while current skill docs and loaders center on `SKILL.md` frontmatter, so the accepted metadata source needs to be clarified.
