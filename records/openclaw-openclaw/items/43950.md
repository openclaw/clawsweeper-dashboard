## Summary

Keep open. Current main still has the reported gap: local skills are loaded for runtime/status/list paths, but search/discovery remains ClawHub-scoped and needs maintainer API/security direction before implementation.

## What This Changes

The issue requests a way to register or auto-index locally authored skill directories so they become discoverable through skill search.

## Best Possible Solution

Define an additive source-aware discovery contract that surfaces local loader results alongside ClawHub results while keeping local entries distinct from registry detail/install/update flows, then update CLI, Gateway protocol, Control UI, docs, and regression tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: local skill roots are loaded by `loadWorkspaceSkillEntries`, while `skills.search`, `openclaw skills search`, and Control UI search call only the ClawHub search path.

## Solution Assessment

Unclear until maintainers choose the product/API shape. The narrowest maintainable direction appears to be extending existing search with source-aware local results rather than adding a separate mutable index with ambiguous trust and precedence semantics.
