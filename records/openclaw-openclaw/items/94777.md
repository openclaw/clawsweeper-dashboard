## Summary

Keep open: the current head fixes the earlier unused-constant and missing-test problems, but its blanket pre/code shielding can restore inline code markup from inside converted table cells into the generated `<pre><code>` fallback, creating nested code HTML instead of plain table text.

## What This Changes

The PR changes Telegram standard HTML rendering to normalize HTML tables into Telegram-compatible pre/code pipe tables before unsupported-tag escaping, with formatter regression tests.

## Best Possible Solution

Use a formatter approach that only skips tables whose table start is inside pre/code, lets table-cell contents flatten through the existing parser, preserves rich-message native tables, and lands one canonical PR for the linked issue.

## Reproduction Assessment

Yes. Current main sends standard HTML text through unsupported-tag escaping before any table fallback, and the linked issue describes raw table tags arriving as escaped entities; the PR body and Mantis screenshots show the intended readable fallback path.

## Solution Assessment

No. The PR is in the right Telegram formatter boundary, but the blanket pre/code shielding is not the best fix because it also shields inline code inside table cells before the table parser can flatten it.
