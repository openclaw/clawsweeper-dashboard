## Summary

Keep this PR open: it targets the right Telegram formatter boundary and has useful proof, but the new pre/code skip is substring-based and can still leave ordinary tables on the escaped-literal path when earlier unsupported tags merely start with `<pre` or `<code`. A narrow repair should make the context check tag-aware before maintainers pick the canonical fix for the linked Telegram table issue.

## What This Changes

The PR changes Telegram standard HTML rendering so raw `<table>` elements are converted to `<pre><code>` pipe-table fallbacks before unsupported-tag escaping, and adds formatter tests for table conversion and pre/code preservation.

## Best Possible Solution

Land one canonical Telegram formatter fix that uses tag-aware pre/code detection, preserves rich-message sanitization, and closes the linked Telegram table issue only after merge.

## Reproduction Assessment

Yes. Current main's standard HTML renderer sends raw tables directly to unsupported-tag escaping, and the PR proof plus Mantis screenshots show the intended readable fallback; the remaining edge is source-reproducible from the substring pre/code guard.

## Solution Assessment

No, not yet. Moving the fix into `renderTelegramHtmlText` is the right boundary, but the pre/code exclusion should use the same tag-aware parsing style as nearby formatter helpers before this is the best fix.
