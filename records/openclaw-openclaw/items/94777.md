## Summary

Keep open: the PR targets a real Telegram table rendering problem, but it applies the conversion in the rich-message sanitizer, which would downgrade opt-in native rich tables and does not fix the default legacy HTML delivery path.

## What This Changes

The branch changes `extensions/telegram/src/format.ts` so `sanitizeTelegramRichHtml` converts every raw HTML table to a `<pre><code>` pipe table before unsupported rich HTML escaping.

## Best Possible Solution

Move raw table fallback to the legacy Telegram HTML/Markdown delivery path while preserving `richMessages: true` native tables, add coverage for both modes, and prove final delivery in Telegram before choosing one linked PR to land.

## Reproduction Assessment

Yes for the PR defect by source inspection: the changed sanitizer is used by rich-message delivery, while existing tests and docs preserve native rich tables there. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The useful fix belongs on the default Telegram HTML/Markdown delivery path while preserving the opt-in rich-message table contract, not by converting every table inside `sanitizeTelegramRichHtml`.
