## Summary

Keep open. The PR targets the right standard Telegram HTML rendering path, but the current branch still normalizes every raw `<table>` before the formatter can respect `<pre>`/`<code>` regions, so existing literal HTML examples can be rewritten instead of displayed as code.

## What This Changes

The branch changes `extensions/telegram/src/format.ts` so `renderTelegramHtmlText(..., { textMode: "html" })` converts raw HTML tables into `<pre><code>` pipe tables before unsupported-tag escaping while leaving the rich-message sanitizer unchanged.

## Best Possible Solution

Land one canonical Telegram fix that converts raw standard HTML tables to readable fallback tables, preserves rich native tables and code/pre literals, and covers those cases with focused formatter tests.

## Reproduction Assessment

## Solution Assessment

No, not yet. Moving the fix to `renderTelegramHtmlText` is the right boundary, but the whole-string regex needs to skip code/pre regions and gain focused regression coverage before merge.
