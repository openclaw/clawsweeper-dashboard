## Summary

Keep open. Current main and the latest release still classify only a single exact silent token as silent, so a token-only doubled payload falls through shared suppression and normalization can leave a visible `NO_REPLY`. The Quiubo plugin-side guard and the closed unmerged #69472 attempt are useful context, but neither fixes the shared OpenClaw core predicate on main.

## What This Changes

Make silent-reply suppression treat payloads made only of repeated whitespace-separated silent tokens as silent instead of leaking a visible token.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, while mixed-content replies and media-bearing payloads keep their existing behavior. Add focused regression coverage for repeated default and custom tokens plus the existing #19537 protections.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current regex behavior returns false for `NO_REPLY\n\nNO_REPLY`, and the current normalization path strips only one trailing token, leaving `NO_REPLY` as deliverable text.

## Solution Assessment

Yes. A narrow core fix in `src/auto-reply/tokens.ts` and its shared normalization/runtime-plan coverage is the best path because the docs and plugin harness contract make silent payload suppression host-owned; plugin-side guards are useful mitigation, not the shared fix.
