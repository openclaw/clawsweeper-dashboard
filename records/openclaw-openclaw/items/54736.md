## Summary

Keep open. Current main and the latest release still use exact-only silent-token matching, so `NO_REPLY\n\nNO_REPLY` is not classified as silent and normalization can leave one visible `NO_REPLY`; related open PRs overlap but have not landed a core fix.

## What This Changes

The issue requests that core silent-reply suppression treat repeated whitespace-separated `NO_REPLY` tokens as silent instead of leaking one to channel plugins.

## Best Possible Solution

Land one shared silent-token classifier that suppresses repeated and format-only token payloads while preserving substantive-message delivery, media handling, and focused regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible: the exact matcher rejects `NO_REPLY\n\nNO_REPLY`, and the current trailing-strip path can leave one `NO_REPLY` as deliverable text.

## Solution Assessment

Yes for the narrow issue. The maintainable fix belongs in the shared silent-token predicate or normalization contract, but maintainers should reconcile it with the overlapping open PRs before landing duplicate policy changes.
