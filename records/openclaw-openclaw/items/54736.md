## Summary

Keep open. Current main and the latest release still use exact-only silent-token matching, so a doubled token-only payload falls through shared suppression and can be normalized into a visible `NO_REPLY`; related open PRs cover broader own-line/reasoning policy and are not a merged core fix for this narrow bug.

## What This Changes

The issue requests that shared silent-reply suppression handle payloads made only of repeated whitespace-separated silent tokens such as `NO_REPLY\n\nNO_REPLY`.

## Best Possible Solution

Update the shared silent-token predicate or equivalent normalization contract so repeated whitespace-separated token-only payloads are suppressed, with focused regression coverage and no broader own-line/trailing-token policy change.

## Reproduction Assessment

Yes. Current main is source-reproducible: the exact matcher rejects `NO_REPLY\n\nNO_REPLY`, and the current trailing-strip path leaves one `NO_REPLY` as deliverable text.

## Solution Assessment

Yes. A narrow shared predicate fix is the best maintainable solution for this issue; broader own-line or reasoning-prefaced suppression should stay with the separate open policy PRs unless maintainers choose to combine them.
