## Label Justifications

- `P2`: This is a reproducible user-visible delivery bug in shared silent-reply handling, but the blast radius is bounded to malformed silent-token outputs.
- `impact:message-loss`: The issue is about the shared channel-delivery suppressor failing to suppress a token-only silent reply payload.

## Summary

Keep open. Current main and the latest release still use an exact single-token silent-reply classifier, so `NO_REPLY\n\nNO_REPLY` is not classified as silent and can normalize down to a visible `NO_REPLY`; the active related PR may cover the same classifier surface, but it has not merged or closed this issue.

## What This Changes

Suppress silent-reply payloads made only of repeated `NO_REPLY` tokens instead of allowing doubled token-only output to reach channel delivery.

## Best Possible Solution

Land or split a core classifier fix that suppresses token-only repeated and wrapped silent markers, with explicit regressions for newline-doubled tokens, markdown wrappers, custom tokens, media payloads, and https://github.com/openclaw/openclaw/issues/19537 preservation.

## Reproduction Assessment

Yes. Source inspection and a read-only regex check against current main show `NO_REPLY\n\nNO_REPLY` misses exact suppression and strips to a still-visible `NO_REPLY`.

## Solution Assessment

Yes for the narrow doubled-token scope. The maintainable fix belongs in the shared silent-token classifier or normalization contract, coordinated with open PR https://github.com/openclaw/openclaw/pull/73153 to avoid duplicate behavior.
