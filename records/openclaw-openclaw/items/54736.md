## Summary

Keep open. Current `main` and `v2026.5.19` still use exact-only silent-token matching, so `NO_REPLY\n\nNO_REPLY` misses shared suppression and can be normalized into a visible `NO_REPLY` before channel delivery.

## What This Changes

The issue asks OpenClaw core to suppress repeated whitespace-separated silent tokens such as `NO_REPLY\n\nNO_REPLY` without suppressing substantive replies.

## Best Possible Solution

Update the shared silent-token predicate or normalization contract so repeated silent-token-only payloads are suppressed, with focused regression tests for default and custom tokens plus substantive-reply preservation.

## Reproduction Assessment

Yes, at source level. Current `main` still uses the exact-token regex and a read-only regex check shows `NO_REPLY\n\nNO_REPLY` misses exact suppression and strips to a visible `NO_REPLY`.

## Solution Assessment

Yes for the doubled-token bug. The narrow maintainable fix belongs in the shared classifier or normalization path, with tests that preserve the existing substantive-reply behavior from the earlier false-positive fix.
