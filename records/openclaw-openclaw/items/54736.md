## Summary

Keep open. Current main and the latest release still use exact-only silent-token classification, so a doubled token-only payload is not suppressed by the shared predicate and can be normalized down to visible `NO_REPLY`; the directly matching fix PR was closed unmerged, and no open canonical item now owns this exact doubled-token bug.

## What This Changes

The issue asks core auto-reply suppression to treat repeated whitespace-separated silent tokens, such as doubled `NO_REPLY`, as a silent-only reply without suppressing substantive text.

## Best Possible Solution

Extend the shared silent-token classifier or normalization contract so repeated whitespace-separated silent-token-only payloads suppress cleanly, while preserving delivery of substantive replies after trailing-token cleanup.

## Reproduction Assessment

Yes, at source level. Current main cannot match `NO_REPLY\n\nNO_REPLY` with the exact single-token regex, and the follow-up normalizer strips only the trailing token, leaving a visible `NO_REPLY`; I did not run tests because this review is read-only.

## Solution Assessment

Yes for the doubled-token bug. The narrow maintainable fix belongs in the shared classifier or normalization path used by channels and runtime delivery, with regression coverage that preserves the older substantive-reply behavior.
