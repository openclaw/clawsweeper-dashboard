## Summary

Keep open. Current main still uses exact-only silent-token matching, so `NO_REPLY\n\nNO_REPLY` is not classified as silent; the later normalization path strips only one trailing token and can leave a visible `NO_REPLY`. The Quiubo plugin-side guard is useful mitigation, and the related #69472 PR is useful prior art, but that PR is closed unmerged and the shared core predicate remains unfixed.

## What This Changes

The issue requests that `isSilentReplyText()` and related silent-reply handling suppress payloads made only of repeated whitespace-separated silent tokens such as doubled `NO_REPLY`.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, while mixed-content replies and media-bearing payloads keep their existing behavior. Add focused regression coverage for repeated default/custom tokens and the #19537 protection cases.
