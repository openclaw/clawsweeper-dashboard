## Summary

Keep open. Current `main` and `v2026.5.3` still use exact-only silent-token matching, so `NO_REPLY\n\nNO_REPLY` falls through shared suppression and can be normalized into a visible `NO_REPLY`; the Quiubo guard and closed unmerged #69472 are useful context but not a core fix.

## What This Changes

The issue asks OpenClaw core to suppress reply payloads made only of repeated silent tokens such as doubled `NO_REPLY`.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, while mixed-content replies and media-bearing payloads keep their existing behavior.

## Reproduction Assessment

Yes. The current-main source path is high-confidence: the exact matcher rejects `NO_REPLY\n\nNO_REPLY`, and the current trailing-strip path leaves one `NO_REPLY` as deliverable text.

## Solution Assessment

Yes. A narrow core fix in the shared silent-token predicate or normalization contract is the best maintainable path; plugin-side guards are useful mitigation, and broader reasoning/trailing-token suppression should stay separate unless maintainers choose it.
