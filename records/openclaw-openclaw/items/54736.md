## Summary

Keep open. Current main and the latest release still use exact single-token matching, so `NO_REPLY\n\nNO_REPLY` falls through shared suppression and can be normalized into a visible `NO_REPLY`; the open related PR targets reasoning-prefixed leaks, not this doubled-token report.

## What This Changes

The issue asks OpenClaw core to suppress silent-reply payloads made only of repeated whitespace-separated silent tokens instead of leaking `NO_REPLY` text to channel plugins.

## Best Possible Solution

Update the shared silent-token predicate or normalization contract to suppress token-only repeated and markdown-wrapped variants, with regression tests that preserve substantive-reply behavior.

## Reproduction Assessment

Yes. Source inspection plus a read-only regex check against the current matcher show `NO_REPLY\n\nNO_REPLY` misses exact suppression and strips down to a visible `NO_REPLY`.

## Solution Assessment

Yes. A narrow shared predicate/normalization fix is the maintainable path because plugin-side guards leave other channel plugins and runtime-plan consumers on the same host-owned policy.
