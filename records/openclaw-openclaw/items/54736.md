## Summary

Keep open. Current main and the latest release still classify silent replies with exact single-token matching, so repeated token-only payloads can miss shared suppression; the recent Quiubo workaround and markdown-wrapped token data are useful proof that plugin-side guards are only mitigation, not a core fix. Related open PRs target adjacent reasoning/trailing-token behavior but do not currently close this issue and choose different semantics.

## What This Changes

The issue asks OpenClaw to suppress repeated token-only silent replies such as `NO_REPLY\n\nNO_REPLY` instead of leaking a visible control token to channel plugins.

## Best Possible Solution

Land one shared silent-token classifier update with focused tests for repeated whitespace-separated tokens, markdown-wrapped token-only variants, custom tokens, and the existing substantive-reply false-positive guard.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: the exact matcher rejects `NO_REPLY\n\nNO_REPLY`, then normalization can strip one trailing token and leave a visible `NO_REPLY`; I did not run tests because this review was read-only.

## Solution Assessment

Yes, a narrow shared predicate or normalization-contract fix is the best direction for this issue. Solving it only in the Quiubo plugin would leave other plugins and runtime-plan consumers on the same broken host-owned policy.
