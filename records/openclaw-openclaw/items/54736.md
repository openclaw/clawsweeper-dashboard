## Summary

Keep open. Current main and the latest release still use exact single-token suppression, so doubled token-only payloads fall through the shared classifier and can normalize into a visible `NO_REPLY`; related open PRs cover overlapping reasoning or own-line policies but have not landed a canonical fix for this issue.

## What This Changes

The issue reports that repeated silent-token outputs such as `NO_REPLY\n\nNO_REPLY` bypass shared suppression and leak visible control text.

## Best Possible Solution

Update the shared silent-token predicate or normalization contract to suppress payloads made only of repeated and token-only wrapped silent markers, with regression tests that preserve substantive-reply behavior and custom-token handling.

## Reproduction Assessment

Yes. Source inspection plus a read-only regex check against the current matcher show `NO_REPLY\n\nNO_REPLY` misses exact suppression and strips down to a visible `NO_REPLY`.

## Solution Assessment

Yes for the narrow repeated-token case. The maintainable fix belongs in the shared token classifier or normalizer, while broader reasoning-preface or own-line policy changes should be coordinated with the existing related PRs.
