## Summary

Keep open. Current main and v2026.5.18 still use exact-only silent-token matching, so doubled token-only payloads can fall through suppression; the exact replacement PR was closed unmerged and the broader related PR has not landed.

## What This Changes

The issue asks OpenClaw core to suppress doubled `NO_REPLY`-only payloads such as `NO_REPLY\n\nNO_REPLY` before channel plugins deliver visible text.

## Best Possible Solution

Update the shared silent-token predicate or normalization contract so repeated whitespace-separated silent-token-only payloads are suppressed, with regression tests that preserve substantive-reply behavior.

## Reproduction Assessment

Yes, at source level. Current main still uses the exact-token regex and a read-only regex check shows `NO_REPLY\n\nNO_REPLY` misses exact suppression and strips to a visible `NO_REPLY`.

## Solution Assessment

Yes for the narrow doubled-token case. The maintainable fix belongs in the shared token classifier or normalization path, with tests for repeated default and custom tokens plus the existing substantive-reply protections.
