## Summary

Keep open. Current main and the latest v2026.5.18 release still use single-token exact suppression, and the read-only reproduction shows a doubled token-only payload falls through and can leave a visible `NO_REPLY`; related PRs are broader, open or unmerged, and do not close this exact issue.

## What This Changes

The issue reports that repeated token-only silent replies such as `NO_REPLY\n\nNO_REPLY` bypass shared suppression and can be delivered visibly by channel plugins.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, with regression tests that preserve substantive-reply and custom-token behavior.

## Reproduction Assessment

Yes. Current main's exact matcher returns false for `NO_REPLY\n\nNO_REPLY`, and the normalizer's trailing-strip path leaves a visible `NO_REPLY` in the read-only reproduction check.

## Solution Assessment

Yes for the narrow repeated-token bug. The maintainable path is to extend the shared silent-token classifier or normalizer with focused tests, not to add a broad `NO_REPLY`-anywhere suppression policy.
