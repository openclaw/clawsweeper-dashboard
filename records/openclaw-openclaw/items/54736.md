## Summary

Keep open. Current main and v2026.5.20 still use exact-only silent-token matching, so a doubled token-only payload falls through shared suppression and can leave a visible `NO_REPLY` after normalization; related PRs are closed or still open/unmerged, so the core bug is not implemented on main.

## What This Changes

The issue requests that shared silent-reply classification suppress repeated whitespace-separated `NO_REPLY` or custom silent-token payloads instead of leaking them to channel delivery.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so repeated whitespace-separated silent-token-only payloads suppress cleanly, with regression coverage for default and custom tokens while preserving the earlier substantive-reply behavior.

## Reproduction Assessment

Yes, at source level. Current main still uses the exact-token regex, and a read-only regex check matching the current exact and trailing patterns shows `NO_REPLY\n\nNO_REPLY` misses exact suppression and strips down to visible `NO_REPLY`.

## Solution Assessment

Yes for the doubled-token bug. The narrow maintainable fix belongs in the shared classifier or normalization contract, with tests that preserve the earlier substantive-reply behavior.
