## Summary

Keep open. Current main and the latest release still let an exact semver `--tag` become a dry-run `targetVersion` without proving npm has published that package, and two open closing PRs already own the candidate fixes.

## What This Changes

The issue reports that `openclaw update --dry-run --tag <version>` can approve a target that apply later rejects with npm `ETARGET` when the version has not propagated to the npm registry.

## Best Possible Solution

Land one canonical update CLI change that makes dry-run and apply share a confirmed-missing npm-target check while preserving non-registry specs, custom registry behavior, and transient network tolerance.

## Reproduction Assessment

Yes, from source inspection: current main resolves exact semver tags locally and returns from dry-run before install, while a live registry check for an absent OpenClaw version-like target returns HTTP 404. I did not run a mutating package-manager update in this read-only review.

## Solution Assessment

Partly. Checking confirmed-missing npm targets in update planning/preflight is the right layer, but automatic GitHub fallback would change update source semantics and should remain a maintainer product decision.
