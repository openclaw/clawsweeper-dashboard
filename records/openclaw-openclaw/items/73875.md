## Summary

Keep open. Current `main` still has the reported ACP runtime-control behavior, and the issue is paired with open, unmerged PR #74048 using closing syntax, so it should remain open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to stop sending unsupported generic runtime config options to Claude ACP adapters and instead send only adapter-advertised config options or degrade gracefully.

## Best Possible Solution

Keep the issue open until PR #74048 or a replacement lands. The desired end state is a capability-aware ACP runtime-control path that sends only explicitly advertised adapter config ids, maps generic OpenClaw concepts to adapter-specific ids where a real contract exists, and keeps generic run timeouts on OpenClaw's own turn-timeout path unless the backend advertises a compatible option.

## Reproduction Assessment

Yes. The issue gives concrete live steps, and current `main` has the static precondition: it builds generic ACP config keys and forwards them when the backend exposes `session/set_config_option` without a strict advertised key list; the pinned Claude adapter source rejects unknown config ids.

## Solution Assessment

Yes. Filtering by advertised adapter support is the narrow maintainable direction because it avoids a Claude-specific core special-case; the existing linked PR still needs normal review, tests, and landing before this issue can close.
