## Summary

Keep open. Current main still converts generic ACP runtime options into ACP config option IDs and forwards them when a backend supports `session/set_config_option` but does not advertise a strict key list; PR #74048 is open and unmerged with closing syntax, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to send only config options supported by the target adapter, avoiding Claude ACP failures from generic keys such as `timeout`, `thinking`, and `approval_policy`.

## Best Possible Solution

Make ACP runtime-control capabilities explicit, filter or gracefully skip unsupported config keys per target adapter, keep run timeouts in OpenClaw's turn-timeout path unless the adapter advertises a timeout config key, and add regression coverage for strict Claude-like backends plus undefined-key backend behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue gives a concrete live path, and current main shows the static precondition: `sessions_spawn` stores generic ACP runtime options, the manager maps them to ACP config IDs, and the acpx wrapper forwards non-Codex options such as `timeout` to the delegate.

## Solution Assessment

Is this the best way to solve the issue? Yes for the requested direction: filter runtime config options by explicit ACP backend support and avoid sending generic keys to strict adapters. The open PR remains the implementation candidate, but it still needs review because current capability docs treat missing `configOptionKeys` as permissive.
