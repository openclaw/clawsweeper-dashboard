## Summary

Keep open. Current main still builds and forwards generic ACP config keys when the backend does not advertise a strict key list, and the linked implementation PR #74048 is open and unmerged, so this should remain open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to avoid sending generic runtime config options to Claude ACP unless the target adapter advertises support for those config IDs.

## Best Possible Solution

Keep the issue open until PR #74048 or a replacement lands; the desired end state is a capability-aware ACP runtime-control path that sends only advertised adapter config ids, maps generic OpenClaw concepts only where a real adapter contract exists, and keeps generic run timeouts on OpenClaw's own turn-timeout path unless the backend advertises a compatible option.

## Reproduction Assessment

Yes. The issue gives concrete live steps, and current main has the static precondition: generic ACP keys are generated and forwarded when `configOptionKeys` is absent, while the pinned Claude ACP adapter rejects unknown config ids.

## Solution Assessment

Yes. Filtering by advertised adapter support is the narrow maintainable direction because it avoids a Claude-specific core special case; the linked PR still needs normal review, tests, and landing before this issue can close.
