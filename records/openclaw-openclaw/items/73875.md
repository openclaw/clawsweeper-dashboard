## Summary

Keep open. Current main still turns generic ACP runtime options into ACP config-option calls when a backend supports `session/set_config_option` but does not advertise a strict key list, and PR #74048 is the open, unmerged implementation candidate with closing syntax.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to avoid sending unsupported generic runtime config options such as `timeout`, `thinking`, and `approval_policy` to strict Claude ACP adapters.

## Best Possible Solution

Keep the issue open until PR #74048 or a replacement lands; the desired end state is an ACP runtime-control contract that applies config options only when the target backend explicitly supports them, while keeping generic run timeouts in OpenClaw's own timeout path unless an adapter advertises a compatible config key.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue provides a concrete live path, and current main shows the static precondition: generic ACP runtime keys are built and forwarded when `session/set_config_option` exists and no strict key list is advertised.

## Solution Assessment

Is this the best way to solve the issue? Yes. Filtering by adapter-advertised support, instead of special-casing Claude, is the narrow maintainable fix; the exact PR still needs normal review because it changes the meaning of missing `configOptionKeys`.
