## Summary

Keep open. Current main still converts OpenClaw runtime options into generic ACP config keys and forwards them whenever the backend supports `session/set_config_option` but has no strict advertised key list; the linked closing PR remains open and unmerged, so this issue should stay open until that PR or a replacement lands.  Item URL: https://github.com/openclaw/openclaw/issues/73875

## What This Changes

The issue asks OpenClaw-managed Claude ACP sessions to stop sending unsupported generic config keys such as `timeout`, `thinking`, and `approval_policy` unless the target ACP adapter advertises support for them.

## Best Possible Solution

Land a reviewed ACP runtime-control fix that derives or exposes adapter-supported config option IDs, sends only supported adapter keys, keeps OpenClaw run timeouts in the runtime timeout path unless the adapter advertises a timeout option, and preserves clear errors or warnings for unsupported permission-related controls. Close this issue only after PR #74048 or an equivalent replacement merges.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The issue gives a concrete live path, and current main shows the static precondition: `sessions_spawn` stores generic ACP runtime options, the manager maps them to ACP config IDs, and the acpx/Claude path can forward unknown keys to a strict adapter.

## Solution Assessment

Is this the best way to solve the issue? Yes for the requested direction: OpenClaw should filter or map config options by explicit adapter support. The exact implementation still needs care because the current no-key capability contract is permissive and `approval_policy` should not be silently ignored.
