## Summary

Keep open. Current main still sends generic ACP runtime config options when a backend supports `session/set_config_option` but does not advertise `configOptionKeys`, and the open, unmerged PR #74048 is already the closing implementation candidate for this report.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to apply only adapter-supported ACP config options, so strict adapters like Claude ACP are not sent generic keys such as `timeout`, `thinking`, or `approval_policy`.

## Best Possible Solution

Keep this issue open until PR #74048 or a replacement lands with tests that prevent strict ACP adapters from receiving unsupported generic OpenClaw config keys, while preserving clear error or warning behavior for advertised strict key lists and permission-related controls.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A focused source-level reproduction is a runtime that supports `session/set_config_option`, returns no `configOptionKeys`, and has persisted runtime options for `thinking`, `approval_policy`, or `timeout`; current main will call `setConfigOption` for each, and the reporter supplied a matching live Claude ACP path.

## Solution Assessment

Is this the best way to solve the issue? Yes in direction: a generic capability-based filter is preferable to special-casing Claude. The linked PR still needs review because it changes advertised-key mismatch behavior and may silently drop permission controls.
