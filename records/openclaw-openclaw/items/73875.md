## Summary

Keep open. Current main still constructs generic ACP config keys and forwards them when ACPX advertises `session/set_config_option` without `configOptionKeys`, and PR #74048 is the open, unmerged closing implementation candidate.

## What This Changes

OpenClaw-managed Claude ACP sessions should avoid sending generic runtime config keys that the Claude ACP adapter does not advertise or support.

## Best Possible Solution

Keep this issue open until PR #74048 or a replacement lands a capability-aware ACP runtime-control path that sends only advertised adapter config IDs and keeps generic runtime keys out of strict backend config calls.

## Reproduction Assessment

Yes. The report gives concrete live steps, and the current source/dependency path statically reproduces the failing precondition: generic keys are built, ACPX advertises the control without strict keys, and the Claude adapter rejects unknown IDs.

## Solution Assessment

Yes. Filtering or skipping by advertised adapter support is the narrow maintainable direction because it avoids a Claude-specific core special case while preserving explicit backend capability contracts.
