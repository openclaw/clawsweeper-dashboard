## Summary

Keep open. Current main still has the reported ACP runtime-control path: generic keys are built, applied before turns, and forwarded through the ACPX Claude path, while the open closing PR remains unmerged and an additional targeted timeout PR only covers part of the reported behavior.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to avoid sending unsupported generic config options such as timeout, thinking, and approval_policy to Claude ACP unless the adapter advertises compatible keys.

## Best Possible Solution

Review and land a focused ACP fix that keeps OpenClaw’s timeout watchdog internal, avoids unadvertised Claude ACP config keys, maps thinking to an advertised effort-style control when supported, and then close this after the merged commit is on main.

## Reproduction Assessment

Yes. Source inspection on current main shows sessions_spawn runTimeoutSeconds can become a timeout config pair and the ACPX Claude path forwards non-Codex timeout controls, while reporter comments add current-stack Claude ACP failures for timeout and thinking overrides.

## Solution Assessment

Yes for keeping this open, but no current open PR fully solves the whole report yet. The safest implementation should be capability-aware without redefining the existing missing-configOptionKeys contract and should handle both timeout filtering and Claude effort mapping.
