## Summary

Keep open. Current main still converts generic ACP runtime options into ACP config option IDs and forwards them whenever a backend supports `session/set_config_option` but does not advertise a strict key list. The linked PR is open and unmerged with closing syntax, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that OpenClaw-managed Claude ACP sessions fail because OpenClaw sends generic runtime config options such as `timeout`, `thinking`, and `approval_policy` to a strict Claude ACP adapter that only supports advertised options.

## Best Possible Solution

Make ACP runtime-control support explicit per backend, filter or gracefully skip unadvertised config keys, keep turn timeouts in OpenClaw's own timeout path unless the adapter advertises a timeout config option, and update the capability type/docs/tests along with the implementation candidate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The report provides a concrete live Claude ACP path, and current main statically satisfies the failure precondition by storing generic runtime options, converting them to ACP config IDs, and forwarding them when the backend has no advertised key list.

## Solution Assessment

Is this the best way to solve the issue? Yes for the requested direction: OpenClaw should not send generic config keys to strict ACP adapters unless support is advertised. The linked PR remains the implementation candidate, but its semantics should be checked against the existing permissive `configOptionKeys` contract.
