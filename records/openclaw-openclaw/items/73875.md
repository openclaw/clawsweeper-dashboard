## Summary

Keep open. Current main still has the reported ACP runtime-control behavior: generic runtime options are persisted, translated into ACP config option keys, and forwarded when a backend reports set_config_option without a strict key list. PR #74048 is an open, unmerged implementation candidate with closing syntax, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Make OpenClaw-managed ACP sessions send only adapter-supported config options, or gracefully skip unsupported generic runtime options, so Claude ACP sessions can start successfully.

## Best Possible Solution

The ACP runtime-control layer should treat supported config IDs as a real capability contract, skip or degrade unsupported generic keys for strict adapters, keep timeout as an OpenClaw execution timeout unless the adapter advertises it, and add regression coverage for strict-key and missing-key backends before closing this issue.

## Reproduction Assessment

Yes. The issue gives a concrete live reproduction path, and current main shows the static precondition: sessions_spawn stores generic runtime options, those options become ACP config IDs, and the manager forwards every pair when no strict key list is present.

## Solution Assessment

Unclear for the open PR until it receives its own review. The direction of filtering or skipping unsupported config options is correct, but the durable fix should align the runtime capability contract and include regression coverage before this issue is considered solved.
