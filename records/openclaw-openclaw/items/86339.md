## Label Justifications

- `P2`: The report describes a normal-priority WebChat session restoration bug with visible context loss, but the transcript is usually still recoverable on disk.
- `impact:session-state`: The affected behavior is active WebChat session context diverging from durable transcript/session state after restart, abort, or reset.
- `impact:message-loss`: The user-visible conversation can omit prior messages even though they remain present in `.jsonl.reset.*` transcript files.

## Summary

Keep open. Current main has improved WebChat history reload and reset-archive indexing, but source review still shows the active WebChat history path reads the current session transcript branch and does not provide a clear user-facing recovery path when useful context has moved into `.jsonl.reset.*` archives; the related open issues overlap, but this report adds distinct restart/reset/archive evidence.

## What This Changes

The issue reports WebChat/dashboard session history disappearing after restart, abort, or reset events even though older transcript content remains recoverable from `.jsonl.reset.*` files.

## Best Possible Solution

Preserve active WebChat context across restart/abort when no explicit reset occurred, and when a reset/new branch is intentional, surface an explicit previous-transcript recovery affordance instead of silently presenting a shorter active branch.

## Reproduction Assessment

No high-confidence end-to-end reproduction was run. Source inspection does show a reproducible reset-archive boundary: reset archives the old transcript, creates a fresh active session file, and WebChat `chat.history` reads only the active session transcript path.

## Solution Assessment

Unclear without maintainer product judgment. The durable fix should distinguish unintentional restart/abort recovery from intentional reset/new-session branching, rather than blindly merging archive content into the active history.
