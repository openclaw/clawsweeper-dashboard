## Summary

Keep open: current main still matches the reported Active Memory reliability gap, but runtime provider failover and p95 latency-budget signaling extend the documented plugin contract and need maintainer direction before implementation.

## What This Changes

Request Active Memory to fail over across configured providers and emit structured latency-budget signals when the selected recall provider is unavailable or slower than the configured timeout.

## Best Possible Solution

Define the Active Memory runtime fallback and latency-observation contract, then reuse existing model fallback/auth-profile cooldown machinery with focused regression tests and docs if maintainers approve the behavior.

## Reproduction Assessment

Yes for current behavior by source inspection: Active Memory resolves one model and calls runEmbeddedPiAgent directly, so provider cooldown/errors return empty recall without runWithModelFallback. No live provider-cooldown run was performed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended contract. If approved, the narrow maintainable path is to reuse runWithModelFallback and auth-profile cooldown semantics rather than adding plugin-specific provider discovery.
