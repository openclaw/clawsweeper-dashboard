## Summary

Keep open: current main has better timeout status/circuit-breaker handling, but it still lacks active-memory runtime provider failover and the requested provider-latency budget signal. The remaining work changes the documented active-memory model fallback contract, so it needs maintainer product/API direction before a repair PR.

## What This Changes

Request active-memory runtime failover across fallback providers plus structured timeout/provider-latency reporting when recall providers are unavailable or too slow.

## Best Possible Solution

Define the active-memory runtime failover and latency-budget contract first; if accepted, reuse runWithModelFallback/auth-profile rotation and add focused status, telemetry, docs, and regression coverage.

## Reproduction Assessment

Yes for the no-failover path by source inspection: active-memory resolves one model and calls runEmbeddedPiAgent directly, so provider cooldown/error does not consult the host fallback runner. The p95 warning absence is also source-confirmed, though no live provider-cooldown run was performed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract. The narrow maintainable implementation would reuse existing runWithModelFallback and auth-profile cooldown semantics rather than adding plugin-specific provider discovery.
