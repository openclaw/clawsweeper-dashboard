## Summary

Keep open: current main still resolves one Active Memory model and catches provider failures without walking the shared model fallback chain, and the requested p95/latency-budget signal is still absent. The behavior is source-reproducible, but changing it extends the documented Active Memory contract and needs maintainer direction.

## What This Changes

The issue requests runtime provider/model failover for Active Memory recall plus structured timeout and latency-budget telemetry.

## Best Possible Solution

Define the Active Memory runtime fallback contract, expose or reuse the shared model-fallback/auth-profile machinery through an appropriate runtime seam, and add structured timeout/circuit-breaker/latency-budget observations with focused regression tests.

## Reproduction Assessment

Yes for the source path, without a live provider-cooldown run. Current main resolves one model, calls runEmbeddedPiAgent directly, and catches provider failures as failed recall without runWithModelFallback; the requested p95 latency-budget signal is absent by source search.

## Solution Assessment

Unclear until maintainers choose the contract. The narrowest maintainable direction is to reuse or expose the shared fallback machinery rather than adding plugin-specific provider discovery, while keeping modelFallback's documented resolution-only meaning explicit.
