## Summary

Keep open: current main still resolves a single Active Memory recall model and calls the embedded runner directly, while the requested runtime failover and latency-budget telemetry would change the documented Active Memory contract and need product/API direction before a focused implementation.

## What This Changes

The issue asks Active Memory to use runtime provider failover for recall subagent failures and to surface structured latency-budget/circuit-breaker signals when provider latency exceeds `timeoutMs`.

## Best Possible Solution

Define the Active Memory fallback and telemetry contract first, then expose or reuse generic model fallback/auth-profile rotation machinery and add structured timeout, circuit-breaker, and latency-budget observations with focused regression coverage.

## Reproduction Assessment

Yes, at source level: Active Memory resolves one model, calls `runEmbeddedPiAgent` directly, and has no p95 latency-budget signal. I did not run a live provider-cooldown or slow-provider reproduction.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable direction would reuse the existing fallback/auth-profile machinery through a supported runtime seam rather than adding plugin-specific provider discovery.
