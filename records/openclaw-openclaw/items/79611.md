## Summary

Keep open: current main and v2026.6.10 still resolve one Active Memory model and run one embedded recall sub-agent without the existing model-fallback chain, while the requested latency-budget/p95 diagnostic is also absent. The remaining work needs maintainer product/API direction because current docs explicitly define `modelFallback` as resolution-only, not runtime failover.

## What This Changes

Request Active Memory to use recoverable provider fallback and structured latency/timeout diagnostics instead of dropping recall when its selected provider is unavailable or too slow.

## Best Possible Solution

Define an Active Memory fallback and latency-telemetry contract that reuses existing model-fallback machinery where appropriate, preserves explicit model pins unless maintainers approve inheritance, and adds focused tests/docs for provider cooldown, recoverable errors, timeout status, and circuit-breaker diagnostics.

## Reproduction Assessment

Yes at source level: current main resolves a single Active Memory provider/model and runs one embedded sub-agent without the model-fallback chain, and no p95 latency diagnostic exists. I did not run a live provider-cooldown repro in this read-only review.

## Solution Assessment

Unclear as a concrete fix: the requested reliability behavior is valid, but the best implementation needs maintainer-chosen fallback inheritance and telemetry semantics. A maintainable fix should reuse existing model-fallback contracts rather than adding a parallel plugin-local fallback stack.
