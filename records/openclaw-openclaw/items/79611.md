## Summary

Keep open: current main still lacks runtime provider failover for active-memory and has no p95/latency-budget signal, but the requested behavior extends the documented active-memory contract and needs maintainer direction before an automated fix.

## What This Changes

Request active-memory recall to fail over through configured providers/auth profiles and emit structured timeout, latency-budget, and circuit-breaker signals when recall is skipped.

## Best Possible Solution

Define the active-memory fallback contract, then reuse the existing model fallback/auth-profile rotation machinery and add structured timeout/circuit-breaker/latency-budget observations with focused regression coverage.

## Reproduction Assessment

Yes for the no-failover path by source inspection: active-memory resolves one model and calls runEmbeddedPiAgent directly, so a provider cooldown/error returns empty recall without consulting runWithModelFallback. The latency-budget warning request is also source-confirmed absent, but the exact p95 behavior is a new design choice.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable implementation would reuse existing runWithModelFallback and auth-profile cooldown semantics rather than creating plugin-specific provider discovery.
