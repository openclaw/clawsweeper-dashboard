## Summary

Keep open: current main still lacks active-memory runtime provider failover and the requested provider-latency p95 warning, while the timeout side is only partially covered by existing status/circuit-breaker handling. The remaining work changes the documented active-memory model fallback contract, so it needs maintainer direction before a fix PR lane.

## What This Changes

The issue requests runtime provider failover and structured timeout/latency-budget signaling for the active-memory plugin.

## Best Possible Solution

Define the active-memory fallback and telemetry contract, then wire the plugin through existing model-fallback/auth-profile machinery with focused regression coverage for provider cooldown, provider errors, timeout classification, and circuit-breaker behavior.

## Reproduction Assessment

Yes for the no-failover path by source inspection: active-memory resolves one model and calls the embedded runner directly, so provider cooldown or structured provider failure returns an empty failed recall without trying host fallbacks. The provider p95 latency-budget behavior is source-confirmed absent, but its exact semantics are a new design choice.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable direction would reuse existing model fallback and auth-profile cooldown semantics rather than adding plugin-local provider discovery, and add structured timeout observations without changing unrelated memory behavior.
