## Summary

Keep this open: current main partially fixes the timeout signaling side with status lines, unavailable diagnostics, and a circuit breaker, but the active-memory recall run still lacks its own outer model-fallback loop and can turn embedded-run failover into a skipped recall.

## What This Changes

The issue asks for active-memory recall to fail over across configured providers and surface structured timeout/latency signals instead of silently skipping recall.

## Best Possible Solution

Add an owner-approved active-memory fallback path that reuses the existing agent/model fallback contract, preserves strict single-model semantics when explicitly requested, and keeps structured timeout, unavailable, and circuit-breaker status coverage.

## Reproduction Assessment

Yes, source-level reproduction is available for the missing outer fallback path: active-memory launches one embedded recall run and catches embedded `FailoverError` instead of driving the model fallback loop. A live degraded-provider reproduction was not run.

## Solution Assessment

Unclear. Reusing the existing model-fallback contract looks cleaner than a plugin-local provider loop, but maintainers need to choose how active-memory should inherit fallback defaults and whether to add rolling p95 latency telemetry.
