## Summary

Keep open: current main still resolves one Active Memory model and runs it directly, so provider cooldown/errors do not use the host fallback chain; the p95/latency-budget signal is also absent. The timeout side is partly mitigated by status lines and a circuit breaker, so the remaining work is a contract/product decision rather than a narrow bug-fix cleanup close.

## What This Changes

The issue requests runtime provider failover plus structured timeout and latency-budget telemetry for the active-memory plugin.

## Best Possible Solution

Define the Active Memory runtime fallback and latency telemetry contract, then reuse the existing runWithModelFallback/auth-profile cooldown machinery and add focused status, docs, and regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main resolves one model, calls runEmbeddedPiAgent directly, and has no Active Memory p95/latency-budget warning path, though I did not run a live provider-cooldown canary.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable direction appears to be reusing runWithModelFallback and existing auth-profile cooldown semantics rather than adding plugin-specific provider discovery.
