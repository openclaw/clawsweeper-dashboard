## Summary

Keep open: current main still resolves Active Memory to one model and calls the embedded runner directly, while runtime provider failover and p95 latency-budget telemetry remain absent. Because current docs explicitly define `modelFallback` as resolution-only, this needs maintainer contract direction rather than cleanup closure.

## What This Changes

Request Active Memory to use runtime provider failover and structured timeout/latency-budget signals instead of skipping recall when the selected provider is unavailable or too slow.

## Best Possible Solution

Define the Active Memory fallback and latency contract, then reuse the generic fallback/auth-profile machinery through a supported seam and add focused timeout, circuit-breaker, telemetry, docs, and regression coverage.

## Reproduction Assessment

Yes for the no-failover path by source inspection: Active Memory resolves one model and calls `runEmbeddedPiAgent` directly, so provider cooldown or FailoverError returns failed/empty recall without host fallback. The p95 behavior is confirmed absent, but its exact warning/short-circuit semantics are a new design choice.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrow maintainable direction is to reuse the existing fallback/auth-profile runner through a generic seam, not to add plugin-specific provider discovery or reinterpret `modelFallback` silently.
