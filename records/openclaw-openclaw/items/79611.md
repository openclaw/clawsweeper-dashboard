## Summary

Keep open: current main still does not provide Active Memory runtime provider failover or the requested provider-latency p95 warning. Timeout status/circuit-breaker handling exists now, but changing failover behavior would alter the documented resolution-only `modelFallback` contract.

## What This Changes

The issue requests active-memory provider failover and better latency-budget timeout signaling for recall runs.

## Best Possible Solution

Define the Active Memory runtime failover and latency-budget contract first; if accepted, reuse existing model fallback/auth-profile rotation machinery and add focused telemetry, docs, and regression coverage.

## Reproduction Assessment

Yes for the missing no-failover path by source inspection: Active Memory resolves one model and calls `runEmbeddedPiAgent` directly, so selected-provider cooldown/error does not consult the outer fallback runner. The requested p95 warning is also absent from source/tests/docs, though no live provider-cooldown run was performed.

## Solution Assessment

Unclear until maintainers choose the contract. The narrowest maintainable implementation would reuse `runWithModelFallback` and existing auth-profile cooldown semantics rather than adding plugin-specific provider discovery.
