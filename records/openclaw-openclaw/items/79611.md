## Summary

Keep open: current main and the latest release still define Active Memory fallback as model-resolution only, while the issue asks for runtime provider failover and latency-budget telemetry that remain missing and need maintainer product/API direction.

## What This Changes

The issue requests runtime provider failover plus clearer timeout, circuit-breaker, and latency diagnostics for Active Memory recall passes.

## Best Possible Solution

Define a maintainer-approved Active Memory fallback contract, preferably reusing the existing model-fallback path where appropriate without changing explicit pins unexpectedly, and add structured timeout/circuit-breaker telemetry for operator diagnosis.

Yes for the missing requested behavior at source level: current main resolves one Active Memory provider/model and invokes one embedded run, so host fallback chaining is not exercised. I did not run a live provider-cooldown repro in this read-only review.

## Solution Assessment

Unclear as an implementation plan: the requested direction is valid, but the best fix needs maintainer-chosen fallback inheritance and latency telemetry semantics first. A narrow implementation should reuse existing model-fallback contracts rather than adding a parallel fallback stack.
