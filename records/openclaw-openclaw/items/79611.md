## Summary

Keep open: current main and the latest release still resolve Active Memory to one provider/model before the embedded run, while the requested runtime failover and latency-budget telemetry remain missing and need maintainer product/API direction.

## What This Changes

The issue requests Active Memory runtime provider failover plus structured timeout and latency-budget telemetry for slow or unavailable providers.

## Best Possible Solution

Define a maintainer-approved Active Memory fallback contract, preferably reusing existing model-fallback seams without changing explicit pins unexpectedly, then add focused timeout/latency diagnostics and regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main resolves one Active Memory provider/model and invokes one embedded run, so host fallback chaining is not exercised. I did not run a live provider-cooldown scenario in this read-only review.

## Solution Assessment

Unclear: adding host fallback inheritance and latency telemetry may be useful, but the best fix needs maintainer-chosen fallback semantics because docs currently define `modelFallback` as model-resolution only.
