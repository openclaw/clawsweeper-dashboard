## Label Justifications

- `P2`: The issue describes a bounded bundled-plugin reliability gap under provider failure and slow providers without evidence of data loss, security impact, crash loop, or unusable core runtime.
- `impact:session-state`: Skipped or timed-out Active Memory recall changes the memory context injected into the session before the user-facing answer.
- `impact:auth-provider`: The remaining failure mode centers on provider/model routing, auth-profile cooldowns, and fallback behavior for the recall sub-agent.

## Summary

Keep open: current main and v2026.6.6 still resolve Active Memory to one model candidate and launch one embedded run, so runtime provider failover and requested latency-budget telemetry remain unimplemented; timeout status lines and the circuit breaker only partially address the silent-drop complaint.

## What This Changes

Request Active Memory to use model/provider failover and structured latency/timeout diagnostics when the selected recall provider is unavailable or too slow.

## Best Possible Solution

Add maintainer-approved Active Memory fallback semantics, preferably reusing the existing model-fallback contract without changing explicit pins unexpectedly, and add focused timeout/latency diagnostics.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: Active Memory resolves one model and calls runEmbeddedAgent once, so embedded provider/auth failures do not walk the host fallback chain; I did not run a live degraded-provider repro.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the best fix should reuse the existing model-fallback contract and needs maintainer-chosen fallback inheritance and latency telemetry semantics first.
