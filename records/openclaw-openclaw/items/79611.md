## Summary

Keep open: current main still lacks active-memory runtime provider failover and p95 latency-budget signaling, but the requested behavior changes the documented contract and needs maintainer product/API direction before implementation.

## What This Changes

Request that active-memory use runtime provider failover and emit structured timeout or latency-budget signals when its selected provider is unavailable or slow.

## Best Possible Solution

Define the active-memory runtime fallback and timeout telemetry contract, then reuse the existing runWithModelFallback/auth-profile machinery and add structured timeout, circuit-breaker, and latency-budget observations with focused regression coverage.

## Reproduction Assessment

Yes, source-level: active-memory resolves a single model and invokes runEmbeddedPiAgent directly, so provider cooldown or FailoverError does not consult runWithModelFallback. The p95 latency-budget signal is also source-confirmed absent, though live provider timing was not rerun.

## Solution Assessment

Unclear until maintainers choose the intended contract. If approved, the narrow maintainable path is to route active-memory through the existing fallback/auth-profile machinery instead of adding plugin-specific provider discovery.
