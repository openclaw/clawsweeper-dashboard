## Summary

Keep open: current main still lacks active-memory runtime provider failover and p95 latency-budget signaling, but the request changes the documented resolution-only fallback contract and needs maintainer product/API direction before a fix is safe.

## What This Changes

Request active-memory to fail over across configured providers/models and emit structured latency-budget/circuit-breaker signals when provider latency exceeds `timeoutMs`.

## Best Possible Solution

Define the active-memory fallback contract, expose or reuse a plugin-safe fallback runner, then add structured provider-failure, timeout, circuit-breaker, and latency-budget observations with focused regression coverage.

## Reproduction Assessment

Yes for the current no-failover behavior by source inspection: active-memory resolves a single model and calls `runEmbeddedPiAgent` directly, and the requested p95 warning is absent from code/docs/tests. A live provider-cooldown reproduction was not run.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable direction is to reuse existing `runWithModelFallback` and auth-profile cooldown semantics through a plugin-safe runtime/API surface rather than adding plugin-specific provider discovery.
