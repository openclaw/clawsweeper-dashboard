## Summary

Keep this PR open; the limiter helper is useful, but the patch is not ready. Production gateway startup never creates or passes the new request limiter, and the HTTP integration keys on the raw socket IP before trusted-proxy resolution, which defeats the stated reverse-proxy use case.

## What This Changes

The PR adds a fixed-window per-IP request limiter module with tests and threads an optional `requestRateLimiter` through gateway HTTP runtime state to return `429` when the check fails.

## Best Possible Solution

Rework this as one coherent Gateway hardening path: define the supported config/schema/docs contract, create and dispose the limiter in production startup, enforce it using `resolveRequestClientIp` after trusted-proxy config is loaded, and add focused HTTP/proxy runtime proof.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the blockers: normal startup never supplies `requestRateLimiter`, and the PR checks `req.socket.remoteAddress` before trusted-proxy client IP resolution.

## Solution Assessment

No. The current branch is not the best merge shape because it adds an optional seam without production wiring and uses the wrong IP source for the reverse-proxy deployment it is meant to protect.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire the limiter into production startup:** `src/gateway/server-runtime-state.ts:61-62`
  - body: Adding `requestRateLimiter` as an optional runtime-state parameter is not enough to enable the feature: the production startup path only creates and passes the existing auth limiter, so normal Gateway HTTP traffic never gets a general request limiter. Please create the limiter from the supported config/default path, pass it here, and dispose it on shutdown before treating this as Gateway flood protection.
  - confidence: 0.91
- **[P1] Resolve client IP before enforcing limits:** `src/gateway/server-http.ts:772`
  - body: This check uses `req.socket.remoteAddress`, which is the reverse proxy address in proxied deployments. With the default loopback exemption, a local proxy bypasses the limiter entirely; with a non-loopback proxy, unrelated users share one bucket. Load the gateway proxy config first and call `resolveRequestClientIp(req, trustedProxies, allowRealIpFallback)` before `check`.
  - confidence: 0.94
