## Summary

Keep open for maintainer/contributor follow-up: current main does not implement general Gateway request limiting, but this PR is not merge-ready because it keys on the raw socket IP before trusted-proxy resolution, lacks a supported config/startup path, and has no real behavior proof. The broader hardening work at https://github.com/openclaw/openclaw/pull/44884 overlaps but is still open, not a shipped replacement.

## What This Changes

The PR adds a fixed-window per-IP Gateway HTTP request limiter module with tests and optional plumbing through `createGatewayHttpServer` and runtime state.

## Best Possible Solution

Land one coherent Gateway request-limit hardening path with documented config, startup/disposal wiring, proxy-aware keying via `resolveRequestClientIp`, focused regression coverage, and redacted real-runtime 429 proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the proposed limiter is keyed before trusted-proxy client-IP resolution, and current main has no normal request-limit config/startup path. I did not run the branch because this review is read-only.

## Solution Assessment

No. The standalone limiter module is useful, but the maintainable fix needs proxy-aware client-IP resolution plus aligned Gateway config, startup, disposal, docs, and proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter checks `req.socket.remoteAddress`, which is the proxy hop in reverse-proxy deployments. Same-host proxies are loopback-exempt and other proxies collapse unrelated users into one bucket, so key the limiter after config load with `resolveRequestClientIp(...)`.
  - confidence: 0.94
- **[P2] Wire request limits through supported gateway config:** `src/gateway/server-runtime-state.ts:64-65`
  - body: This only threads an optional limiter object through runtime state. Without `GatewayConfig`/schema/startup/disposal/docs wiring, ordinary Gateway processes cannot enable the claimed flood protection through supported configuration.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds Gateway flood-protection code, but the current integration can bypass or mis-key reverse-proxy traffic and lacks normal supported enablement.

Concerns:

- **[high] Reverse-proxy requests can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer means loopback reverse proxies are exempt by default and non-loopback proxies make all clients share one bucket, undermining the public-exposure hardening goal.
  - confidence: 0.94
- **[medium] No normal config path enables the hardening:** `src/gateway/server-runtime-state.ts:64`
  - body: The limiter is optional plumbing only; without schema, startup, docs, and shutdown wiring, normal Gateway runs cannot turn on request flood protection safely.
  - confidence: 0.86
