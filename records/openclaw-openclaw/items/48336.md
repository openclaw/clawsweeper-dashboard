## Summary

Keep this PR open for maintainer/security review, but it is not merge-ready: the proposed limiter is keyed before trusted-proxy client-IP resolution, ordinary Gateway startup has no supported request-limit config path on current main, and the external PR still lacks real Gateway runtime proof.

## What This Changes

The PR adds a fixed-window per-IP Gateway HTTP request limiter module, unit tests, and optional plumbing through `createGatewayHttpServer` and runtime state.

## Best Possible Solution

Land one coherent Gateway request-limiting design with proxy-aware client-IP keying, documented config/schema/startup/disposal wiring, HTTP and WebSocket coverage, focused tests, and redacted real-runtime proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the limiter keys on `req.socket.remoteAddress` before trusted-proxy resolution, and current main has no supported general request-limit config/startup path. I did not run the PR branch because this review is read-only.

## Solution Assessment

No. The standalone limiter module is useful, but the maintainable fix needs proxy-aware keying through `resolveRequestClientIp`, aligned Gateway config/startup/docs, and real runtime proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:773`
  - body: The limiter keys requests by `req.socket.remoteAddress`, which is the proxy hop in reverse-proxy deployments. Same-host proxies are loopback-exempt and other proxies make unrelated clients share one bucket, so compute the client IP with the existing trusted-proxy resolver before calling `check`.
  - confidence: 0.94
- **[P2] Wire request limits through supported Gateway config:** `src/gateway/server-runtime-state.ts:63`
  - body: This only threads an optional limiter object through runtime state; normal Gateway startup never creates it from supported config or disposes it. Operators therefore cannot enable the claimed flood protection through the documented Gateway configuration path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds Gateway flood-protection code, but the current integration can bypass or mis-key reverse-proxy traffic and lacks a supported enablement path.

Concerns:

- **[high] Reverse-proxy requests can bypass or share the limiter:** `src/gateway/server-http.ts:773`
  - body: Using the raw socket peer means loopback reverse proxies are exempt by default, while non-loopback proxies make all clients share one counter; that undermines the public-exposure hardening goal.
  - confidence: 0.94
- **[medium] No supported config path enables the hardening:** `src/gateway/server-runtime-state.ts:63`
  - body: The limiter is optional plumbing only; without Gateway schema/startup/disposal/docs wiring, normal Gateway runs cannot turn on request flood protection safely.
  - confidence: 0.86
