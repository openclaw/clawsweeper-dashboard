## Summary

Keep open. Current main does not implement a general Gateway request limiter, and this PR is not ready to merge because the diff keys limits on the raw socket peer before proxy-aware client-IP resolution and does not add a normal config/startup/docs path to enable the hardening. The overlapping open Gateway hardening work in #44884 and the security-labeled tracker #20373 make this a maintainer/security coordination item.

## What This Changes

The branch adds a fixed-window per-IP Gateway HTTP request limiter module with tests and threads an optional `requestRateLimiter` through `createGatewayHttpServer` and `createGatewayRuntimeState`.

## Best Possible Solution

Land one coherent Gateway hardening path: add a documented `gateway.requestRateLimit` config/schema/help surface, create and dispose the limiter in Gateway startup, enforce it after `resolveRequestClientIp(req, trustedProxies, allowRealIpFallback)`, and test direct clients, trusted reverse proxies, loopback proxy sources, `429`/`Retry-After`, and pruning behavior.

## Reproduction Assessment

Yes. The current-main absence is reproducible with the `rg` check for `requestRateLimit`/`createRequestRateLimiter`, and the PR regression is reproducible from the diff by placing the limiter before config-backed client-IP resolution in a same-host reverse-proxy setup.

## Solution Assessment

No. The standalone limiter shape is useful, but the integration is not the best fix because it uses the TCP peer instead of the resolved client IP and leaves the feature opt-in without a normal Gateway config/startup/docs path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter is keyed with `req.socket?.remoteAddress`, which is the proxy hop in reverse-proxy deployments. A same-host proxy is loopback and is exempted entirely, while a non-loopback proxy makes unrelated clients share one bucket; load the config snapshot and use `resolveRequestClientIp(...)` before calling `check()`.
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: The branch only threads an optional limiter object through runtime state. `startGatewayServer` never creates one from Gateway config and the schema/docs do not expose `gateway.requestRateLimit`, so a normal Gateway process will not enable the new flood protection.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch is intended as Gateway exposure hardening, but the current integration can leave reverse-proxy deployments unprotected and has no normal enablement path.

Concerns:

- **[high] Reverse proxy traffic can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer as the key means loopback reverse proxies are exempt by default and non-loopback proxies collapse many users into one bucket, weakening the intended public Gateway flood protection.
  - confidence: 0.94
- **[medium] No normal startup/config path enables the hardening:** `src/gateway/server-runtime-state.ts:61`
  - body: The new limiter is only an optional object passed through runtime state; without schema/startup/docs wiring, ordinary Gateway runs cannot turn on the new request flood protection.
  - confidence: 0.86
