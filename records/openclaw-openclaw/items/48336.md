## Summary

Keep open: current main does not implement general request rate limiting, and this security-hardening PR still has merge-blocking integration gaps plus missing real behavior proof.

## What This Changes

The PR adds a fixed-window per-IP Gateway HTTP request limiter module with tests and threads an optional `requestRateLimiter` through Gateway HTTP runtime state.

## Best Possible Solution

Land one coherent Gateway hardening path with a documented `gateway.requestRateLimit` schema/docs surface, startup creation/disposal, proxy-aware keying via `resolveRequestClientIp`, and regression coverage for direct and trusted reverse-proxy traffic.

## Reproduction Assessment

Yes. Source inspection is enough: in the PR diff, a same-host reverse proxy makes `req.socket.remoteAddress` loopback before trusted proxy headers are resolved, so the loopback-exempt limiter is bypassed.

## Solution Assessment

No. The limiter module is useful, but this branch is not the best merge shape until it is keyed by resolved client IP and wired through normal Gateway config/startup, with the overlap against the broader hardening PR settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter checks the raw TCP peer before loading proxy config. In same-host reverse-proxy deployments that peer is loopback and exempt, while non-loopback proxies make unrelated users share one bucket; move the check after config load and key it with `resolveRequestClientIp(...)`.
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: This only adds an optional limiter object to runtime state. `startGatewayServer` never creates it from config and there is no schema/docs surface, so normal Gateway processes cannot enable the claimed flood protection.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive and currently leaves reverse-proxy deployments unprotected while lacking a normal enablement path.

Concerns:

- **[high] Reverse proxy traffic can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer as the key means loopback reverse proxies are exempt by default and non-loopback proxies collapse many users into one bucket, weakening the intended public Gateway flood protection.
  - confidence: 0.94
- **[medium] No normal startup/config path enables the hardening:** `src/gateway/server-runtime-state.ts:61`
  - body: The new limiter is only an optional object passed through runtime state; without schema/startup/docs wiring, ordinary Gateway runs cannot turn on the new request flood protection.
  - confidence: 0.86
