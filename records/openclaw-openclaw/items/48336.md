## Summary

Keep open: current main still lacks a general all-request Gateway limiter, but this PR is not mergeable as-is because it keys on the raw socket peer before proxy-aware client-IP resolution and lacks a normal config/startup enablement path; the overlap with #44884 and #20373 needs maintainer consolidation.

## What This Changes

The PR adds a fixed-window per-IP Gateway HTTP request limiter module with tests and threads an optional `requestRateLimiter` through Gateway HTTP runtime state.

## Best Possible Solution

Land one coherent Gateway hardening path with a documented `gateway.requestRateLimit` config/schema/docs surface, startup creation/disposal, proxy-aware keying via `resolveRequestClientIp`, and regression coverage for direct and trusted reverse-proxy traffic.

## Reproduction Assessment

Yes. Source inspection of the PR diff gives a high-confidence reproduction path: in a same-host reverse-proxy deployment, `req.socket.remoteAddress` is loopback before trusted proxy headers are resolved, so the loopback-exempt limiter is bypassed.

## Solution Assessment

No. The standalone limiter is a useful building block, but the integration is not the best solution because it uses the TCP peer instead of the resolved client IP and leaves normal Gateway processes without a config/startup path to enable it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter is keyed with `req.socket?.remoteAddress`, which is the proxy hop in reverse-proxy deployments. A same-host proxy is loopback and exempted entirely, while a non-loopback proxy makes unrelated clients share one bucket; load the config snapshot and use `resolveRequestClientIp(...)` before calling `check()`.
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: This only threads an optional limiter object through runtime state. `startGatewayServer` never creates one from Gateway config and the schema/docs do not expose `gateway.requestRateLimit`, so a normal Gateway process will not enable the new flood protection.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because this Gateway exposure hardening patch can leave reverse-proxy deployments unprotected and has no normal enablement path.

Concerns:

- **[high] Reverse proxy traffic can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer as the key means loopback reverse proxies are exempt by default and non-loopback proxies collapse many users into one bucket, weakening the intended public Gateway flood protection.
  - confidence: 0.94
