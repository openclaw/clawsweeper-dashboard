## Label Justifications

- `P1`: Gateway flood protection affects security and availability for users who expose the Gateway beyond loopback.

## Summary

Keep open: the PR is useful Gateway hardening direction, but it is not merge-ready because it keys the limiter by raw socket IP before trusted-proxy resolution, lacks a supported Gateway config/startup lifecycle, and has no after-fix real behavior proof.

## What This Changes

Adds a fixed-window per-IP Gateway request limiter helper, unit tests, and optional `requestRateLimiter` plumbing through the HTTP server/runtime state.

## Best Possible Solution

Land one Gateway hardening path that enforces request limits on the trusted-proxy-resolved client IP, exposes a documented config/startup/disposal lifecycle, and includes redacted runtime proof for `429` plus `Retry-After`.

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:769`

  - confidence: 0.91
