## Summary

Keep open: current main does not implement general Gateway request limiting, and this PR still has security-sensitive proxy-keying, supported-config, and real-runtime-proof blockers. The related https://github.com/openclaw/openclaw/pull/44884 overlaps this area but is still open and not a merged or clearly canonical replacement.

## What This Changes

The branch adds a fixed-window per-IP Gateway HTTP request limiter module, unit tests, and optional HTTP/runtime-state plumbing.

## Best Possible Solution

Land one coherent Gateway request-limiting design with proxy-aware client-IP keying, supported config/schema/startup/disposal wiring, focused tests, docs for any user-facing config, and redacted runtime proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the branch checks the limiter against req.socket.remoteAddress before trusted-proxy resolution, and current main has no supported general request-limit config/startup path. I did not run the PR branch because this review is read-only.

## Solution Assessment

No. The standalone limiter is useful, but the maintainable solution needs proxy-aware keying through resolveRequestClientIp, aligned Gateway config/startup/docs/disposal wiring, and real runtime proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter keys requests by req.socket.remoteAddress, which is the reverse-proxy hop in proxied deployments. Same-host proxies are loopback-exempt and other proxies make unrelated clients share one bucket, so resolve the client IP with the existing trusted-proxy helper before calling check.
  - confidence: 0.94
- **[P2] Wire request limits through supported Gateway config:** `src/gateway/server-runtime-state.ts:65`
  - body: This only threads an optional limiter object through runtime state; normal Gateway startup never creates it from supported config or disposes it. Operators therefore cannot enable the advertised flood protection through the documented Gateway configuration path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch adds Gateway flood-protection code, but the current integration can bypass or mis-key reverse-proxy traffic and lacks a supported enablement path.

Concerns:

- **[high] Reverse-proxy traffic can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer means loopback reverse proxies are exempt by default, while non-loopback proxies make all clients share one counter; that undermines the public-exposure hardening goal.
  - confidence: 0.94
- **[medium] No supported config path enables the hardening:** `src/gateway/server-runtime-state.ts:65`
  - body: The limiter is optional plumbing only; without Gateway schema/startup/disposal/docs wiring, normal Gateway runs cannot turn on request flood protection safely.
  - confidence: 0.86
