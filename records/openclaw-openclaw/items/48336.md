## Label Justifications

- `P1`: Gateway flood protection affects exposed runtime security and availability when users bind the Gateway beyond loopback.

## Summary

Keep open: this targets a real Gateway hardening gap, but it is not merge-ready because proof is missing, proxy traffic is keyed before trusted client-IP resolution, the supported config/startup lifecycle is absent, and the same design space overlaps a broader open Gateway hardening PR.

## What This Changes

The PR adds an optional fixed-window per-IP Gateway HTTP request limiter, unit tests, and plumbing through Gateway HTTP server/runtime state.

## Best Possible Solution

Land one Gateway hardening design that enforces request limits on the trusted-proxy-resolved client IP, exposes a supported config/startup/disposal contract, and includes redacted runtime proof for 429 plus Retry-After behavior.

Yes for the merge-blocking PR defect by source inspection: the PR diff adds requestRateLimiter.check(req.socket?.remoteAddress) before trusted-proxy client IPs are available. I did not run the PR branch because this review is read-only.

No: the PR’s current integration is not the best way to solve Gateway flood protection because it lacks trusted-proxy-aware keying and a supported config/startup lifecycle. The safer direction is one coherent Gateway hardening surface using the existing resolved client-IP contract.
