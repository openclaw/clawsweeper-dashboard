## Summary

Keep open: current main does not implement the requested general Gateway request limiter, and this PR still has blocking proxy-keying, config/startup, security, and real-behavior-proof gaps before it can merge.

## What This Changes

The PR adds a fixed-window Gateway per-IP HTTP request limiter module and tests, then threads an optional limiter through Gateway HTTP runtime state.

## Best Possible Solution

Land one coherent Gateway hardening path with documented `gateway.requestRateLimit` config, startup/disposal wiring, proxy-aware keying, regression coverage, and real-runtime 429 proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the limiter is keyed before trusted-proxy client IP resolution, and current startup/config schema has no normal request-limit enablement path. I did not run the branch because this review is read-only.

## Solution Assessment

No. The standalone limiter is useful, but the maintainable solution needs normal Gateway config/startup wiring and proxy-aware client IP resolution before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter checks `req.socket.remoteAddress`, which is the proxy hop in reverse-proxy deployments. Same-host proxies are loopback-exempt and other proxies collapse unrelated users into one bucket, so key the limiter after config load with `resolveRequestClientIp(...)`.
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: This adds only an optional limiter object passed through runtime state. Without `GatewayConfig`/schema/startup/disposal/docs wiring, ordinary Gateway processes cannot enable the claimed flood protection.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR touches Gateway flood protection and currently leaves reverse-proxy deployments unprotected or cross-throttled while lacking a normal enablement path.

Concerns:

- **[high] Reverse-proxy requests can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer means loopback reverse proxies are exempt by default and non-loopback proxies make all clients share one bucket, undermining the public-exposure hardening goal.
  - confidence: 0.94
- **[medium] No normal config path enables the hardening:** `src/gateway/server-runtime-state.ts:61`
  - body: The limiter is optional plumbing only; without schema/startup/docs wiring, normal Gateway runs cannot turn on request flood protection.
  - confidence: 0.86
