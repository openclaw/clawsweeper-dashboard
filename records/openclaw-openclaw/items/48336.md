## Label Justifications

- `P1`: Gateway flood protection affects security and availability for users who expose the Gateway beyond loopback.
- `merge-risk: 🚨 security-boundary`: The proposed limiter can be bypassed or mis-keyed behind trusted reverse proxies if it uses the raw socket peer instead of resolved client IP.
- `merge-risk: 🚨 availability`: A mis-keyed limiter can make unrelated users behind the same proxy share one bucket and receive cross-user `429` responses.

## Summary

Keep this PR open, but it is not merge-ready: the proposed limiter is security-sensitive, lacks real Gateway runtime proof, and currently keys reverse-proxy traffic on the raw socket peer instead of OpenClaw's trusted-proxy-resolved client IP.

## What This Changes

The PR adds a fixed-window per-IP Gateway HTTP request limiter, unit tests, and optional plumbing through `createGatewayHttpServer` and `createGatewayRuntimeState`.

## Best Possible Solution

Land one Gateway hardening path that enforces request limits on the trusted-proxy-resolved client IP, exposes a documented config/startup/disposal lifecycle, and includes redacted runtime proof for `429` plus `Retry-After`.

## Reproduction Assessment

Yes for the merge-blocking PR defect by source inspection: the diff calls `requestRateLimiter.check(req.socket?.remoteAddress)` before trusted-proxy client IP resolution is available. I did not run the PR branch because this review is read-only.

## Solution Assessment

No: the current PR shape is not the best solution because it lacks trusted-proxy-aware keying and a supported operator lifecycle. The safer path is one coherent Gateway hardening surface using the existing resolved client-IP contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:769`
  - body: This check uses `req.socket.remoteAddress`, which is the reverse proxy hop in proxied deployments. With loopback proxies the default loopback exemption bypasses the limiter entirely, and with non-loopback proxies all clients share one bucket; load the Gateway config and call `resolveRequestClientIp(req, trustedProxies, allowRealIpFallback)` before checking the limiter.
  - confidence: 0.91
- **[P2] Expose a supported startup and config path:** `src/gateway/server-runtime-state.ts:62`
  - body: The PR threads an optional `requestRateLimiter` parameter but does not add Gateway config/schema/docs or startup ownership that creates and disposes it, so normal operators cannot enable the claimed flood protection through the supported Gateway configuration surface.
  - confidence: 0.86
