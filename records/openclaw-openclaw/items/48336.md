## Summary

Keep open: the patch should not merge as-is because it can miss the real client behind reverse proxies and does not provide a normal Gateway config/startup path for the new protection. The overlap with #44884 and #20373 also makes this a security-hardening consolidation decision rather than conservative cleanup.

## What This Changes

The PR adds a fixed-window per-IP Gateway HTTP request limiter with tests and threads an optional `requestRateLimiter` through Gateway HTTP runtime state.

## Best Possible Solution

Land one coherent Gateway hardening path with a documented `gateway.requestRateLimit` config/schema/docs surface, startup creation/disposal, proxy-aware keying via `resolveRequestClientIp`, and regression coverage for direct and trusted reverse-proxy traffic.

## Reproduction Assessment

Yes. Source inspection of the PR diff gives a high-confidence path: a same-host reverse proxy makes `req.socket.remoteAddress` loopback before trusted proxy headers are resolved, so the loopback-exempt limiter is bypassed.

## Solution Assessment

No. The limiter module is a useful building block, but the proposed integration is not the best solution because it keys on the TCP peer and lacks normal Gateway config/startup enablement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter is keyed with `req.socket?.remoteAddress`, which is the proxy hop in reverse-proxy deployments. A same-host proxy is loopback and exempted entirely, while a non-loopback proxy makes unrelated clients share one bucket; load the config snapshot and use `resolveRequestClientIp(...)` before calling `check()`.
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: This only threads an optional limiter object through runtime state. `startGatewayServer` never creates one from Gateway config and the schema/docs do not expose a general request-rate-limit setting, so normal Gateway processes cannot enable the new flood protection.
  - confidence: 0.86
