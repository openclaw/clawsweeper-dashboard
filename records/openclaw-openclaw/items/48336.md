## Summary

Keep this PR open, but do not merge it as submitted. Current main still lacks a general all-request Gateway HTTP rate limiter, and this branch is a plausible source for that work, but the patch keys the limiter before trusted-proxy client IP resolution and does not add the config/startup/docs path needed for normal Gateway runs.

## What This Changes

The PR adds a fixed-window per-IP request rate limiter module with tests and threads an optional requestRateLimiter through the Gateway HTTP/runtime state so over-limit HTTP requests can return 429 with Retry-After.

## Best Possible Solution

Land one coherent Gateway hardening path: add a documented gateway.requestRateLimit config/schema/help surface, create and dispose the limiter in Gateway startup, enforce it after resolveRequestClientIp(req, trustedProxies, allowRealIpFallback), and cover direct clients, trusted reverse proxies, loopback proxy sources, 429 Retry-After, and memory pruning behavior.

## Reproduction Assessment

Yes. The current-main absence is reproducible by searching for requestRateLimit/createRequestRateLimiter and inspecting Gateway startup/config, and the PR bug is reproducible from the diff by placing the rate-limit check before config-backed client IP resolution with a loopback reverse proxy.

## Solution Assessment

No. The limiter module shape is useful, but the integration is not the best fix because it keys on the TCP peer instead of the resolved client IP and leaves startup/config/docs unwired; the safer path is a startup-owned, proxy-aware limiter coordinated with the broader Gateway hardening thread.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter is keyed with req.socket?.remoteAddress, which is the proxy hop in reverse-proxy deployments. A same-host proxy is loopback and is exempted entirely, while a non-loopback proxy makes unrelated clients share one bucket; load the config snapshot and use resolveRequestClientIp(...) before calling check().
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: The branch only threads an optional limiter object through runtime state. startGatewayServer never creates one from gateway config and the schema/docs do not expose gateway.requestRateLimit, so a normal Gateway process will not enable the new flood protection.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch is a security hardening feature, but the submitted integration can make the new protection ineffective in the reverse-proxy deployments it is meant to defend.

Concerns:

- **[high] Reverse proxy traffic can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer as the key means loopback reverse proxies are exempt by default and non-loopback proxies collapse many users into one bucket, which weakens the intended public Gateway flood protection.
  - confidence: 0.94
- **[medium] No normal startup/config path enables the hardening:** `src/gateway/server-runtime-state.ts:61`
  - body: The new limiter is only an optional object passed through runtime state; without schema/startup/docs wiring, ordinary Gateway runs cannot turn on the new request flood protection.
  - confidence: 0.86
