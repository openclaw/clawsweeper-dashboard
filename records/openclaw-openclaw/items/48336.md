## Summary

Keep open: the limiter module is useful, but the PR is not merge-ready because the proposed enforcement keys on the raw socket peer in reverse-proxy deployments, lacks a normal Gateway config/startup path, and has no real behavior proof.

## What This Changes

The branch adds a fixed-window Gateway per-IP HTTP request limiter module/tests and threads an optional `requestRateLimiter` through Gateway HTTP runtime state.

## Best Possible Solution

Land one coherent Gateway hardening path with documented `gateway.requestRateLimit` config/schema/startup/disposal, proxy-aware keying via `resolveRequestClientIp`, regression coverage, and real-runtime proof.

## Reproduction Assessment

Yes for the PR defects. Source inspection is enough: the diff checks `req.socket.remoteAddress` before trusted proxy headers are resolved, and the branch only passes an optional limiter without a startup/config path.

## Solution Assessment

No. The standalone limiter is useful, but the maintainable solution needs normal Gateway config/startup wiring and proxy-aware client IP resolution before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve client IP before enforcing request limits:** `src/gateway/server-http.ts:772`
  - body: The limiter checks the raw TCP peer before loading proxy config. In same-host reverse-proxy deployments that peer is loopback and exempt, while non-loopback proxies make unrelated users share one bucket; move the check after config load and key it with `resolveRequestClientIp(...)`.
  - confidence: 0.94
- **[P2] Wire the limiter into Gateway startup config:** `src/gateway/server-runtime-state.ts:61-62`
  - body: This only adds an optional limiter object to runtime state. `startGatewayServer` never creates it from config and there is no schema/docs surface, so normal Gateway processes cannot enable the claimed flood protection.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive and currently weakens the intended reverse-proxy hardening story by keying on the wrong IP and lacking a normal enablement path.

Concerns:

- **[high] Reverse proxy traffic can bypass or share the limiter:** `src/gateway/server-http.ts:772`
  - body: Using the raw socket peer means loopback reverse proxies are exempt by default and non-loopback proxies collapse many users into one bucket, leaving public Gateway deployments unprotected or causing cross-user throttling.
  - confidence: 0.94
- **[medium] No normal startup/config path enables the hardening:** `src/gateway/server-runtime-state.ts:61`
  - body: The limiter is only an optional object passed through runtime state; without schema/startup/docs wiring, ordinary Gateway runs cannot turn on request flood protection.
  - confidence: 0.86
