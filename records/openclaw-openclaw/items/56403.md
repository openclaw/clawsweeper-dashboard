## Summary

Keep open. The reported default-disabled loop detector is still real on current main, but this PR is not merge-ready because it adds an args-only critical block for generic repeats and leaves user-facing loop-detection default guidance inconsistent.

## What This Changes

The PR enables `tools.loopDetection.enabled` by default, adds loop-detection regression tests, changes exec outcome hashing and generic-repeat escalation, and updates selected config/type/docs default metadata.

## Best Possible Solution

Land one rebased canonical fix that enables loop detection by default, preserves explicit opt-out, keeps current main's stable exec outcome hashing and run-scoped history, updates every config/type/docs surface consistently, and leaves generic-repeat hard aborts warning-only unless no-progress evidence or explicit maintainer policy supports blocking.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: repeated identical calls with omitted loop-detection config still resolve through `enabled: false`, and the unit test asserts that default-disabled behavior.

## Solution Assessment

No as-is. The default-enable direction is useful, but this PR should not hard-block generic repeats without result no-progress evidence, and the docs/config guidance needs complete synchronization before merge.
