## Summary

Keep open. The PR addresses a real current-main default-off gap, but it should not merge as-is because it couples default enabling with generic same-args hard blocking and leaves user-facing docs/changelog surfaces inconsistent.

## What This Changes

The PR enables `tools.loopDetection.enabled` by default, expands loop-detection tests, adjusts exec outcome hashing and generic-repeat escalation, and updates config/type/docs default metadata.

## Best Possible Solution

Land one rebased canonical fix that flips the default to true, preserves explicit opt-out, keeps generic repeats warning-only or gates hard blocks on stable no-progress evidence, and syncs docs, config metadata, generated types, tests, and changelog.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: repeated identical tool calls with omitted loop-detection config still resolve through `enabled: false`, and the default-off unit test asserts that behavior.

## Solution Assessment

No as-is. Enabling loop detection by default is a narrow fix, but this PR also hard-blocks generic repeats without no-progress evidence and leaves docs/changelog incomplete; the safer path is a smaller rebased default flip plus synchronized surfaces.
