## Summary

Keep open. Current main still defaults tool-loop detection off, so this PR addresses a real safety gap, but the branch should not merge as-is because it adds result-agnostic generic hard blocks and leaves the user-facing docs/changelog contract inconsistent amid overlapping loop-abort work.

## What This Changes

The PR flips `tools.loopDetection.enabled` to true by default, adds regression tests, adjusts exec outcome hashing, escalates generic repeats, and updates config/type/docs default metadata.

## Best Possible Solution

Land one rebased canonical fix that flips the default to true, preserves explicit opt-out, keeps generic repeats warning-only or gates hard blocks on stable no-progress evidence, and syncs docs, config metadata, generated types, tests, and changelog.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: repeated identical tool calls with omitted loop-detection config still resolve through `enabled: false`, and the default-off unit test asserts that behavior.

## Solution Assessment

No as-is. Enabling loop detection by default is a narrow maintainable direction, but this PR also hard-blocks generic repeats without no-progress evidence and leaves docs/changelog incomplete; the safer path is a smaller rebased default flip plus synchronized surfaces.
