## Summary

Keep open. Current main still leaves `tools.loopDetection.enabled` off by default, so the intended default-enable fix is not implemented, but this PR is not ready as-is because it adds args-only generic hard blocking, only partially updates the docs, lacks the required changelog entry, and is dirty against its base.

## What This Changes

This PR flips `tools.loopDetection.enabled` to true by default, expands loop-detection tests, normalizes exec outcome hashing, escalates generic repeats to critical, and updates config/type/docs default metadata.

## Best Possible Solution

Land one rebased canonical change that flips the default to true, preserves explicit opt-out, keeps hard blocks tied to no-progress or separately approved hard-abort semantics, and updates docs, config metadata, generated types, tests, and changelog together.

## Reproduction Assessment

Yes, at source level. Current main resolves omitted loop-detection config through `enabled: false`, and the current unit test asserts repeated identical calls are not detected by default.

## Solution Assessment

No as-is. Enabling the default is a narrow maintainable direction, but this branch also changes generic repeat warnings into critical blocks without no-progress evidence and leaves docs/changelog incomplete.
