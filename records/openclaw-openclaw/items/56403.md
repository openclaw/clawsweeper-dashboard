## Summary

Keep open. Current main still ships `tools.loopDetection.enabled` as default-disabled, so the user-visible problem remains, but this PR is not merge-ready because it introduces args-only `generic_repeat` critical blocking for changing outcomes and only partially updates the default-enabled docs/config surface.

## What This Changes

The PR flips tool-loop detection to enabled by default, adds repeated-exec loop regression tests, changes exec outcome hashing, escalates consecutive `generic_repeat` detections to critical, and updates config/type/docs default text.

## Best Possible Solution

Land one rebased canonical fix that enables loop detection by default, preserves explicit opt-out, keeps current main's stable exec outcome hashing and run-scoped history, uses no-progress evidence for critical blocking or leaves generic repeats warning-only, and updates every user-facing config/docs/generated surface consistently.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: omitted loop-detection config resolves to `enabled: false`, and the existing unit test records repeated identical calls with no config and asserts `stuck` remains false.

## Solution Assessment

No as-is. The default-enable direction is narrow and useful, but the PR should not hard-block args-only generic repeats when results are changing, and the docs/config default metadata needs a complete sync before merge.
