## Summary

Keep open. Current main still leaves tool-loop detection disabled by default, so the core fix is not implemented, but this PR is not merge-ready because it adds a generic-repeat critical block without no-progress evidence and leaves user-facing docs/config guidance inconsistent.

## What This Changes

The PR branch enables `tools.loopDetection.enabled` by default, expands loop-detection tests, adjusts exec outcome hashing and generic-repeat escalation, and updates config/type/docs default metadata.

## Best Possible Solution

Land one rebased canonical fix that enables loop detection by default, preserves explicit opt-out, keeps generic repeats warning-only or gates critical blocks on stable no-progress evidence, retains current main's exec/run-scoping improvements, and updates every config, generated, type, docs, and changelog surface consistently.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: repeated identical tool calls with omitted loop-detection config still resolve through `enabled: false`, and the unit test asserts that default-off behavior.

## Solution Assessment

No as-is. The default-enable direction is narrow and useful, but the PR should not hard-block generic repeats without no-progress evidence, and its docs/config updates need to be complete before merge.
