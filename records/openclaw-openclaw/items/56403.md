## Summary

Keep open. Current main still ships `tools.loopDetection.enabled` as `false`, so the core request is not obsolete, and the author supplied useful real-session reproduction context. The PR is not merge-ready because it also hard-blocks `generic_repeat` on consecutive same tool+args calls without requiring no-progress result evidence, and its docs update leaves user-facing loop-detection guidance internally contradictory.

## What This Changes

The PR enables `tools.loopDetection.enabled` by default, adds loop-detection tests, normalizes exec outcome hashing, escalates consecutive `generic_repeat` detections to critical blocking, and updates config/type/docs default metadata.

## Best Possible Solution

Land one rebased canonical fix that enables loop detection by default only with maintainer-approved safety semantics, preserves explicit opt-out, keeps current main's stable exec outcome hashing and run-scoped history, gates critical blocking on no-progress evidence or leaves generic repeats warning-only, and updates every user-facing config, type, generated, and docs surface consistently.

## Reproduction Assessment

Yes. Current main has a source-level and unit-test reproduction: repeated identical calls with omitted config still resolve through `enabled: false`, and the default-disabled test asserts no loop is detected.

## Solution Assessment

No as-is. The default-enable direction is narrow and useful, but this PR should not hard-block generic repeats without result no-progress evidence, and it needs a complete docs/config sync before merge.
