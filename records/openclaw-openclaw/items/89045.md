## Summary

Keep open: this PR targets a real message-loss/session-state bug and is not obsolete, but the current terminal-recovery guard still has a pre-mark race that can fail a fresh recovery operation, and real behavior proof remains mock-only.

## What This Changes

The PR clears terminal failed/timeout/killed session state on visible inbound turns, retries active-run admission after clearing a stale reply operation, adds a terminalRecovery guard, mirrors agent API recovery, and logs zero-count visible dispatch warnings.

## Best Possible Solution

Land a recovery path that atomically identifies and clears only the captured stale terminal operation, preserves session/transcript continuity, and includes focused race coverage plus redacted live channel or gateway proof.

## Reproduction Assessment

Yes, from source inspection for the PR-introduced race: visible pre-dispatch admission can skip immediately, the retry admission is awaited, and the clear resolves the current operation by session id before the new recovery operation is marked. I did not run a live channel repro in this read-only review.

## Solution Assessment

No as written: the recovery direction is plausible and in the right core surfaces, but clearing by session id plus post-await marking does not fully protect concurrent recovery turns. The safer fix is exact-operation clearing or atomic terminal-recovery marking at operation creation/admission time.
