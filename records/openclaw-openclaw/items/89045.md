## Summary

Keep this PR open: the core terminal-session recovery is useful and the real-runtime proof is sufficient, but the current head still has a blocking race where a second visible recovery turn can force-fail the first valid recovery turn when no stale active operation existed at admission time.

## What This Changes

The PR clears recoverable terminal session lifecycle state on visible inbound turns, retries reply admission after clearing matching stale reply operations, and adds zero-count visible dispatch diagnostics with focused tests.

## Best Possible Solution

Keep the recovery design, but mark or otherwise protect every visible terminal recovery admission, including the first no-stale admission, and add a regression test for that race before merge.

## Reproduction Assessment

Yes. Source inspection shows a no-stale terminal snapshot path where the first visible turn owns an unmarked operation and a second visible turn can force-clear it before the session store is cleared.

## Solution Assessment

No. The current approach is close, but the best fix must protect first terminal recovery admissions too, not only operations admitted after clearing a stale leftover.
