## Label Justifications

- `P1`: The linked bug silently drops visible group-chat messages after a terminal session state, which is an urgent channel workflow regression.
- `merge-risk: 🚨 session-state`: The patch changes persisted terminal session recovery and can mis-clear a fresh active operation in a concurrent recovery path.

## Summary

Keep this PR open: it targets a real P1 silent-drop/session-state bug, but the current patch can still fail a fresh recovery turn when two visible inbound messages race against the same terminal session, and the external PR still has only focused-test proof rather than real channel proof.

## What This Changes

The branch clears terminal session lifecycle fields on visible inbound turns, retries reply admission after clearing a stale active reply operation, mirrors recovery in the gateway agent API path, and adds a zero-count visible dispatch warning.

## Best Possible Solution

Make terminal recovery a single owned transition that only clears an operation proven to be the leftover terminal run, add an integrated two-visible-turn regression, then attach redacted real channel proof of recovery.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: two visible inbound dispatches can share a terminal persisted session snapshot, the second can hit the PR's terminal active-run clearing branch, and the busy path still returns a no-op result. I did not run tests because this review was read-only.

## Solution Assessment

No. The PR is aimed at the right owner boundary, but clearing any active operation while the store entry is terminal is too broad; the best fix should distinguish the old stuck operation from a fresh recovery turn and prove that with an integrated regression.
