## Summary

Keep this PR open. The recovery is still needed, but the current patch can force-clear a valid first visible recovery turn when a second visible turn races the same terminal session snapshot and no stale active operation existed.

## What This Changes

The PR clears recoverable terminal lifecycle fields on visible inbound turns, retries reply admission after clearing matching stale operations, adds a terminal-recovery operation flag, and logs zero-count visible dispatch warnings with regression coverage.

## Best Possible Solution

Land the core recovery only after the force-clear path protects every fresh terminal recovery operation, with focused concurrency coverage and maintainer acceptance of the session-state/message-delivery semantics.

## Reproduction Assessment

Yes. From source, seed a terminal session with no active operation, start one visible dispatch so it admits an unmarked operation, then race a second visible dispatch before session recovery clears the store; the second dispatch can force-fail the first.

## Solution Assessment

No. The fix is in the right subsystem, but the best version must protect any visible terminal recovery operation, not only operations admitted after clearing a stale leftover.
