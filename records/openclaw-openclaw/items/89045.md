## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Verify the active operation before force-clearing it:** `src/auto-reply/reply/dispatch-from-config.ts:1254-1257`
  - body: This branch treats the terminal store snapshot as proof that the active operation is the stale failed run, but `admission.activeOperation` can be any active run for the same session key. If a concurrent `/new` or fresh visible turn has already started under a new session id, this will call `forceClearReplyRunBySessionId` on that newer operation and drop a legitimate in-flight reply. Require the active operation's session id to match the terminal entry being recovered, and add a rotated-session race regression.
  - confidence: 0.87
- **[P2] Recheck the fresh entry before clearing lifecycle state:** `src/gateway/server-methods/agent.ts:1786-1789`
  - body: `recoverTerminalSession` is computed from the originally loaded entry, but `buildSessionPatch(freshEntry)` is also called inside the store updater after another writer may have changed the same session id to `running`. In that case `shouldClearTerminalState` remains true and clears `status`, `startedAt`, `endedAt`, `runtimeMs`, and `abortedLastRun` from the fresh running entry. Base the clear decision on `freshEntry` still being terminal, not only on the initial snapshot.
  - confidence: 0.84
