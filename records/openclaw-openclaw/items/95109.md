## Summary

Keep open: current main and v2026.6.9 still run Tavily's strict configured-secret read before the documented `TAVILY_API_KEY` fallback, and open candidate PRs already own the remaining fix path.

## What This Changes

The issue asks OpenClaw to let Tavily `tavily_search` and `tavily_extract` fall back to `process.env.TAVILY_API_KEY` when the configured env SecretRef is unavailable in the active runtime snapshot.

## Best Possible Solution

Land one guarded Tavily resolver fix with regression coverage, then close this issue from the merged PR.

## Reproduction Assessment

Yes. Source inspection shows current main reaches strict Tavily SecretRef normalization before the documented `TAVILY_API_KEY` fallback, and both Tavily search and extract use that resolver.

## Solution Assessment

Yes. A narrow plugin-owned resolver fix is the maintainable boundary; maintainers should prefer a guarded inspect-mode shape so explicit nonmatching SecretRefs are not silently bypassed by ambient env.
