## Summary

Keep open: current main and v2026.6.10 still call Tavily's strict SecretRef normalization before the documented `TAVILY_API_KEY` environment fallback, and the linked fix PRs remain unmerged.

## What This Changes

The issue reports that `tavily_search` and `tavily_extract` fail with `UnresolvedSecretInputError` when Tavily's configured env SecretRef cannot be inline-resolved before the `process.env.TAVILY_API_KEY` fallback runs.

## Best Possible Solution

Land one guarded Tavily-local inspect-mode resolver that only falls back to ambient `TAVILY_API_KEY` when the configured env SecretRef is the matching permitted Tavily credential, with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level. Current main still routes Tavily configured `apiKey` through strict SecretRef normalization before the documented `TAVILY_API_KEY` fallback, so an unresolved configured SecretRef can throw before fallback.

## Solution Assessment

Yes, the right fix boundary is the Tavily plugin resolver. The safest implementation should use inspect-mode SecretRef resolution and preserve explicit nonmatching SecretRef boundaries instead of a broad catch-and-fallback.
