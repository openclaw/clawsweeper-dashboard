## Summary

Current main still has Tavily resolving configured `webSearch.apiKey` through a strict SecretRef helper before reaching the env fallback, so the reported bug remains plausible from source. The issue should stay open because the same reporter has an open candidate fix at https://github.com/openclaw/openclaw/pull/95110.

## What This Changes

The issue reports that Tavily SecretRef-configured API keys throw before the plugin's `process.env.TAVILY_API_KEY` fallback can run.

## Best Possible Solution

Keep this issue open and review https://github.com/openclaw/openclaw/pull/95110 as the candidate plugin-local fix for Tavily SecretRef/env fallback behavior.

## Reproduction Assessment

Source-reproducible: Tavily currently calls a strict SecretRef normalizer before its `process.env.TAVILY_API_KEY` fallback, so an unresolved configured ref can throw first. I did not run a live CLI repro under this read-only review constraint.

## Solution Assessment

Yes, the correct direction is plugin-local: update Tavily's key resolver so an unresolved env-backed SecretRef can fall through to the existing env fallback without changing global strict secret semantics. The exact PR shape still needs review against the Firecrawl inspect-mode pattern.
