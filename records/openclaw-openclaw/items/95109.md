## Summary

Keep open: current main and v2026.6.8 still route Tavily's configured API-key SecretRef through strict resolution before the existing environment fallback, and https://github.com/openclaw/openclaw/pull/95112 is an open closing fix candidate.

## What This Changes

Tavily should fall back to `TAVILY_API_KEY` when `plugins.entries.tavily.config.webSearch.apiKey` is an env-backed SecretRef that is configured but unavailable in the active runtime snapshot.

## Best Possible Solution

Review and land one focused Tavily resolver fix with regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. From source, an unresolved SecretRef object passed to Tavily's `normalizeConfiguredSecret` reaches strict `normalizeResolvedSecretInputString` before the `process.env.TAVILY_API_KEY` fallback can run; I did not run the live CLI command in this read-only review.

## Solution Assessment

Yes. The best path is a plugin-local Tavily resolver fix that lets the intended configured-but-unavailable credential case fall through to the existing env fallback without weakening global strict SecretRef semantics.
