Keep open: current main and v2026.6.8 still route Tavily's configured API-key SecretRef through strict resolution before the existing `TAVILY_API_KEY` fallback, and https://github.com/openclaw/openclaw/pull/95112 is an open candidate fix with closing syntax for this issue.

## What This Changes

The issue asks for Tavily's configured SecretRef API-key path to fall through to `TAVILY_API_KEY` instead of throwing before the fallback.

## Best Possible Solution

Review and land one focused Tavily resolver fix that preserves global strict SecretRef semantics, adds regression coverage for the env-backed SecretRef fallback, and then close this issue after merge.

## Reproduction Assessment

Yes from source: an unresolved SecretRef object passed to Tavily's `normalizeConfiguredSecret` reaches strict `normalizeResolvedSecretInputString` before the `process.env.TAVILY_API_KEY` fallback can run. I did not run the live CLI command in this read-only review.

## Solution Assessment

Yes. The maintainable direction is a plugin-local Tavily resolver change that lets only the intended unresolved credential case fall through to the existing env fallback, with focused regression coverage.
