## Summary

Keep open: current main and the latest release still let Tavily's configured API-key SecretRef throw in strict resolution before the documented `TAVILY_API_KEY` fallback can run, and an open closing PR already owns the fix path.

## What This Changes

The issue reports that Tavily search and extract fail on an unresolved `plugins.entries.tavily.config.webSearch.apiKey` SecretRef instead of falling back to `process.env.TAVILY_API_KEY`.

## Best Possible Solution

Review and land one focused Tavily resolver fix with regression coverage, then close this issue as implemented by the merged PR.

## Reproduction Assessment

Yes. Source inspection shows an unresolved Tavily API-key SecretRef reaches strict `normalizeResolvedSecretInputString` before the existing `TAVILY_API_KEY` fallback can run; I did not run the live command in this read-only review.

## Solution Assessment

Yes. A narrow Tavily resolver fix that treats unresolved configured refs as unavailable only for this fallback path is the smallest maintainable solution; current open PRs are candidate implementations.
