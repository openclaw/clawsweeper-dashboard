## Summary

Keep open: current main and v2026.6.10 still run Tavily's configured SecretRef through strict normalization before the documented `TAVILY_API_KEY` environment fallback, while the linked fix PRs remain open candidate fixes.

## What This Changes

The issue reports that Tavily's plugin-local API-key resolver throws on an unresolved env-backed SecretRef before it can fall back to `process.env.TAVILY_API_KEY`.

## Best Possible Solution

Land one guarded Tavily-local inspect-mode resolver that only falls back to ambient `TAVILY_API_KEY` for the matching permitted env SecretRef, with regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes, at source level. Current main still routes Tavily configured `apiKey` through strict SecretRef normalization before the documented `TAVILY_API_KEY` fallback, so an unresolved configured SecretRef can throw before fallback.

## Solution Assessment

Yes, the right fix boundary is the Tavily plugin resolver. The safest implementation should use inspect-mode SecretRef resolution and preserve explicit nonmatching SecretRef boundaries instead of a broad catch-and-fallback.
