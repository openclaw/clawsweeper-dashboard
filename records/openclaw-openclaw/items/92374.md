## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed on primary channel reply paths, affecting real safety and delivery workflows.
- `impact:security`: Safety, DLP, compliance, and audit plugins using `message_sending` can silently fail open on outbound agent replies.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing messages to be delivered when a hook expected suppression or modification.

## Summary

Keep this open: current main and v2026.6.9 still skip the documented `message_sending` hook when a channel supplies `beforeDeliver`, and the open linked PRs are candidate fixes rather than merged replacements.

## What This Changes

The issue requests that plugin `message_sending` hooks run for channel agent replies even when a channel supplies `beforeDeliver`, including Telegram delivery paths.

## Best Possible Solution

Land one canonical fix that runs `message_sending` exactly once per visible outbound agent reply across shared dispatcher delivery and Telegram preview-finalized streaming, with focused regression tests and Telegram-equivalent proof.

## Reproduction Assessment

Yes. Current-main source deterministically chooses the configured `beforeDeliver` branch that excludes `message_sending`, and Telegram supplies that hook; I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes as a direction: restore the documented pre-send hook through one exactly-once delivery model. A dispatcher-only one-liner is too narrow because Telegram already has delivery-owned hook calls and a streamed-final path outside `deliverReplies`.
