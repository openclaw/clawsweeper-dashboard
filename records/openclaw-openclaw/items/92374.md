## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed on primary channel reply paths, affecting real safety and delivery workflows.
- `impact:security`: Safety, DLP, compliance, and audit plugins using `message_sending` can silently fail open on outbound agent replies.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing messages to be delivered when a hook expected suppression or modification.

## Summary

Keep open: current main and stable v2026.6.8 still skip `message_sending` when a channel supplies `beforeDeliver`, and the linked open PRs are candidate fixes rather than merged replacements.

## What This Changes

The issue reports that documented plugin `message_sending` hooks are skipped on channel agent-reply delivery when a channel provides `beforeDeliver`, leaving Telegram safety, rewrite, and cancel plugins inert.

## Best Possible Solution

Land one canonical fix that runs `message_sending` exactly once per visible reply across shared dispatch, Telegram delivery-owned sends, and streamed finalization, with focused regression tests and Telegram-equivalent proof.

## Reproduction Assessment

Yes. Source inspection on current main shows configured `beforeDeliver` excludes `message_sending`, and Telegram passes identity `beforeDeliver`; I did not run a live Telegram turn in this read-only review.

## Solution Assessment

Yes as a direction: restore the documented hook through one exactly-once pre-send ownership model. A dispatcher-only one-line change is too narrow because Telegram already has delivery-owned hook calls and a streamed-final path outside `deliverReplies`.
