## Label Justifications

- `P1`: A documented outbound interception contract is silently bypassed on primary conversational channel paths affecting real installations.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing unintended or unmodified messages to be delivered.
- `impact:security`: Safety, DLP, and compliance plugins can silently fail open on outbound agent replies.

## Summary

The bug is confirmed in current `main`: both shared dispatcher wrappers omit the documented `message_sending` hook whenever a channel supplies `beforeDeliver`, and Telegram preview finalization has an adjacent visible-send path that needs exactly-once coverage. The newer report is more specific and current than the older related issues, so it should remain open as the canonical repair target.

## What This Changes

Restore the documented plugin `message_sending` interception contract for channel agent replies, including Telegram custom dispatch and preview-finalized delivery.

## Best Possible Solution

Establish one canonical pre-send pipeline that composes channel transformation, `reply_payload_sending`, and `message_sending` exactly once before every visible reply, then cover configured dispatchers and Telegram preview finalization with focused regression tests.

## Reproduction Assessment

Yes from source inspection: on current `main`, any channel-provided `dispatcherOptions.beforeDeliver` deterministically selects a composition that excludes `message_sending`. A live Telegram run was not performed in this read-only review.

## Solution Assessment

Yes, a shared exactly-once delivery fix is the narrowest maintainable solution; a Telegram-only workaround or warning would leave the documented cross-channel plugin contract broken.
