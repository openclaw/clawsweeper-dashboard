## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed on primary channel reply paths, affecting real safety and delivery workflows.
- `impact:security`: Safety, DLP, compliance, and audit plugins using `message_sending` can silently fail open on outbound agent replies.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing messages to be delivered when a hook expected suppression or modification.

## Summary

Keep this open: current main and the latest release still skip the documented `message_sending` hook when a channel supplies `beforeDeliver`, and open linked PRs are candidate fixes rather than merged replacements.

## What This Changes

The issue asks OpenClaw to stop silently bypassing documented `message_sending` plugin hooks on channel agent-reply delivery paths, including Telegram paths that supply `beforeDeliver`.

## Best Possible Solution

Land one canonical fix that runs `message_sending` exactly once per visible outbound reply across shared dispatcher delivery, Telegram delivery-owned sends, and Telegram preview-finalized streaming, with focused regression tests and Telegram-equivalent proof.

## Reproduction Assessment

Yes. The current-main source deterministically selects the configured `beforeDeliver` branch that excludes `message_sending`, and Telegram supplies that hook; no live Telegram run was performed in this read-only review.

## Solution Assessment

Yes as a direction: restore the documented pre-send hook through one exactly-once delivery model. A dispatcher-only one-liner is too narrow because Telegram already has delivery-owned hook calls and a streamed-final path outside `deliverReplies`.
