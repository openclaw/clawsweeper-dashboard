## Summary

Keep open: current main still bypasses `message_sending` whenever a channel supplies `beforeDeliver`, and the latest shipped tag I checked carries the same branch. The linked fixing pull requests are still open, so the issue should remain the canonical target until one lands.

## What This Changes

The issue reports that documented plugin `message_sending` hooks are skipped on channel agent-reply delivery when a channel provides its own `beforeDeliver`, especially Telegram's identity hook.

## Best Possible Solution

Land one canonical shared dispatcher fix that composes channel `beforeDeliver`, `reply_payload_sending`, and `message_sending` exactly once with focused dispatch and Telegram regression coverage, then close redundant fix PRs after the chosen branch lands.

## Reproduction Assessment

Yes from source inspection: current main deterministically chooses a configured `beforeDeliver` branch that excludes `message_sending`, and Telegram supplies that branch-triggering hook. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is shared dispatcher composition, not a Telegram-only workaround or a warning that leaves the documented plugin contract broken. The open PRs should be reviewed against exactly-once ordering and the preview-finalized scope.
