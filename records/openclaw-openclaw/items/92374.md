## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed on primary channel reply paths used by real deployments.
- `impact:security`: Safety, DLP, and compliance plugins that depend on `message_sending` can fail open without operator-visible errors.

## Summary

Keep this open: current main and latest release still omit `message_sending` from the configured `beforeDeliver` chain, and the open closing PRs are implementation candidates rather than merged fixes.

## What This Changes

The issue reports that documented `message_sending` plugin hooks are bypassed on channel agent-reply delivery when a channel supplies its own `beforeDeliver`.

## Best Possible Solution

Land one canonical dispatcher fix that composes channel `beforeDeliver`, `reply_payload_sending`, and `message_sending` exactly once before visible delivery, then close this issue and duplicate fix PRs.

## Reproduction Assessment

Yes, source-level: current main deterministically selects a hook chain that omits `message_sending` whenever `dispatcherOptions.beforeDeliver` exists, and Telegram supplies such a hook. I did not run a live Telegram session in this read-only review.

## Solution Assessment

Yes, the shared dispatcher composition is the right fix boundary; a Telegram-only workaround would leave the documented cross-channel plugin contract fragile.
