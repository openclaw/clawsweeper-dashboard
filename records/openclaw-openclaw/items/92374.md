## Label Justifications

- `P1`: A documented outbound interception contract is silently bypassed on primary channel reply paths affecting real safety/filter plugin workflows.
- `impact:security`: Safety, DLP, and compliance plugins can silently fail open when outbound agent replies bypass `message_sending`.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing unintended or unmodified messages to be delivered.

## Summary

Keep open: current `main` and the latest release still skip the documented `message_sending` gate when a channel supplies `beforeDeliver`, and the issue has open closing PRs that should be reviewed or landed before the issue closes.

## What This Changes

The issue asks OpenClaw to run plugin `message_sending` hooks on channel agent-reply delivery paths, including configured `beforeDeliver` and Telegram preview-finalized paths.

## Best Possible Solution

Review and land a single canonical fix, likely through https://github.com/openclaw/openclaw/pull/93216 if it clears its draft/proof/API concerns, then close this issue after the merge.

## Reproduction Assessment

Yes from source inspection: a configured `beforeDeliver` deterministically selects a chain that excludes `message_sending`, and the current regression test asserts the skip. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, the maintainable direction is a shared exactly-once pre-send pipeline plus Telegram preview-finalized coverage. The issue should not close until one of the open fixing PRs lands or maintainers choose a different canonical path.
