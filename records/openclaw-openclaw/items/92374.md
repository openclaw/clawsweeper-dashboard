## Label Justifications

- `P1`: A shipped, documented outbound plugin seam is silently bypassed on high-volume Telegram agent replies, affecting real safety workflows.
- `impact:security`: The bypass can leave outbound DLP, compliance rewriting, and content-filter controls inactive without operator warning.
- `impact:message-loss`: The hook controls cancellation and rewriting, so bypass can send content that a plugin intended to suppress or alter.

## Summary

Current `main` and the latest release still bypass the documented `message_sending` hook whenever a channel supplies `dispatcherOptions.beforeDeliver`; Telegram supplies that callback on conversational and native-command reply paths, and no canonical fix PR exists.

## What This Changes

Restore documented `message_sending` hook enforcement for channel agent replies, including Telegram, and verify whether streamed preview-finalized replies require the same enforcement.

## Best Possible Solution

Compose channel-specific preprocessing, `reply_payload_sending`, and `message_sending` in one generic exactly-once delivery pipeline, then extend streamed-final enforcement only if a focused regression proves an externally visible send bypasses that pipeline.

## Reproduction Assessment

Yes from deterministic current-main source inspection: Telegram supplies `beforeDeliver`, and that dispatcher branch excludes `buildMessageSendingBeforeDeliver`. A focused dispatcher test can reproduce the failure without Telegram credentials, while the streamed-final subclaim still needs separate proof.

## Solution Assessment

Yes for a generic shared-dispatch composition fix; it is narrower and more maintainable than Telegram-specific logic or a warning-only workaround. Preview-finalized delivery should be changed only if its distinct visible-send path is proven.
