## Label Justifications

- `P1`: A documented outbound interception contract is skipped on primary channel reply paths used by real installations.
- `impact:security`: The skipped hook can silently disable safety, DLP, or compliance plugins before outbound delivery.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing unintended or unmodified outbound messages to be delivered.

## Summary

Keep open: current main still bypasses the documented `message_sending` hook whenever a channel supplies `beforeDeliver`, and an open fixing PR already references this issue with closing syntax, so the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue reports that plugin `message_sending` hooks are skipped on channel agent-reply delivery paths, especially Telegram paths that supply `dispatcherOptions.beforeDeliver`.

## Best Possible Solution

Land one focused shared dispatcher fix that composes channel `beforeDeliver`, `reply_payload_sending`, and `message_sending` exactly once, then split any remaining preview-finalizer or diagnostic warning work into a narrower follow-up if needed.

## Reproduction Assessment

Yes. Current-main source and tests show that channel-provided `beforeDeliver` deterministically selects a composition that excludes `buildMessageSendingBeforeDeliver`; no live Telegram repro was run in this read-only review.

## Solution Assessment

Yes for the central dispatcher bug. The shared dispatcher is the narrow owner boundary because Telegram and native-command paths enter through the same channel-supplied `beforeDeliver` contract; preview-finalized streaming should be split if it needs separate treatment.
