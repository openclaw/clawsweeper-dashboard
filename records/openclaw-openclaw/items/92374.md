## Label Justifications

- `P1`: A documented outbound interception hook is silently skipped on primary channel reply paths that safety plugins may rely on.
- `impact:security`: Safety, DLP, and compliance plugins using `message_sending` can fail open on outbound agent replies.
- `impact:message-loss`: Skipped hook cancellation or rewrite decisions can cause unintended or unmodified messages to be delivered.

## Summary

Keep open: current `main` and the latest `v2026.6.6` release still omit `message_sending` from the configured channel `beforeDeliver` branch, and the linked closing PRs are still open. This should remain the canonical issue until maintainers choose and land one dispatcher fix.

## What This Changes

The issue asks OpenClaw to run plugin `message_sending` hooks on channel agent-reply delivery paths even when the channel supplies its own `beforeDeliver` hook.

## Best Possible Solution

Land one canonical dispatcher fix that composes channel `beforeDeliver`, `reply_payload_sending`, and `message_sending` exactly once before visible delivery, then close this issue and supersede duplicate PRs.

## Reproduction Assessment

Yes, from source inspection: current `main` deterministically excludes `message_sending` whenever `dispatcherOptions.beforeDeliver` is present, and Telegram supplies such a hook. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes: the shared dispatcher composition layer is the narrowest maintainable fix, with Telegram/live proof and exactly-once ordering coverage before merge. Keeping the issue open is correct while the closing PRs remain unmerged.
