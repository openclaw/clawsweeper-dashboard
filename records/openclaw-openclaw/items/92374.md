## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed on primary channel reply paths, affecting real safety and delivery workflows.
- `impact:security`: Safety, DLP, compliance, and audit plugins using `message_sending` can silently fail open on outbound agent replies.
- `impact:message-loss`: Plugin cancellation and rewrite decisions can be ignored, causing messages to be delivered when a hook expected suppression or modification.

## Summary

Keep this open: current main and the latest release still skip the documented `message_sending` hook whenever a channel supplies `beforeDeliver`, and the open linked PRs are candidate fixes rather than merged replacements.

## What This Changes

Restore documented plugin `message_sending` coverage for channel agent replies when channels supply dispatcher `beforeDeliver`, including Telegram direct and streamed-final paths.

## Best Possible Solution

Land one canonical fix that runs `message_sending` exactly once per visible reply across shared dispatch, Telegram delivery-owned sends, and streamed finalization, with focused tests and real or equivalent Telegram proof.

## Reproduction Assessment

Yes. Current-main source deterministically selects the configured `beforeDeliver` branch that excludes `message_sending`, and Telegram supplies such hooks; no live Telegram run was performed in this read-only review.

## Solution Assessment

Yes as a direction: restore the documented hook through one exactly-once pre-send pipeline. A dispatcher-only one-liner is not the best fix because Telegram already has delivery-owned hook calls and a streamed-final path outside `deliverReplies`.
