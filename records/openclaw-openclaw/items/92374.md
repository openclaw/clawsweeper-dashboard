## Label Justifications

- `P1`: A documented outbound interception hook is silently bypassed on primary channel agent replies, affecting real safety and delivery workflows.
- `impact:security`: Safety, DLP, and compliance plugins using `message_sending` can fail open while appearing to be registered normally.
- `impact:message-loss`: Plugin rewrite and cancellation decisions can be ignored, allowing unintended or unmodified messages to be delivered.

## Summary

Keep open: current main and the latest release still omit the documented `message_sending` hook whenever a channel supplies `beforeDeliver`, and this issue already has open closing PRs that should be reviewed instead of closing the issue early.

## What This Changes

The issue asks OpenClaw to restore plugin `message_sending` hook coverage on channel agent-reply delivery paths that use a channel-supplied `beforeDeliver`, especially Telegram.

## Best Possible Solution

Land one canonical shared dispatcher fix that runs channel normalization, `reply_payload_sending`, and `message_sending` exactly once before visible delivery, then close this issue and duplicate PRs after verifying Telegram or equivalent channel behavior.

## Reproduction Assessment

Yes, from source inspection: current main deterministically skips `message_sending` whenever `dispatcherOptions.beforeDeliver` is set, and Telegram supplies such a hook. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, restoring the hook chain in the shared dispatcher layer is the narrowest maintainable fix for the central bug. A Telegram-only workaround or warning would leave the cross-channel plugin contract broken.
