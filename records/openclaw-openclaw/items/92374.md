## Summary

This issue is still necessary: current main and latest v2026.6.6 still skip the documented message_sending hook when a channel supplies beforeDeliver, and the formal closing PR remains open and unmerged.

## What This Changes

The issue reports that plugin message_sending hooks are bypassed when channel agent-reply delivery supplies its own beforeDeliver hook.

## Best Possible Solution

Land one focused shared-dispatcher fix that composes channel beforeDeliver, reply_payload_sending, and message_sending exactly once, then close this issue after the merge.

## Reproduction Assessment

Yes from source inspection. Current main deterministically excludes buildMessageSendingBeforeDeliver whenever dispatcherOptions.beforeDeliver is present, and the adjacent test still asserts runMessageSending is not called in that branch.

## Solution Assessment

Yes. The best repair is the shared dispatcher composition rather than Telegram-specific hook wiring, because the invariant belongs to every channel that supplies beforeDeliver.
