## Summary

Keep this draft PR open. Current main still replays node `exec.*` events from heartbeat-isolated sessions into the system-event queue and requests another `exec-event` heartbeat wake, while the PR’s marker guard is narrow and consistent with the existing isolated-heartbeat session contract. I found no functional or security blocker, but the user-facing fix needs the required `CHANGELOG.md` entry before merge.

## What This Changes

The PR adds a heartbeat-isolated session marker guard to gateway node exec event handling and extends the focused node-event tests with a suppression regression case.

## Best Possible Solution

Land or adapt the marker guard plus regression test, add a single required changelog entry, and preserve existing enqueue/wake behavior for normal non-heartbeat node exec notifications.

## Reproduction Assessment

Yes. A focused reproduction is to mock `loadSessionEntry` for `agent:main:main:heartbeat` with `heartbeatIsolatedBaseSessionKey`, send an `exec.finished` node event, and observe current main still calling `enqueueSystemEvent` and `requestHeartbeatNow` because it ignores the marker.

## Solution Assessment

Yes, after adding the changelog entry. Checking the stored session marker in `server-node-events` before enqueueing/waking is the narrowest maintainable fix for this feedback path and avoids changing scheduler policy or normal exec-event notifications.
