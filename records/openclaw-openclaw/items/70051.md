## Summary

Keep open: the PR targets a real gateway heartbeat self-wake bug, but the current patch is not merge-ready because its guard would rebase ahead of exec-event authorization, its regression test is stale against current main, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a gateway guard to skip exec system-event enqueueing and heartbeat wake requests for heartbeat-isolated sessions, plus a focused server-node-events regression test.

## Best Possible Solution

Move the suppression after successful exec-event authorization, keep it before notify/enqueue/wake work, update the regression to use the current authorized harness, and add redacted live gateway proof showing no second wake.

## Reproduction Assessment

Yes, at source level. Current main stores heartbeatIsolatedBaseSessionKey for isolated heartbeat sessions, while the gateway exec lifecycle handler ignores that marker and still enqueues an exec event plus an exec-event wake after authorization.

## Solution Assessment

No, not as currently proposed. Marker-based suppression is the right narrow direction, but it must run after current exec-event authorization and cleanup, and the regression needs to exercise the authorized path on current main.
