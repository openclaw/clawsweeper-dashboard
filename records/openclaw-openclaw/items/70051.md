## Summary

Keep this draft PR open. Current main still replays node exec lifecycle events through the system-event queue and requests an exec-event heartbeat wake without checking the heartbeat-isolated session marker, and the related heartbeat fixes do not supersede this narrower feedback-loop case. I found no blocking review finding or security concern in the provided two-file diff, but the behavior still needs maintainer review before merge.

## What This Changes

The PR adds a guard in `src/gateway/server-node-events.ts` to suppress node `exec.started`, `exec.finished`, and `exec.denied` system-event replay for marked synthetic heartbeat-isolated sessions, plus a focused regression test in `src/gateway/server-node-events.test.ts`.

## Best Possible Solution

Keep the PR open and either land or adapt the narrow guard plus regression test after maintainer confirmation, preserving the existing enqueue/wake behavior for normal node exec notifications.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to mock `loadSessionEntry` for `agent:main:main:heartbeat` with `heartbeatIsolatedBaseSessionKey` and send an `exec.finished` node event; current main still reaches `enqueueSystemEvent` and `requestHeartbeatNow` because it ignores the marker.

## Solution Assessment

Yes, with maintainer confirmation. The proposed marker check is the narrowest maintainable place to prevent node-event self-wakes for synthetic heartbeat sessions while leaving normal session behavior intact.
