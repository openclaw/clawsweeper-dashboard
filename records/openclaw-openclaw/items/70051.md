## Summary

Keep open: current main still appears to have the heartbeat-isolated exec self-wake path, but this draft PR is stale against the current exec-event authorization path and still lacks after-fix real behavior proof.

## What This Changes

The branch adds a heartbeat-isolated-session guard in gateway node exec-event handling and a focused regression test for suppressing enqueue/wake on a synthetic `:heartbeat` session.

## Best Possible Solution

Rebase the guard onto current main after `authorizeNodeSystemRunEvent`, keep normal non-heartbeat exec notifications unchanged, cover an authorized heartbeat-session exec event, and add redacted live gateway or terminal/log proof.

## Reproduction Assessment

Yes, at source level. Current main writes a `heartbeatIsolatedBaseSessionKey` marker on isolated heartbeat sessions but the gateway exec-event handler ignores that marker and still proceeds to enqueue and wake after authorization; I did not run a live gateway loop in this read-only review.

## Solution Assessment

No, not as currently proposed. Marker-based suppression is the right narrow direction, but it must be placed after current exec-event authorization and cleanup, and the regression should exercise an authorized heartbeat-session event.
