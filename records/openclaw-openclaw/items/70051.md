## Summary

Keep open: the focused bug is plausible, but this PR is not correct as-is against current main because the guard would run before exec-event provenance authorization and the new test uses an outdated heartbeat mock name; the external PR also still lacks after-fix real behavior proof.

## What This Changes

The branch adds a `heartbeatIsolatedBaseSessionKey` guard in `src/gateway/server-node-events.ts` plus regression coverage to skip exec system-event enqueue and heartbeat wake for synthetic heartbeat sessions.

## Best Possible Solution

Rebase the focused suppression onto current main after exec-event authorization, preserve normal non-heartbeat notifications, update the regression to current test helpers with an authorized heartbeat-session event, and provide redacted real gateway proof.

## Reproduction Assessment

Yes, at source level. Current main stores `heartbeatIsolatedBaseSessionKey` for isolated heartbeat sessions, while the exec lifecycle handler ignores that marker and still enqueues plus wakes after authorization; I did not run a live gateway loop.

## Solution Assessment

No, not as currently proposed. Marker-based suppression is the narrow direction, but it must run after `authorizeNodeSystemRunEvent` and terminal cleanup, and the test must be rebased to the current harness.
