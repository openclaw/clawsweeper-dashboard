## Summary

Keep open: the PR targets a real source-level delivery bug, but the implementation over-classifies relayed external contexts as WebChat and the external PR lacks real after-fix behavior proof.

## What This Changes

The PR adds a WebChat-specific bypass for Codex harness source-visible-reply defaults in `dispatch-from-config` and a regression test for direct internal WebChat delivery.

## Best Possible Solution

Narrow the WebChat exception to true internal WebChat turns, preserve Codex message-tool defaults for relayed external direct routes, add regression coverage for that boundary, and provide real after-fix WebChat/Codex proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main applies the Codex `message_tool` default to direct replies and WebChat is not an outbound delivery target. I did not run a live WebChat/Codex session in this read-only review.

## Solution Assessment

No. The direction is right, but the current predicate is too broad because relayed external routes can legitimately carry `Surface: "webchat"` while the active provider remains external.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Constrain the WebChat exception to internal turns:** `src/auto-reply/reply/dispatch-from-config.ts:299-302`
  - body: The new helper treats any WebChat value in Provider, Surface, or OriginatingChannel as internal WebChat. That is too broad: current code documents that `Surface` can be relayed metadata such as WebChat while `Provider` still owns tool routing, and existing dispatch coverage includes `Provider: "telegram", Surface: "webchat", OriginatingChannel: "telegram"` as an external route. In Codex direct sessions this drops the harness `message_tool` default and makes those non-WebChat replies automatic, which regresses the behavior the PR says it preserves.
  - confidence: 0.88
