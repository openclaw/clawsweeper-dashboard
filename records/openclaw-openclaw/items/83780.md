## Summary

Current main already has a foreground gateway entrypoint and systemd-aware restart behavior that avoids the reported detached fork path, so this can close as fixed on main but not yet proven shipped in v2026.5.18.

## What This Changes

The issue asks for a foreground/no-fork systemd mode or documented recommended systemd unit because systemd tracks the wrong process after gateway restart.

## Best Possible Solution

Keep the current foreground `openclaw gateway`/`openclaw gateway run` service path and rely on the canonical generated systemd unit instead of adding another no-fork flag.

## Reproduction Assessment

No live Debian/systemd reproduction was run. Source and tests give a high-signal current-main check: official systemd installs run the foreground gateway and restart handoff returns supervised/no-spawn under systemd markers.

## Solution Assessment

Yes. The maintainable solution is the existing foreground gateway command plus supervisor-aware respawn behavior and documented/generated systemd unit, rather than introducing a second foreground flag.
