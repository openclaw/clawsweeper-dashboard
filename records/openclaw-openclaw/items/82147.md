## Summary

Keep open: current main still has a documented WebChat current-run internal reply path, but source inspection shows the successful `message` tool result is not wired into the existing live WebChat projection/mirror pipeline.

## What This Changes

The issue requests that targetless `message.send` in an active WebChat/TUI `message_tool_only` run either render in the current client or return a clear warning/error instead of reporting invisible success.

## Best Possible Solution

Wire successful `internal-ui` source replies from the `message` tool into the existing `messagingToolSourceReplyPayloads`/final-payload mirror path, preserving external channel delivery and keeping WebChat out of normal outbound routing.

## Reproduction Assessment

Yes at source level: current main returns a successful WebChat `internal-source` send, while the subscriber/attempt path does not record the `sourceReply` payload needed for WebChat projection. A live WebChat/TUI reproduction was not executed in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to complete the existing internal-source reply plumbing rather than adding a new config knob or treating WebChat as a reusable outbound channel. If that path cannot be supported, the runner should fail before committing delivery evidence.
