## Summary

Keep open: current main still lacks the scoped next-heartbeat ghost-run warning, and this PR is the active linked implementation for the remaining issue. The diff looks functionally sound with supplied real run-log proof, but GitHub reports the head as dirty/unmergeable against current main.

## What This Changes

The branch adds a configurable `cron.ghostRunWarningThresholdMs` warning path that logs and persists `possible-main-next-heartbeat-ghost-run` for fast ok main-session `systemEvent` cron handoffs using `wakeMode="next-heartbeat"`, with schema, protocol, Swift model, docs, changelog, and tests.

## Best Possible Solution

Land this after rebase/conflict resolution and maintainer acceptance, keeping the warning scoped to the handoff path while leaving deeper cron delivery semantics to separate work.

## Reproduction Assessment

Yes. Source inspection on current main shows a `main` `systemEvent` job using the non-`now` wake path enqueues the event, requests a heartbeat, and records success without confirmed agent processing or any warning annotation.

## Solution Assessment

Yes for the narrow observability problem. The PR adds a scoped detector at the finished-event/run-log boundary with documented threshold control and regression coverage, avoiding a broader change to cron delivery semantics.
