## Summary

This PR should stay open because it targets a real ACPX test/runtime mismatch, but the proposed helper still computes the wrong dist path and the external-contributor real behavior proof is not observable.

## What This Changes

The PR updates the ACPX config test helper to accept built `dist` MCP server entrypoints when they exist and otherwise fall back to source-mode `tsx` args.

## Best Possible Solution

Mirror the resolver's absolute `dist/mcp/*.js` path construction in the test helper and keep the bridge assertions focused on the two MCP server configs, then add redacted terminal output showing the targeted ACPX config test passing where dist entries exist.

## Reproduction Assessment

Yes for the PR defect: source inspection shows the changed helper receives `src/mcp/...` relative paths, so its `"/src/"` replacement cannot compute the built `dist/mcp/*.js` path. The linked issue's full failure was not rerun because this is a read-only review, but the current-main resolver/test mismatch is clear from source and discussion.

## Solution Assessment

No. Accepting built-dist MCP resolution in the test is the right narrow direction, but this implementation should compute the same absolute dist path as the resolver before checking `fs.existsSync`.
