## Summary

Keep open: current main and v2026.6.8 still provide subagent registration, task-ledger status, active-subagent prompt context, on-demand status, and completion delivery, but not the requested opt-in periodic requester notice while a subagent remains running. This remains a feature/API product decision, with partial overlap but not duplication of the broader watchdog issue.

## What This Changes

The issue requests an opt-in liveness-notification policy for long-running subagent work that periodically tells the requester the run is still active.

## Best Possible Solution

Have maintainers choose a lean default-off liveness-notice contract, or explicitly fold this narrow request into the broader watchdog design while preserving existing pull/status and completion surfaces.

## Reproduction Assessment

Yes from source inspection: the current spawn schema, registration state, task policy, release tag, and docs do not expose or schedule periodic liveness notifications. This is not a broken existing behavior repro; it is source-reproducible feature absence.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. The narrowest maintainable direction appears to be a deterministic default-off notifier derived from existing subagent run or task state, with throttling and terminal cleanup tests.
