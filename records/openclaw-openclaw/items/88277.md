## Summary

Keep open: current main and v2026.6.8 have completion delivery, task-ledger, `subagents`, and `session_status` visibility, but they still do not implement an opt-in periodic requester notice while a native subagent remains running. The request is a feature/API-policy decision, not a cleanup-close candidate.

## What This Changes

Request an opt-in, deterministic periodic “still running” notice for active `runtime: "subagent"` runs delivered to the requester session.

## Best Possible Solution

Define a lean default-off subagent liveness-notice contract that reuses existing subagent run/task state and requester delivery without creating a broad parallel watchdog scheduler.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection confirms current main has pull/status and completion surfaces but no opt-in periodic requester-notice path.

## Solution Assessment

Unclear until maintainers choose the product/API boundary. The narrowest maintainable direction is a deterministic default-off notifier derived from existing subagent run or task state, with throttling and terminal cleanup tests.
