## Summary

Keep open: current main and v2026.6.9 provide pull-side subagent/task observability and completion delivery, but not the requested opt-in periodic requester notice while a native subagent is still running. This remains a feature/product decision because it would add model-facing session/subagent notification policy surface near existing task, heartbeat, and completion paths.

## What This Changes

The issue requests a default-off policy for `sessions_spawn`/native subagent work to send deterministic periodic running notices to the requester while the child run remains active.

## Best Possible Solution

Define one narrow default-off native subagent liveness-notice contract that reuses existing run/task state and requester delivery, or explicitly fold this slice into the broader task-watchdog design before implementation.

## Reproduction Assessment

Not applicable: this is a request for a new opt-in capability, not broken existing behavior. Source inspection on current main and v2026.6.9 found task/status/completion visibility but no timer-driven requester notice policy for running native subagents.

## Solution Assessment
