## Summary

Keep open: current main and the latest release still provide pull-side subagent/task visibility and terminal completion handoff, but not the requested opt-in periodic requester notice while a native subagent remains running. This is a feature/API-policy decision rather than a cleanup-close candidate.

## What This Changes

The issue requests an opt-in `sessions_spawn` policy that emits deterministic periodic running-state notices to the requester for long-running native subagents.

## Best Possible Solution

Keep the issue open until maintainers decide the API shape; if accepted, implement a narrow opt-in deterministic notifier that reuses subagent registry/task-ledger state, throttles per run id, and stops on terminal states without invoking an LLM.

## Reproduction Assessment

Yes for the source-level gap: current main exposes on-demand status and terminal completion delivery, but no opt-in scheduled running-notice parameter or notifier. No live runtime reproduction was run because this is a read-only feature review rather than a broken existing contract.

## Solution Assessment

Unclear as an implementation because the issue intentionally leaves field names open and the timed requester-delivery API needs maintainer product direction. The maintainable shape would be narrow, deterministic, opt-in, registry-backed, and separate from broader watchdog or model-generated heartbeat semantics.
