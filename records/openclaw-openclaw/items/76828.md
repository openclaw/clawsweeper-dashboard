## Summary

Keep open: current main still has a source-level commentary-only path that can leave a final-only turn with no visible user reply, even though the newer Telegram topic final-delivery case is partly covered by current topic fallback behavior and group visible-reply policy docs.

## What This Changes

The issue asks OpenClaw to prevent final-only Telegram/ACP turns from ending silently when the runtime produces only commentary-phase assistant text or when a topic-thread final reply is not visibly delivered.

## Best Possible Solution

Add a terminal no-final guard that emits a safe diagnostic or bounded final fallback without promoting arbitrary commentary, and keep Telegram topic natural delivery aligned with the documented visible-replies policy.

## Reproduction Assessment

Yes, at source level: current main suppresses commentary-phase assistant output and commentary-only fallback payloads, matching the reported silent final-only class. The later Telegram topic final_answer example still needs a config-controlled repro with group visibleReplies set to automatic.

## Solution Assessment

Unclear. Auto-promoting the last commentary chunk would fix silence but risks leaking internal progress text, so the safer maintainer direction is a terminal diagnostic or bounded fallback that preserves commentary suppression.
