## Summary

Keep open: current main still has a credible source-level path where commentary-only assistant output is intentionally suppressed, so a final-only turn with no final-answer chunk can remain silent. The later Telegram topic addendum overlaps with documented group/channel `message_tool` visible-reply policy unless reproduced with `messages.groupChat.visibleReplies: "automatic"`, but that does not resolve the original no-final/final-only failure.

## What This Changes

The issue asks OpenClaw to prevent silent user-visible drops when final-only or Telegram topic delivery sees no deliverable final assistant text.

## Best Possible Solution

Add a terminal no-final guard or constrained diagnostic/fallback for final-only turns that preserves commentary suppression, then separately verify topic final delivery with visible replies explicitly set to automatic.

## Reproduction Assessment

Yes, at source level: current main suppresses commentary-phase assistant output before user-visible delivery, and regression tests assert commentary-only completed messages stay non-visible. I did not run a live Telegram topic turn in this read-only review.

## Solution Assessment

Unclear. Auto-promoting the last commentary chunk would address silence but conflicts with the anti-leak behavior current tests enforce, so the safer direction is a maintainer-approved terminal diagnostic or tightly bounded fallback.
