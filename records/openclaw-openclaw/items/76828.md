## Summary

Keep open: current main still intentionally suppresses commentary-phase assistant output, and I did not find a turn-end guard that surfaces a safe final/fallback when a long turn emits no `final_answer`. The report is credible, but the fix needs maintainer judgment because a broad last-commentary promotion could undo shipped anti-leak behavior.

## What This Changes

The issue requests that `deliveryMode="final_only"` avoid total silence when a Telegram-backed long tool-loop turn emits only commentary-phase chunks and no final-phase chunk.

## Best Possible Solution

Add a narrow turn-end no-final guard for `final_only` sessions that preserves commentary suppression by default while surfacing a safe diagnostic or explicitly approved bounded fallback, with regression coverage for commentary-only long turns.

## Reproduction Assessment

Yes, at source level: current main has direct commentary suppression paths and tests proving commentary-only updates and message_end output do not deliver. I did not run a live Telegram long-loop reproduction, so this is not a full end-to-end reproduction.

## Solution Assessment

Unclear. Auto-promoting the last commentary chunk might solve silence, but it conflicts with shipped commentary-leak prevention; a safer maintainer-chosen fix is a terminal no-final diagnostic or tightly scoped fallback.
