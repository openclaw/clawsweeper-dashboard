## Summary

Keep open: current main still has a credible source-level path where commentary-only assistant output is intentionally suppressed and can leave a final-only user turn silent; the later Telegram topic evidence should be triaged separately from the documented group/tool-only visible-reply policy.

## What This Changes

The issue requests reliable user-visible delivery or diagnostics for Telegram/ACP turns that emit no final-phase reply, plus investigation of topic-thread natural final delivery when preview streaming is off.

## Best Possible Solution

Add a terminal no-final guard or safe diagnostic for final-only turns, then separately verify whether topic-thread final_answer delivery fails when messages.groupChat.visibleReplies is explicitly automatic.

## Reproduction Assessment

Yes, at source level: current main suppresses commentary-phase output and commentary-only fallback payloads, matching the reported silent final-only class. The later topic-thread final_answer claim needs one more config-controlled repro because group chats default to message-tool-only visible replies.

## Solution Assessment

Unclear. Auto-promoting the last commentary chunk would fix silence but risks leaking internal/progress text, so the narrower maintainable fix is a maintainer-approved terminal diagnostic or bounded fallback that preserves commentary suppression.
