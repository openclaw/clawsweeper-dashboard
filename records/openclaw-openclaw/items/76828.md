## Summary

Keep open: current main still has a clear source-level path where commentary-only assistant output is intentionally filtered and commentary-only fallback payloads are rejected, so a final-only turn can finish with no visible reply. The newer Telegram topic addendum looks partly covered by the documented group/tool-only visible-reply policy unless `messages.groupChat.visibleReplies: "automatic"` is set.

## What This Changes

Requests that final-only delivery avoid silent user-visible drops when long tool-loop turns emit only commentary-phase text and no final-answer chunk.

## Best Possible Solution

Add a terminal no-final guard that surfaces a safe diagnostic or bounded maintainer-approved fallback without weakening commentary suppression, and handle topic natural replies separately through the documented visible-reply policy.

## Reproduction Assessment

Yes, at source level: current main filters commentary-phase assistant text before visible delivery and has regression coverage that commentary-only turns produce no fallback payload. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear. Auto-promoting last commentary would solve silence but risks leaking progress/internal text, so the safer direction is a maintainer-approved terminal diagnostic or narrowly bounded fallback.
