## Summary

Keep open for maintainer triage, but not as a strict confirmed core bug yet: current main already has generic Gateway device pairing via `openclaw devices`, while no OpenClaw-owned Coze bridge implementation exists in this repo. The remaining question is whether OpenClaw should improve generic Gateway diagnostics/docs for third-party clients or route this to the bridge/plugin owner.

## What This Changes

The issue asks OpenClaw to support or document coze-bridge Gateway device pairing so `session/prompt` calls stop failing with a not-paired error.

## Best Possible Solution

Keep Gateway pairing generic: verify whether the bridge can create a pending device request and persist the returned device token with the existing protocol, then add only generic docs/diagnostics or a plugin/API design if that flow is insufficient.

## Reproduction Assessment

Yes for the Gateway-side condition: source shows non-Control-UI clients with unapproved device identities can receive pairing-required even when shared auth is disabled. No for the full Coze flow, because the bridge implementation is external and was not run here.

## Solution Assessment

Unclear: adding Coze-specific core behavior is not obviously the best fix. The narrower path is to verify the bridge against the existing Gateway device-pairing protocol and only change OpenClaw if a generic API, diagnostic, or docs gap is proven.
