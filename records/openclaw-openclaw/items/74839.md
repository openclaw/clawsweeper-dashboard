## Summary

Keep open: the OpenRouter hook is the right seam for the linked bug, but the proposed profile drops the existing `minimal` level and still omits documented DeepSeek V4 `max`; it also lacks the required changelog entry and focused regression coverage.

## What This Changes

The PR adds an OpenRouter `resolveThinkingProfile` hook that advertises `off`, `low`, `medium`, `high`, and `xhigh` for cached reasoning-capable OpenRouter models except unsupported proxy routes.

## Best Possible Solution

Land a provider-owned OpenRouter thinking profile that preserves base levels, exposes documented DeepSeek V4 `xhigh`/`max` behavior, and includes focused tests plus a single-line changelog entry.

## Reproduction Assessment

Yes. By source inspection, current main has no OpenRouter `resolveThinkingProfile`, while core validation relies on that provider hook before falling back to base levels; the proposed profile also makes the missing `minimal`/`max` path clear without running the app.

## Solution Assessment

No, not as written. A provider-owned hook is the narrowest maintainable fix, but the level set should preserve `minimal`, include documented DeepSeek V4 `max`, and be covered by OpenRouter profile tests before merge.
