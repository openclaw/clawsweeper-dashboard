## Summary

Keep open. Current main has some cleanup for deleted/reset archives, unreferenced primary transcripts, and optional disk-budget pressure, but it still lacks the requested recycle-based garbage-collection configuration and intentionally preserves old `.bak` archives outside disk-budget cleanup; the related retention-policy issues still need maintainer direction.

## What This Changes

The issue asks to garbage-collect orphaned session archive artifacts, including `.reset`, `.deleted`, `.bak`, and `sessions_saved.json`, using new recycle/retention configuration.

## Best Possible Solution

Define one maintainer-approved session-artifact lifecycle policy, then implement it consistently across `session.maintenance` types/schema, cleanup/runtime behavior, CLI docs, config help, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure enforced maintenance with old `.deleted`, `.reset`, and `.bak` archives; current code and tests show deleted/reset age cleanup while `.bak` remains unless disk-budget pressure is configured, and the proposed config keys are absent.

## Solution Assessment

Unclear as proposed. A recycle-based GC model may be reasonable, but maintainers first need to reconcile hard deletion, archive preservation, default retention, and naming semantics with the related open issues.
