## Summary

Keep open. Current main still only partially handles session-directory cruft: normal maintenance cleans `.deleted` and `.reset` archives, and optional disk-budget cleanup can remove archive artifacts under pressure, but the requested garbage-collection/recycle config is absent and `.jsonl.bak.*` remains intentionally preserved by regression coverage. Related #56609 is still open and asks for the opposite retention posture, so this needs maintainer policy before an automated fix chooses deletion defaults.

## What This Changes

Add session-maintenance garbage collection controls for orphaned `.jsonl.reset.*`, `.jsonl.deleted.*`, `.jsonl.bak.*`, and legacy saved session files, with recycle retention before hard deletion.

## Best Possible Solution

Define one explicit session-artifact retention policy that covers `.jsonl.deleted.*`, `.jsonl.reset.*`, `.jsonl.bak.*`, legacy store backups or `sessions_saved.json` if intentionally supported, and the interaction with disk-budget cleanup. Then update the session maintenance types, strict schema/help, CLI/docs, archive helper callers, and focused regression tests around the chosen retention and recovery behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: the pruning integration test creates an old `.jsonl.bak.*` archive and expects it to remain after maintenance, while source only calls age cleanup for `.deleted` and `.reset` archive reasons.

## Solution Assessment

Unclear. The proposed recycle-and-retention model may be a good direction, but it is not safe to treat as the best fix until maintainers decide whether session archives are disposable, recoverable trash, or permanent history in light of #56609.
