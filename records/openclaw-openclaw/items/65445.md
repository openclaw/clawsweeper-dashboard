## Summary

Keep open. Current main has partial session-artifact cleanup, but it still lacks the requested garbage-collection/recycle config and intentionally preserves `.jsonl.bak.*` outside disk-budget pressure; the open archive-preservation discussion in #56609 also needs maintainer policy before choosing a default deletion model.

## What This Changes

Request garbage collection for orphaned `.jsonl.reset`, `.jsonl.deleted`, `.jsonl.bak`, and `sessions_saved.json` session artifacts, moving them through a recycle path before retention-based deletion.

## Best Possible Solution

Define one maintainer-approved session-artifact retention policy that reconciles archive preservation with disk cleanup, then encode it in `session.maintenance` types, strict schema, docs, cleanup CLI, and store/archive tests. The policy should explicitly cover `.jsonl.deleted.*`, `.jsonl.reset.*`, `.jsonl.bak.*`, legacy store backups, trajectory sidecars, and whether recycle-directory soft deletion is supported or whether existing disk-budget cleanup remains the only pressure valve.

## Reproduction Assessment

Yes. A high-confidence code reproduction path exists: create old `.deleted`, `.reset`, and `.bak` archives under enforced maintenance; current tests already prove old `.deleted` is removed while old `.bak` remains, and the schema rejects the proposed garbage-collection keys.

## Solution Assessment

Unclear as proposed. The recycle-and-retention model may be reasonable, but it is not yet the best maintainable fix until maintainers resolve the conflict with #56609 and decide whether session archives should ever be deleted by default.
