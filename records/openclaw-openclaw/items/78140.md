## Summary

Keep this PR open for maintainer review: the iOS/Swift and TypeScript direction matches the reported failure, but the new `.local` setup-code policy is not aligned with Android's scanner/manual endpoint parser.

## What This Changes

The PR updates Swift/iOS and TypeScript pairing/client paths to allow private-LAN, link-local, and `.local` plaintext mobile pairing, prefer explicit passwords over bootstrap tokens, and refresh docs, changelogs, and targeted tests.

## Best Possible Solution

Land this as the canonical #47887 fix after aligning Android or narrowing the generated/docs scope, preserving tailnet/public `ws://` blocking, and adding final live plus changed-surface proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the iOS manual TLS coercion and Swift/TypeScript bootstrap-over-password paths, and it also shows Android still rejects non-loopback cleartext setup/manual endpoints.

## Solution Assessment

No, not as-is. The iOS/TS approach is narrow and maintainable, but the PR should either update Android to the same private/local allowlist or avoid issuing/documenting setup codes that Android still rejects.
