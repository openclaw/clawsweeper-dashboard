## Summary

Keep open: this draft, protected maintainer PR is the active vehicle for the linked LAN pairing regression, but the current diff misses Android bootstrap handoff persistence after enabling private-LAN cleartext pairing.

## What This Changes

The PR allows private/local mobile `ws://` pairing and explicit-password auth precedence across iOS, Android, TypeScript gateway clients, docs, changelog, and tests.

## Best Possible Solution

Land this as the canonical fix for #47887 after updating Android session persistence to match the new private/local cleartext policy and after maintainer sign-off on the security boundary.

## Reproduction Assessment

Yes, by source inspection. Current main has loopback-only iOS/Android cleartext gates and bootstrap-over-password auth selection; no live device run was performed, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

No, not as currently written. The direction is narrow and consistent, but Android must also persist bootstrap handoff tokens for the newly allowed private/local cleartext endpoints before the fix is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Persist Android bootstrap handoff on private LAN:** `apps/android/app/src/main/java/ai/openclaw/app/node/ConnectionManager.kt:37`
  - body: This change lets private-LAN endpoints run with `tls == null`, but `GatewaySession.shouldPersistBootstrapHandoffTokens()` still persists bootstrap-issued device tokens only for loopback or TLS endpoints. A private-LAN setup-code connection can therefore succeed once and then drop the handoff tokens, causing the next reconnect to fall back to the consumed bootstrap token. Please update the Android session trust/persistence gate to the same private/local host policy and add a session-level regression test.
