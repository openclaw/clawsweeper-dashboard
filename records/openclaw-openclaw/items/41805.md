## Summary

Keep open. Current main still sends ordinary Discord component fallback clicks to the agent as plain event or callback text with sender/routing fields; the structured interaction data requested here remains limited to plugin interactive handlers even though the Discord wrapper and registry preserve the needed raw fields.

## What This Changes

The issue requests structured Discord button/select interaction metadata, including source message id, component/custom id, label, selected values, and clicker identity, in ordinary agent-visible callbacks.

## Best Possible Solution

Keep the Discord plugin as owner and add a typed untrusted component-interaction block to fallback inbound contexts, populated from existing registry and raw interaction fields, with tests covering button and select fallback turns.

## Reproduction Assessment

Yes. The existing Discord monitor test harness can register a component entry and capture lastDispatchCtx after button.run or select.run; current main shows synthesized BodyForAgent text and sender/routing fields without structured component metadata.

## Solution Assessment

Yes. A narrow Discord-plugin change that populates UntrustedStructuredContext on fallback turns is the maintainable path because core already renders structured untrusted JSON and the Discord interaction wrapper already exposes the needed fields.
