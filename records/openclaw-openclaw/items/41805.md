## Summary

Keep open: current main and the latest release still route Discord component clicks to the agent as text with canonical sender fields, while structured message/action metadata is only available to plugin interactive handlers. The remaining work is useful but crosses the shared inbound metadata and trust boundary, so it needs maintainer product/API review before implementation.

## What This Changes

The issue requests structured Discord button/select interaction metadata, including originating message, clicked component, and clicker identity, in agent inbound callbacks.

## Best Possible Solution

Add a portable typed interaction fact to canonical inbound supplemental context, with component kind, originating message id, stable semantic action/callback data, selected values, and explicit trust labeling, while reusing existing sender fields.

## Reproduction Assessment

Yes, at source level. Current main shows sender identity and text reach the agent, but originating message/action metadata is not projected into a structured inbound context field.

## Solution Assessment

No, the proposed Discord-specific envelope duplicates existing sender fields and needs a broader trust decision. The maintainable path is a portable typed interaction fact with untrusted callback data clearly separated from trusted identity.
