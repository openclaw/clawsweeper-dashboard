## Summary

Keep open: current main and the latest release still flatten generic Discord component callbacks into event text plus ordinary sender/message fields, while richer interaction facts are only available to plugin interactive handlers. The remaining work is a portable inbound metadata and trust-boundary decision, not a cleanup close.

## What This Changes

This issue requests structured Discord button/select interaction metadata, including source message, clicked control, callback data, and clicking user, in the agent’s inbound callback context.

## Best Possible Solution

Add a portable typed inbound interaction fact and populate it from Discord component fallback handling, with trusted identity/auth data separated from untrusted callback payloads and covered by focused docs and regression tests.

## Reproduction Assessment

Yes, at source level. Current Discord component fallback code projects callbacks as event text plus ordinary sender/message fields, while no typed interaction object reaches the generic agent context; I did not run a live Discord click in this read-only review.

## Solution Assessment

No, the Discord-specific JSON sketch is useful but not the best final shape. The maintainable solution is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
