## Summary

Keep open: current main and v2026.6.10 still route generic Discord component callbacks to agents as text, while richer interaction facts are only exposed to plugin interactive handlers. This remains a product/API contract decision rather than conservative cleanup.

## What This Changes

The issue requests structured Discord interaction metadata, including message, button, and user identifiers, in agent-facing button callback context.

## Best Possible Solution

Define a portable typed inbound interaction fact, populate it from Discord component fallback handling with explicit trust classification, and cover the contract with Discord docs plus regression tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 format generic Discord component callbacks as text and finalize the agent context from that string, while richer interaction facts only flow to plugin interactive handlers.

## Solution Assessment

No, the issue's Discord-specific JSON sketch captures the user need but is not the best final contract. The maintainable solution is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
