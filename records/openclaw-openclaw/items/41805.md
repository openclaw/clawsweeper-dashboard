## Summary

Keep open: current main and v2026.6.6 still route generic Discord component callbacks as event text plus ordinary sender/message fields, while richer interaction facts are only available to plugin interactive handlers. The remaining work needs a portable inbound metadata schema and trust-boundary decision before implementation.

## What This Changes

Request structured Discord component interaction metadata in agent inbound context for button/select callbacks, including source message, clicked component, and sender identity.

## Best Possible Solution

Add a portable typed interaction fact to canonical inbound supplemental context and populate it from Discord component fallback handling, with trusted identity/auth fields kept separate from untrusted callback payloads and covered by focused docs and regression tests.

## Reproduction Assessment

Yes, at source level. Current dispatch code and tests show Discord component callbacks are projected as formatted text plus ordinary sender/message fields, while no typed interaction object reaches the generic agent context.

## Solution Assessment

No, the Discord-specific JSON example is a useful sketch but not the best final shape. The maintainable solution is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
