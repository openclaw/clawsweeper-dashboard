## Summary

Keep open: current main and the latest release still project generic Discord component callbacks as text plus ordinary sender/message fields; the richer interaction data exists for plugin interactive handlers but is not exposed as a typed agent callback metadata contract.

## What This Changes

The issue requests structured Discord button/select callback metadata in the agent inbound context, including source message, clicked control, callback/custom id, and clicking user details.

## Best Possible Solution

Add a portable typed inbound interaction fact, populate it from Discord component fallback handling, keep trust classification explicit, and cover the new contract with Discord docs plus regression tests.

## Reproduction Assessment

Yes, at source level. Current fallback tests and code show the agent receives text such as `Clicked "Approve".` plus ordinary sender fields, while no typed interaction object reaches the generic agent context; I did not run a live Discord click in this read-only review.

## Solution Assessment

No, the Discord-specific JSON sketch is useful but not the best final shape. The maintainable fix is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
