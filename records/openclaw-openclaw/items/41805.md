## Summary

Keep open: current main and the latest release still route normal Discord component clicks to the agent as text plus flat sender/interaction fields, while plugin callback handlers get only partial structured metadata. The requested stable agent-facing interaction metadata remains useful but needs a maintainer API/product decision before a fix PR.

## What This Changes

Expose structured Discord interaction metadata, including message, button, and user identifiers, to agent callbacks for component clicks.

## Best Possible Solution

Define a stable, bounded Discord component interaction metadata contract, surface it through structured untrusted context for normal agent callbacks, and keep plugin callback context aligned with that contract.

## Reproduction Assessment

Yes, source-level: the Discord component fallback path dispatches plain BodyForAgent text and flat sender/interaction fields, while tests assert the agent receives `Clicked "Approve".`. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix should be a maintainer-approved structured metadata contract rather than ad hoc raw Discord fields or another competing callback-data path.
