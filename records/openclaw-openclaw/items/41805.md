## Summary

Keep open: current main still routes normal Discord component callbacks to the agent as text-only inbound turns, while the structured fields that exist today are limited to plugin interactive handlers. Related work such as https://github.com/openclaw/openclaw/pull/53607 addresses transcript context, not the broader inbound metadata contract requested here.

## What This Changes

The issue requests structured Discord button interaction metadata, including message ID, custom ID, button label, and user identity, in the agent callback context.

## Best Possible Solution

Define a typed Discord interaction metadata contract for normal inbound component callbacks, document it, and add focused Discord component tests without broadening generic adapter metadata into assistant-visible context accidentally.

## Reproduction Assessment

Yes, source-based. Current main’s Discord component fallback tests and dispatch code show callbacks reaching the agent as BodyForAgent text with sender fields, not as a structured interaction metadata envelope.

## Solution Assessment

No current implementation is complete. The plugin interactive handler context is a useful partial surface, but the requested normal agent callback metadata needs a maintainer-approved typed API shape before implementation.
