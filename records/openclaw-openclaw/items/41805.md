## Summary

Keep open: current main still routes ordinary Discord button/select callbacks to the agent as text-only inbound turns, while richer interaction metadata is only exposed to registered plugin interactive handlers. This is a valid Discord channel/API feature request that needs maintainer product and trust-boundary review before implementation.

## What This Changes

The issue requests structured Discord interaction metadata, including source message, clicked button/custom data, and sender identity, in normal agent button callback context.

## Best Possible Solution

Define a typed Discord interaction metadata envelope for normal component callbacks, document trust semantics for sender/message/component fields, and add focused button/select/modal fallback tests without regressing registered plugin interactive handlers.

## Reproduction Assessment

Yes, source-based: current main shows ordinary Discord component callbacks becoming text fallback agent turns, while structured interaction fields remain confined to registered plugin interactive handlers. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear until maintainer API review. The maintainable direction is a typed inbound metadata contract with explicit trust semantics, not embedding identifiers in labels or pushing generic metadata into model context ad hoc.
