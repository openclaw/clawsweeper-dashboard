## Summary

Keep open: current main partially addresses the report by carrying the clicker’s numeric ID and username in canonical sender fields and the component identifier in fallback text, but the agent still lacks structured facts tying the callback to the originating message and semantic action. The remaining change crosses the shared inbound metadata and trust boundary, so maintainers should choose a portable interaction contract before implementation.

## What This Changes

Requests structured Discord button-interaction metadata—such as originating message, component identity, and clicker identity—in the agent’s inbound context.

## Best Possible Solution

Add a portable typed interaction member to canonical inbound supplemental facts containing component kind, originating message ID, and a stable semantic action identifier; reuse existing sender fields and place application callback payloads in explicitly untrusted structured context.

## Reproduction Assessment

Yes, from the exact current-main source and tests: clicker identity and component ID reach the agent through canonical sender fields and formatted text, but the originating message/action relationship is not represented structurally.

## Solution Assessment

No, the proposed Discord-specific envelope is broader and more duplicative than necessary because sender identity already has canonical fields. The narrow maintainable solution is a portable interaction addition to SupplementalContext with explicit trust semantics.
