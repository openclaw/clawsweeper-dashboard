## Label Justifications

## Summary

Keep open: current main and the latest release still route generic Discord component callbacks to the agent as flattened text plus ordinary sender/message fields, while richer interaction facts are only exposed to plugin interactive handlers. The remaining work needs a portable inbound interaction metadata contract and a trust-boundary decision before implementation.

## What This Changes

The issue requests structured Discord button callback metadata in the inbound agent context, including source message, button/custom id, label, and sender identity.

## Best Possible Solution

Add a portable typed inbound interaction fact to canonical supplemental context, populate it from Discord component fallback handling, and document the trust classification with focused Discord regression coverage.

## Reproduction Assessment

Yes, at source level. Current dispatch code and tests show generic Discord component callbacks are projected as event text plus ordinary sender/message fields, while no typed interaction object reaches the agent context.

## Solution Assessment

No, the Discord-specific JSON example is a useful sketch but not the best final shape. The maintainable solution is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
