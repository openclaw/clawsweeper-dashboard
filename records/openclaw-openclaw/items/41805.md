## Summary

Keep open: current main and the latest release still turn ordinary Discord component callbacks into fallback text or raw callbackData for the agent, while the richer interaction metadata exists only on the plugin interactive-handler surface. This is a real, source-reproducible Discord channel/API gap, but exposing message ids, custom payloads, and sender identity to model-facing context needs an explicit trust-boundary/product decision before automation should implement it.

## What This Changes

The issue requests structured Discord button-click metadata, including source message id, button/custom payload identity, label, and clicking user information, in the agent inbound context.

## Best Possible Solution

Define a typed, documented Discord interaction metadata envelope for normal component fallback turns, likely through the existing untrusted structured-context path, while preserving the plugin interactive handler contract.

## Reproduction Assessment

Yes, source-based: current main shows ordinary Discord component callbacks becoming BodyForAgent text or callbackData while richer interaction fields are confined to plugin interactive handlers. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear until maintainer API review. The maintainable direction is a typed inbound metadata contract with explicit trust semantics, not label-encoded IDs or ad hoc prompt text.
