## Summary

Keep open: current main still routes unmatched/general Discord component clicks as text-only inbound turns. Matched plugin callbacks get some structured fields, but the agent fallback lacks the requested interaction metadata, and the related open PR only addresses transcript context rather than the full metadata contract.

## What This Changes

Request to expose Discord component click metadata such as source message id, custom id, label, and clicking user to agent inbound context.

## Best Possible Solution

Define an additive inbound interaction metadata contract for Discord component fallback turns, map the existing Discord custom id, source message, label, selected values, and user fields into it, document the contract, and add focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main converts general Discord component clicks into BodyForAgent/RawBody/CommandBody text and does not attach a structured interaction metadata object. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is useful, but the maintainable fix should be an additive, documented inbound metadata contract rather than ad hoc Discord-only prompt text or the narrower transcript-context change in https://github.com/openclaw/openclaw/pull/53607.
