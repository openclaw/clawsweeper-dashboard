## Summary

Keep open: current main partially exposes the clicker identity through `SenderId`/`SenderUsername`, but the agent fallback path still turns Discord component clicks into text-only inbound turns without structured source message, component/custom id, or interaction metadata. A prior source-message PR was closed after maintainer feedback that the payload shape should be done differently, so this needs a product/API decision rather than automatic cleanup.

## What This Changes

Request to add structured Discord component interaction metadata, including source message, button/custom id, and click user details, to agent inbound button/select callbacks.

## Best Possible Solution

Define a Discord-owned component callback metadata contract that exposes source message id, component kind/label/custom id or callback data, and clicker identity through a bounded, documented agent-context surface without weakening component authorization or shared channel API boundaries.

## Reproduction Assessment

Yes, source-reproducible: current main's component fallback builds the agent-visible turn from `eventText` and does not add structured source message/custom-id metadata, while Discord's typed interaction payload contains those raw facts. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear as proposed. The requested behavior is useful, but the maintainable solution needs maintainer agreement on the payload/API shape and trust boundary before implementation.
