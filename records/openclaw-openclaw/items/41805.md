## Summary

This should stay open: current main still routes normal Discord component fallback callbacks as synthesized text, not as an agent-visible structured interaction metadata object. Sender IDs and plugin-interactive handler metadata exist, but they do not cover the broader fallback path requested here.

## What This Changes

Request structured Discord button/select callback metadata including source message ID, clicked component identity, label/value, and clicker identity in the agent inbound context.

## Best Possible Solution

Define an additive, documented interaction metadata contract for Discord component fallback events, reusing existing registry and Discord payload fields while keeping prompt metadata boundaries stable.

## Reproduction Assessment

Yes, source-reproducible. The active fallback path turns component clicks into event text and finalizes a context without source message ID, component custom ID, label/value metadata, or a structured interaction object.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is useful, but the maintainable fix should be an additive cross-channel-safe metadata contract rather than a Discord-only text workaround.
