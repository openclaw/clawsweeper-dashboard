## Summary

Keep open: current main and v2026.6.10 still route generic Discord component callbacks to agents as formatted inbound text, while richer interaction facts exist only in internal/plugin-interactive paths and need a maintainer-approved typed callback metadata contract.

## What This Changes

The issue requests structured Discord button interaction metadata, including source message ID, clicked button/custom data, label, and Discord user identity, in agent callback context.

## Best Possible Solution

Expose a typed, trust-boundaried inbound interaction metadata contract for agent callbacks, populated first from Discord component fallback handling and covered by Discord docs plus regression tests.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 route ordinary component callbacks to the agent as formatted text while richer interaction details are only present in internal/plugin paths; no live Discord click was run in this read-only review.

## Solution Assessment

Unclear pending product/API review. The request is directionally right, but the best fix is a portable typed inbound interaction fact with explicit trust classification, not an ad hoc Discord-only JSON blob.
