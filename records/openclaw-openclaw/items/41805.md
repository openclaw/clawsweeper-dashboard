## Summary

Keep open: current main still routes unmatched Discord button/select component callbacks to the agent as plain inbound text, while the raw Discord interaction and plugin-interactive path already carry the metadata needed for an additive structured context block.

## What This Changes

Expose structured Discord interaction metadata, including message ID, clicked component identity, and user identity, in agent inbound context for Discord button/select callbacks.

## Best Possible Solution

Add a Discord-owned untrusted structured interaction block to fallback component agent turns, reusing the existing inbound metadata renderer and preserving current callbackData text semantics.

## Reproduction Assessment

Yes, source-reproducible: register a Discord component with callbackData and no matching plugin handler, then inspect lastDispatchCtx in the existing monitor harness. Current source shows BodyForAgent receives only eventText and finalizeInboundContext receives no UntrustedStructuredContext.

## Solution Assessment

Yes. The narrow maintainable fix is to add an additive Discord fallback metadata entry through the existing UntrustedStructuredContext path rather than inventing a new core protocol or changing plugin interactive dispatch.

Keep BodyForAgent/RawBody/CommandBody behavior compatible, preserve plugin interactive handler behavior, and do not treat custom_id/callbackData/labels/values as trusted authorization signals.
