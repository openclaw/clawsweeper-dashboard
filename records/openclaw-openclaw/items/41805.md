## Summary

This should stay open: current main has improved Discord component routing, sender IDs, persisted message IDs, and plugin-interactive handler metadata, but the normal agent fallback path still receives only synthesized text/callback data rather than a structured interaction metadata object with source message ID, clicked component identity, and clicker details.

## What This Changes

Expose structured Discord component interaction metadata to agents when users click buttons or selects.

## Best Possible Solution

Add an additive, documented interaction metadata contract for Discord component fallback events, reusing the existing component registry and interaction payload fields while keeping prompt metadata safe and channel/plugin boundaries intact.

## Reproduction Assessment

Yes, source-reproducible. The active fallback path builds only event text/callback data for the agent turn, while the finalized context lacks a structured interaction object and does not expose the original component message ID as agent metadata.

## Solution Assessment

Unclear without maintainer API direction. The requested direction is useful, but the durable fix should be an additive metadata contract aligned with existing inbound_meta behavior rather than a Discord-only text workaround.
