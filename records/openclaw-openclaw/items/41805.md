## Summary

Keep open: current main still routes ordinary Discord component callbacks to the agent as text or callbackData only. Sender fields now exist as scalar context, but the requested structured interaction envelope with source message id, button/custom payload identity, and click metadata is still only available on the plugin interactive-handler surface.

## What This Changes

Request structured Discord button/select interaction metadata in normal agent callbacks so stateful workflows do not need identifiers embedded in visible labels.

## Best Possible Solution

Define a typed, documented Discord interaction metadata envelope for normal component fallback turns, using the existing untrusted structured-context path where appropriate, and preserve the plugin interactive handler contract.

## Reproduction Assessment

Yes, source-based: current main shows ordinary Discord component callbacks becoming BodyForAgent text or callbackData while richer interaction fields are confined to plugin interactive handlers. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear until maintainer API review. The maintainable direction is a typed inbound metadata contract with explicit trust semantics, not label-encoded IDs or ad hoc prompt text.
