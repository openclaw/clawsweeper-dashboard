## Summary

Keep open. Current main has improved Discord component handling, including hidden `callbackData`/presentation `value` routing and sender identity, but the built-in agent fallback still does not surface structured source-message/component metadata in the inbound context.

## What This Changes

The issue requests structured Discord interaction metadata for button callbacks so agents can identify the originating message, clicked component, and clicking user.

## Best Possible Solution

Add an additive Discord component interaction metadata block to the built-in agent fallback context, reusing stored registry data and preserving current callback routing and dedupe semantics.

## Reproduction Assessment

Yes, by source inspection. The current-main test path for a Discord button click produces a plain model-facing fallback body, while the dispatch context lacks source message/component fields despite the registry already storing them.

## Solution Assessment

Yes, a narrow additive Discord dispatch change is the best repair direction. Use existing typed context or `UntrustedStructuredContext`/`ReplyToId` fields rather than changing `MessageSid` or adding a new global core schema first.
