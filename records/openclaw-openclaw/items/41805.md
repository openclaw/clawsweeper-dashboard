## Summary

Keep open: current main and the latest release still flatten generic Discord component callbacks into event text plus ordinary sender/message fields, while richer interaction facts are only available to plugin interactive handlers. The remaining work is a portable inbound metadata/API decision, not conservative cleanup to close.

## What This Changes

Request structured interaction metadata for Discord button/select callbacks so agents can identify the source message, clicked component, selected value, and clicking user without label-encoded IDs.

## Best Possible Solution

Add a portable typed inbound interaction fact to canonical supplemental context, populate it from Discord component fallback handling, document the trust classification, and cover generic fallback plus plugin-interactive sibling behavior with focused tests.

## Reproduction Assessment

Yes, at source level. Current main routes generic Discord component callbacks as event text plus ordinary sender/message fields, and the canonical inbound context has no typed interaction metadata member.

## Solution Assessment

No, not as the Discord-specific JSON shape alone. The maintainable solution is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
