## Summary

Keep open: current main still routes ordinary Discord component fallback clicks as plain agent text while the raw interaction and plugin handler paths already have the metadata needed for a structured context block.

## What This Changes

Expose Discord button/select interaction metadata, including source message, component/custom id, label or values, and clicker identity, to normal agent callback context.

## Best Possible Solution

Add an additive Discord-owned untrusted structured interaction block to fallback agent turns, with focused regression coverage and docs/changelog updates.

## Reproduction Assessment

Yes. The existing Discord monitor harness can drive a registered button/select click and inspect lastDispatchCtx; source and tests show the fallback only carries plain BodyForAgent text today.

## Solution Assessment

Yes. Reusing the existing UntrustedStructuredContext renderer from the Discord fallback path is the narrowest maintainable approach and avoids a new core contract unless maintainers later want cross-channel interaction metadata.

Keep the implementation inside the Discord plugin unless a tiny generic seam is strictly necessary. For button/select fallback dispatch, add an UntrustedStructuredContext entry labeled for Discord interaction metadata with source/type fields and a payload that includes interaction kind, Discord interaction id, source message id, component registry id, raw custom_id when available, callbackData when present, button/select label, selected values for selects, channel id, and clicker user id/username/tag. Treat labels, callbackData, custom_id, and values as untrusted; do not use them for auth or routing. Preserve existing plugin interactive dispatch behavior and fallback BodyForAgent text.
