## Label Justifications

- `P2`: The missing structured callback association limits Discord component workflows, but callbacks still reach the agent as normal inbound messages and core runtime availability is not broken.

## Summary

Keep open: current main and v2026.6.6 still flatten Discord component clicks into callback text plus sender fields for the generic agent path, while richer interaction facts exist only for plugin interactive handlers. The remaining work is a useful feature, but it needs a maintainer-owned portable inbound metadata and trust-boundary decision before implementation.

## What This Changes

Request to include structured Discord button interaction metadata, such as originating message id, clicked component identity, selected value, and clicking user, in the agent callback context.

## Best Possible Solution

Add a portable typed interaction fact to canonical inbound supplemental context and populate it from Discord component fallback handling, with trusted identity/auth fields kept separate from untrusted callback payloads and covered by docs plus focused regression tests.

## Reproduction Assessment

Yes, at source level. Current tests and dispatch code show Discord button/select callbacks are projected as `BodyForAgent` text with sender fields, while no structured interaction metadata reaches the generic agent context.

## Solution Assessment

No, the Discord-specific JSON example is a useful sketch but not the best final shape. The maintainable solution is a portable typed inbound interaction fact with explicit trust classification and Discord as the first populated channel.
