## Summary

Keep open: current main still routes unmatched Discord button/select callbacks to the agent as text-only fallback turns, even though the interaction payload and plugin-interactive path already have the requested message, custom id, selected value, and user metadata.

## What This Changes

Request structured Discord interaction metadata in agent-facing button/select callback context so workflows can identify the clicked message, component, and user without encoding IDs into labels.

## Best Possible Solution

Add a Discord-owned untrusted structured interaction block to fallback component agent turns while preserving the existing callbackData text and routing behavior.

## Reproduction Assessment

Yes, source-reproducible: register a Discord component with callbackData and no matching plugin handler, then inspect the fallback dispatch context in the existing monitor harness. Current main builds only text fallback fields and finalizes no structured interaction metadata.

## Solution Assessment

Yes, the narrow maintainable fix is an additive UntrustedStructuredContext payload in the Discord fallback path, not a core protocol change or a rewrite of plugin interactive dispatch.
