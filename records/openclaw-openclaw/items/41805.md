## Summary

Keep open: current main has Discord sender IDs and a plugin interactive-handler context, but the normal agent fallback still converts a component click into plain text and does not attach structured component metadata for the agent.

## What This Changes

Request to expose Discord component interaction metadata, including message, button, and user identifiers, to agents when Discord buttons or selects are clicked.

## Best Possible Solution

Add an additive structured Discord component-interaction block to the agent inbound context, preferably through the existing UntrustedStructuredContext path, and document the available fields.

## Repair Work Prompt

Implement a narrow additive fix for https://github.com/openclaw/openclaw/issues/41805. In the Discord message-tool component fallback path, include structured component interaction metadata in the agent inbound context when a button/select/modal fallback is routed to the agent. Prefer using existing MsgContext.UntrustedStructuredContext rather than adding a new config option or changing core plugin boundaries. Include fields needed for stateful workflows: channel `discord`, interaction kind, original component messageId, OpenClaw component id, button/select label, callbackData when present, selected values for selects, actor user id/username/tag, and the Discord interaction id. Do not treat labels or callbackData as trusted instructions, and do not break existing plugin interactive handler semantics. Add focused regression coverage in the Discord component monitor tests and update Discord channel docs plus CHANGELOG.md if user-facing.
