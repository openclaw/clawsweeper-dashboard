## Summary

Keep open: current main still routes normal Discord component fallback callbacks as synthesized text, not as a structured interaction metadata object. Sender IDs and plugin-interactive handler metadata exist, but they do not cover the broader fallback path requested here.

## What This Changes

Request that Discord component callback turns expose structured interaction metadata such as source message ID, button ID/label, and clicking user identity.

## Best Possible Solution

Define an additive, documented interaction metadata contract for Discord component fallback events, reusing existing registry and Discord payload fields while preserving prompt cache boundaries.

## Reproduction Assessment

Yes. The source path from component handler to finalized inbound context shows fallback clicks become text-only agent turns, and current tests assert only BodyForAgent text for those callbacks.

## Solution Assessment

Unclear until maintainers choose the API shape. The direction is useful, but the maintainable fix should be an additive metadata contract rather than Discord-only prompt-field drift.
