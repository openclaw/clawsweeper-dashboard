## Summary

Keep open: current main still converts general Discord component fallback clicks into text-only agent turns, while structured interaction metadata is only available to matched plugin interactive handlers. The remaining work is a useful feature, but it changes the agent-visible inbound metadata contract and needs maintainer API-shape review before implementation.

## What This Changes

The issue requests structured Discord interaction metadata for button callbacks, including source message, clicked component, and Discord user identifiers.

## Best Possible Solution

Define an additive, documented interaction metadata contract for Discord component fallback events, reusing existing registry and Discord payload fields while preserving prompt cache boundaries.

Yes, source-level: current main's fallback path turns Discord component clicks into `BodyForAgent` text and does not serialize source message, component, or structured interaction metadata. I did not run a live Discord click in this read-only review.

## Solution Assessment

Unclear until maintainers choose the API shape. The requested direction is useful, but the maintainable fix should be an additive metadata contract rather than Discord-only prompt-field drift.
