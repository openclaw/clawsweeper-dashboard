## Summary

Keep open. The underlying leak is credible, but this PR currently treats every WebSocket close as cancellation and would cancel in-flight runs during transient reconnects or page refreshes, despite existing late-join support for active sessions.

## What This Changes

The PR imports the existing chat abort helper into the gateway WebSocket close handler, aborts matching `chatAbortControllers` entries on owner connection close, logs that path, and adds a focused unit test.

## Best Possible Solution

Define the disconnect ownership semantics, then implement cancellation in a way that preserves page-refresh/reconnect late joining while still stopping truly abandoned runs promptly.

## Reproduction Assessment

Yes. Source inspection shows current main registers owner connection IDs but the WebSocket close handler does not consume them, and the PR body supplies before/after production-bundle log proof.

## Solution Assessment

No, not as written. The narrow abort helper is the right primitive, but firing it on every socket close conflicts with existing reconnect and late-join behavior; a grace/rebind path or explicit cancellation signal would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve reconnectable runs on transient WS closes:** `src/gateway/server/ws-connection.ts:395-396`
  - body: This loop aborts every run owned by the closing socket. A tab refresh or network flap closes the old socket before the replacement can subscribe, but current chat/agent handlers explicitly register late-joining clients in the same session for active runs. This would turn refresh/reconnect into cancellation, so the disconnect abort needs a grace/rebind path or an explicit user-cancel signal instead of firing on every close.
