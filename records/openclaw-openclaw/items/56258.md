## Best Possible Solution

Port the timeout design onto current main, preserve existing HTTP/WebSocket/session shutdown semantics, let channel-stop timeout ownership preserve final runtime state, and add focused regression coverage plus redacted runtime shutdown proof.

## Reproduction Assessment

Yes for source-level reproduction: current main serially awaits non-HTTP shutdown steps under the shared 25s watchdog, so a hung subsystem can starve later cleanup. I did not live-reproduce the Telegram polling stall or a real hung subsystem in this read-only review.

## Solution Assessment

No as submitted: the timeout direction is reasonable, but wrapping `stopChannel` with an equal 5s outer budget can return before the channel manager records final state. The safer fix keeps channel finalization owned by the channel manager while bounding genuinely hung non-HTTP teardown steps.
