## Summary

Keep open: this is a useful gateway reliability fix, but the current PR is still merge-conflicting, lacks real shutdown proof, and the submitted timeout wrapper can return before channel stop finalization records timeout state.

## What This Changes

The PR adds a 5s per-subsystem gateway shutdown timeout helper, applies it to non-HTTP close steps, adds gateway tests, and updates the changelog.

## Best Possible Solution

Port the per-subsystem timeout design onto current main while preserving channel stop finalization and newer shutdown semantics, then add focused regression coverage plus redacted runtime shutdown proof.

## Reproduction Assessment

Yes, at source level: current main still awaits non-HTTP shutdown steps under the shared 25s watchdog, so a hung subsystem can starve later cleanup. I did not live-reproduce the original Telegram polling stall.

## Solution Assessment

No as submitted: the timeout direction is right, but wrapping stopChannel with the same 5s budget as its own finalization path can return before channel state is recorded. The safer fix is to keep timeout ownership inside the channel manager or use a phase-aware wrapper that preserves finalization.
