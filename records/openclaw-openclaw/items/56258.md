## Summary

Keep open: this is a useful gateway shutdown reliability fix, but the current PR is not merge-ready because it is conflicting, lacks real shutdown proof, and still lets the outer channel-stop timeout race the channel manager's own final state update.

## What This Changes

The PR adds a `raceTimeout` helper, wraps gateway non-HTTP shutdown steps in 5s per-subsystem timeouts with warning logging, adds focused tests, and updates the changelog.

## Best Possible Solution

Port the timeout design onto current main, keep the existing HTTP/WebSocket/session shutdown semantics, make channel-stop timeout ownership preserve final runtime state, and add focused regression coverage plus redacted runtime shutdown proof.

## Reproduction Assessment

Yes for source-level reproduction: current main serially awaits non-HTTP shutdown steps under the shared 25s watchdog, so a hung subsystem can starve later cleanup. I did not live-reproduce the Telegram polling stall or a real hung subsystem in this read-only review.

## Solution Assessment

No as submitted: the timeout direction is right, but wrapping `stopChannel` with the same 5s budget as its own abort-timeout path can return before channel state is finalized. The maintainable fix should preserve the channel manager finalization contract while bounding genuinely hung stop hooks.
