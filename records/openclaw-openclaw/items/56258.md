## Summary

Keep open: the shutdown-starvation bug is still plausible on current main and this PR is relevant, but it is not merge-ready because real shutdown proof is missing and the channel timeout wrapper can bypass channel finalization state.

## What This Changes

The branch adds a shutdown-timeout helper, wraps non-HTTP gateway teardown steps in 5s timeouts, adds gateway shutdown tests, and updates the changelog.

## Best Possible Solution

Port bounded non-HTTP teardown onto current main while keeping channel stop finalization owned by the channel manager and preserving the newer HTTP, WebSocket, session, and restart-drain shutdown semantics.

## Reproduction Assessment

Yes for source-level reproduction: current main serially awaits non-HTTP shutdown steps under the shared 25s watchdog, so an injected hung subsystem can starve later cleanup. I did not live-reproduce the Telegram polling stall or a real production hung subsystem in this read-only review.

## Solution Assessment

No as submitted: the timeout direction is reasonable, but the `stopChannel` wrapper can outrun the channel manager's own final-state handling. The safer fix keeps channel finalization inside the channel manager while bounding genuinely hung non-HTTP teardown steps.
