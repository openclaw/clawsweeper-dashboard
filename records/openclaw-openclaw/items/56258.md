## Best Possible Solution

Port the timeout design onto current main, preserve existing HTTP/WebSocket/session shutdown semantics, keep channel stop timeout ownership inside the channel manager, and add focused regression coverage plus redacted runtime shutdown proof.

## Reproduction Assessment

Yes for source-level reproduction: current main serially awaits non-HTTP shutdown steps under the shared 25s watchdog, so an injected hung subsystem can starve later cleanup. I did not live-reproduce the Telegram polling stall or a real production hung subsystem in this read-only review.

## Solution Assessment

No as submitted: the timeout direction is reasonable, but the branch conflicts with current main and the `stopChannel` wrapper can outrun the channel manager's own final-state handling. The safer fix keeps channel finalization owned by the channel manager while bounding genuinely hung non-HTTP teardown steps.
