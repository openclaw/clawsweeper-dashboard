## Summary

Keep open. The PR addresses a real gateway shutdown reliability gap that current main only partially covers, but the branch is not safe to merge as-is because it can report HTTP shutdown complete while the listener is still bound and can leave timed-out promises unobserved.

## What This Changes

The PR adds a 5s `raceTimeout` helper and tests, then wraps gateway close-handler subsystem stops, WebSocket close, and HTTP server close calls with that timeout plus optional warning logging.

## Best Possible Solution

The desired end state is current main's shutdown result and warning model with bounded non-HTTP teardown steps that cannot starve later cleanup, while HTTP close remains a two-phase listener shutdown that only returns after close succeeds or fails explicitly.

## Reproduction Assessment

Yes. The PR-introduced regressions are reproducible with unit fakes: an HTTP server whose close callback arrives after the 5s timeout shows early shutdown completion, and a promise that rejects after timeout shows the late unhandled-rejection risk.

## Solution Assessment

No. The timeout direction is useful, but this diff is not the best solution because it treats HTTP listener timeout as success and inconsistently observes timed-out promises; adapting current main's bounded-step patterns is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep HTTP close pending through forced connection close:** `src/gateway/server-close.ts:185-189`
  - body: This races `httpServer.close()` against a 5s timeout and lets shutdown complete when the timeout wins. In the in-process restart path, the next gateway can start as soon as `close()` returns, so a listener that is still draining can keep the port bound and make restart fail with `EADDRINUSE`. Preserve current main's grace period, `closeAllConnections()`, force wait, and failure if the listener is still pending.
  - confidence: 0.9
- **[P2] Observe promises after timeout:** `src/gateway/server-close.ts:50-103`
  - body: `raceTimeout` leaves the losing promise dangling, but several call sites pass raw promises such as `bonjourStop`, `canvasHost.close`, `canvasHostServer.close`, and `stopGmailWatcher`. If the timeout wins and the original promise later rejects, that rejection happens outside the surrounding `try/catch` and can trigger Node's unhandled-rejection path during shutdown.
  - confidence: 0.86
