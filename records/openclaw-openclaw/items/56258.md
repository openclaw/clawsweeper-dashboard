## Summary

Keep open: the shutdown hang is a valid gateway reliability problem and the branch contains useful timeout/test work, but it is conflicting, lacks real behavior proof, and the channel timeout wrapper can beat channel-manager finalization during restart.

## What This Changes

The PR adds a 5s gateway shutdown timeout helper, applies it to non-HTTP teardown steps, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rebase onto current main and bound hung non-HTTP cleanup steps while letting stopChannel own channel finalization, or using a gateway budget that cannot win before channel state is recorded, then prove the real shutdown path.

## Reproduction Assessment

Yes, at source level. Current main serially awaits non-HTTP shutdown steps, and the blocking review finding is also source-reproducible because the PR starts an equal 5s gateway race around stopChannel before stopChannel's own finalization timeout can reliably record state.

## Solution Assessment

No, not as currently proposed. Per-subsystem timeouts are a plausible fix, but the channel manager should keep ownership of channel stop timeout/finalization instead of being raced by an equal outer gateway timeout.
