## Summary

Keep open. The reliability problem is real and the PR is useful source work, but current main only partially covers the requested per-subsystem shutdown bound and the proposed branch would regress current HTTP shutdown behavior while leaving late-rejection hazards.

## What This Changes

The PR adds a 5s `raceTimeout` helper and tests, then wraps gateway close-handler subsystem stops, WebSocket close, and HTTP server close calls with that timeout plus optional warning logging.

## Best Possible Solution

Keep current main's shutdown result and warning model, preserve the two-phase HTTP listener close, and add bounded deadlines only to the non-HTTP close-handler steps through a helper that records timeout warnings and observes late promise rejection. Cover hung and late-reject cases in focused gateway tests and add the required changelog entry.

## Reproduction Assessment

Yes. The PR-introduced regressions are reproducible with unit fakes: an HTTP server whose close callback arrives after 5s shows early shutdown completion, and a subsystem promise that rejects after the timeout shows the late unhandled-rejection risk.

## Solution Assessment

No. The timeout direction is useful, but this patch is not the best solution because it treats HTTP listener timeout as success and inconsistently observes timed-out promises; adapting the current `shutdownStep` model is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve two-phase HTTP shutdown:** `src/gateway/server-close.ts:185-192`
  - body: This races `httpServer.close()` against the 5s subsystem timeout and lets shutdown complete when the timeout wins. In the in-process restart path, the next gateway can start while the old listener is still draining, which can fail the new bind with `EADDRINUSE`. Keep current main's grace period, `closeAllConnections()`, force wait, and failure if the listener is still pending.
  - confidence: 0.9
- **[P2] Observe timed-out subsystem promises:** `src/gateway/server-close.ts:50-103`
  - body: `raceTimeout` leaves the losing promise dangling, but several call sites pass raw promises such as `bonjourStop()`, `canvasHost.close()`, `canvasHostServer.close()`, and `stopGmailWatcher()`. If the timeout wins and the original promise later rejects, that rejection happens outside the surrounding `try/catch` and can trigger Node's unhandled-rejection path during shutdown.
  - confidence: 0.86

  - body: This is a user-facing `fix(gateway)` reliability change, but the branch does not add `CHANGELOG.md`. Add a single-line `Unreleased` `### Fixes` entry with eligible contributor credit before this lands.
