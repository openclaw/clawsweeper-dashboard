## Summary

Close as implemented: the reported 2026.5.7 failure came from whole-agent runtime pins forcing non-OpenAI model switches through Codex, while current main scopes runtime policy to provider/model routes and sends unmatched non-OpenAI auto-mode turns to PI.

## What This Changes

The issue asks for dashboard/session model switching to avoid a Codex-runtime plus Ollama-auth mismatch by routing the turn correctly, blocking the selection, or surfacing a clear error.

## Best Possible Solution

Keep the provider/model-scoped runtime policy on main; any remaining broader Codex OAuth or fallback-chain migration work should stay in https://github.com/openclaw/openclaw/issues/75739.

## Reproduction Assessment

Yes for the reported 2026.5.7 behavior by source inspection: that release resolved whole-agent/default agentRuntime into harness selection for non-OpenAI providers. I did not establish a failing current-main reproduction because the current source path no longer selects Codex for stale whole-agent pins.

## Solution Assessment

Yes. Moving runtime policy to provider/model scope and ignoring stale whole-agent pins is the narrow maintainable fix, because it lets Ollama/Gemma use the normal auto/PI route while preserving explicit provider/model runtime pins.
