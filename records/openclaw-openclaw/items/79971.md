The PR stacks a SQLite runtime-state follow-up over `exp-vfs`, moving session export, doctor checks, compaction checkpoints, transcript readers, docs, tests, and scripts toward SQLite transcript truth.

## Best Possible Solution

## Reproduction Assessment

Yes, source-reproducible for the PR-introduced bug: place a legacy `.jsonl` transcript in the legacy sessions directory without matching imported SQLite proof, then confirm the doctor cleanup path and the branch deletes it via `fs.rmSync`. I did not run the destructive path, but the changed source path is direct.

## Solution Assessment

No. Treating SQLite as operational truth is consistent with the database-first refactor, but deleting legacy transcript files should be gated on recorded migration/import proof or remain an archive/rename operation.
