## Summary

Keep open. Current main still has the rejected cron timer tick path logging without re-arming and has no independent watchdog lifecycle, while open PR #73355 is already a same-author implementation candidate with closing syntax for this issue.

## What This Changes

The issue reports that Gateway cron jobs can silently stop firing after about 2.5 days of uptime while manual cron runs and a gateway restart still work.

## Best Possible Solution

Review and land PR #73355 or an equivalent narrow cron-service fix that re-arms after rejected timer ticks, adds deterministic regression coverage for that stalled-chain path, and only adds a watchdog if its lifecycle and duplicate-execution behavior are tested.

## Reproduction Assessment

Unclear. The field report has detailed operational evidence, but a high-confidence live reproduction would require the multi-day macOS launchd uptime scenario; a focused unit test can reproduce the source-level rejected `onTimer` catch-path gap.

## Solution Assessment

Unclear. Re-arming in the catch handlers is the narrowest source-backed fix for the verified recovery gap, while an independent watchdog is broader and should be accepted only with clean lifecycle handling.
