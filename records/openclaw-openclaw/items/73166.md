Keep open. Current `main` still has the cron timer rejected-tick recovery gap, and open PR #73355 already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw cron scheduling to recover when the Gateway timer chain silently stops after long uptime instead of requiring a gateway restart.

## Best Possible Solution

Review and land PR #73355 or an equivalent focused cron-service fix that re-arms rejected timer ticks and adds regression coverage; accept a watchdog only with explicit start/stop lifecycle tests.

## Reproduction Assessment

Unclear. The report has detailed real-world evidence, but high-confidence live reproduction requires a multi-day macOS launchd uptime scenario; a focused unit test can reproduce the rejected `onTimer` catch-path recovery gap.

## Solution Assessment

Unclear. Re-arming in the catch handlers is the narrowest source-backed repair, while a separate watchdog is broader and should be justified by lifecycle-safe tests.
