## Summary

Keep open. Current main still has the rejected cron timer tick recovery gap, and open PR #73355 already targets this issue with closing syntax, so this should remain tracked until that PR or an equivalent focused fix lands.

## What This Changes

The issue reports that Gateway cron jobs can stop firing after multiple days of uptime while manual cron runs and a Gateway restart still work.

## Best Possible Solution

Land a focused cron scheduler recovery fix that re-arms rejected timer ticks with regression coverage, and accept any watchdog only with clear start/stop lifecycle tests.

## Reproduction Assessment

Unclear. The live scenario requires multi-day macOS launchd uptime, but current main has a focused unit-testable path by making `onTimer` reject from a timer callback and observing that the catch handler logs without replacing the timer.

## Solution Assessment

Unclear. Re-arming rejected timer ticks is the narrowest maintainable repair; a separate watchdog may be acceptable, but it needs lifecycle-safe tests and maintainer review.
