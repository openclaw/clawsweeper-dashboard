## Summary

Keep open: the reported abort-before-start bug is valid from current-main source, and the PR’s queue-level cancellation design is the right boundary, but the head still has a blocking diagnostics-state leak that should be fixed before merge.

## What This Changes

The PR adds AbortSignal-aware cancellation to queued ACP SessionActorQueue items, forwards withSessionActor signals into the queue, and adds focused queue regression tests.

## Best Possible Solution

Keep the queue-level cancellation approach, but remove or clean the production diagnostic map so idle actor keys are not retained indefinitely.

## Reproduction Assessment

Yes, from source inspection: current main increments pending on enqueue, decrements only on the queued task’s settle hook, and withSessionActor can reject abort-before-start before that queued task runs. I did not run the repro because this was a read-only review.

## Solution Assessment

Mostly yes: SessionActorQueue is the narrow owner for cancelling an already-enqueued slot while preserving same-key serialization. The diagnostic state should be item-scoped, cleaned when idle, or removed before this is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clean up settled diagnostics:** `src/acp/control-plane/session-actor-queue.ts:213-216`
  - body: Every enqueue stores a diagnosticsBySession entry for the actor key, but onSettle only stamps settleTime and never deletes the entry once pendingBySession and the tail map are idle. In a long-running gateway this retains one diagnostic object for every session key ever queued; please remove this production map or clean/update it only when the settling item still owns the live entry.
  - confidence: 0.84
