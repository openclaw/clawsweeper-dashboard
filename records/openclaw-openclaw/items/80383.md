## Summary

Keep open: the PR addresses a real source-reproducible abort-before-start queue leak, but the current head still has a blocking diagnostics-map issue that can retain one entry per actor key and let older items stamp newer diagnostic state.

## What This Changes

The branch adds AbortSignal-aware cancellation to SessionActorQueue.run, passes withSessionActor's signal into that queue path, and adds focused SessionActorQueue tests.

## Best Possible Solution

Keep the queue-level cancellation design, but remove the production diagnostics map or make it item-owned and cleaned up when the owning queued item settles idle.

## Reproduction Assessment

Yes. Source inspection shows current main increments pending on enqueue, only decrements from onSettle, and withSessionActor can reject an abort-before-start caller while a never-settling predecessor prevents that queued item from reaching onSettle.

## Solution Assessment

Mostly yes. SessionActorQueue is the right owner boundary for cancelling already-enqueued slots while preserving same-key serialization, but the proposed diagnostics map needs cleanup or item ownership before the patch is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clean up diagnostics for settled actor keys:** `src/acp/control-plane/session-actor-queue.ts:210-216`
  - body: Every enqueue stores a diagnosticsBySession entry for the actor key, but onSettle only stamps settleTime and never removes the entry once the key is idle. In a long-running gateway this retains one object for every session key ever queued, and older item hooks can also mutate the newest per-key diagnostic entry. Please remove this production map or make the diagnostic state item-owned and clean it up when the owning item settles.
  - confidence: 0.84
