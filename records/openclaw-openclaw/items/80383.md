## Summary

Keep open: current main still has the source-provable abort-before-start pending cleanup gap, and the PR’s queue-level cancellation is the right owner boundary, but the proposed diagnostics map retains per-session state after settlement.

## What This Changes

The PR adds AbortSignal-aware cancellation for queued ACP SessionActorQueue work, forwards the manager signal into the queue, and adds focused queue regression tests.

## Best Possible Solution

Keep the queue-level cancellation design, but remove or correctly clean up item-scoped diagnostics so idle actor keys are not retained, then land with focused queue and manager coverage.

## Reproduction Assessment

Yes, from source inspection: current main increments pending on enqueue, only decrements from the queued task's onSettle path, and withSessionActor can reject the caller before that task starts. I did not execute the repro because this review is read-only.

## Solution Assessment

Mostly yes: SessionActorQueue is the narrow owner for cancelling an already-enqueued slot while preserving per-key serialization. The added diagnostics state needs cleanup or removal before the PR is the best mergeable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clean up settled diagnostics entries:** `src/acp/control-plane/session-actor-queue.ts:210-216`
  - body: Every enqueue stores a `diagnosticsBySession` entry for the actor key, but `onSettle` only stamps `settleTime` and leaves the entry in the map after `pendingBySession` and the queue tail have gone idle. In a long-running gateway this retains one diagnostic object for every session key ever queued, so please remove the production map or delete/update it when the settled item still owns the live diagnostic entry.
