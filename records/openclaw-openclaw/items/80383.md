## Summary

Keep this PR open: current main still has the abort-before-start pending-count failure path, and the PR targets the right ACP queue boundary. I found one merge-blocking issue in the proposed diagnostics state that should be fixed before landing.

## What This Changes

The PR adds AbortSignal-aware cancellation to SessionActorQueue, passes withSessionActor's signal into the queue, and adds focused SessionActorQueue regression tests.

## Best Possible Solution

Keep the queue-level cancellation design, but make the diagnostic state item-scoped or remove/clean it so idle actor keys are not retained indefinitely.

## Reproduction Assessment

Yes, from source inspection: enqueue increments pending immediately, current cleanup waits for onSettle, and withSessionActor can reject the caller before the queued actor ever starts. I did not run the repro because this was a read-only review.

## Solution Assessment

Mostly yes: SessionActorQueue is the right owner boundary for cancelling an already-enqueued slot while preserving per-key serialization. The proposed diagnostics map needs cleanup or item-scoped guards before this is the best mergeable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid retaining settled diagnostics forever:** `src/acp/control-plane/session-actor-queue.ts:213-216`
  - body: Every enqueue stores a `diagnosticsBySession` entry for the actor key, but `onSettle` only stamps `settleTime` and never removes the entry once the key is idle. Because actor keys are session keys in a long-running gateway, this retains one diagnostic object for every session key ever queued even after `pendingBySession` and the tail map have been cleaned up; please either drop this production map or clean/update it only when the settling item still owns the live diagnostic entry.
  - confidence: 0.84
