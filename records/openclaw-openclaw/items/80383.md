## Summary

Keep open. Current main and v2026.6.6 still lack signal-aware queued-item cancellation, so the PR targets a real ACP queue liveness gap; it now needs a rebase against current main and one small diagnostic-state cleanup before merge.

## What This Changes

The PR adds AbortSignal-aware cancellation to ACP SessionActorQueue so queued work can be marked cancelled and pending counts can decrement before the blocked predecessor settles.

## Best Possible Solution

Land a rebased signal-aware queue cancellation fix that keeps pending-count accounting item-scoped and makes diagnostics either item-scoped or limited to stable per-key facts.

## Reproduction Assessment

Yes, source-reproducible: enqueue a non-settling first task, enqueue a same-key second task through withSessionActor, then abort the second caller before actor start; current main rejects the caller but pending only decrements from onSettle, which cannot run while the predecessor is stuck. I did not execute a test because this review was read-only.

## Solution Assessment

Mostly yes: queue-level cancellation is the right owner boundary because the manager cannot settle a queue item blocked behind a non-settling predecessor. The best merge shape is a rebased patch that keeps the cancellation behavior while guarding diagnostic updates by item identity.
