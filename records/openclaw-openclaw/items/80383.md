## Summary

Keep open: current main still has the abort-before-start queue accounting gap, and the PR’s queue-level cancellation approach is plausible with supplied live proof, but the branch still retains per-key diagnostics indefinitely after queue items settle.

## What This Changes

The PR adds AbortSignal-aware cancellation to SessionActorQueue, forwards withSessionActor’s signal into it, and adds focused queue regression tests.

## Best Possible Solution

Keep the queue-owned cancellation design, but make diagnostics item-scoped or delete per-key diagnostic state when a key becomes idle before merging.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: enqueue increments pending immediately, current cleanup waits for onSettle, and withSessionActor can reject the caller before the queued actor starts. I did not run the repro because this was a read-only review.

## Solution Assessment

Mostly yes: SessionActorQueue is the right owner for cancelling an already-enqueued slot while preserving same-key serialization. The diagnostics map needs cleanup or item-scoped ownership before this is the best mergeable implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clean up settled queue diagnostics:** `src/acp/control-plane/session-actor-queue.ts:210-216`
  - body: Every enqueue stores a diagnosticsBySession entry for the actor key, but onSettle only stamps settleTime and never removes the entry once the key is idle. In a long-running gateway this retains one diagnostic object for every session key ever queued after pendingBySession and the tail map have been cleaned up; make the diagnostic state item-scoped or delete it when the settling item owns the live idle entry.
