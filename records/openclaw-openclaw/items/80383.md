## Summary

Keep open: current main still has a source-reproducible abort-before-start queue accounting bug, and the PR’s queue-level cancellation approach fits the owner boundary, but the current patch adds a production diagnostics map that is never cleaned up and can be stale-mutated by older queue items.

## What This Changes

The PR adds AbortSignal-aware cancellation for queued ACP SessionActorQueue work, forwards withSessionActor signals into the queue, and adds focused SessionActorQueue tests.

## Best Possible Solution

Keep the queue-level cancellation design, but make diagnostic state item-scoped or remove/clean it so idle actor keys are not retained and older queue items cannot mutate newer diagnostics.

## Reproduction Assessment

Yes, from source inspection: current main enqueues pending work immediately, the manager can reject an abort-before-start caller, and queue cleanup waits for the blocked same-key tail to settle. I did not run a live repro because this was a read-only review.

## Solution Assessment

Mostly yes: SessionActorQueue is the right owner for cancelling an already-enqueued slot while preserving same-key serialization. The PR is not mergeable as-is until the new diagnostic map is removed, cleaned, or guarded by item ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid retaining settled diagnostics forever:** `src/acp/control-plane/session-actor-queue.ts:213-216`
  - body: Every enqueue stores a diagnosticsBySession entry for the actor key, but onSettle only stamps settleTime and never removes the entry once the key is idle. Because actor keys are session keys in a long-running gateway, this retains one diagnostic object for every session key ever queued; please either drop this production map or clean/update it only when the settling item still owns the live diagnostic entry.
  - confidence: 0.84
