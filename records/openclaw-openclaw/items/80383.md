## Summary

Keep open: the PR addresses a valid abort-before-start queue accounting bug and has sufficient real behavior proof, but the patch adds production diagnostics state that is never cleaned up after actor keys go idle.

## What This Changes

The PR adds AbortSignal-aware cancellation to SessionActorQueue, forwards withSessionActor signals into the queue, and adds focused SessionActorQueue tests.

## Best Possible Solution

Keep the queue-level cancellation design, but remove or item-scope and clean the diagnostic state so idle actor keys are not retained and older settle hooks cannot rewrite newer snapshots.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main increments pending at enqueue, only decrements on queued-task settlement, and withSessionActor can reject on abort before that queued task ever starts.

## Solution Assessment

Mostly yes: SessionActorQueue is the right owner boundary for cancelling an already-enqueued slot while preserving same-key serialization, but the new diagnostics map needs cleanup or item ownership before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Avoid retaining settled diagnostics forever:** `src/acp/control-plane/session-actor-queue.ts:213-216`
  - body: Every enqueue writes a diagnosticsBySession entry for the actor key, but onSettle only stamps settleTime and never deletes it once pendingBySession and the tail map are idle. In a long-running gateway this retains one diagnostic object per session key ever queued, and an older item settling after a newer enqueue can also stamp the newer snapshot; please drop the production map or make it item-owned and clean it when idle.
  - confidence: 0.84
