## Summary

Keep open: current main still accepts WebChat attachments without a size preflight, serializes them as inline base64 in a WebSocket request, and routes them through the same attachment parser family implicated by the reported stack overflow. An open PR already references this issue with closing syntax, so the issue should stay open until that fix path lands or is replaced.

## What This Changes

WebChat large attachment uploads should either succeed within supported limits or fail with a clear size/type error instead of showing a stack overflow.

## Best Possible Solution

Review and land a fix that covers picker, drop, and paste attachment paths, reconciles client feedback with the gateway size contract, and closes this issue together with https://github.com/openclaw/openclaw/issues/80679 after real browser proof.

## Reproduction Assessment

No, not as a live current-main reproduction in this review. The reporter gives clear steps, current source shows the same inline data URL to JSON.stringify/chat.send path, and https://github.com/openclaw/openclaw/issues/80679 has a matching stack in the same attachment parser family.

## Solution Assessment

Unclear for the open PR as written. A client-side guard is part of the likely fix, but the maintainable solution should cover all WebChat ingress paths and align with the existing gateway size-limit contract instead of only silently skipping picker/drop files over 4 MB.
