## Summary

Keep open: current main still routes WebChat attachments as base64 data inside a `chat.send` WebSocket request, while the documented/code limits imply files above 4 MB should either work or fail with a clear validation error.

## What This Changes

The issue reports that WebChat attachment uploads larger than about 4 MB fail with `Maximum call stack size exceeded` instead of uploading or showing a clear size-limit error.

## Best Possible Solution

Fix WebChat so attachments within the existing server and WebSocket limits send without a stack overflow, and add a clear client/server validation error for payloads that exceed those limits.

## Reproduction Assessment

No, not yet with high confidence: the reporter gives clear steps, and source inspection confirms the current inline base64 WebSocket path, but I did not establish a live browser reproduction or the exact stack frame on current main.

## Solution Assessment

Unclear: the likely fix belongs in WebChat attachment transport or preflight error handling, but the best narrow patch should first capture where the stack overflow occurs rather than only adding a generic size check.
