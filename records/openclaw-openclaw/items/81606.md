## Summary

Keep open: current main still lets WebChat attachments enter the inline data-URL/base64 send path without a size preflight, and the gateway still validates full attachment base64 strings before enforcing the existing size contract. An open fix PR already references this issue with closing syntax, so cleanup should wait for that PR or a replacement fix to land.

## What This Changes

Reports WebChat/Control UI failing with "Maximum call stack size exceeded" for attachments larger than about 4 MB and asks for successful upload or a clear limit message.

## Best Possible Solution

Land one coordinated fix that covers picker, drop, and paste attachment ingress, preserves or clearly surfaces the gateway media size contract, removes stack-prone full-payload validation where needed, and closes the related open attachment-crash reports after real browser/gateway proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The reporter gives concrete steps, current source shows the inline data URL to chat.send path, and the related open report has matching gateway stack evidence in the same attachment parser family.

## Solution Assessment

Unclear for the current proposed PR path. A client-side preflight is useful, but a maintainable fix should cover all WebChat attachment ingress paths and align with the existing gateway size contract instead of only silently dropping picker/drop files over 4 MB.
