## Summary

Keep open: current main still allows WebChat files above the reported threshold to enter the FileReader -> inline base64 `chat.send` -> WebSocket JSON frame path, and the gateway parser still validates the full base64 payload with a regex before size checks. The issue also has an open PR with closing syntax, so it should stay open until that fix path lands or is replaced.

## What This Changes

The issue asks WebChat to avoid a stack overflow and either upload successfully or show a clear limit message for attachments larger than about 4 MB.

## Best Possible Solution

Land one coherent fix that covers picker, drop, and paste preflight/user feedback while preserving the gateway attachment-size contract and removing the stack-prone gateway validation path; then close this issue with the linked parser reports.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. The reporter provides concrete WebChat steps, and current source shows the large inline attachment path through FileReader, `chat.send`, `JSON.stringify(frame)`, and gateway base64 validation.

## Solution Assessment

Unclear for the currently linked UI PR alone: a client-side guard is useful, but a silent 4 MB helper-only cap misses paste and conflicts with the 20 MB gateway contract. The better solution is a coordinated UI preflight/UX plus stack-safe gateway validation, or a narrower maintainer-selected subset that fully closes the reported path.
