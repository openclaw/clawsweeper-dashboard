## Summary

Keep open: current main still routes WebChat attachments as inline base64 and the gateway still validates the full payload with a regex before size enforcement. An open PR already references this issue with closing syntax, so the issue should stay open until that fix path lands or is replaced.

## What This Changes

The issue asks for WebChat uploads larger than about 4 MB to either succeed or fail with a clear size/type-limit message instead of showing “Maximum call stack size exceeded.”

## Best Possible Solution

Land one fix that removes the stack-prone gateway validation and adds consistent WebChat preflight/user feedback across picker, drop, and paste, with focused regression coverage and redacted real browser proof.

## Reproduction Assessment

No, not as a live current-main reproduction. The reporter steps, the related gateway logs in https://github.com/openclaw/openclaw/issues/80679, and current source give a source-reproducible path through WebChat data URLs, inline base64 chat.send attachments, and whole-payload regex validation.

## Solution Assessment

Yes for keeping this issue open until the linked fix path lands. The maintainable repair should combine a stack-safe non-decoding gateway validator with UI preflight/clear feedback aligned with the existing attachment size contract, rather than closing while the open PR is unmerged.
