## Summary

Keep open: current main still lets WebChat attachments become inline base64 WebSocket payloads without a size preflight, and live GitHub state shows an open PR with closing syntax for this issue. Cleanup should wait for that PR, or a replacement, to land.

## What This Changes

The issue reports WebChat failing with `Maximum call stack size exceeded` when sending attachments larger than roughly 4 MB.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/81824 with coverage for picker, drop, and paste attachment paths, aligned user feedback for the gateway size contract, and coordinated resolution of https://github.com/openclaw/openclaw/issues/80679 if the shared parser path is fixed.

## Reproduction Assessment

No, not as a live current-main run. The source path is clear from WebChat file intake through inline base64 `chat.send` serialization, and the related report at https://github.com/openclaw/openclaw/issues/80679 has matching gateway parser logs.

## Solution Assessment

Unclear for the current open PR as written. A client-side preflight is reasonable, but the maintainable fix should cover all WebChat ingress paths and align oversize behavior with the gateway attachment contract.
