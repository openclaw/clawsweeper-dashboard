## Summary

Keep open. The PR addresses a real chat.history truncation problem, but it is stale against current main, misses the current Control UI maxChars override path, and still budgets its UTF-8 fallback against raw text bytes instead of the JSON-serialized message bytes used by the hard placeholder guard.

## What This Changes

The PR raises the old gateway chat.history text cap to 64,000, adds raw UTF-8 byte-aware truncation, and adds gateway regression tests for long ASCII and emoji-heavy assistant text.

## Best Possible Solution

Rebase or replace the PR on current main by implementing serialized-byte-aware truncation in the shared projection path, aligning the Control UI request cap and config/help text with the chosen default, and covering both gateway and UI history behavior.

## Reproduction Assessment

Yes. Source inspection shows current main still truncates projected chat.history text before the serialized-byte guard, and a Node byte check shows the PR-specific raw-byte fallback can still exceed the JSON hard cap for escape-heavy text.

## Solution Assessment

No. The PR fixes part of the reported problem, but the maintainable fix must live in the shared projection path, account for the Control UI maxChars override, and budget truncation against serialized history bytes.
