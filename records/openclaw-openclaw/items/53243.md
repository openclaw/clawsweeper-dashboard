## Summary

Keep open: the underlying WebChat/Gateway history truncation is still source-reproducible, but this PR needs rebase/rework on the current projection path, serialized-byte budgeting, and real behavior proof before merge.

## What This Changes

This PR raises the legacy `chat.history` text cap to 64,000, adds raw UTF-8 byte-aware truncation, and adds gateway regression tests for long ASCII and emoji-heavy assistant history.

## Best Possible Solution

Reimplement the fix in `src/gateway/chat-display-projection.ts`, budget visible prefixes against serialized message size, align the Control UI/config/default cap policy, and cover long ASCII plus byte-heavy text with focused gateway/UI tests.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message can fit under the 128 KiB per-message byte guard, but current main applies the 8,000 default projection cap or the Control UI's 4,000 RPC cap before the byte guard runs.

## Solution Assessment

No. The branch edits the older server-methods helper, budgets raw text bytes instead of serialized message bytes, and does not align the current shared projection/UI cap path.

## Review Findings

Update the WebChat request policy or add focused coverage so this change fixes the user-facing history path, not only direct `chat.history` callers.
  - confidence: 0.86
