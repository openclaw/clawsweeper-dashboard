## Summary

Keep open. The PR targets a real WebChat/Gateway history truncation bug, but it is still conflicting, lacks after-fix real behavior proof, and does not yet fix the current shared projection plus Control UI cap path safely.

## What This Changes

The PR raises the old chat.history text cap to 64,000, adds raw UTF-8 byte-aware truncation before the oversized-message placeholder, and adds gateway regression tests for long ASCII and emoji-heavy assistant history text.

## Best Possible Solution

Rework the fix in src/gateway/chat-display-projection.ts, budget visible prefixes against serialized message size, settle the Control UI/config cap policy, and prove the result with focused gateway/UI tests plus real WebChat or chat.history output.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message can fit under the 128 KiB per-message byte guard, but current main applies the 8,000 default projection cap or the Control UI's 4,000 RPC cap before that guard runs.

## Solution Assessment

No. The branch edits the older server-methods helper, budgets raw text bytes instead of serialized message bytes, and does not align the current shared projection or Control UI cap path.
