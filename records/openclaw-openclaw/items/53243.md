## Summary

Keep open. Current main has not implemented the requested default-cap increase or visible byte-heavy truncation; this PR targets a real WebChat history problem paired with #53242, but its current patch is outdated and still budgets text by raw UTF-8 bytes instead of the serialized bytes used by the hard cap.

## What This Changes

This PR raises the gateway `chat.history` per-text cap, adds raw UTF-8 byte-aware truncation before oversized placeholder replacement, and adds gateway tests for long ASCII and emoji-heavy assistant replies.

## Best Possible Solution

Implement the behavior in the shared chat display projection path, preserve long text only when the fully serialized history message fits the hard cap, truncate byte-heavy and JSON-escape-heavy text to a visible prefix before placeholder replacement, and document the chosen default/config behavior with focused gateway and UI coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: current main truncates normal Control UI history through the 8,000-character default, while the PR-specific defect reproduces with JSON-escape-heavy text such as 64,000 NUL/control characters that fit the PR's raw byte budget but exceed the serialized 128 KiB hard cap.

## Solution Assessment

No. The PR addresses the right user problem, but the current patch should budget against serialized history bytes and move onto the shared projection helper rather than adding raw text-byte truncation to the old server-methods helper.
