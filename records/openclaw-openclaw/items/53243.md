## Summary

Keep open: the underlying chat.history truncation bug still exists on current main, but this PR is not correct as-is because it patches the old server-methods helper and budgets raw UTF-8 text bytes instead of the serialized history entry bytes used by the hard cap.

## What This Changes

This PR raises the old gateway chat.history text cap to 64,000, adds raw UTF-8 byte-aware truncation, and adds gateway regression tests for long ASCII and emoji-heavy assistant replies.

## Best Possible Solution

Implement this in src/gateway/chat-display-projection.ts, preserve long assistant text when the fully serialized history entry fits the hard cap, truncate UTF-8-heavy and JSON-escape-heavy text against the same serialized budget used by the oversized guard, and align the chosen default/config docs, tests, and changelog.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a default Control UI chat.history request for a 50,000-character ASCII assistant message is projected through the current 8,000-character default before byte caps; the PR-specific defect is reproducible with JSON-escape-heavy text that passes raw UTF-8 budgeting but exceeds serialized message bytes.

## Solution Assessment

No. The PR addresses the right user problem, but the best implementation should live in the shared chat display projection path and budget truncation against serialized history bytes rather than raw string bytes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89
