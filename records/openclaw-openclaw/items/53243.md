## Summary

Keep open: the WebChat/chat.history truncation problem is still present on current main, and recent related context makes this PR part of the canonical fix path, but the branch is not merge-ready because it targets an outdated helper, lacks real behavior proof, and still budgets truncation against raw text bytes instead of serialized message bytes.

## What This Changes

The PR raises the chat.history text cap to 64,000, adds raw UTF-8 byte-aware truncation in gateway chat history, and adds gateway regression tests for long ASCII and emoji-heavy assistant text.

## Best Possible Solution

Rework the fix on the shared chat-display-projection path, budget visible prefixes against serialized message size, align the gateway default/config/UI request-cap policy, and cover long ASCII plus byte-heavy text in focused gateway/UI tests.

## Reproduction Assessment

Yes, source-reproducible. Current main can truncate a 50k ASCII assistant message via the 8,000 default projection cap, and ordinary Control UI/WebChat history is capped even lower by the 4,000 maxChars request before the serialized byte guard runs.

## Solution Assessment

No, the PR is not the best solution as-is. The durable fix should live in src/gateway/chat-display-projection.ts and measure truncation against the same JSON-serialized message budget used by the hard omission path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure truncation against serialized history bytes:** `src/gateway/server-methods/chat.ts:425-427`
