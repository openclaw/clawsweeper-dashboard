## Summary

Keep open: this PR targets a real WebChat/Gateway truncation bug, but the branch is now conflicting, edits the older server-methods helper, still budgets raw text bytes instead of serialized message bytes, and lacks external-author real behavior proof.

## What This Changes

The PR raises the `chat.history` text cap to 64,000, adds raw UTF-8 byte-aware truncation in `server-methods/chat.ts`, and adds gateway tests for long ASCII and emoji-heavy assistant history.

## Best Possible Solution

Rework or replace the branch on the shared projection helper, preserve long text that fits the serialized history budget, visibly truncate byte-heavy or escape-heavy text before placeholder replacement, and settle the WebChat default/UI cap policy with focused gateway and UI tests.

## Reproduction Assessment

Yes, source-reproducible. Current main truncates by the 8,000-character default, and Control UI requests a 4,000-character cap, before the 128 KiB serialized message guard can preserve longer assistant text that still fits.

## Solution Assessment

No. The current PR is not the best fix because it edits the older server-methods truncation helper and budgets raw text bytes; the maintainable path is the shared projection helper using the same serialized-byte budget as the hard placeholder gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Budget history truncation with serialized message bytes:** `src/gateway/server-methods/chat.ts:425-427`
  - body: The fallback trims candidates with `Buffer.byteLength(candidate, "utf8")`, but the later oversized gate measures the JSON-serialized message. Escape-heavy text such as quotes, backslashes, or control characters can stay under this raw text budget, expand past the 128 KiB cap when serialized, and still be replaced entirely by the placeholder; budget the prefix against the serialized message bytes instead.
