## Summary

Keep open: the underlying chat-history truncation problem is still source-reproducible on current main, but this PR is not merge-ready because it targets stale gateway plumbing, budgets raw text bytes instead of serialized message bytes, and lacks after-fix real behavior proof.

## What This Changes

The PR raises chat.history's text cap to 64,000, adds raw UTF-8 byte-aware truncation, and adds gateway tests for long ASCII and emoji-heavy assistant history.

## Best Possible Solution

Implement the behavior in src/gateway/chat-display-projection.ts, budget visible prefixes against serialized message size, align the gateway/default/UI cap policy, and cover long ASCII plus byte-heavy text with focused gateway and UI tests.

## Reproduction Assessment

Yes, source-reproducible. Current main applies an 8,000 default cap, or Control UI's 4,000 RPC cap, before the 128 KiB serialized byte guard, so a 50k ASCII assistant message can be sliced even though it fits the byte budget.

## Solution Assessment

No. The PR updates the old server-methods helper and measures raw UTF-8 text bytes, while the durable fix should live in the shared projection helper and use the same serialized-byte accounting as the hard placeholder gate.
