## Summary

Keep open. The underlying WebChat/chat.history truncation problem is still source-reproducible on current main, but this PR is not merge-ready because it targets stale server-methods logic, budgets raw text bytes instead of serialized message bytes, and lacks after-fix real behavior proof.

## What This Changes

The PR raises the chat.history text cap, adds raw UTF-8-aware truncation in gateway chat handling, and adds long ASCII/emoji assistant-history regression tests.

## Best Possible Solution

Implement the fix in the shared chat-display-projection path, budget visible prefixes against serialized message size, align the gateway default/config/UI cap policy, and cover long ASCII plus byte-heavy text with focused gateway and UI tests.

## Reproduction Assessment

Yes, source-reproducible. A 50k ASCII assistant message can fit under the 128 KiB per-message byte guard, but current main applies the 8,000 default projection cap, or the Control UI's 4,000 RPC cap, before the serialized byte guard runs.

## Solution Assessment

No. The PR direction addresses a real bug, but the patch edits the older helper and measures raw UTF-8 bytes while current main projects through chat-display-projection.ts and the hard omission path measures JSON-serialized message bytes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Measure truncation against serialized history bytes:** `src/gateway/server-methods/chat.ts:425-427`
  - body: The fallback trims candidates with `Buffer.byteLength(candidate, "utf8")`, but the later oversized gate measures the JSON-serialized message. Escape-heavy text such as quotes, backslashes, or control characters can stay under this raw text budget, expand past the 128 KiB single-message cap when serialized, and still be replaced entirely by the placeholder; budget the prefix against serialized history bytes instead.
