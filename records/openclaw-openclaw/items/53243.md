## Summary

Keep open: the reported user problem is still valid on current main, but this PR is not merge-ready because the implementation targets the old truncation location and its byte-aware fallback still measures raw string bytes instead of the JSON-serialized bytes used by the hard cap. The paired issue is still open and the merged configurable-limit work does not fully supersede this behavior change.

## What This Changes

This PR raises the old gateway chat.history text cap to 64,000 characters, adds raw UTF-8 byte-aware fallback truncation, and adds gateway regression tests for long ASCII and emoji-heavy assistant replies.

## Best Possible Solution

Implement the behavior in src/gateway/chat-display-projection.ts, preserve long assistant text when the fully serialized history entry fits the hard cap, truncate UTF-8-heavy and JSON-escape-heavy text against the same serialized budget used by the oversized guard, and align the chosen default/config docs and tests with that contract.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a default Control UI chat.history request for a 50,000-character ASCII assistant message is projected through the current 8,000-character default before byte caps. The PR-specific defect is reproducible with JSON-escape-heavy text that passes raw UTF-8 budgeting but exceeds serialized message bytes.

## Solution Assessment

No. The PR addresses the right user problem, but the best current implementation should live in the shared chat display projection path and budget truncation against serialized history bytes rather than raw string bytes.
