## Summary

Keep open. Current main still has a generic resolver path where an explicit directory command treats a loaded channel plugin without a `directory` adapter as installable/missing, so the reported Weixin prompt is consistent with source behavior and not fixed on main.

## What This Changes

The issue reports that `openclaw directory peers list --channel openclaw-weixin` prompts to install `openclaw-weixin` even though the plugin is already installed, enabled, and running.

## Best Possible Solution

Keep the directory/install behavior channel-agnostic: if a trusted installed plugin resolves for the requested channel but lacks `directory` or the requested directory method, the CLI should report unsupported or empty-directory behavior rather than offering to reinstall; only genuinely missing installable channels should enter the installer.

## Reproduction Assessment

Yes. A focused unit reproduction can mock an explicit catalog-backed channel whose loaded plugin lacks `directory` and assert `resolveInstallableChannelPlugin` or the directory CLI does not call `ensureChannelSetupPluginInstalled`; the current code path would call it.

## Solution Assessment

Yes, the best fix is a narrow resolver or directory-CLI guard that distinguishes missing plugin from installed-but-unsupported capability. Do not special-case `openclaw-weixin` or implement WeChat directory data in core.
