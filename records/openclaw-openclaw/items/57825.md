## Summary

Keep this PR open. Current main still has the session-persistence behavior the PR is trying to fix: omitted override fields are deleted whenever the shared helper receives `clearedFields`, and the agent wrapper still passes the full override field list for every wrapper write. The PR’s intended opt-in clear semantics remain useful, but the branch needs rebase/adaptation to the current `attempt-execution.shared.ts` layout.

## What This Changes

The PR adds opt-in clearing for missing session override fields and regression coverage so generic session writes preserve model/provider overrides unless a reset path explicitly clears them.

## Best Possible Solution

Land the intended opt-in clearing behavior in the current shared helper: preserve omitted override fields by default, require an explicit flag for reset/cleanup paths, and keep tests proving both preserve-by-default and explicit clearing semantics.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed a session store entry with `providerOverride` and `modelOverride`, then persist a later entry through the wrapper that omits those fields; `mergeSessionEntry` preserves them first, but `persistSessionEntry()` then deletes them because they are listed in `clearedFields`.

## Solution Assessment

Yes in direction, but not as-is against current main. The opt-in clear flag is the narrow maintainable fix, but it must be moved to `src/agents/command/attempt-execution.shared.ts` and threaded through the current wrapper/reset paths.
