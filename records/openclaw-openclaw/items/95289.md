## Summary

Keep open. The PR is aimed at the right shared Codex app-server refresh path, but it still has a blocking auth-provider issue: the new handler captures a scoped `authProfileStore` while the shared-client cache key ignores scoped-store identity, so a later same-profile shared client acquire can refresh against the wrong store.

## What This Changes

The PR adds shared Codex app-server OAuth refresh handling, forwards scoped auth stores through direct and startup shared-client paths, and adds focused regression tests.

## Best Possible Solution

Keep the shared refresh-handler approach, but make shared-client ownership scoped to the auth store or avoid sharing when a scoped store is supplied, then land after a two-store regression and normal changed/type validation.

## Reproduction Assessment

Yes. Source inspection shows bound Codex conversations use leased shared clients, current main lacks a shared refresh handler, and upstream Codex requires an `accessToken` response; I did not rerun live Telegram in this read-only review.

## Solution Assessment

No. The handler is in the right layer, but the current PR captures scoped auth state on a reused shared client without making that scoped state part of client ownership.
