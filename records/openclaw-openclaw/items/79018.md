## Summary

Keep open: current main still ships the shared-network `openclaw-cli` service with dropped network capabilities, and the Docker/plugin docs make that CLI service the normal post-start command surface. The report has a concrete v2026.5.6 Docker Desktop repro, but the proposed capability-based root cause needs maintainer/security review because current source shows both gateway and CLI drop `NET_RAW`/`NET_ADMIN`.

## What This Changes

The issue asks for Docker Compose or upgrade guidance changes so `openclaw plugins install` works from the stock `openclaw-cli` container instead of failing DNS resolution.

## Best Possible Solution

Live-reproduce the Docker Desktop DNS failure, then choose the least-privilege Compose or docs change that restores plugin install usability without undoing container hardening unnecessarily.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, the issue provides concrete v2026.5.6 Docker Desktop `docker exec` DNS and registry probes, and current source still matches the relevant compose shape; this read-only review did not execute Docker, so the exact root cause remains live-unconfirmed.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Removing capability drops may fix the symptom, but those drops were added as explicit shared-namespace hardening, so maintainers should first confirm the DNS failure and then pick a security-reviewed resolver or documentation path.
