## Summary

Keep open: current main still resolves doctor repairs and Linux user-service install paths from the caller's environment, while the duplicate-service guard only blocks system-level units, so the requested cross-user/root user-service warning remains missing.

## What This Changes

Request a `doctor --fix` warning or confirmation before repairing the caller's home when a running gateway appears to belong to a different Unix user or home.

## Best Possible Solution

Add a doctor preflight or repair guard for discoverable live-gateway home/owner mismatches and document detection limits, while keeping current state-dir semantics unchanged.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection gives a medium-confidence path: run `doctor --fix` from a different Unix home while the live gateway is only another user's systemd user service, because current main scans system units but not that other home.

## Solution Assessment

Yes directionally: a warning or confirmation before mutable repairs is narrower than changing path resolution. The maintainer decision is the detection source, privilege boundary, and whether the prompt gates all writes or only service repair/install.
