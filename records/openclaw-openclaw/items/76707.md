## Summary

Keep open: current main has a guard for system-level Linux gateway units, but it still does not show the requested warning when the live gateway is a root or other-user systemd user service before doctor repairs the caller's home.

## What This Changes

Request a doctor warning or confirmation when the current Unix user/home differs from an already-running gateway's user or config path before applying current-home repairs or installing a separate user service.

## Best Possible Solution

Add a doctor preflight before mutable repairs and service installation that detects safely discoverable live-gateway ownership or config-home mismatches, asks for explicit confirmation, and documents detection limits and privacy handling.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection gives a medium-confidence path: run doctor from one Unix home while the live gateway is represented only by a root or other-user systemd user service, and current main still resolves repairs and user-service discovery to the caller's environment.

## Solution Assessment

Yes directionally: a clear warning or confirmation is narrower than changing the selected state directory implicitly. The maintainer decision is the exact safe detection scope and whether the prompt should gate all mutable repairs or only service install/repair steps.
