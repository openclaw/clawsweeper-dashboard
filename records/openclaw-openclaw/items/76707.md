## Summary

This should stay open: current `main` has good system-level duplicate-service safeguards, but the reported root-user-service/current-user-home mismatch is still plausible because doctor repairs the invoking user’s config and service scope.

## What This Changes

Add a doctor UX guard so `openclaw doctor --fix` warns before repairing the current Unix user’s OpenClaw home when a likely live gateway belongs to another user/home.

## Best Possible Solution

Add a high-signal doctor warning or confirmation gate before service/config repair when the active port listener appears to be an OpenClaw gateway owned by a different Unix user than the current doctor run.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path even though I did not run a live multi-user systemd repro: a non-root doctor run uses the current home and can install the current user’s service while only system-level services are hard-blocked.

## Solution Assessment

Yes, the narrowest maintainable fix is a doctor-side warning/confirmation using existing port/service diagnostics; it should not migrate startup config or scan arbitrary private home directories.
