## Summary

Keep open. Current main has adjacent safeguards for service rewrites and status drift, but I did not find a doctor warning that protects a non-root repair run from targeting the caller's separate Unix home while a live OpenClaw gateway is running under another user.

## What This Changes

Add a high-signal `doctor --fix` warning before repairs target the current Unix home when the live gateway appears to run from another user/home.

## Best Possible Solution

Keep this open and add a conservative pre-repair warning/confirmation that reports the current doctor home/config path versus the detected running gateway context, without trying to repair another user's service automatically.

## Reproduction Assessment

Yes, from source inspection rather than a live root/non-root host: doctor resolves config and service paths from the current process environment, while another user's systemd user service is outside the normal scan/read path.

## Solution Assessment

Yes. A warning/confirmation before repair writes is the narrowest maintainable fix; automatic cross-user repair or sudo inspection would be riskier and outside the issue's requested UX boundary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76707. Add a conservative `openclaw doctor --fix` warning before repair writes when, on Unix/Linux, doctor can detect a likely running OpenClaw gateway whose user/home/config context differs from the current doctor process. Keep the boundary diagnostic-only: do not scan protected homes with sudo, do not repair another user's config or service, do not add a config option, and preserve existing `OPENCLAW_SERVICE_REPAIR_POLICY=external` behavior. Prefer existing helpers such as gateway service/env config path resolution and `inspectPortUsage`/port listener metadata. In interactive repair mode, show a clear warning/confirmation; in non-interactive mode, emit the warning and avoid creating or rewriting a second user service unless an existing update-repair path explicitly owns that operation. Add regression coverage for a non-root current home with a detected OpenClaw listener owned by a different Unix user/root and for ordinary same-user/profile cases. Update `docs/cli/doctor.md` and add a one-line `CHANGELOG.md` entry if behavior changes user-visible output.
