## Summary

Keep open: current main has a guard for detectable system-level gateway services, but it does not cover another Unix user's user-scoped/root gateway home before Doctor repairs the current user's environment.

## What This Changes

Request that `openclaw doctor --fix` warn before repairing the current user's home/service when the live gateway appears to belong to another Unix user or home.

## Best Possible Solution

Add a focused Doctor preflight/repair guard that detects a live or port-owning OpenClaw gateway outside the current effective home, warns before state/service repairs, redacts cross-user paths where needed, and covers the behavior with Linux systemd tests and Doctor docs.

## Reproduction Assessment

Yes, at source level. With the current user's service unloaded and only another user's user-scoped gateway present, current main only blocks system-level units, then can proceed to build/install service repair from the current process environment.

## Solution Assessment

No current fix is in place. The requested warning is the right direction, but the best implementation should avoid broad cross-home scans or raw path disclosure and should align with the existing system-level service guard.
