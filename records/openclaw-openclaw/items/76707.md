## Summary

Keep open: current main has safeguards for system-level Linux gateway units and current-user service repair, but source/docs/tests still do not show the requested warning when a live root or other-user systemd user service is the gateway being maintained.

## What This Changes

The issue requests a clear `doctor --fix` warning before repairs target the caller’s Unix home while the live gateway appears to run from another Unix user/home.

## Best Possible Solution

Add a targeted doctor preflight or repair guard that warns before mutable repairs when a discoverable live gateway belongs to a different Unix user/home, and document the detection limits.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: run doctor from a different Unix home while the live gateway is only represented by another user’s systemd user service, and current main still resolves repairs to the caller’s state dir.

## Solution Assessment

Yes directionally: a clear warning or confirmation before mutable repair is narrower than silently changing the selected state directory. The unresolved maintainer decision is the detection scope and prompt timing.
