## Summary

Keep this open: current `main` has a system-level service guard, but the reported root/other-user systemd user-service case can still flow from a port warning into current-user service repair or install.

## What This Changes

The issue requests a clearer `openclaw doctor --fix` warning when the CLI is repairing the current Unix user's OpenClaw home while a live gateway appears to be running from another Unix user or home.

## Best Possible Solution

Add a conservative doctor warning or confirmation before service repair/install when a running OpenClaw listener or service appears to belong to a different Unix user or home, with focused regression coverage and docs/changelog updates.

## Reproduction Assessment

Yes, at source level. On Linux, if the current user's service is not loaded, a busy OpenClaw port owned by another user is only reported as port diagnostics before doctor can proceed to current-user service install unless a system-level unit is found.

## Solution Assessment

Yes. The narrow maintainable solution is to extend the existing doctor gateway-service guard using current port/process/service evidence, not to migrate homes or add a new policy knob.

## Repair Work Prompt

Fix the UX gap from https://github.com/openclaw/openclaw/issues/76707. In `openclaw doctor --fix`, before installing/repairing/restarting a current-user gateway service, detect when the default/configured gateway port is already occupied by an OpenClaw process owned by a different Unix user or otherwise associated with a different service/config/home, and emit a high-signal warning/confirmation that this doctor run targets the current user's OpenClaw home, not the live gateway. Keep the change conservative: do not migrate config/state between users, do not add a config option, and preserve the existing system-level service guard and `OPENCLAW_SERVICE_REPAIR_POLICY=external` semantics. Add focused tests for the other-user listener case and the existing system-level guard, update user-facing docs if command output or behavior changes, and add a single-line changelog entry because this is user-facing doctor UX.
