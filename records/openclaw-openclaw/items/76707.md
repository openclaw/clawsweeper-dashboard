## Summary

Keep open: current main has safeguards for system-level Linux units and multiple state dirs, but it still does not warn before current-home repairs when the live gateway is another user's user service or root-home environment.

## What This Changes

The issue requests a high-signal doctor warning before applying repairs to the current Unix home when a discoverable live gateway belongs to another Unix user or home.

## Best Possible Solution

Add a doctor preflight or repair guard that detects discoverable live-gateway ownership/path mismatches before mutable repairs and requires explicit confirmation, with docs describing detection limits.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. Source inspection gives a medium-confidence path: run doctor --fix from a non-root home while the live gateway is represented only by another user's systemd user service, and current main still resolves repairs to the caller's state dir without scanning that other user service.

## Solution Assessment

Unclear until maintainers choose the detection scope. A warning/confirmation before mutable repairs is the narrowest direction, but the implementation should decide whether to rely on service environment metadata, root-home scanning, port/process heuristics, or systemd inspection.
