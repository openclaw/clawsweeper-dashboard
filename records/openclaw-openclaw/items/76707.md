## Summary

Keep open: current main and v2026.6.10 still repair and install services from the current process HOME unless they see a system-scope unit, so another Unix user's systemd user gateway can still be missed. The open related PRs address sibling-home scanning and user+system dueling units, but not this exact live-gateway/wrong-home warning.

## What This Changes

Warn or block `openclaw doctor --fix` before it repairs the current Unix home when a live gateway or config appears to belong to another Unix user or home.

## Best Possible Solution

Define the cross-user mismatch policy, then make Doctor compare its current repair target with verified live/supervisor gateway context and warn or block before writes or user-service installs without broad sibling-home disclosure.

## Reproduction Assessment

Yes, source-level: current code derives user service and state paths from the command's HOME and only blocks system-scope gateway services before offering current-user repairs. I did not run a live multi-user systemd repro in this read-only pass.

## Solution Assessment

Unclear until maintainers choose the policy. The existing system-scope guard helps `/etc` units, but the best fix needs verified live/supervisor context for another user's gateway without unsafe cross-home enumeration.
