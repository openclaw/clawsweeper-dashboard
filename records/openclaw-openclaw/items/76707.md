## Summary

Keep open: current main still blocks only detectable system-level duplicate gateway services, not another Unix user's user-scoped/root gateway home, and the related cross-user path-disclosure concern means the fix needs maintainer UX/security policy.

## What This Changes

Request a Doctor warning or confirmation when `doctor --fix` appears to repair the current Unix user's OpenClaw home while a live gateway belongs to another user or home.

## Best Possible Solution

Add a focused Doctor preflight/repair guard that detects a live or port-owning OpenClaw gateway outside the current effective home, warns before state/service repairs, redacts cross-user paths where needed, and documents/tests the Linux policy.

## Reproduction Assessment

Yes, at source level. With the current user's service unloaded and another user's user-scoped gateway present, current main only blocks system-level units and can still build/install a service from the current process environment.

## Solution Assessment

Unclear: the requested warning is the right direction, but the maintainable implementation needs maintainer policy for cross-user detection, confirmation text, and path redaction.
