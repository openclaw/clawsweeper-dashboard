## Summary

Keep this open: current `main` has nearby safeguards for duplicate state dirs, system-level gateway services, and busy ports, but I did not find a pre-repair warning for the specific case where `doctor --fix` is run from one Unix account while the reachable gateway is owned by another account/home.

## What This Changes

The issue asks for `openclaw doctor --fix` to warn before repairing the current Unix user's OpenClaw home when a live gateway appears to belong to another user or home.

## Best Possible Solution

Add an early doctor preflight warning, before repair writes or service install prompts, when the default/current gateway port or readable service metadata indicates a live OpenClaw gateway owned by a different Unix user/home.

## Reproduction Assessment

No live root/non-root host reproduction was run in this read-only review. Source inspection gives a high-confidence path: current doctor repair flows operate on the current process home/config, while existing duplicate-service and state-dir checks do not cover another user's systemd user service under root.

## Solution Assessment

Yes, the requested warning is a narrow maintainable direction. The safest implementation should reuse existing doctor/service/port inspection seams and avoid changing config resolution, ownership repair, or automatic cross-user service management.
