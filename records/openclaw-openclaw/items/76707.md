## Label Justifications

- `P2`: The issue is a bounded doctor/operator safety improvement with meaningful wrong-home side effects but no evidence of a widespread runtime outage.
- `impact:security`: Repairing the wrong Unix home or service can confuse credential and service ownership boundaries during host maintenance.

## Summary

Keep open: current main has a useful guard for Linux system-scope gateway services, but it does not yet cover the reported cross-user/root systemd-user home case or prove that `doctor --fix` warns before earlier config/state repair mutates the wrong OpenClaw home.

## What This Changes

Warn before `openclaw doctor --fix` repairs the current Unix user's OpenClaw home when a live gateway appears to be running from another Unix home or user.

## Best Possible Solution

Add a maintainer-approved doctor preflight that detects or clearly warns about cross-user live gateway/state ownership before mutating config, state, or user services, while preserving documented system-scope and external-supervisor behavior.

## Reproduction Assessment

Yes from source for the remaining gap: current detection covers system unit dirs and the current user's user service, but not another user's or root's systemd-user home; I did not run a destructive live multi-user repro.

## Solution Assessment

No: the current system-scope guard is a useful partial mitigation, but it is too narrow for the reported root/other-user home case and runs after earlier repair paths can already touch state.
