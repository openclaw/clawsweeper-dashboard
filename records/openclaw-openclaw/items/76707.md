## Summary

Keep open: current main has related safeguards for system-level units and multiple state dirs, but the Linux service discovery path does not cover another user's user-systemd gateway such as root's home, so the requested warning is still useful.

## What This Changes

Request a clear `doctor --fix` warning before repairs target the current user's OpenClaw home while a live gateway appears to be running from another Unix user/home.

## Best Possible Solution

Add a targeted doctor safeguard that detects cross-user live gateway evidence before creating or repairing the current user's home/service, then document the operator workflow for root-owned or externally supervised gateways.

## Reproduction Assessment

Yes, source-reproducible. With a non-root operator whose user service is missing, current main only blocks system-level Linux units and only scans the operator's home, so a root user-systemd gateway is not enough to trigger the requested mismatch warning.

## Solution Assessment

Yes. A narrow warning/confirmation in doctor gateway service repair, backed by port-listener or service-discovery evidence, is more maintainable than migrating or editing another user's OpenClaw state.

## Repair Work Prompt

Implement a focused doctor safeguard for issue https://github.com/openclaw/openclaw/issues/76707. When `openclaw doctor --fix`/repair is running on Linux and the current user's gateway service/home is not the apparent live gateway, show a high-signal warning before creating or repairing the current user's service/state. Prefer already-available evidence such as port listener process user/command and existing service-discovery metadata; do not add broad privileged scanning of arbitrary home directories or mutate another user's config/service. Keep core behavior generic, profile-aware, and compatible with intentional isolated gateways. Add regression coverage around a root-owned or different-user OpenClaw listener/user-systemd scenario and ensure the existing system-level unit block still works. Update user-facing docs and add a single-line CHANGELOG.md entry if the behavior/copy changes.
