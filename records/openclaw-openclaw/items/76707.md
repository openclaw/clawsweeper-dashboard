## Summary

Keep open: current main has a guard for system-level Linux gateway services, but the reported root-owned systemd user-service case is still a real gap in the doctor repair path.

## What This Changes

The issue asks `openclaw doctor --fix` to warn before repairing the current Unix user's OpenClaw home when an already-running gateway appears to belong to another user/home.

## Best Possible Solution

Add a Linux doctor preflight that detects an OpenClaw gateway already running under a different Unix user/home or service config path and blocks or strongly confirms current-home service repairs before creating another user service.

## Reproduction Assessment

Yes, source-reproducible: with the current user's service missing, current main only checks system-level units before offering current-user service install, while a root-owned user service is outside that scan. I did not live-run it because the review must not mutate the host.

## Solution Assessment

No, the current solution is too narrow because it handles system-level duplicate services but not another Unix user's user service or live gateway listener. The maintainable fix is a targeted doctor warning/guard before service install or repair, preserving existing external-service policy behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76707. On Linux, when `openclaw doctor --fix` or repair mode would install/repair a current-user gateway service and a local OpenClaw gateway appears to already be running under a different Unix user/home or service config path, emit a high-signal warning and avoid silently creating a second home/service. Start in `src/commands/doctor-gateway-daemon-flow.ts` using existing `inspectPortUsage` listener data plus service/systemd metadata where appropriate; preserve the existing system-level service guard and `OPENCLAW_SERVICE_REPAIR_POLICY=external` behavior. Add regression coverage in `src/commands/doctor-gateway-daemon-flow.test.ts` for `service.isLoaded=false`, `findSystemGatewayServices=[]`, and a busy OpenClaw listener owned by another user, asserting doctor does not proceed silently to `service.install`. Update user-facing docs and `CHANGELOG.md` if behavior changes. Do not add a config option and do not scan arbitrary user homes without an explicit, permission-safe reason.
