## Summary

Keep open. Current `main` already avoids installing a second user service when it sees a system-level Linux gateway unit, but the implemented scans do not cover a gateway running as another Unix user's user-systemd service, so the reported UX gap remains actionable.

## What This Changes

Add a clear warning before `openclaw doctor --fix` repairs the current Unix user’s OpenClaw home when the live gateway appears to belong to another Unix user/home.

## Best Possible Solution

Add a narrow pre-repair diagnostic in the doctor gateway-service/state path that compares the current target home/state/config with detectable live gateway or supervisor metadata and emits a high-signal warning before any write or service install proceeds.

## Reproduction Assessment

Yes, source inspection gives a credible reproduction path: on Linux, run doctor from a user whose service is missing while a root-owned user-systemd gateway owns the real config/state. I did not live-run it because this review is read-only and the scenario requires host service setup.

## Solution Assessment

Unclear. A warning-only guard looks like the narrowest maintainable direction, but the exact detection and blocking policy needs maintainer judgment because cross-user service metadata may not be readable.
