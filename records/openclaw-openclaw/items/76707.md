## Summary

Keep open: current main has guards for system-level Linux duplicate gateway services and port-owner diagnostics, but source/docs/tests still do not show the requested warning before repairing the caller's current home when the live gateway is another Unix user's user service or root home.

## What This Changes

The issue requests a clear `doctor --fix` warning/confirmation when the current Unix user/home differs from a discoverable live gateway user/home.

## Best Possible Solution

Add a doctor preflight or repair guard that warns before mutable current-home repairs when discoverable service, port, or config evidence points to a different live gateway home, and document its Linux permission limits.

## Reproduction Assessment

Yes, with medium confidence from source inspection: run doctor from a different Linux home while the live gateway is represented only by another user's user service/root home, and current main still resolves repairs to the caller's state dir while only system-level units block user-service installation. I did not run a live cross-user systemd setup in this read-only review.

## Solution Assessment

Yes directionally: warning/confirmation before mutable repairs is narrower than silently switching homes or blocking all repairs. The open product question is how broadly doctor should probe other Unix users/root and which repair stages should require confirmation.
