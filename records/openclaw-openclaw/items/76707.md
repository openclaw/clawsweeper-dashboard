## Summary

Keep open: current main still targets the caller's state/config and systemd user service, while the existing Linux safeguard only blocks system-level units; it does not warn for another user's user-service gateway before current-home repairs.

## What This Changes

Request a clear doctor warning or confirmation before `doctor --fix` repairs the current Unix home when a running gateway appears to belong to another user or home.

## Best Possible Solution

Add a doctor preflight or repair guard that detects discoverable live-gateway ownership/path mismatches and asks for explicit confirmation before mutable repairs or service installation, with docs stating detection limits.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: run doctor from a different Unix home while the live gateway is only represented by another user's systemd user service, and current main still resolves repairs to the caller's home.

## Solution Assessment

Unclear at implementation level. The requested warning/confirmation is directionally the narrowest UX fix, but maintainers should decide the supported cross-user detection scope and prompt timing before implementation.
