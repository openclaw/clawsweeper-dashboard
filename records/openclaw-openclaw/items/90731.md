## Summary

This PR is still necessary and its success-path proof is convincing, but it is not merge-ready because a transient dynamic metadata failure can persist an incomplete catalog as a successful cache entry, preventing the intended retry and preserving the gateway thinking rejection.

## What This Changes

The PR hydrates exact allowlisted dynamic model metadata during full catalog loads and uses that metadata for gateway and agent thinking validation.

## Best Possible Solution

Persist a full catalog only after all exact-ref hydration succeeds; partial results may be returned to the current caller but must bypass both persistent and in-memory caches, with a regression test that simulates a real state-cache read after the failure.

## Reproduction Assessment

Yes. Make the first dynamic hydration throw, persist the returned partial catalog under the same cache key, then call `loadModelCatalog` again; the branch returns the partial state-cache hit without invoking the provider hook.

## Solution Assessment

No in its current form. Exact-ref hydration is the correct direction, but partial catalogs must not be persisted as successful state before the hydration-failure flag is evaluated.

- **[P1] Do not persist a partial catalog before rejecting it:** `src/agents/model-catalog.ts:883-888`
  - body: When any dynamic allowlist hydration fails, this writes `sorted` to the persistent agent catalog cache and only afterward clears the promise or throws `PartialModelCatalogError`. The next load hits that persisted partial catalog at the earlier cache read, skips provider hydration, and treats the incomplete metadata as successful, so a transient `/api/show` failure can keep gateway thinking validation off-only. Skip the state-cache write on hydration failure and test a second load using cache behavior that stores the first result.
