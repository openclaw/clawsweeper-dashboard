## Summary

Keep open: the linked bug is valid and this PR targets the right failover boundary, but the patch does not actually make the supplied Moonshot/Kimi payload classify as billing because `rate_limit_reached` is matched before the balance text can win. The PR is also still draft and its supplied proof is source-probe output rather than convincing real behavior proof.

## What This Changes

The PR threads provider identity into failover HTTP-status classification and adds a Moonshot/Kimi 429 billing regression test.

## Best Possible Solution

Keep the narrow failover classifier fix, but change the matcher precedence or add a dedicated Moonshot/Kimi 429 balance-text check so exhausted-balance payloads return `billing` while ordinary 429s remain `rate_limit`.

## Reproduction Assessment

Yes, source-reproducible. Current main's explicit 429 branch returns `rate_limit`, and source inspection shows this PR's own payload still hits the generic rate-limit matcher before the new billing-only branch can apply.

## Solution Assessment

No. The fix belongs in this narrow classifier path, but the implementation must let high-confidence Moonshot/Kimi balance text win before or inside the 429 status override instead of depending on a `billing` message reason that the sample does not produce.
