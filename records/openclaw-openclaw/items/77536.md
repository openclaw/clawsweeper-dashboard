## Summary

Close: current main already implements the gateway-owned package update guard through the merged #75819 fix, shipped in v2026.5.3; this PR is a stale, less safe duplicate and also reverts unrelated diagnostics plugin trust behavior.

## What This Changes

The PR adds service-marker-based gateway process-tree detection in process respawn and removes trusted-official diagnostics plugin plumbing, tests, and a changelog entry.

## Best Possible Solution

Keep the shipped #75819 PID-ancestry package-update guard and handle the external updater design in the separate #77511 maintainer discussion.

## Reproduction Assessment

Yes, source and issue evidence give a high-confidence reproduction path for the original bug: #75691 documents the systemd in-lane update case, and current main now compares the active Gateway runtime PID with the caller ancestry before package mutation. I did not run a live systemd smoke because this was a read-only review.

## Solution Assessment

No, this PR is not the best way to solve it. Current main's #75819 fix is narrower because it uses PID ancestry in the package update command before mutation, while this PR relies on inherited service markers and also removes unrelated diagnostics plugin trust behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Use PID ancestry instead of service markers:** `src/infra/process-respawn.ts:80-86`
  - body: `OPENCLAW_SERVICE_MARKER` is present on the supervised Gateway process itself, not only on child exec lanes. Returning `failed` from `restartGatewayProcessWithFreshPid()` when those markers exist makes ordinary supervised Gateway restarts fall back to in-process restart instead of exiting for systemd/launchd; the shipped fix uses the managed Gateway runtime PID ancestry check in the update command instead.
  - confidence: 0.9
- **[P2] Preserve official diagnostics installs:** `src/plugins/services.ts:27-31`
  - body: This change makes internal diagnostics available only to bundled diagnostics exporters and removes the `trustedOfficialInstall` path. That regresses the current-main behavior for npm-installed `@openclaw/diagnostics-prometheus`, which is intentionally granted the internal diagnostics bus only after official install provenance checks.
  - confidence: 0.93
- **[P3] Restore the unrelated diagnostics changelog entry:** `CHANGELOG.md:55`
  - body: The PR deletes the active changelog entry for the official diagnostics exporter fix even though this branch is about gateway update restart behavior. Dropping that unrelated release note would hide the #76628 user-facing fix from the next changelog.
  - confidence: 0.86
