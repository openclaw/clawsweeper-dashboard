## Label Justifications

- `P1`: Multiple users are blocked in Android onboarding with no workaround when node connectivity is healthy but the operator session is offline.
- `impact:session-state`: The failure comes from conflating operator-session connectivity with device-node readiness during onboarding state handling.

## Summary

Keep open: current `main` and the latest release still require both operator and node connectivity before Android onboarding can finish, so the reported `Connected (operator offline)` state remains blocked. A same-head fork PR with closing syntax is still open, so the issue should stay open until maintainers land, replace, or close that fix path.

## What This Changes

Android onboarding should allow completion once the device node is paired and connected even if the operator session is offline.

## Best Possible Solution

Review and land a narrow Android fix, or replace the open fork PR, so node connectivity is sufficient for final onboarding while disconnected-node states remain blocked and covered by focused unit tests.

## Reproduction Assessment

Yes, source-level: current `main` returns false for `isConnected=false` and `isNodeConnected=true`, while `NodeRuntime` can display that same state as `Connected (operator offline)`. I did not run a device or emulator onboarding pass in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `OnboardingFlow` and `OnboardingFlowLogicTest`; the evidence does not point to a gateway, auth, or protocol change.
