## Label Justifications

- `P1`: Multiple users are blocked in Android onboarding with no workaround when the node is connected but the operator session is offline.
- `impact:session-state`: The bug is caused by the onboarding UI conflating operator-session connectivity with device-node readiness.

## Summary

Keep open: current `main` and latest release `v2026.5.18` still require both operator and node connectivity before Android onboarding can finish, so the reported `Connected (operator offline)` state remains blocked. The discussion includes multiple affected users and a focused fix attempt, but the target-repo PR was closed unmerged and the remaining open same-head PR is only in a fork.

## What This Changes

The issue requests that Android onboarding finish when the device node is paired and connected even if the operator session is offline.

## Best Possible Solution

Ship a small Android app fix that treats node connectivity as sufficient for final onboarding completion, keeps disconnected-node states blocked, updates focused unit coverage, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes, source-level: current `main` returns false for `isConnected=false` and `isNodeConnected=true`, while `NodeRuntime` can display that same state as `Connected (operator offline)`. I did not run a device or emulator onboarding pass in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `OnboardingFlow` and `OnboardingFlowLogicTest`; the evidence does not point to a gateway, auth, or protocol change.
