## Summary

Keep open: current main and the latest release still require both operator and node connectivity before Android onboarding can finish, so the reported `Connected (operator offline)` state remains blocked. The target-repo fix PR at https://github.com/openclaw/openclaw/pull/61269 was closed unmerged, while the same source work remains visible in an external fork PR, so this needs a narrow replacement or revival rather than cleanup closure.

## What This Changes

The issue asks Android onboarding to allow completion when the node is paired and connected even if the operator session is offline.

## Best Possible Solution

Recreate or revive the focused Android fix in the target repository so final onboarding depends on node connectivity, keeps node-disconnected states blocked, removes stale operator-only state usage, and updates the focused logic test plus release notes.

## Reproduction Assessment

Yes, at source level: current main returns false for `canFinishOnboarding(isConnected = false, isNodeConnected = true)`, while `NodeRuntime` explicitly displays `Connected (operator offline)` for that runtime condition. I did not run a live Android onboarding flow in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is an Android app-side finish-gate change and focused unit test update. It should not change gateway pairing, operator auth, reconnect timing, or protocol behavior.
