## Summary

Keep open. Current main still has the Kimi Coding-specific replay gap: native subagent spawn can carry a thinking override into the child run, but the bundled Kimi provider wrapper only sets top-level `thinking` and does not preserve or synthesize `reasoning_content` on replayed assistant tool-call messages. The open Moonshot PR #74846 is useful context, but it does not touch the bundled `extensions/kimi-coding` path reported here. Item URL: https://github.com/openclaw/openclaw/issues/70563

## What This Changes

The issue requests that native subagent spawn succeed with Kimi/Kimi Coding thinking enabled instead of failing when assistant tool-call history replay lacks `reasoning_content`.

## Best Possible Solution

Add a narrow Kimi Coding provider replay repair: when Kimi thinking is enabled, preserve existing provider reasoning metadata or synthesize a Kimi-accepted `reasoning_content` value on replayed assistant tool-call messages, with focused tests for Kimi enabled/disabled thinking and preservation behavior. Keep the behavior provider-owned, extracting a small shared helper only if it avoids duplicating the existing DeepSeek/Moonshot-compatible pattern without broad core subagent changes.

## Reproduction Assessment

Yes. The issue gives a concrete reproduction path: use Kimi/Kimi Coding with thinking enabled, spawn a native subagent, and force a child tool-call continuation that replays assistant tool-call history. Current main source inspection confirms the path can forward thinking into the child while the Kimi wrapper still lacks `reasoning_content` replay repair; live Kimi validation remains the remaining check.

## Solution Assessment

Yes for a narrow provider-owned repair; no for closing as already fixed. The maintainable solution is to patch Kimi's outgoing payload/replay wrapper, or a small shared provider helper it calls, rather than adding a core subagent workaround or relying on the Moonshot-only PR #74846.
