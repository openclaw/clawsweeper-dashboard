## Summary

Keep this issue open. Current main still appears to have the reported Kimi Coding replay gap: native subagent spawning forwards a thinking override into the child run, and the bundled Kimi provider can enable top-level thinking, but its payload wrapper still does not add or preserve `reasoning_content` for replayed assistant tool-call messages. The related #50178 and #74846 are useful context, but they do not currently cover the `extensions/kimi-coding` provider path reported here.

## What This Changes

Fix Kimi/Kimi Coding native subagent spawns so thinking-enabled assistant tool-call replay includes the required `reasoning_content` metadata instead of failing and falling back.

## Best Possible Solution

Keep the fix in the Kimi Coding provider boundary: when Kimi thinking is enabled, patch the outbound replay payload so assistant tool-call messages carry Kimi-accepted `reasoning_content`, while preserving existing provider metadata and keeping the default `off` thinking policy unchanged.

## Reproduction Assessment

Yes. The issue gives a concrete path: enable thinking for `kimi/kimi-code` or `kimi/kimi-for-coding`, spawn a native subagent, and force a tool-call continuation so assistant tool-call history is replayed. Current source inspection confirms that path forwards thinking but lacks Kimi-specific `reasoning_content` replay repair.

## Solution Assessment

No. Current main is not the best solution yet because the Kimi wrapper only sets top-level `thinking`; the narrow maintainable fix is a provider-owned payload patch, analogous to the existing DeepSeek replay backfill, not a core subagent workaround.
