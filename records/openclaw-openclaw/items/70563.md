## Summary

Keep open; current main still exposes the reported provider-owned failure path. Subagent spawning can pass a thinking override into the child run, and the Kimi Coding provider marks the models as reasoning-capable but only writes top-level `thinking` without repairing replayed assistant messages that lack `reasoning_content`.

## What This Changes

The issue reports that native `sessions_spawn` subagents fail for Kimi thinking-enabled runs when replayed assistant tool-call messages lack `reasoning_content`.

## Best Possible Solution

Add a Kimi Coding provider replay repair that preserves existing `reasoning_content` and backfills a Kimi-accepted value on assistant replay messages when thinking is enabled.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `sessions_spawn` can enable thinking for the child run, while the Kimi wrapper sends `thinking: { type: "enabled" }` without repairing assistant replay messages.

## Solution Assessment

Yes, the narrow provider-owned fix is the best direction; patch Kimi Coding payload replay rather than adding provider-specific behavior to core subagent spawning.
