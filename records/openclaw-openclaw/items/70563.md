## Summary

Keep open; current main still has a source-reproducible Kimi Coding replay bug. The Kimi provider can enable thinking for `kimi/kimi-code`, but its payload wrapper only sets top-level `thinking` and does not preserve or backfill `reasoning_content` on replayed assistant tool-call messages, which Kimi’s own docs require for thinking tool-call conversations. ([platform.moonshot.ai](https://platform.moonshot.ai/docs/guide/kimi-k2-5-quickstart))

## What This Changes

The issue requests that native subagent spawns continue working when Kimi thinking is enabled and the replayed tool-call history needs `reasoning_content`.

## Best Possible Solution

Add a Kimi Coding provider replay repair that preserves existing `reasoning_content` and backfills an accepted value when thinking is enabled, with focused provider tests and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible. The current Kimi wrapper can send `thinking: { type: "enabled" }` without repairing replayed assistant tool-call messages, while Kimi’s documented tool-use contract requires `reasoning_content` in that context. ([platform.moonshot.ai](https://platform.moonshot.ai/docs/guide/kimi-k2-5-quickstart))

## Solution Assessment

Yes for a narrow provider-owned fix, not for closing as already fixed. The maintainable path is to patch the Kimi Coding payload wrapper, reusing or extracting a shared replay helper only if it avoids duplicating the existing DeepSeek shape.
