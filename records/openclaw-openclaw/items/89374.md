## Summary

Keep open: current main fixed the release-era CLI native compaction ownership error, but the broader timeout-compaction recovery path can still treat any compacted result as a successful retry without proving the Codex channel session has enough headroom to recover.

## What This Changes

The issue asks OpenClaw to stop reporting timeout compaction as recovered when a Codex-backed channel session remains high-context and unrecoverable.

## Best Possible Solution

Add a bounded Codex timeout-compaction recovery policy that verifies post-compaction headroom or marks the session unrecoverable with explicit reset/rollover handling through the existing Codex binding and session reset contracts.

## Reproduction Assessment

No. The issue has coherent production logs and current source reproduces the decision path, but I did not establish a deterministic current-main live Discord/Codex session repro.

## Solution Assessment

Unclear. The narrow fix should not blindly delete session-store entries; it should first define when compaction failed to restore safe headroom and then use existing Codex reset or explicit recovery messaging semantics.
