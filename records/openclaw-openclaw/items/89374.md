## Summary

Keep open: current main and v2026.6.8 fix narrower Codex automatic-compaction skip handling, but the central timeout-compaction recovery problem still remains source-backed: OpenClaw can log successful timeout compaction, retry the same high-pressure session, and fall through after repeated timeouts without a proved safe rollover/reset.

## What This Changes

This issue asks OpenClaw to recover, clearly report, or safely roll over Codex-backed channel sessions when timeout compaction reports success but leaves the session unrecoverable.

## Best Possible Solution

Keep this as the canonical timeout-compaction recovery tracker, define the safe session rollover/reset policy for unrecoverable Codex channel sessions, and prove a high-context Codex channel session recovers without repeated timeout-compaction loops.

## Reproduction Assessment

No live current-main repro was run here. The code path and tests are source-reproducible, and the issue has multiple release-era field reports with sanitized logs for the same timeout-compaction and unrecovered-session shape.

## Solution Assessment

No, current main is only a partial solution. It fixes the narrower Codex-owned compaction skip and some native-thread pressure cases, but it still lacks the maintainer-approved recovery policy that proves the affected channel session is safe or rolls it over.
