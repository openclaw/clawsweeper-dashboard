## Summary

Keep open. The report is recent and concrete: current main has protections for exact duplicate WebChat submits and redundant final-event reloads, but the code still routes every accepted fresh `chat.send` into `dispatchInboundMessage` and then through runtime plugin loading, and the provided logs describe repeated accepted dispatches rather than only a UI rendering duplicate. The related open WebChat/session issues overlap but do not cleanly supersede this repeated-dispatch/provider-initialization pattern.

## What This Changes

The issue asks maintainers to investigate why Control UI/WebChat repeatedly dispatches the same `agent:main:main` session and reloads provider/plugin runtime work for trivial turns.

## Best Possible Solution

Add a focused WebChat/Gateway trace or regression that sends one trivial message to `agent:main:main` and proves exactly one `chat.send`, one accepted dispatch, and one agent run across normal, reconnect, and post-final event timing. Then fix the specific duplicated dispatch source or plugin-registry reuse miss without broadening core plugin policy; treat `memory_search` and `session_status` as separate tool-policy tuning unless the trace shows they are only being invoked because of duplicate dispatch.

## Reproduction Assessment

No high-confidence repo-local reproduction exists yet. The issue body provides concrete log signatures and environment details, and current tests cover exact same-idempotency duplicates, but there is no deterministic current-main test or live trace for repeated fresh dispatches of `agent:main:main` from Control UI after a trivial message.

## Solution Assessment

Unclear. Existing submit dedupe, idempotency handling, and final-event reload guards are necessary but not sufficient evidence; the narrow maintainable path is to capture the dispatch/run/plugin-load sequence first, then decide whether the fix belongs in Control UI submit/reconnect handling, Gateway dispatch idempotency, plugin registry reuse, or tool policy.
