## Summary

Keep open. Current `main` still has a source-visible path where repeated WebChat sends with fresh idempotency keys can dispatch separate `agent:main:main` runs, and open PR #76446 explicitly targets this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports repeated WebChat/Control UI dispatches of `agent:main:main` that repeatedly initialize providers/plugins and make trivial replies very slow.

## Best Possible Solution

Review and land #76446, or replace it with an equivalent narrow Gateway guard that dedupes active internal WebChat text sends without collapsing attachments, slash commands, explicit delivery routes, or completed intentional repeats.

## Reproduction Assessment

Yes from source inspection: on current `main`, two same-session WebChat `chat.send` text turns with distinct idempotency keys while the first run is active can both register and dispatch. The reporter comments provide concrete logs for the same `agent:main:main` dispatch pattern, but I did not run a live Gateway reproduction here.

## Solution Assessment

Yes. A narrow active-run dedupe keyed to internal WebChat session/message while the original run is still active is the best apparent fix boundary; broader provider/runtime latency work is useful but does not by itself prevent repeated fresh dispatches.
