## Summary

Keep open. The report is fresh, concrete, and current main still only proves same-key/in-flight dedupe; it does not prove that repeated fresh Control UI dispatches of `agent:main:main` are fixed. #72892 is closely related, but this report adds provider/plugin reload and tool-use amplification evidence that could be a broader dispatch or registry-cache path.

## What This Changes

WebChat / Control UI should avoid repeated dispatch of `agent:main:main`, repeated provider/plugin initialization, and heavy tool work for a single trivial message.

## Best Possible Solution

Add a focused WebChat/Gateway trace or regression for one trivial Control UI message to `agent:main:main` that proves exactly one `chat.send`, one accepted dispatch, one agent run, and expected plugin registry reuse across normal, reconnect, and post-final-event timing. Then fix the narrow duplicated dispatch or registry reuse miss; treat `memory_search` and `session_status` as separate tool-policy tuning unless the trace shows they only happen because of duplicate dispatch.
