## Summary

Keep open because an open PR explicitly fixes this issue and current main still only dedupes `chat.send` by the caller-provided idempotency key, so the issue should not be closed before that PR merges.

## What This Changes

The report asks OpenClaw to stop WebChat/Control UI from repeatedly dispatching the same `agent:main:main` turn and reloading providers/plugins for trivial sends.

## Best Possible Solution

Review and land PR #76446, or an equivalent narrow gateway active-send dedupe that collapses duplicate internal WebChat text sends while preserving commands, attachments, explicit routes, non-internal origins, and later intentional repeats.

## Reproduction Assessment

No high-confidence live reproduction was run here. The reporter's logs plus current source give a focused source reproduction: two active internal WebChat `chat.send` calls for the same session/message with different idempotency keys can bypass current main's same-key dedupe and dispatch twice.

## Solution Assessment

Yes, the linked PR's described direction appears to be the narrow maintainable fix: dedupe active duplicate internal WebChat text sends in the gateway rather than broadly changing provider loading or tool policy. Current main alone is not sufficient because it only dedupes same-key retries.
