## What This Changes

The issue reports repeated Control UI/WebChat dispatches for `agent:main:main`, causing repeated provider/plugin initialization, slow replies, and streaming stalls.

## Best Possible Solution

Review #76446 as the canonical fix path, ensuring the final implementation dedupes duplicate active internal WebChat text sends with distinct idempotency keys without collapsing slash commands, attachments, explicit routes, or intentional later repeats.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live repo-local reproduction was established here, but the reporter logs and current `chat.send` source give a clear source-level path for duplicate active sends with fresh idempotency keys to dispatch more than once.

## Solution Assessment

Is this the best way to solve the issue? Yes for the main repeated-dispatch symptom: a narrow Gateway active-send dedupe like #76446 is more robust than relying only on a single Control UI submit guard, but the PR still needs normal review and merge.
