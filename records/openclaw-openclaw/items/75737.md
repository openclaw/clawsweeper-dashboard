## Summary

Keep open. The report has concrete logs and current main still only dedupes `chat.send` by the caller's idempotency key; the focused fix is in open, unmerged PR #76446, which explicitly targets this issue.

## What This Changes

The issue asks OpenClaw to stop WebChat/Control UI from repeatedly dispatching `agent:main:main` and repeatedly reinitializing provider/plugin runtime for trivial sends.

## Best Possible Solution

Land or revise PR #76446, or an equivalent narrow gateway active-send dedupe fix with regression coverage, then close this issue after the fix is on main.

## Reproduction Assessment

No high-confidence live reproduction was run in this read-only review. The focused duplicate-dispatch path is source-reproducible: send the same WebChat text into the same session with a different idempotency key while the first run is active, and current main has no same-session/message active-run guard.

## Solution Assessment

Yes for the central repeated-dispatch bug: the linked PR's narrow gateway active-run dedupe is the right fix boundary. The broader reconnect/model-list latency symptoms should remain follow-up evidence unless that PR's validation proves they are covered too.
