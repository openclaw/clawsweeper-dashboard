## Summary

Keep open. The report is recent, concrete, and not covered by a terminal current-main fix: main has guards for exact duplicate in-flight submits and repeated idempotency keys, but every accepted fresh `chat.send` still enters `dispatchInboundMessage` and the runtime plugin-loading path. The related WebChat issues overlap, but none cleanly supersedes this repeated accepted-dispatch/provider-initialization pattern.

## What This Changes

The issue requests investigation and a fix for WebChat/Control UI repeatedly dispatching `agent:main:main`, reinitializing providers/plugins, and making trivial replies very slow.

## Best Possible Solution

Add a focused WebChat/Gateway trace or regression that sends one trivial message to `agent:main:main` and proves exactly one `chat.send`, one accepted dispatch, and one agent run across normal send, reconnect, and post-final event timing. Then fix the specific duplicate source or plugin-registry reuse miss without broad changes to memory/tool policy unless the trace shows those calls are caused by duplicate dispatch.

No high-confidence repo-local reproduction exists yet. The report includes concrete log signatures and environment details, while current tests only cover exact same-idempotency duplicate sends, not repeated fresh Control UI dispatches of `agent:main:main`.

## Solution Assessment

Unclear. The narrow maintainable approach is to capture the dispatch/run/plugin-load sequence first, then decide whether the fix belongs in Control UI submit/reconnect handling, Gateway idempotency/session state, runtime plugin reuse, or separate tool-policy tuning.
