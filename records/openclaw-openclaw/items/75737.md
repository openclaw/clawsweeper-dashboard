## Summary

Keep open. Current main still dedupes `chat.send` only by the client idempotency key, and open PR #76446 explicitly closes this issue but has not merged, so the issue should stay open until that PR lands or is superseded.

## What This Changes

Ensure a Control UI/WebChat message dispatches only once for `agent:main:main` without repeated provider/plugin initialization from duplicate active sends.

## Best Possible Solution

Review and land or revise #76446 so duplicate active WebChat text sends with distinct idempotency keys reuse the in-flight run while preserving attachments, slash commands, explicit delivery routes, and non-internal origins.

## Reproduction Assessment

No high-confidence live reproduction was run here. The reporter and independent commenter provide concrete logs, and current source shows the same-id-only dedupe path, but I did not establish a failing WebChat/Gateway run locally.

## Solution Assessment

Yes for the active duplicate-send dispatch path: a narrow Gateway active-send dedupe is more maintainable than changing provider loading or adding configuration. It is not yet proven to cover the reconnect/model-list streaming stall, so that symptom may need a follow-up if #76446 does not resolve it.
