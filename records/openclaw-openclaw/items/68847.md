## Summary

Keep open. Current main still has the reported leak path: maybeWakeNodeWithApns stores wake state before APNs registration lookup, returns no-registration without deleting that entry, and the only observed delete path is tied to registered node disconnect cleanup. There is also an open same-author implementation PR, #68848, that explicitly closes this issue, so the issue should stay open while maintainers review or close that PR.

## What This Changes

The issue requests cleanup of nodeWakeById when maybeWakeNodeWithApns returns no-registration for an unregistered or stale nodeId.

## Best Possible Solution

Review #68848 as the canonical fix candidate. If accepted, land it or an equivalent no-registration cleanup with a regression test proving unregistered wake attempts do not leave nodeWakeById entries; if rejected, keep this issue open until that same behavior is addressed in the gateway wake path.

## Reproduction Assessment

Yes. A focused unit reproduction can mock loadApnsRegistration to return null, call maybeWakeNodeWithApns with one or more unregistered nodeIds, and observe that current main leaves nodeWakeById populated because no delete runs on that branch.

## Solution Assessment

Yes. The narrow maintainable fix belongs at the no-registration early return or an equivalent wake-state helper cleanup, with a focused regression test; reviewing #68848 is the best current path because it already targets exactly that behavior.
