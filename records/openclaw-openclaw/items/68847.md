## Summary

Keep open. Current main still has the reported no-registration leak path, and the issue is paired with open same-author PR #68848 that explicitly closes it, so cleanup should wait for maintainer review or landing of that PR rather than closing this issue early.

## What This Changes

The issue requests cleanup of speculative node wake state so unregistered APNs node IDs do not leave permanent `nodeWakeById` entries after `no-registration` wake attempts.

## Best Possible Solution

The gateway wake path should delete speculative wake state on the `no-registration` path, or route that branch through an equivalent helper cleanup, with a focused regression test proving repeated unregistered wake attempts leave no `nodeWakeById` entries.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `loadApnsRegistration` to return null, call `maybeWakeNodeWithApns` with unregistered node IDs, and inspect the exported `nodeWakeById` map; current source leaves entries because the only delete path is registered disconnect cleanup.

## Solution Assessment

Yes. The narrow maintainable solution is a no-registration cleanup plus regression coverage, and #68848 is the current canonical implementation candidate because it explicitly targets that behavior.
