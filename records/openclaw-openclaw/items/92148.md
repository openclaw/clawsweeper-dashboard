## Summary

Keep open: current main does not contain close-level proof that this latest-release DeepSeek provider-routing regression is fixed, and the remaining question is security-sensitive because it concerns sending a provider key across the wrong upstream boundary.

## What This Changes

The issue reports that OpenClaw v2026.6.5 sends a configured DeepSeek model to the OpenAI Responses API instead of DeepSeek OpenAI-compatible chat completions, causing HTTP 401 failures.

## Best Possible Solution

Preserve the documented DeepSeek `openai-completions` route to `api.deepseek.com` for explicit `models.providers.deepseek` overrides, add focused regression coverage for the empty-baseUrl/explicit-model case, then verify with `openclaw models status --probe` using real DeepSeek credentials.

## Reproduction Assessment

No high-confidence live reproduction is available from this read-only review because DeepSeek credentials were not available. Source and docs support the reported routing concern, but the exact current-main 401 path still needs a packaged/live probe.

## Solution Assessment

No, closing is not the best way to solve this issue. The maintainable path is to fix provider/model resolution so configured DeepSeek rows keep the documented chat-completions endpoint, then prove the release/user path with live provider validation.
