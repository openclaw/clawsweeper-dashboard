## What This Changes

The issue requests fixing a regression where a custom/direct DeepSeek OpenAI-compatible provider is routed to OpenAI Responses instead of DeepSeek chat completions, producing HTTP 401.

## Best Possible Solution

Keep this open until a provider owner proves current main with a DeepSeek credential or lands a targeted fix/test that prevents non-OpenAI providers from using the OpenAI Responses endpoint unless explicitly configured for a compatible backend.

## Reproduction Assessment

## Solution Assessment

Unclear: a narrow routing fix may be enough, but the best permanent solution should enforce provider endpoint/API compatibility so third-party credentials cannot silently cross into the OpenAI Responses transport. The adjacent blank-baseUrl fix is relevant but not sufficient proof for this DeepSeek-specific report.
