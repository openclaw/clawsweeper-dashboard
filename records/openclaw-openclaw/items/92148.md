## Summary

Keep open: this is a concrete latest-release provider-routing regression report, and current main does not contain a proven fix. Source and docs say direct DeepSeek should stay on OpenAI-compatible Chat Completions, while the reported log shows the Responses transport path with the DeepSeek key.

## What This Changes

The issue asks OpenClaw to stop routing `deepseek/deepseek-v4-pro` through the OpenAI Responses API in v2026.6.5 and restore the previous DeepSeek Chat Completions route.

## Best Possible Solution

Diagnose the release/package model-resolution path and make `deepseek/deepseek-v4-pro` keep `openai-completions` against `api.deepseek.com` for the reported config, then add focused catalog/transport regression coverage plus a live DeepSeek probe before closing.

## Reproduction Assessment

No high-confidence current-main reproduction was established in this read-only review. The issue has concrete v2026.6.5 logs, but source inspection found the intended DeepSeek catalog route still set to `openai-completions`, so a packaged/live probe is needed.

## Solution Assessment

No, closing or treating the user workaround as enough would be premature. The maintainable fix should preserve the documented DeepSeek Chat Completions route in provider/model resolution rather than requiring users to force routing manually.
