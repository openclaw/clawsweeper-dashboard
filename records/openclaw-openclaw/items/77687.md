## Summary

This looks like a valid current-main doctor false warning: the Gateway auth health check detects that a SecretRef is configured, but it calls the raw auth resolver before materializing that ref, so a resolvable SecretRef can still be treated as a missing token.

## What This Changes

Doctor should stop reporting a configured Gateway auth token SecretRef as unavailable when the SecretRef resolves successfully.

## Best Possible Solution

Materialize the active Gateway auth SecretRef for doctor’s read-only health check, then keep the existing no-plaintext-overwrite behavior only for truly unresolved refs.

## Reproduction Assessment

Yes, we have a high-confidence source-level reproduction path: current main calls resolveGatewayAuth on raw SecretRef-backed config, and that resolver ignores object SecretRefs. I did not run a live exec provider, so this is source_reproducible rather than reproduced.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing Gateway auth SecretRef materialization/resolution path before deciding the token is unavailable. The fix should remain read-only and must not persist the resolved token into config.
