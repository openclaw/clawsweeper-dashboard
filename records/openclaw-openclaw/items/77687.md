## Summary

Keep open: current main still has the source-level doctor false-warning path, and an open PR with closing syntax already targets this issue, so the issue should wait for that PR to merge or be superseded by the related canonical fix work.

## What This Changes

The issue reports that `openclaw doctor` warns `gateway.auth.token` SecretRefs are unavailable even when the SecretRef resolves and Gateway token auth works.

## Best Possible Solution

Land or consolidate the focused doctor fix so resolved SecretRefs suppress the false warning, unresolved refs still warn, and exec refs are either verified through an explicit policy or reported as unverified rather than unavailable.

## Reproduction Assessment

Yes. Current source gives a high-confidence path: configure local token auth with a resolvable `gateway.auth.token` SecretRef and run `openclaw doctor`; the doctor path checks raw auth before SecretRef materialization and reaches the warning branch.

## Solution Assessment

Yes, resolving through the existing SecretRef-aware Gateway token helper is the narrow maintainable direction. The remaining product/security judgment is how doctor should treat exec SecretRefs without silently running side-effecting providers.
