## Summary

Keep open: current `main` still reaches the false unavailable-warning path for SecretRef-backed Gateway tokens, and https://github.com/openclaw/openclaw/pull/77698 is already the paired open fix path but has not merged.

## What This Changes

The issue asks `openclaw doctor` to resolve or recognize a configured `gateway.auth.token` SecretRef before warning that the Gateway token is unavailable.

## Best Possible Solution

Doctor should use the SecretRef-aware Gateway token path and report exec refs as explicitly unverified or opt-in rather than unavailable, with focused regression coverage for resolved SecretRefs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure local token auth with a resolvable `gateway.auth.token` SecretRef and run `openclaw doctor`; `resolveGatewayAuth()` ignores the SecretRef object and the warning branch remains reachable.

## Solution Assessment

Unclear. Reusing the existing SecretRef-aware Gateway token resolver is the right direction, but the final fix should preserve safe exec handling and avoid calling an intentionally unverified exec ref unavailable.
