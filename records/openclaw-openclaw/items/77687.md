## Summary

Keep open: current main still has a source-reproducible doctor bug where the gateway-auth health check detects a SecretRef but does not resolve it before deciding the token is unavailable.

## What This Changes

Doctor should not warn that a SecretRef-backed `gateway.auth.token` is unavailable when the configured SecretRef resolves successfully.

## Best Possible Solution

Reuse the existing gateway auth SecretRef materialization/resolution helper in the doctor gateway-auth health check, preserving the current read-only warning only for genuinely unresolved SecretRefs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: token mode plus a raw SecretRef reaches `resolveGatewayAuth`, which intentionally ignores SecretRef object values and makes doctor emit the unavailable warning. I did not perform a live CLI run during this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to use the existing gateway auth SecretRef resolver before the doctor auth decision rather than adding new config or changing SecretRef semantics.
