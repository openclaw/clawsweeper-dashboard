## Summary

Keep open: the linked bug is credible and source-reproducible, but this PR is not safe to merge because it widens plugin-root containment instead of fixing the explicit SDK resolver/package-entry path and still lacks real packaged-run proof.

## What This Changes

This PR changes `src/plugins/loader.ts` to allow plugin entry, setup-runtime entry, and CLI metadata boundary failures when the escaped realpath appears to belong to a package named `openclaw`.

## Best Possible Solution

Repair host SDK peer imports through explicit SDK/native resolver or package-entry handling, keep plugin entry containment fail-closed, add negative containment tests, and provide redacted packaged Codex-enabled sub-agent proof.

## Reproduction Assessment

Yes from source, but not from a live packaged run in this review. Current main and v2026.6.9 keep the strict loader rejection, runtime registry loading throws on plugin load errors, the Codex plugin imports OpenClaw SDK subpaths, and docs/tests show the host peer symlink shape.

## Solution Assessment

No. This PR is a plausible workaround, but it broadens plugin entry and metadata containment; the safer maintainable fix belongs in explicit SDK resolver or package-entry handling.
