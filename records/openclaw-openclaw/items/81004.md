## Summary

Keep open. Current main still logs Claude live-session turns with provider, model, duration, and raw line count only, and the cron `before_agent_reply` synthetic path returns directly without a comparable cli-backend turn log, so the observability gap is source-confirmed.

## What This Changes

Request adding per-turn output byte count and a short SHA-256 response hash to gateway cli-backend logs, including Claude live-session and synthetic reply paths.

## Best Possible Solution

Add bounded, content-safe CLI backend turn diagnostics through a shared helper used by real Claude live completions and synthetic hook replies.

## Reproduction Assessment

Yes. Source inspection on current main shows the successful Claude live-session log omits output bytes/hash, and the handled synthetic reply path bypasses that log entirely.

## Solution Assessment

Yes. The narrow maintainable fix is to compute byte length and a short hash from the final assistant text at both completion sites, while never logging raw response content.

## Repair Work Prompt

Implement https://github.com/openclaw/openclaw/issues/81004 as a focused diagnostic logging change. Add a small shared helper in the CLI runner logging surface to compute UTF-8 byte length and a 12-hex SHA-256 prefix from final assistant text without logging the text itself. Update `src/agents/cli-runner/claude-live-session.ts` so successful `finishTurn` logs append `outBytes=<n>` and `outHash=<prefix>` for the parsed assistant output. Update `src/agents/cli-runner.ts` so the cron `before_agent_reply` handled branch emits an equivalent `agent/cli-backend` turn-style log for synthetic replies, with provider/model context, durationMs, a clear synthetic marker, and outBytes/outHash; preserve the existing behavior that claimed hooks do not prepare or execute the CLI backend. Add focused tests that capture or mock `cliBackendLog` for both the live-session and `before_agent_reply` paths, including a repeated text/hash assertion, and assert raw assistant text is not logged. Add a CHANGELOG.md entry under the active Changes section because gateway log diagnostics are user-visible. Do not add config flags, do not change session lifecycle behavior, and do not log prompts, raw responses, credentials, or session identifiers.
