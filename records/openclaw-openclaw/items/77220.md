## Summary

Keep open: the PR targets a valid CLI diagnostic bug, but the current diff would misclassify the documented `openclaw browser` CLI as an agent tool when `plugins.allow` excludes `browser`, and the current head still lacks the required changelog entry.

## What This Changes

The branch adds manifest `contracts.tools` lookup to missing CLI command diagnostics, wires it into `runCli`, and adds unit tests for agent-tool names under restrictive `plugins.allow`.

## Best Possible Solution

Land the targeted manifest-backed agent-tool diagnostic after guarding real plugin IDs/CLI command roots such as `browser`, adding regression coverage for that collision, and restoring the single-line Unreleased `### Fixes` changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main lacks tool-owner lookup before the `plugins.allow` fallback, and the PR diff shows a deterministic new collision for the bundled browser plugin because `browser` is both a CLI surface and a declared agent tool.

## Solution Assessment

No, not as currently written. Manifest metadata is the right narrow source for agent-tool ownership, but the branch needs to preserve diagnostics for real plugin command roots before merge.
