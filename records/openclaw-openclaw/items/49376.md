## Summary

Keep open: current main still lacks the requested config-backed retry policy for main LLM requests, and open implementation candidates already cover overlapping parts of the work.

## What This Changes

Add configurable LLM retry/backoff for main conversation requests on transient provider failures such as overloaded, server, network, and timeout errors.

## Best Possible Solution

Land or revise one canonical implementation that adds a bounded, documented LLM retry policy integrated with existing failover classification and proves permanent failures are not retried.

## Reproduction Assessment

Yes for the missing behavior: source inspection on current main shows no agent/defaults or model-provider retry config and no failover decision that retries transient LLM errors with a configurable policy. A live Anthropic 529 is not deterministic, but the missing configuration and control path are directly visible in the schema and runner code.

## Solution Assessment

Unclear for the open implementation candidates: #49800 is a narrow hardcoded overloaded-only mitigation, while #61443 is broader provider-level retry and still open. The best maintainable direction is a first-class, bounded config policy that reuses the existing classifier and is reviewed against duplicate/non-idempotent request risks.
