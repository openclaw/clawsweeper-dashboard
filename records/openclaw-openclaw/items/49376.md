## Summary

Keep open: current main and v2026.6.10 still lack the requested configurable main-LLM retry/backoff contract. Existing behavior covers SDK short retries/caps, cron/channel retries, auth-profile overload rotation, and narrow same-model rate-limit retry, while the open related PRs are partial groundwork rather than the requested public config surface.

## What This Changes

The issue requests an opt-in agent/global LLM retry-with-backoff config for transient provider failures such as Anthropic 529, 503, 500, network, and timeout errors.

## Best Possible Solution

Settle and document the opt-in retry contract, then implement it in the embedded runner/fallback layer with additive config, upgrade-safe schema/docs/tests, and safeguards for idempotency, Retry-After, notification, and fallback ordering.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source and docs inspection against current main and v2026.6.10 shows the requested configurable main-LLM retry policy is absent.

## Solution Assessment

Unclear until maintainers choose the product contract. An opt-in retry policy is plausible, but it must fit existing SDK retry caps, auth-profile cooldowns, model fallback, timeout behavior, idempotency, and notification semantics.
