## Summary

Keep open: current main and v2026.6.10 still do not expose a configurable main-LLM retry/backoff policy for transient provider failures. Existing behavior covers cron retry, channel retry, auth-profile overload rotation/backoff, and narrow same-model rate-limit or idle-timeout retries, which does not satisfy the requested public config contract.

## What This Changes

Request an agent-level or global configurable retry-with-backoff policy for main LLM requests on transient provider errors such as overloaded/529, server errors, network failures, and timeouts.

## Best Possible Solution

Settle and document the opt-in retry contract, then implement it in the embedded runner/fallback layer with additive config, focused upgrade-safe tests, and docs.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection against current main and v2026.6.10 shows the requested configurable main-LLM retry policy is absent.

## Solution Assessment

Unclear until maintainers choose the product contract. An opt-in retry policy is plausible, but it must fit existing SDK retry caps, auth-profile cooldowns, model fallback, timeout behavior, and notification semantics.
