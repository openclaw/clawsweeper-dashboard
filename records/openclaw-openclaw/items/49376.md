## Summary

Keep open: current main and v2026.6.6 still lack configurable retry/backoff for main LLM provider requests, while existing code only provides retry helpers plus auth-profile/fallback handling for overloaded conditions. Related implementation attempts are closed or not merge-ready, so this issue remains the canonical tracker.

## What This Changes

Request configurable agent/global or provider-level retry with backoff for main LLM requests on transient provider errors such as Anthropic 529, 503, and 500.

## Best Possible Solution

Keep this as the canonical tracker and land one opt-in retry contract only after maintainers settle scope, retry classes, timing caps, Retry-After behavior, notifications, and fallback interaction.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main has retry helpers and provider failover/cooldown controls, but no configurable main LLM provider retry policy.

## Solution Assessment

Unclear until maintainers choose the public retry contract. An opt-in provider or agent-level policy is plausible, but it must preserve request timeouts, SDK retry-after caps, auth-profile cooldowns, model fallback, and notification semantics.
