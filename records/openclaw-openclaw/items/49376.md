## Summary

Keep this issue open. Current main still has transient-error classification, SDK retry-after caps, auth-profile rotation, model fallback, and cron retry prior art, but it does not expose the requested configurable main LLM retry policy with retry classes, custom backoff array, max attempts, max wall-clock cap, or retry notifications. The issue is also paired with open implementation candidates, including #49800 using closing syntax, so it should not be closed before maintainer review.

## What This Changes

The issue requests configurable retry with backoff for main LLM conversation requests on transient provider errors such as overloaded, server_error, network, and timeout.

## Best Possible Solution

Add a bounded first-class LLM retry policy, likely under agent defaults or a global model-runtime config, that reuses existing failover classification, supports explicit retry classes, custom backoff delays, max attempts, abort-aware max wall-clock behavior, and optional user-visible retry notification, with docs and focused runner tests proving it does not retry permanent failures.

## Reproduction Assessment

Yes for the missing behavior: current main can be inspected or unit-tested by simulating overloaded/server/timeout failover paths and confirming they rotate profile, fall back, or surface errors without any llmRetry config. A real provider 529 incident is not deterministic, but the relevant gateway behavior is visible in the failover code and schema.

## Solution Assessment

Unclear for the open implementation candidates: #49800 appears to be a useful narrow mitigation, but it is hardcoded and overloaded-only, while this issue asks for configurable retry classes and wall-clock/backoff controls. The maintainable end state should be an explicit config-backed policy integrated with the existing failover classifier rather than an unconfigurable retry loop hidden in one branch.
