## Summary

Keep open: current main and v2026.6.6 still lack a configurable retry/backoff policy for main LLM provider requests. Existing retry infrastructure, SDK retry-after caps, and auth-profile/fallback cooldown controls are related but do not provide the requested public retry contract; the nearest implementation path is still the open, conflicting provider-retry PR at https://github.com/openclaw/openclaw/pull/61443.

## What This Changes

The issue requests configurable LLM request retry with backoff for transient provider errors such as overloaded, server error, network, and timeout failures.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers choose the LLM retry contract and either repair/land the related provider-retry PR or replace it with a narrower implementation that preserves timeout, fallback, auth-cooldown, schema, and docs behavior.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms the missing capability: current main exposes retry helpers, SDK retry-after caps, and overloaded fallback controls, but not configurable LLM provider request retry.

## Solution Assessment

Unclear until maintainers choose the public retry contract. An opt-in provider or agent-level retry policy is plausible, but it must preserve existing timeout, fallback, auth-cooldown, and SDK retry behavior.
