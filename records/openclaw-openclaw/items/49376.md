## Summary

Keep open: current main and v2026.6.8 still do not expose a configurable main-LLM retry/backoff policy for transient provider failures; the existing code covers SDK retry-after caps, auth-profile overload rotation/backoff, and disabled session-runtime retry, which is narrower than this request. The linked cross-repo PR is open and unmerged, and it proposes a hardcoded same-profile overload retry rather than the public config contract requested here.

## What This Changes

The issue requests configurable retry/backoff for main LLM requests on transient provider errors such as overloaded/529, 5xx, network, and timeout failures.

## Best Possible Solution

Keep this issue as the canonical configurable LLM retry/backoff tracker until maintainers choose the public config contract and decide whether https://github.com/BingqingLyu/openclaw/pull/995 should land, narrow, or close.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/config request. Source inspection against current main and v2026.6.8 confirms the requested configurable main-LLM retry policy is absent and the generic session retry is disabled for embedded OpenClaw runs.

## Solution Assessment

Unclear until maintainers choose the product contract. An opt-in retry policy is plausible, but it must be designed around existing SDK retry caps, auth-profile cooldowns, model fallback, timeout behavior, and user notification semantics.
