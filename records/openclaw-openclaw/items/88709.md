## Summary

Keep open: this is a useful auth-provider bug fix with current-head terminal proof, but the PR still over-classifies non-inline auth as inline-key auth, so a stale inline cooldown can block healthy env-first/profile-reference credentials.

## What This Changes

The PR adds provider-level cooldown tracking for inline `models.providers.<id>.apiKey` credentials, gates cooled inline credentials during auth resolution/provider visibility, preserves inline cooldown usage state, and adds auth/runner regression coverage.

## Best Possible Solution

Narrow inline-key cooldown detection to actual literal/env-marker/env SecretRef provider credentials, keep stored profiles and unrelated env-first credentials usable, then land once maintainers accept the fail-closed inline-key contract.

## Reproduction Assessment

Yes. Current main is source-reproducible: no-profile auth/billing failures return before marking cooldown, and provider auth availability accepts inline keys without consulting cooldown state; I did not trigger a live paid-provider 402.

## Solution Assessment

No, not yet. Provider-scoped cooldown is the right repair direction, but the current classifier is too broad and should distinguish real inline credentials from profile-reference or unrelated env-first auth before merge.
