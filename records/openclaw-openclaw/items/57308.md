## Summary

Keep this PR open. Current main still has the Slack interaction liveness gap: block-action and modal lifecycle handlers enqueue system events without waking the routed session, and the PR is a narrow implementation candidate with matching tests and docs. It should get maintainer review because #61374 overlaps the block-action wake area and adds thread-session routing, but this PR is not obsolete and has no concrete blocking review or security finding from this sweep.

## What This Changes

The PR adds successful-enqueue-gated `exec-event` heartbeat wakes for Slack block-action and modal lifecycle system events, extends Slack interaction tests, exposes `requestHeartbeatNow` through heartbeat SDK runtime surfaces, and updates SDK docs plus the changelog.

## Best Possible Solution

Land this PR or a combined Slack interaction fix that wakes the same routed Slack session only after the system event is actually queued, keeps unauthorized and deduped paths quiet, preserves modal lifecycle coverage, and reconciles the block-action thread-routing work from #61374.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: Slack block actions and modal lifecycle events can reach `enqueueSystemEvent` without any `requestHeartbeatNow` call, and a focused unit test can mock both functions to prove the missing wake.

## Solution Assessment

Yes, with one merge-path caveat. Gating `requestHeartbeatNow({ reason: "exec-event", sessionKey })` on the boolean result from `enqueueSystemEvent` is the narrowest maintainable fix for this liveness bug, but the block-action piece should be reconciled with #61374's thread-routing overlap before merge.
