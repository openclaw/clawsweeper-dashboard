## Summary

Keep open. The PR addresses a real current-main Slack liveness bug, but it is not merge-ready because it changes the public Plugin SDK export surface without updating the tracked API baseline hash, and its block-action portion overlaps with the still-open #61374 thread-routing PR.

## What This Changes

The PR updates Slack block-action and modal lifecycle handlers to request a targeted heartbeat wake after successfully queued system events, adds Slack interaction wake tests, exposes the wake helper through Plugin SDK runtime subpaths, and updates docs plus the changelog.

## Best Possible Solution

Land this PR or a combined replacement that keeps the enqueue-success-gated wake for Slack block actions and modal lifecycle events, updates the public SDK baseline for any intentional export change, avoids expanding deprecated compatibility shims unnecessarily, and reconciles the block-action/thread-routing overlap with #61374 before merge.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: Slack block actions and modal lifecycle events can call enqueueSystemEvent without any requestHeartbeatNow call, and the focused interaction unit tests can mock enqueue and wake to lock the behavior.

## Solution Assessment

Mostly yes, but not as submitted. Calling requestHeartbeatNow({ reason: "exec-event", sessionKey }) only after enqueueSystemEvent returns true is the narrow maintainable fix, but the public SDK export needs its baseline hash and the block-action overlap with #61374 should be resolved.
