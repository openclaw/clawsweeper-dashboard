## Summary

Keep open. Current main still routes `channels.telegram.*` config edits through a whole Telegram channel stop/start, and the Telegram bot still creates a fresh in-memory pending group-history map per bot instance, so the reported state loss remains valid despite the related restart-drain mitigation in #71465.

## What This Changes

The issue asks OpenClaw to stop Telegram config hot reloads from destroying in-memory group context, either by applying safe values in place, preserving state across restarts, or restarting only for settings that truly require it.

## Best Possible Solution

Add a Telegram-owned state-retention path, or a generic channel reload seam if maintainers prefer, so Telegram config reloads either apply safe values in place or carry pending group-history state across required stop/start cycles with account/token isolation and trimming to the new `historyLimit`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: change `channels.telegram.historyLimit`, watch it match the Telegram reload prefix, flow into `plan.restartChannels`, call whole-channel stop/start, and recreate `groupHistories` as an empty map in the new bot instance.

## Solution Assessment

Unclear. #71465 is a useful drain mitigation, but it is not the complete fix for Telegram-local pending history; the narrow maintainable repair is state preservation for Telegram group history unless maintainers choose a broader channel reload API.
