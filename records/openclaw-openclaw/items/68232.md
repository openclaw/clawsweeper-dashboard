## Summary

Keep open: current main and the latest release still classify runtime-only Telegram config edits under the broad Telegram channel reload prefix, and the open closing PR that targets this state-loss path has not merged.

## What This Changes

The issue reports that runtime-only Telegram config edits during hot reload restart the Telegram channel and discard in-memory conversation context and group history.

## Best Possible Solution

Land a narrow Telegram reload-classification fix that keeps transport-affecting settings restart-backed while making runtime-only Telegram settings hot no-ops with regression coverage.

## Reproduction Assessment

Yes, at source level: the broad `channels.telegram` reload prefix matches `channels.telegram.historyLimit`, the planner adds Telegram to `restartChannels`, and the reload handler stops and starts that channel. I did not run a live Telegram bot session in this read-only review.

## Solution Assessment

Yes, the best fix direction is a narrow reload-classification change with regression coverage. The issue should stay open until the linked PR lands or an equivalent current-main fix replaces it.
