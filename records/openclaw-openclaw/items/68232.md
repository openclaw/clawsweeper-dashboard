## Summary

Keep open. Current main still routes Telegram config-prefix changes through a channel restart, and each restarted Telegram bot still creates a fresh bot-local pending group-history map, so the merged restart-drain work reduces active-run interruption but does not retain Telegram-local context.

## What This Changes

The issue asks for Telegram config hot reloads to stop destroying in-memory conversation context, especially pending group history and cache-ttl-style context.

## Best Possible Solution

Preserve or hand off Telegram pending group history across restart-required Telegram config reloads, scoped by account/token and history key and trimmed to the new effective `historyLimit`.

## Reproduction Assessment

Yes, source-level: changing `channels.telegram.historyLimit` follows the current path from Telegram reload prefix to `plan.restartChannels`, then `stopChannel`/`startChannel`, then a new bot with an empty `groupHistories` map. I did not run a live Telegram group in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff for pending group history during restart-required reloads; the already-merged restart-drain work is useful but does not retain this bot-local map.
