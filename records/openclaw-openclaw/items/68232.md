## Summary

Current main still has the source-level Telegram state-loss path. The merged restart-drain work reduces active-run disruption, but a `channels.telegram.*` hot reload still restarts the Telegram channel and the replacement bot still starts with a fresh pending group-history map.

## What This Changes

The issue reports that editing `openclaw.json` restarts the Telegram channel and drops Telegram-local in-memory conversation context.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped by account/token/chat/topic, and trim retained entries to the new effective `historyLimit`.

## Reproduction Assessment

Yes, by source inspection. Changing a `channels.telegram.*` path such as `channels.telegram.historyLimit` still reaches `plan.restartChannels`, `stopChannel`/`startChannel`, and a newly constructed Telegram bot with an empty `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff during restart-required Telegram reloads; gateway-level restart draining is useful but does not preserve this bot-local map.
