## Summary

Current main still has the source-level Telegram state-loss path, so this should stay open. The merged restart-drain work reduces active-run damage, but Telegram config changes still restart the channel and each new bot still owns a fresh pending group-history map.

## What This Changes

The issue asks OpenClaw to avoid losing Telegram in-memory group context when config hot reload restarts the Telegram channel.

## Best Possible Solution

Preserve or transfer Telegram pending group history across restart-required Telegram config reloads, scoped by account/token and history key, and trim it to the new effective `historyLimit`.

## Reproduction Assessment

Yes, by source inspection. Changing a `channels.telegram.*` path such as `channels.telegram.historyLimit` follows the current-main path into `plan.restartChannels`, `stopChannel`/`startChannel`, and a newly constructed Telegram bot with an empty `groupHistories` map.

## Solution Assessment

Yes. The narrow maintainable fix is Telegram-owned state retention or handoff during restart-required Telegram reloads; gateway-level restart draining is useful but does not preserve this bot-local map.
